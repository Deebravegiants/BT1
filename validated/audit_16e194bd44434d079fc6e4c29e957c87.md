Audit Report

## Title
Daily Fee Mint Limit Hard Revert Blocks All Price Updates and Permanently Reduces Treasury Fee Yield - (File: contracts/LRTOracle.sol)

## Summary

`_checkAndUpdateDailyFeeMintLimit` unconditionally reverts when `currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay`. Because this check is embedded in `_updateRsETHPrice()`, both `updateRSETHPrice()` and `updateRSETHPriceAsManager()` revert for the remainder of the 24-hour window once the cap is hit. The stale `rsETHPrice` used as `previousTVL` in the next period's fee calculation causes the treasury to permanently receive slightly less rsETH than it would have if updates had proceeded normally.

## Finding Description

`updateRSETHPrice()` (L87–89) and `updateRSETHPriceAsManager()` (L94–96) both call `_updateRsETHPrice()` with no divergent code path. Inside `_updateRsETHPrice()`, when `protocolFeeInETH > 0`, the computed `rsethAmountToMintAsProtocolFee` is passed to `_checkAndUpdateDailyFeeMintLimit` at L303. At L205–206, if `currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay`, the function reverts with `DailyFeeMintLimitExceeded`. This revert unwinds the entire call stack before `rsETHPrice = newRsETHPrice` at L313 is reached, leaving the price stale.

When the period resets the next day, `_updateRsETHPrice()` computes `previousTVL = rsethSupply * rsETHPrice` using the stale price. Because `rsETHPrice` is lower than the true value, `previousTVL` is understated, `rewardAmount` is overstated, and the fee in ETH terms is correct — but `rsethAmountToMintAsProtocolFee = protocolFeeInETH / newRsETHPrice` divides by a higher `newRsETHPrice` than would have applied had the intermediate updates occurred. The treasury therefore receives fewer rsETH tokens in total than it would have across the equivalent unblocked updates. This delta is permanent and unrecoverable.

`setMaxFeeMintAmountPerDay` (L132–135) allows the manager to raise the cap, but this is a privileged action and does not retroactively mint the lost fee for the blocked period.

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.** The rsETH fee that should have been minted to the treasury during the blocked window is never minted. The next period's fee calculation recaptures the ETH-denominated yield but divides by a higher rsETH price, yielding a permanently smaller rsETH amount for the treasury. The shortfall is proportional to the price appreciation that occurred during the blocked window and cannot be recovered after the period resets.

## Likelihood Explanation

`updateRSETHPrice()` is a public, permissionless function. As protocol TVL grows, staking rewards accrue continuously. If `maxFeeMintAmountPerDay` is set conservatively relative to TVL, multiple calls within a single 24-hour window can cumulatively push `currentPeriodMintedFeeAmount` past the cap through entirely normal operations — no attacker action is required. The likelihood increases monotonically with TVL growth and call frequency.

## Recommendation

Return a capped fee amount from `_checkAndUpdateDailyFeeMintLimit` instead of reverting, and use that capped value for the actual mint:

```solidity
function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal returns (uint256 cappedFee) {
    if (block.timestamp >= feePeriodStartTime + 1 days) {
        currentPeriodMintedFeeAmount = 0;
        feePeriodStartTime = getCurrentPeriodStartTime();
    }
    uint256 remaining = maxFeeMintAmountPerDay > currentPeriodMintedFeeAmount
        ? maxFeeMintAmountPerDay - currentPeriodMintedFeeAmount : 0;
    cappedFee = feeAmount > remaining ? remaining : feeAmount;
    currentPeriodMintedFeeAmount += cappedFee;
}
```

Use `cappedFee` in `_updateRsETHPrice()` for the mint, allowing the price update to always proceed regardless of cap state.

## Proof of Concept

1. Deploy with `maxFeeMintAmountPerDay = X` rsETH.
2. Call `updateRSETHPrice()` repeatedly within one 24-hour window. After enough calls, `currentPeriodMintedFeeAmount` reaches `X`.
3. TVL continues to grow (staking rewards). The next call computes `rsethAmountToMintAsProtocolFee > 0`.
4. `_checkAndUpdateDailyFeeMintLimit` at L205 evaluates `currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay` → `true` → reverts.
5. `rsETHPrice` at L313 is never updated. `highestRsethPrice` at L294–296 is never updated.
6. Every subsequent call to `updateRSETHPrice()` or `updateRSETHPriceAsManager()` reverts identically until `block.timestamp >= feePeriodStartTime + 1 days`.
7. After the period resets, the next successful call computes `previousTVL` from the stale `rsETHPrice`, captures the accumulated ETH yield correctly, but mints fewer rsETH to the treasury than would have been minted across the equivalent unblocked updates — the shortfall is permanent.

Foundry test outline: fork mainnet, set a low `maxFeeMintAmountPerDay`, warp time to accumulate yield, call `updateRSETHPrice()` until the revert, assert `rsETHPrice` is unchanged, warp past the reset, call again, assert treasury rsETH balance is less than the sum of per-period mints that would have occurred without the cap hit.
