Audit Report

## Title
Prior User's Leftover ETH Silently Consumed by Subsequent WETH Swap ŌĆö (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay` function in `PeripheryPayments` uses `address(this).balance` ŌĆö the router's total native ETH balance ŌĆö when settling WETH-input swaps. Because this balance is not scoped to the current transaction's `msg.value`, any ETH left over from a prior user's payable call is silently consumed to subsidize the next user's WETH input, causing direct, permanent loss of the prior user's funds.

## Finding Description
The `pay` function contains a three-branch hybrid native/ERC-20 settlement path for WETH: [1](#0-0) 

When `0 < address(this).balance < value`, the middle branch (lines 78ŌĆō81) fires: it wraps **all** of `address(this).balance` into WETH and transfers it to the pool, then pulls only `value - nativeBalance` WETH from `payer`. The critical flaw is that `address(this).balance` is not scoped to the current transaction's `msg.value` ŌĆö it includes ETH deposited by any prior caller who sent ETH via a `payable` entry-point and has not yet called `refundETH`.

`exactInputSingle` is `payable` and stores no per-user ETH accounting: [2](#0-1) 

The `receive()` guard only blocks direct ETH pushes from non-WETH addresses; it does not prevent ETH from accumulating via `msg.value` in payable calls: [3](#0-2) 

`refundETH()` sends the entire `address(this).balance` to `msg.sender` with no per-user accounting, so if another user's swap already consumed the leftover ETH, the original depositor receives nothing: [4](#0-3) 

## Impact Explanation
Direct loss of user principal (ETH). User A sends 0.5 ETH for a 0.3 WETH amountIn swap; 0.2 ETH remains in the router. Before User A calls `refundETH`, User B calls `exactInputSingle(tokenIn=WETH, amountIn=1 ether)` with 0 ETH. The middle branch fires: User A's 0.2 ETH is wrapped and forwarded to the pool, and only 0.8 WETH is pulled from User B. User A's `refundETH` returns nothing ŌĆö 0.2 ETH is permanently lost. The pool receives the correct 1 WETH, so no pool insolvency occurs; the loss falls entirely on User A. This is a direct loss of user principal with no recovery path, meeting Critical/High severity thresholds.

## Likelihood Explanation
Any user who sends ETH with a WETH-input swap and does not atomically batch `refundETH` in the same `multicall` leaves ETH exposed. An attacker can passively monitor the router's ETH balance on-chain and issue a WETH swap immediately after detecting a non-zero balance, front-running the victim's `refundETH`. No privileged role, malicious pool, or non-standard token is required ŌĆö only two sequential calls from unprivileged users.

## Recommendation
Track only the ETH that arrived in the current transaction. Replace the `address(this).balance` read in `pay` with a value derived from `msg.value` passed down through the call stack, or record the pre-call balance at entry and use only the delta. Alternatively, require that WETH swaps using native ETH consume exactly `msg.value` and revert if `msg.value` does not cover the full `amountIn`, eliminating the partial-native hybrid path entirely.

## Proof of Concept
```solidity
function test_priorUserEthConsumedByWethSwap() public {
    // User A: send 0.5 ETH, swap WETH amountIn = 0.3 ether
    // After swap: router holds 0.2 ETH (user A's change, refundETH not yet called)
    uint256 routerEthBefore = address(router).balance; // 0.2 ether

    // User B: swap WETH amountIn = 1 ether, sends 0 ETH
    // pay() middle branch fires: wraps 0.2 ETH (user A's), pulls 0.8 WETH from user B
    vm.prank(userB);
    router.exactInputSingle(ExactInputSingleParams({
        tokenIn: WETH, amountIn: 1 ether, ...
    }));

    // User A calls refundETH ŌĆö receives nothing
    vm.prank(userA);
    router.refundETH();

    assertEq(userA.balance, initialUserABalance - 0.5 ether); // lost 0.2 ETH
    assertEq(address(router).balance, 0);
}
```
The pool receives exactly 1 WETH in both cases, confirming no pool-level invariant is broken ŌĆö the loss is borne entirely by User A whose 0.2 ETH was silently consumed.
