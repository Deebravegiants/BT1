Audit Report

## Title
Hardcoded `account != msg.sender` guard in `_processClaim` permanently freezes unclaimed KERNEL yield for contract recipients unable to self-call — (`contracts/KERNEL/KernelMerkleDistributor.sol`)

## Summary
`_processClaim` unconditionally reverts with `Unauthorized` when `account != msg.sender`, meaning only the account itself can initiate a claim. Any smart contract address included in the merkle tree that lacks an explicit outbound call to `claim` or `claimAndStake` on this distributor — including contracts written before the distributor was deployed — can never collect its allocated KERNEL yield. No admin rescue path, expiry sweep, or alternative claim route exists in the contract.

## Finding Description
At `contracts/KERNEL/KernelMerkleDistributor.sol` lines 311–313, `_processClaim` enforces:

```solidity
if (account != msg.sender) {
    revert Unauthorized();
}
```

Both public entry-points delegate to this function: `claim` at line 261 and `claimAndStake` at line 280. The guard makes `msg.sender == account` a hard precondition for every claim. For EOAs this is always satisfiable. For a smart contract it is satisfiable only if that contract contains logic that explicitly calls `claim`/`claimAndStake` on this specific distributor.

Contracts that legitimately appear in the merkle tree but lack such logic include:
- Contracts deployed before this distributor existed (no function targeting this address)
- Vaults or DAO treasuries with a generic `execute` function whose admin/owner keys are lost or unavailable
- Contracts that were upgraded and had the relevant function removed

The merkle proof at lines 320–323 already cryptographically binds `(index, account, cumulativeAmount)` to the leaf, and `kernel.safeTransfer(account, amountToSend)` at line 263 always sends funds to `account`, not to `msg.sender`. The identity check therefore provides no additional security while introducing a permanent denial-of-service for a class of valid recipients.

The admin functions (lines 356–423) provide no rescue path: there is no sweep function, no expiry mechanism, and no owner-callable override to release frozen allocations.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.** KERNEL tokens allocated to a contract-address recipient that cannot self-call `claim` remain locked in the distributor indefinitely. The tokens are not lost from the contract's balance but are unreachable by the intended recipient and by any third party, satisfying the definition of permanently frozen unclaimed yield within the allowed impact scope.

## Likelihood Explanation
The merkle tree is built off-chain from on-chain activity (deposits, staking, etc.). Any contract that interacted with the protocol — a vault, a DAO treasury, a proxy — may legitimately appear as a leaf. The `IMerkleDistributor` interface and protocol documentation impose no restriction against contract addresses as recipients. Contracts written before this distributor was deployed have no built-in function to call it. The likelihood that at least one such address appears in a live distribution is realistic and not theoretical.

## Recommendation
Remove the blanket identity check and rely solely on the merkle proof, which already cryptographically prevents unauthorized claims:

```solidity
// Remove lines 311-313 entirely
```

Alternatively, add an approved-operator mapping:

```solidity
mapping(address account => mapping(address operator => bool)) public approvedOperators;

if (account != msg.sender && !approvedOperators[account][msg.sender]) {
    revert Unauthorized();
}
```

Option 1 is simpler and has no security regression: the merkle proof binds the leaf to `account`, and funds are always transferred to `account`, not to `msg.sender`.

## Proof of Concept
Deploy a contract with no outbound call capability and include its address in the merkle tree:

```solidity
contract NoCallVault {
    receive() external payable {}
    // No function to call claim() on the distributor
}

contract PoC {
    KernelMerkleDistributor distributor;
    NoCallVault vault; // address included in merkle tree with valid proof

    function test() external {
        uint256 index = 1;
        uint256 amount = 1e18;
        bytes32[] memory proof = /* valid proof for (index, address(vault), amount) */;

        // Any caller other than vault itself reverts with Unauthorized
        try distributor.claim(index, address(vault), amount, proof) {
            revert("should have reverted");
        } catch { }

        // vault cannot call claim — permanently frozen
    }
}
```

The vault's allocation is provably unreachable: the only `msg.sender` that satisfies `account == msg.sender` is `address(vault)` itself, but `NoCallVault` contains no function that produces an outbound call to `claim`.
