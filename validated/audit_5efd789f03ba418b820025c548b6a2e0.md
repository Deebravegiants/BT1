### Title
Permissionless, Caller-Agnostic `verify_foreign_transaction` Fan-Out Queue Can Be Saturated by Any Unprivileged Caller to Block Legitimate Verification Requests — (`crates/contract/src/lib.rs`, `crates/contract/src/pending_requests.rs`)

---

### Summary

`verify_foreign_transaction()` is a permissionless, payable function whose pending-request map key does **not** include the caller's account ID. Any unprivileged account can submit 128 identical foreign-transaction verification requests to saturate the hard cap (`MAX_PENDING_REQUEST_FAN_OUT = 128`) for a specific `VerifyForeignTransactionRequest` key. Once the queue is full, every subsequent legitimate call for the same transaction fails with `PendingRequestQueueFull`, blocking the request lifecycle until the MPC nodes respond and drain the queue. The attacker also receives 128 copies of the MPC-signed verification response, which can be replayed in downstream bridge contracts.

---

### Finding Description

The `verify_foreign_transaction()` entry point is open to any NEAR account with a 1 yoctoNEAR deposit: [1](#0-0) 

The `VerifyForeignTransactionRequest` key stored in `pending_verify_foreign_tx_requests` is constructed from `(domain_id, chain, tx_id, confirmations, extractors)` — it does **not** incorporate the caller's `predecessor_account_id`. This is confirmed by the unit test that explicitly queues Alice's and Bob's identical submissions under the same map entry: [2](#0-1) 

The fan-out queue is capped at 128 entries. `push_pending_yield` panics (reverting the call) when the cap is exceeded: [3](#0-2) 

Contrast this with `sign()`, where the `SignatureRequest` key embeds `tweak = derive_tweak(predecessor_id, path)`, making each caller's request key unique: [4](#0-3) [5](#0-4) 

Because `verify_foreign_transaction` omits the caller from the key, an attacker can occupy all 128 slots for any target transaction before the legitimate caller submits.

---

### Impact Explanation

**Medium — request-lifecycle manipulation that breaks production safety/accounting invariants.**

1. **Selective blocking of bridge verifications.** An attacker can permanently prevent a specific foreign-chain transaction from being verified during any window they choose to maintain the attack. Bridge contracts that gate withdrawals or cross-chain settlements on a `verify_foreign_transaction` response are stalled for that transaction until the attacker stops re-filling the queue.

2. **Attacker receives 128 MPC-signed verification responses.** When the MPC nodes eventually respond, all 128 of the attacker's queued yield promises are resolved with the signed `VerifyForeignTransactionResponse`. The attacker holds 128 valid MPC signatures over the target transaction's payload hash. If any downstream bridge contract does not enforce single-use of such a response (e.g., lacks nonce/receipt tracking), the attacker can replay the response to trigger multiple executions of the same bridge action — a double-spend or duplicate-credit condition.

---

### Likelihood Explanation

- The attack requires only a NEAR account and ~128 × (1 yoctoNEAR + ~7 TGas) per cycle — economically trivial.
- The attacker can monitor the NEAR mempool for `verify_foreign_transaction` calls and front-run them, exactly as described in M-07 for the Yield protocol.
- No privileged access, no threshold collusion, and no TEE bypass is required.
- The queue drains naturally once MPC nodes respond, so the attacker must re-submit 128 calls per MPC response cycle to sustain the DoS — still cheap given NEAR's low gas costs.

---

### Recommendation

1. **Include `predecessor_account_id` in the `VerifyForeignTransactionRequest` key**, mirroring the design of `sign()` where the tweak binds the request to the caller. This makes each caller's queue slot independent, eliminating cross-caller queue saturation.

2. **Alternatively, enforce a per-account submission cap** for `verify_foreign_transaction` so a single account cannot occupy more than a small fraction of the 128-slot queue for any given request key.

3. **Downstream bridge contracts must enforce single-use of `VerifyForeignTransactionResponse`** (e.g., by recording the receipt ID or a nonce) to prevent replay of the 128 duplicate signed responses the attacker receives.

---

### Proof of Concept

```
// Attacker script (pseudo-NEAR CLI):
// Target: a specific Bitcoin tx_id that a bridge is about to verify.

for i in 0..128 {
    near call mpc-contract.near verify_foreign_transaction \
        '{"domain_id": 3, "payload_version": "V1",
          "request": {"Bitcoin": {"tx_id": "<TARGET_TX_ID>",
                                  "confirmations": 2,
                                  "extractors": ["BlockHash"]}}}' \
        --deposit 0.000000000000000000000001 \   // 1 yoctoNEAR
        --gas 300000000000000 \
        --accountId attacker.near
}

// Legitimate bridge call now fails:
near call mpc-contract.near verify_foreign_transaction \
    '{"domain_id": 3, "payload_version": "V1",
      "request": {"Bitcoin": {"tx_id": "<TARGET_TX_ID>",
                              "confirmations": 2,
                              "extractors": ["BlockHash"]}}}' \
    --deposit 0.000000000000000000000001 \
    --accountId bridge.near
// => panics: "Pending-request queue is full for this request key (limit: 128)."

// After MPC nodes respond, attacker holds 128 copies of the signed
// VerifyForeignTransactionResponse and can replay them in any bridge
// contract that lacks single-use enforcement.
```

```rust
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
```
