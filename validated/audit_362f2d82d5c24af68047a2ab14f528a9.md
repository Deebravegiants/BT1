### Title
Hardcoded `only_query = false` in `BroadcastedTransaction` → `ExecutableTransactionInput` Conversion Causes Wrong Transaction Hash and Version in Fee Estimation and Simulation - (`File: crates/apollo_rpc/src/v0_8/api/mod.rs`)

---

### Summary

The `TryFrom<BroadcastedTransaction> for ExecutableTransactionInput` conversion in the RPC layer hardcodes `only_query = false` for all `Invoke` and `DeployAccount` transactions submitted to `starknet_estimateFee` and `starknet_simulateTransactions`. The Starknet protocol requires that fee-estimation and simulation transactions set the 128th bit of the transaction version (the "query bit"), which changes the transaction hash and the version value reported to the contract via `get_execution_info`. Because the bit is never set, the RPC returns authoritative-looking but incorrect fee estimates and simulation traces for any contract whose behavior depends on detecting the query bit.

---

### Finding Description

In `crates/apollo_rpc/src/v0_8/api/mod.rs` lines 329–338, the conversion from a wire-level `BroadcastedTransaction` to the internal `ExecutableTransactionInput` hardcodes `false` for the `only_query` field for every `Invoke` and `DeployAccount` variant:

```rust
impl TryFrom<BroadcastedTransaction> for ExecutableTransactionInput {
    type Error = ErrorObjectOwned;
    fn try_from(value: BroadcastedTransaction) -> Result<Self, Self::Error> {
        // TODO(yair): pass the right value for only_query field.
        match value {
            BroadcastedTransaction::Declare(tx) => Ok(tx.try_into()?),
            BroadcastedTransaction::DeployAccount(tx) => Ok(Self::DeployAccount(tx.into(), false)),
            BroadcastedTransaction::Invoke(tx) => Ok(Self::Invoke(tx.into(), false)),
        }
    }
}
```

The same hardcoded `false` appears for `DeclareV1` in `TryFrom<BroadcastedDeclareTransaction>` at lines 506–508:

```rust
Ok(Self::DeclareV1(
    ...,
    // TODO(yair): pass the right value for only_query field.
    false,
))
```

The `only_query` flag is the direct analog of the `isDeposit` flag in the reported GMX bug. When `only_query = true`, `signed_tx_version` sets the 128th bit of the transaction version:

```rust
pub fn signed_tx_version(
    tx_version: &TransactionVersion,
    transaction_options: &TransactionOptions,
) -> TransactionVersion {
    let query_only_bit = *QUERY_VERSION_BASE;
    if transaction_options.only_query {
        TransactionVersion(tx_version.0 + query_only_bit)
    } else {
        *tx_version
    }
}
```

This modified version is then fed into `get_transaction_hash`, producing a different hash, and is also the value returned to the contract by the `get_execution_info` syscall. The Starknet specification defines that transactions submitted to `starknet_estimateFee` and `starknet_simulateTransactions` are query transactions and must carry the query bit. The `TransactionOptions` struct's own documentation confirms this:

> "Transaction that shouldn't be broadcasted to StarkNet … Using this flag will modify the transaction version by setting the 128-th bit to 1."

Fee estimation and simulation transactions are by definition not broadcast transactions; they are query transactions. The hardcoded `false` means the query bit is never set, so:

1. The transaction hash computed during estimation/simulation is the **broadcast hash**, not the **query hash**.
2. The version value exposed to the contract via `get_execution_info` is the regular version, not the query version.

The conversion is used directly by both `estimate_fee` and `simulate_transactions` in `api_impl.rs`:

```rust
let executable_txns =
    transactions.into_iter().map(|tx| tx.try_into()).collect::<Result<_, _>>()?;
```

---

### Impact Explanation

Any contract that inspects the transaction version via `get_execution_info` to detect whether it is being called in a query context (e.g., to skip side-effects, return mock data, or gate logic) will observe the regular version instead of the query version. This causes `starknet_estimateFee` and `starknet_simulateTransactions` to execute the contract under the wrong mode, producing incorrect fee estimates and simulation traces. The RPC response is authoritative-looking (no error is returned) but reflects execution of the wrong code path. This matches the allowed impact: **"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

---

### Likelihood Explanation

The bug is reachable by any unprivileged caller of the public `starknet_estimateFee` or `starknet_simulateTransactions` RPC endpoints. No special privileges or malicious peer access are required. The TODO comments in the source code confirm the developers are aware the value is wrong. The impact is realized whenever a contract under estimation or simulation branches on the query bit.

---

### Recommendation

Replace the hardcoded `false` with the correct `only_query` value derived from the incoming `BroadcastedTransaction`. For `starknet_estimateFee` and `starknet_simulateTransactions`, all broadcasted transactions are query transactions and should use `only_query = true`:

```rust
impl TryFrom<BroadcastedTransaction> for ExecutableTransactionInput {
    type Error = ErrorObjectOwned;
    fn try_from(value: BroadcastedTransaction) -> Result<Self, Self::Error> {
        match value {
            BroadcastedTransaction::Declare(tx) => Ok(tx.try_into()?),
            BroadcastedTransaction::DeployAccount(tx) => Ok(Self::DeployAccount(tx.into(), true)),
            BroadcastedTransaction::Invoke(tx) => Ok(Self::Invoke(tx.into(), true)),
        }
    }
}
```

Apply the same fix to the `DeclareV1` arm in `TryFrom<BroadcastedDeclareTransaction>`.

---

### Proof of Concept

1. Deploy a contract that reads its own transaction version via `get_execution_info` and branches on whether the query bit (`2^128`) is set — returning different values (e.g., `1` for query, `0` for non-query).
2. Call `starknet_estimateFee` with an `Invoke` transaction targeting that entry point.
3. Observe that the returned fee estimate reflects the non-query branch (version without bit 128 set), even though the Starknet spec requires the query bit to be set for fee estimation.
4. Confirm by calling `starknet_simulateTransactions` and inspecting the execution trace: the contract's `get_execution_info` returns a version without the query bit, proving `only_query = false` was used.

The existing test `simulate_with_query_bit_outputs_same_as_no_query_bit` in `crates/apollo_rpc_execution/src/execution_test.rs` (lines 750–771) asserts that results are identical regardless of the flag for a deprecated contract — but this only holds for contracts that do not inspect the query bit. For contracts that do inspect it, the results diverge, and the hardcoded `false` produces the wrong answer. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

pub fn signed_tx_version(
    tx_version: &TransactionVersion,
    transaction_options: &TransactionOptions,
) -> TransactionVersion {
    // If only_query is true, set the 128-th bit.
    let query_only_bit = *QUERY_VERSION_BASE;
    assert_eq!(
        tx_version.0.to_biguint() & query_only_bit.to_biguint(),
        BigUint::from(0_u8),
        "Requested signed tx version with version that already has query bit set: {tx_version:?}."
    );
    if transaction_options.only_query {
        TransactionVersion(tx_version.0 + query_only_bit)
    } else {
        *tx_version
    }
}
