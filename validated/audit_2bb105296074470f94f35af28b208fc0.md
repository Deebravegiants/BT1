### Title
Missing Cross-Sub-Transaction Deposit Refund Uniqueness Check Allows Double-Claiming of Stake Key Deposit - (`eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubLedgers.hs`)

---

### Summary

In the Dijkstra era, a top-level transaction may contain multiple sub-transactions, each processed sequentially by the `SUBLEDGERS` rule. Each sub-transaction may carry certificates, including `UnRegDepositTxCert`, which claims a stake-key deposit refund. No cross-sub-transaction guard prevents two sub-transactions from each claiming the same deposit refund for the same staking credential. The developers have explicitly acknowledged this gap with a disabled (`xit`) test whose expected predicate failure is marked `error "TODO: predicate failure not yet implemented"`. An unprivileged transaction author can therefore extract twice the deposited ADA from the deposit pot in a single top-level transaction, creating ADA from nothing.

---

### Finding Description

The Dijkstra era introduces nested transactions (`sub_transactions`, field `23` of `transaction_body`). The `SUBLEDGERS` rule folds over the list of sub-transactions, threading `LedgerState` (which includes `CertState`) from one sub-transaction to the next:

```haskell
dijkstraSubLedgersTransition = do
  TRC (env, ledgerState, subTxs) <- judgmentContext
  foldM
    ( \ls subTx ->
        trans @(EraRule "SUBLEDGER" era) $ TRC (env, ls, subTx)
    )
    ledgerState
    subTxs
``` [1](#0-0) 

Each `SUBLEDGER` invocation calls `SUBENTITIES` → `SUBCERTS` → `SUBCERT` (which reuses `Conway.conwayDelegTransition`) to process certificates, including `UnRegDepositTxCert cred deposit`. The `UnRegDepositTxCert` certificate deregisters a staking credential and returns its deposit to the transaction's balance. [2](#0-1) 

The developers have explicitly identified the missing guard in a disabled test:

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
  ...
  let
    subTx1 = mkBasicTx mkBasicTxBody
      & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
    subTx2 = mkBasicTx mkBasicTxBody
      & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
    tx = mkBasicTx mkBasicTxBody
      & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
  submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
``` [3](#0-2) 

The `xit` prefix marks the test as **pending/skipped**, and the body calls `error "TODO: predicate failure not yet implemented"` as the expected failure — confirming that no predicate failure currently rejects this transaction. The transaction is accepted by the ledger when it should be rejected.

The root cause is that no rule in the `SUBLEDGER` / `SUBENTITIES` / `SUBCERTS` / `SUBCERT` chain enforces that a given staking credential's deposit refund can be claimed at most once across all sub-transactions within a single top-level transaction. While `foldM` does thread the updated `CertState` between sub-transactions, the predicate that would detect the second deregistration attempt on an already-deregistered credential is explicitly noted as unimplemented. [4](#0-3) [5](#0-4) 

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

Attack accounting:
- Attacker pays `keyDeposit` once to register a staking credential.
- Attacker submits one top-level transaction containing two sub-transactions, each carrying `UnRegDepositTxCert stakingCred keyDeposit`.
- Each sub-transaction's balance equation includes `keyDeposit` on the consumed side, so each sub-transaction's outputs can be inflated by `keyDeposit`.
- Total ADA received in outputs: `2 × keyDeposit − fees`.
- Total ADA paid: `keyDeposit`.
- Net gain: `keyDeposit − fees` ADA created from nothing.

The deposit pot (`utxosDeposited` in `UTxOState`) is decremented twice for a deposit that was paid only once, producing an invalid ledger state. This constitutes a direct, attacker-controlled loss of ADA from the deposit pot (ultimately backed by reserves) and creation of unbacked ADA in UTxO outputs.

---

### Likelihood Explanation

**High.** The attack requires only:
1. Registering a staking credential (permissionless, costs `keyDeposit`).
2. Constructing a valid top-level Dijkstra transaction with two sub-transactions carrying the same `UnRegDepositTxCert`.

No governance majority, privileged key, or external dependency is needed. The entry path is fully attacker-controlled. The vulnerability is acknowledged in production test code with a `TODO` comment, confirming it is a known gap rather than a theoretical concern.

---

### Recommendation

1. **Implement the missing predicate failure** in the `SUBCERT` / `SUBDELEG` rule chain: when processing `UnRegDepositTxCert` in a sub-transaction, verify that the staking credential is still registered in the current (post-previous-sub-transaction) `CertState`. The `Conway.conwayDelegTransition` already performs this check for top-level transactions; ensure it is correctly enforced when the `CertState` has been mutated by a prior sub-transaction.

2. **Enable the pending test** (`xit` → `it`) in `Test.Cardano.Ledger.Dijkstra.Imp.CertSpec` once the predicate failure is defined, and add the correct expected failure constructor.

3. **Audit all other certificate types** (`UnRegDRepTxCert`, `RetirePoolTxCert`) for the same cross-sub-transaction double-claim pattern.

### Proof of Concept

```
Setup:
  stakingCred ← freshKeyHash
  registerStakeCredential stakingCred   -- pays keyDeposit once
  keyDeposit ← getsPParams ppKeyDepositL

Attack transaction:
  subTx1 = { certs: [UnRegDepositTxCert stakingCred keyDeposit]
            , inputs: [input1]
            , outputs: [addr1 ← value1 + keyDeposit - fee1] }

  subTx2 = { certs: [UnRegDepositTxCert stakingCred keyDeposit]
            , inputs: [input2]
            , outputs: [addr2 ← value2 + keyDeposit - fee2] }

  topTx = { sub_transactions: [subTx1, subTx2] }

Result (current, incorrect):
  topTx is accepted.
  addr1 and addr2 each receive an extra keyDeposit.
  Deposit pot is decremented by 2 × keyDeposit.
  Net ADA created: keyDeposit − fees.

Expected result (after fix):
  topTx is rejected with a predicate failure indicating
  stakingCred is not registered when subTx2 attempts deregistration.
```
