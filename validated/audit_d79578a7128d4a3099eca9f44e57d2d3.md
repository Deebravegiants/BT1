Audit Report

## Title
ETH Deposit Limit Bypass via Missing Amount in `_checkIfDepositAmountExceedesCurrentLimit` - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` omits the incoming `amount` from the ETH branch of its limit check, testing only `totalAssetDeposits > limit` instead of `totalAssetDeposits + amount > limit`. Any unprivileged caller can invoke `depositETH` with an arbitrarily large `msg.value` and bypass the `depositLimitByAsset[ETH_TOKEN]` cap as long as the current total has not yet crossed it, minting rsETH beyond the intended ceiling.

## Finding Description
The guard function at `contracts/LRTDepositPool.sol` L676–682 contains an asymmetric comparison:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct
}
``` [1](#0-0) 

The ETH branch never adds `amount` to `totalAssetDeposits`, so the check only fires if the pool is already over the limit before the deposit arrives. Any deposit that would push the total over the limit — but arrives while the total is still at or below the limit — passes the guard unconditionally.

The call chain is: `depositETH` (L76–93) → `_beforeDeposit` (L648–670) → `_checkIfDepositAmountExceedesCurrentLimit`. `depositETH` is `external payable` with no role restriction beyond `whenNotPaused` and `onlySupportedAsset`. [2](#0-1) [3](#0-2) 

## Impact Explanation
The `depositLimitByAsset` cap is the protocol's primary mechanism for bounding ETH intake. Bypassing it allows rsETH to be minted against ETH deposits that exceed the configured ceiling, breaking the protocol's stated deposit-cap invariant. This maps to **Low – contract fails to deliver promised returns**: the cap promise is violated and excess ETH may accumulate in the pool or NDCs without a corresponding EigenLayer strategy allocation, leaving it unrestaked. A path to **Medium – temporary freezing of funds** exists if the excess ETH cannot be forwarded downstream, but that secondary effect depends on strategy capacity and is not directly proven here.

## Likelihood Explanation
The path is fully permissionless. Any user who calls `depositETH{value: X}()` where `X` exceeds the remaining headroom under the cap triggers the bug. No special role, front-running, or external state manipulation is required. The condition is reachable whenever the ETH deposit limit is set to a finite value and the pool is unpaused.

## Recommendation
Unify the ETH and ERC20 branches to always include `amount`:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [1](#0-0) 

## Proof of Concept
1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 1000 ether`.
2. Prior deposits bring `getTotalAssetDeposits(ETH_TOKEN)` to exactly `1000 ether`.
3. Attacker calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500 ether)` evaluates `1000 ether > 1000 ether` → `false` → guard passes.
5. `_mintRsETH` mints rsETH for 500 ETH; total ETH deposits become `1500 ether`, 50% above the cap.
6. For comparison, `depositAsset(stETH, 500 ether, ...)` at the same state evaluates `1000 ether + 500 ether > 1000 ether` → `true` → correctly reverts with `MaximumDepositLimitReached`.

A Foundry fuzz test can confirm this by fuzzing `amount` over `[1, type(uint128).max]` with `totalAssetDeposits` pinned at the limit and asserting that `_checkIfDepositAmountExceedesCurrentLimit` returns `true` for all non-zero `amount` values — the current code will fail that assertion for every input.
