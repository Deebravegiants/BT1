Audit Report

## Title
Daily Fee Mint Limit Asymmetry Freezes rsETH Price Updates and Treasury Yield During Yield-Generating Periods — (`contracts/LRTOracle.sol`)

## Summary

`_checkAndUpdateDailyFeeMintLimit` uses a strict `>` comparison, so when `currentPeriodMintedFeeAmount == maxFeeMintAmountPerDay`, passing `feeAmount = 0` succeeds while passing any positive `feeAmount` reverts. Because `_updateRsETHPrice()` unconditionally calls `_checkAndUpdateDailyFeeMintLimit(0)` in the no-fee branch and `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` in the fee branch, every yield-generating price update reverts once the daily cap is exactly reached, permanently freezing both the treasury's unclaimed fee yield and the rsETH price until admin intervention.

## Finding Description

**Root cause — strict `>` in the limit guard:** [1](#0-0) 

When `currentPeriodMintedFeeAmount == maxFeeMintAmountPerDay`:
- `feeAmount = 0`: `X + 0 > X` → **false** → passes, `currentPeriodMintedFeeAmount` unchanged.
- `feeAmount > 0`: `X + ε > X` → **true** → reverts with `DailyFeeMintLimitExceeded`.

**Asymmetric call sites in `_updateRsETHPrice()`:** [2](#0-1) 

The `if (protocolFeeInETH > 0)` branch (line 303) passes the real fee amount; the `else` branch (line 310) unconditionally passes `0`. Once the cap is exactly reached, any call where TVL has grown reverts before reaching line 313 (`rsETHPrice = newRsETHPrice`), while any call where TVL is flat succeeds.

**Compounding effect:** Because `rsETHPrice` is never updated during yield-generating periods, `previousTVL = rsethSupply × rsETHPrice` stays stale. [3](#0-2) 

Each subsequent call accumulates a larger `rewardAmount`, producing a larger `rsethAmountToMintAsProtocolFee`. After the daily period resets (`currentPeriodMintedFeeAmount = 0`), the accumulated fee may already exceed `maxFeeMintAmountPerDay`, causing the revert to persist across multiple periods without admin intervention.

**`updateRSETHPriceAsManager()` hits the same path:** [4](#0-3) 

Both public and manager entry points call `_updateRsETHPrice()` and are equally affected.

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

The protocol fee rsETH that should be minted to the treasury is never minted during yield-generating periods once the daily cap is hit. Additionally, `rsETHPrice` is not updated, so `RSETHRateProvider.getLatestRate()` returns a stale rate to all cross-chain consumers. [5](#0-4) 

The compounding effect means the frozen yield grows over time, and the condition can persist across multiple daily periods without admin action to raise `maxFeeMintAmountPerDay`.

## Likelihood Explanation

No attacker action is required. Reaching the daily cap is the intended normal operation of the limit mechanism. Once reached within a period, any TVL increase (staking rewards, rebasing LSTs) causes every subsequent `updateRSETHPrice()` call to revert for the rest of that period. The compounding effect can extend the freeze across multiple periods. Any unprivileged external caller can observe and trigger the revert simply by calling `updateRSETHPrice()` when yield has accrued.

## Recommendation

Skip the `_checkAndUpdateDailyFeeMintLimit` call entirely when there is no fee to mint — there is nothing to enforce:

```solidity
if (protocolFeeInETH > 0) {
    uint256 rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice);
    _checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee);
    if (rsethAmountToMintAsProtocolFee > 0) {
        address treasury = lrtConfig.getContract(LRTConstants.PROTOCOL_TREASURY);
        IRSETH(rsETHTokenAddress).mint(treasury, rsethAmountToMintAsProtocolFee);
        emit FeeMinted(treasury, rsethAmountToMintAsProtocolFee);
    }
}
// No else branch — zero-fee updates don't touch the daily limit at all
rsETHPrice = newRsETHPrice;
```

Alternatively, add a `feeAmount > 0` guard inside `_checkAndUpdateDailyFeeMintLimit` so a zero-amount call never triggers the revert:

```solidity
if (feeAmount > 0 && currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
    revert DailyFeeMintLimitExceeded(...);
}
```

## Proof of Concept

Foundry fork/unit test outline:

```solidity
// 1. Deploy LRTOracle with maxFeeMintAmountPerDay = 1e18
// 2. Simulate prior fee minting: set currentPeriodMintedFeeAmount = 1e18
//    (achieved by calling updateRSETHPrice() until cap is reached)

// Case 1: TVL flat — succeeds
// totalETHInProtocol == previousTVL → protocolFeeInETH = 0
// _checkAndUpdateDailyFeeMintLimit(0): 1e18 + 0 > 1e18 → false → passes
vm.prank(anyone);
lrtOracle.updateRSETHPrice(); // ✓ succeeds, rsETHPrice updated

// Case 2: TVL increased by 1 wei of yield — reverts
// Increase mock asset price by 1 wei so totalETHInProtocol > previousTVL
// protocolFeeInETH > 0 → rsethAmountToMintAsProtocolFee = ε
// _checkAndUpdateDailyFeeMintLimit(ε): 1e18 + ε > 1e18 → true → DailyFeeMintLimitExceeded
vm.prank(anyone);
vm.expectRevert(LRTOracle.DailyFeeMintLimitExceeded.selector);
lrtOracle.updateRSETHPrice(); // ✗ reverts — price frozen, treasury fee never minted

// Case 3: Compounding — next day, accumulated fee > maxFeeMintAmountPerDay
// warp forward 1 day; currentPeriodMintedFeeAmount resets to 0
// but stale rsETHPrice means rewardAmount = N days of yield
// rsethAmountToMintAsProtocolFee > maxFeeMintAmountPerDay → still reverts
vm.warp(block.timestamp + 1 days);
vm.expectRevert(LRTOracle.DailyFeeMintLimitExceeded.selector);
lrtOracle.updateRSETHPrice(); // ✗ still reverts
```
