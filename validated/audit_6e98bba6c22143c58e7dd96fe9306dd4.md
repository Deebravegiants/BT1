### Title
DirectDeposits Coin Sum Absent from Both Sides of `validateValueNotConservedUTxO` — ADA Created from Thin Air - (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, a transaction's `DirectDeposits` field is applied to account balances by `applyDirectDeposits` inside `dijkstraEntitiesTransition`, but the total coin in that map is included on **neither** the consumed nor the produced side of `validateValueNotConservedUTxO`. An unprivileged transaction author can therefore set `DirectDeposits` to an arbitrarily large amount, pass the conservation check, and have that amount credited to attacker-controlled accounts — creating ADA from thin air.

---

### Finding Description

**Step 1 — Rule execution order in `dijkstraLedgerTransition`**

`dijkstraLedgerTransition` runs sub-rules in this order:

1. `SUBLEDGERS` (subtransactions)
2. `ENTITIES` — processes certs, withdrawals, and **direct deposits**
3. `GOV`
4. `UTXOW → UTXO` — runs `validateValueNotConservedUTxO` [1](#0-0) 

Critically, `UTXOW` is invoked with `DijkstraUtxoEnv slot pp (lsCertState ledgerState) originalUtxo` — the **original** pre-ENTITIES `certState` and the original UTxO. [2](#0-1) 

**Step 2 — `dijkstraEntitiesTransition` applies DirectDeposits unconditionally**

After processing certs (which can register new accounts), `dijkstraEntitiesTransition` validates that every target account exists, then calls `applyDirectDeposits`:

```haskell
pure $ certStateAfterCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
``` [3](#0-2) 

`applyDirectDeposits` simply adds each specified coin amount to the matching account balance with no source deduction: [4](#0-3) 

**Step 3 — `validateValueNotConservedUTxO` does not include DirectDeposits**

The conservation check in `dijkstraUtxoTransition`:

```haskell
runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
``` [5](#0-4) 

The **consumed** side is computed by `conwayConsumed` → `getConsumedDijkstraValue` → `getConsumedMaryValue`:

```
consumed = Σ(UTxO inputs) + Σ(withdrawals) + Σ(cert refunds)
``` [6](#0-5) 

The **produced** side is computed by `getProducedDijkstraValue` → `dijkstraProducedValue` → `conwayProducedValue`:

```
produced = Σ(outputs) + fee + Σ(cert deposits) + treasury_donation
``` [7](#0-6) [8](#0-7) 

`directDepositsTxBodyL` appears in **neither** formula. The `DirectDeposits` map is a new Dijkstra-era field that was never wired into the conservation equation. [9](#0-8) 

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

The total ADA tracked by `totalAdaPotsES` includes `rewardsAdaPot = sumBalancesAccounts (certState ^. certDStateL . accountsL)`. [10](#0-9) 

After `applyDirectDeposits` runs, `rewardsAdaPot` increases by `Σ(DirectDeposits)`. Because this increase is not offset by any decrease in UTxO, fees, deposits, treasury, or reserves, the total ADA in the ledger grows by exactly `Σ(DirectDeposits)` per transaction. An attacker can repeat this with any number of transactions, inflating the ADA supply without bound.

---

### Likelihood Explanation

The exploit requires only:
1. A funded UTxO input (any amount sufficient to cover fee)
2. A registered account address (can be self-registered via `RegTxCert` in the same tx, processed in ENTITIES before the direct-deposit check)
3. A crafted `DijkstraTxBody` with a non-empty `DirectDeposits` map

No governance majority, privileged key, or external dependency is needed. The Dijkstra era is a new era in this repository; if deployed, this would be immediately exploitable by any transaction author.

---

### Recommendation

Add `Σ(DirectDeposits)` to the **produced** side of the conservation equation so that the tx author must fund direct deposits from their inputs, exactly as fees and cert deposits are funded. Concretely, extend `dijkstraProducedValue` (or `conwayProducedValue` for the Dijkstra override) to include:

```haskell
<+> inject (fold . unDirectDeposits $ txBody ^. directDepositsTxBodyL)
``` [8](#0-7) 

This mirrors how `treasuryDonationTxBodyL` is already included on the produced side in `conwayProducedValue`. [11](#0-10) 

---

### Proof of Concept

```
Initial state:
  UTxO: { txin0 → 10 ADA }
  Accounts: {}  (empty)

Craft DijkstraTxBody:
  spendInputs    = { txin0 }
  outputs        = [ 9.8 ADA → any address ]
  fee            = 0.2 ADA
  certs          = [ RegTxCert attacker_credential ]   -- registers account
  directDeposits = { attacker_account_address → 1_000_000 ADA }

Execution:
  1. ENTITIES:
     - RegTxCert registers attacker_credential → Accounts = { attacker_credential: 0 }
     - directDepositsMissingAccounts check: attacker_account is now registered → passes
     - applyDirectDeposits: Accounts = { attacker_credential: 1_000_000 ADA }
  2. UTXO:
     - consumed = 10 ADA (inputs) + 0 (withdrawals) + 0 (refunds) = 10 ADA
     - produced = 9.8 ADA (outputs) + 0.2 ADA (fee) + 0 (deposits) = 10 ADA
     - validateValueNotConservedUTxO: 10 == 10 → PASSES

Post-state:
  UTxO:     { txout → 9.8 ADA }
  Accounts: { attacker_credential: 1_000_000 ADA }
  Fees pot: +0.2 ADA

Total ADA delta: +1_000_000 ADA (created from thin air)
```

The attacker spent 0.2 ADA in fees and received 1,000,000 ADA in their account. The conservation check passed because `DirectDeposits` is absent from both sides of the equation. [3](#0-2) [12](#0-11) [8](#0-7)
