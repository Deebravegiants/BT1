### Title
Single Byzantine Participant Can Corrupt `AppPublicKey` CKD Responses Due to Missing Cryptographic Output Verification — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function applies cryptographic output verification only for the `AppPublicKeyPV` variant of CKD requests. The `AppPublicKey` variant receives **no verification at all**. This asymmetry allows a single Byzantine attested participant — strictly below the signing threshold — to submit an arbitrary malicious CKD response that the contract accepts and delivers to the requesting user, bypassing the threshold security model entirely.

---

### Finding Description

**Root cause — two code paths, two different invariants**

The `respond` (signing) function and `respond_ckd` for `AppPublicKeyPV` both enforce cryptographic correctness of the submitted response before resolving the pending yield. `respond_ckd` for `AppPublicKey` does not.

In `respond` (signing), every submitted signature is verified against the expected derived public key before the yield is resolved: [1](#0-0) 

In `respond_ckd`, the match on `app_public_key` applies `ckd_output_check()` only for `AppPublicKeyPV`. The `AppPublicKey` arm is an explicit no-op: [2](#0-1) 

The `AppPublicKeyPV` arm calls `ckd_output_check`, which performs a BLS12-381 pairing check to verify the response is cryptographically consistent with the network public key and the user's app key pair: [3](#0-2) 

For `AppPublicKey`, the contract immediately calls `resolve_yields_for` with whatever `CKDResponse` the caller supplied — no pairing check, no hash check, no threshold agreement check.

**Attacker-controlled entry path**

1. Attacker is a legitimately attested participant (passes `assert_caller_is_attested_participant_and_protocol_active`).
2. A user submits `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(pk)`.
3. Attacker observes the pending request on-chain (NEAR is public).
4. Attacker calls `respond_ckd` with a `CKDResponse { big_y, big_c }` of their choosing — e.g., a G1 point for which they know the discrete log — before any honest participant responds.
5. The contract accepts the response (no verification branch executes), resolves the yield, and returns the attacker-chosen `big_y` / `big_c` to the user.
6. The user receives a derived key that the attacker controls.

The race is winnable: the attacker is an on-chain participant, can submit transactions with high gas priority, and the honest MPC nodes must complete an off-chain multi-round protocol before any of them can call `respond_ckd`. The attacker can act the moment the request appears on-chain.

---

### Impact Explanation

A single Byzantine participant (below the signing threshold) can produce a **confidential key derivation output without the required threshold participant authorization**. If the attacker crafts `big_y` to correspond to a G1 point for which they know the scalar, they effectively control the derived key the user receives. Any assets the user subsequently sends to an address derived from that key are under the attacker's control — constituting direct theft. Even without knowing the discrete log, the attacker can supply a random point, permanently freezing any funds the user locks under the corrupted derived key.

This breaks the core threshold invariant: the MPC system is designed so that no single participant below the threshold can unilaterally determine a cryptographic output. For signing and for `AppPublicKeyPV` CKD, the contract enforces this invariant on-chain. For `AppPublicKey` CKD, it does not.

---

### Likelihood Explanation

**High.** The attacker only needs to be a single attested participant — a legitimate role in the network. No collusion, no key leakage, no TEE break is required. The attack is a straightforward front-run of an on-chain pending request. The honest nodes must complete a multi-round off-chain protocol before submitting; the attacker can submit immediately after the request appears on-chain.

---

### Recommendation

Apply the same cryptographic output verification to `AppPublicKey` responses as is applied to `AppPublicKeyPV`. If a pairing-based check is not possible without the G2 component, one of the following mitigations should be applied:

1. **Require threshold agreement on-chain**: Collect `respond_ckd` calls from a threshold of distinct attested participants and only resolve the yield when a quorum submits the same `(big_y, big_c)` pair.
2. **Restrict submission to the designated leader**: Mirror the `start_keygen_instance` / `start_reshare_instance` pattern and only accept a `respond_ckd` call from the elected leader for the current epoch, reducing the attack surface to a single known node.
3. **Deprecate the unverifiable `AppPublicKey` variant** in favour of `AppPublicKeyPV`, which provides on-chain verifiability.

---

### Proof of Concept

```
1. Contract is Running with participants [A, B, C], threshold 2.
2. User calls request_app_private_key({ app_public_key: AppPublicKey(pk_user), ... }).
   → pending_ckd_requests[ckd_request] = [yield_id]
3. Attacker (participant A) immediately calls:
     respond_ckd(ckd_request, CKDResponse { big_y: G*s_attacker, big_c: G*r_attacker })
   where s_attacker, r_attacker are scalars known to the attacker.
4. Contract executes lib.rs:675 → AppPublicKey arm → no-op.
5. Contract executes lib.rs:684-688 → resolve_yields_for resolves the yield with the attacker's response.
6. User's callback receives { big_y: G*s_attacker, big_c: G*r_attacker }.
7. User derives an address from big_y and deposits funds.
8. Attacker uses s_attacker to spend those funds.
```

The honest participants B and C never get a chance to submit the correct response — the yield is already resolved.
