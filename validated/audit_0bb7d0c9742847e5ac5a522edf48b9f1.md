### Title
TDH2 Partial Decryption Proof Omits Ciphertext/Label Binding, Enabling Cross-Context Replay - (File: `src/cbmpc/crypto/tdh2.cpp`)

### Summary
The Schnorr-like proof of correct partial decryption in the TDH2 implementation hashes only `(Xi, Yi, Zi)` — omitting the ciphertext component `R1`, the label, and the public share `Qi` from the challenge. This is the direct analog of the Futureswap bug: a "signature" (proof) over generic data that carries no context binding, allowing it to be replayed across different ciphertexts and labels.

### Finding Description

In `private_share_t::decrypt`, the partial decryption proof challenge is computed as:

```cpp
ei = ro::hash_number(Xi, Yi, Zi).mod(q);   // line 89
```

where `Xi = x * R1`, `Yi = si * R1`, `Zi = si * G`. The hash does **not** include `R1` (the ciphertext ephemeral), the label `L`, the public share `Qi`, or any session/ciphertext identifier. [1](#0-0) 

The verifier in `check_partial_decryption_helper` reconstructs `Yi` and `Zi` from the supplied ciphertext's `R1`, then checks the same hash:

```cpp
const ecc_point_t& R1 = ciphertext.R1;
ecc_point_t Yi = fi * R1 - ei * Xi;
ecc_point_t Zi = fi * G - ei * Qi;
bn_t ei_test = ro::hash_number(Xi, Yi, Zi).mod(q);   // line 121
``` [2](#0-1) 

Because `R1` is not in the hash, any two ciphertexts sharing the same `R1` point will cause the same `(Xi, Yi, Zi)` triple and therefore the same `ei`. A partial decryption proof generated for ciphertext `ct1` (label `L1`) passes verification for ciphertext `ct2` (label `L2`) whenever `ct2.R1 == ct1.R1`.

**Concrete attack path (public API):**

1. Attacker calls `cbmpc_tdh2_encrypt(pk, m1, L1, &ct1)` choosing randomness `r`, so `ct1.R1 = r·G`. The attacker knows `r`.
2. Attacker submits `ct1` to the decryption service. Honest parties call `cbmpc_tdh2_partial_decrypt(share_i, ct1, L1, &pd_i)`, producing proofs `ei = H(Xi, Yi, Zi)` bound only to `R1 = r·G`.
3. Attacker constructs `ct2` with label `L2` (an unauthorized context) using the **same** `r`, so `ct2.R1 = r·G = ct1.R1`. Because the attacker knows `r`, they can compute `ct2.R2 = r·Gamma` and a valid ciphertext-level proof `(e2, f2)` for `ct2`. `ct2` passes `ciphertext_t::verify(pk, L2)`.
4. Attacker calls `cbmpc_tdh2_combine_additive(pk, pub_shares, L2, {pd_i}, ct2, &plain)`.
5. Inside `combine_additive`: `ciphertext.verify(pk, L2)` passes (ct2 was created with L2). Then `check_partial_decryption_helper(Qi, ct2, curve)` is called — it recomputes `Yi = fi·ct2.R1 - ei·Xi` which equals the original `Yi` (same `R1`), so `H(Xi, Yi, Zi) == ei`. The check passes.
6. `combine_additive` computes `V = Σ Xi = x·R1` and decrypts `ct2` under label `L2`. [3](#0-2) 

The public C API entry points are `cbmpc_tdh2_partial_decrypt` and `cbmpc_tdh2_combine_additive`: [4](#0-3) [5](#0-4) 

### Impact Explanation

`combine_additive` (and `combine` for the AC variant) accepts partial decryptions that were generated for a **different** ciphertext/label pair. The label is the primary access-control mechanism in TDH2 — it is supposed to bind each decryption to a specific authorized context. Because `check_partial_decryption_helper` does not verify that the proof was generated for the current ciphertext's `R1` or label, an attacker who controls the encryptor role can:

- Force the combine step to succeed under an unauthorized label `L2` using proofs obtained under authorized label `L1`.
- Supply an attacker-controlled plaintext (via `ct2`) that the system decrypts and acts upon as if it were legitimately authorized under `L2`.

This is a public-API-reachable validation bypass in TDH2 that causes the system to accept invalid cryptographic output (partial decryptions under the wrong label/ciphertext), matching the "High" impact tier.

### Likelihood Explanation

The attacker must be the encryptor of the original ciphertext `ct1` (to know `r` and reuse `R1`). This is a normal, unprivileged role in TDH2 — any party can call `encrypt`. No threshold collusion, no key leakage, and no transport compromise is required. The attack is fully reachable through the shipped C and C++ public APIs.

### Recommendation

Include `R1` (and optionally the label `L` and public share `Qi`) in the partial decryption challenge hash, matching standard TDH2 practice:

```cpp
// In private_share_t::decrypt:
ei = ro::hash_number(R1, Qi, Xi, Yi, Zi).mod(q);

// In check_partial_decryption_helper:
bn_t ei_test = ro::hash_number(R1, Qi, Xi, Yi, Zi).mod(q);
```

This binds each proof to the specific ciphertext ephemeral `R1` and the verifying public share `Qi`, preventing cross-ciphertext and cross-label replay.

### Proof of Concept

```
// Setup: n=2 additive TDH2 key, curve P-256
public_key_t pk; pub_shares_t Qi; vector<private_share_t> shares;
generate_additive_shares(2, pk, Qi, shares, curve_p256);

// Step 1: attacker encrypts ct1 under label "authorized"
bn_t r = curve_p256.get_random_value();
buf_t iv = gen_random(16); bn_t s = curve_p256.get_random_value();
ciphertext_t ct1 = pk.encrypt("secret-msg", "authorized", r, s, iv);

// Step 2: honest parties produce partial decryptions for ct1
partial_decryption_t pd0, pd1;
shares[0].decrypt(ct1, "authorized", pd0);   // ei0 = H(Xi0, Yi0, Zi0)
shares[1].decrypt(ct1, "authorized", pd1);

// Step 3: attacker creates ct2 with SAME r → same R1, but label "unauthorized"
bn_t s2 = curve_p256.get_random_value();
buf_t iv2 = gen_random(16);
ciphertext_t ct2 = pk.encrypt("attacker-payload", "unauthorized", r, s2, iv2);
// ct2.R1 == ct1.R1 == r*G

// Step 4: replay pd0, pd1 against ct2 — check_partial_decryption_helper passes
buf_t plain;
error_t rv = combine_additive(pk, Qi, "unauthorized", {pd0, pd1}, ct2, plain);
assert(rv == SUCCESS);   // ← passes; label "unauthorized" was never authorized
assert(plain == "attacker-payload");
```

```cpp
error_t combine_additive(const public_key_t& pub_key, const pub_shares_t& Qi, mem_t label,
                         const partial_decryptions_t& partial_decryptions, const ciphertext_t& ciphertext,
                         buf_t& plain) {
  error_t rv = UNINITIALIZED_ERROR;
  const auto& curve = pub_key.Q.get_curve();
  int n = int(Qi.size());
  for (const auto& _Qi : Qi) {
    if (rv = curve.check(_Qi)) return coinbase::error(rv, "combine_additive: check Qi failed");
  }
  if ((int)partial_decryptions.size() != n) return coinbase::error(E_CRYPTO);

  if (rv = ciphertext.verify(pub_key, label)) return rv;

  ecc_point_t V = curve.infinity();
  for (int i = 0; i < n; i++) {
    const partial_decryption_t& partial_decryption = partial_decryptions[i];

    const int rid = partial_decryption.rid;
    if (rid < 1 || rid > n) return coinbase::error(E_CRYPTO);
    if (rv = partial_decryption.check_partial_decryption_helper(Qi[rid - 1], ciphertext, curve)) return rv;

    V += partial_decryption.Xi;
  }

  if (rv = ciphertext.decrypt(V, plain, label)) return rv;
  return SUCCESS;
```

cbmpc_error_t cbmpc_tdh2_partial_decrypt(cmem_t private_share, cmem_t ciphertext, cmem_t label,
                                         cmem_t* out_partial_decryption) {
  try {
    if (!out_partial_decryption) return E_BADARG;
    *out_partial_decryption = cmem_t{nullptr, 0};
    const auto vps = validate_cmem(private_share);
    if (vps) return vps;
    const auto vct = validate_cmem(ciphertext);
    if (vct) return vct;
    const auto vlb = validate_cmem(label);
    if (vlb) return vlb;

    coinbase::buf_t partial;
    const coinbase::error_t rv = coinbase::api::tdh2::partial_decrypt(view_cmem(private_share), view_cmem(ciphertext),
                                                                      view_cmem(label), partial);
    if (rv) return rv;
    return alloc_cmem_from_buf(partial, out_partial_decryption);
  } catch (const std::bad_alloc&) {
    if (out_partial_decryption) *out_partial_decryption = cmem_t{nullptr, 0};
    return E_INSUFFICIENT;
  } catch (...) {
    if (out_partial_decryption) *out_partial_decryption = cmem_t{nullptr, 0};
    return E_GENERAL;
  }
}
