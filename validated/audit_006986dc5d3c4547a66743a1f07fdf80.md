### Title
Token-2022 Transfer-Fee Extension Silently Undercollateralizes Vault in `init_transfer` — (`solana/programs/bridge_token_factory/src/instructions/user/init_transfer.rs`)

---

### Summary

`log_metadata` is fully permissionless and creates the vault PDA for any Token-2022 mint. When a mint has the `TransferFeeConfig` extension enabled, `transfer_checked` in `init_transfer` silently credits the vault with `amount − fee` tokens while the Wormhole message encodes the full `payload.amount`. NEAR credits the full amount to the recipient, permanently breaking the 1:1 backing guarantee for that token.

---

### Finding Description

**Step 1 — Permissionless vault creation via `log_metadata`**

`log_metadata` has no admin or privileged-caller check. The only constraint on the mint is that its `mint_authority` must not equal the bridge authority PDA (i.e., it must be a native, non-bridged token). Any user can supply a Token-2022 mint with `TransferFeeConfig` enabled and the instruction will succeed, creating the vault PDA via `init_if_needed`: [1](#0-0) 

The program entry point confirms no access control: [2](#0-1) 

**Step 2 — `init_transfer` calls `transfer_checked` with the full `payload.amount`**

Once the vault exists, `init_transfer` takes the native-token path and calls `transfer_checked` with `payload.amount` as the transfer amount: [3](#0-2) 

Under Token-2022's `TransferFeeConfig` extension, `transfer_checked(amount=N)` behaves as follows:
- Source account (`from`) is debited the full `N`.
- Destination account (`vault`) receives `N − fee` as spendable balance.
- `fee` tokens are withheld inside the vault account as `withheld_amount` (inaccessible to the bridge program, but harvestable by the fee authority).

**Step 3 — Wormhole message encodes the full `payload.amount`**

Immediately after the transfer, the Wormhole message is posted with the original `payload.amount`, not the actual vault credit: [4](#0-3) 

**Step 4 — NEAR credits the full amount**

The NEAR `fin_transfer_callback` reads `init_transfer.amount` directly from the VAA and credits that amount to the recipient: [5](#0-4) 

There is no post-transfer balance check on the Solana side and no mechanism for NEAR to learn the actual vault credit.

**Step 5 — Withheld tokens can be drained by the fee authority**

The attacker, as the `withdraw_withheld_authority` of the mint, can call Token-2022's permissionless `harvest_withheld_tokens_to_mint` followed by `withdraw_withheld_tokens_from_mint` to extract the withheld tokens from the vault, further reducing the vault's spendable balance below `amount − fee`.

**Contrast with the acknowledged transfer-hook issue**

The Solana `SECURITY.md` explicitly acknowledges transfer hooks as "denial, not fund loss" because missing extra-account-metas cause a runtime revert: [6](#0-5) 

Transfer fees are a different extension: `transfer_checked` succeeds silently, making this a fund-loss path, not a denial path. The EVM `SECURITY.md` explicitly documents fee-on-transfer tokens as unsupported for EVM: [7](#0-6) 

No equivalent disclaimer exists in the Solana `SECURITY.md`.

---

### Impact Explanation

Every `init_transfer` call on a transfer-fee Token-2022 mint deposits `amount − fee` tokens into the vault while NEAR credits `amount`. The vault is permanently undercollateralized by `fee` tokens per call. When any user later redeems on the NEAR→Solana path, `finalize_transfer` calls `transfer_checked` from the vault for the full credited amount: [8](#0-7) 

The vault has insufficient spendable balance, so the last redeemer's `finalize_transfer` reverts, permanently locking their NEAR-side tokens. The attacker additionally recovers the withheld tokens via the fee-authority harvest path, extracting value that was never theirs to take.

---

### Likelihood Explanation

The attack is fully permissionless:
1. Create a Token-2022 mint with `TransferFeeConfig` (standard Solana tooling, no special access).
2. Call `log_metadata` — no admin check, any payer can call it.
3. Call `init_transfer` — standard user-facing instruction.

No privileged role, leaked key, or external dependency compromise is required.

---

### Recommendation

In `init_transfer`, after `transfer_checked` completes, read the vault's actual post-transfer spendable balance and use that value (not `payload.amount`) when constructing the Wormhole message. Alternatively, reject mints that have the `TransferFeeConfig` extension enabled by inspecting the mint's extension list before proceeding, consistent with the existing transfer-hook denial documented in `SECURITY.md`. The Solana `SECURITY.md` should also be updated to explicitly document transfer-fee mints as unsupported (mirroring the EVM `SECURITY.md`).

---

### Proof of Concept

```
1. Create Token-2022 mint M with TransferFeeConfig: fee_rate=100 (1%), max_fee=u64::MAX,
   withdraw_withheld_authority = attacker.

2. Call log_metadata(mint=M, payer=attacker)
   → vault PDA [VAULT_SEED, M] created with zero balance.

3. Attacker mints 1000 tokens of M to their `from` account.

4. Call init_transfer(payload={amount:1000, fee:0, native_fee:0, recipient:"attacker.near", ...})
   → transfer_checked(1000, decimals) executes:
      from.amount   -= 1000   (attacker pays 1000)
      vault.amount  += 990    (vault receives 990 spendable)
      vault.withheld += 10    (10 withheld by token program)
   → Wormhole VAA posted with amount=1000.

5. NEAR processes VAA → credits 1000 tokens to attacker.near.

6. Attacker calls harvest_withheld_tokens_to_mint(sources=[vault]) → 10 tokens moved to mint.
   Attacker calls withdraw_withheld_tokens_from_mint(destination=attacker_ata) → 10 tokens recovered.

7. Assert: vault.amount == 990, NEAR balance == 1000.
   Discrepancy = 10 tokens created on NEAR with no Solana backing.

8. Any subsequent finalize_transfer for 1000 tokens will attempt
   transfer_checked(vault→recipient, 1000) but vault only holds 990 → revert.
   Last redeemer's funds are permanently locked.
```
