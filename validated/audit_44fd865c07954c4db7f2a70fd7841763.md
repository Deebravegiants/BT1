### Title
Missing Foreign-Chain Support Check in `respond_verify_foreign_tx` Allows Fulfillment of Requests for Removed Chains - (File: crates/contract/src/lib.rs)

---

### Summary

`verify_foreign_transaction` enforces that the requested foreign chain is in the supported set before accepting a request. `respond_verify_foreign_tx` performs no equivalent check when fulfilling the request. A foreign chain can be removed from the supported set after a request is submitted, yet the pending request can still be fulfilled — delivering a verified transaction response for a chain the network no longer trusts.

---

### Finding Description

In `verify_foreign_transaction`, the contract explicitly gates on the supported-chain set before enqueuing the request:

```rust
let requested_chain = request.request.chain();
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) {
    env::panic_str(
        &InvalidParameters::ForeignChainNotSupported {
            requested: requested_chain,
        }
        .to_string(),
    );
}
``` [1](#0-0) 

In `respond_verify_foreign_tx`, the only guards are: caller is an attested participant, protocol is running-or-resharing, `accept_requests` flag, and cryptographic signature validity. There is **no check** that the foreign chain in the request is still in the supported set:

```rust
pub fn respond_verify_foreign_tx(
    &mut self,
    request: VerifyForeignTransactionRequest,
    response: VerifyForeignTransactionResponse,
) -> Result<(), Error> {
    let signer = Self::assert_caller_is_signer();
    self.assert_caller_is_attested_participant_and_protocol_active();
    if !self.protocol_state.is_running_or_resharing() { ... }
    if !self.accept_requests { ... }
    // ← no supported-chain check here
    ...
    pending_requests::resolve_yields_for(...)
}
``` [2](#0-1) 

The supported set is computed dynamically by `get_supported_foreign_chains()`, which returns the strict intersection of chains registered by **all** active participants: [3](#0-2) 

This set can shrink at any time: any single participant calling `register_foreign_chain_support` without a chain removes it from the supported set, and a resharing that changes the participant set can also drop chains. [4](#0-3) 

---

### Impact Explanation

The `verify_foreign_transaction` / `respond_verify_foreign_tx` flow is the on-chain anchor for bridge services: a successful response is the authoritative attestation that a foreign-chain transaction occurred and was independently verified by the MPC network. Bridge contracts downstream consume this response to release funds or authorize cross-chain state transitions.

If a chain is removed from the supported set — for example because its RPC providers are compromised, the chain has a security incident, or a resharing introduces a new participant that does not cover the chain — the contract's intent is that no further verifications for that chain should be accepted. The check at submission time enforces this for new requests. The missing check at response time means **in-flight requests bypass the removal entirely**: a node can still call `respond_verify_foreign_tx` and the contract will resolve the yield and return the response to the waiting bridge caller, even though the chain is no longer trusted.

This breaks the request-lifecycle safety invariant — that only currently-supported chains can produce verified responses — and can cause invalid bridge execution or double-spend conditions if the chain was removed precisely because its verification data became unreliable.

---

### Likelihood Explanation

The window of exposure is the time between request submission and response fulfillment, which spans multiple blocks. The trigger — a chain leaving the supported set — requires only a single participant to update its registration (no threshold needed), or occurs automatically during any resharing that changes the participant set. Both events are routine operational occurrences. An adversary who controls one participant account can deliberately deregister a chain immediately after submitting a `verify_foreign_transaction` request, ensuring the response is delivered for a chain the network has just declared unsupported.

---

### Recommendation

Add the same supported-chain guard to `respond_verify_foreign_tx` that already exists in `verify_foreign_transaction`:

```rust
let requested_chain = request.request.chain();
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) {
    return Err(InvalidParameters::ForeignChainNotSupported {
        requested: requested_chain,
    }.into());
}
```

This mirrors the pattern used at request-submission time and closes the lifecycle gap.

---

### Proof of Concept

1. All `n` participants call `register_foreign_chain_support` including chain X (e.g., Bitcoin). `get_supported_foreign_chains()` returns `{Bitcoin, ...}`.
2. A user (or bridge contract) calls `verify_foreign_transaction` for a Bitcoin transaction. The chain-support check passes; a pending yield is created.
3. One participant calls `register_foreign_chain_support` with a set that **omits** Bitcoin. `get_supported_foreign_chains()` now returns `{}` (strict intersection drops Bitcoin).
4. A node calls `respond_verify_foreign_tx` with a valid signature for the pending request.
5. The contract executes lines 705–753 — attested-participant check passes, `accept_requests` is true, signature verifies — and calls `pending_requests::resolve_yields_for`, resolving the yield and returning the response to the bridge caller.
6. The bridge caller receives a "verified Bitcoin transaction" response even though Bitcoin is no longer in the supported set.

The check present at step 2 (`verify_foreign_transaction` lines 533–542) is entirely absent at step 5 (`respond_verify_foreign_tx` lines 691–754), the exact structural analog of the original report's disabled-collateral / mint-debt gap.

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

    pub fn get_supported_foreign_chains(&self) -> dtos::SupportedForeignChains {
        let active_participant_account_ids = self
            .protocol_state
            .active_participants()
            .participants()
            .iter()
            .map(|(account_id, _, _)| account_id.clone())
            .collect::<BTreeSet<_>>();

        let mut foreign_chain_to_node_mapping: BTreeMap<
            &dtos::ForeignChain,
            BTreeSet<dtos::AccountId>,
        > = BTreeMap::new();

        for (account_id, chains) in self
            .node_foreign_chain_support
            .foreign_chain_support_by_node
            .iter()
        {
            for chain in chains.iter() {
                foreign_chain_to_node_mapping
                    .entry(chain)
                    .or_default()
                    .insert(account_id.clone());
            }
        }

        foreign_chain_to_node_mapping
            .into_iter()
            .filter_map(|(foreign_chain, nodes_supporting_chain)| {
                let all_active_nodes_supports_chain =
                    nodes_supporting_chain.is_superset(&active_participant_account_ids);

                if all_active_nodes_supports_chain {
                    Some(foreign_chain)
                } else {
                    None
                }
            })
            .cloned()
            .collect::<BTreeSet<dtos::ForeignChain>>()
            .into()
