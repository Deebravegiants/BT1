### Title
DRep `drepExpiry` Unconditionally Overwritten Without Max-Check Allows Premature Expiry After `ppDRepActivity` Reduction — (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Certs.hs`)

---

### Summary

In `updateVotingDRepExpiries` and `ConwayUpdateDRep`, the `drepExpiry` field of a DRep is unconditionally overwritten with a freshly computed value without checking whether the new value exceeds the existing one. If `ppDRepActivity` is reduced via a governance `ParameterChange` action, any DRep that subsequently votes or submits an update certificate will have their expiry epoch set to a value *earlier* than their current stored expiry. This is the direct Cardano analog of the WithdrawProxy `finalAuctionEnd` overwrite bug: a monotonically-increasing deadline is clobbered by a smaller value when a governing parameter is reduced.

---

### Finding Description

`computeDRepExpiry` produces `(currentEpoch + ppDRepActivity) - numDormantEpochs`. [1](#0-0) 

This value is then assigned unconditionally in two places:

**1. `updateVotingDRepExpiries` — triggered by any DRep vote in a transaction:**

```haskell
Map.adjust
  (drepExpiryL .~ computeDRepExpiry drepActivity currentEpoch numDormantEpochs)
  cred
  dreps
``` [2](#0-1) 

**2. `ConwayUpdateDRep` — triggered by a DRep update certificate:**

```haskell
drepState
  & drepExpiryL
    .~ computeDRepExpiry
      ppDRepActivity
      cgceCurrentEpoch
      (certState ^. certVStateL . vsNumDormantEpochsL)
``` [3](#0-2) 

Neither site guards against the new value being smaller than the existing `drepExpiry`. The `ppDRepActivity` parameter is a fully updatable protocol parameter: [4](#0-3) 

Contrast this with `updateDormantDRepExpiry`, which *does* guard against moving the expiry backward relative to the current epoch:

```haskell
let actualExpiry = binOpEpochNo (+) numDormantEpochs currentExpiry
 in if actualExpiry < currentEpoch
      then currentExpiry
      else actualExpiry
``` [5](#0-4) 

That guard only prevents the expiry from being set to a past epoch; it does not prevent it from being set to a future epoch that is still *earlier* than the DRep's current stored expiry.

---

### Impact Explanation

When `ppDRepActivity` is reduced (e.g., from 100 epochs to 1 epoch) and a DRep subsequently votes or submits an update certificate, their stored `drepExpiry` is overwritten with a much earlier epoch. The `vsActualDRepExpiry` function adds `vsNumDormantEpochs` to the stored expiry to determine whether a DRep is active: [6](#0-5) 

A prematurely expired DRep:
- Has their votes excluded from governance ratification.
- Has their delegated stake excluded from the active voting stake denominator.

Reducing the active voting stake denominator lowers the threshold of absolute stake needed to ratify governance actions. If enough DReps are prematurely expired, a `ParameterChange`, `NewConstitution`, `UpdateCommittee`, `HardForkInitiation`, or `TreasuryWithdrawals` action could be ratified with fewer votes than the protocol parameters require, constituting an unauthorized governance action. This maps to **Medium** (attacker-controlled votes/certificates modify governance state outside design parameters) with a credible path to **Critical** (unauthorized governance action enacted).

---

### Likelihood Explanation

`ppDRepActivity` is a standard updatable Conway protocol parameter. A legitimate governance majority can reduce it at any time. Once reduced, the vulnerability is triggered automatically by any DRep that votes on a subsequent proposal or submits a `ConwayUpdateDRep` certificate — both routine, expected actions. No special access beyond normal DRep participation is required to trigger the incorrect state transition after the parameter change takes effect.

---

### Recommendation

Replace the unconditional assignment in both `updateVotingDRepExpiries` and `ConwayUpdateDRep` with a `max`-guarded update, analogous to the Astaria fix:

```haskell
-- In updateVotingDRepExpiries:
Map.adjust
  (drepExpiryL %~ \cur -> max cur (computeDRepExpiry drepActivity currentEpoch numDormantEpochs))
  cred
  dreps

-- In ConwayUpdateDRep:
drepState
  & drepExpiryL %~ \cur ->
      max cur (computeDRepExpiry ppDRepActivity cgceCurrentEpoch
                 (certState ^. certVStateL . vsNumDormantEpochsL))
```

This ensures that voting or updating a DRep can only extend (or maintain) the expiry, never shorten it, regardless of how `ppDRepActivity` has changed.

---

### Proof of Concept

1. At epoch 0, DRep D registers with `ppDRepActivity = 100`. `drepExpiry` is set to epoch 100.
2. A `ParameterChange` governance action reduces `ppDRepActivity` to `1`. It is ratified and enacted at epoch 5.
3. At epoch 5, D votes on a new proposal. `updateVotingDRepExpiries` fires and sets:
   `drepExpiry = computeDRepExpiry 1 5 0 = (5 + 1) - 0 = 6`
4. D's `drepExpiry` is now epoch 6, down from epoch 100 — a 94-epoch reduction caused by a single vote.
5. After epoch 6, D is considered expired. D's delegated stake is excluded from the active voting stake denominator in all subsequent ratification checks, lowering the effective threshold for all governance actions. [7](#0-6) [8](#0-7)
