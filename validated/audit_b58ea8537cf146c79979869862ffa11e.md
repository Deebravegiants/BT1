### Title
Unguarded `assert sp.sp_source_data` in `new_proof_of_space` Drops Winning Proofs When Full Node Omits `sp_source_data` — (`chia/farmer/farmer_api.py`)

### Summary

When a full node sends a `NewSignagePoint` with `sp_source_data=None` (which is protocol-valid), and the farmer's harvester subsequently sends a `NewProofOfSpace` with `include_source_signature_data=True` or `farmer_reward_address_override` set, the bare `assert sp.sp_source_data` at line 136 raises an unhandled `AssertionError`. This crashes the `new_proof_of_space` coroutine and silently drops the winning proof of space.

### Finding Description

`sp_source_data` is declared optional in the protocol: [1](#0-0) 

The `new_signage_point` handler stores the SP unconditionally with no validation of `sp_source_data`: [2](#0-1) 

Later, in `new_proof_of_space`, when the proof qualifies for a block and the harvester has set either flag, the code executes a bare `assert`: [3](#0-2) 

If `sp.sp_source_data` is `None`, Python's `assert` raises `AssertionError`. There is no `try/except` wrapping this path inside the function. The exception propagates out of the `for sp in sps:` loop, terminating the coroutine before `RequestSignatures` is sent to the harvester or the proof is recorded in `self.farmer.proofs_of_space`.

### Impact Explanation

A malicious (or simply non-upgraded) full node can send every `NewSignagePoint` with `sp_source_data=None`. Any farmer whose harvester uses `include_source_signature_data=True` or `farmer_reward_address_override` will silently drop every winning proof of space for those signage points. The farmer loses block rewards with no error surfaced beyond an unhandled exception in the coroutine. This is a **High** impact: long-lived inability for honest farmers to process valid winning proofs under normal network assumptions.

### Likelihood Explanation

- `sp_source_data` is optional in the wire protocol; any peer that omits it (malicious or simply older/non-conforming) triggers the path.
- `include_source_signature_data` is a harvester-side security feature; operators enabling it are specifically targeted.
- `farmer_reward_address_override` is used by fee-based harvester services, a real deployment pattern.
- No authentication or signature is required to send a `NewSignagePoint`; any connected full-node peer can do it.

### Recommendation

Replace the bare `assert` with a proper guard that either skips source-data attachment (falling back to `sp_src_data = None`) or logs and returns early with a clear error:

```python
if (
    new_proof_of_space.include_source_signature_data
    or new_proof_of_space.farmer_reward_address_override is not None
):
    if sp.sp_source_data is None:
        self.farmer.log.error(
            "Harvester requested source signature data but SP has none; dropping proof"
        )
        return None
    # ... build sp_src_data as before
```
