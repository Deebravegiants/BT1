Audit Report

## Title
Reward Tokens Permanently Frozen When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool` uses a Synthetix-style reward accumulation model where `rewardPerToken()` stops accumulating when `totalKernelStaked == 0`. Any staker can call `initiateWithdrawal` to immediately reduce `totalKernelStaked` to zero during an active reward period, causing all reward tokens corresponding to the zero-staking window to be permanently locked in the contract with no on-chain recovery path.

## Finding Description
The `rewardPerToken()` function returns the frozen `rewardPerTokenStored` whenever `totalKernelStaked == 0`:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L408-414
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;  // accumulation halts
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
```

The `updateReward` modifier also snapshots `updatedAt = lastTimeRewardApplicable()` on every call. Once `totalKernelStaked` is zero, `updatedAt` is frozen at that snapshot, and `rewardRate` tokens per second silently accumulate in the contract balance with no staker to attribute them to.

`initiateWithdrawal` immediately decrements `totalKernelStaked` with no check on whether a reward period is active:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L320-326
function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
    ...
    balanceOf[msg.sender] -= _amount;
    totalKernelStaked -= _amount;   // immediate, unconditional reduction
```

The only guard against zero-staking is in `notifyRewardAmount` (L570: `if (totalKernelStaked == 0) revert NoStakedTokens()`), which only prevents starting a new reward period with no stakers. It does not prevent stakers from exiting during an active period.

There is no `sweep`, `recoverERC20`, or admin rescue function for `rewardsToken` anywhere in the contract. The contract's own NatSpec (L14-23) explicitly acknowledges this limitation but relies entirely on an off-chain deployment assumption — it is not enforced on-chain.

## Impact Explanation
Reward tokens deposited via `notifyRewardAmount` that correspond to any time interval where `totalKernelStaked == 0` are permanently frozen in `KernelDepositPool`. The stuck amount equals `rewardRate × (duration of zero-staking window)` in reward token units. There is no function to recover them. This matches **Medium — Permanent freezing of unclaimed yield**.

## Likelihood Explanation
Any unprivileged staker can trigger this by calling `initiateWithdrawal` for their full balance during an active reward period. No special role, governance access, or coordination is required. In a single-large-staker scenario (common in early protocol deployments), one call suffices. The action is a normal user workflow (withdrawal), making it reachable without any adversarial intent.

## Recommendation
Add an admin-callable rescue function that allows recovery of undistributed reward tokens after `block.timestamp >= finishAt`. Alternatively, track "lost" reward seconds when `totalKernelStaked == 0` and roll them into the next reward period via `notifyRewardAmount`. A minimal on-chain mitigation would be to revert `initiateWithdrawal` if the withdrawal would zero out `totalKernelStaked` while `block.timestamp < finishAt`.

## Proof of Concept
1. Admin calls `notifyRewardAmount(1_000e18)` with `duration = 30 days`. `rewardRate = 1_000e18 / 30 days`.
2. Alice is the only staker: `balanceOf[Alice] = 1_000e18`, `totalKernelStaked = 1_000e18`.
3. After 10 days, Alice calls `initiateWithdrawal(1_000e18)`. The `updateReward(Alice)` modifier correctly snapshots her 10-day earnings. Then `totalKernelStaked` is set to 0.
4. For the remaining 20 days, `rewardPerToken()` returns the frozen `rewardPerTokenStored`. No staker accumulates rewards.
5. `rewardRate × 20 days ≈ 666e18` reward tokens remain in the contract permanently.
6. No function exists to recover them — confirmed by full contract review.

**Foundry test plan:** Deploy `KernelDepositPool`, stake 1000e18 for Alice, call `notifyRewardAmount`, warp 10 days, call `initiateWithdrawal(1000e18)`, warp to `finishAt`, assert `rewardsToken.balanceOf(address(pool)) > 0` and that no call path can reduce it to zero.
