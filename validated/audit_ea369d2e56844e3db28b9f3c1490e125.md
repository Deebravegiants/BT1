### Title
`KernelMerkleDistributor.claim` / `claimAndStake` enforce strict `account == msg.sender`, permanently freezing unclaimed KERNEL yield for smart-contract accounts - (File: contracts/KERNEL/KernelMerkleDistributor.sol)

---

### Summary

`KernelMerkleDistributor` exposes `claim` and `claimAndStake` with an explicit `account` parameter, implying any caller can trigger a claim on behalf of a merkle-leaf address. However, the shared internal helper `_processClaim` unconditionally reverts when `account != msg.sender`. Any address in the merkle tree that is a smart contract (multisig, DAO treasury, protocol vault, etc.) and cannot itself issue the call — or any user who relies on a relayer / meta-transaction forwarder — is permanently unable to collect their KERNEL allocation.

---

### Finding Description

`_processClaim` is the single code path executed by both `claim` and `claimAndStake`:

```solidity
// contracts/KERNEL/KernelMerkleDistributor.sol  lines 311-313
if (account != msg.sender) {
    revert Unauthorized();
}
```

The public function signatures accept a caller-supplied `account`:

```solidity
function claim(
    uint256 index,
    address account,          // ← suggests "claim for this address"
    uint256 cumulativeAmount,
    bytes32[] calldata merkleProof
) external override nonReentrant whenNotPaused { … }

function claimAndStake(
    uint256 index,
    address account,          // ← same
    uint256 cumulativeAmount,
    bytes32[] calldata merkleProof
) external nonReentrant whenNotPaused { … }
```

The `account` parameter is part of the merkle leaf (`keccak256(abi.encodePacked(index, account, cumulativeAmount))`), so it is fixed at snapshot time by the protocol. If that address is a smart contract that cannot itself call `claim` — a Gnosis Safe whose signers are unavailable, a DAO governor, a protocol-owned vault, or any contract without a dedicated `claim` wrapper — the KERNEL tokens committed to that leaf are permanently unclaimable. No approved operator, relayer, or third party can substitute.

The inconsistency mirrors the Tigris M-04 pattern exactly: the function signature advertises the ability to act on behalf of an `account`, but the implementation silently collapses that to a strict self-only check.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

KERNEL tokens allocated to a merkle leaf whose `account` is a smart contract incapable of directly calling `claim` are locked in the distributor forever. The tokens are not lost from the contract, but they are irrecoverable by the rightful beneficiary. This matches the "Permanent freezing of unclaimed yield" impact category.

---

### Likelihood Explanation

**Medium.** Merkle distributions in DeFi routinely include smart-contract addresses: protocol treasuries, multisig safes, DAO governors, and liquidity pools. The Kelp/KERNEL ecosystem itself uses multisig governance. Any such address appearing in a distribution snapshot triggers the freeze. The condition is reachable by any unprivileged external caller who simply passes `account != msg.sender`.

---

### Recommendation

Remove the `account != msg.sender` guard from `_processClaim`. The merkle proof already cryptographically binds the `(index, account, cumulativeAmount)` tuple; no additional caller-identity check is needed for security. Tokens are always transferred to `account`, so there is no risk of a third party redirecting funds:

```solidity
// Remove or replace:
// if (account != msg.sender) {
//     revert Unauthorized();
// }
```

If self-only claiming is intentional for some reason, the `account` parameter should be removed from the public ABI and replaced with `msg.sender` throughout, making the restriction explicit and eliminating the misleading signature.

---

### Proof of Concept

1. Protocol snapshots a merkle tree that includes `account = 0xSafeMultisig` with `cumulativeAmount = 1000e18`.
2. The multisig signers attempt to claim by having a relayer call:
   ```solidity
   kernelMerkleDistributor.claim(index, 0xSafeMultisig, 1000e18, proof);
   ```
3. `_processClaim` executes `if (account != msg.sender)` → `0xSafeMultisig != relayer` → `revert Unauthorized()`.
4. The multisig itself calls `claim(index, 0xSafeMultisig, 1000e18, proof)` — now `msg.sender == 0xSafeMultisig` — this succeeds only if the Safe has a module or transaction that encodes this exact call. If the Safe's execution path is unavailable (e.g., threshold signers lost keys, Safe is deprecated), the tokens are permanently frozen.
5. No alternative entry point exists; `claimAndStake` shares the same `_processClaim` gate. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
