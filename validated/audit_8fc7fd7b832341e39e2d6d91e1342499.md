### Title
Single-Participant `vote_abort` Enables Permanent Keygen/Resharing Blocking — (File: `crates/contract/src/state/key_event.rs`)

### Summary
A single Byzantine participant, strictly below the signing threshold, can permanently prevent key generation or key resharing from completing by repeatedly calling `vote_abort_key_event_instance` on every new attempt. Because `vote_abort` immediately destroys the active `KeyEventInstance` with a single participant's vote — no threshold required — the attacker can abort every attempt the leader starts, indefinitely locking the MPC network in a non-functional state.

### Finding Description

The `KeyEvent::vote_abort` function in `crates/contract/src/state/key_event.rs` unconditionally sets `self.instance = None` upon receiving a single valid participant vote:

```rust
pub fn vote_abort(&mut self, key_event_id: KeyEventId) -> Result<(), Error> {
    let candidate = self.verify_vote(&key_event_id)?;
    if self.instance.as_ref().unwrap().completed.contains(&candidate) {
        return Err(VoteError::VoteAlreadySubmitted.into());
    }
    self.instance = None;   // ← single vote destroys the entire attempt
    Ok(())
}
``` [1](#0-0) 

The only guard is that the aborting participant must not already appear in the `completed` set (i.e., must not have previously voted success for this same attempt). Since each new attempt starts with a fresh, empty `completed` set, a participant who aborted attempt N is free to abort attempt N+1, N+2, … without restriction. [2](#0-1) 

The leader — always the participant with the lowest participant ID — is the only one who can call `start` to open a new attempt: [3](#0-2) 

The attack loop is therefore:

1. Leader calls `start_keygen_instance` / `start_reshare_instance` → attempt N is created.
2. Byzantine participant calls `vote_abort_key_event_instance` with attempt N's `KeyEventId` → `instance = None`.
3. Leader calls `start` again → attempt N+1 is created.
4. Byzantine participant aborts N+1.
5. Repeat indefinitely at negligible cost.

This is confirmed by the node-side code, which sends `VoteAbortKeyEventInstance` transactions whenever a node's computation fails: [4](#0-3) 

The same `vote_abort` path is exercised for both the `Initializing` (keygen) and `Resharing` states: [5](#0-4) [6](#0-5) 

### Impact Explanation

**Critical.** If the contract is in the `Initializing` state (initial key generation), a permanently blocked keygen means the MPC network never produces a distributed key. No threshold signatures can ever be issued, permanently freezing all funds and cross-chain flows that depend on the MPC signing service. If the contract is in the `Resharing` state, a permanently blocked resharing freezes the participant set and prevents key rotation, trapping the network in a state where it cannot safely evolve. Both outcomes satisfy the allowed critical impact: *permanent freezing of funds controlled by the MPC network or chain-signature contract*.

### Likelihood Explanation

**Medium–High.** The attacker must be an attested participant in the proposed participant set — a non-trivial prerequisite. However, only **one** participant out of the full set is required, which is strictly below any meaningful signing threshold. A newly added participant (e.g., during a resharing to expand the set) who turns malicious after being voted in can execute this attack. The on-chain cost per abort is a single NEAR transaction, making sustained blocking economically trivial.

### Recommendation

Replace the single-vote-abort model with a **threshold-of-votes-to-abort** model, mirroring the threshold required for `vote_cancel_resharing`. Specifically:

- Accumulate abort votes in a `BTreeSet` on `KeyEventInstance` (analogous to `cancellation_requests` on `ResharingContractState`).
- Only set `instance = None` when the number of abort votes reaches the reconstruction threshold.
- Alternatively, remove `vote_abort` entirely and rely solely on the block-height timeout (`expires_on`) to clean up failed attempts, since `cleanup_if_timed_out` already handles this path. [7](#0-6) 

### Proof of Concept

```
Setup: 5-of-5 keygen in Initializing state. Participant P1 is Byzantine.

Block B:   Leader (P0) calls start_keygen_instance(KeyEventId{epoch=1, domain=0, attempt=0})
           → instance created, expires_on = B + 1 + timeout_blocks

Block B+1: P1 calls vote_abort_key_event_instance(KeyEventId{epoch=1, domain=0, attempt=0})
           → verify_vote passes (P1 is a participant, attempt ID matches)
           → completed.contains(P1) == false (P1 never voted success)
           → instance = None

Block B+2: Leader calls start_keygen_instance(KeyEventId{epoch=1, domain=0, attempt=1})
           → new instance created

Block B+3: P1 calls vote_abort_key_event_instance(KeyEventId{epoch=1, domain=0, attempt=1})
           → instance = None

... (repeats indefinitely)

Result: The contract remains in Initializing state forever. No key is ever generated.
        No threshold signatures can be issued. All MPC-dependent funds are permanently frozen.
```

The `next_attempt_id` counter increments monotonically with each `start` call: [8](#0-7) 

There is no cap on the number of attempts, no cooldown between aborts, and no mechanism to expel a misbehaving participant mid-keygen, confirming the attack is unbounded.

#[near(serializers=[borsh, json])]
#[derive(Debug, Clone, Default, PartialEq)]
pub struct KeyEventInstance {
    attempt_id: AttemptId,
    /// The block in which KeyEvent::start() was called.
    started_in: BlockHeight,
    /// The block that this attempt expires on. To clarify off-by-one behavior: if the contract were
