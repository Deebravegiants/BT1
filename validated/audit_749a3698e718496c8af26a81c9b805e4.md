### Title
`respond_verify_foreign_tx` Accepts Replayed Signatures from Unrelated Transactions, Enabling Forged Foreign-Chain Verification - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash` using the root public key, but **never checks that `response.payload_hash` is derived from the stored `request`**. A single Byzantine attested participant can replay a valid `VerifyForeignTransactionResponse` produced for a different transaction to resolve any pending foreign-tx verification request, causing the bridge caller to receive a cryptographically valid but semantically incorrect attestation.

---

### Finding Description

In `respond_verify_foreign_tx` the contract performs two independent checks:

1. The `request` key exists in `pending_verify_foreign_tx_requests`.
2. `response.signature` is a valid ECDSA signature over `response.payload_hash` under the root public key. [1](#0-0) 

What is **never checked** is whether `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request, extracted_values }))` for the stored `request`. The `payload_hash` is a free field in the response struct: [2](#0-1) 

The intended derivation is:

```
payload_hash = SHA-256(borsh(ForeignTxSignPayload { request, extracted_values }))
```

as computed by the node in `build_signature_request`: [3](#0-2) 

The contract stores only the `request` key, not the `extracted_values`, so it cannot independently recompute the expected hash. The SDK helper `ForeignChainSignatureVerifier::verify_signature` does perform this check: [4](#0-3) 

But that check lives in the **caller's** contract, not in the MPC contract itself. The MPC contract resolves the yield with whatever `response` the attacker supplies, and the caller receives it as the authoritative result.

---

### Impact Explanation

A Byzantine attested participant observes a legitimately completed `respond_verify_foreign_tx` call for transaction **T2** on the NEAR blockchain (all call arguments are public). They extract the `VerifyForeignTransactionResponse` for T2 (containing `payload_hash_T2` and `sig_T2`). They then call `respond_verify_foreign_tx` with:

- `request` = the pending request for **T1** (a different, possibly unconfirmed transaction)
- `response` = the response originally produced for T2

The contract accepts this because:
- T1 is in `pending_verify_foreign_tx_requests` ✓
- `sig_T2` is a valid signature over `payload_hash_T2` under the root public key ✓

`resolve_yields_for` drains T1's yield queue and delivers T2's response to every caller waiting on T1: [5](#0-4) 

The bridge contract waiting on T1 receives a `VerifyForeignTransactionResponse` with T2's `payload_hash` and T2's signature. Any bridge that does not independently recompute the expected hash (which the SDK helper does, but many callers may not) will treat T1 as verified with T2's extracted values. This enables:

- Attestation of an unconfirmed or non-existent foreign transaction.
- Double-spend: a single confirmed transaction T2 can be replayed to satisfy multiple pending verification requests for different transactions.
- Permanent corruption of the bridge state for all callers queued under T1 (fan-out of up to 128 yields per request key).

This matches the allowed impact: **High — cross-chain replay / forged foreign-chain verification causing invalid bridge execution or double-spend conditions**.

---

### Likelihood Explanation

The attacker needs only to be a single attested participant (no threshold collusion required). All `respond_verify_foreign_tx` call arguments are on-chain and publicly visible. The attacker does not need to forge any cryptographic material — they replay an existing valid signature. Any one of the MPC nodes that has been attested can execute this attack immediately after observing any legitimate response on-chain.

---

### Recommendation

The contract must bind `response.payload_hash` to the stored `request`. Since the contract does not store `extracted_values`, the simplest fix is to include the `extracted_values` in the response and have the contract recompute and verify the hash:

```rust
// In respond_verify_foreign_tx, after retrieving the public key:
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(), // new field
});
let expected_hash = expected_payload.compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
// then verify signature over expected_hash (not response.payload_hash)
```

Alternatively, restructure the signing payload so the contract can verify the `request`-portion of the hash without knowing `extracted_values` (e.g., commit to the request hash separately).

---

### Proof of Concept

1. User A submits `verify_foreign_transaction` for Bitcoin tx `0xAAAA` (pending, unconfirmed). Contract stores `request_A` in `pending_verify_foreign_tx_requests`.
2. MPC nodes legitimately process Bitcoin tx `0xBBBB` (confirmed). One node calls `respond_verify_foreign_tx(request_B, response_B)`. Contract accepts; `response_B = {payload_hash_B, sig_B}` is now public on-chain.
3. Byzantine attested participant calls `respond_verify_foreign_tx(request_A, response_B)`.
4. Contract checks: `request_A` is pending ✓; `sig_B` is valid over `payload_hash_B` under root key ✓. No check that `payload_hash_B` is derived from `request_A`.
5. Contract resolves all yields queued under `request_A` with `response_B`.
6. User A's bridge contract receives `{payload_hash_B, sig_B}` as the attestation for `0xAAAA`. If it does not call `ForeignChainSignatureVerifier::verify_signature` with the expected values for `0xAAAA`, it accepts the forged attestation and executes the bridge action for an unconfirmed transaction. impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
