### Title
Witness-Free Stake Credential Registration Enables Frontrunning Denial of Registration - (File: `eras/shelley/impl/src/Cardano/Ledger/Shelley/TxCert.hs`, `eras/conway/impl/src/Cardano/Ledger/Conway/TxCert.hs`)

---

### Summary

Stake key registration certificates (`ShelleyRegCert` / `ConwayRegCert cred SNothing`) require no witness from the credential being registered. Combined with the DELEG rule's hard rejection of duplicate registrations, any unprivileged attacker can frontrun a victim's pending registration transaction by submitting their own registration for the same credential, causing the victim's transaction to fail with `StakeKeyAlreadyRegisteredDELEG` / `StakeKeyRegisteredDELEG`.

---

### Finding Description

**No witness required for stake registration.** Both the Shelley and Conway implementations explicitly return `Nothing` for the witness requirement on registration certificates:

`eras/shelley/impl/src/Cardano/Ledger/Shelley/TxCert.hs` lines 591–592:
```haskell
-- Registration certificates do not require a witness
ShelleyRegCert _ -> Nothing
``` [1](#0-0) 

`eras/conway/impl/src/Cardano/Ledger/Conway/TxCert.hs` lines 770–771:
```haskell
ConwayRegCert _ SNothing -> Nothing
ConwayRegCert cred (SJust _) -> credKeyHashWitness cred
``` [2](#0-1) 

The design spec explicitly confirms this is intentional:
> "We do not require a witness to register a stake address (besides, of course, any witnesses needed for the transaction that is used to post the certificate)." [3](#0-2) 

**Hard rejection on duplicate registration.** The Shelley DELEG rule rejects any registration if the credential is already present in the accounts map:

```haskell
not (isAccountRegistered cred (ds ^. accountsL)) ?! StakeKeyAlreadyRegisteredDELEG cred
``` [4](#0-3) 

The Conway DELEG rule has the same check:
```haskell
checkStakeKeyNotRegistered stakeCred =
  not (isAccountRegistered stakeCred accounts)
    ?! injectFailure (StakeKeyRegisteredDELEG stakeCred)
``` [5](#0-4) 

**Attack path:**
1. Victim broadcasts a transaction containing `ShelleyRegCert cred` (or `ConwayRegCert cred SNothing`) to register their stake credential.
2. Attacker observes the pending transaction in the mempool (or a block producer below consensus threshold orders transactions adversarially within a block).
3. Attacker submits their own transaction containing the same `ShelleyRegCert cred` — no witness from `cred` is required; the attacker only needs to pay the deposit from their own UTxO.
4. Attacker's transaction is included first.
5. Victim's transaction fails with `StakeKeyAlreadyRegisteredDELEG` / `StakeKeyRegisteredDELEG`.

This is the direct analog of the `LinearVesting.vestFor` frontrunning: the "already registered" guard is the collision check, and the absence of a witness requirement is the open-access entry point.

The Conway code comment acknowledges this is a known transitional gap:
> "we preserve the old behavior of not requiring a witness for staking credential registration, but only during the transitional period of Conway era and only for staking credential registration certificates without a deposit. Future eras will require a witness for registration certificates, because the one without a deposit will be removed." [6](#0-5) 

---

### Impact Explanation

The victim's stake registration transaction fails. Until the victim submits a deregistration transaction (which does require their witness) and then re-registers, their stake credential is not active. During this window:

- The victim cannot delegate their stake to a pool or DRep.
- The victim earns no staking rewards on their ADA.
- The victim must pay additional transaction fees to deregister and re-register.

The attacker pays the deposit (currently 2 ADA on mainnet), which is returned to the victim's reward account upon deregistration — making the attack economically irrational but technically executable. The impact fits **Medium**: attacker-controlled certificates modify deposit accounting and reward accrual outside design parameters (the deposit is paid by a non-owner of the credential, and the victim's reward stream is interrupted).

---

### Likelihood Explanation

The attack requires only:
- Knowledge of the victim's stake credential (publicly visible in the mempool or in a pending transaction).
- Willingness to pay the deposit (2 ADA on mainnet).
- Ability to get a transaction included before the victim's (achievable by any mempool observer with a higher fee, or by a block producer ordering transactions within a block).

No privileged access, governance majority, or key compromise is needed. The attack is cheap to attempt and the attacker recovers nothing (deposit goes to victim), so it is primarily a griefing/DoS vector. Likelihood is **Low-to-Medium** in practice due to economic irrationality, but the code path is fully reachable.

---

### Recommendation

Require a witness from the stake credential for all registration certificates, including `ShelleyRegCert` / `ConwayRegCert cred SNothing`. The Conway codebase already acknowledges this fix is planned for a future era (removal of the no-deposit form). Accelerating this change, or adding a witness requirement to the no-deposit form in Conway, would close the frontrunning window. Alternatively, the DELEG rule could be changed to silently succeed (no-op) rather than fail when a credential is already registered, matching the pool re-registration behavior.

---

### Proof of Concept

```
1. Alice wants to register stake credential C (key hash H).
   Alice builds Tx_A = { inputs: [utxo_alice], certs: [ShelleyRegCert C] }
   Alice broadcasts Tx_A.

2. Bob (attacker) sees Tx_A in the mempool.
   Bob builds Tx_B = { inputs: [utxo_bob], certs: [ShelleyRegCert C] }
   -- No witness from C is required; Bob signs only with his own payment key.
   Bob broadcasts Tx_B with a higher fee.

3. Tx_B is included in block N before Tx_A.
   Ledger state: C is now registered, deposit paid by Bob.

4. Tx_A is applied: DELEG rule fires:
     not (isAccountRegistered C accounts) ?! StakeKeyAlreadyRegisteredDELEG C
   → Tx_A fails. Alice's registration is denied.

5. Alice must now submit Tx_C = { certs: [ShelleyUnRegCert C] } (requires Alice's witness)
   and then Tx_D = { certs: [ShelleyRegCert C] } to recover.
   Alice pays fees for two extra transactions; Bob loses the 2 ADA deposit
   (returned to Alice's reward account on deregistration).
```

getVKeyWitnessShelleyTxCert :: ShelleyTxCert era -> Maybe (KeyHash Witness)
getVKeyWitnessShelleyTxCert = \case
  ShelleyTxCertDelegCert delegCert ->
    case delegCert of
      -- Registration certificates do not require a witness
      ShelleyRegCert _ -> Nothing
      ShelleyUnRegCert cred -> credKeyHashWitness cred
      ShelleyDelegCert cred _ -> credKeyHashWitness cred
  ShelleyTxCertPool poolCert -> Just $ poolCertKeyHashWitness poolCert
  ShelleyTxCertGenesisDeleg genesisCert -> Just $ genesisKeyHashWitness genesisCert
  ShelleyTxCertMir {} -> Nothing
