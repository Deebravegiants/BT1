### Title
Byzantine MPC Participant Can Inject Mismatched `payload_hash` into `respond_verify_foreign_tx`, Delivering Forged Foreign-Chain Verification to Waiting Callers - (File: crates/contract/src/lib.rs)

### Summary
The `respond_verify_foreign_tx` function verifies that `response.signature` is cryptographically valid over `response.payload_hash`, but does **not** verify that `response.payload_hash` is the correct canonical hash for the submitted `request`. A single Byzantine MPC participant (below signing threshold) can obtain a legitimately produced threshold signature for one pending foreign-chain verification request and submit it as a response to a **different** pending request. Every caller waiting on that second request receives a `VerifyForeignTransactionResponse` whose `payload_hash` attests to the wrong foreign-chain data.

### Finding Description

The `respond_verify_foreign_tx` function performs the following signature check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

This only confirms that `response.signature` is a valid ECDSA signature over `response.payload_hash`. It does **not** confirm that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request, values }))` for the specific `request` key being resolved.

The `VerifyForeignTransactionRequest` struct used as the pending-map key contains only `request`, `domain_id`, and `payload_version` — no caller identity: [2](#0-1) 

The `args_into_verify_foreign_tx_request` conversion confirms no caller field is added: [3](#0-2) 

After the signature check passes, `resolve_yields_for` drains **all** queued yields for `request` — including yields from different callers — with the attacker-supplied `response` bytes: [4](#0-3) 

The fan-out design is intentional and documented: different callers submitting the same foreign-chain request are queued under the same key and all receive the same response: [5](#0-4) 

**Attack path:**

1. Requests `A` (Bitcoin tx X) and `B` (Bitcoin tx Y) are both pending in `pending_verify_foreign_tx_requests`.
2. The MPC network legitimately processes request `A`, producing threshold signature `sig_A` over `hash_A = SHA-256(borsh(ForeignTxSignPayload{request_A, values_A}))`.
3. Byzantine participant `P` (one attested node, below signing threshold) calls `respond_verify_foreign_tx(request_B, { payload_hash: hash_A, signature: sig_A })`.
4. The contract checks: `sig_A` is valid over `hash_A` ✓; `request_B` exists in the pending map ✓. Both pass.
5. All callers waiting on `request_B` receive `{ payload_hash: hash_A, signature: sig_A }` — a response that attests to Bitcoin tx X's data, not Bitcoin tx Y's.

The participant does not forge any signature; they reuse a legitimately produced threshold signature for a different request.

### Impact Explanation

The `verify_foreign_transaction` flow is the MPC network's mechanism for trustworthy foreign-chain attestation, specifically designed for bridge inbound flows where NEAR contracts release funds upon verified foreign-chain deposits. A receiving NEAR contract that does not independently recompute and compare `payload_hash` against its expected values (e.g., contracts that only check signature validity) will incorrectly conclude that Bitcoin tx Y was verified by the MPC network, when in fact the signature covers Bitcoin tx X. This enables invalid bridge execution or double-spend conditions.

The `near-mpc-sdk`'s `ForeignChainSignatureVerifier::verify_signature` does perform this check: [6](#0-5) 

However, the contract itself does not enforce it, leaving all callers who do not use the SDK's verifier exposed.

### Likelihood Explanation

A single Byzantine MPC participant — one that is TEE-attested and below the signing threshold — can execute this attack. The participant must have been part of a legitimate signing session for request `A` to obtain `sig_A`, which is a normal operational condition. No threshold collusion is required. The attack is viable whenever two different foreign-chain verification requests are simultaneously pending, which is the common production case for any active bridge.

### Recommendation

In `respond_verify_foreign_tx`, after verifying the signature, recompute the expected `payload_hash` from the `request` and the extracted values included in the response, and assert equality:

```rust
// Recompute expected hash from the request and the signed payload
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(), // requires adding this field
}).compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Alternatively, include the `extracted_values` in the `VerifyForeignTransactionResponse` and enforce the hash binding on-chain. This ensures that a valid signature for request `A` cannot be accepted as a response to request `B`.

### Proof of Concept

1. Alice submits `verify_foreign_transaction(Bitcoin { tx_id: X, confirmations: 2, extractors: [BlockHash] })` → pending as `request_A`.
2. Bob submits `verify_foreign_transaction(Bitcoin { tx_id: Y, confirmations: 2, extractors: [BlockHash] })` → pending as `request_B`.
3. MPC network signs `hash_A = SHA-256(borsh({ request_A, values: [BlockHash([0xAA;32])] }))`, producing `sig_A`.
4. Byzantine participant calls:
   ```
   respond_verify_foreign_tx(
     request = request_B,
     response = { payload_hash: hash_A, signature: sig_A }
   )
   ```
5. Contract accepts: `sig_A` is valid over `hash_A` ✓; `request_B` is in the pending map ✓.
6. Bob's NEAR contract receives `{ payload_hash: hash_A, signature: sig_A }`.
7. Bob's bridge contract, checking only that the signature is valid (without recomputing the expected hash for tx Y), incorrectly concludes that Bitcoin tx Y was verified and releases funds. [7](#0-6) [8](#0-7)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

```rust
/// Resume every yield queued for `request` with `response_bytes`, draining the
/// fan-out map in one pass. Returns `Err(RequestNotFound)` if the map held no entry.
///
/// Resuming a yield that has already timed out is a no-op at the SDK level.
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
