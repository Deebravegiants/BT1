### Title
VRF Key Hash Reference-Count Underflow via Oscillating Re-Registration Allows VRF Uniqueness Bypass — (`eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/Pool.hs`)

### Summary
A pool operator can submit three `RegPool` certificates within one epoch (VRF_A → VRF_B → VRF_A → VRF_B) to cause `psVRFKeyHashes` to silently drop the pool's currently active VRF key. A second pool can then register with that same VRF key, giving two simultaneously active pools an identical VRF key — the exact invariant the Conway-era uniqueness check was designed to enforce.

### Finding Description
In `poolTransition` the re-registration guard at lines 279–282 passes when `sppVrf == stakePoolState ^. spsVrfL` (new VRF equals the **current active** VRF). This is correct in isolation, but it interacts badly with `updateFutureVRFKeyHash` at lines 283–294, which uses `psFutureStakePoolParams` to decide what to remove from `psVRFKeyHashes`.

The critical flaw is that `Map.insert sppVrf (knownNonZeroBounded @1)` (line 292) always **sets** the count to 1 rather than incrementing it. When step 2 re-inserts VRF_A (which is already present with count 1), the count stays at 1 instead of rising to 2. Step 3 then deletes VRF_A with `Map.delete` (line 293), reducing the effective reference count to 0 and removing it from the map — even though the pool's active entry in `psStakePools` still carries VRF_A.

The `PoolReap` transition compounds the problem: `danglingVRFKeyHashes` (lines 138–148) identifies VRF_A as "dangling" (active ≠ future) and `Map.withoutKeys` (line 226) removes it again at epoch boundary, so the inconsistency survives the epoch roll. [5](#0-4) [3](#0-2) 

### Impact Explanation
The claimed impact — *"permanent ledger divergence requiring a hard fork"* — is **not correct**. All nodes apply the same deterministic rules and converge to the same (inconsistent) state; there is no inter-node divergence.

The actual impact is **Medium**: attacker-controlled pool-registration certificates exceed the intended validation limit introduced in protocol version 11. Two pools end up sharing the same active VRF key. Because VRF output is deterministic for a given key and slot, both pools would be elected slot leader for identical slots, allowing the operator to produce more blocks than their combined stake entitles them to and distorting the reward distribution outside design parameters. [6](#0-5) 

### Likelihood Explanation
Requires only three `RegPool` certificates from a single pool operator within one epoch — a routine, low-cost operation. No privileged access, governance majority, or external dependency is needed. The exploit is fully local-testable.

### Recommendation
Replace the fixed-count insert with a proper reference-count increment, and also check the **staged future VRF** in the guard so that re-registering back to the active VRF when a future VRF is already staged is treated as a VRF change (not a no-op):

```haskell
-- Guard: also reject if sppVrf matches the *staged* future VRF of another pool
-- updateFutureVRFKeyHash: use insertWith to increment, not overwrite
Map.insertWith (\_ old -> fromMaybe old (mapNonZero (+1) old))
  sppVrf (knownNonZeroBounded @1)
```

Additionally, `PoolReap`'s `danglingVRFKeyHashes` removal should use `removeVRFKeyHashOccurrence` (decrement) rather than `Map.withoutKeys` (hard delete) to stay consistent with the reference-counting model. [7](#0-6)
