### Title
Missing Upper-Bound Validation on `_feeBps` in `initialize` Allows 100% Fee Configuration, Draining User Deposits - (File: contracts/pools/RSETHPoolV3.sol)

### Summary
`RSETHPoolV3.initialize` accepts `_feeBps` without any upper-bound check, while the post-deployment setter `setFeeBps` enforces a hard cap of 1000 BPS (10%). If `_feeBps` is set to 10 000 (100%) at initialization, every user deposit mints 0 `wrsETH` and the entire deposited ETH/token amount is silently captured as protocol fees, permanently transferring user funds to the fee recipient.

### Finding Description
`setFeeBps` enforces `_feeBps ≤ 1000`: [1](#0-0) 

But `initialize` sets `feeBps` with no validation at all: [2](#0-1) 

The fee is applied inside `viewSwapRsETHAmountAndFee`: [3](#0-2) 

When `feeBps = 10 000`:
- `fee = amount * 10_000 / 10_000 = amount`
- `amountAfterFee = 0`
- `rsETHAmount = 0`

The entire deposited amount is added to `feeEarnedInETH` (or `feeEarnedInToken`), and the user receives 0 `wrsETH`. The `limitDailyMint` modifier passes because `dailyMintAmount + 0 ≤ dailyMintLimit`. [4](#0-3) 

The accumulated fees are then withdrawable exclusively by the `BRIDGER_ROLE` via `withdrawFees`, not by depositors: [5](#0-4) 

### Impact Explanation
Every ETH or token deposited by any user is silently captured as a fee. Users receive 0 `wrsETH` in return. Their funds are not recoverable by them — only the `BRIDGER_ROLE` can extract the accumulated fees. This constitutes direct theft of user funds at-rest.

**Impact: Critical** — direct theft of all user deposits.

### Likelihood Explanation
The pool is deployed and initialized once. A deployer who misconfigures `_feeBps = 10_000` (or any value > 1000 that the setter would reject) creates a permanently broken pool. Because `initialize` is called exactly once and there is no post-deployment path to correct `feeBps` to a value above 1000 (the setter would revert), the misconfiguration is irreversible without a proxy upgrade. The inconsistency between `initialize` and `setFeeBps` makes this a realistic misconfiguration risk.

**Likelihood: Medium** — analogous to the reference report's "users can easily misconfigure inputs."

### Recommendation
Add the same upper-bound guard in `initialize` that `setFeeBps` already enforces:

```solidity
function initialize(..., uint256 _feeBps, ...) external initializer {
    if (_feeBps > 1000) revert InvalidFeeAmount(); // add this
    ...
    feeBps = _feeBps;
    ...
}
```

### Proof of Concept
1. Deploy `RSETHPoolV3` proxy and call `initialize` with `_feeBps = 10_000`.
2. User calls `deposit{value: 1 ether}("ref")`.
3. Inside `viewSwapRsETHAmountAndFee(1 ether)`:
   - `fee = 1 ether * 10_000 / 10_000 = 1 ether`
   - `amountAfterFee = 0`
   - `rsETHAmount = 0`
4. `feeEarnedInETH += 1 ether`; `wrsETH.mint(user, 0)` — user receives nothing.
5. `BRIDGER_ROLE` calls `withdrawFees(bridger)` and receives the 1 ETH.
6. User has lost 1 ETH with no recourse. [6](#0-5)

    function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
