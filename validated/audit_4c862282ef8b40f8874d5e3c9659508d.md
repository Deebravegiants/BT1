### Title
Node Explicitly Bails for Whitelisted Chains (Ethereum, Solana, Ton) Causing Permanent Request Timeout - (File: crates/node/src/providers/verify_foreign_tx/sign.rs)

---

### Summary

The `execute_foreign_chain_request` function in the MPC node explicitly returns an error for `Ethereum`, `Solana`, and `Ton` chain variants. The on-chain contract, however, only checks whether the requested chain appears in the supported-chains registry before accepting and queuing a `verify_foreign_transaction` request. If any of these three chains is present in the registry (reachable via governance vote or node registration), every user request for that chain is accepted by the contract, queued as a yield, and then silently abandoned by every node — causing the yield to time out and the caller's cross-contract promise to fail permanently.

---

### Finding Description

`ForeignChainRpcRequest` is a `#[non_exhaustive]` enum with twelve variants, including `Ethereum`, `Solana`, and `Ton`. [1](#0-0) 

The contract's `verify_foreign_transaction` entry point accepts a request for any chain that appears in `get_supported_foreign_chains()`: [2](#0-1) 

If the chain is present, the request is enqueued as a NEAR yield-resume call and stored in `pending_verify_foreign_tx_requests`. The contract then waits for a node to call `respond_verify_foreign_tx`.

On the node side, `execute_foreign_chain_request` is the single dispatch point for all chain families. It contains three explicit hard-coded bails: [3](#0-2) [4](#0-3) 

Because every node hits the same bail, no node ever calls `respond_verify_foreign_tx` for these chains. The yield-resume timeout (200 blocks, ~`REQUEST_EXPIRATION_BLOCKS`) eventually fires, the pending entry is cleaned up, and the caller's promise resolves as a failure — with no funds returned beyond the 1 yoctoNEAR deposit.

The node-side `chain_is_supported` guard runs *before* the match and passes whenever the chain is in the on-chain registry, so it provides no protection: [5](#0-4) 

The sandbox test suite already demonstrates that `Solana` can be inserted into the supported-chains registry: [6](#0-5) 

---

### Impact Explanation

**Medium.** Once any of the three chains is present in the supported-chains registry, every `verify_foreign_transaction` call for that chain is accepted by the contract (consuming gas and the 1 yoctoNEAR deposit), stored as a pending yield, and then permanently abandoned by all nodes. The request lifecycle invariant — that an accepted request will eventually receive a response — is broken. Any bridge or application that relies on `verify_foreign_transaction` to gate an on-NEAR fund release (the primary stated use-case of the feature) will have its cross-contract call fail on timeout, leaving the bridged funds in limbo. This matches the allowed Medium impact: *"request-lifecycle … manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

`Ethereum` and `Solana` are the two largest chains by TVL and are explicitly named in the `ForeignChainRpcRequest` enum, signalling intent to support them. Governance participants are expected to vote to whitelist major chains. A node operator who configures an Ethereum or Solana RPC endpoint and registers it via `register_available_foreign_chain_config` would cause those chains to appear in the available set, making the contract accept requests that no node can fulfill. No attacker capability beyond submitting a normal `verify_foreign_transaction` call is required once the chain is registered.

---

### Recommendation

1. **Enforce the invariant at the contract level**: add a compile-time or runtime allowlist of chains that the node binary actually handles, and reject governance votes or node registrations for chains not in that list.
2. **Remove the stub variants** (`Ethereum`, `Solana`, `Ton`) from `ForeignChainRpcRequest` until full node-side support is implemented, preventing them from ever appearing in the supported-chains registry.
3. **Alternatively**, replace the `bail!` stubs with a structured `ChainNotYetImplemented` error that is surfaced to the contract as a non-retryable failure, so the yield is resolved immediately rather than left to time out.

---

### Proof of Concept

1. Governance (threshold of participants) votes to whitelist `ForeignChain::Ethereum` via `register_available_foreign_chain_config` / `vote_add_foreign_chain`.
2. `get_supported_foreign_chains()` now includes `Ethereum`.
3. A bridge contract calls `verify_foreign_transaction` with `ForeignChainRpcRequest::Ethereum(...)`, attaching 1 yoctoNEAR and sufficient gas.
4. The contract passes the `supported_chains.contains(&requested_chain)` check and enqueues a yield.
5. Every MPC node's `execute_foreign_chain_request` hits `bail!("ForeignChainRpcRequest::Ethereum is unsupported")` and returns an error — no node ever calls `respond_verify_foreign_tx`.
6. After ~200 blocks the yield times out; `return_verify_foreign_tx_and_clean_state_on_success` is called with `Err(PromiseError::Failed)`.
7. The bridge's cross-contract promise resolves as a failure; any on-NEAR fund release gated on this promise is permanently blocked.
