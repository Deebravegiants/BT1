### Title
Precision Loss in `compute_royalty_amount` Enables NFT Royalty Bypass via Small Offer Amounts - (File: `chia/wallet/nft_wallet/nft_wallet.py`)

---

### Summary

The `compute_royalty_amount` function uses sequential integer division (`//`) that truncates to zero for small offer amounts. When the computed royalty is zero, `make_nft1_offer` silently skips the royalty payment coin and also filters out the corresponding trade price from the `trade_prices_list` passed to the on-chain NFT ownership-layer puzzle. The result is that an NFT owner can intentionally set a small offer price to transfer a royalty-enabled NFT without paying any royalty to the creator — bypassing both the wallet-level and on-chain enforcement paths simultaneously.

---

### Finding Description

**Root cause — `compute_royalty_amount`**

```python
# chia/wallet/nft_wallet/nft_wallet.py  lines 60-68
def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
    royalty = uint64(amount)
    if royalty >= abs(offered_amount):
        raise ValueError("Royalty amount meets or exceeds the offered amount")
    return royalty
```

The expression `abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS` performs two integer truncations. For any `offered_amount` where `abs(offered_amount) * percentage < MAX_ROYALTY_BASIS_POINTS` (= 10 000), the result is 0. Concretely:

| offered\_amount | royalty % | computation | result |
|---|---|---|---|
| 99 mojos | 1 % (100 bp) | 99 × 100 // 10 000 | **0** |
| 9 999 mojos | 0.01 % (1 bp) | 9 999 × 1 // 10 000 | **0** |

The existing test `test_small_amount_truncates_to_zero` in `chia/_tests/wallet/nft_wallet/test_nft_royalty.py` (line 42-44) explicitly confirms this behaviour is present and produces `uint64(0)`.

**Consequence 1 — royalty coin is never created**

Inside `make_nft1_offer` (lines 793-912), `compute_royalty_amount` is called for every offered fungible asset:

```python
# line 798
extra_royalty_amount = compute_royalty_amount(amount, request_side_royalty_split, percentage)
payment_list.append((launcher_id, CreateCoin(address, extra_royalty_amount, [address])))
```

When `extra_royalty_amount == 0`, the `CreateCoin` carries amount 0. Later:

```python
# lines 910-912
payments = royalty_payments[asset] if asset in royalty_payments else []
if sum(p.amount for _, p in payments) == 0:
    continue          # ← royalty coin spend is skipped entirely
```

No royalty coin is created, so no XCH/CAT reaches the creator's address.

**Consequence 2 — `trade_prices_list` passed to the on-chain NFT puzzle is emptied**

When the NFT itself is spent, the wallet builds the solution with:

```python
# lines 893-897
trade_prices_list=[
    list(price)
    for price in trade_prices
    if price[0] * offered_royalty_percentages[asset] // MAX_ROYALTY_BASIS_POINTS != 0
],
```

`price[0]` is `amount // offer_side_royalty_split` (line 768). If the per-NFT price is small enough that `price[0] * royalty_percentage // 10000 == 0`, the entry is dropped. An empty `trade_prices_list` is passed to `NFT_TRANSFER_PROGRAM_DEFAULT` (`NFT_OWNERSHIP_TRANSFER_PROGRAM_ONE_WAY_CLAIM_WITH_ROYALTIES`), which uses this list to assert royalty payments on-chain. With an empty list the puzzle imposes no royalty assertion, so the spend is accepted by consensus without any royalty payment.

**Attack path**

1. Alice owns an NFT with a 1 % royalty payable to Bob (the creator).
2. Alice creates an offer: sell NFT for 99 mojos of XCH.
3. Wallet calls `compute_royalty_amount(-99, 1, 100)` → returns `uint64(0)`.
4. `make_nft1_offer` skips the royalty coin spend (line 911) and passes an empty `trade_prices_list` to the NFT puzzle (line 893-897).
5. Carol (Alice's alt account) accepts the offer.
6. The spend bundle is valid on-chain; the NFT transfers to Carol; Bob receives nothing.
7. Alice has effectively transferred the NFT without paying royalties.

The attack requires no privileged access — any NFT owner can craft such an offer.

---

### Impact Explanation

This is a **High** impact finding under the criterion:

> *Corruption of … offer/trade settlement state … with direct security impact.*

The NFT royalty mechanism is the primary economic protection for NFT creators. By exploiting the truncation-to-zero behaviour, an unprivileged NFT owner can permanently divert royalty payments away from the creator on every secondary-market transfer, causing direct, irreversible financial loss. Because the `trade_prices_list` passed to the on-chain puzzle is empty, the bypass is complete at the consensus layer — it cannot be detected or reversed after the block is confirmed.

---

### Likelihood Explanation

Medium. The attacker must be the current NFT owner and must deliberately choose an offer price below the royalty threshold (e.g., < 100 mojos for a 1 % royalty). The NFT is then sold to a colluding account at a nominal price, after which it can be re-listed at market value. The technique is repeatable on every resale and requires no special tooling beyond the standard Chia wallet RPC.

---

### Recommendation

1. **Round up** the royalty calculation so that any non-zero percentage on a non-zero amount always yields at least 1 mojo:
   ```python
   # ceiling division: (a * b + c - 1) // c
   amount = (abs(offered_amount) * percentage + MAX_ROYALTY_BASIS_POINTS - 1) \
            // royalty_split // MAX_ROYALTY_BASIS_POINTS
   ```
2. **Enforce a minimum royalty floor**: if `percentage > 0` and `offered_amount != 0`, require `royalty >= 1`.
3. Apply the same fix to the `royalty_calculation` static method (line 719) and the `trade_prices_list` filter (line 896), which contain the same truncation pattern.
