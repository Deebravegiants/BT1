### Title
`verify_foreign_transaction()` Discards Caller Identity When Creating Request, Unlike `sign()` and `request_app_private_key()` - (File: crates/contract/src/lib.rs)

### Summary
`verify_foreign_transaction()` calls `check_request_preconditions()` but silently discards the returned `predecessor` account ID. As a result, the stored `VerifyForeignTransactionRequest` carries no caller-identity binding (no `tweak` field), and the MPC network signs the foreign-tx payload with the **root** ForeignTx domain key rather than a caller-specific derived key. This is structurally inconsistent with `sign()` and `request_app_private_key()`, which both capture and use the predecessor to derive a per-caller tweak, and it breaks the request-lifecycle isolation invariant the design explicitly requires.

### Finding Description

`sign()` and `request_app_private_key()` both capture the predecessor returned by `check_request_preconditions()` and embed it into the stored request via tweak/app-id derivation:

```rust
// sign()
let (domain_config, predecessor) = self.check_request_preconditions(...);
let request = SignatureRequest::new(request.domain_id, request.payload, &predecessor, &request.path);
``` [1](#0-0) 

```rust
// request_app_private_key()
let (_, predecessor) = self.check_request_preconditions(...);
let request = CKDRequest::new(request.app_public_key, domain_id, &predecessor, &request.derivation_path);
``` [2](#0-1) 

`verify_foreign_transaction()` does the opposite — it calls `check_request_preconditions()` but **discards the return value entirely**, then converts the args to a `VerifyForeignTransactionRequest` that contains no predecessor and no tweak:

```rust
// verify_foreign_transaction() — predecessor is DISCARDED
self.check_request_preconditions(request.domain_id, DomainPurpose::ForeignTx, ...);
let request = args_into_verify_foreign_tx_request(request);   // no predecessor passed
``` [3](#0-2) 

`args_into_verify_foreign_tx_request` simply copies the three fields with no identity binding:

```rust
pub fn args_into_verify_foreign_tx_request(args: dtos::VerifyForeignTransactionRequestArgs)
    -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
``` [4](#0-3) 

The stored `VerifyForeignTransactionRequest` struct has no `tweak` field at all:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [5](#0-4) 

Consequently, `respond_verify_foreign_tx()` verifies the MPC signature against the **root** public key (`secp_pk`), not a derived key:

```rust
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,   // root key — no tweak applied
)
``` [6](#0-5) 

Compare to `respond()` for regular sign requests, which derives the expected key from the stored tweak:

```rust
let expected_public_key = derive_key_secp256k1(&affine, &request.tweak)...;
``` [7](#0-6) 

The design document explicitly states that `verify_foreign_transaction()` **should** derive a per-caller tweak from `(predecessor_id, derivation_path)` using a domain-separated prefix, and that `VerifyForeignTransactionRequestArgs` should carry a `derivation_path` field: [8](#0-7) 

Neither the `derivation_path` field nor the tweak derivation is present in the production implementation.

The existing test explicitly documents the caller-agnostic behavior as a known property:

```rust
// Then: both yields are queued under the single (caller-agnostic) request key.
``` [9](#0-8) 

### Impact Explanation

Because the stored request key contains no caller identity, two distinct callers (Alice and Bob) who submit the same `ForeignChainRpcRequest` are queued under the **same** map entry and both receive the **same** MPC signature when one response arrives. The signature is issued under the root ForeignTx domain key and is not bound to any specific predecessor account.

Any bridge or omnibridge contract that calls `verify_foreign_transaction()` expecting a caller-specific signed attestation — as the design document describes — will instead receive a signature that any other account could have obtained for the same foreign transaction. An adversary can:

1. Observe Alice's pending `verify_foreign_transaction` request on-chain.
2. Submit the identical request from their own account (paying only 1 yoctoNEAR deposit).
3. Receive the same root-key signature when the MPC network responds.
4. Present that signature to a bridge contract to claim Alice's bridged assets, since the signature carries no binding to Alice's NEAR account.

This breaks the request-lifecycle isolation invariant and enables cross-caller replay of foreign-chain verification attestations, matching the "cross-chain replay / forged foreign-chain verification / invalid bridge execution" impact class.

### Likelihood Explanation

The entry path requires only a 1 yoctoNEAR deposit and sufficient gas — no special role or privilege. Any unprivileged NEAR account can call `verify_foreign_transaction()`. The attacker only needs to observe a pending request (visible on-chain) and submit the same arguments. Likelihood is **Medium**: the attack requires a live bridge contract that relies on caller-specific MPC attestations, but the MPC contract itself is the necessary vulnerable step.

### Recommendation

1. Add a `derivation_path: String` field to `VerifyForeignTransactionRequestArgs` (as the design document specifies).
2. Capture the `predecessor` returned by `check_request_preconditions()` in `verify_foreign_transaction()`.
3. Derive a per-caller tweak using the foreign-tx-specific prefix and store it in `VerifyForeignTransactionRequest`, mirroring the pattern used by `sign()` and `request_app_private_key()`.
4. Update `respond_verify_foreign_tx()` to verify the signature against the derived key (using the stored tweak) rather than the root key.

### Proof of Concept

```
// Attacker observes Alice's pending request on-chain:
// VerifyForeignTransactionRequestArgs { request: Bitcoin(tx_id=[7u8;32], ...), domain_id: 0, ... }

// Attacker submits the identical request from their own account:
testing_env!(VMContextBuilder::new()
    .predecessor_account_id("attacker.near".parse().unwrap())
    .attached_deposit(NearToken::from_yoctonear(1))
    .build());
contract.verify_foreign_transaction(identical_request_args);

// Both Alice and attacker are now queued under the same caller-agnostic key.
// When the MPC network responds, both receive the same root-key signature.
// The attacker presents this signature to the bridge contract to claim Alice's funds.
```

The existing unit test `verify_foreign_transaction__should_queue_duplicates_from_different_callers` already demonstrates that Alice and Bob receive the same signature — the only missing step is showing that the attacker can exploit this in a bridge context. [10](#0-9)

                extractors: vec![BitcoinExtractor::BlockHash],
            }),

        let request = args_into_verify_foreign_tx_request(request_args.clone());

        // When: caller alice submits the request.
        let alice = AccountId::from_str("alice.near").unwrap();
        testing_env!(

                .predecessor_account_id(alice)
                .current_account_id(context.current_account_id.clone())
                .attached_deposit(NearToken::from_yoctonear(1))
                .build()

        // And: caller bob submits the identical request — a different account would today
        // be blocked from receiving a response by alice's submission.
        let bob = AccountId::from_str("bob.near").unwrap();
        testing_env!(
            VMContextBuilder::new()
                .signer_account_id(bob.clone())
                .predecessor_account_id(bob)
                .current_account_id(context.current_account_id.clone())
                .attached_deposit(NearToken::from_yoctonear(1))
                .build()
        );
        contract.verify_foreign_transaction(request_args);

        // Then: both yields are queued under the single (caller-agnostic) request key.
        assert_eq!(
            contract
                .pending_verify_foreign_tx_requests
                .get(&request)
                .map(|q| q.len()),
            Some(2),
            "duplicate foreign-tx requests from different callers should fan out",
        );

        // When: a single valid response is delivered.
        let payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: request.request.clone(),
            values: vec![ExtractedValue::BitcoinExtractedValue(
                BitcoinExtractedValue::BlockHash([42u8; 32].into()),
            )],
        });
        let payload_hash_arr = payload.compute_msg_hash().unwrap().0;
        let secret_key_ec: elliptic_curve::SecretKey<Secp256k1> =
            elliptic_curve::SecretKey::from_bytes(&secret_key.to_bytes()).unwrap();
        let signing_key = SigningKey::from_bytes(&secret_key_ec.to_bytes()).unwrap();
        let (signature, recovery_id) = signing_key
            .sign_prehash_recoverable(&payload_hash_arr)
            .unwrap();
        let response = VerifyForeignTransactionResponse {
            payload_hash: payload.compute_msg_hash().unwrap(),
            signature: dtos::SignatureResponse::Secp256k1(
                dtos::K256Signature::from_ecdsa_recoverable(&signature, recovery_id),
            ),
        };

        with_active_participant_and_attested_context(&contract);
        contract
            .respond_verify_foreign_tx(request.clone(), response)
            .expect("respond_verify_foreign_tx should succeed");

        // Then: both queued yields are drained from the single map entry.
        assert!(
            contract
                .pending_verify_foreign_tx_requests
                .get(&request)
                .is_none()
        );
    }

## Tweak Derivation (Sign vs ForeignTx)

`verify_foreign_transaction()` uses a **different tweak derivation prefix** than `sign()` so the same
`(predecessor_id, derivation_path)` can never yield the same derived key across the two purposes.

Design:

* Keep the existing sign tweak derivation prefix unchanged.
* Introduce a foreign-tx-specific prefix and derive the tweak from the same `(predecessor_id, derivation_path)`
  input using the same hash construction.
* The contract derives the tweak internally from `request.derivation_path` (callers do not submit raw tweaks).

Example:

```rust
const SIGN_TWEAK_DERIVATION_PREFIX: &str =
    "near-mpc-recovery v0.1.0 epsilon derivation:";
const FOREIGN_TX_TWEAK_DERIVATION_PREFIX: &str =
    "near-mpc-recovery v0.1.0 foreign-tx epsilon derivation:";

pub fn derive_sign_tweak(predecessor_id: &AccountId, path: &str) -> Tweak {
    let hash: [u8; 32] = derive_from_path(SIGN_TWEAK_DERIVATION_PREFIX, predecessor_id, path);
    Tweak::new(hash)
}

pub fn derive_foreign_tx_tweak(predecessor_id: &AccountId, path: &str) -> Tweak {
    let hash: [u8; 32] = derive_from_path(FOREIGN_TX_TWEAK_DERIVATION_PREFIX, predecessor_id, path);
    Tweak::new(hash)
}
```

This ensures key material used for validated foreign transactions is **always** distinct from
general-purpose `sign()` keys, even if the same account and derivation path are reused.
