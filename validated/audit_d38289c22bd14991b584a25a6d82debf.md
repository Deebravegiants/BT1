### Title
THRESHOLD(n,n) Blob Accepted in Place of AND Node Causes Silent Wrong Secret Reconstruction — (`src/cbmpc/crypto/secret_sharing.cpp`)

### Summary
`validate_tree` permits `THRESHOLD(threshold, n)` where `threshold == n`, but `share_recursive` and `reconstruct_recursive` use fundamentally different arithmetic for `AND` (additive) vs `THRESHOLD` (Lagrange polynomial). An attacker who controls a serialized `ac_owned_t` blob can substitute an `AND(p0,p1)` node with `THRESHOLD(2,[p0,p1])`. The deserialized structure passes all validation, but `reconstruct` silently returns a wrong value when applied to shares that were generated under AND semantics.

### Finding Description

**`validate_tree` — missing semantic equivalence guard** [1](#0-0) 

The only checks for a `THRESHOLD` node are `threshold >= 1` and `threshold <= n`. `threshold == n` is explicitly allowed. There is no rule that rejects `THRESHOLD(n,n)` or normalizes it to `AND`.

**`ac_owned_t::convert` — deserialization trusts `validate_tree` as the sole gate** [2](#0-1) 

After `convert_node` reconstructs the tree from attacker-supplied bytes, `validate_tree()` is the only check before the structure is accepted. A crafted blob encoding `THRESHOLD(2,[p0,p1])` passes this gate.

**`share_recursive` — AND uses additive, THRESHOLD uses Lagrange** [3](#0-2) 

- `AND`: calls `share_and()` — produces shares `s_i` such that `Σ s_i ≡ S (mod q)`.
- `THRESHOLD`: calls `share_threshold()` — evaluates a degree-`(t-1)` polynomial `f` at each child's PID, where `f(0) = S`.

These are algebraically incompatible. Shares produced by one scheme cannot be correctly reconstructed by the other.

**`reconstruct_recursive` — AND sums, THRESHOLD interpolates** [4](#0-3) 

If AND shares `(s0, s1)` with `s0 + s1 = S` are fed into the THRESHOLD(2,2) branch, Lagrange interpolation at `x=0` computes:

```
result = (s1·pid0 − s0·pid1) / (pid0 − pid1)  (mod q)
```

This equals `S` only when `s0·pid0 = s1·pid1`, which is not true for random additive shares. The function returns `SUCCESS` with a wrong `x`.

### Impact Explanation

An attacker who can supply or modify a serialized `ac_owned_t` blob (e.g., a malicious protocol peer, a tampered stored blob, or a Byzantine participant below threshold) can cause an honest party to reconstruct a wrong secret without any error signal. The wrong secret is silently accepted and used downstream (key derivation, decryption, signing). If the attacker also knows the AND shares and the child PIDs (e.g., as a participant), the wrong output is fully predictable, enabling them to anticipate the derived key material.

### Likelihood Explanation

The precondition — attacker-controlled `ac_owned_t` bytes — is realistic wherever the access structure is transmitted or stored separately from the shares and is not independently authenticated. The substitution is structurally minimal (same leaf names, same child count, only the node type byte and threshold field change), making it easy to craft.

### Recommendation

In `validate_tree`, add a check that rejects `THRESHOLD(n,n)` (i.e., `threshold == n`) with an error, or canonicalize it to `AND` during deserialization. Alternatively, bind the access-structure blob to the shares via a commitment or MAC so that substitution is detectable before reconstruction.

### Proof of Concept

```cpp
// Both structures share the same two leaves p0, p1 and the same secret S.
// 1. Share under AND
ac_t and_ac(and_node, curve);          // AND(p0, p1)
auto and_shares = and_ac.share(q, S);

// 2. Serialize a THRESHOLD(2,[p0,p1]) blob and deserialize it
ac_owned_t thr_blob(thr_node, curve);  // THRESHOLD(2,[p0,p1])
buf_t blob = coinbase::convert(thr_blob);
ac_owned_t swapped;
ASSERT_OK(coinbase::convert(swapped, blob));  // passes validate_tree

// 3. Reconstruct AND shares using the swapped THRESHOLD(2,2) structure
bn_t result;
ASSERT_OK(swapped.reconstruct(q, and_shares, result));  // returns SUCCESS

// 4. result != S  — wrong secret silently accepted
EXPECT_NE(result, S);
```
