### Title
`respond_verify_foreign_tx` accepts an arbitrary `payload_hash` without binding it to the pending request, enabling cross-request signature replay by a single malicious participant - (File: `crates/contract/src/lib.rs`)

### Summary
`respond_verify_foreign_tx` verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash`, but never checks that `response.payload_hash` is the correct canonical hash for the `request` being answered. A single malicious attested MPC participant can replay a valid `(payload_hash, signature)` pair produced for one foreign-transaction verification request to satisfy a completely different pending request, delivering a forged attestation to the waiting caller.

### Finding Description
In `respond_verify_foreign_tx`, the contract performs two independent checks:

1. It looks up the pending queue using `request` as the key.
2. It verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash`.

What it does **not** do is verify that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload::V1 { request: request.request, values: <any_valid_values> }))`. The `payload_hash` field is entirely caller-supplied and is never recomputed from the `request` argument. [1](#0-0) 

The `ForeignTxSignPayload` that nodes are supposed to sign encodes both the original `ForeignChainRpcRequest` (including `tx_id`) and the `ExtractedValue`s observed on-chain: [2](#0-1) 

The contract never reconstructs this payload from `request.request` to confirm the submitted `payload_hash` is consistent with it. The signature check only proves the MPC key signed *some* hash — not the hash that corresponds to the pending request. [3](#0-2) 

After the signature check passes, `resolve_yields_for` drains the pending queue for `request` and delivers the raw `response` bytes (containing the attacker-chosen `payload_hash`) to every waiting yield: [4](#0-3) [5](#0-4) 

### Impact Explanation
A single malicious attested MPC participant can deliver a `VerifyForeignTransactionResponse` where `payload_hash` encodes the extracted values of a **different** foreign transaction than the one the user requested. The user's NEAR contract (e.g., an Omnibridge inbound handler) receives `{payload_hash, signature}` and, if it does not independently recompute the expected hash from the original request, will accept a forged attestation. This enables invalid bridge execution or double-spend conditions: the attacker can attest that a transaction with tx_id A has the block-hash or log values of a different transaction B, causing the bridge to credit a transfer that did not occur or to credit it twice.

This matches the allowed impact: **High — forged foreign-chain verification / cross-chain replay that causes invalid bridge execution or double-spend conditions.**

### Likelihood Explanation
The attacker must be a single attested MPC participant — the exact threat model the threshold system is designed to resist. The system is explicitly designed so that no single participant can unilaterally forge outputs. Obtaining a valid `(payload_hash, signature)` pair for a different request is straightforward: the attacker participated in the MPC computation that produced it, or can observe a previously submitted `respond_verify_foreign_tx` transaction on-chain. No key leakage, TEE attack, or threshold collusion is required.

### Recommendation
Inside `respond_verify_foreign_tx`, recompute the expected `payload_hash` from the `request` argument and the `response.payload_hash`, and reject any response where the hash does not match a valid `ForeignTxSignPayload` for that request. Concretely, the contract should verify:

```
response.payload_hash == SHA-256(borsh(ForeignTxSignPayload::V1 {
    request: request.request,
    values: <values encoded in payload_hash>
}))
```

Because the contract does not store the `values` (only the request key), the simplest fix is to require the responder to also submit the `values` alongside the response, then recompute and compare the hash on-chain before accepting. Alternatively, the contract can store a commitment to the expected payload structure at request-submission time.

### Proof of Concept

**Setup**: Two pending requests exist:
- Request A: `verify_foreign_transaction(Bitcoin, tx_id=[0xAA;32], extractors=[BlockHash])`
- Request B: `verify_foreign_transaction(Bitcoin, tx_id=[0xBB;32], extractors=[BlockHash])`

**Step 1**: The MPC network legitimately processes request B and produces:
```
payload_B = ForeignTxSignPayload::V1 { request: Bitcoin(tx_id=[0xBB;32]), values: [BlockHash([0xCC;32])] }
payload_hash_B = SHA-256(borsh(payload_B))
signature_B = MPC_sign(payload_hash_B)
```

**Step 2**: The malicious participant calls:
```
respond_verify_foreign_tx(
    request = VerifyForeignTransactionRequest { request: Bitcoin(tx_id=[0xAA;32]), ... },
    response = VerifyForeignTransactionResponse {
        payload_hash: payload_hash_B,   // hash of tx_id B's payload
        signature: signature_B,          // valid signature over payload_hash_B
    }
)
```

**Step 3**: The contract:
- Finds the pending queue for request A (tx_id=[0xAA;32]) ✓
- Verifies `signature_B` over `payload_hash_B` against the root public key ✓ (valid)
- Resolves all yields for request A with `{payload_hash_B, signature_B}` ✓

**Step 4**: The user's contract waiting for request A receives an attestation claiming Bitcoin tx_id=[0xAA;32] has block-hash `[0xCC;32]` — which is actually the block-hash of tx_id=[0xBB;32]. The bridge credits the transfer associated with tx_id A using forged on-chain evidence. [6](#0-5) [7](#0-6)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

    all(feature = "abi", not(target_arch = "wasm32")),
    derive(schemars::JsonSchema, borsh::BorshSchema)

pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
