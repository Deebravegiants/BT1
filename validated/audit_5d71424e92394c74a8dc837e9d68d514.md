### Title
Single-Step Ownership Transfer Enables Permanent Loss of Admin Control Over Token-Holding Distributor Contracts - (File: contracts/KERNEL/KernelTop100MerkleDistributor.sol)

### Summary
Multiple production contracts in LRT-rsETH inherit `OwnableUpgradeable` (OpenZeppelin v4 single-step), which exposes a `transferOwnership()` that immediately and irrevocably sets a new owner, and a `renounceOwnership()` that sets the owner to `address(0)`. No confirmation from the new address is required. The most critical instance is `KernelTop100MerkleDistributor`, which holds KERNEL tokens on behalf of users and exposes an owner-only `withdrawTokens()` that can drain the entire token balance to any address.

### Finding Description
`KernelTop100MerkleDistributor`, `KernelMerkleDistributor`, `MerkleDistributor`, and `MerkleBlastPointsDistributor` all inherit `OwnableUpgradeable` directly. None of the production contracts in scope use `Ownable2StepUpgradeable`. [1](#0-0) 

The inherited `transferOwnership()` immediately writes the new owner in a single transaction: [2](#0-1) 

The only guard is `newOwner != address(0)`. Any other wrong address is accepted without confirmation. Additionally, `renounceOwnership()` sets the owner to `address(0)` with no guard at all: [3](#0-2) 

`KernelTop100MerkleDistributor` exposes `withdrawTokens()`, which is gated solely by `onlyOwner` and transfers any amount of any token (including the KERNEL reward tokens held for users) to an arbitrary recipient: [4](#0-3) 

If ownership is accidentally transferred to a wrong address, that address immediately gains the ability to call `withdrawTokens()` and drain all KERNEL tokens. If `renounceOwnership()` is called accidentally, `withdrawTokens()`, `pause()`, `unpause()`, `setKernelDepositPool()`, `setProtocolTreasury()`, and `setFeeInBPS()` are all permanently locked.

The same single-step pattern applies to `KernelMerkleDistributor` (owner controls `setMerkleRoot`, `pause/unpause`, `setKernelDepositPool`, `setProtocolTreasury`): [5](#0-4) 

And to `MerkleDistributor` (owner controls `setMerkleRoot`, `setToken`, `setProtocolTreasury`, `pause/unpause`): [6](#0-5) 

No production contract in scope uses `Ownable2StepUpgradeable`: [7](#0-6) 

### Impact Explanation
**Primary (High): Theft of unclaimed yield.** If the owner of `KernelTop100MerkleDistributor` accidentally calls `transferOwnership(wrongAddress)` and the wrong address is controlled by an attacker, the attacker immediately calls `withdrawTokens(kernel, totalBalance, attacker)` and drains all KERNEL tokens allocated to users as unclaimed yield.

**Secondary (Medium): Permanent freezing of unclaimed yield.** If `renounceOwnership()` is called accidentally on any of the distributor contracts, `withdrawTokens()` and `setMerkleRoot()` are permanently locked. Excess KERNEL tokens held in `KernelTop100MerkleDistributor` cannot be recovered, and future distribution epochs in `KernelMerkleDistributor` / `MerkleDistributor` cannot be published (no new merkle root can be set), permanently freezing unclaimed yield for all future epochs.

### Likelihood Explanation
Ownership transfers are infrequent, high-stakes operations typically performed manually. A single character error in the target address (e.g., copy-paste from a wrong tab, address truncation in a UI) is a realistic operational mistake. The `renounceOwnership()` function is callable by the owner at any time with no additional confirmation. Both paths require only one erroneous transaction from the legitimate owner, with no external attacker involvement needed to trigger the freeze or theft.

### Recommendation
Replace `OwnableUpgradeable` with `Ownable2StepUpgradeable` in all four distributor contracts. `Ownable2StepUpgradeable` requires the nominated address to call `acceptOwnership()` before the transfer takes effect, eliminating the single-point-of-failure. Additionally, consider removing or overriding `renounceOwnership()` in contracts that hold user funds, since permanent renouncement is almost never the intended behavior for a live distributor.

### Proof of Concept
1. `KernelTop100MerkleDistributor` is deployed and funded with 1,000,000 KERNEL tokens for user distribution.
2. The owner intends to transfer ownership to a new multisig at `0xAAAA...AAAA` but accidentally pastes `0xBBBB...BBBB` (an EOA controlled by an attacker who monitors the mempool).
3. Owner calls `transferOwnership(0xBBBB...BBBB)`. The call succeeds immediately; `_owner` is set to `0xBBBB...BBBB` in the same transaction. [8](#0-7) 
4. Attacker calls `withdrawTokens(kernelAddress, 1_000_000e18, attackerAddress)`. [4](#0-3) 
5. All 1,000,000 KERNEL tokens (user unclaimed yield) are transferred to the attacker. Legitimate users can no longer claim their vested rewards.
