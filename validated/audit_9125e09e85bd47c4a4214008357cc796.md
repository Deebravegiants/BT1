### Title
Excess NEAR Deposit Silently Consumed in `submit_participant_info` When Existing Participant Re-submits Attestation - (`File: crates/contract/src/lib.rs`)

---

### Summary

The `submit_participant_info` function is `#[payable]` but only refunds excess attached deposits along one branch of its deposit-handling logic. When an existing protocol participant re-submits their attestation (e.g., after a TEE upgrade or certificate renewal), the branch that performs the refund is skipped entirely, and any NEAR tokens the caller attached are permanently locked in the contract.

---

### Finding Description

`submit_participant_info` computes a boolean gate `attestation_storage_must_be_paid_by_caller` to decide whether to enforce and refund the storage deposit:

```rust
let caller_is_not_participant = self.voter_account().is_err();
let is_new_attestation = matches!(
    attestation_insertion_result,
    ParticipantInsertion::NewlyInsertedParticipant
);

let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... check attached >= cost, then refund diff ...
    if let Some(diff) = attached.checked_sub(cost)
        && diff > NearToken::from_yoctonear(0)
    {
        Promise::new(account_id).transfer(diff).detach();
    }
}

Ok(())
``` [1](#0-0) 

When the caller **is** a current participant (`caller_is_not_participant == false`) **and** the TLS key already exists in `stored_attestations` (`is_new_attestation == false`), the condition evaluates to `false`. The entire `if` block — including the refund — is skipped. The function returns `Ok(())` and any NEAR attached by the caller is absorbed into the contract balance with no path to recovery.

The `add_participant` helper confirms the two return variants that drive `is_new_attestation`:

```rust
Ok(match insertion {
    Some(_previous_attestation) => ParticipantInsertion::UpdatedExistingParticipant,
    None => ParticipantInsertion::NewlyInsertedParticipant,
})
``` [2](#0-1) 

Every other `#[payable]` entry point in the contract correctly refunds excess. `propose_update` does so unconditionally:

```rust
if let Some(diff) = attached.checked_sub(required)
    && diff > NearToken::from_yoctonear(0)
{
    Promise::new(proposer).transfer(diff).detach();
}
``` [3](#0-2) 

And `require_deposit`, used by `sign`, `request_app_private_key`, and `verify_foreign_transaction`, also always refunds:

```rust
Some(diff) => {
    if diff > NearToken::from_yoctonear(0) {
        log!("refund excess deposit {diff} to {predecessor}");
        Promise::new(predecessor.clone()).transfer(diff).detach();
    }
}
``` [4](#0-3) 

`submit_participant_info` is the sole `#[payable]` method that can silently consume a deposit.

---

### Impact Explanation

**Medium.** This breaks the production accounting invariant that excess deposits attached to `#[payable]` contract calls must be refunded. Any NEAR tokens attached by an existing participant during a re-submission are permanently locked in the contract. The funds are not controlled by the MPC network's signing flow, so this does not rise to Critical theft of MPC-controlled assets, but it is a direct, irreversible loss of the caller's own funds caused by a contract logic gap — matching the Medium category of balance/accounting invariant violations.

---

### Likelihood Explanation

Participants must periodically re-submit attestations: TEE certificates expire, Docker image hashes are rotated via `vote_code_hash`, and launcher hashes change. Each re-submission is a normal operational event. Because the function is `#[payable]` and the first submission *does* require a deposit, operators following the same invocation pattern for re-submissions will routinely attach NEAR. The vulnerable path is therefore hit on every routine attestation renewal by an active participant.

---

### Recommendation

Unconditionally refund any excess deposit at the end of `submit_participant_info`, regardless of the `attestation_storage_must_be_paid_by_caller` branch:

```rust
// After the storage-cost block, always refund any unspent deposit.
let attached = env::attached_deposit();
let cost = if attestation_storage_must_be_paid_by_caller {
    // existing cost computation
    env::storage_byte_cost().saturating_mul(storage_used as u128)
} else {
    NearToken::from_yoctonear(0)
};
if let Some(diff) = attached.checked_sub(cost)
    && diff > NearToken::from_yoctonear(0)
{
    Promise::new(account_id).transfer(diff).detach();
}
```

Alternatively, mirror the pattern already used in `require_deposit` and `propose_update` by always computing the refund after the storage check, independent of the branch taken.

---

### Proof of Concept

1. Participant `alice.near` calls `submit_participant_info` for the first time. `is_new_attestation = true`, so `attestation_storage_must_be_paid_by_caller = true`. She attaches 0.1 NEAR; the storage cost is 0.05 NEAR; 0.05 NEAR is refunded. ✓

2. Alice's TEE certificate expires. She calls `submit_participant_info` again with the same TLS key and a new attestation, attaching 0.1 NEAR as before.

3. `add_participant` finds an existing entry for Alice's TLS key and returns `ParticipantInsertion::UpdatedExistingParticipant`. So `is_new_attestation = false`.

4. `self.voter_account()` succeeds because Alice is a current participant. So `caller_is_not_participant = false`.

5. `attestation_storage_must_be_paid_by_caller = false || false = false`. The `if` block is skipped.

6. The function returns `Ok(())`. Alice's 0.1 NEAR is permanently absorbed into the contract balance with no refund issued. [5](#0-4)
