### Title
Unvalidated `response.payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Root-Key Signature Replay - (File: crates/contract/src/lib.rs)

---

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is a valid MPC root-key signature over `response.payload_hash`, but never checks that `response.payload_hash` is actually the hash of `ForeignTxSignPayload` derived from the submitted `request`. Any single Byzantine attested participant can replay a valid root-key signature obtained from any previously completed foreign-tx verification to forge a settlement for any currently pending `verify_foreign_transaction` request, delivering a fabricated `VerifyForeignTransactionResponse` to the original caller and permanently consuming the pending yield slot.

---

### Finding Description

In `respond_verify_foreign_tx` (`crates/contract/src/lib.rs:691-754`), the contract performs the following checks:

1. Caller is an attested participant (`assert_caller_is_attested_participant_and_protocol_active`, line 705).
2. `response.signature` is a valid ECDSA signature over `response.payload_hash` under the **root** public key (lines 718–743).
3. The `request` key exists in `pending_verify_foreign_tx_requests` (inside `resolve_yields_for`, line 749).

What is **absent**: any check that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload::V1 { request: request.request, values: <observed> }))`. The `payload_hash` field is taken verbatim from the caller-supplied `response` object at line 726:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,   // ← entirely caller-controlled
    &secp_pk,
)
```

Contrast this with `respond()` for regular sign requests (lines 600–608), where the payload hash is read from the **contract-stored** `request.payload` — a value committed at `sign()` time and never supplied by the responder:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,   // ← from contract state, not from caller
    &expected_public_key,
)
```

Foreign-tx signing uses a **zero tweak** (root key), as confirmed in `build_signature_request` (`crates/node/src/providers/verify_foreign_tx/sign.rs:43`):

```rust
tweak: Tweak::new([0u8; 32]),
```

This means every `respond_verify_foreign_tx` signature is produced under the same root key. Any such signature, once visible on-chain, is reusable against any other pending foreign-tx request, because the contract never binds the `payload_hash` to the specific `request` being resolved.

The `ForeignTxSignPayloadV1` struct (`crates/near-mpc-contract-interface/src/types/foreign_chain.rs:1499-1502`) does include the `ForeignChainRpcRequest` (containing `tx_id`, `confirmations`, `extractors`) and the `values`. However, it does **not** include `domain_id` or `payload_version`. More critically, the contract never recomputes this hash from the `request` argument — it cannot, because the `values` (extracted from the foreign chain by MPC nodes) are not stored on-chain. The contract therefore has no way to independently derive the expected `payload_hash` and is entirely dependent on the responder supplying the correct one.

---

### Impact Explanation

A single Byzantine attested participant executes the following cross-request replay:

1. **Observe** a completed `respond_verify_foreign_tx` transaction on-chain for request A (tx_id=X). Extract `payload_hash_A` and `sig_A` (a valid root-key ECDSA signature over `payload_hash_A`).

2. **Wait** for any pending `verify_foreign_transaction` request B (tx_id=Y, possibly a different transaction or even a different domain) to appear in `pending_verify_foreign_tx_requests`.

3. **Call** `respond_verify_foreign_tx(request=B, response={ payload_hash: payload_hash_A, signature: sig_A })`.

4. The contract's signature check passes because `sig_A` is a valid root-key signature over `payload_hash_A`.

5. `resolve_yields_for` drains the entire yield queue for request B, delivering `{ payload_hash: payload_hash_A, signature: sig_A }` to every caller waiting on B.

Consequences:
- **Forged bridge attestation**: Every caller of `verify_foreign_transaction(B)` receives a `VerifyForeignTransactionResponse` whose `payload_hash` corresponds to a completely different foreign transaction (A). A bridge contract that does not independently recompute the expected hash (i.e., does not use `ForeignChainSignatureVerifier::verify_signature` from the SDK at `crates/near-mpc-sdk/src/foreign_chain.rs:42-89`) will accept this as proof that transaction B was verified, enabling double-spend or invalid inbound bridge execution.
- **Permanent yield consumption**: The pending yield slot(s) for request B are drained and cannot be re-resolved. The legitimate MPC response for B, when it arrives, will be rejected with `RequestNotFound`. The caller must resubmit, and the attacker can repeat the replay indefinitely, creating a persistent denial of valid settlement for any targeted request.

---

### Likelihood Explanation

- Requires **one** Byzantine attested participant — strictly below the signing threshold.
- No key material needs to be stolen; the attacker only needs to read a past on-chain transaction (publicly available).
- The attacker does not need to be the signing leader for any round; they only need to observe a completed `respond_verify_foreign_tx` call.
- The attack is repeatable: every time the victim resubmits `verify_foreign_transaction(B)`, the attacker can replay again.
- The only prerequisite is that at least one prior foreign-tx verification has been completed on the same domain, providing a reusable `(payload_hash, signature)` pair.

---

### Recommendation

The contract must bind `response.payload_hash` to the `request` being resolved. Since the contract cannot recompute the full `ForeignTxSignPayload` hash (the `values` are off-chain), the fix must be at the **protocol level**: include a unique, on-chain-verifiable request identifier in the signed payload.

Concretely, extend `ForeignTxSignPayloadV1` to include `domain_id` and a unique per-request nonce (e.g., the `receipt_id` stored at request submission time). The contract stores the `receipt_id` when the yield is created and checks `response.payload_hash == SHA-256(borsh(ForeignTxSignPayload::V1 { request, domain_id, receipt_id, values }))` by recomputing the hash prefix from the stored fields and rejecting any response whose hash does not match the stored prefix.

As an interim measure, the contract should at minimum verify that `domain_id` is encoded in the signed payload, preventing cross-domain replays.
