Audit Report

## Title
Missing Chainlink Staleness Check Allows Stale Price to Determine rsETH Mint Amount — (`contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary
`ChainlinkPriceOracle.getAssetPrice` calls `latestRoundData()` but silently discards `updatedAt` and `answeredInRound`, performing no freshness validation. A stale Chainlink price propagates directly into `LRTDepositPool.getRsETHAmountToMint`, causing depositors to receive an rsETH amount computed from an outdated asset/ETH rate rather than the current fair rate.

## Finding Description
In `ChainlinkPriceOracle.getAssetPrice` (line 52), the call to `latestRoundData()` ignores all staleness fields:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
``` [1](#0-0) 

`updatedAt` and `answeredInRound` are never checked. If the Chainlink feed stops updating, the oracle continues returning the last known price indefinitely.

This stale price flows into the minting formula in `LRTDepositPool.getRsETHAmountToMint`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [2](#0-1) 

`getAssetPrice(asset)` is evaluated live from the (potentially stale) Chainlink feed, while `rsETHPrice` is a stored value last updated when `updateRSETHPrice()` was called. `rsETHPrice` is computed in `_updateRsETHPrice()` via `_getTotalEthInProtocol()`, which also calls `getAssetPrice` for each supported asset. [3](#0-2) 

The divergence scenario: suppose the feed was at price `P0` when `rsETHPrice` was last updated. The feed then moves to `P1` and becomes stale there. Now:
- `getAssetPrice(asset)` returns `P1` (stale)
- `rsETHPrice` was computed using `P0`

The ratio `P1 / rsETHPrice` is inconsistent, causing depositors to receive more or fewer rsETH tokens than the fair rate. The `_beforeDeposit` path has no staleness guard: [4](#0-3) 

The full call chain: `depositAsset` → `_beforeDeposit` → `getRsETHAmountToMint` → `lrtOracle.getAssetPrice(asset)` → `IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset)` → `latestRoundData()` with no staleness check. [5](#0-4) 

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A depositor calling `depositAsset` when the Chainlink feed is stale receives an rsETH amount computed from an outdated price rather than the current fair rate. The deposited LST tokens are not lost, but the rsETH minted does not accurately reflect the current asset/ETH exchange rate, violating the protocol's core minting invariant. Existing holders may be diluted if the stale price is higher than the true price, or new depositors may be underpaid if the stale price is lower.

## Likelihood Explanation
No attacker action is required. The condition arises passively from Chainlink infrastructure failure (node outage, heartbeat failure, feed deprecation). Chainlink feeds for major LSTs (stETH, rETH, etc.) have heartbeats of 1–24 hours; a >24-hour staleness is a realistic failure mode. Any depositor transacting during the staleness window is affected. The `minRSETHAmountExpected` slippage parameter in `depositAsset` does not protect against this because the depositor has no independent reference price to compare against.

## Recommendation
Add a staleness check in `ChainlinkPriceOracle.getAssetPrice` using the `updatedAt` return value:

```solidity
(, int256 price,, uint256 updatedAt,) = priceFeed.latestRoundData();
if (block.timestamp - updatedAt > STALENESS_THRESHOLD) {
    revert StalePrice();
}
```

A configurable `STALENESS_THRESHOLD` per asset (matching each feed's documented heartbeat) is recommended. Optionally, also check `answeredInRound >= roundId` to guard against incomplete rounds.

## Proof of Concept
1. Deploy or fork with `LRTDepositPool`, `LRTOracle`, and `ChainlinkPriceOracle` configured for a supported LST (e.g., stETH).
2. Call `updateRSETHPrice()` to record `rsETHPrice` at the current Chainlink price `P0`.
3. Advance the Chainlink mock feed price to `P1 != P0` (simulating a price move), then stop updating it (simulating staleness via `vm.warp` past the heartbeat without updating the mock feed).
4. Call `getRsETHAmountToMint(asset, 1e18)` — it returns an amount based on `P1 / rsETHPrice(P0)`, not the fair rate.
5. Call `depositAsset(asset, 1e18, 0, "")` — it succeeds without any staleness revert, minting the incorrect rsETH amount.
6. Assert that no `StalePrice` revert occurred, confirming the missing guard at `ChainlinkPriceOracle.sol` line 52.
