### Title
Single Attested Participant Can Inject Arbitrary CKD Output for `AppPublicKey` Requests — (`File: crates/contract/src/lib.rs`)

### Summary

In `respond_ckd`, when a pending CKD request uses the `CKDAppPublicKey::AppPublicKey` variant, the contract performs **zero cryptographic verification** of the submitted `CKDResponse`. Any single attested participant can call `respond_ckd` with arbitrary `big_y` and `big_c` values and the contract will accept them, resolving the user's yield with a fraudulent confidential key derivation output — without requiring threshold consensus on the actual computation.

### Finding Description

The `respond_ckd` function in `crates/contract/src/lib.rs` contains a conditional output check:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO VERIFICATION
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For the `AppPublicKeyPV` variant, the contract enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)` via `ckd_output_check`, which cryptographically binds the response to the MPC master secret key. [2](#0-1) 

For the `AppPublicKey` variant (the standard, non-PV path), the contract performs **no verification whatsoever** — any `CKDResponse` with arbitrary `big_y` and `big_c` is accepted and forwarded to the user.

Contrast this with `respond` for ECDSA/EdDSA signatures, which always verifies the submitted signature against the derived public key before resolving the yield: [3](#0-2) 

The authorization check in `respond_ckd` only requires the caller to be a single attested participant: [4](#0-3) 

There is no threshold-level consensus enforced on-chain for the CKD `AppPublicKey` response. The off-chain MPC protocol is supposed to produce the correct output, but the contract does not verify it.

### Impact Explanation

A single compromised attested participant can:

1. Watch the chain for a pending `request_app_private_key` call using `CKDAppPublicKey::AppPublicKey`.
2. Call `respond_ckd` with a fabricated `CKDResponse` — e.g., `big_y = g1 * r` and `big_c = target + app_pk * r` for any attacker-chosen `target` and random `r`.
3. The contract resolves the user's yield with this fraudulent output.
4. The user decrypts `big_c - big_y * app_sk = target`, receiving an attacker-controlled value instead of the real MPC-derived secret `H(pk || app_id) * msk`.

This is a **Critical** impact: unauthorized confidential key derivation output without the required threshold participant authorization. The user's derived "confidential key" is not bound to the MPC network's master secret at all — it is entirely under the attacker's control. Any application secret derived via this path (e.g., application private keys, encryption keys) is compromised.

### Likelihood Explanation

**Medium.** The attack requires only a single compromised attested participant — strictly below the signing threshold. The `AppPublicKey` variant is the standard (non-PV) CKD path used by default. The attacker simply needs to observe a pending CKD request on-chain and race to call `respond_ckd` before the honest nodes do. Since NEAR transactions are public and the attacker is an authorized participant, this is straightforward.

### Recommendation

Apply the same cryptographic binding to the `AppPublicKey` path that the `AppPublicKeyPV` path already has. For the non-PV variant, the contract can verify that `big_c - big_y * app_pk = H(pk || app_id) * msk` using the known MPC public key. Specifically, verify the equation `e(big_c - big_y * app_pk, g2) = e(H(pk || app_id), mpc_pk)` on-chain, or require the response to include a proof of correct computation. Alternatively, require the `AppPublicKey` path to also use the PV variant so the existing `ckd_output_check` pairing equation can be applied.

### Proof of Concept

1. User calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(app_pk)` and `derivation_path = "secret"`. A `CKDRequest` is stored in `pending_ckd_requests`.

2. Malicious attested participant observes the pending request on-chain.

3. Malicious participant calls `respond_ckd(ckd_request, CKDResponse { big_y: g1, big_c: g1 })` — completely fabricated values.

4. The contract executes:
   - `assert_caller_is_signer()` → passes (participant is a valid signer)
   - `assert_caller_is_attested_participant_and_protocol_active()` → passes
   - `match &request.app_public_key { AppPublicKey(_) => {} ... }` → **no check performed**
   - `pending_requests::resolve_yields_for(...)` → resolves the user's yield with `{ big_y: g1, big_c: g1 }`

5. The user receives `CKDResponse { big_y: g1, big_c: g1 }` and computes `big_c - big_y * app_sk = g1 - g1 * app_sk`, which is an attacker-controlled value with no relation to the MPC master secret. [5](#0-4) [6](#0-5)

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
