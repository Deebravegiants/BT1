### Title
CHIP-22 Harvester Fee Quality Threshold Not Enforced — Unconditional Farmer Reward Diversion by Any Connected Harvester - (File: `chia/farmer/farmer_api.py`)

### Summary
The CHIP-22 fee convention allows a third-party harvester to redirect the farmer's block reward (0.25 XCH) to its own address only when a probabilistic `fee_quality` value falls at or below a declared `applied_fee_threshold`. The farmer's code checks this threshold in `notify_farmer_reward_taken_by_harvester_as_fee()` but **only logs a warning on violation** — it never rejects the override. The actual reward address substitution in `_process_respond_signatures()` is unconditional. Any connected harvester can therefore redirect 100% of the farmer's block rewards to an arbitrary address on every won block, regardless of whether the fee quality threshold is met.

### Finding Description
The CHIP-22 fee convention is implemented across two functions in `chia/farmer/farmer_api.py`:

**Step 1 — Threshold check (logging only):**
When a harvester sends `NewProofOfSpace` with `farmer_reward_address_override` set, `new_proof_of_space()` calls `notify_farmer_reward_taken_by_harvester_as_fee()`. [1](#0-0) 

Inside that function, the farmer computes `fee_quality` and compares it to `applied_fee_threshold`. If the threshold is violated, it emits `log.warning(...)` and returns — **no rejection, no abort, no disconnect**. [2](#0-1) 

**Step 2 — Unconditional override acceptance:**
Later, when the harvester's `RespondSignatures` arrives, `_process_respond_signatures()` blindly substitutes the farmer reward address with whatever the harvester supplied, with no reference to whether the fee quality check passed:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
``` [3](#0-2) 

The resulting `DeclareProofOfSpace` carries the harvester-controlled address as `farmer_reward_address`, which the full node uses to mint the 0.25 XCH farmer reward coin. [4](#0-3) 

The `ProofOfSpaceFeeInfo` struct that carries `applied_fee_threshold` is an optional field in `NewProofOfSpace` — a harvester can also omit it entirely, which also only triggers a warning. [5](#0-4) 

### Impact Explanation
A malicious third-party harvester can redirect the farmer's entire block reward (0.25 XCH per won block) to an arbitrary puzzle hash on every block it wins, not just on the probabilistic fraction permitted by the agreed fee rate. This is unauthorized payout redirection of XCH. The farmer has no on-node mechanism to detect or prevent this beyond reading log warnings.

### Likelihood Explanation
CHIP-22 was designed specifically for the third-party harvester use case, meaning farmers who deploy it are explicitly trusting external harvesters. Any such harvester — including one that initially behaves honestly — can silently switch to always setting `farmer_reward_address_override` to its own address. The farmer's node will comply without any enforcement action.

### Recommendation
In `_process_respond_signatures()`, before substituting `farmer_reward_address`, re-derive `fee_quality` from the proof bytes and the signage point challenge hash and verify that `fee_quality <= applied_fee_threshold`. If the check fails, drop the override (use `self.farmer.farmer_target`) or abort block submission for that proof. The check in `notify_farmer_reward_taken_by_harvester_as_fee()` should be the enforcement gate, not merely a logging call.

### Proof of Concept
1. Operator connects their farmer to a third-party harvester service.
2. Harvester wins a block (valid proof of space, `required_iters < sp_interval_iters`).
3. Harvester sends `NewProofOfSpace` with `farmer_reward_address_override = <harvester_wallet>` and `fee_info.applied_fee_threshold = 0xFFFFFFFF` (always passes the log check) — or omits `fee_info` entirely.
4. Farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()`, which logs "Fee threshold passed" (or a warning if omitted), then continues.
5. Farmer sends `RequestSignatures`; harvester responds with `RespondSignatures` also carrying `farmer_reward_address_override = <harvester_wallet>`.
6. `_process_respond_signatures()` sets `farmer_reward_address = <harvester_wallet>` unconditionally.
7. Farmer broadcasts `DeclareProofOfSpace` with the harvester's address; the full node mints the 0.25 XCH farmer reward to the harvester.
8. Farmer receives 0 XCH for the block they won.
