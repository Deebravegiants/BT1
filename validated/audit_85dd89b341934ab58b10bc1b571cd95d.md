Audit Report

## Title
Synthetic Ratio Precision Loss Silently Produces Zero Mid Price, Permanently Blocking Pool Swaps — (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

## Summary

In `AnchoredPriceProvider._getBidAndAskPrice()`, the synthetic ratio computation `Math.mulDiv(mid, ORACLE_DECIMALS, mid2)` at line 267 floors to zero whenever `mid * 1e8 < mid2`. No zero-check follows the division, so `_computeBidAsk(0, spreadBps)` is called, which returns the stall sentinel `(0, type(uint128).max)`. `getBidAndAskPrice()` then reverts with `FeedStalled`, and the pool's `swap()` re-throws as `PriceProviderFailed`, completely blocking all swaps for as long as the price ratio remains below `1e-8`.

## Finding Description

In `_getBidAndAskPrice()`, after both legs pass their individual validity checks, the synthetic ratio is computed:

```solidity
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);  // line 267
spreadBps += spreadBps2;
``` [1](#0-0) 

`ORACLE_DECIMALS = 1e8`. When `mid * 1e8 < mid2`, integer division floors the result to zero. There is no guard for `mid == 0` after this line — execution falls through to `_computeBidAsk(0, spreadBps)` at line 271. [2](#0-1) 

Inside `_computeBidAsk`, `_bandEdge(0, ...)` evaluates to `Math.mulDiv(0, Q64 * edgeFactor, STEP_DENOM, rounding) = 0`, so `refBid == 0` immediately triggers the stall sentinel return:

```solidity
uint256 refBid = _bandEdge(mid, BPS_BASE_U - half, Math.Rounding.Floor);
uint256 refAsk = _bandEdge(mid, BPS_BASE_U + half, Math.Rounding.Ceil);
if (refBid == 0 || refAsk > type(uint128).max || refBid >= refAsk) {
    return (0, type(uint128).max);
}
``` [3](#0-2) 

`getBidAndAskPrice()` then reverts:

```solidity
if (bid == 0 || ask == type(uint128).max) revert FeedStalled();
``` [4](#0-3) 

The pool's `_getBidAndAskPriceX64()` catches this and re-throws as `PriceProviderFailed`: [5](#0-4) 

`swap()` calls `_getBidAndAskPriceX64()` as its very first action (line 228), so every swap reverts: [6](#0-5) 

**Existing guards are insufficient:**
- `_readLeg` checks `mid == 0` per individual feed (line 287), but both individual feed prices are non-zero in the failing scenario. [7](#0-6) 
- The `mid2 == 0` check at line 265 guards against a zero denominator, not a zero ratio result. [8](#0-7) 
- The per-feed `priceGuard` (lines 290–292) operates on individual feed prices, not on the computed ratio, so it cannot prevent ratio underflow. [9](#0-8) 

## Impact Explanation

All swaps on any pool using `AnchoredPriceProvider` in synthetic ratio mode are completely blocked whenever the base/quote price ratio falls below `1e-8`. This is broken core pool functionality: the primary user-facing action (`swap`) becomes permanently unusable until the price ratio recovers. Withdrawals are unaffected (the stall sentinel is designed to fail closed for swaps only), but swap flows are entirely unusable, satisfying the "Broken core pool functionality causing unusable swap flows" impact gate.

## Likelihood Explanation

The condition `mid * 1e8 < mid2` is reachable without any privileged action in two realistic scenarios:

1. **Asymmetric pair deployment**: e.g., base = SHIB at $0.00001 → `mid = 1_000` (8-decimal), quote = BTC at $60,000 → `mid2 = 6_000_000_000_000`. `Math.mulDiv(1000, 1e8, 6e12) = 0`. Any pool deployer (public pool creator) can create such a pool.
2. **Token price crash**: A base token that was previously priced normally crashes by more than 8 orders of magnitude relative to the quote token. The oracle correctly reflects the market price, and the ratio underflows.

The synthetic ratio mode is an explicitly supported and documented feature (e.g., BTC/USD ÷ ETH/USD = BTC/ETH). No privileged action is required to trigger the condition — it arises from normal market price movement reflected by the oracle.

## Recommendation

Add a zero-check for the synthetic ratio result immediately after the division:

```solidity
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
if (mid == 0) return (0, type(uint128).max); // ratio underflows 8-decimal precision
```

This fails closed (same behavior as a stale feed) rather than silently producing a zero price that propagates through `_computeBidAsk`. Alternatively, increase intermediate precision by scaling `mid` before dividing (e.g., using `1e18` instead of `1e8`) and adjusting `_bandEdge`/`STEP_DENOM` accordingly to preserve precision for highly asymmetric pairs.

```solidity

pragma solidity ^0.8.28;
import {Math} from "@openzeppelin/contracts/utils/math/Math.sol";

contract PoC {
    uint256 constant ORACLE_DECIMALS = 1e8;

    function demonstrateZeroRatio() external pure returns (uint256 syntheticMid) {
        uint256 mid  = 1_000;               // SHIB at $0.00001 → 8-decimal = 1000
        uint256 mid2 = 6_000_000_000_000;   // BTC at $60,000  → 8-decimal = 6e12
        // Replicates AnchoredPriceProvider._getBidAndAskPrice() line 267
        syntheticMid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
        // Returns 0: 1e11 / 6e12 = 0
        // → _computeBidAsk(0, spreadBps) → (0, type(uint128).max)
        // → getBidAndAskPrice() reverts FeedStalled
        // → MetricOmmPool.swap() reverts PriceProviderFailed
    }
}
```

Deploy against a fork with a pool configured in synthetic ratio mode with an asymmetric pair. Call `swap()` — it reverts with `PriceProviderFailed` wrapping `FeedStalled` for every caller, confirming complete swap DoS.
