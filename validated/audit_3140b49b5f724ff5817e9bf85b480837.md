Audit Report

## Title
Unbounded Per-User Withdrawal Requests Allow Attacker to Exhaust `assetsCommitted`, Temporarily Freezing Withdrawal Initiation - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager.initiateWithdrawal` places no cap on the number of active withdrawal requests a single user may hold. An attacker holding sufficient rsETH can flood the queue with minimum-amount requests, driving `assetsCommitted[asset]` up to `totalAssets`. Once that threshold is reached, `getAvailableAssetAmount` returns zero and every subsequent `initiateWithdrawal` call from any user reverts with `ExceedAmountToWithdraw`, blocking withdrawal initiation for the full `withdrawalDelayBlocks` window (~8 days). The attacker recovers their capital after the delay and can repeat the cycle indefinitely.

## Finding Description
`initiateWithdrawal` (L150–178) enforces only a minimum-amount check and an availability check before committing assets:

```solidity
// contracts/LRTWithdrawalManager.sol L162-173
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
...
if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();
assetsCommitted[asset] += expectedAssetAmount;
```

`getAvailableAssetAmount` (L599–603) computes availability as:

```solidity
availableAssetAmount = totalAssets > assetsCommitted[asset]
    ? totalAssets - assetsCommitted[asset]
    : 0;
```

There is no per-user request count limit anywhere in the contract. `_addUserWithdrawalRequest` (L744–759) simply appends to `userAssociatedNonces[asset][msg.sender]` without bound. An attacker calls `initiateWithdrawal` in a loop with `minRsEthAmountToWithdraw[asset]` each time; each call increments `assetsCommitted[asset]` by the corresponding `expectedAssetAmount`. When `assetsCommitted[asset] >= totalAssets`, `getAvailableAssetAmount` returns 0 and all subsequent calls from any address revert.

The FIFO unlock loop in `_unlockWithdrawalRequests` (L790–815) processes requests in nonce order, so the attacker's N requests must all be processed before any later legitimate request can be unlocked. The `withdrawalDelayBlocks` default of `8 days / 12 seconds` (L94) means the attacker's requests cannot be unlocked for ~8 days, during which the committed assets remain unavailable to other users.

No existing guard prevents this: `nonReentrant` stops re-entrancy within a single call but not repeated external calls; `whenNotPaused` requires privileged action to invoke; and the `ExceedAmountToWithdraw` check itself is the mechanism the attacker exploits.

## Impact Explanation
**Medium — Temporary freezing of funds.** Legitimate rsETH holders cannot initiate the withdrawal lifecycle for the affected asset for up to 8 days per attack cycle. Their underlying assets (ETH/LSTs) held in the protocol are inaccessible during this window. The attacker's capital is returned intact, enabling indefinite repetition and a sustained denial of withdrawal initiation.

## Likelihood Explanation
**Medium.** The attacker must hold rsETH whose underlying value equals the total available asset balance in the `LRTUnstakingVault`. The capital is fully recovered after the delay, so the net cost is only opportunity cost. On L2 deployments (Arbitrum, Optimism, Unichain) where gas per call is negligible, creating thousands of minimum-amount requests is cheap. A large rsETH holder or competing protocol can sustain this attack at low ongoing cost.

## Recommendation
Add a per-user cap on active (not yet unlocked) withdrawal requests per asset:

```solidity
uint256 public maxWithdrawalRequestsPerUser; // e.g., 10–50

function initiateWithdrawal(address asset, uint256 rsETHUnstaked, string calldata referralId) external ... {
    require(
        userAssociatedNonces[asset][msg.sender].length() < maxWithdrawalRequestsPerUser,
        "TooManyActiveRequests"
    );
    ...
}
```

Alternatively, raise `minRsEthAmountToWithdraw[asset]` to a value that makes flooding economically infeasible given the total available asset balance. Both mitigations can be combined.

## Proof of Concept
1. Determine `X = getAvailableAssetAmount(ETH_TOKEN)` and `minAmt = minRsEthAmountToWithdraw[ETH_TOKEN]`.
2. Attacker acquires rsETH such that `getExpectedAssetAmount(ETH_TOKEN, minAmt) * N ≈ X`.
3. Attacker calls `initiateWithdrawal(ETH_TOKEN, minAmt, "")` N times in a loop. Each call transfers `minAmt` rsETH and increments `assetsCommitted[ETH_TOKEN]`.
4. After N calls, `assetsCommitted[ETH_TOKEN] == X`; `getAvailableAssetAmount(ETH_TOKEN)` returns 0.
5. Any honest user calling `initiateWithdrawal` now reverts with `ExceedAmountToWithdraw`.
6. Operator calls `unlockQueue`; the FIFO loop in `_unlockWithdrawalRequests` processes the attacker's N requests first (L790–815), consuming all unlocked assets before reaching any legitimate request.
7. After `withdrawalDelayBlocks` (~8 days), attacker calls `completeWithdrawal` N times and recovers all assets.
8. Attacker repeats from step 1.
