Looking at the actual code to verify each step of the claimed exploit path.

Audit Report

## Title
Stale `rsETHPrice` Cache Allows Depositors to Extract Accrued Yield Before Price Update - (File: contracts/LRTOracle.sol)

## Summary
`LRTOracle.rsETHPrice` is a cached state variable updated only when `updateRSETHPrice()` is explicitly called. Because `updateRSETHPrice()` is `public whenNotPaused`, any unprivileged caller can deposit at the stale (lower) price, trigger the price update themselves, then initiate a withdrawal at the newly updated (higher) price — extracting yield that belongs to existing rsETH holders. The `expectedAssetAmount` is locked at initiation time and paid out in full at completion, so the withdrawal delay does not neutralise the profit.

## Finding Description
`LRTOracle.rsETHPrice` is a stored state variable written only at line 313 of `contracts/LRTOracle.sol`:

```solidity
rsETHPrice = newRsETHPrice;   // LRTOracle.sol L313
```

`updateRSETHPrice()` (L87–89) is `public whenNotPaused`, callable by any address.

**Deposit path** (`contracts/LRTDepositPool.sol` L520):
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```
`getAssetPrice` reads a live Chainlink value; `rsETHPrice()` returns the stale cached value. While EigenLayer rewards accrue between `updateRSETHPrice()` calls, the true rsETH value exceeds `rsETHPrice`, so the denominator is artificially low and depositors receive more rsETH than fair value.

**Withdrawal initiation path** (`contracts/LRTWithdrawalManager.sol` L593):
```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```
`initiateWithdrawal` (L168–175) computes `expectedAssetAmount` using the live `rsETHPrice` at call time and stores it immutably in `withdrawalRequests[requestId].expectedAssetAmount` (L751–753). `_processWithdrawalCompletion` (L734) pays out exactly `request.expectedAssetAmount`.

**`_calculatePayoutAmount` does not prevent the attack.** At `unlockQueue` time (L833–834):
```solidity
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
```
This takes the minimum of the stored amount and the current recalculation. Because the attacker initiates withdrawal *after* calling `updateRSETHPrice()`, the stored `expectedAssetAmount` already reflects the updated (higher) price. If the price stays flat or rises further, the payout equals the stored amount. The min-check only reduces payout if the price *falls* after initiation — it does not cap the gain from depositing at a stale price.

**`pricePercentageLimit` is insufficient.** The guard at L256–266 only reverts if the price jump in a single `updateRSETHPrice()` call exceeds the configured threshold. It does not prevent the attack when: (a) `pricePercentageLimit == 0` (check is skipped entirely per L256), or (b) the accrued staleness is within the configured limit. The attacker simply times the attack to a staleness window that fits within the limit.

**Exploit sequence:**
1. Wait for `rsETHPrice` to become stale (rewards accrue continuously).
2. Call `depositAsset(stETH, 100e18, ...)` — minted rsETH uses the stale low denominator.
3. Call `updateRSETHPrice()` — `rsETHPrice` rises to reflect accrued rewards.
4. Call `initiateWithdrawal(stETH, 100e18, ...)` — `expectedAssetAmount` is computed at the now-higher `rsETHPrice` and stored.
5. After `withdrawalDelayBlocks` and operator `unlockQueue`, call `completeWithdrawal` — receive the inflated asset amount.

## Impact Explanation
**High — Theft of unclaimed yield.** The attacker's excess rsETH (minted against the stale price) dilutes all existing holders' proportional claim on protocol TVL. The profit is extracted directly from yield that should have accrued to existing depositors. The attack is repeatable every reward cycle and requires no special permissions or capital beyond the deposit itself.

## Likelihood Explanation
EigenLayer rewards accrue continuously, so a staleness window exists between every pair of `updateRSETHPrice()` calls. No privileged access is required: `updateRSETHPrice()` is public, deposits are open, and the attacker controls the timing of both the deposit and the price update. The `pricePercentageLimit` guard limits per-call profit but does not eliminate the attack — it only caps the price jump per invocation, and is entirely bypassed when set to zero.

## Recommendation
Call `_updateRsETHPrice()` (or an equivalent internal price refresh) atomically at the start of `depositAsset()` and `depositETH()` in `LRTDepositPool`, before computing `rsethAmountToMint`. This ensures every deposit uses the current, reward-inclusive price and eliminates the staleness window. Alternatively, compute the mint amount using a freshly derived price rather than the cached `rsETHPrice` state variable.

## Proof of Concept

**Setup (large protocol to minimise dilution noise):**
- rsETH supply: 1,000,000 rsETH
- `rsETHPrice` (stale): 1.000 ETH
- True rsETH value (accrued rewards): 1.010 ETH
- stETH/ETH Chainlink price: 1.000

**Step 1 — Attacker deposits 100 stETH at stale price:**
```
rsETH_minted = (100e18 × 1.000e18) / 1.000e18 = 100 rsETH
Fair amount  = (100e18 × 1.000e18) / 1.010e18 ≈  99.01 rsETH
Excess rsETH = ~0.99 rsETH
```

**Step 2 — Attacker calls `updateRSETHPrice()`:**
- `totalETHInProtocol` ≈ 1,010,100 ETH (original 1,010,000 + 100 deposited)
- `rsethSupply` = 1,000,100
- `newRsETHPrice` ≈ 1,010,100 / 1,000,100 ≈ 1.00999 ETH (≈ 1.010 ETH for large TVL)
- `rsETHPrice` updated to ≈ 1.010 ETH

**Step 3 — Attacker calls `initiateWithdrawal(stETH, 100e18)`:**
```
expectedAssetAmount = 100e18 × 1.010e18 / 1.000e18 = 101 stETH  (stored)
```

**Step 4 — After `withdrawalDelayBlocks`, operator calls `unlockQueue`, attacker calls `completeWithdrawal`:**
- `_calculatePayoutAmount`: min(101, 101) = 101 stETH
- Attacker receives **101 stETH**
- Net profit: **~1 stETH** extracted from existing holders' accrued yield
