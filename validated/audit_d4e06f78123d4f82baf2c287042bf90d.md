Audit Report

## Title
Permissionless `updateRSETHPrice` Combined with Permissionless `verifyCheckpointProofs` Allows Theft of Beacon-Chain Yield — (`contracts/LRTOracle.sol`, `contracts/NodeDelegator.sol`, `contracts/LRTDepositPool.sol`)

## Summary

After an operator calls `NodeDelegator.startCheckpoint()`, an attacker can deposit ETH at the stale (pre-yield) rsETH price, then permissionlessly finalize the checkpoint via `IEigenPod.verifyCheckpointProofs()` and trigger `LRTOracle.updateRSETHPrice()`. Because the attacker's rsETH was minted at the old price before the beacon-chain yield `Y` was credited, the attacker captures a portion of `Y` that should accrue exclusively to pre-existing rsETH holders.

## Finding Description

**Root cause:** `LRTOracle.updateRSETHPrice()` has no access control beyond `whenNotPaused`, and EigenLayer's `verifyCheckpointProofs()` is explicitly permissionless. Together they allow an attacker to control the exact moment yield is reflected in the rsETH price, after having deposited at the stale price.

**Step 1 — Operator starts checkpoint.**
`NodeDelegator.startCheckpoint()` is restricted to `onlyLRTOperator` and calls `eigenPod.startCheckpoint()`. [1](#0-0) 
At this point, beacon-chain yield `Y` is not yet reflected in EigenLayer shares; `recordBeaconChainETHBalanceUpdate` is only called upon checkpoint finalization.

**Step 2 — Attacker deposits at stale price.**
`depositETH()` is open to any caller. The rsETH minted is computed using the last stored `rsETHPrice`, which does not yet include `Y`: [2](#0-1) 
The attacker receives `D / P_old` rsETH for `D` ETH deposited.

**Step 3 — Attacker finalizes the checkpoint (permissionless).**
The EigenLayer interface comment explicitly states "Anyone can call this method to submit proofs towards the current checkpoint": [3](#0-2) 
Finalization credits `Y` ETH worth of new shares to the NDC via `recordBeaconChainETHBalanceUpdate`.

**Step 4 — Attacker triggers price update (permissionless).**
`updateRSETHPrice()` has no role check: [4](#0-3) 
`_getTotalEthInProtocol()` calls `getETHDistributionData()`, which calls `getEffectivePodShares()` on each NDC: [5](#0-4) 
`getEffectivePodShares()` reads `DelegationManager.getWithdrawableShares()` via `NodeDelegatorHelper.getWithdrawableShare()`: [6](#0-5) [7](#0-6) 
This now reflects the newly credited `Y` ETH. The new price becomes `(N + D + Y - fee) / (S + D/P_old)`, which exceeds `P_old`, giving the attacker instant profit.

**Why `pricePercentageLimit` is insufficient:**
The guard is skipped entirely when `pricePercentageLimit == 0`: [8](#0-7) 
Even when non-zero, routine staking rewards (a few days of yield) produce a price increase well within any reasonable percentage limit, so the attack succeeds without triggering the revert.

## Impact Explanation

**High — Theft of unclaimed yield.** Pre-existing rsETH holders earned yield `Y` through their stake. The attacker, by depositing at the stale price and then triggering finalization and price update, dilutes the yield distribution: a portion of `Y` (proportional to the attacker's share of the new total supply) accrues to the attacker's rsETH rather than to pre-existing holders. The attacker profits risk-free and can immediately redeem at the new higher price.

## Likelihood Explanation

**Medium.** The operator must call `startCheckpoint()` first, which is a routine operational step. The window between `startCheckpoint()` and proof submission can be substantial (the operator may not immediately submit proofs). Both `verifyCheckpointProofs` and `updateRSETHPrice` require no special access. The attacker only needs to monitor for `CheckpointCreated` events on-chain. The attack is repeatable every checkpoint cycle.

## Recommendation

1. **Restrict `updateRSETHPrice` to operators/managers**, or add a cooldown after a checkpoint is started before the price can be updated by unprivileged callers.
2. **Alternatively**, snapshot the rsETH total supply at `startCheckpoint()` time and use that supply for yield distribution, so depositors after the snapshot do not benefit from yield earned before their entry.
3. **Ensure `pricePercentageLimit` is always set to a non-zero value** as defense-in-depth, and enforce this in the initializer or a setter invariant.
4. Consider emitting a `CheckpointStarted` event from `NodeDelegator.startCheckpoint()` and pausing deposits until the checkpoint is finalized.

## Proof of Concept

```solidity
// Fork test (Holesky or mainnet fork)
// Preconditions:
//   - NDC has validators with accumulated yield Y = 1 ETH in pod balance
//   - rsETH supply S = 1000 ether, rsETHPrice P_old = 1.05 ether, TVL N = 1050 ether
//   - pricePercentageLimit = 0 (or Y/N < limit)

// 1. Operator starts checkpoint (routine operation)
vm.prank(operator);
nodeDelegator.startCheckpoint(false);
// rsETHPrice is still 1.05 ether; Y not yet reflected in EigenLayer shares

// 2. Attacker deposits 10 ETH at stale price
vm.prank(attacker);
lrtDepositPool.depositETH{value: 10 ether}(0, "");
// Attacker receives 10e18 / 1.05e18 ≈ 9.523 rsETH at old price

// 3. Attacker finalizes checkpoint (permissionless — no access control on verifyCheckpointProofs)
eigenPod.verifyCheckpointProofs(balanceContainerProof, proofs);
// EigenLayer now credits Y = 1 ETH of new shares to NDC via recordBeaconChainETHBalanceUpdate
// getWithdrawableShares() now returns the increased amount

// 4. Attacker triggers price update (permissionless)
lrtOracle.updateRSETHPrice();
// newTVL = 1050 + 10 + 1 = 1061 ETH (minus protocol fee on Y)
// newSupply = 1000 + 9.523 = 1009.523 rsETH
// newPrice ≈ 1061 / 1009.523 ≈ 1.051 ether

// 5. Assert attacker profit
uint256 attackerRsETH = rsETH.balanceOf(attacker); // ≈ 9.523 rsETH
uint256 attackerValue = attackerRsETH * lrtOracle.rsETHPrice() / 1e18; // ≈ 10.009 ETH
assertGt(attackerValue, 10 ether); // Attacker gained ~0.009 ETH of yield
// Pre-existing holders lost a proportional share of the 1 ETH yield
```

The attacker's gain scales with `Y * (attackerShare / totalSupply)`. For large yield accumulations (e.g., after a long period without checkpointing), the stolen amount is material.
