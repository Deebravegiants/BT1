### Title
Mocked-Up Peras Certificate Validation Always Returns `True`, Allowing Any Block Producer to Inject Arbitrary Peras Certificates — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/BlockBody/Internal.hs`)

### Summary
The Dijkstra era's BBODY transition rule calls `validatePerasCert` to authenticate Peras certificates embedded in block bodies. However, `validatePerasCert` is an explicitly documented placeholder that unconditionally returns `True` regardless of the certificate, nonce, or key. Any block producer can embed an arbitrary, cryptographically invalid `PerasCert` in a block body and every honest node will accept it as valid, bypassing the intended Peras consensus-certificate check entirely.

### Finding Description

`validatePerasCert` is defined in the production module `Cardano.Ledger.Dijkstra.BlockBody.Internal` as a stub:

```haskell
-- | Mocked-up Peras certificate validation routine
--
-- NOTE: this function will be replaced with the real implementation from
-- 'cardano-base' once it's ready.
validatePerasCert :: Nonce -> PerasKey -> PerasCert -> Bool
validatePerasCert _ _ _ = True
``` [1](#0-0) 

The companion `PerasKey` type is equally a placeholder — a unit-like constructor carrying no actual key material:

```haskell
data PerasKey = PerasKey
  deriving (Eq, Show, Generic, NoThunks)
``` [2](#0-1) 

Both are exported from the public-facing `BlockBody` module and consumed directly by the production BBODY STS rule in `dijkstraBbodyTransition`:

```haskell
case blockBody ^. perasCertBlockBodyL of
  SNothing -> pure ()
  SJust cert ->
    let nonce = block ^. prevNonceBlockHeaderL
     in validatePerasCert nonce PerasKey cert
          ?! injectFailure (PerasCertValidationFailed cert nonce)
``` [3](#0-2) 

Because `validatePerasCert _ _ _ = True`, the predicate `?!` never fires. The `PerasCertValidationFailed` failure constructor exists and is fully serializable, but it is structurally unreachable at runtime. [4](#0-3) 

This is the only place in the BBODY rule where Peras certificate authenticity is checked; there is no secondary validation path. [5](#0-4) 

### Impact Explanation

**Impact: Critical — Honest nodes accept an invalid block causing permanent ledger divergence requiring a hard fork.**

When the Dijkstra era is activated on-chain, any block producer (no special privilege required) can include a `PerasCert` field containing arbitrary bytes. Every node running this code will accept the block unconditionally because `validatePerasCert` always returns `True`. If a future node version ships the real cryptographic validation from `cardano-base`, it will reject the same blocks that the current code accepted, producing a deterministic split between old and new nodes — a ledger fork that cannot be resolved without a hard fork.

Additionally, the Peras protocol is a finality layer; accepting forged Peras certificates could allow an attacker to manipulate which chain segment is considered finalized, with downstream consequences for settlement guarantees.

### Likelihood Explanation

**Likelihood: High.**

The entry path requires only the ability to produce a block in the Dijkstra era — a normal, unprivileged block-producer role. No key compromise, governance majority, or Sybil attack is needed. The stub is in a production implementation file (not a test module), is exported from the public API, and is wired directly into the mandatory BBODY transition rule. The only precondition is era activation.

### Recommendation

Replace `validatePerasCert` and `PerasKey` with the real cryptographic types and verification logic from `cardano-base` before the Dijkstra era is activated on any network. Until the real implementation is available, the BBODY rule should either:

1. **Reject blocks that include a `PerasCert`** (treat `SJust cert` as an unconditional predicate failure), or
2. **Remove the `perasCertBlockBodyL` field from the block body** until the real validation is ready.

This mirrors the recommendation in the external report: do not ship a placeholder component in a production validation path; use the real implementation or gate the feature entirely.

### Proof of Concept

1. Construct a Dijkstra-era block whose `DijkstraBlockBodyRaw` has `dbbrPerasCert = SJust (PerasCert <arbitrary_bytes>)`.
2. Submit the block to any node running this code.
3. The node reaches `dijkstraBbodyTransition`, evaluates `validatePerasCert nonce PerasKey cert`, receives `True`, and accepts the block without error.
4. The `PerasCertValidationFailed` predicate failure is never raised regardless of the certificate content. [6](#0-5) [7](#0-6) [8](#0-7)
