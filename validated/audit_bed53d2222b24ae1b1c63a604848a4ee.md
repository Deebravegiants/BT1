Based on my thorough code review, this is a **real vulnerability**. Here is the full analysis:

### Title
Unvalidated Identity `AppPublicKey` in CKD Request Nullifies Encryption, Exposing Unblinded Confidential Key — (`crates/contract/src/lib.rs`, `crates/node/src/providers/ckd/sign.rs`)

### Summary

An unprivileged caller can submit a CKD request with `AppPublicKey` set to the BLS12-381 G1 identity point. The contract performs no validation on the `AppPublicKey` variant, and the MPC node's `ElementG1::try_from` accepts the identity. The `ckd()` protocol then computes `C = msk * H(pk, app_id) + y * identity = msk * H(pk, app_id)`, returning the **unblinded** confidential key directly to the attacker, bypassing the encryption mechanism entirely.

### Finding Description

**Step 1 — Contract entry, no guard for `AppPublicKey`:**

In `request_app_private_key`, the contract validates `AppPublicKeyPV` via a pairing check but performs **no validation** on the `AppPublicKey` (single G1 point) variant: [1](#0-0) 

The identity point has a valid BLS12-381 compressed encoding and passes deserialization silently.

**Step 2 — `respond_ckd` also skips validation for `AppPublicKey`:** [2](#0-1) 

The `ckd_output_check` (pairing verification) is only applied to `AppPublicKeyPV`. For `AppPublicKey`, the raw `(big_y, big_c)` is accepted and returned to the caller with no integrity check.

**Step 3 — Node-side: `ElementG1::try_from` accepts the identity:**

In `CKDComputation::compute`, the `AppPublicKey` bytes are converted via: [3](#0-2) 

The underlying conversion uses `blstrs::G1Projective::from_compressed`, which accepts the identity point (point at infinity) as a valid encoding: [4](#0-3) 

There is no `is_identity()` check here.

**Step 4 — Protocol computes unblinded output:**

In `compute_signature_share`, each participant computes:

```
C_i = x_i * H(pk, app_id) + y_i * app_pk
```

With `app_pk = identity`:

```
C_i = x_i * H(pk, app_id) + y_i * 0 = x_i * H(pk, app_id)
``` [5](#0-4) 

After Lagrange interpolation and aggregation across all participants, the coordinator returns:

```
C = msk * H(pk, app_id)   (the raw confidential key, unblinded)
Y = y_agg * G1            (a random point, useless to the attacker)
```

The blinding term `y * app_pk` that normally hides `msk * H(pk, app_id)` from anyone without `app_sk` is completely eliminated.

### Impact Explanation

The CKD protocol's security guarantee is that only the holder of `app_sk` (where `app_pk = app_sk * G1`) can recover the confidential key by computing `C - app_sk * Y`. With `app_pk = identity`, the output `C` **is** the confidential key directly. The attacker receives `msk * H(pk, app_id)` for any `app_id` they choose (by varying `derivation_path`), without possessing any app secret key. This is unauthorized confidential key derivation output — the encryption mechanism is fully bypassed.

### Likelihood Explanation

The attack requires only a standard contract call to `request_app_private_key` with a well-formed identity-point encoding. No threshold collusion, TEE compromise, or privileged access is needed. The identity point's compressed encoding is a fixed, publicly known 48-byte value.

### Recommendation

In `request_app_private_key`, add an explicit rejection of the identity point for the `AppPublicKey` variant, mirroring the existing guard for `AppPublicKeyPV`:

```rust
dtos::CKDAppPublicKey::AppPublicKey(pk) => {
    // Reject identity and off-curve points
    let point = blstrs::G1Projective::try_from(pk)
        .expect("valid encoding");
    if bool::from(point.is_identity()) {
        env::panic_str("app public key must not be the identity");
    }
}
```

Additionally, add a subgroup membership check (torsion-free check) to prevent low-order point attacks. The `AppPublicKeyPV` path already enforces this via the pairing check; the `AppPublicKey` path must be brought to parity.

### Proof of Concept

```rust
// Attacker submits a CKD request with the G1 identity as app_pk
let identity_bytes = blstrs::G1Projective::identity().to_compressed();
let request = CKDRequestArgs {
    derivation_path: "victim_app_id".to_string(),
    app_public_key: CKDAppPublicKey::AppPublicKey(
        Bls12381G1PublicKey(identity_bytes)
    ),
    domain_id: bls_domain_id,
};
// Contract accepts it; MPC nodes compute C = msk * H(pk, app_id)
// Attacker receives (Y, C) where C is the raw confidential key
// Verify: C == msk * hash_app_id_with_pk(&mpc_pk, app_id)
```

The existing test `app_public_key_check__should_accept_identity_key_pair` in `crates/contract/src/primitives/ckd.rs` (lines 479–495) explicitly documents that identity key pairs are accepted — confirming the missing guard is a known gap, not an intentional design choice for the `AppPublicKey` variant.
