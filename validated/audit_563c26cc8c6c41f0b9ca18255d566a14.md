### Title
`initiateWithdrawal` Availability Check Uses Total Protocol Assets While `unlockQueue` Uses Only Vault Balance, Causing Temporary rsETH Freeze — (`contracts/LRTWithdrawalManager.sol`)

---

### Summary

`getAvailableAssetAmount` in `LRTWithdrawalManager` uses `getTotalAssetDeposits` (which includes illiquid EigenLayer assets, NDC balances, and converter balances) to gate `initiateWithdrawal`, while `unlockQueue` uses only `unstakingVault.balanceOf(asset)` (liquid vault assets) to actually service the queue. This accounting mismatch allows users to initiate withdrawals that cannot be unlocked, permanently locking their rsETH in the withdrawal manager until operators manually unstake from EigenLayer — a process with a mandatory 7-day EigenLayer delay.

---

### Finding Description

In `initiateWithdrawal`, the availability check calls `getAvailableAssetAmount`:

```solidity
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();
assetsCommitted[asset] += expectedAssetAmount;
```

`getAvailableAssetAmount` computes:

```solidity
uint256 totalAssets = lrtDepositPool.getTotalAssetDeposits(asset);
availableAssetAmount = totalAssets > assetsCommitted[asset] ? totalAssets - assetsCommitted[asset] : 0;
```

`getTotalAssetDeposits` aggregates across the entire protocol:

```solidity
return (assetLyingInDepositPool + assetLyingInNDCs + effectiveAssetWithEigenLayer
        + assetLyingInConverter + assetLyingUnstakingVault);
```

Under normal operation, the vast majority of assets are staked in EigenLayer via NodeDelegators. `getAvailableAssetAmount` therefore returns a large value — potentially the entire protocol TVL — allowing `initiateWithdrawal` to succeed and lock the user's rsETH.

However, when `unlockQueue` is called to actually service the queue, it uses only the liquid vault balance:

```solidity
return UnlockParams({
    rsETHPrice: lrtOracle.rsETHPrice(),
    assetPrice: lrtOracle.getAssetPrice(asset),
    totalAvailableAssets: unstakingVault.balanceOf(asset)   // ← only liquid vault assets
});
```

Inside `_unlockWithdrawalRequests`, if `availableAssetAmount < payoutAmount`, the loop breaks and the request remains locked:

```solidity
if (availableAssetAmount < payoutAmount) break;
```

The user's rsETH is held in the withdrawal manager with no cancel path. There is no `cancelWithdrawal` function in the contract.

---

### Impact Explanation

**Temporary freezing of funds (Medium).** A user who calls `initiateWithdrawal` when the unstaking vault is empty (the normal state — assets are in EigenLayer) will have their rsETH locked in `LRTWithdrawalManager` with no recourse. They must wait for operators to:
1. Queue an EigenLayer withdrawal (7-day mandatory delay).
2. Complete the withdrawal and move assets to the unstaking vault.
3. Call `unlockQueue`.

During this period the user cannot transfer, use, or recover their rsETH. The `getAvailableAssetAmount` view function signals that assets are available when they are not liquid, creating a false guarantee.

---

### Likelihood Explanation

**High.** The normal operational state of the protocol is that most assets are delegated to EigenLayer strategies via NodeDelegators. The unstaking vault holds assets only transiently, between EigenLayer withdrawal completion and queue unlock. Any user calling `initiateWithdrawal` when the vault is empty — which is the common case — will trigger this condition.

---

### Recommendation

Align the availability check in `initiateWithdrawal` with the actual liquid assets that `unlockQueue` can service. Replace the `getTotalAssetDeposits` call in `getAvailableAssetAmount` with `unstakingVault.balanceOf(asset)`, or add a separate "liquid available" check. Additionally, implement a `cancelWithdrawal` function so users can recover their rsETH if the queue cannot be unlocked within a reasonable time.

---

### Proof of Concept

1. Protocol state: 1 000 ETH staked in EigenLayer via NodeDelegators, 0 ETH in `LRTUnstakingVault`, `assetsCommitted[ETH] = 0`.
2. User calls `initiateWithdrawal(ETH, rsETHAmount)` where `getExpectedAssetAmount` returns 500 ETH.
3. `getAvailableAssetAmount(ETH)` returns `1000 - 0 = 1000 ETH` → check passes.
4. `assetsCommitted[ETH] += 500 ETH`; user's rsETH is transferred to `LRTWithdrawalManager`.
5. Operator calls `unlockQueue(ETH, ...)`.
6. `params.totalAvailableAssets = unstakingVault.balanceOf(ETH) = 0`.
7. Inside `_unlockWithdrawalRequests`: `availableAssetAmount (0) < payoutAmount (500 ETH)` → `break`.
8. `nextLockedNonce` is not advanced; the request remains locked.
9. User's rsETH is frozen in `LRTWithdrawalManager` with no cancel mechanism.
