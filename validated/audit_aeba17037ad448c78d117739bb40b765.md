Audit Report

## Title
FIFO Queue `break` in `_unlockWithdrawalRequests` Permanently Blocks Smaller Requests Behind an Underfunded Large Request - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
`_unlockWithdrawalRequests` processes withdrawal requests in strict FIFO order and `break`s immediately when the unstaking vault cannot cover the payout for the request at `nextLockedNonce`. Because `nextLockedNonce` is never advanced past the blocking request, all subsequent requests â€” regardless of their size or the vault's ability to cover them â€” remain permanently locked. No cancellation or skip mechanism exists, so affected users' rsETH is trapped in the contract with no recovery path.

## Finding Description
`initiateWithdrawal` immediately pulls rsETH from the caller into the contract and assigns a sequential global nonce:

```solidity
// line 166
IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);
// lines 756-757
userAssociatedNonces[asset][msg.sender].pushBack(nextUnusedNonce_);
nextUnusedNonce[asset] = nextUnusedNonce_ + 1;
```

The operator-only `unlockQueue` calls `_unlockWithdrawalRequests`, which iterates from `nextLockedNonce[asset]` upward. At line 800, when the vault balance is insufficient for the current request, the loop `break`s and `nextLockedNonce[asset]` is written back at the blocking nonce:

```solidity
// lines 800, 815
if (availableAssetAmount < payoutAmount) break;
...
nextLockedNonce[asset] = nextLockedNonce_;
```

Every future `unlockQueue` call restarts from the same blocking nonce. `_processWithdrawalCompletion` enforces that a user's oldest nonce must be strictly less than `nextLockedNonce[asset]`:

```solidity
// line 707
if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();
```

Any user whose personal queue front is at a nonce â‰Ą the blocking nonce cannot complete their withdrawal. The contract contains no `cancelWithdrawal`, no refund path, and no operator function to skip a specific nonce or advance the pointer past it.

## Impact Explanation
**Temporary freezing of funds (Medium) / Permanent freezing of funds (Critical).**

In the normal case, once the protocol accumulates enough of the asset to cover the blocking request, the queue can advance â€” this is a **temporary freeze (Medium)**. In an extreme case where the protocol suffers significant slashing for that asset type and can never accumulate enough to cover the blocking request, the freeze becomes **permanent (Critical)**. In both cases, rsETH is already held by the contract and users behind the blocking request have no escape path.

## Likelihood Explanation
**Medium.** The scenario requires only two ordinary users: one who submits a large withdrawal request first, and one (or many) who submit smaller requests afterward. This is a routine ordering that will occur in any active deployment. No attacker capability beyond calling `initiateWithdrawal` is required. The operator has no mechanism to prevent or remediate the freeze without a contract upgrade.

## Recommendation
Replace the `break` on insufficient assets with a `continue` so that smaller requests behind a large one can still be unlocked. Alternatively, introduce a separate skip-nonce mechanism allowing the operator to advance `nextLockedNonce` past a request that cannot currently be satisfied, combined with a user-facing `cancelWithdrawal` that returns rsETH when a request is skipped. At minimum, add a cancellation path so users are never permanently trapped.

## Proof of Concept
1. Deploy `LRTWithdrawalManager` with a supported asset (e.g., ETHx).
2. **User A** calls `initiateWithdrawal(ETHx, 1_000_000e18, "")`. rsETH is transferred to the contract. This request receives nonce 0.
3. **User B** calls `initiateWithdrawal(ETHx, 1e18, "")`. rsETH is transferred to the contract. This request receives nonce 1.
4. The unstaking vault holds only 0.5e18 ETHx â€” insufficient for User A's payout but sufficient for User B's.
5. Operator calls `unlockQueue(ETHx, 2, ...)`. Inside `_unlockWithdrawalRequests` (line 800): nonce 0 â†’ `availableAssetAmount (0.5e18) < payoutAmount (large)` â†’ `break`. `nextLockedNonce[ETHx]` stays at 0.
6. **User B** calls `completeWithdrawal(ETHx, "")`. Inside `_processWithdrawalCompletion` (line 707): `usersFirstWithdrawalRequestNonce = 1`; `nextLockedNonce[ETHx] = 0`; `1 >= 0` â†’ `revert WithdrawalLocked()`.
7. User B's rsETH remains locked in the contract. Even if the vault later accumulates 0.5e18 ETHx (enough for User B), the queue still cannot advance past nonce 0 until User A's full payout is available.
