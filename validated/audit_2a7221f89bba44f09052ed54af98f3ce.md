Audit Report

## Title
Fee Limit Check Before Price Commit in `_updateRsETHPrice` Allows Block Stuffing to Freeze `rsETHPrice` - (File: contracts/LRTOracle.sol)

## Summary
`_updateRsETHPrice()` accumulates all rewards since the last price update into a single fee amount, then calls `_checkAndUpdateDailyFeeMintLimit()` at line 303 before committing `rsETHPrice` at line 313. If two or more consecutive daily periods pass without a successful price update (e.g., via block stuffing), the multi-period fee exceeds `maxFeeMintAmountPerDay`, causing an unconditional revert and leaving `rsETHPrice` permanently stale until the manager raises the daily limit via `setMaxFeeMintAmountPerDay()`.

## Finding Description
In `_updateRsETHPrice()`, `rewardAmount` at line 245 is the **total** ETH gain since the last successful update — not one day's worth. The derived `rsethAmountToMintAsProtocolFee` at line 301 therefore scales linearly with the number of missed periods. At line 303, `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` is called before `rsETHPrice = newRsETHPrice` at line 313.

Inside `_checkAndUpdateDailyFeeMintLimit()`, the period reset at lines 199–202 correctly zeroes `currentPeriodMintedFeeAmount`, but the `feeAmount` argument is the full N-day fee. The check at line 205 evaluates `0 + N×dailyFee > maxFeeMintAmountPerDay`. When N ≥ 2 and `maxFeeMintAmountPerDay` is set to one day's expected fee, this reverts with `DailyFeeMintLimitExceeded` every time. Line 313 is unreachable, so `rsETHPrice` is never updated.

The manager bypass `updateRSETHPriceAsManager()` also calls `_updateRsETHPrice()` and hits the same revert path; recovery requires a separate `setMaxFeeMintAmountPerDay()` call first.

## Impact Explanation
`rsETHPrice` storage is frozen at its last committed value. `LRTDepositPool.getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()` at line 520, so all deposit minting uses a stale (undervalued) exchange rate for the duration of the freeze. This matches the allowed impact **Low. Block stuffing.** — the price oracle fails to deliver its promised updated rate while the freeze condition holds.

## Likelihood Explanation
Block stuffing for 2+ consecutive days on Ethereum mainnet is extremely capital-intensive. However, the same condition is reachable without any attacker: a keeper outage of ≥2 days (infrastructure failure, misconfiguration) produces identical behavior. `maxFeeMintAmountPerDay` is a manager-controlled parameter that can be set tightly to one day's expected fee, making the off-by-one condition easy to hit. The freeze is repeatable each time the gap between updates spans two or more periods.

## Recommendation
Move `rsETHPrice = newRsETHPrice` before the fee check, or replace the reverting check with a non-reverting cap that mints up to `maxFeeMintAmountPerDay` and discards the remainder. Either approach ensures the price is always committed regardless of fee limit state:

```solidity
// Option A: commit price first, then check/mint fee
rsETHPrice = newRsETHPrice;
if (protocolFeeInETH > 0) {
    uint256 rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice);
    _checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee); // revert here is now safe
    ...
}
```
