### Title
Cross-Token Drain via Global-Only Accounting Permanently Freezes Deposited Collateral — (`contracts/L2/RsETHTokenWrapper.sol`)

---

### Summary

`RsETHTokenWrapper` tracks collateral with a single global invariant (`totalSupply` of wrsETH vs. the balance of whichever token is queried), but holds balances of **multiple** allowed tokens. Because `_deposit` mints wrsETH against any allowed token and `_withdraw` burns wrsETH against any allowed token, a user can deposit tokenA, then redeem their wrsETH for tokenB (deposited by the bridger), leaving tokenA permanently locked with zero wrsETH outstanding to reclaim it.

---

### Finding Description

`_deposit` and `_withdraw` share a single ERC-20 supply counter but operate on an arbitrary `asset` chosen by the caller: [1](#0-0) 

```solidity
function _deposit(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    ERC20Upgradeable(_asset).safeTransferFrom(msg.sender, address(this), _amount);
    _mint(_to, _amount);                          // ← mints against global supply
    ...
}
``` [2](#0-1) 

```solidity
function _withdraw(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    _burn(msg.sender, _amount);                   // ← burns from global supply
    ERC20Upgradeable(_asset).safeTransfer(_to, _amount);
    ...
}
```

`depositBridgerAssets` uses `maxAmountToDepositBridgerAsset` to gate how much the bridger may deposit: [3](#0-2) 

```solidity
function maxAmountToDepositBridgerAsset(address _asset) public view returns (uint256) {
    uint256 wrsETHSupply = totalSupply();
    uint256 balanceOfAssetInWrapper = ERC20Upgradeable(_asset).balanceOf(address(this));
    if (balanceOfAssetInWrapper > wrsETHSupply) return 0;
    return wrsETHSupply - balanceOfAssetInWrapper;   // ← only checks THIS token's balance
}
```

When tokenA has already been deposited (`totalSupply = N`, `tokenA.balance = N`, `tokenB.balance = 0`), `maxAmountToDepositBridgerAsset(tokenB)` returns `N − 0 = N`, so the bridger is permitted to deposit N tokenB. The function does not subtract the balance of tokenA already held, so it incorrectly concludes there is room for N more units of collateral.

---

### Impact Explanation

**Concrete call sequence (no malicious actors required):**

| Step | Action | State after |
|------|--------|-------------|
| 1 | User calls `deposit(tokenA, N)` | `totalSupply = N`, `tokenA.bal = N`, `tokenB.bal = 0` |
| 2 | Bridger calls `depositBridgerAssets(tokenB, N)` (passes: `maxAmount = N − 0 = N`) | `totalSupply = N`, `tokenA.bal = N`, `tokenB.bal = N` |
| 3 | User calls `withdraw(tokenB, N)` | `totalSupply = 0`, `tokenA.bal = N`, `tokenB.bal = 0` |
| 4 | **tokenA is permanently locked** — `totalSupply = 0`, no wrsETH exists to redeem it | — |

The N tokenA units are irrecoverable: there is no admin rescue path, no emergency withdrawal, and no way to mint wrsETH without depositing new collateral. The original depositor's funds are permanently frozen.

**Impact:** Critical — Permanent freezing of funds.

---

### Likelihood Explanation

This requires only two conditions that are explicitly supported by the contract design:

1. More than one token is added via `addAllowedToken` (the contract has this function and the `reinitialize` path for exactly this purpose).
2. The bridger performs its normal duty of depositing bridged tokens to collateralize outstanding wrsETH.

Neither actor needs to be malicious. The bridger is acting correctly per its role; the user is calling a public function with a valid allowed token. The flaw is purely in the accounting model.

---

### Recommendation

Replace the global `totalSupply` accounting with **per-token deposit tracking**:

```solidity
mapping(address => uint256) public depositedPerToken;

function _deposit(address _asset, address _to, uint256 _amount) internal {
    ...
    depositedPerToken[_asset] += _amount;
    _mint(_to, _amount);
}

function _withdraw(address _asset, address _to, uint256 _amount) internal {
    require(depositedPerToken[_asset] >= _amount, "insufficient per-token balance");
    depositedPerToken[_asset] -= _amount;
    _burn(msg.sender, _amount);
    ERC20Upgradeable(_asset).safeTransfer(_to, _amount);
}
```

`depositBridgerAssets` should similarly credit `depositedPerToken[_asset]` and `maxAmountToDepositBridgerAsset` should compare against `depositedPerToken[_asset]` rather than the raw ERC-20 balance.

### Proof of Concept

Invariant fuzz test asserting the broken invariant:

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import "../contracts/L2/RsETHTokenWrapper.sol";
import "@openzeppelin/contracts/token/ERC20/ERC20.sol";

contract MockToken is ERC20 {
    constructor(string memory n) ERC20(n, n) { _mint(msg.sender, 1e24); }
}

contract CrossTokenDrainTest is Test {
    RsETHTokenWrapper wrapper;
    MockToken tokenA;
    MockToken tokenB;
    address admin   = address(1);
    address bridger = address(2);
    address user    = address(3);

    function setUp() public {
        tokenA = new MockToken("A");
        tokenB = new MockToken("B");

        wrapper = new RsETHTokenWrapper();
        wrapper.initialize(admin, bridger, address(tokenA));

        vm.prank(admin);
        wrapper.grantRole(wrapper.TIMELOCK_ROLE(), admin);

        vm.prank(admin);
        wrapper.addAllowedToken(address(tokenB));

        tokenA.transfer(user,    1000e18);
        tokenB.transfer(bridger, 1000e18);
    }

    function test_crossTokenDrain() public {
        uint256 N = 100e18;

        // Step 1: user deposits tokenA
        vm.startPrank(user);
        tokenA.approve(address(wrapper), N);
        wrapper.deposit(address(tokenA), N);
        vm.stopPrank();

        // Step 2: bridger deposits tokenB (maxAmount = totalSupply - tokenB.bal = N - 0 = N)
        vm.startPrank(bridger);
        tokenB.approve(address(wrapper), N);
        wrapper.depositBridgerAssets(address(tokenB), N);
        vm.stopPrank();

        // Step 3: user withdraws tokenB, burning all wrsETH
        vm.prank(user);
        wrapper.withdraw(address(tokenB), N);

        // Invariant: sum of all token balances must be <= totalSupply
        uint256 totalBal = tokenA.balanceOf(address(wrapper))
                         + tokenB.balanceOf(address(wrapper));
        uint256 supply   = wrapper.totalSupply();

        // This FAILS: totalBal = N, supply = 0
        assertLe(totalBal, supply, "INVARIANT BROKEN: tokens stranded with no wrsETH");
    }
}
```

Running this test will fail at the final assertion, confirming that N tokenA is permanently locked in the wrapper with `totalSupply = 0`.
