### Title
`validateGuardDatums` Silently Skips Datum Validation for Unresolved Script-Credential Guards in Sub-Transactions — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxow.hs`)

---

### Summary

In the Dijkstra era, `validateGuardDatums` is responsible for enforcing that every Plutus-script credential listed in a sub-transaction's `requiredTopLevelGuards` map carries a datum, while key-hash and native-script credentials must not. When the script backing a `ScriptHashObj` credential cannot be found in the sub-transaction's `ScriptsProvided` map, the function silently falls through with `Nothing -> acc`, skipping the datum-presence check entirely. An attacker-controlled sub-transaction can exploit this gap to omit the datum for a Plutus-script guard credential, bypassing the structural validation that is supposed to gate Plutus guard execution.

---

### Finding Description

`validateGuardDatums` iterates over `requiredTopLevelGuards` and classifies each credential:

```
accum acc cred mbDatum =
  case credScriptHash cred of
    Nothing  -> ...          -- key hash: datum must be SNothing
    Just scriptHash ->
      case Map.lookup scriptHash scripts of
        Just script
          | isNativeScript script -> ...   -- native: datum must be SNothing
          | otherwise             -> ...   -- Plutus: datum must be SJust
        Nothing -> acc                     -- ← silent skip
```

The `Nothing -> acc` branch is reached whenever the script hash is not present in `scripts` — the `ScriptsProvided` map for the sub-transaction. `ScriptsProvided` is populated by `getBabbageScriptsProvided`, which collects scripts from the sub-transaction's own witness set and from UTxO outputs referenced by the sub-transaction's reference inputs. A Plutus script that is neither inlined in a UTxO entry reachable by the sub-transaction's reference inputs nor supplied in the sub-transaction's witness set will be absent from `scripts`, even though it may be perfectly reachable by the top-level transaction (e.g., via the top-level transaction's reference inputs or witness set).

Because `getDijkstraScriptsNeeded` derives `guardingScriptsNeeded` from the top-level transaction's `guardsTxBodyL` — not from the sub-transaction's `requiredTopLevelGuards` — the Plutus guard script is scheduled for execution at the top-level UTXOW level, not the sub-level. The datum that is supposed to accompany that execution is the one stored in `requiredTopLevelGuards`. If that datum is `SNothing` (because `validateGuardDatums` never flagged it), the Plutus guard script receives no datum at execution time, which either causes a `CollectError` / missing-datum failure at a later stage or, depending on the script's internal logic, allows the guard to be satisfied without the intended authorization data. The `getDijkstraScriptsNeeded` function that drives guard-script execution reads only from `guardsTxBodyL` of the transaction body being processed, not from `requiredTopLevelGuards`: [2](#0-1) 

The `MissingRequiredGuards` check in the top-level UTXOW transition confirms that every credential in every sub-transaction's `requiredTopLevelGuards` is present in the top-level `guardsTxBodyL`, but it does not re-validate datum consistency: [3](#0-2) 

---

### Impact Explanation

A sub-transaction author can craft a `requiredTopLevelGuards` entry `{ScriptHashObj ph -> SNothing}` for a Plutus script `ph` that is not reachable through the sub-transaction's own reference inputs or witness set. `validateGuardDatums` silently accepts this. The top-level transaction then lists `ph` in `guardsTxBodyL`, causing `getDijkstraScriptsNeeded` to schedule `ph` for Plutus execution under the `Guarding` purpose. At execution time the datum argument is drawn from the sub-transaction's `requiredTopLevelGuards` entry, which is `SNothing`. Depending on the Plutus evaluator's handling of a missing datum for a `Guarding` purpose, this either:

- causes a `CollectError` / missing-datum predicate failure that is distinct from the intended `SubMalformedGuardDatums` failure, producing a ledger state divergence between implementations that handle the error differently; or
- allows the guard to be evaluated without the datum the script author intended to require, effectively bypassing the authorization data the guard was designed to enforce.

Either outcome constitutes an attacker-controlled transaction exceeding intended validation limits — the datum-presence invariant for Plutus guard credentials is violated without triggering the dedicated predicate failure.

**Allowed impact matched:** *Medium — attacker-controlled transactions exceed intended validation limits.*

---

### Likelihood Explanation

Any unprivileged transaction author who can submit a Dijkstra-era batch transaction (top-level + sub-transactions) can trigger this path. The attacker need only:

1. Choose a Plutus script `ph` that is not referenced by the sub-transaction's reference inputs.
2. Place `ScriptHashObj ph -> SNothing` in the sub-transaction's `requiredTopLevelGuards`.
3. Place `ScriptHashObj ph` in the top-level transaction's `guardsTxBodyL`.
4. Ensure `ph` is reachable by the top-level transaction (e.g., via the top-level reference inputs).

No privileged access, governance majority, or key compromise is required. The path is reachable on any Dijkstra-era node.

---

### Recommendation

Replace the `Nothing -> acc` branch with a conservative rejection: if the script hash cannot be resolved, the datum must be `SNothing` (treating the credential as non-Plutus), or the entry should be flagged as malformed. Alternatively, resolve the script type by also consulting the top-level transaction's `ScriptsProvided` map (which aggregates scripts from all levels including the top-level reference inputs), so that a Plutus script reachable only at the top level is still correctly classified when validating the sub-transaction's `requiredTopLevelGuards`.

```haskell
-- Suggested fix: treat unresolved script as requiring no datum
-- (conservative: reject if datum is present for unknown script)
Nothing ->
  case mbDatum of
    SNothing -> acc
    SJust _  -> Set.insert cred acc
```

Or, pass the aggregated top-level `ScriptsProvided` into `validateGuardDatums` so that Plutus scripts reachable only via top-level reference inputs are correctly identified.

  referenceInputsTxBodyL = [utxoRef]          -- utxoRef holds ph as a reference script

  requiredTopLevelGuards = {ScriptHashObj ph -> SNothing}

-- | Validate that requiredTopLevelGuards datums are consistent with the credential type:
-- Plutus script credentials must have a datum, key/native script credentials must not.
validateGuardDatums ::
  forall era.
  DijkstraEraTxBody era =>
  ScriptsProvided era ->
  TxBody SubTx era ->
  Test (DijkstraSubUtxowPredFailure era)
validateGuardDatums (ScriptsProvided scripts) txBody =
  failureOnNonEmptySet malformed SubMalformedGuardDatums
  where
    malformed =
      Map.foldlWithKey' accum mempty (txBody ^. requiredTopLevelGuardsL)
    accum acc cred mbDatum =
      case credScriptHash cred of
        Nothing ->
          -- Key hash: datum must be SNothing
          case mbDatum of
            SNothing -> acc
            SJust _ -> Set.insert cred acc
        Just scriptHash ->
          case Map.lookup scriptHash scripts of
            Just script
              | isNativeScript script ->
                  -- Native script: datum must be SNothing
                  case mbDatum of
                    SNothing -> acc
                    SJust _ -> Set.insert cred acc
              | otherwise ->
                  -- Plutus script: datum must be SJust
                  case mbDatum of
                    SJust _ -> acc
                    SNothing -> Set.insert cred acc
            Nothing -> acc

  ( EraTx era
  , DijkstraEraTxBody era
  , STxLevel l era ~ STxBothLevels l era

  UTxO era ->
  Tx l era ->
  ScriptsProvided era
getDijkstraScriptsProvided utxo tx =
  withBothTxLevels
    tx
    ( \topTx ->
        ScriptsProvided $
          Map.unions $
            unScriptsProvided (getBabbageScriptsProvided utxo topTx)

dijkstraUtxowTransition = do
  TRC (DijkstraUtxoEnv slot pp certState originalUtxo, u, stAnnTx) <- judgmentContext
  let tx = stAnnTx ^. txStAnnTxG
      scriptsProvided = scriptsProvidedStAnnTx stAnnTx

  let txBody = tx ^. bodyTxL
      subTxs = OMap.elems $ txBody ^. subTransactionsTxBodyL
      witsKeyHashes = keyHashWitnessesTxWits (tx ^. witsTxL)

  -- All lookups use originalUtxo.
  -- A subtx may consume a txout that the top-level tx references, so the UTXO threaded in the state may not contain it.

  -- scriptsNeeded for the top-level tx
  let topScriptsNeeded = scriptsNeededStAnnTx stAnnTx
      topScriptHashesNeeded = getScriptsHashesNeeded topScriptsNeeded
      subStAnnTxs = subTransactionsStAnnTx stAnnTx

  -- scriptsNeeded aggregated across all levels
  let allScriptHashesNeeded =
        Set.unions $
          topScriptHashesNeeded
            : (getScriptsHashesNeeded . scriptsNeededStAnnTx <$> subStAnnTxs)

  {- ∀s ∈ (txscripts txw utxo neededHashes ) ∩ Scriptph1 , validateScript s tx -}
  -- Per-level: phase-1 script validation is per-tx (script execution)
  runTest $ Babbage.validateFailedBabbageScripts tx scriptsProvided topScriptHashesNeeded

  {- neededHashes − dom(refScripts tx utxo) = dom(txwitscripts txw) -}
  -- Aggregated: missing/extraneous scripts across all levels.
  let witnessScripts =
        Map.keysSet (tx ^. witsTxL . scriptTxWitsL)
          <> foldMap (Map.keysSet . (^. witsTxL . scriptTxWitsL)) subTxs
      allRefScriptInputs =
        txBody ^. referenceInputsTxBodyL
          <> txBody ^. inputsTxBodyL
          <> foldMap
            ( \subTx ->
                subTx ^. bodyTxL . referenceInputsTxBodyL
                  <> subTx ^. bodyTxL . inputsTxBodyL
            )
            subTxs
      refScripts = Map.keysSet $ getReferenceScripts originalUtxo allRefScriptInputs
  runTest $ Babbage.babbageMissingScripts pp allScriptHashesNeeded refScripts witnessScripts

  {-  inputHashes ⊆  dom(txdats txw) ⊆  allowed -}
  -- Per-level: datum check for top-level tx's own spend inputs
  runTest $ Alonzo.missingRequiredDatums scriptsProvided originalUtxo tx

  {- dom (txrdmrs tx) = { rdptr txb sp | (sp, h) ∈ scriptsNeeded utxo tx,
                          h ↦ s ∈ txscripts txw, s ∈ Scriptph2} -}

  runTestOnSignal $ Shelley.validateVerifiedWits tx

  {- witsVKeyNeeded utxo tx genDelegs ⊆ witsKeyHashes -}
  runTest $ Shelley.validateNeededWitnesses witsKeyHashes certState originalUtxo txBody

  -- check metadata hash
  {- ((adh = ◇) ∧ (ad= ◇)) ∨ (adh = hashAD ad) -}
  runTestOnSignal $ Shelley.validateMetadata pp tx

  runTest $ Babbage.validateScriptsWellFormed pp tx

  -- Per-level: script integrity is per-tx (depends on that tx's redeemers and language views)

  runTest $ Alonzo.checkScriptIntegrityHash tx pp scriptIntegrity
