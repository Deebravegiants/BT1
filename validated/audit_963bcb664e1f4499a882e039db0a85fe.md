Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing `amount` in Limit Check - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` omits the incoming `amount` from the ETH branch of its limit check, comparing only the pre-deposit total against the cap. Any unprivileged caller can invoke `depositETH` with an arbitrarily large ETH value and push total ETH deposits far above `depositLimitByAsset(ETH_TOKEN)`, rendering the ETH deposit cap completely unenforceable.

## Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` (L676–682), the ETH branch evaluates `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)` without adding `amount`, while the ERC20 branch correctly evaluates `totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)`. [1](#0-0) 

The call chain is: `depositETH` (L76–93) → `_beforeDeposit` (L648–670) → `_checkIfDepositAmountExceedesCurrentLimit`. `depositETH` is `external payable` with no privilege guard beyond `whenNotPaused` and `onlySupportedAsset`. [2](#0-1) [3](#0-2) 

Because `amount` is never added to `totalAssetDeposits` before the comparison, the guard only rejects a deposit when the cap is already exceeded *before* the transaction — it never blocks a deposit that *would* exceed the cap. A single call with `msg.value` equal to any amount passes the check as long as the current total is below the limit.

## Impact Explanation
The ETH deposit cap is a protocol-level safety parameter. Because the check is evaluated on the pre-deposit state only, the cap is entirely unenforceable: one transaction can mint rsETH for an unbounded ETH amount beyond the intended ceiling. This matches **Low — Contract fails to deliver promised returns**: the protocol advertises a deposit limit that is never actually enforced for ETH, causing over-issuance of rsETH relative to the protocol's risk-managed ETH capacity.

## Likelihood Explanation
The vulnerable function is `public payable` with no access control. Any ETH holder can trigger it in a single transaction with no preconditions other than the contract not being paused. The ETH deposit path is the primary deposit route. Likelihood is **High**.

## Recommendation
Mirror the ERC20 branch for ETH in `_checkIfDepositAmountExceedesCurrentLimit`:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [4](#0-3) 

## Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN)` = 100 ether.
2. Current state: `getTotalAssetDeposits(ETH_TOKEN)` = 1 ether.
3. Attacker calls `LRTDepositPool.depositETH{value: 10_000 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 10_000 ether)` evaluates `1 ether > 100 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for 10 000 ETH of value.
6. `getTotalAssetDeposits(ETH_TOKEN)` returns 10 001 ether — 9 901 ether above the cap.

Foundry test plan: deploy `LRTDepositPool` with a mock config setting `depositLimitByAsset(ETH_TOKEN) = 100 ether`, seed 1 ether of existing deposits, call `depositETH{value: 10_000 ether}`, assert no revert occurs and `getTotalAssetDeposits` exceeds 100 ether post-call.

    function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
        uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
        if (asset == LRTConstants.ETH_TOKEN) {
            return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
        }
        return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
    }
