### Title
`register_backup_service()` Does Not Verify the Registered TLS Key Belongs to a TEE-Attested Backup Service - (File: `crates/contract/src/lib.rs`)

### Summary

`register_backup_service()` accepts an arbitrary `Ed25519PublicKey` as the backup service's TLS identity with no TEE attestation, no proof of approved software, and no cryptographic binding between the submitted key and a genuine TEE environment. The MPC node uses this on-chain key as the sole mTLS peer identity when transferring keyshares. A participant who registers an attacker-controlled TLS key causes their MPC node to deliver keyshares to an unapproved entity, breaking the TEE security invariant that is supposed to prevent even operators from directly accessing keyshares outside an approved TEE.

### Finding Description

`register_backup_service()` in `crates/contract/src/lib.rs` stores a caller-supplied `BackupServiceInfo { public_key: Ed25519PublicKey }` with no verification beyond confirming the caller is a participant:

```rust
pub fn register_backup_service(
    &mut self,
    backup_service_info: dtos::BackupServiceInfo,
) -> Result<(), Error> {
    let account_id = Self::assert_caller_is_signer();
    if !self.protocol_state.is_existing_or_prospective_participant(&account_id)? {
        return Err(errors::InvalidState::NotParticipant { ... }.into());
    }
    self.node_migrations.set_backup_service_info(account_id, backup_service_info);
    Ok(())
}
``` [1](#0-0) 

The `BackupServiceInfo` struct carries only a bare TLS public key — no attestation quote, no TEE measurement, no binding to approved software:

```rust
pub struct BackupServiceInfo {
    pub public_key: Ed25519PublicKey,
}
``` [2](#0-1) 

By contrast, MPC nodes must call `submit_participant_info()` with a full TDX attestation quote that the contract verifies against an approved Docker image hash whitelist, a launcher compose hash, a timestamp deadline, and a `ReportData` binding `SHA3-384(tls_public_key || account_public_key)` to the TEE. The backup service registration path has none of these checks. [3](#0-2) 

The project's own documentation explicitly acknowledges this gap:

> "The backup service (soft launch) registers via `register_backup_service()` with only a TLS public key — no attestation, no TEE verification." [4](#0-3) 

The MPC node reads the registered TLS key from the contract and uses it as the trusted mTLS peer identity when sending keyshares. There is no secondary check that the peer is running inside an approved TEE. [5](#0-4) 

### Impact Explanation

The MPC contract is the public-key infrastructure for the backup/migration system. MPC nodes authenticate the backup service peer exclusively by the TLS key stored on-chain. A participant who registers an arbitrary TLS key (one whose private key is held outside any TEE, or inside a TEE running unapproved software) causes their MPC node to deliver AES-256-encrypted keyshares to that unapproved entity. The operator also controls the AES-256 symmetric key (`MPC_BACKUP_ENCRYPTION_KEY_HEX`) shared between the MPC node and backup service, so the operator can decrypt the received keyshares. This breaks the TEE security invariant: the hard-launch design is explicitly intended to prevent even operators from directly accessing keyshares outside an approved TEE environment. A single participant's keyshare is below the signing threshold, so this alone does not enable immediate fund theft, but it violates the production safety invariant that keyshares must only leave the MPC node into a verified TEE.

This maps to: **Medium — participant-state and contract execution-flow manipulation that breaks production safety/accounting invariants** (the TEE-enforced keyshare custody model).

### Likelihood Explanation

Any current or prospective participant can call `register_backup_service()` with an arbitrary TLS key at any time (Running, Initializing, or Resharing state). No special conditions are required beyond being a participant. The call is free (no deposit enforced) and idempotent, so it can be updated at will. [6](#0-5) 

### Recommendation

Require a TEE attestation in `register_backup_service()` analogous to `submit_participant_info()`. Specifically:

1. Extend `BackupServiceInfo` to include a `Dstack` attestation quote (reject `Mock` in production).
2. Verify the quote against a separate `backup_service_tee_state` with its own approved Docker image hash whitelist (voted in by participants via `vote_backup_service_code_hash()`).
3. Verify `ReportData = SHA3-384(tls_public_key || account_public_key)` to cryptographically bind the submitted TLS key to the TEE that generated it.
4. Verify the transaction signer's public key matches `account_public_key` via `env::signer_account_pk()`.

This is exactly the design already specified in `docs/backup-service-attestation-data.md` and `docs/migration-service.md` for the hard-launch phase. It should be enforced before keyshare transfer is enabled in production. [7](#0-6) 

### Proof of Concept

1. Attacker is a legitimate MPC participant with account `attacker.near`.
2. Attacker generates an Ed25519 keypair `(sk_evil, pk_evil)` outside any TEE on a standard server.
3. Attacker calls:
   ```
   register_backup_service({"backup_service_info": {"public_key": "ed25519:<pk_evil>"}})
   ```
   The contract accepts this — `attacker.near` is a participant, no attestation is checked. [8](#0-7) 
4. Attacker's MPC node reads `pk_evil` from the contract via the indexer and establishes an mTLS connection to the attacker's server (which presents a certificate for `pk_evil`).
5. The MPC node sends AES-256-encrypted keyshares to the attacker's server.
6. The attacker, who also controls `MPC_BACKUP_ENCRYPTION_KEY_HEX` (the operator-provided symmetric key), decrypts the keyshares.
7. The attacker now holds their keyshare in plaintext outside any TEE, violating the TEE custody invariant. If the signing threshold is low (e.g., t-of-n where t is small), colluding participants performing the same attack can reconstruct the full private key. [9](#0-8)

1. The node operator calls the contract method `register_backup_service()` to register the backup service's public key in the smart contract. The node, running a NEAR client, has access to this information and uses it for the authentication in the following step.
2. The node operator manually runs `backup-cli get-keyshares` with the MPC node's URL, public key and the symmetric AES-256 key (matching `MPC_BACKUP_ENCRYPTION_KEY_HEX` below) as input. This triggers the following:
    1. The `backup-cli` and MPC node establish a mutually authenticated TLS connection using their P2P keys.
    2. The `backup-cli` requests the keyshares from the MPC node's `GET /get_keyshares` endpoint
    3. The MPC node returns the AES-256 encrypted keyshares. The MPC node uses the node-operator provided symmetric key `MPC_BACKUP_ENCRYPTION_KEY_HEX` for encryption.
    4. The `backup-cli` saves the encrypted keyshares to local storage
