Audit Report

## Title
Reward Tokens Permanently Frozen When `totalKernelStaked` Drops to Zero Mid-Period — (`contracts/KERNEL/KernelDepositPool.sol`)

## Summary
`initiateWithdrawal` has no guard preventing it from being called during an active reward period. If all stakers withdraw, `totalKernelStaked` reaches zero and `rewardPerToken()` freezes at `rewardPerTokenStored`, causing `rewardRate × (finishAt − T_zero)` tokens to accumulate in the contract with no accounting entry. These tokens cannot be claimed by any staker and cannot be recaptured by `notifyRewardAmount` due to its transfer-in delta pattern, resulting in permanent freezing of unclaimed yield.

## Finding Description
`rewardPerToken()` short-circuits to `rewardPerTokenStored` whenever `totalKernelStaked == 0`: [1](#0-0) 

`initiateWithdrawal` decrements `totalKernelStaked` with no check on whether a reward period is active (`block.timestamp < finishAt`): [2](#0-1) 

The `updateReward` modifier snapshots `rewardPerTokenStored` and `updatedAt` **before** the balance change via `_;`: [3](#0-2) 

When the last staker calls `initiateWithdrawal`:
1. `updateReward` runs: `rewardPerTokenStored` is updated to the current value, `updatedAt = block.timestamp`.
2. `totalKernelStaked -= amount` → `totalKernelStaked = 0`.
3. For all subsequent blocks until `finishAt`, `rewardPerToken()` returns the frozen `rewardPerTokenStored`.
4. `rewardRate × (finishAt − updatedAt)` tokens accumulate in the contract with no accounting entry.

When a new staker eventually calls `stake`, `updateReward` runs again with `totalKernelStaked` still 0 at modifier time, so `rewardPerTokenStored` stays frozen and `updatedAt` advances to `lastTimeRewardApplicable()`. The gap-period rewards are silently skipped.

Recovery via a subsequent `notifyRewardAmount` call is impossible because the function measures only the **delta** of the transfer-in: [4](#0-3) 

The already-stuck tokens are not counted as new rewards. There is no admin `recoverStuckRewards` or equivalent function in the contract. The contract's own NatDoc acknowledges this exact failure mode but relies entirely on an off-chain operational promise with no on-chain enforcement: [5](#0-4) 

## Impact Explanation
Reward tokens equal to `rewardRate × (finishAt − T_zero)` — where `T_zero` is the timestamp at which `totalKernelStaked` last hit zero — are permanently locked in the contract. They cannot be claimed by any staker (since `earned()` depends on `balanceOf[_account]`, which is 0 post-withdrawal) and cannot be recaptured by `notifyRewardAmount`. This matches **Medium — Permanent freezing of unclaimed yield**.

## Likelihood Explanation
No special role or privilege is required; any staker can call `initiateWithdrawal` at any time. A single large staker holding the majority of `totalKernelStaked` can trigger this unilaterally. Multiple stakers withdrawing independently for legitimate reasons (market conditions, risk management) can trigger it without coordination. The `withdrawalDelay` only delays the token transfer, not the `totalKernelStaked` decrement, so the freeze occurs at `initiateWithdrawal` time, not `claimWithdrawal` time. [6](#0-5) 

## Recommendation
Add an on-chain guard in `initiateWithdrawal` that prevents `totalKernelStaked` from reaching zero while a reward period is active:

```solidity
// After decrementing totalKernelStaked:
if (totalKernelStaked == 0 && block.timestamp < finishAt) {
    revert CannotLeaveZeroStakeDuringRewardPeriod();
}
```

Alternatively, add an admin-callable `recoverStuckRewards` function that can sweep the unaccounted balance after a period ends, or track `undistributedRewards` explicitly and allow the admin to roll them into the next period.

## Proof of Concept

```solidity
// 1. Admin sets duration and notifies rewards (totalKernelStaked > 0 required)
pool.setRewardsDuration(7 days);
kernelToken.approve(address(pool), 1000e18);
pool.stake(1000e18);                         // Alice stakes
pool.notifyRewardAmount(700e18);             // rewardRate = 700e18 / 7 days

// 2. Mid-period: Alice initiates withdrawal (no guard prevents this)
vm.warp(block.timestamp + 1 days);          // 1 day into the 7-day period
pool.initiateWithdrawal(1000e18);           // totalKernelStaked → 0

// 3. Warp to end of period
vm.warp(pool.finishAt());

// 4. New staker stakes 1 wei to trigger updateReward
kernelToken.approve(address(pool), 1);
pool.stake(1);

// 5. Assert: contract holds more reward tokens than all earned() sums
uint256 contractBalance = rewardsToken.balanceOf(address(pool));
uint256 aliceEarned = pool.earned(alice);
// contractBalance ≈ 700e18 (all rewards), aliceEarned ≈ 100e18 (1 day)
// ~600e18 permanently frozen — unreachable by any staker or admin
assert(contractBalance > aliceEarned);
```

    function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
        if (_amount == 0) revert AmountZero();
        if (balanceOf[msg.sender] < _amount) revert InsufficientStakedBalance();
        if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();

        balanceOf[msg.sender] -= _amount;
        totalKernelStaked -= _amount;

        // Create a withdrawal record
        uint256 withdrawalId = ++withdrawalCounter;
        uint256 unlockTime = block.timestamp + withdrawalDelay;

        withdrawals[withdrawalId] = Withdrawal({
            user: msg.sender, amount: _amount, unlockTime: unlockTime, claimed: false, withdrawalId: withdrawalId
        });
        userWithdrawalIds[msg.sender].push(withdrawalId);

        emit WithdrawalInitiated(msg.sender, _amount, withdrawalId, unlockTime);
    }
