### Title
Unvalidated `ProposalInit.builder` Address Allows Malicious Proposer to Corrupt Sequencer Address in Block Hash and Fee Collection - (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

`is_proposal_init_valid` validates many fields of a received `ProposalInit` (height, timestamp, L1/L2 gas prices, starknet version, version constant commitment, fee proposal) but never validates the `builder` field. This field is used verbatim as `sequencer_address` in the block execution context and is hashed into `PartialBlockHashComponents`. A malicious proposer can set `builder` to any arbitrary address; validators accept the proposal, execute the block with the wrong sequencer address, and commit a wrong block hash.

### Finding Description

`is_proposal_init_valid` in `crates/apollo_consensus_orchestrator/src/validate_proposal.rs` checks the following fields of the incoming `ProposalInit`: [1](#0-0) 

The `ProposalInitValidation` struct that carries the locally-trusted reference values contains `height`, `l1_da_mode`, `l2_gas_price_fri`, `starknet_version`, `block_timestamp_window_seconds`, `previous_proposal_init`, and `fee_actual` — but no `builder` field: [2](#0-1) 

After `is_proposal_init_valid` passes, `convert_to_sn_api_block_info` maps `init.builder` directly to `sequencer_address` with no further check: [3](#0-2) 

This `BlockInfo` is then passed to `PartialBlockHashComponents::new`, which stores `sequencer_address` as a component of the block hash: [4](#0-3) 

`calculate_block_hash` chains the sequencer address directly into the Poseidon hash: [5](#0-4) 

The only address-level check in the consensus manager is on `init.proposer` (the consensus-layer proposer), not on `init.builder` (the block-builder/sequencer address): [6](#0-5) 

The `builder` address is also propagated to state sync: [7](#0-6) 

### Impact Explanation

A malicious proposer (any legitimate consensus participant when it is their turn to propose) can set `ProposalInit.builder` to an arbitrary address — including an attacker-controlled address, zero, or a reserved system address. Because both the proposer and the validator derive the `ProposalCommitment` from the same `init.builder` value, the `ProposalFinMismatch` check passes. The block is committed with:

1. **Wrong `sequencer_address` in block execution context** — the `get_sequencer_address` syscall returns the attacker-supplied address, corrupting any contract logic that depends on it.
2. **Wrong block hash committed to storage** — `PartialBlockHashComponents.sequencer` is wrong, so the final block hash diverges from what honest nodes would compute for the canonical sequencer address.
3. **Fee collection redirected** — `get_sequencer_balance_keys` uses `sequencer_address` to determine where transaction fees are credited; fees flow to the attacker-controlled address.

This matches the allowed impact scopes:
- *Critical. Wrong state … or revert result from blockifier/syscall/execution logic for accepted input* (wrong `get_sequencer_address` syscall result).
- *Critical. Incorrect fee … balance … with economic impact* (fees redirected).

### Likelihood Explanation

Any validator that is legitimately selected as proposer for a round can trigger this. No special privilege beyond being in the active validator set is required. The attack is a single-field modification to the outgoing `ProposalInit` message.

### Recommendation

Add `builder` to `ProposalInitValidation` and enforce it in `is_proposal_init_valid`. The validator should compare `init_proposed.builder` against the locally-configured builder address (or the expected builder for the height/round), rejecting any proposal where the two differ:

```rust
if init_proposed.builder != proposal_init_validation.expected_builder {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.expected_builder, init_proposed.builder
        ),
    ));
}
```

Additionally, `ContractAddress::validate()` (which rejects zero and reserved addresses) should be called on `init.builder` before it is used in `convert_to_sn_api_block_info`.

### Proof of Concept

1. A malicious node is selected as proposer for height H, round R.
2. It constructs `ProposalInit` with `builder = attacker_address` (e.g., `ContractAddress::from(0x1337)`).
3. It computes `PartialBlockHashComponents` and `ProposalCommitment` using `attacker_address` as sequencer.
4. It streams `ProposalPart::Init(init)` followed by transactions and `ProposalPart::Fin(fin)` with the commitment derived from `attacker_address`.
5. The validator calls `is_proposal_init_valid` — passes (no `builder` check).
6. The validator calls `convert_to_sn_api_block_info(&init)` → `sequencer_address = attacker_address`.
7. The validator's batcher executes the block with `attacker_address` as sequencer; `get_sequencer_address` syscalls return `attacker_address`; fees are credited to `attacker_address`.
8. The validator computes the same `ProposalCommitment` as the proposer (both used `attacker_address`) → `ProposalFinMismatch` check passes.
9. `decision_reached` commits the block; storage records a block hash that encodes `attacker_address` as sequencer.
