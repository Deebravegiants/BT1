### Title
Unvalidated `ProposalInit.builder` Field Allows Proposer to Inject Arbitrary Sequencer Address into Block Hash and Fee Recipient — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_proposal_init_valid` validates many fields of a received `ProposalInit` (height, L1/L2 gas prices, timestamp, starknet version, `version_constant_commitment`, `fee_proposal_fri`) but never checks the `builder` field. That field is passed verbatim to `convert_to_sn_api_block_info` as `sequencer_address`, which is then hashed into `PartialBlockHashComponents` and used as the fee-collection address during execution. A malicious proposer can set `builder` to any arbitrary address; every validating node will accept the proposal, commit the block with the wrong sequencer address, produce a wrong partial block hash, and credit sequencer fees to the attacker-controlled address.

---

### Finding Description

**Root cause — missing field in `ProposalInitValidation`**

`ProposalInitValidation` carries the locally-trusted reference values the validator checks against:

```rust
pub(crate) struct ProposalInitValidation {
    pub height: BlockNumber,
    pub block_timestamp_window_seconds: u64,
    pub previous_proposal_init: Option<PreviousProposalInitInfo>,
    pub l1_da_mode: L1DataAvailabilityMode,
    pub l2_gas_price_fri: GasPrice,
    pub starknet_version: StarknetVersion,
    pub fee_actual: Option<GasPrice>,
}
``` [1](#0-0) 

There is no `builder` (or `proposer`) field. `is_proposal_init_valid` therefore has no reference value to compare against.

**The validation gate**

The combined equality check that guards the proposal is:

```rust
if !(init_proposed.height == proposal_init_validation.height
    && init_proposed.l1_da_mode == proposal_init_validation.l1_da_mode
    && init_proposed.l2_gas_price_fri == proposal_init_validation.l2_gas_price_fri)
{
    return Err(ValidateProposalError::InvalidProposalInit(...));
}
``` [2](#0-1) 

`init_proposed.builder` is never compared to anything.

**How `builder` reaches the block hash**

After `is_proposal_init_valid` passes, `initiate_validation` calls `convert_to_sn_api_block_info`:

```rust
Ok(starknet_api::block::BlockInfo {
    block_number: init.height,
    block_timestamp: BlockTimestamp(init.timestamp),
    sequencer_address: init.builder,   // ← proposer-supplied, unvalidated
    ...
})
``` [3](#0-2) 

That `sequencer_address` is then hashed into `PartialBlockHashComponents`:

```rust
sequencer: SequencerContractAddress(block_info.sequencer_address),
``` [4](#0-3) 

and chained into the final block hash:

```rust
.chain(&partial_block_hash_components.sequencer.0)
``` [5](#0-4) 

**Proposer side sets it from static config**

The honest proposer sets `builder` from its own node configuration:

```rust
// TODO(Asmaa): Get it from committee once we have it.
builder_address: self.config.static_config.builder_address,
``` [6](#0-5) 

A malicious proposer simply puts any `ContractAddress` in that field before broadcasting `ProposalInit`.

---

### Impact Explanation

1. **Wrong block hash / state commitment.** `sequencer_address` is a direct input to `calculate_block_hash`. Any deviation produces a different `PartialBlockHash`, which propagates into the `ProposalCommitment` that consensus signs over and that the L1 contract verifies. The committed block hash is wrong.

2. **Wrong fee recipient / economic impact.** During execution the blockifier uses `block_info.sequencer_address` as the address that receives sequencer fees. An attacker-controlled `builder` redirects all sequencer fees for that block to an arbitrary address.

3. **Validator-side acceptance is unconditional.** Because `is_proposal_init_valid` has no reference value for `builder`, every honest validator node will accept the manipulated `ProposalInit`, execute the block with the wrong sequencer address, and commit it — there is no local guard that can catch the deviation.

---

### Likelihood Explanation

The trigger requires the attacker to be the elected proposer for a round, which is a role that rotates among validators. In a permissioned validator set this is a low-frequency but reachable condition. No external user interaction is needed beyond the normal consensus message flow; the attacker simply modifies the `builder` field in the `ProposalInit` protobuf message they broadcast.

---

### Recommendation

Add `builder` (and optionally `proposer`) to `ProposalInitValidation` and check it in `is_proposal_init_valid`:

```rust
pub(crate) struct ProposalInitValidation {
    ...
    pub expected_builder: ContractAddress,  // from static config
}
```

```rust
if init_proposed.builder != proposal_init_validation.expected_builder {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!("builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.expected_builder, init_proposed.builder),
    ));
}
```

`SequencerConsensusContext::validate_current_round_proposal` already constructs `ProposalInitValidation`; it should populate `expected_builder` from `self.config.static_config.builder_address`. [7](#0-6) 

---

### Proof of Concept

1. Attacker controls a validator node that is elected proposer for round R at height H.
2. In `initiate_build`, the attacker patches `builder` to `ContractAddress(attacker_wallet)` before broadcasting `ProposalInit`.
3. Every honest validator calls `is_proposal_init_valid` → passes (no `builder` check).
4. `initiate_validation` calls `convert_to_sn_api_block_info(init)` → `sequencer_address = attacker_wallet`.
5. Batcher executes the block; all sequencer fees are credited to `attacker_wallet`.
6. `PartialBlockHashComponents::new` hashes `attacker_wallet` as the sequencer → wrong `PartialBlockHash`.
7. `proposal_commitment_from(partial_block_hash, fee_proposal)` produces a wrong `ProposalCommitment`.
8. Consensus reaches decision on this commitment; the block is committed with the wrong sequencer address and wrong block hash.
