### Title
Stale `accept_requests` Flag Permits Signing with Expired TEE Attestations — (`crates/contract/src/lib.rs`)

---

### Summary

The `accept_requests` boolean in `MpcContract` is a **cached value** set only when `verify_tee()` is explicitly called by a participant. Between those calls, the flag is never re-evaluated against the live TEE state. If one or more participants' attestations expire in the interval between two `verify_tee()` invocations, the contract continues to accept `sign()`, `respond()`, and `verify_foreign_transaction()` calls as though all attestations are still valid — directly analogous to `.latestRoundData()` returning a cached price rather than the current one.

---

### Finding Description

`accept_requests` is initialized to `true` in `init_running()` and is subsequently updated only inside `verify_tee()`: [1](#0-0) [2](#0-1) 

Every user-facing entry point — `sign()`, `verify_foreign_transaction()`, and the node-facing `respond()` / `respond_ckd()` / `respond_verify_foreign_tx()` — gates on this single cached flag: [3](#0-2) [4](#0-3) [5](#0-4) 

`verify_tee()` itself calls `reverify_and_cleanup_participants()` to inspect the live attestation expiry timestamps stored in `tee_state.stored_attestations`: [6](#0-5) 

But `verify_tee()` is **not called automatically** — it must be invoked explicitly by a participant. There is no on-chain timer, no callback, and no hook that re-evaluates `accept_requests` when block time advances past an attestation's `expiry_timestamp_seconds`. The contract's own sandbox tests confirm that attestations do expire with the passage of blocks: [7](#0-6) 

---

### Impact Explanation

TEE attestations carry a finite `expiry_timestamp_seconds`. Once that timestamp passes, the node is no longer operating inside a verified TEE environment — the security guarantee the MPC network is built on. Because `accept_requests` is never automatically cleared when attestations expire, the contract continues to:

1. Accept new `sign()` / `verify_foreign_transaction()` requests from users.
2. Accept `respond()` calls from nodes whose TEE attestations have lapsed.

A node whose attestation has expired but whose account is still in the participant set can submit signing responses that the contract will accept and relay to callers. If that node has been compromised (the precise scenario TEE expiry is meant to detect and block), it can contribute malicious signature shares to threshold signing rounds without the contract raising any objection. This breaks the production safety invariant that every signing participant must be running inside a verified TEE at the time of signing.

**Allowed impact matched**: *Medium — contract execution-flow manipulation that breaks production safety/accounting invariants* (the invariant being: all active signing participants hold a currently-valid TEE attestation).

---

### Likelihood Explanation

Attestation expiry is a normal, expected event. The `tee_upgrade_deadline_duration_seconds` config parameter explicitly anticipates a grace window for upgrades. During any upgrade cycle — or any period where participants are slow to call `verify_tee()` — the window between the last `verify_tee()` call and the next one can span many blocks. An adversarial or compromised node operator can deliberately delay calling `verify_tee()` to extend this window. No privileged access beyond being an existing participant is required; the attacker simply waits.

---

### Recommendation

Replace the cached `accept_requests` flag with an inline, per-call check against the live attestation state in `tee_state`. Concretely, `check_request_preconditions()` and the `respond*` entry points should call a lightweight read-only variant of `reverify_and_cleanup_participants()` (or a dedicated `are_all_attestations_valid()` predicate) rather than reading the pre-computed boolean. Alternatively, introduce an on-chain expiry deadline that is compared against `env::block_timestamp()` on every request, so the contract self-enforces freshness without requiring an explicit `verify_tee()` call.

---

### Proof of Concept

```
T=0   verify_tee() called → all 3 participants valid → accept_requests = true
T=1   Participant P3's attestation expires (expiry_timestamp_seconds reached)
T=2   No one calls verify_tee()
T=3   User calls sign() → check_request_preconditions reads accept_requests == true → request enqueued
T=4   P3 (expired TEE, potentially compromised) calls respond() →
        accept_requests still true → response accepted → signature delivered to user
```

The contract's own test suite demonstrates that `accept_requests` is only set to `false` when `verify_tee()` is explicitly invoked *after* expiry: [8](#0-7) 

Between `T=1` and the next `verify_tee()` call, every signing path operates on the stale cached value, accepting requests and responses from a node that is no longer TEE-attested.
