### Title
Unbounded Per-Bin Additional Fee Bypasses Factory Owner's Admin Fee Cap — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`setPoolBinAdditionalFees` allows the pool admin to set per-bin `addFeeBuyE6` / `addFeeSellE6` to any value up to `type(uint16).max = 65 535` (≈ 6.55% in E6 units) with no factory-enforced upper bound. The global admin spread fee is explicitly capped by the factory owner via `maxAdminSpreadFeeE6`, but the per-bin additional fee path has no analogous cap, letting the pool admin charge traders fees that exceed the factory owner's intended ceiling.

### Finding Description

The factory enforces a two-layer cap on the global admin spread fee:

- A hard constant `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%) [1](#0-0) 
- A configurable `maxAdminSpreadFeeE6` (≤ hard cap) checked in `setPoolAdminFees`: [2](#0-1) 

`setPoolBinAdditionalFees`, however, passes the caller-supplied `uint16` values straight through to the pool with no cap check:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [3](#0-2) 

The pool stores the values verbatim: [4](#0-3) 

During every swap the per-bin fee is added directly to `baseFeeX64` before the fee-amount calculation:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [5](#0-4) 

The same pattern applies to the sell direction and to the exact-output paths. [6](#0-5) 

`BinState.addFeeBuyE6` is a `uint16`, so the maximum storable value is 65 535, representing 6.5535% in E6 units — with no factory-level ceiling. [7](#0-6) 

### Impact Explanation

Every swap through an affected bin pays `feeAmountScaled = ceil(amountIn × (baseFee + addFee))`. The additional fee is borne entirely by the trader: the excess input is split between LPs and the protocol/admin (via `spreadFeeE6`). A pool admin who sets `addFeeBuyE6 = 65535` imposes an extra ≈ 6.55% fee on every buy-side swap in that bin, directly reducing the trader's output. This is a direct loss of user principal on every swap routed through the bin.

The factory owner can lower `maxAdminSpreadFeeE6` to, say, 1 000 (0.1%) to protect traders, but the per-bin additional fee is completely outside that cap, so the pool admin can still impose 6.55% per-bin fees regardless of the factory owner's intent.

### Likelihood Explanation

The pool admin is a semi-trusted role set at pool creation. Any pool admin — including one that turns adversarial after deployment — can call `setPoolBinAdditionalFees` at any time with no timelock, no cap check, and no prior fee-collection step. The call is a single transaction with immediate effect on all subsequent swaps through the targeted bin.

### Recommendation

Add a hard cap on per-bin additional fees inside `setPoolBinAdditionalFees` (or inside `setBinAdditionalFees` on the pool), analogous to the cap already applied to the global admin spread fee:

```solidity
uint256 constant HARD_MAX_BIN_ADD_FEE_E6 = 200_000; // or a lower value

function setPoolBinAdditionalFees(
    address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6
) external override nonReentrant onlyPoolAdmin(pool) {
    if (addFeeBuyE6 > HARD_MAX_BIN_ADD_FEE_E6) revert BinFeeTooHigh();
    if (addFeeSellE6 > HARD_MAX_BIN_ADD_FEE_E6) revert BinFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, expose a configurable `maxBinAdditionalFeeE6` that the factory owner can adjust, consistent with the existing `maxAdminSpreadFeeE6` pattern.

### Proof of Concept

1. Factory owner sets `maxAdminSpreadFeeE6 = 1_000` (0.1%) to protect traders.
2. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — no revert.
3. A trader calls `swap(recipient, true, amountIn, ...)` routing through bin 0.
4. Inside `_swapToken1ForToken0SpecifiedInput`, the effective buy fee is `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)` ≈ oracle spread + 6.55%.
5. The trader receives ≈ 6.55% less token0 than they would with a 0-additional-fee bin, while the factory owner's 0.1% cap on the global admin fee is completely bypassed.
