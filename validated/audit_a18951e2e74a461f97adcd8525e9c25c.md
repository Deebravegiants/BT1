Audit Report

## Title
`addLiquidity` Missing `whenNotPaused` Guard Allows Deposits Into a Paused Pool — (`metric-core/contracts/MetricOmmPool.sol`)

## Summary
`MetricOmmPool.swap` is protected by `whenNotPaused`, but `MetricOmmPool.addLiquidity` is not. Any unprivileged caller can deposit real token balances into a pool that has been paused at level 1 or 2, precisely when the oracle is known to be stale or unsafe. When the pool unpauses with a corrected oracle price, the newly deposited tokens are immediately exposed to arbitrage at the corrected price, causing direct LP principal loss.

## Finding Description
`MetricOmmPool` maintains a `pauseLevel` state variable (0 = active, 1 = admin-paused, 2 = protocol-paused). The `whenNotPaused` modifier calls `_checkNotPaused()`, which reverts if `pauseLevel != 0`. [1](#0-0) [2](#0-1) 

`swap` correctly applies this guard: [3](#0-2) 

`addLiquidity` does not: [4](#0-3) 

`LiquidityLib.addLiquidity` does not consult the oracle at all — it computes token amounts from existing `binState.token0BalanceScaled` / `token1BalanceScaled` and the pool cursor (`curBinIdx`, `curPosInBin`), then updates `binTotals.scaledToken0` / `scaledToken1` with real token amounts pulled via callback: [5](#0-4) [6](#0-5) 

The bin prices used during swaps are derived from the oracle at swap time. When the pool is paused because the oracle is stale or manipulated, the bin cursor reflects the pre-pause price. A new LP depositing during the pause deposits proportionally to that stale bin state. When the pool unpauses with a corrected oracle price, the bin prices shift, and arbitrageurs immediately execute swaps against the LP's newly deposited tokens at the corrected (lower or higher) price, draining the position.

The pause documentation confirms the intent: pausing "Disables swaps (and generally 'active' trading); use for incidents, upgrades, or market conditions." [7](#0-6) 

The `setPause` path is factory-gated and trusted, but the `addLiquidity` call path is fully permissionless — no role check exists. [8](#0-7) 

## Impact Explanation
When the pool unpauses after a price-disruption event, the oracle-derived bin prices shift to the corrected value. Arbitrageurs can immediately swap against the pool at the corrected price, extracting the newly deposited tokens. `binTotals.scaledToken0` / `scaledToken1` are updated by `LiquidityLib.addLiquidity` during the paused state, so the accounting is real and the tokens are genuinely at risk. The LP suffers a direct loss of deposited principal with no recourse. This is a Medium-severity direct loss of user principal triggered by an unprivileged caller on a core pool function.

## Likelihood Explanation
- Pause events are a normal operational action (admin at level 1, protocol at level 2).
- Any unprivileged user or automated LP bot can call `addLiquidity` at any time with no role requirement.
- Bots and routers that do not read `pauseLevel` before depositing will silently deposit into an unsafe pool.
- The attack window is the entire duration between a pause and its resolution — a realistic and repeatable condition.

## Recommendation
Apply `whenNotPaused` to `addLiquidity`, mirroring the protection already on `swap`:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {
```

`removeLiquidity` should intentionally remain unguarded so existing LPs can always exit a paused pool. [9](#0-8) 

## Proof of Concept
1. Deploy pool with a mutable `priceProvider`.
2. Admin calls `factory.pausePool(pool)` — `pauseLevel` becomes 1; `swap` now reverts with `PoolPaused`.
3. Alice calls `pool.addLiquidity(...)` depositing tokens into bin 0 (the active bin). The call succeeds; `binTotals.scaledToken0` increases; Alice's `_positionBinShares` are recorded.
4. Admin resolves the oracle issue and calls `factory.unpausePool(pool)` — `pauseLevel` returns to 0. The oracle now reports a corrected (materially different) price for token0.
5. Bob immediately calls `pool.swap(...)` buying token0 at the corrected lower price, draining Alice's deposited token0 and paying less token1 than Alice's deposit was worth.
6. Alice calls `removeLiquidity` and receives far less than she deposited — direct loss of principal.

Foundry test plan: fork or unit-test with a `MockPriceProvider` that returns price P1 at pause time and P2 (e.g., 10% lower) at unpause time; assert Alice's `removeLiquidity` proceeds after Bob's arbitrage swap yield less token value than Alice's `addLiquidity` cost.
