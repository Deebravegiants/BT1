Audit Report

## Title
Missing Domain Separation in Merkle Leaf Enables Cross-Chain Proof Replay — (`contracts/KERNEL/KernelTop100MerkleDistributor.sol`)

## Summary
`KernelTop100MerkleDistributor._verifyClaimProof` constructs the merkle leaf as `keccak256(abi.encodePacked(user, amount))` with no `address(this)` or `block.chainid` binding. Because the merkle root is fixed permanently at initialization with no `setMerkleRoot` function, any valid proof for `(user, amount)` on one chain is equally valid on every other chain where the same root is deployed, allowing a user to claim their full allocation on each chain independently.

## Finding Description
In `_verifyClaimProof` at line 293, the leaf is:

```solidity
bytes32 leaf = keccak256(abi.encodePacked(user, amount));
``` [1](#0-0) 

Neither `address(this)` nor `block.chainid` is included. The merkle root is set once at `initialize` and is immutable for the contract's lifetime — there is no `setMerkleRoot` function in this contract: [2](#0-1) 

The claim guard tracks `amountClaimed` only within the local `userClaims` mapping of a single contract instance: [3](#0-2) 

Because the leaf hash is identical on every chain, a proof `P` valid for `(alice, 1000e18)` on chain A is valid on chain B with the same root. The Arbitrum instance has its own independent `userClaims` storage where `amountClaimed[alice] == 0`, so the proof passes and tokens are transferred. The same structural gap exists in `KernelMerkleDistributor._processClaim` and `MerkleDistributor.claim`, both of which hash `(index, account, cumulativeAmount)` without chain or contract binding: [4](#0-3) [5](#0-4) 

## Impact Explanation
**High — Theft of unclaimed yield.**

KERNEL tokens distributed through these contracts are protocol yield/rewards. A user entitled to `X` KERNEL on chain A can replay the identical proof on chain B (and every additional chain where the same root is live) and receive `X` KERNEL per chain. The excess tokens are drawn from the contract's balance funded by the protocol, directly stealing yield that belongs to other recipients or the protocol treasury.

## Likelihood Explanation
LRT-rsETH is a multi-chain protocol with pool contracts deployed on Arbitrum, Optimism, Base, and others. Deploying the same KERNEL reward distribution (same merkle root) across multiple chains is a natural operational step for a cross-chain rewards campaign. The attacker requires only a legitimately obtained proof — no privileged access, no leaked keys, no front-running, and no victim mistakes. The attack is repeatable for every chain where the same root is deployed.

## Recommendation
Include `address(this)` and `block.chainid` in every merkle leaf to bind the proof to a specific contract and chain:

```solidity
// KernelTop100MerkleDistributor._verifyClaimProof
bytes32 leaf = keccak256(abi.encodePacked(address(this), block.chainid, user, amount));
```

Apply the same fix to `KernelMerkleDistributor._processClaim` and `MerkleDistributor.claim`:

```solidity
bytes32 node = keccak256(abi.encodePacked(address(this), block.chainid, index, account, cumulativeAmount));
```

Off-chain tree generation must be updated to hash leaves with the same domain prefix.

## Proof of Concept

1. Protocol generates a merkle tree for the Top-100 KERNEL distribution. Alice's leaf: `keccak256(abi.encodePacked(alice, 1000e18))`. Root = `R`.
2. Protocol deploys `KernelTop100MerkleDistributor` on Ethereum mainnet with root `R`, funded with 100 000 KERNEL.
3. Protocol deploys a second `KernelTop100MerkleDistributor` on Arbitrum with the **same root `R`**, funded with 100 000 KERNEL.
4. Alice calls `claim(1000e18, proof)` on Ethereum. Proof verifies; `userClaims[alice].amountClaimed = 1000e18`.
5. Alice calls `claim(1000e18, proof)` on Arbitrum with the **identical proof**. The Arbitrum contract's `userClaims[alice].amountClaimed == 0`; the leaf hash is identical; proof verifies. Alice receives another 1 000 KERNEL.
6. Alice has claimed 2 000 KERNEL instead of 1 000, stealing 1 000 KERNEL from the Arbitrum contract.

**Foundry fork test plan:** Fork Ethereum and Arbitrum mainnet states. Deploy `KernelTop100MerkleDistributor` on both forks with the same root. Generate a valid proof for Alice off-chain. Call `claim` on the Ethereum fork, assert success. Call `claim` on the Arbitrum fork with the same proof and same parameters, assert success and that Alice's balance increased by `2 * amount`.
