Audit Report

## Title
Stale `processChunk` Snapshot Inflates `unlockedWithdrawalsCount` via Race with `completeWithdrawal`, Permanently Freezing Protocol Assets — (`contracts/utils/UnlockedWithdrawalsInitializer.sol`)

## Summary
`processChunk` records a point-in-time count of unlocked withdrawal requests. If a user calls `completeWithdrawal` after `processChunk` has scanned a request but before `finalizeInitialize2` is called, the live `unlockedWithdrawalsCount` is correctly decremented to zero, but `initialize2` then overwrites it with the stale, higher snapshot value. Because `initialize2` uses `reinitializer(2)` it can never be called again, making the inflation permanent. `hasUnlockedWithdrawals` returns `true` forever and `sweepRemainingAssets` reverts permanently, freezing any residual protocol assets in the contract.

## Finding Description
`processChunk` iterates from `processedIndex[asset]` to `nextLockedNonce[asset]` and counts entries where `withdrawalRequests[requestId].expectedAssetAmount > 0`, accumulating the result in `unlockedCount[asset]`:

```
contracts/utils/UnlockedWithdrawalsInitializer.sol L97-106
```

`_processWithdrawalCompletion` (called by the permissionless `completeWithdrawal`) deletes the request struct — zeroing `expectedAssetAmount` — and decrements the live counter:

```
contracts/LRTWithdrawalManager.sol L712, L717
```

`finalizeInitialize2` reads the stale `unlockedCount` values and passes them to `initialize2`, which **sets** (not adds to) the live counters:

```
contracts/utils/UnlockedWithdrawalsInitializer.sol L117-121
contracts/LRTWithdrawalManager.sol L118-120
```

The `reinitializer(2)` guard on `initialize2` ensures this overwrite is permanent and cannot be corrected without a new upgrade:

```
contracts/LRTWithdrawalManager.sol L115
```

`completeWithdrawal` carries only `nonReentrant` and `whenNotPaused` guards — no `onlyBeforeInitialize2` guard exists — so it is freely callable by any user during the multi-block window between `processChunk` and `finalizeInitialize2`.

Exact race sequence:
1. One unlocked request exists for ETH (nonce 0, `expectedAssetAmount > 0`). `_unlockWithdrawalRequests` already set `unlockedWithdrawalsCount[ETH] = 1` and `nextLockedNonce[ETH] = 1`.
2. Operator calls `processChunk(ETH, 100)` → iterates nonce 0, finds `expectedAssetAmount > 0`, sets `unlockedCount[ETH] = 1`, `processedIndex[ETH] = 1`.
3. User calls `completeWithdrawal(ETH, "")` → `delete withdrawalRequests[requestId]` (zeroes `expectedAssetAmount`) → `unlockedWithdrawalsCount[ETH]-- = 0`. State is now correct.
4. Manager calls `finalizeInitialize2()` → `isAssetComplete` passes (`processedIndex[ETH] == nextLockedNonce[ETH] == 1`) → calls `initialize2(0, 0, 1)` → **SETS** `unlockedWithdrawalsCount[ETH] = 1`, overwriting the correct `0`.
5. No real unlocked requests remain, but `unlockedWithdrawalsCount[ETH]` is permanently `1`.

## Impact Explanation
After the inflated `initialize2` call, `unlockedWithdrawalsCount[asset]` is permanently one (or more) higher than the true number of remaining unlocked requests. Once all real requests are claimed, the counter never reaches zero. `hasUnlockedWithdrawals` permanently returns `true` (L629-631), and `sweepRemainingAssets` permanently reverts at the guard (L403). Any residual assets — rounding dust, direct ETH sends, or future small deposits — held by `LRTWithdrawalManager` are permanently frozen with no recovery path in the current code. This matches **Critical: Permanent freezing of funds**.

## Likelihood Explanation
The vulnerable window spans at least one block and typically many blocks in a multi-step upgrade procedure. Any user who holds an unlocked, delay-passed withdrawal request can trigger this by calling `completeWithdrawal` — a normal, permissionless production action requiring no admin access, front-running, or special knowledge. The condition is easily satisfied on mainnet where unlocked requests routinely exist during upgrade windows.

## Recommendation
Replace the SET semantics in `initialize2` with an ADD-TO approach so that completions processed before `finalizeInitialize2` are not overwritten. Alternatively, have `finalizeInitialize2` read the live `unlockedWithdrawalsCount` directly from the withdrawal manager at call time instead of relying on the stale `unlockedCount` accumulated by `processChunk`. A third option is to add an `onlyBeforeInitialize2` guard to `completeWithdrawal` (or pause the contract) during the initialization window to close the race entirely.

## Proof of Concept
```solidity
// Precondition: one unlocked ETH request (nonce 0) exists
// _unlockWithdrawalRequests already set unlockedWithdrawalsCount[ETH] = 1

// Step 1: Operator snapshots the count
initializer.processChunk(ETH, 100);
// unlockedCount[ETH] = 1, processedIndex[ETH] = 1

// Step 2: User completes their withdrawal (normal permissionless action)
withdrawalManager.completeWithdrawal(ETH, "");
// withdrawalRequests[requestId] deleted, unlockedWithdrawalsCount[ETH] = 0 ✓

// Step 3: Manager finalizes — passes stale count 1
initializer.finalizeInitialize2();
// initialize2(0, 0, 1) called → SETS unlockedWithdrawalsCount[ETH] = 1 ✗

// Step 4: Verify permanent inflation
assert(withdrawalManager.unlockedWithdrawalsCount(ETH) == 1);
assert(withdrawalManager.hasUnlockedWithdrawals(ETH) == true);

// Step 5: sweepRemainingAssets is permanently blocked
vm.expectRevert(PendingWithdrawalsExist.selector);
withdrawalManager.sweepRemainingAssets(ETH);
```
