### Title
`respond_verify_foreign_tx` Accepts Replayed Signatures via Missing `payload_hash`-to-`request` Binding — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies only that the submitted signature is cryptographically valid over the caller-supplied `response.payload_hash` using the root key. It never checks that `response.payload_hash` is the correct hash for the `request` argument. A single Byzantine attested participant can replay any previously observed `(payload_hash, signature)` pair against any pending `verify_foreign_transaction` request, permanently resolving that yield with fabricated verification data.

---

### Finding Description

**Vulnerability class:** Forged foreign-chain verification / cross-chain replay via missing response-to-request binding.

**Root cause — `respond_verify_foreign_tx` in `crates/contract/src/lib.rs`:**

```
// lines 718-734
let payload_hash: [u8; 32] = response.payload_hash.0;   // ← taken from the response, not derived from request

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,                                            // ← root key, no tweak
)
.is_ok()
```

The contract only checks: *"is `signature` a valid ECDSA signature over `payload_hash` under the root key?"* It never checks: *"does `payload_hash` equal `SHA-256(borsh(ForeignTxSignPayloadV1 { request, values }))`?"*

**Contrast with `respond()` (lines 597–608):**

```rust
let expected_public_key =
    derive_key_secp256k1(&affine, &request.tweak)...;   // key derived from request tweak
let payload_hash = request.payload.as_ecdsa()...;       // hash taken FROM the request, not the response
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response, payload_hash, &expected_public_key,
)
```

In `respond()`, the payload hash is extracted from the immutable `request` struct and the key is derived from the request's tweak — both are bound to the specific request. In `respond_verify_foreign_tx()`, both the hash and the key are decoupled from the request.

**`VerifyForeignTransactionRequest` has no tweak or caller binding:**

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs lines 124-128
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
    // ← no tweak, no predecessor, no caller binding
}
```

The node-side signing uses a zero tweak (`Tweak::new([0u8; 32])`) confirming the root key is used:

```rust
// crates/node/src/providers/verify_foreign_tx/sign.rs lines 43
tweak: Tweak::new([0u8; 32]),
```

This means every `verify_foreign_transaction` signature is produced under the same root key with no per-request key derivation, making all past signatures reusable against any future request.

---

### Impact Explanation

**High — Forged foreign-chain verification enabling invalid bridge execution.**

Attack steps (single Byzantine attested participant, no threshold collusion):

1. Observe any past `respond_verify_foreign_tx` call on-chain for request `R1` with response `(H1, sig_H1)` where `H1 = SHA-256(borsh(ForeignTxSignPayloadV1 { request: R1, values: V1 }))`.
2. A new `verify_foreign_transaction` request `R2` is pending (e.g., a bridge contract waiting to release funds for Bitcoin tx `X`).
3. Byzantine node calls `respond_verify_foreign_tx(request=R2, response=(H1, sig_H1))`.
4. Contract checks: `verify_ecdsa_signature(sig_H1, H1, root_key)` → **valid** (signature is genuine, just for a different request).
5. Contract calls `resolve_yields_for` and permanently resolves the yield for `R2` with `(H1, sig_H1)`.
6. Caller of `R2` receives `VerifyForeignTransactionResponse { payload_hash: H1, signature: sig_H1 }` — a valid MPC signature, but over the wrong payload.

The yield for `R2` is now permanently consumed. The legitimate response for `R2` can never be delivered. Any bridge contract that trusts the `payload_hash` in the response (without independently recomputing it from the request) will act on fabricated verification data. Even callers using `ForeignChainSignatureVerifier::verify_signature` from the SDK will have their request permanently poisoned — they receive an `IncorrectPayloadSigned` error and their deposit is lost with no recourse.

---

### Likelihood Explanation

**Medium.** The attacker must be a single TEE-attested MPC participant (below the signing threshold). No key forgery or threshold collusion is needed — the replayed signature is a genuine MPC signature, just for a different request. Past responses are permanently visible on-chain. The attacker only needs to call `respond_verify_foreign_tx` before the honest leader does, which is straightforward given that any attested participant can call this method (no leader-only restriction is enforced by the contract).

---

### Recommendation

In `respond_verify_foreign_tx`, recompute the expected `payload_hash` from the `request` argument on-chain and reject any response whose `payload_hash` does not match. Since `ForeignTxSignPayloadV1` only requires `request` and `values` (the extracted values), the contract cannot recompute the hash without the `values`. The fix is to include the `values` in the `VerifyForeignTransactionResponse` and have the contract compute and verify `SHA-256(borsh(ForeignTxSignPayloadV1 { request, values })) == response.payload_hash` before accepting the response. This mirrors how `respond()` takes the payload hash from the immutable request rather than from the response.

### Proof of Concept

```
// Setup: R1 was previously verified for Bitcoin tx_id=[0x01;32], block_hash=[0xAA;32]
// H1 = SHA-256(borsh(ForeignTxSignPayloadV1 { request: R1, values: [BlockHash([0xAA;32])] }))
// sig_H1 = root_key.sign(H1)  ← this is on-chain from a past respond_verify_foreign_tx call

// Victim: R2 is pending for Bitcoin tx_id=[0x02;32] (a bridge contract waiting to release funds)
// Correct response would be H2 = SHA-256(borsh(ForeignTxSignPayloadV1 { request: R2, values: [BlockHash([0xBB;32])] }))

// Attack: Byzantine node calls respond_verify_foreign_tx with replayed response
respond_verify_foreign_tx(
    request = R2,                          // the pending victim request
    response = { payload_hash: H1,         // hash from R1, not R2
                 signature: sig_H1 }       // valid root-key signature over H1
)

// Contract check (lines 729-734):
//   verify_ecdsa_signature(sig_H1, H1, root_key) → OK  ← passes, sig is genuine
//   payload_hash == expected_hash_for_R2?          ← NEVER CHECKED

// Result: yield for R2 resolved with (H1, sig_H1)
// Bridge contract receives payload_hash=H1 (Bitcoin tx [0x01;32] block hash [0xAA;32])
// instead of payload_hash=H2 (Bitcoin tx [0x02;32] block hash [0xBB;32])
// Bridge contract may release funds based on wrong foreign-chain state
```

/// Resume every yield queued for `request` with `response_bytes`, draining the
/// fan-out map in one pass. Returns `Err(RequestNotFound)` if the map held no entry.
///
/// Resuming a yield that has already timed out is a no-op at the SDK level.
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
