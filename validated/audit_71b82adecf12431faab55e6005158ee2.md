### Title
Expired DRep Can Re-Activate Itself by Submitting a Vote, Bypassing the DRep Activity Requirement — (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Certs.hs`)

---

### Summary

`updateVotingDRepExpiries` in `Certs.hs` unconditionally refreshes the expiry of every DRep that casts a vote in a transaction, without first checking whether that DRep is already expired. The `GOV` rule's voter-validity checks do not verify DRep expiry either. As a result, an expired DRep can submit a vote, have it accepted, and have its expiry silently reset — re-entering the active governance set and influencing ratification in the following epoch.

---

### Finding Description

**Root cause — `updateVotingDRepExpiries` (Certs.hs lines 272–292)**

```haskell
updateVotingDRepExpiries tx currentEpoch drepActivity certState =
  let numDormantEpochs = certState ^. certVStateL . vsNumDormantEpochsL
      updateVSDReps vsDReps =
        Map.foldlWithKey'
          ( \dreps voter _ -> case voter of
              DRepVoter cred ->
                Map.adjust
                  (drepExpiryL .~ computeDRepExpiry drepActivity currentEpoch numDormantEpochs)
                  cred
                  dreps
              _ -> dreps
          )
          vsDReps
          (unVotingProcedures $ tx ^. bodyTxL . votingProceduresTxBodyL)
   in certState & certVStateL . vsDRepsL %~ updateVSDReps
``` [1](#0-0) 

`Map.adjust` updates the `drepExpiry` field for any DRep credential found in `vsDReps`. Expired DReps are **not removed** from `vsDReps`; they remain registered with a past expiry epoch. Therefore `Map.adjust` succeeds and overwrites their expiry with a fresh future value, with no staleness guard.

**Contrast with `updateDormantDRepExpiry` (Certs.hs lines 308–328)**

The sibling function that handles dormant-epoch bumps explicitly refuses to update an already-expired DRep:

```haskell
updateExpiry =
  drepExpiryL
    %~ \currentExpiry ->
        let actualExpiry = binOpEpochNo (+) numDormantEpochs currentExpiry
         in if actualExpiry < currentEpoch
              then currentExpiry   -- ← guard: skip if already expired
              else actualExpiry
``` [2](#0-1) 

No equivalent guard exists in `updateVotingDRepExpiries`.

**`GOV` rule does not check DRep expiry (Gov.hs lines 364–376)**

```haskell
checkVotersAreValid currentEpoch committeeState votes =
  checkDisallowedVotes votes DisallowedVoters $ \gas ->
    \case
      CommitteeVoter {} -> isCommitteeVotingAllowed currentEpoch committeeState (gasAction gas)
      DRepVoter {}      -> isDRepVotingAllowed (gasAction gas)   -- ← action-type only, no expiry check
      StakePoolVoter {} -> isStakePoolVotingAllowed (gasAction gas)
``` [3](#0-2) 

`isDRepVotingAllowed` checks only the governance action type. The `VotersDoNotExist` check verifies membership in `knownDReps = vsDReps certVState`; expired DReps remain in that map. Neither check tests whether `currentEpoch > drepExpiry`. [4](#0-3) 

**Execution order in `LEDGER` (Ledger.hs lines 383–421)**

`updateVotingDRepExpiries` is called before the `GOV` rule, so the expiry is refreshed before the vote is recorded in `gasDRepVotes`. [5](#0-4) 

**Ratification uses a one-epoch-old snapshot**

`dRepAcceptedRatio` checks `reCurrentEpoch > drepExpiry drepState` against `reDRepState`, which is the pulser snapshot from the previous epoch boundary. [6](#0-5) 

So the re-activated DRep's refreshed expiry is invisible to ratification at the end of epoch N, but is fully visible at the end of epoch N+1, when the pulser snapshot captures the updated `drepExpiry`.

**Known discrepancy acknowledged in the test suite**

The test `disableInConformanceIt "expiry is not updated for inactive DReps"` in `EpochSpec.hs` explicitly expects that expired DReps should NOT have their expiry updated, but is disabled pending resolution of formal-spec issue #923. [7](#0-6) 

---

### Impact Explanation

An expired DRep — one that the protocol has designated as inactive and excluded from governance — can unilaterally re-enter the active DRep set by submitting a single vote transaction. In the epoch following the vote, the DRep's refreshed expiry is captured in the pulser snapshot, and the DRep's stake is counted in `dRepAcceptedRatio`. If the DRep controls enough delegated stake, this can tip the DRep acceptance ratio above or below a threshold, causing a governance action (protocol-parameter change, hard-fork initiation, treasury withdrawal, committee update, or constitution change) to be ratified or blocked contrary to the intent of the DRep activity mechanism.

**Impact category:** Critical — unauthorized governance action enacted through an invalid ledger state transition (expired DRep treated as active).

---

### Likelihood Explanation

The attack requires only a registered-but-expired DRep credential and any live governance proposal to vote on. No privileged access, key compromise, or majority collusion is needed. The DRep simply submits a standard `VotingProcedures` transaction. The cost is one transaction fee. Any DRep operator who has let their registration lapse can exploit this to re-enter governance without re-registering or paying a new deposit.

---

### Recommendation

Add an expiry guard to `updateVotingDRepExpiries` mirroring the one already present in `updateDormantDRepExpiry`:

```haskell
DRepVoter cred ->
  Map.adjust
    ( \drepState ->
        let actualExpiry =
              binOpEpochNo (+) numDormantEpochs (drepExpiry drepState)
         in if actualExpiry < currentEpoch
              then drepState  -- DRep is expired; do not refresh
              else drepState & drepExpiryL
                    .~ computeDRepExpiry drepActivity currentEpoch numDormantEpochs
    )
    cred
    dreps
```

This ensures that only DReps whose actual expiry (raw expiry + dormant epochs) has not yet passed receive an expiry refresh when they vote, consistent with the intent of the DRep activity mechanism and the expected behavior documented in the disabled conformance test.

### Proof of Concept

```
1. Set ppDRepActivityL = EpochInterval 2, ppGovActionLifetimeL = EpochInterval 5
2. Register DRep D with significant delegated stake.
3. Submit a governance proposal P (e.g., ParameterChange).
4. Pass 3 epochs without D voting → D is expired
   (drepExpiry D < currentEpoch + numDormantEpochs).
5. Submit a transaction containing VotingProcedures { DRepVoter D → VoteYes on P }.
6. Observe:
   a. GOV rule accepts the vote (no DisallowedVoters failure; D is still in vsDReps).
   b. updateVotingDRepExpiries sets D's drepExpiry to
      currentEpoch + drepActivity - numDormantEpochs (a future epoch).
7. At the next epoch boundary, the pulser snapshot captures D's refreshed expiry.
8. At the epoch boundary after that, dRepAcceptedRatio includes D's stake in the
   denominator and numerator (if VoteYes), potentially pushing the ratio above the
   DRep voting threshold and ratifying P — a governance action that would not have
   been ratified without D's illegitimate re-activation.
```

```haskell
dRepAcceptedRatio RatifyEnv {reDRepDistr, reDRepState, reCurrentEpoch} gasDRepVotes govAction =
  toInteger yesStake %? toInteger totalExcludingAbstainStake
  where
    accumStake (!yes, !tot) drep (CompactCoin stake) =
      case drep of
        DRepCredential cred ->
          case Map.lookup cred reDRepState of
            Nothing -> (yes, tot) -- drep is not registered, so we don't consider it
            Just drepState
              | reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, so we don't consider it
              | otherwise ->
                  case Map.lookup cred gasDRepVotes of
                    -- drep hasn't voted for this action, so we don't count
                    -- the vote but we consider it in the denominator:
                    Nothing -> (yes, tot + stake)
                    Just VoteYes -> (yes + stake, tot + stake)
                    Just Abstain -> (yes, tot)
                    Just VoteNo -> (yes, tot + stake)
        DRepAlwaysNoConfidence ->
          case govAction of
            NoConfidence _ -> (yes + stake, tot + stake)
            _ -> (yes, tot + stake)
        DRepAlwaysAbstain -> (yes, tot)
    (yesStake, totalExcludingAbstainStake) = Map.foldlWithKey' accumStake (0, 0) reDRepDistr
