### Title
Trusted Relayer Can Redirect Fees on Re-Routed Transfers via Unbound `fee_recipient` in `sign_transfer` — (`near/omni-bridge/src/lib.rs`)

---

### Summary

`process_fin_transfer_to_other_chain` stores a pending `TransferMessage` with the calling relayer as the storage `owner` only for accounting purposes. It does not bind the fee recipient. `sign_transfer` accepts `fee_recipient: Option<AccountId>` directly from the caller with no check that the caller is the relayer who created the pending transfer. Any trusted relayer can therefore call `sign_transfer` on any pending transfer and embed their own account as `fee_recipient` in the MPC-signed payload, causing the fee to be paid to them on the destination chain.

---

### Finding Description

**Step 1 — `fin_transfer` creates a pending transfer with no fee_recipient binding**

`fin_transfer` is gated by `#[trusted_relayer]` and passes `env::predecessor_account_id()` (the calling relayer) to `fin_transfer_callback`: [1](#0-0) 

`fin_transfer_callback` constructs a `TransferMessage` and, for non-NEAR destinations, calls `process_fin_transfer_to_other_chain(predecessor_account_id, transfer_message)`: [2](#0-1) 

Inside `process_fin_transfer_to_other_chain`, the `predecessor_account_id` is stored only as the storage `owner` for balance accounting. No `fee_recipient` field is recorded anywhere in the stored `TransferMessage`: [3](#0-2) 

The `TransferMessageStorageValue` struct confirms the stored value contains only `message` and `owner` — no `fee_recipient`: [4](#0-3) 

**Step 2 — `sign_transfer` accepts arbitrary `fee_recipient` from any trusted relayer**

`sign_transfer` is also gated by `#[trusted_relayer]` (permissionlessly achievable by staking NEAR). It reads the stored `TransferMessage` but performs no check that the caller is the relayer who created the pending transfer. The `fee_recipient` parameter is taken directly from the caller and embedded verbatim into the MPC-signed payload: [5](#0-4) 

There is no `require!(env::predecessor_account_id() == transfer_owner, ...)` guard anywhere in `sign_transfer`.

**Step 3 — The signed payload carries `fee_recipient` to the destination chain**

The `TransferMessagePayload` with the attacker-controlled `fee_recipient` is hashed and sent to the MPC signer. The resulting signature is broadcast in a `SignTransferEvent`. On the destination chain (e.g., EVM), `fin_transfer` verifies the MPC signature and pays the `fee_recipient` embedded in the payload. The attacker's account receives the fee.

**Step 4 — `sign_transfer_callback` does not remove the transfer for fee > 0**

For transfers with non-zero fee, the transfer message is not removed after signing, meaning multiple competing `sign_transfer` calls can produce multiple valid signatures. Only one can be used on the destination chain (nonce protection), but the attacker who front-runs gets the fee: [6](#0-5) 

---

### Impact Explanation

This is a **fee-routing divergence** that sends value to the wrong party. Any trusted relayer can steal the fee from any pending transfer created by `process_fin_transfer_to_other_chain` by calling `sign_transfer` with `fee_recipient` set to themselves. The original transfer recipient is unaffected (they receive `amount_without_fee`), but the relayer who performed the `fin_transfer` work loses their fee to the attacker.

Scoped impact: **High — fee-routing divergence that sends value to the wrong party.**

---

### Likelihood Explanation

Becoming a trusted relayer is permissionless (stake NEAR, wait the activation period). Any economically motivated actor operating as a trusted relayer can monitor the chain for `FinTransferEvent` logs emitted by `process_fin_transfer_to_other_chain` and immediately call `sign_transfer` with `fee_recipient=self` before the original relayer does. This is a straightforward front-running attack on NEAR, where transaction ordering is observable.

---

### Recommendation

Bind the `fee_recipient` at the time the pending transfer is created. Two options:

1. **Store the fee_recipient at creation time**: In `process_fin_transfer_to_other_chain`, record `predecessor_account_id` as the `fee_recipient` inside the stored `TransferMessage` (add a `fee_recipient: Option<AccountId>` field). In `sign_transfer`, require that the provided `fee_recipient` matches the stored value (or default to it if `None`).

2. **Restrict `sign_transfer` to the transfer owner**: Add a check `require!(env::predecessor_account_id() == stored_transfer.owner, ...)` so only the relayer who paid for storage (and created the pending transfer) can sign it.

---

### Proof of Concept

```
1. Trusted relayer A calls fin_transfer for a chain-A → chain-B (non-NEAR) transfer with fee > 0.
2. process_fin_transfer_to_other_chain stores TransferMessage with owner=A, no fee_recipient.
3. Trusted relayer B (attacker) observes the FinTransferEvent on-chain.
4. B calls sign_transfer(transfer_id, fee_recipient=Some(B_account), fee=None).
   - #[trusted_relayer] passes (B is staked).
   - No check that B == original relayer.
   - TransferMessagePayload is built with fee_recipient=B_account.
5. MPC signs the payload embedding B_account as fee_recipient.
6. B submits the signature to chain B's fin_transfer.
7. Chain B pays the fee to B_account instead of A.
8. A receives nothing for their fin_transfer work.
```

Integration test assertion: after step 7, `balance(A_fee_account_on_B)` is unchanged and `balance(B_account_on_B)` increased by the fee amount.

    #[payable]
    #[trusted_relayer]
    #[pause(except(roles(Role::DAO)))]
    pub fn sign_transfer(
        &mut self,
        transfer_id: TransferId,
        fee_recipient: Option<AccountId>,
        fee: &Option<Fee>,
    ) -> Promise {
        let transfer_message = self.get_transfer_message(transfer_id);

        if let Some(fee) = &fee {
            require!(
                &transfer_message.fee == fee,
                BridgeError::InvalidFee.as_ref()
            );
        }

        let token_address = self
            .get_token_address(
                transfer_message.get_destination_chain(),
                self.get_token_id(&transfer_message.token),
            )
            .unwrap_or_else(|| {
                env::panic_str(BridgeError::FailedToGetTokenAddress.to_string().as_str())
            });

        let decimals = self
            .token_decimals
            .get(&token_address)
            .near_expect(BridgeError::TokenDecimalsNotFound);
        let amount_to_transfer = Self::normalize_amount(
            transfer_message
                .amount_without_fee()
                .near_expect(BridgeError::InvalidFee),
            decimals,
        );

        require!(
            amount_to_transfer > 0,
            BridgeError::InvalidAmountToTransfer.as_ref()
        );

        let message = DestinationChainMsg::from_json(&transfer_message.msg)
            .and_then(|s| s.destination_msg())
            .unwrap_or_default();

        let transfer_payload = TransferMessagePayload {
            prefix: PayloadType::TransferMessage,
            destination_nonce: transfer_message.destination_nonce,
            transfer_id,
            token_address,
            amount: U128(amount_to_transfer),
            recipient: transfer_message.recipient,
            fee_recipient,
            message,
        };
