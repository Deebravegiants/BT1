### Title
Missing `owner != address(0)` Guard in `MetricOmmPool.addLiquidity()` Permanently Locks LP Tokens — (`metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`MetricOmmPool.addLiquidity()` accepts an arbitrary `owner` address with no zero-address check. If `owner == address(0)`, LP shares are minted to the `(address(0), salt, binIdx)` storage key and the deposited tokens are permanently locked, because `removeLiquidity()` enforces `msg.sender == owner`, which can never be satisfied for `address(0)`. The periphery `MetricOmmPoolLiquidityAdder` carries a `_validateOwner()` guard, but the core pool is a public contract callable directly, making that guard bypassable.

---

### Finding Description

`MetricOmmPool.addLiquidity()` performs two input checks before delegating to `LiquidityLib`:

```solidity
function addLiquidity(
    address owner,          // ← never validated ≠ address(0)
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
    if (deltas.binIdxs.length == 0) return (0, 0);
    if (deltas.binIdxs.length != deltas.shares.length) revert LiquidityDeltaLengthMismatch();
    ...
}
``` [1](#0-0) 

Neither check rejects `owner == address(0)`. The call proceeds: `LiquidityLib.addLiquidity` mints shares into `_positionBinShares` keyed by `(owner, salt, binIdx)`, and the caller's tokens are pulled via the modify-liquidity callback into the pool.

The only exit path for those tokens is `removeLiquidity()`:

```solidity
function removeLiquidity(address owner, ...) external ... {
    ...
    if (msg.sender != owner) revert NotPositionOwner();
    ...
}
``` [2](#0-1) 

Because `msg.sender` can never equal `address(0)` in a normal EVM transaction, the `NotPositionOwner` revert fires unconditionally for any attempt to remove a position whose `owner` is `address(0)`. The deposited tokens are irrecoverable.

The periphery `MetricOmmPoolLiquidityAdder` does guard against this via `_validateOwner(owner)`: [3](#0-2) 

The periphery test confirms the guard exists there: [4](#0-3) 

However, `MetricOmmPool` is a standalone public contract. Any caller — including other contracts, integrators, or users — can invoke `addLiquidity()` directly, bypassing the periphery entirely.

---

### Impact Explanation

A caller who passes `owner == address(0)` to `MetricOmmPool.addLiquidity()` directly:

1. Pays real token0 and/or token1 into the pool via the modify-liquidity callback.
2. Receives LP shares credited to the `(address(0), salt, binIdx)` storage key.
3. Can never recover those tokens: `removeLiquidity(address(0), ...)` always reverts with `NotPositionOwner`.

The deposited principal is permanently locked inside the pool. `binTotals.scaledToken0` / `scaledToken1` are incremented, so the pool's accounting remains consistent, but the underlying tokens are unclaimable. This is a direct, irreversible loss of user principal.

---

### Likelihood Explanation

- The core pool is permissionless and callable by any EOA or contract.
- The periphery guard is not enforced at the pool level.
- Integrators building custom routers or contracts that wrap `addLiquidity()` may omit the zero-address check, especially since the pool's own interface (`IMetricOmmPoolActions`) documents no such restriction.
- A single mistaken or malicious call is sufficient to trigger the loss.

Likelihood is **low** (requires a caller to pass `address(0)` as `owner`), but impact is **high** (permanent, unrecoverable loss of deposited tokens). This maps to **Medium** severity under Sherlock thresholds.

---

### Recommendation

Add an explicit zero-address guard at the top of `MetricOmmPool.addLiquidity()`, mirroring the check already present in the periphery:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {
    if (owner == address(0)) revert InvalidPositionOwner();   // ← add this
    if (deltas.binIdxs.length == 0) return (0, 0);
    if (deltas.binIdxs.length != deltas.shares.length) revert LiquidityDeltaLengthMismatch();
    ...
}
```

This ensures the invariant is enforced at the protocol layer regardless of which entry point is used.

---

### Proof of Concept

```solidity
// Attacker (or any caller) calls the core pool directly, bypassing the periphery guard.
// Assume pool is a deployed MetricOmmPool, token0/token1 are standard ERC20s.

// 1. Caller approves pool to pull tokens (or implements the callback).
// 2. Caller calls addLiquidity with owner = address(0).
pool.addLiquidity(
    address(0),          // owner — zero address, no revert
    uint80(0),           // salt
    deltas,              // some valid bin/share delta
    callbackData,        // callback pays token0/token1 into pool
    ""
);

// 3. Tokens are now in the pool. binTotals updated. Shares at key (address(0), 0, binIdx).

// 4. Attempt to recover — always reverts:
pool.removeLiquidity(
    address(0),          // owner
    uint80(0),
    deltas,
    ""
);
// → NotPositionOwner (msg.sender != address(0) always)

// Tokens are permanently locked.
```
