### Title
`validatePerasCert` Hardcoded to Always Return `True`, Bypassing Peras Certificate Validation in BBODY Rule — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/BlockBody/Internal.hs`)

---

### Summary

In the Dijkstra era, the `validatePerasCert` function used inside the production `BBODY` transition rule is a mocked-up placeholder that unconditionally returns `True`, ignoring all inputs. Any block producer can embed an arbitrary, cryptographically invalid `PerasCert` in a block body, and every honest node will accept it. This is the direct Cardano analog of `MagicLpAggregator.latestRoundData()` returning hardcoded zeros: a function that is supposed to compute a meaningful result instead returns a fixed value, causing downstream validation to behave incorrectly.

---

### Finding Description

`validatePerasCert` is defined in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/BlockBody/Internal.hs`:

```haskell
-- | Mocked-up Peras certificate validation routine
--
-- NOTE: this function will be replaced with the real implementation from
-- 'cardano-base' once it's ready.
validatePerasCert :: Nonce -> PerasKey -> PerasCert -> Bool
validatePerasCert _ _ _ = True
``` [1](#0-0) 

The companion "key" type is also a unit placeholder with no actual cryptographic material:

```haskell
data PerasKey = PerasKey
  deriving (Eq, Show, Generic, NoThunks)
``` [2](#0-1) 

This function is called directly inside the production `dijkstraBbodyTransition` rule in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs`:

```haskell
case blockBody ^. perasCertBlockBodyL of
  SNothing -> pure ()
  SJust cert ->
    let nonce = block ^. prevNonceBlockHeaderL
     in validatePerasCert nonce PerasKey cert ?! injectFailure (PerasCertValidationFailed cert nonce)
``` [3](#0-2) 

The `?!` STS operator emits `PerasCertValidationFailed` only when the predicate is `False`. Because `validatePerasCert _ _ _ = True` is unconditional, the failure branch is **unreachable**. The `PerasCert` type is a raw `ByteArray` newtype with no structural constraints:

```haskell
newtype PerasCert = PerasCert ByteArray
  deriving (Eq, Show, Generic)
  deriving newtype (EncCBOR, DecCBOR)
``` [4](#0-3) 

A block producer can therefore serialize any arbitrary byte sequence as a `PerasCert`, include it in a `DijkstraBlockBody`, and every node running the Dijkstra era will accept the block unconditionally.

The `DijkstraBlockBodyRaw` struct shows that `dbbrPerasCert` is a first-class field of the block body, included in the block body hash and serialized on-chain: [5](#0-4)

### Impact Explanation

**Classification: Critical — Honest nodes accept an invalid block, with potential for permanent ledger divergence requiring a hard fork.**

1. Any block producer (no special privilege required beyond normal block production rights) can embed a cryptographically invalid `PerasCert` in a Dijkstra-era block.
2. All honest nodes running the current production code accept the block, because `validatePerasCert` always returns `True`.
3. Invalid Peras certificates are permanently committed to the chain.
4. When `cardano-base` delivers the real `validatePerasCert` implementation and it is integrated, nodes running the updated code will reject blocks that the current code accepted, causing a deterministic split between updated and non-updated nodes — a ledger divergence that requires a hard fork to resolve.
5. Additionally, the Peras fast-finality protocol itself is undermined: an adversary can inject arbitrary certificate data, potentially manipulating the Peras finality signal seen by downstream consumers.

---

### Likelihood Explanation

- The Dijkstra era is the active development era and will be deployed to mainnet.
- The entry path requires only that a block producer include a `PerasCert` field in a block body — a standard serialization operation with no threshold or key requirement.
- The placeholder is explicitly documented as temporary, meaning it is present in the production codebase by design, but the security gap it creates is not bounded by any other check.
- No governance majority, leaked key, or Sybil attack is required.

---

### Recommendation

1. **Do not ship `validatePerasCert _ _ _ = True` in a mainnet-deployed era.** Either gate the Peras certificate field behind a protocol version check that is only activated once the real implementation is available, or replace the placeholder with the real cryptographic validation from `cardano-base` before the Dijkstra era activates on mainnet.
2. Replace the `PerasKey = PerasKey` unit type with the actual public key type from `cardano-base` so that the validation function has real key material to work with.
3. Add a property-based test asserting that `validatePerasCert` rejects at least one invalid certificate, to prevent future regressions where a placeholder silently replaces a real implementation.

---

### Proof of Concept

The attack path is entirely mechanical:

1. Construct a `DijkstraBlockBody` with `dbbrPerasCert = SJust (PerasCert someArbitraryBytes)` for any `someArbitraryBytes`.
2. Submit the block to the network.
3. The `dijkstraBbodyTransition` rule evaluates `validatePerasCert nonce PerasKey cert`, which returns `True` unconditionally.
4. The `?!` operator does not fire; no `PerasCertValidationFailed` is emitted.
5. The block is accepted and the invalid certificate is permanently recorded in the block body hash.

The `PerasCertValidationFailed` constructor exists in `DijkstraBbodyPredFailure` and is fully serializable, confirming the intent was to reject invalid certificates — but the guard is permanently disabled by the hardcoded return value. [6](#0-5)
