Audit Report

## Title
`RsETHTokenWrapper` 1:1 cross-token mint/burn with no per-token accounting enables cross-token arbitrage theft when multiple altRsETH tokens are active - (File: contracts/L2/RsETHTokenWrapper.sol)

## Summary
`RsETHTokenWrapper` supports multiple allowed altRsETH tokens via `allowedTokens` and mints/burns wrsETH at a strict 1:1 ratio against any of them, with no exchange rate check and no restriction on which token a user may withdraw. Once a second altRsETH token is added via the documented `reinitialize` upgrade path, any unprivileged user can deposit the cheaper token, receive wrsETH 1:1, and immediately withdraw the more valuable token 1:1, extracting the price difference from other depositors. `maxAmountToDepositBridgerAsset` compounds the issue by comparing the total wrsETH supply against the balance of only one specific token, inflating the deposit ceiling by the supply backed by all other tokens.

## Finding Description
The contract explicitly provisions for multiple allowed tokens:

- `initialize` registers the first altRsETH token.
- `reinitialize` (line 47, `DEFAULT_ADMIN_ROLE`) adds a second altRsETH token — this is the documented upgrade path.
- `addAllowedToken` (line 174, `TIMELOCK_ROLE`) can add further tokens.

`_deposit` (lines 134–141) mints exactly `_amount` wrsETH for exactly `_amount` of any allowed token with no exchange rate consulted:

```solidity
ERC20Upgradeable(_asset).safeTransferFrom(msg.sender, address(this), _amount);
_mint(_to, _amount);
```

`_withdraw` (lines 120–128) burns exactly `_amount` wrsETH and transfers exactly `_amount` of **any caller-chosen** allowed token:

```solidity
_burn(msg.sender, _amount);
ERC20Upgradeable(_asset).safeTransfer(_to, _amount);
```

The public `withdraw` and `withdrawTo` functions impose no restriction on which allowed asset is selected. If two allowed tokens trade at different prices (a realistic condition for bridge representations of the same underlying asset), the wrapper becomes a zero-cost swap venue.

`maxAmountToDepositBridgerAsset` (lines 99–110) uses `totalSupply()` — the aggregate wrsETH supply across all altRsETH tokens — but compares it against the balance of only one specific token:

```solidity
uint256 wrsETHSupply = totalSupply();          // ALL wrsETH, from ALL altRsETH tokens
uint256 balanceOfAssetInWrapper = ERC20Upgradeable(_asset).balanceOf(address(this));
return wrsETHSupply - balanceOfAssetInWrapper; // inflated by wrsETH minted for other tokens
```

With 100 wrsETH minted for altRsETH_A and 50 wrsETH minted for altRsETH_B, querying for altRsETH_A returns `150 − 100 = 50`, implying 50 more altRsETH_A can be bridged in — but those 50 wrsETH were already issued against altRsETH_B, breaking the collateralisation invariant.

The identical pattern exists in `contracts/agETH/AGETHTokenWrapper.sol` (lines 90–101, 111–119, 125–131), though that contract currently has no `addAllowedToken` or `reinitialize` function (line 153 comment: "Dont' allow to add other tokens at the moment"), limiting the immediate exploitability there to the single-token case.

## Impact Explanation
**Critical — direct theft of user funds.**

Once two altRsETH tokens are active and trade at any non-zero price differential, an unprivileged attacker can atomically deposit the cheaper token and withdraw the more valuable token, extracting the price difference from other depositors. The attack requires no special role, no flash loan, and no victim mistake — only the protocol being in its intended multi-token state. The stolen value comes directly from other users' deposited collateral.

## Likelihood Explanation
**Medium.**

The precondition is the admin calling `reinitialize` to add a second altRsETH token, which is the explicitly documented upgrade path for the contract (the function exists solely for this purpose). Different bridge representations of the same underlying asset (e.g., LayerZero OFT vs. native bridge) routinely trade at small but non-zero discounts/premiums due to bridge risk, liquidity differences, and redemption delays. Once the second token is added — a routine governance action — the vulnerability is immediately and repeatedly exploitable by any user with no further preconditions.

## Recommendation
1. **Track per-token accounting**: maintain a `mapping(address => uint256) public wrsETHMintedFor` incremented in `_deposit` and decremented in `_withdraw`. Use this per-token balance (not `totalSupply()`) in `maxAmountToDepositBridgerAsset`.
2. **Enforce same-token withdrawal or introduce an oracle**: either require that wrsETH can only be redeemed for the same altRsETH token it was minted against, or introduce an on-chain exchange rate between allowed tokens before permitting cross-token redemption.
3. Apply the same fix to `AGETHTokenWrapper` proactively, before any second token is added.

## Proof of Concept
**Setup**: Admin calls `reinitialize(altRsETH_B)` to add a second allowed token alongside the original `altRsETH_A`. Suppose `altRsETH_B` trades at 1.05× the value of `altRsETH_A`.

**Step 1 — Victim deposits the valuable token:**
```
Alice calls deposit(altRsETH_B, 100e18)
→ wrapper receives 100e18 altRsETH_B
→ Alice receives 100e18 wrsETH
```

**Step 2 — Attacker deposits the cheap token:**
```
Bob calls deposit(altRsETH_A, 100e18)
→ wrapper receives 100e18 altRsETH_A
→ Bob receives 100e18 wrsETH
```

**Step 3 — Attacker withdraws the valuable token:**
```
Bob calls withdraw(altRsETH_B, 100e18)
→ burns 100e18 wrsETH
→ Bob receives 100e18 altRsETH_B (worth 105e18 altRsETH_A equivalent)
```

**Result**: Bob spent 100e18 altRsETH_A and received 100e18 altRsETH_B, netting a 5 ETH-equivalent profit. Alice's 100e18 altRsETH_B is gone; the wrapper holds only 100e18 altRsETH_A against Alice's 100e18 wrsETH.

**Foundry invariant test plan**: Deploy `RsETHTokenWrapper` with two mock ERC20 tokens at different prices. Fuzz `deposit`/`withdraw` call sequences across arbitrary callers. Assert invariant: `sum over all allowed tokens of (token.balanceOf(wrapper)) * price[token] >= totalSupply() * price[canonical]`. The invariant will break as soon as a cross-token withdrawal is executed.
