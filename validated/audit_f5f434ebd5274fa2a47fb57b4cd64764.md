Audit Report

## Title
Stale `rsETHPrice` Used in `getRsETHAmountToMint` Allows Depositors to Receive Excess rsETH at the Expense of Existing Holders - (File: contracts/LRTDepositPool.sol)

## Summary

`LRTOracle` stores the rsETH/ETH exchange rate in a public state variable `rsETHPrice` that is only updated when `updateRSETHPrice()` is explicitly called by an off-chain keeper. Neither `depositETH()` nor `depositAsset()` in `LRTDepositPool` refreshes this price before computing the mint amount. When staking rewards accrue between keeper calls, the stored price is lower than the true price, causing depositors to receive more rsETH than they are entitled to, directly diluting the yield owed to existing rsETH holders.

## Finding Description

`LRTOracle` stores the exchange rate as a plain storage variable:

```solidity
uint256 public override rsETHPrice;
``` [1](#0-0) 

This variable is only written inside `_updateRsETHPrice()`, which is called exclusively through `updateRSETHPrice()` (public, keeper-driven) or `updateRSETHPriceAsManager()` (manager-only): [2](#0-1) [3](#0-2) 

The deposit path is:

`depositETH()` → `_beforeDeposit()` → `getRsETHAmountToMint()`: [4](#0-3) [5](#0-4) 

`getRsETHAmountToMint` divides by the cached `rsETHPrice` with no freshness update: [6](#0-5) 

The true current price is computed inside `_updateRsETHPrice()` by summing all TVL across NodeDelegators (including live EigenLayer pod shares and LST oracle prices) and dividing by total rsETH supply: [7](#0-6) 

Between two keeper calls, ETH staking rewards and LST appreciation increase the true TVL while `rsETHPrice` in storage remains at the old, lower value. Any deposit during this window uses the stale denominator, minting more rsETH than the depositor's contribution warrants.

The `pricePercentageLimit` guard in `_updateRsETHPrice()` only restricts who can *update* the price when the increase exceeds the threshold — it does not prevent deposits from using the stale price, and it does not bound the staleness window: [8](#0-7) 

## Impact Explanation

This is **High — Theft of unclaimed yield**. The excess rsETH minted to the depositor represents a proportional claim on TVL that was earned by existing holders through staking rewards. After `updateRSETHPrice()` is eventually called, the new price reflects the diluted supply, and prior holders receive less yield than they earned. The loss is permanent and proportional to the size of the deposit relative to total TVL and the magnitude of the price discrepancy.

## Likelihood Explanation

`updateRSETHPrice()` is called by an off-chain keeper on a periodic schedule, not atomically with every deposit. The existence of `pricePercentageLimit`, `maxFeeMintAmountPerDay`, and `feePeriodStartTime` confirms the protocol explicitly expects periodic (not per-block) price updates. A sophisticated attacker can monitor `_getTotalEthInProtocol()` on-chain versus the stored `rsETHPrice` and time large deposits to maximize the discrepancy. The exploit requires no special privileges — `depositETH()` and `depositAsset()` are fully public — and is repeatable every keeper interval.

## Recommendation

Call `_updateRsETHPrice()` (or expose it as an internal call accessible from `LRTDepositPool`) at the start of `depositETH()` and `depositAsset()` before computing the mint amount. Alternatively, compute the rsETH mint amount using a freshly calculated price (i.e., `_getTotalEthInProtocol() / rsETHSupply`) rather than the cached `rsETHPrice` storage variable, so that the denominator always reflects the current state of the protocol.

## Proof of Concept

1. Protocol state: 1000 ETH TVL, 1000 rsETH supply → `rsETHPrice = 1.0 ETH/rsETH`.
2. Staking rewards of 10 ETH accrue → true TVL = 1010 ETH, true price = `1.01 ETH/rsETH`.
3. Keeper has not yet called `updateRSETHPrice()`; `rsETHPrice` in storage = `1.0`.
4. Attacker calls `depositETH{value: 100 ETH}(0, "")`.
5. `getRsETHAmountToMint` computes: `(100e18 * 1e18) / 1e18 = 100 rsETH` minted.
6. Correct amount at true price: `100e18 / 1.01e18 ≈ 99.01 rsETH`.
7. Attacker receives `≈ 0.99 rsETH` excess, representing a claim on ~0.99 ETH of the 10 ETH reward pool that belongs to existing holders.
8. When `updateRSETHPrice()` is called, new price = `1110 ETH / 1100 rsETH ≈ 1.009 ETH/rsETH` — lower than the true pre-deposit price of `1.01`, confirming existing holders' yield was diluted.

**Foundry fork test outline:**
- Fork mainnet at a block where rewards have accrued since the last `updateRSETHPrice()` call.
- Record `rsETHPrice` and compute `_getTotalEthInProtocol() / rsETH.totalSupply()` to confirm discrepancy.
- Call `depositETH` with a large value and record rsETH minted.
- Call `updateRSETHPrice()` and verify the new price is lower than the true pre-deposit price, confirming dilution of existing holders.

    function getRsETHAmountToMint(
        address asset,
        uint256 amount
    )
        public
        view
        override
        returns (uint256 rsethAmountToMint)
    {
        // setup oracle contract
        address lrtOracleAddress = lrtConfig.getContract(LRTConstants.LRT_ORACLE);
        ILRTOracle lrtOracle = ILRTOracle(lrtOracleAddress);

        // calculate rseth amount to mint based on asset amount and asset exchange rate
        rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
