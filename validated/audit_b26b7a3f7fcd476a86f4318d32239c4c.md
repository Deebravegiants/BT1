After tracing the full call chain through the actual code, I can confirm this is a real vulnerability. The critical missing piece is in `reorg_rollback`.

### Title
Pool Wallet Zombie After Reorg Below Launcher Height: Missing `self.wallets.pop()` in `reorg_rollback` — (`chia/wallet/wallet_state_manager.py`, `chia/pools/pool_wallet.py`)

---

### Summary

When a blockchain reorg rolls back to a height below the pool wallet's launcher spend height, `reorg_rollback` correctly identifies the wallet for removal and calls `delete_wallet`, but **never removes the wallet from the in-memory `self.wallets` dict**. The pool store is simultaneously emptied by `rollback`. The result is a permanently zombie `PoolWallet` object in memory whose pool store is empty, causing `assert len(all_spends) >= 1` to fire on every subsequent call to `get_current_state()`, permanently corrupting pool membership state for that wallet.

---

### Finding Description

**Step 1 — `reorg_rollback` calls `delete_wallet` but skips `self.wallets.pop`** [1](#0-0) 

```python
remove_ids: list[uint32] = []
for wallet_id, wallet in self.wallets.items():
    if wallet.type() == WalletType.POOLING_WALLET.value:
        ...
        remove: bool = await wallet.rewind(height, action_scope)
        if remove:
            remove_ids.append(wallet_id)
for wallet_id in remove_ids:
    await self.delete_wallet(wallet_id)          # ← DB only
    self.state_changed("wallet_removed", wallet_id)
    # ← self.wallets.pop(wallet_id) is NEVER called
```

**Step 2 — `delete_wallet` only touches the DB, not `self.wallets`** [2](#0-1) 

```python
async def delete_wallet(self, wallet_id: uint32) -> None:
    await self.user_store.delete_wallet(wallet_id)
    await self.puzzle_store.delete_wallet(wallet_id)
    # self.wallets is never touched
```

Compare with the DID wallet removal path, which explicitly calls `self.wallets.pop(remove_id)` after `delete_wallet`: [3](#0-2) 

**Step 3 — `rewind` empties the pool store before returning True** [4](#0-3) 

When `block_height < launcher_height`:
- `history` is fetched (contains the launcher spend at height H)
- `pool_store.rollback(block_height, wallet_id)` deletes all rows where `height > block_height` — including the launcher spend
- `history[0][0] > block_height` → `True` → returns `True`

The pool store for this wallet is now empty.

**Step 4 — `WalletPoolStore.rollback` deletes all rows including the launcher spend** [5](#0-4) 

```python
DELETE FROM pool_state_transitions WHERE height>? AND wallet_id=?
```

If launcher was at height 100 and reorg target is 50, this deletes the launcher row.

**Step 5 — Zombie wallet crashes on next `get_current_state()` call** [6](#0-5) 

```python
async def get_current_state(self) -> PoolWalletInfo:
    history = await self.get_spend_history()
    all_spends = [cs for _, cs in history]
    assert len(all_spends) >= 1   # ← AssertionError: pool store is empty
```

**Step 6 — `new_peak` calls `get_current_state()` with no exception guard** [7](#0-6) 

```python
async def new_peak(self, peak_height: uint32) -> None:
    pool_wallet_info: PoolWalletInfo = await self.get_current_state()  # ← unguarded
```

Every new block peak triggers this unhandled `AssertionError` for the zombie wallet.

**Step 7 — Subsequent `reorg_rollback` calls are also broken**

On the next reorg, `rewind` is called again on the zombie wallet. `get_current_state()` at line 314 fires the same `AssertionError`, which is caught by the `except Exception` block, causing `rewind` to return `False`. The wallet is never added to `remove_ids` again and stays in `self.wallets` forever. [8](#0-7) 

---

### Impact Explanation

After a reorg below the launcher height:
- The pool wallet is removed from the DB (`user_store`, `puzzle_store`) and its pool store is emptied
- The wallet object remains in `self.wallets` indefinitely
- Every new block peak causes an unhandled `AssertionError` in `new_peak`
- Every subsequent reorg silently swallows the error and leaves the zombie in place
- Pool membership state is permanently corrupted: the wallet cannot report its state, cannot process transitions, and cannot be cleanly removed without a restart

This matches **High: Corruption of pool membership state** and **High: Permanent inability for honest wallets to process pool actions or sync updates under normal network assumptions**.

---

### Likelihood Explanation

Reorgs are normal network events. A reorg deep enough to roll back below a pool wallet's launcher height is uncommon but entirely possible (e.g., during initial sync, network partition, or a targeted selfish-mining attack). No special privileges are required — the trigger is receiving a new peak via the standard peer protocol. The wallet node does not need to be misconfigured.

---

### Recommendation

In `reorg_rollback`, after calling `delete_wallet`, explicitly remove the wallet from `self.wallets`:

```python
for wallet_id in remove_ids:
    await self.delete_wallet(wallet_id)
    self.wallets.pop(wallet_id, None)   # ← add this line
    self.state_changed("wallet_removed", wallet_id)
```

This mirrors the pattern already used in the DID wallet removal path. [9](#0-8) 

Additionally, `get_current_state()` should guard against an empty history with a graceful error rather than a bare `assert`, and `new_peak` should wrap its call to `get_current_state()` in a try/except to prevent a single zombie wallet from disrupting all peak processing.

---

### Proof of Concept

```
1. Create a pool wallet; launcher spend confirmed at block height 100.
2. Trigger a reorg to height 50 (e.g., via WalletStateManager.reorg_rollback(50)).
3. Observe:
   a. pool_store.get_spends_for_wallet(wallet_id) → [] (all rows deleted)
   b. wallet_state_manager.wallets[wallet_id] still exists (not popped)
   c. user_store.get_wallet_by_id(wallet_id) → None (deleted from DB)
4. Trigger a new peak (height 51).
5. Observe: AssertionError in PoolWallet.get_current_state() at `assert len(all_spends) >= 1`.
6. Trigger another reorg_rollback(40).
7. Observe: rewind() catches the AssertionError, returns False, wallet remains in self.wallets.
8. Pool wallet is permanently stuck: unreachable via DB, broken in memory.
```

```python
    async def rewind(self, block_height: int, action_scope: WalletActionScope) -> bool:
        """
        Rolls back all transactions after block_height, and if creation was after block_height, deletes the wallet.
        Returns True if the wallet should be removed.
        """
        try:
            history: list[tuple[uint32, CoinSpend]] = await self.wallet_state_manager.pool_store.get_spends_for_wallet(
                self.wallet_id
            )
            prev_state: PoolWalletInfo = await self.get_current_state()
            await self.wallet_state_manager.pool_store.rollback(block_height, self.wallet_id)

            if len(history) > 0 and history[0][0] > block_height:
                return True
            else:
                if await self.get_current_state() != prev_state:
                    await self.update_pool_config(action_scope)
                return False
        except Exception as e:
            self.log.error(f"Exception rewinding: {e}")
            return False
```
