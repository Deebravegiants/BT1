Audit Report

## Title
Unbounded Withdrawal Queue Enables Temporary Freezing of Legitimate Withdrawals - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
`initiateWithdrawal` accepts any amount ≥ `minRsEthAmountToWithdraw[asset]`, which defaults to zero and requires an explicit admin call to set. There is no per-user cap on pending requests. Because `_unlockWithdrawalRequests` processes the global queue in strict FIFO order and uses `break` (not `continue`) when a condition fails, an attacker who floods the queue before a legitimate user forces the operator to exhaust all attacker entries before the legitimate user's withdrawal can be unlocked, temporarily freezing those funds.

## Finding Description
`minRsEthAmountToWithdraw` is declared as a plain mapping that zero-initializes by default; `setMinRsEthAmountToWithdraw` is the only setter and is restricted to `onlyLRTAdmin` (L330–332). With the default of zero, `initiateWithdrawal` accepts any non-zero amount (L162–164).

`_addUserWithdrawalRequest` appends to the global nonce counter and the caller's per-user deque with no cap check (L755–757). The `DoubleEndedQueue.pushBack` only reverts on a uint128 wraparound, which is effectively unlimited.

`_unlockWithdrawalRequests` iterates from `nextLockedNonce[asset]` up to `firstExcludedIndex` (L790). On two conditions it **breaks**, halting the entire loop:
- `block.number < request.withdrawalStartBlock + withdrawalDelayBlocks` (L795)
- `availableAssetAmount < payoutAmount` (L800)

Neither condition skips an entry; both stop processing entirely. Therefore every attacker-inserted entry at the front of the queue must be fully processed before any later legitimate request is reached. The operator controls `firstExcludedIndex` but cannot skip entries — the pointer `nextLockedNonce[asset]` advances only sequentially.

Additionally, each attacker request commits assets via `assetsCommitted[asset] += expectedAssetAmount` (L173), consuming available capacity and potentially blocking new legitimate `initiateWithdrawal` calls (L170) until the attacker's entries are unlocked and burned.

## Impact Explanation
**Temporary freezing of funds (Medium):** A legitimate user who has already submitted a withdrawal request (rsETH transferred to the contract, L166) cannot call `completeWithdrawal` until their nonce is below `nextLockedNonce[asset]` (L707). If the attacker's N entries precede the victim's in the queue, the victim's rsETH is inaccessible until the operator processes all N entries. With `minRsEthAmountToWithdraw == 0`, N is bounded only by total protocol assets divided by 1 wei, making the freeze duration proportional to operator batching throughput. The attacker can sustain the delay by re-submitting after each batch clears, as their rsETH is returned after the 8-day delay.

## Likelihood Explanation
- `minRsEthAmountToWithdraw` defaults to zero; many deployments may leave it unset.
- `initiateWithdrawal` is permissionless — any rsETH holder can call it.
- Even with a small minimum (e.g., 0.001 ETH), an attacker holding 1 ETH of rsETH can submit 1,000 requests. The rsETH is returned after 8 days, so the only cost is gas and opportunity cost.
- The attack is repeatable: after each batch is cleared, the attacker re-submits, sustaining the delay indefinitely at low marginal cost.

## Recommendation
1. **Enforce a non-zero `minRsEthAmountToWithdraw`** at initialization time, or revert in `initiateWithdrawal` if the minimum has not been set for the asset.
2. **Add a per-user cap** on simultaneously pending withdrawal requests: `require(userAssociatedNonces[asset][msg.sender].length() < MAX_PENDING_PER_USER)` before `pushBack` in `_addUserWithdrawalRequest`.
3. Consider a **small non-refundable protocol fee** on `initiateWithdrawal` to raise the economic cost of flooding the queue.

## Proof of Concept
```solidity
// Precondition: minRsEthAmountToWithdraw[asset] == 0 (default)
// Attacker holds N wei of rsETH

for (uint i = 0; i < N; i++) {
    // Each call transfers 1 wei rsETH, commits ~1 wei of asset
    lrtWithdrawalManager.initiateWithdrawal(asset, 1, "");
}

// Victim submits their withdrawal — now at global nonce position N
lrtWithdrawalManager.initiateWithdrawal(asset, largeAmount, ""); // victim's rsETH locked

// Operator must call unlockQueue ceil(N / batchSize) times,
// advancing nextLockedNonce through all N attacker entries,
// before victim's nonce becomes < nextLockedNonce[asset].
// Until then, victim's completeWithdrawal reverts with WithdrawalLocked (L707).
```
