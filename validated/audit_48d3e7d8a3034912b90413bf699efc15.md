### Title
ETH Push-Transfer to Recipient Contract Can Permanently Freeze User Funds After rsETH Is Burned - (File: contracts/LRTWithdrawalManager.sol)

---

### Summary

`LRTWithdrawalManager._processWithdrawalCompletion` pushes ETH to the withdrawing user's address via a low-level `call`. If that address is a contract whose `receive()` reverts (or is absent), the push always fails. Because rsETH is irreversibly burned in a prior, separate `unlockQueue` transaction, the user's principal is destroyed while the ETH remains permanently trapped in the withdrawal manager with no admin recovery path.

---

### Finding Description

The withdrawal lifecycle in `LRTWithdrawalManager` is split across two independent transactions:

**Step 1 — `unlockQueue` (operator-only):**
rsETH held by the contract is burned and the corresponding ETH is pulled from `LRTUnstakingVault` into `LRTWithdrawalManager`.

```solidity
// contracts/LRTWithdrawalManager.sol:305-307
if (rsETHBurned != 0) IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned);
//Take the amount to distribute from vault
unstakingVault.redeem(asset, assetAmountUnlocked);
```

After this transaction completes, the rsETH is gone and the ETH sits in `LRTWithdrawalManager`.

**Step 2 — `completeWithdrawal` / `completeWithdrawalForUser` (user or operator):**
`_processWithdrawalCompletion` attempts to push ETH to the user:

```solidity
// contracts/LRTWithdrawalManager.sol:734
_transferAsset(asset, user, request.expectedAssetAmount);
```

which resolves to:

```solidity
// contracts/LRTWithdrawalManager.sol:877-879
(bool sent,) = payable(to).call{ value: amount }("");
if (!sent) revert EthTransferFailed();
```

If `to` is a contract that reverts on ETH receipt (no `receive()`, or an explicit `revert` in `receive()`), `sent` is `false` and the entire Step 2 transaction reverts. Because Step 1 already committed and cannot be undone, the rsETH is permanently burned while the ETH is permanently stranded.

**No recovery path exists:**
- `sweepRemainingAssets` is gated by `!hasUnlockedWithdrawals(asset)`. The stuck request keeps `unlockedWithdrawalsCount[asset] > 0`, so this function is permanently blocked for that asset.
- `completeWithdrawalForUser` (operator path) sends ETH to the same `user` address and therefore also always reverts.
- There is no `cancelWithdrawal`, no alternative recipient parameter, and no admin escape hatch for stranded ETH.

The developer comment on `completeWithdrawalForUser` acknowledges ETH transfer risk but dismisses it:

```
/// @dev Not expected to be used for ETH; potential gas grief scenarios are non-impactful for ETH
```

This underestimates the severity: the scenario is not mere gas grief but a permanent, irrecoverable fund freeze.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

After `unlockQueue` burns the user's rsETH, the ETH allocated to that user is held in `LRTWithdrawalManager`. If the user's address cannot accept ETH, neither the user nor any operator can ever complete the withdrawal. The ETH is locked in the contract indefinitely with no on-chain recovery mechanism. The user loses their entire principal.

---

### Likelihood Explanation

**Medium.**

Smart-contract wallets, multisigs, and DeFi protocol treasuries that interact with LRT-rsETH may lack a `receive()` function or may have one that conditionally reverts (e.g., access-controlled, paused, or upgraded away). A malicious actor can also deliberately deploy such a contract, initiate a withdrawal, wait for the operator to burn their rsETH, and then use the permanently stuck ETH as leverage — demanding out-of-band compensation before agreeing to upgrade their contract to accept ETH. No privileged access is required; any depositor can trigger this path.

---

### Recommendation

Replace the push-payment pattern with a pull-payment (claimable balance) pattern for ETH withdrawals:

1. In `_processWithdrawalCompletion`, instead of calling `_transferAsset` directly, credit `claimableETH[user] += request.expectedAssetAmount`.
2. Add a separate `claimETH()` function that lets the user pull their balance to any address they specify.

This decouples the accounting step (which must succeed atomically) from the ETH delivery step (which is user-controlled), eliminating the ability of a reverting `receive()` to freeze funds.

---

### Proof of Concept

1. Attacker deploys `MaliciousRecipient` — a contract with `receive() external payable { revert(); }`.
2. `MaliciousRecipient` calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. rsETH is transferred to `LRTWithdrawalManager`; a `WithdrawalRequest` is recorded.
3. Operator calls `unlockQueue(ETH_TOKEN, ...)`. At line 305, rsETH is burned. At line 307, ETH is moved from `LRTUnstakingVault` into `LRTWithdrawalManager`. The request is now "unlocked".
4. `MaliciousRecipient` calls `completeWithdrawal(ETH_TOKEN, "")`. `_processWithdrawalCompletion` reaches line 878: `payable(MaliciousRecipient).call{value: amount}("")` → `receive()` reverts → `sent = false` → `revert EthTransferFailed()`. Entire tx reverts; request remains in queue.
5. Operator calls `completeWithdrawalForUser(ETH_TOKEN, MaliciousRecipient, "")`. Same revert.
6. Operator attempts `sweepRemainingAssets(ETH_TOKEN)`. Reverts: `hasUnlockedWithdrawals(ETH_TOKEN)` is `true`.
7. **Result:** rsETH is permanently burned; ETH is permanently locked in `LRTWithdrawalManager`. No on-chain path exists to recover it. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

    /// @param asset The asset address to sweep
    /// @return transferredAmount The amount swept to treasury
    /// @dev This function can only be called when no unlocked withdrawals exist for the asset
    /// @dev Not expected to be used for ETH; ETH should not accumulate requiring sweeping
    function sweepRemainingAssets(address asset)
        external
        nonReentrant
        onlySupportedAsset(asset)
        onlyLRTManager
        returns (uint256 transferredAmount)
    {
        // Check that all withdrawals are completed
        if (hasUnlockedWithdrawals(asset)) revert PendingWithdrawalsExist();

        uint256 balance = _getAssetBalance(asset);
        if (balance == 0) revert AmountMustBeGreaterThanZero();

        // Transfer to treasury
        address treasury = lrtConfig.getContract(LRTConstants.PROTOCOL_TREASURY);
        _transferAsset(asset, treasury, balance);

        emit RemainingAssetsSwept(asset, balance, treasury);
        return balance;
    }

    function _processWithdrawalCompletion(address asset, address user, string calldata referralId) internal {
        if (userAssociatedNonces[asset][user].empty()) {
            revert NoWithdrawalRequests(user, asset);
        }

        // Retrieve and remove the oldest withdrawal request for the user.
        uint256 usersFirstWithdrawalRequestNonce = userAssociatedNonces[asset][user].popFront();
        // Ensure the request is already unlocked.
        if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();

        bytes32 requestId = getRequestId(asset, usersFirstWithdrawalRequestNonce);
        WithdrawalRequest memory request = withdrawalRequests[requestId];

        delete withdrawalRequests[requestId];

        // Check that the withdrawal delay has passed since the request's initiation.
        if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();

        unlockedWithdrawalsCount[asset]--;

        // If Aave integration is enabled and asset is ETH, withdraw from Aave if needed
        if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
            uint256 contractBalance = address(this).balance;
            if (contractBalance < request.expectedAssetAmount) {
                uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
                _withdrawFromAave(amountNeeded);

                // Verify we have sufficient balance after withdrawal
                uint256 balanceAfter = address(this).balance;
                if (balanceAfter < request.expectedAssetAmount) {
                    revert InsufficientLiquidityForWithdrawal();
                }
            }
        }

        _transferAsset(asset, user, request.expectedAssetAmount);

        emit ReferralIdEmitted(referralId);
        emit AssetWithdrawalFinalized(user, asset, request.rsETHUnstaked, request.expectedAssetAmount);
    }
