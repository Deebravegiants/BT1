### Title
Unrestricted `init()` Allows Any Account to Initialize the MPC Contract with Malicious Participants - (File: `crates/contract/src/lib.rs`)

### Summary
The `init()` function in `MpcContract` lacks a caller restriction. Unlike `init_running()` and `migrate()`, which are both marked `#[private]`, `init()` can be called by any NEAR account on an uninitialized contract. This enables a front-running attack where an adversary initializes the contract with attacker-controlled participants before the legitimate deployer, seizing full control of the MPC network.

### Finding Description
The `init()` function at line 1926 of `crates/contract/src/lib.rs` is decorated with `#[init]` (which prevents re-initialization once state exists) but carries no `#[private]` attribute and no explicit `assert_self()` / predecessor check:

```rust
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
```

In contrast, both sibling initializers are properly restricted:

```rust
#[private]   // ← present on init_running
#[init]
pub fn init_running(...) -> Result<Self, Error> { ... }

#[private]   // ← present on migrate
#[init(ignore_state)]
pub fn migrate() -> Result<Self, Error> { ... }
```

The `#[private]` macro in the NEAR SDK enforces `predecessor_account_id == current_account_id`, meaning only the contract account itself can invoke those functions. `init()` has no such guard.

The production deployment workflow documented in `docs/localnet/localnet.md` and `docs/testnet/setup-guide-for-testnet-with-tee-support.md` performs contract deployment and initialization as **separate transactions**:

```
near contract call-function as-transaction mpc-contract.test.near init \
  file-args /tmp/init_args.json ... sign-as mpc-contract.test.near ...
```

This creates a non-zero window between the `DeployContract` receipt and the `init` call receipt during which any external account can race to call `init()` with attacker-controlled `parameters`.

Because `#[init]` prevents re-initialization once state is written, the legitimate deployer's subsequent `init()` call will fail with "contract is already initialized," leaving the attacker's participant set permanently in place.

### Impact Explanation
**Critical.** An attacker who wins the initialization race supplies an arbitrary `ThresholdParameters` struct naming accounts they control as the full participant set. Those participants then:

1. Run MPC nodes that complete DKG and generate threshold key shares.
2. Collectively hold the signing capability for every domain registered on the contract.
3. Can issue threshold signatures on foreign chains (Bitcoin, Ethereum, etc.) without any legitimate user authorization.

This satisfies the allowed Critical impact: *"Unauthorized transaction execution, threshold signature issuance, or confidential key derivation output without the required participant authorization"* and *"Theft, direct loss, or permanent freezing of funds controlled by the MPC network."*

### Likelihood Explanation
**Medium.** NEAR block times are approximately one second. The attacker only needs to observe the `DeployContract` action in the mempool or in a finalized block and submit a `FunctionCall` to `init` in the next block. No privileged access, leaked key, or TEE bypass is required — only the ability to submit a standard NEAR transaction. The attack is fully permissionless and requires no collusion.

### Recommendation
Add `#[private]` to `init()`, consistent with `init_running()` and `migrate()`:

```rust
#[private]   // ← add this
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> { ... }
```

This restricts the call to transactions where `predecessor_account_id == current_account_id`, i.e., only the contract account itself. Alternatively, combine deployment and initialization into a single batch transaction (one `DeployContract` action followed immediately by a `FunctionCall` to `init` in the same receipt batch), eliminating the race window entirely.

### Proof of Concept

1. Attacker watches the NEAR blockchain for a `DeployContract` action targeting the MPC contract account ID.
2. In the immediately following block, attacker submits:
   ```
   near contract call-function as-transaction <mpc-contract-account> init \
     json-args '{"parameters": {"participants": [...attacker accounts...], "threshold": 1}}' \
     sign-as <attacker-account> ...
   ```
3. The `#[init]` guard writes contract state; the legitimate deployer's subsequent `init()` call panics with "contract is already initialized."
4. Attacker's accounts are now the sole registered participants. Attacker runs MPC nodes, completes DKG via `vote_add_domains` → `start_keygen_instance` → `vote_pk`, and obtains threshold signing capability.
5. Attacker calls `sign()` on behalf of any user payload, producing valid foreign-chain signatures and enabling fund theft. [1](#0-0) [2](#0-1) [3](#0-2)
