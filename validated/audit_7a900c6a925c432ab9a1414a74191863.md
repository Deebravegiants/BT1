### Title
Wrong Gas Config Field Used for `verify_foreign_transaction` Yield Callback, Enabling Permanent Request Failure and Deposit Loss - (File: `crates/contract/src/lib.rs`)

---

### Summary

`verify_foreign_transaction` allocates callback gas by reading `return_signature_and_clean_state_on_success_call_tera_gas` — the gas budget intended for the **sign** callback — instead of a dedicated field for the foreign-tx callback. The `Config` struct has a separate field for the CKD callback but no field for the foreign-tx callback, so the two request types share a single gas knob they should not share. If governance lowers that shared field (a routine optimization for sign throughput), every in-flight `verify_foreign_transaction` yield callback receives insufficient gas, OOGs, and the user's deposit is permanently lost.

---

### Finding Description

In `verify_foreign_transaction` the callback gas is sourced from the wrong config field:

```rust
// crates/contract/src/lib.rs  lines 544-556
let callback_gas = Gas::from_tgas(
    self.config
        .return_signature_and_clean_state_on_success_call_tera_gas,  // ← sign callback field
);

let request = args_into_verify_foreign_tx_request(request);
let callback_args = serde_json::to_vec(&(&request,)).unwrap();
self.enqueue_yield_request(
    method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,  // correct method name
    callback_args,
    callback_gas,   // ← gas from the wrong field
    move |this, id| this.add_verify_foreign_tx_request(request, id),
);
``` [1](#0-0) 

The `Config` struct defines a dedicated field for the CKD callback (`return_ck_and_clean_state_on_success_call_tera_gas`) but has **no** dedicated field for the foreign-tx callback:

```rust
// crates/contract/src/config.rs  lines 53-56
/// Prepaid gas for a `return_signature_and_clean_state_on_success` call.
pub(crate) return_signature_and_clean_state_on_success_call_tera_gas: u64,
/// Prepaid gas for a `return_ck_and_clean_state_on_success` call.
pub(crate) return_ck_and_clean_state_on_success_call_tera_gas: u64,
// ← no return_verify_foreign_tx_and_clean_state_on_success_call_tera_gas
``` [2](#0-1) 

The foreign-tx callback (`return_verify_foreign_tx_and_clean_state_on_success`) deserializes a `VerifyForeignTransactionRequest` — which embeds a full `ForeignChainRpcRequest` enum (Bitcoin tx_id + confirmations + extractors, EVM tx_id + extractors + finality, etc.) — plus a `VerifyForeignTransactionResponse` containing a full ECDSA signature. This is structurally larger and more expensive to deserialize than the `SignatureRequest` + `SignatureResponse` pair handled by the sign callback. [3](#0-2) 

The sign and CKD callbacks each have their own gas field and can be tuned independently. The foreign-tx callback cannot be tuned independently; it is silently coupled to the sign callback budget. [4](#0-3) 

---

### Impact Explanation

When the yield callback OOGs, NEAR runtime delivers `Err(PromiseError::Failed)` to `return_verify_foreign_tx_and_clean_state_on_success`. The `Err(_)` branch pops the pending yield slot and schedules `fail_on_timeout`, which panics with a timeout error. There is no deposit refund on this path: [5](#0-4) 

The user's attached deposit (≥ `MINIMUM_SIGN_REQUEST_DEPOSIT`) is permanently lost. Every queued `verify_foreign_transaction` yield sharing the same request key is affected simultaneously, because `resolve_yields_for` fans out the response to all of them.

This maps to: **Medium — request-lifecycle and balance invariant broken without requiring network-level DoS or threshold collusion.**

---

### Likelihood Explanation

The default value is 7 TGas for both fields, which is currently sufficient. However:

1. Governance can call `update_config` to lower `return_signature_and_clean_state_on_success_call_tera_gas` for sign-throughput optimization — a routine, legitimate action.
2. Because the foreign-tx callback silently reads the same field, that reduction also applies to foreign-tx callbacks, which have larger argument payloads.
3. There is no on-chain guard or documentation warning that this field controls two distinct callbacks with different gas profiles.
4. The attacker-controlled entry path is simply submitting a `verify_foreign_transaction` request after governance has lowered the shared gas field; no privileged access is needed by the attacker. [6](#0-5) 

---

### Recommendation

1. Add a dedicated `return_verify_foreign_tx_and_clean_state_on_success_call_tera_gas` field to both `Config` (internal) and `InitConfig` / public `Config` (interface) structs, with a default value benchmarked against the actual foreign-tx callback cost.
2. Update `verify_foreign_transaction` to read the new dedicated field.
3. Add a migration step to populate the new field from the existing value so deployed contracts are not broken on upgrade.
4. Add a unit/sandbox test that verifies the foreign-tx callback succeeds with the configured gas budget when given a maximally-sized `ForeignChainRpcRequest`.

---

### Proof of Concept

1. Governance votes to lower `return_signature_and_clean_state_on_success_call_tera_gas` from 7 TGas to 3 TGas to reduce sign-request overhead.
2. Alice calls `verify_foreign_transaction` with a Bitcoin request (large `ForeignChainRpcRequest` with multiple extractors). Her deposit is locked.
3. MPC nodes verify the transaction and call `respond_verify_foreign_tx` with a valid response.
4. `resolve_yields_for` calls `env::promise_yield_resume` for Alice's yield, scheduling `return_verify_foreign_tx_and_clean_state_on_success` with only 3 TGas.
5. The callback OOGs while deserializing the large `VerifyForeignTransactionRequest` argument.
6. NEAR runtime delivers `Err(PromiseError::Failed)` to the callback; `pop_oldest_pending_yield` removes Alice's slot and `fail_on_timeout` panics.
7. Alice's transaction fails with a timeout error and her deposit is not refunded. [1](#0-0) [6](#0-5)

    pub fn return_verify_foreign_tx_and_clean_state_on_success(
        &mut self,
        request: VerifyForeignTransactionRequest,
        #[callback_result] response: Result<VerifyForeignTransactionResponse, PromiseError>,
    ) -> PromiseOrValue<VerifyForeignTransactionResponse> {
        match response {
            Ok(response) => PromiseOrValue::Value(response),
            Err(_) => {
                pending_requests::pop_oldest_pending_yield(
                    &mut self.pending_verify_foreign_tx_requests,
                    &request,
                );
                let fail_on_timeout_gas = Gas::from_tgas(self.config.fail_on_timeout_tera_gas);
                let promise = Promise::new(env::current_account_id()).function_call(
                    method_names::FAIL_ON_TIMEOUT.to_string(),
                    vec![],
                    NearToken::from_near(0),
                    fail_on_timeout_gas,
                );
                near_sdk::PromiseOrValue::Promise(promise.as_return())
            }
        }
    }
