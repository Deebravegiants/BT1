### Title
Unconstrained `fee_proposal_fri` During Bootstrap Period Poisons `fee_actual` and All Subsequent Block Commitments — (`File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

The `fee_proposal_fri` bounds check in `is_proposal_init_valid` is explicitly skipped for the first `window_size` blocks because `fee_actual` is `None` during that period. A malicious proposer who controls any block in the bootstrap window can inject an arbitrary `fee_proposal_fri` (e.g., `u128::MAX` or `1`) that validators accept without restriction. Those values are permanently recorded in `fee_proposals_window`, corrupt the `fee_actual` median once the window fills, and propagate into every subsequent `ProposalCommitment` via `Poseidon(partial_block_hash, fee_proposal_fri)`.

### Finding Description

In `is_proposal_init_valid`, the bounds check is guarded by a pattern match that silently passes when `fee_actual` is `None`:

```rust
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{ … }
``` [1](#0-0) 

`fee_actual` is `None` whenever `height < window_size`, because `compute_fee_actual` returns `None` on the `checked_sub` underflow:

```rust
let Some(start) = height.0.checked_sub(window_size) else {
    warn!("Cannot compute fee_actual for height {height}: height is below window_size …");
    return None;
};
``` [2](#0-1) 

The validator constructs `ProposalInitValidation` with `fee_actual: compute_fee_actual(…)` at both the current-round and queued-proposal paths: [3](#0-2) 

So for every block at height `< window_size`, the `if let` arm is never entered and any `fee_proposal_fri` value — including `u128::MAX` or `1` — passes validation.

After the proposal is decided, `finalize_decision` records the accepted value permanently:

```rust
self.record_fee_proposal(height, init.fee_proposal_fri);
``` [4](#0-3) 

Once `window_size` blocks have been committed, `compute_fee_actual` computes the median of those recorded values. If the attacker controlled more than half of the bootstrap blocks, the median equals the injected extreme. Every subsequent proposer is then constrained to stay within `margin_ppt` of that corrupted median: [5](#0-4) 

The corrupted `fee_proposal_fri` is also hashed directly into the `ProposalCommitment` that consensus signs over:

```rust
ProposalCommitment(Poseidon::hash_array(&[partial.0, Felt::from(fee_proposal.0)]))
``` [6](#0-5) 

### Impact Explanation

**Corrupted `fee_actual` → wrong L2 gas price for all future blocks.** `fee_actual` feeds into `calculate_next_l2_gas_price_for_fin`, which sets `self.l2_gas_price` at every `decision_reached`. If the attacker drives `fee_actual` to `u128::MAX`, every subsequent block's L2 gas price is pegged near `u128::MAX`, making all user transactions economically impossible. If driven to `1`, the chain becomes trivially spammable at near-zero cost.

**Wrong `ProposalCommitment` committed to chain.** The commitment `Poseidon(partial_block_hash, fee_proposal_fri)` is the value consensus votes on and the batcher stores. Injected extreme values produce commitments that honest nodes would never have produced under normal operation, yet they are accepted and finalized.

**Permanent, self-reinforcing corruption.** Because `fee_proposal_fri` is bounded to `fee_actual ± margin_ppt` once the window is full, the corrupted baseline cannot be corrected by honest proposers within a single round; recovery requires `window_size` blocks of honest proposals to shift the median back, during which the corrupted price remains in effect.

### Likelihood Explanation

The attack window is the first `window_size` blocks of the chain (or after a restart that clears `fee_proposals_window`). A validator who is the legitimate proposer for more than half of those blocks can execute this without any additional capability. In early-stage or single-sequencer deployments the bootstrap window is entirely controlled by one party. The code comment explicitly acknowledges the unconstrained state ("bounds are not enforced"), confirming this is a known design gap rather than an oversight.

### Recommendation

Replace the silent `if let` skip with an explicit fallback reference. During the bootstrap period, bound `fee_proposal_fri` against the configured `l2_gas_price` (the same value `compute_proposer_fee_proposal` falls back to when `fee_actual` is `None`):

```rust
let reference = proposal_init_validation.fee_actual
    .unwrap_or(proposal_init_validation.l2_gas_price_fri);
let (lower_bound, upper_bound) = fee_proposal_bounds(
    reference,
    VersionedConstants::latest_constants().fee_proposal_margin_ppt,
);
if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
    return Err(…);
}
```

This preserves the existing steady-state behavior while preventing arbitrary injection during bootstrap.

### Proof of Concept

1. Chain starts at height 0; `window_size = N` (e.g., 10).
2. Attacker is the legitimate proposer for blocks 0 … ⌊N/2⌋.
3. For each of those blocks, attacker sets `fee_proposal_fri = u128::MAX` in `ProposalInit`.
4. `is_proposal_init_valid` is called; `proposal_init_validation.fee_actual` is `None` (height < N); the `if let` arm is skipped; the proposal passes.
5. `finalize_decision` calls `record_fee_proposal(height, Some(u128::MAX))` for each block.
6. At height N, `compute_fee_actual` sorts the window and returns the median = `u128::MAX`.
7. `calculate_next_l2_gas_price_for_fin` sets `self.l2_gas_price ≈ u128::MAX`.
8. All future `ProposalInit` messages must carry `fee_proposal_fri` within `margin_ppt` of `u128::MAX`; honest proposers who cannot satisfy this are rejected by validators, stalling the chain or forcing acceptance of the corrupted price.
9. Every `ProposalCommitment` from block N onward encodes `Poseidon(partial_block_hash, ~u128::MAX)`, diverging from any honest-baseline commitment.
