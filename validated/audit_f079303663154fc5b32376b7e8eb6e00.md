### Title
Stale TEE Attestation in `is_caller_an_attested_participant` Allows Expired/Revoked-Attestation Nodes to Submit Threshold Signatures — (`File: crates/contract/src/tee/tee_state.rs`)

---

### Summary

`is_caller_an_attested_participant` checks only that a stored attestation *exists* and that the stored keys match the caller. It never re-verifies whether the attestation is still valid (not expired, image hash still approved). Because `respond`, `respond_ckd`, and `respond_verify_foreign_tx` all gate on this stale check, a node whose TEE attestation has since expired or whose image hash has been revoked can continue submitting threshold signatures for up to the full `verify_tee` polling interval (documented as 7 days).

---

### Finding Description

`is_caller_an_attested_participant` in `tee_state.rs` performs three checks:

1. The caller's account ID is in the active `Participants` set.
2. A `stored_attestations` entry exists for the participant's TLS key.
3. The stored `account_id` and `account_public_key` match the signer. [1](#0-0) 

It does **not** call `reverify_participants`, which is the function that actually checks attestation expiry and image-hash approval: [2](#0-1) 

`reverify_participants` is only invoked from `verify_tee` → `reverify_and_cleanup_participants`. Per the design documentation, nodes call `verify_tee` every 7 days. [3](#0-2) 

All three `respond` entry points call `assert_caller_is_attested_participant_and_protocol_active`, which delegates to the stale check: [4](#0-3) 

The `accept_requests` flag is a *global* gate set by `verify_tee`; it reflects whether the *aggregate* system has enough valid attestations, not whether the *individual* responding node's attestation is currently valid: [5](#0-4) 

Therefore, between the moment a single node's attestation expires (or its image hash is revoked) and the next `verify_tee` call, that node can still call `respond` and have its signature accepted by the contract.

---

### Impact Explanation

A node running software whose TEE attestation has expired or whose image hash has been revoked retains the ability to call `respond` / `respond_ckd` / `respond_verify_foreign_tx` and deliver threshold signatures that the contract will accept. This breaks the core TEE security invariant: only nodes running currently-approved, attested software should be able to contribute to signing. If the node's TEE has been compromised (which is precisely what an expired or revoked attestation signals), it may be running unapproved code with access to its keyshare, and the contract's per-call check will not detect this.

This maps to the **Medium** allowed impact: *participant-state manipulation that breaks production safety/accounting invariants* — specifically the invariant that every signing participant holds a currently-valid TEE attestation.

---

### Likelihood Explanation

- TEE attestations have finite validity periods. Expiry is a routine, predictable event.
- `verify_tee` is called on a 7-day schedule, creating a deterministic window.
- No privileged access or collusion is required; the node simply continues operating after its attestation lapses.
- An adversary who controls a node (or whose node's image hash has been revoked by a governance vote) can exploit this window without any additional capability.

---

### Recommendation

Add an inline re-verification of the stored attestation's current validity inside `is_caller_an_attested_participant` (or in `assert_caller_is_attested_participant_and_protocol_active`), calling the same expiry and image-hash checks that `reverify_participants` performs. This ensures that every `respond` call is gated on the attestation's *current* status, not its status at the time it was submitted.

Alternatively, shorten the `verify_tee` polling interval significantly, or emit an on-chain event when an attestation expires so that `accept_requests` is cleared promptly for the affected node rather than waiting for the next scheduled call.

---

### Proof of Concept

1. Node N is a legitimate participant with a valid TEE attestation stored in `tee_state.stored_attestations`.
2. N's attestation expires (or governance votes to revoke N's image hash).
3. `verify_tee` has not yet been called since the expiry; `accept_requests` remains `true`.
4. N calls `respond(request, valid_signature)`.
5. `assert_caller_is_attested_participant_and_protocol_active` calls `is_caller_an_attested_participant`, which finds the (now-stale) entry in `stored_attestations`, confirms the keys match, and returns `Ok(())`.
6. The `accept_requests` check passes (global flag still `true`).
7. The cryptographic signature check passes (N still holds its keyshare).
8. The contract resolves the pending yield and delivers the signature to the user — signed by a node whose TEE is no longer considered valid by the system. [1](#0-0) [6](#0-5)

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
