### Title
Fee-on-Transfer Token Accounting Mismatch in `initTransfer` Creates Unbacked Cross-Chain Supply — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
`OmniBridge.initTransfer` uses the caller-supplied `amount` parameter for cross-chain accounting without measuring the actual tokens received. For fee-on-transfer (deflationary) ERC-20 tokens, the contract receives fewer tokens than `amount`, yet publishes a cross-chain message crediting the full `amount` to the destination chain, creating unbacked wrapped supply.

### Finding Description
In `initTransfer`, when the token is neither a bridge token nor a custom-minter token, the contract executes:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // user-supplied, not verified against actual receipt
);
``` [1](#0-0) 

Immediately after, the unverified `amount` is forwarded to `initTransferExtension`:

```solidity
initTransferExtension(
    msg.sender, tokenAddress, currentOriginNonce,
    amount, fee, nativeFee, recipient, message, extensionValue
);
``` [2](#0-1) 

And the same `amount` is emitted in the `InitTransfer` event that drives cross-chain settlement: [3](#0-2) 

In `OmniBridgeWormhole`, `initTransferExtension` publishes a Wormhole message encoding the caller-supplied `amount` as the canonical transfer value: [4](#0-3) 

No balance snapshot is taken before/after the `safeTransferFrom` to verify actual receipt. The contract has no token whitelist and no fee-on-transfer guard.

### Impact Explanation
The NEAR side receives a cross-chain message claiming `amount` tokens are locked on EVM and mints `amount` wrapped tokens. The EVM contract actually holds `amount - fee_deducted` tokens. The resulting wrapped supply is unbacked by the shortfall. When any user later redeems via `finTransfer`:

```solidity
IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount);
``` [5](#0-4) 

the contract attempts to release the full `amount` it never actually held. Repeated deposits drain the real token reserve, causing later legitimate redeemers to receive less than their entitlement or have their redemption revert entirely — a backing guarantee break and potential fund loss for honest users.

This maps to: **High — asset-accounting divergence that breaks backing guarantees or sends value to the wrong party**, and potentially **Critical — custody escape of locked bridge assets through settlement failure**.

### Likelihood Explanation
The entry point is the public `initTransfer` function with no token whitelist. Any user can call it with any ERC-20 address, including fee-on-transfer tokens (e.g., USDT with fees enabled, STA, REFLECT-style tokens). No privileged access is required. The attacker simply calls `initTransfer` with a deflationary token and a large `amount`; the cross-chain message is published with the inflated value automatically.

### Recommendation
1. **Measure actual receipt**: Record the contract's token balance before and after `safeTransferFrom`, and use the delta as the canonical `amount` for cross-chain messaging:
   ```solidity
   uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
   IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
   uint256 actualAmount = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
   // use actualAmount in initTransferExtension and InitTransfer event
   ```
2. **Or maintain a token whitelist**: Only allow pre-approved tokens that are known not to have transfer fees, preventing deflationary tokens from being used as bridge assets.

### Proof of Concept
1. Deploy or use any fee-on-transfer ERC-20 (e.g., 1% fee per transfer).
2. Call `OmniBridge.initTransfer(feeToken, 1000e18, 0, 0, "near_recipient", "")`.
3. The contract receives `990e18` tokens (after 1% fee), but `safeTransferFrom` does not revert.
4. `initTransferExtension` is called with `amount = 1000e18`; the Wormhole message encodes `1000e18`.
5. NEAR mints `1000e18` wrapped tokens to `near_recipient`, backed by only `990e18` real tokens.
6. Repeat N times; the shortfall accumulates. The Nth honest redeemer calling `finTransfer` on EVM finds the contract balance insufficient and their redemption fails or steals from other depositors' reserves.
