### Title
Unvalidated `builder` field in `ProposalInit` allows a malicious proposer to commit an arbitrary sequencer address — (File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs)

### Summary

`is_proposal_init_valid` enforces every security-sensitive field of `ProposalInit` except `builder`. The `builder` field is used verbatim as the block's `sequencer_address` in both the batcher's execution context and the committed `BlockHeaderWithoutHash`. Because the commitment is computed from the batcher's output (which itself uses the proposer-supplied `builder`), the `ProposalFinMismatch` check cannot catch a spoofed value. Any validator that wins a proposer slot can redirect transaction fees and corrupt the `get_sequencer_address` syscall result for every transaction in the block.

### Finding Description

`is_proposal_init_valid` validates: `height`, `l1_da_mode`, `l2_gas_price_fri`, `starknet_version`, `version_constant_commitment`, `timestamp`, all four L1 gas prices (FRI + WEI), and `fee_proposal_fri`. The `builder` field is never checked against any locally-trusted value. [1](#0-0) 

The `builder` field flows into two critical paths:

**Path 1 — Transaction execution context.** `initiate_validation` calls `convert_to_sn_api_block_info(init)` and passes the result to `batcher.validate_block(input)`. The batcher uses `block_info.sequencer_address` (derived from `init.builder`) for every transaction it executes: fee collection and the `get_sequencer_address` syscall both
