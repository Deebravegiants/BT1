Audit Report

## Title
Unbounded `merkleProof` Array Enables Arbitrary Gas Consumption via `_processClaim` ŌĆö (`contracts/KERNEL/KernelMerkleDistributor.sol`)

## Summary
`KernelMerkleDistributor._processClaim()` accepts a caller-supplied `bytes32[] calldata merkleProof` with no upper-bound check before passing it to `MerkleProofUpgradeable.verify()`. Because `processProof` iterates unconditionally over every element, any unprivileged caller who holds a valid, unclaimed index can submit a proof of arbitrary length L and force the EVM to execute L `keccak256` operations before the transaction reverts with `InvalidMerkleProof`, consuming gas proportional to L with no state change.

## Finding Description
In `_processClaim` (lines 292ŌĆō346 of `contracts/KERNEL/KernelMerkleDistributor.sol`), the execution path is:

1. O(1) guards: `checkNonZeroAddress`, `currentMerkleRoot != 0`, `1 Ōēż index Ōēż currentIndex`, `account == msg.sender`, `!isClaimed(index, account)` ŌĆö all passable by any address holding a valid unclaimed index.
2. `bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));`
3. `MerkleProofUpgradeable.verify(merkleProof, currentMerkleRoot, node)` ŌĆö delegates to `processProof`, which runs `for (uint256 i = 0; i < proof.length; i++) { computedHash = _hashPair(computedHash, proof[i]); }` over all L elements unconditionally.
4. Returns `false` ŌåÆ `revert InvalidMerkleProof()`.

There is no `require(merkleProof.length <= MAX_PROOF_LENGTH)` or equivalent guard anywhere in the call chain. [1](#0-0) [2](#0-1) 

## Impact Explanation
Gas cost scales linearly with L:
- **Calldata cost**: each 32-byte proof element costs 16 gas/non-zero byte ├Ś 32 bytes = 512 gas; at L = 5,000 that is ~2.56 M gas in calldata alone.
- **Computation cost**: each `_hashPair` call costs ~40 gas; at L = 5,000 that is ~200 K additional gas.
- A single transaction with L Ōēł 9,000 can approach Ethereum's ~30 M block gas limit.

An attacker can repeatedly submit such transactions to stuff blocks and delay or prevent legitimate claimants from having their transactions included. This concretely matches **Medium. Unbounded gas consumption** and **Low. Block stuffing**, both of which are explicitly in the allowed impact scope.

## Likelihood Explanation
Preconditions are minimal: the attacker only needs a valid `index` in the range `[1, currentIndex]` (readable from the public `currentIndex` state variable) and must not have previously claimed under that index. No privileged role, no front-running, no external dependency, and no victim mistake is required. Any address that has never called `claim` satisfies this. The attack is repeatable across multiple blocks. [3](#0-2) 

## Recommendation
Add a proof-length cap before the `verify` call. A Merkle tree over N leaves requires at most `ceil(log2(N))` proof elements. For any realistic distribution (e.g., Ōēż 2┬▓Ōü░ Ōēł 1 M recipients), 20 elements is sufficient:

```solidity
uint256 public constant MAX_PROOF_LENGTH = 20;

// inside _processClaim, before the verify call:
if (merkleProof.length > MAX_PROOF_LENGTH) revert InvalidMerkleProof();
```

This makes the check O(1) and eliminates the attack surface entirely.

## Proof of Concept
```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import "forge-std/Test.sol";
import "../contracts/KERNEL/KernelMerkleDistributor.sol";

contract UnboundedProofGasTest is Test {
    KernelMerkleDistributor distributor;

    function setUp() public {
        // Deploy and initialize distributor with a non-zero merkle root,
        // currentIndex = 1, attacker address registered as index 1 (unclaimed).
    }

    function test_unboundedGasOnInvalidProof() public {
        uint256 L = 5_000;
        bytes32[] memory bigProof = new bytes32[](L);
        for (uint256 i = 0; i < L; i++) {
            bigProof[i] = bytes32(uint256(i + 1));
        }

        uint256 gasBefore = gasleft();
        vm.expectRevert(IMerkleDistributor.InvalidMerkleProof.selector);
        distributor.claim(1, address(this), 1 ether, bigProof);
        uint256 gasUsed = gasBefore - gasleft();

        assertGt(gasUsed, 100_000, "Expected significant gas consumption");
    }
}
```

Running with increasing values of L (1, 10, 100, 1 000, 5 000, 9 000) and plotting `gasUsed` vs L will confirm linear growth, demonstrating that a single transaction with L Ōēł 9 000 can approach the 30 M block gas limit.
