### Title
Missing Price Validation in Chainlink Oracle Allows Stale/Invalid Price to Drive rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but performs zero validation on the returned data: no check that `price > 0`, no staleness check via `updatedAt`, and no `answeredInRound >= roundId` completeness check. This raw price is consumed directly by `LRTOracle` → `LRTDepositPool` to compute rsETH minting amounts for any depositor.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` discards all round metadata and blindly casts the raw `int256 price` to `uint256`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

Three validations are absent:

1. **No `price > 0` check.** Chainlink can return `answer = 0` or a negative `int256` during circuit-breaker events. Casting a negative `int256` to `uint256` in Solidity 0.8.x does not revert — it wraps to a near-`type(uint256).max` value.

2. **No staleness check.** The `updatedAt` timestamp is silently discarded. Each Chainlink feed has a heartbeat (e.g., 1 hour for ETH/stETH). If the feed goes stale — e.g., during network congestion or a Chainlink node outage — the last cached price is returned indefinitely.

3. **No `answeredInRound >= roundId` check.** An incomplete round can return a stale answer from a prior round.

By contrast, the sibling contract `ChainlinkOracleForRSETHPoolCollateral` in the same repo does implement all three checks:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

The unvalidated price flows through `LRTOracle.getAssetPrice()` → `LRTDepositPool.getRsETHAmountToMint()` → `depositAsset()` / `depositETH()`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [3](#0-2) 

---

### Impact Explanation

**Scenario A — Negative/zero price wraps to `uint256` max:**
If Chainlink returns `price <= 0` (circuit breaker, depegging event), `uint256(price)` either returns 0 (user deposits and receives 0 rsETH, losing their LST) or wraps to `~2^256` (if negative), causing `getRsETHAmountToMint` to overflow and revert — permanently blocking deposits for that asset until the oracle recovers.

**Scenario B — Stale price during market volatility:**
If the feed is stale and the real market price has dropped significantly, a depositor can mint rsETH at the inflated stale price, receiving more rsETH than the current fair value of their deposit. This dilutes existing rsETH holders and constitutes theft of unclaimed yield / protocol value.

**Impact: Temporary freezing of funds (deposits blocked) or theft of unclaimed yield.**

---

### Likelihood Explanation

Chainlink feeds do go stale during network congestion, node outages, or when the price moves within the deviation threshold for an extended period. The affected oracle (`ChainlinkPriceOracle`) is the production oracle for LST assets (stETH, etc.) on Ethereum mainnet. Any depositor can trigger the vulnerable path permissionlessly via `depositAsset()`. Likelihood is **Low** (requires an oracle anomaly) but the path is fully permissionless.

---

### Recommendation

Apply the same validation pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    // Per-feed heartbeat check (e.g., store heartbeat per asset):
    // if (block.timestamp - updatedAt > heartbeat[asset]) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Store a per-asset heartbeat (since each feed has a different update frequency) and validate `block.timestamp - updatedAt <= heartbeat[asset]`.

---

### Proof of Concept

1. Assume the stETH/ETH Chainlink feed goes stale (last update was 2 hours ago, heartbeat is 1 hour) and the real stETH price has dropped 5% from the cached value.
2. Attacker calls `LRTDepositPool.depositAsset(stETH, amount, 0, "")`.
3. `_beforeDeposit` → `getRsETHAmountToMint` → `LRTOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)`.
4. `latestRoundData()` returns the stale (inflated) price; no staleness check exists.
5. `rsethAmountToMint = (amount * stalePriceInflated) / rsETHPrice` — attacker receives ~5% more rsETH than fair value.
6. Attacker redeems rsETH later at the correct price, extracting value from the protocol.

The root cause is exclusively in `ChainlinkPriceOracle.getAssetPrice()` at line 52–54. [4](#0-3)
