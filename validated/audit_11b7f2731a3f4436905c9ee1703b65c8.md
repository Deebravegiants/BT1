Audit Report

## Title
RSETH Daily Mint Limit Blocks `_updateRsETHPrice()` When Protocol Fee Minting Is Required, Causing Oracle DoS and Theft of Unclaimed Yield — (File: `contracts/LRTOracle.sol`)

## Summary

`LRTOracle._updateRsETHPrice()` calls `IRSETH.mint()` to collect protocol fees without any error isolation. `RSETH.mint()` applies a `checkDailyMintLimit` modifier that accumulates all rsETH minted in the current period — including ordinary user deposits. Once user deposits exhaust `maxMintAmountPerDay`, every subsequent call to `updateRSETHPrice()` that would generate a non-zero protocol fee reverts entirely, because `rsETHPrice = newRsETHPrice` is written unconditionally after the failing mint. The price oracle is frozen for up to 24 hours, during which depositors receive excess rsETH at the stale lower price, permanently diluting existing holders' unclaimed yield.

## Finding Description

**Two independent daily mint counters with no cross-awareness:**

`LRTOracle` maintains its own oracle-level fee guard, `_checkAndUpdateDailyFeeMintLimit`, which tracks `currentPeriodMintedFeeAmount` against `maxFeeMintAmountPerDay` (`LRTOracle.sol` lines 197–210). This check passes independently of how much rsETH users have minted that day.

The actual mint call that follows routes through `RSETH.mint()` (`LRTOracle.sol` line 306):
```solidity
IRSETH(rsETHTokenAddress).mint(treasury, rsethAmountToMintAsProtocolFee);
```

`RSETH.mint()` applies the `checkDailyMintLimit` modifier (`RSETH.sol` lines 42–56), which tracks `currentPeriodMintedAmount` — a counter that accumulates **all** rsETH minted in the period, including user deposits via `LRTDepositPool._mintRsETH()` → `IRSETH.mint()` (`LRTDepositPool.sol` lines 686–690).

**No error isolation around the mint call:**

`rsETHPrice = newRsETHPrice` is written at line 313, unconditionally after the `IRSETH.mint()` call at line 306, with no `try/catch`. If `IRSETH.mint()` reverts with `DailyMintLimitExceeded`, the entire `_updateRsETHPrice()` call reverts and the price is never updated.

**Exploit path:**
1. Attacker reads `RSETH.remainingDailyMintLimit()` to get remaining capacity `R`.
2. Attacker calls `LRTDepositPool.depositETH{value: R_in_ETH}(...)`, minting exactly `R` rsETH. `currentPeriodMintedAmount == maxMintAmountPerDay`.
3. EigenLayer rewards accrue; `totalETHInProtocol > previousTVL`, so `protocolFeeInETH > 0`.
4. Anyone calls `LRTOracle.updateRSETHPrice()` (public, no access control).
5. `_checkAndUpdateDailyFeeMintLimit` passes (oracle-level counter is within its own limit).
6. `IRSETH.mint(treasury, rsethAmountToMintAsProtocolFee)` fires `checkDailyMintLimit`: `currentPeriodMintedAmount + rsethAmountToMintAsProtocolFee > maxMintAmountPerDay` → `revert DailyMintLimitExceeded`.
7. Entire call reverts. `rsETHPrice` remains at its stale (lower) value.
8. Attacker deposits additional assets at the stale lower price, receiving excess rsETH relative to the true price.
9. After 24 hours `currentPeriodMintedAmount` resets, but the dilution and missed protocol fee are permanent.

## Impact Explanation

The frozen stale price causes new depositors to receive more rsETH than the true price entitles them to. This directly dilutes existing holders' share of accrued, unclaimed yield — constituting **theft of unclaimed yield**. The protocol fee for the frozen period is also permanently unrecoverable. This matches the allowed impact: **High — Theft of unclaimed yield**.

## Likelihood Explanation

- `updateRSETHPrice()` is a public function; any caller can trigger the revert once the limit is reached.
- `remainingDailyMintLimit()` is a public view function, allowing the attacker to precisely measure the remaining capacity before depositing.
- The attacker receives rsETH in return for their deposit, so net capital at risk is only price-movement exposure and gas — not a pure burn.
- The attack is repeatable every 24 hours with the same capital.
- Likelihood: **Medium** — requires meaningful capital but is economically rational given the attacker profits from the stale price window.

## Recommendation

Decouple the price write from the fee mint. Apply a `try/catch` around `IRSETH.mint()` so that a failed fee mint does not block the price update:

```solidity
if (rsethAmountToMintAsProtocolFee > 0) {
    try IRSETH(rsETHTokenAddress).mint(treasury, rsethAmountToMintAsProtocolFee) {
        emit FeeMinted(treasury, rsethAmountToMintAsProtocolFee);
    } catch {
        // fee deferred; roll back oracle counter to avoid phantom accounting
        currentPeriodMintedFeeAmount -= rsethAmountToMintAsProtocolFee;
    }
}
rsETHPrice = newRsETHPrice;
```

Alternatively, move `rsETHPrice = newRsETHPrice` before the fee-minting block so the price is always committed regardless of fee-mint success.

## Proof of Concept

```
1. Deploy/fork with maxMintAmountPerDay = X on RSETH.
2. Call RSETH.remainingDailyMintLimit() → returns R.
3. Attacker calls LRTDepositPool.depositETH{value: R_in_ETH}(0, "").
   → RSETH.currentPeriodMintedAmount == maxMintAmountPerDay.
4. Simulate EigenLayer reward accrual so totalETHInProtocol > previousTVL.
5. Call LRTOracle.updateRSETHPrice().
   → Execution reaches IRSETH.mint(treasury, fee) at LRTOracle.sol:306.
   → RSETH.checkDailyMintLimit reverts with DailyMintLimitExceeded.
   → rsETHPrice is NOT updated.
6. Assert rsETHPrice == old stale value.
7. Attacker calls LRTDepositPool.depositETH{value: V}(0, "").
   → rsethAmountToMint = V * assetPrice / rsETHPrice (stale, lower price → more rsETH).
8. Warp +24h. Call updateRSETHPrice() → succeeds, price updates upward.
9. Assert attacker's rsETH balance is worth more ETH than V deposited.
   → Existing holders' rsETH is worth less than it should be (dilution = theft of unclaimed yield).
```
