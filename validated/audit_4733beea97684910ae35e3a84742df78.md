Audit Report

## Title
Cross-Token Arbitrage via 1:1 Mint/Burn Across Multiple Allowed Tokens - (File: `contracts/L2/RsETHTokenWrapper.sol`)

## Summary
`RsETHTokenWrapper` maintains a whitelist of allowed altRsETH tokens and mints/burns `wrsETH` strictly 1:1 for any of them, with no price accounting. When two allowed tokens simultaneously exist in the whitelist — an expected operational state introduced by `reinitialize` — any unprivileged caller can deposit the cheaper token and withdraw the more expensive one, extracting the price difference from the wrapper's reserves and leaving honest depositors unable to redeem their original token.

## Finding Description
`_deposit` and `_withdraw` each perform only a single guard — `if (!allowedTokens[_asset]) revert TokenNotAllowed()` — before minting or burning `wrsETH` 1:1 against whichever allowed token the caller supplies:

```solidity
// L134-141
function _deposit(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    ERC20Upgradeable(_asset).safeTransferFrom(msg.sender, address(this), _amount);
    _mint(_to, _amount);
    ...
}

// L120-128
function _withdraw(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    _burn(msg.sender, _amount);
    ERC20Upgradeable(_asset).safeTransfer(_to, _amount);
    ...
}
```

The `reinitialize` function (callable by `DEFAULT_ADMIN_ROLE`) was specifically introduced to add a second altRsETH token to the whitelist during bridge migrations, and `addAllowedToken` (callable by `TIMELOCK_ROLE`) can add further tokens at any time. Neither function removes the previously allowed token. The `allowedTokens` mapping therefore holds multiple active entries simultaneously, which is the intended operational state during migration windows.

During any such window, the two bridge representations of rsETH routinely trade at different prices on secondary markets due to liquidity imbalances or bridge-specific risk premiums. Because `_deposit` and `_withdraw` are unrestricted public entry points, any caller can atomically:
1. Call `deposit(cheaperToken, N)` — transfers N cheaper tokens in, mints N `wrsETH`.
2. Call `withdraw(expensiveToken, N)` — burns N `wrsETH`, transfers N expensive tokens out.

No oracle, no price check, and no per-token accounting exists to prevent this. The only existing guard (`allowedTokens`) is satisfied by both tokens.

## Impact Explanation
**Critical — Direct theft of user funds at rest.** Honest users who deposited the higher-value altRsETH token have their collateral drained. The attacker receives more ETH-denominated value than deposited. The wrapper's reserves for the premium token are depleted, and affected users can no longer redeem their `wrsETH` for the token they originally deposited. This matches the allowed impact: *Direct theft of any user funds, whether at-rest or in-motion*.

## Likelihood Explanation
**Medium.** The `reinitialize` function was explicitly introduced to support a second altRsETH token, confirming simultaneous multi-token operation is an intended production scenario. During any bridge migration window — when both old and new altRsETH tokens are simultaneously whitelisted — the two tokens routinely trade at different prices. No privileged access is required to execute the attack once two tokens are whitelisted; `deposit` and `withdraw` are fully public. The attack is atomic, repeatable, and requires no victim interaction.

## Recommendation
1. **Enforce single-token invariant during transitions**: In `reinitialize` and `addAllowedToken`, remove (or require prior removal of) the existing allowed token before adding the new one, so at most one token is ever active at a time.
2. **Or, implement per-token reserve accounting**: Track deposits per token and restrict withdrawals of token X to the amount of token X actually held, preventing cross-token redemption.
3. **At minimum**, add a check in `_withdraw` that `ERC20Upgradeable(_asset).balanceOf(address(this))` is sufficient relative to the amount being withdrawn, and track per-token liabilities separately from the aggregate `wrsETH` supply.

## Proof of Concept
**Setup**: Wrapper holds 1,000 `altRsETH_A` (market price 1.00 ETH each), backing 1,000 `wrsETH`. Admin calls `reinitialize(altRsETH_B)` where `altRsETH_B` trades at 0.95 ETH each.

**Attack** (single transaction or two sequential calls):
```
1. attacker.approve(wrapper, 1000 altRsETH_B)
2. wrapper.deposit(altRsETH_B, 1000)
   → transfers 1,000 altRsETH_B (~950 ETH) into wrapper
   → mints 1,000 wrsETH to attacker
3. wrapper.withdraw(altRsETH_A, 1000)
   → burns 1,000 wrsETH
   → transfers 1,000 altRsETH_A (~1,000 ETH) to attacker
```

**Result**: Attacker profits ~50 ETH. Original `altRsETH_A` depositors cannot redeem.

**Foundry invariant test**: Deploy two mock ERC20 tokens at different prices. Call `reinitialize` to add the second. Assert `invariant: for each allowed token T, wrapper.balanceOf(T) >= wrsETH.totalSupply() * priceOf(T) / priceOf(primaryToken)`. The invariant breaks immediately after the cross-token deposit/withdraw sequence.
