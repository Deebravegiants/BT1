### Title
Forged Foreign-Chain Verification via Cross-Request Payload Hash Substitution — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that a signature is valid over `response.payload_hash` using the root public key, but never checks that `response.payload_hash` is actually derived from the original pending `request`'s transaction data. A single malicious attested MPC participant (strictly below the signing threshold) can take a legitimately-produced threshold signature for one foreign-chain request and submit it as the response to a completely different pending request, causing the contract to resolve that request with a forged payload hash.

---

### Finding Description

In `respond_verify_foreign_tx` the contract performs three checks:

1. Caller is an attested participant.
2. The ECDSA signature in `response` is valid over `response.payload_hash` under the root public key.
3. The `request` key exists in `pending_verify_foreign_tx_requests`. [1](#0-0) 

What is **absent** is any binding between `response.payload_hash` and the `request` argument. The hash is defined as:

```
payload_hash = SHA-256(borsh(ForeignTxSignPayload { request, extracted_values }))
``` [2](#0-1) 

Because `request` (including the foreign-chain `tx_id`) is embedded inside the hash, a hash computed for request B is cryptographically distinct from the hash for request A. The contract never reconstructs or compares this; it simply trusts whatever 32-byte value the responding node supplies.

On the node side, the signing request is built with a **zero tweak** (`Tweak::new([0u8; 32])`), meaning the root key signs directly: [3](#0-2) 

Any attested participant who has observed a completed threshold signing session for request B therefore holds a valid `(payload_hash_B, sig_B)` pair that will pass the contract's signature check for **any** pending request, because the check is purely `verify(sig_B, payload_hash_B, root_pk)`. [4](#0-3) 

The `args_into_verify_foreign_tx_request` conversion that builds the pending-map key from user-supplied arguments does not embed any caller identity or nonce, so the key is fully predictable and stable: [5](#0-4) 

---

### Impact Explanation

A malicious single attested participant (below signing threshold) can resolve any pending `verify_foreign_transaction` request with a `payload_hash` that corresponds to a **different** foreign-chain transaction. The `VerifyForeignTransactionResponse` delivered to the waiting caller contains `payload_hash_B` and `sig_B`. Any bridge contract that:

- verifies `sig_B` over `payload_hash_B` against the root public key (passes), but
- does **not** independently reconstruct the expected payload hash from the original `tx_id` and the extracted values

will accept this as proof that the original transaction was verified, enabling invalid bridge execution or double-spend conditions. This matches the allowed impact: **High — forged foreign-chain verification that causes invalid bridge execution**.

The SDK helper `ForeignChainSignatureVerifier::verify_signature` does perform the binding check: [6](#0-5) 

However, the on-chain contract enforces no such invariant, so bridge contracts that do not use the SDK, or that only check the ECDSA signature, are fully exposed.

---

### Likelihood Explanation

The attacker must be a single attested MPC participant — strictly below the signing threshold. No threshold collusion is required. The attacker participates honestly in the threshold signing for request B (obtaining a valid `(payload_hash_B, sig_B)`), then races to call `respond_verify_foreign_tx(request_A, {payload_hash_B, sig_B})` before any honest node resolves request A. Because `resolve_yields_for` drains the queue on first call, the first responder wins. A malicious participant who monitors the NEAR chain for pending requests and participates in any concurrent signing session has everything needed.

---

### Recommendation

The contract must bind the response to the originating request. Since `payload_hash` encodes the `ForeignChainRpcRequest` inside it, the fix is to verify that the `request` field embedded in the signed payload matches the `request` argument supplied to `respond_verify_foreign_tx`. Concretely:

- Require the responding node to also supply the `extracted_values` (or a commitment to them).
- Recompute `expected_prefix = SHA-256(borsh(request))` and verify it is a prefix of the pre-image, **or**
- Require the node to supply the full `ForeignTxSignPayload`, recompute `SHA-256(borsh(payload))` on-chain, and assert it equals `response.payload_hash` **and** that `payload.request == request`.

At minimum, document that callers **must** use `ForeignChainSignatureVerifier` and that the on-chain contract does not enforce payload-to-request binding.

### Proof of Concept

```
1. Alice calls verify_foreign_transaction(tx_id=A)  → request_A queued
2. Bob   calls verify_foreign_transaction(tx_id=B)  → request_B queued
3. Honest MPC nodes run threshold signing for request_B:
       payload_B      = ForeignTxSignPayloadV1 { request_B, extracted_values_B }
       payload_hash_B = SHA-256(borsh(payload_B))
       sig_B          = threshold_sign(payload_hash_B, root_key)
4. Malicious attested node (participant in step 3) calls:
       respond_verify_foreign_tx(request_A, { payload_hash_B, sig_B })
5. Contract checks:
       verify_ecdsa(sig_B, payload_hash_B, root_pk)  → OK  (valid threshold sig)
       pending_verify_foreign_tx_requests[request_A]  → OK  (Alice's yield exists)
   → resolves Alice's yield with { payload_hash_B, sig_B }
6. Alice's bridge contract receives { payload_hash_B, sig_B }.
   It calls verify_ecdsa(sig_B, payload_hash_B, root_pk) → passes.
   It does NOT check payload_hash_B encodes tx_id=A.
   → Bridge accepts forged proof that tx_id=A was verified on-chain.
```
