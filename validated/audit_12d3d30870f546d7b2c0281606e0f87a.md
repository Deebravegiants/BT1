Audit Report

## Title
Aave v3 Pool Pause Causes Permanent DOS of `completeWithdrawal` for ETH Withdrawers — (File: contracts/LRTWithdrawalManager.sol)

## Summary

`LRTWithdrawalManager._processWithdrawalCompletion` calls `_withdrawFromAave` with no error handling when the contract's ETH balance is insufficient to cover a user's withdrawal. If the Aave v3 WETH pool is paused, `aaveWETHGateway.withdrawETH` reverts unconditionally, propagating the revert through `completeWithdrawal`. Because rsETH is burned during `unlockQueue` (before `completeWithdrawal` is called), affected users lose their rsETH with no path to recover their ETH for the duration of the pause. No admin escape hatch bypasses this: `emergencyWithdrawFromAave` and `setAaveIntegrationEnabled(false)` both also call `_withdrawFromAave` and revert identically.

## Finding Description

**Asymmetric error handling between deposit and withdrawal paths:**

The deposit path in `unlockQueue` (line 311) wraps `depositToAaveExternal` in a `try/catch`, silently continuing on failure: [1](#0-0) 

The withdrawal path in `_processWithdrawalCompletion` (line 724) calls `_withdrawFromAave` with no equivalent protection: [2](#0-1) 

`_withdrawFromAave` calls `aaveWETHGateway.withdrawETH` directly with no error handling: [3](#0-2) 

When the Aave v3 pool is paused, `withdrawETH` reverts unconditionally. This revert propagates through `completeWithdrawal`, freezing all ETH withdrawers whose funds are held in Aave.

**No admin escape hatch works while Aave is paused:**

- `emergencyWithdrawFromAave` (line 560) calls `_withdrawFromAave` → reverts: [4](#0-3) 

- `setAaveIntegrationEnabled(false)` (line 495) calls `_withdrawFromAave` before disabling → reverts: [5](#0-4) 

There is no code path that sets `isAaveIntegrationEnabled = false` without first attempting a withdrawal from Aave.

**Exploit flow:**
1. Aave integration is enabled; `unlockQueue` burns rsETH and deposits ETH to Aave via the try/catch path.
2. Aave governance pauses the WETH pool (a documented, exercised mechanism).
3. User calls `completeWithdrawal` — their rsETH is already burned.
4. `_processWithdrawalCompletion` sees `address(this).balance < request.expectedAssetAmount`, calls `_withdrawFromAave`.
5. `aaveWETHGateway.withdrawETH` reverts due to pool pause.
6. Entire transaction reverts. User cannot recover ETH. Admin functions also revert.

## Impact Explanation

**Medium — Temporary freezing of funds.** Users whose rsETH has already been burned by `unlockQueue` cannot complete ETH withdrawals for the duration of the Aave pause. Their rsETH is gone and their ETH is locked in Aave with no protocol-level fallback. If the pause is extended or the market is deprecated, this escalates to permanent freezing.

## Likelihood Explanation

Aave v3 has a well-documented pool-level and reserve-level pause mechanism exercised by Aave governance and the Aave Guardian. The WETH market on Aave v3 Ethereum is one of the largest markets and has been subject to emergency pauses historically. The Aave integration is opt-in but, once enabled, routes all unlocked ETH into Aave, making every ETH withdrawer dependent on Aave liveness. The trigger is an external governance action on a live, production protocol — not a theoretical or contrived scenario.

## Recommendation

1. **Wrap `_withdrawFromAave` in `_processWithdrawalCompletion` in a `try/catch`**: On failure, pay the user from whatever native ETH balance is available and record the shortfall for later settlement.
2. **Add a force-disable admin function**: A separate function that sets `isAaveIntegrationEnabled = false` *without* calling `_withdrawFromAave`, allowing the protocol to degrade gracefully when Aave is paused and recover ETH manually once the pause lifts.

## Proof of Concept

**Foundry fork test outline:**

```solidity
// Fork Ethereum mainnet at a block where Aave WETH pool is active
// 1. Enable Aave integration, call unlockQueue → rsETH burned, ETH in Aave
// 2. Prank as Aave Guardian: call aavePool.setReservePause(WETH, true)
// 3. Advance blocks past withdrawalDelayBlocks
// 4. Call completeWithdrawal(ETH_TOKEN, requestId, referralId)
// 5. Assert: transaction reverts (DOS confirmed)
// 6. Call emergencyWithdrawFromAave(amount) → also reverts
// 7. Call setAaveIntegrationEnabled(false) → also reverts
// 8. Assert: user's rsETH balance is 0 (already burned), ETH balance unchanged
```

The minimal sequence is: `unlockQueue` (burns rsETH, deposits to Aave) → Aave pool pause → `completeWithdrawal` reverts → all admin escape hatches revert → user funds frozen.
