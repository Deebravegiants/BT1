### Title
Missing Minimum Output (Slippage) Parameter in L2 Pool `deposit()` Functions - (File: contracts/pools/RSETHPoolV3.sol)

### Summary
All L2 pool `deposit()` functions lack a `minRsETHAmount` slippage guard. Users who preview their expected output via `viewSwapRsETHAmountAndFee()` off-chain have no on-chain guarantee they will receive at least that amount when the oracle rate is updated between their query and their transaction's execution.

### Finding Description
The mainnet `LRTDepositPool` correctly accepts a `minRSETHAmountExpected` parameter in both `depositETH()` and `depositAsset()`, enforcing a minimum output check before minting rsETH: [1](#0-0) 

However, the analogous L2 pool `deposit()` functions in `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolNoWrapper`, and `RSETHPool` accept no such parameter: [2](#0-1) [3](#0-2) 

The rsETH amount minted is computed entirely from the oracle rate at execution time: [4](#0-3) 

The L2 oracle (`rsETHOracle`) is a cross-chain rate provider that is updated periodically, not every block. Between the moment a user calls `viewSwapRsETHAmountAndFee()` off-chain and the moment their `deposit()` transaction is mined, the oracle rate can be updated (rsETH appreciates as staking rewards accrue), causing the user to receive fewer wrsETH/rsETH than the UI quoted — with no on-chain recourse. The same pattern exists in `RSETHPoolNoWrapper.deposit()`: [5](#0-4) 

and `RSETHPoolV3ExternalBridge.deposit()`: [6](#0-5) 

### Impact Explanation
**Low.** The user deposits ETH or an LST and receives fewer wrsETH/rsETH tokens than the protocol's own view function quoted them. The deposited value is not stolen — the fewer tokens received are each worth proportionally more ETH — but the contract fails to deliver the promised token quantity. This breaks the user's expectation set by `viewSwapRsETHAmountAndFee()` and can cause downstream failures (e.g., a user depositing to meet a specific collateral threshold receives less than required).

### Likelihood Explanation
**Low.** The rsETH oracle rate on L2 is updated periodically via cross-chain rate broadcasts, not every block. The discrepancy between a user's off-chain quote and their on-chain execution is therefore bounded by the rate change since the last oracle update. Because rsETH appreciates slowly (staking rewards), the shortfall per transaction is small but non-zero and grows with deposit size and time elapsed since the last oracle push. No attacker manipulation is required; the condition arises naturally from normal oracle operation.

### Recommendation
Add a `minRsETHAmount` parameter to all four `deposit()` overloads across `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolNoWrapper`, and `RSETHPool`, mirroring the pattern already used in `LRTDepositPool.depositETH()`. Revert if the computed `rsETHAmount` is below the caller-supplied minimum:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmount)
    external payable nonReentrant whenNotPaused
    limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    require(rsETHAmount >= minRsETHAmount, "Slippage: insufficient output");
    ...
}
```

### Proof of Concept
1. User calls `viewSwapRsETHAmountAndFee(1 ether)` off-chain and sees they will receive `X` wrsETH at the current oracle rate `R`.
2. Before the user's transaction is mined, the L2 oracle is updated to rate `R' > R` (rsETH has appreciated).
3. User's `deposit{value: 1 ether}(referralId)` executes; `rsETHAmount = (1e18 - fee) * 1e18 / R'` which is strictly less than `X`.
4. The user receives fewer wrsETH than quoted with no revert and no recourse.

The gap is proportional to the oracle rate delta and the deposit size. For a 1 ETH deposit and a 0.1% oracle rate increase (plausible over a multi-day oracle update interval), the user receives approximately 0.001 ETH worth fewer wrsETH than expected.

        external
        payable
        nonReentrant
        whenNotPaused
        limitDailyMint(msg.value, ETH_IDENTIFIER)
    {
        if (!isEthDepositEnabled) revert EthDepositDisabled();
        uint256 amount = msg.value;

        if (amount == 0) revert InvalidAmount();

        (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);

        feeEarnedInETH += fee;

    }

    function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
    }
