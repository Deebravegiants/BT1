### Title
Unauthorized `directDeposits` in Dijkstra Sub-Transactions Allow Griefing of Stake Credential Deregistration — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubEntities.hs`)

---

### Summary

In the Dijkstra era, the `SUBENTITIES` ledger rule applies `directDeposits` from sub-transaction bodies to registered account balances with **no authorization check from the target account owner**. Any unprivileged transaction submitter can deposit an arbitrary coin amount into any registered stake account, forcing the account owner to pay an extra withdrawal transaction fee before they can deregister their stake credential.

---

### Finding Description

The `dijkstraSubEntitiesTransition` function in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubEntities.hs` processes the `directDeposits` field of a sub-transaction body:

```haskell
let directDeposits = tx ^. bodyTxL . directDepositsTxBodyL
    accountsAfterSubCerts = certStateAfterSubCerts ^. certDStateL . accountsL
failOnJust (directDepositsMissingAccounts directDeposits accountsAfterSubCerts) $
  injectFailure . SubDirectDepositsToMissingAccounts

pure $ certStateAfterSubCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
``` [1](#0-0) 

The sole validation is `directDepositsMissingAccounts`, which only checks that the target `AccountAddress` credentials are registered. There is **no witness requirement** and **no authorization check** from the target account owner.

`applyDirectDeposits` unconditionally adds coin to `balanceAccountStateL` for every entry in the map:

```haskell
applyDirectDeposits (DirectDeposits dd) =
  updateAccountBalances
    (\amount account -> addCompactCoin amount (account ^. balanceAccountStateL))
    dd
``` [2](#0-1) 

The `balanceAccountStateL` field is the same balance that the `DELEG` rule requires to be zero before a stake credential can be deregistered:

```haskell
checkStakeKeyHasZeroRewardBalance = do
  accountState <- mAccountState
  let balanceCompact = accountState ^. balanceAccountStateL
  guard (balanceCompact /= mempty)
  Just $ fromCompact balanceCompact
failOnJust
  checkStakeKeyHasZeroRewardBalance
  (injectFailure . StakeKeyHasNonZeroAccountBalanceDELEG)
``` [3](#0-2) 

The `directDeposits` field is present in both the top-level and sub-transaction Dijkstra body: [4](#0-3) 

---

### Impact Explanation

An attacker can deposit 1 lovelace into a victim's registered account at any time. Because `balanceAccountStateL` is now non-zero, the victim's `UnRegDepositTxCert` (deregistration) will be rejected with `StakeKeyHasNonZeroAccountBalanceDELEG`. The victim must first submit a withdrawal transaction (paying a transaction fee) to drain the balance before deregistering. The attacker can repeat the deposit immediately after each withdrawal, keeping the victim's account non-zero indefinitely at minimal cost (1 lovelace + fees per round). This modifies the victim's account balance and withdrawal/deregistration flow outside design parameters without any authorization from the account owner.

**Impact class**: Medium — attacker-controlled sub-transactions modify rewards/withdrawals outside design parameters, forcing the victim to pay fees and blocking deregistration.

---

### Likelihood Explanation

The attack requires no special privilege — any transaction submitter on the Dijkstra network can craft a top-level transaction containing a sub-transaction with a `directDeposits` entry targeting any registered credential. The cost per attack round is 1 lovelace plus the transaction fee. The victim's registered credential is publicly observable on-chain, making targeting trivial.

---

### Recommendation

Add an authorization check in the `SUBENTITIES` (and `ENTITIES`) rule for `directDeposits`: require that each target `AccountAddress` credential is either (a) controlled by the sub-transaction author (i.e., a witness for the target credential is present in the sub-transaction), or (b) restrict `directDeposits` to entries whose target credential matches a credential the sub-transaction author controls. This mirrors the existing witness requirement for withdrawals and deregistration certificates.

---

### Proof of Concept

1. Victim registers stake credential `C` and accumulates rewards; then withdraws all rewards so `balanceAccountStateL C = 0`.
2. Victim constructs `UnRegDepositTxCert C deposit` and is ready to submit.
3. Attacker observes `C` on-chain and submits a Dijkstra top-level transaction containing a sub-transaction with:
   ```
   directDeposits = { AccountAddress(network, C) => 1 }
   ```
   funded by the attacker's own UTxO input in the sub-transaction.
4. The `SUBENTITIES` rule applies `applyDirectDeposits`, setting `balanceAccountStateL C = 1`.
5. Victim's deregistration transaction fails: `StakeKeyHasNonZeroAccountBalanceDELEG (Coin 1)`.
6. Victim must submit a withdrawal transaction (paying fees) to drain the 1 lovelace, then retry deregistration.
7. Attacker repeats step 3 after each victim withdrawal, indefinitely blocking deregistration at ~1 lovelace cost per round.
