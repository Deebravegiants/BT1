### Title
Expired DRep Can Vote and Self-Resurrect, Bypassing Activity Expiry Mechanism - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Gov.hs`)

### Summary
The Conway governance `GOV` transition rule validates DRep voters by checking only that the credential exists in the registered DRep map (`vsDReps`) and that the action type permits DRep voting. It does not check whether the DRep is currently active (non-expired). An expired DRep credential holder can submit a vote transaction that passes all ledger checks. The `CERTS` rule then updates the DRep's expiry via `updateVotingDRepExpiries`, effectively resurrecting the expired DRep. In the following epoch, the resurrected DRep's stored votes count toward governance ratification, potentially enacting governance actions that should not have passed.

### Finding Description

**Voter existence check — does not filter expired DReps**

In `conwayGovTransition`, the set of known DReps is built directly from `vsDReps`:

```haskell
knownDReps = vsDReps certVState
``` [1](#0-0) 

`vsDReps` contains **all registered DReps**, including those whose `drepExpiry` has already passed. An expired DRep is never removed from this map until it explicitly unregisters.

The voter existence check (`VotersDoNotExist`) uses `internMap cred knownDReps`, which succeeds for any registered DRep regardless of expiry:

```haskell
DRepVoter cred -> DRepVoter <$> internMap cred knownDReps
``` [2](#0-1) 

**Voter validity check — only checks action type, not DRep expiry**

`checkVotersAreValid` dispatches on voter type. For `CommitteeVoter`, it calls `isCommitteeVotingAllowed currentEpoch committeeState`, which internally checks whether each member is active (not expired, not resigned). For `DRepVoter`, it calls only `isDRepVotingAllowed (gasAction gas)`, which checks only whether the action type permits DRep voting — **no expiry check**:

```haskell
CommitteeVoter {} -> isCommitteeVotingAllowed currentEpoch committeeState (gasAction gas)
DRepVoter {}      -> isDRepVotingAllowed (gasAction gas)
StakePoolVoter {} -> isStakePoolVotingAllowed (gasAction gas)
``` [3](#0-2) 

Compare with the committee path, which does check active status via `isActive`:

```haskell
isActive coldKey validUntil =
  case Map.lookup coldKey hotKeys of
    Just (CommitteeMemberResigned _) -> False
    Just _ -> currentEpoch <= validUntil
    Nothing -> False
``` [4](#0-3) 

**Expiry update resurrects the expired DRep**

`updateVotingDRepExpiries` (called in the `CERTS` rule for the same transaction) iterates over all `DRepVoter` entries in the transaction's voting procedures and updates each DRep's expiry to `computeDRepExpiry drepActivity currentEpoch numDormantEpochs`:

```haskell
DRepVoter cred ->
  Map.adjust
    (drepExpiryL .~ computeDRepExpiry drepActivity currentEpoch numDormantEpochs)
    cred
    dreps
``` [5](#0-4) 

Because the `GOV` rule accepts the expired DRep's vote (no expiry check), the transaction succeeds, and the `CERTS` state change is committed. The DRep's expiry is now set to a future epoch — it is resurrected.

**Resurrected DRep votes count at ratification**

At the next epoch boundary, `dRepAcceptedRatio` checks `reCurrentEpoch > drepExpiry drepState`. With the updated expiry, this check now passes, and the DRep's stored `VoteYes` (recorded in `gasDRepVotes` during the epoch) is counted toward ratification:

```haskell
Just drepState
  | reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, skip
  | otherwise ->
      case Map.lookup cred gasDRepVotes of
        Just VoteYes -> (yes + stake, tot + stake)
``` [6](#0-5) 

The DRep's stake is also present in `reDRepDistr` because `computeDRepDistr` includes all entries in `regDReps` (which contains expired DReps):

```haskell
DRepCredential cred
  | Map.member cred regDReps -> updatedDistr
``` [7](#0-6) 

### Impact Explanation

An expired DRep credential holder can unilaterally resurrect their DRep by submitting a vote transaction. After resurrection, the DRep's votes count in governance ratification. If the resurrected DRep's stake is sufficient to tip the threshold for a governance action (e.g., `TreasuryWithdrawals`, `HardForkInitiation`, `ParameterChange`, `NewConstitution`), that action can be enacted when it otherwise would not have been. This constitutes an **unauthorized governance or treasury action enacted** — a Critical impact.

### Likelihood Explanation

Any holder of an expired DRep credential (key or script) can trigger this. No special privileges, governance majority, or leaked keys are required. The attacker only needs to submit a standard vote transaction. DReps expire routinely after `ppDRepActivity` epochs of inactivity, so the pool of exploitable credentials grows over time. The attack is cheap (only transaction fees) and repeatable.

### Recommendation

Add an expiry check for `DRepVoter` in `checkVotersAreValid`, analogous to the existing check for `CommitteeVoter`. Specifically, look up the DRep's state in `vsDReps` and reject the vote if `currentEpoch > drepExpiry drepState + numDormantEpochs`. Alternatively, filter `knownDReps` at the point of construction (line 472) to exclude expired DReps, so that `VotersDoNotExist` fires for expired DRep credentials.

### Proof of Concept

1. Register a DRep and delegate stake to it.
2. Wait `ppDRepActivity + 1` epochs without voting (DRep expires).
3. Confirm `isDRepExpired drep == True`.
4. Submit a `VoteYes` transaction for an active governance action using the expired DRep credential.
5. Observe the transaction is accepted (no `VotersDoNotExist` or `DisallowedVoters` failure).
6. Observe `drepExpiry` in `vsDReps` is now updated to a future epoch (DRep resurrected).
7. At the next epoch boundary, observe the DRep's `VoteYes` is counted in `dRepAcceptedRatio`, potentially causing the governance action to be ratified.
