### Title
Unbounded Sub-Transaction Count in `SUBLEDGERS` Rule Enables Disproportionate Per-Transaction Validation Work - (File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubLedgers.hs)

---

### Summary

The Dijkstra era introduces nested sub-transactions (`sub_transactions`) inside a top-level transaction body. The `SUBLEDGERS` rule processes each sub-transaction by iterating over the full list with `foldM`, applying the complete `SUBLEDGER` pipeline (certificate processing via `SUBENTITIES`, governance processing via `SUBGOV`, and UTxO witnessing via `SUBUTXOW`) to every sub-transaction. No explicit maximum count on sub-transactions is enforced beyond the overall `maxTxSize` byte limit. Because `maxTxSize` was calibrated for pre-Dijkstra transactions that carry no sub-transactions, a single crafted Dijkstra transaction can trigger N full ledger-rule evaluations for the price of one, where N is bounded only by `maxTxSize / minSubTxSize`. This allows an unprivileged transaction sender to cause validation work that exceeds the intended per-transaction limit.

---

### Finding Description

**Root cause — `dijkstraSubLedgersTransition`:**

```haskell
dijkstraSubLedgersTransition = do
  TRC (env, ledgerState, subTxs) <- judgmentContext
  foldM
    ( \ls subTx ->
        trans @(EraRule "SUBLEDGER" era) $ TRC (env, ls, subTx)
    )
    ledgerState
    subTxs
``` [1](#0-0) 

The signal type for `SUBLEDGERS` is `[StAnnTx SubTx era]` — an unbounded list. [2](#0-1) 

**No maximum count in the CDDL spec:**

```
sub_transactions = nonempty_oset<sub_transaction>
``` [3](#0-2) 

Only a non-empty lower bound is enforced; there is no upper bound.

**No maximum count in the Haskell type:**

```haskell
dtbrSubTransactions :: !(OMap TxId (Tx SubTx era))
``` [4](#0-3) 

The `OMap` has no enforced size limit.

**Each sub-transaction triggers the full `SUBLEDGER` pipeline:**

The `SUBLEDGER` rule calls `SUBENTITIES` (all certificates), `SUBGOV` (all votes and proposals), and `SUBUTXOW` (full UTxO witnessing). [5](#0-4) 

**The only implicit bound is `maxTxSize`:**

```haskell
{- txsize tx ≤ maxTxSize pp -}
runTestOnSignal $ Shelley.validateMaxTxSizeUTxO pp tx
``` [6](#0-5) 

`maxTxSize` bounds the serialized byte count of the whole transaction, but it was designed for pre-Dijkstra transactions that carry no sub-transactions. It does not bound the number of full ledger-rule evaluations triggered per transaction.

**Multiplicative validation work:**

With `maxTxSize = 16 384` bytes and a minimal sub-transaction of ~50 bytes, up to ~327 sub-transactions can be packed into one transaction. Each sub-transaction independently runs `SUBENTITIES` (O(c · log K) for c certificates and K-sized cert state), `SUBGOV` (O(v · log P) for v votes/proposals and P-sized proposals state), and `SUBUTXOW` (O(i · log U) for i inputs and U-sized UTxO). The total validation work grows as N × (c + v + i) × log(state size), which is not proportional to the serialized byte count.

---

### Impact Explanation

**Impact: Medium — Attacker-controlled transactions exceed intended validation limits.**

A single crafted Dijkstra transaction can force every validating node to execute N full sub-ledger rule evaluations (SUBENTITIES + SUBGOV + SUBUTXOW) for the cost of one transaction fee. Because the fee is proportional to serialized size and not to validation work, the attacker pays a fee calibrated for a pre-Dijkstra transaction while imposing N times the validation cost. This exceeds the intended per-transaction validation budget that `maxTxSize` was designed to enforce, and can cause block-validation time to spike beyond the slot interval, threatening liveness.

---

### Likelihood Explanation

**Likelihood: Medium.**

- Any unprivileged user can submit a Dijkstra top-level transaction; no special role or key is required.
- The Dijkstra era is new; `maxTxSize` has not been recalibrated to account for the multiplicative cost of sub-transactions.
- The attacker pays only a byte-proportional fee, not a work-proportional fee.
- The attack requires crafting a valid Dijkstra transaction, which is a non-trivial but achievable engineering task.
- There is no on-chain mechanism that rejects a transaction solely because it contains many sub-transactions.

---

### Recommendation

1. **Introduce a new protocol parameter** `maxSubTransactions :: Word16` (analogous to `maxCollateralInputs`) and enforce it in the `SUBLEDGERS` rule or in the `UTXO` rule before dispatching to `SUBLEDGERS`.
2. **Alternatively**, recalibrate `maxTxSize` for the Dijkstra era to account for the multiplicative validation cost introduced by sub-transactions, and document the safe upper bound.
3. **Add a comment** in `SubLedgers.hs` and in the CDDL spec noting the implicit bound and the assumption that `maxTxSize` is sufficient, so future parameter changes are made with awareness of this constraint.

---

### Proof of Concept

1. Construct a Dijkstra top-level transaction body (`DijkstraTxBodyRaw TopTx`) with `dtbrSubTransactions` set to an `OMap` containing N minimal sub-transactions, where N = ⌊maxTxSize / minSubTxSize⌋ (e.g., N ≈ 327 for `maxTxSize = 16 384` and `minSubTxSize ≈ 50`).
2. Each sub-transaction body (`DijkstraSubTxBodyRaw`) contains the maximum number of certificates and voting procedures that fit within its byte budget.
3. Submit the transaction. Every validating node will execute `dijkstraSubLedgersTransition` [1](#0-0) , calling `trans @(EraRule "SUBLEDGER" era)` N times in sequence via `foldM`.
4. Each call runs the full `SUBLEDGER` pipeline [5](#0-4) , including `SUBENTITIES`, `SUBGOV`, and `SUBUTXOW`.
5. The total validation work is O(N × work_per_subtx), which is not bounded by `maxTxSize` alone, demonstrating that the intended per-transaction validation limit is exceeded.
