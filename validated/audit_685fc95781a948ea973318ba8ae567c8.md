### Title
Missing `ppuWellFormed` Validation for `maxRefScriptSizePerTx` and `maxRefScriptSizePerBlock` Allows Governance to Set Zero, Permanently Disabling All Transactions with Reference Scripts in Dijkstra Era - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/PParams.hs`)

---

### Summary

In the Dijkstra era, the `ppuWellFormed` implementation for `DijkstraEra` does not validate that the new parameters `maxRefScriptSizePerTx` and `maxRefScriptSizePerBlock` are non-zero. A governance `ParameterChange` action setting either of these to `0` would be accepted as well-formed and enacted. Once enacted, every transaction that includes any reference script would permanently fail validation — because `totalRefScriptSize > 0` is always `> maxRefScriptSizePerTx = 0` — with no on-chain recovery path short of a hard fork.

---

### Finding Description

The Dijkstra era introduces `maxRefScriptSizePerTx` and `maxRefScriptSizePerBlock` as proper, updatable protocol parameters (replacing the Conway-era hard-coded constants of 200 KiB and 1 MiB respectively).

The `ppuWellFormed` check for `DijkstraEra` is defined in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/PParams.hs` at the `ConwayEraPParams DijkstraEra` instance:

```haskell
instance ConwayEraPParams DijkstraEra where
  ppuWellFormed _pv ppu =
    and
      [ isValid (/= 0) ppuMaxBBSizeL
      , isValid (/= 0) ppuMaxTxSizeL
      , isValid (/= 0) ppuMaxBHSizeL
      , isValid (/= 0) ppuMaxValSizeL
      , isValid (/= 0) ppuCollateralPercentageL
      , isValid (/= EpochInterval 0) ppuCommitteeMaxTermLengthL
      , isValid (/= EpochInterval 0) ppuGovActionLifetimeL
      , isValid (/= mempty) ppuPoolDepositL
      , isValid (/= zero) ppuGovActionDepositL
      , isValid (/= zero) ppuDRepDepositL
      , isValid ((/= CompactCoin 0) . unCoinPerByte) ppuCoinsPerUTxOByteL
      , ppu /= emptyPParamsUpdate
      , isValid (/= 0) ppuNOptL
      ]
``` [1](#0-0) 

Conspicuously absent from this list are `ppuMaxRefScriptSizePerTxL` and `ppuMaxRefScriptSizePerBlockL`. Both are `Word32` fields and both are fully updatable via governance `ParameterChange` actions in Dijkstra. [2](#0-1) 

The validation logic that enforces the per-transaction limit is:

```haskell
validateAllRefScriptSize pp utxo tx =
  let totalRefScriptSize = batchNonDistinctRefScriptsSize utxo tx
      maxRefScriptSizePerTx = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerTxG
   in failureUnless (totalRefScriptSize <= maxRefScriptSizePerTx) $ ...
``` [3](#0-2) 

And the per-block limit:

```haskell
validateBodyRefScriptsSizeTooBig pp blockBody utxo =
  let ...
      maxSize = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerBlockG
   in totalSize <= maxSize ?! ...
``` [4](#0-3) 

If `maxRefScriptSizePerTx = 0`, then any transaction with `totalRefScriptSize > 0` (i.e., any transaction that uses even a single reference script) will fail with `DijkstraTxRefScriptsSizeTooBig`. If `maxRefScriptSizePerBlock = 0`, any block containing such a transaction fails with `BodyRefScriptsSizeTooBig`.

The `actionWellFormed` check in the GOV rule only calls `ppuWellFormed` for `ParameterChange` actions:

```haskell
actionWellFormed pv ga = failureUnless isWellFormed $ MalformedProposal ga
  where
    isWellFormed = case ga of
      ParameterChange _ ppd _ -> ppuWellFormed pv ppd
      _ -> True
``` [5](#0-4) 

Since `ppuWellFormed` for `DijkstraEra` does not check these two parameters, a `ParameterChange` proposal setting `maxRefScriptSizePerTx = 0` or `maxRefScriptSizePerBlock = 0` passes `actionWellFormed` and can be ratified and enacted normally.

The analogous parameters in Conway (`ppMaxRefScriptSizePerTxG` and `ppMaxRefScriptSizePerBlockG`) were hard-coded constants, so this attack surface did not exist there. The Dijkstra era deliberately promoted them to governance-updatable parameters but omitted the corresponding `ppuWellFormed` guards.

---

### Impact Explanation

**High — Permanent freezing of all reference-script-using transactions, requiring a hard fork to recover.**

Once enacted, `maxRefScriptSizePerTx = 0` causes every transaction that references any UTxO containing a script to fail at the `LEDGER` rule. Reference scripts are used pervasively in DeFi and smart-contract applications. All such transactions are permanently rejected by all honest nodes. The only recovery is a subsequent governance action to raise the parameter back — but if the same governance majority that enacted the zero value is still in control, or if the governance system itself is impaired, recovery requires a hard fork.

---

### Likelihood Explanation

The attack requires a successful `ParameterChange` governance action, which in Conway/Dijkstra requires ratification by DReps, SPOs, and/or the Constitutional Committee depending on the parameter group. `maxRefScriptSizePerTx` and `maxRefScriptSizePerBlock` belong to `'NetworkGroup 'SecurityGroup`, meaning SPO votes are required for ratification. This is not a single-actor attack, but it is reachable by any coalition that can pass a governance action — including a malicious or mistaken governance majority, or a proposal author who crafts a `ParameterChange` that bundles a zero value alongside other legitimate changes. The absence of a `ppuWellFormed` guard means the ledger itself provides no last-resort protection.

---

### Recommendation

Add non-zero guards for `ppuMaxRefScriptSizePerTxL` and `ppuMaxRefScriptSizePerBlockL` inside the `ppuWellFormed` implementation for `DijkstraEra`, mirroring the existing pattern for `ppuMaxBBSizeL`, `ppuMaxTxSizeL`, etc.:

```haskell
, isValid (/= 0) ppuMaxRefScriptSizePerTxL
, isValid (/= 0) ppuMaxRefScriptSizePerBlockL
``` [1](#0-0) 

This mirrors the existing pattern used for `ppuMaxBBSizeL` and `ppuMaxTxSizeL` in the same function, and is consistent with how the Conway era's `ppuWellFormed` protects analogous size-limit parameters. [6](#0-5) 

---

### Proof of Concept

1. In Dijkstra era, a proposal author submits a `ParameterChange` governance action with `ppuMaxRefScriptSizePerTxL = SJust 0`.
2. `actionWellFormed` calls `ppuWellFormed _pv ppu`. The `ppuWellFormed` for `DijkstraEra` does not check `ppuMaxRefScriptSizePerTxL`, so the proposal is accepted as well-formed.
3. The governance action is ratified (requires DRep + SPO + CC votes per the `'NetworkGroup 'SecurityGroup` classification) and enacted at the epoch boundary.
4. From that epoch onward, `validateAllRefScriptSize` computes `maxRefScriptSizePerTx = fromIntegral @Word32 @Int 0 = 0`.
5. Any transaction with `batchNonDistinctRefScriptsSize utxo tx > 0` — i.e., any transaction referencing a UTxO that holds a script — fails with `DijkstraTxRefScriptsSizeTooBig`.
6. All DeFi and smart-contract transactions using reference scripts are permanently frozen. Recovery requires a new governance action to restore a non-zero value, or a hard fork. [7](#0-6) [2](#0-1)
