### Title
Unconditional `farmer_reward_address_override` Acceptance Enables Farmer Block Reward Diversion — (`chia/farmer/farmer_api.py`)

### Summary

A connected harvester can set `RespondSignatures.farmer_reward_address_override` to any arbitrary `bytes32` puzzle hash, and the farmer will unconditionally use it as the `farmer_puzzle_hash` in `DeclareProofOfSpace`, redirecting the 0.25 XCH farmer block reward away from the farmer's configured `farmer_target`.

### Finding Description

In `_process_respond_signatures`, the farmer initializes `farmer_reward_address` from its own configured target, then replaces it without any authorization check: [1](#0-0) 

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
    include_source_signature_data = True
```

The overridden address is then passed directly into `DeclareProofOfSpace.farmer_puzzle_hash`: [2](#0-1) 

The `farmer_reward_address_override` field is a plain optional `bytes32` in the `RespondSignatures` streamable message, with no cryptographic binding to any key the farmer controls: [3](#0-2) 

There is no farmer-side opt-in flag, no allowlist of permitted override addresses, and no signature proving the harvester is authorized to redirect rewards. The `notify_farmer_reward_taken_by_harvester_as_fee` call (referenced in `farmer.py`) only logs a warning and does not block the override.

### Impact Explanation

Every block won by the malicious harvester will have its 0.25 XCH farmer coinbase reward directed to the attacker's puzzle hash instead of the farmer's configured `farmer_target`. This is a complete, per-block diversion of the farmer reward with no on-chain recourse.

### Likelihood Explanation

The precondition is that the attacker controls a **connected, SSL-authenticated harvester process**. Chia's farmer-harvester connection uses mutual TLS, so the attacker must either:
- Operate a harvester that the farmer operator has legitimately connected (e.g., a third-party harvester service marketed as standard), or
- Compromise a legitimate harvester machine.

The attacker also needs valid plot files to produce a proof of space that passes `verify_and_get_quality_string`: [4](#0-3) 

These preconditions are non-trivial but realistic, particularly for third-party harvester operators (CHIP-22 use case) who can ship modified harvester software that sets the override to their own address without the farmer operator's explicit knowledge or consent.

### Recommendation

Add a farmer-side configuration flag (e.g., `allow_farmer_reward_address_override: bool`, defaulting to `false`) and reject any `RespondSignatures` where `farmer_reward_address_override` is set but the flag is disabled. Optionally, restrict permitted override addresses to a configured allowlist. The warning log is insufficient as a security control.

### Proof of Concept

1. Stand up a farmer with `farmer_target` = `FARMER_ADDR`.
2. Connect a harvester with valid plots.
3. Modify the harvester's `RespondSignatures` construction to set `farmer_reward_address_override = ATTACKER_ADDR`.
4. Win a block.
5. Observe that `DeclareProofOfSpace.farmer_puzzle_hash == ATTACKER_ADDR`, not `FARMER_ADDR`.
6. Confirm the 0.25 XCH coinbase is paid to `ATTACKER_ADDR`. [3](#0-2) [5](#0-4)
