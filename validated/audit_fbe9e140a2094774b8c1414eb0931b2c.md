Audit Report

## Title
ETH Deposit Limit Check Omits Incoming Amount, Allowing Cap to Be Exceeded - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric comparison for ETH versus LST assets. For LSTs the incoming `amount` is added to `totalAssetDeposits` before comparing against the cap, but for ETH it is not. As a result, the ETH deposit cap is only enforced after it has already been exceeded, not before, allowing any depositor to push total ETH holdings arbitrarily above the configured limit in a single transaction.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // amount omitted
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct for LSTs
}
``` [1](#0-0) 

The ETH branch returns `true` only when `totalAssetDeposits` **already** exceeds the cap. It never tests whether the incoming deposit would push the total over the cap. The LST branch correctly adds `amount` before the comparison.

`depositETH` (lines 76–93) passes `msg.value` as `depositAmount` to `_beforeDeposit` (lines 648–670), which calls this check at line 661. If the check returns `false`, the deposit proceeds unconditionally. [2](#0-1) [3](#0-2) 

Existing guards (`nonReentrant`, `whenNotPaused`, `onlySupportedAsset`) are unrelated to the limit check and do not compensate for the missing `amount` term.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

`depositLimitByAsset` for ETH is the protocol's risk-management cap on total ETH exposure. Because the check never includes the incoming deposit amount, a single depositor can push `totalAssetDeposits` arbitrarily above the cap in one transaction (e.g., cap = 1 000 ETH, current total = 999 ETH, attacker deposits 10 000 ETH → total becomes 10 999 ETH). The cap is a false invariant: it gives operators and users a false sense of security that ETH exposure is bounded, while in practice it is not. No funds are directly stolen, but the protocol silently accepts far more ETH than its risk parameters allow, matching the allowed impact "Contract fails to deliver promised returns, but doesn't lose value."

## Likelihood Explanation
**Medium.** The attack requires no special role, no flash loan, and no coordination. Any depositor who observes `totalAssetDeposits ≤ depositLimitByAsset` on-chain can send a single `depositETH` call with an arbitrarily large `msg.value`. The condition is trivially observable and the exploit is a single transaction with no preconditions beyond having ETH.

## Recommendation
Add the incoming `amount` to the ETH branch to match the LST branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [4](#0-3) 

## Proof of Concept
1. Admin sets ETH deposit limit to 1 000 ETH via `LRTConfig.updateAssetDepositLimit`.
2. Legitimate users deposit until `getTotalAssetDeposits(ETH_TOKEN) == 999 ETH`.
3. Attacker calls `depositETH{value: 10_000 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `999e18 > 1000e18` → `false` → deposit is not blocked.
5. `_mintRsETH` mints rsETH for 10 000 ETH; `totalAssetDeposits` becomes 10 999 ETH — ~11× the intended cap.
6. All subsequent ETH deposits are blocked (`10999 > 1000` → `true`), but the protocol now holds ~11× its intended ETH exposure.

**Foundry test plan:**
```solidity
function test_ethCapBypass() public {
    // set cap to 1000 ether
    lrtConfig.updateAssetDepositLimit(LRTConstants.ETH_TOKEN, 1000 ether);
    // deposit 999 ether legitimately
    vm.deal(alice, 999 ether);
    vm.prank(alice); depositPool.depositETH{value: 999 ether}(0, "");
    // attacker deposits 10_000 ether — should revert but does not
    vm.deal(attacker, 10_000 ether);
    vm.prank(attacker); depositPool.depositETH{value: 10_000 ether}(0, "");
    // assert total far exceeds cap
    assertGt(depositPool.getTotalAssetDeposits(LRTConstants.ETH_TOKEN), 1000 ether);
}
```
