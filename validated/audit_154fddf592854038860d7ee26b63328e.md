### Title
`setPoolBinAdditionalFees` Lacks Cap Validation, Allowing Pool Admin to Bypass Factory-Enforced Fee Caps — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

The factory enforces fee caps on `setPoolAdminFees` but applies **no cap check** on `setPoolBinAdditionalFees`. A pool admin can set per-bin additional fees to the full `uint16` maximum (65 535 in E6 = 6.5535%) on any bin, bypassing the `maxAdminSpreadFeeE6` guard that is supposed to restrict them.

### Finding Description

The protocol's security model states that the pool admin is "semi-trusted … but restricted by factory-enforced caps." Two admin paths modify effective swap fees:

**Path 1 — `setPoolAdminFees` (cap enforced):** [1](#0-0) 

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
```

**Path 2 — `setPoolBinAdditionalFees` (no cap check):** [2](#0-1) 

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

The pool-level handler also performs no cap check: [3](#0-2) 

```solidity
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    emit BinAdditionalFeesUpdated(bin, addFeeBuyE6, addFeeSellE6);
}
```

The only validation is a bin-range check. `addFeeBuyE6` and `addFeeSellE6` are `uint16`, so they can be set to **65 535** (6.5535 % in E6 units) with no factory-level cap enforcement. This is the direct analog to the external report: one adjustment path (base fee) was fixed with a cap check; the other adjustment path (per-bin additional fee) was not.

### Impact Explanation

Per-bin additional fees are applied on top of the base spread fee during swap execution. A pool admin can set `addFeeBuyE6 = 65535` on the active bin, making every swap through that bin pay an additional 6.5535 % fee beyond what the `maxAdminSpreadFeeE6` cap (20 %) is supposed to allow. Traders receive less output than the oracle/bin curve permits under the protocol's stated fee ceiling, constituting a swap conservation failure and an admin-boundary break. The surplus accrues as spread fees collectible by the admin.

### Likelihood Explanation

The pool admin is a semi-trusted role that is explicitly supposed to be bounded by factory caps. The call path is direct and requires no special conditions: any pool admin can call `setPoolBinAdditionalFees` at any time with `addFeeBuyE6 = type(uint16).max`. No timelock, no co-signer, no additional guard exists.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` (or in `setBinAdditionalFees` on the pool) that enforces a maximum for `addFeeBuyE6` and `addFeeSellE6`, consistent with the existing `maxAdminSpreadFeeE6` cap:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce the cap inside `MetricOmmPool.setBinAdditionalFees` by reading the factory's current `maxAdminSpreadFeeE6`.

### Proof of Concept

1. Factory owner sets `maxAdminSpreadFeeE6 = 200_000` (20 %).
2. Pool admin calls `factory.setPoolAdminFees(pool, 200_000, 0)` — succeeds, capped at 20 %.
3. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — **succeeds with no revert**.
4. Effective buy fee on bin 0 is now `spreadFeeE6 (200_000) + addFeeBuyE6 (65_535) = 265_535` in E6 = 26.5 %, exceeding the 20 % cap.
5. Any trader swapping through bin 0 pays 6.5 % more than the protocol's cap system is supposed to allow; the excess accrues as spread surplus collectible by the admin via `collectPoolFees`. [4](#0-3) [2](#0-1) [3](#0-2)
