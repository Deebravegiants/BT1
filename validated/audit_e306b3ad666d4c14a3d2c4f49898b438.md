### Title
P2P Peer Can Inject Arbitrary Transaction Hashes into Syncing Node Storage, Corrupting RPC Responses — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

### Summary

`TransactionStreamFactory::parse_data_for_block` accepts the `transaction_hash` field from a peer-supplied `FullTransaction` without any validation against the computed hash. The forged hash is written verbatim into the `transaction_metadata` and `transaction_hash_to_idx` storage tables, causing every downstream RPC consumer to serve the attacker-chosen hash as authoritative.

### Finding Description

In `parse_data_for_block`, the `transaction_hash` field of each peer-supplied `FullTransaction` is pushed directly into `block_body.transaction_hashes` with an explicit developer acknowledgment that validation is missing: [1](#0-0) 

The `BlockBody` is then passed to `write_to_storage`, which calls `append_body`: [2](#0-1) 

Inside `write_transactions`, the unvalidated hash is stored in two tables:

1. `transaction_hash_to_idx_table.insert(txn, tx_hash, &transaction_index)` — maps the forged hash to the transaction index.
2. `transaction_metadata_table.append(txn, &transaction_index, &TransactionMetadata { ..., tx_hash: *tx_hash })` — stores the forged hash as the canonical hash for that index. [3](#0-2) 

There is no post-storage cross-check that verifies the stored hashes against the `transaction_commitment` already present in the block header. The header's `transaction_commitment` is stored independently and is never reconciled with the body hashes during p2p sync.

### Impact Explanation

Every RPC endpoint that reads transaction hashes from storage is affected:

- `starknet_getTransactionByHash(real_hash)` → returns `TRANSACTION_HASH_NOT_FOUND` because the real hash has no entry in `transaction_hash_to_idx_table`.
- `starknet_getTransactionByHash(forged_hash)` → returns the transaction body, binding the wrong hash to it.
- `starknet_getTransactionByBlockIdAndIndex` → returns the forged hash as the canonical hash.
- `starknet_getBlockWithFullTransactions` → zips stored transactions with stored hashes, returning forged hashes for every affected transaction. [4](#0-3) [5](#0-4) 

The stored `transaction_commitment` in the header is never verified against the body hashes, so the inconsistency persists silently.

### Likelihood Explanation

Any unauthenticated p2p peer can send a `FullTransaction` with an arbitrary `transaction_hash` field. The p2p sync client only reports a peer as bad for structural violations (wrong count, wrong ordering); hash mismatches are not detected. The `BadPeer` path is never triggered by a forged hash. [6](#0-5) 

### Recommendation

Before pushing `transaction_hash` into `block_body.transaction_hashes`, recompute the hash from the transaction body and `chain_id` and compare:

```rust
let computed_hash = get_transaction_hash(&transaction, chain_id, &TransactionOptions::default());
if computed_hash != transaction_hash {
    return Err(ParseDataError::BadPeer(BadPeerError::InvalidTransactionHash { ... }));
}
```

This is already acknowledged by the developer TODO at line 88. [7](#0-6) 

### Proof of Concept

1. Stand up a node in p2p sync mode.
2. Act as a peer and respond to a `TransactionQuery` with a `FullTransaction` where `transaction_hash` is an arbitrary `Felt` (e.g., `Felt::from(0xdeadbeef)`) that does not equal the computed hash of the transaction body.
3. After the sync client calls `write_to_storage`, query `get_block_transaction_hashes(block_number)` — it returns the injected `Felt`.
4. Call RPC `starknet_getTransactionByHash` with the real computed hash — it returns `TRANSACTION_HASH_NOT_FOUND`.
5. Call RPC `starknet_getTransactionByHash` with the injected `Felt` — it returns the transaction body. [8](#0-7)
