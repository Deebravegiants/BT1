### Title
Proposer Deposits Permanently Locked When Competing Update Proposal Wins - (File: `crates/contract/src/update.rs`)

### Summary

When `do_update` executes a winning update proposal, it unconditionally clears **all** other pending proposals from storage. Because `UpdateEntry` does not record the proposer's account ID or the deposit amount paid at proposal time, there is no mechanism to refund those deposits. Any participant who proposed a non-winning update permanently loses their storage deposit.

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` requires an attached deposit proportional to the size of the proposed update (contract binary or config), calculated by `ProposedUpdates::required_deposit`. Only the excess above the required amount is immediately refunded; the required portion is retained by the contract to cover storage. [1](#0-0) 

The `UpdateEntry` struct stored per proposal contains only the update payload and a `bytes_used` field. It does **not** record the proposer's `AccountId` or the deposit amount paid. [2](#0-1) 

When `do_update` is triggered by a threshold vote, it removes the winning entry and then calls `self.entries.clear()` and `self.vote_by_participant.clear()`, wiping all competing proposals from storage without issuing any refund. [3](#0-2) 

Because the proposer identity and deposit amount are not stored in `UpdateEntry`, the contract has no information with which to issue refunds at cleanup time. The freed storage bytes reduce the contract's storage obligation, but the corresponding NEAR tokens remain locked in the contract balance with no owner.

The `required_deposit` for a 1 MB contract binary is approximately `1_000_000 bytes × 10^-5 NEAR/byte ≈ 10 NEAR`, making this a non-trivial loss per affected proposer. [4](#0-3) 

### Impact Explanation

This breaks the production accounting invariant that deposited funds must be recoverable by their depositor. A participant who proposes a contract upgrade and attaches the required deposit permanently loses that deposit if any competing proposal reaches threshold first. The funds are not burned — they remain in the contract's balance — but they are irrecoverable because the association between deposit and proposer is discarded at proposal time.

This matches the allowed Medium impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

### Likelihood Explanation

The scenario requires only two participants to independently propose different updates — a realistic operational event (e.g., two operators propose different upgrade binaries in parallel). A single Byzantine participant below the signing threshold can also deliberately trigger this: propose a competing update, coordinate threshold votes for it, and cause a legitimate proposer's deposit to be locked. No privileged access, no key leakage, and no network-level attack is required.

### Recommendation

Store the proposer's `AccountId` and the deposit amount inside `UpdateEntry`:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    pub(super) proposer: AccountId,      // add
    pub(super) deposit: NearToken,       // add
}
```

In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and schedule a `Promise::new(entry.proposer).transfer(entry.deposit)` for each non-winning proposal:

```rust
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Refund all non-winning proposers before clearing
    for (_, loser) in self.entries.iter() {
        Promise::new(loser.proposer.clone()).transfer(loser.deposit).detach();
    }
    self.entries.clear();
    self.vote_by_participant.clear();
    // ... rest unchanged
}
```

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB contract binary, attaching ~10 NEAR deposit. `UpdateEntry { update: Contract([...]), bytes_used: 1_000_xxx }` is stored; A's 10 NEAR is retained by the contract.
2. Participant B calls `propose_update` with a different binary, attaching their own deposit.
3. Threshold participants vote for B's proposal. `vote_update` calls `do_update(&B_id, gas)`.
4. `do_update` removes B's entry, then calls `self.entries.clear()` — A's entry is deleted with no refund.
5. A's ~10 NEAR is permanently locked in the contract balance. `A` has no recourse. [5](#0-4)

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
