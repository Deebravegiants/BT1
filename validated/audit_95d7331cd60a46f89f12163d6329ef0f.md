### Title
Stale Removed-Participant Vote Counted Toward Measurement Threshold After Resharing — (`crates/contract/src/tee/measurements.rs`, `crates/contract/src/tee/tee_state.rs`, `crates/contract/src/lib.rs`)

### Summary
`MeasurementVotes::count_votes` counts every entry in `vote_by_account` without filtering by the current participant set. The stale-vote cleanup (`TeeState::clean_non_participant_votes`) is performed by a separate `clean_tee_status` path, not synchronously during the resharing-completion transition. During the window between resharing completion and `clean_tee_status` executing, a removed participant's pre-resharing vote still inflates the count, allowing T−1 current participants to satisfy the threshold check and add a malicious measurement.

### Finding Description

**`count_votes` is unfiltered.** [1](#0-0) 

Every value in `vote_by_account` is counted regardless of whether the voter is still a participant. No cross-check against the current `Participants` set is performed.

**`vote_add_os_measurement` uses this unfiltered count against the current threshold.** [2](#0-1) 

`threshold_parameters` is fetched from the post-resharing state, but `vote_measurement` → `count_votes` still includes stale entries.

**The cleanup function exists but is separate.** [3](#0-2) 

`clean_non_participant_votes` correctly calls `get_remaining_votes` for all three vote maps including `measurement_votes`, but it is only invoked via the `clean_tee_status` path — not inline during the resharing-completion transition. The sandbox test file `crates/contract/tests/sandbox/tee_cleanup_after_resharing.rs` and the presence of `clean_tee_status` as a distinct configurable function (5 references in `config.rs`, 4 in `dto_mapping.rs`) confirm that cleanup is deferred, not atomic with resharing.

**`get_remaining_votes` is never called on the hot path.** [4](#0-3) 

This filter exists and is correct, but it is only reached via `clean_non_participant_votes`, which is not called synchronously when `vote_reshared` returns the new `RunningContractState`.

### Impact Explanation
A Byzantine participant P0 who is about to be removed can pre-cast a vote for a malicious `ContractExpectedMeasurements`. After resharing removes P0, P0's vote persists in `measurement_votes.vote_by_account`. During the window before `clean_tee_status` executes, only T−1 honest current participants need to vote for the same malicious measurement to reach the threshold. The malicious measurement is then added to `AllowedMeasurements`, allowing any node running that unapproved OS/firmware to pass attestation and participate in MPC signing — an attestation authorization bypass.

### Likelihood Explanation
The attack requires only a single participant who knows they will be removed (or who is colluding with the resharing initiator). The window is bounded by NEAR block time between resharing completion and `clean_tee_status` execution. The attacker needs T−1 additional colluding participants, which is strictly below the signing threshold T. The call sequence is deterministic and requires no network-level access.

### Recommendation
Call `self.tee_state.clean_non_participant_votes(&new_participants)` synchronously inside the resharing-completion branch of `vote_reshared` (or its caller in `lib.rs`) before returning the new `RunningContractState`, so stale votes are purged atomically with the participant-set transition. Alternatively, filter `count_votes` against the current `Participants` at call time in `vote_add_os_measurement`.

### Proof of Concept
Deterministic unit test (no network required):

1. Build N=5, T=3 contract with participants P0–P4.
2. Set signer to P0; call `vote_add_os_measurement(malicious_M)` → `measurement_votes` has 1 entry for P0.
3. Mutate `protocol_state` directly to a new `RunningContractState` with participants P1–P4 (T=3), simulating resharing completion **without** calling `clean_tee_status`.
4. Set signer to P1; call `vote_add_os_measurement(malicious_M)` → count = 2 (P0 stale + P1).
5. Set signer to P2; call `vote_add_os_measurement(malicious_M)` → count = 3 ≥ threshold(3) → `malicious_M` added.
6. Assert `allowed_os_measurements()` contains `malicious_M` despite only 2 current-participant votes.

    pub fn get_remaining_votes(&self, participants: &Participants) -> Self {
        let remaining = self
            .vote_by_account
            .iter()
            .filter(|(participant_id, _)| {
                participants.is_participant_given_participant_id(&participant_id.get())
            })
            .map(|(participant_id, vote)| (participant_id.clone(), vote.clone()))
            .collect();
        MeasurementVotes {
            vote_by_account: remaining,
        }
    }
