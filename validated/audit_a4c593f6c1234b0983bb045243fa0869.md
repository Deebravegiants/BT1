### Title
Expired DRep Can Self-Resurrect via Vote Submission, Bypassing Expiry Enforcement — (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Gov.hs`, `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Certs.hs`)

---

### Summary

The Conway `GOV` rule's `checkVotersAreValid` function omits an expiry check for `DRepVoter` credentials, while correctly enforcing expiry for `CommitteeVoter` credentials. Combined with `updateVotingDRepExpiries` in the `CERTS` rule — which runs **before** `GOV` in the ledger transition and unconditionally refreshes the stored expiry of any DRep that appears in the transaction's voting procedures — an expired DRep can submit a vote transaction, have their expiry silently reset to a future epoch, and have their vote counted in ratification as if they were active. This bypasses the DRep expiry mechanism that is designed to exclude inactive participants from governance.

---

### Finding Description

**Root cause 1 — missing expiry check in `checkVotersAreValid`:**

`checkVotersAreValid` in `Gov.hs` dispatches per voter type:

```haskell
checkVotersAreValid currentEpoch committeeState votes =
  checkDisallowedVotes votes DisallowedVoters $ \gas ->
    \case
      CommitteeVoter {} -> isCommitteeVotingAllowed currentEpoch committeeState (gasAction gas)
      DRepVoter {}      -> isDRepVotingAllowed (gasAction gas)   -- ← no expiry check
      StakePoolVoter {} -> isStakePoolVotingAllowed (gasAction gas)
``` [1](#0-0) 

For `CommitteeVoter`, `isCommitteeVotingAllowed` internally calls `votingCommitteeThresholdInternal`, which computes `activeCommitteeSize` by filtering out members where `currentEpoch > validUntil` — expiry is enforced at submission time. [2](#0-1) 

For `DRepVoter`, `isDRepVotingAllowed` only checks the governance action type; it never consults the DRep's stored `drepExpiry` field. [3](#0-2) 

**Root cause 2 — `updateVotingDRepExpiries` unconditionally refreshes expiry:**

`updateVotingDRepExpiries` in `Certs.hs` iterates over every `DRepVoter` entry in the transaction's voting procedures and calls `Map.adjust` to overwrite `drepExpiryL` with a freshly computed future epoch. There is no guard checking whether the DRep is currently expired before performing the update:

```haskell
updateVotingDRepExpiries tx currentEpoch drepActivity certState =
  let numDormantEpochs = certState ^. certVStateL . vsNumDormantEpochsL
      updateVSDReps vsDReps =
        Map.foldlWithKey'
          ( \dreps voter _ -> case voter of
              DRepVoter cred ->
                Map.adjust
                  (drepExpiryL .~ computeDRepExpiry drepActivity currentEpoch numDormantEpochs)
                  cred
                  dreps
              _ -> dreps
          )
          vsDReps
          (unVotingProcedures $ tx ^. bodyTxL . votingProceduresTxBodyL)
   in certState & certVStateL . vsDRepsL %~ updateVSDReps
``` [4](#0-3) 

**Root cause 3 — ordering: `CERTS` runs before `GOV`:**

In `conwayLedgerTransition`, the `CERTS` sub-rule is invoked first (processing certificates and calling `updateVotingDRepExpiries`), and only then is the `GOV` sub-rule invoked (processing votes and calling `checkVotersAreValid`): [5](#0-4) 

The combined effect: by the time `GOV` evaluates the vote, the DRep's expiry has already been silently reset to a future epoch by `CERTS`. The ratification check in `dRepAcceptedRatio` then sees a non-expired DRep and counts the vote:

```haskell
Just drepState
  | reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, skip
  | otherwise -> ...  -- vote counts
``` [6](#0-5)

### Impact Explanation

An expired DRep — one whose `drepExpiry` epoch has passed and who should be excluded from governance — can submit a transaction containing a vote. The `CERTS` rule unconditionally refreshes their expiry before `GOV` validates the vote. The DRep is then treated as active at ratification time, and their delegated stake is counted toward the DRep acceptance ratio. Depending on the stake weight of the expired DRep, this can tip the ratification threshold for `TreasuryWithdrawals`, `ParameterChange`, `HardForkInitiation`, `NewConstitution`, or `UpdateCommittee` proposals, causing governance actions to be enacted that would otherwise fail. This matches the **Medium** allowed impact: attacker-controlled transactions modify governance outcomes (DRep vote weight) outside design parameters.

---

### Likelihood Explanation

The attacker only needs to control a registered-but-expired DRep credential and submit a standard vote transaction. No privileged access, key compromise, or consensus majority is required. Any DRep that has been inactive long enough to expire retains their private key and can exploit this at any time. The barrier is low: the DRep must hold enough delegated stake to influence a close ratification vote.

---

### Recommendation

1. **Add an expiry guard in `checkVotersAreValid`**: For `DRepVoter`, look up the DRep's state in `certVState` and reject the vote if `currentEpoch > drepExpiry drepState` (mirroring the committee member check).

2. **Guard `updateVotingDRepExpiries` against expired DReps**: Before calling `Map.adjust`, check whether the DRep's current stored expiry (adjusted for dormant epochs) has already passed. Only refresh expiry for DReps that are still active.

3. **Align the expiry semantics**: The ratification check in `dRepAcceptedRatio` uses the raw `drepExpiry` field, while `isDRepExpired` in the test library uses `drepExpiry + numDormantEpochs`. Ensure a single, consistent expiry predicate is used across vote submission, expiry refresh, and ratification.

---

### Proof of Concept

1. Register a DRep and wait `ppDRepActivity + 1` epochs without voting so the DRep expires (`currentEpoch > drepExpiry`).
2. Confirm `isDRepExpired` returns `True` and `dRepAcceptedRatio` ignores the DRep's stake.
3. Submit a transaction containing a `DRepVoter` vote on any live governance action (e.g., `InfoAction` or a `TreasuryWithdrawals` proposal).
4. The `CERTS` rule calls `updateVotingDRepExpiries`, setting `drepExpiry := currentEpoch + drepActivity - numDormantEpochs` (a future epoch).
5. The `GOV` rule calls `checkVotersAreValid`; `isDRepVotingAllowed` returns `True` (action type is votable); the vote is stored in `gasDRepVotes`.
6. At the next ratification epoch, `dRepAcceptedRatio` finds `reCurrentEpoch <= drepExpiry drepState` and counts the DRep's stake toward the yes/no ratio.
7. If the DRep's stake is sufficient to cross the `dvtXxx` threshold, the governance action is ratified and enacted despite the DRep having been expired at the time of voting.
