### Title
Unverified `payload_hash` Binding in `respond_verify_foreign_tx` Enables Cross-Request Replay — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted ECDSA signature is valid over `response.payload_hash`, but never verifies that `payload_hash` is actually the canonical hash of the foreign-chain data for the submitted `request`. A single Byzantine attested MPC participant can reuse a valid threshold signature obtained from a prior legitimate signing session to resolve a different, unrelated pending request with an incorrect `payload_hash`, consuming that request's yield slot and delivering a structurally invalid response to the caller.

---

### Finding Description

The contract method `respond_verify_foreign_tx` accepts two caller-supplied arguments: a `VerifyForeignTransactionRequest` (the original request key) and a `VerifyForeignTransactionResponse` (containing `payload_hash` and `signature`). [1](#0-0) 

The contract performs two checks:
1. The `signature` is a valid ECDSA signature over `payload_hash` under the MPC root public key.
2. The `request` key exists in `pending_verify_foreign_tx_requests`.

What it **never** checks is whether `payload_hash` is actually `SHA-256(borsh(ForeignTxSignPayload { request, values }))` for any set of `values` consistent with the submitted `request`. The `payload_hash` field is a free parameter — it only needs to carry a valid signature.

But the contract imposes no such constraint. Any attested participant can call `respond_verify_foreign_tx` with an arbitrary `payload_hash` value, provided they supply a valid signature over it.

**Cross-request replay attack path:**

1. Byzantine attested participant (acting as leader or having observed a prior session) obtains a valid threshold signature `sig_A` over `hash(payload_A)` from a legitimate signing session for pending request A.
2. A separate, unrelated request B is pending in `pending_verify_foreign_tx_requests`.
3. The Byzantine participant calls `respond_verify_foreign_tx(request=B, response={payload_hash=hash(payload_A), signature=sig_A})`.
4. The contract checks: does request B exist? ✓. Is `sig_A` valid over `hash(payload_A)`? ✓. Both pass.
5. Request B's yield is resolved and its pending slot is consumed, delivering `{payload_hash=hash(payload_A), signature=sig_A}` to the caller of B. [5](#0-4) 

---

### Impact Explanation

The caller of request B receives a `VerifyForeignTransactionResponse` whose `payload_hash` encodes the foreign-chain data of request A, not B. Two outcomes follow depending on the caller's behavior:

**If the caller uses `ForeignChainSignatureVerifier::verify_signature` from the SDK:** [6](#0-5) 

The SDK recomputes `expected_payload_hash` from B's request parameters and the caller's expected extracted values, then compares it to `response.payload_hash`. Since `hash(payload_A)` encodes A's `ForeignChainRpcRequest`, the comparison fails and the caller's transaction reverts. Request B's pending slot has been permanently consumed; the caller must resubmit and pay again.

**If the caller does not verify `payload_hash`:** The caller accepts a response whose signature covers foreign-chain data from a completely different transaction (A), not the one they requested (B). This constitutes a forged foreign-chain verification — the caller's contract believes the MPC network attested to B's on-chain state when it actually attested to A's.

This breaks the core safety invariant of the `verify_foreign_transaction` flow: the contract is supposed to guarantee that the returned signature covers the verified state of the specific transaction the caller submitted.

---

### Likelihood Explanation

- Any single attested MPC participant can call `respond_verify_foreign_tx`; the function is not restricted to the designated leader.
- The attacker only needs one valid threshold signature from any prior session — obtained legitimately by participating in a normal signing round.
- TEE attestation is required, but a node that was previously honest and then turns Byzantine (or whose TEE is compromised) satisfies this condition.
- The attack is silent: the contract emits no event distinguishing a replayed response from a legitimate one.

---

### Recommendation

The contract cannot independently recompute `payload_hash` because it has no access to the foreign-chain extracted values. Two mitigations are possible:

1. **Bind `payload_hash` to the request at submission time:** Require callers of `verify_foreign_transaction` to supply a commitment (e.g., a hash of the expected `ForeignChainRpcRequest` parameters) that is stored alongside the pending request. In `respond_verify_foreign_tx`, verify that `payload_hash` is consistent with this stored commitment (at minimum, that the `ForeignChainRpcRequest` embedded in the signed payload matches the stored request).

2. **Enforce SDK-level verification on-chain:** Require the responding node to also submit the full `ForeignTxSignPayload` (not just the hash), allowing the contract to recompute and verify `payload_hash = SHA-256(borsh(payload))` and that `payload.request == request`. The payload can be discarded after verification to avoid storage bloat.

2. Request B (different tx_id, same domain) is pending.

   H_A encodes A's ForeignChainRpcRequest, not B's.
   Caller's verify_signature check fails → caller's transaction reverts.

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
