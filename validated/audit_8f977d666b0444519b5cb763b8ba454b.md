Audit Report

## Title
Unrestricted `sendFunds()` in `FeeReceiver` Enables Stale-Price Deposit to Steal MEV Yield from rsETH Holders - (File: contracts/FeeReceiver.sol)

## Summary

`FeeReceiver.sendFunds()` has no access-control modifier, allowing any caller to flush accumulated MEV/execution-layer rewards into `LRTDepositPool` at will. Because `LRTOracle.updateRSETHPrice()` is also publicly callable and will revert with `PriceAboveDailyThreshold` when the resulting price increase exceeds `pricePercentageLimit`, an attacker can force the reward flush, cause the price update to revert, and then deposit rsETH at the pre-reward stale price — capturing yield that belongs to existing rsETH holders.

## Finding Description

**Root cause — no access control on `sendFunds()`:**

`FeeReceiver.sendFunds()` is declared `external` with no role modifier:

```solidity
// contracts/FeeReceiver.sol L53-58
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

`receiveFromRewardReceiver()` in `LRTDepositPool` is equally unrestricted:

```solidity
// contracts/LRTDepositPool.sol L61
function receiveFromRewardReceiver() external payable { }
```

Once ETH lands in the deposit pool, `getETHDistributionData()` immediately counts it in TVL:

```solidity
// contracts/LRTDepositPool.sol L480
ethLyingInDepositPool = address(this).balance;
```

**Price guard causes revert, leaving `rsETHPrice` stale:**

`updateRSETHPrice()` is `public whenNotPaused`. When the TVL spike from the forced reward dump pushes `newRsETHPrice` above `highestRsethPrice * (1 + pricePercentageLimit)`, the function reverts for non-manager callers:

```solidity
// contracts/LRTOracle.sol L252-265
if (newRsETHPrice > highestRsethPrice) {
    uint256 priceDifference = newRsETHPrice - highestRsethPrice;
    bool isPriceIncreaseOffLimit =
        pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
    if (isPriceIncreaseOffLimit) {
        if (!IAccessControl(address(lrtConfig)).hasRole(LRTConstants.MANAGER, msg.sender)) {
            revert PriceAboveDailyThreshold();
        }
    }
}
```

The revert means `rsETHPrice` (the stored state variable) is **not updated**. It remains at the pre-reward value.

**Deposit uses the stale price:**

`getRsETHAmountToMint` reads the stored `rsETHPrice` directly — it does not recompute it:

```solidity
// contracts/LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

A lower `rsETHPrice` denominator means the attacker receives more rsETH per ETH than the true post-reward exchange rate warrants.

**Exploit sequence:**
1. Attacker calls `FeeReceiver.sendFunds()` — all accumulated MEV ETH moves to `LRTDepositPool`.
2. Attacker calls `LRTOracle.updateRSETHPrice()` — reverts with `PriceAboveDailyThreshold`; `rsETHPrice` stays stale.
3. Attacker calls `LRTDepositPool.depositETH{value: X}(minRSETH, "")` — minted rsETH is computed using the stale (lower) price.
4. Manager eventually calls `updateRSETHPriceAsManager()` — price is updated to the true higher value.
5. Attacker's rsETH is now worth more than they paid, at the expense of all pre-existing holders.

## Impact Explanation

**High — Theft of unclaimed yield.**

MEV/execution-layer rewards represent yield accrued to existing rsETH holders; it should manifest as rsETH price appreciation. By depositing at the stale price before the oracle is updated, the attacker captures a disproportionate share of that appreciation. Every existing holder's claim on the underlying ETH is diluted by the attacker's inflated rsETH balance. The stolen amount scales with the size of the accumulated reward balance and the attacker's deposit size.

## Likelihood Explanation

**Medium.**

- `pricePercentageLimit` is explicitly configurable via `setPricePercentageLimit()` and the guard code is present and enforced, making it highly likely to be set in production.
- MEV/execution-layer rewards accumulate passively in `FeeReceiver`; over days or weeks the balance can be large enough to exceed the daily threshold in a single call.
- The attack requires only two public calls (`sendFunds()` + `updateRSETHPrice()`) and a standard deposit — no special privileges, no flash loan, no front-running of a specific transaction.
- The attack is repeatable each time rewards accumulate past the threshold.

## Recommendation

Restrict `sendFunds()` to an authorized role so only a trusted keeper or manager can trigger the reward flush:

```solidity
function sendFunds() external onlyRole(LRTConstants.MANAGER) {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

Additionally, restrict `receiveFromRewardReceiver()` in `LRTDepositPool` to only accept calls from the registered `FeeReceiver` address, preventing direct ETH injection from arbitrary callers.

## Proof of Concept

```
Setup:
  - FeeReceiver holds 500 ETH in accumulated MEV rewards.
  - rsETHPrice = 1.05e18, highestRsethPrice = 1.05e18.
  - pricePercentageLimit = 1e16 (1%).
  - rsETH totalSupply = S.

Step 1: attacker calls FeeReceiver.sendFunds()
  → 500 ETH transferred to LRTDepositPool via receiveFromRewardReceiver().
  → LRTDepositPool.address(this).balance increases by 500 ETH.
  → getETHDistributionData() now returns ethLyingInDepositPool += 500 ETH.

Step 2: attacker calls LRTOracle.updateRSETHPrice()
  → _getTotalEthInProtocol() returns ~500 ETH more than before.
  → newRsETHPrice = (TVL + 500 ETH) / S >> 1.05e18 * 1.01.
  → isPriceIncreaseOffLimit = true.
  → msg.sender is not MANAGER → revert PriceAboveDailyThreshold().
  → rsETHPrice remains 1.05e18 (stale).

Step 3: attacker calls LRTDepositPool.depositETH{value: Y}(minRSETH, "")
  → getRsETHAmountToMint uses rsETHPrice = 1.05e18 (stale).
  → attacker receives Y / 1.05e18 rsETH.
  → true fair price is (TVL + 500 ETH + Y) / S > 1.05e18.

Step 4: manager calls LRTOracle.updateRSETHPriceAsManager()
  → rsETHPrice updated to true higher value P_new.
  → attacker's rsETH (minted at P_old) is now worth (Y / P_old) * P_new > Y.
  → excess value (Y/P_old * P_new - Y) is extracted from pre-existing holders.

Foundry fork test outline:
  1. Fork mainnet/testnet at a block where FeeReceiver has balance.
  2. vm.prank(attacker); feeReceiver.sendFunds();
  3. vm.expectRevert(PriceAboveDailyThreshold.selector);
     lrtOracle.updateRSETHPrice();
  4. uint256 rsethBefore = rseth.balanceOf(attacker);
     vm.prank(attacker); depositPool.depositETH{value: Y}(0, "");
     uint256 rsethMinted = rseth.balanceOf(attacker) - rsethBefore;
  5. vm.prank(manager); lrtOracle.updateRSETHPriceAsManager();
  6. Assert rsethMinted * lrtOracle.rsETHPrice() / 1e18 > Y (attacker profit).
  7. Assert existing holder's ETH redemption value decreased proportionally.
```
