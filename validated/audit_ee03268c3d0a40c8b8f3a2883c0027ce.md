Audit Report

## Title
LP Share Burn With Zero Token Return Due to Rounding in `removeLiquidity` — (`metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary
`LiquidityLib.removeLiquidity` computes per-bin token amounts owed to a withdrawing LP using integer floor division. When `binState.token0BalanceScaled * sharesToRemove < binTotalSharesVal`, both `amount0Scaled` and `amount1Scaled` evaluate to zero. The function unconditionally burns the LP's shares and clears their position with no revert or skip guard, so the LP permanently loses their position while receiving nothing. The forfeited balance is silently redistributed to remaining LPs.

## Finding Description
At lines 205–206, the per-bin amounts are computed with plain floor division:

```solidity
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
uint256 amount1Scaled = _checkedMul(binState.token1BalanceScaled, sharesToRemove) / binTotalSharesVal;
```

When the product `binState.token0BalanceScaled * sharesToRemove` is less than `binTotalSharesVal`, both results are zero. Immediately after, at lines 210–214, the code unconditionally updates state:

```solidity
binState.token0BalanceScaled -= uint104(amount0Scaled);   // subtracts 0
binState.token1BalanceScaled -= uint104(amount1Scaled);   // subtracts 0
binTotalShares[binIdx] = binTotalSharesVal - sharesToRemove; // shares burned
positionBinShares[posKey] = newUserShares;                   // position cleared
```

There is no guard that reverts or skips the burn when both amounts are zero. The `minimalMintableLiquidity` check at lines 200–202 only prevents leaving a dust position; it does not prevent a full removal that yields zero tokens. At lines 239–240, `_deltasScaledToExternal` applies a second floor division (dividing by `token0ScaleMultiplier`, up to `1e12` for 6-decimal tokens), which can reduce a non-zero `amount0Scaled` to zero external units. Transfers at lines 242–246 are conditional on non-zero amounts, so no tokens are sent. The bin balance is unchanged; the LP's shares are gone.

By contrast, `addLiquidity` at lines 109–110 uses `Math.ceilDiv` for the same computation, creating an asymmetric rounding that always favours the pool over the LP.

## Impact Explanation
An LP who calls `removeLiquidity` under the triggering conditions permanently loses their entire position in the affected bin. Their shares are burned, the bin balance is not reduced, and no tokens are transferred. This is a direct, irreversible loss of user principal — a broken LP withdraw flow causing loss of LP assets — matching the allowed-impact gate for Critical/High severity.

## Likelihood Explanation
The condition is reachable in normal operation without any privileged action. A bin that has been heavily traded can have its scaled balance drained to a few units while retaining many LP shares (swaps do not affect share counts). A small LP holding the minimum `minimalMintableLiquidity` shares removing their full position in such a bin will trigger the rounding. The user can trigger it unknowingly via a router that does not pre-simulate output. Because `removeLiquidity` requires `msg.sender == owner`, the loss cannot be forced on a victim by a third party, but there is no on-chain protection against self-inflicted loss.

## Recommendation
Add a revert before burning shares when both computed amounts are zero:

```solidity
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
uint256 amount1Scaled = _checkedMul(binState.token1BalanceScaled, sharesToRemove) / binTotalSharesVal;

if (amount0Scaled == 0 && amount1Scaled == 0) {
    revert ZeroTokensOut();
}
```

Alternatively, use `Math.ceilDiv` for the LP's share of the bin balance (rounding in the LP's favour, consistent with how `addLiquidity` rounds against the LP at lines 109–110):

```solidity
uint256 amount0Scaled = Math.ceilDiv(_checkedMul(binState.token0BalanceScaled, sharesToRemove), binTotalSharesVal);
uint256 amount1Scaled = Math.ceilDiv(_checkedMul(binState.token1BalanceScaled, sharesToRemove), binTotalSharesVal);
```

## Proof of Concept
**Setup:**
- Pool with a 6-decimal token (USDC) as token0; `TOKEN_0_SCALE_MULTIPLIER = 1e12`.
- Bin 0 has been traded down to `token0BalanceScaled = 500`.
- `binTotalShares[0] = 1_000_000e18`.
- Alice holds `1e18` shares (the minimum `minimalMintableLiquidity`) in bin 0.

**Steps:**
1. Alice calls `removeLiquidity` with `sharesToRemove = 1e18`.
2. `amount0Scaled = (500 * 1e18) / 1_000_000e18 = 0` (floor division).
3. `amount1Scaled = 0`.
4. `binState.token0BalanceScaled -= 0` → unchanged at 500.
5. `binTotalShares[0] = 999_999e18` — Alice's shares burned.
6. `positionBinShares[aliceKey] = 0` — Alice's position erased.
7. `amount0Removed = 0 / 1e12 = 0`; no transfer.
8. Alice receives nothing; the 500 scaled units remain in the bin owned by remaining LPs.
