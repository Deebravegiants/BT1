### Title
Raw ETH Balance Used for Deposit-Limit Accounting Allows Donation Attack to Freeze Deposits — (File: `contracts/LRTDepositPool.sol`)

---

### Summary

`LRTDepositPool.getETHDistributionData()` measures the ETH held in the deposit pool using the raw `address(this).balance` instead of a tracked internal accounting variable. Because the contract exposes an unrestricted `receive()` fallback, any unprivileged actor can donate ETH directly to the contract, artificially inflating the reported total ETH in the protocol. This inflated figure is consumed by the deposit-limit guard, which will then reject all future `depositETH()` calls with `MaximumDepositLimitReached`, temporarily freezing the deposit path for all users.

---

### Finding Description

`LRTDepositPool` exposes a bare `receive()` function that accepts ETH from any sender with no accounting side-effects: [1](#0-0) 

`getETHDistributionData()` then reads the raw native balance of the contract as the amount of ETH "lying in the deposit pool": [2](#0-1) 

The same pattern applies to every NodeDelegator and to the unstaking vault: [3](#0-2) 

These raw balances are summed by `getTotalAssetDeposits()`: [4](#0-3) 

That total is then fed directly into the deposit-limit guard inside `_beforeDeposit()`: [5](#0-4) 

The guard for ETH uses a strict `>` comparison (no `amount` added), so once the raw balance exceeds the configured cap, **every** subsequent `depositETH()` call reverts: [6](#0-5) 

The same raw-balance pattern exists for LST tokens: [7](#0-6) [8](#0-7) 

A secondary consequence is that the inflated balance propagates into `LRTOracle._getTotalEthInProtocol()`, which calls `getTotalAssetDeposits()` for every supported asset and uses the result to compute `newRsETHPrice`. An artificially elevated TVL causes the oracle to record a higher rsETH price and to mint excess protocol-fee rsETH to the treasury, diluting existing rsETH holders: [9](#0-8) [10](#0-9) 

---

### Impact Explanation

**Primary — Medium: Temporary freezing of funds (deposit path).**
An attacker who donates enough ETH to push `address(this).balance` above `depositLimitByAsset(ETH_TOKEN)` causes every `depositETH()` call to revert. The deposit path is frozen until an admin raises the limit via governance. Users cannot enter the protocol during this window.

**Secondary — High: Theft of unclaimed yield (excess fee minting).**
The donated ETH is counted as protocol reward (`totalETHInProtocol − previousTVL`). The oracle mints protocol-fee rsETH proportional to this artificial reward to the treasury, diluting existing rsETH holders' share of the real underlying yield. [11](#0-10) 

---

### Likelihood Explanation

`LRTDepositPool.receive()` is unrestricted and callable by any EOA or contract. The cost of the attack scales with the gap between current total deposits and the configured deposit limit. When the protocol is operating near its cap (a common production state for a popular LRT), the donation required to tip it over is small. The attacker permanently loses the donated ETH (it becomes protocol TVL), but the griefing cost is bounded and the impact (deposit freeze + excess fee minting) is immediate and protocol-wide.

---

### Recommendation

Replace raw-balance reads with an internal accounting variable that is incremented only on legitimate deposits and decremented on withdrawals/transfers. For example:

```solidity
// storage
uint256 public ethDepositedInPool;

// in depositETH / receiveFromNodeDelegator / etc.
ethDepositedInPool += msg.value;

// in getETHDistributionData
ethLyingInDepositPool = ethDepositedInPool;
```

Apply the same pattern to `assetLyingInDepositPool` (use a `mapping(address => uint256) public assetDepositedInPool`) and to the unstaking vault and NodeDelegator balances. This mirrors the fix described in the reference report: compute the reserve from tracked state rather than from the live account balance.

---

### Proof of Concept

```solidity
// Assume depositLimitByAsset[ETH_TOKEN] = 10_000 ether
// Assume current getTotalAssetDeposits(ETH_TOKEN) = 9_999 ether

// Step 1: attacker donates 2 ETH directly to the deposit pool
(bool ok,) = address(lrtDepositPool).call{value: 2 ether}("");
require(ok);

// Step 2: address(this).balance is now 9_999 + 2 = 10_001 ether
// getTotalAssetDeposits(ETH_TOKEN) returns 10_001 ether
// _checkIfDepositAmountExceedesCurrentLimit returns true (10_001 > 10_000)

// Step 3: any user attempting to deposit ETH now reverts
vm.prank(user);
vm.expectRevert(ILRTDepositPool.MaximumDepositLimitReached.selector);
lrtDepositPool.depositETH{value: 1 ether}(0, "");

// Step 4: updateRSETHPrice() is called (publicly callable)
// totalETHInProtocol is inflated by 2 ETH
// rewardAmount = 2 ETH → protocolFeeInETH = 2e18 * feeBPS / 10_000
// excess rsETH is minted to treasury, diluting existing holders
lrtOracle.updateRSETHPrice();
```
