### Title
Missing Caller-Binding in `verify_foreign_transaction` Enables Cross-Caller Signature Replay - (File: `crates/contract/src/lib.rs`, `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

### Summary

The `verify_foreign_transaction()` function produces MPC signatures that are not bound to the caller's identity. Unlike `sign()`, which derives a unique key per caller using `(predecessor_id, path)`, `verify_foreign_transaction()` omits the `derivation_path` field entirely and signs under the root ForeignTx key with no tweak. Any party can submit the same foreign chain tx verification request and receive the same signature, enabling cross-caller replay attacks against bridge contracts that rely on this function.

### Finding Description

**`sign()` correctly binds the signing key to the caller:**

In `crates/contract/src/lib.rs`, the `sign()` function derives a unique signing key per caller by computing a tweak from `(predecessor_id, path)`:

```rust
let request = SignatureRequest::new(
    request.domain_id,
    request.payload,
    &predecessor,   // caller-specific binding
    &request.path,
);
```

The `SignatureRequest` stores a `tweak` field derived from the caller's identity, ensuring each caller gets a unique derived key. [1](#0-0) 

**The design document specifies caller-specific key derivation for `verify_foreign_transaction()`:**

`docs/foreign-chain-transactions.md` explicitly states that `verify_foreign_transaction()` should use a `derivation_path` field and a different tweak prefix so the same `(predecessor_id, derivation_path)` can never yield the same derived key across the two purposes:

```rust
// Design doc intent:
pub struct VerifyForeignTransactionRequestArgs {
    pub request: ForeignChainRpcRequest,
    pub derivation_path: String,  // Key derivation path — caller-specific
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [2](#0-1) 

**The actual implementation omits `derivation_path` entirely:**

The real `VerifyForeignTransactionRequestArgs` in `crates/near-mpc-contract-interface/src/types/foreign_chain.rs` has no `derivation_path`:

```rust
pub struct VerifyForeignTransactionRequestArgs {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
    // NO derivation_path
}
``` [3](#0-2) 

And `VerifyForeignTransactionRequest` (the stored form) has no `tweak` field:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
    // NO tweak
}
``` [4](#0-3) 

**The contract confirms root-key signing with no tweak:**

The `verify_foreign_transaction()` handler calls `args_into_verify_foreign_tx_request(request)` which converts the args into a `VerifyForeignTransactionRequest` with no tweak derivation. [5](#0-4) 

A unit test comment at line 3694 explicitly confirms: *"simulate signature with the root key (no tweak for foreign tx)"*. [6](#0-5) 

**The request is explicitly caller-agnostic:**

The test `verify_foreign_transaction__should_queue_duplicates_from_different_callers` shows that Alice and Bob submitting the same request are queued under a single key and both receive the same response: *"both yields are queued under the single (caller-agnostic) request key."* [7](#0-6) 

This is the direct analog to H-02: just as `fetchVirtualAccount()` used only the caller's address without considering which chain the call came from, `verify_foreign_transaction()` uses only the foreign chain tx ID without binding the resulting signature to the caller's identity.

### Impact Explanation

Any unprivileged party can submit the same `verify_foreign_transaction` request as a legitimate user and receive the identical MPC signature, because the signature is produced under the root ForeignTx key with no caller-specific tweak. Bridge contracts that use this function to authorize user-specific actions (e.g., releasing funds to the caller after verifying a foreign chain deposit) cannot rely on the MPC signature to distinguish between callers. An attacker can:

1. Observe a legitimate user's pending `verify_foreign_transaction` request on-chain.
2. Submit the same request (or simply wait for the pending response to be delivered to both).
3. Receive the identical MPC signature.
4. Call the bridge contract with the signature, impersonating the legitimate user and claiming their funds.

This enables cross-caller replay and front-running attacks that cause invalid bridge execution — matching the High impact category.

### Likelihood Explanation

The `verify_foreign_transaction` function is explicitly designed for bridge use cases (Omnibridge inbound flow, chain abstraction). Bridge contracts are the primary consumers. The design document's specification of `derivation_path` and a caller-specific tweak prefix creates a reasonable expectation among bridge developers that the returned signature is bound to the submitting caller. Any bridge contract built against the documented API that relies on this binding is directly vulnerable. The attacker entry path requires only a standard NEAR account and 1 yoctoNEAR deposit — no special privileges.

### Recommendation

Implement the `derivation_path` field as specified in the design document:

1. Add `derivation_path: String` to `VerifyForeignTransactionRequestArgs` in `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`.
2. Add `tweak: Tweak` to `VerifyForeignTransactionRequest`.
3. In `verify_foreign_transaction()`, derive the tweak from `(predecessor_id, derivation_path)` using the foreign-tx-specific prefix (`"near-mpc-recovery v0.1.0 foreign-tx epsilon derivation:"`) as described in the design doc, and store it in the request.
4. Have MPC nodes use this tweak when signing the `ForeignTxSignPayload`, so the resulting signature is bound to the specific caller and derivation path.

### Proof of Concept

```
1. Alice calls verify_foreign_transaction({tx_id: X, chain: Ethereum, ...})
   → MPC nodes verify tx X on Ethereum
   → MPC signs hash(ForeignTxSignPayload{tx_id: X, values: [...]}) under root ForeignTx key
   → Alice receives signature S

2. Bob (attacker) calls verify_foreign_transaction({tx_id: X, chain: Ethereum, ...})
   → Same request key → queued under same pending entry
   → Bob receives the identical signature S (same root-key signing, no tweak)

3. Bob calls bridge_contract.claim(tx_id=X, signature=S)
   → Bridge verifies S is a valid MPC signature over the ForeignTxSignPayload for tx X ✓
   → Bridge releases funds intended for Alice to Bob ✗
```

The root cause is confirmed by `VerifyForeignTransactionRequest` having no `tweak` field and the test comment "simulate signature with the root key (no tweak for foreign tx)", in direct contrast to `SignatureRequest` which always carries a caller-derived `tweak`. [8](#0-7)

## Tweak Derivation (Sign vs ForeignTx)

`verify_foreign_transaction()` uses a **different tweak derivation prefix** than `sign()` so the same
`(predecessor_id, derivation_path)` can never yield the same derived key across the two purposes.

Design:

* Keep the existing sign tweak derivation prefix unchanged.
* Introduce a foreign-tx-specific prefix and derive the tweak from the same `(predecessor_id, derivation_path)`
  input using the same hash construction.
* The contract derives the tweak internally from `request.derivation_path` (callers do not submit raw tweaks).

Example:

```rust
const SIGN_TWEAK_DERIVATION_PREFIX: &str =
    "near-mpc-recovery v0.1.0 epsilon derivation:";
const FOREIGN_TX_TWEAK_DERIVATION_PREFIX: &str =
    "near-mpc-recovery v0.1.0 foreign-tx epsilon derivation:";

pub fn derive_sign_tweak(predecessor_id: &AccountId, path: &str) -> Tweak {
    let hash: [u8; 32] = derive_from_path(SIGN_TWEAK_DERIVATION_PREFIX, predecessor_id, path);
    Tweak::new(hash)
}

pub fn derive_foreign_tx_tweak(predecessor_id: &AccountId, path: &str) -> Tweak {
    let hash: [u8; 32] = derive_from_path(FOREIGN_TX_TWEAK_DERIVATION_PREFIX, predecessor_id, path);
    Tweak::new(hash)
}
```

This ensures key material used for validated foreign transactions is **always** distinct from
general-purpose `sign()` keys, even if the same account and derivation path are reused.
```
