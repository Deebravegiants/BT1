### Title
Missing Minimum `BlockConfirmations` Enforcement Allows Zero-Confirmation Bitcoin Transaction Signing — (`crates/contract/src/lib.rs`, `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`)

---

### Summary

The `verify_foreign_transaction` endpoint accepts a `BitcoinRpcRequest` with `confirmations: 0`. The `BitcoinInspector` enforces no minimum confirmation floor — the check `block_confirmations_threshold <= transaction_block_confirmation` trivially passes when the caller supplies `0`. An unprivileged caller can therefore obtain a valid MPC threshold signature over an unconfirmed (mempool) Bitcoin transaction, enabling double-spend attacks on any bridge that treats the MPC signature as proof of Bitcoin finality.

---

### Finding Description

`BlockConfirmations` is a plain `u64` newtype with no minimum constraint:

```rust
pub struct BlockConfirmations(pub u64);
```

The ABI schema confirms `minimum: 0.0`, meaning `0` is an accepted wire value.

The contract's `verify_foreign_transaction` method performs no validation of the `confirmations` field inside `BitcoinRpcRequest` before enqueuing the request:

```rust
pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
    // ...
    self.check_request_preconditions(
        request.domain_id,
        DomainPurpose::ForeignTx,
        Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
        MINIMUM_SIGN_REQUEST_DEPOSIT,   // only 1 yoctoNEAR
    );
    // no confirmation-count check
    let request = args_into_verify_foreign_tx_request(request);
    // ... enqueue
}
```

On the node side, `BitcoinInspector::extract` enforces the caller-supplied threshold verbatim:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;

if !enough_block_confirmations {
    return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations { ... });
}
```

When `block_confirmations_threshold = BlockConfirmations(0)`, the condition `0 <= actual_confirmations` is always `true` for any `u64`, including `0` (an unconfirmed mempool transaction). The inspector proceeds to extract values and the signing pipeline produces a valid threshold signature.

The signed payload is `(request, extracted_values)`, where `request` includes `confirmations: 0`. A bridge contract that relies on the MPC signature as proof of Bitcoin finality — without independently re-checking the confirmation count embedded in the signed payload — will accept this signature as authoritative.

---

### Impact Explanation

**High — Cross-chain verification bypass enabling double-spend.**

An attacker who controls a Bitcoin transaction can:
1. Broadcast the transaction to the Bitcoin mempool (0 confirmations).
2. Immediately call `verify_foreign_transaction` with `confirmations: 0`.
3. Receive a valid MPC threshold signature over the unconfirmed transaction.
4. Submit the signature to a bridge contract to claim NEAR-side funds.
5. Double-spend the Bitcoin transaction (replace-by-fee or miner collusion), reversing the on-chain payment.

The bridge has paid out; the Bitcoin payment is gone. This is a direct, concrete loss of funds controlled by the bridge flow that the MPC network is supposed to protect.

---

### Likelihood Explanation

**Medium.** Any unprivileged caller with 1 yoctoNEAR can submit a `verify_foreign_transaction` request. The only additional requirement is that the attacker controls a Bitcoin transaction eligible for double-spending (RBF-flagged transactions, or any transaction before it reaches 1 confirmation). Bitcoin mempool transactions are routinely replaceable. No privileged access, no threshold collusion, and no TEE bypass is required — the vulnerability is exercised entirely through the public contract API.

---

### Recommendation

Enforce a protocol-level minimum confirmation count. The simplest fix is to add a named constant and validate it at the contract boundary before the request is enqueued:

```rust
/// Minimum Bitcoin block confirmations required before the MPC network
/// will sign a foreign-chain transaction. A value of 0 allows signing
/// unconfirmed (mempool) transactions, enabling double-spend attacks.
const MIN_BITCOIN_BLOCK_CONFIRMATIONS: u64 = 1; // or 6 for production safety

// Inside verify_foreign_transaction, before enqueue:
if let ForeignChainRpcRequest::Bitcoin(ref btc_req) = request.request {
    if btc_req.confirmations.0 < MIN_BITCOIN_BLOCK_CONFIRMATIONS {
        env::panic_str("Bitcoin confirmations below minimum required");
    }
}
```

A separate named constant (distinct from any other threshold constant) makes the policy explicit and independently adjustable per chain, mirroring the lesson from the Aave-Lens M-01 finding: reusing an unrelated constant — or in this case, the implicit default of `0` — as a security floor is semantically wrong and operationally dangerous.

---

### Proof of Concept

**Attacker-controlled entry path:**

1. Attacker broadcasts Bitcoin tx `T` to the mempool (0 confirmations, RBF-enabled).
2. Attacker calls:
   ```json
   verify_foreign_transaction({
     "request": {
       "domain_id": <foreign_tx_domain>,
       "payload_version": "V1",
       "request": {
         "Bitcoin": {
           "tx_id": "<T's txid>",
           "confirmations": 0,
           "extractors": ["BlockHash"]
         }
       }
     }
   })
   ```
   with `deposit: 1 yoctoNEAR`.
3. Contract passes `check_request_preconditions` (deposit ✓, gas ✓, domain ✓, `accept_requests` ✓). No confirmation-count check. Request enqueued.
4. MPC nodes call `getrawtransaction(T, verbose=true)`. Bitcoin returns `confirmations: 0` for a mempool tx.
5. `BitcoinInspector::extract`: `0 <= 0` → `enough_block_confirmations = true`. Extraction proceeds; block hash extracted.
6. Nodes sign `(request, [BlockHash(…)])`. Threshold signature produced and returned on-chain.
7. Attacker submits signature to bridge contract, claims NEAR-side funds.
8. Attacker RBF-replaces tx `T` with a conflicting tx `T'` paying themselves. `T` is evicted from the mempool and never confirms.
9. Bridge has paid out; Bitcoin payment is reversed.

- Contract deposit-only gate (no confirmation check): [1](#0-0)

- Inspector confirmation check (trivially passes at 0): [3](#0-2) 
- Minimum deposit constant (only guard at contract entry): [4](#0-3)

        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",

            request
        );

        self.check_request_preconditions(
            request.domain_id,

            MINIMUM_SIGN_REQUEST_DEPOSIT,

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );

            callback_args,
            callback_gas,
