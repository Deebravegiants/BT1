Audit Report

## Title
Token Deposits Default to Zero Fees, Allowing Bypass of ETH Deposit Fees - (File: contracts/pools/RSETHPool.sol)

## Summary
`RSETHPool.sol` maintains two separate fee variables: `feeBps` for ETH deposits and a per-token `tokenFeeBps[token]` mapping for ERC20 deposits. Because `addSupportedToken()` never initializes `tokenFeeBps[token]`, every newly added token defaults to a zero fee. Any unprivileged user can deposit a supported ERC20 token instead of ETH to receive the same `wrsETH` output while paying zero protocol fees, permanently depriving the protocol of its intended fee revenue.

## Finding Description
`RSETHPool.sol` declares two distinct fee storage variables:

- `feeBps` (line 43): applied to native ETH deposits via `viewSwapRsETHAmountAndFee(uint256)` at line 312.
- `tokenFeeBps` (line 88): a per-token mapping applied to ERC20 deposits via `viewSwapRsETHAmountAndFee(uint256, address)` at lines 335–336.

The token deposit path at lines 284–305 calls `viewSwapRsETHAmountAndFee(amount, token)`, which reads `tokenFeeBps[token]` directly (line 335). Because Solidity mappings default to `0`, `fee = amount * 0 / 10_000 = 0` for any token whose fee has not been explicitly set.

`addSupportedToken()` (lines 637–656) only sets `supportedTokenOracle[token]` and `tokenBridge[token]`; it never writes to `tokenFeeBps[token]`. The only mechanism to set a non-zero token fee is the separate admin call `setTokenFeeBps()` (lines 583–594), which requires `DEFAULT_ADMIN_ROLE` and must be called after token addition.

The ETH deposit path (lines 265–278) always applies the global `feeBps`, so the two paths are asymmetric: ETH deposits are fee-bearing by default; token deposits are fee-free by default.

## Impact Explanation
This is **High — Theft of unclaimed yield**. The protocol is designed to collect fee revenue on all deposits. A depositor who would pay `feeBps` on an ETH deposit can instead deposit an equivalent value of a supported ERC20 token (e.g., wstETH) and receive the same `wrsETH` output with zero fee deducted. The protocol permanently loses the fee revenue it intended to collect on that economic value. The loss is unbounded and repeatable for every token deposit made while `tokenFeeBps[token] == 0`.

## Likelihood Explanation
The condition is always present for any token added without a subsequent `setTokenFeeBps()` call. No special privileges, flash loans, or timing are required. The attacker only needs to hold a supported token and call `deposit(token, amount, referralId)` instead of the ETH overload. wstETH is already integrated, making this immediately exploitable on Arbitrum mainnet.

## Recommendation
Initialize `tokenFeeBps[token]` inside `addSupportedToken()` to prevent silent zero-fee defaults:

```solidity
function addSupportedToken(address token, address oracle, address bridge, uint256 _feeBps)
    external onlyRole(TIMELOCK_ROLE)
{
    // ... existing checks ...
    if (_feeBps > 10_000) revert InvalidFeeAmount();
    tokenFeeBps[token] = _feeBps;
    // ...
}
```

Additionally, audit all currently supported tokens and call `setTokenFeeBps()` for any that have not yet had their fee explicitly set.

## Proof of Concept
1. Protocol sets `feeBps = 30` (0.30% ETH deposit fee).
2. Admin calls `addSupportedToken(wstETH, oracle, bridge)` — `tokenFeeBps[wstETH]` remains `0`.
3. Alice wants to acquire `wrsETH` equivalent to 10 ETH of value.
   - **ETH path**: `deposit{value: 10 ETH}("")` → `fee = 10e18 * 30 / 10000 = 0.03 ETH`; Alice receives `rsETHAmount` based on `9.97 ETH`.
   - **Token path**: `deposit(wstETH, wstETHAmount, "")` → `feeBpsForToken = tokenFeeBps[wstETH] = 0`; `fee = 0`; Alice receives `rsETHAmount` based on the full `10 ETH` equivalent.
4. Alice uses the token path, receiving more `wrsETH` and paying zero protocol fee, permanently depriving the protocol of the 0.03 ETH fee.

**Foundry test plan**: Deploy `RSETHPool` with `feeBps = 30`, add wstETH via `addSupportedToken` without calling `setTokenFeeBps`, then assert that `viewSwapRsETHAmountAndFee(amount, wstETH).fee == 0` while `viewSwapRsETHAmountAndFee(amount).fee == amount * 30 / 10_000`. Confirm the token `deposit` call transfers the full `rsETHAmount` with no fee accrued in `feeEarnedInToken[wstETH]`.
