### Title
Single-Participant Abort Enables Indefinite Key-Event Lifecycle Manipulation - (`crates/contract/src/state/key_event.rs`)

### Summary

`vote_abort_key_event_instance` requires only **one** attested participant's vote to immediately destroy the active key-event instance. A single Byzantine participant below the signing threshold can call this repeatedly to prevent key generation or resharing from ever completing, indefinitely stalling the protocol lifecycle.

### Finding Description

The `vote_abort` method in `KeyEvent` sets `self.instance = None` the moment a single participant calls it, with no threshold accumulation: [1](#0-0) 

The public contract entry point `vote_abort_key_event_instance` gates this only on `assert_caller_is_attested_participant_and_protocol_active()` — i.e., any single attested participant in the active set: [2](#0-1) 

After the abort, `instance` is `None` and `next_attempt_id` has already been incremented by the preceding `start` call: [3](#0-2) 

The leader must then call `start_keygen_instance` or `start_reshare_instance` again with the new `attempt_id`. The Byzantine participant immediately calls `vote_abort_key_event_instance` again with the new `KeyEventId`. This cycle repeats at the cost of only NEAR gas per transaction.

The analog to M-07 is direct: just as `buyShareCredFor` allowed anyone to reset another user's share-lock timer, `vote_abort_key_event_instance` allows a single participant to reset the key-event timer — the difference being that the victim here is the entire MPC network rather than one user.

### Impact Explanation

**Medium — contract execution-flow manipulation that breaks production safety invariants.**

1. **Key generation stall (Initializing state):** A Byzantine participant can prevent any new domain from ever completing key generation. New signing domains are permanently blocked.

2. **Resharing stall (Resharing state):** Resharing is triggered either by governance (`vote_new_parameters`) or automatically by `verify_tee` when participants have expired TEE attestations. If resharing cannot complete, the participant set is frozen in a transitional state. As more attestations expire over time, `verify_tee` may eventually set `accept_requests = false`, halting all signature processing: [4](#0-3) 

3. **Signature-request freeze:** Once `accept_requests` is false, all pending and new `sign`, `request_app_private_key`, and `verify_foreign_transaction` calls are rejected, permanently freezing funds that depend on MPC-signed transactions.

### Likelihood Explanation

A single attested participant below the signing threshold is sufficient. Participants are admitted through `vote_new_parameters` (threshold governance), but once admitted, a single one can abuse `vote_abort_key_event_instance`. The cost is only NEAR gas (~a few cents per abort). The attack is especially realistic during a resharing triggered by TEE attestation expiry, where the Byzantine participant has a strong incentive to prevent their own removal from the participant set.

### Recommendation

Replace the single-vote abort with a threshold-based abort. Accumulate abort votes in a `BTreeSet` inside `KeyEventInstance` (analogous to `completed`), and only set `self.instance = None` once the number of abort votes reaches the reconstruction threshold. This mirrors how `vote_pk` and `vote_reshared` require threshold agreement before advancing state.

### Proof of Concept

1. Network is in `Resharing` state (e.g., triggered by `verify_tee` after a participant's attestation expires).
2. Byzantine participant `B` (below threshold, but attested) observes the current `KeyEventId` from contract state.
3. Leader calls `start_reshare_instance(key_event_id_0)` → `instance` is set, `next_attempt_id` becomes `1`.
4. `B` calls `vote_abort_key_event_instance(key_event_id_0)` → `instance = None` immediately. [1](#0-0) 

5. Leader calls `start_reshare_instance(key_event_id_1)` → `instance` is set, `next_attempt_id` becomes `2`.
6. `B` calls `vote_abort_key_event_instance(key_event_id_1)` → `instance = None` again.
7. Steps 5–6 repeat indefinitely. Resharing never completes. The network remains stuck in `Resharing` state. As more attestations expire, `verify_tee` eventually sets `accept_requests = false`: [5](#0-4) 

All signature requests are then rejected, freezing all cross-chain operations that depend on the MPC network.

    pub fn start(&mut self, key_event_id: KeyEventId, timeout_blocks: u64) -> Result<(), Error> {
        self.cleanup_if_timed_out();
        if self.instance.is_some() {
            return Err(KeyEventError::ActiveKeyEvent.into());
        }
        let expected_key_event_id =
            KeyEventId::new(self.epoch_id, self.domain.id, self.next_attempt_id);
        if key_event_id != expected_key_event_id {
            return Err(KeyEventError::KeyEventIdMismatch.into());
        }
        self.verify_leader()?;
        self.instance = Some(KeyEventInstance::new(self.next_attempt_id, timeout_blocks));
        self.next_attempt_id = self.next_attempt_id.next();
        Ok(())
    }

    /// Casts a vote to abort the current keygen instance.
    /// A new instance needs to be started later to start a new keygen attempt.
    pub fn vote_abort(&mut self, key_event_id: KeyEventId) -> Result<(), Error> {
        let candidate = self.verify_vote(&key_event_id)?;
        if self
            .instance
            .as_ref()
            .unwrap()
            .completed
            .contains(&candidate)
        {
            return Err(VoteError::VoteAlreadySubmitted.into());
        }
        self.instance = None;
        Ok(())
    }
