### Title
Hardcoded `only_query = false` in `BroadcastedTransaction`→`ExecutableTransactionInput` Conversion Causes Wrong Transaction Version in RPC Simulation and Fee Estimation — (File: `crates/apollo_rpc/src/v0_8/api/mod.rs`)

---

### Summary

The `TryFrom<BroadcastedTransaction> for ExecutableTransactionInput` implementation unconditionally sets `only_query = false` for every `Invoke` and `DeployAccount` transaction submitted to `starknet_estimateFee` and `starknet_simulateTransactions`. The `only_query` flag is the Starknet-protocol "query bit": when `true` it sets the 128th bit of the transaction version (e.g. `0x100000000000000000000000000000003` for V3), which is what the Starknet spec requires for simulation/estimation calls. Because the flag is hardcoded to `false`, every simulation and fee-estimation call executes with the non-query version (`0x3`), causing the contract's `get_execution_info` syscall to return the wrong transaction version. Any account contract that branches on the query bit — a common pattern for skipping signature verification during simulation — will receive incorrect execution results, and the RPC will return those results as authoritative.

---

### Finding Description

In `crates/apollo_rpc/src/v0_8/api/mod.rs` the conversion from the wire type to the internal execution type is:

```rust
impl TryFrom<BroadcastedTransaction> for ExecutableTransactionInput {
    type Error = ErrorObjectOwned;
    fn try_from(value: BroadcastedTransaction) -> Result<Self, Self::Error> {
        // TODO(yair): pass the right value for only_query field.
        match value {
            BroadcastedTransaction::Declare(tx) => Ok(tx.try_into()?),
            BroadcastedTransaction::DeployAccount(tx) => Ok(Self::DeployAccount(tx.into(), false)),
            BroadcastedTransaction::Invoke(tx) => Ok(Self::Invoke(tx.into(), false)),
        }
    }
}
``` [1](#0-0) 

This `TryFrom` is the sole conversion path used by both `estimate_fee` and `simulate_transactions`:

```rust
let executable_txns =
    transactions.into_iter().map(|tx| tx.try_into()).collect::<Result<_, _>>()?;
``` [2](#0-1) [3](#0-2) 

The `only_query` boolean is then threaded into `ExecutionFlags` and ultimately into `signed_tx_version`, which sets the 128th bit of the transaction version when `only_query = true`:

```rust
pub fn signed_tx_version(
    tx_version: &TransactionVersion,
    transaction_options: &TransactionOptions,
) -> TransactionVersion {
    let query_only_bit = *QUERY_VERSION_BASE;
    if transaction_options.only_query {
        TransactionVersion(tx_version.0 + query_only_bit)
    } else {
        *tx_version
    }
}
``` [4](#0-3) 

The version produced by `signed_tx_version` is what the contract reads via the `get_execution_info` syscall:

```rust
pub fn tx_version_for_get_execution_info(&self) -> TransactionVersion {
    // ...
    tx_context.tx_info.signed_version()
}
``` [5](#0-4) 

Because `only_query` is always `false`, the contract always sees the non-query version (`0x3` for V3, `0x1` for V1) even during simulation. The Starknet specification requires the query bit to be set for simulation/estimation so that contracts can distinguish these calls from real sequenced transactions.

---

### Impact Explanation

Any account contract that inspects the transaction version in `__validate__` or `__execute__` to detect simulation mode (a standard pattern: skip expensive signature verification when the query bit is set) will behave as if it is being sequenced for real. Concretely:

- **Wrong fee estimate**: a contract that skips heavy computation in query mode will over-report gas consumption, returning a fee estimate that is higher than the actual on-chain cost.
- **Spurious simulation failure**: a contract that requires the query bit to be set for simulation (e.g. to allow unsigned calls) will revert during `starknet_simulateTransactions`, even though the real transaction would succeed.
- **Spurious simulation success**: conversely, a contract that rejects the query bit will succeed in simulation but fail on-chain, giving a false-positive fee estimate.

All three outcomes are returned by the RPC as authoritative values with no indication that the query bit was missing.

---

### Likelihood Explanation

The trigger is entirely unprivileged: any caller of `starknet_estimateFee` or `starknet_simulateTransactions` with a V3 `Invoke` or `DeployAccount` transaction reaches the buggy path. The TODO comment (`// TODO(yair): pass the right value for only_query field.`) confirms the defect is known but unresolved. The affected RPC methods are among the most frequently called in any Starknet deployment.

---

### Recommendation

Replace the hardcoded `false` with the correct value derived from the transaction's version field. Per the Starknet spec, a `BroadcastedTransaction` submitted to `estimateFee` or `simulateTransactions` is a query transaction and must use `only_query = true`:

```rust
impl TryFrom<BroadcastedTransaction> for ExecutableTransactionInput {
    type Error = ErrorObjectOwned;
    fn try_from(value: BroadcastedTransaction) -> Result<Self, Self::Error> {
        match value {
            BroadcastedTransaction::Declare(tx) => Ok(tx.try_into()?),
            BroadcastedTransaction::DeployAccount(tx) => Ok(Self::DeployAccount(tx.into(), true)),
            BroadcastedTransaction::Invoke(tx) => Ok(Self::Invoke(tx.into(), true)),
        }
    }
}
```

Alternatively, if the same `TryFrom` is also used for non-query paths (e.g. `add_invoke_transaction`), introduce a separate conversion that accepts an explicit `only_query` parameter and pass the correct value at each call site.

### Proof of Concept

```rust
// Pseudocode – calls starknet_simulateTransactions with a V3 Invoke
// whose __validate__ checks the query bit.
//
// Contract __validate__ (Cairo):
//   let tx_info = get_execution_info().tx_info;
//   // Starknet spec: query bit must be set for simulation.
//   assert(tx_info.version == 0x100000000000000000000000000000003,
//          'TX_INFO_MISMATCH');  // always fires because only_query=false
//   return ();

let invoke = BroadcastedTransaction::Invoke(InvokeTransaction::V3(InvokeTransactionV3 {
    // ... fields ...
}));

// Both calls go through TryFrom which hardcodes only_query = false.
let estimate_result = rpc.estimate_fee(vec![invoke.clone()], vec![], block_id).await;
// → TRANSACTION_EXECUTION_ERROR: TX_INFO_MISMATCH
//   (contract expected query version 0x1000...0003, got 0x3)

let simulate_result = rpc.simulate_transactions(block_id, vec![invoke], vec![]).await;
// → TRANSACTION_EXECUTION_ERROR: TX_INFO_MISMATCH
//   (same root cause; real on-chain execution would succeed because
//    the sequencer uses new_for_sequencing() which also sets only_query=false,
//    but the contract's __validate__ is written to accept only_query=false
//    for real txs and only_query=true for simulation — the simulation
//    path never delivers the expected query version)
```

The existing test `simulate_with_query_bit_outputs_same_as_no_query_bit` only covers deprecated V1 transactions where the version field is not inspected by the test contract; it does not cover V3 contracts that branch on the query bit, leaving the defect undetected. [6](#0-5)
