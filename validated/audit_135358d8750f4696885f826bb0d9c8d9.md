Audit Report

## Title
Unvalidated `latestRoundData` Return Values Allow Stale Prices to Trigger Protocol-Wide Pause - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards `roundId`, `updatedAt`, and `answeredInRound`, performing no staleness validation. A stale (artificially depressed) price fed into the publicly callable `LRTOracle.updateRSETHPrice()` can cause the downside-protection branch to fire, pausing `LRTDepositPool`, `LRTWithdrawalManager`, and `LRTOracle` and temporarily freezing all user deposits and withdrawals.

## Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol` at line 52, `getAssetPrice()` captures only `price` from `latestRoundData()`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
``` [1](#0-0) 

No check is made that `updatedAt != 0` (incomplete round), `answeredInRound >= roundId` (stale round), or that `block.timestamp - updatedAt` is within an acceptable freshness window. By contrast, `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` in the same repository correctly validates all three conditions: [2](#0-1) 

The stale price propagates through two confirmed paths:

**Path 1 — rsETH price update (publicly callable, no access control):**
`LRTOracle.updateRSETHPrice()` (line 87, `public whenNotPaused`) → `_updateRsETHPrice()` → `_getTotalEthInProtocol()` → `getAssetPrice(asset)` → `ChainlinkPriceOracle.getAssetPrice()`. [3](#0-2) [4](#0-3) 

**Path 2 — Deposit minting (user-triggered):**
`LRTDepositPool.depositAsset()` / `depositETH()` → `_beforeDeposit()` → `getRsETHAmountToMint()` → `lrtOracle.getAssetPrice(asset)` → `ChainlinkPriceOracle.getAssetPrice()`. [5](#0-4) 

The downside-protection branch in `_updateRsETHPrice()` compares `newRsETHPrice` (computed from the stale total ETH value) against `highestRsethPrice`. If the stale price is depressed enough to exceed `pricePercentageLimit`, the following executes:

```solidity
if (isPriceDecreaseOffLimit) {
    if (!lrtDepositPool.paused()) lrtDepositPool.pause();
    if (!withdrawalManager.paused()) withdrawalManager.pause();
    _pause();
    return;
}
``` [6](#0-5) 

Existing guards are insufficient: `whenNotPaused` only blocks calls after the protocol is already paused; it does not prevent the stale price from triggering the pause in the first place. The `pricePercentageLimit` check is a protection mechanism, not a staleness guard.

## Impact Explanation
**Medium — Temporary freezing of funds.**

A successful trigger pauses `LRTDepositPool` and `LRTWithdrawalManager`, freezing all user deposits and withdrawals until an admin manually unpauses. This matches the allowed impact "Temporary freezing of funds." The freeze is not permanent because an admin can unpause, but it is externally triggerable by any unprivileged caller with no capital requirement.

Additionally, a stale inflated price on Path 2 causes depositors to receive more rsETH than entitled, diluting existing holders — matching "Contract fails to deliver promised returns" (Low), though the primary impact is the Medium freeze.

## Likelihood Explanation
Chainlink feeds can return stale data during network congestion, heartbeat lapses, or L2 sequencer downtime (Arbitrum, Optimism, Base — all chains where this protocol is deployed). The trigger requires no attacker capital: any address calls the public `updateRSETHPrice()` after a stale round is published. L2 sequencer outages are historically documented, making this a realistic, repeatable scenario requiring no privileged access.

## Recommendation
In `contracts/oracles/ChainlinkPriceOracle.sol`, capture and validate all `latestRoundData` return values, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral.sol`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();
require(answeredInRound >= roundId, "Stale price");
require(updatedAt != 0, "Incomplete round");
require(block.timestamp - updatedAt <= MAX_STALENESS, "Price too old");
require(price > 0, "Non-positive price");
```

Per-feed configurable `MAX_STALENESS` thresholds should be stored and managed by `LRTManager`, and a sequencer uptime feed check should be added for all L2 deployments.

## Proof of Concept
1. Deploy a mock Chainlink feed for stETH/ETH that returns a stale round: `answeredInRound < roundId`, `updatedAt` set to `block.timestamp - 2 days`, and `price` set to a value 5% below the current `highestRsethPrice` (assuming `pricePercentageLimit` is set to e.g. 3%).
2. Register the mock feed via `ChainlinkPriceOracle.updatePriceFeedFor(stETH, mockFeed)` (or fork mainnet with the existing feed in a stale state).
3. Call `LRTOracle.updateRSETHPrice()` from any EOA.
4. `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)`, which returns the stale depressed price without reverting.
5. `newRsETHPrice` is computed below `highestRsethPrice` by more than `pricePercentageLimit`.
6. The condition at `LRTOracle.sol:277` fires: `LRTDepositPool`, `LRTWithdrawalManager`, and `LRTOracle` are all paused.
7. Assert that `lrtDepositPool.paused() == true` and `withdrawalManager.paused() == true`.
8. Attempt a deposit — it reverts, confirming the freeze.
