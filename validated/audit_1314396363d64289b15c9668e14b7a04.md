### Title
Stale `rsETHPrice` After Price Oracle Update Causes Incorrect rsETH Minting - (File: `contracts/LRTOracle.sol`)

---

### Summary
`LRTOracle.updatePriceOracleFor()` and `updatePriceOracleForValidated()` replace the live price feed for a supported asset but never call `updateRSETHPrice()` to refresh the stored `rsETHPrice` state variable. Because `LRTDepositPool.getRsETHAmountToMint()` divides the **live** new-oracle asset price by the **stale** stored `rsETHPrice`, every deposit executed in the window between the oracle swap and the next price update mints an incorrect number of rsETH shares.

---

### Finding Description

`LRTOracle` maintains a stored exchange rate `rsETHPrice` that is updated only when `updateRSETHPrice()` / `_updateRsETHPrice()` is explicitly called. [1](#0-0) 

Two admin-callable functions replace the price oracle for a supported asset without refreshing this stored rate: [2](#0-1) [3](#0-2) 

`LRTDepositPool.getRsETHAmountToMint()` computes the rsETH to mint as:

```
rsethAmountToMint = (amount × lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()
``` [4](#0-3) 

`getAssetPrice(asset)` is a **live** call that immediately reflects the newly registered oracle, while `rsETHPrice()` returns the **stale** stored value that was computed against the old oracle. The two quantities are now inconsistent. [5](#0-4) 

---

### Impact Explanation

**Scenario A — new oracle returns a higher asset price (e.g. stETH oracle updated from 1.00 ETH to 1.05 ETH):**

- `getAssetPrice(stETH)` → 1.05 ETH (live, new oracle)
- `rsETHPrice` → 1.00 ETH (stale, computed under old oracle)
- A depositor of 100 stETH receives `100 × 1.05 / 1.00 = 105 rsETH` instead of the correct `100 × 1.05 / 1.05 = 100 rsETH`
- The 5 extra rsETH dilute all existing holders — this is **theft of unclaimed yield** from current rsETH holders.

**Scenario B — new oracle returns a lower asset price:**

- Depositors receive fewer rsETH shares than the protocol owes them — the **contract fails to deliver promised returns**.

Impact classification: **High** (theft of unclaimed yield) for Scenario A; **Low** (contract fails to deliver promised returns) for Scenario B.

---

### Likelihood Explanation

Oracle updates are routine operational events (e.g. migrating to a new Chainlink feed, switching price adapter implementations). No special attacker capability is required beyond monitoring the mempool for `AssetPriceOracleUpdate` events and submitting a deposit in the same block before `updateRSETHPrice()` is called. `updateRSETHPrice()` is a public function but is not called atomically inside `updatePriceOracleFor()`, leaving an exploitable gap. [6](#0-5) 

---

### Recommendation

`updatePriceOracleFor()` and `updatePriceOracleForValidated()` should call `_updateRsETHPrice()` (or the public `updateRSETHPrice()`) **after** writing the new oracle address, so that `rsETHPrice` is always consistent with the currently registered oracles before any deposit can use the new price feed. This mirrors the recommendation in the reference report: the entry point for the parameter change should atomically trigger the dependent state update.

---

### Proof of Concept

1. Protocol state: stETH oracle returns 1.00 ETH/stETH; `rsETHPrice` = 1.00 ETH; rsETH supply = 1 000 rsETH backed by 1 000 stETH.
2. Admin calls `updatePriceOracleFor(stETH, newOracle)` where `newOracle.getAssetPrice(stETH)` returns 1.05 ETH. `rsETHPrice` remains 1.00 ETH. [3](#0-2) 
3. Attacker immediately calls `depositAsset(stETH, 100e18, 0, "")`.
4. `getRsETHAmountToMint(stETH, 100e18)` = `100e18 × 1.05e18 / 1.00e18` = **105 rsETH**. [4](#0-3) 
5. Correct mint (after price update) would be `100e18 × 1.05e18 / 1.05e18` = **100 rsETH**.
6. The 5 extra rsETH represent a 0.5 % dilution of the existing 1 000 rsETH holders, extracted by the attacker at zero cost beyond the deposit itself.
7. When `updateRSETHPrice()` is eventually called, `rsETHPrice` rises to ~1.05 ETH, but the attacker already holds the over-minted shares. [6](#0-5)
