### Title
Caller-Agnostic `VerifyForeignTransactionRequest` Key Allows Any Unprivileged Account to Exhaust the Fan-Out Queue and Block Legitimate Foreign-Chain Verification Requests - (File: `crates/contract/src/lib.rs`, `crates/contract/src/pending_requests.rs`)

---

### Summary

The `verify_foreign_transaction` endpoint stores pending requests under a key (`VerifyForeignTransactionRequest`) that does **not** include the caller's account ID. Because the fan-out queue for each key is capped at `MAX_PENDING_REQUEST_FAN_OUT = 128`, any unprivileged NEAR account can submit 128 identical requests for a specific foreign-chain transaction at a cost of 128 yoctonear (≈ $0), filling the queue and causing every subsequent legitimate submission for that same transaction to be rejected with `PendingRequestQueueFull`. The contract remains in this blocked state until the MPC nodes respond and drain the queue — at which point the attacker can immediately refill it.

---

### Finding Description

**Root cause — caller-agnostic request key**

`sign()` and `request_app_private_key()` both embed the caller's identity into their stored request key. `SignatureRequest::new()` derives a `tweak` from `(predecessor_id, path)`, so two different callers always produce different map keys:

```rust
// crates/near-mpc-crypto-types/src/sign.rs:118-125
pub fn new(domain: DomainId, payload: Payload, predecessor_id: &AccountId, path: &str) -> Self {
    let tweak = crate::kdf::derive_tweak(predecessor_id, path);
    SignatureRequest { domain_id: domain, tweak, payload }
}
```

`verify_foreign_transaction()` does the opposite. The conversion function `args_into_verify_foreign_tx_request` strips the caller entirely:

```rust
// crates/contract/src/dto_mapping.rs:840-848
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
```

The resulting `VerifyForeignTransactionRequest` struct contains only `(request, domain_id, payload_version)` — no caller field:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs:124-128
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
```

This is explicitly confirmed by the contract's own test comment:

> `"duplicate foreign-tx requests from different callers should fan out"` — `crates/contract/src/lib.rs:3255`

**Fan-out queue cap**

`push_pending_yield` enforces a hard cap of 128 entries per key:

```rust
// crates/contract/src/pending_requests.rs:37,51-57
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;
// ...
if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
    env::panic_str(&RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }.to_string());
}
```

**Attack path**

1. Attacker observes (or predicts) a target foreign-chain `tx_id` that a legitimate bridge user will need to verify.
2. Attacker submits 128 `verify_foreign_transaction` calls with the identical `(request, domain_id, payload_version)` tuple, each attaching 1 yoctonear. Total cost: 128 yoctonear.
3. The queue for that request key is now full.
4. The legitimate user's `verify_foreign_transaction` call panics with `PendingRequestQueueFull` and their yield is never created.
5. When MPC nodes eventually respond and drain the queue, the attacker immediately resubmits 128 requests to refill it, creating a persistent block.

---

### Impact Explanation

This is a **Medium** impact finding under the "request-lifecycle manipulation that breaks production safety/accounting invariants" category.

- Any specific foreign-chain transaction can be permanently prevented from receiving a verified MPC attestation.
- Legitimate bridge users are denied service for targeted transactions without any recourse until the attacker stops.
- The contract state is corrupted in the sense that the queue is occupied by attacker-controlled yields, not legitimate user yields, breaking the expected request-lifecycle accounting invariant.
- No funds are directly stolen, but bridge operations depending on `verify_foreign_transaction` attestations can be indefinitely blocked for specific transactions.

---

### Likelihood Explanation

**High.** The attack requires:
- A standard NEAR account (no privileges).
- 128 yoctonear per blocking cycle (effectively free).
- Knowledge of the target `tx_id`, which is public on the foreign chain.

The attacker can automate refilling the queue on every MPC response, making the block persistent with negligible ongoing cost.

---

### Recommendation

Include the caller's `predecessor_account_id` in the `VerifyForeignTransactionRequest` key, analogous to how `SignatureRequest` derives a caller-specific `tweak`. This ensures each caller has an independent queue slot and no single account can exhaust the shared cap for a given foreign-chain transaction.

Alternatively, enforce a per-account submission limit for `verify_foreign_transaction` requests to bound the number of queue slots any single caller can occupy.

---

### Proof of Concept

The contract's own unit test demonstrates the shared-queue property:

```rust
// crates/contract/src/lib.rs:3208-3263
// alice and bob submit the same request_args from different accounts
contract.verify_foreign_transaction(request_args.clone()); // alice
contract.verify_foreign_transaction(request_args);         // bob

// Both yields land in the SAME map entry
assert_eq!(
    contract.pending_verify_foreign_tx_requests.get(&request).map(|q| q.len()),
    Some(2),
    "duplicate foreign-tx requests from different callers should fan out",
);
```

Extending this to 128 submissions from a single attacker account fills the queue:

```
for _ in 0..128 {
    contract.verify_foreign_transaction(request_args.clone()); // attacker
}
// Queue is now full. The 129th call (from the legitimate user) panics:
// "Pending-request queue is full (limit: 128)"
contract.verify_foreign_transaction(request_args); // legitimate user → REJECTED

pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
```
