### Title
Unguarded `bytes32(memos[0])` in `get_mirror_info` Permanently Drops Mirror Coin States — (`chia/wallet/db_wallet/db_wallet_puzzles.py`)

### Summary

`get_mirror_info` calls `bytes32(memos[0])` at line 108 without a `try/except`. When an on-chain mirror coin's parent spend carries a `CREATE_COIN` memo whose first element is not exactly 32 bytes, `bytes32` raises a raw `ValueError` that is distinct from the function's documented sentinel `ValueError("The provided puzzle and solution do not create a mirror coin")`. This exception propagates through `DataLayerWallet.coin_added` and `WalletStateManager.coin_added` (neither has a guard), reaches the `except Exception` handler in `_add_coin_states`, is classified as a non-retryable error, and the coin state is permanently removed from the retry store. The mirror coin is silently and permanently dropped from wallet tracking.

### Finding Description

**Root cause — `get_mirror_info`, line 108:**

```python
memos: list[bytes] = condition.at("rrrf").as_python()
launcher_id = bytes32(memos[0])          # raises ValueError if len != 32
return launcher_id, [url for url in memos[1:]]
```

`bytes32` (from `chia_rs.sized_bytes`) raises `ValueError` for any input whose length is not exactly 32. There is no `try/except` here. [1](#0-0) 

**Propagation — `DataLayerWallet.coin_added`, line 780:**

```python
launcher_id, urls = get_mirror_info(parent_spend.puzzle_reveal, parent_spend.solution)
```

No exception handling surrounds this call. [2](#0-1) 

**Propagation — `WalletStateManager.coin_added`, line 2405:**

```python
await self.wallets[wallet_id].coin_added(coin, height, peer, coin_data)
```

No exception handling here either. [3](#0-2) 

**Terminal handler — `_add_coin_states`, lines 2227–2235:**

```python
except Exception as e:
    self.log.exception(f"Failed to add coin_state: {coin_state}, error: {e}")
    if rollback_wallets is not None:
        self.wallets = rollback_wallets
    if isinstance(e, (PeerRequestException, aiosqlite.Error)):
        await self.retry_store.add_state(coin_state, peer.peer_node_id, fork_height)
    else:
        await self.retry_store.remove_state(coin_state)   # ← permanent drop
    continue
```

`ValueError` is not a `PeerRequestException` or `aiosqlite.Error`, so the coin state is **permanently removed from the retry store** and never re-processed. [4](#0-3) 

### Impact Explanation

Any unprivileged on-chain actor can spend a coin to create a coin with `puzzle_hash == MIRROR_PUZZLE_HASH` while embedding a `CREATE_COIN` condition whose memo list has a first element shorter or longer than 32 bytes (e.g., `b'short'`). When the victim wallet syncs and calls `coin_added` for that mirror coin, the `ValueError` from `bytes32(memos[0])` causes the coin state to be permanently discarded. The wallet will never track that mirror coin, and any DL store whose mirror URL updates depend on it will permanently lose those updates. The impact is scoped to the specific crafted mirror coin per attack, but the attacker can repeat this for any number of DL stores at negligible cost (one small on-chain transaction per targeted store).

The claim that "all mirror coins at that height" are broken is slightly overstated — the `continue` in the exception handler means only the crafted coin state is dropped, not adjacent ones. The permanent-drop impact on the targeted mirror coin itself is accurate and real.

### Likelihood Explanation

The attack requires only a standard on-chain spend (no key compromise, no peer control, no special privilege). The attacker controls the parent puzzle and solution entirely, so crafting a memo with a non-32-byte first element is trivial. The victim wallet must have a DL wallet active and be subscribed to a store for the mirror coin to be routed through `DataLayerWallet.coin_added`.

### Recommendation

Wrap `bytes32(memos[0])` in `get_mirror_info` with a `try/except (ValueError, TypeError)` and re-raise as the canonical sentinel:

```python
memos: list[bytes] = condition.at("rrrf").as_python()
try:
    launcher_id = bytes32(memos[0])
except (ValueError, TypeError):
    raise ValueError("The provided puzzle and solution do not create a mirror coin")
return launcher_id, [url for url in memos[1:]]
```

This ensures `get_mirror_info` only ever raises its documented `ValueError` with the expected message, which callers can safely catch and handle.

### Proof of Concept

```python
from chia.types.blockchain_format.program import Program, INFINITE_COST
from chia.wallet.db_wallet.db_wallet_puzzles import get_mirror_info, MIRROR_PUZZLE_HASH
from chia.types.condition_opcodes import ConditionOpcode

# Craft a parent puzzle that outputs CREATE_COIN with MIRROR_PUZZLE_HASH
# and a memo list where memos[0] is b'short' (5 bytes, not 32)
parent_puzzle = Program.to(1)  # identity: returns solution as conditions
memo_short = b'short'          # not 32 bytes
conditions = Program.to([
    [ConditionOpcode.CREATE_COIN, MIRROR_PUZZLE_HASH, 1000,
     [memo_short, b'http://example.com']]
])
parent_solution = conditions

try:
    get_mirror_info(parent_puzzle, parent_solution)
    assert False, "Should have raised"
except ValueError as e:
    # BUG: raises raw bytes32 ValueError, not the sentinel message
    assert str(e) == "The provided puzzle and solution do not create a mirror coin", \
        f"Got unexpected message: {e}"
```

This test currently fails because `bytes32(b'short')` raises `ValueError` with a message about wrong byte length, not the expected sentinel — confirming the invariant is violated.
