### Title
Missing DamgardEtAl Honest-Majority Participant-Count Check in `verify_tee` Kickout Path — (`File: crates/contract/src/lib.rs`)

---

### Summary

`verify_tee` can kick out participants with expired TEE attestations and immediately initiate a resharing with the surviving set. The guard it applies — `validate_governance_against_reconstruction` — only checks that the remaining count satisfies the governance threshold and that the governance threshold dominates the raw `ReconstructionThreshold` value `t`. It does **not** apply the DamgardEtAl-specific honest-majority bound `2t − 1 ≤ n`. When a DamgardEtAl domain is active, the surviving participant count can satisfy the governance check while being strictly below the minimum required by the protocol, causing the contract to enter a Resharing state that no set of honest nodes can ever complete — permanently freezing the network's signing capability.

---

### Finding Description

`validate_domain_threshold` in `crates/contract/src/primitives/domain.rs` enforces the DamgardEtAl-specific bound:

```rust
if domain.protocol == Protocol::DamgardEtAl {
    let required = t.checked_mul(2).and_then(|x| x.checked_sub(1))...;
    if required > num_participants {
        return Err(DomainError::InsufficientParticipantsForProtocol { ... });
    }
}
``` [1](#0-0) 

This function is correctly called inside `process_new_parameters_proposal` (the normal governance path) for every effective domain:

```rust
for domain in &effective_domains {
    validate_domain_threshold(domain, new_num_participants)?;
}
``` [2](#0-1) 

However, the `verify_tee` kickout path in `lib.rs` takes a completely different route. After computing `participants_with_valid_attestation`, it only calls:

```rust
ThresholdParameters::validate_governance_against_reconstruction(
    u64::try_from(remaining).expect("participant count fits in u64"),
    current_params.threshold(),
    max_reconstruction_threshold,
)?;
``` [3](#0-2) 

`validate_governance_against_reconstruction` delegates to `validate_threshold`, which checks `governance_threshold ≤ n` and `governance_threshold ≥ max_reconstruction_threshold` (the raw `t`). It never calls `validate_domain_threshold` and therefore never enforces `2t − 1 ≤ n` for DamgardEtAl. [4](#0-3) 

After the guard passes, the code calls `validate_incoming_proposal` (which also has no domain-threshold check) and then `transition_to_resharing_no_checks`:

```rust
current_params.validate_incoming_proposal(&threshold_parameters)?;
let proposed_parameters = ProposedThresholdParameters::new(threshold_parameters, BTreeMap::new());
let res = running_state.transition_to_resharing_no_checks(&proposed_parameters);
``` [5](#0-4) 

`transition_to_resharing_no_checks` performs no domain-threshold validation by design: [6](#0-5) 

---

### Impact Explanation

**Critical — permanent freezing of funds controlled by the MPC network.**

Consider a live deployment with:
- 5 participants, governance threshold = 3
- One active DamgardEtAl domain with `reconstruction_threshold = 3` (requires `2×3−1 = 5` participants; this is a valid initial state since `5 ≤ 5`)

If 2 participants' TEE attestations expire, `remaining = 3`. The guard evaluates:
- `validate_governance_against_reconstruction(3, 3, Some(3))` → `3 ≥ 3` (governance ≥ reconstruction) → **passes**
- DamgardEtAl minimum: `2×3−1 = 5 > 3` → **not checked**

The contract transitions to `Resharing` with only 3 participants. No set of 3 nodes can execute the DamgardEtAl resharing protocol that requires 5 active participants. The resharing can never complete. The contract is stuck in `Resharing` indefinitely, unable to process any signature requests. All funds whose cross-chain control depends on MPC signatures are permanently frozen.

---

### Likelihood Explanation

DamgardEtAl (`V2Secp256k1`) is listed as "Ongoing" in the deployed protocol table. [7](#0-6) 

TEE attestations expire naturally; the contract already has logic to handle expiry-triggered kickouts. Any participant (or any external account that can call `verify_tee` after becoming a participant) can call `verify_tee` at any time:

```rust
pub fn verify_tee(&mut self) -> Result<bool, Error> {
    self.voter_or_panic();
``` [8](#0-7) 

The triggering condition — enough attestations expired so that `remaining ≥ governance_threshold` but `remaining < 2t−1` — is reachable whenever `t ≥ 2` (always true) and the participant set is not at its absolute minimum. An adversary who controls one or more participants can deliberately allow their own attestations to expire to push the surviving count into this window, then call `verify_tee` to lock the network.

---

### Recommendation

In the `TeeValidationResult::Partial` branch of `verify_tee`, after computing `participants_with_valid_attestation`, add a per-domain threshold check identical to the one in `process_new_parameters_proposal`:

```rust
let new_num_participants = u64::try_from(remaining).expect("fits in u64");
for domain in running_state.domains.domains() {
    if let Err(err) = validate_domain_threshold(domain, new_num_participants) {
        log!("Kicking out participants would violate domain threshold ({:?}); refusing kickout.", err);
        self.accept_requests = false;
        return Ok(false);
    }
}
```

This mirrors the existing guard pattern and ensures that the DamgardEtAl honest-majority bound `2t−1 ≤ n` is enforced before any resharing is initiated via the TEE-kickout path, consistent with how `validate_domain_threshold` is applied in `process_new_parameters_proposal`. [9](#0-8) 

---

### Proof of Concept

1. Deploy the contract with 5 participants, governance threshold = 3, and one DamgardEtAl domain with `reconstruction_threshold = 3` (valid: `2×3−1 = 5 ≤ 5`).
2. Allow 2 participants' TEE attestations to expire (naturally or by an adversary controlling those accounts and not renewing).
3. Any remaining participant calls `verify_tee()`.
4. `reverify_and_cleanup_participants` returns `TeeValidationResult::Partial` with 3 surviving participants.
5. `validate_governance_against_reconstruction(3, 3, Some(3))` passes (3 ≥ 3).
6. The DamgardEtAl check `2×3−1 = 5 ≤ 3` is never evaluated.
7. `transition_to_resharing_no_checks` is called; the contract enters `Resharing` with 3 participants.
8. No node can start or complete the DamgardEtAl resharing (requires 5 active participants). The contract is permanently stuck in `Resharing`. All `sign`, `respond`, and `request_app_private_key` calls are blocked. Funds controlled by the MPC network are frozen. [10](#0-9) [11](#0-10)

pub fn validate_domain_threshold(
    domain: &DomainConfig,
    num_participants: u64,
) -> Result<(), Error> {
    let t = domain.reconstruction_threshold.inner();
    if t < MIN_RECONSTRUCTION_THRESHOLD {
        return Err(DomainError::ReconstructionThresholdTooLow.into());
    }
    if t > num_participants {
        return Err(DomainError::ReconstructionThresholdExceedsParticipants {
            threshold: t,
            participants: num_participants,
        }
        .into());
    }
    if domain.protocol == Protocol::DamgardEtAl {
        let required = t
            .checked_mul(2)
            .and_then(|x| x.checked_sub(1))
            .ok_or(DomainError::ReconstructionThresholdOverflow { threshold: t })?;
        if required > num_participants {
            return Err(DomainError::InsufficientParticipantsForProtocol {
                protocol: domain.protocol,
                required,
                participants: num_participants,
            }
            .into());
        }
    }
    Ok(())

    pub fn validate_governance_against_reconstruction(
        num_participants: u64,
        governance: Threshold,
        max_reconstruction_threshold: Option<ReconstructionThreshold>,
    ) -> Result<(), Error> {
        Self::validate_threshold(num_participants, governance)?;
        if let Some(max_reconstruction_threshold) = max_reconstruction_threshold
            && governance.value() < max_reconstruction_threshold.inner()

            return Err(InvalidThreshold::BelowReconstructionThreshold {
                reconstruction_threshold: max_reconstruction_threshold.inner(),
                governance_threshold: governance.value(),
