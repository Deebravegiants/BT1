### Title
No Threshold-Voted Emergency Pause Mechanism Leaves Signing Operations Unstoppable During Active Exploits - (File: `crates/contract/src/lib.rs`)

### Summary
The NEAR MPC contract's only mechanism to halt signing operations (`accept_requests = false`) is exclusively controlled by the `verify_tee()` function, which is tied to TEE attestation state and cannot be triggered on-demand by participants. There is no threshold-voted emergency pause that participants can invoke to immediately stop `sign`, `request_app_private_key`, and `verify_foreign_transaction` operations during an active exploit.

### Finding Description
The `MpcContract` struct contains an `accept_requests: bool` field that gates all user-facing request methods and all node response methods. [1](#0-0) 

This flag is checked in `check_request_preconditions` (called by `sign`, `request_app_private_key`, `verify_foreign_transaction`) and directly in `respond`, `respond_ckd`, and `respond_verify_foreign_tx`: [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

The **only** code path that sets `accept_requests = false` is inside `verify_tee()`, and only when the TEE validation result shows fewer than threshold participants have valid attestations: [6](#0-5) 

`verify_tee()` itself is gated behind `voter_or_panic()`, which requires the caller to be a current participant: [7](#0-6) 

This means `accept_requests` can only be set to `false` as a side-effect of TEE attestation expiry — a passive, slow-moving condition (attestations expire over days/weeks per `tee_upgrade_deadline_duration_seconds`). There is no `vote_pause()` or equivalent function that a threshold of participants can call on-demand to immediately halt all signing operations.

Both `init` and `init_running` unconditionally set `accept_requests: true`: [8](#0-7) [9](#0-8) 

### Impact Explanation
If an exploit is discovered in any of the signing flows — for example, a bug in `verify_foreign_transaction` that allows an attacker to obtain MPC-signed payloads for arbitrary foreign-chain transactions, or a flaw in the ECDSA/EdDSA signing protocol — participants have no fast on-chain mechanism to halt the system. The only recourse is a contract upgrade via `vote_update`, which requires collecting threshold votes and executing the upgrade across multiple NEAR blocks. During that window, the exploit continues to be reachable by any unprivileged caller who can submit a `sign` or `verify_foreign_transaction` request with a 1 yoctoNEAR deposit.

This maps to: **Medium — contract execution-flow manipulation that breaks production safety invariants**, because the absence of an emergency halt means the contract's safety invariant ("participants can stop the system when a critical bug is found") is permanently unenforceable at the speed required to contain an active attack.

### Likelihood Explanation
The likelihood is realistic. The `verify_foreign_transaction` flow involves complex off-chain foreign-chain RPC verification by MPC nodes before signing; any discrepancy between what the contract accepts and what nodes verify creates a window. Similarly, the ECDSA/EdDSA signing paths are cryptographically complex. Historical precedent (Thorchain, referenced in the original report) shows that teams often detect attacks before funds are fully drained but cannot shut down fast enough. The 1 yoctoNEAR deposit barrier is negligible for a motivated attacker.

### Recommendation
Add a threshold-voted emergency pause mechanism:

```rust
// Participants vote to set accept_requests = false immediately
pub fn vote_emergency_pause(&mut self) -> Result<(), Error> {
    Self::assert_caller_is_signer();
    self.voter_or_panic();
    // collect votes; if threshold reached, set accept_requests = false
    ...
}

// Participants vote to resume
pub fn vote_resume(&mut self) -> Result<(), Error> {
    Self::assert_caller_is_signer();
    self.voter_or_panic();
    // collect votes; if threshold reached, set accept_requests = true
    ...
}
```

This mirrors the existing `vote_cancel_resharing` pattern and allows participants to halt the system within a single NEAR block once threshold votes are collected, without waiting for TEE attestations to expire or a full contract upgrade to be deployed.

### Proof of Concept

1. Assume an exploit is discovered in `verify_foreign_transaction` that allows an attacker to obtain valid MPC signatures for arbitrary Bitcoin transactions.
2. The attacker submits repeated `verify_foreign_transaction` calls with 1 yoctoNEAR deposit each.
3. Participants observe the attack and want to halt the system.
4. They cannot call `verify_tee()` to set `accept_requests = false` because all TEE attestations are currently valid — `verify_tee()` only sets the flag to `false` when attestations fail. [10](#0-9) 

5. The only recourse is `vote_update` for a contract upgrade, which requires collecting threshold votes across multiple transactions and blocks.
6. During this window — which could span many NEAR blocks — the attacker continues to drain funds via the exploit.
7. The `accept_requests` flag remains `true` throughout, and every `verify_foreign_transaction` call succeeds at the contract level. [11](#0-10)
