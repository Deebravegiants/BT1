### Title
Single `currentMerkleRoot` Variable Overwrites Previous Distribution Roots, Permanently Freezing Unclaimed KERNEL Yield - (File: `contracts/KERNEL/KernelMerkleDistributor.sol`)

---

### Summary

`KernelMerkleDistributor` stores only the most recent merkle root in a single `currentMerkleRoot` state variable. When `setMerkleRoot` is called, the previous root is permanently overwritten and lost. All claim proofs — regardless of which distribution round (`index`) they target — are verified exclusively against `currentMerkleRoot`. Users who hold valid proofs from a prior round but have not yet claimed will find their proofs permanently invalidated after a new root is set, freezing their unclaimed KERNEL tokens in the contract.

---

### Finding Description

`setMerkleRoot` unconditionally overwrites the single `currentMerkleRoot` variable and increments `currentIndex`: [1](#0-0) 

```solidity
function setMerkleRoot(bytes32 _merkleRootToSet) external onlyOwner {
    ...
    currentMerkleRoot = _merkleRootToSet;   // previous root is lost
    currentMerkleRootIndex++;
    currentIndex++;
    ...
}
```

The internal `_processClaim` function permits claiming for any `index` in the range `[1, currentIndex]`, meaning it accepts indices from all past distribution rounds. However, it verifies every proof against the single `currentMerkleRoot` — the **latest** root only: [2](#0-1) 

```solidity
if (index == 0 || index > currentIndex) {
    revert InvalidIndex();
}
// ...
bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));
if (!MerkleProofUpgradeable.verify(merkleProof, currentMerkleRoot, node)) {
    revert InvalidMerkleProof();   // always checked against latest root
}
```

There is no mapping from a distribution round index to its corresponding root. This is the direct analog of the `fbck_finalize` single-buffer pattern: `currentMerkleRoot` acts as a single slot that is overwritten by each successive call to `setMerkleRoot`, just as `fbck_finalize` was overwritten by each successive pool finalization. [3](#0-2) 

---

### Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

Any user who:
1. Has a valid claim from distribution round N (proof generated under root N), and
2. Has not claimed before round N+1's root is set, and
3. Is not included in round N+1's merkle tree (e.g., was only eligible for round N),

will have their proof permanently invalidated. Their KERNEL tokens remain locked in the contract with no recovery path, because:
- Their proof for index N fails verification against root N+1.
- They have no valid proof for index N+1 (they are not in that tree).
- The contract provides no fallback or root-history lookup.

---

### Likelihood Explanation

**Medium.** `setMerkleRoot` is a routine owner operation expected to be called for each new distribution epoch. It is realistic that:
- Some users from round N will not have claimed before round N+1 is published (e.g., due to inactivity, gas costs, or unawareness).
- Not all round-N recipients will be included in round N+1 (e.g., one-time airdrops, top-100 lists that change).

No attacker action is required; the freeze occurs as a side-effect of normal protocol operation.

---

### Recommendation

Replace the single `currentMerkleRoot` variable with a mapping from distribution index to its corresponding root:

```solidity
mapping(uint256 => bytes32) public merkleRoots;
```

In `setMerkleRoot`, store the root under the new index:
```solidity
currentIndex++;
merkleRoots[currentIndex] = _merkleRootToSet;
```

In `_processClaim`, verify the proof against the root for the specific index being claimed:
```solidity
if (!MerkleProofUpgradeable.verify(merkleProof, merkleRoots[index], node)) {
    revert InvalidMerkleProof();
}
```

This mirrors the recommendation in the original report: use a mapping to get the correct value for each item, rather than relying on a single shared buffer.

---

### Proof of Concept

1. Owner calls `setMerkleRoot(root1)` → `currentMerkleRoot = root1`, `currentIndex = 1`.
2. User A is included in round 1 with cumulative amount 100 KERNEL; they receive proof `P1` (valid under `root1`, for `index = 1`).
3. User A does not claim.
4. Owner calls `setMerkleRoot(root2)` → `currentMerkleRoot = root2`, `currentIndex = 2`. User A is **not** included in round 2.
5. User A calls `claim(index=1, account=A, cumulativeAmount=100, merkleProof=P1)`.
6. Inside `_processClaim`: `index (1) <= currentIndex (2)` passes the range check.
7. `MerkleProofUpgradeable.verify(P1, root2, node)` returns `false` because `P1` was generated under `root1`, not `root2`.
8. Transaction reverts with `InvalidMerkleProof`.
9. User A has no valid proof for `index = 2` (not in that tree). Their 100 KERNEL tokens are permanently frozen in the contract. [4](#0-3) [1](#0-0)
