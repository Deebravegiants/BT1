Audit Report

## Title
Unprivileged Caller Can Trigger Protocol-Wide Pause via Natural Oracle Price Dip — (`contracts/LRTOracle.sol`)

## Summary

`updateRSETHPrice()` is a permissionless `public` function with no role guard. Its internal `_updateRsETHPrice()` logic unconditionally pauses `LRTDepositPool`, `LRTWithdrawalManager`, and `LRTOracle` itself when `newRsETHPrice` drops more than `pricePercentageLimit` below `highestRsethPrice`. Any unprivileged address can call this at a moment of natural oracle price weakness, freezing all deposits and withdrawals until an admin manually unpauses each contract.

## Finding Description

`updateRSETHPrice()` carries no role check: [1](#0-0) 

Inside `_updateRsETHPrice()`, the upside-protection branch correctly reverts for non-managers when the price increase exceeds the threshold: [2](#0-1) 

The downside-protection branch, however, has no equivalent caller-identity check. It executes unconditionally for any caller: [3](#0-2) 

`highestRsethPrice` is a monotonically non-decreasing watermark — it only updates when `newRsETHPrice` exceeds it: [4](#0-3) 

This means the protocol is permanently exposed to the next dip below the all-time peak price. `_pause()` is idempotent but irreversible without admin action: [5](#0-4) 

Unpausing requires `onlyLRTAdmin`: [6](#0-5) 

The asymmetry is the root cause: the upside branch mirrors the correct pattern (revert for non-managers), but the downside branch omits it entirely, allowing any `address` to atomically pause all three contracts in a single transaction.

## Impact Explanation

**Medium — Temporary freezing of funds.**

All three pause calls execute atomically in one transaction from an unprivileged address. After the call: `LRTDepositPool.paused() == true` (no deposits), `LRTWithdrawalManager.paused() == true` (no withdrawals or claims), and `LRTOracle.paused == true` (price updates blocked). User funds are inaccessible until an admin manually calls `unpause()` on each contract. The freeze is temporary (admin can recover) but real and user-impacting, matching the "Temporary freezing of funds" Medium impact class.

## Likelihood Explanation

- `pricePercentageLimit` is admin-configurable (e.g., 1% = `1e16`). A 1% intraday dip in stETH or ETHx is routine market behavior.
- `highestRsethPrice` is the all-time peak, so any prior price high permanently lowers the threshold for triggering the pause.
- No oracle manipulation, MEV, or front-running is required — the attacker only needs to observe a natural price dip and call the public function at that block.
- The attacker pays only gas. The attack is repeatable after each admin unpause.

## Recommendation

Mirror the upside-branch pattern in the downside branch. Inside `_updateRsETHPrice()`, when `isPriceDecreaseOffLimit` is true, check whether `msg.sender` holds `MANAGER` or `PAUSER_ROLE` before executing the pause calls. For unprivileged callers, revert (e.g., `revert PriceBelowDailyThreshold()`) instead of pausing. Alternatively, remove the auto-pause side-effect from `_updateRsETHPrice()` entirely: emit an event on threshold breach and let an off-chain keeper with `PAUSER_ROLE` react via the existing `pause()` function at line 138.

## Proof of Concept

```
Preconditions:
  - pricePercentageLimit = 1e16 (1%)
  - Protocol deployed, rsETH supply > 0

Step 1: Call updateRSETHPrice() at normal price P0.
        → highestRsethPrice = P0

Step 2: Drop mock oracle price by 2% (below the 1% limit).
        → newRsETHPrice = P0 * 0.98

Step 3: Call updateRSETHPrice() as address(1) (unprivileged).
        → diff = P0 - P0*0.98 = P0*0.02
        → isPriceDecreaseOffLimit: 0.02*P0 > 0.01*P0 → true
        → lrtDepositPool.pause() executes
        → withdrawalManager.pause() executes
        → _pause() executes, LRTOracle.paused = true

Assertions:
  assertEq(lrtDepositPool.paused(), true);
  assertEq(withdrawalManager.paused(), true);
  assertEq(lrtOracle.paused(), true);

Step 4 (recovery): Only onlyLRTAdmin can call unpause() on each contract.
```

The call at Step 3 succeeds with no revert because `updateRSETHPrice()` has no role check and the downside branch at lines 277–281 executes unconditionally for any caller. [7](#0-6)
