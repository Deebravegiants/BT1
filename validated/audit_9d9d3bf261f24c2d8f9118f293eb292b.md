### Title
Deposit Refund Misdirection in Meta Transactions Allows Unprivileged User to Drain Relayer Funds - (File: runtime/runtime/src/lib.rs)

### Summary
In NEAR meta transactions (NEP-366), when a relayer funds an inner action with an attached deposit and that inner action fails on the receiver's shard, the deposit refund is sent to the `predecessor_id` of the failing receipt — which is the meta-transaction sender (Alice), not the relayer who actually paid. An unprivileged user can deliberately craft a `DelegateAction` with a large attached balance targeting a call that will fail, causing the relayer to permanently lose the attached funds to the user.

### Finding Description
When a meta transaction is processed, the relayer's account is charged for all inner action deposits at the time the transaction is converted to a receipt. The `DelegateAction` is unwrapped on Alice's shard, and a new action receipt is forwarded to Bob's shard with `predecessor_id = Alice`. If the inner action fails on Bob's shard, the runtime generates a deposit refund receipt via `refund_unspent_gas_and_deposits`:

```rust
if deposit_refund > Balance::ZERO {
    result.new_receipts.push(Receipt::new_balance_refund(
        receipt.balance_refund_receiver(),
        deposit_refund,
    ));
}
``` [1](#0-0) 

The refund destination is determined by `balance_refund_receiver()`:

```rust
pub fn balance_refund_receiver(&self) -> &AccountId {
    self.refund_to().as_ref().unwrap_or_else(|| self.predecessor_id())
}
``` [2](#0-1) 

For the inner action receipt on Bob's shard, `predecessor_id` is Alice — not the relayer. The relayer paid the deposit but receives no refund. Alice receives the full deposit refund.

This behavior is acknowledged in the protocol documentation:

> "But what if the transaction fails execution on Bob's shard? At this point, the predecessor is `Alice` and therefore she receives the token balance refunded, not the relayer. This is something relayer implementations must be aware of since there is a financial incentive for Alice to submit meta transactions that have high balances attached but will fail on Bob's shard." [3](#0-2) 

The protocol provides no mechanism for the relayer to redirect the deposit refund back to itself. The `refund_to` field (via `promise_set_refund_to`) can only be set by the predecessor of a receipt — in this case Alice, not the relayer.

### Impact Explanation
The corrupted protocol values are concrete account balances stored in the state trie:
- **Relayer's account balance**: permanently decreased by the full deposit amount (e.g., 100 NEAR)
- **Alice's account balance**: incorrectly increased by the same deposit amount

This is a direct, irreversible loss of funds for the relayer. The relayer cannot recover the funds because the refund receipt has already been executed and Alice's balance has been credited. The deposit amount is bounded only by what the relayer is willing to attach, which can be arbitrarily large.

### Likelihood Explanation
The attack is straightforward for any NEAR user:
1. Alice crafts a `DelegateAction` containing a `Transfer` of N NEAR to a non-existent account (or a `FunctionCall` with a deposit to a contract that will revert)
2. Alice signs the `DelegateAction` and submits it to a relayer
3. The relayer, after checking the action looks valid, wraps it in a transaction and submits it — paying N NEAR from its own account
4. The inner action fails on the receiver's shard (account does not exist, or contract reverts)
5. The N NEAR deposit refund is sent to Alice

The relayer cannot fully prevent this: even if it checks Alice's intended receiver exists at submission time, the receiver's account could be deleted between the relayer's check and the transaction's execution (TOCTOU). The relayer also cannot verify that a contract call will succeed without executing it.

### Recommendation
The protocol should track the original deposit payer (the relayer/transaction signer) and route deposit refunds to that account rather than to `predecessor_id` when the receipt was generated from a `DelegateAction`. Alternatively, the `DelegateAction` processing should set `refund_to` on the inner action receipt to point to the relayer's account, so that `balance_refund_receiver()` returns the relayer rather than Alice. [4](#0-3) 

### Proof of Concept
1. Alice has a NEAR account with no tokens. She creates a `DelegateAction` containing `Transfer { deposit: 100 NEAR }` to `nonexistent-account.near`.
2. Alice signs the `DelegateAction` and sends it to a relayer.
3. The relayer verifies the action looks reasonable, wraps it in a `SignedTransaction` (relayer is the signer), and submits it. The relayer's account is charged 100 NEAR + gas.
4. On Alice's shard, the `DelegateAction` is unwrapped. A new action receipt is created with `predecessor_id = Alice`, `receiver_id = nonexistent-account.near`, and `deposit = 100 NEAR`.
5. On the receiver's shard, the receipt fails: `ActionErrorKind::AccountDoesNotExist`.
6. `refund_unspent_gas_and_deposits` is called. `deposit_refund = 100 NEAR`. `receipt.balance_refund_receiver()` returns `Alice` (the `predecessor_id`).
7. A `new_balance_refund` receipt is created sending 100 NEAR to Alice.
8. Alice's balance increases by 100 NEAR. The relayer has lost 100 NEAR permanently. [5](#0-4) [6](#0-5)

    pub fn refund_to(&self) -> &Option<AccountId> {
        match self.receipt() {
            ReceiptEnum::Action(_)
            | ReceiptEnum::Data(_)
            | ReceiptEnum::PromiseYield(_)
            | ReceiptEnum::PromiseResume(_)
            | ReceiptEnum::GlobalContractDistribution(_) => &None,
            ReceiptEnum::ActionV2(action_receipt_v2)
            | ReceiptEnum::PromiseYieldV2(action_receipt_v2) => &action_receipt_v2.refund_to,
        }
    }
