Looking at the `MerkleDistributor.sol` `initialize` function and `claim` function for the unset-address analog.

### Title
Unset `token` Address Causes `claim` to Permanently Revert for All Users - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

### Summary
`MerkleDistributor.initialize` explicitly permits `token_` to be `address(0)`. Once a merkle root is set, any user who calls `claim` will receive a revert from `SafeERC20.safeTransfer` because the call target is the zero address, permanently blocking all reward claims until an admin intervenes.

### Finding Description
The `initialize` function in `MerkleDistributor` intentionally skips a zero-address check on `token_`, as documented by the inline comment:

> "token can be set later but not the protocol treasury" [1](#0-0) 

`token` is stored as-is, potentially as `address(0)`: [2](#0-1) 

The `claim` function then unconditionally calls `IERC20(token).safeTransfer(...)` on both the user transfer and the fee transfer, with no guard checking whether `token` has been set: [3](#0-2) 

`SafeERC20.safeTransfer` internally calls `Address.functionCall`, which checks `isContract(token)`. Since `address(0)` has no code, this check fails and the call reverts with `"Address: call to non-contract"`. Every `claim` call fails until the owner calls `setToken`.

This is structurally identical to the reference bug: an address that is intentionally left unset at initialization is used unconditionally in the claim execution path, causing a full revert for all claimants.

By contrast, `KernelMerkleDistributor` and `KernelTop100MerkleDistributor` both enforce a non-zero check on their token address (`_kernel`) during initialization via `UtilLib.checkNonZeroAddress`, so they are not affected. [4](#0-3) 

### Impact Explanation
**Medium — Permanent freezing of unclaimed yield.**

If the contract is deployed with `token_ = address(0)` and a merkle root is published before `setToken` is called, every eligible user's `claim` call reverts. Their allocated token rewards are locked in the contract and inaccessible. The freeze persists until the owner calls `setToken`; if the owner never does (e.g., key loss, operational error), the freeze is permanent.

### Likelihood Explanation
The design explicitly documents and supports deploying with `token_ = address(0)`. This creates a deployment window during which the merkle root may be set (by the owner) and users may attempt to claim, all while `token` remains zero. The likelihood is realistic: the contract is designed for this two-step initialization, and any claim attempt in that window fails.

### Recommendation
Add a zero-address check for `token_` in `initialize`, mirroring the existing check for `_protocolTreasury`:

```solidity
function initialize(address token_, address _protocolTreasury, uint256 _feeInBPS) external initializer {
    if (token_ == address(0)) revert ZeroValueProvided();          // ADD THIS
    if (_protocolTreasury == address(0)) revert ZeroValueProvided();
    ...
}
```

Alternatively, add a guard in `claim` before the transfers:

```solidity
if (token == address(0)) revert ZeroValueProvided();
```

### Proof of Concept
1. Deploy `MerkleDistributor` proxy and call `initialize(address(0), treasury, 100)` — succeeds because no zero-check on `token_`.
2. Owner calls `setMerkleRoot(validRoot)` — succeeds.
3. User calls `claim(1, user, amount, proof)` with a valid proof.
4. Execution reaches line 141: `IERC20(address(0)).safeTransfer(user, amountToSend)`.
5. `SafeERC20` → `Address.functionCall(address(0), ...)` → `isContract(address(0))` returns `false` → reverts.
6. All users are blocked from claiming their allocated tokens. [5](#0-4) [6](#0-5)
