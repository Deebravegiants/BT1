### Title
Fee-on-Transfer Token Accounting Divergence in `initTransfer` Causes Bridge Undercollateralization — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
`OmniBridge.initTransfer` uses the caller-supplied `amount` for cross-chain accounting after calling `safeTransferFrom`, without verifying the actual amount received. For fee-on-transfer ERC-20 tokens, the bridge receives `amount − fee` but emits and forwards `amount` to the destination chain, permanently undercollateralizing the EVM vault.

### Finding Description
In `OmniBridge.initTransfer`, when the token is neither a bridge token nor a custom-minter token, the contract executes:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount
);
``` [1](#0-0) 

Immediately after, the unverified `amount` is forwarded verbatim to both `initTransferExtension` and the `InitTransfer` event:

```solidity
initTransferExtension(
    msg.sender, tokenAddress, currentOriginNonce,
    amount,   // ← user-supplied, not balance-checked
    fee, nativeFee, recipient, message, extensionValue
);
emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress, currentOriginNonce,
    amount,   // ← same unverified value
    fee, nativeFee, recipient, message
);
``` [2](#0-1) 

`OmniBridgeWormhole.initTransferExtension` then publishes a Wormhole message encoding this `amount`: [3](#0-2) 

The NEAR bridge processes the proof and mints/releases the full `amount` to the recipient: [4](#0-3) 

No balance-before/balance-after check is performed anywhere in this path. The actual amount held by the EVM vault is `amount − transferFee`, but the cross-chain record claims `amount`.

### Impact Explanation
Each bridging call with a fee-on-transfer token silently widens the gap between the EVM vault's real balance and the outstanding cross-chain obligations. When the accumulated deficit exceeds the vault's reserves, `finTransfer` calls that attempt `safeTransfer(recipient, amount)` will revert because the vault is short. Users who legitimately bridged tokens to NEAR can never redeem them on EVM — their funds are permanently locked. This satisfies both:

- **High**: Balance-accounting divergence that breaks backing guarantees.
- **Critical** (at scale): Irreversible fund lock for users whose redemption path is frozen.

### Likelihood Explanation
The precondition is that a fee-on-transfer ERC-20 is registered with the bridge (i.e., has a NEAR token mapping). This is not an exotic scenario: USDT has a dormant fee switch that its issuer can enable; other tokens (e.g., STA, PAXG) charge transfer fees by design. An admin registering such a token in good faith is sufficient. Once registered, any unprivileged user calling `initTransfer` triggers the accounting divergence — no special role or key is required.

### Recommendation
Measure the actual received amount using a balance snapshot:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(actualReceived == amount, "fee-on-transfer token not supported");
```

Alternatively, use `actualReceived` in place of `amount` for all downstream accounting (event emission and `initTransferExtension`), and document that fee-on-transfer tokens are unsupported if the stricter check is preferred.

### Proof of Concept
1. Admin registers a fee-on-transfer token `FTT` (1% fee) with the bridge.
2. Attacker calls `initTransfer(FTT, 1000, 0, 0, nearRecipient, "")`.
3. `safeTransferFrom` moves 1000 FTT from attacker; bridge receives 990 FTT (10 taken as fee).
4. `InitTransfer` event and Wormhole message record `amount = 1000`.
5. NEAR relayer processes the proof; NEAR bridge mints 1000 FTT-wrapped tokens to `nearRecipient`.
6. EVM vault now holds 990 FTT but owes 1000 FTT.
7. Repeated calls accumulate the deficit.
8. A legitimate user attempts `finTransfer` to redeem 1000 FTT on EVM; `safeTransfer` reverts — funds are permanently locked.
