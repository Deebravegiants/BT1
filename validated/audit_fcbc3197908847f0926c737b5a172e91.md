### Title
Pool admin can front-run swaps with maximum fees via instant `setPoolAdminFees` — no timelock enforced — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

`MetricOmmPoolFactory.setPoolAdminFees` allows the pool admin to raise `adminSpreadFeeE6` and `adminNotionalFeeE8` to their configured caps instantly, with no timelock or delay. A malicious pool admin can front-run pending swap transactions, spike fees to the maximum, collect the inflated fee revenue, then immediately reset fees to attract the next victim.

---

### Finding Description

`setPoolAdminFees` is callable by the pool admin at any time with no delay: [1](#0-0) 

The only guards are cap checks against `maxAdminSpreadFeeE6` and `maxAdminNotionalFeeE8`, which default to the hard ceiling `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%) and `HARD_MAX_NOTIONAL_FEE_E8 = 1_000_000` (1%): [2](#0-1) 

After the cap check, the function immediately calls `setPoolFees` on the pool with the new totals: [3](#0-2) 

The pool's `spreadFeeE6` and `notionalFeeE8` are live state variables read during every swap: [4](#0-3) 

The swap function reads these values at execution time, not at the time the user submitted the transaction: [5](#0-4) 

The `priceLimitX64` slippage guard in `swap` constrains the execution price but does **not** constrain the fee deducted from the swap output. A fee increase from 0% to 20% reduces the user's received amount without triggering the price limit revert.

By contrast, the price-provider rotation path — a less impactful admin action — correctly enforces a timelock: [6](#0-5) 

No equivalent delay exists for fee changes.

Similarly, `setPoolBinAdditionalFees` has no cap validation at all at the factory level — it passes `uint16` values directly to the pool with zero bounds checking: [7](#0-6) 

---

### Impact Explanation

A malicious pool admin can extract value from every swap:

1. Advertise low fees (e.g., 0.1% admin spread) to attract volume.
2. Observe a pending swap in the mempool.
3. Front-run it with `setPoolAdminFees(pool, 200_000, 1_000_000)` — raising spread fee to 20% and notional fee to 1%.
4. The victim's swap executes and pays the inflated fee; the admin receives the excess via `collectPoolFees`.
5. Call `setPoolAdminFees(pool, low, low)` to reset and repeat.

Users have no on-chain mechanism to protect against this: `priceLimitX64` does not bound fees, and there is no "max fee" parameter in the `swap` calldata.

---

### Likelihood Explanation

The pool admin is a semi-trusted role set at pool creation. The attack requires only two transactions (fee spike + reset) and is repeatable on every swap. On chains with a public mempool the admin can reliably front-run. The admin cap of 20% spread fee is large enough to cause material loss on any non-trivial swap.

---

### Recommendation

Introduce a timelock on `setPoolAdminFees` (and `setPoolBinAdditionalFees`) analogous to the existing `priceProviderTimelock` mechanism. A two-step propose/execute pattern with a minimum delay (e.g., 24–48 hours) would give users time to observe the pending change and avoid the pool before it takes effect. Alternatively, add a `maxFeeE6` parameter to `swap` that reverts if the effective fee exceeds the caller's tolerance.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Assume standard test harness with pool, factory, admin, alice, token0/1 set up.

function test_poolAdmin_frontRuns_swap_with_higher_fees() public {
    // Pool is live with low fees: adminSpreadFeeE6 = 1_000 (0.1%)
    factory.setPoolAdminFees(address(pool), 1_000, 0);

    // Alice approves and prepares a swap (token0 -> token1)
    uint256 aliceToken1Before = token1.balanceOf(alice);

    // Admin sees Alice's pending swap and front-runs: spike fees to 20%
    vm.prank(admin);
    factory.setPoolAdminFees(address(pool), 200_000, 0); // 20% spread fee

    // Alice's swap executes at the inflated fee
    vm.prank(alice);
    pool.swap(alice, true, 1_000e18, 0, "", "");

    uint256 aliceToken1After = token1.balanceOf(alice);

    // Admin resets fees to attract next user
    vm.prank(admin);
    factory.setPoolAdminFees(address(pool), 1_000, 0);

    // Alice received significantly less than she would have at 0.1% fee
    // The difference is captured as admin fee revenue
    assertLt(aliceToken1After - aliceToken1Before, expectedOutputAtLowFee);
}
```
