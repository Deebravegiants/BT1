### Title
Stale Guardrails Script Hash Allows In-Flight Proposals to Bypass Updated Constitutional Constraints at Ratification - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs`)

---

### Summary

The Conway governance system validates the guardrails script hash of `TreasuryWithdrawals` and `ParameterChange` proposals only at submission time (in the `GOV` rule). The `RATIFY` and `ENACT` rules never re-validate this hash against the current constitution. If a `NewConstitution` action is enacted while such a proposal is in-flight, the proposal will be ratified and enacted under the new constitution's guardrails without ever being checked against them, bypassing the updated constitutional constraints.

---

### Finding Description

**Submission-time check (GOV rule):**

In `conwayGovTransition`, when a `TreasuryWithdrawals` or `ParameterChange` proposal is submitted, the `checkGuardrailsScriptHash` function verifies that the `proposalPolicy` field embedded in the action matches the current constitution's `constitutionGuardrailsScriptHash`: [1](#0-0) 

This check is performed once, at proposal submission. The guardrails Plutus script is also executed at this point as a transaction witness.

**Ratification-time checks (RATIFY rule):**

In `ratifyTransition`, the full set of ratification predicates is: [2](#0-1) 

The checks are `prevActionAsExpected`, `validCommitteeTerm`, `not rsDelayed`, `withdrawalCanWithdraw`, and `acceptedByEveryone`. There is no check that the guardrails script hash embedded in the proposal still matches the current constitution's guardrails script hash (`ensConstitution` in the `EnactState`).

**Enactment (ENACT rule):**

The `enactmentTransition` applies `TreasuryWithdrawals` and `ParameterChange` actions directly with no guardrails validation: [3](#0-2) 

**The snapshot mechanism:**

At each epoch boundary, `setFreshDRepPulsingState` snapshots the current `EnactState` (including the current constitution) into the `DRepPulser`: [4](#0-3) 

When a `NewConstitution` action is enacted in epoch N, the new constitution (with its new guardrails script hash) becomes part of the `EnactState`. In epoch N+1's ratification run, the `EnactState` snapshot will contain the new constitution. However, the `ratifyTransition` never compares the `proposalPolicy` field of in-flight `TreasuryWithdrawals`/`ParameterChange` proposals against `ensConstitution ^. constitutionGuardrailsScriptHashL`.

The `Constitution` type stores the guardrails script hash: [5](#0-4) 

And the `EnactState` carries the current constitution: [6](#0-5) 

The `NewConstitution` enactment updates `ensConstitution` in the `EnactState`: [7](#0-6) 

But no code in `ratifyTransition` or `enactmentTransition` reads `ensConstitution` to re-validate in-flight proposals.

---

### Impact Explanation

The guardrails script is the on-chain enforcement mechanism for constitutional constraints on protocol parameter changes and treasury withdrawals. If the community enacts a new constitution with more restrictive guardrails (e.g., lowering the maximum treasury withdrawal, or restricting which protocol parameters can be changed), any in-flight `TreasuryWithdrawals` or `ParameterChange` proposal that was valid under the old guardrails will still be ratified and enacted under the new constitution without the new guardrails ever being applied to it.

This constitutes an **unauthorized governance action being enacted**: a treasury withdrawal or protocol parameter change that the current constitution's guardrails would prohibit is enacted anyway, because the guardrails check is only performed at proposal submission time and is never repeated at ratification or enactment.

**Impact class:** Critical — Unauthorized governance, treasury, or protocol-parameter action is enacted.

---

### Likelihood Explanation

The scenario requires:
1. A `TreasuryWithdrawals` or `ParameterChange` proposal is submitted and accumulates sufficient votes.
2. A `NewConstitution` proposal is enacted (changing the guardrails script hash) before the first proposal is ratified.
3. The first proposal is ratified in the subsequent epoch under the new constitution.

This is a realistic governance sequence. `NewConstitution` is a delaying action, so the in-flight proposal is deferred by one epoch after the constitution change, but it is not invalidated. The attacker can deliberately time their proposal submission to precede a known constitution change, or the scenario can arise organically when the community updates the constitution while proposals are pending.

The attacker only needs to pay the governance action deposit and obtain sufficient DRep/CC/SPO votes — no privileged access is required.

---

### Recommendation

In `ratifyTransition`, before ratifying a `TreasuryWithdrawals` or `ParameterChange` action, re-validate that the `proposalPolicy` field embedded in the action still matches the current constitution's guardrails script hash from the `EnactState`:

```haskell
-- In ratifyTransition, add to the ratification predicate:
&& guardrailsHashStillValid govAction (rsEnactState ^. ensConstitutionL)
```

where:

```haskell
guardrailsHashStillValid :: GovAction era -> Constitution era -> Bool
guardrailsHashStillValid (TreasuryWithdrawals _ proposalPolicy) constitution =
  proposalPolicy == constitutionGuardrailsScriptHash constitution
guardrailsHashStillValid (ParameterChange _ _ proposalPolicy) constitution =
  proposalPolicy == constitutionGuardrailsScriptHash constitution
guardrailsHashStillValid _ _ = True
```

This mirrors the existing `checkGuardrailsScriptHash` logic in the `GOV` rule and ensures that proposals cannot be enacted under a constitution whose guardrails they were never validated against.

### Proof of Concept

1. Constitution C1 is active with guardrails script hash `H1`. Script `H1` permits treasury withdrawals up to 1,000,000 ADA.
2. Attacker submits `TreasuryWithdrawals` for 800,000 ADA with `proposalPolicy = SJust H1`. The GOV rule accepts it (hash matches, script executes successfully). Votes accumulate.
3. Community enacts `NewConstitution` with guardrails script hash `H2`. Script `H2` restricts withdrawals to 100,000 ADA. The `EnactState` now has `ensConstitution` pointing to C2 with hash `H2`.
4. In the next epoch's ratification run, the `DRepPulser` snapshots the new `EnactState` (with C2/H2). The attacker's `TreasuryWithdrawals` proposal has sufficient votes. `ratifyTransition` checks `prevActionAsExpected` (passes — `TreasuryWithdrawals` has no ancestry requirement), `withdrawalCanWithdraw` (passes — treasury has funds), `acceptedByEveryone` (passes — votes are sufficient). No check compares `SJust H1` against `H2`.
5. The `ENACT` rule applies the withdrawal: 800,000 ADA is transferred from the treasury, violating C2's 100,000 ADA limit.

The 800,000 ADA treasury withdrawal is enacted under a constitution whose guardrails explicitly prohibit it.
