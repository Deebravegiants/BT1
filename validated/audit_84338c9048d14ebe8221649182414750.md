### Title
Missing `payload_hash` Binding to Pending Request in `respond_verify_foreign_tx` — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is cryptographically valid over the caller-supplied `response.payload_hash`, but never checks that `response.payload_hash` is the hash that corresponds to the pending `VerifyForeignTransactionRequest`. A single malicious attested participant (strictly below the signing threshold) can replay a legitimately-produced MPC signature from one foreign-chain verification request as the response to a different pending request, delivering a fraudulent `(payload_hash, signature)` pair to the waiting caller.

---

### Finding Description

In `respond` (the regular signing endpoint), the payload hash used for signature verification is read from the **stored request** — it is never accepted from the caller:

```rust
// crates/contract/src/lib.rs  L600
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,          // ← comes from the stored request, not the response
    &expected_public_key,
)
```

In `respond_verify_foreign_tx`, the payload hash is taken **from the caller-supplied response** and is never cross-checked against the pending request:

```rust
// crates/contract/src/lib.rs  L726-733
let payload_hash: [u8; 32] = response.payload_hash.0;   // ← caller-supplied

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,          // ← no check that this matches the pending request
    &secp_pk,
)
.is_ok()
```

After the signature check passes, `pending_requests::resolve_yields_for` is called with the `request` key to drain the yield queue and deliver `serde_json::to_vec(&response)` — which contains the attacker-chosen `payload_hash` — to every waiting caller.

The contract has no mechanism to recompute the expected hash from the stored request (because the extracted values are only known to the MPC nodes), so the check is simply absent.

---

### Impact Explanation

A single malicious attested participant can:

1. Observe that the MPC network has produced a valid threshold signature `sig_Y` over `hash_Y = SHA-256(borsh(ForeignTxSignPayloadV1 { request_Y, extracted_values_Y }))` for some request Y (e.g., a fraudulent or attacker-controlled foreign-chain transaction).
2. Identify a different pending request X in `pending_verify_foreign_tx_requests` (the map is readable on-chain).
3. Call `respond_verify_foreign_tx(request_X, { payload_hash: hash_Y, signature: sig_Y })`.
4. The contract accepts the call: the signature is valid over `hash_Y`, and request X exists in the map.
5. Every caller waiting on request X receives `(hash_Y, sig_Y)` — a valid MPC signature over a hash that encodes a completely different foreign-chain state.

Any bridge or application contract that does not independently recompute and compare the expected `payload_hash` will treat this as a legitimate attestation of the foreign-chain event they requested, enabling forged foreign-chain verification and potential double-spend or invalid bridge execution.

This is exploitable by a **single** attested participant, strictly below the signing threshold.

---

### Likelihood Explanation

- The attacker only needs to be one attested MPC participant — no threshold collusion is required.
- The pending request map is public on-chain, so the attacker can trivially identify target requests.
- The attacker only needs to wait for any legitimate `respond_verify_foreign_tx` call to complete for a different request to obtain a reusable `(hash, signature)` pair.
- Bridge contracts that rely on the MPC network's attestation without re-verifying the payload hash (a reasonable assumption for contracts that trust the MPC network) are directly vulnerable.

---

### Recommendation

Bind the signature verification to the pending request, not to the caller-supplied hash. Because the contract cannot recompute the full `ForeignTxSignPayload` (it does not store the extracted values), the fix should store the expected `payload_hash` alongside the pending request at enqueue time — computed by the node and committed on-chain — and then verify `response.payload_hash == stored_payload_hash` before accepting the response.

Alternatively, mirror the design of `respond`: derive the message to verify from the stored state rather than from the response.

---

### Proof of Concept

1. Alice submits `verify_foreign_transaction` for Bitcoin tx X → `request_X` is stored in `pending_verify_foreign_tx_requests`.
2. Attacker (attested participant) submits `verify_foreign_transaction` for a fraudulent Bitcoin tx Y → `request_Y` is stored.
3. The MPC network honestly processes request Y and produces `sig_Y` over `hash_Y`. The attacker calls `respond_verify_foreign_tx(request_Y, { payload_hash: hash_Y, sig_Y })` to obtain a valid on-chain signature.
4. Attacker calls `respond_verify_foreign_tx(request_X, { payload_hash: hash_Y, signature: sig_Y })`.
5. Contract checks: (a) caller is attested ✓, (b) `sig_Y` is valid over `hash_Y` under the root public key ✓, (c) `request_X` exists in the map ✓ → **accepted**.
6. Alice's waiting promise is resumed with `{ payload_hash: hash_Y, signature: sig_Y }`. Alice's bridge contract, trusting the MPC network, sees a valid signature and releases funds for a transaction that was never confirmed.

The root cause is directly analogous to the reported AUIPC issue: a value (`payload_hash`) that participates in a security-critical check is accepted from the caller without being constrained to the value that the protocol actually computed. [1](#0-0) [2](#0-1) [3](#0-2)

```rust
    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
```

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
```
