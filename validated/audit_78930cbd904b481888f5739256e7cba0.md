## Verdict

### Title
Non-Canonical y-Coordinate Acceptance Enables Ed25519 Point Encoding Malleability — (`src/cbmpc/crypto/ec25519_core.cpp`)

### Summary
`from_bin` accepts the non-canonical encoding y = p (= 2^255−19) without rejecting it, violating RFC 8032 §5.1.3 which requires y < p. Because `fe_t::to_fe` stores the raw value unreduced and all field arithmetic reduces p to 0, the encoding y=p decodes to the same point as y=0. Two distinct 32-byte blobs thus map to the same point, creating encoding malleability.

### Finding Description
In `from_bin` (line 872), after reversing and masking the sign bit, the y coordinate is loaded via `fe_t::to_fe(uint256_t::from_bin(buf))` with no bounds check. When the input encodes y = p = 0x7fffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffed (little-endian: `[0xed, 0xff, …, 0xff, 0x7f]`), `fe_t::to_fe` stores p unreduced in `fe_t::d`. All subsequent field arithmetic (mul, sub, add) reduces p to 0 internally, so the x-coordinate computation is identical to the y=0 case. The function returns `SUCCESS` and stores `R.y = y` with the unreduced value p. The resulting point is (sqrtm1, 0) — a valid order-4 torsion point on the curve. When re-serialized, `to_bin` calls `y.from_fe()` which invokes `fe_freeze`, reducing p to 0, so the point re-encodes as y=0. The missing guard is a single range check: `if (parsed_y >= p) return error(E_FORMAT)`.

### Impact Explanation
Two distinct 32-byte blobs — `[0x00, …, 0x00]` (y=0) and `[0xed, 0xff, …, 0xff, 0x7f]` (y=p) — are both accepted by `from_bin` and decode to the same point. A Byzantine participant below threshold can send y=p to one honest party and y=0 to another. If either party uses the raw received bytes as a transcript or commitment input (e.g., in a hash-to-challenge step), the two honest parties compute different transcripts and diverge. This satisfies the Medium scope: public-API-reachable parsing invariant break causing honest-party divergence or unsafe state acceptance.

### Likelihood Explanation
The attacker only needs to craft a single 32-byte blob with y = p. No cryptographic knowledge is required. The encoding is deterministic and the affected value is unique (only y=p, not a range). Any peer-facing deserialization path that calls `from_bin` is reachable.

### Recommendation
Add a canonical-form check immediately after parsing y in `from_bin`:
```cpp
uint256_t y_val = uint256_t::from_bin(buf);
// RFC 8032 §5.1.3: reject non-canonical y >= p
uint256_t y_frozen = y_val;
fe_freeze(y_frozen);
if (y_frozen != y_val) return coinbase::error(E_FORMAT);
fe_t y = fe_t::to_fe(y_val);
```
This rejects any y in [p, 2^255) before any arithmetic is performed.

static error_t from_bin(point_t& R, mem_t bin) {
  if (bin.size != 32) return coinbase::error(E_FORMAT);

  buf_t buf = bin.rev();
  uint8_t neg = buf[0] >> 7;
  buf[0] &= 0x7f;
  fe_t y = fe_t::to_fe(uint256_t::from_bin(buf));

  // x² = (y² - 1) / (dy² + 1)

  fe_t u, v, w, vxx, check;

  u = y * y;
  v = u * formula_t::get_d();
  u -= fe_t::one();       // u = y^2-1
  v += fe_t::one();       // v = dy^2+1
  w = u * v;              // w = u*v
  fe_t x = w.pow22523();  // x = w^((q-5)/8)
  x *= u;                 // x = u * w^((q-5)/8)

  vxx = x * x;
  vxx *= v;
  check = vxx - u;  // vx^2-u
  if (!check.is_zero()) {
    check = vxx + u;  // vx^2+u
    if (!check.is_zero()) {
      return coinbase::error(E_CRYPTO);
    }
    static const fe_t sqrtm1 =
        fe_t::from_bn(bn_t::from_hex("2b8324804fc1df0b2b4d00993dfbd7a72f431806ad2fe478c4ee1b274a0ea0b0"));
    x *= sqrtm1;
  }

  uint256_t x_val = x.from_fe();
  if (neg != (x_val.w0 & 1)) x = -x;

  R.x = x;
  R.y = y;
  R.z = fe_t::one();
  return 0;
}
