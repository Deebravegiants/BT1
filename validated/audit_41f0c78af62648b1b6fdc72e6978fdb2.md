### Title
Caller-Controlled Finality Threshold Allows MPC Signing of Unfinalized Foreign-Chain Transactions - (File: `crates/contract/src/lib.rs`, `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

The `verify_foreign_transaction` endpoint accepts a caller-supplied `confirmations` (Bitcoin) or `finality` (EVM) parameter with **no minimum enforced at the contract level**. An unprivileged caller can set `confirmations: 0` for Bitcoin or `finality: Latest` for EVM chains, causing the MPC network to produce a valid threshold signature over a payload that attests to a transaction that has not reached any meaningful finality. A bridge contract (e.g., Omnibridge) consuming this signature without re-validating the embedded finality parameter is exposed to double-spend.

---

### Finding Description

The `verify_foreign_transaction` function in `crates/contract/src/lib.rs` accepts a `VerifyForeignTransactionRequestArgs` that includes a chain-specific request. For Bitcoin, this is a `BitcoinRpcRequest` containing a `confirmations: BlockConfirmations` field. `BlockConfirmations` is defined as a plain `u64` newtype with no minimum value constraint:

```rust
pub struct BlockConfirmations(pub u64);
```

The contract's `verify_foreign_transaction` method performs no validation on the `confirmations` value before enqueuing the request for MPC nodes to process:

```rust
pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
    // ... only checks: domain purpose, chain whitelist, gas, deposit
    // NO check on request.request.confirmations minimum value
    self.enqueue_yield_request(...);
}
```

On the node side, `BitcoinInspector::extract()` enforces the threshold as supplied by the caller:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;

if !enough_block_confirmations {
    return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations { ... });
}
```

When `block_confirmations_threshold` is `0`, the condition `0 <= N` is trivially true for any `N ≥ 0`, meaning a transaction with even a single confirmation (or zero, if the RPC returns it) passes the check. The MPC nodes then proceed to sign the payload.

The signed payload (`ForeignTxSignPayloadV1`) does embed the `confirmations` value as part of the request, but the MPC contract itself never enforces a minimum — it delegates that responsibility entirely to the caller. This is the direct analog to the H-01 pattern: just as `OUSGInstantManager._getMintAmount` assumed 1 USDC = 1 USD without validating the actual USDC price, the MPC contract assumes the caller will supply a meaningful finality threshold without enforcing one.

The same class of issue applies to EVM chains: `EvmFinality::Latest` is a valid, accepted value that maps to the weakest possible finality guarantee (`eth_getBlockByNumber("latest", ...)`), and the contract imposes no minimum finality level.

---

### Impact Explanation

**High — Invalid bridge execution / double-spend conditions.**

The primary use case of `verify_foreign_transaction` is the Omnibridge inbound flow: a bridge contract releases NEAR-side assets only after the MPC network attests that a foreign-chain deposit transaction has finalized. If an attacker can obtain a valid MPC threshold signature over a payload with `confirmations: 0` (Bitcoin) or `finality: Latest` (EVM):

1. Attacker broadcasts a Bitcoin deposit transaction.
2. After 1 confirmation (~10 min), attacker calls `verify_foreign_transaction` with `confirmations: 0`.
3. MPC nodes satisfy `0 <= 1`, sign the payload, and return a valid threshold signature.
4. Attacker submits the signature to the bridge contract to claim NEAR-side assets.
5. Attacker double-spends the Bitcoin transaction via a reorg (feasible within 1–2 confirmations).
6. Bridge has disbursed NEAR assets against a Bitcoin transaction that no longer exists on the canonical chain.

For EVM chains, the same attack applies with `finality: Latest` — a transaction in the latest block can be reorganized before it reaches `Safe` or `Finalized` status.

The MPC signature is cryptographically valid and covers the full payload including the weak finality parameter. A bridge contract that trusts the MPC signature without re-reading and enforcing the embedded `confirmations`/`finality` field is directly exploitable.

---

### Likelihood Explanation

**High.** The attack requires no privileged access, no collusion, and no special capability beyond the ability to call `verify_foreign_transaction` (which is open to any NEAR account with 1 yoctoNEAR deposit). The attacker controls the `confirmations` field directly in the request arguments. The condition `0 <= 1` is deterministically satisfied. The only prerequisite is that the target bridge contract does not independently re-validate the finality parameter embedded in the signed payload — a reasonable assumption given that the MPC contract is positioned as the security primitive that enforces finality.

---

### Recommendation

Enforce a protocol-level minimum finality threshold at the contract layer, before the request is enqueued:

- For Bitcoin: reject `BitcoinRpcRequest` where `confirmations < MIN_BITCOIN_CONFIRMATIONS` (e.g., 6). This mirrors the H-01 mitigation of adding a Chainlink USDC/USD oracle to block mints below a price floor.
- For EVM: reject `EvmRpcRequest` where `finality == EvmFinality::Latest` (or enforce `finality >= Safe`).
- Alternatively, expose a per-chain minimum finality policy in the on-chain foreign-chain configuration (`ForeignChainsMetadata`) so that governance can set and update minimums without a contract upgrade.

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
