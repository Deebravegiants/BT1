Audit Report

## Title
ETH Deposit Limit Bypass via Missing Amount in Limit Check - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` omits the incoming `amount` from the ETH branch of its limit check, using `totalAssetDeposits > limit` instead of `totalAssetDeposits + amount > limit`. When total ETH deposits are exactly at the cap, the strict greater-than check evaluates to `false` and every subsequent ETH deposit is accepted, allowing the protocol to absorb ETH indefinitely beyond its configured ceiling.

## Finding Description
At [1](#0-0)  the function branches on asset type:

- **ETH branch (line 679):** `return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset))` — a pre-deposit snapshot with no `amount` term.
- **LST branch (line 681):** `return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset))` — correctly projects the post-deposit total.

`_beforeDeposit` at [2](#0-1)  reverts only when this function returns `true`. Because the ETH branch never adds `amount`, once `totalAssetDeposits == limit` the expression `limit > limit` is permanently `false`, and every subsequent `depositETH` call passes the guard regardless of `msg.value`.

`getTotalAssetDeposits` at [3](#0-2)  aggregates balances across the deposit pool, NDCs, EigenLayer pod shares, the unstaking vault, and the converter — all on-chain and observable, so an attacker can compute the exact top-up needed to reach the limit before exploiting.

**Exploit path:**
1. Observe `totalAssetDeposits(ETH) = limit − X`.
2. Call `depositETH{value: X}()` → check: `(limit − X) > limit` → `false` → succeeds. Now `totalAssetDeposits = limit`.
3. Call `depositETH{value: Y}()` for any `Y > 0` → check: `limit > limit` → `false` → succeeds. Now `totalAssetDeposits = limit + Y`.

Steps 2 and 3 can be executed in consecutive transactions; step 2 can be skipped entirely if the state is already at the limit.

## Impact Explanation
The ETH deposit cap (`depositLimitByAsset[ETH]`) is a protocol-level safety ceiling. Bypassing it allows the protocol to absorb more ETH than intended, violating the invariant `getTotalAssetDeposits(ETH) ≤ depositLimitByAsset[ETH]`. No funds are directly stolen, but the protocol fails to enforce its promised deposit-limit guarantee.

**Impact: Low** — Contract fails to deliver promised returns (deposit cap enforcement), but does not lose value.

## Likelihood Explanation
Any unprivileged external caller can trigger this via the public `depositETH` function. The only precondition — `totalAssetDeposits(ETH)` being exactly at the limit — can be engineered by the attacker with a single preparatory deposit. No special permissions, flash loans, or oracle manipulation are required, and the exploit is repeatable indefinitely once the limit is reached.

**Likelihood: Medium.**

## Recommendation
Include the incoming `amount` in the ETH branch to match the LST branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This unifies both branches and correctly rejects any deposit that would push total deposits above the configured limit.

## Proof of Concept

```
depositLimit  = 1_000 ETH
totalDeposits = 1_000 ETH  (attacker filled to exactly the limit)

// Bypass: deposit beyond the limit
depositETH{value: 500 ETH}()
  _beforeDeposit → _checkIfDepositAmountExceedesCurrentLimit(ETH, 500e18)
    totalAssetDeposits = 1_000 ETH
    check: 1_000 > 1_000 → false → no revert
  totalDeposits = 1_500 ETH  ← 50% over cap
```

**Foundry invariant test plan:**
```solidity
function invariant_ethDepositCapNeverExceeded() public {
    uint256 total = depositPool.getTotalAssetDeposits(ETH_TOKEN);
    uint256 limit = lrtConfig.depositLimitByAsset(ETH_TOKEN);
    assertLe(total, limit, "ETH deposit cap exceeded");
}
```
A fuzzer targeting `depositETH` will break this invariant by first filling deposits to exactly `limit`, then depositing any nonzero amount, confirming the bug at [4](#0-3) .
