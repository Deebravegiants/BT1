Audit Report

## Title
Permissionless `FeeReceiver.sendFunds()` + `LRTOracle.updateRSETHPrice()` Enables MEV Reward Front-Running to Steal Yield from rsETH Holders — (`contracts/FeeReceiver.sol`, `contracts/LRTOracle.sol`)

## Summary
`FeeReceiver.sendFunds()` carries no access control, allowing any caller to inject accumulated MEV/EL rewards into `LRTDepositPool` at will. Because `LRTOracle.updateRSETHPrice()` is equally permissionless and deposit minting uses the last *stored* `rsETHPrice`, an attacker can deposit at the pre-reward price, atomically trigger reward injection and a price update, and redeem after the withdrawal delay at the inflated price — capturing a pro-rata share of yield they did not earn, at the expense of existing rsETH holders.

## Finding Description

**Root cause — three permissionless operations compose into an exploit:**

1. `FeeReceiver.sendFunds()` has no role check: [1](#0-0) 

2. `LRTOracle.updateRSETHPrice()` is guarded only by `whenNotPaused`: [2](#0-1) 

3. `getRsETHAmountToMint` divides by the *stored* `rsETHPrice`, not a freshly computed value: [3](#0-2) 

**Exploit flow (exact math):**

Let T = total ETH in protocol, S = rsETH supply, R = reward in `FeeReceiver`, D = attacker deposit.

- Attacker deposits D ETH → minted `D·S/T` rsETH at stored price `T/S`.
- Attacker calls `sendFunds()` → deposit pool ETH balance rises by R; `getTotalAssetDeposits(ETH)` now returns `T + D + R`.
- Attacker calls `updateRSETHPrice()` → `_updateRsETHPrice()` computes `newRsETHPrice = (T+D+R) / (S + D·S/T)`: [4](#0-3) 
- After the withdrawal delay, attacker redeems `D·S/T` rsETH at the new price. Attacker ETH value = `D·(T+D+R)/(T+D)`, yielding profit `D·R/(T+D)`.
- Existing holders lose `R·D / (S·(T+D))` per rsETH token — exactly the attacker's gain redistributed from them.

**Why `pricePercentageLimit` is insufficient:** [5](#0-4) 

- If `pricePercentageLimit == 0` (default until explicitly set), the check is entirely skipped (`pricePercentageLimit > 0` is false).
- For any reward R small enough that `(newRsETHPrice - highestRsethPrice) ≤ pricePercentageLimit · highestRsethPrice`, the attack succeeds unconditionally.
- An attacker can batch many small-reward injections to accumulate profit while staying under the threshold each time.

## Impact Explanation
**High — Theft of unclaimed yield.** Every rsETH holder who held through the MEV reward accrual period loses a quantifiable fraction of their yield. The loss is `R·D/(S·(T+D))` per token, directly transferred to the attacker. This matches the allowed impact "High. Theft of unclaimed yield."

## Likelihood Explanation
High. Both entry points are fully permissionless. MEV/EL rewards accumulate continuously in `FeeReceiver`. The attack requires no privileged access, no oracle manipulation, no reentrancy, and no victim mistake — only monitoring `address(feeReceiver).balance` and submitting two transactions. The withdrawal delay prevents same-block exit but does not prevent the attack; it only defers profit realization.

## Recommendation
1. **Restrict `FeeReceiver.sendFunds()`** to a privileged role (e.g., `MANAGER` or `OPERATOR`) so reward injection timing is controlled by the protocol.
2. **Alternatively**, call `updateRSETHPrice()` atomically inside `receiveFromRewardReceiver()` so any deposit made before reward injection is priced *after* the reward is reflected, eliminating the timing gap.
3. **Ensure `pricePercentageLimit` is always initialized to a non-zero value** as defense-in-depth; consider reverting in `updateRSETHPrice()` when it is zero.

## Proof of Concept
```solidity
// Foundry fork test (pseudocode)
function testMEVFrontRun() public {
    // FeeReceiver has accumulated 10 ETH of MEV rewards
    vm.deal(address(feeReceiver), 10 ether);

    uint256 priceBefore = lrtOracle.rsETHPrice();

    // Step 1: Attacker deposits 100 ETH at stale price
    vm.prank(attacker);
    depositPool.depositETH{value: 100 ether}(0, "");
    uint256 attackerRsETH = rsETH.balanceOf(attacker);

    // Step 2: Attacker injects reward
    vm.prank(attacker);
    feeReceiver.sendFunds();

    // Step 3: Attacker triggers price update
    vm.prank(attacker);
    lrtOracle.updateRSETHPrice();

    uint256 priceAfter = lrtOracle.rsETHPrice();
    assertGt(priceAfter, priceBefore);

    // Step 4: After withdrawal delay, attacker redeems
    // attackerRsETH * priceAfter / 1e18 > 100 ether  =>  profit = D*R/(T+D)
    uint256 attackerETHValue = attackerRsETH * priceAfter / 1e18;
    assertGt(attackerETHValue, 100 ether);
}
```
The assertion holds for any non-zero `FeeReceiver` balance, provided the price increase stays within `pricePercentageLimit` (or `pricePercentageLimit == 0`).
