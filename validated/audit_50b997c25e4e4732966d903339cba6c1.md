### Title
Incomplete Payload Binding in `respond_verify_foreign_tx` Allows Cross-Request Response Replay - (File: `crates/contract/src/lib.rs`)

### Summary

The `respond_verify_foreign_tx` function verifies that a signature is valid over a caller-supplied `payload_hash`, but never checks that `payload_hash` is actually derived from the `request` fields supplied in the same call. A single attested MPC participant (below threshold) can take a legitimately produced MPC response for one pending foreign-chain verification request and replay it as the response to a different pending request, delivering a `payload_hash` that attests to a different transaction's extracted values.

### Finding Description

`respond_verify_foreign_tx` in `crates/contract/src/lib.rs` accepts two independent arguments: a `VerifyForeignTransactionRequest` (used as a lookup key into `pending_verify_foreign_tx_requests`) and a `VerifyForeignTransactionResponse` (containing `payload_hash` and `signature`). [1](#0-0) 

The signature check at lines 726–734 only verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's root public key. It does **not** recompute the expected hash from the `request` fields and compare it to `response.payload_hash`.

The canonical hash that nodes are supposed to sign is:

```
payload_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request: ForeignChainRpcRequest, values: Vec<ExtractedValue> }))
``` [2](#0-1) 

The `ForeignTxSignPayloadV1` struct binds the original `ForeignChainRpcRequest` to the observed `values`. The contract never reconstructs this struct from `request.request` and verifies the hash matches `response.payload_hash`. [3](#0-2) 

After the signature check passes, the full `response` (including the unverified `payload_hash`) is serialized and delivered to every caller waiting on that `request` key: [4](#0-3) 

### Impact Explanation

A malicious attested MPC participant (below threshold) can:

1. Observe a legitimately produced MPC response `{payload_hash_A, sig_A}` for pending request A (`tx_id: X, domain_id: D`). This response is visible on-chain once submitted.
2. While a different request B (`tx_id: Y, domain_id: D`) is pending, call `respond_verify_foreign_tx(request: B, response: {payload_hash: payload_hash_A, sig: sig_A})`.
3. The contract verifies `sig_A` over `payload_hash_A` using domain D's root key → passes, because the signature is legitimately produced.
4. The contract finds request B in `pending_verify_foreign_tx_requests` → passes.
5. The contract delivers `{payload_hash_A, sig_A}` to all callers waiting on request B.

Callers of request B receive a `VerifyForeignTransactionResponse` whose `payload_hash` attests to transaction X's extracted values (e.g., block hash of tx X), not transaction Y's. Any bridge contract (e.g., Omnibridge) that does not independently reconstruct and verify the hash from its own request fields will accept this as a valid MPC attestation of transaction Y, enabling forged foreign-chain verification and potential double-spend or invalid bridge execution. [5](#0-4) 

### Likelihood Explanation

The attacker must be a single attested MPC participant — below the signing threshold. No threshold collusion is required. The attacker does not forge any signature; they only replay a legitimately produced one. The on-chain transaction history makes previously submitted responses publicly observable. The attack is executable whenever two different `verify_foreign_transaction` requests with different `request` keys are pending on the same domain, which is a normal production condition for any active bridge.

### Recommendation

In `respond_verify_foreign_tx`, after the signature check, recompute the expected `payload_hash` from `request.request` and the `values` embedded in the response, and assert it equals `response.payload_hash`. Since `values` are not currently part of the on-chain response struct, the simplest fix is to include the full `ForeignTxSignPayload` (not just its hash) in the response, or to add a binding check that the `payload_hash` prefix encodes the original `request.request` fields. At minimum, the contract should verify:

```rust
// Pseudocode
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.values.clone(), // values must be added to the response
}).compute_msg_hash()?;
assert_eq!(expected_hash, response.payload_hash, "payload_hash does not match request");
```

This mirrors the fix applied in the Beranames report: ensure all fields of the request struct are included in the payload used for validation.

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

    all(feature = "abi", not(target_arch = "wasm32")),
    derive(schemars::JsonSchema, borsh::BorshSchema)

pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
