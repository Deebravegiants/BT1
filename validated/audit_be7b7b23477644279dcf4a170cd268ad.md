**Tracing the asymmetry:**

`is_gas_key_tx` is set based solely on `nonce_index.is_some()`: [1](#0-0) 

The deploy exclusivity guard is wrapped in `!is_gas_key_tx`: [2](#0-1) 

But the `session_deploy_tx_counts` increment is **not** guarded by `!is_gas_key_tx`: [3](#0-2) 

So a gas key tx carrying a deploy action bypasses the exclusivity check but still increments `session_deploy_tx_counts[signer_id]`. Any subsequent access key tx for the same signer computes `total_deploy_count = snapshot.deploy_tx_count + session_deploy_count > 0` and hits the `Skip` return at line 497.

The same asymmetry exists in `add_chunk_transactions`: gas key deploy txs increment `chunk_account.deploy_tx_count` without any `!is_gas_key_tx` guard: [4](#0-3) 

This means the poisoning can also come from the PTQ snapshot (`snapshot.deploy_tx_count`) if a gas key deploy tx was already included in a prior uncertified block.

The code's own comment acknowledges the non-rollback risk but claims it is mitigated because "only transactions with valid signatures can reach this point": [5](#0-4) 

That mitigation only prevents cheap unsigned spam. An attacker who legitimately controls a gas key for an account can produce a validly-signed gas key deploy tx at negligible cost.

### Title
Gas-key deploy tx poisons deploy exclusivity session state, blocking all access-key transactions for the account — (`chain/client/src/pending_transaction_queue.rs`)

### Summary
`PendingTxSession::check_pending` exempts gas-key transactions from the deploy exclusivity guard (`!is_gas_key_tx` at line 495) but unconditionally increments `session_deploy_tx_counts` for any tx that carries a deploy action (line 529–531). A single validly-signed gas-key transaction containing a `DeployContract` (or any deploy-like) action poisons `session_deploy_tx_counts[signer_id]`, causing every subsequent access-key transaction from that account to return `PendingTxCheckResult::Skip` for the remainder of the chunk production session.

### Finding Description
The invariant the code intends to enforce is: gas-key transactions are exempt from deploy exclusivity constraints. The check at lines 495–502 correctly implements the exemption for the *incoming* gas-key tx. However, the session-state update at lines 529–531 is not symmetrically guarded:

```rust
// lines 529-531 — no !is_gas_key_tx guard
if tx_has_deploy {
    *self.session_deploy_tx_counts.entry(signer_id.clone()).or_insert(0) += 1;
}
```

After this increment, any access-key tx for the same `signer_id` computes:

```
total_deploy_count = snapshot.deploy_tx_count   // from PTQ
                   + session_deploy_count        // now ≥ 1
```

and immediately hits:

```rust
if !is_gas_key_tx {
    if total_deploy_count > 0 {
        return PendingTxCheckResult::Skip;   // line 497
    }
}
```

The same bug exists in `add_chunk_transactions` (lines 266–268): gas-key deploy txs already included in uncertified blocks also increment `deploy_tx_count` in the PTQ, so the poisoning can originate from the persistent PTQ snapshot as well.

### Impact Explanation
An attacker who holds a gas key for an account (including their own account, or one where they were granted a gas key) can submit a gas-key deploy tx to block all access-key transactions from that account from being included in the current chunk. If the runtime rejects the gas-key deploy tx (e.g., because gas-key txs are not permitted to carry deploy actions), the session state is already poisoned and is not rolled back, as explicitly noted in the code comment at lines 516–525. The effect persists for the entire chunk production session. The attacker can repeat this for every session, achieving sustained exclusion of access-key transactions for the targeted account.

### Likelihood Explanation
The attacker needs a validly-signed gas-key transaction, which requires controlling a gas key for the target account. This is a realistic condition: an attacker controlling their own account with a gas key can self-target, and an attacker who was granted a gas key by a victim account can target the victim. The attack requires only one transaction submission per chunk production session.

### Recommendation
Guard the `session_deploy_tx_counts` increment with `!is_gas_key_tx`, mirroring the exclusivity check:

```rust
if tx_has_deploy && !is_gas_key_tx {
    *self.session_deploy_tx_counts.entry(signer_id.clone()).or_insert(0) += 1;
}
```

Apply the same fix to `add_chunk_transactions` at lines 266–268 if gas-key deploy txs should not contribute to the `deploy_tx_count` used for access-key exclusivity decisions.

### Proof of Concept
1. Create account `alice.near` with both an access key and a gas key (nonce_index set).
2. Construct a validly-signed gas-key `SignedTransaction` for `alice.near` containing a `DeployContractAction`.
3. Call `PendingTxSession::check_pending` with this tx and `HasContract::No`. Observe it returns `Admit` (deploy exclusivity check is skipped for gas-key txs) and `session_deploy_tx_counts["alice.near"]` is now 1.
4. Construct a validly-signed access-key `SignedTransaction` for `alice.near` containing a `TransferAction`.
5. Call `check_pending` with this tx. `total_deploy_count = 0 + 1 = 1 > 0`, so the function returns `PendingTxCheckResult::Skip`.

The access-key transfer tx is excluded from the chunk despite being valid.
