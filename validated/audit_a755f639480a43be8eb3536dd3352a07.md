### Title
Uninitialized `dailyMintLimit` After Upgrade Freezes All L2 Pool Deposits - (File: contracts/pools/RSETHPoolV3.sol)

### Summary
The `dailyMintLimit` and `startTimestamp` storage variables introduced in the `RSETHPoolV3` upgrade are zero-initialized for existing deployed contracts. If the required `reinitialize(2)` is not called immediately after the upgrade, the `limitDailyMint` modifier causes every user deposit to revert with `DailyMintLimitExceeded`, temporarily freezing all L2 pool deposits.

### Finding Description

`RSETHPoolV3` (and the same pattern in `RSETHPoolV2`, `RSETHPoolV2ExternalBridge`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`) introduces three new storage variables as part of an upgrade:

- `dailyMintLimit` (line 51)
- `dailyMintAmount` (line 54)
- `startTimestamp` (line 60) [1](#0-0) 

These variables are **not set in `initialize`** (version 1). They are only set via the separate `reinitialize(2)` call: [2](#0-1) 

Both `deposit` functions are gated by the `limitDailyMint` modifier: [3](#0-2) [4](#0-3) 

Inside `limitDailyMint`, the critical check is: [5](#0-4) 

When `dailyMintLimit == 0` (its default zero value after upgrade, before `reinitialize(2)` is called), the condition `dailyMintAmount + rsETHAmount > 0` evaluates to `true` for **any non-zero deposit**, causing an unconditional revert with `DailyMintLimitExceeded`.

Additionally, when `startTimestamp == 0`, `getCurrentDay()` returns `block.timestamp / 1 days` (a large number), so `currentDay > lastMintDay` is always true and `dailyMintAmount` is reset to 0 each call ‚Äî but this does not help, since `0 + rsETHAmount > 0` still reverts. [6](#0-5) 

The same root cause exists in:
- `RSETHPoolV2.reinitialize(3)` / `limitDailyMint`
- `RSETHPoolV2ExternalBridge.reinitialize(4)` / `limitDailyMint`
- `RSETHPoolV3ExternalBridge.reinitialize(4)` / `limitDailyMint`
- `RSETHPoolV3WithNativeChainBridge.reinitialize(2)` / `limitDailyMint` [7](#0-6) [8](#0-7) 

### Impact Explanation

**Impact: Medium ‚Äî Temporary freezing of funds.**

Any user attempting to deposit ETH or supported tokens into the L2 pool during the window between the contract upgrade and the `reinitialize` call will have their transaction revert. No funds are lost (ETH is returned on revert), but the pool is completely non-functional for deposits during this window. The `reinitialize` is a separate admin transaction from the upgrade itself, creating a guaranteed exposure window.

### Likelihood Explanation

**Likelihood: Medium.**

The upgrade and the `reinitialize` call are two separate transactions. Any block mined between them exposes the pool to this freeze. On busy L2 networks (Arbitrum, Base, Optimism), users actively attempt deposits continuously. The window is short but guaranteed to exist. Additionally, if the upgrade script omits the `reinitialize` call entirely (a realistic operational mistake), the freeze is permanent until corrected.

### Recommendation

Initialize `dailyMintLimit` and `startTimestamp` directly within the upgrade's `reinitialize` function, or add a guard in `limitDailyMint` to skip the check when `dailyMintLimit == 0` (treating zero as "no limit enforced"):

```solidity
modifier limitDailyMint(uint256 amount, address token) {
    if (block.timestamp < startTimestamp) {
        revert MintBeforeStartTimestamp();
    }
    // Skip limit enforcement if dailyMintLimit is not yet initialized
    if (dailyMintLimit == 0) {
        _;
        return;
    }
    // ... rest of the check
}
```

Alternatively, ensure the upgrade transaction atomically calls `reinitialize` in the same transaction (e.g., via a multicall or upgrade script that batches both calls).

### Proof of Concept

1. `RSETHPoolV3` proxy is live on an L2 with `_initialized == 1` (post-`initialize`). `dailyMintLimit == 0`, `startTimestamp == 0`.
2. Admin upgrades the implementation to the new `RSETHPoolV3` logic that includes `limitDailyMint`. The `reinitialize(2)` has not yet been called.
3. A user calls `deposit{value: 1 ether}("ref")`.
4. `limitDailyMint(1 ether, ETH_IDENTIFIER)` executes:
   - `block.timestamp < 0` ‚Üí false (passes)
   - `rsETHAmount` is computed as ~`1e18 / rate` (non-zero)
   - `currentDay = block.timestamp / 1 days` ‚Üí large number > `lastMintDay (0)` ‚Üí `dailyMintAmount` reset to 0
   - `0 + rsETHAmount > dailyMintLimit (0)` ‚Üí `rsETHAmount > 0` ‚Üí **true**
   - Reverts: `DailyMintLimitExceeded`
5. All deposits revert until admin calls `reinitialize(2)` with a valid `_dailyMintLimit > 0`. [9](#0-8)

```text
    modifier limitDailyMint(uint256 amount, address token) {
        if (block.timestamp < startTimestamp) {
            revert MintBeforeStartTimestamp();
        }

        uint256 rsETHAmount;

        // Calculate the amount of rsETH that will be minted
        if (token == ETH_IDENTIFIER) {
            (rsETHAmount,) = viewSwapRsETHAmountAndFee(amount);
        } else {
            (rsETHAmount,) = viewSwapRsETHAmountAndFee(amount, token);
        }

        uint256 currentDay = getCurrentDay();

        // If the current day is greater than the last mint day, reset the daily mint amount
        if (currentDay > lastMintDay) {
            lastMintDay = currentDay;
            dailyMintAmount = 0;
        }

        // Check if the daily mint amount plus the amount to mint is greater than the daily mint limit
        if (dailyMintAmount + rsETHAmount > dailyMintLimit) {
            revert DailyMintLimitExceeded();
        }

        dailyMintAmount += rsETHAmount;
        _;
    }
```
