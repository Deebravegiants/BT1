### Title
Permanent `stakedButUnverifiedNativeETH` Inflation from Pre-Verification Beacon Chain Slashing Inflates rsETH Price — (`contracts/NodeDelegator.sol`)

### Summary
`NodeDelegator.stakedButUnverifiedNativeETH` is incremented by 32 ETH on every `stake32Eth` call and is **only ever decremented** inside `verifyWithdrawalCredentials`. If a validator is slashed on the beacon chain before its withdrawal credentials are verified, the EigenPod will reject any subsequent `verifyWithdrawalCredentials` call for that validator (exit epoch is set → `ValidatorIsExitingBeaconChain`). There is no other code path — no admin setter, no emergency correction — that can reduce `stakedButUnverifiedNativeETH`. The phantom 32 ETH per slashed validator permanently inflates `getEffectivePodShares()`, which flows directly into `rsETHPrice` computation.

### Finding Description
`stake32Eth` unconditionally increments `stakedButUnverifiedNativeETH`:

```solidity
// NodeDelegator.sol:166
stakedButUnverifiedNativeETH += 32 ether;
```

The only decrement is:

```solidity
// NodeDelegator.sol:240
stakedButUnverifiedNativeETH -= (validatorFields.length * (32 ether));
```

This decrement is gated on a successful `eigenPod.verifyWithdrawalCredentials(...)` call. EigenLayer's EigenPod rejects credential verification for validators with an exit epoch set (`ValidatorIsExitingBeaconChain`). A beacon-chain-slashed validator has its exit epoch set within the same epoch as slashing. Therefore, `verifyWithdrawalCredentials` will revert for any slashed validator, and `stakedButUnverifiedNativeETH` can never be corrected for it.

`getEffectivePodShares()` returns:
```solidity
return stakedButUnverifiedNativeETH + withdrawableShare;
```

Since the slashed validator was never verified, `withdrawableShare` from EigenLayer is 0 for it, while `stakedButUnverifiedNativeETH` still counts 32 ETH. The phantom ETH propagates through:

`getEffectivePodShares()` → `getETHDistributionData()` → `getTotalAssetDeposits(ETH)` → `_getTotalEthInProtocol()` → `newRsETHPrice = totalETHInProtocol / rsethSupply`

The `pricePercentageLimit` upside check (lines 252–266 of `LRTOracle.sol`) only blocks non-manager callers when the increase exceeds the threshold; the manager can always call `updateRSETHPriceAsManager()` to bypass it. For small numbers of slashed validators relative to TVL, the per-update increase is below any reasonable threshold anyway.

### Impact Explanation
Each slashed-but-unverified validator permanently adds 32 ETH of phantom value to the protocol's reported TVL. `rsETHPrice` is inflated proportionally. Holders who redeem rsETH at the inflated price receive more ETH than their share of actual backing, draining real ETH from the protocol. With N slashed validators, the protocol is insolvent by N × 32 ETH. This satisfies **Critical — Protocol insolvency**.

### Likelihood Explanation
Beacon chain slashing requires validator misbehavior (double voting, surround voting) or a compromised/buggy validator client. For a protocol operating dozens of validators across multiple NDCs, the probability of at least one slashing event over a multi-year horizon is non-negligible. The operator does not need to be malicious — a software bug in the validator client is sufficient. Likelihood is **Low**, but the impact is permanent and unbounded per slashed validator.

### Recommendation
1. Add an admin/operator function to manually correct `stakedButUnverifiedNativeETH` for validators that can be proven slashed on-chain (e.g., by verifying the slashed flag from a beacon state proof).
2. Alternatively, integrate EigenPod's `verifyStaleBalance` flow: after a slashing is detected, call `startCheckpoint` / `verifyCheckpointProofs` to let EigenLayer record the actual balance, then decrement `stakedButUnverifiedNativeETH` accordingly.
3. At minimum, add an emergency admin setter for `stakedButUnverifiedNativeETH` protected by a timelock, so the protocol can correct phantom ETH after a slashing event is confirmed.
