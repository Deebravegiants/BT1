### Title
Dijkstra Batch Transaction Deposit Refund Double-Counting Allows ADA Creation From Thin Air - (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, `getConsumedDijkstraValue` computes the total consumed value for a batch transaction by summing the consumed value of the top-level transaction and each sub-transaction independently, passing the **same original-state** `lookupStakingDeposit` / `lookupDRepDeposit` functions to every sub-transaction. Because these lookup functions are derived from the pre-batch cert state and are never updated as sub-transactions are processed, multiple sub-transactions can each claim the full deposit refund for the same staking credential. The `validateValueNotConservedUTxO` balance check therefore passes with an inflated consumed value, violating the preservation-of-value invariant and enabling ADA creation from thin air. The developers have acknowledged this gap with a disabled test (`xit "Multiple subtransactions cannot get the same refund"`) whose expected predicate failure is explicitly marked `error "TODO: predicate failure not yet implemented"`.

---

### Finding Description

`getConsumedDijkstraValue` in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs` aggregates consumed value across the top-level transaction and all sub-transactions:

```haskell
getConsumedDijkstraValue pp lookupStakingDeposit lookupDRepDeposit utxo txBody =
  withBothTxLevels
    txBody
    ( \topTxBody ->
        txBodyConsumedValue topTxBody <> subTransactionsConsumedValue topTxBody
    )
    txBodyConsumedValue
  where
    txBodyConsumedValue :: forall m. TxBody m era -> Value era
    txBodyConsumedValue = getConsumedMaryValue pp lookupStakingDeposit lookupDRepDeposit utxo
    subTransactionsConsumedValue topTxBody =
      foldMap'
        (getConsumedValue pp lookupStakingDeposit lookupDRepDeposit utxo . view bodyTxL)
        (topTxBody ^. subTransactionsTxBodyL)
``` [1](#0-0) 

The critical flaw: `lookupStakingDeposit` and `lookupDRepDeposit` are closures over the **original** cert state and are passed unchanged to every sub-transaction's `getConsumedValue` call. There is no stateful accumulator that marks a deposit as "already claimed" after the first sub-transaction deregisters a credential.

The Dijkstra UTXO rule invokes the balance check at line 381 using this same original cert state:

```haskell
{- consumed pp utxo₀ txb = produced pp certState txb -}
runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
``` [2](#0-1) 

For a batch containing N sub-transactions each carrying `UnRegDepositTxCert C D` for the same credential `C` with deposit `D`:

- **Consumed** = UTxO inputs + N × D  (each sub-tx independently looks up D from the original state)
- **Produced** = outputs + fee + 0 deposits

The balance check passes when `outputs = inputs + N×D − fee`, allowing the transaction to claim N×D in refunds while only one deposit of D was ever paid.

By contrast, the analogous cross-batch check for **withdrawals** was correctly implemented:

```haskell
validateBatchWithdrawals accounts tx =
  let allWithdrawals =
        Map.unionsWith (<>) $
          unWithdrawals (tx ^. bodyTxL . withdrawalsTxBodyL)
            : [ unWithdrawals $ subTx ^. bodyTxL . withdrawalsTxBodyL
              | subTx <- OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL
              ]
      badWithdrawals = Map.mapMaybeWithKey (\acctAddr withdrawn ->
          let balance = getAccountBalance acctAddr
           in if withdrawn > balance then Just ... else Nothing) allWithdrawals
   in failureOnNonEmptyMap badWithdrawals WithdrawalsExceedAccountBalance
``` [3](#0-2) 

No equivalent cross-sub-transaction guard exists for deposit refunds.

The developers have explicitly acknowledged this gap in the disabled test:

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
    stakingCred <- KeyHashObj <$> freshKeyHash
    _ <- registerStakeCredential stakingCred
    keyDeposit <- getsPParams ppKeyDepositL
    ...
    subTx1 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
    subTx2 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
    tx = mkBasicTx mkBasicTxBody & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
    submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
``` [4](#0-3) 

The `error "TODO: predicate failure not yet implemented"` placeholder — which would cause a runtime crash if the test ran — confirms that the specific predicate failure guarding against this case has not been implemented. The test is disabled (`xit`) precisely because the transaction currently passes validation when it should be rejected.

The same pattern applies to DRep deposit refunds: `lookupDRepDeposit` is derived from `lookupDepositVState` over the original `VState`, and is passed identically to every sub-transaction's consumed calculation. [5](#0-4) 

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An attacker who registers a staking credential (paying deposit D) can craft a Dijkstra batch transaction with N sub-transactions each containing `UnRegDepositTxCert` for the same credential. The balance check accepts consumed = inputs + N×D while produced = outputs + fee, so the attacker can place N×D − fee in outputs while only D was ever deposited. For N = 2 this yields a net gain of D ADA per batch. The attack is repeatable: re-register the credential and repeat. The preservation-of-value invariant — the foundational accounting property of the Cardano ledger — is violated, constituting a direct invalid ledger state transition that creates ADA from thin air.

---

### Likelihood Explanation

The Dijkstra era is production code in this repository. The attack requires only:
1. A funded wallet to pay the initial deposit and transaction fees.
2. The ability to construct a batch transaction with two sub-transactions — a standard unprivileged operation.
3. No special keys, governance majority, or external dependencies.

The disabled test and the `TODO` comment confirm the developers are aware the guard is missing. Any user of the Dijkstra era can trigger this path.

---

### Recommendation

Introduce a cross-sub-transaction deposit-refund guard analogous to `validateBatchWithdrawals`. Before the balance check, accumulate all `UnRegDepositTxCert` and `UnRegDRepTxCert` certificates across the top-level transaction and all sub-transactions. Reject the batch if any credential appears in more than one deregistration certificate (net of same-sub-transaction re-registrations). Alternatively, thread a stateful "already-claimed" set through `getConsumedDijkstraValue` so that `lookupStakingDeposit` returns `Nothing` for a credential whose refund has already been counted by an earlier sub-transaction.

---

### Proof of Concept

1. Register staking credential `C`, paying deposit `D = ppKeyDeposit`.
2. Fund two UTxO entries `input1` (value `V1`) and `input2` (value `V2`).
3. Construct:
   ```
   subTx1 = { inputs: {input1}, certs: [UnRegDepositTxCert C D] }
   subTx2 = { inputs: {input2}, certs: [UnRegDepositTxCert C D] }
   topTx  = { subTransactions: [subTx1, subTx2],
               outputs: [addr ↦ V1 + V2 + 2D − fee] }
   ```
4. Submit `topTx`. The `validateValueNotConservedUTxO` check computes:
   - consumed = V1 + V2 + D (sub-tx1 lookup) + D (sub-tx2 lookup) = V1 + V2 + 2D
   - produced = (V1 + V2 + 2D − fee) + fee = V1 + V2 + 2D ✓
5. The balance check passes. The attacker receives `V1 + V2 + 2D − fee` in outputs, having only deposited `D` and spent `V1 + V2` from UTxO — a net gain of `D` ADA.
