### Title
Missing `kem_ss` size validation in `kem_policy_runtime_kem_t::encapsulate` allows malicious callback to force HKDF derivation from empty IKM, producing a predictable AES key - (`include-internal/cbmpc/internal/protocol/pve_base.h`)

---

### Summary

`kem_policy_runtime_kem_t::encapsulate` blindly forwards the callback's `kem_ss` output to `kem_aead_ciphertext_t::seal`, which passes it directly to HKDF with no size guard. A malicious `encap_fn_t` callback that returns `SUCCESS` with an empty `kem_ss` causes the library to derive a fully deterministic, publicly computable AES-GCM-256 key from empty IKM. Any plaintext (e.g., a private scalar share) encrypted through `pve_base_pke_runtime_kem()` with that callback is then recoverable by anyone who independently evaluates the same HKDF invocation.

---

### Finding Description

**Entrypoint**: `pve_base_pke_runtime_kem().encrypt(...)` → `kem_pve_base_pke_t<kem_policy_runtime_kem_t>::encrypt` → `kem_aead_ciphertext_t<kem_policy_runtime_kem_t>::seal`.

**Step 1 – callback dispatch with no output validation.**

`kem_policy_runtime_kem_t::encapsulate` calls the user-supplied callback and returns its result verbatim:

```cpp
return pub_key.callbacks->encap(pub_key.callbacks->ctx, pub_key.ek_bytes, rho, kem_ct, kem_ss);
``` [1](#0-0) 

There is no check that `kem_ss` is non-empty or of any expected length after the callback returns.

**Step 2 – C API shim does not reject empty `kem_ss`.**

`kem_encap_shim` only rejects negative sizes or a non-null pointer with zero size:

```cpp
if (kem_ss.size < 0 || (kem_ss.size > 0 && !kem_ss.data)) { ... return E_FORMAT; }
``` [2](#0-1) 

`kem_ss.size == 0` (empty buffer) passes this check and is forwarded as an empty `buf_t`.

**Step 3 – `seal` uses `kem_ss` for HKDF with no guard.**

After `encapsulate` returns `SUCCESS`, `seal` immediately feeds `kem_ss` into HKDF:

```cpp
buf_t prk = crypto::hkdf_extract_sha256(mem_t(), kem_ss);
buf_t aes_key = crypto::hkdf_expand_sha256(prk, mem_t("CBMPC|KEM-AEAD|v1|KDF=HKDF-SHA256|AEAD=AES-GCM-256"), 32);
crypto::aes_gcm_t::encrypt(aes_key, mem_t(iv, iv_size), aad, tag_size, plain, aead_ciphertext);
``` [3](#0-2) 

With empty `kem_ss`, `hkdf_extract_sha256(salt=∅, IKM=∅)` produces a fixed, publicly known PRK, and `hkdf_expand` produces a fixed 32-byte AES key. The plaintext is then encrypted under that key.

**Contrast with sibling HSM policies that do validate.**

`kem_policy_rsa_oaep_hsm_t::decapsulate` explicitly rejects wrong-length `kem_ss`:

```cpp
if (kem_ss.size() != expected_ss_size) return coinbase::error(E_CRYPTO, "invalid RSA KEM output size");
``` [4](#0-3) 

`kem_policy_ecdh_p256_hsm_t::decapsulate` similarly rejects wrong-length ECDH output:

```cpp
if (dh.size() != 32) return coinbase::error(E_CRYPTO, "invalid ECDH output size");
``` [5](#0-4) 

`kem_policy_runtime_kem_t` has no equivalent guard on either the encapsulate or decapsulate side.

---

### Impact Explanation

Any private scalar share, Paillier secret fragment, or other secret material passed as `plain` to `pve_base_pke_runtime_kem().encrypt()` with a malicious callback is encrypted under a deterministic AES-GCM-256 key. The key is fully computable offline from the public HKDF label `"CBMPC|KEM-AEAD|v1|KDF=HKDF-SHA256|AEAD=AES-GCM-256"` and empty IKM. The IV is stored in the ciphertext struct. An attacker who observes the ciphertext can decrypt it without any secret material.

---

### Likelihood Explanation

The `pve_runtime_kem_callbacks_t` / `cbmpc_pve_base_kem_t` interface is explicitly designed for HSM drivers and FFI wrappers. A compromised or malicious HSM driver, a buggy wrapper that returns early before writing `kem_ss`, or a deliberately adversarial callback provider can all trigger this path. The documentation only requires that `encap` be deterministic given `rho32`; it does not require a non-empty `kem_ss`, so a callback author may not realize the consequence of returning an empty buffer on success.

---

### Recommendation

Add a minimum-length guard in `kem_policy_runtime_kem_t::encapsulate` immediately after the callback returns, mirroring the pattern already used in `kem_policy_rsa_oaep_hsm_t` and `kem_policy_ecdh_p256_hsm_t`. A reasonable minimum is 16 bytes (128-bit security); 32 bytes matches the rest of the codebase. Additionally, update the `pve_runtime_kem_callbacks_t` documentation to state the minimum required `kem_ss` length, and add the same check in `kem_encap_shim` for the C API path.

### Proof of Concept

```cpp
// Malicious encap callback: returns SUCCESS with empty kem_ss
static error_t evil_encap(void* ctx, mem_t ek_bytes, mem_t rho32,
                           buf_t& kem_ct, buf_t& kem_ss) {
  kem_ct = buf_t("fake-ct");
  kem_ss = buf_t();  // empty — no error returned
  return SUCCESS;
}

// Attacker independently computes the deterministic AES key
buf_t prk = hkdf_extract_sha256(mem_t(), mem_t());  // empty IKM
buf_t aes_key = hkdf_expand_sha256(
    prk, mem_t("CBMPC|KEM-AEAD|v1|KDF=HKDF-SHA256|AEAD=AES-GCM-256"), 32);

// Encrypt a private scalar share using the malicious callback
pve_runtime_kem_callbacks_t cbs;
cbs.encap = evil_encap;
pve_runtime_kem_ek_t ek{mem_t("ek"), &cbs};
buf_t ct;
pve_base_pke_runtime_kem().encrypt(pve_keyref(ek), "label", secret_share, rho, ct);

// Deserialize ct, extract iv, then:
// aes_gcm_t::decrypt(aes_key, iv, "label", tag_size, aead_ciphertext, recovered_plain)
// recovered_plain == secret_share  ✓
```
