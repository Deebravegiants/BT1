### Title
Pre-upgrade buffered receipts without `ReceiptGroupsQueueData` cause unconditional `assert_eq!` panic during dynamic resharding — (`chain/chain/src/resharding/manager.rs`)

---

### Summary

`get_child_congestion_info_not_finalized` assumes that the sum of all `ReceiptGroupsQueue::total_gas()` values across parent shards exactly equals `parent_congestion_info.buffered_receipts_gas()`. This invariant is broken for any shard whose outgoing buffer contains receipts buffered before the bandwidth-scheduler protocol upgrade, because those receipts were stored without writing `ReceiptGroupsQueueData` to the trie. The function silently skips those shards (`continue`), leaves `buffered_receipts_gas > 0`, and then hits the unconditional `assert_eq!(congestion_info.buffered_receipts_gas(), 0)` — a hard panic in production — causing every node to crash at the resharding boundary block.

---

### Finding Description

**Root cause — `buffer_receipt` always increments `buffered_receipts_gas` but only conditionally writes metadata:** [1](#0-0) 

`add_buffered_receipt_gas` is called unconditionally (line 487), but `outgoing_metadatas.update_on_receipt_pushed` is guarded by `receipt.should_update_outgoing_metadatas()` (line 489). For pre-upgrade receipts stored as `ReceiptOrStateStoredReceipt::Receipt`, that predicate returns `false`: [2](#0-1) 

And for `StateStoredReceipt::V0` (the migration variant): [3](#0-2) 

The `use_state_stored_receipt` flag in `ApplyState` is protocol-version-gated. Before the upgrade it is `false`, so `buffer_receipt` wraps receipts as the plain `Receipt` variant — incrementing `buffered_receipts_gas` in `CongestionInfo` but writing **no** `BufferedReceiptGroupsQueueData` trie key.

**Root cause — `get_child_congestion_info_not_finalized` silently skips shards with no metadata:** [4](#0-3) 

When `ReceiptGroupsQueue::load` returns `None` (key absent from trie), the loop does `continue` — no gas is subtracted for that shard. After the loop, the function asserts: [5](#0-4) 

This is a hard `assert_eq!`, not a debug assertion. It panics in release builds.

**Call site — called with real `parent_congestion_info` during resharding:** [6](#0-5) 

`parent_congestion_info` comes from `parent_chunk_extra.congestion_info()` (line 225), which is the live `CongestionInfo` carried in the chunk header — it faithfully reflects the gas of all receipts currently in the outgoing buffer, including pre-upgrade ones.

**Why pre-upgrade receipts can still be in the buffer at resharding time:**

`forward_from_buffer_to_shard` does forward pre-upgrade receipts and decrements `buffered_receipts_gas` when they are forwarded. However, if the target shard is congested, forwarding stops at `ReceiptForwarding::NotForwarded` → `break`: [7](#0-6) 

Any pre-upgrade receipts that could not be forwarded before the resharding epoch boundary remain in the buffer with their gas counted in `CongestionInfo` but with no `ReceiptGroupsQueueData` in the trie.

---

### Impact Explanation

Every node executes the same deterministic resharding logic. When `assert_eq!(congestion_info.buffered_receipts_gas(), 0)` fires, every node panics at the same resharding boundary block. The chain halts permanently at that height until a hotfix is deployed. This is a Critical consensus-level impact.

---

### Likelihood Explanation

The scenario requires two conditions to coincide:

1. At least one shard has buffered receipts (stored pre-upgrade, without `ReceiptGroupsQueueData`) that were not fully drained before the resharding epoch.
2. Dynamic resharding is scheduled for the epoch immediately following (or shortly after) the bandwidth-scheduler upgrade.

An unprivileged user can guarantee condition 1 by sending cross-shard transactions to a congested shard before the upgrade. Condition 2 depends on protocol scheduling, but if resharding and the upgrade land in adjacent epochs (a realistic deployment scenario), the window is real. The attacker does not need any special privileges — only the ability to submit public RPC transactions.

---

### Recommendation

Replace the `continue` + unconditional `assert_eq!` pattern with a fallback that iterates the actual outgoing buffer receipts for shards whose `ReceiptGroupsQueueData` is absent, computing their gas directly (similar to `bootstrap_congestion_info`). Alternatively, block resharding until all outgoing buffers contain only post-upgrade (`StateStoredReceipt::V1`) receipts, or add a migration step that writes `ReceiptGroupsQueueData` for pre-upgrade receipts during the upgrade epoch.

---

### Proof of Concept

Construct a unit test that:
1. Creates a parent trie with `CongestionInfo` having `buffered_receipts_gas > 0`.
2. Populates the outgoing buffer for shard X with plain `Receipt` entries (no `BufferedReceiptGroupsQueueData` trie key written).
3. Calls `ReshardingManager::get_child_congestion_info` (or the private `get_child_congestion_info_not_finalized`) with `retain_mode = RetainMode::Right`.
4. Observes the `assert_eq!(congestion_info.buffered_receipts_gas(), 0)` panic.
