Audit Report

## Title
Cumulative Aave Deposit Rounding Drift Causes `totalETHDepositedToAave` to Permanently Exceed aWETH Balance, Blocking User ETH Withdrawals and Interest Collection — (`contracts/LRTWithdrawalManager.sol`)

## Summary
`_depositToAave` credits `totalETHDepositedToAave` with the full requested `amount`, but Aave v3's scaled-balance arithmetic (`rayDiv`/`rayMul` with rounding) can mint `amount − 1` aWETH, creating a 1-wei drift per deposit-withdraw cycle. After 3 cycles the drift exceeds the 2-wei tolerance in `_checkAaveHealth`, permanently blocking `collectInterestToTreasury`. Even after a single cycle, if the contract holds no idle ETH, `_processWithdrawalCompletion` reverts with `InsufficientLiquidityForWithdrawal` because `_withdrawFromAave` can only deliver `amountNeeded − 1`.

## Finding Description

**Root cause — `_depositToAave` (L897-898):** [1](#0-0) 
The accounting variable is incremented by the requested `amount`, not by the actual aWETH balance delta. Aave v3 mints `floor(amount × RAY / liquidityIndex)` scaled tokens; `balanceOf` returns `floor(scaledBalance × liquidityIndex / RAY)`. The double rounding means `aaveAWETH.balanceOf(address(this))` can be `amount − 1` immediately after deposit whenever `liquidityIndex > RAY` (always true on mainnet after any interest has accrued).

**`_withdrawFromAave` propagates the drift (L912-918):** [2](#0-1) 
`withdrawablePrincipal = min(aaveBalance, totalETHDepositedToAave)`. If `aaveBalance = amount − 1`, then `withdrawnAmount = amount − 1` and `totalETHDepositedToAave -= (amount − 1)` leaves a residue of 1 wei with `aaveBalance = 0`. Each deposit-withdraw cycle adds 1 wei of permanent drift.

**`_checkAaveHealth` tolerates only 2 wei (L931):** [3](#0-2) 
After 3 cycles the drift equals 3 wei, causing `_checkAaveHealth` to return `false` and permanently blocking any function that gates on it (e.g., `collectInterestToTreasury`).

**`_processWithdrawalCompletion` reverts on 1-wei shortfall (L722-730):** [4](#0-3) 
When the contract has no idle ETH, `_withdrawFromAave(amountNeeded)` is capped at `aaveBalance = amountNeeded − 1`. The hard check `balanceAfter < request.expectedAssetAmount` then fires and reverts with `InsufficientLiquidityForWithdrawal`. The user's request was deleted at L712 before this check, but the revert unwinds the deletion — the user is stuck in a retry loop that always fails without operator intervention.

## Impact Explanation

Two concrete Medium-severity impacts are proven:

1. **Temporary freezing of funds** — user ETH withdrawal requests revert deterministically after even a single deposit-withdraw cycle when the contract holds no idle ETH. Recovery requires an operator to call `emergencyWithdrawFromAave` and externally donate the missing wei, constituting operator-dependent manual intervention.
2. **Permanent freezing of unclaimed yield** — after 3 cycles, `_checkAaveHealth` returns `false` unconditionally, blocking `collectInterestToTreasury` indefinitely until the accounting state is corrected.

Both match **Medium — Temporary freezing of funds** and **Medium — Permanent freezing of unclaimed yield** from the allowed impact scope, which supersedes the submitted Low classification.

## Likelihood Explanation

- Aave integration must be enabled (operator-controlled, realistic in production).
- `liquidityIndex > RAY` is guaranteed on mainnet Aave v3 WETH after any interest accrues — from day 1 of deployment.
- The rounding loss is deterministic, not probabilistic; it occurs on every deposit-withdraw cycle.
- No attacker is required: normal operator use of `depositIdleETHToAave` followed by a user calling `completeWithdrawal` is sufficient.
- N > 2 cycles is trivially reached in normal operation.

## Recommendation

1. **Measure actual aWETH received**: after `aaveWETHGateway.depositETH(...)`, read `aaveAWETH.balanceOf(address(this))` and use the balance delta as the increment to `totalETHDepositedToAave`.
2. **Measure actual ETH received**: after `aaveWETHGateway.withdrawETH(...)`, use the contract's ETH balance delta as the decrement to `totalETHDepositedToAave`, not the requested `withdrawnAmount`.
3. Alternatively, remove `totalETHDepositedToAave` as a principal tracker and derive principal directly from the aWETH balance, treating any excess as interest.
4. Increase or remove the fixed 2-wei tolerance in `_checkAaveHealth`, replacing it with a proportional bound (e.g., 1 wei per deposit operation recorded).

## Proof of Concept

```solidity
// Foundry fork test — Aave v3 mainnet fork (liquidityIndex > RAY from day 1)
function test_roundingDrift() public {
    uint256 amount = 1 ether;

    // Cycle 1: deposit → aaveBalance may be amount-1 due to scaled rounding
    vm.deal(address(withdrawalManager), amount);
    vm.prank(operator);
    withdrawalManager.depositIdleETHToAave(amount);

    uint256 aaveBalance = aaveAWETH.balanceOf(address(withdrawalManager));
    uint256 principal   = withdrawalManager.totalETHDepositedToAave();
    // principal == 1e18, aaveBalance == 1e18-1 → drift = 1

    // Simulate withdrawal needing full `amount` with no idle ETH:
    // _withdrawFromAave(1e18) → withdrawablePrincipal = 1e18-1 → withdrawnAmount = 1e18-1
    // balanceAfter = 1e18-1 < 1e18 → InsufficientLiquidityForWithdrawal ✓

    // Cycles 2 and 3 accumulate drift to 3 wei:
    for (uint i = 1; i < 3; i++) {
        vm.deal(address(withdrawalManager), amount);
        vm.prank(operator);
        withdrawalManager.depositIdleETHToAave(amount);
        // trigger a full withdrawal cycle to drain aaveBalance back to 0
        // ...
    }

    // After 3 cycles: principal - aaveBalance == 3 > 2
    assertFalse(withdrawalManager._checkAaveHealth()); // collectInterestToTreasury blocked ✓
}
```
