### Title
Proposer Deposits Permanently Locked When `do_update` Clears All Pending Proposals Without Refund - (`File: crates/contract/src/update.rs`)

---

### Summary

When `do_update` executes a winning contract/config upgrade proposal, it unconditionally clears **all** pending proposals and votes from storage. However, it never refunds the NEAR storage-staking deposits that each proposer attached when calling `propose_update`. Those deposits remain in the contract balance with no on-chain mechanism to retrieve them, permanently locking participant funds.

---

### Finding Description

`propose_update` is a `#[payable]` function that requires the caller to attach a deposit proportional to the size of the proposed update (contract binary or config blob) to cover NEAR storage staking costs:

```rust
// lib.rs:1308-1331
let attached = env::attached_deposit();
let required = ProposedUpdates::required_deposit(&update);
if attached < required { return Err(...) }
let id = self.proposed_updates.propose(update);
// Only excess above `required` is refunded; the `required` portion stays in the contract.
if let Some(diff) = attached.checked_sub(required) && diff > NearToken::from_yoctonear(0) {
    Promise::new(proposer).transfer(diff).detach();
}
``` [1](#0-0) 

The deposit calculation is based on the serialized size of the update payload plus a fixed overhead for 128 participant votes:

```rust
// update.rs:278-295
fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;
    match update {
        Update::Contract(code) => { bytes_used += code.len() as u128; }
        ...
    }
    bytes_used
}
fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
``` [2](#0-1) 

For a full contract binary (~1.5 MB), this deposit is on the order of tens of NEAR tokens per proposal.

When `vote_update` reaches threshold and calls `do_update`, the function removes the winning entry, then **clears all remaining entries and votes** — but issues **zero refunds** to any proposer:

```rust
// update.rs:195-227
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    // ... builds and returns the deploy/config Promise, no refund anywhere
}
``` [3](#0-2) 

There is no `cancel_update`, `withdraw_update_deposit`, or any other on-chain path to recover these deposits. The contract API exposes only `propose_update` and `vote_update` for this flow. [4](#0-3) 

---

### Impact Explanation

Every participant who proposed a competing update loses their entire storage-staking deposit the moment any other proposal reaches threshold and `do_update` fires. The freed storage bytes reduce the contract's storage obligation, but the corresponding NEAR tokens remain in the contract's own balance with no withdrawal path. This permanently breaks the accounting invariant that proposers should recover their deposits when their proposals are superseded or cleared.

For a realistic governance scenario where N participants each propose a different contract binary (~1.5 MB each at ~10 yoctoNEAR/byte), the locked amount is `(N-1) × ~15 NEAR` per upgrade cycle. Over multiple upgrade cycles this accumulates without bound.

This matches the allowed Medium impact: **"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."**

---

### Likelihood Explanation

The trigger is a normal, expected governance operation: multiple participants independently propose contract upgrades (a documented and tested workflow), one reaches threshold, and `do_update` fires. No adversarial behavior is required. The sandbox test `test_propose_update_contract_many` explicitly exercises the multi-proposal scenario and confirms all proposals are cleared after one executes — it does not check for deposit refunds. [5](#0-4) 

---

### Recommendation

In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and issue a `Promise::new(proposer_account_id).transfer(deposit)` for each. This requires storing the proposer's `AccountId` and the exact deposit amount inside `UpdateEntry` at proposal time (analogous to how `PendingAttestation` stores `attached_deposit` for the attestation flow). [6](#0-5) 

Alternatively, add a `cancel_update(id: UpdateId)` endpoint that allows the original proposer to withdraw their proposal and receive a refund, mirroring the pattern used for `remove_update_vote`. [7](#0-6) 

---

### Proof of Concept

1. Three participants A, B, C are in the network (threshold = 2).
2. A calls `propose_update(code_A)` attaching 15 NEAR → deposit stored in contract.
3. B calls `propose_update(code_B)` attaching 15 NEAR → deposit stored in contract.
4. B and C call `vote_update(B's id)` → threshold reached, `do_update` fires.
5. `do_update` removes B's entry, then calls `self.entries.clear()` (removing A's entry) and `self.vote_by_participant.clear()`.
6. No refund is issued to A. A's 15 NEAR is permanently locked in the contract balance.
7. There is no on-chain function A can call to recover the deposit. [3](#0-2) [8](#0-7)

    #[payable]
    #[handle_result]
    pub fn propose_update(
        &mut self,
        #[serializer(borsh)] args: ProposeUpdateArgs,
    ) -> Result<UpdateId, Error> {
        // Only voters can propose updates:
        let proposer = self.voter_or_panic();
        let update: Update = args.try_into()?;

        let attached = env::attached_deposit();
        let required = ProposedUpdates::required_deposit(&update);
        if attached < required {
            return Err(InvalidParameters::InsufficientDeposit {
                attached: attached.as_yoctonear(),
                required: required.as_yoctonear(),
            }
            .into());
        }

        let id = self.proposed_updates.propose(update);

        log!(
            "propose_update: signer={}, id={:?}",
            env::signer_account_id(),
            id,
        );

        // Refund the difference if the proposer attached more than required.
        if let Some(diff) = attached.checked_sub(required)
            && diff > NearToken::from_yoctonear(0)
        {
            Promise::new(proposer).transfer(diff).detach();
        }

        Ok(id)
    }

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

            bytes_used += bytes.len() as u128;
        }

}
