Audit Report

## Title
Chainlink Oracle Staleness Not Validated in `getAssetPrice`, Enabling Stale-Price Deposit Exploitation - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all freshness fields (`updatedAt`, `answeredInRound`, `roundId`), unconditionally accepting whatever price the feed returns. When a Chainlink feed goes stale at an inflated price, an unprivileged depositor can call `LRTDepositPool.depositAsset()` and receive excess rsETH computed against the stale high price, diluting all existing rsETH holders once the oracle corrects.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` (L49–55) destructures the `latestRoundData()` return tuple as:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

`roundId`, `updatedAt`, and `answeredInRound` are all silently discarded. No check is made that `updatedAt != 0` (incomplete round) or that `answeredInRound >= roundId` (stale round). The same repository already contains the correct pattern in `ChainlinkOracleForRSETHPoolCollateral.getRate()` (L30–32):

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The stale price flows directly into `LRTDepositPool.getRsETHAmountToMint()` (L519–520):

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

This is called by `_beforeDeposit()` (L665), which is invoked by the public `depositAsset()` (L99–118). The `minRSETHAmountExpected` slippage guard (L667) only protects the depositor from receiving *less* than expected; it does not prevent receiving *more* due to a stale inflated price. No other guard in the deposit path validates oracle freshness.

`_getTotalEthInProtocol()` in `LRTOracle` also calls `getAssetPrice()` through the same path, meaning `rsETHPrice` is also computed from the stale feed. The attack window is the period between the feed going stale (at an inflated price) and the feed updating to the true lower price.

## Impact Explanation
**Critical — Direct theft of user funds at rest.**

An attacker deposits an LST whose Chainlink feed is stale at an inflated price. They receive rsETH computed against the old high price, representing a larger share of protocol TVL than the assets they contributed. When the feed updates to the true lower price and `updateRSETHPrice()` is called, the recalculated `rsETHPrice` is lower, and every existing rsETH holder's position is worth less. The attacker's excess rsETH is redeemable at the direct expense of existing holders — a concrete, quantifiable theft of funds at rest.

## Likelihood Explanation
**Medium.** Chainlink feeds have documented heartbeat intervals (1–24 hours depending on the feed and network). L2 sequencer outages, network congestion, and oracle node downtime are known, recurring events that cause feeds to miss heartbeats. The protocol is deployed across multiple L2 networks where sequencer downtime is a documented risk. No special permissions are required — any external account can call `depositAsset()`. The condition (stale feed at an inflated price) is realistic and has occurred historically.

## Recommendation
Apply the same staleness and completeness checks already present in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Additionally, add a configurable per-asset `maxStaleness` threshold (`block.timestamp - updatedAt > maxStaleness`) since heartbeat intervals differ across feeds and networks.

## Proof of Concept
**Actors:** Alice (existing rsETH holder, deposited 1 ETH), Bob (attacker).

**Preconditions:**
- stETH/ETH Chainlink feed last updated 4+ hours ago at `1.001e18` (stETH at slight premium).
- True market price of stETH has dropped to `0.95e18` (depeg event).
- `rsETHPrice` = `1.001e18` (last `updateRSETHPrice()` call used the same stale feed).

**Exploit steps:**
1. Bob observes the Chainlink feed is stale (heartbeat missed).
2. Bob calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
3. `getRsETHAmountToMint(stETH, 100e18)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` returns stale `1.001e18`.
4. `rsethAmountToMint = (100e18 * 1.001e18) / 1.001e18 = 100e18` rsETH minted to Bob.
5. Bob's 100 stETH is worth only 95 ETH at true market price; he received rsETH representing 100 ETH of protocol claims.
6. Chainlink feed updates; `updateRSETHPrice()` is called; `_getTotalEthInProtocol()` now prices stETH at `0.95e18`, reducing `rsETHPrice` for all holders.
7. Alice's rsETH is now worth less than 1 ETH. Bob extracted ~5 ETH of value from existing holders.

**Foundry fork test plan:** Fork mainnet/L2 at a block where a target LST Chainlink feed is stale. Deploy or reference the existing `ChainlinkPriceOracle`. Call `getAssetPrice(stETH)` and confirm it returns the stale price without reverting. Then simulate `depositAsset()` and verify the minted rsETH exceeds fair value. Warp time forward, call `updateRSETHPrice()`, and assert `rsETHPrice` has decreased, confirming dilution of pre-existing holders.
