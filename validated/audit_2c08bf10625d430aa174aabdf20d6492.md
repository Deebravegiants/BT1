### Title
Stale `drepDelegs` Entry Causes Silent Erasure of Active DRep Delegation on DRep Unregistration — (`File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Deleg.hs` + `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/GovCert.hs`)

---

### Summary

During the Conway bootstrap phase (protocol version < 10), `processDelegationInternal` intentionally preserves a buggy behavior (issue #4772) where re-delegating a stake credential from DRep A to DRep B only inserts the credential into DRep B's `drepDelegs` reverse-delegation set but **never removes it from DRep A's `drepDelegs`**. When DRep A subsequently unregisters, `ConwayUnRegDRep` in `GovCert.hs` iterates over DRep A's stale `drepDelegs` set and clears the `dRepDelegationAccountStateL` for every credential listed there — including credentials that have already re-delegated to DRep B. This silently destroys those credentials' active DRep delegations, removing their stake from DRep B's voting power without their knowledge or consent.

---

### Finding Description

**Root cause — wrong key retained in `drepDelegs` during re-delegation (bootstrap phase):**

In `processDelegationInternal`, when `preserveIncorrectDelegation` is `True` (i.e., `pvMajor pv < natVersion @10`) and the new delegatee is a credential-based DRep, the `delegVote` branch only inserts the stake credential into the new DRep's `drepDelegs` set. It never calls `unDelegReDelegDRep` to remove the credential from the old DRep's `drepDelegs` set:

```haskell
-- Deleg.hs lines 363-365
| isNothing mAccountState || preserveIncorrectDelegation ->
    certVStateL . vsDRepsL
      %~ Map.adjust (drepDelegsL %~ Set.insert stakeCred) dRepCred
```

The account state's `dRepDelegationAccountStateL` is correctly updated to point to DRep B (line 376), but DRep A's `drepDelegs` set still contains the stake credential.

**Impact trigger — `ConwayUnRegDRep` uses the stale `drepDelegs` to clear delegations:**

When DRep A unregisters, `GovCert.hs` lines 244–254 iterate over `drepDelegs dRepState` and set `dRepDelegationAccountStateL .~ Nothing` for every credential in that set:

```haskell
-- GovCert.hs lines 246-254
let
  clearDRepDelegations delegs accountsMap =
    foldr (Map.adjust (dRepDelegationAccountStateL .~ Nothing)) accountsMap delegs
pure $
  case mDRepState of
    ...
    Just dRepState ->
      certState'
        & certDStateL . accountsL . accountsMapL
          %~ clearDRepDelegations (drepDelegs dRepState)
```

Because DRep A's `drepDelegs` still contains credential C (which has since re-delegated to DRep B), `clearDRepDelegations` sets C's `dRepDelegationAccountStateL` to `Nothing`. C's account state now shows no DRep delegation at all, even though C intended to delegate to DRep B.

**Effect on voting power:**

`computeDRepDistr` in `DRepPulser.hs` (lines 228–241) computes DRep voting power from `accountState ^. dRepDelegationAccountStateL`. After the erasure, C's stake is no longer attributed to DRep B (or any DRep), silently reducing DRep B's voting power.

---

### Impact Explanation

An attacker who registers a DRep, attracts delegators, and then unregisters after those delegators have re-delegated to other DReps will silently zero out those delegators' voting power. This directly manipulates the DRep stake distribution used in `dRepAcceptedRatio` (Ratify.hs lines 258–281) to determine whether governance actions pass. Governance actions that require DRep approval (protocol parameter changes, treasury withdrawals, hard-fork initiations, committee updates, constitutional changes) can be made to fail or pass by artificially deflating the voting power of targeted DReps. This constitutes unauthorized influence over governance action ratification.

---

### Likelihood Explanation

The attack is executable by any unprivileged transaction sender during the Conway bootstrap phase (PV9). The attacker needs only to:
1. Register a DRep (pay the DRep deposit).
2. Attract at least one delegator (or wait for organic delegation).
3. Wait for that delegator to re-delegate to a different DRep.
4. Submit a `ConwayUnRegDRep` certificate to unregister.

No privileged access, key compromise, or supermajority is required. The bootstrap phase is a live, extended period on mainnet before the PV10 hard fork. The `updateDRepDelegations` cleanup only runs at the hard fork transition, so the window of exploitability spans the entire bootstrap phase.

---

### Recommendation

In `processDelegationInternal`, the `preserveIncorrectDelegation` guard should not be applied when the account already exists and has a prior DRep delegation. Specifically, when `mAccountState` is `Just accountState` (i.e., the credential is already registered and has a prior delegation), `unDelegReDelegDRep` must always be called to remove the credential from the old DRep's `drepDelegs`, regardless of the protocol version. The `preserveIncorrectDelegation` path (lines 363–365) should be restricted to the case where `mAccountState` is `Nothing` only, never to the re-delegation case. Alternatively, `ConwayUnRegDRep` should cross-check each credential in `drepDelegs` against the account state before clearing it, only clearing the delegation if the account state still points to the unregistering DRep.

---

### Proof of Concept

1. Bootstrap phase is active (`pvMajor pv < natVersion @10`, i.e., Conway PV9).
2. Attacker registers DRep A.
3. Victim stake credential C registers and delegates to DRep A via `ConwayRegDelegCert`.
   - C's account state: `dRepDelegationAccountStateL = Just (DRepCredential A)`
   - DRep A's `drepDelegs`: `{C}`
4. Victim re-delegates C to DRep B via `ConwayDelegCert C (DelegVote (DRepCredential B))`.
   - `processDelegationInternal` is called with `preserveIncorrectDelegation = True`.
   - C's account state: `dRepDelegationAccountStateL = Just (DRepCredential B)` ✓
   - DRep B's `drepDelegs`: `{C}` ✓
   - DRep A's `drepDelegs`: **`{C}` (stale, not removed)** ✗
5. Attacker submits `ConwayUnRegDRep A refund`.
   - `clearDRepDelegations (drepDelegs dRepStateA)` iterates over `{C}`.
   - Sets C's `dRepDelegationAccountStateL .~ Nothing`.
   - C's account state: `dRepDelegationAccountStateL = Nothing` ✗
6. `computeDRepDistr` now finds `Nothing` for C's DRep delegation; C's stake is not added to DRep B's distribution.
7. DRep B's effective voting power is reduced by C's stake without C's knowledge. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```haskell
  ConwayEraCertState era =>
  -- | Preserve the buggy behavior where DRep delegations are not updated correctly (See #4772)
  Bool ->
  -- | Delegator
  Credential Staking ->
  -- | Account state for the above stake credential
  Maybe (AccountState era) ->
  -- | New delegatee
  Delegatee ->
  CertState era ->
  CertState era
processDelegationInternal preserveIncorrectDelegation stakeCred mAccountState newDelegatee =
  case newDelegatee of
    DelegStake sPool -> delegStake sPool
    DelegVote dRep -> delegVote dRep
    DelegStakeVote sPool dRep -> delegVote dRep . delegStake sPool
  where
    delegStake stakePool cState =
      cState
        & certDStateL . accountsL
          %~ adjustAccountState (stakePoolDelegationAccountStateL ?~ stakePool) stakeCred
        & maybe
          (certPStateL . psStakePoolsL %~ Map.adjust (spsDelegatorsL %~ Set.insert stakeCred) stakePool)
          (\accountState -> certPStateL %~ unDelegReDelegStakePool stakeCred accountState (Just stakePool))
          mAccountState

    delegVote dRep cState =
      let handleReverseDelegation =
            case dRepToCred dRep of
              Just dRepCred
                -- This is the case where we only add the new reverse delegation and do not remove
                -- the old one, which is the behavior that we want:
                --
                -- 1) for new accounts, since there is no old reverse delegation to remove
                --
                -- 2) in the bootstrap phase, in order to preserve the incorrect behavior, where old reverse
                --   delegation for the prior DRep was wrongfully retained. It is important to note
                --   that in case when the new delegation was to a predefined DRep, the reverse
                --   delegations where handled correctly even in the boostrap phase
                --
                -- For reference here is the original bug report:
                --   https://github.com/IntersectMBO/cardano-ledger/issues/4772
                | isNothing mAccountState || preserveIncorrectDelegation ->
                    certVStateL . vsDRepsL
                      %~ Map.adjust (drepDelegsL %~ Set.insert stakeCred) dRepCred
```

```haskell
dRepAcceptedRatio ::
  forall era.
  RatifyEnv era ->
  Map (Credential DRepRole) Vote ->
  GovAction era ->
  Rational
dRepAcceptedRatio RatifyEnv {reDRepDistr, reDRepState, reCurrentEpoch} gasDRepVotes govAction =
  toInteger yesStake %? toInteger totalExcludingAbstainStake
  where
    accumStake (!yes, !tot) drep (CompactCoin stake) =
      case drep of
        DRepCredential cred ->
          case Map.lookup cred reDRepState of
            Nothing -> (yes, tot) -- drep is not registered, so we don't consider it
            Just drepState
              | reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, so we don't consider it
              | otherwise ->
                  case Map.lookup cred gasDRepVotes of
                    -- drep hasn't voted for this action, so we don't count
                    -- the vote but we consider it in the denominator:
                    Nothing -> (yes, tot + stake)
                    Just VoteYes -> (yes + stake, tot + stake)
                    Just Abstain -> (yes, tot)
                    Just VoteNo -> (yes, tot + stake)
        DRepAlwaysNoConfidence ->
          case govAction of
            NoConfidence _ -> (yes + stake, tot + stake)
            _ -> (yes, tot + stake)
        DRepAlwaysAbstain -> (yes, tot)
    (yesStake, totalExcludingAbstainStake) = Map.foldlWithKey' accumStake (0, 0) reDRepDistr
```
