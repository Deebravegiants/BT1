After thorough investigation of the Cardano Ledger codebase for an analog to the Nayms double-counting dividend bug, I found a directly analogous issue in the Dijkstra era's sub-transaction accounting.

---

### Title
Double-Counting of Deposit Refunds Across Sub-Transactions Enables ADA Creation - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

### Summary

In the Dijkstra era, `getConsumedDijkstraValue` computes the total consumed value for a top-level transaction by summing the consumed values of the top-level body and all sub-transaction bodies using the **same initial** `lookupStakingDeposit` / `lookupDRepDeposit` snapshot. If two sub-transactions each contain an `UnRegDepositTxCert` (or `UnRegDRepTxCert`) for the **same credential**, the deposit refund is counted twice in the consumed-value total. A disabled test in the repository (`xit "Multiple subtransactions cannot get the same refund"`) explicitly acknowledges this scenario is unprotected, with the comment `error "TODO: predicate failure not yet implemented"`, indicating the guard has not been implemented.

### Finding Description

**Root cause — `getConsumedDijkstraValue`** in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`:

```haskell
subTransactionsConsumedValue topTxBody =
  foldMap'
    (getConsumedValue pp lookupStakingDeposit lookupDRepDeposit utxo . view bodyTxL)
    (topTxBody ^. subTransactionsTxBodyL)
``` [1](#0-0) 

The same `lookupStakingDeposit` closure — which reflects the ledger state **before** any sub-transaction is applied — is passed to every sub-transaction's consumed-value calculation. There is no inter-sub-transaction state threading: each sub-transaction independently queries the same pre-execution deposit map.

**Contrast with the correctly-deduplicated pool-registration deposit logic** in `shelleyTotalDepositsTxCerts`:

```haskell
numNewRegPoolCerts = Set.size (F.foldl' addNewPoolIds Set.empty certs)
addNewPoolIds regPoolIds = \case
  RegPoolTxCert (StakePoolParams {sppId})
    | not (isRegPoolRegistered sppId || Set.member sppId regPoolIds) -> Set.insert sppId regPoolIds
  _ -> regPoolIds
``` [2](#0-1) 

Pool registrations are explicitly deduplicated across the certificate list. No equivalent deduplication exists for deposit refunds across sub-transactions.

**The DRep deposit counting function** `conwayDRepDepositsTxCerts` also counts every `RegDRepTxCert` occurrence without deduplication:

```haskell
conwayDRepDepositsTxCerts pp certs = nDReps <×> depositPerDRep
  where
    nDReps = getSum @Int (foldMap' (\case RegDRepTxCert {} -> 1; _ -> 0) certs)
``` [3](#0-2) 

**The unimplemented guard** — the test file `eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs` contains:

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
  ...
  subTx1 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
  subTx2 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
  tx = mkBasicTx mkBasicTxBody
         & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
  submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
``` [4](#0-3) 

The `xit` prefix disables the test. The body uses `submitFailingTx` (which asserts the transaction **must** fail) but supplies `error "TODO: predicate failure not yet implemented"` as the expected failure — meaning the specific rejection predicate does not yet exist. The most natural reading is that the transaction currently **succeeds** when it should fail, which is why the test is disabled rather than simply updated with the correct failure constructor.

### Impact Explanation

If the value-conservation check (`consumed = produced`) is evaluated using the inflated consumed value (deposit `D` counted twice = `2D`), an attacker can craft outputs absorbing the extra `D` lovelace. If the sub-transaction processing also uses the pre-execution state snapshot (consistent with how `getConsumedDijkstraValue` works), both sub-transactions' certificate processing would see the credential as still registered, allowing both `UnRegDepositTxCert` certificates to succeed. The net result is that `D` lovelace is created from nothing — a direct, attacker-controlled increase in circulating ADA supply through an invalid ledger state transition.

**Matched impact**: *Critical — Direct creation of ADA through an invalid ledger state transition.*

### Likelihood Explanation

The Dijkstra era is the newest era in the codebase and sub-transactions are a novel feature. The attack requires only:
1. A registered staking credential (anyone can register one for a small deposit).
2. A top-level Dijkstra transaction with two sub-transactions, each containing `UnRegDepositTxCert` for the same credential.

No privileged access, governance majority, or leaked key is required. The attacker is an ordinary transaction sender. The `xit` test confirms the developers are aware the protection is missing.

### Recommendation

1. **Thread state across sub-transactions** in `getConsumedDijkstraValue`: after computing the consumed value for each sub-transaction, update the `lookupStakingDeposit` / `lookupDRepDeposit` closures to reflect credentials already claimed for refund in earlier sub-transactions.
2. **Implement the missing predicate failure** for "multiple sub-transactions claiming the same deposit refund" and re-enable the `xit` test.
3. **Apply the same deduplication pattern** used for pool registrations in `shelleyTotalDepositsTxCerts` to DRep deposit counting in `conwayDRepDepositsTxCerts` when sub-transactions are involved.

### Proof of Concept

1. Register staking credential `cred` with deposit `D` lovelace.
2. Construct `subTx1` with `UnRegDepositTxCert cred D`.
3. Construct `subTx2` with `UnRegDepositTxCert cred D` (same credential, same deposit).
4. Construct top-level Dijkstra transaction `tx` containing `[subTx1, subTx2]` as sub-transactions.
5. `getConsumedDijkstraValue` evaluates `lookupStakingDeposit cred` for both sub-transactions against the same pre-execution state, returning `D` both times → total consumed value is inflated by `D`.
6. Set `tx`'s outputs to absorb the inflated consumed value (i.e., include an extra output of `D` lovelace).
7. If the value-conservation check passes with the inflated consumed value and certificate processing uses the same pre-execution snapshot, `tx` is accepted and `D` lovelace has been created from nothing.
