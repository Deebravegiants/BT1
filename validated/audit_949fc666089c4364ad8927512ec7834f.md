Looking at the vulnerability class — a code path that silently short-circuits and prevents a legitimate operation from completing when one branch of a dual-path system is unimplemented — I searched the NEAR MPC codebase for an analog.

The most direct analog is in the `execute_foreign_chain_request` function in the node's foreign-tx verification provider.

---

### Title
Ton Chain Verification Requests Permanently Unserviceable Despite Contract Acceptance — (File: `crates/node/src/providers/verify_foreign_tx/sign.rs`)

### Summary
The MPC node's `execute_foreign_chain_request` unconditionally bails on `ForeignChainRpcRequest::Ton` while the contract interface accepts Ton as a valid chain variant and the sandbox test suite includes Ton as an expected-to-succeed case. If Ton is registered as a supported chain by enough nodes to meet the signing threshold, users can submit `verify_foreign_transaction` requests that the contract accepts (consuming the deposit), but every MPC node will bail — the requests will always time out and never be fulfilled.

### Finding Description

In `crates/node/src/providers/verify_foreign_tx/sign.rs`, the `execute_foreign_chain_request` function dispatches on each `ForeignChainRpcRequest` variant. For the Ton variant, it unconditionally bails: [1](#0-0) 

Meanwhile, three facts hold simultaneously:

1. `ForeignChainRpcRequest::Ton` is a valid enum variant in the contract interface — the contract accepts it as a legitimate chain type.
2. The contract's `verify_foreign_transaction` accepts Ton requests if Ton is in the supported-chains set, consuming the user's deposit. [2](#0-1) 

3. The sandbox test suite explicitly includes Ton as an expected-to-succeed case: [3](#0-2) 

The supported-chains set is computed by `update_available_chains_config_cache`: a chain becomes "available" once ≥ `signing_threshold` active nodes register it via `register_available_foreign_chain_config`. [4](#0-3) 

If operators follow the test-suite setup and register Ton in their local config, Ton crosses the availability threshold. From that point, every user-submitted Ton verification request passes the contract's chain-support check, is queued with a yield, and then silently fails at every node — producing no signature shares and timing out.

The structural parallel to the reported Notional bug is exact: just as `_claimVaultRewards` returns early with `if (rewardPool.rewardPool == address(0)) return;` when the Gauge branch is used, `execute_foreign_chain_request` bails immediately for the Ton branch, leaving the request lifecycle permanently broken for that chain family.

### Impact Explanation

This is a request-lifecycle safety-invariant violation. The contract's design guarantee is: a request that passes the `supported_chains` check **can** be fulfilled by the network. For Ton, that guarantee is broken — every accepted Ton request will time out. Users lose their deposit and receive no response. The broken invariant is not surfaced to the caller; the request simply expires.

Impact: **Medium** — request-lifecycle manipulation that breaks a production safety invariant without requiring network-level DoS or privileged operator access.

### Likelihood Explanation

Likelihood is conditional on Ton being registered as available. The sandbox test suite treats Ton as a first-class supported chain (line 32 of `foreign_chain_request.rs`), so operators following the documented test setup would naturally include Ton in their local config. Once ≥ `signing_threshold` nodes register Ton, the chain becomes available and any unprivileged user can trigger the broken path by submitting a Ton verification request.

### Recommendation

Either:
1. Implement a Ton chain inspector in the node (add a `ton` inspector to `ForeignChainInspectors` and handle `ForeignChainRpcRequest::Ton` properly), or
2. Prevent Ton from ever entering the available-chains set by rejecting Ton registrations at the contract level, so the contract's `verify_foreign_transaction` rejects Ton requests before they are queued.

### Proof of Concept

1. Operators configure Ton in their local foreign-chain config and call `register_available_foreign_chain_config`.
2. Once ≥ `signing_threshold` nodes register Ton, `update_available_chains_config_cache` marks Ton as available.
3. An unprivileged user calls `verify_foreign_transaction` with `ForeignChainRpcRequest::Ton` and attaches the required deposit.
4. The contract's chain-support check passes; the request is enqueued with a yield.
5. Every MPC node receives the request, calls `execute_foreign_chain_request`, and hits the `bail!("ForeignChainRpcRequest::Ton is unsupported")` branch.
6. No node produces a signature share; the yield times out.
7. `return_verify_foreign_tx_and_clean_state_on_success` fires `fail_on_timeout`, the user's transaction fails, and the deposit is consumed — with no Ton verification result ever returned. [1](#0-0) [5](#0-4)

    pub(crate) fn update_available_chains_config_cache(
        &mut self,
        active_tls_keys: &BTreeSet<dtos::Ed25519PublicKey>,
        threshold: u64,
    ) {
        let mut chain_to_supporter_count: std::collections::BTreeMap<dtos::ForeignChain, u64> =
            std::collections::BTreeMap::new();
        for tls_key in active_tls_keys {
            let Some(chains) = self.foreign_chains_configs.get(tls_key) else {
                continue;
            };
            for chain in chains.iter() {
                if self.rpc_whitelist.entries.is_whitelisted(chain) {
                    let count = chain_to_supporter_count.entry(*chain).or_default();
                    *count = count
                        .checked_add(1)
                        .expect("supporter count bounded by participant set size");
                }
            }
        }
        self.available_foreign_chains = chain_to_supporter_count
            .into_iter()
            .filter_map(|(chain, count)| (count >= threshold).then_some(chain))
            .collect::<BTreeSet<_>>()
            .into();
