### Title
Byzantine Party Sends Equivocating `sid_i` / `c` / `sid_msg` / `h` Values Across Honest Parties in MP-DKG, Causing Inconsistent `Qi` Acceptance - (`src/cbmpc/protocol/ec_dkg.cpp`)

---

### Summary

`key_share_mp_t::dkg` uses `plain_broadcast`, which is implemented as point-to-point sends with no broadcast-consistency guarantee. A Byzantine party `j` can send different `(sid_i, c)` in round 1 and different `(sid_msg, h, Qi, rho, pi)` in round 2 to each honest peer. Because the two consistency guards (`sid_msg._j != sid` and `h._j != h.msg`) are themselves received over the same unauthenticated point-to-point channel, party `j` can tailor each guard value to match the locally-computed expectation of each honest party. Both honest parties pass all checks and accept different `Qi_j` values, producing inconsistent key shares and different public keys.

---

### Finding Description

**Transport model.** `plain_broadcast` calls `group_message(party_set_t::all(), party_set_t::all(), msgs...)`, which calls `send_to_parties` / `receive_from_parties` via `send_impl` → `transport_raw->send(to, msg)`. This is purely point-to-point; there is no broadcast-consistency enforcement. A Byzantine party can send different serialized bytes to each recipient. [1](#0-0) [2](#0-1) 

**Round 1 equivocation.** Party `j` broadcasts `(sid_i, c, h_consistency)`. Because the channel is point-to-point, party `j` sends `(A, c_A, ...)` to party 0 and `(B, c_B, ...)` to party 2. [3](#0-2) 

**Divergent local `sid` and `h`.** After round 1, each honest party independently computes:

```
sid_0 = SHA256(sid_i_0, A, sid_i_2, ...)   // party 0's view
sid_2 = SHA256(sid_i_0, B, sid_i_2, ...)   // party 2's view

h_0   = SHA256(c_0, c_A, c_2, ...)
h_2   = SHA256(c_0, c_B, c_2, ...)
``` [4](#0-3) 

**Round 2 equivocation.** Party `j` sends `(sid_msg=sid_0, h=h_0, Qi_A, rho_A, pi_A)` to party 0 and `(sid_msg=sid_2, h=h_2, Qi_B, rho_B, pi_B)` to party 2. [5](#0-4) 

**Guards bypassed.** The two intended consistency checks compare received values against locally-computed values:

```cpp
if (sid_msg._j != sid)   return error(E_CRYPTO);   // line 152
if (h._j != h.msg)       return error(E_CRYPTO);   // line 153
```

Party `j` sends exactly `sid_0` / `h_0` to party 0 and `sid_2` / `h_2` to party 2, so both comparisons evaluate to `false` (no error) at each honest party. [6](#0-5) 

**Commitment and ZK proof pass.** Party `j` pre-computes two independent, valid commitment/proof pairs:

- `commitment_t(A, pid_j).gen(Qi_A)` → `(c_A, rho_A)`, and `pi_A.prove(Qi_A, x_A, sid_0, j)`
- `commitment_t(B, pid_j).gen(Qi_B)` → `(c_B, rho_B)`, and `pi_B.prove(Qi_B, x_B, sid_2, j)`

Both `commitment_t(sid_i._j, pid_j).set(rho._j, c._j).open(Qi._j)` and `pi._j.verify(Qi._j, sid, j)` succeed at each honest party. [7](#0-6) [8](#0-7) 

**Inconsistent output.** Each honest party stores the `Qi_j` it received and sums all shares:

```
Q_0 = Qi_0 + Qi_A + Qi_2 + ...
Q_2 = Qi_0 + Qi_B + Qi_2 + ...
``` [9](#0-8) 

The two honest parties complete DKG with different public keys and inconsistent key-share sets.

---

### Impact Explanation

The Byzantine party causes honest parties to accept different `Qi_j` values for party `j`, producing divergent `Q` (the joint public key). Subsequent signing rounds will use different public keys, causing honest parties to disagree on signature validity. More critically, the Byzantine party knows both `x_A` and `x_B` (the discrete logs of `Qi_A` and `Qi_B`), giving it full knowledge of the "contribution" each honest party believes party `j` made. Depending on the signing protocol, this can be leveraged to bias or fully control the resulting key material. This meets the **High** bar: attacker-controlled session data is accepted under the wrong transcript, causing honest-party divergence and invalid cryptographic output.

---

### Likelihood Explanation

Any Byzantine transport peer participating in the MP-DKG protocol can execute this attack with no special privileges. It requires only the ability to send different serialized bytes to different parties in two broadcast rounds, which is trivially achievable over any point-to-point channel. No threshold collusion is required; a single Byzantine party suffices.

---

### Recommendation

The protocol must enforce **broadcast consistency** for `plain_broadcast` rounds. Standard approaches:

1. **Echo-broadcast / Dolev-Strong**: after receiving a message from party `j`, each honest party re-broadcasts a hash of what it received; parties abort if they observe disagreement.
2. **Commit-then-reveal with cross-party hash check**: before using `sid_i._j` in the commitment, all parties exchange and agree on a hash of the full `sid_i` vector (similar to the `h_consistency` pattern already used for the curve name, but applied to `sid_i` and `c` values).
3. **Replace `plain_broadcast` with `committed_group_broadcast`** for the first round, which already includes a cross-party hash check (`v` check at line 72 of `committed_broadcast.h`) that would detect equivocation. [10](#0-9)

### Proof of Concept

Three-party simulation (parties 0, 1, 2); party 1 is Byzantine.

**Setup**: Party 1 chooses scalars `x_A`, `x_B`, computes `Qi_A = x_A·G`, `Qi_B = x_B·G`.

**Round 1** — party 1 sends:
- To party 0: `sid_i = A` (random), `c = commitment_t(A, pid_1).gen(Qi_A).msg`
- To party 2: `sid_i = B` (different random), `c = commitment_t(B, pid_1).gen(Qi_B).msg`
- `h_consistency` = correct curve hash (same to both)

**After round 1**:
- Party 0 computes `sid_0 = SHA256(sid_0_own, A, sid_2_own)`, `h_0 = SHA256(c_0, c_A, c_2)`
- Party 2 computes `sid_2 = SHA256(sid_0_own, B, sid_2_own)`, `h_2 = SHA256(c_0, c_B, c_2)`

**Round 2** — party 1 sends:
- To party 0: `sid_msg=sid_0`, `h=h_0`, `Qi=Qi_A`, `rho=rho_A`, `pi=pi_A` (proof of `Qi_A` under `sid_0`)
- To party 2: `sid_msg=sid_2`, `h=h_2`, `Qi=Qi_B`, `rho=rho_B`, `pi=pi_B` (proof of `Qi_B` under `sid_2`)

**Result**: Both honest parties pass all checks. Party 0 stores `key.Qis[name_1] = Qi_A`; party 2 stores `key.Qis[name_1] = Qi_B`. Assert `Q_0 ≠ Q_2` — the assertion holds, confirming inconsistent DKG output accepted by both honest parties.

template <typename... ARGS>
inline error_t committed_group_broadcast(job_mp_t& job, ARGS&... args) {
  error_t rv = UNINITIALIZED_ERROR;

  const crypto::mpc_pid_t& pid = job.get_pid();
  coinbase::crypto::commitment_t com(pid);
  // effectively, it does com.gen(args.msg...)
  std::apply([&com](auto&&... args) { com.gen(args...); },
             map_args_to_tuple([](auto& arg) { return arg.msg; }, args...));

  auto com_rand = job.uniform_msg<buf256_t>(com.rand);
  auto com_msg = job.uniform_msg<buf_t>(com.msg);

  if (rv = job.plain_broadcast(com_msg)) return rv;

  auto v = job.uniform_msg<buf256_t>(crypto::sha256_t::hash(com_msg.all_received()));

  if (rv = job.plain_broadcast(v, com_rand, args...)) return rv;

  for (int i = 0; i < job.get_n_parties(); i++) {
    if (v.msg != v.received(i)) return rv = job.mpc_abort(E_CRYPTO, "received hash mismatch");

    coinbase::crypto::commitment_t com(job.get_pid(i));
    com.set(com_rand.received(i), com_msg.received(i));
    rv = std::apply([&com](auto&&... args) { return com.open(args...); },
                    map_args_to_tuple([i](auto& arg) { return arg.received(i); }, args...));
    if (rv) return rv;
  }

  return SUCCESS;
}
