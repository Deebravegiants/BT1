### Title
Unchecked `sub_epoch_n` Index Into `summaries` List Causes Unhandled `IndexError` During Weight Proof Validation - (File: `chia/full_node/weight_proof.py`)

### Summary
In `_validate_sub_epoch_segments`, the `summaries` list is sized by the number of sub-epoch summaries present in the weight proof (`weight_proof.sub_epochs`), but it is indexed using `sub_epoch_n` values taken directly from attacker-controlled `SubEpochChallengeSegment` data. A malicious peer can craft a `WeightProof` where a segment's `sub_epoch_n` is greater than or equal to `len(summaries)`, triggering an unhandled `IndexError` that crashes the sync validation path for both full nodes and wallets.

### Finding Description

In `_validate_sub_epoch_segments` (`chia/full_node/weight_proof.py`):

```python
summaries = summaries_from_bytes(summaries_bytes)          # len = len(weight_proof.sub_epochs)
...
segments_by_sub_epoch = map_segments_by_sub_epoch(sub_epoch_segments.challenge_segments)
...
for sub_epoch_n, segments in segments_by_sub_epoch.items():   # sub_epoch_n from attacker data
    ...
    if sub_epoch_n > 0:
        prev_ses = summaries[sub_epoch_n - 1]              # IndexError if sub_epoch_n >= len(summaries)
        ...
    if not summaries[sub_epoch_n].reward_chain_hash == ...: # IndexError if sub_epoch_n >= len(summaries)
``` [1](#0-0) 

The `summaries` list is built from `weight_proof.sub_epochs` and has a fixed length equal to the number of sub-epoch summaries in the proof. The `sub_epoch_n` values, however, come from `SubEpochChallengeSegment.sub_epoch_n` fields inside `weight_proof.sub_epoch_segments`, which are parsed directly from attacker-supplied bytes with no upper-bound validation before the index access.

`map_segments_by_sub_epoch` simply groups segments by their `sub_epoch_n` field without any bounds check: [2](#0-1) 

The only guard present checks `len(segments) > max_segments` (segment count per sub-epoch), not whether `sub_epoch_n` is within `[0, len(summaries))`: [3](#0-2) 

The same unchecked index also appears inside `__get_rc_sub_slot`, called from the same loop: [4](#0-3) 

### Impact Explanation

`_validate_sub_epoch_segments` is called from two paths with no surrounding `try/except`:

1. `WeightProofHandler.validate_weight_proof_single_proc` → called during full-node sync
2. `validate_weight_proof_inner` → called from both `WeightProofHandler.validate_weight_proof` and `WalletWeightProofHandler.validate_weight_proof` [5](#0-4) [6](#0-5) 

An unhandled `IndexError` propagates up through the sync machinery. A malicious peer that serves a crafted `WeightProof` can therefore cause a **permanent or long-lived inability for honest nodes and wallets to complete sync**, matching the High impact criterion.

### Likelihood Explanation

Any unprivileged peer on the network can serve a `WeightProof` response during the sync handshake. Crafting a proof with a `SubEpochChallengeSegment` whose `sub_epoch_n` exceeds the number of entries in `sub_epochs` requires only knowledge of the serialization format, which is public. No keys, admin access, or cryptographic breaks are needed.

### Recommendation

Add an explicit bounds check on `sub_epoch_n` before accessing `summaries` inside `_validate_sub_epoch_segments`:

```python
for sub_epoch_n, segments in segments_by_sub_epoch.items():
    if sub_epoch_n >= len(summaries):
        log.error(f"sub_epoch_n {sub_epoch_n} out of range for summaries (len={len(summaries)})")
        return None
    if len(segments) > max_segments:
        ...
```

Apply the same guard in `__get_rc_sub_slot` before `summaries[uint32(segment.sub_epoch_n - 1)]`.

### Proof of Concept

A malicious peer constructs a `WeightProof` where `weight_proof.sub_epochs` contains N entries (so `len(summaries) == N`) but `weight_proof.sub_epoch_segments` contains a `SubEpochChallengeSegment` with `sub_epoch_n = N` (or any value ≥ N). When the victim node calls `validate_weight_proof` on this proof, `_validate_sub_epoch_segments` reaches `summaries[sub_epoch_n]` with `sub_epoch_n == N`, raising `IndexError: list index out of range`, which is unhandled and crashes the sync validation task. def map_segments_by_sub_epoch(
    sub_epoch_segments: list[SubEpochChallengeSegment],
) -> dict[int, list[SubEpochChallengeSegment]]:
    segments: dict[int, list[SubEpochChallengeSegment]] = {}
    curr_sub_epoch_n = -1
    for idx, segment in enumerate(sub_epoch_segments):
        if curr_sub_epoch_n < segment.sub_epoch_n:
            curr_sub_epoch_n = segment.sub_epoch_n
            segments[curr_sub_epoch_n] = []
        segments[curr_sub_epoch_n].append(segment)
    return segments
```
