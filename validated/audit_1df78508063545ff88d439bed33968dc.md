### Title
Griefing Attack Blocks `removeNodeDelegatorContractFromQueue` via Dust Token Transfer - (File: contracts/LRTDepositPool.sol)

### Summary
The `_removeNodeDelegatorContractFromQueue` internal function performs balance checks on the `NodeDelegator` before removing it from the queue. Because these checks rely on live `balanceOf` and native ETH balance reads, any unprivileged external party can permanently block the removal by transferring a dust amount of a supported LST token (or ETH) to the `NodeDelegator` address, causing the checks to revert indefinitely.

### Finding Description
`_removeNodeDelegatorContractFromQueue` (called by both `removeNodeDelegatorContractFromQueue` and `removeManyNodeDelegatorContractsFromQueue`) performs two residue checks before executing the removal:

**Check 1 — ETH balance** (`_checkResidueEthBalance`, line 619):
```solidity
address(nodeDelegatorAddress).balance > maxNegligibleAmount
```

**Check 2 — LST balance** (`_checkResidueLSTBalance`, line 638):
```solidity
assetBalance = IERC20(supportedAssets[i]).balanceOf(nodeDelegatorAddress)
             + INodeDelegator(nodeDelegatorAddress).getAssetBalance(supportedAssets[i]);
assetBalance += INodeDelegator(nodeDelegatorAddress).getAssetUnstaking(supportedAssets[i]);

if (assetBalance > maxNegligibleAmount) {
    revert NodeDelegatorHasAssetBalance(supportedAssets[i], assetBalance);
}
```

`maxNegligibleAmount` is declared as a storage variable but is **never initialized** in `initialize()`, so it defaults to `0`. [1](#0-0) 

With `maxNegligibleAmount == 0`, any non-zero `balanceOf` on the `NodeDelegator` causes a revert. Since ERC-20 `transfer` is permissionless, any external caller can send 1 wei of any supported LST token directly to the `NodeDelegator` address at any time. [2](#0-1) 

Similarly, `NodeDelegator` exposes a `receive()` function (confirmed present in `NodeDelegator.sol`), so 1 wei of ETH can also be force-sent to trigger `_checkResidueEthBalance`. [3](#0-2) 

### Impact Explanation
The admin is permanently unable to call `removeNodeDelegatorContractFromQueue` or `removeManyNodeDelegatorContractsFromQueue` for the targeted `NodeDelegator` as long as the attacker maintains even 1 wei of balance there. This blocks the protocol's ability to rotate out a compromised or misbehaving `NodeDelegator`, which is a security-critical operation. The impact maps to **Medium — Temporary freezing of funds** (the NodeDelegator and its associated assets cannot be dequeued and isolated). [4](#0-3) 

### Likelihood Explanation
The attack requires no special privileges, no flash loan, and no timing precision. Any external account holding 1 wei of a supported LST (e.g., stETH, which is freely obtainable on the open market) can execute this at any time before or after the admin initiates removal. The cost is negligible (1 wei + gas). The attacker can repeat the transfer every time the admin attempts to drain and retry, making the block persistent. [5](#0-4) 

### Recommendation
Remove the `balanceOf` check from the removal path, or replace it with a pull-based approach: transfer all residual LST balances from the `NodeDelegator` to the `LRTDepositPool` (or `LRTUnstakingVault`) atomically within `_removeNodeDelegatorContractFromQueue` before performing any balance assertion. Alternatively, if a dust check is still desired, set a meaningful non-zero `maxNegligibleAmount` during `initialize()` and document that the check is advisory only, not a hard gate on removal. [4](#0-3) 

### Proof of Concept
1. Admin decides to remove `NodeDelegator` at address `NDC_ADDR` from the queue (e.g., because it is being rotated out).
2. Attacker observes the pending admin transaction (or simply acts preemptively) and calls `stETH.transfer(NDC_ADDR, 1)` — costing 1 wei of stETH.
3. Admin's call to `removeNodeDelegatorContractFromQueue(NDC_ADDR)` executes `_checkResidueLSTBalance`:
   - `assetBalance = IERC20(stETH).balanceOf(NDC_ADDR) + ... = 1 + 0 + 0 = 1`
   - `1 > maxNegligibleAmount` → `1 > 0` → `revert NodeDelegatorHasAssetBalance(stETH, 1)`
4. The removal reverts. The attacker repeats step 2 after every admin drain attempt, indefinitely blocking dequeuing of `NDC_ADDR`. [2](#0-1)

```text
    function _removeNodeDelegatorContractFromQueue(address nodeDelegatorAddress) internal {
        // 1. check if node delegator contract is in queue and find Index
        uint256 ndcIndex = _getNDCIndex(nodeDelegatorAddress);

        // 2. revert if node delegator contract has any asset balances.
        // 2.1 check if NDC has native ETH balance in eigen layer or/and in itself.
        _checkResidueEthBalance(nodeDelegatorAddress);
        // 2.2  check if NDC has LST balance
        _checkResidueLSTBalance(nodeDelegatorAddress);

        // 3. remove node delegator contract from queue
        // 3.1 remove from isNodeDelegator mapping
        isNodeDelegator[nodeDelegatorAddress] = 0;
        // 3.2 remove from nodeDelegatorQueue
        nodeDelegatorQueue[ndcIndex] = nodeDelegatorQueue[nodeDelegatorQueue.length - 1];
        nodeDelegatorQueue.pop();

        emit NodeDelegatorRemovedFromQueue(nodeDelegatorAddress);
    }
```

```text
    function _checkResidueLSTBalance(address nodeDelegatorAddress) internal view {
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetsLength = supportedAssets.length;

        uint256 assetBalance;
        for (uint256 i; i < supportedAssetsLength; ++i) {
            if (supportedAssets[i] == LRTConstants.ETH_TOKEN) {
                // this function only checks for residual LST balance
                continue;
            }

            assetBalance = IERC20(supportedAssets[i]).balanceOf(nodeDelegatorAddress)
                + INodeDelegator(nodeDelegatorAddress).getAssetBalance(supportedAssets[i]);
            assetBalance += INodeDelegator(nodeDelegatorAddress).getAssetUnstaking(supportedAssets[i]);

            if (assetBalance > maxNegligibleAmount) {
                revert NodeDelegatorHasAssetBalance(supportedAssets[i], assetBalance);
            }
        }
```
