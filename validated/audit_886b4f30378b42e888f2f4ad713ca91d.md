Audit Report

## Title
ETH Permanently Frozen for Contract Depositors Without `receive()` - (File: contracts/LRTWithdrawalManager.sol)

## Summary

When a smart contract without a `receive()` or `fallback()` function initiates an ETH withdrawal, the operator's `unlockQueue` call irreversibly burns the depositor's rsETH and moves ETH into `LRTWithdrawalManager`. Every subsequent `completeWithdrawal` call then permanently reverts because `_transferAsset` cannot push ETH to the contract address. No recovery path exists: `sweepRemainingAssets` is blocked as long as the stuck withdrawal keeps `unlockedWithdrawalsCount[asset] > 0`, and `completeWithdrawalForUser` still routes to the same undeliverable address.

## Finding Description

**Step 1 — `unlockQueue` (separate transaction, irreversible):**
rsETH is burned and ETH is redeemed from `LRTUnstakingVault` into `LRTWithdrawalManager`. [1](#0-0) 

**Step 2 — `completeWithdrawal` / `completeWithdrawalForUser` (always reverts):**
`_processWithdrawalCompletion` pops the nonce, deletes the request record, decrements `unlockedWithdrawalsCount`, then calls `_transferAsset(asset, user, ...)`. [2](#0-1) 

`_transferAsset` uses a low-level call and reverts with `EthTransferFailed` if the recipient cannot accept ETH: [3](#0-2) 

Because the revert unwinds all state changes within the transaction, the withdrawal request record, the nonce queue entry, and `unlockedWithdrawalsCount` are all restored to their pre-call values. The call will revert identically on every future attempt.

**Sweep is permanently blocked:**
`sweepRemainingAssets` requires `hasUnlockedWithdrawals(asset) == false`, but the stuck withdrawal keeps `unlockedWithdrawalsCount[asset] > 0` indefinitely: [4](#0-3) 

**`completeWithdrawalForUser` provides no relief:**
The operator-callable variant still routes to the original `user` address, not a caller-specified recipient: [5](#0-4) 

## Impact Explanation

**Critical — Permanent freezing of funds.**

The depositor's rsETH is irreversibly burned in Step 1. The corresponding ETH is permanently locked inside `LRTWithdrawalManager` with no on-chain recovery path available to the user or the protocol short of a contract upgrade. This matches the allowed Critical impact: *Permanent freezing of funds*.

## Likelihood Explanation

Smart contract wallets (Gnosis Safe, DAO treasuries, protocol vaults, custom multisigs) are common depositors in LRT protocols. Not all such contracts implement `receive()` or a payable `fallback`. There is no on-chain check in `initiateWithdrawal` that prevents a non-ETH-receivable contract from queuing a withdrawal. The failure is silent until Step 2, after rsETH is already burned, and is fully repeatable for every affected depositor. [6](#0-5) 

## Recommendation

Add an optional `recipient` parameter to `completeWithdrawal` so the depositor can redirect ETH to an address that can receive it:

```solidity
function completeWithdrawal(
    address asset,
    address recipient,   // caller-specified ETH recipient
    string calldata referralId
) external nonReentrant whenNotPaused {
    _processWithdrawalCompletion(asset, msg.sender, recipient, referralId);
}
```

Update `_processWithdrawalCompletion` to accept a separate `recipient` parameter and pass it to `_transferAsset` instead of `user`. Apply the same change to `completeWithdrawalForUser`. This allows a contract depositor to specify an EOA or a contract capable of receiving ETH before the ETH transfer is attempted.

## Proof of Concept

1. Deploy a contract `NoReceive` (no `receive()`, no `fallback()`) that holds rsETH.
2. `NoReceive` calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. rsETH is transferred to `LRTWithdrawalManager`; a withdrawal request is recorded under `userAssociatedNonces[ETH_TOKEN][NoReceive]`.
3. Operator calls `unlockQueue(ETH_TOKEN, ...)`. rsETH is burned; ETH is redeemed into `LRTWithdrawalManager`. `unlockedWithdrawalsCount[ETH_TOKEN]++`. This transaction succeeds and is irreversible.
4. `NoReceive` calls `completeWithdrawal(ETH_TOKEN, "")`. Inside `_processWithdrawalCompletion`, `_transferAsset` executes `payable(NoReceive).call{value: amount}("")`. The call returns `success = false`; the function reverts with `EthTransferFailed`. All state changes in this transaction are rolled back.
5. Repeat step 4 indefinitely — always reverts.
6. Call `sweepRemainingAssets(ETH_TOKEN)` — reverts with `PendingWithdrawalsExist` because `unlockedWithdrawalsCount[ETH_TOKEN] > 0`.
7. Result: rsETH permanently burned, ETH permanently locked.

    function _processWithdrawalCompletion(address asset, address user, string calldata referralId) internal {
        if (userAssociatedNonces[asset][user].empty()) {
            revert NoWithdrawalRequests(user, asset);
        }

        // Retrieve and remove the oldest withdrawal request for the user.
        uint256 usersFirstWithdrawalRequestNonce = userAssociatedNonces[asset][user].popFront();
        // Ensure the request is already unlocked.
        if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();

        bytes32 requestId = getRequestId(asset, usersFirstWithdrawalRequestNonce);
        WithdrawalRequest memory request = withdrawalRequests[requestId];

        delete withdrawalRequests[requestId];

        // Check that the withdrawal delay has passed since the request's initiation.
        if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();

        unlockedWithdrawalsCount[asset]--;

        // If Aave integration is enabled and asset is ETH, withdraw from Aave if needed
        if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
            uint256 contractBalance = address(this).balance;
            if (contractBalance < request.expectedAssetAmount) {
                uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
                _withdrawFromAave(amountNeeded);

                // Verify we have sufficient balance after withdrawal
                uint256 balanceAfter = address(this).balance;
                if (balanceAfter < request.expectedAssetAmount) {
                    revert InsufficientLiquidityForWithdrawal();
                }
            }
        }

        _transferAsset(asset, user, request.expectedAssetAmount);

        emit ReferralIdEmitted(referralId);
        emit AssetWithdrawalFinalized(user, asset, request.rsETHUnstaked, request.expectedAssetAmount);
    }
