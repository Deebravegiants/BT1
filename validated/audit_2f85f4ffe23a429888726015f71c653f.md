### Title
`RSETHPriceFeed` Returns Zero Price When `LRTOracle.rsETHPrice` Is Uninitialized, Enabling Collateral Liquidation Theft in External Lending Protocols - (File: contracts/oracles/RSETHPriceFeed.sol)

---

### Summary

`RSETHPriceFeed.latestRoundData()` and `getRoundData()` compute `answer` by multiplying `RS_ETH_ORACLE.rsETHPrice()` by the ETH/USD Chainlink answer without any zero-check on the rsETH price. Because `LRTOracle.rsETHPrice` is a `uint256` storage variable that defaults to `0` and is never set inside `initialize()`, the price feed returns `answer = 0` during the deployment window before `updateRSETHPrice()` is first called. Any external lending protocol that integrates `RSETHPriceFeed` as a Chainlink-compatible oracle during this window will price rsETH at $0, making all rsETH collateral positions immediately liquidatable at zero cost.

---

### Finding Description

**Root cause — `LRTOracle.rsETHPrice` starts at 0 and is not set in `initialize()`:**

`LRTOracle` declares `rsETHPrice` as a plain `uint256` storage variable: [1](#0-0) 

Its `initialize()` function only sets `lrtConfig` and emits an event — `rsETHPrice` is never written: [2](#0-1) 

**Root cause — `updateRSETHPrice()` is public with no access control:**

Anyone can call `updateRSETHPrice()` at any time: [3](#0-2) 

**Root cause — `_updateRsETHPrice()` can write `rsETHPrice = 0` when `pricePercentageLimit == 0`:**

Inside `_updateRsETHPrice()`, if `rsethSupply > 0` and `totalETHInProtocol == 0`, `newRsETHPrice` is computed as `0`. The downside-protection guard only pauses the protocol when `pricePercentageLimit > 0`; if it is `0` (the Solidity default, never set in `initialize()`), the guard is skipped and `rsETHPrice = 0` is committed to storage: [4](#0-3) [5](#0-4) 

**Root cause — `RSETHPriceFeed` blindly multiplies by `rsETHPrice` with no zero-check:**

Both `latestRoundData()` and `getRoundData()` compute:

```solidity
answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
``` [6](#0-5) [7](#0-6) 

When `rsETHPrice == 0`, `answer` becomes `0` regardless of the ETH/USD Chainlink price. There is no revert, no staleness flag, and no fallback — the feed silently reports rsETH as worthless.

---

### Impact Explanation

**Critical — Direct theft of user funds.**

`RSETHPriceFeed` is a Chainlink `AggregatorV3Interface`-compatible contract explicitly designed to be consumed by external lending protocols (e.g., Aave, Compound forks, Morpho). When it returns `answer = 0`:

- Every rsETH collateral position in any integrated lending protocol becomes instantly undercollateralized (collateral value = $0).
- An attacker can liquidate all such positions at zero or near-zero cost, seizing the underlying collateral assets (ETH, stETH, etc.) that real users deposited.
- The attacker profits by the full value of the liquidated collateral minus any liquidation bonus paid to the protocol — a direct, complete theft of user funds at rest.

---

### Likelihood Explanation

**Medium.**

Two independent trigger conditions exist:

1. **Deployment window (always present):** Every deployment of `LRTOracle` starts with `rsETHPrice = 0`. If `RSETHPriceFeed` is integrated into a lending protocol before `updateRSETHPrice()` is first called — a realistic race condition during launch — the feed returns 0 for the entire window. No attacker action is required to create the condition; only timing is needed.

2. **Public `updateRSETHPrice()` with `pricePercentageLimit = 0`:** Because `pricePercentageLimit` is also `0` by default and `updateRSETHPrice()` is unrestricted, if `totalETHInProtocol` ever reaches 0 while `rsethSupply > 0` (e.g., due to an accounting edge case or a supported-asset list change), any unprivileged caller can invoke `updateRSETHPrice()` to write `rsETHPrice = 0` into storage, permanently corrupting the price feed until an admin intervenes.

---

### Recommendation

1. **In `RSETHPriceFeed`:** Add an explicit zero-check before returning `answer`:
   ```solidity
   uint256 rsETHRate = RS_ETH_ORACLE.rsETHPrice();
   require(rsETHRate > 0, "RSETHPriceFeed: zero rsETH price");
   answer = int256(rsETHRate) * answer / 1e18;
   ```

2. **In `LRTOracle.initialize()`:** Set `rsETHPrice = 1 ether` (the canonical initial rate) so the storage variable is never left at 0 after deployment.

3. **In `LRTOracle._updateRsETHPrice()`:** Add a guard that prevents writing `rsETHPrice = 0` regardless of `pricePercentageLimit`:
   ```solidity
   if (newRsETHPrice == 0) revert InvalidPrice();
   ```

4. **In `LRTOracle.initialize()` or `reinitialize()`:** Require `pricePercentageLimit` to be set to a non-zero value before the oracle is considered operational.

---

### Proof of Concept

```
State at deployment:
  LRTOracle.rsETHPrice = 0          (Solidity default, not set in initialize())
  LRTOracle.pricePercentageLimit = 0 (Solidity default, not set in initialize())

Step 1: RSETHPriceFeed is deployed pointing to LRTOracle.
Step 2: An external lending protocol (e.g., Aave fork) integrates RSETHPriceFeed
        as the rsETH/USD price oracle and accepts rsETH as collateral.
Step 3: Legitimate users deposit rsETH as collateral and borrow against it.
        (updateRSETHPrice() has not yet been called, OR has been reset to 0.)

Step 4: Attacker calls RSETHPriceFeed.latestRoundData():
          ETH_TO_USD.latestRoundData() → answer = 3000e8  (e.g., $3000/ETH)
          RS_ETH_ORACLE.rsETHPrice()   → 0
          answer = int256(0) * 3000e8 / 1e18 = 0
        → rsETH/USD price reported = $0

Step 5: Lending protocol sees all rsETH collateral positions as having $0 value.
        All positions are immediately liquidatable.

Step 6: Attacker calls liquidate() on all rsETH collateral positions,
        receiving the underlying collateral (ETH, stETH, etc.) at zero cost.

Result: Attacker steals 100% of rsETH-backed collateral from all affected users.
```
