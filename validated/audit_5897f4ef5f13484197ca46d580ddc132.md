### Title
MANAGER-callable `setKingToken` strands old KING yield tokens with no self-service recovery path — (`contracts/king-protocol/TokenSwap.sol`)

### Summary

`setKingToken` is restricted to `MANAGER_ROLE`, but after it redirects the `kingToken` storage pointer, any old KING tokens already held by the contract become unreachable through `withdrawKing`. The only recovery path is `emergencyWithdraw`, which is `onlyAdmin`. If admin is a timelock or multisig, yield is frozen for the duration of the delay — with no upper bound guarantee.

---

### Finding Description

`setKingToken` updates the `kingToken` storage variable to a new address: [1](#0-0) 

`withdrawKing` reads the balance of the **current** `kingToken` pointer: [2](#0-1) 

After `setKingToken(newAddr)`, `kingToken` points to `newAddr`. The old KING tokens sitting in the contract are invisible to `withdrawKing` — it will see a zero balance for `newAddr` and revert with `InsufficientBalance`. The only escape hatch is: [3](#0-2) 

This is `onlyAdmin`, not `onlyAdminOrManager`. The MANAGER who deposited assets and accumulated yield has no self-service path to recover the stranded old KING tokens.

---

### Impact Explanation

Old KING tokens representing accrued yield are inaccessible via the normal `withdrawKing` flow after any `setKingToken` call. Recovery requires admin action. If admin is a timelock or multisig with a queuing delay, the yield is frozen for at least that delay period. There is no on-chain time bound guaranteeing recovery, so the freeze can be indefinite from the MANAGER's perspective.

**Impact: Medium — Permanent freezing of unclaimed yield.**

---

### Likelihood Explanation

`setKingToken` is a legitimate operational function the MANAGER is expected to call during a KING token migration or upgrade. The stranding of old tokens is a silent side effect — no revert, no warning. A MANAGER performing a routine token pointer update can accidentally trigger this without any malicious intent. No compromise of any key is required; the MANAGER is using their authorized function.

---

### Recommendation

1. **Drain before redirect**: In `setKingToken`, automatically transfer any remaining old KING balance to a safe address (or revert if balance > 0, forcing the caller to drain first).
2. **Restrict to admin**: If token address changes are rare/sensitive, gate `setKingToken` with `onlyAdmin` instead of `onlyManager`.
3. **Add recovery to manager**: Allow `MANAGER_ROLE` to call `emergencyWithdraw` for the specific old token address, or add a dedicated `withdrawToken(address token, ...)` function accessible to managers.

---

### Proof of Concept

```solidity
// 1. TokenSwap holds 1000e18 old KING tokens (accrued yield from deposits)
// 2. MANAGER calls setKingToken(newKingAddr)  -- legitimate migration step
//    => kingToken now points to newKingAddr
// 3. MANAGER calls withdrawKing(recipient, 1000e18)
//    => kingToken.balanceOf(address(this)) == 0  (newKingAddr balance)
//    => reverts: InsufficientBalance
// 4. Old KING tokens are stranded; only admin can recover via emergencyWithdraw
// 5. If admin is a 48h timelock, yield is frozen for ≥48h with no upper bound
```

Invariant that breaks: after any `setKingToken` call, the sum of old and new KING token balances held by `TokenSwap` must equal the pre-call total. The contract provides no mechanism to enforce or restore this invariant.

        UtilLib.checkNonZeroAddress(token);
        UtilLib.checkNonZeroAddress(recipient);

        if (amount == 0) {
            revert ZeroAmount();
        }

        IERC20(token).safeTransfer(recipient, amount);
    }
