### Title
Unbounded Subtransaction Plutus Execution Units Bypass Block-Level `maxBlockExUnits` Limit — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs`)

---

### Summary

The Dijkstra era introduces nested subtransactions (`sub_transactions`, field 23 of the transaction body). The `BBODY` rule enforces the block-level execution unit cap using `Alonzo.validateExUnits`, which sums `totExUnits` over top-level transactions only. Because `totExUnits` reads redeemers from the top-level transaction's witness set and subtransactions carry their own independent witness sets, subtransaction Plutus execution units are invisible to the block-level cap. An unprivileged transaction author can craft a single top-level transaction containing many subtransactions, each consuming up to `maxTxExUnits` execution units, causing the actual per-block Plutus execution cost to be a large multiple of `maxBlockExUnits` while the block passes the `maxBlockExUnits` check.

---

### Finding Description

**Root cause — `BBODY` block-level ExUnits check ignores subtransaction redeemers**

`dijkstraBbodyTransition` in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs` enforces the block execution-unit cap with the unmodified Alonzo helper:

```haskell
Alonzo.validateExUnits @era txs $ pp ^. ppMaxBlockExUnitsL
``` [1](#0-0) 

`Alonzo.validateExUnits` folds `totExUnits` over the sequence of top-level transactions. `totExUnits` is defined in the Alonzo era as:

```haskell
totExUnits tx = foldMap snd . unRedeemers $ tx ^. witsTxL . rdmrsTxWitsL
```

It reads redeemers exclusively from the top-level transaction's witness set (`dtWits`). In the Dijkstra era, subtransactions are embedded in the top-level transaction body at CBOR map key 23 (`sub_transactions`) and each subtransaction carries its own independent witness set (`dstWits`) with its own redeemers. Those redeemers are never visited by `totExUnits`, so the block-level cap sees only the top-level transaction's execution units.

**Unbounded iteration in `SUBLEDGERS`**

`dijkstraSubLedgersTransition` in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubLedgers.hs` iterates over every subtransaction with a plain `foldM`, applying the full `SUBLEDGER` rule (which includes `SUBUTXOW` → Plutus script execution) to each one, with no explicit count limit:

```haskell
dijkstraSubLedgersTransition = do
  TRC (env, ledgerState, subTxs) <- judgmentContext
  foldM
    ( \ls subTx ->
        trans @(EraRule "SUBLEDGER" era) $ TRC (env, ls, subTx)
    )
    ledgerState
    subTxs
``` [2](#0-1) 

Each subtransaction's execution units are individually bounded by `maxTxExUnits` (checked inside `SUBUTXO`), but the aggregate across all subtransactions in the batch is never compared against `maxBlockExUnits`.

**The only implicit bound is `maxTxSize`**

The `validateMaxTxSizeUTxO` check applies to the serialized size of the top-level transaction, which includes the embedded subtransactions. This provides an implicit upper bound on the number of subtransactions. However, with Babbage-era reference scripts, a subtransaction can reference a large Plutus script stored in a UTxO output without including the script bytes in the transaction body. This means a subtransaction can consume close to `maxTxExUnits` execution units while contributing only a small number of bytes to the transaction size, allowing many such subtransactions to fit within `maxTxSize`. [3](#0-2) 

**Fee calculation is also affected**

The minimum-fee calculation for a Dijkstra transaction uses the same `totExUnits` path. Subtransaction execution units are therefore not reflected in the fee, meaning the attacker pays less than the actual computational cost imposed on validators.

---

### Impact Explanation

This is a **Medium** impact finding matching: *"Attacker-controlled transactions, blocks, certificates, votes, proposals, scripts, witnesses, or serialized inputs exceed intended validation limits."*

`maxBlockExUnits` is the protocol's primary mechanism for bounding the CPU and memory cost of Plutus script execution per block. By submitting a single top-level transaction with N subtransactions each consuming up to `maxTxExUnits` execution units, an attacker causes the actual per-block Plutus execution cost to reach `N × maxTxExUnits`. With typical parameters (`maxTxSize` ≈ 16 KB, minimal subtransaction referencing a script ≈ 100 bytes, `maxTxExUnits` / `maxBlockExUnits` ≈ 0.25), the amplification factor is on the order of 40×. This forces all honest block-producing and validating nodes to perform far more Plutus evaluation work per block than the protocol intends, degrading throughput and potentially causing nodes to fall behind chain tip.

---

### Likelihood Explanation

The Dijkstra era is the newest era in the codebase and subtransactions are a novel feature. The attack requires only the ability to submit a valid transaction — no privileged access, no governance majority, no key compromise. An attacker who understands the execution-unit accounting discrepancy can construct the exploit transaction directly. Likelihood is **Medium**.
