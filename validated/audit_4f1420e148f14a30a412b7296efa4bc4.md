## Analysis

I traced the full value conservation path for the Dijkstra era to determine whether `directDepositsDijkstraTxBodyRawL` can be used to bypass value conservation.

### Step 1: What `directDepositsDijkstraTxBodyRawL` is

It is a simple `Lens'` accessor for the `dtbrDirectDeposits` / `dstbrDirectDeposits` field in `DijkstraTxBodyRaw`. It is present in both top-level and sub-transaction bodies. [1](#0-0) 

`DirectDeposits` is a `Map AccountAddress Coin` — a map of account addresses to coin amounts that are added to account balances when the transaction is applied. [2](#0-1) 

### Step 2: Where `DirectDeposits` is applied

`applyDirectDeposits` is called in the `ENTITIES` rule (and `SUBENTITIES` rule for sub-transactions). It unconditionally **adds** the specified coin amounts to account balances: [3](#0-2) [4](#0-3) 

### Step 3: The value conservation check

The Dijkstra UTXO rule calls `Shelley.validateValueNotConservedUTxO`: [5](#0-4) 

This computes `consumed pp certState utxo txBody == produced pp certState txBody`. For `DijkstraEra`:

**Consumed** (`getConsumedDijkstraValue` → `getConsumedMaryValue`):
- UTxO inputs + minted positive multi-assets + refunds + withdrawals [6](#0-5) 

**Produced** (`getProducedDijkstraValue` → `dijkstraProducedValue` → `conwayProducedValue` → `getProducedMaryValue` → `shelleyProducedValue`):
- Outputs + fee + cert/proposal deposits + treasury donation + burned multi-assets [7](#0-6) [8](#0-7) 

**`DirectDeposits` appears in neither consumed nor produced.**

### Step 4: Consequence

The value conservation equation for Dijkstra is:

```
UTxO_inputs + minted + refunds + withdrawals
  == outputs + fee + deposits + treasury_donation + burned
```

`DirectDeposits` is entirely absent. The ENTITIES rule then adds the specified coin amounts to account balances without those coins being balanced anywhere in the equation. The only validation on `DirectDeposits` in the UTXO rule is a network-ID check: [9](#0-8) 

There is no check that the sum of direct deposits is funded by the transaction's inputs.

### Step 5: Is this exploitable?

The `DirectDeposits` field is fully functional — it is decoded from CBOR (key 25), stored in the transaction body, and applied to account balances. An unprivileged attacker can include any `DirectDeposits` map in a transaction body. The value conservation check will pass regardless of the direct deposit amounts, and the specified ADA will be added to account balances from nowhere.

### Title
`DirectDeposits` Omitted from Value Conservation — ADA Creation in Dijkstra Era (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

### Summary
The `DirectDeposits` field in the Dijkstra era transaction body is applied to account balances by the `ENTITIES` rule but is absent from both sides of the value conservation equation checked by the `UTXO` rule. Any unprivileged submitter can include arbitrary direct-deposit amounts in a transaction, pass the `consumed == produced` check, and have those coins credited to accounts without any corresponding debit from the transaction's inputs.

### Finding Description
`getConsumedDijkstraValue` aggregates UTxO inputs, minted positive multi-assets, refunds, and withdrawals. `getProducedDijkstraValue` / `dijkstraProducedValue` aggregates outputs, fee, cert/proposal deposits, treasury donation, and burned multi-assets. Neither function reads `txBody ^. directDepositsTxBodyL`. The `ENTITIES` rule subsequently calls `applyDirectDeposits`, which calls `addCompactCoin` on each targeted account balance. No prior rule has required those coins to be present in the transaction's inputs.

The only guard on `DirectDeposits` in the UTXO rule is `validateWrongNetworkInDirectDeposit`, which checks network IDs only.

### Impact Explanation
**Critical.** An attacker can create ADA (or, for sub-transactions, any value) out of thin air. A transaction with inputs worth X ADA, outputs worth X − fee ADA, and a `DirectDeposits` map crediting Y ADA to attacker-controlled accounts will pass all ledger checks. The attacker then withdraws Y ADA via a subsequent transaction. This is direct, unbounded ADA creation.

### Likelihood Explanation
Any node operator or wallet user can submit such a transaction. No privileged access, governance majority, or key compromise is required. The `DirectDeposits` field is fully decoded and applied in the current codebase.

### Recommendation
Add the sum of `DirectDeposits` to the **produced** side of the value conservation equation, symmetric to how withdrawals appear on the consumed side. Concretely, `dijkstraProducedValue` and `dijkstraSubTxProducedValue` should include:

```haskell
<> inject (fold . unDirectDeposits $ txBody ^. directDepositsTxBodyL)
``` [7](#0-6) [8](#0-7) 

### Proof of Concept

```
Inputs:  { utxo_in → 10 ADA }
Outputs: { attacker_utxo → 9 ADA }
Fee:     1 ADA
DirectDeposits: { attacker_account → 1_000_000 ADA }
```

`consumed = 10 ADA`, `produced = 9 + 1 = 10 ADA` → check passes.  
`applyDirectDeposits` credits 1 000 000 ADA to `attacker_account`.  
A follow-up withdrawal transaction drains the account.
