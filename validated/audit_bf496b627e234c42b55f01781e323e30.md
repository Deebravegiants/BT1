### Title
Donation Attack via Raw `balanceOf` in `getAssetDistributionData()` Inflates rsETH Price, Enabling Fund Theft - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool.getAssetDistributionData()` uses raw `IERC20(asset).balanceOf(address(this))` and `address(this).balance` to account for assets lying in the deposit pool. These values feed directly into `LRTOracle._getTotalEthInProtocol()`, which computes the stored `rsETHPrice`. Because the contract accepts arbitrary ETH via `receive()` and arbitrary ERC20 transfers, an attacker can donate assets to inflate `rsETHPrice`, causing subsequent depositors to receive drastically fewer (or zero) rsETH shares, with the donated assets accruing to the attacker's existing rsETH position.

### Finding Description

`LRTDepositPool.getAssetDistributionData()` computes the pool's asset balance using raw on-chain balances:

```solidity
// contracts/LRTDepositPool.sol
assetLyingInDepositPool = IERC20(asset).balanceOf(address(this));
// ...
ethLyingInDepositPool = address(this).balance;
```

These values flow into `getTotalAssetDeposits()` → `LRTOracle._getTotalEthInProtocol()` → `_updateRsETHPrice()`:

```solidity
// contracts/LRTOracle.sol
uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
// ...
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

`LRTDepositPool` has an open `receive()` function and accepts arbitrary ERC20 transfers, so any actor can inflate `totalETHInProtocol` without minting rsETH. The permissionless `updateRSETHPrice()` then commits the inflated price to storage.

The only guard against unbounded price increases is `pricePercentageLimit`:

```solidity
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
```

`pricePercentageLimit` is **not initialized** in `initialize()` and defaults to `0`, making `isPriceIncreaseOffLimit` permanently `false` until an admin explicitly calls `setPricePercentageLimit()`. With `pricePercentageLimit == 0`, the price can increase by an arbitrary multiple in a single `updateRSETHPrice()` call.

The rsETH minting formula in `getRsETHAmountToMint()` is:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

If `rsETHPrice` is inflated sufficiently, this expression rounds down to zero.

### Impact Explanation

**Critical — Direct theft of user funds.**

Attack walkthrough (ETH path, `pricePercentageLimit == 0`):

| Step | Action | rsETH Supply | Total ETH | rsETH Price |
|------|--------|-------------|-----------|-------------|
| 1 | Attacker deposits 1 ETH via `depositETH()` | 1 | 1 ETH | 1 ETH |
| 2 | Attacker sends 100 ETH directly to `LRTDepositPool` (`receive()`) | 1 | 101 ETH | — |
| 3 | Attacker calls `updateRSETHPrice()` | 1 | 101 ETH | **101 ETH** |
| 4 | Victim deposits 50 ETH with `minRSETHAmountExpected = 0` → `50e18 / 101e18 = 0` rsETH minted | 1 | 151 ETH | 151 ETH |
| 5 | Attacker redeems 1 rsETH → receives **151 ETH** | 0 | 0 | — |

Attacker net gain: `151 − 1 (deposit) − 100 (donation) = 50 ETH` (the victim's entire deposit).

The victim's 50 ETH is permanently lost; they receive zero rsETH and have no claim on the protocol.

### Likelihood Explanation

**Medium.** Two conditions must hold simultaneously:

1. **`pricePercentageLimit == 0`**: This is the default state. The variable is not set in `initialize()`, so every newly deployed or upgraded `LRTOracle` instance starts with this protection disabled. An admin must call `setPricePercentageLimit()` to enable it.

2. **Victim uses `minRSETHAmountExpected = 0`**: The `depositETH()` and `depositAsset()` functions accept this parameter. Users who pass `0` (e.g., via a frontend that omits slippage protection, or direct contract calls) receive no protection.

Both conditions are realistic in practice, particularly at protocol launch or after an upgrade before the admin configures `pricePercentageLimit`.

### Recommendation

1. **Initialize `pricePercentageLimit` to a non-zero value** in `initialize()` (e.g., `1e16` for 1%) so the protection is active from deployment.

2. **Track deposited assets separately** from raw `balanceOf`. Maintain an internal accounting variable (e.g., `totalDepositedAsset[asset]`) that is incremented only on legitimate deposits and decremented on withdrawals, analogous to how `feeEarnedInETH` is tracked in `RSETHPoolV3`. Use this internal variable instead of `balanceOf` in `getAssetDistributionData()`.

3. **Enforce a non-zero `minRSETHAmountExpected`** in `_beforeDeposit()` to prevent zero-share deposits.

### Proof of Concept

```solidity
// Precondition: pricePercentageLimit == 0 (default)
// Attacker holds 1 rsETH from depositing 1 ETH

// Step 1: Attacker donates 100 ETH directly to LRTDepositPool

require(ok);

// Step 2: Attacker updates rsETH price — no revert because pricePercentageLimit == 0

// Step 3: Victim deposits 50 ETH with no slippage protection

// rsETHPrice is now ~151 ETH/rsETH; attacker recovers 151 ETH
