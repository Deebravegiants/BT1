### Title
Missing Chainlink Price Feed Staleness Validation Enables Oracle Rate Abuse at Deposit - (`contracts/oracles/ChainlinkPriceOracle.sol`)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all validity fields (`roundId`, `updatedAt`, `answeredInRound`), consuming the price with no staleness check. This is the direct analog to the PoolTogether VRF finding: both protocols consume a Chainlink-sourced value without any confirmation/freshness guard, allowing an attacker to exploit a stale external value to game the protocol's accounting.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` reads only the raw `price` field from `latestRoundData()`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

The five return values of `latestRoundData()` are `(roundId, answer, startedAt, updatedAt, answeredInRound)`. The contract silently ignores `updatedAt` (heartbeat freshness) and `answeredInRound` (round completeness). There is no check that `answeredInRound >= roundId` and no check that `block.timestamp - updatedAt <= heartbeat`.

The protocol's own `ChainlinkOracleForRSETHPoolCollateral` contract — used for pool collateral pricing — correctly performs these checks:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

The stale price from `ChainlinkPriceOracle` propagates through the following call chain:

1. `LRTOracle.getAssetPrice(asset)` delegates to `IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset)`, which resolves to `ChainlinkPriceOracle.getAssetPrice()`. [3](#0-2) 

2. `LRTOracle._getTotalEthInProtocol()` calls `getAssetPrice(asset)` for every supported LST to compute the protocol's total ETH value. [4](#0-3) 

3. `LRTDepositPool.getRsETHAmountToMint()` uses both `lrtOracle.getAssetPrice(asset)` and the stored `lrtOracle.rsETHPrice()` to determine how many rsETH tokens to mint per deposit. [5](#0-4) 

4. `LRTDepositPool.depositAsset()` calls `_beforeDeposit()` → `getRsETHAmountToMint()`, so every user deposit is priced using the potentially stale Chainlink value. [6](#0-5) 

---

### Impact Explanation

**Impact: High — Theft of unclaimed yield / protocol insolvency.**

The stored `rsETHPrice` is a snapshot updated by `updateRSETHPrice()`. During the window between price updates, `getAssetPrice(asset)` is called live against the Chainlink feed. If a supported LST (e.g., stETH) depegs but the Chainlink feed has not yet updated (stale at the pre-depeg high price), an attacker can:

- Deposit the depegged LST at the stale inflated price.
- Receive `rsethAmountToMint = (depositAmount × stalePriceHigh) / rsETHPrice` — more rsETH than the LST's actual ETH value warrants.
- The excess rsETH represents a direct dilution of all existing rsETH holders' claims on the underlying ETH pool.

When `updateRSETHPrice()` is subsequently called with the stale price, `_getTotalEthInProtocol()` computes an inflated TVL, causing `rsETHPrice` to be set incorrectly. This can trigger the downside-protection pause (if the corrected price later drops below `highestRsethPrice`) or leave the protocol in an insolvent state where rsETH is backed by less ETH than its price implies. [7](#0-6) 

---

### Likelihood Explanation

**Likelihood: Medium.**

Chainlink feeds go stale in documented, recurring scenarios:
- Network congestion preventing oracle node transactions from landing.
- L2 sequencer downtime (the protocol deploys on multiple L2s).
- Chainlink node operational issues.

The attack window is bounded by the Chainlink heartbeat (typically 1–24 hours for LST/ETH feeds). An attacker monitoring on-chain feed timestamps can detect staleness and execute the deposit atomically. No privileged access is required — `depositAsset()` is open to any user. [6](#0-5) 

---

### Recommendation

Add staleness and validity checks to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > MAX_HEARTBEAT) revert PriceTooOld();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`MAX_HEARTBEAT` should be set per-asset to match the Chainlink feed's documented heartbeat (e.g., 86400 seconds for daily-updated LST feeds).

---

### Proof of Concept

1. Observe that the stETH/ETH Chainlink feed's `updatedAt` timestamp is more than one heartbeat old (feed is stale at price `P_stale > P_actual`).
2. Call `LRTDepositPool.depositAsset(stETH, amount, 0, "")`.
3. Internally, `getRsETHAmountToMint(stETH, amount)` computes:
   - `assetPrice = ChainlinkPriceOracle.getAssetPrice(stETH)` → returns `P_stale` (no staleness revert).
   - `rsethAmountToMint = amount * P_stale / rsETHPrice`.
4. Attacker receives `amount * P_stale / rsETHPrice` rsETH, which is worth more ETH than `amount` of the depegged stETH.
5. Attacker redeems rsETH via `LRTWithdrawalManager` for ETH, extracting the difference at the expense of existing rsETH holders. [1](#0-0) [8](#0-7) [2](#0-1)
