### Title
Sub-Transactions in Dijkstra Era Bypass `maxTxExUnits` Execution-Unit Limit — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`)

---

### Summary

The `SUBUTXO` transition rule for Dijkstra era sub-transactions omits the `validateExUnitsTooBigUTxO` check entirely. An unprivileged transaction sender can embed arbitrarily many sub-transactions, each carrying a near-maximum Plutus execution-unit budget, causing block producers to execute far more script computation than the `maxTxExUnits` protocol parameter is designed to allow — with no corresponding fee obligation for the excess computation.

---

### Finding Description

In the Dijkstra era, a top-level transaction (`TopTx`) may contain nested sub-transactions (`SubTx`) stored in `dtbrSubTransactions`.

**Top-level path (enforced):** `dijkstraUtxoTransition` in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs` calls:

```haskell
{- totExunits tx ≤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
``` [1](#0-0) 

`totExUnits` is defined as:

```haskell
totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL
``` [2](#0-1) 

This only sums redeemers in the **top-level** transaction's own witness set. Sub-transactions carry their own independent witness sets with their own redeemers.

**Sub-transaction path (unenforced):** Each sub-transaction is validated by `dijkstraSubUtxoTransition` in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`. The complete transition rule (lines 231–278) performs validity-interval, input, output, and network-ID checks, but **never calls `validateExUnitsTooBigUTxO`**. [3](#0-2) 

The `DijkstraSubUtxoPredFailure` type defines `SubMaxTxSizeUTxO` (suggesting a size check was at least considered) but defines no `SubExUnitsTooBigUTxO` variant. [4](#0-3) 

The injection function `dijkstraUtxoToDijkstraSubUtxoPredFailure` explicitly marks the execution-unit failure as impossible for sub-transactions:

```haskell
ExUnitsTooBigUTxO _ -> error "Impossible: `ExUnitsTooBigUTxO` for SUBUTXO"
``` [5](#0-4) 

This confirms the omission is structural, not accidental.

**Fee path (also unenforced for sub-transactions):** The minimum-fee check in `dijkstraUtxoTransition` calls `validateFeeTooSmallUTxO pp tx originalUtxo`, which computes the minimum fee using `alonzoMinFeeTx`. That function includes `txscriptfee (pp ^. ppPricesL) allExunits` where `allExunits = totExUnits tx` — again only the top-level redeemers. [6](#0-5) 

Sub-transaction execution units therefore contribute **neither to the per-transaction limit check nor to the fee calculation**.

**Block-level path (also unenforced):** The Alonzo/Conway BBODY rule's `validateExUnits` sums `foldMap totExUnits txs` across all top-level transactions in a block. [7](#0-6) 

Because `totExUnits` ignores sub-transaction redeemers, sub-transaction execution units are also not counted against `maxBlockExUnits`.

---

### Impact Explanation

An attacker crafts a top-level transaction with:
- Zero top-level redeemers (passes `maxTxExUnits` check, pays only the base fee)
- N sub-transactions, each carrying redeemers with execution units approaching `maxTxExUnits`

Total execution units executed by block producers = N × `maxTxExUnits`, with no protocol-level cap. The attacker pays a fee proportional only to the top-level transaction's size and its own (zero) execution units, while forcing block producers to execute an unbounded amount of Plutus script computation. This can delay or halt block production in the Dijkstra era.

This matches: **Medium — Attacker-controlled transactions exceed intended validation limits.**

---

### Likelihood Explanation

The Dijkstra era is not yet deployed on mainnet, reducing immediate risk. However, the exploit path requires only the ability to submit a valid transaction — no privileged access, no key compromise, no governance majority. The attacker pays a minimal fee (base transaction fee only) while causing disproportionate computation. The structural omission (the `error "Impossible"` annotation) means no future code path can accidentally trigger the missing check; it must be explicitly added.

---

### Recommendation

1. Add `validateExUnitsTooBigUTxO pp tx` to `dijkstraSubUtxoTransition`, enforcing `totExUnits subTx ≤ maxTxExUnits pp` for each sub-transaction individually.
2. Add a corresponding `SubExUnitsTooBigUTxO` constructor to `DijkstraSubUtxoPredFailure`.
3. Consider whether the Dijkstra BBODY rule should aggregate sub-transaction execution units into the per-block `maxBlockExUnits` check.
4. Verify that the minimum-fee calculation (`alonzoMinFeeTx` / `validateFeeTooSmallUTxO`) accounts for sub-transaction execution units, or introduce a separate fee component for them.

---

### Proof of Concept

```
1. Construct a Dijkstra TopTx with:
   - One UTxO input (to pay fees)
   - Fee = base fee only (no script fee, since top-level redeemers = 0)
   - dtbrSubTransactions = OMap of N DijkstraSubTx, each with:
       - A Plutus script spending a UTxO
       - Redeemers with ExUnits { mem = maxTxExUnitsMem, steps = maxTxExUnitsSteps }

2. Submit the transaction.

3. The MEMPOOL/LEDGER/UTXO rule validates the top-level transaction:
   - totExUnits topTx = 0  →  passes maxTxExUnits check
   - fee = base fee only   →  passes fee check

4. SUBLEDGERS iterates over each sub-transaction via SUBLEDGER → SUBUTXOW → SUBUTXO.
   dijkstraSubUtxoTransition runs for each sub-transaction:
   - No validateExUnitsTooBigUTxO call → each sub-transaction's ExUnits are unchecked.

5. Block producers execute N × maxTxExUnits worth of Plutus computation,
   far exceeding the protocol-parameter limit, with no rejection.
``` [8](#0-7) [9](#0-8) [10](#0-9)

dijkstraSubUtxoTransition ::
  forall era.
  ( EraTx era
  , EraStake era
  , EraCertState era
  , DijkstraEraTxBody era
  , AlonzoEraTxWits era
  , STS (EraRule "SUBUTXO" era)
  , EraRule "SUBUTXO" era ~ SUBUTXO era
  , InjectRuleFailure "SUBUTXO" Shelley.ShelleyUtxoPredFailure era
  , InjectRuleFailure "SUBUTXO" Allegra.AllegraUtxoPredFailure era
  , InjectRuleFailure "SUBUTXO" Alonzo.AlonzoUtxoPredFailure era
  , InjectRuleFailure "SUBUTXO" Babbage.BabbageUtxoPredFailure era
  , InjectRuleFailure "SUBUTXO" DijkstraUtxoPredFailure era
  ) =>
  TransitionRule (EraRule "SUBUTXO" era)
dijkstraSubUtxoTransition = do
  TRC (SubUtxoEnv slot pp certState originalUtxo (IsValid isValid), utxoState, stAnnTx) <-
    judgmentContext
  let tx = stAnnTx ^. txStAnnTxG

  let txBody = tx ^. bodyTxL

  runTest $ Allegra.validateOutsideValidityIntervalUTxO slot txBody

  sysSt <- liftSTS $ asks systemStart
  ei <- liftSTS $ asks epochInfo
  runTest $ Alonzo.validateOutsideForecast ei slot sysSt tx

  let allSizedOutputs = txBody ^. allSizedOutputsTxBodyF
  let allOutputs = fmap sizedValue allSizedOutputs
  runTest $ Alonzo.validateOutputTooBigUTxO pp allOutputs

  runTest $ Shelley.validateInputSetEmptyUTxO txBody

  let inputs = txBody ^. inputsTxBodyL
  let refInputs = txBody ^. referenceInputsTxBodyL
  runTest $ Shelley.validateBadInputsUTxO originalUtxo (inputs `Set.union` refInputs)
  runTest $ Shelley.validateBadInputsUTxO (utxosUtxo utxoState) inputs

  runTestOnSignal $ Shelley.validateOutputBootAddrAttrsTooBig allOutputs

  runTestOnSignal $ Babbage.validateOutputTooSmallUTxO pp allSizedOutputs

  netId <- liftSTS $ asks networkId
  runTestOnSignal $ Shelley.validateWrongNetwork netId allOutputs
  runTestOnSignal $ Shelley.validateWrongNetworkWithdrawal netId txBody
  runTestOnSignal $ validateWrongNetworkInDirectDeposit netId txBody
  runTestOnSignal $ Alonzo.validateWrongNetworkInTxBody netId txBody

  if isValid
    then do
      newState <-
        Shelley.updateUTxOStateNoFees
          pp
          utxoState
          txBody
          certState
          (utxosGovState utxoState)
          (tellEvent . TotalDeposits (hashAnnotated txBody))
          (\a b -> tellEvent $ TxUTxODiff a b)
      pure $ newState & utxosDonationL <>~ txBody ^. treasuryDonationTxBodyL
    else
      pure utxoState

-- | Validate that total execution units (all transactions) do not exceed block limit.
-- ∑(tx ∈ txs)(totExunits tx) ≤ maxBlockExUnits pp
validateExUnits ::
  forall era.
  ( AlonzoEraTx era
  , InjectRuleFailure "BBODY" AlonzoBbodyPredFailure era
  ) =>
  StrictSeq.StrictSeq (Tx TopTx era) ->
  -- | Max block exunits protocol parameter.
  ExUnits ->
  Rule (EraRule "BBODY" era) 'Transition ()
validateExUnits txs ppMax =
  let txTotal = foldMap totExUnits txs
   in pointWiseExUnits (<=) txTotal ppMax
        ?! injectFailure
          ( TooManyExUnits $
              Mismatch
                { mismatchSupplied = txTotal
                , mismatchExpected = ppMax
                }
          )
