### Title
`respond_verify_foreign_tx` Does Not Verify `payload_hash` Derives From the Original Request — (`crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies only that the MPC signature is valid over the caller-supplied `response.payload_hash`, but never checks that `payload_hash` was actually computed from the pending `request`. A single Byzantine attested participant (below signing threshold) can replay a legitimately-produced `{payload_hash, signature}` pair from one settled request into any other pending `verify_foreign_transaction` request, causing the contract to deliver a forged foreign-chain attestation to the waiting caller.

### Finding Description

When a user calls `verify_foreign_transaction`, the contract stores the request and yields execution waiting for MPC nodes to call `respond_verify_foreign_tx`. The response carries two fields: `payload_hash` (the hash of `(request, extracted_values)` that the nodes observed on-chain) and `signature` (the MPC threshold signature over that hash).

In `respond_verify_foreign_tx` the contract performs only one cryptographic check:

```rust
// crates/contract/src/lib.rs  ~L726-L733
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
```

It confirms that `signature` is a valid ECDSA signature over `payload_hash` under the domain's root public key. It does **not** recompute the expected `payload_hash` from the `request` argument and compare. The `request` is used only as a lookup key to find the pending yield:

```rust
// crates/contract/src/lib.rs  ~L749-L753
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
```

Contrast this with the regular `respond` path, where the payload is always taken from the stored request, never from the node's response:

```rust
// crates/contract/src/lib.rs  ~L600-L607
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
)
```

Because `payload_hash` is entirely node-supplied in the foreign-tx path, a single attested participant can:

1. Observe a legitimately completed `respond_verify_foreign_tx(request_A, {payload_hash_A, sig_A})` on-chain (all data is public).
2. Find any other pending `verify_foreign_transaction` request `request_B`.
3. Call `respond_verify_foreign_tx(request_B, {payload_hash_A, sig_A})`.
4. The contract checks: `sig_A` valid over `payload_hash_A` → **true**. It resolves `request_B`'s yield with `{payload_hash_A, sig_A}`.
5. The caller of `request_B` receives a signed attestation whose `payload_hash` encodes the data of a completely different foreign-chain transaction.

The same `{payload_hash_A, sig_A}` pair can be replayed into every pending request of the same domain, because the contract keeps no record of which `payload_hash` values have already been consumed.

### Impact Explanation

The `verify_foreign_transaction` flow exists specifically to let NEAR contracts react to foreign-chain events without a trusted relayer. The signed `{payload_hash, signature}` tuple is the on-chain proof that a specific foreign transaction was observed. Downstream bridge contracts (e.g., Omnibridge inbound flow) consume this proof to release funds or mint tokens.

By replaying a stale or unrelated `payload_hash`, a Byzantine participant can cause a downstream contract to receive a cryptographically valid MPC signature over the wrong foreign-chain data. If the downstream contract does not independently recompute the expected `payload_hash` from the original request (a step the NEAR MPC contract itself omits), it will accept the forged attestation and execute the corresponding action (fund release, token mint, etc.), constituting a forged foreign-chain verification and potential double-spend or theft.

This maps to: **High — forged foreign-chain verification / cross-chain replay that causes invalid bridge execution.**

### Likelihood Explanation

- The attacker needs only to be a single attested MPC participant (Byzantine participant strictly below the signing threshold). No threshold collusion is required.
- All inputs needed for the replay (`payload_hash`, `signature`, pending request IDs) are publicly visible on-chain.
- The attacker does not need to break any cryptography; they reuse a legitimately produced signature.
- The attack is repeatable: one captured `{payload_hash, sig}` pair can be replayed into every concurrently pending request of the same domain.

### Recommendation

Inside `respond_verify_foreign_tx`, recompute the expected `payload_hash` from the `request` argument (using the same `ForeignTxSignPayload::compute_msg_hash` logic the nodes use) and assert it equals `response.payload_hash` before accepting the response:

```rust
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(), // nodes must also submit extracted values
}).compute_msg_hash()?;

if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Alternatively, include the `payload_hash` in the pending-request map key so that a mismatched hash simply fails the lookup.

### Proof of Concept

1. Alice submits `verify_foreign_transaction(request_A)` (Bitcoin tx `0xAA`). MPC nodes respond legitimately: `respond_verify_foreign_tx(request_A, {payload_hash_A, sig_A})`. Alice's yield resolves. `payload_hash_A` and `sig_A` are now public on-chain.

2. Bob submits `verify_foreign_transaction(request_B)` (Bitcoin tx `0xBB`). The yield is pending.

3. Mallory (a single attested participant) calls:
   ```
   respond_verify_foreign_tx(request_B, {payload_hash_A, sig_A})
   ```

4. Contract execution in `respond_verify_foreign_tx`:
   - `assert_caller_is_attested_participant_and_protocol_active()` → passes (Mallory is attested).
   - `verify_ecdsa_signature(sig_A, payload_hash_A, root_pk)` → **passes** (legitimate signature).
   - `resolve_yields_for(pending_verify_foreign_tx_requests, request_B, {payload_hash_A, sig_A})` → Bob's yield is resolved.

5. Bob's contract receives `{payload_hash_A, sig_A}` — a valid MPC signature over Bitcoin tx `0xAA`'s data, not `0xBB`. If Bob's downstream bridge contract trusts the MPC signature without re-deriving the expected `payload_hash`, it will act on forged attestation data.
