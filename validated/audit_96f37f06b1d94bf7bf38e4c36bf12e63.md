### Title
`view_access_keys` Returns Successful Empty List for Non-Existent Accounts Without Existence Check - (File: runtime/runtime/src/state_viewer/mod.rs)

### Summary

`TrieViewer::view_access_keys` does not verify that the queried account exists before iterating its trie prefix. For any non-existent account ID, the trie iterator finds no keys and returns `Ok(vec![])` — a successful empty list — instead of an `UnknownAccount` error. This is exposed directly through the public `query` RPC (`ViewAccessKeyList`) and `EXPERIMENTAL_view_access_key_list`, allowing any unprivileged caller to receive a misleading "account has no keys" response for an account that does not exist at all.

### Finding Description

`TrieViewer::view_access_keys` in `runtime/runtime/src/state_viewer/mod.rs` iterates the trie prefix for access keys belonging to `account_id` without first calling `get_account` to verify the account exists: [1](#0-0) 

Compare this to `view_state`, which explicitly checks account existence before proceeding: [2](#0-1) 

And `view_account`, which returns `AccountDoesNotExist` for missing accounts: [3](#0-2) 

For a non-existent account, the trie prefix iterator yields zero results, so `view_access_keys` returns `Ok(vec![])`. The `ViewAccessKeyError` enum used by this function has no `AccountDoesNotExist` variant: [4](#0-3) 

This means the `UnknownAccount` variant defined in `RpcViewAccessKeyListError` is unreachable through this code path: [5](#0-4) 

The call chain from the public RPC to the broken function is:

1. `EXPERIMENTAL_view_access_key_list` or `query` (ViewAccessKeyList) → `view_access_key_list_local`
2. → `ClientQuery` → `NightshadeRuntime::query` → `ViewAccessKeyList` branch
3. → `self.view_access_keys(...)` → `TrieViewer::view_access_keys`
4. Returns `Ok(vec![])` for non-existent account → `AccessKeyList { keys: [] }` success response [6](#0-5) [7](#0-6) 

### Impact Explanation

Any unprivileged RPC caller querying `ViewAccessKeyList` for a non-existent account ID receives a successful JSON-RPC response with `{ "keys": [] }` instead of an `UnknownAccount` error. This corrupts the RPC result: callers cannot distinguish between "account exists with no keys" and "account does not exist." Applications relying on this endpoint to verify account existence (e.g., wallets, bridges, indexers) will silently treat non-existent accounts as existing-but-keyless, potentially leading to incorrect authorization decisions or misleading UI state.

### Likelihood Explanation

Trivially reachable by any unprivileged user via a single public JSON-RPC call with any non-existent account ID. No special privileges, transactions, or chain state are required.

### Recommendation

Add an account existence check at the start of `view_access_keys`, mirroring the pattern used in `view_state`:

```rust
pub fn view_access_keys(
    &self,
    state_update: &TrieUpdate,
    account_id: &AccountId,
) -> Result<Vec<(PublicKeyHandle, AccessKey)>, errors::ViewAccessKeyError> {
    // Check account existence first
    if get_account(state_update, account_id)?.is_none() {
        return Err(errors::ViewAccessKeyError::AccountDoesNotExist {
            requested_account_id: account_id.clone(),
        });
    }
    // ... existing iteration logic
}
```

Add `AccountDoesNotExist` to `ViewAccessKeyError` and map it to `QueryError::UnknownAccount` in `from_view_access_key_error`.

    pub fn view_access_keys(
        &self,
        state_update: &TrieUpdate,
        account_id: &AccountId,
    ) -> Result<Vec<(PublicKeyHandle, AccessKey)>, errors::ViewAccessKeyError> {
        let prefix = trie_key_parsers::get_raw_prefix_for_access_keys(account_id);
        let access_keys =
            state_update
                .iter(&prefix)?
                .map(|key| {
                    let key = key?;
                    let key_handle = parse_key_handle_from_access_key_key(&key, account_id)
                        .map_err(|_| errors::ViewAccessKeyError::InternalError {
                            error_message: "Unexpected invalid access key from iterator"
                                .to_string(),
                        })?;
                    if let Some(_index) =
                        parse_nonce_index_from_gas_key_key(&key, account_id, &key_handle).map_err(
                            |_| errors::ViewAccessKeyError::InternalError {
                                error_message: "could not parse nonce index".to_string(),
                            },
                        )?
                    {
                        // This is a gas key nonce, skip it.
                        return Ok(None);
                    }
                    let access_key = near_store::get_access_key_by_handle(
                        state_update,
                        account_id,
                        &key_handle,
                    )?
                    .ok_or_else(|| {
                        near_primitives::errors::StorageError::StorageInconsistentState(format!(
                            "iterator yielded an access-key trie key with no value: {key_handle}"
                        ))
                    })?;

                    Ok(Some((key_handle, access_key)))
                })
                .filter_map_ok(|x| x)
                .collect::<Result<Vec<_>, errors::ViewAccessKeyError>>();
        access_keys
    }
