### Title
Malicious Attested Participant Can Supply Arbitrary `payload_hash` in `respond_verify_foreign_tx` to Deliver Forged Foreign-Chain Verification Results — (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` accepts a caller-supplied `response.payload_hash` and only verifies that `response.signature` is a valid ECDSA signature over that hash under the root public key. It does **not** verify that `response.payload_hash` was correctly derived from the original `request` data. A single malicious attested participant (the leader) can replay a valid signature from any prior signing operation, pairing it with the matching `payload_hash` from that prior operation, to resolve a live pending foreign-chain verification request with fabricated data.

### Finding Description

In `respond_verify_foreign_tx` at `crates/contract/src/lib.rs` lines 691–754, the contract performs the following check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
).is_ok()
``` [1](#0-0) 

The contract verifies only that `response.signature` is valid for `response.payload_hash` under the root key. It does **not** verify that `response.payload_hash` equals `ForeignTxSignPayload::V1 { request: request.request.clone(), values: <actual_extracted_values> }.compute_msg_hash()`.

The correct `payload_hash` is defined as:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))
``` [2](#0-1) 

The `values` (extracted values from the foreign chain) are off-chain data known only to the MPC nodes. The contract has no independent way to compute the expected `payload_hash` from the `request` alone. This creates a trust gap: the contract trusts the responding node to supply the correct `payload_hash`, but does not enforce this binding.

The `response.payload_hash` is entirely caller-controlled. A malicious leader node that has previously participated in any legitimate threshold signing operation possesses a valid `(payload_hash_old, signature_old)` pair. Because the signature is verified against the root key with no tweak (unlike `respond`, which uses a derived key), any previously produced valid signature remains valid indefinitely for its corresponding hash. [3](#0-2) 

Compare this to the client-side SDK verifier (`crates/near-mpc-sdk/src/foreign_chain.rs`), which **does** independently recompute the expected hash and checks `expected_payload_hash == response.payload_hash`: [4](#0-3) 

This check exists only in the off-chain SDK, not in the on-chain contract. The contract itself delivers the response unconditionally once the signature check passes.

### Impact Explanation

**High.** A single malicious attested participant acting as leader can:

1. Replay `(payload_hash_old, signature_old)` from any prior legitimate signing operation.
2. Call `respond_verify_foreign_tx` for a live pending request with `payload_hash = payload_hash_old` and `signature = signature_old`.
3. The contract's signature check passes (the signature is genuinely valid for `payload_hash_old` under the root key).
4. `resolve_yields_for` removes the pending request from `pending_verify_foreign_tx_requests` and delivers the fabricated response to all waiting callers. [5](#0-4) 

**Consequence 1 — Forged verification result**: Any bridge contract or caller that does not independently recompute the expected `payload_hash` from the original request and extracted values will accept the fabricated response as a valid attestation of foreign-chain state. This enables invalid bridge execution (e.g., releasing funds on NEAR for a foreign transaction that was never finalized, or for a different transaction entirely).

**Consequence 2 — Permanent denial of service on the request**: Once `resolve_yields_for` drains the pending queue for the request, the legitimate response (when honest nodes eventually submit it) finds no pending entry and fails with `RequestNotFound`. The request is permanently unresolvable.

This matches the allowed High impact: *"Cross-chain replay, forged foreign-chain verification, light-client-style verification bypass, or participant/attestation authorization bypass that causes invalid bridge execution or double-spend conditions."*

### Likelihood Explanation

**Medium.** The attacker must be a single attested participant (a legitimate MPC node that has passed TEE attestation). This is a realistic threat: the protocol explicitly models Byzantine participants below the signing threshold. The attacker needs only to have participated in any prior signing operation to possess a reusable `(payload_hash, signature)` pair. No threshold collusion is required. The attack is executable by a single node acting as the leader for a foreign-tx signing round.

### Recommendation

1. **Bind `payload_hash` to the request on-chain**: Include the `ForeignChainRpcRequest` (the chain query) as a prefix in the signed payload and verify on-chain that `response.payload_hash` encodes the correct `request`. Since the contract cannot know the extracted `values`, at minimum verify that the hash commits to the correct `request` component (e.g., by requiring nodes to submit `values` separately and computing the hash on-chain, or by using a two-field response structure).

2. **Require threshold agreement on `payload_hash`**: Rather than accepting `payload_hash` from a single leader, require threshold-many attested participants to submit the same `payload_hash` before resolving the request. This mirrors the threshold requirement for the signature itself.

3. **Short-term mitigation**: Enforce in the contract that `response.payload_hash` is not a hash that has been used to resolve any prior request (a replay registry), preventing reuse of old `(payload_hash, signature)` pairs across different requests.

### Proof of Concept

**Setup**: 4-node MPC network, threshold = 3. Node 4 is malicious.

1. **Step 1**: A legitimate `verify_foreign_transaction` request R1 (Bitcoin tx_id = `[0xAA; 32]`) is submitted and processed. The threshold protocol produces `sig_old` over `H_old = SHA256(borsh(ForeignTxSignPayload::V1 { request: R1, values: [BlockHash([0xBB; 32])] }))`. Node 4 records `(H_old, sig_old)`.

2. **Step 2**: A new `verify_foreign_transaction` request R2 (Bitcoin tx_id = `[0xCC; 32]`) is submitted. It enters `pending_verify_foreign_tx_requests`.

3. **Step 3**: Node 4 (malicious leader for R2) calls `respond_verify_foreign_tx(request=R2, response={payload_hash: H_old, signature: sig_old})`.

4. **Step 4**: The contract checks `verify_ecdsa_signature(sig_old, H_old, root_pk)` → **passes** (sig_old is genuinely valid for H_old).

5. **Step 5**: `resolve_yields_for` removes R2 from `pending_verify_foreign_tx_requests` and delivers `{payload_hash: H_old, signature: sig_old}` to all callers waiting on R2.

6. **Step 6**: The bridge contract for R2 receives a response claiming the MPC network attested to `H_old` (which encodes data from R1, not R2). If the bridge contract does not independently recompute the expected hash, it accepts this as valid attestation for tx_id `[0xCC; 32]`.

7. **Step 7**: When honest nodes later submit the correct response for R2, `resolve_yields_for` finds no pending entry and returns `RequestNotFound`. R2 is permanently unresolvable. [6](#0-5) [7](#0-6)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
