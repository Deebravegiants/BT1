### Title
`respond_verify_foreign_tx` Does Not Bind `response.payload_hash` to the Submitted `request`, Enabling Cross-Request Replay by a Single Byzantine Participant - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` function in the MPC contract verifies that the MPC-produced signature is valid over `response.payload_hash`, but it never verifies that `response.payload_hash` was actually derived from the `request` that is being resolved. A single Byzantine attested participant (below threshold) can replay a previously computed threshold signature — produced for a completely different foreign-chain request — to resolve any pending `verify_foreign_transaction` request with fabricated extracted values.

---

### Finding Description

The `respond_verify_foreign_tx` function accepts two independent arguments: a `request` (the original user-submitted `VerifyForeignTransactionRequest`) and a `response` (containing `payload_hash` and `signature`). The contract performs two checks:

1. The caller is an attested participant.
2. The `signature` is valid over `response.payload_hash` using the MPC network's root public key. [1](#0-0) 

What is **not** checked: that `response.payload_hash` is the canonical hash of `ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: request.request, values: <extracted_values> })` for the specific `request` being resolved.

The `payload_hash` is supposed to be `SHA-256(borsh(ForeignTxSignPayload))` where the payload binds together the original `ForeignChainRpcRequest` and the extracted values the MPC nodes observed on-chain. [2](#0-1) 

Because the contract cannot reconstruct the extracted values (they are computed off-chain by MPC nodes querying the foreign chain), it cannot independently verify this binding. The contract simply trusts whatever `payload_hash` the responding participant supplies, as long as a valid threshold signature exists over it.

The `request` key is used only to look up and drain the pending yield queue: [3](#0-2) 

The `response` (including the unverified `payload_hash`) is serialized and returned verbatim to the user's callback.

---

### Impact Explanation

A single Byzantine attested participant can execute a **cross-request replay**:

1. The MPC network legitimately computes a threshold signature for request A, producing `(payload_hash_A, signature_A)`. This is a public on-chain value.
2. A separate user submits request B (same `domain_id`, different `ForeignChainRpcRequest`).
3. The Byzantine participant calls `respond_verify_foreign_tx(request_B, { payload_hash_A, signature_A })`.
4. The contract verifies: is `signature_A` valid over `payload_hash_A` using the MPC root key? **Yes** — the signature is genuine.
5. The contract resolves request B's pending yield with `{ payload_hash_A, signature_A }`.
6. The user who submitted request B receives a `VerifyForeignTransactionResponse` whose `payload_hash` corresponds to request A's extracted values, not request B's.

Any user contract that does not independently reconstruct and compare the expected `payload_hash` (using `ForeignChainSignatureVerifier::verify_signature` from the SDK) will accept this forged response as a valid MPC-attested foreign-chain verification. [4](#0-3) 

The SDK-side verifier does perform this check, but the on-chain contract does not enforce it, leaving user contracts that omit the SDK check exposed to forged foreign-chain verification results. This maps directly to the **High** impact class: *forged foreign-chain verification that causes invalid bridge execution*.

---

### Likelihood Explanation

- Threshold signatures are public on-chain values; any attested participant can observe and replay them.
- Only **one** Byzantine attested participant is required — no threshold collusion needed.
- The attacker does not need to forge a new signature; replaying any previously finalized response suffices.
- All foreign-chain verification requests sharing the same `domain_id` are vulnerable to cross-request replay from one another.
- The window of opportunity exists for every pending request that has not yet been resolved.

---

### Recommendation

The contract should reconstruct the expected `payload_hash` from the `request` and the extracted values provided in the response, and verify it matches `response.payload_hash` before accepting the response. Since the contract cannot independently fetch foreign-chain data, the recommended mitigation is to include the extracted values in the `respond_verify_foreign_tx` call and have the contract recompute and verify the hash:

```rust
// In respond_verify_foreign_tx, after verifying the signature:
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response_values.clone(), // caller must supply extracted values
});
let expected_hash = expected_payload.compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Alternatively, the `VerifyForeignTransactionResponse` should be extended to include the extracted values so the contract can verify the binding on-chain, analogous to how `ForeignChainSignatureVerifier::verify_signature` works in the SDK.

---

### Proof of Concept

1. Alice submits `verify_foreign_transaction(request_B)` for Bitcoin tx `[0xBB; 32]`.
2. The MPC network previously processed request A (Bitcoin tx `[0xAA; 32]`) and produced `(payload_hash_A, signature_A)` — visible on-chain.
3. Mallory (a single Byzantine attested participant) calls:
   ```
   respond_verify_foreign_tx(
       request = request_B,          // correct pending request key
       response = {
           payload_hash: payload_hash_A,   // hash of request A's payload
           signature: signature_A,          // valid threshold sig over payload_hash_A
       }
   )
   ```
4. The contract checks: `verify_ecdsa_signature(signature_A, payload_hash_A, mpc_root_key)` → **valid**.
5. Alice's callback receives `{ payload_hash_A, signature_A }`.
6. Alice's contract, if it does not call `ForeignChainSignatureVerifier::verify_signature`, accepts this as a valid attestation that Bitcoin tx `[0xBB; 32]` was confirmed — but the MPC network actually attested to tx `[0xAA; 32]`. [5](#0-4) [6](#0-5)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
