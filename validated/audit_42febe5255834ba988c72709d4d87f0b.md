### Title
Missing Lower-Bound Timestamp Check in `is_proposal_init_valid` Allows Proposals with Arbitrarily Stale Block Timestamps — (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_proposal_init_valid` enforces only a **future** bound on the proposed block timestamp (`timestamp ≤ now + window`) but has **no past bound** (`timestamp ≥ now − window`). A malicious proposer can submit a `ProposalInit` whose timestamp is arbitrarily far in the past (bounded only by the last committed block's timestamp), causing the validator to look up and accept historical L1 gas prices that may be far below current market rates. Because the validator re-derives L1 prices using the proposer-supplied timestamp, the mismatch check passes and the block is committed with wrong gas prices.

---

### Finding Description

In `is_proposal_init_valid` the two timestamp guards are:

```rust
// lower bound: must not precede the last block
if init_proposed.timestamp < last_block_timestamp { … }

// upper bound: must not be more than `window` seconds in the future
if init_proposed.timestamp > now + proposal_init_validation.block_timestamp_window_seconds { … }
``` [1](#0-0) 

There is no guard of the form `init_proposed.timestamp < now − block_timestamp_window_seconds`. The config field is documented as *"Maximum allowed deviation (seconds) of a proposed block's timestamp from the current time"*, which implies a symmetric window, but only the future half is enforced.

The same asymmetry appears in `try_sync`:

```rust
&& timestamp.0 >= last_block_timestamp
&& timestamp.0 <= now + self.config.static_config.block_timestamp_window_seconds
``` [2](#0-1) 

The `block_timestamp_window_seconds` default is **1 second** in production config: [3](#0-2) 

---

### Impact Explanation

After `is_proposal_init_valid` accepts the stale timestamp, `initiate_validation` converts it directly into `BlockInfo` and passes it to the batcher: [4](#0-3) 

The L1 gas price check inside `is_proposal_init_valid` calls `get_l1_prices_in_fri_and_wei` with `init_proposed.timestamp`: [5](#0-4) 

Because the validator uses the **proposer-supplied** timestamp to re-derive reference prices, a proposer that picks a past timestamp when L1 gas was cheap will have its claimed prices verified against those same cheap historical prices. The margin check then passes: [6](#0-5) 

The `L1GasPriceProvider` ring buffer retains data for many hours (`storage_limit` blocks × ~12 s/block). For any timestamp within that window the provider returns real historical data rather than an error: [7](#0-6) 

Result: a committed block carries a stale timestamp and artificially low `l1_gas_price_fri` / `l1_data_gas_price_fri` / `l1_gas_price_wei` / `l1_data_gas_price_wei`, causing every transaction in that block to be charged incorrect (under-priced) fees — a direct economic impact matching the **Critical: Incorrect fee/gas/L1 gas price effect with economic impact** category.

---

### Likelihood Explanation

The proposer role rotates among committee validators. Any validator that wins a proposer slot can craft a `ProposalInit` with a past timestamp. No special privilege beyond being a scheduled proposer is required. The attack is silent (no on-chain revert, no log anomaly beyond a stale timestamp) and repeatable every time the attacker is selected as proposer.

---

### Recommendation

Add a symmetric lower-bound check immediately after the existing upper-bound check in `is_proposal_init_valid`:

```rust
// Existing upper-bound check (future)
if init_proposed.timestamp > now + proposal_init_validation.block_timestamp_window_seconds {
    return Err(…"Timestamp is in the future"…);
}

// ADD: lower-bound check (past)
if now.saturating_sub(proposal_init_validation.block_timestamp_window_seconds)
    > init_proposed.timestamp
{
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "Timestamp is too far in the past: now={}, \
             block_timestamp_window_seconds={}, proposed={}",
            now,
            proposal_init_validation.block_timestamp_window_seconds,
            init_proposed.timestamp
        ),
    ));
}
```

Apply the same fix to the `try_sync` timestamp guard in `sequencer_consensus_context.rs`. [8](#0-7) 

---

### Proof of Concept

1. Attacker is scheduled as proposer for height H.
2. Attacker picks `T_past = last_block_timestamp + 1` — a timestamp potentially hours in the past.
3. Attacker queries its own `L1GasPriceProvider` for `T_past`; receives historical prices `P_old` (e.g., from a period of low L1 congestion).
4. Attacker broadcasts `ProposalInit { timestamp: T_past, l1_gas_price_fri: P_old, … }`.
5. Each honest validator runs `is_proposal_init_valid`:
   - `T_past >= last_block_timestamp` → **passes** (lower bound only checks last block).
   - `T_past <= now + 1` → **passes** (T_past is in the past, so trivially ≤ now + 1).
   - Validator fetches L1 prices for `T_past` from its own provider → gets the same `P_old`.
   - Margin check on `P_old` vs `P_old` → **passes**.
6. Block is committed with timestamp `T_past` and gas prices `P_old`.
7. All transactions in the block pay fees computed from `P_old` instead of current market prices.
