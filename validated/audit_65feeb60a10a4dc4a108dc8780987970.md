### Title
Hardcoded Test Private Key Used in Production `create_signature_manager()` Enables Validator Precommit-Vote and Identity Forgery — (`File: crates/apollo_signature_manager/src/lib.rs`)

---

### Summary

The production factory function `create_signature_manager()` unconditionally calls `LocalKeyStore::new_for_testing()`, which embeds a well-known ECDSA private key in the binary. The `SignatureManager` is deployed as an unauthenticated `RemoteComponentServer` bound to `0.0.0.0`. Any party that can reach the port — or simply read the source code — possesses the validator's signing key and can produce valid `sign_precommit_vote` and `sign_identification` signatures for arbitrary inputs, binding the wrong signer to every consensus artifact the key protects.

---

### Finding Description

**Root cause — hardcoded key shipped to production**

`crates/apollo_signature_manager/src/lib.rs` exports the production type alias and factory:

```rust
pub use LocalKeyStoreSignatureManager as SignatureManager;

// TODO(Elin): understand how key store would look in production …
pub fn create_signature_manager() -> SignatureManager {
    SignatureManager::new()          // ← calls new_for_testing() unconditionally
}
```

`LocalKeyStoreSignatureManager::new()` delegates to `LocalKeyStore::new_for_testing()`:

```rust
pub(crate) const fn new_for_testing() -> Self {
    const PRIVATE_KEY: PrivateKey = PrivateKey(Felt::from_hex_unchecked(
        "0x608bf2cdb1ad4138e72d2f82b8c5db9fa182d1883868ae582ed373429b7a133",
    ));
    …
    Self { private_key: PRIVATE_KEY, public_key: PUBLIC_KEY }
}
```

The private key is a compile-time constant embedded in every production binary.

**Signing surface exposed over an unauthenticated remote server**

The deployment preset `replacer_signature_manager.json` runs the manager as `LocalExecutionWithRemoteEnabled` with `bind_ip: "0.0.0.0"` and no authentication layer. The `RemoteSignatureManagerServer` accepts two request variants with no caller verification:

- `SignatureManagerRequest::SignPrecommitVote(BlockHash)` — returns `ecdsa_sign(private_key, H(PRECOMMIT_VOTE || block_hash))`
- `SignatureManagerRequest::SignIdentification(PeerId, Challenge)` — returns `ecdsa_sign(private_key, H(INIT_PEER_ID || peer_id || challenge))`

Any caller that can reach the port can request a valid ECDSA signature over an arbitrary `BlockHash` or `(PeerId, Challenge)` pair using the validator's key.

**Broken invariant**

The invariant is: *each validator's signing key is secret and unique; a precommit vote or identity proof is unforgeable without that key.* Both halves are broken: the key is public (source code), and the signing oracle is unauthenticated.

**Exact corrupted value**

Once precommit-vote signature verification is wired in (the state machine currently carries `// TODO(Asmaa): sign the vote` with `signature: RawSignature::default()`), any attacker can produce `(r, s) = ecdsa_sign(0x608bf2…, digest)` for any `BlockHash` they choose. The resulting `Vote { voter: validator_id, signature: forged_sig, proposal_commitment: attacker_chosen }` passes verification against the validator's public key `0x125d56b1…`, binding the wrong signer to an attacker-controlled block commitment.

For `sign_identification`, the same key lets an attacker answer any challenge on behalf of the validator's `PeerId`, enabling P2P-layer impersonation today.

---

### Impact Explanation

Matches: **High — Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

The `sign_precommit_vote` path directly produces the ECDSA signature that will be attached to consensus `Vote` messages. With the key public, an attacker can forge a precommit for any `proposal_commitment` value they choose. When signature verification is activated, a single forged precommit from the validator's key counts toward quorum for an attacker-chosen block, corrupting the decided block hash and all downstream state commitments.

---

### Likelihood Explanation

**High.** The private key is a hex literal in a public source file. No network access, no privilege, and no interaction with the running node is required to obtain it. The remote signing oracle further lowers the bar: an attacker inside the cluster network can call `SignPrecommitVote` directly without knowing the key at all.

---

### Recommendation

1. Remove `LocalKeyStore::new_for_testing()` from the production call path. `create_signature_manager()` must load the private key from a secret store (HSM, sealed Kubernetes Secret, or environment-injected file) and fail hard if none is configured.
2. Add authentication (mTLS or a shared secret) to the `RemoteSignatureManagerServer` so that only the consensus manager component can invoke signing operations.
3. Rotate the exposed key (`0x608bf2cdb1ad4138e72d2f82b8c5db9fa182d1883868ae582ed373429b7a133`) on all deployed nodes immediately.
