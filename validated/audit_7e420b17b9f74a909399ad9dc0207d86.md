### Title
ETH-Recipient Revert in EVM `finTransfer` Permanently Locks User Funds on NEAR — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The EVM `OmniBridge.finTransfer()` function sends native ETH to a user-controlled `payload.recipient` via a low-level call. If the recipient is a smart contract whose `receive()` function uses environment-dependent opcodes (e.g., `block.number`, `block.timestamp`) to conditionally revert, off-chain simulation will succeed while on-chain execution fails. Because the NEAR bridge has no cancel or refund path for pending transfers, the user's locked tokens become permanently unclaimable, and the relayer wastes gas on every retry.

---

### Finding Description

**Bug class:** State machine / finality — off-chain simulation success does not guarantee on-chain success.

**Attacker-controlled entry point:** The `recipient` field in `InitTransferMsg`, supplied by the user when calling `ft_on_transfer` on NEAR (or `initTransfer` on EVM for the reverse direction). This value is embedded verbatim into the signed `TransferMessagePayload` and later used on EVM.

**Root cause in production code:**

In `finTransfer`, after the nonce is marked used and the signature is verified, the contract dispatches ETH to the recipient via a low-level call: [1](#0-0) 

```solidity
if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();
}
```

`payload.recipient` is entirely user-supplied. If it is a smart contract whose `receive()` function uses `block.number`, `block.timestamp`, `COINBASE`, or any other environment-dependent opcode to conditionally revert, the following divergence occurs:

- **Off-chain simulation** (at block N, before the condition triggers): `call` succeeds → simulation reports success.
- **On-chain execution** (at block N+k, after the condition triggers): `call` reverts → `FailedToSendEther` is thrown.

Because Solidity reverts all state changes on failure, the nonce marking (`completedTransfers[payload.destinationNonce] = true`) is also rolled back. The relayer pays gas for the failed transaction and can retry, but every retry fails identically.

**Why the fund lock is irreversible:**

On the NEAR side, the user's tokens are locked in `pending_transfers` when `init_transfer` is called: [2](#0-1) 

The only mechanism to release those tokens is a successful EVM `finTransfer` followed by a `claim_fee` call on NEAR (which requires proof of the EVM `FinTransfer` event). There is no `cancel_transfer`, `refund_transfer`, or timeout-based reclaim function anywhere in the NEAR bridge contract: [3](#0-2) 

If `finTransfer` can never succeed, the tokens remain in `pending_transfers` indefinitely with no on-chain path to recover them.

---

### Impact Explanation

**Critical — Irreversible fund lock:** A user who sets `recipient` to a conditionally-reverting contract causes their own tokens to be permanently locked in the NEAR bridge. No admin function, timeout, or user-callable path exists to reclaim them. This satisfies: *"Irreversible fund lock, frozen redemption path, or permanently unclaimable user or protocol value in bridge flows."*

**Relayer harm (secondary):** The relayer is an unprivileged third party. Every retry of `finTransfer` costs gas with no reimbursement. The relayer cannot distinguish a permanently-failing recipient from a transiently-failing one without additional off-chain tooling.

---

### Likelihood Explanation

- The `recipient` field is a free-form string/address set by the user at transfer initiation time with no on-chain validation of whether it is a contract or what its `receive()` logic does.
- A user who wants to grief a relayer (or who accidentally uses a contract address that conditionally reverts) can trigger this with a single `ft_on_transfer` call on NEAR.
- The condition is reachable on every EVM chain the bridge supports (Ethereum, Arbitrum, Base, etc.).
- Likelihood is **Medium**: requires deliberate construction of a reverting recipient, but the entry path is fully permissionless.

---

### Recommendation

1. **Add a cancel/timeout path on NEAR.** Introduce a `cancel_transfer` function (callable by the transfer owner after a configurable timeout) that removes the entry from `pending_transfers` and refunds the locked tokens. This is the primary mitigation for the fund-lock impact.

2. **Pull-payment pattern for ETH on EVM.** Instead of pushing ETH to `payload.recipient` in `finTransfer`, credit the amount to a claimable balance mapping and let the recipient pull it. This eliminates the revert-on-receive attack surface entirely.

3. **Relayer-side simulation hardening.** Relayers should separately simulate the low-level ETH call (or ERC20 transfer) to `payload.recipient` using a static call before submitting `finTransfer`, and refuse to relay transfers whose recipient is a contract that reverts on receipt.

---

### Proof of Concept

```solidity
// Deploy on target EVM chain
contract MaliciousRecipient {
    uint256 public immutable triggerBlock;
    constructor(uint256 _triggerBlock) { triggerBlock = _triggerBlock; }
    receive() external payable {
        // Succeeds during simulation (block.number < triggerBlock)
        // Reverts on-chain once block.number >= triggerBlock
        require(block.number < triggerBlock, "griefed");
    }
}
```

**Attack steps:**

1. Deploy `MaliciousRecipient(block.number + 100)` on EVM.
2. On NEAR, call `ft_on_transfer` with `InitTransferMsg { recipient: "eth:0x<MaliciousRecipient>", ... }` for a native-ETH transfer. Tokens are locked in `pending_transfers`.
3. Relayer calls `sign_transfer` on NEAR → MPC signature issued.
4. Relayer simulates `finTransfer` at block N < triggerBlock → simulation succeeds.
5. Relayer submits `finTransfer` at block N ≥ triggerBlock → `FailedToSendEther` revert; relayer pays gas; nonce rolled back.
6. Every subsequent retry fails identically. User's tokens remain locked in NEAR `pending_transfers` with no recovery path. [4](#0-3) [5](#0-4)
