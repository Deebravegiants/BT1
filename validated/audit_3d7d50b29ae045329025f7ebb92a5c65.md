### Title
Duplicate `tls_public_key` in Participant Proposal Bypasses Per-Account TEE Attestation Check, Enabling Double Key-Share Acquisition - (File: `crates/contract/src/primitives/participants.rs`, `crates/contract/src/tee/tee_state.rs`)

---

### Summary

`Participants::validate()` checks for duplicate `AccountId` and `ParticipantId` values but never checks for duplicate `tls_public_key` values across participants. Simultaneously, `reverify_and_cleanup_participants` performs TEE attestation lookup keyed exclusively on `tls_public_key`, not on `(account_id, tls_public_key)`. A malicious current participant can therefore propose a resharing that lists two different NEAR accounts sharing the same TLS key. The TEE validation passes for both entries (both resolve to the same registered attestation), `validate_incoming_proposal` passes (distinct account IDs and participant IDs), and the proposal is accepted. After resharing, the attacker's single node holds two independent key shares, reducing the effective reconstruction threshold.

---

### Finding Description

**Root cause 1 — `Participants::validate()` omits TLS-key uniqueness:** [1](#0-0) 

The function inserts `AccountId` and `ParticipantId` into `BTreeSet`s and compares lengths, but `ParticipantInfo` (which contains `tls_public_key`) is never collected into a set. Two entries with distinct account IDs and participant IDs but identical `tls_public_key` values pass validation silently.

**Root cause 2 — `reverify_and_cleanup_participants` resolves attestations by TLS key alone:** [2](#0-1) 

For each proposed participant, the function calls `find_node_id_by_tls_key(&participant_info.tls_public_key)`. If two proposed participants share TLS key K, both resolve to the same stored `NodeId` (the one that legitimately registered K). `reverify_participants` is then called on that single `NodeId` and returns `TeeQuoteStatus::Valid` for both entries. The function never verifies that the resolved `node_id.account_id` matches the account being validated.

**Root cause 3 — `validate_incoming_proposal` does not check TLS-key uniqueness:** [3](#0-2) 

The loop iterates `(account_id, participant_id, info)` tuples and checks ID contiguity and account-ID uniqueness, but never collects `tls_public_key` values into a deduplication set.

**Call chain in `vote_new_parameters`:** [4](#0-3) 

1. `proposal.try_into_contract_type()` → `ThresholdParameters::new()` validates only the governance threshold, not participant TLS-key uniqueness.
2. `reverify_and_cleanup_participants` passes for both `alice.near` (TLS key K) and `alice2.near` (TLS key K) because both resolve to `alice.near`'s registered attestation.
3. `protocol_state.vote_new_parameters()` → `process_new_parameters_proposal()` → `validate_incoming_proposal()` passes because account IDs and participant IDs are distinct.
4. The vote is recorded.

**Why `add_participant` does not prevent this:**

`add_participant` rejects a second account attempting to register the same TLS key: [5](#0-4) 

However, this only prevents `alice2.near` from registering TLS key K in `tee_state`. It does not prevent `alice2.near` from appearing in a `vote_new_parameters` proposal with TLS key K already owned by `alice.near`. The TEE validation in `reverify_and_cleanup_participants` will still resolve K to `alice.near`'s attestation and return `Valid` for `alice2.near`.

---

### Impact Explanation

After the resharing completes, `alice2.near` is a full participant in the new epoch. During DKG/resharing, the attacker's node (identified by TLS key K) receives key-share material for both `alice.near`'s participant ID and `alice2.near`'s participant ID. The attacker therefore holds two independent Shamir shares of the distributed secret key.

In a `t`-of-`n` threshold scheme, the attacker with 2 shares needs only `t − 2` additional colluding participants to produce an unauthorized signature. At the minimum viable configuration (`t = 2`, `n = 3`), the attacker can sign unilaterally without any other participant — a complete threshold bypass. At larger configurations the effective security margin is proportionally reduced.

This matches: **Critical — Bypass of threshold-signature requirements or unauthorized access to MPC key shares, signing capability, or secret material that materially enables forgery or secret recovery.**

---

### Likelihood Explanation

- The attacker must already be a current participant (one legitimate slot).
- The proposal requires **all** proposed participants to vote (`new_num_participants == n_votes`). The attacker controls both `alice.near` and `alice2.near`, so they can cast both votes.
- Legitimate participants running automated node software vote for proposals without manually inspecting `tls_public_key` uniqueness across entries. No on-chain guard rejects the proposal.
- Creating a second NEAR account is trivial and free.
- The attack is fully on-chain and requires no TEE compromise, no key leakage, and no network-level capability.

Likelihood: **Medium** (requires threshold votes from honest participants who do not inspect TLS-key uniqueness, which is realistic for automated node software).

---

### Recommendation

1. **Add TLS-key uniqueness to `Participants::validate()`:**

```rust
let mut tls_keys: BTreeSet<&Ed25519PublicKey> = BTreeSet::new();
for (_, _, info) in &self.participants {
    if !tls_keys.insert(&info.tls_public_key) {
        return Err(InvalidCandidateSet::DuplicateTlsPublicKeys.into());
    }
}
```

2. **Verify account ownership in `reverify_and_cleanup_participants`:** After resolving `node_id` by TLS key, assert `node_id.account_id == account_id` before accepting the attestation as valid for that participant entry.

3. **Add a TLS-key uniqueness check in `validate_incoming_proposal`** as a defence-in-depth measure, mirroring the existing account-ID and participant-ID deduplication.

---

### Proof of Concept

**Setup:** Running contract with participants `[alice.near (ID=0, TLS=K), bob.near (ID=1, TLS=K2)]`, threshold `t=2`.

**Step 1:** Attacker (`alice.near`) creates NEAR account `alice2.near`.

**Step 2:** Attacker calls `submit_participant_info` for `alice.near` with TLS key K — succeeds, attestation stored under K.

**Step 3:** Attacker calls `submit_participant_info` for `alice2.near` with TLS key K — **rejected** (`TlsKeyOwnedByOtherAccount`). `alice2.near` has no attestation in `tee_state`.

**Step 4:** Attacker (as `alice.near`) calls `vote_new_parameters` with proposal:
```
participants: [
  alice.near  (ID=0, TLS=K),   // existing
  bob.near    (ID=1, TLS=K2),  // existing
  alice2.near (ID=2, TLS=K),   // NEW — same TLS key as alice.near
]
threshold: 2
```

**Step 5:** `reverify_and_cleanup_participants` iterates:
- `alice.near` (TLS=K): `find_node_id_by_tls_key(K)` → `alice.near`'s NodeId → `Valid` ✓
- `bob.near` (TLS=K2): → `Valid` ✓
- `alice2.near` (TLS=K): `find_node_id_by_tls_key(K)` → **`alice.near`'s NodeId** → `Valid` ✓ ← **bypass**

All 3 pass → `TeeValidationResult::Full`.

**Step 6:** `validate_incoming_proposal` checks: account IDs `{alice.near, bob.near, alice2.near}` — all distinct ✓. Participant IDs `{0,1,2}` — all distinct ✓. No TLS-key check. Passes.

**Step 7:** `alice.near`, `bob.near`, and `alice2.near` all vote (attacker controls `alice.near` and `alice2.near`; `bob.near` votes automatically). Proposal accepted; resharing begins.

**Step 8:** After resharing, the attacker's node (TLS key K) holds key shares for both participant ID 0 (`alice.near`) and participant ID 2 (`alice2.near`). With `t=2` and 2 shares, the attacker signs unilaterally. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    pub fn validate(&self) -> Result<(), Error> {
        let mut ids: BTreeSet<ParticipantId> = BTreeSet::new();
        let mut accounts: BTreeSet<AccountId> = BTreeSet::new();
        for (acc_id, pid, _) in &self.participants {
            accounts.insert(acc_id.clone());
            ids.insert(*pid);
            if self.next_id.get() <= pid.get() {
                return Err(InvalidCandidateSet::ParticipantIdNotLessThanNextId {
                    id: pid.get(),
                    next_id: self.next_id.get(),
                }
                .into());
            }
        }
        if ids.len() != self.len() {
            return Err(InvalidCandidateSet::DuplicateParticipantIds.into());
        }
        if accounts.len() != self.len() {
            return Err(InvalidCandidateSet::DuplicateAccountIds.into());
        }
        Ok(())
    }

    pub fn validate_incoming_proposal(&self, proposal: &ThresholdParameters) -> Result<(), Error> {
        // ensure the proposed threshold parameters are valid:
        // if performance issue, inline and merge with loop below
        proposal.validate()?;
        let mut old_by_id: BTreeMap<ParticipantId, AccountId> = BTreeMap::new();
        let mut old_by_acc: BTreeMap<AccountId, (ParticipantId, ParticipantInfo)> = BTreeMap::new();
        for (acc, id, info) in self.participants().participants() {
            old_by_id.insert(*id, acc.clone());
            old_by_acc.insert(acc.clone(), (*id, info.clone()));
        }
        let new_participants = proposal.participants().participants();
        let mut new_min_id = u32::MAX;
        let mut new_max_id = 0u32;
        let mut n_old = 0u64;
        for (new_account, new_id, new_info) in new_participants {
            match old_by_acc.get(new_account) {
                Some((old_id, old_info)) => {
                    if new_id != old_id {
                        return Err(InvalidCandidateSet::ParticipantIdChanged {
                            account_id: new_account.clone(),
                            old_id: old_id.get(),
                            new_id: new_id.get(),
                        }
                        .into());
                    }
                    if *new_info != *old_info {
                        return Err(InvalidCandidateSet::ParticipantInfoChanged {
                            account_id: new_account.clone(),
                        }
                        .into());
                    }
                    n_old += 1;
                }
                None => {
                    if let Some(existing_account) = old_by_id.get(new_id) {
                        return Err(InvalidCandidateSet::NewParticipantReusesOldId {
                            account_id: new_account.clone(),
                            new_id: new_id.get(),
                            existing_account_id: existing_account.clone(),
                        }
                        .into());
                    }
                    new_min_id = std::cmp::min(new_min_id, new_id.get());
                    new_max_id = std::cmp::max(new_max_id, new_id.get());
                }
            }
        }
        // assert there are enough old participants
        if n_old < self.threshold().value() {
            return Err(InvalidCandidateSet::InsufficientOldParticipants.into());
        }
        // ensure the new ids are contiguous and unique
        let n_new = proposal.participants().len() as u64 - n_old;
        if n_new > 0 {
            if n_new - 1 != (new_max_id - new_min_id) as u64 {
                return Err(InvalidCandidateSet::NewParticipantIdsNotContiguous.into());
            }
            if new_min_id != self.participants().next_id().get() {
                return Err(InvalidCandidateSet::NewParticipantIdsNotContiguous.into());
            }
            if new_max_id + 1 != proposal.participants().next_id().get() {
                return Err(InvalidCandidateSet::NewParticipantIdsTooHigh.into());
            }
        }
        Ok(())
    }

    pub fn vote_new_parameters(
        &mut self,
        prospective_epoch_id: EpochId,
        proposal: dtos::ProposedThresholdParameters,
    ) -> Result<(), Error> {
        Self::assert_caller_is_signer();
        let proposal: ProposedThresholdParameters = proposal.try_into_contract_type()?;
        log!(
            "vote_new_parameters: signer={}, proposal={:?}",
            env::signer_account_id(),
            proposal,
        );

        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);

        let validation_result = self.tee_state.reverify_and_cleanup_participants(
            proposal.participants(),
            tee_upgrade_deadline_duration,
        );

        let proposed_participants = proposal.participants();
        match validation_result {
            TeeValidationResult::Full => {
                if let Some(new_state) = self
                    .protocol_state
                    .vote_new_parameters(prospective_epoch_id, &proposal)?
                {
                    self.protocol_state = new_state;
                }
                Ok(())
            }
            TeeValidationResult::Partial {
                participants_with_valid_attestation,
            } => {
                let invalid_participants: Vec<_> = proposed_participants
                    .participants()
                    .iter()
                    .filter(|(account_id, _, _)| {
                        !participants_with_valid_attestation
                            .is_participant_given_account_id(account_id)
                    })
                    .collect();

                Err(InvalidParameters::InvalidTeeRemoteAttestation {
                    reason: format!(
                        "The following participants have invalid TEE status: {:?}",
                        invalid_participants
                    ),
                }
                .into())
            }
        }
    }
