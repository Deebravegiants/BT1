Audit Report

## Title
ETH Permanently Frozen When Withdrawal Recipient Contract Lacks `receive()` - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
The two-phase withdrawal design in `LRTWithdrawalManager` commits an irreversible rsETH burn and ETH redemption in `unlockQueue` (Phase 1), then attempts ETH delivery to the original `user` address in `_processWithdrawalCompletion` (Phase 2). If `user` is a contract without a `receive()` fallback, Phase 2 permanently reverts, the rsETH burned in Phase 1 is unrecoverable, and the corresponding ETH is permanently locked inside `LRTWithdrawalManager` with no admin escape path.

## Finding Description
**Phase 1 — `unlockQueue` (operator-triggered, committed):** [1](#0-0) 
`burnFrom` destroys rsETH held by the contract and `unstakingVault.redeem` pulls ETH into `LRTWithdrawalManager`. This transaction is fully committed and irreversible.

**Phase 2 — `_processWithdrawalCompletion` (user- or operator-triggered):** [2](#0-1) 
The function pops the nonce queue, deletes the request, decrements `unlockedWithdrawalsCount`, then calls `_transferAsset`.

**`_transferAsset` failure path:** [3](#0-2) 
When `to` is a contract without `receive()`, `payable(to).call{value: amount}("")` returns `false`, triggering `revert EthTransferFailed()`. The entire Phase 2 transaction reverts, restoring `withdrawalRequests[requestId]`, `unlockedWithdrawalsCount[asset]`, and the nonce queue — but the rsETH burned in Phase 1 is not restored.

Every subsequent call to `completeWithdrawal` or `completeWithdrawalForUser` routes through `_processWithdrawalCompletion` and sends to the same `user` address: [4](#0-3) 
Both paths are permanently broken for this user.

**`sweepRemainingAssets` is permanently blocked:** [5](#0-4) [6](#0-5) 
Because the Phase 2 revert restores `unlockedWithdrawalsCount[asset]` to its pre-call value (still > 0), `hasUnlockedWithdrawals` permanently returns `true`, and `sweepRemainingAssets` permanently reverts with `PendingWithdrawalsExist`. No other ETH egress function exists in the contract.

## Impact Explanation
**Critical — Permanent freezing of funds.** The user's rsETH is permanently destroyed with no compensation. The corresponding ETH is permanently locked inside `LRTWithdrawalManager`. No admin function exists to cancel the stuck request, redirect the payout, or recover the ETH. `sweepRemainingAssets` is also permanently blocked for the affected asset, compounding the impact across all future users of that asset.

## Likelihood Explanation
Any smart contract that holds rsETH and calls `initiateWithdrawal(ETH_TOKEN, …)` without a `receive()` fallback triggers this condition. This includes multisig wallets (e.g., Gnosis Safe with no ETH receiver module), DeFi aggregators, yield vaults, and other on-chain integrators — a realistic and growing class of callers for a liquid restaking protocol. No special privilege is required: the unprivileged depositor initiates the withdrawal, and the operator's routine `unlockQueue` call completes the trap. The operator's own NatSpec comment acknowledges the function is "Not expected to be used for ETH" but provides no guard preventing it. [7](#0-6) 

## Recommendation
**Short term:** Add an admin-only function to cancel a stuck withdrawal request and return the ETH to a configurable address, or add a `withdrawalRecipient` mapping allowing users to designate a separate payout address at `initiateWithdrawal` time.

**Long term:** Add a pre-flight ETH-receivability check at `initiateWithdrawal` time for ETH withdrawals (analogous to `_checkOnERC721Received`): attempt a zero-value call to `msg.sender` and revert with a descriptive error if it fails. This prevents the two-phase commitment from ever entering an unrecoverable state.

## Proof of Concept
```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract NoReceiveWallet {
    // No receive() — ETH delivery always fails

    function step1_initiateWithdrawal(address withdrawalManager, address rsETH, uint256 rsETHAmount) external {
        IERC20(rsETH).approve(withdrawalManager, rsETHAmount);
        ILRTWithdrawalManager(withdrawalManager).initiateWithdrawal(ETH_TOKEN, rsETHAmount, "");
        // rsETH is now held by LRTWithdrawalManager
    }

    // step2: operator calls unlockQueue(ETH_TOKEN, ...) — rsETH burned, ETH in LRTWithdrawalManager (committed)

    function step3_completeWithdrawal(address withdrawalManager) external {
        // Always reverts with EthTransferFailed — NoReceiveWallet cannot receive ETH
        ILRTWithdrawalManager(withdrawalManager).completeWithdrawal(ETH_TOKEN, "");
    }
    // Result: rsETH permanently burned, ETH permanently locked, sweepRemainingAssets permanently blocked
}
```

**Foundry fork test outline:**
1. Deploy `NoReceiveWallet`, fund with rsETH, call `initiateWithdrawal`.
2. Warp/roll past delay; operator calls `unlockQueue` — assert rsETH supply decreased.
3. Call `completeWithdrawal` — assert revert `EthTransferFailed`.
4. Assert `LRTWithdrawalManager` ETH balance unchanged and `hasUnlockedWithdrawals(ETH_TOKEN)` returns `true`.
5. Assert manager call to `sweepRemainingAssets(ETH_TOKEN)` reverts `PendingWithdrawalsExist`.
6. Assert no admin function exists to recover the ETH (enumerate all external/public functions).

```text
    function _processWithdrawalCompletion(address asset, address user, string calldata referralId) internal {
        if (userAssociatedNonces[asset][user].empty()) {
            revert NoWithdrawalRequests(user, asset);
        }

        // Retrieve and remove the oldest withdrawal request for the user.
        uint256 usersFirstWithdrawalRequestNonce = userAssociatedNonces[asset][user].popFront();
        // Ensure the request is already unlocked.
        if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();

        bytes32 requestId = getRequestId(asset, usersFirstWithdrawalRequestNonce);
        WithdrawalRequest memory request = withdrawalRequests[requestId];

        delete withdrawalRequests[requestId];

        // Check that the withdrawal delay has passed since the request's initiation.
        if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();

        unlockedWithdrawalsCount[asset]--;

        // If Aave integration is enabled and asset is ETH, withdraw from Aave if needed
        if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
            uint256 contractBalance = address(this).balance;
            if (contractBalance < request.expectedAssetAmount) {
                uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
                _withdrawFromAave(amountNeeded);

                // Verify we have sufficient balance after withdrawal
                uint256 balanceAfter = address(this).balance;
                if (balanceAfter < request.expectedAssetAmount) {
                    revert InsufficientLiquidityForWithdrawal();
                }
            }
        }

        _transferAsset(asset, user, request.expectedAssetAmount);

        emit ReferralIdEmitted(referralId);
        emit AssetWithdrawalFinalized(user, asset, request.rsETHUnstaked, request.expectedAssetAmount);
    }
```
