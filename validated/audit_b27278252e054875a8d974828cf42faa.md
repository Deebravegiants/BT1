### Title
Unprotected `init` Function Allows Any Caller to Seize Control of the MPC Contract Before Legitimate Initialization - (File: `crates/contract/src/lib.rs`)

### Summary
The `MpcContract::init` function carries no access-control guard. Any NEAR account can call it in the window between contract deployment and the legitimate operator's initialization transaction. A successful front-run lets the attacker install an arbitrary participant set, giving them full control of the MPC network's signing capability.

### Finding Description
The `init` function is the sole public entry point that transitions the contract from `NotInitialized` to `Running`. It is decorated with `#[init]` (which only prevents a second call once state exists) but carries **no `#[private]` attribute and no caller check**:

```rust
// crates/contract/src/lib.rs:1924-1926
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
``` [1](#0-0) 

Compare this with `init_running` and `migrate`, which are both correctly gated with `#[private]`:

```rust
// crates/contract/src/lib.rs:1976-1979
#[private]
#[init]
#[handle_result]
pub fn init_running(...)
``` [2](#0-1) 

```rust
// crates/contract/src/lib.rs:2060-2063
#[private]
#[init(ignore_state)]
#[handle_result]
pub fn migrate() -> Result<Self, Error> {
``` [3](#0-2) 

The deployment scripts explicitly separate deployment from initialization into two distinct transactions using `without-init-call`:

```bash
near contract deploy "$MPC_CONTRACT_ACCOUNT" use-file "$MPC_CONTRACT_PATH" \
  without-init-call ...
# ... then separately:
near contract call-function as-transaction "$MPC_CONTRACT_ACCOUNT" init \
  file-args "$INIT_ARGS_JSON" ...
``` [4](#0-3) 

This creates a race window in which any observer can submit their own `init` call and win.

Additionally, `init` calls `TeeState::with_mocked_participant_attestations`, which inserts `VerifiedAttestation::Mock(MockAttestation::Valid)` for every participant supplied by the caller. This means the attacker's injected participants are immediately treated as fully attested, bypassing the TEE verification requirement that normally gates governance actions:

```rust
// crates/contract/src/tee/tee_state.rs:131-139
tee_state.stored_attestations.insert(
    tls_public_key,
    NodeAttestation {
        node_id,
        verified_attestation: VerifiedAttestation::Mock(
            attestation::MockAttestation::Valid,
        ),
    },
);
``` [5](#0-4) 

### Impact Explanation
An attacker who wins the initialization race installs themselves as the sole participant set. Because mock attestations are granted automatically at `init` time, the attacker's accounts immediately satisfy `assert_caller_is_attested_participant_and_protocol_active`. The attacker can then:

1. Call `vote_add_domains` to register signing domains.
2. Drive key generation (`start_keygen_instance`, `vote_pk`) using their own nodes.
3. Respond to every user `sign()` / `request_app_private_key()` / `verify_foreign_transaction()` call with signatures produced by their own key material.

This constitutes **unauthorized threshold signature issuance and confidential key derivation** — the attacker issues valid-looking MPC signatures without any legitimate participant authorization. Users and downstream protocols that trust the contract's public key have no way to distinguish attacker-issued signatures from legitimate ones.

### Likelihood Explanation
NEAR transactions are publicly visible before finalization. An attacker monitoring the chain for a `deploy` transaction targeting the MPC contract account can immediately broadcast a competing `init` call. The two-transaction deployment pattern used in all deployment scripts (TEE cluster, localnet, testnet guides) guarantees the window exists on every fresh deployment or contract upgrade that resets state. No privileged access, leaked key, or threshold collusion is required — only the ability to submit a NEAR transaction.

### Recommendation
Add `#[private]` to `init`, mirroring the pattern already applied to `init_running` and `migrate`. In NEAR SDK, `#[private]` enforces `predecessor_account_id == current_account_id`, so initialization can only be triggered by the contract account itself (i.e., as part of an atomic `deploy_contract().function_call("init", ...)` batch). All deployment scripts should be updated to use `with-init-call` rather than the two-step `without-init-call` + separate `init` pattern.

### Proof of Concept
1. Attacker watches the NEAR blockchain for a transaction deploying the MPC contract WASM to `mpc-contract.near` using `without-init-call`.
2. Before the operator's `init` transaction is included, the attacker submits:
   ```json
   near contract call-function as-transaction mpc-contract.near init \
     json-args '{
       "parameters": {
         "threshold": 1,
         "participants": {
           "next_id": 1,
           "participants": [
             ["attacker.near", 0, {
               "tls_public_key": "<attacker_ed25519_key>",
               "url": "http://attacker.com"
             }]
           ]
         }
       }
     }' ...
   ```
3. The attacker's `init` call succeeds; the operator's call fails with "state already exists."
4. The contract is now in `Running` state with `attacker.near` as the sole participant, holding a `MockAttestation::Valid` entry in `TeeState`.
5. The attacker calls `vote_add_domains` (passes `assert_caller_is_attested_participant_and_protocol_active`), drives key generation, and begins responding to all user `sign()` requests — issuing signatures under attacker-controlled key material. [6](#0-5) [7](#0-6)

add_node_keys_from_file() {
  local keys_file="$1"
  log "Adding node keys to NEAR accounts using $keys_file"
  [ -f "$keys_file" ] || { err "Missing keys file at $keys_file. Run collect phase first."; exit 1; }

  jq -c '.[]' "$keys_file" | while read -r row; do
    local acct signer responder
    acct="$(echo "$row" | jq -r .account)"
    signer="$(echo "$row" | jq -r .signer_pk)"
    responder="$(echo "$row" | jq -r .responder_pk)"

    log "$acct: add signer key"
    near_add_key_skip_if_exists "$acct" "$signer" "signer"

    log "$acct: add responder key"
    near_add_key_skip_if_exists "$acct" "$responder" "responder"
  done
}

    pub(crate) fn with_mocked_participant_attestations(participants: &Participants) -> Self {
        let mut tee_state = Self::default();

        for (account_id, _, participant_info) in participants.participants() {
            let tls_public_key = participant_info.tls_public_key.clone();
            // TODO(#1087): replace account_public_key with a real account public
            // key passed in by the caller. `Participants` does not currently
            // carry the operator's account public key, so a mocked entry
            // cannot record the real one and we use the TLS key as a unique
            // per-participant placeholder. The mock keeps the
            // participant from being kicked out of an empty `TeeState` until
            // a real `submit_participant_info` call replaces it (keyed by
            // TLS), but any caller-facing check that compares
            // `signer_account_pk` against the stored key will fail until
            // then. #1087 tracks threading real attestations through
            // initialization so this sentinel can go away.
            let node_id = NodeId {
                account_id: account_id.clone(),
                tls_public_key: tls_public_key.clone(),
                // Use tls_public_key as account_public_key instead of hardcoded
                // Ed25519PublicKey::from([0u8; 32]) so that same account public
                // key isn't associated with different tls keys.
                // This is not a fix for above issue: #1087, which should be
                // addressed outside this PR.
                account_public_key: tls_public_key.clone(),
            };

            tee_state.stored_attestations.insert(
                tls_public_key,
                NodeAttestation {
                    node_id,
                    verified_attestation: VerifiedAttestation::Mock(
                        attestation::MockAttestation::Valid,
                    ),
                },
            );
        }

        tee_state
    }
