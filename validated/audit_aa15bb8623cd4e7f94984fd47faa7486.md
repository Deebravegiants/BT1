### Title
`vote_add_domains` Quorum Uses Live Participant Count Instead of a Cached Value, Allowing Resharing to Retroactively Satisfy In-Flight Domain-Addition Proposals - (File: `crates/contract/src/state/running.rs`)

---

### Summary

`vote_add_domains` requires unanimity — every current participant must vote — but the quorum denominator is read from the live `self.parameters.participants().len()` at the moment each vote is cast, not from a value snapshotted when the proposal was first submitted. Because `add_domains_votes` are carried over (filtered) into the new `RunningContractState` after a resharing, a governance-threshold majority can first accumulate votes for a domain addition, then reshare to remove the non-cooperating minority, and finally trigger the domain addition with the carried-over votes that now satisfy the reduced quorum — all without the removed participants' consent.

---

### Finding Description

`RunningContractState::vote_add_domains` records each participant's vote and then checks:

```rust
if self.parameters.participants().len() as u64 == n_votes {
    // transition to Initializing
}
``` [1](#0-0) 

The quorum is `self.parameters.participants().len()` — the **current, live** participant count — not a value frozen at the time the first vote for this proposal was recorded.

When a resharing completes, `RunningContractState::new` is called with the new parameters. It carries `add_domains_votes` forward, filtering out votes from participants who were removed:

```rust
let remaining_add_domain_votes =
    add_domains_votes.get_remaining_votes(parameters.participants());
``` [2](#0-1) 

`parameters_votes` is reset to `ThresholdParametersVotes::default()`, but `add_domains_votes` survives across epoch boundaries. [3](#0-2) 

This means the quorum denominator for an in-flight `vote_add_domains` proposal silently changes whenever the participant set changes via resharing.

---

### Impact Explanation

A governance-threshold majority (≥ 60 % of participants, the minimum allowed by `validate_threshold`) can bypass the unanimity requirement for domain additions:

1. They accumulate votes for a `vote_add_domains` proposal — enough to meet the governance threshold but not unanimity.
2. They use the same majority to execute `vote_new_parameters`, removing the non-cooperating minority via resharing.
3. After resharing, `add_domains_votes` retains exactly the votes of the remaining (majority) participants. The new quorum equals the new participant count, which equals the number of carried-over votes.
4. Any one of the remaining participants calls `vote_add_domains` with the same proposal; the check `participants.len() == n_votes` is now satisfied, and the contract transitions to `Initializing`.

The domain addition passes without the removed participants ever consenting, and without the remaining participants explicitly re-voting after the resharing. This breaks the production safety invariant that domain additions require unanimous agreement of all current participants, and it can be used to add a new signing domain (e.g., a new protocol or a domain with a reconstruction threshold at the governance-threshold floor) that a significant minority of participants objected to.

This maps to the **Medium** allowed impact: *"participant-state or contract execution-flow manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

The attack requires only that a governance-threshold majority (≥ 60 % of participants) coordinate two sequential governance actions: `vote_add_domains` and `vote_new_parameters`. Both actions are normal, unprivileged contract calls available to any current participant. No external dependency, TEE bypass, or network-level capability is needed. The minimum governance threshold enforced by `validate_threshold` is 60 %, so the attack is reachable whenever a 60 % coalition disagrees with the remaining 40 % about adding a domain. [4](#0-3) 

---

### Recommendation

Cache the participant count (or the full participant set identity) at the time the **first** vote for a given `vote_add_domains` proposal is recorded, and use that cached value as the quorum denominator for the lifetime of that proposal. This mirrors how `process_new_parameters_proposal` ties the quorum to the proposed participant count embedded in the proposal itself, rather than reading it from live state.

Concretely, `AddDomainsVotes` should store the participant count (or epoch id) at first-vote time alongside each proposal, and `vote_add_domains` should compare `n_votes` against that frozen value instead of `self.parameters.participants().len()`. [5](#0-4) 

---

### Proof of Concept

**Setup**: 10 participants, governance threshold = 6 (the minimum ≥ 60 % value).

**Step 1** — Six participants (P1–P6) call `vote_add_domains` with proposal `D` (e.g., a new Frost domain). `n_votes = 6`, quorum = `participants.len() = 10` → `6 ≠ 10`, no transition.

**Step 2** — The same six participants call `vote_new_parameters` proposing a new epoch that retains only P1–P6 (removing P7–P10). Six votes meet the governance threshold of 6 → resharing executes.

**Step 3** — After resharing completes, `RunningContractState::new` is called. `add_domains_votes.get_remaining_votes` keeps the 6 votes from P1–P6 (all still participants); votes from P7–P10 are dropped (they never voted for `D` anyway). New state: `add_domains_votes` has 6 entries for proposal `D`, `parameters.participants().len() = 6`.

**Step 4** — P1 calls `vote_add_domains(D)`. `AddDomainsVotes::vote` replaces P1's existing entry (count stays 6). Check: `6 == 6` → `true`. Contract transitions to `Initializing` for domain `D`. [6](#0-5) [7](#0-6) 

Domain `D` is now being key-generated despite P7–P10 never having consented to it, and despite the original quorum of 10 never having been reached.

    pub fn vote_add_domains(
        &mut self,
        domains: Vec<DomainConfig>,
    ) -> Result<Option<InitializingContractState>, Error> {
        if domains.is_empty() {
            return Err(DomainError::AddDomainsMustAddAtLeastOneDomain.into());
        }
        let num_participants = u64::try_from(self.parameters.participants().len())
            .expect("participant count fits in u64");
        for domain in &domains {
            validate_domain_purpose(domain)?;
            validate_domain_threshold(domain, num_participants)?;
        }
        // Keep trust assumptions consistent: a domain must never require more shares to
        // reconstruct than the GovernanceThreshold demands to govern. Route through the
        // canonical helper so the cross-domain invariant has a single source of truth.
        ThresholdParameters::validate_governance_against_reconstruction(
            num_participants,
            self.parameters.threshold(),
            max_reconstruction_threshold(&domains),
        )?;
        let participant = AuthenticatedParticipantId::new(self.parameters.participants())?;
        let n_votes = self.add_domains_votes.vote(domains.clone(), &participant);
        if self.parameters.participants().len() as u64 == n_votes {
            let new_domains = self.domains.add_domains(domains.clone())?;
            Ok(Some(InitializingContractState {
                generated_keys: self.keyset.domains.clone(),
                domains: new_domains,
                epoch_id: self.keyset.epoch_id,
                generating_key: KeyEvent::new(
                    self.keyset.epoch_id,
                    domains[0].clone(),
                    self.parameters.clone(),
                ),
                cancel_votes: BTreeSet::new(),
            }))
        } else {
            Ok(None)
        }
    }

    fn validate_threshold(n_shares: u64, k: Threshold) -> Result<(), Error> {
        if k.value() > n_shares {
            return Err(InvalidThreshold::MaxRequirementFailed {
                max: n_shares,
                found: k.value(),
            }
            .into());
        }
        if k.value() < MIN_THRESHOLD_ABSOLUTE {
            return Err(InvalidThreshold::MinAbsRequirementFailed.into());
        }
        let lower_relative_bound = governance_threshold_lower_relative_bound(n_shares);
        if k.value() < lower_relative_bound {
            return Err(InvalidThreshold::MinRelRequirementFailed {
                required: lower_relative_bound,
                found: k.value(),
            }
            .into());
        }
        let upper_relative_bound = governance_threshold_upper_relative_bound(n_shares);
        if k.value() > upper_relative_bound {
            return Err(InvalidThreshold::MaxRelRequirementFailed {
                max: upper_relative_bound,
                found: k.value(),
            }
            .into());
        }
        Ok(())
    }

pub struct AddDomainsVotes {
    pub(crate) proposal_by_account: BTreeMap<AuthenticatedParticipantId, Vec<DomainConfig>>,
}

impl AddDomainsVotes {
    /// Votes for the proposal, returning the total number of voters so far who
    /// have proposed the exact same domains to add.
    /// If the participant had voted already, this replaces the existing vote.
    pub fn vote(
        &mut self,
        proposal: Vec<DomainConfig>,
        participant: &AuthenticatedParticipantId,
    ) -> u64 {
        if self
            .proposal_by_account
            .insert(participant.clone(), proposal.clone())
            .is_some()
        {
            log!("removed old vote for signer");
        }
        let total = self
            .proposal_by_account
            .values()
            .filter(|&prop| prop == &proposal)
            .count() as u64;
        log!("total votes for proposal: {}", total);
        total
    }

    /// Filters out existing votes no longer in the participant set
    pub fn get_remaining_votes(&self, participants: &Participants) -> Self {
        let remaining_votes = self
            .proposal_by_account
            .iter()
            .filter(|&(participant_id, _vote)| {
                participants.is_participant_given_participant_id(&participant_id.get())
            })
            .map(|(participant_id, vote)| (participant_id.clone(), vote.clone()))
            .collect();
        AddDomainsVotes {
            proposal_by_account: remaining_votes,
        }
    }
