Audit Report

## Title
First-Depositor Inflation Attack via Bootstrap `rsETHPrice` Bypass and Uninitialized `pricePercentageLimit` — (File: `contracts/LRTOracle.sol`)

## Summary
`LRTOracle._updateRsETHPrice()` unconditionally sets `rsETHPrice = 1 ether` and returns early when `rsethSupply == 0`, skipping every price-validation guard. Because `pricePercentageLimit` is never set in `initialize()` and defaults to `0`, the price-increase ceiling is permanently disabled until an admin acts. Combined with an open ETH donation path into `LRTDepositPool` and the absence of a zero-mint revert in `_beforeDeposit`, an attacker can inflate `rsETHPrice` to an arbitrary value at deployment, causing depositors who pass `minRSETHAmountExpected = 0` to receive 0 rsETH for real ETH deposited — a direct theft of funds.

## Finding Description

**Root cause 1 — bootstrap bypass in `_updateRsETHPrice()`**

`LRTOracle.sol` lines 218–222:
```solidity
if (rsethSupply == 0) {
    rsETHPrice = 1 ether;
    highestRsethPrice = 1 ether;
    return;
}
```
When `rsethSupply == 0`, the function sets both prices to `1 ether` and returns immediately. Every downstream guard — the `pricePercentageLimit` ceiling, fee minting, and the downside auto-pause — is skipped entirely. `updateRSETHPrice()` is `public` and `whenNotPaused` only, so any caller can trigger this path.

**Root cause 2 — `pricePercentageLimit` defaults to `0`**

`LRTOracle.initialize()` (lines 64–68) sets only `lrtConfig` and emits an event; it never assigns `pricePercentageLimit`. The price-increase guard at lines 256–257 reads:
```solidity
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
```
Because `pricePercentageLimit == 0`, the short-circuit makes `isPriceIncreaseOffLimit` permanently `false` until an admin calls `setPricePercentageLimit()`. No price increase of any magnitude will revert for an unprivileged caller.

**Root cause 3 — open ETH donation path**

`LRTDepositPool.sol` line 58: `receive() external payable { }` accepts ETH from any address. `getETHDistributionData()` at line 480 returns `address(this).balance` as `ethLyingInDepositPool`, which is immediately included in `_getTotalEthInProtocol()` via `getTotalAssetDeposits(ETH_TOKEN)`. Donated ETH inflates `totalETHInProtocol` used by the oracle with no delay or access control.

**Root cause 4 — zero-rsETH mint silently accepted**

`_beforeDeposit` (lines 667–669) only reverts if `rsethAmountToMint < minRSETHAmountExpected`. When `minRSETHAmountExpected = 0` and `rsethAmountToMint = 0` (due to inflated price), `0 < 0` is `false` — no revert. The victim's ETH is accepted and `_mintRsETH(0)` is called, issuing nothing.

**Exploit chain:**

1. Attacker calls `updateRSETHPrice()` → `rsethSupply == 0` → bootstrap path sets `rsETHPrice = 1e18`, `highestRsethPrice = 1e18`, returns early.
2. Attacker calls `depositETH{value: 1}(0, "")` → mints 1 wei rsETH (100 % of supply).
3. Attacker sends `N` ETH directly to `LRTDepositPool` via `receive()` → `address(this).balance` grows by `N`.
4. Attacker calls `updateRSETHPrice()` again → `rsethSupply = 1`, `totalETHInProtocol ≈ N * 1e18`, `newRsETHPrice = N * 1e18 * 1e18 / 1 = N * 1e36`. `pricePercentageLimit == 0` → `isPriceIncreaseOffLimit = false` → no revert → `rsETHPrice = N * 1e36`.
5. Victim calls `depositETH{value: V}(0, "")` → `rsethAmountToMint = V * 1e18 / (N * 1e36) = 0` (rounds down) → 0 rsETH minted, `V` ETH absorbed.
6. Attacker redeems 1 wei rsETH (100 % of supply) → entitled to entire TVL ≈ `N + V` ETH. Net profit ≈ `V` ETH (donation `N` recovered in full).

## Impact Explanation

**Critical — direct theft of any user funds.** The victim's ETH is permanently absorbed into the protocol with zero rsETH issued. The attacker holds 100 % of the rsETH supply and can redeem it for the entire TVL. The donation is fully recovered; the victim's ETH is the attacker's profit. This matches the allowed impact class "Direct theft of any user funds, whether at-rest or in-motion."

## Likelihood Explanation

The attack window opens at deployment and closes only when an admin explicitly calls `setPricePercentageLimit()` with a non-zero value. There is no on-chain enforcement that this must happen before the first deposit. `updateRSETHPrice()` is public and permissionless. The only cost to the attacker is the ETH donation, which is recovered in full. Any depositor who passes `minRSETHAmountExpected = 0` — the default in many integrations and front-ends — is a viable victim. The attack is repeatable across any deployment of this contract.

## Recommendation

1. **Set a non-zero `pricePercentageLimit` in `initialize()`** (e.g., `1e16` for 1 %) so the price-increase ceiling is active from block 0.
2. **Revert in `_beforeDeposit` if `rsethAmountToMint == 0`** to prevent silent fund loss regardless of oracle state.
3. **Seed the protocol with a non-trivial initial deposit** (or use a virtual-offset pattern as described in the OpenZeppelin ERC-4626 inflation-attack documentation already present in this repo) so the first-depositor attack surface does not exist.
4. **Restrict `receive()`** to known callers (node delegators, reward receivers) to eliminate the open donation vector.
