Audit Report

## Title
Stale `rsETHPrice` Used in Deposit Minting Allows Theft of Unclaimed Yield from Existing rsETH Holders - (File: `contracts/LRTDepositPool.sol`, `contracts/LRTOracle.sol`)

## Summary

`LRTDepositPool.getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()`, a stored state variable that is only updated when `updateRSETHPrice()` is explicitly called. Because neither `depositETH()` nor `depositAsset()` refreshes this price before computing the mint amount, any deposit made while the stored price lags behind the true protocol NAV mints excess rsETH at the expense of existing holders. The attacker then triggers a price update, crystallising their surplus rsETH into real ETH value stolen from prior depositors.

## Finding Description

`getRsETHAmountToMint()` at `contracts/LRTDepositPool.sol` L520 computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.getAssetPrice(asset)` is a live call through `IPriceFetcher` (Chainlink-backed). `lrtOracle.rsETHPrice()` reads the storage variable declared at `contracts/LRTOracle.sol` L28, which is only written inside `_updateRsETHPrice()` (L313: `rsETHPrice = newRsETHPrice`).

`_updateRsETHPrice()` derives the correct price from `_getTotalEthInProtocol()` (L331–349), which sums live Chainlink prices multiplied by total asset deposits across all node delegators. Between two calls to `updateRSETHPrice()`, LST rebases, EigenLayer restaking rewards, and ETH validator rewards increase `totalETHInProtocol` while `rsethSupply` stays constant, so the true NAV per rsETH rises above the stored `rsETHPrice`.

Neither `depositETH()` (L76–93) nor `depositAsset()` (L99–118) calls `updateRSETHPrice()` or any equivalent internal refresh before invoking `_beforeDeposit()` → `getRsETHAmountToMint()`. There is no staleness check, no timestamp guard, and no forced price sync. The `pricePercentageLimit` guard (L256–266) only applies inside `_updateRsETHPrice()` itself and does not block deposits at a stale price; it can only delay the price update for non-managers, which extends the exploitable window.

Exploit flow:
1. Monitor `_getTotalEthInProtocol()` vs. `rsETHPrice * rsethSupply` for a profitable gap.
2. Call `depositETH{value: X}()` — minted rsETH is computed using the stale (lower) `rsETHPrice`, yielding more rsETH than the deposit's true ETH value warrants.
3. Call `updateRSETHPrice()` — the new price reflects the diluted supply; the attacker's surplus rsETH now represents ETH value extracted from prior holders.
4. Hold or redeem the surplus rsETH.

No privileged role is required at any step.

## Impact Explanation

**High — Theft of unclaimed yield.**

Existing rsETH holders accumulate yield as `totalETHInProtocol` grows between price updates. A depositor who acts before `updateRSETHPrice()` is called receives rsETH priced at the stale (lower) rate, effectively claiming a share of unrealised yield. After the price update, the attacker's surplus rsETH represents ETH value extracted from existing holders. The theft scales linearly with deposit size; a flash-loan-sized deposit can capture nearly all accrued-but-unrecorded rewards in a single block. This matches the allowed impact: **High. Theft of unclaimed yield.**

## Likelihood Explanation

**Medium.** `updateRSETHPrice()` is triggered off-chain (keeper/bot/manual). Any gap between reward accrual and the next price update is an exploitable window. The gap is predictable and monitorable entirely on-chain by comparing `_getTotalEthInProtocol()` to `rsETHPrice * rsethSupply`. No privileged access is required; `depositETH()` is fully public and callable by any EOA or contract. The attack is repeatable every reward accrual cycle.

## Recommendation

Call `_updateRsETHPrice()` (or expose an internal equivalent) at the start of `_beforeDeposit()` before `getRsETHAmountToMint()` is invoked. This ensures the mint calculation always uses a fresh NAV. Alternatively, compute the mint amount inline using a freshly derived price rather than reading the stored `rsETHPrice` state variable.

## Proof of Concept

**State before attack:**
```
totalETHInProtocol = 1100 ETH   (100 ETH staking rewards accrued, not yet reflected)
rsethSupply        = 1000 rsETH
rsETHPrice (stored)= 1.000 ETH  (stale — correct value is 1.100 ETH)
```

**Step 1 — Attacker calls `depositETH{value: 10_000 ETH}()`:**
```
rsethAmountToMint = 10_000e18 / 1.000e18 = 10_000 rsETH
correct amount at true price: 10_000 / 1.100 = 9_090.9 rsETH
surplus minted to attacker: 909.1 rsETH
```

**Step 2 — Anyone calls `updateRSETHPrice()`:**
```
totalETHInProtocol = 11_100 ETH
rsethSupply        = 11_000 rsETH
previousTVL        = 11_000 * 1.000 = 11_000 ETH
rewardAmount       = 100 ETH → protocol fee taken on this
newRsETHPrice      = 11_100 / 11_000 = 1.00909 ETH
```

**Step 3 — Attacker's position:**
```
10_000 rsETH × 1.00909 ETH = 10_090.9 ETH
Deposited: 10_000 ETH
Profit:       90.9 ETH  (≈ 90.9% of the 100 ETH accrued reward stolen)

Existing holders' loss:
1000 rsETH × 1.00909 ETH = 1_009.09 ETH
Should have been: 1000 × 1.100 = 1_100 ETH
Loss: 90.9 ETH
```

**Foundry fork test plan:**
1. Fork mainnet at a block where staking rewards have accrued since the last `updateRSETHPrice()` call.
2. Assert `_getTotalEthInProtocol() > rsETHPrice * rsethSupply` (gap exists).
3. Call `depositETH{value: 10_000 ether}(0, "")` from attacker address.
4. Record attacker rsETH balance.
5. Call `updateRSETHPrice()`.
6. Assert attacker rsETH balance × new `rsETHPrice` > 10_000 ether (profit confirmed).
7. Assert existing holder rsETH × new `rsETHPrice` < pre-attack expected value (loss confirmed).
