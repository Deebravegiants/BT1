### Title
Unvalidated `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Payload Substitution by a Single Byzantine Participant - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` accepts a caller-supplied `response.payload_hash` and only checks that the accompanying signature is valid over that hash using the root public key. It never validates that `payload_hash` is actually derived from the foreign chain transaction data in the accompanying `request`. A single malicious attested participant can replay a legitimate `(payload_hash, signature)` pair from a previously resolved request to resolve a completely different pending request, delivering a forged verification attestation to the waiting user.

### Finding Description

The `respond_verify_foreign_tx` function in `crates/contract/src/lib.rs` performs the following validation:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract verifies only that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the root public key. It does **not** verify that `response.payload_hash` is the hash of `ForeignTxSignPayload { request: request.request, values: <extracted_values> }` — i.e., that the payload hash is actually derived from the foreign chain transaction data in the `request` argument.

Contrast this with the regular `respond` function, which derives the expected payload directly from the request object itself:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
)
``` [2](#0-1) 

For `respond`, the payload comes from the request itself. For `respond_verify_foreign_tx`, the payload comes from the response — and is never cross-checked against the request.

After the signature check passes, the full `response` (including the unvalidated `payload_hash`) is serialized and delivered to all pending yields for the given `request` key:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [3](#0-2) 

The `payload_hash` that should be computed on-chain from the foreign chain data is instead accepted verbatim from the caller: [4](#0-3) 

### Impact Explanation

A single malicious attested participant can:

1. Observe a previously completed `respond_verify_foreign_tx` call for transaction `tx_A` on-chain, extracting the public `(payload_hash_A, signature_A)` pair.
2. Wait for a user to submit `verify_foreign_transaction(tx_B)`, creating a pending request.
3. Call `respond_verify_foreign_tx(request = tx_B_request, response = { payload_hash: payload_hash_A, signature: signature_A })`.
4. The contract accepts: `signature_A` is a valid signature over `payload_hash_A` under the root key.
5. The user's pending yield for `tx_B` is resolved with `payload_hash_A` and `signature_A` — tx_A's attestation data.

The user receives a `VerifyForeignTransactionResponse` that appears to be a valid MPC-signed attestation for `tx_B`, but actually encodes `tx_A`'s extracted values (block hash, log data, etc.). Any bridge contract or downstream application that trusts this attestation to authorize a cross-chain action will execute based on `tx_A`'s data, not `tx_B`'s — enabling forged foreign-chain verification and potential double-spend or invalid bridge execution.

This matches the **High** impact category: *"Forged foreign-chain verification, light-client-style verification bypass that causes invalid bridge execution or double-spend conditions."*

### Likelihood Explanation

- Requires only **one** malicious attested participant, well below the signing threshold.
- The `(payload_hash, signature)` replay material is entirely public on-chain from any prior successful `respond_verify_foreign_tx` call.
- No cryptographic forgery is needed; only a valid replay of existing material.
- The attacker does not need to race against honest nodes — they can call `respond_verify_foreign_tx` for `tx_B` before honest nodes do, or after honest nodes fail to respond in time.

### Recommendation

The contract should recompute the expected `payload_hash` from the `request` data and compare it against `response.payload_hash` before accepting the response. Since the extracted values are not stored on-chain, the contract cannot fully recompute the hash. However, it should at minimum bind the `payload_hash` to the request by requiring the MPC nodes to include the `request` data in the signed payload, or by storing a commitment to the expected payload hash at request submission time. Alternatively, follow the pattern of `respond`: derive the payload from the request, not from the response.

### Proof of Concept

1. User submits `verify_foreign_transaction({ request: bitcoin_tx_B, domain_id: D })` — a pending yield is created under key `tx_B_request`.
2. A prior call `respond_verify_foreign_tx(tx_A_request, { payload_hash: H_A, signature: S_A })` is visible on-chain.
3. Malicious participant calls:
   ```
   respond_verify_foreign_tx(
     request = tx_B_request,
     response = { payload_hash: H_A, signature: S_A }
   )
   ```
4. Contract checks: `verify_ecdsa_signature(S_A, H_A, root_pk)` → valid ✓
5. `resolve_yields_for(pending_verify_foreign_tx_requests, tx_B_request, serialize({H_A, S_A}))` → user's yield resolved.
6. User receives `{ payload_hash: H_A, signature: S_A }` — tx_A's attestation delivered as tx_B's verification result.

fn build_signature_request(
    request: &VerifyForeignTxRequest,
    foreign_tx_payload: &dtos::ForeignTxSignPayload,
) -> anyhow::Result<SignatureRequest> {
    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
        foreign_tx_payload.compute_msg_hash()?.into();
    let payload_bytes: BoundedVec<u8, ECDSA_PAYLOAD_SIZE_BYTES, ECDSA_PAYLOAD_SIZE_BYTES> =
        payload_hash.into();

    Ok(SignatureRequest {
        id: request.id,
        receipt_id: request.receipt_id,
        payload: Payload::Ecdsa(payload_bytes),
        tweak: Tweak::new([0u8; 32]),
        entropy: request.entropy,
        timestamp_nanosec: request.timestamp_nanosec,
        domain: request.domain_id,
    })
