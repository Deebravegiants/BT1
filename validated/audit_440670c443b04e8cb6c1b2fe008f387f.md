### Title
Expired Governance Action Can Be Enacted Due to Missing Expiry Guard in `ratifyTransition` - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs`)

---

### Summary

In `ratifyTransition`, the expiry check `gasExpiresAfter < reCurrentEpoch` is placed exclusively in the `else` branch — the path taken when a governance action is **not** ratified. When a governance action **is** accepted by everyone, it is enacted unconditionally, with no check that its `gasExpiresAfter` epoch has not already passed. This allows a governance action to be enacted one epoch after its intended deadline, directly analogous to the missing deadline guard in the GluexRouter `swap()` function.

---

### Finding Description

`ratifyTransition` processes each `GovActionState` in the `RatifySignal` sequence and decides whether to enact or expire it:

```haskell
case rsig of
  gas@GovActionState {gasId, gasExpiresAfter} SSeq.:<| sigs -> do
    let govAction = gasAction gas
    if prevActionAsExpected gas ensPrevGovActionIds
      && validCommitteeTerm govAction ensCurPParams reCurrentEpoch
      && not rsDelayed
      && withdrawalCanWithdraw govAction ensTreasury
      && acceptedByEveryone env st gas
    then do
      -- *** NO expiry check here ***
      newEnactState <- trans @(EraRule "ENACT" era) $ ...
      ...
    else do
      st' <- trans @(RATIFY era) $ TRC (env, st, RatifySignal sigs)
      -- Expiry check is ONLY here, in the non-ratification path
      if gasExpiresAfter < reCurrentEpoch
        then pure $ st' & rsExpiredL %~ Set.insert gasId
        else pure st'
``` [1](#0-0) 

The expiry predicate `gasExpiresAfter < reCurrentEpoch` is evaluated **only** when the action fails ratification. If `acceptedByEveryone` returns `True`, the `then` branch is taken and the action is enacted via `ENACT` with no expiry guard whatsoever.

`gasExpiresAfter` is set at proposal submission time as `addEpochInterval curEpoch expiryInterval`: [2](#0-1) 

The GOV rule does prevent votes from being cast on already-expired actions (`curEpoch <= gasExpiresAfter`): [3](#0-2) 

However, this guard only prevents new votes from being recorded **after** expiry. It does not prevent the RATIFY rule from enacting a proposal whose `gasExpiresAfter` epoch has already passed, using votes that were legitimately cast during the final valid epoch.

The `RatifyState` comment confirms the `rsDelayed` flag is reset to `False` at each epoch boundary before RATIFY runs: [4](#0-3) 

So `rsDelayed = False` is the default state for the first action processed, meaning the `then` branch is reachable for any action that satisfies `acceptedByEveryone`.

---

### Impact Explanation

Any governance action type — `TreasuryWithdrawals`, `HardForkInitiation`, `ParameterChange`, `UpdateCommittee`, `NewConstitution`, `NoConfidence` — can be enacted one epoch past its `gasExpiresAfter` deadline. This constitutes an **unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action being enacted**, which is a **Critical** impact under the allowed scope.

Concretely:
- A `TreasuryWithdrawals` action enacted after expiry causes direct, irreversible loss of ADA from the treasury.
- A `HardForkInitiation` enacted after expiry triggers an unauthorized protocol version change.
- A `ParameterChange` enacted after expiry modifies protocol parameters outside the governance window the community voted within.

The `proposalsApplyEnactment` function in the EPOCH rule applies `rsEnacted` (enacted) and `rsExpired` (expired) sets to the live proposals without any secondary expiry re-check: [5](#0-4) 

Once an action lands in `rsEnacted`, it is applied unconditionally at the epoch boundary.

---

### Likelihood Explanation

The Proposals module documents that at epoch `n` boundary, the DRepPulser contains all proposals and votes from epoch `n-1`: [6](#0-5) 

This means votes cast during epoch `gasExpiresAfter` are included in the pulser evaluated at epoch `gasExpiresAfter + 1` boundary, where `reCurrentEpoch = gasExpiresAfter + 1`. At that boundary, `gasExpiresAfter < reCurrentEpoch` is `True` (expired), yet `acceptedByEveryone` can also be `True` if those last-epoch votes crossed the threshold.

No special privilege is required. Any DRep, SPO, or committee member who can cast the final vote(s) needed to reach the threshold during epoch `gasExpiresAfter` can trigger this. The scenario arises naturally whenever a proposal is near-threshold at its last valid epoch boundary and receives the remaining votes during that epoch. A motivated attacker can deliberately time their vote submission to the final epoch to force enactment of an otherwise-expiring proposal.

---

### Recommendation

Add an expiry guard **before** the ratification check, so that an expired action is always routed to `rsExpired` regardless of vote tallies:

```haskell
case rsig of
  gas@GovActionState {gasId, gasExpiresAfter} SSeq.:<| sigs -> do
    let govAction = gasAction gas
    if gasExpiresAfter < reCurrentEpoch
      then do
        st' <- trans @(RATIFY era) $ TRC (env, st, RatifySignal sigs)
        pure $ st' & rsExpiredL %~ Set.insert gasId
      else if prevActionAsExpected gas ensPrevGovActionIds
        && validCommitteeTerm govAction ensCurPParams reCurrentEpoch
        && not rsDelayed
        && withdrawalCanWithdraw govAction ensTreasury
        && acceptedByEveryone env st gas
      then do
        -- enact
        ...
      else do
        -- not ratified, not expired
        st' <- trans @(RATIFY era) $ TRC (env, st, RatifySignal sigs)
        pure st'
```

This mirrors the intent of `checkVotesAreNotForExpiredActions` in the GOV rule and ensures the expiry deadline is enforced on the enactment path as well.

### Proof of Concept

1. Set `ppGovActionLifetime = EpochInterval 1`.
2. In epoch `E`, submit a `TreasuryWithdrawals` governance action. `gasExpiresAfter = E + 1`.
3. During epoch `E`, cast votes from DReps/SPOs/committee that bring the tally to just below the required threshold.
4. At epoch `E + 1` boundary: `reCurrentEpoch = E + 1`. `gasExpiresAfter = E + 1`. `E + 1 < E + 1` → `False`. Action is **not** marked expired. It remains in proposals.
5. During epoch `E + 1`, cast the final vote(s) that push the tally over the threshold. (This is valid: `checkVotesAreNotForExpiredActions` allows votes while `curEpoch <= gasExpiresAfter`, i.e., `E + 1 <= E + 1`.)
6. At epoch `E + 2` boundary: `reCurrentEpoch = E + 2`. The pulser (containing epoch `E + 1` votes) runs `ratifyTransition`. `acceptedByEveryone` → `True`. The `then` branch is taken. `gasExpiresAfter = E + 1 < E + 2 = reCurrentEpoch` — the action has expired — but the expiry check is never evaluated. The treasury withdrawal is enacted and ADA is transferred out of the treasury. [1](#0-0)
