Audit Report

## Title
LP Funds Permanently Locked When Owner Is USDC/USDT Blacklisted in `removeLiquidity` — (File: metric-core/contracts/libraries/LiquidityLib.sol)

## Summary
`LiquidityLib.removeLiquidity` unconditionally transfers withdrawn tokens to the `owner` address with no alternative `recipient` path. `MetricOmmPool.removeLiquidity` enforces `msg.sender == owner`, so the position owner must be the caller and tokens always go to that same address. If `owner` is USDC/USDT blacklisted after depositing, every future `removeLiquidity` call reverts and rolls back all state changes, permanently locking the LP's principal with no recovery mechanism.

## Finding Description
In `LiquidityLib.removeLiquidity`, all internal state mutations (bin balance decrements, share burns, `binTotals` updates) occur inside the loop before the transfers: [1](#0-0) 

After all state is updated, the function performs direct push transfers to `owner`: [2](#0-1) 

`MetricOmmPool.removeLiquidity` enforces `msg.sender == owner`, so the position owner must be the caller and there is no way to redirect tokens to a different address: [3](#0-2) 

The function signature has no `recipient` parameter: [4](#0-3) 

Positions are keyed by `keccak256(abi.encode(owner, salt, bin))` and are not transferable, so there is no escape hatch: [5](#0-4) 

The extension hooks (`beforeRemoveLiquidity` / `afterRemoveLiquidity`) do not include a `recipient` parameter and cannot redirect the token transfer, which is hardcoded inside `LiquidityLib`: [6](#0-5) 

Because `safeTransfer` to a USDC/USDT-blacklisted address reverts, the entire transaction reverts, rolling back all state changes. The LP's position remains in storage but every subsequent `removeLiquidity` call produces the same revert — funds are permanently locked.

## Impact Explanation
The LP's entire deposited principal is permanently locked inside the pool. The `removeLiquidity` flow becomes completely unusable for the affected address. This satisfies two allowed impact-gate criteria simultaneously: direct loss of user principal and unusable withdraw/liquidity flow. Severity is High.

## Likelihood Explanation
USDC and USDT both implement on-chain address blacklists enforced at the token level. An LP can be blacklisted after depositing (e.g., regulatory action, sanctions compliance, or erroneous listing). USDC/USDT non-standard behavior is explicitly in scope per the contest rules. The consequence per affected LP is irreversible and total.

## Recommendation
Add an optional `recipient` parameter to `removeLiquidity` in both `MetricOmmPool` and `LiquidityLib` so the position owner can direct withdrawn tokens to an address they control that is not blacklisted:

```solidity
function removeLiquidity(
    address owner,
    address recipient,   // <-- new
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata extensionData
) external ...
```

Transfer to `recipient` instead of `owner` inside `LiquidityLib.removeLiquidity`. Alternatively, implement a pull pattern: store owed amounts in an internal mapping keyed by `owner` and expose a separate `claim(address recipient)` function.

## Proof of Concept
1. Pool is deployed with USDC as `token0`.
2. Alice (`owner = alice`) calls `addLiquidity`; her shares are recorded under `keccak256(abi.encode(alice, salt, bin))`.
3. USDC blacklists Alice's address.
4. Alice calls `removeLiquidity(alice, salt, deltas, "")`.
5. `MetricOmmPool` passes `owner = alice` into `LiquidityLib.removeLiquidity`.
6. The library computes `amount0Removed > 0` and executes `IERC20(USDC).safeTransfer(alice, amount0Removed)`.
7. USDC reverts because `alice` is blacklisted.
8. The entire transaction reverts; all state changes are rolled back.
9. Alice's position remains in storage but every future `removeLiquidity` call produces the same revert — her funds are permanently locked.
