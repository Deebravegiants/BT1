### Title
Pool Wallet Singleton State Corruption via Unvalidated Peer-Supplied CoinSpend Solution — (File: chia/wallet/wallet_state_manager.py)

---

### Summary

In `WalletStateManager._add_coin_states`, when processing a spent `POOLING_WALLET` singleton, the wallet fetches the `CoinSpend` from a peer and applies it to local state **before** any solution-level validation. A malicious peer can supply a puzzle-hash-valid but solution-crafted `CoinSpend`, causing the pool wallet's singleton lineage, pool state, and on-disk pool config to be corrupted — redirecting pool communications and permanently breaking the wallet's ability to generate valid state-transition transactions.

---

### Finding Description

Inside `_add_coin_states`, the entire processing of a coin state update is wrapped in a single DB writer transaction:

```python
async with self.db_wrapper.writer():   # line 1792
```

When the coin belongs to a `POOLING_WALLET` and has been spent, the following sequence executes **inside that transaction**: [1](#0-0) 

```
while curr_coin_state.spent_height is not None:
    cs = await fetch_coin_spend_for_coin_state(curr_coin_state, peer)   # ← external call
    success = await singleton_wallet.apply_state_transition(cs, ...)    # ← state write
    ...
    await self.coin_store.set_spent(...)                                 # ← state write
    new_coin_state = await self.wallet_node.get_coin_state(             # ← external call
        [new_singleton_name], peer=peer, ...)
    curr_coin_state = new_coin_state[0]
```

`fetch_coin_spend_for_coin_state` calls `fetch_coin_spend`, which validates **only** that the returned puzzle's tree-hash matches the coin's on-chain `puzzle_hash` and that the `coin_name` field is correct: [2](#0-1) 

The **solution is never validated against the blockchain**. A peer can return any solution bytes it chooses.

`apply_state_transition` then accepts the spend if `new_state.coin.name() == tip_coin.name()` (the coin ID is correct — it is the real on-chain singleton tip, which is public), and immediately writes it to `pool_store`: [3](#0-2) 

After writing, `update_pool_config` is called, which overwrites the on-disk pool config (pool URL, target puzzle hash, owner pubkey) with values parsed from the attacker-controlled solution via `solution_to_pool_state`. [4](#0-3) 

The second external call, `wallet_node.get_coin_state` at line 2119, drives the loop's next iteration. A malicious peer can return a `CoinState` with `spent_height is not None` to force additional iterations, each applying another fake state transition. [5](#0-4) 

---

### Impact Explanation

**High — Corruption of pool membership state with direct security impact.**

1. **Pool config corruption**: `update_pool_config` writes the attacker-chosen `pool_url` and `target_puzzle_hash` to the farmer's config file. The farmer then submits shares to the attacker-controlled pool URL, losing all pool rewards.
2. **Singleton lineage corruption**: `pool_store.add_spend` records the fake spend. All subsequent calls to `get_current_state`, `get_tip`, and `generate_travel_transactions` operate on the corrupted lineage, producing transactions that are invalid on-chain and will never confirm.
3. **`target_state` reset**: If the crafted solution parses to a `PoolState` equal to `self.target_state`, the wallet stops attempting the legitimate state transition (e.g., leaving a pool), permanently locking the farmer in the wrong pool state.

---

### Likelihood Explanation

- Chia wallets connect to full nodes via peer discovery; an attacker can operate a public full node and attract connections.
- The attack triggers automatically during normal pool operations (any singleton spend: joining, leaving, absorbing rewards).
- No privileged access is required; the attacker only needs to be the peer that responds to `request_puzzle_solution`.
- For untrusted peers, coin *states* are validated via Merkle proof, but the *solution* returned by `request_puzzle_solution` is never cross-checked against the chain.

---

### Recommendation

1. **Validate the solution before writing state**: After fetching the `CoinSpend`, run the puzzle against the solution locally (or verify the solution hash against a Merkle proof from the block) before calling `apply_state_transition`.
2. **Follow Checks-Effects-Interactions**: Move all external peer calls (`fetch_coin_spend_for_coin_state`, `get_coin_state`) **before** the `db_wrapper.writer()` block. Collect all required data, validate it, then perform all DB writes atomically.
3. **Validate `get_coin_state` results**: The `assert len(new_coin_state) == 1` at line 2122 is the only guard on the loop-driving external call; replace it with a proper validation that the returned coin state is consistent with the expected singleton lineage.

---

### Proof of Concept

1. Attacker operates a Chia full node and attracts a victim farmer's wallet connection via peer discovery.
2. Victim has a `POOLING_WALLET` singleton currently in `FARMING_TO_POOL` state with a known on-chain coin ID (public blockchain data).
3. Victim's singleton is spent (e.g., absorbing a pool reward — a routine operation).
4. Wallet calls `fetch_coin_spend_for_coin_state` for the spent singleton coin.
5. Attacker returns `RespondPuzzleSolution` with:
   - `puzzle`: the real singleton full puzzle (puzzle hash matches — passes validation)
   - `solution`: crafted bytes that `solution_to_pool_state` parses as `SELF_POOLING` with `pool_url = ""` and `target_puzzle_hash = attacker_address`
6. `apply_state_transition` accepts the spend (coin ID matches tip), writes it to `pool_store`.
7. `update_pool_config` overwrites the pool config with the attacker-chosen state.
8. Farmer's node now believes it is self-pooling; all future pool reward coins go unprocessed; the farmer loses ongoing pool rewards.
9. Attacker returns `spent_height is not None` from `get_coin_state` to drive additional loop iterations, applying further fake transitions before the outer DB transaction commits.

async def fetch_coin_spend(height: uint32, coin: Coin, peer: WSChiaConnection) -> CoinSpend:
    solution_response = await peer.call_api(
        FullNodeAPI.request_puzzle_solution, RequestPuzzleSolution(coin.name(), height)
    )
    if solution_response is None or not isinstance(solution_response, RespondPuzzleSolution):
        raise PeerRequestException(f"Was not able to obtain solution {solution_response}")
    coin_id = coin.name()
    if solution_response.response.puzzle.get_tree_hash() != coin.puzzle_hash:
        raise PeerRequestException(f"Peer returned wrong puzzle hash for coin {coin_id}")
    if solution_response.response.coin_name != coin_id:
        raise PeerRequestException(f"Peer returned wrong coin name in puzzle solution for coin {coin_id}")

    return make_spend(
        coin,
        solution_response.response.puzzle,
        solution_response.response.solution,
    )

    async def apply_state_transition(
        self, new_state: CoinSpend, block_height: uint32, action_scope: WalletActionScope
    ) -> bool:
        """
        Updates the Pool state (including DB) with new singleton spends.
        The DB must be committed after calling this method. All validation should be done here. Returns True iff
        the spend is a valid transition spend for the singleton, False otherwise.
        """
        tip: tuple[uint32, CoinSpend] = await self.get_tip()
        tip_spend = tip[1]

        tip_coin: Coin | None = get_most_recent_singleton_coin_from_coin_spend(tip_spend)
        assert tip_coin is not None
        spent_coin_name: bytes32 = tip_coin.name()

        if spent_coin_name != new_state.coin.name():
            history: list[tuple[uint32, CoinSpend]] = await self.get_spend_history()
            if new_state.coin.name() in [sp.coin.name() for _, sp in history]:
                self.log.info(f"Already have state transition: {new_state.coin.name().hex()}")
            else:
                self.log.warning(
                    f"Failed to apply state transition. tip: {tip_coin} new_state: {new_state} height {block_height}"
                )
            return False

        await self.wallet_state_manager.pool_store.add_spend(self.wallet_id, new_state, block_height)
        tip_spend = (await self.get_tip())[1]
        self.log.info(f"New PoolWallet singleton tip_coin: {tip_spend} farmed at height {block_height}")

        # If we have reached the target state, resets it to None. Loops back to get current state
        for _, added_spend in reversed(
            await self.wallet_state_manager.pool_store.get_spends_for_wallet(self.wallet_id)
        ):
            latest_state: PoolState | None = solution_to_pool_state(added_spend)
            if latest_state is not None:
                if self.target_state == latest_state:
                    self.target_state = None
                    self.next_transaction_fee = uint64(0)
                    self.next_tx_config = DEFAULT_TX_CONFIG
                break

        await self.update_pool_config(action_scope)
        return True
