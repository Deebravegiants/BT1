Audit Report

## Title
Unpermissioned `updateRSETHPrice()` Allows Any Caller to Trigger Protocol-Wide Pause, Temporarily Freezing All User Deposits and Withdrawals - (File: `contracts/LRTOracle.sol`)

## Summary

`LRTOracle.updateRSETHPrice()` is declared `public` with only a `whenNotPaused` guard, allowing any address to invoke it. Inside `_updateRsETHPrice()`, when the computed rsETH price falls more than `pricePercentageLimit` below `highestRsethPrice`, the function unconditionally pauses `LRTDepositPool`, `LRTWithdrawalManager`, and the oracle itself — with no role check on the caller. This is an explicit asymmetry: the upside-price path reverts for non-managers, but the downside-pause path executes for any caller, enabling an unprivileged attacker to freeze all protocol deposits and withdrawals whenever the price-drop condition is naturally met.

## Finding Description

`updateRSETHPrice()` carries no access control beyond `whenNotPaused`:

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

Inside `_updateRsETHPrice()`, the upside path at L260-265 explicitly checks `msg.sender` for the `MANAGER` role before allowing the price update to proceed:

```solidity
if (!IAccessControl(address(lrtConfig)).hasRole(LRTConstants.MANAGER, msg.sender)) {
    revert PriceAboveDailyThreshold();
}
```

The downside path at L270-282 has no equivalent check:

```solidity
if (isPriceDecreaseOffLimit) {
    if (!lrtDepositPool.paused()) lrtDepositPool.pause();
    if (!withdrawalManager.paused()) withdrawalManager.pause();
    _pause();
    return;
}
```

For the oracle to call `lrtDepositPool.pause()` and `withdrawalManager.pause()` successfully, `LRTOracle` must hold `PAUSER_ROLE` on those contracts — which is a required deployment precondition for the downside-protection mechanism to function at all (the same mechanism used by `LRTConfig.pauseAll()`). Given that precondition, any unprivileged EOA can call `updateRSETHPrice()` when the price-drop condition is met and atomically pause all three contracts.

`LRTDepositPool.depositETH()` and `depositAsset()` are gated by `whenNotPaused`. `LRTWithdrawalManager.initiateWithdrawal()`, `completeWithdrawal()`, and `instantWithdrawal()` are all gated by `whenNotPaused`. Unpausing each contract requires `onlyLRTAdmin` — a multisig operation requiring coordination across three separate `unpause()` calls.

## Impact Explanation

An unprivileged attacker who calls `updateRSETHPrice()` when `isPriceDecreaseOffLimit == true` causes an immediate, protocol-wide freeze: no deposits can enter and no withdrawals (standard, instant, or operator-assisted) can exit until an admin multisig executes three separate unpause transactions. Users with matured withdrawal requests cannot complete them during the freeze. This concretely matches the allowed impact: **Medium — Temporary freezing of funds**.

## Likelihood Explanation

The triggering condition — a price drop exceeding `pricePercentageLimit` relative to `highestRsethPrice` — is a real-world event arising from EigenLayer slashing, LST de-pegs, or transient oracle dips. The attacker does not manufacture the condition; they only need to observe on-chain state (public variables `rsETHPrice`, `highestRsethPrice`, `pricePercentageLimit`) and submit a cheap, zero-value call to `updateRSETHPrice()`. No special role, capital, or victim cooperation is required. A monitoring bot can reliably detect and exploit the condition in the same block it arises. **Likelihood: Medium.**

## Recommendation

Apply the same role gate to the downside-pause path that already protects the upside path. The minimal fix is to restrict `updateRSETHPrice()` to authorized callers:

```solidity
// After (recommended)
function updateRSETHPrice() external whenNotPaused onlyLRTManager {
    _updateRsETHPrice();
}
```

Alternatively, separate the pause-triggering logic from the price-update logic: keep price reads public but require a privileged role to execute the pause branch, or emit an event and let an off-chain keeper with the `PAUSER_ROLE` respond.

## Proof of Concept

1. Deploy with `pricePercentageLimit = 1e16` (1%).
2. A slashing event causes `newRsETHPrice` to be 1.5% below `highestRsethPrice`, making `isPriceDecreaseOffLimit == true`.
3. Attacker (any EOA) calls `LRTOracle.updateRSETHPrice()`.
4. `_updateRsETHPrice()` executes L278-280:
   - `lrtDepositPool.pause()` — all deposits blocked.
   - `withdrawalManager.pause()` — all withdrawals blocked.
   - `_pause()` — oracle paused, blocking further public price updates.
5. Alice, whose withdrawal delay has elapsed, calls `completeWithdrawal()` — reverts with `Pausable: paused`.
6. Protocol team must coordinate multisig to call `LRTDepositPool.unpause()`, `LRTWithdrawalManager.unpause()`, and `LRTOracle.unpause()` before any user interaction resumes.

**Foundry test sketch:**
```solidity
function test_anyCallerCanPauseViaUpdateRSETHPrice() public {
    // Setup: price drop condition met
    vm.mockCall(...); // mock oracle to return price 1.5% below highestRsethPrice
    
    address attacker = makeAddr("attacker");
    vm.prank(attacker);
    lrtOracle.updateRSETHPrice(); // no role required
    
    assertTrue(lrtDepositPool.paused());
    assertTrue(lrtWithdrawalManager.paused());
    assertTrue(lrtOracle.paused());

    lrtWithdrawalManager.completeWithdrawal(asset, "");
}
```
