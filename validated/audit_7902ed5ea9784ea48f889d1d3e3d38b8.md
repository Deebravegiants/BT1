Looking at the actual code carefully to validate each step of the claimed attack.

Audit Report

## Title
Depositors Can Sandwich `updateRSETHPrice()` to Steal Yield from Existing rsETH Holders — (File: `contracts/LRTOracle.sol`, `contracts/LRTDepositPool.sol`, `contracts/LRTWithdrawalManager.sol`)

## Summary

`LRTOracle.rsETHPrice` is a cached state variable updated only when `updateRSETHPrice()` is explicitly called. Because this function is public and permissionless, an attacker can deposit at the stale lower price to receive excess rsETH, trigger the price update to reflect accrued yield, and immediately withdraw at the elevated price via `instantWithdrawal`. This transfers yield belonging to existing rsETH holders to the attacker. Without instant withdrawal, the same dilution of existing holders' yield occurs, though the attacker bears 8-day price risk.

## Finding Description

**Root cause:** `rsETHPrice` is a cached value at `LRTOracle.sol:28` that only updates on explicit calls to `updateRSETHPrice()` (`LRTOracle.sol:87-89`). The function carries no access control — any caller can invoke it while the contract is unpaused.

**Deposit at stale price:** `LRTDepositPool.getRsETHAmountToMint` (`LRTDepositPool.sol:520`) computes `rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()`. When `rsETHPrice` is stale-low (yield has accrued but not yet been reflected), the denominator is smaller than fair value, so the attacker receives more rsETH than warranted.

**Price update dilutes existing holders:** Inside `_updateRsETHPrice()` (`LRTOracle.sol:234`), `previousTVL = rsethSupply.mulWad(rsETHPrice)` is computed using the now-inflated supply (which includes the attacker's freshly minted tokens) multiplied by the stale price. The `rewardAmount = totalETHInProtocol - previousTVL` therefore appears to be the full accrued yield, but it is now spread across the attacker's rsETH as well. The new price (`LRTOracle.sol:250`) becomes `(totalETHInProtocol - fee) / rsethSupply`, which is lower than it would have been without the attacker's deposit, directly reducing the per-token yield for original holders.

**Instant withdrawal path (atomic, risk-free):** `LRTWithdrawalManager.instantWithdrawal` (`LRTWithdrawalManager.sol:228`) calls `getExpectedAssetAmount` which reads the freshly updated `rsETHPrice` (`LRTWithdrawalManager.sol:593`). The attacker burns their rsETH and receives the elevated asset amount in the same transaction, with no delay. The only friction is `instantWithdrawalFee` (`LRTWithdrawalManager.sol:237`), which is configurable up to 10% but may be set to 0 or below the yield-capture threshold.

**Standard withdrawal path (delayed, partial risk):** `initiateWithdrawal` stores `expectedAssetAmount` at the post-update price (`LRTWithdrawalManager.sol:168`). At unlock time, `_calculatePayoutAmount` (`LRTWithdrawalManager.sol:824-835`) applies `min(expectedAtInitiation, currentReturn)`, so the attacker's profit is contingent on the price not falling below the initiation price over the 8-day delay. Regardless of the attacker's outcome, the yield dilution to existing holders is permanent and irreversible.

**`pricePercentageLimit` is insufficient:** The guard at `LRTOracle.sol:252-266` reverts non-manager callers only when the price increase exceeds `pricePercentageLimit`. This check is bypassed entirely when `pricePercentageLimit == 0` (the condition is `pricePercentageLimit > 0 && ...`). Even when set, an attacker can time deposits to windows where accrued yield falls within the limit, or execute repeated smaller attacks across multiple update cycles.

## Impact Explanation

**High — Theft of unclaimed yield.**

Concrete example (no fee, no `pricePercentageLimit`):
- Pre-attack: TVL = 1,010 ETH (10 ETH yield accrued), rsETH supply = 1,000, stale `rsETHPrice` = 1.0.
- Attacker deposits 1,000 ETH → minted 1,000 rsETH (fair value ≈ 990 rsETH).
- `updateRSETHPrice()`: new TVL = 2,010, supply = 2,000 → `rsETHPrice` = 1.005.
- `instantWithdrawal` of 1,000 rsETH → attacker receives 1,005 ETH. Net profit: **5 ETH**.
- Original holders' 1,000 rsETH is now worth 1,005 ETH instead of 1,010 ETH — **5 ETH of yield stolen**.

The attack scales linearly with deposit size and the length of the stale window. It requires no privileged access, no oracle manipulation, and no external protocol compromise.

## Likelihood Explanation

**Likelihood: Low for standard withdrawal path; Medium-High when instant withdrawal is enabled.**

Without instant withdrawal, the attacker must lock capital for `withdrawalDelayBlocks` (~57,600 blocks / 8 days, `LRTWithdrawalManager.sol:94`) and bears price risk over that period. The `pricePercentageLimit` caps per-cycle profit when non-zero. These factors reduce practical frequency.

When `isInstantWithdrawalEnabled[asset] == true` and `instantWithdrawalFee` is below the yield-capture threshold, the entire attack collapses into a single flash-loan transaction with no capital lockup and no price risk. Any MEV bot monitoring the mempool for stale price windows can execute this atomically, raising likelihood significantly.

## Recommendation

1. **Update `rsETHPrice` atomically inside `depositETH`/`depositAsset`** before computing `rsethAmountToMint`, so deposits always use a fresh price and cannot exploit stale windows.
2. **Restrict `updateRSETHPrice()` to authorized callers** (e.g., `MANAGER` role or a keeper with a minimum call interval), preventing an attacker from triggering the update at will.
3. **Enforce a minimum `instantWithdrawalFee`** that exceeds the maximum yield capturable in one update cycle, or require a time delay between deposit and instant withdrawal eligibility.
4. **Enforce a deposit fee or entry delay** proportional to the time elapsed since the last price update, so late depositors cannot capture accrued yield for free.

// - Protocol TVL: 1,000 ETH; rsETH supply: 1,000; stale rsETHPrice = 1.0 ETH

// - instantWithdrawal enabled for ETH; instantWithdrawalFee = 0

// Step 1 — Deposit at stale price
LRTDepositPool.depositETH{value: 1000 ether}(0, "")
// getRsETHAmountToMint: 1000e18 * 1e18 / 1.0e18 = 1000e18 rsETH minted
// Attacker holds 1000 rsETH; fair value ≈ 990 rsETH

// Step 2 — Trigger price update
LRTOracle.updateRSETHPrice()
// rsethSupply = 2000, totalETHInProtocol = 2010
// previousTVL = 2000 * 1.0 = 2000; rewardAmount = 10 ETH (diluted across 2000 rsETH)
// newRsETHPrice = 2010 / 2000 = 1.005 ETH

// Step 3 — Instant withdrawal at elevated price
LRTWithdrawalManager.instantWithdrawal(ETH_TOKEN, 1000e18, "")
// assetAmountUnlocked = 1000e18 * 1.005e18 / 1e18 = 1005 ETH
// Attacker receives 1005 ETH; net profit = 5 ETH

// Result:
// Original holders' 1000 rsETH = 1005 ETH (should be 1010 ETH)
