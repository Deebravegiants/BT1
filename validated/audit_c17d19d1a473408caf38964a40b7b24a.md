### Title
`accept_requests` Guard in `respond()` Permanently Blocks In-Flight Signature Delivery After TEE Attestation Expiry - (File: `crates/contract/src/lib.rs`)

### Summary

The `respond()` function in the MPC smart contract applies an `accept_requests` guard that was designed to block **new** signing requests when TEE validation fails. However, this same guard also blocks MPC nodes from delivering signatures for requests that were **already queued** before the TEE state degraded. This is a direct structural analog to the LinearVesting `expiry` check that blocked `mint()` for already-queued payouts: a post-submission state transition permanently prevents completion of a legitimately initiated operation.

### Finding Description

When a user calls `sign()`, the contract enqueues a yield-resume promise and stores the pending request. The MPC nodes then compute the signature off-chain and call `respond()` to deliver it on-chain.

The `respond()` function contains the following guard:

```rust
// crates/contract/src/lib.rs ~line 579
if !self.accept_requests {
    return Err(TeeError::TeeValidationFailed.into());
}
``` [1](#0-0) 

The `accept_requests` flag is set to `false` by `verify_tee()` when a TEE attestation expires and the surviving participant count would violate the threshold relation:

```rust
// crates/contract/src/lib.rs ~line 1737
self.accept_requests = false;
return Ok(false);
``` [2](#0-1) 

The `verify_tee()` function is called routinely by nodes every 7 days (per the design doc) and by any participant at any time. When it fires and sets `accept_requests = false`, every pending yield that was queued before this state change is now unresolvable: `respond()` returns `TeeError::TeeValidationFailed` for every delivery attempt. The NEAR runtime's yield-timeout then fires `fail_on_timeout()`, which panics and fails the original caller's transaction:

```rust
// crates/contract/src/lib.rs ~line 2341
pub fn fail_on_timeout() {
    env::panic_str(&RequestError::Timeout.to_string());
}
``` [3](#0-2) 

The deposit attached to the sign request (minimum 1 yoctoNEAR, but callers may attach more) is not refunded on timeout — `fail_on_timeout()` only panics; there is no transfer back to the caller. [4](#0-3) 

The `check_request_preconditions()` function (called by `sign()`) also checks `accept_requests`, but that only prevents new requests from being queued. The critical gap is that `respond()` applies the same check to **already-queued** requests, for which the user had no opportunity to observe the degraded state before submitting. [5](#0-4) 

### Impact Explanation

Any signing request that is in-flight at the moment `verify_tee()` sets `accept_requests = false` will time out and fail. The user:

1. Cannot obtain the threshold signature they requested.
2. Loses the attached deposit (not refunded on timeout path).
3. Cannot retry until `accept_requests` is restored to `true` (which requires TEE re-attestation and another `verify_tee()` call succeeding).

This breaks the request-lifecycle invariant: a request that passed all preconditions at submission time is permanently blocked from completion by a post-submission state change. This matches the **Medium** allowed impact: *"request-lifecycle manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."*

### Likelihood Explanation

TEE attestations have a finite validity period (certificates expire; the design doc specifies nodes re-attest every 7 days). A single node failing to renew its attestation before expiry — due to a crash, network partition, or deliberate delay by a Byzantine participant below the signing threshold — is sufficient to trigger `TeeValidationResult::Partial`. If the surviving set is too small to satisfy the threshold relation, `accept_requests` is set to `false` and all in-flight requests are frozen. This is a realistic operational scenario, not a theoretical one. [6](#0-5) 

### Recommendation

The `accept_requests` guard in `respond()` should be removed or scoped only to **new** request intake. Delivering a signature for a request that was already accepted and queued does not introduce new trust assumptions — the signing computation was already authorized. The guard in `sign()` (and `request_app_private_key()`, `verify_foreign_transaction()`) is sufficient to stop new work from entering the queue during a degraded TEE state. Alternatively, implement a deposit-refund path in `return_signature_and_clean_state_on_success` when the timeout fires due to a TEE-blocked `respond()`, so users are not penalized for a state transition they could not anticipate.

### Proof of Concept

1. Contract is `Running`, `accept_requests = true`. Alice calls `sign()` with 1 yoctoNEAR deposit. The yield is queued in `pending_signature_requests`.
2. Node B's TEE certificate expires before it renews. Node A calls `verify_tee()`. `reverify_and_cleanup_participants()` returns `TeeValidationResult::Partial` with only Node A valid. The surviving count is below the threshold relation, so `accept_requests` is set to `false`.
3. Node A computes the signature for Alice's request and calls `respond(request, signature)`. The contract executes:
   ```
   if !self.accept_requests {
       return Err(TeeError::TeeValidationFailed.into());
   }
   ```
   The call fails. Node A retries; same result.
4. The NEAR runtime's yield timeout fires. `return_signature_and_clean_state_on_success` receives `Err(PromiseError::Failed)`, pops the pending yield, and schedules `fail_on_timeout()`.
5. `fail_on_timeout()` panics. Alice's original `sign()` transaction fails with `RequestError::Timeout`. Her deposit is not returned. [7](#0-6) [8](#0-7)
