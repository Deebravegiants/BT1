Audit Report

## Title
`assetsCommitted` exceeding `getTotalAssetDeposits` after EigenLayer slashing permanently blocks new withdrawal requests — (`contracts/LRTWithdrawalManager.sol`)

## Summary

When EigenLayer slashing reduces `getTotalAssetDeposits` below the current `assetsCommitted[asset]` value, `getAvailableAssetAmount` returns 0 and every subsequent `initiateWithdrawal` call reverts with `ExceedAmountToWithdraw`. The freeze persists until an operator calls `unlockQueue`, which itself can revert if the unstaking vault holds no assets. This constitutes a temporary freeze of the withdrawal queue for all unprivileged users.

## Finding Description

`initiateWithdrawal` at L166–173 of `contracts/LRTWithdrawalManager.sol` executes:

```solidity
IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();
assetsCommitted[asset] += expectedAssetAmount;
```

`getAvailableAssetAmount` (L599–603) returns:

```solidity
availableAssetAmount = totalAssets > assetsCommitted[asset] ? totalAssets - assetsCommitted[asset] : 0;
```

`totalAssets` is sourced from `LRTDepositPool.getTotalAssetDeposits` (L385–397), which sums `assetStakedInEigenLayer + assetUnstakingFromEigenLayer` alongside other buckets. EigenLayer slashing directly reduces `assetStakedInEigenLayer` at any time, independently of `assetsCommitted`. Once `assetsCommitted[asset] >= totalAssets`, `getAvailableAssetAmount` returns 0, and since `expectedAssetAmount > 0` for any non-zero `rsETHUnstaked`, the strict `>` check at L170 causes every `initiateWithdrawal` call to revert.

The only recovery path is `unlockQueue` (L268–320), which is restricted to `onlyAssetTransferOrOperatorRole` and itself reverts at L297 with `AmountMustBeGreaterThanZero` when `params.totalAvailableAssets == 0` — i.e., when the unstaking vault holds no ETH/LST (assets are still in EigenLayer's delayed-withdrawal queue). This extends the freeze through the full EigenLayer withdrawal delay period.

The developer comment at L147–149 acknowledges: *"There is an edge case were the user withdraws last underlying asset and that asset gets slashed"*, confirming awareness of the slashing interaction but no mitigation is implemented.

## Impact Explanation

All unprivileged users are blocked from calling `initiateWithdrawal` for the affected asset. Their rsETH cannot be redeemed through the queued withdrawal path for the duration of the freeze. This matches **Medium — Temporary freezing of funds**.

## Likelihood Explanation

EigenLayer slashing is a designed protocol feature, not an external compromise. The protocol delegates to multiple node operators via `NodeDelegator`, and any slashing event that reduces `getTotalAssetDeposits` below the current `assetsCommitted` value triggers the freeze. Under normal high-demand conditions, `assetsCommitted` can approach `totalAssets`, meaning even a modest slashing event crosses the threshold. The freeze duration is bounded by the EigenLayer delayed-withdrawal period plus operator responsiveness.

## Recommendation

1. Remove the strict availability check from `initiateWithdrawal` and instead let `unlockQueue` settle payouts at the prevailing price (already capped by `_calculatePayoutAmount`), accepting that some requests may receive proportionally less than committed.
2. Alternatively, allow `initiateWithdrawal` to proceed when `assetsCommitted >= totalAssets` by committing the user to a proportionally reduced amount, so new requests can still be queued without worsening the shortfall.

## Proof of Concept

1. `getTotalAssetDeposits(ETH)` = 100 ETH; `assetsCommitted[ETH]` = 0.
2. Users call `initiateWithdrawal` until `assetsCommitted[ETH]` = 95 ETH; `getAvailableAssetAmount` = 5 ETH.
3. EigenLayer slashing reduces the NDC's strategy balance by 10 ETH; `getTotalAssetDeposits(ETH)` drops to 90 ETH.
4. `getAvailableAssetAmount` = `max(0, 90 − 95)` = **0**.
5. Any user calling `initiateWithdrawal` with any non-zero `rsETHUnstaked` receives `ExceedAmountToWithdraw` (L170).
6. Operator calls `unlockQueue`; if the unstaking vault is empty (assets still in EigenLayer's delayed-withdrawal queue), L297 reverts with `AmountMustBeGreaterThanZero`, extending the freeze until the EigenLayer withdrawal delay completes.

Foundry fork test plan: fork mainnet, deploy with realistic `assetsCommitted` near `totalAssets`, mock EigenLayer strategy to return a reduced share value, call `initiateWithdrawal` and assert revert with `ExceedAmountToWithdraw`, then call `unlockQueue` with empty vault and assert revert with `AmountMustBeGreaterThanZero`.
