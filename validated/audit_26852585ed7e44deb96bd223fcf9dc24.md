### Title
Hardcoded `only_query=false` in `TryFrom<BroadcastedTransaction>` Causes Wrong Transaction Hash in Fee Estimation and Simulation — (`crates/apollo_rpc/src/v0_8/api/mod.rs`)

---

### Summary

The `TryFrom<BroadcastedTransaction> for ExecutableTransactionInput` conversion unconditionally passes `only_query=false` for Invoke and DeployAccount transactions, regardless of whether the client submitted a query-version transaction (version with the high bit set, e.g. `0x100000000000000000000000000000001`). This causes `calc_tx_hash` to compute a hash that does not match what the account contract's `__validate__` entry point expects, making `starknet_estimateFee` and `starknet_simulateTransactions` return wrong results for any query-version transaction.

---

### Finding Description

In `crates/apollo_rpc/src/v0_8/api/mod.rs`, the conversion from `BroadcastedTransaction` to `ExecutableTransactionInput` hardcodes `only_query=false`:

```rust
impl TryFrom<BroadcastedTransaction> for ExecutableTransactionInput {
    type Error = ErrorObjectOwned;
    fn try_from(value: BroadcastedTransaction) -> Result<Self, Self::Error> {
        // TODO(yair): pass the right value for only_query field.
        match value {
            BroadcastedTransaction::Declare(tx) => Ok(tx.try_into()?),
            BroadcastedTransaction::DeployAccount(tx) => Ok(Self::DeployAccount(tx.into(), false)),
            BroadcastedTransaction::Invoke(tx) => Ok(Self::Invoke(tx.into(), false)),
        }
    }
}
``` [1](#0-0) 

The `only_query` boolean is then consumed by `calc_tx_hash`:

```rust
fn calc_tx_hash(self, chain_id: &ChainId) -> ExecutionResult<(Self, TransactionHash)> {
    match self.apply_on_transaction(|tx, only_query| {
        get_transaction_hash(tx, chain_id, &TransactionOptions { only_query })
    }) { ... }
}
``` [2](#0-1) 

`get_transaction_hash` calls `signed_tx_version_from_tx(transaction, transaction_options)` to derive the version that is hashed. When `only_query=true`, this function sets the query bit in the version (producing e.g. `0x100000000000000000000000000000001`); when `only_query=false`, it does not. The version is a direct input to the Pedersen/Poseidon hash chain for every transaction type. [3](#0-2) 

For an InvokeV1 hash, the version is the second element chained: [4](#0-3) 

Because `only_query` is always `false`, the hash computed by the node for a query-version transaction will always be the non-query hash. The account contract's `__validate__` entry point, however, reads the transaction hash via `get_execution_info` syscall, which reflects the hash as computed — i.e., the non-query hash. If the account contract signed and expects the query-version hash (as the Starknet spec requires for `starknet_estimateFee`/`starknet_simulateTransactions`), signature verification inside `__validate__` will fail, producing a wrong simulation trace or a wrong fee estimate.

The TODO comment at line 332 explicitly acknowledges this is unimplemented:
```
// TODO(yair): pass the right value for only_query field.
``` [5](#0-4) 

---

### Impact Explanation

Any unprivileged RPC client calling `starknet_estimateFee` or `starknet_simulateTransactions` with a properly-formed query-version transaction (as the Starknet spec requires) will receive a simulation result where `__validate__` sees the wrong transaction hash. For account contracts that verify the transaction hash as part of signature validation (the standard pattern), this causes validation to fail in simulation even though the real transaction would succeed, or vice versa. The returned fee estimate or simulation trace is therefore wrong for the same accepted input.

This falls under: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

---

### Likelihood Explanation

The Starknet specification mandates that clients send query-version transactions to `starknet_estimateFee` and `starknet_simulateTransactions`. Any compliant client or wallet doing fee estimation will trigger this path. The bug is unconditional — there is no code path that correctly sets `only_query=true` for these RPC endpoints.

---

### Recommendation

Derive `only_query` from the transaction version field before constructing `ExecutableTransactionInput`. Specifically, inspect whether the submitted transaction's version has the query bit set (i.e., `version >= 2^128`) and pass `only_query=true` accordingly. The `signed_tx_version_from_tx` function already knows how to handle this; the fix is to extract the bit from the `BroadcastedTransaction` version before the conversion discards it.

---

### Proof of Concept

1. Construct a `BroadcastedInvokeTransactionV1` with `version = 0x100000000000000000000000000000001` (query bit set).
2. Sign it using the query-version hash (as a compliant client would).
3. Submit it to `starknet_simulateTransactions` without `SKIP_VALIDATE`.
4. Observe that the simulation trace shows `__validate__` failing with a signature error, even though the same transaction submitted with `version = 0x1` and the non-query hash would pass.
5. Confirm by comparing the `transaction_hash` field in the returned trace against `get_transaction_hash(tx, chain_id, &TransactionOptions { only_query: true })` — they will differ.
