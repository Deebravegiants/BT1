### Title
`execute_foreign_chain_request` Unconditionally Bails for Ethereum and Solana Requests Despite Contract Accepting Them - (File: `crates/node/src/providers/verify_foreign_tx/sign.rs`)

---

### Summary

The node-side `execute_foreign_chain_request` function immediately bails for `ForeignChainRpcRequest::Ethereum` and `ForeignChainRpcRequest::Solana` variants, while the on-chain `verify_foreign_transaction` contract method accepts requests for these chain types whenever they appear in the supported-chains set. This creates a permanent request-lifecycle failure: the deposit is consumed, the yield request is enqueued, but no node will ever produce a signature, causing the request to time out and the deposit to be permanently lost.

---

### Finding Description

`ForeignChainRpcRequest` is a `#[non_exhaustive]` enum with twelve variants including `Ethereum` and `Solana`: [1](#0-0) 

The on-chain `verify_foreign_transaction` method gates on `get_supported_foreign_chains()`, which returns the intersection of all active participants' registered chains. If every active node registers `Ethereum` or `Solana` via `register_foreign_chain_config`, the contract will accept requests for those chains: [2](#0-1) 

However, in the node-side execution path, `execute_foreign_chain_request` explicitly bails for both variants: [3](#0-2) 

There is no `ethereum` or `solana` inspector in `ForeignChainInspectors` — the struct only contains `bitcoin`, `abstract_chain`, `bnb`, `starknet`, `base`, `arbitrum`, `hyper_evm`, `polygon`, and `aptos`: [4](#0-3) 

The contract has no mechanism to validate that a chain registered as "supported" is actually executable by the nodes. `register_foreign_chain_config` accepts any `ForeignChain` variant without checking whether the node binary implements it. Once a chain enters the supported set, the contract enqueues a yield request and consumes the deposit: [5](#0-4) 

The deposit is not returned on timeout. The sandbox test confirms that all chain types — including Ethereum and Solana — time out without a response when no node responds: [6](#0-5) 

This is structurally identical to the UniV2 analog: the contract-level interface accepts a request type (`Ethereum`/`Solana`) that the execution layer (`execute_foreign_chain_request`) cannot handle, causing all such requests to silently fail after consuming the deposit.

---

### Impact Explanation

Any user who submits a `verify_foreign_transaction` request for `Ethereum` or `Solana` — when those chains are in the supported set — will have their deposit permanently consumed with no signature ever produced. The request lifecycle invariant (deposit → signature or deposit refund) is broken. This matches the **Medium** allowed impact: *balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants*.

---

### Likelihood Explanation

The trigger requires that all active participants have registered `Ethereum` or `Solana` via `register_foreign_chain_config`. This is plausible during network expansion: operators may configure their `foreign_chains.yaml` to include Ethereum (the most prominent chain) expecting it to be handled, since the contract API and the `ForeignChainRpcRequest` enum both expose it as a first-class variant. There is no compile-time or runtime guard preventing this misconfiguration. Once the chain enters the supported set, any unprivileged user can trigger the deposit-loss condition.

---

### Recommendation

1. **Node-side guard**: Before a node registers a chain as supported via `register_foreign_chain_config`, validate that the chain has a corresponding inspector in `ForeignChainInspectors`. Chains without an inspector should be excluded from the registration set automatically.
2. **Contract-side guard**: Alternatively, maintain a contract-level allowlist of chains that have a known, deployed implementation, and reject `verify_foreign_transaction` for chains not on that list regardless of node registrations.
3. **Explicit documentation**: Mark `ForeignChainRpcRequest::Ethereum` and `ForeignChainRpcRequest::Solana` as unimplemented stubs with a clear comment, and add a compile-time or startup assertion that prevents nodes from registering them.

---

### Proof of Concept

1. All active MPC participants call `register_foreign_chain_config` with `ForeignChain::Ethereum` in their configuration (possible since no validation prevents it).
2. `get_supported_foreign_chains()` now returns `Ethereum` in the supported set.
3. An unprivileged user calls `verify_foreign_transaction` with `ForeignChainRpcRequest::Ethereum(...)` and attaches the required deposit.
4. The contract passes the chain check at line 535 and enqueues the yield request, consuming the deposit.
5. Every MPC node reaches `execute_foreign_chain_request` and hits `bail!("ForeignChainRpcRequest::Ethereum is unsupported")` at line 126.
6. No node ever calls `respond_verify_foreign_tx`. The yield times out, the request is cleaned up, and the deposit is not returned. [3](#0-2)

#[tokio::test]
async fn verify_foreign_transaction__should_timeout_without_response(
    #[case] rpc_request: ForeignChainRpcRequest,
) {
    let chain = rpc_request.chain();
    let setup = SandboxTestSetup::builder()
        .with_foreign_tx_domain()
        .build()
        .await;
    let foreign_tx_key = setup.foreign_tx_key();
    register_foreign_chain_configuration(chain, &setup.contract, &setup.mpc_signer_accounts).await;

    let user = setup.worker.dev_create_account().await.unwrap();

    let request_args = dtos::VerifyForeignTransactionRequestArgs {
        domain_id: dtos::DomainId(foreign_tx_key.domain_id().0),
        payload_version: ForeignTxPayloadVersion::V1,
        request: rpc_request,
    };

    let status = user
        .call(
            setup.contract.id(),
            method_names::VERIFY_FOREIGN_TRANSACTION,
        )
        .args_json(json!({ "request": request_args }))
        .deposit(NearToken::from_yoctonear(1))
        .max_gas()
        .transact_async()
        .await
        .unwrap();

    setup
        .worker
        .fast_forward(SIGNATURE_TIMEOUT_BLOCKS)
        .await
        .unwrap();
    let execution = status.await.unwrap();
    assert!(
        execution.is_failure(),
        "request should time out without a response"
    );
}
