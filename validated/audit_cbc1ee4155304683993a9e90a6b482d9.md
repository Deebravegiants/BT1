Let me look at the connection authentication and how harvesters connect to farmers to assess the "unprivileged" aspect.

## Code Analysis

The vulnerability claim centers on `_sync_started` in `chia/plot_sync/receiver.py`. Let me trace the exact execution path.

### `_sync_started` — ordering of `reset()` vs. validation [1](#0-0) 

```python
async def _sync_started(self, data: PlotSyncStart) -> None:
    if data.initial:
        self.reset()                              # ← (1) reset FIRST
    self._validate_identifier(data.identifier, True)  # ← (2) validate AFTER
    if data.last_sync_id != self._last_sync.sync_id:  # ← (3) check AFTER
        raise InvalidLastSyncIdError(...)
    if data.last_sync_id == data.identifier.sync_id:
        raise SyncIdsMatchError(...)
```

### `reset()` — what it clears [2](#0-1) 

After `reset()`: `_current_sync.next_message_id = 0`, `_last_sync.sync_id = 0`, `_plots = {}`, `_total_plot_size = 0`.

### `_validate_identifier` with `start=True` [3](#0-2) 

When `start=True`, **only `message_id_match`** is checked — `sync_id` is ignored. After `reset()`, `next_message_id = 0`, so any message with `message_id=0` passes.

## Attack Trace

**Precondition**: One full sync has completed. `_last_sync.sync_id = X` (non-zero), `_plots` has entries, `_current_sync = Sync()` (next_message_id=0).

**Attacker sends**: `PlotSyncStart(initial=True, last_sync_id=0, identifier=PlotSyncIdentifier(ts, sync_id=Z, message_id=0))` where `Z ≠ 0`.

**Step-by-step**:

1. `data.initial = True` → `reset()` called:
   - `_current_sync = Sync()` → `next_message_id=0`, `sync_id=0`
   - `_last_sync = Sync()` → `sync_id=0`
   - `_plots.clear()` → **all plots wiped**
   - `_total_plot_size = 0`

2. `_validate_identifier(identifier, start=True)`:
   - `message_id_match = (0 == 0)` → **True** (passes, because reset zeroed `next_message_id`)

3. `data.last_sync_id (0) != self._last_sync.sync_id (0)` → **False** (passes, because reset zeroed `_last_sync.sync_id`)

4. `data.last_sync_id (0) == data.identifier.sync_id (Z)` → **False** if Z≠0 (passes)

5. `_current_sync.sync_id = Z`, state set to `loaded`, `next_message_id` bumped to 1.

**Result**: All guards pass. `_plots` is empty, `_total_plot_size = 0`. The attack can be repeated indefinitely with a new `sync_id` each time, since after each attack `next_message_id` resets to 0 again on the next `reset()` call.

## Authentication Scope

The harvester-farmer connection uses a **private SSL CA** (not the public Chia CA): [4](#0-3) 

The attacker must hold a certificate signed by the farmer's private CA — i.e., must be a harvester that the farmer operator has provisioned. This is not an anonymous internet attacker. However, in multi-harvester deployments, a **compromised harvester** (one that has a valid cert) can execute this attack against the farmer it is connected to.

The farmer creates a `Receiver` per connected harvester peer: [5](#0-4)

## Verdict

### Title
Pre-validation `reset()` in `Receiver._sync_started` allows a connected harvester to wipe all farmer plot state — (`chia/plot_sync/receiver.py`)

### Summary
`_sync_started` calls `reset()` before performing identifier and `last_sync_id` validation. A malicious harvester with a valid SSL certificate can send `PlotSyncStart(initial=True, last_sync_id=0, message_id=0)` at any time to clear `_plots`, `_last_sync`, `_total_plot_size`, and all related state on the farmer's `Receiver`, then pass all subsequent guards because `reset()` has already zeroed the values those guards compare against.

### Finding Description
The ordering bug is in `_sync_started` at lines 210–211:

```python
if data.initial:
    self.reset()                          # destroys _last_sync.sync_id and next_message_id
self._validate_identifier(...)            # now compares against zeroed values
if data.last_sync_id != self._last_sync.sync_id:  # _last_sync.sync_id is 0 after reset
```

After `reset()`, `_last_sync.sync_id = 0` and `_current_sync.next_message_id = 0`. An attacker who sends `last_sync_id=0` and `message_id=0` passes both guards regardless of what the pre-attack sync state was. The attack is repeatable: each new `PlotSyncStart(initial=True, message_id=0)` re-triggers `reset()` and re-zeroes the counters.

### Impact Explanation
- `_plots` is cleared → farmer has zero plots visible to the block-signing and pool-partial path
- `_total_plot_size = 0` → pool sees zero plot count, partials stop
- A compromised harvester can loop this attack to prevent the farmer from ever accumulating plots again, causing a **persistent inability to farm blocks or submit pool partials**

This meets the High threshold: "Permanent or long-lived inability for honest farmers to process valid blocks or pool actions."

### Likelihood Explanation
Requires a harvester with a valid private-CA certificate connected to the target farmer. In multi-harvester setups (e.g., a farm with remote harvesters), a single compromised harvester machine is sufficient. No leaked farmer keys or admin access to the farmer are needed.

### Recommendation
Move all validation **before** calling `reset()`. Validate `last_sync_id` and the identifier against the **pre-reset** state, and only call `reset()` if validation succeeds:

```python
async def _sync_started(self, data: PlotSyncStart) -> None:
    self._validate_identifier(data.identifier, True)
    expected_last_sync_id = uint64(0) if data.initial else self._last_sync.sync_id
    if data.last_sync_id != expected_last_sync_id:
        raise InvalidLastSyncIdError(data.last_sync_id, expected_last_sync_id)
    if data.last_sync_id == data.identifier.sync_id:
        raise SyncIdsMatchError(State.idle, data.last_sync_id)
    if data.initial:
        self.reset()
    ...
```

### Proof of Concept
1. Complete one full sync (farmer `_plots` non-empty, `_last_sync.sync_id = X`).
2. Optionally start a second sync (to test mid-sync variant).
3. Send `PlotSyncStart(initial=True, last_sync_id=0, plot_file_count=0, harvesting_mode=..., identifier=PlotSyncIdentifier(ts, sync_id=1, message_id=0))`.
4. Assert `receiver.plots() == {}` and `receiver.total_plot_size() == 0`.
5. Assert `receiver.last_sync().sync_id == 0` (last sync record also wiped).
