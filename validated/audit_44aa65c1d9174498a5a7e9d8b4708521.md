### Title
Stale Reverse DRep Delegation Allows Previously Nominated DRep to Erase a Delegator's Current Delegation — (`eras/conway/impl/src/Cardano/Ledger/Conway/Rules/GovCert.hs`)

---

### Summary

During the Conway bootstrap phase (`pvMajor pv < natVersion @10`), when a stake credential redelegates from DRep-A to DRep-B, the old reverse-delegation entry (DRep-A → stakeCred in `drepDelegs`) is intentionally **not** removed. When DRep-A later submits a `ConwayUnRegDRep` certificate, the `clearDRepDelegations` helper in `GOVCERT` blindly sets `dRepDelegationAccountStateL .~ Nothing` for every credential still listed in its stale `drepDelegs` set — including credentials that have already moved to DRep-B. This silently erases the delegator's current DRep delegation, reducing DRep-B's active voting stake without the delegator's consent.

---

### Finding Description

**Step 1 — Stale reverse delegation is retained on redelegation (bootstrap phase only)**

In `conwayDelegTransition`, a `ConwayDelegCert` is processed by calling `processDelegationInternal` with `preserveIncorrectDelegation = pvMajor pv < natVersion @10`: [1](#0-0) 

Inside `processDelegationInternal`, the `delegVote` branch only **adds** the new reverse delegation to DRep-B's `drepDelegs`; it does **not** remove the old entry from DRep-A's `drepDelegs` when `preserveIncorrectDelegation` is `True`: [2](#0-1) 

The forward delegation (`dRepDelegationAccountStateL`) is correctly updated to DRep-B, but DRep-A's `drepDelegs` set still contains `stakeCred`. This is the acknowledged bug referenced as issue #4772.

**Step 2 — `ConwayUnRegDRep` uses the stale `drepDelegs` to clear delegations unconditionally**

When DRep-A submits a `ConwayUnRegDRep` certificate, `conwayGovCertTransition` reads DRep-A's `DRepState` (which still lists `stakeCred` in `drepDelegs`) and calls `clearDRepDelegations`: [3](#0-2) 

`clearDRepDelegations` performs no ownership check — it unconditionally sets `dRepDelegationAccountStateL .~ Nothing` for every credential in the set: [4](#0-3) 

Because `stakeCred` is still in DRep-A's stale `drepDelegs`, its forward delegation is overwritten to `Nothing`, even though it was already pointing to DRep-B.

**Step 3 — Test confirms the behavior**

The existing test suite explicitly verifies this outcome in the bootstrap phase: [5](#0-4) 

`expectNothingExpr (lookupDRepDelegation cred accounts)` confirms that after DRep-A unregisters, `stakeCred`'s delegation is `Nothing` — even though it was actively delegated to DRep-B.

---

### Impact Explanation

After DRep-A's unregistration clears `stakeCred`'s `dRepDelegationAccountStateL`, `computeDRepDistr` in `DRepPulser.hs` reads that field to assign stake to DReps: [6](#0-5) 

Since the field is now `Nothing`, `stakeCred`'s entire stake (UTxO balance + rewards) is excluded from every DRep's distribution. DRep-B loses voting weight it legitimately holds. If DRep-A coordinates this across multiple former delegators, it can materially shift the DRep vote ratio for any live governance action — potentially causing an action to pass or fail contrary to the honest outcome. This matches the **Medium** impact class: *attacker-controlled certificates modify governance voting power outside design parameters*, and at sufficient scale reaches **Critical**: *unauthorized governance action is enacted*.

---

### Likelihood Explanation

The attack is executable by any registered DRep during the Conway bootstrap phase (protocol version 9). The attacker must:
1. Have previously attracted delegators who have since redelegated away.
2. Submit a single `ConwayUnRegDRep` certificate (standard, permissionless transaction).
3. Accept the loss of their DRep deposit as the cost of the attack.

No privileged access, key compromise, or consensus majority is required. The bootstrap phase is a live, finite window on mainnet, making this a realistic and time-bounded threat.

---

### Recommendation

`clearDRepDelegations` in `conwayGovCertTransition` must verify that each credential's `dRepDelegationAccountStateL` still points to the unregistering DRep before clearing it. The existing `unDelegReDelegDRep` helper in `VState.hs` already performs this ownership check correctly: [7](#0-6) 

Replace the `clearDRepDelegations` fold in `ConwayUnRegDRep` with a guarded variant that only clears `dRepDelegationAccountStateL` when it equals `Just (DRepCredential cred)`, or refactor to reuse `unDelegReDelegDRep` per credential. This ensures that a DRep's unregistration cannot affect delegators who have already moved to a different DRep.

---

### Proof of Concept

```
Epoch N (bootstrap, PV9):
  1. Alice submits RegDepositDelegTxCert(stakeCred, DelegVote(DRepCredential drepA), deposit)
     → dRepDelegationAccountStateL(stakeCred) = Just (DRepCredential drepA)
     → drepA.drepDelegs = {stakeCred}

  2. Alice submits DelegTxCert(stakeCred, DelegVote(DRepCredential drepB))
     → processDelegationInternal called with preserveIncorrectDelegation = True
     → dRepDelegationAccountStateL(stakeCred) = Just (DRepCredential drepB)  [updated]
     → drepB.drepDelegs = {stakeCred}                                         [added]
     → drepA.drepDelegs = {stakeCred}                                         [NOT removed — stale]

  3. DRep-A submits ConwayUnRegDRep(drepA, refund)
     → clearDRepDelegations({stakeCred}) runs
     → dRepDelegationAccountStateL(stakeCred) := Nothing                      [WRONG: clears drepB delegation]
     → drepB.drepDelegs still contains stakeCred                              [inconsistent]

  4. computeDRepDistr runs at epoch boundary:
     → accountState ^. dRepDelegationAccountStateL = Nothing
     → Alice's stake is NOT counted for drepB (or anyone)
     → drepB's voting power is reduced without Alice's consent
```

```haskell
unDelegReDelegDRep stakeCred accountState mNewDRep =
  fromMaybe (vsDRepsL %~ addNewDelegation) $ do
    dRep@(DRepCredential dRepCred) <- accountState ^. dRepDelegationAccountStateL
    pure $
      -- There is no need to update set of delegations if delegation is unchanged
      if Just dRep == mNewDRep
        then id
        else
          vsDRepsL %~ addNewDelegation . Map.adjust (drepDelegsL %~ Set.delete stakeCred) dRepCred
  where
    addNewDelegation =
      case mNewDRep of
        Just (DRepCredential dRepCred) ->
          Map.adjust (drepDelegsL %~ Set.insert stakeCred) dRepCred
        _ -> id
```
