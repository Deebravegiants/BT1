Audit Report

## Title
Zero-Amount ERC20 Transfer in `MerkleDistributor.claim()` Freezes All User Claims When `feeInBPS == 0` - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary
`MerkleDistributor.claim()` unconditionally calls `safeTransfer(protocolTreasury, fee)` even when `fee == 0`. When `feeInBPS` is set to zero and the distributed token reverts on zero-amount transfers, every call to `claim()` reverts, permanently freezing all unclaimed yield inside the contract. The sibling contracts `KernelMerkleDistributor` and `KernelTop100MerkleDistributor` already guard against this with `if (fee > 0)`, making this an inconsistency with a concrete impact.

## Finding Description
In `MerkleDistributor.claim()`, after computing `fee = (claimableAmount * feeInBPS) / 10_000`, the contract unconditionally executes `IERC20(token).safeTransfer(protocolTreasury, fee)` with no zero-check: [1](#0-0) 

`setFeeInBPS` permits setting `feeInBPS` to `0` — the only guard is `> MAX_FEE_IN_BPS`: [2](#0-1) 

When `feeInBPS == 0`, `fee == 0`, and `safeTransfer(protocolTreasury, 0)` reverts for any ERC20 that disallows zero-amount transfers (e.g., LEND at `0x80fB784B7eD66730e8b1DBd9820aFD29931aab03`). The state update at lines 134–135 has already been written before the revert, but since the entire transaction reverts, the user's claim is not marked as processed — yet the transfer always fails, making the function permanently uncallable under this configuration.

By contrast, both sibling contracts guard the fee transfer: [3](#0-2) [4](#0-3) 

`MerkleDistributor` is missing this guard entirely.

## Impact Explanation
When `feeInBPS == 0` and the distributed token reverts on zero-amount transfers, every call to `claim()` reverts. All tokens allocated to users in the merkle tree are frozen inside the contract — users cannot receive their entitled yield/rewards. This constitutes **permanent freezing of unclaimed yield** (Medium severity). The state is stuck until the owner raises `feeInBPS` to a non-zero value, which changes the economics of every pending claim.

## Likelihood Explanation
- `setFeeInBPS(0)` is a natural "no-fee" configuration for a distribution round; the owner is not acting maliciously.
- `setToken()` allows any ERC20 to be used, including tokens that revert on zero-amount transfers.
- The combination requires no attacker — it is triggered by any user calling `claim()` under this configuration.
- The preconditions are realistic operational choices, not edge cases.

## Recommendation
Add a zero-check before the fee transfer, mirroring the pattern already used in `KernelMerkleDistributor` and `KernelTop100MerkleDistributor`:

```solidity
if (fee > 0) {
    IERC20(token).safeTransfer(protocolTreasury, fee);
}
```

## Proof of Concept
1. Deploy `MerkleDistributor` with LEND (`0x80fB784B7eD66730e8b1DBd9820aFD29931aab03`) as the token — LEND reverts on zero-amount transfers.
2. Owner calls `setFeeInBPS(0)` — valid since `0 <= MAX_FEE_IN_BPS`.
3. Owner sets a merkle root with user allocations via `setMerkleRoot()`.
4. Any user calls `claim(index, account, cumulativeAmount, merkleProof)`.
5. `fee = (claimableAmount * 0) / 10_000 = 0`.
6. `IERC20(token).safeTransfer(protocolTreasury, 0)` reverts because LEND disallows zero-amount transfers.
7. All claims revert; all user funds are frozen in the contract.
