### Summary

An unprivileged attacker can craft a CBOR-encoded `RequireMOf` (Shelley) or `TimelockMOf` (Allegra+) native script with a negative integer for `m`. The CBOR decoder accepts it without error, and the evaluator's `isValidMOf` short-circuits to `True` for any `n <= 0`, making the script unconditionally satisfied with zero witnesses. Funds locked by such a script can be spent by anyone who supplies the script bytes, with no key signatures required.

The "deterministic disagreement between nodes" framing in the question is **incorrect** â€” all nodes run the same decoder and evaluator and will agree. The actual impact is **unauthorized spending of funds locked by a malformed script**.

**Type storage â€” `Int` instead of `Natural`**

`TimelockMOf` does the same, and the source code itself contains an explicit acknowledgment of the problem: [2](#0-1) 

**CBOR decoder â€” no non-negativity guard**

The `DecCBOR (Annotator (MultiSigRaw era))` instance decodes `m` with a bare `decCBOR` call, which resolves to `instance DecCBOR Int` â†’ `decodeInt`. CBOR major type 1 (negative integers) is valid CBOR, so any negative value is accepted without error: [3](#0-2)

For any `n <= 0` â€” including all negative values â€” `isValidMOf` returns `True` immediately, regardless of the sub-script list and regardless of what witnesses are present.

### Impact Explanation

If funds are locked to the address derived from a `RequireMOf (-1) []` (or any `m < 0`) script, **any party who possesses the script bytes** can spend those funds by submitting a transaction with the script in the witness set and zero key signatures. The ledger accepts the transaction because `evalMultiSig`/`evalTimelock` returns `True`.

This is **not** a deterministic disagreement between nodes â€” all nodes run identical code and reach the same conclusion. The impact is **Critical: direct, unauthorized loss of ADA** locked by such a script.

---

### Likelihood Explanation

The attacker cannot unilaterally lock someone else's funds to a malformed script. Exploitation requires one of:

1. **Malicious party in a multi-sig setup**: A participant proposes a script that appears to be "2-of-3" but is CBOR-encoded with `m = -1`. The victim inspects only the script hash (not the raw bytes), locks funds, and the attacker drains them alone.
2. **Attacker-controlled address generation**: The attacker publishes a "safe" multi-sig address (e.g., for a shared wallet or escrow) that is actually backed by a `RequireMOf (-1) []` script.

Both scenarios are realistic in multi-sig wallet tooling. The code comment at line 184 of `Allegra/Scripts.hs` confirms the developers are aware of the behavior but have not added a decoder-level guard.

---

### Recommendation

1. Change the `m` field type from `!Int` to `!Natural` in both `MultiSigMOf` and `TimelockMOf`, or add an explicit non-negativity check in the CBOR decoder (analogous to `decodeNatural`).
2. Add a decoder-level guard in the `DecCBOR (Annotator (MultiSigRaw era))` and `DecCBOR (Annotator (TimelockRaw era))` instances that rejects `m < 0` with a `DecoderErrorCustom`.
3. Remove the comment at `Allegra/Scripts.hs:184` that documents the behavior as acceptable.
