Audit Report

## Title
Wrong Condition in ETH Deposit Limit Check Allows Limit Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses `totalAssetDeposits > limit` for ETH instead of `totalAssetDeposits + amount > limit`, meaning the incoming deposit amount is never factored into the cap check. Any ETH deposit that would push the running total past the configured limit is accepted, while the equivalent ERC-20 path correctly includes `+ amount`. The deposit limit guarantee is broken for ETH.

## Finding Description
In `LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` (L676–682), the ETH branch omits the incoming `amount`:

```solidity
// L678-681
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // missing + amount
}
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct for ERC-20
```

This function is called unconditionally inside `_beforeDeposit` (L661–663), which is the sole guard for the public `depositETH` entry point (L76–93). `depositETH` requires no privileged role — only `nonReentrant`, `whenNotPaused`, and `onlySupportedAsset`. As a result, when `totalAssetDeposits` is just below the cap, a deposit whose `amount` alone would exceed the cap passes the check (`totalAssetDeposits > limit` evaluates `false`) and rsETH is minted for the full over-limit amount.

## Impact Explanation
The ETH deposit cap set by the admin via `lrtConfig.depositLimitByAsset` is not enforced for the deposit that crosses the boundary. rsETH is minted against ETH in excess of the intended protocol limit. No funds are stolen from existing holders in a single transaction, but the protocol fails to deliver its promised deposit-limit invariant. This matches **Low — Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
Any unprivileged depositor can trigger this with a single call to `depositETH` when `totalAssetDeposits` is near the cap. No special role, front-running, or external dependency is required. The condition is reachable on every `depositETH` call when the running total is within one deposit's distance of the limit.

## Recommendation
Apply the same `+ amount` inclusion used for ERC-20 assets:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets ETH deposit limit to 100 ETH via `lrtConfig.depositLimitByAsset`.
2. `getTotalAssetDeposits(ETH_TOKEN)` = 99 ETH.
3. Attacker calls `depositETH{value: 10 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 10 ether)` evaluates `99 > 100` → `false` → no revert.
5. 10 ETH is accepted and rsETH is minted; total becomes 109 ETH — 9 ETH above the cap.
6. For any ERC-20 in the same scenario, step 4 evaluates `99 + 10 > 100` → `true` → revert, correctly blocking the deposit.

Foundry test plan: deploy `LRTDepositPool` on a local fork, set ETH limit to 100 ether, seed 99 ether of prior deposits, call `depositETH{value: 10 ether}`, assert the call succeeds and `getTotalAssetDeposits(ETH_TOKEN)` returns 109 ether, confirming the cap was bypassed.
