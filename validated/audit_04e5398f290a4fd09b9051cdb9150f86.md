Looking at the exact validation logic in `respond_verify_foreign_tx` and the `resolve_yields_for` mechanism, I need to determine whether the contract binds `response.payload_hash` to the specific request being answered.

### Title
Unbound `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay by a Single Byzantine Participant — (`crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies only that the submitted ECDSA signature is valid over the caller-supplied `payload_hash` against the root public key. It does **not** verify that `payload_hash` equals `SHA256(Borsh(ForeignTxSignPayload{request=R2.request, values=<actual_observed_values>}))`. A single Byzantine attested participant can therefore replay a `(payload_hash=H, sig=sig(H))` pair from any prior completed request R1 as the response to a different pending request R2, causing R2's caller to receive a cryptographically valid MPC signature whose hash encodes R1's transaction data rather than R2's.

---

### Finding Description

The validation block in `respond_verify_foreign_tx` is:

```rust
// crates/contract/src/lib.rs:718-747
let payload_hash: [u8; 32] = response.payload_hash.0;
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
).is_ok()
``` [1](#0-0) 

The contract checks: "is `sig` a valid ECDSA signature over `payload_hash` under the root public key?" It does **not** check: "does `payload_hash` equal `SHA256(Borsh(ForeignTxSignPayload{request=R2.request, values=...}))`?"

After this check passes, `resolve_yields_for` is called with the attacker-supplied `response` (which includes the attacker-chosen `payload_hash`) serialized verbatim and delivered to R2's caller:

```rust
// crates/contract/src/lib.rs:749-753
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

The `ForeignTxSignPayload` is defined as `SHA256(Borsh(ForeignTxSignPayloadV1 { request: ForeignChainRpcRequest, values: Vec<ExtractedValue> }))`, binding both the request and the observed values: [3](#0-2) 

The node-side `build_signature_request` correctly computes `payload_hash` from the actual observed values for the specific request: [4](#0-3) 

But the contract never enforces this binding on the response path.

The gate `assert_caller_is_attested_participant_and_protocol_active` requires only a **single** attested participant — not a threshold: [5](#0-4) 

---

### Impact Explanation

A single Byzantine attested MPC participant can:

1. Observe any prior completed `respond_verify_foreign_tx(R1, {payload_hash=H, sig=sig(H)})` call on-chain (public blockchain data).
2. Wait for a new pending request R2 (different `tx_id`, different `ForeignChainRpcRequest`).
3. Call `respond_verify_foreign_tx(R2, {payload_hash=H, sig=sig(H)})` before any honest node responds.
4. The contract accepts: `sig(H)` is valid over `H` under the root key ✓, R2 is a pending request ✓.
5. R2's caller receives `{payload_hash=H, sig=sig(H)}` — a valid MPC root-key signature whose hash encodes R1's `(request, values)`, not R2's.

The caller of R2 receives a cryptographically valid MPC attestation that does not correspond to R2's actual foreign-chain transaction. Any bridge contract that trusts the MPC response without independently recomputing `SHA256(Borsh(ForeignTxSignPayload{R2.request, expected_values}))` and comparing it to `payload_hash` will accept a forged attestation, enabling invalid bridge execution.

The design documentation explicitly states that callers *can* verify by reconstructing the hash locally — but this is not enforced by the contract, and the contract is the trusted attestation layer. [6](#0-5) 

---

### Likelihood Explanation

- Requires being a single attested MPC participant (Byzantine node below threshold). This is within the explicit MPC threat model.
- The replayed `(H, sig(H))` is observable from any prior on-chain `respond_verify_foreign_tx` transaction — no off-chain secret is needed.
- The attacker only needs to race honest nodes to submit the response for R2. Since the attacker controls a node, they observe R2 from the NEAR indexer at the same time as honest nodes and can submit immediately.
- No threshold collusion, no TEE compromise, no key leakage required.

---

### Recommendation

In `respond_verify_foreign_tx`, recompute the expected `payload_hash` from the `request` field and reject if it does not match `response.payload_hash`. Concretely, the contract must store the expected `payload_hash` at request submission time (computed by the node and committed on-chain), or the contract must enforce that `payload_hash` is derived from the request key. The simplest fix is to require the submitting node to also provide the `ForeignTxSignPayload` (or its preimage fields), recompute `SHA256(Borsh(...))` on-chain, and assert equality with `response.payload_hash` before accepting.

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
