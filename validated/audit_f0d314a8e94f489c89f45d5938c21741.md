Audit Report

## Title
`instantWithdrawal` Can Drain Vault Assets Reserved for Queued Withdrawal Requests Due to Static `queuedWithdrawalsBuffer` Not Tracking `assetsCommitted` - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager.instantWithdrawal` checks available vault assets via `LRTUnstakingVault.getAssetsAvailableForInstantWithdrawal`, which subtracts a static, operator-set `queuedWithdrawalsBuffer` from the vault balance. This buffer is never automatically synchronized with `assetsCommitted`, the dynamically tracked amount owed to pending queued withdrawal requests. When the buffer is zero (the default) or stale, any rsETH holder can drain vault assets that are needed to service queued requests, temporarily freezing funds for those users.

## Finding Description
`assetsCommitted[asset]` is incremented in `initiateWithdrawal` at `LRTWithdrawalManager.sol:173` and decremented in `_unlockWithdrawalRequests` at line 802. It accurately reflects the total asset amount owed to pending queued requests at all times.

`instantWithdrawal` (lines 212–253) does not consult `assetsCommitted` at all. Instead, it calls `unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)` (line 231), which computes `vaultBalance - queuedWithdrawalsBuffer[asset]` (LRTUnstakingVault.sol lines 235–237). `queuedWithdrawalsBuffer` is a static value set exclusively by an operator via `setQueuedWithdrawalsBuffer` (lines 199–209) and defaults to `0`.

`unlockQueue` constructs `totalAvailableAssets` as `unstakingVault.balanceOf(asset)` (LRTWithdrawalManager.sol line 849) and immediately reverts if it is zero (line 297). If the vault has been drained by instant withdrawals, `unlockQueue` cannot proceed and queued requests remain locked.

The structural desync: `assetsCommitted` lives in `LRTWithdrawalManager` and grows with every `initiateWithdrawal` call, while `queuedWithdrawalsBuffer` lives in `LRTUnstakingVault` and is only updated by manual operator intervention. There is no on-chain link between them.

## Impact Explanation
Users who called `initiateWithdrawal` have already transferred their rsETH to the withdrawal manager contract. If instant withdrawals drain the vault before `unlockQueue` is called, those users cannot have their requests unlocked until new assets are unstaked from EigenLayer and deposited into the vault. This constitutes **temporary freezing of funds** for an unbounded set of queued withdrawal users, matching the allowed Medium impact class.

## Likelihood Explanation
- `instantWithdrawal` is gated only by `isInstantWithdrawalEnabled[asset]`, a manager toggle. Once enabled, **any rsETH holder** can call it with no additional privilege.
- `queuedWithdrawalsBuffer` defaults to `0`, providing zero protection immediately upon enabling instant withdrawals.
- Even if the operator sets the buffer correctly at one point, it becomes stale as new `initiateWithdrawal` calls increase `assetsCommitted`. The operator must continuously track and update the buffer off-chain.
- No collusion, oracle manipulation, or special role is required beyond the manager having enabled the feature.

**Likelihood: Medium.**

## Recommendation
Replace the static-buffer check in `instantWithdrawal` with a direct comparison against `assetsCommitted` inside `LRTWithdrawalManager`:

```solidity
uint256 vaultBalance = unstakingVault.balanceOf(asset);
uint256 reservedForQueued = assetsCommitted[asset];
uint256 availableForInstant = vaultBalance > reservedForQueued
    ? vaultBalance - reservedForQueued
    : 0;
if (assetAmountUnlocked > availableForInstant) revert CantInstantWithdrawMoreThanAvailable();
```

This eliminates the cross-contract desync entirely and ensures instant withdrawals can never consume assets already committed to pending queued requests.

## Proof of Concept
1. `LRTUnstakingVault` holds 100 ETH (freshly unstaked from EigenLayer).
2. Multiple users have called `initiateWithdrawal`; `assetsCommitted[ETH] = 80 ETH`.
3. `queuedWithdrawalsBuffer[ETH] = 0` (default, operator has not updated it).
4. Manager has set `isInstantWithdrawalEnabled[ETH] = true`.
5. Attacker holds rsETH worth 100 ETH and calls `instantWithdrawal(ETH, rsETHAmount, "")`.
6. `getAssetsAvailableForInstantWithdrawal(ETH)` returns `100 - 0 = 100 ETH`; check passes.
7. `unstakingVault.redeem(ETH, 100 ETH)` drains the vault to 0.
8. Operator calls `unlockQueue(ETH, ...)`: `_createUnlockParams` sets `totalAvailableAssets = unstakingVault.balanceOf(ETH) = 0`; line 297 reverts with `AmountMustBeGreaterThanZero`.
9. The 80 ETH owed to queued withdrawal users cannot be unlocked. Their rsETH is held by the withdrawal manager and they are frozen until new assets arrive from EigenLayer unstaking.

**Foundry test outline:**
- Deploy contracts with ETH as a supported asset.
- Have three users call `initiateWithdrawal` totalling 80 ETH committed.
- Fund the vault with 100 ETH (simulating EigenLayer unstaking completion).
- Enable instant withdrawals; leave `queuedWithdrawalsBuffer` at 0.
- Have an attacker call `instantWithdrawal` for 100 ETH worth of rsETH; assert it succeeds.
- Assert `unstakingVault.balanceOf(ETH) == 0`.
- Call `unlockQueue`; assert it reverts with `AmountMustBeGreaterThanZero`.
