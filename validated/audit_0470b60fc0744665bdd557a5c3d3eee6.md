### Title
Congestion Check Bypassed via `SpiceCoreWriterActor` Lag Window — (`chain/chain/src/spice/chunk_application.rs`, `chain/client/src/rpc_handler.rs`)

### Summary
When `ProtocolFeature::Spice` is enabled, the RPC transaction admission path uses `spice_shard_congestion_info` to check receiver-shard congestion. This function returns `None` when the node does not track the receiver shard AND `DBCol::execution_results` has not yet been populated by `SpiceCoreWriterActor`. When `None` is returned, `validate_tx` skips the congestion check entirely, admitting transactions to a fully congested shard.

### Finding Description

The timing gap is structural:

1. When block B+1 is processed, `record_uncertified_chunks_for_block` updates `DBCol::uncertified_chunks` **synchronously**, causing `get_last_certified_block_header` to return block B as certified.
2. `SpiceCoreWriterActor.handle_processed_block` writes block B's execution results to `DBCol::execution_results` **asynchronously** — it runs in a separate actor and is always at least one message-queue dispatch behind.
3. In the window between (1) and (2), `spice_shard_congestion_info` for a non-tracking node calls `get_execution_result_from_store` which reads `DBCol::execution_results` and returns `None`.
4. `receiver_congestion_info` is `None`; `validate_tx` enters the `if let Some(...)` branch only when `Some`, so the entire `shard_accepts_transactions()` check is skipped.
5. The transaction is admitted to the pool (or forwarded) without any congestion gate.

The question's claim that the handler "falls back to `cur_block.block_congestion_info()`" is incorrect — the `else` branch at line 200 is only taken when `spice_certified_header` is `None` (i.e., Spice is disabled). When Spice is enabled and `spice_shard_congestion_info` returns `None`, `receiver_congestion_info` is simply `None` and the check is skipped entirely. [7](#0-6) 

### Impact Explanation

An unprivileged user with an account on a tracked shard (signer) can submit transactions targeting a fully congested untracked shard (receiver) during the writer lag window. The transaction bypasses `InvalidTxError::ShardCongested` rejection at the RPC layer and enters the mempool. The chunk producer's `congestion_control_accepts_transaction` has the same `None`-means-skip behavior: [8](#0-7) 

So the transaction can also bypass chunk-production-level filtering and be included in a block, sending a receipt to the already-congested shard and worsening congestion.

### Likelihood Explanation

The writer lag window is always present — it is an inherent property of the async actor model. Under normal load the window is brief (one message-queue dispatch), but under high block throughput or CPU pressure it can span multiple blocks. The Spice design explicitly acknowledges execution lag (the test `slow_test_spice_congestion_and_bandwidth_with_execution_lag` uses a 3-block artificial delay). An attacker submitting transactions at high rate will statistically land submissions in this window without any special timing knowledge. [9](#0-8) 

### Recommendation

In `spice_shard_congestion_info`, when `get_execution_result_from_store` returns `None` for a block that `get_last_certified_block_header` has already declared certified, the function should not silently return `None`. Options:
- Return a sentinel `ExtendedCongestionInfo` representing maximum congestion (safe default: reject rather than admit).
- Block the RPC path until the writer catches up (return an error that causes forwarding).
- Cache the last-known congestion info per shard and use it as the fallback instead of `None`.

The `validate_tx` caller should also treat `None` as "unknown = reject" rather than "unknown = admit" when Spice is active and a certified block is available.

### Proof of Concept

Test-loop with `SpiceCoreWriterActor` artificially delayed (e.g., by holding the actor's message queue): congest a receiver shard, then submit a transaction from a tracked shard to the congested untracked shard during the delay window. Assert `process_tx` returns `ProcessTxResponse::ValidTx` (bypass) rather than `ProcessTxResponse::InvalidTx(InvalidTxError::ShardCongested { .. })`. [10](#0-9)

pub fn get_last_certified_block_header(
    chain_store: &ChainStoreAdapter,
    block_hash: &CryptoHash,
) -> Result<Arc<BlockHeader>, Error> {
    let uncertified_chunks = get_uncertified_chunks(chain_store, block_hash)?;
    let oldest_uncertified = find_oldest_uncertified_block_header(chain_store, uncertified_chunks)?;
    if let Some(header) = oldest_uncertified {
        Ok(chain_store.get_block_header(header.prev_hash())?)
    } else {
        // No uncertified-chunks tracking means the block has nothing to
        // certify: genesis, or a pre-spice block at the activation
        // boundary. Both are fully certified by definition.
        let header = chain_store.get_block_header(block_hash)?;
        debug_assert!(
            header.is_genesis() || !header.is_spice(),
            "post-genesis spice blocks should always have uncertified chunks"
        );
        Ok(header)
    }

pub fn spice_shard_congestion_info(
    chain_store: &ChainStoreAdapter,
    shard_layout: &ShardLayout,
    block_header: &BlockHeader,
    shard_id: ShardId,
) -> Option<ExtendedCongestionInfo> {
    let shard_uid = ShardUId::from_shard_id_and_layout(shard_id, shard_layout);
    let chunk_store = chain_store.chunk_store();
    let congestion_info =
        if let Ok(chunk_extra) = chunk_store.get_chunk_extra(block_header.hash(), &shard_uid) {
            chunk_extra.congestion_info()
        } else {
            get_execution_result_from_store(chain_store, block_header.hash(), shard_id)?
                .chunk_extra
                .congestion_info()
        };
    Some(ExtendedCongestionInfo::new(congestion_info, 0))
}
