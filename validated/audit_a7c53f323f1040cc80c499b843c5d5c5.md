Audit Report

## Title
Unbounded Withdrawal Request Queue Enables Spam That Causes Unbounded Gas in `_unlockWithdrawalRequests` - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager.initiateWithdrawal` imposes no per-user or global cap on the number of pending withdrawal requests. The internal `_unlockWithdrawalRequests` function processes them in an unbounded `while` loop with multiple storage reads and writes per iteration. Any rsETH holder can flood the queue with minimum-sized requests, causing `unlockQueue` calls to exhaust the block gas limit and preventing timely processing of legitimate withdrawals.

## Finding Description
`initiateWithdrawal` at [1](#0-0)  checks only that `rsETHUnstaked >= minRsEthAmountToWithdraw[asset]` and that the expected asset amount does not exceed currently available assets. There is no cap on how many requests a single user or all users combined may have outstanding.

Each call routes to `_addUserWithdrawalRequest`, which pushes a new nonce into `userAssociatedNonces[asset][msg.sender]` and increments `nextUnusedNonce[asset]` without any bound check: [2](#0-1) 

The operator-facing `unlockQueue` calls `_unlockWithdrawalRequests`, which iterates from `nextLockedNonce[asset]` up to the caller-supplied `firstExcludedIndex` in an unbounded `while` loop: [3](#0-2) 

Each iteration performs multiple `SLOAD`/`SSTORE` operations on `withdrawalRequests`, `assetsCommitted`, and `unlockedWithdrawalsCount`. The two `break` conditions (withdrawal delay and available asset amount) do not prevent gas exhaustion when the attacker's requests have all aged past `withdrawalDelayBlocks` and sufficient assets exist to cover them. Legitimate users' requests queued behind the spam entries cannot be reached in the same call.

By contrast, `KernelDepositPool.initiateWithdrawal` explicitly guards against this pattern with a per-user cap: [4](#0-3) 

`LRTWithdrawalManager` has no equivalent guard.

## Impact Explanation
**Medium — Unbounded gas consumption / Temporary freezing of funds.**

With a sufficiently large queue of spam entries that have passed the withdrawal delay, a single `unlockQueue` call iterates over all of them, each requiring multiple storage operations. For large enough N, the transaction reverts with out-of-gas. The operator is forced into ever-smaller batches; if the attacker continuously re-fills the queue between batches, legitimate users' withdrawal requests cannot be unlocked in a timely manner, constituting a temporary freeze of their funds. Both "Unbounded gas consumption" (Medium) and "Temporary freezing of funds" (Medium) are explicitly within the allowed impact scope.

## Likelihood Explanation
**Medium.** Any rsETH holder can execute this attack. The attacker's rsETH is returned when requests are eventually processed, so the only sustained cost is gas for `initiateWithdrawal` calls. The `minRsEthAmountToWithdraw` threshold is the sole throttle and can be set to a low value. The attack is repeatable across blocks and across multiple attacker addresses, making it difficult for the operator to outpace the spam.

## Recommendation
1. **Add a per-user pending-request cap** in `initiateWithdrawal`, analogous to `KernelDepositPool.maxNumberOfWithdrawalsPerUser`:
   ```solidity
   if (userAssociatedNonces[asset][msg.sender].length() >= maxPendingWithdrawalsPerUser)
       revert TooManyPendingWithdrawals();
   ```
2. **Add an iteration cap** inside `_unlockWithdrawalRequests` so a single call cannot process more than a protocol-defined maximum number of entries regardless of `firstExcludedIndex`.
3. **Raise `minRsEthAmountToWithdraw`** to a value that makes mass spam economically prohibitive.

## Proof of Concept
1. Attacker acquires N × `minRsEthAmountToWithdraw[asset]` rsETH (N in the hundreds to thousands, spread across blocks or addresses to bypass the `ExceedAmountToWithdraw` check per call).
2. Attacker calls `initiateWithdrawal(asset, minRsEthAmountToWithdraw[asset], "")` N times, creating N entries in the global queue.
3. After `withdrawalDelayBlocks` pass, all N entries are eligible for unlocking.
4. Operator calls `unlockQueue(asset, nextUnusedNonce[asset], ...)`.
5. `_unlockWithdrawalRequests` enters the `while` loop and iterates N times, each performing multiple `SLOAD`/`SSTORE` operations.
6. For sufficiently large N, the transaction reverts with out-of-gas; no legitimate withdrawal is unlocked.
7. Attacker re-initiates new requests between operator batches, keeping the queue perpetually congested.

**Foundry test plan:** Deploy `LRTWithdrawalManager` on a local fork; fund an attacker with sufficient rsETH; loop `initiateWithdrawal` to create 500–1000 minimum-sized requests; advance blocks past `withdrawalDelayBlocks`; call `unlockQueue` with `gas: 30_000_000` and assert the call reverts with out-of-gas or that `gasleft()` after the call is near zero.
