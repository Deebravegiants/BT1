Audit Report

## Title
ETH Donation via Unguarded `receive()` Inflates `rsETHPrice`, Causing Zero-rsETH Mints for Depositors - (File: contracts/LRTDepositPool.sol / contracts/LRTOracle.sol)

## Summary
`LRTDepositPool.receive()` accepts ETH from any caller with no access control, and `getETHDistributionData()` counts the raw `address(this).balance` as protocol TVL. Because `pricePercentageLimit` is never initialized and defaults to zero, the public `updateRSETHPrice()` function applies no cap on price increases. An attacker can donate a large ETH amount, trigger a price update to inflate `rsETHPrice` to an astronomically large value, and cause subsequent depositors who pass `minRSETHAmountExpected = 0` to receive zero rsETH for their ETH, permanently losing their funds while the attacker's single rsETH share represents the entire pool.

## Finding Description

**Root cause 1 — Unguarded ETH balance counted as TVL:**

`receive()` accepts ETH from any address with no restriction: [1](#0-0) 

`getETHDistributionData()` then counts the entire raw balance as protocol-owned ETH: [2](#0-1) 

**Root cause 2 — Public price update with no default rate-limit:**

`updateRSETHPrice()` is callable by any unprivileged address: [3](#0-2) 

The price-increase guard is gated on `pricePercentageLimit > 0`, but `pricePercentageLimit` is never set in `initialize()` and defaults to `0`, so `isPriceIncreaseOffLimit` is always `false`: [4](#0-3) [5](#0-4) 

**Root cause 3 — rsETH minting divides by the now-inflated cached price:**

`getRsETHAmountToMint` uses `lrtOracle.rsETHPrice()` (the cached, inflated value). With `rsETHPrice ≈ 1000e36`, any realistic deposit amount divided by that value truncates to zero: [6](#0-5) 

**Root cause 4 — Zero-rsETH mint is silently accepted:**

The only guard is `rsethAmountToMint < minRSETHAmountExpected`. When the caller passes `minRSETHAmountExpected = 0` (the default in most integrations), `0 < 0` is false and the deposit proceeds, absorbing the caller's ETH with no shares issued: [7](#0-6) 

**Why the daily fee-mint limit does not block the attack:**

When `rsETHPrice` is inflated to ~1000e36, `rsethAmountToMintAsProtocolFee = protocolFeeInETH * 1e18 / newRsETHPrice` truncates to zero regardless of `protocolFeeInBPS`. `_checkAndUpdateDailyFeeMintLimit(0)` evaluates `0 + 0 > maxFeeMintAmountPerDay` which is false for any non-zero limit, so `updateRSETHPrice()` completes successfully. [8](#0-7) 

**Attacker extraction path via `initiateWithdrawal`:**

After the victim deposits, `getExpectedAssetAmount(ETH, 1_wei_rsETH) = 1 * rsETHPrice / assetPrice ≈ 1000e18` (≈ 1000 ETH). `getAvailableAssetAmount(ETH)` returns `getTotalAssetDeposits(ETH) ≈ 1001e18`, so the `ExceedAmountToWithdraw` guard passes. The attacker queues a withdrawal for their 1 wei rsETH expecting ~1000 ETH. When an operator calls `unlockQueue` (normal protocol operations) with loose or default price bounds, the request is processed and the attacker completes the withdrawal after the delay. [9](#0-8) [10](#0-9) 

## Impact Explanation

**Critical — Direct theft of any user funds.**

The victim deposits ETH and receives zero rsETH; their ETH is permanently absorbed into the pool with no claim token issued. The attacker's single rsETH share represents the entire pool value and can be redeemed for the victim's ETH through the normal withdrawal flow. This is a direct, irreversible loss of depositor principal, matching the "Direct theft of any user funds" critical impact class.

## Likelihood Explanation

Any unprivileged external account can execute this attack permissionlessly. The only capital requirement is the donation amount (which the attacker recovers via withdrawal) plus a negligible seed deposit. No privileged role, governance action, or victim mistake beyond passing `minRSETHAmountExpected = 0` (the standard default) is required. The attack is repeatable and can be executed atomically in a single block for the donation and price-update steps.

## Recommendation

1. **Remove or restrict `receive()`**: Replace the open `receive()` with a whitelist of trusted callers (e.g., NodeDelegator, RewardReceiver, LRTConverter) matching the pattern already used by `receiveFromRewardReceiver()`, `receiveFromLRTConverter()`, and `receiveFromNodeDelegator()`. Reject ETH from arbitrary senders.
2. **Set `pricePercentageLimit` in `initialize()`**: Initialize `pricePercentageLimit` to a safe non-zero value (e.g., `1e16` for 1%) so the price-increase guard is active from deployment.
3. **Reject zero-rsETH mints unconditionally**: In `_beforeDeposit`, add `if (rsethAmountToMint == 0) revert ZeroRsETHMinted();` regardless of `minRSETHAmountExpected`.
4. **Exclude untracked ETH from TVL**: Track deposited ETH separately (e.g., a `depositedETH` counter incremented only by legitimate deposit paths) and use that counter instead of `address(this).balance` in `getETHDistributionData()`.
