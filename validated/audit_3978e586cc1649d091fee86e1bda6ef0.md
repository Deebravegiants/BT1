Audit Report

## Title
FIFO Queue `break` Logic Blocks Satisfiable Withdrawal Requests With No User Cancel Path - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`_unlockWithdrawalRequests` halts the entire FIFO unlock loop with a `break` the moment one request's `payoutAmount` exceeds `availableAssetAmount`, preventing all subsequent requests — including ones the vault could fully satisfy — from being unlocked. Because no `cancelWithdrawal()` function exists anywhere in the contract or its interface, users whose requests are queued behind a blocking request have no mechanism to reclaim their locked rsETH.

## Finding Description
When `initiateWithdrawal` is called, rsETH is immediately transferred to the contract and `assetsCommitted[asset]` is incremented by `expectedAssetAmount` ( [1](#0-0) ). The operator-driven `_unlockWithdrawalRequests` loop iterates `nextLockedNonce_` sequentially and executes a hard `break` at line 800 when `availableAssetAmount < payoutAmount` ( [2](#0-1) ). Because `nextLockedNonce[asset]` only advances when a request is successfully processed ( [3](#0-2) ), any request at nonce N that cannot be satisfied freezes nonces N+1, N+2, … regardless of how small those requests are. `_processWithdrawalCompletion` enforces this gate strictly: it reverts with `WithdrawalLocked` if `usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]` ( [4](#0-3) ). No `cancelWithdrawal` function exists in the contract or its interface ( [5](#0-4) ), leaving blocked users with zero recourse.

The critical distinction from a pure liquidity issue: the vault may hold sufficient assets to satisfy all *subsequent* requests, yet the `break` prevents them from being processed. This is a code logic defect, not merely insufficient liquidity.

## Impact Explanation
**Medium — Temporary freezing of funds.** Users whose withdrawal requests are queued behind a blocking request cannot complete their withdrawals and cannot cancel to reclaim their rsETH. Their funds remain locked in the contract until the vault accumulates enough assets to satisfy the blocking request first. This matches the allowed impact "Medium. Temporary freezing of funds."

## Likelihood Explanation
**Medium.** Any rsETH holder can call `initiateWithdrawal` — no privileges required. The blocking condition is triggered when the vault's asset balance drops after a large request is submitted (e.g., EigenLayer slashing, other withdrawals draining the vault). The `break` is deterministic and requires no operator action to trigger. The condition is realistic on a live protocol where vault balances fluctuate.

## Recommendation
1. Replace `break` with `continue` at line 800 of `_unlockWithdrawalRequests` so that smaller subsequent requests are not blocked by one unsatisfiable large request.
2. Add a `cancelWithdrawal()` function that allows a user to remove their own pending (not yet unlocked) request, decrement `assetsCommitted[asset]` by the original `expectedAssetAmount`, and return the locked rsETH to the user.

## Proof of Concept
1. Alice calls `initiateWithdrawal(stETH, largeAmount)`. Vault holds `X` stETH; `expectedAssetAmount = X - 1`. `assetsCommitted[stETH] += X - 1`. Alice's nonce = N.
2. Bob calls `initiateWithdrawal(stETH, 1 wei rsETH)`. Bob's nonce = N+1.
3. Vault balance drops to `X - 2` (slashing or other withdrawal).
4. Operator calls `unlockQueue(stETH, ...)`. Loop reaches nonce N: `payoutAmount ≈ X-1 > availableAssetAmount = X-2` → `break`. `nextLockedNonce[stETH]` stays at N.
5. Bob calls `completeWithdrawal(stETH, "")`. `_processWithdrawalCompletion` pops Bob's nonce N+1, checks `N+1 >= nextLockedNonce[stETH] (= N)` → **reverts `WithdrawalLocked`**.
6. Bob's rsETH remains locked. No `cancelWithdrawal` exists. Bob cannot exit until the vault accumulates enough to satisfy Alice's request first — even though the vault holds `X-2` which is more than enough for Bob's 1 wei request.
