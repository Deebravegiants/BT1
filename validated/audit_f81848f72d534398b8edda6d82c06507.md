Audit Report

## Title
Reward Tokens Permanently Locked When `totalKernelStaked` Drops to Zero During Active Reward Period - (`contracts/KERNEL/KernelDepositPool.sol`)

## Summary
When all stakers exit during an active reward window, `totalKernelStaked` reaches zero. The `rewardPerToken()` function returns the stored value unchanged while the `updateReward` modifier still advances `updatedAt` to the current time. Reward tokens that accrued during the zero-stake gap are permanently unclaimable, with no admin recovery path in the contract.

## Finding Description
`rewardPerToken()` (L408–414) short-circuits on `totalKernelStaked == 0`, returning `rewardPerTokenStored` without accumulating any new rewards. The `updateReward` modifier (L232–242) unconditionally executes `updatedAt = lastTimeRewardApplicable()` regardless of whether `totalKernelStaked` is zero. When the last staker calls `initiateWithdrawal` (L320–338), the modifier runs first with the current non-zero supply, correctly checkpointing `rewardPerTokenStored` and setting `updatedAt = now`. After the call body executes, `totalKernelStaked` becomes 0. For any subsequent elapsed time, `rewardRate` continues to tick but `rewardPerTokenStored` is frozen. When the next staker calls `stake`, the modifier again sets `updatedAt = lastTimeRewardApplicable()`, permanently skipping the entire zero-stake interval. The `notifyRewardAmount` guard (L570) only prevents *starting* a new period with zero stake; it does not prevent all tokens from being unstaked after a period has started. The contract contains no `sweep`, `recoverTokens`, or equivalent admin function to retrieve stranded reward tokens. The NatSpec (L17–22) explicitly acknowledges this behavior but relies solely on an operational mitigation with no code-level enforcement.

## Impact Explanation
Reward tokens (`rewardsToken` ERC-20) accumulate in `KernelDepositPool` and become permanently unclaimable by anyone. The magnitude equals `rewardRate × (duration of zero-stake gap)`, which can represent a substantial fraction of the entire reward budget. This matches **Permanent freezing of unclaimed yield** (Medium severity).

## Likelihood Explanation
Any staker can call `initiateWithdrawal` at any time without restriction during an active reward period. A coordinated or organic full-exit (e.g., in response to a market event or a better yield opportunity elsewhere) is a realistic scenario. No attacker collusion is required; ordinary user behavior is sufficient to trigger the loss. The condition is repeatable across multiple reward periods.

## Recommendation
1. **Track zero-stake intervals explicitly**: When `totalKernelStaked` drops to zero, record the timestamp. When it becomes non-zero again, advance `updatedAt` only to that recorded timestamp, excluding the zero-stake interval from the reward window rather than silently skipping it.
2. **Alternatively**, add a `recoverUnallocatedRewards()` admin function that computes the unallocated balance (`rewardsToken.balanceOf(address(this)) - sum_of_all_earned_rewards`) and transfers it to a treasury address.
3. For the integer-division dust in `notifyRewardAmount` (`receivedAmount % duration` tokens lost per notification): accumulate the remainder and carry it forward into the next `notifyRewardAmount` call.

## Proof of Concept
1. Admin calls `notifyRewardAmount(1_000e18)` with `duration = 30 days`; `rewardRate = 1_000e18 / (30 days)`.
2. Alice is the only staker: `balanceOf[alice] = 100e18`, `totalKernelStaked = 100e18`.
3. After 10 days, Alice calls `initiateWithdrawal(100e18)`. The `updateReward` modifier runs with `totalKernelStaked = 100e18`, correctly checkpointing ~333e18 worth of `rewardPerTokenStored`, then sets `updatedAt = now`. The call body sets `totalKernelStaked = 0`.
4. 10 days pass with no stakers. `rewardRate` ticks but `rewardPerTokenStored` is frozen.
5. Bob calls `stake(1e18)`. The `updateReward` modifier runs: `rewardPerToken()` returns `rewardPerTokenStored` unchanged (because `totalKernelStaked` was 0), then sets `updatedAt = lastTimeRewardApplicable()` (day 20).
6. The ~333e18 reward tokens that accrued during the 10-day zero-stake window (days 10–20) are now permanently stranded in the contract. Alice earned ~333e18 (days 0–10). Bob earns only ~333e18 (days 20–30). The middle third is locked forever with no recovery path.

**Foundry test plan**: Deploy `KernelDepositPool`, call `notifyRewardAmount`, have Alice stake, warp 10 days, have Alice call `initiateWithdrawal`, warp another 10 days, have Bob stake, warp to end of period, have Bob call `getReward`, assert `rewardsToken.balanceOf(address(pool)) > 0` (stranded tokens remain) and that the stranded amount equals approximately `rewardRate * 10 days`.
