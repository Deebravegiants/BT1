### Title
Unsigned Cross-Binding Between `ChunkEndorsementInnerV1` and `ChunkEndorsementMetadata` Allows Endorsement Suppression via Forged P2P Messages - (File: `core/primitives/src/stateless_validation/chunk_endorsement.rs`)

### Summary

`ChunkEndorsementV2` carries two independently-signed payloads — `inner` (containing only `chunk_hash`) and `metadata` (containing `account_id`, `shard_id`, `epoch_id`, `height_created`) — with no cryptographic cross-binding between them. An unprivileged network peer who has observed two valid endorsements from the same validator can splice the `inner`+`signature` from one endorsement with the `metadata`+`metadata_signature` from another, producing a forged endorsement that passes all validation checks but carries an inconsistent `chunk_hash`. This forged message occupies the per-validator cache slot for the target chunk, causing the legitimate endorsement to be silently dropped when it arrives, suppressing that validator's vote and potentially preventing the chunk from reaching the endorsement threshold.

### Finding Description

`ChunkEndorsementV2` is defined with two separate signed payloads:

```
inner             = { chunk_hash, signature_differentiator }
signature         = Sign(inner, validator.key)

metadata          = { account_id, shard_id, epoch_id, height_created }
metadata_signature = Sign(metadata, validator.key)
``` [1](#0-0) 

The `verify()` method checks both signatures against the same public key, but does so **independently** — neither payload commits to any field of the other: [2](#0-1) 

`ChunkEndorsementInnerV1` contains only `chunk_hash` and a static differentiator string; it does not include `shard_id`, `epoch_id`, `height_created`, or `account_id`: [3](#0-2) 

`ChunkEndorsementMetadata` contains routing/validation fields but does not include `chunk_hash`: [4](#0-3) 

Because the two payloads are disjoint, a forged endorsement constructed by combining `inner`+`signature` from validator V's endorsement of chunk C1 with `metadata`+`metadata_signature` from V's endorsement of chunk C2 will satisfy `verify()` for V's public key, since both component signatures are genuine.

The validation path in `validate_chunk_endorsement` calls `validate_chunk_relevant_as_validator` (which uses only the metadata fields) and then `validate_chunk_endorsement_signature` (which calls `endorsement.verify()`). Neither step checks that `inner.chunk_hash` is consistent with `metadata.shard_id`/`height_created`: [5](#0-4) 

Once accepted, the forged endorsement is stored in the `ChunkEndorsementTracker` cache under the key derived from the metadata: [6](#0-5) 

The cache uses a per-validator deduplication check: if an entry already exists for `(ChunkProductionKey, account_id)`, the incoming endorsement is silently dropped: [7](#0-6) 

When the block producer later collects endorsements for the real chunk C2, the forged entry is filtered out because its `chunk_hash` (from C1) does not match C2's actual hash: [8](#0-7) 

The legitimate endorsement from V for C2 was already dropped. V's vote is gone.

### Impact Explanation

An attacker who can inject P2P `ChunkEndorsement` messages can suppress the endorsement votes of targeted validators for specific chunks. If enough validators are targeted, the chunk fails to reach the endorsement threshold, causing it to be excluded from the block. The corrupted protocol value is the **chunk endorsement state** (`ChunkEndorsementsState::is_endorsed`) used by the block producer to decide whether to include a chunk. This is a targeted liveness attack: specific chunks on specific shards can be prevented from being endorsed, stalling block production for those shards.

### Likelihood Explanation

Endorsements are broadcast over the P2P network. Any peer can observe them and craft a forged splice. The attack requires the same validator to be assigned to at least two shards (or two heights) within an observable window — a normal condition in a multi-shard network. The forged message must arrive before the legitimate one, which is achievable by a peer with low-latency connectivity to the block producer. No privileged access is required.

### Recommendation

Bind `inner` and `metadata` cryptographically. The simplest fix is to include the metadata fields (`shard_id`, `epoch_id`, `height_created`, `account_id`) inside `ChunkEndorsementInnerV1` so that a single signature covers the complete endorsement context, eliminating the possibility of cross-splicing. Alternatively, include `inner.chunk_hash` inside `ChunkEndorsementMetadata` so that the metadata signature also commits to the chunk being endorsed. Either approach ensures that a valid endorsement cannot be constructed by combining components from two different endorsements.

### Proof of Concept

1. Observe validator V's `ChunkEndorsementV2` for chunk C1 (shard S1, height H1, epoch E1): extract `(inner_C1, sig_C1, meta_C1, meta_sig_C1)`.
2. Observe validator V's `ChunkEndorsementV2` for chunk C2 (shard S2, height H2, epoch E2): extract `(inner_C2, sig_C2, meta_C2, meta_sig_C2)`.
3. Construct forged endorsement: `ChunkEndorsementV2 { inner: inner_C1, signature: sig_C1, metadata: meta_C2, metadata_signature: meta_sig_C2 }`.
4. `verify(V.public_key())` returns `true`: `sig_C1.verify(inner_C1, V.pk)` ✓ and `meta_sig_C2.verify(meta_C2, V.pk)` ✓.
5. Send forged endorsement to the block producer before V's legitimate C2 endorsement arrives.
6. `validate_chunk_endorsement` accepts it; it is cached under key `{S2, E2, H2}` for account V with `chunk_hash = H(C1)`.
7. V's legitimate C2 endorsement arrives; the deduplication check fires and it is dropped.
8. `collect_chunk_endorsements` for C2 filters out the forged entry (`H(C1) ≠ H(C2)`); V's vote is absent.
9. If sufficient validators are targeted, `endorsement_state.is_endorsed` is `false` and the chunk is not included in the block. [9](#0-8) [10](#0-9)

pub struct ChunkEndorsementV2 {
    // This is the part of the chunk endorsement that signed and included in the block header
    inner: ChunkEndorsementInnerV1,
    // This is the signature of the inner field, to be included in the block header
    signature: Signature,
    // This consists of the metadata for chunk endorsement used in validation
    metadata: ChunkEndorsementMetadata,
    // Metadata signature is used to validate that the metadata is produced by the expected validator
    metadata_signature: Signature,
}

impl ChunkEndorsementV2 {
    fn verify(&self, public_key: &PublicKey) -> bool {
        let inner = borsh::to_vec(&self.inner).unwrap();
        let metadata = borsh::to_vec(&self.metadata).unwrap();
        self.signature.verify(&inner, public_key)
            && self.metadata_signature.verify(&metadata, public_key)
    }
}

        // Check if we have already received chunk endorsement from this validator.
        let key = endorsement.chunk_production_key();
        let account_id = endorsement.account_id();

        {
            let cache = self.chunk_endorsements.lock();
            if cache.peek(&key).is_some_and(|entry| entry.contains_key(account_id)) {
                tracing::debug!(target: "client", ?endorsement, "already received chunk endorsement");
                return Ok(());
            }

        // Validate the chunk endorsement and store it in the cache.
