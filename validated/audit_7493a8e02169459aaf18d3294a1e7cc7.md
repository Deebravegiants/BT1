### Title
ETH Deposit Cap Bypass via Missing Amount in Limit Check - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric validation: for ERC-20 LSTs it correctly tests `totalAssetDeposits + amount > depositLimit`, but for ETH it omits the incoming `amount` and only tests `totalAssetDeposits > depositLimit`. Any unprivileged depositor can therefore submit an ETH deposit of arbitrary size whenever the current total is at or below the limit, bypassing the protocol's deposit cap entirely.

### Finding Description
The function at lines 676-682 of `contracts/LRTDepositPool.sol` reads:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount not included
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

The ETH branch answers the question *"is the current total already over the limit?"* rather than *"would this deposit push the total over the limit?"* Consequently, while `totalAssetDeposits ≤ depositLimit`, the check always returns `false` regardless of how large `amount` is, and `_beforeDeposit` never reverts with `MaximumDepositLimitReached`. A single `depositETH` call can push `getTotalAssetDeposits(ETH)` from just below the cap to an arbitrarily large value. Only after that first over-limit deposit does the check begin to block further deposits.

The parallel LST path correctly includes `amount`, making this an ETH-specific wrong-reference-value defect — the direct structural analog of the external report's wrong-threshold bug.

### Impact Explanation
**Low — Contract fails to deliver promised returns.**

The deposit limit is the protocol's primary on-chain safety valve for ETH exposure. Bypassing it allows:
- Minting rsETH beyond the intended cap in a single transaction (subject only to the separate `RSETH.checkDailyMintLimit` guard, if configured).
- More ETH than intended flowing into NodeDelegators and EigenLayer strategies, violating the risk parameters set by the admin.
- Downstream accounting inconsistencies: `getAssetCurrentLimit` will report 0 remaining capacity after the over-deposit, but the damage is already done.

Because the deposited ETH is real collateral backing the minted rsETH, there is no direct fund theft. The harm is that the protocol operates outside its configured risk envelope, which can escalate to insolvency under EigenLayer slashing if the cap was sized to bound that exposure.

### Likelihood Explanation
**High.** The entry point is the public, permissionless `depositETH` function. No special role, flash loan, or price manipulation is required. Any ETH holder can call it whenever `getTotalAssetDeposits(ETH_TOKEN) ≤ depositLimitByAsset(ETH_TOKEN)`, which is the normal operating state of the protocol.

### Recommendation
Add `amount` to the ETH branch, matching the LST branch:

```diff
    if (asset == LRTConstants.ETH_TOKEN) {
-       return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
+       return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
    }
```

### Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 100 ether`.
2. Protocol currently holds `totalAssetDeposits(ETH) = 99 ether` (just below cap).
3. Attacker calls `depositETH{value: 1000 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH, 1000e18)` evaluates `99e18 > 100e18` → `false` → no revert.
5. 1000 ETH is accepted; `getTotalAssetDeposits(ETH)` becomes 1099 ETH — 10× the intended cap.
6. Subsequent calls revert because `1099e18 > 100e18` is now `true`, but the cap has already been violated by 999 ETH. [1](#0-0) [2](#0-1) [3](#0-2)
