### Title
Missing `token::authority` Constraint on `from` Account Allows Delegate to Drain Victim Token Accounts via `init_transfer` - (File: `solana/programs/bridge_token_factory/src/instructions/user/init_transfer.rs`)

### Summary
The Solana `InitTransfer` instruction accepts a caller-supplied `from` token account with no constraint binding it to the `user` signer as its **owner**. The SPL token `transfer_checked` CPI accepts `user` as authority if `user` is either the owner *or* an approved delegate of `from`. Any attacker who holds a delegate approval on a victim's token account can pass the victim's account as `from`, drain it into the bridge vault, and route the bridged output to an attacker-controlled NEAR address.

### Finding Description
In `InitTransfer`, the `from` account is declared with only mint and token-program constraints:

```rust
#[account(
    mut,
    token::mint = mint,
    token::token_program = token_program,
)]
pub from: Box<InterfaceAccount<'info, TokenAccount>>,
``` [1](#0-0) 

There is no `token::authority = user` constraint, which in Anchor enforces `from.owner == user.key()`. The `process` function then issues:

```rust
transfer_checked(
    CpiContext::new(
        self.token_program.to_account_info(),
        TransferChecked {
            from: self.from.to_account_info(),
            to: vault.to_account_info(),
            authority: self.user.to_account_info(),
            mint: self.mint.to_account_info(),
        },
    ),
    payload.amount...,
    self.mint.decimals,
)?;
``` [2](#0-1) 

The SPL Token program's `transfer_checked` accepts `authority` as valid if it is either the **owner** of `from` *or* an **approved delegate** with sufficient delegated amount. Because the Anchor account struct does not constrain `from.owner == user`, an attacker who holds a delegate approval on any victim token account can supply the victim's account as `from` and their own key as `user` (signer), causing the SPL token program to accept the transfer.

The outgoing Wormhole message records `self.user.key()` as the sender:

```rust
self.common.post_message(payload.serialize_for_near((
    self.common.sequence.sequence,
    self.user.key(),   // attacker's key recorded as sender
    self.mint.key(),
))?)?;
``` [3](#0-2) 

The `recipient` field in `InitTransferPayload` is a free-form string supplied by the caller with no binding to `from`: [4](#0-3) 

So the attacker fully controls both the source of funds (`from` = victim's account) and the destination (`recipient` = attacker's NEAR address).

### Impact Explanation
**Critical.** An attacker drains a victim's SPL token balance and receives the equivalent bridged tokens on NEAR (or another destination chain). The victim's tokens are either locked in the bridge vault (native token path) or burned (bridged token path) — both are irreversible. This constitutes unauthorized withdrawal of user assets through the bridge settlement path.

### Likelihood Explanation
**Medium.** The attacker must hold a delegate approval on the victim's token account. In practice, users grant delegate approvals to DeFi protocols, aggregators, and wallets routinely. Any such approval where the approved address is attacker-controlled (e.g., a compromised protocol, a malicious contract, or a social-engineering scenario) is sufficient. The attack requires no privileged role and is executable by any unprivileged Solana account.

### Recommendation
Add `token::authority = user` to the `from` account constraint in `InitTransfer`:

```rust
#[account(
    mut,
    token::mint = mint,
    token::authority = user,   // <-- add this
    token::token_program = token_program,
)]
pub from: Box<InterfaceAccount<'info, TokenAccount>>,
```

This Anchor constraint checks `from.owner == user.key()` at account validation time, ensuring only the actual owner of the token account can initiate a transfer, regardless of any delegate approvals.

### Proof of Concept
1. Victim `V` holds a native SPL token account `V_ATA` and has previously approved attacker `A` as a delegate (e.g., via interaction with another protocol).
2. Attacker `A` constructs an `init_transfer` instruction with:
   - `from = V_ATA` (victim's token account)
   - `user = A` (attacker, signs the transaction)
   - `payload.recipient = "near:attacker.near"` (attacker's NEAR address)
   - `payload.amount = V_ATA.delegated_amount`
3. Anchor validates `from`: only checks `token::mint = mint` and `token::token_program = token_program` — both pass.
4. `process()` calls `transfer_checked` with `authority = A`. The SPL token program checks: `V_ATA.delegate == Some(A)` and `V_ATA.delegated_amount >= amount` — both true. Transfer succeeds.
5. `V_ATA` is drained into the bridge vault. A Wormhole message is emitted recording `A` as sender and `"near:attacker.near"` as recipient.
6. The NEAR bridge finalizes the transfer, minting/releasing tokens to `attacker.near`.
