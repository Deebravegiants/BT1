### Title
Strict Equality Check on Time-Varying Treasury Value Enables Governance Proposal Griefing — (`eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs`)

---

### Summary

`validateTreasuryValue` enforces a strict equality check between the `currentTreasuryValue` field embedded in a Conway `TxBody` and the actual treasury balance at transaction-processing time. Because the treasury balance is time-varying (it grows at every epoch boundary via treasury expansion and can be incremented by treasury-donation transactions in any block), a proposer who must include `currentTreasuryValue` — as required by any guardrails script that reads `txInfoCurrentTreasuryAmount` — cannot reliably predict the exact treasury value that will be in effect when their transaction is processed. Any mismatch causes an unconditional `ConwayTreasuryValueMismatch` failure. An attacker can exploit this by submitting a treasury-donation transaction to shift the treasury value between the moment the proposer builds their transaction and the moment it is included in a block, permanently preventing the proposal from landing.

---

### Finding Description

`validateTreasuryValue` is called unconditionally for every valid Conway transaction:

```haskell
runTest $ validateTreasuryValue txBody (chainAccountState ^. casTreasuryL)
```

Its implementation is a strict equality gate:

```haskell
validateTreasuryValue txBody actualTreasuryValue =
  case txBody ^. currentTreasuryValueTxBodyL of
    SNothing -> pure ()
    SJust submittedTreasuryValue ->
      failureUnless (submittedTreasuryValue == actualTreasuryValue) $
        ConwayTreasuryValueMismatch
          ( Mismatch
              { mismatchSupplied = submittedTreasuryValue
              , mismatchExpected = actualTreasuryValue
              }
          )
```

The field is optional (`SNothing` skips the check), but it is **mandatory** for any governance proposal whose guardrails script reads `txInfoCurrentTreasuryAmount`. The Plutus script context is populated directly from the tx-body field:

```haskell
txInfoCurrentTreasuryAmount = strictMaybeToMaybe $ txBody ^. currentTreasuryValueTxBodyL
```

If the field is absent, `txInfoCurrentTreasuryAmount` is `Nothing`, and any guardrails script that pattern-matches on `Just treasury` (e.g. `ensureTreasuryReserve`) immediately returns `False`, failing the transaction. Therefore, proposers using such scripts **must** supply `currentTreasuryValue`.

The treasury balance (`chainAccountState ^. casTreasuryL`) changes:
1. **At every epoch boundary** — treasury expansion adds a fraction of reserves to the treasury.
2. **Between blocks** — any transaction that includes a non-zero `treasuryDonation` field increments the treasury; the updated `ChainAccountState` is visible to subsequent blocks.

Because the proposer cannot know which exact treasury value will be in effect when their transaction is processed, the strict equality check is impossible to satisfy reliably. An attacker who observes a pending proposal in the mempool can submit a 1-lovelace treasury donation in the preceding block, shifting the treasury by exactly that amount and causing the proposal to fail with `ConwayTreasuryValueMismatch`. The proposer must rebuild and resubmit; the attacker can repeat indefinitely at the cost of only transaction fees.

---

### Impact Explanation

A governance proposal that relies on a guardrails script checking `txInfoCurrentTreasuryAmount` can be permanently blocked from reaching the ledger. The attacker does not need to control a majority of stake or any privileged role — any ADA holder can submit a treasury donation. This falls under **Medium** impact: attacker-controlled transactions modify treasury donations outside design parameters (the donation feature is not intended to be a governance-censorship primitive), preventing legitimate governance proposals from being enacted.

---

### Likelihood Explanation

The attack is cheap (one small donation per attempt, cost = tx fee + donation amount that goes to the treasury), requires no special access, and is repeatable. Any proposer using a guardrails script that inspects the treasury amount is vulnerable. The window also opens naturally at every epoch boundary without any attacker involvement, meaning even non-adversarial conditions can cause repeated failures near epoch transitions.

---

### Recommendation

Replace the strict equality check with a "snapshot" approach: instead of requiring the proposer to guess the exact treasury value, record the actual treasury value at the time the transaction is processed and expose that value to the Plutus script context directly from the ledger state, without requiring the proposer to supply it in the tx body. If the field must remain in the tx body for other reasons (e.g. fee calculation), accept any value that is `<= actualTreasuryValue` rather than requiring exact equality, analogous to how the Solidity report's mitigation caps `deltaDebt` at `maxRepayment` instead of reverting.

---

### Proof of Concept

1. Proposer queries treasury: `T = 1_000_000_000`.
2. Proposer builds a `TreasuryWithdrawals` governance proposal with a guardrails script that checks `txInfoCurrentTreasuryAmount >= withdrawalAmount`. Sets `currentTreasuryValue = T` in the tx body. Signs and submits to mempool.
3. Attacker observes the pending proposal. Submits a transaction with `treasuryDonation = 1` (1 lovelace) in the next block.
4. Treasury becomes `T + 1 = 1_000_000_001`.
5. Proposer's transaction is processed: `validateTreasuryValue` compares `1_000_000_000 == 1_000_000_001` → `False` → `ConwayTreasuryValueMismatch`. Transaction rejected.
6. Proposer rebuilds with `currentTreasuryValue = 1_000_000_001`. Attacker donates 1 lovelace again. Repeat.
