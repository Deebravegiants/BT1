Audit Report

## Title
Unguarded `sendFunds()` Allows Anyone to Force Early MEV Reward Distribution - (File: `contracts/FeeReceiver.sol`)

## Summary
`FeeReceiver.sendFunds()` carries no access control, allowing any external caller to transfer all accumulated MEV/execution-layer ETH from `FeeReceiver` into `LRTDepositPool` at an arbitrary time. Because `LRTOracle.rsETHPrice` is a stored value that is updated by the public `updateRSETHPrice()`, an attacker can sequence `sendFunds()` → `updateRSETHPrice()` → `instantWithdrawal()` in a single block to cash out at a price that includes rewards the protocol had not yet chosen to distribute, extracting their proportional share of pending rewards ahead of schedule.

## Finding Description
`FeeReceiver.sendFunds()` is `external` with no modifier:

```solidity
// contracts/FeeReceiver.sol L53-58
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

Every other state-changing function in the contract (`setDepositPool`) is gated by `onlyRole(LRTConstants.MANAGER)`. [1](#0-0) 

`LRTDepositPool.receiveFromRewardReceiver()` is equally unguarded: [2](#0-1) 

The ETH sitting in `FeeReceiver` is intentionally **excluded** from TVL until `sendFunds()` is called. The comment in `getETHDistributionData()` confirms this design: [3](#0-2) 

`LRTOracle.rsETHPrice` is a **stored** value, not computed on-the-fly. It is updated only when `updateRSETHPrice()` is called, which is `public whenNotPaused` — callable by anyone: [4](#0-3) 

`LRTWithdrawalManager.instantWithdrawal()` computes the payout using the **current stored** `rsETHPrice` at execution time: [5](#0-4) 

The exploit chain is therefore: `sendFunds()` inflates `getTotalAssetDeposits(ETH_TOKEN)` → `updateRSETHPrice()` recalculates and stores the higher price → `instantWithdrawal()` pays out at the inflated price.

A partial mitigation exists: `updateRSETHPrice()` enforces a `pricePercentageLimit` check that reverts for non-managers if the price jump exceeds the configured threshold. [6](#0-5)  However, this guard is ineffective when (a) `pricePercentageLimit == 0` (unset/disabled), or (b) the accumulated reward is small enough to stay within the allowed band. Both are realistic operating conditions.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but does not lose value.**

The attacker extracts exactly their proportional share of the pending MEV rewards early; no funds are stolen from other users (remaining rsETH holders retain rsETH at the now-higher price). The harm is that the protocol's intended time-averaged reward distribution schedule is bypassed: rewards that should accrue to all holders over time can be selectively captured by a withdrawing attacker at a moment of their choosing. This matches the allowed Low impact: "Contract fails to deliver promised returns, but doesn't lose value."

## Likelihood Explanation
The function is `external` with zero preconditions and costs only gas. The preconditions for the full exploit are: (1) `FeeReceiver` holds a non-trivial ETH balance, (2) `isInstantWithdrawalEnabled[asset]` is `true`, (3) the accumulated reward is within `pricePercentageLimit` (or the limit is unset). Conditions (1) and (3) are routinely satisfied during normal protocol operation; condition (2) is a protocol configuration choice. The attack is repeatable every time rewards accumulate.

## Recommendation
Add the same role guard used by every other privileged function in the contract:

```solidity
function sendFunds() external onlyRole(LRTConstants.MANAGER) {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

Additionally, consider adding a caller whitelist or `onlyFeeReceiver` guard to `LRTDepositPool.receiveFromRewardReceiver()` so that the ETH accounting path cannot be triggered by an arbitrary address sending ETH directly.

## Proof of Concept
Minimal call sequence (single block, no privileged access required):

1. Confirm `FeeReceiver` holds `R` ETH in MEV rewards (balance > 0).
2. Confirm `isInstantWithdrawalEnabled[ETH_TOKEN] == true` and the unstaking vault holds sufficient ETH.
3. Attacker holds `A` rsETH.
4. **Tx 1:** `FeeReceiver.sendFunds()` — no role check; `R` ETH moves to `LRTDepositPool`, increasing `getTotalAssetDeposits(ETH_TOKEN)` by `R`.
5. **Tx 2 (same block):** `LRTOracle.updateRSETHPrice()` — public; recalculates and stores the higher rsETH price that now includes `R`.
   - If `pricePercentageLimit > 0` and the jump exceeds it, this step reverts; exploit is blocked for large reward batches. For small rewards or when the limit is unset, it succeeds.
6. **Tx 3 (same block):** `LRTWithdrawalManager.instantWithdrawal(ETH_TOKEN, A, "")` — `getExpectedAssetAmount` uses the now-inflated stored `rsETHPrice`; attacker receives `A * newRsETHPrice / assetPrice` ETH instead of `A * oldRsETHPrice / assetPrice` ETH.
7. Net gain: attacker captures `A * R / totalRsETHSupply` ETH of rewards that would otherwise have been distributed proportionally to all holders on the protocol's normal schedule.

Foundry fork test plan: fork mainnet, impersonate a whale rsETH holder, execute steps 4–6 in sequence, assert `assetAmountUnlocked` in step 6 exceeds the value computed with the pre-`sendFunds` price.
