### Title
Unimplemented Double Deposit Refund Guard Across Multiple Sub-Transactions in Dijkstra Era - (File: `eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs`)

---

### Summary

The Dijkstra era introduces nested (sub-)transactions. A top-level transaction can embed multiple sub-transactions that are processed sequentially, each sharing the same evolving `CertState`. The protection that prevents multiple sub-transactions from each claiming the deposit refund for the same stake credential is explicitly acknowledged as **not yet implemented** in the codebase. An attacker can craft a top-level transaction whose sub-transactions each carry `UnRegDepositTxCert` for the same credential, potentially claiming the same deposit refund more than once and draining ADA from the deposit pot.

---

### Finding Description

The Dijkstra era's `SUBLEDGERS` → `SUBLEDGER` pipeline processes sub-transactions sequentially, threading `CertState` from one sub-transaction to the next. Each sub-transaction's `SUBENTITIES` rule (in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubEntities.hs`) processes certificates including `UnRegDepositTxCert`. The `SUBUTXO` rule (via `updateUTxOStateNoFees`) computes the deposit refund using the `certState` present in the `UtxoEnv` at the time of processing — which, following the same pattern as the Conway `LEDGER` rule, is the **pre-certificate** state for the current sub-transaction.

The critical gap: after sub-transaction 1 (`subTx1`) successfully unregisters `stakingCred` and claims `keyDeposit` as a refund, sub-transaction 2 (`subTx2`) is processed. The `certState` passed to `SUBUTXOW`/`SUBUTXO` for `subTx2` is the state **after** `subTx1` fully completed. If this state still reflects `stakingCred` as registered (e.g., because the `SUBLEDGER` rule passes the original pre-sub-transaction `certState` to `SUBUTXOW` for all sub-transactions, mirroring the Conway `LEDGER` pattern of passing the unmodified `certState` to `UTXOW`), then `certsTotalRefundsTxBody` for `subTx2` would also return `keyDeposit`, and `utxosDeposited` would be decremented a second time — a double refund.
