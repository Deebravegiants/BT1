### Title
Missing On-Chain Output Verification for Legacy `AppPublicKey` CKD Requests Allows Byzantine Leader to Corrupt Confidential Key Derivation — (`File: crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract verifies the CKD output cryptographically only when the request uses the `AppPublicKeyPV` (publicly verifiable) format. When the legacy `AppPublicKey` (privately verifiable) format is used, the contract performs **no on-chain verification** of the submitted response. A single Byzantine leader node — a participant strictly below the signing threshold — can therefore submit an arbitrary, incorrect CKD response that the contract will unconditionally accept and deliver to the requesting user.

---

### Finding Description

In `crates/contract/src/lib.rs`, the `respond_ckd` function handles the leader node's submission of a completed Confidential Key Derivation result:

```rust
// lib.rs lines 675–682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no verification whatsoever
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For `AppPublicKeyPV` requests, `ckd_output_check` verifies that the submitted `(big_y, big_c)` pair is a valid encryption of the derived key under the master BLS public key and the user-supplied `(pk1, pk2)` pair. For `AppPublicKey` requests, the arm is an empty no-op — the contract unconditionally proceeds to `resolve_yields_for`, delivering whatever `CKDResponse` the leader submitted.

The `AppPublicKey` format is explicitly described as "privately verifiable (legacy)" in the contract README — only the holder of the app's secret key can verify the result off-chain. The contract has no cryptographic means to check it. This is the direct analog of the OETH rebasing design flaw: a required opt-in (on-chain output verification) is absent by design for the legacy path, and the protocol silently accepts unverified output.

**Attack path:**

1. A user submits `request_app_private_key` with `AppPublicKey` format (still fully supported and documented).
2. The threshold CKD protocol runs; all honest participants contribute their BLS shares to the leader.
3. The malicious leader collects the shares, computes the correct output, but calls `respond_ckd` with a fabricated `CKDResponse` — e.g., random `big_y` and `big_c` values.
4. `respond_ckd` passes the `AppPublicKey` branch with no check, calls `resolve_yields_for`, and the fabricated response is delivered to the user's yield-resume callback.
5. The user receives a ciphertext that decrypts to garbage; the CKD request lifecycle completes with a corrupted result and no on-chain error.

The leader role is not fixed: any attested participant can be elected leader for a given request. A single Byzantine participant strictly below the signing threshold is sufficient.

---

### Impact Explanation

The production safety invariant broken is: *every completed CKD request must deliver the correct derived key encrypted under the user's app public key*. For `AppPublicKey` requests, a single malicious leader can violate this invariant undetected at the contract level. The user's application receives a ciphertext it cannot decrypt to a valid key, silently breaking the confidential key derivation flow. Because the contract resolves the yield with the fabricated response, the request is marked complete and cannot be retried under the same request key without a new submission. This maps to **Medium** under the allowed scope: *request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration*.

---

### Likelihood Explanation

The `AppPublicKey` format is still fully supported (documented in the contract README as the legacy path and accepted by `request_app_private_key`). Any attested participant can be elected leader. A single Byzantine participant below the signing threshold — the exact adversary model called out in the scope — can execute this attack on any `AppPublicKey` CKD request they are elected to lead, with no external preconditions.

---

### Recommendation

For `AppPublicKey` requests, add a verifiable binding between the submitted response and the master BLS public key and the user's app public key. The simplest mitigation is to require callers to migrate to `AppPublicKeyPV`, which already has `ckd_output_check` enforced. If the legacy format must be retained, document explicitly that its outputs are unverified on-chain and that users must verify off-chain before trusting the result — and consider emitting a contract-level warning or deprecation log on the `AppPublicKey` branch.

```
1. Deploy contract with a BLS CKD domain (Protocol::ConfidentialKeyDerivation).
2. Submit request_app_private_key with AppPublicKey = <any valid G1 point>.
3. As the elected leader node (any attested participant), call respond_ckd with:
     request  = the pending CKDRequest
     response = CKDResponse { big_y: <random G1>, big_c: <random G1> }
4. Observe: respond_ckd returns Ok(()), the yield resolves, and the user's
   callback receives the fabricated CKDResponse with no contract-level error.
5. Contrast: repeat with AppPublicKeyPV — respond_ckd panics with
   "CKD output check failed" for the same fabricated response.
```
