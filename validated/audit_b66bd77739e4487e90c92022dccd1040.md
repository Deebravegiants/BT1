### Title
`LRTWithdrawalManager._withdrawFromAave` silently returns zero when `totalETHDepositedToAave` is exhausted but residual Aave interest remains, causing ETH withdrawal DoS — (`contracts/LRTWithdrawalManager.sol`)

---

### Summary

When the Aave integration is enabled and all tracked principal (`totalETHDepositedToAave`) has been consumed by prior withdrawals while residual interest still sits in Aave, `_withdrawFromAave` silently returns `0` without pulling any ETH. Any subsequent call to `completeWithdrawal` for ETH then reverts with `InsufficientLiquidityForWithdrawal`, temporarily freezing pending user ETH withdrawals.

---

### Finding Description

`_withdrawFromAave` caps the amount it will pull to `withdrawablePrincipal = min(aaveBalance, totalETHDepositedToAave)`: [1](#0-0) 

When `totalETHDepositedToAave == 0`, `withdrawablePrincipal` collapses to `0` regardless of the actual `aaveBalance`, and the function returns `0` silently: [2](#0-1) 

This state is reachable in normal operation. `_processWithdrawalCompletion` calls `_withdrawFromAave(amountNeeded)` without first collecting interest: [3](#0-2) 

When a user's withdrawal exactly drains the remaining principal (i.e., `amountNeeded == totalETHDepositedToAave`), the accounting sets `totalETHDepositedToAave = 0` while Aave still holds accrued interest. Every subsequent ETH withdrawal that needs Aave liquidity will then:

1. Call `_withdrawFromAave(amountNeeded)` — returns `0` because `totalETHDepositedToAave == 0`.
2. Re-check `balanceAfter < request.expectedAssetAmount` — still true.
3. Revert with `InsufficientLiquidityForWithdrawal`. [4](#0-3) 

The interest sitting in Aave is inaccessible to users because `_withdrawFromAave` intentionally excludes it, and `_collectInterestToTreasury` routes it to the treasury rather than back to the contract: [5](#0-4) 

---

### Impact Explanation

**Temporary freezing of funds (Medium).** Users who have already burned their rsETH via `initiateWithdrawal` cannot complete their ETH withdrawals. Their position is stuck until an external ETH inflow (e.g., from the unstaking vault) replenishes the contract balance. Disabling the Aave integration does not help because `_collectInterestToTreasury` routes the residual interest to the treasury, not to the withdrawal contract.

---

### Likelihood Explanation

**Medium.** The trigger condition — `totalETHDepositedToAave` reaching exactly zero while `aaveBalance > 0` — arises naturally whenever a withdrawal drains the last of the tracked principal while interest has accrued. This is a routine operational state, not an exotic edge case. Any user whose withdrawal request is processed after this point and whose amount exceeds the contract's idle ETH balance will be blocked.

---

### Recommendation

In `_withdrawFromAave`, when `totalETHDepositedToAave == 0` but `aaveBalance > 0`, allow the full `aaveBalance` to be used (treating it as reclaimable liquidity). Alternatively, call `_collectInterestToTreasury()` inside `_processWithdrawalCompletion` before calling `_withdrawFromAave`, so that interest is swept to the treasury first and `aaveBalance` is brought back in line with `totalETHDepositedToAave` before the principal withdrawal is attempted.

```solidity
// Suggested fix in _withdrawFromAave:
uint256 withdrawablePrincipal = totalETHDepositedToAave == 0
    ? aaveBalance                                          // use residual interest if no principal remains
    : (aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave);
```

---

### Proof of Concept

1. Aave integration enabled; `totalETHDepositedToAave = 100 ETH`, `aaveBalance = 101 ETH` (1 ETH accrued interest).
2. User A calls `completeWithdrawal(ETH)` for `100 ETH`; contract idle balance is `0`.
3. `_withdrawFromAave(100)` → `withdrawablePrincipal = min(101, 100) = 100`; withdraws `100 ETH`; `totalETHDepositedToAave = 0`, `aaveBalance = 1 ETH`.
4. User A receives `100 ETH`; contract idle balance returns to `0`.
5. User B calls `completeWithdrawal(ETH)` for any amount > `0`; contract idle balance is `0`.
6. `_withdrawFromAave(amountNeeded)` → `withdrawablePrincipal = min(1, 0) = 0`; `withdrawnAmount = 0`; returns `0`.
7. `balanceAfter = 0 < request.expectedAssetAmount` → `revert InsufficientLiquidityForWithdrawal()`.
8. User B's withdrawal is blocked indefinitely until external ETH is deposited. [6](#0-5) [7](#0-6)

    function _withdrawFromAave(uint256 amount) internal returns (uint256 withdrawnAmount) {
        if (amount == 0) return 0;

        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        if (aaveBalance == 0) revert InsufficientAaveBalance();

        // Only withdraw up to the principal amount (don't use accrued interest for user withdrawals)
        uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave;

        withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
        if (withdrawnAmount == 0) return 0;

        aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
        totalETHDepositedToAave -= withdrawnAmount;

        emit ETHWithdrawnFromAave(withdrawnAmount, totalETHDepositedToAave);
    }

    function _collectInterestToTreasury() internal returns (uint256 interestAmount) {
        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        uint256 principal = totalETHDepositedToAave;

        // Return 0 if no interest or balance is less than principal (accounting for rounding)
        if (aaveBalance <= principal) return 0;

        interestAmount = aaveBalance - principal;

        aaveWETHGateway.withdrawETH(aavePool, interestAmount, address(this));

        address treasury = lrtConfig.getContract(LRTConstants.PROTOCOL_TREASURY);
        (bool sent,) = payable(treasury).call{ value: interestAmount }("");
