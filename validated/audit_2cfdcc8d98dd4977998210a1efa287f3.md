Audit Report

## Title
Stale `rsETHPrice` Sandwich via Public `updateRSETHPrice()` and `instantWithdrawal` ŌĆö (File: `contracts/LRTOracle.sol`)

## Summary
`LRTOracle.updateRSETHPrice()` is an unrestricted public function that writes a freshly computed price to the stored `rsETHPrice` variable. Because both deposits and instant withdrawals read this cached value rather than computing a live price, an attacker can deposit at the stale (lower) price, call `updateRSETHPrice()` to snap the stored price to its true higher value, then immediately call `instantWithdrawal()` at the inflated price ŌĆö extracting accrued yield that belongs to existing rsETH holders. The impact is theft of unclaimed yield.

## Finding Description

**Root cause ŌĆö public price update with no access control:**

`updateRSETHPrice()` carries only a `whenNotPaused` modifier; any unprivileged EOA can call it at will. [1](#0-0) 

`_updateRsETHPrice()` writes the result into the storage slot `rsETHPrice`: [2](#0-1) 

**Deposit reads the stale cached price:**

`getRsETHAmountToMint` divides by `lrtOracle.rsETHPrice()` ŌĆö the stored value, not a freshly computed one. A deposit made while the price is stale mints more rsETH than the depositor's ETH is worth at the true price. [3](#0-2) 

**Instant withdrawal also reads the cached price:**

`getExpectedAssetAmount` multiplies by `lrtOracle.rsETHPrice()`. After the attacker has forced the price update, this returns the higher true price. [4](#0-3) 

**`instantWithdrawal` is gated only by `isInstantWithdrawalEnabled` and vault liquidity:** [5](#0-4) 

**Partial guard ŌĆö `pricePercentageLimit`:**

`_updateRsETHPrice()` contains a check that reverts for non-managers when the new price exceeds `highestRsethPrice` by more than `pricePercentageLimit`: [6](#0-5) 

This guard is **not a complete mitigation** for two reasons:
1. `pricePercentageLimit` defaults to `0` (no default is set in `initialize`); when it is zero the condition `pricePercentageLimit > 0` is false and the entire check is skipped, leaving the function fully open.
2. Even when a non-zero limit is configured (e.g. 1 %), the attack still works for any staleness that falls within that window ŌĆö the limit only caps the per-cycle profit, it does not eliminate the vulnerability.

**`previousTVL` accounting flaw amplifies the issue:**

Inside `_updateRsETHPrice()`, `previousTVL` is computed as `rsethSupply ├Ś rsETHPrice` using the **post-deposit** supply and the **stale** price: [7](#0-6) 

The attacker's freshly minted rsETH is silently folded into the "previous TVL" baseline, so the protocol fee is charged on the combined genuine reward plus the attacker's deposit delta rather than on the genuine reward alone.

**Exploit sequence:**
1. Observe that `rsETHPrice` is stale (true TVL > `rsethSupply ├Ś rsETHPrice`).
2. Call `depositETH(D)` ŌĆö mints `D / P_stale` rsETH (more than fair share).
3. Call `updateRSETHPrice()` ŌĆö stored price rises to `P_actual > P_stale`.
4. Call `instantWithdrawal(asset, D / P_stale)` ŌĆö receives `(D / P_stale) ├Ś P_actual > D` ETH.

No privileged role is required for any of these steps.

## Impact Explanation

The attacker extracts a net profit of approximately `D ├Ś (P_actual ŌłÆ P_stale) / P_stale` ETH per cycle, funded entirely by diluting the accrued staking yield of existing rsETH holders. This is a direct, concrete instance of **theft of unclaimed yield (High)**. With instant withdrawal enabled the entire cycle can be executed atomically or within a single block, making repeated extraction trivial. At scale, continuous draining of the yield pool approaches protocol insolvency.

## Likelihood Explanation

`rsETHPrice` is stale during every interval between keeper calls ŌĆö a normal operational state. The attacker controls exactly when the price snaps to its true value by calling the public `updateRSETHPrice()`. No privileged role is required. The only preconditions are: (a) `isInstantWithdrawalEnabled[asset] == true` and (b) `LRTUnstakingVault` holds sufficient liquidity ŌĆö both expected during normal operation. When `pricePercentageLimit == 0` (the default) there is no per-cycle cap on profit. Likelihood is **Medium** (requires instant withdrawal to be enabled), overall severity **High**.

## Recommendation

1. **Atomically refresh the price inside `depositETH`/`depositAsset`** before computing `rsethAmountToMint`, so every mint uses the freshest price.
2. Alternatively, **compute the mint amount directly from live TVL** (`totalETHInProtocol / rsethSupply`) rather than the cached `rsETHPrice`, eliminating the stale-read window entirely.
3. **Restrict `updateRSETHPrice()` to a keeper/manager role** so the attacker cannot control the exact moment the price snaps to its true value.
4. **Add a deposit-then-withdraw cooldown** (block-number lock on rsETH received from a deposit) to prevent same-block or same-transaction round-trips regardless of price freshness.
5. Ensure `pricePercentageLimit` is always set to a non-zero value as a defence-in-depth measure; document that leaving it at zero disables the only existing guard.

## Proof of Concept

```
Setup:
  TVL = 110 ETH, rsETH supply = 100, rsETHPrice (stored) = 1.00 ETH
  (10 ETH of staking rewards have accrued; price not yet updated)
  pricePercentageLimit = 0  (default ŌĆö guard disabled)
  isInstantWithdrawalEnabled[ETH] = true

Step 1 ŌĆö depositETH(10 ETH):
  rsethAmountToMint = 10e18 * 1e18 / 1e18 = 10 rsETH   ŌåÉ stale price
  TVL = 120 ETH, supply = 110

Step 2 ŌĆö updateRSETHPrice() (called by attacker):
  previousTVL = 110 * 1.00 = 110 ETH
  totalETHInProtocol = 120 ETH
  rewardAmount = 10 ETH ŌåÆ protocolFeeInETH = fee
  newRsETHPrice Ōēł (120 - fee) / 110 Ōēł 1.0909 ETH/rsETH  ŌåÉ stored

Step 3 ŌĆö instantWithdrawal(ETH, 10 rsETH):
  assetAmountUnlocked = 10 * 1.0909 / 1 Ōēł 10.909 ETH
  userAmount Ōēł 10.909 ETH (minus instantWithdrawalFee)

Attacker net: deposited 10 ETH, received Ōēł 10.909 ETH ŌåÆ profit Ōēł 0.909 ETH
Existing holders: each rsETH worth 1.0909 instead of 1.10 ETH ŌåÆ ~0.91 ETH stolen
```

**Foundry test plan:** Fork mainnet, set `pricePercentageLimit = 0`, enable instant withdrawal for ETH, seed the unstaking vault with ETH, simulate staking reward accrual by directly increasing the ETH balance tracked by `getTotalAssetDeposits`, then execute the three-step sequence above in a single test and assert `attacker.balance > initialDeposit`.
