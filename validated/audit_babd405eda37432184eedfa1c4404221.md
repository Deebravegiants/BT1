### Title
Hardcoded `paid_fee_on_l1 = Fee(1)` in Consensus L1 Handler Conversion Binds Wrong Executable Payload — (`crates/apollo_transaction_converter/src/transaction_converter.rs`)

### Summary

When a validator node converts a consensus-received `L1HandlerTransaction` into an executable transaction, it unconditionally substitutes `paid_fee_on_l1 = Fee(1)` instead of the actual fee paid on L1. The `ConsensusTransaction::L1Handler` wire format carries only the raw `L1HandlerTransaction` fields (nonce, contract address, entry-point selector, calldata) and strips `paid_fee_on_l1` entirely. The converter then reconstructs the executable transaction with a hardcoded sentinel, acknowledged in a TODO comment, rather than the real L1 fee. This is the direct sequencer analog of the Gelato relay bug: a forwarded message loses its authoritative origin-bound field, and the receiving side silently substitutes a wrong value.

### Finding Description

**Root cause — `convert_consensus_l1_handler_to_internal_l1_handler`:** [1](#0-0) 

```rust
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),
    )?)
}
```

This function is called on every `ConsensusTransaction::L1Handler` received from a peer proposer: [2](#0-1) 

The `ExecutableL1HandlerTransaction` struct carries `paid_fee_on_l1` as a first-class field: [3](#0-2) 

The blockifier's L1 handler execution path reads `self.paid_fee_on_l1` directly for its fee enforcement check: [4](#0-3) 

```rust
let paid_fee = self.paid_fee_on_l1;
// For now, assert only that any amount of fee was paid.
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(Box::new(
        TransactionFeeError::InsufficientFee { paid_fee, actual_fee: receipt.fee },
    )));
}
```

The proposer path, by contrast, obtains the real `paid_fee_on_l1` from the L1 events provider and passes it through `ExecutableL1HandlerTransaction::create` with the actual value. The Cende serialization path also records the real fee: [5](#0-4) 

### Impact Explanation

The wrong `paid_fee_on_l1` is bound into the `ExecutableL1HandlerTransaction` that the validator's blockifier executes. Under the **current** fee check (`paid_fee == Fee(0)` → reject), `Fee(1)` always passes, so there is no execution divergence between proposer and validator today. However:

1. **Wrong executable payload is committed**: The validator's executable transaction carries `paid_fee_on_l1 = Fee(1)` regardless of the actual L1 fee. This is a structural invariant violation: the executable payload does not faithfully represent the on-chain event.
2. **Fee enforcement is silently bypassed**: The comment "For now, assert only that any amount of fee was paid" is a deliberate weakening. Once the check is tightened to `paid_fee >= actual_fee` (the natural next step, also noted in the blockifier TODO), every validator node will reject all L1 handler transactions with `actual_fee > 1 wei`, while the proposer (using the real fee) accepts them. This produces a `ProposalFinMismatch` on every block containing an L1 handler transaction, halting consensus.
3. **Cende blob divergence**: The proposer's Cende blob records the real `paid_fee_on_l1`; any validator that becomes proposer would record `Fee(1)`, corrupting the central pre-confirmation record.

Impact category: **High — Transaction conversion binds the wrong executable payload** (`paid_fee_on_l1` field of `ExecutableL1HandlerTransaction`).

### Likelihood Explanation

Every L1 handler transaction that passes through the consensus validator path triggers this substitution. No special privilege is required; any L1→L2 message submitted to the Starknet core contract on L1 produces an `L1HandlerTransaction` that will be forwarded over consensus and converted with `Fee(1)`. The trigger is unconditional and unprivileged.

### Recommendation

Carry `paid_fee_on_l1` through the consensus wire format. The `ConsensusTransaction::L1Handler` variant (and its protobuf encoding) should include the `paid_fee_on_l1` field so that `convert_consensus_l1_handler_to_internal_l1_handler` can use the real value instead of the sentinel. Alternatively, the validator can look up the real fee from its local L1 events provider by `tx_hash` before constructing the executable transaction, mirroring how the proposer obtains it.

### Proof of Concept

1. A valid L1→L2 message is submitted on L1 with `paid_fee_on_l1 = X` (any `X > 0`).
2. The proposer's L1 events provider scrapes the event; the proposer builds a block including the L1 handler transaction with `paid_fee_on_l1 = X`.
3. The proposer broadcasts `ConsensusTransaction::L1Handler(tx.tx)` — the `paid_fee_on_l1` field is absent from the wire message.
4. The validator calls `convert_consensus_l1_handler_to_internal_l1_handler(tx)`, which calls `ExecutableL1HandlerTransaction::create(tx, &chain_id, Fee(1))`.
5. The validator's blockifier executes the transaction with `paid_fee_on_l1 = Fee(1)` instead of `X`.
6. Today the check `if paid_fee == Fee(0)` passes for `Fee(1)`, so no divergence occurs. Once the check becomes `if paid_fee < actual_fee`, any L1 handler transaction whose actual execution cost exceeds 1 wei will be rejected by the validator but accepted by the proposer, producing `ProposalFinMismatch` and stalling consensus.
