Audit Report

## Title
Stale Cached `rsETHPrice` Allows New Depositors to Mint Excess rsETH, Stealing Yield from Existing Holders - (File: contracts/LRTDepositPool.sol)

## Summary
`getRsETHAmountToMint` divides a live asset price by a cached `rsETHPrice` storage variable that is only updated via an explicit `updateRSETHPrice()` call. When rebasing LSTs (e.g., stETH) accrue yield between oracle updates, the stale denominator causes new depositors to receive more rsETH than their contribution warrants, permanently diluting existing holders and transferring their unclaimed yield to the attacker.

## Finding Description
`getRsETHAmountToMint` in `LRTDepositPool.sol` computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.getAssetPrice(asset)` calls through to a live external price feed (`IPriceFetcher`), while `lrtOracle.rsETHPrice()` reads a plain storage variable (`uint256 public override rsETHPrice` at `LRTOracle.sol:28`) that is only written inside `_updateRsETHPrice()`, which is only reachable via the separate public `updateRSETHPrice()` call (`LRTOracle.sol:87-89`).

Neither `depositAsset` (`LRTDepositPool.sol:99-118`) nor `depositETH` (`LRTDepositPool.sol:76-93`) calls `updateRSETHPrice()` before invoking `_beforeDeposit` → `getRsETHAmountToMint`. The deposit flow has no price-refresh step.

`getAssetDistributionData` reads `IERC20(asset).balanceOf(address(this))` directly (`LRTDepositPool.sol:444`), so `getTotalAssetDeposits` — and therefore `_getTotalEthInProtocol` inside `_updateRsETHPrice` — reflects the live, post-rebase stETH balance. However, this live TVL is only used when `updateRSETHPrice()` is explicitly called; it is never consulted at deposit time. The minting formula therefore uses a denominator that lags behind the true TVL after every stETH rebase.

Exploit path:
1. Protocol holds 100 stETH, 100 rsETH minted, `rsETHPrice = 1e18`.
2. stETH rebases: deposit pool now holds stETH worth 101 ETH. True rsETH price = `1.01e18`. `rsETHPrice` in storage is still `1e18`.
3. Attacker calls `depositAsset(stETH, 10e18, 0, "")`.
4. `getRsETHAmountToMint` returns `10e18 * 1e18 / 1e18 = 10e18` rsETH. Correct amount: `10e18 / 1.01e18 ≈ 9.9e18`.
5. Attacker receives ~0.099 excess rsETH at no cost.
6. `updateRSETHPrice()` is called: TVL = 111 ETH, supply = 110 rsETH, new price ≈ `1.009e18`. Original 100 rsETH holders now hold ~100.9 ETH worth instead of the 101 ETH they were entitled to — ~0.1 ETH of yield was transferred to the attacker.

Existing guards (`minRSETHAmountExpected`, deposit limits, `nonReentrant`, `whenNotPaused`) do not address the stale-price minting calculation.

## Impact Explanation
**High — Theft of unclaimed yield.** stETH rebase yield accrues to existing rsETH holders as an increase in the backing TVL per rsETH. When a new depositor mints rsETH against a stale (pre-rebase) `rsETHPrice`, they capture a share of that accrued yield without contributing it. The subsequent `updateRSETHPrice()` call permanently dilutes the original holders' share, with no recovery path. This maps directly to the allowed impact: *Theft of unclaimed yield*.

## Likelihood Explanation
stETH rebases approximately every 24 hours. `updateRSETHPrice()` is a separate permissionless call with no atomicity guarantee relative to deposits. Any unprivileged user can call `depositAsset` at any time; no front-running, governance access, or special role is required. The attacker only needs to deposit during the predictable window between a stETH rebase and the next `updateRSETHPrice()` invocation. This condition recurs daily and is trivially observable on-chain.

## Recommendation
At the start of `depositAsset` and `depositETH`, call `updateRSETHPrice()` (or an equivalent internal `_updateRsETHPrice()`) before computing `rsethAmountToMint`. This ensures the price used for minting always reflects the current TVL including any rebased yield. Alternatively, compute `rsETHPrice` on-the-fly within `getRsETHAmountToMint` by calling `_getTotalEthInProtocol().divWad(rsethSupply)` directly, bypassing the cached storage value entirely.

## Proof of Concept
Foundry fork test outline:

```solidity
// Fork mainnet at a block just after a stETH rebase
// 1. Confirm rsETHPrice is stale (pre-rebase value)
uint256 stalePriceBeforeUpdate = lrtOracle.rsETHPrice();

// 2. Confirm live TVL is higher than stalePriceBeforeUpdate * rsethSupply
uint256 liveTVL = lrtOracle._getTotalEthInProtocol(); // via exposed internal or view
uint256 truePrice = liveTVL * 1e18 / rsETH.totalSupply();
assertGt(truePrice, stalePriceBeforeUpdate);

// 3. Attacker deposits stETH without calling updateRSETHPrice first
uint256 attackerRsETHBefore = rsETH.balanceOf(attacker);
vm.prank(attacker);
lrtDepositPool.depositAsset(stETH, 10 ether, 0, "");
uint256 attackerRsETHMinted = rsETH.balanceOf(attacker) - attackerRsETHBefore;

// 4. Compute correct amount
uint256 correctMint = (10 ether * 1e18) / truePrice;
assertGt(attackerRsETHMinted, correctMint); // attacker received excess

// 5. Call updateRSETHPrice and verify original holders are diluted
lrtOracle.updateRSETHPrice();
uint256 newPrice = lrtOracle.rsETHPrice();
// originalHolderValue = originalHolderRsETH * newPrice < pre-rebase entitlement
```
