Audit Report

## Title
No Chainlink Price Staleness Check in `ChainlinkPriceOracle.getAssetPrice()` Allows Stale Prices to Deflate `rsETHPrice` and Dilute Existing Holders - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `price`, performing no staleness validation. A stale-low price for any supported LST feed deflates `rsETHPrice` in storage. Any subsequent depositor of a correctly-priced asset (e.g., ETH) then receives excess rsETH relative to their contribution, permanently diluting the share value of all existing rsETH holders — a theft of their accrued yield.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` at line 52 silently discards `roundId`, `startedAt`, `updatedAt`, and `answeredInRound`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
```

No check of the form `answeredInRound >= roundId`, `updatedAt != 0`, or `block.timestamp - updatedAt <= heartbeat` is performed. This is in direct contrast to `ChainlinkOracleForRSETHPoolCollateral.getRate()` in the same codebase, which explicitly reverts on `answeredInRound < roundID` and `timestamp == 0`.

The stale price flows through `LRTOracle.getAssetPrice()` → `_getTotalEthInProtocol()` → `_updateRsETHPrice()`, which stores the deflated `rsETHPrice` in state. `updateRSETHPrice()` is `public whenNotPaused` — any external caller can trigger it at any time.

The critical attack path exploits the asymmetry in `getRsETHAmountToMint()`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

When stETH's feed is stale-low and stETH represents a fraction `f` of total TVL, `rsETHPrice` is deflated by approximately `f * staleness_factor`. A depositor of ETH (correctly priced at 1e18) then computes: `(amount * 1e18) / deflatedRsETHPrice`, receiving more rsETH than their ETH contribution warrants. The existing holders' share value is permanently diluted.

The `pricePercentageLimit` guard in `_updateRsETHPrice()` (lines 270–282) only triggers a protocol pause when the price drop exceeds the configured threshold. It provides no protection when `pricePercentageLimit == 0` (the default unset state) or when staleness is within the configured limit.

## Impact Explanation

**High — Theft of unclaimed yield.**

When a Chainlink feed for any supported LST goes stale-low:
1. `_getTotalEthInProtocol()` undervalues protocol TVL.
2. `rsETHPrice` is stored at a deflated value.
3. Any depositor of a correctly-priced asset (ETH, or any LST whose feed is current) receives excess rsETH proportional to the deflation.
4. Existing rsETH holders' share value is permanently diluted — their accrued yield is transferred to the new depositor.

This is a direct, concrete theft of unclaimed yield from existing holders, matching the allowed High impact scope.

## Likelihood Explanation

- `updateRSETHPrice()` is publicly callable with no access control — any external actor can trigger a price update at any time.
- Chainlink feeds go stale during network congestion, oracle downtime, or when the deviation threshold is not crossed for extended periods.
- The protocol supports multiple LSTs (stETH, cbETH, rETH, sfrxETH, swETH, ETHx), each with its own Chainlink feed — any single feed going stale is sufficient.
- No heartbeat or `answeredInRound` check exists anywhere in `ChainlinkPriceOracle`, making this a persistent, unconditional gap.
- The `pricePercentageLimit` guard is bypassable when set to 0 and ineffective for small staleness deviations.

## Recommendation

Add staleness validation in `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    // Optionally: if (block.timestamp - updatedAt > HEARTBEAT) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

## Proof of Concept

1. stETH/ETH Chainlink feed goes stale (`answeredInRound < roundId`, `updatedAt` is hours old, stale price is 0.95e18 vs true 1.0e18). stETH represents 20% of protocol TVL.
2. Attacker calls `LRTOracle.updateRSETHPrice()` (public, no access control).
3. `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns 0.95e18, no revert.
4. `newRsETHPrice` is computed ~1% below true value (20% TVL share × 5% staleness); `rsETHPrice` is stored at the deflated rate.
5. Attacker calls `LRTDepositPool.depositETH{value: largeAmount}(0, "")`.
6. `getRsETHAmountToMint()` computes `(largeAmount * 1e18) / deflatedRsETHPrice` → mints ~1% excess rsETH.
7. Attacker holds inflated rsETH position; all existing holders' share value is permanently diluted by the excess minted supply.
8. Repeat on every stale-feed window for compounding dilution.
