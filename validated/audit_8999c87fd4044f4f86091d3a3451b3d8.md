### Title
Zero-Confirmation Bitcoin Finality Check Bypass via User-Controlled `confirmations: 0` â€” (`crates/foreign-chain-inspector/src/bitcoin/inspector.rs`)

### Summary

The `verify_foreign_transaction` flow accepts a user-supplied `confirmations` field in `BitcoinRpcRequest` with no minimum-value enforcement. When a caller sets `confirmations: 0`, the finality guard `block_confirmations_threshold <= transaction_block_confirmation` evaluates to `0 <= N` which is trivially true for any confirmation count, including 0 (mempool). The MPC network then signs the payload for an unconfirmed Bitcoin transaction, breaking the bridge's core safety invariant that only finalized transactions receive a threshold signature.

This is the direct analog of the Smilee delta=0 issue: a boundary value of a user-controlled numeric parameter causes a critical safety check to be silently skipped rather than enforced.

---

### Finding Description

In `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`, the `extract` method enforces finality via:

```rust
let transaction_block_confirmation = rpc_response.confirmations.into();
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;

if !enough_block_confirmations {
    return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations { ... });
}
``` [1](#0-0) 

The `block_confirmations_threshold` is taken directly from the user-supplied `BitcoinRpcRequest.confirmations` field, which flows from `VerifyForeignTransactionRequestArgs` through `verify_foreign_transaction` on the contract without any minimum-value validation: [2](#0-1) 

The contract performs no lower-bound check on `confirmations`. The `BlockConfirmations` type is a plain numeric wrapper: [3](#0-2) 

When `confirmations = 0`, the comparison `0 <= transaction_block_confirmation` is always true for any value returned by the Bitcoin RPC node, including 0 (mempool transactions). Bitcoin's `getrawtransaction` with `verbose=true` returns mempool transactions with `confirmations = 0`, so the check passes and the MPC network proceeds to sign the payload.

The `respond_verify_foreign_tx` path then verifies the signature against the root public key and resolves the yield, delivering a valid threshold signature to the caller: [4](#0-3) 

---

### Impact Explanation

An attacker submits `verify_foreign_transaction` with `confirmations: 0` for a Bitcoin transaction that is still in the mempool (or has only 1 confirmation on a chain that may reorg). The MPC network issues a valid threshold signature attesting that the transaction was observed. The attacker then:

1. Uses the signature to trigger a bridge payout on NEAR (e.g., minting wrapped tokens).
2. Double-spends or causes a reorg of the Bitcoin transaction.
3. Retains both the NEAR-side payout and the original Bitcoin funds.

This is a **High** impact finding: forged foreign-chain verification / light-client-style verification bypass causing invalid bridge execution or double-spend conditions.

---

### Likelihood Explanation

The attack requires only a single unprivileged `verify_foreign_transaction` call with `confirmations: 0` and a 1 yoctoNEAR deposit. No privileged access, threshold collusion, or TEE bypass is needed. The attacker controls the `confirmations` field entirely. Bitcoin mempool transactions are routinely accessible via `getrawtransaction`. The condition is reachable on every deployment where a Bitcoin `ForeignTx` domain is configured.

---

### Recommendation

Enforce a protocol-level minimum on `confirmations` in the contract before enqueuing the request. Reject any `BitcoinRpcRequest` with `confirmations == 0` (or below a configured minimum, e.g., 1 for testnet, 6 for mainnet):

```rust
// In verify_foreign_transaction, before enqueue_yield_request:
if let ForeignChainRpcRequest::Bitcoin(ref btc_req) = request.request {
    if btc_req.confirmations == BlockConfirmations::from(0u64) {
        env::panic_str("Bitcoin confirmations must be at least 1");
    }
}
```

Alternatively, enforce a per-chain minimum confirmation policy in the on-chain `ForeignChainsMetadata` configuration, and validate the user-supplied value against it at request time.

---

### Proof of Concept

1. Deploy the MPC contract with a Bitcoin `ForeignTx` domain configured.
2. Broadcast a Bitcoin transaction `T` to the mempool (0 confirmations).
3. Call `verify_foreign_transaction` with:
   ```json
   {
     "request": {
       "Bitcoin": {
         "tx_id": "<T's txid>",
         "confirmations": 0,
         "extractors": ["BlockHash"]
       }
     },
     "domain_id": <foreign_tx_domain_id>,
     "payload_version": "V1"
   }
   ```
   with `deposit: 1 yoctoNEAR`.
4. The MPC nodes call `getrawtransaction` on `T`, receive `confirmations: 0` from the RPC.
5. The check `0 <= 0` passes (`enough_block_confirmations = true`).
6. MPC signs the payload and calls `respond_verify_foreign_tx`.
7. The caller receives a valid threshold signature for an unconfirmed transaction.
8. Double-spend `T` on Bitcoin; the NEAR-side bridge has already processed the signature.
