Audit Report

## Title
Deposit-Root Freshness Check in `stake32EthValidated` Enables Block-Stuffing to Revert Staking Transactions — (`contracts/NodeDelegator.sol`)

## Summary

`stake32EthValidated` performs an unconditional revert if the live `get_deposit_root()` differs from the caller-supplied `expectedDepositRoot`. Because the ETH2 deposit root changes with every third-party validator deposit, an attacker who stuffs blocks between the operator's off-chain snapshot and on-chain inclusion can force the transaction to revert, leaving 32 ETH idle in the `NodeDelegator` and `stakedButUnverifiedNativeETH` un-incremented until the operator retries.

## Finding Description

`stake32EthValidated` (lines 186–200) carries no access-control modifier of its own; the `onlyLRTOperator` guard lives inside `stake32Eth`, which is only reached after the deposit-root check passes:

```solidity
// contracts/NodeDelegator.sol  L194-199
IETHPOSDeposit depositContract = _getEigenPodManager().ethPOS();
bytes32 actualDepositRoot = depositContract.get_deposit_root();
if (expectedDepositRoot != actualDepositRoot) {
    revert InvalidDepositRoot(expectedDepositRoot, actualDepositRoot);
}
stake32Eth(pubkey, signature, depositDataRoot);
```

`stakedButUnverifiedNativeETH += 32 ether` is at line 166, inside `stake32Eth`. If the outer function reverts before reaching `stake32Eth`, the accounting increment is never applied and 32 ETH remains in the contract.

The ETH2 deposit root (`get_deposit_root()`) is a Merkle root over all historical deposits and changes with every new validator deposit anywhere on the network. An attacker does not need to cause the root change themselves — they only need to delay the operator's transaction (via block stuffing) long enough for any organic third-party deposit to occur, which on a busy network happens within seconds.

## Impact Explanation

Each stuffed attempt causes the operator's `stake32EthValidated` transaction to revert. The 32 ETH already transferred to the `NodeDelegator` remains idle there, and `stakedButUnverifiedNativeETH` is not incremented for the duration until the operator retries with a freshly computed root. This maps to the allowed impact **Low — Block stuffing**, with a secondary temporary-freeze effect on the 32 ETH per attempt.

## Likelihood Explanation

Block stuffing on Ethereum mainnet is expensive but bounded and has been demonstrated on high-value targets. Critically, the attacker does not need to cause the deposit-root change — only delay the operator's transaction. On a busy network the root changes organically multiple times per block, so even a short stuffing window (a few blocks) is sufficient. The attack is repeatable: every retry by the operator creates a new stuffing window.

## Recommendation

1. **Remove the deposit-root check from `stake32EthValidated`** or make it non-reverting (emit an event instead). The `depositDataRoot` passed to the actual EigenPod `stake` call is what governs correctness at the beacon-chain level; the `expectedDepositRoot` check is a belt-and-suspenders guard that is not required for safety.
2. **Add a deadline/expiry parameter** so the operator can bound the valid window and the transaction auto-expires rather than being stuffable indefinitely.
3. **Use `stake32Eth` directly** for cases where deposit-root front-running risk is acceptable, since it does not have this revert path.

## Proof of Concept

```solidity
// Foundry fork test
function test_blockStuffing_stake32EthValidated() public {
    // 1. Operator snapshots deposit root at block B
    bytes32 snapshotRoot = depositContract.get_deposit_root();

    // 2. Third-party deposit changes the root (simulates organic change during block stuffing)
    vm.prank(thirdParty);
    depositContract.deposit{value: 32 ether}(
        thirdPartyPubkey, thirdPartyWithdrawalCreds, thirdPartySig, thirdPartyDataRoot
    );

    // 3. Confirm root has changed
    bytes32 newRoot = depositContract.get_deposit_root();
    assertNotEq(snapshotRoot, newRoot);

    // 4. Operator submits with stale root — reverts
    vm.prank(operator);
    vm.expectRevert(
        abi.encodeWithSelector(INodeDelegator.InvalidDepositRoot.selector, snapshotRoot, newRoot)
    );
    nodeDelegator.stake32EthValidated(pubkey, signature, depositDataRoot, snapshotRoot);

    // 5. stakedButUnverifiedNativeETH unchanged; 32 ETH idle in NodeDelegator
    assertEq(nodeDelegator.stakedButUnverifiedNativeETH(), 0);
}
```
