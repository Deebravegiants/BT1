Audit Report

## Title
Minimum Deposit/Withdrawal Threshold Mismatch Traps Small rsETH Positions - (File: contracts/LRTDepositPool.sol / contracts/LRTWithdrawalManager.sol)

## Summary
`LRTDepositPool` enforces a minimum deposit in asset units via `minAmountToDeposit`, while `LRTWithdrawalManager` enforces a minimum withdrawal in rsETH units via `minRsEthAmountToWithdraw[asset]`. Because rsETH is a yield-bearing token whose price increases monotonically, the rsETH minted from a deposit of exactly `minAmountToDeposit` decreases over time, eventually falling below `minRsEthAmountToWithdraw[asset]`. Users who deposit at the minimum threshold after sufficient yield accrual receive rsETH they cannot redeem through the protocol's own withdrawal system.

## Finding Description
In `LRTDepositPool._beforeDeposit` (L657–658), the deposit guard operates in asset units:

```solidity
if (depositAmount == 0 || depositAmount < minAmountToDeposit) {
    revert InvalidAmountToDeposit();
}
```

The rsETH minted is computed at L519–520:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

As `rsETHPrice()` increases with yield accrual, the same `depositAmount` produces fewer rsETH. In `LRTWithdrawalManager.initiateWithdrawal` (L162–164) and `instantWithdrawal` (L224–226), the withdrawal guard operates in rsETH units:

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```

The two setter functions — `setMinAmountToDeposit` (L282–285 of `LRTDepositPool.sol`) and `setMinRsEthAmountToWithdraw` (L330–332 of `LRTWithdrawalManager.sol`) — have no cross-validation and operate on different units. Once rsETH appreciates enough that `(minAmountToDeposit * assetPrice) / rsETHPrice < minRsEthAmountToWithdraw[asset]`, any user depositing at the minimum threshold receives rsETH that both `initiateWithdrawal` and `instantWithdrawal` will permanently reject.

## Impact Explanation
A user depositing exactly `minAmountToDeposit` receives rsETH that is below `minRsEthAmountToWithdraw[asset]`. Both withdrawal paths revert with `InvalidAmountToWithdraw`. The user's rsETH is not destroyed but is irrecoverable through the protocol's own withdrawal system. This matches the allowed Low impact: **contract fails to deliver promised returns, but does not directly lose value**.

## Likelihood Explanation
No attacker action is required. The condition arises from normal protocol operation: rsETH accrues staking yield, `rsETHPrice` increases, and the rsETH equivalent of `minAmountToDeposit` drifts downward. Any user depositing near the minimum threshold after sufficient yield accrual is affected. The two thresholds are independently configurable with no on-chain enforcement of their relationship, making silent drift the default outcome on a live protocol where parameters are not continuously re-synchronized.

## Recommendation
Add a cross-validation check in both setter functions to ensure that `(minAmountToDeposit * assetPrice) / rsETHPrice >= minRsEthAmountToWithdraw[asset]` at the time of configuration. Alternatively, express `minRsEthAmountToWithdraw` in asset units and convert to rsETH at withdrawal time using the current oracle rate, eliminating the unit mismatch entirely.

## Proof of Concept
1. Deploy with `rsETHPrice = 1e18`, `minAmountToDeposit = 1e15`, `minRsEthAmountToWithdraw[ETH] = 1e15`.
2. Allow yield to accrue until `rsETHPrice = 1.2e18`.
3. Call `depositETH` with `msg.value = 1e15` (passes L657 check).
4. `rsethAmountToMint = (1e15 * 1e18) / 1.2e18 = 8.33e14` rsETH minted.
5. Call `initiateWithdrawal(ETH, 8.33e14, "")` → L162 check: `8.33e14 < 1e15` → reverts `InvalidAmountToWithdraw`.
6. Call `instantWithdrawal(ETH, 8.33e14, "")` → L224 check: same revert.
7. User holds rsETH unredeemable through the protocol. A Foundry fork test can reproduce this by manipulating `rsETHPrice` in the oracle mock and asserting both withdrawal calls revert after a minimum deposit.
