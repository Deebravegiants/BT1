Audit Report

## Title
`LRTDepositPool.getAssetCurrentLimit()` Returns Inflated Deposit Capacity When Daily rsETH Mint Quota Is Exhausted - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool.getAssetCurrentLimit()` reports remaining deposit capacity based solely on the per-asset cap in `LRTConfig.depositLimitByAsset`, without consulting the independent daily rsETH mint cap enforced by `RSETH.checkDailyMintLimit`. When the daily quota is exhausted, the function returns a non-zero value while any deposit attempt will revert with `DailyMintLimitExceeded`, making the view function's output misleading and unreliable as a deposit-sizing signal.

## Finding Description
`getAssetCurrentLimit()` at `contracts/LRTDepositPool.sol:402-409` computes only:

```solidity
function getAssetCurrentLimit(address asset) public view override returns (uint256) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)) {
        return 0;
    }
    return lrtConfig.depositLimitByAsset(asset) - totalAssetDeposits;
}
```

The deposit execution path is:
1. `depositAsset()` / `depositETH()` → `_beforeDeposit()` → `_checkIfDepositAmountExceedesCurrentLimit()` — checks only `lrtConfig.depositLimitByAsset`.
2. `_mintRsETH()` at `contracts/LRTDepositPool.sol:686-690` calls `IRSETH(rsethToken).mint(msg.sender, rsethAmountToMint)`.
3. `RSETH.mint()` applies the `checkDailyMintLimit(amount)` modifier at `contracts/RSETH.sol:42-56`, which reverts with `DailyMintLimitExceeded` if `currentPeriodMintedAmount + amount > maxMintAmountPerDay`.

`getAssetCurrentLimit()` never reads `RSETH.remainingDailyMintLimit()` (`contracts/RSETH.sol:265-272`), which is the authoritative view of the second, independent gate. The two limits are entirely orthogonal: the per-asset cap is cumulative and permanent; the daily mint cap resets every 24 hours. Either can be the binding constraint at any given moment, and `getAssetCurrentLimit()` only reflects one of them.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but does not lose value.**

When `maxMintAmountPerDay` is active and the daily quota is exhausted, `getAssetCurrentLimit()` returns a positive value while every deposit attempt reverts. No funds are permanently lost because the revert is atomic (ETH is returned; LST `safeTransferFrom` unwinds). The concrete harm is: the canonical public view function exposed in `ILRTDepositPool` at `contracts/interfaces/ILRTDepositPool.sol:60` gives a false signal, causing wasted gas and failed transactions for any user or on-chain integrator that sizes deposits against it.

## Likelihood Explanation
`maxMintAmountPerDay` has a dedicated setter, a dedicated view (`remainingDailyMintLimit`), and a dedicated reset-timestamp getter (`getNextDailyLimitResetTimestamp`), all confirming it is an active production control. Any depositor who queries `getAssetCurrentLimit()` near the end of a high-volume day — when the per-asset cap still has headroom but the daily rsETH quota is exhausted — will receive an inflated value and experience a revert. No special privileges or attacker action are required; this is a normal operating condition.

## Recommendation
`getAssetCurrentLimit()` should factor in the remaining daily rsETH mint capacity. Convert `RSETH.remainingDailyMintLimit()` to asset units using the current exchange rate and return the minimum of the two limits:

```solidity
function getAssetCurrentLimit(address asset) public view override returns (uint256) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)) return 0;
    uint256 assetCapRemaining = lrtConfig.depositLimitByAsset(asset) - totalAssetDeposits;

    address rsethToken = lrtConfig.rsETH();
    uint256 rsethDailyRemaining = IRSETH(rsethToken).remainingDailyMintLimit();
    // maxMintAmountPerDay == 0 means no daily cap is enforced
    if (maxMintAmountPerDay != 0 && rsethDailyRemaining == 0) return 0;

    address lrtOracleAddress = lrtConfig.getContract(LRTConstants.LRT_ORACLE);
    ILRTOracle lrtOracle = ILRTOracle(lrtOracleAddress);
    uint256 assetFromDailyLimit = (rsethDailyRemaining * lrtOracle.rsETHPrice()) / lrtOracle.getAssetPrice(asset);

    return assetCapRemaining < assetFromDailyLimit ? assetCapRemaining : assetFromDailyLimit;
}
```

## Proof of Concept
1. Admin calls `RSETH.setMaxMintAmountPerDay(1000 ether)`.
2. During the day, deposits mint 999 ether of rsETH; `RSETH.remainingDailyMintLimit()` returns `1 ether`.
3. The per-asset deposit cap for stETH still has `500 ether` of headroom.
4. User calls `LRTDepositPool.getAssetCurrentLimit(stETH)` → returns `500 ether` (misleading).
5. User calls `depositAsset(stETH, 2 ether, ...)` — within the asset cap, but the rsETH to mint exceeds the daily quota.
6. `_beforeDeposit` passes (asset cap check at `contracts/LRTDepositPool.sol:661` does not revert).
7. `_mintRsETH` at `contracts/LRTDepositPool.sol:686-690` calls `RSETH.mint`.
8. `checkDailyMintLimit` at `contracts/RSETH.sol:50-52` reverts with `DailyMintLimitExceeded`.
9. Transaction reverts despite `getAssetCurrentLimit()` having indicated capacity was available.

        uint256 effectiveDailyMintAmount = (block.timestamp >= periodStartTime + 1 days) ? 0 : currentPeriodMintedAmount;
