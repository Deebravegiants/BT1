### Title
Mempool-Only Atom/Pair Density Guard Absent from Block Validation Allows Malicious Farmer to Exhaust Validator Memory - (`File: chia/full_node/mempool_manager.py`)

---

### Summary

Chia's mempool enforces a density check bounding the number of CLVM atoms and pairs relative to execution cost. This check exists only in `pre_validate_spendbundle()` and is never applied during block validation (`_pre_validate_block` / `_run_block`). A malicious farmer who wins a block slot can craft a block generator that produces a spend bundle with an atom/pair count far exceeding the mempool threshold, causing all validating nodes to allocate excessive memory during block validation, potentially triggering OOM crashes and a chain halt.

---

### Finding Description

**Mempool validation** (`chia/full_node/mempool_manager.py`, `pre_validate_spendbundle`) enforces two density guards after CLVM execution:

```python
if sbc.num_atoms > sbc.cost * 60_000_000 / self.constants.MAX_BLOCK_COST_CLVM:
    raise ValueError("too many atoms")

if sbc.num_pairs > sbc.cost * 60_000_000 / self.constants.MAX_BLOCK_COST_CLVM:
    raise ValueError("too many pairs")
```

At `MAX_BLOCK_COST_CLVM = 11_000_000_000`, this caps atoms/pairs at 60 million for a max-cost transaction. [1](#0-0) 

**Block validation** (`chia/consensus/multiprocess_validation.py`, `_run_block`) calls `run_block_generator2` directly with no atom/pair check afterward:

```python
def _run_block(block, prev_generators, prev_tx_height, constants):
    flags = get_flags_for_height_and_constants(prev_tx_height, constants)
    ...
    return run_block_generator2(
        bytes(block.transactions_generator),
        prev_generators,
        block.transactions_info.cost,
        flags,
        block.transactions_info.aggregated_signature,
        None,
        constants,
    )
```

No `num_atoms` or `num_pairs` check is performed anywhere in `_pre_validate_block`, `validate_block_body`, or `_run_block`. [2](#0-1) [3](#0-2) 

Additionally, mempool validation uses `flags | MEMPOOL_MODE` (stricter CLVM rules), while block validation uses only `get_flags_for_height_and_constants(prev_tx_height)` — no `MEMPOOL_MODE`. The project's own documentation confirms this asymmetry:

> "Atom/pair bounds (mempool only) — This bounds memory usage relative to cost paid." [4](#0-3) 

The codebase explicitly documents that these checks are intentionally mempool-only, but provides no equivalent protection at block validation time.

---

### Impact Explanation

A malicious farmer who wins any block slot can craft a `transactions_generator` CLVM program that:

1. Stays within `MAX_BLOCK_COST_CLVM` (so the cost check passes)
2. Produces a spend bundle with atom/pair counts far above the 60M mempool threshold (e.g., 500M+ atoms)

Every full node that validates this block calls `_run_block` → `run_block_generator2`, which allocates memory proportional to the atom/pair count. At 32 bytes per atom, 500M atoms = ~16 GB of allocations. Nodes with insufficient RAM crash with OOM. Since all honest nodes must validate the block, this causes a network-wide chain halt.

This matches the allowed High impact: **"Permanent or long-lived inability for honest nodes to process valid blocks... under normal network assumptions."**

---

### Likelihood Explanation

Any farmer with any plot (even a single k32 plot) has a nonzero probability of winning a block slot. The attacker needs no special privilege beyond farming participation. The crafted block generator is a valid CLVM program with valid coin spends and signatures — it passes all other block validation checks. The attack is repeatable every time the attacker wins a slot.

---

### Recommendation

Apply the atom/pair density check in block validation, either:

1. Inside `_pre_validate_block` after `_run_block` returns `conds`, mirroring the mempool check:
   ```python
   if conds.num_atoms > conds.cost * 60_000_000 / constants.MAX_BLOCK_COST_CLVM:
       return PreValidationResult(uint16(Err.INVALID_SPEND_BUNDLE.value), ...)
   if conds.num_pairs > conds.cost * 60_000_000 / constants.MAX_BLOCK_COST_CLVM:
       return PreValidationResult(uint16(Err.INVALID_SPEND_BUNDLE.value), ...)
   ```
2. Or enforce the limit inside the Rust `run_block_generator2` itself so it cannot be bypassed at any call site. [5](#0-4) 

---

### Proof of Concept

1. Attacker farms with any amount of space and waits to win a block slot.
2. Instead of using the standard mempool-derived block generator, the attacker crafts a custom `transactions_generator` CLVM program that spends a valid coin (attacker-owned) and, during execution, constructs a deeply nested list structure producing >200M atoms — all within `MAX_BLOCK_COST_CLVM`.
3. The attacker signs the block normally and broadcasts it.
4. Every peer node calls `_pre_validate_block` → `_run_block` → `run_block_generator2`. The Rust executor allocates memory for all atoms/pairs and returns `SpendBundleConditions` with `num_atoms > 200_000_000`. No Python-level check rejects this.
5. Nodes with <8 GB free RAM crash with OOM. The chain halts until nodes are restarted, and the attack can be repeated on the next won slot.
