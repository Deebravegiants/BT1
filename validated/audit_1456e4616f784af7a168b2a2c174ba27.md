### Title
Permanently Locked/Burned Tokens When `normalize_amount` Rounds Transfer Amount to Zero - (`File: near/omni-bridge/src/lib.rs`)

### Summary
When a user initiates a NEAR→foreign-chain transfer via `ft_on_transfer`, the bridge accepts and locks/burns the tokens immediately. However, `sign_transfer` applies `normalize_amount` (floor division) to scale the amount to the destination chain's decimals. If the user's `amount - fee` is smaller than the decimal scaling factor, `normalize_amount` returns 0, causing `sign_transfer` to panic with `InvalidAmountToTransfer`. Because there is no cancel or refund path for a pending transfer whose signing always reverts, the user's tokens are permanently locked in the bridge (non-deployed tokens) or permanently burned (deployed tokens).

### Finding Description

**Step 1 – Tokens are accepted and locked/burned unconditionally.**

`ft_on_transfer` dispatches to `init_transfer`, which calls `init_transfer_internal`: [1](#0-0) 

For deployed tokens, `burn_tokens_if_needed` fires a detached burn promise; for non-deployed tokens, `lock_tokens_if_needed` increments the locked counter. In both cases the transfer message is stored in `pending_transfers`. The only guard at this stage is `fee < amount`: [2](#0-1) 

There is **no** check that `normalize_amount(amount - fee, decimals) > 0`.

**Step 2 – `normalize_amount` uses floor division.** [3](#0-2) 

For any `amount_without_fee < 10^(origin_decimals - decimals)`, this returns 0.

**Step 3 – `sign_transfer` panics, leaving the transfer permanently stuck.** [4](#0-3) 

The panic happens before the MPC signing call, so `sign_transfer_callback` is never reached and the transfer message is never removed from `pending_transfers`. No cancel or refund function exists for pending transfers.

### Impact Explanation

Tokens are irreversibly lost:
- **Deployed (bridge-minted) tokens**: burned via detached promise; supply is destroyed with no corresponding credit on the destination chain.
- **Non-deployed (locked) tokens**: held in the bridge contract with the locked-token counter incremented; `unlock_tokens_if_needed` is only called during successful `fin_transfer` or `claim_fee` flows, neither of which is reachable for a transfer that can never be signed.

This satisfies: **Critical – Irreversible fund lock / permanently unclaimable user value in bridge flows.**

### Likelihood Explanation

The threshold below which the amount normalizes to zero is `10^(origin_decimals - decimals)`. Concrete examples:

| Token decimals (NEAR) | Destination decimals | Threshold (raw units) | Threshold (human) |
|---|---|---|---|
| 24 | 18 | 10^6 | 0.000000000000000001 tokens — dust only |
| 6 | 0 | 10^6 | 1 full token |
| 8 | 0 | 10^8 | 1 full token |

For tokens with small decimal counts (e.g., 6 or 8 decimals on NEAR bridging to a chain that normalizes to 0 decimals), any transfer of less than 1 full token triggers the bug. A user sending 0.5 tokens would have them permanently locked with no error at submission time.

### Recommendation

Add a pre-flight check in `init_transfer` (before accepting tokens) that verifies the normalized amount is non-zero:

```rust
let token_address = self.get_token_address(destination_chain, token_id.clone())
    .near_expect(BridgeError::TokenNotFound);
let decimals = self.token_decimals.get(&token_address)
    .near_expect(BridgeError::TokenDecimalsNotFound);
let normalized = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
```

This mirrors the existing guard in `sign_transfer` but fires before tokens are taken, allowing `ft_on_transfer` to return the full amount to the sender (standard NEP-141 refund).

### Proof of Concept

1. Register a token with `origin_decimals = 6`, `decimals = 0` (e.g., a 6-decimal NEAR token bridging to a chain that normalizes to 0 decimals).
2. Call `ft_transfer_call` on that token with `amount = 500_000` (0.5 tokens), `fee = 0`, valid EVM recipient.
3. `init_transfer` accepts the transfer; `lock_tokens_if_needed` increments the locked counter by 500,000; transfer stored in `pending_transfers`.
4. Relayer calls `sign_transfer` for this transfer ID.
5. `normalize_amount(500_000, Decimals { origin_decimals: 6, decimals: 0 })` = `500_000 / 10^6` = **0**.
6. `require!(0 > 0, ...)` panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`.
7. No MPC call is made; `sign_transfer_callback` never fires; transfer message remains in `pending_transfers` forever.
8. The 500,000 raw token units are permanently locked in the bridge with no recovery path.

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }

    fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
        let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
        amount / (10_u128.pow(diff_decimals))
    }
