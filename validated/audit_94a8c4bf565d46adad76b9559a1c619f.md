Audit Report

## Title
Nested Unbounded Iteration in Public `updateRSETHPrice()` Causes Gas-Based DoS on Price Updates — (File: `contracts/LRTOracle.sol`)

## Summary

`updateRSETHPrice()` is a public, permissionless function that triggers a deeply nested call chain across `supportedAssets` × `nodeDelegatorQueue` × EigenLayer queued withdrawals × strategies per withdrawal. The cumulative gas cost grows multiplicatively with protocol scale and can exceed the Ethereum block gas limit, permanently preventing rsETH price updates and fee minting. The developers themselves acknowledge this concern in a code comment but their mitigation (capping total withdrawals at 80) does not fully bound the multiplicative product.

## Finding Description

The call chain is confirmed in the actual code:

`updateRSETHPrice()` (LRTOracle.sol:87) calls `_updateRsETHPrice()` (LRTOracle.sol:214), which calls `_getTotalEthInProtocol()` (LRTOracle.sol:331). This function iterates over every `supportedAsset` and calls `ILRTDepositPool.getTotalAssetDeposits(asset)` (LRTDepositPool.sol:385), which calls `getAssetDistributionData(asset)` (LRTDepositPool.sol:426). That function iterates over every NDC in `nodeDelegatorQueue` and calls `INodeDelegator(nodeDelegatorQueue[i]).getAssetUnstaking(asset)` (LRTDepositPool.sol:451). `getAssetUnstaking()` (NodeDelegator.sol:405) makes an external call to `_getDelegationManager().getQueuedWithdrawals(address(this))` and then iterates over every queued withdrawal and every strategy within each withdrawal, calling `strategy.sharesToUnderlyingView()` for each non-ETH strategy.

The total complexity is **O(N × M × W × S)** external calls and iterations:
- **N** = `supportedAssets.length` — no hard cap enforced in contract code
- **M** = `nodeDelegatorQueue.length` — bounded by `maxNodeDelegatorLimit` (default 10, admin-raiseable)
- **W** = queued withdrawals per NDC — global total bounded by `maxUncompletedWithdrawalCount` ≤ 80
- **S** = strategies per withdrawal — variable

Critically, the `maxUncompletedWithdrawalCount` cap of 80 is a **global** counter across all NDCs, not per-NDC. `getAssetUnstaking()` is called N × M times (once per asset per NDC), and each call independently fetches all queued withdrawals for that NDC via `getQueuedWithdrawals()`. The developers' own comment at LRTUnstakingVault.sol:151 reads: *"120 is the max number of uncompleted withdrawals that allows us to still perform update rsETH price"* — confirming awareness of the gas concern — but the cap of 80 only addresses W, not the full N × M × W × S product.

With N=10, M=10, W=8 per NDC (80 total), S=5:
- `getQueuedWithdrawals()` external calls: N × M = **100**
- `sharesToUnderlyingView()` external calls: N × M × W × S = **4,000**
- Total: ~4,100 external calls × ~2,500 gas each = ~10.25M gas for external calls alone, plus SLOAD costs, memory allocation for withdrawal arrays, and loop overhead — easily approaching or exceeding the 30M block gas limit.

## Impact Explanation

If `updateRSETHPrice()` reverts due to out-of-gas, `rsETHPrice` becomes permanently stale. Protocol fees are minted inside `_updateRsETHPrice()` — a stale price means no protocol yield is ever captured again, constituting **permanent freezing of unclaimed yield** (Medium). Additionally, `updateRSETHPriceAsManager()` calls the same `_updateRsETHPrice()` internal function, so even the privileged path is blocked. The impact matches: **Medium — Permanent freezing of unclaimed yield** and **Medium — Unbounded gas consumption**.

## Likelihood Explanation

No attacker action is required. Ordinary protocol growth within admin-set parameters is sufficient: more LST assets added via governance, more NDCs deployed for restaking capacity, and EigenLayer queued withdrawals accumulating during normal unstaking operations. Any external caller can invoke `updateRSETHPrice()` at any time, and once the gas cost exceeds the block limit, the function is permanently uncallable by anyone, including the manager via `updateRSETHPriceAsManager()`.

## Recommendation

1. **Cache `getQueuedWithdrawals` results per NDC**: Call `getQueuedWithdrawals()` once per NDC (not once per asset per NDC) and compute all asset unstaking amounts in a single pass over the returned withdrawals.
2. **Decouple asset unstaking accounting from price updates**: Maintain a running `assetUnstaking` counter updated incrementally on `initiateUnstaking` / `completeUnstaking` events rather than recomputing it on every price update.
3. **Introduce a hard cap on `supportedAssets.length`**: Analogous to `maxNodeDelegatorLimit`, enforce a maximum number of supported assets to bound N.

    function getAssetUnstaking(address asset) external view returns (uint256 amount) {
        (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
            _getDelegationManager().getQueuedWithdrawals(address(this));

        for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
            IDelegationManager.Withdrawal memory withdrawal = queuedWithdrawals[withdrawalIndex];

            for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
                IStrategy strategy = withdrawal.strategies[strategyIndex];

                address strategyAsset = address(strategy) == address(lrtConfig.beaconChainETHStrategy())
                    ? LRTConstants.ETH_TOKEN
                    : address(strategy.underlyingToken());

                if (strategyAsset != asset) continue;

                uint256 sharesToUnstake = withdrawalShares[withdrawalIndex][strategyIndex];
                amount += strategyAsset == LRTConstants.ETH_TOKEN
                    ? sharesToUnstake
                    : strategy.sharesToUnderlyingView(sharesToUnstake);
            }
        }
    }
