### Title
NFT Royalty Silently Truncates to Zero via Integer Division, Enabling Royalty Bypass in Offer Settlement - (File: chia/wallet/nft_wallet/nft_wallet.py)

### Summary
`compute_royalty_amount` uses sequential integer floor-division (`//`) that silently truncates to zero for small offered amounts. When the computed royalty is zero, the wallet skips the royalty coin spend entirely and omits the royalty announcement from the spend bundle. Because the on-chain NFT transfer program enforces royalties only through the `trade_prices_list` supplied by the wallet, an unprivileged attacker can craft an offer with a sufficiently small fungible amount to make every royalty round to zero, completing the NFT transfer without paying any royalty to the creator.

### Finding Description

`compute_royalty_amount` in `chia/wallet/nft_wallet/nft_wallet.py` computes the royalty as:

```python
amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
``` [1](#0-0) 

`MAX_ROYALTY_BASIS_POINTS` is 10 000. For any `offered_amount` where `abs(offered_amount) * percentage < 10 000`, the result is 0. The project's own test suite explicitly documents and asserts this truncation:

```python
def test_small_amount_truncates_to_zero() -> None:
    result = compute_royalty_amount(offered_amount=-50, royalty_split=1, percentage=100)
    assert result == uint64(0)
``` [2](#0-1) 

In `make_nft1_offer`, when the total royalty sum is zero the royalty coin spend is skipped:

```python
if sum(p.amount for _, p in payments) == 0:
    continue
``` [3](#0-2) 

Simultaneously, zero-amount royalty payments are excluded from the puzzle announcements that the spend bundle must assert:

```python
for launcher_id, p in payments
    if p.amount > 0
``` [4](#0-3) 

For the complementary path (NFT being offered rather than requested), the `trade_prices_list` passed to the NFT transfer program is filtered to exclude any price whose royalty rounds to zero:

```python
trade_prices_list=[
    list(price)
    for price in trade_prices
    if price[0] * offered_royalty_percentages[asset] // MAX_ROYALTY_BASIS_POINTS != 0
],
``` [5](#0-4) 

When the list is empty, the NFT transfer program receives no trade prices and therefore enforces no royalty payments. The NFT singleton spend is accepted on-chain without any royalty output.

The same truncation exists in the `royalty_calculation` static method used by the `nft_calculate_royalties` RPC, so the UI also reports zero royalties, giving no warning to the creator:

```python
"amount": abs(amount) // len(royalty_assets_dict) * percentage // MAX_ROYALTY_BASIS_POINTS,
``` [6](#0-5) 

### Impact Explanation

An unprivileged attacker can permanently bypass NFT royalties — a creator-defined, on-chain financial right encoded in the singleton's ownership layer — by constructing an offer whose fungible amount is below the truncation threshold. The NFT transfer settles on-chain with no royalty coin created, constituting an unauthorized accounting change affecting NFT singleton-controlled assets and offer settlement. This matches the Critical impact category: *"Unauthorized … offer settlement, or accounting change affecting … NFTs … singleton-controlled assets."*

### Likelihood Explanation

The threshold is easily reachable. For a 1 % royalty (100 basis points), any offered amount below 100 mojos produces zero royalty. For a 0.01 % royalty (1 basis point), any amount below 10 000 mojos (0.00001 XCH) produces zero. CAT tokens are denominated in their own mojo-equivalent units, so small-denomination CAT offers are equally affected. The attacker needs only to create a valid offer at a price below the threshold; no special privileges, keys, or access are required.

### Recommendation

1. **Enforce a minimum royalty floor**: after computing the royalty, if `percentage > 0` and `royalty == 0`, reject the offer construction with an explicit error rather than silently proceeding.
2. **Use ceiling division** for royalty calculation so that any non-zero percentage always produces at least 1 mojo:
   ```python
   import math
   amount = math.ceil(abs(offered_amount) / royalty_split * percentage / MAX_ROYALTY_BASIS_POINTS)
   ```
3. Apply the same fix to `royalty_calculation` so the RPC and UI accurately reflect the enforced amount.

### Proof of Concept

**Concrete numbers** (1 % royalty, `percentage = 100`):

```
offered_amount = -50   # 50 mojos of CAT offered for the NFT
royalty_split  = 1
percentage     = 100   # 1 %

amount = abs(-50) // 1 * 100 // 10000
       = 50 * 100 // 10000
       = 5000 // 10000
       = 0              ← royalty silently becomes zero
```

With `royalty == 0`:
- `payment_sum == 0` → no royalty coin is sent to `OFFER_MOD_HASH`
- No royalty announcement is added to `announcements_to_assert`
- The royalty coin spend block is skipped (`continue`)
- The NFT transfer program receives an empty `trade_prices_list` and enforces nothing

The resulting spend bundle is valid and accepted by consensus. The NFT creator receives 0 royalty despite having set a non-zero percentage in the on-chain puzzle. [1](#0-0) [7](#0-6) [8](#0-7)

def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    """Compute royalty using integer arithmetic, validating against overflow and excessive percentage."""
    if percentage > MAX_ROYALTY_BASIS_POINTS:
        raise ValueError(f"NFT royalty percentage {percentage} exceeds 100% ({MAX_ROYALTY_BASIS_POINTS} basis points)")
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
    royalty = uint64(amount)
    if royalty >= abs(offered_amount):
        raise ValueError("Royalty amount meets or exceeds the offered amount")
    return royalty
```
