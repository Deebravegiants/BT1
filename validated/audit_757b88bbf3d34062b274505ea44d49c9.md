Audit Report

## Title
Missing `+ amount` in ETH Deposit Limit Check Allows Exceeding the Cap - (File: `contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` omits the incoming `amount` from the ETH branch of the deposit limit check, evaluating only whether the current total already exceeds the cap rather than whether the new deposit would push it over. Any unprivileged caller can invoke `depositETH` with an amount that crosses the `depositLimitByAsset` boundary, causing the protocol to accept ETH and mint rsETH beyond the governance-configured cap.

## Finding Description
In `LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` (lines 676–682), the ETH branch returns `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)` — it fires only when the total has *already* exceeded the limit. The LST branch correctly returns `totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)`. Because `_beforeDeposit` (lines 648–670) reverts only when this function returns `true`, the ETH path never reverts for a deposit that would merely *push* the total over the limit. `depositETH` (lines 76–93) is `external payable` with no role restriction, so any user can trigger this path. No other guard in `_beforeDeposit` or `depositETH` compensates for the missing `+ amount` term.

## Impact Explanation
**Low — Contract fails to deliver promised returns.**
The `depositLimitByAsset` cap is a protocol-enforced risk control. For ETH, this invariant is broken at the boundary: the protocol accepts ETH and mints rsETH for amounts that exceed the cap, taking on more EigenLayer exposure than governance intended. No direct fund theft or loss occurs, but the protocol does not deliver the risk-limiting behavior it promises.

## Likelihood Explanation
**High.** `depositETH` is permissionless and requires no special role or precondition beyond the pool being unpaused and ETH being a supported asset. The condition is reachable whenever `totalAssetDeposits` is at or near the configured limit — a normal operational state. Any depositor observing the current total approaching the cap can trivially submit a deposit that crosses it.

## Recommendation
Remove the ETH-specific branch and apply the same arithmetic to both paths:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Governance sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. `getTotalAssetDeposits(ETH_TOKEN)` returns `999 ether`.
3. Attacker calls `depositETH{value: 2 ether}(0, "")`.
4. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 2 ether)`.
5. ETH branch evaluates `999 ether > 1000 ether → false`; no revert.
6. `_mintRsETH` mints rsETH for 2 ETH; `totalAssetDeposits` becomes `1001 ether`, exceeding the cap.
