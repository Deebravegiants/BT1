Audit Report

## Title
Rewards Permanently Locked When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool.rewardPerToken()` returns `rewardPerTokenStored` unchanged whenever `totalKernelStaked == 0`, while the `updateReward` modifier simultaneously advances `updatedAt`. Any rewards accruing during a zero-staked interval are permanently unclaimable. The `notifyRewardAmount` guard only prevents starting a reward period with zero stakers; it does not prevent `totalKernelStaked` from reaching zero mid-period via normal `initiateWithdrawal` calls.

## Finding Description
`rewardPerToken()` at lines 408–414 short-circuits to `rewardPerTokenStored` when `totalKernelStaked == 0`, skipping accumulation for every elapsed second. The `updateReward` modifier at lines 232–234 unconditionally sets `updatedAt = lastTimeRewardApplicable()`, so once time advances past a zero-staked interval, `rewardPerTokenStored` can never retroactively catch up — those rewards are irrecoverably skipped.

The guard at lines 569–570 (`if (totalKernelStaked == 0) revert NoStakedTokens()`) only fires inside `notifyRewardAmount`. It has no effect once a reward period is already running. `initiateWithdrawal` at lines 325–326 immediately decrements both `balanceOf[msg.sender]` and `totalKernelStaked` at initiation time. `claimWithdrawal` at lines 344–379 does not touch `totalKernelStaked` at all. Therefore, if all stakers call `initiateWithdrawal` while a reward period is active, `totalKernelStaked` drops to zero for up to `withdrawalDelay` (≤ `MAX_WITHDRAWAL_DELAY = 30 days`), and all rewards accruing during that window are permanently locked.

The contract's own NatSpec at lines 17–23 acknowledges this exact class of bug and states the mitigation is purely operational ("ensuring there are always some tokens staked… for the entire duration of the reward period"). This is not code-enforced, and normal permissionless user behavior (mass withdrawal initiation) is sufficient to violate the assumption.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.** Reward tokens transferred via `notifyRewardAmount` that accrue during any zero-staked interval are permanently locked in the contract. The `rewardsToken` balance will exceed the sum of all claimable `rewards[user]` values, and there is no admin rescue, sweep, or reallocation mechanism. This matches the allowed impact: *Medium. Permanent freezing of unclaimed yield.*

## Likelihood Explanation
No attacker coordination is required. Any set of stakers independently exercising their normal right to call `initiateWithdrawal` can collectively drive `totalKernelStaked` to zero. The scenario requires: (1) an active reward period (admin action, already done), and (2) all current stakers initiating withdrawal — a routine, permissionless action. The zero-staked window can persist for up to 30 days within a single reward period. This is realistic under organic market conditions (e.g., a price drop prompting mass exits).

## Recommendation
1. **Defer the `totalKernelStaked` decrement to `claimWithdrawal`**, not `initiateWithdrawal`. Stakers remain "economically present" in the reward accounting until they actually receive their tokens, preventing zero-staked gaps.
2. **Alternatively, accumulate unallocated rewards**: when `totalKernelStaked == 0` in `rewardPerToken()`, track the skipped `rewardRate * elapsed` in a separate `unallocatedRewards` variable and fold it back into the next `notifyRewardAmount` call.
3. As a complementary measure, enforce the operational assumption in code: add a check in `initiateWithdrawal` that reverts (or pauses reward accrual) if the withdrawal would bring `totalKernelStaked` to zero during an active reward period.

## Proof of Concept
1. Admin calls `setRewardsDuration(30 days)`, then `notifyRewardAmount(1_000_000e18)` while Alice has `1e18` KERNEL staked. Guard passes (`totalKernelStaked == 1e18 > 0`). `rewardRate = 1_000_000e18 / 30 days`.
2. Alice immediately calls `initiateWithdrawal(1e18)`. Line 326 sets `totalKernelStaked = 0`. Line 325 sets `balanceOf[Alice] = 0`.
3. For the next 30 days, every `updateReward` trigger calls `rewardPerToken()`, which hits the `totalKernelStaked == 0` branch (line 409) and returns `rewardPerTokenStored` unchanged, while line 234 advances `updatedAt` to `lastTimeRewardApplicable()`.
4. After `withdrawalDelay`, Alice calls `claimWithdrawal` and recovers her KERNEL. The `~1_000_000e18` reward tokens remain in the contract. `earned(Alice) == 0` because `balanceOf[Alice] == 0` during the entire reward period. No staker can claim them; no admin function can recover them.

**Foundry test sketch:**
```solidity
function test_rewardsLockedOnZeroStake() public {
    // Setup: Alice stakes, admin notifies rewards
    vm.prank(alice); pool.stake(1e18);
    vm.prank(admin); pool.notifyRewardAmount(1_000_000e18);
    // Alice immediately withdraws
    vm.prank(alice); pool.initiateWithdrawal(1e18);
    assertEq(pool.totalKernelStaked(), 0);
    // Fast-forward through entire reward period
    vm.warp(block.timestamp + 30 days + 1);
    // Alice claims withdrawal
    vm.prank(alice); pool.claimWithdrawal(1);
    // Rewards are locked: earned == 0, contract holds full reward balance
    assertEq(pool.earned(alice), 0);
    assertGt(rewardsToken.balanceOf(address(pool)), 0);
}
```
