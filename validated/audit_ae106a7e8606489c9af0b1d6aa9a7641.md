### Title
Compromised TEE Verifier Cannot Be Disabled — No Mechanism to Unset `tee_verifier_account_id` to `None` - (File: `crates/contract/src/lib.rs`)

### Summary

The `vote_tee_verifier_change` function in `MpcContract` allows participants to rotate the trusted DCAP attestation verifier account, but provides no mechanism to set `tee_verifier_account_id` back to `None`. Once a verifier is voted in, it can only be replaced by another account — it cannot be disabled. If the active verifier contract is found to be buggy or compromised, participants have no emergency path to halt TEE-gated attestation acceptance without first coordinating a replacement verifier deployment and a new threshold vote.

### Finding Description

`MpcContract` stores the trusted TEE verifier as `Option<AccountId>`: [1](#0-0) 

The only mutation path is `vote_tee_verifier_change`, which requires a non-null `candidate_account_id` and explicitly short-circuits if the candidate equals the current verifier: [2](#0-1) 

The inner `TeeVerifierVotes::vote` resolves to `Some(proposal.candidate_account_id)` on threshold crossing — always a concrete `AccountId`, never `None`: [3](#0-2) 

There is no `vote_disable_tee_verifier` or equivalent function anywhere in the contract. The design document confirms the verifier is the sole dispatch target for `submit_participant_info` DCAP verification: [4](#0-3) 

The analogous protection for OS measurements and launcher hashes also prevents removing the last entry: [5](#0-4) [6](#0-5) 

### Impact Explanation

Once `tee_verifier_account_id` is set, every `submit_participant_info` call dispatches a cross-contract `verify_quote` to that account. If the verifier contract contains a logic bug that causes it to return `VerificationResult::Verified` for arbitrary or malformed quotes, malicious nodes can obtain valid stored attestations. Those attestations gate participation in the MPC signing protocol. A node with a fraudulently accepted attestation can participate in threshold signing rounds, potentially enabling **unauthorized threshold signature issuance** or **unauthorized access to MPC key shares** — both within the Critical/High allowed impact scope.

Because participants cannot set the verifier to `None`, the only remediation path is to deploy a new verifier contract and collect a fresh threshold vote. This coordination window — during which the compromised verifier continues to accept all quotes — is the exploitable gap.

### Likelihood Explanation

The design requires the verifier account to have no full-access keys, making post-deployment code replacement impossible under normal conditions. However, a logic vulnerability in the verifier's `verify_quote` implementation (e.g., incorrect DCAP collateral validation, timestamp bypass, or measurement comparison flaw) is a realistic bug class in complex cryptographic verification code. The NEAR MPC team explicitly acknowledges the verifier rotation scenario in the design document, indicating they consider verifier compromise a credible threat model. The inability to disable the verifier without a replacement makes the response window longer than necessary.

### Recommendation

Add a threshold-gated `vote_disable_tee_verifier` function (or extend `vote_tee_verifier_change` to accept an explicit `None` sentinel) that sets `tee_verifier_account_id = None`. When `None`, `submit_participant_info` should either reject Dstack attestations outright or fall back to a safe default (e.g., accept only Mock attestations). This mirrors the fix applied in the referenced SpokePool report: "Added ability for admin to always set messenger to zero address."

### Proof of Concept

1. Participants vote in a verifier via `vote_tee_verifier_change("verifier.near", hash)` — threshold reached, `tee_verifier_account_id = Some("verifier.near")`.
2. A logic bug is discovered in `verifier.near`'s `verify_quote`: it returns `Verified` for any input.
3. Participants attempt to disable the verifier immediately. There is no call they can make — `vote_tee_verifier_change` requires a non-null replacement candidate, and no `vote_disable_tee_verifier` exists.
4. While participants coordinate deploying `verifier-v2.near` and collecting threshold votes, an adversary submits `submit_participant_info` with a fabricated Dstack attestation. The call dispatches to `verifier.near`, which returns `Verified`. `resolve_verification` stores the entry as a valid attestation.
5. The adversary's node is now treated as an attested participant and joins threshold signing rounds, enabling unauthorized signature production. [7](#0-6) [8](#0-7)

```rust
    pub fn vote_tee_verifier_change(
        &mut self,
        candidate_account_id: AccountId,
        expected_code_hash: TeeVerifierCodeHash,
    ) -> Result<(), Error> {
        log!(
            "vote_tee_verifier_change: signer={}, candidate={}, expected_code_hash={}",
            env::signer_account_id(),
            candidate_account_id,
            expected_code_hash,
        );
        self.voter_or_panic();

        // Voting in the already-current verifier is a no-op
        if self.tee_verifier_account_id.as_ref() == Some(&candidate_account_id) {
            return Ok(());
        }

        let threshold_parameters = self.protocol_state.threshold_parameters_or_panic();
        let participant = AuthenticatedParticipantId::new(threshold_parameters.participants())?;

        let proposal = VerifierChangeProposal {
            candidate_account_id,
            expected_code_hash,
        };
        if let Some(new_verifier) =
            self.tee_verifier_votes
                .vote(proposal, participant, threshold_parameters)?
        {
            log!("vote_tee_verifier_change: new verifier = {}", new_verifier);
            self.tee_verifier_account_id = Some(new_verifier);
        }
        Ok(())
    }
```

```rust
    pub fn vote(
        &mut self,
        proposal: VerifierChangeProposal,
        participant: AuthenticatedParticipantId,
        threshold_parameters: &ThresholdParameters,
    ) -> Result<Option<AccountId>, Error> {
        let protocol_threshold = threshold_parameters.threshold().value();
        let participants = threshold_parameters.participants();
        let proposal_hash: ProposalHash = proposal.clone().into();

        let count_usize = {
            let voter_set = self.pending.vote(participant, proposal_hash);
            voter_set.count_for(|p| participants.is_participant_given_participant_id(&p.get()))
        };
        let count = u64::try_from(count_usize).map_err(|e| ConversionError::DataConversion {
            reason: format!("vote count {count_usize} does not fit in u64: {e}"),
        })?;

        if count >= protocol_threshold {
            self.pending.clear();
            Ok(Some(proposal.candidate_account_id))
        } else {
            Ok(None)
        }
    }
```

```markdown
There's a small race worth naming, and it is benign. A `submit_participant_info` call schedules its cross-contract call to the current verifier, and then — before that call executes — a `vote_tee_verifier_change` passes and updates `tee_verifier_account_id` to a different address. The in-flight verification does not redirect to the new verifier: the target account of a cross-contract call is fixed when the call is scheduled, not re-read when it executes.     pub fn remove(&mut self, measurement: &ContractExpectedMeasurements) -> bool {
```
```rust
        let would_remain = self.entries.iter().filter(|e| *e != measurement).count();
        if would_remain == 0 {

        }
```
