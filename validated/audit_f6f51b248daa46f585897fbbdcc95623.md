Missing Vote Signature Verification Allows Unauthenticated Gossipsub Peer to Forge Validator Votes and Corrupt Consensus — (`crates/apollo_consensus/src/single_height_consensus.rs`)

---

### Summary

`SingleHeightConsensus::handle_vote` contains an explicit `// TODO(Asmaa): verify the signature` comment and performs **no cryptographic verification** of the `Vote.signature` field before counting the vote toward quorum. Any unauthenticated gossipsub peer can craft `Vote` messages with a legitimate validator's `ContractAddress` in the `voter` field and arbitrary bytes in `signature`, and those votes will be accepted and counted by the state machine.

---

### Finding Description

The full inbound vote path is:

1. Gossipsub delivers `(Result<Vote, ProtobufConversionError>, BroadcastedMessageMetadata)` to `MultiHeightManager::handle_vote` in `manager.rs`.
2. `manager.handle_vote` performs only protobuf-parse error handling and height-range filtering, then calls `shc.handle_vote(message)`.
3. `SingleHeightConsensus::handle_vote` (the only place where per-vote validation occurs) checks:
   - `vote.height == current_height`
   - `vote.voter` is a member of the committee (address equality only)
   - duplicate / conflict detection via `state_machine.received_vote`
4. The vote is then forwarded directly to the state machine as `StateMachineEvent::Prevote(vote)` or `StateMachineEvent::Precommit(vote)`.

At no point is `vote.signature` verified against the claimed `vote.voter`'s public key. The `verify_precommit_vote_signature` function exists in `apollo_signature_manager` and is tested, but it is **never called** anywhere in `crates/apollo_consensus`. [1](#0-0) 

The state machine's `handle_prevote` and `handle_precommit` insert the vote keyed by `(round, voter)` and immediately accumulate weight toward quorum thresholds: [2](#0-1) [3](#0-2) 

The `// TODO(Asmaa): verify the signature` comment is the only acknowledgement of this gap: [4](#0-3) 

The `verify_precommit_vote_signature` function that should gate this path exists but is unused in consensus: [5](#0-4)

### Impact Explanation

An attacker who can connect as a gossipsub peer (the network is unauthenticated at the message level) can:

1. Enumerate the current validator committee addresses (public information broadcast in `ProposalInit`).
2. For each validator address `V_i`, craft a `Vote { vote_type: Precommit, height: H, round: R, proposal_commitment: C, voter: V_i, signature: <arbitrary> }`.
3. Inject one forged precommit per validator until the quorum threshold is crossed.
4. The state machine fires `SMRequest::DecisionReached` for commitment `C`.
5. `decision_reached` in `sequencer_consensus_context.rs` looks up `C` in `valid_proposals` and, if a matching validated proposal exists, commits the block.

The constraint is that `C` must correspond to a proposal that was actually received and validated by the target node (the `valid_proposals` lookup would panic/fail otherwise). This means the attacker cannot inject a completely fabricated block, but they **can** force the node to decide on a legitimately proposed block before the honest validator set has actually reached quorum — causing the node to commit a block the rest of the network has not agreed on, producing a consensus fork and corrupting the node's chain state. [6](#0-5)

### Likelihood Explanation

- Gossipsub peers are unauthenticated at the vote-message level; no transport-layer credential is required to send a `Vote` protobuf.
- Validator addresses are public (broadcast in `ProposalInit`).
- The attack requires only knowledge of the current committee and the ability to open a gossipsub connection — no validator key material is needed.
- The `// TODO` comment confirms the gap is known but unimplemented, making it a concrete, reachable missing control rather than a theoretical bypass.

---

### Recommendation

Before forwarding a vote to the state machine, `SingleHeightConsensus::handle_vote` must:

1. Resolve the `voter` address to its corresponding ECDSA public key (from the committee roster).
2. Call `verify_precommit_vote_signature` (or an equivalent prevote verifier) against `vote.signature` and the canonical message digest for the vote.
3. Drop and report the peer on verification failure.

The `verify_precommit_vote_signature` utility in `apollo_signature_manager` already implements the ECDSA check and should be wired in here. [5](#0-4)

    pub(crate) fn handle_vote(&mut self, vote: Vote) -> Requests {
        // TODO(Asmaa): verify the signature
        trace!("Received {:?}", vote);
        let height = self.state_machine.height();
        if vote.height != height {
            warn!("Invalid vote height: expected {:?}, got {:?}", height, vote.height);
            return VecDeque::new();
        }
        if !self.committee.members().iter().any(|s| s.address == vote.voter) {
            debug!("Ignoring vote from non validator: vote={:?}", vote);
            return VecDeque::new();
        }

        // Check if vote has already been received.
        match self.state_machine.received_vote(&vote) {
            VoteStatus::Duplicate => {
                // Duplicate - ignore.
                trace_every_n_ms!(
                    DUPLICATE_VOTE_LOG_PERIOD_MS,
                    "Ignoring duplicate vote: {vote:?}"
                );
                return VecDeque::new();
            }
            VoteStatus::Conflict(old_vote, new_vote) => {
                // Conflict - ignore and record.
                warn!("Conflicting votes: old={old_vote:?}, new={new_vote:?}");
                CONSENSUS_CONFLICTING_VOTES.increment(1);
                return VecDeque::new();
            }
            VoteStatus::New => {
                // Vote is new, proceed to process it.
            }
        }

        info!("Accepting {:?}", vote);
        let sm_vote = match vote.vote_type {
            VoteType::Prevote => StateMachineEvent::Prevote(vote),
            VoteType::Precommit => StateMachineEvent::Precommit(vote),
        };
        self.state_machine.handle_event(sm_vote)
    }
