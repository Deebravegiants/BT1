### Title
Farmer Signs Attacker-Controlled Foliage Hashes Without Validating `farmer_reward_puzzle_hash`, Enabling Block Reward Diversion — (`chia/farmer/farmer_api.py`)

---

### Summary

`FarmerAPI.request_signed_values` accepts `foliage_block_data_hash` and `foliage_transaction_block_hash` directly from a full_node peer and forwards them to the harvester for signing — and then co-signs them itself — without ever verifying that the hashes correspond to a `FoliageBlockData` whose `farmer_reward_puzzle_hash` matches the farmer's configured reward address. A malicious full_node peer that receives a valid `DeclareProofOfSpace` (and thus the `quality_string`) can substitute attacker-controlled hashes, obtain valid plot-key signatures over a block that pays the farmer reward to the attacker's address, and broadcast that block to the network.

---

### Finding Description

**Entry point — `request_signed_values`** (`chia/farmer/farmer_api.py`, lines 723–769):

The only guard is a lookup of `quality_string` in `self.farmer.quality_str_to_identifiers`:

```python
if full_node_request.quality_string not in self.farmer.quality_str_to_identifiers:
    self.farmer.log.error(...)
    return None
``` [1](#0-0) 

After that check passes, the function builds a `RequestSignatures` message using the hashes verbatim from the full_node request:

```python
request = harvester_protocol.RequestSignatures(
    plot_identifier,
    challenge_hash,
    sp_hash,
    [full_node_request.foliage_block_data_hash, full_node_request.foliage_transaction_block_hash],
    ...
)
``` [2](#0-1) 

There is no check that `foliage_block_data_hash == hash(foliage_block_data)` and no check that `foliage_block_data.farmer_reward_puzzle_hash == self.farmer.farmer_target`.

**Signing — `_process_respond_signatures`** (lines 934–979):

The farmer co-signs whatever hashes the harvester returned signatures for:

```python
foliage_sig_farmer = AugSchemeMPL.sign(sk, foliage_block_data_hash, agg_pk)
foliage_transaction_block_sig_farmer = AugSchemeMPL.sign(sk, foliage_transaction_block_hash, agg_pk)
``` [3](#0-2) 

**How `quality_string` is obtained by the attacker:**

`quality_str_to_identifiers` is populated when the farmer finds a proof of space:

```python
self.farmer.quality_str_to_identifiers[computed_quality_string] = (
    new_proof_of_space.plot_identifier,
    new_proof_of_space.challenge_hash,
    new_proof_of_space.sp_hash,
    peer.peer_node_id,
)
``` [4](#0-3) 

The farmer then sends `DeclareProofOfSpace` (which contains `quality_string`) to all full_node peers, including the attacker's. The attacker's full_node thus receives the valid `quality_string` it needs.

**Consensus does not restrict `farmer_reward_puzzle_hash`:**

Block header validation only checks that the foliage signature is valid over the foliage block data actually present in the block:

```python
if not AugSchemeMPL.verify(
    header_block.reward_chain_block.proof_of_space.plot_public_key,
    header_block.foliage.foliage_block_data.get_hash(),
    header_block.foliage.foliage_block_data_signature,
):
    return None, ValidationError(Err.INVALID_PLOT_SIGNATURE)
``` [5](#0-4) 

There is no consensus rule that `farmer_reward_puzzle_hash` must equal any particular address (except for the genesis block). A block with an attacker-controlled `farmer_reward_puzzle_hash` and valid plot-key signatures passes all consensus checks.

**The intended validation exists only in test code:**

The test intercept at `test_third_party_harvesters.py` lines 227–244 explicitly validates `foliage_block_data.farmer_reward_puzzle_hash == farmer_reward_address` and `foliage_block_data.get_hash() == foliage_block_data_hash`, but this logic is absent from the production handler. [6](#0-5) 

---

### Impact Explanation

A malicious full_node peer can redirect every block reward the farmer wins to an attacker-controlled address. The farmer's plot key signs the attacker's `FoliageBlockData`, the resulting block is valid under consensus, and the farmer receives nothing. This is a direct, irreversible XCH loss per won block.

---

### Likelihood Explanation

The farmer must be connected to a malicious full_node. This is realistic when:
- The farmer uses a remote/third-party full_node (a common configuration for lightweight setups)
- The farmer's full_node peer is a network-level attacker who has established a connection

The attacker needs no leaked keys, no broken cryptography, and no privileged access — only a peer connection to the farmer.

---

### Recommendation

In `request_signed_values`, before forwarding hashes to the harvester, the farmer must:

1. Require `foliage_block_data` to be present in the request (not `None`).
2. Assert `full_node_request.foliage_block_data.get_hash() == full_node_request.foliage_block_data_hash`.
3. Assert `full_node_request.foliage_block_data.farmer_reward_puzzle_hash == self.farmer.farmer_target`.

This mirrors the validation already present in the test intercept at `test_third_party_harvesters.py` lines 235–244 and should be promoted to the production handler. [7](#0-6)
