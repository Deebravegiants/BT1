Audit Report

## Title
Permanent Freezing of Unclaimed Rewards When `totalKernelStaked` Drops to Zero Mid-Period — (`contracts/KERNEL/KernelDepositPool.sol`)

## Summary

When the sole staker (or a coordinated group holding 100% of `totalKernelStaked`) calls `initiateWithdrawal` for their full balance during an active reward period, `totalKernelStaked` is decremented to zero. From that point forward, `rewardPerToken()` permanently returns the frozen `rewardPerTokenStored` value, and the remaining `(finishAt - block.timestamp) * rewardRate` reward tokens are irrecoverably locked in the contract. No admin function exists to rescue them.

## Finding Description

`rewardPerToken()` short-circuits to `rewardPerTokenStored` whenever `totalKernelStaked == 0`: [1](#0-0) 

`initiateWithdrawal()` applies the `updateReward(msg.sender)` modifier first — correctly snapshotting the exiting staker's earned rewards — then unconditionally decrements `totalKernelStaked`: [2](#0-1) 

Once `totalKernelStaked` reaches zero, every subsequent call to `rewardPerToken()` returns the same frozen value. No staker can accumulate new rewards because `balanceOf` is zero for all accounts, and the formula `balanceOf[_account] * (rewardPerToken() - userRewardPerTokenPaid[_account])` always evaluates to zero: [3](#0-2) 

`notifyRewardAmount()` guards against *starting* a period with zero staked tokens but provides no protection against `totalKernelStaked` reaching zero *after* a period has started: [4](#0-3) 

The contract contains no `recoverERC20`, `sweep`, or any other admin escape hatch. The file ends at line 621 with only `setMaxNumberOfWithdrawalsPerUser`: [5](#0-4) 

The contract's own NatSpec explicitly acknowledges this design limitation and states it is mitigated *operationally*, not on-chain: [6](#0-5) 

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.** The remaining reward tokens for the rest of the active period become permanently unclaimable and unrecoverable inside `KernelDepositPool`. The `rewardsToken` balance sits in the contract forever. This exactly matches the allowed impact: *"Medium. Permanent freezing of unclaimed yield."*

## Likelihood Explanation

The scenario requires a single staker holding 100% of `totalKernelStaked`, or a coordinated group. This is realistic in early deployment (one bootstrap staker before others join) or in a low-participation environment. No privileged role is needed — `initiateWithdrawal` is a public user function callable by any staker. The triggering party also recovers their own principal after `withdrawalDelay`, so there is zero net cost to causing this outcome.

## Recommendation

1. **Add a `recoverERC20` admin function** that allows recovery of excess reward tokens — i.e., `rewardsToken.balanceOf(address(this))` minus the sum of all `rewards[user]` mappings — so stranded rewards can be rescued and re-notified via `notifyRewardAmount`.
2. **Alternatively**, enforce on-chain that `totalKernelStaked` cannot reach zero during an active reward period by reverting `initiateWithdrawal` when `block.timestamp < finishAt && totalKernelStaked - _amount == 0`.
3. The operational note in the NatSpec is insufficient as a security control and should not be relied upon.

## Proof of Concept

```solidity
// Setup: admin sets duration and notifies reward amount (requires totalKernelStaked > 0)
pool.setRewardsDuration(7 days);
rewardsToken.approve(address(pool), 1000e18);
pool.notifyRewardAmount(1000e18);
// rewardRate = 1000e18 / 7 days

// Attacker is the sole staker; initiates withdrawal for full balance mid-period
// updateReward runs first: rewardPerTokenStored updated, attacker's earned() credited
pool.initiateWithdrawal(attackerFullBalance);
// Now totalKernelStaked == 0

// rewardPerToken() is now permanently frozen
assert(pool.rewardPerToken() == pool.rewardPerTokenStored()); // always true hereafter

// After finishAt, remaining rewards are locked with no recovery path
// locked ≈ (finishAt - timestampOfWithdrawal) * rewardRate
assert(rewardsToken.balanceOf(address(pool)) > 0); // permanently stuck

// Attacker claims principal after withdrawalDelay — zero net cost
pool.claimWithdrawal(withdrawalId);
```

        if (_amount == 0) revert AmountZero();
        if (balanceOf[msg.sender] < _amount) revert InsufficientStakedBalance();
        if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();

        balanceOf[msg.sender] -= _amount;
        totalKernelStaked -= _amount;
