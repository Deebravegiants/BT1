Audit Report

## Title
Rewards Permanently Frozen When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
The `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` even when `totalKernelStaked == 0`. If all stakers exit during an active reward period, the reward tokens accrued during the empty interval are permanently skipped and frozen in the contract. No on-chain recovery mechanism exists. The developers explicitly acknowledge this behavior in the contract NatSpec but rely solely on an off-chain operational assumption that tokens will always remain staked — an assumption that cannot be enforced on-chain.

## Finding Description
In `updateReward` (L232–242), `updatedAt` is advanced unconditionally:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    updatedAt = lastTimeRewardApplicable();  // always advances, even when totalKernelStaked == 0
    ...
}
```

`rewardPerToken()` (L408–414) correctly short-circuits when the pool is empty, returning `rewardPerTokenStored` unchanged. However, because `updatedAt` was already advanced, the time window during which no tokens were staked is silently consumed. When the next `stake()` call fires `updateReward`, the formula `rewardRate * (lastTimeRewardApplicable() - updatedAt)` uses the newly advanced `updatedAt`, so the rewards that accrued during the entire empty interval are never added to `rewardPerTokenStored`.

`notifyRewardAmount` (L569–570) guards against starting a new period on an empty pool:
```solidity
if (totalKernelStaked == 0) revert NoStakedTokens();
```
But this guard only prevents *initiating* a period on an empty pool. It does not prevent the pool from becoming empty *after* a period has started. Once `totalKernelStaked` reaches zero mid-period, the rewards for that gap are unrecoverable.

The contract does not inherit `Recoverable` and has no `recoverTokens`, `emergencyWithdraw`, or equivalent function. The frozen reward tokens cannot be retrieved by anyone.

Critically, the developers themselves document this exact behavior in the contract NatSpec (L17–22):
> *"If `totalKernelStaked` ever hits zero during a reward distribution window, any remaining rewards for that period will stay locked in the contract. In this deployment, we're avoiding this issue by ensuring there are always some tokens staked before admin calls the `notifyRewardAmount` function, as well as for the entire duration of the reward period."*

This confirms the behavior is real and acknowledged, but the stated mitigation is a purely off-chain operational assumption with no on-chain enforcement.

## Impact Explanation
Reward tokens transferred into the contract via `notifyRewardAmount` are permanently frozen for any interval where `totalKernelStaked == 0` during an active reward period. There is no admin escape hatch to recover them. This constitutes **permanent freezing of unclaimed yield**, a Medium severity impact per the allowed scope.

## Likelihood Explanation
Any staker can call `initiateWithdrawal` at any time; this immediately decrements `totalKernelStaked` (L325–326). No privileged action or attacker coordination is required — ordinary user withdrawals are sufficient. If all stakers exit during an active reward period (a realistic scenario during market stress, migration, or a competing incentive program), the pool reaches `totalKernelStaked == 0` while `block.timestamp < finishAt`. The protocol's stated mitigation ("ensuring there are always some tokens staked for the entire duration") is not enforceable on-chain and cannot be guaranteed.

## Recommendation
In the `updateReward` modifier, only advance `updatedAt` when `totalKernelStaked > 0`:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    ...
}
```

This is the standard fix for Synthetix-style staking contracts and ensures that time windows with no stakers do not consume the reward budget.

## Proof of Concept
1. Admin calls `notifyRewardAmount(1000e18)` with `totalKernelStaked > 0`. `finishAt = T + duration`, `updatedAt = T`, `rewardRate = 1000e18 / duration`.
2. All stakers call `initiateWithdrawal` for their full balances. After the last withdrawal, `totalKernelStaked == 0`. `updatedAt` was last set to `T+100`.
3. No one stakes for 500 seconds. Rewards worth `rewardRate * 500` tokens accrue but are unassignable.
4. At `T+600`, Alice calls `stake(1)`. `updateReward` fires:
   - `rewardPerToken()` returns `rewardPerTokenStored` unchanged (since `totalKernelStaked == 0` before Alice's balance is added).
   - `updatedAt` is set to `T+600`.
5. The `rewardRate * 500` tokens from step 3 are permanently skipped — `updatedAt` jumped from `T+100` to `T+600` without any corresponding increase in `rewardPerTokenStored`. Those tokens remain in the contract with no distribution or recovery path.

**Foundry test plan:** Deploy `KernelDepositPool`, stake tokens, call `notifyRewardAmount`, warp forward, have all stakers call `initiateWithdrawal`, warp forward again, have a new staker call `stake`, then assert that `rewardsToken.balanceOf(address(pool))` exceeds the sum of all `earned()` values across all accounts — confirming the stranded balance.
