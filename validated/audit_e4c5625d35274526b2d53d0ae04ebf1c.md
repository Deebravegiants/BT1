### Title
`RequireGuard` Key-Hash Credentials Require No Witness Signature, Allowing Unauthorized Script Execution - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The Dijkstra era introduces a `RequireGuard` native script construct intended to require a credential to authorize script execution. For key-hash guard credentials, the ledger checks only that the credential appears in the transaction body's `guardsTxBodyL` field — but never requires a corresponding cryptographic witness (signature) from the key holder. Any transaction author can include an arbitrary key hash in the guards field without the key holder's consent, causing `RequireGuard (KeyHashObj kh)` to evaluate to `True` and bypassing the intended access control.

---

### Finding Description

The Dijkstra era adds a `RequireGuard` constructor to native scripts:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Scripts.hs
evalDijkstraNativeScript keyHashes (ValidityInterval txStart txExp) guards = go
  where
    go = \case
      ...
      RequireGuard cred -> cred `OSet.member` guards   -- line 575
``` [1](#0-0) 

Here `guards` is `txBody ^. guardsTxBodyL`, an `OSet (Credential Guard)` that the **transaction author** populates freely in the tx body. The check is purely: "is this credential in the set I wrote into my own tx body?"

For **script-hash** guard credentials, `getDijkstraScriptsNeeded` correctly adds them to the scripts-needed set, forcing the script to be witnessed and validated:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
guardingScriptsNeeded = AlonzoScriptsNeeded $
  catMaybes $
    zipAsIxItem (txb ^. guardsTxBodyL) $
      \(AsIxItem idx cred) -> (\sh -> (GuardingPurpose (AsIxItem idx sh), sh)) <$> credScriptHash cred
``` [2](#0-1) 

`credScriptHash` returns `Nothing` for `KeyHashObj` credentials, so **key-hash guard credentials are silently dropped** from the scripts-needed set. No `GuardingPurpose` redeemer is required, and no script witness is checked.

For the key-witness side, Dijkstra reuses `getConwayWitsVKeyNeeded`:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/UTxO.hs
getConwayWitsVKeyNeeded utxo txBody =
  getShelleyWitsVKeyNeededNoGov utxo txBody
    `Set.union` Set.map asWitness (txBody ^. reqSignerHashesTxBodyG)
    `Set.union` voterWitnesses txBody
``` [3](#0-2) 

Guard key hashes are **not included** in any of these three sets. The UTXOW rule then enforces:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxow.hs
runTest $ Shelley.validateNeededWitnesses witsKeyHashes certState originalUtxo txBody
``` [4](#0-3) 

Since guard key hashes are absent from the required-witnesses set, the ledger never checks that the key holder signed the transaction. The existing test confirms this behavior — the transaction succeeds with only the guard credential listed in the tx body, no signature from the guard key:

```haskell
-- eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/UtxowSpec.hs
submitTx_ $ tx & bodyTxL . guardsTxBodyL .~ [guardKeyHash]
-- No witness from guardKeyHash is added
``` [5](#0-4) 

The CDDL comment explicitly states the design intent:

```
-- Dijkstra adds guard scripts for enhanced security.
-- A guard script requires a credential to authorize execution.
``` [6](#0-5) 

The implementation contradicts this intent for key-hash credentials.

---

### Impact Explanation

Any UTxO, minting policy, or certificate operation locked by a `RequireGuard (KeyHashObj kh)` native script can be spent, minted against, or certified by any transaction that simply lists `kh` in the `guardsTxBodyL` field — without a signature from the holder of `kh`. This constitutes a direct, attacker-controlled bypass of the script's access control, enabling unauthorized spending of ADA and native assets, or unauthorized minting/burning of tokens, through an invalid ledger state transition.

**Impact class:** Critical — Direct loss, creation, or destruction of ADA or native assets through an invalid ledger state transition.

---

### Likelihood Explanation

The attack requires no privileged access. Any transaction author who can observe a UTxO or minting policy locked by a `RequireGuard (KeyHashObj kh)` script can exploit this immediately by constructing a transaction with the target key hash in the guards field. The `guardsTxBodyL` field is freely writable by the transaction author. No key material, governance role, or special permission is needed.

---

### Recommendation

`getDijkstraScriptsNeeded` (or a parallel function) must add key-hash guard credentials to the required-witness set. Specifically, for each `KeyHashObj kh` in `guardsTxBodyL`, `asWitness kh` must be added to the set returned by `getWitsVKeyNeeded` (analogous to how `reqSignerHashesTxBodyG` key hashes are handled). This ensures that `RequireGuard (KeyHashObj kh)` can only be satisfied when the key holder has actually signed the transaction.

For script-hash guard credentials, the existing `getDijkstraScriptsNeeded` path is correct and requires no change.

---

### Proof of Concept

1. Deploy a UTxO to an address whose payment credential is the hash of the native script `RequireGuard (KeyHashObj kh)` for some key `kh` controlled by Alice.
2. Attacker (Bob) constructs a transaction spending that UTxO, setting `guardsTxBodyL = [KeyHashObj kh]` in the tx body. Bob does **not** add a witness for `kh`.
3. The ledger evaluates `evalDijkstraNativeScript`:
   - `RequireGuard (KeyHashObj kh)` → `(KeyHashObj kh) `OSet.member` {KeyHashObj kh}` → `True`
4. `getDijkstraScriptsNeeded` returns `Nothing` for `KeyHashObj kh` (no script hash), so no script witness is required.
5. `getConwayWitsVKeyNeeded` does not include `kh`, so no key witness is required.
6. `Shelley.validateNeededWitnesses` passes.
7. The transaction is accepted; Bob has spent Alice's funds without her authorization. [7](#0-6) [8](#0-7) [9](#0-8) [10](#0-9)

evalDijkstraNativeScript ::
  (DijkstraEraScript era, NativeScript era ~ DijkstraNativeScript era) =>
  Set.Set (KeyHash Witness) ->
  ValidityInterval ->
  OSet (Credential Guard) ->
  NativeScript era ->
  Bool
evalDijkstraNativeScript keyHashes (ValidityInterval txStart txExp) guards = go
  where
    -- the evaluation will stop as soon as it reaches the required number of valid scripts
    isValidMOf n SSeq.Empty = n <= 0
    isValidMOf n (ts SSeq.:<| tss) =
      n <= 0 || if go ts then isValidMOf (n - 1) tss else isValidMOf n tss
    go = \case
      RequireTimeStart lockStart -> lockStart `lteNegInfty` txStart
      RequireTimeExpire lockExp -> txExp `ltePosInfty` lockExp
      RequireSignature hash -> hash `Set.member` keyHashes
      RequireAllOf xs -> all go xs
      RequireAnyOf xs -> any go xs
      RequireMOf m xs -> isValidMOf m xs
      RequireGuard cred -> cred `OSet.member` guards
      _ -> error "Impossible: All NativeScripts should have been accounted for"
```

getConwayWitsVKeyNeeded ::
  (EraTx era, ConwayEraTxBody era) =>
  UTxO era ->
  TxBody l era ->
  Set.Set (KeyHash Witness)
getConwayWitsVKeyNeeded utxo txBody =
  getShelleyWitsVKeyNeededNoGov utxo txBody
    `Set.union` Set.map asWitness (txBody ^. reqSignerHashesTxBodyG)
    `Set.union` voterWitnesses txBody

voterWitnesses ::
  ConwayEraTxBody era =>
  TxBody l era ->
  Set.Set (KeyHash Witness)
voterWitnesses txb =
  Map.foldrWithKey' accum mempty (unVotingProcedures (txb ^. votingProceduresTxBodyL))
  where
    accum voter _ khs =
      maybe khs (`Set.insert` khs) $
        case voter of
          CommitteeVoter cred -> credKeyHashWitness cred
          DRepVoter cred -> credKeyHashWitness cred
          StakePoolVoter poolId -> Just $ asWitness poolId
