### Title
Stale Governance Votes from Prospective Participants Persist After `vote_cancel_resharing`, Enabling Threshold Bypass for TEE Attestation Whitelisting — (`File: crates/contract/src/lib.rs`)

---

### Summary

When resharing completes successfully via `vote_reshared`, the contract spawns six detached cleanup promises to purge stale votes from participants who are no longer in the new set. When resharing is **cancelled** via `vote_cancel_resharing`, the contract transitions back to the previous Running state but spawns **none** of these cleanup promises. Votes cast by prospective participants (candidates in the cancelled resharing) remain in `tee_state.votes`, `tee_verifier_votes`, and `proposed_updates`, and continue to count toward governance thresholds. An adversarial prospective participant can pre-vote for a malicious TEE image hash during resharing, then benefit from resharing cancellation to have that stale vote count toward the whitelisting threshold with fewer real-participant votes required.

---

### Finding Description

`vote_reshared` (the successful resharing path) spawns six cleanup promises on every Resharing → Running transition: [1](#0-0) 

These cleanups remove stale votes from participants who are no longer in the new set:
- `REMOVE_NON_PARTICIPANT_UPDATE_VOTES` — purges `proposed_updates` votes
- `CLEAN_TEE_STATUS` — purges `tee_state.votes` (code-hash votes)
- `CLEAN_INVALID_ATTESTATIONS` — prunes expired stored attestations
- `CLEANUP_ORPHANED_NODE_MIGRATIONS` — removes orphaned migration data
- `CLEAN_FOREIGN_CHAIN_DATA` — removes foreign-chain data for non-participants
- `REMOVE_NON_PARTICIPANT_TEE_VERIFIER_VOTES` — purges `tee_verifier_votes`

`vote_cancel_resharing` (the cancellation path) performs **none** of these cleanups: [2](#0-1) 

The state machine allows both paths to reach Running: [3](#0-2) 

The test suite explicitly documents that stale votes from removed participants **must not** count toward quorum after a participant-set change: [4](#0-3) 

Yet this invariant is only enforced on the `vote_reshared` path, not on `vote_cancel_resharing`.

---

### Impact Explanation

During resharing, prospective participants (candidates in the new set) are permitted to cast governance votes — including `vote_code_hash` (TEE image hash whitelisting). When resharing is cancelled, the contract reverts to the old Running state, and those prospective participants are no longer in the participant set. However, their votes in `tee_state.votes` are not purged.

**Attack scenario:**

1. Resharing is initiated to add a prospective participant F (adversarial).
2. F votes for a malicious TEE image hash H via `vote_code_hash` during resharing.
3. Old participants vote to cancel resharing via `vote_cancel_resharing` (threshold of old participants required).
4. The contract reverts to the old Running state. F is no longer a participant, but F's vote for H remains in `tee_state.votes`.
5. With F's stale vote counting as 1, only `threshold - 1` real participants need to vote for H instead of `threshold`.
6. If the attacker can convince `threshold - 1` real participants to vote for H (or if F was colluding with them), the malicious hash is whitelisted.
7. Nodes running malicious code can now submit attestations referencing H, pass TEE verification, and become participants in the MPC network.
8. Malicious participants can leak key shares or sign unauthorized transactions.

This is an **attestation authorization bypass**: stale votes from non-participants contribute to governance decisions that control which TEE images are trusted, breaking the production safety invariant documented in the codebase itself. [5](#0-4) 

---

### Likelihood Explanation

- Resharing cancellation is a documented, reachable code path used when resharing stalls (e.g., insufficient nodes online).
- A prospective participant can vote for a malicious hash during resharing without any special privilege beyond being a candidate.
- The attacker only needs to be proposed as a prospective participant (which requires threshold approval from current participants, but does not require the attacker to be trusted beyond that).
- The stale vote persists indefinitely — there is no TTL or automatic cleanup.
- The attack requires `threshold - 1` real participants to also vote for the malicious hash, which limits likelihood but does not eliminate it (e.g., social engineering, or the attacker is colluding with `threshold - 1` participants who are below the signing threshold).

**Likelihood: Medium.**

---

### Recommendation

Apply the same cleanup logic in `vote_cancel_resharing` that is applied in `vote_reshared`. When the cancellation transition to Running state occurs, spawn the same set of detached cleanup promises:

```rust
if let Some(new_state) = self.protocol_state.vote_cancel_resharing()? {
    self.protocol_state = new_state;

    // Mirror the cleanup spawned by vote_reshared:
    Promise::new(env::current_account_id())
        .function_call(method_names::REMOVE_NON_PARTICIPANT_UPDATE_VOTES, ...)
        .detach();
    Promise::new(env::current_account_id())
        .function_call(method_names::CLEAN_TEE_STATUS, ...)
        .detach();
    Promise::new(env::current_account_id())
        .function_call(method_names::CLEAN_INVALID_ATTESTATIONS, ...)
        .detach();
    Promise::new(env::current_account_id())
        .function_call(method_names::CLEANUP_ORPHANED_NODE_MIGRATIONS, ...)
        .detach();
    Promise::new(env::current_account_id())
        .function_call(method_names::CLEAN_FOREIGN_CHAIN_DATA, ...)
        .detach();
    Promise::new(env::current_account_id())
        .function_call(method_names::REMOVE_NON_PARTICIPANT_TEE_VERIFIER_VOTES, ...)
        .detach();
}
```

The existing cleanup functions already handle the Running-state guard and are safe to call as detached promises (failures are non-fatal, as noted in the `vote_reshared` comments). [6](#0-5)

### Proof of Concept

1. Contract is in Running state with participants `{A, B, C, D, E}`, threshold = 3.
2. `vote_new_parameters` is called to add prospective participant F → contract enters Resharing state.
3. F calls `vote_code_hash(malicious_hash)` during resharing. F's vote is stored in `tee_state.votes`.
4. Old participants A, B, C call `vote_cancel_resharing` → threshold reached, contract reverts to Running state with `{A, B, C, D, E}`.
5. `tee_state.votes` still contains F's vote for `malicious_hash`. No cleanup was triggered.
6. A calls `vote_code_hash(malicious_hash)` → 2 votes (F's stale + A's).
7. B calls `vote_code_hash(malicious_hash)` → 3 votes → threshold reached → `malicious_hash` is added to `allowed_docker_image_hashes`.
8. A node running malicious code submits an attestation referencing `malicious_hash`, passes `submit_participant_info`, and becomes an attested participant eligible to participate in threshold signing.

The root cause is the asymmetry between `vote_reshared` (which calls cleanup) and `vote_cancel_resharing` (which does not), confirmed at: [2](#0-1) [1](#0-0)

    /// Stale CodeHashesVotes entries from removed participants must not count toward
    /// quorum after resharing.
    ///
    /// Scenario (N=5, T=3):
    /// 1. P1 and P2 vote for malicious hash before resharing.
    /// 2. Resharing removes P1 and P2. New set: {P3, P4, P5}.
    /// 3. clean_non_participant_votes removes stale votes.
    /// 4. P3 votes for the same hash — only 1 vote, not 3.
    #[test]
    fn test_clean_non_participant_votes_removes_stale_votes() {
        // Build 5 participants
        let mut all_participants = Participants::new();
        let mut account_ids = Vec::new();
        for i in 0..5 {
            let (account_id, info) = gen_participant(i);
            account_ids.push(account_id.clone());
            all_participants.insert(account_id, info).unwrap();
        }

        let mut tee_state = TeeState::default();

        // P0 and P1 vote for a malicious hash before resharing
        let malicious_hash = NodeImageHash::from([0xAA; 32]);
        for account_id in &account_ids[0..2] {
            let mut ctx = VMContextBuilder::new();
            ctx.signer_account_id(account_id.clone());
            testing_env!(ctx.build());
            let auth_id = AuthenticatedParticipantId::new(&all_participants).unwrap();
            tee_state.votes.vote(malicious_hash, &auth_id);
        }
        assert_eq!(tee_state.votes.proposal_by_account.len(), 2);

        // Resharing removes P0 and P1. New participant set: {P2, P3, P4}.
        let new_participants = all_participants.subset(2..5);

        // Clean non-participants (as done by CLEAN_TEE_STATUS after resharing)
        tee_state.clean_non_participant_votes(&new_participants);

        // Stale votes must be removed
        assert_eq!(tee_state.votes.proposal_by_account.len(), 0);

        // P2 votes for the same malicious hash — should be only 1 vote, not 3
        let p2_account = &account_ids[2];
        let mut ctx = VMContextBuilder::new();
        ctx.signer_account_id(p2_account.clone());
        testing_env!(ctx.build());
        let auth_id = AuthenticatedParticipantId::new(&new_participants).unwrap();
        let vote_count = tee_state.votes.vote(malicious_hash, &auth_id);
        assert_eq!(vote_count, 1, "Only the fresh vote from P2 should count");
