### Title
Missing Scalar Field Range Check in BLS12-381 Multiexp Host Functions Enables Scalar Malleability - (File: runtime/near-vm-runner/src/logic/bls12381.rs)

### Summary
The `bls12381_g1_multiexp` and `bls12381_g2_multiexp` host functions accept 32-byte scalars from user-supplied contract input without verifying that the scalar is less than the BLS12-381 scalar field order `r`. Because EC scalar multiplication is inherently modular, a scalar `s` and `s + r` produce identical multiexp outputs. This is the direct analog of the reported Plonk verifier bug: a missing scalar field range check that allows non-canonical scalar representations to pass as valid.

### Finding Description

In `runtime/near-vm-runner/src/logic/bls12381.rs`, the `$g_multiexp` macro-generated function (instantiated as both `g1_multiexp` and `g2_multiexp`) reads a raw 32-byte scalar from user-controlled contract input and passes it directly to `blst_p1_mult` / `blst_p2_mult` with `nbits = BLS_SCALAR_SIZE * 8 = 256`:

```rust
let mut pk_mul = blst::$blst_p::default();
unsafe {
    blst::$blst_p_mult(&mut pk_mul, &pk, scalar_data.as_ptr(), BLS_SCALAR_SIZE * 8);
}
``` [1](#0-0) 

There is no check that `scalar_data` encodes a value less than the BLS12-381 scalar field order:

```
r = 0x73eda753299d7d483339d80809a1d80553bda402fffe5bfeffffffff00000001
``` [2](#0-1) 

The `blst_p1_mult` / `blst_p2_mult` functions do not reduce the scalar modulo `r` before computing the multiplication. Because the group order is `r`, the mathematical identity `s·P = (s mod r)·P` holds for any group point `P`, so a scalar `s ≥ r` produces the same output point as `s mod r`. The host function therefore silently accepts non-canonical scalar representations and returns an identical result.

This is in direct contrast to the `alt_bn128_g1_multiexp` host function, which explicitly rejects out-of-range scalars via `bn::Fr::new`:

```rust
fn decode_fr(raw: &[u8; SCALAR_SIZE]) -> Result<bn::Fr, InvalidInput> {
    let val = decode_u256(raw);
    bn::Fr::new(val).ok_or_else(|| InvalidInput::new("invalid fr", raw))
}
``` [3](#0-2) 

The BLS12-381 host functions have no equivalent guard.

### Impact Explanation

Any smart contract deployed on NEAR that uses `bls12381_g1_multiexp` or `bls12381_g2_multiexp` to verify BLS signatures or zero-knowledge proofs (e.g., Groth16, PLONK) and simultaneously uses the raw scalar bytes as a unique identifier for replay protection is vulnerable. An unprivileged attacker who holds a valid proof or signature with scalar `s` can re-submit it with scalar `s + r` (or `s + k·r` for any `k`). The multiexp output is identical, so the cryptographic check passes, but the raw byte representation differs, bypassing any replay-protection map keyed on the scalar bytes. The corrupted protocol value is the smart contract's storage state (the replay-protection set), which incorrectly records the malleable scalar as a new, unseen value and permits the duplicate action.

### Likelihood Explanation

The BLS12-381 host functions are production-enabled and callable by any deployed contract. As NEAR's ecosystem adopts BLS-based signature aggregation and ZK proof verification (the primary use cases for these host functions), contracts that combine multiexp-based verification with scalar-byte uniqueness checks will appear. The attack requires no special privilege: any user who can call such a contract can craft a malleable scalar. The inconsistency with `alt_bn128_g1_multiexp` (which does enforce the range check) makes it likely that contract authors will assume the BLS12-381 functions behave the same way.

### Recommendation

Add a scalar field range check inside the `$g_multiexp` macro body in `runtime/near-vm-runner/src/logic/bls12381.rs`, immediately after extracting `scalar_data`, to reject any scalar `≥ r`. The BLS12-381 scalar field order `r` is the 255-bit value `0x73eda753299d7d483339d80809a1d80553bda402fffe5bfeffffffff00000001` (little-endian in the 32-byte encoding used by the host function). Return `Ok(None)` (indicating invalid input) if the check fails, consistent with how other invalid inputs are handled. This mirrors the existing behavior of `alt_bn128_g1_multiexp`.

### Proof of Concept

1. Take any valid G1 point `P` (in the G1 subgroup).
2. Choose a scalar `s` in `[1, r-1]`.
3. Call `bls12381_g1_multiexp` with `(P, s)` → result `Q`.
4. Compute `s' = s + r` (a 256-bit value, fits in 32 bytes since `r < 2^255`).
5. Call `bls12381_g1_multiexp` with `(P, s')` → result `Q'`.
6. Observe `Q == Q'`: the host function returns the same point for both scalars.
7. A contract that checks `seen_scalars.contains(s_bytes)` will accept both `s` and `s'` as distinct, allowing the same logical operation to execute twice.

The scalar `r` in little-endian bytes is `[0x01, 0x00, 0x00, 0x00, 0xff, 0xff, 0xff, 0xff, 0xfe, 0x5b, 0xfe, 0xff, 0x02, 0xa4, 0xbd, 0x53, 0x05, 0xd8, 0xa1, 0x09, 0x08, 0xd8, 0x39, 0x33, 0x48, 0x7d, 0x9d, 0x29, 0x53, 0xa7, 0xed, 0x73]`, confirming it fits in 32 bytes and that `s + r` also fits for any `s < r`.         pub(crate) fn $g_multiexp(data: &[u8], version: u32) -> Result<Option<Vec<u8>>> {
            const ITEM_SIZE: usize = $BLS_P_SIZE + BLS_SCALAR_SIZE;
            check_input_size(data, ITEM_SIZE, &format!("{}_multiexp", $bls12381_p))?;

            let mut res_pk = blst::$blst_p::default();

            for item_data in data.chunks_exact(ITEM_SIZE) {
                let (point_data, scalar_data) = item_data.split_at($BLS_P_SIZE);
                debug_assert_eq!(scalar_data.len(), BLS_SCALAR_SIZE);

                let pk = match $parse_p(point_data, version) {
                    Some(pk) => pk,
                    None => return Ok(None),
                };

                if unsafe { blst::$blst_p_in_g(&pk) } != true {
                    return Ok(None);
                }

                let mut pk_mul = blst::$blst_p::default();
                unsafe {
                    blst::$blst_p_mult(&mut pk_mul, &pk, scalar_data.as_ptr(), BLS_SCALAR_SIZE * 8);
                }

                unsafe {
                    blst::$blst_p_add_or_double(&mut res_pk, &res_pk, &pk_mul);
                }
            }

            Ok(Some($serialize_p(&res_pk)))
        }
