### Title
Unbounded Iteration Over Delegators in `ConwayUnRegDRep` Certificate Processing - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/GovCert.hs`)

### Summary
The `ConwayUnRegDRep` case in `conwayGovCertTransition` iterates over an unbounded set of staking credentials (`drepDelegs`) when clearing reverse-delegation pointers during DRep unregistration. Because no protocol parameter caps the number of delegators per DRep, a single certificate can force O(n log m) ledger work where n is unbounded, allowing an attacker to craft a transaction whose validation cost grows without limit.

### Finding Description
`DRepState` stores a `Set (Credential Staking)` called `drepDelegs` that records every staking credential currently delegated to the DRep. [1](#0-0) 

There is no protocol parameter in `ConwayPParams` that bounds the size of this set. [2](#0-1) 

When a `ConwayUnRegDRep` certificate is processed, `conwayGovCertTransition` calls `clearDRepDelegations`, which performs a `foldr` over the entire `drepDelegs` set, executing one `Map.adjust` per delegator against the global accounts map: [3](#0-2) 

The inner `Map.adjust` is O(log m) (m = total accounts), so the total cost of one `ConwayUnRegDRep` certificate is **O(n log m)** where n = `|drepDelegs|` is unbounded. No ledger rule, protocol parameter, or transaction-size check constrains n before this loop executes.

### Impact Explanation
**Medium. Attacker-controlled certificates exceed intended validation limits.**

An attacker who controls a DRep can accumulate an arbitrarily large `drepDelegs` set and then submit a single `ConwayUnRegDRep` certificate. Every honest node must execute the full `clearDRepDelegations` loop to validate the block. With a sufficiently large delegator set, per-block validation time grows without bound, degrading node throughput and potentially causing block producers to miss their slots. This is the direct Cardano analog of the `PermissionsLib.revokeAuthorization` unbounded-loop class: a revocation/unregistration operation that must touch every member of an unbounded collection.

### Likelihood Explanation
Moderate. Accumulating a large `drepDelegs` set requires many staking credentials to delegate to the attacker's DRep. Each delegation requires a key-deposit (currently 2 ADA) and a transaction fee, but the deposits are refunded on unregistration. The net cost to the attacker is only transaction fees, making a large-scale accumulation economically feasible over time. Real popular DReps will also naturally accumulate large delegator sets, making the attack surface broader than just adversarial actors.

### Recommendation
Introduce a protocol parameter (e.g., `ppMaxDRepDelegators`) that caps the number of staking credentials that may delegate to a single DRep. Enforce this cap in the `ConwayDeleg` / `ConwayRegDeleg` delegation rules before inserting into `drepDelegs`. Alternatively, remove `drepDelegs` from `DRepState` entirely and derive the reverse-delegation mapping lazily (e.g., at epoch boundary) rather than maintaining it eagerly and clearing it in O(n) during unregistration.

### Proof of Concept

1. Register a DRep credential `D` with `ConwayRegDRep`.
2. Register N staking credentials `s_1 … s_N` and delegate each to `D` via `ConwayDelegCert … (DelegVote (DRepCredential D))`. Each delegation inserts `s_i` into `D`'s `drepDelegs` set: [4](#0-3) 
3. Submit a transaction containing `ConwayUnRegDRep D refund`. The GOVCERT rule executes:
   ```haskell
   clearDRepDelegations (drepDelegs dRepState) accountsMap
   -- = foldr (Map.adjust ...) accountsMap {s_1, …, s_N}
   ``` [5](#0-4) 
4. Every validating node must execute N `Map.adjust` calls. As N grows (no protocol-level ceiling exists), validation time grows linearly, exceeding intended per-transaction validation limits.
