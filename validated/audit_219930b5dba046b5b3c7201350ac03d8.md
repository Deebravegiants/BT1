Audit Report

## Title
Unauthorized Claim on Behalf of Any Address Enables Permanent Freezing of Reward Tokens - (File: `contracts/utils/MerkleDistributor/MerkleDistributor.sol`)

## Summary
`MerkleDistributor.claim()` accepts a caller-supplied `account` parameter and transfers ERC20 reward tokens directly to that address with no check that `msg.sender == account`. Any unprivileged caller can supply a valid merkle proof for any eligible address and force a token transfer to it. If the target `account` is a contract that cannot handle an unexpected ERC20 receipt and has no rescue path, the tokens are permanently frozen and the claim state is irreversibly consumed.

## Finding Description
`MerkleDistributor.claim()` performs merkle proof verification against the tuple `(index, account, cumulativeAmount)` — all of which are public off-chain data — and then unconditionally transfers tokens to the caller-supplied `account`: [1](#0-0) 

No authorization check of the form `msg.sender == account` exists anywhere in the function. After the transfer, the claim state is permanently consumed: [2](#0-1) 

The token transfer then executes to the attacker-supplied address: [3](#0-2) 

The sibling contract `KernelMerkleDistributor._processClaim()` explicitly guards against this with: [4](#0-3) 

`MerkleDistributor` has no equivalent guard. An attacker reads the public merkle tree, identifies a contract address `account` that is an eligible recipient but cannot safely receive unexpected ERC20 transfers (e.g., a multisig, vault, or DeFi integration contract with no token rescue logic), and calls `claim()` for that address. Tokens arrive at the contract unexpectedly, are unhandled, and are permanently locked. The `AlreadyClaimed` revert at line 115-117 then prevents the legitimate owner from ever re-claiming. [5](#0-4) 

## Impact Explanation
**Critical — Permanent freezing of funds.** The reward tokens are transferred to the intended `account` but arrive unexpectedly and without any handler. If the contract has no rescue function, the tokens are permanently locked. The claim state is also consumed, making the loss irreversible. This matches the allowed Critical impact class: "Permanent freezing of funds."

## Likelihood Explanation
**Medium.** The merkle tree data (index, account, cumulativeAmount, proof) is public off-chain data. No special privilege is required. The attacker only needs to identify one contract address in the tree that cannot safely receive unexpected ERC20 tokens — a realistic condition for any DeFi protocol that integrates with `MerkleDistributor` and holds an eligible address. The attack is repeatable for every such address in the tree.

## Recommendation
Add a caller authorization check inside `claim()`, mirroring the fix already present in `KernelMerkleDistributor._processClaim()`:

```solidity
if (account != msg.sender) {
    revert Unauthorized();
}
```

Alternatively, remove the `account` parameter entirely and derive it from `msg.sender`, eliminating the attack surface entirely.

## Proof of Concept
1. Merkle tree contains leaf: `(index=5, account=VaultContract, cumulativeAmount=1000e18)`.
2. `VaultContract` is a deployed contract with no ERC20 rescue logic and no handler for unexpected token receipts.
3. Attacker calls:
   ```solidity
   merkleDistributor.claim(5, address(VaultContract), 1000e18, validProof);
   ```
4. Proof verification passes (public data). `userClaims[VaultContract].lastClaimedIndex = 5` is set. `safeTransfer(VaultContract, amountToSend)` executes.
5. Tokens arrive at `VaultContract` with no handler — permanently frozen.
6. `VaultContract` later calls `claim()` itself — reverts with `AlreadyClaimed`.

**Foundry test plan:** Deploy `MerkleDistributor` with a mock ERC20. Build a merkle tree with a leaf for a mock contract that has no token handling. Call `claim()` from an unrelated EOA with the valid proof. Assert token balance of mock contract increased and that a subsequent call from the mock contract itself reverts with `AlreadyClaimed`.
