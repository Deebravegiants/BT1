### Title
Unrefunded Excess Deposit in `submit_participant_info` for Existing-Participant Re-submissions — (File: `crates/contract/src/lib.rs`)

---

### Summary

`submit_participant_info` is marked `#[payable]` and accepts arbitrary deposits, but its deposit-refund logic is gated behind a condition that evaluates to `false` for existing participants updating their attestation. Any deposit attached in that code path is silently absorbed by the contract with no refund.

---

### Finding Description

In `crates/contract/src/lib.rs`, `submit_participant_info` computes a boolean guard:

```rust
let caller_is_not_participant = self.voter_account().is_err();
let is_new_attestation = matches!(
    attestation_insertion_result,
    ParticipantInsertion::NewlyInsertedParticipant
);

let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... deposit check and refund ...
}
``` [1](#0-0) 

When an **existing participant** (i.e., already in the protocol's participant set) re-submits their attestation (e.g., after a TEE image upgrade), both `is_new_attestation` and `caller_is_not_participant` are `false`, so `attestation_storage_must_be_paid_by_caller` is `false`. The entire deposit-handling block — including the refund — is skipped. Because the function is `#[payable]`, any deposit the caller attached is permanently transferred to the contract with no return path. [2](#0-1) 

The design document confirms that the only production caller is `mpc-node`'s `periodic_attestation_submission` task, which re-submits on a **1-hour cadence** and on attestation-removal events: [3](#0-2) 

The node's submission path goes through `submit_remote_attestation`: [4](#0-3) 

Test helpers consistently attach `NearToken::from_near(1)` (1 NEAR) when calling `submit_participant_info`: [5](#0-4) 

---

### Impact Explanation

**Medium — Balance accounting invariant broken.**

Every time an existing participant re-submits their attestation with any non-zero deposit attached, that deposit is permanently consumed by the contract. Given the 1-hour re-submission cadence, this is a recurring, compounding fund loss for every active MPC node operator. The contract's own README states "Any excess deposit is automatically refunded" as a general invariant for payable methods, but this invariant is violated for the existing-participant re-submission path. [6](#0-5) 

---

### Likelihood Explanation

**Medium.** The `periodic_attestation_submission` task runs hourly in production. If the node attaches any deposit (even 1 yoctoNEAR) on re-submission — which is the pattern shown in all test fixtures — the loss occurs on every cycle. The `#[payable]` annotation on the function makes it structurally expected to accept deposits, and the node has no on-chain signal telling it "attach zero deposit this time."

---

### Recommendation

Add an unconditional refund of any excess deposit at the end of `submit_participant_info`, outside the `if attestation_storage_must_be_paid_by_caller` block. When `attestation_storage_must_be_paid_by_caller` is `false`, the required cost is zero, so the entire attached deposit should be returned:

```rust
// After the if block:
let attached = env::attached_deposit();
if !attestation_storage_must_be_paid_by_caller && attached > NearToken::from_yoctonear(0) {
    Promise::new(account_id).transfer(attached).detach();
}
```

Alternatively, restructure the deposit logic to always compute `cost` (zero when no storage is charged) and always issue the refund for `attached - cost`.

---

### Proof of Concept

1. Contract is in `Running` state with participant `alice.near` already in the participant set and with a stored attestation.
2. `alice.near` calls `submit_participant_info` with a new attestation and `attached_deposit = 1_000_000_000_000_000_000_000_000` yoctoNEAR (1 NEAR).
3. `add_participant` returns `ParticipantInsertion::UpdatedExistingParticipant` → `is_new_attestation = false`.
4. `self.voter_account()` succeeds for `alice.near` → `caller_is_not_participant = false`.
5. `attestation_storage_must_be_paid_by_caller = false || false = false`.
6. The `if` block is skipped entirely. No refund is issued.
7. Alice's 1 NEAR is permanently held by the contract.
8. This repeats every hour via `periodic_attestation_submission`. [7](#0-6)

            // `saturating_sub`: if a re-submission shrinks the entry, charge nothing
            // rather than underflow. Intentional asymmetry: we do not refund freed bytes

                return Err(InvalidParameters::InsufficientDeposit {
                    attached: attached.as_yoctonear(),
                    required: cost.as_yoctonear(),
                }
                .into());
            }

                && diff > NearToken::from_yoctonear(0)

The only caller of `submit_participant_info` in production is `mpc-node`'s `periodic_attestation_submission` task, which resubmits on a 1-hour cadence and on attestation-removal events. It already polls contract state to confirm the attestation is actually stored, with exponential backoff (100 ms → 60 s, capped at 12 h). That polling-based success criterion is what makes the sync→async change transparent. Under yield-resume the returned `Promise` also resolves with the actual outcome — success, a verifier-rejection error, or a post-DCAP-failure error (each as soon as the verifier answers), or a timeout error after ~200 blocks if it never does — so any future caller that wants to await the result synchronously can, without changing the contract.
