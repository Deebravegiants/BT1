### Title
Single Participant Can Freeze All `verify_foreign_transaction` Requests via Empty Chain Registration - (`File: crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction` entry point rejects any request whose target chain is absent from `get_supported_foreign_chains()`. That function returns the **strict intersection** of every active participant's registered chain list. A single participant calling `register_foreign_chain_support` with an empty list collapses the intersection to the empty set, causing every subsequent `verify_foreign_transaction` call to `env::panic_str` and revert — freezing the entire foreign-chain verification feature for all users.

---

### Finding Description

`verify_foreign_transaction` enforces a chain-support gate before enqueuing the yield-resume request:

```rust
// crates/contract/src/lib.rs  lines 533-542
let requested_chain = request.request.chain();
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) {
    env::panic_str(
        &InvalidParameters::ForeignChainNotSupported {
            requested: requested_chain,
        }
        .to_string(),
    );
}
```

The supported-chain set is computed as the strict intersection of every active participant's `SupportedForeignChains` entry stored in `node_foreign_chain_support`. Any participant can overwrite their own entry at will via:

```rust
// crates/contract/src/lib.rs  lines 972-983
pub fn register_foreign_chain_support(
    &mut self,
    foreign_chain_support: dtos::SupportedForeignChains,
) -> Result<(), Error> {
    let account_id = self.voter_or_panic();
    self.node_foreign_chain_support
        .foreign_chain_support_by_node
        .insert(account_id, foreign_chain_support);
    Ok(())
}
```

There is no lower bound on the submitted set. Submitting an empty `SupportedForeignChains` is accepted without error. Because the intersection of any set with the empty set is empty, the very next call to `get_supported_foreign_chains()` returns `{}`, and every chain fails the `contains` check. The project's own design document acknowledges this root cause explicitly:

> *"Today, `get_supported_foreign_chains()` returns the strict intersection of every active participant's registered chains, and `verify_foreign_transaction` rejects any request whose target chain is not in it. A single node that registers an empty list (or hasn't registered yet) drops every chain — one operator can take the whole feature down."*
> — `docs/design/calculating-supported-foreign-chains.md`

The proposed two-set fix (`whitelisted` / `available`) is marked **Status: Proposed** and is not yet reflected in the production contract code; the `node_foreign_chain_support` field carries a `TODO(#3475)` deprecation note but remains the live data source.

---

### Impact Explanation

Every `verify_foreign_transaction` call reverts with `ForeignChainNotSupported` for every chain. Users who have already submitted a foreign-chain transaction and are waiting for MPC verification receive no signature; their yield-resume promise times out. Any funds locked on the foreign chain pending that signature are effectively frozen for the duration of the attack. The freeze persists until the malicious participant re-registers a non-empty list or a resharing removes them — both of which require governance action that takes time. This matches the **Medium** allowed impact: *request-lifecycle manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration*.

---

### Likelihood Explanation

The attack requires only one participant — strictly below the signing threshold — to call a single, publicly documented contract method with an empty argument. No cryptographic material, no collusion, no special tooling. The call is cheap (one NEAR transaction) and can be repeated after any recovery attempt. The attacker's identity is known on-chain but the damage is immediate and the recovery path (resharing) is slow.

---

### Recommendation

1. **Reject empty registrations**: add a guard in `register_foreign_chain_support` that panics if the submitted `SupportedForeignChains` set is empty.
2. **Adopt the threshold-based availability rule** described in `docs/design/calculating-supported-foreign-chains.md`: a chain is available iff ≥ `signing_threshold` active participants cover it, so no single participant can collapse the set.
3. **Accelerate the deprecation** of `node_foreign_chain_support` (tracked as `TODO(#3475)`) in favour of the new `ForeignChainsMetadata` path, which already implements the threshold-counting logic in `update_available_chains_config_cache`.

---

### Proof of Concept

1. Attacker is participant `alice.near` in a 5-of-7 network.
2. Alice calls:
   ```
   register_foreign_chain_support({ chains: [] })
   ```
   Cost: one NEAR transaction, no deposit required.
3. `node_foreign_chain_support` now contains `alice.near → {}`.
4. `get_supported_foreign_chains()` computes the intersection across all 7 participants; the intersection with `{}` is `{}`.
5. Any user calls `verify_foreign_transaction({ chain: "ethereum", ... })`.
6. Contract evaluates `supported_chains.contains("ethereum")` → `false` → `env::panic_str("ForeignChainNotSupported { requested: ethereum }")`.
7. All `verify_foreign_transaction` requests for every chain revert. Pending yield-resume promises time out. Foreign-chain-locked funds are frozen until governance intervenes.

    pub(crate) fn update_available_chains_config_cache(
        &mut self,
        active_tls_keys: &BTreeSet<dtos::Ed25519PublicKey>,
        threshold: u64,
    ) {
        let mut chain_to_supporter_count: std::collections::BTreeMap<dtos::ForeignChain, u64> =
            std::collections::BTreeMap::new();
        for tls_key in active_tls_keys {
            let Some(chains) = self.foreign_chains_configs.get(tls_key) else {
                continue;
            };
            for chain in chains.iter() {
                if self.rpc_whitelist.entries.is_whitelisted(chain) {
                    let count = chain_to_supporter_count.entry(*chain).or_default();
                    *count = count
                        .checked_add(1)
                        .expect("supporter count bounded by participant set size");
                }
            }
        }
        self.available_foreign_chains = chain_to_supporter_count
            .into_iter()
            .filter_map(|(chain, count)| (count >= threshold).then_some(chain))
            .collect::<BTreeSet<_>>()
            .into();
    }
