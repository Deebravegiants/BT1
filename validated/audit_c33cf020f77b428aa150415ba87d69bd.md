Audit Report

## Title
Raw `balanceOf` in `getAssetDistributionData` allows donation-based deposit-limit DoS and rsETH price inflation — (`contracts/LRTDepositPool.sol`)

## Summary
`LRTDepositPool.getAssetDistributionData()` measures the pool's asset balance via a raw `IERC20(asset).balanceOf(address(this))` call with no internal accounting variable. Any unprivileged actor can donate ERC-20 tokens directly to the pool address, inflating `getTotalAssetDeposits`, which causes `_checkIfDepositAmountExceedesCurrentLimit` to revert all subsequent deposits with `MaximumDepositLimitReached`. The same inflated total propagates into `LRTOracle._getTotalEthInProtocol()`, allowing the public `updateRSETHPrice()` to set an artificially high rsETH price, causing new depositors to receive fewer rsETH tokens than they are entitled to.

## Finding Description

**Root cause — raw balance read:** [1](#0-0) 

`getAssetDistributionData` reads `IERC20(asset).balanceOf(address(this))` directly. There is no internal accounting variable that tracks only legitimately deposited tokens. Any direct ERC-20 transfer to the contract address inflates this value.

**Inflated balance propagates through `getTotalAssetDeposits`:** [2](#0-1) 

The donated amount is included in the sum returned by `getTotalAssetDeposits`.

**DoS path — `_checkIfDepositAmountExceedesCurrentLimit` reverts all deposits:** [3](#0-2) 

For ERC-20 assets, if `totalAssetDeposits + amount > depositLimitByAsset`, the function returns `true`. Once a donation pushes `totalAssetDeposits` to or above the limit, every subsequent `depositAsset` call (even for 1 wei) triggers `MaximumDepositLimitReached` via `_beforeDeposit`: [4](#0-3) 

**Price manipulation path — public `updateRSETHPrice` uses the same inflated total:** [5](#0-4) [6](#0-5) [7](#0-6) 

The inflated `totalETHInProtocol` raises `newRsETHPrice`, causing `getRsETHAmountToMint` to divide by a higher price and mint fewer rsETH tokens to new depositors: [8](#0-7) 

**Existing guard is insufficient for the price path:** The `pricePercentageLimit` check at lines 256–266 only blocks price increases above the configured threshold. When `pricePercentageLimit == 0` (unset/default), the guard is entirely bypassed. Even when set, a donation sized to stay within the threshold manipulates the price silently. There is no guard at all for the DoS path.

## Impact Explanation

**Medium — Temporary freezing of funds:** An attacker donates a small amount of the target ERC-20 token (exactly enough to push `totalAssetDeposits` over `depositLimitByAsset`) and all `depositAsset` calls for that asset revert with `MaximumDepositLimitReached`. The freeze persists until an admin raises the deposit limit. The attacker's cost is bounded by the remaining deposit capacity, which may be trivially small near the limit.

**Low — Contract fails to deliver promised returns:** By donating tokens and calling the permissionless `updateRSETHPrice()`, the attacker inflates the rsETH price. New depositors receive fewer rsETH tokens than the true exchange rate warrants. Existing holders are unaffected; only new depositors are shortchanged.

## Likelihood Explanation

The DoS attack requires only a standard ERC-20 `transfer` to the `LRTDepositPool` address — no approval from the protocol, no flash loan, no privileged role. The exact donation amount needed is trivially computable from on-chain state (`depositLimitByAsset - getTotalAssetDeposits`). The attack is cheap, repeatable after each admin limit increase, and can be executed by any EOA. Likelihood is **high** for the DoS vector. The price manipulation vector is **medium** likelihood, constrained by `pricePercentageLimit` when configured.

## Recommendation

Replace the raw `balanceOf` read with an internal accounting variable incremented only through controlled deposit and transfer paths:

```solidity
mapping(address => uint256) internal _assetBalance;

// In depositAsset (after safeTransferFrom):
_assetBalance[asset] += depositAmount;

// In transferAssetToNodeDelegator (after transfer out):
_assetBalance[asset] -= amount;

// In getAssetDistributionData:
assetLyingInDepositPool = _assetBalance[asset]; // not balanceOf
```

Apply the same pattern to ETH, replacing `address(this).balance` with a tracked variable updated in `depositETH`, `receiveFromRewardReceiver`, `receiveFromLRTConverter`, and `receiveFromNodeDelegator`.

## Proof of Concept

```
// DoS vector
1. stETH deposit limit = 100_000 ether; current getTotalAssetDeposits(stETH) = 99_999 ether.
2. Attacker calls stETH.transfer(address(lrtDepositPool), 2 ether).
   - No protocol approval needed; direct ERC-20 transfer.
3. getAssetDistributionData() now returns assetLyingInDepositPool inflated by 2 ether.
4. getTotalAssetDeposits(stETH) = 100_001 ether > 100_000 ether limit.
5. Any user calling depositAsset(stETH, amount, ...) hits:
   _checkIfDepositAmountExceedesCurrentLimit: 100_001 + amount > 100_000 => true
   => revert MaximumDepositLimitReached.

// Price manipulation vector (when pricePercentageLimit == 0)
6. Attacker calls stETH.transfer(address(lrtDepositPool), X ether).
7. Attacker calls lrtOracle.updateRSETHPrice().
   - _getTotalEthInProtocol() includes the donated X ether.
   - newRsETHPrice = (inflated totalETHInProtocol) / rsethSupply > true price.
   - rsETHPrice is set above true backing.
8. Next depositor calling depositAsset receives:
   rsethAmountToMint = (amount * assetPrice) / inflatedRsETHPrice < fair amount.
```
