### Title
Cross-Domain Replay of `deploy_token` MPC Signature Across All Bridge Chains — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/omni_bridge.cairo`, `aptos/sources/omni_bridge.move`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The `MetadataPayload` borsh encoding used to verify the NEAR MPC signature in every chain's `deploy_token` entry point contains no chain identifier and no contract/program address. An attacker who observes a valid `deploy_token` signature on any one chain can replay it verbatim on every other supported chain, deploying bridged tokens on chains the NEAR MPC never authorized.

---

### Finding Description

Every chain's `deploy_token` function constructs the signed message as:

```
PayloadType::Metadata (1 byte) | token (borsh string) | name | symbol | decimals
```

**EVM** (`OmniBridge.sol`, `deployToken()`):
```solidity

    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) revert InvalidSignature();

None of these encodings include a chain ID, contract address, or program ID. The resulting keccak256 hash is **byte-for-byte identical** across all four chains for the same NEAR token ID.

**Contrast with `fin_transfer`**, which correctly binds the chain by interleaving `omniBridgeChainId` as the OmniAddress tag before both `tokenAddress` and `recipient`: [7](#0-6) 

The `deploy_token` path has no equivalent binding.

---

### Impact Explanation

An attacker who observes a valid `deploy_token` signature on chain A (e.g., Ethereum) can submit it to chain B (e.g., Starknet, Aptos, or Solana) before the legitimate relayer does. The signature passes verification on chain B because the hash is identical.

Consequences:
1. **Unauthorized token deployment**: A bridged token is deployed on a chain the NEAR MPC never authorized for that token. The NEAR side may not have registered the token for that chain, so it will not sign `fin_transfer` payloads for it.
2. **Irreversible fund lock**: Users who call `init_transfer` on chain B with the replay-deployed token lock their assets in the bridge vault. Because the NEAR side does not recognize the token on chain B, no `fin_transfer` signature is ever produced, and the funds are permanently unclaimable.
3. **Cross-instance replay**: If multiple bridge deployments exist on the same chain (e.g., mainnet and a staging deployment), a signature from one is valid on the other, since the contract address is also absent from the hash.

---

### Likelihood Explanation

- The attack requires no privileged access. `deploy_token` is a fully permissionless entry point on all chains.
- The attacker only needs to observe a valid signature on any public chain (e.g., from a mempool or finalized transaction) and submit it to another chain.
- Front-running is straightforward: the attacker monitors the mempool or finalized blocks of chain A and races to submit the same calldata to chain B.
- The same NEAR token ID (e.g., `wrap.near`) is deployed across all chains with identical metadata, so the signed bytes are always identical.

---

### Recommendation

Bind the `MetadataPayload` hash to the destination chain and contract/program address, mirroring the pattern already used in `fin_transfer`. For example, prepend `chain_id` and `contract_address` to the borsh-encoded bytes before hashing:

```
keccak256(chain_id (1 byte) | contract_address (20/32 bytes) | PayloadType::Metadata | token | name | symbol | decimals)
```

This must be applied consistently across EVM (`OmniBridge.sol`), Starknet (`bridge_types.cairo`), Aptos (`bridge_types.move`), and Solana (`deploy_token.rs`), and the NEAR MPC signing logic must include the same fields when producing the signature.

---

### Proof of Concept

1. NEAR MPC signs `MetadataPayload { token: "wrap.near", name: "Wrapped NEAR", symbol: "wNEAR", decimals: 24 }` for deployment on Ethereum. The signature `sig` is broadcast in a transaction to `OmniBridge` on Ethereum.

2. Attacker observes `sig` from the Ethereum mempool or a finalized block.

3. Attacker calls `deploy_token(sig, payload)` on the Starknet `OmniBridge` contract with the identical payload. The Starknet contract computes `keccak256(borsh(payload))` — identical bytes, identical hash — and `verify_eth_signature` passes.

4. `wNEAR` is now deployed on Starknet without NEAR MPC authorization for Starknet. The NEAR side has no record of this Starknet token address.

5. A user calls `init_transfer` on Starknet with the replay-deployed `wNEAR`, locking their tokens. The NEAR side never produces a `fin_transfer` signature for this token on Starknet. The user's funds are permanently locked.
