Audit Report

## Title
Protocol deposit freeze when `rsETHPrice` is set to zero — (`contracts/LRTOracle.sol` / `contracts/LRTDepositPool.sol`)

## Summary

`LRTOracle._updateRsETHPrice()` can write `rsETHPrice = 0` when `totalETHInProtocol` is zero while `rsethSupply` is non-zero. The downside-protection guard that would abort this update is permanently disabled because `pricePercentageLimit` is never initialised in `initialize()` and defaults to `0`. Once `rsETHPrice == 0`, every call to `LRTDepositPool.getRsETHAmountToMint()` panics with a division-by-zero, freezing all new deposits until the state is manually corrected.

## Finding Description

**Root cause 1 — `pricePercentageLimit` is never initialised.**

`LRTOracle.initialize()` only sets `lrtConfig`; `pricePercentageLimit` is left at its Solidity default of `0`: [1](#0-0) 

**Root cause 2 — Downside-protection guard is permanently disabled.**

`_updateRsETHPrice()` evaluates the guard as:

```solidity
bool isPriceDecreaseOffLimit =
    pricePercentageLimit > 0 && diff > pricePercentageLimit.mulWad(highestRsethPrice);
``` [2](#0-1) 

Because `pricePercentageLimit == 0`, the short-circuit makes `isPriceDecreaseOffLimit` always `false`. The `return` that would abort the price update is never reached: [3](#0-2) 

**Root cause 3 — Zero price is written unconditionally.**

When `totalETHInProtocol == 0` and `rsethSupply > 0`, `protocolFeeInETH` is also `0` (the TVL-increase condition on line 244 is false), so:

```solidity
uint256 newRsETHPrice = (0 - 0).divWad(rsethSupply);  // = 0
``` [4](#0-3) 

With the guard bypassed, execution reaches:

```solidity
rsETHPrice = newRsETHPrice;  // = 0
``` [5](#0-4) 

**Root cause 4 — `getRsETHAmountToMint()` has no zero-divisor guard.**

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [6](#0-5) 

With `rsETHPrice == 0` this is a Solidity 0.8 division-by-zero panic. Every call to `depositETH()` and `depositAsset()` routes through `_beforeDeposit()` → `getRsETHAmountToMint()` and reverts: [7](#0-6) 

**Trigger is permissionless.**

`updateRSETHPrice()` is `public` with no role restriction: [8](#0-7) 

Any address can call it the moment `totalETHInProtocol` reaches zero, racing ahead of any admin response.

## Impact Explanation

**Medium — Temporary freezing of funds.**

All new deposits (`depositETH`, `depositAsset`) are blocked for every user until an admin either (a) calls `setPricePercentageLimit()` with a non-zero value and then `updateRSETHPriceAsManager()` after restoring TVL, or (b) directly injects assets to bring `totalETHInProtocol > 0` and re-runs the price update. Existing rsETH holders can still queue withdrawals (the withdrawal path multiplies by `rsETHPrice`, not divides), so funds are not permanently lost — matching the "Medium: Temporary freezing of funds" allowed impact.

## Likelihood Explanation

Two conditions must coincide:

1. `pricePercentageLimit == 0` — this is the **default state** of every deployment where the admin has not explicitly called `setPricePercentageLimit()`. No special attacker capability is required.
2. `totalETHInProtocol == 0` while `rsethSupply > 0` — requires all liquid and staked balances to be simultaneously zero. With EigenLayer slashing enabled post-ELIP-002, a severe slashing event across all supported assets is a realistic (if low-probability) tail risk. The attacker does not need to cause the slashing; they only need to call the permissionless `updateRSETHPrice()` once the condition is met.

Combined likelihood: **Low-Medium**.

## Recommendation

1. **Guard against zero price in `_updateRsETHPrice()`** — add before the final assignment:
   ```solidity
   if (newRsETHPrice == 0) revert ZeroRsETHPrice();
   ```
2. **Initialise `pricePercentageLimit` in `initialize()`** — set a sensible non-zero default (e.g., `1e16` = 1%) so the downside-protection branch is always active from deployment.
3. **Guard the divisor in `getRsETHAmountToMint()`** — add:
   ```solidity
   if (lrtOracle.rsETHPrice() == 0) revert ZeroRsETHPrice();
   ```
   before the division on line 520 of `LRTDepositPool.sol`.

## Proof of Concept

```
Initial state:
  rsethSupply          = 1_000 ether   (users hold rsETH)
  totalETHInProtocol   = 0             (all assets slashed / drained)
  pricePercentageLimit = 0             (never set — default)
  rsETHPrice (stored)  = 1 ether       (last valid price)

1. Attacker calls LRTOracle.updateRSETHPrice()  [public, no role check]
2. rsethSupply > 0  → early-return at line 218 skipped
3. totalETHInProtocol == 0, protocolFeeInETH == 0
   → newRsETHPrice = (0).divWad(1_000e18) = 0
4. newRsETHPrice < highestRsethPrice  → enters downside-protection block
   isPriceDecreaseOffLimit = (0 > 0) && ... = false  → no pause, no return
5. rsETHPrice = 0  ← stored on-chain

6. Any user calls LRTDepositPool.depositETH{value: 1 ether}(0, "")
7. _beforeDeposit → getRsETHAmountToMint:
     (1e18 * assetPrice) / rsETHPrice
   = (1e18 * assetPrice) / 0
   → Panic: division by zero  ← REVERT

All deposits are now frozen until admin intervention.
```
