### Title
Missing `RobustEcdsaTaskId::KeyResharing` in Resharing Network Multiplexer Causes Permanent Key-Resharing Failure for Robust ECDSA Domains — (File: `crates/node/src/coordinator.rs`)

---

### Summary

The network-channel multiplexer in `coordinator.rs` classifies incoming P2P channels as either "resharing" or "running" before dispatching them. The predicate that identifies resharing messages omits `RobustEcdsaTaskId::KeyResharing`, so Robust ECDSA resharing channels are silently routed to the running MPC client, which explicitly rejects them. A compounding omission in `resharing_follower` means the channels would be ignored even if they reached the correct handler. The net effect is that Robust ECDSA key resharing is permanently broken for all follower nodes: the leader initiates the protocol, no follower ever participates, and the resharing never reaches threshold.

---

### Finding Description

**Root cause — coordinator.rs multiplexer (lines 470–474)**

```rust
let is_resharing_message = matches!(
    network_channel.task_id(),
    MpcTaskId::EcdsaTaskId(EcdsaTaskId::KeyResharing { .. })
        | MpcTaskId::EddsaTaskId(EddsaTaskId::KeyResharing { .. })
);
``` [1](#0-0) 

`RobustEcdsaTaskId::KeyResharing` is a fully defined variant of `RobustEcdsaTaskId`: [2](#0-1) 

Because the `matches!` predicate does not include `MpcTaskId::RobustEcdsaTaskId(RobustEcdsaTaskId::KeyResharing { .. })`, every Robust ECDSA resharing channel is sent to `running_sender` instead of `resharing_sender`.

**First rejection — RobustEcdsaSignatureProvider::process_channel (lines 199–200)**

The running MPC client dispatches the channel to `RobustEcdsaSignatureProvider::process_channel`, which explicitly bails:

```rust
RobustEcdsaTaskId::KeyResharing { .. } => {
    anyhow::bail!("Key resharing rejected in normal node operation");
}
``` [3](#0-2) 

**Second omission — resharing_follower in key_events.rs (lines 641–651)**

Even if the channel were correctly routed to `resharing_network_receiver`, the `resharing_follower` function only handles `EcdsaTaskId::KeyResharing` and `EddsaTaskId::KeyResharing`; it silently ignores everything else: [4](#0-3) 

Both omissions are structurally identical to the original report's pattern: a new message type (`RobustEcdsaTaskId::KeyResharing`) was added to the type system but was never registered/handled in the dispatch layer.

---

### Impact Explanation

`RobustEcdsaSignatureProvider` implements `run_key_resharing_client`, meaning Robust ECDSA domains are intended to support key resharing (participant-set rotation and key refresh). Because follower nodes always reject or ignore the resharing channel, the protocol can never reach the required threshold of participants. This means:

1. **Participant-set updates are impossible** for any Robust ECDSA domain — a failed or compromised node cannot be removed.
2. **If the live participant set drops below the signing threshold** (e.g., due to node failures), the Robust ECDSA domain's key shares become permanently inaccessible, freezing any funds or signing capability tied to that domain.

This maps to the allowed Critical impact: *permanent freezing of funds controlled by the MPC network*.

---

### Likelihood Explanation

Key resharing is triggered by governance/operator action, not by an unprivileged caller. However, the bug is deterministic: **every** Robust ECDSA resharing attempt will fail, regardless of who initiates it or how many times it is retried. No adversarial input is required to trigger the failure; normal operational events (node rotation, key refresh) are sufficient. The `RobustEcdsaTaskId::KeyResharing` variant and `run_key_resharing_client` implementation confirm the feature is intended for production use.

---

### Recommendation

1. **Fix the multiplexer predicate** in `coordinator.rs` to include the missing variant:

```rust
let is_resharing_message = matches!(
    network_channel.task_id(),
    MpcTaskId::EcdsaTaskId(EcdsaTaskId::KeyResharing { .. })
        | MpcTaskId::EddsaTaskId(EddsaTaskId::KeyResharing { .. })
        | MpcTaskId::RobustEcdsaTaskId(RobustEcdsaTaskId::KeyResharing { .. })
);
```

2. **Fix `resharing_follower`** in `key_events.rs` to handle `RobustEcdsaTaskId::KeyResharing` and dispatch it to the appropriate resharing computation.

3. **Add an exhaustive match or compile-time guard** on `MpcTaskId` in the multiplexer so that adding a new task type forces an explicit routing decision, preventing the same class of omission in the future.

---

### Proof of Concept

1. Deploy a NEAR MPC cluster with at least one Robust ECDSA domain.
2. Trigger a key resharing event (e.g., via the contract's governance call to initiate resharing).
3. The leader node creates a `NetworkTaskChannel` with `task_id = MpcTaskId::RobustEcdsaTaskId(RobustEcdsaTaskId::KeyResharing { key_event })` and broadcasts a `MpcMessageKind::Start` to all followers.
4. On each follower, the multiplexer evaluates `is_resharing_message` — the predicate returns `false` because `RobustEcdsaTaskId::KeyResharing` is absent.
5. The channel is sent to `running_sender`; `process_channel_task` dispatches it to `RobustEcdsaSignatureProvider::process_channel`, which returns `Err("Key resharing rejected in normal node operation")`.
6. No follower ever participates; the leader's resharing protocol times out without reaching threshold.
7. The Robust ECDSA domain's key shares remain bound to the old participant set indefinitely.
