### Title
Pending Signature Requests Permanently Stranded When `accept_requests` Is Set to `false` After Submission — (`crates/contract/src/lib.rs`)

---

### Summary

The `respond` and `respond_ckd` functions gate response delivery on `self.accept_requests`. When `verify_tee` sets `accept_requests = false` due to TEE validation failure, MPC nodes are blocked from delivering responses to requests that were already queued before the flag changed. Those yield promises will time out and fail even though the MPC network computed valid signatures, breaking the request-lifecycle accounting invariant.

---

### Finding Description

**Root cause — the `accept_requests` flag gates both new submissions and existing-request resolution:**

`sign` creates a NEAR yield promise and stores the request in `pending_signature_requests`: [1](#0-0) 

`check_request_preconditions` correctly gates new submissions on `accept_requests`: [2](#0-1) 

However, `respond` also checks `accept_requests` before resolving an already-queued yield: [3](#0-2) 

And `respond_ckd` does the same: [4](#0-3) 

`verify_tee` sets `accept_requests = false` when fewer than threshold participants have valid TEE attestations: [5](#0-4) 

**Exploit flow:**

1. User calls `sign(...)` → yield promise is created, request is stored in `pending_signature_requests`, user's NEAR call is suspended awaiting the resume.
2. A participant (or the TEE expiry timer) triggers `verify_tee` → TEE validation fails → `accept_requests = false`.
3. MPC nodes finish computing the signature and call `respond(request, response)`.
4. `respond` returns `Err(TeeError::TeeValidationFailed)` at line 579–581 — the response is rejected.
5. The MPC nodes may retry, but `accept_requests` remains `false` until `verify_tee` is called again with sufficient valid attestations.
6. The NEAR yield promise eventually times out; the callback `return_signature_and_clean_state_on_success` is invoked with an error, the pending entry is cleaned up, and the user receives a failure — even though a valid threshold signature was produced.

**Analog to M-07:** In the Yieldy report, a "once per cycle" guard (`currentCycleIndex > lastTokeCycleIndex`) blocked the batch-withdrawal processing for requests submitted after the batch had already been sent, creating a discrepancy between recorded withdrawals and actually processed ones. Here, the `accept_requests = false` guard blocks `respond` from resolving requests that were already recorded in `pending_signature_requests` before the flag changed — the same pattern of a request being durably queued but then made unresolvable by a state-transition condition that was not meant to apply to in-flight work.

---

### Impact Explanation

**Medium.** The `pending_signature_requests` map records a live yield promise for the user's request. The MPC network expends threshold computation to produce a valid signature. The `accept_requests` check in `respond` prevents that signature from ever being delivered. The yield promise expires, the user's call fails, and the 1 yoctoNEAR deposit is consumed — all while the contract's own accounting (a pending entry exists) is inconsistent with the outcome (no response was ever delivered). This breaks the production safety invariant that every queued request with a computed response will be resolved.

---

### Likelihood Explanation

`verify_tee` can be called by any current participant at any time: [6](#0-5) 

TEE attestations expire naturally (configurable `tee_upgrade_deadline_duration_seconds`). Any window where attestations expire while sign requests are in-flight triggers this race without any adversarial action. A Byzantine participant below the signing threshold can also call `verify_tee` strategically to force `accept_requests = false` at a moment when requests are pending, deliberately stranding them.

---

### Recommendation

Remove the `accept_requests` guard from `respond` and `respond_ckd`. The flag's purpose is to prevent **new** requests from entering the system; it should not block MPC nodes from resolving requests that were already accepted and queued. The guard in `check_request_preconditions` is the correct and sufficient enforcement point for new submissions.

### Proof of Concept

1. Deploy contract with 3 participants, threshold 2, one CaitSith domain.
2. User calls `sign(payload, path)` with 1 yoctoNEAR deposit → yield promise `Y` is created, `pending_signature_requests[R] = [Y]`.
3. One participant's TEE attestation expires. Another participant calls `verify_tee` → `accept_requests = false`.
4. MPC nodes complete threshold signing and call `respond(R, sig)`.
5. `respond` returns `TeeError::TeeValidationFailed` at line 579–581; the response is dropped.
6. NEAR yield timeout fires → `return_signature_and_clean_state_on_success(R, Err(...))` is called → `pending_signature_requests` entry is removed → user receives a failure receipt.
7. `verify_tee` is called again with valid attestations → `accept_requests = true` — but the request is gone; the user must resubmit and pay again. [7](#0-6) [8](#0-7) [9](#0-8)

    /// Returns `false` and stops the contract from accepting new signature requests or responses,
    /// in case less than `threshold` participants run in an accepted TEE State.
    #[handle_result]
    pub fn verify_tee(&mut self) -> Result<bool, Error> {
        log!("verify_tee: signer={}", env::signer_account_id());
        // Caller must be a participant (node or operator).
        self.voter_or_panic();
        let ProtocolContractState::Running(running_state) = &mut self.protocol_state else {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        };
        let current_params = running_state.parameters.clone();

        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);

        match self.tee_state.reverify_and_cleanup_participants(
            current_params.participants(),
            tee_upgrade_deadline_duration,
        ) {
            TeeValidationResult::Full => {
                self.accept_requests = true;
                log!("All participants have an accepted Tee status");
                Ok(true)
            }
            TeeValidationResult::Partial {
                participants_with_valid_attestation,
            } => {
                let remaining = participants_with_valid_attestation.len();
                // Defense in depth: the surviving participant set must keep the full
                // threshold relation intact — the GovernanceThreshold must still sit
                // within its bounds for the smaller set (in particular it must not
                // exceed the remaining participant count or the upper cap) and must
                // remain at least every domain's ReconstructionThreshold (the kickout
                // keeps the existing per-domain thresholds). Otherwise we refuse and
                // wait for manual intervention.
                let max_reconstruction_threshold =
                    max_reconstruction_threshold(running_state.domains.domains());
                if let Err(err) = ThresholdParameters::validate_governance_against_reconstruction(
                    u64::try_from(remaining).expect("participant count fits in u64"),
                    current_params.threshold(),
                    max_reconstruction_threshold,
                ) {
                    log!(
                        "Kicking out participants with an invalid TEE status would break the threshold relation ({:?}); {} participants remain with a valid TEE status. This requires manual intervention. We will not accept new signature requests as a safety precaution.",
                        err,
                        remaining,
                    );
                    self.accept_requests = false;
                    return Ok(false);
