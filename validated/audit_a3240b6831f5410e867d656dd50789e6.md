### Title
Missing Deposit-Refund Validation Across Sub-Transactions Allows Double-Claiming of Deposits - (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxCert.hs`)

### Summary

In the Dijkstra era, `dijkstraTotalRefundsTxCerts` sums up deposit-refund amounts directly from certificate fields without validating them against the ledger state. The `getTotalRefundsTxCerts` implementation discards the deposit-lookup functions entirely. Combined with the absence of a cross-sub-transaction deduplication check in the `SUBCERTS`/`SUBENTITIES` rules, an attacker can include the same `UnRegDepositTxCert` (or `UnRegDRepTxCert`) in multiple sub-transactions of a single outer transaction and claim the same deposit refund more than once. The developers explicitly acknowledge this gap with a disabled test whose expected predicate failure is marked `error "TODO: predicate failure not yet implemented"`.

### Finding Description

**Root cause 1 – `dijkstraTotalRefundsTxCerts` ignores ledger-state lookup:**

In every prior era, `getTotalRefundsTxCerts` receives two lookup functions (`lookupStakingDeposit`, `lookupDRepDeposit`) that resolve the *actual* deposit stored in the ledger state for each credential. The Dijkstra instance discards both:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxCert.hs  line 285
getTotalRefundsTxCerts _ _ _ = dijkstraTotalRefundsTxCerts
```

`dijkstraTotalRefundsTxCerts` simply folds over the certificate list and sums whatever `Coin` value the transaction author wrote into each `UnRegDepositTxCert`/`UnRegDRepTxCert` field:

```haskell
-- lines 229-238
dijkstraTotalRefundsTxCerts = foldMap' $ \case
  UnRegDepositTxCert _ deposit -> deposit
  UnRegDRepTxCert  _ deposit -> deposit
  _ -> zero
```

The inline comment claims "ledger rules will validate that they are accurate," but the validation for the multi-sub-transaction case is explicitly unimplemented.

**Root cause 2 – `SUBCERTS`/`SUBENTITIES` rules process sub-transactions sequentially without cross-sub-transaction refund deduplication:**

`dijkstraSubEntitiesTransition` in `SubEntities.hs` processes each sub-transaction's certificates through `SUBCERTS` independently. After the first sub-transaction unregisters a credential and claims its deposit, the ledger state passed to the second sub-transaction no longer contains that credential. The Conway `DELEG` rule's `checkInvalidRefund` guard short-circuits on `mAccountState = Nothing` (credential not found), so the second unregistration may proceed without triggering a predicate failure.

**Root cause 3 – Acknowledged unimplemented check:**

The test suite contains a disabled (`xit`) test that explicitly names this scenario and marks the expected predicate failure as unimplemented:

```haskell
-- eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs  lines 53-75
xit "Multiple subtransactions cannot get the same refund" $ do
  ...
  submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
```

The use of `submitFailingTx` (not `submitTx_`) confirms the developers intend this to be rejected, but the rejection logic does not yet exist.

### Impact Explanation

If both sub-transactions are accepted, the outer transaction's `depositChange` (computed via `certsTotalRefundsTxBody → getTotalRefundsTxCerts → dijkstraTotalRefundsTxCerts`) reflects twice the actual deposit. `updateUTxOState` then subtracts this inflated value from `utxosDeposited`:

```haskell
-- eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/Utxo.hs  line 635
utxosDeposited = utxosDeposited <> depositChange
```

The deposit pot is reduced by twice the deposited amount while only one deposit was ever made, constituting a direct creation of ADA from the deposit pot — matching the **Critical** impact class: *Direct loss, creation, or destruction of ADA or native assets through an invalid ledger state transition*.

### Likelihood Explanation

The attacker entry path is fully unprivileged: any transaction author can craft an outer transaction containing two sub-transactions that each carry an `UnRegDepositTxCert` for the same credential. No special keys, governance majority, or privileged access is required. The only prerequisite is that the Dijkstra era is active on-chain. The attack is deterministic and reproducible.

### Recommendation

1. **Implement the missing predicate failure** for the case where multiple sub-transactions attempt to claim the same deposit refund. Track which credentials have already been unregistered within the current outer-transaction batch and reject any subsequent unregistration of the same credential.

2. **Restore deposit-lookup validation in `getTotalRefundsTxCerts`** for the Dijkstra era. Rather than discarding the lookup functions, use them to verify that the `deposit` field in each `UnRegDepositTxCert`/`UnRegDRepTxCert` matches the value stored in the ledger state, consistent with the Conway-era approach in `conwayTotalRefundsTxCerts`.

3. **Enable and fix the pending test** `"Multiple subtransactions cannot get the same refund"` in `Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs`.

### Proof of Concept

```
1. Register staking credential C with deposit D (keyDeposit).
2. Construct subTx1 = { certs: [UnRegDepositTxCert C D], inputs: [utxo1] }
3. Construct subTx2 = { certs: [UnRegDepositTxCert C D], inputs: [utxo2] }
4. Construct outerTx = { subTransactions: [subTx1, subTx2],
                         outputs: [addr ↦ value1 + value2 + 2*D] }
5. Submit outerTx.

Expected (correct) behaviour : rejected with a predicate failure.
Actual (current) behaviour   : accepted; utxosDeposited decreases by 2*D
                                while only D was ever deposited, creating D

dijkstraSubEntitiesTransition = do
  TRC (subCertsEnv, certState, certificates) <- judgmentContext
  let tx = certsTx subCertsEnv
      pp = certsPParams subCertsEnv
      curEpoch = certsCurrentEpoch subCertsEnv
      withdrawals = tx ^. bodyTxL . withdrawalsTxBodyL
      accounts = certState ^. certDStateL . accountsL

  runTest $ Conway.validateWithdrawalsDelegated accounts tx

  network <- liftSTS $ asks networkId
  let (missingWithdrawals, exceededWithdrawals) =
        case withdrawalsThatExceedAccountBalance withdrawals network accounts of
          Nothing -> (Map.empty, Map.empty)
          Just (missing, exceeded) -> (unWithdrawals missing, exceeded)
  failOnNonEmptyMap missingWithdrawals $
    injectFailure . SubWithdrawalsMissingAccounts . Withdrawals . NEM.toMap
  failOnNonEmptyMap exceededWithdrawals $ injectFailure . SubWithdrawalAmountsExceedAccountBalances

  let certStateBeforeSubCerts =
        certState
          & Conway.updateDormantDRepExpiries tx curEpoch
          & Conway.updateVotingDRepExpiries tx curEpoch (pp ^. ppDRepActivityL)
          & certDStateL . accountsL %~ applyWithdrawals withdrawals
  certStateAfterSubCerts <-
    trans @(EraRule "SUBCERTS" era) $ TRC (subCertsEnv, certStateBeforeSubCerts, certificates)

  let directDeposits = tx ^. bodyTxL . directDepositsTxBodyL
      accountsAfterSubCerts = certStateAfterSubCerts ^. certDStateL . accountsL
  failOnJust (directDepositsMissingAccounts directDeposits accountsAfterSubCerts) $
    injectFailure . SubDirectDepositsToMissingAccounts

  pure $ certStateAfterSubCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
