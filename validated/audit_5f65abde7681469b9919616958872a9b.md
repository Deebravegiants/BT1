### Title
NFT Royalty Integer Division Truncation Silently Zeroes Royalty Payments in Offer Settlement - (File: chia/wallet/nft_wallet/nft_wallet.py)

### Summary
The `compute_royalty_amount` function in `chia/wallet/nft_wallet/nft_wallet.py` uses sequential integer floor division (`//`) to compute NFT royalty payments during offer settlement. When the offered fungible amount is small relative to `MAX_ROYALTY_BASIS_POINTS` (10000), the royalty truncates to exactly 0. When the royalty is 0, the on-chain announcement assertion that enforces royalty payment is silently skipped, allowing an unprivileged offer maker to settle an NFT trade without paying any royalty to the creator.

### Finding Description
`compute_royalty_amount` computes the royalty as:

```python
amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
``` [1](#0-0) 

Python's `//` operator always floors toward negative infinity. When `abs(offered_amount) // royalty_split * percentage < 10000`, the final `// MAX_ROYALTY_BASIS_POINTS` produces 0. The project's own test suite explicitly documents this behavior:

```python
def test_small_amount_truncates_to_zero() -> None:
    result = compute_royalty_amount(offered_amount=-50, royalty_split=1, percentage=100)
    assert result == uint64(0)
``` [2](#0-1) 

When `compute_royalty_amount` returns 0, the downstream code in `make_nft1_offer` that builds the on-chain `AssertPuzzleAnnouncement` conditions filters out zero-amount payments with `if p.amount > 0`:

```python
announcements_to_assert.extend(
    [
        AssertPuzzleAnnouncement(...)
        for launcher_id, p in payments
        if p.amount > 0   # <-- zero royalty silently skipped
    ]
)
``` [3](#0-2) 

This means no on-chain assertion is generated to enforce royalty payment. The offer bundle is valid and settles on-chain without any royalty coin being created for the NFT creator.

The same integer truncation pattern also appears in `royalty_calculation`:

```python
"amount": abs(amount) // len(royalty_assets_dict) * percentage // MAX_ROYALTY_BASIS_POINTS,
``` [4](#0-3) 

### Impact Explanation
An unprivileged offer maker can craft an NFT offer with a fungible amount small enough to cause `compute_royalty_amount` to return 0. The resulting spend bundle omits the royalty `AssertPuzzleAnnouncement`, so the on-chain NFT ownership transfer puzzle does not require any royalty coin to be created. The NFT creator's royalty is permanently lost for that trade. This is an unauthorized accounting change affecting NFT and XCH/CAT offer settlement — royalty diversion via truncation.

Threshold for truncation: `abs(offered_amount) * percentage < 10000`. For a 1% royalty (100 basis points), any offer below 100 mojos triggers this. For a 0.01% royalty (1 basis point), any offer below 10,000 mojos triggers it.

### Likelihood Explanation
Any user can create an offer. The attacker only needs to set the offered fungible amount below the truncation threshold for the target NFT's royalty percentage. No special privileges, leaked keys, or admin access are required. The wallet software will silently construct and submit the offer without royalties.

### Recommendation
Replace the sequential integer floor division with a single multiplication followed by a single division to minimize truncation error, and add a minimum royalty floor (e.g., 1 mojo) when the percentage is non-zero:

```python
amount = abs(offered_amount) * percentage // (royalty_split * MAX_ROYALTY_BASIS_POINTS)
if percentage > 0 and amount == 0:
    amount = 1  # enforce minimum non-zero royalty
```

Additionally, the `if p.amount > 0` filter in `make_nft1_offer` should not silently drop royalty payments — if a royalty is owed but computes to 0, the offer should either be rejected or the minimum enforced.

### Proof of Concept
1. Mint an NFT with `royalty_percentage = 100` (1%).
2. Create an offer: offer 50 mojos of XCH for the NFT.
3. `compute_royalty_amount(-50, 1, 100)` → `50 // 1 * 100 // 10000` = `5000 // 10000` = **0**.
4. The `if p.amount > 0` guard at line 853 skips the `AssertPuzzleAnnouncement` for the royalty.
5. The resulting spend bundle settles on-chain with no royalty coin created for the NFT creator.
6. The NFT creator receives 0 mojos in royalties despite having a 1% royalty configured. [5](#0-4)

def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    """Compute royalty using integer arithmetic, validating against overflow and excessive percentage."""
    if percentage > MAX_ROYALTY_BASIS_POINTS:
        raise ValueError(f"NFT royalty percentage {percentage} exceeds 100% ({MAX_ROYALTY_BASIS_POINTS} basis points)")
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
    royalty = uint64(amount)
    if royalty >= abs(offered_amount):
        raise ValueError("Royalty amount meets or exceeds the offered amount")
    return royalty
```
