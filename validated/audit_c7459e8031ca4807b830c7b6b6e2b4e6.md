### Title
Unchecked Detached Refund Promises Allow Silent Permanent Locking of Excess Deposits - (File: crates/contract/src/lib.rs)

---

### Summary

Multiple payable contract entry points use `Promise::new(...).transfer(diff).detach()` to return excess deposits to callers. The `.detach()` call is the direct NEAR analog of Solidity's deprecated `.transfer()` / `.send()` pattern: the promise is fired without its result ever being observed. If the refund transfer fails at runtime, the excess deposit is silently and permanently locked inside the MPC contract with no recovery path.

---

### Finding Description

Three distinct code sites share the same pattern:

**Site 1 — `require_deposit()`, called from every user-facing request method (`sign`, `request_app_private_key`, `verify_foreign_transaction`)** [1](#0-0) 

```rust
Some(diff) => {
    if diff > NearToken::from_yoctonear(0) {
        log!("refund excess deposit {diff} to {predecessor}");
        Promise::new(predecessor.clone()).transfer(diff).detach();
    }
}
```

**Site 2 — `submit_participant_info()`, callable by any unprivileged account** [2](#0-1) 

```rust
if let Some(diff) = attached.checked_sub(cost)
    && diff > NearToken::from_yoctonear(0)
{
    Promise::new(account_id).transfer(diff).detach();
}
```

**Site 3 — `propose_update()`, callable by any current participant** [3](#0-2) 

```rust
if let Some(diff) = attached.checked_sub(required)
    && diff > NearToken::from_yoctonear(0)
{
    Promise::new(proposer).transfer(diff).detach();
}
```

In NEAR SDK, `.detach()` schedules the promise as a fire-and-forget action receipt. The parent transaction completes successfully regardless of whether the child transfer receipt succeeds or fails. There is no callback, no error log on failure, and no on-chain mechanism to recover the locked tokens.

A `Transfer` action receipt fails when the recipient account does not exist at the time the receipt is applied. Because NEAR allows account deletion (`DeleteAccount` action), a smart-contract caller that attaches excess deposit and is subsequently deleted — as part of a legitimate multi-step DeFi flow — will trigger this failure path. The excess deposit remains in the MPC contract's balance permanently.

The minimum required deposit for `sign()` is 1 yoctoNEAR. [4](#0-3) 

A caller that attaches, say, 1 NEAR (a common round-number mistake or a contract that forwards its full balance) would have ~1 NEAR silently locked if the refund fails.

---

### Impact Explanation

If the detached refund promise fails, the excess deposit is permanently frozen inside the MPC contract. The caller has no on-chain recourse: the parent transaction already succeeded, no error is surfaced, and the contract exposes no admin withdrawal function for arbitrary locked balances. This breaks the production accounting invariant that every excess deposit is returned to its sender, and constitutes permanent freezing of user funds held by the MPC contract.

**Impact: Medium** — balance/accounting invariant broken without requiring network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

The trigger condition — recipient account absent at promise-execution time — is reachable via a legitimate NEAR operation (`DeleteAccount`). A smart contract that:
1. calls `sign()` (or `submit_participant_info()`) with excess deposit as part of a larger atomic flow, and
2. is deleted in a subsequent receipt within the same transaction batch or shortly after,

will silently lose the refund. This pattern appears in bridge contracts, proxy contracts, and one-shot factory contracts that self-destruct after use. No privileged access is required; any unprivileged contract caller can reach Site 1 via `sign()`.

---

### Recommendation

Replace `.detach()` with a checked promise chain. The idiomatic NEAR SDK fix is to return the transfer promise (or chain it with `.then()`) so that a failed refund causes the enclosing receipt to fail and the deposit is not consumed:

```rust
// Instead of:
Promise::new(predecessor.clone()).transfer(diff).detach();

// Use (where the function can return a Promise):
return Promise::new(predecessor.clone()).transfer(diff);

// Or, if the function must return a value, chain a callback that panics on failure:
Promise::new(predecessor.clone())
    .transfer(diff)
    .then(Promise::new(env::current_account_id())
        .function_call("assert_refund_succeeded", vec![], NearToken::from_yoctonear(0), Gas::from_tgas(5)));
```

Alternatively, adopt the pattern used by the NEAR SDK's `#[payable]` refund helper, which returns the refund promise as the function's return value so the runtime enforces delivery.

---

### Proof of Concept

1. Deploy a one-shot proxy contract `proxy.near` that forwards calls to the MPC contract.
2. `proxy.near` calls `sign(request)` on the MPC contract, attaching 1 NEAR (excess above the 1 yoctoNEAR minimum).
3. `require_deposit()` fires `Promise::new("proxy.near").transfer(1_NEAR - 1_yocto).detach()`.
4. In the same transaction batch (or a subsequent transaction before the refund receipt is applied), `proxy.near` executes `DeleteAccount { beneficiary_id: "attacker.near" }`.
5. When the detached transfer receipt is applied, `proxy.near` no longer exists; the `Transfer` action fails.
6. The ~1 NEAR excess deposit remains permanently in the MPC contract's balance. No error is surfaced to any party. The `sign()` call itself succeeded normally.
