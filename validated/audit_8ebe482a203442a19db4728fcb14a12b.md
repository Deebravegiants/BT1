Audit Report

## Title
Downside Protection in `_updateRsETHPrice()` Compares Against All-Time High, Causing Temporary Freeze of User Withdrawals on Legitimate EigenLayer Slashing - (File: contracts/LRTOracle.sol)

## Summary

`LRTOracle._updateRsETHPrice()` pauses the entire protocol whenever the computed rsETH price drops more than `pricePercentageLimit` below `highestRsethPrice` (the all-time high). Because the threshold is anchored to the ATH rather than the previous price, a legitimate EigenLayer slashing event that causes even a modest percentage drop from the current price can exceed the threshold and trigger a protocol-wide pause. Any unprivileged address can call the public `updateRSETHPrice()` to trigger this path, temporarily freezing funds for users who have already transferred rsETH into `LRTWithdrawalManager` via `initiateWithdrawal`.

## Finding Description

`_updateRsETHPrice()` in `contracts/LRTOracle.sol` (lines 269–282) implements downside protection by comparing the new price against `highestRsethPrice`:

```solidity
if (newRsETHPrice < highestRsethPrice) {
    uint256 diff = highestRsethPrice - newRsETHPrice;
    bool isPriceDecreaseOffLimit =
        pricePercentageLimit > 0 && diff > pricePercentageLimit.mulWad(highestRsethPrice);

    if (isPriceDecreaseOffLimit) {
        if (!lrtDepositPool.paused()) lrtDepositPool.pause();
        if (!withdrawalManager.paused()) withdrawalManager.pause();
        _pause();
        return;
    }
```

The threshold is `pricePercentageLimit × highestRsethPrice`. Since `highestRsethPrice` is the all-time high and is never reset downward, a protocol that has appreciated (e.g., from 1.0 ETH to 1.5 ETH per rsETH) will have a proportionally larger absolute threshold. A slashing event that causes a drop exceeding this threshold — even if the current price has been stable at the ATH for months — triggers the auto-pause.

`updateRSETHPrice()` carries no access control:

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

Any address can call it immediately after slashing is reflected on-chain. When the auto-pause fires, `LRTWithdrawalManager` is paused, blocking both `completeWithdrawal` (line 183, `whenNotPaused`) and `unlockQueue` (line 279, `whenNotPaused`). Users who already called `initiateWithdrawal` — which transfers rsETH into the contract at line 166 — cannot recover their funds until an admin calls `unpause()` (line 352, `onlyLRTAdmin`).

The oracle itself is also paused, so the public `updateRSETHPrice()` (which has `whenNotPaused`) cannot be called again until admin intervention. The only bypass is `updateRSETHPriceAsManager()` (line 94), which is restricted to `onlyLRTManager` and does not automatically unpause the withdrawal manager.

## Impact Explanation

**Temporary freezing of funds (Medium).** Users who have initiated withdrawals — transferring rsETH to `LRTWithdrawalManager` via `initiateWithdrawal` — cannot call `completeWithdrawal` or have their requests processed via `unlockQueue` while the protocol is paused. Their rsETH is held in the contract with no trustless exit path. Recovery requires an admin (`onlyLRTAdmin`) to manually call `unpause()` on `LRTWithdrawalManager` and `LRTOracle`. This matches the allowed impact class "Temporary freezing of funds."

## Likelihood Explanation

EigenLayer operator slashing is an explicitly documented and first-class risk of restaking, confirmed by the presence of `SlashingLib.sol` in this repository. When slashing reduces the ETH value of assets held by `NodeDelegator`, `_getTotalEthInProtocol()` returns a lower value, lowering `newRsETHPrice`. Since `updateRSETHPrice()` is public, any user, keeper, or bot can trigger this path immediately after slashing is reflected on-chain. No special privileges or victim mistakes are required. The scenario is repeatable for every slashing event whose magnitude exceeds `pricePercentageLimit` from the ATH.

## Recommendation

1. **Compare against the previous price, not the all-time high.** Replace `highestRsethPrice` with `previousPrice` (already computed at line 228) as the baseline for the downside threshold. This ensures the guard triggers only on sudden, anomalous drops from the last known price, not on cumulative drift from the ATH.
2. **Use a separate, larger downside threshold.** Introduce a dedicated `maxDownsidePercentage` calibrated to realistic slashing magnitudes, distinct from the upside `pricePercentageLimit`.
3. **Emit an event instead of auto-pausing.** Allow the price update to proceed and emit a warning event, letting governance decide whether to pause, rather than auto-pausing on every price update that crosses the threshold.

## Proof of Concept

1. Protocol launches; rsETH price grows from 1.0 ETH to 1.5 ETH. `highestRsethPrice = 1.5e18`.
2. Admin sets `pricePercentageLimit = 1e16` (1%).
3. Downside threshold = `1.5e18 × 1% = 0.015e18`.
4. User calls `initiateWithdrawal`, transferring rsETH to `LRTWithdrawalManager` (line 166).
5. An EigenLayer operator is slashed; ETH value of restaked assets drops 1.5%. New rsETH price = `1.4775e18`. Diff = `0.0225e18 > 0.015e18`.
6. Any address calls `updateRSETHPrice()`.
7. `_updateRsETHPrice()` detects `isPriceDecreaseOffLimit = true`, pauses `LRTDepositPool`, `LRTWithdrawalManager`, and `LRTOracle`.
8. User cannot call `completeWithdrawal` (reverts on `whenNotPaused`). Operator cannot call `unlockQueue` (reverts on `whenNotPaused`).
9. rsETH remains locked in `LRTWithdrawalManager` until admin calls `unpause()`.
