### Title
`respond_verify_foreign_tx` Does Not Verify `payload_hash` Corresponds to the Submitted Request - (`File: crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the ECDSA signature is valid over `response.payload_hash`, but never checks that `response.payload_hash` is actually the hash of a `ForeignTxSignPayload` encoding the correct pending `request`. A single Byzantine attested participant can replay any previously-observed valid signature (over any hash) to resolve an unrelated pending `verify_foreign_transaction` request, delivering a cryptographically-valid but semantically-wrong response to the waiting user.

---

### Finding Description

In `respond_verify_foreign_tx` (lines 691–754 of `crates/contract/src/lib.rs`), the contract performs the following check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;   // taken from the response, not the request

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,          // only checks sig over this caller-supplied hash
    &secp_pk,
)
.is_ok()
```

The `payload_hash` is taken verbatim from the MPC node's response. The contract verifies that the signature is valid over that hash, but **never verifies that the hash encodes the correct pending request**. The `request` parameter is used only to look up the pending yield queue — it plays no role in the hash check.

Contrast this with the regular `respond` function (lines 563–651), where the payload hash is derived from the on-chain `request` itself:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(signature_response, payload_hash, &expected_public_key)
```

The `ForeignTxSignPayload` that nodes are supposed to sign is:

```
payload_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))
```

The contract has `request` but not `values`, so it cannot recompute the hash. However, it also makes no attempt to verify even the weaker property that the hash encodes the correct `request`. The missing check is: **`response.payload_hash` must be the hash of a payload whose `request` field matches the pending `VerifyForeignTransactionRequest`**.

---

### Impact Explanation

A single Byzantine attested participant can:

1. Observe any previously-submitted valid `VerifyForeignTransactionResponse` (all responses are public on-chain).
2. Call `respond_verify_foreign_tx(request = B, response = {payload_hash = hash_A, signature = sig_A})` where `B` is any currently-pending request and `(hash_A, sig_A)` is a valid signature pair from a prior, unrelated request.
3. The contract accepts the call because `sig_A` is a valid ECDSA signature over `hash_A` under the domain key, and request `B` exists in the pending queue.
4. The pending yield for request `B` is resolved and removed from `pending_verify_foreign_tx_requests` with the wrong response.
5. The user who submitted request `B` receives `VerifyForeignTransactionResponse { payload_hash: hash_A, signature: sig_A }`. SDK-side verification (`ForeignChainSignatureVerifier::verify_signature`) will fail because `hash_A ≠ H(ForeignTxSignPayload::V1 { request: B, values: ... })`.
6. The user's deposit is consumed, the pending slot is gone, and the user must resubmit and pay again.

This breaks the request-lifecycle and accounting invariants of the `verify_foreign_transaction` flow without requiring threshold-level collusion.

---

### Likelihood Explanation

- **Attacker role**: Any single attested MPC participant (below the signing threshold).
- **Required material**: Any previously-accepted `VerifyForeignTransactionResponse` — these are emitted on-chain and trivially observable.
- **Trigger condition**: At least one pending `verify_foreign_transaction` request exists (routine in production).
- **No cryptographic forgery needed**: The attacker replays an existing valid signature; no new threshold signing is required.

---

### Recommendation

The contract must bind the `payload_hash` to the pending request. Two complementary approaches:

1. **Include `values` in the response** so the contract can recompute `SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))` and compare it against `response.payload_hash`. This is the strongest fix.

2. **Alternatively**, require the MPC nodes to include the `request` field separately in the response and verify it matches the pending request before accepting the signature. This at least prevents cross-request replay even if `values` are not verified on-chain.

The fix mirrors how `respond` handles regular signatures: derive the hash from the on-chain request, not from the caller-supplied response.
