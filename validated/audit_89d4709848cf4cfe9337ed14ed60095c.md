### Title
Hardcoded Minimum Trie-Value Size in `gas_key_transfer_exec_fee` Causes Systematic Gas Undercharging for `GasKeyFunctionCall` Keys - (File: core/parameters/src/cost.rs)

### Summary

`gas_key_transfer_exec_fee()` computes the execution fee for `TransferToGasKey` and `WithdrawFromGasKey` actions using `AccessKey::min_gas_key_borsh_len()` — the borsh-serialized size of the smallest possible gas key (`GasKeyFullAccess`) — as a hardcoded estimate of the trie value length. When the target key is a `GasKeyFunctionCall` with a long `receiver_id` and/or many method names, the actual trie value is substantially larger, causing systematic gas undercharging for those actions.

### Finding Description

`gas_key_transfer_exec_fee` in `core/parameters/src/cost.rs` computes the per-byte execution fee component as:

```rust
let estimated_value_len = AccessKey::min_gas_key_borsh_len();
let per_byte = cfg
    .fee(ActionCosts::gas_key_byte)
    .exec_fee()
    .checked_mul((trie_key_len + estimated_value_len) as u64)
    .unwrap();
``` [1](#0-0) 

`AccessKey::min_gas_key_borsh_len()` is explicitly documented as "the size for `GasKeyFullAccess` (the smallest gas key variant)": [2](#0-1) 

However, `GasKeyFunctionCall` keys also contain a `FunctionCallPermission` struct with a variable-length `receiver_id` (up to 64 bytes) and `method_names` (up to 2000 bytes total per protocol limits): [3](#0-2) 

This same underestimating function is used for both `TransferToGasKey` and `WithdrawFromGasKey` in `exec_fee()`: [4](#0-3) 

And in the host function path (`promise_batch_action_transfer_to_gas_key`): [5](#0-4) 

### Impact Explanation

The `gas_key_byte` execution fee is `101_435_400` gas per byte. A `GasKeyFunctionCall` key with a 64-byte `receiver_id` and 2000 bytes of method names is ~2068 bytes larger than `GasKeyFullAccess`. This yields up to ~210 billion gas of undercharging per `TransferToGasKey` or `WithdrawFromGasKey` action. The corrupted protocol value is the `gas_burnt` field in the action receipt outcome — it is systematically lower than the actual trie I/O work performed by the receiver shard. Validators executing these actions are undercompensated, and an attacker can fill chunk space with underpriced trie reads/writes.

**Impact: Low** — the undercharging per action is bounded by the maximum `FunctionCallPermission` size and does not break consensus or state correctness; it only corrupts gas accounting for these specific action types.

### Likelihood Explanation

**Likelihood: Medium** — any unprivileged user can create a `GasKeyFunctionCall` key with maximum-length `receiver_id` and method names, then repeatedly issue `TransferToGasKey` or `WithdrawFromGasKey` transactions targeting it. No special privileges are required. The `GasKeyFunctionCall` permission type is a standard, supported variant. [6](#0-5) 

### Recommendation

Replace the hardcoded `AccessKey::min_gas_key_borsh_len()` estimate in `gas_key_transfer_exec_fee` with the actual borsh-serialized length of the target `AccessKey` value. Since the exec fee is computed at the point where the receiver's access key is known (the action is applied on the receiver shard), the actual key size can be obtained from the trie and used directly. Alternatively, add a separate fee parameter for the `GasKeyFunctionCall` overhead bytes so the fee schedule can be updated via protocol upgrade without hardcoding a structural assumption about the smallest key variant.

### Proof of Concept

1. User A creates a `GasKeyFunctionCall` key on their account with `receiver_id` = 64-char string and `method_names` = list of 256-char names totaling 2000 bytes.
2. User A (or anyone) submits a `TransferToGasKey` action targeting that key.
3. At execution, `gas_key_transfer_exec_fee` is called with `estimated_value_len = AccessKey::min_gas_key_borsh_len()` (~27 bytes for `GasKeyFullAccess`), ignoring the actual ~2095-byte `GasKeyFunctionCall` value.
4. The receipt outcome records `gas_burnt` that is ~`(2095 - 27) * 101_435_400 ≈ 210 billion gas` less than the actual trie I/O cost.
5. Repeating this across many transactions fills chunk gas budget with underpriced trie operations. [7](#0-6) [2](#0-1)

/// Exec fee for TransferToGasKey / WithdrawFromGasKey actions.
/// Based on the access key trie key length + estimated value length (what the
/// receiver needs to read/write in the trie).
pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
}
```
