### Title
Stale Reverse-Delegation Authorization Not Revoked on DRep Redelegation Enables Governance Vote-Power Erasure During Bootstrap Phase â€” (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Deleg.hs`)

### Summary

During the Conway bootstrap phase (protocol version < 10), when a stake credential redelegates from DRep A to DRep B, the reverse-delegation entry in DRep A's `drepDelegs` set is intentionally **not removed**. When DRep A subsequently unregisters, the `ConwayUnRegDRep` handler uses that stale `drepDelegs` set to clear `dRepDelegationAccountStateL` for every listed delegator â€” including delegators who have already moved to DRep B. This incorrectly nullifies the vote delegation of those delegators, silently reducing DRep B's governance voting power in subsequent epochs.

---

### Finding Description

**Root cause â€” `processDelegationInternal` (bootstrap path):** [1](#0-0) 

When `preserveIncorrectDelegation = True` (set to `pvMajor pv < natVersion @10` by the caller), the branch at line 363 only **inserts** the delegator into the new DRep's `drepDelegs` and never removes it from the old DRep's `drepDelegs`:

```haskell
| isNothing mAccountState || preserveIncorrectDelegation ->
    certVStateL . vsDRepsL
      %~ Map.adjust (drepDelegsL %~ Set.insert stakeCred) dRepCred
```

The old DRep's `drepDelegs` therefore retains a stale entry for the delegator even after they have moved to DRep B.

**Trigger â€” `ConwayUnRegDRep` in `conwayGovCertTransition`:** [2](#0-1) 

```haskell
clearDRepDelegations delegs accountsMap =
  foldr (Map.adjust (dRepDelegationAccountStateL .~ Nothing)) accountsMap delegs
pure $
  case mDRepState of
    Nothing -> certState'
    Just dRepState ->
      certState'
        & certDStateL . accountsL . accountsMapL
          %~ clearDRepDelegations (drepDelegs dRepState)
```

`clearDRepDelegations` iterates over `drepDelegs dRepState` and sets `dRepDelegationAccountStateL .~ Nothing` for every listed credential. Because the stale entry was never removed, the delegator who already moved to DRep B has their `dRepDelegationAccountStateL` wiped to `Nothing`.

**Confirmed by the test suite:** [3](#0-2) 

The test explicitly documents that in bootstrap phase, after the old DRep unregisters, `lookupDRepDelegation cred accounts` returns `Nothing` even though the delegator had already moved to DRep B â€” the delegation is silently erased.

**Stake distribution consequence:**

`dRepAcceptedRatio` builds the DRep stake distribution from accounts' `dRepDelegationAccountStateL`. Once that field is `Nothing`, the delegator's stake is excluded from every DRep's distribution in the next epoch snapshot, permanently reducing DRep B's voting power until the delegator re-delegates. [4](#0-3) 

---

### Impact Explanation

An attacker who registers as a DRep, accumulates delegators, and then waits for those delegators to redelegate to a competing DRep can unregister their own DRep at any time. The unregistration transaction â€” submitted by an unprivileged actor â€” silently clears the vote delegation of every delegator who had previously moved away. This reduces the competing DRep's stake-weighted voting power in all subsequent ratification rounds. If the competing DRep was near a ratification threshold, this can:

- Prevent a legitimate governance action (e.g., a treasury withdrawal, protocol-parameter change, or hard-fork initiation) from reaching the required DRep acceptance ratio, causing it to expire without enactment.
- Conversely, allow a governance action that would otherwise have been blocked by the competing DRep's stake to pass.

This maps to **Medium** (attacker-controlled certificate modifies governance voting power outside design parameters) and potentially **Critical** (unauthorized governance/treasury/protocol-parameter action is enacted or a legitimate one is blocked).

---

### Likelihood Explanation

- Registering as a DRep is permissionless (requires only a deposit).
- The attacker does not need to solicit delegations; any delegator who naturally delegates and later redelegates creates the precondition.
- The unregistration transaction is a single, valid, fee-paying transaction with no special privileges.
- The attack is only possible while the network is in the bootstrap phase (protocol version major < 10). On any network still running at version 9, this is immediately exploitable.

---

### Recommendation

When `ConwayDelegCert` processes a redelegation during the bootstrap phase, the old DRep's `drepDelegs` entry for the delegator should be removed before inserting into the new DRep's set â€” mirroring the post-bootstrap behavior in `unDelegReDelegDRep`. Alternatively, `clearDRepDelegations` in `ConwayUnRegDRep` should cross-check each listed delegator's current `dRepDelegationAccountStateL` before clearing it, and only clear it if the account is still actually delegating to the unregistering DRep.

---

### Proof of Concept

1. Network is in bootstrap phase (protocol version 9).
2. Attacker submits `RegDRepTxCert attackerDRep deposit SNothing` â€” registers as DRep A.
3. Victim submits `RegDepositDelegTxCert victim (DelegVote (DRepCredential attackerDRep)) deposit` â€” delegates to DRep A.
4. Victim submits `DelegTxCert victim (DelegVote (DRepCredential legitimateDRep))` â€” redelegates to DRep B. Due to the bootstrap bug, DRep A's `drepDelegs` still contains `victim`.
5. Attacker submits `UnRegDRepTxCert attackerDRep deposit` â€” unregisters DRep A. `clearDRepDelegations` iterates over DRep A's stale `drepDelegs`, finds `victim`, and sets `victim`'s `dRepDelegationAccountStateL .~ Nothing`.
6. At the next epoch boundary, the stake snapshot excludes `victim`'s stake from DRep B's distribution.
7. DRep B's `dRepAcceptedRatio` is lower than it should be; governance actions that required DRep B's stake to pass the threshold now fail ratification.

```haskell
dRepAcceptedRatio RatifyEnv {reDRepDistr, reDRepState, reCurrentEpoch} gasDRepVotes govAction =
  toInteger yesStake %? toInteger totalExcludingAbstainStake
  where
    accumStake (!yes, !tot) drep (CompactCoin stake) =
      case drep of
        DRepCredential cred ->
          case Map.lookup cred reDRepState of
            Nothing -> (yes, tot) -- drep is not registered, so we don't consider it
            Just drepState
              | reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, so we don't consider it
              | otherwise ->
                  case Map.lookup cred gasDRepVotes of
                    -- drep hasn't voted for this action, so we don't count
                    -- the vote but we consider it in the denominator:
                    Nothing -> (yes, tot + stake)
                    Just VoteYes -> (yes + stake, tot + stake)
                    Just Abstain -> (yes, tot)
                    Just VoteNo -> (yes, tot + stake)
        DRepAlwaysNoConfidence ->
          case govAction of
            NoConfidence _ -> (yes + stake, tot + stake)
            _ -> (yes, tot + stake)
        DRepAlwaysAbstain -> (yes, tot)
    (yesStake, totalExcludingAbstainStake) = Map.foldlWithKey' accumStake (0, 0) reDRepDistr
```
