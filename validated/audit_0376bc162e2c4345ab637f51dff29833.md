Audit Report

## Title
Stale `AGETHRateReceiver` Rate Causes Structural Undercollateralization in `AGETHTokenWrapper`, Permanently Freezing Last Redeemers' Funds — (`contracts/agETH/AGETHPoolV3.sol`, `contracts/cross-chain/CrossChainRateReceiver.sol`, `contracts/agETH/AGETHTokenWrapper.sol`)

## Summary

`CrossChainRateReceiver.getRate()` returns the stored `rate` with no staleness enforcement despite tracking `lastUpdated`. When a LayerZero message is delayed, `AGETHPoolV3.deposit()` mints more wrapper tokens than the ETH deposited can back at the true L1 rate. The bridger, operating correctly off-chain, can only return as much altAgETH as the ETH actually purchases on L1 at the current rate. The resulting shortfall is a permanent, irrecoverable undercollateralization in `AGETHTokenWrapper`: the last redeemers hold wrapper tokens that can never be redeemed because the backing altAgETH does not exist in the contract.

## Finding Description

**Root cause — no staleness guard in `getRate()`:**

`CrossChainRateReceiver` stores `lastUpdated` (line 16) but `getRate()` (lines 103–105) blindly returns `rate` with no check against any maximum staleness window. `AGETHRateReceiver` inherits this without adding any guard.

**Over-minting path:**

`AGETHPoolV3.viewSwapAgETHAmountAndFee()` (lines 160–169) computes:
```
agETHAmount = amountAfterFee * 1e18 / agETHToETHrate;
```
where `agETHToETHrate = getRate()`. Because agETH accrues staking yield monotonically, any stale rate is always lower than the true rate, making the division yield a larger `agETHAmount` than the correct rate would produce. `deposit()` (lines 121–127) then calls `agETH.mint(msg.sender, agETHAmount)`, minting the inflated amount directly to the user.

**Bridger cannot cover the gap:**

`AGETHTokenWrapper.depositBridgerAssets()` (lines 143–151) is the mechanism by which the bridger backs already-minted wrapper tokens with altAgETH bridged from L1. The bridger takes the ETH from the pool, purchases agETH on L1 at the current (correct, higher) rate, and bridges it back. The altAgETH received equals `ETH * 1e18 / correctRate`, which is strictly less than the `agETHAmount_stale` already minted. The `maxAmountToDepositBridgerAsset()` cap (lines 90–101) allows the bridger to deposit up to `totalSupply() - balanceOfAssetInWrapper`, but the bridger physically holds less altAgETH than that cap, so the gap persists after the deposit.

**Permanent freeze:**

`_withdraw()` (lines 111–119) burns wrapper tokens and transfers altAgETH 1:1 via `safeTransfer`. Once the altAgETH balance is exhausted, all subsequent `withdraw()` calls revert with an ERC20 transfer failure. There is no admin function to burn excess wrapper tokens, mint additional altAgETH, or otherwise close the gap. `setIsEthDepositEnabled()` can pause future deposits but cannot retroactively correct the already-minted supply.

**No existing guard is sufficient:**

- `CrossChainRateReceiver`: `lastUpdated` is recorded but never compared to any threshold anywhere in the production contracts.
- `AGETHPoolV3`: no rate-deviation check, no circuit breaker, no automatic pause on stale oracle.
- `AGETHTokenWrapper`: `depositBridgerAssets()` enforces only an upper cap, not a lower bound ensuring full collateralization.

## Impact Explanation

**Critical — Permanent freezing of funds.**

Users who deposited during a stale-rate window hold wrapper tokens that can never be redeemed. The undercollateralization is structural: the backing altAgETH simply does not exist in the contract, and no existing on-chain function can create it or destroy the excess supply. This matches the allowed impact "Permanent freezing of funds."

## Likelihood Explanation

- LayerZero message delays and drops are a documented, real-world occurrence requiring no attacker action.
- agETH's rate is monotonically non-decreasing (staking yield), so any stale rate is always lower than the current rate, always producing over-minting — the direction of the error is deterministic.
- The exploit is triggered by a normal, unprivileged `deposit()` call; no special role or capital beyond the deposit itself is required.
- The bridger's correct off-chain behavior is what causes the gap — a non-malicious, protocol-correct actor deterministically produces the undercollateralization.
- No automatic staleness-based pause or rate-deviation guard exists in any of the three contracts.

## Recommendation

1. **Enforce a staleness threshold in `getRate()`**: revert if `block.timestamp - lastUpdated > MAX_STALENESS` (e.g., 1 hour).
2. **Add a staleness check in `AGETHPoolV3.viewSwapAgETHAmountAndFee()`** before consuming the oracle rate, reverting or pausing deposits if the rate is stale.
3. **Add a minimum-rate guard**: revert if the returned rate deviates more than a configured percentage below the previously accepted rate.
4. **Automatically pause deposits** (`isEthDepositEnabled = false`) when the rate has not been updated within the staleness window, requiring an explicit admin re-enable after the oracle is refreshed.

## Proof of Concept

```solidity
// Foundry fork test (pinned block where LZ message is delayed)
function testStaleRateUndercollateralization() external {
    // 1. Simulate stale rate: true L1 rate is 1.10e18, oracle still returns 1.05e18
    vm.mockCall(
        address(agETHOracle),
        abi.encodeWithSelector(IOracle.getRate.selector),
        abi.encode(1.05e18)
    );

    // 2. Unprivileged user deposits 1 ETH — public call, no special role needed
    vm.deal(user, 1 ether);
    vm.prank(user);
    pool.deposit{value: 1 ether}("ref");
    // agETHAmount_stale ≈ 0.952e18 wrapper tokens minted

    // 3. Bridger takes ETH to L1, buys agETH at correct rate 1.10e18,
    //    bridges back altAgETH — correct amount is 1e18 * 1e18 / 1.10e18 ≈ 0.909e18
    uint256 correctAltAgETH = 1e18 * 1e18 / 1.10e18;
    vm.prank(bridger);
    wrapper.depositBridgerAssets(altAgETH, correctAltAgETH);

    // 4. Wrapper is undercollateralized
    uint256 supply  = wrapper.totalSupply();           // ≈ 0.952e18
    uint256 backing = IERC20(altAgETH).balanceOf(address(wrapper)); // ≈ 0.909e18
    assertGt(supply, backing); // gap ≈ 0.043e18 — permanent, no on-chain remedy

    // 5. Last redeemer is permanently frozen
    vm.prank(user);
    vm.expectRevert(); // safeTransfer fails — insufficient altAgETH in wrapper
    wrapper.withdraw(altAgETH, supply);
}
```
