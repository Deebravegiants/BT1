### Title
Excess Attached Deposit Not Refunded in `submit_participant_info` When Caller Is an Existing Participant — (`File: crates/contract/src/lib.rs`)

### Summary

`submit_participant_info` is a `#[payable]` function that silently consumes any attached NEAR deposit when the caller is an **existing participant** updating their attestation. The refund branch is gated behind a boolean flag that is `false` in this code path, so any NEAR attached to the call is permanently locked in the contract with no mechanism for recovery.

### Finding Description

The function `submit_participant_info` computes a flag to decide whether the caller must pay for storage:

```rust
let caller_is_not_participant = self.voter_account().is_err();
let is_new_attestation = matches!(
    attestation_insertion_result,
    ParticipantInsertion::NewlyInsertedParticipant
);

let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... cost check and refund of excess ...
    if let Some(diff) = attached.checked_sub(cost)
        && diff > NearToken::from_yoctonear(0)
    {
        Promise::new(account_id).transfer(diff).detach();
    }
}
``` [1](#0-0) 

When `attestation_storage_must_be_paid_by_caller` evaluates to `false` — i.e., when `!is_new_attestation && !caller_is_not_participant`, meaning an **existing participant re-submits or updates their attestation** — the entire `if` block is skipped. Because the function is marked `#[payable]`, the NEAR runtime accepts any attached deposit without complaint, but the contract never returns it. The deposit is absorbed into the contract's balance with no accounting entry and no refund path. [2](#0-1) 

Every other `#[payable]` entry point in the contract either calls the shared `require_deposit` helper (which always refunds the excess) or has an explicit unconditional refund:

- `sign`, `request_app_private_key`, `verify_foreign_transaction` → `check_request_preconditions` → `require_deposit` (lines 122–141) always refunds.
- `propose_update` → explicit `checked_sub` + `transfer` at lines 1327–1331. [3](#0-2) [4](#0-3) 

`submit_participant_info` is the only `#[payable]` function that has a conditional refund path that can be bypassed.

### Impact Explanation

Any NEAR tokens attached to a `submit_participant_info` call by an existing participant are **permanently locked** in the MPC contract. There is no withdrawal function, no admin sweep, and no subsequent call that would release them. This constitutes a direct, irreversible loss of user funds — matching the "permanent freezing of funds controlled by the MPC network" criterion.

### Likelihood Explanation

MPC participant nodes call `submit_participant_info` periodically to refresh their TEE attestation quotes. Node automation scripts or operator tooling may attach a small deposit as a safety margin (mirroring the pattern used for new-participant submissions). Any such deposit on a re-submission call is silently lost. The trigger condition — being an existing participant — is the **normal steady-state** for all active nodes, making this the common code path rather than an edge case.

### Recommendation

Add an unconditional refund of any excess deposit at the end of `submit_participant_info`, mirroring the pattern already used in `propose_update` and `require_deposit`:

```rust
// After the conditional storage-cost block:
let attached = env::attached_deposit();
if attached > NearToken::from_yoctonear(0) {
    Promise::new(account_id).transfer(attached).detach();
}
```

Or, restructure the function so that `require_deposit(NearToken::from_yoctonear(0), &account_id)` is always called, which already handles the refund-of-excess logic uniformly. [3](#0-2) 

### Proof of Concept

1. Participant `alice.near` is already registered (her attestation exists in `tee_state`).
2. Alice's node refreshes its TEE quote and calls `submit_participant_info` with `1 NEAR` attached (e.g., because the operator script always attaches a deposit to cover potential storage costs).
3. Inside the function, `attestation_insertion_result` is `ParticipantInsertion::UpdatedParticipant` (not `NewlyInsertedParticipant`), so `is_new_attestation = false`.
4. `self.voter_account()` succeeds because Alice is a participant, so `caller_is_not_participant = false`.
5. `attestation_storage_must_be_paid_by_caller = false || false = false`.
6. The `if attestation_storage_must_be_paid_by_caller { ... }` block is skipped entirely.
7. The function returns `Ok(())`. Alice's `1 NEAR` is now permanently held by the contract with no refund issued. [5](#0-4)
