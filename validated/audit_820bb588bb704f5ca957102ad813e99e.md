### Title
Pool Admin Can Instantly Set Uncapped Per-Bin Fees Without Timelock, Front-Running Swapper Inputs — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`setPoolBinAdditionalFees` allows the pool admin to change `addFeeBuyE6` / `addFeeSellE6` on any bin to any `uint16` value (0–65535, i.e. up to 6.5535% in E6 units) with immediate effect and no cap validation. The protocol's own trust model states the pool admin "cannot exceed caps or bypass timelocks," yet this function enforces neither. A pool admin can front-run a pending swap by spiking the per-bin fee to `type(uint16).max`, causing the swapper to pay a materially higher fee than expected, then reset the fee in the same block.

### Finding Description

The protocol enforces two safety constraints on pool-admin fee changes:

1. **Cap enforcement** — `setPoolAdminFees` checks `newAdminSpreadFeeE6 > maxAdminSpreadFeeE6` and reverts with `AdminFeeTooHigh`.
2. **Timelock** — `proposePoolPriceProvider` / `executePoolPriceProviderUpdate` require waiting `priceProviderTimelock[pool]` seconds before the change takes effect.

`setPoolBinAdditionalFees` applies **neither** constraint:

```solidity
// MetricOmmPoolFactory.sol L450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

The pool-level handler also performs no cap check:

```solidity
// MetricOmmPool.sol L464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external
    onlyFactory
    nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    emit BinAdditionalFeesUpdated(bin, addFeeBuyE6, addFeeSellE6);
}
```

These per-bin fees are directly injected into every swap through the bin walk:

```solidity
// MetricOmmPool.sol L1177
params.baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6)
```

Setting `addFeeSellE6 = 65535` adds ≈6.55% on top of the base spread fee, immediately and atomically, with no on-chain warning period.

### Impact Explanation

A pool admin can:
1. Observe a pending swap in the mempool targeting the active bin.
2. Call `setPoolBinAdditionalFees(pool, curBinIdx, 65535, 65535)` in a higher-priority transaction.
3. The victim's swap executes with an effective fee up to 6.55% higher than the quoted rate, paying more token-in than the oracle/bin curve would normally require.
4. The admin resets the fee to 0 in the same block.

The excess fee accrues as LP fee inside the bin (not to the admin directly), but the swapper suffers a direct, quantifiable loss of input tokens relative to the price they observed. This is a swap conservation failure: the trader pays more than the bin curve permits at the quoted spread.

**Severity: Medium** — requires a semi-trusted pool admin acting maliciously, but the README explicitly states this class of finding is valid: *"The finding can be valid only if the Pool admin can bypass the caps or honestly configured timelocks and it qualifies for Medium or higher severity."* Both conditions are met.

### Likelihood Explanation

Any pool whose admin is an EOA or a compromised multisig can trigger this. The attack is atomic (single block), requires no special setup, and leaves no persistent trace beyond the fee reset. Pools with mutable admins and active swap volume are the primary target surface.

### Recommendation

Apply the same two-layer protection used for other admin-controlled parameters:

1. **Cap**: Validate `addFeeBuyE6` and `addFeeSellE6` against `maxAdminSpreadFeeE6` (or a dedicated per-bin cap) inside `setPoolBinAdditionalFees`, mirroring the check in `setPoolAdminFees`.
2. **Timelock**: Introduce a propose/execute pattern for per-bin fee changes, analogous to `proposePoolPriceProvider` / `executePoolPriceProviderUpdate`, so users have advance notice before a fee spike takes effect.

### Proof of Concept

```solidity
// Attacker = pool admin
// 1. Observe victim's swap tx in mempool (e.g. sell token0 into bin 0)
// 2. Front-run with fee spike
factory.setPoolBinAdditionalFees(pool, 0, 0, 65535); // addFeeSellE6 = max

// 3. Victim's swap executes: pays ~6.55% extra fee on token0 input
//    swap(SwapParams{zeroForOne: true, ...})

// 4. Admin resets fee in same block
factory.setPoolBinAdditionalFees(pool, 0, 0, 0);
```

The victim's `amountCalculatedScaled` (token0 input required) is inflated by `Math.ceilDiv(amountInScaled * currBinSellFeeX64, ONE_X64)` where `currBinSellFeeX64` now includes the spiked `addFeeSellE6`, causing the callback to transfer more token0 than the pre-swap quote indicated.
