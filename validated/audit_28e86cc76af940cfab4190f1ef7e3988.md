### Title
OCert Cold-Key Signature Missing Network Discriminator Enables Cross-Network Operational Certificate Replay - (File: libs/cardano-protocol-tpraos/src/Cardano/Protocol/TPraos/OCert.hs)

---

### Summary

The `OCertSignable` type's `getSignableRepresentation` encodes only `(vk_hot || counter || kes_period)` — with no network identifier — as the payload signed by a pool's cold key. Because `ShelleyBase` carries `networkId` in `Globals` but the `OCERT` transition rule never reads it during signature verification, an operational certificate (OCert) issued on one Cardano network is cryptographically valid on any other network where the same cold key is registered. An attacker who obtains a pool's KES hot key from one network can replay that network's OCert on a second network, producing blocks as the legitimate pool without possessing the cold key.

---

### Finding Description

**Root cause — `OCertSignable` payload:** [1](#0-0) 

```haskell
data OCertSignable c
  = OCertSignable !(KES.VerKeyKES (KES c)) !Word64 !KESPeriod

instance Crypto c => SignableRepresentation (OCertSignable c) where
  getSignableRepresentation (OCertSignable vk counter period) =
    runByteBuilder ...
      $ BS.byteStringCopy (KES.rawSerialiseVerKeyKES vk)
        <> BS.word64BE counter
        <> BS.word64BE (fromIntegral $ unKESPeriod period)
```

The signed bytes are exactly `vk_hot || counter || kes_period`. No genesis hash, no network magic, no `networkId` is included.

**Verification site — `OCERT` transition rule:** [2](#0-1) 

The rule calls:
```haskell
verifySignedDSIGN vkcold (ocertToSignable $ bheaderOCert bhb) tau ?! InvalidSignatureOCERT n c0
```
and then checks only the KES-period window and the counter monotonicity (`m <= n`). It never calls `liftSTS $ asks networkId`.

**`networkId` is available but unused:**

`Globals` carries `networkId :: !Network` and `ShelleyBase = ReaderT Globals Identity`. [3](#0-2) 

Every other security-sensitive rule reads it — UTXO, POOL, GOV, CERTS, BBODY all call `liftSTS $ asks networkId` — but `OCERT` does not. [4](#0-3) 

**Formal spec confirms the omission:**

The OCert rule in the Shelley formal spec verifies:
```
V_{vk_cold}(serialised(vk_hot, n, c0))_tau
```
No network discriminator appears in the signed message. [5](#0-4)

### Impact Explanation

**Impact: High — Deterministic disagreement between honest nodes / unauthorized block production**

Attack scenario:
1. Pool operator registers the same cold key on mainnet and a testnet (common operational practice).
2. On testnet the pool has issued OCert with counter `N`; on mainnet the stored counter is `M < N`.
3. Attacker compromises the testnet node and obtains the KES hot key and VRF key (both kept online).
4. Attacker presents the testnet OCert (`counter = N`) in mainnet block headers. The OCERT rule accepts it: the cold-key signature is valid, `M ≤ N`, and the cold key is registered on mainnet.
5. Attacker signs mainnet block-header bodies with the stolen KES hot key (the KES key material is network-agnostic; the KES signature covers `BHBody` which is chain-specific, but the attacker *holds the key* and can sign fresh mainnet `BHBody` values).
6. Attacker produces mainnet blocks as the legitimate pool.

If the legitimate pool operator is simultaneously producing blocks for the same slots, two valid competing chains exist. All honest nodes deterministically accept whichever chain they see first, causing a persistent fork. Resolving the equivocation may require a hard fork.

---

### Likelihood Explanation

**Likelihood: Medium**

- Running the same cold key on mainnet and a public testnet (Preview, PreProd) is standard practice for Cardano stake pool operators; the prerequisite is widely met.
- The testnet OCert counter exceeding the mainnet counter is plausible whenever a pool is more active on testnet (key rotations, testing).
- Stealing the KES hot key requires compromising the pool's online block-producing node, which is a realistic but non-trivial attacker capability.
- No privileged access beyond the pool's own online infrastructure is required.

---

### Recommendation

Include the genesis hash (or `networkId`) in `OCertSignable.getSignableRepresentation` so that a cold-key signature is bound to a specific network:

```haskell
-- In OCertSignable, add a Network or genesis-hash field:
data OCertSignable c
  = OCertSignable !(KES.VerKeyKES (KES c)) !Word64 !KESPeriod !Network

instance Crypto c => SignableRepresentation (OCertSignable c) where
  getSignableRepresentation (OCertSignable vk counter period nw) =
    runByteBuilder ... $
      BS.byteStringCopy (KES.rawSerialiseVerKeyKES vk)
        <> BS.word64BE counter
        <> BS.word64BE (fromIntegral $ unKESPeriod period)
        <> encodeNetwork nw          -- network discriminator
```

`ocertToSignable` and the `OCERT` transition rule would need to supply the `networkId` from `Globals` when constructing the signable value. This is a consensus-layer change requiring a hard fork to deploy.

```haskell
ocertTransition =
  judgmentContext >>= \(TRC (env, cs, BHeader bhb sigma)) -> do
    let OCert vk_hot n c0@(KESPeriod c0_) tau = bheaderOCert bhb
        vkcold = bheaderVk bhb
        hk = hashKey vkcold
        s = bheaderSlotNo bhb
    kp@(KESPeriod kp_) <- liftSTS $ kesPeriod s

    maxKESiterations <- liftSTS $ asks maxKESEvo

    c0 <= kp ?! KESBeforeStartOCERT c0 kp
    kp_
      < c0_
        + fromIntegral maxKESiterations
          ?! KESAfterEndOCERT kp c0 maxKESiterations

    let t = if kp_ >= c0_ then kp_ - c0_ else 0 -- this is required to prevent an
    -- arithmetic underflow, in the
    -- case of kp_ < c0_ we get the
    -- above `KESBeforeStartOCERT`
    -- predicate failure in the
    -- transition.
    verifySignedDSIGN vkcold (ocertToSignable $ bheaderOCert bhb) tau ?! InvalidSignatureOCERT n c0
    verifySignedKES () vk_hot t bhb sigma ?!: InvalidKesSignatureOCERT kp_ c0_ t

    case currentIssueNo env cs hk of
      Nothing -> do
        failBecause $ NoCounterForKeyHashOCERT hk
        pure cs
      Just m -> do
        m <= n ?! CounterTooSmallOCERT m n
        pure (Map.insert hk n cs)
```
