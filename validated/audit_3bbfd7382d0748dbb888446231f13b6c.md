Audit Report

## Title
Zero-Address `owner` in `addLiquidity` Permanently Locks LP Principal - (File: metric-core/contracts/libraries/LiquidityLib.sol)

## Summary
`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address and passes it directly to `LiquidityLib.addLiquidity` without any zero-address guard. When `owner = address(0)`, real tokens are pulled from the caller via the modify-liquidity callback and credited to the pool, but the resulting LP shares are minted to the `address(0)` position key. Because `removeLiquidity` enforces `msg.sender == owner` and `msg.sender` can never be `address(0)` on-chain, the deposited principal is permanently locked with no recovery path.

## Finding Description
`MetricOmmPool.addLiquidity` passes `owner` directly into `LiquidityLib.addLiquidity` with no zero-address validation:

```solidity
// metric-core/contracts/MetricOmmPool.sol L182-196
function addLiquidity(
  address owner,
  ...
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
  ...
  (amount0Added, amount1Added) = LiquidityLib.addLiquidity(
    _liquidityContext(), owner, salt, deltas, callbackData, ...
  );
}
```

Inside `LiquidityLib.addLiquidity`, the position key is computed as `keccak256(abi.encode(owner, salt, bin))` (L72, L256-258). Shares are written to `positionBinShares[posKey]` and `binTotalShares[binIdx]` is incremented (L120-121), then the callback fires on `msg.sender` pulling real tokens into the pool (L147-148). `binTotals` is updated to reflect the deposit (L134-138), making pool accounting internally consistent but the LP claim inaccessible.

`removeLiquidity` enforces `msg.sender == owner` before any state change (L206). Since `msg.sender` is never `address(0)` on-chain, the `NotPositionOwner` revert fires unconditionally for any attempt to remove the zero-address position. No admin, factory, or emergency function provides a recovery path.

The periphery `MetricOmmPoolLiquidityAdder` calls `_validateOwner(owner)` (L65) which reverts with `InvalidPositionOwner` for zero-address, but this guard exists only in the periphery — the core pool, which is the canonical production surface, has no equivalent check.

## Impact Explanation
Any tokens deposited via `addLiquidity(address(0), ...)` are permanently irrecoverable. The depositor loses 100% of their principal. This constitutes a direct loss of user principal with no on-chain recovery mechanism, meeting the Critical/High threshold under the allowed impact gate (direct loss of user principal, broken core liquidity withdraw flow).

## Likelihood Explanation
The `addLiquidity` interface explicitly supports the operator pattern where `msg.sender != owner`, making it natural for routers, aggregators, or integration contracts to accept a user-supplied `owner` parameter. A buggy or malicious router that passes a user-supplied or misconfigured `owner = address(0)` silently triggers permanent fund loss. The factory validates `admin != address(0)` and `adminFeeDestination != address(0)` at pool creation, making the omission of an equivalent guard on the liquidity path inconsistent and easy to overlook. The call is reachable by any unprivileged external caller with no special preconditions.

## Recommendation
Add a zero-address check at the entry of `MetricOmmPool.addLiquidity` (or equivalently at the top of `LiquidityLib.addLiquidity`):

```solidity
if (owner == address(0)) revert InvalidPositionOwner();
```

This mirrors the guard already present in `MetricOmmPoolLiquidityAdder._validateOwner` and is consistent with the factory's existing `InvalidAdmin` / `InvalidAdminFeeDestination` checks.

## Proof of Concept

1. Deploy a contract implementing `IMetricOmmModifyLiquidityCallback` that transfers tokens to the pool in `metricOmmModifyLiquidityCallback`.
2. Call:
   ```solidity
   pool.addLiquidity(
       address(0),   // owner = zero address
       0,            // salt
       deltas,       // valid binIdxs/shares arrays
       callbackData,
       ""
   );
   ```
3. `LiquidityLib.addLiquidity` computes `posKey = keccak256(abi.encode(address(0), 0, binIdx))` and writes shares there; `binTotals` is updated; callback fires and real tokens are transferred into the pool.
4. Any subsequent call to `pool.removeLiquidity(address(0), 0, deltas, "")` reverts with `NotPositionOwner` because `msg.sender != address(0)`.
5. The deposited tokens are permanently locked; no admin, factory, or owner function provides a recovery path.
