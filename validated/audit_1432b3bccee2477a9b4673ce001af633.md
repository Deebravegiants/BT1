### Title
Non-Conforming NEP-141 `storage_balance_of` Response Permanently Blocks ERC-20 Emulated Transfers - (`runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs`)

---

### Summary

The `WalletContract::nep_141_storage_balance_callback` function strictly deserializes the `storage_balance_of` response from a NEP-141 token contract as `Option<StorageBalance>`. If the token contract returns a response that is syntactically valid but does not match the exact expected JSON schema (e.g., integer values instead of string-encoded U128 amounts), deserialization fails and the ERC-20 emulated transfer is permanently blocked for that token — even though the token contract is otherwise functional and the transfer would have succeeded.

---

### Finding Description

When a user with an eth-implicit account (wallet contract) submits an ERC-20 `transfer(address,uint256)` call via `rlp_execute`, the wallet contract maps it to a NEP-141 `ft_transfer` call. Before doing so, it first calls `storage_balance_of` on the token contract to check whether the receiver is registered (NEP-145 storage management standard).

The response is handled in `nep_141_storage_balance_callback`:

```rust
PromiseResult::Successful(value) => match serde_json::from_slice(&value) {
    Ok(x) => x,
    Err(_) => {
        return PromiseOrValue::Value(ExecuteResponse {
            success: false,
            success_value: None,
            error: Some("Unexpected response from NEP-141 storage_balance_of".into()),
        });
    }
},
``` [1](#0-0) 

The target type is `Option<StorageBalance>`, where `StorageBalance` (from `near_contract_standards`) requires `total` and `available` to be JSON strings encoding U128 values. If a NEP-141 token contract returns integer values instead of string values (e.g., `{"total": 1250000000000000000000, "available": 0}` instead of `{"total": "1250000000000000000000", "available": "0"}`), or any other non-conforming but otherwise valid JSON, `serde_json::from_slice` returns an error and the callback immediately returns `success: false`.

The ERC-20 emulated transfer path is initiated here:

```rust
Promise::new(token_id.clone())
    .function_call(
        "storage_balance_of".into(),
        storage_balance_args,
        NearToken::from_yoctonear(0),
        NEP_141_STORAGE_BALANCE_OF_GAS,
    )
    .then(ext.nep_141_storage_balance_callback(...))
``` [2](#0-1) 

There is no fallback path. If the deserialization fails, the transfer is aborted with no retry mechanism. The user's nonce is already incremented before this call: [3](#0-2) 

So every subsequent attempt with a new nonce will fail identically.

---

### Impact Explanation

Any user holding tokens in a NEP-141 contract whose `storage_balance_of` returns a non-conforming JSON response cannot transfer those tokens via the wallet contract's ERC-20 emulation interface. The `ExecuteResponse.success` value is incorrectly set to `false`, and the `ft_transfer` call is never dispatched. The tokens remain in the NEP-141 contract but are permanently inaccessible through the ERC-20 emulation path — the only transfer interface available to eth-implicit account users who interact via Ethereum tooling.

---

### Likelihood Explanation

The NEP-145 storage management standard specifies that `storage_balance_of` should return `Option<StorageBalance>` with U128 amounts encoded as strings. However, not all deployed NEP-141 token contracts strictly follow this encoding. A contract returning integer-encoded amounts (a common deviation) or any other non-standard but valid JSON will trigger this failure. Additionally, an attacker can deliberately deploy a token contract that returns a non-conforming `storage_balance_of` response to grief users who hold those tokens in their eth-implicit wallet accounts.

---

### Recommendation

Instead of hard-failing on deserialization error, treat an unparseable `storage_balance_of` response as `None` (unregistered) and proceed with the `storage_deposit` + `ft_transfer` path. This mirrors the intent of the `safeTransfer` pattern: tolerate non-conforming but otherwise functional external contracts.

```rust
PromiseResult::Successful(value) => {
    serde_json::from_slice(&value).unwrap_or(None)
}
```

This ensures that even if the token contract returns a non-standard response, the wallet contract still attempts the transfer (with a storage deposit if needed), rather than silently blocking it forever.

---

### Proof of Concept

1. Deploy a NEP-141 token contract whose `storage_balance_of` returns `{"total": 1250000000000000000000, "available": 0}` (integer values, not strings — a common non-conforming implementation).
2. Mint tokens to an eth-implicit wallet contract account.
3. Submit an ERC-20 `transfer(address,uint256)` via `rlp_execute` targeting the token contract.
4. The wallet contract calls `storage_balance_of` on the token contract.
5. `nep_141_storage_balance_callback` receives the response, `serde_json::from_slice::<Option<StorageBalance>>` fails because `U128` requires string-encoded values.
6. The callback returns `ExecuteResponse { success: false, error: Some("Unexpected response from NEP-141 storage_balance_of") }`.
7. The `ft_transfer` is never dispatched. The user's nonce is incremented. Every retry fails identically.
8. The user's tokens are permanently inaccessible via the ERC-20 emulation path. [4](#0-3) [5](#0-4)
