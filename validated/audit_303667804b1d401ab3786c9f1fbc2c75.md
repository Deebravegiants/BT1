### Title
Missing `maxCollateralInputs` Enforcement in Alonzo UTXO Transition Rule - (File: `eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Rules/Utxo.hs`)

### Summary
The Alonzo era `utxoTransition` function contains a comment indicating that the `maxCollateralInputs` protocol-parameter check should be enforced, but the actual `runTest $ validateTooManyCollateralInputs` call is absent. An unprivileged transaction sender can craft a transaction with an arbitrarily large number of collateral inputs, bypassing the `maxCollateralInputs` resource limit that the formal specification mandates.

### Finding Description

In `eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Rules/Utxo.hs`, the `utxoTransition` function runs all per-transaction UTXO checks. After enforcing `totExUnits tx ≤ maxTxExUnits pp`, the code contains a comment that mirrors the formal specification's requirement, but the corresponding `runTest` call is entirely absent:

```haskell
  {-   totExunits tx ≤ maxTxExUnits pp    -}
  runTest $ validateExUnitsTooBigUTxO pp tx

  {-   ‖collateral tx‖  ≤  maxCollInputs pp   -}
  -- ← NO runTest $ validateTooManyCollateralInputs pp txBody here

  updatedGovState <-
    trans @(EraRule "UTXOS" era) $ ...
``` [1](#0-0) 

The `validateTooManyCollateralInputs` function is defined and exported from the same module: [2](#0-1) 

The Alonzo formal specification explicitly requires both checks side-by-side in the UTxO-inductive rule:

```
totExunits tx ≤ maxTxExUnits pp   ‖collateral tx‖ ≤ maxCollateralInputs pp
``` [3](#0-2) 

By contrast, the Babbage era's `babbageUtxoValidation` and the Dijkstra era's `dijkstraUtxoTransition` both correctly call `runTest $ Alonzo.validateTooManyCollateralInputs pp txBody`: [4](#0-3) [5](#0-4) 

The `maxCollateralInputs` protocol parameter is defined to limit the total number of collateral inputs and, consequently, the number of additional VKey signatures that must be verified during validation: [6](#0-5) 

The `totExUnits` check is enforced via `runTest` (a dynamic check that always runs), while the collateral-count check is simply absent — not even downgraded to `runTestOnSignal` (a static/signal-only check): [7](#0-6) 

### Impact Explanation

**Medium.** An attacker-controlled transaction can include more collateral inputs than `maxCollateralInputs` permits. Each collateral input requires a VKey signature check during Phase-1 validation. Without the guard, the ledger accepts transactions that the formal specification and every subsequent era (Babbage, Conway, Dijkstra) would reject. This constitutes attacker-controlled transactions exceeding intended validation limits — specifically the `maxCollateralInputs` resource-limit parameter — outside design parameters. Because all Alonzo-era nodes share the same missing check, there is no inter-node divergence, but the protocol invariant that bounds per-transaction signature-verification work is violated.

### Likelihood Explanation

**Low-to-Medium.** The Alonzo era is historical (September 2021 – June 2022); new transactions cannot be submitted to it on mainnet today. However, the code path remains active for chain replay and historical block validation. Any tooling or node that re-validates the Alonzo portion of the chain will silently accept transactions that exceed `maxCollateralInputs`. The presence of the comment `{- ‖collateral tx‖ ≤ maxCollInputs pp -}` with no corresponding call strongly indicates an unintentional omission rather than a deliberate design choice.

### Recommendation

Add the missing enforcement call immediately after the `validateExUnitsTooBigUTxO` check in `utxoTransition`, matching the pattern used in Babbage and Dijkstra:

```haskell
  {-   totExunits tx ≤ maxTxExUnits pp    -}
  runTest $ validateExUnitsTooBigUTxO pp tx

  {-   ‖collateral tx‖  ≤  maxCollInputs pp   -}
  runTest $ validateTooManyCollateralInputs pp txBody
``` [1](#0-0) 

### Proof of Concept

1. Obtain the `maxCollateralInputs` value from the Alonzo-era protocol parameters (e.g., `3`).
2. Construct an Alonzo-era transaction with `maxCollateralInputs + N` (e.g., `10`) collateral inputs, each backed by a valid VKey UTxO and a corresponding signature.
3. Submit the transaction to an Alonzo-era ledger state.
4. Observe that `utxoTransition` does **not** call `validateTooManyCollateralInputs`, so the transaction passes Phase-1 validation despite violating the `maxCollateralInputs` bound.
5. Confirm that the same transaction submitted to a Babbage-era ledger state is correctly rejected with `TooManyCollateralInputs`.

The root cause is the absent `runTest $ validateTooManyCollateralInputs pp txBody` at line 570 of `eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Rules/Utxo.hs`, contradicting the formal specification at `eras/alonzo/formal-spec/utxo.tex` line 662 and the implementations in Babbage (`eras/babbage/impl/src/Cardano/Ledger/Babbage/Rules/Utxo.hs` line 412) and Dijkstra (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs` line 418).

validateTooManyCollateralInputs ::
  AlonzoEraTxBody era =>
  PParams era ->
  TxBody TopTx era ->
  Test (AlonzoUtxoPredFailure era)
validateTooManyCollateralInputs pp txBody =
  failureUnless (numColl <= maxColl) $
    TooManyCollateralInputs Mismatch {mismatchSupplied = numColl, mismatchExpected = maxColl}
  where
    maxColl, numColl :: Word16
    maxColl = pp ^. ppMaxCollateralInputsL
    numColl = fromIntegral . Set.size $ txBody ^. collateralInputsTxBodyL
