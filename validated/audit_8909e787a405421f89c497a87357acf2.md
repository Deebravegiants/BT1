### Title
Attacker Can Permanently Lock Reward Tokens by Staking Minimal Amount Before `notifyRewardAmount` Then Immediately Withdrawing - (File: contracts/KERNEL/KernelDepositPool.sol)

---

### Summary

`KernelDepositPool.notifyRewardAmount()` guards against unallocated rewards by requiring `totalKernelStaked > 0`, but there is no minimum stake amount enforced. An attacker can satisfy this guard with 1 wei of KERNEL, allow the admin to call `notifyRewardAmount`, then immediately call `initiateWithdrawal(1)` to reduce `totalKernelStaked` back to zero. Once `totalKernelStaked == 0`, `rewardPerToken()` is permanently frozen and all undistributed rewards for the period are locked in the contract forever.

---

### Finding Description

The `notifyRewardAmount` function contains the following guard:

```solidity
if (totalKernelStaked == 0) revert NoStakedTokens();
``` [1](#0-0) 

This is intended to prevent the admin from starting a reward period with no stakers, which would cause rewards to be unclaimable. However, the `stake` function only requires `_amount > 0` with no minimum:

```solidity
function stake(uint256 _amount) external nonReentrant updateReward(msg.sender) {
    if (_amount == 0) revert AmountZero();
``` [2](#0-1) 

And `initiateWithdrawal` reduces `totalKernelStaked` **immediately** (before the withdrawal delay):

```solidity
balanceOf[msg.sender] -= _amount;
totalKernelStaked -= _amount;
``` [3](#0-2) 

The `rewardPerToken()` function returns `rewardPerTokenStored` unchanged whenever `totalKernelStaked == 0`:

```solidity
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
``` [4](#0-3) 

The `updateReward` modifier updates `updatedAt` to `lastTimeRewardApplicable()` on every user action:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    updatedAt = lastTimeRewardApplicable();
``` [5](#0-4) 

When the attacker calls `initiateWithdrawal(1)`, `updateReward` runs, setting `updatedAt = now` and `rewardPerTokenStored` unchanged (since `totalKernelStaked` is still 1 at modifier entry, then drops to 0). From this point forward, `rewardPerToken()` returns the frozen `rewardPerTokenStored`. When the next user eventually stakes, `updateReward` runs again with `totalKernelStaked == 0`, so `rewardPerTokenStored` is still not updated and `updatedAt` is advanced again — permanently skipping over the zero-staked window. All rewards that should have accrued during that window are irrecoverably locked.

The contract's own NatSpec acknowledges this risk but relies on an off-chain operational assumption rather than on-chain enforcement:

```
* @dev If `totalKernelStaked` ever hits zero during a reward distribution window, any remaining rewards
*      for that period will stay locked in the contract.
``` [6](#0-5) 

This is the direct analog to the Timeswap `yIncrease` bug: a critical accumulator (`totalKernelStaked`) can be set to an arbitrarily small value (1 wei) that satisfies the guard check but causes the protocol to permanently malfunction for that reward period.

---

### Impact Explanation

All reward tokens deposited via `notifyRewardAmount` for the current period are permanently locked in the `KernelDepositPool` contract. No staker — present or future — can ever claim the rewards that should have accrued during the zero-staked window. This is **permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

Medium. The attacker needs only 1 wei of KERNEL (negligible cost) and must time their `initiateWithdrawal` call after the admin's `notifyRewardAmount`. This can be achieved by:
- Monitoring the mempool and front-running `notifyRewardAmount` with a `stake(1)` + back-running with `initiateWithdrawal(1)`, or
- Being the only staker at the time the admin calls `notifyRewardAmount` (e.g., at protocol launch when staking is sparse), then withdrawing immediately after.

The reward amount locked could be substantial (the entire period's reward allocation).

---

### Recommendation

Replace the `totalKernelStaked == 0` guard in `notifyRewardAmount` with a meaningful minimum threshold:

```solidity
uint256 public constant MIN_STAKED_FOR_REWARDS = 1e18; // e.g., 1 KERNEL

function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
    if (_amount == 0) revert AmountZero();
    if (totalKernelStaked < MIN_STAKED_FOR_REWARDS) revert InsufficientStakedTokens();
    ...
}
```

This mirrors the Timeswap recommendation of enforcing a minimum meaningful value (`2**32 / 10000`) for the initial pool state parameter.

---

### Proof of Concept

1. Attacker calls `stake(1)` → `totalKernelStaked = 1` (satisfies guard).
2. Admin calls `notifyRewardAmount(1_000_000e18)` → passes `totalKernelStaked > 0`, sets `rewardRate = 1_000_000e18 / duration`, `finishAt = block.timestamp + duration`.
3. Attacker calls `initiateWithdrawal(1)`:
   - `updateReward` runs: `rewardPerTokenStored` updated (tiny amount for 1 wei), `updatedAt = block.timestamp`.
   - `totalKernelStaked = 0`.
4. For all future calls, `rewardPerToken()` returns the frozen `rewardPerTokenStored`.
5. When the next legitimate user calls `stake(largeAmount)`, `updateReward` runs with `totalKernelStaked == 0` → `rewardPerTokenStored` still frozen, `updatedAt` advanced to now.
6. All 1,000,000 reward tokens allocated for the period from step 3 to step 5 are permanently locked in the contract. No user can ever claim them.
