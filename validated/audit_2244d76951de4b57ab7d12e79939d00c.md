### Title
Stub `error` in `toPlutusScriptPurpose` for PlutusV4 Causes Unhandled Runtime Exception During Script Execution - (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxInfo.hs`)

---

### Summary

The `EraPlutusTxInfo 'PlutusV4 DijkstraEra` instance in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxInfo.hs` implements `toPlutusScriptPurpose` as a hard `error` stub. Because `EraPlutusContext DijkstraEra` simultaneously declares PlutusV4 as a **supported language** via `mkSupportedLanguage` and routes `DijkstraPlutusV4` scripts through the same `toPlutusWithContext → toPlutusArgs → toPlutusScriptPurpose` execution pipeline used by all other language versions, any transaction that causes a PlutusV4 script to be evaluated will throw an impure Haskell exception rather than returning a graceful `Left (ContextError …)`. This crashes the validating node rather than cleanly rejecting the transaction, creating a deterministic ledger-divergence vector.

A block producer who crafts a block that bypasses or precedes the `eraMaxLanguage` check (e.g., by exploiting a deserialization path that constructs a `DijkstraPlutusV4` value without going through the UTxO validation rule), or any future code path that calls `mkPlutusWithContext` on a `DijkstraPlutusV4` script, will unconditionally crash the node with an unhandled `ErrorCall`.

### Impact Explanation

`error` in Haskell throws an impure `ErrorCall` exception. Unlike `Left (ContextError …)`, it is **not** caught by the `Either` monad used throughout the script-context pipeline. Unless there is an explicit `catch`/`try` at the top of the block-validation loop (which is not present in the standard STS evaluation path in `libs/small-steps/src/Control/State/Transition/Extended.hs`), the exception propagates and terminates the node process. Nodes that crash on a given block diverge from nodes that do not encounter the same code path, satisfying the **High — deterministic disagreement between honest nodes from script/witness validation** impact criterion.

---

### Likelihood Explanation

The `EraPlutusContext DijkstraEra` instance explicitly registers PlutusV4 as supported (`mkSupportedLanguage PlutusV4 = Just …`) and `mkPlutusScript` accepts `SPlutusV4` without restriction. Any transaction that carries a `DijkstraPlutusV4` script and reaches the two-phase script execution step will trigger the stub. The `eraMaxLanguage = PlutusV3` guard may prevent this in the standard UTxO validation path, but the inconsistency between the two settings means the guard is not enforced at the `EraPlutusContext` layer, leaving the crash reachable through any alternative entry point (deserialization, future era upgrade code, or a block that passes the `eraMaxLanguage` check through a protocol-version negotiation edge case).

---

### Recommendation

1. **Immediate**: Change `toPlutusScriptPurpose` to return a proper `Left (ContextError …)` value (e.g., a new `PlutusV4NotYetImplemented` constructor in `DijkstraContextError`) instead of calling `error`. This converts a node-crashing impure exception into a graceful transaction rejection.

2. **Consistency fix**: Either set `eraMaxLanguage = PlutusV4` (once the implementation is complete) or change `mkSupportedLanguage PlutusV4` to return `Nothing` to match the `eraMaxLanguage = PlutusV3` gate. The two settings must agree.

3. **Complete the implementation**: Provide a real `toPlutusScriptPurpose` for `'PlutusV4 DijkstraEra`, analogous to `Conway.transPlutusPurposeV3` used for PlutusV3.
