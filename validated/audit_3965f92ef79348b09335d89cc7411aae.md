### Title
Wrong-Curve Public Share Accepted in MP-DKG Round 2 Due to Missing Curve-Identity Guard - (`src/cbmpc/protocol/ec_dkg.cpp`)

---

### Summary

In `key_share_mp_t::dkg`, a Byzantine peer can broadcast a public key share `Qi._j` serialized on a different curve (e.g., ed25519 in a secp256k1 session). The code comment at line 157 states *"curve check of Qi._j is done inside the zk verify function"*, but `uc_dl_t::verify` infers the curve from `Q.get_curve()` — the attacker-controlled point itself — and only checks that the point is on *its own* curve, not the session's expected curve. A valid ZK proof for the injected curve passes verification, and the wrong-curve point is stored in `key.Qis`.

---

### Finding Description

**Deserialization embeds the curve from the wire.**

`ecc_point_t::convert` reads the curve code from the serialized message:

```cpp
void ecc_point_t::convert(coinbase::converter_t& converter) {
  ecurve_t c = curve;
  c.convert(converter);   // reads uint16_t openssl_code from wire
  if (!c) return;
  convert_fixed_curve(converter, c);
}
``` [1](#0-0) 

`ecurve_t::convert` calls `ecurve_t::find(curve_code)`, which iterates `g_curves[]` (which includes `ed25519_info`) and returns whichever curve matches the wire-supplied `openssl_code`:

```cpp
ecurve_t ecurve_t::find(int openssl_id) {
  for (int i = 0; i < sizeof(g_curves) / sizeof(g_curves[0]); i++) {
    ecurve_t curve = ecurve_t(g_curves[i]);
    if (curve.type() == ecurve_type_e::ossl && !curve.get_group()) continue;
    if (openssl_id == curve.get_openssl_code()) return curve;
  }
  ...
}
``` [2](#0-1) 

`convert_fixed_curve` then calls `curve.check(*this)` where `curve` is the *wire* curve, not the session curve:

```cpp
if (rv = curve.check(*this)) { converter.set_error(rv); return; }
``` [3](#0-2) 

`ecurve_t::check` only verifies `point.get_curve() == *this` — i.e., the point is on its own curve — not the session's expected curve:

```cpp
error_t ecurve_t::check(const ecc_point_t& point) const {
  if (point.get_curve() != *this) return crypto::error("EC-point of wrong curve");
  if (!point.is_in_subgroup()) return crypto::error("EC-point is not on curve");
  ...
}
``` [4](#0-3) 

**ZK verify infers the curve from the attacker-controlled point.**

`uc_dl_t::verify` does:

```cpp
ecurve_t curve = Q.get_curve();   // ← attacker-controlled
const mod_t& q = curve.order();
if (rv = curve.check(Q)) return ...;  // trivially passes: Q is on its own curve
const auto& G = curve.generator();   // ← wrong generator
...
if (A_sum != z_sum * G - e_sum * Q) return ...;
``` [5](#0-4) 

The attacker generates a valid ZK proof for the injected curve; verification passes because both `G` and `Q` are on the injected curve.

**The missing guard in `key_share_mp_t::dkg`.**

After the commitment is opened, the code relies entirely on the ZK verify for curve validation:

```cpp
if (rv = crypto::commitment_t(sid_i._j, job.get_pid(j)).set(rho._j, c._j).open(Qi._j)) return rv;

// curve check of Qi._j is done inside the zk verify function
if (rv = pi._j.verify(Qi._j, sid, j)) return rv;
``` [6](#0-5) 

There is no `if (Qi._j.get_curve() != curve) return error(E_CRYPTO)` guard. The comment is factually wrong.

The wrong-curve point is then stored and summed:

```cpp
for (int j = 0; j < n; j++) {
    key.Qis[job.get_name(j)] = Qi._j;
}
key.Q = SUM(key.Qis);
``` [7](#0-6) 

`ecc_point_t::add` has only a `cb_assert` (debug-only) for curve mismatch:

```cpp
ecc_point_t ecc_point_t::add(const ecc_point_t& val1, const ecc_point_t& val2) {
  cb_assert(val1.curve == val2.curve && "ecc_point_t::add: curve mismatch");
  ...
}
``` [8](#0-7) 

In a release build where `cb_assert` is a no-op, the addition proceeds with undefined behaviour across incompatible internal representations (OpenSSL `EC_POINT*` vs. ed25519 `ecp_storage_t*`), potentially producing a garbage `key.Q` that the DKG returns as SUCCESS.

**The `h_consistency` check does not close the gap.**

The consistency hash only covers the curve *name string*, not the curve of the actual points:

```cpp
h_consistency._i = crypto::sha256_t::hash(std::string(curve.get_name()));
``` [9](#0-8) 

The attacker sends the correct curve name in `h_consistency` while sending a point on a different curve in `Qi`.

---

### Impact Explanation

A single Byzantine peer below threshold can inject a public key share on an arbitrary supported curve (ed25519, P-256, P-384, P-521) into a secp256k1 (or any other) DKG session. The ZK proof for the injected curve passes verification. In a release build the DKG may return SUCCESS with `key.Qis[j]` holding a wrong-curve point and `key.Q` holding a garbage value derived from mixing incompatible curve arithmetic. This violates the invariant that all public shares are on the session curve and corrupts the group public key, satisfying the High-impact criterion: *"attacker-controlled points accepted under the wrong curve."*

---

### Likelihood Explanation

Any single Byzantine participant in an MP-DKG session can trigger this. No threshold collusion is required. The attacker only needs to craft a serialized `ecc_point_t` with a different `openssl_code` and a matching valid ZK proof for that curve — both are straightforward to produce.

---

### Recommendation

Add an explicit curve-identity check immediately after deserializing each peer's `Qi._j`, before the ZK verify call:

```cpp
if (Qi._j.get_curve() != curve)
    return coinbase::error(E_CRYPTO, "ec_dkg: Qi from wrong curve");
```

Remove the misleading comment. The same pattern should be applied in `key_share_mp_t::refresh` for `R._j[l]` (line 222 comment: *"Curve check of R._j[l] is done inside the zk verify function further below"*) and in `dkg_or_refresh_ac` (line 390 comment: *"Verifying that R values are on the curve and subgroup is done in the zk verify function"*). ### Proof of Concept

In a 3-party secp256k1 DKG session, party 2 (Byzantine) does the following instead of the honest round-2 broadcast:

1. Pick `x_bad` ← random scalar mod ed25519 order.
2. Compute `Qi_bad = x_bad * G_ed25519`.
3. Serialize `Qi_bad` with `openssl_code = NID_ED25519 (1087)`.
4. Generate `pi_bad` = `uc_dl_t::prove(Qi_bad, x_bad, sid, 2)` (valid ed25519 proof).
5. Commit to `Qi_bad` in round 1 (commitment is curve-agnostic).
6. Broadcast `(sid_msg, h, Qi_bad, rho, pi_bad)` in round 2.

Honest parties 0 and 1:
- Deserialize `Qi_bad` → `ecc_point_t` with `curve = ed25519`. ✓ (passes `convert_fixed_curve` check)
- `pi_bad.verify(Qi_bad, sid, 2)` → uses ed25519 generator → **returns SUCCESS**. ✓
- Store `key.Qis["p2"] = Qi_bad` (ed25519 point).
- `key.Q = SUM(key.Qis)` → mixes secp256k1 + ed25519 points → UB / garbage in release build.

Assert: `uc_dl_t::verify` returns `SUCCESS` for an ed25519 point in a secp256k1 session when the proof was generated for ed25519. The missing guard `Qi._j.get_curve() != curve` is never evaluated.
