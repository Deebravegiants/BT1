Audit Report

## Title
ETH Deposit Limit Not Enforced at Exact Boundary Due to Strict Inequality - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses a strict `>` comparison for ETH deposits, meaning when `totalAssetDeposits` equals exactly `depositLimitByAsset(ETH_TOKEN)`, the check returns `false` and the deposit is not blocked. Any unprivileged depositor can call `depositETH` at this boundary to mint rsETH beyond the configured cap. The ERC-20 branch correctly uses `totalAssetDeposits + amount > limit`, which always reverts for non-zero amounts when the limit is fully consumed.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676â€“682, `_checkIfDepositAmountExceedesCurrentLimit` has two branches:

```solidity
// L678-681
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));  // strict >
}
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
```

The ETH branch omits `amount` entirely and uses strict `>`. When `totalAssetDeposits == depositLimit`, the expression evaluates to `false`, so `_beforeDeposit` (lines 661â€“663) does not revert and the deposit proceeds. The ERC-20 branch includes `amount`, so `totalAssetDeposits + amount > limit` is always `true` for any non-zero deposit when the limit is fully consumed â€” correctly blocking it.

`getAssetCurrentLimit` (lines 402â€“409) already treats `totalAssetDeposits == limit` as zero remaining capacity (returns `limit - totalAssetDeposits = 0`), creating an observable inconsistency: the public view reports zero capacity while the enforcement gate still admits ETH deposits.

## Impact Explanation
An unprivileged depositor can mint rsETH against ETH that exceeds the protocol's configured deposit ceiling. The deposit limit is the primary mechanism preventing over-acceptance of ETH. Bypassing it at the exact boundary violates the protocol's stated deposit constraints without any loss of funds to existing users. This maps to **Low â€” contract fails to deliver promised returns, but doesn't lose value**, which is within the allowed impact scope.

## Likelihood Explanation
The boundary condition `totalAssetDeposits == depositLimit` is reachable in two ways without requiring privileged access: (1) an admin sets `depositLimitByAsset` to exactly the current TVL to freeze deposits â€” immediately exploitable by any depositor; (2) a sequence of normal deposits naturally lands the cumulative total at the limit, after which any subsequent `depositETH` call bypasses the cap. Scenario (1) is directly exploitable the moment the admin action completes. Likelihood is **Low** but non-negligible given realistic admin usage patterns.

## Recommendation
Replace the strict inequality with a non-strict one in the ETH branch of `_checkIfDepositAmountExceedesCurrentLimit` (line 679):

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits >= lrtConfig.depositLimitByAsset(asset));
}
```

This aligns the enforcement gate with `getAssetCurrentLimit`, which already treats `totalAssetDeposits == depositLimit` as zero remaining capacity.
