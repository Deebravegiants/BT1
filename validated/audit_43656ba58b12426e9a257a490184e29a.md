### Title
`UseGlobalContract` Action Lacks Account-Type Restriction, Allowing Replacement of ETH-Implicit Wallet Contract - (File: `runtime/runtime/src/global_contracts.rs`)

### Summary
The `UseGlobalContract` action can be applied to any account type, including ETH-implicit accounts, without restriction. ETH-implicit accounts are supposed to always use the wallet contract (which enforces Ethereum signature verification). An unprivileged attacker can replace the wallet contract of any ETH-implicit account with an arbitrary global contract, breaking the core security invariant and enabling fund theft.

### Finding Description
The `use_global_contract` function performs no check on the account type of the receiver: [1](#0-0) 

`validate_use_global_contract_action` only validates the contract identifier format, with no account-type gate: [2](#0-1) 

`check_account_existence` for `UseGlobalContract` only verifies the account exists, not its type: [3](#0-2) 

ETH-implicit accounts are created with the wallet contract deployed as a global contract, enforcing that only the owner of the corresponding Ethereum private key can execute actions: [4](#0-3) 

The documentation confirms this invariant — ETH-implicit accounts cannot be deleted, nor can a full access key be added, and they can only be used by calling the wallet contract: [5](#0-4) 

Since `UseGlobalContract` has no account-type restriction, any unprivileged user can send a transaction with `receiver_id` = ETH-implicit account and `actions = [UseGlobalContract(malicious_contract_hash)]`, replacing the wallet contract with an arbitrary global contract. The corrupted state is the `AccountContract` field of the ETH-implicit account, which changes from `AccountContract::Global(wallet_contract_hash)` to `AccountContract::Global(malicious_contract_hash)`. [6](#0-5) 

### Impact Explanation
After replacing the wallet contract:
1. The ETH-implicit account's Ethereum signature verification is bypassed — the wallet contract that enforces `rlp_execute` signature checks is gone.
2. The attacker can send a second transaction FROM their own account TO the ETH-implicit account with a `FunctionCall` action targeting a method in the malicious contract.
3. The malicious contract executes in the context of the ETH-implicit account and can issue transfer promises, draining the account's balance to the attacker.

The exact corrupted protocol value is `account.contract` (the `AccountContract` stored in the trie for the ETH-implicit account), which determines what code executes when the account receives function calls.

### Likelihood Explanation
The attack requires:
1. Any NEAR account with a full access key (trivially obtainable).
2. Deploying a malicious global contract (permissionless, costs only storage fees).
3. Sending two transactions.

No validator, node admin, or trusted-service privilege is required. The attack is fully executable by any unprivileged external user via public RPC transactions.

### Recommendation
Add an account-type check in `use_global_contract` (or in `validate_use_global_contract_action`) to reject `UseGlobalContract` actions targeting ETH-implicit accounts. ETH-implicit accounts should only have their contract set during account creation via `action_implicit_account_creation_transfer`. A check analogous to the existing `OnlyImplicitAccountCreationAllowed` guard for `CreateAccount` should be added:

```rust
pub(crate) fn use_global_contract(...) -> Result<(), RuntimeError> {
    // Reject UseGlobalContract on ETH-implicit accounts; their contract
    // is fixed to the wallet contract and must not be overwritten.
    if account_id.get_account_type() == AccountType::EthImplicitAccount {
        result.result = Err(ActionErrorKind::UseGlobalContractNotAllowed {
            account_id: account_id.clone(),
        }.into());
        return Ok(());
    }
    // ... existing logic
}
```

### Proof of Concept
1. Attacker deploys a malicious global contract containing a `steal_funds()` method that issues a transfer promise to the attacker's account.
2. Attacker sends transaction: `signer_id=attacker.near`, `receiver_id=0x<victim_eth_implicit>`, `actions=[UseGlobalContract(malicious_hash)]`.
3. `use_global_contract` executes without any account-type check, replacing the wallet contract with the malicious contract.
4. Attacker sends transaction: `signer_id=attacker.near`, `receiver_id=0x<victim_eth_implicit>`, `actions=[FunctionCall("steal_funds", gas=300T, deposit=0)]`.
5. The malicious contract executes in the ETH-implicit account's context and transfers all funds to the attacker. [7](#0-6) [2](#0-1) [3](#0-2)

pub(crate) fn use_global_contract(
    state_update: &mut TrieUpdate,
    account_id: &AccountId,
    account: &mut Account,
    contract_identifier: &GlobalContractIdentifier,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    let key = TrieKey::GlobalContractCode { identifier: contract_identifier.clone().into() };
    if !state_update.contains_key(&key, AccessOptions::DEFAULT)? {
        result.result = Err(ActionErrorKind::GlobalContractDoesNotExist {
            identifier: contract_identifier.clone(),
        }
        .into());
        return Ok(());
    }
    clear_account_contract_storage_usage(state_update, account_id, account)?;
    if account.contract().is_local() {
        state_update.remove(TrieKey::ContractCode { account_id: account_id.clone() });
    }
    let contract = match contract_identifier {
        GlobalContractIdentifier::CodeHash(code_hash) => AccountContract::Global(*code_hash),
        GlobalContractIdentifier::AccountId(id) => AccountContract::GlobalByAccount(id.clone()),
    };
    account.set_storage_usage(
        account.storage_usage().checked_add(contract_identifier.len() as u64).ok_or_else(|| {
            StorageError::StorageInconsistentState(format!(
                "Storage usage integer overflow for account {}",
                account_id
            ))
        })?,
    );
    account.set_contract(contract);
    Ok(())
}
