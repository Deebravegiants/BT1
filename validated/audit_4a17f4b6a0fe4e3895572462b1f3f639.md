### Title
`maxBlockExUnits` Ceiling Bypassed via Sub-Transaction Execution Units in Dijkstra Era — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs`)

---

### Summary

In the Dijkstra era, the block-level execution-unit ceiling (`maxBlockExUnits`) is enforced by summing `totExUnits` over only the top-level transactions in a block. Sub-transactions, which carry their own independent witness sets and redeemers, are never included in this sum. A block producer can therefore craft a single top-level transaction containing many sub-transactions, each loaded with execution units up to `maxTxExUnits`, causing the actual computation required to validate the block to far exceed `maxBlockExUnits` while the ledger check passes.

---

### Finding Description

The `maxBlockExUnits` protocol parameter is intended to bound the total Plutus script execution cost per block. The check is implemented in `validateExUnits`:

```haskell
-- eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Rules/Bbody.hs:158-167
validateExUnits txs ppMax =
  let txTotal = foldMap totExUnits txs
   in pointWiseExUnits (<=) txTotal ppMax ?! ...
``` [1](#0-0) 

`totExUnits` is defined as:

```haskell
-- eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Tx.hs:390-394
totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL
``` [2](#0-1) 

This reads only from the **top-level** transaction's witness set. It does not traverse sub-transactions.

The Dijkstra BBODY rule calls this same function unchanged:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs:361
Alonzo.validateExUnits @era txs $ pp ^. ppMaxBlockExUnitsL
``` [3](#0-2) 

However, Dijkstra sub-transactions are full transactions with their own witness sets, including redeemers. The CDDL schema confirms this:

```
sub_transaction =
  [sub_transaction_body, transaction_witness_set, auxiliary_data/ nil]
``` [4](#0-3) 

The `transaction_witness_set` field includes `redeemers`, meaning sub-transactions carry their own execution-unit budgets. [5](#0-4) 

The contrast with the reference-script size limit is instructive. For that limit, the developers explicitly wrote `batchNonDistinctRefScriptsSize` to aggregate across sub-transactions:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs:271-277
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum (foldMap' (Sum . txNonDistinctRefScriptsSize utxo)
                       (tx ^. bodyTxL . subTransactionsTxBodyL))
``` [6](#0-5) 

No equivalent aggregation exists for execution units. The `validateAllRefScriptSize` check in the Dijkstra LEDGER rule uses `batchNonDistinctRefScriptsSize` (which includes sub-transactions), but the block-level `validateExUnits` uses `totExUnits` (which does not). [7](#0-6) 

---

### Impact Explanation

The `maxBlockExUnits` parameter exists to bound the CPU and memory cost of validating a block, ensuring nodes can keep up with the chain. By bypassing this ceiling, a block producer can force all honest nodes to execute arbitrarily more Plutus script computation than the protocol intends to allow per block. This constitutes attacker-controlled transactions exceeding intended validation limits, matching the **Medium** impact category.

---

### Likelihood Explanation

The attacker must be a block producer (slot leader), which is a realistic role for any sufficiently staked SPO. No privileged key or governance majority is required. The attack requires only constructing a valid top-level transaction with many sub-transactions, each carrying redeemers up to `maxTxExUnits`. This is straightforward to craft once the Dijkstra era is active.

---

### Recommendation

Replace the `totExUnits` call in the block-level check with a batch-aware aggregation that sums execution units across both the top-level transaction and all its sub-transactions, analogous to how `batchNonDistinctRefScriptsSize` was introduced for reference-script size. Specifically, define a `batchTotExUnits` function:

```haskell
batchTotExUnits tx =
  totExUnits tx
    + foldMap totExUnits (tx ^. bodyTxL . subTransactionsTxBodyL)
```

and use it in `validateExUnits` when operating in the Dijkstra era.

---

### Proof of Concept

1. Set `maxBlockExUnits = ExUnits { mem = M, steps = S }` and `maxTxExUnits = ExUnits { mem = m, steps = s }` where `m < M` and `s < S`.
2. Craft `N` sub-transactions, each with redeemers totalling `ExUnits { mem = m, steps = s }` (just at the per-tx limit).
3. Wrap them in a single top-level transaction with zero redeemers of its own.
4. The top-level tx passes `validateExUnitsTooBigUTxO` (0 ≤ maxTxExUnits).
5. `validateExUnits` in BBODY computes `foldMap totExUnits [topTx] = ExUnits 0 0`, which passes `≤ maxBlockExUnits`.
6. The actual execution cost is `N × ExUnits { mem = m, steps = s }`. With `N = ⌈M/m⌉ + 1`, the real cost exceeds `maxBlockExUnits` while the ledger check passes.
7. All honest nodes must execute all sub-transaction scripts to validate the block, consuming computation far beyond the intended ceiling.
