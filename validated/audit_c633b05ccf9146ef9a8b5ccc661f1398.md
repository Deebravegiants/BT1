## Analysis

### Tracing `threshold_parameters()` During Resharing

The first claim is confirmed. During `Resharing`, `threshold_parameters()` returns the **old** running state's parameters: [1](#0-0) 

```rust
ProtocolContractState::Resharing(resharing_contract_state) => {
    Ok(&resharing_contract_state.previous_running_state.parameters)
}
```

### Tracing `vote_update_foreign_chain_providers` During Resharing

The function calls `voter_or_panic()` first, then fetches `threshold_parameters()` (which returns OLD params during resharing), then authenticates the caller against the OLD participant set: [2](#0-1) 

So a departing participant P — still in the OLD set — passes both `voter_or_panic()` and `AuthenticatedParticipantId::new(threshold_parameters.participants())`. Their vote is recorded and counted against the OLD threshold.

### Tracing `ProviderVotes::vote` Count Logic

The `count_for` predicate filters against `threshold_parameters.participants()`, which during resharing is the OLD set: [3](#0-2) 

P's vote counts. If P + (OLD_THRESHOLD − 1) other participants vote the same entry, the chain is whitelisted immediately and pending rows are cleared.

### Tracing `clean_foreign_chain_data`

After resharing completes, `clean_foreign_chain_data` is spawned as a detached promise: [4](#0-3) 

Its implementation only calls `votes.retain(participants)` (clears **pending** votes) and `remove_stale_configs` (clears node config registrations). It does **not** touch `AllowedProviders.entries` — the applied whitelist: [5](#0-4) 

The whitelist entry persists. This is confirmed by the existing test `clean_foreign_chain_data__should_drop_departed_participant_contribution_from_cache`, which only tests the `available_foreign_chains` cache, not the whitelist entries themselves. [6](#0-5) 

### Why This Does Not Constitute a Vulnerability

The proof-of-concept path is technically reachable, but it fails the bounty-grade bar for the following reasons:

**1. Requires (OLD_THRESHOLD − 1) colluding new participants.**
P alone cannot tip the balance. P needs OLD_THRESHOLD − 1 other participants to vote the identical `(providers, quorum)` pair. Those co-voters are all continuing participants in the new epoch. If they are honest, they will not vote for a malicious provider entry. The attack therefore requires (OLD_THRESHOLD − 1) malicious new participants — a collusion requirement.

**2. The whitelist entry does not directly authorize signing.**
The whitelist controls which RPC providers nodes trust for foreign-chain receipt verification. It is a prerequisite for `available_foreign_chains`, which additionally requires the new epoch's reconstruction threshold of participants to have registered support for the chain: [7](#0-6)

    pub(crate) fn update_available_chains_config_cache(
        &mut self,
        active_tls_keys: &BTreeSet<dtos::Ed25519PublicKey>,
        threshold: u64,
    ) {
        let mut chain_to_supporter_count: std::collections::BTreeMap<dtos::ForeignChain, u64> =
            std::collections::BTreeMap::new();
        for tls_key in active_tls_keys {
            let Some(chains) = self.foreign_chains_configs.get(tls_key) else {
                continue;
            };
            for chain in chains.iter() {
                if self.rpc_whitelist.entries.is_whitelisted(chain) {
                    let count = chain_to_supporter_count.entry(*chain).or_default();
                    *count = count
                        .checked_add(1)
                        .expect("supporter count bounded by participant set size");
                }
            }
        }
        self.available_foreign_chains = chain_to_supporter_count
            .into_iter()
            .filter_map(|(chain, count)| (count >= threshold).then_some(chain))
            .collect::<BTreeSet<_>>()
            .into();
    }
