### Title
DNS Seeder `good_peers` Poisoning via Unauthenticated `NewPeak` Height Claim — (`chia/seeder/crawler.py`, `chia/seeder/peer_record.py`)

### Summary
An unprivileged remote peer that completes the Chia TLS handshake and sends a single `NewPeak` message with `height >= minimum_height` is unconditionally marked as a reliable peer and inserted into the `good_peers` table served by the DNS seeder. No chain validation, weight proof, or block hash check is performed. The `is_reliable()` threshold of 1 success out of ≤3 tries is satisfied by a single attacker-controlled connection.

### Finding Description
In `Crawler.new_peak` (`crawler.py:342–344`), the only condition checked before calling `crawl_store.peer_connected_hostname(host, True)` is `request.height >= self.minimum_height`. This triggers `peer_connected` → `PeerReliability.update(True, age)`, which increments `tries` to 1 and `successes` to 1.

`PeerReliability.is_reliable()` (`peer_record.py:94`) returns `True` when `tries > 0 and tries <= 3 and successes * 2 >= tries`. After one attacker-controlled success, all three conditions hold. On the next crawl cycle, `save_to_db` → `load_reliable_peers_to_db` iterates all peers, calls `is_reliable()`, and inserts matching peers into the `good_peers` SQLite table. The DNS seeder reads exclusively from this table via `get_good_peers`.

An attacker needs only to:
1. Have their IP gossiped into the crawler's peer list (trivially achieved by having any existing node include them in a `RespondPeers` message, or by being reachable when the crawler connects).
2. Accept the crawler's outbound TCP/TLS connection.
3. Send one `NewPeak(height=minimum_height)`.

### Impact Explanation
New nodes bootstrapping from the DNS seeder receive attacker IPs. With enough attacker-controlled addresses in `good_peers` (achievable by running many cheap VPS instances and getting them gossiped into the crawler), bootstrapping nodes can be eclipsed, preventing them from receiving valid blocks or sync updates. This matches the **High** impact category: long-lived inability for honest nodes to sync under normal network assumptions.

### Likelihood Explanation
The attack requires no special privileges. Getting an IP into the crawler's peer list is straightforward — any existing node can gossip attacker IPs via `RespondPeers`. The crawler connects outbound to all discovered peers. The attacker only needs to respond to the connection and send one message. The `is_reliable()` threshold of 1/1 is extremely low.

### Recommendation
1. **Remove the permissive `tries <= 3` branch** from `is_reliable()`, or raise the minimum success count significantly (e.g., require `stat_2h` or `stat_8h` thresholds to be met instead).
2. **Do not use `NewPeak` height alone as a connectivity signal** for `good_peers` eligibility. At minimum, require multiple independent successful connections over time before a peer is considered reliable.
3. Consider requiring a peer to pass a block header or weight proof challenge before being marked as connected with `True`.

### Proof of Concept
```python
# Minimal PoC sketch:
# 1. Spin up a TCP server that completes the Chia TLS+handshake protocol.
# 2. On connection from crawler, respond to RequestPeers with an empty list.
# 3. Send NewPeak(height=minimum_height, ...) unsolicited.
# 4. After one crawl cycle (save_to_db -> load_reliable_peers_to_db),
#    query the crawler DB: SELECT * FROM good_peers WHERE ip = '<attacker_ip>';
#    -> attacker IP appears after a single connection.
```

The root cause is the combination of: (a) no chain validation in `Crawler.new_peak`, and (b) `is_reliable()` returning `True` after a single success.
