Audit Report

## Title
Uninitialized `minRsEthAmountToWithdraw` Enables Queue Flooding and Unbounded Gas Consumption in `_unlockWithdrawalRequests` - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`minRsEthAmountToWithdraw` defaults to `0` for every asset because `initialize()` never sets it, making the dust-amount guard in `initiateWithdrawal` vacuously ineffective. Any rsETH holder can flood the FIFO withdrawal queue with 1-wei requests. Because `_unlockWithdrawalRequests` iterates the queue sequentially, legitimate withdrawals queued behind the dust entries are temporarily frozen, and a sufficiently large flood causes the operator's unlock transaction to revert out-of-gas.

## Finding Description
`minRsEthAmountToWithdraw` is declared as a plain mapping with no initialization:

```solidity
// L35
mapping(address asset => uint256) public minRsEthAmountToWithdraw;
```

`initialize()` (L90–98) sets only `withdrawalDelayBlocks` and `lrtConfig`; it never assigns a value to `minRsEthAmountToWithdraw` for any asset. The guard in `initiateWithdrawal` is:

```solidity
// L162–164
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```

When the mapping value is `0`, the condition `rsETHUnstaked < 0` is always `false` for `uint256`, so any amount ≥ 1 wei passes. Each accepted call appends a new entry to the global FIFO queue (L756–757) with no per-user cap.

The operator's unlock path calls `_unlockWithdrawalRequests` (L770–816), which runs a `while` loop from `nextLockedNonce[asset]` to `firstExcludedIndex` (L790–814). Each iteration performs multiple storage reads and writes (`withdrawalRequests`, `assetsCommitted`, `unlockedWithdrawalsCount`, `nextLockedNonce`). The only early-exit conditions are:

- Withdrawal delay not yet elapsed (L795) — passes after the delay period.
- Insufficient available assets (L800) — dust requests have negligible payout, so this rarely triggers.

An attacker submitting N dust requests forces the operator to iterate through all N entries before reaching any legitimate request queued afterward. With a large enough N, the loop exhausts the block gas limit in a single transaction. Even with a bounded `firstExcludedIndex`, the operator must issue many transactions to drain the dust entries, temporarily freezing legitimate users' rsETH (transferred to the contract at L166 and irrecoverable until the operator processes all preceding entries).

## Impact Explanation
**Medium — Unbounded gas consumption / Temporary freezing of funds.**

Legitimate withdrawers whose requests are queued behind a flood of dust entries cannot complete their withdrawals until the operator processes every preceding entry. The operator's unlock transaction gas cost scales linearly with the number of queued entries; a sufficiently large flood causes the transaction to revert out-of-gas, stalling the entire withdrawal queue for the affected asset. Both "Medium. Unbounded gas consumption" and "Medium. Temporary freezing of funds" are concrete, in-scope impacts.

## Likelihood Explanation
Any rsETH holder — an unprivileged, externally reachable role — can call `initiateWithdrawal` with 1-wei amounts. The attacker's net cost is only gas per call plus a temporary lock-up of negligible rsETH (recovered after the withdrawal delay). No special role, coordination, or oracle manipulation is required. The attack is repeatable and cheap relative to the disruption caused.

## Recommendation
1. Initialize `minRsEthAmountToWithdraw` to a sensible non-zero default (e.g., `1e15` wei) for each supported asset in `initialize()`, or require it to be set before an asset is usable for withdrawals.
2. Enforce the minimum strictly so that a zero value is never treated as "no minimum":
   ```solidity
   uint256 minAmount = minRsEthAmountToWithdraw[asset];
   if (rsETHUnstaked == 0 || (minAmount > 0 && rsETHUnstaked < minAmount)) {
       revert InvalidAmountToWithdraw();
   }
   ```
   Or simply ensure `minRsEthAmountToWithdraw[asset]` is always non-zero before the asset is supported.
3. Consider adding a per-user cap on pending withdrawal requests to further limit queue flooding.

## Proof of Concept
1. Admin deploys `LRTWithdrawalManager` and supports asset ETH. `minRsEthAmountToWithdraw[ETH]` is `0` (never set in `initialize()`).
2. Attacker acquires 10,000 wei of rsETH (negligible cost).
3. Attacker calls `initiateWithdrawal(ETH, 1, "")` 10,000 times. Each call passes the check at L162 (`1 != 0` and `1 < 0` is false). Each call appends a new entry to the queue (L756–757).
4. Legitimate user calls `initiateWithdrawal(ETH, 1 ether, "")`. Their request is at nonce 10,000.
5. After the withdrawal delay, operator calls `unlockQueue` with `firstExcludedIndex = nextUnusedNonce[ETH]` (10,001). The `_unlockWithdrawalRequests` while loop at L790 iterates all 10,001 entries, each requiring multiple storage ops, exhausting the block gas limit and reverting.
6. Even if the operator batches in chunks by passing a smaller `firstExcludedIndex`, the legitimate user's withdrawal at nonce 10,000 is delayed until all 10,000 dust entries are drained, temporarily freezing their funds.

**Foundry test sketch:**
```solidity
function testQueueFlood() public {
    // Attacker mints/acquires rsETH dust
    for (uint i = 0; i < 10_000; i++) {
        vm.prank(attacker);
        withdrawalManager.initiateWithdrawal(ETH, 1, "");
    }
    // Legitimate user queues
    vm.prank(legitimateUser);
    withdrawalManager.initiateWithdrawal(ETH, 1 ether, "");
    // Advance blocks past delay
    vm.roll(block.number + withdrawalDelayBlocks + 1);
    // Operator attempts to unlock all — expect OOG or extreme gas usage
    uint256 gasBefore = gasleft();
    withdrawalManager.unlockQueue(ETH, 10_001, ...);
    assertGt(gasBefore - gasleft(), 10_000 * STORAGE_OP_COST);
}
```
