Audit Report

## Title
Dust Withdrawal Request Spam Causes Unbounded Gas in `_unlockWithdrawalRequests` and Temporary Freezing of Legitimate User Funds - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`minRsEthAmountToWithdraw` is a storage mapping that defaults to `0` for every asset and is never initialized in `initialize()`, meaning any amount ≥ 1 wei passes the only guard in `initiateWithdrawal`. There is no per-user cap on queued requests. An attacker can flood the FIFO queue with thousands of dust entries; because `_unlockWithdrawalRequests` advances `nextLockedNonce` strictly in order, legitimate users' requests cannot be unlocked until every preceding attacker entry is processed, causing unbounded gas consumption and temporary freezing of legitimate users' rsETH already held by the contract.

## Finding Description

**Root cause 1 — zero minimum enforced by default.**
`minRsEthAmountToWithdraw` is declared at line 35 as a plain mapping with no initialization in `initialize()` (lines 90–98). Its Solidity default is `0`, so the guard at line 162:

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```

collapses to `rsETHUnstaked == 0`, accepting any amount ≥ 1 wei.

**Root cause 2 — no per-user request cap.**
`_addUserWithdrawalRequest` (lines 744–760) unconditionally appends to `userAssociatedNonces[asset][msg.sender]` and increments the global `nextUnusedNonce[asset]`. There is no check on how many pending requests a single address may hold.

**Exploit path through `_unlockWithdrawalRequests`.**
The while-loop at lines 790–814 iterates from `nextLockedNonce_` up to `firstExcludedIndex`. For a dust request of 1 wei rsETH, `_calculatePayoutAmount` at line 833 computes:

```solidity
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
```

Integer division rounds this to `0`. The break condition at line 800:

```solidity
if (availableAssetAmount < payoutAmount) break;
```

is `availableAssetAmount < 0`, which is never true for a `uint256`, so the loop does **not** break on dust entries. Each dust iteration reads a storage slot (`withdrawalRequests[requestId]`), performs arithmetic, and writes back (`assetsCommitted`, `request.expectedAssetAmount`, `unlockedWithdrawalsCount`), consuming ~5 000–10 000 gas per iteration. With N = 10 000 dust entries, a single `unlockQueue` call requires ~50–100 M gas, exceeding the block gas limit. Even when the operator batches calls with a smaller `firstExcludedIndex`, `nextLockedNonce` must advance through every attacker nonce before any legitimate nonce at position N can be reached.

**Existing checks are insufficient.**
`setMinRsEthAmountToWithdraw` (line 330) exists but is admin-gated and never called during deployment. The `onlySupportedAsset` and `onlySupportedStrategy` modifiers do not constrain amount or request count. The `ExceedAmountToWithdraw` check at line 170 only prevents over-commitment of available assets, not dust amounts.

## Impact Explanation

**Medium — Temporary freezing of funds and unbounded gas consumption.**

Legitimate users transfer rsETH to the contract at line 166 upon calling `initiateWithdrawal`. Those funds remain locked in the contract until their nonce is reached by `nextLockedNonce`. Because the queue is strictly FIFO and there is no mechanism to skip attacker nonces, legitimate users' withdrawals are delayed by a period proportional to the number of spam entries. This matches the allowed impact "Medium. Temporary freezing of funds" and "Medium. Unbounded gas consumption."

## Likelihood Explanation

**Medium.** The attacker only needs rsETH, obtainable permissionlessly by depositing into the protocol. At 1 wei per request, the economic cost is negligible — only gas fees. No privileged role is required; `initiateWithdrawal` is a public function. The default `minRsEthAmountToWithdraw = 0` is the live on-chain state unless the admin has explicitly called `setMinRsEthAmountToWithdraw` for every supported asset after deployment. The attack is repeatable and can be executed by any address.

## Recommendation

1. **Enforce a non-zero minimum in `initialize()`**: Set a meaningful `minRsEthAmountToWithdraw` for every supported asset during initialization, or add a `require(minRsEthAmountToWithdraw[asset] > 0)` guard in `initiateWithdrawal` so the mapping cannot remain at its default.
2. **Add a per-user pending request cap**: In `_addUserWithdrawalRequest`, add `require(userAssociatedNonces[asset][msg.sender].length() < MAX_REQUESTS_PER_USER)` to bound the number of simultaneous pending requests per address.
3. **Add a `maxIterations` parameter to `_unlockWithdrawalRequests`**: Allow the operator to bound gas per call independently of `firstExcludedIndex`, so batching does not require processing attacker entries before legitimate ones.

## Proof of Concept

1. Attacker acquires N wei of rsETH (e.g., N = 10 000, one per call) by depositing into the protocol.
2. Attacker calls `initiateWithdrawal(asset, 1, "")` 10 000 times. Each call passes the `rsETHUnstaked == 0` check because `minRsEthAmountToWithdraw[asset] == 0`. Each call transfers 1 wei rsETH to the contract and appends a new entry to the global queue.
3. Legitimate user calls `initiateWithdrawal(asset, largeAmount, "")`. Their request is assigned nonce 10 000.
4. Operator calls `unlockQueue(asset, 10001, ...)`. `_unlockWithdrawalRequests` loops from nonce 0 to 10 000. For each dust entry, `_calculatePayoutAmount` returns 0, the `availableAssetAmount < payoutAmount` break is never triggered, and the loop processes all 10 000 attacker entries — consuming ~50–100 M gas, exceeding the block gas limit.
5. Even if the operator batches calls (e.g., 500 entries per call), 20 separate `unlockQueue` transactions are required to exhaust the attacker's nonces before the legitimate user's request at nonce 10 000 can be unlocked. During this period, the legitimate user's rsETH remains locked in the contract with no recourse.
