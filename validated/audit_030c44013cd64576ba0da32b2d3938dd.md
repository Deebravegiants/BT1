### Title
Native Fee (STRK) Permanently Locked in Starknet Bridge Contract with No Withdrawal Mechanism — (File: starknet/src/omni_bridge.cairo)

### Summary
The Starknet `OmniBridge` contract collects `native_fee` as STRK tokens from users during `init_transfer` and holds them in the contract itself. No function exists to withdraw or claim these tokens. Relayers are compensated on NEAR via minted wrapped tokens, but the actual STRK paid by users is permanently locked with no recovery path.

### Finding Description
In `starknet/src/omni_bridge.cairo`, the `init_transfer` function transfers `native_fee` STRK tokens from the caller directly into the bridge contract (`get_contract_address()`): [1](#0-0) 

The destination is the contract itself — not a relayer, not a fee vault, not any claimable address. The `IOmniBridge` interface exposes no `claim_native_fee`, `withdraw`, or equivalent function: [2](#0-1) 

The `IUpgradeable` interface only adds `upgrade`. There is no other exit path for these STRK tokens.

On the NEAR side, when the relayer finalizes the transfer, `send_fee_internal` compensates the relayer by **minting** wrapped native tokens — it does not release the locked STRK from Starknet: [3](#0-2) 

This means:
- User pays `native_fee` STRK on Starknet → locked forever in the contract.
- Relayer receives freshly minted wrapped STRK on NEAR → unbacked by the locked STRK.
- The locked STRK is permanently unclaimable.

### Impact Explanation
Every user who pays a non-zero `native_fee` on Starknet suffers an irreversible loss of STRK tokens. The tokens accumulate in the bridge contract with no withdrawal path. This is a permanent fund lock in the fee flow, matching the Critical impact criterion: *"permanently unclaimable user or protocol value in bridge, token, fee, vault, fast-transfer, or UTXO flows."*

### Likelihood Explanation
High. Paying a non-zero `native_fee` is the standard mechanism for users to incentivize relayers to process their transfers. Any user who does so on Starknet will have their STRK permanently locked. The code path is unconditional and requires no special conditions.

### Recommendation
Add a privileged `claim_native_fee` function to the Starknet contract that allows an authorized role (e.g., `DEFAULT_ADMIN_ROLE` or a designated fee recipient) to withdraw accumulated STRK native fees:

```cairo
fn claim_native_fee(ref self: ContractState, recipient: ContractAddress, amount: u128) {
    self.accesscontrol.assert_only_role(DEFAULT_ADMIN_ROLE);
    let native_token = self.strk_token_address.read();
    let success = IERC20Dispatcher { contract_address: native_token }
        .transfer(recipient, amount.into());
    assert(success, 'ERR_FEE_CLAIM_FAILED');
}
```

Alternatively, align with the EVM design where `nativeFee` is forwarded to the messaging layer rather than held in the contract.

### Proof of Concept
1. User calls `init_transfer` on Starknet with `native_fee = 100e18` (100 STRK).
2. `transfer_from(caller, get_contract_address(), 100e18)` executes — 100 STRK enters the bridge contract.
3. Relayer observes the event and calls `sign_transfer` on NEAR with `fee_recipient = relayer_account`.
4. NEAR's `send_fee_internal` calls `ext_token::mint(relayer_account, 100e18, None)` — relayer receives 100 wrapped STRK on NEAR.
5. The 100 STRK in the Starknet contract has no exit path. It is permanently locked.
6. Repeated for every user paying a non-zero native fee — STRK accumulates and is never recoverable. [1](#0-0) [4](#0-3)
