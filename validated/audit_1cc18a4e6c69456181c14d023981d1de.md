### Title
Missing `payload_hash`-to-`request` Binding in `respond_verify_foreign_tx` Enables Byzantine Participant to Forge Foreign Chain Verification — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` contract method verifies that the submitted ECDSA signature is valid over the caller-supplied `response.payload_hash`, but it never verifies that `payload_hash` was actually derived from the `request` argument. A single Byzantine attested participant (below the signing threshold) who possesses any prior valid threshold signature can reuse it as a response to a different pending foreign-chain verification request, causing the contract to attest to foreign-chain data that was never verified for that request.

---

### Finding Description

**Root cause — missing constraint in `respond_verify_foreign_tx`:** [1](#0-0) 

The function performs two checks:
1. The caller is an attested participant.
2. `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's root public key.

It does **not** check that:

```
response.payload_hash == SHA-256(borsh(ForeignTxSignPayload { request, values }))
```

for any `values` consistent with `request`. The `payload_hash` field is entirely attacker-controlled.

**The intended binding** is defined in `ForeignTxSignPayload::compute_msg_hash`: [2](#0-1) 

The node-side code correctly computes and signs this hash: [3](#0-2) 

But the contract never re-derives or checks this binding when accepting the response.

**Exploit path (single Byzantine leader, no threshold collusion required):**

1. Attacker is an attested MPC participant and acts as the signing leader for a legitimate request **R1** (e.g., "verify Bitcoin tx T1, extract block hash BH1"). The MPC protocol produces a valid threshold signature **S1** over **H1** = `SHA-256(borsh(ForeignTxSignPayload{R1, [BH1]}))`.
2. A separate, unrelated request **R2** is pending on-chain (e.g., "verify Bitcoin tx T2, extract block hash BH2"). The attacker does not query the foreign chain for R2 at all.
3. The attacker calls `respond_verify_foreign_tx(request=R2, response={payload_hash=H1, signature=S1})`.
4. The contract checks: S1 is a valid signature over H1 ✓; R2 is a pending request ✓. It resolves R2's yield with `{payload_hash=H1, signature=S1}`.
5. The bridge contract waiting on R2 receives a response claiming the MPC network verified R2, but the signed payload actually encodes R1's data (T1's block hash BH1, not T2's BH2).

The attacker needs only to have been the leader of **any** prior signing session to obtain a reusable threshold signature. No threshold collusion is required.

**Why the SDK-level check does not fully mitigate this:**

The SDK's `ForeignChainSignatureVerifier::verify_signature` does check `expected_payload_hash == response.payload_hash`: [4](#0-3) 

However, this check is performed by the **caller contract** (e.g., the bridge), not by the MPC contract itself. The MPC contract resolves the yield and returns the forged `{payload_hash, signature}` pair unconditionally. Any caller that trusts the returned `payload_hash` without re-deriving it from the request (a reasonable assumption given the contract is supposed to be the authoritative verifier) is vulnerable. The design document explicitly states the response is intended to be trusted: [5](#0-4) 

---

### Impact Explanation

**Impact class:** High — Forged foreign-chain verification / invalid bridge execution.

A bridge contract using `verify_foreign_transaction` to gate fund releases (e.g., Omnibridge inbound flow) can be made to release funds for a foreign-chain transaction that was never verified. The attacker forges the binding between a pending request and the signed payload, causing the MPC network's attestation to cover the wrong transaction data. This directly enables double-spend or unauthorized fund release conditions on the NEAR side.

---

### Likelihood Explanation

**Likelihood:** Medium.

- The attacker must be an attested MPC participant (Byzantine, below threshold). This is within the stated attacker model ("Byzantine participant strictly below the signing threshold").
- The attacker must have been the signing leader for at least one prior legitimate session to obtain a reusable threshold signature. In a live network processing many requests, this is trivially achievable.
- The target request R2 must be pending at the time of the attack. In a bridge with active traffic, pending requests are always present.
- The attack is a single on-chain transaction and leaves no on-chain trace distinguishing it from a legitimate response.

---

### Recommendation

Add an on-chain check in `respond_verify_foreign_tx` that binds `response.payload_hash` to `request`. Two options:

**Option A (preferred):** Require the response to include the extracted `values`, compute `expected_hash = SHA-256(borsh(ForeignTxSignPayload{request, values}))` on-chain, and assert `response.payload_hash == expected_hash` before accepting the signature.

**Option B:** Store the expected `payload_hash` in the pending request map at enqueue time (requires nodes to agree on it before signing, which is already the case in the MPC protocol) and verify it on response.

---

### Proof of Concept

```rust
// Attacker (Byzantine leader) has previously participated in signing session for R1,
// obtaining threshold signature S1 over H1 = SHA-256(borsh(ForeignTxSignPayload{R1, [BH1]})).

// R2 is a different pending request (e.g., verify Bitcoin tx T2).
let r2 = VerifyForeignTransactionRequest {
    domain_id: foreign_tx_domain_id,
    payload_version: ForeignTxPayloadVersion::V1,
    request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
        tx_id: BitcoinTxId([0xT2; 32]),  // different tx
        confirmations: BlockConfirmations(1),
        extractors: vec![BitcoinExtractor::BlockHash],
    }),
};

// Attacker submits R1's signature as a response to R2.
// The contract only checks: is S1 a valid signature over H1? YES.
// It does NOT check: does H1 correspond to R2? NO CHECK EXISTS.
contract.respond_verify_foreign_tx(
    r2,
    VerifyForeignTransactionResponse {
        payload_hash: H1,   // hash of R1's payload, not R2's
        signature: S1,      // valid threshold signature over H1
    },
);
// R2's yield resolves with R1's data. Bridge releases funds for T2 based on T1's block hash.
```

The contract's signature check at line 729 passes because S1 is genuinely valid over H1. The missing constraint — that H1 must equal `SHA-256(borsh(ForeignTxSignPayload{R2, values}))` — is never enforced. [6](#0-5)
