### Title
`verify_foreign_transaction` Discards Caller Identity, Enabling Any Party to Obtain Root-Key Signatures for Foreign Chain Transactions - (File: crates/contract/src/lib.rs)

### Summary
The `verify_foreign_transaction` function discards the caller's predecessor account ID when constructing the `VerifyForeignTransactionRequest`. Unlike `sign()` and `request_app_private_key()`, which bind the caller's identity to the request (ensuring derived keys are caller-specific), `verify_foreign_transaction` constructs the request with no caller binding. Any unprivileged actor can submit an identical request for the same foreign chain transaction and receive the same root-key MPC signature via the fan-out mechanism, enabling unauthorized bridge execution.

### Finding Description
In `lib.rs`, `check_request_preconditions` returns `(DomainConfig, AccountId)` — the `AccountId` being the caller's predecessor. In `sign()` and `request_app_private_key()`, the predecessor is captured and used to construct a caller-bound request:

```rust
// sign() — predecessor is captured and bound into the request
let (domain_config, predecessor) = self.check_request_preconditions(...);
let request = SignatureRequest::new(
    request.domain_id, request.payload, &predecessor, &request.path
);
``` [1](#0-0) 

In `verify_foreign_transaction()`, the return value is completely discarded:

```rust
// verify_foreign_transaction() — return value dropped, predecessor never used
self.check_request_preconditions(
    request.domain_id,
    DomainPurpose::ForeignTx,
    Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
    MINIMUM_SIGN_REQUEST_DEPOSIT,
);
// ...
let request = args_into_verify_foreign_tx_request(request); // no caller identity
``` [2](#0-1) 

The resulting `VerifyForeignTransactionRequest` contains no caller identity. Because `resolve_yields_for` fans out the MPC response to **all** queued yields for the same request key, any party who submits an identical `VerifyForeignTransactionRequestArgs` will receive the same root-key signature. [3](#0-2) 

Compounding this, `respond_verify_foreign_tx` verifies the signature against the **root** public key — not a caller-derived key — making the issued signature broadly applicable rather than caller-specific: [4](#0-3) 

The contrast with `sign()` and `request_app_private_key()` is deliberate in those functions and absent here, indicating this is an oversight rather than a design choice. [5](#0-4) 

### Impact Explanation
An attacker who observes a legitimate `verify_foreign_transaction` submission (all NEAR transactions are public) can submit the same request. When MPC nodes respond, both the legitimate user and the attacker receive the root-key signature for the foreign chain transaction. The attacker can then present this signature to a bridge or downstream contract to claim bridged funds or execute the authorized foreign-chain action before the legitimate user does. This constitutes unauthorized bridge execution and potential double-spend conditions — matching the **High** impact tier: *cross-chain verification bypass causing invalid bridge execution*.

### Likelihood Explanation
All NEAR transactions are publicly visible on-chain. An attacker only needs to:
1. Monitor the chain for `verify_foreign_transaction` calls.
2. Re-submit the same `VerifyForeignTransactionRequestArgs`.
3. Pay the negligible 1 yoctoNEAR deposit (`MINIMUM_SIGN_REQUEST_DEPOSIT`). [6](#0-5) 

No privileged access, key material, or threshold collusion is required. The attack is straightforward and directly profitable whenever bridged funds are involved.

### Recommendation
Capture and bind the caller's predecessor identity into the `VerifyForeignTransactionRequest`, consistent with how `sign()` and `request_app_private_key()` operate. The request should include the caller's account ID so that the MPC-issued signature is specific to the original requester, preventing unauthorized parties from obtaining or reusing it. Concretely, change:

```rust
self.check_request_preconditions(...);
```

to:

```rust
let (_, predecessor) = self.check_request_preconditions(...);
```

and pass `predecessor` into `args_into_verify_foreign_tx_request` (or a new variant) so it is embedded in the request key stored in `pending_verify_foreign_tx_requests`.

### Proof of Concept
1. User A calls `verify_foreign_transaction` with `VerifyForeignTransactionRequestArgs` for Ethereum transaction `TX_ID`, intending to bridge funds to NEAR.
2. Attacker B observes the on-chain call and submits an identical `verify_foreign_transaction` call for the same `TX_ID`, paying 1 yoctoNEAR.
3. Both requests are stored under the **same** `VerifyForeignTransactionRequest` key in `pending_verify_foreign_tx_requests`, each with their own `YieldIndex`.
4. MPC nodes verify `TX_ID` on Ethereum and call `respond_verify_foreign_tx` with the root-key signature.
5. `resolve_yields_for` fans out the response: both A and B receive the identical root-key signature via their respective yield-resume callbacks.
6. Attacker B immediately presents the signature to the bridge contract to claim the bridged funds.
7. When User A attempts to claim, the bridge contract rejects the duplicate, and A loses their bridged funds despite having initiated the legitimate request.

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
