### Title
Missing RPC Quorum Enforcement in `FanOut` Inspector Allows Single-Provider Verification Bypass — (`crates/foreign-chain-inspector/src/lib.rs`)

### Summary

The `FanOut<Inspector>` implementation that drives foreign-chain transaction verification accepts a result from a **single** RPC provider even when the on-chain `ChainEntry.quorum` requires multiple providers to agree. Because the quorum value is never wired into `FanOut`, a single compromised or manipulated RPC provider whose peers return transient errors (timeouts, rate-limit 429s) can unilaterally drive the verification result accepted by each MPC node, enabling the threshold of nodes to sign a fraudulent foreign-chain payload.

### Finding Description

The design requires that each MPC node fan out a foreign-chain verification query to all its locally configured RPC providers and accept the result only when at least `rpc_quorum(C)` of them return the same response. This is documented explicitly:

> "A node considers the foreign TX verified iff at least the per-chain RPC quorum of providers agreed. If fewer than the quorum agree, the node errors out."
> — `docs/design/allowing-per-node-foreign-chain-rpc-configuration.md`, line 121

The on-chain quorum is stored in `ChainEntry.quorum`: [1](#0-0) 

However, the `FanOut<Inspector>` struct that implements the fan-out has **no quorum field**: [2](#0-1) 

Its `extract` logic accepts any result as long as all *substantive* (non-transient) verdicts agree — even if only **one** inspector succeeds and all others return transient errors: [3](#0-2) 

The `ForeignChainInspectors::build()` function constructs `FanOut` from the node's local config without ever consulting `ChainEntry.quorum`: [4](#0-3) 

The test suite explicitly validates and enshrines this single-provider acceptance behavior: [5](#0-4) 

The design docs acknowledge this gap as "implementation requirement, not current behavior": [6](#0-5) 

### Impact Explanation

When a threshold of MPC nodes each run `execute_foreign_chain_request` for the same `verify_foreign_transaction` request, each node's `FanOut` independently accepts a single provider's response if the others fail transiently. If a single widely-used RPC provider is compromised and returns a forged verification result (e.g., claiming a bridge deposit transaction exists and is finalized when it does not), and the other providers for those nodes return transient errors (timeouts, 429 rate-limit responses), every node in the signing set accepts the forged result. The nodes then each produce a signature share over the fraudulent `ForeignTxSignPayload`, the threshold is reached, and the MPC contract returns a valid threshold signature attesting to a foreign-chain event that never occurred. This enables invalid bridge execution — e.g., minting bridged assets on NEAR for a deposit that was never made on the foreign chain. [7](#0-6) 

### Likelihood Explanation

A popular RPC provider (e.g., Alchemy, Infura, Ankr) is likely configured by most or all node operators. If that provider is compromised at the infrastructure level, or if an attacker exploits its API to return forged responses for specific transaction IDs while causing other providers to hit rate limits (returning `RpcRequestFailed`, which is classified as transient), the attack succeeds without any threshold collusion among MPC participants. The `RpcRequestFailed` transient classification covers rate-limit 429 responses, which an attacker controlling one provider can trivially trigger against competing providers by exhausting shared API quotas. [8](#0-7) 

### Recommendation

1. Add a `quorum: NonZeroUsize` field to `FanOut<Inspector>` and enforce it: after collecting all results, require that the number of agreeing successful responses is `≥ quorum` before returning `Ok`. If fewer than `quorum` providers succeed (even if they all agree), return a non-transient sub-quorum error.
2. Wire the on-chain `ChainEntry.quorum` value through `ForeignChainInspectors::build()` into each `FanOut` instance so the network-voted quorum is actually enforced at runtime.
3. Treat sub-quorum outcomes as terminal (non-retryable) as the design already requires, to prevent retry loops from masking the security failure.

### Proof of Concept

**Setup**: Node A has 3 Bitcoin RPC providers configured: `provider-good-1`, `provider-attacker`, `provider-good-2`. On-chain `ChainEntry.quorum = 2`.

**Attack**:
1. Attacker submits `verify_foreign_transaction` for a Bitcoin `tx_id` that does not exist (or has fewer confirmations than required).
2. `provider-attacker` returns a forged success: `BlockHash = 0xdeadbeef...`, confirmations = 6.
3. `provider-good-1` and `provider-good-2` return `RpcRequestFailed` (transient) — attacker triggers this by exhausting their rate limits via unrelated requests.
4. `FanOut::extract` sees: 1 success, 2 transient errors. Since `successes.first()` is `Some` and all successes agree (trivially, there is only one), it returns `Ok(BlockHash(0xdeadbeef...))`.
5. The quorum of 2 is never checked. Node A produces a signature share.
6. Repeated across a threshold of nodes (all using `provider-attacker`), the MPC contract receives a complete threshold signature over the fraudulent payload. [9](#0-8)
