### Title
Missing G2 Subgroup Membership Check in `alt_bn128_pairing_check` Allows Cofactor Attack on ZK-SNARK Verifier Contracts - (`runtime/near-vm-runner/src/logic/alt_bn128.rs`)

### Summary

`decode_g2` only verifies that a supplied G2 point lies on the twist curve; it never verifies that the point is in the prime-order r-torsion subgroup. The `bn` crate does not perform this check internally either. An attacker can supply a G2 point that is on the twist but outside the subgroup, causing `pairing_check` to compute a pairing over a non-subgroup element and potentially return `true` for a forged proof.

### Finding Description

`decode_g2` calls `bn::AffineG2::new(x, y)`, which only checks that `(x, y)` satisfies the twist curve equation `Y² = X³ + 3/(i+9)` over Fq2. It does **not** multiply by the cofactor or check that `r * P' = O`: [1](#0-0) 

`pairing_check` then passes the decoded (potentially non-subgroup) point directly to `bn::pairing_batch`: [2](#0-1) 

The `bn` crate (paritytech/bn) does not perform subgroup membership checks internally; `AffineG2::new` is a curve-membership-only check. For BN128, G1 has cofactor 1 (all curve points are in the subgroup), but the G2 twist has a large non-trivial cofactor h2. Points on the twist outside the r-torsion subgroup exist and are accepted by `decode_g2`.

This directly contradicts the documented contract, which explicitly promises:

> "If point coordinates are not on curve, **point is not in the subgroup**, scalar is not in the field or data are wrong serialized... the function returns `AltBn128InvalidInput`." [3](#0-2) 

The contrast with the BLS12-381 implementation in the same codebase is stark — `bls12381::pairing_check` explicitly calls `blst_p1_affine_in_g1` and `blst_p2_affine_in_g2` before computing the pairing: [4](#0-3) 

No equivalent guard exists anywhere in the alt_bn128 path.

### Impact Explanation

For BN128, the Ate pairing with final exponentiation maps to μ_r (the r-th roots of unity in Fq12\*). For a non-subgroup G2 point Q' ∈ E'(Fq2) \ G2, the Miller loop + final exponentiation still produces an element of μ_r, but it need not equal 1 only when the pairing equation is genuinely satisfied. Crafted non-subgroup points can produce `Gt::one()` for inputs that do not satisfy the pairing equation. This is the well-known cofactor attack on BN128 pairings, which is why EIP-197 explicitly mandates subgroup membership checks.

A ZK-SNARK verifier contract (e.g., a Groth16 verifier) deployed on NEAR that calls `alt_bn128_pairing_check` can be fooled: an attacker submits a forged proof containing a G2 point outside the subgroup, `pairing_check` returns `true`, the contract writes `proof_valid = true` to storage, and the resulting state root reflects this incorrect state. All nodes compute the same wrong result (since they all use the same `bn` crate), so there is no consensus split — but the accepted state root is cryptographically incorrect.

### Likelihood Explanation

- The alt_bn128 host function is enabled in production.
- Any unprivileged user can deploy a ZK-SNARK verifier contract and submit transactions calling it.
- Constructing a non-subgroup G2 point on the BN128 twist is straightforward: take any point Q ∈ E'(Fq2) and compute Q' = Q - (h2_inv mod r) * (r * Q) or simply pick a random point on the twist and verify r * Q' ≠ O.
- The attack requires no special privileges, no validator access, and no node compromise.

### Recommendation

Add an explicit subgroup membership check in `decode_g2` after the curve-membership check. The standard approach for BN128 is to verify `r * P' == O` (point at infinity) after decoding. Alternatively, use cofactor clearing: replace the decoded point with `h2 * P'` and verify it is non-zero (or reject if `r * P' ≠ O`). This mirrors what the BLS12-381 path already does via `blst_p2_affine_in_g2`.

### Proof of Concept

1. Pick any point Q ∈ E'(Fq2) such that r * Q ≠ O (i.e., Q is on the twist but not in the r-torsion subgroup). Such points exist because the twist has order r * h2 with h2 > 1.
2. Encode Q as a 128-byte G2 input (two Fq2 coordinates, little-endian).
3. Call `alt_bn128_pairing_check` with a crafted pair `(G1_gen, Q)`.
4. `decode_g2` accepts Q (it satisfies the twist equation).
5. `bn::pairing_batch` computes the Ate pairing without subgroup checks.
6. For appropriately crafted Q (exploiting the cofactor structure), the result equals `Gt::one()`, and `pairing_check` returns `true`.
7. Assert: `alt_bn128::pairing_check([(G1_gen, Q)])` returns `true` despite Q ∉ G2.

The invariant "pairing_check must return false for any input containing a G2 point not in the prime-order subgroup" is violated because `decode_g2` performs no subgroup check and the `bn` crate provides none. [1](#0-0) [2](#0-1)

pub(crate) fn pairing_check(
    elements: &[[u8; PAIRING_CHECK_ELEMENT_SIZE]],
) -> Result<bool, InvalidInput> {
    let elements: Vec<(bn::G1, bn::G2)> = elements
        .iter()
        .map(|chunk| {
            let (g1, g2) = stdx::split_array(chunk);
            let g1 = decode_g1(g1)?;
            let g2 = decode_g2(g2)?;
            Ok((g1, g2))
        })
        .collect::<Result<Vec<_>, InvalidInput>>()?;

    let res = bn::pairing_batch(&elements) == bn::Gt::one();

    Ok(res)
}

fn decode_g2(raw: &[u8; 2 * POINT_SIZE]) -> Result<bn::G2, InvalidInput> {
    let (x, y) = stdx::split_array(raw);
    let x = decode_fq2(x)?;
    let y = decode_fq2(y)?;
    if x.is_zero() && y.is_zero() {
        Ok(bn::G2::zero())
    } else {
        bn::AffineG2::new(x, y)
            .map_err(|_err| InvalidInput::new("invalid g2", raw))
            .map(bn::G2::from)
    }
}
