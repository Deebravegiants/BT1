### Title
Missing Qi_self binding in EdDSA-2p `attach_private_scalar` allows substitution of arbitrary share scalar — (`src/cbmpc/api/eddsa2pc.cpp`)

### Summary
`attach_private_scalar` in the EdDSA-2p API only verifies internal consistency (`x*G == caller_Qi`), but never checks that the caller-supplied `public_share_compressed` matches the original Qi_self from DKG. Because the public blob produced by `detach_private_scalar` does not store Qi_self, any valid (x′, x′·G) pair is accepted, allowing a caller to substitute an arbitrary scalar into the key blob and proceed to `sign`.

### Finding Description
The `key_blob_v1_t` for EdDSA-2p (`src/cbmpc/api/eddsa2pc.cpp`) contains only `{version, role, curve, Q_compressed, x_share}`. When `detach_private_scalar` produces the public blob, it sets `x_share = q` as a sentinel and preserves `Q_compressed`, but **does not record Qi_self** (the party's individual share public point).

When `attach_private_scalar` is later called, it:
1. Decodes `public_share_compressed` from the caller argument into `Qi_self`.
2. Checks `x * G == Qi_self`.
3. Writes `x` into the blob and returns success.

Step 2 is a self-referential check: both `x` and `Qi_self` are supplied by the caller. There is no anchor to the original DKG share because the blob carries no record of it. Any (x′, x′·G) pair satisfies the check.

The MP variant (`src/cbmpc/api/eddsa_mp.cpp`) correctly fixes this by storing `Qis_compressed` in the blob and enforcing `public_share_compressed == blob.Qis_compressed[party_name]` before the scalar check. [5](#0-4) 

### Impact Explanation
A caller who holds the public blob (e.g., from a backup store) can call `attach_private_scalar(pub_blob, x′, x′·G)` with any x′ ∈ [1, q). The resulting blob passes `deserialize_key_blob` (x′ is in range, Q parses correctly) and is accepted by `sign`. The signing protocol then runs with x′ instead of the original x_share. Since Q ≠ x′·G + Qi_other, the produced signature does not verify under Q — but the API returns `SUCCESS` and outputs a signature. This is accepted invalid cryptographic output: the library asserts a successful signing operation while the output is cryptographically wrong.

### Likelihood Explanation
The API is explicitly designed for backup/restore workflows (PVE, HSM escrow). Any party that can obtain the public blob — which is described as "safe to persist as public-only material" — can perform the substitution. No secret material is required beyond the public blob and a chosen scalar.

### Recommendation
Store the original Qi_self in the public blob produced by `detach_private_scalar`, mirroring the MP approach. In `attach_private_scalar`, reject any `public_share_compressed` that does not byte-equal the stored value before performing the `x*G` check. Concretely, add a `Qi_compressed` field to `key_blob_v1_t`, populate it in `detach_private_scalar`, and add the equality guard in `attach_private_scalar` analogous to:
```cpp
if (public_share_compressed != mem_t(pub.Qi_compressed))
    return coinbase::error(E_BADARG, "public_share_compressed mismatch key blob");
```

### Proof of Concept
1. Both parties run `dkg(...)` → each holds `key_blob_i` with original shares x1, x2 and Q = x1·G + x2·G.
2. P1 calls `detach_private_scalar(key_blob_1)` → `pub_blob` (contains Q, no Qi_self), `x_original`.
3. P1 picks x′ = q−1, computes Qi′ = (q−1)·G.
4. P1 calls `attach_private_scalar(pub_blob, x′_bytes, Qi′_compressed)` → returns `SUCCESS`, produces `tampered_blob`.
5. P1 calls `sign(job, tampered_blob, msg)` with P2 using its legitimate `key_blob_2` → protocol completes, returns `SUCCESS` and a 64-byte signature.
6. Verify the signature against Q: verification fails, confirming the signature was produced with the wrong share scalar while the API reported success.
