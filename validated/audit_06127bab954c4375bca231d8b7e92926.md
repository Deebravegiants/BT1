### Title
Per-Asset Cap in `maxAmountToDepositBridgerAsset` Allows Aggregate Over-Collateralization, Stranding Bridger Funds on Token Removal — (`contracts/L2/RsETHTokenWrapper.sol`)

---

### Summary

`maxAmountToDepositBridgerAsset` computes a per-asset cap of `totalSupply - balance[_asset]` independently for each allowed token. With two allowed tokens, the BRIDGER_ROLE can fill each token's individual cap simultaneously, causing total collateral held by the wrapper to reach `N × totalSupply` (where N = number of allowed tokens). When `removeAllowedToken` is subsequently called on one token, that token's balance is permanently stranded with no recovery path.

---

### Finding Description

The cap formula in `maxAmountToDepositBridgerAsset` is:

```
return wrsETHSupply - balanceOfAssetInWrapper;   // line 109
``` [1](#0-0) 

It measures only the balance of the queried token against `totalSupply()`. It does not subtract the balances of any other allowed tokens. Therefore, for two allowed tokens tokenA and tokenB both starting at zero balance:

- `maxAmountToDepositBridgerAsset(tokenA)` = `totalSupply - 0` = `totalSupply`
- `maxAmountToDepositBridgerAsset(tokenB)` = `totalSupply - 0` = `totalSupply`

The BRIDGER_ROLE can call `depositBridgerAssets` for each token up to its individual cap:

```solidity
function depositBridgerAssets(address _asset, uint256 _amount) external onlyRole(BRIDGER_ROLE) {
    if (maxAmountToDepositBridgerAsset(_asset) < _amount) revert CannotDeposit();
    ERC20Upgradeable(_asset).safeTransferFrom(msg.sender, address(this), _amount);
    emit BridgerDeposited(_asset, msg.sender, _amount);
}
``` [2](#0-1) 

After both deposits: `balance[tokenA] + balance[tokenB] = 2 × totalSupply`. The aggregate invariant `sum(all token balances) ≤ totalSupply` is violated.

When `removeAllowedToken(tokenB)` is later called (a routine TIMELOCK_ROLE operation, e.g., deprecating a bridge route):

```solidity
function removeAllowedToken(address _asset) external onlyRole(TIMELOCK_ROLE) {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    allowedTokens[_asset] = false;
    emit TokenRemoved(_asset);
}
``` [3](#0-2) 

The tokenB balance is permanently locked. `_withdraw` enforces `allowedTokens[_asset]`:

```solidity
function _withdraw(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
``` [4](#0-3) 

There is no `rescueTokens`, no admin sweep, and no recovery path for the stranded tokenB. The contract has no function to retrieve ERC-20 tokens other than through `_withdraw`, which is gated on `allowedTokens`.

---

### Impact Explanation

- The BRIDGER_ROLE's excess tokenB collateral is permanently frozen in the contract after `removeAllowedToken`.
- The aggregate collateral cap is silently violated: the contract accepts `N × totalSupply` worth of collateral for N allowed tokens, which is not the intended design (the cap is meant to prevent over-collateralization).
- Users holding wrsETH are not directly harmed (they can still redeem via tokenA if sufficient balance exists), so this fits **Low — Contract fails to deliver promised returns, but doesn't lose value**: the bridger's deposited collateral is irrecoverable, and the cap mechanism fails to enforce its stated purpose.

---

### Likelihood Explanation

- The BRIDGER_ROLE is an operational role that routinely calls `depositBridgerAssets` to collateralize minted wrsETH.
- `addAllowedToken` / `removeAllowedToken` are expected lifecycle operations (e.g., migrating to a new bridge token).
- No malicious intent is required — the over-collateralization occurs through normal, independent use of each token's cap.
- The combination of two allowed tokens + bridger deposits + token removal is a realistic operational sequence.

---

### Recommendation

Replace the per-asset cap with an aggregate cap across all allowed tokens. Track the total collateral deposited via `depositBridgerAssets` in a storage variable, and enforce:

```solidity
uint256 totalBridgerDeposited; // sum across all assets

function maxAmountToDepositBridgerAsset(address _asset) public view returns (uint256) {
    if (!allowedTokens[_asset]) return 0;
    uint256 supply = totalSupply();
    if (totalBridgerDeposited >= supply) return 0;
    return supply - totalBridgerDeposited;
}
```

Additionally, add an admin token-rescue function so that stranded collateral from removed tokens can be recovered.

---

### Proof of Concept

```solidity
// Setup: totalSupply = 100e18 wrsETH (minted via MINTER_ROLE)
// tokenA and tokenB are both in allowedTokens
// balance[tokenA] = 0, balance[tokenB] = 0

// Step 1: Bridger deposits tokenA up to its cap
uint256 capA = wrapper.maxAmountToDepositBridgerAsset(tokenA);
// capA == 100e18 (totalSupply - 0)
vm.prank(bridger);
wrapper.depositBridgerAssets(tokenA, capA);
// balance[tokenA] = 100e18

// Step 2: Bridger deposits tokenB up to its cap
uint256 capB = wrapper.maxAmountToDepositBridgerAsset(tokenB);
// capB == 100e18 (totalSupply - 0, tokenB balance still 0)
vm.prank(bridger);
wrapper.depositBridgerAssets(tokenB, capB);
// balance[tokenB] = 100e18

// Assert: total collateral = 200e18 > totalSupply = 100e18
assert(tokenA.balanceOf(address(wrapper)) + tokenB.balanceOf(address(wrapper)) > wrapper.totalSupply());

// Step 3: Admin removes tokenB
vm.prank(timelockAdmin);
wrapper.removeAllowedToken(tokenB);

// Step 4: tokenB is permanently stranded — no withdrawal path
vm.expectRevert(RsETHTokenWrapper.TokenNotAllowed.selector);
vm.prank(bridger);
wrapper.withdraw(tokenB, 100e18); // reverts: TokenNotAllowed
// 100e18 tokenB locked forever with no recovery function
```
