### Title
Caller Deposit Permanently Lost When Intermediate Cross-Contract Call Fails in `nep_141_storage_balance_callback` / `address_check_callback` — (`runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs`)

### Summary

The `WalletContract` in `near-wallet-contract` tracks an external caller's attached NEAR deposit via `CallerDeposit` and is designed to refund it if the downstream cross-contract call fails. However, two intermediate callbacks — `nep_141_storage_balance_callback` and `address_check_callback` — have early-return failure paths that discard `caller_deposit` without issuing a refund. Any NEAR tokens attached by an external caller are permanently stuck in the wallet contract's balance.

### Finding Description

When an external caller (e.g., a relayer) calls `rlp_execute` with an attached NEAR deposit for an ERC-20 transfer, the following promise chain is constructed:

```
storage_balance_of → nep_141_storage_balance_callback → rlp_execute_callback
```

`CallerDeposit` is created in `inner_rlp_execute` to track the caller's deposit: [1](#0-0) 

The deposit is forwarded through the chain as `caller_deposit: Option<CallerDeposit>`. The terminal callback `rlp_execute_callback` correctly refunds the caller on failure: [2](#0-1) 

However, `nep_141_storage_balance_callback` has two early-return paths that silently drop `caller_deposit` without refunding:

**Path 1 — `storage_balance_of` call fails:** [3](#0-2) 

**Path 2 — JSON deserialization of the response fails:** [4](#0-3) 

Both paths return `PromiseOrValue::Value(ExecuteResponse { success: false, ... })` without creating a refund promise. The deposit, already transferred to the wallet contract's account when `rlp_execute` was called, is never returned.

The same defect exists in `address_check_callback` for the EOA base-token-transfer path: [5](#0-4) [6](#0-5) 

### Impact Explanation

The caller's NEAR deposit is permanently locked in the wallet contract's account balance. There is no recovery path: the wallet contract has no admin withdrawal function, and the NEAR runtime's automatic deposit refund only applies to the receipt that panics — not to a deposit consumed by a prior receipt (`rlp_execute`). The corrupted protocol value is the caller's NEAR account balance.

### Likelihood Explanation

Any external account (relayer or user) can call `rlp_execute` with a deposit. The `storage_balance_of` call can fail if the token contract does not exist, panics, or runs out of gas — all realistic conditions. The `address_check_callback` failure path is reachable if the address registrar contract is unavailable. No special privileges are required.

### Recommendation

In every early-return failure branch of `nep_141_storage_balance_callback` and `address_check_callback`, issue a refund promise before returning, mirroring the pattern already used in `rlp_execute_callback`:

```rust
if let Some(CallerDeposit { account_id, yocto_near }) = caller_deposit {
    let refund_promise = env::promise_batch_create(&account_id);
    env::promise_batch_action_transfer(
        refund_promise,
        NearToken::from_yoctonear(yocto_near.into()),
    );
}
```

This must be added to all `return PromiseOrValue::Value(...)` paths in both callbacks where `caller_deposit` is in scope but unused.

### Proof of Concept

1. External relayer calls `rlp_execute` on a wallet contract with `deposit = 5 NEAR` for an ERC-20 transfer to an unregistered receiver.
2. `inner_rlp_execute` creates `CallerDeposit { account_id: relayer, yocto_near: 5e24 }` and schedules `storage_balance_of → nep_141_storage_balance_callback`. [7](#0-6) 
3. The NEP-141 token contract's `storage_balance_of` call fails (e.g., contract panics or does not exist).
4. `nep_141_storage_balance_callback` is invoked with `PromiseResult::Failed`. It returns early at line 205–209 without touching `caller_deposit`.
5. The 5 NEAR deposit remains in the wallet contract's balance. The relayer's balance is permanently reduced by 5 NEAR with no recovery path.
