Audit Report

## Title
Race Condition in `processChunk` / `finalizeInitialize2` Inflates `unlockedWithdrawalsCount`, Permanently Blocking `sweepRemainingAssets` — (`contracts/utils/UnlockedWithdrawalsInitializer.sol`)

## Summary

`processChunk` snapshots `expectedAssetAmount > 0` at call time and accumulates the count in `unlockedCount[asset]`. If a user calls `completeWithdrawal` for any already-counted request before `finalizeInitialize2` executes, that request is deleted from `withdrawalRequests` and `unlockedWithdrawalsCount` is decremented, but `unlockedCount` in the initializer is never corrected. `finalizeInitialize2` then passes the stale, inflated value to `initialize2`, which overwrites the live counter. After all real withdrawals drain the counter to 1, `hasUnlockedWithdrawals` permanently returns `true` and `sweepRemainingAssets` is permanently blocked, freezing any remaining protocol balance in the withdrawal manager.

## Finding Description

**Root cause — stale snapshot in `processChunk`:** [1](#0-0) 

For every nonce `i` in `[start, limit)`, the function reads `withdrawalRequests[requestId].expectedAssetAmount` at call time. If non-zero, `added` is incremented and the result is permanently accumulated into `unlockedCount[asset]`. There is no mechanism to subtract from `unlockedCount` if a request is subsequently completed.

**`completeWithdrawal` deletes the request and decrements the live counter, but never touches `unlockedCount`:** [2](#0-1) 

`delete withdrawalRequests[requestId]` zeroes `expectedAssetAmount`. `unlockedWithdrawalsCount[asset]--` decrements the live counter. Neither operation touches `unlockedCount` in the initializer contract.

**`finalizeInitialize2` blindly passes the stale accumulated count:** [3](#0-2) 

**`initialize2` overwrites the live counter with the stale value (one-time, non-reversible via `reinitializer(2)`):** [4](#0-3) 

**`sweepRemainingAssets` is permanently gated on `hasUnlockedWithdrawals`:** [5](#0-4) 

**`hasUnlockedWithdrawals` is a pure counter check with no floor guard:** [6](#0-5) 

**Exploit flow:**
1. Operator calls `processChunk(asset, N)` — counts N unlocked requests, stores in `unlockedCount[asset] = N`.
2. A user (or operator on behalf of a user) calls `completeWithdrawal` for one of those N requests — `unlockedWithdrawalsCount[asset]` decrements to `N-1`; `unlockedCount` remains `N`.
3. Manager calls `finalizeInitialize2` — passes stale `N` to `initialize2`, which sets `unlockedWithdrawalsCount[asset] = N`.
4. Only `N-1` real withdrawals remain. After all are completed: `unlockedWithdrawalsCount[asset] = N - (N-1) = 1`.
5. `hasUnlockedWithdrawals(asset)` returns `true` forever. `sweepRemainingAssets` always reverts with `PendingWithdrawalsExist`.

**Existing guards are insufficient:** `onlyBeforeInitialize2` only prevents double-initialization; it does not prevent `completeWithdrawal` from executing during the window. `whenNotPaused` on `completeWithdrawal` would block this only if the operator manually pauses — not enforced by code. `reinitializer(2)` on `initialize2` makes the stale write permanent.

## Impact Explanation

After `initialize2` sets `unlockedWithdrawalsCount[asset]` to a value 1 (or more) higher than the true remaining count, every subsequent `completeWithdrawal` decrements it. Once all real withdrawals are exhausted the counter sits at ≥1, `hasUnlockedWithdrawals` returns `true` forever, and `sweepRemainingAssets` always reverts. Any remaining protocol balance (rounding residuals, uncommitted assets, or any ETH/LST held by the withdrawal manager) is permanently frozen in the contract with no non-upgrade recovery path. This matches **Critical — Permanent freezing of funds**.

## Likelihood Explanation

The initialization window is not atomic: `processChunk` is called by an operator in one or more transactions; `finalizeInitialize2` is called by a manager in a separate transaction. The gap can span many blocks. During this window, `completeWithdrawal` is callable by any user whose withdrawal delay (default 8 days / ~57,600 blocks) has passed — a routine, expected user action requiring no special role, front-running, or key compromise. On a live deployment with active users, at least one completion during the window is highly probable. The only mitigation is a manual pause before `processChunk` and unpause after `finalizeInitialize2`, which is not enforced by the code.

## Recommendation

1. **Recount at finalization time**: replace the accumulated `unlockedCount` with a live recount in `finalizeInitialize2` using `getUnlockedWithdrawalsCount` (which re-reads live `withdrawalRequests` state), eliminating the stale-snapshot window entirely.
2. **Enforce atomicity via pause**: require the contract to be paused during the entire `processChunk` → `finalizeInitialize2` sequence (enforced in code, not just documentation).
3. **Alternatively**, track completions that occur during the window and subtract them from `unlockedCount` before calling `initialize2`.

## Proof of Concept

```solidity
// Setup: 3 requests unlocked for asset A (nonces 0,1,2), nextLockedNonce=3
// unlockedWithdrawalsCount[A] = 3 (set by prior unlockQueue calls)

// Step 1: operator calls processChunk(A, 3)
//   → unlockedCount[A] = 3, processedIndex[A] = 3

// Step 2: unlockQueue advances nextLockedNonce to 4 (nonce 3 unlocked)
//   → unlockedWithdrawalsCount[A] = 4

// Step 3: operator calls processChunk(A, 1)
//   → unlockedCount[A] = 4, processedIndex[A] = 4

// Step 4: user calls completeWithdrawal(A) for nonce 0
//   → delete withdrawalRequests[keccak(A,0)]  (expectedAssetAmount → 0)
//   → unlockedWithdrawalsCount[A] = 3

// Step 5: manager calls finalizeInitialize2()
//   → isAssetComplete: processedIndex[A]=4 >= nextLockedNonce[A]=4 → true
//   → initialize2(4, ...) called
//   → unlockedWithdrawalsCount[A] = 4  (overwritten with stale value)

// Now only 3 real withdrawals remain (nonces 1,2,3).
// After users complete all 3: unlockedWithdrawalsCount[A] = 4 - 3 = 1
// hasUnlockedWithdrawals(A) == true  ← permanently
// sweepRemainingAssets(A) reverts forever with PendingWithdrawalsExist

assert(withdrawalManager.unlockedWithdrawalsCount(A) == 1);
assert(withdrawalManager.hasUnlockedWithdrawals(A) == true);
vm.expectRevert(LRTWithdrawalManager.PendingWithdrawalsExist.selector);
withdrawalManager.sweepRemainingAssets(A);
```

A Foundry fork test against the deployed contracts can reproduce this by: (1) forking mainnet post-deployment, (2) impersonating an operator to call `processChunk`, (3) impersonating a user with a matured withdrawal to call `completeWithdrawal`, (4) impersonating a manager to call `finalizeInitialize2`, then (5) completing all remaining withdrawals and asserting `sweepRemainingAssets` reverts.
