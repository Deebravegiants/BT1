### Title
Missing Chain ID in `DelegateAction` Signed Payload Enables Cross-Fork Replay of Meta Transactions - (File: `core/primitives/src/action/delegate.rs`)

### Summary
`DelegateAction` (NEP-366 meta transactions) and `DelegateActionV2` (NEP-611 gas keys) are off-chain signed messages that authorize a relayer to execute actions on behalf of a user. Neither the signed payload nor the `SignableMessage` wrapper includes a chain/network identifier. If the NEAR chain forks after deployment, a signed `DelegateAction` is cryptographically valid on both forks, allowing any party who observed the signed message to replay it on the other fork and execute the inner actions (transfers, key additions, etc.) without the user's consent on that chain.

### Finding Description

`DelegateAction` is the NEAR analog of ERC20Permit: a user signs it off-chain and hands it to a relayer, who submits it on-chain. The struct contains:

```rust
pub struct DelegateAction {
    pub sender_id: AccountId,
    pub receiver_id: AccountId,
    pub actions: Vec<NonDelegateAction>,
    pub nonce: Nonce,
    pub max_block_height: BlockHeight,
    pub public_key: PublicKey,
}
``` [1](#0-0) 

No chain ID field is present. The hash signed by the user is computed as:

```rust
pub fn get_nep461_hash(&self) -> CryptoHash {
    let signable = SignableMessage::new(&self, SignableMessageType::DelegateAction);
    let bytes = borsh::to_vec(&signable).expect("Failed to deserialize");
    hash(&bytes)
}
``` [2](#0-1) 

`SignableMessage` prepends only a `MessageDiscriminant` (a NEP number encoded as a `u32`), with no chain ID:

```rust
pub struct MessageDiscriminant {
    discriminant: u32,  // just a NEP number, no chain identifier
}
``` [3](#0-2) 

The same omission applies to `DelegateActionV2` / `VersionedDelegateActionPayload::get_nep461_hash()`. [4](#0-3) 

At execution time, `apply_delegate_action` verifies the signature and checks `max_block_height`, `sender_id`, and the access key — but never validates any chain identifier:

```rust
pub(crate) fn apply_delegate_action(...) {
    if !signed_delegate_action.verify() { ... }
    if apply_state.block_height > delegate_action.max_block_height() { ... }
    if delegate_action.sender_id().as_str() != sender_id.as_str() { ... }
    validate_delegate_action_key(...)?;
    // No chain ID check anywhere
``` [5](#0-4) 

### Impact Explanation

After a post-deployment hard fork of NEAR, both chains share the same account state, nonces, and block heights up to the fork point. A `DelegateAction` signed on chain A (new fork) with `nonce = N` and `max_block_height = H` is byte-for-byte identical and cryptographically valid on chain B (old fork), because the signed hash covers no chain-specific data. Any party who received the signed `DelegateAction` off-chain (e.g., a relayer operating on both forks) can submit it to the other fork. The inner actions execute with `sender_id` as the predecessor — corrupting the victim's account state (balance drained, access keys added/deleted, arbitrary contract calls made) on the unintended chain.

The corrupted protocol values are: account balance (via `Transfer` inner actions), access key set (via `AddKey`/`DeleteKey` inner actions), account nonce, and any contract state mutated by inner `FunctionCall` actions.

### Likelihood Explanation

Hard forks of live blockchains are uncommon but not unprecedented. The window of vulnerability is bounded by `max_block_height` (set by the user, typically a few hundred blocks). Any relayer or observer who received the signed `DelegateAction` off-chain before the fork can perform the replay with no special privileges — only a public RPC call is needed. The attack requires no validator access.

### Recommendation

Include the chain ID (the `chain_id` string from genesis / `StatusResponse`) in the `DelegateAction` signed payload, or include it in the `SignableMessage` / `MessageDiscriminant` scheme. The ETH-implicit Wallet Contract already demonstrates the correct pattern: it hard-codes `CHAIN_ID` and rejects any transaction where `tx.chain_id != Some(CHAIN_ID)`. [6](#0-5) 

Short term: add a `chain_id: String` field to `DelegateAction` and `DelegateActionV2`, include it in `get_nep461_hash()`, and validate it in `apply_delegate_action`.

Long term: standardize chain-ID binding in the `SignableMessage` / NEP-461 scheme so all future off-chain signed message types inherit the protection automatically.

### Proof of Concept

1. NEAR mainnet forks at block height `F`. Both chains share identical state for all accounts up to block `F`.
2. Alice signs a `DelegateAction` on the new fork (chain A) authorizing relayer R to transfer 100 NEAR to Bob. She sets `nonce = N` (her current access-key nonce + 1) and `max_block_height = F + 200`.
3. R submits the meta transaction on chain A; it executes successfully, advancing Alice's nonce to `N` on chain A.
4. R (or any observer of the signed payload) submits the **identical** `SignedDelegateAction` bytes on chain B (old fork). Alice's nonce on chain B is still `N - 1` (the fork state), so the nonce check passes. `max_block_height` is still in the future. The signature verifies because the hash covers no chain-specific data.
5. `apply_delegate_action` on chain B executes the inner `Transfer`, draining 100 NEAR from Alice's chain-B account without her authorization. [7](#0-6) [8](#0-7)

impl SignedDelegateAction {
    pub fn verify(&self) -> bool {
        let delegate_action = &self.delegate_action;
        let hash = delegate_action.get_nep461_hash();
        let public_key = &delegate_action.public_key;

        self.signature.verify(hash.as_ref(), public_key)
    }

    pub fn sign(singer: &Signer, delegate_action: DelegateAction) -> Self {
        let signature = singer.sign(delegate_action.get_nep461_hash().as_bytes());
        Self { delegate_action, signature }
    }

pub(crate) fn apply_delegate_action(
    state_update: &mut TrieUpdate,
    apply_state: &ApplyState,
    action_receipt: &VersionedActionReceipt,
    sender_id: &AccountId,
    signed_delegate_action: VersionedSignedDelegateActionRef<'_>,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    if !signed_delegate_action.verify() {
        result.result = Err(ActionErrorKind::DelegateActionInvalidSignature.into());
        return Ok(());
    }
    let delegate_action = signed_delegate_action.delegate_action();
    if apply_state.block_height > delegate_action.max_block_height() {
        result.result = Err(ActionErrorKind::DelegateActionExpired.into());
        return Ok(());
    }
    if delegate_action.sender_id().as_str() != sender_id.as_str() {
        result.result = Err(ActionErrorKind::DelegateActionSenderDoesNotMatchTxReceiver {
            sender_id: delegate_action.sender_id().clone(),
            receiver_id: sender_id.clone(),
        }
        .into());
        return Ok(());
    }

    validate_delegate_action_key(state_update, apply_state, delegate_action, result)?;
    if result.result.is_err() {
        // Validation failed. Need to return Ok() because this is not a runtime error.
        // "result.result" will be return to the User as the action execution result.
        return Ok(());
    }
```
