### Title
Missing Minimum Confirmation Threshold Enforcement in `verify_foreign_transaction` Enables Double-Spend via Premature Signing of Unfinalized Bitcoin Transactions - (File: `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`, `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

The `verify_foreign_transaction` contract method accepts a caller-supplied `BlockConfirmations(0)` threshold with no minimum floor enforced anywhere in the contract or node. The `BitcoinInspector` check is trivially satisfied when the threshold is zero, causing the MPC network to sign a Bitcoin transaction that has not achieved meaningful finality. In bridge flows (e.g., Omnibridge inbound), this enables a double-spend: an attacker obtains a valid MPC threshold signature for a Bitcoin transaction after only one confirmation, claims NEAR-side funds, then reverses the Bitcoin transaction via a natural or induced reorganization.

---

### Finding Description

`BlockConfirmations` is a plain `u64` newtype with no minimum constraint: [1](#0-0) 

The ABI schema confirms `"minimum": 0.0` is the only bound: [2](#0-1) 

The contract's `verify_foreign_transaction` entry point performs no validation on the `confirmations` field of a `BitcoinRpcRequest`: [3](#0-2) 

The `BitcoinRpcRequest` struct stores the caller-supplied value verbatim: [4](#0-3) 

On the node side, `execute_foreign_chain_request` passes the caller-supplied `request.confirmations.0` directly to the inspector as `block_confirmations`: [5](#0-4) 

Inside `BitcoinInspector::extract`, the only finality gate is:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
if !enough_block_confirmations {
    return Err(...);
}
``` [6](#0-5) 

When `block_confirmations_threshold = 0`, the predicate `0 <= actual_confirmations` is trivially true for any transaction that `getrawtransaction` returns (i.e., any transaction with ≥ 1 confirmation, since `blockhash` is required for deserialization). The MPC network then proceeds to produce a threshold signature over the payload hash without any finality guarantee.

The existing test suite explicitly confirms that an empty extractor list and a zero-threshold request are accepted without error: [7](#0-6) 

---

### Impact Explanation

The primary use case for `verify_foreign_transaction` is the Omnibridge inbound flow: a bridge contract calls this method to obtain an MPC attestation that a Bitcoin deposit transaction finalized, then releases NEAR-side tokens. If the bridge contract (or any caller) supplies `confirmations: 0` or `confirmations: 1`, the MPC network signs after a single confirmation. Bitcoin transactions with one confirmation are routinely reorganized during normal chain operation and can be deliberately reversed by a miner with sufficient hash power. An attacker who obtains the MPC signature at one confirmation can:

1. Claim NEAR-side bridge funds using the valid MPC signature.
2. Collude with a miner (or exploit a natural reorg) to reverse the Bitcoin deposit transaction.
3. Retain both the NEAR-side tokens and the original Bitcoin.

This is a direct double-spend enabled by missing minimum output/quality validation — the exact vulnerability class of the reference report (hardcoded `minAmountOut = 0`). The MPC signature is irrevocable once issued; there is no on-chain mechanism to claw it back after a reorg.

Impact maps to: **High — cross-chain double-spend condition caused by invalid bridge execution**.

---

### Likelihood Explanation

The attacker-controlled entry path is fully reachable by any unprivileged NEAR account:

- `verify_foreign_transaction` is a public, payable method requiring only a 1 yoctoNEAR deposit.
- The `confirmations` field is caller-supplied with no contract-side floor.
- A bridge contract that does not independently enforce a minimum (or a malicious bridge contract) is sufficient to trigger the condition.

Bitcoin reorganizations of 1–2 blocks occur naturally several times per year on mainnet; deliberate 1-block reorgs are economically feasible for large mining pools. The likelihood is **Medium**: the attacker needs a cooperating or negligent bridge contract and a Bitcoin reorg, but neither requires threshold-level collusion within the MPC network itself.

---

### Recommendation

Enforce a protocol-level minimum confirmation threshold in the contract's `verify_foreign_transaction` method before enqueuing the request. A reasonable floor for Bitcoin is 6 confirmations; the exact value should be a governance-controlled parameter rather than a hardcoded constant. Reject any `BitcoinRpcRequest` whose `confirmations` field is below the floor:

```rust
// In verify_foreign_transaction, after chain support check:
if let ForeignChainRpcRequest::Bitcoin(ref btc_req) = request.request {
    if btc_req.confirmations.0 < MIN_BITCOIN_CONFIRMATIONS {
        env::panic_str("Bitcoin confirmations below protocol minimum");
    }
}
```

Similarly, consider whether `EvmFinality::Latest` should be accepted for bridge-purpose domains, since `Latest` blocks are also subject to reorganization.

---

### Proof of Concept

A malicious or negligent bridge contract submits:

```json
{
  "request": {
    "Bitcoin": {
      "tx_id": "<attacker_deposit_txid>",
      "confirmations": 0,
      "extractors": ["BlockHash"]
    }
  },
  "domain_id": <foreign_tx_domain>,
  "payload_version": 1
}
```

The contract enqueues the request without validation. Each MPC node calls `execute_foreign_chain_request`, which calls `BitcoinInspector::extract` with `block_confirmations_threshold = 0`. The check `0 <= actual_confirmations` passes for any confirmed transaction. The node signs the payload. The threshold signature is returned to the bridge contract, which releases NEAR-side tokens. The attacker then reverses the Bitcoin transaction via a reorg, retaining both assets.
