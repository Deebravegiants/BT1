Audit Report

## Title
rsETH Price Inflation via ETH Donation and Permissionless Price Update - (File: contracts/LRTDepositPool.sol / contracts/LRTOracle.sol)

## Summary
An attacker who is the first depositor can send ETH directly to `LRTDepositPool` via its open `receive()` function, then call the permissionless `LRTOracle.updateRSETHPrice()` to inflate `rsETHPrice` by an arbitrary multiple. Subsequent depositors receive drastically fewer rsETH tokens due to integer-division truncation, while the attacker's seed position represents a disproportionate share of the pool, enabling direct theft of depositor funds.

## Finding Description

**Root cause chain:**

1. `LRTDepositPool.receive()` accepts arbitrary ETH from any sender with no accounting: [1](#0-0) 

2. `getETHDistributionData()` uses raw `address(this).balance`, which includes donated ETH, as the canonical ETH-in-pool figure: [2](#0-1) 

3. `_getTotalEthInProtocol()` in `LRTOracle` calls `getTotalAssetDeposits(ETH)` which resolves to the above balance, so donated ETH inflates `totalETHInProtocol`: [3](#0-2) 

4. `updateRSETHPrice()` is fully permissionless — any caller can trigger a price update: [4](#0-3) 

5. The only guard against a large price jump is `pricePercentageLimit`, which is **zero by default** (never set in `initialize()`), making the condition always `false`: [5](#0-4) 

6. The new price is accepted and stored unconditionally: [6](#0-5) [7](#0-6) 

7. The minting formula divides by the inflated `rsETHPrice`, causing integer truncation to zero or near-zero for victim deposits: [8](#0-7) 

8. The slippage guard in `_beforeDeposit` only reverts if `rsethAmountToMint < minRSETHAmountExpected`; when the victim passes `0`, even 1 wei satisfies the check: [9](#0-8) 

**Why existing checks fail:**
- `minAmountToDeposit` is `0` by default (not set in `initialize()`), so the only deposit-size guard is `depositAmount == 0`: [10](#0-9) 
- `pricePercentageLimit == 0` disables the price-jump guard entirely for unprivileged callers.
- Protocol fee rounding: even if `protocolFeeInBPS > 0`, at the inflated price `rsethAmountToMintAsProtocolFee` rounds to 0, so `_checkAndUpdateDailyFeeMintLimit(0)` is called and does not revert.

## Impact Explanation

**Critical — Direct theft of depositor funds.**

The attacker inflates `rsETHPrice` so that a victim's large ETH deposit mints only 1 wei of rsETH. The attacker's 1 wei rsETH then represents 50% of total supply, entitling them to half the combined pool (which includes the victim's ETH). The victim recovers only ~50% of their deposit; the attacker extracts the difference as profit.

## Likelihood Explanation

**Medium.** The attack is most effective at protocol launch or when rsETH supply is near zero. Required conditions: (1) attacker is first depositor or front-runs the victim; (2) attacker donates ETH ≈ equal to the victim's intended deposit; (3) victim calls `depositETH` with `minRSETHAmountExpected = 0`. All three are realistic: the protocol starts with zero supply, the donation is a plain ETH transfer, and many integrations or naive users omit the slippage parameter. The attack is repeatable whenever supply is low.

## Recommendation

1. **Replace `address(this).balance` with an internal accounting variable** incremented only by legitimate deposit paths (`depositETH`, `receiveFromNodeDelegator`, etc.), so direct ETH transfers are excluded from `totalETHInProtocol`.
2. **Set a non-zero `pricePercentageLimit`** in `initialize()` (e.g., 1% = `1e16`) to cap permissionless price updates.
3. **Set a meaningful `minAmountToDeposit`** in `initialize()` to raise the cost of the seed deposit.
4. **Require `minRSETHAmountExpected > 0`** in `depositETH` / `depositAsset` to force callers to declare slippage tolerance.

## Proof of Concept

```
Precondition: ETH is a supported asset with depositLimit = 100e18.
              pricePercentageLimit = 0 (default). minAmountToDeposit = 0 (default).

Step 0 — Attacker calls updateRSETHPrice() (rsethSupply == 0):
  rsETHPrice = 1e18, highestRsethPrice = 1e18

Step 1 — Attacker calls depositETH{value: 1 wei}(0, ""):
  rsethAmountToMint = (1 * 1e18) / 1e18 = 1 wei rsETH
  Pool ETH balance = 1 wei, rsETH supply = 1 wei

Step 2 — Attacker sends 10e18 - 1 wei ETH directly to LRTDepositPool (receive()):
  Pool ETH balance = 10e18 wei

Step 3 — Attacker calls LRTOracle.updateRSETHPrice():
  totalETHInProtocol = 10e18
  previousTVL = 1 * 1e18 = 1e18
  newRsETHPrice = 10e18 * 1e18 / 1 = 10e36
  pricePercentageLimit == 0 → isPriceIncreaseOffLimit = false → price accepted
  rsETHPrice = 10e36

Step 4 — Victim calls depositETH{value: 19e18}(0, ""):
  rsethAmountToMint = (19e18 * 1e18) / 10e36 = 1 wei rsETH  ← truncated
  minRSETHAmountExpected = 0 → 1 >= 0 → passes
  Pool ETH balance = 29e18, rsETH supply = 2 wei

Step 5 — Attacker redeems 1 wei rsETH (50% of supply):
  Attacker receives ≈ 14.5 ETH
  Attacker spent: 10e18 ETH total → profit ≈ 4.5 ETH
  Victim deposited 19e18 ETH, recovers ≈ 14.5 ETH → loss ≈ 4.5 ETH
```

Foundry test plan: deploy `LRTConfig`, `LRTDepositPool`, `LRTOracle`, `RSETH` with ETH as a supported asset; execute the five-step sequence above in a single test function; assert `rsETHPrice == 10e36` after Step 3 and `rsethAmountToMint == 1` for the victim's 19 ETH deposit in Step 4.
