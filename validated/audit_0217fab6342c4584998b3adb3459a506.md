### Title
Malicious Harvester Can Unconditionally Redirect Farmer Block Rewards via Unenforced `farmer_reward_address_override` — (File: `chia/farmer/farmer_api.py`, `chia/farmer/farmer.py`)

---

### Summary

The CHIP-22 third-party harvester fee mechanism allows a connected harvester to redirect the farmer's block reward by setting `farmer_reward_address_override` in protocol messages. The farmer's fee-quality enforcement is purely advisory — it only logs warnings on violation but never rejects the override. A malicious harvester can unconditionally redirect any block reward to an arbitrary address with no cryptographic or protocol-level enforcement stopping it.

---

### Finding Description

CHIP-22 (implemented in `chia/farmer/farmer_api.py` and `chia/farmer/farmer.py`) allows third-party harvesters to claim a fee by setting `farmer_reward_address_override` in two protocol messages:

1. `harvester_protocol.NewProofOfSpace.farmer_reward_address_override` [1](#0-0) 
2. `harvester_protocol.RespondSignatures.farmer_reward_address_override` [2](#0-1) 

**Step 1 — `new_proof_of_space` handler (advisory-only check):**

When `NewProofOfSpace` arrives with `farmer_reward_address_override != None`, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()`:

```python
if new_proof_of_space.farmer_reward_address_override is not None:
    self.farmer.notify_farmer_reward_taken_by_harvester_as_fee(sp, new_proof_of_space)
``` [3](#0-2) 

Inside `notify_farmer_reward_taken_by_harvester_as_fee`, the fee quality is computed and compared against the harvester-supplied threshold. When the check fails — including when `fee_info` is `None` entirely — the function **only logs a warning and returns `None`**. It never signals rejection back to the caller:

```python
else:
    self.log.warning(
        "Harvester illegitimately took reward by failing to provide its fee rate ..."
    )
``` [4](#0-3) 

The `new_proof_of_space` handler ignores the return value and continues processing unconditionally.

**Step 2 — `_process_respond_signatures` (no check at all):**

When the harvester's `RespondSignatures` arrives, the farmer unconditionally substitutes the harvester-supplied address as the block reward destination with zero fee-quality verification:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
    include_source_signature_data = True
``` [5](#0-4) 

This address is then placed directly into `DeclareProofOfSpace` and broadcast to the full node:

```python
return farmer_protocol.DeclareProofOfSpace(
    ...
    farmer_reward_address,
    ...
)
``` [6](#0-5) 

**Attack path:**

A malicious harvester connected via mutual TLS (the normal third-party harvester scenario CHIP-22 targets) can:

1. Send `NewProofOfSpace` with `farmer_reward_address_override` set to the attacker's address and `fee_info.applied_fee_threshold = 0xFFFFFFFF` (always passes the advisory check), or omit `fee_info` entirely.
2. The farmer logs a warning but continues, sends `RequestSignatures` back.
3. The harvester responds with `RespondSignatures` also carrying `farmer_reward_address_override` pointing to the attacker's address.
4. The farmer uses the attacker's address in `DeclareProofOfSpace` — the block reward is diverted.

There is no cross-check that the address in `RespondSignatures` matches what was in `NewProofOfSpace`, and no fee-quality gate in `_process_respond_signatures`.

---

### Impact Explanation

**HIGH** — Unauthorized reward diversion affecting XCH. Every block won by the farmer while connected to a malicious harvester has its farmer reward (currently 0.25 XCH) redirected to an attacker-controlled address. The farmer suffers direct, permanent financial loss with no on-chain recourse. The full node and consensus layer accept the `DeclareProofOfSpace` as valid because the cryptographic signatures are correct — only the reward destination is wrong.

---

### Likelihood Explanation

The attacker must be a connected harvester (mutual TLS is required). In the third-party harvester model that CHIP-22 explicitly enables, farmers are expected to connect to external harvesters they do not fully control. Any such harvester can exploit this silently. The farmer's only signal is a log warning, which is easily missed in production.

---

### Recommendation

1. **Enforce the fee-quality gate, not just log it.** `notify_farmer_reward_taken_by_harvester_as_fee` should return a boolean; `new_proof_of_space` should abort processing (return `None`) when the check fails.
2. **Re-verify in `_process_respond_signatures`.** Before accepting `RespondSignatures.farmer_reward_address_override`, confirm that the corresponding `NewProofOfSpace` also carried the same override and that the fee-quality check passed for that proof.
3. **Reject `farmer_reward_address_override` when `fee_info` is absent.** Currently the farmer only logs a warning; it should treat a missing `fee_info` as an invalid override and drop the proof.

---

### Proof of Concept

```
Attacker (malicious third-party harvester):

1. Connect to farmer via mutual TLS as NodeType.HARVESTER.

2. On receiving NewSignagePointHarvester, send NewProofOfSpace with:
     farmer_reward_address_override = <attacker_puzzle_hash>
     fee_info = ProofOfSpaceFeeInfo(applied_fee_threshold=0xFFFFFFFF)
   → Farmer calls notify_farmer_reward_taken_by_harvester_as_fee(),
     fee_quality (≤ 0xFFFFFFFF always), logs INFO, continues.

3. Farmer sends RequestSignatures back to harvester.

4. Harvester responds with RespondSignatures:
     farmer_reward_address_override = <attacker_puzzle_hash>

5. Farmer executes _process_respond_signatures():
     farmer_reward_address = response.farmer_reward_address_override
     → DeclareProofOfSpace(farmer_reward_address=<attacker_puzzle_hash>)

6. Full node accepts the block; 0.25 XCH reward is paid to attacker.
   Farmer receives nothing. No on-chain evidence of tampering.
```
