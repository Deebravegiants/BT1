Audit Report

## Title
`claimAndStake()` Always Reverts Due to Missing `STAKE_FOR_ROLE` Grant on Distributor Contracts - (`contracts/KERNEL/KernelMerkleDistributor.sol`, `contracts/KERNEL/KernelTop100MerkleDistributor.sol`)

## Summary
Both `KernelMerkleDistributor` and `KernelTop100MerkleDistributor` expose a `claimAndStake()` function that calls `KernelDepositPool.stakeFor()`. That function is gated by `onlyRole(STAKE_FOR_ROLE)`, but no code in the codebase ever grants either distributor contract that role. Every call to `claimAndStake()` reverts unconditionally, rendering the atomic claim-and-stake feature completely non-functional.

## Finding Description
`KernelDepositPool.stakeFor()` enforces `onlyRole(STAKE_FOR_ROLE)` as its first effective check:

```solidity
// KernelDepositPool.sol L296-L303
function stakeFor(address _account, uint256 _amount)
    external
    nonReentrant
    onlyRole(STAKE_FOR_ROLE)   // ← reverts if caller lacks role
    updateReward(_account)
{ ... }
```

`KernelDepositPool.initialize()` only grants `DEFAULT_ADMIN_ROLE` — `STAKE_FOR_ROLE` is never assigned to any address:

```solidity
// KernelDepositPool.sol L259-L271
_setupRole(DEFAULT_ADMIN_ROLE, _admin);
// no STAKE_FOR_ROLE grant
```

`KernelMerkleDistributor.claimAndStake()` calls `stakeFor()` directly after processing the claim:

```solidity
// KernelMerkleDistributor.sol L280-L284
uint256 amountToStake = _processClaim(index, account, cumulativeAmount, merkleProof);
IKernelDepositPool(kernelDepositPool).stakeFor(account, amountToStake); // ← always reverts
```

`KernelTop100MerkleDistributor.claimAndStake()` does the same:

```solidity
// KernelTop100MerkleDistributor.sol L371
kernelDepositPool.stakeFor(user, amountToStake); // ← always reverts
```

Neither distributor's `initialize()` contains any `grantRole` call for `STAKE_FOR_ROLE`. A codebase-wide search for `STAKE_FOR_ROLE` confirms it appears only in `KernelDepositPool.sol` (definition + usage) and in the interface comment in `KernelMerkleDistributor.sol` — never in a `grantRole` call. The ERC-20 `forceApprove` calls in both distributors' `initialize()` functions address token allowance for the `safeTransferFrom` inside `stakeFor()`, but the role check fires before the transfer and causes the revert regardless.

## Impact Explanation
Every invocation of `claimAndStake()` on either distributor reverts at the `onlyRole(STAKE_FOR_ROLE)` check. No state is mutated and no funds are lost, but the advertised atomic claim-and-stake feature is entirely non-functional from deployment. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
The failure is deterministic and unconditional from the moment of deployment. Any user who calls `claimAndStake()` with a valid merkle proof will receive a revert. The broken state persists until an admin separately calls `grantRole(STAKE_FOR_ROLE, distributorAddress)` on `KernelDepositPool`. The likelihood of a user encountering the broken state is high, as the function is publicly advertised.

## Recommendation
Grant `STAKE_FOR_ROLE` to each distributor contract as part of the deployment sequence. The cleanest approach is to call `grantRole(STAKE_FOR_ROLE, distributorAddress)` on `KernelDepositPool` from the admin account after deploying each distributor, or add a dedicated setter on `KernelDepositPool` (restricted to `DEFAULT_ADMIN_ROLE`) that grants the role to a specified address. Alternatively, each distributor's `initialize()` could accept the pool address and call `IAccessControl(pool).grantRole(STAKE_FOR_ROLE, address(this))` — provided the deployer holds `DEFAULT_ADMIN_ROLE` at that point.

## Proof of Concept
1. Deploy `KernelDepositPool` with `initialize(admin, kernelToken, rewardToken)`.
2. Deploy `KernelMerkleDistributor` with `initialize(kernel, poolAddress, treasury, fee)`.
3. Admin calls `setMerkleRoot(root)` on the distributor.
4. Fund the distributor with KERNEL tokens.
5. A user with a valid proof calls `claimAndStake(index, account, amount, proof)`.
6. `_processClaim` succeeds and returns `amountToStake`.
7. `IKernelDepositPool(kernelDepositPool).stakeFor(account, amountToStake)` is called.
8. `KernelDepositPool.stakeFor()` executes `onlyRole(STAKE_FOR_ROLE)` — the distributor does not hold this role → transaction reverts with `AccessControl` error.
9. Repeat steps 5–8 for `KernelTop100MerkleDistributor.claimAndStake()` — identical revert.

Foundry test sketch:
```solidity
function test_claimAndStakeReverts() public {
    // setup: deploy pool, distributor, fund distributor, set merkle root
    vm.prank(user);
    vm.expectRevert(); // AccessControl revert
    distributor.claimAndStake(index, user, amount, proof);
}
```
