Audit Report

## Title
Uninitialized `minRsEthAmountToWithdraw` allows dust-request flooding to temporarily freeze legitimate withdrawals — (File: `contracts/LRTWithdrawalManager.sol`)

## Summary

`minRsEthAmountToWithdraw[asset]` is never set in `initialize()`, defaulting to `0` for every asset. The guard in `initiateWithdrawal()` therefore collapses to a single `rsETHUnstaked == 0` check, accepting any amount ≥ 1 wei. An unprivileged rsETH holder can flood the FIFO withdrawal queue with dust entries; because `_unlockWithdrawalRequests()` breaks on the first entry whose delay has not yet elapsed, all legitimate requests queued behind the dust entries are temporarily frozen until the operator drains every preceding entry across many expensive transactions.

## Finding Description

`LRTWithdrawalManager` declares the per-asset minimum as a mapping:

```solidity
mapping(address asset => uint256) public minRsEthAmountToWithdraw;
``` [1](#0-0) 

`initialize()` never writes to it, leaving every asset at the Solidity default of `0`: [2](#0-1) 

The guard in `initiateWithdrawal()` is:

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
``` [3](#0-2) 

When `minRsEthAmountToWithdraw[asset] == 0`, the second condition `rsETHUnstaked < 0` is always false for `uint256`, so any value ≥ 1 wei passes unchecked.

`_unlockWithdrawalRequests()` iterates the queue in strict ascending nonce order and **breaks** (not continues) on the first entry whose delay has not elapsed:

```solidity
if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) break;
``` [4](#0-3) 

The nonce pointer is only advanced past entries that are fully processed: [5](#0-4) 

Any request at nonce `N` cannot be unlocked until all requests at nonces `0 … N-1` have been processed. Dust entries inserted before a legitimate request block it until the operator drains every preceding entry.

`setMinRsEthAmountToWithdraw` also accepts `0` as a valid value (no zero-check), so even a post-deployment admin configuration can be silently reset: [6](#0-5) 

`LRTDepositPool.minAmountToDeposit` is similarly never set in `initialize()`, defaulting to `0`, so the attacker can acquire rsETH in arbitrarily small increments: [7](#0-6) 

The deposit guard mirrors the same pattern: [8](#0-7) 

## Impact Explanation

**Temporary freezing of funds (Medium).** Legitimate users whose `initiateWithdrawal()` calls are queued after the attacker's dust entries cannot have their requests unlocked — and therefore cannot call `completeWithdrawal()` — until the operator processes every preceding dust entry. With `withdrawalDelayBlocks` already set to `8 days / 12 seconds` (~57,600 blocks), inserting N dust entries ahead of a legitimate request extends the effective freeze by however many `unlockQueue` batches are needed to drain N entries.

**Unbounded gas consumption (Medium).** Each `unlockQueue` call processes a bounded batch. The total number of operator transactions and total ETH spent on gas scales linearly with the number of dust requests created. At ~5,000–10,000 gas per loop iteration and a 30 M gas block limit, 1,000,000 dust entries require ~167–333 separate operator calls. [9](#0-8) 

## Likelihood Explanation

Any address holding rsETH can execute this attack. rsETH is freely obtainable by calling `LRTDepositPool.depositETH()` or `depositAsset()`. Because `minAmountToDeposit` also defaults to `0`, the attacker can acquire rsETH in arbitrarily small increments. The attacker's only cost is gas for creating the dust requests and the temporary lock-up of their own rsETH (which they eventually recover). The cost asymmetry strongly favours the attacker: N dust requests cost the attacker N `initiateWithdrawal` transactions, while forcing the operator to spend gas on N `unlockQueue` iterations and blocking all legitimate users queued behind those entries.

## Recommendation

1. **Enforce a non-zero minimum in `initialize()`**: Set a sensible default for `minRsEthAmountToWithdraw` for each supported asset during initialisation (e.g., equivalent to ~0.001 ETH worth of rsETH).
2. **Guard `setMinRsEthAmountToWithdraw` against zero**: Add `if (minRsEthAmountToWithdraw_ == 0) revert InvalidMinAmount();` analogous to guards already present elsewhere in the codebase.
3. **Guard `setMinAmountToDeposit` against zero** in `LRTDepositPool` for the same reason.

## Proof of Concept

```
// Setup: minRsEthAmountToWithdraw[ETH_TOKEN] == 0 (default, never set)
//        minAmountToDeposit == 0 (default, never set)

1. Attacker calls LRTDepositPool.depositETH{value: 1 ether}(0, "")
   → receives ~1e18 rsETH (exact amount depends on exchange rate)

2. Attacker approves LRTWithdrawalManager to spend rsETH.

3. Attacker calls initiateWithdrawal(ETH_TOKEN, 1, "") 1,000,000 times
   (splitting 1e18 rsETH into 1e6 requests of 1 wei each).
   Each call passes the guard: (1 == 0) → false, (1 < 0) → false → no revert.
   Each call pushes a new entry into the FIFO queue at nonces 0 … 999,999.

4. Victim calls initiateWithdrawal(ETH_TOKEN, 1e18, "") → queued at nonce 1,000,000.

5. After withdrawalDelayBlocks pass, operator calls unlockQueue(ETH_TOKEN, ...).
   _unlockWithdrawalRequests iterates from nextLockedNonce=0.
   Must process all 1,000,000 dust entries before reaching the victim's entry.
   → Victim's withdrawal is locked until the operator drains all dust entries
     across ~167–333 expensive transactions.
   → Victim cannot call completeWithdrawal() until their request is unlocked.

Foundry test sketch:
- Deploy contracts with default initialization (no setMinRsEthAmountToWithdraw call).
- vm.prank(attacker); for (uint i; i < 1_000_000; i++) withdrawalManager.initiateWithdrawal(ETH_TOKEN, 1, "");
- vm.prank(victim); withdrawalManager.initiateWithdrawal(ETH_TOKEN, 1e18, "");
- vm.roll(block.number + withdrawalDelayBlocks + 1);
- Assert: victim's nonce (1_000_000) >= nextLockedNonce[ETH_TOKEN] → withdrawal still locked.
```
