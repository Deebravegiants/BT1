### Title
Access Key Nonce Reset via DeleteKey+AddKey Enables Transaction Replay — (`runtime/runtime/src/access_keys.rs`)

---

### Summary

`add_regular_key` unconditionally resets the access key nonce to `initial_nonce_value(block_height) = (block_height - 1) * 1_000_000` whenever a key is added. A batched transaction containing `[DeleteKey(K), AddKey(K)]` removes the `AddKeyAlreadyExists` guard (the key is gone by the time AddKey runs) and resets the nonce to the same value it had when the key was first created at that block height. Any previously signed transaction whose nonce falls in the now-valid range `((H-1)*1e6, H*1e6)` passes the monotonic nonce check and is accepted.

---

### Finding Description

**Root cause — `add_regular_key` nonce reset:** [1](#0-0) 

```rust
pub(crate) fn initial_nonce_value(block_height: BlockHeight) -> Nonce {
    (block_height - 1) * near_primitives::account::AccessKey::ACCESS_KEY_NONCE_RANGE_MULTIPLIER
}
``` [2](#0-1) 

Every call to `add_regular_key` unconditionally overwrites the stored nonce with `initial_nonce_value(block_height)`, regardless of what the key's nonce was before deletion.

**The `AddKeyAlreadyExists` guard is bypassed by DeleteKey in the same tx:** [3](#0-2) 

The guard reads the key from the current `TrieUpdate`. After `DeleteKey` executes in the same action list, the key is absent from the trie update, so `AddKey` for the same public key succeeds.

**The nonce check only requires `tx_nonce > current_nonce`:** [4](#0-3) 

After the reset, `current_nonce = (H-1)*1e6`. Any transaction signed with nonce `(H-1)*1e6 + k` (k ≥ 1) passes.

**The `NonceTooLarge` upper-bound check does not block the replay:** [5](#0-4) 

`upper_bound = H * 1e6`. A replayed nonce of `(H-1)*1e6 + 1` satisfies `tx_nonce < upper_bound`.

**No cross-block transaction deduplication exists:** [6](#0-5) 

`UniqueChunkTransactions` only deduplicates within a single chunk. There is no global check preventing the same transaction hash from being included in a later block after a nonce reset.

---

### Impact Explanation

**Scenario A — replay of a not-yet-included transaction (confirmed exploitable):**

1. Key K created at block H; nonce = `(H-1)*1e6`.
2. Attacker signs T1 (e.g., a payment) with nonce `(H-1)*1e6 + 1` but does not submit it.
3. Attacker submits T2 = `[DeleteKey(K), AddKey(K)]` with nonce `(H-1)*1e6 + 1`.
4. T2 is included: the runtime first sets `access_key.nonce = (H-1)*1e6 + 1` (transaction verification), then executes actions — DeleteKey removes K, AddKey re-adds K with nonce `(H-1)*1e6`.
5. After T2: `access_key.nonce = (H-1)*1e6`.
6. Attacker submits T1; nonce check `(H-1)*1e6 + 1 > (H-1)*1e6` passes → T1 is accepted.

**Scenario B — double-spend of an already-included transaction:**

1. Alice signs T1 (payment to Bob, nonce N) and gives it to Bob.
2. Bob submits T1; it is included; `access_key.nonce = N`.
3. Alice later submits `[DeleteKey(K), AddKey(K)]` with nonce N+1 (for any legitimate reason, e.g., key rotation to the same key).
4. After inclusion: `access_key.nonce = (H-1)*1e6`.
5. Bob re-submits T1 (nonce N > `(H-1)*1e6`); it is accepted again.
6. Bob receives the payment a second time.

The `transaction_validity_period` (86 400 blocks ≈ 1 day on mainnet) is the only time bound on how long after the original signing the replay can occur. [7](#0-6) 

---

### Likelihood Explanation

The prerequisite is that the account owner deletes and re-adds the **same** public key. This is unusual but protocol-permitted and can happen accidentally during key-rotation scripts that re-use the same keypair. The `AddKeyAlreadyExists` guard gives no protection because the delete and add are batched in one transaction. Once the nonce is reset, any party holding a previously signed transaction (e.g., a payment recipient) can replay it within the validity window.

---

### Recommendation

`add_regular_key` must not blindly reset the nonce to `initial_nonce_value`. When a key is re-added, the nonce should be set to `max(initial_nonce_value(block_height), previous_nonce + 1)` — but since the previous nonce is not retained after deletion, the safest fix is to set the nonce to `initial_nonce_value(block_height + 1)` (i.e., the start of the *next* block's range), ensuring no previously valid nonce in the current block's range can be reused. Alternatively, track the high-water-mark nonce for deleted keys in a separate trie column.

---

### Proof of Concept

```
Block H: Key K created → access_key.nonce = (H-1)*1_000_000

Step 1: Sign T1 = Transfer(to=Bob, amount=X), nonce=(H-1)*1e6+1, key=K
        (do not submit yet)

Step 2: Submit T2 = [DeleteKey(K), AddKey(K)], nonce=(H-1)*1e6+1, key=K
        → T2 included in block H
        → runtime sets access_key.nonce = (H-1)*1e6+1 (tx verification)
        → DeleteKey(K) executes: K removed from trie
        → AddKey(K) executes: K re-added, nonce = initial_nonce_value(H) = (H-1)*1e6
        → Final state: access_key.nonce = (H-1)*1e6

Step 3: Submit T1 (nonce=(H-1)*1e6+1)
        → verify_nonce: (H-1)*1e6+1 > (H-1)*1e6 ✓
        → NonceTooLarge: (H-1)*1e6+1 < H*1e6 ✓
        → T1 accepted; Bob receives X tokens

Step 4: Submit T3 = [DeleteKey(K), AddKey(K)], nonce=(H-1)*1e6+2
        → nonce reset to (H-1)*1e6 again

Step 5: Re-submit T1 → accepted again; Bob receives X tokens a second time
```

An integration test can assert `ProcessTxResponse::ValidTx` for T1 after the nonce reset, confirming the replay is accepted by the runtime.

pub fn check_transaction_validity_period(
    chain_store: &ChainStoreAdapter,
    prev_block_header: &BlockHeader,
    base_block_hash: &CryptoHash,
    transaction_validity_period: BlockHeightDelta,
) -> Result<(), InvalidTxError> {
    let base_header =
        chain_store.get_block_header(base_block_hash).map_err(|_| InvalidTxError::Expired)?;

    metrics::CHAIN_VALIDITY_PERIOD_CHECK_DELAY
        .observe(prev_block_header.height().saturating_sub(base_header.height()) as f64);

    // First check the distance between blocks
    if prev_block_header.height() > base_header.height() + transaction_validity_period {
        return Err(InvalidTxError::Expired);
    }
