### Title
Dijkstra Sub-Transaction Deposit Refund Double-Counting in `getConsumedDijkstraValue` — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, `getConsumedDijkstraValue` computes the total consumed value for a batch transaction by summing the consumed values of the top-level body and all sub-transactions. The `lookupStakingDeposit` closure — a static snapshot of the pre-transaction ledger state — is passed **unchanged** to every sub-transaction's consumed-value calculation. If two sub-transactions both carry an `UnRegDepositTxCert` for the same staking credential, both independently resolve the same deposit amount, inflating the total consumed value by one full deposit. The developers have explicitly acknowledged this gap in a disabled test with the comment `"TODO: predicate failure not yet implemented"`.

---

### Finding Description

**Root cause — `getConsumedDijkstraValue`** [1](#0-0) 

```haskell
getConsumedDijkstraValue pp lookupStakingDeposit lookupDRepDeposit utxo txBody =
  withBothTxLevels
    txBody
    ( \topTxBody ->
        txBodyConsumedValue topTxBody <> subTransactionsConsumedValue topTxBody
    )
    txBodyConsumedValue
  where
    txBodyConsumedValue :: forall m. TxBody m era -> Value era
    txBodyConsumedValue = getConsumedMaryValue pp lookupStakingDeposit lookupDRepDeposit utxo
    subTransactionsConsumedValue topTxBody =
      foldMap'
        (getConsumedValue pp lookupStakingDeposit lookupDRepDeposit utxo . view bodyTxL)
        (topTxBody ^. subTransactionsTxBodyL)
```

`lookupStakingDeposit` is a pure function over the **pre-transaction** `CertState`. It is called independently for every sub-transaction body. There is no stateful accumulator that marks a credential's deposit as "already claimed" after the first sub-transaction resolves it. Consequently, `foldMap'` over `subTransactionsTxBodyL` can count the same deposit `D` multiple times — once per sub-transaction that carries an `UnRegDepositTxCert` for the same credential.

**Contrast with the single-body case**

Within a single transaction body, `shelleyTotalRefundsTxCerts` correctly prevents double-counting by threading a `regCreds` set through a left fold: [2](#0-1) 

That stateful guard is absent across sub-transaction boundaries in `getConsumedDijkstraValue`.

**Developer acknowledgement**

The developers have already written — and then disabled — a test that directly exercises this scenario: [3](#0-2) 

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
  ...
  subTx1 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
  subTx2 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
  tx = mkBasicTx mkBasicTxBody & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
  submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
```

The `xit` prefix disables the test; the `error "TODO: predicate failure not yet implemented"` placeholder confirms that **no predicate failure has been wired up** to reject this case. The preservation-of-value check therefore does not guard against it.

**Produced-value side**

`dijkstraProducedValue` sums outputs of the top-level body and all sub-transactions: [4](#0-3) 

Because `consumed = produced` is the only balance gate in the UTXO rule, an attacker who inflates `consumed` by `D` can balance the equation by placing an extra `D` in the outputs of the second sub-transaction — outputs that are added to the UTxO set — while the deposit pot is only reduced by `D` once.

---

### Impact Explanation

The preservation-of-value invariant (`consumed = produced`) is the ledger's primary accounting guard. When two sub-transactions both resolve the same deposit refund `D`, `consumed` is inflated by `D`. To satisfy the check, `produced` must also be inflated by `D`, which is achieved by placing `D` lovelace in the outputs of the second sub-transaction. The deposit pot is debited only once (by the first successful deregistration), so the net effect is the creation of `D` lovelace from nothing — a direct violation of ADA conservation. This maps to the **Critical** impact class: *direct creation of ADA through an invalid ledger state transition*.

A secondary guard exists: the DELEG rule processes certificates sequentially and would reject the second `UnRegDepositTxCert` for an already-deregistered credential. However, the primary preservation-of-value check is structurally broken, and the developers have explicitly noted that the dedicated predicate failure for this case is not yet implemented. Any future refactoring that reorders UTXO vs. CERT validation, or any path that bypasses the DELEG rule for sub-transactions, would make the exploit directly reachable.

---

### Likelihood Explanation

The Dijkstra era is the newest era (protocol version 12) and sub-transactions are a novel feature. An unprivileged transaction author can craft a top-level transaction containing two sub-transactions that each carry `UnRegDepositTxCert` for the same staking credential. No privileged access, governance majority, or key compromise is required. The attacker controls the transaction body entirely. Likelihood is **medium** given that Dijkstra is not yet on mainnet, but the attack surface is fully attacker-controlled once it is.

---

### Recommendation

`getConsumedDijkstraValue` must thread a stateful accumulator across sub-transaction boundaries, analogous to the `regCreds` set in `shelleyTotalRefundsTxCerts`. Concretely:

1. Before iterating over sub-transactions, initialise a mutable set of "already-claimed staking credentials" and "already-claimed DRep credentials".
2. When computing the refund contribution of each sub-transaction, consult and update this set so that a credential whose deposit was already counted in a prior sub-transaction (or in the top-level body) contributes zero to subsequent sub-transactions.
3. Alternatively, add an explicit UTXO predicate failure (analogous to `BabbageNonDisjointRefInputs`) that rejects any batch transaction where the same credential appears in `UnRegDepositTxCert` across more than one sub-transaction or between the top-level body and a sub-transaction.

The disabled test at `eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs:53` should be re-enabled once the predicate failure is implemented.

---

### Proof of Concept

The existing disabled test is the proof of concept: [3](#0-2) 

1. Register `stakingCred`, paying deposit `D` into the deposit pot.
2. Construct `subTx1` with `UnRegDepositTxCert stakingCred D` and `subTx2` with the same certificate.
3. Wrap both in a top-level Dijkstra transaction via `subTransactionsTxBodyL`.
4. `getConsumedDijkstraValue` calls `lookupStakingDeposit stakingCred` twice (once per sub-transaction), returning `D` both times, so `consumed` includes `2D` from refunds.
5. To satisfy `consumed = produced`, place `D` extra lovelace in the outputs of `subTx2`.
6. The deposit pot is reduced by `D` (first deregistration succeeds); the UTxO gains `D` extra lovelace — net creation of `D` ADA.

The `error "TODO: predicate failure not yet implemented"` in the test confirms that no ledger rule currently rejects step 4.

shelleyTotalRefundsTxCerts pp lookupDeposit = snd . F.foldl' accum (mempty, Coin 0)
  where
    keyDeposit = pp ^. ppKeyDepositL
    accum (!regCreds, !totalRefunds) cert =
      case lookupRegStakeTxCert cert of
        Just k ->
          -- Need to track new delegations in case that the same key is later deregistered in
          -- the same transaction.
          (Set.insert k regCreds, totalRefunds)
        Nothing ->
          case lookupUnRegStakeTxCert cert of
            Just cred
              -- We first check if there was already a registration certificate in this
              -- transaction.
              | Set.member cred regCreds -> (Set.delete cred regCreds, totalRefunds <+> keyDeposit)
              -- Check for the deposit left during registration in some previous
              -- transaction. This de-registration check will be matched first, despite being
              -- the last case to match, because registration is not possible without
              -- de-registration.
              | Just deposit <- lookupDeposit cred -> (regCreds, totalRefunds <+> deposit)
            _ -> (regCreds, totalRefunds)
