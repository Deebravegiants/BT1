Audit Report

## Title
Zero rsETH Minted for Non-Zero Deposit Due to Integer Division Truncation - (File: contracts/LRTDepositPool.sol)

## Summary
`getRsETHAmountToMint` performs integer division that truncates to zero when `amount * assetPrice < rsETHPrice`. Because `_beforeDeposit` has no explicit zero-mint guard and the slippage check `rsethAmountToMint < minRSETHAmountExpected` is trivially bypassed by passing `minRSETHAmountExpected = 0`, a deposit of sufficiently small size succeeds: the asset is transferred in and zero rsETH is minted to the depositor.

## Finding Description

**Root cause — `getRsETHAmountToMint` (L520):**

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

When `amount * assetPrice < rsETHPrice`, Solidity integer division truncates to `0`.

**`_beforeDeposit` has no zero-mint guard (L657–669):**

```solidity
if (depositAmount == 0 || depositAmount < minAmountToDeposit) {
    revert InvalidAmountToDeposit();
}
// ...
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

`minAmountToDeposit` is a storage variable that defaults to `0` (no initialization in `initialize()`), so any `depositAmount >= 1` passes the first check. If `rsethAmountToMint == 0` and `minRSETHAmountExpected == 0`, the condition `0 < 0` is false — no revert.

**`depositAsset` then transfers the asset and mints zero (L114–115):**

```solidity
IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
_mintRsETH(rsethAmountToMint);  // rsethAmountToMint == 0
```

**`RSETH.mint(address, 0)` does not revert (L229–240, L42–56):**

The `checkDailyMintLimit` modifier with `amount = 0` does not revert (the condition `currentPeriodMintedAmount + 0 > maxMintAmountPerDay` is false under normal operation), and OpenZeppelin's `_mint(to, 0)` is a no-op. The call succeeds, minting nothing.

## Impact Explanation

The depositor's asset is transferred into the pool but zero rsETH is minted. The depositor receives no share of the protocol's yield-bearing collateral. The protocol retains the collateral without issuing a corresponding claim token. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation

- `rsETHPrice` starts at `1e18` and monotonically increases as rewards accrue (`_updateRsETHPrice` in `LRTOracle.sol`). After any meaningful protocol operation, `rsETHPrice > 1e18` is the normal state.
- With `rsETHPrice = 1.05e18` and `assetPrice = 1e18`, depositing `1 wei` gives `(1 * 1e18) / 1.05e18 = 0`.
- `minAmountToDeposit` defaults to `0` and may not be configured by operators.
- Any caller who passes `minRSETHAmountExpected = 0` (the default for automated integrations or naive users) is vulnerable.
- The oracle sanity check in `updatePriceOracleForValidated` bounds asset prices to `[1e16, 1e19]` but does not prevent truncation — it only constrains the ratio.
- The attack is repeatable by any unprivileged external caller via the public `depositAsset` function.

## Recommendation

Add an explicit zero-mint guard in `_beforeDeposit`:

```solidity
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

if (rsethAmountToMint == 0) {
    revert MinimumAmountToReceiveNotMet();
}

if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

Alternatively, enforce a non-zero `minAmountToDeposit` large enough to guarantee at least 1 wei of rsETH is minted at any realistic `rsETHPrice`.
