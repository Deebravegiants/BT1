### Title
No Completed-Transaction Registry Enables Cross-Chain Replay of Foreign-Tx Attestations — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction` endpoint produces a threshold-signed attestation that a specific foreign-chain transaction occurred. After the attestation is delivered and the pending-request map entry is removed, the contract retains no record that the transaction was ever processed. Any unprivileged NEAR account can immediately re-submit the identical `tx_id`, causing the MPC network to produce a second (and third, etc.) valid signed attestation for the same foreign event. In a bridge flow — the primary stated use case — this enables a depositor to obtain multiple independent MPC-signed proofs for a single on-chain deposit and redeem bridge funds more than once.

---

### Finding Description

`verify_foreign_transaction` accepts a `VerifyForeignTransactionRequestArgs` containing `{request: ForeignChainRpcRequest, domain_id, payload_version}`, where `ForeignChainRpcRequest` carries the foreign-chain transaction identifier (`BitcoinTxId`, `EvmTxId`, etc.). [1](#0-0) 

The request is stored in `pending_verify_foreign_tx_requests` as a temporary yield-resume queue. When `respond_verify_foreign_tx` is called by an attested MPC node, `resolve_yields_for` drains the queue and **removes the map entry entirely**: [2](#0-1) [3](#0-2) 

After removal, the contract holds **no persistent record** that this `tx_id` was ever attested. The `args_into_verify_foreign_tx_request` conversion function performs no uniqueness check: [4](#0-3) 

The `VerifyForeignTransactionRequest` struct used as the map key contains only `{request, domain_id, payload_version}` — no caller identity, no nonce, no "already-finalized" flag: [5](#0-4) 

Because the pending map is purely ephemeral, re-submitting the same `tx_id` after the first attestation completes is indistinguishable from a fresh first-time request. The MPC network will re-verify the foreign transaction and produce a new, independently valid `VerifyForeignTransactionResponse{payload_hash, signature}`.

---

### Impact Explanation

The design document explicitly identifies the primary use case as the **Omnibridge inbound flow**: a bridge contract on NEAR mints tokens only after receiving an MPC-signed attestation that a foreign-chain deposit finalized. [6](#0-5) 

With no completed-transaction registry, an attacker who deposited once on the foreign chain can:

1. Call `verify_foreign_transaction(tx_id=X)` → receive `response_1{payload_hash_H, sig_1}`.
2. Immediately call `verify_foreign_transaction(tx_id=X)` again → receive `response_2{payload_hash_H, sig_2}` (same hash, fresh signature).
3. Submit both attestations to the bridge contract in separate transactions.

Any bridge contract that does not independently track used `payload_hash` values will process both claims, minting double the bridged tokens. Because the `payload_hash` is deterministically derived from `(request, observed_values)`, both responses carry the **same** `payload_hash`, so a bridge contract that only checks "is this hash valid?" (rather than "has this hash been used?") is fully exploitable. The MPC contract's design places no obligation on callers to implement such tracking, and the contract README describes the feature as a drop-in replacement for a trusted relayer — implying the MPC attestation is itself the security boundary.

Impact classification: **High** — cross-chain replay that causes double-spend conditions in bridge execution flows.

---

### Likelihood Explanation

The entry path is the public, permissionless `verify_foreign_transaction` method, gated only by a 1 yoctonear deposit: [7](#0-6) 

No privileged role, no threshold collusion, and no leaked key material is required. Any NEAR account that has previously obtained a successful attestation can replay it an arbitrary number of times. The only cost is 1 yoctonear per submission plus gas. Likelihood: **High**.

---

### Recommendation

Introduce a persistent `LookupSet<VerifyForeignTransactionRequest>` (or equivalent keyed on `(chain, tx_id, domain_id)`) that records every successfully attested request. In `verify_foreign_transaction`, reject any submission whose key is already present in this set before enqueuing the yield. Populate the set inside `return_verify_foreign_tx_and_clean_state_on_success` on the success branch, so only confirmed attestations are recorded. This mirrors the invariant the original report recommends: track the "loss event" (here, the completed attestation) so it cannot be re-applied.

---

### Proof of Concept

```
// Round 1
alice.call("verify_foreign_transaction", {
    request: { Bitcoin: { tx_id: X, confirmations: 1, extractors: [BlockHash] } },
    domain_id: 0,
    payload_version: 1
}, deposit=1yN)
// MPC nodes respond → alice receives response_1{payload_hash: H, sig: S1}
// pending_verify_foreign_tx_requests entry for tx_id=X is REMOVED

// Round 2 — identical call, no rejection
alice.call("verify_foreign_transaction", {
    request: { Bitcoin: { tx_id: X, confirmations: 1, extractors: [BlockHash] } },
    domain_id: 0,
    payload_version: 1
}, deposit=1yN)
// MPC nodes respond again → alice receives response_2{payload_hash: H, sig: S2}

// alice now holds two independently valid MPC-signed attestations for the same deposit X
// She submits both to the bridge contract → double-spend
```

The contract's own test confirms that after `respond_verify_foreign_tx` the map entry is absent and a fresh submission would be accepted as a new request: [8](#0-7)

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
