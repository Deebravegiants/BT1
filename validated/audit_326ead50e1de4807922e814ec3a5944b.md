Audit Report

## Title
Missing `amount` in ETH Deposit Limit Check Allows Deposits Beyond Configured Cap - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric comparison: for LST assets it correctly tests `totalAssetDeposits + amount > depositLimit`, but for ETH it omits `amount` and only tests `totalAssetDeposits > depositLimit`. As a result, any ETH deposit is accepted as long as the running total has not already crossed the cap, regardless of how large `msg.value` is, allowing total ETH holdings to be pushed arbitrarily above the configured limit in a single transaction.

## Finding Description
The sole deposit-cap guard is `_checkIfDepositAmountExceedesCurrentLimit` (lines 676–682). The ETH branch at line 679 evaluates `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)`, which returns `true` only when the cap has **already** been exceeded before the deposit. The incoming `amount` is never added, so the check is always `false` whenever `totalAssetDeposits <= depositLimit`, no matter how large the deposit is. [1](#0-0) 

The LST branch on line 681 correctly includes `amount` in the sum, making the asymmetry clear. `_beforeDeposit` (lines 661–663) reverts on `true` from this function, but because the ETH branch never returns `true` for an oversized deposit, the revert never fires. [2](#0-1) 

`depositETH` is fully public and guarded only by `whenNotPaused` and `nonReentrant`, requiring no special role or precondition. [3](#0-2) 

## Impact Explanation
The ETH deposit cap — a protocol-level safety control — is not enforced for any individual ETH deposit. An unprivileged depositor can mint rsETH for an arbitrarily large ETH amount in one call, pushing total ETH holdings well beyond the configured limit. This matches the allowed impact: **Low — contract fails to deliver promised returns (deposit limit not enforced)**, as the protocol's own invariant is violated without direct loss of user funds.

## Likelihood Explanation
The entry path is a single public `payable` function callable by any Ethereum address. No special role, front-running, or external dependency is required. Any depositor who sends ETH while `totalAssetDeposits <= depositLimit` bypasses the cap automatically. The condition is trivially satisfied at any point before the cap is organically breached.

## Recommendation
Add `amount` to the ETH branch, mirroring the LST branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
Assume `depositLimitByAsset[ETH] = 1000 ether` and `getTotalAssetDeposits(ETH) = 999 ether`.

1. Attacker calls `depositETH{value: 500 ether}(0, "")`.
2. `_checkIfDepositAmountExceedesCurrentLimit(ETH, 500 ether)` evaluates `999 ether > 1000 ether` → `false`.
3. No revert; `_mintRsETH` mints rsETH for 500 ETH.
4. `getTotalAssetDeposits(ETH)` is now `1499 ether`, 49.9% above the configured cap.
5. For comparison, a 500-token LST deposit under the same conditions evaluates `999 + 500 > 1000` → `true` and reverts with `MaximumDepositLimitReached`.

**Foundry test plan:** Deploy `LRTDepositPool` on a local fork, set `depositLimitByAsset[ETH] = 1000 ether`, seed the pool to `999 ether` total deposits, then call `depositETH{value: 500 ether}` from an unprivileged address and assert the call succeeds and `getTotalAssetDeposits(ETH)` returns `1499 ether`.

        uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
