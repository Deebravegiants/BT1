Audit Report

## Title
`instantWithdrawal` Ignores `assetsCommitted`, Allowing Drain of Vault Assets Reserved for Queued Withdrawal Requests - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`instantWithdrawal` checks only `unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)`, which is computed as `vaultBalance - queuedWithdrawalsBuffer[asset]`. Because `queuedWithdrawalsBuffer` defaults to zero and is never automatically updated when `initiateWithdrawal` increments `assetsCommitted[asset]`, any rsETH holder can drain the entire `LRTUnstakingVault` balance via `instantWithdrawal` even when those assets are already committed to pending queued withdrawal requests. This causes `unlockQueue` to find zero vault balance and be unable to unlock any queued requests, temporarily freezing the rsETH held in the `WithdrawalManager` on behalf of queued-withdrawal users.

## Finding Description
When `initiateWithdrawal` is called, it increments `assetsCommitted[asset]` and enforces that the new commitment does not exceed `getAvailableAssetAmount`, which is `getTotalAssetDeposits(asset) - assetsCommitted[asset]`. `getTotalAssetDeposits` includes `assetLyingUnstakingVault`, so vault assets are counted as backing the commitment.

However, `instantWithdrawal` performs a completely separate availability check at line 231:

```solidity
if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
    revert CantInstantWithdrawMoreThanAvailable();
}
```

`getAssetsAvailableForInstantWithdrawal` in `LRTUnstakingVault.sol` (L235–237) returns:

```solidity
uint256 vaultBalance = balanceOf(asset);
uint256 reservedBuffer = queuedWithdrawalsBuffer[asset];
availableAmount = reservedBuffer >= vaultBalance ? 0 : vaultBalance - reservedBuffer;
```

`queuedWithdrawalsBuffer` is set exclusively by `onlyLRTOperator` via `setQueuedWithdrawalsBuffer` (L199–209 in `LRTUnstakingVault.sol`) and is **never automatically updated** when `initiateWithdrawal` is called. Its default value is `0`. The `instantWithdrawal` function never reads `assetsCommitted`, so it has no knowledge of how much of the vault balance is already spoken for by queued withdrawal requests.

With `queuedWithdrawalsBuffer == 0` (the default), the entire vault balance is reported as available for instant withdrawal, even if 100% of it is committed via `assetsCommitted`.

After the drain, `unlockQueue` calls `_createUnlockParams` which reads `unstakingVault.balanceOf(asset)` as `totalAvailableAssets` (L849). With vault balance at zero, `params.totalAvailableAssets == 0` triggers the revert at L297:

```solidity
if (params.totalAvailableAssets == 0) revert AmountMustBeGreaterThanZero();
```

The rsETH held in the `WithdrawalManager` on behalf of queued-withdrawal users is frozen until the operator manually replenishes the vault.

## Impact Explanation
**Temporary freezing of funds — Medium.** Users who called `initiateWithdrawal` have their rsETH locked in the `WithdrawalManager` and cannot complete their withdrawal until the operator replenishes the `LRTUnstakingVault` (e.g., by completing an EigenLayer withdrawal cycle, which carries a multi-day delay). This matches the allowed impact class "Temporary freezing of funds."

## Likelihood Explanation
- Instant withdrawals must be enabled by the LRT Manager (`setInstantWithdrawalEnabled`), which is a routine operational action.
- `queuedWithdrawalsBuffer` is `0` by default and requires an explicit operator call to set; many deployments will leave it at zero.
- No special privilege is required for the attacker — any rsETH holder can call `instantWithdrawal`.
- The attacker does not lose funds; they simply redeem rsETH at the current rate.
- The attack is repeatable each time the vault is replenished.

## Recommendation
1. **Automatic buffer accounting**: When `initiateWithdrawal` increments `assetsCommitted[asset]`, also increase `queuedWithdrawalsBuffer[asset]` in the unstaking vault by the same amount; decrease it when the request is unlocked (`_unlockWithdrawalRequests` decrements `assetsCommitted`) or cancelled.
2. **Cross-check in `instantWithdrawal`**: Before redeeming from the vault, verify that `assetAmountUnlocked ≤ unstakingVault.balanceOf(asset) - assetsCommitted[asset]`, analogous to how `getAvailableAssetAmount` protects `initiateWithdrawal`.

## Proof of Concept
```
State: LRTUnstakingVault holds 100 ETH, deposit pool holds 0 ETH.
       queuedWithdrawalsBuffer[ETH] = 0 (default).
       Instant withdrawals enabled for ETH.

1. Alice calls initiateWithdrawal(ETH, rsETHAmount_A):
   - rsETH transferred to WithdrawalManager.
   - assetsCommitted[ETH] += 100  →  getAvailableAssetAmount(ETH) = 0.

2. Bob calls instantWithdrawal(ETH, rsETHAmount_B):
   - getAssetsAvailableForInstantWithdrawal(ETH) = 100 - 0 = 100  ✓ (passes)
   - assetsCommitted[ETH] is never consulted.
   - Bob burns rsETH, receives 100 ETH from vault.
   - LRTUnstakingVault balance = 0.

3. Operator calls unlockQueue(ETH, ...):
   - totalAvailableAssets = unstakingVault.balanceOf(ETH) = 0.
   - Reverts with AmountMustBeGreaterThanZero (L297).
   - Alice's rsETH remains locked in the WithdrawalManager indefinitely
     until the operator replenishes the vault.
```

**Foundry test plan**: Deploy `LRTWithdrawalManager` and `LRTUnstakingVault` with mocked config; seed vault with 100 ETH; call `initiateWithdrawal` as Alice; call `instantWithdrawal` as Bob for the full vault balance; assert vault balance is 0; assert `unlockQueue` reverts with `AmountMustBeGreaterThanZero`; assert Alice's withdrawal request remains locked.
