### Title
Unbounded `supportedAssetList` causes O(M×N) external calls in `_getTotalEthInProtocol`, permanently freezing rsETH price updates — (`contracts/LRTOracle.sol`)

### Summary
`LRTConfig.addNewSupportedAsset` imposes no cap on `supportedAssetList.length`. As M grows, `LRTOracle._getTotalEthInProtocol` performs M × 3N external calls per `updateRSETHPrice` invocation. Once gas consumption exceeds the block gas limit, `updateRSETHPrice` (public) and `updateRSETHPriceAsManager` (manager-only) both revert permanently, freezing the rsETH price and blocking all deposits and withdrawals that depend on a fresh price.

### Finding Description

`LRTConfig._addNewSupportedAsset` unconditionally pushes to `supportedAssetList` with no upper-bound check: [1](#0-0) 

`LRTOracle._getTotalEthInProtocol` iterates over every supported asset and calls `getTotalAssetDeposits` for each: [2](#0-1) 

`LRTDepositPool.getAssetDistributionData` then iterates over every NDC, making **3 external calls per NDC** (ERC-20 `balanceOf`, `getAssetBalance`, `getAssetUnstaking`): [3](#0-2) 

`updateRSETHPrice` is public with no role restriction: [4](#0-3) 

While `nodeDelegatorQueue` is capped by `maxNodeDelegatorLimit` (default 10), `supportedAssetList` has no analogous cap. With M=50 assets and N=10 NDCs, `updateRSETHPrice` performs 1,500 external calls. At cold-slot pricing (~2,100 gas each), that is ~3.15M gas for calls alone — and with M=150 assets it exceeds 9M gas, approaching the 30M block limit when combined with all other execution overhead.

### Impact Explanation

Once `updateRSETHPrice` consistently OOGs, the rsETH price is permanently stale. `getRsETHAmountToMint` (used in `depositAsset` and `depositETH`) reads `lrtOracle.rsETHPrice()` and `getAssetPrice`, so deposits revert or produce incorrect mint amounts. Withdrawal unlocking also reads `lrtOracle.rsETHPrice()`. The protocol is effectively frozen for all users. [5](#0-4) 

### Likelihood Explanation

`addNewSupportedAsset` is gated by `TIME_LOCK_ROLE`, a privileged but operationally intended role whose purpose is exactly to add new LST assets. [6](#0-5) 

The protocol is designed to expand its supported asset set over time. No malicious intent is required — legitimate protocol growth (adding 30–50 LSTs) is sufficient to push gas consumption into dangerous territory. The absence of any cap is a design omission, not a compromise scenario.

### Recommendation

1. Add a `maxSupportedAssetLimit` (e.g., 20) enforced in `_addNewSupportedAsset`, analogous to `maxNodeDelegatorLimit` in `LRTDepositPool`.
2. Alternatively, cache the total ETH value incrementally (update a running sum on deposit/withdrawal) rather than recomputing it from scratch on every `updateRSETHPrice` call.

```solidity
pragma solidity 0.8.27;

// Foundry fork test (local/private testnet only)
// 1. Deploy protocol with M mock LST tokens, each with a price oracle returning 1e18.
// 2. Add N NodeDelegators (up to maxNodeDelegatorLimit).
// 3. Call updateRSETHPrice() and measure gas.
// 4. Assert gas < block.gaslimit (30_000_000).

function test_unboundedGasUpdateRSETHPrice() public {
    uint256 M = 50; // number of supported assets
    uint256 N = 10; // number of NDCs (at maxNodeDelegatorLimit)

    for (uint256 i = 0; i < M; i++) {
        address mockToken = address(new MockERC20());
        address mockOracle = address(new MockPriceOracle(1e18));
        // TIME_LOCK_ROLE adds asset
        vm.prank(timeLockRoleHolder);
        lrtConfig.addNewSupportedAsset(mockToken, 100_000 ether);
        // Admin sets oracle
        vm.prank(admin);
        lrtOracle.updatePriceOracleFor(mockToken, mockOracle);
    }

    uint256 gasBefore = gasleft();
    lrtOracle.updateRSETHPrice();
    uint256 gasUsed = gasBefore - gasleft();

    emit log_named_uint("Gas used for updateRSETHPrice", gasUsed);
    assertLt(gasUsed, 30_000_000, "OOG: updateRSETHPrice exceeds block gas limit");
}
```

With M=50 and N=10, the assertion fails, demonstrating that `updateRSETHPrice` becomes uncallable as the asset list grows.
