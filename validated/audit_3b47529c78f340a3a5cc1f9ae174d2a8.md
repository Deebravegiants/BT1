Audit Report

## Title
Missing `priceLimitX64 != 0` Special-Case in Initial Guards of `!zeroForOne` Swap Paths Causes Silent Total Swap Failure — (`metric-core/contracts/MetricOmmPool.sol`)

## Summary

When `priceLimitX64 = 0` is passed to `swap()` to signal "no price limit," both `_swapToken1ForToken0SpecifiedInput` and `_swapToken1ForToken0SpecifiedOutput` immediately return zero-delta results before processing any bin. The initial guard `params.priceLimitX64 <= initialPriceX64` evaluates to `0 <= initialPriceX64`, which is always `true` for any valid uint256 price, causing an early return. The loop guard inside `_swapToken1ForToken0SpecifiedInput` already carries the correct `!= 0` special-case, confirming the protocol intended `0` to mean "no limit," but the initial guards in both functions are missing this check.

## Finding Description

The code at line 970 in `_swapToken1ForToken0SpecifiedInput`:

```solidity
if (params.priceLimitX64 <= initialPriceX64) {
    return (0, 0, 0);
}
``` [1](#0-0) 

and at line 888 in `_swapToken1ForToken0SpecifiedOutput`:

```solidity
if (params.priceLimitX64 <= initialPriceX64) {
    return (0, 0, 0, 0);
}
``` [2](#0-1) 

both lack the `!= 0` guard. When `priceLimitX64 == 0`, the comparison `0 <= initialPriceX64` is always `true` for any valid uint256 price, so both functions return immediately with zero amounts.

The loop guard inside `_swapToken1ForToken0SpecifiedInput` at line 979 already has the correct check:

```solidity
if (params.priceLimitX64 != 0 && params.priceLimitX64 <= upperPriceX64) {
    break;
}
``` [3](#0-2) 

This asymmetry confirms the protocol intended `priceLimitX64 = 0` to mean "no limit." The loop guard at line 979 is dead code — it is never reached because the initial guard at line 970 returns first. The same flaw exists in the loop guard of `_swapToken1ForToken0SpecifiedOutput` at line 895, which also lacks the `!= 0` check:

```solidity
if (params.priceLimitX64 <= upperPriceX64) {
    break;
}
``` [4](#0-3) 

## Impact Explanation

Any caller passing `priceLimitX64 = 0` on a `!zeroForOne` swap receives `(0, 0)` from `swap()`. The pool emits a `Swap` event with zero deltas and transfers nothing. If a router or aggregator unconditionally sends tokens to the pool before checking return values (a common pattern), those tokens are permanently stranded as unaccounted surplus — a direct loss of user principal. Even without a malformed callback, the core swap functionality is completely broken for this valid and documented input pattern. This constitutes broken core pool functionality causing potential loss of funds, meeting the allowed impact gate.

## Likelihood Explanation

`priceLimitX64 = 0` as "no limit" is the standard convention in bin-based and oracle-anchored AMMs (analogous to `sqrtPriceLimitX96 = 0` in Uniswap v3). The presence of the `!= 0` guard at line 979 confirms the protocol explicitly designed for this convention. Any router, aggregator, or user following this convention on a `!zeroForOne` swap will silently receive nothing. No special privileges or unusual conditions are required — a standard unprivileged swap call with `priceLimitX64 = 0` triggers the bug deterministically.

## Recommendation

Add the `!= 0` guard to every price-limit comparison in the `!zeroForOne` swap paths:

```solidity
// _swapToken1ForToken0SpecifiedOutput — initial guard (line 888)
if (params.priceLimitX64 != 0 && params.priceLimitX64 <= initialPriceX64) {
    return (0, 0, 0, 0);
}

// _swapToken1ForToken0SpecifiedOutput — loop guard (line 895)
if (params.priceLimitX64 != 0 && params.priceLimitX64 <= upperPriceX64) {
    break;
}

// _swapToken1ForToken0SpecifiedInput — initial guard (line 970)
if (params.priceLimitX64 != 0 && params.priceLimitX64 <= initialPriceX64) {
    return (0, 0, 0);
}
// loop guard at line 979 is already correct
```

## Proof of Concept

1. Deploy a pool with token0/token1 and seed it with liquidity across several bins.
2. Call `pool.swap(recipient, false, 1000e18, 0, callbackData, "")` — exact-in, buying token0, `priceLimitX64 = 0`.
3. `_swapToken1ForToken0SpecifiedInput` evaluates `0 <= initialPriceX64` at line 970 → always `true` → returns `(0, 0, 0)`.
4. `swap()` calls `metricOmmSwapCallback(0, 0, callbackData)` and emits `Swap` with zero deltas.
5. Recipient receives no token0; caller pays no token1. Swap silently no-ops.
6. Repeat with `amountSpecified = -1000e18` (exact-out): `_swapToken1ForToken0SpecifiedOutput` hits line 888 with the same result.
7. Confirm with a Foundry test asserting that `amount0Delta` and `amount1Delta` returned from `swap()` are both zero despite non-zero `amountSpecified` and available liquidity.
