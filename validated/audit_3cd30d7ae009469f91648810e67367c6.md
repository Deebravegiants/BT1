### Title
Consensus Proposal Validation Path Bypasses Gateway `nonce_data_availability_mode` / `fee_data_availability_mode` / `paymaster_data` / `account_deployment_data` Checks, Allowing a Malicious Proposer to Inject OS-Invalid Transactions — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

The gateway's `StatelessTransactionValidator::validate()` enforces four OS-level invariants on every user-submitted transaction: `nonce_data_availability_mode = L1`, `fee_data_availability_mode = L1`, `paymaster_data` is empty, and `account_deployment_data` is empty. These checks exist explicitly because the Starknet OS enforces the same invariants and will fail any transaction that violates them. The consensus proposal-validation path (`handle_proposal_part` → `convert_consensus_tx_to_internal_consensus_tx`) performs no equivalent checks. A malicious proposer can therefore stream `ConsensusTransaction` objects that violate these invariants directly into the batcher. The blockifier's `perform_pre_validation_stage` does not check DA mode or those fields, so it executes the transactions and produces a state diff. Honest validators accept the proposal (commitment matches), the block is committed, and the OS later fails those transactions during proving — producing a divergence between the sequencer's committed state and the provable state.

---

### Finding Description

**Gateway path (enforces the invariants)**

`StatelessTransactionValidator::validate()` is called for every transaction that enters via `Gateway::add_tx`:

```
validate_contract_address
validate_empty_account_deployment_data   ← OS invariant
validate_empty_paymaster_data            ← OS invariant
validate_resource_bounds
validate_tx_size
validate_nonce_data_availability_mode    ← OS invariant (must be L1)
validate_fee_data_availability_mode      ← OS invariant (must be L1)
validate_client_side_proving_allowed
validate_proof_facts_and_proof_consistency
```

The comments in the source are explicit:

> "The Starknet OS enforces that the nonce data availability mode is L1. We add this validation here in the gateway to prevent transactions from failing the OS."
> "The Starknet OS enforces that the deployer data is empty. We add this validation here in the gateway to prevent transactions from failing the OS."

**Consensus path (missing the invariants)**

When a validator processes a received proposal, `handle_proposal_part` handles each `ProposalPart::Transactions` batch:

```rust
// validate_proposal.rs ~line 602-604
let conversion_results =
    futures::future::join_all(txs.into_iter().map(|tx| {
        transaction_converter.convert_consensus_tx_to_internal_consensus_tx(tx)
    }))
```

`convert_consensus_tx_to_internal_consensus_tx` delegates to `convert_rpc_tx_to_internal`, which:
- For `Invoke`: extracts `proof_data` if `proof_facts` is non-empty, then converts — **no DA-mode check, no paymaster check, no account_deployment_data check**.
- For `Declare`: validates `compiled_class_hash` only.
- For `DeployAccount`: computes `contract_address` only.

The converted transactions are forwarded to the batcher via `send_txs_for_proposal`. The batcher executes them through the blockifier. The blockifier's `perform_pre_validation_stage` checks nonce, fee bounds (if `charge_fee`), and `validate_proof_facts` — but **does not check DA mode, paymaster data, or account_deployment_data**:

```rust
// account_transaction.rs ~line 355-372
pub fn perform_pre_validation_stage<S: State + StateReader>(
    &self,
    state: &mut S,
    tx_context: &TransactionContext,
) -> TransactionPreValidationResult<()> {
    let tx_info = &tx_context.tx_info;
    Self::handle_nonce(state, tx_info, self.execution_flags.strict_nonce_check)?;
    if self.execution_flags.charge_fee {
        self.check_fee_bounds(tx_context)?;
        verify_can_pay_committed_bounds(state, tx_context).map_err(Box::new)?;
    }
    self.validate_proof_facts(&tx_context.block_context, state)?;
    Ok(())
}
```

The blockifier executes the transaction, produces a state diff, and the validator computes a `ProposalCommitment`. If the commitment matches the proposer's `ProposalFin.proposal_commitment`, the proposal is accepted and the block is committed. The OS then fails those transactions during proving (DA mode ≠ L1, non-empty paymaster/account_deployment_data), producing a state that diverges from the sequencer's committed state.

**Exact broken invariant**: Every transaction committed to a Starknet block must satisfy `nonce_data_availability_mode = L1`, `fee_data_availability_mode = L1`, `paymaster_data = []`, and `account_deployment_data = []`. The consensus path does not enforce this invariant, while the gateway path does.

---

### Impact Explanation

A malicious proposer (any node that wins a consensus round) can craft a `ConsensusTransaction::RpcTransaction(RpcInvokeTransactionV3 { nonce_data_availability_mode: L2, ... })` and stream it in a `ProposalPart::Transactions` batch. Honest validators will:
1. Convert it without error (no DA-mode check in `convert_rpc_tx_to_internal`).
2. Send it to the batcher, which executes it via the blockifier (no DA-mode check in `perform_pre_validation_stage`).
3. Accept the proposal when the commitment matches.
4. Commit the block.

The OS will then fail the transaction during proving. The sequencer's committed state includes the transaction's storage/nonce changes; the proven state does not. This is a wrong committed state / wrong receipt result — matching the "Critical. Wrong state, receipt, event … or revert result from blockifier/syscall/execution logic for accepted input" impact category.

---

### Likelihood Explanation

Any node that is elected proposer for a round can trigger this. In a BFT network with up to 1/3 Byzantine validators, a malicious node will be elected proposer in roughly 1/N rounds (where N is the validator-set size). No special privilege beyond being a consensus participant is required. The attack is deterministic and requires only constructing a well-formed `ConsensusTransaction` with a non-L1 DA mode field.

---

### Recommendation

Apply the same OS-invariant checks to transactions arriving through the consensus path. The cleanest fix is to add a lightweight validation step inside `handle_proposal_part` (or inside `convert_consensus_tx_to_internal_consensus_tx`) that rejects any `RpcTransaction` whose `nonce_data_availability_mode`, `fee_data_availability_mode`, `paymaster_data`, or `account_deployment_data` violates the OS invariants — mirroring `validate_nonce_data_availability_mode`, `validate_fee_data_availability_mode`, `validate_empty_paymaster_data`, and `validate_empty_account_deployment_data` from `StatelessTransactionValidator`.

---

### Proof of Concept

A malicious proposer node constructs and streams the following proposal:

```rust
// Malicious proposer builds a proposal containing an OS-invalid transaction.
let invalid_tx = ConsensusTransaction::RpcTransaction(
    RpcTransaction::Invoke(RpcInvokeTransaction::V3(RpcInvokeTransactionV3 {
        nonce_data_availability_mode: DataAvailabilityMode::L2, // violates OS invariant
        fee_data_availability_mode:   DataAvailabilityMode::L2, // violates OS invariant
        // ... all other fields valid ...
    }))
);
// Proposer streams: ProposalPart::Init, ProposalPart::Transactions([invalid_tx]), ProposalPart::Fin
```

On the validator side, `handle_proposal_part` receives `ProposalPart::Transactions`:

```rust
// validate_proposal.rs ~602
transaction_converter.convert_consensus_tx_to_internal_consensus_tx(invalid_tx)
// → calls convert_rpc_tx_to_internal → no DA-mode check → Ok(internal_tx)
```

The internal transaction is forwarded to the batcher. The blockifier's `perform_pre_validation_stage` runs — no DA-mode check — and executes the transaction. The validator computes the commitment, it matches the proposer's `ProposalFin`, and the block is committed. The OS subsequently fails the transaction during proving, producing a divergent state.

    /// gateway to prevent transactions from failing the OS.
    fn validate_empty_account_deployment_data(
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        let account_deployment_data = match tx {
            RpcTransaction::DeployAccount(_) => return Ok(()),
            RpcTransaction::Declare(RpcDeclareTransaction::V3(tx)) => &tx.account_deployment_data,
            RpcTransaction::Invoke(RpcInvokeTransaction::V3(tx)) => &tx.account_deployment_data,
        };

        if account_deployment_data.is_empty() {
            Ok(())
        } else {
            Err(StatelessTransactionValidatorError::NonEmptyField {
                field_name: "account_deployment_data".to_string(),
            })
        }
    }

    /// The Starknet OS enforces that the nonce data availability mode is L1. We add this validation
    /// here in the gateway to prevent transactions from failing the OS.
    fn validate_nonce_data_availability_mode(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        let expected_da_mode = DataAvailabilityMode::L1;
        let da_mode = *tx.nonce_data_availability_mode();
        if da_mode != expected_da_mode {
            return Err(StatelessTransactionValidatorError::InvalidDataAvailabilityMode {
                field_name: "nonce".to_string(),
