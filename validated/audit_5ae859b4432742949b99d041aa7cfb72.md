### Title
Unverified CKD Response for `AppPublicKey` Variant Allows Single Byzantine Participant to Deliver Forged Confidential Key - (File: `crates/contract/src/lib.rs`)

### Summary

In `respond_ckd`, when a pending CKD request uses the legacy `AppPublicKey` variant, the contract performs **zero cryptographic verification** of the submitted `CKDResponse`. A single malicious attested participant (below the signing threshold) can call `respond_ckd` with an arbitrary forged response, which the contract unconditionally accepts and delivers to the waiting user via the yield-resume mechanism — analogous to the external report's cached-balance bypass that allowed receiving two NFTs for the price of one.

### Finding Description

`respond_ckd` in `crates/contract/src/lib.rs` branches on the `CKDAppPublicKey` variant of the request:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO verification
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` cryptographically verifies the response against the BLS12-381 master public key. For the legacy `AppPublicKey` variant, the arm is an empty no-op — the response is accepted unconditionally. The contract then immediately calls `resolve_yields_for`, which removes the pending request from the map and resumes every queued yield with the (potentially forged) response bytes. [2](#0-1) 

The `AppPublicKey` variant is still actively supported and described as the "legacy" path in the contract README: [3](#0-2) 

By contrast, `respond` (for ECDSA/EdDSA signatures) always verifies the signature cryptographically before calling `resolve_yields_for`, so a single participant cannot forge a signature response: [4](#0-3) 

The asymmetry is the root cause: threshold-signed responses are verified on-chain; `AppPublicKey` CKD responses are not.

### Impact Explanation

A single Byzantine attested participant (strictly below the signing threshold) can:

1. Observe a pending `request_app_private_key` with `AppPublicKey` in `pending_ckd_requests`.
2. Call `respond_ckd` with a `CKDResponse` containing a BLS12-381 point they control.
3. The contract accepts the forged response without any check.
4. `resolve_yields_for` drains the queue and resumes the user's yield with the forged bytes.
5. The user receives a private key whose corresponding public key the attacker knows.

If the user subsequently uses that forged private key to sign foreign-chain transactions or decrypt data, the attacker — who knows the key — can co-sign transactions, drain any funds sent to the derived address, or decrypt the user's confidential data. This is unauthorized confidential key derivation output delivered without the required threshold-participant authorization.

Allowed impact match: **Critical — confidential key derivation output without required participant authorization.**

### Likelihood Explanation

- Only **one** attested participant needs to be Byzantine; no threshold collusion is required.
- The attacker's entry path is the public `respond_ckd` method, callable by any attested participant.
- The `AppPublicKey` variant is still in production use (marked "legacy," not removed).
- The attacker simply races the legitimate MPC response; since `resolve_yields_for` removes the queue entry on first call, whichever `respond_ckd` call lands first wins. [5](#0-4) 

### Recommendation

1. **Deprecate `AppPublicKey` immediately** and reject new `request_app_private_key` submissions that use it, requiring callers to migrate to `AppPublicKeyPV` (which is publicly verifiable on-chain).
2. Until deprecation is complete, **require threshold-many `respond_ckd` votes** for `AppPublicKey` requests before resuming the yield — mirroring how threshold signatures prevent a single node from forging a response.
3. At minimum, add prominent documentation warning that `AppPublicKey` CKD outputs carry no on-chain integrity guarantee and that callers **must** verify the response themselves before use.

### Proof of Concept

```
1. Alice calls request_app_private_key({app_public_key: AppPublicKey(pk), ...})
   → contract stores yield Y in pending_ckd_requests[R]

2. Malicious attested participant Eve calls:
   respond_ckd(request=R, response=CKDResponse{big_y: Eve_controlled_point, big_c: ...})

3. respond_ckd reaches the match arm:
       AppPublicKey(_) => {}   // no check
   then calls resolve_yields_for → removes R, resumes Y with Eve's bytes

4. Alice's yield fires: she receives Eve's forged CKDResponse as if it were
   the legitimate MPC output.

5. Alice derives a private key from Eve's response and uses it to sign a
   Bitcoin transaction, sending funds to a derived address.
   Eve, knowing the private key, sweeps those funds.
```

The legitimate MPC `respond_ckd` call (if it arrives later) returns `RequestNotFound` because `resolve_yields_for` already removed the queue entry, so the forgery is undetectable at the contract level. [6](#0-5)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
