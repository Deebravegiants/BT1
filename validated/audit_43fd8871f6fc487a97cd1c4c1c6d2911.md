### Title
Missing Sender Validation on `ContractCodeResponse::V1` Allows Any Peer to Inject Fake Contract Code into Chunk Validators - (File: chain/client/src/stateless_validation/validate.rs)

### Summary

Before the `SignedContractCodeResponse` protocol feature is enabled, `validate_contract_code_response` accepts `ContractCodeResponse::V1` messages without validating the sender's identity. Any peer on the network can send a routed message carrying a `ContractCodeResponse::V1` with arbitrary fake contract code bytes to a chunk validator. The validator stores those bytes without checking they came from the actual chunk producer, corrupting the contract code cache used for chunk state witness validation.

### Finding Description

`validate_contract_code_response` in `chain/client/src/stateless_validation/validate.rs` conditionally skips all sender-identity checks: [1](#0-0) 

When `ProtocolFeature::SignedContractCodeResponse` is not yet enabled, the function only calls `validate_chunk_relevant` (height/epoch window checks) and returns `Ok(ChunkRelevance::Relevant)` with no verification of who sent the response. `ContractCodeResponse::V1` carries no signature at all — its `verify_signature` method unconditionally returns `false`: [2](#0-1) 

The handler `handle_contract_code_response` in `PartialWitnessActor` calls `validate_contract_code_response` and, on success, stores the received code bytes directly into the `partial_witness_tracker`: [3](#0-2) 

The network dispatch path in `receive_routed_message` forwards any `T1MessageBody::ContractCodeResponse` to the `partial_witness_adapter` without any sender check: [4](#0-3) 

Critically, `PeerMessage::Routed` is permitted on T2 (general) connections, and the tier check in `handle_msg_ready` applies only to the outer `PeerMessage` variant, not to the inner `TieredMessageBody`. A T2 peer can therefore wrap a `T1MessageBody::ContractCodeResponse` inside a routed message, have it forwarded hop-by-hop to the target validator, and have it processed as if it were a legitimate chunk-producer response: [5](#0-4) 

The routing signature (`msg.verify()`) only authenticates the routing envelope author (any peer with a valid key), not the content: [6](#0-5) 

By contrast, `ContractCodeRequest` and `ChunkContractAccesses` already require a valid chunk-validator or chunk-producer signature respectively: [7](#0-6) [8](#0-7) 

The `SignedContractCodeResponse` feature was introduced precisely to close this gap: [9](#0-8) 

### Impact Explanation

An attacker sends a routed `ContractCodeResponse::V1` message containing wrong or empty contract code bytes, targeting a specific `ChunkProductionKey` (epoch_id, shard_id, height_created — all public information). The validator's `partial_witness_tracker` stores those bytes under that key. When the validator later attempts to reconstruct and validate the chunk state witness, it uses the injected wrong code, causing witness validation to fail. The validator does not emit a `ChunkEndorsement`. If the attacker targets enough validators simultaneously, the chunk fails to accumulate the required endorsement stake, and the block producer cannot include the chunk — a concrete DoS on chunk production.

The corrupted value is the `AccessedContractCodes` entry in `PartialEncodedStateWitnessTracker`, which directly determines whether `ChunkEndorsement` messages are produced and whether chunks are included in blocks.

### Likelihood Explanation

The `ChunkProductionKey` is deterministic and publicly observable. Any peer with a valid peer key (no stake required) can construct and sign a routing envelope, embed a `ContractCodeResponse::V1` body, and route it to any validator. The attack requires no validator privileges, no admin access, and no special configuration. The only timing constraint is that the fake response must arrive after the validator has begun tracking the chunk (after receiving `ChunkContractAccesses`), which is also observable from the network.

### Recommendation

Apply the same sender-identity check unconditionally, regardless of protocol version, by requiring that `ContractCodeResponse` carries a verifiable chunk-producer signature before storing any code bytes. Until `SignedContractCodeResponse` is universally enabled, add a fallback check that at minimum verifies the network-layer sender peer ID maps to a known chunk producer for the relevant shard and epoch, mirroring the pattern already used for `ChunkContractAccesses` and `ContractCodeRequest`.

### Proof of Concept

1. Attacker node connects to the network as a regular T2 peer (no stake required).
2. Attacker observes or predicts the `ChunkProductionKey` for an upcoming chunk (epoch_id, shard_id, height_created are all derivable from public chain state).
3. Attacker waits until the target validator has received a `ChunkContractAccesses` message for that key (observable from the network).
4. Attacker constructs a `ContractCodeResponse::V1` with the correct `ChunkProductionKey` but containing garbage or empty `compressed_contracts`.
5. Attacker wraps it in a `PeerMessage::Routed` with `T1MessageBody::ContractCodeResponse` body, signs the routing envelope with its own peer key, and routes it to the target validator's peer ID.
6. The validator's `handle_peer_message` → `receive_routed_message` dispatches the body to `PartialWitnessActor::handle_contract_code_response`.
7. `validate_contract_code_response` passes (chunk is relevant, `SignedContractCodeResponse` not enabled, no signature check).
8. `store_accessed_contract_codes` writes the garbage bytes into the tracker under the target key.
9. When the validator attempts chunk state witness validation, it uses the wrong contract code, validation fails, and no `ChunkEndorsement` is emitted for that chunk.

pub fn validate_contract_code_response(
    epoch_manager: &dyn EpochManagerAdapter,
    response: &ContractCodeResponse,
    store: &Store,
) -> Result<ChunkRelevance, Error> {
    let key = response.chunk_production_key();
    require_relevant!(validate_chunk_relevant(epoch_manager, key, store)?);
    let protocol_version = epoch_manager.get_epoch_protocol_version(&key.epoch_id)?;
    if ProtocolFeature::SignedContractCodeResponse.enabled(protocol_version) {
        validate_witness_contract_code_response_signature(epoch_manager, response)?;
    }

    Ok(ChunkRelevance::Relevant)
}
