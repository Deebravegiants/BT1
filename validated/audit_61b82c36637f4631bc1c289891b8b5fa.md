### Title
Receipt-Proof Ordering Not Validated in State Sync Header — Malicious Peer Can Cause Wrong State Root Propagation - (`chain/chain/src/chain_update.rs`, `chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_shard_state` validates that each `from_shard_id` in `incoming_receipts_proofs` is unique (pigeonhole principle) and that each receipt set has a valid Merkle proof, but it does **not** validate the **order** of `receipt_proofs` within a block. A malicious peer can reorder the `receipt_proofs` (and their paired `root_proofs`) within any block entry, pass all checks, and cause `set_state_finalize` to apply receipts in a different order than honest nodes used. Because receipt execution order affects the state trie (especially when the gas limit is hit), this produces a wrong `new_root`. Every subsequent `set_state_finalize_on_height` call chains off this wrong root, corrupting the entire finalization sequence up to `sync_hash`.

---

### Finding Description

**Step 1 — How honest nodes store incoming receipts.**

During normal block processing, `postprocess_block` calls `save_incoming_receipt` with receipts that came from `collect_incoming_receipts_from_chunks`, which applies `shuffle_receipt_proofs` keyed on `block.header().prev_hash()` before storing. [1](#0-0) [2](#0-1) 

So the DB stores receipt proofs in the **shuffled** order that was actually used for chunk application.

**Step 2 — How the state sync header is generated.**

`get_shard_state` reads those shuffled receipts from the DB via `get_incoming_receipts_for_shard` and builds `root_proofs` in the same iteration order as `receipt_proofs`. [3](#0-2) 

The `root_proofs[i][j]` entry is the Merkle proof for `receipt_proofs[i][j]`; the pairing is positional, not keyed by `from_shard_id`.

**Step 3 — What `set_shard_state` validates.**

The validation loop (checks 4a–4g) enforces:
- Continuous block chain (4b)
- Count of `receipt_proofs` equals `chunks_included` (4c)
- Uniqueness of `from_shard_id` within a block (4d, pigeonhole)
- Merkle proof validity for each receipt set (4e, 4f) [4](#0-3) 

**What is NOT validated**: the **order** of `receipt_proofs` within a block. A malicious peer can permute the list (e.g., swap shard 1 and shard 2 entries) and simultaneously permute `root_proofs[i]` in the same way. Every individual Merkle proof still verifies because each proof is tied to its own receipt set hash, not to a position.

**Step 4 — How `set_state_finalize` uses the receipts.**

`collect_receipts_from_response` flattens `receipt_proof_responses` in iteration order — outer loop over blocks, inner loop over `receipt_proofs` within each block — with no shuffling applied. [5](#0-4) [6](#0-5) 

If the peer reordered the proofs, the resulting `Vec<Receipt>` differs from the honest shuffled order, and `apply_chunk` computes a different `new_root`.

**Step 5 — Propagation through `set_state_finalize_on_height`.**

Every subsequent call reads `chunk_extra` from the previous block and chains off its `state_root`. [7](#0-6) 

A wrong initial root from `set_state_finalize` propagates to every block between `height_included` and `sync_hash`.

---

### Impact Explanation

The syncing node stores wrong `chunk_extra` state roots for all blocks up to `sync_hash`. After state sync completes, the node:
- Holds a state trie that diverges from the canonical chain state.
- Produces chunk extras with wrong state roots, causing it to reject valid chunks (whose `prev_state_root` matches the honest root, not the corrupted one).
- Cannot participate correctly in consensus or chunk validation.

This is "accepts forged state" in the consensus path: the node treats a wrong state root as the valid post-sync state.

---

### Likelihood Explanation

The attacker only needs to be a peer reachable by the syncing node during state sync — an unprivileged node operator. State sync is a standard production code path triggered whenever a node falls behind. The attack requires crafting a `ShardStateSyncResponseHeader` with permuted `receipt_proofs`/`root_proofs` within one or more blocks. All Merkle checks pass. The wrong state root materializes whenever the gas limit is hit during receipt processing (common on a loaded network) or when receipts have order-dependent effects.

---

### Recommendation

In `set_shard_state`, after verifying uniqueness of `from_shard_id` (check 4d), enforce that the order of `receipt_proofs` within each block matches the canonical shuffled order. Specifically, after collecting all `receipt_proofs` for a block, apply `shuffle_receipt_proofs` with `get_receipts_shuffle_salt(block)` and assert the result equals the provided order. Alternatively, sort the proofs by `from_shard_id` before hashing and enforce that the peer provides them in that canonical order, then apply the shuffle deterministically inside `set_state_finalize` rather than relying on peer-supplied ordering. [8](#0-7) 

---

### Proof of Concept

1. Run a node that will serve state sync responses.
2. Intercept or mock the `ShardStateSyncResponseHeader` it sends.
3. For any block entry in `incoming_receipts_proofs` that has ≥2 `receipt_proofs`, swap two entries and swap the corresponding entries in `root_proofs[i]`.
4. Deliver the modified header to a syncing node. `set_shard_state` accepts it (all Merkle checks pass, `from_shard_id` uniqueness holds).
5. The syncing node calls `set_state_finalize`, which calls `collect_receipts_from_response` on the reordered proofs, applies the chunk with receipts in the wrong order, and stores a wrong `chunk_extra.state_root`.
6. All `set_state_finalize_on_height` calls for subsequent heights chain off this wrong root.
7. Verify: the `chunk_extra` state root stored for `height_included` differs from the one stored by an honest node that processed the same block normally. [9](#0-8) [10](#0-9)

```rust
pub fn shuffle_receipt_proofs<ReceiptProofType>(
    receipt_proofs: &mut Vec<ReceiptProofType>,
    shuffle_salt: &CryptoHash,
) {
    let mut slice = [0u8; 32];
    slice.copy_from_slice(shuffle_salt.as_ref());
    let mut rng: ChaCha20Rng = SeedableRng::from_seed(slice);
    receipt_proofs.shuffle(&mut rng);
}
```
