### Title
Daily Mint Limit Exhaustion + Block Stuffing Enables Sustained Deposit DoS — (File: `contracts/pools/RSETHPoolV2.sol`)

---

### Summary

The `limitDailyMint` modifier imposes a global daily cap with no per-user sub-limit. On a low-fee L2, an attacker can exhaust the entire daily cap in one transaction, then use block stuffing to guarantee they are first to exhaust it again at the next day boundary — repeating indefinitely.

---

### Finding Description

The `limitDailyMint` modifier resets `dailyMintAmount` to zero whenever `currentDay > lastMintDay`, then checks the global cap: [1](#0-0) 

There is no per-depositor sub-limit, no minimum deposit interval, and no minimum deposit size. A single call to `deposit()` can consume the entire `dailyMintLimit` in one transaction. [2](#0-1) 

The attack proceeds as follows:

1. **Day N opens** (`currentDay > lastMintDay` → `dailyMintAmount` resets to 0).
2. Attacker calls `deposit()` with enough ETH so that `rsETHAmount ≥ dailyMintLimit`. The modifier sets `dailyMintAmount = dailyMintLimit`. All subsequent `deposit()` calls revert with `DailyMintLimitExceeded` for the rest of the day.
3. Attacker fills the remaining ~86 400 seconds of blocks with cheap filler transactions (block stuffing). On low-base-fee L2s this is inexpensive.
4. **Day N+1 opens.** Because the attacker controls block inclusion via stuffing, they are guaranteed to be the first transaction in the new day, exhausting the limit again before any legitimate user can deposit.
5. Repeat indefinitely.

The block stuffing step is the critical enabler: without it, a legitimate user could race the attacker at the day boundary. With it, the attacker holds exclusive access to the first slot of every new day.

`getCurrentDay()` is purely timestamp-based: [3](#0-2) 

Time advances normally even during block stuffing, so the day does reset — but the attacker's stuffed blocks ensure no competing transaction lands before theirs.

---

### Impact Explanation

**Low. Block stuffing.** Legitimate users are unable to call `deposit()` for the entire daily window, repeated across consecutive days. Their ETH is not lost (they still hold it), but the pool is rendered functionally unusable for deposits for as long as the attacker sustains the campaign.

---

### Likelihood Explanation

Realistic on any L2 where the base fee is low enough that filling a day's worth of blocks costs less than the fee revenue the attacker sacrifices. The attacker receives `wrsETH` worth approximately `(1 - feeBps/10000) × dailyMintLimit` ETH per day, so the net cost is only `feeBps/10000 × dailyMintLimit` ETH per day plus block-stuffing gas. On chains like Optimism or Base at low-congestion base fees, block stuffing for 24 hours can cost a few dollars, while the fee cost on a large daily limit is also modest.

---

### Recommendation

1. **Per-user daily sub-limit**: Track `dailyMintAmountPerUser[msg.sender]` and cap each address's daily contribution to a fraction of `dailyMintLimit`.
2. **Minimum deposit size**: Require `msg.value >= minDepositAmount` to raise the capital cost of exhausting the limit in one shot.
3. **Partial fill instead of revert**: Allow a deposit up to the remaining limit rather than reverting entirely, so the attacker cannot atomically exhaust the cap in one transaction without leaving residual capacity.

---

### Proof of Concept

```solidity
// Economic model (pseudo-test, local fork)
// Preconditions: L2 base fee < threshold, attacker has dailyMintLimit ETH

for (uint day = 0; day < N; day++) {
    // 1. Wait for day boundary (or warp in test)
    vm.warp(pool.startTimestamp() + day * 1 days + 1);

    // 2. Exhaust daily limit in one deposit
    uint256 needed = computeETHForLimit(pool.dailyMintLimit(), pool.feeBps(), oracle.getRate());
    pool.deposit{value: needed}("attacker");
    // dailyMintAmount == dailyMintLimit; all further deposits revert

    // 3. Block-stuff remainder of day (simulated by asserting no user deposit succeeds)
    vm.expectRevert(RSETHPoolV2.DailyMintLimitExceeded.selector);
    pool.deposit{value: 1 ether}("user");

    // 4. Assert attacker net cost = fee only
    uint256 fee = needed * pool.feeBps() / 10_000;
    assertLt(fee, blockStuffingCostThreshold); // profitable on low-fee L2
}
```

The loop asserts that across `N` days the attacker's total cost (fee + stuffing gas) is below the economic threshold, confirming sustained profitability on low-fee L2s.
