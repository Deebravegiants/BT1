Audit Report

## Title
Stale `rsETHPrice` in `getRsETHAmountToMint()` Allows Depositors to Receive Excess rsETH, Diluting Existing Holders' Yield - (File: contracts/LRTDepositPool.sol)

## Summary
`getRsETHAmountToMint()` in `LRTDepositPool` divides by the cached `rsETHPrice` state variable from `LRTOracle`, which is only updated on explicit calls to `updateRSETHPrice()`. As staking rewards accrue between updates, the stored price falls below the true price, causing any depositor to receive more rsETH than their contribution warrants. The excess rsETH permanently dilutes existing holders' share of the TVL.

## Finding Description
`getRsETHAmountToMint()` at `contracts/LRTDepositPool.sol:520` computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.rsETHPrice()` returns the public state variable `rsETHPrice` declared at `contracts/LRTOracle.sol:28`, which is only written in `_updateRsETHPrice()` (line 313). This internal function is only reachable via the public `updateRSETHPrice()` (line 87) or manager-only `updateRSETHPriceAsManager()` (line 94).

Neither `depositETH()` (`contracts/LRTDepositPool.sol:76-93`) nor `depositAsset()` (lines 99-118) calls `updateRSETHPrice()` before computing the mint amount. The helper `_beforeDeposit()` (lines 648-670) is declared `private view`, making it structurally incapable of triggering a price update. The call chain is:

```
depositETH() / depositAsset()
  → _beforeDeposit() [private view]
    → getRsETHAmountToMint() [public view]
      → lrtOracle.rsETHPrice()  ← stale cached value
```

Between any two calls to `updateRSETHPrice()`, ETH staking rewards cause the true price (`totalETHInProtocol / rsethSupply`) to exceed the stored `rsETHPrice`. A depositor who deposits during this window receives `deposit * assetPrice / rsETHPrice_stale`, which is greater than the correct `deposit * assetPrice / rsETHPrice_actual`. The excess rsETH inflates `rsethSupply`; when `updateRSETHPrice()` is eventually called, the new price is computed as `(totalETHInProtocol - fee) / rsethSupply` (line 250), and the inflated denominator permanently lowers the price for all existing holders.

An additional aggravating factor: if the accrued yield causes the true price to exceed `highestRsethPrice` by more than `pricePercentageLimit`, non-manager callers receive `PriceAboveDailyThreshold` (lines 260-265), preventing them from updating the price and extending the staleness window indefinitely until a manager acts.

## Impact Explanation
**High — Theft of unclaimed yield.** Existing rsETH holders' accrued staking yield is partially transferred to new depositors who mint against the stale price. After `updateRSETHPrice()` is called, the settled price is lower than it would have been had the deposit used the correct price, because the supply denominator is inflated. The stolen yield is proportional to deposit size and the duration of price staleness, and the effect is permanent and irreversible.

## Likelihood Explanation
**Medium.** ETH staking rewards accrue every block, so `rsETHPrice` is perpetually stale between updates. No special setup is required — any depositor calling `depositETH()` or `depositAsset()` during a staleness window benefits. Depositors have a direct financial incentive to avoid calling `updateRSETHPrice()` before depositing. The `pricePercentageLimit` guard can further extend the exploitable window by blocking public price updates when yield accumulation is large.

## Recommendation
Remove the `view` modifier from `_beforeDeposit()` and call `_updateRsETHPrice()` (or an equivalent on-the-fly TVL/supply computation) atomically before `getRsETHAmountToMint()` is invoked. This ensures the rsETH price always reflects the current state at the moment of deposit, eliminating the staleness window entirely.

## Proof of Concept
**Initial state:** TVL = 100 ETH, rsETH supply = 100, `rsETHPrice` = 1e18.

1. 1 ETH of staking rewards accrue. True price = 101/100 = 1.01e18. `updateRSETHPrice()` is NOT called.
2. Attacker calls `depositETH{ value: 1 ether }(0, "")`.
3. `getRsETHAmountToMint(ETH, 1e18)` computes `1e18 * 1e18 / 1e18 = 1e18` → mints **1.0000 rsETH**.
4. Correct amount at true price: `1e18 * 1e18 / 1.01e18 ≈ 0.9901 rsETH`.
5. Attacker received **~0.0099 rsETH excess**.

**After `updateRSETHPrice()` is called:**
- TVL = 102 ETH, supply = 101 rsETH → settled price ≈ **1.0099e18**.
- Correct price (if attacker got 0.9901 rsETH): 102 / 100.9901 ≈ **1.0100e18**.
- Existing holders' rsETH is permanently worth ~0.01% less than it should be.

The exploit is repeatable every time rewards accrue before a price update and scales linearly with deposit size and staleness duration. A Foundry fork test can confirm this by: (1) forking mainnet, (2) advancing blocks to accrue rewards without calling `updateRSETHPrice()`, (3) depositing and recording rsETH minted, (4) calling `updateRSETHPrice()`, and (5) asserting the settled price is lower than it would be with a correctly-sized deposit.
