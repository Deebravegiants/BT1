Audit Report

## Title
`instantWithdrawal` Bypasses `assetsCommitted` Accounting, Enabling Temporary Freezing of Pending Withdrawal Requests - (File: contracts/LRTWithdrawalManager.sol)

## Summary

`instantWithdrawal` checks `getAssetsAvailableForInstantWithdrawal` on the `LRTUnstakingVault`, which deducts only the manually-set `queuedWithdrawalsBuffer`, not the protocol-tracked `assetsCommitted`. Because `queuedWithdrawalsBuffer` is a static operator-set value that does not automatically mirror `assetsCommitted`, any rsETH holder can drain vault assets already promised to users with pending queued withdrawal requests, causing `unlockQueue` to revert and temporarily freezing those users' funds.

## Finding Description

`initiateWithdrawal` enforces a committed-assets check before accepting a queued request:

```solidity
// LRTWithdrawalManager.sol L168-173
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();
assetsCommitted[asset] += expectedAssetAmount;
```

`getAvailableAssetAmount` (L599-602) computes `lrtDepositPool.getTotalAssetDeposits(asset) - assetsCommitted[asset]`, so `assetsCommitted` is the authoritative on-chain record of assets already spoken for.

`instantWithdrawal` (L228-235) uses a completely separate check:

```solidity
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
ILRTUnstakingVault unstakingVault = ...;
if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
    revert CantInstantWithdrawMoreThanAvailable();
}
unstakingVault.redeem(asset, assetAmountUnlocked);
```

`getAssetsAvailableForInstantWithdrawal` in `LRTUnstakingVault` (L235-237) returns `vaultBalance - queuedWithdrawalsBuffer[asset]`. The `queuedWithdrawalsBuffer` is set manually by an operator via `setQueuedWithdrawalsBuffer` and has no automatic linkage to `assetsCommitted`. Whenever `assetsCommitted[asset] > queuedWithdrawalsBuffer[asset]` — which is the normal state unless the operator proactively updates the buffer after every `initiateWithdrawal` call — instant withdrawals can redeem vault assets that are already committed to queued requests.

`unlockQueue` (L284-307) reads `unstakingVault.balanceOf(asset)` as `totalAvailableAssets` and then calls `unstakingVault.redeem(asset, assetAmountUnlocked)`. If the vault has been drained, `totalAvailableAssets == 0` causes an immediate revert at L297 (`AmountMustBeGreaterThanZero`), and no queued requests can be processed.

## Impact Explanation

**Medium — Temporary freezing of funds.** Users who called `initiateWithdrawal` have their expected asset amounts tracked in `assetsCommitted`. If instant withdrawals drain the `LRTUnstakingVault` of those committed assets, `unlockQueue` cannot process pending requests. The freeze persists until the operator replenishes the vault from the deposit pool or waits for EigenLayer's multi-day withdrawal delay to expire.

## Likelihood Explanation

**Medium.** The only precondition is that `isInstantWithdrawalEnabled[asset]` is `true` — a legitimate operational state set by the manager. Once enabled, any rsETH holder can call `instantWithdrawal` without any privileged role. The gap between `assetsCommitted` and `queuedWithdrawalsBuffer` arises naturally after each `initiateWithdrawal` call unless the operator manually updates the buffer in the same block, making the window continuously exploitable during normal operation.

## Recommendation

In `instantWithdrawal`, add a check against `getAvailableAssetAmount(asset)` (which accounts for `assetsCommitted`) before redeeming from the vault, mirroring the guard in `initiateWithdrawal`:

```solidity
if (assetAmountUnlocked > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();
```

No decrement of `assetsCommitted` is needed here because rsETH is burned immediately rather than held pending unlock. Alternatively, `getAssetsAvailableForInstantWithdrawal` in `LRTUnstakingVault` could be updated to accept and subtract the current `assetsCommitted` value, but the simpler fix is the availability check in `instantWithdrawal` itself.

## Proof of Concept

1. Protocol has 100 ETH in `LRTUnstakingVault`. `queuedWithdrawalsBuffer[ETH] = 0`.
2. User A calls `initiateWithdrawal(ETH, rsETHFor80ETH, "")`. `assetsCommitted[ETH] = 80 ETH`. `getAvailableAssetAmount(ETH)` now returns 20 ETH.
3. User B holds rsETH equivalent to 100 ETH. `isInstantWithdrawalEnabled[ETH] == true`.
4. User B calls `instantWithdrawal(ETH, rsETHFor100ETH, "")`. `getAssetsAvailableForInstantWithdrawal(ETH)` returns `100 - 0 = 100 ETH` (ignores `assetsCommitted`). The call succeeds, burning User B's rsETH and draining all 100 ETH from the vault.
5. Operator calls `unlockQueue(ETH, ...)`. `unstakingVault.balanceOf(ETH) == 0`, so `params.totalAvailableAssets == 0` and the call reverts with `AmountMustBeGreaterThanZero`.
6. User A's withdrawal request is frozen. The operator must replenish the vault from the deposit pool or wait for EigenLayer's withdrawal delay (potentially days) before `unlockQueue` can succeed.
