### Title
User-Controlled Refund Amount in Dijkstra Subtransaction `UnRegDepositTxCert` Lacks Multi-Subtransaction Double-Claim Validation — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxCert.hs`)

---

### Summary

In the Dijkstra era, `UnRegDepositTxCert` certificates embed the refund amount directly in the user-supplied certificate. The function `dijkstraTotalRefundsTxCerts` reads this amount verbatim from the certificate without consulting the ledger state. The comment in the code explicitly states "ledger rules will validate that they are accurate," but the test file for the Dijkstra CERT rule contains a disabled test (`xit`) with the comment `error "TODO: predicate failure not yet implemented"` for the case where multiple subtransactions attempt to claim the same deposit refund for the same credential. This is the direct analog to the Sablier bug: a user-controlled parameter is assumed to be validated, but the validation is absent.

---

### Finding Description

In `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxCert.hs`, the function `dijkstraTotalRefundsTxCerts` computes the total refunds for a set of certificates by reading the deposit amount directly from each `UnRegDepositTxCert` or `UnRegDRepTxCert` certificate:

```haskell
-- | Unlike previous eras, we no longer need to lookup refunds from the ledger state,
-- since all of the certificates specify the actual refund and ledger rules will validate
-- that they are accurate.
dijkstraTotalRefundsTxCerts ::
  ( Foldable f, ConwayEraTxCert era ) =>
  f (TxCert era) -> Coin
dijkstraTotalRefundsTxCerts = foldMap' $ \case
  UnRegDepositTxCert _ deposit -> deposit
  UnRegDRepTxCert _ deposit -> deposit
  _ -> zero
```

The `EraTxCert` instance for `DijkstraEra` wires this in as the canonical refund computation, ignoring protocol parameters and ledger state entirely:

```haskell
getTotalRefundsTxCerts _ _ _ = dijkstraTotalRefundsTxCerts
```

The design intent is that the DELEG/CERT rules validate the user-supplied amount against the stored deposit. However, the test file `eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs` contains two disabled tests (`xit`) that expose the gap:

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
  ...
  subTx1 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
  subTx2 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
  tx = mkBasicTx mkBasicTxBody
         & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
  submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
```

The `error "TODO: predicate failure not yet implemented"` is the expected predicate failure value passed to `submitFailingTx`. Its absence means no production predicate failure has been defined or implemented to reject this case. The test is disabled precisely because the check does not yet exist.

The attack surface is the Dijkstra nested-transaction (`subTransactions`) feature: a top-level transaction embeds multiple subtransactions, each containing an `UnRegDepositTxCert` for the **same** credential with the **same** user-specified deposit amount. Because `dijkstraTotalRefundsTxCerts` sums the user-supplied amounts from all certificates across all subtransactions, and because no predicate failure prevents two subtransactions from both claiming the same credential's deposit, the deposit pot can be drained by the difference between the double-counted refund and the single actual deposit.

---

### Impact Explanation

**Critical — Direct loss of ADA through an invalid ledger state transition.**

The deposit pot (`utxosDeposited`) is a real ADA balance maintained by the ledger. If two subtransactions both include `UnRegDepositTxCert stakingCred D` for the same credential, the preservation-of-value equation credits `2 * D` on the consumed side while only `D` was ever deposited. The extra `D` lovelace is drawn from the deposit pot, constituting a direct, attacker-controlled extraction of ADA from the ledger's accounting invariant.

---

### Likelihood Explanation

Any unprivileged transaction sender can construct a top-level Dijkstra transaction with two or more subtransactions each containing `UnRegDepositTxCert` for the same credential. No special privilege, key compromise, or governance majority is required. The attacker only needs to have previously registered a stake credential (paying the `keyDeposit`) and then submit the malformed batch transaction. The entry path is fully reachable from a normal user transaction.

---

### Recommendation

1. Implement the missing predicate failure in the Dijkstra CERT/DELEG rule that rejects any batch transaction (top-level + subtransactions) where the same credential appears in more than one `UnRegDepositTxCert` across the entire batch.
2. Enable and complete the two `xit` tests in `CertSpec.hs` once the predicate failure is defined.
3. Consider whether `dijkstraTotalRefundsTxCerts` should validate the user-supplied amount against the stored deposit (as the Conway DELEG rule does) rather than relying solely on a separate batch-level check.

---

### Proof of Concept

```
1. Register stakingCred, paying keyDeposit D.
2. Construct:
     subTx1 = { certs: [UnRegDepositTxCert stakingCred D] }
     subTx2 = { certs: [UnRegDepositTxCert stakingCred D] }
     topTx  = { subTransactions: [subTx1, subTx2] }
3. Submit topTx.
4. dijkstraTotalRefundsTxCerts sums D + D = 2D on the consumed side.
5. Only D was in the deposit pot for stakingCred.
6. The extra D is extracted from the deposit pot, violating the deposit invariant.
