Audit Report

## Title
ETH Deposit Limit Check Omits Incoming Amount, Allowing Cap Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` evaluates the ETH deposit cap against the pre-deposit total only, never adding the incoming `amount`. Any unprivileged caller can therefore exceed the configured ETH cap by an arbitrary amount in a single `depositETH` call. The ERC20 branch of the same function correctly includes `amount` in its comparison, confirming the asymmetry is unintentional.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682, `_checkIfDepositAmountExceedesCurrentLimit` branches on asset type:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount omitted
}
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct
```

The ETH branch returns `true` (triggering `MaximumDepositLimitReached`) only when the running total **already** exceeds the cap. The incoming `amount` is never considered. This check is invoked by `_beforeDeposit` (line 661), which is called unconditionally by `depositETH` (line 87). `depositETH` carries no role restriction — only `whenNotPaused` and `onlySupportedAsset` — making it reachable by any external account. As long as `totalAssetDeposits ≤ depositLimitByAsset(ETH)`, the guard always passes regardless of `msg.value`.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

The ETH deposit cap is a risk-management invariant. Bypassing it allows the protocol to absorb an unbounded ETH amount in a single transaction, violating `getTotalAssetDeposits(ETH) ≤ depositLimitByAsset(ETH)`. No user funds are stolen and no funds are frozen; the protocol simply operates outside its declared safety envelope.

## Likelihood Explanation
**Medium.** The entry point is permissionless. The only precondition is that `totalAssetDeposits(ETH) ≤ depositLimitByAsset(ETH)`, which is trivially observable on-chain and true during normal operation. No special role, flash loan, or oracle manipulation is required. The bypass is repeatable until the admin intervenes.

## Recommendation
Mirror the ERC20 branch in the ETH branch:

```diff
 if (asset == LRTConstants.ETH_TOKEN) {
-    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
+    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
 }
```

## Proof of Concept
1. Admin sets `depositLimitByAsset(ETH) = 1000 ether`.
2. Normal usage brings `getTotalAssetDeposits(ETH)` to `999 ether`.
3. Attacker calls `depositETH{value: 5000 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `999 ether > 1000 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for the full `5000 ether`.
6. `getTotalAssetDeposits(ETH)` is now `5999 ether` — nearly 6× the cap — with no protocol-level rejection.

**Foundry test sketch:**
```solidity
function test_ethCapBypass() public {
    // set limit to 1000 ETH, deposit 999 ETH normally
    // then deposit 5000 ETH as attacker
    vm.deal(attacker, 5000 ether);
    vm.prank(attacker);
    depositPool.depositETH{value: 5000 ether}(0, "");
    assertGt(depositPool.getTotalAssetDeposits(ETH_TOKEN), 1000 ether);
}
```
