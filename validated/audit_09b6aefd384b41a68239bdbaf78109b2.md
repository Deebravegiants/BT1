Audit Report

## Title
Missing Minimum Output Slippage Guard in L2 Pool `deposit()` Functions - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
All L2 RSETHPool deposit entry points compute rsETH output solely from the live oracle rate at execution time and expose no `minRsETHAmountExpected` parameter. If the protocol's cross-chain oracle rate is updated between a user's transaction submission and execution, the user silently receives fewer rsETH than previewed with no ability to revert. The L1 `LRTDepositPool` already guards against this exact scenario via `_beforeDeposit`.

## Finding Description
In `RSETHPoolV3.deposit(string)` (ETH path, lines 246–265) and `RSETHPoolV3.deposit(address,uint256,string)` (token path, lines 271–293), the rsETH output is computed by `viewSwapRsETHAmountAndFee` (lines 299–308), which divides the fee-adjusted input by the live `rsETHToETHrate` from the oracle:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

Because `rsETHToETHrate` is read at execution time and no minimum output check exists, a user who previews the deposit at rate R and submits a transaction may have it execute at rate R′ > R after a routine protocol oracle update, receiving fewer rsETH with no revert path. The same pattern is present in `RSETHPool`, `RSETHPoolNoWrapper`, `RSETHPoolV2`, and `RSETHPoolV2NBA`.

By contrast, `LRTDepositPool._beforeDeposit` (lines 648–670) explicitly reverts with `MinimumAmountToReceiveNotMet` if `rsethAmountToMint < minRSETHAmountExpected`, confirming the protocol recognizes this as a real deposit-flow risk on L1. No equivalent guard exists in any L2 pool variant.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but does not lose value.**

A depositor who previews the swap off-chain (or via `viewSwapRsETHAmountAndFee`) and submits a transaction may receive materially fewer rsETH than expected if the oracle rate advances before inclusion. The shortfall is not extracted by an attacker; it remains as higher ETH-per-rsETH backing. The user suffers a silent reduction with no on-chain recourse.

## Likelihood Explanation
**Low-Medium.** The rsETH oracle rate is updated by the protocol via cross-chain messaging on a regular cadence. No privileged attacker action is required — a routine rate update landing before a user's deposit is sufficient. On L2 networks, sequencer ordering means user transactions can be included after a rate update without any adversarial action. The condition is repeatable on every oracle update cycle.

## Recommendation
Add a `minRsETHAmountExpected` parameter to every `deposit()` overload in all L2 pool contracts (`RSETHPoolV3`, `RSETHPool`, `RSETHPoolNoWrapper`, `RSETHPoolV2`, `RSETHPoolV2NBA`). After computing `rsETHAmount` via `viewSwapRsETHAmountAndFee`, revert if `rsETHAmount < minRsETHAmountExpected`. This mirrors the protection already present in `LRTDepositPool._beforeDeposit` (lines 667–669).

## Proof of Concept
1. Current oracle rate: `1.05e18`. User calls `viewSwapRsETHAmountAndFee(1 ether)` off-chain, expects `≈ 0.952 rsETH` after fee.
2. User submits `RSETHPoolV3.deposit{value: 1 ether}("ref")`.
3. Before inclusion, the protocol's cross-chain rate propagation updates the oracle to `1.10e18`.
4. Transaction executes: `rsETHAmount = 1e18 * 1e18 / 1.10e18 ≈ 0.909 rsETH` — ~4.5% less than previewed.
5. No revert occurs; user receives `0.909 rsETH` with no ability to reject the worse rate.

**Foundry fork test plan:** Deploy a mock oracle returning `1.05e18`. Call `deposit{value: 1 ether}` in the same block after updating the mock oracle to `1.10e18`. Assert `wrsETH.balanceOf(user) < expectedAmount` and confirm no revert occurred.
