### Title
No Staleness Check on Chainlink `latestRoundData()` Allows Stale Price to Corrupt rsETH Exchange Rate - (File: contracts/oracles/ChainlinkPriceOracle.sol)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls Chainlink's `latestRoundData()` but silently discards every return value except the raw `price`. There is no check on `updatedAt` (heartbeat staleness), no check that `answeredInRound >= roundId` (stale round), and no check that `price > 0`. A stale or zero price propagates directly into `LRTOracle._getTotalEthInProtocol()`, corrupting the rsETH/ETH exchange rate that is stored on-chain and used for all deposits and withdrawals.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` is the price source for every supported LST collateral asset (stETH, cbETH, rETH, etc.):

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L49-55
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();   // updatedAt and answeredInRound silently dropped
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
``` [1](#0-0) 

The three return values that guard against stale data — `roundId`, `updatedAt`, and `answeredInRound` — are all discarded. No revert is triggered if the feed has not been updated within its heartbeat window, if the round is incomplete, or if the price is zero or negative.

By contrast, the sibling contract `ChainlinkOracleForRSETHPoolCollateral` in the same repository applies all three guards:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L27-32
(uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
    AggregatorV3Interface(oracle).latestRoundData();
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0)            revert IncompleteRound();
if (ethPrice <= 0)             revert InvalidPrice();
``` [2](#0-1) 

The stale price from `ChainlinkPriceOracle` flows into `LRTOracle._getTotalEthInProtocol()`, which sums `assetPrice × assetBalance` for every supported asset:

```solidity
// contracts/LRTOracle.sol L336-343
for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
    address asset = supportedAssets[assetIdx];
    uint256 assetER = getAssetPrice(asset);          // ← stale price enters here
    uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
    totalETHInProtocol += totalAssetAmt.mulWad(assetER);
    ...
}
``` [3](#0-2) 

The corrupted `totalETHInProtocol` is then used to compute and persist `rsETHPrice`:

```solidity
// contracts/LRTOracle.sol L250
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
``` [4](#0-3) 

`updateRSETHPrice()` is a public function with no access control, so any external caller can trigger the price update at will:

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [5](#0-4)

### Impact Explanation

**Scenario A — Stale price is lower than actual (e.g., LST depegged but Chainlink hasn't updated):**
- `totalETHInProtocol` is understated → `rsETHPrice` is deflated.
- An attacker calls `updateRSETHPrice()` to lock in the deflated price, then deposits LST collateral and receives more rsETH shares than the true value warrants.
- When the price corrects, the attacker redeems at the true rate, extracting value from existing rsETH holders (share dilution / theft of yield).

**Scenario B — Stale price is higher than actual (e.g., LST price dropped but Chainlink hasn't updated):**
- `rsETHPrice` is inflated.
- An attacker calls `updateRSETHPrice()` to lock in the inflated price, then initiates a withdrawal, receiving more underlying ETH than their rsETH shares are actually worth.
- This constitutes direct theft of assets from the pool.

Both scenarios map to the allowed impact scope: **theft of unclaimed yield (High)** in Scenario A and **direct theft of user funds (Critical)** in Scenario B.

---

### Likelihood Explanation

Chainlink feeds operate on a heartbeat (e.g., 1 hour for ETH/USD, up to 24 hours for some LST feeds) plus a deviation threshold. During the heartbeat window, the on-chain price can diverge from the true market price by the full deviation band without triggering an update. This is a known, recurring condition — not a hypothetical. Because `updateRSETHPrice()` is publicly callable with no access control, an attacker can monitor mempool/off-chain prices and call the function at the exact moment the stale price is most favorable, requiring no special privileges.

---

### Recommendation

1. **Short term:** Add staleness and validity guards to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:
   ```solidity
   (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
       priceFeed.latestRoundData();
   require(answeredInRound >= roundId, "Stale price");
   require(updatedAt != 0, "Incomplete round");
   require(price > 0, "Invalid price");
   require(block.timestamp - updatedAt <= MAX_STALENESS, "Price too old");
   ```
2. **Long term:** Restrict `updateRSETHPrice()` to trusted callers (e.g., a keeper with `MANAGER` role) or add a circuit-breaker that refuses to update the stored price if any individual feed is stale, preventing opportunistic timing attacks.

---

### Proof of Concept

1. Chainlink's stETH/ETH feed has a 24-hour heartbeat. At hour 23, the true stETH price has dropped 2% but the on-chain feed still reports the old price.
2. Attacker observes the discrepancy off-chain.
3. Attacker calls `LRTOracle.updateRSETHPrice()` (no access control). `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)`, which returns the stale (inflated) price. `rsETHPrice` is set above its true value.
4. Attacker immediately calls `LRTWithdrawalManager` to queue a withdrawal, receiving a claim priced at the inflated `rsETHPrice`.
5. When Chainlink updates (hour 24), the true price is reflected. The attacker's withdrawal claim was locked in at the inflated rate, extracting excess ETH from the pool at the expense of remaining rsETH holders.

The entire attack path is permissionless: steps 3–4 require only standard ERC-20 and deposit-pool interactions available to any unprivileged user.
