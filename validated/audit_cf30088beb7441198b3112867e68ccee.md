### Title
Malicious Peer Can Corrupt DL Singleton Root via Fabricated Solution in `singleton_removed` — (`chia/data_layer/data_layer_wallet.py`)

### Summary

`singleton_removed` extracts `root` and `inner_puzzle_hash` from CREATE_COIN hints in the CLVM output of a peer-supplied solution. `fetch_coin_spend` validates the puzzle reveal against the coin's puzzle hash but **never validates the solution**. A malicious peer can return the correct puzzle reveal with a crafted solution whose CREATE_COIN hints contain arbitrary bytes32 values, permanently corrupting the wallet's `SingletonRecord` and breaking all subsequent update spends.

---

### Finding Description

**Call chain:**

`_add_coin_states` (wallet_state_manager.py:2124) → `fetch_coin_spend_for_coin_state` → `fetch_coin_spend` → `singleton_removed`

**Step 1 — `fetch_coin_spend` validates puzzle, not solution.** [1](#0-0) 

The peer-supplied puzzle reveal is authenticated against `coin.puzzle_hash`. The solution, however, is accepted verbatim with no on-chain validation.

**Step 2 — `singleton_removed` runs the puzzle with the peer-supplied solution and blindly trusts the resulting hints.** [2](#0-1) 

`root` and `inner_puzzle_hash` are read directly from `condition[3][1]` and `condition[3][2]` — the CREATE_COIN memos — with no cross-check against the actual on-chain puzzle hash of the child coin.

**Step 3 — The corrupted values are persisted.** [3](#0-2) 

`root`, `inner_puzzle_hash`, and the derived `lineage_proof` (which is computed from the attacker-controlled `inner_puzzle_hash` and `root`) are all written to the DB.

**Step 4 — `create_update_state_spend` uses the corrupted record.** [4](#0-3) 

`singleton_record.inner_puzzle_hash` is used to look up the derivation record; if the attacker sets it to a non-existent hash, every subsequent call raises `ValueError` immediately. If set to a valid-but-wrong hash, `current_full_puz` is reconstructed with the wrong `root`, producing a coin ID that does not exist on-chain, and the spend is rejected by consensus.

---

### Impact Explanation

A malicious peer can permanently break the DL wallet owner's ability to update their Data Layer store. The on-chain singleton is unaffected, but the wallet's local state is corrupted such that every `create_update_state_spend` either raises an authorization error or produces a spend bundle referencing a non-existent coin. Recovery requires manual SQLite repair of the `singleton_records` table. This is a **High** impact: corruption of Data Layer root/store state with direct security impact, and long-lived inability to perform valid Data Layer updates.

---

### Likelihood Explanation

Any peer the wallet connects to can execute this attack. Chia wallets connect to full nodes discovered via DNS seeders and peer exchange. An attacker who operates a reachable full node, or who performs a MITM on an untrusted connection, can trigger this on any wallet that tracks a DL singleton. No keys, admin access, or broken cryptography are required.

---

### Recommendation

After `run_with_cost` produces the CREATE_COIN condition, cross-validate the extracted `root` and `inner_puzzle_hash` against the actual child coin's puzzle hash:

```python
expected_full_puz_hash = create_host_fullpuz(
    inner_puzzle_hash, root, singleton_record.launcher_id
).get_tree_hash_precalc(inner_puzzle_hash)

if full_puzzle_hash != expected_full_puz_hash:
    self.log.warning("Hints in CREATE_COIN do not match child puzzle hash; ignoring spend")
    return
```

This ensures that `root` and `inner_puzzle_hash` are consistent with the child coin's actual on-chain puzzle hash, which is already authenticated via the coin record.

### Proof of Concept

```python
# 1. Wallet tracks launcher_id; parent singleton record exists in dl_store.
# 2. Malicious peer intercepts RequestPuzzleSolution for the spent singleton coin.
# 3. Peer returns:
#    - puzzle_reveal: correct (matches coin.puzzle_hash — enforced by fetch_coin_spend)
#    - solution: crafted with valid lineage proof (known from chain) + inner solution
#      that produces CREATE_COIN(actual_child_puz_hash, odd_amount,
#                                [launcher_id, EVIL_ROOT, EVIL_INNER_PUZ_HASH])
# 4. singleton_removed runs the puzzle locally, extracts EVIL_ROOT and EVIL_INNER_PUZ_HASH.
# 5. add_singleton_record stores SingletonRecord(root=EVIL_ROOT, inner_puzzle_hash=EVIL_INNER_PUZ_HASH).
# 6. get_latest_singleton returns the record with EVIL_ROOT.
# 7. create_update_state_spend reconstructs current_full_puz with EVIL_ROOT → wrong puzzle hash
#    → Coin(parent, wrong_puz_hash, amount) does not exist on-chain → spend rejected.
# Assert: all subsequent update spends fail until manual DB repair.
```
