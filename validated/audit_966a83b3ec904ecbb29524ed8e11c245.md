### Title
Missing `payload_hash`-to-`request` Binding in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay — (File: `crates/contract/src/lib.rs`)

---

### Summary

`MpcContract::respond_verify_foreign_tx` verifies that `response.payload_hash` carries a valid root-key ECDSA signature, but never checks that `response.payload_hash` is actually derived from the `request` supplied in the same call. A single Byzantine MPC participant (strictly below the signing threshold) who has already obtained a valid signature for one pending foreign-tx request can replay that signature against a *different* pending request, causing the original caller to receive a forged `VerifyForeignTransactionResponse` that attests to the wrong foreign-chain transaction.

---

### Finding Description

`respond_verify_foreign_tx` performs two independent checks and then resolves the pending yield:

```
1. Verify sig(response.payload_hash) is valid under the root public key.
2. Look up `request` in pending_verify_foreign_tx_requests and drain it.
```

There is no step that binds `response.payload_hash` to `request.request` (the `ForeignChainRpcRequest` that was originally submitted). The contract stores the full `VerifyForeignTransactionRequest` as the map key, but when a response arrives it only asks "is this signature valid for *some* hash?" — not "is this hash the SHA-256 of a `ForeignTxSignPayload` whose embedded `ForeignChainRpcRequest` equals `request.request`?" [1](#0-0) 

Compare this with `respond`, which signs over a *derived* key whose tweak encodes the request's `predecessor` and `path`, implicitly binding the signature to the specific request. For foreign-tx, the node code uses a zero tweak (root key), so no such implicit binding exists: [2](#0-1) 

The `ForeignTxSignPayload` that nodes actually sign does embed the `ForeignChainRpcRequest`: [3](#0-2) 

But the contract never reconstructs or checks any part of that payload from the stored `request`.

---

### Impact Explanation

**High — Forged foreign-chain verification causing invalid bridge execution.**

A bridge contract that calls `verify_foreign_transaction` for Bitcoin tx `[0xAA;32]` will receive a `VerifyForeignTransactionResponse` whose `payload_hash` actually attests to Bitcoin tx `[0xBB;32]`. The bridge cannot distinguish this from a legitimate response: the signature is cryptographically valid under the root key, and the `payload_hash` is a well-formed 32-byte value. Any downstream logic that gates fund release on the `payload_hash` (e.g., by re-deriving it from the expected tx-id and comparing) will be deceived into releasing funds for the wrong transaction, enabling a double-spend or theft of bridged assets.

---

### Likelihood Explanation

**Realistic for a single Byzantine participant below the signing threshold.**

The attacker only needs to:
1. Be an attested MPC participant (no threshold collusion required).
2. Have two different foreign-tx requests pending simultaneously — routine in any active bridge deployment.
3. Participate in the legitimate threshold signing of Request B to obtain the final `(payload_hash_B, sig_B)` pair. The leader node assembles and broadcasts the final signature; any participant in that round receives it.
4. Call `respond_verify_foreign_tx(request_A, response_B)` before any honest node responds to Request A.

No key-share extraction, TEE attack, or network-level DoS is required.

---

### Recommendation

The contract must verify that `response.payload_hash` is consistent with `request.request`. Since the contract cannot independently reproduce the extracted `values`, the recommended fix is to include the `ForeignChainRpcRequest` as a separately committed field in the signed payload and verify it on-chain:

**Option A (preferred):** Change the signing scheme so that nodes sign `SHA-256(borsh(request_key) || SHA-256(borsh(values)))`, where `request_key` is the full `VerifyForeignTransactionRequest`. The contract can then verify the first component independently.

**Option B:** Add a `request_hash: Hash256` field to `VerifyForeignTransactionResponse` (committed as `SHA-256(borsh(ForeignChainRpcRequest))`), have nodes include it in the signed payload, and have the contract verify `request_hash == SHA-256(borsh(request.request))` before accepting the response.

Either approach closes the gap analogous to the `FluidLocker::provideLiquidity` fix: just as that fix validates `amount <= getAvailableBalance()` to bind the liquidity operation to the actual staked state, these fixes bind `response.payload_hash` to the actual pending request.

---

### Proof of Concept

**Setup:** Two pending requests exist:
- `request_A = VerifyForeignTransactionRequest { request: Bitcoin { tx_id: [0xAA;32], ... }, domain_id, payload_version }`
- `request_B = VerifyForeignTransactionRequest { request: Bitcoin { tx_id: [0xBB;32], ... }, domain_id, payload_version }`

**Step 1 — Attacker participates in legitimate signing of `request_B`:**
The MPC network produces `(payload_hash_B, sig_B)` where `payload_hash_B = SHA-256(borsh(ForeignTxSignPayloadV1 { request: Bitcoin([0xBB;32]), values: [BlockHash([0x42;32])] }))`. The attacker, as a participant, receives `sig_B`.

**Step 2 — Attacker calls `respond_verify_foreign_tx` with mismatched arguments:**
```
respond_verify_foreign_tx(
    request  = request_A,          // pending in the map
    response = VerifyForeignTransactionResponse {
        payload_hash: payload_hash_B,   // hash of request_B's payload
        signature:    sig_B,            // valid sig over payload_hash_B
    }
)
```

**Step 3 — Contract accepts the call:**
- `verify_ecdsa_signature(sig_B, payload_hash_B, root_pk)` → `Ok` ✓
- `pending_verify_foreign_tx_requests.get(request_A)` → found ✓
- `resolve_yields_for(request_A, serialize(response_B))` → yields for `request_A` are drained with `response_B`

**Step 4 — Alice's bridge contract receives `response_B`:**
Alice's `verify_foreign_transaction` call for tx `[0xAA;32]` resolves with `payload_hash_B` (attesting to tx `[0xBB;32]`). Any bridge logic that trusts this response will execute as if tx `[0xBB;32]` was the verified transaction, enabling invalid bridge execution. [4](#0-3) [5](#0-4)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
