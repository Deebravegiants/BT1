### Title
Fee-on-Transfer Token Mis-Accounting in `depositAsset` Leads to rsETH Over-Minting — (File: contracts/LRTDepositPool.sol)

---

### Summary

`LRTDepositPool.depositAsset` calculates the rsETH amount to mint from the caller-supplied `depositAmount` **before** the token transfer occurs, and never reconciles against the actual balance received. If a fee-on-transfer token is ever added as a supported LST asset, every depositor will receive more rsETH than the real value they contributed, diluting all existing rsETH holders.

---

### Finding Description

In `depositAsset`, the execution order is:

1. `rsethAmountToMint = _beforeDeposit(asset, depositAmount, minRSETHAmountExpected)` — computes rsETH to mint using the nominal `depositAmount`.
2. `IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount)` — pulls tokens; for a fee-on-transfer token the contract receives `depositAmount − fee`.
3. `_mintRsETH(rsethAmountToMint)` — mints rsETH calculated from the **nominal** amount, not the **actual** received amount. [1](#0-0) 

`_beforeDeposit` delegates to `getRsETHAmountToMint`, which multiplies `amount` (the nominal `depositAmount`) by the oracle price: [2](#0-1) 

No pre/post balance check is performed anywhere in the deposit path to detect a shortfall caused by a transfer fee.

`getTotalAssetDeposits` later reads the **real** on-chain `balanceOf`, so the rsETH price denominator reflects the true (lower) asset balance, while the numerator (total rsETH supply) was inflated at mint time. [3](#0-2) 

Note: the `safeTransferFrom` wrapper correctly handles tokens that do not return a bool (e.g., USDT), so that half of the original report's issue is already mitigated. The remaining gap is the fee-on-transfer accounting. [4](#0-3) 

---

### Impact Explanation

If a fee-on-transfer token is added as a supported asset:

- The depositor receives rsETH priced against `depositAmount` but only contributes `depositAmount − fee` worth of assets.
- `rsETHPrice()` (used by the oracle) is computed as `totalAssets / totalRsETHSupply`. Because `totalRsETHSupply` is inflated relative to `totalAssets`, the price per rsETH drops.
- Every existing rsETH holder suffers a proportional loss in redemption value — this is a direct, quantifiable theft of existing holders' funds.
- Repeated deposits amplify the insolvency gap.

**Impact class**: Critical — direct theft of user funds (existing rsETH holders' redemption value is permanently reduced).

---

### Likelihood Explanation

The current supported assets (stETH, rETH, cbETH, etc.) are not fee-on-transfer tokens, so the bug is dormant today. However:

- The `onlySupportedERC20Token` modifier only checks whether the asset is whitelisted in `LRTConfig`; it imposes no structural constraint against fee-on-transfer tokens.
- The LRT team has an established pattern of adding new LST/LRT assets over time.
- A single admin `addNewSupportedAsset` call is sufficient to activate the vulnerability for all future depositors.

**Likelihood**: Low (requires a future governance decision to add a fee-on-transfer token).

---

### Recommendation

After the `safeTransferFrom` call, measure the actual received amount and use it — not the caller-supplied `depositAmount` — for rsETH minting:

```solidity
uint256 balanceBefore = IERC20(asset).balanceOf(address(this));
IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
uint256 actualReceived = IERC20(asset).balanceOf(address(this)) - balanceBefore;

// Recompute rsETH to mint from actualReceived, not depositAmount
uint256 rsethAmountToMint = getRsETHAmountToMint(asset, actualReceived);
if (rsethAmountToMint < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet();
_mintRsETH(rsethAmountToMint);
```

Additionally, document in `LRTConfig.addNewSupportedAsset` that fee-on-transfer and rebasing tokens must not be added without corresponding contract-level safeguards.

---

### Proof of Concept

1. Governance adds a token with a 2 % transfer fee as a supported asset.
2. Attacker calls `depositAsset(feeToken, 1_000e18, 0, "")`.
3. `_beforeDeposit` computes `rsethAmountToMint` for **1 000** tokens at the current oracle price.
4. `safeTransferFrom` pulls 1 000 tokens; the contract receives **980** tokens (2 % fee deducted).
5. `_mintRsETH` mints rsETH sized for 1 000 tokens.
6. `getTotalAssetDeposits` now reports 980 tokens; `rsETHPrice` drops proportionally.
7. All pre-existing rsETH holders can redeem fewer assets per rsETH than before the deposit — the 20-token shortfall is socialised across all holders.
8. The attacker repeats to amplify the drain. [5](#0-4)
