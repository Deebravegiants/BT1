### Title
Unbounded Recursive Iteration Over Active Governance Proposals in `ratifyTransition` — (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs`)

---

### Summary

The Conway-era `RATIFY` rule processes every active governance proposal at each epoch boundary through an unbounded recursive traversal. No protocol-level cap on the number of simultaneously active proposals exists. An unprivileged actor who pays the `govActionDeposit` for each submission can inflate the active-proposals sequence, causing the epoch-boundary computation to grow linearly (and the `reorderActions` sort to grow O(n log n)) with the number of proposals, with no hard ceiling enforced by the ledger rules.

---

### Finding Description

`ratifyTransition` in `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs` is a recursive STS rule that consumes the entire `RatifySignal` — a `StrictSeq (GovActionState era)` — one element at a time:

```haskell

  gas@GovActionState {gasId, gasExpiresAfter} SSeq.:<| sigs -> do
    ...
    trans @(RATIFY era) $ TRC (env, st', RatifySignal sigs)   -- recursive call
    ...
  SSeq.Empty -> pure $ ...
``` [1](#0-0) 

Before this recursive descent, `runConwayRatify` (called from `finishDRepPulser`) first passes the entire sequence through `reorderActions`:

```haskell

reorderActions = SS.fromList . sortOn (actionPriority . gasAction) . toList
``` [2](#0-1) 

This materialises the full sequence as a list, sorts it O(n log n), and rebuilds it — all proportional to the number of active proposals.

`finishDRepPulser` then calls `runConwayRatify` with the full snapshot of proposals:

---

### Impact Explanation

All of the above operations execute synchronously at the epoch boundary inside the `EPOCH` → `NEWEPOCH` transition, which every honest node must complete deterministically. Because `ratifyTransition` is not tail-recursive (it performs work on the result of the recursive call before returning), a sufficiently large proposals sequence can:

1. Cause epoch-boundary processing time to grow proportionally to the number of active proposals, potentially exceeding the inter-block slot budget and causing nodes to fall behind the chain tip.
2. In the extreme, exhaust the Haskell runtime stack (the recursion depth equals the number of proposals), causing a runtime exception that crashes the node process.

If nodes with different runtime stack configurations diverge in whether they crash or complete the epoch transition, this constitutes a deterministic disagreement between honest nodes at the era-transition boundary — matching the **Medium** allowed impact: *attacker-controlled proposals exceed intended validation limits*.

---

### Likelihood Explanation

The attacker entry point is the `GOV` rule, which accepts `ProposalProcedure` items from any transaction author. No privileged role is required. The economic barrier is `govActionDeposit` (100,000 ADA on mainnet as of the example parameters), but:

- Deposits are **refunded** on expiry, so the attacker only needs to keep the deposit locked for `govActionLifetime` epochs (~30 days at 6 epochs).
- A well-funded attacker (or a coalition) can recycle deposits across epochs to maintain a large standing pool of active proposals.
- The `govActionDeposit` is itself a governance parameter; if it is ever lowered, the cost of the attack drops proportionally.
- There is no hard protocol cap that would reject a transaction solely because too many proposals are already active.

Likelihood is **Low** under current mainnet parameters but rises if `govActionDeposit` is reduced or if the attacker has substantial ADA holdings.

---

### Recommendation

Add a protocol parameter `maxActiveGovActions :: Word32` (analogous to `maxTxSize` or `maxCollateralInputs`) and enforce it in the `GOV` rule when a new `ProposalProcedure` is submitted:

```haskell
-- In the GOV transition rule:
runTest $ failureUnless
  (proposalsSize currentProposals < fromIntegral (pp ^. ppMaxActiveGovActionsL))
  TooManyActiveGovActions
```

This mirrors the existing pattern for `maxCollateralInputs` and `maxTxSize` and directly addresses the unbounded-collection problem identified in the external report.

---

### Proof of Concept

1. Set `govActionDeposit` to a low value via a governance parameter-change action.
2. Submit a large number of `InfoAction` or `TreasuryWithdrawals` proposals across many transactions over several epochs, each paying the deposit.
3. At the next epoch boundary, `finishDRepPulser` calls `reorderActions` (O(n log n) sort) followed by the recursive `ratifyTransition` (O(n) recursion depth) over all n active proposals.
4. With n large enough, epoch-boundary processing time grows beyond the inter-block slot window, or the Haskell stack is exhausted, causing node crashes or stalls — matching the "for loop over dynamic array" resource-exhaustion pattern from the external report.
