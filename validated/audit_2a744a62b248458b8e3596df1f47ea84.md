Audit Report

## Title
`ChainlinkPriceOracle::getAssetPrice()` Discards All Staleness Indicators, Enabling Over-Minting of rsETH at Stale Prices - (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary
`ChainlinkPriceOracle::getAssetPrice()` calls `latestRoundData()` but silently discards `updatedAt`, `answeredInRound`, and `roundId`, accepting any price regardless of age or round completeness. This stale price flows directly into `LRTDepositPool::getRsETHAmountToMint()`, allowing any unprivileged depositor to mint rsETH at an inflated exchange rate during oracle downtime or LST depeg events, extracting value from existing rsETH holders. The protocol already implements correct staleness validation in `ChainlinkOracleForRSETHPoolCollateral`, confirming awareness of the pattern.

## Finding Description
`ChainlinkPriceOracle::getAssetPrice()` destructures only the `price` field from `latestRoundData()`:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52
(, int256 price,,,) = priceFeed.latestRoundData();
``` [1](#0-0) 

No check is performed for:
- `answeredInRound < roundId` (incomplete/stale round)
- `updatedAt == 0` (round never finalized)
- `block.timestamp - updatedAt > threshold` (heartbeat staleness)
- `price <= 0` (invalid price)

By contrast, `ChainlinkOracleForRSETHPoolCollateral::getRate()` — used for pool collateral pricing — performs all three structural checks:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L30-32
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

The stale price propagates through `LRTOracle::getAssetPrice()`: [3](#0-2) 

Into `LRTDepositPool::getRsETHAmountToMint()`, which determines rsETH minted per deposit: [4](#0-3) 

And into `_getTotalEthInProtocol()`, which computes the rsETH/ETH exchange rate used as the denominator in the same mint calculation: [5](#0-4) 

No existing guard in `depositAsset()` or `_beforeDeposit()` checks oracle freshness; the only check is `minRSETHAmountExpected`, which an attacker sets to 0 or to the inflated amount. [6](#0-5) 

## Impact Explanation
**Critical — Direct theft of user funds.**

When a Chainlink LST/ETH feed goes stale with a last reported price above the current market price (e.g., during an LST depeg or oracle node downtime), an attacker deposits the depegged LST and receives more rsETH than the deposited collateral is worth. The excess rsETH can be sold or redeemed, extracting ETH value from existing rsETH holders. The `rsETHPrice` denominator is also computed using the same stale prices via `_getTotalEthInProtocol()`, compounding the miscalculation. This matches the allowed impact: **Critical — Direct theft of any user funds**.

## Likelihood Explanation
Chainlink LST/ETH feeds carry documented heartbeat intervals (1–24 hours depending on the feed). Network congestion, oracle node failures, or rapid market movement can cause feeds to lag beyond their heartbeat. LST depeg events (e.g., stETH in 2022) are historically documented and create exactly the stale-high-price condition required. The exploit requires no privileges: any address can call `depositAsset()` on a non-paused contract. The window persists until the feed updates or the contract is paused by an operator.

## Recommendation
Add staleness validation in `ChainlinkPriceOracle::getAssetPrice()`, mirroring `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > stalenessThreshold[asset]) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`stalenessThreshold` should be configurable per asset and set to the feed's documented heartbeat interval (e.g., 3600 seconds for a 1-hour heartbeat feed, with a small buffer).

## Proof of Concept
**Foundry fork test outline:**

1. Fork mainnet at a block where a supported LST/ETH Chainlink feed is within its heartbeat.
2. Warp `block.timestamp` forward by `heartbeat + 1` without advancing the feed (simulating oracle downtime). The feed's `updatedAt` is now stale but `latestRoundData()` still returns the last price.
3. Call `LRTDepositPool.depositAsset(lstAddress, 100e18, 0, "")` as an unprivileged attacker who holds 100 LST tokens.
4. Observe that `ChainlinkPriceOracle.getAssetPrice()` returns the stale (inflated) price without reverting.
5. Compute `rsethAmountToMint = (100e18 * stalePrice) / rsETHPrice` — attacker receives rsETH in excess of the fair value of the deposited LST.
6. Assert that the attacker's rsETH balance exceeds the fair-value equivalent, confirming over-minting.

The root cause is exclusively at `contracts/oracles/ChainlinkPriceOracle.sol` line 52, where all staleness fields are silently discarded. [7](#0-6)
