### Title
Incorrect Hardcoded Guard in Protocol Parameter Update Acceptance Silently Rejects Valid Updates — (`File: eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/Ppup.hs`)

---

### Summary

The `votedFuturePParams` function in the Shelley PPUP rule contains an incorrect hardcoded guard that checks `ppMaxTxSize + ppMaxBHSize < ppMaxBBSize`. This check is semantically wrong: block header size (`ppMaxBHSize`) is not part of block body size (`ppMaxBBSize`); they are independent limits on separate block components. As a result, any protocol parameter update that reaches genesis-node quorum but whose proposed values satisfy `ppMaxTxSize + ppMaxBHSize >= ppMaxBBSize` is **silently rejected** — even though the update is otherwise valid and has achieved the required quorum. In the hard-fork scenario, this silent rejection causes consensus to advance the era while the ledger does not, producing permanent ledger divergence.

---

### Finding Description

In `votedFuturePParams`, after computing the new protocol parameters `ppNew` from a quorum-approved update, the following guard is applied:

```haskell
guard $
  toInteger (ppNew ^. ppMaxTxSizeL) + toInteger (ppNew ^. ppMaxBHSizeL)
    < toInteger (ppNew ^. ppMaxBBSizeL)
pure ppNew
``` [1](#0-0) 

The code itself carries an explicit developer acknowledgment that this check is wrong:

> "TODO: Remove this incorrect check from the code and the spec. It is incorrect because block header size is not part of the block body size, therefore this relation makes no sense." [2](#0-1) 

The same incorrect check is mirrored in the formal specification: [3](#0-2) 

The `votedFuturePParams` function is called from `ppupTransitionNonEmpty` (the `PPUP` STS rule, active through Babbage) to compute `sgsFuturePParams`. If the guard returns `Nothing`, the future protocol parameters are set to `PotentialPParamsUpdate Nothing`, meaning the update is dropped at the epoch boundary even though quorum was met. [4](#0-3) 

The correct semantic constraint — that a single transaction must fit inside a block body — is only `ppMaxTxSize < ppMaxBBSize`. Adding `ppMaxBHSize` to the left-hand side is a category error: the block header is a separate serialized structure from the block body, and `chainChecks` enforces their limits independently. [5](#0-4) 

---

### Impact Explanation

**Scenario — Hard Fork Update Silently Rejected:**

The Cardano ADR and errata both document the following split-brain mechanism: consensus decides to enact a hard fork (major protocol version bump) based on the protocol parameter update state two stability windows before the epoch end. The ledger, however, only applies the update at the epoch boundary via the `NEWPP` rule, which calls `votedFuturePParams`. If the incorrect guard rejects the update at that point, consensus changes the era while the ledger does not change the major protocol version — leaving the chain in a permanent split-brain state requiring a hard fork to recover. [6](#0-5) [7](#0-6) 

**Scenario — Silent Rejection of Valid Parameter Update:**

Any quorum-approved update proposing, for example, a large `ppMaxTxSize` (e.g., 90 000 bytes) while `ppMaxBBSize` is 90 112 bytes would be silently dropped because `90000 + 1100 = 91100 > 90112`. The genesis nodes receive no error; the update simply does not take effect. This constitutes an unauthorized modification of the governance outcome — a valid enacted update is suppressed by an incorrect hardcoded check.

---

### Likelihood Explanation

The check is triggered whenever a quorum-approved protocol parameter update proposes values where `ppMaxTxSize + ppMaxBHSize >= ppMaxBBSize`. This is a realistic configuration: if governance ever proposes to raise `ppMaxTxSize` to a value close to `ppMaxBBSize` (a natural desire to allow larger transactions), the incorrect guard silently blocks it. The risk is highest during hard-fork proposals that bundle parameter changes. The developers have already flagged this with a `TODO` and linked it to issue [#4251](https://github.com/IntersectMBO/cardano-ledger/issues/4251), confirming it is a known live defect.

---

### Recommendation

Remove the incorrect guard from `votedFuturePParams` in `eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/Ppup.hs` (lines 236–238). The only semantically correct size constraint — that a transaction must fit within a block body — is already enforced separately. Simultaneously, remove the corresponding condition from the formal specification (`eras/shelley/formal-spec/epoch.tex`, lines 776–778) to keep spec and implementation consistent. [8](#0-7) 

---

### Proof of Concept

1. Construct a `ProposedPPUpdates` signed by a quorum of genesis keys proposing:
   - `ppMaxTxSize = 89000` (valid: fits in a block body of 90112)
   - `ppMaxBHSize = 1100` (unchanged)
   - `ppMaxBBSize = 90112` (unchanged)
2. Submit the update in the correct voting window via the `PPUP` rule.
3. Observe that `votedFuturePParams` evaluates `89000 + 1100 = 90100 < 90112` — this particular example passes. Now change `ppMaxTxSize = 89100`: `89100 + 1100 = 90200 >= 90112` — the `guard` returns `Nothing`.
4. The `sgsFuturePParams` field is set to `PotentialPParamsUpdate Nothing`; at the epoch boundary the `NEWPP` rule finds no update to apply.
5. If the update also contained a major protocol version bump, consensus has already committed to the hard fork, but the ledger has not applied it — permanent divergence. [9](#0-8)
