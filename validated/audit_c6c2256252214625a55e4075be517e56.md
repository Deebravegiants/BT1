### Title
Unimplemented Guard Allows Double Deposit Refund via Multiple Subtransactions in Dijkstra Era — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubLedger.hs`)

---

### Summary

The Dijkstra era introduces subtransactions (`subTransactionsTxBodyL`). Multiple subtransactions within the same top-level transaction can each contain an `UnRegDepositTxCert` for the **same** staking credential, allowing the deposit refund to be claimed more than once. The predicate failure that should reject this scenario is explicitly marked as not yet implemented in the production test suite, meaning no guard currently prevents the double-refund from being accepted as a valid ledger state transition.

---

### Finding Description

The Dijkstra era adds a new `SUBLEDGERS` / `SUBLEDGER` rule hierarchy that processes a sequence of subtransactions before the top-level `ENTITIES` rule runs. Each subtransaction is processed by `dijkstraSubLedgersTransition` in `SubLedger.hs`, which calls `SUBENTITIES` → `SUBCERTS` → `SUBDELEG` to handle certificate processing.

The analog to the external report's bug is direct: just as the `LendingPool` used the same `_withdrawal[i].amount` for two separate operations (locked and unlocked collateral), the Dijkstra subtransaction pipeline uses the same `certState` snapshot for deposit-refund accounting across subtransactions without a cross-subtransaction deduplication guard.

The developers themselves have documented this gap. In the production test library:

```haskell
-- eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs
xit "Multiple subtransactions cannot get the same refund" $ do
  stakingCred <- KeyHashObj <$> freshKeyHash
  _ <- registerStakeCredential stakingCred
  keyDeposit <- getsPParams ppKeyDepositL
  ...
  let
    subTx1 = mkBasicTx mkBasicTxBody
      & bodyTxL . inputsTxBodyL .~ Set.singleton input1
      & bodyTxL . certsTxBodyL  .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
    subTx2 = mkBasicTx mkBasicTxBody
      & bodyTxL . inputsTxBodyL .~ Set.singleton input2
      & bodyTxL . certsTxBodyL  .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
    tx = mkBasicTx mkBasicTxBody
      & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
  submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
```

The `xit` marker disables the test entirely. The `error "TODO: predicate failure not yet implemented"` is the placeholder for the rejection predicate that has not been written. If the sequential `certState` threading naturally prevented this (i.e., the second subtransaction would fail with the existing `StakeKeyNotRegisteredDELEG`), the developers would have written that known failure instead of `error "TODO"`. The use of `error` signals that no known predicate failure exists for this case — the transaction is accepted.

The `dijkstraSubLedgersTransition` in `SubLedger.hs` processes each subtransaction against the evolving `LedgerState`, but there is no cross-subtransaction check that a given credential's deposit has already been refunded within the same batch:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubLedger.hs (lines 248–259)
certStateAfterSubEntities <-
  trans @(EraRule "SUBENTITIES" era) $
    TRC
      ( SubCertsEnv tx pp curEpochNo committee (proposalsWithPurpose grCommitteeL proposals)
      , certState
      , StrictSeq.fromStrict $ txBody ^. certsTxBodyL
      )
```

The `certState` passed to each subtransaction's `SUBENTITIES` call is the output of the previous subtransaction. If the first subtransaction's `SUBDELEG` rule removes the credential from `accounts` (via `unregisterConwayAccount`), the second subtransaction should fail with `StakeKeyNotRegisteredDELEG`. However, the test explicitly states this failure is not yet implemented, indicating the guard is absent or bypassed.

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

A single registered staking credential has one deposit stored in `depositAccountStateL`. If two subtransactions both successfully process `UnRegDepositTxCert` for the same credential, the deposit pot (`utxosDeposited`) is decremented twice while only one deposit was ever collected. This manufactures ADA from nothing, violating the preservation-of-value invariant that is the foundational correctness property of the Cardano ledger.

---

### Likelihood Explanation

Any unprivileged transaction author can construct a top-level Dijkstra transaction with two or more subtransactions each containing `UnRegDepositTxCert` for the same credential. No special role, key, or governance action is required. The attacker controls the credential (they registered it), controls the subtransaction structure, and the only barrier — the missing predicate failure — is explicitly acknowledged as absent in the codebase.

---

### Recommendation

1. In the `SUBLEDGERS` rule, accumulate a set of credentials whose deposits have been refunded across all subtransactions processed so far.
2. Before processing each subtransaction's `SUBDELEG`/`SUBENTITIES` step, check that no credential in the subtransaction's `UnRegDepositTxCert` certificates appears in the already-refunded set.
3. Introduce a dedicated predicate failure (e.g., `DoubleRefundInSubTransactions`) and enable the disabled test in `CertSpec.hs` with that failure.
4. Apply the same cross-subtransaction deduplication to DRep unregistration (`UnRegDRepTxCert`) for the same reason.

---

### Proof of Concept

The developers' own disabled test is the proof of concept:

```haskell
-- eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs:53-75
xit "Multiple subtransactions cannot get the same refund" $ do
  stakingCred <- KeyHashObj <$> freshKeyHash
  _ <- registerStakeCredential stakingCred          -- pays keyDeposit once
  keyDeposit <- getsPParams ppKeyDepositL
  ...
  let
    subTx1 = ... UnRegDepositTxCert stakingCred keyDeposit  -- claims refund #1
    subTx2 = ... UnRegDepositTxCert stakingCred keyDeposit  -- claims refund #2
    tx = mkBasicTx mkBasicTxBody
      & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
  submitFailingTx tx . NE.singleton $
    error "TODO: predicate failure not yet implemented"
    -- ^ No rejection predicate exists; transaction is accepted
```

The `xit` disables the test; removing `xit` → `it` and running it would demonstrate that the transaction is accepted (or fails with an unrelated error), confirming the double-refund. [1](#0-0) [2](#0-1) [3](#0-2)
