### Title
Malicious Harvester Unconditionally Diverts Farmer Block Rewards via Unenforced `farmer_reward_address_override` — (File: chia/farmer/farmer_api.py)

---

### Summary

The Chia farmer accepts a `farmer_reward_address_override` field from any connected harvester in both `NewProofOfSpace` and `RespondSignatures` protocol messages. When set, this field unconditionally replaces the farmer's configured reward address (`farmer_target`) in the `DeclareProofOfSpace` message sent to the full node. The only check performed is a log-only warning; the override is never rejected. A malicious third-party harvester with valid plots can redirect 100% of the farmer's block rewards (0.25 XCH per block) to an arbitrary attacker-controlled address with no cryptographic or protocol-level enforcement stopping it.

---

### Finding Description

**Dual-path reward address control with missing enforcement**

The CHIP-22 fee convention (implemented in this codebase) introduces a second, less-restricted path for setting the farmer reward address: the `farmer_reward_address_override` field in harvester protocol messages. The farmer's own configured address (`farmer_target`) is the "restricted" path — only the farmer operator can change it. The override is the "open" path — any connected harvester can set it to any `bytes32` value.

**Path 1 — Farmer's configured address (restricted):**
The farmer's reward address is set by the operator in config and stored as `self.farmer.farmer_target`.

**Path 2 — Harvester override (unenforced):**

In `FarmerAPI.new_proof_of_space`, when a harvester sends `NewProofOfSpace` with `farmer_reward_address_override` set, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee`:

```python
if new_proof_of_space.farmer_reward_address_override is not None:
    self.farmer.notify_farmer_reward_taken_by_harvester_as_fee(sp, new_proof_of_space)
``` [1](#0-0) 

`notify_farmer_reward_taken_by_harvester_as_fee` checks the fee quality threshold and logs a warning if it is invalid — but **does not reject the override, does not return an error, and does not halt processing**:

```python
else:
    self.log.warning(
        "Harvester illegitimately took reward by failing to provide its fee rate ..."
    )
``` [2](#0-1) 

Then in `_process_respond_signatures`, the override from `RespondSignatures` is accepted unconditionally:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
    include_source_signature_data = True
``` [3](#0-2) 

This `farmer_reward_address` is then placed directly into `DeclareProofOfSpace` and broadcast to the full node:

```python
return farmer_protocol.DeclareProofOfSpace(
    ...
    farmer_reward_address,
    ...
)
``` [4](#0-3) 

The `farmer_reward_address_override` field is defined in both `NewProofOfSpace` and `RespondSignatures` as an optional `bytes32` with no protocol-level constraint: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

**Critical — Unauthorized reward diversion of XCH**

A malicious third-party harvester with valid plots can:

1. Connect to a farmer (a normal, unprivileged operation — harvesters connect over TLS with no special credentials beyond a valid certificate).
2. Send `NewProofOfSpace` with `farmer_reward_address_override` set to the attacker's own puzzle hash.
3. The farmer logs a warning but continues processing.
4. The farmer sends `RequestSignatures` back to the harvester.
5. The harvester responds with `RespondSignatures` also carrying `farmer_reward_address_override` pointing to the attacker's address.
6. The farmer builds `DeclareProofOfSpace` with the attacker's address as `farmer_reward_address` and broadcasts it to the full node.
7. The full node includes the block with the attacker's puzzle hash as the farmer reward recipient.
8. The 0.25 XCH farmer reward is paid to the attacker, not the farmer.

This can be applied to every block the harvester wins, permanently and silently diverting all farmer rewards. The farmer operator has no in-protocol mechanism to prevent or detect this in real time — only post-hoc log inspection.

---

### Likelihood Explanation

Any third-party harvester service (a legitimate business model in Chia) can exploit this. No privileged keys, no admin access, and no cryptographic break are required. The attacker only needs:
- A valid TLS certificate (obtained normally when setting up a harvester).
- Plots that pass the proof-of-space filter (required to win blocks, but this is the harvester's normal function).

The CHIP-22 fee convention was designed to allow harvesters to take a fee, but the farmer has zero enforcement of the fee threshold — only logging. A malicious harvester can set the override to 100% of the reward on every block, not just blocks where the fee quality threshold is legitimately met.

---

### Recommendation

The farmer must enforce the fee threshold as a hard gate, not a soft log. Specifically:

1. In `new_proof_of_space`, if `farmer_reward_address_override` is set and `fee_quality > fee_threshold` (or `fee_info is None`), **reject the proof** and return `None` rather than proceeding.
2. In `_process_respond_signatures`, cross-check that the `farmer_reward_address_override` in `RespondSignatures` matches what was declared in the corresponding `NewProofOfSpace`, and that the fee threshold was valid at the time of the original proof submission.
3. Provide a farmer-side configuration option to disable `farmer_reward_address_override` entirely for operators who do not use third-party harvesters.

---

### Proof of Concept

```python
# Malicious harvester intercepts new_proof_of_space and injects override
async def intercept_farmer_new_proof_of_space(
    self, request: harvester_protocol.NewProofOfSpace, peer: WSChiaConnection
) -> None:
    # Override the farmer reward address to attacker's address
    # fee_info=None means no threshold is provided — farmer only logs a warning
    request = dataclasses.replace(
        request,
        farmer_reward_address_override=ATTACKER_PUZZLE_HASH,
        fee_info=None,  # No fee threshold — farmer logs warning but does NOT reject
    )
    await FarmerAPI.new_proof_of_space(farmer.server.api, request, peer)

# Malicious harvester also injects override in RespondSignatures
async def intercept_harvester_request_signatures(
    self, request: harvester_protocol.RequestSignatures
) -> Message | None:
    result_msg = await HarvesterAPI.request_signatures(harvester.server.api, request)
    response = dataclasses.replace(
        RespondSignatures.from_bytes(result_msg.data),
        farmer_reward_address_override=ATTACKER_PUZZLE_HASH,
    )
    return make_msg(ProtocolMessageTypes.respond_signatures, response)
```
