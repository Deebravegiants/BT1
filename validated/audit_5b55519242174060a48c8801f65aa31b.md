### Title
Broadcast Channel Lag in `SignRequestStorage::get` / `CKDRequestStorage::get` / `VerifyForeignTransactionRequestStorage::get` Causes Permanent Liveness Failure for Targeted Requests — (`crates/node/src/storage.rs`)

### Summary

All three storage `get()` methods subscribe to a `tokio::sync::broadcast` channel of capacity 500, then enter a loop waiting for a matching ID notification. When `RecvError::Lagged` is returned, the code logs and calls `continue`, re-entering `rx.recv().await` **without re-checking RocksDB**. If the target request's notification was among the dropped messages, the `get()` future hangs indefinitely — the request is durably stored in RocksDB but the waiting task never unblocks.

---

### Finding Description

All three storage structs are constructed with a broadcast channel of capacity 500: [1](#0-0) [2](#0-1) [3](#0-2) 

The `get()` pattern for all three is identical. Taking `SignRequestStorage::get` as the canonical example:

1. Subscribe to the broadcast channel.
2. Check RocksDB — if present, return immediately.
3. Enter a loop calling `rx.recv().await`.
4. On `RecvError::Lagged`, log and `continue` — **no RocksDB re-check**.
5. On `Ok(added_id)`, compare to the target; break if equal. [4](#0-3) 

The `Lagged` arm: [5](#0-4) 

Tokio's broadcast semantics: when `Lagged(n)` is returned, the receiver's internal cursor is advanced past the `n` dropped messages. Those messages are **permanently lost** to this receiver. If the target request's `add_sender.send(request.id)` call was one of those dropped messages, no future `recv()` call on this receiver will ever yield that ID. [6](#0-5) 

The same flaw exists identically in `CKDRequestStorage::get` and `VerifyForeignTransactionRequestStorage::get`: [7](#0-6) [8](#0-7) 

---

### Impact Explanation

A follower node calls `get(id)` when it receives a protocol message from the leader for a request its own indexer has not yet indexed. The comment in the code confirms this is the intended use case: [9](#0-8) 

If the follower's `get()` future hangs:
- The follower never participates in the MPC round.
- The leader's round fails (insufficient participants).
- The leader retries up to `MAX_ATTEMPTS_PER_REQUEST_AS_LEADER = 10` times: [10](#0-9) 

- After 10 failed attempts, the request is dropped with `DropReason::MaxAttemptsExceeded`: [11](#0-10) 

The user's signing/CKD/foreign-tx-verification request is **permanently and silently dropped** from the queue. The request remains in RocksDB but is never processed.

---

### Likelihood Explanation

The exploit requires an attacker to submit more than 500 on-chain requests in a burst such that they are indexed by the follower between the moment the follower's `get()` subscribes and the moment the target request's `add()` notification is sent. This is achievable by:

1. Submitting 501+ sign/CKD/verify-foreign-tx requests to the NEAR chain contract (all legitimate, unprivileged calls).
2. Ensuring the target victim request is indexed after the burst.

The 500-message channel capacity is a fixed constant with no backpressure or dynamic sizing. Under realistic mainnet load with many concurrent requests, the window is reachable without any special privileges.

---

### Recommendation

After receiving `RecvError::Lagged`, re-check RocksDB before continuing to wait. The corrected `Lagged` arm should be:

```rust
broadcast::error::RecvError::Lagged(msg_n) => {
    tracing::info!("{msg_n} messages lagged during sign_request channel recv");
    // Re-check DB: the target notification may have been dropped.
    if let Some(request_ser) = self.db.get(DBCol::SignRequest, &key)? {
        return Ok(serde_json::from_slice(&request_ser)?);
    }
    continue;
}
```

Apply the same fix to `CKDRequestStorage::get` and `VerifyForeignTransactionRequestStorage::get`.

### Proof of Concept

```rust
#[tokio::test]
async fn test_lag_causes_get_to_hang() {
    let dir = tempfile::tempdir().unwrap();
    let db = Arc::new(SecretDB::new(dir.path(), [1; 16]).unwrap());
    let storage = Arc::new(SignRequestStorage::new(db.clone()).unwrap());

    // Create the target request (not yet added).
    let target = SignatureRequest { id: CryptoHash(rand::random()), /* ... */ };

    // Spawn get() for the target — it subscribes, finds nothing in DB, enters loop.
    let storage2 = storage.clone();
    let target_id = target.id;
    let get_fut = tokio::spawn(async move { storage2.get(target_id).await });

    // Yield to let get() subscribe and enter the recv loop.
    tokio::task::yield_now().await;

    // Add 501 unrelated requests to overflow the broadcast channel (capacity=500).
    for _ in 0..501 {
        let req = SignatureRequest { id: CryptoHash(rand::random()), /* ... */ };
        storage.add(&req);
    }

    // Now add the target — its notification is sent AFTER the channel overflowed.
    // The get() receiver already lagged; the target's ID was dropped.
    storage.add(&target);

    // get() should resolve, but it hangs — demonstrate with a timeout.
    let result = tokio::time::timeout(
        std::time::Duration::from_secs(2),
        get_fut,
    ).await;

    // This assertion FAILS (timeout), proving the liveness break.
    assert!(result.is_ok(), "get() hung indefinitely despite request being in RocksDB");
}
```

The target request IS present in RocksDB after `storage.add(&target)`, but `get()` never returns because the notification was dropped during the lag window and the `Lagged` handler does not re-check the database. [12](#0-11)

    pub async fn get(&self, id: SignatureId) -> Result<SignatureRequest, anyhow::Error> {
        let key = borsh::to_vec(&id)?;
        let mut rx = self.add_sender.subscribe();
        if let Some(request_ser) = self.db.get(DBCol::SignRequest, &key)? {
            return Ok(serde_json::from_slice(&request_ser)?);
        }
        loop {
            let added_id = match rx.recv().await {
                Ok(added_id) => added_id,
                Err(e) => match e {
                    broadcast::error::RecvError::Closed => {
                        metrics::SIGN_REQUEST_CHANNEL_FAILED.inc();
                        return Err(anyhow::anyhow!("Error in sign_request channel recv, {e}"));
                    }
                    broadcast::error::RecvError::Lagged(msg_n) => {
                        tracing::info!("{msg_n} messages lagged during sign_request channel recv");
                        continue;
                    }
                },
            };
            if added_id == id {
                break;
            }
        }
        let request_ser = self.db.get(DBCol::SignRequest, &key)?.unwrap();
        Ok(serde_json::from_slice(&request_ser)?)
    }
