Audit Report

## Title
Asymmetric ETH Deposit-Limit Check Allows Limit Overshoot and Blocks All Subsequent ETH Deposits — (`contracts/LRTDepositPool.sol`)

## Summary

`_checkIfDepositAmountExceedesCurrentLimit` uses `totalAssetDeposits > depositLimit` for ETH but `totalAssetDeposits + amount > depositLimit` for ERC20 tokens. The missing `+ amount` in the ETH branch allows a single deposit to push total deposits above the configured cap, after which every subsequent ETH deposit reverts with `MaximumDepositLimitReached` until an admin manually raises the limit.

## Finding Description

In `contracts/LRTDepositPool.sol` at lines 676–682:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← no + amount
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [1](#0-0) 

The ETH branch evaluates only the *current* total against the limit, not the total *after* the incoming deposit. This means:

- When `totalAssetDeposits == depositLimit`, the check returns `false` and the deposit proceeds, pushing total deposits to `depositLimit + msg.value`.
- Every subsequent call evaluates `depositLimit + msg.value > depositLimit` → `true` → `MaximumDepositLimitReached` revert.

The check at lines 661–663 that calls this function and reverts on `true` is correct in structure; the flaw is entirely in the comparison logic of the ETH branch. [2](#0-1) 

## Impact Explanation

**Medium — Temporary freezing of funds (new ETH deposits).**

Once the overshoot occurs, all new ETH deposits are blocked at the protocol level until a MANAGER-role account calls `updateAssetDepositLimit` to raise the cap. The depositor who triggered the overshoot received rsETH and retains redemption rights; ETH already in the pool remains accessible to operators. The impact is a protocol-wide DoS on the ETH deposit path of indefinite duration (no automatic recovery), combined with a silent violation of the configured deposit cap.

## Likelihood Explanation

Moderate. The precondition `totalAssetDeposits == depositLimit` is reachable in normal operation: the cap is a governance-set round number and deposits accumulate toward it. No privileged role, front-running, or external dependency is required — any ordinary user calling `depositETH` at the right moment triggers it. The resulting broken state is sticky and requires manual admin intervention to resolve.

## Recommendation

Add `amount` to the ETH branch to match the ERC20 logic:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [3](#0-2) 

## Proof of Concept

```solidity
// Foundry fork test
function test_ethDepositLimitOvershoot() public {
    uint256 limit = lrtConfig.depositLimitByAsset(LRTConstants.ETH_TOKEN);

    // Arrange: fill pool to exactly the limit via prior deposits
    // (use vm.store or a helper to set getTotalAssetDeposits == limit)

    // Step 1: deposit 1 wei — should revert MaximumDepositLimitReached but does NOT
    depositPool.depositETH{value: 1}(0, "");

    // Step 2: verify overshoot
    assertGt(depositPool.getTotalAssetDeposits(LRTConstants.ETH_TOKEN), limit);

    // Step 3: any subsequent ETH deposit is now permanently blocked
    vm.expectRevert(ILRTDepositPool.MaximumDepositLimitReached.selector);
    depositPool.depositETH{value: 1 ether}(0, "");
}
```
