### Title
Small-Prime Factor Bypass in `check_integer_with_small_primes` Allows Malicious Paillier Modulus Acceptance — (`include-internal/cbmpc/internal/zk/small_primes.h`, `src/cbmpc/zk/zk_paillier.cpp`)

### Summary
`check_integer_with_small_primes` stops testing divisors when `small_prime > alpha` where `alpha = 8192`. The `small_primes[]` array extends to 104,729. Primes in the range (8192, 104729] — starting with 8209 — are present in the array but are never tested. A malicious protocol peer can construct N = p·q with p = 8209 (known to them), generate a valid `valid_paillier_t` ZK proof (since they know phi(N)), and cause the honest verifier to accept a structurally weak Paillier modulus. The attacker then decrypts any ciphertext the honest party encrypts under N, recovering private key material.

### Finding Description
In `include-internal/cbmpc/internal/zk/small_primes.h`, `check_integer_with_small_primes` iterates over `small_primes[]` and breaks when `small_prime > alpha`:

```cpp
static error_t check_integer_with_small_primes(const bn_t& prime, int alpha) {
  for (int i = 0; i < small_primes_count; i++) {
    int small_prime = small_primes[i];
    if (small_prime > alpha) break;          // stops at 8192
    if (mod_t::mod(prime, small_prime) == 0) return coinbase::error(E_CRYPTO);
  }
  return SUCCESS;
}
```

`param_t::alpha = 1 << 13 = 8192`. The array contains 10,000 primes up to 104,729. The 1028th entry is 8191, the 1029th is 8209. Every prime from 8209 to 104,729 is silently skipped.

All callers pass `param::alpha` (= 8192):
- `valid_paillier_t::verify` (line 34)
- `valid_paillier_interactive_t::verify` (line 87)
- `paillier_zero_t::verify` (line 147)
- `two_paillier_equal_t::verify` (lines 254, 258)
- `pdl_t::verify` (line 498)
- `paillier_pedersen_equal_t::verify` (line 339)

### Impact Explanation
A malicious party (Byzantine peer in 2P ECDSA, DKG, or refresh) generates N = 8209 · q where q is a large prime. They know phi(N) = 8208 · (q-1) and compute N^{-1} mod phi(N). They produce valid sigma values for `valid_paillier_t::verify`. The honest party accepts N, sets `paillier_valid_key = verified` and `paillier_no_small_factors = verified`, then encrypts their ECDSA private key share under N. The attacker decrypts using phi(N), recovering the honest party's key share. This constitutes full private key recovery — Critical impact.

### Likelihood Explanation
The attacker is a single malicious peer in a 2P protocol. No threshold collusion is required. The construction of N = 8209 · q is trivial. The only probabilistic obstacle is the `coprime(rho_prod, N)` check, which fails with probability ~1/8209; the attacker can iterate over different q values until it passes. The attack is fully deterministic once a suitable N is found.

### Recommendation
Change the `alpha` cutoff passed to `check_integer_with_small_primes` to cover the full array, or replace the `alpha` parameter with a check against all 10,000 entries unconditionally. The simplest fix is to remove the early-break condition and always iterate the full array, or to set the cutoff to `small_primes[small_primes_count - 1]` (104,729). Alternatively, align `param_t::alpha` with the actual array maximum.

### Proof of Concept

   expected ~1 retry per 8209 candidates).
3. Set N = p * q. Compute phi_N = (p-1)*(q-1). Compute N_inv = N^{-1} mod phi_N.
4. For each i in [0, t): compute rho[i] = drbg(hash(N, sid, aux)).gen_bn(N);

   => verify() returns SUCCESS; paillier_valid_key = verified; paillier_no_small_factors = verified.
6. Honest party encrypts secret x under N: c = Enc_N(x).
