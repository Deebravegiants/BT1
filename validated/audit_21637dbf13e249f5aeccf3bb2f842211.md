### Title
Unverified CKD Response for Legacy `AppPublicKey` Requests Enables Single Byzantine Participant to Forge Confidential Key Derivation Output - (File: crates/contract/src/lib.rs)

### Summary

The `respond_ckd` function in the MPC contract applies on-chain cryptographic verification of the CKD output **only** for the `AppPublicKeyPV` (publicly verifiable) variant. For the legacy `AppPublicKey` variant, the response is accepted and delivered to the caller with **zero on-chain verification**. A single Byzantine attested participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary forged `CKDResponse` for any pending `AppPublicKey` request and have it delivered to the user as if it were the legitimate MPC output.

### Finding Description

In `respond_ckd` (lines 653–689 of `crates/contract/src/lib.rs`), after confirming the caller is an attested participant, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKey`, the arm is a no-op. The contract then unconditionally calls `resolve_yields_for`, which removes the pending request from the map and delivers the attacker-supplied `response` bytes to every queued yield for that request key. [2](#0-1) 

Contrast this with `respond` (signature path), which always runs `verify_ecdsa_signature` / `verify_eddsa_signature` and returns `Err(InvalidSignature)` on failure: [3](#0-2) 

The `ckd_output_check` function verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which is only computable on-chain when `pk2` (the G2 component) is present — i.e., only for `AppPublicKeyPV`. [4](#0-3) 

For `AppPublicKey` (a single G1 point), the contract has no way to run the pairing check without the app's secret scalar, so the check is omitted entirely. However, the contract still accepts and delivers whatever `CKDResponse` the responding participant supplies.

**Exploit path:**

1. A user submits `request_app_private_key` with `AppPublicKey` (the legacy variant), attaching 1 yoctonear.
2. The request is stored in `pending_ckd_requests` under the derived `CKDRequest` key.
3. A single Byzantine attested participant calls `respond_ckd(request, forged_response)` where `forged_response` contains attacker-chosen `big_y` and `big_c` values.
4. The contract passes all checks (participant is attested, protocol is running, domain is BLS12-381) and calls `resolve_yields_for`, which delivers the forged bytes to the user's yield-resume promise.
5. The user's transaction receives the forged `CKDResponse` as if it were the legitimate MPC output. [5](#0-4) 

### Impact Explanation

The user receives an attacker-controlled confidential key derivation output. If the user subsequently uses the derived key material (e.g., to encrypt secrets, derive a wallet key, or authenticate to a service), the attacker — who chose `big_y` and `big_c` — may be able to recover or control the derived secret. This constitutes **unauthorized confidential key derivation output without the required participant authorization** (the threshold is t-of-n; the attack requires only 1 Byzantine participant). This maps directly to the Critical impact category: "Unauthorized … confidential key derivation output without the required participant authorization."

### Likelihood Explanation

The `AppPublicKey` (legacy) variant is still accepted by both `request_app_private_key` and `respond_ckd` in production. Any single attested participant who turns Byzantine can execute this attack against every in-flight `AppPublicKey` CKD request. No collusion above threshold is required. The attacker needs only to be an active, attested participant — a realistic assumption for a Byzantine node strictly below the signing threshold.

### Recommendation

Apply the same response-verification discipline used in `respond` to `respond_ckd` for all request variants. For `AppPublicKey`, on-chain pairing verification is not possible without the app's G2 component; therefore, the contract should either:

1. **Deprecate and reject** `AppPublicKey` requests in `respond_ckd`, requiring all new requests to use `AppPublicKeyPV` (which supports on-chain verification via `ckd_output_check`), or
2. **Require threshold-many matching responses** before delivering a CKD result for `AppPublicKey` requests, so that a single Byzantine participant cannot unilaterally forge the output.

### Proof of Concept

```
// Attacker is an attested participant (account: "byzantine.near")
// Victim submitted: request_app_private_key({ app_public_key: AppPublicKey(pk1), ... })
// CKDRequest key is deterministic from (app_public_key, domain_id, predecessor, path)

// Attacker constructs a forged response with arbitrary big_y, big_c:
let forged_response = CKDResponse {
    big_y: Bls12381G1PublicKey([0xAA; 48]),  // attacker-chosen
    big_c: Bls12381G1PublicKey([0xBB; 48]),  // attacker-chosen
};

// Attacker calls respond_ckd — passes all checks, no pairing verification runs:
contract.respond_ckd(victim_ckd_request, forged_response);
// → resolve_yields_for delivers forged bytes to victim's pending yield
// → victim's transaction resolves with attacker-controlled CKDResponse
```

The `AppPublicKey` arm in `respond_ckd` is a silent no-op, directly analogous to the Pods Finance vault's `shares = 0` path: a structural bypass that allows a meaningful output (the CKD result) to be delivered without the cryptographic accounting (the pairing check) that should gate it. [6](#0-5)

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
