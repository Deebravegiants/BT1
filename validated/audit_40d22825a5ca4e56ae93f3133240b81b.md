### Title
Unprotected `init` Allows Any Caller to Front-Run Contract Initialization with Attacker-Controlled Participants - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `init` function in the NEAR MPC contract lacks access controls. Unlike `init_running` (which carries `#[private]`), `init` can be called by any NEAR account before the legitimate deployer does so. An attacker who monitors the chain for contract deployment can front-run the deployer's `init` call, seeding the contract with attacker-controlled participant accounts and a threshold of 1. Because `init` bootstraps participants with `TeeState::with_mocked_participant_attestations`, those attacker-controlled accounts immediately pass the contract's attestation checks, allowing the attacker to respond to every `sign`, `respond_ckd`, and `respond_verify_foreign_tx` request with attacker-generated output.

---

### Finding Description

`init` is the one-time bootstrap entry point for the MPC contract. It is decorated with `#[init]` (preventing re-initialization) but **not** with `#[private]` (which would restrict callers to the contract account itself). [1](#0-0) 

Compare with `init_running`, which carries both attributes: [2](#0-1) 

The test suite explicitly verifies that `init_running` rejects external callers: [3](#0-2) 

No equivalent test or guard exists for `init`.

Inside `init`, all supplied participants are immediately granted mocked-valid attestations: [4](#0-3) 

The inline TODO acknowledges this is a known gap:

> `// TODO(#1087): Every participant must have a valid attestation, otherwise we risk participants being immediately kicked out once contract transitions into running.`

Because `accept_requests` is set to `true` and the mocked attestations satisfy `assert_caller_is_attested_participant_and_protocol_active`, the attacker's nodes can immediately call `respond`, `respond_ckd`, and `respond_verify_foreign_tx`. [5](#0-4) 

The production deployment flow calls `init` as a **separate transaction** from contract deployment, creating the front-running window: [6](#0-5) 

---

### Impact Explanation

**Critical — Unauthorized threshold signature issuance and bypass of threshold-signature requirements.**

An attacker who wins the race to call `init` becomes the sole controller of the MPC network:

1. The attacker supplies their own NEAR accounts as participants with `threshold = 1`.
2. `with_mocked_participant_attestations` grants those accounts valid-looking attestation state, bypassing TEE checks.
3. The attacker's node calls `vote_add_domains` to register signing domains.
4. Every user `sign()` request is answered by the attacker's node with a signature produced by attacker-controlled key material.
5. The attacker can sign arbitrary foreign-chain transactions (Bitcoin, Ethereum, Solana) on behalf of any user who calls the contract.
6. The legitimate deployer's subsequent `init` call fails ("contract already initialized"), and the contract cannot be re-initialized without a full redeployment.

---

### Likelihood Explanation

**High.** NEAR block times are ~1 second. The deployment and initialization are separate transactions (confirmed by the localnet and TEE-localnet documentation). Any attacker watching the chain for a new MPC contract deployment can submit their `init` call in the immediately following block. No privileged access, leaked keys, or TEE capabilities are required — only a funded NEAR account and knowledge of the contract's ABI (which is public).

---

### Recommendation

Add `#[private]` to `init`, matching the protection already applied to `init_running`:

```rust
#[private]
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> { ... }
```

Alternatively, perform deployment and initialization atomically in a single NEAR transaction using `DeployContractAction` followed immediately by `FunctionCallAction` for `init`, so no uninitialized window exists on-chain.

Long-term, resolve `TODO(#1087)` so that initial participants must supply real TEE attestations rather than receiving mocked ones, eliminating the bootstrapping trust gap.

---

### Proof of Concept

1. Legitimate deployer submits `DeployContractAction` for the MPC WASM to `mpc-contract.near`. Contract state is `NotInitialized`.
2. Attacker observes the deployment receipt on-chain and immediately submits:
   ```json
   {
     "parameters": {
       "participants": { "attacker.near": { ... } },
       "threshold": 1
     }
   }
   ```
   to `mpc-contract.near::init`.
3. Attacker's `init` executes first (next block). Contract transitions to `Running` with `attacker.near` as the sole participant, holding mocked-valid attestations and `accept_requests: true`.
4. Legitimate deployer's `init` call arrives and panics: contract is already initialized.
5. Attacker calls `vote_add_domains` (as the sole participant, threshold = 1, vote passes immediately) to register a Secp256k1 signing domain.
6. Any user calling `sign()` on the contract receives a signature produced by the attacker's key material. The attacker can use this to authorize arbitrary transactions on Bitcoin, Ethereum, or any other supported foreign chain.
