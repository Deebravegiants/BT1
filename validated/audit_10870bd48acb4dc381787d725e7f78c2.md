### Title
Silent Floor-to-Zero in `ethValueInWithdrawal` Causes Converter Assets to Vanish from Protocol Accounting — (`contracts/LRTConverter.sol`)

---

### Summary

`LRTConverter.ethValueInWithdrawal` is a single scalar that is the **sole accounting mechanism** for all assets held in the converter. The floor-to-zero subtraction in `transferAssetToDepositPool` can silently set it to `0` while assets from a *different* asset type remain in the converter, making those assets completely invisible to `getETHDistributionData()` and causing `rsETHPrice` to be understated.

---

### Finding Description

`ethValueInWithdrawal` is incremented per-asset when assets leave the deposit pool: [1](#0-0) 

It is decremented with a floor at zero when assets return: [2](#0-1) 

This single number tracks the combined ETH value of **all** assets currently in the converter. It is the only source of truth for converter holdings: [3](#0-2) [4](#0-3) 

**Concrete scenario (no malicious intent required):**

| Step | Action | `ethValueInWithdrawal` |
|------|--------|----------------------|
| 1 | `transferAssetFromDepositPool(stETH, 100e18)` at oracle price 1.0 | `100e18` |
| 2 | `transferAssetFromDepositPool(cbETH, 100e18)` at oracle price 1.0 | `200e18` |
| 3 | stETH oracle price updates to 2.1 | `200e18` |
| 4 | `transferAssetToDepositPool(stETH, 100e18)` → `assetValue = 210e18 > 200e18` | **`0`** (floored) |

After step 4, the converter still holds 100 cbETH (worth ~100 ETH), but `ethValueInWithdrawal = 0`. `getETHDistributionData()` returns `ethLyingInConverter = 0`. The 100 cbETH is completely invisible to `_getTotalEthInProtocol()`. [5](#0-4) 

`rsETHPrice` is then computed as `(totalETHInProtocol - fee) / rsethSupply` with `totalETHInProtocol` understated by ~100 ETH. [6](#0-5) 

---

### Impact Explanation

While the cbETH is not permanently lost (it will be re-counted when eventually transferred back), during the window:

- `rsETHPrice` is understated
- New depositors receive more rsETH than the true backing ratio warrants, diluting existing holders
- The protocol's reported TVL is incorrect

This matches **Low — Contract fails to deliver promised returns, but doesn't lose value**.

---

### Likelihood Explanation

- Requires two or more assets to be in the converter simultaneously — a normal operational state when the protocol is actively managing multiple LSTs
- Requires one asset's oracle price to appreciate enough that its return value exceeds the total `ethValueInWithdrawal` — realistic for LSTs over time (stETH accrues staking rewards continuously)
- The `ASSET_TRANSFER_ROLE` operator does not need to act maliciously; the bug fires through a routine "return assets to pool" call
- No external dependency or governance action is needed beyond normal price oracle updates [7](#0-6) [8](#0-7) 

---

### Recommendation

Track `ethValueInWithdrawal` **per asset** rather than as a single aggregate, and revert (or cap the decrement) if the returned value would exceed what was recorded for that asset:

```solidity
mapping(address => uint256) public ethValueInWithdrawalByAsset;

// in transferAssetFromDepositPool:
ethValueInWithdrawalByAsset[_asset] += assetValue;

// in transferAssetToDepositPool:
uint256 recorded = ethValueInWithdrawalByAsset[_asset];
uint256 decrement = assetValue > recorded ? recorded : assetValue;
ethValueInWithdrawalByAsset[_asset] -= decrement;
```

Alternatively, revert if `assetValue > ethValueInWithdrawal` to surface the accounting inconsistency rather than silently masking it.

---

### Proof of Concept

```solidity
// Local fork test (no mainnet)
function test_floorToZeroMasksConverterAssets() public {
    // Setup: two supported assets, oracle price = 1e18 each
    uint256 A = 100e18;

    // Step 1: move stETH and cbETH to converter
    vm.prank(assetTransferRole);
    converter.transferAssetFromDepositPool(stETH, A);   // ethValueInWithdrawal = 100e18
    vm.prank(assetTransferRole);
    converter.transferAssetFromDepositPool(cbETH, A);   // ethValueInWithdrawal = 200e18

    // Step 2: stETH oracle price appreciates to 2.1e18
    mockOracle.setPrice(stETH, 2.1e18);

    // Step 3: return stETH — assetValue = 210e18 > 200e18 → floors to 0
    vm.prank(assetTransferRole);
    converter.transferAssetToDepositPool(stETH, A);

    assertEq(converter.ethValueInWithdrawal(), 0);  // cbETH still in converter!

    // Step 4: oracle price update understates rsETHPrice
    oracle.updateRSETHPrice();
    uint256 price = oracle.rsETHPrice();

    // True backing includes 100 cbETH @ 1e18 = 100 ETH, but it is invisible
    // price is lower than the true backing ratio
    assertTrue(price < trueBackingRatio);
}
```
