Audit Report

## Title
Permissionless `sendFunds()` With No Caller Incentive Enables Yield Dilution of Existing rsETH Holders — (File: contracts/FeeReceiver.sol)

## Summary
`FeeReceiver.sendFunds()` has no access control and no caller reward, so MEV/execution-layer rewards can accumulate in `FeeReceiver` indefinitely without being reflected in the protocol's TVL. Because `LRTOracle.rsETHPrice` is a stored state variable updated only on explicit calls to `updateRSETHPrice()`, and because `_getTotalEthInProtocol()` never includes `FeeReceiver.balance`, an attacker can call `updateRSETHPrice()` while rewards sit uncollected, lock in an understated price, then deposit to mint excess rsETH — permanently diluting the yield owed to existing holders.

## Finding Description
`FeeReceiver.sendFunds()` is entirely permissionless:

```solidity
// contracts/FeeReceiver.sol L53-58
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

`getETHDistributionData()` explicitly excludes `FeeReceiver.balance` from TVL:

```solidity
// contracts/LRTDepositPool.sol L464-466
/// @dev rewards are not accounted here
/// it will automatically be accounted once it is moved from feeReceiver/rewardReceiver to depositPool
```

`_getTotalEthInProtocol()` sums only `getTotalAssetDeposits()` across supported assets, which routes through `getETHDistributionData()` for ETH and never reads `FeeReceiver.balance`:

```solidity
// contracts/LRTOracle.sol L341-343
uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
```

`rsETHPrice` is a stored state variable updated only when `updateRSETHPrice()` is called. `updateRSETHPrice()` is public and callable by any unprivileged user when the protocol is not paused:

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

`getRsETHAmountToMint()` reads the stored `rsETHPrice` directly:

```solidity
// contracts/LRTDepositPool.sol L519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

**Exploit path (all steps callable by an unprivileged external user):**
1. Observe `FeeReceiver.balance` growing (MEV rewards arriving).
2. Call `LRTOracle.updateRSETHPrice()` — price is computed without the accumulated rewards, storing an understated value in `rsETHPrice`.
3. Call `LRTDepositPool.depositETH{value: X}(0, "")` — `getRsETHAmountToMint()` divides by the understated `rsETHPrice`, minting excess rsETH.
4. Call `FeeReceiver.sendFunds()` — rewards enter TVL.
5. Call `LRTOracle.updateRSETHPrice()` — price rises, but the excess rsETH minted in step 3 permanently dilutes existing holders' share of the reward.

No existing guard prevents this: `depositETH` only enforces a `minRSETHAmountExpected` floor (protecting the depositor from receiving *less*, not from receiving *more*), and there is no check that `rsETHPrice` was computed with a fully-flushed `FeeReceiver`.

## Impact Explanation
**High — Theft of unclaimed yield.** Existing rsETH holders permanently lose a portion of MEV/execution-layer yield to a depositor who enters while rewards are stranded in `FeeReceiver`. The dilution is irreversible once the excess rsETH is minted. The magnitude scales with the ratio of uncollected rewards to TVL and the deposit volume during the delay window. This matches the allowed impact: *High. Theft of unclaimed yield.*

## Likelihood Explanation
**Low-Medium.** The protocol team is expected to call `sendFunds()` periodically off-chain, but there is no on-chain enforcement, no keeper incentive, and no minimum call frequency. A sophisticated depositor who monitors `FeeReceiver.balance` can time a large deposit to exploit the window. The attacker bears no capital risk beyond normal deposit gas costs and can repeat the attack whenever rewards accumulate. During periods of high MEV activity, the reward-to-TVL ratio can be significant enough to make the attack economically worthwhile.

## Recommendation
1. **Auto-flush on deposit:** At the start of `LRTDepositPool.depositETH()` and `depositAsset()`, call `FeeReceiver.sendFunds()` (or an equivalent internal flush) so rewards are always counted in TVL before new rsETH is minted.
2. **Caller incentive:** Reward the `sendFunds()` caller with a small fixed percentage (e.g., 0.1–0.5%) of the forwarded balance, analogous to liquidation bonuses in lending protocols, to create on-chain incentive for timely flushing.
3. **Keeper automation:** Deploy a Gelato/Chainlink Automation keeper that calls `sendFunds()` whenever `FeeReceiver.balance` exceeds a configurable threshold, funded by a portion of the rewards.

## Proof of Concept
Foundry fork test sequence:

```solidity
// 1. Simulate MEV rewards arriving in FeeReceiver
vm.deal(address(feeReceiver), 10 ether);

// 2. Attacker calls updateRSETHPrice() — price excludes the 10 ETH
lrtOracle.updateRSETHPrice();
uint256 understatedPrice = lrtOracle.rsETHPrice();

// 3. Attacker deposits 100 ETH — receives excess rsETH at understated price
uint256 rsethBefore = rsETH.balanceOf(attacker);
lrtDepositPool.depositETH{value: 100 ether}(0, "");
uint256 excessRseth = rsETH.balanceOf(attacker) - rsethBefore;
// excessRseth > 100 ether * 1e18 / fairPrice

// 4. Anyone calls sendFunds() — 10 ETH enters TVL
feeReceiver.sendFunds();

// 5. Price update — price rises, but excess rsETH already minted
lrtOracle.updateRSETHPrice();
uint256 newPrice = lrtOracle.rsETHPrice();
assertGt(newPrice, understatedPrice);

// 6. Assert existing holders' effective ETH value per rsETH is lower than
//    it would have been had sendFunds() been called before the deposit
```

The test demonstrates that `excessRseth` is strictly greater than zero and that the final price is lower than the price that would have resulted had `sendFunds()` been called before step 3, confirming permanent yield dilution for pre-existing holders.
