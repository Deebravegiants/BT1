### Title
Missing Cryptographic Output Verification for `AppPublicKey` CKD Responses Allows Single Byzantine Participant to Deliver Malicious Key Material - (`File: crates/contract/src/lib.rs`)

### Summary

`respond_ckd` in `MpcContract` applies a BLS12-381 pairing-based output check (`ckd_output_check`) only for `AppPublicKeyPV` requests, while silently accepting any arbitrary `CKDResponse` for the legacy `AppPublicKey` variant. A single Byzantine attested participant acting as the computation leader can call `respond_ckd` with a fully attacker-controlled `(big_y, big_c)` pair for any pending `AppPublicKey` request, bypassing the threshold protocol entirely. The contract resolves the yield and delivers the malicious key material to the user with no on-chain recourse.

### Finding Description

In `respond_ckd`, the match on `request.app_public_key` has an explicit empty arm for `AppPublicKey`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the BLS pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, cryptographically binding the response to the network's master public key and the user's ephemeral key pair. [2](#0-1) 

For `AppPublicKey`, no analogous check exists. The contract proceeds directly to `pending_requests::resolve_yields_for`, serializing and delivering whatever `(big_y, big_c)` the caller supplied. [3](#0-2) 

The `AppPublicKey` variant is still actively accepted at `request_app_private_key` (described as "legacy" but not gated or deprecated): [4](#0-3) 

The CKD design document explicitly states the security requirement that is violated: *"No single node in the MPC network should be capable of computing s. This avoids key leakage in the case a single TEE is compromised."* [5](#0-4) 

The contrast with `respond` (ECDSA/EdDSA) is instructive — that function always verifies the signature against the derived public key before resolving: [6](#0-5) 

This is the direct analog of `_mint()` vs `_safeMint()`: `AppPublicKeyPV` is the "safe" path with the output check; `AppPublicKey` is the "unsafe" path that skips it.

### Impact Explanation

A Byzantine leader node submits `big_y = G1` (the generator, a valid non-identity point) and `big_c = K` for any attacker-chosen point `K`. The user decrypts `s' = big_c - a · big_y = K - a · G1`. The attacker does not know `a`, so they do not know `s'` directly. However, they can also submit `big_y = 0` (the identity point, which is a valid compressed G1 encoding) and `big_c = K`, causing the user to compute `s' = K - a · 0 = K`. Now the attacker knows `K = s'` and can compute `KDF(K)` — the user's derived key — breaking the "only known by app" confidentiality requirement.

Even in the non-identity case, the user receives incorrect key material with no on-chain mechanism to detect the corruption. The user's deposit is consumed, the pending request is cleared, and the yield is resolved with the malicious payload. The safety invariant that the delivered CKD output is the correct output of the threshold protocol is broken by a single participant.

### Likelihood Explanation

The attacker must be a single attested participant (TEE-verified node) acting as the computation leader for the targeted request. This is a realistic Byzantine-below-threshold scenario: one compromised or malicious node in the participant set suffices. The `AppPublicKey` variant remains the default legacy path used by the example CLI and existing integrations. [7](#0-6) 

### Recommendation

1. **Deprecate `AppPublicKey`** and require all new requests to use `AppPublicKeyPV`, which has the on-chain pairing check. Emit a warning or reject `AppPublicKey` submissions at `request_app_private_key`.
2. **If `AppPublicKey` must be retained**, add an equivalent output check. Since `AppPublicKey` provides only a G1 point (`pk1 = a·G1`) without the G2 counterpart needed for the pairing equation, the check cannot be applied directly. The minimal fix is to require callers to also supply `pk2 = a·G2` (i.e., upgrade to `AppPublicKeyPV`) so `ckd_output_check` can be applied uniformly in `respond_ckd`.
3. **Mirror the pattern from `respond`**: just as `respond` always verifies the signature before resolving, `respond_ckd` should always verify the CKD output before resolving, regardless of the key variant.

### Proof of Concept

1. User calls `request_app_private_key` with `AppPublicKey(pk1)` where `pk1 = a·G1`.
2. Byzantine leader node (single attested participant) calls `respond_ckd` with:
   - `request`: the matching `CKDRequest` (publicly visible on-chain)
   - `response`: `CKDResponse { big_y: G1_identity, big_c: K }` for attacker-chosen `K`
3. The `AppPublicKey(_) => {}` arm executes — no check runs.
4. `resolve_yields_for` resolves the yield with `serde_json::to_vec(&response)`.
5. The user's `return_ck_and_clean_state_on_success` callback fires, returning `K` as `big_c` and the identity as `big_y`.
6. User computes `s' = K - a · 0 = K`. The attacker knows `K` and therefore `KDF(K)`.

The existing test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` already demonstrates that `respond_ckd` accepts `CKDResponse { big_y: [1u8;48], big_c: [2u8;48] }` — completely arbitrary byte arrays — for an `AppPublicKey` request without any rejection: [8](#0-7)

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
