### Title
No minimum withdrawal amount enforced by default causes dust withdrawal requests where gas cost exceeds asset value, permanently trapping user funds - (File: `contracts/LRTWithdrawalManager.sol`)

### Summary
`minRsEthAmountToWithdraw[asset]` defaults to zero for every asset in `LRTWithdrawalManager`. Because the guard condition is `rsETHUnstaked < minRsEthAmountToWithdraw[asset]`, any non-zero rsETH amount passes when the minimum is zero. Users can therefore queue withdrawal requests for arbitrarily small amounts. Once the operator calls `unlockQueue`, the rsETH is burned from the contract. The user must then call `completeWithdrawal` to receive the corresponding tiny asset amount. When the gas cost of `completeWithdrawal` exceeds the asset value, the user has no economic incentive to claim, leaving the asset permanently stranded in the withdrawal manager and blocking `sweepRemainingAssets` for that asset.

### Finding Description
`LRTWithdrawalManager` stores the per-asset minimum as a mapping that is never initialised in `initialize`:

```solidity
// LRTWithdrawalManager.sol line 35
mapping(address asset => uint256) public minRsEthAmountToWithdraw;
```

The guard in `initiateWithdrawal` is:

```solidity
// LRTWithdrawalManager.sol line 162-164
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```

When `minRsEthAmountToWithdraw[asset] == 0` (the default), the second condition is `rsETHUnstaked < 0`, which is always false for `uint256`. Any non-zero rsETH amount is accepted.

The withdrawal lifecycle then proceeds:
1. `initiateWithdrawal` — rsETH is transferred from the user to the withdrawal manager and `assetsCommitted` is increased.
2. `unlockQueue` (operator-only) — rsETH held by the contract is **burned** and the corresponding asset is pulled from `LRTUnstakingVault` into the withdrawal manager.
3. `completeWithdrawal` — the user must call this to receive the asset.

For a dust request (e.g., 1 wei rsETH → 1 wei ETH), step 2 permanently destroys the user's rsETH. Step 3 requires a transaction that costs tens of thousands of gas on Ethereum mainnet (≫ 1 wei in value). The user has no economic incentive to call `completeWithdrawal`, so the tiny asset amount remains in the withdrawal manager indefinitely.

`sweepRemainingAssets` cannot recover these stranded assets because it reverts when `unlockedWithdrawalsCount[asset] > 0`:

```solidity
// LRTWithdrawalManager.sol line 403
if (hasUnlockedWithdrawals(asset)) revert PendingWithdrawalsExist();
```

`unlockedWithdrawalsCount` is incremented in `_unlockWithdrawalRequests` for every unlocked request and decremented only in `_processWithdrawalCompletion`. Unclaimed dust requests keep the counter positive, permanently blocking the sweep path.

The setter exists but is never called during deployment:

```solidity
// LRTWithdrawalManager.sol line 330-333
function setMinRsEthAmountToWithdraw(address asset, uint256 minRsEthAmountToWithdraw_) external onlyLRTAdmin {
    minRsEthAmountToWithdraw[asset] = minRsEthAmountToWithdraw_;
    emit MinAmountToWithdrawUpdated(asset, minRsEthAmountToWithdraw_);
}
```

### Impact Explanation
Users who create dust withdrawal requests have their rsETH burned by the operator's `unlockQueue` call, but receive no economic benefit because the gas cost of `completeWithdrawal` exceeds the asset value. The corresponding tiny asset amount is stranded in the withdrawal manager and cannot be swept to the treasury while any unclaimed dust withdrawal exists. This matches **Low — Contract fails to deliver promised returns**: the protocol burns the user's rsETH but cannot economically deliver the promised underlying asset.

### Likelihood Explanation
The minimum is zero by default and must be explicitly set by the LRT admin for each supported asset. Any user can call `initiateWithdrawal` with a tiny amount at any time without admin action. The condition is reachable by any unprivileged rsETH holder on the supported withdrawal path.

### Recommendation
Enforce a non-zero minimum in `initialize` for each supported asset, or revert in `setMinRsEthAmountToWithdraw` if the caller attempts to set the minimum to zero. A reasonable floor (e.g., 0.001 ETH worth of rsETH) ensures that the gas cost of `completeWithdrawal` is always economically justified.

```solidity
// In initialize or a dedicated initializer:
for (uint256 i; i < supportedAssets.length; i++) {
    minRsEthAmountToWithdraw[supportedAssets[i]] = MIN_WITHDRAWAL_FLOOR; // e.g. 1e15
}
```

Additionally, add a zero-check in the setter:

```solidity
function setMinRsEthAmountToWithdraw(address asset, uint256 minRsEthAmountToWithdraw_) external onlyLRTAdmin {
    if (minRsEthAmountToWithdraw_ == 0) revert MinAmountCannotBeZero();
    minRsEthAmountToWithdraw[asset] = minRsEthAmountToWithdraw_;
    emit MinAmountToWithdrawUpdated(asset, minRsEthAmountToWithdraw_);
}
```

### Proof of Concept
1. Admin deploys `LRTWithdrawalManager` and never calls `setMinRsEthAmountToWithdraw` for ETH.
2. Alice holds 1 wei of rsETH and calls `initiateWithdrawal(ETH_TOKEN, 1, "")`. The check `1 < 0` is false, so the call succeeds. Alice's 1 wei rsETH is transferred to the withdrawal manager.
3. After `withdrawalDelayBlocks`, the operator calls `unlockQueue(ETH_TOKEN, ...)`. `_unlockWithdrawalRequests` processes Alice's request: `payoutAmount = 1 * rsETHPrice / assetPrice ≈ 1 wei`. The rsETH is burned; 1 wei ETH is pulled from `LRTUnstakingVault` into the withdrawal manager. `unlockedWithdrawalsCount[ETH_TOKEN]` becomes 1.
4. Alice's `completeWithdrawal` would cost ~50,000–100,000 gas (≈ 0.001–0.003 ETH at 20–30 gwei), far exceeding 1 wei. Alice does not call it.
5. The 1 wei ETH is permanently stranded. `sweepRemainingAssets(ETH_TOKEN)` reverts with `PendingWithdrawalsExist` because `unlockedWithdrawalsCount[ETH_TOKEN] == 1`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
