### Title
Attacker-Controlled `addr` Field in Routed `StateHeaderRequest`/`StatePartRequest` Causes Snapshot Host to Initiate Outbound Tier3 Connections to Arbitrary Addresses - (File: `chain/network/src/peer_manager/network_state/mod.rs`)

### Summary
The `addr` field embedded in `StateHeaderRequest` and `StatePartRequest` routed messages is accepted from any connected peer without any validation that it matches the actual sender's TCP connection address. A connected unprivileged peer can forge an arbitrary `addr`, causing the snapshot host to initiate an outbound Tier3 TCP connection to an attacker-controlled IP:port and deliver state data there.

### Finding Description
`StateHeaderRequest` and `StatePartRequest` each carry an `addr: std::net::SocketAddr` field that is intended to be the requester's own public address so the snapshot host can open a direct Tier3 connection back to deliver the response.

When a snapshot host receives a routed `T2MessageBody::StateHeaderRequest` or `T2MessageBody::StatePartRequest`, the handler in `receive_routed_message` constructs a `Tier3Request` by blindly copying `request.addr` into `peer_info.addr`:

```rust
T2MessageBody::StateHeaderRequest(request) => {
    self.peer_manager_adapter.send(Tier3Request {
        peer_info: PeerInfo {
            id: msg_author,
            addr: Some(request.addr),   // ← attacker-supplied, never validated
            ...
        },
        ...
    });
}
```

The `Tier3Request` is then dispatched to `PeerManagerActor::handle<Tier3Request>`, which:
1. Computes the state header/part response.
2. Calls `transport.connect_to_peer(&clock, request.peer_info.clone(), tcp::Tier::T3)` — initiating an **outbound TCP connection to the attacker-supplied address**.
3. Sends the full state response over that connection.

There is no check that `request.addr` corresponds to the actual TCP source address of the peer that sent the routed message. The `addr` field is a free-form payload field that any peer can set to any value.

Additionally, the rate-limit table explicitly marks these message types as unrated:
```rust
T2MessageBody::StatePartRequest(_) => None, // TODO
T2MessageBody::StateHeaderRequest(_) => None, // TODO
```

### Impact Explanation
- **SSRF (Server-Side Request Forgery)**: Any connected peer can cause the snapshot host to initiate outbound TCP connections to arbitrary IP:port combinations, including internal network addresses not reachable from the public internet. This can be used to probe internal services, bypass firewall rules, or interact with services that trust connections from the node's IP.
- **State data misdirection**: The full state header or state part (potentially megabytes of data) is delivered to the attacker-controlled address rather than the legitimate requester, wasting bandwidth and leaking data to unintended recipients.
- **No rate limit**: The absence of any rate limit on routed `StateHeaderRequest`/`StatePartRequest` messages amplifies the impact, allowing repeated connection attempts to arbitrary targets.

The corrupted protocol value is the **Tier3 connection target address** and the **destination of state part/header data** — both are determined by an unauthenticated, peer-supplied field rather than the authenticated connection metadata.

### Likelihood Explanation
Any peer that can establish a Tier2 connection to a snapshot host (i.e., any node on the public network) can send a routed `StateHeaderRequest` or `StatePartRequest` with a forged `addr`. No validator or operator privileges are required. The routed message passes signature verification (the `RoutedMessage` signature covers the author's PeerId, not the `addr` field), so the forgery is undetected.

### Recommendation
Do not trust the `addr` field from the message payload. Instead, derive the callback address from the authenticated TCP connection metadata (the actual source address of the peer that delivered the routed message), or require the `addr` to be signed by the author's key and validated against the known peer address. At minimum, add a rate limit on routed `StateHeaderRequest`/`StatePartRequest` messages (the existing `// TODO` comments acknowledge this gap).

### Proof of Concept
1. Establish a Tier2 connection to a snapshot host node.
2. Craft a signed `RoutedMessage` with body `T2MessageBody::StateHeaderRequest(StateHeaderRequest { shard_id: <valid>, sync_hash: <valid>, addr: <target_ip>:<target_port> })`, where `<target_ip>:<target_port>` is any address (e.g., an internal service or a controlled listener).
3. Send the routed message to the snapshot host.
4. The snapshot host processes it in `receive_routed_message` at `chain/network/src/peer_manager/network_state/mod.rs:1005–1016`, constructs a `Tier3Request` with `peer_info.addr = <target_ip>:<target_port>`, and dispatches it.
5. `PeerManagerActor::handle<Tier3Request>` at `chain/network/src/peer_manager/peer_manager_actor.rs:1634–1643` calls `transport.connect_to_peer(…, tcp::Tier::T3)` targeting the attacker-supplied address and sends the state response there.
