Audit Report

## Title
`setPoolAdminFeeDestination` Fails to Flush Accrued Fees Before Overwriting Destination, Causing Old Destination to Lose Owed Admin Fees - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

## Summary
`setPoolAdminFeeDestination` overwrites `poolAdminFeeDestination[pool]` without first calling `collectFees` for the old destination. Any spread-fee surplus and notional fees that accrued under the old destination are permanently redirected to the new destination on the next `collectPoolFees` call. The old destination has no recourse and the loss is irreversible.

## Finding Description
`setPoolAdminFees` (L408–435) correctly calls `collectFees` with the current `poolAdminFeeDestination[pool]` before updating fee rates, ensuring the old destination receives all fees accrued at the old rates: [1](#0-0) 

`setPoolAdminFeeDestination` (L438–447) performs no such flush — it directly overwrites the mapping: [2](#0-1) 

`collectPoolFees` is permissionless and reads `poolAdminFeeDestination[pool]` at call time: [3](#0-2) 

Inside `collectFees`, the admin's share of the spread-fee surplus is computed from the live pool balance minus bin totals minus notional fees, and transferred directly to `adminFeeDestination_`: [4](#0-3) [5](#0-4) 

Because the surplus accumulates continuously with every swap and is only realized at collection time, all surplus that built up while the old destination was active is swept to the new destination on the next collection call.

## Impact Explanation
The old admin fee destination loses all admin-share spread fees and notional fees that accrued between the last `collectPoolFees` call and the `setPoolAdminFeeDestination` call. These are real ERC-20 token balances held by the pool. The new destination receives tokens it did not earn. This constitutes a direct, irreversible loss of owed protocol/admin fees, meeting the Critical/High direct loss threshold under the allowed impact gate.

## Likelihood Explanation
The pool admin is a semi-trusted actor who legitimately calls `setPoolAdminFeeDestination` during treasury rotations. Unlike `setPoolAdminFees`, which enforces collection automatically, `setPoolAdminFeeDestination` provides no guard. Any treasury rotation without a prior manual `collectPoolFees` call triggers the loss. Since `collectPoolFees` is permissionless, a third party (keeper, bot, or adversary) can call it immediately after the destination change, making the race window reliably exploitable and the loss guaranteed without atomic protection.

## Recommendation
Mirror the pattern from `setPoolAdminFees` by flushing accrued fees to the old destination before overwriting it:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    // Flush accrued fees to the old destination before switching
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]
    );
    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

## Proof of Concept
1. Pool is created with `adminFeeDestination = oldTreasury`.
2. Swaps occur; spread-fee surplus accumulates (`balance > binTotals + notionalFees`).
3. Pool admin calls `setPoolAdminFeeDestination(pool, newTreasury)` without a prior `collectPoolFees` — this is the normal treasury rotation flow.
4. `poolAdminFeeDestination[pool]` is now `newTreasury`.
5. Anyone calls `collectPoolFees(pool)` (permissionless).
6. `collectFees` computes the full accumulated surplus and sends the admin share to `newTreasury`.
7. `oldTreasury` receives nothing despite having earned the fees during steps 1–3.

A Foundry test can confirm this by: deploying a pool, executing swaps to build surplus, calling `setPoolAdminFeeDestination` to a new address, then calling `collectPoolFees` and asserting `oldTreasury.balance == 0` while `newTreasury.balance > 0`.
