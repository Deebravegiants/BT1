Audit Report

## Title
Stale Cached `rsETHPrice` Allows Depositors to Mint Excess rsETH, Diluting Accrued Yield of Existing Holders - (File: contracts/LRTDepositPool.sol)

## Summary

`LRTDepositPool.getRsETHAmountToMint` divides by `lrtOracle.rsETHPrice()`, which is a lazily-updated stored state variable in `LRTOracle`. Because rewards accrue continuously but `rsETHPrice` is only refreshed on explicit calls to `updateRSETHPrice()`, any depositor can mint rsETH at a below-true-value price during the staleness window, capturing a portion of yield that belongs to existing rsETH holders.

## Finding Description

`getRsETHAmountToMint` computes the mint amount as:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [1](#0-0) 

`lrtOracle.rsETHPrice()` resolves to the public state variable:

```solidity
uint256 public override rsETHPrice;
``` [2](#0-1) 

This variable is only updated inside `_updateRsETHPrice()`, which is triggered by the public `updateRSETHPrice()` or the manager-only `updateRSETHPriceAsManager()`:

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [3](#0-2) 

The true price is computed inside `_updateRsETHPrice()` as:

```solidity
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
``` [4](#0-3) 

and only written to storage at the end of that function:

```solidity
rsETHPrice = newRsETHPrice;
``` [5](#0-4) 

The deposit flow (`depositETH` / `depositAsset`) calls `_beforeDeposit` → `getRsETHAmountToMint` → reads the stale `rsETHPrice` — there is no atomic price refresh before minting. [6](#0-5) 

A compounding factor: when a large reward event pushes the true price above `highestRsethPrice` by more than `pricePercentageLimit`, the public `updateRSETHPrice()` reverts for non-manager callers:

```solidity
if (!IAccessControl(address(lrtConfig)).hasRole(LRTConstants.MANAGER, msg.sender)) {
    revert PriceAboveDailyThreshold();
}
``` [7](#0-6) 

This extends the staleness window precisely when the price gap — and therefore the exploitable yield — is largest.

## Impact Explanation

**High — Theft of unclaimed yield.**

Existing rsETH holders continuously accrue yield (EigenLayer restaking rewards, LST appreciation) reflected in the true `rsETHPrice = totalETHInProtocol / rsethSupply`. A depositor who mints at the stale lower price receives more rsETH than their deposit is worth at the true exchange rate. When `updateRSETHPrice()` is subsequently called, the inflated supply causes the new price to be lower than it would have been without the deposit, permanently diluting the yield of all prior holders. The stolen value per deposit is proportional to `(truePrice − stalePrice) / truePrice × depositAmount`.

## Likelihood Explanation

**Medium.** Rewards accrue continuously from EigenLayer strategies and LST appreciation, so a staleness window exists after every reward event. `updateRSETHPrice()` is not called atomically with deposits, and no on-chain mechanism forces a refresh. The attack requires no special privileges — any depositor can execute it. The `pricePercentageLimit` guard makes the window longer and the yield gap larger after significant reward events, increasing both feasibility and profitability.

## Recommendation

Either (a) call `_updateRsETHPrice()` (or an equivalent live computation) atomically at the start of every deposit before computing `getRsETHAmountToMint`, ensuring the price is always fresh at mint time; or (b) replace the stored `rsETHPrice` read in `getRsETHAmountToMint` with an inline live computation of `totalETHInProtocol / rsethSupply` so the mint ratio always reflects the current state of the protocol.

## Proof of Concept

1. Protocol state: 1 000 ETH TVL, 1 000 rsETH supply → stored `rsETHPrice = 1.0 ETH`.
2. EigenLayer rewards of 10 ETH accrue → true price is `1 010 / 1 000 = 1.01 ETH`; stored `rsETHPrice` remains `1.0 ETH`.
3. Attacker calls `depositETH{value: 100 ether}(0, "")` before anyone calls `updateRSETHPrice()`.
4. `getRsETHAmountToMint` computes `100e18 * 1e18 / 1.0e18 = 100 rsETH` minted to attacker.
5. Fair amount at true price: `100 / 1.01 ≈ 99.01 rsETH`; attacker received `≈ 0.99 rsETH` excess.
6. `updateRSETHPrice()` is called: TVL = 1 110 ETH, supply = 1 100 rsETH → new price = `1 110 / 1 100 ≈ 1.009 ETH` instead of `1.01 ETH`. Existing holders permanently lost `≈ 0.001 ETH/rsETH` of yield.
7. The attack is repeatable every reward cycle and scales linearly with deposit size.
