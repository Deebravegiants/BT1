### Title
Missing RSA Modulus Bit-Length Validation in `base_pke_rsa_ek_from_modulus` Allows Sub-2048-bit Key Acceptance - (`src/cbmpc/api/pve_base_pke.cpp`)

### Summary

`base_pke_rsa_ek_from_modulus` validates only the byte-length of the supplied modulus (256 bytes), not its bit-length (2048 bits). A 256-byte buffer whose most significant byte is `0x00` passes the byte-length guard, but `BN_bin2bn` produces a BIGNUM smaller than 2^2047. The resulting sub-2048-bit RSA key is accepted, stored with the `rsa_oaep_2048` type tag, and used for PVE encryption without error.

### Finding Description

The validation in `base_pke_rsa_ek_from_modulus` is:

```cpp
constexpr int kExpectedModulusBytes = coinbase::crypto::RSA_KEY_LENGTH / 8;  // 256
if (modulus.size != kExpectedModulusBytes)
    return coinbase::error(E_BADARG, "modulus must be exactly 256 bytes (RSA-2048)");

BIGNUM* n_bn = BN_bin2bn(modulus.data, modulus.size, nullptr);
if (BN_is_zero(n_bn)) { ... }   // only zero-check
pk.set(n_bn, e_bn);             // no bit-length check
``` [1](#0-0) 

There is no call to `BN_num_bits(n_bn)` to assert the value equals 2048. `rsa_pub_key_t::set` passes the BIGNUM directly to `EVP_PKEY_fromdata` with no further size enforcement: [2](#0-1) 

`rsa_pub_key_t::size()` returns `EVP_PKEY_get_size(ptr)`, which reflects the actual BIGNUM size, not the expected 256: [3](#0-2) 

`encrypt_oaep_with_seed` derives the padding width from `size() * 8`, so a 255-byte effective modulus produces a 255-byte padded buffer, which then passes the `n_size == in.size` check in `encrypt_raw` and completes successfully: [4](#0-3) 

The resulting key blob is tagged `rsa_oaep_2048` despite the key being sub-2048-bit.

### Impact Explanation

An API caller who supplies a 256-byte modulus with one or more leading zero bytes — and who knows the factorization of that modulus (trivially true if they constructed it) — obtains a PVE encryption key blob that the library accepts as valid RSA-2048. Any plaintext encrypted under that blob is encrypted under a weaker-than-declared key. In the HSM integration scenario this API targets, a malicious or compromised modulus source causes honest software to silently encrypt PVE key-share material under a key the attacker can factor, enabling full recovery of the protected secret. The `rsa_oaep_2048` type tag on the blob provides false assurance to downstream consumers.

### Likelihood Explanation

The function is a public C and C++ API (`cbmpc_pve_base_pke_rsa_ek_from_modulus` / `base_pke_rsa_ek_from_modulus`) reachable without any privilege. The only existing negative tests cover wrong byte-length and all-zero inputs; no test covers a 256-byte buffer with a leading zero byte. [5](#0-4) 

### Recommendation

After `BN_bin2bn`, add an explicit bit-length check before calling `pk.set`:

```cpp
if (BN_num_bits(n_bn) != coinbase::crypto::RSA_KEY_LENGTH) {
    BN_free(n_bn);
    return coinbase::error(E_BADARG, "modulus must be exactly 2048 bits");
}
```

This ensures a 256-byte buffer with any leading zero bytes is rejected with `E_BADARG` rather than silently accepted.

### Proof of Concept

```cpp
// 256-byte buffer, first byte 0x00, remaining bytes set to 0xFF
// (or any non-zero pattern the attacker controls/can factor)
std::array<uint8_t, 256> modulus{};
modulus[0] = 0x00;
for (int i = 1; i < 256; ++i) modulus[i] = 0xFF;

buf_t ek;
error_t rv = coinbase::api::pve::base_pke_rsa_ek_from_modulus(
    mem_t(modulus.data(), 256), ek);

// Expected: E_BADARG (modulus is not 2048 bits)
// Actual:   SUCCESS  — sub-2048-bit key accepted and tagged rsa_oaep_2048
assert(rv == E_BADARG);  // FAILS
```

void rsa_pub_key_t::set(RSA_BASE*& rsa, const BIGNUM* n, const BIGNUM* e) {
  cb_assert(n && e);
  OSSL_PARAM_BLD* param_bld = OSSL_PARAM_BLD_new();
  OSSL_PARAM_BLD_push_BN(param_bld, "n", n);
  OSSL_PARAM_BLD_push_BN(param_bld, "e", e);
  OSSL_PARAM* params = OSSL_PARAM_BLD_to_param(param_bld);

  scoped_ptr_t<EVP_PKEY_CTX> ctx = EVP_PKEY_CTX_new_from_name(NULL, "RSA", NULL);
  cb_assert(EVP_PKEY_fromdata_init(ctx) > 0);
  cb_assert(EVP_PKEY_fromdata(ctx, &rsa, EVP_PKEY_PUBLIC_KEY, params) > 0);

  OSSL_PARAM_free(params);
  OSSL_PARAM_BLD_free(param_bld);
}
