### Title
Unbounded `stored_attestations` growth via permissionless `submit_participant_info` enables linear-scan DoS of critical MPC threshold-signing protocol functions — (`crates/contract/src/tee/tee_state.rs`)

---

### Summary

Any NEAR account can call `submit_participant_info` to insert entries into the `stored_attestations` `IterableMap`. The function `lookup_node_id_by_signer_pk` performs a **full linear scan** over this map to find a node by its `account_public_key`. Because the map is indexed by TLS public key but the lookup is on a different field (`account_public_key`), no direct key access is possible. If an attacker floods `stored_attestations` with enough entries, every call to `lookup_node_id_by_signer_pk` exhausts the NEAR gas budget, causing `vote_pk`, `vote_reshared`, `start_keygen_instance`, and `start_reshare_instance` to fail with out-of-gas errors. This permanently freezes the MPC network during any key-generation or resharing epoch.

---

### Finding Description

**Root cause — linear scan over an unbounded map:**

`lookup_node_id_by_signer_pk` iterates over every entry in `stored_attestations` to find the one whose `account_public_key` matches the caller's signer key:

```rust
pub(crate) fn lookup_node_id_by_signer_pk(
    &self,
    signer_account_pk: &Ed25519PublicKey,
) -> Result<&NodeId, AttestationCheckError> {
    self.stored_attestations
        .iter()
        .find(|(_, attestation)| attestation.node_id.account_public_key == *signer_account_pk)
        .map(|(_, attestation)| &attestation.node_id)
        .ok_or(AttestationCheckError::AttestationNotFound)
}
``` [1](#0-0) 

The `TeeState` struct's own documentation acknowledges this danger explicitly:

> "Attestations are stored for any valid participant that has submitted one, not just for the currently active participants. **Callers must not assume this map is small; use the key-indexed accessors rather than scanning the whole collection.**" [2](#0-1) 

Yet `lookup_node_id_by_signer_pk` violates this guidance by scanning the whole collection.

**Permissionless growth path:**

`submit_participant_info` is a `#[payable]` method with no participant restriction. Any NEAR account can call it with a unique TLS public key and a valid attestation (including `Attestation::Mock(MockAttestation::Valid)`, which is accepted in the current "transitioning period" where both TEE and non-TEE modes are supported):

```rust
#[payable]
#[handle_result]
pub fn submit_participant_info(
    &mut self,
    proposed_participant_attestation: dtos::Attestation,
    tls_public_key: dtos::Ed25519PublicKey,
) -> Result<(), Error> {
``` [3](#0-2) 

Each call with a fresh TLS key inserts a new entry into `stored_attestations`: [4](#0-3) 

The only barrier is a storage-staking deposit, which is proportional to entry size (~100 bytes) and costs on the order of 0.001 NEAR per entry — trivially cheap.

**Critical path affected:**

`lookup_node_id_by_signer_pk` is called inside the participant-authentication check (`assert_caller_is_attested_participant_and_protocol_active`), which gates every critical MPC protocol method:

- `vote_pk` — casts a vote for the generated public key during key generation
- `vote_reshared` — casts a vote for successful resharing
- `start_keygen_instance` — starts a new key-generation attempt
- `start_reshare_instance` — starts a new resharing attempt [5](#0-4) [6](#0-5) 

**No effective cleanup path:**

`clean_invalid_attestations` only removes entries whose attestation has expired or whose image hash is no longer on the allowlist. Attacker-submitted mock attestations with no expiry are never removed by this sweep: [7](#0-6) 

`clean_non_participant_votes` removes votes but explicitly leaves `stored_attestations` untouched: [8](#0-7) 

---

### Impact Explanation

When `stored_attestations` contains enough entries to exhaust the 300 TGas NEAR transaction budget during the linear scan, every call to `vote_pk` and `vote_reshared` by legitimate MPC participants fails with an out-of-gas error. The MPC network cannot:

- Complete an in-progress key-generation epoch (Initializing state → network never becomes operational)
- Complete an in-progress resharing epoch (Resharing state → network permanently stuck, unable to rotate participants or respond to key compromise)

All cross-chain assets whose signatures require the MPC network are permanently frozen. This matches the **Critical** impact criterion: *permanent freezing of funds controlled by the MPC network*.

---

### Likelihood Explanation

- **Attack cost:** Each entry costs ~0.001 NEAR in storage staking. Causing OOG at 300 TGas requires on the order of tens to low hundreds of entries (each `IterableMap` iteration involves a persistent-storage read costing ~5 TGas). Total attack cost: well under 1 NEAR.
- **Permissionless entry point:** `submit_participant_info` has no caller restriction. Any NEAR account can call it.
- **Mock attestations accepted in production:** The codebase explicitly states the network is in a "transitioning period" where both TEE and non-TEE modes are supported, meaning `Attestation::Mock(MockAttestation::Valid)` passes verification today.
- **No hard cap:** There is no upper bound on `stored_attestations` size.

---

### Recommendation

1. **Add a secondary index:** Maintain a `LookupMap<Ed25519PublicKey, Ed25519PublicKey>` mapping `account_public_key → tls_public_key`. Update it on every `add_participant` call. Replace the linear scan in `lookup_node_id_by_signer_pk` with an O(1) lookup through this index.

2. **Cap `stored_attestations`:** Enforce a maximum number of entries (e.g., bounded by the maximum possible participant set size times a small multiplier). Reject `submit_participant_info` calls that would exceed the cap.

3. **Restrict `submit_participant_info` to candidates:** Require the caller to be either a current participant or a prospective participant in a pending resharing proposal, eliminating the fully-open insertion path.
