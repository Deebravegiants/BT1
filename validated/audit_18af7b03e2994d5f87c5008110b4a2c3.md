### Title
rsETH Price Inflation via ETH Donation Enables First-Depositor Theft - (`contracts/LRTOracle.sol` / `contracts/LRTDepositPool.sol`)

---

### Summary

`LRTOracle._updateRsETHPrice()` computes the rsETH price as `totalETHInProtocol / rsethSupply`, where `totalETHInProtocol` includes `address(LRTDepositPool).balance` directly. Because `LRTDepositPool` has an unrestricted `receive()` function and `updateRSETHPrice()` is public, an attacker who is the first depositor can donate ETH to inflate `rsETHPrice` to an arbitrarily large value. When `pricePercentageLimit` is unset (defaults to 0, bypassing the price-increase guard), subsequent depositors who pass `minRSETHAmountExpected = 0` receive 0 rsETH while their ETH is permanently absorbed into the pool.

---

### Finding Description

**Root cause 1 — raw balance used in TVL:**

`getETHDistributionData()` reads `address(this).balance` without any donation-exclusion accounting: [1](#0-0) 

Any ETH sent directly to the contract (via `receive()`) immediately increases the reported TVL.

**Root cause 2 — public price update with no limit when `pricePercentageLimit == 0`:**

`updateRSETHPrice()` is callable by anyone: [2](#0-1) 

The price-increase guard is conditioned on `pricePercentageLimit > 0`: [3](#0-2) 

`pricePercentageLimit` is never set in `initialize()`, so it defaults to `0`, making `isPriceIncreaseOffLimit` always `false` and the guard a no-op: [4](#0-3) 

**Root cause 3 — rsETH mint amount calculation:** [5](#0-4) 

With an inflated `rsETHPrice`, `rsethAmountToMint` rounds to 0 for any deposit smaller than the inflated price.

**Root cause 4 — zero-rsETH deposit is not rejected when `minRSETHAmountExpected = 0`:** [6](#0-5) 

If the caller passes `minRSETHAmountExpected = 0`, the check `rsethAmountToMint < 0` is never true, so the deposit proceeds, ETH is retained by the pool, and 0 rsETH is minted.

---

### Impact Explanation

A victim who deposits Y ETH with `minRSETHAmountExpected = 0` while the price is inflated to `(1 + X) * 1e18` receives 0 rsETH when `Y < (1 + X)`. Their ETH is permanently locked in the pool. The attacker, holding the only 1 wei of rsETH, can later redeem it for `1 + X + Y` ETH, profiting by Y ETH.

This is **direct theft of user funds** — Critical impact.

---

### Likelihood Explanation

The attack requires:
1. The attacker is the first depositor (rsETH supply = 0 or 1 wei) — realistic at protocol launch or after a full withdrawal cycle.
2. `pricePercentageLimit == 0` — this is the **default state** since `initialize()` never sets it.
3. A victim calls `depositETH` with `minRSETHAmountExpected = 0` — common when integrators or scripts omit slippage protection.

No front-running is required. The attacker can set up the inflated state in advance and simply wait for a victim.

---

### Recommendation

1. **Virtual offset / dead shares**: Mint a small amount of rsETH to a dead address at initialization (e.g., 1000 wei) so the supply is never 1 wei, making the rounding attack economically infeasible.
2. **Exclude donations from TVL**: Track deposited ETH in a separate accounting variable rather than using `address(this).balance` directly in `getETHDistributionData()`.
3. **Require `pricePercentageLimit > 0` at initialization**: Set a non-zero default in `initialize()` so the price-increase guard is always active.
4. **Reject zero-rsETH mints**: In `_beforeDeposit`, revert if `rsethAmountToMint == 0` regardless of `minRSETHAmountExpected`.

---

### Proof of Concept

```
State: rsETH totalSupply = 0, pricePercentageLimit = 0 (default)

1. updateRSETHPrice() → rsETHPrice = 1e18 (zero-supply branch)

2. Attacker calls depositETH{value: 1}(0, "")
   → rsethAmountToMint = 1 * 1e18 / 1e18 = 1
   → Attacker holds 1 wei rsETH
   → LRTDepositPool.balance = 1 wei

3. Attacker sends 100_000e18 ETH directly to LRTDepositPool (receive())
   → LRTDepositPool.balance = 100_000e18 + 1

4. Attacker calls updateRSETHPrice()
   → totalETHInProtocol ≈ 100_000e18
   → rsethSupply = 1
   → rsETHPrice = 100_000e18 * 1e18 / 1 = 100_000e36

5. Victim calls depositETH{value: 50_000e18}(0, "")
   → rsethAmountToMint = 50_000e18 * 1e18 / 100_000e36 = 0
   → minRSETHAmountExpected = 0 → no revert
   → Victim's 50_000e18 ETH absorbed, 0 rsETH minted

6. Attacker redeems 1 wei rsETH via withdrawal
   → Receives ≈ 150_000e18 ETH (1 + 100_000e18 + 50_000e18)
   → Net profit: 50_000e18 ETH (victim's deposit)
