### Title
Governance Deadlock via Unrestricted Maximum Threshold Permanently Freezes Contract Administration — (File: `crates/contract/src/primitives/thresholds.rs`)

---

### Summary

The NEAR MPC contract allows the `GovernanceThreshold` to be set to `n` (100% of participants). All administrative governance actions — including the only mechanism to lower the threshold — require this same threshold of votes. If even one participant becomes permanently unavailable, no governance action can ever pass again. There is no emergency admin bypass, no multisig owner, and the code explicitly blocks forwarded calls (e.g., through a multisig). The code itself acknowledges this risk but does not address it. When combined with the `accept_requests = false` path in `verify_tee()`, which the code labels "requires manual intervention" with no such mechanism existing, the contract can be permanently frozen with no recovery path.

---

### Finding Description

**Root Cause 1 — Unrestricted upper bound on GovernanceThreshold:**

`governance_threshold_upper_relative_bound(n)` returns `n` (100%), meaning a valid deployment can set GovernanceThreshold equal to the total participant count. [1](#0-0) 

The comment in the code explicitly acknowledges the risk: *"would be a discussion subject to drop this upper bound down not to have problems with smart contract being locked if t = n and if an operator stops voting."* Despite this acknowledgment, the upper bound remains at 100%.

**Root Cause 2 — All governance actions are gated behind the same threshold:**

Every critical administrative action requires `threshold` votes from current participants:
- `vote_new_parameters` (change participant set / lower threshold) — requires `n_old >= threshold` old participants
- `vote_update` (contract code/config upgrade) — requires `threshold` votes
- `vote_add_domains`, `vote_code_hash`, `vote_cancel_resharing` — all require threshold votes [2](#0-1) [3](#0-2) [4](#0-3) 

**Root Cause 3 — No emergency admin bypass exists; multisig forwarding is explicitly blocked:**

`assert_caller_is_signer()` enforces that all governance methods must be called directly from the participant's own NEAR account, explicitly rejecting calls forwarded through another contract such as a multisig. [5](#0-4) 

**Root Cause 4 — `accept_requests = false` has no recovery path when governance is deadlocked:**

`verify_tee()` sets `accept_requests = false` and logs "This requires manual intervention" when kicking out participants with expired TEE attestations would break the threshold relation. But there is no out-of-band mechanism for that intervention — recovery requires `vote_new_parameters` (resharing), which itself requires threshold votes. [6](#0-5) 

**Deadlock scenario:**

1. GovernanceThreshold is set to `n` (valid per `governance_threshold_upper_relative_bound`).
2. One participant goes offline, loses their key, or has their TEE attestation expire permanently.
3. `vote_new_parameters` to remove the unavailable participant requires `n` votes; only `n-1` are available — the proposal can never pass.
4. No other governance action can pass either (contract upgrade, threshold change, domain addition).
5. If TEE attestations expire for enough participants, `verify_tee()` sets `accept_requests = false`.
6. The contract is permanently frozen: no new signature requests accepted, no governance recovery possible.

---

### Impact Explanation

**Medium.** The governance system is a single point of failure for all administrative actions. A permanently unavailable participant (below the signing threshold) can deadlock the entire governance mechanism, preventing contract upgrades, participant set changes, threshold adjustments, and TEE image whitelisting. When combined with the `accept_requests = false` path, the contract stops accepting new cross-chain signature requests with no recovery path. This breaks production safety/accounting invariants — the MPC network's operational continuity — without relying on network-level DoS or privileged operator access.

---

### Likelihood Explanation

**Medium.** Setting GovernanceThreshold = `n` is a valid, explicitly supported configuration (the code allows it and the comment acknowledges the risk without preventing it). Participants can become permanently unavailable through key loss, infrastructure failure, or TEE attestation expiry. TEE attestations have explicit expiry timestamps and must be renewed periodically; a coordinated expiry of enough attestations is a natural operational event. A single Byzantine participant (strictly below the signing threshold) can also deliberately trigger this by voting for GovernanceThreshold = `n` and then going offline.

---

### Recommendation

1. **Lower the `governance_threshold_upper_relative_bound`** to a value strictly less than `n` (e.g., 80% or `n-1`) to ensure at least one participant can be unavailable without deadlocking governance. [1](#0-0) 

2. **Implement an out-of-band emergency admin path** — analogous to the UMA recommendation — such as a designated multisig account (e.g., a NEAR DAO or foundation multisig) that can call a restricted set of emergency functions (`set_governance_threshold`, `remove_participant`) without going through the participant voting mechanism. This would require relaxing the `assert_caller_is_signer()` restriction for a specific emergency role. [5](#0-4) 

3. **Provide a concrete recovery path for `accept_requests = false`** — the log message "This requires manual intervention" should correspond to an actual on-chain mechanism, not just an operational note. [7](#0-6) 

---

### Proof of Concept

**Step 1:** Deploy the contract with `n = 5` participants and `GovernanceThreshold = 5` (valid: `governance_threshold_upper_relative_bound(5) = 5`).

**Step 2:** Participant 5 loses their NEAR account key and goes permanently offline.

**Step 3:** Any participant attempts `vote_new_parameters` to remove participant 5 and lower the threshold to 4. The proposal requires `n_old >= threshold = 5` old participants to vote. Only 4 are available. `validate_incoming_proposal` returns `InsufficientOldParticipants`. [8](#0-7) 

**Step 4:** `vote_update` to deploy a patched contract also requires `threshold = 5` votes. Only 4 are available. The update never executes. [9](#0-8) 

**Step 5:** Participant 5's TEE attestation expires. A participant calls `verify_tee()`. The surviving 4 participants cannot form a valid set (GovernanceThreshold 5 > remaining 4), so `accept_requests` is set to `false`. All new `sign()` calls revert with `TeeValidationFailed`. No recovery path exists. [10](#0-9)

        proposal: &ProposedThresholdParameters,

        // ensure the proposal is valid against the current parameters
        self.parameters
            .validate_incoming_proposal(proposal.parameters())?;

        // Validate effective per-domain thresholds (updates override, absent
        // domains keep theirs) against the proposed participant count.

            u64::try_from(proposal.participants().len()).expect("participant count fits in u64");
        let threshold_updates = proposal.per_domain_thresholds();
        // Reject unknown domain IDs: the loop below iterates existing domains, so
        // an unknown ID would otherwise be silently ignored here (it's caught at
        // the resharing transition, but we fail fast at vote acceptance).
        for id in threshold_updates.keys() {
            if self.domains.get_domain_by_domain_id(*id).is_none() {
                return Err(DomainError::UnknownDomainInProposal { domain_id: *id }.into());
            }
        }

            .domains
            .domains()
            .iter()
            .map(|domain| {
                let effective_threshold = threshold_updates
                    .get(&domain.id)
                    .copied()
                    .unwrap_or(domain.reconstruction_threshold);
                DomainConfig {
                    reconstruction_threshold: effective_threshold,
                    ..domain.clone()
                }
            })
            .collect();
        for domain in &effective_domains {
            validate_domain_threshold(domain, new_num_participants)?;
