### Title
Single Attested Participant Can Deliver Arbitrary CKD Output for `AppPublicKey` Requests Without Threshold Authorization — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd()` in `MpcContract` performs **no cryptographic verification** of the `CKDResponse` when the pending request used the `AppPublicKey` variant. A single Byzantine attested participant (strictly below the signing threshold) can call `respond_ckd` with any arbitrary `big_y` / `big_c` values, and the contract will accept and deliver them to the waiting user. This is the direct analog of the "unlimited minting" class: just as a single `registeredMinter` could mint unlimited tokens by bypassing the supply cap, a single participant here can produce unlimited unverified CKD outputs, bypassing the threshold requirement entirely.

---

### Finding Description

In `respond_ckd()`, after the caller is confirmed to be an attested participant, the contract branches on the request's `app_public_key` variant:

```rust
// crates/contract/src/lib.rs  lines 675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check whatsoever
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract calls `ckd_output_check`, which verifies the BLS12-381 pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, cryptographically binding the response to the network master key and the app's public key. [2](#0-1) 

For `AppPublicKey` (the legacy / privately-verifiable variant), the arm is a bare `{}` — the response bytes are serialized and delivered to the user's yield with **zero validation**: [3](#0-2) 

The `respond` and `respond_verify_foreign_tx` methods always verify the cryptographic output before resolving yields. `respond_ckd` is the only respond path that conditionally skips verification. [4](#0-3) 

The `AppPublicKey` variant is the default / legacy form accepted by `request_app_private_key`, and is the variant used in the existing e2e test `ckd_response__passes_cryptographic_verification`: [5](#0-4) 

The unit test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` confirms the contract accepts a completely fabricated `CKDResponse{big_y:[1u8;48], big_c:[2u8;48]}` for an `AppPublicKey` request without complaint: [6](#0-5) 

---

### Impact Explanation

**Critical — Confidential key derivation output without the required participant authorization.**

The CKD protocol for `AppPublicKey` works as follows: the user holds app secret key `a`, submits `A = a·G1`. The MPC network computes `big_c = H(pk‖app_id)·msk + A·y` and `big_y = y·G1`. The user recovers the confidential key as `big_c − a·big_y = H(pk‖app_id)·msk`.

Because the contract performs no verification, a single Byzantine participant can submit:
- `big_y = G1_identity` (the identity point, which is accepted — confirmed by `app_public_key_check__should_accept_identity_key_pair`)
- `big_c = K` for any arbitrary G1 point `K`

The user then computes `K − a·identity = K`, receiving an attacker-chosen confidential key. The attacker fully controls the derived secret. Any downstream use of that key (e.g., signing transactions, encrypting data, deriving further keys) is compromised. This breaks the core security guarantee of the CKD feature: that the derived key is the unique threshold-computed value bound to the network master key.

---

### Likelihood Explanation

**Medium-High.** The `AppPublicKey` variant is the default/legacy form and is the variant most callers use (it requires only a single G1 point, not a (G1, G2) pair). Any single attested participant — one node out of the full set, well below the signing threshold — can race the honest coordinator's `respond_ckd` call. Because NEAR transaction ordering is public and the pending CKD request map is on-chain, a malicious participant can observe a new `request_app_private_key` call and immediately submit a forged `respond_ckd` before the honest coordinator does. The first `respond_ckd` that matches the request key drains the yield queue and delivers its response.

---

### Recommendation

Apply `ckd_output_check` unconditionally for both variants, or add an equivalent binding check for `AppPublicKey`. For the privately-verifiable variant, the check must use the G1 app public key directly. Alternatively, deprecate `AppPublicKey` in favor of `AppPublicKeyPV` (which already has on-chain verification) and reject `AppPublicKey` requests at the contract level until a verifiable equivalent is designed.

---

### Proof of Concept

1. User calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(A)` and 1 yoctoNEAR deposit. A pending CKD request is stored on-chain.

2. A single Byzantine attested participant (account `evil.near`) observes the pending request on-chain.

3. `evil.near` calls `respond_ckd(ckd_request, CKDResponse { big_y: G1_identity_bytes, big_c: attacker_chosen_K_bytes })`.

4. The contract executes:
   - `assert_caller_is_signer()` — passes (signer == predecessor).
   - `is_running_or_resharing()` — passes.
   - `accept_requests` — passes.
   - `assert_caller_is_attested_participant_and_protocol_active()` — passes (evil.near is an attested participant).
   - `match AppPublicKey(_) => {}` — **no check executed**.
   - `resolve_yields_for(...)` — the user's yield is resolved with the attacker's bytes.

5. The user's callback fires with `CKDResponse { big_y: identity, big_c: K }`. The user computes `K − a·identity = K` and uses the attacker-controlled key as their confidential key.

The existing unit test already demonstrates step 4 succeeds with fabricated bytes: [7](#0-6)

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

/// Verify that a CKD response (AppPublicKey variant) is mathematically correct.
#[tokio::test]
#[expect(non_snake_case)]
async fn ckd_response__passes_cryptographic_verification() {
    // given
    let (cluster, running) =
        common::must_setup_cluster(common::CKD_VERIFICATION_PORT_SEED, |_| {}).await;

    let bls_domain = running
        .domains
        .domains
        .iter()
        .find(|d| {
            Curve::from(d.protocol) == Curve::Bls12381 && matches!(d.purpose, DomainPurpose::CKD)
        })
        .expect("no Bls12381 CKD domain found")
        .clone();

    let mpc_pk = common::must_get_bls_public_key(&running, bls_domain.id);
    let user = cluster.default_user_account().clone();

    let mut rng = rand::rngs::StdRng::seed_from_u64(1);
    let private_key = Scalar::random(&mut rng);
    let app_public_key = CKDAppPublicKey::AppPublicKey(Bls12381G1PublicKey::from(
        &(G1Projective::generator() * private_key),
    ));

    // when
    let outcome = cluster
        .send_ckd_request(bls_domain.id, app_public_key, &user)
        .await
        .expect("CKD request transaction failed");

    // then
    assert!(
        outcome.is_success(),
        "CKD request failed: {:?}",
        outcome.failure_message()
    );

    let response: serde_json::Value = outcome.json().expect("failed to deserialize CKD response");
    let big_y: Bls12381G1PublicKey =
        serde_json::from_value(response["big_y"].clone()).expect("failed to parse big_y");
    let big_c: Bls12381G1PublicKey =
        serde_json::from_value(response["big_c"].clone()).expect("failed to parse big_c");

    assert!(
        verify_ckd(&user, DERIVATION_PATH, &mpc_pk, private_key, &big_y, &big_c)
            .expect("verify_ckd failed"),
        "CKD response failed cryptographic verification"
    );
}
