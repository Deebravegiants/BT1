### Title
Missing `state_diff_commitment` Verification in P2P Sync Allows Unauthenticated Peer to Corrupt Stored `ThinStateDiff` — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

### Summary
The p2p sync client accepts `SignedBlockHeader` messages without verifying the block signature, and then accepts `StateDiffChunk` messages whose assembled content is never checked against the `state_diff_commitment` stored in that header. Any unauthenticated p2p peer can therefore inject a completely fabricated `ThinStateDiff` for any block number, which is durably written to storage and served by all subsequent RPC queries.

### Finding Description
**Step 1 — Header injection (no signature check).**
`HeaderStreamBuilder::parse_data_for_block` (`header.rs:82-123`) accepts a `SignedBlockHeader` if and only if the block number matches the expected sequence and `signatures.len() == 1`. The cryptographic function `verify_block_signature` (which checks `poseidon_hash(block_hash, state_diff_commitment)` against the sequencer public key) is **never invoked**. The attacker therefore controls every field of the stored `BlockHeader`, including `state_diff_length` and `state_diff_commitment`. [1](#0-0) 

**Step 2 — State diff injection (no commitment check).**
`StateDiffStreamBuilder::parse_data_for_block` (`state_diff.rs:62-107`) reads `target_state_diff_len` from the header the attacker just stored, then loops consuming `StateDiffChunk` messages until `current_state_diff_len == target_state_diff_len`. After the loop, `validate_deprecated_declared_classes_non_conflicting` is the only integrity check — it only detects duplicate deprecated class hashes. The assembled `ThinStateDiff` is **never hashed and compared against `state_diff_commitment`**. If the chunks are well-formed (no duplicate keys, correct total length), `Ok(Some((result, block_number)))` is returned. [2](#0-1) 

**Step 3 — Durable write.**
`write_to_storage` calls `append_state_diff(self.1, self.0)` unconditionally, committing the attacker-controlled diff to the node's MDBX database. [3](#0-2) 

The `verify_block_signature` helper that would close this gap exists but is unused in the sync client: [4](#0-3) 

### Impact Explanation
Once a fabricated `ThinStateDiff` is stored for block N, every subsequent RPC call that reads state at or after block N (`get_storage_at`, `get_class_hash_at`, `get_nonce`, `get_class`, etc.) returns attacker-chosen values. Contract storage slots, deployed class hashes, nonces, and compiled class mappings are all fields of `ThinStateDiff` and are all writable by the attacker via `StateDiffChunk::ContractDiff`, `StateDiffChunk::DeclaredClass`, and `StateDiffChunk::DeprecatedDeclaredClass`. This constitutes an authoritative-looking wrong RPC value (High) and wrong storage value / class hash served to callers (Critical).

### Likelihood Explanation
The p2p sync client connects to unauthenticated peers. Any node that can establish a p2p connection and respond to SQMR header and state-diff queries can execute this attack. No validator key, sequencer operator access, or any other privilege is required. The attacker only needs to send well-formed protobuf messages with the correct total `len()` count — a trivial constraint.

### Recommendation
1. **Verify the block signature** in `HeaderStreamBuilder::parse_data_for_block` using `verify_block_signature` with the known sequencer public key before calling `write_to_storage`. Reject and report the peer on failure.
2. **Verify the state diff commitment** in `StateDiffStreamBuilder::parse_data_for_block` after assembling the full `ThinStateDiff`: compute `calculate_state_diff_hash(&result)` and compare it against `block_header.state_diff_commitment`. Reject and report the peer on mismatch.
