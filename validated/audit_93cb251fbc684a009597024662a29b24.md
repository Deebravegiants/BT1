### Title
`respond_ckd()` Skips Output Verification for `AppPublicKey` Variant, Allowing a Single Malicious Participant to Forge CKD Responses - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd()` performs on-chain cryptographic output verification only for the `AppPublicKeyPV` variant of `CKDAppPublicKey`. The `AppPublicKey` variant receives **no verification at all**. A single malicious attested participant (strictly below the signing threshold) can call `respond_ckd()` with an arbitrary fraudulent response for any pending `AppPublicKey` CKD request, bypassing the threshold requirement entirely and delivering an attacker-controlled encrypted key to the user.

---

### Finding Description

The `respond_ckd()` function in `crates/contract/src/lib.rs` contains an incomplete variant check that is the direct analog of the ERC721/ERC1155 issue in the reference report:

```rust
// crates/contract/src/lib.rs lines 675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO VERIFICATION
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract calls `ckd_output_check(...)` to verify the response against the BLS12-381 root public key and the user's app public key pair. For `AppPublicKey`, the arm is a no-op — the contract unconditionally proceeds to `resolve_yields_for`, resolving the pending yield with whatever response the caller supplied.

The `CKDAppPublicKey` enum has exactly two variants:

- `AppPublicKey` — described as "privately verifiable, legacy": a single G1 point; the contract cannot verify the encrypted output on-chain.
- `AppPublicKeyPV` — "publicly verifiable": a `{pk1, pk2}` pair that enables on-chain verification via `ckd_output_check`. [2](#0-1) 

The rest of `respond_ckd()` correctly enforces that the caller is an attested participant and that the domain key is `Bls12381`, but these checks do not compensate for the missing output verification on the `AppPublicKey` branch. [3](#0-2) 

---

### Impact Explanation

**Critical — Confidential key derivation output without required participant authorization.**

A single malicious attested participant can:

1. Observe a pending `CKDRequest` with `AppPublicKey` in `pending_ckd_requests` (contract state is public).
2. Construct an arbitrary `CKDResponse` — e.g., a ciphertext encrypting a known-bad or attacker-chosen key.
3. Call `respond_ckd(request, fraudulent_response)` before honest nodes complete the off-chain threshold protocol.
4. The contract accepts the response with no verification and resolves the yield, delivering the fraudulent encrypted key to the user.

The user receives an encrypted blob that is not the correct threshold-derived key. If the user uses the resulting key to control on-chain assets (the primary use-case for CKD), those assets are permanently inaccessible (frozen) or, if the attacker can arrange for the ciphertext to decrypt to a key they know, stolen. This breaks the threshold guarantee: instead of requiring `t`-of-`n` participants to agree on the derived key, a single participant suffices.

---

### Likelihood Explanation

**Medium-High.** Every attested participant is a potential attacker. The attack requires no collusion, no leaked key material, and no privileged operator access — only the ability to submit a NEAR transaction as an attested participant. The race window (between the user's `request_app_private_key()` call and the honest nodes' `respond_ckd()`) is observable on-chain and can be won by any participant monitoring the chain. `AppPublicKey` is the legacy/default variant and is likely used by existing integrations.

---

### Recommendation

Add on-chain output verification for the `AppPublicKey` variant, or reject `AppPublicKey` requests at the `request_app_private_key()` entry point if on-chain verification is not possible. At minimum, document that `AppPublicKey` requests carry no on-chain integrity guarantee and that callers must use `AppPublicKeyPV` for security-sensitive key derivation. The safest fix is to deprecate `AppPublicKey` and require `AppPublicKeyPV` for all new requests, mirroring the pattern already applied to `AppPublicKeyPV`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {
        // Either reject or add equivalent verification
        env::panic_str("AppPublicKey variant is not supported; use AppPublicKeyPV");
    }
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

### Proof of Concept

**Setup**: Contract is in `Running` state with a `CKD` domain (Bls12381). Attacker is an attested participant.

1. **User submits request** (legitimate):
   ```json
   request_app_private_key({
     "derivation_path": "my-app",
     "app_public_key": { "AppPublicKey": "<user_g1_point>" },
     "domain_id": 2
   })
   ```
   Contract stores the request in `pending_ckd_requests`.

2. **Attacker observes** the pending `CKDRequest` on-chain (domain_id, app_id, app_public_key, tweak are all public).

3. **Attacker calls** `respond_ckd` with a fraudulent response before honest nodes:
   ```json
   respond_ckd({
     "request": { /* exact CKDRequest from step 1 */ },
     "response": { /* attacker-chosen CKDResponse */ }
   })
   ```

4. **Contract execution** (`crates/contract/src/lib.rs` lines 675–682):
   - Matches `AppPublicKey(_)` → empty arm, no verification.
   - Falls through to `resolve_yields_for(&mut self.pending_ckd_requests, &request, ...)`.
   - Yield is resolved; user's promise receives the fraudulent response.

5. **Result**: User receives an attacker-controlled encrypted key. The threshold protocol is bypassed by a single participant. [4](#0-3)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
