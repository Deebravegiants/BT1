Audit Report

## Title
Missing Token Decimal Normalization in `viewSwapRsETHAmountAndFee` Causes Severe rsETH Underpayment for Non-18-Decimal Tokens - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
`viewSwapRsETHAmountAndFee(uint256 amount, address token)` in `RSETHPoolV3` multiplies a raw token amount directly by a Chainlink-normalized 18-decimal rate without first scaling the token amount to 18 decimals. For any non-18-decimal token added as supported collateral (e.g., USDT with 6 decimals), depositors receive `10^(18 - tokenDecimals)` times fewer wrsETH than owed while their full token amount is permanently retained by the pool. The identical flaw exists in five additional pool contracts.

## Finding Description
`ChainlinkOracleForRSETHPoolCollateral.getRate()` normalizes the Chainlink answer to 18 decimal precision:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L34
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
```

This returns the ETH value of **1 whole token** expressed with 18 decimal places (e.g., `3.33e14` for 1 USDT ≈ 0.000333 ETH).

`viewSwapRsETHAmountAndFee` then computes:

```solidity
// contracts/pools/RSETHPoolV3.sol L324-334
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;
uint256 rsETHToETHrate = getRate();
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

`amountAfterFee` is in the token's **native raw units** (e.g., `1e6` for 1 USDT), but `tokenToETHRate` is calibrated to price **1 whole token** (i.e., `1e6` raw USDT = 1 whole USDT). The formula treats `amountAfterFee` as if it were already in 18-decimal units, producing a result `10^(18 - tokenDecimals)` times too small.

Contrast with the ETH-only overload at L299-308, which correctly applies `amountAfterFee * 1e18 / rsETHToETHrate` because ETH is already 18 decimals. No analogous normalization exists for the token path.

The same uncorrected pattern is confirmed at:
- `contracts/pools/RSETHPool.sol` L346
- `contracts/pools/RSETHPoolNoWrapper.sol` L311
- `contracts/pools/RSETHPoolV3ExternalBridge.sol` L452
- `contracts/pools/RSETHPoolV3WithNativeChainBridge.sol` L370
- `contracts/agETH/AGETHPoolV3.sol` L194

`addSupportedToken` in `RSETHPoolV3` imposes no restriction on token decimals and no existing guard in `viewSwapRsETHAmountAndFee` checks or normalizes decimals.

## Impact Explanation
**Critical — Direct theft of user funds.**

For 1 USDT deposit (`1e6` raw units, ETH price $3000, rsETH ≈ 1.05 ETH):
- `tokenToETHRate ≈ 3.33e14`
- `rsETHToETHrate ≈ 1.05e18`
- Actual minted: `1e6 * 3.33e14 / 1.05e18 = 317 wei` of wrsETH (worth ~$0)
- Correct amount: `≈ 3.17e14 wei` (~$0.001 worth)

The user's USDT is transferred into the pool via `safeTransferFrom` before `viewSwapRsETHAmountAndFee` is called, and there is no refund path. The pool accumulates the full token value while minting negligible wrsETH. This matches the "Direct theft of any user funds" critical impact class.

## Likelihood Explanation
`addSupportedToken` in `RSETHPoolV3` requires `TIMELOCK_ROLE`; in `AGETHPoolV3` it requires `DEFAULT_ADMIN_ROLE`. Adding a non-18-decimal token is a legitimate governance action — the contract imposes no decimal restriction and is explicitly designed to accept arbitrary ERC-20 collateral. Once any non-18-decimal token is added (a routine protocol expansion decision), the public `deposit(address token, uint256 amount, string referralId)` entry point is immediately exploitable by any depositor with no further preconditions. The admin action is not an attack; the bug is in the calculation logic that silently activates for every subsequent depositor of that token.

## Recommendation
Normalize `amountAfterFee` to 18 decimals before applying the rate ratio in all affected pool contracts:

```solidity
import { IERC20Metadata } from "@openzeppelin/contracts/token/ERC20/extensions/IERC20Metadata.sol";

function viewSwapRsETHAmountAndFee(uint256 amount, address token)
    public view onlySupportedToken(token)
    returns (uint256 rsETHAmount, uint256 fee)
{
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;

    uint8 tokenDecimals = IERC20Metadata(token).decimals();
    uint256 normalizedAmount = amountAfterFee * 1e18 / 10 ** uint256(tokenDecimals);

    uint256 rsETHToETHrate = getRate();
    uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();

    rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
}
```

Apply the identical fix to `RSETHPool`, `RSETHPoolNoWrapper`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, and `AGETHPoolV3`.

## Proof of Concept
**Foundry fork test outline:**

1. Fork mainnet; deploy or reference `RSETHPoolV3` with a live `ChainlinkOracleForRSETHPoolCollateral` wrapping the USDT/ETH Chainlink feed.
2. Grant `TIMELOCK_ROLE` to test address; call `addSupportedToken(USDT, chainlinkUSDTOracle)`.
3. Fund test user with 1000 USDT (`1e9` raw units).
4. User approves pool and calls `deposit(USDT, 1e9, "ref")`.
5. Assert: `wrsETH.balanceOf(user)` equals `317_000` (≈ `1e9 * 3.33e14 / 1.05e18`) instead of the correct `≈ 3.17e17`.
6. Assert: `USDT.balanceOf(pool)` increased by `1e9` — funds are retained with no recovery mechanism for the depositor.
7. Compute ratio: `correctAmount / actualAmount ≈ 1e12`, confirming the `10^(18-6)` underpayment factor.
