### Title
Attacker Can DOS Legacy-Mode Withdrawal Transactions via Direct Deposits Manipulating Account Balance - (File: `libs/cardano-ledger-core/src/Cardano/Ledger/State/Account.hs`)

### Summary

In the Dijkstra era, the ledger introduces a `directDepositsTxBodyL` field that allows any unprivileged transaction sender to increase the `balanceAccountStateL` of any registered account. Simultaneously, legacy-mode withdrawal validation (used for Plutus V2 transactions) enforces a strict equality check: the withdrawal amount must exactly equal the current account balance. An attacker can observe a victim's pending legacy-mode withdrawal transaction, submit a direct deposit of a negligible amount to the victim's account, and cause the victim's transaction to fail with `IncompleteWithdrawals`.

### Finding Description

**Root cause — strict equality check in `withdrawalsThatDoNotDrainAccounts`:** [1](#0-0) 

The function `withdrawalsThatDoNotDrainAccounts` enforces:
```haskell
withdrawalAmount == fromCompact (account ^. balanceAccountStateL)
```
Any deviation — even 1 lovelace — causes the transaction to fail.

**Root cause — attacker-controlled balance increase via `applyDirectDeposits`:** [2](#0-1) 

`applyDirectDeposits` unconditionally adds to `balanceAccountStateL` for any registered account. The only validation is that the target account exists: [3](#0-2) 

There is no restriction on who may deposit to whom.

**Root cause — legacy mode selects the strict equality path:** [4](#0-3) 

When `legacyMode = True` (triggered by Plutus V2 script inputs), `withdrawalsThatDoNotDrainAccounts` is used. When `legacyMode = False`, the looser `withdrawalsThatExceedAccountBalance` (`<=`) is used instead.

**Ledger rule enforcement in `testIncompleteAndMissingWithdrawals`:** [5](#0-4) 

**Attack path:**
1. Victim Alice has account balance `X` and builds a legacy-mode (Plutus V2) withdrawal transaction specifying exactly `X`.
2. Alice broadcasts the transaction to the mempool.
3. Attacker observes Alice's pending transaction and submits a transaction with `directDepositsTxBodyL` depositing `1` lovelace to Alice's account.
4. If the attacker's transaction is included first, Alice's balance becomes `X + 1`.
5. Alice's withdrawal transaction fails with `IncompleteWithdrawals` because it specifies `X ≠ X + 1`.
6. The attacker can repeat this indefinitely as long as Alice rebuilds and rebroadcasts.

This is confirmed by the test suite: [6](#0-5) 

### Impact Explanation

An unprivileged attacker can permanently prevent any legacy-mode (Plutus V2) withdrawal transaction from being included on-chain by continuously depositing negligible amounts to the victim's account. The victim's withdrawal amount becomes stale on every attempt. While the victim's funds are not destroyed, their ability to withdraw rewards via legacy-mode scripts is indefinitely blocked as long as the attacker continues the attack. This matches the **Medium** impact category: attacker-controlled transactions modify rewards/withdrawals outside design parameters.

### Likelihood Explanation

- Dijkstra era is the latest production era in this repository.
- Any registered account is a valid target; no special relationship is required.
- The attacker's cost per disruption is 1 lovelace (the minimum direct deposit), making sustained attacks economically trivial.
- Plutus V2 scripts are widely deployed across the Cardano ecosystem, making legacy-mode transactions common.

### Recommendation

1. **Remove the strict equality requirement for legacy-mode withdrawals in Dijkstra.** Replace `withdrawalsThatDoNotDrainAccounts` (exact equality) with `withdrawalsThatExceedAccountBalance` (≤) uniformly, regardless of `legacyMode`. The strict drain requirement was designed to prevent partial withdrawals, but this invariant can be maintained by zeroing the balance after withdrawal rather than by rejecting non-exact amounts.
2. **Alternatively, restrict `directDepositsTxBodyL` to prevent deposits to accounts that have pending withdrawal transactions**, or add a protocol-level snapshot mechanism (analogous to the TRST-H-2 fix) that freezes the balance used for withdrawal validation at the time the transaction is submitted.
3. **At minimum**, document that any application using Plutus V2 withdrawal scripts that hardcode the exact balance is vulnerable to this griefing attack and should migrate to Plutus V3 (non-legacy mode).

### Proof of Concept

```
1. Register account for credential C → balance = 0
2. Trigger epoch boundary reward → balance = R (e.g., 1_000_000 lovelace)
3. Alice builds legacy-mode tx:
     withdrawalsTxBodyL = Withdrawals [(accountAddress_C, R)]
     inputsTxBodyL      = [plutusV2ScriptInput]   -- triggers legacyMode
4. Alice broadcasts tx (mempool)
5. Attacker submits tx:
     directDepositsTxBodyL = DirectDeposits [(accountAddress_C, Coin 1)]
6. Attacker's tx included in block N → balance_C = R + 1
7. Alice's tx included in block N+1 → FAILS:
     IncompleteWithdrawals { mismatchSupplied = R, mismatchExpected = R+1 }
8. Repeat from step 3 indefinitely at cost of 1 lovelace per disruption.
```

The `IncompleteWithdrawals` failure constructor and the `withdrawalsThatDoNotDrainAccounts` strict equality check are the precise ledger enforcement points that make this attack possible. [7](#0-6) [8](#0-7)
