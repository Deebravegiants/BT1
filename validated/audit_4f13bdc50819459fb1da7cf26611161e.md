### Title
`vote_cancel_resharing` Returns `RunningContractState` With Stale `parameters_votes`, Allowing Resharing Re-Trigger With Fewer Than Required Votes - (`crates/contract/src/state/resharing.rs`)

---

### Summary

`vote_cancel_resharing` clones and returns `previous_running_state` without clearing its `parameters_votes` field. Because `ResharingContractState::vote_new_parameters` accumulates re-proposal votes directly into `previous_running_state.parameters_votes`, those votes survive the cancellation and are present in the returned `RunningContractState`. A subsequent `vote_new_parameters` call on that state counts the stale votes, allowing the resharing transition to fire with fewer new votes than the full proposed-participant set requires.

---

### Finding Description

**Vote accumulation during resharing re-proposal**

`ResharingContractState::vote_new_parameters` delegates directly to `self.previous_running_state.process_new_parameters_proposal(proposal)`: [1](#0-0) 

`process_new_parameters_proposal` records each vote into `self.parameters_votes`: [2](#0-1) 

So after K participants vote for re-proposal P during resharing, `previous_running_state.parameters_votes` holds K entries.

**`vote_cancel_resharing` does not clear `parameters_votes`**

When threshold cancellation votes are reached, the function clones `previous_running_state` and sets only `previously_cancelled_resharing_epoch_id`: [3](#0-2) 

`parameters_votes` is never reset. The K stale re-proposal votes are carried into the returned `RunningContractState`.

**Epoch ID alignment means the stale votes are immediately usable**

Re-proposal votes were cast with epoch ID = `resharing.prospective_epoch_id().next()`. After cancellation, `previously_cancelled_resharing_epoch_id` is set to `resharing.prospective_epoch_id()`, so: [4](#0-3) 

`returned_state.prospective_epoch_id()` = `cancelled_epoch_id.next()` = `resharing.prospective_epoch_id().next()` — exactly the epoch ID the stale votes were cast for. The epoch-mismatch guard in `vote_new_parameters` passes without error.

**Transition fires with one additional vote**

The transition condition is `new_num_participants == n_votes`: [5](#0-4) 

If (N−1) votes for proposal P were accumulated during resharing, a single subsequent `vote_new_parameters(cancelled_epoch_id.next(), P)` call on the returned running state pushes `n_votes` to N and immediately produces a new `ResharingContractState`.

---

### Impact Explanation

The governance invariant — that transitioning to resharing requires all N proposed participants to cast fresh votes in the Running state — is broken. An attacker who participates in the re-proposal voting during resharing and then triggers cancellation (or waits for others to cancel) can cause a new resharing to begin with only one new vote, bypassing the full-quorum requirement for the governance transition. This is a contract execution-flow manipulation that breaks a production safety/accounting invariant (Medium scope per the program rules).

The resharing protocol itself still requires all proposed participants to complete `vote_reshared`, so this does not directly enable unauthorized key derivation or signature issuance. The harm is the governance bypass: the participant set can be changed with fewer fresh votes than the protocol requires.

---

### Likelihood Explanation

The scenario is reachable in normal operation:

1. A resharing is in progress.
2. Some or all proposed participants vote for re-proposal P (legitimate, e.g., because the original proposed set is partially offline).
3. Threshold participants of the previous running state vote to cancel resharing (legitimate governance action).
4. The returned `RunningContractState` carries the stale votes.
5. One participant calls `vote_new_parameters` with the same proposal P and the correct epoch ID — the transition fires immediately.

No special privileges, leaked keys, or network-level access are required. All steps are reachable through normal contract calls by participants.

---

### Recommendation

Clear `parameters_votes` before returning the running state from `vote_cancel_resharing`. The simplest fix is to call `RunningContractState::new(...)` (which initialises `parameters_votes` to `Default::default()`) instead of cloning `previous_running_state` directly:

```rust
// resharing.rs – vote_cancel_resharing
let running_state = if threshold_cancellation_votes_reached {
    let mut fresh = RunningContractState::new(
        self.previous_running_state.domains.clone(),
        self.previous_running_state.keyset.clone(),
        self.previous_running_state.parameters.clone(),
        self.previous_running_state.add_domains_votes.clone(),
    );
    fresh.previously_cancelled_resharing_epoch_id = Some(self.prospective_epoch_id());
    Some(fresh)
} else {
    None
};
```

This mirrors the pattern already used when a successful re-proposal creates a new `ResharingContractState`: [6](#0-5)

   → n_votes becomes N  →  transition fires  →  new ResharingContractState produced

    pub fn vote_cancel_resharing(&mut self) -> Result<Option<RunningContractState>, Error> {
        let previous_running_participants = self.previous_running_state.parameters.participants();
        let authenticated_candidate = AuthenticatedAccountId::new(previous_running_participants)?;
        self.cancellation_requests.insert(authenticated_candidate);

        let cancellation_votes_count = self.cancellation_requests.len() as u64;
        let previous_running_threshold = self.previous_running_state.parameters.threshold();

        let threshold_cancellation_votes_reached: bool =
            cancellation_votes_count >= previous_running_threshold.value();

        let running_state = if threshold_cancellation_votes_reached {
            let mut previous_running_state = self.previous_running_state.clone();
            let prospective_epoch_id = self.prospective_epoch_id();
            previous_running_state.previously_cancelled_resharing_epoch_id =
                Some(prospective_epoch_id);

            Some(previous_running_state)
        } else {
            None
        };

        Ok(running_state)
    }
