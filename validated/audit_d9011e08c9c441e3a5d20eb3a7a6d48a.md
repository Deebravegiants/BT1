Audit Report

## Title
Deterministic UID Collision in `bridgeTokenToL1` Enables Permanent Freezing of Bridged Funds — (`contracts/bridges/SonicChainNativeTokenBridge.sol`)

## Summary
`SonicChainNativeTokenBridge.bridgeTokenToL1` derives a `uint96` withdrawal UID entirely from values that are either block-level constants or caller-supplied parameters. When the same caller (`msg.sender`) submits two bridge calls in the same block with identical `recipient`, `amount`, and `tx.gasprice`, the hash inputs are byte-for-byte identical, producing the same UID. The duplicate UID is passed to `sonicBridge.withdraw`, and on L1 `SonicBridgeReceiver.claimAndTransferToVault` uses it as the sole replay-prevention key, causing the second withdrawal to be permanently unclaimable.

## Finding Description
The UID is computed at `SonicChainNativeTokenBridge.sol` lines 92–100:

```solidity
uint96 uid = uint96(
    uint256(
        keccak256(
            abi.encodePacked(
                block.timestamp, block.number, msg.sender, recipient, amount, tx.gasprice, bridgeReceiver
            )
        )
    ) % type(uint96).max
);
```

Every input is either a block-level constant (`block.timestamp`, `block.number` — identical for all transactions in the same block), an immutable (`bridgeReceiver`), or a caller-supplied parameter (`msg.sender`, `recipient`, `amount`, `tx.gasprice`). There is no nonce, no contract-internal counter, and no source of entropy that distinguishes two calls with identical parameters in the same block.

The zero-avoidance fallback at lines 103–105 uses an even smaller input set (`block.number`, `msg.sender`, `tx.gasprice`, `bridgeReceiver`), making collisions there strictly more likely.

The UID is passed directly to the Sonic bridge at line 115:
```solidity
sonicBridge.withdraw(uid, originalToken, amount);
```

On L1, `SonicBridgeReceiver.claimAndTransferToVault` (lines 79–82) uses the UID as the sole deduplication key:
```solidity
if (claimedWithdrawals[withdrawalId]) revert WithdrawalAlreadyClaimed();
claimedWithdrawals[withdrawalId] = true;
```

**Concrete collision path:** `RSETHPoolV3` (or any contract integrating the bridge) calls `bridgeTokenToL1` twice in the same block with the same `recipient` (which the NatDoc explicitly notes is "informational only" and in practice resolves to L1Vault — a fixed address), the same `amount`, and the same `tx.gasprice`. Because `msg.sender` is the pool contract in both calls, all seven hash inputs are identical, and both calls produce the same `uid`. Both `sonicBridge.withdraw` calls succeed (if the Sonic bridge does not enforce UID uniqueness at the protocol level), registering two withdrawal records under one ID. On L1, the first `claimAndTransferToVault(uid, ...)` succeeds and sets `claimedWithdrawals[uid] = true`. The second call reverts with `WithdrawalAlreadyClaimed`, and the corresponding tokens — already burned/locked on Sonic — are permanently unclaimable with no admin override for individual withdrawal IDs.

The `% type(uint96).max` truncation (modulo by `2^96 − 1` rather than `2^96`) also introduces a marginal modular bias, but the deterministic collision described above is the primary issue.

## Impact Explanation
**Critical — Permanent freezing of funds.** Tokens transferred from users to the bridge contract on Sonic are burned/locked by `sonicBridge.withdraw`. If the Sonic bridge accepts both `withdraw` calls (registering two withdrawals under the same UID), the second user's tokens are irrecoverably frozen: the Sonic-side burn is final, and the L1-side claim is permanently blocked by `claimedWithdrawals[uid] = true`. There is no per-withdrawal admin recovery path in `SonicBridgeReceiver`; `emergencyRecover` can only move tokens already held by the contract, not tokens stuck in the Sonic bridge under an already-claimed ID.

## Likelihood Explanation
The collision is deterministic, not probabilistic. It requires only that the same caller address submits two bridge calls in the same block with matching `recipient`, `amount`, and `tx.gasprice`. This is a realistic condition for any pool contract that batches bridge operations (e.g., two users triggering auto-bridging via `RSETHPoolV3` in the same block with the same withdrawal amount). No privileged access, no oracle manipulation, and no victim mistake is required. The `recipient` field is documented as "informational only," meaning it is likely a fixed address (L1Vault) in all pool-initiated calls, further reducing the parameter space that must match.

## Recommendation
Replace the hash-based UID with a monotonically incrementing contract-internal counter:

```solidity
uint96 private _nextUid;

function bridgeTokenToL1(address recipient, uint256 amount) external payable override nonReentrant {
    ...
    uint96 uid = ++_nextUid;
    sonicBridge.withdraw(uid, originalToken, amount);
    ...
}
```

This guarantees global uniqueness regardless of block context, caller parameters, gas price, or validator timestamp manipulation, and eliminates both the deterministic collision and the modular bias introduced by `% type(uint96).max`.

## Proof of Concept
1. Deploy `SonicChainNativeTokenBridge` with a mock `sonicBridge` that accepts duplicate UIDs (records both withdrawals independently).
2. In a single block, have `PoolContract` call `bridgeTokenToL1(L1_VAULT, 1e18)` twice with the same `tx.gasprice`.
3. Assert both calls succeed on Sonic (same UID emitted in both `TokensBridgedToL1` events).
4. On L1, call `claimAndTransferToVault(uid, token, 1e18, proof1)` — succeeds, `claimedWithdrawals[uid] = true`.
5. Call `claimAndTransferToVault(uid, token, 1e18, proof2)` — reverts with `WithdrawalAlreadyClaimed`.
6. Confirm the second user's `1e18` tokens are permanently unclaimable despite being burned on Sonic.
