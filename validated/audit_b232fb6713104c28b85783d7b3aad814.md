### Title
Permanently Locked Funds When Normalized Transfer Amount Rounds to Zero ŌĆö (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

`init_transfer` burns or locks user tokens and stores the transfer message before any decimal-normalization check is performed. The normalization check only occurs later in `sign_transfer`, where it can permanently panic with `InvalidAmountToTransfer`. Because no cancel or refund path exists for pending transfers, any user who sends an amount (minus fee) smaller than the destination chain's decimal precision unit suffers an irreversible fund lock.

---

### Finding Description

**Step 1 ŌĆö `init_transfer` commits funds without normalization validation.**

`init_transfer` only checks that `fee < amount`: [1](#0-0) 

It then calls `init_transfer_internal`, which burns deployed tokens or locks native tokens: [2](#0-1) 

At this point the user's tokens are gone from their account and committed to the bridge.

**Step 2 ŌĆö `sign_transfer` applies floor-division normalization and panics if the result is zero.**

`normalize_amount` uses integer floor division to scale the NEAR-side amount down to the destination chain's decimal precision: [3](#0-2) 

`sign_transfer` then rejects the transfer if the normalized value is zero: [4](#0-3) 

**Step 3 ŌĆö No cancel or refund path exists.**

`remove_transfer_message` is only called inside `sign_transfer_callback` (on success when fee is zero) and inside `claim_fee_callback`. There is no user-callable cancel function. The transfer message stays in `pending_transfers` indefinitely, and the burned/locked tokens are unrecoverable.

The code comment itself acknowledges the dust-lock problem for the zero-fee case: [5](#0-4) 

---

### Impact Explanation

**Irreversible fund lock / permanently unclaimable user value in bridge flows.**

For any token pair where `origin_decimals > decimals` (e.g., a NEAR-native token with 24 decimals bridged to an EVM chain where it is registered with 6 decimals, giving a divisor of 10┬╣ŌüĖ), any transfer where `amount ŌłÆ fee < 10┬╣ŌüĖ` yocto-units will:

1. Burn or lock the user's tokens at `init_transfer` time.
2. Permanently fail at `sign_transfer` time with `InvalidAmountToTransfer`.
3. Leave the transfer message in `pending_transfers` with no recovery path.

The tokens are irretrievably lost.

---

### Likelihood Explanation

**Moderate.** The condition is reachable through normal public protocol inputs (`ft_transfer_call` ŌåÆ `ft_on_transfer` ŌåÆ `init_transfer`). No privileged role is required. The only prerequisite is that the user sends an amount below the normalization threshold, which is a realistic scenario for:

- Tokens with large decimal differences (24 vs 6 is a common NEAR-to-EVM pairing).
- Small-value transfers or transfers where a large fee reduces `amount ŌłÆ fee` below the threshold.
- Automated integrations that do not pre-validate the normalized amount.

---

### Recommendation

Add a normalization check inside `init_transfer` (before burning/locking tokens) that mirrors the check in `sign_transfer`:

```rust
// In init_transfer, after building transfer_message and before init_transfer_internal:
let token_address = self.get_token_address(
    transfer_message.get_destination_chain(),
    self.get_token_id(&transfer_message.token),
);
if let Some(addr) = token_address {
    if let Some(decimals) = self.token_decimals.get(&addr) {
        let normalized = Self::normalize_amount(
            transfer_message.amount_without_fee()
                .near_expect(BridgeError::InvalidFee),
            decimals,
        );
        require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
    }
}
```

Alternatively, add a user-callable `cancel_transfer` function that refunds locked/burned tokens for transfers that have not yet been signed, so that stuck transfers can be recovered.

---

### Proof of Concept

1. A NEAR-native token is registered with `origin_decimals = 24`, `decimals = 6` (divisor = 10┬╣ŌüĖ).
2. User calls `ft_transfer_call` with `amount = 5 ├Ś 10┬╣ŌüĘ` (0.5 units in NEAR precision), `fee = 0`.
3. `init_transfer` passes the only guard (`fee=0 < amount=5├Ś10┬╣ŌüĘ`). Tokens are burned via `burn_tokens_if_needed`.
4. Transfer message is stored in `pending_transfers`.
5. Trusted relayer calls `sign_transfer`.
6. `normalize_amount(5├Ś10┬╣ŌüĘ, {24, 6}) = 5├Ś10┬╣ŌüĘ / 10┬╣ŌüĖ = 0` (floor division).
7. `require!(0 > 0, BridgeError::InvalidAmountToTransfer)` panics.
8. No callback fires, no refund is issued, transfer message stays in `pending_transfers`.
9. User's tokens are permanently burned with no recovery path. [4](#0-3) [6](#0-5) [2](#0-1)
