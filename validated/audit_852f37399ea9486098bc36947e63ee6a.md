### Title
Acceptance of `EvmFinality::Latest` / `SolanaFinality::Processed` Enables Forged Foreign-Chain Verification via Block Reorganization — (File: `crates/foreign-chain-inspector/src/evm/inspector.rs`)

---

### Summary

The MPC foreign-chain verification pipeline accepts caller-supplied finality levels including `EvmFinality::Latest` and `SolanaFinality::Processed` — the weakest, non-finalized states — without any minimum-finality enforcement at the contract or node layer. This is the direct analog of the reported "spot price" vulnerability: the system reads and signs over an instantaneous, reorganizable chain state instead of a robust, finalized one. An unprivileged caller can obtain a valid MPC threshold signature attesting to a foreign transaction that is subsequently reorganized out of existence, enabling a double-spend against any bridge contract that consumes the signed attestation.

---

### Finding Description

`EvmFinality` is a caller-controlled enum with three variants: `Latest`, `Safe`, and `Finalized`. [1](#0-0) 

`SolanaFinality` similarly exposes `Processed`, `Confirmed`, and `Finalized`. [2](#0-1) 

The on-chain entry point `verify_foreign_transaction` validates only that the target chain is supported and the domain exists. It performs **no check on the finality level** supplied in the request. [3](#0-2) 

On the node side, `EvmInspector::verify_finality_level` maps `EthereumFinality::Latest` directly to `FinalityTag::Latest` and passes it to `eth_getBlockByNumber`. It then checks only that the returned "latest" head block number is ≥ the receipt's block number — a pure spot check against the current chain tip. [4](#0-3) 

The subsequent `verify_block_is_canonical` call also operates at the moment of the RPC query; it cannot guarantee the block remains canonical after the query returns. [5](#0-4) 

Once both checks pass, the MPC nodes sign a `ForeignTxSignPayloadV1` that encodes the full request (including `finality: Latest`) and the extracted values. [6](#0-5) 

The signed payload is then returned on-chain as a valid threshold signature. The finality level is embedded in the payload, but there is no protocol-level enforcement preventing `Latest` from being used, and no warning is emitted.

The conversion layer confirms `Latest` is fully round-tripped without rejection: [7](#0-6)

### Impact Explanation

The MPC network's primary stated use case is the Omnibridge inbound flow, where Chain Signatures attest that a foreign transaction **finalized successfully**. [8](#0-7) 

When `EvmFinality::Latest` is used:

1. The MPC network produces a valid threshold signature over `(request={finality:Latest, tx_id:X}, values={block_hash:H})`.
2. The attacker presents this signature to a bridge contract on NEAR to claim inbound funds.
3. A miner/validator reorganizes the EVM chain, removing transaction `X` from the canonical chain.
4. The MPC signature remains cryptographically valid — it was produced correctly — but it now attests to a transaction that no longer exists.
5. The attacker has claimed bridge funds for a non-existent deposit: a double-spend.

This matches the allowed impact: **High — forged foreign-chain verification that causes double-spend / invalid bridge execution conditions.**

---

### Likelihood Explanation

- `EvmFinality::Latest` is a first-class, publicly documented variant in the contract ABI, accessible to any unprivileged NEAR account that can call `verify_foreign_transaction` with a 1 yoctoNEAR deposit.
- No governance vote, operator permission, or threshold collusion is required.
- EVM chains (especially L2s and sidechains like Polygon, BNB, HyperEVM, Abstract) have shallow finality windows where reorgs are realistic.
- The attack requires only that the attacker can influence or wait for a reorg on the target chain — a well-known capability for miners/validators and MEV actors.

**Likelihood: High.**

---

### Recommendation

1. **Enforce a minimum finality level in the contract.** Reject `EvmFinality::Latest` and `SolanaFinality::Processed` in `verify_foreign_transaction` before the request is enqueued. Only `Safe`/`Finalized` (EVM) and `Confirmed`/`Finalized` (Solana) should be accepted for bridge-security use cases.
2. **Alternatively, enforce at the node layer.** In `verify_finality_level`, treat `EthereumFinality::Latest` as an error and return `ForeignChainInspectionError::NotFinalized` rather than proceeding.
3. **Document the security model clearly** if `Latest` is intentionally kept for non-bridge use cases, and add a contract-level `DomainConfig` flag that restricts which finality levels are permitted per domain.

    async fn verify_finality_level(
        &self,
        receipt_block_number: U64,
        finality: EthereumFinality,
    ) -> Result<(), ForeignChainInspectionError> {
        let finality_tag = match finality {
            EthereumFinality::Finalized => FinalityTag::Finalized,
            EthereumFinality::Safe => FinalityTag::Safe,
            EthereumFinality::Latest => FinalityTag::Latest,
        };
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Tag(finality_tag),
            ReturnFullTransactionObjects::from(false),
        );
        let head: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        if head.number < receipt_block_number {
            return Err(ForeignChainInspectionError::NotFinalized);
        }
        Ok(())
    }

    async fn verify_block_is_canonical(
        &self,
        receipt_block_number: U64,
        receipt_block_hash: H256,
    ) -> Result<(), ForeignChainInspectionError> {
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Number(receipt_block_number),
            ReturnFullTransactionObjects::from(false),
        );
        let canonical: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        let hash_matches = canonical.hash == receipt_block_hash;
        let height_matches = canonical.number == receipt_block_number;
        if !hash_matches || !height_matches {
            return Err(ForeignChainInspectionError::NonCanonicalBlock {
                block_number: receipt_block_number.as_u64(),
                receipt_hash: receipt_block_hash.into(),
                canonical_hash: canonical.hash.into(),
            });
        }
        Ok(())
    }
