### Title
Single Attested Participant Can Forge CKD Response for `AppPublicKey` Variant, Bypassing Threshold Requirement — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract performs **no cryptographic verification** of the CKD response when the request uses the `AppPublicKey` (privately verifiable, legacy) variant. A single malicious attested participant can submit a fabricated `CKDResponse` for any pending `AppPublicKey` CKD request, bypassing the threshold cooperation requirement and delivering attacker-controlled key material to the user.

---

### Finding Description

In `respond_ckd`, the contract conditionally verifies the response only for the `AppPublicKeyPV` variant: [1](#0-0) 

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← empty: no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For `AppPublicKey`, the match arm is intentionally empty — no on-chain verification is performed. The contract then immediately resolves all queued yields with whatever `CKDResponse` was submitted: [2](#0-1) 

Because `resolve_yields_for` drains the **entire** fan-out queue on the first successful call, the first attested participant to call `respond_ckd` wins — honest or malicious.

The `AppPublicKey` variant is still actively supported as the "legacy" path: [3](#0-2) 

A malicious attested participant can:
1. Monitor the NEAR blockchain for pending `CKDRequest` entries using `AppPublicKey`.
2. Immediately call `respond_ckd` with a crafted `CKDResponse { big_y: attacker_point, big_c: ElGamal_encrypt(attacker_chosen_key, user_app_public_key) }`.
3. The contract accepts it (no verification) and delivers the fake response to the user via `resolve_yields_for`.
4. The user receives attacker-controlled key material.

Since the user's `app_public_key` is public (submitted on-chain in the request), the attacker can encrypt any value under it, producing a `big_c` that the user can successfully decrypt — getting the attacker's chosen key instead of the MPC-derived key.

The contrast with `AppPublicKeyPV` is explicit in the contract README: [4](#0-3) 

The `AppPublicKeyPV` variant was specifically designed to enable on-chain verification. For `AppPublicKey`, no such protection exists.

---

### Impact Explanation

This is a **critical bypass of the threshold requirement for confidential key derivation**. The security invariant of CKD is that the output requires threshold-many nodes to cooperate. For `AppPublicKey` requests, a single malicious attested participant can substitute its own output without any threshold cooperation. The user receives an attacker-controlled key instead of the MPC-derived key. If the attacker encrypts a key they know under the user's `app_public_key`, they can later use that key to decrypt any data the user encrypts with it — breaking the confidentiality guarantee of CKD entirely.

This maps to: **Critical — Unauthorized confidential key derivation output without the required participant authorization** and **Bypass of threshold-signature requirements**.

---

### Likelihood Explanation

Any single attested participant can execute this attack. The attacker needs only to:
1. Be an attested participant (a legitimate node in the network, below the signing threshold).
2. Monitor for pending `AppPublicKey` CKD requests on-chain.
3. Submit a fake response before the honest leader node.

No threshold cooperation, no privileged access, and no key material is required beyond being an attested participant. The attack is a simple race condition against the honest leader's `respond_ckd` submission. [5](#0-4) 

---

### Recommendation

- **Deprecate `AppPublicKey`** and require `AppPublicKeyPV` for all new CKD requests, since only `AppPublicKeyPV` enables on-chain verification via `ckd_output_check`.
- Alternatively, implement a **multi-response accumulation mechanism** where threshold-many nodes must submit matching `CKDResponse` values before the result is accepted and delivered.
- At minimum, document clearly that `AppPublicKey` requests carry no on-chain protection against single-node forgery, so integrators can make an informed choice.

---

### Proof of Concept

1. User submits `request_app_private_key` with `AppPublicKey` variant (app_public_key = BLS12-381 G1 point `P`).
2. Malicious attested participant observes the pending `CKDRequest` on-chain.
3. Attacker crafts `CKDResponse { big_y: G1_point, big_c: ElGamal_encrypt(attacker_known_key, P) }`.
4. Attacker calls `respond_ckd(request, fake_response)` before honest nodes submit.
5. Contract accepts — `AppPublicKey` branch is empty, no check fires.
6. `resolve_yields_for` drains the queue, delivering `fake_response` to the user.
7. User decrypts `big_c` using their secret key and obtains `attacker_known_key`.
8. User uses `attacker_known_key` believing it is the MPC-derived key; attacker knows this key and can use it to decrypt any data the user subsequently encrypts with it. [6](#0-5)

```rust
        let signer = Self::assert_caller_is_signer();

        if !self.protocol_state.is_running_or_resharing() {

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }

        }
```

```rust
/// Resume every yield queued for `request` with `response_bytes`, draining the
/// fan-out map in one pass. Returns `Err(RequestNotFound)` if the map held no entry.
///
/// Resuming a yield that has already timed out is a no-op at the SDK level.
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
```
