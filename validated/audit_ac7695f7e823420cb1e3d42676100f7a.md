### Title
`respond_verify_foreign_tx` Does Not Bind `payload_hash` to the Submitted `request`, Enabling Cross-Request Response Replay - (File: `crates/contract/src/lib.rs`)

### Summary

In `respond_verify_foreign_tx`, the contract verifies only that `response.signature` is a valid MPC signature over `response.payload_hash`, but never checks that `payload_hash` is the hash of a `ForeignTxSignPayload` derived from the `request` argument supplied in the same call. A single Byzantine MPC participant (below the signing threshold) can take a legitimately-produced MPC-signed response for one pending request and submit it as the response for a different pending request, permanently consuming that request's yield with wrong data.

### Finding Description

`respond_verify_foreign_tx` in `crates/contract/src/lib.rs` (lines 692–754) performs the following checks:

1. Caller is an attested participant (`assert_caller_is_attested_participant_and_protocol_active`).
2. `response.signature` is a valid ECDSA signature over `response.payload_hash` under the MPC root public key.
3. Calls `resolve_yields_for(&request, serialized_response)` to drain all queued yields for `request`. [1](#0-0) 

What is **never** checked: that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request: request.request, values: <any> }))`. The `payload_hash` is accepted as-is, as long as the signature over it is valid under the MPC root key.

The `ForeignTxSignPayload` that MPC nodes sign is:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request: ForeignChainRpcRequest, values: Vec<ExtractedValue> }))
``` [2](#0-1) 

Because the `request` field inside the signed payload is the `ForeignChainRpcRequest` (tx_id, extractors, finality, etc.), a response signed for Request A commits to Request A's transaction data. However, the contract never recomputes the expected hash from the `request` parameter to confirm the response belongs to it.

The pending-request map key is `VerifyForeignTransactionRequest { request, domain_id, payload_version }` — it does not include the caller's account ID (by design, for fan-out). Two different callers submitting different requests (different `tx_id`, different `confirmations`, etc.) produce different map keys. [3](#0-2) 

A Byzantine MPC participant can:

1. Observe that Request B (`tx_id=Y`) is pending in the contract.
2. Obtain a valid MPC-signed response `response_A = { payload_hash_A, sig_A }` produced for Request A (`tx_id=X`). This response is either already on-chain (from a prior `respond_verify_foreign_tx` call) or visible to the participant during the MPC signing protocol.
3. Call `respond_verify_foreign_tx(request=B, response=response_A)`.
4. The contract verifies `sig_A` over `payload_hash_A` → valid (it is a genuine MPC signature).
5. `resolve_yields_for` drains Request B's yield queue with `response_A`. [4](#0-3) 

Request B's yield is now permanently consumed. The bridge contract waiting on Request B receives `{ payload_hash_A, sig_A }`. When it reconstructs the expected payload hash from its own request data (Request B), the hashes do not match, and the bridge operation fails. The yield cannot be retried. [5](#0-4) 

### Impact Explanation

A single Byzantine MPC participant (below the signing threshold) can permanently fail any pending `verify_foreign_transaction` request by replaying a valid MPC response from a different request. The victim's yield is consumed with wrong data; the bridge contract's `ForeignChainSignatureVerifier::verify_signature` rejects the mismatched `payload_hash`, and the bridge operation fails permanently with no retry path. If the bridge contract locked funds on the foreign chain before calling `verify_foreign_transaction`, those funds may be permanently frozen.

This matches: **Medium — request-lifecycle manipulation that breaks production safety/accounting invariants.**

### Likelihood Explanation

- Requires only **one** Byzantine attested MPC participant (below signing threshold).
- No key forgery is needed; the attacker reuses a legitimately-produced MPC signature.
- Valid responses for other requests are observable on-chain (prior `respond_verify_foreign_tx` calls) or visible to the participant during the MPC signing round.
- The attacker can front-run the legitimate response submission for Request B.

### Recommendation

In `respond_verify_foreign_tx`, recompute the expected `payload_hash` from the `request` parameter and the `values` embedded in the response, and assert it matches `response.payload_hash` before resolving the yield. Concretely:

```rust
// Recompute expected hash from the request key and the values in the response
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.values.clone(), // values must be included in the response DTO
});
let expected_hash = expected_payload.compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Alternatively, include the `VerifyForeignTransactionRequest` key (or its hash) inside the signed `ForeignTxSignPayload` so that a response signed for Request A is cryptographically invalid for Request B.

### Proof of Concept

```
// Setup: two different pending requests
contract.verify_foreign_transaction(request_args_A); // tx_id=X, confirmations=6
contract.verify_foreign_transaction(request_args_B); // tx_id=Y, confirmations=1

let response_A = VerifyForeignTransactionResponse {
    payload_hash: SHA256(borsh(ForeignTxSignPayload{request_A, values_A})),

// Byzantine participant submits response_A as the answer for request_B
contract.respond_verify_foreign_tx(request_B, response_A);

// → resolve_yields_for drains request_B's yield with response_A

// ForeignChainSignatureVerifier reconstructs expected_hash from request_B → mismatch

    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
