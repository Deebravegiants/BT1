Audit Report

## Title
Unconditional `updatedAt` advance in `updateReward` modifier when `totalKernelStaked == 0` permanently freezes streamed rewards - (File: `contracts/KERNEL/KernelDepositPool.sol`)

## Summary

The `updateReward` modifier in `KernelDepositPool` unconditionally advances `updatedAt = lastTimeRewardApplicable()` even when `totalKernelStaked == 0`. Because `rewardPerToken()` returns the stored accumulator unchanged when supply is zero, any elapsed time during a zero-supply window is consumed without crediting any staker. Rewards that streamed during that gap are permanently stranded in the contract and cannot be claimed by any address.

## Finding Description

The `updateReward` modifier executes two unconditional assignments before any function body runs:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();   // returns stored value unchanged when supply == 0
    updatedAt = lastTimeRewardApplicable();    // always advances regardless of supply
    ...
}
``` [1](#0-0) 

`rewardPerToken()` correctly short-circuits when supply is zero, returning `rewardPerTokenStored` unchanged:

```solidity
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;
    }
    ...
}
``` [2](#0-1) 

However, the modifier still executes `updatedAt = lastTimeRewardApplicable()` unconditionally. This means the time interval `[updatedAt_before, now]` is permanently consumed: `rewardPerTokenStored` does not increase, yet `updatedAt` jumps forward, discarding `rewardRate * elapsed` worth of rewards.

The zero-supply state is reachable via `initiateWithdrawal`, which decrements `totalKernelStaked` after the modifier has already run:

```solidity
function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
    ...
    balanceOf[msg.sender] -= _amount;
    totalKernelStaked -= _amount;   // can reach 0 after modifier has already advanced updatedAt
    ...
}
``` [3](#0-2) 

The only on-chain guard against this scenario is in `notifyRewardAmount`, which reverts if `totalKernelStaked == 0` at the time a new reward period is started:

```solidity
if (totalKernelStaked == 0) revert NoStakedTokens();
``` [4](#0-3) 

This guard does not prevent stakers from exiting to zero during an already-active reward period. The contract's own NatSpec explicitly acknowledges the issue and relies entirely on off-chain operational discipline:

```
* @dev If `totalKernelStaked` ever hits zero during a reward distribution window, any remaining rewards
*      for that period will stay locked in the contract. In this deployment, we're avoiding this issue by
*      ensuring there are always some tokens staked...
``` [5](#0-4) 

No on-chain mechanism prevents the last staker from calling `initiateWithdrawal` for their full balance during an active reward period.

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

When `totalKernelStaked` drops to zero mid-period, every subsequent call that triggers `updateReward` (including the next `stake`) advances `updatedAt` without accumulating `rewardPerTokenStored`. The rewards for the entire zero-supply gap are permanently stranded in the contract balance. They are not recoverable by any user or admin function, as no sweep or recovery mechanism exists in the contract.

## Likelihood Explanation

Any unprivileged staker who is the last remaining depositor can trigger this by calling `initiateWithdrawal` for their full balance during an active reward period. No special role, governance action, or external dependency is required. The scenario is realistic whenever staking participation is low or a single large staker exits. It is repeatable across multiple reward periods.

## Recommendation

Do not advance `updatedAt` when `totalKernelStaked == 0`. The modifier should preserve the time window so rewards are credited once stakers return:

```diff
modifier updateReward(address _account) {
+   if (totalKernelStaked > 0) {
        rewardPerTokenStored = rewardPerToken();
        updatedAt = lastTimeRewardApplicable();
+   }

    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }

    _;
}
```

This ensures the elapsed window is only consumed when there are stakers to receive it, preserving pending rewards for future participants.

## Proof of Concept

1. Admin calls `notifyRewardAmount` with `rewardRate > 0` and `finishAt = now + 7 days` (requires `totalKernelStaked > 0`, so Alice must already be staked).
2. Alice is the only staker (`totalKernelStaked = Alice's balance`).
3. Alice calls `initiateWithdrawal(fullBalance)`:
   - `updateReward` runs first: `totalKernelStaked > 0` at modifier entry, so `rewardPerToken()` accumulates correctly up to this point, and `updatedAt = block.timestamp`.
   - Function body then executes: `totalKernelStaked -= _amount` → becomes 0.
4. 3 days pass. `rewardRate * 3 days` worth of rewards stream but are credited to no accumulator.
5. Bob calls `stake(amount)`:
   - `updateReward` runs: `totalKernelStaked == 0` at modifier entry (Bob not yet added), so `rewardPerToken()` returns `rewardPerTokenStored` unchanged, but `updatedAt` advances to `block.timestamp`.
   - The 3-day reward window is permanently consumed.
   - Function body then executes: `totalKernelStaked += _amount`.
6. Bob calls `getReward()` — receives only rewards from step 5 onward, not the 3-day gap.
7. The contract holds `rewardRate * 3 days` tokens that can never be claimed by any address.

**Foundry test plan:** Deploy `KernelDepositPool`, call `notifyRewardAmount` with Alice staked, have Alice call `initiateWithdrawal` for full balance, `vm.warp` forward 3 days, have Bob `stake`, then assert `earned(Bob) < rewardRate * 3 days` and that the contract's reward token balance exceeds all claimable rewards.
