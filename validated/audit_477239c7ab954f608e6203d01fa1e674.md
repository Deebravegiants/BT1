### Title
Unbounded Iteration Over All Registered DReps During Governance Proposal Validation - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Certs.hs`)

---

### Summary

The `updateDormantDRepExpiry` function performs an unbounded `Map.map` over **every registered DRep** in the ledger state whenever any transaction contains at least one governance proposal and `numDormantEpochs > 0`. There is no protocol-enforced upper bound on the number of registered DReps. This creates an asymmetry: a minimally-sized governance proposal transaction can force all validating nodes to perform O(n) work proportional to the total number of DReps in the global ledger state, not proportional to the transaction's own size or declared resource budget.

---

### Finding Description

`updateDormantDRepExpiry` in `Certs.hs` applies `Map.map updateExpiry` to the entire `vsDReps` map unconditionally when `numDormantEpochs ≠ 0`:

```haskell
updateDormantDRepExpiry currentEpoch vState =
  if numDormantEpochs == EpochNo 0
    then vState
    else
      vState
        & vsNumDormantEpochsL .~ EpochNo 0
        & vsDRepsL %~ Map.map updateExpiry   -- iterates ALL DReps
``` [1](#0-0) 

This is invoked by `updateDormantDRepExpiries`, which fires whenever the transaction body contains any governance proposal:

```haskell
updateDormantDRepExpiries tx currentEpoch =
  let hasProposals = not . OSet.null $ tx ^. bodyTxL . proposalProceduresTxBodyL
   in if hasProposals
        then certVStateL %~ updateDormantDRepExpiry currentEpoch
        else id
``` [2](#0-1) 

This is called in two places during transaction validation:

1. In `conwayCertsTransition` (the `CERTS` rule, pre-hardfork path): [3](#0-2) 

2. In `conwayLedgerTransition` (the `LEDGER` rule, post-hardfork path): [4](#0-3) 

The `vsDReps` map lives in `VState`:

```haskell
data VState era = VState
  { vsDReps :: !(Map (Credential DRepRole) DRepState)
  , ...
  }
``` [5](#0-4) 

There is **no protocol parameter** that caps the total number of registered DReps. The only barrier is the `ppDRepDeposit` economic cost. The Conway `PParams` record contains no `maxDReps` field: [6](#0-5) 

The parallel `updateVotingDRepExpiries` function (also called in the same rules) is bounded by the transaction's own `votingProcedures` map — it only touches DReps that voted in the current transaction — and is therefore safe: [7](#0-6) 

The contrast is exact: `updateVotingDRepExpiries` is bounded by transaction content; `updateDormantDRepExpiry` is bounded only by the global ledger state.

---

### Impact Explanation

**Medium — Attacker-controlled governance proposal transactions exceed intended validation limits.**

The `maxTxSize` protocol parameter bounds the byte size of a transaction, and `maxTxExUnits` / `maxBlockExUnits` bound Plutus script execution. Neither bounds the ledger-state traversal triggered by `updateDormantDRepExpiry`. A single minimal governance proposal transaction (well within `maxTxSize`) forces every validating node to traverse and rewrite the entire DRep map. If the DRep population is large, this traversal can consume CPU time disproportionate to the transaction's declared resource footprint, causing block validation to exceed the slot time budget. Nodes that cannot complete validation within the slot window may orphan the block or fall behind the chain tip, constituting a resource-limit violation outside design parameters.

---

### Likelihood Explanation

**Low-to-medium.** An attacker must register a large number of DReps, each requiring a `ppDRepDeposit` payment (currently 500 ADA on mainnet). This is a capital-intensive setup. However, the attack is amplified by the dormancy mechanism: the attacker need only wait for a period with no governance proposals (incrementing `numDormantEpochs`), then submit a single proposal to trigger the full traversal. The attack is repeatable as long as the attacker's DReps remain registered. The cost is a one-time deposit (refundable upon DRep deregistration), not a recurring fee.

---

### Recommendation

1. **Add a protocol parameter** (e.g., `maxDReps :: Word32`) to cap the total number of registered DReps, analogous to `maxCollateralInputs` for collateral.
2. **Alternatively, apply the pulsing pattern** already used for reward calculation (`PulsingRewUpdate` / `RewardPulser`) to spread the DRep expiry update across multiple blocks rather than forcing it synchronously during a single transaction's validation.
3. **At minimum**, add a ledger-rule check that rejects governance proposal transactions when `numDormantEpochs > 0` and `|vsDReps|` exceeds a safe threshold, or move the dormant-expiry update to the epoch boundary (where unbounded work is already expected and budgeted).

---

### Proof of Concept

1. Register **N** DReps (paying N × `ppDRepDeposit` ADA). No protocol rule prevents this.
2. Allow **k > 0** epochs to pass with zero governance proposals, so `numDormantEpochs` accumulates to `k`.
3. Submit a governance proposal transaction (minimum valid size, well within `maxTxSize`).
4. Every validating node executes `updateDormantDRepExpiry`, which calls `Map.map updateExpiry` over all **N** DRep entries in `vsDReps` — O(N) work triggered by an O(1)-sized transaction field (`proposalProcedures` containing a single proposal).
5. With N sufficiently large, the per-transaction validation cost grows unboundedly relative to the transaction's byte size, exceeding the implicit per-slot validation budget.

The root cause is structurally identical to the reported M-03: a loop over an unbounded collection (all DReps) is triggered by a transaction-level signal (presence of any proposal), with no offset/length guard and no upper bound on the collection size.
