Audit Report

## Title
Sandwich Attack on `updateRSETHPrice()` Enables Theft of Accrued Yield via `instantWithdrawal` — (File: contracts/LRTOracle.sol, contracts/LRTWithdrawalManager.sol)

## Summary

`LRTOracle.updateRSETHPrice()` is a permissionless public function that updates the stored `rsETHPrice` to reflect accrued restaking rewards. Because `LRTDepositPool.depositETH()` mints rsETH at the current stale stored price and `LRTWithdrawalManager.instantWithdrawal()` redeems rsETH at the updated stored price with no block-delay requirement, an attacker can sandwich any `updateRSETHPrice()` transaction in the public mempool to capture the yield increment that should accrue proportionally to all existing rsETH holders.

## Finding Description

**Root cause:** Three independently correct design decisions compose into an exploitable sequence:

1. `updateRSETHPrice()` is `public whenNotPaused` with no access restriction (`LRTOracle.sol:87-89`), making it observable and sandwichable from the public mempool.

2. `getRsETHAmountToMint()` in `LRTDepositPool.sol:520` computes mint amount as `(amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()` — using the **stored** (stale) `rsETHPrice`. A deposit made before `updateRSETHPrice()` executes receives more rsETH than the post-update price would grant.

3. `instantWithdrawal()` in `LRTWithdrawalManager.sol:228` calls `getExpectedAssetAmount()`, which at line 593 computes `amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset)` — using the **current** (updated) `rsETHPrice`. There is no block-number check, no deposit-time lock, and no same-block restriction.

**Exploit path:**

Let `T` = total ETH in protocol (including accrued but unpriced rewards), `S` = rsETH supply, `P_old` = stale stored price, `P_new_true = T/S` = true post-update price.

- **Front-run:** Attacker calls `depositETH{value: X}()`. Minted rsETH = `X / P_old`. The attacker's X ETH is added to `totalETHInProtocol`.
- **Target tx:** `updateRSETHPrice()` executes. New price: `P_new = (T + X) / (S + X/P_old)`. Since `T > P_old * S` (rewards have accrued), `P_new > P_old`.
- **Back-run:** Attacker calls `instantWithdrawal(ETH, X/P_old)`. Receives `(X/P_old) * P_new` ETH minus `instantWithdrawalFee`.
- **Profit:** `X * (P_new/P_old - 1) = X * (T - P_old*S) / (P_old*S + X)`. As X grows large relative to `P_old*S`, profit approaches the full reward amount `T - P_old*S`.

**Why existing guards fail:**

- `pricePercentageLimit` (`LRTOracle.sol:256-266`): This guard reverts non-manager callers if the price increase exceeds the threshold. The attacker is sandwiching the *manager's or bot's* call to `updateRSETHPrice()`, not calling it themselves. The manager's call succeeds regardless of the limit (line 263 exempts `MANAGER` role callers). The attacker's own transactions (`depositETH`, `instantWithdrawal`) never invoke this guard.
- `instantWithdrawalFee` (`LRTWithdrawalManager.sol:237`): Capped at 10% (`feeBasisPoints > 1000` reverts). For a yield increment of even 0.5%, a sufficiently large deposit still yields net profit after a 0.1% fee.
- `onlyInstantWithdrawalAllowed` modifier (`LRTWithdrawalManager.sol:78-81`): Requires `isInstantWithdrawalEnabled[asset]` to be true. This is a manager-set flag, not a per-transaction guard. When the feature is live (its intended operational state), the attack path is fully open.
- `CantInstantWithdrawMoreThanAvailable` check (`LRTWithdrawalManager.sol:231-233`): Limits withdrawal to vault liquidity, bounding attack scale but not preventing it.

## Impact Explanation

**High — Theft of unclaimed yield.**

Every `updateRSETHPrice()` call that reflects accrued EigenLayer restaking rewards can be sandwiched to extract the full yield increment from existing rsETH holders. Legitimate long-term stakers receive zero yield for that period. The attack is repeatable on every price-update cycle, constituting a persistent, automated drain on protocol yield. This maps exactly to the allowed impact: *High — Theft of unclaimed yield*.

## Likelihood Explanation

**Medium.**

- `updateRSETHPrice()` is permissionless and expected to be called regularly by bots or keepers from the public mempool, making it reliably observable.
- `instantWithdrawal` is a live, manager-enabled feature (`isInstantWithdrawalEnabled[asset]`). When enabled for ETH (the primary asset), the full attack path is open to any EOA or contract.
- The `LRTUnstakingVault` must hold sufficient ETH for the back-run withdrawal; this is an operational prerequisite the protocol maintains to service normal instant withdrawals.
- MEV infrastructure (flashbots, private relays) makes atomic sandwich execution routine on Ethereum mainnet.
- The `instantWithdrawalFee` reduces but does not eliminate profitability; at 0.1% fee and a 1% yield increment, a 1000 ETH deposit yields ~9 ETH net profit.

## Recommendation

1. **Same-block deposit lock:** Record `block.number` at deposit time per user (in `LRTDepositPool` or rsETH token), and revert in `instantWithdrawal` if the caller deposited in the same block.
2. **Restrict `updateRSETHPrice()` to a privileged role** and submit price-update transactions via a private relay (e.g., Flashbots Protect) to remove mempool visibility.
3. **Time-weighted price:** Delay the effect of `updateRSETHPrice()` by at least one block, or apply the price update only at the start of the next block, so front-running within the same block is structurally impossible.

## Proof of Concept

**Foundry fork test outline:**

```solidity
// Fork mainnet at block N where EigenLayer rewards have accrued but rsETHPrice is stale.
// 1. Record rsETHPrice = P_old = lrtOracle.rsETHPrice()
// 2. Attacker calls lrtDepositPool.depositETH{value: 1000 ether}("") 
//    → receives 1000e18 / P_old rsETH at stale price
// 3. Call lrtOracle.updateRSETHPrice()
//    → rsETHPrice updated to P_new > P_old
// 4. Attacker calls lrtWithdrawalManager.instantWithdrawal(ETH_TOKEN, rsETHReceived, "")
//    → receives rsETHReceived * P_new / 1e18 ETH (minus fee)
// 5. Assert: attacker ETH balance > 1000 ether (net of fee)
//    Assert: (attackerFinalBalance + fee) - 1000 ether ≈ 1000e18 * (P_new - P_old) / P_old
//    Assert: existing rsETH holders' redeemable value decreased by the stolen amount
```

Steps 2–4 can be executed within a single `vm.roll` block to confirm no block-delay protection exists. The profit is bounded by `min(rewardAmount, vaultLiquidity)` and reduced by `instantWithdrawalFee`, both of which are verifiable on-chain.
