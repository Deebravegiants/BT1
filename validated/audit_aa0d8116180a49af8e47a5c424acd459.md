Audit Report

## Title
ETH Branch of `_checkIfDepositAmountExceedesCurrentLimit` Omits `amount`, Allowing Bypass of `depositLimitByAsset` Cap - (File: contracts/LRTDepositPool.sol)

## Summary

`_checkIfDepositAmountExceedesCurrentLimit` at line 676 of `contracts/LRTDepositPool.sol` evaluates the ETH deposit limit as `totalAssetDeposits > limit`, omitting the incoming `amount` from the comparison. The ERC-20 branch correctly evaluates `totalAssetDeposits + amount > limit`. As a result, any unprivileged depositor can push total ETH deposits arbitrarily above the configured `depositLimitByAsset` cap in a single `depositETH` call, as long as the running total has not already crossed the limit before the transaction.

## Finding Description

`_checkIfDepositAmountExceedesCurrentLimit` (L676–682) branches on whether the asset is native ETH:

```solidity
// contracts/LRTDepositPool.sol L676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount never added
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));  // ← correct for ERC-20
}
``` [1](#0-0) 

The ETH branch returns `false` (i.e., "not exceeded") for any deposit as long as the running total has not already crossed the limit, regardless of how large the incoming `amount` is. `depositETH` (L76–93) calls `_beforeDeposit` (L648–670), which calls this check and reverts with `MaximumDepositLimitReached` only when the function returns `true`. [2](#0-1) 

Because `amount` is never included in the ETH branch, the revert never fires for a deposit that would push the total over the limit. There are no other guards in `depositETH` or `_beforeDeposit` that compensate for this omission. [3](#0-2) 

## Impact Explanation

`depositLimitByAsset[ETH_TOKEN]` is the protocol's safety cap on total ETH exposure. Because the check omits `amount`, a single depositor can send an arbitrarily large ETH value and the cap is never enforced. The protocol mints rsETH for the full amount and forwards ETH to EigenLayer beyond the intended ceiling. No funds are stolen or frozen, but the contract fails to enforce the deposit-limit guarantee it advertises. This matches the **Low — contract fails to deliver promised returns** impact tier.

## Likelihood Explanation

The entry point is the public, payable `depositETH` function — no role, whitelist, or special condition is required. [4](#0-3) 

Any depositor who holds enough ETH and observes that `totalAssetDeposits` is below the limit can trigger this in a single transaction. The condition is trivially observable on-chain via `getAssetCurrentLimit` or `getTotalAssetDeposits`. Likelihood is **High**. [5](#0-4) 

## Recommendation

Add `amount` to the ETH branch, mirroring the ERC-20 branch:

```diff
  if (asset == LRTConstants.ETH_TOKEN) {
-     return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
+     return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
  }
``` [6](#0-5) 

## Proof of Concept

**Setup:**
- `depositLimitByAsset[ETH_TOKEN] = 1000 ether`
- `getTotalAssetDeposits(ETH_TOKEN) = 999 ether`

**Steps:**
1. Attacker calls `depositETH{value: 500 ether}(0, "")`.
2. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500 ether)`.
3. ETH branch evaluates `999 ether > 1000 ether` → `false`.
4. Check passes; 500 ETH is accepted and rsETH is minted.
5. Total ETH in protocol is now `1499 ether`, ~49.9% above the configured cap.

**Equivalent Foundry test sketch:**
```solidity
function test_ethDepositBypassesLimit() public {
    // set limit to 1000 ether, pre-fill to 999 ether via prior deposits
    vm.deal(attacker, 500 ether);
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 500 ether}(0, "");
    assertGt(lrtDepositPool.getTotalAssetDeposits(ETH_TOKEN), 1000 ether);
}
```

The same call with any ERC-20 LST would correctly evaluate `999 + 500 > 1000` → `true` and revert with `MaximumDepositLimitReached`. [1](#0-0)
