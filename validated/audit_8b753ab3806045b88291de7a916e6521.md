### Title
`respond_verify_foreign_tx` Does Not Bind `response.payload_hash` to the Pending `request`, Enabling Single-Participant Cross-Request Replay for Forged Foreign-Chain Verification - (File: `crates/contract/src/lib.rs`)

### Summary

The `respond_verify_foreign_tx` function verifies that `response.signature` is valid for `response.payload_hash` under the root public key, but never verifies that `response.payload_hash` was actually derived from the pending `request` data. A single Byzantine MPC participant (below threshold) can replay any previously produced `(payload_hash, signature)` pair from a prior on-chain `verify_foreign_transaction` response to resolve a completely different pending request with fabricated verification data, enabling forged foreign-chain verification and invalid bridge execution.

### Finding Description

In `respond_verify_foreign_tx` (`crates/contract/src/lib.rs` lines 691–754), the contract performs the following checks:

1. Caller is an attested participant (`assert_caller_is_attested_participant_and_protocol_active`)
2. Protocol is running or resharing
3. `accept_requests` is true
4. Retrieves the root public key for `request.domain_id`
5. Verifies `response.signature` is valid for `response.payload_hash` under the **root key** (no per-user tweak)
