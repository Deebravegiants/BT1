### Title
`peer_nonce` Tracking Keyed Only by `PeerId`, Not `(PeerId, CommitteeId)`, Allows Cross-Committee Nonce Pollution to Silently Drop Legitimate P2P Consensus Messages — (`crates/apollo_propeller/src/engine.rs`)

---

### Summary

The Propeller engine's replay-protection cache `peer_nonce` is a `LruCache<PeerId, u64>` keyed solely by publisher identity. When a peer participates in multiple committees simultaneously, a high nonce finalized in committee A is written back to the global `peer_nonce[publisher]` entry. Any subsequent message from the same peer in committee B that carries a lower nonce (sent earlier but delayed in transit) is then silently dropped as "too old," even though it is a fully legitimate message for committee B. The codebase itself acknowledges the defect with an explicit TODO: `// TODO(guyn): track nonces separately for each committee.`

---

### Finding Description

**Defective state structure**

```rust
// crates/apollo_propeller/src/engine.rs, line 125
// TODO(guyn): track nonces separately for each committee.
/// Nonce per peer. LRU cache is used as a passive garbage collection mechanism.
peer_nonce: LruCache<PeerId, u64>,
```

The mapping is `PeerId → u64`. The correct key is `(PeerId, CommitteeId) → u64`.

**Admission gate in `handle_unit` (lines 316–324)**

```rust
if !self.message_to_unit_tx.contains_key(&message_key) {
    let nonce = self.peer_nonce.get(&claimed_publisher).copied().unwrap_or(0);
    if nonce >= claimed_nonce {
        warn_every_n_ms!(2000, "Message nonce is too old, dropping unit …");
        return;
    }
    …
}
```

The check reads `peer_nonce[claimed_publisher]` without consulting `claimed_committee_id`. A nonce written by committee A's expiry path therefore gates admission for committee B.

**Write-back path in `handle_state_manager_message` (lines 479–491)**

```rust
for key in expired_keys {
    let new_nonce = self
        .peer_nonce
        .peek(&key.publisher)
        .copied()
        .unwrap_or(0)
        .max(key.nonce);
    self.peer_nonce.put(key.publisher, new_nonce);   // ← committee_id discarded
}
```

When a finalized message expires from the TTL cache `messages_to_ignore_units_from`, the engine advances `peer_nonce[publisher]` to `max(current, expired_nonce)`. The `key.committee_id` field is present in `MessageKey` but is never used as part of the write-back key.

**Structural parallel to the external bug**

| External (Rewards) | Sequencer (Propeller) |
|---|---|
| `lastEpochClaimedStaker[address]` | `peer_nonce[publisher]` |
| Claiming token A updates the epoch for all tokens | Finalizing a message in committee A updates the nonce for all committees |
| Staker cannot claim token B for already-claimed epochs | Publisher's messages in committee B are dropped as "too old" |

---

### Impact Explanation

A peer that is a member of two or more committees (normal in a multi-committee deployment) can have its legitimate messages silently dropped by the receiving node. The dropped messages are P2P consensus units (proposals, votes, or data-availability shards) that the Tendermint state machine depends on. Dropping enough of them stalls consensus for the affected height, preventing the sequencer from sequencing any transactions until a timeout forces a round change or the peer retransmits with a fresh (higher) nonce.

This maps to **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing** (the P2P admission gate rejects valid consensus content before it can influence sequencing) and to the **p2p/sync routing** pivot explicitly listed in the audit scope.

---

### Likelihood Explanation

- **Trigger is unprivileged and non-malicious.** Any honest peer that is registered in two committees and whose messages experience ordinary network reordering or delay can trigger the bug.
- **The developers already identified it.** The inline TODO at line 123 confirms the defect is known and unresolved.
- **Nonces are nanosecond timestamps.** A message sent at time T₁ in committee B and a message sent at time T₂ > T₁ in committee A will have N_B < N_A. If the committee A message is finalized and expires from the TTL cache before the committee B message arrives (possible under any non-trivial network jitter), `peer_nonce[publisher]` is set to N_A and the committee B message is rejected.

---

### Recommendation

Change `peer_nonce` to be keyed by `(PeerId, CommitteeId)`:

```rust
// Before
peer_nonce: LruCache<PeerId, u64>,

// After
peer_nonce: LruCache<(PeerId, CommitteeId), u64>,
```

Update every read and write to include the committee dimension:

```rust
// handle_unit – read
let nonce = self.peer_nonce
    .get(&(claimed_publisher, claimed_committee_id))
    .copied()
    .unwrap_or(0);

// handle_state_manager_message – write-back
self.peer_nonce.put((key.publisher, key.committee_id), new_nonce);
```

### Proof of Concept

1. Register peer `P` in `COMMITTEE_A` and `COMMITTEE_B` on the same engine instance.
2. Peer `P` broadcasts message `M_A` in `COMMITTEE_A` with nonce `T2` (a later timestamp).
3. Peer `P` broadcasts message `M_B` in `COMMITTEE_B` with nonce `T1 < T2` (an earlier timestamp, delayed in transit).
4. `M_A` is finalized and its TTL expires → `handle_state_manager_message` sets `peer_nonce[P] = T2`.
5. `M_B` arrives at the engine → `handle_unit` reads `peer_nonce[P] = T2`, sees `T2 >= T1`, and silently drops `M_B` with "Message nonce is too old."
6. The consensus layer never receives the committee B message; if this affects a quorum threshold, the height stalls.
