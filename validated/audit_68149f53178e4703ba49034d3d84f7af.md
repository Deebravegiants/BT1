### Title
Stale Plutus `CostModel` in Cached `StAnnTx` Reused After Protocol Parameter Update, Bypassing New Execution Limits and Causing Ledger Divergence - (`File: eras/shelley/impl/src/Cardano/Ledger/Shelley/API/Mempool.hs`)

### Summary

`reapplyValidatedTx` guards stale-annotation reuse only by comparing `pvMajor` (major protocol version). Protocol parameter updates — including cost-model changes — routinely occur at epoch boundaries **without** a major-version bump. The `StAnnTx` for Alonzo/Babbage/Conway eras embeds a fully-baked `PlutusWithContext` per script, each carrying a `pwcCostModel` field captured at mempool-validation time. When `reapplyValidatedTx` is called after a cost-model update, it reuses the stale `PlutusWithContext` with the old cost model, violating the `StAnnTx` safety invariant and enabling two concrete harms: (1) a transaction whose Plutus scripts would exceed the new execution limits is silently accepted, and (2) nodes that hold the transaction in their mempool diverge from nodes that do not, because the latter perform full revalidation with the current cost model.

### Finding Description

**Vulnerability class:** Stale cached value (computed once from mutable ledger state, never refreshed when that state changes) — the direct Cardano analog of the `DOMAIN_SEPARATOR` / `block.chainid` pattern.

**Step 4 — The `StAnnTx` safety invariant is violated for cost-model changes.**

The invariant documented in `EraTx` states:

> "It is critical to only store here information that satisfies the following property: if the ledger state changes in a way that makes the annotated value stale, then some other predicate check in the STS rules must independently cause the transaction to be rejected. Stale annotations must never lead to a transaction being silently accepted." [7](#0-6) 

For cost-model changes there is **no independent predicate check**. The only check is Plutus execution itself, which uses `pwcCostModel` from the cached `PlutusWithContext`. If the cost model becomes stricter (higher per-operation costs), the actual execution units computed under the new model would exceed the declared `pwcExUnits`, causing a script failure — but with the stale model the old (lower) costs are used and the check passes silently.

### Impact Explanation

**High — Deterministic ledger divergence between honest nodes.**

- Node A holds the transaction in its mempool as a `ValidatedTx` (stale `pwcCostModel` = old model). When the block arrives, `reapplyValidatedTx` reuses the stale annotation → Plutus execution passes → block accepted.
- Node B does not hold the transaction (e.g., just synced, or the tx was evicted). It calls `internalApplyTxWithValidation ValidateAll`, which calls `mkStAnnTx` fresh → Plutus execution uses the new, stricter cost model → script exceeds declared execution units → block rejected.

Two honest nodes reach different ledger states from the same valid block, satisfying the "Deterministic disagreement between honest nodes from ledger rule evaluation" impact criterion.

**Medium — Attacker-controlled transaction exceeds intended validation limits.**

A transaction sender who crafts a Plutus script consuming execution units that are valid under the current cost model but would be invalid under a stricter future model can have that transaction included in a block after the cost-model update, bypassing the new limits on nodes that hold it in their mempool.

### Likelihood Explanation

Cost-model updates are a routine part of Cardano governance (they accompany each new Plutus language version and periodic parameter updates) and do not require a major protocol version bump. The `pvMajor`-only guard in `reapplyValidatedTx` is therefore bypassed on every such update. Any transaction with Plutus scripts that was in the mempool at the time of the update is affected. No special attacker capability is required beyond submitting a standard Plutus-bearing transaction.

### Recommendation

1. **Extend the staleness guard in `reapplyValidatedTx`** to compare the full `ProtVer` (both major and minor), or compare the cost-model hash/fingerprint stored in `ValidatedTx` against the current protocol parameters. If they differ, fall through to `internalApplyTxWithValidation ValidateAll`.

2. **Store the cost-model snapshot** used to build `StAnnTx` inside `ValidatedTx` alongside `vtProtocolVersion`, and invalidate the cached annotation whenever the cost models change.

3. **Alternatively**, follow the Alonzo `internalReapplyValidatedTx` pattern (slot-number check) and extend it to also check cost-model identity, so that any protocol-parameter change that affects Plutus execution triggers full revalidation.

### Proof of Concept

1. Submit a Plutus V2 transaction `Tx` to the mempool in epoch `E`. The node validates it, producing `ValidatedTx { vtStAnnTx = stAnnTx_E, vtProtocolVersion = PV{major=9, minor=0}, ... }`. The `stAnnTx_E` contains `PlutusWithContext { pwcCostModel = CM_E, ... }` where `CM_E` is the cost model at epoch `E`.

2. At the epoch `E+1` boundary, a governance action enacts a protocol parameter update that increases Plutus V2 cost-model parameters (no major version bump; `PV{major=9, minor=1}`).

3. A block producer includes `Tx` in a block in epoch `E+1`.

4. **Node A** (has `Tx` in mempool): `reapplyValidatedTx` checks `pvMajor(PV{9,0}) == pvMajor(PV{9,1})` → `True` → calls `defaultReapplyValidatedTx` → passes `stAnnTx_E` (with `CM_E`) to the UTXOS rule → Plutus script executes under `CM_E` → passes → block accepted.

5. **Node B** (does not have `Tx` in mempool): calls `internalApplyTxWithValidation ValidateAll` → `mkStAnnTx` builds fresh `PlutusWithContext { pwcCostModel = CM_{E+1} }` → Plutus script executes under `CM_{E+1}` → actual execution units exceed declared units → `ValidationTagMismatch` / script failure → block rejected.

Nodes A and B now hold divergent ledger states from the same block, requiring a hard fork to reconcile. [8](#0-7) [9](#0-8) [3](#0-2)
