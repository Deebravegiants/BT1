### Title
`FanOut` Inspector Accepts Single-Provider Result Without Enforcing `ChainEntry.quorum` Minimum — (`File: crates/foreign-chain-inspector/src/lib.rs`)

### Summary

The `FanOut` inspector, which fans out foreign-chain RPC queries to multiple providers, does not enforce the on-chain `ChainEntry.quorum` minimum. When some providers return transient errors and only one returns a non-transient success, `FanOut` accepts that single provider's result unconditionally — regardless of what the voted-in quorum requires. A single malicious or compromised RPC provider can therefore cause a node to produce a signature share attesting to a forged foreign-chain state whenever the other providers are transiently unavailable.

### Finding Description

The design contract is explicit:

> "Each node fans the query out to its whitelisted providers for C and accepts a result only when ≥ `rpc_quorum(C)` return the same response. If fewer agree, the node errors out and produces no signature share." [1](#0-0) 

The `ChainEntry.quorum` value is voted in on-chain alongside the provider list and is the authoritative minimum agreement count. [2](#0-1) 

However, `FanOut` is constructed with only a `NonEmptyVec<Inspector>` — the quorum value is never passed in and is never checked:

```rust
pub struct FanOut<Inspector> {
    inspectors: NonEmptyVec<Inspector>,
}
``` [3](#0-2) 

The acceptance logic inside `FanOut::extract` is:

```rust
if let Some(first_values) = successes.first() {
    let all_successes_agree = successes.iter().all(|(_, v)| v == &first_values.1);
    if !all_successes_agree {
        return Err(ForeignChainInspectionError::InspectorResponseMismatch);
    }
    let (_, first) = successes.into_iter().next().expect("checked non-empty");
    return Ok(first);
}
``` [4](#0-3) 

If `successes` contains exactly **one** entry (because all other providers returned transient errors), `all_successes_agree` is trivially `true` and the single result is returned as authoritative — with no check that the count of agreeing providers meets `ChainEntry.quorum`.

Transient errors are explicitly tolerated and do not count as substantive verdicts:

```rust
Err(err) if err.is_transient() => {
    first_transient_error.get_or_insert(err);
}
``` [5](#0-4) 

`ForeignChainInspectionError::is_transient` includes `ClientError`, `RpcRequestFailed`, `NotFinalized`, and `NotEnoughBlockConfirmations` — all common real-world failure modes. [6](#0-5) 

The `FanOut` is built from the node's local config providers without ever reading or storing the on-chain quorum:

```rust
fn build_fanout<I>(
    chain_config: Option<&ForeignChainConfig>,
    new_inspector: impl Fn(...) -> anyhow::Result<I>,
) -> anyhow::Result<Option<FanOut<I>>> {
    ...
    Ok(Some(FanOut::new(inspectors)))
}
``` [7](#0-6) 

The quorum value from `ChainEntry` is never retrieved, stored, or passed anywhere in the signing flow. [8](#0-7) 

### Impact Explanation

A node that accepts a single malicious provider's forged verification result will compute a `ForeignTxSignPayload` from that forged data and produce a signature share over it. If a threshold of nodes are similarly affected — each with their other providers transiently unavailable — the MPC network produces a valid threshold signature attesting that a foreign-chain transaction (which may not exist, or may have failed) was successfully verified. This enables invalid bridge execution: an attacker can claim funds on NEAR for a foreign-chain deposit that never occurred, or for a transaction that was reverted.

This matches the allowed impact: **High — forged foreign-chain verification that causes invalid bridge execution or double-spend conditions.**

### Likelihood Explanation

The attack requires two simultaneous conditions:

1. **One controlled/compromised RPC provider** — the attacker either operates a whitelisted provider or compromises one. The on-chain whitelist is voted in by threshold participants, so a provider that was legitimately voted in can later be compromised without immediate detection.

2. **Other providers transiently unavailable** — transient failures (`ClientError`, `RpcRequestFailed`, rate-limit 429, timeout) are routine in production RPC infrastructure. The attacker does not need to actively cause them; they can simply wait for a window when other providers are degraded, or exploit rate limits by flooding them with unrelated requests just before submitting the target `verify_foreign_transaction`.

Because transient failures are common and the quorum bypass requires no threshold-level collusion (only one provider needs to be malicious), the likelihood is **moderate to high** in a production environment with multiple RPC providers.

### Recommendation

Pass the on-chain `ChainEntry.quorum` value into `FanOut` and enforce it explicitly:

```rust
pub struct FanOut<Inspector> {
    inspectors: NonEmptyVec<Inspector>,
    quorum: usize, // minimum number of agreeing non-transient successes required
}

// In extract():
if successes.len() < self.quorum {
    return Err(ForeignChainInspectionError::InsufficientQuorum {
        required: self.quorum,
        got: successes.len(),
    });
}
```

The quorum value must be sourced from the on-chain `ChainEntry.quorum` (streamed via the indexer whitelist task described in PR 3 of the whitelist rollout) and threaded through `ForeignChainInspectors::build` into each `FanOut` instance. A result with fewer than `quorum` agreeing non-transient successes must be treated as a terminal non-retryable failure, consistent with the design requirement. [9](#0-8) 

### Proof of Concept

**Setup**: A chain `C` is whitelisted with `ChainEntry { providers: [P1, P2, P3], quorum: 2 }`. A node has all three providers configured in its local `ForeignChainsConfig`. `FanOut` is built with three inspectors (one per provider), but `quorum=2` is never stored.

**Attack**:
1. Attacker controls provider `P1` (legitimately whitelisted or compromised after the fact).
2. User submits `verify_foreign_transaction` for a non-existent or failed transaction `T`.
3. Attacker's `P1` returns a forged success: `Ok([BlockHash(0xdeadbeef...)])`.
4. `P2` and `P3` return `RpcRequestFailed` (transient) — e.g., due to a brief outage or rate-limit.
5. `FanOut::extract` collects: `successes = [(0, [BlockHash(0xdeadbeef...)])]`, `first_transient_error = Some(RpcRequestFailed(...))`.
6. `inspectors_split_between_success_and_failure` is `false` (no non-transient errors).
7. `successes.first()` is `Some(...)`, `all_successes_agree` is trivially `true` (only one entry).
8. `FanOut` returns `Ok([BlockHash(0xdeadbeef...)])` — the forged value.
9. The node computes `ForeignTxSignPayload::V1 { request: T, values: [BlockHash(0xdeadbeef...)] }` and produces a signature share.
10. If `signing_threshold` nodes are similarly affected, the MPC network emits a valid threshold signature attesting that transaction `T` finalized with block hash `0xdeadbeef...` — enabling the attacker to claim bridge funds for a transaction that never existed. [10](#0-9) [11](#0-10)
