### Title
`CREATE3Factory` Salt Does Not Bind to `msg.sender`, Enabling Reorg-Based Deployment Hijack - (File: contracts/utils/CREATE3Factory.sol)

### Summary

`CREATE3Factory.deploy` accepts a caller-supplied `salt` without incorporating `msg.sender` into it. Because the deployed address is fully determined by `(factory_address, salt)`, any actor can race to claim a salt during a chain reorg and deploy a malicious contract at the address the protocol expected to be legitimate.

### Finding Description

`CREATE3Factory.deploy` passes the raw caller-supplied `salt` directly to `CREATE3.deploy`: [1](#0-0) 

Inside `CREATE3.deploy`, a proxy is first deployed via `create2` keyed on that salt: [2](#0-1) 

The final contract address is then derived from the proxy's `create`-based nonce (always 1): [3](#0-2) 

Because the salt is not namespaced to `msg.sender`, any external caller can invoke `CREATE3Factory.deploy` with an identical salt and substitute bytecode. There is no access control on `deploy`: [4](#0-3) 

### Impact Explanation

If the protocol team uses `CREATE3Factory` to deploy a core contract (e.g., a new `NodeDelegator`, vault, or bridge receiver) and a reorg occurs between the deployment transaction and a subsequent fund-transfer transaction, an attacker can:

1. Observe the pending `deploy(salt, legitimateCode)` in the mempool.
2. Submit `deploy(salt, maliciousCode)` with higher gas (or exploit the reorg window).
3. Claim the proxy address for that salt; the protocol's deployment reverts.
4. The protocol's fund-transfer transaction re-executes against the attacker-controlled contract at the same address.

This maps to **direct theft of user funds** (Critical) if the hijacked contract is a fund-holding component, or at minimum **temporary freezing of funds** (Medium) if the protocol deployment is blocked and must be redeployed at a different address while funds are in transit.

### Likelihood Explanation

The LRT-rsETH protocol targets Ethereum mainnet and L2s (Arbitrum, Optimism, etc.). Optimistic rollups are subject to reorgs during fraud-proof windows. The `deploy` function is unrestricted (`external payable`, no role check), so any EOA or contract can call it with any salt. An attacker only needs to know the salt in advance (observable from the mempool or from a public deployment script).

### Recommendation

Bind the salt to `msg.sender` inside `CREATE3Factory.deploy` so that only the original deployer can claim a given salt:

```solidity
function deploy(bytes32 salt, bytes memory creationCode) external payable returns (address deployed) {
    bytes32 guardedSalt = keccak256(abi.encodePacked(msg.sender, salt));
    deployed = CREATE3.deploy(guardedSalt, creationCode, msg.value);
    emit ContractDeployed(guardedSalt, deployed);
}
```

`getDeployed` should be updated to accept a `deployer` parameter and apply the same transformation so callers can still predict addresses off-chain.

### Proof of Concept

1. Protocol team broadcasts `CREATE3Factory.deploy(SALT, legitimateNodeDelegatorBytecode)`.
2. Attacker sees the transaction in the mempool and broadcasts `CREATE3Factory.deploy(SALT, maliciousCode)` with higher gas.
3. Attacker's transaction lands first; the proxy for `SALT` is now owned by the attacker's deployment.
4. Protocol's transaction reverts with `"DEPLOYMENT_FAILED"` (proxy address already has code).
5. Protocol team, unaware of the hijack, sends ETH or LST tokens to the pre-computed address (now the attacker's contract).
6. Attacker drains the funds via the malicious contract's sweep function.

On a reorg-susceptible chain the same outcome is achieved even if the protocol's transaction confirmed first: the reorg reverts it, the attacker's replacement transaction lands, and the protocol's subsequent fund-transfer re-executes against the malicious contract. [5](#0-4) [6](#0-5)

import { CREATE3 } from "./CREATE3.sol";

/// @title CREATE3Factory
/// @notice Factory contract for deploying contracts using CREATE3
/// @dev This factory can be deployed at the same address on multiple chains
contract CREATE3Factory {
    /// @notice Emitted when a contract is deployed
    event ContractDeployed(bytes32 indexed salt, address indexed deployedAddress);

    /// @notice Deploy a contract using CREATE3
    /// @param salt The salt for deterministic address generation
    /// @param creationCode The contract creation code with constructor parameters
    /// @return deployed The address of the deployed contract
    function deploy(bytes32 salt, bytes memory creationCode) external payable returns (address deployed) {
        deployed = CREATE3.deploy(salt, creationCode, msg.value);
        emit ContractDeployed(salt, deployed);
    }

    /// @notice Get the deployed address for a given salt
    /// @param salt The salt used for deployment
    /// @return The deterministic address
    function getDeployed(bytes32 salt) external view returns (address) {
        return CREATE3.getDeployed(salt);
    }
}

    function deploy(bytes32 salt, bytes memory creationCode, uint256 value) internal returns (address deployed) {
        bytes memory proxyChildBytecode = PROXY_BYTECODE;

        address proxy;
        assembly {
            // Deploy a new contract with our pre-made bytecode via CREATE2.
            // We start 32 bytes into the code to avoid copying the byte length.
            proxy := create2(0, add(proxyChildBytecode, 32), mload(proxyChildBytecode), salt)
        }
        require(proxy != address(0), "DEPLOYMENT_FAILED");

        deployed = getDeployed(salt);
        (bool success,) = proxy.call{ value: value }(creationCode);
        require(success && deployed.code.length != 0, "INITIALIZATION_FAILED");
    }
