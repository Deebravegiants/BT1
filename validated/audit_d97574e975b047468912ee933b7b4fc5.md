### Title
Missing Minimum Output Slippage Protection in L2 Pool `deposit()` Functions - (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPoolV3WithNativeChainBridge.sol, contracts/pools/RSETHPoolNoWrapper.sol, contracts/pools/RSETHPool.sol)

---

### Summary

Every public `deposit()` function across all L2 pool contracts computes the wrsETH/rsETH output amount at execution time using a live oracle rate, but accepts no `minRsETHAmountExpected` parameter. A user has no on-chain mechanism to bound the minimum wrsETH they will receive, leaving them exposed to oracle rate updates that occur between transaction submission and execution.

---

### Finding Description

The L2 pool contracts expose two `deposit()` overloads — one for native ETH and one for ERC-20 LSTs. Both compute the output amount by calling `viewSwapRsETHAmountAndFee()`, which reads `getRate()` from the oracle at execution time:

```solidity
// RSETHPoolV3.sol – ETH deposit
function deposit(string memory referralId) external payable ... {
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    wrsETH.mint(msg.sender, rsETHAmount);          // no minimum check
}

// RSETHPoolV3.sol – token deposit
function deposit(address token, uint256 amount, string memory referralId) external ... {
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
    wrsETH.mint(msg.sender, rsETHAmount);          // no minimum check
}
``` [1](#0-0) [2](#0-1) 

The rate computation is:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [3](#0-2) 

Because rsETH is a yield-bearing token, `rsETHToETHrate` monotonically increases over time. Any oracle update that raises the rate between a user's transaction submission and its on-chain execution will silently reduce the wrsETH minted for that user, with no revert.

The same pattern is present in every other L2 pool variant:

- `RSETHPoolV3ExternalBridge.deposit()` [4](#0-3) 
- `RSETHPoolV3WithNativeChainBridge.deposit()` [5](#0-4) 
- `RSETHPoolNoWrapper.deposit()` [6](#0-5) 
- `RSETHPool.deposit()` (Arbitrum) [7](#0-6) 

This is in direct contrast to the L1 `LRTDepositPool`, which explicitly enforces a caller-supplied minimum:

```solidity
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [8](#0-7) 

The L2 pools provide no equivalent protection.

---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A depositor who observes a quoted rate off-chain and submits a transaction may receive materially fewer wrsETH than expected if the oracle is updated in the same block or a subsequent block before their transaction is included. Because wrsETH is redeemable at the current rate, the user's position value in ETH terms is preserved, but the number of wrsETH shares received is lower than the user intended to accept. The user cannot enforce their own acceptable exchange rate on-chain.

---

### Likelihood Explanation

The rsETH/ETH oracle rate is propagated from L1 to each L2 via cross-chain messaging (LayerZero/CCIP). These rate updates are permissionless to relay once the cross-chain message is in flight, and they occur regularly as EigenLayer rewards accrue. On any L2 with a public mempool, a rate-update relay transaction and a user deposit transaction can land in the same block, with the rate update ordered first. No privileged access is required to trigger this ordering; any block builder or sequencer can do so.

---

### Recommendation

Add a `minRsETHAmountExpected` parameter to every public `deposit()` overload in all L2 pool contracts, and revert if the computed output falls below it:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountExpected)
    external payable nonReentrant whenNotPaused ...
{
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(msg.value);
    if (rsETHAmount < minRsETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    wrsETH.mint(msg.sender, rsETHAmount);
}
```

This mirrors the protection already present in `LRTDepositPool._beforeDeposit()` on L1 and is consistent with the pattern used by Uniswap V2's router for output-amount slippage protection.

---

### Proof of Concept

1. The current rsETH/ETH oracle rate on a given L2 is `1.05e18` (1 rsETH = 1.05 ETH).
2. Alice submits `deposit{value: 1 ether}("ref")` expecting to receive approximately `0.952 wrsETH`.
3. Before Alice's transaction is included, a cross-chain rate relay updates the oracle to `1.10e18`.
4. Alice's transaction executes: `rsETHAmount = 1e18 * 1e18 / 1.10e18 ≈ 0.909e18`.
5. Alice receives `0.909 wrsETH` instead of the `0.952 wrsETH` she observed off-chain — a ~4.5% shortfall — with no revert and no recourse.

The `viewSwapRsETHAmountAndFee` function confirms the rate is read live at execution time with no floor check: [3](#0-2)
