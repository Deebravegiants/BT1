### Title
Gas-Key TransactionV1 with Near-`u64::MAX` Nonce Admitted to Mempool but Permanently Rejected at Chunk Production — (`chain/chain/src/runtime/mod.rs`)

---

### Summary

`can_verify_and_charge_tx` unconditionally passes `block_height = None` to `verify_and_charge_gas_key_tx_ephemeral`, which causes `verify_nonce` to skip the `NonceTooLarge` upper-bound check. An attacker with a valid gas key account can submit a TransactionV1 with `nonce = u64::MAX - 1` and a valid `nonce_index`, receive `ProcessTxResponse::ValidTx` from the RPC, and have the transaction inserted into the mempool — while `prepare_transactions_extra` (chunk production) passes `Some(next_block_height)` and permanently rejects the same transaction with `NonceTooLarge`. The transaction is never included in a block.

---

### Finding Description

**Step 1 — RPC admission path skips the upper-bound check.**

In `NightshadeRuntime::can_verify_and_charge_tx`, the code explicitly hardcodes `block_height = None` before calling into the gas-key verifier: [1](#0-0) 

This `None` is forwarded directly to `verify_and_charge_gas_key_tx_ephemeral` → `verify_nonce`.

**Step 2 — `verify_nonce` gates the upper-bound check on `block_height`.** [2](#0-1) 

When `block_height` is `None`, the entire `NonceTooLarge` guard is bypassed. A nonce of `u64::MAX - 1` passes the lower-bound check (`tx_nonce > current_nonce`) and the function returns `Ok(())`.

**Step 3 — RPC returns `ValidTx` and inserts into the pool.**

`process_tx_internal` calls `can_verify_and_charge_tx`; on `Ok(())` it inserts the transaction into the sharded pool and returns `ProcessTxResponse::ValidTx`: [3](#0-2) 

**Step 4 — Chunk production enforces the upper-bound check.**

`prepare_transactions_extra` passes `Some(next_block_height)` to the same verifier: [4](#0-3) 

For any realistic block height `h`, `u64::MAX - 1 >= h * ACCESS_KEY_NONCE_RANGE_MULTIPLIER`, so `verify_nonce` returns `Err(NonceTooLarge)`. The transaction is discarded at every chunk production attempt and never included in a block.

**Step 5 — Apply path also enforces the check.**

The runtime's `apply` path in `lib.rs` similarly passes `Some(block_height)`: [5](#0-4) 

---

### Impact Explanation

The RPC incorrectly returns `ValidTx` for a gas-key transaction that is permanently unexecutable. The transaction occupies a mempool slot for the full transaction validity period. An attacker with a funded gas key account can repeatedly submit such transactions (each with a distinct `nonce_index` or after the prior one expires), polluting the pool and displacing legitimate transactions. The RPC contract is violated: `ValidTx` is supposed to mean the transaction can be included in a block.

---

### Likelihood Explanation

The attacker only needs:
1. A NEAR account with a gas key (standard on-chain operation).
2. Sufficient gas key balance to pass the balance check in `verify_and_charge_gas_key_tx_ephemeral`.
3. A single RPC call with `nonce = u64::MAX - 1` and a valid `nonce_index`.

No validator privileges, no special configuration, no social engineering. The path is directly reachable through the public `broadcast_tx_async` / `send_tx` RPC endpoints.

---

### Recommendation

In `can_verify_and_charge_tx`, use the current head block height (already available via `self.chain_store.head()` in the RPC handler, or passed as a parameter) as the `block_height` argument instead of `None`. This aligns the RPC admission check with the chunk-production check and prevents transactions with permanently-invalid nonces from entering the pool.
