### Title
`recoverFrozenFunds()` Misses rsETH Locked in the Withdrawal Queue, Allowing Blocked Users to Receive Underlying Assets - (File: contracts/RSETH.sol)

### Summary
`RSETH.recoverFrozenFunds()` only recovers the rsETH balance held directly in a blocked user's wallet. It does not account for rsETH that the user has already deposited into `LRTWithdrawalManager` via `initiateWithdrawal()`. Because `completeWithdrawal()` transfers LST/ETH (not rsETH) to the user and contains no block-enforcement check, a blocked user retains the ability to complete a pending withdrawal and receive the underlying asset, defeating the purpose of the freeze mechanism.

### Finding Description

When a user calls `initiateWithdrawal()`, their rsETH is pulled into `LRTWithdrawalManager`:

```solidity
IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);
``` [1](#0-0) 

The rsETH now sits in `LRTWithdrawalManager`, not in the user's wallet.

If the manager subsequently calls `blockUserTransfers()` on the user:

```solidity
transfersBlockedUntil[account] = blockedUntil;
``` [2](#0-1) 

and the admin then calls `recoverFrozenFunds()` to seize the user's rsETH:

```solidity
uint256 accountBalance = balanceOf(from);
super._transfer(from, custodyAddress, accountBalance);
``` [3](#0-2) 

`balanceOf(from)` returns **0** (or only the residual wallet balance) because the rsETH was already transferred to `LRTWithdrawalManager`. The rsETH sitting in the withdrawal manager on behalf of the user is invisible to `recoverFrozenFunds()`.

Meanwhile, `completeWithdrawal()` contains no block-enforcement check:

```solidity
function completeWithdrawal(address asset, string calldata referralId) external nonReentrant whenNotPaused {
    _processWithdrawalCompletion(asset, msg.sender, referralId);
}
``` [4](#0-3) 

`_processWithdrawalCompletion` ultimately calls `_transferAsset(asset, user, request.expectedAssetAmount)`, which transfers LST or ETH — not rsETH — so `_enforceNotBlocked` is never invoked:

```solidity
_transferAsset(asset, user, request.expectedAssetAmount);
``` [5](#0-4) 

The rsETH in `LRTWithdrawalManager` is burned during `unlockQueue()` from `address(this)` (the manager itself, which is not blocked):

```solidity
if (rsETHBurned != 0) IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned);
``` [6](#0-5) 

The `burnFrom` call enforces `_enforceNotBlocked(account)` where `account = address(this)` (LRTWithdrawalManager), which is not blocked, so it succeeds.

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

The `recoverFrozenFunds()` + `blockUserTransfers()` mechanism is designed to allow the protocol to fully freeze and seize a user's rsETH (e.g., for sanctions compliance). When a user has a pending withdrawal, `recoverFrozenFunds()` silently recovers zero (or a partial amount), and the user can still call `completeWithdrawal()` to receive the full underlying LST/ETH value. No third-party funds are lost, but the freeze mechanism fails to deliver its stated guarantee.

### Likelihood Explanation

Any user who has called `initiateWithdrawalManager.initiateWithdrawal()` before being blocked is in this state. The withdrawal delay is 8 days, so there is a substantial window during which a user can have a pending withdrawal when a block is applied. No front-running is required — the user need not anticipate the block.

### Recommendation

1. `recoverFrozenFunds()` should also cancel and recover rsETH held in `LRTWithdrawalManager` on behalf of the blocked user (e.g., by iterating `userAssociatedNonces` and burning the queued rsETH, crediting the committed assets back).
2. `completeWithdrawal()` (or `_processWithdrawalCompletion`) should check whether the recipient is currently blocked and revert if so, consistent with the intent of the freeze mechanism.

### Proof of Concept

1. Alice holds 100 rsETH and calls `LRTWithdrawalManager.initiateWithdrawal(stETH, 100e18, "")`. Her 100 rsETH is transferred to `LRTWithdrawalManager`; `RSETH.balanceOf(Alice) == 0`.
2. Manager calls `RSETH.blockUserTransfers([Alice])`. Alice's rsETH transfers are now blocked until `block.timestamp + 1 days`.
3. Admin calls `RSETH.recoverFrozenFunds(Alice)`. `balanceOf(Alice) == 0`, so `accountBalance = 0` and nothing is recovered. [7](#0-6) 
4. After the withdrawal delay, operator calls `unlockQueue(stETH, ...)`. The 100 rsETH is burned from `LRTWithdrawalManager` (not blocked), and stETH is allocated to Alice's request. [8](#0-7) 
5. Alice calls `completeWithdrawal(stETH, "")`. No block check is performed; she receives the full stETH amount. [4](#0-3) 

Alice successfully receives her underlying assets despite being blocked, and the admin's `recoverFrozenFunds()` call recovered nothing.
