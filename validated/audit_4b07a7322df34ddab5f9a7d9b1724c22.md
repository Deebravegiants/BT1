### Title
Stale-pending fallback in `read_pending_data` serves `l1_data_gas_price = 0` to RPC clients — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

### Summary
When the cached pending data is stale (its `parent_block_hash` does not match the latest committed block), `read_pending_data` constructs a synthetic `DeprecatedPendingBlock`. Because `DeprecatedPendingBlock` has no `l1_data_gas_price` field and the `l1_data_gas_price()` accessor unconditionally returns zero for the `Deprecated` variant, any RPC call with `block_id=pending` during this window receives `l1_data_gas_price = {price_in_wei: 0, price_in_fri: 0}`.

### Finding Description
`read_pending_data` falls back to `PendingBlockOrDeprecated::Deprecated(DeprecatedPendingBlock { ... })` when pending data is stale. The `l1_data_gas_price()` method on `PendingBlockOrDeprecated` returns `GasPricePerToken::default()` (all zeros) for the `Deprecated` arm — this was intentional for historical blocks that predate blob DA, but the stale fallback applies it to modern networks. The `PendingBlockHeader` built in `get_block` reads `block.l1_data_gas_price()` without any guard, so the zero value is served directly to callers. The same fallback also zeroes `l2_gas_price` and forces `l1_da_mode = Calldata`.

### Impact Explanation
Any unprivileged client calling `starknet_getBlockWithTxHashes`, `starknet_estimateFee`, or `starknet_simulateTransactions` with `block_id=pending` during a stale-pending window receives an authoritative-looking `PendingBlockHeader` with `l1_data_gas_price = 0`. Fee estimates for blob-DA transactions will undercount the data-gas component, potentially by the full blob fee. This is a **High** impact: RPC fee estimation returns an authoritative-looking wrong value.

### Likelihood Explanation
Stale-pending windows occur naturally whenever the pending data source (feeder gateway or internal pending block producer) lags behind committed blocks — e.g., during network hiccups, restarts, or high block production rates. No privileged access is required; any RPC client can trigger this path.

### Recommendation
Replace the `DeprecatedPendingBlock` fallback with a `PendingBlock` (the `Current` variant) that copies all gas price fields from the latest committed block header, including `l1_data_gas_price`, `l2_gas_price`, and `l1_da_mode`. Alternatively, copy the missing fields explicitly into the deprecated struct and override the `l1_data_gas_price()` / `l2_gas_price()` / `l1_da_mode()` accessors to read them.

### Proof of Concept
1. Set the node's pending data to a stale value (parent hash does not match the latest committed block).
2. Call `starknet_getBlockWithTxHashes` with `block_id = "pending"`.
3. Observe `result.l1_data_gas_price = { price_in_wei: "0x0", price_in_fri: "0x0" }` even though the latest committed block has non-zero `l1_data_gas_price`.
4. Call `starknet_estimateFee` with a blob-DA transaction and `block_id = "pending"`; the returned fee will be missing the data-gas component entirely.
