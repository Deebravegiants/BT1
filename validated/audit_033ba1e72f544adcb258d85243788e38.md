### Title
Single Attested Participant Can Deliver Arbitrary CKD Response for `AppPublicKey`-Type Requests, Bypassing Threshold Requirement — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in `MpcContract` performs **no cryptographic verification** of the response payload when the pending request carries the `CKDAppPublicKey::AppPublicKey` variant. Every other respond path — `respond` (ECDSA/EdDSA) and `respond_ckd` with `AppPublicKeyPV` — verifies the submitted output against the MPC network's public key before resolving user yields. The `AppPublicKey` branch is a deliberate no-op. A single attested participant (Byzantine, strictly below the signing threshold) can race the legitimate nodes, submit an arbitrary `CKDResponse` containing key material they control, and have the contract deliver that forged key to the waiting user — bypassing the threshold requirement entirely.

---

### Finding Description

**Root cause — missing output check in `respond_ckd`:** [1](#0-0) 

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no verification
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For `AppPublicKeyPV`, `ckd_output_check` cryptographically confirms the response is consistent with the threshold protocol output (BLS12-381 pairing check). For `AppPublicKey`, the arm is empty — the contract unconditionally proceeds to resolve all queued user yields with whatever `big_y`/`big_c` the caller supplied.

**Contrast with the other respond paths:**

`respond` (ECDSA/EdDSA) always verifies the submitted signature against the derived public key before resolving yields: [2](#0-1) 

`respond_ckd` with `AppPublicKeyPV` runs `ckd_output_check` and panics on failure: [3](#0-2) 

**Authorization gate that is present but insufficient:**

The function does require the caller to be an attested participant: [4](#0-3) 

`is_caller_an_attested_participant` checks account ID, TLS key, and Ed25519 account key against stored attestations: [5](#0-4) 

This gate restricts the attack to attested participants, but the system's explicit security goal is resistance to Byzantine participants **below** the signing threshold. A single compromised or malicious TEE node satisfies the gate and can then submit any `CKDResponse` it wishes.

**Yield resolution is first-writer-wins:**

`resolve_yields_for` drains all queued yields for the request on the first successful call: [6](#0-5) 

Once the attacker's transaction lands, the request is removed from `pending_ckd_requests`. Any subsequent legitimate `respond_ckd` from the honest majority finds no pending entry and silently discards the correct output.

---

### Impact Explanation

**Impact: Critical — Bypass of threshold-signature requirements / unauthorized confidential key derivation output.**

The attacker constructs a `CKDResponse` whose `big_y`/`big_c` correspond to a BLS12-381 key pair they generated locally. The contract delivers this forged key material to the user via the yield/resume callback. The user, believing they received a threshold-derived key, deposits assets to addresses controlled by that key. The attacker, holding the matching private key, can drain those assets at will.

This directly matches the allowed impact: *"Critical. Bypass of threshold-signature requirements or unauthorized access to MPC key shares, signing capability, or secret material that materially enables forgery or secret recovery."*

---

### Likelihood Explanation

**Likelihood: Medium.**

The attacker must be an attested participant — they must operate a TEE node that has passed `submit_participant_info` and `verify_tee`. This is a meaningful barrier. However, once inside the participant set, the exploit requires only:

1. Monitoring the NEAR chain for `request_app_private_key` calls with `AppPublicKey` type.
2. Constructing a fake `CKDResponse` (trivial — any two G1 points).
3. Submitting `respond_ckd` before the honest nodes do.

No threshold collusion, no key leakage, and no privileged operator access is required beyond the single attested-participant role. The race window is the block-processing latency between the user's request landing and the first honest `respond_ckd` transaction being included.

---

### Recommendation

1. **Add an output check for `AppPublicKey`:** Require the caller to supply a proof (e.g., a BLS signature over the request ID using the derived key) that the contract can verify against the MPC network's BLS12-381 public key. This mirrors what `ckd_output_check` does for `AppPublicKeyPV`.

2. **Alternatively, deprecate `AppPublicKey` in favour of `AppPublicKeyPV`:** Since `AppPublicKeyPV` already provides on-chain verifiability, `AppPublicKey` can be removed or restricted to a grace-period migration path.

3. **Enforce consistent check ordering across all `respond*` methods:** The ordering of `is_running_or_resharing`, `accept_requests`, and `assert_caller_is_attested_participant_and_protocol_active` differs between `respond`, `respond_ckd`, and `respond_verify_foreign_tx`. Centralising these guards into a single helper (analogous to `check_request_preconditions` for user-facing methods) would make the security surface auditable and prevent future omissions.

### Proof of Concept

```
1. Attacker operates a TEE node, submits valid attestation via submit_participant_info,
   passes verify_tee — becomes an attested participant.

2. User calls request_app_private_key({
       derivation_path: "m/44'/60'/0'",
       app_public_key: AppPublicKey(<attacker-chosen G1 point>),
       domain_id: <BLS domain>
   }) with 1 yoctoNEAR deposit.
   → Contract stores a pending CKD yield for the user.

3. Attacker observes the transaction on-chain.
   Attacker generates a fresh BLS12-381 key pair (sk_evil, pk_evil).
   Attacker constructs:
       fake_response = CKDResponse {
           big_y: G1::generator() * sk_evil,   // attacker-controlled
           big_c: G1::generator() * sk_evil,   // arbitrary
       }

4. Attacker calls respond_ckd(ckd_request, fake_response).
   Contract checks:
     - signer == predecessor ✓ (attacker's own account)
     - is_running_or_resharing ✓
     - accept_requests ✓
     - is_caller_an_attested_participant ✓ (attacker is attested)
     - AppPublicKey branch → no-op ✓ (no verification)
   → resolve_yields_for delivers fake_response to the user's pending yield.
   → Pending CKD entry is removed.

5. Honest nodes' respond_ckd calls arrive later — find no pending entry, silently discard.

6. User receives fake_response, derives an address from big_y, deposits funds.
   Attacker uses sk_evil to sweep the funds.
``` [7](#0-6)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
