### Title
Arbitrary `payload_hash` Substitution in `respond_verify_foreign_tx` Allows a Single Byzantine Node to Forge Foreign-Chain Verification Responses — (File: `crates/contract/src/lib.rs`)

---

### Summary

`verify_foreign_transaction` validates the foreign-chain request at submission time and stores it in `pending_verify_foreign_tx_requests`. However, `respond_verify_foreign_tx` only validates that the provided `response.payload_hash` carries a valid MPC signature; it never validates that `payload_hash` is actually the correct hash of the *stored* request. A single Byzantine MPC node can therefore recycle a legitimately-produced `(payload_hash_A, sig_A)` pair from request A and submit it as the response to a completely different pending request B, causing the user's contract to receive a signed attestation for the wrong foreign-chain transaction.

---

### Finding Description

**Submission path — validation present:**

`verify_foreign_transaction` (lines 519–557) enforces:
1. The domain exists and has `ForeignTx` purpose (`check_request_preconditions`).
2. The requested chain is in the on-chain supported-chains whitelist.
3. The request is stored in `pending_verify_foreign_tx_requests` keyed by the full `VerifyForeignTransactionRequest`. [1](#0-0) 

**Response path — validation missing:**

`respond_verify_foreign_tx` (lines 692–754) only checks:
1. The caller is an attested participant.
2. `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's root public key.
3. `request` exists in `pending_verify_foreign_tx_requests`.

It does **not** verify that `response.payload_hash` is the correct hash of the stored request. [2](#0-1) 

The `VerifyForeignTransactionResponse` carries only `payload_hash` and `signature`; the contract has no way to recompute the expected hash from the stored request alone, and it never attempts to do so. [3](#0-2) 

The entire unvalidated `response` (including the substituted `payload_hash`) is then serialised and returned to the user's contract via `resolve_yields_for`. [4](#0-3) 

The node-side code confirms that `payload_hash` is computed as `hash(ForeignChainRpcRequest, extracted_values)` and is therefore request-specific: [5](#0-4) 

---

### Impact Explanation

A user's bridge contract calls `verify_foreign_transaction` to obtain a signed attestation that a specific foreign-chain deposit (e.g., a Bitcoin transaction with a particular `tx_id` and `confirmations`) occurred before releasing NEAR-side funds. If a Byzantine node substitutes `payload_hash_A` (attesting to a *different* transaction, possibly a smaller deposit or one belonging to a different user) as the response to the user's pending request B, the user's contract receives a cryptographically valid signature over the wrong transaction. Any bridge contract that does not independently recompute and verify the `payload_hash` against its own expected request will accept this forged attestation and release funds incorrectly, enabling theft or double-spend conditions.

This matches the allowed impact: **"Cross-chain replay, forged foreign-chain verification, light-client-style verification bypass, or participant/attestation authorization bypass that causes invalid bridge execution or double-spend conditions."**

---

### Likelihood Explanation

The attack requires only **one** Byzantine MPC node (strictly below the signing threshold). The node participates normally in the threshold signing for request A, obtaining a valid `(payload_hash_A, sig_A)`. It then calls `respond_verify_foreign_tx(request_B, {payload_hash_A, sig_A})`. No threshold collusion is needed; the signature was legitimately produced by the full network for request A. The malicious node merely redirects it. Any attested participant whose operator is adversarial can execute this at any time two foreign-tx requests are simultaneously pending.

---

### Recommendation

Inside `respond_verify_foreign_tx`, after verifying the signature, recompute the expected `payload_hash` from the stored request and the extracted values provided in the response, and assert equality before resolving the yield. Concretely:

1. Extend `VerifyForeignTransactionResponse` to include the `extracted_values` field alongside `payload_hash` and `signature`.
2. In `respond_verify_foreign_tx`, recompute `expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: stored_request.request.clone(), values: response.extracted_values.clone() }).compute_msg_hash()` and assert `expected_hash == response.payload_hash` before calling `resolve_yields_for`.

This mirrors the fix in the reference report: validate the critical parameter (`payload_hash`) on the *respond* path with the same rigour applied on the *submit* path.

---

### Proof of Concept

1. User U1 calls `verify_foreign_transaction(request_A)` — a Bitcoin tx with `tx_id = [0xAA; 32]`, `confirmations = 6`. The contract stores `request_A` and issues a yield.
2. User U2 calls `verify_foreign_transaction(request_B)` — a Bitcoin tx with `tx_id = [0xBB; 32]`, `confirmations = 1` (a much smaller/different deposit). The contract stores `request_B` and issues a yield.
3. The MPC network processes request A. All nodes (including the Byzantine node) compute `payload_hash_A = hash(request_A, {BlockHash: 0xCC…})` and collectively produce `sig_A`.
4. The Byzantine node, instead of calling `respond_verify_foreign_tx(request_A, {payload_hash_A, sig_A})`, calls `respond_verify_foreign_tx(request_B, {payload_hash_A, sig_A})`.
5. The contract checks: is `sig_A` valid for `payload_hash_A` under the domain's public key? **Yes** (the network signed it). Does `request_B` exist in `pending_verify_foreign_tx_requests`? **Yes**. No check on whether `payload_hash_A` corresponds to `request_B`.
6. The contract resolves U2's yield with `{payload_hash_A, sig_A}`.
7. U2's bridge contract receives a valid MPC signature attesting to `request_A`'s transaction (the large deposit), not `request_B`'s. If U2's contract does not recompute and verify the `payload_hash`, it releases funds as if the large deposit occurred, enabling theft.

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }

    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
