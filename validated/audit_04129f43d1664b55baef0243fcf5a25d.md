### Title
Proposer Deposits Permanently Locked When `do_update` Clears All Pending Proposals — (File: `crates/contract/src/update.rs`)

---

### Summary

`propose_update` requires a NEAR deposit to cover storage staking for the proposal payload (which can include full contract bytecode). When any update is executed via `do_update`, **all** pending proposals are cleared — but the deposits paid by the other proposers are never refunded. Because `UpdateEntry` does not record the proposer's identity, there is no mechanism to return these funds, and no sweep/recovery function exists on the contract.

---

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` enforces a deposit equal to `ProposedUpdates::required_deposit(&update)`, which is computed as `env::storage_byte_cost() × bytes_used`. For a contract-code update, `bytes_used` includes the full bytecode plus overhead for up to 128 participant votes, making the required deposit potentially several NEAR for a realistic wasm binary. [1](#0-0) 

The deposit is accepted into the contract's balance. Only the excess above `required` is refunded to the proposer. [2](#0-1) 

The `UpdateEntry` struct that is stored contains only the update payload and a byte-count; the proposer's `AccountId` is **not** recorded. [3](#0-2) 

When `do_update` is called (triggered by `vote_update` reaching threshold), it removes the winning entry and then calls `self.entries.clear()` and `self.vote_by_participant.clear()`, wiping every other pending proposal. [4](#0-3) 

No refund is issued to the proposers of the cleared entries. Because the proposer identity was never stored, the contract cannot know whom to refund. The freed storage cost returns to the contract's own balance, where it remains inaccessible — there is no `sweep`, `withdraw`, or recovery function on the contract. [5](#0-4) 

---

### Impact Explanation

Every participant whose proposal is cleared by a competing update permanently loses their storage deposit. For a contract-code update (e.g., a 500 KB wasm), the required deposit is on the order of several NEAR (≈ `500_000 bytes × ~10 yN/byte = 5 NEAR`). Multiple competing proposals can accumulate in the contract simultaneously; when one wins, all others are silently cleared and their deposits are irrecoverable. This breaks the production accounting invariant that participants are made whole after a governance round, and constitutes a permanent, non-recoverable loss of participant funds controlled by the MPC contract.

---

### Likelihood Explanation

The scenario is a normal governance event: two or more participants independently propose different contract upgrades (e.g., one proposes a code update, another proposes a config update). This is explicitly supported by the data model (`IterableMap<UpdateId, UpdateEntry>` holds multiple entries). When one proposal reaches threshold and is executed, the others are cleared. No adversarial action is required — honest, independent governance participation is sufficient to trigger the loss.

---

### Recommendation

1. **Store the proposer's `AccountId` in `UpdateEntry`** so that refunds can be issued when entries are cleared.
2. **Refund cleared proposals in `do_update`**: iterate over all remaining entries before clearing them and schedule `Promise::new(entry.proposer).transfer(entry.deposit)` for each.
3. **Alternatively**, add a permissioned `sweep()` function (callable by participants via threshold vote) that transfers any excess contract balance above the minimum storage reserve to a designated account, analogous to the `sweep()` fix in the referenced report.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB contract binary, attaching ~5 NEAR deposit. The deposit is accepted; `UpdateEntry { update: Contract(bytes), bytes_used: N }` is stored with no record of A's identity.
2. Participant B calls `propose_update` with a config update, attaching a smaller deposit.
3. Participants vote for B's update; `vote_update` reaches threshold and calls `do_update(&id_B, gas)`.
4. Inside `do_update`: `self.entries.remove(&id_B)` removes B's entry; `self.entries.clear()` removes A's entry. No refund is issued to A.
5. A's ~5 NEAR deposit remains in the contract's balance permanently. There is no function to recover it. [6](#0-5) [7](#0-6)

    #[payable]
    #[handle_result]
    pub fn propose_update(
        &mut self,
        #[serializer(borsh)] args: ProposeUpdateArgs,
    ) -> Result<UpdateId, Error> {
        // Only voters can propose updates:
        let proposer = self.voter_or_panic();
        let update: Update = args.try_into()?;

        let attached = env::attached_deposit();
        let required = ProposedUpdates::required_deposit(&update);
        if attached < required {
            return Err(InvalidParameters::InsufficientDeposit {
                attached: attached.as_yoctonear(),
                required: required.as_yoctonear(),
            }
            .into());
        }

        let id = self.proposed_updates.propose(update);

        log!(
            "propose_update: signer={}, id={:?}",
            env::signer_account_id(),
            id,
        );

        // Refund the difference if the proposer attached more than required.
        if let Some(diff) = attached.checked_sub(required)
            && diff > NearToken::from_yoctonear(0)
        {
            Promise::new(proposer).transfer(diff).detach();
        }

        Ok(id)
    }

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }
