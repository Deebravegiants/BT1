### Title
Excess NEAR Deposit Permanently Locked in Contract on Existing-Participant Attestation Re-submission - (File: `crates/contract/src/lib.rs`)

### Summary

The `submit_participant_info` function is `#[payable]` and contains two deposit-handling code paths. Only one path (new participant or non-participant caller) refunds excess NEAR. The other path — an existing participant re-submitting their attestation — silently accepts any attached deposit and returns `Ok(())` without issuing any refund, permanently locking the NEAR in the contract.

### Finding Description

`submit_participant_info` is marked `#[payable]` and computes a boolean gate:

```rust
let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;
``` [1](#0-0) 

When this is `true`, the function correctly checks the deposit against actual storage cost and refunds any surplus:

```rust
if let Some(diff) = attached.checked_sub(cost)
    && diff > NearToken::from_yoctonear(0)
{
    Promise::new(account_id).transfer(diff).detach();
}
``` [2](#0-1) 

When this is `false` — i.e., when `is_new_attestation == false` (`UpdatedExistingParticipant`) **and** `caller_is_not_participant == false` (caller is an active participant) — the entire deposit-handling block is skipped and the function returns `Ok(())`: [3](#0-2) 

The `ParticipantInsertion::UpdatedExistingParticipant` variant is explicitly defined and returned by `tee_state.add_participant` when the TLS key already exists in `stored_attestations`: [4](#0-3) 

Because the function is `#[payable]`, the NEAR runtime accepts any attached deposit. With no refund path for the re-submission case, any NEAR attached to such a call is absorbed into the contract's balance with no mechanism for recovery. [5](#0-4) 

Compare this with the correct pattern used by `require_deposit` (called from `sign`, `request_app_private_key`, `verify_foreign_transaction`), which always refunds excess: [6](#0-5) 

`submit_participant_info` does not use `require_deposit`; it has its own ad-hoc deposit logic that is incomplete.

### Impact Explanation

Any NEAR tokens attached to a re-submission call by an existing participant are permanently locked in the MPC contract with no withdrawal path. This breaks the production accounting invariant that `#[payable]` functions must either consume exactly the required amount or refund the surplus. The locked funds are directly controlled by the MPC contract account and cannot be recovered without a contract upgrade.

This matches the allowed impact: **Medium — balance or contract execution-flow manipulation that breaks production safety/accounting invariants.**

### Likelihood Explanation

Participants must periodically re-submit TEE attestations as quotes expire (governed by `tee_upgrade_deadline_duration`). This is a routine, expected operation. Any participant who accidentally attaches NEAR (e.g., copy-pasting a prior `submit_participant_info` call that included a deposit for initial storage) will silently lose those funds. The scenario requires no adversarial setup — it is a normal operational action with a missing refund path.

### Recommendation

Two options, mirroring the external report's recommendations:

1. **Require zero deposit on re-submission:** When `attestation_storage_must_be_paid_by_caller` is `false`, assert `env::attached_deposit() == NearToken::from_yoctonear(0)` and panic otherwise.

2. **Refund unconditionally:** Restructure the deposit logic so that any excess above the computed `cost` (or above zero when no storage is charged) is always refunded to the caller, regardless of which branch is taken. The simplest fix is to call `require_deposit(NearToken::from_yoctonear(0), &account_id)` at the top of the function (which refunds everything when the minimum is zero) and then separately charge for storage when needed.

### Proof of Concept

1. Deploy the contract to a NEAR sandbox.
2. Have `participant_A` call `submit_participant_info` with a valid attestation (new participant). This succeeds and charges storage.
3. Have `participant_A` call `submit_participant_info` again (re-submission / attestation renewal) with `1 NEAR` attached.
4. Observe: the call succeeds (`Ok(())`), `participant_A`'s balance decreases by `1 NEAR`, and the contract's balance increases by `1 NEAR`. No refund is issued.
5. Confirm there is no contract method to withdraw or recover the locked NEAR.
