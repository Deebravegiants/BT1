### Title
Proposal Timestamp Has No Lower Bound Relative to Current Time, Enabling Retroactive L1 Gas Price Selection — (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_proposal_init_valid` enforces that `ProposalInit.timestamp` does not exceed `now + block_timestamp_window_seconds` but imposes **no lower bound relative to the current wall-clock time**. A malicious proposer can set a block timestamp arbitrarily far in the past (bounded only by the previous block's timestamp and the L1 gas price sample window). Because the validator recomputes the expected L1 gas prices using the *proposer-supplied* timestamp, it will accept historical prices that may be substantially lower than current prices, corrupting fee accounting for every transaction in the block.

---

### Finding Description

In `is_proposal_init_valid`, the timestamp window check is asymmetric:

```rust
// Lower bound: only requires timestamp >= previous block's timestamp
if init_proposed.timestamp < last_block_timestamp {
    return Err(...);
}
// Upper bound: timestamp must not be too far in the future
if init_proposed.timestamp > now + proposal_init_validation.block_timestamp_window_seconds {
    return Err(...);
}
``` [1](#0-0) 

There is no check of the form `init_proposed.timestamp >= now - some_past_window`. The config field `block_timestamp_window_seconds` is documented as "Maximum allowed deviation (seconds) of a proposed block's timestamp from the current time" but is only applied as a future cap. [2](#0-1) 

Immediately after the timestamp check, the validator looks up the reference L1 gas prices using the **proposer-supplied** timestamp:

```rust
let (l1_gas_prices_fri, l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(
    l1_gas_price_provider,
    init_proposed.timestamp,   // ← proposer-controlled past timestamp
    proposal_init_validation.previous_proposal_init.as_ref(),
    gas_price_params,
).await;
``` [3](#0-2) 

The validator then checks that the proposer's prices are within `l1_gas_price_margin_percent` of the reference it just computed for that past timestamp: [4](#0-3) 

Because the reference is anchored to the proposer's chosen timestamp, a proposer who picks a past time when L1 gas was cheaper will have their lower prices accepted as valid.

The accepted timestamp flows directly into `convert_to_sn_api_block_info` as `BlockTimestamp(init.timestamp)`, which becomes the authoritative block timestamp used for execution: [5](#0-4) 

The same asymmetric check is present in the sync path (`try_sync`): [6](#0-5) 

---

### Impact Explanation

**Incorrect L1 gas price with economic impact (Critical).**

A malicious proposer selects a past timestamp T_past at which L1 gas prices were lower. The validator computes its reference prices at T_past and accepts the proposer's matching (lower) prices. Every transaction in the block is then executed with those artificially low L1 gas prices, reducing fee revenue and potentially allowing transactions that would be rejected at current prices to succeed. The corrupted prices are committed to the block hash and propagated to state sync.

Additionally, the block timestamp visible to contracts via `get_block_timestamp()` is set to T_past, which can affect any time-sensitive contract logic.

---

### Likelihood Explanation

The proposer role rotates among consensus participants. Any consensus participant who becomes proposer for a round can exploit this without any additional privilege. The only practical constraint is that T_past must be ≥ the previous block's timestamp and within the L1 gas price sample window (which can span many hours of L1 history). The attack requires no special tooling beyond crafting a `ProposalInit` with a past timestamp and matching historical prices.

---

### Recommendation

Add a lower-bound check symmetric to the existing upper-bound check. For example:

```rust
// Reject timestamps more than block_timestamp_window_seconds in the past
if now.saturating_sub(proposal_init_validation.block_timestamp_window_seconds) > init_proposed.timestamp {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "Timestamp is too far in the past: now={}, block_timestamp_window_seconds={}, proposed={}",
            now, proposal_init_validation.block_timestamp_window_seconds, init_proposed.timestamp
        ),
    ));
}
```

This matches the documented intent of `block_timestamp_window_seconds` ("maximum allowed deviation from the current time") and closes the retroactive gas-price selection window. The same fix should be applied in `try_sync`.

---

### Proof of Concept

1. Observe that L1 gas prices were lower at time T_past (e.g., 2 hours ago).
2. When it is the attacker's turn to propose, craft a `ProposalInit` with:
   - `timestamp = T_past`
   - `l1_gas_price_fri` / `l1_data_gas_price_fri` / `l1_gas_price_wei` / `l1_data_gas_price_wei` set to the historical values at T_past (within `l1_gas_price_margin_percent` of what the validator will compute for T_past).
3. `is_proposal_init_valid` passes: `T_past >= last_block_timestamp` ✓ and `T_past <= now + window` ✓.
4. The validator calls `get_l1_prices_in_fri_and_wei(..., T_past, ...)`, obtains the same historical prices, and the `within_margin` check passes.
5. `initiate_validation` calls `convert_to_sn_api_block_info`, embedding T_past and the lower prices into the block's `BlockInfo`.
6. All transactions execute with the lower L1 gas prices; the block is committed with a past timestamp and reduced fees. [7](#0-6) [8](#0-7)

```rust
async fn initiate_validation(
    batcher: Arc<dyn BatcherClient>,
    state_sync_client: SharedStateSyncClient,
    init: &ProposalInit,
    proposal_id: ProposalId,
    timeout_plus_margin: Duration,
    clock: &dyn Clock,
    compare_retrospective_block_hash: bool,
) -> ValidateProposalResult<()> {
    let chrono_timeout = chrono::Duration::from_std(timeout_plus_margin)
        .expect("Can't convert timeout to chrono::Duration");

    let input = ValidateBlockInput {
        proposal_id,
        deadline: clock.now() + chrono_timeout,
        retrospective_block_hash: retrospective_block_hash(
            batcher.clone(),
            state_sync_client,
            init,
            compare_retrospective_block_hash,
        )
        .await
        .map_err(ValidateProposalError::from)?,
        block_info: convert_to_sn_api_block_info(init)?,
    };
    debug!("Initiating validate proposal: input={input:?}");
    batcher.validate_block(input.clone()).await.map_err(|err| {
        ValidateProposalError::Batcher(
            format!("Failed to initiate validate proposal {input:?}."),
            err,
        )
    })?;
    Ok(())
}
