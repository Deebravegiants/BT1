### Title
Development TCB Measurements Compiled Into Production Builds Enabling Attestation Authorization Bypass - (File: `crates/mpc-attestation/src/attestation.rs`)

### Summary

The `default_measurements()` function in `crates/mpc-attestation/src/attestation.rs` statically compiles both production (`tcb_info.json`) and development (`tcb_info_dev.json`) TCB measurements into every build. A node running development-environment software whose TEE measurements match `tcb_info_dev.json` will pass attestation verification and be accepted as a valid MPC participant in production, bypassing the TEE security boundary. The codebase itself acknowledges this with an explicit security TODO comment.

### Finding Description

`default_measurements()` returns a static array of two `ExpectedMeasurements` entries:

```rust
// crates/mpc-attestation/src/attestation.rs  L259-L266
pub fn default_measurements() -> &'static [ExpectedMeasurements] {
    static MEASUREMENTS: [ExpectedMeasurements; 2] = [
        include_measurements!("assets/tcb_info.json"),
        // TODO(#1433): Security - remove dev measurements from production builds after testing is complete
        include_measurements!("assets/tcb_info_dev.json"),
    ];
    &MEASUREMENTS
}
``` [1](#0-0) 

This slice is passed as `accepted_measurements` into `verify_measurements()`, which accepts an attestation if its measurements appear anywhere in the slice:

```rust
fn verify_measurements(
    measurements: &ExpectedMeasurements,
    allowed_measurements: &[ExpectedMeasurements],
) -> Result<(), VerificationError> {
    if !allowed_measurements.contains(measurements) {
        return Err(VerificationError::MeasurementsNotAllowed);
    }
    Ok(())
}
``` [2](#0-1) 

Because `tcb_info_dev.json` measurements are included unconditionally (no `#[cfg(debug_assertions)]`, no feature flag, no build-time environment check), a `DstackAttestation` whose RTMR/MRTD values match the dev TCB profile passes `verify_measurements` in production.

The `Attestation` enum itself is also not feature-gated and includes a `Mock(MockAttestation)` variant alongside `Dstack`:

```rust
pub enum Attestation {
    Dstack(DstackAttestation),
    Mock(MockAttestation),
}
``` [3](#0-2) 

`MockAttestation::Valid` always returns `Ok(())` from `verify_constraints` with no hardware checks, and the conversion path from node to contract DTO for `MockAttestation` is also not feature-gated: [4](#0-3) 

### Impact Explanation

The TEE attestation system is the sole mechanism that ensures every MPC participant is running approved, unmodified software inside a genuine Intel TDX enclave. Accepting dev measurements in production means a node running development-environment software (or any software whose measurements were compiled into `tcb_info_dev.json`) can call `submit_participant_info` on the MPC contract, pass `add_participant` attestation verification, and be registered as a valid participant. Once registered, such a node participates in threshold signing rounds. If enough dev-measurement nodes are accepted (up to but not exceeding the threshold), they can observe partial key-share material exchanged over the P2P mesh and influence signing outputs. This constitutes a **participant/attestation authorization bypass** that breaks the production safety invariant that all signers must be running verified TEE software.

### Likelihood Explanation

The dev measurements are compiled into every production binary unconditionally. Any operator or attacker who can run a node whose hardware measurements match `tcb_info_dev.json` — or who can submit a `MockAttestation::Valid` (which requires no hardware at all and always passes) — can register as a participant. The explicit `TODO(#1433): Security` comment confirms the team is aware this is unresolved. The attack requires only the ability to call `submit_participant_info` on the NEAR contract, which is a public, permissionless entry point.

### Recommendation

1. Remove `include_measurements!("assets/tcb_info_dev.json")` from `default_measurements()` in production builds. Gate it behind a compile-time feature flag (e.g., `#[cfg(feature = "dev-measurements")]`) that is never enabled in release/production CI pipelines.
2. Gate the `Attestation::Mock` variant and all its handling paths behind `#[cfg(any(test, feature = "mock-attestation"))]` so that `MockAttestation::Valid` cannot be submitted to a production contract.
3. Add a CI check that asserts the production binary does not contain dev measurement bytes or mock attestation code paths.

### Proof of Concept

1. Obtain the dev TCB measurement values from `crates/mpc-attestation/assets/tcb_info_dev.json`.
2. Run an MPC node (or craft a raw NEAR transaction) that calls `submit_participant_info` with either:
   - A `DstackAttestation` whose MRTD/RTMR fields match the dev measurements, **or**
   - A `MockAttestation::Valid` (no hardware required).
3. `add_participant` calls `Attestation::verify_locally` → `MockAttestation::verify` → `verify_constraints` → returns `Ok(())` unconditionally for `MockAttestation::Valid`, or passes `verify_measurements` for dev-measurement `DstackAttestation` because `default_measurements()` includes `tcb_info_dev.json`.
4. The node is stored in `tee_state.stored_attestations` as a valid participant.
5. On the next `vote_new_parameters` that includes this node, `reverify_and_cleanup_participants` re-runs the same `default_measurements()` path and again accepts the dev-measurement node, yielding `TeeValidationResult::Full`.
6. The dev-measurement node is now a recognized MPC participant and receives P2P signing messages containing partial key-share material. [5](#0-4) [6](#0-5)
