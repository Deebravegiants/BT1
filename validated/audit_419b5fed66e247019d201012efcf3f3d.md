### Title
Caller-Controlled Finality Parameter Allows Signing of Unfinalized Foreign-Chain Transactions — (`crates/contract/src/lib.rs`, `crates/node/src/providers/verify_foreign_tx/sign.rs`)

---

### Summary

`verify_foreign_transaction` accepts the finality level and confirmation count directly from the unprivileged caller without enforcing any minimum security threshold. An attacker can submit a request with `EvmFinality::Latest` (for any EVM chain) or `BlockConfirmations(0)` (for Bitcoin), causing the MPC network to produce a valid threshold signature attesting to a transaction that has not reached sufficient finality. This is the direct analog of the BondOracleAdapter selecting an attacker-created pool with tick spacing 1: in both cases the protocol accepts a caller-supplied "quality" parameter without a floor, and the resulting attestation is based on a manipulable, low-security data source.

---

### Finding Description

The `ForeignChainRpcRequest` enum carries the finality requirement as a caller-supplied field:

- `EvmRpcRequest.finality: EvmFinality` — variants `Latest`, `Safe`, `Finalized`
- `BitcoinRpcRequest.confirmations: BlockConfirmations` — a plain `u64` with no minimum [1](#0-0) [2](#0-1) 

The contract's `verify_foreign_transaction` entry point performs only two checks: that the domain is a `ForeignTx` domain and that the chain is in the supported set. It does **not** validate the finality level or confirmation count: [3](#0-2) 

On the node side, `execute_foreign_chain_request` similarly only calls `chain_is_supported` before dispatching to the inspector, passing the caller-supplied finality value verbatim: [4](#0-3) 

The EVM inspector's `verify_finality_level` maps `EvmFinality::Latest` directly to `FinalityTag::Latest` — the weakest possible tag, representing the chain tip with no reorg protection: [5](#0-4) 

The Bitcoin inspector compares the caller-supplied threshold against the RPC-reported confirmation count with `block_confirmations_threshold <= transaction_block_confirmation`. With `threshold = 0`, this is always true: [6](#0-5) 

The signed payload (`ForeignTxSignPayload::V1`) embeds the full request — including the weak finality field — alongside the extracted values. The MPC network's signature therefore attests to "this transaction was verified at `Latest` finality" or "with 0 confirmations", which is a valid but security-vacuous statement. [7](#0-6) 

---

### Impact Explanation

A NEAR bridge contract that calls `verify_foreign_transaction` and then acts on the returned MPC signature (e.g., minting wrapped tokens or releasing collateral) will receive a cryptographically valid threshold signature from the MPC network. If the bridge contract does not independently re-validate the `finality` field embedded in the signed payload, it will treat the signature as proof of a finalized foreign-chain event. The attacker can:

1. Broadcast a Bitcoin or EVM transaction.
2. Immediately submit `verify_foreign_transaction` with `confirmations: 0` or `finality: Latest`.
3. Receive a valid MPC signature before the transaction is confirmed.
4. Present the signature to the bridge contract to claim NEAR-side assets.
5. Double-spend or reorganize the foreign-chain transaction.

This constitutes **forged foreign-chain verification enabling double-spend conditions** — matching the High impact category.

---

### Likelihood Explanation

- Any unprivileged account can call `verify_foreign_transaction` with a deposit of 1 yoctoNEAR.
- No governance vote, operator action, or threshold collusion is required.
- The attack is fully self-contained: the attacker controls the `finality` field in the request struct.
- EVM chains (Abstract, BNB, Base, Arbitrum, HyperEVM, Polygon) all share the same `EvmFinality::Latest` path; Bitcoin accepts `BlockConfirmations(0)`.
- The window between transaction broadcast and finality is well-known and routinely exploited in bridge attacks.

---

### Recommendation

Enforce a minimum finality level on-chain in `verify_foreign_transaction` before enqueuing the request. Concretely:

- For EVM chains: reject requests with `EvmFinality::Latest`; require at minimum `Safe` or `Finalized` (configurable per chain via the `ForeignChainRpcWhitelist` / `ChainEntry`).
- For Bitcoin: enforce a protocol-defined minimum `BlockConfirmations` (e.g., 6) stored in the on-chain chain configuration, and reject requests below that floor.

The per-chain `ChainEntry` already carries a `quorum` field for RPC agreement; a `min_finality` / `min_confirmations` field should be added alongside it and validated in `verify_foreign_transaction` before the request is enqueued.

            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );

    /// Checks that the receipt's block has reached the requested finality level — i.e. that the
    /// head of the chain at `finality` is at or past `receipt_block_number`.
    async fn verify_finality_level(
        &self,
        receipt_block_number: U64,
        finality: EthereumFinality,
    ) -> Result<(), ForeignChainInspectionError> {
        let finality_tag = match finality {
            EthereumFinality::Finalized => FinalityTag::Finalized,
            EthereumFinality::Safe => FinalityTag::Safe,
            EthereumFinality::Latest => FinalityTag::Latest,
        };
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Tag(finality_tag),
            ReturnFullTransactionObjects::from(false),
        );
        let head: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        if head.number < receipt_block_number {
            return Err(ForeignChainInspectionError::NotFinalized);
        }
        Ok(())
    }
