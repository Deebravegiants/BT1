### Title
Byzantine MPC Coordinator Submits `K256Signature` with Inconsistent `recovery_id` / `big_r` Y-Parity, Causing Bridge Transaction Failures - (File: crates/near-mpc-crypto-types/src/conversions/k256.rs)

---

### Summary

The `TryFrom<&K256Signature> for k256::ecdsa::Signature` conversion silently discards the y-parity prefix byte of `big_r.affine_point[0]`. Because the contract's `respond` path calls `verify_ecdsa_signature` through this conversion, it never validates that `recovery_id` is consistent with the y-parity encoded in `big_r`. A Byzantine MPC coordinator (a single attested participant, strictly below the signing threshold) can assemble a valid `(r, s)` signature from the MPC protocol, then flip the prefix byte of `big_r` and set a mismatched `recovery_id` before calling `respond`. The contract accepts the signature, the user receives a `SignatureResponse` with wrong y-parity metadata, and any downstream bridge contract that calls `ecrecover` recovers the wrong address, permanently blocking the bridge execution.

---

### Finding Description

`K256AffinePoint` is a raw `[u8; 33]` byte array with no validation at the struct level. [1](#0-0) 

The `TryFrom<&K256Signature> for k256::ecdsa::Signature` conversion extracts `r` from bytes `[1..33]` of `big_r.affine_point`, completely ignoring byte `[0]` (the SEC1 prefix that encodes y-parity: `0x02` = even, `0x03` = odd): [2](#0-1) 

The contract's `respond` function accepts a `dtos::SignatureResponse` from any attested participant and calls `verify_ecdsa_signature` to validate it: [3](#0-2) 

Because `verify_ecdsa_signature` uses the above conversion (which ignores the prefix byte), it validates only that `(r, s)` satisfies the ECDSA equation for the given public key and message hash. It does **not** validate that:
1. `big_r.affine_point[0]` is a valid SEC1 prefix (`0x02` or `0x03`), and
2. `recovery_id` is consistent with the y-parity encoded in that prefix byte.

The `recovery_id` field receives only a trivial range check (`> MAX_RECOVERY_ID`) in the node-side helper, not a consistency check against `big_r`: [4](#0-3) 

The full `K256Signature` (including the unchecked `big_r` prefix and `recovery_id`) is then serialised and returned to the user via `resolve_yields_for`: [5](#0-4) 

---

### Impact Explanation

Ethereum and EVM-compatible bridge contracts use `ecrecover(hash, v, r, s)` (where `v = recovery_id + 27`) to verify that the MPC network signed a payload. If `recovery_id` is flipped (0 ↔ 1), `ecrecover` returns the address derived from the *other* candidate R point, which does not match the MPC network's public key. The bridge contract rejects the call, and the user's funds locked in the bridge cannot be released. Because the NEAR yield is already resolved with the corrupted signature, the user cannot obtain a corrected signature without submitting an entirely new request — which the Byzantine coordinator can corrupt again indefinitely.

This maps to the allowed impact: **High — cross-chain verification bypass that causes invalid bridge execution**.

---

### Likelihood Explanation

The MPC coordinator role is held by the leader node for each signing round. A single Byzantine participant acting as coordinator has the full assembled `(R, s)` signature (it collects all shares), so it can trivially flip `big_r.affine_point[0]` and set a mismatched `recovery_id` before calling `respond`. No threshold collusion is required; one Byzantine node below the signing threshold suffices. The attack is silent: the contract logs only that the signature is valid, and the user has no way to distinguish a correctly-formed response from a corrupted one until they attempt to use it on the foreign chain.

---

### Recommendation

**Short term:** In `verify_ecdsa_signature` (or in the contract's `respond` handler), after accepting the signature, assert that `big_r.affine_point[0]` is either `0x02` or `0x03`, and that `recovery_id & 1` equals `big_r.affine_point[0] & 1` (i.e., the y-parity bits agree). Reject the response if they differ.

**Long term:** Canonicalise `K256Signature` at the boundary where it enters the system (deserialization or the `respond` call) by deriving `recovery_id` deterministically from the full `big_r` point and the known public key (as `brute_force_recovery_id` already does on the honest node path), rather than trusting the caller-supplied value.

---

### Proof of Concept

1. A Byzantine MPC node wins the coordinator role for a signing round.
2. The MPC protocol completes; the coordinator assembles the valid signature `(R, s)` where `R` has even y-coordinate (prefix `0x02`).
3. The Byzantine coordinator constructs a `K256Signature` with:
   - `big_r.affine_point[0]` = `0x03` (flipped to odd-y prefix)
   - `big_r.affine_point[1..33]` = correct x-coordinate of R (unchanged)
   - `s` = correct s scalar (unchanged)
   - `recovery_id` = `1` (consistent with the flipped prefix, but wrong for the actual R)
4. The coordinator calls `respond(request, crafted_response)`.
5. The contract calls `verify_ecdsa_signature`: `TryFrom<&K256Signature>` extracts `r` from bytes `[1..33]` (same x-coordinate), ignores byte `[0]`, and calls `from_scalars(r, s)`. Verification passes.
6. The contract resolves the pending yield; the user receives `SignatureResponse` with `recovery_id = 1`.
7. The user submits the signature to an Ethereum bridge contract: `ecrecover(hash, 28, r, s)` recovers the address for the *odd-y* R point, which is not the MPC network's address.
8. The bridge contract rejects the call. The user's funds remain locked. [2](#0-1) [6](#0-5) [7](#0-6)

impl TryFrom<&K256Signature> for k256::ecdsa::Signature {
    type Error = CryptoConversionError;
    fn try_from(dto: &K256Signature) -> Result<Self, Self::Error> {
        // r is the x-coordinate from the compressed R point (bytes [1..33])
        let r = k256::FieldBytes::from_slice(&dto.big_r.affine_point[1..]);
        let s = k256::FieldBytes::from_slice(&dto.s.scalar);
        k256::ecdsa::Signature::from_scalars(*r, *s)
            .map_err(|_| CryptoConversionError::InvalidSignature)
    }
}

    pub fn respond(
        &mut self,
        request: SignatureRequest,
        response: dtos::SignatureResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!("respond: signer={}, request={:?}", &signer, &request);

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain)?;

        let signature_is_valid = match (&response, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                // generate the expected public key
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");
                let affine = *k256::PublicKey::try_from(&secp_pk)
                    .expect("stored key is always valid")
                    .as_affine();
                let expected_public_key =
                    derive_key_secp256k1(&affine, &request.tweak).map_err(RespondError::from)?;

                let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");

                // Check the signature is correct
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    payload_hash,
                    &expected_public_key,
                )
                .is_ok()
            }
            (
                dtos::SignatureResponse::Ed25519 { signature },
                PublicKeyExtended::Ed25519 {
                    edwards_point: public_key_edwards_point,
                    ..
                },
            ) => {
                let derived_public_key_edwards_point = derive_public_key_edwards_point_ed25519(
                    &public_key_edwards_point,
                    &request.tweak,
                );
                let derived_public_key_32_bytes =
                    dtos::Ed25519PublicKey::from(derived_public_key_edwards_point.compress());

                let message = request.payload.as_eddsa().expect("Payload is not EdDSA");

                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    message,
                    &derived_public_key_32_bytes,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_signature_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

fn k256_signature_response(
    big_r: AffinePoint,
    s: Scalar,
    recovery_id: u8,
) -> anyhow::Result<dtos::SignatureResponse> {
    if recovery_id > MAX_RECOVERY_ID {
        anyhow::bail!("Invalid Recovery Id: recovery id larger than 3.");
    }
    Ok(dtos::SignatureResponse::Secp256k1(dtos::K256Signature {
        big_r: dtos::K256AffinePoint::from(big_r),
        s: dtos::K256Scalar::from(s),
        recovery_id,
    }))
}
