### Title
P2P Sync Class Stream Accepts Mismatched Sierra Class Without Hash Verification, Leaving Correct Class Hash Unregistered in Class Manager — (`crates/apollo_p2p_sync/src/client/class.rs`)

### Summary

During p2p class sync, `ClassStreamBuilder::write_to_storage` calls `class_manager_client.add_class(class)` but never verifies that the class hash returned by the class manager matches the expected class hash from the state diff. Because `ClassManager::add_class` computes and stores the class under its own content-derived hash, a malicious peer can supply a `wrong_sierra_class` alongside a `correct_class_hash` (one that is legitimately in the state diff). The class manager stores `wrong_sierra_class` under `wrong_class_hash` (computed from content), and `correct_class_hash` is never registered. The node's class manager is then permanently missing the class for `correct_class_hash`, causing all future contract executions that require it to fail or use wrong code.

### Finding Description

**Entrypoint:** An unauthenticated p2p peer sends `(ApiContractClass::ContractClass(wrong_sierra), correct_class_hash)` during SQMR class sync.

**Step 1 — `parse_data_for_block`** reads the stored state diff and validates that the peer-supplied `class_hash` exists in `state_diff.class_hash_to_compiled_class_hash`: [1](#0-0) 

The check at line 144 only verifies the peer-supplied `class_hash` is declared in the state diff. It does **not** verify that the `contract_class` content actually hashes to `class_hash`. The result is `declared_classes_result = {correct_class_hash: wrong_sierra_class}`.

**Step 2 — `write_to_storage`** iterates over the map and calls `add_class` with only the class content, discarding the expected hash entirely: [2](#0-1) 

The TODO at line 39 explicitly acknowledges the missing check. The returned `ClassHashes` value (which contains the actual stored hash) is thrown away.

**Step 3 — `ClassManager::add_class`** computes the class hash from the class content and stores the class under that computed hash: [3](#0-2) 

`wrong_sierra_class` is stored under `wrong_class_hash = wrong_sierra_class.calculate_class_hash()`. `correct_class_hash` is never written.

**Contrast with the central sync path**, which performs the check and panics on mismatch: [4](#0-3) 

The p2p sync path has no equivalent guard.

### Impact Explanation

After the attack, the class manager has no entry for `correct_class_hash`. Any contract whose class was declared in that block (and whose class hash is `correct_class_hash`) will fail to execute — the class manager returns `None` for `get_executable(correct_class_hash)`. This constitutes **wrong compiled class / contract code selected for execution**, which is a Critical impact per the allowed scope. The node's view of the chain diverges from the canonical state.

### Likelihood Explanation

Any node running in p2p sync mode is exposed. The attacker only needs to be a reachable p2p peer (no authentication required). The peer must supply a `class_hash` that is present in the already-stored state diff for the target block, which is public information. The attack is deterministic and requires no brute force.

### Recommendation

After `add_class` returns successfully, compare the returned `class_hash` against the expected key from the state diff and treat a mismatch as a `BadPeerError`, mirroring the central sync path:

```rust
// In write_to_storage, replace the while-loop body with:
loop {
    match class_manager_client.add_class(class.clone()).await {
        Ok(class_hashes) => {
            if class_hashes.class_hash != class_hash {
                return Err(P2pSyncClientError::BadPeer(
                    BadPeerError::ClassHashMismatch {
                        expected: class_hash,
                        got: class_hashes.class_hash,
                    }
                ));
            }
            break;
        }
        Err(err) => { warn!(...); }
    }
}
```

### Proof of Concept

1. Construct a `wrong_sierra` whose `calculate_class_hash()` returns `wrong_class_hash ≠ correct_class_hash`.
2. Store a state diff containing `correct_class_hash` in `class_hash_to_compiled_class_hash`.
3. Feed `(ApiContractClass::ContractClass(wrong_sierra), correct_class_hash)` to `ClassStreamBuilder` via a mock `ClientResponsesManager`.
4. Assert `write_to_storage` returns `Ok(())` (it does — no error is raised).
5. Assert `class_manager.get_executable(correct_class_hash)` returns `None` (the correct class was never stored).
6. Assert `class_manager.get_executable(wrong_class_hash)` returns `Some(...)` (the wrong class was stored instead).
