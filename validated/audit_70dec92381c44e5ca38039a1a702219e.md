Audit Report

## Title
Uninitialized `rate=0` in `CrossChainRateReceiver` Causes Division-by-Zero, Temporarily Freezing All L2 Pool Deposits - (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`CrossChainRateReceiver.rate` is a `uint256` storage variable that defaults to `0` at deployment, and `getRate()` returns it with no zero-check. Every L2 pool variant calls `viewSwapRsETHAmountAndFee`, which divides by this rate to compute the wrsETH mint amount. Before the first LayerZero `lzReceive()` call populates `rate`, any deposit attempt reverts with a division-by-zero panic, temporarily freezing all deposits.

## Finding Description

`CrossChainRateReceiver` declares `rate` as a plain `uint256` storage variable, defaulting to `0`:

```solidity
uint256 public rate;   // default = 0
```

`getRate()` returns it unconditionally with no zero-check or staleness guard:

```solidity
function getRate() external view returns (uint256) {
    return rate;
}
```

`rate` is only written inside `lzReceive()`, which requires a LayerZero cross-chain message to be relayed. `RSETHRateReceiver`'s constructor sets no initial rate:

```solidity
constructor(uint16 _srcChainId, address _rateProvider, address _layerZeroEndpoint) {
    rateInfo = RateInfo({ tokenSymbol: "rsETH", baseTokenSymbol: "ETH" });
    srcChainId = _srcChainId;
    rateProvider = _rateProvider;
    layerZeroEndpoint = _layerZeroEndpoint;
}
```

Every pool's `viewSwapRsETHAmountAndFee` divides by the oracle rate without a zero-check:

```solidity
uint256 rsETHToETHrate = getRate();          // returns 0 before first LZ message
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;  // division by zero → revert
```

This pattern is present identically in `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, and `RSETHPoolNoWrapper`. Notably, `RSETHPoolV3.viewSwapAssetToPremintedRsETH` does include a zero-check (`if (rsETHToETHrate == 0) revert UnsupportedOracle()`), confirming the protocol is aware of the risk but failed to apply the same guard in `viewSwapRsETHAmountAndFee`.

In `RSETHPoolV3`, the `limitDailyMint` modifier calls `viewSwapRsETHAmountAndFee` before the function body executes, meaning the division-by-zero panic fires even before the `isEthDepositEnabled` guard in the function body is reached. For `RSETHPoolNoWrapper`, if `isEthDepositEnabled` is set to `true` at initialization, the division-by-zero is reached directly in the function body.

## Impact Explanation

**Medium — Temporary freezing of funds.** All `deposit()` calls on any L2 pool whose `rsETHOracle` is a `CrossChainRateReceiver`-derived contract revert with a division-by-zero panic until the first LayerZero rate message is successfully relayed. Users cannot deposit ETH or LSTs during this window. No funds are permanently lost, but the freeze window can be extended by LayerZero congestion or insufficient gas on the destination chain.

## Likelihood Explanation

This is triggered automatically on every new deployment of `RSETHRateReceiver` paired with any L2 pool, with no attacker action required. Any ordinary depositor calling `deposit()` before the first `lzReceive()` will encounter the revert. Given the protocol is deployed across 15+ L2 chains, this window occurs on every new deployment or oracle contract redeployment. The precondition (rate == 0) is the default state and requires no special setup.

## Recommendation

Set a safe non-zero initial rate in the `RSETHRateReceiver` constructor, e.g., `rate = 1 ether` (representing a 1:1 rsETH/ETH ratio). Alternatively, add a zero-check guard in `getRate()` that reverts with a descriptive error (e.g., `RateNotInitialized()`), and add a corresponding zero-check in every `viewSwapRsETHAmountAndFee` before dividing — consistent with the guard already present in `viewSwapAssetToPremintedRsETH`.

## Proof of Concept

1. Deploy `RSETHRateReceiver` on an L2 chain. `rate` is `0`.
2. Deploy `RSETHPoolV3` with `rsETHOracle = address(rateReceiver)` and `_isEthDepositEnabled = true`.
3. Call `reinitialize` with a `startTimestamp` in the past or present so the `limitDailyMint` modifier does not revert on the timestamp check.
4. Call `pool.deposit{value: 1 ether}("ref")` before any LayerZero message arrives.
5. `limitDailyMint` modifier calls `viewSwapRsETHAmountAndFee(1 ether)` → calls `getRate()` → returns `0`.
6. `rsETHAmount = amountAfterFee * 1e18 / 0` → EVM division-by-zero panic → revert.
7. All deposits are frozen until `lzReceive()` is called by the LayerZero endpoint.

Foundry test sketch:
```solidity
function test_depositRevertsWhenRateZero() public {
    RSETHRateReceiver receiver = new RSETHRateReceiver(srcChainId, rateProvider, lzEndpoint);
    // rate == 0 at this point
    assertEq(receiver.getRate(), 0);

    // Deploy pool with receiver as oracle
    // ... initialize pool with isEthDepositEnabled=true, startTimestamp in past ...

    vm.expectRevert(); // division by zero panic
    pool.deposit{value: 1 ether}("ref");
}
```
