### Title
Excess Deposit Permanently Locked in `submit_participant_info` When Existing Participant Re-submits Attestation - (File: crates/contract/src/lib.rs)

### Summary

`submit_participant_info` is marked `#[payable]` and contains conditional deposit-handling logic. When an existing participant re-submits their attestation (an update, not a new entry), the entire deposit-check-and-refund block is skipped. Any NEAR tokens attached to that call are silently absorbed by the contract with no refund path and no withdrawal mechanism, permanently locking them.

### Finding Description

The function `submit_participant_info` computes a boolean flag to decide whether to enforce a deposit:

```rust
let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... check attached >= cost, refund diff ...
}
``` [1](#0-0) 

When `is_new_attestation = false` **and** `caller_is_not_participant = false` (i.e., an existing participant refreshes their TEE attestation), the `if` block is never entered. Because the function is declared `#[payable]`, NEAR's runtime does not reject a non-zero `attached_deposit`; it simply credits the contract. The deposit is neither validated nor refunded. [2](#0-1) 

The comment inside the block explains an intentional asymmetry for *storage bytes*, but says nothing about ignoring an arbitrary monetary deposit: [3](#0-2) 

No other code path in the contract provides a way to withdraw funds that were deposited this way. The contract has no owner-withdraw or sweep function.

### Impact Explanation

Any NEAR tokens attached by a participant during an attestation update are permanently frozen inside the chain-signature contract. There is no recovery path: the contract exposes no privileged withdrawal endpoint, and the deposit is not tracked in any refundable ledger. This breaks the production accounting invariant that every yoctoNEAR deposited into a `#[payable]` function must either be consumed for its stated purpose (storage cost) or returned to the sender.

This maps to the **Medium** allowed impact: *"Balance … manipulation that breaks production safety/accounting invariants."*

### Likelihood Explanation

MPC node operators run automated software that calls `submit_participant_info` periodically to refresh TEE attestations (e.g., after a TEE image upgrade). Automation scripts or SDK wrappers that always attach a small safety deposit (a common NEAR pattern to satisfy `#[payable]` guards) will silently lose those tokens on every re-submission. The sandbox test helper itself omits a deposit, but unit tests throughout the codebase attach `NearToken::from_near(1)` to the same call, demonstrating that callers routinely attach deposits. [4](#0-3) [5](#0-4) 

### Recommendation

Add an unconditional refund of any excess deposit **outside** the `if attestation_storage_must_be_paid_by_caller` block, mirroring the pattern already used in `require_deposit` and `propose_update`:

```rust
// After the conditional storage-charge block:
let attached = env::attached_deposit();
if attached > NearToken::from_yoctonear(0) {
    Promise::new(account_id).transfer(attached).detach();
}
```

Alternatively, restructure the logic so the refund path is always reached regardless of whether storage was charged.

### Proof of Concept

1. Deploy the contract and initialize it with participant `alice.near`.
2. `alice.near` calls `submit_participant_info` a second time (attestation update) with `attached_deposit = 1 NEAR`.
3. At this point: `is_new_attestation = false`, `caller_is_not_participant = false`, so `attestation_storage_must_be_paid_by_caller = false`.
4. The `if` block is skipped entirely — no refund promise is created.
5. The contract's balance increases by 1 NEAR; `alice.near`'s balance decreases by 1 NEAR.
6. No contract method exists to recover those funds.

            // rather than underflow. Intentional asymmetry: we do not refund freed bytes
            // either — the caller already paid for the larger entry, and we'd rather
            // accept that asymmetry than open a refund path for payload-shrinking games.
            let storage_used = env::storage_usage().saturating_sub(initial_storage);
            let cost = env::storage_byte_cost().saturating_mul(storage_used as u128);
            let attached = env::attached_deposit();

                return Err(InvalidParameters::InsufficientDeposit {
                    attached: attached.as_yoctonear(),
                    required: cost.as_yoctonear(),

                && diff > NearToken::from_yoctonear(0)
