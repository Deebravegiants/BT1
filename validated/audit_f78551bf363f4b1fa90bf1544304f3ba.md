Audit Report

## Title
`out0Scaled` Not Updated When Fallback Clamps `totalIn1Scaled` — Trader Receives Less Token0 Than Owed - (File: metric-core/contracts/libraries/SwapMath.sol)

## Summary
In `buyToken0InBinSpecifiedIn`, when both the analytical solution and the first proportional refinement still underestimate the required token1 input (i.e., `totalIn1Scaled < amountSpecifiedRemainingScaled && targetPos < maxFinalBinPos` remains true after the first refinement pass), the fallback at lines 620–622 forcibly sets `totalIn1Scaled = amountSpecifiedRemainingScaled` without updating `out0Scaled`. The pool then consumes the trader's full token1 input but credits only the token0 amount computed for the underestimated `targetPos`, causing a direct loss of token0 output to the trader.

## Finding Description

**Code path:**

`MetricOmmPool._swapToken1ForToken0SpecifiedInput` → `SwapMath.buyToken0InBinSpecifiedIn`

The function enters the `else` branch at line 579 because the full bin costs more token1 than the trader's input. It then calls `computeAnalyticalTargetPosForBuyToken0` (line 581) to find a partial `targetPos`, recomputes `out0Scaled` and `totalIn1Scaled` for that position (lines 591–596), and checks whether the analytical estimate still underestimates (line 598).

If it does, a first proportional refinement scales `targetPos` upward (lines 600–617) and recomputes `out0Scaled` and `totalIn1Scaled`. If after this refinement `totalIn1Scaled` is **still** less than `amountSpecifiedRemainingScaled` and `targetPos < maxFinalBinPos`, the fallback fires:

```solidity
// lines 620-622
if (totalIn1Scaled < state.amountSpecifiedRemainingScaled && targetPos < maxFinalBinPos) {
    totalIn1Scaled = state.amountSpecifiedRemainingScaled;
}
```

After this assignment `totalIn1Scaled == amountSpecifiedRemainingScaled`, so the guard at line 624 (`totalIn1Scaled > amountSpecifiedRemainingScaled`) is **false** and the proportional rescaling block (lines 624–634) — which is the only place `out0Scaled` is reduced to match the actual input — is **skipped entirely**.

The function then settles with:
- `totalIn1Scaled` = full trader input (line 643 drains `amountSpecifiedRemainingScaled` to zero)
- `out0Scaled` = amount computed for the underestimated `targetPos` (never updated)
- `binState.token0BalanceScaled -= out0Scaled` (pool retains the extra token0)
- `state.amountCalculatedScaled += out0Scaled` (trader credited the smaller amount)

**Why existing guards fail:** The only guard that rescales `out0Scaled` is the block at lines 624–634, which is unreachable after the line-621 assignment because the equality makes the `>` condition false. No other code path updates `out0Scaled` before the final settlement.

## Impact Explanation

The trader pays `amountSpecifiedRemainingScaled` token1 but receives fewer token0 than the oracle/bin curve permits. The shortfall remains locked in the pool's `token0BalanceScaled`, accruing to LPs rather than the trader. This is a direct, repeatable loss of user principal on every swap that triggers the double-underestimate path. It constitutes a swap conservation failure: the pool receives the full owed input but does not deliver the full owed output.

Severity: **Medium** (loss is bounded by the magnitude of the two-step underestimate, which grows with bin width and input size, but does not drain the entire pool in a single call).

## Likelihood Explanation

The path is reachable by any unprivileged trader calling `swap` with `zeroForOne = false` (token1 → token0, exact input). It requires:
1. The full bin cost exceeds the trader's input (partial-bin swap).
2. The analytical quadratic approximation underestimates the required input — possible due to integer `Math.sqrt` truncation and `ceilDiv` rounding in `computeAnalyticalTargetPosForBuyToken0`.
3. The first proportional refinement still underestimates — possible because the position-to-input relationship is quadratic, so linear proportional scaling can remain below the true value.

Wide price ranges (`upperPriceX64 >> lowerPriceX64`), large token0 balances, and mid-range `currBinPos` values increase the quadratic correction term and make double-underestimate more likely. No special privileges or malicious setup are required.

## Recommendation

Replace the fallback at lines 620–622 with a direct advance to `maxFinalBinPos` and a full recomputation of `out0Scaled`, mirroring the pattern used in the `totalIn1Scaled <= amountSpecifiedRemainingScaled` fast path at line 577–578:

```solidity
if (totalIn1Scaled < state.amountSpecifiedRemainingScaled && targetPos < maxFinalBinPos) {
    targetPos = maxFinalBinPos;
    out0Scaled = calculateOutputToken0FromBinPosition(
        binState.token0BalanceScaled, currBinPos, targetPos
    );
    finalPriceX64 = calculatePriceAtBinPosition(
        lowerPriceX64, upperPriceX64, targetPos, Math.Rounding.Ceil
    );
    avgPriceX64 = calculateArithmeticMean(startingPriceX64, finalPriceX64);
    in1WithoutFeeScaled = calculateRequiredToken(out0Scaled, avgPriceX64);
    totalIn1Scaled = grossInputWithBinFeeCeil(in1WithoutFeeScaled, onePlusBuyFeeX64);
}
```

This ensures `out0Scaled` always corresponds to the `targetPos` and `totalIn1Scaled` that are actually settled.
