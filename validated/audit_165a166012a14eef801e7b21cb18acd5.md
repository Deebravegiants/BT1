### Title
Missing Caller Identity Binding in `verify_foreign_transaction` Enables Unauthorized Root-Key Attestation of Foreign Chain Transactions - (File: `crates/contract/src/lib.rs`, `crates/node/src/providers/verify_foreign_tx/sign.rs`)

---

### Summary

`verify_foreign_transaction` discards the caller's identity entirely and the MPC nodes sign the resulting payload with a **zero tweak** (root key). Any unprivileged NEAR account can submit a verification request for any foreign chain transaction and receive a valid root-key MPC attestation signature. This is the direct analog of clickjacking: just as clickjacking exploits missing security controls (X-Frame-Options headers) to let an attacker trigger privileged actions attributed to a victim, here the missing caller-identity binding lets an attacker trigger a privileged root-key signing operation and receive the attestation, enabling front-running of bridge deposit claims.

---

### Finding Description

**1. Caller identity is silently discarded in `verify_foreign_transaction`**

`check_request_preconditions` returns `(domain_config, predecessor)`, but `verify_foreign_transaction` discards the entire return value:

```rust
// lib.rs lines 526-531
self.check_request_preconditions(
    request.domain_id,
    DomainPurpose::ForeignTx,
    Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
    MINIMUM_SIGN_REQUEST_DEPOSIT,
);
// return value (domain_config, predecessor) is dropped
```

Compare with `sign()`, which binds the predecessor to the request key via tweak derivation:

```rust
// lib.rs lines 379-384
let request = SignatureRequest::new(
    request.domain_id,
    request.payload,
    &predecessor,   // caller identity bound here
    &request.path,
);
```

**2. `VerifyForeignTransactionRequest` has no caller field**

The struct used as the pending-request map key contains only chain data — no predecessor, no derivation path, no tweak:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
```

**3. Node signs with a hard-coded zero tweak (root key)**

`build_signature_request` in the node provider sets `tweak: Tweak::new([0u8; 32])`, meaning the MPC network always signs foreign-tx payloads with the **undifferentiated root key**:

```rust
// sign.rs lines 39-47
Ok(SignatureRequest {
    id: request.id,
    receipt_id: request.receipt_id,
    payload: Payload::Ecdsa(payload_bytes),
    tweak: Tweak::new([0u8; 32]),   // root key — no caller derivation
    ...
})
```

**4. Contract verifies against root key, not a derived key**

`respond_verify_foreign_tx` verifies the signature against `secp_pk` (the raw root public key), confirming no tweak is applied:

```rust
// lib.rs lines 728-734
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,   // root key, no derive_key_secp256k1 call
)
```

**5. Design specified caller-specific key derivation — it was never implemented**

`docs/foreign-chain-transactions.md` lines 254–286 explicitly specified that `verify_foreign_transaction` must use a caller-specific tweak derived from `(predecessor_id, derivation_path)` with a foreign-tx-specific prefix (`"near-mpc-recovery v0.1.0 foreign-tx epsilon derivation:"`), and that `VerifyForeignTransactionRequestArgs` must carry a `derivation_path` field. Neither the field nor the derivation was implemented.

**6. Fan-out delivers the same root-key signature to every caller**

The pending-request map is keyed on the caller-agnostic `VerifyForeignTransactionRequest`. Multiple callers submitting the same foreign-tx request are queued under the same key and all receive the identical root-key signature when one `respond_verify_foreign_tx` is delivered. This is confirmed by the test `verify_foreign_transaction__should_queue_duplicates_from_different_callers`.

---

### Impact Explanation

**High — forged foreign-chain verification enabling invalid bridge execution.**

The `verify_foreign_transaction` flow is the trust anchor for the Omnibridge inbound path: a NEAR bridge contract accepts the MPC root-key signature as proof that a specific foreign-chain transaction finalized. Because the signature carries no caller identity and is signed with the root key:

1. Alice deposits 1 ETH to the bridge on Ethereum (tx_id = X).
2. Bob (attacker, any NEAR account) submits `verify_foreign_transaction` for tx_id X before Alice does.
3. MPC nodes verify the transaction and sign `hash(request, extracted_values)` with the root key.
4. Bob receives the valid root-key attestation.
5. Bob presents the attestation to the NEAR bridge contract.
6. If the bridge contract checks only "is this a valid MPC root-key signature over this tx_id?" (the natural check given the API), Bob claims Alice's deposit.

Even if the bridge contract checks extracted values (e.g., Ethereum recipient address), the attacker still receives a valid root-key signature over a transaction they did not initiate, and the root key is shared across all foreign-tx verifications — a single compromised or replayed attestation affects the entire bridge domain.

---

### Likelihood Explanation

**Medium.** The entry point is fully open to any unprivileged NEAR account with 1 yoctoNEAR deposit. No special role, key, or collusion is required. The attacker only needs to observe a pending foreign-chain deposit (publicly visible on-chain) and submit the `verify_foreign_transaction` call before the legitimate user. Front-running on NEAR is straightforward given public mempool visibility. The attack requires no TEE access, no threshold collusion, and no social engineering.

---

### Recommendation

1. **Add `derivation_path` to `VerifyForeignTransactionRequestArgs`** and include `predecessor_id` in `VerifyForeignTransactionRequest` (as the design doc specified).
2. **Derive the tweak** from `(predecessor_id, derivation_path)` using the foreign-tx-specific prefix (`"near-mpc-recovery v0.1.0 foreign-tx epsilon derivation:"`) in the contract, and pass it through to the node's `build_signature_request` instead of `Tweak::new([0u8; 32])`.
3. **Verify the response against the derived key** in `respond_verify_foreign_tx`, mirroring the `respond()` pattern that calls `derive_key_secp256k1(&affine, &request.tweak)`.
4. **Include the caller identity in the pending-request map key** so that different callers' requests are isolated and cannot fan-out to unintended recipients.

## Tweak Derivation (Sign vs ForeignTx)

`verify_foreign_transaction()` uses a **different tweak derivation prefix** than `sign()` so the same
`(predecessor_id, derivation_path)` can never yield the same derived key across the two purposes.

Design:

* Keep the existing sign tweak derivation prefix unchanged.
* Introduce a foreign-tx-specific prefix and derive the tweak from the same `(predecessor_id, derivation_path)`
  input using the same hash construction.
* The contract derives the tweak internally from `request.derivation_path` (callers do not submit raw tweaks).

Example:

```rust
const SIGN_TWEAK_DERIVATION_PREFIX: &str =
    "near-mpc-recovery v0.1.0 epsilon derivation:";
const FOREIGN_TX_TWEAK_DERIVATION_PREFIX: &str =
    "near-mpc-recovery v0.1.0 foreign-tx epsilon derivation:";

pub fn derive_sign_tweak(predecessor_id: &AccountId, path: &str) -> Tweak {
    let hash: [u8; 32] = derive_from_path(SIGN_TWEAK_DERIVATION_PREFIX, predecessor_id, path);
    Tweak::new(hash)
}

pub fn derive_foreign_tx_tweak(predecessor_id: &AccountId, path: &str) -> Tweak {
    let hash: [u8; 32] = derive_from_path(FOREIGN_TX_TWEAK_DERIVATION_PREFIX, predecessor_id, path);
    Tweak::new(hash)
}
```

This ensures key material used for validated foreign transactions is **always** distinct from
general-purpose `sign()` keys, even if the same account and derivation path are reused.
