### Title
Missing Minimum-Length Validation on KEM Shared Secret Allows Zero-Entropy DEM Key in PVE Custom-KEM Path - (`src/cbmpc/c_api/pve_base_pke.cpp`)

### Summary

The guard in `kem_encap_shim` (and symmetrically in `kem_decap_shim`) does not reject a callback-returned `kem_ss` of `{nullptr, 0}`. The condition `kem_ss.size < 0 || (kem_ss.size > 0 && !kem_ss.data)` is false when `size == 0` and `data == nullptr`, so the empty value is forwarded into `kem_aead_ciphertext_t::seal`, which derives a fully deterministic, publicly-computable AES-GCM-256 key via HKDF-SHA256 over an empty IKM. The encryption returns `SUCCESS` and produces a structurally valid PVE ciphertext whose DEM key has zero entropy.

### Finding Description

**Exact guard (both shims):** [1](#0-0) 

The condition is:
```
size < 0  ||  (size > 0 && !data)
```
It is `false` for `{nullptr, 0}`, so the shim proceeds to: [2](#0-1) 

`buf_t(nullptr, 0)` constructs a valid empty buffer (size=0, `assign_short` copies 0 bytes — no UB). [3](#0-2) 

The empty `buf_t` is returned as `out_kem_ss` to `kem_policy_runtime_kem_t::encapsulate`, which returns it unchanged to `kem_aead_ciphertext_t::seal`: [4](#0-3) 

`hkdf_extract_sha256(mem_t(), kem_ss)` is called with `kem_ss` as an empty `mem_t` (size=0). OpenSSL's HKDF-Extract with empty IKM is well-defined and produces a fixed, deterministic PRK. The AES-GCM-256 key derived from it is therefore a fixed, publicly-known constant. The encryption completes and returns `SUCCESS`. [5](#0-4) 

The same bypass exists in `kem_decap_shim`: [6](#0-5) 

So a matching `decap` callback that also returns `{nullptr, 0}` causes decryption to succeed with the same zero-entropy key, completing the end-to-end path.

**Contrast with the HSM-backed RSA path**, which correctly validates the shared-secret size: [7](#0-6) 

No equivalent minimum-size check exists for the generic runtime-KEM path.

### Impact Explanation

A malicious or buggy `cbmpc_pve_base_kem_t::encap` callback that returns `{nullptr, 0}` for `out_kem_ss` causes `cbmpc_pve_encrypt_with_kem` to produce a PVE ciphertext whose AES-GCM-256 key is derived from an empty shared secret. The key is:

```
HKDF-Expand(HKDF-Extract(zero_salt_32, ""), "CBMPC|KEM-AEAD|v1|...", 32)
```

This is a fixed, globally-known constant. Any party that knows the HKDF label (which is hardcoded in the public header) can decrypt the ciphertext without the private key. The PVE layer is used to protect private key shares and secret scalars in the MPC protocol; a zero-entropy DEM key means those secrets are effectively unprotected.

### Likelihood Explanation

The `cbmpc_pve_base_kem_t` callback interface is a public C API surface explicitly designed for third-party KEM integrations (HSMs, FFI wrappers). A buggy HSM driver that returns `{nullptr, 0}` on a soft error without setting a non-zero error code, or a malicious callback provider, can trigger this path. The library gives no indication that anything is wrong — it returns `CBMPC_SUCCESS` and a well-formed ciphertext.

### Recommendation

Add a minimum-length check in both `kem_encap_shim` and `kem_decap_shim` immediately after the existing guard, rejecting any `kem_ss` with `size == 0`:

```cpp
// existing guard
if (kem_ss.size < 0 || (kem_ss.size > 0 && !kem_ss.data)) { ... return E_FORMAT; }
// add:
if (kem_ss.size == 0) { return coinbase::error(E_FORMAT, "KEM shared secret must be non-empty"); }
```

Additionally, consider enforcing a minimum entropy threshold (e.g., ≥ 16 bytes) consistent with the security level of the AES-GCM-256 DEM.
