### Title
Block Stuffing Delays L2 Burn, Causing CCIP Mint to Revert on `i_maxSupply` Cap — (`contracts/ccip/WrappedRSETH.sol`)

---

### Summary

When `WrappedRSETH` is deployed with a non-zero `i_maxSupply` and `totalSupply` is near that cap, an attacker can use block stuffing on L2 to delay a pending burn transaction long enough for a CCIP-triggered `mint` to arrive first. The mint reverts with `MaxSupplyExceeded`, causing the CCIP message to fail and temporarily freezing the bridged rsETH.

---

### Finding Description

`WrappedRSETH.mint` enforces a hard supply cap: [1](#0-0) 

```solidity
function mint(address account, uint256 amount) external override onlyMinter validAddress(account) {
    if (i_maxSupply != 0 && totalSupply() + amount > i_maxSupply) revert MaxSupplyExceeded(totalSupply() + amount);
    _mint(account, amount);
}
```

`i_maxSupply` is set immutably at construction time: [2](#0-1) 

The CCIP bridge flow originates in `L1VaultV2.bridgeRsETHToL2UsingCCIP`, which locks rsETH on L1 and dispatches a CCIP message. On L2, the CCIP OffRamp calls `WrappedRSETH.mint` to deliver the tokens: [3](#0-2) 

The attack sequence:

1. `totalSupply` is at `i_maxSupply - X` (near cap).
2. A legitimate user submits a burn transaction on L2 to reduce `totalSupply` by `X`.
3. Attacker observes the pending burn and stuffs L2 blocks with high-gas transactions, delaying the burn.
4. A CCIP-triggered `mint` of amount `≥ X` arrives and executes before the burn.
5. `totalSupply() + amount > i_maxSupply` → `MaxSupplyExceeded` revert.
6. The CCIP OffRamp marks the message as failed; rsETH remains locked in the CCIP bridge on L1.

The `allowOutOfOrderExecution: true` flag set in the CCIP message means there is no ordering guarantee that would prevent this race: [4](#0-3) 

---

### Impact Explanation

The CCIP message fails and rsETH is temporarily frozen in the L1 bridge. CCIP's `manuallyExecute` retry mechanism means the freeze is **temporary** (not permanent): once the burn executes and `totalSupply` drops below `i_maxSupply`, the failed message can be retried. However, during the window between the failed mint and the successful retry, the bridged funds are inaccessible to the recipient on L2. This constitutes **temporary freezing of funds** and falls squarely within the "Low. Block stuffing" impact scope.

---

### Likelihood Explanation

- **Precondition 1:** `i_maxSupply` must be set to a non-zero value at deployment. This is a constructor parameter; if `0`, the check is bypassed entirely and no vulnerability exists.
- **Precondition 2:** `totalSupply` must be near `i_maxSupply`. This is a realistic operational state if the cap is set conservatively.
- **Precondition 3:** Block stuffing on L2 is expensive but feasible, especially on chains with low block gas limits or during low-fee periods. The attacker must sustain the stuffing only for the duration of the CCIP finality window (minutes to hours).
- **Precondition 4:** The burn must be visible in the mempool. On sequencer-based L2s with private mempools this is harder, but not impossible (e.g., via public RPC observation or sequencer MEV).

Likelihood is **Low** given the cost and preconditions, but the impact is real and the code path is concrete.

---

### Recommendation

1. **Remove or make `i_maxSupply` adjustable by governance** so it can be raised if the cap is approached, preventing the race condition from being exploitable.
2. **Alternatively, deploy `WrappedRSETH` with `maxSupply_ = 0`** (unlimited) if no hard cap is operationally required, which disables the check entirely.
3. **Add a CCIP message retry handler** that automatically re-queues failed mints rather than requiring manual intervention, reducing the freeze window.

---

### Proof of Concept

```solidity
// Foundry fuzz test (local fork, unmodified contracts)
function testBlockStuffingMintRevert(uint256 burnAmount) public {
    // Setup: deploy WrappedRSETH with maxSupply = 1000e18
    uint256 cap = 1000e18;
    WrappedRSETH token = new WrappedRSETH("wrsETH", "wrsETH", 18, cap, owner);
    vm.prank(owner); token.grantMintAndBurnRoles(address(ccipOffRamp));
    vm.prank(owner); token.grantMintAndBurnRoles(address(burner));

    // Mint to near cap: totalSupply = cap - 10e18
    vm.prank(address(ccipOffRamp));
    token.mint(user, cap - 10e18);

    // Simulate block stuffing: burn tx is delayed (not yet executed)
    // burnAmount = 10e18 (would bring totalSupply back to cap - 10e18)

    // CCIP mint of 10e18 arrives first (before burn)
    vm.prank(address(ccipOffRamp));
    vm.expectRevert(WrappedRSETH.MaxSupplyExceeded.selector);
    token.mint(recipient, 10e18); // totalSupply(990e18) + 10e18 = cap → reverts

    // rsETH is now locked on L1; burn has not yet executed
    // After burn executes, retry succeeds:
    vm.prank(address(burner));
    token.burn(user, 10e18); // totalSupply drops to 980e18

    vm.prank(address(ccipOffRamp));
    token.mint(recipient, 10e18); // now succeeds
}
```

This demonstrates the revert path at [5](#0-4)  and confirms the temporary freeze, with recovery only after manual retry post-burn.
