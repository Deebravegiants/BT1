### Title
Fee-on-Transfer Token Undercollateralization in `initTransfer` — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` accepts any ERC20 token, calls `safeTransferFrom` for the caller-specified `amount`, then emits `InitTransfer` (and, in the Wormhole variant, publishes a Wormhole message) encoding that same caller-specified `amount` — without ever checking the contract's actual balance before and after the transfer. For fee-on-transfer tokens the vault receives `amount − fee` but the cross-chain message credits `amount`, permanently undercollateralizing the bridge and eventually locking legitimate withdrawers' funds.

---

### Finding Description

In `OmniBridge.initTransfer`, when `tokenAddress` is neither a bridge token nor a custom-minter token, the contract executes:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // caller-supplied, not verified against actual receipt
);
``` [1](#0-0) 

Immediately after, the function emits the `InitTransfer` event (and, in `OmniBridgeWormhole`, publishes a Wormhole message) using the same caller-supplied `amount`:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,          // ← not the actual received amount
    fee,
    nativeFee,
    recipient,
    message
);
``` [2](#0-1) 

The Wormhole extension encodes the same `amount` into the cross-chain payload:

```solidity
Borsh.encodeUint128(amount),
``` [3](#0-2) 

No balance snapshot is taken before the transfer, and no balance check is performed after it. For any ERC20 that silently deducts a transfer fee from the recipient (fee-on-transfer tokens), the bridge vault receives `amount − fee` while the cross-chain message asserts `amount`.

The NEAR bridge's `fin_transfer_callback` trusts the proof-verified event amount unconditionally and credits the full `amount` to the destination recipient: [4](#0-3) 

There is no reconciliation step that compares the emitted amount against the vault's actual balance.

---

### Impact Explanation

Every `initTransfer` call with a fee-on-transfer token creates a deficit equal to the transfer fee:

- The EVM vault holds `amount − fee`.
- The destination chain (NEAR or another EVM) releases `amount` to the recipient.
- The bridge is undercollateralized by `fee` per call.

Repeated calls drain the vault's real reserves while the bridge's accounting believes it is fully backed. When legitimate users later bridge back from NEAR to EVM, `finTransfer` attempts to release the full credited amount:

```solidity
IERC20(payload.tokenAddress).safeTransfer(
    payload.recipient,
    payload.amount
);
``` [5](#0-4) 

The vault has insufficient balance; the transfer reverts. Affected users cannot redeem their bridged assets — a **permanent, irreversible fund lock** matching the Critical impact class.

---

### Likelihood Explanation

- `initTransfer` is a fully public, permissionless function — no role or whitelist restricts which ERC20 address a caller may supply.
- Fee-on-transfer tokens are deployed on mainnet (e.g., tokens with auto-liquidity or reflection mechanics). An attacker does not need to deploy a custom token; any existing fee-on-transfer ERC20 suffices.
- The attacker profits by receiving the full `amount` on the destination chain while only spending `amount − fee` on the source chain, with the deficit borne by future withdrawers.
- No privileged access, leaked key, or external oracle compromise is required.

---

### Recommendation

Record the vault balance before and after `safeTransferFrom` and use the **actual received amount** — not the caller-supplied `amount` — in the emitted event and cross-chain message:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(actualReceived > 0, "OmniBridge: zero received");
// use actualReceived (cast to uint128) in the event and cross-chain payload
```

This must be applied to both the `InitTransfer` event emission and the `initTransferExtension` payload in `OmniBridgeWormhole`. If fee-on-transfer tokens are intentionally unsupported, add an explicit check `require(actualReceived == amount, "OmniBridge: fee-on-transfer not supported")`.

---

### Proof of Concept

1. Deploy or identify a fee-on-transfer ERC20 `FoT` that deducts 10 % on every transfer.
2. Attacker holds 1 000 `FoT` and approves `OmniBridge`.
3. Attacker calls `initTransfer(FoT, 1000e18, 0, 0, "near:recipient", "")`.
4. `safeTransferFrom` moves 1 000 tokens; vault receives 900 (10 % fee deducted by the token).
5. `InitTransfer` event is emitted with `amount = 1000e18`.
6. NEAR bridge processes the proof and releases 1 000 `FoT`-equivalent tokens to `near:recipient`.
7. Vault deficit: 100 tokens per call. After 10 calls the vault is empty.
8. Any user who previously bridged legitimate `FoT` tokens and tries to bridge back calls `finTransfer`; the `safeTransfer` at line 351–355 reverts because the vault balance is 0 — **permanent fund lock**. [6](#0-5)

    function initTransfer(
        address tokenAddress,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message
    ) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
        currentOriginNonce += 1;
        if (fee >= amount) {
            revert InvalidFee();
        }

        uint256 extensionValue;
        if (tokenAddress == address(0)) {
            if (fee != 0) {
                revert InvalidFee();
            }
            extensionValue = msg.value - amount - nativeFee;
        } else {
            extensionValue = msg.value - nativeFee;
            if (customMinters[tokenAddress] != address(0)) {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    customMinters[tokenAddress],
                    amount
                );
                ICustomMinter(customMinters[tokenAddress]).burn(
                    tokenAddress,
                    amount
                );
            } else if (isBridgeToken[tokenAddress]) {
                BridgeToken(tokenAddress).burn(msg.sender, amount);
            } else {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    address(this),
                    amount
                );
            }
        }

        initTransferExtension(
            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message,
            extensionValue
        );

        emit BridgeTypes.InitTransfer(
            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message
        );
    }
