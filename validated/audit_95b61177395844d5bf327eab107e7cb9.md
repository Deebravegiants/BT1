Audit Report

## Title
Rewards Permanently Locked When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: `contracts/KERNEL/KernelDepositPool.sol`)

## Summary
In `KernelDepositPool.sol`, the `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` even when `totalKernelStaked == 0`. If all stakers exit via `initiateWithdrawal` during an active reward window, the rewards accrued during the zero-staked interval are permanently skipped and locked in the contract when the next staker arrives. The contract's own NatSpec acknowledges this behavior and relies solely on an off-chain operational mitigation that is not enforced by code.

## Finding Description
The `rewardPerToken()` function freezes at `rewardPerTokenStored` whenever `totalKernelStaked == 0`:

```solidity
// L408-414
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
```

The `updateReward` modifier always advances `updatedAt` regardless:

```solidity
// L232-242
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    updatedAt = lastTimeRewardApplicable(); // always advances
    ...
}
```

The exploit sequence:

1. `initiateWithdrawal` is called by the last staker. `updateReward` runs: `rewardPerTokenStored` is checkpointed correctly, `updatedAt` is set to T1. Then `totalKernelStaked` drops to 0 (L326).
2. Time passes T1 → T2. `rewardRate` is still active (`finishAt` is in the future), but `rewardPerToken()` returns the frozen `rewardPerTokenStored`. No rewards accumulate for anyone.
3. A new staker calls `stake`. `updateReward` runs: since `totalKernelStaked` is still 0 at this point, `rewardPerToken()` again returns the frozen value — but `updatedAt` is advanced to T2. Then `totalKernelStaked` increases (L285).
4. The `rewardRate × (T2 - T1)` tokens are permanently skipped. No user's `earned()` will ever account for this interval.

The guard in `notifyRewardAmount` (`if (totalKernelStaked == 0) revert NoStakedTokens()` at L570) only prevents starting a new reward period with zero stakers; it does not prevent stakers from exiting after a period has already started. Any user can call `initiateWithdrawal` — no privileged access is required.

The NatSpec at L18–22 explicitly acknowledges this design limitation and states the mitigation is purely operational ("ensuring there are always some tokens staked").

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.** Reward tokens (`rewardsToken`) are permanently locked in the `KernelDepositPool` contract with no recovery mechanism. The amount lost equals `rewardRate × (duration of zero-staked interval)`. For a 30-day reward window, this can represent a substantial fraction of the total reward allocation. This matches the allowed impact: *Medium. Permanent freezing of unclaimed yield.*

## Likelihood Explanation
No privileged access is required. Any combination of stakers calling `initiateWithdrawal` is sufficient to drain `totalKernelStaked` to zero. This can occur organically (loss of confidence, better yield elsewhere) or via a coordinated exit. The `withdrawalDelay` is admin-configurable and can be set to as little as 1 second (the only lower bound is `> 0` per L599). The scenario is realistic and repeatable across every reward period.

## Recommendation
Advance `updatedAt` only when `totalKernelStaked > 0`, so the zero-staked interval's rewards are not silently consumed when the next staker arrives:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

This preserves the gap in `updatedAt`, so when the next staker arrives and `totalKernelStaked` becomes non-zero, the full elapsed reward interval (including the zero-staked gap) is correctly accounted for in `rewardPerToken()`. Alternatively, when `totalKernelStaked` drops to zero, snapshot the remaining undistributed rewards and roll them into the next `notifyRewardAmount` call.

## Proof of Concept

Foundry test sequence (local fork or unit test):

1. Deploy `KernelDepositPool`, initialize with `kernelToken` and `rewardsToken`.
2. Alice stakes `100e18` KERNEL. Admin calls `notifyRewardAmount(1_000e18)` with `duration = 30 days`. Assert `rewardRate = 1_000e18 / 30 days`, `finishAt = block.timestamp + 30 days`.
3. `vm.warp(block.timestamp + 1 days)`. Alice calls `initiateWithdrawal(100e18)`. Assert `totalKernelStaked == 0`, `updatedAt == block.timestamp`.
4. `vm.warp(block.timestamp + 15 days)`. Assert `rewardPerToken() == rewardPerTokenStored` (frozen). Assert `totalKernelStaked == 0`.
5. Bob calls `stake(1e18)`. Inside `updateReward`: `rewardPerToken()` returns frozen value (still 0 staked), `updatedAt` advances to `block.timestamp` (T2). Then `totalKernelStaked = 1e18`.
6. Assert `userRewardPerTokenPaid[Bob] == rewardPerTokenStored` (the frozen value from step 3).
7. `vm.warp(finishAt)`. Bob calls `getReward()`. Assert Bob only earned rewards for the last ~14 days, not the 15-day zero-staked gap.
