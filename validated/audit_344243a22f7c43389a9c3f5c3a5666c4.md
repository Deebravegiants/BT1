### Title
Stale Cross-Chain Rate in `CrossChainRateReceiver` Enables Oracle Drift Arbitrage on L2 Deposit Pools - (File: `contracts/cross-chain/CrossChainRateReceiver.sol`)

---

### Summary

`CrossChainRateReceiver.getRate()` returns the stored `rate` with no staleness check. All L2 pool contracts (`RSETHPoolV3`, `RSETHPool`, `RSETHPoolNoWrapper`, `RSETHPoolV3ExternalBridge`) call this function to determine how much wrsETH/rsETH to mint per ETH deposited. Because rsETH is a yield-bearing token that monotonically appreciates, any delay in the LayerZero rate update leaves the L2 oracle quoting a rate lower than the true L1 rate. An unprivileged depositor can exploit this gap to mint more wrsETH than deserved, diluting existing holders and extracting their accrued yield.

---

### Finding Description

**Root cause — no staleness guard in `CrossChainRateReceiver.getRate()`:** [1](#0-0) 

The function unconditionally returns `rate`. The companion field `lastUpdated` is recorded on every `lzReceive` call but is never consulted by `getRate()`. [2](#0-1) 

The rate is only refreshed when a LayerZero message arrives from L1 via `MultiChainRateProvider.updateRate()`, which is a permissionless but manually triggered function: [3](#0-2) 

**How the stale rate is consumed — deposit minting formula:**

Every L2 pool computes the wrsETH amount as:

```
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate
```

where `rsETHToETHrate` comes directly from the oracle: [4](#0-3) [5](#0-4) 

The same pattern is present in `RSETHPool` and `RSETHPoolV3ExternalBridge`: [6](#0-5) [7](#0-6) 

Because rsETH only appreciates (staking rewards accumulate on L1), the true rate is always ≥ the last pushed rate. A stale oracle therefore always quotes a rate that is **too low**, causing the division to yield **more** wrsETH than the depositor is entitled to.

---

### Impact Explanation

**Impact: High — Theft of unclaimed yield from existing wrsETH/rsETH holders.**

Existing holders' share of the underlying ETH is diluted by the excess wrsETH minted to the attacker. Concretely:

- Suppose the true L1 rate is 1.06 ETH/rsETH but the stale L2 oracle still reports 1.05 ETH/rsETH (a ~0.95% drift after, e.g., several days without an update).
- Attacker deposits 1 ETH (fee aside): receives `1e18 / 1.05e18 ≈ 0.9524 wrsETH`.
- Fair amount would be `1e18 / 1.06e18 ≈ 0.9434 wrsETH`.
- Excess: ~0.009 wrsETH, worth ~0.0095 ETH at the true rate.
- Once the rate is updated on L2, the attacker's wrsETH is worth more than they paid; the shortfall is borne by all existing holders.

The attacker can then bridge wrsETH to L1 as rsETH and redeem through the L1 withdrawal mechanism, realising the profit. The deposit fee (`feeBps`) does not prevent this: it is charged on the ETH input, not on the rate gap, so any drift exceeding `feeBps` is profitable. [8](#0-7) 

---

### Likelihood Explanation

**Likelihood: Medium.**

- rsETH accrues staking rewards continuously (~3–5% APY). After 24 hours without a rate push, the drift is ~0.01%; after 7 days, ~0.07%. These figures can exceed a low `feeBps` setting.
- `updateRate()` on `MultiChainRateProvider` is permissionless but requires the caller to pay LayerZero fees. There is no on-chain enforcement of a maximum staleness window.
- LayerZero message delays, network congestion, or simply no one paying to push the update create windows of opportunity.
- The attacker needs no special role; only a standard ETH deposit is required. [9](#0-8) 

---

### Recommendation

1. **Add a staleness check in `CrossChainRateReceiver.getRate()`**: revert (or return a sentinel) if `block.timestamp - lastUpdated` exceeds a configurable `maxStaleness` (e.g., 24 hours).

```solidity
function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

2. **Enforce `feeBps ≥ expected maximum drift`**: document and enforce that the deposit fee must exceed the maximum expected oracle drift over the staleness window, mirroring the Liquity rule that redemption fees must exceed the oracle deviation threshold.

3. **Automate rate pushes**: use a keeper or Chainlink Automation to call `updateRate()` at a fixed cadence, ensuring the L2 rate never drifts beyond an acceptable bound.

---

### Proof of Concept

1. Observe that the L2 `CrossChainRateReceiver.lastUpdated` is, say, 48 hours old. The stored `rate` is `1.05e18` while the true L1 `rsETHPrice` (from `LRTOracle`) is `1.06e18`.
2. Call `RSETHPoolV3.deposit{value: 10 ether}("")` on the L2 pool.
3. Pool calls `getRate()` → returns stale `1.05e18`.
4. `rsETHAmount = 10e18 * 1e18 / 1.05e18 = 9.5238e18` wrsETH minted.
5. Fair amount: `10e18 * 1e18 / 1.06e18 = 9.4340e18` wrsETH.
6. Excess: `~0.0898e18` wrsETH (~0.09 ETH of value at true rate).
7. Attacker bridges wrsETH to L1 as rsETH; after the rate is updated on L2, redeems rsETH on L1 for `~10.09 ETH`, having deposited only `10 ETH`.
8. The `~0.09 ETH` profit is extracted from existing wrsETH holders' accrued yield. [10](#0-9) [11](#0-10)
