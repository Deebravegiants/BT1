### Title
Validator Exit Mid-Checkpoint Creates Temporary ETH Accounting Gap That Triggers Protocol-Wide Pause — (`contracts/NodeDelegator.sol`, `contracts/LRTOracle.sol`)

---

### Summary

When a validator exits mid-checkpoint, the 32 ETH swept to the EigenPod is temporarily invisible to `getETHDistributionData`. This causes `getEffectivePodShares` to undercount protocol ETH, which can push `_updateRsETHPrice` to compute a `newRsETHPrice` below `highestRsethPrice` by more than `pricePercentageLimit`, triggering a protocol-wide pause that freezes all user deposits and withdrawals until an admin manually unpauses.

---

### Finding Description

**Step 1 — ETH accounting in `getETHDistributionData`**

`LRTDepositPool.getETHDistributionData` counts ETH in two buckets:

```
ethLyingInNDCs   += nodeDelegatorQueue[i].balance          // NDC contract balance only
ethStakedInEigenLayer += INodeDelegator(...).getEffectivePodShares()
``` [1](#0-0) 

The EigenPod is a **separate contract** from the NDC. Its ETH balance is never added to `ethLyingInNDCs`.

**Step 2 — `getEffectivePodShares` depends on `withdrawableShare`**

```solidity
function getEffectivePodShares() external view override returns (uint256 ethStaked) {
    uint256 withdrawableShare =
        NodeDelegatorHelper.getWithdrawableShare(lrtConfig, IStrategy(lrtConfig.beaconChainETHStrategy()));
    return stakedButUnverifiedNativeETH + withdrawableShare;
}
``` [2](#0-1) 

`withdrawableShare` is read live from `DelegationManager.getWithdrawableShares`, which reflects EigenLayer's current share accounting for the NDC.

**Step 3 — The accounting gap when a validator exits mid-checkpoint**

EigenLayer's checkpoint mechanism works as follows:
- `startCheckpoint` records the pod's ETH balance at that moment.
- `verifyCheckpointProofs` (permissionless — "Anyone can call this method") applies per-validator balance deltas.
- At finalization, any ETH in the pod **not already awarded shares** is credited.

When a validator exits mid-checkpoint:
1. Their beacon-chain balance drops to 0 gwei.
2. The 32 ETH is swept to the EigenPod (execution layer).
3. Anyone calls `eigenPod.verifyCheckpointProofs()` with the 0-balance `BalanceProof`. EigenLayer applies a **−32 ETH share delta**, immediately reducing `withdrawableShare`.
4. The swept 32 ETH sits in the EigenPod — **not** in the NDC's `balance`, **not** yet credited as shares (that only happens at checkpoint finalization).

During the window between step 3 and finalization, `getEffectivePodShares` is understated by ~32 ETH per exited validator.

**Step 4 — Price drop and pause trigger in `_updateRsETHPrice`**

```solidity
uint256 totalETHInProtocol = _getTotalEthInProtocol();   // uses getTotalAssetDeposits → getETHDistributionData → getEffectivePodShares
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);

if (newRsETHPrice < highestRsethPrice) {
    uint256 diff = highestRsethPrice - newRsETHPrice;
    bool isPriceDecreaseOffLimit =
        pricePercentageLimit > 0 && diff > pricePercentageLimit.mulWad(highestRsethPrice);
    if (isPriceDecreaseOffLimit) {
        if (!lrtDepositPool.paused()) lrtDepositPool.pause();
        if (!withdrawalManager.paused()) withdrawalManager.pause();
        _pause();
        return;   // rsETHPrice is NOT updated; protocol is frozen
    }
}
``` [3](#0-2) 

`updateRSETHPrice()` is `public whenNotPaused` — callable by anyone. [4](#0-3) 

**Complete permissionless attack path:**
1. Operator starts a checkpoint (`onlyLRTOperator`).
2. A validator exits; 32 ETH is swept to the EigenPod.
3. **Attacker** calls `eigenPod.verifyCheckpointProofs(...)` with the 0-balance proof (permissionless per EigenLayer spec).
4. **Attacker** calls `LRTOracle.updateRSETHPrice()` (public).
5. `newRsETHPrice` is understated; if the drop exceeds `pricePercentageLimit`, `LRTDepositPool`, `LRTWithdrawalManager`, and `LRTOracle` are all paused.

---

### Impact Explanation

**Medium — Temporary freezing of funds.**

All user deposits and withdrawals are halted. The pause is not self-healing: it persists until an `onlyLRTAdmin` account manually calls `unpause()` on each contract. The ETH is not lost — it is recovered when the checkpoint is finalized and the next `updateRSETHPrice` call is made — but users are locked out for the entire duration.

---

### Likelihood Explanation

- Validator exits are routine operational events; they are not attacker-controlled but are observable on-chain.
- `verifyCheckpointProofs` is permissionless; any EOA can submit the 0-balance proof.
- `updateRSETHPrice` is public; any EOA can call it immediately after.
- Exploitability scales with protocol size: for a protocol with N validators, a single exit causes a ~1/N price drop. With `pricePercentageLimit = 1%` (1e16), fewer than 100 active validators suffice for a single exit to trigger the pause. Multiple simultaneous exits amplify the effect at any scale.
- No front-running, no privileged access, no governance capture required.

---

### Recommendation

1. **Include EigenPod ETH in accounting.** Add `address(eigenPod).balance` to `getEffectivePodShares` (or to `getETHDistributionData`) so swept-but-not-yet-checkpointed ETH is not invisible.
2. **Alternatively**, track the EigenPod's balance at checkpoint start and subtract it from the share delta to avoid double-counting, mirroring EigenLayer's own internal logic.
3. **Tighten the pause trigger.** Consider requiring the price to remain below `highestRsethPrice - pricePercentageLimit` for more than one oracle update before pausing, to avoid single-block transient drops triggering a freeze.
