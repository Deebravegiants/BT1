### Title
`accept_requests` Not Reset to `true` After Resharing Completes Following a TEE-Triggered Halt - (File: `crates/contract/src/lib.rs`)

### Summary

When `verify_tee()` determines that kicking out expired-attestation participants would break the governance-threshold relation, it sets `self.accept_requests = false` and returns without initiating resharing. If participants subsequently initiate and complete a manual resharing via `vote_new_parameters()` + `vote_reshared()`, the `accept_requests` flag is never reset to `true` inside `vote_reshared()`. The contract re-enters the Running state with a healthy new participant set but permanently refuses all new `sign()` / `request_app_private_key()` calls until an explicit `verify_tee()` call is made.

---

### Finding Description

`MpcContract::verify_tee()` contains two branches for `TeeValidationResult::Partial`:

1. **Threshold relation intact** → sets `accept_requests = true`, initiates automatic resharing.
2. **Threshold relation broken** → sets `accept_requests = false`, returns `Ok(false)` *without* initiating resharing. [1](#0-0) 

After path 2, the contract remains in `ProtocolContractState::Running`. Participants can still call `vote_new_parameters()` (which uses `assert_caller_is_signer()`, not the `accept_requests`-gated check) to manually propose a new participant set and initiate resharing. [2](#0-1) 

`vote_reshared()` transitions the contract back to Running by constructing a brand-new `RunningContractState` via `ResharingContractState::vote_reshared()`: [3](#0-2) 

However, `vote_reshared()` in `lib.rs` never touches `self.accept_requests`: [4](#0-3) 

`accept_requests` is a field on `MpcContract` itself, not on `RunningContractState`, so the state-machine transition does not carry a reset. After resharing concludes, `accept_requests` remains `false` even though the network is now healthy with a valid participant set. [5](#0-4) 

---

### Impact Explanation

With `accept_requests = false`, every call to `sign()` and `request_app_private_key()` is rejected at `check_request_preconditions()`. The MPC network is fully operational after resharing (new keyshares, valid TEE attestations, threshold met), yet no user can obtain a chain signature. This permanently freezes the signing service until an operator notices and manually calls `verify_tee()` again — a non-obvious recovery step that is not part of the normal post-resharing flow. This matches the **Medium** allowed impact: *request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants*.

---

### Likelihood Explanation

**Medium.** The trigger is a realistic operational sequence:

1. One or more participants' TEE attestations expire (happens routinely; attestations have a finite validity window).
2. Any participant calls `verify_tee()` — a routine health-check call — and the surviving set is below the threshold relation bound. `accept_requests` is set to `false`.
3. Operators respond by proposing a new participant set via `vote_new_parameters()` (standard governance).
4. Resharing completes via `vote_reshared()`.
5. The contract is Running with a healthy set, but `accept_requests` is still `false`.

No adversarial coordination is required; this is the expected recovery path after a TEE-expiry event that breaks the threshold relation.

---

### Recommendation

Reset `accept_requests = true` inside `vote_reshared()` immediately after the protocol state transitions to Running:

```rust
if let Some(new_state) = self.protocol_state.vote_reshared(key_event_id)? {
    self.protocol_state = new_state;
+   self.accept_requests = true;   // resharing succeeded; network is healthy
    self.recompute_available_foreign_chains();
    // ... cleanup promises
}
```

Alternatively, derive `accept_requests` lazily from the TEE state on every request check rather than caching it as a persistent flag, eliminating the class of bugs where the flag diverges from reality.

---

### Proof of Concept

```
1. Deploy contract; 5 participants, GovernanceThreshold = 5, ReconstructionThreshold = 5.
2. Let participant[4]'s TEE attestation expire.
3. Call verify_tee() from participant[0].
   → TeeValidationResult::Partial, 4 survivors, threshold relation broken (5 > 4).
   → accept_requests = false. Contract stays Running.
4. Participants [0..4] call vote_new_parameters() proposing a 4-participant set
   (dropping participant[4]). Threshold votes accumulate → Resharing state.
5. All 4 participants call start_reshare() then vote_reshared() for each domain.
   → Resharing completes. Contract transitions to Running with new epoch.
   → accept_requests is still false (never reset in vote_reshared()).
6. Any user calls sign(...).
   → Rejected: accept_requests == false.
   → Network is healthy but permanently refuses signing until verify_tee() is called.
``` [6](#0-5) [7](#0-6) [8](#0-7)

    pub fn verify_tee(&mut self) -> Result<bool, Error> {
        log!("verify_tee: signer={}", env::signer_account_id());
        // Caller must be a participant (node or operator).
        self.voter_or_panic();
        let ProtocolContractState::Running(running_state) = &mut self.protocol_state else {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        };
        let current_params = running_state.parameters.clone();

        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);

        match self.tee_state.reverify_and_cleanup_participants(
            current_params.participants(),
            tee_upgrade_deadline_duration,
        ) {
            TeeValidationResult::Full => {
                self.accept_requests = true;
                log!("All participants have an accepted Tee status");
                Ok(true)
            }
            TeeValidationResult::Partial {
                participants_with_valid_attestation,
            } => {
                let remaining = participants_with_valid_attestation.len();
                // Defense in depth: the surviving participant set must keep the full
                // threshold relation intact — the GovernanceThreshold must still sit
                // within its bounds for the smaller set (in particular it must not
                // exceed the remaining participant count or the upper cap) and must
                // remain at least every domain's ReconstructionThreshold (the kickout
                // keeps the existing per-domain thresholds). Otherwise we refuse and
                // wait for manual intervention.
                let max_reconstruction_threshold =
                    max_reconstruction_threshold(running_state.domains.domains());
                if let Err(err) = ThresholdParameters::validate_governance_against_reconstruction(
                    u64::try_from(remaining).expect("participant count fits in u64"),
                    current_params.threshold(),
                    max_reconstruction_threshold,
                ) {
                    log!(
                        "Kicking out participants with an invalid TEE status would break the threshold relation ({:?}); {} participants remain with a valid TEE status. This requires manual intervention. We will not accept new signature requests as a safety precaution.",
                        err,
                        remaining,
                    );
                    self.accept_requests = false;
                    return Ok(false);
                }

                // here, we set it to true, because at this point, we have at least `threshold`
                // number of participants with an accepted Tee status.
                self.accept_requests = true;

                // do we want to adjust the threshold?
                //let n_participants_new = new_participants.len();
                //let new_threshold = (3 * n_participants_new + 4) / 5; // minimum 60%
                //let new_threshold = new_threshold.max(2); // but also minimum 2
                let new_threshold = usize::try_from(current_params.threshold().value())
                    .expect("threshold value fits in usize");

                let threshold_parameters = ThresholdParameters::new(
                    participants_with_valid_attestation,
                    Threshold::new(new_threshold as u64),
                )
                .expect("Require valid threshold parameters"); // this should never happen.
                current_params.validate_incoming_proposal(&threshold_parameters)?;
                // This resharing only changes the participant set, so the
                // per-domain reconstruction-threshold updates map is empty.
                let proposed_parameters =
                    ProposedThresholdParameters::new(threshold_parameters, BTreeMap::new());
                let res = running_state.transition_to_resharing_no_checks(&proposed_parameters);
                if let Some(resharing) = res {
                    self.protocol_state = ProtocolContractState::Resharing(resharing);
                }

                Ok(true)
            }
        }
    }
