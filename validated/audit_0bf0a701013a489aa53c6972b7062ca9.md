### Title
`respond_verify_foreign_tx` Accepts Caller-Supplied `payload_hash` Without Binding It to the Pending Request — (File: `crates/contract/src/lib.rs`)

### Summary

The `respond_verify_foreign_tx` function verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash`, but never verifies that `response.payload_hash` is actually derived from the `request` argument. A single Byzantine MPC participant (strictly below the signing threshold) can reuse a legitimately-produced signature from one completed signing round to resolve a completely different pending request, causing the contract to deliver a forged foreign-chain verification attestation to the waiting caller.

### Finding Description

`respond_verify_foreign_tx` performs exactly two security checks before resolving the pending yield:

1. The caller is an attested participant (`assert_caller_is_attested_participant_and_protocol_active`).
2. `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's root public key. [1](#0-0) 

What it does **not** check is that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: request.request, values: <actual extracted values> })))`. The `request` parameter (which carries the chain type, `tx_id`, extractors, `domain_id`, and `payload_version`) is used only as a map key to locate the pending yield — it is never used to recompute or bound-check the hash. [2](#0-1) 

The correct payload structure that nodes sign is: [3](#0-2) 

The hash commits to both the `ForeignChainRpcRequest` (chain identity, tx id, extractors) and the `values` (extracted observations). The contract never reconstructs this hash from the `request` it already holds, so any 32-byte value that carries a valid MPC signature is accepted as the attestation for any pending request.

Compare this with the analogous `respond` function for plain signing, which derives the payload directly from the request object and cannot be substituted: [4](#0-3) 

The client-side SDK does perform the missing check (`expected_payload_hash == response.payload_hash`), but this is not enforced by the contract: [5](#0-4) 

Bridge contracts that do not use the SDK verifier, or that trust the contract's acceptance as sufficient, receive no on-chain protection.

### Impact Explanation

A Byzantine participant can deliver a forged foreign-chain verification attestation to any waiting caller without producing a new threshold signature. The `payload_hash` returned to the caller will not correspond to the transaction the caller asked to verify. Any bridge contract that gates fund release on the MPC attestation (the primary use-case described in the design doc — Omnibridge inbound flow) and that does not independently recompute the expected hash will accept the forged attestation and release funds for a transaction that was never verified.

This matches the allowed impact: **High — forged foreign-chain verification that causes invalid bridge execution or double-spend conditions.**

### Likelihood Explanation

The attacker is a single Byzantine MPC participant (below the signing threshold). No collusion is required. The attack is fully observable from public on-chain state:

- Pending requests are stored in `pending_verify_foreign_tx_requests` and are publicly readable.
- Once the honest leader submits `respond_verify_foreign_tx` for request R1, the `(payload_hash=H1, signature=sig_over_H1)` pair is visible on-chain.
- The Byzantine participant immediately submits `respond_verify_foreign_tx(request=R2, response={payload_hash=H1, signature=sig_over_H1})` for any other pending request R2.

The only precondition is that at least two requests are pending simultaneously, which is the normal production state for any active bridge.

### Recommendation

The contract must bind the accepted `payload_hash` to the `request` it is resolving. Because the contract does not know the extracted `values`, the cleanest fix is to include the extracted values in the response and have the contract recompute and verify the hash:

```rust
// In respond_verify_foreign_tx, after extracting payload_hash from response:
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(), // add to response DTO
})
.compute_msg_hash()
.map_err(|_| RespondError::InvalidPayloadHash)?;

if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Alternatively, if adding `values` to the response DTO is undesirable, the contract can enforce that `payload_hash` is a commitment to the `request` by requiring the hash to be computed over at least the request fields (accepting that the values portion is trusted from the nodes), and documenting that bridge contracts **must** use `ForeignChainSignatureVerifier::verify_signature` to complete the check.

### Proof of Concept

1. User A submits `verify_foreign_transaction` for Bitcoin tx `T1` → pending request `R1`.
2. User B submits `verify_foreign_transaction` for Bitcoin tx `T2` → pending request `R2`.
3. The MPC network honestly processes `R1`; the leader submits `respond_verify_foreign_tx(request=R1, response={payload_hash=H1, signature=σ1})`. `R1` is resolved; `H1` and `σ1` are now visible on-chain.
4. The Byzantine participant (any single attested node) submits `respond_verify_foreign_tx(request=R2, response={payload_hash=H1, signature=σ1})`.
5. The contract checks: (a) caller is attested ✓; (b) `σ1` is a valid signature over `H1` under the domain key ✓. It resolves `R2` with `{payload_hash=H1, signature=σ1}`.
6. User B's bridge contract receives the response. `H1` is the hash of `T1`'s verification, not `T2`'s. Any bridge that does not call `ForeignChainSignatureVerifier::verify_signature` (or equivalent) accepts the forged attestation and releases funds for `T2` without `T2` ever having been verified.     pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
