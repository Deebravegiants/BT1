### Title
TDH2 Partial Decryption ZK Proof Missing Ciphertext/Label Binding Enables Cross-Ciphertext Replay - (File: src/cbmpc/crypto/tdh2.cpp)

### Summary
The Schnorr-style ZK proof embedded in every TDH2 partial decryption (`ei = H(Xi, Yi, Zi)`) does not commit to the ciphertext, its label, or the public key. Because the proof only implicitly depends on `R1` through `Yi = si * R1`, a partial decryption produced for one ciphertext is cryptographically valid for any other ciphertext that shares the same `R1`. An attacker who controls n−1 parties can choose `r` freely, construct two ciphertexts with identical `R1 = r·G` but different plaintexts/labels, induce the honest party to decrypt the first, and replay that partial decryption to decrypt the second without the honest party's knowledge or consent.

### Finding Description

**Root cause — `private_share_t::decrypt` in `src/cbmpc/crypto/tdh2.cpp` lines 82–90:**

```cpp
Xi = x * R1;
bn_t si = curve.get_random_value();
ecc_point_t Yi = si * R1;
ecc_point_t Zi = si * G;
ei = ro::hash_number(Xi, Yi, Zi).mod(q);   // ← no ciphertext, no label, no Qi
MODULO(q) fi = si + x * ei;
```

The challenge `ei` is `H(Xi, Yi, Zi)`. It does not include `R1`, the ciphertext body `c`, the label `L`, or the party's public share `Qi`.

**Verification — `partial_decryption_t::check_partial_decryption_helper` lines 117–122:**

```cpp
const ecc_point_t& R1 = ciphertext.R1;
ecc_point_t Yi = fi * R1 - ei * Xi;
ecc_point_t Zi = fi * G  - ei * Qi;
bn_t ei_test = ro::hash_number(Xi, Yi, Zi).mod(q);   // same omission
if (ei != ei_test) return coinbase::error(E_CRYPTO);
```

The verifier recomputes `Yi` using `ciphertext.R1`, but then hashes only `(Xi, Yi, Zi)`. Any ciphertext that carries the same `R1` will produce the same `Yi` from the same `(Xi, fi, ei)` triple, so the proof passes unchanged.

**End-to-end exploit path (n = 2, one honest party H, one malicious party M):**

1. M picks scalar `r`, computes `R1 = r·G`, `R2 = r·Γ`.
2. M calls `encrypt(pk, P, L, CT)` using that `r` → valid ciphertext `CT` with `R1`, label `L`.
3. H calls `partial_decrypt(private_share, CT, L, pd_H)` → `pd_H = (rid_H, Xi=x·R1, ei, fi)`. The ciphertext validity check (`ciphertext.verify`) passes because `CT` is well-formed.
4. M calls `encrypt(pk, P', L', CT')` using the **same** `r` → valid ciphertext `CT'` with the **same** `R1`, different label `L'` and different plaintext `P'`.
5. M calls `partial_decrypt(private_share_M, CT', L', pd_M)` → M's own valid partial decryption for `CT'`.
6. M calls `combine_additive(pk, pub_shares, L', [pd_M, pd_H], CT', plaintext)`.
   - `ciphertext.verify(pk, L')` passes (CT' is valid under L').
   - `check_partial_decryption_helper(Qi_H, CT', curve)` for `pd_H`: recomputes `Yi = fi·R1 − ei·Xi` using `CT'.R1 = R1` (same point) → identical `Yi`, identical hash → **accepts**.
7. `V = Xi_M + Xi_H = x_M·R1 + x_H·R1 = (x_M+x_H)·R1 = x·R1 = r·Q` → correct decryption key for `CT'` → M recovers `P'`.

H never called `partial_decrypt` for `CT'` or label `L'`.

The same attack applies to the access-structure path (`combine` in lines 155–178) because it calls the same `check_partial_decryption_helper`.

### Impact Explanation

A below-threshold coalition (n−1 malicious parties in the additive case; fewer in the AC case) can obtain a valid TDH2 decryption of any ciphertext they construct, provided they can induce the honest party to produce a single partial decryption for a ciphertext sharing the same `R1`. Because the attacker freely chooses `r`, they can always arrange this. The label-based access-control guarantee of TDH2 — that a partial decryption for label `L` cannot be used to decrypt under label `L'` — is entirely defeated. This maps to the allowed impact scope: "A single malicious peer or below-threshold coalition obtains a valid … TDH2 decryption result without the required honest participants" and "Attacker-controlled … ciphertexts … labels … are accepted under the wrong … label."

### Likelihood Explanation

The public C++ API (`coinbase::api::tdh2::partial_decrypt`, `combine_additive`, `combine_ac`) and the C ABI (`cbmpc_tdh2_partial_decrypt`, `cbmpc_tdh2_combine_additive`) are both fully reachable without any privileged access. The attacker only needs to be one of the n parties in a TDH2 deployment (n−1 of them, in the additive case). Constructing two ciphertexts with the same `R1` requires only calling `encrypt` twice with the same `r`, which is trivial. No cryptographic hardness assumption is broken.

### Recommendation

Bind the ZK proof challenge to the ciphertext and label by including `R1`, `c`, and `L` in the hash:

```cpp
// in private_share_t::decrypt:
ei = ro::hash_number(Xi, Yi, Zi, R1, ciphertext.c, label).mod(q);

// in check_partial_decryption_helper:
bn_t ei_test = ro::hash_number(Xi, Yi, Zi, R1, ciphertext.c, ciphertext.L).mod(q);
```

This makes each partial decryption cryptographically bound to the specific ciphertext and label for which it was produced, preventing any cross-ciphertext replay.

### Proof of Concept

```cpp
// Pseudocode — all calls use the public API
auto [pk, pub_shares, priv_shares] = tdh2_dkg(n=2, curve_secp256k1);

// Attacker picks r and builds two ciphertexts with the same R1
bn_t r = pick_scalar();
buf_t iv1 = random_iv(), iv2 = random_iv();
ciphertext_t CT  = pk.encrypt("secret_P",  "label_L",  r, random_s(), iv1);
ciphertext_t CT2 = pk.encrypt("secret_P2", "label_L2", r, random_s(), iv2);
// CT.R1 == CT2.R1 == r*G  ✓

// Honest party H decrypts CT under label_L
partial_decryption_t pd_H;
priv_shares[H].decrypt(CT, "label_L", pd_H);   // succeeds, H consented

// Attacker M produces its own partial decryption for CT2
partial_decryption_t pd_M;
priv_shares[M].decrypt(CT2, "label_L2", pd_M);

// Attacker replays pd_H for CT2 — check_partial_decryption_helper accepts it
buf_t plaintext;
combine_additive(pk, pub_shares, "label_L2", {pd_M, pd_H}, CT2, plaintext);
// plaintext == "secret_P2"  — H never consented to decrypt CT2 / label_L2
assert(plaintext == "secret_P2");

error_t partial_decryption_t::check_partial_decryption_helper(const ecc_point_t& Qi, const ciphertext_t& ciphertext,
                                                              ecurve_t curve) const {
  error_t rv = UNINITIALIZED_ERROR;

  if (rv = curve.check(Qi))
    return coinbase::error(rv, "partial_decryption_t::check_partial_decryption_helper: check Qi failed");
  if (rv = curve.check(Xi))
    return coinbase::error(rv, "partial_decryption_t::check_partial_decryption_helper: check Xi failed");

  const auto& G = curve.generator();
  const mod_t& q = curve.order();
  if (!q.is_in_range(ei) || !q.is_in_range(fi)) return coinbase::error(E_CRYPTO);

  const ecc_point_t& R1 = ciphertext.R1;
  ecc_point_t Yi = fi * R1 - ei * Xi;
  ecc_point_t Zi = fi * G - ei * Qi;

  bn_t ei_test = ro::hash_number(Xi, Yi, Zi).mod(q);
  if (ei != ei_test) return coinbase::error(E_CRYPTO);

  return SUCCESS;
}

```cpp
error_t combine_additive(const public_key_t& pub_key, const pub_shares_t& Qi, mem_t label,
                         const partial_decryptions_t& partial_decryptions, const ciphertext_t& ciphertext,
                         buf_t& plain) {
  error_t rv = UNINITIALIZED_ERROR;
  const auto& curve = pub_key.Q.get_curve();
  int n = int(Qi.size());
  for (const auto& _Qi : Qi) {
    if (rv = curve.check(_Qi)) return coinbase::error(rv, "combine_additive: check Qi failed");
  }
  if ((int)partial_decryptions.size() != n) return coinbase::error(E_CRYPTO);

  if (rv = ciphertext.verify(pub_key, label)) return rv;

  ecc_point_t V = curve.infinity();
  for (int i = 0; i < n; i++) {
    const partial_decryption_t& partial_decryption = partial_decryptions[i];

    const int rid = partial_decryption.rid;
    if (rid < 1 || rid > n) return coinbase::error(E_CRYPTO);
    if (rv = partial_decryption.check_partial_decryption_helper(Qi[rid - 1], ciphertext, curve)) return rv;

    V += partial_decryption.Xi;
  }

  if (rv = ciphertext.decrypt(V, plain, label)) return rv;
  return SUCCESS;
}
