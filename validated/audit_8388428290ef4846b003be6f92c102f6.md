Audit Report

## Title
Missing `msg.sender` Authorization in `MerkleDistributor.claim` Allows Anyone to Force Fee-Extracting Claims on Behalf of Any User - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary

`MerkleDistributor.claim` accepts an arbitrary `account` parameter with no check that `account == msg.sender`. Any external caller can supply a victim's address, a valid public Merkle proof, and trigger a claim on the victim's behalf. Because the contract deducts a protocol fee (up to 10%) before transferring tokens to `account`, the victim permanently loses the fee portion of their unclaimed yield without ever consenting to the claim.

## Finding Description

`MerkleDistributor.claim` at lines 97–147 accepts four caller-controlled parameters: `index`, `account`, `cumulativeAmount`, and `merkleProof`. The Merkle proof is verified against the tuple `(index, account, cumulativeAmount)` — this binds the data to the victim's address, but it does **not** bind the transaction sender to that address. [1](#0-0) 

There is no check that `account == msg.sender` anywhere in the function. After proof verification, the contract deducts a fee of up to 10% (`MAX_FEE_IN_BPS = 1000`) and sends it to `protocolTreasury`, then sends the remainder to `account`. [2](#0-1) 

The victim's `userClaims[account].cumulativeAmount` is permanently updated to `cumulativeAmount`, consuming the entire allocation. [3](#0-2) 

The sibling contract `KernelMerkleDistributor._processClaim` correctly guards against this with an explicit check at lines 311–313: [4](#0-3) 

`MerkleDistributor` has no equivalent guard. The fee cap is: [5](#0-4) 

**Exploit flow:**
1. Attacker retrieves victim's public Merkle proof data `(index, victimAddress, cumulativeAmount, proof)` from the off-chain distribution data.
2. Attacker calls `merkleDistributor.claim(index, victimAddress, cumulativeAmount, proof)` directly.
3. Contract verifies the proof (passes, since the data is valid for the victim).
4. Contract computes `fee = (claimableAmount * feeInBPS) / 10_000` and sends it to `protocolTreasury`.
5. Contract sends `claimableAmount - fee` to `victimAddress`.
6. `userClaims[victimAddress].cumulativeAmount` is set to `cumulativeAmount` — the victim can never re-claim the fee portion.

## Impact Explanation

**High — Theft of unclaimed yield.** The victim is entitled to `claimableAmount` tokens from the distribution. After a forced claim, they permanently receive only `claimableAmount - fee`. With `feeInBPS = 1000`, this is a 10% permanent loss of their entire unclaimed allocation. The fee is irrecoverably redirected to `protocolTreasury`. The victim's claim state is fully consumed, so no recovery path exists. This matches the allowed impact "High. Theft of unclaimed yield" exactly.

## Likelihood Explanation

- Merkle proofs are public data published off-chain; any attacker can obtain valid proofs for every user in the tree.
- The attack requires no privileged access, no special role, and no capital at risk.
- A single EOA can iterate over all users in the Merkle tree and force-claim for all of them in one block, extracting up to 10% of the entire distribution.
- The only precondition is `feeInBPS > 0`, which is a configurable protocol parameter and the reason the fee mechanism exists.

## Recommendation

Add a caller authorization check identical to the one in `KernelMerkleDistributor._processClaim`:

```solidity
if (account != msg.sender) {
    revert Unauthorized();
}
```

Insert this check at the top of `MerkleDistributor.claim`, before the Merkle proof verification, mirroring the pattern at: [4](#0-3) 

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

// Foundry fork test (mainnet fork at deployment block)
contract ForcedClaimTest is Test {
    MerkleDistributor distributor = MerkleDistributor(0x2DDB11443bD9Ceb92d4951A05f55eb7096EB53d3);
    address victim = 0x<VICTIM>;
    address attacker = address(0xBEEF);

    function testForcedClaim() public {
        // Obtain victim's proof from off-chain distribution data
        uint256 index = 5;
        uint256 cumulativeAmount = 1000e18;
        bytes32[] memory proof = ...; // fetched from public distribution API

        uint256 feeInBPS = distributor.feeInBPS(); // e.g. 1000 = 10%
        uint256 expectedFee = (cumulativeAmount * feeInBPS) / 10_000;

        address treasury = distributor.protocolTreasury();
        address token = distributor.token();

        uint256 victimBalBefore = IERC20(token).balanceOf(victim);
        uint256 treasuryBalBefore = IERC20(token).balanceOf(treasury);

        // Attacker forces claim on behalf of victim
        vm.prank(attacker);
        distributor.claim(index, victim, cumulativeAmount, proof);

        // Victim received less than their full allocation
        assertEq(IERC20(token).balanceOf(victim) - victimBalBefore, cumulativeAmount - expectedFee);
        // Fee went to treasury, not attacker
        assertEq(IERC20(token).balanceOf(treasury) - treasuryBalBefore, expectedFee);
        // Victim's claim is fully consumed — cannot re-claim
        assertTrue(distributor.isClaimed(index, victim));
    }
}
```
