### Title
Rejected L1 Handler Transactions Incorrectly Committed During Catch-Up, Permanently Suppressing Re-Proposal of L1 Messages — (`crates/apollo_l1_events/src/l1_events_provider.rs`)

---

### Summary

When the `L1EventsProvider` enters catch-up state, the `rejected_txs` parameter is silently dropped inside `accept_commit_while_catching_up`. Every consumed L1 handler transaction—including those that were rejected by the blockifier—is forwarded to `apply_commit_block` with an empty rejected set, so the transaction manager marks them all as `Committed`. Rejected L1 handler transactions that should remain `Pending` and be re-proposed are instead permanently removed from the proposable pool, causing the corresponding L1 messages to be lost.

---

### Finding Description

**Normal (non-catch-up) path — correct behaviour**

`commit_proposal_and_block` in the batcher calls `l1_events_provider_client.commit_block` with two distinct sets:

- `consumed_l1_handler_tx_hashes` — every L1 handler tx that was executed (success **or** failure)
- `rejected_l1_handler_tx_hashes` — the subset that failed execution [1](#0-0) 

`L1EventsProvider::commit_block` receives both sets and, on the normal path, calls `apply_commit_block` with both: [2](#0-1) 

`apply_commit_block` correctly partitions consumed txs: those that appear in `rejected_txs` are routed to `commit_txs` as *rejected* (kept `Pending`); the rest are marked `Committed`: [3](#0-2) 

**Catch-up path — the bug**

When the provider is in catch-up state, `commit_block` calls `accept_commit_while_catching_up` and passes **only** `committed_txs`; `rejected_txs` is silently discarded: [4](#0-3) 

Inside `accept_commit_while_catching_up`, `apply_commit_block` is called with `Default::default()` (an empty set) as `rejected_txs`: [5](#0-4) 

A developer TODO comment acknowledges the gap:

```rust
// TODO(guyn): check what about rejected txs here and in the backlog?
Equal => self.apply_commit_block(committed_txs, Default::default()),
```

The same omission applies to the backlog drain path: [6](#0-5) 

Because `rejected_txs` is empty, `apply_commit_block`'s partition produces no `rejected_and_consumed` entries; every consumed L1 handler tx—including those that failed execution—is passed to `commit_txs` as a *committed* transaction and permanently marked `Committed`.

**Root cause of the data loss**

`accept_commit_while_catching_up` only accepts `committed_txs`; the `CommitBlock` struct stored in the backlog likewise carries only `committed_txs` and `height`. There is no field for `rejected_txs`, so the information is structurally unrecoverable once a block enters the backlog: [7](#0-6) 

**Contrast with expected behaviour**

The design intent is explicit in the flow diagram and in the test `validate_rejected_transaction_accepted_after_rollback`: rejected L1 handler transactions must remain `Pending` so they can be re-proposed in a subsequent block: [8](#0-7) 

---

### Impact Explanation

A rejected L1 handler transaction that should remain `Pending` is permanently promoted to `Committed` during catch-up. It is removed from the proposable index and will never be included in any future block. The corresponding L1→L2 message is silently lost. This is a **High** impact: the admission layer permanently rejects a valid (re-proposable) transaction before sequencing, matching the allowed impact category *"Mempool/gateway/RPC admission… rejects valid transactions before sequencing."*

---

### Likelihood Explanation

Catch-up state is entered whenever `commit_block` arrives for a height that differs from the provider's current height—a routine occurrence after any node restart, sync lag, or round-trip delay between the batcher and the L1 events provider. Any block that contains at least one rejected L1 handler transaction and is processed during catch-up triggers the bug. No privileged access or adversarial peer is required.

---

### Recommendation

1. **Extend `accept_commit_while_catching_up`** to accept a `rejected_txs: IndexSet<TransactionHash>` parameter and forward it to `apply_commit_block` instead of `Default::default()`.
2. **Extend the `CommitBlock` backlog struct** to store `rejected_txs` alongside `committed_txs`, so the information survives backlog queuing.
3. **Update `add_commit_block_to_backlog`** to accept and persist `rejected_txs`.
4. **Update the backlog drain loop** to pass the stored `rejected_txs` to `apply_commit_block`.

---

### Proof of Concept

```
Setup
─────
• L1EventsProvider is in CatchingUp state (e.g., after a restart where the
  batcher is ahead of the provider).
• An L1 handler transaction T was included in block N, executed, and rejected
  by the blockifier (e.g., insufficient gas).

Execution
─────────
1. Batcher calls:
     l1_events_provider_client.commit_block(
         consumed = {T, ...},   // consumed_l1_handler_tx_hashes — includes T
         rejected = {T},        // rejected_l1_handler_tx_hashes
         height  = N,
     )

2. L1EventsProvider::commit_block sees state == CatchingUp and calls:
     self.accept_commit_while_catching_up({T, ...}, N)
   — rejected = {T} is dropped here.

3. accept_commit_while_catching_up calls:
     self.apply_commit_block({T, ...}, Default::default())
   — rejected_txs is empty.

4. apply_commit_block partitions:
     rejected_and_consumed = {}   (empty — T is not in the empty rejected set)
     committed_txs         = {T, ...}

5. tx_manager.commit_txs(&[T, ...], &[]) marks T as Committed.

Result
──────
T is permanently Committed. It is removed from the proposable index.
The L1 message is never re-proposed and is silently lost.

        consumed_txs: IndexSet<TransactionHash>,
