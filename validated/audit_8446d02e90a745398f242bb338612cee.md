Audit Report

## Title
Dust Withdrawal Queue Inflation Enables Unbounded Gas Consumption and Blocks `sweepRemainingAssets` — (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
`minRsEthAmountToWithdraw` defaults to `0` for every asset, meaning the only effective guard in `initiateWithdrawal` is `rsETHUnstaked == 0`. Any rsETH holder can flood the withdrawal queue with arbitrarily many 1-wei requests. Once unlocked, each request can only be retired one-at-a-time via `completeWithdrawalForUser`, forcing O(N) operator transactions before `sweepRemainingAssets` can execute, and imposing unbounded cumulative gas costs on the operator.

## Finding Description
**Root cause — no enforced minimum withdrawal amount**

`minRsEthAmountToWithdraw` is a plain mapping that defaults to `0`:

```solidity
// L35
mapping(address asset => uint256) public minRsEthAmountToWithdraw;
```

The guard in `initiateWithdrawal` (L162–164) is:

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```

When `minRsEthAmountToWithdraw[asset] == 0`, the condition `rsETHUnstaked < 0` is vacuously false for `uint256`, so any amount ≥ 1 wei passes. `setMinRsEthAmountToWithdraw` (L330–333) requires an explicit admin call; there is no initializer-time enforcement.

**Queue inflation**

Each accepted `initiateWithdrawal` call appends a new entry to the global FIFO queue via `_addUserWithdrawalRequest` (L744–759) and increments `nextUnusedNonce[asset]`. With 1 wei of rsETH per call, `assetsCommitted[asset]` increases by at most 1 wei per request (and may increase by 0 if `getExpectedAssetAmount` rounds down), so the `getAvailableAssetAmount` guard at L170 is effectively bypassed for dust amounts. An attacker holding rsETH from a prior deposit can create thousands of dust entries.

**Impact 1 — `_unlockWithdrawalRequests` O(N) loop**

`unlockQueue` calls `_unlockWithdrawalRequests` (L790–815), which iterates from `nextLockedNonce` to `firstExcludedIndex`. Each iteration performs multiple storage reads and writes (`withdrawalRequests`, `assetsCommitted`, `unlockedWithdrawalsCount`). While the operator controls `firstExcludedIndex` and can batch calls, N dust entries still require O(N) total storage operations spread across O(N) transactions, each paying gas.

**Impact 2 — `sweepRemainingAssets` blocked until O(N) completions**

`sweepRemainingAssets` (L395–414) is gated on:

```solidity
if (hasUnlockedWithdrawals(asset)) revert PendingWithdrawalsExist();
```

`hasUnlockedWithdrawals` (L629–631) returns `true` whenever `unlockedWithdrawalsCount[asset] > 0`. This counter is incremented once per request in `_unlockWithdrawalRequests` (L809) and decremented only in `_processWithdrawalCompletion` (L717). If the attacker never calls `completeWithdrawal`, the operator's only recourse is to call `completeWithdrawalForUser` (L192–204) once per dust entry — O(N) separate transactions — before `sweepRemainingAssets` can succeed.

## Impact Explanation
**Medium. Unbounded gas consumption**: The operator is forced to pay O(N) gas across O(N) `completeWithdrawalForUser` transactions to drain the dust queue. With 1 ETH deposited, an attacker receives ~1e18 rsETH, enabling up to 1e18 dust requests before `assetsCommitted` saturates `getAvailableAssetAmount`. Even a few thousand requests imposes a significant and unbounded operational gas burden.

**Medium. Temporary freezing of unclaimed yield**: `sweepRemainingAssets` — the mechanism for recovering residual LST balances to the treasury — is blocked for as long as any dust withdrawal remains unlocked but unclaimed, temporarily freezing protocol yield.

## Likelihood Explanation
Any rsETH holder (obtained by depositing ETH or LSTs into `LRTDepositPool`) can call `initiateWithdrawal` with 1 wei of rsETH at any time the contract is unpaused. No special privileges are required. The attacker's cost is proportional to N (gas for N `initiateWithdrawal` calls + N wei of rsETH), while the operator's remediation cost is also O(N) but may occur at higher gas prices and requires active monitoring. The attack is repeatable as long as the attacker holds rsETH.

## Recommendation
1. **Enforce a non-zero minimum in the initializer**: Set `minRsEthAmountToWithdraw` to a meaningful floor (e.g., 0.001 rsETH = 1e15) for every supported asset during `initialize`, not just via the optional `setMinRsEthAmountToWithdraw` admin setter.
2. **Add a per-user request cap**: Limit the number of open withdrawal requests per user (analogous to `MAX_WITHDRAWALS_PER_USER` patterns used elsewhere in the ecosystem).
3. **Batch-complete dust withdrawals**: Add an operator function that completes multiple withdrawal requests for a given user in a single call, reducing the O(N) transaction overhead.

## Proof of Concept
```
1. Attacker deposits 1 ETH into LRTDepositPool → receives ~1e18 rsETH.
2. Attacker calls initiateWithdrawal(ETH, 1, "") 10,000 times.
   - minRsEthAmountToWithdraw[ETH] == 0, so no revert.
   - expectedAssetAmount ≈ 0–1 wei per call; assetsCommitted increases negligibly.
3. After withdrawalDelayBlocks pass, operator calls unlockQueue(ETH, 10001, ...).
   - _unlockWithdrawalRequests iterates 10,000 times, each doing 3+ storage ops.
   - Operator must split into many smaller calls to stay within block gas limit.
4. unlockedWithdrawalsCount[ETH] is now 10,000.
5. Manager calls sweepRemainingAssets(ETH) → reverts: PendingWithdrawalsExist.
6. Manager must call completeWithdrawalForUser(ETH, attacker, "") 10,000 times
   before sweepRemainingAssets can succeed.

Foundry test sketch:
- Deploy contracts on a local fork.
- Mint rsETH to attacker (or deposit ETH).
- Loop: vm.prank(attacker); withdrawalManager.initiateWithdrawal(ETH, 1, "");
- vm.roll(block.number + withdrawalDelayBlocks + 1);
- operator calls unlockQueue in batches.
- Assert: sweepRemainingAssets reverts with PendingWithdrawalsExist.
- Assert: operator must call completeWithdrawalForUser N times to unblock.
```
