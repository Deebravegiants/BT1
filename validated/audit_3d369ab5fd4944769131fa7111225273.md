### Title
Propose-Update Deposits for Non-Winning Proposals Are Permanently Locked in the Contract - (File: `crates/contract/src/update.rs`)

### Summary
When a contract update is executed via `do_update`, all competing (non-winning) update proposals are cleared from storage without refunding the NEAR deposits that their proposers paid. Those deposits accumulate in the contract's balance with no withdrawal path, permanently freezing participant funds.

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` requires a NEAR deposit proportional to the storage consumed by the proposal: [1](#0-0) 

The required deposit is computed in `crates/contract/src/update.rs` as `env::storage_byte_cost() * bytes_used`, where `bytes_used` includes the full contract binary size plus a 128-account-ID overhead: [2](#0-1) 

The `UpdateEntry` struct even stores `bytes_used` explicitly, implying it was intended to be used for a refund: [3](#0-2) 

When `do_update` is called after threshold votes are reached, it removes the winning entry and then unconditionally clears **all** remaining entries and votes: [4](#0-3) 

No refund is issued to the proposers of the cleared entries. The `bytes_used` field stored in each `UpdateEntry` is never consulted for a refund. There is no `withdraw` or `reclaim_deposit` function anywhere in the contract.

### Impact Explanation

Every participant who proposed a competing update loses their deposit permanently. For a contract-code proposal carrying a typical WASM binary (hundreds of KB), the required deposit is on the order of several NEAR tokens (NEAR storage costs ~10^19 yoctoNEAR per byte). With N competing proposals, up to (N-1) × deposit_per_proposal NEAR is frozen in the contract balance with no recovery path. This constitutes permanent freezing of funds controlled by the chain-signature contract, matching the Critical impact tier.

### Likelihood Explanation

Contract upgrades are a routine governance operation. It is normal for multiple participants to independently propose updates (e.g., different config values or different code versions) before consensus is reached. Every such governance cycle where more than one proposal exists will silently lock the losing proposers' deposits. The condition is reachable by any participant without any special privilege or collusion.

### Recommendation

In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and issue a `Promise::new(proposer_account).transfer(required_deposit(entry.bytes_used))` for each one. The proposer account must be tracked per entry (add a `proposer: AccountId` field to `UpdateEntry` at proposal time). Alternatively, store the deposit amount directly in `UpdateEntry` and refund it on clearing.

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB WASM binary. Required deposit ≈ 5 NEAR. Excess is refunded; exactly 5 NEAR is retained by the contract.
2. Participant B calls `propose_update` with a different WASM binary. Another 5 NEAR is retained.
3. Threshold participants vote for B's proposal via `vote_update`.
4. `vote_update` calls `do_update`:
   - `self.entries.remove(&id_B)` — removes B's entry.
   - `self.entries.clear()` — silently drops A's entry. [5](#0-4) 
   - No transfer to A is issued.
5. A's 5 NEAR deposit is now permanently locked in the contract balance. There is no function in the contract that can recover it. [6](#0-5)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}
