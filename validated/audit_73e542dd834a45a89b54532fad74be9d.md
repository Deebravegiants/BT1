Audit Report

## Title
Unrestricted `claim` Caller Allows Anyone to Force-Claim on Behalf of Any User, Causing Permanent Fee-Based Yield Loss — (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary
The `claim` function in `MerkleDistributor.sol` accepts an arbitrary `account` parameter but performs no check that `msg.sender == account`. Any external caller with a valid merkle proof for a target user can trigger that user's claim, causing the victim to permanently lose up to 10% of their entitled tokens to the protocol fee. The claim is then marked consumed and cannot be re-executed.

## Finding Description
The `claim` function at L97–147 of `contracts/utils/MerkleDistributor/MerkleDistributor.sol` verifies the merkle proof for `account` but never checks that `msg.sender == account`: [1](#0-0) 

After proof verification, a fee is deducted and sent to `protocolTreasury`, with only the remainder transferred to `account`: [2](#0-1) 

The user's `lastClaimedIndex` and `cumulativeAmount` are then permanently updated, consuming the claim: [3](#0-2) 

By contrast, `KernelMerkleDistributor._processClaim` explicitly guards against this pattern: [4](#0-3) 

`MerkleDistributor.sol` has no equivalent guard. An attacker who observes Alice's index, address, cumulative amount, and merkle proof (all derivable from public off-chain data) can call `claim(aliceIndex, alice, aliceCumulativeAmount, aliceMerkleProof)` directly. Alice receives `claimableAmount - fee` instead of `claimableAmount`, and the fee is irrecoverably sent to the treasury.

## Impact Explanation
**High — Theft of unclaimed yield.** With `MAX_FEE_IN_BPS = 1000` (10%), a user entitled to 1,000 tokens permanently loses up to 100 tokens per forced claim. The loss is irreversible: the claim state is updated and the victim cannot reclaim the fee. The attack is repeatable across every eligible address in the merkle tree in a single block. [5](#0-4) 

## Likelihood Explanation
**High.** Merkle trees and proofs are computed from public off-chain data and routinely published by protocols. No privileged role, leaked key, or governance action is required. The attack path is a single public function call with no preconditions beyond knowing the victim's index, address, and cumulative amount — all of which are public. The attacker's only cost is gas.

## Recommendation
Add an authorization check at the top of `claim`, identical to the one in `KernelMerkleDistributor._processClaim`:

```solidity
if (account != msg.sender) {
    revert Unauthorized();
}
```

Additionally, add `nonReentrant` to `claim` for defense-in-depth, as it is currently absent unlike in `KernelMerkleDistributor`. [6](#0-5) 

## Proof of Concept

```solidity
// Setup: Alice is entitled to 1000e18 tokens; feeInBPS = 1000 (10%).
// Attacker observes Alice's merkle proof from the public tree.

merkleDistributor.claim(aliceIndex, alice, 1000e18, aliceMerkleProof);

// Result:
//   Alice receives: 900e18 tokens (90%)
//   Protocol treasury receives: 100e18 tokens (10%)
//   Alice's claim is permanently consumed — she cannot reclaim the 100e18 fee.
//   Attacker cost: gas only.

// Attacker repeats for every address in the merkle tree in a single block.
```

**Foundry test plan:** Deploy `MerkleDistributor` with `feeInBPS = 1000`. Build a two-leaf merkle tree for Alice and Bob. Call `claim` from an attacker EOA passing Alice's valid proof. Assert Alice received `900e18`, treasury received `100e18`, and a subsequent call from Alice reverts with `AlreadyClaimed`.
