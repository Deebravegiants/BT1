### Title
Multiple Sub-Transactions Can Claim the Same Deposit Refund, Draining the Deposit Pot - (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxCert.hs`)

### Summary

In the Dijkstra era, `dijkstraTotalRefundsTxCerts` reads deposit refund amounts directly from certificate fields rather than from the ledger state. The value-conservation check for each sub-transaction uses this function independently. No batch-level guard prevents multiple sub-transactions from each claiming the same deposit refund for the same credential. The developers have explicitly acknowledged this gap with a disabled (`xit`) test whose body reads `error "TODO: predicate failure not yet implemented"`.

### Finding Description

**Root cause — `dijkstraTotalRefundsTxCerts` ignores ledger state:** [1](#0-0) 

```haskell
-- | Unlike previous eras, we no longer need to lookup refunds from the ledger state,
--   since all of the certificates specify the actual refund and ledger rules will validate
--   that they are accurate.
dijkstraTotalRefundsTxCerts = foldMap' $ \case
  UnRegDepositTxCert _ deposit -> deposit   -- reads amount from cert, not state
  UnRegDRepTxCert  _ deposit   -> deposit
  _ -> zero
```

The `getTotalRefundsTxCerts` instance for `DijkstraEra` discards both lookup callbacks entirely: [2](#0-1) 

```haskell
getTotalRefundsTxCerts _ _ _ = dijkstraTotalRefundsTxCerts
```

**How the value-conservation check uses this per sub-transaction:**

`updateUTxOStateNoFees` is called for every sub-transaction in `dijkstraSubUtxoTransition`: [3](#0-2) 

```haskell
totalRefunds  = certsTotalRefundsTxBody pp certState txBody
totalDeposits = certsTotalDepositsTxBody pp certState txBody
depositChange = totalDeposits <-> totalRefunds
```

For Dijkstra era, `certsTotalRefundsTxBody` resolves to `dijkstraTotalRefundsTxCerts`, so the refund amount is taken from the certificate field, not from the stored deposit in `certState`. Each sub-transaction's value-conservation check therefore passes independently, even when two sub-transactions both carry `UnRegDepositTxCert stakingCred D` for the same credential.

**No batch-level guard exists:**

Compare with withdrawals, where a batch-level check explicitly prevents the total across all sub-transactions from exceeding the original balance: [4](#0-3) 

No equivalent guard exists for deposit refunds across sub-transactions.

**Developer acknowledgement — disabled test with explicit TODO:** [5](#0-4) 

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
  ...
  submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
```

The `xit` marker disables the test because the predicate failure that should reject this transaction has not been implemented. The phrase "not yet implemented" indicates the check is absent, not merely mis-labelled.

**Attack path:**

1. Attacker registers `stakingCred`, paying deposit `D` into `utxosDeposited`.
2. Attacker constructs a top-level transaction with `N` sub-transactions, each containing `UnRegDepositTxCert stakingCred D` and outputs inflated by `D`.
3. Each sub-transaction's value-conservation check passes independently (reads `D` from the certificate).
4. `utxosDeposited` is decremented by `D` for each sub-transaction → total decrement `N × D`.
5. Only `D` was ever deposited; the attacker extracts `(N−1) × D` ADA that was never theirs.

### Impact Explanation

`utxosDeposited` is the ledger's accounting of all outstanding deposit obligations. Draining it by more than was deposited breaks the invariant `sumObligation (obligationCertState certState) == utxosDeposited`. ADA is effectively created from nothing (or destroyed from the deposit pot), constituting a **direct loss/creation of ADA through an invalid ledger state transition**. [6](#0-5) 

### Likelihood Explanation

The Dijkstra era introduces sub-transactions as a new primitive. The attack requires only:
- One registered staking credential (trivially obtained).
- A single top-level transaction with multiple sub-transactions (no privileged role, no governance majority, no key leakage).

The developers have already identified the gap (the `xit` test) but have not yet implemented the guard.

### Recommendation

Add a batch-level check analogous to `validateBatchWithdrawals`: before processing sub-transactions, aggregate all `UnRegDepositTxCert` / `UnRegDRepTxCert` refund claims across the entire batch and verify that no credential appears more than once. Alternatively, thread the evolving `CertState` through the sub-transaction loop so that `certsTotalRefundsTxBody` for sub-transaction `i` sees the state after sub-transactions `0…i−1` have been applied, making a double-claim fail the value-conservation check of the second sub-transaction.

### Proof of Concept

```
1. registerStakeCredential stakingCred   -- deposit D stored in utxosDeposited
2. subTx1 = mkBasicTx ... & certsTxBodyL .~ [UnRegDepositTxCert stakingCred D]
            -- outputs inflated by D; value-conservation passes (reads D from cert)
3. subTx2 = mkBasicTx ... & certsTxBodyL .~ [UnRegDepositTxCert stakingCred D]
            -- same cert; value-conservation passes again (reads D from cert)
4. tx = mkBasicTx ... & subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
5. submitTx_ tx
   -- utxosDeposited decremented by 2D; attacker's outputs contain 2D; only D was deposited
```

This is exactly the scenario the disabled test at `eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs:53` describes, with the predicate failure left as `error "TODO: predicate failure not yet implemented"`.

```haskell
-- | For each account, the total withdrawals across the entire batch should not exceed the original account balance.
-- Unregistered accounts are treated as having 0 balance.
validateBatchWithdrawals ::
  ( EraTx era
  , EraAccounts era
  , DijkstraEraTxBody era
  ) =>
  Accounts era ->
  Tx TopTx era ->
  Test (DijkstraUtxoPredFailure era)
validateBatchWithdrawals accounts tx =
  let allWithdrawals =
        Map.unionsWith (<>) $
          unWithdrawals (tx ^. bodyTxL . withdrawalsTxBodyL)
            : [ unWithdrawals $ subTx ^. bodyTxL . withdrawalsTxBodyL
              | subTx <- OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL
              ]
      badWithdrawals =
        Map.mapMaybeWithKey
          ( \acctAddr withdrawn ->
              let balance = getAccountBalance acctAddr
               in if withdrawn > balance
                    then Just Mismatch {mismatchSupplied = withdrawn, mismatchExpected = balance}
                    else Nothing
          )
          allWithdrawals
   in failureOnNonEmptyMap badWithdrawals WithdrawalsExceedAccountBalance
```
