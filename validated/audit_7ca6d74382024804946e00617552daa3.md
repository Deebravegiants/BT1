### Title
Wrong Availability Factor in `verify_foreign_transaction` Uses Unanimity Instead of Signing Threshold — (`File: crates/contract/src/lib.rs`)

### Summary

`verify_foreign_transaction()` gates foreign-chain requests on `get_supported_foreign_chains()`, which requires **all** active participants to have registered support for a chain (unanimity). The correct factor — per the protocol's own design specification — is the **signing threshold** (t-of-n), exposed by `get_available_foreign_chains()`. A single Byzantine participant strictly below the signing threshold can exploit this wrong factor to permanently block all `verify_foreign_transaction` requests for any chain, breaking the production request-lifecycle safety invariant without any threshold-level collusion.

---

### Finding Description

`verify_foreign_transaction()` calls `get_supported_foreign_chains()` to decide whether to accept a request: [1](#0-0) 

`get_supported_foreign_chains()` computes the supported set using a strict **superset check** — every active participant must appear in the set of nodes that registered support for the chain: [2](#0-1) 

The filter condition `nodes_supporting_chain.is_superset(&active_participant_account_ids)` means **n-of-n** (unanimity). If even one active participant has not called `register_foreign_chain_config` for a chain, the chain is absent from the returned set and every `verify_foreign_transaction` call for it panics with `ForeignChainNotSupported`.

The protocol's own design document and the separately implemented `get_available_foreign_chains()` both specify that the correct factor is the **signing threshold** (t-of-n): [3](#0-2) 

The design document explicitly states:

> `verify_foreign_transaction(C)` is **rejected unless `C` is available** … computed from the per-participant registrations … ≥ signing threshold active participants currently cover. [4](#0-3) 

The test suite confirms the behavioral difference:
- `get_supported_foreign_chains__should_exclude_chains_not_supported_by_all` — unanimity rule.
- `get_available_foreign_chains__should_include_chain_when_at_least_threshold_participants_cover_it` — threshold rule. [5](#0-4) [6](#0-5) 

---

### Impact Explanation

**Medium — request-lifecycle manipulation that breaks a production safety invariant.**

The invariant broken is: *"a `verify_foreign_transaction` request is accepted if and only if ≥ signing_threshold active participants cover the requested chain."* With the wrong factor (unanimity), the effective invariant becomes n-of-n, which is strictly stronger than intended. A single non-registering participant collapses the available set to empty for any chain they omit, causing the contract to reject every inbound bridge verification request for that chain. For the Omnibridge use-case this means users who have already committed funds on the foreign chain cannot obtain the MPC attestation needed to complete the inbound flow — a permanent request-lifecycle freeze reachable by one Byzantine participant below the signing threshold.

---

### Likelihood Explanation

Any single active MPC participant can trigger this by simply not calling `register_foreign_chain_config` for a target chain (or by calling it and omitting the chain). No key material, threshold collusion, or privileged access beyond being one participant is required. The entry path is a normal, unprivileged contract call (or its absence). The condition is stable: once the participant stops registering, every subsequent `verify_foreign_transaction` call for that chain fails until the participant re-registers or is removed from the participant set via a resharing vote.

---

### Recommendation

Replace the `get_supported_foreign_chains()` call inside `verify_foreign_transaction()` with `get_available_foreign_chains()`, which already implements the correct threshold-based availability check. The migration path is already documented and the replacement function already exists in the contract:

```rust
// Current (wrong factor — unanimity):
let supported_chains = self.get_supported_foreign_chains();

// Correct (threshold factor):
let supported_chains = self.get_available_foreign_chains();
```

Ensure `recompute_available_foreign_chains()` is called on every state-changing event (participant registration, resharing completion, node migration) so the cached available set stays consistent — the existing call sites already do this for the new function.

---

### Proof of Concept

1. Deploy the contract with 4 participants and signing threshold 3.
2. Whitelist Bitcoin via `vote_update_foreign_chain_providers`.
3. Have participants 0, 1, 2 call `register_foreign_chain_config` with Bitcoin. Participant 3 does **not** register.
4. Call `verify_foreign_transaction` with a Bitcoin request.
5. **Observed**: the call panics with `ForeignChainNotSupported` because `get_supported_foreign_chains()` requires all 4 participants to have registered (unanimity), and participant 3 has not.
6. **Expected**: the call should succeed because 3 ≥ threshold (3) participants cover Bitcoin, satisfying `get_available_foreign_chains()`.

The test `get_available_foreign_chains__should_include_chain_when_at_least_threshold_participants_cover_it` already proves that `get_available_foreign_chains()` would return Bitcoin as available in step 4, confirming the wrong factor is the sole cause of the rejection. [6](#0-5)

**File:** docs/design/calculating-supported-foreign-chains.md (L31-50)
```markdown

`verify_foreign_transaction(C)` is **rejected unless `C` is available**: the contract fails fast
instead of accepting a request that can't reach the signing threshold and letting it time out. The
rejection is temporary — `C` becomes serviceable again as soon as enough nodes report coverage.

The legacy `get_supported_foreign_chains()` (the intersection rule) is **to be deprecated** in favour
of the two views above.

## Why two sets

**Whitelisted** is a stable, operator-visible commitment: it changes only by vote,
never flaps with node churn, and no single operator can change it. Every node is
expected to cover every whitelisted chain.

**Available** avoids spending resources on a request the network can't fulfill: one
for a chain without a signing threshold's worth of coverage is rejected up front, not
attempted and left to time out. Because it requires `signing_threshold` — not all —
participants, the network tolerates up to `n − signing_threshold` nodes
stopping coverage.
```
