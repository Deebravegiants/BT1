Audit Report

## Title
ETH Deposit Limit Check Missing Incoming Amount Allows Cap Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses an asymmetric comparison for ETH versus ERC20 tokens. The ETH branch omits the incoming deposit `amount` from the limit comparison, meaning the check only detects an already-exceeded cap rather than a would-be-exceeded cap. Any unprivileged depositor can push total ETH deposits above the configured ceiling when the cap is exactly reached.

## Finding Description
At `contracts/LRTDepositPool.sol` lines 676–682, the ETH branch evaluates `totalAssetDeposits > limit` while the ERC20 branch evaluates `totalAssetDeposits + amount > limit`. When `totalAssetDeposits == depositLimit`, the ETH branch returns `false` (not exceeded), the deposit proceeds, and `totalAssetDeposits` ends up at `depositLimit + msg.value`. The ERC20 path would have returned `true` and reverted. The call path is fully unprivileged: `depositETH` (L76) → `_beforeDeposit` (L648) → `_checkIfDepositAmountExceedesCurrentLimit` (L676), with no additional guards between them.

## Impact Explanation
The deposit limit is a protocol-level risk-management invariant. Bypassing it allows total ETH TVL to exceed the intended ceiling. No funds are directly stolen or frozen, but the protocol fails to enforce its stated deposit cap guarantee. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
The boundary condition (`totalAssetDeposits == depositLimit`) is reachable in normal operation given that the deposit pool is a high-traffic entry point and the limit is a fixed configured value. Any unprivileged user can call `depositETH` at that moment with any nonzero `msg.value`. The condition is repeatable: each subsequent deposit while `totalAssetDeposits >= depositLimit` also bypasses the cap (since `totalAssetDeposits > limit` remains `false` only at exact equality, but once exceeded, subsequent calls would be blocked — however the first breach is sufficient). **Likelihood: Medium.**

## Recommendation
Apply the same `+ amount` inclusion used for ERC20 tokens to the ETH branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = X`.
2. Prior `depositETH` calls accumulate total ETH deposits to exactly `X`.
3. Attacker calls `depositETH{value: Y}(minRSETH, "")` for any `Y > 0`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `X > X` → `false` → no revert.
5. `_mintRsETH` executes; `getTotalAssetDeposits(ETH_TOKEN)` now returns `X + Y`.
6. Foundry test: set limit to `1 ether`, deposit `1 ether` via a prior call, then assert a second `depositETH{value: 1}()` does **not** revert — confirming the cap is bypassed. The equivalent ERC20 path with the same setup would revert with `MaximumDepositLimitReached`. [1](#0-0) [2](#0-1) [3](#0-2)
