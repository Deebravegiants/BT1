### Title
Stale Votes from Removed Participants Count Toward TEE Code Hash / Launcher Hash / OS Measurement Whitelisting Threshold — (`crates/contract/src/tee/proposal.rs`, `crates/contract/src/tee/measurements.rs`)

### Summary

`CodeHashesVotes`, `LauncherHashVotes`, and `MeasurementVotes` count all stored votes without filtering by current participants at tally time. If the post-resharing cleanup promise (`clean_tee_status`) fails silently, stale votes from removed participants persist and are counted toward the threshold, allowing a malicious TEE image hash to be whitelisted with fewer than threshold current participants.

### Finding Description

After a resharing event, `vote_reshared` in `lib.rs` spawns a detached promise to call `clean_tee_status`: [1](#0-0) 

`clean_tee_status` calls `tee_state.clean_non_participant_votes(participants)`: [2](#0-1) 

Because this is a `.detach()`-ed promise, if it fails (e.g., gas exhaustion from many stored votes), the main `vote_reshared` transaction still succeeds and stale votes remain in storage.

The critical flaw is that `CodeHashesVotes.count_votes()` counts **all** stored votes without filtering by current participants: [3](#0-2) 

The same unfiltered pattern exists in `LauncherHashVotes.count_votes()`: [4](#0-3) 

And `MeasurementVotes.count_votes()`: [5](#0-4) 

This is architecturally inconsistent with the newer `TeeVerifierVotes.vote()`, which **does** filter stale votes at tally time using `count_for`: [6](#0-5) 

And `ProviderVotes.vote()`, which also filters at tally time: [7](#0-6) 

The codebase itself acknowledges this risk in a test comment: *"simulating a missed `clean_tee_status`"* — but only tests that `ForeignChainRpcWhitelist` and `TeeVerifierVotes` are safe. The `CodeHashesVotes`, `LauncherHashVotes`, and `MeasurementVotes` paths have no equivalent at-tally-time guard. [8](#0-7) 

### Impact Explanation

An attacker who controlled `t-1` participants before resharing (below signing threshold) could have those participants vote for a malicious Docker image hash. After resharing removes those participants, if `clean_tee_status` fails, the stale `t-1` votes persist. A single attacker-controlled participant in the new epoch then casts the final vote, crossing the threshold. The malicious image hash is whitelisted, allowing unauthorized code to run inside TEE nodes — directly enabling unauthorized access to MPC key shares and potential secret recovery.

This maps to the allowed impact: **Bypass of threshold-signature requirements or unauthorized access to MPC key shares, signing capability, or secret material that materially enables forgery or secret recovery.**

### Likelihood Explanation

- The attacker needs `t-1` participants before resharing (below threshold — no signing capability required).
- The attacker needs 1 participant in the new epoch.
- The `clean_tee_status` detached promise must fail. `clean_non_participant_votes` iterates all stored votes (unbounded), and the gas budget is a fixed config value (`clean_tee_status_tera_gas`). A participant set with many pending votes can exhaust this budget, causing silent failure.
- The inconsistency with `TeeVerifierVotes` and `ProviderVotes` shows this is an unintentional design gap, not a deliberate trade-off.

### Recommendation

Apply the same at-tally-time participant filtering used in `TeeVerifierVotes` and `ProviderVotes` to `CodeHashesVotes`, `LauncherHashVotes`, and `MeasurementVotes`. Pass the current `ThresholdParameters` (or `Participants`) into the `vote()` method and use `count_for(|p| participants.is_participant_given_participant_id(&p.get()))` instead of the unguarded `count_votes()`. This makes correctness independent of whether the cleanup promise succeeds.

### Proof of Concept

1. Network has `n=5`, `t=3`. Attacker controls participants P1 and P2.
2. P1 and P2 call `vote_code_hash(malicious_hash)`. `CodeHashesVotes` stores 2 votes. Threshold not yet reached.
3. Resharing removes P1 and P2. New set: {P3, P4, P5}.
4. `vote_reshared` succeeds; `clean_tee_status` detached promise fails (gas exhausted). Stale votes for P1 and P2 remain in `CodeHashesVotes.proposal_by_account`.
5. Attacker's participant P3 (now in new set) calls `vote_code_hash(malicious_hash)`.
6. `CodeHashesVotes.count_votes()` counts P1 + P2 + P3 = 3 votes = threshold.
7. `malicious_hash` is inserted into `AllowedDockerImageHashes`. Unauthorized TEE image is now whitelisted. [9](#0-8) [10](#0-9)

```rust
    /// Stale CodeHashesVotes entries from removed participants must not count toward
    /// quorum after resharing.
    ///
    /// Scenario (N=5, T=3):
    /// 1. P1 and P2 vote for malicious hash before resharing.
    /// 2. Resharing removes P1 and P2. New set: {P3, P4, P5}.
    /// 3. clean_non_participant_votes removes stale votes.
    /// 4. P3 votes for the same hash — only 1 vote, not 3.
    #[test]
    fn test_clean_non_participant_votes_removes_stale_votes() {
        // Build 5 participants
        let mut all_participants = Participants::new();
        let mut account_ids = Vec::new();
        for i in 0..5 {
            let (account_id, info) = gen_participant(i);
            account_ids.push(account_id.clone());
            all_participants.insert(account_id, info).unwrap();
        }

        let mut tee_state = TeeState::default();

        // P0 and P1 vote for a malicious hash before resharing
        let malicious_hash = NodeImageHash::from([0xAA; 32]);
        for account_id in &account_ids[0..2] {
            let mut ctx = VMContextBuilder::new();
            ctx.signer_account_id(account_id.clone());
            testing_env!(ctx.build());
            let auth_id = AuthenticatedParticipantId::new(&all_participants).unwrap();
            tee_state.votes.vote(malicious_hash, &auth_id);
        }
        assert_eq!(tee_state.votes.proposal_by_account.len(), 2);

        // Resharing removes P0 and P1. New participant set: {P2, P3, P4}.
        let new_participants = all_participants.subset(2..5);

        // Clean non-participants (as done by CLEAN_TEE_STATUS after resharing)
        tee_state.clean_non_participant_votes(&new_participants);

        // Stale votes must be removed
        assert_eq!(tee_state.votes.proposal_by_account.len(), 0);

        // P2 votes for the same malicious hash — should be only 1 vote, not 3
        let p2_account = &account_ids[2];
        let mut ctx = VMContextBuilder::new();
        ctx.signer_account_id(p2_account.clone());
        testing_env!(ctx.build());
        let auth_id = AuthenticatedParticipantId::new(&new_participants).unwrap();
        let vote_count = tee_state.votes.vote(malicious_hash, &auth_id);
        assert_eq!(vote_count, 1, "Only the fresh vote from P2 should count");
```
