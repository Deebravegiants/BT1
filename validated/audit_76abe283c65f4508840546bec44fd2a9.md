### Title
Wrong `l1_data_gas_price`, `l2_gas_price`, and `l1_da_mode` in `starknet_estimateFee` During Pending-Data Staleness Window — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

---

### Summary

When the in-memory pending data is stale (its `parent_block_hash` does not match the latest committed block hash), `read_pending_data` synthesises a fallback `DeprecatedPendingBlock`. That fallback copies only `l1_gas_price` from the latest header; it omits `l1_data_gas_price`, `l2_gas_price`, and `l1_da_mode` entirely, because `DeprecatedPendingBlock` has no fields for them. Any call to `starknet_estimateFee` (or `starknet_simulateTransactions`, `starknet_traceBlockTransactions`) with `block_id = Pending` that lands during this window receives a fee estimate computed with `l1_data_gas_price = NonzeroGasPrice::MIN`, `l2_gas_price = NonzeroGasPrice::MIN`, and `use_kzg_da = false`, regardless of what the real pending block contains.

---

### Finding Description

**`read_pending_data` fallback — missing fields** [1](#0-0) 

When `pending_data.block.parent_block_hash() != latest_header.block_hash`, the function returns a synthetic `DeprecatedPendingBlock` built with `..Default::default()`. The struct has no `l1_data_gas_price`, `l2_gas_price`, or `l1_da_mode` fields: [2](#0-1) 

Consequently, the accessor methods on `PendingBlockOrDeprecated` return hard-coded defaults for the `Deprecated` variant: [3](#0-2) 

- `l1_data_gas_price()` → `GasPricePerToken::default()` (both components = 0)
- `l2_gas_price()` → `GasPricePerToken::default()` (both components = 0)
- `l1_da_mode()` → `L1DataAvailabilityMode::Calldata`

**Propagation into `estimate_fee`**

`estimate_fee` calls `read_pending_data` and immediately converts the result: [4](#0-3) 

The conversion faithfully copies whatever the accessors return: [5](#0-4) 

**Block-context construction clamps zeros to `NonzeroGasPrice::MIN`** [6](#0-5) 

Zero prices are silently replaced with `NonzeroGasPrice::MIN`. The `use_kzg_da` flag is derived from the (wrong) `l1_da_mode`: [7](#0-6) 

If the real pending block uses `BLOB` mode, `use_kzg_da` is forced to `false`, and the data-gas component of the fee is computed with the wrong price and the wrong DA mode.

**The staleness window is a normal, recurring condition**

The pending data is updated by a background sync loop. Every time a new block is committed to storage, there is a window — potentially several seconds — during which `pending_data.parent_block_hash` still points to the previous block while `latest_header.block_hash` has already advanced. Any `starknet_estimateFee` request that arrives during this window hits the fallback path.

---

### Impact Explanation

`starknet_estimateFee` with `block_id = Pending` returns an authoritative-looking `FeeEstimation` struct with:

- `l1_data_gas_price` = `NonzeroGasPrice::MIN` instead of the real pending-block value
- `l2_gas_price` = `NonzeroGasPrice::MIN` instead of the real pending-block value
- `overall_fee` computed with wrong gas prices and wrong `use_kzg_da`

A client that uses this estimate to set `max_fee` or resource bounds may submit a transaction that is under-funded (reverts on-chain) or over-funded. The same wrong context is used by `starknet_simulateTransactions` and `starknet_traceBlockTransactions` when `block_id = Pending`.

This matches: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

---

### Likelihood Explanation

The staleness window is a normal, recurring event on every block (~2–10 seconds on Starknet mainnet). No special privileges are required. Any unprivileged client that polls `starknet_estimateFee` with `block_id = Pending` at a moderate rate will regularly receive wrong estimates. The effect is most pronounced when the real pending block uses `BLOB` DA mode (post-0.13.1), because `use_kzg_da` is forced to `false` and the data-gas price is wrong simultaneously.

---

### Recommendation

Replace the `DeprecatedPendingBlock` fallback with a `PendingBlock` (the `Current` variant) that copies all relevant fields from the latest committed header, including `l1_data_gas_price`, `l2_gas_price`, and `l1_da_mode`:

```rust
// In read_pending_data, else branch:
Ok(PendingData {
    block: PendingBlockOrDeprecated::Current(PendingBlock {
        parent_block_hash: latest_header.block_hash,
        l1_gas_price: latest_header.block_header_without_hash.l1_gas_price,
        l1_data_gas_price: latest_header.block_header_without_hash.l1_data_gas_price,
        l2_gas_price: latest_header.block_header_without_hash.l2_gas_price,
        l1_da_mode: latest_header.block_header_without_hash.l1_da_mode,
        timestamp: latest_header.block_header_without_hash.timestamp,
        sequencer_address: latest_header.block_header_without_hash.sequencer,
        starknet_version: latest_header.block_header_without_hash.starknet_version.to_string(),
        ..Default::default()
    }),
    ...
})
```

---

### Proof of Concept

1. Set up a node with one committed block whose header has `l1_data_gas_price.price_in_wei = 1_000_000` and `l1_da_mode = BLOB`.
2. Write pending data into the `Arc<RwLock<PendingData>>` with `parent_block_hash` set to a **different** hash (simulating the staleness window) and the same non-zero `l1_data_gas_price`.
3. Call `starknet_estimateFee` with `block_id = Pending` on any transaction that touches storage (so data gas is non-zero).
4. Assert that the returned `l1_data_gas_price` equals `NonzeroGasPrice::MIN` (not `1_000_000`) and that `overall_fee` is wrong.

The existing unit tests in `crates/apollo_rpc/src/v0_8/api/test.rs` already confirm the fallback behaviour — they explicitly assert `l1_data_gas_price: GasPricePerToken::default()` and `l1_da_mode: L1DataAvailabilityMode::Calldata` when the pending block is deprecated: [8](#0-7)
