### Title
`DelegateAction` Signature Hash Excludes Chain/Network Identifier, Enabling Cross-Network Replay of Meta Transactions - (File: `core/primitives/src/action/delegate.rs`)

### Summary
The `DelegateAction` (NEP-366 meta transaction) signature scheme hashes the action payload without any chain or network identifier. A `SignedDelegateAction` produced for one NEAR network (e.g., mainnet) is cryptographically valid on any other NEAR network (e.g., testnet) where the same account and key exist with a compatible nonce, allowing an unprivileged attacker to replay the signed inner actions on the unintended network.

### Finding Description

`DelegateAction::get_nep461_hash()` constructs the signed digest by wrapping the action in a `SignableMessage` that contains only a 4-byte NEP discriminant and the Borsh-serialized action body: [1](#0-0) 

`SignableMessage` itself contains no chain ID, genesis hash, or network identifier: [2](#0-1) 

The `DelegateAction` struct fields are `sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, and `public_key` — none of which bind the signature to a specific network: [3](#0-2) 

`SignedDelegateAction::verify()` therefore accepts any signature that is valid over this chain-agnostic hash: [4](#0-3) 

By contrast, regular `SignedTransaction` embeds a `block_hash` that implicitly binds it to a specific chain, so ordinary transactions are not affected. The `DelegateAction` path has no equivalent binding.

### Impact Explanation

An attacker who observes a `SignedDelegateAction` submitted to one NEAR network can wrap it in a fresh relayer transaction and submit it to any other NEAR network. If the victim's account exists on the target network with the same key and a nonce at or below the one in the delegate action, the inner actions execute under the victim's identity on that network. Concretely corrupted values include: account balance (via `Transfer`), access key set (via `AddKey`/`DeleteKey`), and contract state (via `FunctionCall`/`DeployContract`).

### Likelihood Explanation

Developers routinely use the same account names and key pairs on both mainnet and testnet. A relayer that receives a `SignedDelegateAction` off-chain (the normal meta-transaction flow) can trivially replay it on a second network. The `max_block_height` bound provides only a time window, not a network binding. The nonce check provides replay protection within a single network but not across networks.

### Recommendation

Include a chain/network identifier in the signed payload. The most robust approach is to add the genesis block hash or a canonical chain-ID string to `SignableMessage` or directly to `DelegateAction`, so that the discriminant uniquely identifies both the message type and the target network. This mirrors the fix applied to the `Trust` contract in the referenced report (hashing the chain ID alongside the signature data).

### Proof of Concept

1. Alice signs a `DelegateAction` on NEAR testnet to transfer 10 NEAR to Bob. The hash is `SHA256(discriminant_366 || borsh(delegate_action))` — no network field.
2. The relayer submits it to testnet. An observer (or the relayer itself) captures the `SignedDelegateAction`.
3. The attacker wraps the identical `SignedDelegateAction` in a new outer transaction addressed to Alice's account on mainnet and submits it.
4. On mainnet, `apply_delegate_action` calls `signed_delegate_action.verify()`, which recomputes `get_nep461_hash()` over the same chain-agnostic bytes and finds the signature valid.
5. The inner actions execute on mainnet under Alice's identity, draining her mainnet balance or mutating her mainnet account state without her authorization. [5](#0-4) [6](#0-5)

impl SignedDelegateAction {
    pub fn verify(&self) -> bool {
        let delegate_action = &self.delegate_action;
        let hash = delegate_action.get_nep461_hash();
        let public_key = &delegate_action.public_key;

        self.signature.verify(hash.as_ref(), public_key)
    }

    pub fn sign(singer: &Signer, delegate_action: DelegateAction) -> Self {
        let signature = singer.sign(delegate_action.get_nep461_hash().as_bytes());
        Self { delegate_action, signature }
    }

impl<'a, T: BorshSerialize> SignableMessage<'a, T> {
    pub fn new(msg: &'a T, ty: SignableMessageType) -> Self {
        let discriminant = ty.into();
        Self { discriminant, msg }
    }

    pub fn sign(&self, signer: &Signer) -> Signature {
        let bytes = borsh::to_vec(&self).expect("Failed to deserialize");
        let hash = hash(&bytes);
        signer.sign(hash.as_bytes())
    }
}
