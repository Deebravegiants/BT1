Based on my investigation of the `MetricOmmPool.sol` contract, I found a direct structural analog to M-02.

### Title
Exact-Output Swaps Silently Deliver Less Token Than Requested — (`metric-core/contracts/MetricOmmPool.sol`)

### Summary

When a caller requests an exact-output swap (`amountSpecified < 0`) but the pool holds less liquidity than the requested output, the pool silently caps the output to whatever is available and proceeds. No revert occurs, no minimum-output guard exists, and the caller's callback receives a smaller-than-requested delta with no on-chain signal that the fill was partial.

### Finding Description

In `_swapToken1ForToken0SpecifiedOutput`, before any bin iteration begins, the requested output is unconditionally clamped to the pool's total available token0:

```solidity
// MetricOmmPool.sol  _swapToken1ForToken0SpecifiedOutput
uint256 totalAvailableToken0Scaled = binTotals.scaledToken0;
if (amountOutScaled > totalAvailableToken0Scaled) {
    amountOutScaled = totalAvailableToken0Scaled;   // silent cap
}
``` [1](#0-0) 

The capped `amountOutScaled` is then returned as the actual output, and `_executeSwap` propagates it back as `amount0DeltaScaled`:

```solidity
(amountInScaled, amountOutScaled, protocolFeeScaled, feeExclusiveInputScaled) =
    _swapToken1ForToken0SpecifiedOutput(amountOutScaled, params);
amount0DeltaScaled = -int256(amountOutScaled);   // less than requested
amount1DeltaScaled =  int256(amountInScaled);
``` [2](#0-1) 

The outer `swap` function then transfers the reduced token0 to the recipient and calls the callback with the reduced deltas:

```solidity
transferToken0(recipient, uint256(-amount0Delta));   // less than -amountSpecified
IMetricOmmSwapCallback(msg.sender).metricOmmSwapCallback(amount0Delta, amount1Delta, callbackData);
``` [3](#0-2) 

The `swap` signature accepts only `priceLimitX64` as a slippage guard:

```solidity
function swap(
    address recipient,
    bool zeroForOne,
    int128 amountSpecified,
    uint128 priceLimitX64,      // price guard only — no minAmountOut
    bytes calldata callbackData,
    bytes calldata extensionData
) external whenNotPaused nonReentrant(PoolActions.SWAP) returns (int128, int128)
``` [4](#0-3) 

`priceLimitX64` guards against price movement but does **not** guard against insufficient pool depth. If the pool is deep enough at the right price but simply does not hold enough tokens, the price limit is never breached and the partial fill goes through silently.

### Impact Explanation

A router or aggregator that calls `swap` with `amountSpecified = -X` to obtain exactly X tokens (e.g., to repay a flash loan, to meet a minimum fill for a user order, or to satisfy a downstream protocol) will:

1. Receive fewer than X tokens transferred to `recipient`.
2. Have its callback invoked with a smaller-than-expected `amount0Delta`, causing it to pay less token1 than anticipated.
3. Receive no revert and no on-chain indication that the fill was partial.

If the callback or the caller's downstream logic assumes the full X tokens were received (a reasonable assumption for an exact-output swap), it may proceed with an incorrect state — for example, attempting to repay a loan with insufficient funds, or crediting a user with tokens that were never received. This is a direct analog to M-02: just as a liquidator could receive zero collateral after burning their own tokens, a swap caller can receive materially less output than specified with no protection.

### Likelihood Explanation

Any pool that has been heavily traded or has thin liquidity relative to a requested exact-output amount is vulnerable. An attacker can also deliberately drain a pool's token0 balance in a prior transaction (or in the same block) to force a subsequent victim's exact-output swap to partially fill. The trigger requires no privileged access — any caller of `swap` with `amountSpecified < 0` is affected.

### Recommendation

Add a `minAmountOut` parameter to `swap` (analogous to the `minCollateralAmount` recommended in M-02). After `_executeSwap` returns, verify that the actual output meets the caller's minimum:

```solidity
function swap(
    address recipient,
    bool zeroForOne,
    int128 amountSpecified,
    uint128 priceLimitX64,
    uint128 minAmountOut,          // NEW: minimum acceptable output
    bytes calldata callbackData,
    bytes calldata extensionData
) external ...
```

After `_executeSwap`:

```solidity
// For zeroForOne exact-output: amount1Delta is negative (output)
if (amountSpecified < 0 && uint256(-amount1Delta) < minAmountOut) revert InsufficientOutput();
// For !zeroForOne exact-output: amount0Delta is negative (output)
if (amountSpecified < 0 && uint256(-amount0Delta) < minAmountOut) revert InsufficientOutput();
```

Alternatively, revert inside `_swapToken*ForToken*SpecifiedOutput` when the available balance is less than the requested output, rather than silently capping it.

### Proof of Concept

```solidity
// Scenario: pool holds only 0.5e18 token0 (scaledToken0 = 0.5e18 * scale)
// Attacker or normal user calls:
pool.swap(
    recipient,
    false,           // zeroForOne = false → buy token0 with token1
    -1e18,           // exact output: wants 1e18 token0
    0,               // no price limit
    ""               // no callbackData
);
// Result:
//   recipient receives only 0.5e18 token0 (silently capped)
//   callback is called with amount0Delta = -0.5e18 (not -1e18)
//   no revert, no event indicating partial fill
//   caller's downstream logic that expected 1e18 token0 operates on wrong state
```

The root cause is the unconditional silent cap at `metric-core/contracts/MetricOmmPool.sol` lines 872–875, combined with the absence of any `minAmountOut` guard in the public `swap` interface.
