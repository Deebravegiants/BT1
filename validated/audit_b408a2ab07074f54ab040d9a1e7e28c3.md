### Title
Missing Cryptographic Output Validation for `AppPublicKey` CKD Variant Allows Malicious Node to Forge Derived Key Response - (File: `crates/contract/src/lib.rs`)

### Summary

In `respond_ckd`, the contract applies a cryptographic pairing check (`ckd_output_check`) only for the `AppPublicKeyPV` variant of `CKDAppPublicKey`. The `AppPublicKey` variant receives **no output validation whatsoever**. A single malicious attested participant (Byzantine node below the signing threshold) can call `respond_ckd` with an arbitrary `CKDResponse` for any pending `AppPublicKey` CKD request, causing the user to receive a forged derived key whose corresponding private key is known to the attacker.

### Finding Description

In `crates/contract/src/lib.rs`, the `respond_ckd` function validates the response only for the `AppPublicKeyPV` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically binds the response to the MPC network's secret key and the user's app public key: [2](#0-1) 

For `AppPublicKey`, the analogous invariant is `big_c = H(app_id, pk) * msk + app_pk * y` and `big_y = g * y`. The contract enforces **none of this**. Any `(big_y, big_c)` pair is accepted and forwarded to the user as the authoritative CKD response.

The `AppPublicKey` variant is the standard (non-privacy-preserving) CKD path used in production: [3](#0-2) 

The `resolve_yields_for` call that follows immediately delivers the unvalidated response to all waiting callers: [4](#0-3) 

### Impact Explanation

A malicious attested participant can craft a `CKDResponse` where they choose `y` freely and set `big_y = g * y`, `big_c = app_pk * y + g * z` for an attacker-chosen scalar `z`. The user decrypts this with their private key `x` (where `app_pk = g * x`) and recovers `big_c - big_y * x = g * z`. The attacker knows `z`, so they know the user's derived private key. Any assets the user subsequently controls with that derived key can be stolen by the attacker.

This satisfies the **Critical** impact class: unauthorized confidential key derivation output without the required participant authorization, materially enabling secret recovery.

### Likelihood Explanation

The attacker must be a single attested participant — a Byzantine node strictly below the signing threshold. The `assert_caller_is_attested_participant_and_protocol_active` guard is the only barrier: [5](#0-4) 

No threshold quorum is required to call `respond_ckd`. One compromised or malicious TEE node that has passed attestation can race to submit a forged response before the honest leader does. Given that the MPC network is designed to tolerate Byzantine participants below threshold, this is a realistic attacker model explicitly in scope.

### Recommendation

Apply the same pairing-based output check to the `AppPublicKey` variant. For `AppPublicKey(pk1)`, the verifiable invariant is:

```
e(big_c, g2) = e(big_y, pk2_derived) · e(H(app_id, mpc_pk), mpc_pk_g2)
```

where `pk2_derived` is derived from `pk1`. Alternatively, require the `AppPublicKey` variant to be deprecated in favor of `AppPublicKeyPV`, which already has the check. At minimum, document explicitly why `AppPublicKey` is safe without an on-chain check — which it is not, given a single-node response model.

### Proof of Concept

1. User calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(user_pk1)` and `domain_id` pointing to a BLS12-381 CKD domain. A pending request is queued.
2. Attacker (a single attested participant) constructs a forged response: pick scalar `y`, set `big_y = g1 * y`, pick scalar `z` (the target private key), set `big_c = user_pk1 * y + g1 * z`.
3. Attacker calls `respond_ckd(request, CKDResponse { big_y, big_c })`.
4. The `AppPublicKey` branch executes with no check; `resolve_yields_for` delivers the forged response to the user.
5. User decrypts: `big_c - big_y * user_sk = g1 * z`. The user believes `z` is their derived private key. The attacker also knows `z` and can sign on behalf of the user on any chain using that key. [6](#0-5)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
