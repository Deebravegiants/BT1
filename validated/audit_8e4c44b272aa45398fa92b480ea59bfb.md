### Title
Statistical Security Reduction in EdDSA 2PC Key Refresh via Floor Rounding of Random Buffer Size — (File: src/cbmpc/protocol/hd_keyset_eddsa_2p.cpp)

---

### Summary

`key_share_eddsa_hdmpc_2p_t::refresh` computes the per-share random buffer size using `bits_to_bytes_floor` (floor division) instead of `bits_to_bytes` (ceiling division). For curves whose group order is not byte-aligned — specifically ed25519 with a 255-bit order — this silently truncates the refresh randomness by up to 7 bits, reducing the statistical security of the key-share refresh below the intended `SEC_P_STAT`-bit level.

---

### Finding Description

In `src/cbmpc/protocol/hd_keyset_eddsa_2p.cpp`, lines 66–71:

```cpp
int rand_bitlen = q.get_bits_count() + SEC_P_STAT;
int rand_size = bits_to_bytes_floor(rand_bitlen);   // ← floor, not ceiling
if (rv = agree_random(job, 2 * rand_bitlen, rand)) return rv;

bn_t r_x = bn_t::from_bin(rand.take(rand_size)) % q;
bn_t r_k = bn_t::from_bin(rand.skip(rand_size).take(rand_size)) % q;
``` [1](#0-0) 

The design intent is to sample `rand_bitlen = |q| + SEC_P_STAT` bits for each of `r_x` and `r_k`, so that reducing modulo `q` yields a distribution within statistical distance `2^{-SEC_P_STAT}` of uniform. The correct byte count for that is `bits_to_bytes(rand_bitlen)` (ceiling), defined as:

```cpp
inline int bits_to_bytes(int bits) { return (bits + 7) >> 3; }   // ceiling
inline int bits_to_bytes_floor(int bits) { return bits >> 3; }   // floor
``` [2](#0-1) 

For ed25519, `q.get_bits_count()` = 255 (via `BN_num_bits`, which is the standard floor-log₂ + 1):

```cpp
int bn_t::get_bits_count() const { return BN_num_bits(*this); }
``` [3](#0-2) 

Concrete arithmetic for ed25519 (`SEC_P_STAT` = 64):

| Expression | Value |
|---|---|
| `rand_bitlen` | 255 + 64 = **319** |
| `bits_to_bytes_floor(319)` | 319 >> 3 = **39** bytes = 312 bits |
| `bits_to_bytes(319)` | (319+7) >> 3 = **40** bytes = 320 bits |
| Actual statistical security | 312 − 255 = **57 bits** |
| Intended statistical security | 319 − 255 = **64 bits** |

`agree_random` correctly generates `bits_to_bytes(2 × 319) = 80` bytes, so the buffer is large enough; the truncation happens only when `rand.take(rand_size)` slices 39 bytes instead of 40. The last 2 bytes of the 80-byte buffer are silently discarded.

For secp256k1 (`|q|` = 256), `rand_bitlen` = 320, which is byte-aligned, so floor = ceiling and no truncation occurs. The bug is specific to curves with non-byte-aligned orders, i.e., ed25519.

---

### Impact Explanation

The `refresh` protocol is the proactive-security primitive for `key_share_eddsa_hdmpc_2p_t`: it re-randomizes both parties' shares so that compromise of old shares does not compromise new ones. The security proof requires that `r_x` and `r_k` are within statistical distance `2^{-SEC_P_STAT}` of uniform modulo `q`. With the floor rounding, the actual statistical distance is `2^{-57}` instead of `2^{-64}`, a 7-bit shortfall below the library's own security parameter. The refresh produces key shares that do not meet the specified security level — invalid cryptographic output with security impact reachable through the public protocol API.

---

### Likelihood Explanation

The bug is deterministically triggered on every call to `key_share_eddsa_hdmpc_2p_t::refresh` when the curve is ed25519. No adversarial input is required; the truncation is a compile-time arithmetic constant. The function is a first-class protocol entry point for EdDSA 2PC HD key management.

---

### Recommendation

Replace the floor variant with the ceiling variant on line 67:

```diff
- int rand_size = bits_to_bytes_floor(rand_bitlen);
+ int rand_size = bits_to_bytes(rand_bitlen);
```

This matches the pattern used everywhere else in the codebase when converting a bit-length security parameter to a byte buffer size (e.g., `drbg.cpp` line 45, `ro.cpp` line 14, `agree_random.cpp` lines 31–32). [4](#0-3) [5](#0-4)

### Proof of Concept

```
Curve: ed25519
q.get_bits_count() = 255   (BN_num_bits of the ed25519 group order)
SEC_P_STAT         = 64

rand_bitlen        = 255 + 64 = 319
rand_size (floor)  = 319 >> 3 = 39 bytes  →  312 bits fed to r_x, r_k
rand_size (ceil)   = (319+7)>>3 = 40 bytes →  320 bits (correct)

Statistical distance of r_x mod q from uniform:
  floor path:  2^{-(312-255)} = 2^{-57}   (7 bits below spec)
  ceiling path: 2^{-(320-255)} = 2^{-65}  (meets SEC_P_STAT=64)

agree_random generates bits_to_bytes(638)=80 bytes; the last 2 bytes
are never consumed, confirming the truncation is not a buffer issue
but a security-parameter shortfall.
```
