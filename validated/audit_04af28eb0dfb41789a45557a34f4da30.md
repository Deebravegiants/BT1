### Title
Excess NEAR Deposit Silently Absorbed When Existing Participant Re-submits Attestation - (File: crates/contract/src/lib.rs)

### Summary
`submit_participant_info` is marked `#[payable]` and accepts an attached NEAR deposit, but only performs deposit validation and refund logic when `attestation_storage_must_be_paid_by_caller` is true. When an existing participant re-submits their attestation (e.g., for a TEE upgrade), this flag is false and any attached NEAR is silently absorbed by the contract with no refund.

### Finding Description

`submit_participant_info` computes a boolean gate that controls whether deposit accounting runs:

```rust
let caller_is_not_participant = self.voter_account().is_err();
let is_new_attestation = matches!(
    attestation_insertion_result,
    ParticipantInsertion::NewlyInsertedParticipant
);
let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... deposit check and refund ...
}
```

When the caller is an **existing participant** (`caller_is_not_participant == false`) **and** the submission is an **update** rather than a new insertion (`is_new_attestation == false`), the entire deposit block is skipped. Because the function is `#[payable]`, NEAR attached to the call is accepted by the runtime unconditionally. No refund is issued and no error is raised.

This is the direct analog of the reported Rocket Pool bug: a conditional branch that handles the "already has credit / already a participant" case omits the check on the incoming value, causing any attached funds to be silently lost. [1](#0-0) [2](#0-1) 

### Impact Explanation

Any NEAR tokens attached to a re-submission call by an existing participant are permanently transferred to the contract account with no mechanism for recovery. The contract has no withdrawal function for accumulated surplus deposits. This breaks the production accounting invariant that `#[payable]` functions must either consume the exact required deposit or refund the remainder — a pattern correctly enforced in every other payable entry point (`sign`, `request_app_private_key`, `verify_foreign_transaction`, `propose_update`) via `require_deposit`. [3](#0-2) [4](#0-3) 

**Impact: Medium** — Balance/accounting invariant broken; participant NEAR is permanently lost to the contract without network-level DoS or operator misconfiguration.

### Likelihood Explanation

TEE upgrade cycles require existing participants to re-submit attestations. This is a routine, expected operation. A participant who habitually attaches a small safety deposit (e.g., to cover potential storage growth) or who copies a prior call that included a deposit will silently lose those funds. The path is reachable by any existing participant with no special privileges. [5](#0-4) 

### Recommendation

Add an unconditional excess-deposit refund at the end of `submit_participant_info`, mirroring the pattern used in `require_deposit`:

```rust
// After the conditional storage-cost block:
let attached = env::attached_deposit();
let consumed = if attestation_storage_must_be_paid_by_caller { cost } else { NearToken::from_yoctonear(0) };
if let Some(diff) = attached.checked_sub(consumed)
    && diff > NearToken::from_yoctonear(0)
{
    Promise::new(account_id).transfer(diff).detach();
}
```

Alternatively, reject any non-zero deposit when `!attestation_storage_must_be_paid_by_caller` with an explicit error, consistent with the principle of least surprise.

### Proof of Concept

1. Participant `alice.near` is already registered (passes `voter_account()` check).
2. Alice re-submits her attestation after a TEE firmware upgrade, attaching `1_000_000_000_000_000_000_000_000` yoctoNEAR (1 NEAR) as a precaution.
3. `attestation_insertion_result` is `ParticipantInsertion::UpdatedExistingParticipant` (not `NewlyInsertedParticipant`), so `is_new_attestation = false`.
4. `caller_is_not_participant = false` because Alice is a current voter.
5. `attestation_storage_must_be_paid_by_caller = false || false = false`.
6. The `if attestation_storage_must_be_paid_by_caller { … }` block is skipped entirely.
7. The function returns `Ok(())`. Alice's 1 NEAR is now held by the contract with no refund path. [6](#0-5)

fn require_deposit(minimum_deposit: NearToken, predecessor: &AccountId) {
    let deposit = env::attached_deposit();
    match deposit.checked_sub(minimum_deposit) {
        None => {
            env::panic_str(
                &InvalidParameters::InsufficientDeposit {
                    attached: deposit.as_yoctonear(),
                    required: minimum_deposit.as_yoctonear(),
                }
                .to_string(),
            );

            if diff > NearToken::from_yoctonear(0) {
                log!("refund excess deposit {diff} to {predecessor}");
                Promise::new(predecessor.clone()).transfer(diff).detach();
            }
        }
    }
