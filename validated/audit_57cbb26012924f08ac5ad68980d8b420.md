### Title
`respond_verify_foreign_tx` Accepts Cross-Request Replayed Signatures, Enabling Forged Foreign-Chain Attestations - (File: crates/contract/src/lib.rs)

### Summary
The `respond_verify_foreign_tx` function verifies that a submitted signature is valid against the provided `payload_hash` using the root public key, but never verifies that `payload_hash` is actually derived from the `request` argument. A single Byzantine MPC participant (attested, strictly below the signing threshold) can replay any previously computed valid foreign-tx signature against a different pending request, causing the contract to deliver a forged attestation to every caller waiting for that request.

### Finding Description
In `respond_verify_foreign_tx` the contract performs:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,   // root public key — no tweak
)
.is_ok()
``` [1](#0-0) 

The check confirms only that `signature` is a valid ECDSA signature over `payload_hash` under the root key. It does **not** verify that `payload_hash = SHA-256(borsh(ForeignTxSignPayload{ request, values }))` for the `request` that was passed in.

Compare this with the regular `respond` function, which correctly extracts the payload directly from the request object and verifies the signature against it:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
)
``` [2](#0-1) 

The canonical payload for foreign-tx responses is `ForeignTxSignPayloadV1 { request, values }`, Borsh-serialized and SHA-256 hashed: [3](#0-2) 

Because the foreign-tx signing domain uses a zero tweak (root key, no derivation), every valid `(payload_hash, signature)` pair produced for **any** foreign-tx request is a valid pair under the same root public key: [4](#0-3) 

After the (insufficient) signature check passes, `resolve_yields_for` drains the entire fan-out queue for `request_Y` and delivers `response_X` to every waiting caller: [5](#0-4) 

The SDK helper `ForeignChainSignatureVerifier::verify_signature` does check `expected_payload_hash == response.payload_hash`, but it requires the caller to supply `expected_extracted_values` — values that are determined by the MPC nodes, not known to the caller in advance: [6](#0-5) 

Bridge contracts that rely on the MPC service precisely to avoid independent foreign-chain queries cannot perform this check.

### Impact Explanation
A single Byzantine attested participant (below the signing threshold) can:

1. Observe a previously computed valid response `response_X` for request `request_X` (e.g., Bitcoin tx `[0xAA;32]`).
2. Wait for a different request `request_Y` (e.g., Bitcoin tx `[0xBB;32]`) to appear in `pending_verify_foreign_tx_requests`.
3. Call `respond_verify_foreign_tx(request_Y, response_X)`.
4. The contract accepts it — `signature_X` is valid over `payload_hash_X` under the root key.
5. All callers waiting for `request_Y` receive `response_X`, whose `payload_hash` commits to Bitcoin tx X, not Bitcoin tx Y.

Any bridge contract that uses the returned `(payload_hash, signature)` to authorize a cross-chain transfer for Bitcoin tx Y is now operating on a forged attestation. This enables invalid bridge execution and potential double-spend conditions — the attacker can attest to a transaction that was never verified, or re-use an attestation for a transaction that has already been consumed.

This matches: **High — forged foreign-chain verification causing invalid bridge execution or double-spend conditions.**

### Likelihood Explanation
- Requires only **one** Byzantine attested MPC participant, strictly below the signing threshold.
- The attacker needs only to have observed one prior legitimate `respond_verify_foreign_tx` call (trivially available on-chain or via node participation).
- The forged call must arrive before the legitimate nodes respond; given the attacker is an active participant, this is a realistic race.
- No threshold collusion, no key leakage, no TEE attack required.

### Recommendation
The contract must bind the response to the request. The most direct fix is to require the responder to supply the extracted `values` alongside the response, then recompute and compare the hash on-chain:

```rust
// In respond_verify_foreign_tx, after receiving `values`:
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values,
});
let expected_hash = expected_payload.compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

This mirrors the binding already enforced in `respond` for regular signatures, where the payload is taken from the request, not from the response.

    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
