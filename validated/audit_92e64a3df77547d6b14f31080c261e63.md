Audit Report

## Title
ETH Deposit Limit Check Omits Incoming Amount, Allowing Cap Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses `totalAssetDeposits > depositLimit` for ETH but `totalAssetDeposits + amount > depositLimit` for ERC-20 assets. The missing `amount` in the ETH branch means any depositor can push the ETH total above the configured cap in a single `depositETH` call. No funds are stolen, but the protocol mints rsETH beyond the intended ceiling.

## Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` (lines 676–682), the ETH branch returns `true` (blocked) only when `totalAssetDeposits > depositLimit`, ignoring the incoming `amount`. The ERC-20 branch correctly tests `totalAssetDeposits + amount > depositLimit`. Because `depositETH` passes `msg.value` as `depositAmount` to `_beforeDeposit` (line 87), which calls `_checkIfDepositAmountExceedesCurrentLimit` (line 661), the guard never accounts for the new ETH being added. When `totalAssetDeposits == depositLimit`, the check evaluates `depositLimit > depositLimit` → `false` → no revert, and the deposit proceeds, minting rsETH for the full `amount` and permanently overshooting the cap.

## Impact Explanation
The ETH deposit cap is a protocol-level safety ceiling. Because the check is incomplete, any depositor can exceed it in a single transaction. The protocol mints rsETH for ETH beyond the limit, violating the invariant the cap is meant to enforce. This matches the allowed impact: **Low — Contract fails to deliver promised returns (deposit limit not enforced for ETH).**

## Likelihood Explanation
`depositETH` is a public payable function with no role restriction. The condition is met naturally as the ETH pool approaches its configured limit, a normal operational state. No front-running, special privilege, or external dependency is required.

## Recommendation
Add `amount` to the ETH comparison to match the ERC-20 path:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 1000 ether`.
2. Through normal deposits, `getTotalAssetDeposits(ETH_TOKEN)` reaches exactly `1000 ether`.
3. Any user calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `1000 ether > 1000 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for 500 ETH; total ETH in protocol becomes `1500 ether`, 50% above the cap.
6. The equivalent ERC-20 deposit would evaluate `1000 + 500 > 1000` → `true` → reverted correctly.

Foundry test sketch:
```solidity
function test_ethDepositBypassesLimit() public {
    vm.deal(alice, 500 ether);
    // fill pool to exactly the limit via prior deposits
    _fillEthPoolToLimit(1000 ether);
    // this should revert but does not
    vm.prank(alice);
    depositPool.depositETH{value: 500 ether}(0, "");
    assertGt(depositPool.getTotalAssetDeposits(ETH_TOKEN), 1000 ether);
}
```
