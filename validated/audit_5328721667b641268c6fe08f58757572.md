### Title
Attacker-Controlled `DirectDeposits` Can Permanently Freeze Stake Key Deposits by Blocking Deregistration - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs`)

---

### Summary

In the Dijkstra era, any unprivileged transaction sender can include `DirectDeposits` in a transaction body to increase the balance of any registered stake account without authorization from the account owner. Because stake key deregistration (`ConwayUnRegCert` / `UnRegTxCert`) requires the account balance to be exactly zero, an attacker can persistently front-run a victim's deregistration attempt by depositing 1 lovelace into the victim's account, permanently blocking recovery of the key deposit.

---

### Finding Description

The Dijkstra era introduces `DirectDeposits` — a map from `AccountAddress` to `Coin` embedded in the transaction body — that allows any transaction sender to credit ADA directly into any registered stake account's balance. The only validations applied are:

1. The target account must be registered (`directDepositsMissingAccounts` check).
2. The network ID of the target address must match the chain's network ID (`validateWrongNetworkInDirectDeposit`).

No witness, signature, or authorization from the account owner is required.

In `dijkstraEntitiesTransition`:

Stake key deregistration in both the Conway and Shelley DELEG rules enforces a hard zero-balance precondition:

```haskell
checkStakeKeyHasZeroRewardBalance = do
  accountState <- mAccountState
  let balanceCompact = accountState ^. balanceAccountStateL
  guard (balanceCompact /= mempty)
  Just $ fromCompact balanceCompact
failOnJust
  checkStakeKeyHasZeroRewardBalance
  (injectFailure . StakeKeyHasNonZeroAccountBalanceDELEG)
``` [3](#0-2) 

The same check exists in the Shelley DELEG rule: [4](#0-3) 

The ENTITIES transition rule processes state in this order: (1) apply withdrawals, (2) process certificates (including deregistration), (3) apply direct deposits. This ordering means that within a single transaction, direct deposits from the same transaction cannot interfere with deregistration. However, an attacker operating across transactions can front-run the victim's deregistration transaction in the mempool.

The victim cannot escape by combining a withdrawal with deregistration in one transaction, because:

- **Legacy mode**: `withdrawalsThatDoNotDrainAccounts` requires the withdrawal to exactly equal the balance. If the attacker front-runs with a 1-lovelace direct deposit, the balance becomes `W+1`, and the victim's withdrawal of `W` fails.
- **Non-legacy mode**: `withdrawalsThatExceedAccountBalance` allows partial withdrawals (`amount ≤ balance`). The victim's withdrawal of `W` succeeds against a balance of `W+1`, but the remaining balance is 1 lovelace, causing the deregistration cert to fail.

In both modes, the attacker can always keep exactly 1 lovelace in the victim's account, permanently blocking deregistration.

---

### Impact Explanation

A stake key deregistration returns the key deposit (currently 2 ADA in Conway, set by `ppKeyDeposit`). By permanently blocking deregistration, the attacker permanently freezes the victim's key deposit. The victim has no on-chain recourse: they cannot atomically drain an attacker-controlled, continuously-updated balance and deregister in the same transaction. Recovery of the frozen deposit would require a protocol change (hard fork) to either remove the zero-balance precondition or add authorization requirements to `DirectDeposits`.

**Impact category: High — Permanent freezing of funds/deposits where recovery requires a hard fork.**

---

### Likelihood Explanation

- The attacker requires no special privilege — any transaction sender on the Dijkstra network can submit a `DirectDeposits` transaction targeting any registered stake credential.
- The cost per attack is one transaction fee plus 1 lovelace (which goes to the victim's account, not burned). The attacker can repeat this indefinitely.
- The victim's account address is publicly derivable from their registered stake credential, which is on-chain.
- The attack is deterministic and requires no timing luck beyond standard mempool front-running, which is straightforward on Cardano given the predictable slot structure.

**Likelihood: High.**

---

### Recommendation

Add an authorization requirement to `DirectDeposits`: require a witness (key signature or script satisfaction) from the target account's staking credential before crediting the deposit. This mirrors the authorization model for withdrawals. Alternatively, remove the zero-balance precondition from stake key deregistration and instead drain the balance to the deregistering party's UTxO as part of the deregistration transition, analogous to the recommendation in the original report.

---

### Proof of Concept

1. Alice registers stake credential `cred_A` (pays 2 ADA key deposit). Her account balance is 0.
2. Alice submits a `ConwayUnRegCert cred_A (SJust 2_ADA)` transaction to deregister and recover her deposit.
3. Bob observes Alice's transaction in the mempool. Bob submits a Dijkstra transaction with:
   ```
   directDeposits = { AccountAddress Mainnet (AccountId cred_A) : 1 }
   ```
   This transaction has no special witnesses — Bob only needs to sign for his own UTxO inputs.
4. Bob's transaction is included in a block before Alice's. Alice's account balance is now 1 lovelace.
5. Alice's deregistration transaction fails: `StakeKeyHasNonZeroAccountBalanceDELEG 1`.
6. Alice retries with a withdrawal of 1 lovelace + deregistration in the same transaction. Bob front-runs again with another 1-lovelace direct deposit.
7. Alice's 2 ADA key deposit is permanently frozen.

**Entry path**: `DijkstraTxBody.directDepositsTxBodyL` → `ENTITIES.dijkstraEntitiesTransition` → `applyDirectDeposits` → account balance increased → `DELEG.checkStakeKeyHasZeroRewardBalance` blocks deregistration. [6](#0-5) [7](#0-6) [3](#0-2)

dijkstraEntitiesTransition = do
  TRC (EntitiesEnv legacyMode certsEnv, certState, certificates) <- judgmentContext
  let Conway.CertsEnv tx pp curEpoch _committee _committeeProposals = certsEnv
      withdrawals = tx ^. bodyTxL . withdrawalsTxBodyL
      accounts = certState ^. certDStateL . accountsL

  runTest $ Conway.validateWithdrawalsDelegated accounts tx

  network <- liftSTS $ asks networkId

  validateWithdrawals legacyMode network withdrawals accounts

  let certStateBeforeCerts =
        certState
          & Conway.updateDormantDRepExpiries tx curEpoch
          & Conway.updateVotingDRepExpiries tx curEpoch (pp ^. ppDRepActivityL)
          & certDStateL . accountsL %~ applyWithdrawals withdrawals
  certStateAfterCerts <-
    trans @(EraRule "CERTS" era) $ TRC (certsEnv, certStateBeforeCerts, certificates)

  let directDeposits = tx ^. bodyTxL . directDepositsTxBodyL
      accountsAfterCerts = certStateAfterCerts ^. certDStateL . accountsL
