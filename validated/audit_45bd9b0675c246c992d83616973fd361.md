### Title
ETH Deposit Limit Bypass Due to Missing Per-Deposit Amount Validation — (File: `contracts/LRTDepositPool.sol`)

### Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies different logic for ETH vs ERC20 assets. For ETH it omits the incoming deposit amount from the comparison, allowing any unprivileged depositor to push total ETH deposits arbitrarily beyond the configured `depositLimitByAsset` in a single transaction.

### Finding Description
In `LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` (lines 676–682), the ETH branch checks only whether the *current* total already exceeds the limit, without adding the incoming deposit amount:

```solidity
// ETH branch — deposit amount `amount` is ignored
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
}
// ERC20 branch — correctly includes the incoming amount
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
```

For ERC20 assets the check is `totalAssetDeposits + amount > limit`, which correctly rejects any deposit that would breach the cap. For ETH the check is `totalAssetDeposits > limit`, which returns `false` (i.e., "not exceeded") whenever the running total is at or below the limit — regardless of how large the incoming deposit is. Because `depositETH` passes `msg.value` as `amount` to `_beforeDeposit`, which calls this function, the `amount` parameter is silently discarded for ETH.

This is the direct analog of the bonding-curve bug: two quantities that must be jointly validated (`totalAssetDeposits + depositAmount` vs `depositLimitByAsset`) are not compared together for ETH, so the limit can be overshot by an arbitrary margin in one call.

### Impact Explanation
An unprivileged user can deposit an arbitrarily large amount of ETH in a single transaction whenever `totalAssetDeposits[ETH] ≤ depositLimitByAsset[ETH]`. The deposit limit is the protocol's primary mechanism for controlling risk exposure and growth rate. Bypassing it means the protocol accepts more ETH than operators intended, potentially exceeding the amount that can be safely delegated to EigenLayer strategies, and violating the invariant that `getTotalAssetDeposits(ETH) ≤ depositLimitByAsset[ETH]` that the rest of the system relies on (e.g., `getAssetCurrentLimit` returning 0 while deposits still succeed).

**Impact class**: Low — Contract fails to deliver promised returns (the deposit cap), but deposited funds are not directly stolen or frozen.

### Likelihood Explanation
The entry point `depositETH` is public and payable with no access control. Any user can call it with an arbitrarily large `msg.value`. The only precondition is that `totalAssetDeposits[ETH] ≤ depositLimitByAsset[ETH]` at the time of the call, which is the normal operating state of the protocol. No front-running, flash loans, or privileged keys are required.

### Recommendation
Add the incoming deposit amount to the ETH branch, matching the ERC20 logic:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept
1. Assume `depositLimitByAsset[ETH_TOKEN] = 100_000 ether` and `getTotalAssetDeposits(ETH_TOKEN) = 99_999 ether`.
2. Attacker calls `depositETH{value: 50_000 ether}(0, "")`.
3. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 50_000 ether)` evaluates `99_999 ether > 100_000 ether` → `false` → deposit is not blocked.
4. `_mintRsETH` mints rsETH for the full 50,000 ETH.
5. After the call `getTotalAssetDeposits(ETH_TOKEN) = 149_999 ether`, 49,999 ETH above the configured limit, while `getAssetCurrentLimit(ETH_TOKEN)` now returns `0` — the limit is retroactively shown as exhausted but was never enforced at deposit time. [1](#0-0) [2](#0-1) [3](#0-2)
