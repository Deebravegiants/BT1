### Title
UTXO Fast-Path Settlement Skips `finalised_utxo_transfers` Guard, Enabling Double-Credit on Re-Submission — (`near/omni-bridge/src/lib.rs`)

---

### Summary

When `utxo_fin_transfer_fast` settles a Near-destination UTXO transfer, it calls `remove_fast_transfer` to erase the fast-transfer entry but **never** calls `add_fin_utxo_transfer`. The UTXO ID is therefore absent from `finalised_utxo_transfers` after settlement. A subsequent submission of the same UTXO ID finds no fast-transfer entry and no finalization record, passes `add_fin_utxo_transfer` without error, and proceeds to pay the recipient a second time.

---

### Finding Description

**Entry point** — `ft_on_transfer` dispatches to `utxo_fin_transfer` when the message variant is `BridgeOnTransferMsg::UtxoFinTransfer`. [1](#0-0) 

**Connector gate** — `utxo_fin_transfer` enforces that `sender_id` (the account that called `ft_transfer_call` on the token) equals the DAO-registered connector. [2](#0-1) 

**Fast-path branch** — if a matching fast-transfer entry exists, execution is handed to `utxo_fin_transfer_fast` and the function returns immediately, skipping `add_fin_utxo_transfer`. [3](#0-2) 

**The gap** — inside `utxo_fin_transfer_fast`, when the destination is `ChainKind::Near`, the fast-transfer entry is **removed** from `fast_transfers` and the function returns `Value(U128(0))`. It never inserts the UTXO ID into `finalised_utxo_transfers`. [4](#0-3) [5](#0-4) 

**Normal-path guard** — on a second submission of the same UTXO ID, `get_fast_transfer_status` returns `None` (entry was removed), so execution falls through to `add_fin_utxo_transfer`. Because `finalised_utxo_transfers` has no record of this UTXO ID, the `require!` inside `add_fin_utxo_transfer` passes. [6](#0-5) 

The call then proceeds to `utxo_fin_transfer_to_near`, which pays the recipient a second time. [7](#0-6) 

**Contrast with the non-Near fast path** — when the destination is not Near, `mark_fast_transfer_as_finalised` is called instead of `remove_fast_transfer`, leaving the entry in `fast_transfers` with `finalised = true`. A re-submission would then hit the `FastTransferAlreadyFinalised` guard. Only the Near-destination branch has this gap. [8](#0-7) 

---

### Impact Explanation

Each UTXO origin event is backed by exactly one on-chain UTXO output. After the fast path settles it, the bridge holds the connector-supplied tokens and has repaid the relayer. Because `finalised_utxo_transfers` is not updated, the bridge's own replay-protection set is empty for that UTXO ID. A second connector submission causes the bridge to mint/transfer the same amount again to the recipient, creating unbacked supply. This is a direct double-credit of native UTXO-backed tokens.

---

### Likelihood Explanation

The connector is a separate contract whose `verify_deposit` function is callable by relayers (the test shows any account with the relayer role can call it). If the connector does not maintain its own per-UTXO finalization set — a reasonable assumption for a relay contract that delegates replay protection to the bridge — any account that can call `verify_deposit` can trigger the second submission. The bridge is the authoritative source of truth for settlement uniqueness; the connector should not be required to compensate for a missing bridge-side guard.

---

### Recommendation

In `utxo_fin_transfer_fast`, after `remove_fast_transfer` for the Near-destination branch, immediately insert the UTXO ID into `finalised_utxo_transfers`:

```rust
let amount = if fast_transfer.get_destination_chain() == ChainKind::Near {
    self.remove_fast_transfer(&fast_transfer.id());
    // ADD: record settlement so re-submission is rejected
    self.add_fin_utxo_transfer(&utxo_fin_transfer_msg.get_transfer_id(origin_chain));
    fast_transfer.amount
} else { ... }
```

This mirrors the normal-path invariant: every settled UTXO ID is permanently recorded in `finalised_utxo_transfers` regardless of which settlement branch was taken. [9](#0-8) 

---

### Proof of Concept

```
State setup:
  - Register fast transfer for (utxo_id=X, recipient=Near(alice), amount=100, relayer=R)

Tx 1 — connector calls ft_transfer_call(bridge, 100, UtxoFinTransfer{utxo_id=X, recipient=Near(alice), ...})
  → utxo_fin_transfer: fast transfer found → utxo_fin_transfer_fast
  → destination == Near → remove_fast_transfer(X)   // fast_transfers[X] deleted
  → send_tokens(token, R, 100)                       // relayer repaid
  → return Value(0)                                  // bridge keeps 0 extra tokens
  → finalised_utxo_transfers: {} (X NOT inserted)

Tx 2 — connector calls ft_transfer_call(bridge, 100, UtxoFinTransfer{utxo_id=X, recipient=Near(alice), ...})
  → utxo_fin_transfer: get_fast_transfer_status(X) == None (removed)
  → add_fin_utxo_transfer(X): X not in set → succeeds, inserts X
  → utxo_fin_transfer_to_near(alice, 100)
  → alice receives 100 tokens (second credit)

Assert: alice.balance == 200  (should be 100)
        relayer.balance == 100 (correctly repaid once)
        Total tokens paid out == 200 from a single 100-token UTXO deposit
```
