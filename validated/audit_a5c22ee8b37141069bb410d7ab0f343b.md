### Title
Deposit Silently Consumed in `submit_participant_info` When Existing Participant Re-submits Attestation - (File: crates/contract/src/lib.rs)

### Summary
`submit_participant_info` is marked `#[payable]` and enforces a deposit-and-refund flow only when `attestation_storage_must_be_paid_by_caller` is `true`. When an existing participant re-submits their attestation (e.g., during a TEE upgrade), that condition evaluates to `false` and the entire deposit-handling block is skipped — including any refund. Any NEAR attached by the caller in that code path is silently absorbed by the contract with no mechanism to recover it.

### Finding Description
In `crates/contract/src/lib.rs`, `submit_participant_info` is declared `#[payable]`, meaning the NEAR runtime will accept any attached deposit without panicking. The function computes a conditional:

```rust
let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... storage cost check and excess refund ...
}
// No else branch — no refund when condition is false
Ok(())
```

When an existing participant (one for whom `self.voter_account().is_ok()`) re-submits their attestation and the insertion result is not `ParticipantInsertion::NewlyInsertedParticipant`, both `is_new_attestation` and `caller_is_not_participant` are `false`. The `if` block is never entered, so:
- No deposit is required (correct).
- No deposit is refunded (incorrect — any attached value is permanently locked in the contract).

This is structurally identical to the Optimism finding: a `#[payable]` entry point accepts value that the underlying logic does not consume, and no refund path exists for that branch.

### Impact Explanation
Any NEAR token attached by an existing participant when re-submitting their attestation is permanently locked in the MPC contract. The contract has no withdrawal or sweep mechanism for such stranded deposits. This is a direct, irreversible loss of the caller's funds. The amount is bounded only by what the caller attaches; during a TEE-upgrade cycle, operators may attach the same deposit they used for their initial submission (covering storage costs), which can be non-trivial.

### Likelihood Explanation
TEE upgrades are a routine, expected operation: every time a new MPC node image is approved, all participants must re-submit their attestations via `submit_participant_info`. Operator tooling or scripts that always attach a deposit (mirroring the initial-submission flow) will silently lose funds on every such re-submission. The function's `#[payable]` annotation gives no indication that attaching a deposit is harmful in the re-submission case.

### Recommendation
Add an unconditional refund of any excess deposit at the end of `submit_participant_info`, mirroring the pattern already used in `require_deposit` and `propose_update`:

```rust
// After the conditional block, always refund any attached deposit
// that was not consumed by storage costs.
let attached = env::attached_deposit();
if attached > NearToken::from_yoctonear(0) {
    Promise::new(account_id).transfer(attached).detach();
}
```

Alternatively, remove `#[payable]` from `submit_participant_info` entirely and handle the storage-cost deposit in a separate, dedicated method, so the runtime panics if any deposit is attached when none is needed.

### Proof of Concept
1. Participant Alice is already registered; `self.voter_account()` returns `Ok(...)`.
2. A new TEE image is approved; Alice must re-submit her attestation.
3. Alice's operator script calls `submit_participant_info` with `1 NEAR` attached (same as her initial submission).
4. `add_participant` returns `ParticipantInsertion::UpdatedParticipant` (not `NewlyInsertedParticipant`), so `is_new_attestation = false`.
5. `caller_is_not_participant = false` because Alice is already a participant.
6. `attestation_storage_must_be_paid_by_caller = false || false = false`.
7. The `if` block is skipped; no refund is issued.
8. The function returns `Ok(())`. Alice's `1 NEAR` is permanently locked in the contract. [1](#0-0) [2](#0-1)
