### Title
`estimate_minimal_gas_vector` Understates DA Gas for Declare V2/V3 Transactions Due to Missing `n_compiled_class_hash_updates` — (`File: crates/blockifier/src/fee/gas_usage.rs`)

### Summary

`estimate_minimal_gas_vector` deliberately sets `n_compiled_class_hash_updates: 0` for all `Declare` transactions, even though Declare V2/V3 transactions always write one compiled class hash entry to state. A self-annotated `TODO` in the source acknowledges the correct value is `1`. This causes the pre-validation fee floor (`check_fee_bounds`) to accept Declare V2/V3 transactions whose resource bounds are below the true minimum execution cost, mirroring the external report's pattern of an optimistic benchmark that understates real-world costs.

---

### Finding Description

In `estimate_minimal_gas_vector`, the `StateChangesCount` for `Transaction::Declare` is:

```rust
Transaction::Declare(_) => StateChangesCount {
    n_storage_updates: 1,
    n_class_hash_updates: 0,
    // TODO(Yoni): BLOCKIFIER-RESET: should be 1.
    n_compiled_class_hash_updates: 0,   // ← acknowledged wrong value
    n_modified_contracts: 1,
},
``` [1](#0-0) 

The test helper `declare_expected_state_changes_count` confirms the actual on-chain footprint for Declare V2/V3 is:

```rust
StateChangesCount {
    n_storage_updates: 1,
    n_modified_contracts: 1,
    n_compiled_class_hash_updates: 1,   // ← the real value
    ..
}
``` [2](#0-1) 

`get_onchain_data_segment_length` adds `n_compiled_class_hash_updates * 2` field elements to the DA segment: [3](#0-2) 

By setting this to `0`, the estimate omits 2 field elements of DA cost. `get_da_gas_cost` then computes a lower blob-gas (KZG DA) or L1-gas (calldata DA) floor, and `os_kzg_da_resources` also receives a shorter `data_segment_length`, further reducing the OS-step estimate.

The understated vector is fed directly into `check_fee_bounds`, which is called from `perform_pre_validation_stage` on every Declare V2/V3 transaction: [4](#0-3) [5](#0-4) 

---

### Impact Explanation

A submitter of a Declare V2/V3 transaction can set `resource_bounds` to a value strictly between the understated floor and the true minimum. `check_fee_bounds` passes (understated floor ≤ user bounds), so the transaction is not rejected at pre-validation. During execution the actual DA gas consumed exceeds the user's bounds; the post-execution fee check reverts the transaction and charges the user their declared bounds — which are less than the true cost. The sequencer therefore processes a Declare transaction whose resource bounds were provably insufficient, violating the invariant that `check_fee_bounds` is supposed to enforce.

Matching impact: **High — pre-validation (admission) accepts Declare V2/V3 transactions with resource bounds below the true minimum execution cost.**

---

### Likelihood Explanation

The understatement is small in absolute terms (2 field elements of DA), but it is structural and unconditional: every Declare V2/V3 transaction is affected regardless of calldata size, account type, or chain configuration. The TODO comment confirms the team is aware the value is wrong. An attacker who inspects the source or observes the floor returned by `estimate_minimal_gas_vector` can trivially craft a Declare V2/V3 transaction that exploits the gap.

---

### Recommendation

Change `n_compiled_class_hash_updates` from `0` to `1` for the `Transaction::Declare` arm in `estimate_minimal_gas_vector`, as the TODO already prescribes:

```rust
Transaction::Declare(_) => StateChangesCount {
    n_storage_updates: 1,
    n_class_hash_updates: 0,
    n_compiled_class_hash_updates: 1,  // V2/V3 always writes compiled_class_hash
    n_modified_contracts: 1,
},
``` [6](#0-5) 

This aligns the estimate with the actual state changes recorded in `declare_expected_state_changes_count` and with the DA cost formula in `get_onchain_data_segment_length`.

---

### Proof of Concept

1. Observe that `estimate_minimal_gas_vector` returns a `GasVector` for a Declare V3 transaction with `n_compiled_class_hash_updates = 0`.
2. Compute the true minimum by calling the same function with `n_compiled_class_hash_updates = 1`; the difference is `get_da_gas_cost({n_compiled_class_hash_updates:1, ...}, use_kzg_da)` — 2 blob-gas units (KZG) or `2 * SHARP_GAS_PER_DA_WORD` L1-gas units (calldata).
3. Construct a Declare V3 transaction whose `AllResourceBounds.l1_data_gas.max_amount` equals the understated floor (i.e., the value returned by the current `estimate_minimal_gas_vector`).
4. Call `perform_pre_validation_stage` — `check_fee_bounds` passes.
5. Execute the transaction — the actual DA gas consumed exceeds the user's `l1_data_gas` bound; the post-execution fee check reverts the transaction and charges the user their declared (insufficient) bounds.

The test helper `declare_expected_state_changes_count` already documents the correct state-change count for V2/V3 and can be used to construct the exact numeric gap. [7](#0-6) [8](#0-7)

/// Returns an estimated lower bound for the gas required by the given account transaction.
pub fn estimate_minimal_gas_vector(
    block_context: &BlockContext,
    tx: &AccountTransaction,
    gas_usage_vector_computation_mode: &GasVectorComputationMode,
) -> GasVector {
    // TODO(Dori, 1/8/2023): Give names to the constant VM step estimates and regression-test them.
    let BlockContext { block_info, versioned_constants, .. } = block_context;
    let state_changes_by_account_tx = match &tx.tx {
        // We consider the following state changes: sender balance update (storage update) + nonce
        // increment (contract modification) (we exclude the sequencer balance update and the ERC20
        // contract modification since it occurs for every tx).
        Transaction::Declare(_) => StateChangesCount {
            n_storage_updates: 1,
            n_class_hash_updates: 0,
            // TODO(Yoni): BLOCKIFIER-RESET: should be 1.
            n_compiled_class_hash_updates: 0,
            n_modified_contracts: 1,
        },
        Transaction::Invoke(_) => StateChangesCount {
            n_storage_updates: 1,
            n_class_hash_updates: 0,
            n_compiled_class_hash_updates: 0,
            n_modified_contracts: 1,
        },
        // DeployAccount also updates the address -> class hash mapping.
        Transaction::DeployAccount(_) => StateChangesCount {
            n_storage_updates: 1,
            n_class_hash_updates: 1,
            n_compiled_class_hash_updates: 0,
            n_modified_contracts: 1,
        },
    };

    // TODO(Yoni): BLOCKIFIER-RESET: reuse TransactionReceipt code.
    let data_segment_length = get_onchain_data_segment_length(&state_changes_by_account_tx);
    let os_steps_for_type = versioned_constants
        .os_resources_for_tx_type(&tx.tx_type(), tx.extended_calldata_length())
        .n_steps
        + versioned_constants.os_kzg_da_resources(data_segment_length).n_steps;

    let resources = ExtendedExecutionResources {
        vm_resources: ExecutionResources { n_steps: os_steps_for_type, ..Default::default() },
        ..Default::default()
    };
    let da_gas_cost = get_da_gas_cost(&state_changes_by_account_tx, block_info.use_kzg_da);
    let vm_resources_cost = get_extended_vm_resources_cost(
        versioned_constants,
        &resources,
        0,
        gas_usage_vector_computation_mode,
    );
    da_gas_cost.checked_add(vm_resources_cost).unwrap_or_else(|| {
        panic!(
            "Overflow in minimal gas estimation; attempted to add {da_gas_cost:?} to \
             {vm_resources_cost:?}"
        )
    })
}
