### Title
Unguarded Double-Refund of Stake Credential Deposit via Multiple Sub-Transactions - (File: `eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs` / `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Ledger.hs`)

### Summary

The Dijkstra era introduces nested sub-transactions. A single top-level transaction can contain multiple sub-transactions, each of which can carry certificates including stake credential de-registration (`UnRegDepositTxCert`). No predicate currently prevents two sub-transactions within the same top-level transaction from each claiming the deposit refund for the **same** staking credential. The deposit is paid once but refunded twice (or more), directly draining ADA from the deposit pot.

---

### Finding Description

The Dijkstra era's `LEDGER` rule processes all sub-transactions sequentially through `SUBLEDGERS` before processing the top-level transaction body:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Ledger.hs
LedgerState utxoStateAfterSubLedgers certStateAfterSubLedgers <-
  trans @(EraRule "SUBLEDGERS" era) $
    TRC
      ( SubLedgerEnv slot mbCurEpochNo txIx pp chainAccountState
          originalUtxo          -- ← original UTxO snapshot, not updated
          (tx ^. isValidTxL)
      , ledgerState
      , subStAnnTxs
      )
``` [1](#0-0) 

The `originalUtxo` is captured once before any sub-transaction is applied and is threaded through the environment for all witness/validation lookups. The deposit-refund accounting in `updateUTxOStateNoFees` uses `certsTotalRefundsTxBody`, which calls `shelleyTotalRefundsTxCerts` with a `lookupDeposit` function. If the state snapshot used for that lookup is the **original** pre-sub-transaction state (because `originalUtxo` is the reference), both sub-transactions independently see the credential as still registered and each independently compute a full refund.

The deposit-refund path in `updateUTxOStateNoFees` is:

```haskell
-- eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/Utxo.hs
totalRefunds = certsTotalRefundsTxBody pp certState txBody
totalDeposits = certsTotalDepositsTxBody pp certState txBody
depositChange = totalDeposits <-> totalRefunds
...
utxosDeposited = utxosDeposited <> depositChange
``` [2](#0-1) 

The developers themselves have acknowledged this gap with a disabled test that explicitly marks the required predicate failure as unimplemented:

```haskell
-- eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs
xit "Multiple subtransactions cannot get the same refund" $ do
  ...
  submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
``` [3](#0-2) 

The `xit` prefix disables the test precisely because the guard does not exist yet. The analog to the external report's "double normalization" is exact: a value (the deposit refund) that should be applied at most once is applied once per sub-transaction that carries the same `UnRegDepositTxCert`, because the per-sub-transaction refund calculation does not see the credential as already consumed by a sibling sub-transaction.

---

### Impact Explanation

Each `UnRegDepositTxCert` for the same credential in a different sub-transaction independently reduces `utxosDeposited` by the full deposit amount. With `N` sub-transactions each carrying the same de-registration certificate, the deposit pot is reduced by `N × deposit` while only `1 × deposit` was ever collected. The excess `(N-1) × deposit` is created from nothing and credited to the attacker's balance, constituting **direct creation of ADA** through an invalid ledger state transition.

This matches: **Critical. Direct loss, creation, or destruction of ADA or native assets through an invalid ledger state transition.**

---

### Likelihood Explanation

The Dijkstra era is in active development and sub-transactions are a new, reachable feature. Any unprivileged transaction sender can craft a top-level transaction with multiple sub-transactions. No special role, key, or governance threshold is required. The only prerequisite is having previously registered a staking credential (paying the deposit), which is a normal user action. The attack is deterministic and repeatable.

---

### Recommendation

Before the Dijkstra era is deployed to mainnet, implement a predicate in the `SUBLEDGERS` (or `SUBLEDGER`) rule that tracks which staking credentials have already been de-registered by earlier sub-transactions within the same top-level transaction, and rejects any subsequent sub-transaction that attempts to claim a refund for the same credential. This is the same deduplication logic already applied within a single transaction's certificate list in `shelleyTotalRefundsTxCerts` (tracking `regCreds` across the cert sequence), extended to span sub-transaction boundaries. [4](#0-3) 

---

### Proof of Concept

1. Register staking credential `C`, paying deposit `D` (e.g., 2 ADA).
2. Construct sub-transaction `subTx1` containing `UnRegDepositTxCert C D`.
3. Construct sub-transaction `subTx2` containing `UnRegDepositTxCert C D`.
4. Submit top-level transaction `tx` with `subTransactionsTxBodyL = OMap.fromFoldable [subTx1, subTx2]`.
5. Because no predicate prevents it (as confirmed by the `xit` test with `error "TODO: predicate failure not yet implemented"`), both sub-transactions succeed.
6. `utxosDeposited` is decremented by `D` twice; the attacker's reward account receives `2D` while only `D` was deposited — net gain of `D` ADA from the deposit pot. [3](#0-2) [5](#0-4)

```haskell
shelleyTotalRefundsTxCerts pp lookupDeposit = snd . F.foldl' accum (mempty, Coin 0)
  where
    keyDeposit = pp ^. ppKeyDepositL
    accum (!regCreds, !totalRefunds) cert =
      case lookupRegStakeTxCert cert of
        Just k ->
          -- Need to track new delegations in case that the same key is later deregistered in
          -- the same transaction.
          (Set.insert k regCreds, totalRefunds)
        Nothing ->
          case lookupUnRegStakeTxCert cert of
            Just cred
              -- We first check if there was already a registration certificate in this
              -- transaction.
              | Set.member cred regCreds -> (Set.delete cred regCreds, totalRefunds <+> keyDeposit)
              -- Check for the deposit left during registration in some previous
              -- transaction. This de-registration check will be matched first, despite being
              -- the last case to match, because registration is not possible without
              -- de-registration.
              | Just deposit <- lookupDeposit cred -> (regCreds, totalRefunds <+> deposit)
            _ -> (regCreds, totalRefunds)
