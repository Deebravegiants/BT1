### Title
ETH Deposit Limit Check Excludes New Deposit Amount, Allowing Limit Bypass - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` applies a weaker deposit-cap check for ETH than for ERC-20 assets. The ETH branch omits the incoming deposit amount from the comparison, so the protocol-wide ETH deposit ceiling can be breached by any depositor.

### Finding Description
Inside `_checkIfDepositAmountExceedesCurrentLimit` there is a conditional branch on `asset == LRTConstants.ETH_TOKEN`:

```solidity
// contracts/LRTDepositPool.sol  lines 676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← new amount NOT added
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← new amount IS added
}
``` [1](#0-0) 

For every ERC-20 asset the guard correctly evaluates `totalAssetDeposits + amount > limit`, blocking a deposit that would push the total over the cap. For ETH the guard only evaluates `totalAssetDeposits > limit`, which is `false` whenever the running total is exactly at (or one wei below) the cap. A depositor can therefore send an arbitrarily large ETH amount and the check will pass, minting rsETH and pushing `totalAssetDeposits` well above the configured ceiling.

The ETH deposit entry point is `depositETH`, which calls `_beforeDeposit`, which calls this function: [2](#0-1) [3](#0-2) 

### Impact Explanation
The deposit limit is a risk-management ceiling that bounds the protocol's exposure to any single asset. Bypassing it for ETH allows the protocol to accept more ETH than the governance-approved cap, inflating rsETH supply beyond the intended maximum and potentially undermining the collateral assumptions that back rsETH. This maps to **Low – contract fails to deliver promised returns (the deposit cap guarantee), but does not directly lose value**.

### Likelihood Explanation
Any unprivileged ETH depositor can trigger this the moment `totalAssetDeposits` for ETH equals the configured `depositLimitByAsset`. No special role, front-running, or oracle manipulation is required. The condition arises naturally as the pool fills up, making exploitation straightforward and likely.

### Recommendation
Add the incoming `amount` to the ETH branch, mirroring the ERC-20 logic:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept
1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 100 ether`.
2. Protocol accumulates `totalAssetDeposits = 100 ether` (exactly at the cap).
3. Attacker calls `depositETH{value: 50 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `100 ether > 100 ether` → `false` → deposit is **not** blocked.
5. `getTotalAssetDeposits(ETH_TOKEN)` now returns `150 ether`, 50 % above the governance-approved ceiling, and the attacker receives rsETH for the full 50 ETH. [1](#0-0)
