### Title
`tierRefScriptFee` Rounds Down While `txscriptfee` Rounds Up — Inconsistent Rounding Allows Underpayment of Reference-Script Fees - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Tx.hs`)

---

### Summary

The Conway minimum-fee calculation is composed of two additive components. The Plutus execution-unit component (`txscriptfee`) rounds its rational result **up** (`ceiling`), as required by the Alonzo formal specification. The reference-script tiered-pricing component (`tierRefScriptFee`) rounds its rational result **down** (`floor`). Because the ledger's fee-sufficiency check is `minFee ≤ theFee`, the floor-rounded minimum is what the ledger enforces. An attacker can therefore submit a transaction whose fee is 1 lovelace below the true cost of its reference scripts, and the ledger will accept it.

---

### Finding Description

**`txscriptfee` — rounds up (correct)**

The Alonzo formal specification (Figure 2, `eras/alonzo/formal-spec/utxo.tex` line 34) defines:

```
txscriptfee (pr_mem, pr_steps)(mem, steps) = ceiling(pr_mem * mem + pr_steps * steps)
```

The implementation faithfully uses `ceiling`:

```haskell
-- libs/cardano-ledger-core/src/Cardano/Ledger/Plutus/ExUnits.hs:185-190
txscriptfee Prices {prMem, prSteps} ExUnits {exUnitsMem = m, exUnitsSteps = s} =
  Coin $
    ceiling $
      (fromIntegral m * unboundRational prMem)
        + (fromIntegral s * unboundRational prSteps)
```

**`tierRefScriptFee` — rounds down (inconsistent)**

The Conway reference-script fee component uses `floor` instead:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Tx.hs:131-133
go !acc !curTierPrice !n
  | n < sizeIncrement =
      Coin $ floor (acc + toRational n * curTierPrice)
```

Both functions are summed together inside `getConwayMinFeeTx`:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Tx.hs:103-112
getConwayMinFeeTx pp tx refScriptsSize =
  alonzoMinFeeTx pp tx <+> refScriptsFee
  where
    refScriptsFee =
      tierRefScriptFee
        (unboundRational $ pp ^. ppRefScriptCostMultiplierG)
        (fromIntegral @Word32 @Int . unNonZero $ pp ^. ppRefScriptCostStrideG)
        refScriptCostPerByte
        refScriptsSize
```

The ledger enforces the fee check as `minFee ≤ theFee`:

```haskell
-- eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Rules/Utxo.hs:295-298
failureUnless
  (minFee <= theFee)
  (FeeTooSmallUTxO Mismatch {mismatchSupplied = theFee, mismatchExpected = minFee})
```

Because `minFee` is computed with `floor` for the reference-script portion, the enforced minimum is 1 lovelace lower than the true rational cost whenever `acc + toRational n * curTierPrice` is non-integer.

---

### Impact Explanation

**Medium. Attacker-controlled transactions modify fees outside design parameters.**

The `ppMinFeeRefScriptCostPerByteL` parameter is a `NonNegativeInterval` (a rational), and `n` (the remaining byte count within the final tier) is an integer. Their product is generically non-integer. Whenever it is non-integer, `floor` yields a value 1 lovelace below `ceiling`. The attacker pays this floor-rounded fee; the ledger accepts the transaction. Over many transactions the protocol systematically collects less fee revenue than the governance-approved parameters intend. The discrepancy is bounded at 1 lovelace per transaction but is deterministically exploitable on every transaction whose reference-script size lands in a non-integer position.

---

### Likelihood Explanation

High. Any Conway transaction that includes reference scripts and whose total reference-script byte count produces a non-integer rational fee will trigger the underpayment. With `ppMinFeeRefScriptCostPerByteL` set to a non-integer rational (e.g., the genesis default of `15` lovelace/byte is integer, but governance can update it to any `NonNegativeInterval`), or with any byte count that is not a multiple of the denominator of the price, the fractional part is non-zero. An attacker can trivially choose a reference-script size to guarantee a non-integer result and pay 1 lovelace less than the true cost on every submission.

---

### Recommendation

Replace `floor` with `ceiling` in `tierRefScriptFee` to match the rounding convention established by `txscriptfee` and the Alonzo formal specification:

```diff
-- eras/conway/impl/src/Cardano/Ledger/Conway/Tx.hs
 go !acc !curTierPrice !n
   | n < sizeIncrement =
-      Coin $ floor   (acc + toRational n * curTierPrice)
+      Coin $ ceiling (acc + toRational n * curTierPrice)
```

This ensures the integer minimum fee is always ≥ the true rational cost, consistent with the protocol's design intent and with `txscriptfee`.

---

### Proof of Concept

Let `ppMinFeeRefScriptCostPerByteL = 1 % 3` (1/3 lovelace per byte, a valid `NonNegativeInterval`). Set `ppRefScriptCostStrideG` large enough that the entire reference-script size falls in the first tier (i.e., `n < sizeIncrement`). Choose a reference-script total size of `n = 1` byte.

True rational fee = `0 + (1 % 1) * (1 % 3)` = `1/3` lovelace.

- `ceiling(1/3)` = **1 lovelace** (what `txscriptfee` convention requires)
- `floor(1/3)` = **0 lovelace** (what `tierRefScriptFee` currently computes)

The attacker submits a transaction paying `alonzoMinFeeTx + 0` lovelace. The ledger's `feesOK` check passes because `minFee = alonzoMinFeeTx + 0 ≤ theFee`. The true minimum should have been `alonzoMinFeeTx + 1`. The attacker underpays by 1 lovelace. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
