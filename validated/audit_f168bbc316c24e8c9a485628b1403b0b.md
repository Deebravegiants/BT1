### Title
Manager Can Permanently Freeze User rsETH Withdrawals via Renewable Transfer Block - (File: contracts/RSETH.sol)

### Summary
The `RSETH` contract allows the LRT Manager to block all rsETH transfers to and from any user address for 24 hours via `blockUserTransfers()`. Because this block can be refreshed before expiry, the manager can keep a user's rsETH permanently frozen, preventing them from ever calling `initiateWithdrawal()` or `instantWithdrawal()` in `LRTWithdrawalManager`. This is the direct analog to the MasterChef finding: an admin-controlled access revocation that freezes user funds.

### Finding Description

`RSETH.blockUserTransfers()` is callable by any address holding the `MANAGER` role:

```solidity
function blockUserTransfers(address[] calldata accounts) external onlyLRTManager {
    uint256 blockedUntil = block.timestamp + 1 days;
    ...
    transfersBlockedUntil[account] = blockedUntil;
```

The NatSpec explicitly states: *"Re-applying the block before expiry refreshes the hold to `block.timestamp + 1 days`"*. There is no cap on how many times this can be applied.

The overridden `_transfer` in `RSETH` enforces the block on both sender and receiver:

```solidity
function _transfer(address from, address to, uint256 amount) internal override {
    _enforceNotBlocked(from);
    _enforceNotBlocked(to);
    super._transfer(from, to, amount);
}
```

`LRTWithdrawalManager.initiateWithdrawal()` requires a `safeTransferFrom` of rsETH from the user:

```solidity
IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);
```

This triggers `_transfer` → `_enforceNotBlocked(from)` → `revert TransfersBlocked(account, blockedUntil)`.

`instantWithdrawal()` calls `IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked)`, which also calls `_enforceNotBlocked(account)` before burning.

The user cannot escape by transferring their rsETH to a different address either, since `_transfer` blocks both sender and receiver.

### Impact Explanation

A user holding rsETH is completely unable to:
1. Call `initiateWithdrawal()` to queue a withdrawal for underlying LST/ETH
2. Call `instantWithdrawal()` to immediately redeem rsETH for underlying assets
3. Transfer rsETH to any other address to circumvent the block

By refreshing the block every 24 hours, the manager can make this freeze permanent. The user's entire rsETH position — representing their staked ETH — is rendered illiquid indefinitely. This constitutes a **temporary (extendable to permanent) freezing of funds**.

### Likelihood Explanation

The `MANAGER` role is a privileged but operationally active role in the protocol (it sets whitelists, fees, mint limits, etc.). The `blockUserTransfers` function is explicitly designed for use against users. A malicious or compromised manager can target any non-exempt user. The mechanism requires only repeated calls to a single function with no on-chain resistance.

### Recommendation

1. Enforce a hard cap on the total number of times a user's transfers can be blocked within a rolling window, or
2. Ensure that a blocked user can still call `initiateWithdrawal()` and `instantWithdrawal()` by exempting the `LRTWithdrawalManager` as the `to` address from the block check, or
3. Add a governance-controlled time-lock before the block can be renewed, preventing indefinite extension by a single role.

### Proof of Concept

1. User holds 10 rsETH and calls `LRTWithdrawalManager.initiateWithdrawal(ETH_TOKEN, 10e18, "")`.
2. Manager front-runs (or calls at any time) `RSETH.blockUserTransfers([user])`, setting `transfersBlockedUntil[user] = block.timestamp + 1 days`.
3. `initiateWithdrawal` calls `IERC20(rsETH).safeTransferFrom(user, withdrawalManager, 10e18)`.
4. `RSETH._transfer` calls `_enforceNotBlocked(user)` → reverts with `TransfersBlocked`.
5. User tries `instantWithdrawal` → `burnFrom(user, ...)` → `_enforceNotBlocked(user)` → reverts.
6. User tries to transfer rsETH to a fresh address → `_transfer` → `_enforceNotBlocked(user)` → reverts.
7. Manager calls `blockUserTransfers([user])` again before the 24-hour window expires, refreshing `transfersBlockedUntil[user]` indefinitely.
8. User's rsETH is permanently frozen with no on-chain recourse.

    function blockUserTransfers(address[] calldata accounts) external onlyLRTManager {
        uint256 blockedUntil = block.timestamp + 1 days;
        uint256 length = accounts.length;

        for (uint256 i = 0; i < length; ++i) {
            address account = accounts[i];

            if (isPermanentlyExempt[account] || account == address(0)) continue;

            uint256 prevBlockedUntil = transfersBlockedUntil[account];

            if (blockedUntil != prevBlockedUntil) {
                transfersBlockedUntil[account] = blockedUntil;
                emit UserTransfersBlocked(account, blockedUntil);
            }
        }
    }
```

    function _transfer(address from, address to, uint256 amount) internal override {
        _enforceNotBlocked(from);
        _enforceNotBlocked(to);
        super._transfer(from, to, amount);
    }

    /// @dev Reverts if `account` is currently blocked (used for transfers, mints, and burns)
    function _enforceNotBlocked(address account) internal {
        // Addresses that are permanently exempt can never be blocked
        if (isPermanentlyExempt[account]) return;

        // Check if the account has an active transfer block
        uint256 blockedUntil = transfersBlockedUntil[account];
        if (blockedUntil == 0) return;

        if (block.timestamp < blockedUntil) revert TransfersBlocked(account, blockedUntil);

        // Auto-clean up expired block
        delete transfersBlockedUntil[account];
    }
```
