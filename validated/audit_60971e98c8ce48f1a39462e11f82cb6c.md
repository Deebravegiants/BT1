### Title
User-Controlled `EvmFinality::Latest` Enables Forged Foreign-Chain Verification via Reorg - (File: `crates/foreign-chain-inspector/src/evm/inspector.rs`)

### Summary
The `verify_foreign_transaction` flow accepts a user-controlled `EvmFinality::Latest` parameter that maps directly to `FinalityTag::Latest` (the most recent, easily-reorgable block tip) when querying EVM chains. No minimum finality is enforced at the contract or node layer. An Ethereum validator — or any party capable of causing a short reorg — can submit a deposit transaction, obtain a valid MPC signature while the block is temporarily canonical, claim bridge funds on NEAR, and then reorg the deposit block, achieving a double-spend without any threshold-level collusion inside the MPC network.

### Finding Description

In `verify_finality_level`, the user-supplied finality value is mapped without any floor check:

```rust
let finality_tag = match finality {
    EthereumFinality::Finalized => FinalityTag::Finalized,
    EthereumFinality::Safe     => FinalityTag::Safe,
    EthereumFinality::Latest   => FinalityTag::Latest,   // ← spot tip, no economic finality
};
let head: GetBlockByNumberResponse = self
    .client
    .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
    .await?;
if head.number < receipt_block_number {
    return Err(ForeignChainInspectionError::NotFinalized);
}
``` [1](#0-0) 

`FinalityTag::Latest` is the direct analog of Uniswap V3's `slot0`: the most recent data point, carrying zero economic finality guarantee. The `verify_block_is_canonical` check that follows is a point-in-time snapshot — it confirms the block is canonical *at the moment of the RPC call* but provides no protection against a reorg that occurs after the MPC signature is issued. [2](#0-1) 

The user-supplied `EvmFinality` value flows from the public contract method all the way to the inspector without any minimum-finality guard at either the contract layer or the node layer:

- **Contract layer** (`verify_foreign_transaction`): validates domain purpose, gas, deposit, and chain support — but never inspects the `finality` field. [3](#0-2) 

- **Node layer** (`verify_foreign_tx` provider): passes the finality value directly to `inspector.extract(transaction_id, finality, extractors)` without any floor check. [4](#0-3) 

The `EvmFinality` enum explicitly exposes `Latest` as a valid variant accepted by the public API:

```rust
pub enum EvmFinality {
    Latest,
    Safe,
    Finalized,
}
```

### Impact Explanation

For bridge use cases — the primary motivation for `verify_foreign_transaction` as stated in the design doc — an attacker who can cause a short reorg of the `Latest` block can:

1. Deposit on Ethereum (block N, `Latest`)
2. Obtain an MPC signature attesting to the deposit
3. Claim bridge funds on NEAR using the signature
4. Reorg block N (removing the deposit transaction)
5. Repeat — a classic double-spend

This maps to the allowed impact: **High — forged foreign-chain verification / double-spend conditions**. The MPC network issues a cryptographically valid signature over a transaction that no longer exists on the canonical chain, causing invalid bridge execution.

### Likelihood Explanation

**Medium.** On Ethereum post-merge, reorging the `Latest` block requires being a validator (32 ETH stake) or using MEV techniques (e.g., proposer-builder separation allows sophisticated actors to cause single-slot reorgs). This is far easier than reorging a `Finalized` block (which requires 2/3 of validators to attest and has never happened on mainnet). The attack is realistic for well-resourced bridge attackers, who have strong economic incentive proportional to bridge TVL.

### Recommendation

Enforce a minimum finality level at the contract or node layer. For bridge use cases, reject any `verify_foreign_transaction` request that specifies `EvmFinality::Latest` (and optionally `EvmFinality::Safe` depending on risk tolerance). The contract's `verify_foreign_transaction` handler should validate the `finality` field before enqueuing the request, or the node should refuse to sign responses for `Latest`-finality requests.

### Proof of Concept

1. Attacker is an Ethereum validator (or has access to MEV infrastructure capable of causing a single-slot reorg).
2. Attacker submits a deposit transaction on Ethereum; it lands in block N.
3. Attacker calls `verify_foreign_transaction` with `EvmFinality::Latest` and the deposit's `tx_id`.
4. MPC nodes call `eth_getBlockByNumber(Latest)` → see block N; `verify_finality_level` passes (`N ≤ Latest`).
5. MPC nodes call `eth_getBlockByNumber(N)` → hash matches receipt; `verify_block_is_canonical` passes.
6. MPC threshold signs the payload; signature is returned on-chain.
7. Attacker submits the signature to the NEAR bridge contract and claims funds.
8. Attacker causes a reorg of block N (proposes a competing block without the deposit transaction).
9. The deposit transaction no longer exists on the canonical Ethereum chain.
10. Attacker has claimed NEAR bridge funds without a corresponding on-chain deposit — a double-spend enabled by the same root cause as H-10: using the most recent, easily-manipulable data point (`Latest` block / `slot0`) instead of a committed, economically-final reference (`Finalized` block / TWAP).

    /// Checks that the receipt's block has reached the requested finality level — i.e. that the
    /// head of the chain at `finality` is at or past `receipt_block_number`.
    async fn verify_finality_level(
        &self,
        receipt_block_number: U64,
        finality: EthereumFinality,
    ) -> Result<(), ForeignChainInspectionError> {
        let finality_tag = match finality {
            EthereumFinality::Finalized => FinalityTag::Finalized,
            EthereumFinality::Safe => FinalityTag::Safe,
            EthereumFinality::Latest => FinalityTag::Latest,
        };
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Tag(finality_tag),
            ReturnFullTransactionObjects::from(false),
        );
        let head: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        if head.number < receipt_block_number {
            return Err(ForeignChainInspectionError::NotFinalized);
        }
        Ok(())
    }

    async fn verify_block_is_canonical(
        &self,
        receipt_block_number: U64,
        receipt_block_hash: H256,
    ) -> Result<(), ForeignChainInspectionError> {
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Number(receipt_block_number),
            ReturnFullTransactionObjects::from(false),
        );
        let canonical: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        let hash_matches = canonical.hash == receipt_block_hash;
        let height_matches = canonical.number == receipt_block_number;
        if !hash_matches || !height_matches {
            return Err(ForeignChainInspectionError::NonCanonicalBlock {
                block_number: receipt_block_number.as_u64(),
                receipt_hash: receipt_block_hash.into(),
                canonical_hash: canonical.hash.into(),
            });
        }
        Ok(())
    }

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
