Looking at the full call chain across the Starknet contract, the parser, and the NEAR bridge callback.

**Step 1 — Starknet `log_metadata` is permissionless and accepts any token address** [1](#0-0) 

`log_metadata(token)` has no access control. It calls `name/symbol/decimals` on whatever `token` address is passed, then emits the event with `address: token` as a `#[key]` field (→ `keys[1]`) and `from_address = bridge_contract`.

**Step 2 — Parser separates `token_address` (keys[1]) from `emitter_address` (from_address)** [2](#0-1) 

`token_address = OmniAddress::Strk(H256(keys[1]))` and `emitter_address = OmniAddress::Strk(H256(*from_address))` are two independent fields. There is no assertion that they are equal.

**Step 3 — `deploy_token_callback` validates only `emitter_address`, then registers `token_address`** [3](#0-2) 

The factory check passes because `emitter_address` is the bridge contract (which IS the registered factory). Then `deploy_token_internal` is called with `metadata.token_address` — the value from `keys[1]`, which is the arbitrary address the attacker passed to `log_metadata`.

**Step 4 — The MPC prover passes `from_address` (the real bridge contract) as the emitter** [4](#0-3) 

`starknet_log.from_address` is the contract that emitted the event — always the bridge contract — so the MPC-verified proof correctly carries the bridge address as `from_address`. The attacker does not need to forge anything.

**Conclusion**

The invariant `token_address == emitter_address` (i.e., the registered token must be the bridge itself, or at minimum must have been deployed by the bridge) is never enforced. The `log_metadata` function is documented as intentionally permissionless: [5](#0-4) 

This design is correct for native-token discovery, but the NEAR side's `deploy_token_callback` uses `LogMetadata` proofs to register bridge tokens without verifying that `token_address` was actually deployed by the bridge factory.

### Title
Permissionless `log_metadata` allows arbitrary Starknet contract to be registered as bridge token on NEAR — (`near/omni-bridge/src/lib.rs`, `near/omni-types/src/starknet/events.rs`)

### Summary
An unprivileged attacker can register any Starknet contract address as a bridge token on NEAR by calling the permissionless `log_metadata(fake_token)` on the Starknet bridge, then submitting the resulting proof to NEAR's `deploy_token`. The factory check in `deploy_token_callback` validates only `emitter_address` (the bridge contract, which is the registered factory), but registers `token_address` from `keys[1]` — the attacker-controlled argument.

### Finding Description
`log_metadata` on Starknet is intentionally permissionless and accepts any `ContractAddress`. It emits a `LogMetadata` event where `keys[1] = token` (the caller-supplied address) and `from_address = bridge_contract`. The NEAR parser (`parse_log_metadata`) faithfully separates these into `token_address` and `emitter_address`. `deploy_token_callback` checks only that `emitter_address` matches the registered factory — which always passes because the bridge IS the factory — and then calls `deploy_token_internal(&metadata.token_address, ...)` with the attacker-supplied address. No check enforces that `token_address` was deployed by the bridge or equals the emitter.

### Impact Explanation
An arbitrary Starknet contract (e.g., a fake ERC20 whose `transfer_from` always returns `true`) gets registered as a bridge token on NEAR. A subsequent `init_transfer` on Starknet with that fake token causes the bridge to emit a valid `InitTransfer` event (the fake token's `transfer_from` succeeds without locking real value). The MPC prover verifies the Starknet transaction and NEAR mints the corresponding wrapped tokens — unbacked by any real locked value. This is unauthorized creation of wrapped bridge assets.

### Likelihood Explanation
The attack requires only: (1) deploying a minimal fake ERC20 on Starknet (trivial), (2) calling the public `log_metadata` function (no privilege needed), (3) submitting the MPC proof to NEAR (any relayer can do this). No key compromise, no colluding signers, no chain reorg.

### Recommendation
In `deploy_token_callback`, add a check that `metadata.token_address == metadata.emitter_address` (i.e., the token being registered IS the bridge/factory contract itself, which is the correct semantic for native-token registration), or alternatively enforce that `token_address` is present in the bridge's own deployed-token registry. A cleaner fix is to use a dedicated privileged event (e.g., a new `RegisterNativeToken` event emitted only by an admin function) rather than the permissionless `LogMetadata` event for the token-registration path.

            // 2. New style: name and symbol are ByteArray values (ERC20 ABI).

            let call_data: Array<felt252> = array![];
            let mut res = syscalls::call_contract_syscall(
                token, selector!("name"), call_data.span(),

                .unwrap_syscall();

                );
                utils::felt252_to_string(name)
