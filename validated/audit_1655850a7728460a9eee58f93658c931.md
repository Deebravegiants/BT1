### Title
ETH Deposit Limit Check Omits Deposit Amount, Allowing Any Single Deposit to Bypass the Cap — (File: `contracts/LRTDepositPool.sol`)

---

### Summary

`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric branch for ETH versus ERC20 tokens. The ERC20 branch correctly includes the incoming deposit amount in the comparison, but the ETH branch omits it entirely. Any single ETH deposit can therefore push total protocol ETH deposits arbitrarily above the configured `depositLimitByAsset` cap.

---

### Finding Description

In `_checkIfDepositAmountExceedesCurrentLimit`:

```solidity
// contracts/LRTDepositPool.sol  lines 676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount absent
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct
}
``` [1](#0-0) 

The ETH branch evaluates `totalAssetDeposits > limit`, which is `false` whenever the current total is at or below the limit — regardless of how large `amount` is. The ERC20 branch evaluates `totalAssetDeposits + amount > limit`, which correctly rejects a deposit that would push the total over the cap.

This function is called from `_beforeDeposit`, which is invoked by the public `depositETH` entry point:

```solidity
// contracts/LRTDepositPool.sol  lines 87, 661-663
uint256 rsethAmountToMint = _beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, minRSETHAmountExpected);
...
if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
    revert MaximumDepositLimitReached();
}
``` [2](#0-1) [3](#0-2) 

The inconsistency is also visible in `getAssetCurrentLimit`, which returns `0` when `totalAssetDeposits >= limit` (using `>`), while the deposit guard only blocks when `totalAssetDeposits > limit` (strictly greater), creating a one-deposit window where the view function reports zero capacity but the guard still allows a deposit. [4](#0-3) 

---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

The `depositLimitByAsset` cap is the protocol's primary mechanism for controlling how much ETH is accepted into the restaking system at any given time (e.g., during phased rollouts or risk-management windows). Because the ETH branch ignores `amount`, a single depositor can push total ETH deposits to any value above the cap in one transaction. No funds are stolen and no user loses principal, but the protocol accepts more ETH than its configured risk limit permits, violating the invariant that `getTotalAssetDeposits(ETH) ≤ depositLimitByAsset(ETH)`.

---

### Likelihood Explanation

**High.** The entry path is the standard, permissionless `depositETH()` function. Any depositor who observes that `totalAssetDeposits` is at or near the limit can send a large `msg.value` in a single call. No special role, flash loan, or multi-step setup is required.

---

### Recommendation

Add `amount` to the ETH branch to match the ERC20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

---

### Proof of Concept

1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 1 000 ETH`.
2. Current `getTotalAssetDeposits(ETH_TOKEN) = 999 ETH`.
3. Depositor calls `depositETH{value: 500 ETH}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500 ETH)`:
   - ETH branch: `999 > 1000` → `false` → guard does **not** revert.
5. Deposit succeeds; `getTotalAssetDeposits(ETH_TOKEN)` becomes `1 499 ETH`, 49.9 % above the cap.
6. Identical call with an ERC20 token at the same state: `999 + 500 > 1000` → `true` → `MaximumDepositLimitReached` revert.

The asymmetry is the direct, reachable root cause. A branch-coverage test for `depositETH` at the limit boundary would have exposed this immediately, matching the "untested branch" vulnerability class of the external report.
