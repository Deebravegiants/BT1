### Title
Block-Level Reference Script Size Limit Bypass via Subtransaction Loop — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`, `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Bbody.hs`)

---

### Summary

In the Dijkstra era, the per-transaction reference script size check correctly sums reference scripts across the top-level transaction **and all its subtransactions** via `batchNonDistinctRefScriptsSize`. However, the block-level check (`validateBodyRefScriptsSizeTooBig`) uses `totalRefScriptSizeInBlock`, which calls `txNonDistinctRefScriptsSize` — a function that only counts the **top-level transaction's** reference scripts and is blind to subtransaction reference scripts. An attacker can craft a top-level transaction whose subtransactions collectively carry large reference scripts, causing the actual total reference script processing in a block to exceed `ppMaxRefScriptSizePerBlockG` while the block-level predicate passes.

---

### Finding Description

**The loop analog.** `dijkstraSubLedgersTransition` in `SUBLEDGERS` iterates over a user-controlled list of subtransactions with `foldM`, calling `trans @(EraRule "SUBLEDGER" era)` for each one — a full ledger state transition that includes UTxO witness validation, certificate processing, and Plutus script evaluation. This is the direct structural analog to "calls inside a loop" from the external report: each iteration is an expensive sub-rule invocation driven by attacker-supplied data. [1](#0-0) 

**The per-transaction check (correct).** `dijkstraLedgerTransition` calls `validateAllRefScriptSize`, which delegates to `batchNonDistinctRefScriptsSize`. That function sums the top-level transaction's reference scripts **plus** every subtransaction's reference scripts, and checks the combined total against `ppMaxRefScriptSizePerTxG`. [2](#0-1) [3](#0-2) 

**The block-level check (incomplete).** `validateBodyRefScriptsSizeTooBig` in the Conway/Dijkstra BBODY rule calls `totalRefScriptSizeInBlock`, which iterates over the block's top-level transactions and calls `txNonDistinctRefScriptsSize` for each one. `txNonDistinctRefScriptsSize` only inspects the top-level transaction's `referenceInputsTxBodyL ∪ inputsTxBodyL` — it has no knowledge of subtransactions. [4](#0-3) [5](#0-4) 

**The gap.** The two checks use different accounting functions:

| Check | Function used | Counts subtransaction ref scripts? |
|---|---|---|
| Per-tx (`LEDGER`) | `batchNonDistinctRefScriptsSize` | **Yes** |
| Per-block (`BBODY`) | `txNonDistinctRefScriptsSize` | **No** |

A top-level transaction whose body contains zero reference inputs but whose subtransactions each carry reference scripts up to `ppMaxRefScriptSizePerTxG` will pass the block-level check with a contribution of **0 bytes**, while actually forcing every validating node to deserialize and process up to `ppMaxRefScriptSizePerTxG` bytes of reference scripts per subtransaction.

---

### Impact Explanation

**Medium — Attacker-controlled transactions exceed intended validation limits.**

The reference script size limits were introduced specifically to prevent resource-exhaustion attacks (documented in `docs/adr/2024-08-14_009-refscripts-fee-change.md`): a block containing K top-level transactions, each with subtransactions carrying reference scripts, can force nodes to process up to K × `ppMaxRefScriptSizePerTxG` bytes of reference scripts while the block-level predicate only observes the top-level reference script bytes (potentially 0). With default parameters (`ppMaxRefScriptSizePerTxG` = 200 KiB, `ppMaxRefScriptSizePerBlockG` = 1 MiB), a single block could force ~5× the intended maximum reference script deserialization work per block. [6](#0-5) 

---

### Likelihood Explanation

Any unprivileged transaction author can submit a top-level Dijkstra transaction with subtransactions. The attacker only needs to:
1. Produce UTxO outputs containing large reference scripts (one-time setup).
2. Craft a top-level transaction whose subtransactions reference those outputs as reference inputs.
3. Keep the top-level transaction's own reference inputs empty so the block-level check sees 0 bytes.

No privileged keys, governance majority, or external dependency compromise is required.

---

### Recommendation

Replace `txNonDistinctRefScriptsSize` with `batchNonDistinctRefScriptsSize` (or an equivalent that recurses into subtransactions) inside `totalRefScriptSizeInBlock` when operating in the Dijkstra era (protocol version ≥ 12). Alternatively, add a Dijkstra-specific override of `validateBodyRefScriptsSizeTooBig` in the Dijkstra BBODY rule that uses `batchNonDistinctRefScriptsSize` for each top-level transaction. [7](#0-6) 

---

### Proof of Concept

```
Setup:
  - Produce N UTxO outputs, each carrying a reference script of size S bytes
    (S chosen so that N × S > ppMaxRefScriptSizePerBlockG, e.g. N=6, S=200 KiB)

Craft top-level transaction T:
  - inputsTxBodyL        = { some ordinary UTxO to pay fees }
  - referenceInputsTxBodyL = ∅   ← block-level check sees 0 bytes for T
  - subTransactionsTxBodyL = {
        subTx_1: referenceInputsTxBodyL = { output_1 }  (S bytes)
        subTx_2: referenceInputsTxBodyL = { output_2 }  (S bytes)
        ...
        subTx_N: referenceInputsTxBodyL = { output_N }  (S bytes)
    }

Block-level check (BBODY):
  totalRefScriptSizeInBlock sees txNonDistinctRefScriptsSize(T) = 0
  → predicate passes (0 ≤ ppMaxRefScriptSizePerBlockG)

Per-tx check (LEDGER):
  batchNonDistinctRefScriptsSize(T) = 0 + N×S
  → if N×S ≤ ppMaxRefScriptSizePerTxG the per-tx check also passes

Actual node work:
  Each subtransaction forces deserialization of S bytes of reference scripts.
  Total = N×S >> ppMaxRefScriptSizePerBlockG.
```

The `foldM` loop in `dijkstraSubLedgersTransition` processes each subtransaction sequentially; the block-level guard that was designed to cap this work is bypassed because `totalRefScriptSizeInBlock` never calls `batchNonDistinctRefScriptsSize`. [1](#0-0) [8](#0-7) [4](#0-3)

validateBodyRefScriptsSizeTooBig ::
  forall era.
  ( AlonzoEraTx era
  , BabbageEraTxBody era
  , InjectRuleFailure "BBODY" ConwayBbodyPredFailure era
  , EraBlockBody era
  , ConwayEraPParams era
  ) =>
  PParams era ->
  BlockBody era ->
  UTxO era ->
  Rule (EraRule "BBODY" era) 'Transition ()
validateBodyRefScriptsSizeTooBig pp blockBody utxo =
  let protVer = pp ^. ppProtocolVersionL
      txs = blockBody ^. txSeqBlockBodyL
      totalSize = totalRefScriptSizeInBlock protVer txs utxo
      maxSize = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerBlockG
   in totalSize
        <= maxSize
          ?! injectFailure
            ( BodyRefScriptsSizeTooBig $
                Mismatch
                  { mismatchSupplied = totalSize
                  , mismatchExpected = maxSize
                  }

totalRefScriptSizeInBlock ::
  (AlonzoEraTx era, BabbageEraTxBody era) => ProtVer -> StrictSeq (Tx TopTx era) -> UTxO era -> Int
totalRefScriptSizeInBlock protVer txs (UTxO utxo)
  | pvMajor protVer <= natVersion @10 =
      getSum $ foldMap' (Monoid.Sum . txNonDistinctRefScriptsSize (UTxO utxo)) txs
  | otherwise =
      snd $ F.foldl' accum (utxo, 0) txs
  where
    accum (!accUtxo, !accSum) tx =
      let updatedUtxo = accUtxo `Map.union` unUTxO toAdd
          toAdd
            | IsValid True <- tx ^. isValidTxL = txouts $ tx ^. bodyTxL
            | otherwise = collOuts $ tx ^. bodyTxL
       in (updatedUtxo, accSum + txNonDistinctRefScriptsSize (UTxO accUtxo) tx)

It was identified a while ago that there is an overhead associated with deserializing scripts, so much so that it would be possible to create a very large script that was fairly expensive to deserialize, but very cheap to execute. This opened up an attack vector when such a Plutus script would be used as a reference script. This problem was exacerbated by the fact that there was no real limit on the total size of reference scripts that could be used in a transaction, thus being limited only by the size of the transaction itself. Therefore this opened up Cardano to a DDoS attack where an attacker could submit many such transactions that would cost very little, but would be expensive for a `cardano-node` to validate.

In order to prevent such an attack a `"minFeeRefScriptCostPerByte"` protocol parameter was introduced in the Conway era. The idea was fairly simple: we would calculate the total size of reference scripts used by a transaction and multiply it by the value specified by this protocol parameter. Result would be added to the transaction fee. However, in order for this approach to be a definite deterrent of such attacks this parameter would have to be set to a fairly high value. Community was not very keen on having this value set to a high enough value, since that could make reference scripts almost as expensive to use as regular scripts, thus significantly reducing their usability. This attack forced us to make a quick decision on implementing a definite prevention of such attacks, while trying to not have a drastic impact on the common use case that DApp developers rely on so much.
