### Title
Caller-Agnostic `VerifyForeignTransactionRequest` Map Key Allows Any Unprivileged Caller to Exhaust the Per-Request Fan-Out Queue and Block Legitimate Foreign-Transaction Verifications - (File: `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

The `VerifyForeignTransactionRequest` struct used as the map key for `pending_verify_foreign_tx_requests` contains no caller identity (no predecessor account, no tweak). Any unprivileged account can submit 128 identical `verify_foreign_transaction` calls for the same foreign-chain transaction, saturating the `MAX_PENDING_REQUEST_FAN_OUT = 128` cap for that key. Every subsequent legitimate user who tries to verify the same transaction receives `PendingRequestQueueFull` and is rejected until the queue drains — a condition the attacker can sustain indefinitely at negligible cost.

---

### Finding Description

The contract maintains three separate fan-out maps for pending requests. Each map is keyed by the request struct, and the per-key queue is hard-capped at `MAX_PENDING_REQUEST_FAN_OUT = 128`.

For `sign` and `request_app_private_key`, the stored key includes a **tweak** derived from `hash(predecessor_id, path)`:

```rust
// crates/near-mpc-crypto-types/src/sign.rs
pub struct SignatureRequest {
    pub tweak: Tweak,   // ← hash(predecessor_id, path) — caller-specific
    pub payload: Payload,
    pub domain_id: DomainId,
}
``` [1](#0-0) 

This means two different callers submitting the same payload and path produce **different** map keys, so one caller cannot fill another's queue.

For `verify_foreign_transaction`, the stored key is:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [2](#0-1) 

There is **no predecessor account and no tweak**. The key is entirely determined by the foreign-chain transaction content. The contract's own unit test confirms this explicitly:

```
// "both yields are queued under the single (caller-agnostic) request key"
// "a different account would today be blocked from receiving a response by alice's submission"
``` [3](#0-2) 

The fan-out queue enforces the cap in `push_pending_yield`:

```rust
if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
    env::panic_str(&RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }.to_string());
}
``` [4](#0-3) 

The cap is 128: [5](#0-4) 

An attacker who submits 128 `verify_foreign_transaction` calls for a specific `(ForeignChainRpcRequest, domain_id, payload_version)` fills the queue entirely. Any subsequent caller submitting the same request is rejected with `PendingRequestQueueFull`. The attacker can immediately re-saturate the queue after it drains (either via MPC response or per-yield timeout), sustaining the block indefinitely.

---

### Impact Explanation

**Medium — request-lifecycle manipulation that breaks production safety/accounting invariants.**

A user whose foreign-chain transaction verification is blocked cannot complete any cross-chain operation that depends on the MPC network confirming that transaction. The attacker does not need to steal funds directly; blocking the verification prevents the user from unlocking or acting on assets that are contingent on the verified foreign-chain event. The block can be sustained for as long as the attacker is willing to pay gas, with no upper bound on duration.

---

### Likelihood Explanation

**Medium.** Each blocking cycle requires 128 NEAR transactions, each attaching 1 yoctoNEAR deposit plus ~10 TGas. The deposit cost is negligible (128 yoctoNEAR ≈ $0). The gas cost per cycle is the only real expense. An attacker targeting a high-value cross-chain operation (e.g., a large bridge withdrawal that requires foreign-tx verification) has strong economic incentive to sustain the attack. The target transaction ID is observable on-chain, so the attacker can identify and target specific victims. [6](#0-5) 

---

### Recommendation

Incorporate the caller's identity into the `VerifyForeignTransactionRequest` map key, mirroring the approach used for `SignatureRequest`. Concretely:

1. Derive a tweak from `(predecessor_id, derivation_path)` inside `verify_foreign_transaction` (using the existing `derive_tweak` or a foreign-tx-specific prefix as already documented in `docs/foreign-chain-transactions.md`).
2. Store that tweak in `VerifyForeignTransactionRequest` so the map key is caller-specific.
3. Each caller then has an independent queue slot; one caller cannot exhaust another's queue.

The design document already anticipates this (`FOREIGN_TX_TWEAK_DERIVATION_PREFIX`), but the production struct and contract method do not yet implement it. [7](#0-6) 

---

### Proof of Concept

1. Alice submits `verify_foreign_transaction({ request: Bitcoin(tx_id=[7u8;32], ...), domain_id: 0, payload_version: V1 })` with 1 yoctoNEAR deposit.
2. Attacker observes Alice's pending request on-chain (the `VerifyForeignTransactionRequest` key is fully public).
3. Attacker submits 127 more identical `verify_foreign_transaction` calls from their own account, each with 1 yoctoNEAR deposit. All 128 slots (Alice's + 127 attacker's) are now occupied under the single caller-agnostic key.
4. Bob attempts to submit the same request. The contract calls `push_pending_yield`, finds `queue.len() == 128 >= MAX_PENDING_REQUEST_FAN_OUT`, and panics with `PendingRequestQueueFull { limit: 128 }`. Bob's request is rejected.
5. MPC nodes respond, draining the queue. Attacker immediately submits 128 new requests, re-saturating the queue before Bob can retry.
6. Bob is indefinitely blocked from getting the foreign transaction verified.

The test `verify_foreign_transaction__should_queue_duplicates_from_different_callers` at `crates/contract/src/lib.rs:3209` already demonstrates steps 1–3 as intended behavior, confirming the caller-agnostic key is production code, not a test artifact. [8](#0-7)
