Audit Report

## Title
Multi-Token Accounting Flaw in `maxAmountToDepositBridgerAsset` Enables Bridger Fund Drain and Permanent Collateral Lock — (`contracts/L2/RsETHTokenWrapper.sol`)

## Summary

`maxAmountToDepositBridgerAsset` computes available deposit capacity as `totalSupply() - balanceOf(_asset)`, checking only the queried token's balance against the global supply. When a second allowed token is added via `reinitialize` or `addAllowedToken`, the function returns the full `totalSupply()` as available capacity for the new token, even though the wrapper is already fully collateralized by the original token. The bridger, acting in good faith on this signal, deposits the new token; unprivileged users then burn their wrsETH to withdraw the new token, draining the bridger's deposit and permanently locking the original collateral with no remaining wrsETH to redeem it.

## Finding Description

`maxAmountToDepositBridgerAsset` at lines 99–110 computes:

```solidity
uint256 wrsETHSupply = totalSupply();
uint256 balanceOfAssetInWrapper = ERC20Upgradeable(_asset).balanceOf(address(this));
if (balanceOfAssetInWrapper > wrsETHSupply) return 0;
return wrsETHSupply - balanceOfAssetInWrapper;
``` [1](#0-0) 

The invariant enforced is: `balanceOf(_asset) ≤ totalSupply()`. The correct invariant for a multi-token wrapper is: `Σ balanceOf(t) for all allowed t ≤ totalSupply()`. The function ignores all other allowed tokens' balances entirely.

`reinitialize` (lines 47–49) and `addAllowedToken` (lines 174–176) both call `_addAllowedToken`, which sets `allowedTokens[_asset] = true` without any accounting adjustment. [2](#0-1) [3](#0-2) 

`depositBridgerAssets` (lines 162–170) uses `maxAmountToDepositBridgerAsset` as its sole guard and does not mint wrsETH, so the bridger's deposit of tokenB adds collateral without increasing `totalSupply()`. [4](#0-3) 

`_withdraw` (lines 120–128) allows burning wrsETH to receive **any** allowed token with no constraint on which token the caller originally deposited. [5](#0-4) 

**Exploit path:**

1. Wrapper initialized with tokenA. Users deposit 1000e18 tokenA → `totalSupply() = 1000e18`, `balanceOf(tokenA) = 1000e18`.
2. Admin calls `reinitialize(tokenB)` (intended v2 upgrade). `allowedTokens[tokenB] = true`.
3. `maxAmountToDepositBridgerAsset(tokenB)` returns `1000e18 - 0 = 1000e18`. The wrapper is already fully backed, but the signal is wrong.
4. Bridger calls `depositBridgerAssets(tokenB, 1000e18)` in good faith. Contract now holds 1000e18 tokenA + 1000e18 tokenB; `totalSupply()` remains 1000e18.
5. Any wrsETH holder calls `withdraw(tokenB, 1000e18)`. Burns 1000e18 wrsETH, receives 1000e18 tokenB.
6. Result: `totalSupply() = 0`, contract holds 1000e18 tokenA with no wrsETH left to redeem it, bridger's tokenB is fully drained.

No existing check prevents this. `depositBridgerAssets` only checks `maxAmountToDepositBridgerAsset`, which is the flawed function. `_withdraw` has no cross-token accounting.

## Impact Explanation

Two Critical impacts are concretely realized:

- **Direct theft of bridger's tokenB**: unprivileged wrsETH holders drain the bridger's entire tokenB deposit by withdrawing it instead of their original tokenA.
- **Permanent freezing of tokenA**: once all wrsETH is burned for tokenB, the 1000e18 tokenA held by the contract is irrecoverable — no wrsETH remains to redeem it, and no function allows recovery without minting new wrsETH.

Both match the allowed Critical scope: *Direct theft of any user funds* and *Permanent freezing of funds*.

## Likelihood Explanation

No malicious actor is required among the privileged roles. The admin adding a second collateral token is the explicit purpose of `reinitialize` (the v2 upgrade path). The bridger depositing tokenB after it is added is the expected, designed behavior — `maxAmountToDepositBridgerAsset` is the bridger's authoritative signal for how much to deposit. Both privileged actors behave exactly as intended. The actual exploitation step (withdrawing tokenB) is performed by any unprivileged wrsETH holder acting rationally. No key compromise, governance capture, or external dependency failure is required.

## Recommendation

`maxAmountToDepositBridgerAsset` must sum collateral across all allowed tokens:

```solidity
uint256 totalCollateral = 0;
for each t in allowedTokensSet {
    totalCollateral += ERC20Upgradeable(t).balanceOf(address(this));
}
if (totalCollateral >= wrsETHSupply) return 0;
return wrsETHSupply - totalCollateral;
```

This requires maintaining an enumerable set of allowed tokens (e.g., OpenZeppelin `EnumerableSet.AddressSet`) alongside the existing `allowedTokens` mapping, updated in `_addAllowedToken` and `removeAllowedToken`. Alternatively, restrict the wrapper to a single collateral token and remove the multi-token path entirely.

## Proof of Concept

```solidity
// Foundry unit test — no fork required
function test_multiTokenDrain() public {
    // 1. Setup
    wrapper.initialize(admin, bridger, address(tokenA));

    // 2. User deposits tokenA
    vm.startPrank(user);
    tokenA.approve(address(wrapper), 1000e18);
    wrapper.deposit(address(tokenA), 1000e18);
    vm.stopPrank();
    assertEq(wrapper.totalSupply(), 1000e18);
    assertEq(tokenA.balanceOf(address(wrapper)), 1000e18);

    // 3. Admin adds tokenB via reinitialize
    vm.prank(admin);
    wrapper.reinitialize(address(tokenB));

    // 4. Capacity signal is wrong: returns 1000e18 for tokenB
    assertEq(wrapper.maxAmountToDepositBridgerAsset(address(tokenB)), 1000e18);

    // 5. Bridger deposits tokenB in good faith
    vm.startPrank(bridger);
    tokenB.approve(address(wrapper), 1000e18);
    wrapper.depositBridgerAssets(address(tokenB), 1000e18);
    vm.stopPrank();

    // 6. User withdraws tokenB (not tokenA)
    vm.prank(user);
    wrapper.withdraw(address(tokenB), 1000e18);

    // 7. Bridger's tokenB drained; tokenA permanently locked
    assertEq(tokenB.balanceOf(user), 1000e18);
    assertEq(tokenA.balanceOf(address(wrapper)), 1000e18); // locked forever
    assertEq(wrapper.totalSupply(), 0);
}
```
