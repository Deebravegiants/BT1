Looking at the external bug's invariant: an intermediary validates the **first argument** of a call against `msg.sender`, but approval functions place the *approvee* (not the approver) as the first argument — so any caller can self-authorize. The Sequencer analog is: `is_proposal_init_valid` validates many `ProposalInit` fields against locally-trusted values, but the `builder` field — which becomes the `sequencer_address` in every transaction's execution context and in the block hash — is never checked against any expected value.

---

### Title
Unvalidated `builder` field in `ProposalInit` lets the designated proposer inject an arbitrary `sequencer_address` into block execution context and `PartialBlockHash` — (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_proposal_init_valid` enforces `height`, `l1_da_mode`, `l2_gas_price_fri`, `timestamp`, `starknet_version`, `version_constant_commitment`, all four L1 gas prices, and `fee_proposal_fri`. It never checks `init.builder`. `convert_to_sn_api_block_info` maps `init.builder` directly to `BlockInfo.sequencer_address`, which is (a) returned to every contract via the `get_execution_info` syscall and (b) hashed into `PartialBlockHashComponents.sequencer` and therefore into the `ProposalCommitment` that consensus votes on. A malicious designated proposer can set `builder` to any address; validators accept the proposal, execute the block under the wrong sequencer identity, and commit a wrong block hash.

---

### Finding Description

**Root cause — missing field in `ProposalInitValidation`**

`ProposalInitValidation` carries the locally-trusted reference values the validator uses to check the proposer's `ProposalInit`: [1](#0-0) 

`builder` is absent. `is_proposal_init_valid` therefore never compares `init.builder` against anything: [2](#0-1) 

**Propagation — `builder` becomes `sequencer_address`**

`convert_to_sn_api_block_info` is called for both the proposer path (`initiate_build`) and the validator path (`initiate_validation`): [3](#0-2) 

Line 332 sets `sequencer_address: init.builder`. Whatever the proposer wrote in `builder` is now the sequencer address for the entire block.

**Execution context exposure**

Every transaction's `get_execution_info` syscall returns `block_info.sequencer_address`: [4](#0-3) [5](#0-4) 

**Block hash corruption**

`PartialBlockHashComponents` includes `sequencer: SequencerContractAddress`: [6](#0-5) 

Because both proposer and validator derive `sequencer_address` from the same unchecked `init.builder`, they compute identical (but wrong) `PartialBlockHash` values and agree on the `ProposalCommitment`. Consensus votes on and finalises a block whose hash encodes the attacker-chosen sequencer address.

**Contrast with `proposer` validation**

`proposer` is checked against the committee in `handle_proposal`: [7](#0-6) 

and again in `handle_proposal` inside `SingleHeightConsensus`: [8](#0-7) 

`builder` receives no equivalent check anywhere.

---

### Impact Explanation

1. **Wrong execution state** — every contract that reads `get_execution_info().block_info.sequencer_address` (fee-collection contracts, access-control guards, any contract that routes logic on the sequencer identity) receives the attacker-chosen value. This is a wrong state/revert result from blockifier/syscall execution logic for accepted input.

2. **Wrong block hash / state commitment** — `PartialBlockHashComponents.sequencer` is set to the injected address, so the `PartialBlockHash`, `ProposalCommitment`, and ultimately the on-chain block hash are all wrong. L1 anchoring and proof verification operate over a corrupted commitment.

3. **Fee misdirection** — if the protocol routes sequencer fee receipts to `sequencer_address`, fees for the entire block are sent to the attacker-controlled address.

---

### Likelihood Explanation

The trigger is the designated proposer for any height/round. Any validator in the BFT committee becomes the proposer on a rotating schedule; no additional privilege is required beyond holding a validator key. The attack is silent: validators accept the proposal normally (all checked fields pass), the batcher executes under the injected address, and the commitment comparison at fin succeeds because both sides used the same unchecked `init.builder`.

---

### Recommendation

Add `expected_builder: ContractAddress` to `ProposalInitValidation` (populated from the node's own `static_config.builder_address` or from the committee definition) and add a check inside `is_proposal_init_valid`:

```rust
if init_proposed.builder != proposal_init_validation.expected_builder {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected {:?}, proposed {:?}",
            proposal_init_validation.expected_builder,
            init_proposed.builder
        ),
    ));
}
```

This mirrors the existing pattern used for `height`, `l1_da_mode`, and `l2_gas_price_fri`. [9](#0-8) 

---

### Proof of Concept

```
1. Attacker holds a validator key and is the designated proposer for height H, round R.

2. Attacker constructs ProposalInit {
       height: H, round: R,
       proposer: <attacker's legitimate validator address>,  // passes committee check
       builder: <attacker-controlled address>,              // NOT checked by validator
       ... (all other fields set legitimately) ...
   }

3. Attacker streams ProposalPart::Init(init) to all validators.

4. Each validator calls is_proposal_init_valid():
   - height ✓, l1_da_mode ✓, l2_gas_price_fri ✓, timestamp ✓,
     starknet_version ✓, version_constant_commitment ✓,
     L1 gas prices ✓, fee_proposal_fri ✓
   - builder: NOT CHECKED → passes

5. Each validator calls initiate_validation() → convert_to_sn_api_block_info(init):
       sequencer_address = init.builder  // = attacker-controlled address

6. Batcher executes all transactions with sequencer_address = attacker address.
   - Every get_execution_info() syscall returns attacker address as sequencer.
   - PartialBlockHashComponents.sequencer = attacker address.

7. Validator computes batcher_block_commitment from the wrong PartialBlockHash.
   Proposer's ProposalFin carries the same wrong commitment (computed identically).
   built_block == received_fin.proposal_commitment → proposal accepted.

8. Consensus reaches decision on a block whose hash encodes the attacker's address
   as sequencer. Block is committed, state is updated, L1 anchor is wrong.
```

#[instrument(level = "warn", skip_all, fields(?proposal_init_validation, ?init_proposed))]
async fn is_proposal_init_valid(
    proposal_init_validation: &ProposalInitValidation,
    init_proposed: &ProposalInit,
    clock: &dyn Clock,
    l1_gas_price_provider: Arc<dyn L1GasPriceProviderClient>,
    gas_price_params: &GasPriceParams,
) -> ValidateProposalResult<()> {
    let now: u64 = clock.unix_now();
    let last_block_timestamp =
        proposal_init_validation.previous_proposal_init.as_ref().map_or(0, |info| info.timestamp);
    if init_proposed.timestamp < last_block_timestamp {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "Timestamp is too old: last_block_timestamp={}, proposed={}",
                last_block_timestamp, init_proposed.timestamp
            ),
        ));
    }
    if init_proposed.timestamp > now + proposal_init_validation.block_timestamp_window_seconds {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "Timestamp is in the future: now={}, block_timestamp_window_seconds={}, \
                 proposed={}",
                now,
                proposal_init_validation.block_timestamp_window_seconds,
                init_proposed.timestamp
            ),
        ));
    }
    if init_proposed.starknet_version != proposal_init_validation.starknet_version {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "starknet_version mismatch: expected={:?}, proposed={:?}",
                proposal_init_validation.starknet_version, init_proposed.starknet_version
            ),
        ));
    }
    // `version_constant_commitment` is proposer-supplied (network-derived). It is not yet a real
    // commitment (see `expected_version_constant_commitment`): the only valid value is the
    // sentinel, so reject anything else. Enforcing the same value the proposer emits keeps the two
    // sides in lockstep, so a real value cannot ship on one side without the other.
    let expected_commitment = expected_version_constant_commitment();
    if init_proposed.version_constant_commitment != expected_commitment {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "version_constant_commitment mismatch: expected={expected_commitment:?}, \
                 proposed={:?}",
                init_proposed.version_constant_commitment
            ),
        ));
    }
    if !(init_proposed.height == proposal_init_validation.height
        && init_proposed.l1_da_mode == proposal_init_validation.l1_da_mode
        && init_proposed.l2_gas_price_fri == proposal_init_validation.l2_gas_price_fri)
    {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            "ProposalInit validation failed".to_string(),
        ));
    }
