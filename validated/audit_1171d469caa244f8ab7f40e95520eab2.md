Audit Report

## Title
Missing Minimum Withdrawal Amount Allows Dust Request Queue Flooding, Temporarily Freezing Legitimate Withdrawals - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`minRsEthAmountToWithdraw` is never initialized in `initialize()` and defaults to `0` for every asset. The guard in `initiateWithdrawal` collapses to `rsETHUnstaked == 0` when the mapping is zero, allowing any amount ≥ 1 wei to pass. An unprivileged rsETH holder can flood the sequential withdrawal queue with dust entries, forcing the operator to process all of them before any later legitimate request can be unlocked, temporarily freezing those users' funds.

## Finding Description
`minRsEthAmountToWithdraw` is declared as a mapping at line 35 and is never assigned in `initialize()` (lines 90–98), so it defaults to `0` for every asset. The guard at lines 162–164:

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```

reduces to `rsETHUnstaked == 0` when the mapping value is zero, meaning any amount ≥ 1 wei is accepted. Each accepted call transfers rsETH from the caller (line 166) and appends a new nonce to the global queue via `_addUserWithdrawalRequest` (lines 755–757). There is no per-user pending-request cap anywhere in the contract.

The operator's `unlockQueue` calls `_unlockWithdrawalRequests` (lines 790–815), which iterates strictly from `nextLockedNonce[asset]` upward, one entry at a time, and cannot skip entries. Every dust entry must be individually processed — reading two storage slots, computing a payout, and writing back several storage values — before `nextLockedNonce` advances past it. A legitimate request at nonce N+1 cannot be unlocked until all N preceding dust entries are processed. With a sufficiently large dust flood, a single `unlockQueue` call exceeds the block gas limit, and the operator must issue many transactions to drain the queue, during which the legitimate user's rsETH remains locked in the contract and `completeWithdrawal` reverts with `WithdrawalLocked`.

The admin setter `setMinRsEthAmountToWithdraw` (lines 330–333) exists but is not called during deployment, providing no protection unless proactively invoked before the first withdrawal.

## Impact Explanation
Once a user calls `initiateWithdrawal`, their rsETH is transferred to the contract (line 166) and they cannot recover it except through `completeWithdrawal`, which requires their nonce to be below `nextLockedNonce[asset]`. Because `nextLockedNonce` advances only through `unlockQueue`, and `unlockQueue` must iterate through every preceding dust entry in order, the legitimate user's funds are inaccessible for the duration of the drain operation. This constitutes **Medium — Temporary freezing of funds**.

## Likelihood Explanation
The attacker requires only rsETH (obtainable by depositing ETH into `LRTDepositPool`) and the ability to call `initiateWithdrawal` repeatedly. No privileged role is needed. The cost per dust entry is one transaction plus 1 wei of rsETH. On any chain where gas is inexpensive, flooding thousands of entries is realistic. Every fresh deployment is vulnerable by default until an admin explicitly calls `setMinRsEthAmountToWithdraw`.

## Recommendation
1. In `initialize()`, set a non-zero default for `minRsEthAmountToWithdraw` for every supported asset (e.g., `1e15` wei rsETH, ~0.001 rsETH).
2. Alternatively, enforce a protocol-wide floor directly in `initiateWithdrawal` independent of the per-asset mapping, so the contract is safe even before admin configuration.
3. Add a per-user pending-request cap to bound queue growth regardless of amount.

## Proof of Concept
1. Deploy `LRTWithdrawalManager` without calling `setMinRsEthAmountToWithdraw`; confirm `minRsEthAmountToWithdraw[stETH] == 0`.
2. Attacker acquires a small amount of rsETH (e.g., 10 000 wei) via `LRTDepositPool`.
3. Attacker calls `initiateWithdrawal(stETH, 1, "")` 10 000 times; each call transfers 1 wei rsETH and pushes nonces 0–9 999.
4. Legitimate user calls `initiateWithdrawal(stETH, 1e18, "")`, landing at nonce 10 000.
5. Operator calls `unlockQueue(stETH, 10001, ...)`. The while-loop at lines 790–815 must iterate all 10 001 entries; at sufficient scale this exceeds the block gas limit.
6. Legitimate user calls `completeWithdrawal(stETH, "")` — reverts with `WithdrawalLocked` because `nextLockedNonce[stETH]` has not yet reached nonce 10 000.
7. Foundry fuzz test: fuzz the number of dust entries N, assert that after N dust entries the legitimate user's `completeWithdrawal` reverts until the operator issues ⌈N/batch_size⌉ `unlockQueue` transactions.
