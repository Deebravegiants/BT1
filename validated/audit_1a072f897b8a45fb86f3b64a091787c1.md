### Title
Updating `LRT_UNSTAKING_VAULT` via `setContract` Strands Funds and Freezes User Withdrawals - (File: contracts/LRTConfig.sol)

### Summary
`LRTConfig._setContract()` updates the `LRT_UNSTAKING_VAULT` address with no check on the old vault's balance. When the admin points the protocol to a new vault, all ETH/LSTs held in the old vault are stranded there while `LRTWithdrawalManager` immediately begins querying the new (empty) vault. Every pending withdrawal unlock and every instant withdrawal reverts, temporarily freezing user funds.

### Finding Description
`LRTConfig.setContract()` delegates to `_setContract()`, which performs only a zero-address check and a duplicate-value guard before overwriting `contractMap[key]`:

```solidity
function _setContract(bytes32 key, address val) private {
    UtilLib.checkNonZeroAddress(val);
    if (contractMap[key] == val) {
        revert ValueAlreadyInUse();
    }
    contractMap[key] = val;          // no balance check on old contract
    emit SetContract(key, val);
}
``` [1](#0-0) 

Contrast this with `updateAssetStrategy`, which explicitly iterates every NDC and reverts if the old strategy still holds funds:

```solidity
if (assetStrategy[asset] != address(0)) {
    ...
    uint256 ndcBalance = IStrategy(assetStrategy[asset]).userUnderlyingView(ndcs[i]);
    if (ndcBalance > 0) {
        revert CannotUpdateStrategyAsItHasFundsNDCFunds(ndcs[i], ndcBalance);
    }
``` [2](#0-1) 

No equivalent guard exists for `LRT_UNSTAKING_VAULT`.

`LRTWithdrawalManager.unlockQueue()` resolves the vault at call time from `lrtConfig`:

```solidity
ILRTUnstakingVault unstakingVault =
    ILRTUnstakingVault(lrtConfig.getContract(LRTConstants.LRT_UNSTAKING_VAULT));
UnlockParams memory params = _createUnlockParams(lrtOracle, unstakingVault, asset);
...
if (params.totalAvailableAssets == 0) revert AmountMustBeGreaterThanZero();
``` [3](#0-2) 

`_createUnlockParams` reads `unstakingVault.balanceOf(asset)`:

```solidity
return UnlockParams({
    rsETHPrice: lrtOracle.rsETHPrice(),
    assetPrice: lrtOracle.getAssetPrice(asset),
    totalAvailableAssets: unstakingVault.balanceOf(asset)
});
``` [4](#0-3) 

After the vault address is swapped, the new vault has zero balance, so `totalAvailableAssets == 0` and `unlockQueue` always reverts. The subsequent `unstakingVault.redeem()` call that would transfer assets to the withdrawal manager is never reached:

```solidity
unstakingVault.redeem(asset, assetAmountUnlocked);
``` [5](#0-4) 

`instantWithdrawal()` is equally broken: it calls `unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)` on the new vault, which returns 0, causing an immediate revert:

```solidity
if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
    revert CantInstantWithdrawMoreThanAvailable();
}
unstakingVault.redeem(asset, assetAmountUnlocked);
``` [6](#0-5) 

Funds that `NodeDelegator.completeUnstaking()` previously forwarded to the old vault via `receiveFromNodeDelegator()` are now inaccessible to the withdrawal pipeline:

```solidity
_getUnstakingVault().receiveFromNodeDelegator{ value: address(this).balance - balancesBefore[i] }();
``` [7](#0-6) 

### Impact Explanation
All users who have queued withdrawal requests via `initiateWithdrawal()` are unable to have those requests unlocked (`unlockQueue` reverts) and therefore cannot call `completeWithdrawal`. Instant withdrawals also revert. ETH and LSTs sitting in the old vault are stranded until the admin manually migrates them (e.g., reverting the address change, or routing funds through `transferAssetToNodeDelegator` back into the system). This is a **temporary freezing of user funds** (Medium impact).

Additionally, `LRTDepositPool.getTotalAssetDeposits()` counts `assetLyingUnstakingVault` from the new (empty) vault, understating total protocol TVL and distorting the rsETH price used in withdrawal calculations. [8](#0-7) 

### Likelihood Explanation
The admin must call `setContract(LRT_UNSTAKING_VAULT, newVault)` — a legitimate operational action during a vault upgrade or migration. There is no malicious intent required. The protocol already demonstrates awareness of this class of risk for strategies (`updateAssetStrategy` reverts if funds remain), but the same protection is absent for the vault. Likelihood is low but realistic during any vault upgrade.

### Recommendation
Add a balance check before overwriting `LRT_UNSTAKING_VAULT`, mirroring the pattern used in `updateAssetStrategy`:

```solidity
if (key == LRTConstants.LRT_UNSTAKING_VAULT) {
    address oldVault = contractMap[key];
    if (oldVault != address(0)) {
        // Check ETH and all supported LSTs
        require(oldVault.balance == 0, "OldVaultHasETH");
        address[] memory assets = getSupportedAssetList();
        for (uint256 i; i < assets.length; ++i) {
            if (assets[i] == ETH_TOKEN) continue;
            require(IERC20(assets[i]).balanceOf(oldVault) == 0, "OldVaultHasAssets");
        }
    }
}
```

Alternatively, implement an atomic migration function that transfers all balances from the old vault to the new one before updating the pointer.

### Proof of Concept
1. Users call `LRTWithdrawalManager.initiateWithdrawal()` — rsETH is locked in the withdrawal manager; ETH/LSTs flow through `NodeDelegator.completeUnstaking()` into `LRTUnstakingVault` (old vault).
2. Admin calls `LRTConfig.setContract(LRT_UNSTAKING_VAULT, newVault)` — no balance check, succeeds immediately.
3. `LRTWithdrawalManager.unlockQueue(asset, ...)` is called:
   - Resolves `unstakingVault` = `newVault` (balance = 0).
   - `_createUnlockParams` returns `totalAvailableAssets = 0`.
   - Reverts: `AmountMustBeGreaterThanZero()`.
4. `LRTWithdrawalManager.instantWithdrawal(asset, ...)` is called:
   - `newVault.getAssetsAvailableForInstantWithdrawal(asset)` = 0.
   - Reverts: `CantInstantWithdrawMoreThanAvailable()`.
5. All user withdrawal requests are frozen. Funds remain in the old vault with no automated path to recover them into the new withdrawal pipeline.
