### Title
`LRTDepositPool::depositETH`/`depositAsset` uses stale `rsETHPrice`, allowing depositors to mint excess rsETH at existing holders' expense - (File: contracts/LRTDepositPool.sol)

---

### Summary

`LRTDepositPool.depositETH()` and `depositAsset()` compute the rsETH mint amount using the stored `rsETHPrice` from `LRTOracle`, which is only updated when `updateRSETHPrice()` is explicitly called. Neither deposit function triggers a price update before computing the mint amount. When rewards have accrued since the last price update, the stored price is stale (lower than the true price), causing depositors to receive more rsETH than they are entitled to, diluting existing holders' unclaimed yield.

---

### Finding Description

`LRTDepositPool.getRsETHAmountToMint()` computes the rsETH mint amount as:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [1](#0-0) 

`lrtOracle.rsETHPrice()` is a stored state variable in `LRTOracle`: [2](#0-1) 

It is only updated when `updateRSETHPrice()` or `updateRSETHPriceAsManager()` is explicitly called: [3](#0-2) 

Neither `depositETH` nor `depositAsset` calls `updateRSETHPrice()` before invoking `_beforeDeposit` → `getRsETHAmountToMint`: [4](#0-3) [5](#0-4) 

The `_updateRsETHPrice()` internal function computes the new price from current TVL and mints protocol fees. Between calls, `rsETHPrice` is frozen at its last recorded value: [6](#0-5) 

As restaking rewards accrue (TVL increases), the true rsETH/ETH rate rises above the stored `rsETHPrice`. Any deposit made before the next `updateRSETHPrice()` call uses the lower stale price, minting more rsETH than the deposited ETH value warrants. The excess rsETH represents yield that belongs to existing holders.

Additionally, `updateRSETHPrice()` has a `pricePercentageLimit` guard that reverts for non-managers if the price increase exceeds the configured threshold: [7](#0-6) 

This means that after a large reward accrual, only a manager can update the price, extending the window during which the stale price is exploitable by any depositor.

---

### Impact Explanation

**High. Theft of unclaimed yield.**

When `rsETHPrice` is stale (lower than the true rate), a depositor receives:

```
rsETH_minted = deposit_ETH / stale_rsETHPrice   >   deposit_ETH / true_rsETHPrice
```

The excess rsETH is backed by yield that already belongs to existing holders. When the price is eventually updated, the attacker's rsETH is worth more than their deposit, at the direct expense of existing holders whose proportional share of TVL is diluted.

---

### Likelihood Explanation

**Medium.** The price update is a separate off-chain-triggered operation with no on-chain enforcement before deposits. Any depositor can observe the last `RsETHPriceUpdate` event, compute the accrued-but-unrecorded yield, and time a deposit to exploit the stale price. The `pricePercentageLimit` guard can extend the exploitable window further by blocking public price updates after large reward accruals.

---

### Recommendation

Call `_updateRsETHPrice()` (or expose an internal equivalent) at the start of `depositETH` and `depositAsset` before computing the mint amount, ensuring the price reflects current TVL before any new shares are issued. This mirrors the fix applied in the Licredity report: accrue state at the start of the function that modifies shares.

---

### Proof of Concept

1. Protocol state: 100 ETH TVL, 100 rsETH supply, `rsETHPrice = 1.00 ETH` (last updated).
2. Staking rewards accrue: TVL rises to 105 ETH. `updateRSETHPrice()` has not been called.
3. Attacker calls `depositETH{value: 1 ether}(0, "")`.
4. `getRsETHAmountToMint` computes: `1e18 * 1e18 / 1.00e18 = 1.0 rsETH` minted.
5. True price would give: `1e18 * 1e18 / 1.05e18 ≈ 0.952 rsETH`.
6. Attacker receives `~0.048 rsETH` excess, representing ~5% of the accrued yield.
7. When `updateRSETHPrice()` is later called, new price = `106 ETH / 101 rsETH ≈ 1.0495 ETH`. Attacker's 1.0 rsETH is worth `1.0495 ETH` on a `1 ETH` deposit.
8. Existing holders' yield is permanently diluted by the excess rsETH issued.
