### Title
Strict Equality Check in `validateTreasuryValue` Enables Governance Proposal DoS via Treasury Donations — (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs`)

---

### Summary

The `validateTreasuryValue` function in the Conway LEDGER rule enforces a strict equality (`==`) between the optional `currentTreasuryValue` field in a transaction body and the actual on-chain treasury balance. Because any unprivileged transaction can include a `treasuryDonation` (field 22 in the Conway transaction body), an attacker can front-run any governance-proposal transaction that carries `currentTreasuryValue` by donating even 1 lovelace to the treasury, permanently invalidating the victim's transaction until they abandon the safety field.

---

### Finding Description

In `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs`, the `conwayLedgerTransitionTRC` function calls:

```haskell
runTest $ validateTreasuryValue txBody (chainAccountState ^. casTreasuryL)
``` [1](#0-0) 

The validation function is:

```haskell
validateTreasuryValue txBody actualTreasuryValue =
  case txBody ^. currentTreasuryValueTxBodyL of
    SNothing -> pure ()
    SJust submittedTreasuryValue ->
      failureUnless (submittedTreasuryValue == actualTreasuryValue) $
        ConwayTreasuryValueMismatch ...
``` [2](#0-1) 

The `actualTreasuryValue` is the live treasury balance at the moment the transaction is applied. The treasury balance is increased by any transaction that includes a `treasuryDonation` (CDDL field `? 22 : positive_coin`): [3](#0-2) 

There is no minimum donation size beyond `positive_coin` (> 0). The check uses `==` rather than `<=`, so any increase — even 1 lovelace — causes the predicate to fail with `ConwayTreasuryValueMismatch`: [4](#0-3) 

The `currentTreasuryValue` field was introduced specifically as a safety mechanism for governance proposals (e.g., `TreasuryWithdrawals`) so that a proposer can assert the treasury state their proposal is based on: [5](#0-4) 

---

### Impact Explanation

An attacker can permanently prevent any transaction that includes `currentTreasuryValue` from being applied. Because the field is used by governance-proposal transactions to assert the treasury state, the attacker can:

1. Observe a pending governance proposal transaction carrying `currentTreasuryValue = T` in the mempool.
2. Submit a transaction with `treasuryDonation = 1` (1 lovelace), making the on-chain treasury `T + 1`.
3. The victim's transaction is rejected with `ConwayTreasuryValueMismatch`.
4. The victim updates to `currentTreasuryValue = T + 1` and rebroadcasts; the attacker front-runs again.

This matches the **Medium** allowed impact: attacker-controlled transactions modify treasury donations outside design parameters, and the strict equality gate — intended as a safety check — is weaponised to block governance proposals indefinitely. While the field is optional (omitting it bypasses the check), wallet software and tooling that always includes it for safety is rendered unable to submit governance proposals as long as the attacker persists.

---

### Likelihood Explanation

The attack requires only:
- Mempool visibility (standard for any full node).
- The ability to submit a transaction with `treasuryDonation ≥ 1` lovelace — no special role, key, or stake is needed.
- Repeated front-running to sustain the DoS.

The cost per front-run is 1 lovelace plus the transaction fee, making sustained attacks economically trivial.

---

### Recommendation

Replace the strict equality with a `<=` check, mirroring the pattern already used for withdrawal validation in the Dijkstra era (`withdrawalsThatExceedAccountBalance`):

```haskell
-- Current (vulnerable):
failureUnless (submittedTreasuryValue == actualTreasuryValue) $

-- Fixed:
failureUnless (submittedTreasuryValue <= actualTreasuryValue) $
``` [6](#0-5) 

This preserves the safety intent (the proposer cannot assert a treasury value *larger* than reality, which would allow a `TreasuryWithdrawals` proposal to appear to leave more in the treasury than actually exists) while tolerating any increase caused by concurrent donations.

---

### Proof of Concept

```
Block N-1 state: treasury = 1_000_000 ADA

Victim builds Tx_V:
  currentTreasuryValue = 1_000_000 ADA
  proposalProcedures   = [TreasuryWithdrawals ...]

Attacker observes Tx_V in mempool, submits Tx_A:
  treasuryDonation = 1   -- 1 lovelace

Block N includes Tx_A first:
  treasury = 1_000_000 ADA + 1 lovelace

Block N then attempts Tx_V:
  validateTreasuryValue:
    submittedTreasuryValue (1_000_000 ADA)
      == actualTreasuryValue (1_000_000 ADA + 1 lovelace)
    → FALSE → ConwayTreasuryValueMismatch → Tx_V rejected

Victim retries with updated value; attacker front-runs again.
Governance proposal is permanently blocked.
```
