### Title
Harvester `farmer_reward_address_override` in `RespondSignatures` Unconditionally Redirects Farmer Block Rewards Without Enforcement — (`chia/farmer/farmer_api.py`)

### Summary

A connected harvester (including third-party harvesters explicitly supported by CHIP-22) can set `farmer_reward_address_override` to an arbitrary puzzle hash in its `RespondSignatures` message. The farmer unconditionally substitutes this address for its configured `farmer_target` in the `DeclareProofOfSpace` it broadcasts to the full node, permanently redirecting the 0.25 XCH farmer block reward to the attacker's address. The only guard — `notify_farmer_reward_taken_by_harvester_as_fee()` — is advisory logging only and is not even invoked on the `RespondSignatures` path, making it trivially bypassable.

### Finding Description

**Protocol field:**

`RespondSignatures.farmer_reward_address_override: bytes32 | None` [1](#0-0) 

**Unconditional substitution in `_process_respond_signatures()`:**

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
    include_source_signature_data = True
``` [2](#0-1) 

The resulting `farmer_reward_address` is placed directly into `DeclareProofOfSpace` and broadcast to the full node: [3](#0-2) 

**Advisory-only guard (not invoked on the `RespondSignatures` path):**

`notify_farmer_reward_taken_by_harvester_as_fee()` is called only when processing `NewProofOfSpace`: [4](#0-3) 

That function only emits log warnings regardless of whether the fee threshold is valid or invalid — it never rejects the override or halts processing: [5](#0-4) 

**Attack path bypassing even the advisory log:**

A harvester can send `NewProofOfSpace` with `farmer_reward_address_override = None` (no log triggered), wait for the farmer's `RequestSignatures`, then return `RespondSignatures` with `farmer_reward_address_override` set to the attacker's puzzle hash. The farmer processes this in `_process_respond_signatures()` with zero validation and no logging, and broadcasts `DeclareProofOfSpace` with the attacker's address as the farmer reward destination.

### Impact Explanation

Every block won by the compromised harvester redirects the 0.25 XCH farmer coinbase reward to an attacker-controlled address. The farmer operator's configured `xch_target_address` / `farmer_target` is silently bypassed. This is a direct, on-chain, irreversible transfer of XCH away from the legitimate farmer. The impact matches the allowed Critical/High scope: **unauthorized reward diversion affecting XCH**. [6](#0-5) 

### Likelihood Explanation

CHIP-22 explicitly enables third-party harvesters to connect to a farmer using the private CA certificate. A farmer operator who sets up a third-party harvester (e.g., DrPlotter or any GPU harvester) grants that harvester the ability to send `RespondSignatures`. The operator expects the CHIP-22 fee convention to be enforced, but it is not — the harvester can redirect 100% of farmer rewards, not just the agreed fee percentage. No cryptographic break or key theft is required; only a valid private-CA-signed TLS certificate (which the farmer operator issues to the harvester) is needed. [7](#0-6) 

### Recommendation

1. **Enforce the fee convention, not just log it.** Before accepting `farmer_reward_address_override` from either `NewProofOfSpace` or `RespondSignatures`, compute `calculate_harvester_fee_quality` and reject (return `None` / drop the proof) if `fee_quality > applied_fee_threshold` or if `fee_info` is absent.
2. **Validate consistency.** If `farmer_reward_address_override` appears in `RespondSignatures`, verify it matches the value declared in the corresponding `NewProofOfSpace`. A mismatch should be treated as a protocol violation and the harvester connection dropped.
3. **Apply the check on both message paths.** Currently `notify_farmer_reward_taken_by_harvester_as_fee()` is only called from `new_proof_of_space()`; the `_process_respond_signatures()` path has no guard at all.

### Proof of Concept

```
1. Attacker operates a harvester with a valid private-CA cert (standard CHIP-22 setup).
2. Harvester wins a block and sends NewProofOfSpace to the farmer with
   farmer_reward_address_override = None  (no advisory log triggered).
3. Farmer sends RequestSignatures back to the harvester.
4. Harvester returns RespondSignatures with
   farmer_reward_address_override = <attacker_puzzle_hash>.
5. Farmer executes _process_respond_signatures():
     farmer_reward_address = self.farmer.farmer_target   # legitimate address
     if response.farmer_reward_address_override is not None:
         farmer_reward_address = response.farmer_reward_address_override  # attacker's address
6. Farmer broadcasts DeclareProofOfSpace with farmer_reward_address = attacker_puzzle_hash.
7. Full node includes the block; 0.25 XCH farmer reward is paid to attacker_puzzle_hash.
8. Farmer operator receives nothing. No error is logged. No rejection occurs.
``` [8](#0-7) [9](#0-8)

    @metadata.request()
    async def respond_signatures(self, response: harvester_protocol.RespondSignatures) -> None:
        request = self._process_respond_signatures(response)
        if request is None:
            return None

        message: Message | None = None
        if isinstance(request, DeclareProofOfSpace):
            self.farmer.state_changed("proof", {"proof": request, "passed_filter": True})
            message = make_msg(ProtocolMessageTypes.declare_proof_of_space, request)
        if isinstance(request, SignedValues):
            message = make_msg(ProtocolMessageTypes.signed_values, request)
        await self.farmer.server.send_to_all([message], NodeType.FULL_NODE)
