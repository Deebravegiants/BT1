### Title
Negative `required` in `MultiSigMOf` bypasses all witness checks, making any UTxO locked by such a script spendable by anyone — (`eras/shelley/impl/src/Cardano/Ledger/Shelley/Scripts.hs`)

### Summary

`MultiSigMOf` stores its threshold as `!Int`, not `Natural`. Neither the JSON parser (`shelleyBasedEraNativeScriptJSONParser`) nor the CBOR decoder validates that the threshold is non-negative. When `m < 0`, `isValidMOf` short-circuits to `True` immediately, making the script unconditionally valid regardless of the witness set.

### Finding Description

**1. JSON parser — no lower-bound guard**

`shelleyBasedEraNativeScriptJSONParser` parses `"required"` directly as `Int` via Aeson's `FromJSON Int`, which accepts any JSON integer including negatives:

```haskell
"MOf" ->
  mkRequireMOf @era
    <$> o .: "required"          -- Int, no validation
    <*> (StrictSeq.fromList <$> o .: "scripts")
``` [1](#0-0) 

**2. CBOR decoder — same gap**

The CBOR decoder for `MultiSigRaw` also decodes `m` as bare `Int` with no non-negativity check:

```haskell
3 -> do
  m <- decCBOR          -- Int; CBOR major-type-1 negative integers accepted
  multiSigs <- sequence <$> decCBOR
  pure (3, MultiSigMOf m <$> multiSigs)
``` [2](#0-1) 

**3. `isValidMOf` — always `True` for any `n ≤ 0`**

```haskell
isValidMOf n StrictSeq.Empty = n <= 0
isValidMOf n (msig StrictSeq.:<| msigs) =
  n <= 0 || if go msig then isValidMOf (n - 1) msigs else isValidMOf n msigs
``` [3](#0-2) 

With `n = -1`, the first branch of every recursive call is `(-1) <= 0 = True`, so the function returns `True` immediately regardless of the witness set or the sub-script list.

**4. `validateMultiSig` — the only gate**

```haskell
validateMultiSig tx =
  evalMultiSig $ Set.map witVKeyHash (tx ^. witsTxL . addrTxWitsL)
``` [4](#0-3) 

With an empty `addrTxWitsL`, `evalMultiSig Set.empty (RequireMOf (-1) _)` calls `isValidMOf (-1) _` = `True`. The UTXOW rule's `validateFailedNativeScripts` therefore passes. [5](#0-4) 

**5. Formal spec already flags this**

The formal spec defines `RequireMOf m ts` with `m ∈ ℕ` (naturals), and the embedded Plutus reference code even carries the comment `"n.b., should also check that this is >= 0"`. The implementation uses `Int` instead of `Natural`, violating the spec's type invariant. [6](#0-5) 

### Impact Explanation

Any UTxO whose payment credential is the hash of a `MultiSigMOf (-1) scripts` script can be spent by submitting a transaction with the script in the witness set and **zero key witnesses**. The ledger accepts the spend as valid. This constitutes an invalid ledger state transition that directly destroys the owner's ADA.

The same script hash can be used as a staking credential, so rewards accumulated under such a credential are equally unprotected.

### Likelihood Explanation

The attacker cannot retroactively affect existing, legitimately-constructed scripts (different hash). Exploitation requires that ADA be locked to a negative-threshold script. This can happen via:

- Social engineering: attacker distributes a "multisig template" JSON file with `"required": -1` to a victim who locks funds to the resulting address.
- Tooling bug: any tool that constructs `MOf` scripts from user-supplied integers without validation could produce such a script.
- Direct CBOR crafting: the attacker can also craft the script directly in CBOR without going through the JSON parser.

Likelihood is low for accidental occurrence but non-zero for targeted social engineering.

### Recommendation

1. Change `MultiSigMOf !Int` to `MultiSigMOf !Natural` (or `!Word`) in `MultiSigRaw`, propagating the type change through `mkRequireMOf`, `getRequireMOf`, `ShelleyEraScript`, and all era instances.
2. If keeping `Int` for compatibility, add an explicit guard in `shelleyBasedEraNativeScriptJSONParser` and in the CBOR decoder: reject any `m < 0` with a parse/decode failure.
3. Add the same guard in `evalMultiSig`/`isValidMOf` as a defence-in-depth assertion.
