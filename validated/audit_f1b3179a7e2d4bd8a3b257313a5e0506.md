### Title
Byzantine Party Equivocates `sid_i` Across Honest Parties to Open Commitment to Substituted `Qi` in Multi-Party DKG — (`src/cbmpc/protocol/ec_dkg.cpp`)

---

### Summary

In `key_share_mp_t::dkg`, a Byzantine party j can send a different local session identifier `sid_i._j` to each honest party in round 1. Because the commitment is later verified using the per-party-received `sid_i._j` directly — without any cross-party consistency check — the Byzantine party can commit to a different `Qi` value to each honest party. All verification checks pass at every honest party, but honest parties end up with inconsistent views of the public key `Q`.

---

### Finding Description

The protocol flow in `key_share_mp_t::dkg` is:

**Round 1** — each party broadcasts `(sid_i, c, h_consistency)`: [1](#0-0) 

**After round 1** — global `sid` is derived from all received local sids: [2](#0-1) 

**Round 2 verification** — commitment is opened using the per-party-received `sid_i._j`: [3](#0-2) 

The critical issue is that `sid_i._j` — the local sid received from party j — is used directly as `external_sid` in `commitment_t(sid_i._j, job.get_pid(j))` without first verifying that all honest parties received the same value from party j.

`plain_broadcast` provides **no consistency guarantee**. It is implemented as independent point-to-point sends: [4](#0-3) [5](#0-4) 

A Byzantine party controlling its own transport can send `in[i]` with different content to each recipient `i`. The `uniform_msg_t::pack` method always serializes the same `msg` for all recipients when called by an honest party, but a Byzantine peer is not constrained by this. [6](#0-5) 

The two existing guards in round 2 do **not** prevent the attack:

1. `sid_msg._j != sid` — checks that party j's claimed global sid matches the local party's computed sid. Since party j knows what `sid_i` it sent to each honest party, it can compute the correct global sid for each and send it individually. Each honest party's check passes independently. [7](#0-6) 

2. `h._j != h.msg` — checks that party j's hash of all commitments matches the local party's hash. Since party j sent different `c._j` to different parties, each honest party has a different `h.msg`, and party j can send the matching value to each. [8](#0-7) 

---

### Impact Explanation

**Concrete attack (3-party DKG, party 2 Byzantine):**

- **Round 1:** Party 2 sends `(sid_A, c_A)` to party 0 and `(sid_B, c_B)` to party 1, where `c_A = commitment_t(sid_A, pid_2).gen(Qi_A)` and `c_B = commitment_t(sid_B, pid_2).gen(Qi_B)`, with `Qi_A ≠ Qi_B` and party 2 knowing both discrete logs `x_A`, `x_B`.

- **After round 1:**
  - Party 0 computes `sid_0 = sha256(sid_i_0, sid_A, sid_i_1)`
  - Party 1 computes `sid_1 = sha256(sid_i_0, sid_B, sid_i_1)`

- **Round 2:** Party 2 sends to party 0: `(sid_0, sha256(c_0, c_A, c_1), Qi_A, rho_A, pi_A)` where `pi_A` proves `Qi_A = x_A * G` under `sid_0`. Sends to party 1: `(sid_1, sha256(c_0, c_B, c_1), Qi_B, rho_B, pi_B)` where `pi_B` proves `Qi_B = x_B * G` under `sid_1`.

- **Result:** All checks pass at both honest parties. Party 0 stores `Q_0 = Qi_0 + Qi_A + Qi_1`; party 1 stores `Q_1 = Qi_0 + Qi_B + Qi_1`. The honest parties hold inconsistent public keys and inconsistent key shares. Party 2 can subsequently participate in threshold signing with party 0 using `x_A` (producing valid signatures under `Q_0`) and with party 1 using `x_B` (producing valid signatures under `Q_1`), effectively operating two independent key pairs against the two honest parties.

The same pattern exists in `schnorr_mp.cpp` signing, which has an identical `commitment_t(sid_i._j, ...).open(...)` structure: [9](#0-8) 

---

### Likelihood Explanation

Exploitability requires a Byzantine participant who controls their own transport implementation — a standard assumption for any Byzantine party in a multi-party protocol. No threshold collusion is needed; a single Byzantine party suffices. The attack requires only the ability to send different byte sequences to different peers in round 1, which is trivially achievable by any party that controls its own network stack.

---

### Recommendation

Before using `sid_i._j` as `external_sid` in the commitment verification, the protocol must verify that all honest parties received the same `sid_i._j` from party j. This can be achieved by:

1. **Consistency hash over received `sid_i` values:** After round 1, have each party broadcast `sha256(sid_i._js)` (the hash of all received local sids) and verify that all parties agree on this hash before proceeding to round 2. This is analogous to the `h` check already used for commitments.

2. **Alternatively:** Include `sid_i._j` in the global `sid` derivation and then verify the global `sid` is consistent across all parties before using it — but the current `sid_msg` check is insufficient because it is checked pairwise, not collectively.

The `h_consistency` check in round 1 only covers `curve.get_name()` and does not bind `sid_i._j`: [10](#0-9)

### Proof of Concept

```
3-party DKG: parties {0, 1, 2}, party 2 is Byzantine.

Round 1 (honest parties broadcast uniformly; party 2 equivocates):
  Party 2 → Party 0: sid_i_A, c_A = HMAC(rand_A; Qi_A || sid_A || pid_2)
  Party 2 → Party 1: sid_i_B, c_B = HMAC(rand_B; Qi_B || sid_B || pid_2)
  (Qi_A ≠ Qi_B, party 2 knows x_A = dlog(Qi_A), x_B = dlog(Qi_B))

After round 1:
  sid_at_0 = sha256(sid_i_0, sid_A, sid_i_1)
  sid_at_1 = sha256(sid_i_0, sid_B, sid_i_1)

Round 2 (party 2 sends tailored messages):
  Party 2 → Party 0: sid_at_0, h_0=sha256(c_0,c_A,c_1), Qi_A, rho_A, pi_A(x_A, sid_at_0)
  Party 2 → Party 1: sid_at_1, h_1=sha256(c_0,c_B,c_1), Qi_B, rho_B, pi_B(x_B, sid_at_1)

Verification at party 0:
  sid_msg._2 == sid_at_0  ✓
  h._2 == h_0             ✓
  commitment_t(sid_A, pid_2).set(rho_A, c_A).open(Qi_A)  ✓
  pi_A.verify(Qi_A, sid_at_0, 2)  ✓

Verification at party 1:
  sid_msg._2 == sid_at_1  ✓
  h._2 == h_1             ✓
  commitment_t(sid_B, pid_2).set(rho_B, c_B).open(Qi_B)  ✓
  pi_B.verify(Qi_B, sid_at_1, 2)  ✓

Output:
  Party 0: Q_0 = Qi_0 + Qi_A + Qi_1  (party 2 can sign here with x_A)
  Party 1: Q_1 = Qi_0 + Qi_B + Qi_1  (party 2 can sign here with x_B)
  → Inconsistent DKG outputs; party 2 holds two independent key pairs.
```
