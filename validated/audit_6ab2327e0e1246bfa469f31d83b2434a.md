### Title
Harvester Can Silently Redirect Farmer Block Reward via Mismatched `farmer_reward_address_override` in `RespondSignatures` — (`chia/farmer/farmer_api.py`)

### Summary

The `_process_respond_signatures` function unconditionally accepts `farmer_reward_address_override` from a harvester's `RespondSignatures` message and propagates it as `DeclareProofOfSpace.farmer_puzzle_hash`. The full node then embeds this value verbatim as `farmer_reward_puzzle_hash` in the block's `FoliageBlockData`. Because the farmer's notification path (`notify_farmer_reward_taken_by_harvester_as_fee`) is gated on `NewProofOfSpace.farmer_reward_address_override`, a malicious harvester can send `NewProofOfSpace` with `farmer_reward_address_override=None` (bypassing the notification) and then send `RespondSignatures` with `farmer_reward_address_override=attacker_ph`, silently redirecting the block reward with no farmer-side validation or alert.

---

### Finding Description

**Step 1 — Protocol message definitions**

`RespondSignatures` carries an independent `farmer_reward_address_override` field: [1](#0-0) 

`NewProofOfSpace` carries a separate, independent copy of the same field: [2](#0-1) 

These two fields are never cross-validated anywhere in the farmer.

**Step 2 — Notification is gated on `NewProofOfSpace`, not `RespondSignatures`**

In `new_proof_of_space`, the farmer's only awareness of a reward redirect is: [3](#0-2) 

A harvester that sends `NewProofOfSpace` with `farmer_reward_address_override=None` triggers no notification. The farmer then sends `RequestSignatures` to the harvester.

**Step 3 — Unchecked override in `_process_respond_signatures`**

When the harvester replies with `RespondSignatures`, the farmer blindly accepts whatever `farmer_reward_address_override` is present: [4](#0-3) 

There is no check that this value matches what was declared in `NewProofOfSpace`, and no notification is issued at this point.

**Step 4 — Attacker address flows into `DeclareProofOfSpace`**

The overridden address is placed directly into `DeclareProofOfSpace.farmer_puzzle_hash`: [5](#0-4) 

**Step 5 — Full node uses it verbatim**

In `declare_proof_of_space`, the full node assigns `farmer_ph` directly from the request with no independent validation: [6](#0-5) 

This `farmer_ph` is then passed to `create_unfinished_block`: [7](#0-6) 

The resulting `FoliageBlockData.farmer_reward_puzzle_hash` is the attacker's address. The farmer then unknowingly co-signs this foliage data in the second round (`RequestSignedValues` → `RespondSignatures`), permanently committing the block reward to the attacker.

---

### Impact Explanation

A connected harvester can permanently redirect the farmer's block reward coin to an arbitrary attacker-controlled puzzle hash. The farmer receives no notification (because the notification path checks `NewProofOfSpace.farmer_reward_address_override`, not `RespondSignatures.farmer_reward_address_override`), and the farmer's own signing key co-signs the foliage data that commits to the attacker's address. This is an unauthorized payout redirection of XCH block rewards — Critical/High impact under the allowed scope.

---

### Likelihood Explanation

Any harvester connected to the farmer can execute this attack on every block win. No leaked keys or external compromise is required — only a malicious or compromised harvester process. The farmer has no defense: it cannot distinguish a legitimate CHIP-22 fee arrangement from a silent redirect because the two protocol messages are never cross-validated.

---

### Recommendation

In `_process_respond_signatures`, validate that `response.farmer_reward_address_override` matches the value stored from the corresponding `NewProofOfSpace` (which should be cached alongside the proof-of-space entry in `self.farmer.proofs_of_space`). If the values differ, reject the response and log an error. Additionally, the farmer should always emit a warning/notification whenever `RespondSignatures.farmer_reward_address_override` is non-`None`, regardless of what `NewProofOfSpace` declared.

---

### Proof of Concept

1. Harvester sends `NewProofOfSpace` with `farmer_reward_address_override=None` — farmer logs nothing, sends `RequestSignatures`.
2. Harvester replies with `RespondSignatures` where `farmer_reward_address_override=attacker_bytes32`.
3. `_process_respond_signatures` sets `farmer_reward_address = attacker_bytes32` at line 918.
4. `DeclareProofOfSpace.farmer_puzzle_hash = attacker_bytes32` is sent to the full node.
5. Full node sets `farmer_ph = attacker_bytes32` at line 1069 and calls `create_unfinished_block` with it at line 1151.
6. The resulting `unfinished_block.foliage.foliage_block_data.farmer_reward_puzzle_hash == attacker_bytes32`.
7. Farmer signs the foliage hash (which commits to attacker's address) in the second round.
8. Block is finalized; farmer reward coin is created with attacker's puzzle hash.
