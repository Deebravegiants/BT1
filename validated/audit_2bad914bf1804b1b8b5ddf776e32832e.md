### Title
Unprivileged Caller Can Disrupt Signing Operations via `clean_invalid_attestations` — (File: crates/contract/src/lib.rs)

---

### Summary

Any unprivileged NEAR account can call `MpcContract::clean_invalid_attestations()` to remove participant attestations from `stored_attestations`. Because every participant-gated method (`respond`, `respond_ckd`, `respond_verify_foreign_tx`, `vote_pk`, `vote_reshared`, etc.) is gated by `assert_caller_is_attested_participant_and_protocol_active`, which reads `stored_attestations`, an attacker who triggers this cleanup at a strategically chosen moment — immediately after a whitelist change or after the TEE upgrade deadline expires — can evict all participants whose attestations have just become stale, preventing them from completing in-flight signing rounds and causing pending user requests to time out.

---

### Finding Description

`clean_invalid_attestations` is explicitly documented as "Callable by anyone while the protocol is in `Running`" and carries no authorization guard: [1](#0-0) 

The function delegates to `TeeState::clean_invalid_attestations`, which iterates `stored_attestations` and drops every entry whose attestation fails re-verification against the current whitelist and the `tee_upgrade_deadline_duration` window.

Every participant-gated method reaches `is_caller_an_attested_participant`, which performs a mandatory lookup in `stored_attestations`: [2](#0-1) 

If the entry is absent the call returns `AttestationCheckError::AttestationNotFound` and the method panics. This gate is enforced on `respond`, `respond_ckd`, `respond_verify_foreign_tx`, `vote_pk`, `vote_reshared`, and `start_keygen_instance` / `start_reshare_instance` via: [3](#0-2) 

**Attack path:**

1. Governance participants vote in a whitelist change (e.g., a new Docker image hash replaces the old one). This is a normal, expected governance action.
2. The `tee_upgrade_deadline_duration` grace period begins. During this window, nodes running the old image are still considered valid.
3. The attacker monitors the contract. The moment the grace period expires, all existing attestations that reference the old image hash fail re-verification.
4. The attacker immediately calls `clean_invalid_attestations(max_scan = <large value>)` — no deposit, no participant membership required.
5. All affected participants' entries are purged from `stored_attestations`.
6. Those participants can no longer call `respond()` or any other attestation-gated method until they upgrade their software and re-submit via `submit_participant_info`.
7. Every pending `sign`, `request_app_private_key`, or `verify_foreign_transaction` request that was awaiting a response times out and is rejected with `RequestError::Timeout`.

The contrast with `remove_non_participant_update_votes` — which explicitly restricts callers to participants or the contract itself — makes the missing guard on `clean_invalid_attestations` a clear inconsistency: [4](#0-3) 

---

### Impact Explanation

Pending signing requests time out and are permanently lost from the user's perspective. Users who submitted `sign()` calls with a 1 yoctoNEAR deposit receive a timeout error instead of a signature, breaking any cross-chain transaction they were constructing. If the attacker repeats the call each time participants re-submit (e.g., by front-running re-submissions), the disruption can be sustained for the entire upgrade window. This maps to **request-lifecycle and participant-state manipulation that breaks production safety invariants** — the invariant that an in-progress signing round can always be completed by the threshold set of attested participants.

---

### Likelihood Explanation

Medium. The precondition — attestations becoming stale — is a routine, governance-driven event (whitelist rotation, TEE software upgrade). Any observer of the NEAR chain can detect the exact block at which the grace period expires and submit the call in the same block. No privileged access, no leaked keys, and no collusion are required. The only cost is a small gas fee.

---

### Recommendation

Apply the same authorization pattern used by `remove_non_participant_update_votes`: restrict `clean_invalid_attestations` to calls originating from a current participant or from the contract itself (a self-call). Alternatively, gate it behind `voter_or_panic()` so only attested participants can trigger the sweep. This preserves the cleanup utility while eliminating the unprivileged-caller attack surface.

---

### Proof of Concept

```rust
// Attacker account: "attacker.near" — not a participant, no deposit needed.
// Precondition: governance has rotated the docker-image whitelist and the
// tee_upgrade_deadline_duration has just elapsed, making existing attestations stale.

testing_env!(
    VMContextBuilder::new()
        .signer_account_id("attacker.near".parse().unwrap())
        .predecessor_account_id("attacker.near".parse().unwrap())
        .build()
);

// Step 1: attacker evicts all stale attestations in one call.
let removed = contract
    .clean_invalid_attestations(10_000)
    .expect("no auth check — succeeds for any caller");
assert!(removed > 0, "participant attestations were purged");

// Step 2: a legitimate participant now tries to respond to a pending sign request.
// The call panics because stored_attestations no longer contains their entry.
let result = std::panic::catch_unwind(std::panic::AssertUnwindSafe(|| {
    contract.respond(pending_request, valid_response).unwrap();
}));
assert!(result.is_err(), "respond panics with AttestationNotFound");

// Step 3: the yield-resume promise times out → user receives RequestError::Timeout.
```

    /// Prunes up to `max_scan` stored attestations that fail re-verification (expired or
    /// referencing stale whitelists). Returns the number of entries removed. Callable by
    /// anyone while the protocol is in `Running`.
    #[handle_result]
    pub fn clean_invalid_attestations(&mut self, max_scan: u32) -> Result<u32, Error> {
        log!(
            "clean_invalid_attestations: signer={}, max_scan={}",
            env::signer_account_id(),
            max_scan
        );
        // Running-only: keygen / resharing may reference attestations that have not yet
        // been activated, so cleanup is off-limits during those phases.
        if !matches!(self.protocol_state, ProtocolContractState::Running(_)) {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }
        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);
        Ok(self
            .tee_state
            .clean_invalid_attestations(tee_upgrade_deadline_duration, max_scan as usize))
    }
