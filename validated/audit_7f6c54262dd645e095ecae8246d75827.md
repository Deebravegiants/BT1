### Title
RPC Congestion Check Bypassed at Resharding Epoch Boundary via Shard Layout Mismatch — (`chain/client/src/rpc_handler.rs`)

### Summary

In `process_tx_internal`, the `shard_layout` used to compute `receiver_shard` is derived from the **next** epoch (the epoch of the block that will be built on top of the current head), while `block_congestion_info()` on `cur_block` is keyed by the **current** epoch's shard IDs. At a resharding boundary, these two sets of shard IDs diverge: the new layout introduces child shard IDs that do not exist in the current block's chunk headers. The lookup returns `None`, and `validate_tx` unconditionally skips the congestion check when `receiver_congestion_info` is `None`, admitting the transaction to the mempool regardless of the child shard's congestion state.

### Finding Description

The exact code path in `process_tx_internal` (non-Spice path):

**Step 1** — epoch and layout resolution:
```
epoch_id = get_epoch_id_from_prev_block(&head.last_block_hash)
```
When `head.last_block_hash` is the last block of epoch N, `is_next_block_epoch_start` returns `true`, so `get_epoch_id_from_prev_block` returns epoch N+1's `EpochId`. [1](#0-0) 

**Step 2** — shard layout from new epoch:
```
shard_layout = get_shard_layout(&epoch_id)   // epoch N+1 layout, with child shards
receiver_shard = shard_layout.account_id_to_shard_id(receiver_id)  // new child shard_id
``` [2](#0-1) 

**Step 3** — congestion lookup against the old block:
```
cur_block.block_congestion_info().get(&receiver_shard).copied()
```
`cur_block` is the last block of epoch N. Its `block_congestion_info()` iterates over chunk headers keyed by epoch N's shard IDs (e.g., `{0, 1, 2, 3}`). The new child shard ID (e.g., `4` or `5`) does not exist in that map, so `.get(&new_child_shard_id)` returns `None`. [3](#0-2) [4](#0-3) 

**Step 4** — congestion check silently skipped:
```rust
if let Some(congestion_info) = receiver_congestion_info {
    // ... congestion check ...
}
```
With `receiver_congestion_info = None`, the entire congestion gate is bypassed. `validate_transaction` proceeds without any congestion rejection. [5](#0-4) 

`BlockCongestionInfo` is a `BTreeMap<ShardId, ExtendedCongestionInfo>`. The key is a plain `ShardId` (not a `ShardUId`), so there is no version disambiguation — a child shard ID from epoch N+1 simply has no entry in epoch N's map. [6](#0-5) 

### Impact Explanation

Any unprivileged user can submit a transaction whose `receiver_id` maps to a congested child shard at the exact moment the node's head is the last block of epoch N (i.e., the next block will be the first of epoch N+1 with the new V3 shard layout). The RPC-level congestion gate — the only early-rejection mechanism exposed to users — is silently bypassed. The transaction enters the mempool without the `InvalidTx(ShardCongested)` or `InvalidTx(ShardStuck)` response that should have been returned.

The chunk producer still applies congestion control at inclusion time, so the transaction may ultimately be excluded from blocks. However:
- The RPC admission invariant is violated: the node claims the transaction is valid (`ValidTx`) when it should be rejected.
- The mempool is polluted with transactions that will be dropped at chunk production time.
- Users receive no early feedback, and automated senders can flood the mempool at the boundary window.

### Likelihood Explanation

The trigger window is narrow — it applies only while `head.last_block_hash` is the last block of epoch N during a resharding epoch transition. However:
- The window is deterministic and predictable (epoch length is known).
- Any unprivileged user can submit transactions via the public RPC endpoint.
- Dynamic resharding (V3 layouts) is an active production feature.
- No special privileges or validator access are required.

### Recommendation

In `process_tx_internal`, when looking up congestion info for the receiver shard, use the **current block's** shard layout (epoch N's layout) rather than the next epoch's layout. Specifically, compute `receiver_shard` using `get_shard_layout` for the **current** block's epoch (i.e., `cur_block.header().epoch_id()`), not `get_epoch_id_from_prev_block`. Alternatively, when the next epoch has a different shard layout, map the new child shard ID back to its parent shard ID before looking up congestion info in the current block's map.

### Proof of Concept

1. Configure a test-loop environment with dynamic resharding enabled and a short epoch length.
2. Congest a parent shard (e.g., by dropping chunks or filling delayed receipts) so `block_congestion_info` for that shard shows `missed_chunks_count >= max_congestion_missed_chunks`.
3. Advance the chain to the last block of epoch N (the resharding boundary block).
4. At this exact head, submit a transaction whose `receiver_id` maps to a **child** shard of the congested parent under the new epoch N+1 layout.
5. Assert that `process_tx` returns `ProcessTxResponse::ValidTx` instead of `ProcessTxResponse::InvalidTx(ShardStuck { .. })`.
6. Confirm that the same transaction submitted one block earlier (still in epoch N, using the old layout) correctly returns `InvalidTx(ShardStuck)`.

The root cause is the layout mismatch at lines 175–200 of `chain/client/src/rpc_handler.rs`: `shard_layout` is resolved for epoch N+1 but `block_congestion_info()` is keyed by epoch N's shard IDs. [7](#0-6)

pub struct BlockCongestionInfo {
    /// The per shard congestion info. It's important that the data structure is
    /// deterministic because the allowed shard id selection depends on the
    /// order of shard ids in this map. Ideally it should also be sorted by shard id.
    shards_congestion_info: BTreeMap<ShardId, ExtendedCongestionInfo>,
}

impl BlockCongestionInfo {
    pub fn new(shards_congestion_info: BTreeMap<ShardId, ExtendedCongestionInfo>) -> Self {
        Self { shards_congestion_info }
    }

    pub fn iter(&self) -> impl Iterator<Item = (&ShardId, &ExtendedCongestionInfo)> {
        self.shards_congestion_info.iter()
    }

    pub fn all_shards(&self) -> Vec<ShardId> {
        self.shards_congestion_info.keys().copied().collect()
    }

    pub fn get(&self, shard_id: &ShardId) -> Option<&ExtendedCongestionInfo> {
        self.shards_congestion_info.get(shard_id)
    }
