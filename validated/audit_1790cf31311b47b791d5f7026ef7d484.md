### Title
Deposit for Superseded `propose_update` Entries Is Permanently Locked With No Refund Path — (`File: crates/contract/src/update.rs`, `crates/contract/src/lib.rs`)

---

### Summary

When `propose_update` is called, the proposer attaches a NEAR deposit to cover storage costs for the update entry. When any update reaches threshold and `do_update` executes, it unconditionally clears **all** pending update entries and votes — including proposals from other participants who paid their own deposits. Those deposits are never returned. There is no `remove_update` function, no per-entry deposit tracking, and no refund path for the cleared proposals.

---

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` requires a deposit proportional to the storage cost of the update payload: [1](#0-0) 

The excess above `required` is refunded immediately, but the exact `required` amount is retained by the contract as a storage stake for the entry.

When any update reaches threshold and `vote_update` triggers `do_update`, the implementation in `crates/contract/src/update.rs` clears **every** entry and every vote unconditionally: [2](#0-1) 

Specifically, lines 198–200 call `self.entries.clear()` and `self.vote_by_participant.clear()`, wiping all competing proposals. The storage those entries occupied is freed back to the contract's available balance, but the original depositors of the cleared proposals receive nothing. No deposit amount is stored per entry, and no refund promise is issued for the cleared entries.

The only existing removal path is `remove_update_vote` (lib.rs:1397–1404), which removes a vote record but does not remove the update entry and does not refund the deposit. [3](#0-2) 

---

### Impact Explanation

**Medium.** This breaks the production safety/accounting invariant that a participant's storage deposit is returned when the storage it covers is freed. For a contract-code update, the deposit can be substantial — a 1 MB WASM payload costs roughly 10 NEAR at current storage pricing. Every time a threshold update executes while competing proposals exist, the competing proposers permanently lose their deposits to the contract with no recovery path. This is not a griefing-only issue: the locked funds are real NEAR tokens belonging to legitimate participants, and the contract provides no mechanism to reclaim them.

---

### Likelihood Explanation

**Medium.** In any active governance period where multiple participants independently propose updates (e.g., competing config changes or contract upgrades), the scenario arises naturally. The contract does not prevent multiple simultaneous proposals, and the clearing behavior is unconditional on every successful `do_update` call.

---

### Recommendation

1. Store the depositor's `AccountId` and the exact deposit amount inside `UpdateEntry`.
2. In `do_update`, before calling `self.entries.clear()`, iterate over all entries **except** the one being executed and issue `Promise::new(depositor).transfer(deposit)` for each.
3. Alternatively, add a `remove_update` method (restricted to the original proposer) that removes the entry and refunds the deposit, giving proposers a self-service recovery path analogous to the VestingTrustee workaround described in the reference report.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB contract payload, attaching ~10 NEAR deposit. Entry `id=0` is stored.
2. Participant B calls `propose_update` with a config change, attaching a smaller deposit. Entry `id=1` is stored.
3. Threshold participants vote for `id=1` via `vote_update(1)`.
4. `do_update(&id=1, gas)` executes: it removes entry `id=1`, then calls `self.entries.clear()` (removing entry `id=0`) and `self.vote_by_participant.clear()`.
5. Participant A's ~10 NEAR deposit is now held by the contract with no entry, no refund promise, and no callable method to recover it. [4](#0-3) [5](#0-4)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }
