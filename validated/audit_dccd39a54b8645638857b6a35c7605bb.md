### Title
ETH Deposit Limit Check Omits New Deposit Amount, Allowing Unlimited Single-Transaction ETH Deposits Beyond Configured Cap - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric check for ETH versus LST assets. The ETH branch omits the incoming deposit amount from the comparison, so any single ETH deposit can push total protocol ETH holdings arbitrarily far above the configured `depositLimitByAsset` cap. The deposit limit is the protocol's only mechanism for matching ETH supply to market liquidity and EigenLayer strategy capacity; bypassing it is the direct analog of the "lack of supply limit" finding in the external report.

### Finding Description

In `LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit`:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // ← amount NOT included
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← amount included
}
```

For every LST the check is `totalAssetDeposits + amount > limit`, correctly blocking any deposit that would breach the cap. For ETH the check is `totalAssetDeposits > limit`, which only asks whether the cap has *already* been exceeded before this transaction. As long as the running total is at or below the limit, the check returns `false` (not exceeded) regardless of how large `amount` is.

Concrete example with `depositLimitByAsset[ETH_TOKEN] = 100 ETH`:
- Current total = 99 ETH
- User calls `depositETH` with `msg.value = 10 000 ETH`
- Check: `99 > 100` → `false` → deposit is accepted
- New total = 10 099 ETH — 100× the intended cap

The function is called unconditionally from `_beforeDeposit`, which is the sole guard in `depositETH`. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

`depositLimitByAsset` is the protocol's supply cap, set by the manager to match EigenLayer strategy capacity and secondary-market liquidity for rsETH. Bypassing it for ETH allows the protocol to accumulate an unbounded amount of ETH in a single block:

1. All deposited ETH is eventually routed to EigenLayer via `NodeDelegator`. EigenLayer strategies have their own capacity limits; excess ETH that cannot be deployed sits idle in the deposit pool or NDCs, diluting the effective yield backing rsETH.
2. The rsETH exchange rate (`getRsETHAmountToMint`) is computed from `getTotalAssetDeposits`, which includes the over-limit ETH. Minting rsETH against ETH that cannot be productively deployed degrades the backing ratio for all existing holders.
3. The withdrawal queue (`LRTWithdrawalManager`) sizes its commitments against available assets; a sudden large ETH surplus that cannot be unstaked from EigenLayer on demand can temporarily freeze withdrawals for other users.

Impact classification: **Low — contract fails to deliver promised returns** (the deposit cap guarantee is broken); escalates toward **Medium — temporary freezing of funds** if the excess ETH cannot be deployed or withdrawn. [4](#0-3) [5](#0-4) 

### Likelihood Explanation

The entry point is the public, permissionless `depositETH` function. Any user holding sufficient ETH can trigger this in a single transaction. No special role, flash loan, or multi-step setup is required. The condition is met whenever `getTotalAssetDeposits(ETH_TOKEN)` is at or below the configured limit, which is the normal operating state of the protocol. [3](#0-2) 

### Recommendation

Add the incoming `amount` to the ETH branch, matching the LST logic:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

Additionally, monitor `depositLimitByAsset[ETH_TOKEN]` against EigenLayer strategy capacity and secondary-market rsETH liquidity, and update it proactively as conditions change — mirroring the external report's recommendation to track the supply limit against market conditions.

### Proof of Concept

1. Manager sets `depositLimitByAsset[ETH_TOKEN] = 100 ether` via `LRTConfig.updateAssetDepositLimit`.
2. Protocol accumulates 99 ETH through normal deposits (`getTotalAssetDeposits(ETH_TOKEN) = 99 ether`).
3. Attacker calls `LRTDepositPool.depositETH{value: 10_000 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 10_000 ether)` evaluates `99 ether > 100 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for the full 10 000 ETH.
6. `getTotalAssetDeposits(ETH_TOKEN)` is now 10 099 ETH — 100× the intended cap — while the LST cap would have correctly blocked any deposit exceeding 1 ETH remaining headroom. [1](#0-0) [6](#0-5)
