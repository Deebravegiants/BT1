Audit Report

## Title
Pool Admin Can Set Per-Bin Additional Fees Without Any Cap Enforcement, Bypassing Protocol Fee Limits — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

## Summary
`setPoolBinAdditionalFees` in `MetricOmmPoolFactory` forwards `addFeeBuyE6` / `addFeeSellE6` directly to the pool with zero cap validation, while `setPoolAdminFees` enforces explicit caps against `maxAdminSpreadFeeE6`. Because per-bin additional fees are additive to the global spread fee in swap math, a pool admin can push the effective swap fee for any bin beyond the protocol's 20% hard ceiling (`HARD_MAX_SPREAD_FEE_E6 = 200_000`), causing traders to pay more than the protocol-sanctioned maximum.

## Finding Description
`setPoolAdminFees` enforces caps before updating fees: [1](#0-0) 

`setPoolBinAdditionalFees` performs no such check — it passes values straight through: [2](#0-1) 

`MetricOmmPool.setBinAdditionalFees` only validates the bin index, not the fee magnitude: [3](#0-2) 

In swap math, the per-bin additional fee is **additive** to `baseFeeX64` (which is derived from the global spread fee): [4](#0-3) 

The same additive pattern appears in `_executeSwap` for `buyToken0InBinSpecifiedIn`: [5](#0-4) 

`BinState` stores `addFeeBuyE6` and `addFeeSellE6` as `uint16` (max 65 535 ≈ 6.5535% in E6 units): [6](#0-5) 

The protocol hard cap is `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%), enforced only on global fees: [7](#0-6) 

`setFeeCaps` enforces this hard ceiling for global admin and protocol spread fees but has no analogous cap for per-bin additional fees: [8](#0-7) 

The documentation for `setPoolBinAdditionalFees` acknowledges the interaction with the global spread fee but imposes no cap: [9](#0-8) 

## Impact Explanation
This is an **admin-boundary break**: the pool admin role is semi-trusted only within caps. The per-bin fee path has no analogous guard to the global fee cap system. A pool admin can set `addFeeBuyE6 = addFeeSellE6 = 65535` on any bin, making the effective swap fee for that bin ≈ 26.55% when the global spread fee is at the 20% hard cap. Traders swapping through the affected bin receive fewer tokens than the oracle price and protocol fee ceiling permit — the excess is retained as LP spread surplus. This constitutes **bad-price execution** and a **swap conservation failure** directly reducing trader token output beyond the protocol-sanctioned limit.

## Likelihood Explanation
The pool admin has direct, immediate access to `setPoolBinAdditionalFees` — no timelock, no governance vote, no second step. The call path is three lines with no revert for any fee value ≤ 65 535. The active bin (bin 0 or whichever bin the cursor sits in) is the highest-traffic target; setting its additional fee to max immediately affects all live swaps through that bin.

## Recommendation
Add cap validation in `setPoolBinAdditionalFees` consistent with `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6  > maxAdminBinAdditionalFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminBinAdditionalFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce `globalSpreadFeeE6 + addFeeBuyE6 ≤ HARD_MAX_SPREAD_FEE_E6` at call time, mirroring the invariant already maintained for global fees.

## Proof of Concept
```
1. Pool deployed with spreadFeeE6 = 200_000 (20%, at the hard cap).
2. Pool admin calls:
       factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535);
   → No revert. BinState[0].addFeeBuyE6 = 65535, addFeeSellE6 = 65535.

3. Trader calls swap (zeroForOne = false, buying token0):
   Effective buy fee in SwapMath.buyToken0InBinSpecifiedIn:
       feeX64 = baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)
             ≈ spreadFeeX64(20%) + 6.5535%
             ≈ 26.5535% total

4. Trader pays ~26.55% above oracle mid-price instead of the
   protocol-capped 20%. Excess ~6.55% is retained in the pool
   as LP surplus — trader loses real token1 beyond what the
   protocol's fee ceiling permits.
```
