### Title
Wrong Reference Script Size Used in Minimum Fee Check for Dijkstra Batch Transactions — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, the minimum fee validation for a batch transaction (top-level tx + sub-transactions) uses only the top-level transaction's reference script size when computing the required minimum fee. Sub-transactions can carry their own reference inputs pointing to large reference scripts, but those sizes are excluded from the fee floor calculation. A `batchNonDistinctRefScriptsSize` helper that correctly aggregates reference script sizes across the entire batch is defined in the same file but is never wired into the fee check. This is a direct structural analog to the external report: in a multi-step process, the wrong intermediate value (top-level-only script size) is used in a limit check that should use the combined output of all steps (total batch script size).

---

### Finding Description

The Dijkstra era introduces nested sub-transactions (`subTransactionsTxBodyL`). Each sub-transaction body (`DijkstraSubTxBodyRaw`) carries its own `dstbrReferenceInputs` field, meaning sub-transactions can reference UTxO entries that contain large Plutus or native reference scripts.

The minimum fee check in `dijkstraUtxoTransition` is:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs, line 373
{- minfee pp txTop utxo₀ ≤ txfee txb -}
runTest $ Shelley.validateFeeTooSmallUTxO pp tx originalUtxo
```

`validateFeeTooSmallUTxO` calls `getMinFeeTxUtxo pp tx utxo`. For `DijkstraEra` this resolves to:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs, line 141
instance EraUTxO DijkstraEra where
  ...
  getMinFeeTxUtxo = getConwayMinFeeTxUtxo
```

`getConwayMinFeeTxUtxo` is:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/UTxO.hs, lines 174-175
getConwayMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ txNonDistinctRefScriptsSize utxo tx
```

`txNonDistinctRefScriptsSize` only inspects the top-level transaction's inputs and reference inputs:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/UTxO.hs, lines 183-187
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
```

Meanwhile, the same file defines a batch-aware variant that is never used in the fee check:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs, lines 263-277
-- | Total size of reference scripts across a top-level transaction and all its subtransactions.
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
```

The comment explicitly states its purpose. It is never called from `getMinFeeTxUtxo` or from `dijkstraUtxoTransition`.

---

### Impact Explanation

The tiered reference-script fee (`minFeeRefScriptCostPerByte`, introduced in Conway to prevent DDoS via large reference scripts) is computed from the reference script size passed to `getMinFeeTx`. Because `getConwayMinFeeTxUtxo` passes only the top-level script size, the fee floor for a Dijkstra batch transaction ignores all reference scripts loaded by sub-transactions.

An attacker can construct a batch transaction where:
- The top-level transaction has zero or minimal reference scripts (low computed `minfee`).
- One or more sub-transactions reference large Plutus scripts via `dstbrReferenceInputs`.
- The declared `txfee` satisfies the underestimated floor.

The node accepts and fully validates the transaction—including deserializing and processing the large sub-transaction reference scripts—while collecting a fee that does not cover that cost. This is the same DDoS vector that motivated the Conway reference-script fee change (documented in `docs/adr/2024-08-14_009-refscripts-fee-change.md`), now re-opened for Dijkstra batch transactions.

This falls under: **Medium. Attacker-controlled transactions exceed intended validation limits or modify fees outside design parameters.**

---

### Likelihood Explanation

Any unprivileged transaction sender can craft such a transaction. No special role, key, or governance majority is required. The attacker only needs to:
1. Identify or create UTxO entries containing large reference scripts.
2. Reference those UTxO entries from sub-transaction `referenceInputs`.
3. Set `txfee` to satisfy the underestimated minimum fee.

The Dijkstra era is the newest era and sub-transactions are a new feature, so this path has not been exercised by existing tooling or tests.

---

### Recommendation

Override `getMinFeeTxUtxo` in the `EraUTxO DijkstraEra` instance to use `batchNonDistinctRefScriptsSize` instead of delegating to `getConwayMinFeeTxUtxo`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
instance EraUTxO DijkstraEra where
  ...
  getMinFeeTxUtxo pp tx utxo =
    getMinFeeTx pp tx $ batchNonDistinctRefScriptsSize utxo tx
```

This ensures the tiered reference-script fee covers all reference scripts loaded by the batch, consistent with the intent of the Conway ADR and the existing `batchNonDistinctRefScriptsSize` helper.

---

### Proof of Concept

1. Produce a UTxO entry containing a large Plutus V3 script as a reference script (e.g., 50 KiB).
2. Construct a Dijkstra top-level transaction with:
   - A normal spend input (no reference scripts).
   - One sub-transaction whose `referenceInputs` includes the large-script UTxO entry.
   - `txfee` set to `getConwayMinFeeTxUtxo pp topTx utxo` (which ignores the sub-tx reference script).
3. Submit the transaction. The `validateFeeTooSmallUTxO` check passes because `getMinFeeTxUtxo` returns the underestimated value.
4. The node deserializes and processes the 50 KiB reference script from the sub-transaction at a cost far exceeding the collected fee.

The discrepancy between the correct fee (using `batchNonDistinctRefScriptsSize`) and the accepted fee (using `txNonDistinctRefScriptsSize`) grows linearly with the total size of reference scripts in sub-transactions, up to the per-transaction and per-block size limits.
