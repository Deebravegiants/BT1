### Title
Empty `eligible_proposers` Causes Unconditional Panic in Production Proposer Selection — (`File: crates/apollo_staking/src/staking_manager.rs`)

### Summary

`fetch_and_build_committee` builds and caches a `Committee` struct without validating that `eligible_proposers` is non-empty. In both production deployment configs, `use_only_actual_proposer_selection = true`, which routes every `get_proposer` call directly to `get_actual_proposer`. That function contains an unconditional `assert!(!self.eligible_proposers.is_empty())` — so once a committee with an empty eligible-proposer list is cached, every subsequent consensus round panics and the node halts permanently.

### Finding Description

`fetch_and_build_committee` in `crates/apollo_staking/src/staking_manager.rs` follows this sequence:

1. Fetches stakers from the on-chain contract and validates they are non-empty and non-zero-weight. [1](#0-0) 
2. Selects committee members via `select_committee`, which silently returns an empty `Vec` when `committee_size = 0`. [2](#0-1) 
3. Computes `eligible_proposers` by intersecting committee member addresses with config stakers that have `can_propose = true`. [3](#0-2) 
4. **Builds and returns the `Committee` with no check that `eligible_proposers` is non-empty.** [4](#0-3) 

The result is cached and reused for the entire epoch. [5](#0-4) 

`get_proposer` in production always calls `get_actual_proposer` because `use_only_actual_proposer_selection` is `true` in both production configs: [6](#0-5) [7](#0-6) 

`get_actual_proposer` unconditionally panics on an empty list: [8](#0-7) 

The non-`use_only_actual_proposer_selection` path has a graceful `EmptyCommittee` error return, but that guard is bypassed entirely in production. [9](#0-8) 

The test suite explicitly documents and confirms the panic: [10](#0-9) 

`eligible_proposers` becomes empty in any of these reachable scenarios:

- All `ConfiguredStaker` entries in the config have `can_propose = false`.
- The config staker addresses do not overlap with the addresses returned by the staking contract for the current epoch (e.g., after an epoch rotation that adds new on-chain stakers not yet reflected in the operator config).
- `committee_size = 0` in the config, making `committee_members` empty, which makes `eligible_proposers` empty regardless of `can_propose` flags.

The default `StakingManagerDynamicConfig` ships with `stakers: Vec::new()`, meaning an unconfigured or partially configured node hits this immediately. [11](#0-10) 

There is no `#[validate]` attribute on `CommitteeConfig` or `StakingManagerDynamicConfig` that would reject an empty eligible-proposer set at config-load time. [12](#0-11) 

### Impact Explanation

Once the bad committee is cached, every call to `get_proposer` for that epoch panics. The consensus state machine calls `get_proposer` at the start of every round to determine the proposer. A panic in an async task crashes the consensus manager, halting block production permanently for the affected node. Because the committee is cached, restarting without fixing the config re-enters the same panic immediately.

This matches: **High — consensus decisions broken, rendering the sequencer useless.**

### Likelihood Explanation

`use_only_actual_proposer_selection = true` is hardcoded in both production deployment configs. The `eligible_proposers` list can become empty through a config/contract address mismatch that arises naturally during epoch rotation when new on-chain stakers are not yet reflected in the operator config. The default config ships with an empty stakers list, so any node that starts without a fully populated `default_committee` hits this on the first epoch.

### Recommendation

Add a non-empty check for `eligible_proposers` inside `fetch_and_build_committee`, returning a `CommitteeProviderError` (analogous to the existing `EmptyStakerSet` guard) rather than allowing a broken `Committee` to be cached:

```rust
// After computing eligible_proposers:
if eligible_proposers.is_empty() {
    return Err(CommitteeProviderError::EmptyEligibleProposers);
}
```

Additionally, add a config-load-time `#[validate(length(min = 1))]` on the eligible-proposer subset of `CommitteeConfig.stakers` (those with `can_propose = true`) so misconfiguration is caught before the node starts.

### Proof of Concept

The existing test `get_actual_proposer_no_eligible_panics` already demonstrates the panic path end-to-end with `#[should_panic]`:

```
set_stakers(&mut contract, EPOCH_1, vec![STAKER_1, STAKER_2, STAKER_3]);
// All stakers have can_propose = false in config
let committee = committee_manager.get_committee(E1_H1).await.unwrap(); // succeeds, caches bad committee
let _ = committee.get_actual_proposer(E1_H1, 0);                       // PANICS
``` [10](#0-9) 

To reproduce the production path, set `use_only_actual_proposer_selection = true` (as in both production configs) and call `committee.get_proposer(height, round)` — it routes to `get_actual_proposer` and panics identically. [13](#0-12)

    fn calculate_eligible_proposers(
        &self,
        committee_members: &StakerSet,
        config_stakers: &[ConfiguredStaker],
    ) -> Vec<ContractAddress> {
        let eligible_stakers: HashSet<ContractAddress> = config_stakers
            .iter()
            .filter(|staker| staker.can_propose)
            .map(|staker| staker.address)
            .collect();

        // Iterate over the committee to build the list of eligible proposers, keeping the order

            .map(|staker| staker.address)

    }
