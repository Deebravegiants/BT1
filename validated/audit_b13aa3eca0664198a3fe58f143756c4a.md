Audit Report

## Title
Uninitialized `tokenFeeBps` in `addSupportedToken` Allows Zero-Fee Deposits - (File: contracts/pools/RSETHPool.sol)

## Summary
`addSupportedToken` (L637) adds a token to the protocol but never initializes `tokenFeeBps[token]`, leaving it at the Solidity default of 0. Because `setTokenFeeBps` requires `DEFAULT_ADMIN_ROLE` while `addSupportedToken` requires `TIMELOCK_ROLE`, there is no atomic path to set both in one transaction. Any user who deposits between token activation and fee configuration pays zero protocol fee, permanently depriving the protocol of that fee revenue.

## Finding Description
`RSETHPool.sol` maintains `mapping(address token => uint256 feeBps) public tokenFeeBps` (L88). When `addSupportedToken` (L637–656) is called by `TIMELOCK_ROLE`, it sets `supportedTokenOracle[token]` and `tokenBridge[token]` but never writes to `tokenFeeBps[token]`. The token is immediately live for deposits via the `onlySupportedToken` modifier (L100–103), which only checks `supportedTokenOracle[token] != address(0)`.

`viewSwapRsETHAmountAndFee(amount, token)` (L326–347) reads `tokenFeeBps[token]` directly at L335–336:
```solidity
uint256 feeBpsForToken = tokenFeeBps[token];
fee = amount * feeBpsForToken / 10_000;
```
With `feeBpsForToken == 0`, `fee == 0` and `rsETHAmount` is computed on the full `amount`. `feeEarnedInToken[token]` is never incremented (L300).

The only way to set the fee is `setTokenFeeBps` (L583–594), which requires `DEFAULT_ADMIN_ROLE` — a structurally distinct role from `TIMELOCK_ROLE`. These two roles cannot be exercised atomically in a single transaction by design, guaranteeing a multi-block window during which any caller of `deposit(token, amount, referralId)` (L284–305) pays zero fee.

Existing guards are insufficient: `nonReentrant`, `whenNotPaused`, and `onlySupportedToken` all pass normally; none check whether `tokenFeeBps[token]` has been initialized.

## Impact Explanation
Every deposit made in the window between `addSupportedToken` and `setTokenFeeBps` pays zero protocol fee. The protocol permanently loses the fee revenue on those deposits — the depositor receives rsETH calculated on the full token amount rather than `amount - fee`. This constitutes **High — Theft of unclaimed yield**: fee revenue the protocol is entitled to is instead captured by depositors as excess rsETH.

## Likelihood Explanation
**Medium.** Adding a new supported token is a routine, recurring protocol operation. The `AddSupportedToken` event (L136, emitted at L655) is publicly observable on-chain. Any user monitoring for this event can immediately call `deposit` before `DEFAULT_ADMIN_ROLE` issues a separate `setTokenFeeBps` transaction. The role separation makes the gap structural and unavoidable under the current design. The attack requires no special privileges, no flash loan, and no oracle manipulation — only a standard ERC-20 `approve` + `deposit` call sequence.

## Recommendation
Add a `_feeBps` parameter to `addSupportedToken` and set `tokenFeeBps[token] = _feeBps` within the same function body, eliminating the zero-fee window entirely. Validate `_feeBps <= 10_000` consistent with the existing check in `setTokenFeeBps` (L591). This makes fee initialization atomic with token activation and removes the dependency on a subsequent `DEFAULT_ADMIN_ROLE` transaction.

## Proof of Concept
1. `TIMELOCK_ROLE` calls `addSupportedToken(newToken, oracle, bridge)` — `tokenFeeBps[newToken]` remains `0`.
2. Attacker observes the `AddSupportedToken` event on-chain and calls `deposit(newToken, 100e18, "")` in the next block.
3. `viewSwapRsETHAmountAndFee(100e18, newToken)` computes `fee = 100e18 * 0 / 10_000 = 0`; attacker receives rsETH for the full `100e18` with no fee deduction; `feeEarnedInToken[newToken]` remains `0`.
4. `DEFAULT_ADMIN_ROLE` calls `setTokenFeeBps(newToken, 50)` — too late; all prior deposits in the window paid zero fee.
5. Protocol permanently lost fee revenue on all deposits in the window.

**Foundry test sketch:**
```solidity
function test_zeroFeeWindowOnTokenAdd() public {
    vm.prank(timelockRole);
    pool.addSupportedToken(address(newToken), address(oracle), address(bridge));

    // tokenFeeBps[newToken] == 0 at this point
    (, uint256 fee) = pool.viewSwapRsETHAmountAndFee(100e18, address(newToken));
    assertEq(fee, 0); // zero fee confirmed

    vm.prank(attacker);
    newToken.approve(address(pool), 100e18);
    pool.deposit(address(newToken), 100e18, "");
    assertEq(pool.feeEarnedInToken(address(newToken)), 0); // no fee collected

    vm.prank(adminRole);
    pool.setTokenFeeBps(address(newToken), 50); // too late
}
```
