### Title
Unchecked Detached NEAR Token Refund Transfers Can Permanently Lock Excess Deposits in the Contract - (File: `crates/contract/src/lib.rs`)

---

### Summary
The MPC contract uses `Promise::transfer(diff).detach()` to refund excess deposits to callers. Because the promise is detached, its success or failure is never observed. If the refund transfer fails — for example, because the predecessor account (a smart contract) is deleted between the call and the promise's execution — the excess deposit is silently retained by the MPC contract with no recovery path for the user.

---

### Finding Description

Two locations in `crates/contract/src/lib.rs` issue refund transfers using the fire-and-forget `.detach()` pattern:

**Location 1 — `require_deposit` (called by every user-facing request method):**

```rust
// lib.rs:134-138
Some(diff) => {
    if diff > NearToken::from_yoctonear(0) {
        log!("refund excess deposit {diff} to {predecessor}");
        Promise::new(predecessor.clone()).transfer(diff).detach();
    }
}
```

This helper is invoked by `check_request_preconditions`, which is called by `sign()`, `request_app_private_key()`, and `verify_foreign_transaction()`. Any smart contract that calls these methods with an excess deposit relies on this refund path.

**Location 2 — `submit_participant_info`:**

```rust
// lib.rs:844-848
if let Some(diff) = attached.checked_sub(cost)
    && diff > NearToken::from_yoctonear(0)
{
    Promise::new(account_id).transfer(diff).detach();
}
```

In both cases, `.detach()` means the NEAR runtime's receipt result is never inspected. If the target account is deleted (or otherwise unreachable) before the scheduled receipt executes, the transfer receipt fails. In NEAR's receipt model, the failed transfer amount is returned to the contract's own balance — not to the original depositor — and since there is no callback registered, the contract has no mechanism to re-attempt or log the failure. The excess deposit is permanently absorbed into the contract's balance.

---

### Impact Explanation

This breaks the production accounting invariant that excess deposits must be returned to their senders. Any smart contract that:

1. Calls `sign()`, `request_app_private_key()`, `verify_foreign_transaction()`, or `submit_participant_info()` with an attached deposit larger than the required minimum, **and**
2. Has its account deleted (or becomes unreachable) before the refund receipt executes,

will permanently lose the excess deposit. The funds are not destroyed — they accumulate in the MPC contract's balance — but they are irrecoverable by the original sender. There is no sweep, no admin withdrawal, and no retry mechanism for failed detached promises.

This maps to the **Medium** allowed impact: *balance or contract execution-flow manipulation that breaks production safety/accounting invariants*.

---

### Likelihood Explanation

The entry path is fully unprivileged: any account (including a smart contract) can call `sign()` with an excess deposit. The failure condition — account deletion between call dispatch and receipt execution — is a realistic NEAR pattern for ephemeral or upgradeable proxy contracts. Cross-contract call chains where the intermediate caller is a short-lived contract are a documented NEAR use case. The minimum deposit is only 1 yoctoNEAR, but callers may attach larger amounts (e.g., to satisfy storage costs or due to frontend rounding), making the lost amount non-trivial in practice.

---

### Recommendation

Replace the detached fire-and-forget pattern with a checked promise chain. Return the refund promise as part of the function's return value so that the NEAR runtime propagates failures back to the caller, or attach a failure callback that re-credits the sender:

```rust
// Instead of:
Promise::new(predecessor.clone()).transfer(diff).detach();

// Use (return the promise so failure propagates):
Promise::new(predecessor.clone()).transfer(diff)
// or attach a then() callback to handle failure
```

For `require_deposit`, since it is a helper called mid-function (not at the tail), the refund promise should be composed with the main yield promise using `.and()` or returned as a `PromiseOrValue`. Alternatively, track unrefunded amounts in contract state and expose a separate `claim_refund` method.

---

### Proof of Concept

1. Deploy a proxy smart contract `proxy.near` that calls `sign()` on the MPC contract, attaching `10 NEAR` (far above the 1 yoctoNEAR minimum).
2. `require_deposit` computes `diff = 10 NEAR - 1 yoctoNEAR` and schedules `Promise::new("proxy.near").transfer(diff).detach()`.
3. Before the refund receipt executes (within the same block or the next), delete `proxy.near` via `DeleteAccount`.
4. The transfer receipt fails; NEAR returns the `diff` to the MPC contract's balance.
5. No callback fires. The `diff` is permanently locked in the MPC contract. The original depositor has no recourse.
