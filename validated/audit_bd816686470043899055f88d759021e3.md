### Title
Unprivileged `updateRate()` Broadcasts Zero Rate to All L2 Chains, Temporarily Freezing Pool Operations — (`contracts/cross-chain/RSETHMultiChainRateProvider.sol`)

### Summary
`MultiChainRateProvider.updateRate()` has no access control and no zero-rate guard. If called when `LRTOracle.rsETHPrice == 0` (the default storage value before `updateRSETHPrice()` is ever called), it broadcasts `0` to every registered `RSETHRateReceiver`. Any `RSETHPoolV3` using that receiver as its oracle then reverts with division-by-zero on every deposit.

### Finding Description

**Entrypoint — `updateRate()` is permissionless:**

`MultiChainRateProvider.updateRate()` carries only a `nonReentrant` modifier; there is no role check. [1](#0-0) 

**Source of zero — `rsETHPrice` default storage value:**

`LRTOracle.rsETHPrice` is a plain `uint256` storage slot. It is never set in `initialize()`. Until `updateRSETHPrice()` / `_updateRsETHPrice()` is called for the first time, it remains `0`. [2](#0-1) [3](#0-2) 

**Rate read with no zero-check:**

`RSETHMultiChainRateProvider.getLatestRate()` returns `ILRTOracle(rsETHPriceOracle).rsETHPrice()` directly, with no guard against zero. [4](#0-3) 

**Zero is encoded and sent to every receiver:**

`updateRate()` encodes `latestRate` (which is `0`) and calls `ILayerZeroEndpoint.send` for every registered receiver with no validation. [5](#0-4) 

**Receiver stores zero unconditionally:**

`CrossChainRateReceiver.lzReceive()` decodes the payload and writes `rate = _rate` with no zero-value check. [6](#0-5) 

**Division by zero in `RSETHPoolV3`:**

`viewSwapRsETHAmountAndFee` divides by `rsETHToETHrate` (which is now `0`), reverting every call. [7](#0-6) 

The `deposit()` function invokes `viewSwapRsETHAmountAndFee` inside the `limitDailyMint` modifier before any state change, so every deposit reverts. [8](#0-7) 

### Impact Explanation

All L2 `RSETHPoolV3` deposits revert with division-by-zero for as long as the receiver holds `rate == 0`. This is a **temporary** freeze of L2 pool operations — the admin can recover by calling `updateRSETHPrice()` on L1 (setting a non-zero price) and then calling `updateRate()` again to broadcast the correct value. The freeze is therefore **not permanent**, which means the claimed impact of "permanent freezing of unclaimed yield" is overstated. The correct scoped impact is **Medium — Temporary freezing of funds** (L2 pool deposits).

### Likelihood Explanation

The precondition — `rsETHPrice == 0` — is only satisfied during the window between `LRTOracle` deployment/upgrade and the first successful `updateRSETHPrice()` call. In a well-operated deployment this window is short, but it is real and requires no privilege to exploit. The attacker only needs to pay the LayerZero messaging fee.

### Recommendation

1. Add a zero-rate guard in `updateRate()`:
   ```solidity
   uint256 latestRate = getLatestRate();
   require(latestRate > 0, "Rate must be non-zero");
   ```
2. Add a zero-rate guard in `CrossChainRateReceiver.lzReceive()`:
   ```solidity
   require(_rate > 0, "Received zero rate");
   ```
3. Consider restricting `updateRate()` to an authorized keeper role to prevent griefing entirely.

### Proof of Concept

```solidity
// 1. Deploy LRTOracle (rsETHPrice == 0, updateRSETHPrice never called)
// 2. Deploy RSETHMultiChainRateProvider pointing at LRTOracle
// 3. Register an RSETHRateReceiver on a test L2
// 4. Anyone calls:
rateProvider.updateRate{value: fee}();
// => RSETHRateReceiver.rate == 0

// 5. On L2:
pool.deposit{value: 1 ether}("ref");
// => reverts: division by zero in viewSwapRsETHAmountAndFee
//    (rsETHAmount = amountAfterFee * 1e18 / 0)
```
