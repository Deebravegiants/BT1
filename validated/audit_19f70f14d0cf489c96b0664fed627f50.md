### Title
Class-Level `_streamable_fields` Permanent Mutation via CHIP-0029 RPC Request Corrupts All Subsequent Non-CHIP-0029 Deserialization — (`chia/wallet/util/clvm_streamable.py`)

---

### Summary

`json_deserialize_with_clvm_streamable` unconditionally overwrites the class-level `_streamable_fields` attribute on any compound `Streamable` type it processes. An unprivileged caller sending a single CHIP-0029 RPC request with `translation_layer=BLIND_SIGNER_TRANSLATION` permanently replaces the canonical `convert_function`s on the target class with translation-layer-specific ones. All subsequent non-CHIP-0029 callers that invoke `from_json_dict` directly on that class — as the `marshal` decorator does — will silently use the wrong convert functions, causing deserialization failures or silent data corruption for every honest wallet client thereafter.

---

### Finding Description

**Root cause — line 135 of `chia/wallet/util/clvm_streamable.py`:**

```python
setattr(streamable_type, '_streamable_fields', tuple(new_streamable_fields))
```

`_streamable_fields` is a `ClassVar[StreamableFields]` set once at class-decoration time by the `@streamable` decorator:

```python
# chia/util/streamable.py line 649
cls._streamable_fields = create_fields(cls)
```

The `setattr` call in `json_deserialize_with_clvm_streamable` writes directly to the class object — not an instance — permanently replacing the canonical `convert_function` for every field with a `functools.partial` that bakes in the caller's `translation_layer`. There is no save/restore, no locking, and no copy-on-write.

**Attack path through the `marshal` decorator (`chia/rpc/util.py` lines 40–56):**

```python
(
    request_class.from_json_dict(request)          # non-CHIP-0029: direct call
    if not request.get("CHIP-0029", False)
    else json_deserialize_with_clvm_streamable(    # CHIP-0029: mutates class, then calls from_json_dict
        request, request_hint,
        translation_layer=ALL_TRANSLATION_LAYERS[request["translation"]] ...
    )
)
```

Step-by-step:

1. Attacker sends `{"CHIP-0029": true, "translation": "CHIP-0028", ...}` to any of the 121 `@marshal`-decorated wallet RPC endpoints.
2. `json_deserialize_with_clvm_streamable` enters the `elif hasattr(streamable_type, "streamable_fields")` branch (line 119), builds new fields with `translation_layer=BLIND_SIGNER_TRANSLATION` baked into each `convert_function`, and calls `setattr(streamable_type, '_streamable_fields', ...)` (line 135).
3. The class-level `_streamable_fields` is now permanently replaced.
4. Any subsequent non-CHIP-0029 request causes the `marshal` decorator to call `request_class.from_json_dict(request)` directly (line 44).
5. `from_json_dict` → `streamable_from_dict` → `klass.streamable_fields()` → returns the mutated `cls._streamable_fields` with BLIND_SIGNER_TRANSLATION `convert_function`s.
6. Standard JSON field values are fed into translation-layer-specific converters: for nested `clvm_streamable` types, the converter calls `json_deserialize_with_clvm_streamable(value, streamable_type=T, translation_layer=BLIND_SIGNER_TRANSLATION)`. If `value` is a standard JSON dict (not a hex string), this recurses into the wrong branch, producing incorrect objects or raising exceptions.

The code itself acknowledges the design flaw with a comment at line 113:

```python
# This function is flawed for compound types because it's highjacking the function_to_convert_one_item func
# which does not call back to it. More examination is needed.
```

---

### Impact Explanation

Once a single CHIP-0029 request is processed, the class-level `_streamable_fields` for the targeted request type is permanently corrupted for the lifetime of the wallet process. Every subsequent honest non-CHIP-0029 caller invoking `from_json_dict` on that type will either:

- Receive silently incorrect deserialized objects (if the translation-layer converter happens to accept the standard JSON format but interprets it differently), or
- Receive an exception and have their request rejected entirely.

This satisfies the **High** impact criterion: *permanent or long-lived inability for honest wallets to process valid RPC requests under normal network assumptions*. The wallet's signing pipeline (`gather_signing_info`, `apply_signatures`, `submit_transactions`) is directly in scope, as all three endpoints use `@marshal` and involve compound Streamable types with nested `clvm_streamable` fields.

---

### Likelihood Explanation

- The RPC server is accessible to any local or authenticated RPC client.
- The attacker needs only to send one well-formed CHIP-0029 request with `"translation": "CHIP-0028"` — no special privileges, no key material, no cryptographic work.
- The mutation is triggered on the first such request and persists until process restart.
- The `ALL_TRANSLATION_LAYERS` dict in `rpc/util.py` (line 31) already maps `"CHIP-0028"` to `BLIND_SIGNER_TRANSLATION`, so no novel input is required.

---

### Recommendation

Replace the destructive `setattr` pattern with a local, non-mutating approach: build the modified fields list and pass it directly to a custom deserialization call rather than overwriting the class attribute. The class-level `_streamable_fields` must remain the canonical, immutable set established by `@streamable`. A thread-local or call-scoped override mechanism should be used instead.

### Proof of Concept

```python
import dataclasses
from chia.util.streamable import Streamable, streamable
from chia.wallet.util.clvm_streamable import (
    clvm_streamable, json_deserialize_with_clvm_streamable,
)
from chia.wallet.util.blind_signer_tl import BLIND_SIGNER_TRANSLATION

@clvm_streamable
@dataclasses.dataclass(frozen=True)
class Inner(Streamable):
    a: str

@streamable
@dataclasses.dataclass(frozen=True)
class Outer(Streamable):
    inner: Inner

# Capture original convert_function
original_cf = Outer._streamable_fields[0].convert_function

# Step 1: CHIP-0029 caller with translation layer — mutates Outer._streamable_fields
json_deserialize_with_clvm_streamable(
    {"inner": "ffff613180"},   # hex-encoded CLVM blob
    Outer,
    translation_layer=BLIND_SIGNER_TRANSLATION,
)

# Step 2: Verify class-level mutation persists
mutated_cf = Outer._streamable_fields[0].convert_function
assert original_cf is not mutated_cf, "Class was mutated"

# Step 3: Non-CHIP-0029 caller uses from_json_dict directly (as marshal does)
# Standard JSON format for Inner is {"a": "1"}, not a hex string
try:
    result = Outer.from_json_dict({"inner": {"a": "1"}})
    # If no exception: check result is correct
    assert result == Outer(inner=Inner(a="1")), f"Silent corruption: got {result}"
except Exception as e:
    print(f"Honest non-CHIP-0029 request rejected: {e}")
    # This demonstrates the DoS: valid requests now fail
```

**Expected outcome:** Either an exception (DoS for honest callers) or a silently incorrect `result` object, demonstrating that the class-level mutation from the attacker's CHIP-0029 request corrupts all subsequent non-CHIP-0029 deserialization.
