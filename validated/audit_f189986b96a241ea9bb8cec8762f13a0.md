Audit Report

## Title
Aave WETH Reserve Pause Blocks All ETH Withdrawal Completions With No Admin Recovery Path - (File: contracts/LRTWithdrawalManager.sol)

## Summary

When `isAaveIntegrationEnabled` is true and the Aave WETH reserve is paused by the Aave guardian, every call to `completeWithdrawal` for ETH reverts because `_withdrawFromAave` calls `aaveWETHGateway.withdrawETH` with no try/catch or fallback. Critically, every admin recovery path (`emergencyWithdrawFromAave`, `setAaveIntegrationEnabled(false)`, `configureAaveIntegration`) also routes through `_withdrawFromAave` or `_collectInterestToTreasury`, both of which call `aaveWETHGateway.withdrawETH`, so no on-chain escape hatch exists while the reserve is paused. Users whose rsETH was already burned in `unlockQueue` cannot receive their ETH for the entire duration of the pause.

## Finding Description

**Root cause:** `_withdrawFromAave` (L905â€“921) calls `aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this))` at L917 unconditionally, with no try/catch. When the Aave WETH reserve is paused, this call reverts.

**Exploit path:**

1. Operator calls `unlockQueue(ETH_TOKEN, ...)` â€” rsETH is burned at L305 (`IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned)`), ETH is redeemed from the vault, and deposited to Aave via `depositToAaveExternal` at L311.
2. Aave guardian calls `setReservePause(WETH, true)` on the Aave pool â€” a standard, documented Aave v3 guardian action.
3. User calls `completeWithdrawal(ETH_TOKEN, referralId)` â†’ `_processWithdrawalCompletion` â†’ at L722â€“724, since `address(this).balance < request.expectedAssetAmount`, calls `_withdrawFromAave(amountNeeded)` â†’ `aaveWETHGateway.withdrawETH` reverts â†’ entire transaction reverts.
4. Because the transaction reverts, the `popFront()` at L705 and `delete withdrawalRequests[requestId]` at L712 are also reverted, so the withdrawal request is preserved â€” but the user's rsETH is already permanently burned.

**Why existing checks fail â€” no admin escape hatch:**

- `emergencyWithdrawFromAave` (L551â€“563): calls `_collectInterestToTreasury()` (which calls `aaveWETHGateway.withdrawETH` at L954) and then `_withdrawFromAave(amount)` at L560 â€” both revert when the reserve is paused.
- `setAaveIntegrationEnabled(false)` (L486â€“497): when `aaveBalance > 0`, calls `_collectInterestToTreasury()` at L490 and `_withdrawFromAave(aaveBalance)` at L495 â€” both revert when the reserve is paused. The flag `isAaveIntegrationEnabled` is never set to `false` (L503), so the integration cannot be disabled.
- `configureAaveIntegration` (L438â€“448): reconfiguration path also calls `_collectInterestToTreasury()` and `_withdrawFromAave` â€” same revert.

There is no code path that sets `isAaveIntegrationEnabled = false` without first successfully withdrawing from Aave.

## Impact Explanation

**Temporary freezing of funds (Medium).** Users who have initiated withdrawals and had their rsETH burned by the operator in `unlockQueue` cannot complete their ETH withdrawals for the entire duration of the Aave WETH reserve pause. Their rsETH is gone and their ETH is locked in Aave. If the Aave reserve is deprecated or the pause is indefinite, this escalates to **permanent freezing of funds (Critical)**.

## Likelihood Explanation

Aave v3 guardian pauses are a documented, production-exercised mechanism (e.g., the November 2023 Aave CRV incident). The WETH reserve on mainnet is a high-value target. The LRT manager explicitly enables the Aave integration, making this a live production risk whenever `isAaveIntegrationEnabled == true` and ETH withdrawals are pending. No attacker capability is required â€” the Aave guardian acts independently, and any user with a pending unlocked ETH withdrawal is affected.

## Recommendation

1. In `_processWithdrawalCompletion`, wrap the `_withdrawFromAave` call in a try/catch. On failure, revert with a descriptive error (e.g., `AaveWithdrawalFailed`) so the user can retry later without losing their request.
2. Add a separate admin function that sets `isAaveIntegrationEnabled = false` **without** attempting to withdraw from Aave first, allowing the protocol to source ETH from other means once the flag is cleared.
3. `emergencyWithdrawFromAave` should have a direct low-level path (e.g., calling `aavePool.withdraw(WETH, amount, address(this))` directly and then unwrapping WETH) that bypasses the gateway and works even when the gateway is paused.
4. `_collectInterestToTreasury` should also be wrapped in a try/catch in the `setAaveIntegrationEnabled(false)` and `configureAaveIntegration` paths so that disabling the integration does not depend on a successful Aave withdrawal.

## Proof of Concept

**Minimal call sequence (local fork test):**

```solidity
// 1. Setup: operator unlocks ETH queue, ETH deposited to Aave
withdrawalManager.unlockQueue(ETH_TOKEN, firstExcludedIndex, ...);
// rsETH burned, ETH in Aave

// 2. Aave guardian pauses WETH reserve
vm.prank(AAVE_GUARDIAN);
aavePool.setReservePause(WETH_ADDRESS, true);

// 3. User attempts to complete withdrawal
vm.prank(user);
vm.expectRevert(); // aaveWETHGateway.withdrawETH reverts
withdrawalManager.completeWithdrawal(ETH_TOKEN, "ref");

// 4. Admin emergency withdrawal also fails
vm.prank(pauserRole);
vm.expectRevert();
withdrawalManager.emergencyWithdrawFromAave(type(uint256).max);

// 5. Admin cannot disable integration either
vm.prank(lrtManager);
vm.expectRevert(); // _withdrawFromAave inside setAaveIntegrationEnabled reverts
withdrawalManager.setAaveIntegrationEnabled(false);
// isAaveIntegrationEnabled remains true â€” no recovery path
```
