Audit Report

## Title
Rewards Permanently Frozen When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
The `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` even when `totalKernelStaked == 0`. If all stakers exit during an active reward period, the reward tokens accrued during the empty interval are permanently frozen in the contract. The contract itself acknowledges this limitation in its NatSpec but relies on an operational assumption — that tokens will always remain staked — which cannot be enforced at the code level.

## Finding Description
The `updateReward` modifier at lines 232–242 always executes:

```solidity
rewardPerTokenStored = rewardPerToken();
updatedAt = lastTimeRewardApplicable();   // unconditional
```

`rewardPerToken()` (lines 408–414) correctly short-circuits when `totalKernelStaked == 0`, returning `rewardPerTokenStored` unchanged. However, `updatedAt` is still advanced to the current time. When the next `stake()` call triggers `updateReward`, the formula `rewardRate * (lastTimeRewardApplicable() - updatedAt)` uses the newly advanced `updatedAt`, silently skipping all rewards that accrued during the empty window.

`notifyRewardAmount` (line 570) guards against starting a reward period on an empty pool with `if (totalKernelStaked == 0) revert NoStakedTokens()`, but this does not protect against the pool becoming empty *after* a period has started. `initiateWithdrawal` (lines 325–326) immediately decrements `totalKernelStaked` with no restriction during an active reward period.

The contract has no `recoverTokens`, `emergencyWithdraw`, or equivalent function, and does not inherit any `Recoverable` base. The frozen reward tokens are permanently unrecoverable.

Notably, the contract's own NatSpec (lines 18–22) explicitly acknowledges this behavior:
> *"If `totalKernelStaked` ever hits zero during a reward distribution window, any remaining rewards for that period will stay locked in the contract. In this deployment, we're avoiding this issue by ensuring there are always some tokens staked…"*

This operational mitigation is not enforceable at the code level; any user can call `initiateWithdrawal` at any time.

## Impact Explanation
Reward tokens transferred into the contract via `notifyRewardAmount` are permanently frozen for any interval where `totalKernelStaked == 0` during an active reward period. There is no admin or user mechanism to recover them. This matches **Medium — Permanent freezing of unclaimed yield**.

## Likelihood Explanation
Any unprivileged staker can call `initiateWithdrawal` at any time to reduce `totalKernelStaked`. If all stakers exit during an active reward period — a realistic scenario during market stress, migration, or coordinated exit — the pool reaches `totalKernelStaked == 0` while `block.timestamp < finishAt`. No privileged action or attacker coordination is required; ordinary user withdrawals are sufficient. The protocol's own comment confirms this is a known reachable state.

## Recommendation
In the `updateReward` modifier, only advance `updatedAt` when `totalKernelStaked > 0`:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    ...
}
```

This is the standard fix for Synthetix-style staking contracts and ensures the empty-pool time window does not consume the reward budget.

## Proof of Concept

1. Admin calls `notifyRewardAmount(1000e18)` with `totalKernelStaked > 0`. `finishAt = T + duration`, `updatedAt = T`, `rewardRate = 1000e18 / duration`.
2. All stakers call `initiateWithdrawal` for their full balances. After the last withdrawal, `totalKernelStaked == 0`. `updatedAt` was last set to `T+100`.
3. No one stakes for 500 seconds. Rewards worth `rewardRate * 500` tokens accrue but are unassignable.
4. At `T+600`, Alice calls `stake(1)`. `updateReward` fires:
   - `rewardPerToken()` returns `rewardPerTokenStored` unchanged (since `totalKernelStaked == 0` before Alice's balance is added).
   - `updatedAt` is set to `T+600`.
5. The `rewardRate * 500` tokens are permanently skipped — `updatedAt` jumped from `T+100` to `T+600` with no corresponding increase in `rewardPerTokenStored`. No mechanism exists to distribute or recover them.

**Foundry test plan:** Deploy `KernelDepositPool`, call `notifyRewardAmount`, have all stakers call `initiateWithdrawal`, warp 500 seconds, have a new staker call `stake(1)`, then assert that `rewardPerTokenStored` did not increase during the empty window and that the contract holds more reward tokens than are attributable to any staker.
