Audit Report

## Title
Stale Chainlink Price Accepted Without Staleness Validation in `ChainlinkPriceOracle.getAssetPrice()` Enables Excess rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` and returns the price with no staleness, round-completeness, or sign checks. Because `LRTOracle.updateRSETHPrice()` is `public` and permissionless, an attacker can persist a stale-derived `rsETHPrice` to storage and immediately deposit a non-stale asset to receive more rsETH than the true TVL warrants, permanently diluting existing holders. The same codebase already implements the required checks in `ChainlinkOracleForRSETHPoolCollateral.getRate()`, confirming this is a code-level omission, not an accepted design choice.

## Finding Description

**Root cause — missing validation in `ChainlinkPriceOracle.getAssetPrice()`:**

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L49-55
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();          // roundID, updatedAt, answeredInRound all discarded
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

No check on `answeredInRound < roundID`, `updatedAt == 0`, `block.timestamp - updatedAt > heartbeat`, or `price <= 0`. Contrast with `ChainlinkOracleForRSETHPoolCollateral.getRate()` (L30-32), which already enforces all three guards.

**Exploit call chain:**

1. `LRTOracle.updateRSETHPrice()` — `public whenNotPaused`, no role check (L87-89).
2. → `_updateRsETHPrice()` → `_getTotalEthInProtocol()` (L331-349) iterates every supported asset and calls `getAssetPrice(asset)` via `ChainlinkPriceOracle`, accepting any stale value.
3. `newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply)` (L250) is computed from the stale TVL.
4. `rsETHPrice = newRsETHPrice` (L313) persists the undervalued price to storage.
5. Attacker calls `LRTDepositPool.depositETH()` or `depositAsset()` for a non-stale asset. `getRsETHAmountToMint` (L520) computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

The numerator uses the live (correct) price of the deposited asset; the denominator uses the stale-derived `rsETHPrice`. Because `rsETHPrice` is artificially low, the quotient is artificially high — the attacker receives excess rsETH.

**Why existing guards are insufficient:**

The `pricePercentageLimit` downside guard (L270-282) only pauses the protocol when the price drop *exceeds* the configured threshold. A stale price that causes a sub-threshold decrease (e.g., 0.93% when the limit is 1%) passes through silently. If `pricePercentageLimit` is zero (unset), there is no protection at all. The upside guard (L252-266) is irrelevant because the stale price causes a *decrease*, not an increase.

## Impact Explanation

**High — Theft of unclaimed yield.**

When a Chainlink feed for any supported L1 asset (e.g., stETH/ETH) is stale and its last reported price is below the true market price, `rsETHPrice` is written to storage at a value below its true level. Any depositor of a non-stale asset (e.g., ETH) immediately after receives more rsETH than the true exchange rate warrants. When the feed self-corrects and `rsETHPrice` is updated upward, the attacker's excess rsETH is already minted and represents a permanent dilution of all existing rsETH holders' accrued yield. The loss is proportional to the staleness magnitude multiplied by the stale asset's share of TVL.

## Likelihood Explanation

**Medium.** Chainlink feeds go stale in documented, recurring scenarios: network congestion preventing keeper transactions, price drift below the deviation threshold (e.g., stETH/ETH 0.5% threshold not triggered for hours), or temporary RPC issues. No special privileges are required — any EOA can call `updateRSETHPrice()`. The attacker only needs to monitor `updatedAt` on-chain and act before the feed self-corrects. The attack is repeatable every time a feed enters a stale window.

## Recommendation

Add round-completeness, staleness, and sign checks in `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already present in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
uint256 public constant MAX_STALENESS = 3600; // per-feed heartbeat

function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundID, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    if (answeredInRound < roundID) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();
    if (price <= 0) revert InvalidPrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Each supported asset's feed should have its own configurable `MAX_STALENESS` matching the Chainlink-documented heartbeat for that feed.

## Proof of Concept

**Setup:** Protocol holds 100 stETH (true price 1.08 ETH, stale price 1.07 ETH) and 100 rETH (true price 1.10 ETH, not stale). rsethSupply = 200.

1. True `rsETHPrice` = (100×1.08 + 100×1.10) / 200 = 218/200 = **1.09 ETH**.
2. Stale `rsETHPrice` = (100×1.07 + 100×1.10) / 200 = 217/200 = **1.085 ETH**.
3. Attacker calls `LRTOracle.updateRSETHPrice()` (no access control). `rsETHPrice` is written as 1.085 ETH.
4. Attacker calls `LRTDepositPool.depositETH{ value: 1 ether }(0, "")`.
5. `getRsETHAmountToMint(ETH, 1e18)` = (1e18 × 1e18) / 1.085e18 = **0.9217 rsETH**.
6. True amount should be: (1e18 × 1e18) / 1.09e18 = **0.9174 rsETH**.
7. Attacker receives **0.0043 rsETH excess** (~0.47% overmint) per 1 ETH deposited.
8. Next block, Chainlink feed updates. A keeper calls `updateRSETHPrice()`; `rsETHPrice` rises to 1.09 ETH. Attacker's excess rsETH is now worth more than deposited. Loss is borne by all existing holders.

**Foundry fork test plan:** Fork mainnet, mock `latestRoundData()` on the stETH/ETH feed to return a price last updated 23 hours ago (within 24-hour heartbeat, so no external revert), call `updateRSETHPrice()`, record `rsETHPrice`, deposit 1 ETH, assert `rsethAmountToMint > trueAmountToMint`, then restore the feed and call `updateRSETHPrice()` again to confirm `rsETHPrice` rises and the attacker's rsETH is worth more than deposited.
