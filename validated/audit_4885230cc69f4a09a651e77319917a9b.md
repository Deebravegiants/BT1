### Title
Hardcoded Reference Script Fee Parameters and Size Limits in Conway Era Bypass On-Chain Governance Adjustability — (File: `eras/conway/impl/src/Cardano/Ledger/Conway/PParams.hs`)

---

### Summary

In the Conway era, four critical reference-script parameters are compiled as immutable constants rather than on-chain protocol parameters. They govern per-transaction and per-block size limits and the tiered fee formula for reference scripts. Because they are hardcoded, the governance system cannot adjust them via a `ParameterChange` proposal; the only recourse is a hard fork. If the hardcoded values prove insufficient to deter reference-script fee abuse, attacker-controlled transactions can pay fees below the actual computational cost to validators, and the network has no in-protocol remedy.

---

### Finding Description

The `ConwayEraPParams ConwayEra` instance in `eras/conway/impl/src/Cardano/Ledger/Conway/PParams.hs` implements four parameter getters as compile-time constants:

```haskell
ppMaxRefScriptSizePerTxG    = L.to . const $ 200 * 1024          -- 204,800 bytes
ppMaxRefScriptSizePerBlockG = L.to . const $ 1024 * 1024         -- 1,048,576 bytes
ppRefScriptCostMultiplierG  = L.to . const . fromJust $ boundRational 1.2
ppRefScriptCostStrideG      = L.to . const $ knownNonZeroBounded @25_600
``` [1](#0-0) 

These constants feed directly into two enforcement points:

1. **Per-transaction size check** — `validateRefScriptSize` in the `LEDGER` rule reads `ppMaxRefScriptSizePerTxG` and rejects any transaction whose total reference-script bytes exceed 204,800.
2. **Per-block size check** — `validateBodyRefScriptsSizeTooBig` in the `BBODY` rule reads `ppMaxRefScriptSizePerBlockG` and rejects any block whose aggregate reference-script bytes exceed 1,048,576.
3. **Tiered fee formula** — the fee calculation uses `ppRefScriptCostStrideG` (25,600-byte tier width) and `ppRefScriptCostMultiplierG` (1.2× per-tier price increase) to compute the minimum fee for reference-script usage. [2](#0-1) 

The `ppuWellFormed` check — the gate that rejects malformed `ParameterChange` governance proposals — validates only the parameters that are actual protocol-parameter fields. It has no entry for these four values because they are not fields at all. [3](#0-2) 

The `ConwayEraPParams` type-class definition exposes these four as `SimpleGetter` accessors, not as updatable `HKD`-wrapped fields, confirming they are structurally excluded from the protocol-parameter update mechanism. [4](#0-3) 

ADR-009 explicitly acknowledges the design choice: *"we had to hard code some values, which will be turned into proper protocol parameters in the next era."* [5](#0-4) 

The Dijkstra era corrects this by promoting all four to proper `THKD`-wrapped protocol parameters (`dppMaxRefScriptSizePerBlock`, `dppMaxRefScriptSizePerTx`, `dppRefScriptCostStride`, `dppRefScriptCostMultiplier`), and its `ppuWellFormed` instance validates them. [6](#0-5) [7](#0-6) 

---

### Impact Explanation

**Vulnerability class**: hardcoded limits that bypass intended configurable constraints — direct analog to the external report.

**Fee manipulation (Medium)**: The tiered fee formula uses the hardcoded multiplier (1.2) and stride (25,600 bytes). If these values are insufficient to price reference-script usage above the actual deserialization cost to validators, any transaction submitter can craft transactions that are cheap to submit but expensive to validate. Because the values cannot be changed via a `ParameterChange` governance action, the network has no in-protocol response; a hard fork is the only remedy. This allows attacker-controlled transactions to pay fees outside the intended adjustable design parameters.

**Size-limit immutability**: If the 200 KiB per-transaction or 1 MiB per-block caps need tightening in response to a newly discovered attack vector, governance cannot act. Conversely, if legitimate DApp use cases require larger reference scripts, governance cannot relax the limits either.

Both effects match the allowed impact: *"Medium — Attacker-controlled transactions… exceed intended validation limits or modify fees… outside design parameters."*

---

### Likelihood Explanation

The June 2024 DDoS attack on Cardano mainnet demonstrated that reference-script fee abuse is a real, exploited threat vector. The hardcoded values were chosen reactively under release-cycle pressure. The ADR itself flags them as temporary. Any transaction submitter — no privileged role required — can craft transactions with reference scripts sized to exploit the tiered pricing formula. If the multiplier or stride proves insufficient, the attack surface is open for the entire Conway era with no governance remedy.

---

### Recommendation

1. **Immediate (Conway)**: Document the hardcoded values prominently in the ledger changelog and node release notes so that node operators and the community can monitor whether the values remain sufficient. Establish an off-chain emergency process (e.g., a rapid hard-fork proposal) if the values prove inadequate.
2. **Structural (already done in Dijkstra)**: The Dijkstra era correctly promotes all four values to proper protocol parameters. Ensure the upgrade path from Conway to Dijkstra initialises these parameters from the Conway hardcoded values so there is no discontinuity.
3. **Validation**: Add a `ppuWellFormed` check in Dijkstra that rejects zero-valued or obviously unsafe values for `maxRefScriptSizePerTx`, `maxRefScriptSizePerBlock`, `refScriptCostStride`, and `refScriptCostMultiplier`.

---

### Proof of Concept

1. In Conway era, call `pp ^. ppMaxRefScriptSizePerTxG` on any `PParams ConwayEra` value — it always returns `204800` regardless of the genesis or any enacted governance action.
2. Submit a `ParameterChange` governance proposal attempting to set a new reference-script size limit. The proposal will be accepted by the `GOV` rule (because `ppuWellFormed` does not check these fields), but after enactment the `LEDGER` rule will still enforce the hardcoded `200 * 1024` constant — the enacted value has no effect.
3. Craft a transaction with reference scripts totalling just under 204,800 bytes. The fee is computed using the hardcoded `1.2` multiplier and `25,600`-byte stride. If the resulting fee is below the actual node deserialization cost, the transaction is valid and cheap to spam.
4. Contrast with Dijkstra: `pp ^. ppMaxRefScriptSizePerTxG` reads `dppMaxRefScriptSizePerTx` from the live protocol-parameter state, which governance can update via a `ParameterChange` proposal. [1](#0-0) [8](#0-7) [7](#0-6)
