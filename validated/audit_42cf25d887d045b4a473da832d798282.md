Audit Report

## Title
LP Liquidity Permanently Locked When Owner Address Is USDC-Blacklisted — (`metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary
`LiquidityLib.removeLiquidity` unconditionally transfers pool tokens to the position `owner` address with no alternative recipient path. When a pool token is USDC and the LP's address is subsequently blacklisted, every `removeLiquidity` call reverts atomically, permanently locking the LP's deposited principal. No position-transfer primitive or pull-based claim mechanism exists to recover the funds.

## Finding Description
In `LiquidityLib.removeLiquidity`, after computing the LP's pro-rata share of bin balances, the function unconditionally calls `safeTransfer` to `owner`: [1](#0-0) 

At the pool level, `MetricOmmPool.removeLiquidity` enforces `msg.sender == owner`, preventing any third party from calling on behalf of the blacklisted LP: [2](#0-1) 

The position key permanently binds the LP claim to the original depositor address: [3](#0-2) 

All state mutations (bin balance decrements, share burns) and the `safeTransfer` execute atomically with no `try/catch`. A revert on the USDC transfer rolls back the entire call, leaving shares recorded in `_positionBinShares` but permanently unreachable. No position-transfer primitive, alternative recipient parameter, or admin rescue function exists in the scoped contracts.

## Impact Explanation
Direct, permanent loss of LP principal. An LP who deposited into a USDC pool and is later blacklisted loses their entire deposited principal with no recovery path. This meets the "Critical/High direct loss of user principal" and "broken core pool functionality causing loss of funds or unusable withdraw/liquidity flows" impact gates.

## Likelihood Explanation
USDC blacklisting is explicitly carved out as in-scope non-standard ERC20 behavior per the contest rules. The scenario requires only: (1) a pool with USDC as `token0` or `token1` (common, as USDC is a primary quote asset), and (2) the LP's address being blacklisted after deposit (e.g., due to sanctions). No attacker action is required — the blacklisting is an external event. Both conditions are realistic and have occurred on-chain in other protocols.

## Recommendation
Decouple the withdrawal recipient from the position owner. Add an optional `recipient` parameter to `removeLiquidity` (defaulting to `owner` when zero-address), or implement a pull-based model where computed amounts are credited to an internal balance mapping claimable to any non-blacklisted address:

```solidity
function removeLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    address recipient,   // new: where tokens are sent; defaults to owner if address(0)
    bytes calldata extensionData
) external returns (uint256 amount0Removed, uint256 amount1Removed);
```

## Proof of Concept
1. Pool is deployed with USDC as `token0`.
2. LP calls `addLiquidity`; shares are minted to `owner = LP_ADDRESS`.
3. USDC blacklists `LP_ADDRESS`.
4. LP calls `removeLiquidity(LP_ADDRESS, salt, deltas, "")`.
5. `LiquidityLib.removeLiquidity` computes `amount0Removed > 0` and calls `IERC20(USDC).safeTransfer(LP_ADDRESS, amount0Removed)`.
6. USDC reverts because `LP_ADDRESS` is blacklisted.
7. Entire transaction reverts; LP's shares remain in `_positionBinShares` permanently inaccessible.
8. No alternative call path exists to recover the funds.
