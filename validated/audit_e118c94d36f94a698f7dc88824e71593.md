### Title
Caller-Agnostic `VerifyForeignTransactionRequest` Queue Key Enables Per-Transaction Fan-Out Exhaustion, Permanently Blocking Legitimate Bridge Requests - (File: `crates/contract/src/pending_requests.rs`, `crates/contract/src/dto_mapping.rs`)

---

### Summary

The `pending_verify_foreign_tx_requests` map uses a `VerifyForeignTransactionRequest` key that contains no caller identity. Any unprivileged account can fill the 128-slot fan-out queue for a specific foreign-chain transaction by submitting 128 identical `verify_foreign_transaction` calls. Once the cap is reached, every subsequent submission from any other account for that same transaction is rejected with `PendingRequestQueueFull`. Because NEAR's yield-resume timeout frees slots one at a time (~200 blocks each), the attacker can continuously re-fill freed slots, permanently locking out legitimate bridge users at negligible cost.

---

### Finding Description

`verify_foreign_transaction` stores pending yields in `pending_verify_foreign_tx_requests`, a `LookupMap<VerifyForeignTransactionRequest, Vec<YieldIndex>>`. The map key is constructed by `args_into_verify_foreign_tx_request`:

```rust
// crates/contract/src/dto_mapping.rs:840-848
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
``` [1](#0-0) 

The `VerifyForeignTransactionRequest` struct contains only `request`, `domain_id`, and `payload_version` — **no caller account ID**:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs:124-128
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [2](#0-1) 

This is in direct contrast to `SignatureRequest`, whose key incorporates the caller's identity via `derive_tweak(predecessor_id, path)`:

```rust
// crates/near-mpc-crypto-types/src/sign.rs:118-125
pub fn new(domain: DomainId, payload: Payload, predecessor_id: &AccountId, path: &str) -> Self {
    let tweak = crate::kdf::derive_tweak(predecessor_id, path);
    SignatureRequest { domain_id: domain, tweak, payload }
}
``` [3](#0-2) 

The fan-out queue is capped at `MAX_PENDING_REQUEST_FAN_OUT = 128` per key. `push_pending_yield` panics with `PendingRequestQueueFull` when this cap is exceeded:

```rust
// crates/contract/src/pending_requests.rs:51-58
if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
    env::panic_str(
        &RequestError::PendingRequestQueueFull {
            limit: MAX_PENDING_REQUEST_FAN_OUT,
        }
        .to_string(),
    );
}
``` [4](#0-3) 

On yield timeout, `return_verify_foreign_tx_and_clean_state_on_success` calls `pop_oldest_pending_yield`, which removes **only one** slot (FIFO head) per timeout event:

```rust
// crates/contract/src/pending_requests.rs:97-111
pub(crate) fn pop_oldest_pending_yield<K>(requests: &mut LookupMap<K, Vec<YieldIndex>>, request: &K) {
    ...
    queue.remove(0);
    if queue.is_empty() { requests.remove(request); }
}
``` [5](#0-4) 

The codebase itself acknowledges the caller-agnostic nature of the key in a test comment:

> "caller bob submits the identical request — a different account would today be blocked from receiving a response by alice's submission" [6](#0-5) 

The test confirms both alice and bob share the same queue entry:

> "both yields are queued under the single (caller-agnostic) request key" [7](#0-6) 

---

### Impact Explanation

An attacker who fills the 128-slot queue for a specific `VerifyForeignTransactionRequest` key causes every subsequent `verify_foreign_transaction` call for that same foreign-chain transaction to revert with `PendingRequestQueueFull`. Because the attacker can re-fill freed slots as they time out one-by-one (~200 NEAR blocks ≈ ~200 seconds per slot), the lock can be maintained indefinitely. This permanently prevents any legitimate user from completing a bridge operation that depends on verifying that specific foreign-chain transaction, breaking the request-lifecycle and execution-flow invariants of the foreign-chain bridge without requiring any privileged access or network-level DoS.

---

### Likelihood Explanation

The attack requires only a NEAR account and 128 × 1 yoctoNEAR deposits (effectively zero cost). The attacker needs to know the target foreign-chain transaction ID in advance, which is publicly observable on the foreign chain. The attack is fully automatable: a script monitors the foreign chain for pending bridge transactions and front-runs legitimate users by filling the queue before their `verify_foreign_transaction` call lands. No threshold collusion, TEE access, or privileged role is required.

---

### Recommendation

Include the caller's `predecessor_account_id` in the `VerifyForeignTransactionRequest` map key, analogous to how `SignatureRequest` incorporates the caller via `derive_tweak`. Alternatively, enforce a per-account submission limit within the fan-out queue so a single account cannot occupy more than a small fraction of the 128 slots. A rate-limit or per-account nonce on `verify_foreign_transaction` submissions would also mitigate the cycling attack.

---

### Proof of Concept

```
Attacker (any NEAR account):

1. Observe target Bitcoin tx_id = [0xAB; 32] that victim wants to bridge.

2. Submit 128 identical calls:
   verify_foreign_transaction({
     domain_id: <ForeignTx domain>,
     payload_version: V1,
     request: Bitcoin { tx_id: [0xAB;32], confirmations: 2, extractors: [BlockHash] }
   })
   with 1 yoctoNEAR deposit each.

   → pending_verify_foreign_tx_requests[key].len() == 128

3. Victim submits the same request:
   → push_pending_yield panics: "Pending-request queue is full (limit: 128)"
   → Victim's bridge operation fails.

4. After ~200 NEAR blocks, the oldest attacker yield times out:
   → pop_oldest_pending_yield removes slot 0 → queue.len() == 127

5. Attacker immediately submits one more call:
   → queue.len() == 128 again

6. Repeat step 4-5 indefinitely.
   → Victim can never submit; bridge operation is permanently blocked.

Cost per cycle: 1 yoctoNEAR. No privileged access required.
```

The root cause is in `crates/contract/src/dto_mapping.rs` (`args_into_verify_foreign_tx_request`) which drops the caller identity, and `crates/contract/src/pending_requests.rs` (`push_pending_yield`) which enforces a hard cap on the resulting caller-agnostic key. [1](#0-0) [8](#0-7)
