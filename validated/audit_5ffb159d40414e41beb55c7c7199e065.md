### Title
`is_caller_an_attested_participant` Uses Stale Attestation State Without Re-Verification, Allowing Expired-TEE Nodes to Participate in Threshold Signing and Key Events - (File: `crates/contract/src/tee/tee_state.rs`)

---

### Summary

The `assert_caller_is_attested_participant_and_protocol_active` guard — which gates every node-facing operation including `respond`, `vote_pk`, `vote_reshared`, `start_keygen_instance`, `start_reshare_instance`, and `vote_abort_key_event_instance` — delegates to `is_caller_an_attested_participant`, which checks only that a stored attestation *exists* and that the key identifiers match. It never calls `re_verify` (or `reverify_participants`) to check whether the stored attestation is still valid at the current block timestamp, whether the node's image hash is still in the allowed list, or whether the measurements are still accepted. This is the direct analog of the FraxLend pattern: a critical authorization check uses cached/stale state instead of re-evaluating the time-sensitive invariant at the point of use.

---

### Finding Description

`is_caller_an_attested_participant` (the inner check called by `assert_caller_is_attested_participant_and_protocol_active`) performs three static checks: [1](#0-0) 

1. Is the caller in the participants list?
2. Is there a stored `NodeAttestation` for the caller's TLS key?
3. Do the `account_id` and `account_public_key` fields match?

It does **not** call `re_verify` on the stored `VerifiedAttestation`. Compare this to `reverify_participants`, which does call `re_verify` with the current block timestamp and the current allowed-hash/measurement lists: [2](#0-1) 

`re_verify` checks expiry, image-hash allowlist membership, and measurement validity: [3](#0-2) 

The only mechanism that updates the `accept_requests` flag based on re-verification is `verify_tee`, which must be **explicitly called** by a participant: [4](#0-3) 

`verify_tee` is not called automatically before any signing or key-event operation. Between calls to `verify_tee`, the `accept_requests` flag reflects stale TEE state.

Furthermore, `accept_requests` is checked in `respond`, `respond_ckd`, and `respond_verify_foreign_tx`: [5](#0-4) 

But it is **not** checked in `vote_pk`, `vote_reshared`, `start_keygen_instance`, `start_reshare_instance`, or `vote_abort_key_event_instance` — those functions rely solely on `assert_caller_is_attested_participant_and_protocol_active`, which uses the stale stored attestation: [6](#0-5) 

The `accept_requests` flag itself is also stale: it is set to `true` when `verify_tee` last ran and found a full or sufficient quorum, but it is never automatically invalidated when an individual attestation expires between `verify_tee` calls.

---

### Impact Explanation

A node whose TEE attestation has expired (or whose image hash has been removed from the allowed list via `vote_remove_launcher_hash` or a governance action) can still:

1. Call `respond` / `respond_ckd` / `respond_verify_foreign_tx` to deliver threshold signature shares and have them accepted by the contract — as long as `accept_requests` is still `true` (i.e., `verify_tee` has not yet been called since the attestation became invalid).
2. Call `vote_pk`, `vote_reshared`, `start_keygen_instance`, `start_reshare_instance`, and `vote_abort_key_event_instance` regardless of `accept_requests`, because those functions have no `accept_requests` guard at all.

This breaks the core TEE invariant: only nodes currently running in a verified, unexpired TEE environment should be able to contribute to threshold signatures and key events. A node with an expired attestation may be running unverified or compromised software. Its continued participation in signing means threshold signatures can be produced with contributions from a node that the system's own policy has declared untrusted.

The impact maps to: **Unauthorized threshold signature issuance without the required participant authorization** (the TEE attestation is the authorization mechanism), and **bypass of threshold-signature requirements** in the sense that the TEE-validity precondition is not enforced at the point of use.

---

### Likelihood Explanation

- TEE attestations have a finite expiry (`expiry_timestamp_seconds` in `ValidatedDstackAttestation`; the design doc references a 7-day default and proposes 1 day).
- `verify_tee` is not called automatically; it requires an explicit participant call. Any gap between attestation expiry and the next `verify_tee` call is an open window.
- An image hash can be removed from the allowed list via governance (`vote_remove_launcher_hash`) without any automatic re-verification of existing participants.
- A node operator who wants to continue participating after their attestation expires (e.g., to delay a forced resharing) can simply refrain from calling `verify_tee` and continue submitting `respond` calls.
- The window is bounded only by how frequently honest participants call `verify_tee`, which is not enforced on-chain.

---

### Recommendation

**Short term**: Add a call to `reverify_participants` (or inline the `re_verify` check) inside `is_caller_an_attested_participant`, so that every invocation of `assert_caller_is_attested_participant_and_protocol_active` re-validates the stored attestation against the current block timestamp and current allowed-hash/measurement lists before permitting the operation. This mirrors the fix recommended for FraxLend: call the state-update function (`updateExchangeRate` / `reverify_participants`) inside every function that relies on that state for authorization.

**Long term**: Document the invariant "only nodes with currently valid TEE attestations may call node-facing methods" as an explicit precondition on every function that calls `assert_caller_is_attested_participant_and_protocol_active`, and add a fuzz/property test that verifies no node with an expired attestation can successfully call `respond` or any key-event vote.

---

### Proof of Concept

1. Network is `Running` with 5 participants, threshold 3. All attestations are valid; `accept_requests = true`.
2. Node Eve's attestation expires at block timestamp `T`.
3. No participant calls `verify_tee` between `T` and `T + Δ`.
4. At `T + 1`, Eve calls `respond(request, valid_signature)`.
5. The contract executes:
   - `assert_caller_is_attested_participant_and_protocol_active` → calls `is_caller_an_attested_participant` → finds Eve's stored attestation, checks key match → returns `Ok(())`. **No `re_verify` is called.**
   - `accept_requests` is `true` (stale; `verify_tee` has not run).
   - The signature is cryptographically valid.
   - `resolve_yields_for` delivers the signature to the waiting user.
6. Eve's expired-TEE contribution is accepted as a valid threshold share, completing a signature that the system's own policy should have blocked.
7. Eve can repeat this for any pending request during the window `[T, T+Δ]`.
8. For key-event votes (`vote_pk`, `vote_reshared`), the same path applies with no `accept_requests` guard at all, so Eve can influence key generation or resharing outcomes even after `accept_requests` is set to `false` by a subsequent `verify_tee` call. [1](#0-0) [7](#0-6) [8](#0-7) [9](#0-8)

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
