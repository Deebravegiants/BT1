### Title
Reward Sniping via Stake-Before-Distribution / Unstake-After-Claim Bypasses Long-Term Staker Yield - (File: contracts/KERNEL/KernelDepositPool.sol)

---

### Summary
`KernelDepositPool` has no minimum staking period and no lock on reward claiming. An unprivileged user can observe a pending `notifyRewardAmount` transaction in the mempool, front-run it with a large `stake()`, let rewards accrue, claim them via `getReward()`, and immediately exit via `initiateWithdrawal()`. This is the structural analog of the Trove bad-debt avoidance: a participant times their entry/exit around a redistribution event to capture gains that should belong to long-term participants, leaving those participants with less than they are owed.

---

### Finding Description

`KernelDepositPool` is a Synthetix-style staking contract. Rewards are distributed continuously at `rewardRate` tokens per second, proportional to each staker's share of `totalKernelStaked`. [1](#0-0) 

When `notifyRewardAmount` is called by the admin, it sets a new `rewardRate` over the configured `duration`: [2](#0-1) 

Three properties combine to create the vulnerability:

**1. No minimum staking period.** `stake()` immediately increases `balanceOf` and `totalKernelStaked`, so the staker begins earning at the full rate from the very next second. [3](#0-2) 

**2. Rewards are claimable at any time.** `getReward()` has no staking-duration requirement; it transfers whatever `rewards[msg.sender]` has accumulated. [4](#0-3) 

**3. `initiateWithdrawal()` snapshots rewards before decrementing stake.** The `updateReward(msg.sender)` modifier runs first, writing the full accrued amount into `rewards[msg.sender]`. Only then are `balanceOf` and `totalKernelStaked` decremented. The attacker can therefore call `initiateWithdrawal()` to lock in their reward snapshot, then call `getReward()` to collect — all before the `withdrawalDelay` expires. [5](#0-4) 

The `withdrawalDelay` only delays the return of the principal KERNEL tokens; it does **not** delay reward claiming. [6](#0-5) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

Long-term stakers are diluted. The attacker's large transient stake inflates `totalKernelStaked` during the reward window, reducing `rewardPerToken` for everyone else. After the attacker exits, the remaining stakers' share of future rewards increases, but the rewards already distributed to the attacker are permanently gone. The attacker extracts yield that was intended for committed stakers without bearing the corresponding staking risk.

---

### Likelihood Explanation

**Medium.** The attack requires:
- Monitoring the mempool for `notifyRewardAmount` (straightforward with standard MEV tooling).
- Sufficient capital to stake a meaningful fraction of `totalKernelStaked`.
- The reward amount must exceed the opportunity cost of locking KERNEL tokens for `withdrawalDelay` (up to 30 days per `MAX_WITHDRAWAL_DELAY`). [7](#0-6) 

For large reward distributions relative to the existing staked pool, the attack is highly profitable even with a multi-day delay. The attack is permissionless and requires no privileged access.

---

### Recommendation

1. **Minimum staking period before reward eligibility.** Track each user's `stakeTimestamp` and only allow `getReward()` after a minimum duration (e.g., matching `withdrawalDelay`).
2. **Reward vesting / time-weighted accrual.** Only count rewards earned after a configurable warm-up window following a stake event.
3. **Snapshot-based reward distribution.** Distribute rewards based on stake balances at a fixed snapshot block rather than continuously, preventing same-block entry/exit exploitation.

---

### Proof of Concept

```
Setup:
  totalKernelStaked = 1_000 KERNEL (existing stakers)
  Admin prepares notifyRewardAmount(100_000 rewardTokens) tx in mempool

Step 1 — Attacker front-runs:
  attacker.stake(9_000 KERNEL)
  → totalKernelStaked = 10_000 KERNEL
  → attacker owns 90% of the pool

Step 2 — notifyRewardAmount executes:
  rewardRate = 100_000 / duration

Step 3 — Attacker waits 1 day (T = 86_400 s):
  attacker accrues ≈ 100_000 * (86_400/duration) * 0.90 reward tokens

Step 4 — Attacker calls initiateWithdrawal(9_000):
  updateReward(attacker) snapshots full accrued rewards into rewards[attacker]
  balanceOf[attacker] -= 9_000; totalKernelStaked -= 9_000

Step 5 — Attacker calls getReward():
  receives ≈ 90% of rewards distributed so far

Step 6 — After withdrawalDelay, attacker calls claimWithdrawal():
  receives 9_000 KERNEL principal back

Net result: attacker extracted ~90% of 1-day rewards; original 1,000-KERNEL stakers
received only ~10% instead of 100%.
``` [8](#0-7) [9](#0-8) [10](#0-9)

    function claimWithdrawal(uint256 _withdrawalId) external nonReentrant {
        Withdrawal storage withdrawal = withdrawals[_withdrawalId];

        if (withdrawal.user == address(0)) {
            revert WithdrawalDoesNotExist();
        }

        if (withdrawal.user != msg.sender) {
            revert NotYourWithdrawal();
        }

        if (block.timestamp < withdrawal.unlockTime) {
            revert WithdrawalNotReady();
        }

        if (withdrawal.claimed) {
            revert WithdrawalAlreadyClaimed();
        }

        withdrawal.claimed = true;

        // Remove the withdrawal ID from the user's list of withdrawal IDs
        uint256[] storage userWithdrawalIdsArray = userWithdrawalIds[msg.sender];
        for (uint256 i = 0; i < userWithdrawalIdsArray.length; ++i) {
            if (userWithdrawalIdsArray[i] == _withdrawalId) {
                userWithdrawalIdsArray[i] = userWithdrawalIdsArray[userWithdrawalIdsArray.length - 1];
                userWithdrawalIdsArray.pop();
                break;
            }
        }

        // Transfer the KERNEL tokens from contract to user
        kernelToken.safeTransfer(msg.sender, withdrawal.amount);

        emit WithdrawalClaimed(msg.sender, withdrawal.amount, _withdrawalId);
    }

    function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
        if (_amount == 0) revert AmountZero();

        // Prevent starting a reward period when no tokens are staked to avoid unallocated rewards
        if (totalKernelStaked == 0) revert NoStakedTokens();

        // Transfer reward tokens into the contract
        uint256 balanceBefore = rewardsToken.balanceOf(address(this));
        rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
        uint256 balanceAfter = rewardsToken.balanceOf(address(this));
        // Calculate the actual amount of tokens received in case of a transfer fee (tax)
        uint256 receivedAmount = balanceAfter - balanceBefore;

        if (block.timestamp >= finishAt) {
            rewardRate = receivedAmount / duration;
        } else {
            uint256 remaining = (finishAt - block.timestamp) * rewardRate;
            rewardRate = (receivedAmount + remaining) / duration;
        }

        if (rewardRate == 0) revert RewardRateZero();

        finishAt = block.timestamp + duration;
        updatedAt = block.timestamp;

        emit NotifyRewardAmount(receivedAmount, finishAt);
