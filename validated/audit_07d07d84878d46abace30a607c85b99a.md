### Title
Global Contract Owner Can Steal Pre-Funded NEAR From Deterministic Accounts via Code Update Before `DeterministicStateInit` — (File: `runtime/runtime/src/deterministic_account_id.rs`)

### Summary

When a `DeterministicStateInit` action references a global contract by `GlobalContractIdentifier::AccountId`, the deterministic account ID is derived from the deployer's account ID — not from the actual code hash. Because the global contract owner can update the code at any time via a new `DeployGlobalContractAction`, the deterministic account ID remains stable while the code silently changes. A user who pre-funds the deterministic address and then calls `DeterministicStateInit` will have their NEAR locked under malicious code that the factory owner controls and can drain.

### Finding Description

`derive_near_deterministic_account_id` computes the account ID by Keccak256-hashing the borsh-serialized `DeterministicAccountStateInit`, which contains a `GlobalContractIdentifier` field:

```rust
pub struct DeterministicAccountStateInitV1 {
    pub code: GlobalContractIdentifier,   // ← AccountId variant: only the name, not the code bytes
    pub data: BTreeMap<Vec<u8>, Vec<u8>>,
}
``` [1](#0-0) 

When the `AccountId` variant is used, the serialized bytes contain only the deployer's account name. The actual WASM bytes are not part of the hash input:

```rust
pub fn derive_near_deterministic_account_id(state_init: &DeterministicAccountStateInit) -> AccountId {
    let data = borsh::to_vec(&state_init).expect("borsh must not fail");
    let hash = sha3::Keccak256::digest(&data);
    format!("0s{}", hex::encode(&hash[12..32])).parse().unwrap()
}
``` [2](#0-1) 

Meanwhile, `GlobalContractDeployMode::AccountId` is explicitly designed to allow the owner to overwrite the code at any time:

> "Contract is deployed under the owner account id. Users will be able reference it by that account id. **This allows the owner to update the contract for all its users.**" [3](#0-2) 

When `DeterministicStateInit` executes, `use_global_contract` sets the account's contract to `AccountContract::GlobalByAccount(id)` — a live pointer to whatever code the owner currently has deployed, not a snapshot:

```rust
let contract = match contract_identifier {
    GlobalContractIdentifier::CodeHash(code_hash) => AccountContract::Global(*code_hash),
    GlobalContractIdentifier::AccountId(id) => AccountContract::GlobalByAccount(id.clone()),
};
account.set_contract(contract);
``` [4](#0-3) 

The `DeterministicStateInit` action handler only deploys the contract if the account has no contract yet (`account.contract().is_none()`), and does not validate that the current global code matches any expected hash: [5](#0-4) 

The protocol explicitly supports pre-funding a deterministic account before initialization (tested in `test_deterministic_state_init_prepay_for_storage`). Once pre-funded, the account exists on-chain with a balance but no contract. The deterministic account has no access keys, so the only way to move funds out is through the contract code itself.

### Impact Explanation

**Corrupted protocol value: the NEAR balance of the deterministic account.**

Attack steps:
1. Attacker (`factory.near`) deploys a legitimate global contract with `GlobalContractDeployMode::AccountId`.
2. Victim computes `det_account = derive_near_deterministic_account_id({ code: AccountId("factory.near"), data: {...} })` and pre-funds it with a significant NEAR balance (e.g., for storage staking of a large data set).
3. Attacker replaces the global contract with malicious WASM containing a `drain()` method that calls `promise_batch_action_transfer` to send all funds to the caller.
4. Victim submits `DeterministicStateInit` — the account is initialized with the malicious code (the deterministic account ID has not changed).
5. Attacker calls `drain()` on the deterministic account from any account; all pre-funded NEAR is transferred to the attacker.

The deterministic account has no keys, so the victim cannot recover the funds without going through the contract. The corrupted DB entry is the account balance trie node for `det_account`.

### Likelihood Explanation

- The `AccountId` deploy mode is a first-class, documented feature intended for upgradeable factory patterns (NEP-616).
- Pre-funding deterministic accounts before initialization is an explicitly supported and tested workflow.
- The attacker only needs to own the global contract account — no validator or node-admin privilege is required.
- The attack window is any time between the victim's pre-fund transfer and the `DeterministicStateInit` execution; in a cross-shard scenario this window spans multiple blocks.
- The victim has no on-chain way to verify that the code has not changed between pre-funding and initialization.

### Recommendation

When `GlobalContractIdentifier::AccountId` is used inside `DeterministicStateInit`, resolve the current code hash at the time of initialization and include it in the account ID derivation (analogous to including `tokenToPool[asset]` in the salt in the BetFactory fix). Concretely, `derive_near_deterministic_account_id` should, for the `AccountId` variant, look up the current code hash from state and hash that instead of (or in addition to) the account ID string. Alternatively, document that `DeterministicStateInit` with `AccountId` mode carries this trust assumption and enforce that pre-funding is only safe with `CodeHash` mode.

### Proof of Concept

```
1. factory.near deploys global contract V1 (legitimate) with AccountId mode.
   → TrieKey::GlobalContractCode { AccountId("factory.near") } = V1_WASM

2. victim computes:
   state_init = DeterministicAccountStateInit::V1 {
       code: GlobalContractIdentifier::AccountId("factory.near"),
       data: { "key" => [0u8; 100_000] },
   }
   det_account = derive_near_deterministic_account_id(&state_init)
   // det_account is fixed regardless of what V1/V2 WASM is stored

3. victim sends Transfer(100 NEAR) → det_account
   → det_account now exists with 100 NEAR, no contract

4. factory.near redeploys with V2_WASM (contains drain() that calls
   promise_batch_action_transfer(all_balance, attacker.near))
   → TrieKey::GlobalContractCode { AccountId("factory.near") } = V2_WASM

5. victim submits DeterministicStateInit targeting det_account
   → use_global_contract sets det_account.contract = GlobalByAccount("factory.near")
   → V2_WASM is now the live code on det_account

6. factory.near calls det_account::drain()
   → 100 NEAR transferred to factory.near
   → victim's balance: 0
```
