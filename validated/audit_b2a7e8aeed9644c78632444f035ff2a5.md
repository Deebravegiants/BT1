### Title
Unbounded-downward `BlockConfirmations` Allows Caller to Bypass Bitcoin Finality and Obtain MPC Signature for Unconfirmed Transactions - (`crates/foreign-chain-inspector/src/bitcoin/inspector.rs`)

### Summary

The `BlockConfirmations` field in `BitcoinRpcRequest` is a plain `u64` with no enforced minimum. Neither the on-chain contract (`verify_foreign_transaction`) nor the off-chain Bitcoin inspector rejects a caller-supplied value of `0`. Because the inspector's finality gate is `block_confirmations_threshold <= transaction_block_confirmation`, setting `confirmations: 0` makes the check trivially true for every transaction, including mempool-only (unconfirmed) ones. The MPC network then produces a valid threshold signature attesting to the existence of a transaction that has never been mined, enabling forged foreign-chain verification and double-spend conditions.

### Finding Description

`BlockConfirmations` is defined as a newtype over `u64`: [1](#0-0) 

The ABI schema enforces only `"minimum": 0.0` — the natural floor of an unsigned integer — with no protocol-level minimum of 1 or higher. The field flows directly from the user's `VerifyForeignTransactionRequestArgs` into `BitcoinRpcRequest`: [2](#0-1) 

The on-chain `verify_foreign_transaction` entry point calls `check_request_preconditions`, which validates domain purpose, gas, and deposit — but performs **no validation of the `confirmations` value**: [3](#0-2) 

The request is then enqueued and forwarded verbatim to the MPC nodes. Each node's Bitcoin inspector uses the caller-supplied value as the threshold: [4](#0-3) 

When `block_confirmations_threshold = 0`, the comparison `0 <= transaction_block_confirmation` is always `true` for any `u64`, so the finality gate is never triggered. The inspector proceeds to verify block canonicality and extract values, then the MPC nodes collectively sign the `ForeignTxSignPayloadV1` — which embeds the `confirmations: 0` request verbatim: [5](#0-4) 

### Impact Explanation

A caller obtains a valid MPC threshold signature over a payload that asserts a Bitcoin transaction was "verified" with zero required confirmations. Any downstream NEAR contract or bridge that trusts this signature — without independently re-checking the `confirmations` field embedded in the signed payload — will accept it as proof of a finalized Bitcoin event. The attacker can:

1. Broadcast a Bitcoin transaction spending their own UTXO.
2. Immediately call `verify_foreign_transaction` with `confirmations: 0` before any block is mined.
3. Receive a valid MPC signature.
4. Claim the corresponding NEAR-side asset.
5. Double-spend the Bitcoin transaction (RBF or via a competing chain tip), invalidating the original transaction while keeping the NEAR-side funds.

This matches the allowed High impact: **forged foreign-chain verification that causes invalid bridge execution or double-spend conditions**.

### Likelihood Explanation

The attack requires only a 1 yoctoNEAR deposit and a standard NEAR transaction — no privileged access, no threshold collusion, no TEE bypass. The `confirmations` field is fully attacker-controlled and the contract accepts `0` without complaint. Any user of the `verify_foreign_transaction` API can trigger this immediately.

### Recommendation

Enforce a protocol-level minimum confirmation count at the contract boundary, before the request is enqueued. A reasonable minimum for Bitcoin finality is 6 confirmations. Add a validation step in `verify_foreign_transaction` (or in a dedicated `validate_bitcoin_request` helper) that rejects any `BitcoinRpcRequest` with `confirmations < MIN_BITCOIN_CONFIRMATIONS`:

```rust
const MIN_BITCOIN_CONFIRMATIONS: u64 = 6;

if request.confirmations.0 < MIN_BITCOIN_CONFIRMATIONS {
    env::panic_str("Bitcoin confirmations below minimum required");
}
```

Alternatively, make `BlockConfirmations` a validated newtype that rejects construction with a value below the minimum, analogous to how `ReconstructionThreshold` enforces `t >= MIN_RECONSTRUCTION_THRESHOLD`: [6](#0-5) 

### Proof of Concept

```json
{
  "request": {
    "request": {
      "Bitcoin": {
        "tx_id": "<txid of a mempool-only transaction>",
        "confirmations": 0,
        "extractors": ["BlockHash"]
      }
    },
    "domain_id": 3,
    "payload_version": 1
  }
}
```

1. Attacker broadcasts a Bitcoin transaction (0 confirmations, still in mempool).
2. Attacker calls `verify_foreign_transaction` with `confirmations: 0` and the mempool `tx_id`.
3. Each MPC node's Bitcoin inspector evaluates `0 <= rpc_response.confirmations` → `true` for any transaction the Bitcoin node knows about (including mempool).
4. Nodes sign `ForeignTxSignPayloadV1 { request: Bitcoin { confirmations: 0, ... }, values: [BlockHash(...)] }`.
5. Attacker receives a valid MPC signature and uses it to claim NEAR-side funds.
6. Attacker RBF-replaces the Bitcoin transaction, double-spending the original output. [7](#0-6)
