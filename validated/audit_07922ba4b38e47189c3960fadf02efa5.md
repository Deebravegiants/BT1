### Title
Unvalidated `payload_hash` in `respond_verify_foreign_tx` Enables Forged Foreign-Chain Verification - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash`, but never validates that `response.payload_hash` is actually derived from the submitted `request`. A single Byzantine attested participant (strictly below the signing threshold) who has obtained a legitimately produced threshold signature for one request can replay it as a response to a different pending request, causing the contract to resolve that yield with a `payload_hash` that does not correspond to the caller's actual foreign-chain transaction.

---

### Finding Description

When a user calls `verify_foreign_transaction`, the contract stores a `VerifyForeignTransactionRequest` (containing `request`, `domain_id`, `payload_version`) as a pending yield key and waits for a node to call `respond_verify_foreign_tx`.

The intended invariant is that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload::V1 { request, extracted_values }))` — i.e., the hash commits to both the original request and the values the MPC nodes actually observed on the foreign chain.

In `respond_verify_foreign_tx`, the contract performs:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

This only checks that `signature` is a valid ECDSA signature over `payload_hash` under the root public key. It does **not** check that `payload_hash` is the canonical hash of a `ForeignTxSignPayload` whose embedded `request` field matches the `request` argument passed to `respond_verify_foreign_tx`.

The `request` argument is used solely as a lookup key into `pending_verify_foreign_tx_requests` to resolve the correct yield:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

The `VerifyForeignTransactionResponse` type carries only `payload_hash` and `signature` — the extracted values are never returned: [3](#0-2) 

The node-side code confirms the intended derivation: `payload_hash` is computed as `foreign_tx_payload.compute_msg_hash()` where `foreign_tx_payload` embeds the original `request` and the observed `values`: [4](#0-3) 

But this derivation is enforced only off-chain by honest nodes. The contract never re-derives or constrains `payload_hash` from `request`.

---

### Impact Explanation

A Byzantine attested participant who participated in a threshold signing session for **request A** obtains a valid threshold signature `sig_A` over `hash_A = SHA-256(borsh({request_A, values_A}))`. They then call:

```
respond_verify_foreign_tx(
    request = request_B,          // a different pending request
    response = { payload_hash: hash_A, signature: sig_A }
)
```

The contract accepts this because `sig_A` is a valid signature over `hash_A`. It resolves the yield for `request_B` with `{payload_hash: hash_A, signature: sig_A}`.

The caller of `verify_foreign_transaction(request_B)` receives a response whose `payload_hash` does not correspond to their transaction. Since the extracted values are not included in the response, the caller cannot independently reconstruct the expected hash to detect the mismatch. A bridge contract that uses this response to authorize fund releases (e.g., "Bitcoin tx_id_B was confirmed with N confirmations and block hash H") is deceived into accepting a forged attestation.

This maps to: **High — forged foreign-chain verification that causes invalid bridge execution**.

---

### Likelihood Explanation

- Requires a single Byzantine attested participant (strictly below threshold) — the attacker does not need to compromise a threshold of nodes.
- The attacker must have participated in at least one legitimate threshold signing session to obtain a valid signature. This is a normal operational condition for any participant.
- The attack is executable any time two or more `verify_foreign_transaction` requests are pending simultaneously, which is a routine production condition.
- No special privileges, leaked keys, or physical TEE attacks are required.

---

### Recommendation

In `respond_verify_foreign_tx`, after verifying the signature, recompute the expected `payload_hash` from the `request` field and the `payload_version`, and assert that `response.payload_hash` matches. Since the contract does not know the extracted values, it cannot compute the exact hash. The correct fix is to include the extracted values in the response so the contract can verify:

```
expected_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))
assert_eq!(response.payload_hash, expected_hash)
```

Alternatively, the `ForeignTxSignPayload` (including `values`) should be included in the response so the contract can independently compute and verify the hash, then verify the signature over it. This ensures the `payload_hash` is bound to the specific `request` that was submitted.

---

### Proof of Concept

```
// Setup: two pending requests exist
verify_foreign_transaction(request_A)  // pending
verify_foreign_transaction(request_B)  // pending

// Byzantine node participates in threshold signing for request_A,
// obtains (hash_A, sig_A) where hash_A = SHA-256(borsh({request_A, values_A}))

// Byzantine node submits sig_A as response to request_B:
respond_verify_foreign_tx(
    request = request_B,
    response = VerifyForeignTransactionResponse {
        payload_hash: hash_A,   // hash of a DIFFERENT request's payload
        signature: sig_A,       // valid threshold signature over hash_A
    }
)

// Contract checks: verify_ecdsa_signature(sig_A, hash_A, root_pk) => OK
// Contract resolves yield for request_B with {payload_hash: hash_A, sig_A}
// Caller of request_B receives a response attesting to the wrong transaction
```
