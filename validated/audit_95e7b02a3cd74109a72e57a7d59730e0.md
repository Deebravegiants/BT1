Audit Report

## Title
Missing Staleness Check in `CrossChainRateReceiver.getRate()` Enables Over-Minting of wrsETH on L2 â€” (File: `contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary
`CrossChainRateReceiver` stores the rsETH/ETH rate pushed from L1 via LayerZero and records `lastUpdated` on receipt, but `getRate()` returns the stored `rate` unconditionally with no freshness validation. `RSETHPoolV3` uses this oracle for every deposit, computing `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate`. When the rate is stale and lower than the true L1 rate, depositors receive more `wrsETH` than they are entitled to, and can redeem the excess on L1 at the correct higher rate, extracting value from existing rsETH holders.

## Finding Description
`RSETHRateReceiver` extends `CrossChainRateReceiver` and is the `rsETHOracle` wired into `RSETHPoolV3`. When a LayerZero message arrives, `lzReceive` stores the rate and timestamps it: [1](#0-0) 

`getRate()` then returns the stored value with no freshness guard: [2](#0-1) 

`lastUpdated` is written on every update but is never read anywhere in the contract or by any caller. `RSETHPoolV3.getRate()` delegates directly to this oracle: [3](#0-2) 

Both deposit paths call `viewSwapRsETHAmountAndFee`, which computes: [4](#0-3) 

A stale lower rate produces a larger `rsETHAmount` for the same ETH input. The minting then executes unconditionally: [5](#0-4) 

The rate originates from `LRTOracle.rsETHPrice()` on L1, which increases monotonically as staking rewards accrue: [6](#0-5) 

The rate is pushed by an off-chain keeper calling `updateRate()`: [7](#0-6) 

If the keeper stalls or LayerZero delivery fails, the L2 receiver holds a rate lower than the true L1 rate for an unbounded period, with no on-chain mechanism to detect or reject it. By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` â€” used for collateral tokens in the same pool â€” explicitly checks for stale rounds and zero timestamps before returning a price: [8](#0-7) 

No equivalent protection exists for the cross-chain rsETH rate.

## Impact Explanation
**High â€” Theft of unclaimed yield.**

When the stale rate is `1.00e18` and the true L1 rate is `1.05e18`, a depositor of 1 ETH receives `1.0 wrsETH` instead of `~0.952 wrsETH`. After bridging back to L1 and redeeming at the correct rate, the attacker recovers `~1.05 ETH` â€” a ~5% profit per ETH deposited, extracted from yield that belongs to existing rsETH holders. The `dailyMintLimit` caps per-day exposure but does not prevent the attack; it only bounds the maximum loss per staleness window. Repeated across multiple days of keeper downtime, this degrades the backing ratio for all holders and can push the protocol toward insolvency.

## Likelihood Explanation
**Low.** The attack window opens only when the off-chain keeper fails to call `updateRate()` or when LayerZero fails to relay the message. These are realistic but not routine failure modes. The rsETH rate increases monotonically, so any prolonged outage creates a profitable and growing gap. The attack itself requires no special privileges â€” any unprivileged user can call `deposit()` on `RSETHPoolV3` during the stale window.

## Recommendation
Add a configurable `maxStaleness` parameter to `CrossChainRateReceiver` and revert in `getRate()` if the stored rate is too old:

```solidity
uint256 public maxStaleness; // e.g. 24 hours, set by owner

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

This mirrors the staleness-rejection pattern already present in `ChainlinkOracleForRSETHPoolCollateral`. [9](#0-8)  The `maxStaleness` value should be set conservatively (e.g., 24â€“48 hours) and updatable by the owner to accommodate expected keeper cadence.

## Proof of Concept
1. On L1, `LRTOracle.rsETHPrice()` is updated to `1.05e18` (5% staking appreciation).
2. The off-chain keeper or LayerZero relay fails; `RSETHRateReceiver.rate` on the L2 deployment remains at `1.00e18`. `lastUpdated` is hours old, but `getRate()` returns `1.00e18` without complaint.
3. Attacker calls `RSETHPoolV3.deposit{value: 1 ether}("")` on the L2 chain.
4. `viewSwapRsETHAmountAndFee(1e18)` computes `rsETHAmount = 1e18 * 1e18 / 1.00e18 = 1.0 wrsETH` instead of the correct `~0.952 wrsETH`. [10](#0-9) 
5. `wrsETH.mint(msg.sender, 1.0e18)` executes. [11](#0-10) 
6. Attacker bridges `wrsETH` to L1 and redeems at the true rate of `1.05e18`, receiving `~1.05 ETH` for the `1 ETH` deposited â€” a ~5% profit extracted from existing holders.
7. The protocol has issued more rsETH claims than the ETH it received, degrading the backing ratio.

**Foundry fork test plan:** Fork an L2 deployment; mock `CrossChainRateReceiver.rate` to a value lower than the current L1 `rsETHPrice`; call `RSETHPoolV3.deposit` and assert that `wrsETH` minted exceeds the amount that would be minted at the correct rate; confirm `lastUpdated` is stale and `getRate()` returns the stale value without reverting.
