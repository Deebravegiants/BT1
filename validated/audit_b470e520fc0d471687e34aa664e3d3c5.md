### Title
Single Attested Participant Can Inject Arbitrary CKD Response for `AppPublicKey` Variant Without Threshold Authorization - (File: `crates/contract/src/lib.rs`)

### Summary
The `respond_ckd` function applies no cryptographic verification to the CKD response when the request uses the `AppPublicKey` (privately-verifiable, legacy) variant. Any single attested participant can race to call `respond_ckd` with fabricated `big_y`/`big_c` values before the legitimate leader, causing the contract to deliver a wrong confidential key to the requester. This is the direct analog of the tBTC front-running pattern: just as any observer could replay a fraud-proof call to steal the liquidation reward, any attested participant can race to resolve a pending CKD request with a fabricated payload, bypassing the threshold-authorization requirement entirely.

### Finding Description

`respond_ckd` in `crates/contract/src/lib.rs` handles two sub-variants of CKD requests differently:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no verification at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract performs a BLS12-381 pairing check (`ckd_output_check`) that cryptographically binds the response to the MPC network's public key and the requester's ephemeral key pair. For `AppPublicKey`, the branch is a no-op: the contract unconditionally accepts whatever `big_y` and `big_c` the caller supplies. [2](#0-1) 

The only gate on `respond_ckd` is `assert_caller_is_attested_participant_and_protocol_active`, which requires the caller to be a registered, TEE-attested participant — a single participant, not a threshold. [3](#0-2) 

Once `respond_ckd` succeeds, `resolve_yields_for` drains the entire fan-out queue for that request key and resumes every waiting yield with the (fabricated) response bytes. The legitimate leader's subsequent `respond_ckd` call returns `RequestNotFound` because the map entry is already gone. [4](#0-3) 

The existing unit test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` confirms this: it passes `big_y: [1u8; 48], big_c: [2u8; 48]` — cryptographically meaningless bytes — and the contract accepts them without complaint. [5](#0-4) 

The `AppPublicKey` variant is documented as the "legacy" format but remains fully supported and accepted by the contract. [6](#0-5) 

### Impact Explanation

A single Byzantine attested participant (strictly below the signing threshold) can:

1. Monitor `pending_ckd_requests` on-chain for any `AppPublicKey`-variant request.
2. Call `respond_ckd` with arbitrary `big_y` and `big_c` values before the legitimate leader submits the real MPC output.
3. The contract resolves the yield and delivers the fabricated key material to the requester.
4. The legitimate leader's `respond_ckd` fails with `RequestNotFound`; the correct key is never delivered.

The requester receives a wrong confidential key. Because the `AppPublicKey` variant is "privately verifiable," the requester can only detect the corruption after attempting to use the key — by which point the request lifecycle is permanently closed. This constitutes **confidential key derivation output produced without the required threshold-participant authorization**, matching the Critical impact tier.

### Likelihood Explanation

- The attacker needs only one attested participant account — no threshold collusion.
- Pending CKD requests are stored in contract state (`pending_ckd_requests`) and are publicly readable via view calls.
- NEAR's transaction ordering is deterministic per shard; a participant who submits a `respond_ckd` transaction in the same block as the leader, or slightly earlier, wins the race.
- The `AppPublicKey` variant is the legacy default and is likely used by existing integrations.

### Recommendation

Apply the same `ckd_output_check` pairing verification to the `AppPublicKey` variant, or — since the `AppPublicKey` variant lacks a G2 component needed for the pairing — require the response to carry a BLS signature from a threshold of participants over `(app_id, big_y, big_c)` before the contract resolves the yield. Alternatively, deprecate and remove the `AppPublicKey` variant entirely, requiring all callers to migrate to `AppPublicKeyPV`.

### Proof of Concept

```
1. Alice submits request_app_private_key with AppPublicKey variant.
   → pending_ckd_requests[CKDRequest{app_id, domain_id, app_public_key}] = [yield_id]

2. Malicious participant Eve (single attested node) reads the pending request
   from contract state via a view call.

3. Eve calls respond_ckd(request, CKDResponse { big_y: [0u8;48], big_c: [0u8;48] })
   before the legitimate leader.

4. Contract executes:
     assert_caller_is_attested_participant_and_protocol_active() → Ok (Eve is attested)
     match AppPublicKey(_) => {}                                 → no verification
     resolve_yields_for(...)                                     → resumes Alice's yield
                                                                    with fabricated bytes

5. Alice's contract callback receives { big_y: [0;48], big_c: [0;48] }.
   The legitimate leader's respond_ckd returns Err(RequestNotFound).

6. Alice's TEE application decrypts the fabricated ciphertext with her private key
   and obtains a garbage secret — the correct MPC-derived key is never delivered.
```

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}

  - **Privately verifiable** (legacy): a single G1 point, e.g. `"bls12381g1:<base58>"` or `{"AppPublicKey": "bls12381g1:<base58>"}`.
  - **Publicly verifiable**: a pair of points `(pk1, pk2) = (a·G1, a·G2)`, passed as `{"AppPublicKeyPV": {"pk1": "bls12381g1:<base58>", "pk2": "bls12381g2:<base58>"}}`. This allows anyone to verify the encrypted result on-chain without the app's secret key.
- `domain_id` (integer): identifies the master key to use for deriving the ckd, and must correspond to bls12381.
