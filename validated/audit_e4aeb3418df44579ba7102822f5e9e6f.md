Audit Report

## Title
Fee Baseline Uses Last Stored Price Instead of Peak Price, Charging Fees on Slashing Recovery — (`contracts/LRTOracle.sol`)

## Summary
`_updateRsETHPrice()` computes the protocol fee baseline as `previousTVL = rsethSupply * rsETHPrice` using the last stored price, not `highestRsethPrice`. When the rsETH price drops within `pricePercentageLimit` (no pause fires, `rsETHPrice` is written down to the depressed value), then later recovers above the prior peak, the fee is charged on the full delta from the depressed price — including the recovery of previously-slashed value — rather than only on genuine new yield above the prior peak. This causes excess rsETH to be minted to the treasury, diluting all existing rsETH holders.

## Finding Description
In `_updateRsETHPrice()`, the fee baseline is set at line 234:

```solidity
uint256 previousTVL = rsethSupply.mulWad(rsETHPrice);
```

`rsETHPrice` is the last stored price, overwritten at line 313 on every non-pausing call. `highestRsethPrice` is tracked separately and updated only upward (lines 294–296), but is **never used** in the fee calculation.

The downside-pause guard at lines 270–282 only fires when `diff > pricePercentageLimit.mulWad(highestRsethPrice)`. When the drop is within the limit, execution falls through and line 313 writes `rsETHPrice` to the depressed value. On the subsequent recovery call:

- `previousTVL = rsethSupply × depressedPrice` (line 234)
- `totalETHInProtocol` = TVL at recovered price
- `rewardAmount = totalETHInProtocol - previousTVL` (line 245) — includes both slashing recovery and genuine new yield
- `protocolFeeInETH = rewardAmount * protocolFeeInBPS / 10_000` (line 246)

The fee is inflated by the recovery portion. `highestRsethPrice` is available but unused as a fee baseline.

**Concrete trace (PoC values):**
- `rsethSupply = 1000e18`, `rsETHPrice = 1.1e18`, `highestRsethPrice = 1.1e18`, `protocolFeeInBPS = 1000`, `pricePercentageLimit = 0.20e18`
- Step 1: price drops to `0.9e18`. `diff = 0.2e18`, limit = `0.20 × 1.1 = 0.22e18`. `0.2 < 0.22` → no pause. `rsETHPrice` written to `0.9e18`.
- Step 2: price recovers to `1.2e18`. `previousTVL = 1000 × 0.9 = 900e18`. `rewardAmount = 1200 - 900 = 300e18`. `protocolFeeInETH = 30e18`. Correct fee (yield above peak only): `(1200 - 1100) × 10% = 10e18`. Excess fee: `20e18` ETH-equivalent minted to treasury.

## Impact Explanation
Excess rsETH is minted to the treasury (lines 304–307). Because rsETH is a share token, minting extra shares to the treasury dilutes every existing holder's proportional claim on the underlying ETH. The dilution is direct and quantifiable — in the PoC, the treasury receives 3× the legitimate fee, with the excess `20e18` ETH-equivalent transferred from rsETH holders to the treasury. This constitutes **theft of unclaimed yield** (High severity).

## Likelihood Explanation
- `protocolFeeInBPS > 0` is the normal operational state (manager-configurable, capped at 1500 BPS).
- EigenLayer slashing or LST de-pegging events can cause small, within-limit price drops.
- `pricePercentageLimit` is set by admin to allow small drops without pausing; any drop smaller than the configured limit writes down `rsETHPrice`.
- `updateRSETHPrice()` is a public, permissionless function (line 87) — any external caller can trigger the fee mint once the price has recovered.
- No attacker-controlled input is required; the condition arises from normal market events and the existing price-tracking logic.

## Recommendation
Replace the fee baseline with the maximum of the last stored price and the all-time-high price, so fees are only charged on genuine new yield above the previous peak:

```solidity
uint256 feeBaselinePrice = rsETHPrice > highestRsethPrice ? rsETHPrice : highestRsethPrice;
uint256 previousTVL = rsethSupply.mulWad(feeBaselinePrice);
```

This anchors `previousTVL` to the peak TVL when the price recovers from a slashing event, ensuring `rewardAmount` only captures growth that is genuinely new yield above the prior high-water mark.

## Proof of Concept
Foundry state-based unit test (no fork required):

1. Deploy `LRTOracle` with mock `rsETH` (supply `1000e18`), mock deposit pool, mock withdrawal manager, and `LRTConfig` with `protocolFeeInBPS = 1000`.
2. Set `pricePercentageLimit = 0.20e18`, `rsETHPrice = 1.1e18`, `highestRsethPrice = 1.1e18`.
3. Set mock TVL to `900e18`. Call `updateRSETHPrice()`. Assert no fee minted, `rsETHPrice == 0.9e18`, `highestRsethPrice == 1.1e18`.
4. Set mock TVL to `1200e18`. Call `updateRSETHPrice()`. Capture `rsethAmountToMintAsProtocolFee` from the `FeeMinted` event.
5. Assert `rsethAmountToMintAsProtocolFee == 10e18 / newRsETHPrice` (correct, peak-anchored fee), not `30e18 / newRsETHPrice` (inflated, depressed-baseline fee).

The test will fail on step 5 against the current code, confirming the bug.
