### Title
Incomplete TEE Attestation Validity Check in `is_caller_an_attested_participant` Allows Expired/Revoked-Attestation Participants to Submit Threshold Signatures — (File: `crates/contract/src/tee/tee_state.rs`)

---

### Summary

`is_caller_an_attested_participant` verifies only the **identity fields** of a stored attestation (account ID and account public key) but never calls `re_verify` to confirm the attestation is still **currently valid** (not expired, not using a revoked image hash, not failing measurement checks). This is the direct analog of the NFT collection check that verifies `collection.key` but ignores `collection.verified`: the existence and ownership of the attestation record is confirmed, but its current validity status is not.

---

### Finding Description

`assert_caller_is_attested_participant_and_protocol_active` is the per-call gate used by every node-facing mutating method — `respond`, `respond_ckd`, `respond_verify_foreign_tx`, `vote_pk`, `vote_reshared`, `vote_abort_key_event_instance`, `start_keygen_instance`, and `start_reshare_instance`. [1](#0-0) 

It delegates to `is_caller_an_attested_participant`, which performs three checks:

1. The signer's account ID is in the active participant set.
2. A `NodeAttestation` record exists in `stored_attestations` keyed by the participant's TLS public key.
3. The stored `account_id` and `account_public_key` match the transaction signer. [2](#0-1) 

**What is never checked**: whether `stored_attestations[tls_key].verified_attestation` still passes `re_verify` — i.e., whether the attestation has expired, whether the MPC image hash it was issued for is still on the allowlist, or whether the measurements still match.

The `re_verify` path exists and is used by `reverify_participants`: [3](#0-2) 

But `reverify_participants` is only invoked from `reverify_and_cleanup_participants`, which is only called from `verify_tee()` (a separate, explicitly-triggered governance method) and `vote_new_parameters`. It is **never called** inside `is_caller_an_attested_participant`. [4](#0-3) 

The global `accept_requests` flag is a coarse gate that is set to `false` only when `verify_tee()` determines that **fewer than threshold** participants have valid attestations. [5](#0-4) 

This means: as long as at least `threshold` participants still have valid attestations, `accept_requests` remains `true`, and a participant whose individual attestation has expired or whose image hash has been revoked can still pass `is_caller_an_attested_participant` and submit responses.

---

### Impact Explanation

A participant node whose TEE attestation has expired (or whose MPC image hash was removed from the allowlist via governance) can continue to call `respond`, `respond_ckd`, `respond_verify_foreign_tx`, and key-event votes (`vote_pk`, `vote_reshared`) without any per-call validity check rejecting it. The node may no longer be running approved MPC software inside a genuine TEE enclave, yet it can contribute its share to threshold signature production. If enough such participants accumulate before `verify_tee()` is called and resharing completes, the TEE security guarantee — that every signing participant is running verified, approved code — is violated for live signing operations. This constitutes a participant/attestation authorization bypass that breaks the production safety invariant of the TEE-enforced signing model.

---

### Likelihood Explanation

Attestations have a finite expiry (`DEFAULT_EXPIRATION_DURATION_SECONDS`). The `verify_tee()` call is not automatic; it must be triggered by a participant. The window between attestation expiry and `verify_tee()` being called (and resharing completing) is a realistic, recurring operational gap — especially during high-load periods or when the triggering participant is itself offline. Image hash revocations (via `vote_add_image_hash` governance) create the same window. The scenario requires no privileged access: any participant whose attestation has lapsed is automatically in this state.

---

### Recommendation

`is_caller_an_attested_participant` should call `re_verify` on the retrieved `NodeAttestation` before returning `Ok(())`, using the current block timestamp and the live allowlists, so that an expired or revoked attestation is rejected at the point of each sensitive call — not only when `verify_tee()` happens to be invoked. Alternatively, introduce a per-participant validity flag that is cleared by `reverify_and_cleanup_participants` and checked inside `is_caller_an_attested_participant`.

---

### Proof of Concept

1. Participant `A` submits a valid TEE attestation at time `T`. It is stored in `stored_attestations`.
2. At time `T + DEFAULT_EXPIRATION_DURATION_SECONDS + 1`, the attestation expires. `reverify_participants` would return `TeeQuoteStatus::Invalid` for `A`.
3. No one calls `verify_tee()` yet (or it was called before expiry). `accept_requests` remains `true`.
4. `A` calls `respond(request, response)`. The contract executes `assert_caller_is_attested_participant_and_protocol_active`, which calls `is_caller_an_attested_participant`. This function finds `A`'s entry in `stored_attestations`, confirms `account_id` and `account_public_key` match, and returns `Ok(())` — without ever calling `re_verify`.
5. The `accept_requests` check passes (it is still `true`).
6. `A`'s signature share is accepted and resolves a pending yield, contributing to a threshold signature issued while `A` is no longer in a verified TEE state. [6](#0-5) [7](#0-6)

            &allowed_mpc_docker_image_hashes,
            &allowed_launcher_compose_hashes,
            &allowed_measurements,
        ) {
            Ok(()) => TeeQuoteStatus::Valid,
            Err(err) => TeeQuoteStatus::Invalid(err.to_string()),
        }

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
