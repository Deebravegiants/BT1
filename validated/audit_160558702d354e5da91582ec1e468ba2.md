### Title
Hardcoded `Fee(1)` in L1Handler Conversion Always Produces Wrong `paid_fee_on_l1` for Every L1→L2 Message — (File: `crates/apollo_transaction_converter/src/transaction_converter.rs`)

### Summary
`convert_consensus_l1_handler_to_internal_l1_handler` unconditionally passes `Fee(1)` (1 wei) as `paid_fee_on_l1` when converting every L1Handler transaction from its consensus wire form to the executable form the blockifier runs. The actual fee paid on L1 to trigger the message is never read or forwarded. This is the direct sequencer analog of the Derby exchange-rate bug: a quantity that should track a per-message value always returns a fixed constant, corrupting every downstream fee and resource-accounting calculation that depends on it.

### Finding Description

In `crates/apollo_transaction_converter/src/transaction_converter.rs`, the function `convert_consensus_l1_handler_to_internal_l1_handler` is called from `convert_consensus_tx_to_internal_consensus_tx` for every `ConsensusTransaction::L1Handler` received during proposal validation:

```rust
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),
    )?)
}
``` [1](#0-0) 

`paid_fee_on_l1` is the fee the L1 sender deposited on Ethereum to fund the L1→L2 message. The blockifier uses it as the **maximum fee budget** the L1 handler execution may consume. By hardcoding it to `Fee(1)` (1 wei) for every message, regardless of what was actually paid on L1:

1. Any L1 handler whose execution cost exceeds 1 wei will be charged or rejected incorrectly.
2. The fee refund calculation (`paid_fee_on_l1 − actual_execution_fee`) is wrong for every L1 handler.
3. Bouncer resource accounting for L1 handler transactions receives a corrupted fee value.

The call site that routes every `ConsensusTransaction::L1Handler` through this function is:

```rust
ConsensusTransaction::L1Handler(tx) => {
    let internal_tx = self.convert_consensus_l1_handler_to_internal_l1_handler(tx)?;
    Ok((InternalConsensusTransaction::L1Handler(internal_tx), None))
}
``` [2](#0-1) 

This path is exercised during `validate_proposal` → `handle_proposal_part` → `convert_consensus_tx_to_internal_consensus_tx` for every L1Handler in a validated block. [3](#0-2) 

### Impact Explanation

Every L1Handler transaction that passes through the consensus validation path is executed by the blockifier with `paid_fee_on_l1 = 1 wei`. The blockifier uses this as the fee budget cap. Real L1→L2 messages carry fees orders of magnitude larger (typically thousands of gwei). The result is:

- **Wrong execution outcome**: L1 handlers whose actual gas cost exceeds 1 wei may be incorrectly bounded, producing wrong execution results or reverts that would not occur with the correct fee.
- **Wrong fee/balance accounting**: The fee charged to the L1 sender and any refund computed from `paid_fee_on_l1 − actual_fee` are incorrect for every L1 handler in every block.
- **Wrong receipt/event data**: Execution receipts and events derived from the fee calculation carry the corrupted value, producing an authoritative-looking wrong RPC view.

This matches the allowed impact: *"Critical. Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."*

### Likelihood Explanation

The trigger is unprivileged: any Ethereum account that sends an L1→L2 message to the Starknet core contract causes an L1Handler transaction to be included in a block. The code path is unconditional — there is no branch that reads the real `paid_fee_on_l1` from the wire transaction. Every L1Handler in every block is affected.

### Recommendation

Read the actual fee paid on L1 from the `L1HandlerTransaction` wire struct (it is available in the `paid_fee_on_l1` field of the on-chain event or the transaction itself) and pass it to `executable_transaction::L1HandlerTransaction::create` instead of the hardcoded `Fee(1)`.

### Proof of Concept

1. Submit an L1→L2 message on Ethereum with a fee of, e.g., 10 000 gwei.
2. The sequencer includes the resulting L1Handler in a block.
3. During `validate_proposal`, `convert_consensus_l1_handler_to_internal_l1_handler` is called and produces an executable transaction with `paid_fee_on_l1 = Fee(1)`.
4. The blockifier executes the handler with a 1-wei fee budget instead of 10 000 gwei.
5. The execution receipt, fee deduction, and any refund all reflect the wrong value; the on-chain state and RPC responses carry the corrupted fee data. [1](#0-0)
