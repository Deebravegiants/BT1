Audit Report

## Title
Stale `rsETHPrice` Cache Enables Unprivileged Depositor to Extract Accrued Yield via Sandwich of Public Price Update - (File: contracts/LRTOracle.sol, contracts/LRTDepositPool.sol)

## Summary

`LRTOracle` stores the rsETH/ETH exchange rate in the state variable `rsETHPrice`, which is only updated on explicit calls to the public, permissionless `updateRSETHPrice()`. `LRTDepositPool.getRsETHAmountToMint()` divides the deposit value by this stale cached price, so a depositor who acts before a price update receives more rsETH than the true exchange rate warrants. Because the attacker controls the timing of the price update, they can deposit at the stale lower price, trigger the update, and then initiate a withdrawal at the now-higher price, extracting the difference as profit at the expense of existing rsETH holders.

## Finding Description

**Root cause — stale price read in deposit path:**

`LRTDepositPool.getRsETHAmountToMint()` reads `lrtOracle.rsETHPrice()` directly:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [1](#0-0) 

`rsETHPrice` is a cached state variable that is only written inside `_updateRsETHPrice()`: [2](#0-1) [3](#0-2) 

No call to `updateRSETHPrice()` or `_updateRsETHPrice()` exists anywhere in the deposit path (`depositETH` → `_beforeDeposit` → `getRsETHAmountToMint`). [4](#0-3) [5](#0-4) 

**Root cause — permissionless price update:**

`updateRSETHPrice()` carries no access control:

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [6](#0-5) 

**Root cause — withdrawal uses price at initiation time:**

`initiateWithdrawal()` calls `getExpectedAssetAmount()`, which reads the cached `rsETHPrice` at the moment of the call:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
``` [7](#0-6) 

The resulting `expectedAssetAmount` is stored in the `WithdrawalRequest` struct: [8](#0-7) 

**Root cause — `_calculatePayoutAmount` takes the minimum, not the maximum:**

When the operator calls `unlockQueue`, `_calculatePayoutAmount` pays `min(expectedAssetAmount, currentReturn)`:

```solidity
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
``` [9](#0-8) 

Because the attacker initiates withdrawal *after* calling `updateRSETHPrice()`, the stored `expectedAssetAmount` already reflects the higher post-update price. Since staking rewards cause the price to be monotonically non-decreasing, `currentReturn` at unlock time will be ≥ `expectedAssetAmount`, so the attacker receives the full higher amount.

**Existing guard — `pricePercentageLimit` — is insufficient:**

The guard only fires when `pricePercentageLimit > 0` AND the price increase exceeds the configured threshold. When `pricePercentageLimit == 0` (unset), the condition short-circuits and the guard is entirely bypassed:

```solidity
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
``` [10](#0-9) 

Even when set, any price movement within the threshold is freely exploitable.

**Exploit flow:**

1. Staking rewards accrue; true TVL rises but `rsETHPrice` remains stale at the old lower value.
2. Attacker calls `depositETH{value: X}(...)` — minted rsETH is computed using the stale lower price, so the attacker receives more rsETH than the true rate warrants.
3. Attacker calls `lrtOracle.updateRSETHPrice()` — `rsETHPrice` is now updated to the higher true rate.
4. Attacker calls `initiateWithdrawal(asset, rsETHAmount, ...)` — `expectedAssetAmount` is computed at the new higher price and stored.
5. After `withdrawalDelayBlocks`, the operator's routine `unlockQueue` call processes the request; `_calculatePayoutAmount` returns `expectedAssetAmount` (the higher value).
6. Attacker calls `completeWithdrawal` and receives more assets than deposited.

## Impact Explanation

**High — Theft of unclaimed yield.**

Existing rsETH holders accumulate yield as staking rewards increase the protocol's TVL, reflected in a rising `rsETHPrice`. An attacker who deposits at the stale lower price receives more rsETH than their deposit is worth at the true current rate. After the price update, the attacker's rsETH is worth more than they paid; the difference is extracted directly from the yield that should have accrued to pre-existing holders. This matches the allowed impact "Theft of unclaimed yield."

## Likelihood Explanation

**Medium-High.**

- `updateRSETHPrice()` is public with no access control; the attacker fully controls the timing of the price update.
- Ethereum validator rewards and EigenLayer restaking rewards accrue continuously, so the stale-price window reopens after every reward cycle.
- No special privileges, leaked keys, or external protocol compromise are required — only standard `depositETH`, `updateRSETHPrice`, and `initiateWithdrawal` calls.
- The `pricePercentageLimit` guard is bypassable when unset and only limits large single-step increases; small but profitable movements are always exploitable.
- The only external dependency is the operator's routine `unlockQueue` call, which is expected to run regularly as part of normal protocol operation.

## Recommendation

1. **Atomically refresh the price inside the deposit flow.** Call `_updateRsETHPrice()` (or an equivalent internal live-TVL read) at the start of `depositETH()` / `depositAsset()` before computing `rsethAmountToMint`. This eliminates the stale-price window entirely.

2. **Alternatively, compute the mint amount from live TVL directly** rather than from the cached `rsETHPrice` state variable, so the deposit always uses the real-time exchange rate.

3. **Restrict `updateRSETHPrice()`** to trusted callers (e.g., a keeper or the manager role) so that the timing of price updates cannot be weaponized by an unprivileged attacker.

## Proof of Concept

**Setup:**
- Protocol TVL = 1000 ETH, rsETH supply = 1000, `rsETHPrice` = 1.000 ETH/rsETH.
- Staking rewards of 10 ETH accrue → true TVL = 1010 ETH, true price = 1.010 ETH/rsETH.
- `rsETHPrice` is still cached at 1.000 (not yet updated).

**Attack steps:**

```solidity
// Step 1: Deposit 100 ETH at stale price 1.000
// getRsETHAmountToMint = (100e18 * 1e18) / 1.000e18 = 100 rsETH
// At true price 1.010, attacker should only receive ~99.01 rsETH
lrtDepositPool.depositETH{value: 100 ether}(0, "");

// Step 2: Trigger the public price update
// New TVL = 1110 ETH, new supply = 1100 rsETH
// newRsETHPrice = 1110/1100 ≈ 1.00909 ETH/rsETH
lrtOracle.updateRSETHPrice();

// Step 3: Initiate withdrawal of 100 rsETH at the new price
// expectedAssetAmount = 100 * 1.00909 / 1.0 ≈ 100.909 ETH  (stored in WithdrawalRequest)
lrtWithdrawalManager.initiateWithdrawal(ETH_TOKEN, 100e18, "");

// Step 4: After withdrawalDelayBlocks, operator calls unlockQueue (routine operation)
// _calculatePayoutAmount: currentReturn ≈ 100.909, expectedAssetAmount ≈ 100.909
// min(100.909, 100.909) = 100.909 ETH → attacker receives 100.909 ETH

// Net profit: ~0.909 ETH extracted from existing holders' accrued yield
// Repeatable every reward cycle
```

**Foundry fork test plan:**
1. Fork mainnet at a block where rewards have accrued since the last `updateRSETHPrice` call.
2. Impersonate an unprivileged EOA.
3. Call `depositETH` with a meaningful ETH value.
4. Call `lrtOracle.updateRSETHPrice()`.
5. Call `initiateWithdrawal` for the full rsETH balance.
6. Roll forward `withdrawalDelayBlocks` blocks.
7. Impersonate the operator and call `unlockQueue`.
8. Call `completeWithdrawal` and assert that the received ETH exceeds the deposited ETH.
