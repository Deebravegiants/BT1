### Title
Unauthenticated Mirror Index Poisoning via Arbitrary Memo in Mirror-Puzzle-Hash Coin — (`chia/data_layer/data_layer_wallet.py`)

### Summary

`DataLayerWallet.coin_added` blindly trusts `memos[0]` from any parent spend that creates a coin at the mirror puzzle hash as the authoritative `launcher_id`. An unprivileged attacker who holds any XCH can craft a standard wallet spend that creates a mirror-puzzle-hash output with `memos[0]` set to a known tracked `launcher_id` and `memos[1]` set to an attacker-controlled URL. When the block is confirmed, the victim's wallet processes the coin, calls `get_mirror_info`, and inserts the fake mirror record into `dl_store`, associating the attacker's URL with the victim's tracked DL store. The DataLayer service subsequently syncs peer URLs from this corrupted index.

### Finding Description

**Step 1 — Victim subscribes to `launcher_id` as a hint.**

When the victim tracks a DL launcher, `_track_new_launcher_id` calls:

```python
await self.wallet_state_manager.add_interested_puzzle_hashes([launcher_id], [self.id()])
``` [1](#0-0) 

This causes the full node to deliver any coin whose hint (`memos[0]`) equals `launcher_id` to the victim's wallet.

**Step 2 — Attacker crafts a standard wallet spend.**

The attacker submits a standard wallet spend bundle that creates a coin with:
- `puzzle_hash = create_mirror_puzzle().get_tree_hash()` (the mirror puzzle hash — a public, well-known value)
- `memos = [tracked_launcher_id, b'http://evil.example.com']`

No special permissions are required. Any XCH holder can do this.

**Step 3 — `coin_added` is triggered.**

### Impact Explanation

The wallet's mirror index is corrupted: `dl_store.mirrors` now contains a record associating the attacker's URL with the victim's tracked `launcher_id`. The DataLayer service will attempt to sync data for that store from the attacker-controlled server. This causes:

1. **Data sync disruption**: The DataLayer service may preferentially contact the attacker's server, missing legitimate data updates from real mirrors.
2. **Information disclosure**: The attacker's server learns which store IDs the victim is syncing, what generations are requested, and timing of sync activity.
3. **Persistent corruption**: The fake mirror record persists until the attacker's coin is spent (which only the attacker controls), or the victim manually removes it.

This matches the High impact category: *"Corruption of... Data Layer root/store state with direct security impact."*

### Likelihood Explanation

- Tracked `launcher_id` values are public on-chain.
- The attack requires only a small amount of XCH to pay transaction fees.
- No special access, leaked keys, or privileged position is needed.
- The attack is silent — the victim's wallet logs no warning about the fake mirror.

### Recommendation

In `coin_added`, before calling `add_mirror`, verify that the parent spend is actually a DL singleton spend (or a known legitimate mirror-creating spend) by checking that the parent coin's puzzle hash matches a known DL singleton puzzle hash pattern. Alternatively, validate that `memos[0]` corresponds to a launcher coin that actually exists on-chain and is a valid DL singleton launcher before accepting it as a `launcher_id`. At minimum, `get_mirror_info` should not be called on arbitrary parent spends without first confirming the parent is a DL-related spend.
