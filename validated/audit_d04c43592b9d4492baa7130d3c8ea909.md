### Title
UTXO Fast-Path Near Settlement Omits `finalised_utxo_transfers` Guard, Enabling Double-Credit — (`near/omni-bridge/src/lib.rs`)

---

### Summary

`utxo_fin_transfer_fast` handles the Near-destination fast-path by calling `remove_fast_transfer`, which **deletes** the entry from `fast_transfers` but never inserts the UTXO ID into `finalised_utxo_transfers`. A subsequent call to `utxo_fin_transfer` with the same UTXO ID finds no fast-transfer entry and no finalization record, so it passes `add_fin_utxo_transfer` and pays the recipient a second time.

---

### Finding Description

**Step 1 — Normal path inserts into `finalised_utxo_transfers`.**

When no fast transfer exists, `utxo_fin_transfer` calls `add_fin_utxo_transfer` before paying the recipient: [1](#0-0) 

`add_fin_utxo_transfer` enforces uniqueness via a `require!` on `LookupSet::insert`: [2](#0-1) 

**Step 2 — Fast-path Near branch calls `remove_fast_transfer`, not `add_fin_utxo_transfer`.**

Inside `utxo_fin_transfer_fast`, the Near-destination branch removes the entry from `fast_transfers` and returns immediately. It never touches `finalised_utxo_transfers`: [3](#0-2) 

`remove_fast_transfer` only removes from `fast_transfers` and refunds storage — no write to `finalised_utxo_transfers`: [4](#0-3) 

**Step 3 — Second submission succeeds.**

After the fast-path call, `fast_transfers` no longer contains the UTXO ID and `finalised_utxo_transfers` is still empty for it. A second `utxo_fin_transfer` call with the same `UtxoFinTransferMsg`:

- `get_fast_transfer_status` returns `None` (entry was removed) → skips fast path
- `add_fin_utxo_transfer` succeeds (UTXO ID absent from `finalised_utxo_transfers`)
- `utxo_fin_transfer_to_near` is called → recipient receives tokens again [5](#0-4) 

**Step 4 — Contrast with non-Near fast path (correctly protected).**

For non-Near destinations, `mark_fast_transfer_as_finalised` is called instead, leaving the entry in `fast_transfers` with `finalised = true`. A second call hits the `require!(!fast_transfer_status.finalised, ...)` guard and reverts. This asymmetry is why the existing `fails_on_double_finalization` test (which uses `base_eoa_address()`) passes, while the Near-destination case is unguarded: [6](#0-5) 

**Step 5 — Connector entry point is public.**

The only UTXO connector in the repository, `MockUtxoConnector`, exposes `verify_deposit` with no access control — any account can call it: [7](#0-6) 

The bridge only checks `sender_id == config.connector`; it does not restrict who may call the connector itself: [8](#0-7) 

---

### Impact Explanation

An attacker who can call `verify_deposit` on the connector twice with the same `UtxoFinTransferMsg` (same `utxo_id`, Near recipient) causes the bridge to credit the recipient twice from its locked token pool. The first call reimburses the fast-transfer relayer; the second call pays the recipient in full again. This is an unauthorized release of locked UTXO-backed tokens — unbacked supply is created.

---

### Likelihood Explanation

The mock connector (the only connector present in the repository) has no replay protection. The bridge contract itself is the intended authoritative replay guard via `finalised_utxo_transfers`, but that guard is bypassed for the Near fast-path. Any caller of `verify_deposit` can trigger this with two identical submissions after a fast transfer has been pre-registered.

---

### Recommendation

In `utxo_fin_transfer_fast`, after `remove_fast_transfer` for the Near-destination branch, also insert the UTXO transfer ID into `finalised_utxo_transfers`:

```rust
// Near destination fast path
self.remove_fast_transfer(&fast_transfer.id());
// ADD: record finalization so re-submission is rejected
self.add_fin_utxo_transfer(&utxo_fin_transfer_msg.get_transfer_id(origin_chain));
```

This mirrors the invariant enforced by the normal (non-fast) path and closes the gap that the non-Near branch avoids via `mark_fast_transfer_as_finalised`.

---

### Proof of Concept

```
1. Relayer calls fast_fin_transfer (ft_transfer_call → bridge) for UTXO ID X, Near recipient R.
   → fast_transfers[X] = { relayer, finalised: false }

2. Connector calls verify_deposit(amount, msg{utxo_id: X, recipient: Near(R)})
   → ft_transfer_call → bridge.utxo_fin_transfer
   → get_fast_transfer_status(X) = Some(status)
   → utxo_fin_transfer_fast: remove_fast_transfer(X)   // fast_transfers[X] deleted
   → send_tokens(relayer, amount)                       // relayer reimbursed
   → finalised_utxo_transfers: EMPTY for X

3. Connector calls verify_deposit(amount, msg{utxo_id: X, recipient: Near(R)}) again
   → ft_transfer_call → bridge.utxo_fin_transfer
   → get_fast_transfer_status(X) = None                // entry was removed
   → add_fin_utxo_transfer(X) succeeds                 // X not in set
   → utxo_fin_transfer_to_near(R, amount)              // recipient paid AGAIN

Assert: recipient balance == 2 × amount  (double-credit)
```

    fn remove_fast_transfer(&mut self, fast_transfer_id: &FastTransferId) {
        let storage_usage = env::storage_usage();
        let fast_transfer = self
            .fast_transfers
            .remove(fast_transfer_id)
            .map(storage::FastTransferStatusStorage::into_main)
            .near_expect(BridgeError::TransferNotExist);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(&fast_transfer.storage_owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances
                .insert(&fast_transfer.storage_owner, &storage);
        }
    }
