### Title
Stale Chainlink Price Accepted in `ChainlinkPriceOracle.getAssetPrice()` Enables Over-Minting of rsETH at Depositors' Expense - (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all staleness-related return values (`updatedAt`, `answeredInRound`). When a Chainlink LST/ETH feed is stale and the actual market price has dropped below the reported price, any depositor can call `LRTDepositPool.depositAsset()` and receive more rsETH than the deposited asset is actually worth, diluting all existing rsETH holders.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the Chainlink price but only destructures the `price` field:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

Neither `updatedAt` (timestamp of last update) nor `answeredInRound` (round completeness) is checked. [1](#0-0) 

This price is consumed directly by `LRTDepositPool.getRsETHAmountToMint()`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [2](#0-1) 

`depositAsset()` calls `_beforeDeposit()` → `getRsETHAmountToMint()` and then mints rsETH to the depositor: [3](#0-2) 

The same stale price also feeds `LRTOracle._getTotalEthInProtocol()`, which inflates the computed TVL and therefore the rsETH price used as the denominator in the mint calculation: [4](#0-3) 

By contrast, the pool-side oracle wrapper `ChainlinkOracleForRSETHPoolCollateral` does check `answeredInRound < roundID`, demonstrating the team is aware of the pattern but did not apply it to the core deposit oracle: [5](#0-4) 

---

### Impact Explanation

When a Chainlink LST/ETH feed is stale and the actual price has fallen below the reported price, a depositor receives rsETH computed against the inflated stale price. The excess rsETH is backed by less ETH value than the protocol accounts for. When `updateRSETHPrice()` is next called with the corrected price, the rsETH price drops, socialising the loss across all existing holders. This constitutes theft of yield from existing rsETH holders.

**Impact: High — Theft of unclaimed yield / dilution of existing rsETH holders.**

---

### Likelihood Explanation

Chainlink feeds can go stale during network congestion, oracle node outages, or when the price has not moved enough to trigger a deviation update. LST/ETH feeds (e.g., stETH/ETH) are generally stable but are not immune. The attack requires no special privileges — any depositor can trigger it the moment a stale feed is detected. The combination of staleness and a meaningful price drop is low-probability but not negligible, and the absence of any on-chain guard means the window is fully exploitable whenever it occurs.

**Likelihood: Low.**

---

### Recommendation

Add staleness and round-completeness checks to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    require(answeredInRound >= roundId, "ChainlinkPriceOracle: stale price");
    require(updatedAt != 0, "ChainlinkPriceOracle: incomplete round");
    require(price > 0, "ChainlinkPriceOracle: invalid price");
    require(block.timestamp - updatedAt <= MAX_STALENESS, "ChainlinkPriceOracle: price too old");

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

A configurable `MAX_STALENESS` per asset (e.g., 3600 seconds for ETH/USD, 86400 seconds for LST/ETH) should be set by the LRT manager.

---

### Proof of Concept

1. Assume stETH/ETH Chainlink feed last updated 2 hours ago at `1.05 ETH/stETH`. Actual market price has since dropped to `0.95 ETH/stETH` (e.g., slashing event).
2. `rsETHPrice` is currently `1.04 ETH/rsETH` (last oracle update).
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
4. `getRsETHAmountToMint` computes: `1000e18 * 1.05e18 / 1.04e18 ≈ 1009.6 rsETH`.
5. True value of 1000 stETH at market price: `1000 * 0.95 = 950 ETH`.
6. True rsETH equivalent: `950 / 1.04 ≈ 913.5 rsETH`.
7. Attacker receives `≈ 96 rsETH` in excess, each backed by no real ETH value.
8. When `updateRSETHPrice()` is called with the corrected stETH price, `totalETHInProtocol` drops, rsETH price falls, and all existing holders absorb the loss. [1](#0-0) [2](#0-1)
