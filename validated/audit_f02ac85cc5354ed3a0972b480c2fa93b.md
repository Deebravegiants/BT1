### Title
Unvalidated `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Response Replay by a Single Byzantine Participant - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` accepts a caller-supplied `response.payload_hash` and only verifies that the signature is valid over that hash. It never checks that `response.payload_hash` is actually derived from the `request` parameter that is used as the pending-request map key. A single malicious attested participant (below threshold) can replay a valid `(payload_hash, signature)` pair from any previously completed foreign-tx request against a different pending request, permanently consuming that request and delivering a forged response to every caller waiting on it.

### Finding Description

In `respond_verify_foreign_tx`, the signature validity check reads `payload_hash` from the **response** rather than recomputing it from the **request**:

```rust
// crates/contract/src/lib.rs  lines 718-734
let payload_hash: [u8; 32] = response.payload_hash.0;   // ← caller-supplied
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,          // verified against THIS, not against request
    &secp_pk,
).is_ok()
``` [1](#0-0) 

Contrast this with `respond` (regular signatures), where the hash is taken directly from the stored request payload — a value the caller cannot alter:

```rust
// crates/contract/src/lib.rs  lines 600-608
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,           // ← taken from request, not from response
    &expected_public_key,
).is_ok()
``` [2](#0-1) 

The correct hash for a foreign-tx response is `SHA-256(borsh(ForeignTxSignPayload { request, values }))`, where `values` are the extracted on-chain values. The contract stores only the `request` as the map key; it never stores or recomputes the expected `payload_hash`. Because `payload_hash` is fully attacker-controlled in the response struct, the only binding between `request` and `response` is the map lookup — which succeeds as long as the `request` key exists, regardless of what `payload_hash` is supplied. [3](#0-2) 

The `VerifyForeignTransactionResponse` type carries both fields independently: [4](#0-3) 

### Impact Explanation

**Medium → High.** A single Byzantine participant (below signing threshold) can:

1. Observe any previously completed `respond_verify_foreign_tx` call on-chain and extract its `(payload_hash_B, signature_B)` pair — both are public.
2. Identify a currently pending request `request_A` (a different foreign-chain transaction).
3. Call `respond_verify_foreign_tx(request_A, { payload_hash_B, signature_B })`.
4. The contract passes all checks (valid attested participant, valid signature over `payload_hash_B`) and calls `resolve_yields_for` on `request_A`, permanently draining every yield queued under it. [5](#0-4) 

Every caller whose yield was queued under `request_A` now receives `response_B`. Callers using `ForeignChainSignatureVerifier::verify_signature` from the SDK will detect the mismatch between `payload_hash_B` and the hash they expect for `request_A`: [6](#0-5) 

However, the contract has already irrevocably consumed the pending-request entry. Callers must resubmit and pay the deposit again, and the malicious participant can repeat the attack indefinitely. Callers that do not use the SDK's verifier (e.g., custom bridge integrations that only check the ECDSA signature) will silently accept the forged `payload_hash`, enabling invalid bridge execution — a forged foreign-chain verification.

### Likelihood Explanation

The attacker must be a single attested MPC participant (below threshold). Attestation is a TEE-based check, but the attack requires only one compromised or malicious node. The required `(payload_hash, signature)` material is freely available from any prior on-chain `respond_verify_foreign_tx` call. No key material, threshold collusion, or network-level access is needed beyond holding a valid participant slot.

### Recommendation

**Short term:** Derive `payload_hash` from the `request` inside the contract rather than accepting it from the caller. Since the contract does not know the extracted `values`, the nodes should include them in the response so the contract can recompute and verify the hash:

```rust
// Pseudocode
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.values.clone(),   // nodes must supply values
}).compute_msg_hash()?;
require!(expected_hash == response.payload_hash, "payload_hash mismatch");
```

**Long term:** Add an invariant test asserting that `respond_verify_foreign_tx` rejects a response whose `payload_hash` does not correspond to the submitted `request`. Document the binding requirement between `request` and `payload_hash` in the contract API.

### Proof of Concept

1. Alice submits `verify_foreign_transaction(request_A)` (e.g., Bitcoin tx_id `[0xAA; 32]`). The contract queues a yield under `request_A`.
2. An honest node previously completed `respond_verify_foreign_tx(request_B, response_B)` for a different transaction (tx_id `[0xBB; 32]`). `response_B = { payload_hash_B, signature_B }` is visible on-chain.
3. Malicious participant Eve (a single attested node) calls:
   ```
   respond_verify_foreign_tx(request_A, response_B)
   ```
4. The contract checks:
   - Eve is an attested participant ✓
   - `request_A` is in `pending_verify_foreign_tx_requests` ✓
   - `verify_ecdsa_signature(signature_B, payload_hash_B, root_pk)` ✓ (valid for request_B)
   - **No check that `payload_hash_B` is derived from `request_A`** ✗
5. `resolve_yields_for` drains all yields under `request_A`, delivering `response_B` to Alice.
6. Alice's transaction receives `payload_hash_B` (the hash of tx_id `[0xBB; 32]`'s data), not the hash of her tx_id `[0xAA; 32]`. Her request is permanently consumed. If Alice's bridge contract does not independently verify `payload_hash`, it accepts a proof for the wrong transaction. [7](#0-6)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
