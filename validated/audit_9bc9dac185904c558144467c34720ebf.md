### Title
Presignature Consumed Before Foreign Chain Verification Succeeds, Causing Permanent Loss on Failure - (File: crates/node/src/providers/verify_foreign_tx/sign.rs)

### Summary
In `make_verify_foreign_tx_leader`, the leader node irrevocably removes a presignature from its store **before** executing the foreign chain RPC verification. If `execute_foreign_chain_request` fails for any reason (RPC timeout, chain temporarily unavailable, transaction not found, unsupported chain), the presignature is permanently lost with no recovery path. Any unprivileged caller can trigger this by submitting `verify_foreign_transaction` requests that are expected to fail verification.

### Finding Description

In `crates/node/src/providers/verify_foreign_tx/sign.rs`, `make_verify_foreign_tx_leader` executes in this order:

```rust
// Step 1: presignature is IRREVOCABLY removed from the store
let (presignature_id, presignature) = domain_data.presignature_store.take_owned().await;
let participants = presignature.participants.clone();
let channel = self.ecdsa_signature_provider.new_channel_for_task(...)?;

// Step 2: foreign chain RPC verification — can fail for many reasons
let response_payload = self
    .execute_foreign_chain_request(
        &foreign_tx_request.request,
        foreign_tx_request.payload_version,
    )
    .await?;   // <-- if this returns Err, `presignature` is dropped and gone
``` [1](#0-0) 

`take_owned()` removes the presignature from the persistent store. The local variable `presignature` is never returned to the store on error. `execute_foreign_chain_request` can fail due to:
- A 5-second hard timeout (`FOREIGN_CHAIN_INSPECTION_TIMEOUT`) on any RPC call
- An RPC provider being unreachable or returning an unexpected response
- The transaction not existing or not meeting finality requirements
- The chain being removed from the supported set between request submission and processing [2](#0-1) 

By contrast, the **follower** path does not have this problem: it calls `execute_foreign_chain_request` first, and only consumes the presignature inside `make_signature_follower_given_request` → `FollowerSignComputation::compute` → `take_unowned`, which is called after verification succeeds. [3](#0-2) 

The regular ECDSA `make_signature_leader` path does not have this issue either, because there is no fallible intermediate step between `take_owned()` and the signing computation itself. [4](#0-3) 

### Impact Explanation

Presignatures are expensive cryptographic artifacts: each one requires two Beaver triples (themselves requiring OT extension) plus a multi-round presignature protocol across all participants. The presignature store is finite and shared between regular `sign()` requests and `verify_foreign_transaction` requests. [5](#0-4) 

When the leader's presignature pool is drained:
- All pending `sign()` and `verify_foreign_transaction` requests on that leader stall until new presignatures are generated
- Pending sign requests time out and their on-chain yield callbacks fire with failure, permanently failing those user requests
- The contract's `return_signature_and_clean_state_on_success` callback handles the timeout by calling `fail_on_timeout`, meaning the user's request is permanently dropped [6](#0-5) 

This breaks the production safety invariant that a presignature is only consumed when a signing operation is successfully completed. It maps to the **Medium** allowed impact: *request-lifecycle manipulation that breaks production safety/accounting invariants*.

### Likelihood Explanation

The entry point is the public `verify_foreign_transaction` contract method, callable by any NEAR account with a 1 yoctoNEAR deposit. [7](#0-6) 

An attacker submits many `verify_foreign_transaction` requests pointing to a transaction ID that does not exist on the target chain, or to a chain whose RPC providers are temporarily slow. Each request causes the leader to:
1. Dequeue a presignature
2. Attempt the RPC call (times out after 5 seconds)
3. Return `Err`, dropping the presignature permanently

The cost per wasted presignature is 1 yoctoNEAR. The presignature buffer size is a small fixed number (`desired_presignatures_to_buffer`), so a modest number of requests suffices to exhaust it. [8](#0-7) 

### Recommendation

Move `take_owned()` to **after** `execute_foreign_chain_request` succeeds. The corrected order in `make_verify_foreign_tx_leader` should be:

1. Fetch the foreign tx request from the store.
2. Execute and verify the foreign chain request (fallible).
3. Build the sign request payload.
4. Only then call `presignature_store.take_owned()`.
5. Open the P2P channel and proceed with signing.

This mirrors the invariant already correctly maintained in the follower path and in the regular ECDSA leader path.

### Proof of Concept

```
1. Attacker calls verify_foreign_transaction(request={chain: Bitcoin, tx_id: [0xff;32], ...})
   with 1 yoctoNEAR deposit. tx_id does not exist on Bitcoin mainnet.

2. Contract enqueues the request (passes all on-chain checks — chain is supported,
   domain exists, deposit is sufficient).

3. Leader node picks up the request in make_verify_foreign_tx_leader:
   a. presignature_store.take_owned() → presignature P is removed from the DB.
   b. execute_foreign_chain_request() → Bitcoin RPC returns "tx not found" or times
      out after 5 s.
   c. Function returns Err. `presignature` (P) is dropped. P is gone forever.

4. Attacker repeats N times (N = desired_presignatures_to_buffer).
   Leader's presignature pool is now empty.

5. Any legitimate sign() or verify_foreign_transaction() request assigned to this
   leader now blocks indefinitely waiting for take_owned() to return a new presignature,
   which takes significant time to regenerate (triple generation is the bottleneck).
   Pending requests time out on-chain and are permanently failed.
``` [9](#0-8)
