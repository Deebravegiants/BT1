### Title
Byzantine Representative Bypasses `h_all` Consistency Check via Wrong Reference Value in `dkg_or_refresh_ac` — (`src/cbmpc/protocol/ec_dkg.cpp`)

### Summary

In `key_share_mp_t::dkg_or_refresh_ac`, the `h_all` consistency check at line 377 compares each received `h_all` value against `h_all.received(representative_quorum_pid_index)` — the value received *from* the representative — rather than against `h_all.msg`, the locally computed value. A Byzantine representative can exploit this by sending different `c_all` commitment hashes to different honest parties in round 1, then sending crafted `h_all` values in round 2 that make the check trivially pass. Both honest parties proceed with different views of the commitment set, accept different `Rs` openings, and compute inconsistent key shares.

### Finding Description

**Protocol flow in `dkg_or_refresh_ac`:**

1. Each quorum party broadcasts its commitment hash `c_all` (line 358).
2. Each party locally computes `h_all._i = sha256(all_received_c_s, quorum_pids, sid)` (line 367).
3. Each party broadcasts its `h_all._i` (line 369).
4. The consistency check (line 377):

```cpp
if (h_all._j != h_all.received(representative_quorum_pid_index))
    return coinbase::error(E_CRYPTO, "h_all mismatch");
``` [1](#0-0) 

**The `received()` semantics** (from `uniform_msg_t`):

```cpp
T& received(int index) {
    return (index == job->get_party_idx()) ? msg : inbox[index];
}
``` [2](#0-1) 

When the executing party `i` is **not** the representative, `h_all.received(representative_quorum_pid_index)` resolves to `inbox[representative_quorum_pid_index]` — the value received *from* the Byzantine representative — not the locally computed `h_all.msg`. The check therefore only verifies that all received values agree with the representative's claimed value, never that any received value matches the local computation.

When `j == representative_quorum_pid_index` in the loop, the check degenerates to `inbox[rep] != inbox[rep]`, which is always false — the representative's own value is never validated against anything.

**Contrast with the correct pattern** used elsewhere in the codebase:

- `schnorr_mp.cpp`: `if (h._j != h._i) return coinbase::error(E_CRYPTO);` [3](#0-2) 
- `ecdsa_mp.cpp`: `if (h_gen._j != h_gen.msg) return coinbase::error(E_CRYPTO);` [4](#0-3) 

Both compare against the locally computed value (`h._i` / `h_gen.msg`). The `dkg_or_refresh_ac` check does not.

**`representative_quorum_pid_index` is always the last quorum party** (by index), set unconditionally in the setup loop:

```cpp
for (int j = 0; j < n; j++) {
    ...
    if (quorum_party_set.has(j)) {
        ...
        representative_quorum_pid_index = j;
    }
}

### Impact Explanation

A Byzantine representative (the highest-indexed quorum party) can equivocate on its commitment hash `c_all` in round 1, sending `c_all_A` (committing to `Rs_A`) to party 0 and `c_all_B` (committing to `Rs_B`) to party 1. Honest parties then compute different `h_all` values. In round 2, the representative sends `h_all_1` to party 0 and `h_all_0` to party 1. Both parties' checks pass because each sees all received values agreeing with the representative's value. The representative then opens `c_all_A` to party 0 and `c_all_B` to party 1 with valid ZK proofs for each. Both commitment opens and ZK verifications succeed. The result is that party 0 and party 1 accumulate different `ac_pub_all` contributions from the representative, compute different `Qis` and different `x_i` shares, and the DKG outputs an inconsistent key: the shares held by honest parties do not correspond to any single consistent secret.

This satisfies the "High" impact criterion: a public API reachable validation bypass in DKG creates accepted invalid cryptographic output (inconsistent key shares across honest parties).

### Likelihood Explanation

The attack requires only that the highest-indexed quorum party is Byzantine — a single below-threshold Byzantine participant. No threshold collusion is needed. The manipulation is purely at the network/message layer (sending different bytes to different parties), which is the standard Byzantine adversary model for MPC protocols. The attack is deterministic and requires no brute force.

### Recommendation

Replace the reference value in the `h_all` check from the representative's received value to the locally computed value, matching the correct pattern used in `schnorr_mp.cpp` and `ecdsa_mp.cpp`:

```cpp
// Current (incorrect):
if (h_all._j != h_all.received(representative_quorum_pid_index))
    return coinbase::error(E_CRYPTO, "h_all mismatch");

// Correct:
if (h_all._j != h_all._i)
    return coinbase::error(E_CRYPTO, "h_all mismatch");
```

This ensures every received `h_all` is compared against the locally computed hash of the locally received `c_all` values, making equivocation on `c_all` detectable regardless of which party is the representative.

### Proof of Concept

Simulated 3-party quorum (parties 0, 1, 2; representative = party 2):

**Round 1 — `c_all` broadcast:**
- Party 2 creates two valid commitments: `com_A` (to `Rs_A`, `pi_A`) and `com_B` (to `Rs_B`, `pi_B`).
- Party 2 sends `c_all_A = com_A.msg` to party 0; `c_all_B = com_B.msg` to party 1.

**Local `h_all` computation:**
- Party 0: `h_all_0 = sha256({0:c_0, 1:c_1, 2:c_all_A}, quorum_pids, sid)`
- Party 1: `h_all_1 = sha256({0:c_0, 1:c_1, 2:c_all_B}, quorum_pids, sid)`
- `h_all_0 ≠ h_all_1` (different `c_all` from party 2).

**Round 2 — `h_all` broadcast:**
- Party 0 broadcasts `h_all_0`; party 1 broadcasts `h_all_1`.
- Party 2 sends `h_all_1` to party 0 and `h_all_0` to party 1.

**Check at party 0 (i=0):**
- j=1: `received(1)=h_all_1` vs `received(2)=h_all_1` → equal, no error ✓
- j=2: `received(2)=h_all_1` vs `received(2)=h_all_1` → equal, no error ✓
- Party 0 proceeds. Its local `h_all.msg = h_all_0` was never compared.

**Check at party 1 (i=1):**
- j=0: `received(0)=h_all_0` vs `received(2)=h_all_0` → equal, no error ✓
- j=2: `received(2)=h_all_0` vs `received(2)=h_all_0` → equal, no error ✓
- Party 1 proceeds. Its local `h_all.msg = h_all_1` was never compared.

**Round 2 openings:**
- Party 2 sends `(rho_A, Rs_A, pi_A)` to party 0 → `com_A.open(Rs_A)` succeeds, `pi_A.verify(Rs_A)` succeeds.
- Party 2 sends `(rho_B, Rs_B, pi_B)` to party 1 → `com_B.open(Rs_B)` succeeds, `pi_B.verify(Rs_B)` succeeds.

**Result:** Party 0 accumulates `Rs_A` from party 2; party 1 accumulates `Rs_B`. Their final `Qis` and `x_i` shares are inconsistent — they correspond to different keys. The DKG completes without error on both honest parties, but the output key material is corrupted.
