### Title
Single Participant Can Freeze All `verify_foreign_transaction` Requests via Strict-Intersection Chain Support Check - (`File: crates/contract/src/lib.rs`)

---

### Summary

`verify_foreign_transaction` gates on `get_supported_foreign_chains()`, which requires **every** active participant to have registered a chain. A single Byzantine participant (strictly below the signing threshold) can register an empty chain-support list, causing the strict-intersection check to return an empty set and permanently rejecting all foreign-chain verification requests — even for chains that a signing-threshold-worth of honest participants still cover. The correct threshold-based check (`get_available_foreign_chains()`) exists in the contract but is never consulted by `verify_foreign_transaction`.

---

### Finding Description

`verify_foreign_transaction` performs the following chain-availability check:

```rust
let requested_chain = request.request.chain();
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) {
    env::panic_str(
        &InvalidParameters::ForeignChainNotSupported { requested: requested_chain }.to_string(),
    );
}
``` [1](#0-0) 

`get_supported_foreign_chains()` computes the **strict intersection** of all active participants' registered chains:

```rust
let all_active_nodes_supports_chain =
    nodes_supporting_chain.is_superset(&active_participant_account_ids);
if all_active_nodes_supports_chain { Some(foreign_chain) } else { None }
``` [2](#0-1) 

If **any single** active participant registers an empty set via `register_foreign_chain_support`, the superset check fails for every chain, and `get_supported_foreign_chains()` returns an empty set. The function returns false (panics) without ever consulting `get_available_foreign_chains()`, which uses the correct threshold-based rule: a chain is available if ≥ `signing_threshold` participants cover it. [3](#0-2) 

The contract already contains the correct threshold-based implementation in `update_available_chains_config_cache`:

```rust
.filter_map(|(chain, count)| (count >= threshold).then_some(chain))
``` [4](#0-3) 

But `verify_foreign_transaction` never calls `get_available_foreign_chains()`. The design document explicitly acknowledges this as a known defect: *"A single node that registers an empty list (or hasn't registered yet) drops every chain — one operator can take the whole feature down."* [5](#0-4) 

---

### Impact Explanation

A single Byzantine participant (strictly below the signing threshold, so threshold-based security is not violated) calls `register_foreign_chain_support` with an empty `SupportedForeignChains` set. After this single on-chain transaction, `get_supported_foreign_chains()` returns an empty set for every chain. Every subsequent call to `verify_foreign_transaction` — regardless of which chain is requested — panics with `ForeignChainNotSupported`. All pending and future foreign-chain verification requests are permanently frozen until the Byzantine participant re-registers a non-empty set or is removed via resharing. This breaks the request-lifecycle invariant: requests for chains that a signing-threshold-worth of honest participants cover are rejected at the contract gate, preventing any bridge inbound flow or foreign-chain attestation from completing.

**Impact class:** Medium — request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants without requiring network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

`register_foreign_chain_support` is callable by any active participant with a single direct NEAR transaction (no special privilege beyond being a participant):

```rust
pub fn register_foreign_chain_support(
    &mut self,
    foreign_chain_support: dtos::SupportedForeignChains,
) -> Result<(), Error> {
    let account_id = self.voter_or_panic();
    self.node_foreign_chain_support
        .foreign_chain_support_by_node
        .insert(account_id, foreign_chain_support);
    Ok(())
}
``` [6](#0-5) 

The threshold is designed to tolerate up to `n − signing_threshold` Byzantine participants. A network with `n=4, threshold=3` tolerates one Byzantine node — yet that single node can freeze the entire foreign-chain feature. The attack requires no collusion, no key material, and no special tooling: one transaction with an empty JSON array.

---

### Recommendation

Replace the `get_supported_foreign_chains()` call inside `verify_foreign_transaction` with `get_available_foreign_chains()`, which already implements the correct threshold-based rule. The design document (issue #3434) already proposes this migration:

```rust
// Before (strict intersection — broken):
let supported_chains = self.get_supported_foreign_chains();

// After (threshold-based — correct):
let supported_chains = self.get_available_foreign_chains();
``` [1](#0-0) 

The `get_available_foreign_chains()` cache is already maintained correctly by `recompute_available_foreign_chains()` and `update_available_chains_config_cache()`, so no additional computation is needed. [7](#0-6) 

---

### Proof of Concept

Setup: 4-participant network, signing threshold = 3. All 4 participants have registered Bitcoin support. Bitcoin is available.

1. Byzantine participant (participant 4, below threshold) calls:
   ```json
   register_foreign_chain_support({ "foreign_chain_support": [] })
   ```
2. `get_supported_foreign_chains()` now computes: `{Bitcoin: {p1,p2,p3}}.is_superset({p1,p2,p3,p4})` → `false`. Returns `{}`.
3. Any user calls `verify_foreign_transaction` for Bitcoin.
4. Contract executes: `supported_chains.contains(&Bitcoin)` → `false` → panics with `ForeignChainNotSupported`.
5. `get_available_foreign_chains()` would return `{Bitcoin}` (3 ≥ threshold=3 participants cover it), but it is never consulted.

The existing unit test `get_supported_foreign_chains__should_exclude_chains_not_supported_by_all` confirms this behavior is the current production logic. [8](#0-7)

    pub(crate) fn update_available_chains_config_cache(
        &mut self,
        active_tls_keys: &BTreeSet<dtos::Ed25519PublicKey>,
        threshold: u64,
    ) {
        let mut chain_to_supporter_count: std::collections::BTreeMap<dtos::ForeignChain, u64> =
            std::collections::BTreeMap::new();
        for tls_key in active_tls_keys {
            let Some(chains) = self.foreign_chains_configs.get(tls_key) else {
                continue;
            };
            for chain in chains.iter() {
                if self.rpc_whitelist.entries.is_whitelisted(chain) {
                    let count = chain_to_supporter_count.entry(*chain).or_default();
                    *count = count
                        .checked_add(1)
                        .expect("supporter count bounded by participant set size");
                }
            }
        }
        self.available_foreign_chains = chain_to_supporter_count
            .into_iter()
            .filter_map(|(chain, count)| (count >= threshold).then_some(chain))
            .collect::<BTreeSet<_>>()
            .into();
    }
