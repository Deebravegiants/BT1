Audit Report

## Title
`RSETHPriceFeed` Returns ETH/USD `updatedAt` Instead of Composite Staleness, Masking Stale rsETH Oracle Prices — (File: `contracts/oracles/RSETHPriceFeed.sol`)

## Summary
`RSETHPriceFeed` implements `AggregatorV3Interface` as a composite feed multiplying the ETH/USD Chainlink price by the rsETH/ETH rate from `LRTOracle`. Both `latestRoundData()` and `getRoundData()` correctly compute the composite `answer`, but return `updatedAt`, `startedAt`, `roundId`, and `answeredInRound` exclusively from the ETH/USD Chainlink feed. Because `LRTOracle` stores no timestamp alongside `rsETHPrice`, the rsETH/ETH component's staleness is structurally undetectable by any consumer performing a standard Chainlink staleness check.

## Finding Description
`RSETHPriceFeed.latestRoundData()` (lines 63–70) and `getRoundData()` (lines 53–61) both follow the same pattern:

```solidity
(roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.latestRoundData();
answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
// updatedAt is still from ETH_TO_USD — rsETH staleness is never reflected
```

`RS_ETH_ORACLE.rsETHPrice()` reads the `rsETHPrice` state variable in `LRTOracle` (line 28), which is written only inside `_updateRsETHPrice()` (line 313) with no accompanying timestamp. The `IRSETHOracle` interface (lines 22–24) exposes only `rsETHPrice()` — there is no `lastUpdated` or equivalent. `updateRSETHPrice()` (line 87) is public and permissionless but has no on-chain enforcement of a maximum update interval.

The result: `updatedAt` in the returned tuple reflects only when the ETH/USD Chainlink round was last updated (typically every few minutes), not when the rsETH/ETH rate was last computed. A consumer checking `require(updatedAt > block.timestamp - maxStaleness)` will always pass this check even if `rsETHPrice` has not been refreshed for hours or days.

Existing guards are insufficient: the `whenNotPaused` modifier on `updateRSETHPrice()` only prevents updates when paused; it does not enforce freshness. The downside-protection pause logic (lines 270–281) only triggers if `updateRSETHPrice()` is actually called and the new price deviates beyond `pricePercentageLimit` — it provides no protection when the function is simply not called.

## Impact Explanation
Any protocol consuming `RSETHPriceFeed` as a Chainlink-compatible feed will perform the standard staleness check against `updatedAt`. Because `updatedAt` is sourced from the ETH/USD feed (which updates independently and frequently), this check will always pass regardless of how stale `rsETHPrice` is.

- **Stale inflated price (rsETH/ETH has fallen, e.g., slashing):** A lending protocol accepts an inflated rsETH/USD price. Borrowers can over-borrow against rsETH collateral, leaving the protocol under-collateralized — **temporary freezing of funds** for lenders unable to recover collateral, or worse, direct loss.
- **Stale deflated price (rsETH/ETH has risen):** Incorrect liquidations of healthy rsETH positions — **temporary freezing of user funds**.

**Impact: Medium — Temporary freezing of funds**, as claimed.

## Likelihood Explanation
`updateRSETHPrice()` is permissionless but relies entirely on off-chain keepers. There is no on-chain heartbeat enforcement. Keeper failure, delay, or front-running during volatile periods (precisely when staleness matters most) is a realistic operational scenario. The ETH/USD Chainlink feed will continue updating independently, making the staleness completely invisible to consumers. No privileged access or attacker action is required — passive keeper failure is sufficient.

## Recommendation
1. Add a `rsETHPriceLastUpdated` timestamp to `LRTOracle`, written every time `rsETHPrice` is updated in `_updateRsETHPrice()`.
2. Expose it via `IRSETHOracle`: `function rsETHPriceLastUpdated() external view returns (uint256)`.
3. In `RSETHPriceFeed`, return `min(ethToUSD_updatedAt, rsETH_lastUpdated)` as `updatedAt`:

```solidity
function latestRoundData()
    external view
    returns (uint80 roundId, int256 answer, uint256 startedAt, uint256 updatedAt, uint80 answeredInRound)
{
    (roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.latestRoundData();
    answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
    uint256 rsETHUpdatedAt = RS_ETH_ORACLE.rsETHPriceLastUpdated();
    if (rsETHUpdatedAt < updatedAt) updatedAt = rsETHUpdatedAt;
}
```

## Proof of Concept
1. Deploy `LRTOracle` and call `updateRSETHPrice()` at `T=0`; `rsETHPrice` is set (e.g., 1.05e18).
2. Advance time to `T = 25 hours`. Do not call `updateRSETHPrice()`. The ETH/USD Chainlink feed updates normally; its `updatedAt` advances to `T = 25 hours`.
3. Call `RSETHPriceFeed.latestRoundData()`:
   - `answer` = `1.05e18 * ethUSDPrice / 1e18` (stale rsETH/ETH rate)
   - `updatedAt` = `T = 25 hours` (from ETH/USD, appears fresh)
4. A lending protocol's staleness check `require(updatedAt > block.timestamp - 1 hours)` passes.
5. A borrower deposits rsETH and borrows at the inflated valuation. If the true rsETH/ETH rate has dropped (e.g., slashing), the protocol is immediately under-collateralized.

**Foundry fork test outline:** Fork mainnet, mock `LRTOracle.rsETHPrice` to a fixed value, `vm.warp` forward 25 hours without calling `updateRSETHPrice()`, call `RSETHPriceFeed.latestRoundData()`, assert `updatedAt` is within the last hour (it will be, from ETH/USD), and assert `answer` still uses the 25-hour-old rsETH rate — demonstrating the staleness is invisible.
