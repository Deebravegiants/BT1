### Title
Missing CKD Response Integrity Check for `AppPublicKey` Variant Allows Byzantine Coordinator to Deliver Fraudulent Key Material — (`File: crates/contract/src/lib.rs`)

---

### Summary

In `respond_ckd()`, the contract performs a cryptographic pairing-based output check (`ckd_output_check`) only when the request uses the `AppPublicKeyPV` variant. When the request uses the legacy `AppPublicKey` variant — the common path for most users — the response `(big_y, big_c)` is accepted and delivered to the caller with **no on-chain integrity verification**. A single Byzantine MPC participant acting as coordinator can submit an arbitrary fraudulent CKD response for any `AppPublicKey` request, and the contract will accept and return it without detection.

---

### Finding Description

**Root cause — conditional check with an empty arm:**

In `crates/contract/src/lib.rs`, `respond_ckd()` branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract verifies the pairing equation `e(big_c, G2) = e(big_y, app_pk2) · e(H(pk, app_id), pk)` via `ckd_output_check`: [2](#0-1) 

For `AppPublicKey` — the legacy variant used by the majority of callers — the `(big_y, big_c)` response fields are accepted and forwarded to the user with **zero cryptographic verification**.

**The `AppPublicKey` variant is the default and most common path.** The `request_app_private_key` entry point also only validates the `AppPublicKeyPV` variant at submission time: [3](#0-2) 

**Attacker-controlled entry path:**

1. A user submits `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(pk)` (the legacy path). The contract enqueues the request.
2. MPC nodes run the CKD protocol. The coordinator aggregates shares and calls `respond_ckd`.
3. A Byzantine participant who is elected coordinator for this session calls `respond_ckd` with arbitrary `big_y` and `big_c` values of their choosing.
4. The contract's `respond_ckd` passes `assert_caller_is_attested_participant_and_protocol_active()`, then hits the `AppPublicKey(_) => {}` arm — no check — and calls `pending_requests::resolve_yields_for`, delivering the fraudulent response to the user. [4](#0-3) 

The `AppPublicKeyPV` path is protected because the pairing check verifies the response is a valid encryption of `msk · H(pk, app_id)` under the user's public key. The `AppPublicKey` path has no equivalent protection.

---

### Impact Explanation

The CKD protocol's purpose is to derive application-specific private keys that users rely on to decrypt data or control funds on foreign chains. When a Byzantine coordinator delivers a fraudulent `(big_y, big_c)`:

- The user receives a key that does **not** equal `msk · H(pk, app_id)`.
- Any assets sent to an address derived from the expected key become permanently inaccessible — the user holds a wrong key and cannot reconstruct the correct one.
- The contract's integrity guarantee for the CKD flow is broken: the on-chain response is accepted as valid despite being cryptographically invalid.

This maps to the allowed impact: **Medium — contract execution-flow manipulation that breaks production safety/accounting invariants** (fraudulent key material delivered and accepted on-chain), with potential escalation to **Critical** if the user has pre-funded a foreign-chain address derived from the expected key (permanent freezing of funds controlled by the MPC-derived key).

---

### Likelihood Explanation

- The `AppPublicKey` variant is the **legacy and default** path; the CLI documentation and existing tests show it is the common usage pattern.
- A single Byzantine participant below the signing threshold suffices — no collusion is required. The coordinator role is deterministic per request, so a malicious participant can target any request for which they are elected coordinator.
- The attacker is already an attested participant (a valid precondition within the Byzantine-below-threshold threat model), so no additional privilege is needed beyond their normal role.

---

### Recommendation

Apply `ckd_output_check` unconditionally for both variants in `respond_ckd`. For the `AppPublicKey` variant, the check can be performed by treating `pk1` as the user's G1 public key (already stored in the request). Alternatively, require all new CKD requests to use `AppPublicKeyPV` and deprecate the unverifiable `AppPublicKey` path, or add a separate pairing-based integrity check for the legacy variant that verifies the response is consistent with the network public key and `app_id`.

The analogous fix in the PrimeX report was to make the oracle check unconditional regardless of position type. The same principle applies here: the cryptographic output check must be applied to **all** CKD responses, not only those using the publicly-verifiable variant.

---

### Proof of Concept

1. User Alice calls `request_app_private_key` with:
   ```json
   { "derivation_path": "mykey", "app_public_key": "bls12381g1:<alice_pk1>", "domain_id": 0 }
   ```
   This enqueues a `CKDRequest` with `app_public_key = AppPublicKey(alice_pk1)`.

2. Malicious coordinator (an attested MPC participant) calls `respond_ckd` with:
   ```rust
   CKDResponse { big_y: arbitrary_g1_point, big_c: arbitrary_g1_point }
   ```

3. `respond_ckd` executes:
   - `assert_caller_is_attested_participant_and_protocol_active()` → passes (caller is a valid participant).
   - `match &request.app_public_key { AppPublicKey(_) => {} ... }` → **empty arm, no check**.
   - `pending_requests::resolve_yields_for(...)` → fraudulent response delivered to Alice.

4. Alice receives `(big_y, big_c)` that do not satisfy `e(big_c, G2) = e(H(pk, app_id), pk) · e(big_y, alice_pk2)`. Her derived key is wrong. Any funds she pre-sent to a foreign-chain address controlled by the expected key are permanently frozen. [5](#0-4) [6](#0-5)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
