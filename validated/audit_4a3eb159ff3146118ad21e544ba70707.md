### Title
Deposit Cap Enforced via Raw `balanceOf` Allows Anyone to Permanently DOS New Deposits - (File: `contracts/LRTDepositPool.sol`)

### Summary
`LRTDepositPool` enforces its per-asset deposit cap by reading the live ERC-20 `balanceOf` the contract rather than a tracked internal counter. Any unprivileged user can directly transfer supported LST tokens to `LRTDepositPool` to artificially inflate the measured total, pushing it past `depositLimitByAsset`, and causing every subsequent `depositAsset()` call to revert with `MaximumDepositLimitReached`.

### Finding Description
`getAssetDistributionData()` computes the pool's share of total deposits as:

```solidity
assetLyingInDepositPool = IERC20(asset).balanceOf(address(this));
``` [1](#0-0) 

This value feeds directly into `getTotalAssetDeposits()`:

```solidity
return (assetLyingInDepositPool + assetLyingInNDCs + effectiveAssetWithEigenLayer
        + assetLyingInConverter + assetLyingUnstakingVault);
``` [2](#0-1) 

Which is then compared against the configured cap in `_checkIfDepositAmountExceedesCurrentLimit()`:

```solidity
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
``` [3](#0-2) 

And enforced in `_beforeDeposit()`, called by every `depositAsset()` invocation:

```solidity
if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
    revert MaximumDepositLimitReached();
}
``` [4](#0-3) 

Because `LRTDepositPool` holds no access control on token receipt (it is a plain ERC-20 transfer target), any address can call `IERC20(stETH).transfer(depositPool, amount)` directly, bypassing `depositAsset()` entirely. The donated tokens are immediately counted as "deposited" by the cap logic, yet no rsETH is minted for them and no internal accounting entry is created. The same flaw exists for the ETH path via `address(this).balance` in `getETHDistributionData()`:

```solidity
ethLyingInDepositPool = address(this).balance;
``` [5](#0-4) 

and for the unstaking vault component:

```solidity
assetLyingUnstakingVault = IERC20(asset).balanceOf(lrtUnstakingVault);
``` [6](#0-5) 

### Impact Explanation
All calls to `depositAsset()` (and `depositETH()`) revert with `MaximumDepositLimitReached` for the targeted asset until an admin raises `depositLimitByAsset`. The deposited tokens are irrecoverable by the attacker (no withdrawal path exists for unsolicited transfers), making this a one-way griefing attack. Legitimate users are temporarily frozen out of the deposit function — **Temporary freezing of funds (Medium)**.

### Likelihood Explanation
The attack requires only holding a supported LST (stETH, ETHx) and calling a standard ERC-20 `transfer`. No privileged access, no flash loan, no front-running dependency. The cost equals the value of tokens donated, which becomes viable whenever LST prices fall or the remaining cap headroom is small. The `depositLimitByAsset` values are set to `100_000 ether` per asset at initialization: [7](#0-6) 

meaning the attack cost scales with how much headroom remains, but near the cap boundary the cost can be negligible.

### Recommendation
Replace the raw `balanceOf` reads with an internal accounting variable (e.g., `totalDeposited[asset]`) that is incremented only inside `depositAsset()` / `depositETH()` and decremented on withdrawals. The `balanceOf` call can still be used for sanity checks or operator views, but must not gate the deposit cap. This mirrors the fix applied to Connext's `RoutersFacet` in PR 2444.

### Proof of Concept
1. Protocol has `depositLimitByAsset[stETH] = 100_000e18` and `getTotalAssetDeposits(stETH)` currently returns `99_999e18`.
2. Attacker calls `IERC20(stETH).transfer(depositPool, 2e18)` directly (no `depositAsset()` call).
3. `getAssetDistributionData()` now reads `balanceOf(depositPool)` as `2e18` higher than before.
4. `getTotalAssetDeposits(stETH)` returns `100_001e18 > 100_000e18`.
5. Any subsequent `depositAsset(stETH, any_amount, ...)` call hits `_checkIfDepositAmountExceedesCurrentLimit` → returns `true` → `_beforeDeposit` reverts `MaximumDepositLimitReached`.
6. All stETH deposits are blocked until admin calls `updateAssetDepositLimit` to raise the cap. [8](#0-7) [9](#0-8)
