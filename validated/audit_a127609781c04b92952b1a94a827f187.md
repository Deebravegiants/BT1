### Title
Cross-Chain Replay of `deploy_token` MPC Signature Permanently Blocks Legitimate Token Deployment - (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `aptos/sources/bridge_types.move`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The `MetadataPayload` signed by the NEAR MPC for `deploy_token` does not include the destination chain ID in the Borsh-encoded bytes that are hashed and verified. The identical signed bytes are accepted on every chain (EVM, Starknet, Aptos, Solana). An unprivileged attacker who observes a valid `deploy_token` signature for chain A can immediately replay it on chains B, C, and D. Because each chain enforces a one-time deployment guard (`ERR_TOKEN_ALREADY_DEPLOYED` / `ERR_TOKEN_EXIST`), the replay permanently prevents the protocol from ever legitimately deploying that token on the replayed chains, freezing the redemption path for all users who attempt to bridge that token there.

---

### Finding Description

Every chain's `fin_transfer` path correctly binds the signed payload to the destination chain by embedding `chain_id` (the OmniAddress tag) twice in the Borsh encoding — once before `token_address` and once before `recipient`. The `deploy_token` path omits this binding entirely.

Because the Borsh bytes are identical across all four chains, the `keccak256` hash is identical, and the NEAR MPC signature is valid on all of them simultaneously.

---

### Impact Explanation

**Permanent token deployment freeze (Critical/High):**

1. NEAR MPC signs `MetadataPayload{token="usdc.near", name, symbol, decimals}` for Ethereum deployment. The signed bytes are chain-agnostic.
2. Attacker replays the same `(signature, payload)` on Starknet, Aptos, and Solana before the protocol does.
3. Each chain's one-time guard (`ERR_TOKEN_ALREADY_DEPLOYED` on Starknet/Aptos, `ERR_TOKEN_EXIST` on EVM) fires on all subsequent legitimate calls.
4. The token is now permanently un-deployable on those chains through the normal protocol flow.
5. Any user who later bridges `usdc.near` to Starknet/Aptos/Solana will find no valid bridge token contract registered on the NEAR side (because NEAR's `bind_token` was never called for the attacker-deployed instance), making their bridged funds permanently unclaimable.

The attacker-deployed token contract is orphaned: it exists on-chain but has no NEAR-side binding, so the bridge cannot mint or unlock against it. The legitimate deployment path is permanently closed.

---

### Likelihood Explanation

- The attacker needs only to observe a single valid `deploy_token` transaction on any one chain (fully public on-chain data) and submit the same calldata to the other chains before the protocol does.
- No privileged access, no leaked keys, no colluding MPC signers required.
- The window is the time between the first legitimate deployment and the protocol's subsequent deployments on other chains — typically hours to days during a new token rollout.
- The attack is cheap (one transaction per target chain) and irreversible.

---

### Recommendation

Embed the destination chain ID in the `MetadataPayload` Borsh encoding, mirroring the pattern already used in `TransferMessagePayload`. Concretely:

**EVM** — add `bytes1(omniBridgeChainId)` to the `borshEncoded` concatenation in `deployToken()`.

**Starknet** — add a `chain_id: u8` parameter to `MetadataPayload.to_borsh()` and append it (e.g., after the payload type byte), then pass `self.omni_bridge_chain_id.read()` at the call site in `deploy_token()`.

**Aptos** — add a `chain_id: u8` parameter to `metadata_to_borsh()` and push it into the buffer, then pass `state.chain_id` at the call site in `deploy_token()`.

**Solana** — add the chain ID constant to `DeployTokenPayload::serialize_for_near()`.

**NEAR MPC signer** — update the signing logic to include the target chain ID in the metadata payload hash so that a signature for chain A is cryptographically distinct from one for chain B.

---

### Proof of Concept

```
1. NEAR MPC signs MetadataPayload for "usdc.near" → Ethereum.
   Signed bytes: [0x01, len(usdc.near), "usdc.near", len(USD Coin), "USD Coin", len(USDC), "USDC", 0x06]
   Signature: sig_mpc

2. Attacker calls Starknet deploy_token(sig_mpc, MetadataPayload{token="usdc.near",...})
   → _verify_borsh_signature hashes identical bytes → recovers same signer → passes
   → near_to_starknet_token["usdc.near"] = <attacker-deployed address>

3. Protocol later calls Starknet deploy_token(sig_mpc, same payload)
   → assert(existing_token.is_zero(), 'ERR_TOKEN_ALREADY_DEPLOYED') → REVERTS

4. "usdc.near" is permanently un-deployable on Starknet.
   Users bridging USDC to Starknet receive nothing; funds are locked on NEAR.
