### Title
Cross-EVM-Chain Replay of `deployToken` MPC Signature Due to Missing Chain Binding — (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

The `deployToken` function in `OmniBridge.sol` verifies an MPC signature over a borsh-encoded payload that contains **no chain-specific identifier**. In contrast, `finTransfer` in the same contract correctly embeds `omniBridgeChainId` twice in its signed payload. Because the same `nearBridgeDerivedAddress` (derived from the NEAR MPC key) is used across every EVM deployment of `OmniBridge`, a valid `deployToken` signature observed on one EVM chain (e.g., Ethereum) can be replayed verbatim on any other EVM chain running `OmniBridge` (Arbitrum, Base, BNB, Polygon, Abstract, HyperEVM), causing unauthorized token deployment on the target chain.

---

### Finding Description

In `OmniBridge.sol`, `deployToken` constructs the signed message as:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [1](#0-0) 

The payload contains only: a 1-byte type prefix, the NEAR token string, name, symbol, and decimals. **No `omniBridgeChainId` and no contract address are included.**

Compare with `finTransfer`, which correctly binds the signature to the destination chain by embedding `omniBridgeChainId` in the token address and recipient address positions:

```solidity
bytes1(omniBridgeChainId),
Borsh.encodeAddress(payload.tokenAddress),
Borsh.encodeUint128(payload.amount),
bytes1(omniBridgeChainId),
Borsh.encodeAddress(payload.recipient),
``` [2](#0-1) 

The `omniBridgeChainId` field exists in contract storage and is set at initialization time, so it is available to `deployToken` — it is simply not used. [3](#0-2) 

The only replay guard in `deployToken` is:

```solidity
require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST");
``` [4](#0-3) 

This guard only prevents re-deployment on the **same** chain. It provides no protection against replay on a **different** chain where the token has not yet been deployed.

---

### Impact Explanation

An attacker who observes a valid `deployToken` call on chain A can replay the identical `(signatureData, metadata)` tuple on chain B. The signature check passes because:

1. The MPC-derived address (`nearBridgeDerivedAddress`) is the same across all EVM `OmniBridge` deployments — it is derived from the NEAR chain's MPC key, not from any per-chain secret.
2. The signed bytes are byte-for-byte identical on every chain (no chain ID, no contract address in the payload).

Consequences on the target chain B:
- A bridge token contract is deployed and registered in `nearToEthToken` / `ethToNearToken` / `isBridgeToken` without the protocol's authorization.
- Any subsequent legitimate `deployToken` call for the same NEAR token on chain B reverts with `ERR_TOKEN_EXIST`, permanently blocking the protocol from deploying the token with intended configuration (e.g., a custom minter via `addCustomToken`, or different decimal normalization).
- If the protocol intended to use a `customMinter` for this token on chain B (for a wrapped asset with special burn/mint logic), that path is foreclosed; `finTransfer` will fall through to the plain `IBridgeToken.mint` path instead, potentially breaking backing guarantees.

This fits the **High** impact category: insufficiently-bound MPC signatures that bypass the deployment execution gate, with potential to break asset-identity and token-mapping guarantees on the target chain.

---

### Likelihood Explanation

The attack requires only:
1. Monitoring any public EVM chain for a `deployToken` transaction (trivially done via event indexing or mempool watching).
2. Submitting the same calldata to `OmniBridge` on a different EVM chain.

No privileged access, no leaked keys, no colluding MPC signers. The attacker is fully unprivileged. The Omni Bridge is live on multiple EVM chains simultaneously, so the replay surface is large and persistent.

---

### Recommendation

Include `omniBridgeChainId` in the borsh-encoded payload that is signed by the MPC for `deployToken`, mirroring the pattern already used in `finTransfer`:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),          // bind to destination chain
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

The NEAR-side MPC signing logic must be updated in parallel to include the destination chain ID when producing `deployToken` signatures, so that existing signatures are invalidated on all other chains.

---

### Proof of Concept

1. Token `"usdc.near"` is authorized for deployment on Ethereum. The MPC produces signature `S` over `keccak256(borsh(Metadata | "usdc.near" | "USD Coin" | "USDC" | 6))`.
2. A relayer submits `deployToken(S, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})` on Ethereum. The transaction succeeds; a `BridgeToken` proxy is deployed at address `T_eth`.
3. An attacker observes this transaction and extracts `(S, metadata)`.
4. The attacker calls `deployToken(S, metadata)` on Arbitrum's `OmniBridge`. The signed bytes are identical; `ECDSA.recover` returns `nearBridgeDerivedAddress`; the check passes.
5. A new `BridgeToken` proxy `T_arb` is deployed on Arbitrum and registered in `nearToEthToken["usdc.near"] = T_arb`.
6. When the protocol later attempts to deploy `"usdc.near"` on Arbitrum with a custom minter (e.g., for a native USDC integration), the call reverts with `ERR_TOKEN_EXIST`. The protocol is locked into the attacker-triggered plain `BridgeToken` deployment.
