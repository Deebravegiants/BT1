### Title
Cross-Chain Replay of `deployToken` Signature Due to Missing Chain ID Binding - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
The `deployToken` function in `OmniBridge.sol` verifies a signature over a Borsh-encoded message that omits the destination chain ID (`omniBridgeChainId`). Because all EVM deployments of `OmniBridge` share the same `nearBridgeDerivedAddress` (derived from the same NEAR MPC key at path `"bridge-1"`), a valid `deployToken` signature obtained for one EVM chain can be replayed verbatim on any other EVM chain without modification.

### Finding Description
In `OmniBridge.sol`, `deployToken` constructs the signed message as:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [1](#0-0) 

No chain identifier is included. In contrast, `finTransfer` encodes `omniBridgeChainId` twice in its signed message, making it chain-specific:

```solidity
bytes1(omniBridgeChainId),   // token chain binding
...
bytes1(omniBridgeChainId),   // recipient chain binding
``` [2](#0-1) 

On the NEAR side, `log_metadata_callback` signs a `MetadataPayload` that also contains no chain ID:

```rust
let metadata_payload = MetadataPayload {
    prefix: PayloadType::Metadata,
    token: token_id.to_string(),
    name: metadata.name,
    symbol: metadata.symbol,
    decimals: metadata.decimals,
};
let payload = near_sdk::env::keccak256_array(
    borsh::to_vec(&metadata_payload).near_expect(BridgeError::Borsh),
);
ext_signer::ext(self.mpc_signer.clone())
    .sign(SignRequest { payload, path: SIGN_PATH.to_owned(), key_version: 0 })
``` [3](#0-2) 

`SIGN_PATH = "bridge-1"` is a single fixed path used for all signing operations across all chains. [4](#0-3) 

Because the NEAR MPC derives a deterministic Ethereum address from a fixed key path, `nearBridgeDerivedAddress` is identical on Ethereum, Arbitrum, Base, BNB, Polygon, HyperEVM, and Abstract. A signature valid on one chain is cryptographically valid on all others.

The only replay protection in `deployToken` is the idempotency check:

```solidity
require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST");
``` [5](#0-4) 

This check is per-chain state, so it does not prevent replay across chains.

### Impact Explanation
An unprivileged attacker who observes a `deployToken` transaction on any EVM chain can:

1. Extract the `signatureData` and `MetadataPayload` from the public transaction.
2. Call `deployToken` on every other EVM chain with the identical arguments.
3. A legitimate `BridgeToken` proxy is deployed on each target chain, setting `isBridgeToken`, `ethToNearToken`, and `nearToEthToken` mappings.
4. When the NEAR bridge later attempts to legitimately deploy the same token on those chains, the call reverts with `ERR_TOKEN_EXIST`.
5. The NEAR bridge's internal accounting (`token_id_to_address`, `token_decimals`) has no record of these attacker-triggered deployments, so it cannot sign `finTransfer` messages for the correct token address on those chains.
6. The token is permanently stranded on those chains — no `finTransfer` can ever mint to it — unless an admin manually calls `addCustomToken` to re-register it.

This constitutes unauthorized creation of wrapped bridge assets through a deployment verification failure, and a permanently frozen redemption path for that token on the affected chains.

### Likelihood Explanation
The attack requires no privileges. The attacker only needs to:
- Monitor any EVM chain for `deployToken` transactions (public mempool or block explorer).
- Replay the same calldata on other EVM chains.

This is trivially automatable. Any token deployment on any EVM chain immediately exposes all other EVM chains to this replay.

### Recommendation
Include `omniBridgeChainId` in the `deployToken` signed message, matching the pattern already used in `finTransfer`:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),          // bind to this chain
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

The corresponding `MetadataPayload` on the NEAR side must also include the destination chain kind, and `log_metadata_callback` must accept and embed it before signing.

### Proof of Concept

```
1. NEAR bridge calls log_metadata("usdc.near") → MPC signs MetadataPayload{token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6} → emits SignTransferEvent with signature S.

2. Relayer submits deployToken(S, {token:"usdc.near",...}) on Ethereum (chain ID 1).
   → Token deployed at address T_eth. nearToEthToken["usdc.near"] = T_eth on Ethereum.

3. Attacker observes S and the payload from the Ethereum transaction.

4. Attacker calls deployToken(S, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6}) on Arbitrum.
   → ECDSA.recover(keccak256(borshEncoded), S) == nearBridgeDerivedAddress ✓ (same key, no chain binding)
   → isBridgeToken[nearToEthToken["usdc.near"]] == false ✓ (fresh Arbitrum state)
   → Token deployed at address T_arb. nearToEthToken["usdc.near"] = T_arb on Arbitrum.

5. NEAR bridge later tries to deploy "usdc.near" on Arbitrum legitimately:
   → deployToken(S', {...}) reverts: "ERR_TOKEN_EXIST"

6. NEAR bridge has no record of T_arb in its token_id_to_address mapping.
   → Cannot sign finTransfer for T_arb on Arbitrum.
   → "usdc.near" is permanently unclaimable on Arbitrum without admin intervention.
```
