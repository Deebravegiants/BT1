### Title
Missing Zero-Rate Validation in `lzReceive` Enables Temporary Freezing of All Pool Deposits — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

---

### Summary

`CrossChainRateReceiver.lzReceive` writes any decoded `_rate` value — including zero — directly to storage without validation. Because `RSETHPoolV2.viewSwapRsETHAmountAndFee` and `RSETHPoolV3.viewSwapRsETHAmountAndFee` divide by `rsETHToETHrate` with no zero-guard, a single cross-chain message carrying `_rate=0` causes every subsequent `deposit` call to revert with a Solidity division-by-zero panic, freezing all pool deposits until the next valid update arrives.

---

### Finding Description

**Root cause — `CrossChainRateReceiver.lzReceive` (line 95):**

`lzReceive` correctly authenticates the message (endpoint, source chain, source address) but performs no sanity check on the decoded rate value before persisting it:

```solidity
uint256 _rate = abi.decode(_payload, (uint256));
rate = _rate;          // ← no require(_rate > 0)
lastUpdated = block.timestamp;
``` [1](#0-0) 

**Propagation — `RSETHPoolV2.viewSwapRsETHAmountAndFee` (line 233):**

```solidity
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;  // panic if rate == 0
``` [2](#0-1) 

The same pattern exists in `RSETHPoolV3.viewSwapRsETHAmountAndFee`: [3](#0-2) 

**`deposit` is fully blocked:** the `limitDailyMint` modifier calls `viewSwapRsETHAmountAndFee` *before* the function body, so the revert fires even before any state change: [4](#0-3) 

**Inconsistency confirming developer awareness:** `RSETHPoolV3.viewSwapAssetToPremintedRsETH` *does* guard against zero:

```solidity
if (rsETHToETHrate == 0) revert UnsupportedOracle();
``` [5](#0-4) 

This guard is absent in both `viewSwapRsETHAmountAndFee` overloads.

---

### Impact Explanation

Once `rate` is set to zero, every call to `RSETHPoolV2.deposit` and `RSETHPoolV3.deposit` (ETH and ERC-20 paths) reverts with a division-by-zero panic. No user can deposit until the next valid cross-chain rate update overwrites the zero. This is a **temporary freezing of funds** (all pool deposits).

---

### Likelihood Explanation

The trigger requires a legitimately authenticated LayerZero message (correct endpoint, correct source chain, correct `rateProvider` address) carrying `_rate=0`. This can arise from:

- A transient oracle fault on the provider chain (e.g., the underlying price feed returning 0 during an upgrade or edge-case state)
- A bug in the `CrossChainRateProvider` that sends 0 before the oracle is initialized

No admin compromise, key leak, or malicious actor is required — the message passes all three authentication checks in `lzReceive`. The missing guard is the sole root cause.

---

### Recommendation

Add a zero-value guard in `lzReceive` before writing to storage:

```solidity
require(_rate > 0, "Rate must be non-zero");
rate = _rate;
``` [6](#0-5) 

Additionally, add a matching guard in both `viewSwapRsETHAmountAndFee` functions (V2 and V3) consistent with the existing guard in `viewSwapAssetToPremintedRsETH`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import "forge-std/Test.sol";
import "contracts/cross-chain/CrossChainRateReceiver.sol";
import "contracts/pools/RSETHPoolV2.sol";

contract ZeroRateFreeze is Test {
    // Minimal concrete receiver
    CrossChainRateReceiverImpl receiver; // concrete subclass of abstract
    RSETHPoolV2 pool;
    address lzEndpoint = address(0xE0);
    address rateProvider = address(0xA0);
    uint16 srcChain = 101;

    function setUp() public {
        receiver = new CrossChainRateReceiverImpl(address(this));
        receiver.updateLayerZeroEndpoint(lzEndpoint);
        receiver.updateRateProvider(rateProvider);
        receiver.updateSrcChainId(srcChain);

        // Deploy pool pointing to receiver as oracle
        pool = new RSETHPoolV2Impl();
        pool.initialize(address(this), address(this), address(wrsETH), 0, address(receiver));
    }

    function test_zeroRateFreezeDeposit() public {
        // Step 1: LZ endpoint delivers _rate=0
        bytes memory srcAddressBytes = abi.encodePacked(rateProvider);
        vm.prank(lzEndpoint);
        receiver.lzReceive(srcChain, srcAddressBytes, 0, abi.encode(uint256(0)));

        assertEq(receiver.getRate(), 0);

        // Step 2: Any deposit reverts with division-by-zero
        vm.deal(address(this), 1 ether);
        vm.expectRevert(); // Panic: division by zero
        pool.deposit{value: 1 ether}("ref");
    }

    function testFuzz_zeroRateFreezeDeposit(uint256 _rate) public {
        vm.assume(_rate <= 1e15); // near-zero rates also cause extreme minting
        bytes memory srcAddressBytes = abi.encodePacked(rateProvider);
        vm.prank(lzEndpoint);
        receiver.lzReceive(srcChain, srcAddressBytes, 0, abi.encode(_rate));

        if (_rate == 0) {
            vm.deal(address(this), 1 ether);
            vm.expectRevert();
            pool.deposit{value: 1 ether}("ref");
        }
    }
}
```
