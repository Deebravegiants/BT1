### Title
`initialize2` Overwrites `unlockedWithdrawalsCount` Incremented by Concurrent `unlockQueue`, Causing Underflow Revert on Legitimate `completeWithdrawal` — (`contracts/LRTWithdrawalManager.sol`)

---

### Summary

`initialize2` **assigns** (not adds to) `unlockedWithdrawalsCount[asset]`. If an operator calls `unlockQueue` in the window between `processChunk` finishing its scan and `finalizeInitialize2` being called, the M requests unlocked by that `unlockQueue` call are silently dropped from the counter. After N completions (N = seeded value), the counter reaches zero, and the next legitimate `completeWithdrawal` hits a Solidity 0.8 checked-arithmetic underflow revert.

---

### Finding Description

`UnlockedWithdrawalsInitializer.processChunk` scans requests in `[0, nextLockedNonce_at_scan_time)` and accumulates a count N. [5](#0-4) 

`finalizeInitialize2` then calls `initialize2(N, ...)`, which **sets** the counter:

```solidity
unlockedWithdrawalsCount[lrtConfig.getLSTToken(LRTConstants.ST_ETH_TOKEN)] = unlockedWithdrawalsCountSTETH;
unlockedWithdrawalsCount[lrtConfig.getLSTToken(LRTConstants.ETHX_TOKEN)] = unlockedWithdrawalsCountETHx;
unlockedWithdrawalsCount[LRTConstants.ETH_TOKEN] = unlockedWithdrawalsCountETH;
``` [1](#0-0) 

`unlockQueue` carries no `onlyBeforeInitialize2` guard and can be called at any time by an operator: [2](#0-1) 

If `unlockQueue` runs after `processChunk` but before `finalizeInitialize2`, `_unlockWithdrawalRequests` increments `unlockedWithdrawalsCount` from 0 to M for M newly unlocked requests: [6](#0-5) 

`initialize2` then overwrites the counter to N, discarding M. Now N+M requests are unlocked but the counter is N. After N completions the counter is 0, and the (N+1)th call to `_processWithdrawalCompletion` hits:

```solidity
unlockedWithdrawalsCount[asset]--;   // underflow → revert (Solidity 0.8 checked arithmetic)
``` [3](#0-2) 

The `WithdrawalLocked` guard does **not** protect against this: the nonce is below `nextLockedNonce`, so the request is legitimately unlocked, yet the decrement still reverts. [7](#0-6) 

---

### Impact Explanation

Users whose requests were unlocked by the post-scan `unlockQueue` call cannot complete their withdrawals. Their rsETH was already burned inside `unlockQueue` (line 305), and the corresponding assets sit in the contract, but every `completeWithdrawal` call reverts. There is no admin setter to correct `unlockedWithdrawalsCount` for ETHx/stETH/ETH after `initialize2` has been called (it is a `reinitializer(2)`, callable exactly once); recovery requires a contract upgrade. Funds are not lost but are inaccessible until an upgrade is deployed.

**Impact: Low — Contract fails to deliver promised returns, but doesn't lose value.**

---

### Likelihood Explanation

The vulnerable window is the period between `processChunk` completing its scan and `finalizeInitialize2` being called. `unlockQueue` is a routine operator action performed as part of normal withdrawal processing. No malicious intent is required; a well-meaning operator calling `unlockQueue` during the upgrade window is sufficient to trigger the bug. The window may span multiple blocks or even days depending on operational cadence.

---

### Recommendation

Change `initialize2` to **add** to the existing counter rather than overwrite it:

```solidity
unlockedWithdrawalsCount[...] += unlockedWithdrawalsCountSTETH;
unlockedWithdrawalsCount[...] += unlockedWithdrawalsCountETHx;
unlockedWithdrawalsCount[LRTConstants.ETH_TOKEN] += unlockedWithdrawalsCountETH;
```

This makes the seeding idempotent with respect to any `unlockQueue` calls that ran before `initialize2`. Alternatively, add an `onlyBeforeInitialize2` guard to `unlockQueue` so no new unlocks can occur during the initialization window.

---

### Proof of Concept

```solidity
// Pseudocode unit test (local fork / Foundry)

// State before initialize2:
// - 3 requests unlocked by a prior unlockQueue call (nonces 0,1,2)
// - nextLockedNonce[asset] = 3
// - unlockedWithdrawalsCount[asset] = 3  (set by that unlockQueue)

// processChunk scans [0,3) → counts 3 → unlockedCount = 3

// Operator calls unlockQueue again → unlocks nonces 3,4
// unlockedWithdrawalsCount[asset] = 5, nextLockedNonce = 5

// finalizeInitialize2 calls initialize2(3, ...)
// → unlockedWithdrawalsCount[asset] = 3  (overwrites 5)

// Users complete nonces 0,1,2 → counter = 0
// User tries to complete nonce 3 (legitimately unlocked):
//   usersFirstWithdrawalRequestNonce=3 < nextLockedNonce=5 → passes lock check
//   unlockedWithdrawalsCount[asset]-- → 0-- → REVERT (underflow)

assertReverts(completeWithdrawal(asset, ...));  // nonce 3 is stuck
assertReverts(completeWithdrawal(asset, ...));  // nonce 4 is stuck
```
