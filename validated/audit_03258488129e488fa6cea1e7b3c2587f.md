### Title
VRF Key Hash Reference-Count Underflow in `poolReapTransition` Allows Duplicate VRF Keys Post-v11 — (`eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/PoolReap.hs`)

---

### Summary

`poolReapTransition` uses `Map.withoutKeys` to remove "dangling" VRF key hashes (old VRF keys superseded by a pool re-registration) from `psVRFKeyHashes`. `Map.withoutKeys` deletes the key entirely regardless of its reference count. When a dangling VRF key is still held by a second pool that did **not** re-register, the count is zeroed out incorrectly. The key disappears from the map, and the post-v11 uniqueness guard (`Map.notMember sppVrf psVRFKeyHashes`) subsequently passes for that key, allowing a new pool to register with it and creating a duplicate VRF key in the active pool set.

---

### Finding Description

**Step 1 — Pre-v11 state (allowed by old rules)**

Pool P1 and pool P2 both register with VRF key V before the v11 hardfork. Duplicate VRF keys were permitted pre-v11.

**Step 2 — v11 hardfork: `populateVRFKeyHashes`**

`populateVRFKeyHashes` (HardFork.hs lines 107–125) counts VRF keys from both `psStakePools` and `psFutureStakePoolParams`:

```
psVRFKeyHashes = { V → 2 }
``` [1](#0-0) 

**Step 3 — P1 re-registers with fresh VRF key W (post-v11)**

In `poolTransition`, the re-registration branch (Pool.hs lines 283–295) handles the case where `psFutureStakePoolParams` has no prior entry for P1 (`Nothing` branch) by inserting W with count 1:

```
psVRFKeyHashes = { V → 2, W → 1 }
psFutureStakePoolParams[P1] = { sppVrf = W }
``` [2](#0-1) 

**Step 4 — Epoch boundary: `poolReapTransition` computes `danglingVRFKeyHashes`**

The merge at lines 138–148 finds pools where `psStakePools[pool].spsVrf ≠ psFutureStakePoolParams[pool].sppVrf`. For P1: V ≠ W → emits `Just V`. P2 has no future entry → `Map.dropMissing` silently drops it. Result:

```
danglingVRFKeyHashes = { V }
``` [3](#0-2) 

**Step 5 — `Map.withoutKeys` deletes V entirely**

Lines 224–227 apply:

```haskell
& certPStateL . psVRFKeyHashesL
  %~ ( removeVRFKeyHashOccurrences retiredVRFKeyHashes
         . (`Map.withoutKeys` danglingVRFKeyHashes)
     )
```

`Map.withoutKeys psVRFKeyHashes { V }` removes V unconditionally. The count of 2 is irrelevant to `Map.withoutKeys`. After the epoch boundary:

```
psVRFKeyHashes = { W → 1 }   -- V is gone
psStakePools[P2].spsVrf = V  -- P2 still actively uses V
``` [4](#0-3) 

**Step 6 — P3 registers with V, bypassing the uniqueness guard**

The new-pool registration guard (Pool.hs lines 265–267):

```haskell
Map.notMember sppVrf psVRFKeyHashes
  ?! injectFailure (VRFKeyHashAlreadyRegistered sppId sppVrf)
```

V is absent from `psVRFKeyHashes`, so `Map.notMember V psVRFKeyHashes = True`. The check passes. P3 registers with V. Both P2 and P3 are now active pools with VRF key V, violating the invariant the v11 hardfork was designed to enforce. [5](#0-4) 

**Root cause**

`Map.withoutKeys` is a set-difference operation that removes keys entirely. The correct operation for decrementing a reference-counted map is `removeVRFKeyHashOccurrence`, which already exists in the same function and is used correctly for retired pools. The dangling-key path uses the wrong primitive. [6](#0-5) 

---

### Impact Explanation

The VRF uniqueness invariant (`psVRFKeyHashes` accurately reflects the number of active pools using each VRF key) is broken. After the bug fires, `psVRFKeyHashes` no longer contains V even though P2 actively uses it. Any new pool can then register with V, producing two simultaneously active pools sharing the same VRF key. Because VRF output determines slot-leadership eligibility, two pools sharing the same VRF key are eligible for exactly the same slots. If the same VRF secret key is held by both operators (e.g., one operator controls both P2 and P3, or P3 copies P2's public key to register and then obtains the secret key), both can produce valid blocks for the same slot, causing a fork. Even without secret-key sharing, the ledger's `psVRFKeyHashes` invariant is permanently corrupted, and the v11 uniqueness guarantee is silently voided for the affected key.

---

### Likelihood Explanation

The sequence is fully attacker-controlled via ordinary pool certificates submitted in transactions:

1. Register P1 and P2 with the same VRF key V before v11 (no special privilege required; duplicate VRF keys were legal pre-v11).
2. After the v11 hardfork activates, submit a `RegPool` certificate for P1 with a new VRF key W.
3. Wait one epoch boundary (POOLREAP fires automatically).
4. Submit a `RegPool` certificate for P3 with VRF key V.

All four steps are standard, unprivileged pool-operator actions. No governance majority, leaked key, or Sybil attack is required. The sequence is deterministic and locally testable.

---

### Recommendation

Replace `Map.withoutKeys danglingVRFKeyHashes` with `removeVRFKeyHashOccurrences (Set.toList danglingVRFKeyHashes)` in `poolReapTransition`. This reuses the already-correct decrement-and-remove-on-zero logic that is applied to retired pools, ensuring that a dangling VRF key whose count is still > 1 (because other pools still use it) is decremented rather than deleted.

Additionally, note that `Set.fromList` in the `danglingVRFKeyHashes` computation deduplicates VRF keys, so if two pools simultaneously re-register away from the same VRF key V in the same epoch, V would only be decremented once instead of twice. The `Set` should be replaced with a list (or multiset) to correctly handle that edge case as well.
