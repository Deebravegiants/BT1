## Vulnerability Analysis

**The gap is real and concrete.** The PTQ adjusts the balance check via `paid_from_balance` but has no corresponding `paid_from_allowance` field. The allowance check in `check_and_compute_new_allowance` always reads the certified-state allowance=A, regardless of how many pending transactions have already consumed it.

**Exact attack sequence:**

1. Attacker holds FunctionCall key with `allowance=A`, account balance `B ≥ 2A`
2. Tx1 (nonce=1, cost=A) submitted → included in block B1 (not yet certified)
3. PTQ records: `paid_from_balance=A`, `max_nonce=1`
4. Tx2 (nonce=2, cost=A) submitted via RPC:
   - Balance check: `available = B - A ≥ A` → passes
   - Nonce check: `effective_nonce = max(0, 1) = 1`, nonce=2 > 1 → passes
   - Allowance check: `check_and_compute_new_allowance(allowance=A, cost=A)` → `new_allowance=0` → **passes** (certified state allowance=A, no PTQ adjustment)
5. Tx2 admitted to mempool, included in block B2
6. B1 certified: tx1 executes, allowance → 0
7. B2 certified: tx2 executes, allowance=0, cost=A → `NotEnoughAllowance` runtime failure

**Why the chunk producer also admits tx2:** When the chunk producer processes tx2, it reads certified state (allowance=A, since tx1 is included but not yet certified), applies PTQ constraints (paid_from_balance=A, max_nonce=1), and calls `check_and_compute_new_allowance(A, A)=0` → passes. Tx2 is included. [8](#0-7)

### Title
FunctionCall Access Key Allowance Not Tracked in PendingConstraints — Two Transactions Can Each Consume Full Allowance Before Certification - (`runtime/runtime/src/verifier.rs`, `chain/client/src/pending_transaction_queue.rs`)

### Summary
`PendingConstraints` has no `paid_from_allowance` field. The PTQ tracks `paid_from_balance` (which adjusts the balance check) but never tracks consumed allowance. `check_and_compute_new_allowance` always reads the certified-state allowance, so a second transaction costing the full allowance passes RPC and chunk-production validation while the first is pending, resulting in both being included in blocks with one failing at runtime.

### Finding Description
`verify_and_charge_tx_ephemeral` applies `pending.paid_from_balance` to reduce available balance but calls `check_and_compute_new_allowance(access_key, ..., total_cost)` with the raw certified-state access key. Since `PendingConstraints` has no allowance field and `PendingTransactionQueue` tracks no per-key allowance consumption, the allowance check is always performed against the certified-state value. An attacker can submit tx1 (cost=A, allowance=A), wait for it to be included (PTQ records paid_from_balance=A), then submit tx2 (cost=A): the balance check is reduced by A but the allowance check still sees allowance=A and passes.

### Impact Explanation
Both transactions are admitted to the mempool and included in separate blocks. Tx2 fails at runtime with `NotEnoughAllowance`. The network includes a predictably-failing transaction, wasting block space. The attacker pays gas for both transactions but achieves double-inclusion of allowance-consuming transactions.

### Likelihood Explanation
Requires: (1) FunctionCall key with finite allowance=A; (2) account balance ≥ 2A; (3) timing window between tx1 block inclusion and tx2 submission (before tx1 is certified). This is straightforward to engineer deliberately and can occur naturally under normal network latency.

### Recommendation
Add a `paid_from_allowance` field to `PendingConstraints` and `PendingAccount`. In `add_chunk_transactions`, for access key transactions using a FunctionCall key, record `total_cost` as `paid_from_allowance` keyed by `(account_id, public_key)`. In `verify_and_charge_tx_ephemeral`, subtract `pending.paid_from_allowance` from the certified-state allowance before calling `check_and_compute_new_allowance`, analogous to how `paid_from_balance` adjusts the balance check.

### Proof of Concept
Test-loop test: create account with FunctionCall key (allowance=A, balance=2A+ε). Submit tx1 (cost=A, nonce=1). Wait for tx1 to be included in a block (but not certified). Submit tx2 (cost=A, nonce=2) via RPC. Assert tx2 returns `ValidTx` (not `InvalidTx`). Run until both blocks are certified. Assert tx2's execution outcome is `Failure(NotEnoughAllowance)`.

            cost: total_cost,
        })
    })?;

}
