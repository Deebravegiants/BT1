### Title
Sub-transaction Plutus ExUnits Bypass `maxTxExUnits` Limit in Dijkstra Era — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`)

---

### Summary

The Dijkstra era introduces nested ("sub") transactions. Sub-transactions can carry Plutus redeemers with their own `ExUnits` budgets. The top-level UTXO rule validates only the top-level transaction's `totExUnits` against `ppMaxTxExUnitsL`. The `SUBUTXO` transition rule never calls `validateExUnitsTooBigUTxO`, and `totExUnits` is a standalone function that reads only `witsTxL . rdmrsTxWitsL` of the top-level transaction — it never traverses `subTransactionsTxBodyL`. An unprivileged transaction author can therefore embed sub-transactions whose redeemers declare arbitrarily large `ExUnits`, bypassing the per-transaction resource cap and causing validator nodes to spend computation that is neither bounded by the protocol parameter nor reflected in the minimum fee.

---

### Finding Description

**Root cause — `dijkstraSubUtxoTransition` omits the ExUnits check**

`dijkstraSubUtxoTransition` in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs` performs the following validations for each sub-transaction: [1](#0-0) 

The rule checks validity interval, forecast, output sizes, input set, bad inputs, network IDs, and output minimums — but it never calls `validateExUnitsTooBigUTxO`. There is no per-sub-transaction ExUnits cap.

**The top-level check does not cover sub-transactions**

The top-level `dijkstraUtxoTransition` does call the check: [2](#0-1) 

However, `Alonzo.validateExUnitsTooBigUTxO` delegates to `totExUnits tx`: [3](#0-2) 

`totExUnits` is a standalone (non-overridable) function that reads only the top-level transaction's witness redeemers:

```haskell
totExUnits = foldMap snd . Map.elems . unRedeemers . (^. witsTxL . rdmrsTxWitsL)
```

Sub-transactions reside in `subTransactionsTxBodyL` and carry their own `witsTxL`. They are invisible to `totExUnits`.

**Sub-transactions provably carry redeemers**

`validateBatchCollateral` — called unconditionally in `dijkstraUtxoTransition` — explicitly inspects sub-transaction redeemers: [4](#0-3) 

This confirms sub-transactions are a legitimate carrier of Plutus redeemers with `ExUnits`. The collateral check was added for sub-transaction redeemers, but the corresponding ExUnits cap was not.

**Incorrect "Impossible" assertion**

The injection helper for `SUBUTXO` failures contains: [5](#0-4) 

This `error` call asserts that `ExUnitsTooBigUTxO` can never arise for a sub-transaction. Given that sub-transactions demonstrably carry redeemers, this assumption is incorrect and the guard is never reached — it is simply never checked.

**Block-level cap is equally bypassed**

`validateExUnits` in the BBODY rule also uses `foldMap totExUnits txs`: [6](#0-5) 

Sub-transaction ExUnits are excluded here as well, so `ppMaxBlockExUnitsL` is also bypassable.

---

### Impact Explanation

An attacker-controlled transaction author can submit a Dijkstra top-level transaction whose top-level redeemers declare `ExUnits 0 0` (trivially passing `validateExUnitsTooBigUTxO`) while embedding one or more sub-transactions whose redeemers declare `ExUnits` up to `maxBound :: Natural`. The Plutus interpreter is invoked with those budgets; actual CPU/memory consumption is bounded only by the script's real execution cost, not by any ledger-enforced cap. Because `totExUnits` omits sub-transaction redeemers, the minimum-fee formula (`txscriptfee prices totExUnits`) also undercharges the attacker. The result is that validator nodes spend computation resources exceeding the intended per-transaction and per-block limits while the attacker pays fees calibrated only to the (near-zero) top-level ExUnits. This matches the allowed **Medium** impact: *attacker-controlled transactions exceed intended validation limits and modify fees outside design parameters*.

---

### Likelihood Explanation

The Dijkstra era is the next scheduled era and its code is present in the production repository. Any unprivileged user who can submit a transaction can trigger this path — no privileged key, governance majority, or external dependency is required. The construction is straightforward: embed a computationally expensive Plutus script in a sub-transaction with a large declared ExUnits budget and a near-zero top-level ExUnits declaration. Likelihood is **Medium** (era not yet live on mainnet, but the code path is fully reachable once deployed).

---

### Recommendation

1. **Short term**: In `dijkstraSubUtxoTransition`, add a call to `validateExUnitsTooBigUTxO` (or a Dijkstra-specific equivalent) for each sub-transaction, using the same `ppMaxTxExUnitsL` bound.
2. **Short term**: Extend `totExUnits` (or introduce a Dijkstra-specific `totBatchExUnits`) to sum ExUnits across all sub-transactions and use it in both the UTXO and BBODY checks.
3. **Short term**: Remove or correct the `error "Impossible: ExUnitsTooBigUTxO for SUBUTXO"` guard in `dijkstraUtxoToDijkstraSubUtxoPredFailure`; it is reachable once the check is added.
4. **Long term**: Audit all resource-limit checks (`maxTxSize`, `maxCollateralInputs`, `maxRefScriptSizePerTx`) to confirm they account for the full batch of sub-transactions, not only the top-level transaction.

---

### Proof of Concept

```
1. Compile a Plutus V4 script S that loops for exactly N steps
   (where N = exUnitsSteps maxTxExUnits).

2. Lock a UTxO U with script S in the Dijkstra UTxO set.

3. Construct a sub-transaction subTx:
     inputs  = { U }
     redeemer for U = (datum, ExUnits { mem = maxBound, steps = N })

4. Construct a top-level Dijkstra transaction topTx:
     subTransactions = { subTx }
     redeemers       = {}          -- empty: totExUnits topTx = ExUnits 0 0
     fee             = minfee pp topTx   -- computed without sub-tx ExUnits

5. Submit topTx.

Outcome:
  - validateExUnitsTooBigUTxO passes (totExUnits topTx = 0 ≤ maxTxExUnits).
  - dijkstraSubUtxoTransition runs without any ExUnits check.
  - The Plutus interpreter executes S for N steps inside subTx.
  - The attacker pays a fee reflecting 0 script ExUnits.
  - Nodes spend N Plutus steps of computation per such transaction,
    unbounded by the maxTxExUnits protocol parameter.
```

validateExUnitsTooBigUTxO pp tx =
  failureUnless (pointWiseExUnits (<=) totalExUnits maxTxExUnits) $
    ExUnitsTooBigUTxO Mismatch {mismatchSupplied = totalExUnits, mismatchExpected = maxTxExUnits}
  where
    maxTxExUnits = pp ^. ppMaxTxExUnitsL
    -- This sums up the ExUnits for all embedded Plutus Scripts anywhere in the transaction:
    totalExUnits = totExUnits tx
