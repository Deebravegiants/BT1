Audit Report

## Title
Stale `ethValueInWithdrawal` Snapshot Causes Persistent rsETH Price Understatement - (`contracts/LRTConverter.sol`)

## Summary
`LRTConverter.ethValueInWithdrawal` is snapshotted at the LST oracle price when `transferAssetFromDepositPool()` is called, but the LST is simultaneously zeroed out of LST-denominated accounting. Because LST prices (stETH, rETH, ETHx, sfrxETH) continuously increase as staking rewards accrue, the stale snapshot persistently understates the ETH value of converter holdings throughout the unstaking window. `_getTotalEthInProtocol()` consumes this stale value verbatim, causing rsETH to be priced below its true NAV, allowing new depositors to receive excess rsETH and diluting existing holders' accrued yield.

## Finding Description
**Step 1 — Snapshot on transfer:**
When an operator calls `transferAssetFromDepositPool`, the ETH value is frozen at the current oracle price:
```solidity
// LRTConverter.sol L140
ethValueInWithdrawal += (_amount * lrtOracle.getAssetPrice(_asset)) / 1e18;
```
The LST tokens are simultaneously excluded from LST-denominated accounting:
```solidity
// LRTDepositPool.sol L460
assetLyingInConverter = 0; // assets in converter are accounted in their eth value => getETHDistributionData()
```

**Step 2 — Stale value consumed in ETH accounting:**
`getETHDistributionData()` reads the frozen snapshot directly:
```solidity
// LRTDepositPool.sol L498-499
address lrtConverter = lrtConfig.getContract(LRTConstants.LRT_CONVERTER);
ethLyingInConverter = ILRTConverter(lrtConverter).ethValueInWithdrawal();
```

**Step 3 — Price calculation uses stale total:**
`_getTotalEthInProtocol()` iterates all supported assets including ETH_TOKEN (price = 1e18), so `ethValueInWithdrawal` flows directly into `totalETHInProtocol`:
```solidity
// LRTOracle.sol L339-343
uint256 assetER = getAssetPrice(asset);
uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
```
Then rsETH price is computed from this understated total:
```solidity
// LRTOracle.sol L250
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

**Why existing guards fail:**
- The `pricePercentageLimit` guard only triggers on large single-update price jumps; it does not detect the slow, continuous drift caused by stale `ethValueInWithdrawal`.
- `transferAssetToDepositPool` uses the **current** price to reduce `ethValueInWithdrawal` (L160-163), which means the stale-high snapshot is reduced by a current-higher price, potentially zeroing `ethValueInWithdrawal` prematurely and compounding the understatement.
- There is no mechanism to refresh `ethValueInWithdrawal` between transfer and claim.

## Impact Explanation
**High — Theft of unclaimed yield.**

stETH (and other rebasing/appreciating LSTs) continuously accrue staking rewards, causing their ETH price to drift upward. For every day the LST sits in the converter with a stale snapshot, the gap between `ethValueInWithdrawal` and the true ETH value of those holdings widens. When `updateRSETHPrice()` is called, rsETH is priced against the understated total. Any depositor who mints rsETH during this window receives more rsETH than their ETH contribution warrants — at the direct expense of existing rsETH holders whose accrued staking yield is diluted. This is a continuous, passive transfer of yield from existing holders to new depositors, matching the "Theft of unclaimed yield" impact class.

## Likelihood Explanation
No attacker capability is required. The conditions are normal protocol operations:
1. Operator calls `transferAssetFromDepositPool` (routine rebalancing).
2. Lido withdrawal queue takes 1–14 days (normal operating condition).
3. stETH accrues ~4% APY continuously — price drift is guaranteed, not probabilistic.
4. Any user calling `depositETH` or `depositAsset` during the unstaking window passively benefits.

The only precondition is that the converter holds LSTs awaiting unstaking, which is the converter's primary purpose. This is not an edge case.

## Recommendation
Replace the ETH-value snapshot with a per-asset token-amount tracker. Compute the ETH value dynamically at price-query time:

```solidity
// LRTConverter.sol
mapping(address => uint256) public lstAmountInConverter;

function transferAssetFromDepositPool(address _asset, uint256 _amount) external ... {
    lstAmountInConverter[_asset] += _amount;
    IERC20(_asset).safeTransferFrom(...);
}

function transferAssetToDepositPool(address _asset, uint256 _amount) external ... {
    lstAmountInConverter[_asset] -= _amount;
    IERC20(_asset).safeTransfer(...);
}
```

In `getETHDistributionData()`, compute the live ETH value:
```solidity
address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
for (uint i; i < supportedAssets.length; i++) {
    address asset = supportedAssets[i];
    if (asset == LRTConstants.ETH_TOKEN) continue;
    uint256 amt = ILRTConverter(lrtConverter).lstAmountInConverter(asset);
    if (amt > 0) {
        ethLyingInConverter += (amt * lrtOracle.getAssetPrice(asset)) / 1e18;
    }
}
```
Also update `_sendEthToDepositPool` to reduce the appropriate per-asset bucket rather than a global ETH counter.

## Proof of Concept
**Minimal call sequence (no fork required):**

1. Deploy protocol with 100,000 rsETH in supply, rsETH price = 1.05 ETH.
2. Operator calls `transferAssetFromDepositPool(stETH, 10_000e18)` when stETH price = 1.05 ETH.
   - `ethValueInWithdrawal = 10_500 ETH` (snapshot)
   - stETH balance excluded from stETH accounting
3. Advance time 7 days. stETH oracle price updates to 1.058 ETH (≈0.76% increase, consistent with ~4% APY).
   - Actual ETH value of converter = 10,580 ETH
   - `ethValueInWithdrawal` = 10,500 ETH (stale, 80 ETH short)
4. Call `updateRSETHPrice()`.
   - `_getTotalEthInProtocol()` uses 10,500 ETH instead of 10,580 ETH
   - rsETH price = (totalETH - 80 ETH) / 100,000 rsETH → understated by 0.0008 ETH/rsETH
5. New depositor calls `depositETH(1000 ETH)`.
   - Receives `1000 / understated_price` rsETH — excess rsETH minted at existing holders' expense
6. Repeat for each `updateRSETHPrice()` call during the unstaking window; the dilution compounds.
