### Title
Attached NEAR Deposit Silently Consumed and Not Refunded on Attestation Re-Submission by Existing Participant — (`File: crates/contract/src/lib.rs`)

---

### Summary

`submit_participant_info` is a `#[payable]` function that accepts NEAR token deposits. Its deposit-refund logic is gated behind a boolean flag `attestation_storage_must_be_paid_by_caller`. When an **existing participant** (a current voter) **re-submits** their attestation (an update, not a first-time insertion), this flag evaluates to `false` and the entire deposit-handling block is skipped. Any NEAR tokens attached to the call are silently consumed by the contract with no refund issued and no error raised.

---

### Finding Description

In `crates/contract/src/lib.rs`, `submit_participant_info` computes:

```rust
let caller_is_not_participant = self.voter_account().is_err();
let is_new_attestation = matches!(
    attestation_insertion_result,
    ParticipantInsertion::NewlyInsertedParticipant
);

let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... deposit check and refund logic ...
    if let Some(diff) = attached.checked_sub(cost)
        && diff > NearToken::from_yoctonear(0)
    {
        Promise::new(account_id).transfer(diff).detach();
    }
}
``` [1](#0-0) 

When `is_new_attestation == false` (the TLS key already exists in `stored_attestations`, so `add_participant` returns `ParticipantInsertion::UpdatedExistingParticipant`) **and** `caller_is_not_participant == false` (the caller is already a registered voter), the flag is `false` and the `if` block is entirely bypassed. [2](#0-1) 

The function is declared `#[payable]`, so the NEAR runtime accepts any attached deposit without restriction. [3](#0-2) 

Because the deposit-handling block is skipped, `env::attached_deposit()` is never read, and no `Promise::transfer` is scheduled. The attached NEAR is absorbed into the contract's balance permanently. [4](#0-3) 

The `add_participant` path that produces `UpdatedExistingParticipant` is confirmed reachable: the TLS-key-keyed `stored_attestations` map simply overwrites the previous entry when the same account re-submits. [5](#0-4) 

---

### Impact Explanation

Any NEAR tokens attached to a re-submission call by an existing participant are permanently locked in the contract with no recovery path. The contract's accounting invariant — that a `#[payable]` function either consumes the deposit for its stated purpose or refunds the excess — is violated for this code path. This constitutes a direct, irreversible loss of the caller's NEAR balance and breaks the production safety/accounting invariant of the contract.

This maps to the **Medium** allowed impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."*

---

### Likelihood Explanation

The MPC node's `periodic_attestation_submission` task resubmits on a 1-hour cadence and, per the operational documentation, attaches `0` deposit by default. However:

1. The function is publicly callable by any account that holds a valid access key for the participant account.
2. Operators are explicitly documented to call `submit_participant_info` manually (e.g., when the node fails to submit automatically), and may attach a deposit out of caution or habit (the first-time submission requires a deposit for storage staking).
3. Any tooling or script that attaches a non-zero deposit to a re-submission — a common defensive pattern on NEAR — will silently lose those tokens.

The likelihood is **low-to-medium**: the default automated path is safe, but the manual operator path and any third-party tooling are exposed.

---

### Recommendation

Unconditionally read and refund any excess deposit at the end of `submit_participant_info`, regardless of whether `attestation_storage_must_be_paid_by_caller` is true or false. The simplest fix is to move the refund logic outside the conditional block:

```rust
let attached = env::attached_deposit();

if attestation_storage_must_be_paid_by_caller {
    let storage_used = env::storage_usage().saturating_sub(initial_storage);
    let cost = env::storage_byte_cost().saturating_mul(storage_used as u128);
    if attached < cost {
        return Err(InvalidParameters::InsufficientDeposit { ... }.into());
    }
    // cost is what we keep; refund the rest below
    let cost_to_keep = cost;
    if let Some(diff) = attached.checked_sub(cost_to_keep)
        && diff > NearToken::from_yoctonear(0)
    {
        Promise::new(account_id).transfer(diff).detach();
    }
} else {
    // No storage cost; refund the entire deposit
    if attached > NearToken::from_yoctonear(0) {
        Promise::new(account_id).transfer(attached).detach();
    }
}
```

This mirrors the pattern already used in `require_deposit`, `propose_update`, and the `submit_participant_info` storage-charging branch itself. [6](#0-5) 

---

### Proof of Concept

1. Participant `alice.near` has already submitted a valid attestation; her TLS key is present in `stored_attestations`. She is a registered voter (`voter_account()` returns `Ok`).
2. Alice's node (or an operator script) calls `submit_participant_info` with a fresh attestation and attaches `1 NEAR` as a deposit (e.g., following the documented manual-submission procedure that requires a deposit for first-time joiners).
3. `add_participant` succeeds and returns `ParticipantInsertion::UpdatedExistingParticipant`. [7](#0-6) 
4. `is_new_attestation` is `false`; `caller_is_not_participant` is `false`; therefore `attestation_storage_must_be_paid_by_caller` is `false`. [8](#0-7) 
5. The `if attestation_storage_must_be_paid_by_caller` block is skipped entirely. No refund Promise is created.
6. The function returns `Ok(())`. Alice's `1 NEAR` is permanently absorbed into the contract balance with no log entry and no error.
