Audit Report

## Title
Token Deposit Yields Zero wrsETH/rsETH Due to Integer Division Truncation — (`contracts/pools/RSETHPoolV3.sol`, `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV3WithNativeChainBridge.sol`, `RSETHPoolNoWrapper.sol`)

## Summary
The token-deposit path in all four L2 pool contracts computes the output amount via plain integer division in `viewSwapRsETHAmountAndFee(amount, token)`. When `amountAfterFee * tokenToETHRate < rsETHToETHrate`, Solidity truncates the result to zero. Because the token transfer occurs before the output is computed and no zero-output guard exists, a depositor's tokens are permanently absorbed by the pool while they receive zero wrsETH or rsETH in return.

## Finding Description
In `RSETHPoolV3.sol`, the overloaded `viewSwapRsETHAmountAndFee(uint256 amount, address token)` computes:

```solidity
// RSETHPoolV3.sol L324-334
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;
uint256 rsETHToETHrate = getRate();
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

When `amountAfterFee * tokenToETHRate < rsETHToETHrate`, `rsETHAmount` is truncated to zero. The deposit function at L282–292 only guards against `amount == 0`; it does not guard against a zero output:

```solidity
// RSETHPoolV3.sol L282-292
if (amount == 0) revert InvalidAmount();
IERC20(token).safeTransferFrom(msg.sender, address(this), amount);
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
feeEarnedInToken[token] += fee;
wrsETH.mint(msg.sender, rsETHAmount);   // mints 0 — no revert
```

The token transfer is irrevocable. `wrsETH.mint(msg.sender, 0)` (or `rsETH.safeTransfer(msg.sender, 0)` in `RSETHPoolNoWrapper`) succeeds silently. The identical pattern is confirmed at:
- `RSETHPoolV3ExternalBridge.sol` L401–411, L442–452
- `RSETHPoolV3WithNativeChainBridge.sol` L318–328, L360–370
- `RSETHPoolNoWrapper.sol` L260–270, L301–311

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A depositor who sends a dust amount of a supported ERC-20 token (especially one with fewer than 18 decimals) receives zero wrsETH/rsETH. Their tokens are permanently transferred into the pool and become part of the bridgeable balance with no recourse. The protocol does not lose value; the depositor does. This matches the allowed impact "Low. Contract fails to deliver promised returns, but doesn't lose value."

## Likelihood Explanation
**Low.** The affected deposit amounts are sub-cent for most tokens (e.g., for USDC with `tokenToETHRate ≈ 3e14` and `rsETHToETHrate ≈ 1.05e18`, the threshold is `amountAfterFee < 3500` base units, i.e., < 0.0035 USDC). Gas costs far exceed the value lost in normal usage. However, the path is fully permissionless, requires no special privileges, and can be triggered inadvertently by integrations or scripts that do not pre-simulate output.

## Recommendation
Add an explicit zero-output revert immediately after computing the swap output, either inside `viewSwapRsETHAmountAndFee` or at the call site in each `deposit` function:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
if (rsETHAmount == 0) revert InvalidAmount();
```

Additionally, consider adding a per-token minimum deposit amount analogous to `minAmountToDeposit` in `LRTDepositPool` (L657–658), which already enforces `depositAmount >= minAmountToDeposit` before minting.

## Proof of Concept
1. A supported token with 6 decimals (e.g., USDC) is added to `RSETHPoolV3` via `addSupportedToken` with a valid oracle returning `tokenToETHRate ≈ 3e14`.
2. Any user calls `deposit(usdcAddress, 1000, "")` — depositing 1000 USDC base units (0.001 USDC).
3. `viewSwapRsETHAmountAndFee(1000, usdcAddress)` computes:
   - `fee = 1000 * feeBps / 10_000` → 0 (assuming `feeBps < 10`)
   - `amountAfterFee = 1000`
   - `rsETHAmount = 1000 * 3e14 / 1.05e18 = 3e17 / 1.05e18 = 0`
4. `IERC20(usdc).safeTransferFrom(msg.sender, pool, 1000)` executes — tokens leave the user.
5. `wrsETH.mint(msg.sender, 0)` executes — user receives nothing.
6. The 1000 USDC base units remain in the pool, credited to no one.

Foundry fuzz test plan: fuzz `amount` in `[1, threshold)` for each supported token, assert `rsETHAmount == 0` from `viewSwapRsETHAmountAndFee`, then call `deposit` and assert user wrsETH balance is unchanged while pool token balance increases.
