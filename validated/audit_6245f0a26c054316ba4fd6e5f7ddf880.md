### Title
Inconsistent CKD Response Validation Across `app_public_key` Variants Allows Single Attested Participant to Forge Key Derivation Output - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_ckd` validates the CKD response cryptographically only when `request.app_public_key` is the `AppPublicKeyPV` variant. When the `AppPublicKey` variant is used, the validation branch is entirely empty — a single malicious attested participant can call `respond_ckd` with any arbitrary `CKDResponse`, bypassing the `ckd_output_check` that guards the `AppPublicKeyPV` path, and the contract will accept and return the forged key material to the user.

### Finding Description

In `respond_ckd`, after confirming the caller is an attested participant, the contract performs a `match` on `request.app_public_key`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no validation whatsoever
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the response against the app public key and the domain's BLS12-381 root key. For `AppPublicKey`, the arm is a no-op — the `CKDResponse` is accepted unconditionally and immediately forwarded to `pending_requests::resolve_yields_for`, which resolves the pending yield and delivers the response to the waiting caller. [2](#0-1) 

This is structurally identical to the reference vulnerability: one branch of a conditional omits a critical component (the cryptographic output check) that all other branches include, creating an exploitable inconsistency.

### Impact Explanation

A single malicious attested participant (strictly below the signing threshold) can:

1. Observe a pending `CKDRequest` with `app_public_key = AppPublicKey(...)` in the contract state.
2. Call `respond_ckd` with that request and an arbitrary, attacker-crafted `CKDResponse` (e.g., a forged BLS12-381 derived key).
3. The contract skips `ckd_output_check`, accepts the response, and resolves the yield — delivering the forged key material to the user.

The user receives a confidential derived key that was never produced by the honest MPC threshold computation. This constitutes **unauthorized confidential key derivation output without the required participant authorization**, matching the Critical allowed impact: *"Unauthorized transaction execution, threshold signature issuance, or confidential key derivation output without the required participant authorization."*

### Likelihood Explanation

Any single attested participant (one node operator) can exploit this. No threshold collusion is required. The `AppPublicKey` variant is a valid, user-submittable request type exposed through the contract's public API. The attacker only needs to be faster than the honest nodes in calling `respond_ckd` after a matching request appears on-chain — a realistic race condition in a live network.

### Recommendation

Apply `ckd_output_check` (or an equivalent cryptographic validation) consistently for **all** `app_public_key` variants. If `AppPublicKey` genuinely carries no verifiable proof, the contract should either:

- Reject `AppPublicKey` requests in `respond_ckd` until a validation path is defined, or
- Require that `AppPublicKey` requests can only be resolved after threshold-many matching `respond_ckd` calls (a quorum gate), analogous to how the signing threshold guards `respond`.

The fix mirrors the reference report's recommendation: handle the omitted component consistently across all branches.

### Proof of Concept

1. Deploy the contract in `Running` state with a BLS12-381 CKD domain.
2. User calls `request_app_private_key` with `app_public_key = AppPublicKey(some_pk)`.
3. Malicious attested participant (node A, acting alone) constructs a `CKDResponse` containing an arbitrary forged BLS12-381 key share output.
4. Node A calls `respond_ckd(request, forged_response)`.
5. The `match` arm for `AppPublicKey` is a no-op; `ckd_output_check` is never called.
6. `resolve_yields_for` resolves the pending yield with the forged response.
7. The user's promise callback receives the forged key material — a confidential derived key that was never produced by the honest threshold protocol. [3](#0-2)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
