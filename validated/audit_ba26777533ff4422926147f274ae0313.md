### Title
Missing Minimum rsETH Output Parameter in L2 Pool `deposit()` Functions Leaves Users Without Slippage Protection - (File: `contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`, `contracts/pools/RSETHPoolV2.sol`, `contracts/pools/RSETHPoolV2ExternalBridge.sol`)

---

### Summary

All L2 pool `deposit()` functions compute the rsETH mint amount from a live oracle rate at execution time but accept no `minRSETHAmountExpected` parameter. If the oracle rate increases between the moment a user submits a transaction and the moment it is included in a block, the user receives fewer wrsETH tokens than they observed when constructing the transaction, with no on-chain mechanism to revert. The L1 `LRTDepositPool` already enforces this protection, making the omission in the L2 pool contracts an inconsistency with demonstrated protocol intent.

---

### Finding Description

Every L2 pool deposit entry point follows the same pattern: compute `rsETHAmount` from `viewSwapRsETHAmountAndFee`, which divides the deposit amount by `getRate()` fetched from the oracle at execution time, then mint that amount unconditionally.

`RSETHPoolV3.sol` – ETH deposit:
```solidity
function deposit(string memory referralId) external payable ... {
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);   // no minimum check
}
``` [1](#0-0) 

`RSETHPoolV3.sol` – token deposit:
```solidity
function deposit(address token, uint256 amount, string memory referralId) external ... {
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
    feeEarnedInToken[token] += fee;
    wrsETH.mint(msg.sender, rsETHAmount);   // no minimum check
}
``` [2](#0-1) 

The same pattern is present in `RSETHPoolV3ExternalBridge.sol`: [3](#0-2) 

And in `RSETHPoolV2.sol` and `RSETHPoolV2ExternalBridge.sol`: [4](#0-3) [5](#0-4) 

The rsETH amount is computed as:
```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [6](#0-5) 

where `rsETHToETHrate` is read from the oracle at execution time. Because rsETH is a yield-bearing token, its ETH rate increases monotonically over time. Any increase in the rate between tx submission and inclusion means the user receives fewer wrsETH tokens than they expected.

**Contrast with L1 `LRTDepositPool`**, which explicitly enforces a caller-supplied minimum:
```solidity
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [7](#0-6) 

Both `depositETH` and `depositAsset` on L1 accept and enforce `minRSETHAmountExpected`: [8](#0-7) [9](#0-8) 

The L2 pool contracts provide no equivalent protection.

---

### Impact Explanation

**Low. Contract fails to deliver promised returns, but doesn't lose value.**

A user calls `viewSwapRsETHAmountAndFee` off-chain to preview the rsETH they will receive, then submits `deposit()`. If the oracle rate is updated (or naturally drifts upward due to accrued staking rewards) before the transaction is mined, the user receives fewer wrsETH tokens than the preview showed. The user cannot specify a minimum acceptable output, so the transaction cannot revert to protect them. The deposited ETH/LST is not stolen — it is held by the pool — but the user receives less rsETH than the protocol implicitly promised at the time of their decision.

---

### Likelihood Explanation

The rsETH/ETH oracle rate increases continuously as staking rewards accrue (~4% APY). On L2 chains where transactions may be delayed or reordered, the gap between preview and execution can be non-trivial. Additionally, the oracle address itself can be replaced via `setRSETHOracle` (TIMELOCK_ROLE), and the fee basis points can be changed via `setFeeBps` (TIMELOCK_ROLE) — either change between tx submission and execution would silently reduce the user's output. No attacker is required; natural oracle rate drift is sufficient to trigger the discrepancy on every deposit.

---

### Recommendation

Add a `uint256 minRSETHAmountExpected` parameter to every public `deposit()` overload in all L2 pool contracts, mirroring the pattern already used in `LRTDepositPool._beforeDeposit()`:

```solidity
function deposit(string memory referralId, uint256 minRSETHAmountExpected)
    external payable nonReentrant whenNotPaused limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    ...
}
```

Apply the same guard to the token-deposit overloads in `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV2`, and `RSETHPoolV2ExternalBridge`.

---

### Proof of Concept

1. User calls `viewSwapRsETHAmountAndFee(1 ether)` off-chain and observes `rsETHAmount = X`.
2. User submits `deposit{value: 1 ether}("ref")`.
3. Before the tx is mined, the oracle rate increases (e.g., due to a scheduled oracle update or natural yield accrual).
4. `deposit()` executes, computes a new (lower) `rsETHAmount = X - delta`, and mints `X - delta` wrsETH to the user.
5. The user receives fewer wrsETH than expected with no revert and no recourse.

The same scenario applies to token deposits in `RSETHPoolV3.deposit(address token, uint256 amount, string memory referralId)` and its counterpart in `RSETHPoolV3ExternalBridge`. [10](#0-9)

        uint256 amount,
        address token
    )
        public
        view
        onlySupportedToken(token)

    {
        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();
