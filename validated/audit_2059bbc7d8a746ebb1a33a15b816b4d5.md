### Title
Byzantine New Participant Can Forge `AppPublicKey` CKD Responses During Resharing ‚Äî (`crates/contract/src/lib.rs`)

### Summary
During resharing, `respond_ckd` authorizes callers via `active_participants()`, which returns the **new** proposed participant set. The `AppPublicKey` variant of a CKD request has no on-chain cryptographic output check. A single Byzantine new participant (attested, in the proposed set) can call `respond_ckd` with arbitrary `(big_y, big_c)` for any in-flight `AppPublicKey` CKD request that was submitted before resharing began, resolving the yield with a forged response.

### Finding Description

`respond_ckd` in `crates/contract/src/lib.rs` (lines 654‚Äì689) has two independent weaknesses that compose into an exploitable path:

1. **No output check for `AppPublicKey`**: The match at line 675‚Äì682 is a no-op for the `AppPublicKey` variant. Only `AppPublicKeyPV` triggers `ckd_output_check`. [1](#0-0) 

2. **Wrong participant set used during resharing**: `assert_caller_is_attested_participant_and_protocol_active` calls `self.protocol_state.active_participants()`, which during `Resharing` returns `state.resharing_key.proposed_parameters().participants()` ‚Äî the **new** set, not the old set that holds the key shares needed to compute a correct CKD response. [6](#0-5) 

A new participant who is attested but does not hold the old key shares passes both guards and can call `resolve_yields_for` with arbitrary bytes, draining the yield queue for any pending `AppPublicKey` CKD request. [7](#0-6) 

### Impact Explanation

The user's pending CKD yield is consumed with a forged `(big_y, big_c)`. The user receives a cryptographically invalid key derivation result. Because `resolve_yields_for` removes the entry from `pending_ckd_requests`, the user cannot retry ‚Äî the request lifecycle is permanently terminated with a bad output. The `AppPublicKey` variant is a documented production feature (described as "legacy" in the README but fully supported). [8](#0-7) 

### Likelihood Explanation

The window is the resharing period. Any single Byzantine participant in the **new** proposed set who is attested can exploit this. They do not need to collude with any other participant. The attack requires only: (1) a pending `AppPublicKey` CKD request exists, (2) the contract is in `Resharing` state, (3) the attacker is an attested new participant. All three conditions are reachable in normal operation.

### Recommendation

Two independent fixes, either sufficient alone:

1. **Restrict `respond_ckd` to old participants during resharing**: Add a separate check that during `Resharing`, the caller must be in `previous_running_state.parameters.participants()`, not the proposed new set. This mirrors the pattern already used in `authenticate_update_vote` and `vote_cancel_resharing`. [9](#0-8) 

2. **Add an output check for `AppPublicKey`**: The `AppPublicKey` variant carries a G1 point `pk`. The contract can verify `e(big_c, g2) = e(pk, g2)^y ¬∑ e(H(app_id), network_pk)` using the same pairing infrastructure already used for `AppPublicKeyPV`. This would make forging impossible regardless of who calls `respond_ckd`.

### Proof of Concept

The existing unit test at `lib.rs:3403‚Äì3441` already demonstrates that `respond_ckd` with `AppPublicKey` and garbage `big_y`/`big_c` (`[1u8;48]`, `[2u8;48]`) succeeds when the caller is an active attested participant. [10](#0-9) 

To prove the resharing variant: transition the contract to `Resharing` with a new participant set, mock attestations for the new set, set the signer context to a new participant not in the old set, and call `respond_ckd` with the same garbage values ‚Äî the call succeeds and the yield is resolved, as `active_participants()` returns the new set and the `AppPublicKey` branch performs no verification.

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
