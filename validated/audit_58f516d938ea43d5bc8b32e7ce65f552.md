Audit Report

## Title
Missing `priceLimitX64 != 0` Guard in Initial Price-Limit Check Silently Blocks All `!zeroForOne` Swaps with No Price Limit — (File: `metric-core/contracts/MetricOmmPool.sol`)

## Summary
Both `_swapToken1ForToken0SpecifiedInput` and `_swapToken1ForToken0SpecifiedOutput` perform an initial early-exit check `if (params.priceLimitX64 <= initialPriceX64)` without the `priceLimitX64 != 0` guard that is explicitly present in the inner swap loop of `_swapToken1ForToken0SpecifiedInput`. When a caller passes `priceLimitX64 == 0` (the conventional "no upper price limit" sentinel), the condition `0 <= initialPriceX64` is always true because `initialPriceX64` is derived from a strictly positive oracle price, causing both functions to immediately return zero amounts. The swap transaction completes without reverting but transfers nothing to the recipient.

## Finding Description

The inner loop of `_swapToken1ForToken0SpecifiedInput` explicitly treats `priceLimitX64 == 0` as "no limit":

```solidity
if (params.priceLimitX64 != 0 && params.priceLimitX64 <= upperPriceX64) {
    break;
}
``` [1](#0-0) 

But the initial guard in both `!zeroForOne` functions lacks this protection:

```solidity
// _swapToken1ForToken0SpecifiedInput (L970)
if (params.priceLimitX64 <= initialPriceX64) { return (0, 0, 0); }

// _swapToken1ForToken0SpecifiedOutput (L888)
if (params.priceLimitX64 <= initialPriceX64) { return (0, 0, 0, 0); }
``` [2](#0-1) [3](#0-2) 

`initialPriceX64` is computed in `_getInitialStateForSwap` via `SwapMath.calculatePriceAtBinPosition(lowerPriceX64, upperPriceX64, curPosInBinCache, posRounding)`, where `lowerPriceX64` and `upperPriceX64` are derived from `params.midPriceX64 = sqrt(bid * ask)`. [4](#0-3) 

`_getBidAndAskPriceX64` enforces `bid > 0` and `bid < ask`, so `midPriceX64 > 0` always, and `initialPriceX64` is a weighted average of two positive values — it is always strictly greater than zero. [5](#0-4) 

`calculatePriceAtBinPosition` returns `(lowerPriceX64 * maxSubCurrPos + upperPriceX64 * position) / MAX_POS_BIN`, which is always `> 0` when both bounds are positive. [6](#0-5) 

The `swap` entry point imposes no non-zero requirement on `priceLimitX64`: [7](#0-6) 

Exploit flow:
1. Caller invokes `swap(recipient, false, amountSpecified, 0, callbackData, "")`.
2. `_executeSwap` routes to `_swapToken1ForToken0SpecifiedInput` or `_swapToken1ForToken0SpecifiedOutput` with `params.priceLimitX64 = 0`.
3. `initialPriceX64 > 0` always → `0 <= initialPriceX64` → `true` → function returns `(0, 0, 0)` / `(0, 0, 0, 0)` immediately.
4. Back in `swap`: `amount0Delta = 0`, `amount1Delta = 0`. `if (amount0Delta < 0)` is false — no token0 sent to recipient. Callback is invoked with `(0, 0)` — caller pays nothing. `if (amount1Delta > 0 && ...)` is false — no `IncorrectDelta` revert. [8](#0-7) 

The symmetric `zeroForOne` path is **not** affected: its initial check is `if (params.priceLimitX64 >= initialPriceX64)`, which evaluates to `0 >= initialPriceX64` → false when `priceLimitX64 == 0`, so it does not early-exit. [9](#0-8) 

## Impact Explanation
Every `!zeroForOne` swap (token1 → token0) submitted with `priceLimitX64 == 0` silently returns `(0, 0)`. The recipient receives zero token0, the caller pays nothing, and pool state is unchanged. This constitutes broken core pool swap functionality: the entire `!zeroForOne` swap path is unusable for any router or integrator following the Uniswap V3 convention of passing `0` for "no price limit." The `MetricOmmSimpleRouter` periphery contract is directly affected.

## Likelihood Explanation
Medium. The `swap` function accepts `priceLimitX64` as a raw `uint128` with no non-zero requirement. The inner loop's explicit `!= 0` guard is strong evidence the protocol intended `0` to mean "no limit." Any router or integrator built on this convention will silently fail on all `!zeroForOne` swaps. The condition is trivially reachable by any unprivileged caller.

## Recommendation
Add the `priceLimitX64 != 0` guard to the initial check in both functions, matching the inner loop pattern:

```solidity
// _swapToken1ForToken0SpecifiedInput
if (params.priceLimitX64 != 0 && params.priceLimitX64 <= initialPriceX64) {
    return (0, 0, 0);
}

// _swapToken1ForToken0SpecifiedOutput
if (params.priceLimitX64 != 0 && params.priceLimitX64 <= initialPriceX64) {
    return (0, 0, 0, 0);
}
```

## Proof of Concept
1. Deploy a pool with token0/token1 and add liquidity across bins.
2. Call `pool.swap(recipient, false, 1e18, 0, callbackData, "")` — `zeroForOne = false`, `priceLimitX64 = 0`.
3. `_executeSwap` routes to `_swapToken1ForToken0SpecifiedInput` with `params.priceLimitX64 = 0`.
4. `_getInitialStateForSwap` computes `initialPriceX64 > 0` from the current bin position and oracle-derived `midPriceX64`.
5. `0 <= initialPriceX64` → `true` → function returns `(0, 0, 0)` immediately.
6. `amount0Delta = 0`, `amount1Delta = 0`. Callback invoked with `(0, 0)`. No `IncorrectDelta` revert.
7. `swap` returns `(0, 0)`. Recipient receives 0 token0. Swap silently fails.
8. Confirm by asserting `token0.balanceOf(recipient) == 0` and `token1.balanceOf(caller)` is unchanged after the call.
