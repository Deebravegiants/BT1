### Title
Missing On-Chain Response Verification for `AppPublicKey` CKD Variant Allows Single Malicious Participant to Forge Key Derivation Output - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` in the MPC contract applies a cryptographic pairing check (`ckd_output_check`) only when the request uses the `AppPublicKeyPV` variant. When the request uses the `AppPublicKey` variant, the contract performs **no verification** of the submitted response before resolving the pending yield. A single malicious attested participant acting as the CKD leader can submit an arbitrary forged `(big_y, big_c)` pair for any `AppPublicKey`-type CKD request, and the contract will accept and deliver it unconditionally — bypassing the threshold correctness guarantee.

---

### Finding Description

In `respond_ckd`, after the caller is authenticated as an attested participant, the response is conditionally verified:

```rust
// crates/contract/src/lib.rs lines 675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically binds the response to the MPC network's public key and the user's ephemeral key pair. [2](#0-1) 

For `AppPublicKey`, the arm is an empty block — the contract immediately proceeds to resolve the pending yield with whatever `(big_y, big_c)` the caller supplied: [3](#0-2) 

The `AppPublicKey` variant is the original, widely-used CKD mode (it is the only mode exercised in the E2E verification test): [4](#0-3) 

The caller of `respond_ckd` must pass `assert_caller_is_attested_participant_and_protocol_active`, so the attacker must be a legitimately attested participant. However, only **one** participant (the leader) calls `respond_ckd` per request — the contract does not aggregate or cross-check responses from multiple nodes. [5](#0-4) 

---

### Impact Explanation

The CKD protocol's threshold guarantee is that the correct encrypted secret is produced only when a threshold of honest participants cooperate. The contract's role is to enforce that the submitted result actually satisfies the protocol equation before delivering it to the caller. For `AppPublicKeyPV` this enforcement exists; for `AppPublicKey` it is absent.

A single Byzantine attested participant who becomes the CKD leader for an `AppPublicKey` request can:

1. Participate honestly in the threshold computation (so the threshold is not violated at the cryptographic layer).
2. Discard the correct result and call `respond_ckd` with an arbitrary forged `(big_y, big_c)`.
3. The contract resolves the pending yield with the forged response; the requesting application receives a wrong derived key.

Because the derived key is the output of the CKD protocol — the secret the TEE application will use to protect sensitive material — delivering a wrong key constitutes **confidential key derivation output produced without the required threshold participant authorization**. The requesting application has no way to distinguish a correct from a forged response at the contract level; off-chain verification requires the application to already possess its private key `a` and the MPC public key, which is an additional burden not enforced by the protocol.

Impact class: **Critical** — confidential key derivation output without the required participant authorization (single Byzantine participant below signing threshold).

---

### Likelihood Explanation

Any attested participant can be elected leader for a CKD session. The attack requires only one compromised or malicious node — strictly below the signing threshold — and no collusion. The `AppPublicKey` variant is the default and most common CKD mode. Likelihood: **High**.

---

### Recommendation

Apply an equivalent on-chain binding check for the `AppPublicKey` variant. Because `AppPublicKey` supplies only `pk1` (a G1 point) and not `pk2`, the existing `ckd_output_check` pairing equation cannot be reused directly. Options:

1. **Require `AppPublicKeyPV` for all new CKD requests** and deprecate the unverifiable `AppPublicKey` variant.
2. **Add a G1-only binding check** for `AppPublicKey`: verify that `big_c = a·big_y + hash·MPC_pk` holds in G1 (requires a different equation derivable from the protocol spec).
3. At minimum, **document explicitly** that `AppPublicKey` requests carry no on-chain response integrity guarantee and that callers must verify off-chain, so that integrators are not misled into assuming the contract enforces correctness for both variants.

---

### Proof of Concept

1. Attacker controls one attested participant node (below threshold).
2. A user submits a `request_app_private_key` call with `CKDAppPublicKey::AppPublicKey(pk1)`.
3. The threshold CKD computation runs; the attacker's node is elected leader.
4. Instead of submitting the correct `(big_y, big_c)`, the attacker calls `respond_ckd` with `big_y = G1_generator` and `big_c = G1_generator` (arbitrary values).
5. `respond_ckd` passes `assert_caller_is_attested_participant_and_protocol_active`, enters the `AppPublicKey` arm (empty block), and calls `resolve_yields_for` with the forged response.
6. The contract resolves the yield; the user's application receives `(big_y=G1, big_c=G1)` and derives a wrong key — with no contract-level rejection. [6](#0-5)

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
