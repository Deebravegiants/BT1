Audit Report

## Title
rsETH Transfers Bypass Pause — Missing `whenNotPaused` in `RSETH._transfer` (File: contracts/RSETH.sol)

## Summary

`RSETH._transfer` (L287–291) does not include the `whenNotPaused` modifier, while both `mint` (L229–240) and `burnFrom` (L245–248) do. As a result, any rsETH holder can call the standard ERC20 `transfer` or `transferFrom` functions to move tokens freely while the contract is paused, defeating the intended operational freeze.

## Finding Description

`RSETH` inherits `PausableUpgradeable` and gates its two privileged state-changing paths:

- `mint` at L235 carries `whenNotPaused`. [1](#0-0) 
- `burnFrom` at L245 carries `whenNotPaused`. [2](#0-1) 

The internal `_transfer` override at L287–291, which is the sole hook invoked by every ERC20 `transfer` and `transferFrom` call, only enforces the per-address block mechanism and then delegates to `super._transfer` with no pause check: [3](#0-2) 

Because `ERC20Upgradeable.transfer` and `transferFrom` both route through `_transfer`, the pause state is never consulted for ordinary token movement. The existing `_enforceNotBlocked` calls inside `_transfer` are irrelevant to the pause — they only check the per-address `transfersBlockedUntil` mapping. [4](#0-3) 

## Impact Explanation

When the contract is paused in response to a security incident, rsETH holders retain full ability to transfer tokens via standard ERC20 calls. This means the protocol cannot achieve a complete operational freeze: minting and burning are halted, but token movement continues unimpeded. The contract fails to deliver its promised invariant (pause halts all rsETH operations) without causing direct fund loss. This maps to **Low: Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation

The `PAUSER_ROLE` can pause at any time; pausing is a realistic operational action during incidents. Once paused, the bypass requires no privileged access, no special knowledge, and no preconditions beyond holding rsETH — a standard `transfer` call suffices. Any rsETH holder can exploit this immediately and repeatedly. [5](#0-4) 

## Recommendation

Add `whenNotPaused` to the `_transfer` override:

```solidity
function _transfer(address from, address to, uint256 amount)
    internal
    override
    whenNotPaused
{
    _enforceNotBlocked(from);
    _enforceNotBlocked(to);
    super._transfer(from, to, amount);
}
```

This ensures all token movement is consistently halted alongside `mint` and `burnFrom` when the contract is paused. [3](#0-2) 

## Proof of Concept

1. Deploy or fork with the RSETH contract.
2. An address holding `PAUSER_ROLE` calls `RSETH.pause()`. [6](#0-5) 
3. Any rsETH holder calls `RSETH.transfer(recipient, amount)`.
4. `ERC20Upgradeable.transfer` calls `_transfer(msg.sender, recipient, amount)`.
5. `_transfer` runs `_enforceNotBlocked` (passes — address is not blocked) then `super._transfer` — no pause check is performed. [3](#0-2) 
6. Transfer succeeds. Confirm with a Foundry test:

```solidity
function test_transferBypassesPause() public {
    // Setup: mint rsETH to alice
    vm.prank(minter);
    rsETH.mint(alice, 1e18);

    // Pause the contract
    vm.prank(pauser);
    rsETH.pause();
    assertTrue(rsETH.paused());

    // Alice transfers while paused — should revert if pause is enforced, but succeeds
    vm.prank(alice);
    rsETH.transfer(bob, 1e18); // succeeds, demonstrating the bypass
    assertEq(rsETH.balanceOf(bob), 1e18);
}
```

    function _enforceNotBlocked(address account) internal {
        // Addresses that are permanently exempt can never be blocked
        if (isPermanentlyExempt[account]) return;

        // Check if the account has an active transfer block
        uint256 blockedUntil = transfersBlockedUntil[account];
        if (blockedUntil == 0) return;

        if (block.timestamp < blockedUntil) revert TransfersBlocked(account, blockedUntil);

        // Auto-clean up expired block
        delete transfersBlockedUntil[account];
    }
