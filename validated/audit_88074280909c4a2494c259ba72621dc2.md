### Title
Reentrancy via ERC-777 `tokensToSend` Hook Corrupts `currentOriginNonce` in `initTransfer`, Causing Duplicate-Nonce Events and Permanent Fund Lock - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
`OmniBridge.initTransfer` increments `currentOriginNonce` before the external `safeTransferFrom` call but reads it again from storage at emit time. An ERC-777 token's `tokensToSend` hook gives an attacker a reentrant call into `initTransfer` that increments the nonce a second time. Both the reentrant and outer calls then emit `InitTransfer` with the same nonce. The NEAR side processes both events; the second overwrites the first in `pending_transfers`, leaving one transfer's tokens permanently locked on EVM with no NEAR credit.

### Finding Description
In `OmniBridge.initTransfer`, `currentOriginNonce` is incremented at line 381 before any external call. However, the `InitTransfer` event at line 427–436 reads `currentOriginNonce` directly from storage rather than from a local variable captured before the external call. [1](#0-0) 

The function then performs an external `safeTransferFrom` call for non-bridge, non-custom-minter tokens: [2](#0-1) 

ERC-777 tokens implement ERC-20 compatibility and fire a `tokensToSend` hook on the sender's registered implementer during `transferFrom`. This hook executes before the transfer completes, giving the attacker a callback mid-execution. During this callback, the attacker calls `initTransfer` again (reentrancy). The reentrant call increments `currentOriginNonce` from N to N+1 and emits `InitTransfer` with nonce N+1. When the outer call resumes, it reads `currentOriginNonce` from storage — now N+1 — and emits a second `InitTransfer` also with nonce N+1: [3](#0-2) 

There is no `nonReentrant` guard on `initTransfer`. The contract imports `AccessControlUpgradeable`, `UUPSUpgradeable`, and `SelectivePausableUpgradable` but no `ReentrancyGuardUpgradeable`. [4](#0-3) 

On the NEAR side, `pending_transfers` is keyed by `TransferId { origin_chain, origin_nonce }`: [5](#0-4) 

When the NEAR prover processes both EVM events with the same `origin_nonce`, the second `init_transfer` call overwrites the first entry in `pending_transfers`. The first transfer's tokens are locked on EVM but its pending entry is overwritten and never signed or finalized.

### Impact Explanation
**Critical — Irreversible fund lock.** The attacker's reentrant transfer locks real ERC-20 tokens in the bridge vault on EVM. The NEAR side's `pending_transfers` map is overwritten by the duplicate-nonce event, so the first transfer's entry is permanently lost. No relayer can sign or finalize it. The locked tokens are unrecoverable without an admin intervention or contract upgrade.

### Likelihood Explanation
ERC-777 tokens are deployed on mainnet (e.g., imBTC, CHAI). Any such token that is neither a bridge token nor registered with a custom minter goes through the vulnerable `safeTransferFrom` path. An attacker needs only to hold an ERC-777 token, register a `tokensToSend` implementer via ERC-1820, and call `initTransfer` twice — once from the hook. No privileged access is required.

### Recommendation
Capture `currentOriginNonce` in a local variable immediately after incrementing it, and use that local variable in the `emit` statement:

```solidity
currentOriginNonce += 1;
uint64 originNonce = currentOriginNonce; // capture before any external call
...
IERC20(tokenAddress).safeTransferFrom(...);
...
emit BridgeTypes.InitTransfer(msg.sender, tokenAddress, originNonce, ...);
```

Additionally, add OpenZeppelin's `ReentrancyGuardUpgradeable` and apply `nonReentrant` to `initTransfer` and `initTransfer1155` as defense-in-depth.

### Proof of Concept

1. Deploy an ERC-777 token `T` (not a bridge token, no custom minter).
2. Register an attacker contract `A` as the `tokensToSend` implementer for `T` via ERC-1820.
3. Call `OmniBridge.initTransfer(T, amount1, fee1, 0, "near:victim.near", "")` from `A`.
   - `currentOriginNonce` increments to N.
   - `safeTransferFrom(A, bridge, amount1)` fires `A.tokensToSend`.
4. Inside `tokensToSend`, `A` calls `OmniBridge.initTransfer(T, amount2, fee2, 0, "near:attacker.near", "")`.
   - `currentOriginNonce` increments to N+1.
   - Inner `safeTransferFrom` completes (no further hook).
   - Inner call emits `InitTransfer(..., nonce=N+1, ...)`.
5. Outer call resumes, reads `currentOriginNonce` = N+1 from storage.
   - Outer call emits `InitTransfer(..., nonce=N+1, ...)`.
6. Two events with `originNonce=N+1` are on-chain. Nonce N is never emitted.
7. NEAR prover submits both events. The second `init_transfer` on NEAR overwrites `pending_transfers[{Eth, N+1}]` with the outer call's data.
8. The inner call's `amount2` tokens are locked in the bridge vault with no pending transfer entry → permanently unclaimable. [6](#0-5)

                revert InvalidFee();

            extensionValue = msg.value - amount - nativeFee;

            extensionValue = msg.value - nativeFee;
            if (customMinters[tokenAddress] != address(0)) {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,

                    amount

                    amount
                );

                    msg.sender,
                    address(this),
                    amount
                );

            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message,
