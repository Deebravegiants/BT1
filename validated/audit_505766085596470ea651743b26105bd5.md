### Title
`starknet_getStorageAt` returns `CONTRACT_NOT_FOUND` for a pending-only deployed contract with a zero storage value — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

### Summary
When a contract is deployed exclusively in the pending block and a caller queries a storage slot whose value is zero (either explicitly written as zero or never written), the RPC handler incorrectly returns `CONTRACT_NOT_FOUND` instead of `Felt::ZERO`.

### Finding Description
In `get_storage_at` (`api_impl.rs` lines 340–384), after `execution_utils::get_storage_at` returns `Felt::ZERO`, the handler performs a contract-existence guard at line 375. This guard calls `txn.get_state_reader()?.get_class_hash_at(state_number, &contract_address)` — a query against the **committed** (non-pending) storage only. For a contract that was deployed in the pending block but not yet committed, this lookup returns `None`, and the handler converts that into a `CONTRACT_NOT_FOUND` RPC error.

The zero-value branch fires in two distinct sub-cases:
1. The pending `storage_diffs` contains an explicit `(key → 0)` entry for the contract.
2. The pending `storage_diffs` has no entry for the key at all (slot never written; default is zero).

In both cases the contract is validly deployed in the pending state diff, and the correct answer per the Starknet spec is `Felt::ZERO`.

The sibling function `maybe_get_class_hash_at` (lines 1649–1675) already demonstrates the correct pattern: it reads `pending_state_diff.deployed_contracts` and passes them to `execution_utils::get_class_hash_at`, which searches the pending diff before falling back to committed state. `get_storage_at` omits this step entirely.

### Impact Explanation
Any unprivileged RPC caller can trigger this by querying `starknet_getStorageAt` with `Tag::Pending` for a contract deployed in the current pending block. The node returns an authoritative `CONTRACT_NOT_FOUND` error for a contract that legitimately exists in the pending view. This is a wrong RPC result for a valid pending state, fitting the "High — RPC pending view returns an authoritative-looking wrong value" impact category.

### Likelihood Explanation
The condition is trivially reachable: any contract deployed in the pending block whose storage slot has never been written (the common case immediately after deployment) will exhibit this bug on every query. No special privileges are required.

### Recommendation
In `get_storage_at`, when `block_id` is `Tag::Pending`, also read `pending_state_diff.deployed_contracts` and `pending_state_diff.replaced_classes`, then replace the raw `txn.get_state_reader()?.get_class_hash_at(...)` call with `execution_utils::get_class_hash_at(...)` passing those pending collections — exactly as `maybe_get_class_hash_at` already does.

// 4. Call starknet_getStorageAt(addr, key, Tag::Pending).
