The vulnerability is real and reachable. Here is the full trace:

### Title
`starknet_estimateFee` with `block_id=Pending` Returns Wrong `l2_gas_price` (= `NonzeroGasPrice::MIN`) When Pending Data Falls Back to `DeprecatedPendingBlock` — (`crates/apollo_rpc/src/pending.rs`, `crates/apollo_rpc_execution/src/lib.rs`)

---

### Summary

When a node's pending data is stale (parent hash mismatch), `read_pending_data` synthesizes a fallback `DeprecatedPendingBlock`. That type has no `l2_gas_price` field, so `PendingBlockOrDeprecated::l2_gas_price()` unconditionally returns `GasPricePerToken::default()` (both `price_in_wei=0` and `price_in_fri=0`). `create_block_context` then substitutes `NonzeroGasPrice::MIN` (= `GasPrice(1)`) for both ETH and STRK L2 gas prices. Any unprivileged caller of `starknet_estimateFee` with `block_id=Pending` submitting a v3 transaction receives a dramatically underestimated `overall_fee` and a wrong `l2_gas_price` in the `FeeEstimation` response.

---

### Finding Description

**Step 1 — Fallback `DeprecatedPendingBlock` is synthesized**

`read_pending_data` checks whether the cached pending block's parent hash matches the latest stored block hash. When it does not (stale pending data — a normal, recurring operational state), it constructs a fallback:

```rust
Ok(PendingData {
    block: PendingBlockOrDeprecated::Deprecated(DeprecatedPendingBlock {
        parent_block_hash: latest_header.block_hash,
        eth_l1_gas_price: latest_header...l1_gas_price.price_in_wei,
        strk_l1_gas_price: latest_header...l1_gas_price.price_in_fri,
        ..Default::default()   // ← no l2_gas_price field exists in this struct
    }),
    ...
})
``` [1](#0-0) 

`DeprecatedPendingBlock` has no `l2_gas_price` field at all: [2](#0-1) 

**Step 2 — `l2_gas_price()` returns zero for `Deprecated` variant**

```rust
pub fn l2_gas_price(&self) -> GasPricePerToken {
    match self {
        // In older versions, L2 gas price was 0.
        PendingBlockOrDeprecated::Deprecated(_) => GasPricePerToken::default(),
        PendingBlockOrDeprecated::Current(block) => block.l2_gas_price,
    }
}
``` [3](#0-2) 

**Step 3 — Zero is forwarded through `client_pending_data_to_execution_pending_data`**

```rust
l2_gas_price: client_pending_data.block.l2_gas_price(),  // → (0, 0)
``` [4](#0-3) 

**Step 4 — `create_block_context` silently substitutes `NonzeroGasPrice::MIN`**

```rust
l2_gas_price: NonzeroGasPrice::new(l2_gas_price.price_in_wei)
    .unwrap_or(NonzeroGasPrice::MIN),   // GasPrice(0) → GasPrice(1)
// strk side:
l2_gas_price: NonzeroGasPrice::new(l2_gas_price.price_in_fri)
    .unwrap_or(NonzeroGasPrice::MIN),   // GasPrice(0) → GasPrice(1)
``` [5](#0-4) 

`NonzeroGasPrice::MIN` is `GasPrice(1)`: [6](#0-5) 

**Step 5 — Wrong price is embedded in `FeeEstimation`**

`tx_execution_output_to_fee_estimation` reads `l2_gas_price` directly from the `BlockContext` built above, so both the `l2_gas_price` field and the `overall_fee` (which includes the l2_gas component priced at 1 instead of the real price, e.g. ~10⁹ fri) are wrong: [7](#0-6) 

**Step 6 — The public RPC handler wires it all together**

```rust
let maybe_pending_data = if let BlockId::Tag(Tag::Pending) = block_id {
    Some(client_pending_data_to_execution_pending_data(
        read_pending_data(&self.pending_data, &storage_txn).await?,
        ...
    ))
} else { None };
``` [8](#0-7) 

The same flaw applies to `simulate_transactions` (lines 1079–1086) and `estimate_message_fee` (lines 1437–1444) when called with `block_id=Pending`.

---

### Impact Explanation

For any v3 transaction with non-trivial L2 gas consumption, the returned `l2_gas_price` is `GasPrice(1)` instead of the real network price (typically ~10⁹ fri). The `overall_fee` is correspondingly wrong — potentially off by nine or more orders of magnitude for the L2 gas component. Clients that use this estimate to set `resource_bounds.l2_gas.max_price_per_unit` will either:
- Set it too low → transaction rejected at sequencer admission, or
- Receive a misleading "cheap" estimate that does not reflect actual execution cost.

This is a **High** impact: `starknet_estimateFee` returns an authoritative-looking wrong value to any caller.

---

### Likelihood Explanation

The trigger condition — pending data being stale (parent hash mismatch) — is a **normal, recurring operational state** that occurs every time a new block is accepted before the pending data feed updates. It is not an adversarial condition; any unprivileged RPC client that happens to call `starknet_estimateFee` with `block_id=Pending` during this window receives the wrong estimate. No special privileges, keys, or coordination are required.

---

### Recommendation

In `read_pending_data`, when constructing the fallback `DeprecatedPendingBlock`, use a `PendingBlock` (Current variant) instead, populating `l2_gas_price` from the latest stored block header's `l2_gas_price` field. Alternatively, add a `l2_gas_price` field to `DeprecatedPendingBlock` with a sensible default sourced from the latest header, and update `l2_gas_price()` to return it. A guard in `create_block_context` that logs a warning (or returns an error) when `l2_gas_price` is zero for a pending block on a modern network would also help surface the issue.

### Proof of Concept

Conceptual Rust unit test (production path, no mocks):

```rust
// 1. Write one block to storage with a real l2_gas_price (e.g. 1_000_000_000 fri).
// 2. Set pending_data to a DeprecatedPendingBlock whose parent_block_hash
//    does NOT match the stored block hash (stale pending data).
// 3. Call estimate_fee with block_id=Tag::Pending and a v3 InvokeTransaction.
// 4. Assert:
assert_eq!(fee_estimation.l2_gas_price, GasPrice(1)); // NonzeroGasPrice::MIN
// Real price should be 1_000_000_000, not 1.
// overall_fee is therefore wrong by ~9 orders of magnitude on the l2_gas component.
```

The fallback path in `read_pending_data` (lines 1573–1594) is exercised whenever `pending_data.block.parent_block_hash() != latest_header.block_hash`, which is the common case during block transitions.
