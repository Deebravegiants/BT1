### Title
Permissionless `init` Function Allows Frontrunning to Corrupt Initial Participant Set and Seize MPC Network Control - (File: crates/contract/src/lib.rs)

### Summary
The MPC contract's `init` function carries no access control. Any NEAR account can call it before the legitimate deployer does. Because the deployment transaction and the `init` call are submitted as two separate transactions, a racing attacker can insert their own `init` call in between, installing themselves as the sole participant with a threshold of 1 and thereby gaining unilateral signing authority over the entire MPC network.

### Finding Description
`init` is decorated only with `#[init]` (which prevents *re*-initialization) and `#[handle_result]`. There is no `#[private]` guard, no `assert_caller_is_signer` check, and no restriction on who the predecessor or signer account must be. [1](#0-0) 

The NEAR SDK `#[init]` attribute solely checks that the contract state storage key does not yet exist; it imposes no caller restriction. Any account that submits a valid `ThresholdParameters` payload before the deployer's own `init` transaction is included will become the contract's authoritative participant set.

The deployment pipeline in the E2E cluster helper issues two separate, non-atomic async calls: [2](#0-1) 

The shell-based localnet launcher does the same: [3](#0-2) 

Between the `deploy_contract` call and the `init_contract` call there is at least one block boundary during which the contract is deployed but uninitialized. Any NEAR account that observes the deployment transaction (e.g., a malicious validator, a block-explorer bot, or any node watching the chain) can submit its own `init` call targeting the same contract account. NEAR validators are free to order transactions within a block; a colluding or racing validator can ensure the attacker's `init` lands first.

The attacker supplies a `ThresholdParameters` containing only their own account as the single participant and a threshold of 1: [4](#0-3) 

Once `init` succeeds, the `#[init]` guard permanently blocks any subsequent `init` call, locking the legitimate deployer out. [5](#0-4) 

### Impact Explanation
After a successful frontrun the attacker is the only registered participant with a governance threshold of 1. They can:

- Call `start_keygen_instance` / `vote_pk` alone to generate a key they fully control.
- Call `respond` / `respond_verify_foreign_tx` / `respond_ckd` to resolve any pending user request with an attacker-chosen signature.
- Issue threshold signatures for arbitrary payloads on every supported chain (ECDSA, EdDSA, CKD) without any honest-node participation.

This satisfies the **Critical** impact class: *Unauthorized transaction execution, threshold signature issuance, or confidential key derivation output without the required participant authorization* and *Bypass of threshold-signature requirements*.

### Likelihood Explanation
- Deployment and initialization are always two separate transactions in every documented flow (E2E cluster, localnet shell script, testnet setup guide).
- NEAR has no guaranteed intra-block ordering; a validator that sees the deployment receipt can include a competing `init` in the very next block before the deployer's `init` arrives.
- No special capability is required: any NEAR account with enough NEAR for gas can make the call.
- The attack window is short (one or a few blocks) but the attacker only needs to win the race once; the `#[init]` guard then permanently prevents recovery.

### Recommendation
1. **Atomic deploy-and-init**: Use a NEAR batch transaction that combines `DeployContract` and `FunctionCall(init)` actions in a single atomic unit. NEAR guarantees that all actions in a batch either all succeed or all fail, eliminating the race window entirely.
2. **Caller restriction (defense-in-depth)**: Add an explicit check inside `init` that `env::predecessor_account_id() == env::current_account_id()` (i.e., the contract account itself must be the caller, which is only possible via a batch deploy action), or record the deployer's account at deploy time and assert it matches.

### Proof of Concept
1. Legitimate deployer submits `DeployContract` for the MPC WASM → included in block N.
2. Attacker observes block N, constructs `ThresholdParameters { participants: {attacker: id=0, info=…}, threshold: 1 }`.
3. Attacker submits `init(parameters=<above>, init_config=None)` targeting the newly deployed contract account → included in block N+1 before the deployer's own `init`.
4. Contract state is now `Running` with attacker as sole participant, threshold = 1.
5. Deployer's `init` call in block N+1 (or later) panics: *"The contract has already been initialized"*.
6. Attacker calls `vote_add_domains([…])` alone (threshold met with 1 vote), drives keygen, and obtains the root public key whose secret share only they hold.
7. All subsequent `sign` / `verify_foreign_transaction` / `request_app_private_key` requests are resolved exclusively by the attacker. [6](#0-5)

    #[handle_result]
    #[init]
    pub fn init(
        parameters: dtos::ThresholdParameters,
        init_config: Option<dtos::InitConfig>,
    ) -> Result<Self, Error> {
        let parameters: ThresholdParameters = parameters.try_into_contract_type()?;
        // Log participant count and hash - full parameters exceed NEAR's 16KB log limit at ~100 participants
        let params_hash = env::sha256_array(borsh::to_vec(&parameters).unwrap());
        log!(
            "init: signer={}, num_participants={}, parameters_hash={:?}, init_config={:?}",
            env::signer_account_id(),
            parameters.participants().len(),
            params_hash,
            init_config,
        );
        parameters.validate()?;

        // TODO(#1087): Every participant must have a valid attestation, otherwise we risk
        // participants being immediately kicked out once contract transitions into running.
        let initial_participants = parameters.participants();
        let tee_state = TeeState::with_mocked_participant_attestations(initial_participants);

        Ok(Self {
            protocol_state: ProtocolContractState::Running(RunningContractState::new(
                DomainRegistry::default(),
                Keyset::new(EpochId::new(0), Vec::new()),
                parameters,
                AddDomainsVotes::default(),
            )),
            pending_signature_requests: LookupMap::new(StorageKey::PendingSignatureRequestsV4),
            pending_ckd_requests: LookupMap::new(StorageKey::PendingCKDRequestsV3),
            pending_verify_foreign_tx_requests: LookupMap::new(
                StorageKey::PendingVerifyForeignTxRequestsV2,
            ),
            proposed_updates: ProposedUpdates::default(),
            config: init_config.map(Into::into).unwrap_or_default(),
            tee_state,
            accept_requests: true,
            node_migrations: NodeMigrations::default(),
            metrics: Default::default(),
            node_foreign_chain_support: Default::default(),
            foreign_chains: Lazy::new(
                StorageKey::ForeignChainMetadata,
                ForeignChainsMetadata::default(),
            ),
            tee_verifier_account_id: None,
            tee_verifier_votes: TeeVerifierVotes::default(),
        })
    }
