### Title
Fee-on-Transfer Token Accounting Divergence in `initTransfer` Creates Unbacked Minted Supply on NEAR — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

`OmniBridge.initTransfer` accepts any ERC-20 token from any unprivileged caller and records the caller-supplied `amount` in the cross-chain event without verifying how many tokens the contract actually received. For fee-on-transfer (deflationary) tokens, the contract receives `amount − fee_taken`, but the bridge event and the downstream NEAR mint are both for the full `amount`, creating unbacked wrapped supply.

### Finding Description

`initTransfer` is a public, unpermissioned entry point. For tokens that are neither bridge-deployed nor custom-minter-managed, the contract executes a plain `safeTransferFrom`: [1](#0-0) 

Immediately after, the function emits `InitTransfer` with the caller-supplied `amount` — not the actual balance delta: [2](#0-1) 

There is no pre/post balance snapshot, no token whitelist, and no restriction preventing a fee-on-transfer token from being used. The `initTransfer` function's only guards are a pause flag and a `fee >= amount` revert: [3](#0-2) 

The `OmniBridgeWormhole` extension propagates the same unchecked `amount` verbatim into the Wormhole message payload: [4](#0-3) 

The NEAR bridge then finalizes the transfer and mints exactly `amount` wrapped tokens to the recipient, while the EVM vault holds only `amount − transfer_fee`.

### Impact Explanation

Every `initTransfer` call with a fee-on-transfer token produces a permanent backing shortfall: the EVM vault holds less collateral than the NEAR-side supply. An attacker can repeat this to inflate the wrapped supply arbitrarily. When legitimate holders later redeem on the EVM side, the vault is undercollateralized and redemptions fail — irreversible fund lock for honest users. This matches the **Critical** impact category: unauthorized creation of unbacked wrapped bridge assets and broken backing guarantees.

### Likelihood Explanation

The entry point is fully public and requires no privilege. Fee-on-transfer tokens (e.g., USDT on some chains, STA, REFLECT-style tokens) are common. The attacker only needs to call `initTransfer` with such a token; no front-running, key compromise, or operator collusion is required.

### Recommendation

- **Short term**: Record the contract's token balance before and after `safeTransferFrom` and use the actual delta as the bridged amount, both in the event and in the cross-chain message.
- **Long term**: Maintain an explicit allowlist of supported tokens and reject `initTransfer` calls for tokens not on the list, or document and enforce that only standard ERC-20 tokens (no transfer fees) are supported.

### Proof of Concept

1. Alice deploys or uses an existing fee-on-transfer ERC-20 token `FOT` (e.g., 1% fee on every transfer).
2. Alice calls `OmniBridge.initTransfer(FOT, 1000, 0, 0, "alice.near", "")`.
3. `safeTransferFrom(alice, bridge, 1000)` executes; the bridge receives 990 tokens (1% fee taken by the token contract).
4. `InitTransfer` event is emitted with `amount = 1000`.
5. The Wormhole/NEAR relayer picks up the event and calls `fin_transfer` on NEAR, minting 1000 wrapped FOT to `alice.near`.
6. The EVM vault holds 990 FOT but 1000 wrapped FOT exist on NEAR — 10 tokens of unbacked supply per call.
7. Alice repeats to drain the vault's backing ratio to zero; subsequent legitimate redemptions via `finTransfer` on EVM fail because the vault cannot cover the full supply.
