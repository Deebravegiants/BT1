### Title
Cross-Request Replay of `VerifyForeignTransactionResponse` via Missing Payload-Hash Binding in `respond_verify_foreign_tx` — (File: crates/contract/src/lib.rs)

---

### Summary

`respond_verify_foreign_tx` validates that the submitted signature is cryptographically valid over `response.payload_hash` against the root public key, but does **not** verify that `response.payload_hash` actually encodes the `request` being resolved. A single Byzantine attested participant (below the signing threshold) can replay a previously observed, legitimately produced `VerifyForeignTransactionResponse` against a completely different pending `verify_foreign_transaction` request, causing the downstream caller to receive a forged foreign-chain attestation.

---

### Finding Description

In `respond_verify_foreign_tx` the contract performs two checks before resolving all queued yields:

1. The signature in `response` is valid over `response.payload_hash` against the **root public key** (no tweak).
2. The `request` key exists in `pending_verify_foreign_tx_requests`. [1](#0-0) 

What is **absent** is any binding between `response.payload_hash` and the `request` argument. The `payload_hash` is supposed to be `hash(ForeignTxSignPayloadV1 { request: ForeignChainRpcRequest, values: Vec<ExtractedValue> })`, but the contract never reconstructs or checks this hash.

Compare this with the regular `respond` function, where the signature is verified against a **derived public key** computed from `request.tweak` and the signature is checked against `request.payload` — both fields come directly from the stored request, so the signature is cryptographically bound to that specific request: [2](#0-1) 

For `respond_verify_foreign_tx`, the foreign-tx domain uses a **zero tweak** (`Tweak::new([0u8; 32])`), so the root public key is used for all foreign-tx signatures: [3](#0-2) 

This means any valid `(payload_hash, signature)` pair produced by the MPC network for **any** foreign-tx request is also a valid pair that will pass the signature check in `respond_verify_foreign_tx` when submitted against **any other** pending foreign-tx request.

The `resolve_yields_for` helper atomically removes the pending entry and resumes all queued yields with the supplied response bytes — so the forged response is delivered to every caller waiting on request B: [4](#0-3) 

The `VerifyForeignTransactionRequest` key itself contains no caller identity — it is caller-agnostic by design — so any attested participant can call `respond_verify_foreign_tx` for any pending request: [5](#0-4) 

---

### Impact Explanation

A downstream bridge contract (e.g., Omnibridge) calls `verify_foreign_transaction(bitcoin_tx_id=B)` and awaits a `VerifyForeignTransactionResponse`. After the attack, it receives `{payload_hash_A, sig_A}` — a hash and signature that correspond to a **different** foreign-chain transaction (bitcoin_tx_id=A). The MPC contract has formally attested to this response; the signature is valid against the root MPC public key.

Any downstream contract that verifies only the ECDSA signature (without also checking that `payload_hash` encodes the expected `(request_B, extracted_values_B)`) will accept this as a legitimate attestation for transaction B and may authorize a bridge action (e.g., minting tokens) that should not be authorized. This constitutes **forged foreign-chain verification enabling invalid bridge execution**.

The `ForeignChainSignatureVerifier` in the SDK does perform the payload-hash correspondence check: [6](#0-5) 

However, the MPC contract itself provides no such guarantee, so any downstream contract that omits or incorrectly implements this check is silently vulnerable.

---

### Likelihood Explanation

- **Attacker role**: A single Byzantine attested participant, strictly below the signing threshold. No collusion required.
- **Required material**: A previously produced `VerifyForeignTransactionResponse` — this is publicly observable on-chain after any successful `respond_verify_foreign_tx` call.
- **Required condition**: At least one pending `verify_foreign_transaction` request for a different transaction exists at the time of the replay.
- **Barrier**: The attacker must be a registered, attested participant. However, the MPC threat model explicitly considers Byzantine participants below the threshold.

---

### Recommendation

In `respond_verify_foreign_tx`, bind the signature check to the specific request being resolved. One concrete approach: require the caller to also supply the `extracted_values` that were used to compute `payload_hash`, reconstruct `hash(ForeignTxSignPayloadV1 { request, values })` on-chain, and assert it equals `response.payload_hash`. This ensures the signature is over the correct `(request, values)` pair and cannot be replayed across requests.

Alternatively, mirror the design of `respond`: derive a per-request tweak from the `VerifyForeignTransactionRequest` fields and verify the signature against the derived key, making cross-request replay cryptographically impossible.

---

### Proof of Concept

```
1. Alice calls verify_foreign_transaction(bitcoin_tx_id=A) → pending request_A queued.

2. MPC nodes verify tx A, compute:
     payload_hash_A = hash(ForeignTxSignPayloadV1 { request: {tx_id=A, ...}, values: [block_hash_A] })
   and produce sig_A (threshold ECDSA over payload_hash_A with root key).

3. Honest node calls respond_verify_foreign_tx(request_A, {payload_hash_A, sig_A}).
   Contract: sig_A valid over payload_hash_A ✓, request_A exists ✓ → resolves, Alice receives response.

4. Bob calls verify_foreign_transaction(bitcoin_tx_id=B) → pending request_B queued.

5. Byzantine participant (single node, below threshold) observes {payload_hash_A, sig_A} on-chain
   and calls respond_verify_foreign_tx(request_B, {payload_hash_A, sig_A}).

6. Contract checks:
     - sig_A valid over payload_hash_A against root public key? ✓ (same key, no tweak)
     - request_B exists in pending_verify_foreign_tx_requests? ✓
     - payload_hash_A corresponds to request_B? ← NOT CHECKED

7. Contract resolves request_B; Bob's yield is resumed with {payload_hash_A, sig_A}.

8. Bob's downstream bridge contract receives a valid MPC signature attesting to
   payload_hash_A (which encodes tx_id=A, not tx_id=B).
   If the bridge only verifies the ECDSA signature, it authorizes a bridge action for tx B
   based on attestation data from tx A — forged foreign-chain verification.
```

/// Resume every yield queued for `request` with `response_bytes`, draining the
/// fan-out map in one pass. Returns `Err(RequestNotFound)` if the map held no entry.
///
/// Resuming a yield that has already timed out is a no-op at the SDK level.
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
