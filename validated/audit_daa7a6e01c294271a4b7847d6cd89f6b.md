Audit Report

## Title
Stale Cross-Chain Rate in `CrossChainRateReceiver.getRate()` Enables Oracle Drift Arbitrage on L2 Deposit Pools - (File: `contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`CrossChainRateReceiver.getRate()` unconditionally returns the stored `rate` without consulting `lastUpdated`, providing no staleness protection. All L2 pool contracts (`RSETHPoolV3`, `RSETHPool`, `RSETHPoolNoWrapper`, `RSETHPoolV3ExternalBridge`) use this rate as the sole divisor in their minting formula. Because rsETH is a monotonically appreciating yield-bearing token, any delay in the LayerZero rate push leaves the L2 oracle quoting a rate lower than the true L1 rate, allowing any depositor to mint more wrsETH than deserved and extract accrued yield from existing holders.

## Finding Description

**Root cause — `CrossChainRateReceiver.getRate()` has no staleness guard:**

`getRate()` at line 103–105 of `contracts/cross-chain/CrossChainRateReceiver.sol` returns `rate` unconditionally:

```solidity
function getRate() external view returns (uint256) {
    return rate;
}
```

The contract records `lastUpdated` on every `lzReceive` call (line 97) but never reads it in `getRate()`. There is no `maxStaleness` parameter or any revert path based on time elapsed since the last update.

**Rate update mechanism — permissionless but not enforced on-chain:**

`MultiChainRateProvider.updateRate()` (lines 108–113) is callable by anyone willing to pay LayerZero fees, but there is no on-chain mechanism that enforces a maximum staleness window. If no one pays to push an update, the L2 rate drifts indefinitely.

**How the stale rate is consumed — deposit minting formula:**

Every L2 pool's `viewSwapRsETHAmountAndFee()` calls `getRate()` and uses the result as the divisor:

```solidity
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

This pattern is confirmed in `RSETHPoolV3` (lines 299–308), `RSETHPool` (lines 311–320), `RSETHPoolNoWrapper` (lines 277–286), and `RSETHPoolV3ExternalBridge` (lines 418–427). A stale (lower) rate causes the division to yield a larger `rsETHAmount` than the depositor is entitled to.

**Existing guards are insufficient:**

- `feeBps` is charged on the ETH input amount, not on the rate gap. Any oracle drift exceeding `feeBps / 10_000` is profitable.
- `dailyMintLimit` caps total daily minting volume but does not prevent rate-gap exploitation within that cap.
- `whenNotPaused` and `nonReentrant` are unrelated to staleness.

## Impact Explanation

**Impact: High — Theft of unclaimed yield.**

rsETH continuously accrues staking rewards on L1. The appreciation between two rate pushes represents yield that belongs to existing wrsETH holders. When a depositor mints against a stale (lower) rate, they receive excess wrsETH. Once the rate is updated on L2, that excess wrsETH is redeemable for more ETH than was deposited. The shortfall is borne by all existing holders whose proportional claim on the underlying ETH is diluted. This matches the allowed impact "High. Theft of unclaimed yield."

## Likelihood Explanation

- rsETH accrues ~3–5% APY continuously; after 24 hours without a push, drift is ~0.01%; after 7 days, ~0.07%. These figures can exceed a low `feeBps` setting.
- `updateRate()` requires the caller to pay LayerZero fees; there is no keeper or on-chain enforcement of a maximum staleness window.
- LayerZero message delays, network congestion, or simply no one paying to push the update create windows of opportunity.
- The attacker requires no special role — only a standard ETH deposit via any public pool function.

## Recommendation

1. **Add a staleness check in `CrossChainRateReceiver.getRate()`**: revert if `block.timestamp - lastUpdated` exceeds a configurable `maxStaleness` (e.g., 24 hours).

```solidity
function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

2. **Enforce `feeBps ≥ maximum expected drift`**: document and enforce that the deposit fee must exceed the maximum expected oracle drift over the staleness window.

3. **Automate rate pushes**: use a keeper or Chainlink Automation to call `updateRate()` at a fixed cadence, ensuring the L2 rate never drifts beyond an acceptable bound.

## Proof of Concept

1. Observe that `CrossChainRateReceiver.lastUpdated` is 48+ hours old on the target L2. The stored `rate` is `1.05e18`; the true L1 `rsETHPrice` is `1.06e18`.
2. Call `RSETHPoolV3.deposit{value: 10 ether}("")`.
3. Pool calls `getRate()` → returns stale `1.05e18` (no staleness check).
4. `rsETHAmount = 10e18 * 1e18 / 1.05e18 = 9.5238e18` wrsETH minted.
5. Fair amount: `10e18 * 1e18 / 1.06e18 = 9.4340e18` wrsETH.
6. Excess: `~0.0898e18` wrsETH (~0.09 ETH of value at true rate).
7. After the rate is updated on L2, attacker bridges wrsETH to L1 as rsETH and redeems for `~10.09 ETH`, having deposited only `10 ETH`.
8. The `~0.09 ETH` profit is extracted from existing wrsETH holders' accrued yield.

**Foundry fork test plan:**
- Fork the target L2 at a block where `lastUpdated` is stale.
- Warp `block.timestamp` forward by 48 hours without calling `updateRate()`.
- Call `deposit()` and record `rsETHAmount` minted.
- Assert `rsETHAmount > amountAfterFee * 1e18 / trueRate`, confirming excess minting.
- Verify the excess is redeemable for more ETH than deposited after a rate update.
