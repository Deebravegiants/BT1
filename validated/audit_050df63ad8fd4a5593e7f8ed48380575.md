Audit Report

## Title
Missing Decimal Normalization in `viewSwapRsETHAmountAndFee` Causes Severe rsETH Under-Minting for Non-18 Decimal Tokens - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
`RSETHPoolV3.viewSwapRsETHAmountAndFee(uint256 amount, address token)` multiplies `amountAfterFee` (in the token's native decimals) directly against `tokenToETHRate` (always 1e18-scaled), without first normalizing the token amount to 18 decimals. For any supported token with fewer than 18 decimals, every depositor receives a negligible rsETH amount — off by a factor of `10^(18 - tokenDecimals)` — while the full token deposit is retained by the pool. The only reverse-swap path is restricted to `OPERATOR_ROLE`, so affected users cannot self-rescue.

## Finding Description
The token-deposit formula at line 334 of `contracts/pools/RSETHPoolV3.sol` is:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

Both `rsETHToETHrate` (from `InterimRSETHOracle` or `ChainlinkOracleForRSETHPoolCollateral`) and `tokenToETHRate` are always normalized to 1e18 precision. `WETHOracle.getRate()` returns `1e18` directly; `ChainlinkOracleForRSETHPoolCollateral.getRate()` explicitly normalizes via `uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals())`; `InterimRSETHOracle` enforces `rate >= 1e18`. However, `amountAfterFee` is in the token's native decimals with no normalization step.

For a 6-decimal token (e.g., USDC), depositing `1e6` units (1 USDC):
- `tokenToETHRate` ≈ `3.33e14` (1/3000 ETH, 1e18-scaled)
- `rsETHToETHrate` ≈ `1.05e18`
- Computed: `1e6 × 3.33e14 / 1.05e18 ≈ 317 wei`
- Correct: `1e12 × 317 ≈ 3.17e14 wei`

The shortfall is `10^12×`. The pool receives the full USDC deposit via `safeTransferFrom` at line 284, mints 317 wei of rsETH, and the depositor has no user-accessible path to recover the stranded value. `swapAssetToPremintedRsETH` (the only reverse path) is gated by `onlyRole(OPERATOR_ROLE)` at line 422. `addSupportedToken` at line 541 has no decimal guard, so any sub-18-decimal token can be listed by `TIMELOCK_ROLE`.

The companion view `viewSwapAssetToPremintedRsETH` at line 400 has the symmetric inverse bug (`tokenAmount = rsETHAmount * rsETHToETHrate / tokenToETHRate`), meaning the operator-only rescue would return only the tiny rsETH amount's worth of tokens, not the full deposit — unless the operator manually accounts for the discrepancy.

## Impact Explanation
Any user who calls `deposit(token, amount, referralId)` after a sub-18-decimal token is listed receives a negligible rsETH amount. The deposited tokens are held by the pool with no user-accessible redemption path. This constitutes **temporary freezing of funds** (Medium), as the operator can intervene but the user cannot act unilaterally. The impact is immediate and automatic for every depositor of that token once listed.

## Likelihood Explanation
`addSupportedToken` is gated by `TIMELOCK_ROLE`, requiring a governance decision to list a sub-18-decimal token. The contract contains no technical barrier (no decimal check) preventing such a listing, and the pool is explicitly designed to be multi-asset. USDC, USDT, and WBTC are natural candidates. Likelihood is low (governance action required), but the consequence is automatic and affects every depositor of that token once listed. The depositing user requires no special privileges.

## Recommendation
Normalize `amountAfterFee` to 18 decimals before applying the rate formula in `viewSwapRsETHAmountAndFee`:

```solidity
uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 normalizedAmount = amountAfterFee * 10 ** (18 - tokenDecimals);
rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
```

Apply the symmetric inverse normalization in `viewSwapAssetToPremintedRsETH`:

```solidity
uint8 tokenDecimals = IERC20Metadata(token).decimals();
tokenAmount = (rsETHAmount * rsETHToETHrate / tokenToETHRate) / 10 ** (18 - tokenDecimals);
```

Alternatively, enforce `IERC20Metadata(token).decimals() == 18` inside `addSupportedToken` to prevent listing sub-18-decimal tokens entirely.

## Proof of Concept
1. Deploy a mock ERC-20 with 6 decimals and a mock oracle returning `3.33e14` (≈ USDC/ETH rate in 1e18 precision).
2. Call `addSupportedToken(mockUSDC, mockOracle)` as `TIMELOCK_ROLE`.
3. Approve `1e6` units and call `deposit(mockUSDC, 1e6, "")` as an unprivileged user.
4. Observe `wrsETH.balanceOf(msg.sender) == 317` instead of the expected `~3.17e14`.
5. Confirm the pool holds `1e6` USDC units and the user has no callable function to recover them.

Foundry test sketch:
```solidity
function test_underpayUSDC() public {
    MockERC20 usdc = new MockERC20(6);
    MockOracle oracle = new MockOracle(3.33e14); // ~1/3000 ETH
    vm.prank(timelockRole);
    pool.addSupportedToken(address(usdc), address(oracle));
    usdc.mint(user, 1e6);
    vm.startPrank(user);
    usdc.approve(address(pool), 1e6);
    pool.deposit(address(usdc), 1e6, "");
    vm.stopPrank();
    assertLt(wrsETH.balanceOf(user), 1000); // receives ~317 wei, not ~3.17e14
}
```
