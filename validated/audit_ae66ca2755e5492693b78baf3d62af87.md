### Title
SPICE + Resharding: Receiver-Shard Congestion Check Bypassed for Child Shard Transactions During Epoch Transition — (`chain/client/src/rpc_handler.rs`)

### Summary
When SPICE is enabled and a resharding epoch boundary has been crossed, `process_tx_internal` computes `receiver_shard` using the **current epoch's shard layout** (child shard IDs) but queries congestion info from the **last certified block**, which may still be in the **previous epoch** (parent shard IDs). `spice_shard_congestion_info` returns `None` because the child shard UID/ID does not exist in the certified block's `ChunkExtra` or `execution_results`. `validate_tx` then skips the congestion check entirely, admitting the transaction to the mempool without any congestion rejection.

### Finding Description
In `process_tx_internal`:
- `shard_layout` = current epoch's layout (child shards exist)
- `receiver_shard` = child shard ID (from current layout)
- `spice_certified_header` = last certified block (possibly in previous epoch, parent shards only)

`spice_shard_congestion_info` is called with the current epoch's `shard_layout` and the previous epoch's `certified_header`. It constructs a `ShardUId` for the child shard using the current layout, then looks it up in the certified block — which only contains parent shard entries. Both the `ChunkExtra` lookup and the `execution_results` fallback return `None`. `validate_tx` receives `None` and skips the `if let Some(congestion_info)` block entirely.

### Impact Explanation
A transaction targeting a congested child shard (e.g., the left child inheriting the parent's full delayed-receipt backlog) is admitted to the mempool without any `ShardCongested` or `ShardStuck` rejection. The RPC-level congestion gate — the primary mechanism for preventing mempool flooding during congestion — is silently bypassed for the duration of the resharding transition window (while the certified block remains in the previous epoch).

### Likelihood Explanation
The scenario requires SPICE enabled + a resharding epoch boundary + the certified block lagging behind the resharding block. The existing test `test_spice_certified_results_across_resharding` explicitly confirms this lag occurs in practice. Any unprivileged user can submit a transaction to a child-shard account immediately after the epoch boundary to trigger the bypass.

### Recommendation
In `process_tx_internal`, when `spice_shard_congestion_info` returns `None` for a receiver shard that exists in the current epoch's layout, fall back to the parent shard's congestion info from the certified block (using `get_prev_shard_id_from_prev_hash` or equivalent) rather than treating `None` as "no congestion." Alternatively, use the certified block's own epoch's shard layout when querying `spice_shard_congestion_info`, and map the parent shard's congestion info to the child shard.

### Proof of Concept
1. Enable SPICE (`protocol_feature_spice`).
2. Configure a resharding boundary (V2 layout with one shard splitting into two children).
3. Congest the parent shard before the split (fill delayed receipts to above `reject_tx_congestion_threshold`).
4. Advance to the new epoch (resharding takes effect).
5. Introduce an endorsement propagation delay (as in `test_spice_certified_results_across_resharding`) so the certified block remains in the previous epoch.
6. Submit a transaction to a `receiver_id` that maps to the child shard.
7. Assert `process_tx` returns `ProcessTxResponse::ValidTx` instead of `ProcessTxResponse::InvalidTx(InvalidTxError::ShardCongested { .. })`.

pub fn spice_shard_congestion_info(
    chain_store: &ChainStoreAdapter,
    shard_layout: &ShardLayout,
    block_header: &BlockHeader,
    shard_id: ShardId,
) -> Option<ExtendedCongestionInfo> {
    let shard_uid = ShardUId::from_shard_id_and_layout(shard_id, shard_layout);
    let chunk_store = chain_store.chunk_store();
    let congestion_info =
        if let Ok(chunk_extra) = chunk_store.get_chunk_extra(block_header.hash(), &shard_uid) {
            chunk_extra.congestion_info()
        } else {
            get_execution_result_from_store(chain_store, block_header.hash(), shard_id)?
                .chunk_extra
                .congestion_info()
        };
    Some(ExtendedCongestionInfo::new(congestion_info, 0))
}
