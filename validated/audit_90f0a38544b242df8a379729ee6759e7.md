Audit Report

## Title
Missing Stale/Zero Price Validation in `ChainlinkPriceOracle.getAssetPrice()` Enables Protocol Freeze - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards `updatedAt`, `answeredInRound`, and `roundId`, and performs no zero-price guard. A stale or zero Chainlink answer flows unchecked into `LRTOracle._updateRsETHPrice()`, which is reachable by any unprivileged caller via the public `updateRSETHPrice()`. Depending on configuration, this causes either a temporary protocol-wide freeze (Scenario A) or deflation of rsETH price enabling theft of unclaimed yield (Scenario B).

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` (L49–55) reads only `answer` from `latestRoundData()`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

No check on `updatedAt` (heartbeat staleness), `answeredInRound >= roundId` (incomplete round), or `price > 0` (zero answer). By contrast, the sibling contract `ChainlinkOracleForRSETHPoolCollateral.getRate()` (L26–37) performs all three checks and reverts with `StalePrice`, `IncompleteRound`, or `InvalidPrice`.

The unchecked price propagates through:
1. `LRTOracle.updateRSETHPrice()` (L87–89) — `public whenNotPaused`, no role check
2. → `_updateRsETHPrice()` (L214) → `_getTotalEthInProtocol()` (L331)
3. → `getAssetPrice(asset)` (L339) → `ChainlinkPriceOracle.getAssetPrice()`

**Scenario A (`pricePercentageLimit > 0`):** A stale, artificially low price reduces `totalETHInProtocol`, causing `newRsETHPrice < highestRsethPrice` by more than `pricePercentageLimit`. The downside-protection branch (L270–282) executes `lrtDepositPool.pause()`, `withdrawalManager.pause()`, and `_pause()`, freezing all deposits and withdrawals until admin intervention.

**Scenario B (`pricePercentageLimit == 0`, the default uninitialized state):** If `price == 0`, `uint256(0) * 1e18 / 10**decimals == 0`. The affected asset contributes 0 ETH to TVL. `newRsETHPrice` is deflated and written to storage (L313). New depositors mint rsETH at the deflated price, receiving more shares than entitled, diluting existing holders — constituting theft of unclaimed yield. The downside-protection branch does not trigger because `pricePercentageLimit > 0` is false.

## Impact Explanation

- **Scenario A:** Temporary freezing of funds (Medium). All user deposits and withdrawals are halted until an admin manually unpauses. Funds are not lost but are inaccessible.
- **Scenario B:** Theft of unclaimed yield (High). Existing rsETH holders are diluted when new depositors mint at an artificially deflated price. This is a direct, concrete loss of yield to current holders.

Both impacts are within the allowed scope. Scenario B is the higher-severity path.

## Likelihood Explanation

`updateRSETHPrice()` requires no privileges — any EOA can call it. Chainlink feeds for LST/ETH pairs (stETH/ETH, cbETH/ETH) have heartbeats of 1–24 hours; missed updates during network congestion are documented behavior. A zero answer (`price == 0`) can occur when a feed is deprecated or during an oracle node outage. No special attacker capability is required beyond monitoring the feed and calling the public function at the right moment.

## Recommendation

Apply the same validation pattern already present in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > maxOracleDelay[asset]) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Use a per-asset configurable `maxOracleDelay` since different Chainlink feeds have different heartbeat intervals.

## Proof of Concept

**Scenario A (Temporary freeze):**
1. Deploy on a fork where `pricePercentageLimit > 0` is set.
2. Mock the stETH/ETH Chainlink feed to return a stale answer (e.g., `updatedAt` = 25 hours ago, `answeredInRound < roundId`), with `price` significantly below current market.
3. Call `LRTOracle.updateRSETHPrice()` from any EOA.
4. `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns stale low price without reverting.
5. `newRsETHPrice` drops below `highestRsethPrice` beyond `pricePercentageLimit`.
6. `lrtDepositPool.pause()`, `withdrawalManager.pause()`, `_pause()` execute — protocol frozen.

**Scenario B (Yield theft):**
1. Deploy on a fork where `pricePercentageLimit == 0` (default).
2. Mock the Chainlink feed to return `price = 0`.
3. Call `LRTOracle.updateRSETHPrice()` from any EOA.
4. `rsETHPrice` is updated to a deflated value (asset with zero price contributes 0 to TVL).
5. Attacker deposits immediately after at the deflated price, receiving excess rsETH shares.
6. When price corrects on next update, attacker's excess shares represent stolen yield from existing holders.
