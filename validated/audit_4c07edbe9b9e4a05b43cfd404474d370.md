### Title
Bouncer `accumulated_weights` Not Decremented for Validator-Excluded Transactions in `close_block` — (`crates/blockifier/src/blockifier/concurrent_transaction_executor.rs`)

---

### Summary

The `Bouncer::accumulated_weights` field is the Sequencer's direct analog to `_lockedETH`. It accumulates resource weights for every committed transaction but is never decremented. In the validator flow, when the proposer's `ProposalFin.executed_transaction_count` is smaller than the number of transactions the validator has already committed, `close_block(final_n_executed_txs)` correctly limits the state commitment to `final_n_executed_txs` transactions, but the `Bouncer` still holds accumulated weights for **all** committed transactions. The resulting inflated `bouncer_weights` is embedded in `BlockExecutionArtifacts` and propagated to the blob/proof pipeline, producing incorrect resource accounting for the finalized block.

---

### Finding Description

**Accumulator that only grows — `Bouncer::update`**

`Bouncer::update` only ever adds to `accumulated_weights`; there is no subtract path: [1](#0-0) 

**State is trimmed; bouncer is not — `ConcurrentTransactionExecutor::close_block`**

```
close_block(final_n_executed_txs)
  ├─ commit_chunk_and_recover_block_state(final_n_executed_txs)  ← state trimmed to N txs
  └─ finalize_block(&worker_executor.bouncer, ...)               ← bouncer has weights for ALL committed txs
``` [2](#0-1) 

`commit_chunk_and_recover_block_state(final_n_executed_txs)` commits only the first N transactions to the block state, so `to_state_diff()` is correct. But `worker_executor.bouncer` has accumulated weights for every transaction up to `n_committed_txs`, which can be strictly greater than `final_n_executed_txs`.

**`execution_data` is trimmed; `block_summary.bouncer_weights` is not — `build_block_inner`**

```rust
// block_summary.bouncer_weights = inflated (all committed txs)
let block_summary = spawn_blocking(move || {
    lock_executor(&executor).close_block(final_n_executed_txs_nonopt)
}).await...?;

let mut execution_data = std::mem::take(&mut self.execution_data);
if let Some(final_n_executed_txs) = final_n_executed_txs {
    // execution_data correctly trimmed
    execution_data.remove_last_txs(&remove_tx_hashes);
}
// bouncer_weights from block_summary is used as-is — NOT recomputed
Ok(BlockExecutionArtifacts::new(block_summary, execution_data, final_n_executed_txs_nonopt).await)
``` [3](#0-2) 

`BlockExecutionArtifacts::new` takes `block_summary.bouncer_weights` directly: [4](#0-3) 

**Inflated weights propagate to the blob**

`finished_proposal_info_from_artifacts` reads `artifacts.bouncer_weights` and passes it downstream: [5](#0-4) 

In `finalize_decision`, the inflated `bouncer_weights` are forwarded to `cende_ambassador.prepare_blob_for_next_height`: [6](#0-5) 

---

### Impact Explanation

The blob sent to Aerospike contains `bouncer_weights` that include `l1_gas`, `sierra_gas`, `proving_gas`, `n_events`, `state_diff_size`, and `n_txs` from transactions that were **not** included in the finalized block. The proof system consumes these weights to account for block resource usage. Inflated weights cause the proof system to believe the block consumed more resources than it actually did, producing incorrect resource accounting with direct economic impact (over-reported proving gas, incorrect fee-market signals, and potential proof-cost overcharges). The discrepancy is also inconsistent between the proposer node (whose weights are always correct, since `final_n_executed_txs == n_executed_txs`) and any validator node that committed extra transactions before receiving `ProposalFin`.

---

### Likelihood Explanation

This condition — validator commits more transactions than the proposer's `final_n_executed_txs` — is a **normal, expected scenario** explicitly documented in the code: [7](#0-6) 

No privileged access or malicious peer is required. Any block where the proposer closes before the validator finishes executing all streamed transactions triggers the bug. The existing test `test_validate_block_excluded_txs` confirms this path is exercised: [8](#0-7) 

---

### Recommendation

After `close_block(final_n_executed_txs)` returns, recompute `bouncer_weights` from only the `final_n_executed_txs` transactions rather than using the bouncer's accumulated total. One approach: expose a `trim_to(n: usize)` method on `Bouncer` that recomputes `accumulated_weights` from a provided slice of per-transaction weights, or pass the per-transaction weight log through `BlockExecutionSummary` so `BlockExecutionArtifacts::new` can sum only the included transactions. At minimum, `BlockExecutionArtifacts` should assert that `bouncer_weights.n_txs == final_n_executed_txs` to surface the discrepancy.

---

### Proof of Concept

1. Validator receives a stream of 5 transactions from the proposer and commits all 5 to the concurrent executor (bouncer accumulates weights for 5 txs).
2. Proposer sends `ProposalFin { executed_transaction_count: 3, ... }`.
3. `handle_proposal_part` calls `truncate_to_executed_txs(content, 3)` and then `batcher.finish_proposal({ final_n_executed_txs: 3 })`.
4. `build_block_inner` calls `close_block(3)`: state is committed for 3 txs, but bouncer still holds weights for 5 txs.
5. `execution_data.remove_last_txs([tx4_hash, tx5_hash])` correctly removes txs 4 and 5 from execution data.
6. `BlockExecutionArtifacts::new(block_summary, execution_data, 3)` is called: `l2_gas_used` is correctly summed from 3 txs, but `bouncer_weights` from `block_summary` reflects 5 txs.
7. `decision_reached` retrieves these artifacts; `finalize_decision` passes the inflated `bouncer_weights` to `prepare_blob_for_next_height`.
8. The blob records, e.g., `n_txs: 5`, `sierra_gas: X+Y` (where Y is the gas of the two excluded txs) — values inconsistent with the 3-transaction block that was actually committed.

    pub fn close_block(
        &mut self,
        final_n_executed_txs: usize,
    ) -> TransactionExecutorResult<BlockExecutionSummary> {
        log::info!("Worker executor: Closing block.");
        let worker_executor = &self.worker_executor;
        worker_executor.scheduler.halt();

        let n_committed_txs = worker_executor.scheduler.get_n_committed_txs();
        assert!(
            final_n_executed_txs <= n_committed_txs,
            "Close block requested with {final_n_executed_txs} transactions, but only \
             {n_committed_txs} transactions were committed."
        );

        let mut state_after_block =
            worker_executor.commit_chunk_and_recover_block_state(final_n_executed_txs);
        finalize_block(
            &worker_executor.bouncer,
            &mut state_after_block,
            &self.worker_executor.block_context,
        )
    }
