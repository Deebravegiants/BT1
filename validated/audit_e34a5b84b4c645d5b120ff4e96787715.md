### Title
Byzantine Equivocation via Degenerate Echo-Broadcast Thresholds Breaks DKG Public-Key Consistency at n=3 — (`crates/threshold-signatures/src/protocol/echo_broadcast.rs`, `crates/threshold-signatures/src/dkg.rs`)

---

### Summary

`echo_ready_thresholds` returns `(0, 0)` for any `n ≤ 3`. With these thresholds, the delivery condition `count > 2 * ready_t` collapses to `count > 0`, meaning a **single READY vote** from any participant causes immediate delivery. This destroys the Agreement property of Byzantine Reliable Broadcast: a Byzantine participant can send different SEND messages to different honest nodes, and each honest node will deliver the value echoed by the *other* honest node — not the value it received. In the DKG (`do_keyshare`), this allows a single Byzantine participant to cause the two honest nodes to derive different `VerifiableSecretSharingCommitment` values for the Byzantine sender, producing divergent `verifying_key` outputs and permanently breaking the public-key consistency invariant.

---

### Finding Description

**Root cause 1 — Degenerate thresholds at n=3:** [1](#0-0) 

`echo_ready_thresholds(3)` returns `(echo_t=0, ready_t=0)`. The delivery check is: [2](#0-1) 

With `ready_t = 0`, `2 * ready_t = 0`, so `count > 0` is satisfied by the very first READY vote received. This means delivery happens after a single READY message, with no cross-checking.

**Equivocation trace (n=3: P_0, P_1, P_b Byzantine):**

For P_b's broadcast session, P_b sends `SEND(A)` to P_0 and `SEND(B)` to P_1 (using `send_private` instead of `send_many`).

- P_0 receives `SEND(A)` → echoes `ECHO(A)` to all → simulates own echo → `count(A)=1 > 0` → sends `READY(A)`, sets `finish_echo=true`
- P_0 simulates own `READY(A)` → `count(A)=1 > 0` → amplifies `READY(A)`, sets `finish_amplification=true`
- P_0 receives `ECHO(B)` from P_1 → **skipped** (`finish_echo=true`)
- P_0 receives `READY(B)` from P_1 → `count(B)=1 > 2*0=0` → **P_0 delivers B**

Symmetrically, P_1 delivers A.

**P_0 and P_1 deliver different values for P_b's broadcast.** The Agreement property is violated. [3](#0-2) 

**Root cause 2 — Commitment hash sent via unauthenticated point-to-point:**

In `do_keyshare`, the commitment hash is sent via `chan.send_many` (not reliable broadcast): [4](#0-3) 

A Byzantine participant can use `send_private` to send **different hashes** to different honest nodes. The `verify_commitment_hash` check: [5](#0-4) 

only verifies that the received commitment matches the hash *that node* received — it cannot detect that a different hash was sent to another node.

**Full attack (n=3, P_b Byzantine):**

1. P_b generates two polynomials `f_A`, `f_B` with different constant terms; computes `commitment_A`, `commitment_B`, `proof_A`, `proof_B`.
2. P_b computes `hash_A = H(P_b, commitment_A, session_id)` and `hash_B = H(P_b, commitment_B, session_id)`.
3. **Round 2 (hash send):** P_b sends `hash_B` to P_0 and `hash_A` to P_1 (swapped, via `send_private`).
4. **Round 3 (commitment broadcast):** P_b sends `SEND(commitment_A, proof_A)` to P_0 and `SEND(commitment_B, proof_B)` to P_1.
5. Due to (0,0) thresholds: P_0 delivers `(commitment_B, proof_B)`; P_1 delivers `(commitment_A, proof_A)`.
6. P_0 checks: `H(P_b, commitment_B, session_id) == hash_B` → **PASS**; `proof_B` valid → **PASS**.
7. P_1 checks: `H(P_b, commitment_A, session_id) == hash_A` → **PASS**; `proof_A` valid → **PASS**.
8. P_0 computes `verifying_key_0 = commitment_B[0]·G + ...`; P_1 computes `verifying_key_1 = commitment_A[0]·G + ...`.
9. `verifying_key_0 ≠ verifying_key_1`. [6](#0-5) 

**Round 5 share validation does not catch this:** P_b sends `share_B(P_0)` consistent with `commitment_B` to P_0, and `share_A(P_1)` consistent with `commitment_A` to P_1. Both `validate_received_share` calls pass. [7](#0-6) 

**`broadcast_success` does not catch this:** P_b sends the same `(true, session_id)` to both honest nodes in the final round, so both pass the success check. [8](#0-7) 

The self-consistency check at line 311 only fires when `sid == participants.index(me)` (i.e., for a node's own session), not for P_b's session: [9](#0-8) 

---

### Impact Explanation

Both honest nodes complete DKG without error but hold **different public keys**. The MPC network cannot produce a valid threshold signature: any signature produced by P_0's share is invalid under P_1's public key and vice versa. If one node's key is registered on-chain, the other node's signing attempts permanently produce invalid signatures, **freezing all funds** controlled by that MPC key. This matches the Critical impact: *permanent freezing of funds controlled by the MPC network* and *bypass of threshold-signature requirements*.

---

### Likelihood Explanation

The minimum valid DKG configuration (`n=3`) is a realistic deployment scenario. The Byzantine participant is a single node acting below the signing threshold (`MaxMalicious=1`). No collusion, no key leakage, no TEE compromise, and no network-level DoS is required. The attack is fully deterministic and requires only control over one participant's outgoing messages.

---

### Recommendation

1. **Fix `echo_ready_thresholds` for n=3:** The (0,0) branch is mathematically incorrect for Byzantine fault tolerance. The BRB Agreement property requires `3 * MaxFaulty + 1 ≤ N`. For n=3, `MaxFaulty=0`, meaning the protocol only tolerates crash faults, not Byzantine ones. Either enforce `n ≥ 4` for DKG, or handle n=3 with `MaxFaulty=0` thresholds that still require all-honest agreement (e.g., require all n echoes).

2. **Broadcast the commitment hash reliably:** Replace `chan.send_many(wait_round_1, &commitment_hash)` with `do_broadcast` so that all honest nodes are guaranteed to receive the same hash for each sender. This closes the second attack vector independently. [10](#0-9) 

---

### Proof of Concept

Deterministic simulation with n=3 (P_0, P_1, P_b):

```
1. P_b generates f_A(x) = a0 + a1*x and f_B(x) = b0 + b1*x with a0 ≠ b0
2. Computes commitment_A, commitment_B, proof_A, proof_B, hash_A, hash_B
3. Round 1: P_b sends same session_id to P_0 and P_1 (honest broadcast)
4. Round 2: P_b.send_private(P_0, hash_B); P_b.send_private(P_1, hash_A)
5. Round 3: P_b sends SEND(commitment_A, proof_A) to P_0 only
             P_b sends SEND(commitment_B, proof_B) to P_1 only
6. Echo broadcast with (0,0) thresholds:
   - P_0 echoes A, simulates echo, sends READY(A), receives READY(B) from P_1 → delivers B
   - P_1 echoes B, simulates echo, sends READY(B), receives READY(A) from P_0 → delivers A
7. Round 4: both verify_commitment_hash and verify_proof_of_knowledge pass for each node
8. Round 5: P_b sends share_B(P_0) to P_0, share_A(P_1) to P_1; both validate_received_share pass
9. Assert: output_P0.public_key ≠ output_P1.public_key  ← invariant broken
```

fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    // case where no malicious parties are assumed: when n <= 3/
    // In this case the echo and ready thresholds are both 0
    // later we compare if we have collected more votes than these thresholds
    if n <= 3 {
        return (0, 0);
    }
    // we should always have n >= 3*threshold + 1
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
