Audit Report

## Title
`instantWithdrawal` Drains Committed Assets From `LRTUnstakingVault` Due to Unlinked Accounting Systems - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary

`LRTWithdrawalManager.instantWithdrawal` gates vault access via `LRTUnstakingVault.getAssetsAvailableForInstantWithdrawal`, which subtracts a static, manually-set `queuedWithdrawalsBuffer` from the vault balance. This buffer is never automatically synchronized with `assetsCommitted`, the live variable that tracks assets already promised to queued-withdrawal users. Because `instantWithdrawal` also never updates `assetsCommitted`, any unprivileged rsETH holder can drain assets that are committed to pending queued withdrawals, causing `unlockQueue` to revert and temporarily freezing those users' funds.

## Finding Description

Two independent accounting systems govern the same pool of assets in `LRTUnstakingVault`:

**System 1 — Queued withdrawal commitments (`assetsCommitted`):**
`initiateWithdrawal` transfers the caller's rsETH to the withdrawal manager and increments `assetsCommitted[asset]` by the expected payout: [1](#0-0) 

`assetsCommitted` is only decremented inside `_unlockWithdrawalRequests` when the privileged `unlockQueue` is called: [2](#0-1) 

**System 2 — Instant withdrawal vault protection (`queuedWithdrawalsBuffer`):**
`instantWithdrawal` checks a completely separate limit: [3](#0-2) 

`getAssetsAvailableForInstantWithdrawal` returns `vaultBalance − queuedWithdrawalsBuffer[asset]`: [4](#0-3) 

`queuedWithdrawalsBuffer` is a static value set only by the operator via `setQueuedWithdrawalsBuffer`; its default is `0`: [5](#0-4) 

**The gap:** `instantWithdrawal` never reads or updates `assetsCommitted`. With `queuedWithdrawalsBuffer` at its default of `0`, the entire vault balance is exposed to instant withdrawals regardless of how much is committed to queued users. Even if the operator sets the buffer once, it becomes stale the moment any queued withdrawal is initiated or completed.

After the vault is drained, `unlockQueue` fetches `totalAvailableAssets` from the vault and immediately reverts: [6](#0-5) 

This prevents `nextLockedNonce` from advancing and `assetsCommitted` from being decremented, leaving queued users' rsETH locked in the withdrawal manager with no path to completion until the vault is replenished.

## Impact Explanation

**Medium — Temporary freezing of funds.** Queued-withdrawal users' rsETH is transferred to the withdrawal manager at `initiateWithdrawal` time and cannot be recovered by the user. If the unstaking vault is subsequently drained by instant withdrawals before `unlockQueue` is called, `unlockQueue` reverts on the empty vault, and affected users cannot complete their withdrawals. The freeze persists until the operator re-unstakes from EigenLayer and replenishes the vault, a process subject to EigenLayer's withdrawal delay (currently ~7 days). Because the protocol can eventually replenish the vault, the freeze is temporary rather than permanent, matching the **Medium / Temporary freezing of funds** impact class.

## Likelihood Explanation

**Medium.** Three conditions must hold simultaneously, all of which are routine operational states:
1. `isInstantWithdrawalEnabled[asset]` is `true` — a normal operational state set by the manager.
2. The unstaking vault holds a non-trivial balance — normal after any EigenLayer unstaking cycle.
3. `queuedWithdrawalsBuffer[asset]` is zero or stale relative to `assetsCommitted[asset]` — the default and the common case between operator updates.

No privileged key compromise is required. Any rsETH holder can call `instantWithdrawal`.

## Recommendation

Replace the static `queuedWithdrawalsBuffer` with a dynamic computation that reads `assetsCommitted` directly from `LRTWithdrawalManager`:

```solidity
function getAssetsAvailableForInstantWithdrawal(address asset)
    external view returns (uint256 availableAmount)
{
    uint256 vaultBalance = balanceOf(asset);
    uint256 committed = ILRTWithdrawalManager(withdrawalManager).assetsCommitted(asset);
    availableAmount = vaultBalance > committed ? vaultBalance - committed : 0;
}
```

This ensures the instant withdrawal check always reflects the live committed amount, eliminating the stale-buffer gap.

## Proof of Concept

1. **Setup:** `isInstantWithdrawalEnabled[stETH] = true`; `queuedWithdrawalsBuffer[stETH] = 0` (default).
2. **Alice** calls `initiateWithdrawal(stETH, rsETHAmount)`. `assetsCommitted[stETH] += 100e18`. Alice's rsETH is transferred to the withdrawal manager and locked.
3. **Operator** unstakes 100 stETH from EigenLayer; 100 stETH lands in `LRTUnstakingVault`.
4. **Bob** (any rsETH holder) calls `instantWithdrawal(stETH, rsETHAmount2)`. `getAssetsAvailableForInstantWithdrawal` returns `100e18 − 0 = 100e18`, so the check at L231 passes. Bob's rsETH is burned, and `unstakingVault.redeem` sends 100 stETH to Bob. Vault balance → 0.
5. **Operator** calls `unlockQueue(stETH, ...)`. `_createUnlockParams` reads `totalAvailableAssets` from the now-empty vault → 0. The check at L297 (`if (params.totalAvailableAssets == 0) revert AmountMustBeGreaterThanZero()`) reverts. `nextLockedNonce` is never advanced; `assetsCommitted` is never decremented.
6. **Alice** calls `completeWithdrawal(stETH, ...)`. Her request is still in the locked state. Her rsETH is held in the withdrawal manager; she cannot withdraw until the operator replenishes the vault via a new EigenLayer unstaking cycle.
