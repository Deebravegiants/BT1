### Title
Unguarded `MpcContract::init()` Allows Any Caller to Frontrun Contract Initialization and Seize Full Network Control - (File: `crates/contract/src/lib.rs`)

### Summary

`MpcContract::init()` lacks the `#[private]` access-control attribute present on every other privileged initializer in the same file. Any NEAR account can call it on a freshly deployed but not-yet-initialized contract, installing themselves as the sole participant with a threshold of 1 and thereby seizing complete governance and signing authority over the MPC network.

### Finding Description

The contract exposes three initializer functions. Two of them — `init_running` and `migrate` — carry the `#[private]` attribute, which restricts the caller to `predecessor_account_id == current_account_id` (i.e., only the contract itself, via a self-Promise). The primary `init` function does not:

The deployment procedure in every documented flow — localnet, testnet, TEE-localnet, and the devnet tooling — deploys the WASM in one transaction and calls `init` in a separate, subsequent transaction:

```bash
# Step 1 – deploy (separate tx)
near deploy mpc-contract.test.near ...
# Step 2 – init (separate tx, exploitable window here)
near contract call-function as-transaction mpc-contract.test.near init ...
``` [4](#0-3) [5](#0-4) 

Between those two transactions there is a block-level window in which any observer can submit their own `init` call. Because `#[init]` panics on re-initialization, the first successful call wins and permanently locks out the legitimate deployer.

The attacker's `init` payload sets:
- `participants` = `{ attacker.near }` (sole participant)
- `threshold` = `1`

After that call succeeds, the attacker is the only recognized participant in `RunningContractState`:

```rust

        AddDomainsVotes::default(),

``` [6](#0-5) 

### Impact Explanation

With threshold = 1 and sole participant = attacker:

1. **Unauthorized key generation**: The attacker calls `vote_add_domains` (participant-only, but they are the only participant) to add signing domains, then `start_keygen_instance` / `vote_pk` to complete DKG alone. The resulting key shares are entirely under the attacker's control.
2. **Unauthorized signature issuance**: The attacker can respond to any `sign` or `request_app_private_key` request, producing valid threshold signatures without any legitimate node's involvement.
3. **Permanent governance capture**: All governance votes (`vote_new_parameters`, `propose_update`, `vote_update`) require only the attacker's single vote to pass. The attacker can upgrade the contract to arbitrary code, drain any deposited funds, or permanently freeze the network.

This matches the allowed Critical impact: *unauthorized threshold signature issuance* and *bypass of threshold-signature requirements*.

### Likelihood Explanation

- NEAR transactions are publicly visible in real time; any mempool observer can detect the deploy transaction and submit a frontrunning `init` before the deployer's follow-up transaction is included.
- The attack requires only one cheap function call with no deposit.
- All documented and scripted deployment flows perform deploy and init as separate transactions, leaving the window open on every fresh deployment or contract account reset.
- The sandbox test `init_running_rejects_external_callers_pre_initialization` explicitly confirms that `init_running` is protected — but no analogous test exists for `init`, confirming the gap was not caught. [7](#0-6) 

### Recommendation

Add `#[private]` to `init`, matching the pattern already used by `init_running` and `migrate`:

```rust
#[private]
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
```

Because the standard deployment flow already signs the `init` call `as mpc-contract.near` (i.e., `predecessor_account_id == current_account_id`), adding `#[private]` is backward-compatible with all existing scripts. Alternatively, batch the `DeployContract` action and the `init` `FunctionCall` action into a single NEAR batch transaction, eliminating the inter-transaction window entirely.

### Proof of Concept

1. Observe the NEAR chain for a `DeployContract` action targeting the MPC contract account.
2. In the next block, submit:
   ```json
   near contract call-function as-transaction <mpc-contract-account> init \
     json-args '{
       "parameters": {
         "participants": {
           "next_id": 1,
           "participants": [["attacker.near", 0, {"tls_public_key": "<attacker_p2p_key>", "url": "https://attacker.example"}]]
         },
         "threshold": 1
       }
     }' \
     sign-as attacker.near ...
   ```
3. The deployer's subsequent `init` call panics with "The contract has already been initialized".
4. The attacker, now the sole participant with threshold 1, calls `vote_add_domains` to add a Secp256k1 signing domain, completes DKG alone via `start_keygen_instance` + `vote_pk`, and can thereafter respond to any `sign` request with a valid MPC signature — without any legitimate node's participation. async fn init_running_rejects_external_callers_pre_initialization() {
    let (worker, contract) = init().await;
    let number_of_participants = 2;
    let (accounts, participants) = gen_accounts(&worker, number_of_participants).await;

    let threshold_parameters = ThresholdParameters::new(
        participants.clone(),
        Threshold::new(number_of_participants as u64),
    )
    .unwrap();

    let init_running_args = serde_json::json!({
            "domains": [],
            "next_domain_id": 0,
            "keyset": Keyset::new(EpochId::new(2), vec![]),
            "parameters": threshold_parameters,
    });

    let execution_error = accounts[0]
        .call(contract.id(), method_names::INIT_RUNNING)
        .max_gas()
        .args_json(init_running_args)
        .transact()
        .await
        .unwrap()
        .into_result()
        .expect_err("method is private and not callable from participant account.");

    let error_message = format!("{:?}", execution_error);

    let expected_error_message = "Smart contract panicked: Method init_running is private";

    assert!(
        error_message.contains(expected_error_message),
        "init_running call was accepted by external caller. expected method to be private. {:?}",
        error_message
    )
}
