### Title
Unvalidated `payload_hash` in `respond_verify_foreign_tx` Enables Single-Participant Forged Foreign-Chain Verification Response - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies only that `response.signature` is a valid ECDSA signature over `response.payload_hash` using the root public key. It does **not** verify that `response.payload_hash` is the canonical hash of `ForeignTxSignPayload{request, values}` for the specific `request` being resolved. A single malicious attested participant can therefore supply a valid `(payload_hash, signature)` pair harvested from any previously completed signing round and use it to resolve a completely different pending request, delivering a misleading attestation to every waiting caller.

---

### Finding Description

In `respond_verify_foreign_tx` the contract performs the following check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The check confirms only that `signature` is a valid ECDSA signature over the caller-supplied `payload_hash`. It does **not** recompute the expected hash from the pending `request` and compare it. The `payload_hash` field is therefore fully attacker-controlled as long as the attacker can supply a matching signature.

Because the MPC network signs `msg_hash = SHA-256(borsh(ForeignTxSignPayload{request, values}))`: [2](#0-1) 

…and because every attested participant observes the final threshold signature produced for every request they participate in, a single malicious participant can:

1. Participate in (or observe) the signing of request **Y**, obtaining a valid `(payload_hash_Y, signature_Y)`.
2. Call `respond_verify_foreign_tx(request = X, response = {payload_hash_Y, signature_Y})` for a different pending request **X**.
3. The contract passes the signature check (the signature is genuinely valid over `payload_hash_Y`).
4. `resolve_yields_for` drains the entire fan-out queue for request **X**, delivering `{payload_hash_Y, signature_Y}` to every caller waiting on **X**. [3](#0-2) 

The SDK-level helper `ForeignChainSignatureVerifier::verify_signature` **does** perform the missing check — it recomputes `expected_payload_hash` from `(request, expected_extracted_values)` and asserts `expected_payload_hash == response.payload_hash` before accepting the signature: [4](#0-3) 

The on-chain contract, however, skips this step entirely, leaving the binding between `request` and `payload_hash` unenforced at the protocol layer.

---

### Impact Explanation

Every caller whose yield is resolved by the forged response receives a `VerifyForeignTransactionResponse` whose `payload_hash` attests to a **different** foreign-chain transaction than the one they requested. Bridge contracts that consume this response without independently recomputing and comparing the expected hash — a step the SDK helper performs but that is not mandated by the contract — will accept the forged attestation as proof that their target transaction was verified. This enables:

- **Forged foreign-chain verification**: a bridge contract is made to believe that an arbitrary foreign-chain transaction (e.g., a deposit) was confirmed, when it was not.
- **Cross-chain double-spend / invalid bridge execution**: an attacker can reuse the attestation of one confirmed deposit to unlock funds for a second, unconfirmed deposit on the NEAR side.

This matches the allowed High impact: *"Cross-chain replay, forged foreign-chain verification, light-client-style verification bypass, or participant/attestation authorization bypass that causes invalid bridge execution or double-spend conditions."*

---

### Likelihood Explanation

- Only **one** attested participant is required; no threshold collusion is needed.
- Every participant naturally observes the final signature for every request they co-sign, so harvesting a valid `(payload_hash, signature)` pair requires no special capability beyond normal protocol participation.
- The attack is silent: the contract emits no event distinguishing a legitimate response from a forged one.
- Bridge contracts that rely on the MPC contract as a trusted oracle and skip the SDK-level payload-hash re-verification step are directly exploitable.

Likelihood: **Medium** (requires one malicious attested participant; the opportunity arises whenever two or more `verify_foreign_transaction` requests are pending simultaneously).

---

### Recommendation

The contract must bind `payload_hash` to the pending `request`. Because the contract does not store the extracted `values`, it cannot recompute the hash itself. Two complementary mitigations should be applied:

1. **Include `payload_hash` in the pending-request key**: store `(request, payload_hash)` as the map key when the request is enqueued, requiring the responding node to commit to the hash at submission time. This requires the user to supply the expected `payload_hash` upfront, or the contract to derive it from a deterministic subset of the request.

2. **Alternatively, require the responding node to supply the full `ForeignTxSignPayload` (including `values`)** so the contract can recompute `SHA-256(borsh(payload))` on-chain and compare it against `response.payload_hash` before resolving yields. This is the pattern already implemented in the SDK verifier.

Until one of these is in place, the contract should at minimum document that callers **must** use `ForeignChainSignatureVerifier::verify_signature` from the SDK to re-verify the `payload_hash` before acting on the response.

### Proof of Concept

```
Setup:
  - Contract is Running with participants [P1 (honest), P2 (malicious)].
  - Two pending requests exist:
      X = verify_foreign_transaction(Bitcoin tx_id = 0xAAAA...)
      Y = verify_foreign_transaction(Bitcoin tx_id = 0xBBBB...)

Step 1 – Honest signing of Y:
  MPC network signs Y → produces (payload_hash_Y, sig_Y) where
  payload_hash_Y = SHA-256(borsh(ForeignTxSignPayload{request=Y, values=[block_hash_B]}))

Step 2 – P2 observes (payload_hash_Y, sig_Y) from the signing round.

Step 3 – P2 calls:
  respond_verify_foreign_tx(
      request  = X,                                    // ← correct key for pending request X
      response = { payload_hash: payload_hash_Y,       // ← hash of Y's payload
                   signature:    sig_Y }               // ← valid sig over payload_hash_Y
  )

Step 4 – Contract validation:
  verify_ecdsa_signature(sig_Y, payload_hash_Y, root_pk) → OK  ✓
  resolve_yields_for(pending_verify_foreign_tx_requests, X, response) → drains X's queue

Step 5 – Every caller waiting on X receives:
  VerifyForeignTransactionResponse {
      payload_hash: payload_hash_Y,   // attests to Bitcoin tx 0xBBBB, NOT 0xAAAA
      signature:    sig_Y,
  }

Step 6 – A bridge contract that does not re-verify payload_hash accepts this as proof
  that Bitcoin tx 0xAAAA was confirmed and releases funds.

impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }
