Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing `amount` in Limit Check - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric check: for ERC20 assets it correctly evaluates `totalAssetDeposits + amount > limit`, but for ETH it silently drops `amount` and evaluates only `totalAssetDeposits > limit`. This means the ETH deposit cap is checked against the pre-deposit balance rather than the post-deposit balance, allowing a single ETH deposit to push the protocol arbitrarily beyond its configured cap.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));  // amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [1](#0-0) 

The `amount` parameter (equal to `msg.value`) is passed in from `_beforeDeposit` at line 661 but is never used in the ETH branch. [2](#0-1) 

Call chain:
1. `depositETH` (line 76) passes `msg.value` to `_beforeDeposit` [3](#0-2) 
2. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)` [4](#0-3) 
3. The ETH branch returns `totalAssetDeposits > limit` — the incoming `amount` is discarded [5](#0-4) 
4. If `false`, `_mintRsETH` mints rsETH for the full `msg.value` regardless of how far it exceeds the cap [6](#0-5) 

No existing guard compensates for this omission. The `nonReentrant`, `whenNotPaused`, and `onlySupportedAsset` modifiers are unrelated to the limit check.

## Impact Explanation
The ETH deposit cap (`depositLimitByAsset[ETH_TOKEN]`) is rendered ineffective for any single deposit made while `totalAssetDeposits ≤ limit`. The protocol accepts ETH beyond its intended cap and mints rsETH for the full over-limit amount. This directly matches **Low — Contract fails to deliver promised returns**: the protocol's advertised deposit-limit guarantee is not enforced for ETH. If the cap exists for backing-ratio or risk-management reasons, the severity can escalate, but the baseline confirmed impact is the broken limit invariant.

## Likelihood Explanation
`depositETH` is a public, payable, permissionless function. No role, privilege, or special condition is required. Any depositor can observe `getTotalAssetDeposits(ETH_TOKEN)` on-chain, confirm it is below the limit, and submit a single transaction with an arbitrarily large `msg.value`. The condition is trivially detectable and the exploit is repeatable until the pre-deposit total itself exceeds the limit.

## Recommendation
Remove the ETH-specific branch and apply the same `totalAssetDeposits + amount` check uniformly:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
Precondition: `depositLimitByAsset[ETH_TOKEN] = 1000 ether`, `getTotalAssetDeposits(ETH_TOKEN) = 999 ether`.

1. Attacker calls `depositETH{value: 500 ether}(0, "")`.
2. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500 ether)` evaluates `999 ether > 1000 ether` → `false` → no revert.
3. `_mintRsETH` mints rsETH for 500 ETH; total ETH in protocol becomes 1499 ETH — 49.9% above the cap.
4. The equivalent ERC20 call would evaluate `999 + 500 > 1000` → `true` → `MaximumDepositLimitReached` revert.

**Foundry test plan:**
```solidity
function test_ETHLimitBypass() public {
    // set limit to 1000 ether, seed pool with 999 ether of prior deposits
    vm.deal(attacker, 500 ether);
    vm.prank(attacker);
    depositPool.depositETH{value: 500 ether}(0, "");
    assertGt(depositPool.getTotalAssetDeposits(ETH_TOKEN), lrtConfig.depositLimitByAsset(ETH_TOKEN));
}
```
