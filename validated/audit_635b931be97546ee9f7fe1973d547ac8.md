### Title
Missing zero-amount guard in `bridgeAssetsViaNativeBridge` allows silent zero-value ArbSys withdrawal tickets тАФ (`contracts/pools/RSETHPool.sol`)

### Summary
`RSETHPool.bridgeAssetsViaNativeBridge()` does not revert when `getETHBalanceMinusFees()` returns zero. The zero value flows through `ArbitrumMessenger.sendETHToL1ViaBridge` (which only checks `msg.value == value`, satisfied when both are 0) and into `ArbSys.withdrawEth{value: 0}`, which creates a phantom zero-value withdrawal ticket without reverting. A misleading `BridgedETHToL1ViaNativeBridge` event with `amount = 0` is then emitted.

### Finding Description
`RSETHPool.bridgeAssetsViaNativeBridge()` reads `ethBalanceMinusFees` from `getETHBalanceMinusFees()` and passes it directly to the messenger without any zero-value check:

```solidity
uint256 ethBalanceMinusFees = getETHBalanceMinusFees();
IL2Messenger(messenger).sendETHToL1ViaBridge{ value: ethBalanceMinusFees }(
    l2Bridge, l1VaultETHForL2Chain, ethBalanceMinusFees
);
emit BridgedETHToL1ViaNativeBridge(l1VaultETHForL2Chain, ethBalanceMinusFees);
``` [1](#0-0) 

`ArbitrumMessenger.sendETHToL1ViaBridge` only validates `msg.value == value`; when both are 0 this check passes:

```solidity
if (msg.value != value) revert MismatchedMsgValue();
IArbitrumMessenger(l2bridge).withdrawEth{ value: value }(target);
``` [2](#0-1) 

Arbitrum's `ArbSys.withdrawEth` precompile does not revert on zero-value calls; it registers a withdrawal ticket with 0 ETH. The contract already defines `error ZeroBridgeAmount()` and uses it in `bridgeTokens()`, but omits it here: [3](#0-2) [4](#0-3) 

The same no-argument pattern is present in `RSETHPoolNoWrapper` and `RSETHPoolV3ExternalBridge`, both of which share the identical missing guard: [5](#0-4) [6](#0-5) 

### Impact Explanation
Each zero-value call wastes gas, registers a phantom ArbSys withdrawal ticket, and emits a `BridgedETHToL1ViaNativeBridge` event with `amount = 0`. Off-chain accounting systems or keepers that rely on this event to track bridged amounts will record a spurious bridge cycle. No ETH is lost, but the contract fails to deliver its promised behavior (bridging a nonzero amount). This matches **Low тАФ Contract fails to deliver promised returns, but doesn't lose value**.

### Likelihood Explanation
The `BRIDGER_ROLE` is required, so an arbitrary attacker cannot trigger this. However, the role is held by an automated keeper/relayer. A keeper that re-triggers `bridgeAssetsViaNativeBridge` after a successful bridge (e.g., on a schedule, or because it cannot distinguish a completed cycle from a pending one due to the missing ticket ID in the event) will silently execute zero-value bridge calls on every subsequent invocation until new ETH accumulates. This is a realistic operational scenario for any keeper-driven bridge cycle.

### Recommendation
Add a zero-amount guard at the top of `bridgeAssetsViaNativeBridge()` in `RSETHPool`, `RSETHPoolNoWrapper`, and `RSETHPoolV3ExternalBridge`, consistent with the guard already present in `RSETHPoolV2ExternalBridge.bridgeAssetsViaNativeBridge(uint256 amount)`:

```solidity
uint256 ethBalanceMinusFees = getETHBalanceMinusFees();
if (ethBalanceMinusFees == 0) revert ZeroBridgeAmount();
```
