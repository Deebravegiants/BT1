The core of this question is whether `submit_promise_resume_data` in `RuntimeExt` can be called twice with the same `data_id` in a single function execution, bypassing the guard and emitting two `PromiseResume` receipts. Let me trace the exact code path.

---

### Title
Double `PromiseResume` Receipt Emission via Repeated `promise_yield_resume` on Same `data_id` — (`runtime/runtime/src/ext.rs`)

### Summary

`RuntimeExt::submit_promise_resume_data` guards against duplicate resume by checking `has_yield_receipt_in_state || has_yield_status_in_state`. However, after the first successful call it only **overwrites** the status to `ResumeInitiated` — it does **not** remove the yield receipt from the trie, and `has_promise_yield_status` checks only for key **existence**, not value. Both conditions remain `true` on the second call, so a second `PromiseResume` receipt is unconditionally emitted for the same `data_id`.

### Finding Description

`submit_promise_resume_data` in `ext.rs`:

```rust
fn submit_promise_resume_data(&mut self, data_id: CryptoHash, data: Vec<u8>)
    -> Result<bool, VMLogicError>
{
    let has_yield_receipt_in_state =
        has_promise_yield_receipt(self.trie_update, self.account_id.clone(), data_id)?;
    let has_yield_status_in_state =
        has_promise_yield_status(self.trie_update, &self.account_id, data_id)?;

    if has_yield_receipt_in_state || has_yield_status_in_state {
        self.receipt_manager.create_promise_resume_receipt(data_id, data);
        set_promise_yield_status(&mut self.trie_update, &self.account_id, data_id,
            PromiseYieldStatus::ResumeInitiated);
        return Ok(true);
    }
    Ok(false)
}
``` [1](#0-0) 

**Call 1** with `data_id`:
- `has_yield_receipt_in_state` = `true` — the `PromiseYieldReceipt` trie key exists.
- `has_yield_status_in_state` = `true` — the `PromiseYieldStatus` key exists with value `Yielded`.
- Guard passes → `create_promise_resume_receipt` is called, status is set to `ResumeInitiated`.

**Call 2** with the same `data_id` (within the same WASM execution):
- `has_yield_receipt_in_state` = **still `true`** — `submit_promise_resume_data` never removes the `PromiseYieldReceipt` trie key.
- `has_yield_status_in_state` = **still `true`** — `has_promise_yield_status` only checks key existence via `contains_key`, not the value:

```rust
pub fn has_promise_yield_status(trie: &dyn TrieAccess, receiver_id: &AccountId,
    data_id: CryptoHash) -> Result<bool, StorageError>
{
    trie.contains_key(
        &TrieKey::PromiseYieldStatus { receiver_id: receiver_id.clone(), data_id },
        AccessOptions::DEFAULT,
    )
}
``` [2](#0-1) 

The key still exists (now holding `ResumeInitiated`), so `contains_key` returns `true`. The guard passes again, and a **second** `PromiseResume` receipt is appended to `receipt_manager` for the same `data_id`.

Similarly, `has_promise_yield_receipt` only checks key existence: [3](#0-2) 

The `PromiseYieldReceipt` key is never removed inside `submit_promise_resume_data`, so it remains present for the second check.

The `TrieUpdate` is a write-ahead overlay that reflects all pending writes within the same execution context, so the status write from call 1 is visible to call 2 — but since the check is existence-only, `ResumeInitiated` is indistinguishable from `Yielded` at the guard level.

### Impact Explanation

Two `PromiseResume` (data) receipts with the same `data_id` are emitted into `result.new_receipts`. When the runtime later processes them:
- The first receipt delivers data to the waiting yield receipt and triggers the callback.
- The second receipt delivers data for the same `data_id` again. Depending on runtime handling, this either executes the callback a second time (double-spend of attached deposits, double state mutation) or causes an inconsistent state entry, in either case producing a **different state root** than the canonical one-resume-per-yield invariant requires.

This breaks the protocol invariant that exactly one `PromiseResume` receipt is emitted per `data_id`, directly causing divergent state roots across validators — a consensus-breaking outcome.

### Likelihood Explanation

An unprivileged user can:
1. Deploy a contract that calls `promise_yield_create` in one receipt, storing the returned `data_id` in contract storage.
2. In a subsequent receipt, call `promise_yield_resume(data_id, ...)` twice in the same WASM function body.

No validator or operator privileges are required. The attacker controls the contract WASM and can freely call any host function any number of times within a single execution.

### Recommendation

The guard in `submit_promise_resume_data` must distinguish between `Yielded` and `ResumeInitiated`. Replace the existence check with a value check:

```rust
let status = get_promise_yield_status(self.trie_update, &self.account_id, data_id)?;
let is_resumable = status == Some(PromiseYieldStatus::Yielded)
    || has_yield_receipt_in_state; // for cross-shard receipts not yet in status
```

Or, more robustly, **remove** the `PromiseYieldStatus` key (or set it to a terminal value that `has_promise_yield_status` explicitly excludes) after the first resume, so the second call finds no valid entry.

### Proof of Concept

A Rust unit test in `runtime/runtime/src/tests/apply.rs` would:
1. Deploy a contract whose exported function calls `promise_yield_resume` twice with the same hardcoded `data_id` (obtained from a prior yield-create call stored in state).
2. Apply the chunk containing that receipt.
3. Assert `result.new_receipts` contains **at most one** `PromiseResume` receipt for that `data_id` — the test would fail, confirming the double-emission. [1](#0-0) [2](#0-1) [3](#0-2)
