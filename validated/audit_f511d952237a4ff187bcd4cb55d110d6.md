### Title
`get_supported_foreign_chains()` Strict All-Participant Intersection Allows a Single Below-Threshold Participant to Permanently Block All Foreign Chain Verification Requests — (File: `crates/contract/src/lib.rs`)

---

### Summary

`verify_foreign_transaction` gates on `get_supported_foreign_chains()`, which requires **every** active participant to have registered support for a chain. A single Byzantine participant below the signing threshold can register an empty foreign-chain support list, causing the intersection to become empty and permanently rejecting all `verify_foreign_transaction` calls for every chain. This is the direct analog of the EIP-721 finding: an overly strict capability check (all-participant intersection instead of threshold-participant coverage) rejects requests that are fully serviceable by the network.

---

### Finding Description

`get_supported_foreign_chains()` computes the set of chains that **all** active participants have registered:

```rust
// crates/contract/src/lib.rs:2203-2214
foreign_chain_to_node_mapping
    .into_iter()
    .filter_map(|(foreign_chain, nodes_supporting_chain)| {
        let all_active_nodes_supports_chain =
            nodes_supporting_chain.is_superset(&active_participant_account_ids);

        if all_active_nodes_supports_chain {
            Some(foreign_chain)
        } else {
            None
        }
    })
```

`verify_foreign_transaction` then hard-rejects any request whose chain is absent from this set:

```rust
// crates/contract/src/lib.rs:533-542
let requested_chain = request.request.chain();
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) {
    env::panic_str(
        &InvalidParameters::ForeignChainNotSupported { requested: requested_chain }
            .to_string(),
    );
}
```

The node-side path mirrors this: `execute_foreign_chain_request` calls `chain_is_supported(...)` before any RPC work, which also reads the same all-participant intersection from the contract.

```rust
// crates/node/src/providers/verify_foreign_tx/sign.rs:122
chain_is_supported(&self.foreign_chain_policy_reader, request).await?;
```

The actual cryptographic requirement is only that a **signing threshold** of participants can serve the chain — not all of them. The all-participant intersection is a stricter check than the protocol requires, exactly as ERC-165 was a stricter check than EIP-721 required for `onERC721Received`.

The project's own design document explicitly acknowledges this root cause:

> "Today, `get_supported_foreign_chains()` returns the **strict intersection** of every active participant's registered chains, and `verify_foreign_transaction` rejects any request whose target chain is not in it. A single node that registers an empty list (or hasn't registered yet) drops **every** chain — one operator can take the whole feature down."
> — `docs/design/calculating-supported-foreign-chains.md`

The proposed fix (threshold-based `available` set) is documented but **not yet implemented** in the production contract code.

---

### Impact Explanation

A single Byzantine participant below the signing threshold calls `register_foreign_chain_config` (or `register_available_foreign_chain_config`) with an empty chain list. Because `get_supported_foreign_chains()` requires `nodes_supporting_chain.is_superset(&active_participant_account_ids)`, the intersection immediately becomes empty. Every subsequent `verify_foreign_transaction` call panics with `ForeignChainNotSupported`, regardless of which chain is requested. The entire foreign-chain verification and bridge-attestation flow is frozen for all users until the malicious participant re-registers or is removed via governance — a process that itself requires threshold voting and significant coordination time.

This matches the allowed Medium impact: **request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants** (valid, fully-serviceable requests are permanently rejected) without requiring network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

The attack requires only a single active participant to submit one on-chain transaction (`register_foreign_chain_config` with an empty payload). No key material, TEE access, or threshold collusion is needed. The call is idempotent and authenticated only by participant membership — any one of the `n` participants can execute it unilaterally. In a production network with multiple operators, the probability that at least one participant is adversarial, coerced, or compromised is non-negligible, especially given the economic incentive to disable a competing bridge flow.

---

### Recommendation

Replace the all-participant intersection in `get_supported_foreign_chains()` with a threshold-based availability check, as described in `docs/design/calculating-supported-foreign-chains.md`: a chain is available if at least `signing_threshold` active participants have registered coverage for it. This matches the actual cryptographic requirement and eliminates the single-participant veto. The `verify_foreign_transaction` gate should be updated to call the new `get_available_foreign_chains()` view instead of `get_supported_foreign_chains()`.

---

### Proof of Concept

1. Participant `P_evil` (one of `n` active participants, below signing threshold) calls:
   ```
   register_foreign_chain_config({ foreign_chain_support: [] })
   ```
2. `node_foreign_chain_support.foreign_chain_support_by_node[P_evil]` is now an empty set.
3. For every chain `C`, `nodes_supporting_chain` for `C` no longer contains `P_evil`, so `nodes_supporting_chain.is_superset(&active_participant_account_ids)` is `false`.
4. `get_supported_foreign_chains()` returns an empty `BTreeSet`.
5. Any user calling `verify_foreign_transaction` for Bitcoin, Ethereum, Solana, or any other chain receives:
   ```
   ForeignChainNotSupported { requested: <chain> }
   ```
   and the request is permanently rejected — no signature is ever produced, no bridge attestation is ever returned. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
