## Code Trace

**Step 1: `request_app_private_key` with `AppPublicKey(O)`** [1](#0-0) 

The `AppPublicKey(_) => {}` arm performs **zero validation** on the submitted G1 point. Any point — including the identity — is accepted unconditionally. The `AppPublicKeyPV` arm calls `app_public_key_check`, but `AppPublicKey` does not.

**Step 2: MPC node computation with identity `app_pk`** [2](#0-1) 

Each node computes `big_c = big_s + app_pk * y`. If `app_pk = O` (identity), then `app_pk * y = O`, so `big_c = big_s = x_i * H(pk || app_id)`. After Lagrange combination, the coordinator produces `big_c = msk * H(pk || app_id)` — the raw BLS signature — with no masking term.

**Step 3: `respond_ckd` output validation** [3](#0-2) 

Again, `AppPublicKey(_) => {}` — **no output check**. The `AppPublicKeyPV` arm calls `ckd_output_check` (the pairing equation verifier), but `AppPublicKey` skips it entirely.

**Step 4: Confirmed behavior of identity points** [4](#0-3) 

The existing test `app_public_key_check__should_accept_identity_key_pair` explicitly documents that identity key pairs pass the pairing check. The comment on `request_app_private_key` says "identity points are accepted in `AppPublicKeyPV`" — but `AppPublicKey` has no check at all, so identity is also accepted there.

**Step 5: The math**

From the CKD protocol:
- `big_c = big_s + app_pk * y`
- If `app_pk = O`: `big_c = big_s + O = big_s = msk * H(pk || app_id)`
- Caller receives `(big_y, big_c)` where `big_c` IS the raw BLS signature, no private scalar needed to "decrypt" [5](#0-4) 

The `unmask` function computes `big_c - a * big_y`. With `big_c = big_s` and any `big_y`, the caller already has `big_s` directly — `a` is irrelevant.

### Title
Missing Input Validation on `AppPublicKey` Variant Allows Identity-Point Bypass of CKD Encryption — (`crates/contract/src/lib.rs`)

### Summary
An unprivileged caller can submit a CKD request using the `AppPublicKey` variant with the BLS12-381 G1 identity point as `app_public_key`. Because neither `request_app_private_key` nor `respond_ckd` validates the `AppPublicKey` arm, honest MPC nodes compute `big_c = msk * H(pk || app_id)` (the raw BLS signature) with no encryption masking. The contract accepts the response without any output check, delivering the raw derived key in plaintext on-chain.

### Finding Description
`request_app_private_key` validates `AppPublicKeyPV` inputs via `app_public_key_check` but applies `AppPublicKey(_) => {}` — a no-op. Similarly, `respond_ckd` calls `ckd_output_check` only for `AppPublicKeyPV` responses. The `AppPublicKey` path has no guard at either the request or response stage.

When `app_pk = O`:
- Node computes: `big_c = big_s + O * y = big_s = msk * H(pk || app_id)`
- Contract accepts: no pairing check, no subgroup check, no output verification
- Caller receives: `big_c` = raw BLS signature, readable directly without any private scalar [1](#0-0) [3](#0-2) 

### Impact Explanation
The CKD protocol's confidentiality invariant — that the derived key `msk * H(pk || app_id)` is always masked by `app_pk * y` before delivery — is broken. The raw BLS signature is written to on-chain storage in plaintext, visible to MPC nodes and all chain observers. The caller recovers the derived key without possessing any private scalar. This breaks the "confidential" property of CKD for the `AppPublicKey` variant.

### Likelihood Explanation
The attack requires only a standard `request_app_private_key` call with a well-formed identity-point encoding (48 bytes with the infinity flag set). No threshold collusion, no TEE compromise, no special access. Honest nodes process the request correctly per their protocol logic and produce the unmasked output. The contract accepts it.

### Recommendation
Add a non-identity check in the `AppPublicKey` arm of `request_app_private_key`:

```rust
dtos::CKDAppPublicKey::AppPublicKey(pk) => {
    // Reject the identity point — it would bypass encryption masking
    let point = env::bls12381_p1_decompress(pk);
    // verify point != identity, e.g. via a known-identity byte pattern check
    // or add a host-level is_identity predicate
}
```

Alternatively, add an output-side pairing check analogous to `ckd_output_check` for `AppPublicKey` responses, verifying `e(big_c, g2) = e(hash_point, pk) * e(big_y, app_pk_g2)` — but this requires the caller to also supply a G2 key, effectively collapsing `AppPublicKey` into `AppPublicKeyPV`.

fn identity_point_bypasses_ckd_encryption() {
    let (context, mut contract, secret_key) = basic_setup(Curve::Bls12381, &mut OsRng);
    let SharedSecretKey::Bls12381(msk) = secret_key else { unreachable!() };

    // G1 identity point (compressed encoding: 0xc0 followed by zeros)
    let mut identity_bytes = [0u8; 48];
    identity_bytes[0] = 0xc0;
    let identity_pk = dtos::Bls12381G1PublicKey(identity_bytes);

    let request = CKDRequestArgs {
        derivation_path: "x".to_string(),
        app_public_key: CKDAppPublicKey::AppPublicKey(identity_pk),
        domain_id: dtos::DomainId::default(),
    };
    let ckd_request = CKDRequest::new(
        request.app_public_key.clone(),
        request.domain_id,
        &context.predecessor_account_id,
        &request.derivation_path,
    );
    contract.request_app_private_key(request);

    // Honest node computes big_c = big_s + O*y = big_s
    let big_s = hash_app_id_with_pk(&key_package.public_key, ckd_request.app_id.as_ref()) * msk;
    let big_y = ckd::ElementG1::generator() * ckd::Scalar::random(OsRng);
    let response = CKDResponse {
        big_y: (&big_y).into(),
        big_c: (&big_s).into(), // unmasked — identity * y = identity
    };

    with_active_participant_and_attested_context(&contract);
    // Accepted — no output check for AppPublicKey
    contract.respond_ckd(ckd_request, response.clone()).unwrap();
    // big_c == msk * H(pk || app_id): raw BLS signature, no decryption needed
}
