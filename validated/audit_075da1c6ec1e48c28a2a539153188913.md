### Title
Unbounded Iteration Over DRep Delegators During `ConwayUnRegDRep` Certificate Processing Causes Excessive Block Validation Cost - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/GovCert.hs`)

---

### Summary

When a DRep unregisters via a `ConwayUnRegDRep` certificate, the `GOVCERT` transition rule iterates over the entire `drepDelegs` set — the set of all stake credentials delegated to that DRep — to clear their delegation pointers. There is no protocol-level cap on the size of this set. An attacker can register a DRep, accumulate an arbitrarily large number of delegators (by registering many stake credentials and delegating them to the DRep), and then submit an unregistration certificate that forces the ledger to perform an unbounded O(N) traversal during block validation, exceeding intended validation limits.

---

### Finding Description

In `conwayGovCertTransition`, the `ConwayUnRegDRep` case executes the following:

```haskell
clearDRepDelegations delegs accountsMap =
  foldr (Map.adjust (dRepDelegationAccountStateL .~ Nothing)) accountsMap delegs
...
certState'
  & certDStateL . accountsL . accountsMapL
    %~ clearDRepDelegations (drepDelegs dRepState)
``` [1](#0-0) 

`drepDelegs` is a `Set (Credential Staking)` stored directly in `DRepState`:

```haskell
data DRepState = DRepState
  { drepExpiry :: !EpochNo
  , drepAnchor :: !(StrictMaybe Anchor)
  , drepDeposit :: !(CompactForm Coin)
  , drepDelegs :: !(Set (Credential Staking))
  }
``` [2](#0-1) 

This set grows monotonically as stake credentials delegate to the DRep via `DelegVote` or `DelegStakeVote` delegatees: [3](#0-2) 

There is no protocol parameter or ledger rule that caps the number of delegators per DRep. The `clearDRepDelegations` `foldr` iterates once per delegator with no early-exit or batching mechanism. The entire traversal must complete atomically within the block that includes the `ConwayUnRegDRep` certificate.

The analog to the original report is exact: just as `_updateWithdrawalQueue` looped over an unbounded queue until funds ran out, `clearDRepDelegations` loops over an unbounded delegator set until all entries are cleared.

---

### Impact Explanation

**Medium — Attacker-controlled certificate exceeds intended validation limits.**

A block containing a `ConwayUnRegDRep` certificate for a DRep with N delegators requires O(N) `Map.adjust` operations on the accounts map during the `GOVCERT` rule. With no cap on N, an attacker can force arbitrarily expensive block validation. This can:

1. Cause block-producing nodes to exceed their slot-time budget, resulting in missed slot leadership.
2. Cause non-producing nodes to spend disproportionate time validating a single transaction, degrading overall chain throughput.
3. In the worst case, if validation time diverges across node implementations or hardware, it could contribute to temporary chain forks — matching the "deterministic disagreement" impact class.

The deposit refund (`drepDeposit`) is returned to the DRep owner on unregistration, so the attacker recovers their DRep deposit. The cost to the attacker is only the stake key deposits for each delegator account, which are also refundable upon stake key deregistration.

---

### Likelihood Explanation

**Realistic.** The attack requires:
1. Registering a DRep (one transaction, one deposit).
2. Registering N stake credentials and delegating each to the DRep (N transactions, each paying a stake key deposit currently ~2 ADA on mainnet).
3. Submitting the `ConwayUnRegDRep` certificate.

All deposits are recoverable. The attacker can use a single wallet with many derived keys. There is no protocol-level guard preventing a DRep from accumulating thousands or millions of delegators. The `drepDelegs` set is already used in production (e.g., `queryRegisteredDRepStakeDistr` iterates it) and is expected to be large for popular DReps. [4](#0-3) 

---

### Recommendation

Introduce a protocol parameter (e.g., `ppMaxDRepDelegators`) capping the number of delegators per DRep, enforced in the `DELEG` rule when processing `DelegVote`/`DelegStakeVote`. Alternatively, remove the `drepDelegs` reverse-index from `DRepState` and replace `clearDRepDelegations` with a lazy/incremental cleanup strategy (similar to how the reward calculation was made incremental via `DRepPulser`), so that DRep unregistration does not require a full O(N) scan in a single block. [5](#0-4) 

---

### Proof of Concept

**Entry path** (unprivileged transaction sender):

1. Submit `ConwayRegDRep cred deposit SNothing` — register attacker DRep. Cost: `ppDRepDeposit` (refundable).
2. For i = 1..N: submit `ConwayRegCert stakeCred_i (SJust ppKeyDeposit)` followed by `ConwayDelegCert stakeCred_i (DelegVote (DRepCredential cred))`. Each `DelegVote` appends `stakeCred_i` to `drepDelegs` via `Map.adjust (drepDelegsL %~ Set.insert stakeCred) dRepCred`. Cost: N × `ppKeyDeposit` (refundable).
3. Submit `ConwayUnRegDRep cred ppDRepDeposit`. The `GOVCERT` rule executes `clearDRepDelegations (drepDelegs dRepState)` — a `foldr` over N entries. [3](#0-2) [6](#0-5) 

At step 3, the ledger performs N `Map.adjust` calls on the accounts map with no bound check, no batching, and no early termination. All deposits from steps 1–2 are recoverable by the attacker after the attack, making the net cost of the attack approach zero.
