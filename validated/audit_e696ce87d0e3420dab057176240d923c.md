The full attack chain is now clear. Let me trace every guard precisely.

**Step 1 — Peer-supplied solution, no solution validation in `fetch_coin_spend`:**

`fetch_coin_spend` validates only the puzzle hash and coin name, never the solution. [1](#0-0) 

For the launcher coin, `coin.puzzle_hash == SINGLETON_LAUNCHER_HASH` — a fixed, well-known constant. A malicious peer must return the correct puzzle (trivially satisfied) but can return **any solution it chooses**.

**Step 2 — `solution_to_pool_state` is pure deserialization, no cryptographic check:**

`solution_to_pool_state` reads the pool state directly from the solution bytes with no verification that the solution was actually executed on-chain. [2](#0-1) [3](#0-2) 

**Step 3 — `_add_coin_states` accepts the peer-supplied spend and calls `PoolWallet.create`:**

The only guard before `PoolWallet.create` is `assert pool_state is not None` — i.e., the solution must be parseable as a `PoolState`. No signature check, no puzzle-execution check. [4](#0-3) 

**Step 4 — `PoolWallet.create` stores the crafted spend verbatim:**

The function matches by `spend.coin.name() == launcher_coin_id` and stores the spend directly. No verification of the solution against the on-chain puzzle execution. [5](#0-4) 

**Step 5 — All downstream state derives from the stored spend:**

`get_current_state()` calls `get_delayed_puz_info_from_launcher_spend(all_spends[0])` and `solution_to_pool_state(full_spend)` on the stored (crafted) spend, then computes `p2_singleton_puzzle_hash` from the attacker-controlled `delayed_seconds`/`delayed_puzhash`. [6](#0-5) 

`update_pool_config` then writes the attacker-controlled `target_puzzle_hash`, `owner_public_key`, and `p2_singleton_puzzle_hash` to the pool config file. [7](#0-6)

**Impact analysis — why not Critical (reward diversion):**

The pool protocol reads `target_puzzle_hash` from the **on-chain** singleton state, not from the wallet's local config. Corrupting the local config does not redirect on-chain rewards.

Travel transactions generated from the corrupted local state would fail on-chain: the wallet reconstructs the inner puzzle from the stored (crafted) solution, producing a puzzle hash that does not match the actual on-chain singleton coin's `puzzle_hash`. The spend would be rejected by consensus.

**Actual impact — High:**

- **Wrong `p2_singleton_puzzle_hash`**: wallet subscribes to the wrong puzzle hash for farming rewards; the user's farming rewards accumulate at the correct on-chain address but the wallet never sees them and cannot claim them.
- **Wrong `owner_pubkey` / `target_puzzle_hash` in pool config**: pool protocol authentication and payout configuration are corrupted.
- **Persistent wallet state corruption**: all subsequent `get_current_state()` calls return attacker-controlled parameters; the wallet cannot generate valid state-transition transactions.

This is a concrete, locally-testable path: a malicious full-node peer the wallet connects to during pool wallet creation or sync can trigger it with no keys or admin access required.

### Title
Malicious peer can corrupt `PoolWallet` initial state by returning a crafted launcher spend solution — (`chia/pools/pool_wallet.py`, `chia/wallet/util/wallet_sync_utils.py`)

### Summary
`fetch_coin_spend` validates the puzzle hash of a peer-supplied `RespondPuzzleSolution` but never validates the solution. For the singleton launcher coin the puzzle is a fixed constant (`SINGLETON_LAUNCHER`), so any peer can supply an arbitrary solution. `solution_to_pool_state` deserializes the pool state from that solution without any cryptographic check. `PoolWallet.create` stores the resulting `CoinSpend` as the canonical initial state. All downstream state — `p2_singleton_puzzle_hash`, pool config `target_puzzle_hash`, `owner_pubkey` — derives from the attacker-controlled solution.

### Finding Description
In `WalletStateManager._add_coin_states`, when a singleton launcher coin is detected as a child of a tracked coin, the wallet fetches the launcher spend from the connected peer via `fetch_coin_spend_for_coin_state`. `fetch_coin_spend` checks that `puzzle.get_tree_hash() == coin.puzzle_hash` (trivially satisfied for the launcher since its puzzle hash is a fixed constant) and that `coin_name` matches, but **does not validate the solution in any way**. The solution is passed directly to `solution_to_pool_state`, which reads the `PoolState` from the solution's key-value list by pure deserialization. If the parsed state is non-`None`, `PoolWallet.create` is called with `[launcher_spend]` and stores the spend verbatim. No guard verifies that the solution matches what was actually executed on-chain.

### Impact Explanation
A malicious peer returns a `RespondPuzzleSolution` with the correct `coin_name` and correct `puzzle` (SINGLETON_LAUNCHER) but a crafted `solution` encoding attacker-chosen `target_puzzle_hash`, `owner_pubkey`, `delayed_seconds`, and `delayed_puzhash`. The wallet stores this as the canonical initial state. Consequences:

1. `p2_singleton_puzzle_hash` is computed from the attacker's `delayed_seconds`/`delayed_puzhash` → wallet subscribes to the wrong puzzle hash → farming rewards are never seen by the wallet and cannot be claimed.
2. Pool config is written with attacker-controlled `target_puzzle_hash` and `owner_public_key` → pool protocol authentication and payout configuration are permanently corrupted for this pool wallet.
3. All travel transactions generated from the corrupted state reference the wrong inner puzzle and fail on-chain, leaving the user unable to change pool state through the wallet.

Direct fund theft via reward redirection is not achievable because the pool reads `target_puzzle_hash` from the blockchain and travel transactions built from the corrupted state are rejected by consensus. The impact is **High**: persistent corruption of wallet sync state and pool membership state with direct security impact.

### Likelihood Explanation
Any full-node peer the wallet connects to can execute this attack. No keys, admin access, or prior relationship is required. The attack triggers during normal pool wallet creation or wallet sync whenever a singleton launcher spend is encountered.

### Recommendation
In `fetch_coin_spend` (or immediately after), execute the puzzle against the solution using the CLVM interpreter and verify that the resulting conditions match the expected coin creation (correct puzzle hash, amount, and announcement). Alternatively, in `_add_coin_states`, after fetching the launcher spend, re-derive the expected `PoolState` from the actual child singleton coin's `puzzle_hash` by uncurrying it and comparing against the solution-parsed state; reject the spend if they disagree.

def pool_state_from_extra_data(extra_data: Program) -> PoolState | None:
    state_bytes: bytes | None = None
    try:
        for key, value in extra_data.as_python():
            if key == b"p":
                state_bytes = value
                break
        if state_bytes is None:

        return PoolState.from_bytes(state_bytes)
