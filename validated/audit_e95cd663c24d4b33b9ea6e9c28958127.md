### Title
Detached `burn.and(mint)` in `swap_migrated_token` with no failure callback permanently destroys user tokens on mint failure - (File: `near/omni-bridge/src/lib.rs`)

### Summary
`swap_migrated_token` burns old bridge tokens and mints new ones via a detached NEAR promise chain with no failure callback. Because `ft_on_transfer` returns `U128(0)` (no refund) before the async burn/mint executes, if the `mint` call on the new token fails for any reason (e.g., user has no storage registration on the new token contract, new token is paused, gas exhaustion), the old tokens are permanently destroyed and the user receives nothing.

### Finding Description
`migrate_deployed_token` registers a token migration mapping (`old_token → new_token`) in `migrated_tokens` without checking that the bridge is paused or that the new token contract is ready to accept mints. After registration, any user can trigger `swap_migrated_token` by calling `ft_transfer_call` on the old token with `msg: "SwapMigratedToken"`.

Inside `ft_on_transfer`, the `SwapMigratedToken` branch calls `swap_migrated_token(...).detach()` and immediately returns `PromiseOrPromiseIndexOrValue::Value(U128(0))` — telling the old token contract that zero tokens should be refunded. The old token contract therefore considers the transfer complete and the bridge now holds the user's tokens. [1](#0-0)

`burn.and(mint)` schedules both cross-contract calls independently. In NEAR, `Promise::and` does not provide atomicity — if `burn` succeeds but `mint` panics (e.g., `internal_deposit` panics because the user has no storage registration on the new token), the burn is not rolled back. There is no `.then(callback)` to detect failure and restore the user's position.

The `mint` function on `OmniToken` calls `token.internal_deposit(&account_id, amount.into())`, which panics if the account has no registered storage: [3](#0-2) 

`migrate_deployed_token` itself performs no check that the bridge is paused or that the new token is ready: [4](#0-3) 

### Impact Explanation
If the `mint` call fails after `burn` succeeds, the user's old bridge tokens are permanently destroyed and no new tokens are minted. This is an irreversible fund loss — the user's value is permanently unclaimable. This falls squarely under **Critical: Irreversible fund lock, frozen redemption path, or permanently unclaimable user or protocol value**.

### Likelihood Explanation
The trigger is reachable by any unprivileged user who holds old tokens and calls `ft_transfer_call` with `SwapMigratedToken`. The failure condition (no storage registration on the new token) is a normal operational state — users are not automatically registered on the new token contract when a migration is declared. A user who migrates immediately after `migrate_deployed_token` is called, before registering storage on the new token, will permanently lose their tokens. This is a realistic and likely scenario during any token migration event.

### Recommendation
1. **Short term:** Add a failure callback to `swap_migrated_token`. Instead of `burn.and(mint).detach()`, chain a `.then(Self::ext(...).swap_migrated_token_callback(...))` that re-mints old tokens to the user if the mint failed. Alternatively, use a sequential promise (`burn.then(mint_callback)`) so the mint only fires after burn succeeds and can be rolled back.
2. **Short term:** In `migrate_deployed_token`, verify that the new token contract is ready (e.g., bridge has storage registered on it) before registering the migration.
3. **Long term:** Add integration tests simulating `SwapMigratedToken` when the user has no storage on the new token, and when the new token contract is paused.

### Proof of Concept
1. DAO calls `migrate_deployed_token(Eth, old_token, new_token)` — bridge is not paused.
2. Alice holds 1000 `old_token` and has NOT registered storage on `new_token`.
3. Alice calls `old_token.ft_transfer_call(bridge, 1000, '{"SwapMigratedToken": null}')`.
4. Old token transfers 1000 to bridge; bridge's `ft_on_transfer` fires.
5. Bridge calls `swap_migrated_token(alice, old_token, 1000).detach()` and returns `U128(0)` — old token considers transfer complete, no refund.
6. Async: `burn(1000)` on `old_token` succeeds — bridge's 1000 old tokens are destroyed.
7. Async: `mint(alice, 1000, None)` on `new_token` panics — `internal_deposit` panics because Alice has no storage registration on `new_token`.
8. No callback exists to detect the failure. Alice's 1000 tokens are permanently gone. [5](#0-4) [6](#0-5)

    fn swap_migrated_token(
        &mut self,
        sender_id: AccountId,
        old_token: AccountId,
        amount: U128,
    ) -> Promise {
        let new_token = self
            .migrated_tokens
            .get(&old_token)
            .near_expect(BridgeError::TokenNotMigrated);

        let burn = ext_token::ext(old_token).burn(amount);
        let mint = ext_token::ext(new_token).mint(sender_id, amount, None);

        burn.and(mint)
    }
