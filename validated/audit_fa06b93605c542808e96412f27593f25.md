### Title
Division-Before-Multiplication in `scaledMinDeposit` Systematically Under-Enforces Minimum UTxO Value — (`eras/mary/impl/src/Cardano/Ledger/Mary/TxOut.hs`)

---

### Summary

The `scaledMinDeposit` function in the Mary/Allegra era computes the minimum ADA required in a multi-asset UTxO output by first performing a truncating integer division (`quot mv 27`) to derive a per-word cost, then multiplying that truncated value by the actual UTxO size. This division-before-multiplication pattern is the exact analog of the external report's vulnerability: it causes the enforced minimum to be systematically lower than the mathematically correct value, allowing any unprivileged transaction sender to create multi-asset UTxO entries with slightly less ADA than the true design minimum.

---

### Finding Description

In `eras/mary/impl/src/Cardano/Ledger/Mary/TxOut.hs`, the `scaledMinDeposit` function is defined as:

```haskell
scaledMinDeposit v (Coin mv)
  | isAdaOnly v = Coin mv
  | otherwise = Coin $ max mv (coinsPerUTxOWord * (utxoEntrySizeWithoutVal + size v))
  where
    utxoEntrySizeWithoutVal = 6 + 14 + 7  -- = 27
    coinSize = 0
    coinsPerUTxOWord = quot mv (utxoEntrySizeWithoutVal + coinSize)  -- truncating division first
``` [1](#0-0) 

The intended formula (from the design document at `doc/explanations/min-utxo-mary.rst`) is the proportion:

```
minAda(u) = max(minUTxOValue, minUTxOValue * (utxoEntrySizeWithoutVal + size v) / adaOnlyUTxOSize)
``` [2](#0-1) 

The implementation instead computes:

```
coinsPerUTxOWord = quot mv 27          -- truncating: mv mod 27 lovelace lost here
result           = coinsPerUTxOWord * (27 + size v)   -- multiplies the already-truncated value
```

This is the same structural error as the external report: **division before multiplication** causes the intermediate result to lose up to `(27 + size v − 1)` lovelace relative to the mathematically exact value.

**Concrete example** with `mv = 1,000,000` lovelace (1 ADA) and `size v = 11` words (one policy, no asset name):

| Computation | Value |
|---|---|
| Exact: `1,000,000 × 38 / 27` | `1,407,407.407…` → floor = **1,407,407** |
| Implementation: `(1,000,000 ÷ 27) × 38 = 37,037 × 38` | **1,407,406** |
| Shortfall | **1 lovelace** |

The golden tests in `eras/shelley-ma/test-suite/src/Test/Cardano/Ledger/Mary/Golden.hs` confirm the implementation returns `Coin 1407406`, not `Coin 1407407`. [3](#0-2) 

The comment in the source code explicitly acknowledges the direction: `-- how much ada does a Word64 of UTxO space cost, calculated from minAdaValue PP -- round down`. [4](#0-3) 

The rounding is in favor of the **transaction sender** (the UTxO creator), not the protocol — the inverse of the intended design, which is to protect the ledger from unbounded UTxO growth.

---

### Impact Explanation

The minimum UTxO value constraint exists to bound the total size of the UTxO set: `max UTxO entries ≤ total ADA / minUTxOValue`. By under-enforcing this constraint by up to `(utxoEntrySizeWithoutVal + size v − 1)` lovelace per entry, an attacker can create multi-asset UTxO entries that carry slightly less ADA than the true design minimum. This allows the UTxO set to grow marginally beyond the intended bound, modifying the effective deposit requirement outside design parameters.

This maps to: **Medium — Attacker-controlled transactions exceed intended validation limits or modify deposits outside design parameters.**

---

### Likelihood Explanation

The entry path is fully unprivileged: any user submitting a transaction with multi-asset outputs in the Mary/Allegra era triggers this code path. The deviation is deterministic and systematic — every multi-asset output is affected. The shortfall is small (1–2 lovelace for typical parameters) but is present for every such output ever created.

---

### Recommendation

Reorder the arithmetic to perform multiplication before division, preserving full precision:

```haskell
-- Instead of:
coinsPerUTxOWord = quot mv (utxoEntrySizeWithoutVal + coinSize)
result = coinsPerUTxOWord * (utxoEntrySizeWithoutVal + size v)

-- Use:
result = mv * (utxoEntrySizeWithoutVal + size v) `quot` (utxoEntrySizeWithoutVal + coinSize)
```

This matches the pattern used in the Babbage era's `babbageMinUTxOValue`, which correctly performs multiplication before division:

```haskell
Coin $ fromIntegral (constantOverhead + sizedSize sizedTxOut) * fromIntegral cpb
``` [5](#0-4) 

---

### Proof of Concept

The existing golden test suite already demonstrates the under-enforcement. The test at line 87–94 of `Test/Cardano/Ledger/Mary/Golden.hs` asserts `Coin 1407406` for a one-policy, zero-asset-name output with `minUTxO = Coin 1000000`. The mathematically correct floor value is `Coin 1407407` (`floor(1000000 * 38 / 27) = floor(1407407.407) = 1407407`). The 1-lovelace shortfall is the direct result of the division-before-multiplication ordering. [6](#0-5)
