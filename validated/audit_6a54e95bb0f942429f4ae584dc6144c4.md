### Title
Cross-Request Signature Replay in `respond_verify_foreign_tx` Allows Forged Foreign-Chain Verification — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is valid over `response.payload_hash`, but never verifies that `response.payload_hash` is the correct hash of the `request` data. Because `payload_hash` is entirely caller-supplied, a single Byzantine MPC participant (below signing threshold) can replay a threshold signature obtained from one legitimate signing session to forge the verification result of a completely different pending foreign-transaction request.

---

### Finding Description

The analog to the original report's pattern is exact: the original contract checked one authorization dimension (`_gameId` membership) but not the relationship between two entities (`_questionId` ↔ `_gameId`). Here, `respond_verify_foreign_tx` checks one dimension (signature validity over `payload_hash`) but not the relationship between two entities (`payload_hash` ↔ `request` data).

In `respond_verify_foreign_tx`:


The contract never computes or checks:

```
response.payload_hash == SHA-256(borsh(ForeignTxSignPayload::V1 {
    request: request.request,
    values: <extracted values>,
}))
```

Contrast this with the regular `respond` function, which correctly verifies the signature against `request.payload` (the payload embedded in the request itself, not a caller-supplied hash):

```rust
// crates/contract/src/lib.rs  lines 600-608
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,           // ← taken from the request, not from the response
    &expected_public_key,
)
```

The `ForeignTxSignPayloadV1` struct that nodes sign also omits `domain_id`:

### Impact Explanation

A single Byzantine MPC participant (below signing threshold) can:

1. Submit a legitimate `verify_foreign_transaction` for transaction **T2** (a transaction they control or whose outcome they know).
2. The MPC network (with threshold cooperation from honest nodes) produces `sig_T2 = Sign(hash(T2_data))`. The malicious node learns this signature by participating in the protocol.
3. The malicious node calls `respond_verify_foreign_tx` with:
   - `request` = the pending request for **T1** (a different, victim transaction)
   - `response.payload_hash` = `hash(T2_data)`
   - `response.signature` = `sig_T2`
4. The contract accepts: `sig_T2` is a valid threshold signature over `hash(T2_data)` under the domain's public key.
5. T1's pending yield is drained and resolved with the forged response.

Consequences:
- **Forged foreign-chain verification**: The user who submitted T1 receives a `VerifyForeignTransactionResponse` whose `payload_hash` corresponds to T2's data, not T1's. Any bridge contract that does not independently recompute and compare the expected hash (using `ForeignChainSignatureVerifier::verify_signature`) will accept this as proof that T1 was verified.
- **Permanent DoS of the target request**: `resolve_yields_for` removes the entry from `pending_verify_foreign_tx_requests` and resumes all queued yields. T1 can never be legitimately responded to again; the user's deposit is consumed and the bridge operation is permanently blocked.
- **Selective targeting**: The attacker can choose which pending requests to poison, enabling targeted disruption of specific bridge flows.

---

### Likelihood Explanation

- The attacker only needs to be a single attested MPC participant — strictly below the signing threshold.
- No key material needs to be leaked; the attacker simply reuses a threshold signature they legitimately observed during a signing round they participated in.
- The attack requires no special timing or network conditions: the attacker can collect signatures over time and replay them at will.
- The `respond_verify_foreign_tx` entry point is callable by any attested participant, not just the leader of a specific signing session.

---

### Recommendation

Inside `respond_verify_foreign_tx`, after verifying the signature, recompute the expected payload hash from the request data and assert it matches `response.payload_hash`:

```rust
// Recompute the canonical payload hash from the request
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: /* extracted values must be included in the request or response in a bound form */,
});
let expected_hash = expected_payload.compute_msg_hash()
    .map_err(|_| RespondError::PayloadHashComputationFailed)?;

if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Alternatively, include the `domain_id` and the full `ForeignChainRpcRequest` in the signed payload (`ForeignTxSignPayloadV1`) so that the signature itself cryptographically binds the response to the specific request and domain, making cross-request replay impossible even if the contract check were absent.

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
```
