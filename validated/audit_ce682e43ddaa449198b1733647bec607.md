### Title
ETH Rewards Sitting in FeeReceiver Are Excluded from TVL, Allowing Depositors to Mint rsETH at an Understated Price - (File: contracts/LRTOracle.sol)

### Summary
`LRTOracle._getTotalEthInProtocol()` computes the rsETH price using only ETH tracked inside `LRTDepositPool`, `NodeDelegator` contracts, the unstaking vault, and the converter. ETH that has accumulated inside `FeeReceiver.sol` (MEV / execution-layer rewards) is never counted until an explicit `sendFunds()` call moves it to the deposit pool. Because `sendFunds()` is permissionless, an attacker can atomically deposit at the artificially low price, then trigger the price correction, diluting existing rsETH holders' accrued yield.

---

### Finding Description

`LRTOracle._getTotalEthInProtocol()` drives the rsETH price:

```solidity
// LRTOracle.sol L331-L348
function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
    address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
    address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
    ...
    for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
        uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
        totalETHInProtocol += totalAssetAmt.mulWad(assetER);
        ...
    }
}
``` [1](#0-0) 

For ETH, `getTotalAssetDeposits` delegates to `getETHDistributionData()`, which the protocol itself annotates:

```solidity
// LRTDepositPool.sol L464-L466
/// @dev rewards are not accounted here
/// it will automatically be accounted once it is moved from feeReceiver/rewardReceiver to depositPool
function getETHDistributionData() public view override returns (...) {
    ethLyingInDepositPool = address(this).balance;
    ...
    ethLyingInUnstakingVault = lrtUnstakingVault.balance;
    ethLyingInConverter = ILRTConverter(lrtConverter).ethValueInWithdrawal();
    // FeeReceiver.balance is never read here
}
``` [2](#0-1) 

`FeeReceiver.sendFunds()` is the only mechanism to move these rewards into the TVL, and it carries **no access-control modifier**:

```solidity
// FeeReceiver.sol L53-L58
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
``` [3](#0-2) 

Similarly, `updateRSETHPrice()` is a public, permissionless function:

```solidity
// LRTOracle.sol L87-L89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [4](#0-3) 

The rsETH price is computed as:

```solidity
// LRTOracle.sol L250
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
``` [5](#0-4) 

Because `FeeReceiver.balance` is absent from `totalETHInProtocol`, `newRsETHPrice` is lower than the true backing value whenever unclaimed rewards sit in `FeeReceiver`.

---

### Impact Explanation

When rewards accumulate in `FeeReceiver` (e.g., 10 ETH), the rsETH price is understated. A depositor who mints rsETH at this lower price receives more shares than the true backing warrants. When `sendFunds()` is subsequently called and the price is updated, the new depositor's rsETH appreciates at the expense of all pre-existing holders — their accrued yield is diluted. This is a **theft of unclaimed yield** from existing rsETH holders.

**Impact: High — Theft of unclaimed yield.**

---

### Likelihood Explanation

- MEV and execution-layer rewards flow into `FeeReceiver` continuously; there is always a non-zero window between reward receipt and `sendFunds()`.
- Both `sendFunds()` and `updateRSETHPrice()` are permissionless, so the attacker controls the exact timing of the price correction.
- No privileged access, leaked keys, or external protocol compromise is required.
- The attack is repeatable every reward cycle.

**Likelihood: Medium** — requires monitoring the `FeeReceiver` balance and acting before operators flush it, but is fully permissionless and repeatable.

---

### Recommendation

Include `FeeReceiver.balance` (and the EigenLayer reward receiver balance) in `getETHDistributionData()` so that `_getTotalEthInProtocol()` always reflects the true protocol TVL, regardless of whether `sendFunds()` has been called. Alternatively, restrict `sendFunds()` to a privileged role and call it atomically before every `updateRSETHPrice()` invocation.

---

### Proof of Concept

1. Protocol TVL (excluding FeeReceiver): 1 000 ETH. rsETH supply: 952 rsETH. rsETH price: 1.05 ETH.
2. FeeReceiver accumulates 10 ETH in MEV rewards (not yet moved).
3. Attacker calls `LRTDepositPool.depositETH{value: 100 ether}(...)`. Price used = 1.05 ETH → attacker receives `100 / 1.05 ≈ 95.24 rsETH`.
4. Attacker calls `FeeReceiver.sendFunds()` — 10 ETH moves to deposit pool; TVL becomes 1 110 ETH.
5. Attacker calls `LRTOracle.updateRSETHPrice()` — new price = `1110 / (952 + 95.24) ≈ 1.060 ETH`.
6. Attacker's 95.24 rsETH is now worth `95.24 × 1.060 ≈ 100.95 ETH` — a gain of ~0.95 ETH extracted from existing holders' yield.
7. The attack scales linearly with the size of accumulated rewards relative to TVL.

        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);

        uint256 supportedAssetCount = supportedAssets.length;

            // assetER is in 1e18 precision (1.0 = 1e18)

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {

        }
