### Title
Missing `payload_hash`-to-`request` Binding in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay by a Single Byzantine Participant - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that the submitted `response.signature` is a valid ECDSA signature over `response.payload_hash` using the MPC root public key, but it never verifies that `response.payload_hash` is the correct canonical hash for the submitted `request`. A single attested MPC participant (strictly below the signing threshold) can replay a valid `(payload_hash, signature)` pair obtained from any prior `respond_verify_foreign_tx` execution to satisfy a completely different pending foreign-tx verification request, causing the contract to issue a forged attestation.

### Finding Description

In `respond_verify_foreign_tx`, the signature check is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract confirms only that `signature` is a valid ECDSA signature over `payload_hash` under the root key. It does **not** verify that `payload_hash == SHA-256(borsh(ForeignTxSignPayload { request, extracted_values }))` for the specific `request` being resolved.

Contrast this with `respond` (for `sign`), where the payload is taken directly from the `request` struct and the signature is verified against a **derived** key that commits to the request's path and predecessor:

```rust
let expected_public_key =
    derive_key_secp256k1(&affine, &request.tweak).map_err(RespondError::from)?;
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
)
``` [2](#0-1) 

For `respond_verify_foreign_tx`, the root key is used (no tweak, no derivation), and `payload_hash` is caller-supplied in the response. The `VerifyForeignTransactionResponse` struct carries both fields independently:

```rust
pub struct VerifyForeignTransactionResponse {
    pub payload_hash: Hash256,
    pub signature: SignatureResponse,
}
``` [3](#0-2) 

After the signature check passes, `resolve_yields_for` removes the pending request and resumes all queued yields with the (unverified) response bytes:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [4](#0-3) 

The pending-request key (`VerifyForeignTransactionRequest`) contains no caller account ID — it is caller-agnostic by design for the fan-out feature:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [5](#0-4) 

The correct binding check is only performed **off-chain** in the client-side SDK (`near-mpc-sdk`), not in the contract:

```rust
let expected_payload_hash = expected_payload.compute_msg_hash()...;
let payload_is_correct = expected_payload_hash == response.payload_hash;
if !payload_is_correct {
    return Err(VerifyForeignChainError::IncorrectPayloadSigned { ... });
}
``` [6](#0-5) 

The contract itself never performs this check.

### Impact Explanation

A single Byzantine MPC participant (below the signing threshold) who has observed any prior valid `(payload_hash_R1, signature_R1)` pair — produced by the MPC network for a legitimate request R1 — can call:

```
respond_verify_foreign_tx(request = R2, response = { payload_hash: payload_hash_R1, signature: signature_R1 })
```

The contract will:
1. Verify `signature_R1` is valid over `payload_hash_R1` under the root key → **passes** (it is valid).
2. Find R2 in the pending map → **passes** (R2 is legitimately pending).
3. Drain all yields queued for R2 with the forged response.

Every caller waiting on R2 receives `{ payload_hash: payload_hash_R1, signature: signature_R1 }` — a response that attests to the extracted values of R1 (a different transaction, different chain data) rather than R2. Any bridge contract that does not independently re-verify `payload_hash` against its own expected values will accept this as a valid attestation of R2's foreign-chain state, enabling invalid bridge execution or double-spend conditions.

This maps to the allowed impact: **High — forged foreign-chain verification that causes invalid bridge execution**.

### Likelihood Explanation

The attacker must be a single attested MPC participant (below threshold). Such a participant legitimately participates in the signing protocol and therefore observes the final threshold signature for every request the network processes. No additional collusion, key leakage, or privileged access is required. The attack is executable by any one of the N participants at any time a target request R2 is pending and a prior signature for any R1 is available.

### Recommendation

Inside `respond_verify_foreign_tx`, after the signature check, recompute the expected `payload_hash` from the `request` and the `extracted_values` included in the response, and assert equality before resolving yields:

```rust
// Recompute the expected hash from the request and the node-supplied extracted values
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(),
}).compute_msg_hash()?;

if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

This mirrors the binding already enforced in `respond` via key derivation (the tweak commits the signature to the specific request), and matches the check already present in the client-side `ForeignChainSignatureVerifier::verify_signature`. [7](#0-6) 

### Proof of Concept

1. MPC network processes request R1 (Bitcoin tx_id = `[1u8; 32]`, extractor = BlockHash). Nodes sign `H1 = SHA-256(borsh(ForeignTxSignPayload { R1, [block_hash_A] }))`. Byzantine participant P observes the final signature `sig_H1`.

2. Bridge contract submits `verify_foreign_transaction(R2)` where R2 is Bitcoin tx_id = `[2u8; 32]`. R2 is queued in `pending_verify_foreign_tx_requests`.

3. P calls `respond_verify_foreign_tx(R2, { payload_hash: H1, signature: sig_H1 })`.

4. Contract checks: `verify_ecdsa_signature(sig_H1, H1, root_pk)` → valid. R2 found in pending map. Yields for R2 resolved with `{ payload_hash: H1, signature: sig_H1 }`.

5. Bridge contract receives the response and interprets `H1` as the attestation for tx_id `[2u8; 32]`. It accepts the bridge action for a transaction that was never actually verified. [8](#0-7) [9](#0-8)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
