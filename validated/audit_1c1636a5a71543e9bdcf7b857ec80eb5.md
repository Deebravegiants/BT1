Audit Report

## Title
Uninitialized `tokenFeeBps` Mapping Allows Zero-Fee Token Deposits After `addSupportedToken()` - (File: contracts/pools/RSETHPool.sol)

## Summary
`RSETHPool.sol` never initializes `tokenFeeBps[token]` when a new token is added via `addSupportedToken()`, leaving it at the Solidity default of zero. Any depositor who calls `deposit(token, amount, referralId)` in the window between `addSupportedToken()` and a subsequent `setTokenFeeBps()` call pays 0% fee and receives the full token value converted to rsETH. The protocol permanently loses the fee revenue from all deposits made in that window.

## Finding Description
`addSupportedToken()` (L637–656) stores the oracle and bridge for a new token but never sets `tokenFeeBps[token]`:

```solidity
supportedTokenList.push(token);
supportedTokenOracle[token] = oracle;
tokenBridge[token] = bridge;
// tokenFeeBps[token] remains 0
emit AddSupportedToken(token, oracle, bridge);
```

Setting the per-token fee is a separate privileged call, `setTokenFeeBps()` (L583–594), gated by `DEFAULT_ADMIN_ROLE` — a different role from the `TIMELOCK_ROLE` required by `addSupportedToken()`. This structural separation makes atomic initialization impossible in a single transaction.

The deposit path at L284–305 calls `viewSwapRsETHAmountAndFee(amount, token)` (L326–347), which reads `tokenFeeBps[token]` directly:

```solidity
uint256 feeBpsForToken = tokenFeeBps[token]; // 0 until setTokenFeeBps is called
fee = amount * feeBpsForToken / 10_000;      // always 0
uint256 amountAfterFee = amount - fee;       // full amount
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate; // no fee deducted
```

`feeEarnedInToken[token] += fee` accumulates nothing (L300), and the depositor receives rsETH equivalent to the full token value. No existing guard prevents deposits during this window — `onlySupportedToken` passes as soon as `addSupportedToken()` completes, and `whenNotPaused` is unrelated to fee initialization.

## Impact Explanation
Every token deposit made between `addSupportedToken()` and `setTokenFeeBps()` pays zero protocol fee. The protocol's `feeEarnedInToken[token]` permanently records nothing for those deposits, and the fee revenue is irrecoverable. This matches **High — Theft of unclaimed yield** in the allowed impact scope.

## Likelihood Explanation
The window is structurally guaranteed on every new token listing: `addSupportedToken()` (`TIMELOCK_ROLE`) and `setTokenFeeBps()` (`DEFAULT_ADMIN_ROLE`) are separate roles that cannot be batched atomically. The window spans at minimum one block and potentially hours or longer if the admin delays. Any unprivileged depositor — including a mempool-monitoring attacker — can call `deposit(token, amount, referralId)` immediately after `addSupportedToken()` confirms, with no special capability required beyond holding the token.

## Recommendation
Accept a `_feeBps` parameter in `addSupportedToken()` and initialize `tokenFeeBps[token]` atomically:

```diff
function addSupportedToken(
    address token,
    address oracle,
-   address bridge
+   address bridge,
+   uint256 _feeBps
) external onlyRole(TIMELOCK_ROLE) {
    // existing checks ...
+   if (_feeBps > 10_000) revert InvalidFeeAmount();
    supportedTokenList.push(token);
    supportedTokenOracle[token] = oracle;
    tokenBridge[token] = bridge;
+   tokenFeeBps[token] = _feeBps;
+   emit TokenFeeBpsSet(token, _feeBps);
    emit AddSupportedToken(token, oracle, bridge);
}
```

This mirrors how `feeBps` is set atomically for ETH deposits in `initialize()` (L248) and eliminates the uninitialized window entirely.

## Proof of Concept
1. Admin calls `addSupportedToken(rETH, rETHOracle, rETHBridge)` — `tokenFeeBps[rETH]` is now 0.
2. Attacker (or any depositor) immediately calls `deposit(rETH, 100e18, "")`.
3. Inside `viewSwapRsETHAmountAndFee(100e18, rETH)`: `feeBpsForToken = 0`, `fee = 0`, `amountAfterFee = 100e18`, rsETH minted at full value.
4. `feeEarnedInToken[rETH] += 0` — protocol earns nothing.
5. Admin later calls `setTokenFeeBps(rETH, 50)` — too late; prior deposits settled at 0%.

**Foundry fork test plan**: Fork Arbitrum mainnet, deploy/upgrade `RSETHPool`, call `addSupportedToken` with a mock token and oracle, immediately call `deposit` as an unprivileged address, assert `feeEarnedInToken[token] == 0` and that the depositor received rsETH equal to the full unfeeed conversion. Then call `setTokenFeeBps` and repeat the deposit to confirm fees are now collected, demonstrating the window is real and bounded.
