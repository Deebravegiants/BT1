The code has two distinct paths in `parse_metadata_account`. I need to evaluate each separately.

### Title
Missing `metadata.mint` Cross-Check in `parse_metadata_account` Allows Token-2022 Mint to Steal Name/Symbol from Arbitrary Metaplex Account — (`solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs`)

---

### Summary

`LogMetadata::parse_metadata_account` verifies the provided metadata account's **address** but never verifies that the deserialized `metadata.mint` field equals `self.mint.key()`. For the classic SPL path this is harmless because the expected address is derived from the mint key (PDA binding). For the Token-2022 "third-party metadata pointer" path the expected address is read directly from the attacker-controlled `MetadataPointer` extension, so an attacker can point their mint at any existing Metaplex account (e.g., USDT's) and the bridge will emit a Wormhole VAA claiming the attacker's mint is "Tether USD / USDT."

---

### Finding Description

`process()` has two branches:

**Classic SPL (safe):** The expected metadata address is computed as:
```rust
Pubkey::find_program_address(
    &[METADATA_SEED, MetaplexID.as_ref(), &self.mint.key().to_bytes()],
    &MetaplexID,
).0
``` [1](#0-0) 

Because the PDA seeds include `self.mint.key()`, the expected address is cryptographically bound to the mint. The attacker can only supply their own mint's metadata account. This path is safe from cross-mint substitution.

**Token-2022 third-party metadata pointer (vulnerable):** The expected address is taken directly from the mint's `MetadataPointer` extension:
```rust
} else if metadata_pointer.metadata_address.0 != Pubkey::default() {
    // Third-party metadata
    self.parse_metadata_account(metadata_pointer.metadata_address.0)?
``` [2](#0-1) 

The attacker creates the Token-2022 mint, so they fully control `metadata_pointer.metadata_address`. They can set it to the Metaplex PDA of any existing token (e.g., USDT's canonical PDA `find_program_address([b"metadata", MetaplexID, usdt_mint], MetaplexID)`).

Inside `parse_metadata_account`, the only checks are:
1. `require_keys_eq!(metadata.key(), address, ...)` — passes because the attacker set the pointer to USDT's PDA and provides that account.
2. `if metadata.owner == &MetaplexID` — passes because USDT's metadata is owned by MetaplexID.
3. `MplMetadata::try_deserialize(...)` — succeeds, returning `name = "Tether USD"`, `symbol = "USDT"`.

```rust
if metadata.owner == &MetaplexID {
    let data = metadata.try_borrow_data()?;
    let metadata = MplMetadata::try_deserialize(&mut data.as_ref())?;
    Ok((metadata.name.clone(), metadata.symbol.clone()))
``` [3](#0-2) 

There is no `require_keys_eq!(metadata.mint, self.mint.key(), ...)` check. The deserialized `metadata.mint` (USDT's mint) is never compared against `self.mint.key()` (the attacker's mint).

The resulting `LogMetadataPayload` then encodes the attacker's mint pubkey alongside USDT's name/symbol:
```rust
let payload = LogMetadataPayload {
    token: self.mint.key(),   // attacker's mint
    name: ...,                // "Tether USD"
    symbol: ...,              // "USDT"
    decimals: self.mint.decimals,
}
``` [4](#0-3) 

This is posted as a Wormhole VAA. On NEAR, `deploy_token_callback` verifies only that the emitter is a known factory (the Solana bridge program), then calls `deploy_token_internal` with `name = "Tether USD"`, `symbol = "USDT"` for the attacker's mint address: [5](#0-4) 

---

### Impact Explanation

An unprivileged attacker can register their worthless Token-2022 mint on NEAR under the name and symbol of any well-known token (USDT, USDC, wBTC, etc.). NEAR deploys a wrapped token contract for the attacker's mint with the impersonated name/symbol. Users who bridge the attacker's token receive a NEAR token labeled "Tether USD / USDT" that is backed by the attacker's worthless mint. This is a concrete asset-identity / token-mapping divergence that breaks backing guarantees and enables impersonation-based fund loss.

---

### Likelihood Explanation

- Requires no privileged access: creating a Token-2022 mint with an arbitrary `MetadataPointer` extension is a standard, permissionless Solana operation.
- The only on-chain constraint on the mint is `!mint.mint_authority.contains(authority.key)` (i.e., not a bridge-wrapped token), which the attacker trivially satisfies by using their own key as mint authority.
- The target Metaplex metadata account (e.g., USDT's) already exists on-chain; no crafting is needed.
- The attack is a single transaction followed by a standard Wormhole relay.

---

### Recommendation

In `parse_metadata_account`, after deserializing the Metaplex account, add a cross-check:

```rust
if metadata.owner == &MetaplexID {
    let data = metadata.try_borrow_data()?;
    let mpl = MplMetadata::try_deserialize(&mut data.as_ref())?;
    // Ensure the metadata account actually belongs to the mint being registered
    require_keys_eq!(
        mpl.mint,
        self.mint.key(),
        ErrorCode::InvalidTokenMetadataAddress,
    );
    Ok((mpl.name.clone(), mpl.symbol.clone()))
}
```

This closes the Token-2022 third-party pointer path. The classic SPL path is already safe due to PDA derivation from the mint key.

### Proof of Concept

1. Attacker identifies USDT's Metaplex metadata PDA:
   ```
   usdt_metadata_pda = find_program_address(
       [b"metadata", MetaplexID, usdt_mint_pubkey], MetaplexID
   )
   ```

2. Attacker creates a Token-2022 mint with `MetadataPointer` extension pointing to `usdt_metadata_pda`:
   ```
   spl-token create-token --program-id TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb \
       --metadata-address <usdt_metadata_pda>
   ```

3. Attacker calls `log_metadata` with their Token-2022 mint and passes `usdt_metadata_pda` as the `metadata` account.

4. `parse_metadata_account` is called with `address = usdt_metadata_pda`. The `require_keys_eq!` passes. The owner check passes. `MplMetadata::try_deserialize` returns `name = "Tether USD"`, `symbol = "USDT"`.

5. A Wormhole VAA is posted: `{token: attacker_mint, name: "Tether USD", symbol: "USDT", decimals: attacker_decimals}`.

6. Relayer submits the VAA to NEAR `deploy_token`. NEAR deploys a wrapped token for `attacker_mint` labeled "Tether USD / USDT".
