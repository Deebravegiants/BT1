### Title
Fee-on-Transfer Token Undercollateralization in `initTransfer` - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
`OmniBridge.initTransfer` records and emits the caller-supplied `amount` as the locked quantity without verifying the actual balance received. For any ERC-20 token that deducts a transfer fee internally (fee-on-transfer tokens), the bridge locks fewer tokens than it announces, causing the destination chain to mint or release more tokens than are held in custody.

### Finding Description
In `OmniBridge.sol`, the `initTransfer` function handles non-bridge, non-custom-minter ERC-20 tokens with a plain `safeTransferFrom`:

```solidity
// lines 406-411
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount          // requested, not verified
    );
}
```

Immediately after, the function emits the `InitTransfer` event using the same caller-supplied `amount`:

```solidity
// lines 427-436
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,             // ← emitted as locked amount, no balance-delta check
    fee,
    nativeFee,
    recipient,
    message
);
``` [1](#0-0) [2](#0-1) 

There is no balance-before / balance-after check anywhere in the function. `safeTransferFrom` succeeds as long as the token does not revert, but for fee-on-transfer tokens it credits the contract with `amount − fee_taken`, not `amount`. The event — which is the sole source of truth consumed by the NEAR prover to authorize minting or releasing tokens on the destination chain — still claims the full `amount` was locked.

### Impact Explanation
When the NEAR bridge processes the proof of the `InitTransfer` event it reads `amount` from the event log and mints or releases exactly that many tokens to the recipient. The EVM bridge holds only `amount − transfer_fee`. Every such transfer permanently widens the gap between EVM custody and destination-chain supply, breaking the 1:1 backing guarantee. Repeated transfers drain the EVM reserve until legitimate redemptions fail (irreversible fund lock for later redeemers) or the destination supply becomes unbacked (unauthorized creation of wrapped assets).

This maps directly to the allowed impact categories:
- **Critical** – Unauthorized creation of wrapped bridge assets through settlement failure.
- **Critical** – Irreversible fund lock / permanently unclaimable value for later redeemers.

### Likelihood Explanation
`initTransfer` is a fully public, permissionless entry point. The only prerequisite is that the token in question is registered on both sides of the bridge. Token registration on EVM (`deployToken`) requires an MPC-signed `MetadataPayload`, but this is a normal operational step performed for every legitimate token. Several widely-used ERC-20 tokens (e.g., tokens with built-in redistribution or reflection mechanics) implement fee-on-transfer. Once any such token is registered through the standard flow, every ordinary user bridging it triggers the accounting divergence — no malicious intent is required.

### Recommendation
Record the actual balance delta rather than trusting the caller-supplied `amount`:

```solidity
} else {
    uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
    uint256 received = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
    require(received == amount, "ERR_FEE_ON_TRANSFER_TOKEN");
}
```

Either enforce `received == amount` (rejecting fee-on-transfer tokens outright, which is the safest option) or use `received` as the canonical locked amount in the emitted event and all downstream accounting. Document the chosen policy explicitly.

### Proof of Concept
1. A fee-on-transfer ERC-20 token `FoT` (10 % fee) is registered through the normal `deployToken` / `bind_token` flow.
2. Alice calls `initTransfer(FoT, 1000, 0, 0, "near:alice.near", "")`.
3. `safeTransferFrom` transfers 1000 from Alice; `FoT` internally deducts 100, so the bridge receives 900.
4. The `InitTransfer` event is emitted with `amount = 1000`.
5. The NEAR prover verifies the event and mints 1000 wrapped `FoT` to Alice on NEAR.
6. The EVM bridge holds only 900 `FoT`. The 100-token shortfall is permanent.
7. After 10 such transfers the bridge holds 9000 `FoT` but has issued 10000 wrapped tokens. The last redeemer cannot withdraw. [3](#0-2)

    function initTransfer(
        address tokenAddress,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message
    ) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
        currentOriginNonce += 1;
        if (fee >= amount) {
            revert InvalidFee();
        }

        uint256 extensionValue;
        if (tokenAddress == address(0)) {
            if (fee != 0) {
                revert InvalidFee();
            }
            extensionValue = msg.value - amount - nativeFee;
        } else {
            extensionValue = msg.value - nativeFee;
            if (customMinters[tokenAddress] != address(0)) {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    customMinters[tokenAddress],
                    amount
                );
                ICustomMinter(customMinters[tokenAddress]).burn(
                    tokenAddress,
                    amount
                );
            } else if (isBridgeToken[tokenAddress]) {
                BridgeToken(tokenAddress).burn(msg.sender, amount);
            } else {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    address(this),
                    amount
                );
            }
        }

        initTransferExtension(
            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message,
            extensionValue
        );

        emit BridgeTypes.InitTransfer(
            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message
        );
    }
