### Title
`deploy_token` Metadata Signature Lacks Chain-ID Binding — Cross-Chain Replay Deploys Tokens on Unintended Chains - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `aptos/sources/bridge_types.move`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The MPC-signed `MetadataPayload` used by `deploy_token` across all destination chains (EVM, Starknet, Aptos, Solana) does not include any destination chain identifier in the signed message. A signature produced for deploying a token on chain A is byte-identical to a valid signature for deploying the same token on chain B. Any unprivileged observer can extract the signature from a public transaction on chain A and replay it on chain B, deploying the token there without authorization.

---

### Finding Description

When NEAR's `log_metadata_callback` requests an MPC signature, it signs a Borsh-encoded `MetadataPayload` containing only:

```
[PayloadType::Metadata | token_id | name | symbol | decimals]
```

No destination chain ID is included. [1](#0-0) 

This same chain-ID-free layout is reproduced identically on every destination chain:

### Impact Explanation

An attacker who observes a valid `deployToken` transaction on chain A (e.g., Ethereum) can extract the signature from calldata and submit it to `deployToken` on chain B (e.g., Arbitrum, Base, BNB, Polygon, Starknet, Aptos, Solana). The result:

1. **Unauthorized token deployment**: A bridge-wrapped token is deployed and registered in the bridge's token mappings on chain B without the protocol's authorization for that chain.
2. **Permanent griefing of legitimate deployment**: Once the token is registered (`nearToEthToken[metadata.token]` is set), any subsequent legitimate `deployToken` call for the same NEAR token ID on chain B reverts with `ERR_TOKEN_EXIST`. The legitimate deployment path is permanently blocked.
3. **Unbacked supply risk**: The attacker can then submit the `DeployToken` event proof from chain B to NEAR's `deploy_token`, which will pass the factory check (the emitter IS the legitimate factory) and register the token on NEAR as deployed on chain B. This opens a bridge path to chain B that was never authorized, potentially creating unbacked wrapped supply if the chain B infrastructure is not ready.

---

### Likelihood Explanation

- The signature is fully public in transaction calldata the moment the first `deployToken` succeeds on any chain.
- The attacker needs no privileged access, no key material, and no special tooling — only the ability to read on-chain data and submit a transaction.
- All destination-chain `deployToken` functions are permissionless entry points.
- The attack is profitable as a griefing vector (blocking competitors or disrupting protocol rollout) and requires zero cost beyond gas.

---

### Recommendation

Include the destination chain ID in the `MetadataPayload` Borsh encoding, mirroring the pattern already used correctly in `TransferMessagePayload`. On NEAR's signing side, add the target `chain_kind` to the payload before hashing:

```rust
let metadata_payload = MetadataPayload {
    prefix: PayloadType::Metadata,
    destination_chain: chain_kind,  // ADD THIS
    token: token_id.to_string(),
    name: metadata.name,
    symbol: metadata.symbol,
    decimals: metadata.decimals,
};
```

Each destination chain must then include its own chain ID in the Borsh encoding it hashes for signature verification, exactly as `transfer_message_to_borsh` already does with `chain_id`.

---

### Proof of Concept

1. Relayer calls `deployToken(sig, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})` on Ethereum mainnet. Transaction is mined; `sig` is visible in calldata.
2. Attacker copies `sig` and calls `deployToken(sig, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})` on Arbitrum.
3. Arbitrum's `OmniBridge.deployToken` reconstructs the identical Borsh bytes (no chain ID), hashes them, recovers the same MPC address, and passes signature verification.
4. A `BridgeToken` proxy for USDC is deployed and registered on Arbitrum's bridge without authorization.
5. The legitimate relayer's subsequent attempt to deploy USDC on Arbitrum reverts with `ERR_TOKEN_EXIST`.
6. Attacker submits the Arbitrum `DeployToken` event proof to NEAR's `deploy_token`; NEAR registers USDC as deployed on Arbitrum, opening an unauthorized bridge path.
