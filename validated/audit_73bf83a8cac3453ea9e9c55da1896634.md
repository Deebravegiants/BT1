Audit Report

## Title
Cross-Chain rsETH Amount Mismatch Due to Oracle Rate Divergence Between L2 Mint and L1 Deposit Causes L2 Wrapper Insolvency — (`contracts/L1Vault.sol`)

## Summary

The L2 pool (`RSETHPoolV2`/`RSETHPoolV3`) immediately mints `wrsETH` to depositors using the rsETH/ETH oracle rate at L2 deposit time (R1). The `L1Vault` later mints rsETH via `LRTDepositPool` using the rsETH price at L1 deposit time (R2). Because rsETH is a yield-bearing token whose price monotonically increases, R2 ≥ R1 under normal operation. When R2 > R1, the L1 mints strictly fewer rsETH than the L2 already committed, leaving `RsETHTokenWrapper` undercollateralized. Users who attempt to redeem `wrsETH` after the deficit is reached have their `safeTransfer` revert, permanently freezing their funds.

## Finding Description

**L2 mint path — `RSETHPoolV2.deposit` (ETH):**

`wrsETH` is minted immediately at the current L2 oracle rate: [1](#0-0) 

`rsETHAmount = amountAfterFee * 1e18 / R1` where `R1 = IOracle(rsETHOracle).getRate()` at L2 deposit time. [2](#0-1) 

**L2 mint path — `RSETHPoolV3.deposit` (tokens):**

Same pattern for token deposits: [3](#0-2) [4](#0-3) 

**L1 mint path — `L1Vault.depositAssetForL1Vault`:**

The MANAGER calls this after the bridge finality delay. rsETH is minted at the current L1 oracle rate R2: [5](#0-4) 

`rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()` where `rsETHPrice` is the L1 oracle value at the time of this call: [6](#0-5) 

**The structural gap:**

The mandatory multi-step sequence — bridge finality (7 days for optimistic rollups) + `unwrapWstETH()` + `depositAssetForL1Vault()` + `bridgeRsETHToL2()` — introduces a time window during which `updateRSETHPrice()` can be called by anyone: [7](#0-6) 

`_updateRsETHPrice()` recomputes rsETH price from total ETH in protocol divided by rsETH supply. As staking rewards accrue, this value increases monotonically. Any call to `updateRSETHPrice()` between L2 deposit and L1 deposit raises R2 above R1.

**Broken invariant in `RsETHTokenWrapper`:**

`_withdraw` performs a strict 1:1 burn-and-transfer: [8](#0-7) 

`maxAmountToDepositBridgerAsset` confirms the invariant the protocol relies on (`rsETH_balance >= wrsETH_totalSupply`): [9](#0-8) 

When L2 minted X wrsETH but L1 only bridges Y < X rsETH, the wrapper holds a deficit of (X − Y). The `safeTransfer` in `_withdraw` reverts for any redeemer once the rsETH balance is exhausted.

**No automatic remediation exists.** The only recovery path is the BRIDGER manually calling `depositBridgerAssets` with additional rsETH sourced externally, which requires ongoing protocol capital expenditure proportional to deposit volume and yield accrual rate.

## Impact Explanation

The `RsETHTokenWrapper` on L2 becomes structurally undercollateralized with every deposit processed during any period of rsETH price appreciation. The deficit is permanent absent manual top-up. Users who deposited legitimately and hold `wrsETH` find their `withdraw()` calls reverting once the rsETH balance is exhausted — their funds are permanently frozen in the wrapper with no on-chain recovery path. At scale, the cumulative deficit constitutes protocol insolvency on L2. This matches **Critical: Permanent freezing of funds** and **Critical: Protocol insolvency**.

## Likelihood Explanation

rsETH price increases continuously as Ethereum staking rewards accrue (~4–5% APY). `updateRSETHPrice()` is permissionlessly callable by any address. The cross-chain processing delay is structurally unavoidable: optimistic rollup finality alone is 7 days, plus MANAGER operational latency. Over 7 days at 5% APY, the rate increases by ~0.096%, meaning every $1M of L2 deposits generates ~$960 of unrecoverable deficit. The deficit accumulates with every deposit batch and compounds over time. No special attacker capability is required — normal user deposits on L2 combined with normal yield accrual on L1 are sufficient to trigger the condition.

## Recommendation

1. **Commit the L2 oracle rate cross-chain.** Record `R1` at L2 deposit time and pass it as a `minRsETHExpected` parameter through the bridge message. `depositAssetForL1Vault` should revert if `getRsETHAmountToMint` returns less than the committed amount, forcing the protocol to cover the shortfall before bridging.
2. **Two-phase commit.** Delay wrsETH minting on L2 until L1 confirms the actual rsETH minted amount, accepting that users wait for L1 finality. This eliminates the rate divergence entirely.
3. **At minimum**, add a protocol reserve mechanism that automatically tops up the wrapper deficit before each `bridgeRsETHToL2` call, funded by protocol fees, so the invariant `rsETH_balance >= wrsETH_totalSupply` is maintained at all times.

## Proof of Concept

```solidity
// Differential model test — no fork required
function testCrossChainRateMismatch() external pure {
    uint256 depositAmount = 1e18;          // 1 ETH deposited on L2
    uint256 R1 = 1.05e18;                  // L2 oracle rate at deposit time
    uint256 R2 = 1.06e18;                  // L1 oracle rate after 7-day bridge delay + yield accrual

    // RSETHPoolV2: wrsETH minted to user on L2
    uint256 wrsETHMinted = depositAmount * 1e18 / R1;  // 0.952380...e18

    // L1Vault: rsETH minted via LRTDepositPool at L1 rate
    uint256 rsETHMintedOnL1 = depositAmount * 1e18 / R2; // 0.943396...e18

    // RsETHTokenWrapper deficit
    assert(rsETHMintedOnL1 < wrsETHMinted);
    // deficit = wrsETHMinted - rsETHMintedOnL1 = ~0.008984e18 per 1 ETH deposit

    // First redeemer withdraws rsETHMintedOnL1 wrsETH → succeeds
    // Remaining wrsETH holders (deficit amount) → _withdraw safeTransfer reverts
    // Funds permanently frozen
}
```

Steps to reproduce on a fork:
1. Deploy `RSETHPoolV2` with `rsETHOracle` returning R1 = 1.05e18.
2. User calls `deposit{value: 1e18}("")` → 0.9523e18 wrsETH minted.
3. ETH bridged to `L1Vault` (simulate transfer).
4. Call `LRTOracle.updateRSETHPrice()` to advance price to R2 = 1.06e18.
5. MANAGER calls `L1Vault.depositETHForL1VaultETH()` → 0.9433e18 rsETH minted.
6. MANAGER calls `L1Vault.bridgeRsETHToL2(0.9433e18, ...)` → wrapper receives 0.9433e18 rsETH.
7. Assert `wrsETH.totalSupply()` (0.9523e18) > `rsETH.balanceOf(wrapper)` (0.9433e18).
8. First redeemer withdraws 0.9433e18 → succeeds.
9. Second redeemer attempts to withdraw remaining 0.009e18 → `safeTransfer` reverts (insufficient rsETH balance).

        UtilLib.checkNonZeroAddress(token);

        uint256 tokenBalance = IERC20(token).balanceOf(address(this));
        uint256 rsETHAmountToMint = lrtDepositPool.getRsETHAmountToMint(token, tokenBalance);

            revert InvalidMinRSETHAmountExpected();

        lrtDepositPool.depositAsset(token, tokenBalance, rsETHAmountToMint, "");

        emit AssetDepositForL1Vault(token, tokenBalance, rsETHAmountToMint);
    }
