### Title
Missing `request`↔`payload_hash` Binding in `respond_verify_foreign_tx` Allows Cross-Request Signature Replay — (`crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted ECDSA signature is valid over `response.payload_hash`, but never checks that `response.payload_hash` was derived from the `request` parameter passed in the same call. A Byzantine MPC node (single participant, below signing threshold) can replay a legitimately-produced signature from a prior request R1 to resolve a different pending request R2, causing the contract to deliver a false foreign-chain attestation to R2's caller.

---

### Finding Description

The verification logic in `respond_verify_foreign_tx` performs two independent checks:

1. **Signature validity**: `sig` is valid over `response.payload_hash` under the domain's root public key.
2. **Request existence**: `request` exists as a key in `pending_verify_foreign_tx_requests`. [1](#0-0) 

There is no check that `response.payload_hash == SHA-256(borsh(ForeignTxSignPayload { request, values }))`. The `request` key is used only to look up and drain the pending yield queue; it is never used to validate what was actually signed.

The canonical payload that the MPC network signs is:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request: ForeignChainRpcRequest, values: Vec<ExtractedValue> }))
``` [2](#0-1) 

Because `values` (the extracted on-chain data) are not stored in the contract and are not part of the `request` key, the contract has no way to reconstruct the expected `payload_hash` from `request` alone — but it also makes no attempt to do so. The `response.payload_hash` field is fully attacker-controlled.

---

### Impact Explanation

**Attack sequence:**

1. Byzantine MPC node (single, below threshold) observes a legitimately completed response for request R1 (BNB tx_id_1): `{payload_hash: H_R1, signature: sig_R1}`. Both are public on-chain.
2. A different user submits request R2 (BNB tx_id_2, same domain). R2 is now pending in `pending_verify_foreign_tx_requests`.
3. Byzantine node calls `respond_verify_foreign_tx(request=R2, response={payload_hash=H_R1, signature=sig_R1})`.
4. Contract checks: is `sig_R1` valid over `H_R1` under the root key? **Yes** — it was produced by the threshold network for R1.
5. Contract resolves R2's yield with `{payload_hash=H_R1, signature=sig_R1}`.
6. R2's caller receives a `VerifyForeignTransactionResponse` that attests to R1's transaction data, not R2's.

The contract has delivered a false attestation: it resolved R2's yield with a signature that was produced for a completely different transaction.

**Downstream impact:** Any bridge contract that calls `verify_foreign_transaction` and acts on the resolved yield without re-verifying `payload_hash == hash(expected_request + expected_values)` will accept the forged attestation. The SDK's `ForeignChainSignatureVerifier::verify_signature` does perform this check client-side: [3](#0-2) 

However, this is a caller-side SDK helper, not enforced by the contract. A bridge contract that trusts the contract's yield resolution as the attestation (without calling `verify_signature`) is fully exploitable.

---

### Likelihood Explanation

- **Attacker capability required**: A single Byzantine attested MPC participant. `respond_verify_foreign_tx` only requires `assert_caller_is_attested_participant_and_protocol_active()` — no threshold, no multi-party agreement. [4](#0-3) 
- **Signature material**: The recycled `(sig_R1, H_R1)` pair is fully public on-chain from the prior legitimate response.
- **Precondition**: R2 must be pending. Any user can submit R2 to create this precondition.
- **No forgery needed**: The signature is genuine — it was produced by the threshold network. The attack is a replay, not a cryptographic break.

---

### Recommendation

Add a binding check inside `respond_verify_foreign_tx` that verifies `response.payload_hash` was derived from the `request` field. Since `values` (extracted data) are not stored in the contract, the response must also carry them, or the contract must store them at request time. The minimal fix is to require the responder to also submit the `extracted_values` alongside the response, then verify:

```rust
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(),
}).compute_msg_hash()?;
assert_eq!(expected_hash, response.payload_hash, "payload_hash does not match request");
```

This mirrors the binding already enforced client-side in `ForeignChainSignatureVerifier::verify_signature`. [5](#0-4)

### Proof of Concept

```rust
// Unit test sketch (extend existing test harness in crates/contract/src/lib.rs)
fn respond_verify_foreign_tx__should_reject_recycled_signature_for_different_request() {
    // Setup: two distinct BNB requests, R1 and R2
    let r1 = VerifyForeignTransactionRequest { request: ForeignChainRpcRequest::Bnb(EvmRpcRequest { tx_id: EvmTxId([0x01; 32]), .. }), .. };
    let r2 = VerifyForeignTransactionRequest { request: ForeignChainRpcRequest::Bnb(EvmRpcRequest { tx_id: EvmTxId([0x02; 32]), .. }), .. };

    // Queue both
    contract.verify_foreign_transaction(r1_args);
    contract.verify_foreign_transaction(r2_args);

    // Produce a legitimate response for R1
    let payload_r1 = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: r1.request.clone(), values: extracted_values.clone() });
    let h_r1 = payload_r1.compute_msg_hash().unwrap();
    let (sig_r1, rec) = signing_key.sign_prehash_recoverable(&h_r1.0).unwrap();
    let response_r1 = VerifyForeignTransactionResponse { payload_hash: h_r1, signature: Secp256k1(sig_r1) };

    // Legitimate respond for R1
    contract.respond_verify_foreign_tx(r1.clone(), response_r1.clone()).unwrap();

    // ATTACK: replay R1's response against R2
    // Currently SUCCEEDS — should FAIL
    let result = contract.respond_verify_foreign_tx(r2.clone(), response_r1);
    assert!(result.is_err(), "contract must reject payload_hash that does not correspond to request R2");
}
```

The test currently passes (the contract accepts the recycled response), demonstrating the missing binding check. [6](#0-5)
