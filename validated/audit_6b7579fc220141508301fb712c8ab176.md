### Title
`validatePerasCert` Unconditionally Returns `True`, Permanently Bypassing Peras Certificate Validation in the BBODY Rule — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/BlockBody/Internal.hs`)

---

### Summary

The Dijkstra-era BBODY transition rule calls `validatePerasCert` to validate any Peras certificate present in a block body. However, the production implementation of `validatePerasCert` is a stub that unconditionally returns `True` regardless of its inputs (nonce, key, certificate bytes). This is a direct structural analog to the Maia bug: just as `bytes1(0x02) & 0x0F` always evaluated to `bytes1(0x02)` regardless of the `_hasFallbackToggled` boolean, `validatePerasCert _ _ _ = True` always evaluates to `True` regardless of the certificate content. The result is that the `PerasCertValidationFailed` predicate failure can never be triggered, and any block producer can include an arbitrary or malformed Peras certificate in a Dijkstra block and have it unconditionally accepted by all honest nodes.

---

### Finding Description

**Root cause — stub always returns `True`:** [1](#0-0) 

```haskell
-- | Mocked-up Peras certificate validation routine
--
-- NOTE: this function will be replaced with the real implementation from
-- 'cardano-base' once it's ready.
validatePerasCert :: Nonce -> PerasKey -> PerasCert -> Bool
validatePerasCert _ _ _ = True
```

**Call site in the BBODY transition rule:** [2](#0-1) 

```haskell
case blockBody ^. perasCertBlockBodyL of
  SNothing -> pure ()
  SJust cert ->
    let nonce = block ^. prevNonceBlockHeaderL
     in validatePerasCert nonce PerasKey cert
          ?! injectFailure (PerasCertValidationFailed cert nonce)
```

The `?!` operator fails the rule if and only if the condition is `False`. [3](#0-2) 

Because `validatePerasCert _ _ _ = True`, the condition is always `True`, so `PerasCertValidationFailed` is structurally unreachable. The entire Peras certificate validation branch is a no-op.

**Compounding issue — `PerasKey` is also a stub:** [4](#0-3) 

```haskell
data PerasKey = PerasKey
  deriving (Eq, Show, Generic, NoThunks)
```

The key passed to `validatePerasCert` at the call site is the unit-like constructor `PerasKey` — not a key derived from any protocol state or block header. Even if `validatePerasCert` were replaced with a real implementation, it would receive a dummy key, making the check semantically meaningless unless the key source is also corrected.

**Encoding/decoding of `PerasCert` is fully attacker-controlled:** [5](#0-4) 

```haskell
newtype PerasCert = PerasCert ByteArray
  deriving (Eq, Show, Generic)
  deriving newtype (EncCBOR, DecCBOR)
```

`PerasCert` is an opaque byte array with no structural constraints. A block producer can serialize any byte sequence as a `PerasCert` and it will be decoded and accepted without any validation.

---

### Impact Explanation

This matches the allowed impact: **"Critical. Honest nodes accept an invalid block or transaction causing permanent ledger divergence requiring a hard fork."**

When the real Peras certificate validation is eventually deployed (replacing the stub), nodes running the new implementation would reject blocks containing invalid Peras certificates that were previously accepted by all nodes running the stub. This creates a permanent split in ledger history between nodes on different software versions, requiring a hard fork to reconcile. Additionally, during the stub period, any block producer can include arbitrary bytes as a Peras certificate with zero validation cost, polluting the chain with semantically invalid data that future validators must handle.

---

### Likelihood Explanation

The entry path requires only a block producer (below consensus threshold) to set the optional `perasCertBlockBodyL` field in a Dijkstra block body to any non-null value. The encoding path confirms this field is serialized as a nullable CBOR field: [6](#0-5) 

No special privilege, key material, or governance action is required. Any block producer can trigger this on every block they produce.

---

### Recommendation

1. Replace `validatePerasCert _ _ _ = True` with the real implementation from `cardano-base` before the Dijkstra era is activated on any network.
2. Replace the `PerasKey = PerasKey` stub with a real key type sourced from protocol state or the block header.
3. Add a compile-time or CI guard (e.g., a `TODO` lint or a failing test) that prevents deployment of the Dijkstra era while `validatePerasCert` remains a stub.

---

### Proof of Concept

A block producer constructs a `DijkstraBlockBody` with `dbbrPerasCert = SJust (PerasCert arbitraryBytes)` for any `arbitraryBytes`. During `dijkstraBbodyTransition`, the rule reaches:

```haskell
validatePerasCert nonce PerasKey cert ?! injectFailure (PerasCertValidationFailed cert nonce)
```

Since `validatePerasCert _ _ _ = True`, the `?!` check passes unconditionally. The block is accepted. The `PerasCertValidationFailed` failure is never emitted regardless of the certificate content. [7](#0-6)
