Audit Report

## Title
Stale Cached `rsETHPrice` Used in Deposit Minting Allows Yield Theft from Existing rsETH Holders - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTOracle` stores the rsETH/ETH exchange rate as a persistent state variable `rsETHPrice` that is only updated by explicit calls to `updateRSETHPrice()`. The deposit flow in `LRTDepositPool` — both `depositETH()` and `depositAsset()` — never calls `updateRSETHPrice()` or `_updateRsETHPrice()` before computing the mint amount, so any deposit made while the stored price lags behind the true on-chain TVL mints excess rsETH, permanently diluting the yield share of all existing rsETH holders.

## Finding Description
`LRTOracle.rsETHPrice` is a plain storage variable updated only by `updateRSETHPrice()` / `_updateRsETHPrice()`:

```solidity
// LRTOracle.sol L28
uint256 public override rsETHPrice;

// LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

The deposit entry points call `_beforeDeposit()`, which calls `getRsETHAmountToMint()`, which divides by the stored `rsETHPrice`:

```solidity
// LRTDepositPool.sol L519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

A grep of `LRTDepositPool.sol` for `updateRSETHPrice` or `_updateRsETHPrice` returns **zero matches** — the price is never refreshed atomically within the deposit flow. `_beforeDeposit` is `private view` and performs only limit and minimum-amount checks before returning the stale-price-derived mint amount:

```solidity
// LRTDepositPool.sol L648-670
function _beforeDeposit(...) private view returns (uint256 rsethAmountToMint) {
    ...
    rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
    if (rsethAmountToMint < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet();
}
```

Between keeper calls to `updateRSETHPrice()`, EigenLayer strategy balances and EigenPod rewards increase `totalETHInProtocol`. The true price `P_true = totalETHInProtocol / rsethSupply` rises above the stored `P_stored`. Because `rsethAmountToMint = deposit / P_stored`, using the lower stale denominator inflates the rsETH minted. When `updateRSETHPrice()` is eventually called, the new price is computed over the now-inflated supply, permanently locking in the dilution of existing holders.

## Impact Explanation
**High — Theft of unclaimed yield.** Every deposit made during a staleness window extracts a portion of accrued-but-not-yet-priced staking rewards from existing rsETH holders. The excess rsETH minted represents a real economic transfer: rewards that had accrued to existing holders are redistributed to the depositor. The effect is permanent and compounds with deposit size, staleness duration, and reward accrual rate. This maps directly to the allowed impact "High. Theft of unclaimed yield."

## Likelihood Explanation
**Medium.** No privileged access is required. The staleness window is structurally guaranteed — there is always a non-zero gap between reward accrual events in EigenLayer and the next keeper call to `updateRSETHPrice()`. A sophisticated attacker can monitor `totalETHInProtocol` on-chain, compute the true current price, compare it to the stored `rsETHPrice`, and deposit whenever the gap is profitable. The attack is repeatable on every keeper cycle and requires only a standard ETH deposit transaction.

## Recommendation
Call `_updateRsETHPrice()` (or its public wrapper) atomically at the start of `_beforeDeposit()` in `LRTDepositPool`, before `getRsETHAmountToMint()` is invoked. This requires removing the `view` modifier from `_beforeDeposit` and the calling deposit functions. Alternatively, compute the mint amount inline from live TVL (`totalETHInProtocol / rsethSupply`) rather than from the stored `rsETHPrice` state variable, so no separate update call is needed.

## Proof of Concept
1. At time T, `rsETHPrice = 1.05e18`. EigenLayer strategies accrue rewards; true price rises to `1.06e18`. Keeper has not yet called `updateRSETHPrice()`.
2. Attacker calls `LRTDepositPool.depositETH{value: 100 ether}(0, "")`.
3. `_beforeDeposit` → `getRsETHAmountToMint` executes: `rsethAmountToMint = (100e18 × 1e18) / 1.05e18 ≈ 95.238 rsETH`.
4. Correct mint at true price: `100e18 / 1.06e18 ≈ 94.340 rsETH`. Excess: `≈ 0.898 rsETH`.
5. Keeper calls `updateRSETHPrice()`. New price = `(totalETH + 100e18) / (rsethSupply + 95.238e18)` — lower than it would have been had the deposit used the correct price — permanently diluting all prior holders.

**Foundry fork test plan:** Fork mainnet, snapshot state where `totalETHInProtocol / rsethSupply > rsETHPrice`. Call `depositETH` as an unprivileged address. Assert `balanceOf(attacker) > depositAmount / truePriceAtSnapshot`. Then call `updateRSETHPrice()` and assert the new `rsETHPrice` is lower than `(totalETHInProtocol_before_deposit) / rsethSupply_before_deposit`, confirming dilution.
