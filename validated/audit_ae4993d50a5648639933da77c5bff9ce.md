Audit Report

## Title
Missing Staleness and Validity Checks on Chainlink `latestRoundData()` Return Values - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` silently discards all validation fields from `latestRoundData()`, accepting stale, zero, or incomplete-round prices without reversion. Because `LRTOracle.updateRSETHPrice()` is a public, permissionless function, any caller can lock in a stale-price-derived `rsETHPrice` at a moment of their choosing, causing new depositors to receive an incorrect number of rsETH tokens and diluting or shortchanging existing holders.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the LST/ETH price as:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

All four non-price return values (`roundId`, `startedAt`, `updatedAt`, `answeredInRound`) are discarded. No check is performed for `price <= 0`, `answeredInRound < roundId`, or `block.timestamp > updatedAt + threshold`. This is in direct contrast to `ChainlinkOracleForRSETHPoolCollateral.getRate()`, which is in the same repository and correctly validates all three conditions before using the price:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

The unvalidated price flows through `LRTOracle.getAssetPrice()` → `_getTotalEthInProtocol()` → `_updateRsETHPrice()`, which writes the computed value to the `rsETHPrice` storage variable. [3](#0-2) 

`updateRSETHPrice()` is public with only a `whenNotPaused` guard — no role check, no access control: [4](#0-3) 

The `pricePercentageLimit` circuit-breaker in `_updateRsETHPrice()` is gated on `pricePercentageLimit > 0`, meaning it is entirely inactive at the default value of zero: [5](#0-4) 

With `pricePercentageLimit == 0`, even a large stale-price deviation is accepted without revert or pause. The computed `rsETHPrice` is then used directly by `LRTDepositPool.depositAsset()` to determine how many rsETH tokens to mint per unit of deposited asset. [6](#0-5) 

`RSETHPriceFeed.latestRoundData()` has the same defect — it passes through all validation fields from `ETH_TO_USD.latestRoundData()` without checking them, compounding the exposure for any downstream consumer of that feed. [7](#0-6) 

## Impact Explanation

**High — Theft of unclaimed yield.**

When the Chainlink LST/ETH feed is stale and the true price has risen (e.g., stETH has accrued more ETH value since the last heartbeat), `_getTotalEthInProtocol()` understates TVL. `rsETHPrice` is set below its true value. New depositors calling `depositAsset()` immediately after receive more rsETH than the true asset value warrants, diluting the share of every existing rsETH holder. The yield that existing holders had accrued — reflected in the true (higher) TVL — is redistributed to new depositors. This is a concrete, quantifiable transfer of value from existing holders to new depositors, matching the "Theft of unclaimed yield" impact class.

The inverse scenario (inflated stale price) shortchanges new depositors rather than existing holders, matching "Contract fails to deliver promised returns."

## Likelihood Explanation

**Low.** Chainlink feeds miss heartbeats during network congestion, sequencer downtime on L2 deployments, or feed deprecation. The attacker requires no privileged access and does not need to cause the staleness — only to observe it on-chain and call the public `updateRSETHPrice()` at the right moment. The window per staleness event is narrow (minutes to hours), but the attack is repeatable across every future staleness event and requires no capital at risk.

## Recommendation

Apply the same validation pattern already present in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
uint256 private constant STALENESS_THRESHOLD = 3600; // tune per feed heartbeat

function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (price <= 0) revert InvalidPrice();
    if (answeredInRound < roundId) revert StalePrice();
    if (block.timestamp > updatedAt + STALENESS_THRESHOLD) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Additionally, add equivalent validation inside `RSETHPriceFeed.latestRoundData()` for the inner `ETH_TO_USD.latestRoundData()` call. Ensure `pricePercentageLimit` is set to a non-zero value at deployment to activate the existing circuit-breaker as a secondary defense.

## Proof of Concept

1. Deploy a fork of mainnet. Confirm `pricePercentageLimit == 0` (default).
2. Advance `block.timestamp` by 2 hours without triggering a Chainlink feed update (simulate a heartbeat miss by not calling the mock aggregator's `updateAnswer`).
3. Call `LRTOracle.updateRSETHPrice()` from an unprivileged EOA. Observe that it succeeds and writes a stale-price-derived value to `rsETHPrice`.
4. Have a second EOA call `LRTDepositPool.depositAsset(stETH, amount, 0, "")`. Record the rsETH minted.
5. Advance time again, trigger a correct feed update, and call `updateRSETHPrice()` again. Observe `rsETHPrice` corrects upward.
6. Compute the rsETH minted in step 4 versus what would have been minted at the correct price. The difference represents the dilution transferred from existing holders to the step-4 depositor.

A Foundry fork test using `vm.warp` and a mock `AggregatorV3Interface` that returns a fixed stale `updatedAt` timestamp can reproduce this deterministically without requiring a live staleness event.

        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)

                ++assetIdx;
