### Title
Missing `max_l2_gas_amount` Admission Check for Declare Transactions in `StatelessTransactionValidator` — (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

---

### Summary

The `validate_resource_bounds` function in `StatelessTransactionValidator` enforces a `max_l2_gas_amount` ceiling on Invoke and DeployAccount transactions but explicitly skips the identical check for Declare transactions. Any unprivileged user can submit a Declare transaction whose `l2_gas.max_amount` exceeds the configured gateway limit, bypassing the admission guard that protects the other two transaction types.

---

### Finding Description

In `crates/apollo_gateway/src/stateless_transaction_validator.rs`, the `validate_resource_bounds` function reads:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
``` [1](#0-0) 

The empty `if let RpcTransaction::Declare(_) = tx {}` arm is a deliberate no-op: the `max_l2_gas_amount` bound is never evaluated for Declare transactions. The companion test `valid_l2_gas_amount_on_declare` explicitly asserts that a Declare transaction with `max_amount: GasAmount(200)` passes validation even when `max_l2_gas_amount` is configured to `100`, confirming the gap is load-bearing in the current test suite: [2](#0-1) 

The `StatefulTransactionValidator::validate_resource_bounds` only checks the L2 gas *price* threshold, not the max *amount*, so there is no downstream guard that compensates for the missing stateless check: [3](#0-2)

### Impact Explanation

**Impact: High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

A Declare transaction with `l2_gas.max_amount` set to an arbitrarily large value (e.g., `GasAmount(u64::MAX)`) passes the gateway's stateless validation and is forwarded to the mempool. The gateway's `max_l2_gas_amount` policy — which exists precisely to prevent admission of transactions with unreasonable gas bounds — is fully bypassed for this transaction type. The blockifier's `check_fee_bounds` (called during execution, not at admission) would only reject the transaction if the sender cannot cover `max_amount × max_price_per_unit`, but a sender who sets `max_price_per_unit = 1` and holds a modest balance can still pass that check with a `max_amount` far above the configured gateway ceiling.

---

### Likelihood Explanation

**Likelihood: Medium.**

The trigger requires no privilege: any user submitting a Declare transaction can set `l2_gas.max_amount` to any value. The gap is acknowledged by a TODO comment and is confirmed by an existing test that asserts the out-of-limit value is accepted. The only friction is that Declare transactions are less frequent than Invoke transactions, but the attack surface is fully open.

---

### Recommendation

Remove the special-casing of Declare transactions in `validate_resource_bounds`. Apply the same `max_l2_gas_amount` ceiling uniformly across all transaction types:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

Update the test `valid_l2_gas_amount_on_declare` to expect a rejection when `max_amount` exceeds the configured limit, and add a positive-flow test for Declare transactions whose `max_amount` is within the limit.

---

### Proof of Concept

1. Configure the gateway with `max_l2_gas_amount = 100`.
2. Construct a Declare V3 transaction with `resource_bounds.l2_gas.max_amount = GasAmount(200)` and any non-zero `max_price_per_unit`.
3. Call `StatelessTransactionValidator::validate(&tx)`.
4. Observe `Ok(())` — the transaction is admitted despite exceeding the configured ceiling.
5. The same transaction submitted as an Invoke type returns `Err(MaxGasAmountTooHigh { gas_amount: GasAmount(200), max_gas_amount: 100 })`.

The existing test at lines 173–201 of `stateless_transaction_validator_test.rs` already encodes step 4 as an expected-pass assertion, confirming the invariant is broken for Declare transactions. [2](#0-1)
