Audit Report

## Title
LP shares permanently burned with zero token return in `removeLiquidity` due to unchecked floor rounding — (`metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary
`LiquidityLib.removeLiquidity` applies floor division at two successive levels when converting LP shares to native token amounts. When either rounding stage produces zero, all share-burning and accounting state mutations still execute, but no tokens are transferred to the LP. The function returns silently with the LP holding fewer shares and receiving nothing, while the untracked scaled surplus is later swept as protocol/admin spread fees via `collectFees`.

## Finding Description
**Root cause — two independent floor-rounding paths:**

**Level 1 (share-to-scaled):** At lines 205–206, the LP's proportional scaled claim is computed with integer floor division:
```solidity
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
uint256 amount1Scaled = _checkedMul(binState.token1BalanceScaled, sharesToRemove) / binTotalSharesVal;
```
When `binState.token0BalanceScaled * sharesToRemove < binTotalSharesVal`, `amount0Scaled` truncates to 0.

**Level 2 (scaled-to-native):** At lines 239–240 and 274–276, `_deltasScaledToExternal` is called with `Math.Rounding.Floor`:
```solidity
(amount0Removed, amount1Removed) =
  _deltasScaledToExternal(totalToken0ToRemoveScaled, totalToken1ToRemoveScaled, ctx, Math.Rounding.Floor);
// ...
deltaAmount0 = scaledDeltaAmount0 / ctx.token0ScaleMultiplier;
```
For USDC (6 decimals, `token0ScaleMultiplier = 10^12`), any `totalToken0ToRemoveScaled < 10^12` produces `amount0Removed = 0`.

**State mutations are unconditional:** Lines 210–214 always execute regardless of whether computed amounts are zero:
```solidity
binState.token0BalanceScaled -= uint104(amount0Scaled);
binState.token1BalanceScaled -= uint104(amount1Scaled);
binTotalShares[binIdx] = binTotalSharesVal - sharesToRemove;
positionBinShares[posKey] = newUserShares;
```

**Transfer is silently skipped:** Lines 242–247 only transfer when amounts are positive:
```solidity
if (amount0Removed > 0) { IERC20(ctx.token0).safeTransfer(owner, amount0Removed); }
if (amount1Removed > 0) { IERC20(ctx.token1).safeTransfer(owner, amount1Removed); }
```

**Existing guard is insufficient:** The `MinimalLiquidity` check at lines 200–202 only guards the *remaining* shares after removal, not whether the *removed* shares yield any tokens.

**Accounting invariant break (scaled-to-native path):** `binTotals.scaledToken0` is decremented at line 232 while the actual ERC-20 balance is unchanged. The resulting surplus is computed at lines 385–388 in `collectFees`:
```solidity
uint256 surplus0Scaled =
  balance0() * TOKEN_0_SCALE_MULTIPLIER - uint256(binTotals.scaledToken0) - notionalFee0AmountScaled;
```
This surplus is distributed to admin and protocol fee receivers — the LP's burned value is transferred to fee collectors.

**Asymmetry with `addLiquidity`:** `addLiquidity` uses `Math.Rounding.Ceil` at lines 109 and 142, ensuring the pool always receives at least the full scaled amount. `removeLiquidity` uses `Math.Rounding.Floor` with no compensating guard, creating a one-sided rounding regime that systematically disadvantages LPs.

## Impact Explanation
An LP burns non-zero shares and receives zero tokens. In the share-to-scaled path, the burned value is silently redistributed to remaining LPs. In the scaled-to-native path, the burned value becomes untracked surplus collected by the fee receiver — a direct, unrecoverable transfer of LP principal to the fee collector with no protocol-level revert. The maximum loss per removal in the scaled-to-native path is `token0ScaleMultiplier - 1` scaled units (just under 1 native unit of token0, e.g., just under 1 USDC for a 6-decimal token). This breaks the core solvency invariant that pool balances must always cover all LP claims, and constitutes a direct loss of LP principal with broken core pool withdraw functionality.

## Likelihood Explanation
The condition is reachable by any unprivileged LP in any USDC/USDT pool (6-decimal tokens, `token0ScaleMultiplier = 10^12`). Any removal whose proportional scaled claim is below `10^12` (i.e., worth less than 1 native unit) silently yields zero. This is reachable with small positions, high total-share bins, or any LP whose proportional claim falls below the rounding threshold. No external attacker is required; the LP is the transaction initiator. The condition is repeatable across any number of removal transactions.

## Recommendation
Add a guard after computing `amount0Removed` and `amount1Removed` that reverts when shares are non-zero but both native amounts are zero:
```solidity
if (sharesToRemove > 0 && amount0Removed == 0 && amount1Removed == 0) {
    revert ZeroTokensForShares();
}
```
Alternatively, enforce a minimum per-bin removal that guarantees at least 1 native unit of at least one token is returned, analogous to the `MinimalLiquidity` guard on the add path. The asymmetry between `addLiquidity` (Ceil) and `removeLiquidity` (Floor) should also be documented or mitigated.

## Proof of Concept
**Setup:** USDC (6 decimals) / WETH (18 decimals) pool. `token0ScaleMultiplier = 10^12`. `MINIMAL_MINTABLE_LIQUIDITY = 1000`.

1. LP adds `1000` shares to bin `+1` (above active bin, token0-only). Initial scaled balance: `token0BalanceScaled = 1`. `binTotalShares[1] = 1000`.
2. LP calls `removeLiquidity` with `sharesToRemove = 1000` (full exit, `newUserShares = 0`, passes `MinimalLiquidity` check at line 200).
3. Level 1: `amount0Scaled = (1 * 1000) / 1000 = 1`. Non-zero, passes Level 1.
4. Level 2: `amount0Removed = 1 / 10^12 = 0`. Rounds to zero.
5. Lines 210–214 execute: `binState.token0BalanceScaled -= 1`, `binTotals.scaledToken0 -= 1`, `positionBinShares[posKey] = 0`.
6. Lines 242–243: `amount0Removed == 0`, no `safeTransfer` executes.
7. LP receives 0 USDC. The 1 scaled unit is now untracked surplus, collectible as fees via `collectFees` at lines 385–388.
