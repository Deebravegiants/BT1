Audit Report

## Title
Contract Without `receive()` Permanently Freezes ETH After rsETH Burn — (`contracts/LRTWithdrawalManager.sol`)

## Summary
`LRTWithdrawalManager` accepts ETH withdrawal requests from any caller, including contracts without a `receive()` function. Once the operator calls `unlockQueue`, the depositor's rsETH is burned and ETH is redeemed from the unstaking vault. Any subsequent attempt to complete the withdrawal via `completeWithdrawal` or `completeWithdrawalForUser` permanently reverts because `_transferAsset` uses a raw `call{value}` that fails for contracts without `receive()`. The ETH is permanently frozen inside the manager with no recovery path.

## Finding Description

**Step 1 — `initiateWithdrawal` accepts any caller:**
`initiateWithdrawal` pulls rsETH via `safeTransferFrom` and records the request keyed to `msg.sender` with no check that the caller can receive ETH. [1](#0-0) 

The nonce is stored against `msg.sender` in `userAssociatedNonces`: [2](#0-1) 

**Operator step — `unlockQueue` burns rsETH irrevocably:**
`unlockQueue` burns all rsETH for unlocked requests in bulk and redeems ETH from the unstaking vault. There is no mechanism to skip a single user's request. [3](#0-2) 

**Step 2 — `_processWithdrawalCompletion` hard-codes the recipient as `user`:**
The ETH is always sent to the original `user` address with no alternative recipient parameter. [4](#0-3) 

**`_transferAsset` reverts for contracts without `receive()`:** [5](#0-4) 

Because the entire transaction reverts, `unlockedWithdrawalsCount[asset]--` at line 717 is also rolled back, keeping the counter permanently above zero. [6](#0-5) 

**`completeWithdrawalForUser` provides no relief:**
The operator-assisted path still routes ETH to the original `user` address. The NatDoc comment on line 191 even acknowledges this function is "not expected to be used for ETH." [7](#0-6) 

**`sweepRemainingAssets` is permanently blocked:**
`sweepRemainingAssets` requires `unlockedWithdrawalsCount[asset] == 0`. The stuck, unlocked-but-uncompletable request keeps this counter non-zero indefinitely. [8](#0-7) 

## Impact Explanation

A contract depositor without `receive()` that initiates an ETH withdrawal will have its rsETH permanently burned and the corresponding ETH permanently frozen inside `LRTWithdrawalManager` with no recovery path for either the user or the protocol. This is a concrete **Critical — Permanent freezing of funds** impact.

## Likelihood Explanation

DeFi vaults, aggregators, and multisig contracts routinely hold liquid restaking tokens and interact with withdrawal managers. Any such contract that omits a `receive()` function and calls `initiateWithdrawal(ETH_TOKEN, ...)` triggers this path. No privileged role is required — only a standard unprivileged `initiateWithdrawal` call. The scenario is realistic and repeatable.

## Recommendation

Add a `receiver` parameter to `completeWithdrawal` (and optionally `completeWithdrawalForUser`) so that a contract caller can designate an ETH-capable address to receive the funds:

```solidity
function completeWithdrawal(
    address asset,
    address receiver,
    string calldata referralId
) external nonReentrant whenNotPaused {
    _processWithdrawalCompletion(asset, msg.sender, receiver, referralId);
}
```

`_processWithdrawalCompletion` would send assets to `receiver` rather than hard-coding `user`, eliminating the permanent freeze while preserving the authorization check that only the original depositor (or operator) can trigger completion.

## Proof of Concept

1. Deploy `VaultNoReceive` — a contract holding rsETH with no `receive()` function.
2. `VaultNoReceive` approves `LRTWithdrawalManager` and calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. rsETH is transferred to the manager. ✅
3. Operator calls `unlockQueue(ETH_TOKEN, ...)`. rsETH is burned; ETH is redeemed from the unstaking vault. ✅
4. `VaultNoReceive` calls `completeWithdrawal(ETH_TOKEN, "")`. `_transferAsset` executes `payable(VaultNoReceive).call{value: amount}("")` → returns `false` → reverts with `EthTransferFailed`. ❌
5. Operator calls `completeWithdrawalForUser(ETH_TOKEN, address(VaultNoReceive), "")`. Same revert. ❌
6. `unlockedWithdrawalsCount[ETH_TOKEN]` remains `> 0`; `sweepRemainingAssets(ETH_TOKEN)` reverts with `PendingWithdrawalsExist`. ETH is permanently frozen. ❌
