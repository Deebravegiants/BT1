### Title
`accept_requests` flag incorrectly blocks `respond`/`respond_ckd`/`respond_verify_foreign_tx` from resolving already-queued pending requests — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `accept_requests` boolean, designed to gate **new** user requests when TEE validation degrades, is also enforced inside the node-side `respond`, `respond_ckd`, and `respond_verify_foreign_tx` methods. When `accept_requests` is set to `false` by `verify_tee`, nodes are prevented from submitting responses to **already-queued** pending requests, causing those requests to exhaust the yield-resume timeout window and fail — breaking the request-lifecycle invariant that an accepted request will eventually be fulfilled or explicitly rejected.

---

### Finding Description

**Root cause — `check_request_preconditions` (new-request gate):** [1](#0-0) 

The flag is checked here to block *new* submissions. This is the intended use.

**Root cause — `respond` (existing-request resolution, also gated):** [2](#0-1) 

**Root cause — `respond_ckd` (same pattern):** [3](#0-2) 

**Root cause — `respond_verify_foreign_tx` (same pattern):** [4](#0-3) 

All three node-side resolution methods return `TeeError::TeeValidationFailed` when `accept_requests == false`, even for requests that were already accepted and are sitting in the pending maps.

**How `accept_requests` is set to `false`:**

`verify_tee` sets the flag when kicking out expired-attestation participants would leave fewer than `threshold` valid participants: [5](#0-4) 

The design comment says *"We will not accept new signature requests as a safety precaution"*, but the implementation also blocks resolution of **existing** requests.

**The yield-resume timeout consequence:**

Each `sign`/`request_app_private_key`/`verify_foreign_transaction` call creates a yield via `promise_yield_create`. If no `promise_yield_resume` lands within ~200 blocks, the runtime fires the yield-callback with `Err(PromiseError::Failed)`. Because `respond` is blocked by `accept_requests = false`, no resume ever lands, and every in-flight request times out and fails. [6](#0-5) 

---

### Impact Explanation

**Medium.** This breaks the request-lifecycle safety invariant: a request that was accepted (deposit taken, yield registered, request stored in `pending_signature_requests` / `pending_ckd_requests` / `pending_verify_foreign_tx_requests`) can no longer be fulfilled by nodes. All in-flight requests silently time out. For `verify_foreign_transaction` requests that are part of a bridge flow where the foreign-chain side has already executed, the timeout failure breaks the cross-chain execution flow and can cause invalid bridge state. Deposits (1 yoctonear) are refunded on timeout, so direct monetary loss per request is negligible, but the contract execution-flow invariant is violated and bridge liveness is broken without any network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

**Low.** The condition requires: (a) enough participant attestations to expire simultaneously that the surviving valid set falls below threshold, and (b) a participant to call `verify_tee` while that condition holds. Both are realistic operational events (attestation expiry is a scheduled 7-day window; `verify_tee` is called periodically by nodes). No malicious intent is required — the flag is set by normal protocol operation.

---

### Recommendation

Remove the `accept_requests` check from `respond`, `respond_ckd`, and `respond_verify_foreign_tx`. The flag's purpose is to prevent **new** requests from entering the queue when TEE security is degraded; it should not prevent nodes from resolving requests that were already accepted before the flag changed. The three lines at: [2](#0-1) [3](#0-2) [4](#0-3) 

should be removed. The `accept_requests` guard in `check_request_preconditions` is sufficient to block new submissions.

---

### Proof of Concept

1. User calls `sign(request)` with 1 yoctonear deposit. The contract is `Running`, `accept_requests = true`. A yield is created; the request is stored in `pending_signature_requests`.
2. Attestations for two of three participants expire. A participant calls `verify_tee`. Because kicking out two participants would drop the set below threshold, `verify_tee` sets `self.accept_requests = false` and returns `Ok(false)`.
3. MPC nodes complete the threshold signing computation and call `respond(request, response)`.
4. `respond` hits the guard at line 579–581: `if !self.accept_requests { return Err(TeeError::TeeValidationFailed.into()); }`. The response is rejected.
5. No `promise_yield_resume` is ever called for the pending yield. After ~200 blocks the NEAR runtime fires the yield-callback `return_signature_and_clean_state_on_success` with `Err(PromiseError::Failed)`.
6. The user's `sign` call resolves with a timeout error. The 1 yoctonear deposit is refunded, but the signature is never produced.
7. Any `verify_foreign_transaction` requests in flight during the same window fail identically, breaking any bridge flow that depended on them.
