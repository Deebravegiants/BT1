### Title
`respond_verify_foreign_tx` Does Not Bind `response.payload_hash` to `request` Content — (`crates/contract/src/lib.rs`)

### Summary

The `respond_verify_foreign_tx` contract method verifies that the MPC signature is valid over the caller-supplied `response.payload_hash`, but never checks that `response.payload_hash` is actually the canonical hash of the `request.request` content. A single Byzantine MPC leader node (below the signing threshold) can misroute a legitimately-produced MPC signature — obtained for one pending foreign-chain verification request — as the response to a different pending request, causing the contract to resolve that second request with a `payload_hash` that does not correspond to the user's actual transaction.

### Finding Description

In `respond_verify_foreign_tx` (`crates/contract/src/lib.rs`, lines 692–754), the contract performs two checks:

1. The caller is an authenticated, attested participant.
2. `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's root public key. [1](#0-0) 

What is **never** checked is whether `response.payload_hash` equals `ForeignTxSignPayload::V1 { request: request.request, values: ... }.compute_msg_hash()`. The `payload_hash` is entirely caller-supplied and is only verified to have a valid signature over it — not that it commits to the specific `ForeignChainRpcRequest` in the `request` argument.

The signed payload type `ForeignTxSignPayloadV1` contains only the `ForeignChainRpcRequest` and extracted values: [2](#0-1) 

It does **not** include `domain_id` or `payload_version` from the `VerifyForeignTransactionRequest`. This means a valid MPC signature produced for request A (with `payload_hash_A`) can be submitted as the response for request B (with a different `ForeignChainRpcRequest`) within the same domain, and the contract's signature check will pass because the signature is valid over `payload_hash_A` under the same domain key.

The contract then resolves request B's pending yield with `{payload_hash_A, sig_A}`: [3](#0-2) 

The SDK's `ForeignChainSignatureVerifier::verify_signature` does perform this binding check on the client side: [4](#0-3) 

But the contract itself enforces no such invariant, leaving the binding entirely to the caller's discretion.

### Impact Explanation

A bridge contract that calls `verify_foreign_transaction` to gate inbound bridge operations (e.g., minting tokens after a foreign-chain deposit) and does not independently recompute and compare the expected `payload_hash` will accept the misrouted response as a valid MPC attestation for the wrong transaction. This enables a Byzantine leader to cause the bridge to process a fraudulent inbound transfer — a forged foreign-chain verification leading to invalid bridge execution or double-spend conditions.

**Impact: High** — forged foreign-chain verification that causes invalid bridge execution.

### Likelihood Explanation

The attack requires exactly one Byzantine MPC participant to act as the signing leader for the affected round. Leader selection is deterministic and rotates among participants, so an attacker controlling any single participant node can wait until they are elected leader. No threshold collusion is required: the MPC network collectively produces the signature for request A honestly; the Byzantine leader then submits it against request B unilaterally via a single on-chain call.

**Likelihood: Medium** — requires one Byzantine participant node, no threshold collusion.

### Recommendation

In `respond_verify_foreign_tx`, after verifying the signature, recompute the expected payload hash from the `request` content and assert it equals `response.payload_hash`:

```rust
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    // values are not available on-chain; see note below
}).compute_msg_hash()?;
assert_eq!(expected_hash, response.payload_hash, "payload_hash does not match request");
```

Because the `values` (extracted chain data) are not stored on-chain, the contract cannot fully recompute the hash. The recommended long-term fix is to include the `domain_id` and a commitment to the originating `VerifyForeignTransactionRequest` (e.g., its Borsh hash) inside `ForeignTxSignPayloadV1`, so the signed payload cryptographically binds to the specific request. This makes cross-request misrouting detectable via the signature check alone, without requiring the contract to know the extracted values.

### Proof of Concept

1. User A submits `verify_foreign_transaction` for Bitcoin `tx_id=0xAA` (a real, confirmed transaction). Request A is queued.
2. User B submits `verify_foreign_transaction` for Bitcoin `tx_id=0xBB` (a non-existent or unconfirmed transaction). Request B is queued.
3. The MPC network honestly processes request A, producing `sig_A` over `hash(ForeignTxSignPayload::V1 { request: Bitcoin(tx_id=0xAA), values: [BlockHash=0x...] })` = `payload_hash_A`.
4. The Byzantine leader calls `respond_verify_foreign_tx(request=B, response={payload_hash=payload_hash_A, signature=sig_A})`.
5. The contract verifies: `sig_A` is valid over `payload_hash_A` under the domain key → **passes**.
6. The contract resolves request B's pending yield with `{payload_hash_A, sig_A}`.
7. User B's bridge contract receives a `VerifyForeignTransactionResponse` containing a valid MPC signature. If it does not recompute the expected hash for `tx_id=0xBB` and compare, it accepts this as proof that `tx_id=0xBB` was verified, and mints tokens for a deposit that never occurred.
