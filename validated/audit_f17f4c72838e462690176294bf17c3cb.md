### Title
Permanent `stakedButUnverifiedNativeETH` Inflation via Pre-Verification Validator Exit Causes Double-Counted ETH in `getEffectivePodShares()` — (`contracts/NodeDelegator.sol`)

---

### Summary

`stakedButUnverifiedNativeETH` is incremented in `stake32Eth` and decremented **only** in `verifyWithdrawalCredentials`. EigenLayer's `IEigenPod.verifyWithdrawalCredentials` explicitly reverts (`ValidatorIsExitingBeaconChain`) if the validator already has an exit epoch set. If a validator exits the beacon chain before its credentials are verified, the 32 ETH returned to the EigenPod is captured by a subsequent checkpoint and awarded as `withdrawableShare`, while `stakedButUnverifiedNativeETH` can never be decremented. `getEffectivePodShares()` then returns `stakedButUnverifiedNativeETH + withdrawableShare = 64 ETH` for a single 32 ETH validator, permanently inflating the rsETH price and causing protocol insolvency.

---

### Finding Description

**Step 1 — Stake increments the tracker:**

`stake32Eth` unconditionally adds 32 ETH to `stakedButUnverifiedNativeETH`: [1](#0-0) 

**Step 2 — The only decrement path is permanently blocked:**

`verifyWithdrawalCredentials` is the sole place `stakedButUnverifiedNativeETH` is decremented: [2](#0-1) 

However, EigenLayer's EigenPod enforces that validators proven via `verifyWithdrawalCredentials` **must not have an exit epoch set already** (documented in the interface and enforced with `ValidatorIsExitingBeaconChain`): [3](#0-2) [4](#0-3) 

Once a validator exits, calling `NodeDelegator.verifyWithdrawalCredentials` will revert at the EigenPod level, so `stakedButUnverifiedNativeETH` is **permanently stuck** at 32 ETH for that validator.

**Step 3 — Checkpoint awards shares for the returned ETH:**

When the validator exits, 32 ETH is returned to the EigenPod address as execution-layer ETH (emitting `NonBeaconChainETHReceived`). Because the validator was never verified, it is `INACTIVE` in EigenPod — `activeValidatorCount = 0`. When `startCheckpoint` is called, `podBalanceGwei` captures the 32 ETH sitting in the pod, `proofsRemaining = 0` (no active validators to prove), so the checkpoint finalizes immediately and awards 32 ETH in shares. `withdrawableShare` becomes 32 ETH. [5](#0-4) 

**Step 4 — `getEffectivePodShares()` double-counts:**

```
getEffectivePodShares() = stakedButUnverifiedNativeETH + withdrawableShare
                        = 32 ETH + 32 ETH = 64 ETH   ← only 32 ETH exists
``` [6](#0-5) 

**Step 5 — Inflated value propagates to rsETH price:**

`getETHDistributionData` calls `getEffectivePodShares()` for each NDC: [7](#0-6) 

`getTotalAssetDeposits` aggregates this: [8](#0-7) 

`_getTotalEthInProtocol` in `LRTOracle` uses it to compute `rsETHPrice`: [9](#0-8) [10](#0-9)

### Impact Explanation

The rsETH price is permanently inflated by 32 ETH per affected validator. The protocol reports more ETH backing than actually exists. When rsETH holders redeem at the inflated price, the protocol cannot cover redemptions — this is **protocol insolvency**. The inflation is irreversible: there is no code path that can decrement `stakedButUnverifiedNativeETH` for an exited-but-unverified validator.

---

### Likelihood Explanation

The trigger condition — a validator exiting before `verifyWithdrawalCredentials` is called — is realistic:

- **Slashing**: A validator slashed on the beacon chain is forced to exit. The slashing can occur in the window between `stake32Eth` and the operator calling `verifyWithdrawalCredentials`.
- **Voluntary early exit**: The operator may voluntarily exit a validator before completing the EigenLayer credential verification flow.
- **Beacon chain forced exits**: EIP-7002 and related mechanisms can force exits.

The window between staking and credential verification is non-zero and operationally unavoidable.

---

### Recommendation

Add a mechanism to decrement `stakedButUnverifiedNativeETH` for validators that exited without credential verification. One approach: introduce a `recoverUnverifiedStake(bytes calldata pubkey)` function that, given proof that the validator is `WITHDRAWN` on the beacon chain and `INACTIVE` in EigenPod (i.e., never verified), decrements `stakedButUnverifiedNativeETH` by 32 ETH. This function should verify the validator's EigenPod status is `INACTIVE` and that the pod has received the corresponding ETH (via checkpoint shares), preventing double-counting.

Alternatively, during `startCheckpoint`, subtract `stakedButUnverifiedNativeETH` from the pod balance before awarding shares, so that ETH from unverified exits is not double-counted.
