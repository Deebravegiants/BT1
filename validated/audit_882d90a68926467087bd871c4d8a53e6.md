Audit Report

## Title
Permissionless `updateRate()` with no monotonicity guard in `CrossChainRateReceiver` allows attacker to push a temporarily depressed rsETH price to L2 and extract excess wrsETH — (`contracts/cross-chain/RSETHMultiChainRateProvider.sol` / `contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`MultiChainRateProvider.updateRate()` is callable by any address with no role restriction. When a slashing event reduces TVL by an amount within `pricePercentageLimit`, `LRTOracle` writes the lower `rsETHPrice` without pausing. An attacker can immediately call `updateRate()` to broadcast this depressed rate to all L2 `CrossChainRateReceiver` contracts, then deposit ETH on L2 at the artificially low rate to receive more `wrsETH` than fair value. When the price recovers, the attacker's excess `wrsETH` dilutes the yield owed to pre-existing holders.

## Finding Description

**Root cause — three missing guards compose into an exploit:**

**1. Permissionless `updateRate()`.**
`MultiChainRateProvider.updateRate()` carries only a `nonReentrant` modifier; any EOA or contract can call it at any time. [1](#0-0) 

It reads the currently stored `rsETHPrice` from `LRTOracle` via `RSETHMultiChainRateProvider.getLatestRate()` and broadcasts it to every registered receiver. [2](#0-1) 

**2. `LRTOracle` writes a lower price without pausing for within-limit drops.**
Inside `_updateRsETHPrice()`, when `newRsETHPrice < highestRsethPrice` but `diff <= pricePercentageLimit.mulWad(highestRsethPrice)`, the code skips the pause branch, emits events, and falls through to `rsETHPrice = newRsETHPrice`. [3](#0-2) [4](#0-3) 

**3. `CrossChainRateReceiver.lzReceive()` stores any incoming rate unconditionally.**
There is no check that `_rate >= rate` (the previously stored value). Any rate pushed by the authorized provider is accepted and written directly. [5](#0-4) 

**Exploit flow:**

- **Step 1:** A validator slashing event reduces `totalETHInProtocol`. `updateRSETHPrice()` (also permissionless) is called; `rsETHPrice` drops from, e.g., 1.05e18 to 1.03e18 (1.9% drop, within a 5% `pricePercentageLimit`). No pause is triggered.
- **Step 2:** Attacker calls `rateProvider.updateRate{value: lzFee}()`. The depressed 1.03e18 rate is sent to every L2 `CrossChainRateReceiver` via LayerZero.
- **Step 3:** `lzReceive()` writes `rate = 1.03e18` with no monotonicity check.
- **Step 4:** Attacker calls `RSETHPoolV2.deposit()` on L2. `viewSwapRsETHAmountAndFee()` computes `rsETHAmount = amountAfterFee * 1e18 / 1.03e18`, minting ~1.9% more `wrsETH` per ETH than the fair 1.05e18 rate would yield. The `dailyMintLimit` caps magnitude but does not prevent the exploit. [6](#0-5) 
- **Step 5:** When TVL recovers (insurance payout, remaining stake), `rsETHPrice` returns to ≥1.05e18. The attacker's excess `wrsETH` now represents a larger share of the recovered TVL than the deposited ETH warranted, extracting yield from pre-existing holders.

**Why existing guards are insufficient:**
- `pricePercentageLimit` only pauses for *large* drops; small-but-real slashing events are explicitly allowed to pass through.
- `dailyMintLimit` on L2 limits per-day magnitude but does not prevent the directional exploit.
- `nonReentrant` on `updateRate()` prevents re-entrancy but not permissionless invocation.
- No access control on `updateRate()` and no monotonicity guard on `lzReceive()`.

## Impact Explanation

**High — Theft of unclaimed yield.**

Pre-existing `wrsETH` holders are entitled to the full per-token recovery of TVL after a minor slashing event. By minting excess `wrsETH` at the depressed rate, the attacker captures a portion of that recovery. The per-token value that existing holders receive upon the price returning to its prior level is permanently reduced by the dilution. The magnitude is bounded by `dailyMintLimit` and the size of the price dip, but is non-zero and directly extractable on every qualifying slashing event.

## Likelihood Explanation

- EigenLayer validator slashing is a known, realistic occurrence for a restaking protocol.
- Both `updateRSETHPrice()` and `updateRate()` are permissionless — no privileged access is required.
- The attacker only needs to monitor on-chain oracle state and act within the price-recovery window; no front-running of admin transactions is required.
- The attack is repeatable on every qualifying slashing event and across every registered L2 chain.
- The only cost is the LayerZero messaging fee paid to `updateRate()`.

## Recommendation

1. **Add a monotonicity guard in `CrossChainRateReceiver.lzReceive()`** — reject any incoming rate lower than the currently stored value:
   ```solidity
   require(_rate >= rate, "Rate cannot decrease");
   ``` [5](#0-4) 

2. **Add access control to `MultiChainRateProvider.updateRate()`** — restrict callers to a trusted keeper/manager role so that rate pushes to L2 are gated:
   ```solidity
   function updateRate() external payable onlyRole(RATE_UPDATER_ROLE) nonReentrant { ... }
   ``` [7](#0-6) 

3. **Propagate the L1 pause state to L2** — when `LRTOracle` detects any price decrease (even within the limit), pause L2 pool deposits until the price recovers above the previously stored rate.
