### Title
Missing CKD Output Verification for `AppPublicKey` Variant Allows Single Byzantine Participant to Forge Key Derivation Responses - (File: `crates/contract/src/lib.rs`)

### Summary
`respond_ckd` applies a full pairing-based cryptographic check to `AppPublicKeyPV` responses but performs **no check whatsoever** on `AppPublicKey` responses. A single Byzantine attested participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary forged `CKDResponse`, which the contract accepts unconditionally and delivers to the waiting user, permanently resolving and removing the request from the pending queue.

### Finding Description
In `respond_ckd` (`crates/contract/src/lib.rs`, lines 675–682), the response verification is asymmetric across the two `CKDAppPublicKey` variants:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For `AppPublicKeyPV`, `ckd_output_check` (`crates/contract/src/primitives/ckd.rs`, lines 80–101) enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, mpc_pk)`, which cryptographically binds the response to the MPC network's threshold key and the user's app key. For `AppPublicKey`, the arm is an empty `{}` — no binding, no validity check.

The `AppPublicKey` variant is the primary CKD variant (used in the e2e test `ckd_response__passes_cryptographic_verification`). The contract's own unit test at line 3424 demonstrates the gap explicitly: it passes `CKDResponse { big_y: [1u8; 48], big_c: [2u8; 48] }` (not valid G1 points, not a threshold-computed output) and expects `respond_ckd` to succeed.

After the empty arm, `resolve_yields_for` (`crates/contract/src/pending_requests.rs`, lines 74–87) drains the entire fan-out queue for the request key and resumes every waiting yield with the forged bytes. The request is then removed from `pending_ckd_requests`, so no legitimate response can ever be submitted for it.

The only gate before the match is `assert_caller_is_attested_participant_and_protocol_active()` (line 666), which requires the caller to be an attested participant — a single node, not a threshold quorum.

### Impact Explanation
A single Byzantine attested participant (below the signing threshold) can:
1. Monitor the chain for any pending `AppPublicKey` CKD request.
2. Call `respond_ckd` with an arbitrary `CKDResponse` (e.g., random G1 points).
3. The contract accepts it without verification and resolves all queued yields with the forged data.
4. The user receives fabricated `(big_y, big_c)` values; decrypting with their private key yields garbage — no valid confidential key is derived.
5. The request is permanently removed from the pending map; the user's deposit is consumed and no retry is possible.

This breaks the threshold-signature guarantee for the CKD flow: the threshold is meant to ensure that no single participant can unilaterally determine the output, but for `AppPublicKey` requests the contract enforces no such constraint at response time.

### Likelihood Explanation
The `AppPublicKey` variant is the standard (non-publicly-verifiable) CKD path and is the variant exercised in the primary e2e verification test. Any single attested participant — a role that every active MPC node holds — can exploit this without any collusion. The attack requires only monitoring the NEAR chain for pending CKD requests and submitting a `respond_ckd` call before the legitimate leader does.

### Recommendation
The contract cannot replicate the `AppPublicKeyPV` pairing check for `AppPublicKey` because it lacks the user's private scalar. Two mitigations are possible:

1. **Require `AppPublicKeyPV` for all on-chain CKD requests.** Since `AppPublicKeyPV` is publicly verifiable and the contract already enforces it correctly, deprecate `AppPublicKey` as an on-chain request variant. Users who need confidentiality can still use `AppPublicKeyPV` with an ephemeral key pair.
2. **Add a threshold-vote mechanism for CKD responses**, analogous to how `vote_pk` requires a quorum before a public key is accepted, so that a single participant cannot unilaterally resolve a CKD request.

### Proof of Concept

1. User calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(pk1)`.
2. Byzantine attested participant calls:
   ```rust
   contract.respond_ckd(
       ckd_request,
       CKDResponse {
           big_y: dtos::Bls12381G1PublicKey([0xAB; 48]), // arbitrary bytes
           big_c: dtos::Bls12381G1PublicKey([0xCD; 48]), // arbitrary bytes
       },
   )
   ```
3. The `AppPublicKey(_) => {}` arm executes — no check runs.
4. `resolve_yields_for` resumes all queued yields with the forged serialized response.
5. The user's `request_app_private_key` transaction resolves with `{ big_y: 0xAB…, big_c: 0xCD… }`.
6. The request is removed from `pending_ckd_requests`; no legitimate response can be submitted.
7. The user's deposit is consumed and the derived key is unusable.

The contract's own test (`respond_ckd__should_succeed_when_response_is_valid_and_request_exists`, line 3403) already demonstrates this path succeeds with `[1u8; 48]` / `[2u8; 48]` garbage values for the `AppPublicKey` variant. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
