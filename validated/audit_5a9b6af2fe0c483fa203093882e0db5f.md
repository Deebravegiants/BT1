### Title
Permanently Locked Native ETH When EVM Recipient Cannot Receive ETH in `finTransfer` - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
`OmniBridge.finTransfer` delivers native ETH to the recipient via a bare `.call{value: amount}("")`. If the recipient address is a contract without a `receive()` or `payable fallback`, the call returns `success = false` and the function reverts with `FailedToSendEther`. Because there is no fallback redemption path (no `dropMessage`-equivalent, no admin rescue function, no alternative delivery route), the ETH that was locked in the bridge during the originating `initTransfer` becomes permanently unclaimable.

### Finding Description

In `OmniBridge.finTransfer`, after verifying the MPC signature, the contract dispatches native ETH as follows:

```solidity
if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();
}
``` [1](#0-0) 

The `payload.recipient` and `payload.amount` are cryptographically bound by the MPC signature verified at line 311. [2](#0-1) 

Because the entire transaction reverts on failure, `completedTransfers[payload.destinationNonce]` (set at line 287) is also reverted, so the nonce is not consumed. [3](#0-2) 

However, the ETH that was deposited into the bridge during the originating `initTransfer` call (where `msg.value` included `amount + nativeFee`) remains in the bridge's balance with no per-transfer accounting to recover it: [4](#0-3) 

The bridge exposes no function to redirect or refund a stuck native-ETH transfer to an alternative address. The only ETH egress for a specific transfer is the `finTransfer` path itself, which is permanently blocked when the recipient is non-payable. The contract's `receive()` function means the bridge silently holds the ETH indefinitely. [5](#0-4) 

The `message` field that enables callback-style delivery exists only for `isBridgeToken` paths; the native-ETH branch has no equivalent hook. [6](#0-5) 

### Impact Explanation

**Critical — Irreversible fund lock.**

Any native ETH bridged from NEAR to EVM where the EVM recipient is a contract without ETH-receive capability is permanently stranded in the bridge. The MPC signature fixes the recipient; no relayer retry, no user action, and no unprivileged call can redirect the funds. Recovery requires a privileged contract upgrade, which is outside the user's control.

### Likelihood Explanation

**Medium.** The scenario arises naturally with:
- Smart-contract wallets or multisigs that do not implement `receive()` (e.g., early Gnosis Safe versions, account-abstraction wallets)
- Contracts that conditionally reject ETH (e.g., based on internal state at delivery time)
- Contracts that were payable when the transfer was initiated on NEAR but were upgraded to non-payable before `finTransfer` executes on EVM

The NEAR-side `initTransfer` accepts any string as the EVM recipient, and the bridge performs no pre-flight check that the target can receive ETH. Any user bridging native ETH to a contract address is exposed.

### Recommendation

1. **Wrap-and-pull pattern**: Instead of pushing ETH directly to the recipient, credit the amount to a per-recipient claimable balance mapping. The recipient (or anyone on their behalf) can then call a `claimNativeETH(address recipient)` function that attempts the transfer, allowing retries without locking funds.
2. **Fallback to WETH**: If the bare `.call` fails, automatically wrap the ETH into WETH and transfer the WETH token to the recipient, ensuring delivery always succeeds.
3. **Pre-flight check**: Reject `initTransfer` on NEAR (or at the MPC signing layer) if the EVM recipient is a known non-payable contract, though this is difficult to enforce cross-chain.

### Proof of Concept

1. NEAR user calls `initTransfer` on NEAR specifying an EVM multisig contract (no `receive()`) as recipient for 1 ETH.
2. MPC signs `TransferMessagePayload` with `tokenAddress = address(0)`, `amount = 1e18`, `recipient = <multisig>`.
3. Relayer calls `finTransfer(sig, payload)` on EVM.
4. `payload.recipient.call{value: 1e18}("")` returns `(false, "")` because the multisig has no payable fallback.
5. `revert FailedToSendEther()` rolls back the transaction; `completedTransfers[nonce]` is reverted.
6. The 1 ETH remains in the bridge's balance. Every subsequent retry with the same signed payload produces the same revert. The ETH is permanently locked with no unprivileged recovery path.
