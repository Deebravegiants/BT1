### Title
`respond_verify_foreign_tx` Accepts Node-Supplied `payload_hash` Without Binding It to the Original Request, Enabling Cross-Request Signature Replay - (File: `crates/contract/src/lib.rs`)

### Summary

`MpcContract::respond_verify_foreign_tx` verifies the MPC signature against a `payload_hash` value that is **supplied by the responding node in the response itself**, not derived from the stored `VerifyForeignTransactionRequest`. This is structurally identical to the reported slippage bug: the "protection" value (`payload_hash`) is taken from the same source being protected (the node's response), making the check self-referential and ineffective against cross-request replay by a single malicious participant.

### Finding Description

In `respond_verify_foreign_tx`, the contract extracts `payload_hash` directly from `response.payload_hash` and verifies the signature against it:

```rust
// crates/contract/src/lib.rs lines 726-734
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract never checks that `response.payload_hash` was correctly derived from the `request` argument (which contains the foreign chain tx ID, extractors, and finality). The `payload_hash` is fully attacker-controlled by the single responding node.

Contrast this with `respond` (regular signing), where the payload is taken from the **stored request** — not from the response — and the signature is verified against a **derived** key (using `request.tweak`), providing both payload binding and domain separation:

```rust
// crates/contract/src/lib.rs lines 600-607
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,  // derived key, not root key
)
``` [2](#0-1) 

`respond_verify_foreign_tx` also verifies against the **root** public key (`secp_pk` directly from `near_public_key`), not a per-request derived key. This eliminates domain separation across all foreign-tx requests, enabling cross-request replay. [3](#0-2) 

The `VerifyForeignTransactionResponse` type confirms that `payload_hash` is a free field in the response, not a commitment derived on-chain:

```rust
pub struct VerifyForeignTransactionResponse {
    pub payload_hash: Hash256,
    pub signature: SignatureResponse,
}
``` [4](#0-3) 

### Impact Explanation

A single malicious attested participant node can replay a valid MPC signature produced for foreign-tx request **R_A** (tx_id X) to resolve a different pending request **R_B** (tx_id Y):

1. The MPC network legitimately signs hash `H_A` (derived from tx_id X) for request R_A.
2. The malicious node calls `respond_verify_foreign_tx(R_B, { payload_hash: H_A, signature: S_A })`.
3. The contract verifies `verify_ecdsa_signature(S_A, H_A, root_pk)` → **passes** (S_A is valid over H_A).
4. The contract resolves the yield for R_B with the forged response.
5. The caller of R_B receives `payload_hash = H_A` (corresponding to tx_id X, not Y) and a valid MPC signature over it.

Any downstream contract that trusts the MPC contract's `verify_foreign_transaction` response to authorize a bridge action (e.g., releasing funds upon confirmation of a deposit tx) will act on the wrong transaction. This enables forged foreign-chain verification and potential double-spend or invalid bridge execution.

This matches the **High** allowed impact: *"Cross-chain replay, forged foreign-chain verification, light-client-style verification bypass, or participant/attestation authorization bypass that causes invalid bridge execution or double-spend conditions."*

### Likelihood Explanation

- Requires exactly **one** malicious attested participant node — strictly below the signing threshold.
- The malicious node needs only a valid MPC signature over any hash (legitimately produced by the network for any prior foreign-tx request).
- No threshold collusion, no TEE break, no network-level attack is required.
- The `assert_caller_is_attested_participant_and_protocol_active()` guard is the only barrier, and the attacker satisfies it by being a legitimate participant. [5](#0-4) 

### Recommendation

1. **Bind `payload_hash` to the request on-chain**: Compute `payload_hash` deterministically from the stored `VerifyForeignTransactionRequest` inside `respond_verify_foreign_tx` (using the same `ForeignTxSignPayload::compute_msg_hash` logic the nodes use), and reject any response whose `payload_hash` does not match.

2. **Use a derived key for domain separation**: Mirror `respond`'s pattern — derive a per-request key from the root key using a tweak derived from the request, so a signature produced for one request cannot verify under a different request's key.

### Proof of Concept

```
// Setup: two pending verify_foreign_transaction requests exist:
//   R_A: { tx_id: X, domain_id: D }  → pending yield Y_A
//   R_B: { tx_id: Y, domain_id: D }  → pending yield Y_B

// Step 1: MPC network legitimately signs H_A = hash(R_A payload) → signature S_A
// (threshold nodes cooperate normally for R_A)

// Step 2: Malicious attested node calls:
respond_verify_foreign_tx(
    request = R_B,                          // resolves yield Y_B
    response = {
        payload_hash: H_A,                  // hash of R_A, not R_B
        signature: S_A,                     // valid over H_A against root key
    }
)

// Step 3: Contract check:
//   verify_ecdsa_signature(S_A, H_A, root_pk) → OK  ← always passes
//   resolve_yields_for(pending_verify_foreign_tx_requests, R_B, response) → resolves Y_B

// Result: caller of R_B (tx_id Y) receives a signed attestation for H_A (tx_id X).
// Any bridge contract that gates fund release on this attestation is deceived.
```
