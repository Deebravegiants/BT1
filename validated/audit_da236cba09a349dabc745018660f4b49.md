### Title
`directDeposits` in Dijkstra Sub-Transactions Are Not Included in Preservation-of-Value Check, Enabling Arbitrary ADA Creation - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, sub-transactions can carry a `directDeposits` field that adds ADA directly to registered account balances. However, `directDeposits` is never included in either the **consumed** or **produced** side of the preservation-of-value (PoV) balance check. The UTXO rule therefore accepts transactions where `directDeposits` credits accounts with ADA that was never deducted from any UTxO input, creating ADA out of thin air.

---

### Finding Description

The Cardano ledger's core invariant is that for every valid transaction, `consumed == produced`. This is enforced in the UTXO rule via `validateValueNotConservedUTxO`.

In the Dijkstra era, the `DijkstraSubTxBodyRaw` type carries a `dstbrDirectDeposits :: !DirectDeposits` field. [1](#0-0) 

`DirectDeposits` is a map from `AccountAddress` to `Coin`, and `applyDirectDeposits` adds each coin amount to the matching account's balance. [2](#0-1) 

The `SUBENTITIES` rule applies these deposits unconditionally (after only checking that target accounts are registered): [3](#0-2) 

However, the sub-transaction's **produced value** function `dijkstraSubTxProducedValue` only accounts for outputs, certificate deposits, treasury donations, and burned multi-assets — **not** `directDeposits`: [4](#0-3) 

Similarly, the sub-transaction's **consumed value** is computed via `getConsumedMaryValue`, which accounts for UTxO inputs, withdrawals, refunds, and minted multi-assets — again, **not** `directDeposits`: [5](#0-4) 

The batch-level PoV check aggregates consumed and produced across the top-level transaction and all sub-transactions: [6](#0-5) 

Because `directDeposits` appears in neither the consumed nor the produced sum, the PoV equation balances regardless of how much ADA is specified in `directDeposits`. The coin credited to account balances is never deducted from any UTxO input.

The `DirectDeposits` type definition confirms it is a plain coin map with no self-balancing mechanism: [7](#0-6) 

---

### Impact Explanation

**Critical. Direct creation of ADA through an invalid ledger state transition.**

An attacker can include a sub-transaction with `directDeposits = {addr: N lovelace}` for any N. The UTXO rule's `consumed == produced` check passes because `directDeposits` is absent from both sides. The `SUBENTITIES` rule then credits N lovelace to `addr`'s account balance. This N lovelace was never deducted from any UTxO input, violating the Full Preservation of ADA invariant:

> `Circulation + Deposits + Fees + Treasury + Rewards + Reserves` must remain constant. [8](#0-7) 

The inflated account balance can subsequently be withdrawn as a normal reward withdrawal, permanently increasing the total ADA supply.

---

### Likelihood Explanation

Any unprivileged transaction submitter can exploit this. The only precondition is that the target `AccountAddress` is registered (checked by `directDepositsMissingAccounts`). An attacker can register their own account address and then submit a top-level Dijkstra transaction containing a sub-transaction with `directDeposits` crediting their own account with an arbitrary amount. No privileged access, governance majority, or key compromise is required. [3](#0-2) 

---

### Recommendation

`directDeposits` must be included in the preservation-of-value accounting. Since `directDeposits` transfers ADA from the UTxO to account balances, the total amount of `directDeposits` should be added to the **produced** side of the balance equation, analogous to how `treasuryDonationTxBodyL` is already included:

```haskell
dijkstraSubTxProducedValue pp isRegPoolId txBody =
  sumAllValue (txBody ^. outputsTxBodyL)
    <> inject (  getTotalDepositsTxBody pp isRegPoolId txBody
              <> txBody ^. treasuryDonationTxBodyL
              <> fold (unDirectDeposits (txBody ^. directDepositsTxBodyL))  -- ADD THIS
              )
    <> burnedMultiAssets txBody
```

This ensures that any ADA credited to accounts via `directDeposits` must be funded by UTxO inputs, preserving the total ADA supply invariant. [4](#0-3) 

The same fix must be applied to the top-level transaction's `directDeposits` (`dtbrDirectDeposits`) in `dijkstraProducedValue` or `conwayProducedValue` override for `DijkstraEra`.

---

### Proof of Concept

**Entry path:**

1. Attacker registers an account address `addr` (standard stake key registration).
2. Attacker constructs a Dijkstra top-level transaction containing one sub-transaction.
3. The sub-transaction body has:
   - `dstbSpendInputs` = some UTxO input worth exactly the fee (e.g., 1 ADA)
   - `dstbOutputs` = one output returning 1 ADA to attacker (balanced)
   - `dstbDirectDeposits` = `{addr: 1_000_000_000_000 lovelace}` (1 million ADA)
4. The top-level transaction pays the required fee from its own inputs.
5. The UTXO rule evaluates `consumed == produced` for the sub-transaction:
   - consumed = 1 ADA (input)
   - produced = 1 ADA (output) + 0 deposits + 0 treasury donation + 0 burned
   - `1 ADA == 1 ADA` ✓ — check passes; `directDeposits` is absent from both sides.
6. The `SUBENTITIES` rule applies `directDeposits`, crediting `addr` with 1,000,000,000,000 lovelace.
7. Attacker submits a withdrawal transaction draining `addr`'s balance.
8. 1 million ADA has been created from nothing and transferred to the attacker. [4](#0-3) [9](#0-8)
