### Title
Reward precision loss causes irrecoverable accumulation of yield in `notifyRewardAmount` - (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary
`KernelDepositPool.notifyRewardAmount()` transfers the full `receivedAmount` into the contract but computes `rewardRate` via integer division, permanently locking the residual `receivedAmount % duration` (or `(receivedAmount + remaining) % duration`) with no recovery path. This residual accumulates on every invocation.

### Finding Description
In `notifyRewardAmount`, the full reward amount is pulled into the contract via `safeTransferFrom`: [1](#0-0) 

Then `rewardRate` is set by integer division: [2](#0-1) 

Both branches suffer the same defect:
- **New period (line 580):** `rewardRate = receivedAmount / duration` — residual `receivedAmount % duration` is never distributed.
- **Mid-period top-up (line 583):** `rewardRate = (receivedAmount + remaining) / duration` — residual `(receivedAmount + remaining) % duration` is never distributed.

Because `finishAt = block.timestamp + duration` is set after each call, the contract will only ever pay out `rewardRate * duration` tokens. The remainder is permanently locked with no sweep, rescue, or carry-forward mechanism anywhere in the contract. [3](#0-2) 

### Impact Explanation
**Medium — Permanent freezing of unclaimed yield.**

Every call to `notifyRewardAmount` silently discards up to `duration - 1` reward tokens (e.g., with `duration = 7 days = 604800 s`, up to 604799 token-wei per call). These tokens are transferred into the contract and counted in its balance but are never reachable by any staker or admin function. The loss compounds across every reward epoch, permanently reducing the yield delivered to stakers relative to what was deposited.

### Likelihood Explanation
**High.** The precision loss occurs on every single invocation of `notifyRewardAmount` for any non-zero `receivedAmount % duration`. Since this is the routine operational function for funding reward epochs, it fires repeatedly throughout the protocol's lifetime.

### Recommendation
Carry the residual forward into the next epoch by tracking it explicitly:

```solidity
uint256 residual = receivedAmount % duration; // or (receivedAmount + remaining) % duration
// store residual and add it to receivedAmount on the next call
```

Alternatively, after computing `rewardRate`, compute the exact distributable amount as `rewardRate * duration` and refund or roll over `receivedAmount - rewardRate * duration` rather than leaving it stranded.

### Proof of Concept

```solidity
// duration = 604800 (7 days in seconds)
// receivedAmount = 604799  (one less than duration)
// rewardRate = 604799 / 604800 = 0  → revert RewardRateZero (edge case)

// receivedAmount = 604800 + 1 = 604801
// rewardRate = 604801 / 604800 = 1
// distributed = 1 * 604800 = 604800
// stuck = 604801 - 604800 = 1 token-wei per epoch

// With a realistic 18-decimal token and reward = 1e18:
// rewardRate = 1e18 / 604800 = 1_653_439_153_439
// distributed = 1_653_439_153_439 * 604800 = 999_999_999_999_907_200
// stuck per epoch = 1e18 - 999_999_999_999_907_200 = 92_800 token-wei
// Over 52 epochs/year: 4_825_600 token-wei permanently locked, accumulating indefinitely
```

The stuck tokens sit in `rewardsToken.balanceOf(address(this))` but are never reachable by `getReward`, `setRewardsDuration`, or any other function in `KernelDepositPool`. [4](#0-3)

    function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
        if (_amount == 0) revert AmountZero();

        // Prevent starting a reward period when no tokens are staked to avoid unallocated rewards
        if (totalKernelStaked == 0) revert NoStakedTokens();

        // Transfer reward tokens into the contract
        uint256 balanceBefore = rewardsToken.balanceOf(address(this));
        rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
        uint256 balanceAfter = rewardsToken.balanceOf(address(this));
        // Calculate the actual amount of tokens received in case of a transfer fee (tax)
        uint256 receivedAmount = balanceAfter - balanceBefore;

        if (block.timestamp >= finishAt) {
            rewardRate = receivedAmount / duration;
        } else {
            uint256 remaining = (finishAt - block.timestamp) * rewardRate;
            rewardRate = (receivedAmount + remaining) / duration;
        }

        if (rewardRate == 0) revert RewardRateZero();

        finishAt = block.timestamp + duration;
        updatedAt = block.timestamp;

        emit NotifyRewardAmount(receivedAmount, finishAt);
    }
