### Title
Propose-Update Storage Deposits Permanently Locked on `do_update` Execution — (`File: crates/contract/src/update.rs`)

---

### Summary

The `propose_update` function in the MPC contract collects a NEAR deposit from the proposer to cover on-chain storage costs. When any update reaches threshold and `do_update` is called, it clears **all** pending update entries — freeing their storage — but never refunds the deposits paid by the proposers of the non-executed entries. Because `UpdateEntry` does not record the proposer's account ID or the deposit amount, those NEAR tokens are permanently locked in the contract account with no withdrawal path.

---

### Finding Description

`propose_update` is a `#[payable]` function that requires a deposit calculated from the storage footprint of the proposed update: [1](#0-0) 

The required deposit is computed as: [2](#0-1) [3](#0-2) 

Only the excess above `required` is refunded to the proposer; the exact `required` amount is retained by the contract. The `UpdateEntry` stored on-chain contains only the update payload and `bytes_used` — no proposer account ID and no deposit amount: [4](#0-3) 

When `vote_update` reaches threshold, it calls `do_update`, which removes the winning entry and then **clears all remaining entries and votes**: [5](#0-4) 

`self.entries.clear()` frees the on-chain storage for every non-executed proposal, but no refund is issued to any of the proposers whose entries were discarded. Because the proposer identity and deposit amount are not stored in `UpdateEntry`, there is no data available to reconstruct who should be refunded, even in a future contract upgrade that adds a refund path.

---

### Impact Explanation

Every time a governance update is successfully executed, all NEAR tokens deposited by the proposers of the **other** pending updates are permanently locked in the contract account. The contract has no `withdraw` function or any other mechanism to release these accumulated deposits. Over multiple governance cycles (each of which is a normal, expected operation), the locked balance grows monotonically. This breaks the production accounting invariant that storage deposits are returned when the corresponding storage is freed.

**Impact: Medium** — balance/accounting invariant broken; funds permanently locked without a contract upgrade.

---

### Likelihood Explanation

This condition is triggered by every successful `vote_update` execution, which is a routine governance operation. Any governance cycle in which more than one update is proposed simultaneously (a realistic scenario when participants disagree on which update to apply) will lock the deposits of all non-winning proposers. Likelihood is **Medium-High** given that multi-proposal governance rounds are expected in production.

---

### Recommendation

1. Add a `proposer: AccountId` and `deposit: NearToken` field to `UpdateEntry` so the contract can track who paid what.
2. In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and issue `Promise::new(entry.proposer).transfer(entry.deposit)` for each one.
3. Alternatively, maintain a separate `IterableMap<UpdateId, (AccountId, NearToken)>` that maps each proposal to its proposer and deposit, and drain it with refunds inside `do_update`.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB contract blob, paying deposit `D_A` (≈ storage cost for ~1 MB + overhead).
2. Participant B calls `propose_update` with a config change, paying deposit `D_B`.
3. Threshold participants vote for B's update via `vote_update(id_B)`.
4. `do_update` executes: it removes entry B, then calls `self.entries.clear()` (removing entry A) and `self.vote_by_participant.clear()`.
5. Storage for both entries is freed; the NEAR protocol unlocks the corresponding tokens back to the contract account.
6. `D_A` is now held by the contract account. There is no function to return it to Participant A.
7. Participant A's deposit is permanently locked. [6](#0-5) [7](#0-6)

    #[payable]
    #[handle_result]
    pub fn propose_update(
        &mut self,
        #[serializer(borsh)] args: ProposeUpdateArgs,
    ) -> Result<UpdateId, Error> {
        // Only voters can propose updates:
        let proposer = self.voter_or_panic();
        let update: Update = args.try_into()?;

        let attached = env::attached_deposit();
        let required = ProposedUpdates::required_deposit(&update);
        if attached < required {
            return Err(InvalidParameters::InsufficientDeposit {
                attached: attached.as_yoctonear(),
                required: required.as_yoctonear(),
            }
            .into());
        }

        let id = self.proposed_updates.propose(update);

        log!(
            "propose_update: signer={}, id={:?}",
            env::signer_account_id(),
            id,
        );

        // Refund the difference if the proposer attached more than required.
        if let Some(diff) = attached.checked_sub(required)
            && diff > NearToken::from_yoctonear(0)
        {
            Promise::new(proposer).transfer(diff).detach();
        }

        Ok(id)
    }

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }
