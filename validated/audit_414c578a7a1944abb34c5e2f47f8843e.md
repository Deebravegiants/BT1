Audit Report

## Title
ETH Deposit Limit Check Omits Incoming Amount, Allowing Cap Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric comparison for ETH versus ERC20 assets: the ETH branch evaluates `totalAssetDeposits > limit` while the ERC20 branch correctly evaluates `totalAssetDeposits + amount > limit`. Because the incoming deposit amount is excluded from the ETH comparison, the cap is only enforced after it has already been breached, not before. Any unprivileged depositor can push ETH holdings past the configured `depositLimitByAsset` ceiling.

## Finding Description
The function at `contracts/LRTDepositPool.sol` lines 676–682 is the sole gate enforcing the per-asset deposit cap before rsETH is minted:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // amount omitted
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct
}
```

`_beforeDeposit` (lines 648–670) calls this function and reverts with `MaximumDepositLimitReached` only when it returns `true`. For ETH, the function returns `true` only when the cap is **already** exceeded — not when the incoming deposit would exceed it. A deposit that brings the total from `limit - 1 wei` to `limit + msg.value` passes the check and proceeds to mint rsETH. The public `depositETH` function (lines 76–93) is `payable` with no role restriction, making this reachable by any caller.

## Impact Explanation
The `depositLimitByAsset` cap is the protocol's primary mechanism for bounding ETH exposure. With the broken check, any depositor can receive rsETH for ETH that exceeds the governance-approved ceiling. No funds are stolen or frozen, but the contract fails to deliver its promised deposit-limit guarantee. This matches the allowed impact: **Low — contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
`depositETH` is a fully permissionless `external payable` function. No role, whitelist, or precondition is required beyond the contract not being paused. Any user who observes `getTotalAssetDeposits(ETH_TOKEN)` approaching `depositLimitByAsset(ETH_TOKEN)` on-chain can immediately exploit the gap. The condition is easily monitored and the exploit is repeatable until governance intervenes. Likelihood is **High**.

## Recommendation
Remove the ETH-specific branch and unify the comparison:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

`getTotalAssetDeposits` already handles ETH correctly via `getETHDistributionData`, so the special-casing of `ETH_TOKEN` is unnecessary.

## Proof of Concept
1. Set `depositLimitByAsset(ETH_TOKEN) = 100 ether` and assume `getTotalAssetDeposits(ETH_TOKEN) = 100 ether - 1 wei`.
2. Call `depositETH{value: 10 ether}(0, "")` from any EOA.
3. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 10 ether)` evaluates `(100 ether - 1 wei) > 100 ether` → `false` → no revert.
4. rsETH is minted for 10 ETH; `getTotalAssetDeposits(ETH_TOKEN)` becomes `110 ether - 1 wei`, ~10 ETH over the cap.
5. The equivalent ERC20 path evaluates `(100 ether - 1 wei) + 10 ether > 100 ether` → `true` → correctly reverts.

**Foundry test sketch:**
```solidity
function test_ethDepositBypassesLimit() public {
    uint256 limit = 100 ether;
    // configure limit and seed pool to limit - 1 wei
    vm.deal(attacker, 10 ether);
    vm.prank(attacker);
    depositPool.depositETH{value: 10 ether}(0, "");
    assertGt(depositPool.getTotalAssetDeposits(ETH_TOKEN), limit);
}
```
