### Title
Identity-Point `AppPublicKeyPV` Bypasses ElGamal Encryption, Exposing Per-App-ID BLS Secret On-Chain — (`crates/contract/src/primitives/ckd.rs` / `crates/threshold-signatures/src/confidential_key_derivation/protocol_pv.rs`)

---

### Summary

An unprivileged caller can submit a `CKDRequestArgs` with `AppPublicKeyPV{pk1: G1::identity(), pk2: G2::identity()}`. Both the contract-side and node-side `app_public_key_check` functions accept this pair because the identity trivially satisfies the pairing equation. The MPC nodes then compute `big_c = big_s + identity * y = big_s`, publishing the raw per-app-id BLS secret `H(pk || app_id) * msk` in plaintext on-chain. The `ckd_output_check` also passes for the same algebraic reason. Any chain observer can read `big_c` directly as the app's confidential key without possessing the private scalar `a`.

---

### Finding Description

**Step 1 — Contract-side `app_public_key_check` accepts identity pair.** [1](#0-0) 

The check verifies `e(pk1, -g2) * e(g1, pk2) = 1`. With `pk1 = pk2 = identity`, both pairings evaluate to 1, so the product is 1. The test at line 483 explicitly documents this as accepted behavior: [2](#0-1) 

**Step 2 — Node-side `app_public_key_check` also accepts identity pair.** [3](#0-2) 

`check_valid_point_g1` and `check_valid_point_g2` only test `is_on_curve() & is_torsion_free()`: [4](#0-3) 

The identity point is on the curve and is torsion-free (it is the neutral element of the prime-order subgroup), so both checks return `true`. The subsequent `multi_miller_loop` evaluates to 1 for the same reason as above.

**Step 3 — MPC computation with identity `pk1` strips the ElGamal mask.** [5](#0-4) 

Each participant computes `big_c = big_s + app_pk.pk1 * y`. With `app_pk.pk1 = identity`, `identity * y = identity`, so `big_c = big_s`. After Lagrange aggregation across all participants, the coordinator holds `big_c = H(pk || app_id) * msk` — the raw BLS signature — with no blinding term.

**Step 4 — `aggregated_output_check` and contract-side `ckd_output_check` both pass.**

The check is `e(big_c, -g2) * e(big_y, pk2) * e(hash_point, pk) = 1`. With `pk2 = identity`, the middle term is 1. The equation reduces to `e(big_c, -g2) * e(hash_point, pk) = 1`, which holds because `big_c = hash_point * msk` and `pk = msk * g2` (bilinearity: `e(hash_point * msk, -g2) * e(hash_point, msk * g2) = 1`). [6](#0-5) [7](#0-6) 

**Step 5 — The `unmask` operation is a no-op.** [8](#0-7) 

`unmask(a)` computes `big_c - a * big_y`. With `big_c = big_s` already, any observer who reads `big_c` from the chain has the secret directly, without needing `a`.

---

### Impact Explanation

The CKD protocol's security model is that `big_s = H(pk || app_id) * msk` is hidden by ElGamal encryption: `big_c = big_s + y * pk1`. With `pk1 = identity`, the encryption term vanishes and `big_c = big_s` is published on-chain in the `CKDResponse`. Any third-party chain observer — not just the requester — can read the app's confidential key directly. The entire confidentiality guarantee of the CKD protocol is broken for the targeted `app_id`. The exposed value `H(pk || app_id) * msk` is the secret material the protocol is designed to protect.

This is unauthorized access to secret material (the per-app-id BLS signature) that the protocol explicitly guarantees is never revealed to any entity other than the application itself.

---

### Likelihood Explanation

The attack requires no special privileges: any NEAR account can call `request_app_private_key` with an identity-point `AppPublicKeyPV`. The identity point encoding is publicly known (all-zero compressed encoding with the infinity flag set). The test `app_public_key_check__should_accept_identity_key_pair` confirms the bypass is reachable in the current codebase without any modification.

---

### Recommendation

Add an explicit identity-point rejection in both `app_public_key_check` functions before the pairing check:

- **Contract** (`crates/contract/src/primitives/ckd.rs`): After decompressing `pk1` and `pk2`, check that neither is the identity point. The NEAR host's `bls12381_p1_decompress` / `bls12381_p2_decompress` accept the identity encoding, so an explicit byte-level or post-decompression identity check is required.
- **Node** (`crates/threshold-signatures/src/confidential_key_derivation/protocol_pv.rs`): Add `|| app_pk.pk1.is_identity().into() || app_pk.pk2.is_identity().into()` to the early-return guard in `app_public_key_check`.

The same guard should be applied in `aggregated_output_check` for `big_y` (an identity `big_y` with a non-identity `pk2` would also degenerate the output check).
