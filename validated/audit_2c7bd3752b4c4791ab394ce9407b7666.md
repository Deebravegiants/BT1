### Title
Sub-Transaction ExUnits Excluded from Block-Level and Per-Transaction Resource Limit Checks in Dijkstra Era — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs`)

---

### Summary

In the Dijkstra era, the `totExUnits` function used by both the per-transaction (`maxTxExUnits`) and per-block (`maxBlockExUnits`) ExUnits validation checks only sums execution units from the **top-level transaction's** redeemers. Sub-transactions embedded in a Dijkstra top-level transaction carry their own witnesses and redeemers, whose ExUnits are never aggregated into these checks. An unprivileged transaction author can craft a top-level transaction with zero top-level script ExUnits but embed many sub-transactions each consuming up to `maxTxExUnits`, causing the actual block-level computation to far exceed `maxBlockExUnits` without triggering any predicate failure.

---

### Finding Description

The Dijkstra era introduces nested transactions: a `Tx TopTx era` may contain an `OMap TxId (Tx SubTx era)` in its body (`dtbrSubTransactions`). Each sub-transaction carries its own `TxWits`, including its own `Redeemers` map with per-script `ExUnits` budgets.

The block-level ExUnits check in `dijkstraBbodyTransition` delegates directly to the Alonzo-era `validateExUnits`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs, line 361
Alonzo.validateExUnits @era txs $ pp ^. ppMaxBlockExUnitsL
```

`validateExUnits` folds `totExUnits` over the sequence of top-level transactions:

```haskell
-- eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Rules/Bbody.hs, lines 158-160
validateExUnits txs ppMax =
  let txTotal = foldMap totExUnits txs
   in pointWiseExUnits (<=) txTotal ppMax
```

`totExUnits` is defined in the Alonzo era and sums ExUnits only from the redeemers in the transaction's own witness set:

```haskell
-- eras/alonzo/formal-spec/utxo.tex, line 16
totExUnits tx = ∑_{r ∈ txrdmrs tx} exunits r
```

Sub-transaction redeemers live in each sub-transaction's own `TxWits` (`dstWits`), not in the top-level `TxWits`. The Dijkstra era does not override `totExUnits` (confirmed: `grep` for `totExUnits` returns zero matches in any Dijkstra source file). Therefore, sub-transaction ExUnits are invisible to both:

1. **Per-transaction check** (`validateExUnitsTooBigUTxO`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`, line 415):
   ```haskell
   {- totExunits tx ≤ maxTxExUnits pp -}
   runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
   ```

2. **Per-block check** (`dijkstraBbodyTransition`, line 361 above).

---

### Impact Explanation

`maxBlockExUnits` is the protocol parameter that bounds the total Plutus execution budget per block, ensuring that block validation time and memory usage remain within the limits required for honest nodes to keep up with the chain. By embedding N sub-transactions each carrying up to `maxTxExUnits` ExUnits, an attacker can cause a single top-level transaction to consume N × `maxTxExUnits` actual computation while the block-level check observes only the top-level ExUnits (potentially zero). A block containing such a transaction would require far more CPU and memory to validate than the protocol parameters permit, exceeding intended validation limits.

**Matched impact**: *Medium — Attacker-controlled transactions exceed intended validation limits.*

---

### Likelihood Explanation

The Dijkstra era is the current development era and is not yet deployed on mainnet. However, the code is present in the repository and the flaw is structural: no override of `totExUnits` exists for the Dijkstra era, and the BBODY and UTXO rules reuse the Alonzo implementation unchanged. Any transaction author can craft a top-level transaction with an `OMap` of sub-transactions, each with a full `maxTxExUnits` redeemer budget, without requiring any privileged access, governance action, or key compromise.

---

### Recommendation

Override `totExUnits` (or the ExUnits aggregation step) in the Dijkstra era to recursively include the ExUnits of all embedded sub-transactions. Concretely, the block-level `validateExUnits` call and the per-transaction `validateExUnitsTooBigUTxO` call should operate on a Dijkstra-aware `totExUnits` that folds over `subTransactionsTxBodyL` and accumulates sub-transaction redeemer ExUnits alongside the top-level redeemers. This mirrors how `getConsumedDijkstraValue` and `dijkstraProducedValue` already recursively fold over sub-transactions for value-conservation accounting.

---

### Proof of Concept

**Root cause chain:**

1. `dijkstraBbodyTransition` calls `Alonzo.validateExUnits @era txs (pp ^. ppMaxBlockExUnitsL)`. [1](#0-0) 

2. `validateExUnits` folds `totExUnits` over top-level transactions only. [2](#0-1) 

3. `totExUnits` sums redeemers from the transaction's own witness set; sub-transaction witnesses are not consulted. [3](#0-2) 

4. Sub-transactions are stored in `dtbrSubTransactions` in the top-level body and carry independent `dstWits` with their own `Redeemers`. [4](#0-3) [5](#0-4) 

5. The Dijkstra UTXO rule also applies the Alonzo per-transaction check without sub-transaction ExUnits. [6](#0-5) 

6. No Dijkstra-era override of `totExUnits` exists (grep confirms zero matches in `eras/dijkstra/`).

**Attack scenario:**
- Construct a `DijkstraTx` with `dtbrSubTransactions` containing K `DijkstraSubTx` entries.
- Each sub-transaction's `dstWits` contains a redeemer with `ExUnits = maxTxExUnits`.
- The top-level transaction carries zero redeemers.
- `totExUnits topTx = mempty`; the block-level check passes.
- Actual Plutus execution budget consumed = K × `maxTxExUnits`, which can be made arbitrarily larger than `maxBlockExUnits` by increasing K (bounded only by `maxTxSize`).
