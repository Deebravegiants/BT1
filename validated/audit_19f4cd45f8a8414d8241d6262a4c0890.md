### Title
Sub-transaction ExUnits bypass `maxTxExUnits` and `maxBlockExUnits` limits in Dijkstra era — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`, `eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Rules/Bbody.hs`)

---

### Summary

The Dijkstra era introduces nested ("sub") transactions embedded inside a top-level transaction body. The `maxTxExUnits` and `maxBlockExUnits` protocol-parameter checks are computed exclusively from the top-level transaction's redeemers via `totExUnits`. Sub-transactions carry their own witnesses and redeemers whose declared `ExUnits` are never aggregated into that check. An unprivileged transaction submitter can therefore craft a top-level transaction whose top-level redeemers declare zero ExUnits (passing the limit check) while embedding arbitrarily many sub-transactions each declaring up to `maxTxExUnits` worth of Plutus execution budget, causing validators to expend computational resources far beyond the intended per-transaction and per-block caps.

---

### Finding Description

**Root cause — `totExUnits` ignores sub-transaction redeemers**

`validateExUnitsTooBigUTxO` in the Dijkstra UTXO rule calls `totExUnits tx` on the top-level transaction only: [1](#0-0) 

`totExUnits` is defined in the Alonzo era and folds only over the transaction's own witness redeemers:

```haskell
totExUnits = foldMap snd . Map.elems . unRedeemers . (^. witsTxL . rdmrsTxWitsL)
```

Sub-transactions are stored in `dtbrSubTransactions` inside the top-level body: [2](#0-1) 

Each sub-transaction carries its own `TxWits` (and therefore its own redeemers with declared `ExUnits`): [3](#0-2) 

Sub-transactions are confirmed to support Plutus scripts via `dsastPlutusScriptsWithContext`: [4](#0-3) 

**The `SUBUTXO` rule has no ExUnits check**

`dijkstraSubUtxoTransition` validates outputs, inputs, network IDs, and validity intervals, but contains no call to `validateExUnitsTooBigUTxO` or any equivalent: [5](#0-4) 

**`maxBlockExUnits` is equally bypassed**

The Alonzo BBODY rule aggregates `totExUnits` across all transactions in the block: [6](#0-5) 

Because `totExUnits` never descends into sub-transactions, the block-level cap is also bypassed.

**`maxTxSize` does cover sub-transactions — creating an asymmetry**

`sizeDijkstraTxF` serialises the full body (which includes sub-transactions at CBOR key 23) for the size check: [7](#0-6) 

So size is bounded but ExUnits are not — a direct asymmetry in the validation rules.

**No `maxSubTxExUnits` parameter exists**

The Dijkstra PParams define only `dppMaxTxExUnits` and `dppMaxBlockExUnits`: [8](#0-7) 

There is no separate cap for sub-transaction execution budgets.

---

### Impact Explanation

`maxTxExUnits` and `maxBlockExUnits` are the protocol's primary mechanism for bounding the CPU and memory cost of Plutus script evaluation per transaction and per block. By embedding N sub-transactions each declaring `ExUnits = (maxMem, maxSteps)` while keeping the top-level redeemers at zero, an attacker causes validators to execute Plutus scripts consuming up to `N × maxTxExUnits` resources for a single transaction that passes all ledger checks. Within a 16 KB `maxTxSize` budget, hundreds of sub-transaction redeemer entries can be packed, multiplying the intended resource cap by the same factor. This matches the allowed Medium impact: **attacker-controlled transactions exceed intended validation limits**.

---

### Likelihood Explanation

The Dijkstra era is the active development era and nested transactions are its headline feature. Any transaction submitter can construct a top-level transaction with embedded sub-transactions — no privileged role, governance majority, or key compromise is required. The attack requires only knowledge of the CBOR transaction format and the ability to pay the standard transaction fee (which is computed from `maxTxSize`, not from ExUnits).

---

### Recommendation

Aggregate sub-transaction ExUnits into the top-level check. The simplest fix is to extend `totExUnits` (or introduce a Dijkstra-specific override) to recursively sum the redeemer ExUnits of all sub-transactions:

```haskell
-- In dijkstraUtxoTransition, replace:
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
-- with a batch-aware check:
let batchExUnits = totExUnits tx
      <> foldMap totExUnits (tx ^. bodyTxL . subTransactionsTxBodyL)
failureUnless (pointWiseExUnits (<=) batchExUnits (pp ^. ppMaxTxExUnitsL)) $
  injectFailure (ExUnitsTooBigUTxO ...)
```

The same aggregation must be applied in the BBODY `validateExUnits` call so that `maxBlockExUnits` correctly accounts for sub-transaction scripts.

---

### Proof of Concept

1. Construct a Dijkstra top-level transaction with **zero** redeemers in its own `TxWits` (so `totExUnits tx = (0, 0)` — passes `maxTxExUnits` check).
2. Embed K sub-transactions in `dtbrSubTransactions`, each spending a UTxO locked by a Plutus script and each declaring `ExUnits = (ppMaxTxExUnits.mem, ppMaxTxExUnits.steps)` in its own `TxWits` redeemers.
3. Submit the transaction. `validateExUnitsTooBigUTxO` checks only the top-level `totExUnits = (0,0) ≤ maxTxExUnits` — **passes**.
4. The `SUBUTXOW` rule executes each sub-transaction's Plutus scripts with the declared budget. Total resources consumed ≈ `K × maxTxExUnits`, far exceeding the intended per-transaction cap.
5. Repeat across multiple transactions in the same block to similarly exceed `maxBlockExUnits`.

-- | Validate that total execution units (all transactions) do not exceed block limit.
-- ∑(tx ∈ txs)(totExunits tx) ≤ maxBlockExUnits pp
validateExUnits ::
  forall era.
  ( AlonzoEraTx era
  , InjectRuleFailure "BBODY" AlonzoBbodyPredFailure era
  ) =>
  StrictSeq.StrictSeq (Tx TopTx era) ->
  -- | Max block exunits protocol parameter.
  ExUnits ->
  Rule (EraRule "BBODY" era) 'Transition ()
validateExUnits txs ppMax =
  let txTotal = foldMap totExUnits txs
   in pointWiseExUnits (<=) txTotal ppMax
        ?! injectFailure
          ( TooManyExUnits $
              Mismatch
                { mismatchSupplied = txTotal
                , mismatchExpected = ppMax
                }
          )
