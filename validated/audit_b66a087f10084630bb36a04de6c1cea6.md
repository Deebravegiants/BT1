### Title
Missing Cross-Sub-Transaction Deposit Refund Guard Enables Double-Claiming of ADA Deposits in Dijkstra Era - (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubLedgers.hs`)

---

### Summary

The Dijkstra era introduces nested transactions (sub-transactions) processed sequentially by the `SUBLEDGERS` rule. The rule threads `LedgerState` through each sub-transaction via `foldM`, but no cross-sub-transaction guard prevents the same staking credential from being deregistered — and its deposit refunded — in two different sub-transactions within the same top-level transaction. The developers have explicitly acknowledged this gap with a disabled (`xit`) test and a `TODO` comment stating the predicate failure is not yet implemented. An unprivileged user can exploit this to receive twice the deposit refund while only having paid it once, directly creating ADA from nothing.

---

### Finding Description

**Root cause — `SUBLEDGERS` rule (`SubLedgers.hs`)**

The `dijkstraSubLedgersTransition` function folds over all sub-transactions, threading `LedgerState` through each one:

```haskell
dijkstraSubLedgersTransition = do
  TRC (env, ledgerState, subTxs) <- judgmentContext
  foldM
    ( \ls subTx ->
        trans @(EraRule "SUBLEDGER" era) $ TRC (env, ls, subTx)
    )
    ledgerState
    subTxs
``` [1](#0-0) 

Each `SUBLEDGER` invocation runs `SUBENTITIES` (which processes certificates, including `UnRegDepositTxCert`) against the accumulated `certState`. After sub-transaction 1 deregisters credential `cred` and claims refund `D`, the updated `certState` is passed to sub-transaction 2. However, no implemented predicate failure prevents sub-transaction 2 from also issuing `UnRegDepositTxCert cred D` and claiming the same refund a second time.

**Developer acknowledgement — disabled test (`CertSpec.hs`)**

The test suite contains a skipped test (`xit`) that directly describes this scenario:

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
    stakingCred <- KeyHashObj <$> freshKeyHash
    _ <- registerStakeCredential stakingCred
    keyDeposit <- getsPParams ppKeyDepositL
    ...
    let
      subTx1 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
      subTx2 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
      tx = mkBasicTx mkBasicTxBody & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
    submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
``` [2](#0-1) 

`xit` marks the test as pending (never executed). The placeholder `error "TODO: predicate failure not yet implemented"` confirms that no rejection predicate exists yet. Because `submitFailingTx` expects a failure, the `xit` suppression implies the transaction currently **succeeds** — i.e., both sub-transactions claim the refund.

**Sub-transaction processing flow**

The `SUBLEDGER` rule for each sub-transaction runs `SUBENTITIES` (certificate processing) before `SUBUTXOW` (UTxO/balance accounting): [3](#0-2) 

The balance conservation check in `SUBUTXOW` → `SUBUTXO` accounts for the refund credited to each sub-transaction individually. With no cross-sub-transaction deduplication, both sub-transactions independently pass balance conservation while each claiming `D` lovelace in refunds.

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An attacker who has registered a staking credential (paying deposit `D`) can craft a single top-level transaction containing two sub-transactions that both issue `UnRegDepositTxCert` for the same credential. If both succeed, the attacker receives `2D` in refunds while only `D` was ever deposited. The difference `D` is net-created ADA, violating the ledger's conservation-of-value invariant. At current mainnet key deposit values (2,000,000 lovelace), each such transaction yields a profit of 2 ADA. The attack is repeatable at scale: register many credentials, then double-claim all of them in a single transaction batch.

---

### Likelihood Explanation

**High.** The entry path requires no privileged access, no leaked keys, and no governance majority. Any user who can submit a valid Dijkstra-era transaction can trigger this. The only prerequisite is owning a registered staking credential, which is itself a normal user action. The developers have already identified the scenario and noted the missing guard, confirming the attack surface is real and reachable.

---

### Recommendation

1. In the `SUBLEDGERS` or `SUBLEDGER` rule, track which credentials have been deregistered across the sub-transaction sequence and reject any subsequent `UnRegDepositTxCert` for an already-deregistered credential within the same top-level transaction.
2. Alternatively, enforce at the `SUBDELEG` level that `UnRegDepositTxCert` for a credential that is no longer present in `certState` (because a prior sub-transaction removed it) produces a concrete predicate failure (e.g., `SubCredentialNotRegistered`).
3. Enable and complete the `xit "Multiple subtransactions cannot get the same refund"` test once the predicate failure is implemented.

---

### Proof of Concept

```
1. Alice registers staking credential `cred` in a prior transaction, paying deposit D = ppKeyDeposit.
2. Alice constructs:
     subTx1 = mkBasicTx mkBasicTxBody
               & bodyTxL . inputsTxBodyL  .~ {input1}
               & bodyTxL . certsTxBodyL   .~ [UnRegDepositTxCert cred D]
     subTx2 = mkBasicTx mkBasicTxBody
               & bodyTxL . inputsTxBodyL  .~ {input2}
               & bodyTxL . certsTxBodyL   .~ [UnRegDepositTxCert cred D]
     topTx  = mkBasicTx mkBasicTxBody
               & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
3. Alice submits topTx.
4. SUBLEDGERS processes subTx1: cred is deregistered, D is refunded to subTx1's outputs.
5. SUBLEDGERS processes subTx2: no implemented guard rejects the second UnRegDepositTxCert;
   D is refunded again to subTx2's outputs.
6. Alice receives 2D total. Net ADA created: D.
```

The exact code path is: `LEDGER` → `SUBLEDGERS` (`foldM` over `[subTx1, subTx2]`) → `SUBLEDGER` (×2) → `SUBENTITIES` → `SUBCERTS` → `SUBCERT` → `SUBDELEG` (no rejection on second call because no cross-sub-tx guard exists).
