### Title
Unverified `transaction_output` from p2p peers written to storage without `receipt_commitment` check — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

### Summary
`TransactionStreamFactory::parse_data_for_block` accepts `transaction_output` fields (`actual_fee`, `execution_status`, `messages_sent`, `execution_resources`) verbatim from an unauthenticated p2p peer and writes them to storage via `append_body` with no verification against the `receipt_commitment` already stored in the block header. Any p2p peer can therefore inject a fabricated receipt for any synced block, causing `starknet_getTransactionReceipt` to return an authoritative-looking wrong value.

### Finding Description

`parse_data_for_block` collects `FullTransaction` messages from the network and pushes each `transaction_output` directly into `block_body.transaction_outputs`: [1](#0-0) 

The only guard is a count check against `n_transactions` from the stored header: [2](#0-1) 

The block header already contains a `receipt_commitment` field: [3](#0-2) 

The receipt commitment is a Poseidon root over per-transaction hashes that bind `actual_fee`, `messages_sent`, `execution_status` (revert reason), and `gas_consumed`: [4](#0-3) 

After `parse_data_for_block` returns, `write_to_storage` calls `append_body` unconditionally: [5](#0-4) 

`append_body` stores the peer-supplied outputs as-is with no commitment check: [6](#0-5) 

The header sync path also does not verify the block signature — it only checks that exactly one signature is present: [7](#0-6) 

This means a malicious peer can also supply a header with a fabricated `receipt_commitment` that matches its fabricated outputs, making the stored state internally consistent and undetectable by a downstream client.

### Impact Explanation

Any node that syncs via p2p will store and serve fabricated receipts. `starknet_getTransactionReceipt` reads directly from the stored `transaction_output`: [8](#0-7) 

A caller receives an authoritative-looking receipt with wrong `actual_fee`, wrong `execution_status` (e.g., `Succeeded` for a reverted transaction), wrong `messages_sent`, or wrong `execution_resources`. This is a **High** RPC-authority impact per the allowed scope.

### Likelihood Explanation

The attacker only needs to be a reachable p2p peer — no privileged role is required. The p2p network is open. The fabricated data passes all existing checks (correct transaction count, valid protobuf encoding) and is never reported as a `BadPeer` error, so the peer is not disconnected or retried. The corruption is permanent until the node resyncs from a different peer.

### Recommendation

After collecting all `transaction_output` values for a block, compute the receipt commitment using `calculate_receipt_commitment` and compare it against the `receipt_commitment` stored in the block header. If they differ, return `ParseDataError::BadPeer` and report the peer. This mirrors the existing `state_diff_length` guard in `StateDiffStreamBuilder`: [9](#0-8) 

Additionally, verify block header signatures (not just their count) before trusting the `receipt_commitment` stored in the header.

### Proof of Concept

```rust
// In parse_data_for_block, after the while loop collects all outputs:
use starknet_api::block_hash::receipt_commitment::{calculate_receipt_commitment, ReceiptElement};
use starknet_types_core::hash::Poseidon;

let receipt_elements: Vec<ReceiptElement> = block_body.transaction_hashes
    .iter()
    .zip(block_body.transaction_outputs.iter())
    .map(|(hash, output)| ReceiptElement {
        transaction_hash: *hash,
        transaction_output: output.into(), // TransactionOutputForHash
    })
    .collect();

let computed = calculate_receipt_commitment::<Poseidon>(&receipt_elements);
let stored = storage_reader
    .begin_ro_txn()?
    .get_block_header(block_number)?
    .expect("header must exist")
    .receipt_commitment;

if Some(computed) != stored {
    return Err(ParseDataError::BadPeer(BadPeerError::InvalidReceiptCommitment { block_number }));
}
```

A unit test feeding `parse_data_for_block` a `FullTransaction` with `actual_fee=Fee(0)` and `execution_status=Succeeded` for a block whose stored header has the real `receipt_commitment` would fail the above check, confirming the gap is real and the fix is effective.
