### Title
Unsigned `ContractCodeResponse` (V1) Accepted Without Sender Verification, Enabling Witness Poisoning to Block Chunk Endorsement — (File: `chain/client/src/stateless_validation/validate.rs`)

---

### Summary

Before the `SignedContractCodeResponse` protocol feature is activated, any connected peer can send a `ContractCodeResponse::V1` message to a chunk validator with arbitrary contract code bytes. The handler accepts it without verifying the sender's identity or that the responder is the legitimate chunk producer. The injected bytes are merged into the assembling `ChunkStateWitness.main_state_transition.base_state`, causing the state root check inside `validate_chunk_state_witness_impl` to fail. The chunk validator never emits a `ChunkEndorsement`, and if enough validators are poisoned, the chunk cannot be included in a block, preventing transaction confirmation.

---

### Finding Description

`validate_contract_code_response` in `chain/client/src/stateless_validation/validate.rs` gates sender verification behind a protocol feature flag:

```rust
pub fn validate_contract_code_response(...) -> Result<ChunkRelevance, Error> {
    let key = response.chunk_production_key();
    require_relevant!(validate_chunk_relevant(epoch_manager, key, store)?);
    let protocol_version = epoch_manager.get_epoch_protocol_version(&key.epoch_id)?;
    if ProtocolFeature::SignedContractCodeResponse.enabled(protocol_version) {
        validate_witness_contract_code_response_signature(epoch_manager, response)?;
    }
    Ok(ChunkRelevance::Relevant)
}
``` [1](#0-0) 

When `SignedContractCodeResponse` is not yet enabled, the only check is `validate_chunk_relevant` — which verifies that the `ChunkProductionKey` is relevant to this validator, not that the sender is the legitimate chunk producer. `ContractCodeResponse::V1` carries no identity field and no signature: [2](#0-1) 

The handler in `PartialWitnessActor` unconditionally decompresses and stores whatever code is received:

```rust
fn handle_contract_code_response(&self, response: ContractCodeResponse) -> Result<(), Error> {
    if !validate_contract_code_response(...).is_relevant() { return Ok(()); }
    let key = response.chunk_production_key().clone();
    let contracts = response.decompress_contracts()?;
    self.partial_witness_tracker.store_accessed_contract_codes(key, contracts)
}
``` [3](#0-2) 

The stored contracts are then merged directly into the witness's `base_state` trie values at assembly time:

```rust
let PartialState::TrieValues(values) = &mut witness.mut_main_state_transition().base_state;
values.extend(accessed_contracts.into_iter().map(|code| code.0.into()));
``` [4](#0-3) 

When `validate_chunk_state_witness_impl` replays the chunk using these trie values, the computed post-state root diverges from `state_witness.main_state_transition().post_state_root`, returning `Error::InvalidChunkStateWitness`: [5](#0-4) 

The chunk validator then never calls `send_chunk_endorsement_to_block_producers`, so no `ChunkEndorsement` is emitted for that chunk. [6](#0-5) 

---

### Impact Explanation

The corrupted protocol value is `ChunkStateWitness.main_state_transition.base_state` (the trie values used to replay chunk execution). Corruption causes the state root check to fail, suppressing `ChunkEndorsement` emission. Block producers collect endorsements via `collect_chunk_endorsements`; if the endorsement stake threshold is not met, the chunk cannot be included in a block: [7](#0-6) 

An attacker targeting all chunk validators for a shard can prevent any chunk from being endorsed, halting transaction confirmation for that shard. This matches the "Network not being able to confirm new transactions" impact class.

---

### Likelihood Explanation

- **Entry path**: Any peer connected to the NEAR P2P network can send a routed message. The routing layer verifies only the routing-layer signature of the forwarding peer, not the content of the payload. `ContractCodeResponse::V1` carries no identity, so any peer can forge one.
- **Targeting**: Chunk validator assignments are public (queryable from epoch manager). The `ChunkProductionKey` (shard_id, epoch_id, height_created) is predictable from the current chain head.
- **Timing**: `store_accessed_contract_codes` uses `create_if_not_exists: false`, so the malicious response must arrive after the first witness part creates a cache entry. The attacker monitors the network for `PartialEncodedStateWitness` messages (which are broadcast to validators) and immediately sends the poisoned response. The window spans the entire witness assembly period (hundreds of milliseconds to seconds).
- **Scope**: The attack is effective for every chunk in every epoch until `SignedContractCodeResponse` is activated.

---

### Recommendation

Remove the protocol-version gate around sender verification in `validate_contract_code_response`. The `validate_witness_contract_code_response_signature` check should be applied unconditionally for V2 messages, and V1 messages should be rejected outright (or the gate should be enforced at the network ingress layer so V1 messages are never delivered once the feature is active). Additionally, `handle_contract_code_response` should verify that the response corresponds to a pending `ContractCodeRequest` before storing any code. [8](#0-7) 

---

### Proof of Concept

1. Connect a non-validator peer to the NEAR network.
2. Observe `PartialEncodedStateWitness` messages on the wire to learn the current `ChunkProductionKey` (shard_id, epoch_id, height_created) and the target chunk validator account IDs.
3. Construct a `ContractCodeResponse::V1` with the correct `ChunkProductionKey` but with arbitrary (wrong) compressed contract bytes.
4. Send it as a routed message to each chunk validator for the target shard, signed with the attacker's own peer key (satisfies the routing-layer signature check).
5. The validators accept the message (no sender check), merge the wrong bytes into `base_state`, fail the state root check during witness validation, and emit no `ChunkEndorsement`.
6. The block producer cannot collect sufficient endorsement stake; the chunk is not included; all transactions in that chunk are not confirmed.

fn validate_witness_contract_code_response_signature(
    epoch_manager: &dyn EpochManagerAdapter,
    response: &ContractCodeResponse,
) -> Result<(), Error> {
    let Some(responder) = response.responder() else {
        return Err(Error::Other(
            "Unsigned contract code response in epoch where signature is required".to_owned(),
        ));
    };
    let key = response.chunk_production_key();
    let chunk_producers =
        epoch_manager.get_epoch_chunk_producers_for_shard(&key.epoch_id, key.shard_id)?;
    if !chunk_producers.contains(responder) {
        return Err(Error::Other(format!(
            "Contract code response responder {responder} is not a chunk producer for shard {} in epoch {:?}",
            key.shard_id, key.epoch_id,
        )));
    }
    let validator = epoch_manager.get_validator_by_account_id(&key.epoch_id, responder)?;
    if !response.verify_signature(validator.public_key()) {
        return Err(Error::Other("Invalid witness contract code response signature".to_owned()));
    }
    Ok(())
}

    pub fn collect_chunk_endorsements(
        &self,
        chunk_header: &ShardChunkHeader,
    ) -> Result<ChunkEndorsementsState, Error> {
        let shard_id = chunk_header.shard_id();
        let epoch_id =
            self.epoch_manager.get_epoch_id_from_prev_block(chunk_header.prev_block_hash())?;

        let height_created = chunk_header.height_created();
        let key = ChunkProductionKey { shard_id, epoch_id, height_created };

        let chunk_validator_assignments = self.epoch_manager.get_chunk_validator_assignments(
            &epoch_id,
            chunk_header.shard_id(),
            chunk_header.height_created(),
        )?;

        // Get the chunk_endorsements for the chunk from our cache.
        // Note that these chunk endorsements are already validated as part of process_chunk_endorsement.
        // We can safely rely on the following details
        //    1. The chunk endorsements are from valid chunk_validator for this chunk.
        //    2. The chunk endorsements signatures are valid.
        //    3. We still need to validate if the chunk_hash matches the chunk_header.chunk_hash()
        let mut cache = self.chunk_endorsements.lock();
        let entry = cache.get_or_insert(key, || HashMap::new());
        let validator_signatures = entry
            .into_iter()
            .filter(|(_, (chunk_hash, _))| chunk_hash == chunk_header.chunk_hash())
            .map(|(account_id, (_, signature))| (account_id, signature.clone()))
            .collect();

        // [perf] The lock here is held over the `compute_endorsement_state` call. If that constitutes an
        // expensive call AND we get into lock contention here, one needs to investigate into
        // cloning the account_id or using the RwLock.
        Ok(chunk_validator_assignments.compute_endorsement_state(validator_signatures))
    }
