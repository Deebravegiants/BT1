### Title
Single-participant unverified CKD response injection via `AppPublicKey` variant bypasses threshold invariant — (`crates/contract/src/lib.rs`)

### Summary

`respond_ckd` performs **no cryptographic verification** of the response when the pending request used `CKDAppPublicKey::AppPublicKey`. A single attested participant can call `respond_ckd` with arbitrary garbage `big_y`/`big_c` values, the contract accepts it unconditionally, and `resolve_yields_for` immediately drains the entire pending queue — fanning out the unverified garbage to every queued caller.

### Finding Description

In `respond_ckd`, the match on `request.app_public_key` has a completely empty arm for the `AppPublicKey` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

`ckd_output_check` enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, cryptographically binding the response to the `app_id` and the master BLS key. This check is **only invoked for `AppPublicKeyPV`**. [2](#0-1) 

After the match, `resolve_yields_for` is called unconditionally. It removes the entire queue entry and calls `env::promise_yield_resume` for every queued yield with the provided (unverified) response bytes — no threshold count, no leader check, no second opinion: [3](#0-2) 

The only guard before this point is `assert_caller_is_attested_participant_and_protocol_active`, which verifies only that the signer is an attested member of the participant set — a single-participant check, not a threshold check: [4](#0-3) 

### Impact Explanation

A single malicious attested participant (strictly below threshold) can:

1. Observe a `request_app_private_key` call with `AppPublicKey` variant on-chain.
2. Race the legitimate MPC computation by calling `respond_ckd` with `big_y = [1u8; 48]`, `big_c = [2u8; 48]` (or any bytes).
3. The contract accepts the call, drains the entire pending queue, and resumes every waiting yield with the garbage payload.
4. All callers receive a cryptographically invalid `(big_y, big_c)` pair from which no valid key can be derived.
5. The legitimate MPC response (when it arrives) hits `RequestNotFound` — the queue is already gone.

This breaks the invariant that a CKD output must be the product of a threshold computation bound to the `app_id` and the master BLS key. A single participant unilaterally resolves the request lifecycle with arbitrary content.

### Likelihood Explanation

The attacker needs only to be an attested participant — a role that is a prerequisite for operating in the MPC network at all. No collusion, no key leakage, no network-level access is required. The attack is a direct on-chain transaction. The existing unit test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` already demonstrates the exact exploit path with `[1u8; 48]` / `[2u8; 48]` garbage values and asserts success: [5](#0-4) 

### Recommendation

For the `AppPublicKey` variant, the contract cannot perform a pairing-based output check (no G2 component is available). The mitigations are:

1. **Deprecate `AppPublicKey` entirely** and require all callers to use `AppPublicKeyPV`, which carries the G2 component needed for on-chain verification.
2. **Or**, enforce a threshold-of-participants agreement before `resolve_yields_for` is called — require `t` distinct attested participants to submit matching `(request, response)` pairs before the yield is resumed.
3. **Or**, restrict `respond_ckd` to the designated protocol leader for each request, so at least the off-chain leader-election provides a single-point-of-accountability.

### Proof of Concept

The existing test at `crates/contract/src/lib.rs:3404` is already a working PoC. It:
- Submits `request_app_private_key` with `CKDAppPublicKey::AppPublicKey`
- Calls `respond_ckd` with `big_y = [1u8; 48]`, `big_c = [2u8; 48]`
- Asserts `Ok(_)` and that the pending queue is drained [6](#0-5)

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
