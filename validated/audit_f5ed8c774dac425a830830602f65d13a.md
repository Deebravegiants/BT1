### Title
Unrestricted `clean_invalid_attestations` Allows Any Caller to Prune Active Participant Attestations, Disrupting MPC Signing Operations â€” (File: `crates/contract/src/lib.rs`)

---

### Summary

`clean_invalid_attestations` is a public, permissionless state-mutating function that removes participant attestations from `stored_attestations`. Every other post-reshare cleanup function in the same contract is either `#[private]` (contract-self-call only) or explicitly checks that the caller is a current participant. This inconsistency lets any unprivileged NEAR account call `clean_invalid_attestations` at will, removing attestations of participants whose certificates have expired or whose image-hash reference has become stale. Because `is_caller_an_attested_participant` checks only for the *existence* of a stored attestation (not its validity), participants with a recently-expired attestation can still call `respond`, `vote_pk`, `vote_reshared`, etc. â€” until an attacker removes their entry. Once removed, those participants are locked out of every function gated by `assert_caller_is_attested_participant_and_protocol_active`, and a subsequent `verify_tee` call will treat them as invalid, potentially halting signing.

---

### Finding Description

`clean_invalid_attestations` is declared as a plain `pub fn` with no caller restriction: [1](#0-0) 

Its own doc-comment acknowledges the absence of access control: *"Callable by anyone while the protocol is in `Running`."*

Compare this with every other cleanup function spawned after resharing:

| Function | Access control |
|---|---|
| `clean_tee_status` | `#[private]` |
| `clean_foreign_chain_data` | `#[private]` |
| `remove_non_participant_tee_verifier_votes` | `#[private]` |
| `remove_non_participant_update_votes` | checks `predecessor == current_account_id` OR participant |
| **`clean_invalid_attestations`** | **none** | [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

The underlying `TeeState::clean_invalid_attestations` iterates `stored_attestations`, calls `reverify_participants` on each entry, and permanently removes any entry that fails re-verification (expired timestamp, stale docker-image hash, stale launcher hash, or stale OS measurement): [6](#0-5) 

The critical asymmetry is in `is_caller_an_attested_participant`, which gates every node-facing mutating method (`respond`, `vote_pk`, `vote_reshared`, `start_keygen_instance`, `start_reshare_instance`, `vote_abort_key_event_instance`). It checks only for the **existence** of a stored attestation entry â€” it does **not** re-verify whether the attestation is still within its validity window: [7](#0-6) 

Therefore, a participant whose attestation has just expired (or whose image hash has become stale) can still call `respond` and other node methods â€” the entry is still present in `stored_attestations`. An attacker who calls `clean_invalid_attestations` before the participant renews removes that entry, immediately locking the participant out of all attested-participant-gated functions.

---

### Impact Explanation

**Medium â€” participant-state and request-lifecycle manipulation breaking production safety invariants.**

1. An attacker calls `clean_invalid_attestations(max_scan: 4294967295)` (no upper-bound is enforced on `max_scan`).
2. All attestations that fail `re_verify` at that moment are permanently deleted from `stored_attestations`.
3. Affected participants can no longer call `respond`, `vote_pk`, `vote_reshared`, etc., because `is_caller_an_attested_participant` returns `AttestationNotFound`.
4. Pending signing requests that were waiting for those participants' `respond` calls time out.
5. If a participant or the contract subsequently calls `verify_tee`, `reverify_and_cleanup_participants` treats every participant with no stored attestation as invalid (the `find_node_id_by_tls_key` branch returns `false`), potentially triggering an unwanted resharing or setting `accept_requests = false`, halting all new signing requests. [8](#0-7) [9](#0-8) 

---

### Likelihood Explanation

**Medium.** Attestations expire on a fixed schedule and image-hash transitions (docker upgrade windows) create predictable windows during which some participants' attestations reference a hash that has just become stale. An attacker monitoring the chain can observe when `allowed_docker_image_hashes` changes or when block timestamps cross an attestation's expiry, then immediately call `clean_invalid_attestations` before affected participants submit renewals via `submit_participant_info`. No privileged access, no collusion, and no TEE attack is required â€” only a standard NEAR account and a single transaction.

---

### Recommendation

Apply the same access-control pattern used by every other cleanup function. Either:

1. Mark `clean_invalid_attestations` `#[private]` so it can only be invoked as a self-call (spawned via `Promise::new(env::current_account_id()).function_call(...)`), consistent with `clean_tee_status` and `clean_foreign_chain_data`; or
2. Require the caller to be a current participant or the contract itself, consistent with `remove_non_participant_update_votes`.

Additionally, enforce an upper bound on the `max_scan` parameter to prevent a single call from scanning the entire attestation store.

---

### Proof of Concept

```
// Any NEAR account can execute this:
near call <mpc_contract_account> clean_invalid_attestations \
  '{"max_scan": 4294967295}' \
  --accountId attacker.near

// Result:
// - All attestations that fail re_verify at this block are deleted from stored_attestations.
// - Affected participants can no longer call respond/vote_pk/vote_reshared.
// - Next verify_tee call by any participant treats deleted entries as invalid,
//   potentially triggering resharing or setting accept_requests = false.
```

The attacker times the call to the moment a docker-image upgrade deadline passes (observable on-chain via `allowed_docker_image_hashes`) or when a participant's attestation timestamp crosses its expiry, ensuring the maximum number of active participants are affected before they can renew.

            return Err(InvalidState::ProtocolStateNotRunning.into());

            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);

        match self.tee_state.reverify_and_cleanup_participants(
            current_params.participants(),
            tee_upgrade_deadline_duration,
        ) {
            TeeValidationResult::Full => {
                self.accept_requests = true;
                log!("All participants have an accepted Tee status");
                Ok(true)
            }
            TeeValidationResult::Partial {
                participants_with_valid_attestation,
            } => {
                let remaining = participants_with_valid_attestation.len();
                // Defense in depth: the surviving participant set must keep the full
                // threshold relation intact â€” the GovernanceThreshold must still sit
                // within its bounds for the smaller set (in particular it must not
                // exceed the remaining participant count or the upper cap) and must
                // remain at least every domain's ReconstructionThreshold (the kickout
                // keeps the existing per-domain thresholds). Otherwise we refuse and
                // wait for manual intervention.
                let max_reconstruction_threshold =
                    max_reconstruction_threshold(running_state.domains.domains());
                if let Err(err) = ThresholdParameters::validate_governance_against_reconstruction(
                    u64::try_from(remaining).expect("participant count fits in u64"),
                    current_params.threshold(),
                    max_reconstruction_threshold,
                ) {
                    log!(
                        "Kicking out participants with an invalid TEE status would break the threshold relation ({:?}); {} participants remain with a valid TEE status. This requires manual intervention. We will not accept new signature requests as a safety precaution.",
                        err,
                        remaining,
                    );
                    self.accept_requests = false;
                    return Ok(false);
                }

                // here, we set it to true, because at this point, we have at least `threshold`
                // number of participants with an accepted Tee status.
                self.accept_requests = true;

                // do we want to adjust the threshold?
                //let n_participants_new = new_participants.len();
                //let new_threshold = (3 * n_participants_new + 4) / 5; // minimum 60%
                //let new_threshold = new_threshold.max(2); // but also minimum 2
                let new_threshold = usize::try_from(current_params.threshold().value())
                    .expect("threshold value fits in usize");

                let threshold_parameters = ThresholdParameters::new(
                    participants_with_valid_attestation,
                    Threshold::new(new_threshold as u64),
                )
                .expect("Require valid threshold parameters"); // this should never happen.
                current_params.validate_incoming_proposal(&threshold_parameters)?;
                // This resharing only changes the participant set, so the
                // per-domain reconstruction-threshold updates map is empty.
                let proposed_parameters =
                    ProposedThresholdParameters::new(threshold_parameters, BTreeMap::new());
                let res = running_state.transition_to_resharing_no_checks(&proposed_parameters);
                if let Some(resharing) = res {
                    self.protocol_state = ProtocolContractState::Resharing(resharing);
                }

                Ok(true)
            }
        }

    pub fn clean_invalid_attestations(
        &mut self,
        tee_upgrade_deadline_duration: Duration,
        max_scan: usize,
    ) -> u32 {
        let has_invalid_attestation = |node_id: &NodeId| {
            !matches!(
                self.reverify_participants(node_id, tee_upgrade_deadline_duration),
                TeeQuoteStatus::Valid
            )
        };

        // Materialize candidates before any mutation to avoid iterator invalidation.
        let invalid_tls_keys: Vec<Ed25519PublicKey> = self
            .stored_attestations
            .iter()
            .take(max_scan)
            .filter(|(_, node_attestation)| has_invalid_attestation(&node_attestation.node_id))
            .map(|(tls_pk, _)| tls_pk.clone())
            .collect();

        let removed = u32::try_from(invalid_tls_keys.len())
            .expect("u32 should always be convertible from usize on wasm32");

        for tls_pk in invalid_tls_keys {
            self.stored_attestations.remove(&tls_pk);
        }
        removed
    }

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
