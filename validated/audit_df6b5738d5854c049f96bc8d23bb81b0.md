### Title
Unvalidated `ProposalInit.builder` Field Allows Malicious Proposer to Redirect All Block Transaction Fees to an Arbitrary Address — (`File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

The `builder` field of `ProposalInit` is a proposer-supplied address that is mapped directly to `sequencer_address` in `BlockInfo` and used as the fee recipient for every transaction in the block. The validator's `is_proposal_init_valid` function never checks this field against any locally-trusted reference value. Any validator-turn proposer can therefore set `builder` to an arbitrary address, causing all transaction fees in that block to be credited to that address and committing a block whose `PartialBlockHash` encodes the wrong sequencer address.

---

### Finding Description

**Root cause — `is_proposal_init_valid` omits `builder`**

`is_proposal_init_valid` validates `height`, `l1_da_mode`, `l2_gas_price_fri`, all four L1 gas prices (within a margin), `starknet_version`, `version_constant_commitment`, `timestamp` bounds, and `fee_proposal_fri` bounds. The `builder` field is never mentioned. [1](#0-0) 

`ProposalInitValidation` — the struct that carries the locally-trusted reference values — has no `builder` field at all: [2](#0-1) 

**Propagation — `builder` becomes `sequencer_address` in `BlockInfo`**

`convert_to_sn_api_block_info` maps `init.builder` directly to `sequencer_address` without any guard: [3](#0-2) 

This `BlockInfo` is passed to the batcher via `ProposeBlockInput` for both the proposer (`initiate_build`) and the validator (`initiate_validation`): [4](#0-3) 

**Fee theft — `sequencer_address` is the fee recipient**

The blockifier's `execute_fee_transfer` uses `block_context.block_info.sequencer_address` as the `recipient` calldata argument in the ERC-20 transfer call for every charged transaction: [5](#0-4) 

The Starknet OS Cairo program confirms the same pattern: [6](#0-5) 

**Wrong block hash — `sequencer` is hashed into `PartialBlockHashComponents`**

`PartialBlockHashComponents::new` stores `block_info.sequencer_address` as the `sequencer` field: [7](#0-6) 

`calculate_block_hash` chains `sequencer` into the Poseidon hash: [8](#0-7) 

**Commitment check does not catch the manipulation**

The validator computes `batcher_block_commitment` from the batcher's output (which already used the attacker-supplied `builder`) and compares it to `received_fin.proposal_commitment`. Because both sides used the same `builder`, the comparison passes: [9](#0-8) 

---

### Impact Explanation

Every account transaction in the block pays its fee to `builder` (the attacker-controlled address) instead of the legitimate sequencer. The state diff records the fee token balance increase at the wrong address. The `PartialBlockHash` — and therefore the `ProposalCommitment` — encodes the wrong `sequencer_address`, so the committed block hash is permanently wrong. This matches two critical impact categories:

- **Critical — Incorrect fee/balance with economic impact**: all transaction fees in the block are stolen.
- **Critical — Wrong state/receipt/storage value**: the fee token storage diff and the block hash both reflect the attacker's address.

---

### Likelihood Explanation

Any validator whose turn it is to propose can trigger this with zero additional privilege. The Tendermint round-robin means every validator eventually proposes. No special key, no governance vote, no timelock — just a modified `ProposalInit.builder` field in the first message of the proposal stream. All other validators accept the proposal because `is_proposal_init_valid` never checks `builder`.

---

### Recommendation

Add `builder` (the expected sequencer address) to `ProposalInitValidation` and enforce it inside `is_proposal_init_valid`:

```rust
// In ProposalInitValidation:
pub builder: ContractAddress,

// In is_proposal_init_valid:
if init_proposed.builder != proposal_init_validation.builder {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.builder, init_proposed.builder
        ),
    ));
}
```

The reference value for `builder` should be derived from the node's own configuration (the sequencer contract address for the current height), not from the incoming `ProposalInit`.

---

### Proof of Concept

1. Attacker controls validator node `V` and waits for its proposer turn at height `H`.
2. `V` constructs `ProposalInit` with `builder = attacker_address` (any address the attacker controls).
3. `V` streams `ProposalPart::Init(init)` to all peers.
4. Each peer calls `is_proposal_init_valid` — passes, because `builder` is not checked.
5. Each peer calls `initiate_validation` → `convert_to_sn_api_block_info(init)` → `sequencer_address = attacker_address`.
6. The batcher executes all transactions with `sequencer_address = attacker_address`; every `execute_fee_transfer` sends fees to `attacker_address`.
7. The batcher computes `PartialBlockHashComponents { sequencer: attacker_address, … }` → `partial_block_hash`.
8. `V` sends `ProposalFin { proposal_commitment: Poseidon(partial_block_hash, fee_proposal), … }`.
9. Each validator computes the same `batcher_block_commitment` (same `builder` was used) → `built_block == received_fin.proposal_commitment` → proposal accepted.
10. Block is committed: all fees for height `H` are in `attacker_address`; the block hash encodes the wrong sequencer.
