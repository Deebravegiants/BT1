### Title
Single Attested Participant Can Deliver Fabricated CKD Response for `AppPublicKey` Requests — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract omits cryptographic verification of the CKD response when the request uses the `AppPublicKey` (privately verifiable) variant. A single malicious attested participant — strictly below the signing threshold — can call `respond_ckd` with a fabricated `CKDResponse`, which the contract accepts without any check, permanently consuming the pending request and delivering garbage to the requester. The analog to the external report is: just as `modifyPosition()` was never called after liquidation (leaving debt state stale and exploitable), `ckd_output_check` is never called for `AppPublicKey` responses (leaving the response correctness invariant unenforced and exploitable by a single participant).

---

### Finding Description

In `respond_ckd`, the contract branches on the request's `app_public_key` variant before calling `resolve_yields_for`:

```rust
// crates/contract/src/lib.rs:675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}

pending_requests::resolve_yields_for(          // always reached
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
```

For `AppPublicKeyPV` requests the contract calls `ckd_output_check`, which verifies the BLS pairing relationship between `(big_y, big_c)` and the master public key. For `AppPublicKey` requests the arm is empty — any `CKDResponse` is serialised and delivered to the waiting yield without any mathematical check.

`resolve_yields_for` then removes the map entry unconditionally:

```rust
// crates/contract/src/pending_requests.rs:74-87
let resumed = requests
    .remove(request)          // entry gone after first call
    .unwrap_or_default()
    ...
```

The attack path for a single Byzantine participant `P_evil`:

1. `P_evil` observes a pending `CKDRequest` keyed on `AppPublicKey` in `pending_ckd_requests`.
2. `P_evil` immediately calls `respond_ckd` with a fabricated `CKDResponse` (e.g. all-zero G1 points). No MPC computation is required.
3. The contract passes the `assert_caller_is_attested_participant_and_protocol_active` check (participant is legitimate), skips the empty `AppPublicKey` arm, and calls `resolve_yields_for`.
4. The pending request is removed and the fabricated bytes are delivered to the user's yield.
5. Honest nodes subsequently attempt `respond_ckd` for the same request and receive `RequestNotFound` — the slot is already gone.
6. The user decrypts garbage with their private key `a` and cannot re-request under the same parameters because the yield has already resolved.

The root cause is structurally identical to the external report: a critical update (`ckd_output_check`) is absent from the code path that finalises the operation, so the operation completes with incorrect state accepted as valid.

---

### Impact Explanation

**Medium — request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants.**

Any pending CKD request submitted with the `AppPublicKey` variant is permanently corruptible by a single malicious attested participant. The user receives a wrong encrypted key, cannot detect the corruption until after decryption (the contract has no way to signal it), and cannot re-request because the yield slot has been consumed. The confidential key derivation safety invariant — that the delivered `(big_y, big_c)` pair is a valid encryption of the MPC-derived secret — is broken for the entire `AppPublicKey` request class without any threshold collusion.

---

### Likelihood Explanation

**Medium.** The attacker must be a single attested participant in the production MPC network. No threshold collusion is required. The attack window is the interval between a user submitting `request_app_private_key` and the honest nodes completing the MPC computation and calling `respond_ckd`. Because the attacker needs zero computation (they fabricate the response), they have a structural time advantage over honest nodes who must run the full BLS CKD protocol. The `AppPublicKey` variant is the legacy path still accepted by the contract and used by existing integrations.

---

### Recommendation

Apply `ckd_output_check` unconditionally, independent of the `app_public_key` variant. For `AppPublicKey` requests the check can be reformulated using only the G1 component (`pk1 = a·G1`) that is already present in the request, verifying the pairing relationship `e(big_c - big_s, G2) == e(big_y, master_pk_g2)` where `big_s` is derived from the master key and `app_id`. Alternatively, deprecate `AppPublicKey` and require all new requests to use `AppPublicKeyPV`, which already carries the G2 component needed for on-chain verification.

### Proof of Concept

```rust
// Attacker is an attested participant; user submitted request_app_private_key
// with AppPublicKey (privately verifiable variant).

// Step 1 – user's pending request is visible on-chain.
let ckd_request = CKDRequest::new(
    CKDAppPublicKey::AppPublicKey(user_app_pk),
    domain_id,
    &user_account_id,
    &derivation_path,
);
assert!(contract.get_pending_ckd_request(&ckd_request).is_some());

// Step 2 – attacker fabricates a response (no MPC computation needed).
let fabricated = CKDResponse {
    big_y: dtos::Bls12381G1PublicKey([0u8; 48]),
    big_c: dtos::Bls12381G1PublicKey([0u8; 48]),
};

// Step 3 – attacker calls respond_ckd; contract accepts without verification.
with_attacker_as_attested_participant_context(&contract);
contract
    .respond_ckd(ckd_request.clone(), fabricated)
    .expect("accepted: no ckd_output_check for AppPublicKey");

// Step 4 – pending request is gone; honest nodes get RequestNotFound.
assert!(contract.get_pending_ckd_request(&ckd_request).is_none());

// Step 5 – user receives garbage bytes; decryption with private key `a` yields
// a random point, not the MPC-derived secret.
```
