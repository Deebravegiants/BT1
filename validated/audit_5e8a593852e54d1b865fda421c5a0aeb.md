Audit Report

## Title
Stale `rsETHPrice` at `initiateWithdrawal` Permanently Caps User Payout, Causing Loss of Unclaimed Yield - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`initiateWithdrawal` reads `lrtOracle.rsETHPrice()` via `getExpectedAssetAmount` without first calling `updateRSETHPrice()`, storing a stale (lower) `expectedAssetAmount` in the withdrawal request. When `unlockQueue` later processes the request with the refreshed price, `_calculatePayoutAmount` returns `min(stale_expectedAssetAmount, current_return)`, permanently capping the user's payout at the stale value. Because no `cancelWithdrawal` function exists and the rsETH is already transferred to the contract at initiation, the user has no recovery path and permanently forfeits the yield that accrued between the last price update and their withdrawal initiation.

## Finding Description
`initiateWithdrawal` calls `getExpectedAssetAmount` at line 168, which reads `lrtOracle.rsETHPrice()` directly from storage at line 593 without triggering a price refresh:

```solidity
// LRTWithdrawalManager.sol:593
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

The result is written permanently into storage at lines 751–753:

```solidity
withdrawalRequests[requestId] = WithdrawalRequest({
    rsETHUnstaked: rsETHUnstaked, expectedAssetAmount: expectedAssetAmount, withdrawalStartBlock: block.number
});
```

`rsETHPrice` in `LRTOracle` is a stored variable updated only when `updateRSETHPrice()` / `_updateRsETHPrice()` is explicitly called (lines 87–89, 313). Because rsETH continuously accrues restaking yield, the stored price drifts below the true current value between keeper updates.

When the operator later calls `unlockQueue`, `_unlockWithdrawalRequests` invokes `_calculatePayoutAmount` at line 798:

```solidity
// LRTWithdrawalManager.sol:833–834
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
```

Because `expectedAssetAmount` (stale, lower) < `currentReturn` (fresh, higher), the payout is capped at the stale value. This result is written back to storage at line 804 (`request.expectedAssetAmount = payoutAmount`) and is the exact amount transferred to the user at line 734 (`_transferAsset(asset, user, request.expectedAssetAmount)`).

The `assetsCommitted` accounting at line 802 releases the original committed amount, meaning the difference between the fresh and stale values remains in the protocol's asset pool — redistributed to remaining rsETH holders — rather than going to the withdrawing user.

`updateRSETHPrice()` is `public` but can revert for non-manager callers when the price increase exceeds `pricePercentageLimit` (lines 260–265 of `LRTOracle.sol`), meaning a user cannot always self-rescue by calling it before initiating withdrawal. No `cancelWithdrawal` function exists anywhere in the contract, confirmed by code search.

## Impact Explanation
The withdrawing user permanently loses the yield that accrued between the last `updateRSETHPrice()` call and their `initiateWithdrawal` call. Their rsETH is burned at the stale (lower) price, and the corresponding yield is redistributed to remaining rsETH holders. This is a permanent, unrecoverable transfer of unclaimed yield away from the user — matching the **High: Theft of unclaimed yield** impact class. The submitter's Low classification understates the impact; the PoC demonstrates a concrete, permanent 5 stETH loss of accrued yield with no recourse.

## Likelihood Explanation
Any rsETH holder who calls `initiateWithdrawal` during a price-stale window is affected. The window exists between every pair of consecutive `updateRSETHPrice()` calls, which in practice can span hours to days. The path is fully permissionless — no special role or attacker is required. The scenario is worsened when `pricePercentageLimit` is set and the accumulated yield exceeds it, because non-manager callers cannot self-rescue by calling `updateRSETHPrice()` before initiating. This is a routine, repeatable condition for any ordinary withdrawer.

## Recommendation
Call `updateRSETHPrice()` (or inline `_updateRsETHPrice()`) at the top of `initiateWithdrawal` before `getExpectedAssetAmount` is invoked. To avoid blocking withdrawals if the public `updateRSETHPrice()` reverts due to `pricePercentageLimit`, wrap the call in a try/catch or use a dedicated internal helper that skips the threshold revert for the withdrawal path. Alternatively, redesign `_calculatePayoutAmount` to use `max(expectedAssetAmount, currentReturn)` so that price appreciation after initiation benefits the user rather than the protocol.

## Proof of Concept
1. `rsETHPrice` in `LRTOracle` is `1.05e18` (stale; true value is `1.10e18` after recent yield accrual). `pricePercentageLimit` is set such that the ~4.8% increase exceeds the threshold, so non-managers cannot call `updateRSETHPrice()`.
2. User calls `initiateWithdrawal(stETH, 100e18, "")`. rsETH is transferred to the contract.
3. `getExpectedAssetAmount` computes `100e18 × 1.05e18 / 1e18 = 105e18` and stores it as `expectedAssetAmount` (line 593, 751–753).
4. Manager calls `updateRSETHPriceAsManager()` — price updates to `1.10e18`.
5. Operator calls `unlockQueue`; `_calculatePayoutAmount` computes `currentReturn = 100e18 × 1.10e18 / 1e18 = 110e18` (line 833).
6. `min(105e18, 110e18) = 105e18` is written back to `request.expectedAssetAmount` (line 804).
7. User calls `completeWithdrawal`; receives 105 stETH (line 734) instead of 110 stETH — permanently losing 5 stETH of accrued yield with no recourse. The 5 stETH difference remains in the protocol asset pool.

**Foundry fork test plan:** Deploy against a mainnet fork. Manipulate `rsETHPrice` storage slot to a value below the current computed price. Call `initiateWithdrawal` as a normal user. Advance blocks past `withdrawalDelayBlocks`. Call `updateRSETHPrice()` as manager to refresh price. Call `unlockQueue` as operator. Assert `withdrawalRequests[requestId].expectedAssetAmount` equals the stale-price amount, not the fresh-price amount. Call `completeWithdrawal` and assert the user received the lesser amount.
