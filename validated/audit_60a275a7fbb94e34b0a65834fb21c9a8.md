### Title
Caller-Agnostic Request Key in `verify_foreign_transaction` Enables Queue-Saturation Attack That Blocks Bridge Transaction Verification - (File: `crates/contract/src/lib.rs`)

### Summary

The `verify_foreign_transaction` function stores pending requests under a key that contains no caller identity. Any unprivileged account can pre-fill the per-key yield queue to its hard cap (`MAX_PENDING_REQUEST_FAN_OUT = 128`) for any observable foreign transaction ID, causing every subsequent legitimate `verify_foreign_transaction` call for that same transaction to panic and revert. This permanently blocks the bridge verification lifecycle for a targeted transaction until the attacker's queue is drained, and the attacker can repeat the fill immediately.

### Finding Description

`verify_foreign_transaction` in `MpcContract` enqueues a yield-resume promise and stores the caller's slot in `pending_verify_foreign_tx_requests`, a `LookupMap<VerifyForeignTransactionRequest, Vec<YieldIndex>>`. [1](#0-0) 

The map key type is `VerifyForeignTransactionRequest`: [2](#0-1) 

It contains only `{request, domain_id, payload_version}` — **no caller account ID**. This is by design for the fan-out feature (multiple callers sharing one MPC computation), as confirmed by the test comment: [3](#0-2) 

The queue is bounded by a hard cap enforced in `push_pending_yield`: [4](#0-3) 

When the cap is reached, the function calls `env::panic_str`, reverting the entire transaction. Because the key is caller-agnostic, **any account** can fill the 128-slot queue for any specific `(tx_id, domain_id, payload_version)` triple before the legitimate bridge service submits its own request.

By contrast, the `sign()` path is not vulnerable: `SignatureRequest::new` folds the caller's `predecessor_id` into a `tweak` via `derive_tweak`, making each caller's key unique: [5](#0-4) [6](#0-5) 

`verify_foreign_transaction` has no equivalent caller-binding step.

### Impact Explanation

`verify_foreign_transaction` is the primary mechanism for the Omnibridge inbound flow: a bridge service submits a foreign-chain transaction ID and the MPC network signs an attestation that the transaction finalized. If the queue for a specific `tx_id` is saturated, the bridge service's call panics and the inbound bridge transaction cannot be attested. Funds deposited on the foreign chain remain locked until the attacker's 128 slots are drained and the bridge service can retry — at which point the attacker can immediately re-saturate the queue. This is a targeted, persistent request-lifecycle manipulation that breaks the bridge's production safety invariant (every finalized inbound transaction must eventually be attestable).

### Likelihood Explanation

Foreign transaction IDs are publicly visible on-chain the moment a transaction is broadcast. An attacker monitoring the foreign chain can submit 128 `verify_foreign_transaction` calls in a single NEAR block batch before any bridge service reacts. The cost is 128 × `MINIMUM_SIGN_REQUEST_DEPOSIT` (described in the contract as "a small deposit"). The attacker's 128 requests are processed normally by MPC nodes and the attacker receives 128 copies of the same valid signature — the deposit is the only economic cost, and it is bounded and predictable.

### Recommendation

Bind the queue key to the caller's account ID, mirroring the `sign()` path. The simplest fix is to include `env::predecessor_account_id()` in the stored request key for `verify_foreign_transaction`, so each caller occupies its own queue slot and cannot saturate another caller's slot. If cross-caller fan-out is still desired, implement a per-`(caller, request)` sub-key with an independent cap, and aggregate fan-out only at respond time.

### Proof of Concept

1. Attacker observes Bitcoin `tx_id = X` finalize on the Bitcoin network.
2. Attacker submits 128 calls to `verify_foreign_transaction` with `{request: Bitcoin(tx_id=X), domain_id: D, payload_version: V1}`, each with the required deposit.
3. `pending_verify_foreign_tx_requests[VerifyForeignTransactionRequest{...}]` reaches length 128.
4. Bridge service submits `verify_foreign_transaction` for the same `tx_id = X`.
5. `push_pending_yield` evaluates `queue.len() >= 128` → `env::panic_str("Pending-request queue is full …")`. [7](#0-6) 

6. Bridge service's NEAR transaction reverts; the inbound bridge attestation is blocked.
7. MPC nodes eventually drain the attacker's 128 slots; attacker immediately re-submits 128 requests, maintaining the DoS indefinitely at bounded cost.

```rust
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
```
