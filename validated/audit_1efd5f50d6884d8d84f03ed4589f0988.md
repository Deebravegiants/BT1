### Title
Unvalidated CKD Response for `AppPublicKey` Requests Allows Single Byzantine Node to Deliver Forged Key Material - (File: `crates/contract/src/lib.rs`)

### Summary
The `respond_ckd` contract method skips all cryptographic output validation when a user's `request_app_private_key` call uses the `AppPublicKey` variant. A single attested-but-malicious MPC participant (strictly below the signing threshold) can race to call `respond_ckd` with an arbitrary, self-chosen `CKDResponse`, and the contract will accept and deliver it to the user without any check. The user receives key material whose discrete log is known to the attacker, enabling decryption of user-encrypted data or theft of assets controlled by that key.

### Finding Description

`respond_ckd` in `crates/contract/src/lib.rs` branches on the variant of `app_public_key` stored in the request:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ŌåÉ no validation at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

When the `AppPublicKeyPV` variant is used, `ckd_output_check` enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) ┬Ę e(hash_point, public_key)`, which proves the response was produced using the MPC master secret and the correct `app_id`. This check is cryptographically binding: only a party that knows the master secret (i.e., the threshold-many cooperating nodes) can produce a passing response. [2](#0-1) 

When the `AppPublicKey` variant is used, the branch body is empty ŌĆö the contract unconditionally proceeds to `resolve_yields_for`, delivering whatever `big_c` and `big_y` the caller supplied. [3](#0-2) 

`resolve_yields_for` drains the entire pending-yield queue for the request in one call, so the first `respond_ckd` invocation that matches the request key wins and permanently resolves all waiting callers. [4](#0-3) 

The `AppPublicKey` variant is the legacy/default format accepted by `CKDRequestArgs` deserialization (a bare G1 key string maps to `AppPublicKey`), making it the path most existing integrations will use. [5](#0-4) 

### Impact Explanation

A single Byzantine MPC participant (below the signing threshold) can:

1. Observe a pending `CKDRequest` with `AppPublicKey` on-chain.
2. Construct an arbitrary `CKDResponse` ŌĆö e.g., `big_y = r┬ĘG1`, `big_c = r┬Ęapp_pk1` for a self-chosen scalar `r` ŌĆö whose discrete log is fully known to the attacker.
3. Call `respond_ckd(request, forged_response)` before the honest leader submits the legitimate result.
4. The contract accepts the forged response with no error, resolves the yield, and returns the attacker-controlled key to the user.

The user receives key material they believe is their confidentially derived private key but which the attacker can fully reconstruct. Any data encrypted to that key is readable by the attacker; any assets gated by that key are stealable. This constitutes **unauthorized confidential key derivation output without the required participant authorization** ŌĆö a Critical impact under the allowed scope.

### Likelihood Explanation

- The attacker need only be one attested MPC participant, which is the minimum possible collusion level (strictly below threshold).
- The `AppPublicKey` variant is the default deserialization path, so the majority of real user requests are vulnerable.
- The attack is a simple front-run: monitor the NEAR chain for `request_app_private_key` receipts and submit `respond_ckd` with a forged response in the next block. No special cryptographic capability is required beyond being an enrolled node.

### Recommendation

Remove the empty `AppPublicKey` branch and require all CKD responses to be validated. The simplest fix is to mandate that callers always supply `AppPublicKeyPV` (which carries the G2 component needed for the pairing check), and reject `AppPublicKey` requests at the `request_app_private_key` entry point. If the `AppPublicKey` variant must be retained for backward compatibility, the contract must either (a) derive an equivalent verifiable commitment from the G1 key alone, or (b) require the responding node to include a zero-knowledge proof of correct computation that the contract can verify without the G2 component.

### Proof of Concept

1. **User submits request** (legacy `AppPublicKey` format):
   ```json
   { "derivation_path": "my/path",
     "app_public_key": "<base64-G1-point>",
     "domain_id": 0 }
   ```
   Contract stores `CKDRequest { app_public_key: AppPublicKey(user_g1), app_id: H(alice, "my/path"), domain_id: 0 }`.

2. **Attacker (single enrolled node) constructs forged response**: Choose scalar `r`; set `big_y = r┬ĘG1`, `big_c = r┬Ęuser_g1`. The attacker knows `r`, so they know the "private key" the user will derive.

3. **Attacker calls `respond_ckd`** with the correct `request` key and the forged `CKDResponse`. The `AppPublicKey` branch executes the empty arm ŌĆö no `ckd_output_check` is called.

4. **Contract resolves the yield** via `resolve_yields_for`, delivering `{ big_y, big_c }` to the user.

5. **User decrypts** using the received key material, believing it is their private MPC-derived key. The attacker, knowing `r`, can independently compute the same key and decrypt all user data or drain any assets controlled by it.

The honest nodes' legitimate response, arriving later, hits `RequestNotFound` and is discarded ŌĆö the attacker's forged answer is already final.

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
