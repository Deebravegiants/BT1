Audit Report

## Title
Uninitialized `rsETHPrice` Causes Division-by-Zero on All Deposit Calls Until `updateRSETHPrice()` Is Invoked - (File: contracts/LRTOracle.sol)

## Summary
`LRTOracle.initialize()` never assigns `rsETHPrice`, leaving it at the Solidity default of `0`. Every deposit call in `LRTDepositPool` reaches `getRsETHAmountToMint()`, which divides by `lrtOracle.rsETHPrice()`. Until `updateRSETHPrice()` is explicitly called, all deposits revert with a division-by-zero panic. No user funds are lost because the revert prevents any state change, but the protocol cannot accept deposits during this window.

## Finding Description
`LRTOracle.initialize()` sets only `lrtConfig` and emits an event; `rsETHPrice` is never written:

```solidity
// contracts/LRTOracle.sol L64-68
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    lrtConfig = ILRTConfig(lrtConfigAddr);
    emit UpdatedLRTConfig(lrtConfigAddr);
}
```

`rsETHPrice` is a plain storage variable that defaults to `0`:

```solidity
// contracts/LRTOracle.sol L28
uint256 public override rsETHPrice;
```

`LRTDepositPool.getRsETHAmountToMint()` performs an unconditional division by `lrtOracle.rsETHPrice()`:

```solidity
// contracts/LRTDepositPool.sol L519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

This function is called from `_beforeDeposit()` (L665), which is called unconditionally by both `depositETH()` (L87) and `depositAsset()` (L111). When `rsETHPrice == 0`, the EVM raises a division-by-zero panic and the entire transaction reverts.

`_updateRsETHPrice()` does set `rsETHPrice = 1 ether` for the zero-supply case (L218-221), but only when explicitly triggered. `updateRSETHPrice()` is `public whenNotPaused` (L87-89), so any caller — including the depositor who just encountered the revert — can invoke it to seed the price.

Existing checks (`minAmountToDeposit`, deposit limit checks) all execute before `getRsETHAmountToMint()` and do not guard against a zero `rsETHPrice`.

## Impact Explanation
Between contract deployment (or any proxy upgrade that resets storage) and the first call to `updateRSETHPrice()`, every invocation of `depositETH()` or `depositAsset()` reverts. No user funds are at risk of loss because the revert prevents any state change. The protocol fails to deliver its core promised service — accepting deposits — during this window.

**Impact: Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
The window is present on every fresh deployment or storage-resetting upgrade. Because `updateRSETHPrice()` is permissionless, any user who encounters the revert can self-remediate immediately by calling it directly. The window is short in practice but is a real, reachable failure mode for any depositor who acts before the price is seeded. No special privileges or attacker capability are required to trigger the revert — a normal depositor calling `depositETH()` or `depositAsset()` is sufficient.

## Recommendation
Initialize `rsETHPrice` (and `highestRsethPrice`) to `1 ether` inside `LRTOracle.initialize()`, mirroring the logic already present in `_updateRsETHPrice()` for the zero-supply case:

```solidity
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    lrtConfig = ILRTConfig(lrtConfigAddr);
+   rsETHPrice = 1 ether;
+   highestRsethPrice = 1 ether;
    emit UpdatedLRTConfig(lrtConfigAddr);
}
```

## Proof of Concept
1. Deploy `LRTOracle` proxy and call `initialize(lrtConfigAddr)`. `rsETHPrice` is `0`.
2. Do **not** call `updateRSETHPrice()`.
3. Call `LRTDepositPool.depositETH{value: 1 ether}(0, "")`.
4. Execution reaches `getRsETHAmountToMint()` → `(1e18 * assetPrice) / 0` → EVM division-by-zero panic → revert.
5. Call `updateRSETHPrice()` (permissionless, no role required). `rsETHPrice` is now `1 ether`.
6. Repeat step 3 — deposit succeeds.

A Foundry test can reproduce this by deploying the proxy, skipping `updateRSETHPrice()`, and asserting that `depositETH` reverts with a `Panic(0x12)` error, then asserting it succeeds after `updateRSETHPrice()` is called.
