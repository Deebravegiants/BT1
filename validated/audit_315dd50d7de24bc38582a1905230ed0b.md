### Title
`directDeposits` Omitted from Preservation-of-Value Check Enables Unbounded ADA Creation — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, the new `directDeposits` field (a map from registered account addresses to coin amounts) is applied to account balances in the `ENTITIES` and `SUBENTITIES` rules, but is **never included in either the consumed or produced side** of the preservation-of-value check (`consumed == produced`). An unprivileged transaction sender can include arbitrarily large `directDeposits` in a balanced transaction (where UTxO inputs equal outputs + fee + cert deposits) and the ledger will accept it, crediting the specified accounts with ADA that was never deducted from any pot. This is unbounded ADA creation.

---

### Finding Description

**Vulnerability class:** Accounting bypass — a new accounting field (`directDeposits`) is applied to ledger state but is absent from both sides of the preservation-of-value equation, making it invisible to the balance check.

**Analog to the external report:** In Trader Joe, certain addresses were exempt from `_cacheFees` (debt initialization), so their debt remained at zero while `accTokenPerShare` grew, letting them collect all historical fees for free. Here, `directDeposits` are exempt from the `consumed`/`produced` accounting equation entirely, so they are applied to account balances without any corresponding deduction from any pot.

**Root cause — `getConsumedDijkstraValue` (consumed side):** [1](#0-0) 

`getConsumedDijkstraValue` aggregates UTxO inputs + withdrawals + refunds for the top-level body and all sub-transaction bodies via `getConsumedMaryValue`. `directDeposits` are not read from the tx body here.

**Root cause — `dijkstraProducedValue` (produced side):** [2](#0-1) 

`dijkstraProducedValue` sums outputs + fee + cert deposits + treasury donation for the top-level body, then adds the same for each sub-transaction via `dijkstraSubTxProducedValue`. `directDeposits` are absent. [3](#0-2) 

`dijkstraSubTxProducedValue` also omits `directDeposits`.

**The preservation check that passes silently:** [4](#0-3) 

`validateValueNotConservedUTxO` calls `consumed` → `getConsumedValue` = `getConsumedDijkstraValue`, and `getProducedValue` = `getProducedDijkstraValue`. Neither path touches `directDeposits`, so the check passes for any balanced transaction regardless of how large the direct deposits are.

**The state mutation that creates ADA:** [5](#0-4) 

After the preservation check has already passed, `applyDirectDeposits` unconditionally adds the specified coin amounts to the target account balances. [6](#0-5) 

`applyDirectDeposits` calls `updateAccountBalances` with an `addCompactCoin` updater — no deduction from any other pot occurs.

The same omission exists in `SUBENTITIES` for sub-transactions: [7](#0-6) 

The only validation applied to `directDeposits` is a network-ID check and a registration check: [8](#0-7) [9](#0-8) 

Neither check verifies that the deposited coin is funded by the transaction's UTxO inputs.

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An attacker can submit a Dijkstra-era transaction that is perfectly balanced (UTxO inputs = outputs + fee + cert deposits) and simultaneously carries `directDeposits` of any magnitude. The preservation check passes, and the specified account balances are inflated by the deposit amounts. The attacker then withdraws the inflated balance via a normal withdrawal transaction. This is unbounded, permissionless ADA minting with no cost beyond the transaction fee.

---

### Likelihood Explanation

Any unprivileged user who can submit a Dijkstra-era transaction can trigger this. No special role, key, or governance threshold is required. The only prerequisite is that the target account address is registered (which the attacker can do themselves for the cost of the key deposit). Once the Dijkstra era is active on mainnet, this is trivially exploitable.

---

### Recommendation

Include `directDeposits` in the **produced** side of the preservation-of-value equation, analogously to how certificate deposits are included. Concretely, `dijkstraProducedValue` and `dijkstraSubTxProducedValue` should add `inject (fold (unDirectDeposits (txBody ^. directDepositsTxBodyL)))` to their result. This ensures that the ADA credited to account balances via direct deposits is deducted from the transaction's UTxO inputs, preserving the total ADA supply.

---

### Proof of Concept

```
1. Register stake credential C (pay keyDeposit D_k).
2. Construct a Dijkstra top-level transaction T:
     inputs  = { utxo_in }   -- value V
     outputs = { utxo_out }  -- value V - fee
     fee     = fee
     directDeposits = { AccountAddress(C) -> N }   -- N = 1_000_000_000_000 lovelace
3. Submit T.
   - validateValueNotConservedUTxO: consumed = V, produced = (V - fee) + fee = V  ✓
   - directDepositsMissingAccounts: C is registered  ✓
   - applyDirectDeposits: account balance of C += N
4. Submit withdrawal transaction W:
     withdrawals = { AccountAddress(C) -> N }
   - W moves N lovelace from the account balance into a UTxO output.
5. Net result: N lovelace created from nothing.
   Repeat steps 2–4 to drain the entire ADA supply.
```

```haskell
getConsumedDijkstraValue pp lookupStakingDeposit lookupDRepDeposit utxo txBody =
  withBothTxLevels
    txBody
    ( \topTxBody ->
        txBodyConsumedValue topTxBody <> subTransactionsConsumedValue topTxBody

    txBodyConsumedValue
```

```haskell
directDepositsMissingAccounts ::
  EraAccounts era =>
  DirectDeposits ->
  Accounts era ->
  Maybe DirectDeposits
directDepositsMissingAccounts (DirectDeposits dds) accounts
  | Map.foldrWithKey' checkRegistered True dds = Nothing
  | otherwise = Just $ DirectDeposits $ Map.foldrWithKey' collectMissing Map.empty dds
  where
    isRegistered (AccountAddress _ (AccountId credential)) =
      isAccountRegistered credential accounts
    checkRegistered addr _ acc = acc && isRegistered addr
    collectMissing addr amount acc
      | isRegistered addr = acc
      | otherwise = Map.insert addr amount acc
```
