### Title
Proposal Deposits Permanently Lost When `do_update` Clears Non-Winning Entries — (`File: crates/contract/src/update.rs`)

### Summary

When a contract update reaches threshold and `do_update` is called, it clears **all** pending proposals and votes but never refunds the storage-staking deposits attached by proposers of the non-winning entries. Those NEAR tokens are permanently stranded in the contract's balance with no recovery path.

### Finding Description

`propose_update` requires a deposit proportional to the bytes stored: [1](#0-0) 

The required amount is computed as `env::storage_byte_cost() * bytes_used`, where `bytes_used` includes the full contract binary size plus per-vote overhead: [2](#0-1) 

The excess above `required` is refunded immediately, but the `required` portion is retained by the contract: [3](#0-2) 

The `UpdateEntry` struct stores only `update` and `bytes_used` — **not** the proposer's `AccountId` or the deposit amount: [4](#0-3) 

When `do_update` executes the winning proposal it calls `self.entries.clear()` and `self.vote_by_participant.clear()`, wiping every competing proposal: [5](#0-4) 

Because the proposer identity and deposit amount are not stored in `UpdateEntry`, there is no information available to issue refunds. The freed storage bytes return to the contract's own balance, but the original depositors receive nothing. No refund path exists anywhere in the codebase for this scenario.

### Impact Explanation

Every participant who proposed a competing update loses their storage-staking deposit permanently. For a full contract binary (~1.5 MB), the required deposit is on the order of 15 NEAR (at ~1 NEAR per 100 KB). With multiple participants each proposing their own update — a realistic governance scenario — the aggregate loss can be substantial. The NEAR accumulates silently in the contract's balance with no mechanism to recover it. This breaks the accounting invariant that a proposer's deposit is returned when their proposal is superseded or invalidated.

**Allowed impact matched**: *Medium — balance/accounting invariant broken without relying on network-level DoS or operator misconfiguration.*

### Likelihood Explanation

The trigger is the normal governance flow: multiple participants propose competing updates, one reaches threshold, and `do_update` fires. This is an expected operational pattern (the README explicitly notes that all proposals are cleared on execution). Every contract upgrade cycle where more than one proposal exists silently destroys the non-winning deposits. Likelihood is **Medium**.

### Recommendation

Store the proposer's `AccountId` and the deposit amount inside `UpdateEntry`:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    pub(super) proposer: AccountId,   // add
    pub(super) deposit: NearToken,    // add
}
```

In `do_update`, before calling `self.entries.clear()`, iterate over the remaining entries and issue `Promise::new(entry.proposer).transfer(entry.deposit)` for each one. This mirrors the pattern already used in `propose_update` for excess-deposit refunds.

### Proof of Concept

1. Participant A calls `propose_update` with a 1.5 MB contract binary, attaching ~15 NEAR deposit. Entry `id=0` is stored.
2. Participant B calls `propose_update` with a different binary, attaching ~15 NEAR deposit. Entry `id=1` is stored.
3. Threshold participants vote for `id=1`. `vote_update` calls `do_update(&id=1, gas)`.
4. Inside `do_update`:
   - `self.entries.remove(&id=1)` — winning entry removed.
   - `self.entries.clear()` — entry `id=0` (Participant A's proposal) deleted with no refund.
   - `self.vote_by_participant.clear()` — votes cleared.
5. Participant A's ~15 NEAR is now part of the contract's balance. No refund is issued. No recovery function exists. [6](#0-5)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}
