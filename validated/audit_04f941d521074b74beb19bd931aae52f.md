Audit Report

## Title
`maxNumberOfWithdrawalsPerUser` Uninitialized to Zero Blocks All Withdrawals Until Admin Intervenes — (`contracts/KERNEL/KernelDepositPool.sol`)

## Summary
`KernelDepositPool.initialize()` does not set `maxNumberOfWithdrawalsPerUser`, leaving it at the Solidity default of `0`. The `initiateWithdrawal()` guard evaluates `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser` as `0 >= 0 == true` for every caller, causing an unconditional revert. Any KERNEL tokens staked before the admin calls `setMaxNumberOfWithdrawalsPerUser` are frozen and unrecoverable until that admin action occurs.

## Finding Description
`maxNumberOfWithdrawalsPerUser` is declared as a plain storage variable with no default: [1](#0-0) 

`initialize()` sets `kernelToken`, `rewardsToken`, and the admin role, but never assigns `maxNumberOfWithdrawalsPerUser`: [2](#0-1) 

`initiateWithdrawal()` uses a `>=` comparison against this zero value. A fresh user with zero open withdrawals satisfies `0 >= 0`, triggering `WithdrawalLimitReached` unconditionally: [3](#0-2) 

`stake()` has no corresponding guard, so tokens flow in freely while the exit path is bricked: [4](#0-3) 

The only remedy is the admin calling `setMaxNumberOfWithdrawalsPerUser`, which itself enforces `_maxNumberOfWithdrawalsPerUser > 0`, confirming `0` is never a valid operational value: [5](#0-4) 

There is no on-chain mechanism preventing users from staking while `maxNumberOfWithdrawalsPerUser == 0`, and no guard in `initiateWithdrawal()` that distinguishes between "limit not yet configured" and "limit reached."

## Impact Explanation
**Medium — Temporary freezing of funds.** All KERNEL tokens staked between deployment and the admin's `setMaxNumberOfWithdrawalsPerUser` call are frozen. The freeze is not permanent because the admin can unblock it, but the window is real and exploitable. Any number of users may stake during this window with no ability to withdraw. This matches the allowed impact: *Medium — Temporary freezing of funds*.

## Likelihood Explanation
Moderate. The deployment sequence (deploy → initialize → users stake → admin forgets setter) is a realistic operational oversight. The contract provides no mechanism to prevent staking while `maxNumberOfWithdrawalsPerUser == 0`, and there is no documentation or on-chain guard warning users that withdrawals are disabled. No attacker action is required — only admin inaction.

## Recommendation
Set a safe default for `maxNumberOfWithdrawalsPerUser` inside `initialize()`, or add a parameter to `initialize()` for it. Alternatively, add a guard in `stake()` that reverts if `maxNumberOfWithdrawalsPerUser == 0`:

```solidity
// Option A: set in initialize
maxNumberOfWithdrawalsPerUser = SOME_DEFAULT; // e.g., MAX_WITHDRAWALS_PER_USER or a passed-in param

// Option B: guard in stake
if (maxNumberOfWithdrawalsPerUser == 0) revert WithdrawalsNotConfigured();
```

## Proof of Concept
```solidity
// 1. Deploy proxy, call initialize(admin, kernelToken, rewardToken)
//    maxNumberOfWithdrawalsPerUser == 0 at this point

// 2. User approves and stakes
kernelToken.approve(address(pool), 1e18);
pool.stake(1e18); // succeeds — no guard

// 3. User tries to withdraw
pool.initiateWithdrawal(1e18);
// REVERTS: WithdrawalLimitReached
// userWithdrawalIds[user].length (0) >= maxNumberOfWithdrawalsPerUser (0) → true

// 4. Tokens are locked until admin calls:
pool.setMaxNumberOfWithdrawalsPerUser(10); // only then does step 3 succeed
```

        external
