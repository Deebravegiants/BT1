Audit Report

## Title
Donation Attack Blocks Bridger Collateralization via `depositBridgerAssets` — (`contracts/agETH/AGETHTokenWrapper.sol`)

## Summary

`maxAmountToDepositBridgerAsset` computes available deposit capacity using a live `ERC20.balanceOf` call rather than internal accounting. An attacker holding altAgETH can transfer tokens directly to the wrapper address, inflating `balanceOfAssetInWrapper` above `agETHSupply` and forcing the function to return 0. Because `depositBridgerAssets` gates strictly on this return value, the bridger cannot collateralize agETH minted on L2 via `mint`, leaving those tokens unredeemable until privileged admin intervention.

## Finding Description

`maxAmountToDepositBridgerAsset` reads the wrapper's altAgETH balance directly from the token contract:

```solidity
// L94-100
uint256 agETHSupply = totalSupply();
uint256 balanceOfAssetInWrapper = ERC20Upgradeable(_asset).balanceOf(address(this));
if (balanceOfAssetInWrapper > agETHSupply) return 0;
return agETHSupply - balanceOfAssetInWrapper;
```

`depositBridgerAssets` enforces this cap with a hard revert:

```solidity
// L143-146
function depositBridgerAssets(address _asset, uint256 _amount) external onlyRole(BRIDGER_ROLE) {
    if (maxAmountToDepositBridgerAsset(_asset) < _amount) {
        revert CannotDeposit();
    }
```

**Exploit path:**

1. Normal state: `balance == supply` (fully collateralized after prior bridger deposits).
2. Attacker calls `altAgETH.transfer(address(wrapper), D)` for some donation amount `D`. No wrapper function is invoked; no agETH is minted. Now `balance = supply + D`, so `maxAmountToDepositBridgerAsset` returns 0.
3. MINTER_ROLE calls `wrapper.mint(user, M)` to issue agETH on L2. Now `supply = supply + M`, `balance = supply_old + D`. If `D >= M`, `balance >= supply` still holds and the cap remains 0.
4. Bridger calls `depositBridgerAssets(asset, M)` to collateralize — reverts with `CannotDeposit` because `maxAmountToDepositBridgerAsset` returns 0 (or `M - D` if `D < M`, but attacker can top up).
5. Attacker repeats donation after each admin recovery attempt.

**Why no self-healing exists:**

- `_withdraw` burns agETH and transfers altAgETH out — both sides decrease equally; the surplus `D` persists.
- `_deposit` mints agETH 1:1 with deposited altAgETH — both sides increase equally; surplus persists.
- `mint` (MINTER_ROLE) increases supply without touching balance — the only workaround, but it introduces uncollateralized agETH and requires privileged action.
- No `recoverTokens` or equivalent function exists in the contract.

## Impact Explanation

agETH minted on L2 via `mint` is intended to be backed by the bridger depositing altAgETH through `depositBridgerAssets`. When this path is blocked, the agETH minted on L2 has no backing in the wrapper. Users holding that agETH call `withdraw`/`withdrawTo`, which burns their agETH and attempts to transfer altAgETH out — but the wrapper's altAgETH balance is insufficient to cover the uncollateralized portion. Those users' funds are frozen until admin intervention. This matches **Medium — Temporary freezing of funds**.

## Likelihood Explanation

- **Permissions required:** None. Any altAgETH holder can call `altAgETH.transfer(wrapperAddress, amount)`.
- **Precondition:** The wrapper must be at or near full collateralization (`balance >= supply`), which is the normal post-bridger state.
- **Cost:** The attacker must donate altAgETH equal to the mint amount they wish to block. Those tokens are permanently locked in the contract (no recovery path for the attacker either), making large-scale sustained griefing economically costly but not impossible for a motivated adversary.
- **Repeatability:** The attacker can re-donate after each admin `mint`-based recovery attempt, sustaining the freeze.

## Recommendation

Replace the live `balanceOf` read with an internal accounting variable (e.g., `uint256 private _trackedBalance`) that is incremented only inside `_deposit` and `depositBridgerAssets`, and decremented only inside `_withdraw`. Direct ERC-20 transfers to the contract address will then have no effect on the computed cap.

Additionally, add a privileged `recoverExcessTokens` function allowing the admin to drain `balanceOf(address(this)) - _trackedBalance` to prevent donated tokens from permanently skewing accounting.

## Proof of Concept

```solidity
function test_donationBlocksBridger() public {
    // State: 100e18 agETH minted, 100e18 altAgETH in wrapper (fully collateralized)
    assertEq(wrapper.totalSupply(), altAgETH.balanceOf(address(wrapper)));

    // Step 1: attacker donates 1 wei — no wrapper function called
    vm.prank(attacker);
    altAgETH.transfer(address(wrapper), 1);

    // maxAmountToDepositBridgerAsset now returns 0
    assertEq(wrapper.maxAmountToDepositBridgerAsset(address(altAgETH)), 0);

    // Step 2: minter issues 10e18 agETH on L2 (supply now > balance by 10e18 - 1)
    vm.prank(minter);
    wrapper.mint(user, 10e18);

    // Step 3: attacker donates 10e18 - 1 more to keep balance >= supply
    vm.prank(attacker);
    altAgETH.transfer(address(wrapper), 10e18 - 1);

    // maxAmountToDepositBridgerAsset still returns 0
    assertEq(wrapper.maxAmountToDepositBridgerAsset(address(altAgETH)), 0);

    // Step 4: bridger cannot collateralize — reverts for any positive amount
    vm.prank(bridger);
    vm.expectRevert(AGETHTokenWrapper.CannotDeposit.selector);
    wrapper.depositBridgerAssets(address(altAgETH), 1);

    // Step 5: user tries to withdraw their L2-minted agETH — fails (insufficient balance)
    vm.prank(user);
    vm.expectRevert();
    wrapper.withdraw(address(altAgETH), 10e18);
}
```
