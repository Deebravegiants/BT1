### Title
Expired TEE Attestation Not Re-Verified on `respond()` — Participant Can Contribute to Threshold Signing After Attestation Expiry - (File: crates/contract/src/tee/tee_state.rs)

### Summary

The `is_caller_an_attested_participant()` function, which gates every `respond()`, `vote_pk()`, `vote_reshared()`, and related participant-only calls, checks only that an attestation **entry exists** in `stored_attestations`. It never calls `re_verify()` to confirm the stored attestation is still valid (not expired, docker-image hash still on the allowlist, measurements still accepted). A participant whose attestation has expired — but whose entry has not yet been swept by `clean_invalid_attestations` or `verify_tee` — can continue to call `respond()` and contribute to threshold signing, breaking the protocol invariant that every signing participant must hold a currently-valid TEE attestation.

---

### Finding Description

**Root cause — `is_caller_an_attested_participant` skips validity re-check** [1](#0-0) 

The function checks four things: (1) caller is in the participant set, (2) a `NodeAttestation` entry exists for the caller's TLS key, (3) the stored `account_id` matches the caller, (4) the stored `account_public_key` matches the signer key. It does **not** call `re_verify()` on the stored `VerifiedAttestation`. Expiry, docker-image-hash membership, launcher-compose-hash membership, and OS-measurement membership are all ignored at call time.

**`respond()` relies on this check without adding its own validity gate** [2](#0-1) 

`respond()` calls `assert_caller_is_attested_participant_and_protocol_active()` (which internally calls `is_caller_an_attested_participant`) and checks `accept_requests`, but `accept_requests` is a coarse flag updated only when `verify_tee()` is explicitly invoked — it is not updated per-call.

**Validity re-check exists but is only invoked lazily** [3](#0-2) 

`reverify_participants()` does call `re_verify()` and correctly checks expiry and allowlist membership. It is called from `reverify_and_cleanup_participants()`, which is called from `verify_tee()` and `vote_new_parameters()` — but neither is called automatically on every `respond()`.

**`clean_invalid_attestations` is bounded and not automatic** [4](#0-3) 

The sweep is bounded by `max_scan` and is only triggered automatically (with `max_scan = 100`) at the end of a resharing. Between resharings, expired entries persist until someone explicitly calls the endpoint. [5](#0-4) 

---

### Impact Explanation

The protocol's security model requires that every node contributing to a threshold signature is running approved software inside a verified TEE. An expired attestation means the node's TEE status is unverified — the node may have updated to a non-approved image, had its enclave compromised, or rotated to a different software stack. Despite this, the node can continue to call `respond()` and have its signature contribution accepted by the contract, breaking the invariant:

> *All participants contributing to threshold signing must hold a currently-valid TEE attestation.*

This is a **participant-state invariant violation** that breaks production safety guarantees. The window of exposure is bounded by the attestation lifetime (default 7 days per `DEFAULT_EXPIRATION_DURATION_SECONDS`) minus the cleanup cadence, which can be days in the absence of a resharing event.

**Impact class**: Medium — participant-state manipulation that breaks a production safety invariant without requiring network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

- Attestations have a finite lifetime. In a long-running network without resharing, entries will naturally expire.
- `clean_invalid_attestations` is not called automatically except post-resharing (with a bounded scan of 100 entries).
- `verify_tee()` must be called explicitly by a participant; it is not triggered by `respond()`.
- A participant whose attestation has just expired has a window of up to 7 days (or until the next resharing + cleanup sweep) during which they can continue signing.
- No special privileges or collusion are required — the participant already holds valid key shares from their legitimate onboarding.

---

### Recommendation

Add an inline validity re-check inside `is_caller_an_attested_participant()` (or inside `assert_caller_is_attested_participant_and_protocol_active()`):

```rust
// After confirming the attestation entry exists:
let tee_upgrade_deadline = Duration::from_secs(config.tee_upgrade_deadline_duration_seconds);
match self.reverify_participants(&attestation.node_id, tee_upgrade_deadline) {
    TeeQuoteStatus::Valid => {}
    TeeQuoteStatus::Invalid(reason) => {
        return Err(AttestationCheckError::AttestationExpiredOrInvalid(reason));
    }
}
```

This mirrors the check already performed in `reverify_and_cleanup_participants` and closes the gap between the lazy sweep and the per-call gate. Alternatively, enforce that `verify_tee()` must have been called within a configurable recency window before `respond()` is accepted.

---

### Proof of Concept

1. Network is in `Running` state; participant `alice.near` has a valid attestation stored in `tee_state.stored_attestations`, keyed by her TLS public key. `accept_requests = true`.

2. Time advances past `alice`'s attestation's `expiry_timestamp_seconds` (e.g., 7 days). No resharing occurs; `clean_invalid_attestations` has not been called for `alice`'s entry; `verify_tee()` has not been called.

3. A user submits a `sign()` request. The request is accepted (deposit ≥ 1 yoctoNEAR, `accept_requests = true`). [6](#0-5) 

4. `alice.near` calls `respond(request, valid_signature)`. The contract executes:
   - `assert_caller_is_attested_participant_and_protocol_active()` → calls `is_caller_an_attested_participant()` → finds `alice`'s entry in `stored_attestations`, checks key match → **returns `Ok(())`** (no expiry check).
   - `accept_requests` is `true` → passes.
   - Signature is cryptographically verified → passes (alice has valid key shares).
   - `resolve_yields_for` resumes the user's yield with the signature. [7](#0-6) 

5. The threshold signature is delivered to the user. Alice's TEE attestation was expired at the time of signing; the contract never detected this.

6. Only when `verify_tee()` or `clean_invalid_attestations` is eventually called does the contract discover and act on the expired attestation. [8](#0-7)

    /// reverifies stored participant attestations.
    pub(crate) fn reverify_participants(
        &self,
        node_id: &NodeId,
        tee_upgrade_deadline_duration: Duration,
    ) -> TeeQuoteStatus {
        let allowed_mpc_docker_image_hashes =
            self.get_allowed_mpc_docker_image_hashes(tee_upgrade_deadline_duration);
        let allowed_launcher_compose_hashes = self.get_allowed_launcher_compose_hashes();
        let allowed_measurements = self.get_accepted_measurements();

        let participant_attestation = self.stored_attestations.get(&node_id.tls_public_key);
        let Some(participant_attestation) = participant_attestation else {
            return TeeQuoteStatus::Invalid("participant has no attestation".to_string());
        };

        // Verify the attestation quote
        let time_stamp_seconds = Self::current_time_seconds();
        match participant_attestation.verified_attestation.re_verify(
            time_stamp_seconds,
            &allowed_mpc_docker_image_hashes,
            &allowed_launcher_compose_hashes,
            &allowed_measurements,
        ) {
            Ok(()) => TeeQuoteStatus::Valid,
            Err(err) => TeeQuoteStatus::Invalid(err.to_string()),
        }
    }

    pub fn clean_invalid_attestations(
        &mut self,
        tee_upgrade_deadline_duration: Duration,
        max_scan: usize,
    ) -> u32 {
        let has_invalid_attestation = |node_id: &NodeId| {
            !matches!(
                self.reverify_participants(node_id, tee_upgrade_deadline_duration),
                TeeQuoteStatus::Valid
            )
        };

        // Materialize candidates before any mutation to avoid iterator invalidation.
        let invalid_tls_keys: Vec<Ed25519PublicKey> = self
            .stored_attestations
            .iter()
            .take(max_scan)
            .filter(|(_, node_attestation)| has_invalid_attestation(&node_attestation.node_id))
            .map(|(tls_pk, _)| tls_pk.clone())
            .collect();

        let removed = u32::try_from(invalid_tls_keys.len())
            .expect("u32 should always be convertible from usize on wasm32");

        for tls_pk in invalid_tls_keys {
            self.stored_attestations.remove(&tls_pk);
        }
        removed
    }

    /// Returns Ok(()) if the caller has at least one participant entry
    /// whose TLS key matches an attested node belonging to the caller account.
    ///
    /// Handles multiple participants per account and supports legacy mock nodes.
    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
