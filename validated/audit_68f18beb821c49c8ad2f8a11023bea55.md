### Title
Unbounded `refScriptCostMultiplier` Protocol Parameter Causes Unbounded Fee Inflation and Potential Node Hang in `tierRefScriptFee` - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Tx.hs`)

---

### Summary

The `tierRefScriptFee` function in `eras/conway/impl/src/Cardano/Ledger/Conway/Tx.hs` computes the reference-script fee component of the minimum transaction fee using an exponential growth formula. In the Dijkstra era, the `refScriptCostMultiplier` protocol parameter that drives this exponential growth is typed as `PositiveInterval` — a rational in `(0, +∞)` with no upper bound — and the `ppuWellFormed` check in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/PParams.hs` imposes **no upper-bound constraint** on it. A governance action that sets this parameter to an astronomically large value (e.g., `maxBound :: Word64 % 1`) causes `tierRefScriptFee` to compute a minimum fee that overflows `Coin` (a `Word64`-backed `Integer` that is then `floor`-ed into a `Coin`), or — if the `Rational` arithmetic is done lazily — causes the recursive loop to run for an extremely long time before terminating, effectively hanging every node that validates any transaction containing reference scripts.

---

### Finding Description

**Root cause — no upper-bound check on `refScriptCostMultiplier`:**

`PositiveInterval` is defined as `BoundedRatio PositiveInterval Word64` with `maxBound = BoundedRatio (maxBound % 1)`, i.e., up to `18446744073709551615 / 1`. [1](#0-0) 

The `ppuWellFormed` implementation for `DijkstraEra` checks only that certain parameters are non-zero; it contains **no check** on `ppuRefScriptCostMultiplierL`: [2](#0-1) 

**How the fee is computed:**

`getConwayMinFeeTx` reads `ppRefScriptCostMultiplierG` and passes it as a `Rational` to `tierRefScriptFee`: [3](#0-2) 

`tierRefScriptFee` recurses once per `sizeIncrement`-byte chunk, multiplying `curTierPrice` by `multiplier` each iteration: [4](#0-3) 

With `multiplier = maxBound @Word64 % 1 ≈ 1.8 × 10^19` and `sizeIncrement = 25600`, a transaction with even a single 25 KiB reference script causes `curTierPrice` to become `15 * 1.8e19` after the first tier — a `Rational` whose numerator is a multi-hundred-digit `Integer`. The `floor` call then produces a `Coin` value far exceeding the total ADA supply (`45 × 10^15` lovelace), making the computed `minFee` impossible to satisfy. Every transaction that touches any reference script will fail the `feesOK` check with `FeeTooSmallUTxO`, permanently freezing all DApps that rely on reference scripts until the parameter is corrected via another governance action.

Additionally, if `sizeIncrement` is set to a very small value (also governable via `ppuRefScriptCostStrideL`, typed `NonZero Word32`, with no upper-bound check either), the number of recursive iterations grows proportionally, potentially causing the computation to run for an impractical amount of time.

**Governance entry path (Dijkstra era):**

In the Dijkstra era, `refScriptCostMultiplier` is a proper protocol parameter (tag 37) updatable via a `ParameterChange` governance action: [5](#0-4) 

A governance coalition that meets the `ppTechnicalGroup` / `ppNetworkGroup` DRep and SPO voting thresholds can ratify such an action. Once enacted, the parameter takes effect at the next epoch boundary and cannot be reversed until another governance action is ratified — a process that takes at minimum one governance action lifetime (default 6 epochs ≈ 30 days).

---

### Impact Explanation

**High — Permanent freezing of funds/DApp operations requiring a hard fork or emergency governance action to recover.**

Once the parameter is enacted:
- Every transaction that includes any reference script will compute a `minFee` that exceeds the total ADA supply, causing `feesOK` to reject it with `FeeTooSmallUTxO`.
- All DApps using reference scripts (the majority of Plutus-based DApps on Cardano) become non-functional.
- UTxOs locked at Plutus script addresses that require reference scripts to spend become permanently frozen until the parameter is corrected.
- Correcting the parameter requires a new governance action to be proposed, voted on, and enacted — a process spanning multiple epochs.

This matches the allowed impact: **"High. Permanent freezing of funds, deposits, rewards, or withdrawals where recovery requires a hard fork."** (or at minimum a multi-epoch governance recovery period).

---

### Likelihood Explanation

**Medium.** Requires a governance coalition to ratify a `ParameterChange` action. This is not a single-actor attack — it requires meeting the DRep and/or SPO voting thresholds for the `NetworkGroup`/`TechnicalGroup` parameters. However:
- The Dijkstra era makes `refScriptCostMultiplier` a first-class governable parameter with no guardrail.
- A malicious or negligent governance coalition (e.g., a temporary majority of DRep votes) could enact this.
- The absence of any `ppuWellFormed` check means the ledger itself will accept and apply any `PositiveInterval` value, including `maxBound`.
- There is no on-chain guardrail script enforced by the ledger rules for this parameter.

---

### Recommendation

Add an upper-bound check on `ppuRefScriptCostMultiplierL` inside `ppuWellFormed` for `DijkstraEra` (and any future era that exposes this parameter). A reasonable upper bound would be a value that keeps `tierRefScriptFee` from producing a `Coin` exceeding `maxLovelaceVal` even for the maximum allowed reference script size (`ppMaxRefScriptSizePerTx`). For example:

```haskell
-- In ppuWellFormed for DijkstraEra:
, isValid (<= someReasonableUpperBound) ppuRefScriptCostMultiplierL
```

Similarly, add a lower-bound check ensuring `refScriptCostMultiplier > 1` is not required but that the value is within a range that keeps fee computation tractable and bounded. The same analysis applies to `ppuRefScriptCostStrideL` — a very small stride combined with a large multiplier amplifies the problem.

---

### Proof of Concept

1. A governance coalition proposes and ratifies a `ParameterChange` action setting `refScriptCostMultiplier` to `maxBound @Word64 % 1` (≈ `1.8 × 10^19`), which is a valid `PositiveInterval`.

2. `ppuWellFormed` in `DijkstraEra` accepts this update — no check on `ppuRefScriptCostMultiplierL` exists: [6](#0-5) 

3. At the next epoch boundary, `dijkstraApplyPPUpdates` applies the new value: [7](#0-6) 

4. Any transaction with a reference script calls `getConwayMinFeeTx`, which calls `tierRefScriptFee` with `multiplier = 1.8e19`. For a 25 KiB reference script (one tier), `curTierPrice` after tier 1 = `15 * 1.8e19 = 2.7e20` lovelace — far exceeding the total ADA supply of `4.5e16` lovelace. [4](#0-3) 

5. The `feesOK` check rejects every such transaction with `FeeTooSmallUTxO`, as no user can supply `2.7e20` lovelace in fees. All reference-script-dependent UTxOs are frozen.

```haskell
tierRefScriptFee multiplier sizeIncrement
  | multiplier <= 0 || sizeIncrement <= 0 = error "Size increment and multiplier must be positive"
  | otherwise = go 0
  where
    go !acc !curTierPrice !n
      | n < sizeIncrement =
          Coin $ floor (acc + toRational n * curTierPrice)
      | otherwise =
          go (acc + sizeIncrementRational * curTierPrice) (multiplier * curTierPrice) (n - sizeIncrement)
    sizeIncrementRational = toRational sizeIncrement
```
