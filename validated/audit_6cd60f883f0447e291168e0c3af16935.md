### Title
Proposal Deposit Permanently Lost When `do_update` Clears All Competing Proposals — (`File: crates/contract/src/update.rs`)

### Summary

When `vote_update` reaches threshold and `do_update` executes, it silently clears **all** pending proposals and their associated storage, but never refunds the NEAR deposits that competing proposers paid to store their proposals. This is a direct accounting analog to the `amend_bid_order` missing-refund bug: a committed resource (the storage deposit) is reduced/freed without returning the corresponding value to the depositor.

### Finding Description

`propose_update` is a `#[payable]` function that requires each proposer to attach a deposit proportional to the size of their update payload: [1](#0-0) 

The required deposit is computed from `bytes_used(update)`, which for a contract-code update includes the full WASM binary size: [2](#0-1) 

Any excess above the required amount is refunded at proposal time: [3](#0-2) 

However, when threshold votes are reached and `do_update` executes, it removes the winning entry **and clears every other pending entry** without issuing any refund to those other proposers: [4](#0-3) 

The storage bytes are freed, but the NEAR tokens that were deposited to cover that storage remain in the contract permanently. There is no `cancel_proposal` or equivalent mechanism that would allow a proposer to reclaim their deposit.

The sandbox test `test_propose_update_contract_many` explicitly demonstrates the scenario where multiple proposals coexist and all non-winning ones are silently deleted: [5](#0-4) 

### Impact Explanation

Each participant who proposes a contract-code update must deposit approximately `storage_byte_cost × (sizeof(UpdateEntry) + 128 × sizeof(AccountId) + wasm_size)` NEAR. For a typical WASM binary (~500 KB), this is several NEAR tokens per proposal. When any one proposal wins, all competing proposals are cleared and their depositors permanently lose their funds. This breaks the production accounting invariant that a deposit paid to store a resource is returned when that resource is freed.

**Impact class:** Medium — balance/accounting invariant broken, causing permanent loss of NEAR tokens held by the MPC contract on behalf of participant proposers.

### Likelihood Explanation

The scenario is a normal operational condition: the test suite itself (`test_propose_update_config`) shows each participant independently proposing their own update. With N participants each proposing a different update, N−1 proposers lose their deposits every time an update is executed. No adversarial action is required; the loss occurs as a natural consequence of the governance flow.

### Recommendation

Track the proposer's account ID alongside each `UpdateEntry`. In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and issue a `Promise::new(proposer).transfer(deposit)` refund for each cleared proposal. The deposit amount to refund is `required_deposit(entry.bytes_used)`.

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB WASM, attaching ~5 NEAR deposit. Proposal ID = 0.
2. Participant B calls `propose_update` with a different 500 KB WASM, attaching ~5 NEAR deposit. Proposal ID = 1.
3. Threshold participants vote for proposal 1 → `vote_update` calls `do_update(&id=1, ...)`.
4. `do_update` removes entry 1, then calls `self.entries.clear()` which removes entry 0.
5. No refund is issued to Participant A. Their ~5 NEAR deposit is permanently retained by the contract.
6. Participant A has no recourse — there is no cancel or reclaim function. [4](#0-3) [6](#0-5)

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}
