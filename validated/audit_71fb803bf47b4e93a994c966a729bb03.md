### Title
Unvalidated `builder` Field in `ProposalInit` Allows Malicious Proposer to Redirect Sequencer Fees and Corrupt Block Header — (`File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_proposal_init_valid()` validates many fields of `ProposalInit` (timestamp, starknet_version, version_constant_commitment, height, l1_da_mode, l2_gas_price_fri, L1 gas prices, fee_proposal_fri) but does **not** validate the `builder` field. The `builder` field is directly mapped to `sequencer_address` in `BlockInfo` via `convert_to_sn_api_block_info()` and passed to the batcher for block execution. A malicious proposer can set `builder` to any arbitrary address; validators will accept the proposal, execute the block with the attacker-controlled sequencer address, and commit the corrupted block header to chain.

---

### Finding Description

`ProposalInit` carries a `builder` field (the address of the block builder/sequencer): [1](#0-0) 

`convert_to_sn_api_block_info()` maps `init.builder` directly to `sequencer_address` in the `BlockInfo` passed to the batcher: [2](#0-1) 

This `BlockInfo` is forwarded to the batcher in `initiate_validation()`: [3](#0-2) 

The `is_proposal_init_valid()` function validates `height`, `l1_da_mode`, `l2_gas_price_fri`, `starknet_version`, `version_constant_commitment`, all four L1 gas prices, and `fee_proposal_fri`, but the `builder` field is absent from every check: [4](#0-3) 

The `ProposalInitValidation` struct — the source of all expected values used in `is_proposal_init_valid` — has no `builder` field at all: [5](#0-4) 

The honest proposer sets `builder` from its own configured `builder_address`: [6](#0-5) 

Because no validator-side expected value exists and no check is performed, any value a proposer places in `builder` passes validation unchanged.

---

### Impact Explanation

The `sequencer_address` derived from `builder` is used in three ways:

1. **Fee collection** — the blockifier credits transaction fees to `sequencer_address`. A malicious proposer redirects all fees in the block to an attacker-controlled address.
2. **Block header** — `sequencer_address` is stored in the committed block header and propagated to state sync, producing a permanently wrong on-chain record.
3. **Cairo contract visibility** — the OS hint `guess_block_info` exposes `sequencer_address` to executing contracts; contracts that branch on the sequencer address (e.g., fee-token contracts, access-control logic) receive the wrong value. [7](#0-6) 

This matches: **Critical — Incorrect fee, gas, resource accounting, refund, balance, or L1 gas price effect with economic impact** and **Critical — Wrong state, receipt, event, or storage value from blockifier/syscall/execution logic**.

---

### Likelihood Explanation

Any validator that wins a proposer slot can craft a `ProposalInit` with an arbitrary `builder`. No special privilege beyond normal proposer rights is required. The attack is silent — the proposal passes all existing validation checks and produces a valid `ProposalFinMismatch`-free commitment. The corrupted sequencer address is committed to chain and cannot be rolled back without a revert.

---

### Recommendation

Add `builder` (expected sequencer address) to `ProposalInitValidation` and enforce it in `is_proposal_init_valid()`:

```rust
// In ProposalInitValidation:
pub builder: ContractAddress,

// In is_proposal_init_valid():
if init_proposed.builder != proposal_init_validation.builder {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.builder, init_proposed.builder
        ),
    ));
}
```

The expected `builder` value should be populated from the node's own configured `builder_address` when constructing `ProposalInitValidation` in `validate_proposal()` inside `sequencer_consensus_context.rs`. [8](#0-7) 

---

### Proof of Concept

1. A malicious validator wins a proposer slot at height H.
2. It constructs `ProposalInit { builder: attacker_address, ... }` with all other fields valid.
3. It streams the proposal to peer validators.
4. Each validator calls `is_proposal_init_valid()` — passes (no `builder` check).
5. Each validator calls `initiate_validation()` → `convert_to_sn_api_block_info(init)` → `sequencer_address = attacker_address` → `batcher.validate_block(input)`.
6. The batcher executes the block with `sequencer_address = attacker_address`; all transaction fees are credited to `attacker_address`.
7. `finish_proposal` returns a valid `partial_block_hash`; `proposal_commitment_from(hash, fee_proposal)` matches `ProposalFin.proposal_commitment` — no `ProposalFinMismatch`.
8. Consensus commits the block. The block header on-chain records `sequencer_address = attacker_address`. All fees for height H are permanently redirected. [9](#0-8)
