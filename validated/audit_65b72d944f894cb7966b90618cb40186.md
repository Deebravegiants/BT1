Audit Report

## Title
Nested NDC × Asset Loop in `getAssetUnstaking()` Causes Unbounded Gas Consumption in `updateRSETHPrice()` and `depositAsset()`/`depositETH()` — (File: `contracts/NodeDelegator.sol`, `contracts/LRTDepositPool.sol`, `contracts/LRTOracle.sol`)

## Summary
`NodeDelegator.getAssetUnstaking()` issues an external call to EigenLayer's `DelegationManager.getQueuedWithdrawals()` and iterates over all returned withdrawals and strategies on every invocation. This function is called inside a nested loop — once per NDC per supported asset — in `LRTDepositPool.getAssetDistributionData()`, which is itself called once per supported asset in `LRTOracle._getTotalEthInProtocol()`. The public, unrestricted `LRTOracle.updateRSETHPrice()` and every user `depositAsset()`/`depositETH()` call traverse this full nested path, producing gas costs that scale multiplicatively with protocol growth. The developers explicitly acknowledge the gas ceiling in `LRTUnstakingVault.sol`.

## Finding Description
**Root cause — `NodeDelegator.getAssetUnstaking()` (L405–427):**
Every call issues one external call to `_getDelegationManager().getQueuedWithdrawals(address(this))` and then iterates over all returned `queuedWithdrawals` and all `withdrawal.strategies` within each. There is no caching; the full withdrawal queue is fetched and scanned on every invocation.

**First nesting layer — `LRTDepositPool.getAssetDistributionData()` (L446–456):**
For a given asset, this function loops over the entire `nodeDelegatorQueue` (up to `maxNodeDelegatorLimit = 10`) and calls `getAssetUnstaking(asset)` on each NDC. This produces up to 10 external calls to EigenLayer per asset.

**Second nesting layer — `LRTOracle._getTotalEthInProtocol()` (L331–349):**
This function loops over every entry in `supportedAssetList` and calls `ILRTDepositPool.getTotalAssetDeposits(asset)` → `getAssetDistributionData(asset)` for each. With N supported assets, the total external calls to EigenLayer become `N_assets × N_NDCs` and total inner iterations become `N_assets × N_NDCs × N_withdrawals × N_strategies`.

**Public entry point — `LRTOracle.updateRSETHPrice()` (L87–89):**
Declared `public whenNotPaused` with no role restriction; any address can call it.

**User deposit entry point — `LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit()` (L676–682):**
Called on every `depositAsset()`/`depositETH()`, it invokes `getTotalAssetDeposits(asset)` → `getAssetDistributionData(asset)`, triggering the NDC loop and `getAssetUnstaking()` for the deposited asset.

**Developer acknowledgment — `LRTUnstakingVault.sol` (L151–153):**
The comment `"120 is the max number of uncompleted withdrawals that allows us to still perform update rsETH price"` and the hard cap `if (_maxUncompletedWithdrawalCount > 80) revert` confirm the developers are aware that gas consumption is bounded only by operational caps, not by the contract logic itself.

Existing guards (`maxNodeDelegatorLimit = 10`, `maxUncompletedWithdrawalCount ≤ 80`) are administrative parameters, not protocol-enforced gas ceilings. They can be raised by the manager, and even at current values the gas cost is significant. As the protocol adds more supported assets or NDCs, the cost grows multiplicatively without any code-level bound.

## Impact Explanation
**Medium — Unbounded gas consumption / Temporary freezing of funds.**
If the cumulative gas cost of `updateRSETHPrice()` approaches or exceeds the Ethereum block gas limit (~30M gas), the function becomes uncallable, freezing the rsETH price at a stale value and breaking the fee-minting mechanism. Every `depositAsset()`/`depositETH()` call also traverses the same path (for a single asset), so deposits revert if gas exceeds the block limit, temporarily freezing user access to the protocol. Both impacts match allowed scope entries: "Medium. Unbounded gas consumption" and "Medium. Temporary freezing of funds."

## Likelihood Explanation
**Medium.** No attacker action is required. The gas cost grows naturally as the protocol scales: more supported assets, more NDCs, or more concurrent EigenLayer unstaking operations all increase it multiplicatively. `updateRSETHPrice()` is callable by any unprivileged address. The developer-acknowledged cap of 80 uncompleted withdrawals per NDC, combined with 10 NDCs and multiple supported assets, already produces hundreds of EigenLayer external calls and thousands of in-memory iterations per `updateRSETHPrice()` invocation. The same withdrawal data for each NDC is fetched redundantly once per supported asset with no caching.

## Recommendation
1. **Cache `getQueuedWithdrawals()` per NDC per transaction**: fetch once per NDC and compute per-asset amounts in a single pass, eliminating the `N_assets × N_NDCs` external call multiplication.
2. **Maintain a running `totalUnstaking` counter per asset per NDC**: increment on `initiateUnstaking`, decrement on `completeUnstaking`, eliminating the need to iterate over EigenLayer's withdrawal queue at read time entirely.
3. **Apply a hard cap on `supportedAssetList` length** in addition to the existing NDC and withdrawal caps, to bound the outer loop in `_getTotalEthInProtocol()`.

## Proof of Concept
Call chain for `updateRSETHPrice()` (callable by any user, no role required):

```
LRTOracle.updateRSETHPrice()                          // public, no access control
  └─ _updateRsETHPrice()
       └─ _getTotalEthInProtocol()                    // loops over N_assets (e.g. 5)
            └─ ILRTDepositPool.getTotalAssetDeposits(asset)
                 └─ getAssetDistributionData(asset)   // loops over N_NDCs (e.g. 10)
                      └─ INodeDelegator.getAssetUnstaking(asset)
                           └─ DelegationManager.getQueuedWithdrawals(ndc)
                                // external call, returns up to 80 withdrawals
                                └─ for each withdrawal (N_w):
                                     for each strategy (N_s):
                                         ...
```

Concrete gas estimate at operational caps (5 assets × 10 NDCs × 8 withdrawals × 2 strategies):
- External calls to EigenLayer: **50**
- Inner iterations: **800**
- Each `getQueuedWithdrawals` call on mainnet costs ~20k–50k gas; 50 calls alone = 1M–2.5M gas before any iteration logic.

Foundry fork test plan:
1. Fork Ethereum mainnet.
2. Deploy/configure 5 supported assets, 10 NDCs, and queue 8 withdrawals per NDC via `initiateUnstaking`.
3. Call `LRTOracle.updateRSETHPrice()` from an unprivileged EOA and measure gas with `vm.expectCall` and `gasleft()` snapshots.
4. Repeat with 10 assets and 10 NDCs to demonstrate multiplicative growth toward block gas limit.
