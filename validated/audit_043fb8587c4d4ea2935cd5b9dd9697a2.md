### Title
Expired Governance Actions Can Be Ratified and Enacted Due to Missing Expiry Guard in Ratification Branch — (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs`)

---

### Summary

In the Conway `RATIFY` rule, the expiry check for a `GovActionState` is placed exclusively in the **non-ratification** branch of `ratifyTransition`. When all voting conditions are satisfied, a governance action is enacted with no check on whether `gasExpiresAfter < reCurrentEpoch`. A proposal that has already expired can therefore be ratified and enacted — including `TreasuryWithdrawals`, `HardForkInitiation`, `UpdateCommittee`, `ParameterChange`, and `NewConstitution` — bypassing the intended lifecycle constraint. A disabled conformance test explicitly documents this divergence from the formal specification.

---

### Finding Description

`ratifyTransition` in `Ratify.hs` processes each `GovActionState` from the `RatifySignal` sequence. The structure is:

```haskell
if prevActionAsExpected gas ensPrevGovActionIds
  && validCommitteeTerm govAction ensCurPParams reCurrentEpoch
  && not rsDelayed
  && withdrawalCanWithdraw govAction ensTreasury
  && acceptedByEveryone env st gas
  then do
      -- ENACT — no expiry check here
      newEnactState <- trans @(EraRule "ENACT" era) ...
      trans @(RATIFY era) $ TRC (env, st', RatifySignal sigs)
  else do
      st' <- trans @(RATIFY era) $ TRC (env, st, RatifySignal sigs)
      -- Expiry check ONLY in the else branch
      if gasExpiresAfter < reCurrentEpoch
        then pure $ st' & rsExpiredL %~ Set.insert gasId
        else pure st'
``` [1](#0-0) 

The expiry predicate `gasExpiresAfter < reCurrentEpoch` is evaluated **only** when the action fails ratification. If all five conditions in the `if` guard are satisfied, the action is enacted unconditionally — even when `gasExpiresAfter < reCurrentEpoch`.

**Concrete attack path using the `delayingAction` mechanism:**

`NewConstitution`, `UpdateCommittee`, `HardForkInitiation`, and `NoConfidence` are all *delaying actions*: [2](#0-1) 

When one of these is enacted, `rsDelayed` is set to `True`, blocking all lower-priority proposals in the same epoch. `rsDelayed` is reset to `False` at the start of each new epoch (initialized in `finishDRepPulser`): [3](#0-2) 

**Step-by-step scenario (treasury drain):**

| Epoch | Event |
|-------|-------|
| N | Submit `NewConstitution` proposal A and `TreasuryWithdrawals` proposal B, both with `gasExpiresAfter = N+1` (lifetime = 1 epoch). Both receive sufficient yes-votes. |
| N+1 boundary | RATIFY runs. A (priority 2) is ratified first; `rsDelayed ← True`. B (priority 5) is skipped due to delay. Expiry check on B: `N+1 < N+1` → **False** → B is not expired yet. |
| N+2 boundary | RATIFY runs fresh (`rsDelayed = False`). B is processed. All voting conditions still hold. B is **ratified and enacted** — but `gasExpiresAfter = N+1 < reCurrentEpoch = N+2` → B **should have expired**. |

The `ENACT` rule for `TreasuryWithdrawals` then subtracts the withdrawal amount from `ensTreasury` and accumulates it in `ensWithdrawals`, which is later applied to the real treasury in `applyEnactedWithdrawals`: [4](#0-3) [5](#0-4) 

**Disabled conformance test confirms the bug:**

The test `"An action expires when delayed enough even after being ratified"` in `RatifySpec.hs` is explicitly disabled with a reference to formal-ledger-specifications issue #923: [6](#0-5) 

The test expects `gai3` (a `NewConstitution` action) to **not** be enacted after expiry, but the current Haskell implementation enacts it. The test is disabled because the implementation diverges from the formal specification.

---

### Impact Explanation

**Critical — Unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action is enacted.**

The most severe case is `TreasuryWithdrawals`: ADA is transferred out of the treasury after the proposal's authorized lifetime has elapsed. This constitutes a direct, unauthorized loss of ADA from the protocol treasury through an invalid ledger state transition.

For `HardForkInitiation`, `UpdateCommittee`, `ParameterChange`, and `NewConstitution`, the same mechanism allows unauthorized protocol-level changes to be enacted after the community's implicit consent window (the proposal lifetime) has closed.

---

### Likelihood Explanation

The trigger condition — a delaying action (priority ≤ 3) being ratified in the same epoch as a lower-priority proposal that is at or near its expiry — is a realistic, non-exotic scenario. It requires no privileged access: any participant with sufficient voting stake can submit and vote on governance proposals. The scenario becomes more probable as governance activity increases and multiple proposals of different priorities are in flight simultaneously. The formal specification explicitly prohibits this behavior, confirming it is unintended.

---

### Recommendation

Add `gasExpiresAfter >= reCurrentEpoch` as a guard in the ratification condition so that an expired action is never enacted:

```haskell
if gasExpiresAfter >= reCurrentEpoch          -- not yet expired
  && prevActionAsExpected gas ensPrevGovActionIds
  && validCommitteeTerm govAction ensCurPParams reCurrentEpoch
  && not rsDelayed
  && withdrawalCanWithdraw govAction ensTreasury
  && acceptedByEveryone env st gas
  then do
      ...
  else do
      st' <- trans @(RATIFY era) $ TRC (env, st, RatifySignal sigs)
      if gasExpiresAfter < reCurrentEpoch
        then pure $ st' & rsExpiredL %~ Set.insert gasId
        else pure st'
```

This aligns the implementation with the formal specification and the intent of the disabled conformance tests.

---

### Proof of Concept

The disabled conformance test at `RatifySpec.hs:1685–1709` is a direct proof of concept. With `ppGovActionLifetimeL .~ EpochInterval 2`, a chain of four `NewConstitution` proposals (`gai0`–`gai3`) all receive yes-votes. Each enacted constitution is a delaying action, pushing the next one to the following epoch. By the time `gai3` is processed, its `gasExpiresAfter` has passed, yet the current implementation enacts it. The test asserts `getLastEnactedConstitution \`shouldReturn\` SJust (GovPurposeId gai2)` (i.e., `gai3` should expire, not be enacted), but this assertion fails against the current code — which is why the test is disabled.
