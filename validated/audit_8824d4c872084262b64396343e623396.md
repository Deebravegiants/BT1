The code path is concrete and the missing check is confirmed. Here is the full analysis:

**Tracing the attack path:**

1. `parse_data_for_block` in `state_diff.rs` reads `target_state_diff_len` from the stored block header's `state_diff_length` field. [1](#0-0) 

2. It accumulates peer-supplied `StateDiffChunk` messages until `current_state_diff_len == target_state_diff_len`. The only guard on a `DeclaredClass` chunk is the duplicate-key check (`ConflictingStateDiffParts`). There is **no verification of the assembled state diff against the `state_diff_commitment` stored in the header**. [2](#0-1) 

3. The assembled `ThinStateDiff` is written directly to storage via `append_state_diff` with no commitment check. [3](#0-2) 

4. The block header itself is also accepted from a p2p peer. The header sync only checks block number ordering and signature count — it does **not** cryptographically verify the signature against the sequencer's public key. [4](#0-3) 

5. `calculate_state_diff_hash` exists and covers `class_hash_to_compiled_class_hash` in its Poseidon chain, so a commitment check *would* catch a wrong `compiled_class_hash`. [5](#0-4) 

6. The stored `compiled_class_hash` is later read by `get_compiled_class_hash` in the execution state reader and used to look up the actual compiled class from the class manager. A wrong value causes wrong class selection or a lookup failure. [6](#0-5) 

**Conclusion:**

The attack is fully reachable via unauthenticated p2p messages (an explicitly allowed attack vector). The malicious peer:
- Sends a fake `SignedBlockHeader` with `state_diff_length=1` (no signature verification is performed)
- Sends one `DeclaredClass` chunk with `class_hash=H`, `compiled_class_hash=WRONG_CASM`
- The length check passes; no commitment check is performed; the wrong value is stored

The `apollo_committer` does have a `verify_state_diff_hash` config flag, but that is in the committer path (block production), not in the p2p sync client path. [7](#0-6)

### Title
Missing state-diff commitment verification in p2p sync client allows malicious peer to corrupt `class_hash_to_compiled_class_hash` — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

### Summary
The p2p sync client assembles a `ThinStateDiff` from peer-supplied `StateDiffChunk` messages and writes it to storage without verifying the result against the `state_diff_commitment` stored in the block header. A malicious peer can inject a `DeclaredClass` chunk with an arbitrary `compiled_class_hash`, corrupting the mapping used by execution and RPC.

### Finding Description
`parse_data_for_block` in `state_diff.rs` validates only that the total chunk length equals `target_state_diff_len` from the stored header. It never calls `calculate_state_diff_hash` on the assembled diff and never compares the result to `BlockHeader::state_diff_commitment`. The header itself is also accepted without cryptographic signature verification (`header.rs` checks only block number ordering and signature count). A peer therefore controls both the `state_diff_length` anchor and the chunk content.

### Impact Explanation
The corrupted `class_hash_to_compiled_class_hash` entry is read by `get_compiled_class_hash` in `apollo_rpc_execution/src/state_reader.rs` and used to look up the compiled class artifact from the class manager. A wrong hash causes the wrong CASM/native artifact to be selected for execution, or causes a `StateReadError`, affecting all RPC calls that execute or simulate contracts declared in the corrupted block.

### Likelihood Explanation
Any peer the node connects to for p2p sync can mount this attack. No privileged access is required. The node has no mechanism to detect the corruption after the fact because the execution path reads the stored diff directly without re-checking the commitment.

### Recommendation
After assembling the full `ThinStateDiff` (after the `current_state_diff_len == target_state_diff_len` check), compute `calculate_state_diff_hash(&result)` and compare it to `block_header.state_diff_commitment`. Return `BadPeerError` on mismatch. Additionally, verify the block signature in `header.rs` using `verify_block_signature` before storing headers received from external peers.

### Proof of Concept
1. Peer sends `SignedBlockHeader` for block N with `state_diff_length = 1` and any `state_diff_commitment` value (no signature check is performed).
2. Peer sends `StateDiffChunk::DeclaredClass(DeclaredClass { class_hash: H, compiled_class_hash: WRONG_CASM })`.
3. `current_state_diff_len` reaches 1 == `target_state_diff_len`; loop exits.
4. `validate_deprecated_declared_classes_non_conflicting` passes (no deprecated classes).
5. `write_to_storage` calls `append_state_diff(N, result)` storing `{H → WRONG_CASM}`.
6. Any subsequent call to `get_compiled_class(H)` reads `WRONG_CASM` from storage and passes it to the class manager, returning the wrong artifact or an error.
