### Title
`respond_verify_foreign_tx` Accepts Signature Over Arbitrary `payload_hash` Without Binding It to the Submitted `request` — (`crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies only that the ECDSA signature is valid over the caller-supplied `payload_hash`, but never checks that `payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request, observed_values }))` for the specific `request` being resolved. A single Byzantine attested participant can replay a valid `(payload_hash_A, sig_A)` produced for request A and submit it as the response to a different pending request B, causing the contract to drain all queued yields for request B with fabricated observation data.

---

### Finding Description

In `respond_verify_foreign_tx` the only cryptographic check performed is:

```rust
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,   // taken verbatim from response.payload_hash.0
    &secp_pk,
)
``` [1](#0-0) 

The `payload_hash` value is taken directly from the caller-supplied `response.payload_hash` with no check that it was derived from the `request` argument passed in the same call. After the signature check passes, the contract immediately resolves every queued yield for `request` with the full `response` blob:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

`resolve_yields_for` removes the entry from the map and resumes every queued yield in one pass: [3](#0-2) 

The correct binding is defined in `ForeignTxSignPayload::compute_msg_hash`, which hashes `(request, observed_values)` together: [4](#0-3) 

The node-side code correctly computes `payload_hash = response.0.0.compute_msg_hash()?` before constructing the respond args: [5](#0-4) 

But the contract never re-derives or validates this relationship.

---

### Impact Explanation

**Attack path (single Byzantine attested participant, no threshold collusion):**

1. Attacker is an attested participant (passes `assert_caller_is_attested_participant_and_protocol_active`).
2. Attacker observes any prior on-chain `respond_verify_foreign_tx` transaction for request A and extracts `(payload_hash_A, sig_A)` — NEAR transactions are public.
3. A victim bridge contract submits `verify_foreign_transaction(request_B)` for a different foreign transaction (e.g., a different Bitcoin tx_id). The yield is queued in `pending_verify_foreign_tx_requests`.
4. Attacker calls `respond_verify_foreign_tx(request_B, { payload_hash_A, sig_A })`.
5. Contract checks `verify_ecdsa_signature(sig_A, payload_hash_A, root_pk)` — **passes** (the signature is genuinely valid over `payload_hash_A`).
6. Contract calls `resolve_yields_for` for `request_B`, draining all queued yields with `{ payload_hash_A, sig_A }` as the response data.
7. The pending entry for `request_B` is permanently removed; the honest nodes' legitimate response can never be delivered.

**Two distinct harms:**

- **Forged verification delivered to callers**: Callers of `request_B` receive `payload_hash_A`, which encodes the observed values for a completely different foreign transaction. Callers that do not independently re-derive and compare the expected hash (i.e., do not use `ForeignChainSignatureVerifier::verify_signature` from the SDK) will accept this as a valid MPC attestation of `request_B`'s foreign-chain state, enabling invalid bridge execution. [6](#0-5) 

- **Permanent DoS on the pending request**: Even callers that do use the SDK verifier will receive `IncorrectPayloadSigned` and cannot retry — the contract has already consumed the yield slot. The legitimate response from honest nodes arrives to find no pending entry and returns `RequestNotFound`.

---

### Likelihood Explanation

The precondition is that the attacker is an attested participant. In the TEE-based deployment this requires passing DCAP attestation and being voted in, which is a meaningful barrier. However, once inside the participant set, the attack requires only:

- Reading one prior on-chain `respond_verify_foreign_tx` transaction (trivially observable).
- Submitting one NEAR transaction before honest nodes respond to the target request.

No threshold collusion, no key material, and no cryptographic forgery are needed. The race window is bounded by NEAR block time; an attacker who also controls when `request_B` is submitted (e.g., the attacker is also the bridge operator or a user) can eliminate the race entirely by submitting the forged response in the same block as the request.

---

### Recommendation

The contract must bind `payload_hash` to `request` before accepting the response. The cleanest fix is to require the responder to submit the full `observed_values` alongside the hash, then recompute and compare:

```rust
// In respond_verify_foreign_tx, after the signature check:
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.observed_values.clone(), // new field
})
.compute_msg_hash()
.map_err(|_| RespondError::InvalidPayloadHash)?;

if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Alternatively, if adding `observed_values` to the response DTO is undesirable, the contract can require the responder to pass the `ForeignTxSignPayload` directly and compute the hash on-chain, then verify the signature over the computed hash rather than the caller-supplied one.

### Proof of Concept

```rust
// Unit test sketch (no threshold collusion required):
// 1. Set up contract with a ForeignTx domain and a pending request_B.
// 2. Obtain (payload_hash_A, sig_A) by signing any ForeignTxSignPayload
//    for a *different* request_A with the root key.
// 3. Call respond_verify_foreign_tx(request_B, { payload_hash_A, sig_A }).
// 4. Assert the call succeeds (Ok(())).
// 5. Assert pending_verify_foreign_tx_requests no longer contains request_B
//    (yields were drained with the wrong hash).
// 6. Assert the response delivered to the caller contains payload_hash_A,
//    not SHA-256(borsh(ForeignTxSignPayload { request_B, ... })).
```

The existing test infrastructure in `crates/contract/src/lib.rs` (e.g., `respond_verify_foreign_tx__should_succeed_when_response_is_valid_and_request_exists`) already demonstrates how to construct a valid `(payload_hash, sig)` pair for one request; the PoC simply reuses that pair against a second, distinct pending request. pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}

impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }
