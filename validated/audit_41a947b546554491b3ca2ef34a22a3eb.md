### Title
Missing `payload_hash`-to-`request` Binding in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay — (`File: crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is valid for `response.payload_hash` against the root public key, but never verifies that `response.payload_hash` is actually derived from the submitted `request`. A single malicious attested participant (strictly below signing threshold) can replay a legitimately-obtained signature from a previous foreign-chain verification request to resolve a completely different pending request, delivering a forged verification response to the waiting caller.

---

### Finding Description

In `respond_verify_foreign_tx`, the contract performs these checks:

1. Caller is an attested participant.
2. Protocol is `Running` or `Resharing`.
3. `accept_requests` is `true`.
4. `response.signature` is a valid ECDSA signature over `response.payload_hash` under the **root** public key of the domain.
5. A pending yield exists for `request` in `pending_verify_foreign_tx_requests`. [1](#0-0) 

What is **absent** is any check that `response.payload_hash` is the SHA-256 of `borsh(ForeignTxSignPayload::V1 { request: request.request, values: <observed> })`. The `payload_hash` is taken entirely from the node-supplied response and is never cross-checked against the on-chain `request` key.

Contrast this with the regular `respond` path, where the payload is taken directly from the user-submitted `request.payload` and the signing key is derived using `request.tweak`, binding the signature cryptographically to the request: [2](#0-1) 

For foreign-tx verification the node code uses a zero tweak (root key, no derivation): [3](#0-2) 

Because the root key is shared across all foreign-tx requests in the same domain, any valid `(payload_hash, signature)` pair produced for request R1 is also a valid pair that the contract will accept when submitted against a different pending request R2.

**Attack path (single malicious attested participant, no threshold collusion):**

1. The MPC network legitimately processes request R1 (e.g., "verify Bitcoin tx A, extract block-hash") and produces signature S1 over hash H1. The response is observable on-chain.
2. A user submits request R2 (e.g., "verify Bitcoin tx B, extract block-hash"), creating a pending yield.
3. The malicious node calls `respond_verify_foreign_tx(R2, { payload_hash: H1, signature: S1 })`.
4. The contract verifies S1 is valid for H1 under the root key — it is. The pending yield for R2 is resolved and the caller receives `{ payload_hash: H1, signature: S1 }`.
5. The caller of R2 holds a root-key signature that was produced for Bitcoin tx A, not Bitcoin tx B.

No signature forgery is required. The attacker only needs to observe a previously produced response (public on-chain) and submit it against any pending request in the same domain.

The `ForeignTxSignPayload` does not include `domain_id`, but different domains use different root keys, so cross-domain replay is blocked by the key-mismatch check. Within a single domain, however, any prior signature is replayable. [4](#0-3) 

---

### Impact Explanation

A bridge or NEAR contract that calls `verify_foreign_transaction` to attest that a specific foreign-chain event occurred receives a `VerifyForeignTransactionResponse` whose `payload_hash` does not correspond to the transaction it requested. If the consuming contract verifies only that the signature is valid for the returned `payload_hash` (a natural and common pattern), it will accept a proof of the wrong event. This enables:

- **Forged foreign-chain verification**: the MPC network's root-key signature is presented as attestation of an event it never verified.
- **Invalid bridge execution / double-spend**: an inbound bridge flow (foreign chain → NEAR) could be triggered by replaying the attestation of a previously-processed transaction against a new pending request, causing the bridge to credit funds for a transaction that was never independently verified.

This matches the allowed High impact: *"Cross-chain replay, forged foreign-chain verification … that causes invalid bridge execution or double-spend conditions."*

---

### Likelihood Explanation

- Requires only **one** malicious attested participant — strictly below the signing threshold.
- No cryptographic capability beyond observing on-chain data is needed; the attacker reuses a legitimately produced signature.
- Any pending `verify_foreign_transaction` request in the same domain is a valid target.
- The window of opportunity is the entire lifetime of a pending yield (up to ~200 blocks per the NEAR yield-resume timeout).

---

### Recommendation

In `respond_verify_foreign_tx`, after verifying the signature, recompute the expected payload hash from the `request` fields and assert equality with `response.payload_hash`. Because the `extracted_values` are not known on-chain, the binding must be enforced differently — for example:

- **Include the `VerifyForeignTransactionRequest` (or its hash) as a domain separator inside `ForeignTxSignPayload`**, so that H1 for R1 is structurally distinct from any hash producible for R2.
- Alternatively, **include the NEAR receipt ID of the originating `verify_foreign_transaction` call** in the signed payload; the contract can then verify that `response.payload_hash` encodes the correct receipt ID.

Either approach ensures that a signature produced for R1 cannot satisfy the verification check for R2, eliminating the cross-request replay vector.

### Proof of Concept

```
1. Deploy contract; configure a ForeignTx domain (domain_id = D).
2. Alice calls verify_foreign_transaction({ domain_id: D, request: BitcoinRpcRequest { tx_id: TX_A, ... } })
   → pending yield Y_A created; request key K_A stored in pending_verify_foreign_tx_requests.
3. MPC nodes legitimately process K_A; malicious node observes the on-chain respond_verify_foreign_tx
   call that resolves Y_A: response = { payload_hash: H_A, signature: S_A }.
4. Bob calls verify_foreign_transaction({ domain_id: D, request: BitcoinRpcRequest { tx_id: TX_B, ... } })
   → pending yield Y_B created; request key K_B stored.
5. Malicious node calls respond_verify_foreign_tx(K_B, { payload_hash: H_A, signature: S_A }).
   - assert_caller_is_attested_participant_and_protocol_active() → passes (node is attested).
   - verify_ecdsa_signature(S_A, H_A, root_pk_D) → passes (S_A is a valid signature for H_A).
   - resolve_yields_for(pending_verify_foreign_tx_requests, K_B, serialize({ H_A, S_A })) → Y_B resolved.
6. Bob's transaction resumes with VerifyForeignTransactionResponse { payload_hash: H_A, signature: S_A }.
   H_A encodes TX_A's block-hash, not TX_B's. Bob's bridge contract, checking only signature validity,
   accepts this as attestation of TX_B and credits the corresponding inbound transfer.
```

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
