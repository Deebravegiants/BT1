Audit Report

## Title
Stale Cross-Chain Rate Consumed Without Freshness Check Allows Depositors to Mint Excess rsETH — (File: contracts/cross-chain/CrossChainRateReceiver.sol)

## Summary

`CrossChainRateReceiver` stores the rsETH/ETH exchange rate received asynchronously from L1 via LayerZero and records a `lastUpdated` timestamp, but neither `getRate()` nor any caller ever validates that timestamp against a maximum staleness threshold. Both `RSETHPoolV3` and `RSETHPoolNoWrapper` divide deposited ETH by this potentially stale rate to compute rsETH output, so any window between a real L1 price increase and the next L2 rate update allows an unprivileged depositor to receive more rsETH than the current L1 price entitles them to, extracting value from existing rsETH holders.

## Finding Description

`CrossChainRateReceiver.lzReceive()` writes the incoming rate and a timestamp unconditionally:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol L95-97
rate = _rate;
lastUpdated = block.timestamp;
```

`getRate()` returns the stored value with no freshness guard:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol L103-105
function getRate() external view returns (uint256) {
    return rate;
}
```

`RSETHPoolV3.viewSwapRsETHAmountAndFee()` and `RSETHPoolNoWrapper.viewSwapRsETHAmountAndFee()` both call `getRate()` and divide by it:

```solidity
// RSETHPoolV3.sol L304-307 / RSETHPoolNoWrapper.sol L282-285
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

Because rsETH continuously accrues restaking rewards, the L1 price rises monotonically between keeper updates. During any update gap the stored `rate` is lower than the true current rate, so the division yields a larger `rsETHAmount` than is warranted. The `dailyMintLimit` in `RSETHPoolV3` caps total daily issuance but does not prevent the per-deposit over-issuance within that cap. `whenNotPaused` is a reactive admin control, not a preventive staleness guard. No other check in either pool contract consults `lastUpdated`.

**Exploit path:**
1. L1 rsETH price rises from 1.05 ETH to 1.10 ETH; L2 `rate` remains 1.05e18.
2. Attacker calls `RSETHPoolV3.deposit{value: 100 ether}("ref")` or `RSETHPoolNoWrapper.deposit("ref")`.
3. `viewSwapRsETHAmountAndFee(100 ether)` computes `rsETHAmount = 100e18 * 1e18 / 1.05e18 ≈ 95.24 rsETH` instead of the correct `≈ 90.91 rsETH`.
4. Attacker receives ≈ 4.33 excess rsETH (≈ 4.76 ETH of value at the current rate).
5. Attacker bridges the excess rsETH to L1 and redeems it, extracting ETH from the protocol.

## Impact Explanation

**High — Theft of unclaimed yield.** The excess rsETH minted at the stale rate represents restaking yield that has accrued to existing rsETH holders but has not yet been reflected on L2. Each such deposit transfers a portion of that yield to the attacker. The attack is repeatable across every update gap and across both pool contracts, making cumulative extraction proportional to the frequency and magnitude of L1 price increases.

## Likelihood Explanation

The rate update depends entirely on an off-chain keeper dispatching a LayerZero message. Any delay — network congestion, keeper downtime, or deliberate inaction — creates an exploitable window. Because rsETH accrues rewards continuously, the L2 rate is perpetually slightly stale between updates. An attacker needs only to monitor the L1 rsETH price (e.g., via `LRTOracle`) and the L2 `CrossChainRateReceiver.rate`, then call `deposit()` when a discrepancy exists. No privileged access is required; the attack is executable by any EOA.

## Recommendation

Enforce a maximum staleness threshold in both pool contracts before accepting the rate. Expose `lastUpdated` through the oracle interface (or a separate `IRateReceiver` interface) and revert in `viewSwapRsETHAmountAndFee` if the rate is too old:

```solidity
uint256 constant MAX_RATE_AGE = 24 hours;
require(block.timestamp - IRateReceiver(rsETHOracle).lastUpdated() <= MAX_RATE_AGE, "Stale rate");
```

Alternatively, add the staleness check directly inside `CrossChainRateReceiver.getRate()` so all consumers are protected automatically. The `lastUpdated` field already exists at line 16 of `CrossChainRateReceiver.sol` but is never read by any caller.

## Proof of Concept

**Foundry fork test outline:**

```solidity
// 1. Fork the target L2 at a block where rate < current L1 price.
// 2. Record CrossChainRateReceiver.rate (stale) and the true L1 rate from LRTOracle.
// 3. Compute expected rsETH at stale rate vs. correct rate.
// 4. Call RSETHPoolV3.deposit{value: 100 ether}("ref") as an unprivileged address.
// 5. Assert wrsETH.balanceOf(attacker) == 100e18 * 1e18 / staleRate (over-issuance confirmed).
// 6. Assert the difference vs. 100e18 * 1e18 / currentRate > 0 (excess rsETH quantified).
// 7. Repeat for RSETHPoolNoWrapper.deposit to confirm both pools are affected.
```

The same test can be run as a differential fuzz test: for any `rate_stale < rate_current`, assert that `deposit()` with `rate_stale` always mints strictly more rsETH than with `rate_current`, and that the difference is non-zero and proportional to the price gap.
