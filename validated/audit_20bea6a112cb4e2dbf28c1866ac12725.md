**Key finding**: The `schnorr_2p` `key_blob_v1_t` struct does not store `Qi` (the per-party share point) in the blob — only the global `Q_compressed`. ### Title
Missing Session-Binding Check in `attach_private_scalar` Allows Cross-Session Key Blob Acceptance — (`src/cbmpc/api/schnorr2pc.cpp`)

### Summary
`schnorr_2p::attach_private_scalar` validates only that the caller-supplied scalar `x` satisfies `x*G == Qi_provided`, but never checks that `Qi_provided` matches the share point that was established for the public blob during its DKG session. Because the schnorr_2p public blob does not store the per-party Qi, the function has no stored value to compare against. An API caller who controls two independent DKG sessions can mix session B's scalar and Qi into session A's public blob and receive `SUCCESS`, producing a structurally valid but cryptographically invalid key blob.

### Finding Description

The `schnorr_2p` `key_blob_v1_t` struct serializes only `{version, role, curve, Q_compressed, x_share}`: [1](#0-0) 

After `detach_private_scalar`, the public blob retains `Q_compressed` (the global key) but carries no per-party Qi. [2](#0-1) 

`attach_private_scalar` then validates only the self-consistency of the caller-supplied `(x, Qi)` pair: [3](#0-2) 

There is no check that `Qi_provided` equals the Qi that was committed to during session A's DKG. The function then writes `x` into the blob and returns `SUCCESS`: [4](#0-3) 

By contrast, `schnorr_mp::attach_private_scalar` stores `Qis_compressed` in the blob and enforces the session-binding check explicitly: [5](#0-4) 

### Impact Explanation

The output of `attach_private_scalar` is a blob accepted as valid by the library, with `Q` from session A and `x_share` from session B. This blob passes all downstream deserialization checks in `blob_to_key` (range check on `x_share`, point decode of `Q`): [6](#0-5) 

The blob can be passed to `sign`. The signing protocol will execute interactively with P2 (who holds session A's `x2_A`), consuming both parties' protocol state, before the final BIP340 verify step detects the mismatch. The accepted-bad-output is the cross-session blob returned with `SUCCESS` from `attach_private_scalar`; this is a public API validation bypass that produces invalid cryptographic output.

### Likelihood Explanation

Any API caller who participates in (or observes the outputs of) two independent DKG sessions can perform this cross-session substitution without any privileged access. The call sequence is entirely within the documented public API surface (`detach_private_scalar` → `attach_private_scalar`). No threshold collusion, leaked keys, or compromised host memory is required.

### Recommendation

Store the per-party `Qi_self` in the schnorr_2p public blob (analogous to `Qis_compressed` in the schnorr_mp blob), and in `attach_private_scalar` enforce that the caller-supplied `public_share_compressed` equals the stored `Qi_self` before accepting the scalar. This mirrors the existing guard in `schnorr_mp::attach_private_scalar`.
