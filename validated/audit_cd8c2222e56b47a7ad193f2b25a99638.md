Audit Report

## Title
Stale `rsETHPrice` Used in Deposit Minting Allows Theft of Unclaimed Yield from Existing rsETH Holders - (File: `contracts/LRTDepositPool.sol`, `contracts/LRTOracle.sol`)

## Summary
`LRTDepositPool.getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()`, a stored state variable that is only updated when `updateRSETHPrice()` is explicitly called. Because neither `depositETH()` nor `depositAsset()` refreshes this price before computing the mint amount, any deposit made while staking rewards have accrued since the last update mints more rsETH than the depositor's ETH value justifies, diluting the unrealised yield that belongs to existing holders.

## Finding Description
`getRsETHAmountToMint` computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.getAssetPrice(asset)` is a live Chainlink-backed call, but `lrtOracle.rsETHPrice()` reads the stored state variable `uint256 public override rsETHPrice` (LRTOracle.sol L28), which is only written inside `_updateRsETHPrice()`. That internal function is reached only via the public `updateRSETHPrice()` or the manager-only `updateRSETHPriceAsManager()` — neither is called inside `depositETH()` (LRTDepositPool.sol L76-93) or `depositAsset()` (LRTDepositPool.sol L99-118).

Between two `updateRSETHPrice()` calls, EigenLayer restaking yield and LST rebases increase `totalETHInProtocol` while `rsethSupply` remains constant, so the true price (`totalETHInProtocol / rsethSupply`) rises above the stored `rsETHPrice`. A depositor who acts in this window receives rsETH priced at the stale lower rate, minting a surplus that represents a claim on the unrealised yield of existing holders.

The `minRSETHAmountExpected` slippage parameter protects the depositor from receiving *less* than expected; it does not prevent the depositor from receiving *more* than warranted. The `pricePercentageLimit` guard inside `_updateRsETHPrice()` (LRTOracle.sol L252-266) limits how fast the stored price can be updated, but it does not constrain how stale the stored price may be at deposit time — it can actually extend the exploitable window by delaying the price update for non-managers.

Exploit path (no privileged role required):
1. Monitor `_getTotalEthInProtocol()` vs. `rsETHPrice * rsethSupply` for a profitable gap.
2. Call `LRTDepositPool.depositETH{value: X}(0, "")` — mints rsETH at the stale lower price.
3. Wait for anyone (keeper, bot, or the attacker) to call `LRTOracle.updateRSETHPrice()`.
4. Attacker's rsETH is now worth more than the deposited ETH; the surplus was extracted from existing holders' unrealised yield.

## Impact Explanation
**High — Theft of unclaimed yield.** Existing rsETH holders accumulate yield as `totalETHInProtocol` grows between price updates. A deposit made before the update dilutes that unrealised yield: the attacker's surplus rsETH captures ETH value that should have accrued exclusively to prior holders. The theft scales linearly with deposit size; a flash-loan-sized deposit can capture the majority of all rewards accrued since the last `updateRSETHPrice()` call in a single transaction.

## Likelihood Explanation
**Medium.** `updateRSETHPrice()` is triggered off-chain by a keeper or bot, not on every block. Any gap between reward accrual and the next price update is an exploitable window. The gap is observable on-chain by comparing `ILRTDepositPool.getTotalETHInPool()` against `rsETHPrice * rsethSupply`. No privileged access is required; `depositETH()` is fully public and `nonReentrant`/`whenNotPaused` guards do not prevent this attack.

## Recommendation
Call `_updateRsETHPrice()` (or an equivalent internal price refresh) at the start of `_beforeDeposit()`, before `getRsETHAmountToMint()` is invoked. This ensures every deposit uses a price that reflects the current `totalETHInProtocol`. Alternatively, compute the mint amount inline using a freshly derived price (`totalETHInProtocol / rsethSupply`) rather than reading the stored `rsETHPrice` state variable.

## Proof of Concept

**Initial state:**
```
totalETHInProtocol = 1 100 ETH  (100 ETH staking rewards accrued, not yet reflected)
rsethSupply        = 1 000 rsETH
rsETHPrice (stored)= 1.000 ETH  (stale — true value is 1.100 ETH)
```

**Step 1 — Attacker calls `depositETH{value: 10_000 ETH}(0, "")`:**
```
rsethAmountToMint = 10_000e18 * 1e18 / 1.000e18 = 10_000 rsETH
correct amount    = 10_000 / 1.100              =  9_090.9 rsETH
surplus minted    =                                  909.1 rsETH
```

**Step 2 — Anyone calls `LRTOracle.updateRSETHPrice()`:**
```
totalETHInProtocol = 11_100 ETH
rsethSupply        = 11_000 rsETH
newRsETHPrice      = 11_100 / 11_000 = 1.00909 ETH
```

**Step 3 — Attacker redeems:**
```
10_000 rsETH × 1.00909 = 10_090.9 ETH   (deposited 10_000 ETH → profit 90.9 ETH)
```

**Existing holders' loss:**
```
1_000 rsETH × 1.00909 = 1_009.09 ETH   (should have been 1_000 × 1.100 = 1_100 ETH)
loss = 90.9 ETH  (≈ 90.9 % of the 100 ETH accrued reward stolen)
```

**Foundry fork test outline:**
1. Fork mainnet at a block where `rsETHPrice` is known to lag `totalETHInProtocol`.
2. Record `rsETHPrice` and compute `totalETHInProtocol / rsethSupply` to confirm the gap.
3. Impersonate an EOA; call `depositETH{value: largeAmount}(0, "")`.
4. Assert `rsETH.balanceOf(attacker) > largeAmount * 1e18 / (totalETHInProtocol / rsethSupply)`.
5. Call `updateRSETHPrice()`; assert attacker's rsETH redeemable value exceeds `largeAmount`.
6. Assert existing holders' rsETH redeemable value is less than `rsethSupply * (totalETHInProtocol_before / rsethSupply_before)`.
