Audit Report

## Title
Deposit output amount silently rounds to zero with no output guard, causing depositor fund loss — (`contracts/pools/RSETHPool.sol`, `contracts/pools/RSETHPoolNoWrapper.sol`, `contracts/pools/RSETHPoolV3.sol`, `contracts/agETH/AGETHPoolV3.sol`)

## Summary

Every `deposit` function across all four pool contracts computes the output token amount via integer division that can silently produce zero. When it does, the depositor's ETH or ERC-20 tokens are accepted and permanently absorbed into the pool's bridgeable balance, while the depositor receives zero rsETH/agETH in return. No output-side guard exists to reject such a deposit.

## Finding Description

All pool contracts share the same output-amount formula. For ETH deposits:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

For token deposits:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

Because `rsETHToETHrate` is always ≥ `1e18` (rsETH accrues value over time), any ETH deposit where `amountAfterFee < rsETHToETHrate / 1e18` produces `rsETHAmount = 0` via integer truncation. The same applies to token deposits when `amountAfterFee * tokenToETHRate < rsETHToETHrate`.

The deposit functions guard only against a zero *input*:

```solidity
if (amount == 0) revert InvalidAmount();
```

There is no corresponding guard on the *output*. After computing a zero output, execution continues unconditionally:

- `RSETHPool.sol` L275: `IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount)` — transfers 0 silently.
- `RSETHPoolNoWrapper.sol` L241: `rsETH.safeTransfer(msg.sender, rsETHAmount)` — transfers 0 silently.
- `RSETHPoolV3.sol` L262: `wrsETH.mint(msg.sender, rsETHAmount)` — mints 0 silently.
- `AGETHPoolV3.sol` L125: `agETH.mint(msg.sender, agETHAmount)` — mints 0 silently.

For ETH deposits, `msg.value` is already in the contract before the output is computed. For token deposits, `safeTransferFrom` executes at lines `RSETHPool.sol:296`, `RSETHPoolNoWrapper.sol:262`, `RSETHPoolV3.sol:284`, and `AGETHPoolV3.sol:145` — before the output amount is computed — so the tokens are already taken when the zero output is produced. No rescue or recovery function exists anywhere in the contracts. The deposited assets become part of the pool's bridgeable balance and are eventually bridged to L1 with no depositor claim attached.

## Impact Explanation

**Low — Contract fails to deliver promised returns.**

A depositor who sends a dust ETH amount (e.g., 1 wei) or a sufficiently small token amount receives 0 rsETH/agETH. Their deposited assets are absorbed into the pool's bridgeable balance with no recovery path. The protocol does not lose funds, but the depositor does. This matches the allowed Low impact: "Contract fails to deliver promised returns, but doesn't lose value."

## Likelihood Explanation

**Low.** For currently supported 18-decimal ETH-correlated LSTs, the truncation threshold is a few wei of ETH — amounts no rational user would intentionally deposit. However, `addSupportedToken` imposes no restriction on token decimals; if a 6-decimal token were added, the truncation threshold rises to potentially thousands of base units, making accidental zero-output deposits realistic for normal users. The exploit is triggerable by any unprivileged depositor with no special preconditions beyond sending a dust amount.

## Recommendation

Add an output-amount guard immediately after computing `rsETHAmount` / `agETHAmount` in every deposit function and its corresponding view function:

```solidity
if (rsETHAmount == 0) revert InvalidAmount();
```

Apply this in all eight affected locations: both `deposit` overloads and both `viewSwap*AmountAndFee` overloads in each of the four pool contracts. This mirrors the zero-amount guard already present on the input side and allows callers to detect the condition off-chain before submitting a transaction.

## Proof of Concept

**ETH deposit path (`RSETHPool.sol`):**

1. `rsETHToETHrate` = `1.05e18` (rsETH has accrued 5% above peg).
2. Depositor calls `deposit{value: 1}("")` (1 wei ETH).
3. `viewSwapRsETHAmountAndFee(1)`:
   - `fee = 1 * feeBps / 10_000 = 0`
   - `amountAfterFee = 1`
   - `rsETHAmount = 1 * 1e18 / 1.05e18 = 0` (integer truncation)
4. `feeEarnedInETH += 0`.
5. `IERC20(address(wrsETH)).safeTransfer(msg.sender, 0)` — succeeds silently.
6. Depositor's 1 wei is now in the pool's ETH balance, counted as bridgeable, and will be sent to L1. Depositor holds no rsETH and has no claim.

**Foundry fuzz test plan:**

```solidity
function testFuzz_ZeroOutputDeposit(uint256 amount) public {
    // Bound amount to the truncation range: 1 to (rsETHToETHrate / 1e18) - 1
    uint256 rate = pool.getRate(); // e.g., 1.05e18
    amount = bound(amount, 1, rate / 1e18 - 1);
    
    (uint256 rsETHAmount,) = pool.viewSwapRsETHAmountAndFee(amount);
    assertEq(rsETHAmount, 0);
    
    uint256 balanceBefore = wrsETH.balanceOf(address(depositor));
    vm.deal(address(depositor), amount);
    vm.prank(depositor);
    pool.deposit{value: amount}("");
    
    // Depositor received nothing
    assertEq(wrsETH.balanceOf(address(depositor)), balanceBefore);
    // Pool absorbed the ETH
    assertGt(address(pool).balance, 0);
}
```
