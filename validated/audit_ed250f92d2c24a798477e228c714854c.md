### Finding Description
`buyToken0InBinSpecifiedIn` uses a three-stage approach to find the correct bin position for an exact-input token1→token0 swap:

1. **Analytical quadratic solution** — accurate but subject to integer rounding.
2. **Linear proportional refinement** (lines 598–618) — scales `targetPos` by `ceil(delta × remaining / totalIn1Scaled)` and recomputes `out0Scaled` and `totalIn1Scaled`.
3. **Forced assignment** (lines 620–622) — if `totalIn1Scaled` is still below `remaining`, sets `totalIn1Scaled = remaining` without touching `out0Scaled`.

The refinement is linear, but the cost function `totalIn1Scaled(targetPos)` is quadratic (product of a linear `out0Scaled` and a linear `avgPrice`). When the bin has a non-trivial price spread, the linear refinement systematically underestimates the required input, leaving `totalIn1Scaled < remaining` after step 2. The forced assignment then fires.

After the forced assignment, `totalIn1Scaled == remaining`, so the scale-down guard at line 624 (`if (totalIn1Scaled > remaining)`) evaluates to false and is skipped. `out0Scaled` retains the value computed for the smaller, under-scaled `targetPos`.

The final state update at lines 643–644 then:
- Deducts all of `remaining` from `state.amountSpecifiedRemainingScaled` (full input consumed).
- Adds the under-computed `out0Scaled` to `state.amountCalculatedScaled` (reduced output credited).
