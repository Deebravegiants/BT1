### Title
`KernelDepositPool` Has No Mechanism to Update Stale `kernelToken`/`rewardsToken` References After Token Migration, Permanently Freezing Staked Funds — (File: `contracts/KERNEL/KernelDepositPool.sol`)

---

### Summary

`KernelDepositPool` stores `kernelToken` and `rewardsToken` as plain storage variables set once at initialization, with no admin setter to update them. If either token is migrated to a new contract address and the old contract is paused or deprecated, all staked KERNEL principal locked in pending withdrawals and all accrued but unclaimed rewards become permanently unrecoverable.

---

### Finding Description

`KernelDepositPool.initialize()` sets `kernelToken` and `rewardsToken` once and never again:

```solidity
// contracts/KERNEL/KernelDepositPool.sol:259-271
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    ...
    kernelToken  = IERC20(_kernelToken);
    rewardsToken = IERC20(_rewardToken);
}
```

A grep across the entire codebase for `setKernelToken`, `setRewardsToken`, `updateKernelToken`, or `updateRewardsToken` returns zero matches. No upgrade path for these two addresses exists anywhere in the contract.

The two critical withdrawal paths both hard-depend on these stale references:

```solidity
// contracts/KERNEL/KernelDepositPool.sol:376
kernelToken.safeTransfer(msg.sender, withdrawal.amount);   // claimWithdrawal()

// contracts/KERNEL/KernelDepositPool.sol:387
rewardsToken.safeTransfer(msg.sender, rewardAmount);       // getReward()
```

If the KERNEL token is ever migrated to a new address (e.g., a security patch, rebranding, or supply-model change) and the old contract is paused as part of that migration, every `claimWithdrawal()` call will revert. Because the withdrawal delay can be up to `MAX_WITHDRAWAL_DELAY = 30 days`, users who have already initiated withdrawals are fully exposed: their principal is already debited from `balanceOf` and `totalKernelStaked`, but the transfer out will permanently fail. ### Impact Explanation

**Critical — Permanent freezing of funds.**  
Any KERNEL tokens that have been staked and for which a withdrawal has been initiated (but not yet claimed) are debited from the user's `balanceOf` and `totalKernelStaked` at `initiateWithdrawal()` time. If `kernelToken` becomes stale before `claimWithdrawal()` is called, the tokens are gone from the user's accounting but can never be transferred back. There is no escape hatch, no emergency withdrawal, and no admin rescue function in the contract.

**Medium — Permanent freezing of unclaimed yield.**  
All accrued rewards tracked in `rewards[user]` are paid out exclusively via `rewardsToken.safeTransfer()`. If `rewardsToken` becomes stale, every `getReward()` call reverts and accumulated yield is permanently frozen. [5](#0-4) [6](#0-5) 

---

### Likelihood Explanation

**Medium.**  
The KERNEL token is a recently launched token in the Kelp DAO ecosystem. Token migrations are a well-documented pattern in DeFi (bug fixes, supply-model changes, rebranding). The `KernelDepositPool` is deployed on BSC (confirmed in `README.md`) where the KERNEL token is also deployed. The withdrawal delay of up to 30 days creates a guaranteed window during which users are exposed: once `initiateWithdrawal()` is called, the user cannot cancel and must wait for `unlockTime` before calling `claimWithdrawal()`. Any migration occurring within that window permanently freezes the principal. [7](#0-6) [8](#0-7) 

---

### Recommendation

**Short term:** Add admin-controlled setter functions for both `kernelToken` and `rewardsToken`, gated behind `DEFAULT_ADMIN_ROLE`, with a non-zero address check. Before allowing the update, validate that no pending (unclaimed) withdrawals exist, or provide a migration path that settles them first.

**Long term:** Develop and test a coordinated migration strategy for the KERNEL token and `KernelDepositPool` together, ensuring that any token address change is atomic with the settlement of all in-flight withdrawals and reward balances.

---

### Proof of Concept

1. Alice calls `stake(1000e18)` → `kernelToken.safeTransferFrom(alice, pool, 1000e18)` succeeds; `balanceOf[alice] = 1000e18`.
2. Alice calls `initiateWithdrawal(1000e18)` → `balanceOf[alice] = 0`, `totalKernelStaked -= 1000e18`; a `Withdrawal` record is created with `unlockTime = block.timestamp + withdrawalDelay` (up to 30 days).
3. The KERNEL token is migrated to a new contract address; the old KERNEL contract is paused as part of the migration.
4. After `unlockTime`, Alice calls `claimWithdrawal(withdrawalId)`.
5. Execution reaches `kernelToken.safeTransfer(alice, 1000e18)` — this calls `transfer()` on the old, paused KERNEL contract, which reverts.
6. Alice's 1000 KERNEL tokens are permanently frozen in `KernelDepositPool`. Her `balanceOf` is already zero; she cannot re-initiate a withdrawal. There is no admin rescue path. [9](#0-8) [5](#0-4) [6](#0-5)

    function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
        if (_amount == 0) revert AmountZero();
        if (balanceOf[msg.sender] < _amount) revert InsufficientStakedBalance();
        if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();

        balanceOf[msg.sender] -= _amount;
        totalKernelStaked -= _amount;

        // Create a withdrawal record
        uint256 withdrawalId = ++withdrawalCounter;
        uint256 unlockTime = block.timestamp + withdrawalDelay;

        withdrawals[withdrawalId] = Withdrawal({
            user: msg.sender, amount: _amount, unlockTime: unlockTime, claimed: false, withdrawalId: withdrawalId
        });
        userWithdrawalIds[msg.sender].push(withdrawalId);

        emit WithdrawalInitiated(msg.sender, _amount, withdrawalId, unlockTime);
    }

    function claimWithdrawal(uint256 _withdrawalId) external nonReentrant {
        Withdrawal storage withdrawal = withdrawals[_withdrawalId];

        if (withdrawal.user == address(0)) {
            revert WithdrawalDoesNotExist();
        }

        if (withdrawal.user != msg.sender) {
            revert NotYourWithdrawal();
        }

        if (block.timestamp < withdrawal.unlockTime) {
            revert WithdrawalNotReady();
        }

        if (withdrawal.claimed) {
            revert WithdrawalAlreadyClaimed();
        }

        withdrawal.claimed = true;

        // Remove the withdrawal ID from the user's list of withdrawal IDs
        uint256[] storage userWithdrawalIdsArray = userWithdrawalIds[msg.sender];
        for (uint256 i = 0; i < userWithdrawalIdsArray.length; ++i) {
            if (userWithdrawalIdsArray[i] == _withdrawalId) {
                userWithdrawalIdsArray[i] = userWithdrawalIdsArray[userWithdrawalIdsArray.length - 1];
                userWithdrawalIdsArray.pop();
                break;
            }
        }

        // Transfer the KERNEL tokens from contract to user
        kernelToken.safeTransfer(msg.sender, withdrawal.amount);

        emit WithdrawalClaimed(msg.sender, withdrawal.amount, _withdrawalId);
    }
