### Title
`LRTDepositPool._beforeDeposit` Allows Zero rsETH Minting — Depositor Loses Assets Without Receiving Shares - (File: contracts/LRTDepositPool.sol)

### Summary

`LRTDepositPool._beforeDeposit` computes `rsethAmountToMint` via integer division and only guards against it being less than `minRSETHAmountExpected`. When a caller passes `minRSETHAmountExpected = 0`, the guard is trivially satisfied even when `rsethAmountToMint = 0`. The depositor's ETH or LST is transferred into the protocol, but zero rsETH is minted in return.

### Finding Description

`getRsETHAmountToMint` computes:

```
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

This is integer division. Whenever `amount * assetPrice < rsETHPrice`, the result truncates to zero.

`_beforeDeposit` then checks:

```solidity
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

There is no explicit `require(rsethAmountToMint > 0)`. If the caller supplies `minRSETHAmountExpected = 0` (a valid argument), the condition `0 < 0` is false and execution continues. For `depositETH`, the ETH is already in the contract via `msg.value`; for `depositAsset`, the LST is pulled via `safeTransferFrom`. In both cases `_mintRsETH(0)` is called, minting nothing to the depositor.

The condition `amount * assetPrice < rsETHPrice` is satisfied for any deposit where `amount < rsETHPrice / assetPrice`. Since rsETH appreciates over time (rsETHPrice grows above 1e18), this threshold rises continuously. With `minAmountToDeposit` initialized to zero and only settable by admin, any non-zero dust deposit can trigger this path.

### Impact Explanation

A depositor who calls `depositETH` or `depositAsset` with `minRSETHAmountExpected = 0` and a deposit amount below the integer-division threshold loses their entire deposited value — ETH or LST is absorbed into the protocol TVL, increasing the share price for all existing rsETH holders, while the depositor receives zero rsETH. This is a direct, permanent loss of the depositor's funds with no recovery path.

Impact: **Low** — Contract fails to deliver promised returns; the depositor loses value while the protocol does not.

### Likelihood Explanation

- `minAmountToDeposit` is initialized to `0` and is only set by admin; in the default state any dust deposit is accepted.
- rsETH price grows above 1e18 as yield accrues, so the zero-mint threshold rises over time.
- Any depositor who omits or zeroes `minRSETHAmountExpected` — including integrating contracts that do not set slippage — is silently vulnerable.
- No privileged role or front-running is required; the condition is reachable by any unprivileged depositor in normal protocol operation.

Likelihood: **Low** (dust amounts only in the non-front-run case; larger amounts require price manipulation).

### Recommendation

Add an explicit zero-share guard immediately after computing `rsethAmountToMint` in `_beforeDeposit`:

```solidity
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

if (rsethAmountToMint == 0) {
    revert ZeroRsETHMinted();
}

if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

This mirrors the fix recommended in the referenced Spearbit report and ensures no depositor can lose assets while receiving zero shares.

### Proof of Concept

Assume:
- `rsETHPrice = 1.05e18` (rsETH has appreciated 5%)
- `assetPrice(ETH) = 1e18`
- `minAmountToDeposit = 0` (default)

A depositor calls:
```
depositETH{value: 1}(minRSETHAmountExpected: 0, referralId: "")
```

Execution path:
1. `_beforeDeposit(ETH, 1, 0)` is called.
2. `depositAmount = 1 >= minAmountToDeposit = 0` → passes.
3. `getRsETHAmountToMint(ETH, 1)` → `(1 * 1e18) / 1.05e18 = 0` (integer division).
4. `rsethAmountToMint (0) < minRSETHAmountExpected (0)` → `false` → no revert.
5. `_mintRsETH(0)` → `IRSETH.mint(msg.sender, 0)` → depositor receives 0 rsETH.
6. The 1 wei ETH remains in the deposit pool, increasing TVL and benefiting all existing rsETH holders.

The depositor permanently loses 1 wei with no recourse. [1](#0-0) [2](#0-1) [3](#0-2)

        uint256 minRSETHAmountExpected

        private

            revert InvalidAmountToDeposit();
        }

        if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
            revert MaximumDepositLimitReached();
        }

        rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

        if (rsethAmountToMint < minRSETHAmountExpected) {
            revert MinimumAmountToReceiveNotMet();
        }
    }
