### Title
`compare_retrospective_block_hash` Disabled in Production Config Silently Propagates Unverified Batcher Block Hash into Block Execution Context — (`crates/apollo_deployments/resources/app_configs/consensus_manager_config.json`)

---

### Summary

The production deployment config sets `compare_retrospective_block_hash: false`, disabling the cross-check between the batcher's locally-computed block hash and the state sync's canonical block hash. When this flag is `false`, `retrospective_block_hash()` in `utils.rs` silently returns the batcher's hash without verifying it against state sync. That hash is then embedded in `ValidateBlockInput.retrospective_block_hash` and used to seed the `get_block_hash` syscall execution context for every validated block.

---

### Finding Description

In `crates/apollo_consensus_orchestrator/src/utils.rs`, the `retrospective_block_hash` function fetches the block hash from both the batcher and state sync, but only enforces agreement when `compare_retrospective_block_hash` is `true`:

```rust
// utils.rs L379-396
let state_sync_block_hash = state_sync_client.get_block_hash(block_number).await?;
let batcher_block_hash   = batcher_client.get_block_hash(block_number).await?;

if compare_retrospective_block_hash && state_sync_block_hash != batcher_block_hash {
    // returns Err(HashMismatch)
}
Ok(Some(BlockHashAndNumber { number: block_number, hash: batcher_block_hash }))
```

When the flag is `false`, `state_sync_block_hash` is fetched and then silently discarded; `batcher_block_hash` is returned unconditionally regardless of divergence. [1](#0-0) 

The production app config explicitly sets this flag to `false`:

```json
// consensus_manager_config.json L60
"consensus_manager_config.context_config.dynamic_config.compare_retrospective_block_hash": false
``` [2](#0-1) 

The schema default is `true`: [3](#0-2) 

The returned hash is passed directly into `ValidateBlockInput.retrospective_block_hash` inside `initiate_validation`, which is called on every proposal validation path:

```rust
// validate_proposal.rs L455-466
let input = ValidateBlockInput {
    proposal_id,
    deadline: ...,
    retrospective_block_hash: retrospective_block_hash(
        batcher.clone(), state_sync_client, init, compare_retrospective_block_hash,
    ).await?,
    block_info: convert_to_sn_api_block_info(init)?,
};
batcher.validate_block(input.clone()).await?;
``` [4](#0-3) 

The `retrospective_block_hash` field in `ValidateBlockInput` is the `STORED_BLOCK_HASH_BUFFER`-blocks-back hash that the blockifier stores in the execution context to serve `get_block_hash` syscall calls. If the batcher's hash diverges from the canonical chain, every contract calling `get_block_hash` during that block's execution receives the wrong value.

---

### Impact Explanation

If the batcher's locally-computed block hash for block `N - STORED_BLOCK_HASH_BUFFER` diverges from the canonical chain (e.g., due to a reorg in the sequencer's own committed chain, a non-finality incident, or a hash-computation bug), the wrong hash is silently embedded in the execution context. Every `get_block_hash` syscall executed during that block returns the wrong value. This is a wrong syscall/execution result from blockifier logic for accepted input — matching the Critical impact tier.

Because `compare_retrospective_block_hash` is a **dynamic config** (updated live via `config_manager_client`), the check can be toggled off at any time without a node restart, and the production baseline is already `false`. [5](#0-4) [6](#0-5) 

---

### Likelihood Explanation

The batcher and state sync diverge whenever:
- The sequencer's own committed chain forks (revert + rebuild scenario)
- A non-finality incident causes state sync to lag or diverge
- A bug in the batcher's async commitment pipeline produces a stale or wrong hash

With the flag disabled, none of these divergences are detected or blocked. The analog to the external bug is exact: the external system used 30 confirmations instead of finalized blocks and was "historically safe" but offered no guarantees in edge cases. Here, the batcher's locally-committed hash is used instead of the cross-verified canonical hash, and the production config has permanently disabled the guard.

---

### Recommendation

1. Set `compare_retrospective_block_hash: true` in the production app config (`consensus_manager_config.json` line 60). The schema default is already `true`; the production override should be removed or flipped.
2. Consider making the flag non-dynamic (static config) so it cannot be silently toggled off at runtime.
3. If the flag must remain `false` for operational reasons (e.g., state sync lag), use the state sync hash as the authoritative source rather than the batcher hash, since state sync represents the canonical chain.

---

### Proof of Concept

1. Production node starts with `compare_retrospective_block_hash: false` (line 60 of `consensus_manager_config.json`).
2. A revert-and-rebuild cycle causes the batcher to recompute block hashes; due to async commitment ordering, block `N - STORED_BLOCK_HASH_BUFFER` temporarily holds a stale hash in the batcher's cache.
3. A new proposal arrives for height `N`. `retrospective_block_hash()` fetches both hashes; they differ. Because the flag is `false`, the mismatch is silently ignored and `batcher_block_hash` (the stale value) is returned.
4. `initiate_validation` passes this stale hash as `ValidateBlockInput.retrospective_block_hash` to the batcher.
5. The blockifier seeds the execution context with the stale hash. Every contract calling `get_block_hash(N - STORED_BLOCK_HASH_BUFFER)` during block `N` receives the wrong value.
6. The proposal is accepted by consensus (the `ProposalFin` comparison checks `partial_block_hash`, not the retrospective hash), so the wrong syscall result is committed to the canonical chain.
