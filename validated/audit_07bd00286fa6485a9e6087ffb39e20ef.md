### Title
User-Controlled `confirmations` Parameter in `verify_foreign_transaction` Bypasses Bitcoin Finality Enforcement - (File: crates/contract/src/lib.rs, crates/foreign-chain-inspector/src/bitcoin/inspector.rs)

### Summary

The `verify_foreign_transaction` contract method accepts a user-supplied `BlockConfirmations` value inside `BitcoinRpcRequest` with no minimum threshold enforced anywhere in the contract or node pipeline. An unprivileged caller can set `confirmations: 0`, causing MPC nodes to sign a Bitcoin transaction that has zero on-chain confirmations (i.e., is still in the mempool or has just been mined), producing a valid MPC-attested signature over an unfinalized foreign-chain event. Any bridge contract consuming this signature without independently re-checking the `confirmations` field will process a double-spendable inbound transfer.

### Finding Description

`BitcoinRpcRequest` carries a `confirmations: BlockConfirmations` field that is entirely caller-controlled: [1](#0-0) 

The on-chain `verify_foreign_transaction` entry point performs no validation of this field — it only checks domain purpose, gas, deposit, and whether the chain is whitelisted: [2](#0-1) 

The MPC node's `execute_foreign_chain_request` passes the caller-supplied value directly to the Bitcoin inspector as the finality threshold: [3](#0-2) 

Inside the inspector, the check is `block_confirmations_threshold <= transaction_block_confirmation`. With `threshold = 0`, this is trivially satisfied for any `u64` confirmation count, including 0 (mempool transaction): [4](#0-3) 

The signed payload (`ForeignTxSignPayloadV1`) does include the full `ForeignChainRpcRequest` (and therefore the `confirmations` value), so the hash commits to it: [5](#0-4) 

However, the MPC contract itself never validates the `confirmations` value before accepting the request, and the ABI schema confirms the only constraint is `minimum: 0`: [6](#0-5) 

### Impact Explanation

An attacker who controls a Bitcoin transaction can:

1. Broadcast a Bitcoin transaction sending funds to a bridge deposit address.
2. Immediately call `verify_foreign_transaction` with `confirmations: 0` before any block confirmation occurs.
3. MPC nodes query `getrawtransaction` — the transaction is visible in the mempool with 0 confirmations; `0 <= 0` passes the finality check.
4. The MPC network produces a valid threshold signature over `SHA256(borsh(ForeignTxSignPayloadV1 { request: Bitcoin{confirmations:0,...}, values: [...] }))`.
5. The attacker submits this signature to the NEAR bridge contract to claim the inbound transfer.
6. The attacker then double-spends the original Bitcoin transaction (RBF or miner collusion), invalidating the deposit.

The bridge has paid out NEAR-side funds for a Bitcoin transaction that never finalized. This is a **cross-chain verification bypass enabling double-spend**, matching the High impact tier.

Even if the bridge contract checks the `confirmations` field in the signed payload, the MPC network's willingness to sign with `confirmations: 0` means the security guarantee is entirely delegated to every downstream consumer — a single bridge contract that omits this check loses funds.

### Likelihood Explanation

- Any unprivileged NEAR account can call `verify_foreign_transaction` with a 1 yoctoNEAR deposit.
- No special role, key, or threshold collusion is required.
- Bitcoin mempool transactions are immediately visible via `getrawtransaction` on most nodes.
- The attack is profitable whenever the bridge's NEAR-side payout exceeds the cost of the Bitcoin transaction fee.
- The localnet example itself uses `confirmations: 1`, showing the field is expected to be set by callers with no protocol-level floor. [7](#0-6) 

### Recommendation

Enforce a protocol-level minimum confirmation threshold for Bitcoin requests. The minimum should be set by governance (e.g., stored in the on-chain `ForeignChainRpcWhitelist` per-chain config) and validated in `verify_foreign_transaction` before the request is enqueued:

```rust
// In verify_foreign_transaction, after chain support check:
if let ForeignChainRpcRequest::Bitcoin(ref btc_req) = request.request {
    let min_confirmations = self.get_min_bitcoin_confirmations(); // from on-chain config
    if btc_req.confirmations.0 < min_confirmations {
        env::panic_str("Bitcoin confirmations below protocol minimum");
    }
}
```

Alternatively, strip the `confirmations` field from the user request entirely and have the MPC nodes apply a hardcoded or governance-controlled minimum, similar to how EVM finality levels (`Finalized`, `Safe`) are an enum with no user-controlled numeric value.

The MPC network proceeds to sign the payload, producing a valid `VerifyForeignTransactionResponse` for an unconfirmed transaction. [8](#0-7) [9](#0-8) [10](#0-9)
