Audit Report

## Title
Stale `rsETHPrice` Enables Depositors to Dilute and Steal Accrued Yield from Existing rsETH Holders - (File: `contracts/LRTOracle.sol`)

## Summary
`LRTOracle` stores the rsETH exchange rate in `rsETHPrice`, which is only updated when `updateRSETHPrice()` is explicitly called. Because `LRTDepositPool.getRsETHAmountToMint()` reads this stored value directly without triggering a refresh, any depositor can mint rsETH at a stale (undervalued) price when rewards have accrued, then exit after a price update at the corrected higher price, capturing a proportional share of yield that belonged to existing holders.

## Finding Description
`LRTOracle.updateRSETHPrice()` is declared `public` with no access restriction:

```solidity
// LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

The stored `rsETHPrice` is therefore only as fresh as the last explicit call. `LRTDepositPool.getRsETHAmountToMint()` reads this value directly:

```solidity
// LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`_beforeDeposit()` calls `getRsETHAmountToMint()` without first refreshing the price:

```solidity
// LRTDepositPool.sol L665
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
```

When staking rewards cause the actual TVL to exceed `rsethSupply × rsETHPrice`, the stored price is lower than the true per-share value. A depositor at this moment receives more rsETH than their ETH is worth at the true price.

When `_updateRsETHPrice()` subsequently runs, it computes `previousTVL` using the post-deposit supply multiplied by the still-stale price:

```solidity
// LRTOracle.sol L234
uint256 previousTVL = rsethSupply.mulWad(rsETHPrice);
```

Because the attacker's minted rsETH is already included in `rsethSupply`, the detected `rewardAmount` equals only the original accrued reward (not the attacker's deposit), and the new price is computed across the enlarged supply — diluting the yield across all holders including the attacker's newly minted shares.

The `pricePercentageLimit` guard at lines 252–266 of `LRTOracle.sol` is not a sufficient mitigation: if the accrued reward is within the configured limit, the attacker calls `updateRSETHPrice()` themselves; if it exceeds the limit, the attacker simply deposits at the stale price and waits for the protocol's keeper or manager to trigger the update.

## Impact Explanation
**Theft of unclaimed yield — High severity.** Existing rsETH holders accumulate yield as protocol TVL grows above `rsethSupply × rsETHPrice`. When an attacker deposits at the stale price, the subsequent price update is shared across a larger supply, so each pre-existing holder receives less yield than they were entitled to. The attacker captures the difference upon withdrawal. The stolen amount scales linearly with the attacker's deposit size relative to the existing supply, and with the magnitude of accrued rewards.

## Likelihood Explanation
**Medium.** The attack is fully permissionless and requires no privileged access. The attacker only needs to monitor publicly readable on-chain state for reward accrual, hold sufficient ETH capital, and tolerate the `LRTWithdrawalManager` queue delay (approximately 7 days for EigenLayer-backed withdrawals). At 5% APY, the opportunity cost of locking 1,000 ETH for 7 days is approximately 0.96 ETH, while stealing half of 10 ETH in accrued rewards yields a net profit of approximately 4 ETH. The attack is repeatable every reward cycle and becomes more profitable the longer `rsETHPrice` goes without an update.

## Recommendation
Call `_updateRsETHPrice()` (or enforce a freshness check on `rsETHPrice`) inside `_beforeDeposit()` before invoking `getRsETHAmountToMint()`. This ensures every deposit uses the current exchange rate and eliminates the stale-price window. Alternatively, make `_beforeDeposit()` non-`view` and invoke the internal price update there. A secondary hardening measure is to restrict `updateRSETHPrice()` to a trusted keeper role so that the price cannot be updated on-demand by an attacker immediately after a large deposit.

## Proof of Concept
**Setup:**
- rsETH total supply: 1,000 rsETH
- Stored `rsETHPrice`: 1.000 ETH/rsETH (stale)
- Actual protocol TVL: 1,010 ETH (10 ETH in accrued rewards via `receiveFromRewardReceiver()`)

**Steps:**

1. Attacker calls `LRTDepositPool.depositETH{value: 1000 ether}(minRSETH, "")`.
   - `getRsETHAmountToMint` reads stale `rsETHPrice = 1.000`.
   - Mints **1,000 rsETH** to attacker. New supply: 2,000 rsETH.

2. Attacker calls `LRTOracle.updateRSETHPrice()` (assuming price increase is within `pricePercentageLimit`).
   - `_getTotalEthInProtocol()` returns 2,010 ETH.
   - `previousTVL = 2000 × 1.000 = 2000 ETH`.
   - `rewardAmount = 10 ETH`; fee deducted.
   - `newRsETHPrice = 2010 / 2000 = 1.005 ETH/rsETH`.

3. Attacker initiates withdrawal of 1,000 rsETH via `LRTWithdrawalManager`.

4. After queue delay, attacker claims **1,005 ETH**. Profit: **+5 ETH**.

**Result:** Existing holders' 1,000 rsETH is worth 1,005 ETH instead of the 1,010 ETH they were owed. **5 ETH of accrued yield stolen** — half of all pending rewards captured in a single sequence.
