### Title
Sub-transaction Plutus Script ExUnits Excluded from Minimum Fee Calculation in Dijkstra Era - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

---

### Summary

In the Dijkstra era, the minimum fee check for a top-level transaction only accounts for the top-level transaction's Plutus script execution units (`ExUnits`). Sub-transactions embedded in the top-level transaction body can carry their own Plutus scripts and redeemers, but their `ExUnits` are never included in the minimum fee calculation. An unprivileged transaction sender can therefore submit a batch with arbitrarily expensive sub-transaction Plutus scripts while paying only the size-proportional fee for those scripts, bypassing the `txscriptfee` component of the protocol-mandated minimum fee.

---

### Finding Description

The Dijkstra era introduces nested ("sub") transactions. A top-level `DijkstraTxBodyRaw` carries a `dtbrSubTransactions` field containing an ordered map of `Tx SubTx era` values. [1](#0-0) 

Sub-transactions have their own `TxWits` (witness set), which the CDDL spec and the `SUBUTXOW` rule confirm can include redeemers and Plutus scripts. [2](#0-1) 

The `SUBUTXOW` transition rule (`dijkstraSubUtxowTransition`) validates sub-transaction scripts, checks redeemer completeness via `Alonzo.hasExactSetOfRedeemers`, and verifies the script integrity hash — confirming that sub-transaction Plutus scripts are genuinely executed. [3](#0-2) 

The Dijkstra UTXO transition rule enforces the minimum fee for the top-level transaction at line 373:

```haskell
{- minfee pp txTop utxo₀ ≤ txfee txb -}
runTest $ Shelley.validateFeeTooSmallUTxO pp tx originalUtxo
``` [4](#0-3) 

`validateFeeTooSmallUTxO` calls `getMinFeeTxUtxo pp tx utxo`, which for Dijkstra resolves to `getConwayMinFeeTxUtxo` → `getConwayMinFeeTx`:

```haskell
getConwayMinFeeTx pp tx refScriptsSize =
  alonzoMinFeeTx pp tx <+> refScriptsFee
``` [5](#0-4) 

`alonzoMinFeeTx` computes the script-execution fee component as:

```haskell
alonzoMinFeeTx pp tx =
  (tx ^. sizeTxF <×> ...) <+> (pp ^. ppTxFeeFixedL) <+> txscriptfee (pp ^. ppPricesL) allExunits
  where
    allExunits = totExUnits tx
``` [6](#0-5) 

`totExUnits` is defined as:

```haskell
totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL
```

This folds only over the **top-level transaction's** witness set redeemers. Sub-transaction redeemers live in each sub-transaction's own `TxWits` and are never visited here. [7](#0-6) 

Sub-transactions are processed by `dijkstraSubUtxoTransition`, which explicitly calls `updateUTxOStateNoFees` — no fee is collected and no fee check is performed: [8](#0-7) 

The `updateUTxOStateNoFees` function is documented as intentionally omitting fee collection for sub-transactions: [9](#0-8) 

The per-transaction ExUnits limit check (`validateExUnitsTooBigUTxO`) also only inspects the top-level transaction's redeemers, so sub-transaction ExUnits are not bounded by `maxTxExUnits` either: [10](#0-9) 

The block-level ExUnits check (`validateExUnits` in BBODY) similarly uses `foldMap totExUnits txs`, which only sums top-level redeemers across all transactions in the block: [11](#0-10) 

---

### Impact Explanation

The `txscriptfee` component of the minimum fee formula is `ceiling(prMem * mem + prSteps * steps)` applied to the total ExUnits. By placing all expensive Plutus scripts in sub-transactions, an attacker pays only the byte-size fee for those scripts (because sub-transactions are serialized inside the top-level body and thus counted in `sizeTxF`), but pays **zero** `txscriptfee` for their execution cost. The attacker can therefore execute arbitrarily expensive Plutus scripts at a fraction of the protocol-mandated cost, modifying fees outside design parameters.

This matches the allowed impact: **Medium — attacker-controlled transactions modify fees outside design parameters.**

---

### Likelihood Explanation

Any unprivileged Dijkstra-era transaction sender can construct a top-level transaction with sub-transactions containing Plutus scripts. No special role, key, or governance action is required. The Dijkstra era is the current development-stage successor to Conway, so this is reachable on any network running Dijkstra-era protocol rules. Likelihood is **medium** — the era is not yet on mainnet, but the code path is fully reachable by any transaction author once deployed.

---

### Recommendation

The minimum fee calculation for a Dijkstra-era top-level transaction must aggregate `ExUnits` across all sub-transactions in addition to the top-level transaction's own redeemers. Concretely, `getConwayMinFeeTx` (or a Dijkstra-specific override of `getMinFeeTx`) should sum sub-transaction redeemer ExUnits:

```haskell
-- Pseudocode for a Dijkstra-aware totExUnits
dijkstraTotExUnits tx =
  totExUnits tx  -- top-level redeemers
  <> foldMap totExUnits (OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL)
```

This aggregated value should be used in place of `totExUnits tx` when computing `alonzoMinFeeTx` for a Dijkstra top-level transaction. The per-transaction and per-block ExUnits limit checks should be updated analogously.

---

### Proof of Concept

1. Construct a Dijkstra-era top-level transaction `txTop` with an empty top-level witness set (no redeemers, so `totExUnits txTop = ExUnits 0 0`).
2. Embed one or more sub-transactions `txSub₁ … txSubₙ`, each spending a Plutus-script-locked UTxO with a redeemer declaring large ExUnits (e.g., `ExUnits maxMem maxSteps`).
3. Set `txTop.fee` to exactly `alonzoMinFeeTx pp txTop` — which equals `a * size(txTop) + b + txscriptfee(prices, ExUnits 0 0)` = `a * size(txTop) + b`.
4. Submit `txTop`. The `validateFeeTooSmallUTxO` check passes because `minFee = a * size + b` and the declared fee equals that amount.
5. The `SUBUTXOW` rule executes the sub-transaction Plutus scripts (consuming `maxMem * maxSteps` execution units) without any fee being charged for those units.

The attacker has executed expensive Plutus scripts while paying only the byte-size fee, bypassing the `txscriptfee` component entirely. [4](#0-3) [6](#0-5) [12](#0-11)

dijkstraSubUtxowTransition = do
  TRC (env@(SubUtxoEnv _ pp certState originalUtxo _), utxoState, stAnnTx) <-
    judgmentContext
  let tx = stAnnTx ^. txStAnnTxG
      txBody = tx ^. bodyTxL
      witsKeyHashes = keyHashWitnessesTxWits (tx ^. witsTxL)
      scriptsProvided = scriptsProvidedStAnnTx stAnnTx

  {- ∀[ (vk , σ) ∈ vKeySigs ] isSigned vk (txidBytes txId) σ -}
  runTestOnSignal $ Shelley.validateVerifiedWits tx

  let scriptsNeeded = scriptsNeededStAnnTx stAnnTx
      scriptHashesNeeded = getScriptsHashesNeeded scriptsNeeded

  {- ∀[ s ∈ p1ScriptsNeeded ] validP1Script vKeyHashesProvided txVldt s -}
  runTest $ Babbage.validateFailedBabbageScripts tx scriptsProvided scriptHashesNeeded

  {- vKeyHashesNeeded ⊆ vKeyHashesProvided -}
  runTest $ Shelley.validateNeededWitnesses witsKeyHashes certState originalUtxo txBody

  {- dataHashesNeeded ⊆ mapˢ hash dataProvided -}
  runTest $ Alonzo.missingRequiredDatums scriptsProvided originalUtxo tx

  {- txADhash ≡ map hash txAuxData -}
  runTestOnSignal $ Shelley.validateMetadata pp tx

  let scriptIntegrity = mkScriptIntegrity pp tx (plutusLanguagesUsedStAnnTx stAnnTx)
  runTest $ Alonzo.checkScriptIntegrityHash tx pp scriptIntegrity

  runTest $ Alonzo.hasExactSetOfRedeemers tx scriptsProvided scriptsNeeded

  runTest $
    Babbage.validateScriptsWellFormedTxOuts
      pp
      (tx ^. witsTxL . scriptTxWitsL)
      (tx ^. bodyTxL . outputsTxBodyL)

  runTest $ validateGuardDatums scriptsProvided txBody

  trans @(EraRule "SUBUTXO" era) $ TRC (env, utxoState, stAnnTx)
