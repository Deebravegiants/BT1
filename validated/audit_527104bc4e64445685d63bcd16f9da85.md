Looking at the `InitTransfer` account struct and `process` function in full:

### Title
Missing `from` Account Ownership Constraint in `InitTransfer` Allows SPL Token Delegate to Bridge Victim's Tokens — (`solana/programs/bridge_token_factory/src/instructions/user/init_transfer.rs`)

---

### Summary

The `InitTransfer` account struct does not enforce that the `from` token account is owned by `user`. Because SPL Token's `transfer_checked` accepts an approved delegate as a valid authority, any attacker who holds a delegate approval on a victim's token account can call `init_transfer`, lock the victim's tokens in the bridge vault, and receive the equivalent amount on NEAR — with themselves recorded as the sender.

---

### Finding Description

The `from` account is declared with only two constraints:

```rust
#[account(
    mut,
    token::mint = mint,
    token::token_program = token_program,
)]
pub from: Box<InterfaceAccount<'info, TokenAccount>>,
``` [1](#0-0) 

There is no `token::authority = user` constraint. The only checks are that `from` uses the correct mint and token program. No check is made that `from.owner == user`.

In the native-token path (`vault` is `Some`), the CPI is:

```rust
transfer_checked(
    CpiContext::new(
        self.token_program.to_account_info(),
        TransferChecked {
            from: self.from.to_account_info(),
            to: vault.to_account_info(),
            authority: self.user.to_account_info(),   // signer = attacker
            mint: self.mint.to_account_info(),
        },
    ),
    ...
)?;
``` [2](#0-1) 

SPL Token's `transfer_checked` succeeds when `authority` is either the **owner** of `from` *or* an **approved delegate** with sufficient delegated amount. The program passes `self.user` as `authority` without verifying that `user` is the owner of `from`.

After the transfer, the Wormhole message is posted with `self.user.key()` as the `sender` field:

```rust
self.common.post_message(payload.serialize_for_near((
    self.common.sequence.sequence,
    self.user.key(),   // attacker's pubkey
    self.mint.key(),
))?)?;
``` [3](#0-2) 

The serialized message encodes `params.1` (i.e., `self.user.key()`) as the `sender` field in the cross-chain payload: [4](#0-3) 

The same missing constraint applies to the burn path (bridged tokens), where `authority: self.user` is passed to `burn` without verifying ownership of `from`. [5](#0-4) 

---

### Impact Explanation

A victim's tokens are irreversibly locked in the bridge vault on Solana. The Wormhole VAA credits the attacker as the sender, so the attacker — not the victim — can claim the bridged tokens on NEAR. The victim loses their tokens with no recourse through the bridge protocol.

---

### Likelihood Explanation

**Prerequisite**: The victim must have previously called SPL Token `approve` granting the attacker (or a contract the attacker controls) delegate authority over their token account. This is a real-world scenario: users routinely grant delegate approvals to DeFi protocols, aggregators, or other contracts. If any such approved party is malicious or compromised, they can exploit this path. The prerequisite limits but does not eliminate the attack surface — likelihood is **medium-low**.

---

### Recommendation

Add `token::authority = user` to the `from` account constraint:

```rust
#[account(
    mut,
    token::mint = mint,
    token::authority = user,          // <-- add this
    token::token_program = token_program,
)]
pub from: Box<InterfaceAccount<'info, TokenAccount>>,
```

This ensures Anchor rejects any `from` account not directly owned by the signing `user`, eliminating the delegate-based attack path for both the `transfer_checked` and `burn` branches.

---

### Proof of Concept

1. Victim creates a token account holding 1,000 tokens for a registered native mint (vault exists).
2. Victim calls SPL Token `approve`, granting the attacker's keypair delegate authority for 1,000 tokens.
3. Attacker constructs an `init_transfer` transaction:
   - `from` = victim's token account
   - `user` = attacker's keypair (signer)
   - `vault` = the bridge vault PDA for the mint
4. Anchor constraint validation passes: `from.mint == mint` ✓, `from.token_program == token_program` ✓. No ownership check.
5. `transfer_checked` CPI succeeds: SPL Token accepts the attacker as a valid delegate authority.
6. 1,000 tokens move from victim's account into the bridge vault.
7. Wormhole message is posted with attacker's pubkey as `sender` and attacker's chosen NEAR address as `recipient`.
8. On NEAR, the prover verifies the VAA and credits the attacker with 1,000 bridged tokens.
9. Victim's tokens are permanently locked; attacker holds the equivalent on NEAR.
