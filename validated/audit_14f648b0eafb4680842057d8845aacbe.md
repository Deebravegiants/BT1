### Title
Unverified CKD Response for `AppPublicKey` Variant Allows Single Byzantine Participant to Forge Confidential Key Derivation Output - (File: `crates/contract/src/lib.rs`)

### Summary

The `respond_ckd` function in `MpcContract` performs cryptographic output verification only for the `AppPublicKeyPV` variant of `CKDAppPublicKey`. When a user submits a `request_app_private_key` using the `AppPublicKey` (legacy, privately-verifiable) variant, the contract accepts any `CKDResponse` from a single attested participant without any on-chain proof that the response was honestly computed using the MPC network's shared key. This is structurally identical to the reported RAACToken bug: the proper verification interface (`ckd_output_check`) is bypassed for one code path, and the unverified data is delivered directly to the waiting user.

### Finding Description

In `respond_ckd` (`crates/contract/src/lib.rs`, lines 653–689), after confirming the caller is an attested participant, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no verification at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
pending_requests::resolve_yields_for(&mut self.pending_ckd_requests, &request,
    serde_json::to_vec(&response).unwrap())
```

For `AppPublicKeyPV`, `ckd_output_check` enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which proves the response was computed using the network's BLS12-381 master key. For `AppPublicKey`, the empty arm `{}` means **no such proof is required**. The response is serialized and immediately delivered to every yield waiting on that request key via `resolve_yields_for`.

The attacker-controlled entry path:
1. A user calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(pk)` and attaches 1 yoctoNEAR. The request is queued in `pending_ckd_requests`.
2. A single malicious attested participant (strictly below the signing threshold) calls `respond_ckd` with the correct `CKDRequest` key and an arbitrary fabricated `CKDResponse { big_y: ..., big_c: ... }`.
3. The contract passes all guards (signer check, protocol-running check, TEE check, participant check) and reaches the `match` arm for `AppPublicKey`, which does nothing.
4. `resolve_yields_for` drains the entire fan-out queue, delivering the forged response to every caller who submitted that request.

The `AppPublicKey` variant is documented as "privately verifiable, legacy" — the intent is that the user verifies the response off-chain using their own secret. But the contract never enforces that the response was honestly produced by the threshold network; it only enforces that *some* attested participant submitted *something*. [1](#0-0) [2](#0-1) 

### Impact Explanation

A single Byzantine attested participant (one node, below threshold) can forge a `CKDResponse` for any pending `AppPublicKey`-type CKD request and have the contract deliver it to the user as if it were the honest MPC output. The user's application receives a fake confidential key whose value the attacker chose. If the user encrypts data to this key or uses it as a secret, the attacker — who knows the fake key — can decrypt or impersonate. This is unauthorized confidential key derivation output without the required threshold participant authorization, matching the Critical impact tier: "Unauthorized … confidential key derivation output without the required participant authorization." [3](#0-2) 

### Likelihood Explanation

Any single attested participant can trigger this. Becoming an attested participant requires submitting a valid TEE attestation via `submit_participant_info`, which is a normal operational step for any MPC node. Once attested, the node can call `respond_ckd` at any time while the protocol is running. The `AppPublicKey` variant is the legacy path still accepted by the contract (the ABI snapshot confirms both variants remain in the public interface). A malicious node operator — or a node whose TEE is compromised — can exploit this without any collusion. [4](#0-3) [5](#0-4) 

### Recommendation

Apply `ckd_output_check` (or an equivalent proof of honest computation) to the `AppPublicKey` variant as well, or reject `AppPublicKey`-type requests in `respond_ckd` entirely and require callers to migrate to `AppPublicKeyPV`. If the `AppPublicKey` variant must remain for backward compatibility, the contract should at minimum require that the response passes a weaker consistency check (e.g., that `big_y` lies on the curve and is non-trivial), and document clearly that the on-chain path provides no integrity guarantee — though the correct fix is to enforce the same pairing check for both variants. [6](#0-5) [7](#0-6) 

### Proof of Concept

1. Deploy the contract in Running state with a BLS12-381 domain and at least one attested participant (the attacker's node).
2. A victim calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(victim_pk)`.
3. The attacker's node (attested, but acting alone — below threshold) calls:
   ```rust
   respond_ckd(
       ckd_request,   // matches the victim's pending request key
       CKDResponse {
           big_y: Bls12381G1PublicKey([0xAA; 48]),  // attacker-chosen fake key
           big_c: Bls12381G1PublicKey([0xBB; 48]),
       }
   )
   ```
4. The contract executes the empty `AppPublicKey` arm, calls `resolve_yields_for`, and the victim's yield resumes with the forged response.
5. The victim's contract or application receives `big_y = [0xAA; 48]` as its "confidential key" — a value the attacker fully controls — with no on-chain error or rejection.

The existing unit test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` (lib.rs line 3404) already demonstrates this path: it passes `CKDResponse { big_y: [1u8; 48], big_c: [2u8; 48] }` — arbitrary bytes — and the contract accepts them without complaint, confirming no verification occurs for the `AppPublicKey` variant. [8](#0-7)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
