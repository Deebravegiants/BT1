Audit Report

## Title
Stale Chainlink Price Accepted Without Staleness Validation Enables Deposit-Time Rate Manipulation — (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards the `updatedAt` timestamp and `answeredInRound` values, accepting arbitrarily stale prices as valid. This stale price propagates directly into the rsETH minting formula in `LRTDepositPool`, allowing an unprivileged depositor to receive more rsETH than their deposited assets are worth when a Chainlink LST/ETH feed lags behind the true market price, diluting all existing rsETH holders.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` at L49–55 fetches price data from Chainlink but only reads the `price` field, discarding `updatedAt` and `answeredInRound`:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52
(, int256 price,,,) = priceFeed.latestRoundData();
```

The same codebase already implements the correct staleness pattern in `ChainlinkOracleForRSETHPoolCollateral.getRate()` (L26–37), which validates `answeredInRound < roundID` and `timestamp == 0` before using the price. The absence of these checks in `ChainlinkPriceOracle` is a code-level deficiency, not an oracle data quality issue.

The stale price propagates through the following confirmed call chain:

1. `LRTDepositPool.depositAsset()` (L99–118) → `_beforeDeposit()` (L648–670) → `getRsETHAmountToMint()` (L506–521)
2. `getRsETHAmountToMint()` computes: `rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()`
3. `lrtOracle.getAssetPrice(asset)` (L156–158) delegates to `IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset)`, which resolves to `ChainlinkPriceOracle`
4. `lrtOracle.rsETHPrice()` is a stored state variable last set by a prior `updateRSETHPrice()` call — it is not recomputed at deposit time

The critical asymmetry: the numerator (`getAssetPrice(asset)`) is fetched live from the stale Chainlink feed at deposit time, while the denominator (`rsETHPrice`) is a stored value reflecting a prior true price. If the feed is stale at an inflated price and `rsETHPrice` has not yet been updated to reflect that inflation, the attacker receives excess rsETH. The `pricePercentageLimit` guard in `_updateRsETHPrice()` only triggers when `updateRSETHPrice()` is called, not during deposits, and only for price moves exceeding the configured threshold — it provides no protection during the deposit itself.

Additionally, `updateRSETHPrice()` (L87–89) is a public, permissionless function that calls `_getTotalEthInProtocol()` (L331–349), which also calls `getAssetPrice()` for each supported asset. A stale price here inflates the computed TVL, potentially minting excess protocol fee rsETH to the treasury.

## Impact Explanation

**Critical — Direct theft of user funds.**

When a Chainlink LST/ETH feed (e.g., stETH/ETH, ETHx/ETH) goes stale at a price above the current market, an attacker deposits LST tokens and receives rsETH computed at the inflated stale price. The excess rsETH represents a claim on ETH value never deposited. Upon redemption via `LRTWithdrawalManager`, the attacker extracts more ETH-equivalent value than contributed, with the shortfall borne by all existing rsETH holders whose share of the underlying TVL is diluted. This matches the allowed Critical impact: direct theft of user funds at rest.

## Likelihood Explanation

**Medium.** Chainlink LST/ETH feeds operate on heartbeat intervals (typically 1 hour) and deviation thresholds. During periods of high network congestion, oracle node issues, or rapid market downturns, feeds can lag behind true market prices for minutes to hours. The attacker's entry path — `depositAsset()` on `LRTDepositPool` — is fully permissionless and requires no privileged access. The attacker only needs to monitor the on-chain `updatedAt` timestamp of the relevant Chainlink feed and act when it lags behind the current market price. The attack is repeatable across any supported LST asset with a stale feed.

## Recommendation

Add staleness validation to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    // Optionally enforce a maximum age:
    // if (block.timestamp - updatedAt > MAX_STALENESS_SECONDS) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

## Proof of Concept

1. Deploy a fork of mainnet. Identify a supported LST asset (e.g., stETH) and its Chainlink feed address registered in `ChainlinkPriceOracle.assetPriceFeed`.
2. Warp time or mock the Chainlink aggregator so that `latestRoundData()` returns a stale `updatedAt` (e.g., 2 hours ago) with a price 2% above the current true market price. Confirm `ChainlinkPriceOracle.getAssetPrice(stETH)` returns the inflated value without reverting.
3. Record the current `LRTOracle.rsETHPrice()` (stored, reflecting true prior price).
4. Call `LRTDepositPool.depositAsset(stETH, amount, 0, "")` as an unprivileged EOA. Verify that `getRsETHAmountToMint` uses the inflated asset price, minting excess rsETH proportional to the price discrepancy.
5. Allow the oracle to update to the true price. Call `LRTOracle.updateRSETHPrice()` to refresh `rsETHPrice` to the correct value.
6. Initiate and complete withdrawal via `LRTWithdrawalManager`. Confirm the attacker redeems more ETH-equivalent value than deposited, with the difference extracted from existing rsETH holders' share of TVL.

**Foundry invariant test:** Assert that for any deposit, `rsethMinted * rsETHPrice_before <= depositAmount * trueAssetPrice` holds. With a mocked stale feed returning an inflated price, this invariant breaks, confirming the vulnerability.
