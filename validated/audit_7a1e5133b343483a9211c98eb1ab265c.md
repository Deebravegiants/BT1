### Title
Unchecked u128 Overflow in `denormalize_amount` / `normalize_amount` Causes Irreversible Fund Lock for Tokens with Large Decimal Differences - (File: near/omni-bridge/src/lib.rs)

### Summary
`denormalize_amount` and `normalize_amount` perform `amount * 10_u128.pow(diff_decimals)` and `amount / 10_u128.pow(diff_decimals)` respectively, with no overflow guard. When `origin_decimals − decimals ≥ 39`, the intermediate `10_u128.pow(diff_decimals)` exceeds `u128::MAX` (≈ 3.4 × 10³⁸). In a NEAR WASM release build with overflow-checks enabled this panics and reverts the transaction; without them it silently wraps, producing a wildly incorrect amount. Either outcome permanently blocks finalization of the affected transfer, locking user funds.

### Finding Description
`denormalize_amount` is defined as:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount * (10_u128.pow(diff_decimals))   // ← no checked_mul / checked_pow
}
``` [1](#0-0) 

`normalize_amount` has the identical structural flaw:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))   // ← 10^diff_decimals itself can overflow
}
``` [2](#0-1) 

`u128::MAX ≈ 3.4028 × 10³⁸`, so `10_u128.pow(39)` already overflows. A token whose `origin_decimals − decimals ≥ 39` triggers this on every call.

The `Decimals` struct stores both fields as bare `u8` with no range validation in `add_token`: [3](#0-2) 

The `bind_token_callback` path writes `deploy_token.decimals` and `deploy_token.origin_decimals` directly from the prover result into storage: [4](#0-3) 

Those values originate from the on-chain `DeployToken` event. On Solana, `decimals = min(MAX_ALLOWED_DECIMALS=9, origin_decimals)` and `origin_decimals` is the raw SPL mint decimals: [5](#0-4) 

A Solana SPL token with `origin_decimals = 48` (unusual but protocol-unrestricted) yields `diff_decimals = 39`, triggering the overflow.

The overflow is reached through two public, permissionless call paths:

1. **`fin_transfer_callback`** — called after proof verification, calls `denormalize_amount` on the prover-supplied amount: [6](#0-5) 

2. **`sign_transfer`** — calls `normalize_amount` to compute the amount to embed in the MPC-signed payload: [7](#0-6) 

### Impact Explanation
- **With overflow-checks = true** (recommended for NEAR): `10_u128.pow(39)` panics → `fin_transfer_callback` reverts → the transfer can never be finalized → user funds are permanently locked on the source chain. This satisfies **"Irreversible fund lock, frozen redemption path, or permanently unclaimable user value."**
- **With overflow-checks = false**: the multiplication wraps to a small incorrect value. `denormalize_amount` returns a near-zero amount; `normalize_amount` returns a near-zero or near-MAX amount. Either breaks the backing guarantee and sends value to the wrong party.

### Likelihood Explanation
Low. No mainstream token uses 48+ decimals. However, the protocol imposes no upper bound on `origin_decimals` at registration time, so a legitimately deployed SPL token with extreme decimals would silently arm this path for every user who bridges it. No privileged role is required to trigger the overflow once such a token is registered.

### Recommendation
Replace the bare arithmetic with checked variants and revert with a descriptive error on overflow:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff: u32 = decimals.origin_decimals
        .checked_sub(decimals.decimals)
        .expect("origin_decimals < decimals") as u32;
    let multiplier = 10_u128.checked_pow(diff)
        .expect("decimal multiplier overflows u128");
    amount.checked_mul(multiplier)
        .expect("denormalized amount overflows u128")
}
```

Additionally, enforce `origin_decimals >= decimals` and `origin_decimals - decimals <= 38` inside `add_token` at registration time.

### Proof of Concept
1. Deploy an SPL token on Solana with `decimals = 48`.
2. The Solana bridge factory emits `DeployTokenResponse { decimals: 9, origin_decimals: 48 }`.
3. A relayer calls `bind_token` on NEAR with the Wormhole VAA; `bind_token_callback` stores `Decimals { decimals: 9, origin_decimals: 48 }`.
4. A user transfers 1 unit of that token from Solana → NEAR; the source chain emits an `InitTransfer` event with the normalized amount.
5. A relayer calls `fin_transfer` on NEAR; `fin_transfer_callback` calls `denormalize_amount(amount, Decimals { decimals: 9, origin_decimals: 48 })`.
6. `diff_decimals = 39`; `10_u128.pow(39)` overflows → panic → transaction reverts → transfer permanently unfinalizeable → user funds locked.
