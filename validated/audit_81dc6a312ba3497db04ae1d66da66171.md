Audit Report

## Title
`checkDailyMintLimit` Reverts All Mints When `maxMintAmountPerDay` Is Unset (Zero) - (File: contracts/RSETH.sol)

## Summary
`RSETH.mint()` unconditionally applies the `checkDailyMintLimit` modifier, which reverts whenever `currentPeriodMintedAmount + amount > maxMintAmountPerDay`. Because `maxMintAmountPerDay` defaults to `0` and is never assigned in `initialize()` or `reinitialize()`, every call to `mint()` reverts with `DailyMintLimitExceeded` until a separate `setMaxMintAmountPerDay()` transaction is confirmed. Any user deposit attempted during this window fails entirely.

## Finding Description
`RSETH.reinitialize()` sets `periodStartTime` and `custodyAddress` but leaves `maxMintAmountPerDay` at its Solidity default of `0` (lines 109–117). The `checkDailyMintLimit` modifier at line 50 evaluates `currentPeriodMintedAmount + amount > maxMintAmountPerDay`; with `maxMintAmountPerDay == 0` this reduces to `amount > 0`, which is always true for any real deposit, causing an unconditional `revert DailyMintLimitExceeded`. The modifier has no zero-value bypass, unlike `remainingDailyMintLimit()` which explicitly returns `0` when `maxMintAmountPerDay == 0` (line 266). `mint()` is the terminal step of every user deposit: `depositETH()` → `_mintRsETH()` → `RSETH.mint()` and `depositAsset()` → `_mintRsETH()` → `RSETH.mint()`. The gap between `reinitialize()` being confirmed and `setMaxMintAmountPerDay()` being confirmed is the vulnerable window. A chain reorganisation that reorders a user deposit transaction ahead of `setMaxMintAmountPerDay()` is one concrete trigger, but the window also exists in normal multi-transaction deployment sequences.

## Impact Explanation
All user deposits (ETH and LST) revert during the window. Because `depositAsset()` transfers tokens before calling `_mintRsETH()`, the revert rolls back the token transfer atomically, so no funds are permanently lost. The impact is **temporary freezing of user deposits**: no rsETH can be minted and no deposit can succeed until `setMaxMintAmountPerDay()` is confirmed on-chain. This matches **Medium — Temporary freezing of funds**.

## Likelihood Explanation
The window opens the moment `reinitialize()` is confirmed and closes only when `setMaxMintAmountPerDay()` is confirmed. On Ethereum mainnet the window spans at least one block (the block gap between the two manager transactions). On L2 chains (Arbitrum, Optimism) where the protocol also operates, chain reorganisations are more frequent and can reorder transactions within the same block. No attacker action is required; any user deposit submitted during the window triggers the failure. Likelihood is **Low-Medium**.

## Recommendation
Set `maxMintAmountPerDay` atomically inside `reinitialize()` to eliminate the gap:

```solidity
function reinitialize(
    uint256 _periodStartTime,
    address _custodyAddress,
    uint256 _maxMintAmountPerDay
) external reinitializer(2) onlyLRTManager {
    if (_periodStartTime > block.timestamp || _periodStartTime <= block.timestamp - 1 days) {
        revert PeriodStartTimeShouldBeWithin24Hours();
    }
    require(_maxMintAmountPerDay > 0, "zero limit");
    periodStartTime = _periodStartTime;
    maxMintAmountPerDay = _maxMintAmountPerDay;
    emit PeriodStartTimeSet(_periodStartTime);
    emit MaxMintAmountPerDayUpdated(_maxMintAmountPerDay);
    _setCustodyAddress(_custodyAddress);
}
```

Alternatively, add a zero-value bypass in `checkDailyMintLimit` (treating `maxMintAmountPerDay == 0` as uncapped), consistent with the existing behaviour of `remainingDailyMintLimit()`.

## Proof of Concept
**Foundry fork test outline:**
1. Deploy and call `reinitialize(periodStart, custodyAddr)` — do NOT call `setMaxMintAmountPerDay`.
2. Have Alice call `depositETH{value: 1 ether}(0, "")`.
3. Observe revert with `DailyMintLimitExceeded(rsethAmountToMint, 0)`.
4. Call `setMaxMintAmountPerDay(1_000 ether)`.
5. Repeat Alice's deposit — observe success.

**Chain-reorg scenario:**
1. Block N: manager sends tx A (`setMaxMintAmountPerDay(1_000 ether)`) and Alice sends tx B (`depositETH`).
2. Reorg reorders tx B before tx A in the same block.
3. Tx B executes with `maxMintAmountPerDay == 0` → `checkDailyMintLimit` reverts → Alice's deposit fails.
4. All concurrent depositors are equally affected until tx A is re-included.
