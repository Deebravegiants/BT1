### Title
Multiple Sub-Transactions Can Claim the Same Deposit Refund, Creating ADA From Nothing ŌĆö (`eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs`)

---

### Summary

In the Dijkstra era, a top-level transaction may contain multiple sub-transactions. Each sub-transaction is processed through `updateUTxOStateNoFees`, which independently computes deposit refunds via `certsTotalRefundsTxBody`. No batch-level guard prevents two sub-transactions from each claiming the refund for the same stake credential deregistration. The developers have explicitly acknowledged this gap with a disabled (`xit`) test titled **"Multiple subtransactions cannot get the same refund"** whose expected predicate failure is marked `error "TODO: predicate failure not yet implemented"`. This means the protection does not yet exist in the production code.

---

### Finding Description

The Dijkstra era introduces nested (sub-)transactions. A top-level transaction body may carry a `subTransactionsTxBodyL` field containing an ordered set of sub-transactions. Each sub-transaction is validated and applied independently through the `SubLedger` / `SubLedgers` rules, which call `updateUTxOStateNoFees` ŌĆö the fee-free variant of the UTxO state updater. [1](#0-0) 

Inside `updateUTxOStateNoFees`, the refund for each sub-transaction is computed as:

```haskell
totalRefunds = certsTotalRefundsTxBody pp certState txBody
``` [2](#0-1) 

The `certState` supplied here reflects the ledger state at the time each sub-transaction is evaluated. If the state is not correctly threaded between sub-transactions ŌĆö or if the refund calculation is performed against the pre-batch state ŌĆö then two sub-transactions that each carry `UnRegDepositTxCert stakingCred keyDeposit` for the **same** credential will each independently compute a non-zero refund, reducing `utxosDeposited` twice while only one deposit was ever paid.

The Dijkstra UTXO rule does implement a batch-level guard for **withdrawals** (`validateBatchWithdrawals`), which aggregates all withdrawals across the top-level transaction and every sub-transaction and checks them against the account balance: [3](#0-2) 

No equivalent batch-level guard exists for **deposit refunds**. The developers are aware of this gap: the test file for Dijkstra certificate rules contains a disabled test that explicitly documents the missing protection: [4](#0-3) 

The test is disabled with `xit` and the expected predicate failure is `error "TODO: predicate failure not yet implemented"` ŌĆö confirming that no rejection rule for this scenario has been implemented yet.

A second disabled test (`xit "Subtransaction consumes correct refund after keyDeposit is changed"`) further shows that the interaction between sub-transaction refund accounting and protocol-parameter changes is also unresolved: [5](#0-4) 

---

### Impact Explanation

**Critical ŌĆö Direct creation of ADA through an invalid ledger state transition.**

Suppose `keyDeposit = 2 ADA`. An attacker:

1. Registers a stake credential, paying 2 ADA into `utxosDeposited`.
2. Submits a top-level transaction containing two sub-transactions, each carrying `UnRegDepositTxCert stakingCred (Coin 2_000_000)`.

If both sub-transactions independently compute and apply a 2 ADA refund, `utxosDeposited` is reduced by 4 ADA while only 2 ADA was ever deposited. The extra 2 ADA appears in the UTxO outputs of the sub-transactions, violating the preservation-of-value invariant. This constitutes direct, attacker-controlled creation of ADA from the deposit pot ŌĆö a Critical ledger impact.

---

### Likelihood Explanation

The Dijkstra era is experimental but is present in the production repository and is being actively developed toward deployment. Any unprivileged user who can submit a valid top-level transaction can craft this attack. No special role, key, or governance threshold is required. The attack requires only:

- One prior stake credential registration (costs `keyDeposit` ADA, fully recovered if the attack succeeds).
- A top-level transaction with two sub-transactions sharing the same `UnRegDepositTxCert`.

The disabled test and the explicit `TODO` comment confirm the developers are aware the check is missing, making this a known, unmitigated gap.

---

### Recommendation

1. **Add a batch-level refund guard** analogous to `validateBatchWithdrawals`: before applying any sub-transaction, aggregate all `UnRegDepositTxCert` (and `UnRegDRepTxCert`) certificates across the entire batch (top-level + all sub-transactions) and verify that no credential appears more than once as a deregistration target.

2. **Enable and complete the disabled tests** in `CertSpec.hs` by implementing the missing predicate failure (e.g., `DuplicateDeregInBatch`) and wiring it into the `SubLedgers` / `SubUtxo` validation pipeline.

3. **Audit `updateUTxOStateNoFees`** to confirm that the `certState` passed to each successive sub-transaction is the post-application state of the previous sub-transaction, not the pre-batch snapshot.

---

### Proof of Concept

The developers' own disabled test is the proof of concept:

```haskell
-- eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs

xit "Multiple subtransactions cannot get the same refund" $ do
  stakingCred <- KeyHashObj <$> freshKeyHash
  _ <- registerStakeCredential stakingCred          -- pays keyDeposit once
  keyDeposit <- getsPParams ppKeyDepositL
  ...
  let
    subTx1 = mkBasicTx mkBasicTxBody
      & bodyTxL . inputsTxBodyL  .~ Set.singleton input1
      & bodyTxL . certsTxBodyL   .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
    subTx2 = mkBasicTx mkBasicTxBody
      & bodyTxL . inputsTxBodyL  .~ Set.singleton input2
      & bodyTxL . certsTxBodyL   .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
    tx = mkBasicTx mkBasicTxBody
      & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
  submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
  -- ^^^ disabled because the rejection rule does not exist yet
``` [4](#0-3) 

Both `subTx1` and `subTx2` carry an `UnRegDepositTxCert` for the same `stakingCred`. Because no batch-level deduplication guard exists for deposit refunds (unlike the `validateBatchWithdrawals` guard that exists for reward withdrawals), each sub-transaction independently claims the full `keyDeposit` refund, extracting twice the deposited amount from `utxosDeposited`. -- | Like 'updateUTxOState', but does not collect fees. This is used for sub-transactions
-- where fees are not applicable.
updateUTxOStateNoFees ::
  ( EraTxBody era
  , EraStake era
  , EraCertState era
  , Monad m
  ) =>
  PParams era ->
  UTxOState era ->
  TxBody l era ->
  CertState era ->
  GovState era ->
  (Coin -> m ()) ->
  (UTxO era -> UTxO era -> m ()) ->
  m (UTxOState era)
updateUTxOStateNoFees pp utxos txBody certState govState depositChangeEvent txUtxODiffEvent = do
  let UTxOState {utxosUtxo, utxosDeposited, utxosFees, utxosDonation} = utxos
      UTxO utxo = utxosUtxo
      !utxoAdd = txouts txBody -- These will be inserted into the UTxO
      {- utxoDel  = txins txb ŌŚü utxo -}
      !(utxoWithout, utxoDel) = extractKeys utxo (txBody ^. inputsTxBodyL)
      {- newUTxO = (txins txb Ōŗ¬ utxo) Ōł¬ outs txb -}
      newUTxO = utxoWithout `Map.union` unUTxO utxoAdd
      deletedUTxO = UTxO utxoDel
      totalRefunds = certsTotalRefundsTxBody pp certState txBody
      totalDeposits = certsTotalDepositsTxBody pp certState txBody
      depositChange = totalDeposits <-> totalRefunds
  depositChangeEvent depositChange
  txUtxODiffEvent deletedUTxO utxoAdd
  pure $!
    UTxOState
      { utxosUtxo = UTxO newUTxO
      , utxosDeposited = utxosDeposited <> depositChange
      , utxosFees = utxosFees
      , utxosGovState = govState
      , utxosInstantStake =
          deleteInstantStake deletedUTxO (addInstantStake utxoAdd (utxos ^. instantStakeL))
      , utxosDonation = utxosDonation
      }

-- | For each account, the total withdrawals across the entire batch should not exceed the original account balance.
-- Unregistered accounts are treated as having 0 balance.
validateBatchWithdrawals ::
  ( EraTx era
  , EraAccounts era
  , DijkstraEraTxBody era
  ) =>
  Accounts era ->
  Tx TopTx era ->
  Test (DijkstraUtxoPredFailure era)
validateBatchWithdrawals accounts tx =
  let allWithdrawals =
        Map.unionsWith (<>) $
          unWithdrawals (tx ^. bodyTxL . withdrawalsTxBodyL)
            : [ unWithdrawals $ subTx ^. bodyTxL . withdrawalsTxBodyL
              | subTx <- OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL
              ]
      badWithdrawals =
        Map.mapMaybeWithKey
          ( \acctAddr withdrawn ->
              let balance = getAccountBalance acctAddr
               in if withdrawn > balance
                    then Just Mismatch {mismatchSupplied = withdrawn, mismatchExpected = balance}
                    else Nothing
          )
          allWithdrawals
   in failureOnNonEmptyMap badWithdrawals WithdrawalsExceedAccountBalance
