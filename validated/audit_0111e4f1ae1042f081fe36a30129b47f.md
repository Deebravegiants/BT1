### Title
Unconstrained Direct Deposits Allow Arbitrary Lovelace Creation — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs`)

### Summary
The Dijkstra era introduces a `directDeposits` field in the transaction body that credits arbitrary amounts to registered accounts. These amounts are applied to account balances in the `ENTITIES` rule but are absent from both the `consumed` and `produced` sides of `validateValueNotConservedUTxO`. Any unprivileged sender can include a `DirectDeposits` map crediting any registered account with any amount of lovelace, and the ledger will accept the transaction while permanently increasing the total lovelace supply.

### Finding Description
`dijkstraEntitiesTransition` applies `applyDirectDeposits` to account balances after processing certificates: [5](#0-4) 

`applyDirectDeposits` adds each deposit amount to the matching account balance: [8](#0-7) 

The UTxO conservation check (`validateValueNotConservedUTxO`) is computed from `getConsumedDijkstraValue` and `getProducedDijkstraValue`. Neither function references `directDepositsTxBodyL`: [9](#0-8) 

The `consumed` side includes withdrawals (which correctly drain accounts into the UTxO balance), but the `produced` side has no corresponding term for direct deposits (which should drain the UTxO balance into accounts). The asymmetry means direct deposits are entirely off-books.

### Impact Explanation
**Critical — Direct creation of ADA through an invalid ledger state transition.**

An attacker with one registered account can include `DirectDeposits = {accountC → N lovelace}` in any valid transaction. The conservation check passes because `N` appears nowhere in the equation. After the transaction is applied, `accountC` holds `N` additional lovelace that was not sourced from any existing UTxO output, account balance, deposit pot, treasury, or reserves. Total lovelace in the system increases by `N`. This can be repeated in every block, inflating supply without bound.

### Likelihood Explanation
The attack requires only a registered account and the ability to submit a transaction — both available to any unprivileged participant. No governance action, key compromise, or privileged role is needed. The path is direct and deterministic.

### Recommendation
Direct deposits must be included in the `produced` side of the value conservation equation. Specifically, `getProducedDijkstraValue` (and `dijkstraSubTxProducedValue` for sub-transactions) must add `inject (fold . unDirectDeposits $ txBody ^. directDepositsTxBodyL)` to the produced value, mirroring how withdrawals are added to the consumed value. This ensures that every lovelace credited to an account via a direct deposit is sourced from the transaction's UTxO inputs.

### Proof of Concept
1. Register account `C` (e.g., via a `RegKey` certificate in a prior transaction).
2. Construct a transaction with:
   - Any valid UTxO input/output pair that balances fees (e.g., send 2 ADA to self, pay 0.17 ADA fee).
   - `directDeposits = { C → 1_000_000_000 lovelace }` (1000 ADA).
   - No withdrawals.
3. Submit the transaction. `validateValueNotConservedUTxO` passes because direct deposits are absent from both sides of the equation.
4. After the transaction, account `C` has 1000 ADA more than before. Total lovelace in the system has increased by 1000 ADA.
5. Repeat to inflate supply arbitrarily.

getConsumedDijkstraValue ::
  forall era l.
  ( DijkstraEraTxBody era
  , EraUTxO era
  , Value era ~ MaryValue
  , STxLevel l era ~ STxBothLevels l era
  ) =>
  PParams era ->
  (Credential Staking -> Maybe Coin) ->
  (Credential DRepRole -> Maybe Coin) ->
  UTxO era ->
  TxBody l era ->
  Value era
getConsumedDijkstraValue pp lookupStakingDeposit lookupDRepDeposit utxo txBody =
  withBothTxLevels
    txBody
    ( \topTxBody ->
        txBodyConsumedValue topTxBody <> subTransactionsConsumedValue topTxBody
    )
    txBodyConsumedValue
  where
    txBodyConsumedValue :: forall m. TxBody m era -> Value era
