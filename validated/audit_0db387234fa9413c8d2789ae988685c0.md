Audit Report

## Title
Unprotected `_withdrawFromAave` Call in `_processWithdrawalCompletion` Causes DoS on ETH Withdrawals - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
When Aave integration is enabled, `completeWithdrawal` calls `_withdrawFromAave` inside `_processWithdrawalCompletion` without any try/catch protection. If Aave's pool is paused or WETH liquidity is exhausted, `aaveWETHGateway.withdrawETH` reverts, causing every ETH `completeWithdrawal` call to revert and temporarily freezing user funds. The protocol already applies try/catch to the symmetric Aave deposit call in `unlockQueue`, making this omission a clear inconsistency in the protocol's own code.

## Finding Description
In `_processWithdrawalCompletion` (lines 719–732), when `isAaveIntegrationEnabled` is true and the contract's ETH balance is below `request.expectedAssetAmount`, the code calls `_withdrawFromAave(amountNeeded)` directly:

```solidity
// lines 719–732
if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
    uint256 contractBalance = address(this).balance;
    if (contractBalance < request.expectedAssetAmount) {
        uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
        _withdrawFromAave(amountNeeded);          // no try/catch
        uint256 balanceAfter = address(this).balance;
        if (balanceAfter < request.expectedAssetAmount) {
            revert InsufficientLiquidityForWithdrawal();
        }
    }
}
```

`_withdrawFromAave` (lines 905–921) calls `aaveWETHGateway.withdrawETH` with no error handling:

```solidity
aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
```

Aave v3 enforces a global guardian-controlled pause; any call to `withdrawETH` while paused reverts unconditionally. High WETH utilization can also cause the call to revert due to insufficient pool liquidity.

By contrast, the deposit path in `unlockQueue` (lines 310–317) wraps the equivalent Aave call in try/catch:

```solidity
try this.depositToAaveExternal(assetAmountUnlocked) { }
catch (bytes memory reason) {
    emit AaveDepositFailed(assetAmountUnlocked, reason);
}
```

Because `unlockQueue` deposits idle ETH into Aave, the contract's own ETH balance is structurally kept near zero, making the Aave withdrawal path the critical dependency for every subsequent `completeWithdrawal` call. The missing protection means a single Aave pause event blocks all ETH withdrawal completions for all users simultaneously. [1](#0-0) [2](#0-1) [3](#0-2) 

## Impact Explanation
Users who have already burned rsETH via `initiateWithdrawal` cannot complete their ETH withdrawal while Aave is paused or illiquid. Their ETH is locked in the protocol until Aave's state normalizes. This is a concrete **temporary freezing of funds**, which is an allowed Medium impact in the scope.

## Likelihood Explanation
Aave v3's guardian pause has been exercised on mainnet historically. During market stress, WETH utilization in Aave can spike to near 100%, making `withdrawETH` revert. When `isAaveIntegrationEnabled` is true, the `unlockQueue` flow deposits unlocked ETH into Aave, making the contract structurally dependent on Aave for every ETH `completeWithdrawal`. No privileged actor or attacker action is required — the condition is triggered by Aave's own operational state. Likelihood is **Medium**.

## Recommendation
Wrap the `_withdrawFromAave` call in `_processWithdrawalCompletion` in a try/catch, mirroring the pattern already used in `unlockQueue`. If the withdrawal fails, revert with a descriptive error so users and operators know to retry once Aave is operational:

```solidity
try this.withdrawFromAaveExternal(amountNeeded) {
} catch {
    revert AaveWithdrawalFailed();
}
```

Alternatively, expose a view function checking Aave's pause status and available WETH liquidity before attempting the withdrawal, and surface a clear revert reason.

## Proof of Concept
1. `isAaveIntegrationEnabled` is `true`; operator has called `unlockQueue`, depositing idle ETH into Aave via `depositToAaveExternal`. Contract's own ETH balance is near zero.
2. User calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, ...)`, burning rsETH; request is queued.
3. Operator calls `unlockQueue`; ETH is redeemed from `LRTUnstakingVault` and deposited to Aave.
4. Aave's guardian pauses the pool (or WETH utilization reaches ~100%).
5. User calls `completeWithdrawal(ETH_TOKEN, ...)`.
6. `_processWithdrawalCompletion` detects `address(this).balance < request.expectedAssetAmount`.
7. `_withdrawFromAave(amountNeeded)` → `aaveWETHGateway.withdrawETH(...)` reverts due to Aave pause.
8. `completeWithdrawal` reverts; user cannot retrieve their ETH until Aave unpauses.

**Foundry fork test plan**: Fork mainnet with Aave v3 deployed; set `isAaveIntegrationEnabled = true`; simulate `unlockQueue` depositing ETH to Aave; impersonate Aave guardian and call `pool.pause()`; call `completeWithdrawal` and assert it reverts.

    function _withdrawFromAave(uint256 amount) internal returns (uint256 withdrawnAmount) {
        if (amount == 0) return 0;

        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        if (aaveBalance == 0) revert InsufficientAaveBalance();

        // Only withdraw up to the principal amount (don't use accrued interest for user withdrawals)
        uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave;

        withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
        if (withdrawnAmount == 0) return 0;

        aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
        totalETHDepositedToAave -= withdrawnAmount;

        emit ETHWithdrawnFromAave(withdrawnAmount, totalETHDepositedToAave);
    }
