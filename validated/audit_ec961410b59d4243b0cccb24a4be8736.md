### Title
Missing Transaction Hash Validation in P2P Sync Allows Malicious Peer to Corrupt Stored BlockBody — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

---

### Summary

`TransactionStreamFactory::parse_data_for_block` accepts `FullTransaction` messages from untrusted p2p peers and stores the `transaction`, `transaction_output`, and `transaction_hash` fields directly into `BlockBody` without verifying that the hash actually corresponds to the transaction's type or content. An explicit developer TODO acknowledges this gap. A malicious peer can therefore inject a `Transaction::Invoke` body paired with a hash that belongs to an `L1Handler` (or any other type mismatch), causing the node's storage to permanently hold a `BlockBody` where `transactions[i]` and `transaction_hashes[i]` are inconsistent. Every downstream consumer — `starknet_getTransactionByHash`, `starknet_getTransactionReceipt`, `starknet_getTransactionByBlockIdAndIndex`, and simulation/tracing — reads this corrupted data verbatim and returns authoritative-looking wrong values.

---

### Finding Description

In `parse_data_for_block`, the only guard on incoming peer data is a count check: the loop runs until `current_transaction_len == target_transaction_len`, where `target_transaction_len` comes from the already-stored block header's `n_transactions` field. [1](#0-0) 

Inside the loop, the three fields of each `FullTransaction` are pushed unconditionally: [2](#0-1) 

The comment on line 88 is the developer's own acknowledgement that this validation is missing:

```
// TODO(eitan): Validate transaction hash from untrusted sources
block_body.transaction_hashes.push(transaction_hash);
```

The protobuf deserialization layer (`TryFrom<protobuf::TransactionInBlock> for (Transaction, TransactionHash)`) decodes the `txn` oneof field and the `transaction_hash` field independently — there is no cross-check between them: [3](#0-2) 

The resulting `BlockBody` is written to storage as-is: [4](#0-3) 

The block hash commits to a `transaction_commitment` (a Merkle root over transaction hashes), but the p2p sync client never validates the received transaction hashes against this commitment after storing the body: [5](#0-4) 

No post-storage cross-validation exists anywhere in the p2p sync pipeline.

---

### Impact Explanation

Once the corrupted `BlockBody` is committed, every RPC handler reads it directly from storage without re-hashing:

- **`starknet_getTransactionByHash`**: looks up the transaction index by the supplied hash, then returns the stored `Transaction` variant at that index. If the stored variant is `Invoke` but the hash belongs to an `L1Handler`, the caller receives an Invoke body for an L1Handler hash — wrong type, wrong `sender_address`, wrong `calldata`, wrong fee bounds. [6](#0-5) 

- **`starknet_getTransactionReceipt`**: reads the stored transaction to determine `tx_version` and whether to call `calc_msg_hash()`. A type mismatch (e.g., stored as `L1Handler` but body is `Invoke`) causes `calc_msg_hash()` to be called on the wrong struct, producing a wrong `msg_hash` in the receipt. [7](#0-6) 

- **`apollo_state_sync`'s `get_block`**: classifies transactions as `l1_transaction_hashes` vs `account_transaction_hashes` based on the stored `Transaction` variant, not the hash. A mismatched body causes the wrong hash to be placed in the wrong bucket, propagating the corruption to the batcher's `add_sync_block` path. [8](#0-7) 

---

### Likelihood Explanation

The p2p sync protocol is unauthenticated at the transaction-content level. Any node that the victim connects to can serve malicious `TransactionsResponse` messages. In a permissionless p2p network, an attacker can either run a peer node or perform an eclipse attack. The attack requires no special privileges — only the ability to respond to a `TransactionsRequest` query for a block the victim is syncing.

---

### Recommendation

Inside `parse_data_for_block`, after decoding each `FullTransaction`, recompute the transaction hash from the transaction body and the node's `chain_id`, then reject the peer (return `BadPeerError`) if the computed hash does not match the received `transaction_hash`. The function `get_transaction_hash` in `crates/starknet_api/src/transaction_hash.rs` already implements this computation for all transaction variants. [9](#0-8) 

Additionally, after collecting all transaction hashes for a block, validate the resulting `transaction_commitment` against the value stored in the block header to provide a second layer of defence.

---

### Proof of Concept

1. Run a p2p peer node that responds to `TransactionsRequest` for block N.
2. For each transaction in block N, send a `TransactionsResponse` where the `txn` oneof is set to `InvokeV3` (with arbitrary calldata/sender) but `transaction_hash` is copied from the real `L1Handler` transaction hash for that position.
3. The victim node's `parse_data_for_block` accepts the message (count matches `n_transactions`), pushes the mismatched pair, and calls `append_body`.
4. Query `starknet_getTransactionByHash(<real_l1_handler_hash>)` on the victim's RPC — the response returns an `INVOKE` transaction object instead of `L1_HANDLER`, with the attacker-chosen `sender_address`, `calldata`, and resource bounds.
5. Query `starknet_getTransactionReceipt(<real_l1_handler_hash>)` — `msg_hash` in the receipt is computed from the wrong (Invoke) struct, returning a fabricated value.

pub fn get_transaction_hash(
    transaction: &Transaction,
    chain_id: &ChainId,
    transaction_options: &TransactionOptions,
) -> Result<TransactionHash, StarknetApiError> {
    let transaction_version = &signed_tx_version_from_tx(transaction, transaction_options);
    match transaction {
        Transaction::Declare(declare) => match declare {
            DeclareTransaction::V0(declare_v0) => {
                get_declare_transaction_v0_hash(declare_v0, chain_id, transaction_version)
            }
            DeclareTransaction::V1(declare_v1) => {
                get_declare_transaction_v1_hash(declare_v1, chain_id, transaction_version)
            }
            DeclareTransaction::V2(declare_v2) => {
                get_declare_transaction_v2_hash(declare_v2, chain_id, transaction_version)
            }
            DeclareTransaction::V3(declare_v3) => {
                get_declare_transaction_v3_hash(declare_v3, chain_id, transaction_version)
            }
        },
        Transaction::Deploy(deploy) => {
            get_deploy_transaction_hash(deploy, chain_id, transaction_version)
        }
        Transaction::DeployAccount(deploy_account) => match deploy_account {
            DeployAccountTransaction::V1(deploy_account_v1) => {
                get_deploy_account_transaction_v1_hash(
                    deploy_account_v1,
                    chain_id,
                    transaction_version,
                )
            }
            DeployAccountTransaction::V3(deploy_account_v3) => {
                get_deploy_account_transaction_v3_hash(
                    deploy_account_v3,
                    chain_id,
                    transaction_version,
                )
            }
        },
        Transaction::Invoke(invoke) => match invoke {
            InvokeTransaction::V0(invoke_v0) => {
                get_invoke_transaction_v0_hash(invoke_v0, chain_id, transaction_version)
            }
            InvokeTransaction::V1(invoke_v1) => {
                get_invoke_transaction_v1_hash(invoke_v1, chain_id, transaction_version)
            }
            InvokeTransaction::V3(invoke_v3) => {
                get_invoke_transaction_v3_hash(invoke_v3, chain_id, transaction_version)
            }
        },
        Transaction::L1Handler(l1_handler) => {
            get_l1_handler_transaction_hash(l1_handler, chain_id, transaction_version)
        }
    }
