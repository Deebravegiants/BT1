### Title
Unguarded `init` Allows Any External Caller to Hijack Participant Set Before Legitimate Initialization - (File: `crates/contract/src/lib.rs`)

### Summary

The `init` function in `MpcContract` is marked `#[init]` but **not** `#[private]`. Any external NEAR account can call it in the window between contract deployment and the legitimate operator's initialization call. A successful front-run permanently installs an attacker-controlled participant set, giving the attacker full control over threshold signing for all foreign chains.

### Finding Description

The `init` function is the sole entry point for bootstrapping the MPC contract from the `NotInitialized` state into `Running`:

```rust
// crates/contract/src/lib.rs:1924-1940
#[handle_result]
#[init]                    // ← one-shot, but NOT #[private]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
    let parameters: ThresholdParameters = parameters.try_into_contract_type()?;
    ...
    parameters.validate()?;
    ...
    Ok(Self {
        protocol_state: ProtocolContractState::Running(RunningContractState::new(
            ...
            parameters,   // ← attacker-supplied participants stored permanently
            ...
        )),
        ...
    })
}
``` [1](#0-0) 

The `#[init]` attribute only prevents a second call once state is written; it does **not** restrict who may make the first call. The `#[private]` attribute (which enforces `predecessor_account_id == current_account_id`) is absent here, even though it is correctly applied to the analogous `init_running` and `migrate` functions:

```rust
// init_running — correctly guarded
#[private]
#[init]
pub fn init_running(...) -> Result<Self, Error> { ... }

// migrate — correctly guarded
#[private]
#[init(ignore_state)]
pub fn migrate() -> Result<Self, Error> { ... }
``` [2](#0-1) [3](#0-2) 

The ABI snapshot confirms the asymmetry: `init` carries only the `"init"` modifier, while `init_running` carries both `"init"` and `"private"`: [4](#0-3) 

The deployment flow documented in the README and all deployment scripts is:

1. Deploy contract WASM (no init call in the same transaction).
2. Call `init` in a separate transaction. [5](#0-4) [6](#0-5) 

This two-step pattern creates a window — at minimum one NEAR block — during which the contract is deployed but uninitialized, and `init` is callable by anyone.

### Impact Explanation

**Critical — Unauthorized threshold signature issuance.**

An attacker who wins the race installs an arbitrary `ThresholdParameters` (subject only to the structural checks in `parameters.validate()`, which enforce threshold bounds but impose no identity constraints). The attacker can supply two accounts they control as participants with threshold 2 (the minimum). Once `init` succeeds:

- The contract is in `Running` state with attacker-controlled participants.
- All subsequent governance votes (`vote_add_domains`, `vote_new_parameters`, `vote_update`, etc.) require a threshold of those participants — i.e., the attacker alone.
- The attacker's nodes perform DKG, generating MPC key shares they fully control.
- The attacker can issue threshold signatures for arbitrary foreign-chain transactions (Bitcoin, Ethereum, etc.) without any legitimate participant authorization.
- The legitimate operator cannot recover: `init` is one-shot; there is no `setOwner`-equivalent that an external party can call to reclaim the contract. A full redeploy is required, and any funds already routed through the hijacked contract are at risk. [7](#0-6) 

### Likelihood Explanation

**High.** The NEAR blockchain is public; every transaction is visible in real time. An attacker running a NEAR node or monitoring the RPC can observe the contract deployment transaction and submit a competing `init` call targeting the same contract account in the very next block. The attack requires no privileged access, no leaked secrets, and no collusion — only the ability to submit a NEAR transaction faster than the legitimate operator. The deployment scripts show the deploy and init steps are always separate shell commands with no atomicity guarantee. [8](#0-7) 

### Recommendation

Add `#[private]` to `init`, consistent with `init_running` and `migrate`:

```rust
#[handle_result]
#[init]
#[private]          // ← add this
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
```

This restricts the call to `predecessor_account_id == current_account_id`, meaning only the contract account itself (via a self-call or a batch transaction that deploys and initializes atomically) can invoke it. Alternatively, combine deployment and initialization into a single atomic batch transaction using NEAR's `deploy_contract` + `function_call` action batch, which eliminates the window entirely.

### Proof of Concept

```
1. Attacker monitors NEAR RPC for a DeployContract action targeting the
   known MPC contract account (e.g., v1.signer-prod.testnet).

2. In the next block, attacker submits:
     near contract call-function as-transaction <mpc_contract_account> init \
       json-args '{
         "parameters": {
           "threshold": 2,
           "participants": {
             "next_id": 2,
             "participants": [
               ["attacker-node-1.near", 0, {"tls_public_key": "<atk_pk1>", "url": "https://attacker1:13001"}],
               ["attacker-node-2.near", 1, {"tls_public_key": "<atk_pk2>", "url": "https://attacker2:13002"}]
             ]
           }
         }
       }' \
       prepaid-gas '300.0 Tgas' attached-deposit '0 NEAR' \
       sign-as attacker.near network-config mainnet sign-with-keychain send

3. init() succeeds (parameters.validate() passes: 2 participants, threshold 2 = 100% ≥ 60%).
   Contract is now Running with attacker-controlled participants.

4. Legitimate operator's init call fails: "#[init] method called on already-initialized contract".

5. Attacker's two nodes perform DKG via vote_add_domains + vote_pk.
   Attacker now holds all key shares and can sign arbitrary foreign-chain transactions.

    fn validate_threshold(n_shares: u64, k: Threshold) -> Result<(), Error> {
        if k.value() > n_shares {
            return Err(InvalidThreshold::MaxRequirementFailed {
                max: n_shares,
                found: k.value(),
            }
            .into());
        }
        if k.value() < MIN_THRESHOLD_ABSOLUTE {
            return Err(InvalidThreshold::MinAbsRequirementFailed.into());
        }
        let lower_relative_bound = governance_threshold_lower_relative_bound(n_shares);
        if k.value() < lower_relative_bound {
            return Err(InvalidThreshold::MinRelRequirementFailed {
                required: lower_relative_bound,
                found: k.value(),
            }
            .into());
        }
        let upper_relative_bound = governance_threshold_upper_relative_bound(n_shares);
        if k.value() > upper_relative_bound {
            return Err(InvalidThreshold::MaxRelRequirementFailed {
                max: upper_relative_bound,
                found: k.value(),
            }
            .into());
        }
        Ok(())
    }
