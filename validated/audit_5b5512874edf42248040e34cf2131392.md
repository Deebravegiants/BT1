### Title
Missing Early Return in `validate_delegate_action_key` Allows Nonce Consumption on Failed Delegate Action - (File: runtime/runtime/src/actions.rs)

### Summary

In `validate_delegate_action_key`, when a `FunctionCall` delegate action carries `deposit > 0`, the `DepositWithFunctionCall` error is written to `result.result` but execution is **not** halted (before `FixDelegateActionDepositWithFunctionCallError`, protocol version 85). Subsequent receiver-id and method-name checks can then either overwrite the error with a different one, or — when both checks pass — fall through to the nonce-update block, permanently incrementing the access-key nonce in the trie even though the action ultimately fails.

### Finding Description

`validate_delegate_action_key` in `runtime/runtime/src/actions.rs` validates a `DelegateAction` signed with a `FunctionCall` access key. The check sequence is:

1. If `deposit > 0` → set `result.result = Err(DepositWithFunctionCall)`. Before the fix, **no `return`**.
2. If `receiver_id` mismatches → overwrite `result.result = Err(ReceiverMismatch)`, then `return Ok(())`.
3. If method name mismatches → overwrite `result.result = Err(MethodNameMismatch)`, then `return Ok(())`.
4. **Nonce update** — `access_key.nonce` or the gas-key nonce row is written to the trie.

When `deposit > 0` **and** the receiver and method name both match (or no method-name restriction exists), neither check 2 nor check 3 fires. Execution falls through to step 4, committing the incremented nonce to state. The caller (`apply_delegate_action`) then sees `result.result.is_err()` and returns without emitting a receipt — but the nonce is already persisted. [1](#0-0)

### Impact Explanation

Two concrete protocol values are corrupted:

1. **Access-key nonce (DB entry)**: The trie entry for the signer's access key is written with `nonce = delegate_nonce` even though the action fails. Any subsequent `DelegateAction` from the same sender/key must use a strictly higher nonce; the consumed nonce slot is permanently lost.

2. **Execution outcome / RPC result**: When `deposit > 0` and the receiver mismatches, the reported error is `ReceiverMismatch` instead of `DepositWithFunctionCall`, giving callers a false signal about why the action was rejected.

### Likelihood Explanation

Trigger conditions at protocol version 84:
- A `DelegateAction` signed with a `FunctionCall` access key.
- The inner `FunctionCall` action carries `deposit > 0`.
- The `receiver_id` in the delegate action matches the access key's `receiver_id`.
- Either no `method_names` restriction exists, or the method name matches.

A user who mistakenly attaches a deposit to a meta-transaction (a common mistake when porting regular transactions to the delegate pattern) will silently lose a nonce. A malicious relayer holding such a signed action can replay it to grief the sender's nonce sequence. The stable protocol version is 86 (fix active), but the unfixed code path remains live for protocol version 84 blocks.

### Recommendation

The fix (`FixDelegateActionDepositWithFunctionCallError`, protocol version 85) is already in place. Ensure the legacy path at protocol version 84 is not reachable on any live network. Additionally, consider adding a defensive assertion that `result.result.is_ok()` before executing the nonce-update block, so future refactors cannot reintroduce the same class of bug.

### Proof of Concept

1. Create a `FunctionCall` access key for `alice` with `receiver_id = "bob.near"`, `method_names = []`.
2. Have `alice` sign a `DelegateAction` whose inner action is `FunctionCall { method_name: "foo", deposit: 1 yoctoNEAR, receiver_id: "bob.near" }` with nonce `N+1`.
3. Relayer submits the action at protocol version 84.
4. `validate_delegate_action_key` sets `DepositWithFunctionCall`, skips the early return, passes the receiver check, passes the method-name check (empty list), and writes nonce `N+1` to the trie.
5. `apply_delegate_action` sees `result.result.is_err()` and returns without a receipt.
6. Querying `alice`'s access key shows `nonce = N+1`; the next valid `DelegateAction` must use `N+2`. The action failed but the nonce was consumed.
