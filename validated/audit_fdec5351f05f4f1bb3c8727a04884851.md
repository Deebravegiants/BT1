### Title
Harvester `farmer_reward_address_override` Fee Threshold Unenforced — Unconditional Farmer Reward Diversion - (File: chia/farmer/farmer_api.py)

### Summary

The CHIP-22 harvester fee convention allows a third-party harvester to redirect the farmer's block reward to itself by setting `farmer_reward_address_override`. The farmer's only guard — `notify_farmer_reward_taken_by_harvester_as_fee` — is **log-only**: it never rejects or blocks the override regardless of whether the fee quality check passes or fails. A malicious harvester can unconditionally divert 100% of the farmer's XCH block reward to an attacker-controlled address on every won block.

### Finding Description

When a harvester sends `NewProofOfSpace` with a non-`None` `farmer_reward_address_override`, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee` and then **unconditionally continues** the block-making flow: [1](#0-0) 

`notify_farmer_reward_taken_by_harvester_as_fee` computes a `fee_quality` from the proof bytes and compares it against the harvester-supplied `applied_fee_threshold`. If `fee_quality > fee_threshold`, it emits a `log.warning`. It **never returns early, raises, or rejects the override**: [2](#0-1) 

A malicious harvester trivially bypasses even the warning by setting `applied_fee_threshold = 0xFFFFFFFF` (max `uint32`), making `fee_quality <= fee_threshold` always true, so the farmer logs "Fee threshold passed" and proceeds silently.

Later, in `_process_respond_signatures`, the `farmer_reward_address_override` from the harvester's `RespondSignatures` is accepted with no validation whatsoever: [3](#0-2) 

The resulting `DeclareProofOfSpace` carries the attacker's puzzle hash as `farmer_reward_address`, which the full node uses to create the farmer reward coin: [4](#0-3) 

The protocol message that enables this is: [5](#0-4) 

### Impact Explanation

Every time the farmer wins a block, the farmer reward coin (1/8 of the block reward, e.g. 0.25 XCH at current halving) is created at the attacker's puzzle hash instead of the farmer's configured `farmer_target`. The farmer receives nothing. This is a direct, permanent, on-chain diversion of XCH to an unauthorized address. The farmer has no on-chain or protocol-level recourse once the block is confirmed.

This matches the **High** impact category: *"Bypass of wallet, daemon, keychain, RPC, pool, or Data Layer authorization that enables unauthorized signing, key use, coin control, payout redirection."*

### Likelihood Explanation

Any third-party harvester that connects to a farmer can exploit this. The CHIP-22 feature is explicitly designed to allow third-party harvesters (e.g. DrPlotter-style services), so the attack surface is real and reachable by an unprivileged network peer that the farmer has chosen to connect to. No key compromise, admin access, or cryptographic break is required — the harvester only needs to set two fields in a protocol message it is already authorized to send.

### Recommendation

The farmer must enforce a configurable maximum fee threshold. When `farmer_reward_address_override` is set, the farmer should:

1. Reject the override (drop the block-making flow and log an error) if `fee_quality > applied_fee_threshold` rather than merely warning.
2. Allow the farmer operator to configure a `max_harvester_fee_threshold` in `config.yaml`; reject any override whose `applied_fee_threshold` exceeds this value.
3. Apply the same enforcement in `_process_respond_signatures` when `response.farmer_reward_address_override` is non-`None`, cross-checking it against the threshold recorded when `NewProofOfSpace` was first received.

### Proof of Concept

1. A malicious harvester connects to a victim farmer.
2. On receiving a `NewSignagePointHarvester`, the harvester finds a valid proof of space.
3. Instead of sending a normal `NewProofOfSpace`, it sends one with:
   - `farmer_reward_address_override = <attacker_puzzle_hash>`
   - `fee_info = ProofOfSpaceFeeInfo(applied_fee_threshold=0xFFFFFFFF)`
4. The farmer calls `notify_farmer_reward_taken_by_harvester_as_fee`: `fee_quality <= 0xFFFFFFFF` is always true → logs "Fee threshold passed" → **no rejection**.
5. The farmer sends `RequestSignatures` to the harvester.
6. The harvester returns `RespondSignatures` with `farmer_reward_address_override = <attacker_puzzle_hash>`.
7. `_process_respond_signatures` sets `farmer_reward_address = attacker_puzzle_hash` unconditionally.
8. `DeclareProofOfSpace` is submitted to the full node with the attacker's address as `farmer_reward_address`.
9. The block is confirmed; the farmer reward coin is created at the attacker's address. The farmer receives 0 XCH for the won block.
