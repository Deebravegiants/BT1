### Title
Unverified CKD Response Accepted for Legacy `AppPublicKey` Variant Allows Single Malicious Participant to Corrupt User Key Derivation - (File: `crates/contract/src/lib.rs`)

---

### Summary

In `respond_ckd`, when a CKD request uses the legacy `CKDAppPublicKey::AppPublicKey` variant, the contract performs **no cryptographic verification** of the submitted `CKDResponse`. A single malicious attested participant can call `respond_ckd` with arbitrary fabricated `big_y` and `big_c` values, and the contract will accept the response, consume the user's pending request, and deliver the garbage output to all waiting callers. The `AppPublicKeyPV` variant is protected by an on-chain BLS pairing check (`ckd_output_check`), but the legacy variant has an explicit empty match arm that skips all verification.

---

### Finding Description

**Root cause — `respond_ckd`, lines 675–682:**

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no verification
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}

pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract enforces `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)` via `ckd_output_check` in `crates/contract/src/primitives/ckd.rs`. [2](#0-1) 

For `AppPublicKey`, the match arm is an empty no-op. The contract then unconditionally calls `resolve_yields_for`, which delivers the response to every queued yield for that request key and removes the pending entry. [3](#0-2) 

**Why the TEE attestation check does not prevent this:**

`assert_caller_is_attested_participant_and_protocol_active` (line 666) only verifies that the caller's account has a previously stored attestation entry in `tee_state.stored_attestations`. It checks the stored `account_public_key` against the transaction signer key — it does **not** verify that the current call originates from within a TEE enclave. [4](#0-3) 

A participant who obtained a valid attestation (e.g., by running the correct node software once) can subsequently craft and submit a `respond_ckd` transaction with arbitrary `big_y`/`big_c` values from outside the TEE, using their stored NEAR account key. The contract's attestation check passes because it only reads the stored entry.

**Attack path:**

1. Attacker is a legitimate MPC participant with a valid stored attestation.
2. A user submits `request_app_private_key` with `AppPublicKey` (the legacy G1-only variant). The pending request is queued under `pending_ckd_requests`.
3. Before the honest leader submits the real response, the attacker crafts a `respond_ckd` call with `big_y = [0u8; 48]` and `big_c = [0u8; 48]` (or any arbitrary bytes).
4. The contract accepts the call: the `AppPublicKey` match arm is a no-op, `resolve_yields_for` fires, the pending entry is removed, and the garbage `(big_y, big_c)` is delivered to the user's yield.
5. The user receives a fake CKD output. The real threshold-computed key is never delivered.

The `AppPublicKey` variant is still actively supported and documented as the "privately verifiable (legacy)" path. [5](#0-4) 

The unit test at line 3403 confirms the contract accepts `CKDResponse { big_y: [1u8; 48], big_c: [2u8; 48] }` (clearly fabricated values) without any cryptographic check when `AppPublicKey` is used. [6](#0-5) 

---

### Impact Explanation

**Classification: Medium — request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants.**

The production safety invariant violated is: *every CKD response delivered to a caller must be a cryptographically valid output of the threshold BLS protocol, i.e., `C = msk · H(pk, app_id) + a·Y`.*

Consequences:
- The user's pending CKD request is permanently consumed. The yield is resolved with garbage; the user cannot recover the real derived key from this response.
- Any application (e.g., a TEE app) that uses the derived key to access secrets or assets on a foreign chain receives an invalid key and cannot proceed.
- The attacker can repeat this for every pending `AppPublicKey` CKD request, systematically denying all legacy-variant users their derived keys.
- Because the `AppPublicKey` variant is "privately verifiable," the user has no on-chain mechanism to detect the forgery; they must attempt decryption locally to discover the response is garbage.

This does not meet the Critical threshold because the attacker does not obtain the real derived key or any MPC key material — they only deliver garbage, not a real unauthorized key derivation output.

---

### Likelihood Explanation

**Low.** The attacker must be a legitimate MPC participant with a valid stored TEE attestation. Becoming a participant requires governance approval (threshold vote). However, only **one** participant needs to be malicious — no threshold collusion is required. The attack requires crafting a NEAR transaction outside the TEE using the participant's account key, which is operationally straightforward once the participant's account key is available.

---

### Recommendation

1. **Deprecate and reject `AppPublicKey` in `respond_ckd`**: Require all new CKD responses to use `AppPublicKeyPV`, which has an enforceable on-chain pairing check. Add a `panic` or `Err` for the `AppPublicKey` arm in `respond_ckd` to prevent unverified responses from being accepted.

2. **If `AppPublicKey` must remain**: Implement a threshold-based response collection mechanism — require at least `t` participants to submit the same `(big_y, big_c)` before the response is delivered, analogous to how threshold signing requires `t`-of-`n` agreement. This prevents a single Byzantine participant from unilaterally delivering a fake response.

---

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }

- `app_public_key`: the ephemeral public key for the CKD request. Two formats are supported:
  - **Privately verifiable** (legacy): a single G1 point, e.g. `"bls12381g1:<base58>"` or `{"AppPublicKey": "bls12381g1:<base58>"}`.
  - **Publicly verifiable**: a pair of points `(pk1, pk2) = (a·G1, a·G2)`, passed as `{"AppPublicKeyPV": {"pk1": "bls12381g1:<base58>", "pk2": "bls12381g2:<base58>"}}`. This allows anyone to verify the encrypted result on-chain without the app's secret key.
- `domain_id` (integer): identifies the master key to use for deriving the ckd, and must correspond to bls12381.
