### Title
Meta Transaction `SignedDelegateAction` Front-Running Invalidates Relayer's Transaction - (`File: runtime/runtime/src/actions.rs`)

### Summary

The NEAR meta transaction (NEP-366) system allows a user to sign a `DelegateAction` off-chain and hand it to a relayer, who wraps it in an outer transaction and pays gas. Because the `SignedDelegateAction` is fully visible in the mempool inside the relayer's pending transaction, an unprivileged attacker can extract it, wrap it in their own outer transaction, and submit it first. When the attacker's transaction executes, the sender's access-key nonce is advanced. The legitimate relayer's transaction then fails with `DelegateActionInvalidNonce`, wasting the relayer's gas and disrupting the meta-transaction flow. Additionally, the `signer_id` field of the generated inner receipt is taken from the outer transaction's signer, so the attacker can substitute their own account as `signer_id` in the inner receipt.

### Impact Explanation

An attacker who observes the relayer's pending transaction in the mempool can:

1. Extract the `SignedDelegateAction` (signature is valid and reusable by anyone).
2. Wrap it in their own outer transaction (attacker becomes the outer signer/relayer).
3. Submit with sufficient gas priority to land first.

Consequences:
- The sender's access-key nonce is advanced by the attacker's transaction.
- The legitimate relayer's transaction fails with `DelegateActionInvalidNonce`, burning the relayer's gas with no benefit.
- The `signer_id` of the inner receipt is the attacker's account instead of the original relayer's account, which can affect contracts that inspect `env::signer_account_id()`.

The corrupted protocol values are: the relayer's NEAR balance (gas wasted on a failed receipt) and the `signer_id` field of the inner action receipt.

### Likelihood Explanation

NEAR transactions are publicly visible in the mempool before inclusion. Extracting a `SignedDelegateAction` from a pending transaction requires only standard RPC access. Submitting a competing transaction is a standard unprivileged operation. The attack is cheap (attacker pays only their own gas) and repeatable, making it a viable griefing vector against any relayer service.

### Recommendation

Mirror the OpenZeppelin recommendation from the original report: treat a nonce-already-used failure as a non-fatal condition. In `apply_delegate_action`, if `validate_delegate_action_key` returns `DelegateActionInvalidNonce` and the inner actions have already been executed (i.e., the nonce was consumed by an identical `SignedDelegateAction`), the relayer's outer transaction should not be charged full gas for a failure that was caused by front-running. More concretely:

- Consider adding a "nonce already used by identical payload" fast-path that returns a cheaper error code, so relayers can detect and handle this case.
- Relayer implementations should monitor for competing submissions of the same `SignedDelegateAction` and avoid resubmitting after the nonce is consumed.
- Protocol-level: consider binding the `SignedDelegateAction` to a specific outer-transaction signer (relayer account) so that only the designated relayer can submit it, preventing third-party extraction and reuse.

### Proof of Concept

1. Alice creates and signs a `DelegateAction` with nonce `N` and hands it to Relayer. [6](#0-5) 

2. Relayer broadcasts an outer transaction containing the `SignedDelegateAction`. [7](#0-6) 

3. Attacker observes the pending transaction, extracts the `SignedDelegateAction`, wraps it in their own outer transaction (attacker as signer), and broadcasts with higher priority.

4. Attacker's transaction executes first. `validate_delegate_action_key` passes (nonce `N > current_nonce`), nonce is advanced to `N`. [3](#0-2) 

5. Relayer's transaction executes next. `validate_delegate_action_key` fails: `N <= N` → `DelegateActionInvalidNonce`. [2](#0-1) 

6. Relayer's gas is burned; the relayer receives no compensation. The inner receipt's `signer_id` is the attacker's account, not the relayer's. [8](#0-7)

pub struct SignedDelegateAction {
    pub delegate_action: DelegateAction,
    pub signature: Signature,
}

impl SignedDelegateAction {
    pub fn verify(&self) -> bool {
        let delegate_action = &self.delegate_action;
        let hash = delegate_action.get_nep461_hash();
        let public_key = &delegate_action.public_key;

        self.signature.verify(hash.as_ref(), public_key)
    }

    pub fn sign(singer: &Signer, delegate_action: DelegateAction) -> Self {
        let signature = singer.sign(delegate_action.get_nep461_hash().as_bytes());
        Self { delegate_action, signature }
    }
}
