### Title
Missing Binding Between `response.payload_hash` and `request` in `respond_verify_foreign_tx` Allows Cross-Request Signature Replay - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is a valid MPC signature over `response.payload_hash`, but it never verifies that `response.payload_hash` was actually derived from the `request` supplied in the same call. A single Byzantine participant (below the signing threshold) can replay a legitimately-produced MPC signature for foreign transaction X to resolve a pending verification request for a completely different foreign transaction Y, causing the contract to return a forged attestation to the original caller.

---

### Finding Description

The `respond_verify_foreign_tx` function in `crates/contract/src/lib.rs` performs two independent checks:

1. It looks up the pending request by `VerifyForeignTransactionRequest { request, domain_id, payload_version }`.
2. It verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash` using the domain's root public key. [1](#0-0) 

Critically, there is **no check** that `response.payload_hash` was computed from the `request` field of the call. The canonical hash that MPC nodes are supposed to sign is:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))
``` [2](#0-1) 

The `ForeignTxSignPayloadV1` contains `request` and `values` — but neither the `domain_id` nor any binding to the pending-request map key is included in the hash. The contract never reconstructs or checks this hash from the `request` it already holds in state. [3](#0-2) 

The `VerifyForeignTransactionRequest` struct used as the pending-map key contains `{ request, domain_id, payload_version }`: [4](#0-3) 

The conversion from user args to this key drops the caller's identity entirely: [5](#0-4) 

This is structurally identical to the H-07 pattern: a field that participates in request lookup (`request` content) is not bound into the signed artifact (`payload_hash`), so a valid signature produced for one context can be injected into a different context.

---

### Impact Explanation

A single Byzantine participant (below the signing threshold) can:

1. Observe a legitimately-produced MPC signature `(payload_hash_X, sig_X)` for foreign transaction X — this is publicly visible on-chain when the honest network responds to any prior request.
2. Submit `verify_foreign_transaction` for a **different** foreign transaction Y (as an ordinary user, no privilege required).
3. Call `respond_verify_foreign_tx` with `request = pending_request_Y`, `response.payload_hash = payload_hash_X`, `response.signature = sig_X`.
4. The contract verifies `sig_X` over `payload_hash_X` — valid — and resolves the pending request for Y, returning `(payload_hash_X, sig_X)` to the original caller.

The caller of `verify_foreign_transaction(Y)` receives an attestation that is cryptographically valid but attests to foreign transaction X, not Y. Any bridge contract that does not independently recompute the expected `payload_hash` from the known `(request, extracted_values)` pair — using the SDK's `ForeignChainSignatureVerifier::verify_signature` — will accept this forged attestation and execute based on incorrect foreign-chain state. [6](#0-5) 

The SDK helper does perform this check, but the **contract itself does not enforce it**, leaving all callers that skip the SDK verification exposed to forged bridge execution or double-spend.

---

### Likelihood Explanation

- `respond_verify_foreign_tx` is callable by **any single attested participant** — no threshold agreement is required to submit a response.
- A valid signature for any prior foreign-tx verification is permanently observable on-chain once submitted.
- The attacker only needs to be one attested participant acting alone, which is explicitly within the Byzantine-participant-below-threshold threat model.
- The attacker can self-generate the target pending request by calling `verify_foreign_transaction` themselves, requiring only a 1 yoctoNEAR deposit. [7](#0-6) 

---

### Recommendation

Include the `request` content in the signed payload hash verification inside `respond_verify_foreign_tx`. Concretely, the contract should require the caller to also supply the `extracted_values` (the `Vec<ExtractedValue>` observed from the foreign chain), reconstruct `ForeignTxSignPayload::V1 { request, values }`, recompute `expected_hash = SHA-256(borsh(...))`, and assert `response.payload_hash == expected_hash` before accepting the response.

This mirrors the fix recommended in H-07: include the missing context identifier (`batchId` there; `request` content here) in the hash that is verified, so that a signature produced for one context cannot be replayed into another.

---

### Proof of Concept

```rust
// Setup: honest MPC network has already responded to a request for foreign tx X.
// The on-chain response (payload_hash_X, sig_X) is publicly visible.

// Step 1: Byzantine participant submits a new verify request for foreign tx Y.
contract.verify_foreign_transaction(VerifyForeignTransactionRequestArgs {
    domain_id: foreign_tx_domain_id,
    payload_version: ForeignTxPayloadVersion::V1,
    request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
        tx_id: tx_id_Y,  // different from tx_id_X
        confirmations: 2.into(),
        extractors: vec![BitcoinExtractor::BlockHash],
    }),
});

// Step 2: Byzantine participant calls respond with the signature for tx X,
// but the pending request for tx Y.
contract.respond_verify_foreign_tx(
    VerifyForeignTransactionRequest {
        domain_id: foreign_tx_domain_id,
        payload_version: ForeignTxPayloadVersion::V1,
        request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
            tx_id: tx_id_Y,  // pending request for Y
            ...
        }),
    },
    VerifyForeignTransactionResponse {
        payload_hash: payload_hash_X,  // hash of tx X's payload
        signature: sig_X,              // valid MPC signature over payload_hash_X
    },
);
// Contract accepts: sig_X is valid over payload_hash_X.
// Caller of verify_foreign_transaction(Y) receives attestation for tx X.
// No error is returned; the forged response is delivered.
``` [8](#0-7)
