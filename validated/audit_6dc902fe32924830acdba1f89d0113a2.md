### Title
No Arbitrum Sequencer Liveness Check in `EvmInspector` Allows Signing of Stale/Reorganized Transaction Data - (File: `crates/foreign-chain-inspector/src/evm/inspector.rs`)

---

### Summary

The `EvmInspector` used for Arbitrum (and other L2 chains: Base, Polygon, BNB, HyperEvm, Abstract) does not check whether the L2 sequencer is active before trusting RPC data. This is the direct analog of M-4: the same generic EVM inspector code is reused for all EVM-family chains without any Arbitrum-specific sequencer liveness guard. When the Arbitrum sequencer is down, RPC nodes continue serving the last known state. A user can submit a `verify_foreign_transaction` request with `EvmFinality::Latest` for an Arbitrum transaction, receive a valid MPC threshold signature over that payload, and then exploit a sequencer restart reorg to produce a signed attestation for a transaction that no longer exists on-chain.

---

### Finding Description

`EvmInspector::extract` in `crates/foreign-chain-inspector/src/evm/inspector.rs` performs two checks before signing:

1. `verify_finality_level` — calls `eth_getBlockByNumber("latest"|"safe"|"finalized")` and asserts `head.number >= receipt_block_number`.
2. `verify_block_is_canonical` — re-fetches the canonical block at the receipt's height and compares hashes.

Neither check queries whether the Arbitrum sequencer is currently live. When the Arbitrum sequencer goes offline, Arbitrum RPC nodes continue to serve the last committed state. The `eth_getBlockByNumber("latest")` call returns the last block produced before the outage. A transaction included in that stale "latest" block passes both checks because:

- `head.number` (stale latest) is still `>= receipt_block_number` (the block the tx was in).
- The canonical block at that height still matches the receipt's `block_hash` (no reorg has happened yet).

The `ArbitrumInspector` is a thin wrapper over this same `EvmInspector<Client, Arbitrum>` code path with no additional sequencer-uptime guard. The node dispatches to it at `crates/node/src/providers/verify_foreign_tx/sign.rs` lines 218–238 by converting `EvmFinality` directly to `EthereumFinality` and calling `.extract(transaction_id, finality, extractors)`.

The `EvmFinality::Latest` variant is fully accepted by the contract's `VerifyForeignTransactionRequestArgs` — there is no restriction preventing a caller from requesting `Latest` finality for Arbitrum. The `ForeignTxSignPayloadV1` that the MPC network signs encodes the `ForeignChainRpcRequest` (including the caller-chosen `finality: EvmFinality::Latest`) and the extracted values, so the resulting signature is a durable, on-chain-stored attestation.

---

### Impact Explanation

**High — forged foreign-chain verification enabling invalid bridge execution / double-spend.**

The primary use case for `verify_foreign_transaction` is the Omnibridge inbound flow: a user deposits on a foreign chain, the MPC network attests to the deposit, and the NEAR-side bridge contract releases funds. If the MPC network signs a `ForeignTxSignPayloadV1` for a transaction that is subsequently reorganized away by the Arbitrum sequencer on restart, the attacker holds a valid threshold signature attesting to a deposit that never settled. They can present this signature to the NEAR bridge contract to claim funds for a deposit that does not exist on Arbitrum, constituting a direct double-spend of bridge funds.

---

### Likelihood Explanation

- Arbitrum sequencer outages have occurred historically (e.g., December 2021, June 2023).
- `EvmFinality::Latest` is a caller-controlled parameter with no contract-level restriction for Arbitrum.
- The attacker only needs to: (a) have a transaction included in an Arbitrum block not yet posted to L1, (b) trigger or wait for a sequencer outage, (c) submit `verify_foreign_transaction` during the outage window, and (d) benefit from the sequencer reorg on restart.
- No privileged access, TEE compromise, or threshold collusion is required — the attack is fully reachable by an unprivileged contract caller.

---

### Recommendation

1. **Reject `EvmFinality::Latest` for known L2 sequencer chains** (`Arbitrum`, `Base`, `Polygon`, `Bnb`, `HyperEvm`, `Abstract`) at the contract level or in the inspector, requiring at minimum `Safe` (batch posted to L1) or `Finalized`.
2. **Add a sequencer-uptime probe** for Arbitrum (and Base): before trusting any RPC response, query the Chainlink Sequencer Uptime Feed (`0xFdB631F5EE196F0ed6FAa767959853A9F217697D` on Arbitrum One) and reject the request if the sequencer is reported down or within the grace period after restart — exactly the fix recommended in M-4.
3. **Enforce a grace period** after sequencer restart (analogous to `GRACE_PERIOD_TIME` in `ArbiChainlinkOracle.sol`) to avoid signing data from the first blocks after a sequencer comes back online, which may still reflect pre-reorg state.

### Proof of Concept

```
1. Attacker submits tx A on Arbitrum (e.g., a bridge deposit of 100 ETH).
   tx A lands in block N, which has NOT yet been posted to L1.

2. Arbitrum sequencer goes offline.
   RPC nodes freeze at block N; eth_getBlockByNumber("latest") returns block N.

3. Attacker calls verify_foreign_transaction({
       request: ForeignChainRpcRequest::Arbitrum(EvmRpcRequest {
           tx_id: tx_A_hash,
           finality: EvmFinality::Latest,   // caller-controlled, no restriction
           extractors: [EvmExtractor::Log { log_index: 0 }],
       }),
       domain_id: foreign_tx_domain_id,
       payload_version: V1,
   })

4. MPC nodes query Arbitrum RPC:
   - eth_getTransactionReceipt(tx_A) → receipt with block_number=N, block_hash=H
   - eth_getBlockByNumber("latest")  → block N  (stale, sequencer is down)
   - verify_finality_level: N >= N  ✓
   - eth_getBlockByNumber(N)         → block N, hash=H  (canonical check passes)
   - Transaction status = success    ✓
   All checks pass. MPC signs ForeignTxSignPayloadV1{request, [Log(...)]}

5. Sequencer restarts, reorgs block N (tx A is dropped or reordered away).
   tx A no longer exists on Arbitrum.

6. Attacker presents the MPC threshold signature to the NEAR bridge contract,
   claiming the 100 ETH deposit. Bridge releases funds.
   Attacker has claimed funds for a deposit that does not exist on Arbitrum.
```

The root cause is in `EvmInspector::verify_finality_level` [1](#0-0) , which compares block numbers against a stale RPC head without any sequencer-liveness guard. The `ArbitrumInspector` is dispatched at [2](#0-1)  using the same code path. The `EvmFinality::Latest` variant that enables the attack is unrestricted in the contract interface [3](#0-2) , and the `ForeignChainRpcRequest::Arbitrum` variant is a first-class supported chain [4](#0-3) .

    async fn verify_finality_level(
        &self,
        receipt_block_number: U64,
        finality: EthereumFinality,
    ) -> Result<(), ForeignChainInspectionError> {
        let finality_tag = match finality {
            EthereumFinality::Finalized => FinalityTag::Finalized,
            EthereumFinality::Safe => FinalityTag::Safe,
            EthereumFinality::Latest => FinalityTag::Latest,
        };
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Tag(finality_tag),
            ReturnFullTransactionObjects::from(false),
        );
        let head: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        if head.number < receipt_block_number {
            return Err(ForeignChainInspectionError::NotFinalized);
        }
        Ok(())
    }
