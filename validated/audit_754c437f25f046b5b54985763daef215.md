### Title
`respond_verify_foreign_tx` Does Not Bind `payload_hash` to the Submitted `request` — (`crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` on-chain handler verifies only that the MPC signature is cryptographically valid over the caller-supplied `response.payload_hash`. It never checks that `payload_hash` is actually the SHA-256 hash of `ForeignTxSignPayload::V1 { request, extracted_values }` for the specific `request` being resolved. A single attested MPC participant (strictly below the signing threshold) can therefore recycle a legitimately-produced threshold signature from one foreign-chain verification request to close out a completely different pending request, delivering a forged attestation to the waiting NEAR contract.

---

### Finding Description

`respond_verify_foreign_tx` in `crates/contract/src/lib.rs` performs the following checks before resolving the yield:

```rust
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,   // ← taken verbatim from the caller-supplied response
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract then immediately resolves the yield for the caller-supplied `request`:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

There is no step that recomputes `expected_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))` and asserts `response.payload_hash == expected_hash`. The contract trusts the caller to supply a `payload_hash` that is consistent with the `request` it is resolving.

The correct hash construction is defined in the interface crate:

```rust
impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
}
``` [3](#0-2) 

The SDK-side verifier (`ForeignChainSignatureVerifier::verify_signature`) does perform this check, but it is an off-chain helper for callers — the on-chain contract does not enforce it:

```rust
let payload_is_correct = expected_payload_hash == response.payload_hash;
if !payload_is_correct {
    return Err(VerifyForeignChainError::IncorrectPayloadSigned { ... });
}
``` [4](#0-3) 

**Attack path (single node, below threshold):**

1. User A submits `verify_foreign_transaction(request_A)` → yield A is queued.
2. User B submits `verify_foreign_transaction(request_B)` → yield B is queued.
3. The MPC network legitimately processes request A and produces `(payload_hash_A, sig_A)` where `payload_hash_A = SHA-256(borsh(ForeignTxSignPayload::V1 { request_A, values_A }))`.
4. A single malicious attested participant calls `respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: sig_A })`.
5. The contract verifies `sig_A` over `payload_hash_A` — this passes because the signature is a genuine threshold signature.
6. Yield B is resolved with `{ payload_hash: payload_hash_A, signature: sig_A }`.
7. User B's NEAR contract receives an attestation that encodes request A's foreign-chain data, not request B's.

The malicious node does not need to forge a threshold signature; it only needs to be the first caller of `respond_verify_foreign_tx` for request B, using a recycled signature from any prior legitimate operation on the same domain key.

---

### Impact Explanation

A NEAR bridge contract (e.g., Omnibridge inbound flow) that calls `verify_foreign_transaction` and then acts on the returned `VerifyForeignTransactionResponse` without independently recomputing the expected `payload_hash` will accept the forged attestation as proof that request B's foreign-chain transaction finalized. This enables:

- **Invalid bridge execution**: a deposit on foreign chain A is used to unlock funds on NEAR for a completely different transaction B.
- **Double-spend**: the same threshold signature (produced once for transaction A) can be replayed to resolve multiple pending requests for different transactions, each time unlocking NEAR-side funds.

This matches the allowed impact: *"Cross-chain replay, forged foreign-chain verification … that causes invalid bridge execution or double-spend conditions."*

---

### Likelihood Explanation

- Only a single attested MPC participant is required — strictly below the signing threshold.
- No cryptographic forgery is needed; the attacker reuses a legitimately produced threshold signature.
- The attacker only needs to win a race to be the first caller of `respond_verify_foreign_tx` for the target request, which is straightforward given direct NEAR RPC access.
- The primary use case of `verify_foreign_transaction` is bridge inbound flows where financial value is at stake, making this an attractive target.

---

### Recommendation

Inside `respond_verify_foreign_tx`, after extracting `payload_hash` from the response, the contract must verify that it is consistent with the pending request. Because the contract does not store `extracted_values` on-chain, the simplest fix is to require the responder to also supply the `extracted_values` and have the contract recompute and assert the hash:

```rust
// Pseudocode for the fix
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(), // new field in response
});
let expected_hash = expected_payload.compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Alternatively, the contract can enforce that `payload_hash` is derived from the `request` by having nodes sign a commitment that includes the `request` identifier, and verifying that commitment on-chain before accepting the response.

---

### Proof of Concept

```
1. Alice submits verify_foreign_transaction(bitcoin_tx_id=0xAAAA, confirmations=1)
   → pending_verify_foreign_tx_requests[req_A] = [yield_A]

2. Bob submits verify_foreign_transaction(bitcoin_tx_id=0xBBBB, confirmations=1)
   → pending_verify_foreign_tx_requests[req_B] = [yield_B]

3. MPC network legitimately signs req_A:
   payload_hash_A = SHA256(borsh(ForeignTxSignPayloadV1 { req_A, [BlockHash(0xAAAA_block)] }))
   sig_A = threshold_sign(payload_hash_A)

4. Malicious node calls:
   respond_verify_foreign_tx(
     request = req_B,                          // Bob's request
     response = { payload_hash: payload_hash_A, // Alice's hash
                  signature:    sig_A }         // Alice's signature
   )

5. Contract checks: verify_ecdsa(sig_A, payload_hash_A, root_pk) → OK
   → yield_B resolved with { payload_hash_A, sig_A }

6. Bob's bridge contract receives the response, sees a valid MPC signature,
   and unlocks NEAR-side funds for bitcoin_tx_id=0xBBBB — which was never
   actually verified by the MPC network.
``` [5](#0-4) [6](#0-5)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
