### Title
Sub-transaction reference script sizes excluded from tiered fee calculation in Dijkstra era — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

### Summary

In the Dijkstra era, the minimum-fee check (`validateFeeTooSmallUTxO`) applied to a batched transaction only accounts for reference scripts referenced by the **top-level** transaction body. Reference scripts pulled in by **sub-transactions** are silently excluded from the tiered fee (`tierRefScriptFee`) calculation, even though the node must deserialize and validate all of them. Because sub-transactions carry no fee field of their own, the entire batch fee is declared in the top-level tx — yet the fee guard only measures the top-level tx's ref-script footprint.

### Finding Description

`dijkstraUtxoTransition` enforces the fee invariant at line 373:

```haskell
runTest $ Shelley.validateFeeTooSmallUTxO pp tx originalUtxo
```

`validateFeeTooSmallUTxO` calls `getMinFeeTxUtxo`, which for `DijkstraEra` is bound to `getConwayMinFeeTxUtxo`. That function computes the ref-script size via `txNonDistinctRefScriptsSize`, which unions only `referenceInputsTxBodyL` and `inputsTxBodyL` of the **top-level** tx body — it never iterates over `subTransactionsTxBodyL`.

The function `batchNonDistinctRefScriptsSize` was introduced precisely to aggregate ref-script sizes across the whole batch (top-level + all sub-txs). It is used in `validateAllRefScriptSize` (LEDGER rule) to enforce the 200 KiB per-tx size cap, but it is **never used** in the fee calculation path.

An attacker can therefore:
1. Craft a top-level `DijkstraTxBody` with zero or minimal reference inputs (paying zero or minimal tiered ref-script fee).
2. Embed sub-transactions that each reference large Plutus scripts (up to the 200 KiB batch cap).
3. Set `dtbTxfee` to exactly the minimum fee computed by `getConwayMinFeeTxUtxo` (top-level only).
4. Submit the transaction. The fee check passes; the node must still deserialize and validate all sub-tx reference scripts.

### Impact Explanation

This is a **Medium** impact finding: attacker-controlled transactions modify fees outside design parameters. The tiered pricing mechanism (`tierRefScriptFee`) was introduced specifically to make large-script DDoS attacks economically infeasible. By routing large scripts through sub-transactions, an attacker bypasses the tiered surcharge entirely. At the 200 KiB batch cap, the tiered fee for sub-tx scripts alone would be on the order of thousands of ADA at current parameters — all of which can be avoided. This enables systematic, cheap submission of computationally expensive batched transactions, recreating the economic conditions of the June 2024 DDoS attack within the Dijkstra era's new batching model.

### Likelihood Explanation

The exploit requires only the ability to submit a valid Dijkstra-era transaction with sub-transactions — no privileged access, no governance majority, no leaked keys. The construction is straightforward: place large reference-script UTxO inputs in sub-tx bodies rather than the top-level body. The discrepancy between `batchNonDistinctRefScriptsSize` (used for size cap) and `txNonDistinctRefScriptsSize` (used for fee) is directly observable in the source code.

### Recommendation

Replace the fee-check call in `dijkstraUtxoTransition` with one that uses `batchNonDistinctRefScriptsSize` instead of the Conway-inherited `getConwayMinFeeTxUtxo`. Concretely, override `getMinFeeTxUtxo` in the `EraUTxO DijkstraEra` instance to use a Dijkstra-specific function:

```haskell
getDijkstraMinFeeTxUtxo :: (EraTx era, DijkstraEraTxBody era, ConwayEraPParams era)
  => PParams era -> Tx TopTx era -> UTxO era -> Coin
getDijkstraMinFeeTxUtxo pp tx utxo =
  getMinFeeTx pp tx (batchNonDistinctRefScriptsSize utxo tx)
```

and bind `getMinFeeTxUtxo = getDijkstraMinFeeTxUtxo` in the `EraUTxO DijkstraEra` instance (currently at line 141 of `Dijkstra/UTxO.hs`).

### Proof of Concept

```

  let largeScript = <200 KiB Plutus script stored in UTxO>
```
