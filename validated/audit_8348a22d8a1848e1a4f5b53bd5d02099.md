Audit Report

## Title
Depositors Mint rsETH at Stale Price, Diluting Accumulated Yield of Existing Holders - (File: `contracts/LRTOracle.sol`, `contracts/LRTDepositPool.sol`)

## Summary

`LRTOracle.rsETHPrice` is a stored value updated only when `updateRSETHPrice()` is explicitly called. Because `LRTDepositPool.depositAsset` and `depositETH` mint rsETH using this stored price without first refreshing it, any depositor can deposit during a staleness window — after underlying assets (e.g., stETH) have appreciated but before the price is updated — and receive more rsETH than their fair share. The excess rsETH is funded by diluting the accumulated yield that rightfully belongs to existing holders.

## Finding Description

`LRTOracle.updateRSETHPrice()` is public and permissionless: [1](#0-0) 

`_updateRsETHPrice()` computes the new price as:

```
newRsETHPrice = (totalETHInProtocol - protocolFeeInETH) / rsethSupply
``` [2](#0-1) 

Critically, `previousTVL` is computed using the **stored** (stale) `rsETHPrice`, not the true current value: [3](#0-2) 

Deposits mint rsETH using the stale stored price: [4](#0-3) 

Neither `depositETH` nor `depositAsset` calls `updateRSETHPrice()` before computing `rsethAmountToMint`: [5](#0-4) [6](#0-5) 

**Exploit flow (concrete numbers):**

| State | totalETHInProtocol | rsethSupply | rsETHPrice |
|---|---|---|---|
| After stETH rebase, before update | 101 ETH | 100 rsETH | 1.00 (stale) |
| After attacker deposits 100 ETH at stale price | 201 ETH | 200 rsETH | 1.00 (stale) |
| After `updateRSETHPrice()` | 201 ETH | 200 rsETH | 1.005 |

- `previousTVL` = 200 × 1.00 = 200 ETH; `rewardAmount` = 201 − 200 = 1 ETH (the rebase)
- `newRsETHPrice` = 201 / 200 = **1.005 ETH**
- Attacker's 100 rsETH → 100.5 ETH (**+0.5 ETH profit**)
- Original holders' 100 rsETH → 100.5 ETH (should have been 101 ETH; **−0.5 ETH loss**)

Without the attacker, `newRsETHPrice` = 101 / 100 = 1.01 ETH, giving original holders the full 1 ETH rebase yield. The attacker captures exactly half of the accumulated yield by depositing at the stale price.

**Why existing guards fail:**

The `pricePercentageLimit` check only blocks `updateRSETHPrice()` calls where the price increase exceeds the configured threshold: [7](#0-6) 

For a normal daily stETH rebase (~0.01%), the price increase is far below any reasonable limit, so the attacker can freely call `updateRSETHPrice()` themselves after depositing. The `minRSETHAmountExpected` slippage parameter protects only the depositor, not existing holders.

## Impact Explanation

**High — Theft of unclaimed yield.** Existing rsETH holders accumulate yield as underlying LSTs appreciate between oracle updates. An attacker depositing at the stale price before the update captures a proportional share of that accumulated yield. The loss is direct and quantifiable: for every 1 ETH of accumulated yield and an equal-sized attacker deposit, existing holders lose 50% of their pending yield. The attack is repeatable on every oracle update cycle (at minimum daily for stETH rebases), making cumulative losses significant.

## Likelihood Explanation

**High.** stETH rebases every ~24 hours on a predictable schedule, creating a recurring staleness window. The attack requires no special permissions — `depositAsset`, `depositETH`, and `updateRSETHPrice()` are all callable by any unprivileged external user. An MEV bot can monitor the stETH rebase schedule on-chain and execute the deposit atomically in the same block as the rebase, before any oracle update. No oracle compromise, admin collusion, or victim mistake is required. [1](#0-0) 

## Recommendation

Call `_updateRsETHPrice()` (or `updateRSETHPrice()`) atomically at the start of `_beforeDeposit` before computing `rsethAmountToMint`. This ensures every deposit uses the freshest possible price, eliminating the stale-price deposit window. Alternatively, introduce a deposit fee or a time-weighted average price (TWAP) for the rsETH rate to make the attack economically unattractive.

    function _updateRsETHPrice() internal {
        address rsETHTokenAddress = lrtConfig.rsETH();
        uint256 rsethSupply = IRSETH(rsETHTokenAddress).totalSupply();

        if (rsethSupply == 0) {
            rsETHPrice = 1 ether;
            highestRsethPrice = 1 ether;
            return;
        }

        if (highestRsethPrice == 0) {
            highestRsethPrice = rsETHPrice;
        }

        uint256 previousPrice = rsETHPrice;

        // get total ETH in the protocol (normalized to 1e18)
        uint256 totalETHInProtocol = _getTotalEthInProtocol();

        // calculate previousTVL using rsethSupply multiplied by rsETHPrice
        uint256 previousTVL = rsethSupply.mulWad(rsETHPrice);

        IPausable lrtDepositPool = IPausable(lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL));
        IPausable withdrawalManager = IPausable(lrtConfig.getContract(LRTConstants.LRT_WITHDRAW_MANAGER));

        // determine if the protocol is active (not paused)
        bool protocolPaused = lrtDepositPool.paused() || withdrawalManager.paused() || paused;

        // only take fee if TVL increased and protocol is not paused
        uint256 protocolFeeInETH = 0;
        if (!protocolPaused && totalETHInProtocol > previousTVL) {
            uint256 rewardAmount = totalETHInProtocol - previousTVL;
            protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
        }

        // compute new rsETH price based on total ETH minus fee
        uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
