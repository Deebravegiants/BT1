### Title
Partial `ft_on_transfer` Return Leaves Tokens Permanently Stuck in Bridge Contract — (`near/omni-bridge/src/lib.rs`)

---

### Summary

`is_refund_required` in the NEAR bridge only triggers a corrective action when the resolved `ft_transfer_call` amount is exactly `0` (full failure). When a recipient's `ft_on_transfer` returns a **partial** refund, the bridge marks the transfer as successful, but the partially returned tokens accumulate in the bridge contract with no recovery mechanism, causing an irreversible fund lock and `locked_tokens` accounting divergence.

---

### Finding Description

`send_tokens` dispatches tokens to a NEAR recipient via `ft_transfer_call` whenever a non-empty `msg` is present: [1](#0-0) 

After the call resolves, the callback reads the promise result and calls `is_refund_required`: [2](#0-1) 

In NEAR's `ft_transfer_call` protocol, the token contract's `ft_resolve_transfer` returns the **amount actually used** (i.e., `total − refund`). The bridge's refund logic therefore handles exactly two cases:

| Scenario | Resolved amount | `is_refund_required` | Outcome |
|---|---|---|---|
| Recipient used all tokens | `= total` | `false` | ✓ success |
| Recipient used 0 tokens (panic / full return) | `= 0` | `true` | ✓ refund/burn |
| **Recipient used partial tokens** | `0 < N < total` | **`false`** | **✗ stuck tokens** |

In the partial case, `is_refund_required` returns `false`, so `fin_transfer_send_tokens_callback` proceeds to the success branch — it sends the fee to the fee recipient and emits `FinTransferEvent` — without accounting for the `(total − N)` tokens that the token contract has already refunded back to the bridge: [3](#0-2) 

The same gap exists in `resolve_fast_transfer` and `resolve_utxo_fin_transfer`: [4](#0-3) [5](#0-4) 

---

### Impact Explanation

**For non-deployed (locked) tokens** (EVM → NEAR path): `unlock_tokens_if_needed` decrements `locked_tokens` by the full transfer amount before `send_tokens` is called. If the recipient returns `K` tokens, those `K` tokens land back in the bridge's fungible-token balance but `locked_tokens` is not re-incremented. The bridge now holds excess tokens that are permanently unclaimable — there is no admin `rescue_tokens` function in the contract. The `locked_tokens` counter is understated by `K`, creating a backing-guarantee divergence.

**For deployed (bridge) tokens**: The bridge mints tokens to the recipient. Partially returned tokens accumulate as unbacked supply held by the bridge contract with no burn path triggered.

This satisfies: **Critical — Irreversible fund lock / permanently unclaimable protocol value in bridge flows.**

---

### Likelihood Explanation

The attack path is fully unprivileged:

1. Attacker deploys a NEAR contract whose `ft_on_transfer` returns a partial amount (e.g., `amount / 2`).
2. Attacker initiates a transfer on EVM specifying that NEAR contract as `recipient` and a non-empty `message`.
3. The relayer finalizes the transfer on NEAR via `fin_transfer` → `process_fin_transfer_to_near` → `send_tokens` → `ft_transfer_call`.
4. The malicious `ft_on_transfer` returns `amount / 2`; the token contract refunds `amount / 2` to the bridge.
5. `fin_transfer_send_tokens_callback` sees resolved amount `= amount / 2 ≠ 0`, treats the transfer as successful, and emits `FinTransferEvent`.
6. `amount / 2` tokens are permanently stuck in the bridge.

The attacker sacrifices their source-chain tokens (griefing), but the same outcome can occur accidentally with any legitimate recipient contract that legitimately returns partial tokens (e.g., a DEX that only partially fills an order). Likelihood is **medium-low** for deliberate exploitation, **medium** for accidental occurrence with composable recipient contracts.

---

### Recommendation

After `ft_transfer_call` resolves, compare the resolved amount against the sent amount. If `resolved_amount < sent_amount`, the difference `(sent_amount − resolved_amount)` has been returned to the bridge and must be re-locked (for non-deployed tokens) or burned (for deployed tokens). Concretely, extend `is_refund_required` (or replace it) with a function that returns the actual unused amount, and handle the partial case in all three callbacks (`fin_transfer_send_tokens_callback`, `resolve_fast_transfer`, `resolve_utxo_fin_transfer`).

---

### Proof of Concept

```
1. Deploy malicious NEAR contract `evil.near`:
   fn ft_on_transfer(sender, amount, msg) -> U128 {
       // Return half — keep half, refund half
       U128(amount.0 / 2)
   }

2. On EVM, call OmniBridge.initTransfer(
       token = USDC,
       amount = 1000,
       recipient = "evil.near",
       message = "trigger"   // non-empty → ft_transfer_call path
   )

3. Relayer calls fin_transfer on NEAR bridge.
   → send_tokens calls ft_transfer_call(evil.near, 1000, "trigger")
   → evil.near.ft_on_transfer returns U128(500)
   → token contract refunds 500 to bridge
   → ft_transfer_call promise resolves with used_amount = 500

4. fin_transfer_send_tokens_callback:
   is_refund_required(true) → 500 == 0 → false
   → success branch: fee sent, FinTransferEvent emitted
   → 500 tokens sit in bridge, locked_tokens understated by 500
   → no recovery path exists
``` [6](#0-5) [7](#0-6)

    fn is_refund_required(is_ft_transfer_call: bool) -> bool {
        if is_ft_transfer_call {
            match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
                Ok(value) => {
                    if let Ok(amount) = near_sdk::serde_json::from_slice::<U128>(&value) {
                        // Normal case: refund if the used token amount is zero
                        // The amount can be zero if the `ft_on_transfer` in the receiver contract returns an amount instead of `0`, or if it panics.
                        amount.0 == 0
                    } else {
                        // Unexpected case: don't refund
                        false
                    }
                }
                // Unexpected case: don't refund
                Err(_) => false,
            }
        } else {
            // Not ft_transfer_call: don't refund
            false
        }
    }
