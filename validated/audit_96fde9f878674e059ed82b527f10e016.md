### Title
`deployToken` / `deploy_token` Signature Lacks Chain-ID Binding, Enabling Cross-Chain Replay — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/omni_bridge.cairo`)

### Summary
The NEAR MPC-signed `deployToken` message is hashed over only `(PayloadType, token, name, symbol, decimals)` with no destination chain identifier. The same signature accepted on Ethereum can be submitted verbatim to any other OmniBridge deployment (Arbitrum, BSC, Starknet, etc.) that shares the same `nearBridgeDerivedAddress`, deploying a bridge token on a chain NEAR never authorized.

### Finding Description
`finTransfer` correctly embeds `omniBridgeChainId` twice in its Borsh-encoded payload before hashing, binding the signature to one specific destination chain. `deployToken` performs no equivalent binding.

**EVM — `OmniBridge.sol` `deployToken` (lines 142–153):** [1](#0-0) 

The Borsh blob is `PayloadType.Metadata || token || name || symbol || decimals`. No `omniBridgeChainId`, no `block.chainid`, no `address(this)`.

**EVM — `OmniBridge.sol` `finTransfer` for contrast (lines 289–311):** [2](#0-1) 

`omniBridgeChainId` appears at positions 5 and 8 of the blob, binding the signature to one chain.

**Starknet — `MetadataPayloadTrait::to_borsh` (bridge_types.cairo lines 36–44):** [3](#0-2) 

Identical omission: no `chain_id` argument, unlike `TransferMessagePayloadTrait::to_borsh` which receives and encodes `chain_id` at lines 67 and 70. [4](#0-3) 

**Starknet — `deploy_token` calls `to_borsh()` with no chain argument (omni_bridge.cairo line 205):** [5](#0-4) 

**Starknet — `fin_transfer` passes `omni_bridge_chain_id` (omni_bridge.cairo lines 252–254):** [6](#0-5) 

The asymmetry is present in both EVM and Starknet implementations.

### Impact Explanation
An attacker who observes a valid `deployToken` calldata on chain A can submit the identical `(signatureData, metadata)` tuple to `deployToken` on chain B. Because the hash is chain-agnostic, `ECDSA.recover(hashed, signatureData) == nearBridgeDerivedAddress` passes on chain B. The token is registered in `nearToEthToken` / `near_to_starknet_token` on chain B without NEAR ever authorizing it there.

Concrete harms:

1. **Permanent blocking of legitimate deployment.** Once the replayed token occupies the `nearToEthToken[metadata.token]` slot, the `require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST")` guard prevents NEAR from ever deploying the canonical version on that chain. There is no admin function to evict a token registered via `deployToken` (only `removeCustomToken` exists, which only covers admin-added tokens). This permanently freezes the redemption path for users bridging that token to the affected chain. [7](#0-6) 

2. **Unauthorized token mapping.** The replayed deployment creates a `nearToEthToken` entry on chain B that NEAR did not authorize, potentially causing NEAR's relayer to route `finTransfer` messages to an unintended token contract.

3. **Decimal mismatch.** If NEAR intended to deploy the token on chain B with different normalized decimals (e.g., after a token upgrade), the replayed deployment locks in the chain-A decimals permanently, breaking backing guarantees for any subsequent `finTransfer`.

### Likelihood Explanation
The attack requires no privilege. The signature is permanently visible in the calldata of the original `deployToken` transaction on chain A. Any unprivileged actor can copy it and call `deployToken` on any other OmniBridge deployment. The same `nearBridgeDerivedAddress` is shared across all EVM chains (it is derived from the NEAR MPC key, not per-chain). The attack is cheap (one transaction) and irreversible.

### Recommendation
Embed the destination chain identifier in the Borsh-encoded metadata payload before hashing, mirroring the pattern already used in `finTransfer`:

**EVM:**
```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),          // chain binding
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

**Starknet (`MetadataPayloadTrait::to_borsh`):**
```cairo
fn to_borsh(self: @MetadataPayload, chain_id: u8) -> ByteArray {
    ...
    borsh_bytes.append_byte(chain_id);   // chain binding
    ...
}
```
and update the `deploy_token` call site to pass `self.omni_bridge_chain_id.read()`, matching the `fin_transfer` pattern exactly.

The NEAR side must correspondingly include the destination chain ID when constructing the message it signs for `deployToken`.

### Proof of Concept
```
Chain A (Ethereum, omniBridgeChainId = 1):
  tx = OmniBridgeA.deployToken(sig, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})
  // NEAR MPC signed keccak256(0x01 || borsh("usdc.near") || borsh("USD Coin") || borsh("USDC") || 0x06)

Chain B (Arbitrum, omniBridgeChainId = 2, same nearBridgeDerivedAddress):
  OmniBridgeB.deployToken(sig, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})
  // Hash is identical — sig verifies — USDC deployed on Arbitrum without NEAR authorization
  // nearToEthToken["usdc.near"] now occupied on Arbitrum
  // NEAR's future attempt to deploy USDC on Arbitrum reverts with ERR_TOKEN_EXIST
  // Users bridging USDC from NEAR to Arbitrum have no valid redemption path
```
