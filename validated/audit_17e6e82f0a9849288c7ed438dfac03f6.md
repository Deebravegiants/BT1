### Title
Unauthenticated P2P Peer Can Inject `state_diff_length=0` Header to Commit Empty `ThinStateDiff`, Corrupting Node State and RPC Results — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

---

### Summary

`HeaderStreamBuilder::parse_data_for_block` accepts any `SignedBlockHeader` from a p2p peer without cryptographic signature verification and without cross-checking `state_diff_length` against `state_diff_commitment`. A malicious peer can send a header with `state_diff_length=0`. When `StateDiffStreamBuilder::parse_data_for_block` subsequently runs for that block, it reads `target_state_diff_len=0` from storage, the accumulation `while` loop is skipped entirely, and `ThinStateDiff::default()` (an empty diff) is committed via `write_to_storage` → `append_state_diff`. This permanently corrupts the syncing node's state for that block, causing all subsequent `starknet_getStateUpdate` / `starknet_getStorageAt` RPC calls for that block to return authoritative-looking wrong values.

---

### Finding Description

**Step 1 — Header injection with no signature verification.**

`HeaderStreamBuilder::parse_data_for_block` performs only two checks on the received `SignedBlockHeader`:

1. Block number matches the expected value.
2. `signatures.len() == ALLOWED_SIGNATURES_LENGTH` (length only, no cryptographic check). [1](#0-0) 

The function `verify_block_signature` exists in the codebase and takes `sequencer_pub_key`, `signature`, `state_diff_commitment`, and `block_hash` as inputs, but it is never called in this path. [2](#0-1) 

There is also an explicit TODO acknowledging that `state_diff_length` is not validated against `concatenated_counts` (which encodes the commitment): [3](#0-2) 

A malicious peer can therefore send a `SignedBlockHeader` with `state_diff_length=Some(0)` and `state_diff_commitment=Some(<non-zero hash>)` and it will be stored verbatim.

**Step 2 — `parse_data_for_block` skips the accumulation loop when `target_state_diff_len = 0`.**

`StateDiffStreamBuilder::parse_data_for_block` reads `target_state_diff_len` directly from the stored header's `state_diff_length` field: [4](#0-3) 

The accumulation loop guard is:

```rust
while current_state_diff_len < target_state_diff_len {
``` [5](#0-4) 

When `target_state_diff_len = 0`, the condition `0 < 0` is immediately false. The loop body never executes — no messages are consumed from the peer stream. The post-loop exact-match check (`current_state_diff_len != target_state_diff_len` → `0 != 0`) also passes silently. [6](#0-5) 

The function returns `Ok(Some((ThinStateDiff::default(), block_number)))`.

**Step 3 — Empty diff is committed to storage.**

`write_to_storage` calls `append_state_diff(self.1, self.0)` with the empty `ThinStateDiff` and advances the state marker: [7](#0-6) 

There is no post-commit verification of the committed diff against the `state_diff_commitment` stored in the header. The state marker is permanently advanced past the corrupted block.

---

### Impact Explanation

Any RPC call that reads state for the affected block — `starknet_getStateUpdate`, `starknet_getStorageAt`, contract balance queries, fee estimation — will return an empty or zero result instead of the real state. The node's storage is permanently corrupted for that block number; the state marker advances, so the node will not re-sync the block. This fits the allowed High impact: **RPC execution returns an authoritative-looking wrong value**.

---

### Likelihood Explanation

The p2p sync client connects to unauthenticated peers by design. Any node that can establish a p2p connection (no credentials required) can serve crafted headers. The attack requires only sending a single malformed `SignedBlockHeader` message with `state_diff_length=0`. No cryptographic material needs to be forged because signature verification is absent.

---

### Recommendation

1. **Verify the block signature cryptographically** in `HeaderStreamBuilder::parse_data_for_block` using the existing `verify_block_signature` function before storing any header. The sequencer public key must be obtained from a trusted source (e.g., L1 contract or configuration).
2. **Verify the accumulated state diff against `state_diff_commitment`** at the end of `StateDiffStreamBuilder::parse_data_for_block` before returning `Ok(Some(...))`. Compute `calculate_state_diff_hash(&result)` and compare it to the `state_diff_commitment` stored in the header.
3. **Reject `state_diff_length=0` with a non-zero `state_diff_commitment`** as an additional sanity guard in the header parsing step.

---

### Proof of Concept

```rust
// In a unit test for StateDiffStreamBuilder:
// 1. Store a header with state_diff_length=Some(0) and
//    state_diff_commitment=Some(<non-zero StateDiffCommitment>).
// 2. Drive parse_data_for_block with a Fin (DataOrFin(None)) message.
// 3. Assert the returned ThinStateDiff is ThinStateDiff::default() (empty).
// 4. Assert the stored header's state_diff_commitment is non-zero.
// => The committed state diff is empty while the header claims a non-empty commitment.
```

The existing test suite already uses `state_diff_length: Some(0)` in headers without triggering any rejection, confirming the path is live: [8](#0-7)
