### Title
Unrefunded Attached Deposit in `submit_participant_info` Re-submission Path Permanently Traps Caller NEAR — (File: crates/contract/src/lib.rs)

---

### Summary

`submit_participant_info` is marked `#[payable]` and therefore accepts any attached NEAR deposit. Its deposit-check-and-refund block is guarded by the boolean `attestation_storage_must_be_paid_by_caller`. When an **existing participant** re-submits their attestation (e.g., after a TEE upgrade), that flag is `false` and the entire deposit block is skipped — meaning any NEAR attached to the call is silently absorbed by the contract with no refund and no accounting entry. There is no contract-level withdrawal mechanism, so the funds are permanently frozen.

---

### Finding Description

In `crates/contract/src/lib.rs`, `submit_participant_info` is declared `#[payable]`: [1](#0-0) 

After the attestation is stored, the code computes a guard flag: [2](#0-1) 

The deposit check and refund are only executed inside the `if attestation_storage_must_be_paid_by_caller` block: [3](#0-2) 

The flag is `false` — and the entire block is skipped — when **both** of the following hold:

1. `is_new_attestation` is `false`: the call is a re-submission that updates an existing entry (`ParticipantInsertion::UpdatedExistingParticipant`).
2. `caller_is_not_participant` is `false`: the caller is already a recognized participant (`self.voter_account()` succeeds).

In that path, `env::attached_deposit()` is never read, never validated, and never refunded. Any NEAR sent with the call is silently credited to the contract's balance and cannot be recovered.

---

### Impact Explanation

The NEAR MPC contract has no withdrawal or sweep function for its own balance. Funds deposited in the re-submission path are permanently frozen inside the chain-signature contract. This matches the allowed impact: **permanent freezing of funds controlled by the chain-signature contract**.

---

### Likelihood Explanation

Existing participants must periodically re-submit attestations — for example, after a TEE image upgrade or when their quote expires and `verify_tee` kicks them out. This is a routine, recurring operation. A frontend, automation script, or operator copy-pasting a call that previously required a deposit (e.g., the initial `submit_participant_info` call, or a `sign` call) can trivially attach a non-zero deposit. Because the function is `#[payable]`, wallets and tooling will not warn the caller that a deposit is unexpected. The scenario is directly analogous to the Dexter report's "Bob calls `approve` but incorrectly sends 10 Tezos."

---

### Recommendation

**Short term:** At the top of `submit_participant_info`, or at the start of the `else` branch (when `attestation_storage_must_be_paid_by_caller` is `false`), assert that `env::attached_deposit() == NearToken::from_yoctonear(0)` and panic if it is not, or unconditionally refund any deposit that exceeds the actual storage cost (including zero cost for a pure update).

**Long term:** Audit every `#[payable]` function in the contract to confirm that every reachable code path either (a) enforces a minimum deposit and refunds the excess, or (b) explicitly rejects any non-zero deposit. The `require_deposit` helper already implements the correct pattern for `sign`, `request_app_private_key`, and `verify_foreign_transaction`; the same discipline should be applied uniformly.

---

### Proof of Concept

1. Alice is an existing MPC participant. Her attestation is already stored (`ParticipantInsertion::UpdatedExistingParticipant` will be returned on re-submission).
2. Alice's TEE image is upgraded; she must call `submit_participant_info` again with her new quote.
3. Alice (or her automation script) mistakenly attaches `1 NEAR` to the call — a common mistake when reusing a transaction template that previously required a deposit.
4. The call succeeds. `attestation_storage_must_be_paid_by_caller` evaluates to `false` (existing participant, not a new entry). The `if` block at line 826 is skipped entirely.
5. `env::attached_deposit()` is never read. The `1 NEAR` is credited to the MPC contract's balance.
6. No refund promise is issued. Alice's `1 NEAR` is permanently frozen in the contract. [4](#0-3)
