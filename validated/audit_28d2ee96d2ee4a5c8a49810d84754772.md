Audit Report

## Title
Unbounded Staleness in `CrossChainRateReceiver.getRate()` Enables Over-Issuance of rsETH/wrsETH on L2 Pools — (File: `contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary
`CrossChainRateReceiver` stores a `lastUpdated` timestamp on every rate push from L1 but `getRate()` returns the stored `rate` unconditionally with no staleness check. Both L2 deposit pools (`RSETHPoolV3`, `RSETHPoolNoWrapper`) call `getRate()` on every deposit to compute how many rsETH/wrsETH tokens to issue. When the L2 oracle is stale and lower than the true L1 rate, any depositor can obtain more rsETH/wrsETH than the current protocol rate entitles them to, capturing yield that belongs to existing holders.

## Finding Description
`CrossChainRateReceiver.lzReceive()` stores both `rate` and `lastUpdated` on every successful update:

```solidity
rate = _rate;
lastUpdated = block.timestamp;
``` [1](#0-0) 

`getRate()` ignores `lastUpdated` entirely and returns the stored value unconditionally:

```solidity
function getRate() external view returns (uint256) {
    return rate;
}
``` [2](#0-1) 

`lastUpdated` is written but never read anywhere in the contract — there is no maximum-age enforcement. Both pool variants consume this oracle on every deposit via `viewSwapRsETHAmountAndFee`:

```solidity
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [3](#0-2) [4](#0-3) 

`RSETHPoolV3.deposit()` mints wrsETH directly using this computed amount: [5](#0-4) 

`RSETHPoolNoWrapper.deposit()` transfers pre-existing rsETH from the pool using the same computed amount: [6](#0-5) 

When `rsETHToETHrate` is stale and lower than the true current rate, the division `amountAfterFee * 1e18 / rsETHToETHrate` yields a larger `rsETHAmount`, issuing more tokens than the deposited ETH justifies at the true rate. The `dailyMintLimit` guard in `RSETHPoolV3` caps total daily issuance but does not prevent the stale-rate over-issuance within that cap; `RSETHPoolNoWrapper` has no such cap at all. [7](#0-6) 

The rate update path (`MultiChainRateProvider.updateRate()`) is triggered off-chain and is not automated on-chain, making oracle staleness an operational reality rather than a theoretical edge case. [8](#0-7) 

## Impact Explanation
**High — Theft of unclaimed yield.**

rsETH appreciates continuously as EigenLayer staking rewards accrue. The appreciation between the last oracle update and the current block is "unclaimed yield" belonging to existing rsETH/wrsETH holders. A depositor who mints at a stale lower rate receives more shares than the current backing justifies; those excess shares are redeemable at the true (higher) rate once the oracle updates, extracting the accrued yield from the pool's reserves. The loss is direct, quantifiable, and proportional to deposit size and the magnitude of the rate drift. This matches the allowed impact class "High — Theft of unclaimed yield."

## Likelihood Explanation
**Medium.** The off-chain bot calling `updateRate()` is the sole mechanism for keeping the L2 oracle fresh. Any lapse — downtime, gas exhaustion, network congestion, LayerZero message delay — leaves the oracle stale. rsETH appreciates on every block that EigenLayer rewards accrue, so the stale rate will almost always be lower than the true rate after any meaningful delay. No special privileges are required: a standard `deposit()` call from any EOA is sufficient. The discrepancy is observable by reading both `LRTOracle.rsETHPrice` on L1 and `CrossChainRateReceiver.rate` on L2, making targeted exploitation straightforward for a monitoring bot.

## Recommendation
**Short term:** Add a staleness guard in `CrossChainRateReceiver.getRate()`:

```solidity
uint256 public constant MAX_RATE_AGE = 24 hours;

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= MAX_RATE_AGE, "Rate is stale");
    return rate;
}
```

This causes deposits to revert when the oracle has not been refreshed within the window, preventing exploitation of stale rates.

**Long term:** Automate `updateRate()` calls on L1 via a keeper or Chainlink Automation job on a fixed cadence (e.g., every 4–8 hours), and alert on any gap exceeding the staleness threshold.

## Proof of Concept
1. At time T, L1 `rsETHPrice` = 1.05 ETH. `MultiChainRateProvider.updateRate()` is called; L2 `CrossChainRateReceiver.rate` = 1.05e18, `lastUpdated` = T.
2. EigenLayer rewards accrue. At T+12h, L1 `rsETHPrice` = 1.10 ETH. The off-chain bot is down; `updateRate()` is not called. L2 oracle still returns 1.05e18.
3. Attacker calls `RSETHPoolV3.deposit{value: 1.05 ether}("")`.
4. Pool computes: `rsETHAmount = 1.05e18 * 1e18 / 1.05e18 = 1.0e18`. At the true rate of 1.10 ETH/rsETH, 1.05 ETH should only buy `1.05/1.10 ≈ 0.9545 wrsETH`. Attacker receives ~0.0455 wrsETH in excess.
5. Bot recovers; oracle updates to 1.10e18. Attacker's 1.0 wrsETH is redeemable for 1.10 ETH — a risk-free profit of ~0.05 ETH extracted from existing holders' accrued yield.
6. The same attack applies to `RSETHPoolNoWrapper.deposit()` with no daily mint cap, scaling linearly with deposit size and repeatable across every oracle-staleness window.
