### Title
Unsound `valid_paillier_t` ZK Proof Due to Insufficient Small-Primes Bound Allows Attacker to Submit Paillier Key with Known Factorization - (`include-internal/cbmpc/internal/zk/small_primes.h`, `src/cbmpc/zk/zk_paillier.cpp`)

---

### Summary

`valid_paillier_t::verify` calls `check_integer_with_small_primes(N, param::alpha)` where `param::alpha = 8192`. This check only rejects N if it has a prime factor ≤ 8192. A Byzantine peer can construct N = p·q with p = 8209 (a prime just above the bound), compute phi(N) = (p−1)(q−1) from the known factorization, derive valid sigma values, and cause `verify()` to return `SUCCESS` — accepting a Paillier key whose modulus is trivially factorable.

---

### Finding Description

**`param::alpha` is 8192, not a security-adequate bound:** [1](#0-0) 

`log_alpha = 13`, so `alpha = 1 << 13 = 8192`. This is the bound passed to `check_integer_with_small_primes`.

**The check breaks early, leaving primes 8193–104729 unchecked:** [2](#0-1) 

The loop iterates over `small_primes[10000]` (the first 10,000 primes, up to 104,729) but `break`s as soon as `small_prime > alpha`. Any prime p with 8192 < p ≤ 104,729 is never tested.

**`verify()` calls the check with this insufficient bound:** [3](#0-2) 

**The core check is `sigma[i]^N mod N == rho[i]`:** [4](#0-3) 

This is a proof of knowledge of phi(N): only someone who knows phi(N) can compute N-th roots mod N. The security assumption is that N is hard to factor. But if N = p·q with p = 8209, the attacker knows the factorization trivially.

**The honest prover computes sigma using `N^{-1} mod phi(N)`:** [5](#0-4) 

The attacker replicates this exactly: they know phi(N) = (p−1)(q−1), compute N_inv = N^{-1} mod phi(N), and compute sigma[i] = rho[i]^{N_inv} mod N. By Euler's theorem, sigma[i]^N ≡ rho[i] (mod N). The rho[i] values are deterministic from N, session_id, and aux — all values the attacker controls or observes.

**Only `sigma` is serialized (attacker-controlled):** [6](#0-5) 

The `paillier_valid_key` and `paillier_no_small_factors` flags are not serialized; they are set by `verify()` itself. The attacker's only input is `sigma[]`.

**The coprime check does not block the attack:** [7](#0-6) 

The probability that any rho[i] is divisible by p = 8209 is ≈ 1/8209 per iteration, negligible over `param::t` iterations.

### Impact Explanation

`valid_paillier_t::verify` sets `paillier_valid_key = zk_flag::verified` for a modulus N = p·q where p = 8209. This modulus is trivially factorable by trial division. The Paillier encryption scheme over such an N provides no semantic security: any ciphertext encrypted under this key can be decrypted by anyone who factors N (including the attacker). In DKG and 2-party ECDSA contexts (confirmed usage in `src/cbmpc/protocol/ecdsa_2p.cpp`), honest parties may encrypt sensitive values (e.g., secret key shares, randomness) under the accepted Paillier key, exposing them to the attacker. Additionally, all downstream ZK proofs that gate on `paillier_valid_key == verified` inherit the broken security assumption.

---

### Likelihood Explanation

The attack requires only:
1. Choosing p = 8209 and a large prime q to form a 2048-bit N
2. Computing phi(N) and N_inv (trivial arithmetic)
3. Replaying the deterministic rho derivation (public inputs only)
4. Submitting the crafted sigma array over the protocol peer boundary

No threshold collusion, no host compromise, and no RNG weakness is required. The attacker is a single Byzantine peer.

---

### Recommendation

Increase `log_alpha` (and thus `alpha`) so that `check_integer_with_small_primes` covers all primes up to the security-required bound. The standard recommendation for Paillier key validation (e.g., per CL/GG protocols) is alpha ≥ 2^17 (131072) or higher. Alternatively, perform a Miller-Rabin primality test on N's small-factor candidates, or use a dedicated safe-prime generation check. At minimum, `alpha` must exceed the largest prime in `small_primes[]` that the protocol intends to exclude, or the loop must not break early.

### Proof of Concept

```
1. Choose p = 8209 (prime; 8209 > 8192 = alpha, so check_integer_with_small_primes passes)
2. Choose q = large safe prime such that p ∤ (q−1) and N = p·q is 2048 bits
3. Compute phi_N = (p−1)·(q−1) = 8208·(q−1)
4. Compute N_inv = N^{−1} mod phi_N  [exists since gcd(N, phi_N) = 1]
5. Derive rho[i] = drbg.gen_bn(N) using k = hash(N, session_id, aux)  [same DRBG as verifier]
6. Compute sigma[i] = rho[i]^{N_inv} mod N  for i = 0..param::t−1
7. Serialize sigma[] into valid_paillier_t and send to honest verifier
8. Verifier calls valid_paillier_t::verify():
   - check_integer_with_small_primes(N, 8192): 8209 > 8192, loop breaks → SUCCESS
   - sigma[i]^N mod N == rho[i]: true by Euler's theorem → SUCCESS
   - coprime(rho_prod, N): passes with probability ≥ 1 − t/8209 ≈ 1
   - paillier_valid_key = zk_flag::verified  ← accepted
9. N = p·q with p = 8209 is now accepted as a valid Paillier key
10. Factor N by trial division: N mod 8209 == 0 → recover p, q, phi_N, decrypt any ciphertext
```

static error_t check_integer_with_small_primes(const bn_t& prime, int alpha) {
  for (int i = 0; i < small_primes_count; i++) {
    int small_prime = small_primes[i];
    if (small_prime > alpha) break;
    if (mod_t::mod(prime, small_prime) == 0) return coinbase::error(E_CRYPTO);
