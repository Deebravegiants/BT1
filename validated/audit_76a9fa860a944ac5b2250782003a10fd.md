### Title
Propose-Update Deposits Permanently Locked on `do_update` Entry Sweep - (File: `crates/contract/src/update.rs`)

### Summary
When `vote_update` reaches threshold and triggers `do_update`, **all** pending update proposals are cleared from storage, but the NEAR deposits paid by every non-winning proposer are never refunded. Because `UpdateEntry` does not record the proposer's account ID, there is no mechanism to return those funds — they remain locked in the contract indefinitely.

### Finding Description
`propose_update` requires each proposer to attach a deposit sized to cover the storage cost of their `UpdateEntry`: [1](#0-0) 

The required amount is computed from `bytes_used` (code length + overhead for 128 participant-vote slots) and kept in the contract: [2](#0-1) 

`UpdateEntry` stores only the update payload and `bytes_used`; the proposer's account ID is **not recorded**: [3](#0-2) 

When `vote_update` crosses the threshold it calls `do_update`, which removes the winning entry and then **bulk-clears every other pending entry and all votes** with no refund logic: [4](#0-3) 

Because the proposer identity was never stored, there is no way to reconstruct who paid what. The only other deposit-related path in the governance flow is `remove_update_vote`, which removes a vote record but does not touch the proposal entry or its deposit: [5](#0-4) 

### Impact Explanation
Every participant who proposed a competing update loses their full storage deposit permanently when any update is executed. For a 1 MB contract binary the required deposit is on the order of 10 NEAR (≈ `10^6 bytes × 10 yoctoNEAR/byte`). With `n` participants each proposing a different binary, `(n-1) × deposit` is locked in the contract forever with no recovery path. This breaks the production accounting invariant that storage deposits are returned when the storage they cover is freed.

### Likelihood Explanation
The trigger is ordinary protocol operation: any quorum of participants voting on a contract upgrade. In practice, multiple participants may each propose their own preferred binary before consensus forms. The loss is deterministic — it occurs on every successful `vote_update` execution that clears competing proposals — and requires no adversarial action beyond normal governance participation.

### Recommendation
1. Add a `proposer: AccountId` field to `UpdateEntry` so the deposit owner is always known.
2. In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and schedule `Promise::new(entry.proposer).transfer(entry.deposit)` for each one.
3. Alternatively, store `(proposer, deposit)` in a separate side-map keyed by `UpdateId` and drain it in `do_update`.

### Proof of Concept
1. Participant A calls `propose_update` with a 1 MB contract binary, attaching ~10 NEAR deposit. `UpdateEntry { update: Contract([...]), bytes_used: ~1_000_000 }` is stored; A's 10 NEAR is retained by the contract.
2. Participant B calls `propose_update` with a different binary, attaching ~10 NEAR.
3. Participants vote for B's proposal until threshold is reached via `vote_update`.
4. `do_update` executes: it removes B's entry, then calls `self.entries.clear()` — silently dropping A's entry — and `self.vote_by_participant.clear()`. No refund is issued to A.
5. A's ~10 NEAR remains in the contract balance with no recovery path, since `UpdateEntry` never recorded A's account ID. [6](#0-5) [7](#0-6)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }
