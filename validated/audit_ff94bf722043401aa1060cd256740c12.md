### Title
Unconstrained `BlockConfirmations` Allows MPC Attestation of Unfinalized Bitcoin Transactions - (File: `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

### Summary
The `verify_foreign_transaction` contract method accepts a `BitcoinRpcRequest` with `confirmations: 0` (or any arbitrarily low value) without enforcing any protocol-level minimum. An unprivileged caller can obtain a valid MPC-signed attestation for a Bitcoin transaction that has not achieved meaningful finality. Any bridge contract that relies on these attestations without independently re-validating the caller-supplied confirmation threshold is exposed to double-spend or chain-reorganization attacks.

### Finding Description
`BlockConfirmations` is a plain `u64` newtype with no minimum-value invariant:

```rust
pub struct BlockConfirmations(pub u64);
```

The ABI schema explicitly permits zero: `"minimum": 0.0`.

`verify_foreign_transaction` performs no validation on the `confirmations` field — it only checks gas attachment, deposit, domain purpose, and whether the chain is whitelisted:

```rust
pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
    self.check_request_preconditions(
        request.domain_id,
        DomainPurpose::ForeignTx,
        Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
        MINIMUM_SIGN_REQUEST_DEPOSIT,
    );
    let requested_chain = request.request.chain();
    let supported_chains = self.get_supported_foreign_chains();
    if !supported_chains.contains(&requested_chain) { ... }
    // ← no check that confirmations >= 1 or any safe minimum
    ...
}
```

On the node side, `BitcoinInspector::extract` enforces only that the actual confirmation count meets the caller-supplied threshold:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
```

With `block_confirmations_threshold = 0`, the condition `0 <= actual` is trivially true for every confirmed transaction. The node then calls `verify_block_is_canonical` and, if that passes, signs the payload. The signed `ForeignTxSignPayloadV1` embeds the original `ForeignChainRpcRequest` — including `confirmations: 0` — so the attestation is cryptographically valid and will pass `respond_verify_foreign_tx`'s signature check.

**Concrete attack path:**
1. Attacker submits `verify_foreign_transaction` with `BitcoinRpcRequest { tx_id: X, confirmations: 0, extractors: [BlockHash] }`.
2. MPC nodes query Bitcoin RPC; any confirmed transaction (even 1 confirmation) satisfies `0 <= 1`.
3. Nodes sign `SHA-256(borsh(ForeignTxSignPayload { request: {confirmations:0, ...}, values: [block_hash] }))`.
4. `respond_verify_foreign_tx` verifies the ECDSA signature against the root public key and resolves the yield — no minimum-confirmation check occurs here either.
5. The caller receives a valid MPC-signed attestation for a 1-confirmation Bitcoin transaction.
6. A bridge contract that does not re-inspect the `confirmations` field in the signed payload releases NEAR-side funds.
7. The Bitcoin transaction is reorganized; the attacker has double-spent.

### Impact Explanation
The MPC network's stated purpose is to "sign payloads only after verifying a specific foreign-chain transaction." Allowing `confirmations: 0` breaks the production safety invariant that attestations represent finalized foreign-chain state. A bridge contract that trusts the MPC attestation without re-validating the caller-supplied confirmation threshold will release funds for a transaction that may never finalize, enabling a direct double-spend loss. This matches the allowed **High** impact: foreign-chain verification bypass causing invalid bridge execution or double-spend conditions.

### Likelihood Explanation
The entry point is the public `verify_foreign_transaction` method, callable by any NEAR account with a 1 yoctoNEAR deposit. No privileged access is required. The attacker only needs to know a Bitcoin transaction ID and submit a request with `confirmations: 0`. The attack is straightforward and repeatable.

### Recommendation
Enforce a protocol-level minimum on `BlockConfirmations` at the contract boundary. Options:
1. Add a `MIN_BITCOIN_CONFIRMATIONS` constant (e.g., `1` or `6`) and reject requests below it in `verify_foreign_transaction`.
2. Change `BlockConfirmations` to a validated newtype that panics or errors on construction with `0`.
3. Expose a per-chain minimum-confirmation configuration in the on-chain foreign-chain policy, and validate the caller-supplied value against it before enqueuing the request.

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
