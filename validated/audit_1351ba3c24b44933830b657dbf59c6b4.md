### Title
Pool Protocol Signed Payloads Lack Chain-Binding (Genesis Challenge), Enabling Cross-Fork Payout Redirection - (File: chia/protocols/pool_protocol.py)

### Summary
The Chia pool protocol's signed request payloads (`PostFarmerPayload`, `PutFarmerPayload`, `PostPartialPayload`, `AuthenticationPayload`) contain no chain-binding identifier (i.e., no `genesis_challenge`). In a fork scenario, a signed `PUT /farmer` or `POST /farmer` request captured on one chain can be replayed verbatim on the other chain within the same authentication-token time window, allowing an attacker to overwrite a farmer's `payout_instructions` on the forked chain with a stale or chain-inappropriate address, diverting pool rewards.

### Finding Description

Every pool protocol payload is a `Streamable` struct whose hash is signed by the farmer's owner or authentication key. None of these structs include a network identifier:

`PostFarmerPayload` fields: `launcher_id`, `authentication_token`, `authentication_public_key`, `payout_instructions`, `suggested_difficulty`. [1](#0-0) 

`PutFarmerPayload` fields: `launcher_id`, `authentication_token`, `authentication_public_key`, `payout_instructions`, `suggested_difficulty`. [2](#0-1) 

`PostPartialPayload` fields: `launcher_id`, `authentication_token`, `proof_of_space`, `sp_hash`, `end_of_sub_slot`, `harvester_id`. [3](#0-2) 

`AuthenticationPayload` fields: `method_name`, `launcher_id`, `target_puzzle_hash`, `authentication_token`. [4](#0-3) 

None of these include `genesis_challenge` (Chia's chain identifier). Compare with how `genesis_challenge` is used everywhere else in the protocol to bind coins and puzzles to a specific chain — e.g., `pool_reward_prefix = bytes32(genesis_challenge[:16] + b"\x00" * 16)` in pool puzzle construction. [5](#0-4) 

The farmer signs these payloads and sends them to the pool:

```python
signature: G2Element = AugSchemeMPL.sign(owner_sk, post_farmer_payload.get_hash())
``` [6](#0-5) 

```python
signature: G2Element = AugSchemeMPL.sign(owner_sk, put_farmer_payload.get_hash())
``` [7](#0-6) 

The only time-based protection is `authentication_token`, which is `int(time.time() / 60) / timeout` and validated with a window of `±timeout` units. [8](#0-7) 

In a fork scenario, both chains run simultaneously, so the same `authentication_token` value is valid on both chains at the same time. The signed payload is byte-for-byte identical and will pass signature verification on either chain.

### Impact Explanation

**High — Pool payout redirection.**

Attack scenario:
1. A contentious Chia hard fork occurs (Chia has already had `HARD_FORK_2_0` and `HARD_FORK_3_0`). Both chains continue.
2. A farmer updates their `payout_instructions` on the forked chain to address B (their forked-chain address) via `PUT /farmer`.
3. An attacker captures the farmer's earlier `PUT /farmer` signed request from the mainnet chain (where `payout_instructions` = address A, e.g., an exchange address only valid on mainnet).
4. Within the same `authentication_token` time window, the attacker replays the mainnet-signed `PutFarmerRequest` to the pool on the forked chain.
5. The pool on the forked chain accepts it (valid signature, valid token, same `launcher_id`) and overwrites `payout_instructions` back to address A.
6. All subsequent pool rewards on the forked chain are sent to address A, which may be unspendable or exchange-controlled on the forked chain — permanently diverting the farmer's forked-chain pool rewards.

The `launcher_id` is the same on both chains (the singleton was created before the fork), so the pool cannot distinguish which chain the request originated from.

### Likelihood Explanation

Requires a live Chia fork where both chains continue operating. Chia has already executed multiple hard forks. The attack window per signed message is bounded by the `authentication_token` timeout (typically minutes), but since both chains run simultaneously, the window is always open for any freshly signed message. An attacker only needs to passively observe HTTPS pool traffic (or run a malicious pool on the forked chain) to collect valid signed payloads.

### Recommendation

Include `genesis_challenge` (the chain's unique identifier) in every signed pool protocol payload. For example, add a `genesis_challenge: bytes32` field to `PostFarmerPayload`, `PutFarmerPayload`, `PostPartialPayload`, and `AuthenticationPayload`. The pool server must verify that the `genesis_challenge` in the payload matches the chain it is operating on before accepting any signed request. This is analogous to EIP-712's `chainId` domain separator.

### Proof of Concept

1. Farmer on mainnet sends `PUT /farmer` with `payout_instructions = "xch1mainnet_exchange_addr..."`, signed by `owner_sk` over `PutFarmerPayload.get_hash()`.
2. Fork occurs. Farmer sends a new `PUT /farmer` on the forked chain with `payout_instructions = "xch1fork_addr..."`.
3. Attacker captures the mainnet `PutFarmerRequest` (same `authentication_token` still valid since both chains run at the same wall-clock time).
4. Attacker POSTs the captured `PutFarmerRequest` JSON to the forked-chain pool's `/farmer` endpoint.
5. Pool verifies: `AugSchemeMPL.verify(owner_pubkey, payload.get_hash(), signature)` — passes, because the payload bytes and signature are identical.
6. Pool updates `payout_instructions` to the mainnet exchange address on the forked chain.
7. All forked-chain pool rewards are diverted to the mainnet exchange address.

The missing guard is explicit: `PutFarmerPayload` has no `genesis_challenge` field. [9](#0-8)
