### Title
Cross-Epoch `delete_all(StateSyncNewChunks)` Permanently Prevents Epoch N+1 Sync Hash Computation — (`chain/chain/src/state_sync/utils.rs`)

---

### Summary

When epoch N's sync hash is found, `on_new_header` calls `store_update.delete_all(DBCol::StateSyncNewChunks)` which erases **every** entry in the column globally. If epoch N+1's first block header was already processed (writing its seed entry via `on_new_epoch`) before that `delete_all` fires, the seed entry is silently destroyed. Every subsequent header in epoch N+1 then fails to find its predecessor's entry in `StateSyncNewChunks`, `save_epoch_new_chunks` returns `false` for every block in the epoch, and the epoch N+1 sync hash is never written — permanently.

---

### Finding Description

**`on_new_epoch`** seeds the `StateSyncNewChunks` column for the first block of a new epoch: [1](#0-0) 

It writes `StateSyncNewChunks[first_block_of_epoch_N+1.hash()] = [0,...,0]`.

**`on_new_header`**, when it finds epoch N's sync hash, unconditionally wipes the entire column: [2](#0-1) 

`delete_all` has no epoch filter — it removes every key in `DBCol::StateSyncNewChunks`.

**`save_epoch_new_chunks`** requires the predecessor entry to exist: [3](#0-2) 

If `StateSyncNewChunks[header.prev_hash()]` is absent, it returns `false` immediately with a comment attributing the miss to epoch sync. No entry is written for the current block either, so the gap propagates forward to every subsequent block in the epoch.

**Exact call sequence that triggers the bug:**

1. Epoch N+1's first block header `B1` arrives → `update_sync_hashes` → `prev_header.epoch_id() != header.epoch_id()` → `on_new_epoch` → writes `StateSyncNewChunks[B1.hash()] = [0,...,0]`. Store update committed.
2. A later epoch-N header arrives → `on_new_header` → sync hash found → `delete_all(StateSyncNewChunks)` → **B1's entry is gone**.
3. Epoch N+1's second block `B2` arrives → `save_epoch_new_chunks` reads `StateSyncNewChunks[B1.hash()]` → `None` → returns `false`. Nothing written for B2.
4. Epoch N+1's third block `B3` → reads `StateSyncNewChunks[B2.hash()]` → `None` (never written) → returns `false`. Nothing written for B3.
5. This repeats for every block in epoch N+1. `update_sync_hashes` checks `get_current_epoch_sync_hash(epoch_N+1_id)` each time and finds `None`, so it keeps calling `on_new_header`, which keeps returning early at step 3. [4](#0-3) 

There is no recovery path. The column is keyed by block hash, not epoch, so there is no way to reconstruct the cumulative new-chunk counts without re-processing every header in the epoch from scratch — which the code never does.

---

### Impact Explanation

The epoch N+1 sync hash entry in `DBCol::StateSyncHashes` is never written. Any node that needs to state-sync to epoch N+1 (a node catching up, a node that fell behind, or a node bootstrapping via epoch sync) will call `find_sync_hash` → `get_sync_hash` → `get_current_epoch_sync_hash(epoch_N+1_id)` and receive `None` indefinitely. [5](#0-4) 

State sync for epoch N+1 is permanently stalled on the affected node. The node cannot advance past the epoch N sync point and will be stuck or forced into a data-reset cycle.

---

### Likelihood Explanation

The triggering condition — epoch N+1's first block header being processed before epoch N's sync hash is found — is a routine occurrence during header sync. Header sync downloads headers from the network tip backwards; it is normal for headers from the next epoch to arrive and be committed before the current epoch's sync hash computation completes. No adversarial peer behavior is required; the ordering arises from ordinary network propagation and the sequential header-processing loop.

---

### Recommendation

Replace the global `delete_all` with a targeted deletion that only removes entries belonging to epoch N (i.e., iterate and delete only keys whose block hashes are known to belong to epoch N), or restructure the column to include the epoch ID in the key so that `delete_all` can be scoped per epoch. Alternatively, re-seed the epoch N+1 entry immediately after the `delete_all` if an entry for the next epoch's first block already exists in the DB.

---

### Proof of Concept

Write a test-loop integration test:
1. Configure a two-shard network with a short epoch length.
2. Process all headers for epoch N+1's first block before allowing epoch N's sync hash to be finalized.
3. Assert that after epoch N's sync hash is written, `DBCol::StateSyncNewChunks` contains no entry for epoch N+1's first block.
4. Continue processing epoch N+1 headers and assert that `get_current_epoch_sync_hash(epoch_N+1_id)` remains `None` indefinitely — confirming permanent stall.

The invariant to assert: `delete_all(StateSyncNewChunks)` must not remove any entry whose block hash belongs to an epoch later than the epoch for which the sync hash was just found. [6](#0-5)

pub(crate) fn update_sync_hashes<T: ChainStoreAccess>(
    chain_store: &T,
    store_update: &mut StoreUpdate,
    header: &BlockHeader,
) -> Result<(), Error> {
    let sync_hash = chain_store.get_current_epoch_sync_hash(header.epoch_id());
    if sync_hash.is_some() || header.height() == chain_store.get_genesis_height() {
        return Ok(());
    }

    let prev_header = match chain_store.get_block_header(header.prev_hash()) {
        Ok(h) => h,
        // During epoch sync, we save headers whose prev headers might not exist, so we just do nothing in this case.
        // This means that we might not be able to state sync for this epoch, but for now this is not a problem.
        Err(Error::DBNotFoundErr(_)) => return Ok(()),
        Err(e) => return Err(e),
    };

    if prev_header.height() == chain_store.get_genesis_height() {
        on_new_epoch(store_update, header);
        return Ok(());
    }
    if prev_header.epoch_id() != header.epoch_id() {
        // Here we remove any sync hashes stored for old epochs after saving [0,...,0] in the StateSyncNewChunks
        // column for this block. This means we will no longer remember sync hashes for these old epochs, which
        // should be fine as we only care to state sync to (and provide state parts for) the latest state
        on_new_epoch(store_update, header);
        return remove_old_epochs(&chain_store.store(), store_update, header, &prev_header);
    }

    on_new_header(chain_store, store_update, header)
}

impl Chain {
    /// Find the hash that should be used as the reference point when requesting state sync
    /// headers and parts from other nodes for the epoch the block with hash `block_hash` belongs to.
    /// If syncing to the state of that epoch (the new way), this block hash might not yet be known,
    /// in which case this returns None. If syncing to the state of the previous epoch (the old way),
    /// it's the hash of the first block in that epoch.
    pub fn get_sync_hash(&self, block_hash: &CryptoHash) -> Result<Option<CryptoHash>, Error> {
        if block_hash == self.genesis().hash() {
            // We shouldn't be trying to sync state from before the genesis block
            return Ok(None);
        }
        let header = self.get_block_header(block_hash)?;
        Ok(self.chain_store.get_current_epoch_sync_hash(header.epoch_id()))
    }
