### Title
`vote_cancel_keygen` Drops `previously_cancelled_resharing_epoch_id`, Enabling Resharing Epoch-ID Reuse — (`crates/contract/src/state/initializing.rs`, `crates/contract/src/state/running.rs`)

---

### Summary

When keygen is cancelled via `vote_cancel_keygen` after a resharing was previously cancelled, the `previously_cancelled_resharing_epoch_id` guard stored in `RunningContractState` is silently discarded. The new `RunningContractState` produced by `vote_cancel` always sets this field to `None`, allowing the next resharing proposal to reuse the already-cancelled epoch ID. This breaks the epoch-uniqueness invariant the field was specifically designed to enforce, and can cause nodes to confuse a new resharing with the old cancelled one, leading to stale key-share or presignature material being used and signing failures.

---

### Finding Description

`RunningContractState` carries `previously_cancelled_resharing_epoch_id: Option<EpochId>` to prevent epoch-ID reuse across resharing attempts: [1](#0-0) 

`prospective_epoch_id()` reads this field to skip any already-used epoch: [2](#0-1) 

When `vote_add_domains` transitions to `InitializingContractState`, it stores only `epoch_id: self.keyset.epoch_id` — the `previously_cancelled_resharing_epoch_id` is never copied into the initializing state: [3](#0-2) 

`InitializingContractState` has no field for it: [4](#0-3) 

When `vote_cancel` (called by `vote_cancel_keygen`) creates the new `RunningContractState`, it calls `RunningContractState::new()`, which unconditionally sets `previously_cancelled_resharing_epoch_id: None`: [5](#0-4) [6](#0-5) 

The guard is therefore silently erased on every keygen-cancellation path.

---

### Impact Explanation

**Epoch-ID reuse in resharing.** Consider the following realistic sequence:

| Step | Contract State | `previously_cancelled_resharing_epoch_id` |
|------|---------------|------------------------------------------|
| 1 | Running (`epoch_id = 1`) | `None` |
| 2 | Resharing starts (`prospective = 2`), then cancelled | `Some(2)` |
| 3 | `vote_add_domains` → Initializing (`epoch_id = 1`) | *field not stored* |
| 4 | `vote_cancel_keygen` → Running (`epoch_id = 1`) | **`None`** ← bug |
| 5 | New resharing proposed → `prospective_epoch_id() = 1.next() = 2` | epoch 2 **reused** |

Epoch 2 was already used for the cancelled resharing. Nodes that participated in that cancelled resharing may have stored partial key-share material keyed by `(epoch_id=2, domain_id, attempt_id=0)`. When a new resharing starts with the same epoch ID and the same initial attempt ID, those nodes can find and act on stale local state. Additionally, the asset-cleanup logic in `delete_stale_triples_and_presignatures` triggers only on epoch-ID *change*; if the epoch ID is reused, stale presignatures tied to old key shares are not wiped, and using them with new key shares produces invalid signatures, breaking the signing flow.

This breaks the production safety invariant that epoch IDs are globally unique — an invariant the `previously_cancelled_resharing_epoch_id` field was explicitly introduced to enforce.

---

### Likelihood Explanation

The sequence requires three consecutive governance operations that each individually occur in normal production operation:

1. A resharing is cancelled (e.g., a node goes offline mid-resharing — documented in the E2E test `test_cancellation_of_resharing`).
2. Operators then call `vote_add_domains` to add a new signing domain (a routine operational step).
3. Keygen stalls (e.g., the same or another node goes offline) and is cancelled via `vote_cancel_keygen`.

All three steps are explicitly tested and expected in production. No adversarial action is required; any participant can trigger `vote_cancel_keygen` with a valid `next_domain_id`.

---

### Recommendation

`InitializingContractState` should carry the `previously_cancelled_resharing_epoch_id` from the `RunningContractState` it was created from, and `vote_cancel` should pass it through when constructing the new `RunningContractState`:

```rust
// In InitializingContractState, add:
pub previously_cancelled_resharing_epoch_id: Option<EpochId>,

// In RunningContractState::vote_add_domains, populate it:
Ok(Some(InitializingContractState {
    previously_cancelled_resharing_epoch_id:
        self.previously_cancelled_resharing_epoch_id,
    // ... existing fields ...
}))

// In InitializingContractState::vote_cancel, thread it through:
return Ok(Some(RunningContractState {
    previously_cancelled_resharing_epoch_id:
        self.previously_cancelled_resharing_epoch_id,
    // ... via a new constructor or direct field set ...
}));
```

Alternatively, add a dedicated constructor to `RunningContractState` that accepts `previously_cancelled_resharing_epoch_id` as a parameter, distinct from `RunningContractState::new()` which always sets it to `None`.

---

### Proof of Concept

1. Deploy the contract and reach `Running { epoch_id: 1, previously_cancelled_resharing_epoch_id: None }`.
2. Vote to reshard (prospective epoch 2); cancel it via `vote_cancel_resharing`. Contract is now `Running { epoch_id: 1, previously_cancelled_resharing_epoch_id: Some(2) }`.
3. Call `vote_add_domains` with all participants. Contract enters `Initializing { epoch_id: 1 }` — `previously_cancelled_resharing_epoch_id` is gone.
4. Call `vote_cancel_keygen`. Contract returns to `Running { epoch_id: 1, previously_cancelled_resharing_epoch_id: None }`.
5. Call `vote_new_parameters` with `prospective_epoch_id: 2`. The contract accepts it (because `prospective_epoch_id() = 1.next() = 2`), reusing the previously cancelled epoch 2.
6. Nodes that stored partial key-share state for epoch 2 from step 2 now face a new resharing under the same epoch ID, violating the uniqueness invariant and potentially causing stale-asset confusion and signing failures.

    pub fn prospective_epoch_id(&self) -> EpochId {
        match self.previously_cancelled_resharing_epoch_id {
            // If `cancelled_epoch_id`, then a resharing has already
            // been attempted but was cancelled.
