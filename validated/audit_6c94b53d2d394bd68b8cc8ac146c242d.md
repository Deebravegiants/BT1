### Title
Multiple Sub-transactions Can Claim the Same Deposit Refund in Dijkstra Era - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, `getConsumedDijkstraValue` computes the total consumed value for a top-level transaction by summing the consumed values of all sub-transactions using a **single, unmodified snapshot** of the ledger state (`lookupStakingDeposit`). Because this snapshot is never updated between sub-transactions, two or more sub-transactions each containing `UnRegDepositTxCert` for the **same staking credential** will each independently resolve the deposit lookup and each contribute the full deposit amount to the consumed-value total. The deposit is thus counted N times for N sub-transactions, while only one deposit exists in the ledger. The developers have already identified this as a problem and left a disabled test with the comment `"TODO: predicate failure not yet implemented"`, confirming the guard is absent.

---

### Finding Description

**Root cause — `getConsumedDijkstraValue`**

`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs` lines 65–91:

```haskell
getConsumedDijkstraValue pp lookupStakingDeposit lookupDRepDeposit utxo txBody =
  withBothTxLevels
    txBody
    ( \topTxBody ->
        txBodyConsumedValue topTxBody <> subTransactionsConsumedValue topTxBody
    )
    txBodyConsumedValue
  where
    txBodyConsumedValue = getConsumedMaryValue pp lookupStakingDeposit lookupDRepDeposit utxo
    subTransactionsConsumedValue topTxBody =
      foldMap'
        (getConsumedValue pp lookupStakingDeposit lookupDRepDeposit utxo . view bodyTxL)
        (topTxBody ^. subTransactionsTxBodyL)
```

The `lookupStakingDeposit :: Credential Staking -> Maybe Coin` closure is a **frozen snapshot** of the ledger state captured before any sub-transaction is processed. `foldMap'` iterates over every sub-transaction and calls `getConsumedValue` with the **same** `lookupStakingDeposit` for each one. There is no state update between iterations.

**Refund calculation per sub-transaction — `dijkstraTotalRefundsTxCerts`**

`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxCert.hs` lines 229–238:

```haskell
dijkstraTotalRefundsTxCerts = foldMap' $ \case
  UnRegDepositTxCert _ deposit -> deposit
  UnRegDRepTxCert _ deposit -> deposit
  _ -> zero
```

Each sub-transaction's refund is computed purely from the certificates it carries. No cross-sub-transaction deduplication is performed.

The test is marked `xit` (skipped) because the predicate failure that should reject this transaction **does not exist yet**. The `error "TODO: predicate failure not yet implemented"` confirms no rejection path is in place.

**Attack scenario**

1. Attacker registers staking credential `C` paying deposit `D`.
2. Attacker constructs top-level Dijkstra transaction with two sub-transactions:
   - `subTx1`: spends `input1`, contains `UnRegDepositTxCert C D`
   - `subTx2`: spends `input2`, contains `UnRegDepositTxCert C D`
3. `getConsumedDijkstraValue` computes:
   - `subTx1` consumed: `value(input1) + D`
   - `subTx2` consumed: `value(input2) + D`
   - Total consumed: `value(input1) + value(input2) + 2D`
4. Preservation-of-value check (`consumed == produced`) passes if outputs equal `value(input1) + value(input2) + 2D − fee`.
5. Attacker receives `D` extra ADA that was never legitimately theirs — the deposit pot is drained by `D` beyond what was deposited.

---

### Impact Explanation

This is a **Critical** impact: **Direct loss of ADA through an invalid ledger state transition**.

An attacker can drain the deposit pot by a factor of `(N − 1) × D` per transaction, where `N` is the number of sub-transactions each carrying `UnRegDepositTxCert` for the same credential. Because the Dijkstra era allows an `OMap` of sub-transactions (bounded only by transaction size limits), the multiplier can be significant. The deposit pot is a shared ledger resource; its unauthorized depletion constitutes direct destruction of ADA value.

---

### Likelihood Explanation

- The Dijkstra era is the newest era and its sub-transaction processing rules are still under active development (multiple `xit` and `xdescribe` markers in the test suite).
- The vulnerability requires only an unprivileged transaction sender — no special role, key, or governance majority is needed.
- The attack is deterministic and reproducible: any node running the Dijkstra ledger rules would accept the malformed transaction identically.
- The developers have already identified the exact scenario in a test but have not yet implemented the blocking predicate failure, making the window of exposure concrete and bounded only by when the fix is shipped.

---

### Recommendation

1. **In `getConsumedDijkstraValue`**: track which staking and DRep credentials have already contributed a refund across sub-transactions. Use a stateful fold (e.g., `foldl'` with an accumulator `Set`) instead of `foldMap'`, so that the second occurrence of the same credential in a different sub-transaction returns `Nothing` from the effective lookup.

2. **Implement the missing predicate failure**: the disabled test in `Test.Cardano.Ledger.Dijkstra.Imp.CertSpec` (`xit "Multiple subtransactions cannot get the same refund"`) should be completed with a concrete `DijkstraSubCertsPredFailure` (or equivalent) that rejects any top-level transaction where the union of `UnRegDepositTxCert` credentials across all sub-transactions contains duplicates.

3. **Cross-check `UnRegDRepTxCert`**: the same pattern applies to DRep deposit refunds via `UnRegDRepTxCert`; the fix must cover both credential types.

### Proof of Concept

```
Ledger state:
  deposit pot: D (from registration of credential C)
  accounts: { C → registered, deposit = D }

Attacker constructs:
  input1 = UTxO entry worth V1
  input2 = UTxO entry worth V2

  subTx1 = TxBody { inputs = {input1}, certs = [UnRegDepositTxCert C D] }
  subTx2 = TxBody { inputs = {input2}, certs = [UnRegDepositTxCert C D] }

  topTx = TxBody {
    subTransactions = OMap { subTx1, subTx2 },
    outputs = [TxOut attacker_addr (V1 + V2 + 2D - fee)]
  }

getConsumedDijkstraValue evaluation (same lookupStakingDeposit for both):
  subTx1: consumed = V1 + D   (lookupStakingDeposit C = Just D ✓)
  subTx2: consumed = V2 + D   (lookupStakingDeposit C = Just D ✓, state not updated)
  total consumed = V1 + V2 + 2D

produced = (V1 + V2 + 2D - fee) + fee = V1 + V2 + 2D

consumed == produced → UTXO preservation check PASSES

Attacker receives V1 + V2 + 2D - fee in outputs.
Deposit pot loses D that was never legitimately refundable twice.
```
