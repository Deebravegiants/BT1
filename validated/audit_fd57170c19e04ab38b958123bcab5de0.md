### Title
Harvester Can Unconditionally Redirect Farmer Block Rewards Without Authorization - (File: chia/farmer/farmer_api.py)

### Summary
The Chia farmer protocol accepts a `farmer_reward_address_override` field from any connected harvester in `RespondSignatures` messages and unconditionally redirects the farmer's XCH block reward to the harvester-supplied address. No cryptographic authorization is required, and the only fee-quality convention check (CHIP-22) is logging-only and never enforced. A malicious third-party harvester can divert 100% of the farmer's block rewards to an attacker-controlled address.

### Finding Description

The `RespondSignatures` protocol message includes an optional `farmer_reward_address_override` field: [1](#0-0) 

In `_process_respond_signatures()` inside `FarmerAPI`, the farmer unconditionally replaces its own configured reward address with whatever the harvester supplies: [2](#0-1) 

This overridden address is then passed directly into `DeclareProofOfSpace`, which is broadcast to the full node and determines where the XCH block reward is sent: [3](#0-2) 

The same override field also exists in `NewProofOfSpace`: [4](#0-3) 

When a `NewProofOfSpace` arrives with an override, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee`, which only logs a warning — it never rejects or blocks the override: [5](#0-4) 

Critically, even when the fee quality check fails (harvester is taking more than agreed), the function only emits a `log.warning` and returns normally. The override is still applied. There is no enforcement path that rejects the `DeclareProofOfSpace` with the attacker-supplied address.

### Impact Explanation

When a farmer wins a block, the `farmer_reward_puzzle_hash` embedded in the foliage block data determines where the XCH coinbase reward is sent. By supplying an arbitrary `farmer_reward_address_override`, a malicious harvester causes the farmer to construct and sign a `DeclareProofOfSpace` that pays the block reward to the attacker's address instead of the farmer's configured address. This is an unauthorized payout redirection of XCH block rewards — a High-impact outcome under the allowed scope.

### Likelihood Explanation

CHIP-22 explicitly supports third-party harvesters connecting to a farmer. Any harvester that establishes a connection can send `RespondSignatures` with an arbitrary `farmer_reward_address_override`. No special privilege, leaked key, or admin access is required. The attacker only needs to be a connected harvester peer. The farmer has no mechanism to reject the override cryptographically.

### Recommendation

- **Short term:** In `_process_respond_signatures`, reject (return `None`) any `RespondSignatures` where `farmer_reward_address_override` is set but the fee quality check fails, rather than only logging a warning. The farmer should enforce the CHIP-22 fee quality convention, not merely observe it.
- **Long term:** Require a cryptographic proof (e.g., a BLS signature over the override address using the plot key) from the harvester to authorize any `farmer_reward_address_override`. This mirrors the `dtf_program_signer` pattern: the harvester must prove it is authorized to redirect the reward before the farmer accepts the override.

### Proof of Concept

1. Attacker operates a third-party harvester and connects it to a victim farmer.
2. When the harvester finds a valid proof of space, it sends `NewProofOfSpace` with `farmer_reward_address_override = attacker_puzzle_hash` and `fee_info = None` (or any `applied_fee_threshold`).
3. The farmer calls `notify_farmer_reward_taken_by_harvester_as_fee`, which logs a warning but does not block the flow.
4. The farmer sends `RequestSignatures` to the harvester; the harvester responds with `RespondSignatures` also containing `farmer_reward_address_override = attacker_puzzle_hash`.
5. In `_process_respond_signatures`, the farmer sets `farmer_reward_address = attacker_puzzle_hash` with no further check.
6. The farmer broadcasts `DeclareProofOfSpace` with `farmer_reward_puzzle_hash = attacker_puzzle_hash` to the full node.
7. The full node farms the block; the XCH coinbase reward is paid to the attacker's address, not the farmer's.

    def notify_farmer_reward_taken_by_harvester_as_fee(
        self, sp: farmer_protocol.NewSignagePoint, proof_of_space: harvester_protocol.NewProofOfSpace
    ) -> None:
        """
        Apply a fee quality convention (see CHIP-22: https://github.com/Chia-Network/chips/pull/88)
        given the proof and signage point. This will be tested against the fee threshold reported
        by the harvester (if any), and logged.
        """
        assert proof_of_space.farmer_reward_address_override is not None

        challenge_str = str(sp.challenge_hash)

        ph_prefix = self.config["network_overrides"]["config"][self.config["selected_network"]]["address_prefix"]
        farmer_reward_puzzle_hash = encode_puzzle_hash(proof_of_space.farmer_reward_address_override, ph_prefix)

        self.log.info(
            f"Farmer reward for challenge '{challenge_str}' "
            + f"taken by harvester for reward address '{farmer_reward_puzzle_hash}'"
        )

        fee_quality = calculate_harvester_fee_quality(proof_of_space.proof.proof, sp.challenge_hash)
        fee_quality_rate = float(fee_quality) / float(0xFFFFFFFF) * 100.0

        if proof_of_space.fee_info is not None:
            fee_threshold = proof_of_space.fee_info.applied_fee_threshold
            fee_threshold_rate = float(fee_threshold) / float(0xFFFFFFFF) * 100.0

            if fee_quality <= fee_threshold:
                self.log.info(
                    f"Fee threshold passed for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
            else:
                self.log.warning(
                    f"Invalid fee threshold for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
                self.log.warning(
                    "Harvester illegitimately took a fee reward that "
                    + "did not belong to it or it incorrectly applied the fee convention."
                )
        else:
            self.log.warning(
                "Harvester illegitimately took reward by failing to provide its fee rate "
                + f"for challenge '{challenge_str}'. "
                + f"Fee quality was {fee_quality_rate:.3f}% ({fee_quality} or 0x{fee_quality:08x})"
            )
