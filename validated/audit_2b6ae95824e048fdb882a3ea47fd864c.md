The critical path is:

1. `validate_received_state_from_peer` calls `request_and_validate_additions` passing only `coin_state.coin.puzzle_hash` — not the full coin identity.
2. `request_and_validate_additions` asks the peer for additions at that puzzle_hash, then calls `validate_additions(additions_res.coins, additions_res.proofs, additions_root)`.
3. `validate_additions` verifies that `hash_coin_ids([c.name() for c in coin_list])` is in the Merkle tree — where `coin_list` is whatever the peer returned, **not** `coin_state.coin`.

5. Back in `validate_received_state_from_peer`, if the result is `True`, the coin_state (containing the fabricated coin) is accepted. **There is no check that `coin_state.coin` appears in the peer's returned coin list.**

The missing guard is explicit: after `validate_additions_result` is `True`, the code never asserts `coin_state.coin in additions_res.coins`.

For the unspent-coin path (`spent_height is None`), `validate_block_inclusion` is also called, which anchors the header block to the weight proof. This means the `additions_root` is the real canonical one. The attacker therefore cannot forge the root — but they **can** serve the real block's Merkle proof for a real coin C_real (same puzzle_hash, different parent/amount) while the coin_state they sent contains C_fake.

4. Wallet calls `request_and_validate_additions(peer, ..., wallet_ph, real_additions_root)` → attacker returns C_real with a valid Merkle proof.
5. `validate_additions` returns `True` (C_real is genuinely in the block).
6. `validate_block_inclusion` returns `True` (real block is in weight proof).

### Title
Wallet accepts phantom coin states via puzzle-hash-only Merkle proof — (`chia/wallet/wallet_node.py`, `chia/wallet/util/wallet_sync_utils.py`)

### Summary
`validate_received_state_from_peer` validates that *some* coin with `coin_state.coin.puzzle_hash` exists in a block's additions Merkle tree, but never verifies that `coin_state.coin` itself is among the validated coins. An untrusted peer can serve a fabricated `CoinState` for a non-existent coin and back it with a valid Merkle proof for a different real coin sharing the same puzzle hash, causing the wallet to accept and store the phantom coin.

### Finding Description

`validate_received_state_from_peer` passes only `coin_state.coin.puzzle_hash` to `request_and_validate_additions`: [1](#0-0) 

`request_and_validate_additions` asks the peer for additions, validates the returned coin list against the Merkle root, and returns a bare `bool`: [2](#0-1) 

`validate_additions` verifies `hash_coin_ids([c.name() for c in coin_list])` against the proof — where `coin_list` is the peer-supplied list, not `coin_state.coin`: [3](#0-2) 

After the call returns `True`, the caller immediately proceeds to accept the coin state with no check that `coin_state.coin` appears in the validated list: [4](#0-3) 

The validated coin list is never returned to the caller; it is silently discarded inside `request_and_validate_additions`.

### Impact Explanation

An untrusted peer can inject arbitrary phantom coins into the wallet's coin store. The wallet's confirmed balance is inflated by the phantom amount. The wallet cannot spend the phantom coin (it does not exist on-chain), but the corrupted coin record persists until a re-sync from a trusted source. This constitutes an accounting change affecting XCH and corruption of wallet sync state with direct security impact.

### Likelihood Explanation

The attacker only needs to:
- Run a reachable Chia full node (untrusted peer)
- Spend a minimal amount (e.g., 1 mojo) to create a real coin with the target wallet's puzzle hash in any block
- Serve the fabricated `CoinState` and the real block's Merkle proof

No key material, cryptographic break, or privileged access is required. The attack is fully local-testable.

### Recommendation

After `validate_additions` returns `True`, verify that `coin_state.coin` is present in the coin list returned by the peer. Concretely, `request_and_validate_additions` should return the validated coin list (or a set of validated coin names) rather than a bare `bool`, and `validate_received_state_from_peer` should assert `coin_state.coin.name() in validated_coin_names` before accepting the state.

async def request_and_validate_additions(
    peer: WSChiaConnection,
    peer_request_cache: PeerRequestCache,
    height: uint32,
    header_hash: bytes32,
    puzzle_hash: bytes32,
    additions_root: bytes32,
) -> bool | None:
    if peer_request_cache.in_additions_in_block(header_hash, puzzle_hash):
        return True
    additions_request = RequestAdditions(height, header_hash, [puzzle_hash])
    additions_res: RespondAdditions | RejectAdditionsRequest | None = await peer.call_api(
        FullNodeAPI.request_additions, additions_request
    )
    if additions_res is None or isinstance(additions_res, RejectAdditionsRequest):
        log.info(
            f"Failed to obtain additions for height {height} header hash {header_hash} "
            f"puzzle hash {puzzle_hash} from peer {peer.peer_node_id} / {peer.peer_info.host} "
            f"version {peer.version} response: {additions_res}"
        )
        return None
    result: bool = validate_additions(
        additions_res.coins,
        additions_res.proofs,
        additions_root,
    )
    peer_request_cache.add_to_additions_in_block(header_hash, puzzle_hash, height)
    return result
