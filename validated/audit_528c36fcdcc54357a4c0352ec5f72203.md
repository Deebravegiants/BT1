### Title
Unverified CKD Output for `AppPublicKey` Variant Allows Single Byzantine Participant to Deliver Forged Key Material - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract verifies the CKD output **only** for the `AppPublicKeyPV` (publicly verifiable) variant. For the `AppPublicKey` (privately verifiable, legacy) variant, **no output verification is performed**. A single malicious participant (strictly below the signing threshold) who front-runs the honest leader can call `respond_ckd` with an arbitrary forged `CKDResponse`, which the contract unconditionally accepts and delivers to the requesting application.

---

### Finding Description

In `respond_ckd`, the contract applies `ckd_output_check` only for the `AppPublicKeyPV` branch:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For the `AppPublicKey` arm, the `CKDResponse` fields `big_y` and `big_c` are accepted without any cryptographic verification. The function then immediately resolves the pending yield and delivers the response to the caller: [2](#0-1) 

By contrast, the regular `respond` function for ECDSA/EdDSA signatures always verifies the signature against the expected derived public key before resolving: [3](#0-2) 

The `respond_ckd` entry point is callable by **any single attested participant**; there is no threshold requirement on the contract side for the respond call itself — the threshold is enforced only in the off-chain MPC protocol that is supposed to produce the response. The contract does not enforce that the response was produced by the threshold protocol. [4](#0-3) 

The `AppPublicKey` variant is the legacy/default variant still accepted by the contract: [5](#0-4) 

---

### Impact Explanation

An application that calls `request_app_private_key` with the `AppPublicKey` variant receives an encrypted blob `(big_y, big_c)` that it decrypts with its own secret key to recover its deterministic application secret. If a malicious participant delivers forged `(big_y, big_c)` values, the application decrypts garbage. Any funds, signing keys, or secrets the application derives from that key become permanently inaccessible — the application cannot reconstruct the correct key, and the attacker does not know it either. This constitutes **permanent freezing of funds** controlled through the chain-signature CKD flow.

---

### Likelihood Explanation

The `AppPublicKey` variant is the legacy default and is still accepted by the contract. Any attested participant can monitor the NEAR chain for pending CKD requests (the pending map is public contract state). A malicious participant who observes a new `AppPublicKey` CKD request can immediately submit a forged `respond_ckd` call, racing the honest leader. Because `resolve_yields_for` removes the request on first resolution, the first caller wins. The malicious participant needs only to be faster than the honest leader — a realistic condition given that NEAR transaction ordering is observable in the mempool.

---

### Recommendation

1. **Deprecate `AppPublicKey`**: Require all new CKD requests to use `AppPublicKeyPV`, which has proper on-chain output verification via `ckd_output_check`.
2. **Add threshold enforcement on-chain**: Require at least `t` participants to submit matching `CKDResponse` values before resolving the yield, analogous to how the signing threshold is enforced in the off-chain protocol.
3. **At minimum, document** that `AppPublicKey` provides no on-chain integrity guarantee and that a single malicious participant can forge the response.

---

### Proof of Concept

1. App submits `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(app_pk)` and 1 yoctoNEAR deposit.
2. Malicious attested participant observes the pending `CKDRequest` in contract state.
3. Malicious participant calls `respond_ckd(request, CKDResponse { big_y: [0u8; 48].into(), big_c: [0u8; 48].into() })` before the honest leader.
4. Contract executes lines 675–682: the `AppPublicKey` arm is a no-op — no `ckd_output_check` is called.
5. `resolve_yields_for` resolves the yield and delivers the forged response to the app.
6. App decrypts `big_c - big_y * a` (where `a` is its secret key) and obtains garbage instead of the correct MPC-derived secret.
7. Any funds or keys the app controls via this derived secret are permanently frozen.
