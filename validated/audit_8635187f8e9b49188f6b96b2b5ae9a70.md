### Title
Token Decimal Assumption in `viewSwapRsETHAmountAndFee` Causes Incorrect rsETH Minting for Non-18-Decimal Tokens - (`contracts/pools/RSETHPoolNoWrapper.sol`, `contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolV3WithNativeChainBridge.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`, `contracts/pools/RSETHPool.sol`)

---

### Summary

Every L2 pool contract's `viewSwapRsETHAmountAndFee(uint256 amount, address token)` computes the rsETH output as:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

This formula silently assumes `amountAfterFee` is expressed in 18 decimals. For any ERC-20 token whose `decimals()` differs from 18 (e.g., USDC at 6, WBTC at 8), the result is off by a factor of `10^(18 - tokenDecimals)`, causing users to receive a drastically wrong amount of rsETH.

---

### Finding Description

The token-deposit variant of `viewSwapRsETHAmountAndFee` is present identically across all pool contracts: [1](#0-0) 

```solidity
// rate of token in ETH
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();

// Calculate the final rsETH amount
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

The ETH-deposit sibling formula makes the 18-decimal assumption explicit: [2](#0-1) 

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

Here `1e18` normalises the ETH wei amount so the division by `rsETHToETHrate` (also 1e18-scaled) yields an 18-decimal rsETH quantity. The token formula omits this normalisation step entirely. `tokenToETHRate` is the price of **one whole token** expressed in 1e18 precision (e.g., 1 USDC ≈ 0.0003 ETH → `tokenToETHRate = 3e14`), but `amountAfterFee` is in the token's **native** decimals (6 for USDC). The product `amountAfterFee * tokenToETHRate` therefore has `6 + 18 = 24` effective decimal places, and after dividing by `rsETHToETHrate` (18 decimals) the result carries only 6 decimal places — not the 18 that rsETH requires.

The same flaw appears in every pool variant: [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

The `addSupportedToken` function (restricted to `TIMELOCK_ROLE`) accepts any ERC-20 address and any oracle without validating that the token has 18 decimals: [7](#0-6) 

---

### Impact Explanation

**Impact: Low — Contract fails to deliver promised returns (severe under-minting) for any non-18-decimal token.**

If a 6-decimal token such as USDC is added as a supported collateral:

- User deposits 1,000 USDC (`amountAfterFee = 1e9` in 6-decimal units).
- `tokenToETHRate` ≈ `3e14` (0.0003 ETH per USDC, 1e18-scaled).
- `rsETHToETHrate` ≈ `1.05e18`.
- Computed: `rsETHAmount = 1e9 * 3e14 / 1.05e18 ≈ 285,714` (≈ 2.857e5 units).
- Correct: `1,000 USDC × 0.0003 ETH / 1.05 ETH/rsETH ≈ 0.2857 rsETH = 2.857e17` units.
- **Shortfall: user receives ~10¹² times fewer rsETH than owed.**

The user's deposited tokens are transferred to the pool and locked, but the rsETH minted is negligible — effectively a near-total loss of the deposited value. The pool retains the tokens, creating an insolvency-adjacent state for that token's depositors.

---

### Likelihood Explanation

**Likelihood: Low.**

All tokens currently supported in these pools are Liquid Staking Tokens (wstETH, etc.) which universally use 18 decimals, so the bug is latent today. Triggering it requires a `TIMELOCK_ROLE` holder to add a non-18-decimal token via `addSupportedToken`. This is a legitimate administrative action (not a key compromise), and the protocol's design explicitly supports arbitrary ERC-20 tokens. The absence of any decimal validation in `addSupportedToken` makes accidental misconfiguration plausible as the protocol expands to new collateral types.

---

### Recommendation

Normalise `amountAfterFee` to 18 decimals before applying the rate conversion, or scale the result accordingly:

```solidity
uint8 tokenDecimals = IERC20Metadata(token).decimals();
rsETHAmount = amountAfterFee
    * tokenToETHRate
    * (10 ** (18 - tokenDecimals))   // normalise to 18 dec
    / rsETHToETHrate;
```

Alternatively, add a guard in `addSupportedToken` that reverts if `IERC20Metadata(token).decimals() != 18`, making the assumption explicit and enforced on-chain.

---

### Proof of Concept

1. Admin calls `addSupportedToken(USDC, usdcOracle, bridge)` on `RSETHPoolNoWrapper`.
2. `usdcOracle.getRate()` returns `3e14` (≈ 0.0003 ETH per USDC, 1e18-scaled).
3. User calls `deposit(USDC, 1_000e6, "ref")` — transfers 1,000 USDC (1e9 units).
4. `viewSwapRsETHAmountAndFee(1e9, USDC)` executes:
   - `fee = 1e9 * feeBps / 10_000` (small)
   - `amountAfterFee ≈ 1e9`
   - `rsETHToETHrate = 1.05e18`
   - `rsETHAmount = 1e9 * 3e14 / 1.05e18 ≈ 285_714`
5. `rsETH.safeTransfer(msg.sender, 285_714)` — user receives `285_714` wei of rsETH (≈ 2.857e-13 rsETH).
6. Correct amount: `≈ 2.857e17` wei of rsETH (≈ 0.2857 rsETH).
7. User loses ~99.9999999999% of their deposited value; the USDC remains locked in the pool.
