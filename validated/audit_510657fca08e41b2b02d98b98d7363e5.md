### Title
Deposit Refund in Meta-Transaction Inner Receipt Sent to Delegate Sender (Alice) Instead of Relayer Who Paid - (`runtime/runtime/src/lib.rs`)

### Summary
In NEAR's meta-transaction (NEP-366) flow, when an inner action receipt fails on the receiver's shard, the attached deposit is refunded to the `predecessor_id` of the inner receipt (Alice, the delegate sender), not to the relayer who actually funded the deposit. An unprivileged user can exploit this to steal the relayer's deposited NEAR tokens by crafting a meta transaction with a large attached deposit that is designed to fail.

### Finding Description
The vulnerability class from the external report is: **a token transfer going to an intermediary account (`msg.sender`) instead of the original funder**, because the code uses the immediate caller's identity rather than the true economic originator.

The exact analog exists in nearcore's meta-transaction deposit refund path.

**Call chain in meta transactions:**
1. Relayer submits a transaction wrapping Alice's `SignedDelegateAction` (relayer pays all gas and deposits)
2. On Alice's shard, the delegate action is unwrapped and a new inner action receipt is created with `predecessor_id = Alice`
3. The inner receipt is forwarded to Bob's shard for execution
4. If the inner action fails on Bob's shard, `refund_unspent_gas_and_deposits` is called

In `runtime/runtime/src/lib.rs`, the deposit refund is sent to `receipt.balance_refund_receiver()`:

```rust
if deposit_refund > Balance::ZERO {
    result.new_receipts.push(Receipt::new_balance_refund(
        receipt.balance_refund_receiver(),   // ŌåÉ returns predecessor_id() = Alice
        deposit_refund,
    ));
}
```

The `balance_refund_receiver()` method in `core/primitives/src/receipt.rs` returns `refund_to` if set, otherwise falls back to `predecessor_id()`:

```rust
pub fn balance_refund_receiver(&self) -> &AccountId {
    self.refund_to().as_ref().unwrap_or_else(|| self.predecessor_id())
}
```

For the inner receipt, `predecessor_id` is **Alice** (the delegate sender), not the relayer. The relayer paid the deposit when the transaction was converted to a receipt, but Alice is the predecessor of the inner receipt because she forwarded it from her shard to Bob's shard. The deposit refund therefore goes to Alice, not the relayer.

The nearcore documentation explicitly acknowledges this in `docs/architecture/how/meta-tx.md`:

> "But what if the transaction fails execution on Bob's shard? At this point, the predecessor is `Alice` and therefore she receives the token balance refunded, not the relayer. This is something relayer implementations must be aware of since there is a **financial incentive for Alice to submit meta transactions that have high balances attached but will fail on Bob's shard**."

The `promise_set_refund_to` host function (`runtime/near-vm-runner/src/logic/logic.rs:2383`) exists to redirect deposit refunds, but the meta-transaction protocol does not automatically use it to route refunds back to the relayer.

### Impact Explanation
An unprivileged user (Alice) can steal the relayer's deposited NEAR tokens. The corrupted protocol value is the account balance: Alice's balance is incorrectly credited with the relayer's deposit, and the relayer's balance is not restored. The amount stolen equals the full attached deposit on the failing inner action. Since there is no upper bound on the deposit Alice can specify in her `DelegateAction`, the loss can be arbitrarily large (up to the relayer's balance).

### Likelihood Explanation
Any user of a meta-transaction relayer can trigger this. The attack requires:
1. Alice creates a `DelegateAction` with a large `deposit` on an inner `FunctionCall` or `Transfer` action
2. Alice ensures the inner action will fail on Bob's shard (e.g., targeting a non-existent account, a contract that panics, or a method that does not exist)
3. The relayer submits the meta transaction, paying the deposit
4. The inner receipt fails; the deposit refund is sent to Alice

This is a straightforward, low-effort attack requiring only the ability to craft a `SignedDelegateAction` and submit it to a relayer.

### Recommendation
When constructing the inner action receipt from a `DelegateAction`, the runtime should set `refund_to` on the inner receipt to the relayer's account ID (the `signer_id` of the outer receipt), so that `balance_refund_receiver()` returns the relayer rather than Alice. Alternatively, the `action_delegate` processing code should call `set_refund_to` on the spawned inner receipt to point to the outer receipt's `signer_id` (the relayer).
