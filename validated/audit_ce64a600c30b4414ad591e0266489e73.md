### Title
Missing `freeze_authority` Check on Native Token Registration Allows Permanent Vault Lock - (`File: solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs`)

---

### Summary
The `log_metadata` instruction registers native Solana SPL tokens for bridging by creating a vault PDA that will hold locked tokens. It checks that the mint's `mint_authority` is not the bridge's authority PDA, but performs no check on the mint's `freeze_authority`. If a registered token has an active `freeze_authority`, that authority can freeze the vault account, permanently blocking all `finalize_transfer` (unlock) operations and irreversibly locking every user's bridged funds.

---

### Finding Description

`log_metadata` accepts any SPL token whose `mint_authority` is not the bridge authority PDA. The only constraint on the mint is:

```rust
#[account(
    constraint = !mint.mint_authority.contains(authority.key),
    mint::token_program = token_program,
)]
pub mint: Box<InterfaceAccount<'info, Mint>>,
``` [1](#0-0) 

There is no constraint of the form `mint.freeze_authority == COption::None`. The instruction then creates a vault PDA that will custody all locked tokens for this mint:

```rust
#[account(
    init_if_needed,
    payer = common.payer,
    token::mint = mint,
    token::authority = authority,
    seeds = [VAULT_SEED, mint.key().as_ref()],
    ...
)]
pub vault: Box<InterfaceAccount<'info, TokenAccount>>,
``` [2](#0-1) 

When users bridge native tokens via `init_transfer`, their tokens are transferred into this vault:

```rust
transfer_checked(
    CpiContext::new(..., TransferChecked {
        from: self.from.to_account_info(),
        to: vault.to_account_info(),
        ...
    }),
    ...
)?;
``` [3](#0-2) 

When tokens return from NEAR via `finalize_transfer`, the vault is the source of the unlock:

```rust
transfer_checked(
    CpiContext::new_with_signer(..., TransferChecked {
        from: vault.to_account_info(),
        to: self.token_account.to_account_info(),
        ...
    }, ...),
    ...
)?;
``` [4](#0-3) 

If the token's `freeze_authority` issues a `FreezeAccount` instruction against the vault PDA, every subsequent `transfer_checked` CPI from or to that vault will fail with `AccountFrozen`. The SPL token program rejects all token instructions on frozen accounts.

---

### Impact Explanation

**Critical — Irreversible fund lock.** All tokens deposited into the vault by users who bridged to NEAR become permanently inaccessible. `finalize_transfer` (the only redemption path back to Solana) unconditionally calls `transfer_checked` from the vault; a frozen vault causes every such call to fail. There is no admin escape hatch or alternative unlock path in the program. The nonce is consumed before the transfer attempt, so even retrying after a hypothetical thaw would require a new signed payload from NEAR. The recipient's ATA could also be frozen independently, blocking that side of the transfer as well. [5](#0-4) 

---

### Likelihood Explanation

Any unprivileged actor can call `log_metadata` with a token they control that has an active `freeze_authority`. Once users deposit into the vault, the `freeze_authority` holder (which may be the token issuer, a multisig, or even a compromised upgrade authority) can freeze the vault. This is not a theoretical edge case: many real SPL tokens retain a `freeze_authority` for compliance or upgrade reasons. The registration step is permissionless, so no privileged bridge operator needs to be involved. [1](#0-0) 

---

### Recommendation

Add a constraint in `LogMetadata` that rejects mints with an active `freeze_authority`:

```rust
#[account(
    constraint = !mint.mint_authority.contains(authority.key),
    constraint = mint.freeze_authority == COption::None
        @ ErrorCode::FreezeAuthorityNotAllowed,
    mint::token_program = token_program,
)]
pub mint: Box<InterfaceAccount<'info, Mint>>,
```

For Token-2022 mints, also verify that the `PermanentDelegate` and `TransferHook` extensions (which can similarly block transfers) are absent or safe. Add a corresponding `ErrorCode::FreezeAuthorityNotAllowed` variant. [1](#0-0) 

---

### Proof of Concept

1. Attacker deploys an SPL token `T` with `freeze_authority = attacker_keypair`.
2. Attacker calls `log_metadata` with mint `T`. The constraint `!mint.mint_authority.contains(authority.key)` passes (attacker holds mint authority, not the bridge). Vault PDA is created.
3. Victim calls `init_transfer` with mint `T` and amount `X`. `transfer_checked` moves `X` tokens from victim's ATA into the vault. Wormhole message is posted; NEAR credits the victim.
4. Attacker calls SPL Token `FreezeAccount` on the vault PDA using `attacker_keypair` as `freeze_authority`.
5. Victim (or relayer) calls `finalize_transfer` to redeem on Solana. The `transfer_checked` CPI at line 103 of `finalize_transfer.rs` fails with `AccountFrozen`. The nonce has already been marked used. Victim's `X` tokens are permanently locked in the frozen vault with no recovery path. [4](#0-3)
