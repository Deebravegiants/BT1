### Title
Unverified CKD Response for `AppPublicKey` Variant Allows Byzantine Participant to Bypass Threshold Requirement and Corrupt Key Derivation Output - (File: crates/contract/src/lib.rs)

---

### Summary

The `respond_ckd` function in the MPC contract performs a cryptographic output check **only** for the `AppPublicKeyPV` variant. For the `AppPublicKey` (privately-verifiable, legacy) variant, the check arm is empty — any arbitrary `CKDResponse` is accepted without verification. A single Byzantine MPC participant (below threshold) can race the legitimate coordinator to submit a crafted response, resolving the user's pending CKD yield with an attacker-controlled output, bypassing the threshold requirement entirely.

---

### Finding Description

In `respond_ckd`, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← empty: no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` enforces the BLS12-381 pairing equation `e(C, G2) = e(Y, app_pk2) · e(H(pk, app_id), msk_pk)`, which cryptographically proves the response encodes the correct threshold-derived secret. [2](#0-1) 

For `AppPublicKey`, the empty arm `{}` means **no such proof is required**. The contract proceeds directly to `resolve_yields_for`, delivering whatever `(big_y, big_c)` the caller supplied. [3](#0-2) 

The only gate before this branch is `assert_caller_is_attested_participant_and_protocol_active`, which verifies the caller is a registered, attested participant — not that the response is cryptographically correct. [4](#0-3) 

The existing unit test for the `AppPublicKey` path confirms this: it submits `big_y = [1u8; 48]` and `big_c = [2u8; 48]` — bytes that are not valid BLS12-381 G1 points — and the contract accepts them without complaint. [5](#0-4) 

The `AppPublicKey` variant is the default/legacy format, accepted as a plain G1 point string, and is the variant used in the majority of documented examples and the e2e test suite. [6](#0-5) 

---

### Impact Explanation

**Threshold bypass / corrupted key derivation output.**

The CKD protocol requires a threshold of participants to contribute shares; the coordinator aggregates them and submits the final `(Y, C)`. The contract is supposed to be the last line of defense ensuring the submitted output is consistent with the MPC master secret. For `AppPublicKeyPV` this holds. For `AppPublicKey` it does not.

A Byzantine participant (strictly below threshold) can:

1. Observe a pending `AppPublicKey` CKD request on-chain (the `CKDRequest` is public).
2. Race the legitimate coordinator by calling `respond_ckd` with an attacker-chosen `CKDResponse { big_y, big_c }`.
3. The contract accepts it — no pairing check, no proof of correct computation.
4. `resolve_yields_for` fires the yield callback, delivering the attacker's response to the waiting user.

The user receives `(Y', C')` instead of the honest `(Y, C)`. The user's derived secret becomes `C' − a·Y'`. Because the attacker controls both `Y'` and `C'`, and the user's ephemeral scalar `a` is unknown to the attacker, the attacker cannot in general predict the resulting key — but they have definitively prevented the user from obtaining the correct threshold-derived key. If the user does not independently verify the response (which the design delegates to the app layer for this variant), they silently derive and use a wrong key. The pending yield slot is consumed; the user must re-submit and pay again.

Additionally, if the attacker sets `big_y` to the G1 identity point (which the contract does not reject for this variant), then `C' − a·0 = C'`, and the attacker fully controls the user's derived secret — a complete key-derivation hijack.

This breaks the core safety invariant: **a CKD output must be the product of threshold-many honest participants' shares, not a single Byzantine actor's choice**.

Impact: **Medium** — request-lifecycle and key-derivation-output manipulation that breaks a production safety invariant, reachable by a single Byzantine participant below threshold without any privileged access.

---

### Likelihood Explanation

- The `AppPublicKey` variant is the legacy default; most existing integrations use it.
- Any attested MPC participant can call `respond_ckd` — the contract does not restrict this to the coordinator role.
- The pending request and its `CKDRequest` key are observable on-chain the moment `request_app_private_key` is called.
- The race window is the time between the user's request landing on-chain and the honest coordinator completing the MPC round and submitting. A Byzantine node that skips the MPC round entirely can respond immediately.
- No leaked keys, no operator access, no threshold collusion required — one Byzantine participant suffices.

Likelihood: **Medium**.

---

### Recommendation

1. **Add the pairing check for `AppPublicKey` requests.** The `ckd_output_check` function already exists and is correct; it only needs the G2 component of the app public key. Require callers using `AppPublicKey` to also supply a G2 companion point (i.e., migrate to `AppPublicKeyPV`), or derive the check differently.

2. **Deprecate `AppPublicKey` in favour of `AppPublicKeyPV`.** The publicly-verifiable variant already provides the necessary on-chain guarantee. Existing callers can be migrated; the legacy path can be removed after a transition period.

3. **Enforce coordinator-only submission.** Include the coordinator's identity in the `CKDRequest` (derivable from the participant list and a deterministic leader-election rule) and assert `signer == coordinator` inside `respond_ckd`, preventing any other participant from racing the coordinator.

### Proof of Concept

```
// 1. User submits request with AppPublicKey variant
contract.request_app_private_key(CKDRequestArgs {
    derivation_path: "mykey".to_string(),
    app_public_key: CKDAppPublicKey::AppPublicKey(user_g1_pk),
    domain_id: bls_domain_id,
});

// 2. Byzantine participant observes the pending CKDRequest on-chain.
//    They skip the MPC round and immediately call respond_ckd with
//    an attacker-chosen response — no valid pairing required.
contract.respond_ckd(
    ckd_request,                          // matches the pending request key
    CKDResponse {
        big_y: Bls12381G1PublicKey([0x40, 0, 0, ..., 0]),  // G1 identity
        big_c: Bls12381G1PublicKey(attacker_chosen_bytes),  // arbitrary point
    },
);
// Contract accepts: AppPublicKey arm is empty, no ckd_output_check called.

// 3. Yield resolves. User receives (big_y=identity, big_c=attacker_chosen).
//    User computes: sig = big_c - a * big_y = big_c - a*0 = big_c.
//    Attacker knows big_c, therefore knows sig, therefore knows the
//    user's derived key = HKDF(sig).
```

```rust
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
