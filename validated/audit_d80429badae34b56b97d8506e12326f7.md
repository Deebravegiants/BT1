### Title
ETH Deposit Limit Check Omits Deposit Amount, Allowing Unlimited ETH Deposits — (File: `contracts/LRTDepositPool.sol`)

### Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` applies the deposit-cap check asymmetrically: for ERC-20 assets it correctly adds the incoming `amount` to the running total before comparing against the limit, but for ETH it omits `amount` entirely. Any unprivileged depositor can therefore bypass the ETH deposit cap in a single transaction.

### Finding Description
The function at the heart of the deposit guard is:

```solidity
// contracts/LRTDepositPool.sol  lines 676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // ← amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← amount included
}
``` [1](#0-0) 

For ETH the guard only asks "has the cap already been breached?" — it never asks "would this deposit breach the cap?" The `amount` parameter (i.e. `msg.value`) is silently discarded. The ERC-20 branch correctly includes `amount` in the comparison.

`depositETH` calls `_beforeDeposit`, which calls this function and reverts on `true`:

```solidity
// contracts/LRTDepositPool.sol  lines 648-670
function _beforeDeposit(address asset, uint256 depositAmount, uint256 minRSETHAmountExpected) private view returns (uint256 rsethAmountToMint) {
    ...
    if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
        revert MaximumDepositLimitReached();
    }
    ...
}
``` [2](#0-1) 

Because the ETH branch never evaluates `depositAmount`, the revert is never triggered for any single ETH deposit so long as the pre-deposit total is ≤ the configured limit.

### Impact Explanation
The ETH deposit limit is a risk-management control (e.g., capping EigenLayer exposure). With the check broken, a single depositor can push the protocol's ETH holdings arbitrarily above the intended ceiling in one call. The protocol then holds more ETH in EigenLayer strategies than governance intended, violating the promised cap and potentially increasing slashing exposure beyond the risk parameters set by the admin.

**Impact: Low — Contract fails to deliver promised returns** (the deposit limit guarantee is not upheld for ETH).

### Likelihood Explanation
The entry point `depositETH` is public and permissionless. No special role, flash loan, or multi-step setup is required. Any depositor who sends ETH while `totalAssetDeposits ≤ depositLimitByAsset[ETH]` will bypass the cap. Likelihood is **High**.

### Recommendation
Include the deposit amount in the ETH branch, mirroring the ERC-20 branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept
1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 1 000 ETH`.
2. Current `getTotalAssetDeposits(ETH_TOKEN)` returns `999 ETH`.
3. Attacker calls `depositETH{value: 50 000 ETH}(0, "")`.
4. Inside `_checkIfDepositAmountExceedesCurrentLimit`: `999 > 1 000` → `false` → no revert.
5. `_mintRsETH` mints rsETH for the full 50 000 ETH.
6. Protocol now holds 50 999 ETH in EigenLayer — 50× the intended cap — with no further check preventing it. [3](#0-2) [1](#0-0)
