### Title
Zero-Amount ERC20 Transfer in `MerkleDistributor.claim()` Freezes All User Claims When `feeInBPS == 0` - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

### Summary
`MerkleDistributor.claim()` unconditionally calls `safeTransfer(protocolTreasury, fee)` even when `fee == 0`. For ERC20 tokens that revert on zero-amount transfers (e.g., LEND), this permanently blocks all users from claiming their tokens whenever `feeInBPS` is set to zero.

### Finding Description
In `MerkleDistributor.claim()`, after computing `fee = (claimableAmount * feeInBPS) / 10_000`, the contract always executes `IERC20(token).safeTransfer(protocolTreasury, fee)` with no guard for the zero case:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);

// Send the fee to the protocol treasury
IERC20(token).safeTransfer(protocolTreasury, fee);  // ŌåÉ no zero-check
``` [1](#0-0) 

The `setFeeInBPS` function permits setting `feeInBPS` to `0` ŌĆö the only guard is `if (_feeInBPS > MAX_FEE_IN_BPS)`: [2](#0-1) 

When `feeInBPS == 0`, `fee == 0`, and the unconditional `safeTransfer(protocolTreasury, 0)` will revert for any token that disallows zero-amount transfers. This makes `claim()` completely unusable for every user of that distributor instance.

By contrast, the sibling contracts `KernelMerkleDistributor` and `KernelTop100MerkleDistributor` correctly guard the fee transfer:

```solidity
if (fee > 0) {
    kernel.safeTransfer(protocolTreasury, fee);
}
``` [3](#0-2) [4](#0-3) 

`MerkleDistributor` is missing this guard entirely.

### Impact Explanation
When `feeInBPS == 0` and the distributed token reverts on zero-amount transfers, every call to `claim()` reverts. All tokens allocated to users in the merkle tree are frozen inside the contract ŌĆö users cannot receive their entitled yield/rewards. This constitutes **permanent freezing of unclaimed yield** (Medium severity) because the state is stuck until the owner intervenes by raising `feeInBPS` to a non-zero value, which itself changes the economics of every pending claim.

### Likelihood Explanation
- The owner can set `feeInBPS = 0` at any time via `setFeeInBOS` ŌĆö a natural configuration for a "no-fee" distribution round.
- The `token` address is configurable via `setToken()`, meaning any ERC20 (including tokens that revert on zero transfers) can be used.
- The combination of `feeInBPS == 0` and a zero-transfer-reverting token is realistic and requires no attacker ŌĆö it is triggered by any user calling `claim()`.

### Recommendation
Add a zero-check before the fee transfer, mirroring the pattern already used in `KernelMerkleDistributor` and `KernelTop100MerkleDistributor`:

```solidity
if (fee > 0) {
    IERC20(token).safeTransfer(protocolTreasury, fee);
}
``` [5](#0-4) 

### Proof of Concept
1. Deploy `MerkleDistributor` with a token that reverts on zero-amount transfers (e.g., LEND at `0x80fB784B7eD66730e8b1DBd9820aFD29931aab03`).
2. Owner calls `setFeeInBPS(0)` ŌĆö a valid call since `0 <= MAX_FEE_IN_BPS`.
3. Owner sets a merkle root with user allocations.
4. Any user calls `claim(index, account, cumulativeAmount, merkleProof)`.
5. `fee = (claimableAmount * 0) / 10_000 = 0`.
6. `IERC20(token).safeTransfer(protocolTreasury, 0)` reverts because LEND disallows zero-amount transfers.
7. All claims revert; all user funds are frozen in the contract.
