### Title
Stale Votes from Removed Participants Inflate `LauncherHashVotes` Count, Allowing Sub-Threshold Launcher Hash Approval — (File: `crates/contract/src/lib.rs`)

---

### Summary

`vote_add_launcher_hash` and `vote_remove_launcher_hash` both rely on `LauncherHashVotes::count_votes` to determine whether the governance threshold (or unanimity) has been reached. `count_votes` iterates over **all** entries in the `vote_by_account` map without filtering for current participants. If participants who voted are later removed via resharing, their votes persist in the map and are still counted, while the threshold denominator (`self.threshold()?.value()` / `total_participants`) reflects only the **current** participant set. This is the direct analog of the `trove-manager` bug: the denominator is computed from live state, but the numerator silently includes stale, unapplied entries.

---

### Finding Description

**Root cause — `count_votes` does not filter by current participants:** [1](#0-0) 

```rust
fn count_votes(&self, action: &LauncherVoteAction) -> u64 {
    u64::try_from(
        self.vote_by_account
            .values()
            .filter(|a| *a == action)
            .count(),
    )
    .expect("participant count should not overflow u64")
}
```

The map key is `AuthenticatedParticipantId`; entries for removed participants are never evicted by this function.

**Call site in `vote_add_launcher_hash`:** [2](#0-1) 

```rust
let participant = AuthenticatedParticipantId::new(threshold_parameters.participants())?;
let action = LauncherVoteAction::Add(launcher_hash);
let votes = self.tee_state.vote_launcher(action, &participant);
// ...
if votes >= self.threshold()?.value() {
    let added = self.tee_state.add_launcher_image(...);
}
```

`self.threshold()?.value()` is the **current** governance threshold, but `votes` includes stale entries from participants removed in a prior resharing.

**Call site in `vote_remove_launcher_hash`:** [3](#0-2) 

```rust
let votes = self.tee_state.vote_launcher(action, &participant);
let total_participants = threshold_parameters.participants().len() as u64;
if votes >= total_participants {
    let removed = self.tee_state.remove_launcher_image(&launcher_hash);
}
```

Same pattern: `total_participants` is current, `votes` is cumulative.

**`get_remaining_votes` exists but its call site after resharing is unconfirmed:** [4](#0-3) 

The method exists to filter stale votes, but unlike `add_domains_votes` (whose post-resharing cleanup is confirmed by a dedicated sandbox test), no equivalent cleanup path for `LauncherHashVotes` was found in the resharing completion flow.

---

### Impact Explanation

**`vote_add_launcher_hash` — High (attestation authorization bypass):**

A launcher image hash controls which TEE binary is trusted to hold MPC key shares. Adding a malicious launcher hash to the allowed set means a node running that binary can pass `submit_participant_info` attestation checks and be accepted as a valid MPC participant. If the threshold check fires on inflated stale votes, a malicious launcher hash can be whitelisted with fewer than the required number of **current** participants agreeing — bypassing the multi-party governance control that protects key-share confidentiality.

**`vote_remove_launcher_hash` — Medium (safety invariant broken):**

Unanimity is required for removal precisely because removing a hash invalidates attestations of nodes still running that launcher. Stale votes from removed participants can satisfy `votes >= total_participants` without all current participants consenting, causing premature removal and disrupting live nodes' attestation validity, which in turn can trigger `verify_tee` to halt request acceptance.

---

### Likelihood Explanation

The exploit path for `vote_add_launcher_hash` requires:

1. One or more participants vote for a target `launcher_hash` (attacker-controlled or socially engineered).
2. Those participants are subsequently removed via a resharing (a normal governance operation).
3. Their stale votes persist in `LauncherHashVotes` (no confirmed cleanup).
4. A single new/remaining participant votes for the same hash, pushing the raw count ≥ threshold.

Step 4 requires only **one** current participant to vote — far below the intended threshold. In a long-running network with participant turnover, steps 1–3 are routine. The attacker does not need to be above threshold; they only need to have been a participant at some prior point.

---

### Recommendation

1. **Filter before counting:** In `LauncherHashVotes::count_votes` (or in `vote`), accept a `&Participants` argument and skip entries whose key is not in the current participant set — mirroring the pattern already used in `TeeVerifierVotes::retain` and `ProviderVotes::retain`.
2. **Cleanup on resharing:** Call `get_remaining_votes` (already implemented) on `tee_state.launcher_votes` during the resharing completion path (`vote_reshared`), analogous to the confirmed cleanup of `add_domains_votes`.
3. **Audit `CodeHashesVotes`:** The same `count_votes`-without-filter pattern exists in `CodeHashesVotes`; apply the same fix. [5](#0-4)

### Proof of Concept

**Setup:** 5 participants A, B, C, D, E; governance threshold = 3.

1. D and E call `vote_add_launcher_hash(malicious_hash)`.  
   → `count_votes` = 2; threshold = 3; hash **not** added.

2. Governance resharing removes D and E, adds F and G.  
   → Current participants: A, B, C, F, G; threshold = 3.  
   → D and E's entries remain in `vote_by_account` (no confirmed cleanup).

3. F calls `vote_add_launcher_hash(malicious_hash)`.  
   → `count_votes` iterates all entries: D ✓, E ✓, F ✓ → returns **3**.  
   → `3 >= threshold(3)` → `add_launcher_image(malicious_hash)` executes.

**Result:** `malicious_hash` is now in the allowed launcher set. Only **1** current participant (F) voted for it; D and E are no longer participants. The governance threshold of 3 current-participant votes was never actually reached. [6](#0-5) [7](#0-6)

    /// Returns a new `LauncherHashVotes` containing only votes from current participants.
    pub fn get_remaining_votes(&self, participants: &Participants) -> Self {
        let remaining = self
            .vote_by_account
            .iter()
            .filter(|(participant_id, _)| {
                participants.is_participant_given_participant_id(&participant_id.get())
            })
            .map(|(participant_id, vote)| (participant_id.clone(), vote.clone()))
            .collect();
        LauncherHashVotes {
            vote_by_account: remaining,
        }
    }
}
