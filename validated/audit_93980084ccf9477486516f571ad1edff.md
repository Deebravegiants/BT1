### Title
Sub-Transaction ExUnits Excluded from Per-Transaction and Per-Block Limits and Fee Calculation in Dijkstra Era — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The Dijkstra era introduces nested transactions: a top-level `TopTx` may embed an ordered map of sub-transactions (`SubTx`) in its body. The `totExUnits` function, which is reused unchanged from the Alonzo era, only reads redeemers from the **top-level** transaction's witness set. Sub-transaction redeemers are never summed. As a result, the per-transaction ExUnits cap (`maxTxExUnits`), the per-block ExUnits cap (`maxBlockExUnits`), and the minimum-fee script-execution component all ignore the ExUnits declared in sub-transactions. An unprivileged transaction submitter can craft a single top-level transaction carrying many sub-transactions, each with a large Plutus ExUnits budget, and bypass both resource limits while underpaying fees.

---

### Finding Description

**`totExUnits` is blind to sub-transaction redeemers.**

`totExUnits` is defined in `eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Tx.hs`:

```haskell
totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL
```

It folds only over the redeemers attached to the single `Tx` value it receives. In the Dijkstra era a `TopTx` carries sub-transactions in `dtbrSubTransactions :: OMap TxId (Tx SubTx era)` inside the body, each with its own `TxWits` containing its own redeemers. `totExUnits` never traverses `subTransactionsTxBodyL`, so sub-transaction ExUnits are invisible to every caller.

**Three enforcement points all call `totExUnits` directly:**

1. **Per-transaction limit** — `dijkstraUtxoTransition` in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs` line 414–415:
   ```haskell
   {- totExunits tx ≤ maxTxExUnits pp -}
   runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
   ```
   `validateExUnitsTooBigUTxO` calls `totExUnits tx`, which returns only the top-level ExUnits.

2. **Per-block limit** — `dijkstraBbodyTransition` in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs` line 361:
   ```haskell
   Alonzo.validateExUnits @era txs $ pp ^. ppMaxBlockExUnitsL
   ```
   `Alonzo.validateExUnits` is `foldMap totExUnits txs`, again missing sub-transaction ExUnits.

3. **Minimum fee** — `getMinFeeTxUtxo = getConwayMinFeeTxUtxo` in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs` line 141, which calls `getConwayMinFeeTx` → `alonzoMinFeeTx` → `txscriptfee prices (totExUnits tx)`. Sub-transaction script fees are not added to the minimum fee.

**Contrast with the ref-scripts fix.** The codebase already contains `batchNonDistinctRefScriptsSize` in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs` lines 264–277, which correctly sums ref-script sizes across the top-level transaction and all sub-transactions:

```haskell
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum (foldMap' (Sum . txNonDistinctRefScriptsSize utxo)
                       (tx ^. bodyTxL . subTransactionsTxBodyL))
```

No analogous aggregation exists for ExUnits.

---

### Impact Explanation

**Allowed impact matched:** *Attacker-controlled transactions exceed intended validation limits or modify fees outside design parameters (Medium).*

- **ExUnits limit bypass.** A submitter places N sub-transactions each declaring `maxTxExUnits` ExUnits in their redeemers. The top-level transaction declares zero ExUnits. The per-transaction check passes (0 ≤ maxTxExUnits). The block check accumulates only the top-level ExUnits across all transactions in the block, so the block can contain N × maxTxExUnits of actual script work while the check sees only 0. Block validation time is unbounded relative to the `maxBlockExUnits` parameter.

- **Fee underpayment.** The minimum fee's script component is `txscriptfee prices (totExUnits topTx)`. Sub-transaction script fees are zero in this calculation. A submitter pays only for the top-level scripts while the node executes all sub-transaction scripts.

Both effects are directly controlled by the transaction author with no privileged access required.

---

### Likelihood Explanation

Any Dijkstra-era transaction submitter can exploit this. No special role, key, or governance threshold is needed. The attacker only needs to construct a valid top-level transaction with sub-transactions that carry Plutus redeemers. The `DijkstraSubTxBodyRaw` type accepts a `dstbrScriptIntegrityHash` and the sub-transaction witness set accepts redeemers, so the path is fully supported by the serialization format. The CDDL confirms `sub_transaction_body` includes `? 11 : script_data_hash` and `transaction_witness_set` includes `? 5 : redeemers`.

---

### Recommendation

Define a Dijkstra-specific ExUnits aggregation function that mirrors `batchNonDistinctRefScriptsSize`:

```haskell
batchTotExUnits :: (AlonzoEraTxWits era, DijkstraEraTxBody era) => Tx TopTx era -> ExUnits
batchTotExUnits tx =
  totExUnits tx
    <> foldMap totExUnits (tx ^. bodyTxL . subTransactionsTxBodyL)
```

Replace the three call sites:

1. In `dijkstraUtxoTransition`, replace `Alonzo.validateExUnitsTooBigUTxO pp tx` with a Dijkstra-specific variant that uses `batchTotExUnits`.
2. In `dijkstraBbodyTransition`, replace `Alonzo.validateExUnits @era txs` with a variant that folds `batchTotExUnits` over the block's top-level transactions.
3. In `getMinFeeTxUtxo` for `DijkstraEra`, replace `getConwayMinFeeTxUtxo` with a Dijkstra-specific function that passes `batchTotExUnits tx` to the script-fee component.

Additionally, audit whether `batchNonDistinctRefScriptsSize` (already defined but unused in the fee path) should replace `txNonDistinctRefScriptsSize` in `getMinFeeTxUtxo` for the same reason.

### Proof of Concept

1. Construct a Dijkstra-era top-level transaction with an empty redeemer map in its own witness set (`totExUnits topTx = ExUnits 0 0`).
2. Embed K sub-transactions, each with a Plutus redeemer declaring `ExUnits M S` where `M > maxTxExUnits.exUnitsMem` or `S > maxTxExUnits.exUnitsSteps`.
3. Submit the transaction. The check at line 415 of `Utxo.hs` evaluates `pointWiseExUnits (<=) (ExUnits 0 0) maxTxExUnits` → passes.
4. The block check at line 361 of `Bbody.hs` accumulates `ExUnits 0 0` for this transaction → passes even if K × ExUnits M S >> maxBlockExUnits.
5. The fee check at line 373 of `Utxo.hs` computes `minfee` using `alonzoMinFeeTx` with `totExUnits = ExUnits 0 0`, so the script-fee component is zero regardless of sub-transaction ExUnits.
