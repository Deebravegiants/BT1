Audit Report

## Title
Stale `rsETHPrice` denominator in `getRsETHAmountToMint` allows depositors to capture accrued LST yield from existing holders — (File: `contracts/LRTDepositPool.sol`)

## Summary

`LRTDepositPool.getRsETHAmountToMint()` divides a live Chainlink asset price by a cached `rsETHPrice` state variable that is only refreshed when `updateRSETHPrice()` is explicitly called. Because LST prices (e.g., stETH/ETH) increase continuously with staking rewards, any depositor who deposits between two oracle update calls receives more rsETH than their proportional share of protocol assets, permanently diluting the yield owed to existing rsETH holders.

## Finding Description

`getRsETHAmountToMint` at [1](#0-0)  computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

- `lrtOracle.getAssetPrice(asset)` at [2](#0-1)  delegates to a live `IPriceFetcher` (Chainlink), returning the **current** asset/ETH rate.
- `lrtOracle.rsETHPrice` at [3](#0-2)  is a plain storage variable written only at [4](#0-3)  inside `_updateRsETHPrice()`, which is triggered only by explicit calls to `updateRSETHPrice()` or `updateRSETHPriceAsManager()`.

`_beforeDeposit`, called by the permissionless `depositAsset` at [5](#0-4) , invokes `getRsETHAmountToMint` without first refreshing `rsETHPrice`. No guard in the deposit path forces a price sync before minting.

**Exploit path:**
1. `rsETHPrice` is set at time T0 when stETH/ETH = P0.
2. stETH/ETH rises to P1 > P0 (continuous staking rewards); `updateRSETHPrice()` has not been called.
3. Attacker calls `depositAsset(stETH, amount, 0, "")`.
4. Minted rsETH = `amount × P1 / P0` — the numerator uses the fresh price P1 while the denominator uses the stale P0.
5. Attacker receives `amount × (P1/P0 − 1)` excess rsETH relative to their true proportional share.
6. When `updateRSETHPrice()` is eventually called, the new rsETH price is computed over the inflated supply, permanently reducing per-token value for all pre-existing holders.

The `pricePercentageLimit` guard at [6](#0-5)  only restricts how much `rsETHPrice` can jump in a single update call; it does not prevent the stale-denominator mismatch during deposits.

## Impact Explanation

Existing rsETH holders' unclaimed staking yield is permanently transferred to new depositors who time their deposits during the staleness window. When `updateRSETHPrice()` is called post-deposit, the new price is diluted by the excess rsETH supply, and existing holders can never recover the lost yield. This matches **High — Theft of unclaimed yield** in the allowed impact scope.

## Likelihood Explanation

- stETH and other LSTs accrue rewards every block, so the numerator/denominator diverge continuously.
- `updateRSETHPrice()` is called on a periodic keeper schedule, not on every deposit, guaranteeing a persistent staleness window.
- `depositAsset` is fully permissionless — no special role, flash loan, or front-running is required; a depositor passively benefits simply by depositing while the price is stale.
- The attack is repeatable across every keeper interval and scales linearly with deposit size and time elapsed since the last update.

**Likelihood: High.**

## Recommendation

Inside `_beforeDeposit` (or at the start of `depositAsset`), call `lrtOracle.updateRSETHPrice()` before invoking `getRsETHAmountToMint`. This ensures the denominator is computed from the same live asset prices as the numerator, eliminating the mismatch. Alternatively, rewrite `getRsETHAmountToMint` to compute rsETH price on-the-fly from `_getTotalEthInProtocol() / rsethSupply` using current prices rather than reading the cached `rsETHPrice` storage variable.

## Proof of Concept

**State before deposit:**
- Protocol holds 100 stETH; `rsethSupply = 100`; `rsETHPrice = 1.050e18` (set when stETH/ETH = 1.050)
- Chainlink stETH/ETH feed now returns `1.060e18`; `updateRSETHPrice()` has not been called

**Steps:**
1. Alice calls `depositAsset(stETH, 10e18, 0, "")`.
2. `getRsETHAmountToMint` computes: `10e18 × 1.060e18 / 1.050e18 = 10.09523...e18` rsETH minted.
3. Alice receives **10.0952 rsETH** instead of the fair **~10.0 rsETH**.

**After deposit:**
- `rsethSupply = 110.0952`; protocol holds 110 stETH at 1.060 = 116.6 ETH
- `updateRSETHPrice()` called: `newRsETHPrice = 116.6e18 / 110.0952e18 ≈ 1.05908e18`
- Existing 100 rsETH worth `100 × 1.05908 = 105.908 ETH`; fair value = `106.0 ETH`
- **Existing holders permanently lost ~0.092 ETH** of staking yield to Alice

**Foundry fork test plan:** Fork mainnet, set `rsETHPrice` to a value slightly below the current live stETH/ETH price (simulating a stale update), call `depositAsset` as an unprivileged address, assert that `rsethAmountToMint > amount`, then call `updateRSETHPrice()` and assert that the per-token ETH value of the original holders decreased relative to a baseline where `updateRSETHPrice()` was called before the deposit.
