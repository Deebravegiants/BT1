### Title
Single Attested Participant Can Deliver Forged CKD Response for `AppPublicKey` Variant, Bypassing Threshold Authorization — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` contract method performs **no cryptographic verification** of the submitted `CKDResponse` when the pending request uses the `AppPublicKey` (legacy/private) variant. A single Byzantine attested participant — strictly below the reconstruction threshold — can call `respond_ckd` with arbitrary `(big_y, big_c)` values, permanently consuming the user's pending CKD request and delivering forged key material, without the threshold number of participants ever cooperating.

---

### Finding Description

In `crates/contract/src/lib.rs`, the `respond_ckd` function dispatches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO VERIFICATION
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For the `AppPublicKeyPV` (publicly verifiable) variant, `ckd_output_check` enforces the pairing equation `e(C, G2) = e(H(pk, app_id), pk) · e(Y, A2)`, which cryptographically proves the response encodes the correct MPC secret. For the `AppPublicKey` (legacy/private) variant, the arm is a no-op — **any** `CKDResponse` is accepted unconditionally.

After the match, `resolve_yields_for` is called unconditionally:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

`resolve_yields_for` **removes** the pending request entry and resumes all queued yield promises with the supplied bytes. Once called, the request is permanently consumed — there is no retry path. [3](#0-2) 

The `respond` function for ECDSA/EdDSA signatures does not share this flaw — it always verifies the signature cryptographically before resolving: [4](#0-3) 

The `AppPublicKey` variant is still actively supported and documented as the primary (legacy) interface: [5](#0-4) 

---

### Impact Explanation

**Impact: Critical — Confidential key derivation output without the required participant authorization.**

The CKD reconstruction threshold is at least 2 (`MIN_RECONSTRUCTION_THRESHOLD`): [6](#0-5) 

A single attested participant (threshold − 1 below the required quorum) can:

1. Observe a pending `AppPublicKey` CKD request on-chain.
2. Immediately call `respond_ckd` with arbitrary `CKDResponse { big_y: ..., big_c: ... }` before the legitimate threshold computation completes.
3. The contract accepts the forged response, removes the pending request, and delivers the forged `(big_y, big_c)` to the user via the yield-resume callback.

The user receives key material that does not encode `msk · H(pk, app_id)`. Their derived application key is wrong. The pending request is permanently consumed — no re-submission is possible under the same yield ID. This permanently breaks the request lifecycle and delivers a CKD output that was never authorized by the threshold of participants.

---

### Likelihood Explanation

**Likelihood: Medium.**

- The attacker must be a single TEE-attested participant in the running MPC network — a realistic Byzantine-participant threat model that threshold cryptography is specifically designed to tolerate.
- No collusion is required; one participant suffices.
- The attack window is the interval between a CKD request appearing on-chain and the legitimate MPC computation completing. A participant who monitors the chain can race the honest nodes.
- The `AppPublicKey` variant is the documented legacy path and is likely used by existing integrations.

---

### Recommendation

Apply the same cryptographic output check to the `AppPublicKey` variant that is already applied to `AppPublicKeyPV`. For the private variant the check must use the user's public key `A` (the G1 point stored in the request) to verify `e(C − msk·H(pk,app_id), G2) = e(Y, A)`, or equivalently verify the BLS signature embedded in the response against the known MPC public key. If an on-chain pairing check for the private variant is not feasible, the contract should require the response to be submitted by a quorum of participants (threshold-many independent `respond_ckd` calls agreeing on the same response bytes) before resolving the yield, mirroring the off-chain threshold guarantee.

---

### Proof of Concept

1. User calls `request_app_private_key` with `AppPublicKey(pk_g1)` and `domain_id` pointing to the BLS12-381 CKD domain. A pending entry is created in `pending_ckd_requests`.

2. Attacker (a single attested participant) observes the pending request on-chain and immediately calls:

```rust
contract.respond_ckd(
    ckd_request.clone(),
    CKDResponse {
        big_y: Bls12381G1PublicKey([0u8; 48]),  // arbitrary forged Y
        big_c: Bls12381G1PublicKey([0u8; 48]),  // arbitrary forged C
    },
)
```

3. The `AppPublicKey` arm is a no-op; `ckd_output_check` is never called.

4. `resolve_yields_for` removes the pending request and resumes the user's yield with the forged bytes.

5. The user's callback receives `(big_y=0, big_c=0)`. Decryption yields `sig = 0 − a·0 = 0`, which fails BLS verification. The user's CKD request is permanently consumed with no recourse.

The existing sandbox test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` demonstrates that arbitrary `[1u8; 48]` / `[2u8; 48]` byte arrays are accepted without error for the `AppPublicKey` variant, confirming the absence of any response validation: [7](#0-6)

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
```

/// Validates the per-domain reconstruction threshold against the participant
/// count. Universal bound `2 <= t <= n` plus, for `DamgardEtAl`, the
/// honest-majority bound `2t - 1 <= n`.
pub fn validate_domain_threshold(
    domain: &DomainConfig,
    num_participants: u64,
) -> Result<(), Error> {
    let t = domain.reconstruction_threshold.inner();
    if t < MIN_RECONSTRUCTION_THRESHOLD {
        return Err(DomainError::ReconstructionThresholdTooLow.into());
    }
    if t > num_participants {
        return Err(DomainError::ReconstructionThresholdExceedsParticipants {
            threshold: t,
            participants: num_participants,
        }
        .into());
    }
