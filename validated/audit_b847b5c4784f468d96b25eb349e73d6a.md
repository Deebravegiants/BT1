### Title
Missing Subgroup Check on `c_tilde[i]` Enables Range Proof Forgery via Order-2 Element - (`src/cbmpc/zk/zk_pedersen.cpp`)

---

### Summary

`range_pedersen_t::verify` accepts `c_tilde[i] = p-1`, an order-2 element of Z_p* that is outside the safe-prime subgroup of order `p_tag`. Squaring it yields `c_tilde2[i] = 1`, which decouples the commitment from the response and allows an attacker to forge a valid range proof for any value `x` satisfying `q ≤ x < q·2^128`.

---

### Finding Description

The verifier at line 128 checks only that `c_tilde[i]` lies in the open interval `(0, p)`: [1](#0-0) 

The element `p-1 ≡ -1 (mod p)` satisfies `0 < p-1 < p`, so this check passes. Because `p` is a safe prime with `p ≡ 3 (mod 4)`, the element `-1` is NOT a quadratic residue and is NOT in the subgroup of order `p_tag`. The code explicitly acknowledges skipping the subgroup check: [2](#0-1) 

Squaring at line 131 gives `(p-1)^2 mod p = 1`. The verification equation then becomes:

```
∏_i (1 · c^{e_i})^{ρ_i} = g^{∑ d[i]·ρ_i} · h^{∑ f[i]·ρ_i}  (mod p)
```

i.e., `c^{∑ e_i·ρ_i} = g^{∑ d[i]·ρ_i} · h^{∑ f[i]·ρ_i}`. [3](#0-2) 

If the attacker holds a commitment `c = g^x · h^r (mod p)` with `q ≤ x < q·2^128`, they can satisfy this equation exactly by setting:

- `d[i] = x` when `e_i = 1`, `d[i] = 0` when `e_i = 0`
- `f[i] = r` when `e_i = 1`, `f[i] = 0` when `e_i = 0`

Then `∑ d[i]·ρ_i = x·∑ e_i·ρ_i` and `∑ f[i]·ρ_i = r·∑ e_i·ρ_i`, so the equation holds for **all** random `ρ_i` chosen by the verifier. The range check `d[i] < q_with_slack` at line 124 passes because `x < q·2^128 = q_with_slack`. The challenge `e` is a Fiat-Shamir hash of the attacker-controlled `c_tilde` array, so the attacker computes it honestly after fixing all `c_tilde[i] = p-1`. [4](#0-3) 

The same structural flaw exists in `range_pedersen_interactive_t::verify` at line 225, where `c_tilde[i]` is squared in-place without a subgroup check. [5](#0-4) 

---

### Impact Explanation

`range_pedersen_t` is embedded in `paillier_range_exp_slack_t`, which is in turn embedded in `pdl_t`: [6](#0-5) [7](#0-6) 

A Byzantine participant can forge a `paillier_range_exp_slack` proof (and therefore a PDL proof) for a Paillier ciphertext whose plaintext `x` satisfies `q ≤ x < q·2^128`. Since `x mod q` can be chosen freely within `[0, q)` by picking the appropriate multiple of `q`, this constitutes a **key-share substitution**: the attacker can force the effective ECDSA key share to any value of their choosing while the honest verifier accepts the proof. This meets the Critical/High impact bar (forged ZK proof in DKG/signing path, accepted invalid cryptographic output).

---

### Likelihood Explanation

The attack requires no brute force, no threshold collusion, and no side-channel. It is a deterministic, single-message forgery. Any malicious peer that participates in a protocol round invoking `pdl_t::verify` or `paillier_range_exp_slack_t::verify` can execute it immediately. Likelihood is **High**.

---

### Recommendation

Add a subgroup membership check on each `c_tilde[i]` before squaring. The existing helper already implements this: [8](#0-7) 

Insert `if (rv = params.check_safe_prime_subgroup(c_tilde[i])) return rv;` immediately after the `check_open_range` call at line 128 (and the equivalent location in `range_pedersen_interactive_t::verify`). The spec comment claiming subgroup checks can be skipped as an "optimization" is incorrect for soundness — the squaring trick is only sound when `c_tilde[i]` is already in the order-`p_tag` subgroup, because only then does `c_tilde[i]^2` uniquely represent a subgroup element.

const bn_t& p = pedersen_commitment_params_t::get().p;
for (int i = 0; i < range_pedersen_t::t; i++)
    forged.c_tilde[i] = p - 1;

// 3. Compute the Fiat-Shamir challenge honestly
forged.e = crypto::ro::hash_string(p, q, g, h, c, forged.c_tilde, sid, aux).bitlen128();

    if (forged.e.get_bit(i)) { forged.d[i] = x; forged.f[i] = r; }
    else                     { forged.d[i] = 0; forged.f[i] = 0; }
}

// 5. Verify accepts despite x >= q
assert(forged.verify(q, c, sid, aux) == SUCCESS);  // passes — vulnerability confirmed
