### Title
Missing `curve.check()` on Public Key Q in `schnorr_2p::blob_to_key()` Allows Invalid EC Point to Enter BIP340 Signing Protocol — (`File: src/cbmpc/api/schnorr2pc.cpp`)

### Summary

The `blob_to_key()` function in `src/cbmpc/api/schnorr2pc.cpp` parses the combined public key `Q` from the key blob using `from_bin()` but omits the `curve.check()` call that every other key-blob deserializer in the codebase performs. This is the direct structural analog to the report's bug class: a validation function exists and is called everywhere else, but is silently absent in one critical path. An attacker who can supply a crafted key blob (e.g., by tampering with unprotected blob storage) can inject an invalid or attacker-chosen public key into the BIP340 Schnorr signing protocol, causing the protocol to produce signatures that verify under the wrong key.

### Finding Description

**Root cause — dead validation at the public API boundary.**

`blob_to_key()` in `schnorr2pc.cpp` (lines 29–41):

```cpp
static error_t blob_to_key(const key_blob_v1_t& blob, coinbase::mpc::schnorr2p::key_t& key) {
  if (blob.role > 1) return coinbase::error(E_FORMAT, "invalid key blob role");
  if (static_cast<curve_id>(blob.curve) != curve_id::secp256k1)
    return coinbase::error(E_FORMAT, "invalid key blob curve");

  key.role = static_cast<coinbase::mpc::party_t>(static_cast<int32_t>(blob.role));
  key.curve = coinbase::crypto::curve_secp256k1;
  const auto& q = key.curve.order();
  if (!q.is_in_range(blob.x_share)) return coinbase::error(E_FORMAT, "invalid key blob");
  key.x_share = blob.x_share;

  return key.Q.from_bin(key.curve, blob.Q_compressed);  // ← no curve.check() called
}
``` [1](#0-0) 

Every other key-blob deserializer in the codebase calls `curve.check()` after `from_bin()`:

- `ecdsa2pc.cpp` `blob_to_key()`:
  ```cpp
  if (const error_t rv = key.Q.from_bin(curve, blob.Q_compressed)) return rv;
  if (curve.check(key.Q)) return coinbase::error(E_FORMAT, "invalid key blob");
  ``` [2](#0-1) 

- `eddsa2pc.cpp` `blob_to_key()`:
  ```cpp
  error_t rv = key.Q.from_bin(key.curve, blob.Q_compressed);
  if (rv) return coinbase::error(rv, "invalid key blob");
  if (key.curve.check(key.Q)) return coinbase::error(E_FORMAT, "invalid key blob");
  ``` [3](#0-2) 

- `ecdsa_mp.cpp` `deserialize_key_blob()` and `eddsa_mp.cpp` `deserialize_key_blob()` both call `curve.check(Q)` after `from_bin()`. [4](#0-3) [5](#0-4) 

`ecurve_t::check()` enforces four invariants that `from_bin()` alone does not guarantee:

```cpp
error_t ecurve_t::check(const ecc_point_t& point) const {
  if (!point.valid()) return crypto::error("EC-point invalid");
  if (point.get_curve() != *this) return crypto::error("EC-point of wrong curve");
  if (!point.is_in_subgroup()) return crypto::error("EC-point is not on curve");
  if (!thread_local_store_allow_ecc_infinity)
    if (point.is_infinity()) return crypto::error("EC-point is infinity");
  return SUCCESS;
}
``` [6](#0-5) 

The missing call means:
1. **Infinity accepted as Q.** If `from_bin` returns success for an infinity encoding (implementation-dependent), the signing protocol proceeds with the identity element as the public key.
2. **No `x_share * G == Q` binding.** All MP deserializers verify `blob.x_share * G == Qi_self`. `schnorr_2p` does not. An attacker can craft a blob where `x_share` is valid but `Q` is an attacker-chosen point, causing the protocol to produce signatures that verify under the attacker's key.

**Entry path:**

`cbmpc_schnorr_2p_sign` (C ABI) → `coinbase::api::schnorr_2p::sign()` → `deserialize_key_blob()` → `blob_to_key()` → `key.Q.from_bin(...)` (no `curve.check()`). [7](#0-6) 

The same path is taken by `refresh()`, `get_public_key_compressed()`, `extract_public_key_xonly()`, `get_public_share_compressed()`, and `detach_private_scalar()`.

### Impact Explanation

An attacker who can write to the key blob storage (which `SECURE_USAGE.md` explicitly notes is unprotected by the library) can craft a blob with an attacker-chosen `Q`. The signing protocol will then produce BIP340 Schnorr signatures that verify under the attacker's key rather than the legitimate key. This constitutes a key-substitution attack: the honest party signs a message believing it is signing under their real key, but the resulting signature is valid under a different key entirely. This falls under the "High" impact category: public API reachable validation bypass in signing creates accepted invalid cryptographic output.

### Likelihood Explanation

The key blob is explicitly documented as unprotected by the library. Applications that do not wrap blobs in AEAD (as recommended but not enforced) expose this path to any attacker with write access to blob storage. The missing check is a single-line omission that is inconsistent with every other deserializer in the same codebase, making it likely an oversight rather than intentional design.

### Recommendation

Add `curve.check(key.Q)` immediately after the `from_bin` call in `blob_to_key()` in `src/cbmpc/api/schnorr2pc.cpp`, matching the pattern used in `ecdsa2pc.cpp` and `eddsa2pc.cpp`:

```cpp
if (const error_t rv = key.Q.from_bin(key.curve, blob.Q_compressed)) return rv;
if (key.curve.check(key.Q)) return coinbase::error(E_FORMAT, "invalid key blob");
return SUCCESS;
```

### Proof of Concept

1. Generate a legitimate `schnorr_2p` key blob via `dkg()`.
2. Deserialize the blob, replace `Q_compressed` with the compressed encoding of an attacker-controlled secp256k1 point `Q'` (any valid point), re-serialize.
3. Call `coinbase::api::schnorr_2p::sign()` with the tampered blob. `blob_to_key()` will accept `Q'` without error because `curve.check()` is never called.
4. The signing protocol runs with `Q'` as the public key. The resulting signature verifies under `Q'`, not the legitimate key.
5. Contrast: supplying the same tampered blob to `coinbase::api::ecdsa_2p::sign()` or `coinbase::api::eddsa_2p::sign()` returns `E_FORMAT` because those paths call `curve.check()`. [1](#0-0) [2](#0-1) [6](#0-5)
