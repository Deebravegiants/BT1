Audit Report

## Title
Pool Admin Can Set Uncapped Per-Bin Additional Fees, Bypassing the Protocol Fee Cap System - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

## Summary
`setPoolBinAdditionalFees` allows the pool admin to set per-bin additional spread fees (`addFeeBuyE6`, `addFeeSellE6`) with no upper-bound validation. Every other fee-setting path in the factory enforces a hard cap against `maxAdminSpreadFeeE6` / `HARD_MAX_SPREAD_FEE_E6`, but this path is silently exempt. A pool admin can push the effective per-bin fee above the protocol's hard ceiling of 20%, causing traders to receive fewer tokens than the protocol guarantees.

## Finding Description
The factory enforces a two-tier fee cap hierarchy. The hard constant `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%) is immutable, and the owner-settable `maxAdminSpreadFeeE6` is bounded by it. The pool-admin path `setPoolAdminFees` enforces `if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh()` before applying any fee change.

The parallel pool-admin path `setPoolBinAdditionalFees` contains no such check:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

The parameters are typed `uint16` (max 65,535 ≈ 6.55% in E6 units). Per the interface NatSpec, these fees are applied **on top of the base spread**. `BinDataLibrary` packs them as raw `uint16` with no validation. With a base spread already at `HARD_MAX_SPREAD_FEE_E6 = 200_000`, a pool admin can add up to 65,535 more, yielding an effective fee of 265,535 E6 (≈26.55%) — 33% above the hard ceiling.

## Impact Explanation
This is an admin-boundary break: the pool admin role is explicitly bounded by the factory's cap system, yet `setPoolBinAdditionalFees` provides an uncapped escape hatch. Traders swapping through an affected bin pay a higher effective spread fee than the protocol's hard cap permits, directly reducing their token output. The cap invariant (`total spread fee ≤ HARD_MAX_SPREAD_FEE_E6`) is broken for any pool whose admin sets a non-zero bin additional fee. This constitutes a direct, quantifiable loss of swap output for users.

## Likelihood Explanation
Any pool admin — including one who turns adversarial post-deployment — can call `setPoolBinAdditionalFees` at any time with `addFeeBuyE6 = type(uint16).max` (65,535). There is no timelock, no owner approval, and no cap check. The call requires only the `onlyPoolAdmin` modifier, which is the role whose fee-setting power is supposed to be bounded by the factory.

## Recommendation
Add the same cap enforcement to `setPoolBinAdditionalFees` that exists in `setPoolAdminFees`. At minimum, revert if either parameter exceeds `maxAdminSpreadFeeE6`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

A stricter fix would define a separate `HARD_MAX_BIN_ADDITIONAL_FEE_E6` constant ensuring the sum of base spread fee and bin additional fee cannot exceed `HARD_MAX_SPREAD_FEE_E6`.

## Proof of Concept
1. Factory owner deploys a pool with `maxAdminSpreadFeeE6 = 200_000` and `spreadFeeE6 = 200_000`.
2. Pool admin calls:
   ```solidity
   factory.setPoolBinAdditionalFees(pool, 0, 65_535, 65_535);
   ```
   No revert — `setPoolBinAdditionalFees` has no cap check.
3. A trader swaps through bin 0. The effective spread fee is `200_000 + 65_535 = 265_535 E6` (≈26.55%), exceeding `HARD_MAX_SPREAD_FEE_E6 = 200_000`.
4. The trader receives fewer tokens than the protocol's fee ceiling permits, with no on-chain mechanism to prevent or detect the violation.
