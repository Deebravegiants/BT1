Audit Report

## Title
Missing L2 Sequencer Uptime Check Enables Stale-Price Exploitation on Deposit - (File: contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol)

## Summary

`ChainlinkOracleForRSETHPoolCollateral.getRate()` calls Chainlink's `latestRoundData()` without verifying the L2 sequencer is online or that the returned price is fresh. No sequencer uptime feed integration exists anywhere in the protocol contracts. Because `RSETHPoolNoWrapper` (explicitly documented for Arbitrum/Unichain), `RSETHPool`, `RSETHPoolV3`, and related variants all consume this oracle to compute how much rsETH/wrsETH to mint per unit of collateral, an attacker can deposit at a stale pre-downtime price immediately after sequencer resumption and receive more rsETH than their collateral is worth.

## Finding Description

`ChainlinkOracleForRSETHPoolCollateral.getRate()` fetches the price as follows:

```solidity
(uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
    AggregatorV3Interface(oracle).latestRoundData();

if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [1](#0-0) 

The three existing checks are insufficient for L2 deployments:
- `answeredInRound < roundID`: passes because the last completed round before sequencer downtime is technically complete.
- `timestamp == 0`: passes because the round has a non-zero timestamp.
- `ethPrice <= 0`: passes because the stale price is a valid positive number.

No sequencer uptime feed is integrated anywhere in the protocol contracts — a grep for `sequencer|uptime|L2_SEQUENCER|ArbitrumSequencer|gracePeriod` returns zero matches in the `contracts/` directory. Additionally, the `updatedAt` return value (captured as `timestamp`) is never compared against `block.timestamp`, so there is no staleness window check either.

This oracle is consumed directly in the deposit minting path. In `RSETHPool.viewSwapRsETHAmountAndFee`:

```solidity
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [2](#0-1) 

The same pattern is present in `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, and `RSETHPoolNoWrapper`. `RSETHPoolNoWrapper` is explicitly documented as the pool for Arbitrum and Unichain: [3](#0-2) 

## Impact Explanation

**Critical — Direct theft of user funds / protocol insolvency.**

When the L2 sequencer resumes after downtime during which the collateral token's price dropped, the Chainlink feed still returns the pre-downtime (inflated) price. An attacker depositing collateral at this stale rate receives `rsETHAmount = amountAfterFee * stalePriceHigh / rsETHRate`, which is proportionally more rsETH than the collateral's actual current value. This directly dilutes existing rsETH/wrsETH holders' claims and can render the pool undercollateralized relative to the minted supply — matching the "Direct theft of user funds" and "Protocol insolvency" critical impact classes.

## Likelihood Explanation

**Medium.** Arbitrum has experienced documented sequencer outages. The attack requires no special privileges — `deposit()` is a public, permissionless function. Arbitrum's delayed inbox mechanism allows an attacker to pre-stage a deposit transaction during downtime, guaranteeing inclusion at the stale price the moment the sequencer resumes. The attacker only needs to monitor sequencer status and act immediately on resumption.

## Recommendation

1. Integrate Chainlink's L2 Sequencer Uptime Feed in `ChainlinkOracleForRSETHPoolCollateral.getRate()`. Revert if the sequencer is reported as down (`answer != 0`).
2. Enforce a minimum grace period (e.g., 3600 seconds) after sequencer recovery before accepting oracle data.
3. Add a `updatedAt` staleness check: `if (block.timestamp - timestamp > heartbeatInterval) revert StalePrice()`.

```solidity
(, int256 seqAnswer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
if (seqAnswer != 0) revert SequencerDown();
if (block.timestamp - startedAt < GRACE_PERIOD) revert GracePeriodNotOver();
```

## Proof of Concept

1. Deploy a fork of the Arbitrum mainnet at a block where the Chainlink collateral feed last updated before a sequencer outage.
2. Advance `block.timestamp` to simulate sequencer downtime during which the collateral token's market price drops (e.g., from 1.0 ETH to 0.7 ETH).
3. Call `ChainlinkOracleForRSETHPoolCollateral.getRate()` — it returns the pre-downtime price (1.0 ETH). All three existing checks pass.
4. Call `RSETHPoolNoWrapper.deposit(token, largeAmount)` as an unprivileged attacker. The minting formula `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate` uses the stale 1.0 ETH rate.
5. Assert that the attacker received ~43% more rsETH than the collateral's current market value warrants, while the pool's backing is now insufficient to cover all outstanding rsETH claims. [1](#0-0) [4](#0-3)
