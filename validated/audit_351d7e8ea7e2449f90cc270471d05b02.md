Audit Report

## Title
Cross-Token Arbitrage via Shared wrsETH Supply Across Untracked Per-Token Reserves in `_withdraw` — (File: contracts/L2/RsETHTokenWrapper.sol)

## Summary
`RsETHTokenWrapper` mints and burns wrsETH 1:1 against any token in `allowedTokens` with no per-token reserve accounting. Because the total wrsETH supply is shared across all allowed altRsETH tokens, a user who deposits a lower-value allowed token implicitly acquires a claim on the reserves of any other allowed token. When two or more tokens are allowed and a price discrepancy exists, an attacker can deposit the cheaper token and withdraw the more valuable one, draining honest depositors' reserves.

## Finding Description
`_deposit` (lines 134–141) accepts any `allowedTokens[_asset]` token, transfers it in, and mints `_amount` wrsETH. `_withdraw` (lines 120–128) accepts any `allowedTokens[_asset]` token, burns `_amount` wrsETH, and transfers that token out. Neither function tracks how much of each specific token backs the outstanding wrsETH supply. The single `allowedTokens` mapping (line 24) and the `reinitialize` function (lines 47–49) confirm the contract is designed to hold multiple distinct altRsETH tokens simultaneously. `addAllowedToken` (lines 174–176) further allows governance to expand the set at any time. With two tokens allowed, the invariant `tokenA_balance + tokenB_balance == totalSupply()` holds globally, but there is no invariant `tokenX_balance >= tokenX_deposits` per token. Any wrsETH holder — regardless of which token they deposited — can redeem against whichever token's balance is largest.

## Impact Explanation
**Critical — Direct theft of user funds.** An attacker can atomically drain the contract's entire balance of the higher-value altRsETH token by depositing an equivalent nominal amount of the lower-value token. Honest depositors of the higher-value token are left with only the cheaper token and cannot recover their original asset, constituting permanent loss of funds at rest.

## Likelihood Explanation
**Medium.** The multi-token design is confirmed live (`reinitialize(2)` already executed). A price discrepancy between two bridge-wrapped rsETH tokens is a realistic and historically observed condition during bridge incidents, bridge version deprecations, or liquidity crises. The attack requires no privileges, no flash loan, and no victim interaction — only holding the cheaper token and calling the two public functions `deposit` and `withdraw`. It is fully repeatable until the cheaper token's balance is exhausted or the discrepancy closes.

## Recommendation
Introduce explicit per-token reserve tracking:

```solidity
mapping(address => uint256) public tokenReserves;
```

In `_deposit`, increment `tokenReserves[_asset] += _amount`. In `_withdraw`, require `tokenReserves[_asset] >= _amount` before decrementing and transferring. This ensures withdrawals of a given token are bounded by deposits of that same token, eliminating cross-token claims. Alternatively, restrict the contract to a single canonical token at any time, or enforce that each user's wrsETH balance is tagged to the token they deposited.

## Proof of Concept
**Setup:** Two tokens allowed — `tokenA` (depegged, 0.9 ETH/unit) and `tokenB` (canonical, 1.0 ETH/unit). Contract holds 1000 `tokenB` from honest depositors; `totalSupply()` of wrsETH = 1000.

1. Attacker acquires 1000 `tokenA` (~900 ETH cost).
2. Attacker calls `deposit(tokenA, 1000)`:
   - `allowedTokens[tokenA]` → true, passes.
   - Contract receives 1000 `tokenA`, mints 1000 wrsETH to attacker.
   - `totalSupply()` = 2000; contract holds 1000 `tokenA` + 1000 `tokenB`.
3. Attacker calls `withdraw(tokenB, 1000)`:
   - `allowedTokens[tokenB]` → true, passes.
   - Burns 1000 wrsETH, transfers 1000 `tokenB` to attacker.
4. Attacker net: spent ~900 ETH, received 1000 `tokenB` worth ~1000 ETH. Profit ~100 ETH.
5. Contract now holds only 1000 `tokenA`; original `tokenB` depositors cannot recover `tokenB`.

**Foundry invariant test plan:** Deploy `RsETHTokenWrapper` with two mock ERC20 tokens at different prices. Add both via `addAllowedToken`. Seed the contract with `tokenB` deposits. Run invariant: `tokenB.balanceOf(wrapper) >= sum of tokenB deposits - sum of tokenB withdrawals`. The invariant breaks immediately when the attacker executes steps 2–3 above.
