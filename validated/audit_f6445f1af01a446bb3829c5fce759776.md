### Title
Sub-Transaction ExUnits Bypass `maxTxExUnits` and `maxBlockExUnits` Limits, Enabling Underpaid Script Execution - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

---

### Summary

In the Dijkstra era, the `totExUnits` function only sums ExUnits from the top-level transaction's redeemers. Sub-transactions embedded in a Dijkstra top-level transaction carry their own witnesses and redeemers with independent ExUnits budgets, but these are never counted against `maxTxExUnits` or `maxBlockExUnits`. An unprivileged transaction sender can craft a top-level transaction whose top-level ExUnits are within protocol limits while embedding sub-transactions whose combined ExUnits budgets far exceed those limits, causing nodes to execute expensive Plutus scripts without paying the corresponding fees and without triggering any protocol-level rejection.

---

### Finding Description

**Root cause — `totExUnits` is blind to sub-transaction redeemers**

`totExUnits` is defined in `eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Tx.hs`:

```haskell
totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL
``` [1](#0-0) 

This reads only from the top-level transaction's `witsTxL`. In the Dijkstra era, a top-level `DijkstraTx` carries `dtWits` (top-level witnesses) and a `dtbrSubTransactions` map of sub-transactions, each of which carries its own `dstWits :: TxWits era` containing independent redeemers with their own ExUnits budgets. [2](#0-1) 

Sub-transactions are annotated with `dsastPlutusScriptsWithContext`, confirming they execute Plutus scripts independently: [3](#0-2) 

**Per-transaction ExUnits check misses sub-transactions**

In `dijkstraUtxoTransition`, the only ExUnits check is:

```haskell
{- totExunits tx ≤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
``` [4](#0-3) 

`validateExUnitsTooBigUTxO` calls `totExUnits tx`, which only reads the top-level tx's redeemers: [5](#0-4) 

Sub-transaction redeemers are never passed to this check.

**Per-block ExUnits check misses sub-transactions**

In `dijkstraBbodyTransition`:

```haskell
Alonzo.validateExUnits @era txs $ pp ^. ppMaxBlockExUnitsL
``` [6](#0-5) 

`validateExUnits` uses `foldMap totExUnits txs`, summing only top-level ExUnits across all transactions in the block: [7](#0-6) 

**Fee calculation also misses sub-transaction ExUnits**

`getConwayMinFeeTx` (reused by Dijkstra) calls `alonzoMinFeeTx`, which prices only `totExUnits tx`:

```haskell
alonzoMinFeeTx pp tx =
  ... <+> txscriptfee (pp ^. ppPricesL) allExunits
  where allExunits = totExUnits tx
``` [8](#0-7) 

Sub-transactions have no fee field of their own; their script execution cost is entirely unpriced.

**Contrast with reference-script size accounting**

The Dijkstra era correctly accounts for sub-transactions when computing reference script sizes via `batchNonDistinctRefScriptsSize`: [9](#0-8) 

No equivalent aggregation exists for ExUnits.

---

### Impact Explanation

An attacker submits a Dijkstra top-level transaction whose top-level redeemers declare `ExUnits {0, 0}` (passing the `maxTxExUnits` check) while embedding N sub-transactions each declaring `ExUnits { exUnitsMem = 14_000_000, exUnitsSteps = 10_000_000_000 }` (the current `maxTxExUnits` values). The block-level check sums only the top-level ExUnits (zero), so the block also passes `maxBlockExUnits`. Nodes must execute all sub-transaction Plutus scripts up to their declared budgets, consuming N × `maxTxExUnits` of CPU and memory per top-level transaction, while the attacker pays fees priced at zero ExUnits. This allows attacker-controlled transactions to exceed intended validation limits and to underpay fees relative to actual script execution cost, matching the **Medium** allowed impact: *"Attacker-controlled transactions… exceed intended validation limits or modify fees… outside design parameters."*

---

### Likelihood Explanation

Any unprivileged transaction sender can construct this transaction. No special keys, governance access, or privileged roles are required. The Dijkstra era is the next production era after Conway; the attack surface is reachable as soon as the era activates. The construction is straightforward: embed sub-transactions with expensive always-succeeding Plutus scripts and large ExUnits budgets. The top-level tx passes all existing checks.

---

### Recommendation

Extend `totExUnits` (or introduce a Dijkstra-specific variant) to recursively sum ExUnits across all sub-transactions:

```haskell
totBatchExUnits :: (EraTx era, AlonzoEraTxWits era, DijkstraEraTxBody era)
                => Tx TopTx era -> ExUnits
totBatchExUnits tx =
  totExUnits tx
    <> foldMap totExUnits (tx ^. bodyTxL . subTransactionsTxBodyL)
```

Use `totBatchExUnits` in:
1. `dijkstraUtxoTransition` for the per-transaction ExUnits check
2. `dijkstraBbodyTransition` for the per-block ExUnits aggregation
3. `getConwayMinFeeTx` / `alonzoMinFeeTx` for fee calculation in the Dijkstra era

This mirrors the existing pattern for reference script size, where `batchNonDistinctRefScriptsSize` already aggregates across sub-transactions.

---

### Proof of Concept

1. Deploy an always-succeeding Plutus V3 script to a UTxO (sub-transaction spending purpose).
2. Construct a Dijkstra top-level transaction:
   - Top-level redeemers: empty (zero ExUnits declared at top level).
   - Sub-transactions: embed K sub-transactions, each spending a script-locked UTxO with a redeemer declaring `ExUnits { exUnitsMem = 14_000_000, exUnitsSteps = 10_000_000_000 }`.
3. Submit the transaction. Observe:
   - `validateExUnitsTooBigUTxO` passes (top-level ExUnits = 0 ≤ `maxTxExUnits`).
   - `validateExUnits` in BBODY passes (block total = 0 ≤ `maxBlockExUnits`).
   - Fee check passes (fee priced at 0 ExUnits).
   - Nodes execute K × `maxTxExUnits` worth of Plutus computation.
4. Pack multiple such transactions into a block to multiply the effect across the block limit.

The attacker pays only the byte-size fee for the transaction, while forcing nodes to execute script budgets totalling K × N × `maxTxExUnits` per block, where K is sub-transactions per top-level tx and N is top-level txs per block.
