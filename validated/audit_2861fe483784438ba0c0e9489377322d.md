Looking at the full `do_update` flow and deposit handling in `crates/contract/src/update.rs` and `crates/contract/src/lib.rs`.

**Key facts established:**

1. `propose_update` collects a deposit calculated as `env::storage_byte_cost() * bytes_used` — a storage deposit, not a fee. [1](#0-0) 

2. `UpdateEntry` stores only `update` and `bytes_used` — **no proposer account ID is recorded**. [2](#0-1) 

3. `do_update` removes the winning entry, then calls `self.entries.clear()` and `self.vote_by_participant.clear()` — wiping all non-winning proposals with no refund logic anywhere. [3](#0-2) 

4. `propose_update` only refunds the *excess* over the required deposit; the required portion is kept by the contract with no mechanism to return it later. [4](#0-3) 

5. The existing test `test_proposed_updates_do_update_clears_all_state` explicitly confirms all entries are wiped, but asserts nothing about deposit refunds — the deposit loss is a known side-effect treated as correct behavior. [5](#0-4)

### Title
Non-Winning Proposal Deposits Permanently Lost on `do_update` — (`crates/contract/src/update.rs`)

### Summary
When `do_update` executes a winning proposal, it unconditionally clears all other pending proposals via `self.entries.clear()`. Because `UpdateEntry` does not record the proposer's account ID and no refund promise is emitted for cleared entries, the storage deposits paid by non-winning proposers are permanently absorbed into the contract balance with no recovery path.

### Finding Description
`propose_update` requires a deposit equal to `env::storage_byte_cost() * bytes_used` — a NEAR storage deposit intended to cover the on-chain storage cost of the proposal. When a competing proposal wins and `do_update` is called:

- Line 196: `self.entries.remove(id)` removes the winning entry.
- Lines 199–200: `self.entries.clear()` and `self.vote_by_participant.clear()` wipe all remaining proposals and votes.

No code path iterates over the cleared entries to issue `Promise::transfer` refunds to their proposers. The `UpdateEntry` struct does not even store the proposer's `AccountId`, making a refund structurally impossible without a redesign. The freed storage NEAR is released back to the contract's own balance and stays there permanently. [3](#0-2) [6](#0-5) 

### Impact Explanation
Every non-winning proposer permanently loses their storage deposit. For a contract binary update the deposit is `code.len() + ~8 KB overhead` multiplied by `env::storage_byte_cost()` (~10 yoctoNEAR/byte on mainnet). A 1 MB contract binary yields a required deposit of roughly 10 NEAR. With multiple competing proposals this loss scales linearly. The NEAR is irrecoverable: there is no admin withdrawal, no expiry-based refund, and no re-proposal mechanism that returns the original deposit.

### Likelihood Explanation
This triggers in normal governance operation whenever two or more participants propose different updates and one reaches threshold. It does not require malicious intent: a participant who proposes a config update while another participant's contract-binary proposal is pending will lose their deposit the moment the binary proposal wins. The sandbox test `test_propose_update_contract_many` and `test_propose_update_config` both exercise multi-proposal scenarios without any deposit-refund assertions, confirming the loss is reachable in standard usage. [7](#0-6) 

### Recommendation
1. Add a `proposer: AccountId` field to `UpdateEntry` at proposal time.
2. In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and emit `Promise::new(entry.proposer).transfer(required_deposit(entry.bytes_used))` for each.
3. Alternatively, maintain a separate `IterableMap<UpdateId, (AccountId, NearToken)>` tracking proposer-to-deposit, and drain it with refunds inside `do_update`.

### Proof of Concept
```rust
// Unit-level proof (no sandbox needed):
// 1. Participant A proposes update_A, paying deposit_A = required_deposit(bytes_used(&update_A))
// 2. Participant B proposes update_B, paying deposit_B
// 3. Threshold participants vote for update_B → do_update(&update_B_id) is called
// 4. entries.clear() silently drops update_A's entry
// 5. No Promise::transfer(deposit_A) is ever emitted to A's account
// 6. contract_balance increases by deposit_A permanently

let mut proposed_updates = ProposedUpdates::default();
let update_a = Update::Contract(vec![0u8; 100_000]); // ~100KB binary
let deposit_a = ProposedUpdates::required_deposit(&update_a); // ~1 NEAR
let id_a = proposed_updates.propose(update_a);

let update_b = Update::Config(some_config());
let id_b = proposed_updates.propose(update_b);

// threshold votes for B
proposed_updates.do_update(&id_b, Gas::from_tgas(100));

// id_a entry is gone, deposit_a is gone, no refund promise was created
assert!(!proposed_updates.entries.contains_key(&id_a)); // passes
// assert: no Promise::transfer to proposer_a was emitted — confirmed by absence of any such code
```
