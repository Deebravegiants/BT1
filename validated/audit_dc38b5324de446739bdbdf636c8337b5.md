### Title
Caller-Agnostic `VerifyForeignTransactionRequest` Key Allows Any Party to Receive Foreign-Chain Attestation Signatures — (File: `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

### Summary
`VerifyForeignTransactionRequest`, used as the pending-request map key, contains no caller identity. Any unprivileged account can submit the same foreign-chain transaction request as a legitimate bridge user and receive the identical MPC-signed attestation via the fan-out mechanism, breaking the request-lifecycle accounting invariant and enabling unauthorized bridge execution.

### Finding Description

**Contrast with `SignatureRequest`:**

`SignatureRequest` binds to a specific caller by computing a `tweak` from `(predecessor_id, path)`:

```rust
// crates/near-mpc-crypto-types/src/sign.rs:118-125
impl SignatureRequest {
    pub fn new(domain: DomainId, payload: Payload, predecessor_id: &AccountId, path: &str) -> Self {
        let tweak = crate::kdf::derive_tweak(predecessor_id, path);
        SignatureRequest { domain_id: domain, tweak, payload }
    }
}
```

`VerifyForeignTransactionRequest` contains only `(request, domain_id, payload_version)` — no caller identity, no derivation path, no tweak:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs:124-128
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
```

**Conversion strips caller identity:**

`args_into_verify_foreign_tx_request` converts the user-supplied args to the stored key without including `env::predecessor_account_id()`:

```rust
// crates/contract/src/dto_mapping.rs:840-848
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
```

**Fan-out to all callers under the same key:**

`push_pending_yield` appends any caller's yield index to the same queue entry, and `resolve_yields_for` drains the entire queue on a single `respond_verify_foreign_tx` call, delivering the same `VerifyForeignTransactionResponse` to every caller who submitted the same `(request, domain_id, payload_version)` tuple:

```rust
// crates/contract/src/pending_requests.rs:50-59
let queue = requests.entry(request).or_default();
// ...
queue.push(YieldIndex { data_id });
```

This is explicitly tested and confirmed in `verify_foreign_transaction__should_queue_duplicates_from_different_callers` (lib.rs:3208–3298), where Alice and Bob both receive the same response.

**Root key used — no per-caller derivation:**

The MPC node builds the signing request with a zero tweak:

```rust
// crates/node/src/providers/verify_foreign_tx/sign.rs:39-47
Ok(SignatureRequest {
    id: request.id,
    receipt_id: request.receipt_id,
    payload: Payload::Ecdsa(payload_bytes),
    tweak: Tweak::new([0u8; 32]),   // root key, no derivation
    ...
})
```

`respond_verify_foreign_tx` verifies against the root public key:

```rust
// crates/contract/src/lib.rs:728-734
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,   // root key, no tweak applied
)
.is_ok()
```

**Design document deviation:**

`docs/foreign-chain-transactions.md` (lines 98–110, 254–283) explicitly planned for a `derivation_path` field in `VerifyForeignTransactionRequestArgs` and a `tweak` field in `VerifyForeignTransactionRequest`, derived with a foreign-tx-specific prefix (`"near-mpc-recovery v0.1.0 foreign-tx epsilon derivation:"`). The actual production structs omit both fields entirely, meaning the attestation is not bound to any specific caller.

**Response carries no caller binding:**

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs:147-150
pub struct VerifyForeignTransactionResponse {
    pub payload_hash: Hash256,
    pub signature: SignatureResponse,
}
```

The response contains only `(payload_hash, signature)`. No caller account ID, no derivation path, no tweak. Any party who receives this response can present it to a bridge contract.

### Impact Explanation

The `verify_foreign_transaction` feature is the MPC network's primary mechanism for the Omnibridge inbound flow (foreign chain → NEAR). A bridge contract that gates fund releases on a valid `VerifyForeignTransactionResponse` cannot distinguish between the legitimate requester and an attacker who submitted the same request and received the fanned-out response. Because the attestation signature is over `SHA-256(borsh(ForeignTxSignPayload{request, extracted_values}))` with no caller binding, any party holding the response can claim the bridged funds. This breaks the request-lifecycle accounting invariant — the attestation should be bound to the party who initiated the bridge operation — and enables unauthorized bridge execution and double-spend conditions.

### Likelihood Explanation

The attack requires no special privileges. The attacker only needs to:
1. Monitor the NEAR blockchain for `verify_foreign_transaction` calls (all call arguments are public on-chain).
2. Submit the identical `VerifyForeignTransactionRequestArgs` before the MPC nodes respond.
3. Receive the fanned-out `VerifyForeignTransactionResponse`.

The Omnibridge inbound flow is the explicitly stated primary use case in the design document, making this a realistic and high-value attack target.

### Recommendation

1. Restore the `derivation_path` field to `VerifyForeignTransactionRequestArgs` as described in the design document.
2. Derive a `tweak` from `(predecessor_id, derivation_path)` using the foreign-tx-specific prefix inside `args_into_verify_foreign_tx_request`, and include it in `VerifyForeignTransactionRequest` so the map key is caller-specific.
3. Pass the tweak to the MPC node signing request (replacing `Tweak::new([0u8; 32])`) and verify the signature against the derived key in `respond_verify_foreign_tx`, mirroring the `respond` flow.
4. Include the derived public key or caller account ID in `VerifyForeignTransactionResponse` so bridge contracts can verify the intended recipient without relying solely on the foreign transaction data.

### Proof of Concept

```
1. Alice calls verify_foreign_transaction({
       request: Bitcoin(tx_id=[7u8;32], confirmations=2, extractors=[BlockHash]),
       domain_id: D,
       payload_version: V1
   }) with 1 yoctoNEAR deposit.

2. Bob (attacker) observes Alice's pending request on-chain and calls
   verify_foreign_transaction with the identical arguments.

3. Both Alice and Bob are queued under the same VerifyForeignTransactionRequest key
   (no caller identity in the key).

4. MPC nodes call respond_verify_foreign_tx, which drains the entire queue,
   delivering the same VerifyForeignTransactionResponse to both Alice and Bob.

5. Bob presents the response to the Omnibridge contract before Alice,
   claiming Alice's bridged funds. The bridge contract cannot distinguish
   Bob's response from Alice's — both are cryptographically identical.

## Tweak Derivation (Sign vs ForeignTx)

`verify_foreign_transaction()` uses a **different tweak derivation prefix** than `sign()` so the same
`(predecessor_id, derivation_path)` can never yield the same derived key across the two purposes.

Design:

* Keep the existing sign tweak derivation prefix unchanged.
* Introduce a foreign-tx-specific prefix and derive the tweak from the same `(predecessor_id, derivation_path)`
  input using the same hash construction.
* The contract derives the tweak internally from `request.derivation_path` (callers do not submit raw tweaks).

Example:

```rust
const SIGN_TWEAK_DERIVATION_PREFIX: &str =
    "near-mpc-recovery v0.1.0 epsilon derivation:";
const FOREIGN_TX_TWEAK_DERIVATION_PREFIX: &str =
    "near-mpc-recovery v0.1.0 foreign-tx epsilon derivation:";

pub fn derive_sign_tweak(predecessor_id: &AccountId, path: &str) -> Tweak {
    let hash: [u8; 32] = derive_from_path(SIGN_TWEAK_DERIVATION_PREFIX, predecessor_id, path);
    Tweak::new(hash)
}

pub fn derive_foreign_tx_tweak(predecessor_id: &AccountId, path: &str) -> Tweak {
    let hash: [u8; 32] = derive_from_path(FOREIGN_TX_TWEAK_DERIVATION_PREFIX, predecessor_id, path);
    Tweak::new(hash)
}
