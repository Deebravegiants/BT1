### Title
Propose-Update Storage Deposits Are Permanently Locked in the Contract — (File: `crates/contract/src/update.rs`)

---

### Summary

Every call to `propose_update` requires the proposer to attach a NEAR deposit that covers the on-chain storage cost of the uploaded contract code or config blob. When `do_update` executes, it clears all stored entries and frees that storage — but the deposit is never returned to the proposer. No withdrawal or refund path exists anywhere in the contract for these accumulated balances. The funds are permanently locked.

---

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` enforces a deposit equal to `ProposedUpdates::required_deposit(&update)`: [1](#0-0) 

`required_deposit` is calculated as: [2](#0-1) 

For a typical contract binary (~500 KB) plus the 128-participant-vote overhead baked into `bytes_used`, this amounts to several NEAR tokens per proposal: [3](#0-2) 

Excess above the required amount is refunded to the proposer: [4](#0-3) 

However, the exact required deposit is retained by the contract. When `do_update` is later called, it clears **all** entries and votes — freeing the storage — but issues **no refund** to any proposer: [5](#0-4) 

This means:
1. The winning proposer's deposit is lost when their update executes.
2. Every competing proposer's deposit is also lost when `entries.clear()` wipes their entry.

A grep across the entire contract for any withdrawal, treasury, or refund path for these accumulated balances returns only the excess-refund logic in `require_deposit` and `propose_update` — there is no general withdrawal function.

---

### Impact Explanation

**Medium.** This breaks the balance/accounting invariant of the chain-signature contract. Every governance upgrade cycle permanently locks several NEAR tokens in the contract with no recovery path. The funds are controlled by the MPC contract and are frozen there indefinitely. This matches the allowed Medium impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

**Medium.** Contract upgrades are a routine governance operation. Every time participants vote in a new contract version or config change, the proposer's deposit (and any competing proposers' deposits) are silently forfeited. This is not a theoretical edge case — it occurs on every successful `vote_update` execution.

---

### Recommendation

In `do_update`, before clearing entries, iterate over all stored `UpdateEntry` records and issue a `Promise::new(proposer_account).transfer(entry_deposit)` for each one. The proposer's account ID must be stored alongside the `UpdateEntry` at proposal time. Alternatively, track a single "active proposer + deposit" pair and refund only that proposer on execution, since competing proposals are speculative.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB contract binary. `required_deposit` ≈ 5 NEAR. A attaches exactly 5 NEAR; no excess is refunded.
2. Participant B calls `propose_update` with a different binary. B attaches 5 NEAR.
3. Threshold participants vote for A's update via `vote_update`.
4. `do_update` is triggered: `entries.clear()` removes both A's and B's entries, freeing ~10 NEAR of storage. Neither A nor B receives any refund.
5. The contract's balance increases by ~10 NEAR with no mechanism to recover it. [6](#0-5) [7](#0-6)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}
