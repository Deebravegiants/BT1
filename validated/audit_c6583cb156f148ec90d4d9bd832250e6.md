### Title
Unbounded Domain Iteration in `process_new_parameters_proposal` Enables Permanent Resharing Freeze - (File: `crates/contract/src/state/running.rs`)

### Summary
`process_new_parameters_proposal` iterates over the entire `DomainRegistry` on every `vote_new_parameters` call with no cap on the number of registered domains. Because `vote_add_domains` imposes no `MAX_DOMAIN_COUNT` limit, a sufficiently large domain set causes every `vote_new_parameters` transaction to exhaust NEAR's 300 TGas block limit, permanently blocking resharing and participant-set rotation.

### Finding Description
`process_new_parameters_proposal` in `crates/contract/src/state/running.rs` performs two unbounded passes over `self.domains.domains()`:

1. A linear scan to reject unknown domain IDs from the caller-supplied `threshold_updates` map: [1](#0-0) 

2. A full `.iter().map(…).collect()` to build `effective_domains`, followed by a second full pass to call `validate_domain_threshold` on every entry: [2](#0-1) 

The `DomainRegistry` is an unbounded `Vec<DomainConfig>` with no enforced ceiling: [3](#0-2) 

A search of the entire contract source confirms there is no `MAX_DOMAIN_COUNT` constant anywhere — contrast this with the fan-out queue, which was explicitly capped at `MAX_PENDING_REQUEST_FAN_OUT = 128` precisely to prevent the same class of gas-exhaustion bug: [4](#0-3) 

Domains are added via `vote_add_domains`, which requires unanimous participant consensus: [5](#0-4) 

Once enough domains accumulate through legitimate governance, every subsequent call to `vote_new_parameters` will exceed 300 TGas and panic. Because NEAR rolls back state on panic, the contract is left unchanged and the call can never succeed regardless of how much gas is attached.

### Impact Explanation
**Medium — participant-state and contract execution-flow invariant broken.**

If `vote_new_parameters` is permanently unexecutable:
- Resharing is blocked forever; the participant set can never rotate.
- If the current participant set later falls below the signing threshold (node failures, key loss, etc.), no new threshold signatures can be produced and all funds controlled by the MPC network are permanently frozen.
- Malicious current participants retain indefinite signing authority with no governance recourse.

This matches the allowed impact: *"participant-state or contract execution-flow manipulation that breaks production safety/accounting invariants."*

### Likelihood Explanation
Low-to-medium. Reaching a domain count that saturates 300 TGas requires many successive unanimous `vote_add_domains` rounds. However:
- There is no protocol-level barrier preventing it.
- The contract already demonstrates awareness of this class of bug (the `MAX_PENDING_REQUEST_FAN_OUT` cap), yet omitted an equivalent guard for domains.
- A single `vote_add_domains` call accepts an unbounded `Vec<DomainConfig>`, so a large batch can be proposed in one round. [6](#0-5) 

### Recommendation
Add a `MAX_DOMAIN_COUNT` constant (analogous to `MAX_PENDING_REQUEST_FAN_OUT`) and enforce it inside `DomainRegistry::add_domains`: [7](#0-6) 

The limit should be derived from empirical gas measurements of `process_new_parameters_proposal` with a worst-case domain list, leaving headroom below 300 TGas. Additionally, `vote_add_domains` should reject proposals whose size would push the total domain count above the cap.

### Proof of Concept

1. Deploy the contract and initialize it with `N` participants.
2. Repeatedly call `vote_add_domains` (with all `N` participants voting) to add batches of domains until `DomainRegistry::domains` holds `D` entries where `D` is large enough that two full linear scans over `Vec<DomainConfig>` exceed 300 TGas.
3. Any participant now calls `vote_new_parameters` with a valid proposal.
4. The call panics with gas exhaustion inside `process_new_parameters_proposal` at the `effective_domains` construction loop.
5. Repeat with maximum gas — the call always panics; resharing is permanently blocked.
6. If participants subsequently drop below the signing threshold, no signatures can be produced and all MPC-controlled funds are frozen.
