### Title
Malicious Harvester Can Unconditionally Redirect Farmer Block Rewards via Unenforced CHIP-22 Fee Convention — (`chia/farmer/farmer_api.py`, `chia/farmer/farmer.py`)

---

### Summary

The CHIP-22 fee convention allows third-party harvesters to redirect the farmer reward address by setting `farmer_reward_address_override`. The farmer is supposed to enforce a fee quality threshold before honoring this override, but the implementation only logs warnings and never rejects. A malicious connected harvester can redirect 100% of farmer block rewards to an arbitrary address with no cryptographic proof of entitlement.

---

### Finding Description

**Two unenforced paths exist:**

**Path 1 — `NewProofOfSpace` with `fee_info=None`:**

In `new_proof_of_space`, when a harvester sends `farmer_reward_address_override` set, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee`. That function checks `fee_info` and the fee quality threshold, but in all failure cases — including `fee_info=None` — it only emits `log.warning` and returns normally. Processing is never halted. [1](#0-0) [2](#0-1) 

**Path 2 — `RespondSignatures` with `farmer_reward_address_override`:**

`RespondSignatures` (the harvester's reply to `RequestSignatures`) carries its own `farmer_reward_address_override` field. In `_process_respond_signatures`, this field is accepted **unconditionally** — there is no fee quality check, no `fee_info` validation, and no call to `notify_farmer_reward_taken_by_harvester_as_fee`: [3](#0-2) 

The resulting `DeclareProofOfSpace` is sent to the full node with the attacker-controlled address as `farmer_reward_address`: [4](#0-3) 

Note that `RespondSignatures` has no `fee_info` field at all — the fee convention is entirely absent from this message type: [5](#0-4) 

---

### Impact Explanation

The full node uses the `farmer_reward_address` from `DeclareProofOfSpace` to construct the foliage block. The farmer then signs this foliage (via `request_signed_values` → `_process_respond_signatures`) without verifying that the reward address matches its own configured `farmer_target`. The signed block pays the farmer reward to the attacker's address. This is a direct, complete payout redirection — 100% of the farmer reward per won block.

---

### Likelihood Explanation

CHIP-22 explicitly introduces third-party harvesters as a supported use case. Any operator using a third-party harvester service is exposed. The attack requires only a connected harvester peer — no key leakage, no admin compromise. The harvester simply sets `farmer_reward_address_override` in its `RespondSignatures` reply to any address it controls.

---

### Recommendation

In `_process_respond_signatures`, before honoring `response.farmer_reward_address_override`, the farmer must:
1. Retrieve the stored `NewProofOfSpace` for this proof (already cached in `self.farmer.proofs_of_space`)
2. Call `calculate_harvester_fee_quality` against the proof and challenge
3. Verify that `fee_info` is present and that `fee_quality <= fee_info.applied_fee_threshold`
4. **Reject** (return `None`) if the check fails, rather than logging and continuing

The same enforcement must be applied in `new_proof_of_space` — `notify_farmer_reward_taken_by_harvester_as_fee` must return a boolean and the caller must abort when it returns `False`.

---

### Proof of Concept

1. Connect a malicious harvester to a farmer.
2. When the harvester receives `NewSignagePointHarvester`, respond with `NewProofOfSpace` containing a valid proof and `farmer_reward_address_override=<attacker_address>`, `fee_info=None`.
3. Farmer logs `"Harvester illegitimately took reward by failing to provide its fee rate"` but does **not** reject — it sends `RequestSignatures` back.
4. Harvester replies with `RespondSignatures` containing `farmer_reward_address_override=<attacker_address>`.
5. `_process_respond_signatures` sets `farmer_reward_address = <attacker_address>` with no check.
6. `DeclareProofOfSpace` is broadcast to the full node with `farmer_reward_address=<attacker_address>`.
7. The full node builds a block paying the farmer reward to `<attacker_address>`.
8. The farmer signs the foliage (which embeds `<attacker_address>`) via `request_signed_values`.
9. Block is finalized; farmer reward is paid to the attacker. [6](#0-5) [7](#0-6)

```python
    def notify_farmer_reward_taken_by_harvester_as_fee(
        self, sp: farmer_protocol.NewSignagePoint, proof_of_space: harvester_protocol.NewProofOfSpace
    ) -> None:
        """
        Apply a fee quality convention (see CHIP-22: https://github.com/Chia-Network/chips/pull/88)
        given the proof and signage point. This will be tested against the fee threshold reported
        by the harvester (if any), and logged.
        """
        assert proof_of_space.farmer_reward_address_override is not None

        challenge_str = str(sp.challenge_hash)

        ph_prefix = self.config["network_overrides"]["config"][self.config["selected_network"]]["address_prefix"]
        farmer_reward_puzzle_hash = encode_puzzle_hash(proof_of_space.farmer_reward_address_override, ph_prefix)

        self.log.info(
            f"Farmer reward for challenge '{challenge_str}' "
            + f"taken by harvester for reward address '{farmer_reward_puzzle_hash}'"
        )

        fee_quality = calculate_harvester_fee_quality(proof_of_space.proof.proof, sp.challenge_hash)
        fee_quality_rate = float(fee_quality) / float(0xFFFFFFFF) * 100.0

        if proof_of_space.fee_info is not None:
            fee_threshold = proof_of_space.fee_info.applied_fee_threshold
            fee_threshold_rate = float(fee_threshold) / float(0xFFFFFFFF) * 100.0

            if fee_quality <= fee_threshold:
                self.log.info(
                    f"Fee threshold passed for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
            else:
                self.log.warning(
                    f"Invalid fee threshold for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
                self.log.warning(
                    "Harvester illegitimately took a fee reward that "
                    + "did not belong to it or it incorrectly applied the fee convention."
                )
        else:
            self.log.warning(
                "Harvester illegitimately took reward by failing to provide its fee rate "
                + f"for challenge '{challenge_str}'. "
                + f"Fee quality was {fee_quality_rate:.3f}% ({fee_quality} or 0x{fee_quality:08x})"
            )
```
