### Title
Single-Participant Forced Resharing via `verify_tee` Bypasses Threshold Consensus — (`File: crates/contract/src/lib.rs`)

---

### Summary

`verify_tee()` allows a single participant to unilaterally force a resharing that permanently changes the active participant set — without any threshold vote — by exploiting the TEE attestation expiry mechanism. This is the direct analog of `LightClient.force`: both functions allow a minority actor to trigger a critical state transition that normally requires threshold consensus.

---

### Finding Description

The normal path for changing the participant set is `vote_new_parameters`, which requires **all** proposed participants to vote (with threshold current-epoch participants first): [1](#0-0) 

`verify_tee`, however, is callable by **any single participant** (`voter_or_panic()`) and, when `reverify_and_cleanup_participants` returns `TeeValidationResult::Partial` (i.e., some participants lack valid attestations), it directly invokes `transition_to_resharing_no_checks` — bypassing the threshold vote entirely: [2](#0-1) [3](#0-2) 

The only guard is a structural check that the surviving set still satisfies the threshold relation: [4](#0-3) 

If that check passes, `transition_to_resharing_no_checks` is called with no further authorization: [5](#0-4) 

The attestation validity is determined by `reverify_and_cleanup_participants`, which marks a participant invalid if their stored attestation has expired or is missing: [6](#0-5) 

---

### Impact Explanation

**Critical.** An attacker who controls a subset of participant accounts can manipulate the participant set without threshold consensus:

1. Attacker controls `k` participant accounts (where `k` is below the signing threshold).
2. Attacker ensures their own attestations remain valid; honest participants' attestations expire (naturally or through operational disruption).
3. Attacker calls `verify_tee()` — a single transaction from one of their accounts.
4. `reverify_and_cleanup_participants` returns `Partial`, excluding the honest participants.
5. The threshold-relation check passes if the remaining (attacker-controlled) set still satisfies `GovernanceThreshold ≥ max(ReconstructionThreshold)`.
6. `transition_to_resharing_no_checks` fires, entering a resharing that excludes honest participants.
7. After resharing completes, the attacker controls a majority (or all) of the new participant set and can issue unauthorized threshold signatures for any future signing request.

This maps directly to the `LightClient.force` scenario: a minority actor forces a state transition that installs a new "committee" they control, gaining authority over all future operations.

---

### Likelihood Explanation

**Medium-High.** The preconditions are:

- Attacker must be a current participant (not unprivileged, but the scope explicitly covers "Byzantine participant strictly below the signing threshold").
- Honest participants' attestations must have expired. Attestations carry `expiry_timestamp_seconds` and the design mandates renewal every 7 days. Any operational failure (node downtime, key rotation lag) causes expiry.
- The attacker does not need to forge any cryptographic material — they only need to call `verify_tee()` at the right moment.

The design doc confirms nodes are expected to call `verify_tee` periodically: [7](#0-6) 

This means the function is a normal operational call, making its misuse indistinguishable from legitimate use.

---

### Recommendation

1. **Short term**: Require `verify_tee` to collect threshold votes from current participants before triggering a resharing, mirroring the `vote_new_parameters` flow. A single caller should only be able to *query* TEE status, not *force* a participant-set change.
2. **Medium term**: Separate the "check and report" behavior (read-only) from the "kick and reshare" behavior (state-mutating), and gate the latter behind threshold consensus.
3. **Long term**: Document the liveness/safety tradeoff explicitly: if attestation renewal is a liveness requirement, the consequence of missed renewal (forced resharing by a single peer) must be acknowledged and mitigated.

---

### Proof of Concept

**Setup**: 5 participants, governance threshold 3, reconstruction threshold 2. Attacker controls participants A, B, C. Honest participants D, E fail to renew their attestations after 7 days.

**Step 1**: Attacker's attestations remain valid; D and E's attestations expire.

**Step 2**: Attacker calls `verify_tee()` from account A (single transaction):

```
contract.verify_tee()  // signer = A, a current participant
```

**Step 3**: `reverify_and_cleanup_participants` returns `Partial { participants_with_valid_attestation: [A, B, C] }`.

**Step 4**: Threshold-relation check: `remaining=3 >= governance_threshold=3` ✓ and `3 >= reconstruction_threshold=2` ✓ → check passes.

**Step 5**: `transition_to_resharing_no_checks` fires with proposed set `{A, B, C}`. [8](#0-7) 

**Step 6**: Resharing completes. New epoch has only `{A, B, C}` — all attacker-controlled. Attacker now meets threshold (3/3) and can issue signatures for any request without honest-participant participation.

The sandbox test `test_verify_tee_triggers_resharing_and_kickout_on_expired_attestation` confirms this exact flow succeeds with a single caller: [9](#0-8)

        if let Some(first_domain) = self.domains.get_domain_by_index(0) {

                    self.keyset.clone(),
                    self.parameters.clone(),
                    self.add_domains_votes.clone(),
                ),
                reshared_keys: Vec::new(),
                resharing_key: KeyEvent::new(

                    proposal.parameters().clone(),
                ),

                per_domain_thresholds: proposal.per_domain_thresholds().clone(),
            })

            // New parameters were proposed, but we have no keys, so directly
            // transition into Running state but bump the EpochId. With no
            // domains the per-domain threshold updates have nothing to apply to

                self.domains.clone(),
                Keyset::new(self.keyset.epoch_id.next(), Vec::new()),
                proposal.parameters().clone(),
                self.add_domains_votes.clone(),
            );
            None
        }

    pub fn reverify_and_cleanup_participants(
        &mut self,
        participants: &Participants,
        tee_upgrade_deadline_duration: Duration,
    ) -> TeeValidationResult {
        self.allowed_docker_image_hashes
            .cleanup_expired_hashes(tee_upgrade_deadline_duration);

        let participants_with_valid_attestation: Vec<_> = participants
            .participants()
            .iter()
            .filter(|(_, _, participant_info)| {
                // Use the stored NodeId (keyed by TLS public key) so the real
                // `account_public_key` participates in re-verification. If
                // there is no stored attestation for this TLS key, the
                // participant is invalid.
                let Some(node_id) = self.find_node_id_by_tls_key(&participant_info.tls_public_key)
                else {
                    return false;
                };

                let tee_status =
                    self.reverify_participants(&node_id, tee_upgrade_deadline_duration);

                matches!(tee_status, TeeQuoteStatus::Valid)
            })
            .cloned()
            .collect();

        if participants_with_valid_attestation.len() != participants.len() {
            let participants_with_valid_attestation =
                Participants::init(participants.next_id(), participants_with_valid_attestation);

            TeeValidationResult::Partial {
                participants_with_valid_attestation,
            }
        } else {
            TeeValidationResult::Full
        }
    }
