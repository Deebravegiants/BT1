### Title
Deposits for Non-Executed Update Proposals Are Permanently Lost When Any Update Is Executed - (File: `crates/contract/src/update.rs`)

---

### Summary

When `do_update` executes a winning governance proposal, it unconditionally clears **all** pending proposals and their associated storage. However, the NEAR deposits paid by proposers of the non-executed proposals are never refunded. Because `UpdateEntry` does not record the proposer's account ID, there is no mechanism to issue refunds, and those funds are permanently locked in the contract.

---

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` requires a deposit proportional to the storage consumed by the proposal:

```rust
let required = ProposedUpdates::required_deposit(&update);
// ...
// Refund the difference if the proposer attached more than required.
if let Some(diff) = attached.checked_sub(required) && diff > NearToken::from_yoctonear(0) {
    Promise::new(proposer).transfer(diff).detach();
}
``` [1](#0-0) 

The deposit is calculated by `bytes_used`, which includes the full WASM binary size for contract updates plus a fixed overhead for 128 participant votes:

```rust
fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;
    match update {
        Update::Contract(code) => { bytes_used += code.len() as u128; }
        // ...
    }
    bytes_used
}
``` [2](#0-1) 

When threshold votes are reached and `do_update` is called, it clears **all** entries — including proposals that were never executed:

```rust
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    // ...
}
``` [3](#0-2) 

Critically, `UpdateEntry` stores only the update payload and `bytes_used` — **not the proposer's account ID**:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
}
``` [4](#0-3) 

Because the proposer is not recorded, there is no way to issue refunds when entries are cleared. The NEAR deposited for non-executed proposals is permanently locked in the contract.

---

### Impact Explanation

For a contract update containing a typical MPC WASM binary (~500 KB–2 MB), the required deposit is approximately 5–20 NEAR per proposal (at NEAR's storage cost of ~10^19 yoctoNEAR/byte). In a governance scenario where multiple participants propose competing updates, all but one proposer permanently lose their deposits when any update reaches threshold. There is no cancel/withdraw path for proposals, and no refund is issued during `do_update`. This constitutes a **direct, permanent loss of funds** held by the chain-signature contract.

---

### Likelihood Explanation

Competing update proposals are a realistic governance scenario: participants may disagree on which contract version or config to apply, leading to multiple simultaneous proposals. Each proposer must attach a non-trivial deposit. The loss is triggered automatically and silently whenever any proposal reaches threshold — no attacker action is required beyond submitting a competing proposal. Likelihood is **moderate**.

---

### Recommendation

1. Add a `proposer: AccountId` field to `UpdateEntry` so the depositor is always recorded.
2. In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and issue `Promise::new(entry.proposer).transfer(refund_amount)` for each non-executed proposal.
3. Alternatively, expose a `cancel_proposal(id: UpdateId)` method that allows a proposer to withdraw their own proposal and receive a refund, callable only by the original proposer.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB contract binary, depositing ~10 NEAR (the required storage deposit).
2. Participant B calls `propose_update` with a different 1 MB contract binary, depositing ~10 NEAR.
3. A threshold of participants votes for B's proposal via `vote_update`.
4. `vote_update` calls `proposed_updates.do_update(&id, ...)`.
5. `do_update` removes B's entry, then calls `self.entries.clear()` — erasing A's entry.
6. No refund is issued to A. A's ~10 NEAR deposit is permanently locked in the contract.
7. A has no recourse: there is no `cancel_proposal` method, and the proposer field is not stored, so even a future contract upgrade cannot reconstruct who to refund. [3](#0-2) [5](#0-4)

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}
