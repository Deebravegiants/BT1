### Title
Unvalidated `ProposalInit.builder` Allows Any Legitimate Proposer to Redirect All Block Fees to an Arbitrary Sequencer Address - (File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs)

---

### Summary

`ProposalInit.builder` is the proposer-supplied field that becomes the `sequencer_address` in every transaction's execution context and in the block hash. The proposal-validation path (`is_proposal_init_valid`) never checks this field against any locally-trusted value. A legitimate consensus participant acting as proposer can set `builder` to any address; every validator will re-execute the block with that address as the sequencer, causing all transaction fees to be credited to the attacker-controlled address. The `ProposalCommitment` comparison still passes because both sides derive the commitment from the same proposer-supplied `builder`.

---

### Finding Description

`convert_to_sn_api_block_info` maps `init.builder` directly to `sequencer_address` in the `BlockInfo` that is handed to the blockifier:

```rust
// crates/apollo_consensus_orchestrator/src/utils.rs:329-347
Ok(starknet_api::block::BlockInfo {
    block_number: init.height,
    block_timestamp: BlockTimestamp(init.timestamp),
    sequencer_address: init.builder,   // ← proposer-supplied, never validated
    gas_prices: GasPrices { ... },
    ...
})
``` [1](#0-0) 

This `BlockInfo` is used in two places:

1. **Proposal build path** (`initiate_build` → `propose_block`): the proposer sets `builder: args.builder_address` from its own config.
2. **Proposal validate path** (`initiate_validation` → `validate_block`): the validator calls `convert_to_sn_api_block_info(&init)` with the **network-received** `init`, using whatever `builder` the proposer sent. [2](#0-1) 

`is_proposal_init_valid` checks `height`, `l1_da_mode`, `l2_gas_price_fri`, `starknet_version`, `version_constant_commitment`, all four L1 gas prices, and `fee_proposal_fri`. The `builder` field is absent from `ProposalInitValidation` and is never compared against any locally-configured value:

```rust
// ProposalInitValidation has no builder field
pub struct ProposalInitValidation {
    pub height: BlockNumber,
    pub block_timestamp_window_seconds: u64,
    pub previous_proposal_init: Option<PreviousProposalInitInfo>,
    pub l1_da_mode: L1DataAvailabilityMode,
    pub l2_gas_price_fri: GasPrice,
    pub starknet_version: StarknetVersion,
    pub fee_actual: Option<GasPrice>,
    // builder is absent
}
``` [3](#0-2) 

The validator node **does** have a locally-configured `builder_address` in `ContextStaticConfig`:

```rust
builder_address: self.config.static_config.builder_address,
``` [4](#0-3) 

But this value is only used when the node is the proposer. When validating an incoming proposal, the validator never checks `init_proposed.builder` against `self.config.static_config.builder_address`.

The `ProposalCommitment` comparison at the end of `validate_proposal` does **not** protect against this:

```rust
if built_block != received_fin.proposal_commitment {
    return Err(ValidateProposalError::ProposalFinMismatch);
}
``` [5](#0-4) 

Both the proposer and the validator derive the commitment from the same `builder` value (the one in the `ProposalInit`). The `PartialBlockHashComponents` includes `sequencer: SequencerContractAddress(block_info.sequencer_address)`, so the hash is internally consistent — but it encodes the attacker's address. [6](#0-5) 

---

### Impact Explanation

**Incorrect fee/balance effect with economic impact (Critical).**

In Starknet execution, transaction fees are transferred from the user's account to `sequencer_address`. Because `sequencer_address` is set from the unvalidated `init.builder`, a malicious proposer can redirect every fee payment in the block to an arbitrary address they control. Additionally, the `get_execution_info` syscall exposes `sequencer_address` to contracts:

```rust
sequencer_address: Felt::from(block_info.sequencer_address),
``` [7](#0-6) 

Any contract that gates logic on `get_execution_info().block_info.sequencer_address` (e.g., sequencer-only admin functions) will observe the attacker's address, producing wrong execution results. The committed block hash also permanently encodes the wrong sequencer address.

---

### Likelihood Explanation

Any legitimate consensus participant who is selected as the proposer for a round can exploit this. No special capability beyond being a scheduled proposer is required. The `proposer` field is checked in `handle_proposal` (ensuring only the designated proposer's message is processed), but `builder` is a separate, unchecked field. The attack is silent — no error is raised, no metric is incremented, and the block is committed normally. [8](#0-7) 

---

### Recommendation

Add `builder: ContractAddress` to `ProposalInitValidation`, populate it from `self.config.static_config.builder_address` in `validate_proposal` inside `SequencerConsensusContext`, and add a check in `is_proposal_init_valid`:

```rust
if init_proposed.builder != proposal_init_validation.builder {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.builder, init_proposed.builder
        ),
    ));
}
```

This mirrors the existing pattern used for `height`, `l1_da_mode`, and `l2_gas_price_fri`. [9](#0-8) 

---

### Proof of Concept

1. Attacker is a legitimate validator with address `V` in the committee.
2. In a round where `V` is the designated proposer, attacker constructs `ProposalInit` with `builder = ATTACKER_WALLET` (any address they control) instead of the legitimate `builder_address`.
3. Attacker streams the proposal to all validators.
4. Each validator calls `is_proposal_init_valid` — no check on `builder`, returns `Ok(())`.
5. Each validator calls `initiate_validation` → `convert_to_sn_api_block_info` → `sequencer_address = ATTACKER_WALLET`.
6. All transactions execute; fees are credited to `ATTACKER_WALLET`.
7. `PartialBlockHashComponents` encodes `sequencer = ATTACKER_WALLET`; the commitment matches the proposer's `ProposalFin`.
8. `validate_proposal` returns `Ok(built_block)`. Consensus reaches decision. Block is committed with `sequencer_address = ATTACKER_WALLET` and all fees stolen. [10](#0-9) [11](#0-10)
