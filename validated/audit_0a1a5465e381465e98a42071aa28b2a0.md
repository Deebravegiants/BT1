### Title
Missing Minimum Output Protection in L2 Pool `deposit()` Functions Exposes Users to Oracle Rate Slippage — (`File: contracts/pools/RSETHPoolV2.sol`)

---

### Summary

All L2 deposit pool contracts (`RSETHPoolV2.sol`, `RSETHPoolV2NBA.sol`, `RSETHPoolV2ExternalBridge.sol`, `RSETHPoolV3.sol`, `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV3WithNativeChainBridge.sol`, `RSETHPoolNoWrapper.sol`) expose a `deposit()` function that converts ETH or LSTs into rsETH/wrsETH using a live oracle rate, but accept no `minRsETHAmountExpected` parameter. If the oracle rate changes between transaction submission and execution, users silently receive fewer rsETH tokens than expected with no on-chain recourse.

---

### Finding Description

Every L2 pool `deposit()` function follows the same pattern:

1. Accept ETH (or token) from the caller.
2. Call `getRate()` → `IOracle(rsETHOracle).getRate()` to fetch the live rsETH/ETH exchange rate.
3. Compute `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate`.
4. Mint that amount of wrsETH/rsETH to the caller.

No floor check on the minted amount exists anywhere in this path.

**Affected deposit path in `RSETHPoolV2.sol`:**

```solidity
function deposit(string memory referralId) external payable nonReentrant whenNotPaused limitDailyMint(msg.value) {
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();

    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);  // uses live oracle
    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);                                     // no minimum check
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
``` [1](#0-0) 

The rate computation:

```solidity
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();                    // live oracle, can change
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate; // no floor enforced
}
``` [2](#0-1) 

The same pattern is replicated verbatim across `RSETHPoolV3.sol` `deposit(string)` and `deposit(address,uint256,string)`: [3](#0-2) 

And in `RSETHPoolNoWrapper.sol` (Arbitrum/Unichain variant): [4](#0-3) 

**Contrast with the L1 mainnet pool**, which already enforces a minimum output:

```solidity
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [5](#0-4) 

The L1 `_beforeDeposit` helper accepts `minRSETHAmountExpected` as an explicit parameter and reverts if the computed mint amount falls below it. [6](#0-5) 

The L2 pools have no equivalent guard.

---

### Impact Explanation

**Low — Contract fails to deliver promised returns.**

When the oracle rate rises between the moment a user signs a transaction and the moment it is included in a block, `rsETHToETHrate` is larger, so `rsETHAmount` is smaller. The user's ETH is consumed by the pool and they receive fewer wrsETH tokens than the off-chain quote showed. Because there is no minimum-output parameter, the transaction does not revert; the shortfall is silently accepted. The user cannot recover the difference.

---

### Likelihood Explanation

The rsETH oracle rate is updated periodically by the protocol's oracle infrastructure. On L2 chains with public mempools (Arbitrum, Base, Linea, etc.), a pending deposit transaction is visible before inclusion. Any oracle update that lands in the same block before the deposit — whether from a scheduled update or a searcher-triggered update — changes the effective rate. This is a routine occurrence on active chains and requires no privileged access beyond normal oracle operation.

---

### Recommendation

Add a `uint256 minRsETHAmountExpected` parameter to every `deposit()` overload in all L2 pool contracts and revert if the computed amount falls below it, mirroring the existing L1 guard:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountExpected)
    external payable nonReentrant whenNotPaused limitDailyMint(msg.value)
{
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();

    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);

    if (rsETHAmount < minRsETHAmountExpected) revert MinimumAmountToReceiveNotMet();

    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
```

Apply the same change to the token-based `deposit(address,uint256,string,uint256)` overloads in `RSETHPoolV3.sol`, `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV3WithNativeChainBridge.sol`, and `RSETHPoolNoWrapper.sol`.

---

### Proof of Concept

1. Oracle reports rsETH/ETH rate = 1.05e18 (1 rsETH = 1.05 ETH).
2. User submits `deposit{value: 1 ether}("ref")` expecting ≈ 0.952 wrsETH.
3. Before the tx is included, the oracle is updated to 1.10e18.
4. `viewSwapRsETHAmountAndFee(1 ether)` now returns `rsETHAmount = 1e18 * 1e18 / 1.10e18 ≈ 0.909e18`.
5. User receives ≈ 0.909 wrsETH instead of ≈ 0.952 wrsETH — a ~4.5% shortfall — with no revert.
6. The ETH is locked in the pool; the user has no recourse.
