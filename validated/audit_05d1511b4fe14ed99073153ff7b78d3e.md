### Title
Missing CKD Output Verification for `AppPublicKey` Variant Allows Single Participant to Forge Key Derivation Response - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_ckd` in the MPC contract applies a cryptographic output-validity check only for the `AppPublicKeyPV` variant of a CKD request. For the `AppPublicKey` variant, the check is entirely absent. A single malicious attested participant — well below the signing threshold — can call `respond_ckd` with an arbitrary `CKDResponse` for any pending `AppPublicKey` request, and the contract will accept and deliver the fraudulent key to the user without any cryptographic verification.

### Finding Description

`respond_ckd` enforces the following guards before resolving a pending CKD yield:

1. Protocol state is `Running` or `Resharing`.
2. `accept_requests` is `true`.
3. Caller is an attested participant (`assert_caller_is_attested_participant_and_protocol_active`).
4. **Conditionally**: if the request carries `AppPublicKeyPV`, run `ckd_output_check`; if it carries `AppPublicKey`, do nothing. [1](#0-0) 

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

After this match, `resolve_yields_for` is called unconditionally, serialising whatever `response` was supplied and resuming every queued yield for that request key. [2](#0-1) 

The `ckd_output_check` helper (imported at the top of `lib.rs`) exists precisely to verify that the derived output is consistent with the master public key and the app identifier. [3](#0-2) 

For `AppPublicKeyPV` requests this check is enforced; for `AppPublicKey` requests it is silently skipped — a structural parallel to the external report's pattern where a safety invariant is only enforced when the system is not already in a "bad" state.

### Impact Explanation

The CKD protocol is designed so that producing a correct derivation output requires t-of-n cooperation among MPC participants. The on-chain `ckd_output_check` is the contract's enforcement point for that threshold requirement. By skipping it for `AppPublicKey` requests, a **single** attested participant (1-of-n, far below the reconstruction threshold) can:

- Call `respond_ckd` with an arbitrary `CKDResponse` (e.g., a key the attacker controls).
- The contract accepts it, resolves all queued yields, and delivers the attacker-chosen key material to every waiting caller.
- The user believes they received a legitimately MPC-derived application key and may use it to control funds on a foreign chain — funds the attacker can also spend.

This is a threshold bypass for confidential key derivation: one participant achieves what should require t participants. Impact maps to **Critical — confidential key derivation output without the required participant authorization / bypass of threshold-signature requirements**.

### Likelihood Explanation

Any single participant that has obtained a valid TEE attestation and is registered in the contract can exploit this. TEE attestation is a per-node operational requirement, not a high bar for a Byzantine participant already inside the network. The `AppPublicKey` variant is a standard user-facing request path, so pending requests of this type will exist in production. No collusion above threshold is required.

### Recommendation

Apply `ckd_output_check` (or an equivalent derivation-consistency check) for the `AppPublicKey` variant as well, using the public key embedded in the variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

If the `AppPublicKey` variant intentionally omits the app public key in a form usable for on-chain verification, the protocol should be redesigned so that every `respond_ckd` path has a cryptographic output check, or the `AppPublicKey` variant should be removed in favour of `AppPublicKeyPV`.

### Proof of Concept

1. User submits `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(legitimate_pk)` and a valid deposit. The contract queues a yield and stores the `CKDRequest` in `pending_ckd_requests`.
2. Attacker (a single attested participant) constructs a `CKDResponse` containing a key pair they control.
3. Attacker calls `respond_ckd(request, attacker_response)`. All guards pass: protocol is running, `accept_requests` is true, caller is attested. The `AppPublicKey` arm executes with no body.
4. `resolve_yields_for` resumes the user's yield with the attacker-supplied response.
5. The user's callback receives the attacker-controlled key and uses it to derive a foreign-chain address, depositing funds. The attacker, who knows the private key, drains those funds. [4](#0-3)

```rust
    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
```
