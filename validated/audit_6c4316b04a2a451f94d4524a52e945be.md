### Title
Missing Zero-Address Validation for `owner` in `addLiquidity` Permanently Locks LP Principal - (File: `metric-core/contracts/MetricOmmPool.sol`)

### Summary

`MetricOmmPool.addLiquidity` accepts `owner = address(0)` without validation. Because `removeLiquidity` enforces `msg.sender == owner`, any shares minted to the zero-address position key are permanently unredeemable, locking the deposited tokens forever.

### Finding Description

`MetricOmmPool.addLiquidity` passes the caller-supplied `owner` directly to `LiquidityLib.addLiquidity` with no zero-address guard: [1](#0-0) 

Inside `LiquidityLib.addLiquidity`, shares are credited to `_positionBinKey(address(0), salt, bin)` and tokens are pulled from `msg.sender` via the modify-liquidity callback: [2](#0-1) [3](#0-2) 

When `removeLiquidity` is later called, the core pool enforces: [4](#0-3) 

Since no EOA or contract can have `msg.sender == address(0)`, the position is permanently unredeemable. Tokens are pulled from the caller and credited to the pool's `binTotals`, but the corresponding LP shares can never be burned.

The periphery contract `MetricOmmPoolLiquidityAdder` does guard against this via `_validateOwner`: [5](#0-4) 

However, the core pool itself has no such guard. Any direct caller of the core pool — including third-party routers or integrators that bypass the periphery — can trigger this path.

This is the direct analog to the Celo "Invalid Beneficiary" bug: just as `setPaymentDelegation` stored `address(0)` as a beneficiary that later caused a mint revert and epoch payment loss, `addLiquidity(address(0), ...)` stores an unreachable owner that permanently locks LP principal.

### Impact Explanation

Direct, permanent loss of user principal. Tokens deposited into a zero-address-owned position are irrecoverable. The pool's `binTotals` correctly account for the tokens (they are not "lost" from the pool's accounting perspective), but the LP shares are unredeemable — the underlying tokens can never be withdrawn by any party. The pool remains solvent but the depositor's claim is permanently destroyed. [6](#0-5) 

### Likelihood Explanation

Low-to-medium. Requires a caller to supply `owner = address(0)`, which is a mistake rather than a deliberate attack. However:

- The core pool is a public, permissionless interface callable by any contract or EOA.
- Third-party integrators who interact with the core pool directly (not through `MetricOmmPoolLiquidityAdder`) have no on-chain protection.
- The inconsistency between the periphery guard and the absent core guard increases the probability of integration errors.
- No privileged role is required; any unprivileged caller can trigger this.

### Recommendation

Add a zero-address check at the top of `MetricOmmPool.addLiquidity`, before delegating to `LiquidityLib`:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {
    require(owner != address(0), InvalidOwner()); // <-- add this
    if (deltas.binIdxs.length == 0) return (0, 0);
    ...
}
```

This mirrors the guard already present in `MetricOmmPoolLiquidityAdder._validateOwner` and closes the gap between the periphery and core layers.

### Proof of Concept

1. Deploy a pool and a caller contract that implements `IMetricOmmModifyLiquidityCallback` and holds token0/token1.
2. Call `pool.addLiquidity(address(0), 0, deltas, callbackData, "")` directly on the core pool.
3. The callback fires on `msg.sender`; tokens are transferred into the pool; shares are minted to `keccak256(abi.encode(address(0), uint80(0), int8(binIdx)))`.
4. Attempt `pool.removeLiquidity(address(0), 0, deltas, "")` — reverts with `NotPositionOwner` because `msg.sender != address(0)`.
5. No further call can ever redeem those shares. The deposited tokens are permanently locked in the pool. [7](#0-6) [8](#0-7)
