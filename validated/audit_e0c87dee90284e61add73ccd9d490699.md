Audit Report

## Title
`MultiChainRateProvider.updateRate()` Reverts When `msg.value` Is Insufficient to Cover All Receiver Fees in the Loop - (File: contracts/cross-chain/MultiChainRateProvider.sol)

## Summary

`updateRate()` estimates and forwards LayerZero fees per receiver inside a single loop with no pre-flight check that `msg.value` covers the cumulative total. Because LayerZero fees are dynamic, a caller who correctly queries `estimateTotalFee()` off-chain can still trigger a mid-loop revert if fees rise before execution, leaving all rate receivers un-updated and the entire transaction reverted.

## Finding Description

In `MultiChainRateProvider.updateRate()` (lines 119–134), each loop iteration calls `estimateFees` and immediately forwards the result via `send{ value: estimatedFee }`. The contract's ETH balance decreases with each iteration. There is no guard of the form `require(msg.value >= totalFee)` before the loop begins.

The `estimateTotalFee()` view function (lines 154–173) exists as an off-chain helper but provides zero on-chain enforcement. Because LayerZero fees are dynamic (they depend on gas prices on the destination chain at the moment of the call), the fee returned by `estimateTotalFee()` at block N can differ from the sum of fees computed inside `updateRate()` at block N+k. If the difference causes `address(this).balance < estimatedFee` on any iteration, the EVM reverts the entire transaction, including the `rate` and `lastUpdated` writes at lines 111–113.

`updateRate()` carries no access control (`external payable`, no `onlyOwner` or similar modifier), so any caller — including automated keeper bots — can invoke it.

## Impact Explanation

When the transaction reverts, none of the L2 `RateReceiver` contracts receive the updated rsETH/ETH exchange rate. L2 pool contracts that depend on this rate for minting wrsETH continue using the stale value until a successful `updateRate()` call is made. No principal is lost and the revert is not permanent, but the contract fails to deliver its core promised function: cross-chain rate synchronization. This matches the allowed Low impact: **"Contract fails to deliver promised returns, but doesn't lose value."**

## Likelihood Explanation

`updateRate()` is permissionless and is expected to be called by keeper bots. LayerZero fees fluctuate with destination-chain gas prices. With multiple receivers registered, the probability that at least one fee increases between off-chain estimation and on-chain execution grows with the number of receivers. This is a realistic, repeatable failure mode under normal network congestion — not a contrived edge case.

## Recommendation

Perform a two-pass approach: first accumulate all estimated fees into a memory array and validate `msg.value >= totalFee`, then execute all `send` calls using the pre-computed values:

```solidity
function updateRate() external payable nonReentrant {
    uint256 latestRate = getLatestRate();
    bytes memory _payload = abi.encode(latestRate);
    uint256 rateReceiversLength = rateReceivers.length;

    uint256 totalFee;
    uint256[] memory fees = new uint256[](rateReceiversLength);
    for (uint256 i; i < rateReceiversLength;) {
        uint16 dstChainId = uint16(rateReceivers[i]._chainId);
        (fees[i],) = ILayerZeroEndpoint(layerZeroEndpoint)
            .estimateFees(dstChainId, address(this), _payload, false, bytes(""));
        totalFee += fees[i];
        unchecked { ++i; }
    }
    require(msg.value >= totalFee, "Insufficient ETH for fees");

    rate = latestRate;
    lastUpdated = block.timestamp;

    for (uint256 i; i < rateReceiversLength;) {
        uint16 dstChainId = uint16(rateReceivers[i]._chainId);
        bytes memory remoteAndLocalAddresses =
            abi.encodePacked(rateReceivers[i]._contract, address(this));
        ILayerZeroEndpoint(layerZeroEndpoint).send{ value: fees[i] }(
            dstChainId, remoteAndLocalAddresses, _payload,
            payable(msg.sender), address(0x0), bytes("")
        );
        unchecked { ++i; }
    }
    emit RateUpdated(rate);
}
```

## Proof of Concept

1. Deploy `MultiChainRateProvider` with two rate receivers on different chains.
2. Call `estimateTotalFee()` off-chain → returns `X` wei (sum of `fee_0 + fee_1`).
3. Submit `updateRate{ value: X }()`. Between block submission and inclusion, destination-chain gas prices rise, making `fee_1` increase by 1 wei.
4. Inside the loop: iteration 0 calls `estimateFees` → `fee_0` (unchanged), `send{ value: fee_0 }` succeeds, balance is now `X - fee_0`.
5. Iteration 1 calls `estimateFees` → `fee_1 + 1` (increased). `send{ value: fee_1 + 1 }` is attempted but `address(this).balance == X - fee_0 == fee_1 < fee_1 + 1` → EVM reverts with insufficient balance.
6. Entire transaction reverts; `rate`, `lastUpdated`, and all receiver updates are rolled back. No receiver gets the new rate.
