### Title
Single asset without a configured price oracle permanently blocks `LRTOracle#updateRSETHPrice` - (File: contracts/LRTOracle.sol)

---

### Summary

`LRTOracle._getTotalEthInProtocol()` iterates over every asset in `lrtConfig.getSupportedAssetList()` and calls `getAssetPrice(asset)` for each. If any single supported asset has no price oracle configured (`assetPriceOracle[asset] == address(0)`), the call reverts with `AssetOracleNotSupported`, causing the entire `updateRSETHPrice()` to revert. Because `LRTConfig.addNewSupportedAsset()` adds an asset to the supported list without atomically requiring a corresponding oracle entry in `LRTOracle`, a window exists — or a permanent state if the oracle is never set — where `updateRSETHPrice()` is completely blocked.

---

### Finding Description

`LRTOracle._getTotalEthInProtocol()` is the private function that computes total protocol TVL in ETH. It unconditionally calls `getAssetPrice(asset)` for every asset in the supported list:

```solidity
// contracts/LRTOracle.sol L336-L348
for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
    address asset = supportedAssets[assetIdx];
    uint256 assetER = getAssetPrice(asset);   // <-- reverts if oracle not set
    uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
    totalETHInProtocol += totalAssetAmt.mulWad(assetER);
    unchecked { ++assetIdx; }
}
```

`getAssetPrice` carries the `onlySupportedOracle` modifier:

```solidity
// contracts/LRTOracle.sol L40-L45
modifier onlySupportedOracle(address asset) {
    if (assetPriceOracle[asset] == address(0)) {
        revert AssetOracleNotSupported();
    }
    _;
}
```

`LRTConfig.addNewSupportedAsset()` pushes an asset into `supportedAssetList` and sets `isSupportedAsset[asset] = true`, but it does **not** require or set a price oracle in `LRTOracle`:

```solidity
// contracts/LRTConfig.sol L99-L118
function addNewSupportedAsset(address asset, uint256 depositLimit)
    external onlyRole(LRTConstants.TIME_LOCK_ROLE)
{
    _addNewSupportedAsset(asset, depositLimit);
}

function _addNewSupportedAsset(address asset, uint256 depositLimit) private {
    ...
    isSupportedAsset[asset] = true;
    supportedAssetList.push(asset);
    depositLimitByAsset[asset] = depositLimit;
    emit AddedNewSupportedAsset(asset, depositLimit);
}
```

The oracle is configured separately via `LRTOracle.updatePriceOracleFor()`, which is a distinct admin transaction. Between the two transactions — or permanently if the oracle is never set — `assetPriceOracle[newAsset] == address(0)`, and every call to `updateRSETHPrice()` reverts.

---

### Impact Explanation

`updateRSETHPrice()` is a public function that:

1. **Mints protocol fees as rsETH** — blocked, causing permanent loss of accrued yield for the treasury.
2. **Updates `rsETHPrice`** — the stored price used by `LRTDepositPool.getRsETHAmountToMint()` for every deposit becomes stale, causing incorrect rsETH minting amounts for all depositors.
3. **Enforces price-drop protection** — the circuit-breaker that pauses the protocol on a large price drop cannot fire, leaving users unprotected.

Impact: **Medium — Temporary (potentially extended) freezing of unclaimed yield** and stale-price mis-accounting for all depositors during the blocked window.

---

### Likelihood Explanation

Adding a new LST collateral is a routine protocol governance action. The oracle configuration is a separate transaction, creating a mandatory gap. Any keeper or user calling `updateRSETHPrice()` during that gap — or if the oracle transaction is simply forgotten — will cause a revert. The gap can last from seconds to indefinitely. No attacker action is required; the condition arises from normal protocol operation.

---

### Recommendation

Enforce atomicity between asset registration and oracle configuration. Two options:

1. **Require oracle at registration time**: Extend `addNewSupportedAsset` (or a wrapper) to accept a `priceOracle` address and call `LRTOracle.updatePriceOracleFor(asset, priceOracle)` in the same transaction.

2. **Skip assets with no oracle in the loop**: In `_getTotalEthInProtocol()`, check `assetPriceOracle[asset] != address(0)` before calling `getAssetPrice`, and `continue` if the oracle is absent. This mirrors the recommendation in the referenced report (skip bad elements rather than reverting the whole loop).

---

### Proof of Concept

```solidity
// Scenario: admin adds a new LST but has not yet set its oracle

// Step 1: Admin adds new asset (TIME_LOCK_ROLE action)
lrtConfig.addNewSupportedAsset(address(newLST), 50_000 ether);
// assetPriceOracle[newLST] == address(0) at this point

// Step 2: Any caller (including keeper bots) tries to update the rsETH price
lrtOracle.updateRSETHPrice();
// Reverts with: AssetOracleNotSupported
// because _getTotalEthInProtocol() calls getAssetPrice(newLST)
// and onlySupportedOracle modifier fires

// Consequence: rsETHPrice is frozen at its last value.
// Protocol fee minting is blocked.
// Price-drop circuit breaker cannot trigger.
// All deposits use a stale rsETHPrice until the oracle is set.
```

    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }

        if (depositLimit == 0) {
