Audit Report

## Title
`SwapAllowlistExtension.beforeSwap()` silently ignores `recipient`, allowing allowlisted swapper to route output tokens to any non-allowlisted address - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap()` receives `recipient` as its second parameter but leaves it unnamed and never validates it. Only `sender` is checked against the allowlist. Because `MetricOmmPool.swap()` unconditionally transfers output tokens to the caller-controlled `recipient` after the extension check passes, any allowlisted swapper can direct swap output to an address the pool admin never approved.

## Finding Description

`MetricOmmPool.swap()` accepts a caller-controlled `recipient` and passes it alongside `msg.sender` to `_beforeSwap()`: [1](#0-0) 

After `_beforeSwap` returns, the pool unconditionally transfers output tokens to `recipient` with no further access check: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap()` receives `recipient` as its second positional argument but leaves it unnamed (discarded) and only validates `sender`: [3](#0-2) 

The asymmetry with `DepositAllowlistExtension.beforeAddLiquidity()` confirms the pattern is inconsistent — the deposit extension correctly ignores `sender` and checks `owner` (the address that holds the economic position): [4](#0-3) 

The `IMetricOmmExtensions` interface confirms `recipient` is available to the hook and is intentionally passed through the call chain, making the omission a code-level defect rather than a design limitation.

## Impact Explanation

Any pool deploying `SwapAllowlistExtension` for KYC/compliance gating has its access control broken for the output leg. An allowlisted swapper calls `pool.swap(nonAllowlistedRecipient, ...)`, the `beforeSwap` check passes (only `sender` is verified), and the pool transfers real ERC-20 output tokens to `nonAllowlistedRecipient`. The non-allowlisted address receives actual token balances without ever being approved by the pool admin. This is a direct break of the access-control invariant the extension is designed to enforce — the pool admin's explicit restriction on who may receive output tokens is bypassed by an unprivileged path (any existing allowlist entry). The `MetricOmmSimpleRouter` amplifies the issue: if the router is allowlisted, any end-user can call `router.exactInputSingle(...)` with any `recipient` and the allowlist check passes unconditionally, since the router is `msg.sender` of `swap()`.

## Likelihood Explanation

Any pool configured with `SwapAllowlistExtension` is affected. The trigger requires only a single allowlisted address (including the router if allowlisted) to call `swap()` with a non-allowlisted `recipient`. No privileged cooperation is needed beyond holding an existing allowlist entry. The `recipient` parameter is fully caller-controlled with no on-chain validation anywhere in the call path.

## Recommendation

Add a `recipient` check inside `beforeSwap`, mirroring how `beforeAddLiquidity` checks `owner`:

```solidity
function beforeSwap(address sender, address recipient, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

Alternatively, explicitly document that `recipient` is intentionally unchecked and that pool admins must not rely on `SwapAllowlistExtension` to restrict who can receive output tokens.

## Proof of Concept

1. Pool is deployed with `SwapAllowlistExtension` attached.
2. Pool admin calls `setAllowedToSwap(pool, alice, true)` — Alice is allowlisted; Bob is not.
3. Alice calls `pool.swap(bob, zeroForOne, amountSpecified, priceLimitX64, callbackData, extensionData)`.
4. `_beforeSwap(alice, bob, ...)` is called → `SwapAllowlistExtension.beforeSwap(alice, bob, ...)`.
5. The check `allowedSwapper[pool][alice]` is `true` → no revert. The `recipient` check `allowedSwapper[pool][bob]` is never evaluated because the second parameter is unnamed and discarded.
6. Pool executes the swap and calls `transferToken1(bob, outputAmount)`.
7. Bob (non-allowlisted) receives real ERC-20 output tokens.

The missing check is confirmed at: [5](#0-4)
