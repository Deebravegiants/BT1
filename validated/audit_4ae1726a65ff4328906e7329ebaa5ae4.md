### Title
Hardcoded Governance Threshold Upper Bound Permits Permanent Contract Governance Freeze — (File: `crates/contract/src/primitives/thresholds.rs`)

---

### Summary

The `governance_threshold_upper_relative_bound` function hardcodes the maximum allowed governance threshold to exactly `n` (100% of participants). This mirrors the external report's pattern: a hardcoded limit that permits a dangerous configuration. If participants vote to set `t = n` and any single participant subsequently becomes unavailable, no governance vote can ever reach threshold again, permanently freezing resharing, key rotation, domain management, and contract upgrades.

---

### Finding Description

In `crates/contract/src/primitives/thresholds.rs`, the upper bound on the governance threshold is hardcoded to return the full participant count:

```rust
/// Upper bound on the GovernanceThreshold for `n` participants:
/// Currently set to 100% of participants but would be a discussion subject
/// to drop this upper bound down not to have problems with smart contract
/// being locked if t = n and if an operator stops voting
pub(crate) fn governance_threshold_upper_relative_bound(n: u64) -> u64 {
    n
}
``` [1](#0-0) 

This value is consumed directly in `validate_threshold`, which is the single gate used by `ThresholdParameters::new`, `ThresholdParameters::validate`, and `validate_governance_against_reconstruction` — every path that validates a governance threshold change:

```rust
let upper_relative_bound = governance_threshold_upper_relative_bound(n_shares);
if k.value() > upper_relative_bound {
    return Err(InvalidThreshold::MaxRelRequirementFailed { ... }.into());
}
``` [2](#0-1) 

Because the upper bound is `n`, a valid `ThresholdParameters` with `threshold = n` can be constructed and voted in through `vote_new_parameters`. Once accepted, every subsequent governance vote (`vote_new_parameters`, `vote_add_domains`, `vote_update`, `verify_tee`-triggered resharing) requires all `n` participants to agree. If even one participant crashes, loses connectivity, or is decommissioned, the quorum can never be reached again.

The code comment is self-incriminating: *"would be a discussion subject to drop this upper bound down not to have problems with smart contract being locked if t = n and if an operator stops voting."* The risk is acknowledged but the hardcoded value was never changed.

---

### Impact Explanation

With `t = n` set and one participant unavailable:

- `vote_new_parameters` can never pass → no resharing, no participant rotation, no threshold change.
- `vote_add_domains` can never pass → no new signing domains.
- `vote_update` can never pass → no contract upgrades, including security patches.
- `verify_tee`-triggered resharing (TEE kickout path) requires governance votes that also cannot pass.

The MPC network's governance is permanently frozen. Because key rotation and participant removal are impossible, any key compromise or participant failure becomes unrecoverable. Funds whose signatures depend on the MPC network's continued operation are effectively permanently frozen — matching the Critical/Medium allowed impact of "permanent freezing of funds controlled by the MPC network" and "contract execution-flow manipulation that breaks production safety invariants."

---

### Likelihood Explanation

The trigger requires threshold-many participants to vote for `t = n`. This is a realistic governance mistake: operators seeking "maximum security" may propose and vote for unanimity without understanding the liveness consequence. The code's own comment confirms the development team is aware this is a plausible misconfiguration. Once set, a single node failure (hardware, network, key loss) is sufficient to trigger the freeze — a routine operational event.

---

### Recommendation

Lower the hardcoded upper bound. A common safe ceiling is `floor(n * 2/3)` or `n - 1` (requiring all-but-one), preventing a single-node failure from permanently halting governance. The bound should be a named constant or a storage-configurable parameter set at initialization, not an inline `n`:

```rust
pub(crate) fn governance_threshold_upper_relative_bound(n: u64) -> u64 {
    // e.g., n - 1, or (2 * n) / 3, stored/configured at init
    n.saturating_sub(1).max(governance_threshold_lower_relative_bound(n))
}
``` [1](#0-0) 

---

### Proof of Concept

1. Deploy the contract with `n = 3` participants and governance threshold `t = 2` (valid, 67%).
2. Call `vote_new_parameters` with a proposal setting `t = 3` (= `n`). `ThresholdParameters::new` accepts it because `governance_threshold_upper_relative_bound(3) == 3`.
3. Two of the three participants vote for the proposal; it passes and the new threshold is stored.
4. Participant 3 goes offline (hardware failure).
5. Any subsequent `vote_new_parameters`, `vote_add_domains`, or `vote_update` call can collect at most 2 votes — one short of the required 3.
6. The contract is permanently frozen: no resharing, no key rotation, no upgrades, no participant removal. All MPC-controlled assets are unrecoverable. [3](#0-2) [4](#0-3)

    fn validate_threshold(n_shares: u64, k: Threshold) -> Result<(), Error> {
        if k.value() > n_shares {
            return Err(InvalidThreshold::MaxRequirementFailed {
                max: n_shares,
                found: k.value(),
            }
            .into());
        }
        if k.value() < MIN_THRESHOLD_ABSOLUTE {
            return Err(InvalidThreshold::MinAbsRequirementFailed.into());
        }
        let lower_relative_bound = governance_threshold_lower_relative_bound(n_shares);
        if k.value() < lower_relative_bound {
            return Err(InvalidThreshold::MinRelRequirementFailed {
                required: lower_relative_bound,
                found: k.value(),
            }
            .into());
        }
        let upper_relative_bound = governance_threshold_upper_relative_bound(n_shares);
        if k.value() > upper_relative_bound {
            return Err(InvalidThreshold::MaxRelRequirementFailed {
                max: upper_relative_bound,
                found: k.value(),
            }
            .into());
        }
        Ok(())
    }

        Self::validate_threshold(num_participants, governance)?;
        if let Some(max_reconstruction_threshold) = max_reconstruction_threshold
