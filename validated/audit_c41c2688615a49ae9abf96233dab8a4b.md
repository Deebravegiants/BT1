### Title
`validateGuardDatums` Silently Skips Datum Enforcement for Plutus Guard Credentials Whose Script Is Not in Sub-Transaction's `scriptsProvided` - (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxow.hs`)

---

### Summary

In the Dijkstra era, `validateGuardDatums` is supposed to enforce that every Plutus-script credential listed in a sub-transaction's `requiredTopLevelGuards` field carries a datum (`SJust`). However, when the referenced script hash is not present in the sub-transaction's own `scriptsProvided`, the function silently returns the accumulator unchanged (`Nothing -> acc`), completely skipping the datum-presence check. Because scripts provided at the top-level transaction level are never included in a sub-transaction's `scriptsProvided`, an attacker can craft a sub-transaction that declares a Plutus guard credential with no datum (`SNothing`) and have it accepted by the ledger without triggering `SubMalformedGuardDatums`.

---

### Finding Description

**Vulnerability class:** Incomplete guard/protection mechanism — a validation check is declared and wired into the transition rule, but its enforcement has a reachable gap that allows the protected invariant to be violated.

**Root cause — `validateGuardDatums`, line 235:**

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxow.hs
validateGuardDatums (ScriptsProvided scripts) txBody =
  failureOnNonEmptySet malformed SubMalformedGuardDatums
  where
    malformed =
      Map.foldlWithKey' accum mempty (txBody ^. requiredTopLevelGuardsL)
    accum acc cred mbDatum =
      case credScriptHash cred of
        Nothing ->
          case mbDatum of
            SNothing -> acc
            SJust _  -> Set.insert cred acc
        Just scriptHash ->
          case Map.lookup scriptHash scripts of
            Just script
              | isNativeScript script ->
                  case mbDatum of { SNothing -> acc; SJust _ -> Set.insert cred acc }
              | otherwise ->
                  case mbDatum of { SJust _ -> acc; SNothing -> Set.insert cred acc }
            Nothing -> acc   -- ← silent skip: no datum check performed
``` [1](#0-0) 

The `Nothing -> acc` branch at line 235 means: if the script hash is not found in `scripts`, the credential is **not** added to `malformed`, regardless of whether `mbDatum` is `SNothing` or `SJust`. For a Plutus-script credential with `SNothing`, this is the wrong outcome — it should be flagged as malformed.

**Why the script is routinely absent from `scriptsProvided`:**

`scriptsProvided` for a sub-transaction is computed by `scriptsProvidedDijkstraStAnnTx`, which for a `SubTx` returns only `dsastScriptsProvided` — the scripts from the sub-transaction's own witness set and its own reference inputs. [2](#0-1) 

Scripts provided at the top-level transaction (in the top-level witness set or via top-level reference inputs) are **never** merged into a sub-transaction's `scriptsProvided`. Because guard scripts are typically provided at the top level (they are executed there via `GuardingPurpose`), their hashes will almost always be absent from the sub-transaction's `scriptsProvided`, making the `Nothing -> acc` branch the common path for any Plutus guard credential.

**How `getDijkstraScriptsNeeded` adds guard scripts to the top-level execution context:** [3](#0-2) 

Guard Plutus scripts are executed at the top level with `GuardingPurpose`. The datum from `requiredTopLevelGuards` is the datum the sub-transaction author is supposed to supply for that execution. Bypassing `validateGuardDatums` allows a sub-transaction to declare `SNothing` for a Plutus guard credential, meaning the guard script runs without the datum the sub-transaction was required to provide.

**The `MissingRequiredGuards` check does not compensate:**

The top-level `dijkstraUtxowTransition` checks that every credential in every sub-transaction's `requiredTopLevelGuards` is present in the top-level `guardsTxBodyL`: [4](#0-3) 

This check only verifies credential presence, not datum correctness. It does not substitute for `validateGuardDatums`.

---

### Impact Explanation

**Medium. Attacker-controlled sub-transactions bypass intended validation limits.**

A sub-transaction author can submit a `requiredTopLevelGuards` map containing a Plutus-script credential with `SNothing` (no datum) and have it accepted by the ledger, violating the invariant that Plutus guard credentials must carry a datum. The guard Plutus script then executes at the top level without the datum the sub-transaction was supposed to supply. Depending on the guard script's logic, this can cause:

- The guard script to fail, permanently blocking the top-level transaction (denial of service against the top-level transaction author).
- The guard script to succeed without the expected datum check, bypassing the guard's intended authorization logic — potentially allowing unauthorized spending, minting, certificate submission, or governance actions that the guard was designed to prevent.

---

### Likelihood Explanation

Any unprivileged transaction submitter can craft a sub-transaction with a `requiredTopLevelGuards` entry of the form `{ScriptHashObj sh -> SNothing}` where `sh` is a Plutus script hash not present in the sub-transaction's own witness set or reference inputs. This is a normal serialized ledger input. No privileged access, key compromise, or consensus majority is required. The triggering condition (script absent from sub-tx `scriptsProvided`) is the default case for guard scripts, making this straightforwardly reachable.

---

### Recommendation

Replace the `Nothing -> acc` branch with a check that treats an unknown script hash as a Plutus script (the conservative assumption), or alternatively look up the script from the full aggregated scripts map (including top-level scripts) rather than only the sub-transaction's own `scriptsProvided`. A minimal fix:

```haskell
Nothing ->
  -- Script not locally provided; conservatively require SJust
  -- (guard scripts are Plutus scripts provided at the top level)
  case mbDatum of
    SJust _  -> acc
    SNothing -> Set.insert cred acc
```

A more principled fix would pass the full aggregated `ScriptsProvided` (top-level + sub-tx) to `validateGuardDatums` so that the script type can always be determined.

---

### Proof of Concept

1. Deploy a Plutus V3 guard script `sh` at the top level (in the top-level tx's witness set or via a reference input of the top-level tx).
2. Construct a sub-transaction with:
   - `requiredTopLevelGuards = Map.singleton (ScriptHashObj sh) SNothing`
   - No witness for `sh` in the sub-transaction's own witness set.
   - No reference input in the sub-transaction that resolves to `sh`.
3. Construct a top-level transaction that:
   - Embeds the sub-transaction.
   - Includes `ScriptHashObj sh` in `guardsTxBodyL`.
   - Provides `sh` in its own witness set.
   - Provides a redeemer for `GuardingPurpose` pointing to `sh`.
4. Submit the top-level transaction.

**Expected (correct) behavior:** `validateGuardDatums` should reject the sub-transaction with `SubMalformedGuardDatums {ScriptHashObj sh}` because a Plutus script credential has `SNothing` datum.

**Actual behavior:** `Map.lookup sh scripts` returns `Nothing` (script not in sub-tx's `scriptsProvided`), so `Nothing -> acc` is taken, the credential is not added to `malformed`, and the sub-transaction passes `validateGuardDatums` with no error. The guard Plutus script subsequently executes at the top level without the datum the sub-transaction was required to supply. [5](#0-4) [6](#0-5) [7](#0-6)
