### Title
Stale `getAssetUnstaking()` Valuation Allows Over-Commitment in `initiateWithdrawal()`, Causing Temporary Freezing of User Funds After Operator Slashing — (`contracts/NodeDelegator.sol`)

---

### Summary

`NodeDelegator.getAssetUnstaking()` reads the current withdrawable shares from EigenLayer's `getQueuedWithdrawals()` and converts them via `strategy.sharesToUnderlyingView()`. At the time a user calls `LRTWithdrawalManager.initiateWithdrawal()`, this value reflects the pre-slash amount. If the delegated operator is subsequently slashed before `completeUnstaking()` is called, the vault receives fewer tokens than the committed amount, and the user's withdrawal request cannot be unlocked until the rsETH price oracle reflects the slashing — temporarily freezing user funds.

---

### Finding Description

**Step 1 — `getAssetUnstaking()` returns current-maxMagnitude-scaled shares**

`NodeDelegator.getAssetUnstaking()` calls `_getDelegationManager().getQueuedWithdrawals(address(this))` and uses the second return value (`withdrawalShares`) directly:

```solidity
uint256 sharesToUnstake = withdrawalShares[withdrawalIndex][strategyIndex];
amount += strategyAsset == LRTConstants.ETH_TOKEN
    ? sharesToUnstake
    : strategy.sharesToUnderlyingView(sharesToUnstake);
``` [1](#0-0) 

EigenLayer's `getQueuedWithdrawals` returns the **current** withdrawable shares (i.e., `scaledShares × currentMaxMagnitude`). Before a slash, this equals the full queued amount. After a slash, it is reduced. The vulnerability window is between when `initiateWithdrawal()` is called (pre-slash) and when the slash occurs.

**Step 2 — `initiateWithdrawal()` commits based on inflated total**

`getAvailableAssetAmount()` calls `getTotalAssetDeposits()`, which sums `getAssetUnstaking()` across all NDCs:

```solidity
assetUnstakingFromEigenLayer += INodeDelegator(nodeDelegatorQueue[i]).getAssetUnstaking(asset);
``` [2](#0-1) 

```solidity
availableAssetAmount = totalAssets > assetsCommitted[asset] ? totalAssets - assetsCommitted[asset] : 0;
``` [3](#0-2) 

When a user calls `initiateWithdrawal()`, `assetsCommitted[asset]` is incremented by `expectedAssetAmount` computed from the pre-slash total:

```solidity
assetsCommitted[asset] += expectedAssetAmount;
``` [4](#0-3) 

**Step 3 — Slash reduces vault receipts**

`completeUnstaking()` calls EigenLayer's `completeQueuedWithdrawal`, which applies `SlashingLib.scaleForCompleteWithdrawal(scaledShares, slashingFactor)` internally. The NDC receives fewer tokens than the pre-slash `getAssetUnstaking()` reported, and transfers only the slashed amount to the vault:

```solidity
assets[i].safeTransfer(
    address(_getUnstakingVault()), assets[i].balanceOf(address(this)) - balancesBefore[i]
);
``` [5](#0-4) 

**Step 4 — `unlockQueue()` is bounded by actual vault balance**

`unlockQueue()` uses `unstakingVault.balanceOf(asset)` as `totalAvailableAssets`:

```solidity
totalAvailableAssets: unstakingVault.balanceOf(asset)
``` [6](#0-5) 

`_unlockWithdrawalRequests()` breaks if `availableAssetAmount < payoutAmount`:

```solidity
if (availableAssetAmount < payoutAmount) break;
``` [7](#0-6) 

`payoutAmount` is `min(request.expectedAssetAmount, currentRsETHPrice * rsETHUnstaked / assetPrice)`. If the rsETH price oracle has not yet been updated to reflect the slashing, `payoutAmount` remains at the pre-slash level (e.g., 100 ETH), while the vault only holds the slashed amount (e.g., 80 ETH). The request cannot be unlocked, and the user's funds are frozen.

---

### Impact Explanation

Users who called `initiateWithdrawal()` before the slash — committing based on the pre-slash `getAssetUnstaking()` value — have their withdrawal requests stuck in the queue. The freeze persists until the rsETH price oracle is updated to reflect the slashing event, at which point `payoutAmount` drops to match the available vault balance and `unlockQueue()` can proceed. This is a **temporary freezing of funds** (Medium impact per scope).

The `InsufficientLiquidityForWithdrawal` revert path described in the question is a secondary effect specific to ETH + Aave integration:

```solidity
if (balanceAfter < request.expectedAssetAmount) {
    revert InsufficientLiquidityForWithdrawal();
}
``` [8](#0-7) 

The primary and broader impact is the queue freeze, which affects all asset types.

---

### Likelihood Explanation

- EigenLayer operator slashing is a legitimate, non-adversarial protocol event (not "operator compromise").
- The LRT-rsETH protocol explicitly supports multiple operators across multiple NDCs, increasing exposure.
- The rsETH price oracle is updated off-chain by operators; there is a realistic delay between a slash event and oracle update.
- No attacker action is required — the freeze is a natural consequence of slashing during the withdrawal delay window.

---

### Recommendation

1. **In `getAssetUnstaking()`**: Apply a conservative discount or use `getWithdrawableShares()` (which already accounts for current maxMagnitude) rather than raw `sharesToUnderlyingView` on the returned shares.
2. **In `initiateWithdrawal()`**: Consider excluding in-flight EigenLayer withdrawals (`assetUnstakingFromEigenLayer`) from the available amount calculation, or apply a slashing haircut buffer.
3. **In `unlockQueue()`**: Allow partial unlocking at the slashed amount when `payoutAmount > availableAssetAmount` by using `min(payoutAmount, availableAssetAmount)` rather than breaking entirely.

### Proof of Concept

```
Fork test outline (Mainnet fork, post-slashing-upgrade):

1. Deploy/configure LRT-rsETH with one NDC delegated to operator O.
2. NDC deposits 100 ETH worth of stETH into EigenLayer strategy.
3. Operator calls initiateUnstaking() → queues withdrawal of 100 stETH.
4. User calls initiateWithdrawal(stETH, rsETH_for_100_stETH) → assetsCommitted += 100 stETH.
5. EigenLayer slashes operator O by 20% → maxMagnitude drops from 1e18 to 0.8e18.
6. Operator calls completeUnstaking() → vault receives 80 stETH (not 100).
7. Operator calls unlockQueue(stETH, ...) with current oracle prices (not yet updated).
   → totalAvailableAssets = 80 stETH
   → payoutAmount = 100 stETH (oracle stale)
   → 80 < 100 → break → request NOT unlocked
8. Assert: user's withdrawal request remains locked (temporary freeze).
9. Update rsETH oracle to reflect slashing → payoutAmount drops to 80 stETH.
10. Retry unlockQueue() → request unlocked, user receives 80 stETH.
```
