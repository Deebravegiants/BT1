### Title
Attacker-Controlled `fischlin_params_t.b` Bypasses Fischlin Security in `uc_dl_t::verify`, Allowing Trivially Forgeable DL Proofs in 2-Party DKG — (`src/cbmpc/zk/zk_ec.cpp`, `include-internal/cbmpc/internal/zk/fischlin.h`)

---

### Summary

`fischlin_params_t::convert` serializes `rho` and `b` over the wire but omits `t`. The verifier (`uc_dl_t::verify`) deserializes `params` from the received blob and derives `b_mask` directly from the attacker-supplied `b`. The only guard is `params.check()`, which enforces `b * rho >= SEC_P_COM` but does **not** enforce that `b` equals the expected value. A Byzantine P1 can send `b=1, rho=128` (product = 128 = `SEC_P_COM`, passes check), reducing the Fischlin bit-mask to `1` — meaning only 1 bit of each hash must be zero. This is trivially achievable by brute-force over `e`, requiring ~2 trials per iteration. P2 in `dkg_2p_t::step4_output_p2` accepts the forged proof and outputs a joint key `Q = Q1 + Q2` for a `Q1` whose discrete log P1 need not know.

---

### Finding Description

**Serialization gap — `t` is dropped:**

`fischlin_params_t::convert` only serializes `rho` and `b`: [1](#0-0) 

`t` is never sent or received. The receiver reconstructs `params` with whatever `rho` and `b` the sender chose.

**`uc_dl_t` deserializes `params` verbatim from the wire:** [2](#0-1) 

The default constructor sets `params{32, 4, 9}`, but after deserialization `params` is fully overwritten by the sender's values.

**`check()` only enforces `b * rho >= SEC_P_COM`, not `b == expected`:** [3](#0-2) 

`SEC_P_COM = 128`. With `b=1, rho=128`: `1 * 128 = 128 >= 128` — passes.

**`verify` derives `b_mask` from the attacker-supplied `b`:** [4](#0-3) 

`b_mask = (1 << 1) - 1 = 1`. The Fischlin condition at line 89–90 becomes `h & 1 == 0` — a 50% probability per trial, trivially satisfied by trying 2 values of `e` per iteration.

**`step4_output_p2` calls `pi_1.verify` with no independent parameter check:** [5](#0-4) 

There is no guard asserting `pi_1.params.b == 4` (or any expected value) before or after the call.

---

### Impact Explanation

P1 can commit to any `Q1` (including a point for which it does not know the discrete log) in step 1, then forge a `uc_dl_t` proof with `b=1` that passes `verify`. P2 accepts and outputs `Q = Q1 + Q2`. The DL proof — the sole mechanism ensuring P1 knows `x1` — provides no soundness guarantee. The resulting joint key is established without the required proof of knowledge, violating the DKG security property. This is a **High** impact validation bypass: attacker-controlled serialized data causes honest P2 to accept invalid cryptographic output from a public protocol-peer boundary.

---

### Likelihood Explanation

The attack requires only that a Byzantine P1 serialize a `uc_dl_t` struct with `params.rho=128, params.b=1` and construct 128 `(e_i, z_i)` pairs each satisfying `H(...) & 1 == 0`. Each pair requires ~2 hash evaluations. Total work: ~256 SHA-256 calls. No special knowledge or threshold collusion is needed — P1 is a single Byzantine participant, which is the standard adversary model for 2-party DKG.

---

### Recommendation

In `uc_dl_t::verify` (and analogously in `uc_batch_dl_finite_difference_impl_t::verify`), after deserializing `params`, assert that `params.b` and `params.rho` match the expected protocol constants before computing `b_mask`. The simplest fix is to add a check such as:

```cpp
if (params.b != expected_b || params.rho != expected_rho)
    return coinbase::error(E_CRYPTO, "uc_dl_t::verify: params mismatch");
```

Alternatively, do not serialize `params` at all — hardcode the expected values in the verifier and reconstruct `params` locally, so the verifier is never influenced by attacker-supplied security parameters.

2. For each i in [0, 128):
   a. Pick random r_i, compute A_i = r_i * G.
   b. Compute common_hash = H(G, Q1, A, sid, 1).
   c. For e_tag in {0, 1, 2, ...}:
      - Compute z_i = r_i + e_tag * x1 (mod q)  [or any z_i in range]
      - h = hash32bit_for_zk_fischlin(common_hash, i, e_tag, z_i) & 1
      - If h == 0: set e[i]=e_tag, z[i]=z_i, break.
      - Expected ~2 trials per i.
3. Serialize pi_1 (params.rho=128, params.b=1, A, e, z) into msg3.
4. P2 calls step4_output_p2:
   - com.id(sid1, p1_pid).open(Q1) → SUCCESS (commitment was valid)
   - pi_1.verify(Q1, sid, 1):
     - params.check(): 1*128=128>=128 → SUCCESS
     - b_mask = 1
     - For each i: h & 1 == 0 → SUCCESS (by construction)
     - Linear combination check passes (z_i chosen consistently)
   - Q = Q1 + Q2 returned → SUCCESS
5. P2 accepts a DL proof that required no knowledge of x1.
