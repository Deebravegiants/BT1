Audit Report

## Title
Publicly Callable `updateRSETHPrice()` and `FeeReceiver.sendFunds()` Enable Sandwich Attack to Steal Yield from rsETH Holders — (File: `contracts/LRTOracle.sol`)

## Summary
`LRTOracle.updateRSETHPrice()` is callable by any unprivileged user with no access control, and `FeeReceiver.sendFunds()` is similarly unrestricted. An attacker can deposit at a stale (pre-reward) rsETH price, trigger reward injection via `sendFunds()`, then trigger the price update — capturing a proportional share of accumulated yield that belongs to pre-existing rsETH holders. The attacker's profit comes directly from diluting the reward accrued to existing holders.

## Finding Description

**Root cause:** `updateRSETHPrice()` carries no role guard and is callable by anyone:

```solidity
// contracts/LRTOracle.sol:87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

`FeeReceiver.sendFunds()` is also unrestricted:

```solidity
// contracts/FeeReceiver.sol:53-58
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

Deposits use the stale cached `rsETHPrice` directly:

```solidity
// contracts/LRTDepositPool.sol:519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

Withdrawals similarly use the cached price:

```solidity
// contracts/LRTWithdrawalManager.sol:592-593
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

The price update computes `previousTVL = rsethSupply × rsETHPrice`. Because the attacker's deposit increases `rsethSupply` before the price update, the `previousTVL` baseline rises, which reduces the computed `rewardAmount` attributed to existing holders:

```solidity
// contracts/LRTOracle.sol:234,244-246
uint256 previousTVL = rsethSupply.mulWad(rsETHPrice);
...
if (!protocolPaused && totalETHInProtocol > previousTVL) {
    uint256 rewardAmount = totalETHInProtocol - previousTVL;
    protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
}
```

**Exploit flow:**
1. Attacker observes accumulated MEV rewards in `FeeReceiver` (publicly visible on-chain) and a stale `rsETHPrice`.
2. Attacker calls `LRTDepositPool.depositETH{value: X}(...)` — minted at the stale price, attacker receives rsETH proportional to the pre-reward TVL.
3. Attacker calls `FeeReceiver.sendFunds()` — moves accumulated rewards into the deposit pool, increasing TVL without updating the price.
4. Attacker calls `LRTOracle.updateRSETHPrice()` — the new price is computed over the enlarged supply (including attacker's shares), so the attacker's rsETH appreciates alongside existing holders' rsETH.
5. After the 8-day withdrawal delay, attacker redeems rsETH at the higher price, extracting a share of yield that should have accrued solely to pre-existing holders.

**Why existing guards fail:**
- `pricePercentageLimit` only reverts if the price increase exceeds the configured threshold. When rewards are within the band, the attack proceeds unimpeded. When `pricePercentageLimit == 0`, the guard is entirely absent (line 257: `pricePercentageLimit > 0 && ...`).
- The `minRSETHAmountExpected` slippage parameter in `depositETH` protects the depositor, not existing holders.
- The 8-day withdrawal delay defers profit but does not prevent it.

## Impact Explanation
**High — Theft of unclaimed yield.** Pre-existing rsETH holders accumulate yield through staking rewards and MEV fees. An attacker who deposits just before triggering the price update dilutes those holders' share of the pending reward. The attacker exits with more ETH than they deposited; the shortfall comes directly from yield owed to pre-existing holders. The attack scales linearly with deposit size and accumulated reward magnitude, and is repeatable every reward cycle.

## Likelihood Explanation
**Medium.** MEV/execution-layer rewards accumulate in `FeeReceiver` continuously and are publicly visible on-chain. Staking rewards increase `getEffectivePodShares()` automatically over time. No privileged role is required — any EOA or contract can execute all three steps. The only friction is the 8-day withdrawal delay, which defers but does not eliminate profit. The `pricePercentageLimit` guard reduces per-transaction profit but does not eliminate the attack, and is bypassable when the limit is unset.

## Recommendation
1. **Atomically refresh the price on every deposit and withdrawal.** Call `_updateRsETHPrice()` at the start of `depositETH`, `depositAsset`, and `initiateWithdrawal` so the price used for minting/sizing always reflects current TVL before the user's action changes supply.
2. **Restrict `FeeReceiver.sendFunds()` to an authorized role** (e.g., `MANAGER`) so reward injection cannot be weaponised as part of a sandwich.
3. As defence-in-depth, enforce a minimum deposit lock-up period so same-block or same-transaction deposit-then-price-update sequences are impossible.

## Proof of Concept

**Setup:**
- Protocol TVL = 10,000 ETH; rsETH supply = 10,000; `rsETHPrice` = 1.0 ETH (stale, not yet updated after rewards arrived).
- `FeeReceiver` holds 100 ETH in accumulated MEV rewards.
- Protocol fee = 10%.

**Attack steps:**

1. Attacker calls `LRTDepositPool.depositETH{value: 1000 ETH}(...)`.
   - `rsethAmountToMint = 1000 × 1.0 / 1.0 = 1000 rsETH` (stale price used).
   - New supply = 11,000 rsETH; TVL = 11,000 ETH; `rsETHPrice` still = 1.0.

2. Attacker calls `FeeReceiver.sendFunds()`.
   - 100 ETH moves to deposit pool; TVL = 11,100 ETH.

3. Attacker calls `LRTOracle.updateRSETHPrice()`.
   - `previousTVL = 11,000 × 1.0 = 11,000 ETH`.
   - `rewardAmount = 11,100 − 11,000 = 100 ETH`.
   - `protocolFeeInETH = 100 × 10% = 10 ETH`.
   - `newRsETHPrice = (11,100 − 10) / 11,000 ≈ 1.009 ETH`.

4. Attacker's 1,000 rsETH is now worth `1,000 × 1.009 = 1,009 ETH`.
   - Attacker profit: ~9 ETH extracted from the 90 ETH net reward pool.
   - Pre-existing holders (10,000 rsETH) receive only ~81 ETH of the 90 ETH net reward instead of the full 90 ETH.

**Foundry test plan:**
```solidity
function testSandwichYieldTheft() public fork {
    // 1. Seed FeeReceiver with 100 ETH (simulating accumulated MEV)
    vm.deal(address(feeReceiver), 100 ether);

    // 2. Record existing holders' rsETH value before attack
    uint256 priceBefore = lrtOracle.rsETHPrice();

    // 3. Attacker deposits 1000 ETH at stale price
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 1000 ether}(0, "");

    // 4. Attacker triggers reward injection
    vm.prank(attacker);
    feeReceiver.sendFunds();

    // 5. Attacker triggers price update
    vm.prank(attacker);
    lrtOracle.updateRSETHPrice();

    uint256 priceAfter = lrtOracle.rsETHPrice();

    // 6. Assert attacker's rsETH is worth more than deposited
    uint256 attackerRsETH = rsETH.balanceOf(attacker);
    uint256 attackerETHValue = attackerRsETH * priceAfter / 1e18;
    assertGt(attackerETHValue, 1000 ether, "Attacker extracted yield");

    // 7. Assert existing holders received less than full reward
    // Full net reward = 90 ETH for 10000 rsETH => price should be 1.009
    // With dilution, existing holders get less
    assertLt(priceAfter - priceBefore, 90 ether * 1e18 / 10_000 ether, "Yield diluted");
}
```
