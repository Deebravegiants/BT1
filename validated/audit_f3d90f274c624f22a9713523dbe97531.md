### Title
`finTransferExtension` Forwards Full `msg.value` to Wormhole After ETH Already Sent to Recipient, Permanently Breaking Native ETH Finalization — (`File: evm/src/omni-bridge/contracts/OmniBridgeWormhole.sol`)

### Summary

`OmniBridgeWormhole.finTransferExtension` passes the original `msg.value` to `_wormhole.publishMessage`, but `finTransfer` has already spent `payload.amount` of that ETH sending it to the recipient. After that spend, the contract no longer holds `msg.value` worth of ETH, so the Wormhole call either reverts (no pre-existing contract balance) or silently drains the contract's accumulated ETH reserve (pre-existing balance ≥ `payload.amount`).

### Finding Description

`OmniBridge.finTransfer` is `payable`. When `payload.tokenAddress == address(0)` (native ETH), it first transfers `payload.amount` ETH to the recipient:

```solidity
(bool success, ) = payload.recipient.call{value: payload.amount}("");
``` [1](#0-0) 

Then it calls `finTransferExtension(payload)`: [2](#0-1) 

In `OmniBridgeWormhole`, that override unconditionally forwards the **original, unmodified** `msg.value` to Wormhole:

```solidity
_wormhole.publishMessage{value: msg.value}(
    wormholeNonce,
    messagePayload,
    _consistencyLevel
);
``` [3](#0-2) 

At that point the contract's balance from this call is only `msg.value − payload.amount` (the Wormhole fee portion). Attempting to forward the full `msg.value` requires `payload.amount` extra ETH that must come from the contract's pre-existing balance.

The developers already understood this pattern for `initTransferExtension`, which correctly receives a pre-computed `value` parameter (= `msg.value − nativeFee`) and forwards only that:

```solidity
_wormhole.publishMessage{value: value}(...)
``` [4](#0-3) 

No equivalent subtraction exists for `finTransferExtension`.

### Impact Explanation

Two outcomes depending on the contract's pre-existing ETH balance:

1. **No pre-existing balance (normal state):** Every native-ETH `finTransfer` call reverts. The recipient never receives their ETH. The redemption path for native ETH bridged from NEAR to EVM via Wormhole is permanently frozen — a Critical irreversible fund-lock.

2. **Contract holds accumulated ETH** (from `receive()` or excess `msg.value` in other calls): The call succeeds but silently drains `payload.amount` from the contract's reserve on every native-ETH finalization. Repeated legitimate finalizations drain the entire contract ETH balance — a Critical unauthorized custody escape of protocol-held value.

### Likelihood Explanation

Any native ETH transfer initiated on NEAR and destined for an EVM chain using the Wormhole variant triggers this path. No special attacker privilege is required; the relayer (or any caller) supplies a valid MPC-signed payload obtained through the normal bridge flow. The bug fires on every such finalization.

### Recommendation

`finTransferExtension` should receive the residual value (after the ETH-to-recipient transfer) as an explicit parameter, mirroring `initTransferExtension`. In `OmniBridge.finTransfer`, compute `extensionValue = (payload.tokenAddress == address(0)) ? msg.value - payload.amount : msg.value` and pass it to `finTransferExtension`. In `OmniBridgeWormhole.finTransferExtension`, forward that parameter instead of `msg.value`.

### Proof of Concept

1. Deploy `OmniBridgeWormhole` with a real (or mock) Wormhole that charges `messageFee = F`.
2. Obtain a valid MPC signature for a native-ETH `finTransfer` with `payload.tokenAddress = address(0)` and `payload.amount = A`.
3. Call `finTransfer(sig, payload)` with `msg.value = A + F`.
4. The contract sends `A` ETH to `payload.recipient` (balance now `F`).
5. `finTransferExtension` calls `publishMessage{value: A + F}(...)` — requires `A + F` ETH but only `F` is available.
6. Transaction reverts; recipient never receives ETH. If the contract held `≥ A` ETH in reserve, the call succeeds and `A` ETH is drained from the reserve instead.
