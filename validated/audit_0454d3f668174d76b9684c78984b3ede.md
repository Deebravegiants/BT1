### Title
Meta-Transaction Deposit Refund Misdirected to Delegate Sender Instead of Relayer on Inner Receipt Failure - (File: runtime/runtime/src/actions.rs)

### Summary
In `apply_delegate_action`, the inner receipt spawned from a `DelegateAction` sets `predecessor_id` to `sender_id` (Alice/the delegate signer). When this inner receipt fails on the receiver's shard, the deposit refund is routed to `predecessor_id` = Alice — not to the relayer who actually paid the deposit. An unprivileged user can exploit this to extract deposited NEAR tokens from a relayer.

### Finding Description
When `apply_delegate_action` constructs the inner receipt from a `DelegateAction`, it sets:

```rust
let new_receipt = Receipt::V0(ReceiptV0 {
    predecessor_id: sender_id.clone(),   // Alice — the delegate signer
    receiver_id: delegate_action.receiver_id().clone(),
    ...
    receipt: ReceiptEnum::Action(ActionReceipt {
        signer_id: action_receipt.signer_id().clone(), // Relayer
        ...
    }),
});
``` [1](#0-0) 

The `balance_refund_receiver()` function returns `refund_to` if set, otherwise falls back to `predecessor_id`:

```rust
pub fn balance_refund_receiver(&self) -> &AccountId {
    self.refund_to().as_ref().unwrap_or_else(|| self.predecessor_id())
}
``` [2](#0-1) 

For `ReceiptEnum::Action` (V0), `refund_to()` always returns `&None` — there is no mechanism to override the refund destination:

```rust
ReceiptEnum::Action(_) | ... => &None,
ReceiptEnum::ActionV2(action_receipt_v2) => &action_receipt_v2.refund_to,
``` [3](#0-2) 

Therefore, when the inner receipt fails on the receiver's shard, `refund_unspent_gas_and_deposits` calls:

```rust
result.new_receipts.push(Receipt::new_balance_refund(
    receipt.balance_refund_receiver(),  // resolves to predecessor_id = Alice
    deposit_refund,
));
``` [4](#0-3) 

The relayer paid the deposit (deducted from its account at transaction processing time), but the refund goes to Alice. The protocol documentation explicitly acknowledges this broken invariant:

> "But what if the transaction fails execution on Bob's shard? At this point, the predecessor is `Alice` and therefore she receives the token balance refunded, not the relayer. This is something relayer implementations must be aware of since there is a **financial incentive for Alice to submit meta transactions that have high balances attached but will fail on Bob's shard**." [5](#0-4) 

The code comment in `apply_delegate_action` also acknowledges this:

> "If something goes wrong, deposit is refunded to the predecessor, this is sender_id/Sender in DelegateAction." [6](#0-5) 

### Impact Explanation
The relayer's NEAR token balance is permanently reduced by the full deposit amount, while Alice's balance increases by the same amount. This is a direct, concrete balance corruption: the relayer loses tokens it paid, and the attacker gains them. The corrupted protocol value is the relayer's account balance in the runtime state trie.

### Likelihood Explanation
The attack requires only that Alice submit a `DelegateAction` with a large attached deposit targeting a function call that will fail on the receiver's shard (e.g., calling a non-existent method, or a method that panics). This is the normal meta-transaction submission flow — Alice sends a `SignedDelegateAction` off-chain to a relayer. The relayer is expected to verify the action, but the protocol itself provides no protection: the deposit routing is wrong regardless of what the relayer checks. Any user interacting with a relayer service can trigger this.

### Recommendation
When `apply_delegate_action` constructs the inner receipt, it should either:
1. Set `predecessor_id` to the relayer (the outer receipt's `predecessor_id`) so that deposit refunds on failure return to the relayer, or
2. Use `ReceiptEnum::ActionV2` with `refund_to` set to the relayer's account ID, so `balance_refund_receiver()` returns the relayer rather than Alice.

The gas refund already correctly targets the relayer via `signer_id`. The deposit refund should follow the same logic.

### Proof of Concept
1. Alice creates a `DelegateAction` with `deposit: 10 NEAR` targeting `non_existing_method` on Bob's contract.
2. Alice signs it and submits it off-chain to a relayer.
3. The relayer wraps it in a transaction; the relayer's account is charged 10 NEAR for the deposit.
4. The outer receipt executes on Alice's shard: `apply_delegate_action` creates an inner receipt with `predecessor_id = Alice`, `signer_id = relayer`.
5. The inner receipt reaches Bob's shard; the function call fails.
6. `refund_unspent_gas_and_deposits` issues a deposit refund to `balance_refund_receiver()` = `predecessor_id` = Alice.
7. Alice receives 10 NEAR. The relayer permanently loses 10 NEAR.

This mirrors the external report exactly: an intermediary (the relayer) pays for a resource, but the refund is routed to the wrong party (Alice instead of the relayer) due to an incorrect address being used as the refund destination.
