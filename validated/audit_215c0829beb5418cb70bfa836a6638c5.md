### Title
Sub-transactions in Dijkstra era bypass ExUnits limits and fee requirements, enabling disproportionate validator CPU consumption — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`)

---

### Summary

The Dijkstra era introduces nested ("sub") transactions processed by the `SUBUTXO` rule. The `dijkstraSubUtxoTransition` function explicitly omits fee validation, ExUnits limit checks, and collateral requirements. If sub-transaction ExUnits are not aggregated into the outer transaction's fee and block-budget accounting, an attacker can craft a cheap outer transaction that embeds sub-transactions carrying maximum-ExUnits Plutus scripts, forcing validators to execute expensive scripts at a cost far below the intended minimum fee.

---

### Finding Description

The `dijkstraSubUtxoTransition` in `SubUtxo.hs` runs the following checks:

- `validateOutsideValidityIntervalUTxO`
- `validateOutsideForecast` ← only meaningful when redeemers are present, confirming sub-transactions can carry Plutus witnesses
- `validateOutputTooBigUTxO`
- `validateInputSetEmptyUTxO`
- `validateBadInputsUTxO`
- `validateOutputBootAddrAttrsTooBig`
- `validateOutputTooSmallUTxO`
- Network-ID checks

It does **not** call:
- `feesOK` — no minimum fee or collateral check
- `validateExUnitsTooBigUTxO` — no per-sub-transaction ExUnits cap
- `validateMaxTxSizeUTxO` — no sub-transaction size cap (the `SubMaxTxSizeUTxO` failure constructor exists but is never raised)
- `validateValueNotConservedUTxO` — no value-conservation check [1](#0-0) 

State is updated via `updateUTxOStateNoFees`, which is explicitly documented as "used for sub-transactions where fees are not applicable": [2](#0-1) 

The `dijkstraUtxoToDijkstraSubUtxoPredFailure` mapping hard-errors on `ExUnitsTooBigUTxO`, `FeeTooSmallUTxO`, `InsufficientCollateral`, `NoCollateralInputs`, and `TooManyCollateralInputs` with `error "Impossible: … for SUBUTXO"`, confirming these checks are structurally absent by design: [3](#0-2) 

The outer `UTXO` rule's `feesOK` (Alonzo/Babbage) gates collateral and ExUnits accounting on the **outer** transaction's redeemer map: [4](#0-3) 

If `totExUnits` for the outer transaction does not aggregate sub-transaction redeemer ExUnits, the outer transaction's fee and the block-level `maxBlockExUnits` budget are computed without accounting for sub-transaction script execution cost.

The `AlonzoEraTxWits era` constraint on the `STS (SUBUTXO era)` instance, combined with the presence of `validateOutsideForecast` (which is a no-op unless redeemers are present), strongly indicates sub-transactions can carry Plutus redeemers: [5](#0-4) 

---

### Impact Explanation

An attacker pays only the outer transaction's minimum fee (proportional to outer-tx size and outer-tx ExUnits) while embedding sub-transactions that each declare the maximum allowed ExUnits for Plutus scripts. Validators must execute all sub-transaction scripts. If sub-transaction ExUnits are not counted against `maxBlockExUnits`, the effective per-block script execution budget is unbounded, exceeding the intended validation limit. This matches the allowed impact: **attacker-controlled transactions exceed intended validation limits**.

---

### Likelihood Explanation

The Dijkstra era is the newest era in this repository and sub-transactions are a novel feature. Any user who can submit a valid outer transaction can embed sub-transactions. No privileged role, governance majority, or key compromise is required. The cost to the attacker is a single outer transaction fee.

---

### Recommendation

1. Aggregate sub-transaction ExUnits into the outer transaction's `totExUnits` computation so that `validateExUnitsTooBigUTxO` and `maxBlockExUnits` accounting cover the full execution budget consumed by the block.
2. Alternatively, add an explicit `validateExUnitsTooBigUTxO` call inside `dijkstraSubUtxoTransition` with a per-sub-transaction ExUnits cap derived from the protocol parameters.
3. Verify that `getMinFeeTxUtxo` for the outer transaction includes sub-transaction script fees; if not, extend the fee formula accordingly.

---

### Proof of Concept

1. Deploy a Plutus script that loops to consume the maximum ExUnits budget.
2. Construct a Dijkstra sub-transaction spending a UTxO locked by that script, with redeemers set to the maximum ExUnits.
3. Wrap it in an outer transaction whose own redeemer map is empty (zero ExUnits declared), keeping the outer fee minimal.
4. Submit the outer transaction. The outer `feesOK` and `validateExUnitsTooBigUTxO` pass because the outer redeemer map is empty. `dijkstraSubUtxoTransition` processes the sub-transaction without any ExUnits or fee check, forcing validators to execute the maximum-ExUnits script at the cost of only the outer transaction's base fee. dijkstraSubUtxoTransition = do
  TRC (SubUtxoEnv slot pp certState originalUtxo (IsValid isValid), utxoState, stAnnTx) <-
    judgmentContext
  let tx = stAnnTx ^. txStAnnTxG

  let txBody = tx ^. bodyTxL

  runTest $ Allegra.validateOutsideValidityIntervalUTxO slot txBody

  sysSt <- liftSTS $ asks systemStart
  ei <- liftSTS $ asks epochInfo
  runTest $ Alonzo.validateOutsideForecast ei slot sysSt tx

  let allSizedOutputs = txBody ^. allSizedOutputsTxBodyF
  let allOutputs = fmap sizedValue allSizedOutputs
  runTest $ Alonzo.validateOutputTooBigUTxO pp allOutputs

  runTest $ Shelley.validateInputSetEmptyUTxO txBody

  let inputs = txBody ^. inputsTxBodyL
  let refInputs = txBody ^. referenceInputsTxBodyL
  runTest $ Shelley.validateBadInputsUTxO originalUtxo (inputs `Set.union` refInputs)
  runTest $ Shelley.validateBadInputsUTxO (utxosUtxo utxoState) inputs

  runTestOnSignal $ Shelley.validateOutputBootAddrAttrsTooBig allOutputs

  runTestOnSignal $ Babbage.validateOutputTooSmallUTxO pp allSizedOutputs

  netId <- liftSTS $ asks networkId
  runTestOnSignal $ Shelley.validateWrongNetwork netId allOutputs
  runTestOnSignal $ Shelley.validateWrongNetworkWithdrawal netId txBody
  runTestOnSignal $ validateWrongNetworkInDirectDeposit netId txBody
  runTestOnSignal $ Alonzo.validateWrongNetworkInTxBody netId txBody

  if isValid
    then do
      newState <-
        Shelley.updateUTxOStateNoFees
          pp
          utxoState
          txBody
          certState
          (utxosGovState utxoState)
          (tellEvent . TotalDeposits (hashAnnotated txBody))
          (\a b -> tellEvent $ TxUTxODiff a b)
      pure $ newState & utxosDonationL <>~ txBody ^. treasuryDonationTxBodyL

      pure utxoState
