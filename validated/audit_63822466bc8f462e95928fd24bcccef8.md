### Title
Single Byzantine Participant Can Permanently Block All `verify_foreign_transaction` Requests via Empty Chain Registration — (File: `crates/contract/src/lib.rs`)

### Summary

The `get_supported_foreign_chains()` function computes the **strict intersection** of every active participant's registered foreign-chain sets. A single Byzantine participant (strictly below the signing threshold) can register an empty `SupportedForeignChains` set, collapsing the intersection to empty and causing every `verify_foreign_transaction` call to be rejected with `ForeignChainNotSupported`. This permanently blocks the entire foreign-chain verification request lifecycle until the participant re-registers or is removed via resharing.

### Finding Description

`get_supported_foreign_chains()` iterates `node_foreign_chain_support.foreign_chain_support_by_node` and keeps only chains for which **every** active participant has registered support: [1](#0-0) 

The filter condition `nodes_supporting_chain.is_superset(&active_participant_account_ids)` means that if any single participant's entry is missing a chain — or is entirely empty — that chain is excluded from the result.

`register_foreign_chain_support` accepts any `SupportedForeignChains` value, including an empty set, from any authenticated participant: [2](#0-1) 

`verify_foreign_transaction` gates on this computed set and panics if the requested chain is absent: [3](#0-2) 

The project's own design documentation explicitly acknowledges this root cause:

> "A single node that registers an empty list (or hasn't registered yet) drops **every** chain — one operator can take the whole feature down."



The analog to the external report is direct: just as a malicious ERC721 token reverting `approve()` blocks liquidation and halts epoch processing, a single Byzantine participant calling `register_foreign_chain_support({})` collapses the supported-chain intersection to empty and halts all foreign-chain verification request processing.

### Impact Explanation

All `verify_foreign_transaction` calls are rejected with `ForeignChainNotSupported` for every chain. Pending requests that were accepted before the attack time out without producing a signature. The Omnibridge inbound flow and any bridge service depending on `verify_foreign_transaction` is completely halted. This is a **request-lifecycle manipulation that breaks production safety/accounting invariants** — matching the Medium allowed impact scope.

### Likelihood Explanation

Any single active MPC participant can execute this with one on-chain transaction costing ~1 yoctoNEAR. No collusion, no leaked keys, and no privileged access beyond being a participant is required. The participant is Byzantine but strictly below the signing threshold (production threshold is a strict minority of the participant set). The call is idempotent and the damage persists until the participant re-registers or is removed via a full resharing ceremony.

### Recommendation

Replace the strict-intersection rule with the threshold-based **available-set** model already designed and tracked in the codebase: a chain is available if ≥ `signing_threshold` active participants cover it, not all of them. `verify_foreign_transaction` should gate on `get_available_foreign_chains()` (the threshold-based view) rather than `get_supported_foreign_chains()` (the intersection view). This is already proposed in `docs/design/calculating-supported-foreign-chains.md` and should be prioritised as a security fix, not merely a liveness improvement. [4](#0-3) 

### Proof of Concept

1. Network is Running with participants `[A, B, C]`, threshold = 2. All three have registered `{Bitcoin, Ethereum}` via `register_foreign_chain_support`. `get_supported_foreign_chains()` returns `{Bitcoin, Ethereum}`.
2. Participant `A` (Byzantine, below threshold) submits:
   ```
   register_foreign_chain_support({ foreign_chain_support: [] })
   ```
3. `node_foreign_chain_support.foreign_chain_support_by_node` now contains `A → {}`, `B → {Bitcoin, Ethereum}`, `C → {Bitcoin, Ethereum}`.
4. `get_supported_foreign_chains()` evaluates: for Bitcoin, `{B, C}` is not a superset of `{A, B, C}` → excluded. Same for Ethereum. Result: `{}`.
5. Any user calling `verify_foreign_transaction({ chain: Bitcoin, ... })` hits:
   ```
   env::panic_str("ForeignChainNotSupported { requested: Bitcoin }")
   ``` [5](#0-4) 
6. The foreign-chain verification feature is completely blocked. Participants `B` and `C` cannot override `A`'s registration. Recovery requires `A` to re-register or a full resharing to remove `A` from the participant set.

        &mut self,
        foreign_chain_support: dtos::SupportedForeignChains,
    ) -> Result<(), Error> {
        let account_id = self.voter_or_panic();

        self.node_foreign_chain_support
            .foreign_chain_support_by_node
            .insert(account_id, foreign_chain_support);

        Ok(())
    }
