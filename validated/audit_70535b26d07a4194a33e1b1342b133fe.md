### Title
Single Participant Can Indefinitely Delay Key Generation and Resharing via Repeated `vote_abort_key_event_instance` - (File: `crates/contract/src/state/key_event.rs`, `crates/contract/src/lib.rs`)

### Summary

A single Byzantine participant, strictly below the signing threshold, can indefinitely stall the MPC network's key generation (`Initializing` state) or key resharing (`Resharing` state) by repeatedly calling `vote_abort_key_event_instance` immediately after each new attempt is started by the leader. Unlike `vote_cancel_resharing` (which requires threshold votes), a single `vote_abort` immediately terminates the active key event instance with no quorum requirement.

### Finding Description

The `KeyEvent::vote_abort` method in `crates/contract/src/state/key_event.rs` terminates the active key event instance upon receipt of a **single** participant's abort vote. This is confirmed by the contract's own test suite:

```rust
// check that vote_abort immediately causes failure.
env.set_signer(candidates.iter().next().unwrap());
state.vote_abort(key_event).unwrap();
assert!(!state.resharing_key.is_active());
```

The public method `vote_abort_key_event_instance` is gated only by `assert_caller_is_attested_participant_and_protocol_active`, meaning any single attested participant — regardless of threshold — can call it. The leader-only restriction applies only to `start_keygen_instance` / `start_reshare_instance`, not to abort votes.

The attack loop is:

1. Leader calls `start_keygen_instance` / `start_reshare_instance` (attempt N).
2. Byzantine participant immediately calls `vote_abort_key_event_instance(attempt N)`.
3. The instance is destroyed; `next_attempt_id` increments.
4. Leader retries with attempt N+1.
5. Byzantine participant aborts attempt N+1.
6. Repeat indefinitely.

This is structurally identical to the ERC20 `distribute(1)` attack: a low-cost, single-actor call resets a time-bounded process, preventing it from ever completing. In the ERC20 case the reset target is `endTimestamp`; here it is the active `KeyEventInstance`.

The asymmetry with `vote_cancel_resharing` (which requires threshold votes) makes the abort path the weaker link: cancellation of the entire resharing requires quorum, but aborting individual attempts within that resharing requires only one participant.

### Impact Explanation

- **Key generation blocked**: During `Initializing` state, if keygen never completes, the MPC network never enters `Running` state and can never produce any signature. All user `sign()` calls will time out indefinitely.
- **Resharing blocked**: During `Resharing` state, participant set changes (adding/removing nodes, threshold changes, TEE-expiry-triggered evictions) can never finalize. The network is frozen at the previous participant set.
- Both outcomes break the contract's core execution-flow invariant: that a threshold of honest participants is sufficient to advance the protocol.

This matches the allowed impact: **Medium — contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.**

### Likelihood Explanation

Any single attested participant who turns Byzantine (e.g., a compromised node, a departing participant who wants to block their own removal via resharing, or an economically motivated attacker) can execute this attack. The attacker only needs to monitor the chain for `start_keygen_instance` / `start_reshare_instance` transactions and respond with `vote_abort_key_event_instance` in the next block. The cost is one NEAR transaction per attempt — negligible compared to the damage caused.

### Recommendation

Require a **threshold of abort votes** before terminating a key event instance, mirroring the quorum requirement for `vote_cancel_resharing`. Alternatively, introduce a **minimum active duration** before an abort vote is accepted (e.g., the instance must have been active for at least `K` blocks), so that a single participant cannot abort an instance that has not yet had time to complete. A third option is to track per-participant abort votes and only abort when `f+1` participants (where `f` is the maximum tolerated Byzantine count) have voted to abort.

### Proof of Concept

The contract's own test in `crates/contract/src/state/resharing.rs` demonstrates the single-vote abort:

```rust
// check that vote_abort immediately causes failure.
env.advance_block_height(1);
env.set_signer(&leader.0);
state.start(key_event.next_attempt(), 0).unwrap();
let key_event = state.resharing_key.current_key_event_id().unwrap();
// A single non-leader participant aborts the instance:
env.set_signer(candidates.iter().next().unwrap());
state.vote_abort(key_event).unwrap();
assert!(!state.resharing_key.is_active()); // instance is gone
```

```rust
    /// Start a new key event instance as the leader, if one isn't already active.
    /// The leader is always the participant with the lowest participant ID.
    pub fn start(&mut self, key_event_id: KeyEventId, timeout_blocks: u64) -> Result<(), Error> {
        self.cleanup_if_timed_out();
        if self.instance.is_some() {
            return Err(KeyEventError::ActiveKeyEvent.into());
        }
        let expected_key_event_id =
            KeyEventId::new(self.epoch_id, self.domain.id, self.next_attempt_id);
        if key_event_id != expected_key_event_id {
            return Err(KeyEventError::KeyEventIdMismatch.into());
        }
        self.verify_leader()?;
        self.instance = Some(KeyEventInstance::new(self.next_attempt_id, timeout_blocks));
        self.next_attempt_id = self.next_attempt_id.next();
        Ok(())
    }
```
