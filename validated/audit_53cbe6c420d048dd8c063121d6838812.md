### Title
Unchecked Chainlink Oracle Response: Missing Staleness and Validity Checks Allow Stale/Zero Price to Corrupt rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary

`ChainlinkPriceOracle.getAssetPrice` calls `latestRoundData()` but discards all validation fields (`updatedAt`, `answeredInRound`, `roundId`), and does not check that `price > 0`. A stale or zero Chainlink price is silently accepted and propagated into rsETH minting calculations, enabling depositors to mint rsETH at incorrect rates or causing deposits to produce zero rsETH.

### Finding Description

`ChainlinkPriceOracle.getAssetPrice` fetches the Chainlink price as follows:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52-54
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

Three fields returned by `latestRoundData` — `updatedAt` (staleness timestamp), `answeredInRound`, and `roundId` — are entirely ignored. No check is performed that `price > 0`.

By contrast, the same codebase's `ChainlinkOracleForRSETHPoolCollateral.getRate()` correctly validates all three conditions:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L30-32
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The stale price returned by `ChainlinkPriceOracle.getAssetPrice` flows directly into rsETH minting:

```solidity
// contracts/LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.getAssetPrice(asset)` delegates to `IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset)`, which resolves to `ChainlinkPriceOracle.getAssetPrice` for Chainlink-backed assets.

### Impact Explanation

**Stale price → protocol insolvency / theft from rsETH holders (Critical):** If the Chainlink oracle goes stale (e.g., due to network congestion or oracle downtime) while an LST depegs, the last reported (inflated) price remains in effect. A depositor can call `depositAsset` with depegged LST and receive rsETH computed at the stale pre-depeg price. This mints excess rsETH backed by less-than-equivalent collateral, diluting all existing rsETH holders and causing protocol insolvency.

**Zero price → deposit DoS / zero rsETH minted (Medium):** If `price == 0` is returned, `getAssetPrice` returns 0, making `getRsETHAmountToMint` return 0. The depositor's assets are transferred in but 0 rsETH is minted (or the call reverts on `minRSETHAmountExpected`), temporarily freezing deposits for the affected asset.

**Negative price → revert DoS (Medium):** If `price < 0`, casting to `uint256` produces a near-maximal value; the subsequent multiplication overflows and reverts under Solidity 0.8.x checked arithmetic, bricking `getAssetPrice` and all deposit paths for that asset.

### Likelihood Explanation

Chainlink oracle staleness is a documented, historically observed failure mode (network congestion, sequencer downtime on L2, oracle deprecation). The absence of any staleness check means the vulnerability is triggered passively whenever the oracle fails to update within its heartbeat window — no attacker action is required beyond submitting a normal deposit transaction during the stale window. The `minRSETHAmountExpected` slippage parameter does not mitigate this because the depositor sets it themselves and benefits from the inflated rate.

### Recommendation

Apply the same validation pattern already present in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

if (answeredInRound < roundId) revert StalePrice();
if (updatedAt == 0) revert IncompleteRound();
if (price <= 0) revert InvalidPrice();
// Optional: if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();
```

### Proof of Concept

1. Chainlink oracle for stETH/ETH goes stale at price `1e18` (1:1 peg) due to network congestion.
2. stETH depegs to `0.5e18` on-chain (real market price).
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
4. `getRsETHAmountToMint` calls `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns stale `1e18`.
5. `rsethAmountToMint = (1000e18 * 1e18) / rsETHPrice` — computed at the stale 1:1 rate.
6. Attacker receives rsETH worth ~1000 ETH while depositing stETH worth only ~500 ETH.
7. Existing rsETH holders are diluted; protocol becomes insolvent by ~500 ETH.
