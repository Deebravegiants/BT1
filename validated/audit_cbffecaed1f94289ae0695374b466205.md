Audit Report

## Title
Pool Admin Can Set Uncapped Bin Additional Fees Instantly to Frontrun Swaps — (`metric-core/contracts/MetricOmmPoolFactory.sol` / `metric-core/contracts/MetricOmmPool.sol`)

## Summary
`MetricOmmPoolFactory.setPoolBinAdditionalFees()` forwards caller-supplied `addFeeBuyE6`/`addFeeSellE6` values to the pool with no upper-bound cap and no timelock. A pool admin can atomically set either value to `type(uint16).max` (65 535 ≈ 6.55% in E6 units) and frontrun a pending user swap, causing the trader to pay up to 6.55% more input tokens than expected, with the excess retained as LP balance inside the bin.

## Finding Description
`setPoolBinAdditionalFees()` passes values directly to the pool: [1](#0-0) 

`setBinAdditionalFees()` performs only a bin-index range check before writing: [2](#0-1) 

These per-bin fees are added directly to the oracle-derived base fee at swap execution time for both buy directions: [3](#0-2) [4](#0-3) 

And for the sell direction: [5](#0-4) 

By contrast, `setPoolAdminFees()` enforces explicit caps before writing: [6](#0-5) 

No equivalent guard exists for bin additional fees. The pool admin is semi-trusted only within caps and timelocks; the complete absence of a cap for `addFeeBuyE6`/`addFeeSellE6` constitutes an admin-boundary break.

## Impact Explanation
Every swap crossing a manipulated bin pays up to `65535/1e6 ≈ 6.55%` additional fee on top of the oracle spread fee. The excess input is retained inside the bin as LP balance — a direct loss of user principal. This meets the "admin-boundary break: pool admin exceeds caps" criterion and the "direct loss of user principal" threshold.

## Likelihood Explanation
The pool admin role can act unilaterally and atomically in a single transaction. No timelock, no governance delay, and no cap guard exists between the admin and the fee change. A malicious admin can observe a large pending swap in the mempool and frontrun it in the same block. Even a non-malicious admin raising fees for legitimate reasons can accidentally frontrun a user whose transaction was already in flight. The attack is repeatable on every swap.

## Recommendation
1. **Add a hard cap** on `addFeeBuyE6` and `addFeeSellE6` in `setPoolBinAdditionalFees()`, analogous to `maxAdminSpreadFeeE6`/`maxAdminNotionalFeeE8` already enforced for pool-level admin fees:
   ```solidity
   if (addFeeBuyE6 > maxAdminBinFeeE6) revert AdminFeeTooHigh();
   if (addFeeSellE6 > maxAdminBinFeeE6) revert AdminFeeTooHigh();
   ```
2. **Add a timelock or two-step propose/execute pattern** for bin additional fee changes, consistent with the timelock already applied to price-provider updates (`proposePoolPriceProvider`/`executePoolPriceProviderUpdate`).

## Proof of Concept
1. Pool is live; `curBinIdx = 0`; `addFeeBuyE6 = 0`.
2. User submits `swap(recipient, false, 1_000e18, priceLimitX64, ...)` — buying token0 with token1.
3. Pool admin observes the pending transaction and frontruns with:
   ```solidity
   factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535);
   ```
4. User's swap executes with effective fee = `baseFee + 0.065535`, paying ~6.55% more token1 than expected for the same token0 output (exact-output) or receiving ~6.55% fewer token0 (exact-input).
5. The extra token1 is retained in the bin as LP balance; the user has no recourse.

  function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external
    onlyFactory
    nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)

    s.addFeeSellE6 = addFeeSellE6;

  }
