### Title
`vote_new_parameters` and `vote_add_domains` Lack Participant Membership Checks, Allowing Any NEAR Account to Cast Governance Votes — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `vote_new_parameters` and `vote_add_domains` functions in the MPC contract enforce only that the transaction signer equals the predecessor (`assert_caller_is_signer`), but perform **no check that the caller is a current protocol participant**. Any NEAR account — including one with no role in the MPC network — can submit votes for resharing proposals or domain additions. This is the direct analog of the `createGauge4Pool()` vulnerability: a state-mutating governance function callable by an unprivileged external account with unsanitized caller identity.

---

### Finding Description

`vote_new_parameters` (lib.rs line 886) and `vote_add_domains` (lib.rs line 948) both open with only:

```rust
Self::assert_caller_is_signer();
``` [1](#0-0) [2](#0-1) 

`assert_caller_is_signer` only verifies `signer_account_id == predecessor_account_id`, blocking contract-forwarded calls but **not** verifying participant membership: [3](#0-2) 

Compare this to `register_foreign_chain_support`, which correctly calls `voter_or_panic()` — a helper that both enforces the signer check **and** asserts the caller is a current participant: [4](#0-3) 

And `voter_or_panic` itself: [5](#0-4) 

The `vote_new_parameters` function does validate that the **proposed** participants have valid TEE attestations via `reverify_and_cleanup_participants`, but this validates the *proposal content*, not the *caller's identity*: [6](#0-5) 

The vote is then forwarded to the state-level `vote_new_parameters` in `running.rs`, which stores it keyed by `env::signer_account_id()`: [7](#0-6) 

Unlike `vote_update` — which explicitly filters the stored vote map to count only current participants before checking the threshold — the state-level `vote_new_parameters` path does not have a confirmed equivalent filter visible in the code reviewed. `vote_update`'s explicit filtering pattern is: [8](#0-7) 

If `ThresholdParametersVotes` counts **all** stored votes (not only those from current participants), an external attacker controlling enough NEAR accounts can accumulate votes toward the threshold without being a participant.

---

### Impact Explanation

If the state-level vote counting in `ThresholdParametersVotes` does not filter by current participant membership:

- An attacker controlling multiple NEAR accounts can submit votes for a `vote_new_parameters` proposal that reduces the active participant set (e.g., removing nodes that have valid TEE attestations but whose removal would weaken the network).
- If the attacker's votes count toward the threshold, the contract transitions to `Resharing` state with an attacker-chosen participant set, breaking the invariant that only current participants govern resharing.
- For `vote_add_domains`, an attacker could trigger key generation for an attacker-chosen domain configuration, corrupting the domain registry.

This maps to: **participant-state and contract execution-flow manipulation that breaks production safety/accounting invariants** (Medium impact scope).

Even if the state-level implementation does filter by participant membership, the missing check at the `lib.rs` entry point allows any account to write into `parameters_votes` storage, constituting state pollution and a latent vulnerability if filtering is ever relaxed.

---

### Likelihood Explanation

The entry path requires only a valid NEAR account and a direct (non-forwarded) call. No TEE, no key material, no threshold collusion is needed to *submit* the vote. The TEE attestation barrier applies only to the *proposed* participants, not to the caller. Any external account can reach this code path on mainnet today.

---

### Recommendation

Add a participant membership check at the start of both `vote_new_parameters` and `vote_add_domains`, consistent with the pattern used by `register_foreign_chain_support`:

```rust
pub fn vote_new_parameters(...) -> Result<(), Error> {
    let _voter = self.voter_or_panic(); // replaces bare assert_caller_is_signer
    ...
}

pub fn vote_add_domains(...) -> Result<(), Error> {
    let _voter = self.voter_or_panic(); // replaces bare assert_caller_is_signer
    ...
}
```

Additionally, confirm that `ThresholdParametersVotes` counts only votes from current participants (mirroring the explicit filter in `vote_update`) and add a test asserting that a non-participant call to `vote_new_parameters` is rejected.

---

### Proof of Concept

1. Attacker creates a fresh NEAR account `attacker.near` with no MPC role.
2. Attacker calls `vote_new_parameters` directly (signer == predecessor, satisfying `assert_caller_is_signer`) with a proposal that removes one legitimate participant but retains enough TEE-attested nodes to pass `reverify_and_cleanup_participants`.
3. The call succeeds and the vote is stored in `parameters_votes` keyed by `attacker.near`.
4. If `ThresholdParametersVotes` counts all stored votes, the attacker repeats from additional accounts until the threshold is reached, triggering a `Resharing` state transition with an attacker-chosen participant set — without any legitimate participant having voted.
5. The resharing proceeds, potentially locking out the removed participant and breaking the threshold-signature invariant.
