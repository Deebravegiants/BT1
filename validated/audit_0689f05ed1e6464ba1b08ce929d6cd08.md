### Title
Missing Transaction-Hash Binding in Starknet Receipt Verification Allows Single Malicious Provider to Forge Verification — (`crates/foreign-chain-inspector/src/starknet/inspector.rs`, `crates/foreign-chain-rpc-interfaces/src/starknet.rs`)

---

### Summary

`StarknetInspector::extract` queries a provider with `hash_A` but never verifies that the returned receipt actually belongs to `hash_A`. Because `GetTransactionReceiptResponse` carries no `transaction_hash` field, a single malicious provider can substitute the receipt of any canonical, succeeded transaction (`hash_B`) in response to a query for a non-existent `hash_A`. The fan-out's mismatch guard does not fire in this case because honest providers return a transient `ClientError` ("transaction not found") for the non-existent hash, and the fan-out explicitly ignores transient errors when at least one substantive `Ok` verdict exists.

---

### Finding Description

**Missing hash binding (root cause 1 & 2):**

`GetTransactionReceiptResponse` contains no `transaction_hash` field: [1](#0-0) 

`StarknetInspector::extract` sends the request with `hash_A` but performs zero cross-check on the response: [2](#0-1) 

After the RPC call returns, the code checks finality, canonicality of the block, and execution status — all of which are properties of the *returned* receipt, not of the *requested* hash: [3](#0-2) 

**Fan-out transient-error bypass (root cause 3 & 4):**

`ClientError` (the variant produced when a JSON-RPC server returns an error response, e.g. "Transaction hash not found") is classified as **transient**: [4](#0-3) 

The fan-out logic ignores transient errors when at least one `Ok` verdict exists. Specifically, `inspectors_split_between_success_and_failure` is only `true` when both `successes` and `non_transient_errors` are non-empty; transient errors are silently discarded: [5](#0-4) 

**Combined exploit path:**

1. Attacker submits a sign request for `hash_A` — a transaction hash that does not exist on Starknet.
2. The node's fan-out queries all configured providers with `hash_A`.
3. Honest providers return a JSON-RPC error ("Transaction hash not found") → `ClientError` → **transient** → silently dropped.
4. The attacker's provider returns the receipt of `hash_B` (a real, `AcceptedOnL1`, `Succeeded`, canonical transaction).
5. `verify_block_is_canonical` passes because `hash_B`'s block is genuinely canonical.
6. `execution_status == Succeeded` passes.
7. The fan-out sees `successes = [attacker_ok]`, `non_transient_errors = []` → returns `Ok(receipt_of_hash_B)`.
8. The node produces a signature share for a transaction it never verified.

Note: for a **reverted** `hash_A`, honest providers would return `TransactionFailed` (non-transient), which would trigger `InspectorResponseMismatch`. The bypass only works when `hash_A` is **absent** (non-existent), causing honest providers to emit the transient `ClientError`.

---

### Impact Explanation

A single malicious provider can cause an MPC node to produce a signature share for an arbitrary non-existent Starknet transaction, substituting the log/block-hash of any real canonical transaction. If the attacker's provider is used by a threshold of MPC nodes (e.g., a widely-used provider shared across operators), a full threshold signature is issued for a bridge operation that was never actually executed on Starknet, enabling invalid bridge execution or double-spend conditions.

---

### Likelihood Explanation

- The attacker needs to control one RPC provider that is configured by a threshold of MPC node operators. In the current pre-whitelist deployment, operators independently configure providers; a popular shared provider (e.g., a major RPC-as-a-service) used by enough operators satisfies this precondition without any threshold collusion among MPC participants.
- The attacker only needs to submit a sign request for a non-existent hash — trivially achievable by any user of the bridge contract.
- No cryptographic break, no TEE compromise, and no threshold collusion among MPC participants is required.

---

### Recommendation

1. **Add `transaction_hash` to `GetTransactionReceiptResponse`** and assert `receipt.transaction_hash == requested_hash` immediately after deserialization in `StarknetInspector::extract`. This is the primary fix and closes the binding gap regardless of fan-out behavior.

2. **Reclassify "transaction not found" as non-transient.** A JSON-RPC error meaning "this hash does not exist" is a deterministic, permanent answer — not a transient network failure. Either catch the specific Starknet error code and map it to `TransactionNotFound` (already defined as non-transient), or treat all `ClientError` variants that carry a JSON-RPC `Call` error as non-transient. This ensures honest providers' "not found" responses are treated as substantive verdicts that conflict with the attacker's `Ok`, triggering `InspectorResponseMismatch`.

// Mock ClientT that ignores the requested hash and always returns hash_B's receipt
struct ForgedProvider { canonical_receipt: GetTransactionReceiptResponse }

impl ClientT for ForgedProvider {
    async fn request<R, P>(&self, method: &str, _params: P) -> Result<R, Error>
    where R: DeserializeOwned {
        // Regardless of which hash was requested, return hash_B's receipt
        let json = serde_json::to_value(&self.canonical_receipt).unwrap();
        Ok(serde_json::from_value(json).unwrap())
    }
    // ...
}

#[tokio::test]
async fn forged_receipt_accepted_for_nonexistent_hash() {
    let canonical_receipt = GetTransactionReceiptResponse {
        block_hash: KNOWN_CANONICAL_BLOCK_HASH,
        block_number: KNOWN_CANONICAL_BLOCK_NUMBER,
        events: vec![/* real event from hash_B */],
        finality_status: StarknetFinalityStatus::AcceptedOnL1,
        execution_status: StarknetExecutionStatus::Succeeded,
    };

    // Fan-out: one honest provider (returns ClientError for nonexistent hash_A),
    //          one attacker provider (returns hash_B's receipt for any query)
    let honest = HonestProvider; // returns ClientError("Transaction hash not found")
    let attacker = ForgedProvider { canonical_receipt };
    let fan_out = FanOut::new(NonEmptyVec::new(vec![
        StarknetInspector::new(honest),
        StarknetInspector::new(attacker),
    ]));

    let result = fan_out.extract(
        NONEXISTENT_HASH_A,
        StarknetFinality::AcceptedOnL1,
        vec![StarknetExtractor::Log { log_index: 0 }],
    ).await;

    // Asserts Ok — the forged receipt from hash_B is returned as if it were hash_A's
    assert!(result.is_ok(), "fan-out accepted forged receipt: {:?}", result);
}
