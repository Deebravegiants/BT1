### Title
Gateway Skips `__validate__` Signature Check for Nonce-1 Invoke When Deploy-Account Is In Mempool — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` introduces a special case that unconditionally suppresses the `__validate__` entry-point call (and therefore all signature verification) for any invoke transaction whose nonce is 1 and whose sender address appears in the mempool's account-tracking state. An unprivileged attacker can exploit this to inject a signature-less invoke into the mempool for any victim account that has a pending `deploy_account` transaction, then use the mempool's fee-escalation path to evict the victim's legitimate nonce-1 invoke and replace it with the attacker's unsigned one. The batcher later rejects the unsigned transaction, but the victim's valid transaction is permanently lost.

---

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` (lines 429–461) fires when three conditions are simultaneously true:

1. The incoming transaction is an `Invoke` with `nonce == 1`.
2. The on-chain nonce read from `gateway_fixed_block_state_reader` is `0` (account not yet deployed).
3. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

When all three hold, the function returns `true`, which propagates back to `run_pre_validation_checks` and then to `run_validate_entry_point`: [2](#0-1) 

`ExecutionFlags { validate: false, … }` is set, and `StatefulValidator::perform_validations` short-circuits before calling `__validate__`: [3](#0-2) 

The code comment claims the mempool check is "sufficient" because the account must have a `deploy_account` or future-nonce transactions that already passed validation: [4](#0-3) 

This reasoning is incomplete. `account_tx_in_pool_or_recent_block` returns `true` for **any** account present in the mempool's `staged` or `committed` maps, or in the transaction pool: [5](#0-4) [6](#0-5) 

Critically, the `committed` map is populated for **any** account whose transaction was committed to a block — not only accounts whose `deploy_account` is still pending. The mempool retains these records across multiple blocks (controlled by `committed_nonce_retention_block_count`): [7](#0-6) 

The `ValidationArgs` struct passed to `mempool_client.validate_tx` carries no signature field, so the mempool's own validation also cannot catch the forgery: [8](#0-7) 

---

### Impact Explanation

**Admission of invalid transactions (High):** The gateway accepts an invoke transaction that carries an arbitrary or empty signature, because `__validate__` is never executed at the gateway layer. The transaction is forwarded to the mempool and queued for sequencing.

**Eviction of valid transactions (High):** The mempool's fee-escalation logic (`validate_fee_escalation`) accepts a replacement transaction for the same `(address, nonce)` pair if the incoming tip or gas price is higher. It does not inspect the signature. An attacker who pays a marginally higher fee can therefore replace the victim's legitimate nonce-1 invoke with the attacker's unsigned one. The victim's transaction is permanently removed from the mempool. [9](#0-8) 

When the batcher later executes the attacker's transaction, `AccountTransaction::new_for_sequencing` sets `validate: true`: [10](#0-9) 

`__validate__` is called, the signature check fails, and the transaction is rejected — but the victim's original invoke is already gone.

---

### Likelihood Explanation

The attack requires no privileged access. Any observer of the public mempool can:

1. Detect a pending `deploy_account` for victim address `A` (nonce 0, account nonce 0 in state).
2. Craft an invoke for `A` with nonce 1, arbitrary calldata, and a blank or random signature.
3. Set a fee slightly above the victim's nonce-1 invoke (if one exists) to trigger fee escalation.
4. Submit to the gateway — the three conditions in `skip_stateful_validations` are satisfied, `__validate__` is skipped, and the transaction is admitted.

The window is open for the entire duration that the `deploy_account` remains unconfirmed (potentially many blocks), making this reliably exploitable rather than a narrow race.

---

### Recommendation

**Short term:** Remove the `skip_stateful_validations` bypass entirely, or replace it with a check that verifies the signature against the class hash declared in the pending `deploy_account` transaction. The class hash is available in the mempool at the time of the check, so offline signature verification is feasible without requiring the account to be deployed.

**Long term:** Avoid special cases that suppress cryptographic validation. The UX goal (allowing a nonce-1 invoke to be submitted alongside a `deploy_account`) can be achieved by deferring the invoke's execution until after the `deploy_account` is confirmed, without skipping signature verification at admission time.

---

### Proof of Concept

```
# Precondition: victim Alice has submitted deploy_account for address A (nonce=0).
# A is now in the mempool; gateway state reader shows nonce=0 for A.

# Step 1 – attacker crafts a forged invoke
forged_invoke = InvokeV3 {
    sender_address: A,
    nonce: 1,
    calldata: <arbitrary>,
    signature: [],          # empty / random — will not be checked
    resource_bounds: { l2_gas: { max_price: alice_price + 1, ... } },  # outbid Alice
    ...
}

# Step 2 – attacker submits to gateway
POST /gateway/add_transaction  body=forged_invoke

# Gateway path:
#   get_nonce_from_state(A) → 0          (account not deployed)
#   validate_by_mempool(...)             (checks nonce/fee only, no signature)
#   skip_stateful_validations(...)
#     tx.nonce() == 1  ✓
#     account_nonce == 0  ✓
#     account_tx_in_pool_or_recent_block(A) == true  ✓  (deploy_account is in pool)
#   → skip_validate = true
#   run_validate_entry_point(skip_validate=true)
#     ExecutionFlags { validate: false }
#     __validate__ is NOT called
#   → transaction accepted, forwarded to mempool

# Step 3 – mempool fee-escalation evicts Alice's nonce-1 invoke (if present)
#   validate_fee_escalation: forged tip > Alice's tip → Alice's tx removed

# Step 4 – batcher executes the block
#   deploy_account(A, nonce=0) → A is deployed
#   forged_invoke(A, nonce=1) → __validate__ called with validate=true
#     signature check fails → transaction rejected
#   Alice's nonce-1 invoke is gone; she must resubmit
```
