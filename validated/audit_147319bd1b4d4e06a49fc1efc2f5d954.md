### Title
Silent `parse().ok()` on `fee_recipient` in Wormhole FinTransfer VAA Permanently Freezes Pending Transfer Fee — (`near/omni-prover/wormhole-omni-prover-proxy/src/parsed_vaa.rs`)

---

### Summary

A guardian-signed Wormhole FinTransfer VAA whose Borsh payload carries a syntactically invalid NEAR `AccountId` string in `fee_recipient` causes `claim_fee_callback` to panic **before** `remove_transfer_message` is called, permanently locking the pending transfer's fee with no alternative redemption path.

---

### Finding Description

**Step 1 — Silent `None` production in the prover proxy**

In `TryInto<FinTransferMessage> for ParsedVAA`:

```rust
Ok(FinTransferMessage {
    transfer_id: transfer.transfer_id,
    fee_recipient: transfer.fee_recipient.parse().ok(),   // ← silently None on bad input
    ...
})
``` [1](#0-0) 

`transfer.fee_recipient` is a raw `String` deserialized from the Borsh payload with no prior validation: [2](#0-1) 

`.parse::<AccountId>().ok()` converts any parse failure (uppercase letters, length > 64, invalid chars, etc.) into `None` without returning an error. The `TryInto` impl succeeds and returns `Ok(ProverResult::FinTransfer(...))` with `fee_recipient: None`. [3](#0-2) 

**Step 2 — Panic before state mutation in `claim_fee_callback`**

```rust
let fee_recipient = fin_transfer.fee_recipient.unwrap_or_else(|| {
    env::panic_str(BridgeError::FeeRecipientNotSetOrEmpty.to_string().as_str());
});
// ... checks ...
let transfer_message = self.remove_transfer_message(fin_transfer.transfer_id); // ← never reached
``` [4](#0-3) 

The `unwrap_or_else` panic fires at line 1083–1085. `remove_transfer_message` at line 1098 is never executed. The NEAR callback reverts with no state change; the `TransferMessage` remains in `pending_transfers` indefinitely.

**Step 3 — No alternative redemption path**

`sign_transfer_callback` only removes a transfer when `fee.is_zero()`: [5](#0-4) 

There is no admin/DAO function to forcibly remove a stuck pending transfer. The `finalised_transfers` set is populated by `fin_transfer_callback`, not `claim_fee_callback`, so the transfer is never marked finalized either.

---

### Impact Explanation

Every pending `TransferMessage` with a non-zero fee that is paired with a Wormhole FinTransfer VAA carrying an invalid `fee_recipient` string is permanently frozen. The fee tokens remain locked in the bridge contract with no callable path to release them. This satisfies the **Critical — Irreversible fund lock / permanently unclaimable protocol value** impact class.

---

### Likelihood Explanation

The source-chain contracts (EVM/Solana) encode `fee_recipient` as a raw byte string in the Borsh payload. If those contracts do not enforce NEAR `AccountId` validity rules (lowercase, ≤ 64 chars, valid charset) before emitting the event, any user who supplies a malformed string triggers this path. Wormhole guardians sign the event as-is; the NEAR prover proxy accepts any guardian-signed VAA. The only gating on the NEAR side is the `#[trusted_relayer]` check on `claim_fee`, meaning a trusted relayer must submit the VAA — but the relayer acts in good faith on a legitimately guardian-signed message. The root cause is entirely in the NEAR parsing layer.

---

### Recommendation

Replace the silent `.parse().ok()` with a hard error so that an invalid `fee_recipient` causes `verify_vaa_callback` to return `Err(...)` and the entire proof submission is rejected before any state is touched:

```rust
fee_recipient: if transfer.fee_recipient.is_empty() {
    None
} else {
    Some(transfer.fee_recipient.parse().map_err(stringify)?)
},
```

This mirrors the pattern already used for `recipient` in `TryInto<InitTransferMessage>`: [6](#0-5) 

---

### Proof of Concept

1. Craft a Borsh-encoded `FinTransferWh` payload where `fee_recipient = "INVALID_ACCOUNT!!"` (uppercase + special chars).
2. Wrap it in a mock `ParsedVAA` and call `parsed_vaa.try_into::<FinTransferMessage>()` — observe it returns `Ok` with `fee_recipient: None`.
3. Feed this `ProverResult::FinTransfer` into a unit test of `claim_fee_callback` — observe it panics at `unwrap_or_else` before any storage mutation.
4. Confirm `pending_transfers` still contains the entry after the panic.
5. Confirm no other public function can remove that entry when `fee > 0`.
