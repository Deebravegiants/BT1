Audit Report

## Title
`_fillBids` subtracts the wrong bin's `lengthE6` when walking downward, corrupting `binAvgExecPriceX64` and `cumulativeAvgExecPriceX64` for all bid bins below `curBinIdx` when bins have heterogeneous lengths — (`metric-periphery/contracts/lens/MetricOmmPoolDataProvider.sol`)

## Summary

In `_fillBids`, the downward-walking loop reads `lenAbove` from bin `b+1` and subtracts it from `walkDistE6` to derive the lower boundary of bin `b`. The correct subtraction is `lengthE6` of bin `b` itself. When bins have different `lengthE6` values, every `lowerX64`, `upperX64`, `mStartX64`, `mEndX64`, `binAvgExecPriceX64`, and `cumulativeAvgExecPriceX64` for bins below `curBinIdx` is wrong, corrupting the bid depth ladder returned by `getLiquidityDepth`.

## Finding Description

`walkDistE6` is initialized to `curBinDistFromProvidedPriceE6`, the lower boundary of `curBinIdx`. The loop at lines 492–526 walks downward:

```solidity
(,, uint16 lenAbove,,) = PoolStateLibrary._binState(pool, int8(b + 1)); // reads bin b+1
walkDistE6 -= int256(uint256(lenAbove));                                 // subtracts b+1's length

(, uint104 t1, uint16 lengthE6,, uint16 addFeeSellE6) = PoolStateLibrary._binState(pool, binIdx); // reads bin b
uint256 lowerX64 = _priceFromMidAndDistE6(midPriceX64, walkDistE6, ...);
uint256 upperX64 = _priceFromMidAndDistE6(midPriceX64, walkDistE6 + int256(uint256(lengthE6)), ...);
```

The invariant for contiguous bins is `lower_boundary(b) = lower_boundary(b+1) - lengthE6(b)`. The code instead computes `walkDistE6 = lower_boundary(b+1) - lengthE6(b+1)`, which equals `lower_boundary(b)` only when `lengthE6(b) == lengthE6(b+1)`. [1](#0-0) 

By contrast, `_fillAskRow` correctly advances `ctx.cumDistE6` by the current bin's own length: [2](#0-1) 

The error accumulates across iterations and can change sign, making the bid ladder non-monotone and internally inconsistent.

## Impact Explanation

`getLiquidityDepth` is the on-chain and off-chain integrator interface for reading bid depth. On-chain routers and aggregators call it to derive `priceLimitX64` before calling `swap`. With corrupted `binAvgExecPriceX64` / `cumulativeAvgExecPriceX64`:

- If the reported bid price is lower than actual, the integrator sets `priceLimitX64` below the true execution price, causing the swap to execute at a price worse than intended — direct loss of token1 output relative to expectation (bad-price execution).
- If the reported bid price is higher than actual, the integrator sets `priceLimitX64` above the true execution price, causing the swap to revert — broken core swap functionality.

Both outcomes are reachable depending on the sign of `lengthE6(b+1) - lengthE6(b)`, which is pool-configuration-dependent.

## Likelihood Explanation

Bins with heterogeneous `lengthE6` are a normal, supported pool configuration — the factory does not require uniform bin lengths. Any unprivileged caller (trader, router, aggregator, UI) invoking `getLiquidityDepth` on such a pool is affected. No privileged access or malicious setup is required; the bug triggers on any public call when at least two adjacent bid bins have different `lengthE6` values.

## Recommendation

Remove the separate read of bin `b+1` and instead subtract the current bin `b`'s own `lengthE6` from `walkDistE6`, mirroring the correct pattern in `_fillAskRow`:

```solidity
// Before (wrong):
(,, uint16 lenAbove,,) = PoolStateLibrary._binState(pool, int8(b + 1));
walkDistE6 -= int256(uint256(lenAbove));
(, uint104 t1, uint16 lengthE6,, uint16 addFeeSellE6) = PoolStateLibrary._binState(pool, binIdx);

// After (correct):
(, uint104 t1, uint16 lengthE6,, uint16 addFeeSellE6) = PoolStateLibrary._binState(pool, binIdx);
walkDistE6 -= int256(uint256(lengthE6));
```

## Proof of Concept

```solidity
function test_fillBids_wrongBoundaries_heterogeneousLengths() public {
    // Deploy pool with bins: curBinIdx=0 (lengthE6=10_000), bin -1 (lengthE6=20_000)
    // Seed liquidity in both bins
    // Call getLiquidityDepth(pool, 2)
    LiquidityDepth memory depth = dataProvider.getLiquidityDepth(address(pool), 2);

    // Bin -1 upper boundary should equal lower boundary of bin 0 = curBinDistFromProvidedPriceE6 (= 0 at mid)
    // Code computes walkDistE6 = 0 - 10_000 = -10_000 (wrong; should be 0)
    // upperX64 = price at -10_000 + 20_000 = +10_000 (above current bin's lower boundary — wrong)

    // Correct: lowerDist = -20_000, upperDist = 0, avg = -10_000
    // Code:    lowerDist = -10_000, upperDist = +10_000, avg = 0 (wrong)
    uint256 expectedAvg = _priceFromMidAndDistE6(midPrice, -10_000); // correct midpoint of bin -1
    assertEq(depth.bids[1].binAvgExecPriceX64, expectedAvg); // fails with current code
}
```
