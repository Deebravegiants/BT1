Audit Report

## Title
`_calculatePayoutAmount` silently reduces user payout below initiation-time `expectedAssetAmount` on adverse oracle price movements — (`contracts/LRTWithdrawalManager.sol`)

## Summary
`initiateWithdrawal` records an `expectedAssetAmount` computed from oracle prices at initiation time. `_calculatePayoutAmount`, called during `unlockQueue`, returns `min(expectedAtInitiation, currentReturn)`. If the rsETH/asset price ratio moves adversely between initiation and unlock through normal oracle operation, the stored `expectedAssetAmount` is overwritten with the lower value and the user receives fewer assets than the amount committed at initiation — without any slashing event occurring.

## Finding Description
In `initiateWithdrawal` (L168), `getExpectedAssetAmount` computes `rsETHUnstaked * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset)` at the current block and stores it as `request.expectedAssetAmount`. [1](#0-0) 

When the operator later calls `unlockQueue`, `_unlockWithdrawalRequests` calls `_calculatePayoutAmount` with oracle prices at unlock time:

```solidity
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
``` [2](#0-1) 

If `currentReturn < expectedAssetAmount` (rsETH price dropped or asset price rose), the function returns `currentReturn`, and L804 overwrites the stored field:

```solidity
request.expectedAssetAmount = payoutAmount; // payoutAmount < original expectedAssetAmount
``` [3](#0-2) 

`_processWithdrawalCompletion` then transfers the overwritten (reduced) amount to the user. [4](#0-3) 

The operator's `minimumRsEthPrice`/`maximumAssetPrice` bounds in `unlockQueue` only gate whether the unlock proceeds at all; within any accepted price window, a lower rsETH price or higher asset price produces `currentReturn < expectedAssetAmount` and the user receives less. [5](#0-4) 

The NatDoc for `initiateWithdrawal` mentions slashing as the only edge case for receiving less, but the code silently reduces payout on any adverse price movement. [6](#0-5) 

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.** Users receive fewer assets than the `expectedAssetAmount` committed and locked at initiation without any slashing event. The field name `expectedAssetAmount` and the `getExpectedAssetAmount` function name imply a guaranteed minimum, but the contract does not honour it when oracle prices move adversely during the withdrawal delay. No funds are lost from the protocol; only the user's expected return is silently reduced.

## Likelihood Explanation
The withdrawal delay is `8 days / 12 seconds` blocks (~57,600 blocks). [7](#0-6) 

Over an 8-day window, rsETH/asset price ratios routinely fluctuate due to LST rebasing, EigenLayer reward accrual, and market conditions. No privileged action or oracle compromise is needed — only a normal oracle update between initiation and unlock. Any user who initiates a withdrawal is exposed. Likelihood is **Medium**.

## Recommendation
At unlock time, pay out `max(expectedAtInitiation, currentReturn)` rather than `min(...)`, so users always receive at least the rate they were shown at initiation absent a genuine slashing event. Slashing can be handled as a separate, explicitly documented exception with its own code path. Alternatively, if the `min` behavior is intentional, rename `expectedAssetAmount` to `estimatedAssetAmount` and update the `initiateWithdrawal` NatDoc to clearly state that the amount is not a floor guarantee and can be reduced by adverse price movements during the delay period.

## Proof of Concept
```solidity
// Foundry fork/fuzz test outline:
// 1. Record oracle prices P_rsETH_init, P_asset_init at block N
// 2. User calls initiateWithdrawal(asset, rsETHAmount)
//    → expectedAssetAmount = rsETHAmount * P_rsETH_init / P_asset_init stored
// 3. vm.roll(block.number + withdrawalDelayBlocks + 1)
// 4. Mock oracle: set P_rsETH_unlock < P_rsETH_init (within operator's minimumRsEthPrice bound)
//    → currentReturn = rsETHAmount * P_rsETH_unlock / P_asset_init < expectedAssetAmount
// 5. Operator calls unlockQueue(asset, ..., minimumRsEthPrice=P_rsETH_unlock, ...)
//    → _calculatePayoutAmount returns currentReturn
//    → request.expectedAssetAmount overwritten with currentReturn
// 6. User calls completeWithdrawal(asset)
//    → user receives currentReturn < original expectedAssetAmount
// Assert: amountReceived < expectedAssetAmountAtInitiation  ← invariant broken without slashing
```
