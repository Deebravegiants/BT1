Audit Report

## Title
FIFO Queue `break` With No Cancel Mechanism Causes Temporary Freezing of User rsETH - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
`LRTWithdrawalManager` processes withdrawal requests in strict FIFO order. When `_unlockWithdrawalRequests` encounters a request it cannot service (insufficient vault assets or delay not elapsed), it `break`s out of the loop, leaving all subsequent requests permanently locked until the blocking request is serviced. No cancel or refund path exists anywhere in the contract, so users whose requests sit behind a blocking entry have their rsETH frozen in the contract with no user-accessible exit.

## Finding Description
`initiateWithdrawal` transfers rsETH from the caller into the contract and records a `WithdrawalRequest` in the FIFO queue, incrementing `assetsCommitted[asset]`: [1](#0-0) 

`_unlockWithdrawalRequests` iterates the queue and uses a hard `break` — not `continue` — on two conditions: the withdrawal delay has not elapsed, or the vault has insufficient assets for the front-of-queue request: [2](#0-1) 

Because `nextLockedNonce[asset]` is only advanced inside the loop body and written back after the loop exits, any request at position N that triggers a `break` leaves all requests at N+1, N+2, … with their nonces still `>= nextLockedNonce[asset]`. [3](#0-2) 

`_processWithdrawalCompletion` enforces that a request's nonce must be strictly less than `nextLockedNonce[asset]` before it can be completed, so those later requests are permanently blocked from completion: [4](#0-3) 

A grep search for `cancel`, `refund`, and `skip` across the entire contract returns no matches — there is no alternative exit path from the "pending" state.

## Impact Explanation
Any user whose withdrawal request is queued behind a request that cannot be unlocked has their rsETH held by the contract with no user-accessible exit. The user cannot recover their rsETH, cannot re-initiate a withdrawal for a different asset, and cannot force the queue to skip the blocking entry. This constitutes **temporary (potentially extended) freezing of user funds**, matching the allowed Medium impact: *"Temporary freezing of funds."* The duration of the freeze is bounded only by EigenLayer's unstaking cycle (days to weeks per cycle), making it materially harmful.

## Likelihood Explanation
The scenario requires no privileged access. `initiateWithdrawal` is open to any user. The precondition — a large request at the front of the queue with insufficient vault assets to cover it — arises naturally during normal operation: EigenLayer unstaking is asynchronous and the vault balance fluctuates. Any user can place a large request first (or simply be first in the queue), and any subsequent user is then exposed. The condition is repeatable and does not require attacker coordination; it can arise from ordinary usage patterns.

## Recommendation
Add a `cancelWithdrawal(address asset)` function that allows a user to remove their own pending (not yet unlocked) withdrawal request and recover their rsETH:
- Verify the caller has a pending request and that its nonce is `>= nextLockedNonce[asset]` (i.e., not yet unlocked).
- Remove the nonce from `userAssociatedNonces[asset][msg.sender]`.
- Delete the `withdrawalRequests[requestId]` entry.
- Decrease `assetsCommitted[asset]` by `request.expectedAssetAmount`.
- Transfer `request.rsETHUnstaked` back to the caller via `safeTransfer`.

This closes the only missing exit from the "pending" state without disrupting the FIFO ordering for other users.

## Proof of Concept
```solidity
// 1. User A initiates a large withdrawal — placed at nonce 0 (front of queue)
vm.prank(userA);
withdrawalManager.initiateWithdrawal(stETH, 1_000_000e18, "");

// 2. User B initiates a smaller withdrawal — placed at nonce 1
vm.prank(userB);
withdrawalManager.initiateWithdrawal(stETH, 1e18, "");

// 3. Unstaking vault holds only 0.5e18 stETH — enough for B, not for A
// unlockQueue iterates: nonce 0 → availableAssetAmount < payoutAmount → break
// nonce 1 is never reached; nextLockedNonce[stETH] remains 0
withdrawalManager.unlockQueue(stETH, 2, minAsset, minRsETH, maxAsset, maxRsETH);
// → rsETHBurned == 0, assetAmountUnlocked == 0

// 4. User B attempts to complete — reverts WithdrawalLocked
// because nonce 1 >= nextLockedNonce[stETH] (which is still 0)
vm.prank(userB);
withdrawalManager.completeWithdrawal(stETH, ""); // ← revert WithdrawalLocked

// 5. No cancelWithdrawal exists — B's rsETH is frozen until A's request
//    can be serviced (requires additional EigenLayer unstaking cycles)
```
The test can be run as a Foundry fork test against a mainnet/testnet deployment of `LRTUnstakingVault` with a controlled vault balance, or as a unit test with a mock vault returning a fixed `totalAssets` below the first request's `payoutAmount`.
