### Title
Irreversible Fund Lock When `ft_transfer`/`mint` Fails During `fin_transfer` Without a Message - (File: `near/omni-bridge/src/lib.rs`)

### Summary
When `fin_transfer` delivers tokens to a NEAR recipient without a message (`msg.is_empty()`), the `fin_transfer_send_tokens_callback` does not check whether the underlying `ft_transfer` or `mint` cross-contract call actually succeeded. If the token transfer fails (e.g., token contract paused, minting cap reached, or any other token-level restriction), the locked-token accounting is permanently decremented, the transfer is marked as finalized (preventing any retry), and the user's funds are irreversibly locked in the bridge with no on-chain recovery path.

### Finding Description

**Bug class**: State machine — the callback that handles the result of `send_tokens()` only inspects the promise result for the `ft_transfer_call` path. For the `ft_transfer` and `mint` (no-message) paths, it unconditionally treats any outcome as success.

**Execution flow in `process_fin_transfer_to_near()`**:

1. `unlock_tokens_if_needed()` decrements `locked_tokens` — this state change is committed to the NEAR state trie when the `fin_transfer_callback` receipt executes.
2. `add_fin_transfer()` inserts the transfer ID into `finalised_transfers` — also committed, preventing any future retry.
3. `send_tokens()` dispatches either `ft_transfer` (native tokens) or `mint` (deployed/bridged tokens) as a cross-contract call.
4. `.then(fin_transfer_send_tokens_callback(..., is_ft_transfer_call = !msg.is_empty(), ...))` is chained.

When `msg` is empty, `is_ft_transfer_call = false`. In `fin_transfer_send_tokens_callback()`, the refund/revert branch is gated on `is_refund_required(is_ft_transfer_call)`:

```rust
fn is_refund_required(is_ft_transfer_call: bool) -> bool {
    if is_ft_transfer_call {
        // checks promise result ...
    } else {
        false   // ← unconditional: no promise result check at all
    }
}
```

If `ft_transfer` or `mint` fails (promise result = `Err`), the callback still enters the `else` (success) branch, emits `FinTransferEvent`, and never calls `revert_lock_actions()`. The committed state changes — decremented `locked_tokens` and finalized transfer ID — are permanent.

**Contrast with the `ft_transfer_call` path**: when `msg` is non-empty, `is_ft_transfer_call = true` and `is_refund_required()` reads the promise result; if the receiver returned `U128(0)` (rejected), `revert_lock_actions()` and `remove_fin_transfer()` are called correctly. The no-message path has no equivalent guard.

### Impact Explanation

**For native NEAR tokens (e.g., wNEAR) bridged EVM → NEAR**:
- `unlock_tokens_if_needed()` permanently decrements `locked_tokens[Eth][token]`.
- The physical tokens remain in the bridge contract.
- The transfer ID is in `finalised_transfers`; re-submitting the same proof panics with `TransferAlreadyFinalised`.
- Result: tokens are irreversibly locked in the bridge with no on-chain recovery path.

**For deployed/bridged tokens (e.g., a bridged ERC-20) bridged EVM → NEAR**:
- `unlock_tokens_if_needed()` returns `LockAction::Unchanged` (no accounting change on NEAR side).
- `mint()` fails; the user never receives tokens.
- The EVM-side tokens are already burned or locked in the EVM bridge.
- The transfer ID is finalized; no retry is possible.
- Result: user's EVM assets are permanently destroyed/locked with no NEAR-side credit.

Both cases satisfy the allowed impact: **Critical — Irreversible fund lock, frozen redemption path, or permanently unclaimable user value in bridge flows.**

### Likelihood Explanation

The `ft_transfer` or `mint` call can fail under realistic, non-adversarial conditions:

- **Token contract paused**: Many production tokens (USDC, USDT, and others) implement a pause mechanism. If the token is paused between the storage-balance check and the actual transfer receipt execution, the transfer fails silently.
- **Minting cap / allowance**: Analogous to the USDC `mintingAllowedAmount` in the reference report — if the deployed token has a supply cap or per-minter allowance, `mint()` can revert.
- **Token contract upgrade or bug**: Any panic in the token contract during `ft_transfer` or `mint` triggers this path.
- **wNEAR withdrawal failure**: `near_withdraw_callback` panics on failure; `fin_transfer_send_tokens_callback` is chained after the entire `send_tokens()` promise and still receives `is_ft_transfer_call = false`, so no revert occurs.

No privileged role or attacker is required. Any ordinary user bridging tokens to NEAR is exposed if the token contract fails at delivery time.

### Recommendation

In `fin_transfer_send_tokens_callback()`, check the promise result of `send_tokens()` unconditionally — not only for the `ft_transfer_call` path. A minimal fix:

```rust
pub fn fin_transfer_send_tokens_callback(...) {
    // Check if the send_tokens promise itself failed
    let send_failed = match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
        Err(_) => true,
        Ok(_) if !is_ft_transfer_call => false,
        Ok(value) => serde_json::from_slice::<U128>(&value)
            .map(|v| v.0 == 0)
            .unwrap_or(false),
    };

    if send_failed {
        self.burn_tokens_if_needed(...);
        self.revert_lock_actions(&lock_actions);
        self.remove_fin_transfer(...);
        // emit FailedFinTransferEvent
    } else {
        // existing success branch
    }
}
```

Additionally, consider not committing `unlock_tokens_if_needed()` and `add_fin_transfer()` until after the token transfer is confirmed, or use a two-phase approach where the transfer ID is only finalized in the callback on confirmed success.

### Proof of Concept

**Scenario: native NEAR token (wNEAR) bridged EVM → NEAR, token contract paused at delivery**

1. User bridges 1000 wNEAR from Ethereum to NEAR. EVM bridge locks 1000 wNEAR; `locked_tokens[Eth][wNEAR] = 1000` on NEAR.
2. Relayer calls `fin_transfer()` on NEAR bridge.
3. `fin_transfer_callback()` → `process_fin_transfer_to_near()` executes:
   - `unlock_tokens_if_needed(Eth, wNEAR, 1000)` → `locked_tokens[Eth][wNEAR] = 0` **(committed)**
   - `add_fin_transfer(transfer_id)` → transfer marked finalized **(committed)**
   - `send_tokens(wNEAR, recipient, 1000, "")` → dispatches `ft_transfer(recipient, 1000)` cross-contract call
4. wNEAR contract is paused; `ft_transfer` receipt fails.
5. `fin_transfer_send_tokens_callback(transfer_message, fee_recipient, is_ft_transfer_call=false, ...)` runs:
   - `is_refund_required(false)` → `false` immediately
   - Enters success branch; emits `FinTransferEvent`
   - `revert_lock_actions()` is **never called**
6. Final state:
   - `locked_tokens[Eth][wNEAR] = 0` (permanently decremented)
   - `finalised_transfers` contains `transfer_id` (retry impossible)
   - 1000 wNEAR physically remain in bridge contract, inaccessible to the user
   - EVM-side tokens remain locked

        &mut self,
