### Title
`reset()` Called Before `last_sync_id` Validation Allows Malicious Harvester to Wipe Farmer's Plot State — (`chia/plot_sync/receiver.py`)

### Summary
In `Receiver._sync_started`, when `data.initial=True`, `reset()` is unconditionally called **before** the `last_sync_id` integrity check. `reset()` zeroes out `_last_sync.sync_id` and clears `_plots`, so the subsequent guard `data.last_sync_id != self._last_sync.sync_id` compares `0 != 0` and trivially passes. A malicious harvester peer with an established connection can exploit this to wipe the farmer's entire plot view for that harvester in a single message.

### Finding Description

The execution order in `_sync_started` is:

```
Line 210-211: if data.initial: self.reset()          ← destroys _last_sync and _plots
Line 212:     self._validate_identifier(...)          ← only checks message_id==0 (start=True path)
Line 213-214: if data.last_sync_id != self._last_sync.sync_id: raise  ← now 0!=0 → False, no raise
Line 215-216: if data.last_sync_id == data.identifier.sync_id: raise  ← 0==X → False if X≠0
``` [1](#0-0) 

`reset()` sets `self._last_sync = Sync()` (with `sync_id=uint64(0)`) and calls `self._plots.clear()`: [2](#0-1) 

`_validate_identifier` with `start=True` only checks `message_id_match`, not `sync_id_match`. After `reset()`, `_current_sync.next_message_id=0`, so `identifier.message_id=0` passes unconditionally: [3](#0-2) 

The `last_sync_id` guard at line 213 was designed to require the sender to prove knowledge of the previous sync's ID. But since `reset()` already zeroed `_last_sync.sync_id`, the attacker simply sends `last_sync_id=0` and the check passes. [4](#0-3) 

### Impact Explanation

After the call completes, `_plots` is empty and `_current_sync.sync_id = X` (attacker-chosen). The farmer's plot count for that harvester drops to zero mid-session. If the attacker follows up with a valid `PlotSyncDone`, the empty state is committed to `_last_sync`. Farming halts for that harvester; pool `plot_count` RPC returns 0 for it.

### Likelihood Explanation

Any peer that has established a HARVESTER-type connection to the farmer can send this message. No signature or shared secret is checked at the `_sync_started` layer — the only guard was the `last_sync_id` check, which is neutralized by the ordering bug. A malicious harvester operator in a multi-harvester farm (e.g., a pool participant running their own harvester pointed at a shared farmer) can trigger this with a single crafted protocol message.

### Recommendation

Capture `_last_sync.sync_id` before calling `reset()`, and validate `data.last_sync_id` against the **pre-reset** value:

```python
async def _sync_started(self, data: PlotSyncStart) -> None:
    expected_last_sync_id = self._last_sync.sync_id   # capture BEFORE reset
    if data.initial:
        if data.last_sync_id != expected_last_sync_id:
            raise InvalidLastSyncIdError(data.last_sync_id, expected_last_sync_id)
        self.reset()
    self._validate_identifier(data.identifier, True)
    if not data.initial and data.last_sync_id != self._last_sync.sync_id:
        raise InvalidLastSyncIdError(data.last_sync_id, self._last_sync.sync_id)
    ...
```

### Proof of Concept

```python
# Receiver pre-state: _plots has N entries, _last_sync.sync_id = 5
receiver._plots = {"plot1.plot": ..., "plot2.plot": ...}
receiver._last_sync.sync_id = uint64(5)

# Attacker sends:
msg = PlotSyncStart(
    identifier=PlotSyncIdentifier(timestamp=now, sync_id=6, message_id=0),
    initial=True,
    last_sync_id=uint64(0),   # does NOT match pre-reset _last_sync.sync_id=5
    plot_file_count=uint32(0),
    harvesting_mode=uint8(0),
)
await receiver.sync_started(msg)

# After the call:
assert len(receiver.plots()) == 0   # _plots wiped
assert receiver._last_sync.sync_id == 0  # reset to zero
# No error response was sent to the attacker
```

The `last_sync_id=0` would have been rejected by `InvalidLastSyncIdError` if checked against the pre-reset value of 5, but `reset()` runs first, making the check compare `0 != 0` → no error.
