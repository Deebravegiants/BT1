Audit Report

## Title
Permissionless `updateRSETHPrice()` Enables Sandwich Attack to Extract Staking Yield - (File: contracts/LRTOracle.sol)

## Summary
`LRTOracle.updateRSETHPrice()` carries no access-control modifier beyond `whenNotPaused`, allowing any caller to trigger a price update. When `isInstantWithdrawalEnabled` is active for an asset, an attacker can atomically deposit at the stale (lower) price, trigger the oracle update, and instantly withdraw at the new (higher) price, extracting a portion of the accumulated staking yield that should have accrued to all rsETH holders.

## Finding Description
`updateRSETHPrice()` is unconditionally public:

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

`_updateRsETHPrice()` computes the new price as `(totalETHInProtocol − protocolFeeInETH).divWad(rsethSupply)` and writes it to the `rsETHPrice` storage slot (L250, L313). Both user-facing price-sensitive operations read this same slot:

- **Deposit** (`LRTDepositPool.getRsETHAmountToMint`, L520): `rsethAmountToMint = (amount * assetPrice) / rsETHPrice`
- **Instant withdrawal** (`LRTWithdrawalManager.getExpectedAssetAmount`, L593): `underlyingToReceive = rsETHUnstaked * rsETHPrice / assetPrice`

The exploit path when `isInstantWithdrawalEnabled[asset] == true`:

1. **Front-run:** Attacker calls `LRTDepositPool.depositETH{value: X}(...)`. The deposited ETH lands in `LRTDepositPool` and is immediately counted in `totalETHInProtocol` via `getETHDistributionData()` → `address(this).balance`. rsETH is minted at the stale `rsETHPrice`.

2. **Trigger:** Attacker calls `LRTOracle.updateRSETHPrice()`. `_getTotalEthInProtocol()` now includes the attacker's X ETH in the deposit pool balance. The new price is:

   `newPrice = (T + X) / (S + X/oldPrice)`

   where T = pre-deposit TVL (inflated by accumulated rewards), S = pre-deposit rsETH supply. Because `oldPrice < T/S`, the attacker's deposit dilutes the full price jump, but `newPrice` is still strictly greater than `oldPrice`, making the withdrawal profitable.

3. **Back-run:** Attacker calls `LRTWithdrawalManager.instantWithdrawal(asset, rsETHMinted, "")`. `getExpectedAssetAmount` reads the freshly written `rsETHPrice` (now `newPrice`), burns the attacker's rsETH, and redeems ETH from `LRTUnstakingVault`.

**Why existing guards are insufficient:**

- `pricePercentageLimit`: Only fires when `newRsETHPrice > highestRsethPrice` AND the increase exceeds the limit. If `pricePercentageLimit == 0` (no cap), or the accumulated yield is within the configured limit, the public call succeeds. The attacker can also wait for a period where the price is recovering from a prior dip (below `highestRsethPrice`), in which case the guard never triggers regardless of the jump size (L252-266).
- `instantWithdrawalFee`: Reduces profit but does not eliminate it; a sufficiently large price increment (e.g., after infrequent updates) makes the attack net-positive.
- `nonReentrant` on both `depositETH` and `instantWithdrawal`: Prevents re-entrancy within a single call but does not prevent a multi-call atomic bundle via a private relay.
- `getAssetsAvailableForInstantWithdrawal`: Caps the withdrawal to vault balance minus `queuedWithdrawalsBuffer`, limiting per-attack scale but not preventing the attack.

## Impact Explanation
**High — Theft of unclaimed yield.**

The attacker captures the yield increment (the gap between `oldPrice` and `newPrice`) on their deposited principal. This profit is extracted directly from `LRTUnstakingVault`, reducing the ETH available for legitimate withdrawers and diluting the yield that should have been shared across all rsETH holders. The attack is repeatable every time rewards accumulate and the price has not yet been updated.

## Likelihood Explanation
**Medium.**

- `updateRSETHPrice()` is callable by any EOA or contract with no privilege requirement.
- Staking rewards accumulate continuously and predictably; the price increment is observable off-chain by monitoring EigenLayer strategy balances.
- The attack requires `isInstantWithdrawalEnabled[asset] == true` and sufficient ETH in `LRTUnstakingVault` — both are normal operating conditions when the instant-withdrawal feature is live.
- All three steps can be bundled atomically via Flashbots or another private relay, leaving no mempool footprint and requiring no flash loan.
- If `pricePercentageLimit == 0` or the accumulated yield is within the limit, the public call is unconditionally available.

## Recommendation
1. **Restrict `updateRSETHPrice()` to authorized callers** (e.g., `onlyLRTManager` or a dedicated keeper role). The existing `updateRSETHPriceAsManager()` already demonstrates the intended privileged path; the public entry point should be removed or gated identically.
2. **Add a deposit/withdrawal cooldown**: rsETH minted in block N should not be redeemable via `instantWithdrawal` until block N+k (e.g., k ≥ 1), breaking the atomicity of the sandwich.
3. **Enforce a minimum `instantWithdrawalFee`** that exceeds the maximum single-update price increment permitted by `pricePercentageLimit`, ensuring the attack is never net-profitable.
4. **Update the oracle frequently** (at least once per day) to minimize the accumulated yield available for extraction in a single sandwich.

## Proof of Concept
Assume: `rsETHPrice` (stale) = 1.01 ETH, accumulated rewards make the fair price 1.02 ETH, `isInstantWithdrawalEnabled[ETH] = true`, `instantWithdrawalFee` = 5 bps, `pricePercentageLimit = 0`, `LRTUnstakingVault` holds ≥ 1010 ETH.

**Step 1 — Deposit at stale price (front-run):**
```
LRTDepositPool.depositETH{value: 1000 ETH}(0, "")
// rsETH minted = 1000 / 1.01 ≈ 990.099 rsETH
// Attacker's 1000 ETH now in LRTDepositPool.balance
```

**Step 2 — Trigger price update:**
```
LRTOracle.updateRSETHPrice()
// totalETHInProtocol includes attacker's 1000 ETH
// newPrice = (T + 1000) / (S + 990.099)  [between 1.01 and 1.02, e.g. ~1.01991 for large S]
```

**Step 3 — Instant withdrawal at new price (back-run):**
```
LRTWithdrawalManager.instantWithdrawal(ETH_TOKEN, 990.099e18, "")
// ETH returned = 990.099 × 1.01991 ≈ 1009.82 ETH (from LRTUnstakingVault)
// Fee (5 bps) ≈ 0.505 ETH
// Net received ≈ 1009.31 ETH
// Profit ≈ 9.31 ETH extracted from other rsETH holders' yield
```

All three calls are bundled in a single Flashbots bundle. The attack is repeatable each time rewards accumulate. A Foundry fork test can confirm by: (1) forking mainnet, (2) simulating reward accrual by directly increasing EigenLayer strategy share values, (3) executing the three-step sequence from an attacker EOA, and (4) asserting `attackerETHAfter > attackerETHBefore + gasUsed`.
