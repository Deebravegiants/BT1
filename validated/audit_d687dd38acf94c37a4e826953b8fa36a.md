### Title
Caller-Agnostic `verify_foreign_transaction` Request Key Enables Queue Saturation Attack Blocking Bridge Transaction Verification - (File: `crates/contract/src/pending_requests.rs`)

---

### Summary

The `VerifyForeignTransactionRequest` struct used as the map key in `pending_verify_foreign_tx_requests` contains no caller identity — only `request`, `domain_id`, and `payload_version`. Combined with the hard cap of `MAX_PENDING_REQUEST_FAN_OUT = 128` entries per key, an unprivileged attacker can pre-fill the queue for any specific foreign-chain transaction before the legitimate bridge user submits, causing the user's `verify_foreign_transaction` call to panic with `PendingRequestQueueFull`. The attacker can sustain this at negligible cost (128 × 1 yoctonear per refill cycle), permanently blocking a targeted bridge deposit from being processed.

---

### Finding Description

**Root cause — caller-agnostic request key:**

`VerifyForeignTransactionRequest` is defined without any caller field:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,   // contains tx_id — fully public
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [1](#0-0) 

This is the exact struct used as the `LookupMap` key in `pending_verify_foreign_tx_requests`. The contract's own test acknowledges the design:

> "a different account would today be blocked from receiving a response by alice's submission" [2](#0-1) 

**Root cause — bounded queue with panic on overflow:**

`push_pending_yield` enforces a hard cap of `MAX_PENDING_REQUEST_FAN_OUT = 128` and panics (aborting the transaction) when exceeded:

```rust
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

pub(crate) fn push_pending_yield<K>(...) {
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(&RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }.to_string());
    }
    queue.push(YieldIndex { data_id });
}
``` [3](#0-2) 

**Contrast with `sign` and `request_app_private_key`:**

For `sign`, the `SignatureRequest` key includes a `tweak` derived from `(predecessor_id, path)`, making it caller-specific — an attacker cannot pre-fill another user's queue:

```rust
pub fn new(domain: DomainId, payload: Payload, predecessor_id: &AccountId, path: &str) -> Self {
    let tweak = crate::kdf::derive_tweak(predecessor_id, path);
    ...
}
``` [4](#0-3) 

For `request_app_private_key`, the `CKDRequest` key includes `app_id` derived from `(predecessor_id, derivation_path)` and an ephemeral `app_public_key` unknown to the attacker: [5](#0-4) 

Only `verify_foreign_transaction` uses a fully public, caller-agnostic key.

**Attack path:**

The `verify_foreign_transaction` function converts the args to a `VerifyForeignTransactionRequest` and enqueues a yield under that key:

```rust
let request = args_into_verify_foreign_tx_request(request);
...
move |this, id| this.add_verify_foreign_tx_request(request, id)
``` [6](#0-5) 

Because the key is derived entirely from the foreign-chain tx_id (public on-chain), `domain_id`, and `payload_version`, an attacker can compute it before the victim submits.

---

### Impact Explanation

`verify_foreign_transaction` is the primary mechanism for the Omnibridge inbound flow: a user deposits funds on a foreign chain (Bitcoin, Ethereum, etc.) and calls `verify_foreign_transaction` to obtain the MPC signature that releases funds on NEAR.

An attacker who observes the foreign-chain tx_id (available as soon as the transaction is broadcast to the mempool — minutes before confirmation) can:

1. Submit 128 `verify_foreign_transaction` calls for that tx_id at 1 yoctonear each (128 yoctonear ≈ $0 cost).
2. The victim's subsequent call panics with `PendingRequestQueueFull`.
3. As queued yields time out, the attacker refills the queue, sustaining the block indefinitely.

The victim's bridge deposit is stuck on the foreign chain for as long as the attacker maintains the attack. This breaks the core production safety invariant of the bridge: *a confirmed foreign-chain deposit must be processable on NEAR*. The impact is **Medium** per the allowed scope: request-lifecycle manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

- Foreign-chain tx_ids are fully public the moment a transaction enters the mempool.
- For Bitcoin, the window between broadcast and confirmation is ~10–60 minutes — ample time to pre-fill the queue.
- The attack costs 128 yoctonear per refill cycle, making sustained blocking economically trivial.
- No privileged access, TEE compromise, or threshold collusion is required — any NEAR account can call `verify_foreign_transaction`.

---

### Recommendation

Include the caller's `predecessor_account_id` in the `VerifyForeignTransactionRequest` key (analogous to how `SignatureRequest` includes the `tweak` derived from the caller). This makes each queue entry caller-specific, preventing cross-caller queue saturation:

```rust
pub struct VerifyForeignTransactionRequest {
    pub caller: AccountId,          // add this
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
```

Alternatively, enforce a per-caller rate limit or per-caller sub-queue so one account cannot exhaust the shared cap for a given tx_id.

---

### Proof of Concept

```
1. Alice broadcasts Bitcoin tx_id = 0xDEADBEEF... (public in mempool).

2. Attacker observes tx_id and submits 128 calls to verify_foreign_transaction
   with {request: Bitcoin(tx_id=0xDEADBEEF...), domain_id: X, payload_version: V1}
   at 1 yoctonear each. Total cost: 128 yoctonear ≈ $0.

3. pending_verify_foreign_tx_requests[key].len() == 128.

4. Alice waits for Bitcoin confirmation (~10 min) and submits
   verify_foreign_transaction for the same tx_id.

5. push_pending_yield panics:
   "Pending-request queue is full for this request key (limit: 128)."
   Alice's transaction is aborted; her bridge deposit is stuck.

6. Attacker monitors the queue. As yields time out, attacker refills to 128.
   Alice can never submit successfully. Her funds remain locked on Bitcoin.
```

/// Maximum number of concurrent yield-resume promises that can be queued for a single
/// request key (i.e. the number of duplicate submissions whose responses fan out from
/// one MPC reply).
///
/// The ceiling is needed because `respond*` drains the entire queue in one call: every
/// queued yield triggers a host-side `promise_yield_resume`, paid for out of the
/// responder's 300 TGas budget. Without a cap, an attacker could enqueue enough
/// duplicates to make `respond*` run out of gas and strand every queued caller.
///
/// 128 is validated empirically by the sandbox test
/// `test_contract_request_duplicate_requests_fan_out`, which fills the queue to this
/// cap across all four signature schemes and confirms `respond*` drains it inside its
/// 300 TGas budget.
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
```
