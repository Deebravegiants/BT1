Audit Report

## Title
`_checkAndUpdateDailyFeeMintLimit` revert permanently deadlocks `_updateRsETHPrice`, freezing `rsETHPrice` and enabling yield dilution - (File: `contracts/LRTOracle.sol`)

## Summary
`LRTOracle._updateRsETHPrice()` calls `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` before writing `rsETHPrice = newRsETHPrice`. When the single-update fee exceeds `maxFeeMintAmountPerDay`, the call reverts with `DailyFeeMintLimitExceeded`, leaving `rsETHPrice` permanently frozen at a stale lower value. The deadlock is self-reinforcing because the stale price understates `previousTVL`, inflating the fee on every subsequent attempt. While frozen, new depositors receive excess rsETH at the expense of existing holders.

## Finding Description
In `_updateRsETHPrice()`, the fee check at line 303 precedes the price write at line 313:

```solidity
// contracts/LRTOracle.sol L299-313
if (protocolFeeInETH > 0) {
    uint256 rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice);
    _checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee); // reverts here
    ...
}
rsETHPrice = newRsETHPrice; // never reached
```

`_checkAndUpdateDailyFeeMintLimit` at lines 197–209 resets the daily counter if a day has passed, then unconditionally reverts if the fee exceeds the cap:

```solidity
if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
    revert DailyFeeMintLimitExceeded(...);
}
```

Both public entry points (`updateRSETHPrice()` at L87 and `updateRSETHPriceAsManager()` at L94) call `_updateRsETHPrice()` and are equally blocked. The deadlock is self-reinforcing: `previousTVL = rsethSupply.mulWad(rsETHPrice)` uses the stale frozen price, so `rewardAmount = totalETHInProtocol - previousTVL` grows larger on every subsequent attempt, making `rsethAmountToMintAsProtocolFee` even larger than the cap. The daily counter reset does not help because the single-update fee exceeds the cap itself.

While `rsETHPrice` is frozen, `getRsETHAmountToMint` at line 520 of `LRTDepositPool.sol` divides by the stale lower price:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

A stale lower `rsETHPrice` causes new depositors to receive more rsETH than they are entitled to, directly diluting existing holders' unclaimed yield.

## Impact Explanation
**High — Theft of unclaimed yield.** Existing rsETH holders have already earned yield reflected in the true higher TVL. While `rsETHPrice` is frozen at a stale lower value, every new deposit mints excess rsETH to the depositor, diluting the yield that existing holders have already accrued. The protocol fee (treasury yield) is also permanently unclaimable until admin intervention. The deadlock does not resolve on its own.

## Likelihood Explanation
No attacker action is required. The condition occurs naturally whenever `maxFeeMintAmountPerDay` is smaller than the fee generated in a single update interval (a function of TVL × fee rate × elapsed time), or whenever any gap in price updates (keeper downtime, gas spikes, brief pause-unpause cycle) allows rewards to accumulate beyond the daily cap for a single call. At large TVL (e.g., 500,000 ETH) with a 10% protocol fee and a modest absolute cap, the condition is easily reached. The deadlock then persists indefinitely until an admin calls `setMaxFeeMintAmountPerDay`.

## Recommendation
1. **Decouple the price write from the fee check.** Write `rsETHPrice = newRsETHPrice` unconditionally before or independently of `_checkAndUpdateDailyFeeMintLimit`. The price update must never be gated on a secondary accounting check.
2. **Cap fee minting instead of reverting.** When the fee exceeds the daily cap, mint only up to `maxFeeMintAmountPerDay - currentPeriodMintedFeeAmount` and carry the remainder forward, or skip fee minting for that update while still writing the new price.
3. **Set `maxFeeMintAmountPerDay` relative to TVL and fee rate**, not as an absolute value, to avoid the cap being breached under normal growth.

## Proof of Concept
```
State:
  rsETHPrice = 1.00e18 (stale, last updated 2 days ago)
  rsethSupply = 500,000e18
  totalETHInProtocol = 502,700e18 (2 days of ~5% APY rewards)
  protocolFeeInBPS = 1000 (10%)
  maxFeeMintAmountPerDay = 100e18 rsETH

Step 1: Anyone calls updateRSETHPrice()
  previousTVL = 500,000e18 * 1.00e18 / 1e18 = 500,000e18
  rewardAmount = 502,700e18 - 500,000e18 = 2,700e18
  protocolFeeInETH = 2,700e18 * 1000 / 10,000 = 270e18
  newRsETHPrice = (502,700e18 - 270e18) / 500,000e18 ≈ 1.00486e18
  rsethAmountToMintAsProtocolFee = 270e18 / 1.00486e18 ≈ 268.7e18

Step 2: _checkAndUpdateDailyFeeMintLimit(268.7e18)
  currentPeriodMintedFeeAmount = 0 (new day)
  0 + 268.7e18 > 100e18 → revert DailyFeeMintLimitExceeded ✗

Step 3: rsETHPrice NOT updated. Remains at 1.00e18.

Step 4: Next day — same call:
  previousTVL still uses stale rsETHPrice = 1.00e18
  rewardAmount is now even larger (3 days of rewards)
  rsethAmountToMintAsProtocolFee > 268.7e18 > 100e18 → revert again ✗

Step 5: New depositor calls depositETH(1 ETH):
  rsethAmountToMint = 1e18 * 1e18 / 1.00e18 = 1.000e18 rsETH
  (correct: 1e18 * 1e18 / 1.00486e18 ≈ 0.9952e18 rsETH)
  → Depositor receives ~0.0048 rsETH excess per ETH at existing holders' expense.
```

Foundry fork test plan: deploy with `maxFeeMintAmountPerDay = 100e18`, warp 2 days, call `updateRSETHPrice()`, assert revert with `DailyFeeMintLimitExceeded`, assert `rsETHPrice` unchanged, then call `depositETH` and assert minted amount exceeds the fair share computed against the true new price.
