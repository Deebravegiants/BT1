### Title
Stale Votes from Removed Participants Counted in `vote_code_hash` Threshold Check — (File: `crates/contract/src/tee/proposal.rs`)

---

### Summary

`CodeHashesVotes::count_votes()` tallies all stored votes without filtering for current participants. After a resharing that removes participants, their votes persist in `proposal_by_account` until the `clean_tee_status` promise runs. If that promise fails (e.g., gas exhaustion), stale votes from removed participants are permanently counted toward the threshold in subsequent `vote_code_hash` calls, allowing a malicious Docker image hash to be whitelisted with fewer than the required number of current-participant votes.

---

### Finding Description

`CodeHashesVotes::count_votes()` in `crates/contract/src/tee/proposal.rs` counts every entry in `proposal_by_account` that matches the proposal, with no participant-membership filter:

```rust
fn count_votes(&self, proposal: &NodeImageHash) -> u64 {
    self.proposal_by_account
        .values()
        .filter(|&prop| prop == proposal)
        .count() as u64
}
``` [1](#0-0) 

This raw count is returned by `CodeHashesVotes::vote()` and consumed directly in `MpcContract::vote_code_hash()`:

```rust
let votes = self.tee_state.vote(code_hash, &participant);
if votes >= self.threshold()?.value() {
    self.tee_state.whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
}
``` [2](#0-1) 

The intended cleanup of stale votes is performed by `clean_tee_status`, a **private promise** scheduled after resharing completes. The `vote_update` function's own comment acknowledges that this promise can fail:

> "This ensures correctness even if the cleanup promise in `MpcContract::vote_reshared()` fails." [3](#0-2) 

By contrast, every other vote-counting path in the contract explicitly filters for current participants:
- `ThresholdParametersVotes::n_votes()` filters by `participants.participants()`.
- `TeeVerifierVotes::vote()` calls `count_for(|p| participants.is_participant_given_participant_id(...))`.
- `ProviderVotes::vote()` similarly filters. [4](#0-3) [5](#0-4) 

`CodeHashesVotes::count_votes()` is the only vote-counting path that omits this filter.

---

### Impact Explanation

If `clean_tee_status` fails after a resharing that removes participants who had already voted for a malicious Docker image hash, those stale votes remain in `proposal_by_account`. A single remaining current participant can then call `vote_code_hash` with the same hash, causing `count_votes()` to return a total that includes the stale votes and meets the threshold. The malicious image hash is then added to `allowed_docker_image_hashes`.

A node running that malicious image can subsequently submit a valid TEE attestation (`submit_participant_info`), pass `verify_tee`, and be voted into the participant set. Once a participant, it joins threshold signing rounds. This breaks the TEE attestation security model: the governance threshold for image approval is bypassed, enabling an unauthorized node image to gain signing capability.

This maps to the **Medium** allowed impact: *participant-state or contract execution-flow manipulation that breaks production safety/accounting invariants*.

---

### Likelihood Explanation

The attack requires:
1. At least two colluding participants (below threshold) to vote for a malicious hash before a resharing that removes them.
2. The `clean_tee_status` promise to fail (e.g., insufficient gas attached to the callback, or a NEAR runtime error).

Both conditions are realistic. NEAR promise callbacks can fail due to gas limits, and the codebase itself acknowledges this failure mode for the analogous `vote_reshared` cleanup. The attacker does not need to be above the signing threshold — they only need to be removed participants whose votes were cast before resharing.

---

### Recommendation

Apply the same participant-membership filter used by all other vote-counting paths. Replace the raw `count_votes` call inside `CodeHashesVotes::vote()` with a filtered count, or pass the current `Participants` set into `vote()` and filter there:

```rust
pub fn vote(
    &mut self,
    proposal: NodeImageHash,
    participant: &AuthenticatedParticipantId,
    current_participants: &Participants,
) -> u64 {
    self.proposal_by_account.insert(participant.clone(), proposal);
    self.proposal_by_account
        .iter()
        .filter(|(p, h)| {
            **h == proposal
                && current_participants.is_participant_given_participant_id(&p.get())
        })
        .count() as u64
}
```

This makes the threshold check in `vote_code_hash` resilient to `clean_tee_status` promise failures, matching the defensive pattern already used by `ThresholdParametersVotes::n_votes()` and `TeeVerifierVotes::vote()`.

---

### Proof of Concept

**Setup**: 5 participants `{P1, P2, P3, P4, P5}`, governance threshold = 3.

1. **Pre-resharing**: P1 and P2 call `vote_code_hash(malicious_hash)`. Both votes are stored in `CodeHashesVotes.proposal_by_account`. Count = 2 < 3; hash not yet whitelisted.

2. **Resharing**: A `vote_new_parameters` resharing removes P1 and P2. New participant set = `{P3, P4, P5}`, threshold = 3. `vote_reshared` schedules `clean_tee_status` as a promise.

3. **Promise failure**: `clean_tee_status` fails (gas exhaustion). Stale votes for P1 and P2 remain in `proposal_by_account`.

4. **Exploit**: P3 calls `vote_code_hash(malicious_hash)`. `CodeHashesVotes::vote()` inserts P3's vote and calls `count_votes(malicious_hash)`, which iterates all three entries (P1, P2, P3) and returns 3.

5. **Threshold met**: `3 >= 3` → `whitelist_tee_proposal(malicious_hash, ...)` is called. The malicious image hash is now in `allowed_docker_image_hashes`.

6. **Consequence**: A node running the malicious image submits a valid attestation, passes `verify_tee`, and can be voted into the participant set, gaining access to threshold signing rounds. [6](#0-5) [7](#0-6) [8](#0-7)

            self.tee_state

        }

        Ok(())
    }

    /// Stale CodeHashesVotes entries from removed participants must not count toward
    /// quorum after resharing.
    ///
    /// Scenario (N=5, T=3):
    /// 1. P1 and P2 vote for malicious hash before resharing.
    /// 2. Resharing removes P1 and P2. New set: {P3, P4, P5}.
    /// 3. clean_non_participant_votes removes stale votes.
    /// 4. P3 votes for the same hash — only 1 vote, not 3.
    #[test]
    fn test_clean_non_participant_votes_removes_stale_votes() {
        // Build 5 participants
        let mut all_participants = Participants::new();
        let mut account_ids = Vec::new();
        for i in 0..5 {
            let (account_id, info) = gen_participant(i);
            account_ids.push(account_id.clone());
            all_participants.insert(account_id, info).unwrap();
        }

        let mut tee_state = TeeState::default();

        // P0 and P1 vote for a malicious hash before resharing
        let malicious_hash = NodeImageHash::from([0xAA; 32]);
        for account_id in &account_ids[0..2] {
            let mut ctx = VMContextBuilder::new();
            ctx.signer_account_id(account_id.clone());
            testing_env!(ctx.build());
            let auth_id = AuthenticatedParticipantId::new(&all_participants).unwrap();
            tee_state.votes.vote(malicious_hash, &auth_id);
        }
        assert_eq!(tee_state.votes.proposal_by_account.len(), 2);

        // Resharing removes P0 and P1. New participant set: {P2, P3, P4}.
        let new_participants = all_participants.subset(2..5);

        // Clean non-participants (as done by CLEAN_TEE_STATUS after resharing)
        tee_state.clean_non_participant_votes(&new_participants);

        // Stale votes must be removed
        assert_eq!(tee_state.votes.proposal_by_account.len(), 0);

        // P2 votes for the same malicious hash — should be only 1 vote, not 3
        let p2_account = &account_ids[2];
        let mut ctx = VMContextBuilder::new();
        ctx.signer_account_id(p2_account.clone());
        testing_env!(ctx.build());
        let auth_id = AuthenticatedParticipantId::new(&new_participants).unwrap();
        let vote_count = tee_state.votes.vote(malicious_hash, &auth_id);
        assert_eq!(vote_count, 1, "Only the fresh vote from P2 should count");
