### Title
Missing `maxCollateralInputs` Enforcement in Alonzo UTXO Transition Allows Unbounded Collateral Inputs - (File: `eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Rules/Utxo.hs`)

---

### Summary

The Alonzo `utxoTransition` function contains a comment indicating that the `‖collateral tx‖ ≤ maxCollInputs pp` check should be enforced, and the validator function `validateTooManyCollateralInputs` is defined and exported — but the `runTest` call is entirely absent. An attacker can submit Alonzo-era transactions with an arbitrarily large number of collateral inputs, bypassing the `maxCollateralInputs` protocol parameter limit and forcing validators to verify more signatures than the protocol intends.

---

### Finding Description

In `eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Rules/Utxo.hs`, the `utxoTransition` function ends with:

```haskell
{-   totExunits tx ≤ maxTxExUnits pp    -}
runTest $ validateExUnitsTooBigUTxO pp tx

{-   ‖collateral tx‖  ≤  maxCollInputs pp   -}
                                              ← NO runTest call here

updatedGovState <-
  trans @(EraRule "UTXOS" era) $
    TRC (UtxosEnv slot pp certState, utxosGovState utxos, stAnnTx)
``` [1](#0-0) 

The comment at line 570 is a direct transcription of the formal specification predicate, and the function that implements it is fully defined:

```haskell
validateTooManyCollateralInputs ::
  AlonzoEraTxBody era =>
  PParams era ->
  TxBody TopTx era ->
  Test (AlonzoUtxoPredFailure era)
validateTooManyCollateralInputs pp txBody =
  failureUnless (numColl <= maxColl) $
    TooManyCollateralInputs Mismatch {mismatchSupplied = numColl, mismatchExpected = maxColl}
``` [2](#0-1) 

The function is also exported from the module: [3](#0-2) 

In contrast, the Babbage `babbageUtxoValidation` correctly calls this check:

```haskell
{-   ‖collateral tx‖  ≤  maxCollInputs pp   -}
runTest $ Alonzo.validateTooManyCollateralInputs pp txBody
``` [4](#0-3) 

The Alonzo `utxoTransition` is the only era-specific transition that omits this call despite having the specification comment in place.

---

### Impact Explanation

The `maxCollateralInputs` protocol parameter was introduced specifically to bound the number of additional VKey signatures that must be verified when processing collateral inputs. Each collateral input must be VKey-locked (enforced by `validateScriptsNotPaidUTxO`), meaning each one requires a cryptographic signature verification. Without the `validateTooManyCollateralInputs` check in the Alonzo transition, an attacker can craft a transaction with an arbitrarily large collateral input set — e.g., hundreds or thousands of entries — forcing every validating node to perform unbounded signature verification work for a single transaction. This allows attacker-controlled transactions to exceed the intended per-transaction validation cost limits set by the protocol parameter, matching the **Medium** impact class: *attacker-controlled transactions exceed intended validation limits*.

---

### Likelihood Explanation

Any unprivileged transaction sender can construct an Alonzo-era transaction with an arbitrarily large `collateralInputsTxBodyL` set. No special privilege, key, or governance action is required. The attacker only needs to own enough UTxO entries to populate the collateral set. The missing check is a straightforward omission — the comment is present, the function is defined and exported, but the `runTest` call was never added. Nodes replaying Alonzo-era history from genesis are affected, and any deployment still operating in the Alonzo era is directly vulnerable.

---

### Recommendation

Add the missing `runTest` call immediately after the `validateExUnitsTooBigUTxO` check in `utxoTransition`, mirroring the Babbage implementation:

```haskell
{-   totExunits tx ≤ maxTxExUnits pp    -}
runTest $ validateExUnitsTooBigUTxO pp tx

{-   ‖collateral tx‖  ≤  maxCollInputs pp   -}
runTest $ validateTooManyCollateralInputs pp txBody   -- ADD THIS LINE
```

This aligns the Alonzo implementation with both the formal specification and the Babbage implementation.

---

### Proof of Concept

1. **Specification mandates the check.** The Alonzo formal spec (`eras/alonzo/formal-spec/utxo.tex`, line 662) includes `‖collateral tx‖ ≤ maxCollateralInputs pp` as a required predicate in the UTXO rule. [5](#0-4) 

2. **The validator function exists and is exported** from `Cardano.Ledger.Alonzo.Rules.Utxo` at line 31, and is fully implemented at lines 470–481. [2](#0-1) 

3. **The comment is present but the call is absent.** Lines 567–574 of `utxoTransition` show `validateExUnitsTooBigUTxO` is called, the collateral comment appears, and then execution immediately proceeds to `trans @(EraRule "UTXOS" era)` with no intervening `runTest`. [1](#0-0) 

4. **Babbage correctly enforces the limit** at line 412 of `babbageUtxoValidation`, confirming the intended pattern. [4](#0-3) 

5. **Attack path:** Craft an Alonzo transaction with `collateralInputsTxBodyL` containing N >> `maxCollateralInputs` VKey-locked UTxO entries, each with a valid witness. Submit to a node in the Alonzo era. The node will pass all checks (fee, ExUnits, collateral sufficiency) and proceed to verify N signatures for collateral, with no upper bound enforced by the ledger rule.
