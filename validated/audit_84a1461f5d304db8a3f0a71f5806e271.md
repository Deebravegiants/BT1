### Title
`compare_retrospective_block_hash = false` Silently Passes a Divergent L1-Anchor Hash to the Batcher, Allowing a Proposal Built on a Wrong Chain Root to Be Accepted — (`crates/apollo_consensus_orchestrator/src/utils.rs`)

---

### Summary

`ContextDynamicConfig` contains a boolean flag `compare_retrospective_block_hash`. When set to `false`, the cross-check between the batcher's view and the state-sync's view of the retrospective block hash is silently skipped. The function still returns `batcher_block_hash` as the authoritative value and passes it to `validate_block` / `propose_block`. If the batcher's local chain has diverged from the canonical chain (different retrospective block hash), the divergent hash is embedded in `ValidateBlockInput.retrospective_block_hash` and used by the blockifier to anchor the block. Every validator that also has `compare_retrospective_block_hash = false` will accept the proposal, committing a block whose L1-anchor hash is wrong. This is the direct Sequencer analog of the `callOnDepositCallback = false` misconfiguration: a single boolean flag disables a critical safety check, and the system silently proceeds with a corrupted value.

---

### Finding Description

In `retrospective_block_hash()`:

```rust
if compare_retrospective_block_hash && state_sync_block_hash != batcher_block_hash {
    // ... error path
    return Err(RetrospectiveBlockHashError::HashMismatch { ... });
}
Ok(Some(BlockHashAndNumber { number: block_number, hash: batcher_block_hash }))
```

When `compare_retrospective_block_hash` is `false`, the `if` branch is never entered regardless of whether `state_sync_block_hash != batcher_block_hash`. The function unconditionally returns `batcher_block_hash`.

This return value flows directly into `ValidateBlockInput.retrospective_block_hash` (in `initiate_validation`) and `ProposeBlockInput.retrospective_block_hash` (in `initiate_build`), which the batcher passes to the blockifier's `create_block_builder`. The blockifier uses this hash as the L1-anchor for the block being built or validated.

The flag is part of `ContextDynamicConfig`, which is a live-updatable configuration pushed via the `ConfigManager`. It defaults to `true` in code, but the production deployment config template exposes it as a substitution variable (`$$$_CONSENSUS_MANAGER_CONFIG-CONTEXT_CONFIG-DYNAMIC_CONFIG-COMPARE_RETROSPECTIVE_BLOCK_HASH_$$$`), meaning it can be set to `false` in any deployment without any compile-time or startup-time guard.

---

### Impact Explanation

**Critical — Wrong state/block commitment.**

The retrospective block hash is the L1-anchored block hash embedded in every new block header. If a divergent hash is used:

1. The `PartialBlockHashComponents` computed by `BlockExecutionArtifacts::new` will incorporate the wrong retrospective hash.
2. The `ProposalCommitment` (partial block hash) derived from it will be wrong.
3. All validators that also have the flag disabled will compute the same wrong commitment and vote for it.
4. The committed block will carry an incorrect L1-anchor, breaking the chain's verifiable link to L1 state.

This is not merely a liveness issue — it is a state-commitment integrity issue. The wrong `retrospective_block_hash` is embedded in the block header that gets committed to storage via `commit_proposal_and_block`, and the resulting `PartialBlockHash` is what consensus signs over.

---

### Likelihood Explanation

**Medium.** The default value is `true`, so the check is enabled by default. However:

- The flag is in `ContextDynamicConfig`, which is live-updatable at runtime without a node restart.
- The production deployment template leaves the value as an unresolved substitution variable, meaning any deployment that fails to set it (or explicitly sets it to `false` for debugging/migration purposes) silently disables the check.
- There is no startup validation, no warning log when the flag is `false`, and no secondary guard that would catch the divergence downstream.
- The flag was introduced as a migration aid (to tolerate temporary batcher/state-sync divergence), making it plausible that operators set it to `false` during upgrades and forget to re-enable it.

---

### Recommendation

1. **Remove the bypass entirely, or make it a hard error.** The comparison between batcher and state-sync block hashes is a safety invariant, not an optional optimization. If the two sources disagree, the node should halt or refuse to build/validate, not silently proceed with one source's value.

2. **If the flag must be retained** (e.g., for migration), emit a `warn!` or `error!` log every time a mismatch is detected but suppressed, and add a startup/config-load assertion that logs a prominent warning when `compare_retrospective_block_hash = false`.

3. **Add a config-load validation** in `validate_dynamic_config` that rejects or warns on `compare_retrospective_block_hash = false` in production deployments.

4. **Replace the substitution variable** in the deployment template with a hardcoded `true` so the safe default cannot be accidentally omitted.

---

### Proof of Concept

**Setup:** Two nodes, both with `compare_retrospective_block_hash = false` in their `ContextDynamicConfig`. The batcher on both nodes has a divergent local chain (e.g., after a revert that was not reflected in state-sync), so `batcher_block_hash != state_sync_block_hash` for the retrospective block.

**Trigger path (proposer):**

1. `build_proposal` → `initiate_build` → `wait_for_retrospective_block_hash` → `retrospective_block_hash(…, compare=false)`.
2. `state_sync_block_hash = H_correct`, `batcher_block_hash = H_wrong`.
3. The `if compare_retrospective_block_hash && …` branch is skipped.
4. Returns `Ok(Some(BlockHashAndNumber { hash: H_wrong }))`.
5. `ProposeBlockInput { retrospective_block_hash: Some(H_wrong), … }` is sent to the batcher.
6. The batcher builds the block with `H_wrong` as the L1-anchor.
7. `BlockExecutionArtifacts::new` computes `PartialBlockHashComponents` using `H_wrong`.
8. The proposer streams the proposal with `ProposalFin { proposal_commitment: C_wrong }`.

**Trigger path (validator):**

1. `validate_proposal` → `initiate_validation` → `retrospective_block_hash(…, compare=false)`.
2. Same divergence: returns `H_wrong`.
3. `ValidateBlockInput { retrospective_block_hash: Some(H_wrong) }` sent to batcher.
4. Validator re-executes with `H_wrong`, computes the same `C_wrong`.
5. `built_block == received_fin.proposal_commitment` → passes the fin-mismatch check.
6. Consensus votes for and commits the block with the wrong L1-anchor.

**Corrupted value:** `ValidateBlockInput.retrospective_block_hash` = `H_wrong` instead of `H_correct`, propagating into the committed `PartialBlockHashComponents` and the final `ProposalCommitment`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6)

pub(crate) async fn retrospective_block_hash(
    batcher_client: Arc<dyn BatcherClient>,
    state_sync_client: SharedStateSyncClient,
    init: &ProposalInit,
    compare_retrospective_block_hash: bool,
) -> RetrospectiveBlockHashResult<Option<BlockHashAndNumber>> {
    if let Some(required_height) = (init.height.0 + 1).checked_sub(STORED_BLOCK_HASH_BUFFER) {
        // Just verify that the batcher has done calculating it.
        batcher_client.get_block_hash(BlockNumber(required_height)).await?;
    }

    let retrospective_block_number = init.height.0.checked_sub(STORED_BLOCK_HASH_BUFFER);

    let Some(block_number) = retrospective_block_number else {
        info!(
            "Retrospective block number is less than {STORED_BLOCK_HASH_BUFFER}, setting None as \
             expected."
        );
        return Ok(None);
    };

    let block_number = BlockNumber(block_number);

    // First try from state sync - assuming it takes longer to this one to be ready.
    let state_sync_block_hash = state_sync_client.get_block_hash(block_number).await?;

    // Then try from batcher.
    let batcher_block_hash = batcher_client.get_block_hash(block_number).await?;

    if compare_retrospective_block_hash && state_sync_block_hash != batcher_block_hash {
        warn!(
            "Retrospective block hashes mismatch for block {block_number}: state sync block hash: \
             {state_sync_block_hash:?}, batcher block hash: {batcher_block_hash:?}"
        );
        CONSENSUS_RETROSPECTIVE_BLOCK_HASH_MISMATCH.increment(1);
        return Err(RetrospectiveBlockHashError::HashMismatch {
            block_number,
            state_sync_block_hash,
            batcher_block_hash,
        });
    }
    Ok(Some(BlockHashAndNumber { number: block_number, hash: batcher_block_hash }))
}

async fn initiate_validation(
    batcher: Arc<dyn BatcherClient>,
    state_sync_client: SharedStateSyncClient,
    init: &ProposalInit,
    proposal_id: ProposalId,
    timeout_plus_margin: Duration,
    clock: &dyn Clock,
    compare_retrospective_block_hash: bool,
) -> ValidateProposalResult<()> {
    let chrono_timeout = chrono::Duration::from_std(timeout_plus_margin)
        .expect("Can't convert timeout to chrono::Duration");

    let input = ValidateBlockInput {
        proposal_id,
        deadline: clock.now() + chrono_timeout,
        retrospective_block_hash: retrospective_block_hash(
            batcher.clone(),
            state_sync_client,
            init,
            compare_retrospective_block_hash,
        )
        .await
        .map_err(ValidateProposalError::from)?,
        block_info: convert_to_sn_api_block_info(init)?,
    };
    debug!("Initiating validate proposal: input={input:?}");
    batcher.validate_block(input.clone()).await.map_err(|err| {
        ValidateProposalError::Batcher(
            format!("Failed to initiate validate proposal {input:?}."),
            err,
        )
    })?;
    Ok(())
}
