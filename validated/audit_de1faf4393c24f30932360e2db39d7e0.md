### Title
Mathematical Incompatibility Between Governance Threshold and DamgardEtAl Participant Count Requirement — (`crates/node/src/providers/robust_ecdsa/presign.rs`)

---

### Summary

The node's `compute_thresholds` function uses the governance threshold directly as `num_signers` for DamgardEtAl (Robust ECDSA) presignature generation, while simultaneously computing `MaxMalicious = (governance_threshold - 1) / 2` via integer division. When the governance threshold is even, `num_signers` exceeds `2 * MaxMalicious + 1` — the exact participant count the DamgardEtAl protocol requires. The internal guard uses `<=` instead of `==`, silently passing the mismatch through to the cryptographic layer, where it either causes permanent presignature generation failure (DoS) or, if the protocol tolerates `N > 2t+1`, enables a split-view attack by a single Byzantine coordinator to recover the secret key.

---

### Finding Description

`compute_thresholds` in `crates/node/src/providers/robust_ecdsa/presign.rs` computes two values from the same governance threshold:

```rust
fn compute_thresholds(
    governance_threshold: u64,
    num_running_participants: usize,
) -> anyhow::Result<(usize, MaxMalicious)> {
    let governance_threshold: usize = governance_threshold.try_into()?;
    let num_signers = get_number_of_signers(governance_threshold, num_running_participants)?;
    let robust_ecdsa_threshold =
        translate_threshold(governance_threshold, num_running_participants)?;
    anyhow::ensure!(
        robust_ecdsa_threshold
            .value()
            .checked_mul(2)
            .and_then(|v| v.checked_add(1))
            .is_some_and(|v| v <= num_signers)   // ← uses <=, not ==
    );
    Ok((num_signers, robust_ecdsa_threshold))
}
``` [1](#0-0) 

`get_number_of_signers` returns `threshold` unchanged, so `num_signers = governance_threshold`. `translate_threshold` computes:

```rust
Ok(MaxMalicious::from((number_of_signers - 1) / 2))
``` [2](#0-1) 

The integer division creates a non-linear transformation. For **even** `governance_threshold`:

| governance_threshold | num_signers | MaxMalicious | 2t+1 | num_signers == 2t+1? |
|---|---|---|---|---|
| 5 | 5 | 2 | 5 | ✓ |
| **6** | **6** | **2** | **5** | **✗ (6 > 5)** |
| 7 | 7 | 3 | 7 | ✓ |
| **8** | **8** | **3** | **7** | **✗ (8 > 7)** |

The guard `2 * MaxMalicious + 1 <= num_signers` passes silently for even thresholds (e.g., `5 <= 6`), so the mismatch is never caught. The node then selects `num_signers` participants and calls the presign protocol with `MaxMalicious` that does not match the participant count.

The DamgardEtAl protocol's own README explicitly states:

> Both presigning and signing require **exactly** N = 2t + 1 participants. This constraint is enforced at initialization and prevents split-view attacks where different subsets sign different messages using shares from the same presignature. [3](#0-2) 

The security documentation further warns:

> Do **not** allow N₁ > 2t + 1 … Allowing larger presigning sets or smaller signing sets enables split-view and presignature-reuse attacks when a coordinator can run parallel or partially overlapping signing sessions.
> a novel split-view attack exists that can extract the secret key using as few as **2t + 2 presigning participants**, with as few as two signing sessions. [4](#0-3) 

The contract's own `validate_domain_threshold` enforces `2t - 1 <= n` for DamgardEtAl domains using the per-domain `reconstruction_threshold`, but the node ignores this and uses the governance threshold instead — a mismatch explicitly flagged as a TODO:

```rust
// TODO(#3164): once the node supports per-domain thresholds, this should
// take the domain-specific threshold instead of the single governance threshold.
``` [5](#0-4) [6](#0-5) 

---

### Impact Explanation

Two concrete outcomes depending on whether the presign protocol enforces `N = 2t+1` at runtime:

**Path A — Protocol enforces exact count (DoS):** Presignature generation for every DamgardEtAl domain permanently fails whenever the governance threshold is even. All signing requests for those domains are permanently unserviceable. Funds locked behind DamgardEtAl-signed foreign-chain transactions are permanently frozen. This matches: *Critical — permanent freezing of funds controlled by the MPC network.*

**Path B — Protocol tolerates N > 2t+1 (Key Recovery):** A single Byzantine MPC node acting as coordinator can run two parallel signing sessions over different subsets of the `N = 2t+2` presigning participants, using shares from the same presignature. The security documentation explicitly states this recovers the secret key in two sessions. This matches: *Critical — unauthorized access to MPC key shares / secret recovery that materially enables forgery.*

---

### Likelihood Explanation

The governance threshold is set by the MPC participants through `vote_new_parameters`. The contract permits any value in `[ceil(0.6n), n]`. For `n = 10`, valid values are `{6, 7, 8, 9, 10}` — half of which are even. An adversarial participant below the signing threshold can propose an even governance threshold; other participants may vote for it legitimately without awareness of the cryptographic constraint. The vulnerability also manifests from the initial deployment if operators choose an even threshold (a valid and natural choice, e.g., 6-of-10). No special privilege is required beyond participating in governance.

---

### Recommendation

Replace the permissive `<=` guard in `compute_thresholds` with an exact equality check, and derive `num_signers` from `MaxMalicious` rather than from the governance threshold:

```rust
fn compute_thresholds(
    governance_threshold: u64,
    num_running_participants: usize,
) -> anyhow::Result<(usize, MaxMalicious)> {
    let governance_threshold: usize = governance_threshold.try_into()?;
    let robust_ecdsa_threshold =
        translate_threshold(governance_threshold, num_running_participants)?;
    // Derive num_signers from MaxMalicious, not from governance_threshold
    let num_signers = robust_ecdsa_threshold.value()
        .checked_mul(2)
        .and_then(|v| v.checked_add(1))
        .ok_or_else(|| anyhow::anyhow!("overflow computing num_signers"))?;
    anyhow::ensure!(num_signers <= num_running_participants, ...);
    Ok((num_signers, robust_ecdsa_threshold))
}
```

Additionally, resolve the underlying design issue by implementing the per-domain threshold support tracked in TODO(#3164), so the node uses `reconstruction_threshold` (validated by the contract for DamgardEtAl) rather than the governance threshold.

### Proof of Concept

Given a deployment with `n = 10` participants and `governance_threshold = 6` (valid: `ceil(0.6 * 10) = 6`):

1. `compute_thresholds(6, 10)` is called at node startup.
2. `num_signers = get_number_of_signers(6, 10) = 6`.
3. `MaxMalicious = translate_threshold(6, 10) = (6 - 1) / 2 = 2`.
4. Guard: `2 * 2 + 1 = 5 <= 6` → **passes silently**.
5. Node selects 6 participants for presignature generation.
6. Presign protocol is invoked with `N = 6`, `MaxMalicious = 2`, but requires exactly `N = 2*2+1 = 5`.
7. **Path A:** Protocol rejects `N != 2t+1` at initialization → presignature generation fails permanently for all DamgardEtAl domains → all signing requests for those domains are unserviceable.
8. **Path B:** Protocol accepts `N = 6 > 5` → Byzantine coordinator runs two signing sessions over different 5-participant subsets of the 6-participant presigning group, using the same presignature → secret key recovered in two sessions.

```rust
/// Validates the per-domain reconstruction threshold against the participant
/// count. Universal bound `2 <= t <= n` plus, for `DamgardEtAl`, the
/// honest-majority bound `2t - 1 <= n`.
pub fn validate_domain_threshold(
    domain: &DomainConfig,
    num_participants: u64,
) -> Result<(), Error> {
    let t = domain.reconstruction_threshold.inner();
    if t < MIN_RECONSTRUCTION_THRESHOLD {
        return Err(DomainError::ReconstructionThresholdTooLow.into());
    }
    if t > num_participants {
        return Err(DomainError::ReconstructionThresholdExceedsParticipants {
            threshold: t,
            participants: num_participants,
        }
        .into());
    }
    if domain.protocol == Protocol::DamgardEtAl {
        let required = t
            .checked_mul(2)
            .and_then(|x| x.checked_sub(1))
            .ok_or(DomainError::ReconstructionThresholdOverflow { threshold: t })?;
        if required > num_participants {
            return Err(DomainError::InsufficientParticipantsForProtocol {
                protocol: domain.protocol,
                required,
                participants: num_participants,
            }
            .into());
        }
    }
    Ok(())
}
