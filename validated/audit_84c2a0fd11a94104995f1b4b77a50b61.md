### Title
Native ETH (`nativeFee`) Permanently Accumulates in `OmniBridge` with No Withdrawal Mechanism — (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

`OmniBridge.sol` is a `payable` contract with a bare `receive()` function. Every call to `initTransfer` and `initTransfer1155` retains the caller-supplied `nativeFee` ETH inside the contract — it is never forwarded to Wormhole and never paid to any relayer. No admin function exists to withdraw this accumulated ETH. The value is permanently locked until an upgrade is deployed.

---

### Finding Description

`initTransfer` splits `msg.value` into two parts:

```
// ERC-20 path
extensionValue = msg.value - nativeFee;   // forwarded to Wormhole
// nativeFee stays in the contract
``` [1](#0-0) 

`initTransferExtension` in `OmniBridgeWormhole` forwards only `value` (i.e., `extensionValue`) to Wormhole:

```solidity
_wormhole.publishMessage{value: value}(wormholeNonce, payload, _consistencyLevel);
``` [2](#0-1) 

The `nativeFee` portion of `msg.value` is therefore silently retained in the contract on every `initTransfer` call. The same applies to `initTransfer1155`:

```
extensionValue = msg.value - nativeFee;
``` [3](#0-2) 

Additionally, the contract exposes a bare `receive()`:

```solidity
receive() external payable {}
``` [4](#0-3) 

This allows arbitrary ETH to be force-sent to the contract. Scanning the entire contract, there is no `withdrawETH`, `rescueETH`, or any other function that moves ETH out to an admin or treasury address. [5](#0-4) 

The relayer compensation for EVM-originated transfers is handled entirely on the NEAR side (via the NEAR bridge's storage balance), so the ETH `nativeFee` collected on EVM has no on-chain disbursement path at all.

---

### Impact Explanation

Every `initTransfer` / `initTransfer1155` call with `nativeFee > 0` deposits ETH into the contract that can never leave. Over time this constitutes a growing pool of locked user-paid fees. This is a **fee-routing divergence that sends value to the wrong party** (the contract instead of the relayer or protocol treasury), matching the High impact criterion. The `receive()` function compounds the issue by allowing additional ETH to be trapped.

---

### Likelihood Explanation

`nativeFee` is a standard, documented parameter of the public `initTransfer` API. Any user bridging ERC-20 tokens who sets `nativeFee > 0` (the normal case for incentivising relayers) triggers the lock. No special conditions are required.

---

### Recommendation

Add an admin-gated ETH withdrawal function:

```solidity
function withdrawETH(
    address payable recipient,
    uint256 amount
) external onlyRole(DEFAULT_ADMIN_ROLE) {
    (bool success, ) = recipient.call{value: amount}("");
    require(success, "ETH transfer failed");
}
```

---

### Proof of Concept

1. Deploy `OmniBridgeWormhole` (or use the live contract).
2. Call `initTransfer(someERC20, amount, fee, nativeFee, recipient, "")` with `msg.value = nativeFee + wormholeFee` and `nativeFee = 1 ether`.
3. Observe: `address(bridge).balance` increases by `1 ether`; Wormhole receives only `wormholeFee`.
4. Attempt any withdrawal — no function exists. The 1 ETH is locked until an upgrade is deployed.
