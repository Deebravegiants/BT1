Audit Report

## Title
Stranded ETH from Payable Non-WETH Calls Is Capturable by Any Subsequent WETH Swap Caller — (`metric-periphery/contracts/base/PeripheryPayments.sol`, `metric-periphery/contracts/MetricOmmSimpleRouter.sol`, `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary
Every user-facing entry point in `MetricOmmSimpleRouter` and `MetricOmmPoolLiquidityAdder` is `payable` but performs no check that `msg.value == 0` when the input token is not WETH. ETH sent alongside a non-WETH call is silently retained by the contract. The `pay()` helper in `PeripheryPayments` unconditionally consumes the contract's entire native-ETH balance when settling a WETH-input swap, bypassing the actual payer entirely. Any subsequent caller who performs a WETH-input swap or WETH-leg liquidity add can drain the stranded ETH at zero cost.

## Finding Description
`PeripheryPayments.pay()` (lines 73–84) contains the following branch:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);   // payer bypassed
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

When `address(this).balance >= value`, the contract wraps its own accumulated ETH and forwards it to the pool. The `payer` argument is completely ignored — no `transferFrom` is issued against the actual caller.

The `receive()` fallback (line 32–34) reverts for any sender other than WETH, which blocks plain ETH transfers but does **not** block ETH sent as `msg.value` alongside a function call. All four swap entry points in `MetricOmmSimpleRouter` (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, lines 67/92/130/154) and all four liquidity entry points in `MetricOmmPoolLiquidityAdder` (`addLiquidityExactShares` ×2, `addLiquidityWeighted` ×2, lines 56/71/88/123) are `payable` with no guard of the form `if (msg.value > 0 && tokenIn != WETH) revert`. ETH sent alongside a non-WETH call is silently retained.

`refundETH()` (lines 58–63) is a separate, optional call. It does not protect against front-running between the victim's transaction and an attacker's subsequent transaction.

Exploit path:
1. Victim calls `exactInputSingle{value: 1 ether}(pool=USDC/DAI, tokenIn=USDC, ...)`. Swap executes via USDC `transferFrom`; 1 ETH remains in `MetricOmmSimpleRouter`.
2. Attacker calls `exactInputSingle{value: 0}(pool=WETH/DAI, tokenIn=WETH, amountIn=1e18, ...)`.
3. Pool calls `metricOmmSwapCallback` → `_justPayCallback` → `pay(WETH, attacker, pool, 1e18)`.
4. `address(this).balance == 1e18 >= 1e18` → contract wraps victim's ETH → transfers 1 WETH to pool → attacker receives DAI output at zero cost.

## Impact Explanation
Direct loss of user ETH principal. A victim who accidentally sends ETH with a non-WETH call loses that ETH permanently unless they call `refundETH()` in the same multicall batch. An attacker who observes the stranded ETH on-chain can immediately drain it via a WETH-input swap or WETH-leg liquidity add, receiving the full swap output at zero cost. This meets the Sherlock threshold for direct loss of user principal (High severity).

## Likelihood Explanation
Low. The attack requires a user error: sending ETH alongside a non-WETH call. However, this is a realistic mistake because all entry points are `payable` and the WETH-via-ETH pattern is explicitly supported and documented. Once ETH is stranded on-chain, front-running it is trivial and requires no special privileges — any unprivileged caller can execute the theft in the same or next block.

## Recommendation
1. **Primary fix**: Add `if (msg.value > 0 && params.tokenIn != WETH) revert ETHNotAccepted();` at the top of each swap and liquidity function that does not use native ETH.
2. **Alternative**: Remove `payable` from functions that have no WETH leg. For functions that may or may not use ETH, enforce `msg.value == 0` when the relevant token is not WETH.
3. **Defense-in-depth**: In `pay()`, if `token == WETH` and `nativeBalance > 0` but `msg.value == 0` (i.e., no ETH was sent in the current call), revert or emit an event to surface unexpected ETH accumulation.
