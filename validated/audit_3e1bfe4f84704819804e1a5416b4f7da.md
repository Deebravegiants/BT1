Audit Report

## Title
First Depositor rsETH Price Inflation via ETH Donation to LRTDepositPool - (File: contracts/LRTDepositPool.sol, contracts/LRTOracle.sol)

## Summary
An attacker who deposits first can mint a minimal rsETH supply (1 wei), then donate a large ETH amount directly to `LRTDepositPool` via its open `receive()` function. Because `getETHDistributionData()` counts `address(this).balance` as protocol ETH and `updateRSETHPrice()` is a permissionless public function with no price-increase guard when `pricePercentageLimit == 0` (the default), the attacker can inflate `rsETHPrice` arbitrarily. Subsequent depositors receive 0 rsETH due to integer division truncation, permanently losing their deposited ETH while the attacker—holding 100% of rsETH supply—redeems all pool assets.

## Finding Description

**Root cause — three cooperating weaknesses:**

**1. Open ETH donation path.**
`LRTDepositPool` accepts arbitrary ETH via an unrestricted `receive()` fallback: [1](#0-0) 

`getETHDistributionData()` counts `address(this).balance` directly as protocol ETH, so any donated ETH is immediately reflected in `totalETHInProtocol`: [2](#0-1) 

**2. Permissionless price update with no guard at default state.**
`updateRSETHPrice()` is `public` with no access control: [3](#0-2) 

The price-increase guard is only active when `pricePercentageLimit > 0`. Since `initialize` never sets it, it defaults to `0`, making `isPriceIncreaseOffLimit` permanently `false` until an admin explicitly configures it: [4](#0-3) [5](#0-4) 

**3. No minimum rsETH output enforced at protocol level.**
`_beforeDeposit` only checks `rsethAmountToMint < minRSETHAmountExpected`. If the caller passes `0` for `minRSETHAmountExpected`, a deposit yielding 0 rsETH is accepted: [6](#0-5) 

`RSETH.mint(victim, 0)` succeeds: the `checkDailyMintLimit(0)` modifier does not revert when `maxMintAmountPerDay > 0` and the daily cap is not yet reached, and OpenZeppelin's `_mint` with amount 0 emits a Transfer event without reverting.

**Exploit flow:**

- **Step 1:** Attacker calls `depositETH{value: 1}(0, "")`. With `rsETHPrice = 1e18` (set by `_updateRsETHPrice` when supply is 0), `getRsETHAmountToMint` yields `(1 * 1e18) / 1e18 = 1` wei rsETH. Supply = 1 wei. [7](#0-6) [8](#0-7) 

- **Step 2:** Attacker sends `D` ETH directly to `LRTDepositPool` via `receive()`. Pool balance = `1 + D`.

- **Step 3:** Attacker calls `lrtOracle.updateRSETHPrice()`. `_getTotalEthInProtocol()` calls `getTotalAssetDeposits(ETH_TOKEN)` → `getETHDistributionData()` → `address(this).balance = 1 + D`. New price: `newRsETHPrice = (1 + D) * 1e18 / 1 ≈ D * 1e18`. No revert because `pricePercentageLimit == 0`. [9](#0-8) 

- **Step 4:** Victim calls `depositETH{value: V}(0, "")`. `rsethAmountToMint = (V * 1e18) / (D * 1e18) = V / D`. If `D ≥ V`, this truncates to 0. `_mintRsETH(0)` succeeds; victim's `V` ETH is absorbed into the pool with no shares issued.

- **Step 5:** Attacker holds 100% of rsETH supply (1 wei) and redeems all ETH in the pool (`1 + D + V`), netting the victim's full deposit `V` at a cost of `D ≈ V`.

## Impact Explanation

**Critical — Direct theft of user funds.** The victim deposits `V` ETH and receives 0 rsETH, permanently losing their entire deposit. The attacker recovers both their donation and the victim's deposit upon withdrawal. This is not a rounding edge case; with `D = V`, the victim's deposit is completely stolen with no recourse.

## Likelihood Explanation

**High.** All required conditions are either default state or permissionless actions:
1. Being the first depositor — realistic at protocol launch.
2. Sending ETH to `receive()` — permissionless.
3. Calling `updateRSETHPrice()` — permissionless public function.
4. `pricePercentageLimit == 0` — the default since `initialize` does not set it.
5. `maxMintAmountPerDay > 0` — required for the protocol to accept any deposits at all; this is a necessary operational precondition, not an attacker-controlled variable.

No privileged access, oracle compromise, or governance capture is required. The attack is executable atomically or across a few transactions.

## Recommendation

1. **Seed the pool at initialization** with a non-trivial ETH deposit minted to a dead address (e.g., `1e15` wei) to make price manipulation economically infeasible.
2. **Reject unaccounted ETH donations** by tracking expected ETH balance and excluding surplus from `totalETHInProtocol`, or by removing the open `receive()` fallback and only accepting ETH through named entry points (`receiveFromRewardReceiver`, `receiveFromNodeDelegator`, etc.).
3. **Set `pricePercentageLimit` in `initialize`** to a non-zero value so the price-increase guard is active from deployment.
4. **Enforce a minimum rsETH output > 0** at the protocol level in `_beforeDeposit`, independent of the caller-supplied `minRSETHAmountExpected`, to revert any deposit that would yield 0 rsETH.

## Proof of Concept

```solidity
// Precondition: protocol deployed, maxMintAmountPerDay > 0, pricePercentageLimit == 0 (default)

// Step 1: updateRSETHPrice() → rsETHPrice = 1e18 (supply == 0)
lrtOracle.updateRSETHPrice();

// Step 2: Attacker deposits 1 wei ETH → mints 1 wei rsETH
lrtDepositPool.depositETH{value: 1}(0, "");
// rsETH supply = 1 wei, pool ETH balance = 1 wei

// Step 3: Attacker donates D = 1 ETH directly to pool
(bool ok,) = address(lrtDepositPool).call{value: 1 ether}("");
require(ok);
// pool ETH balance = 1 + 1e18

// Step 4: Attacker calls updateRSETHPrice() (public, no guard since pricePercentageLimit == 0)
lrtOracle.updateRSETHPrice();
// totalETHInProtocol ≈ 1e18, rsethSupply = 1
// newRsETHPrice = 1e18 * 1e18 / 1 = 1e36

// Step 5: Victim deposits 1 ETH with minRSETHAmountExpected = 0
lrtDepositPool.depositETH{value: 1 ether}(0, "");
// rsethAmountToMint = (1e18 * 1e18) / 1e36 = 1 → rounds to 0 if donation > deposit
// _mintRsETH(0) succeeds; victim receives 0 rsETH, loses 1 ETH

// Step 6: Attacker redeems 1 wei rsETH (100% of supply)
// Pool contains: 1 + 1e18 + 1e18 ≈ 2e18 wei
// Attacker profit ≈ 1 ETH (victim's full deposit)
```
