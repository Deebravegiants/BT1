### Title
`queuedWithdrawalsBuffer` Is a Static Value Disconnected from `assetsCommitted`, Allowing Instant Withdrawals to Drain Vault Below Queued Obligations - (File: `contracts/LRTUnstakingVault.sol`, `contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTUnstakingVault.getAssetsAvailableForInstantWithdrawal` uses a manually-set static buffer (`queuedWithdrawalsBuffer`) to determine how much of the vault is "reserved" for queued withdrawals. This buffer is never automatically updated when `assetsCommitted` in `LRTWithdrawalManager` grows. As a result, instant withdrawals can drain the vault below the amount needed to service pending queued withdrawal requests, temporarily freezing those users' funds.

---

### Finding Description

When a user calls `initiateWithdrawal`, the `LRTWithdrawalManager` increases `assetsCommitted[asset]` by the expected payout amount and locks the user's rsETH in the contract. [1](#0-0) 

The `assetsCommitted` mapping is the protocol's authoritative record of total pending obligations to queued withdrawal users.

When a user calls `instantWithdrawal`, the contract checks how much is available in the `LRTUnstakingVault` via `getAssetsAvailableForInstantWithdrawal`: [2](#0-1) 

That function computes availability as: [3](#0-2) 

The `reservedBuffer` here is `queuedWithdrawalsBuffer[asset]`, which is a **static value set manually by an operator**: [4](#0-3) 

Critically, `queuedWithdrawalsBuffer` is **never automatically updated** when `assetsCommitted` grows. The `LRTUnstakingVault` has no reference to `LRTWithdrawalManager.assetsCommitted` at all. The two accounting systems are entirely disconnected.

When `unlockQueue` is later called to service queued withdrawals, it reads the vault's raw balance as the available amount: [5](#0-4) 

It then calls `unstakingVault.redeem(asset, assetAmountUnlocked)` to pull funds from the vault: [6](#0-5) 

If instant withdrawals have drained the vault below the amount needed to service queued requests, `unlockQueue` will unlock fewer (or zero) requests, and queued withdrawal users cannot complete their withdrawals.

---

### Impact Explanation

Users who called `initiateWithdrawal` have already transferred their rsETH to the `LRTWithdrawalManager` contract. If the vault is drained by instant withdrawals, those users cannot complete their withdrawals until the vault is replenished by the operator (via EigenLayer unstaking). This constitutes a **temporary freezing of funds** for queued withdrawal users.

**Impact: Medium — Temporary freezing of funds.**

---

### Likelihood Explanation

The scenario requires:
1. `assetsCommitted` to exceed `queuedWithdrawalsBuffer` — this happens naturally as more users queue withdrawals without the operator updating the buffer.
2. Any rsETH holder to call `instantWithdrawal` to drain the vault down to `vaultBalance - queuedWithdrawalsBuffer`.

Both conditions are reachable by unprivileged users without any admin compromise. The operator must manually keep `queuedWithdrawalsBuffer` in sync with `assetsCommitted`, which is an off-chain operational dependency with no on-chain enforcement.

**Likelihood: Medium.**

---

### Recommendation

Replace the static `queuedWithdrawalsBuffer` with a dynamic check against the actual `assetsCommitted` from `LRTWithdrawalManager`. In `getAssetsAvailableForInstantWithdrawal`, query `ILRTWithdrawalManager(withdrawalManager).assetsCommitted(asset)` as the reserved amount instead of the manually-set buffer:

```solidity
function getAssetsAvailableForInstantWithdrawal(address asset) external view returns (uint256) {
    uint256 vaultBalance = balanceOf(asset);
    uint256 committed = ILRTWithdrawalManager(withdrawalManager).assetsCommitted(asset);
    return committed >= vaultBalance ? 0 : vaultBalance - committed;
}
```

This ensures the reserved amount always reflects actual pending obligations, analogous to the recommendation in the original report to distinguish available vs. locked funds.

---

### Proof of Concept

1. Protocol has 1000 ETH in `LRTUnstakingVault`. Operator sets `queuedWithdrawalsBuffer[ETH] = 200`.
2. Users call `initiateWithdrawal` for a total of 800 ETH worth of rsETH. `assetsCommitted[ETH] = 800`.
3. `getAssetsAvailableForInstantWithdrawal` returns `1000 - 200 = 800` (uses static buffer, not `assetsCommitted`).
4. An attacker (or any rsETH holder) calls `instantWithdrawal` for 800 ETH worth of rsETH. The vault now holds 200 ETH.
5. Operator calls `unlockQueue`. `_createUnlockParams` reads `unstakingVault.balanceOf(ETH) = 200`.
6. `_unlockWithdrawalRequests` can only allocate 200 ETH against 800 ETH of queued requests — the remaining 600 ETH of queued withdrawals are blocked.
7. The 600 ETH of queued users' rsETH is locked in `LRTWithdrawalManager` with no path to completion until the operator manually replenishes the vault. [7](#0-6) [8](#0-7)
