### Title
Missing Staleness and Price Validity Checks in `ChainlinkPriceOracle.getAssetPrice()` - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `price`, performing no staleness check and no `price > 0` validation before casting `int256` to `uint256`. This stale or zero price propagates directly into rsETH exchange rate computation.

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the Chainlink price as follows:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

The `updatedAt` timestamp is silently discarded. There is no check of the form `require(updatedAt >= block.timestamp - heartbeat, "stale price")`. If a Chainlink feed stops updating — due to network congestion, feed deprecation, or sequencer downtime — the last reported price is returned as if it were current.

Additionally, `price` is cast directly to `uint256` with no `require(price > 0)` guard. A zero price silently returns 0, understating total ETH in the protocol. [1](#0-0) 

Contrast this with `ChainlinkOracleForRSETHPoolCollateral.getRate()`, which at least checks `if (ethPrice <= 0) revert InvalidPrice()` and `if (timestamp == 0) revert IncompleteRound()`, but is used only for the pool collateral path, not for the core rsETH pricing path. [2](#0-1) 

### Impact Explanation

The stale/zero price flows through the following call chain:

1. `ChainlinkPriceOracle.getAssetPrice()` returns a stale or zero price.
2. `LRTOracle.getAssetPrice()` returns it unchanged.
3. `LRTOracle._getTotalEthInProtocol()` multiplies it by the total deposited asset amount.
4. `LRTOracle._updateRsETHPrice()` uses the resulting `totalETHInProtocol` to compute `newRsETHPrice = (totalETHInProtocol - fee) / rsethSupply`.
5. `rsETHPrice` is stored and used for all subsequent deposits and withdrawals. [3](#0-2) [4](#0-3) 

**Stale inflated price scenario:** If a feed's last reported price is higher than the true current price (e.g., the asset depegged after the last update), `totalETHInProtocol` is overstated, `rsETHPrice` is inflated, and a depositor receives fewer rsETH tokens than the true value of their deposit — a loss for depositors.

**Stale deflated price scenario:** If the last reported price is lower than the true current price, `rsETHPrice` is understated, and a depositor receives more rsETH than the true value of their deposit — a loss for the protocol and existing rsETH holders (dilution/insolvency).

**Zero price scenario:** A zero price for any supported asset causes `_getTotalEthInProtocol()` to undercount ETH, potentially triggering the downside protection pause (`isPriceDecreaseOffLimit`) and freezing deposits and withdrawals. [5](#0-4) 

**Impact classification:** High — theft of unclaimed yield / share mis-accounting for existing rsETH holders; Medium — temporary freezing of funds via spurious pause trigger.

### Likelihood Explanation

Chainlink feeds have documented heartbeat intervals (e.g., 1 hour for ETH/USD). During periods of low volatility, feeds may not update for the full heartbeat window. More critically, if a feed is deprecated or a sequencer goes offline, the last price can persist indefinitely. The public `updateRSETHPrice()` function can be called by anyone at any time, meaning an attacker can deliberately time the call to coincide with a stale price window. [6](#0-5) 

### Recommendation

1. Capture `updatedAt` from `latestRoundData()` and require it to be within the feed's heartbeat:
   ```solidity
   (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
       priceFeed.latestRoundData();
   require(updatedAt >= block.timestamp - MAX_STALENESS, "stale price");
   require(price > 0, "invalid price");
   ```
2. Store a per-feed `maxStaleness` parameter, or use a protocol-wide constant matching the shortest Chainlink heartbeat among all registered feeds.
3. Align `ChainlinkPriceOracle` with the stricter checks already present in `ChainlinkOracleForRSETHPoolCollateral`.

### Proof of Concept

1. Chainlink's ETH/stETH feed (or any supported asset feed) has not updated for longer than its heartbeat (e.g., 24 hours during low volatility).
2. Attacker calls `LRTOracle.updateRSETHPrice()` (no access control).
3. `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)`, which returns the stale price without any staleness check.
4. `rsETHPrice` is set based on the stale price.
5. Attacker deposits assets and mints rsETH at the incorrect exchange rate, profiting at the expense of existing rsETH holders. [7](#0-6) [8](#0-7)
