### Title
`respond_ckd` Skips Cryptographic Output Verification for `AppPublicKey` Variant, Allowing a Single Byzantine Node to Deliver Forged CKD Output — (`File: crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` contract method enforces a cryptographic output check (`ckd_output_check`) only when the pending request uses the `AppPublicKeyPV` variant of `CKDAppPublicKey`. When the request uses the legacy `AppPublicKey` variant, the check arm is entirely empty — the contract accepts and delivers any `CKDResponse` without any on-chain verification. A single Byzantine attested participant (below the signing threshold) can race honest nodes, call `respond_ckd` first with an attacker-chosen response, and cause the user to receive a forged confidential key derivation output that was never computed by the threshold network.

---

### Finding Description

**Root cause — incomplete branch coverage in `respond_ckd`:**

```rust
// crates/contract/src/lib.rs  lines 675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For comparison, `respond` (the signature analogue) always verifies the cryptographic output before resolving the yield: [2](#0-1) 

And `respond_verify_foreign_tx` likewise always verifies: [3](#0-2) 

**Exploit flow:**

1. A user submits `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(user_pk)`. The request enters `pending_ckd_requests`.
2. A single Byzantine attested participant (one compromised node, below threshold) observes the pending request on-chain.
3. Before honest nodes finish the threshold CKD computation, the attacker calls `respond_ckd` with the legitimate `CKDRequest` struct but an attacker-chosen `CKDResponse { big_y: ..., big_c: ... }`.
4. The contract passes all authorization checks (`assert_caller_is_signer`, `assert_caller_is_attested_participant_and_protocol_active`, `is_running_or_resharing`, `accept_requests`).
5. The `match` arm for `AppPublicKey` is empty — no output check fires.
6. `resolve_yields_for` drains the entire pending queue for that request, delivering the forged response to the user. [4](#0-3) 

The `resolve_yields_for` call drains all queued yields for the request in one shot, so the first `respond_ckd` call wins — honest nodes that arrive later find no pending entry and their responses are silently ignored. [5](#0-4) 

---

### Impact Explanation

**Critical — unauthorized confidential key derivation output without required participant authorization.**

The `AppPublicKey` variant is described as "privately verifiable, legacy" in the ABI documentation: [6](#0-5) 

However, "privately verifiable" means the user is expected to verify off-chain using their private key. The contract's failure to enforce any on-chain check means:

- A **single** Byzantine attested participant (strictly below the signing threshold) can unilaterally resolve any pending `AppPublicKey` CKD request with an arbitrary response.
- The threshold MPC computation is entirely bypassed — the forged output was never computed by the network.
- The user receives `big_y` and `big_c` values chosen by the attacker. If the user does not perform rigorous off-chain verification against the MPC network's public key, they will use a key that the attacker knows or controls.
- This directly violates the core security property: confidential key derivation output must require threshold participant authorization.

---

### Likelihood Explanation

**High.** The `AppPublicKey` variant is still a supported, non-deprecated code path (it appears in the ABI and is exercised in tests). Any single attested participant — a role reachable without privileged access beyond joining the network — can exploit this. The attacker only needs to:

1. Be an attested participant (a legitimate network role).
2. Observe a pending `AppPublicKey` CKD request on-chain (public state).
3. Submit `respond_ckd` before honest nodes complete the threshold computation.

No threshold collusion, no leaked keys, and no privileged operator access are required.

---

### Recommendation

Apply `ckd_output_check` (or an equivalent validity check) unconditionally for all `CKDAppPublicKey` variants, mirroring the pattern used in `respond` and `respond_verify_foreign_tx`:

```rust
// respond_ckd — apply check to ALL variants
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(app_pk) => {
        if !ckd_output_check_legacy(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

If a cryptographic check for the `AppPublicKey` variant is not feasible (e.g., because the plain `AppPublicKey` does not carry enough information for on-chain verification), the `AppPublicKey` variant should be deprecated and removed from the contract's accepted inputs, forcing all callers to use `AppPublicKeyPV`.

### Proof of Concept

**Setup:** A running contract with ≥1 attested participant (the attacker) and a pending `AppPublicKey` CKD request from a user.

**Steps:**

1. User calls `request_app_private_key` with `AppPublicKey(user_pk)` and 1 yoctoNEAR deposit. The request is queued in `pending_ckd_requests`.
2. Attacker (single attested participant) constructs a `CKDRequest` matching the pending request and a forged `CKDResponse { big_y: [0u8; 48], big_c: [0u8; 48] }`.
3. Attacker calls `respond_ckd(forged_request, forged_response)`.
4. Contract passes all authorization checks and enters the `match` block. The `AppPublicKey` arm is empty — no panic.
5. `resolve_yields_for` resolves the user's yield with the forged response.
6. User's callback receives `CKDResponse { big_y: [0u8; 48], big_c: [0u8; 48] }` — an attacker-controlled value — as their "confidential key."
7. Honest nodes' subsequent `respond_ckd` calls find no pending entry and are silently discarded.

**Expected (correct) behavior:** The contract should reject the forged response with "CKD output check failed," as it does for `AppPublicKeyPV`.

**Observed behavior:** The contract accepts and delivers the forged response with no error.

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
