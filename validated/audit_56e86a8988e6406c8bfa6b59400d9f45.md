Audit Report

## Title
Daily Mint Limit Exhaustion + Block Stuffing Enables Sustained Deposit DoS — (File: `contracts/pools/RSETHPoolV2.sol`)

## Summary
The `limitDailyMint` modifier enforces only a global daily cap with no per-user sub-limit and no minimum deposit size, allowing a single attacker transaction to exhaust the entire `dailyMintLimit`. On a low-fee L2, the attacker can then use block stuffing to guarantee they are the first depositor at each day boundary, sustaining the DoS indefinitely at low cost.

## Finding Description
The `limitDailyMint` modifier at lines 72–94 of `contracts/pools/RSETHPoolV2.sol` resets `dailyMintAmount` to zero when `currentDay > lastMintDay`, then checks the global cap:

```solidity
if (dailyMintAmount + rsETHAmount > dailyMintLimit) {
    revert DailyMintLimitExceeded();
}
dailyMintAmount += rsETHAmount;
```

There is no per-depositor tracking, no minimum deposit interval, and no minimum deposit size (only `amount == 0` is rejected at line 210). A single call to `deposit()` with sufficient ETH can set `dailyMintAmount == dailyMintLimit`, causing all subsequent deposits to revert with `DailyMintLimitExceeded` for the remainder of the day.

`getCurrentDay()` (lines 246–248) is purely timestamp-based:

```solidity
return (block.timestamp - startTimestamp) / 1 days;
```

Time advances normally even during block stuffing, so the day does reset — but the attacker's stuffed blocks ensure no competing transaction lands before theirs at the day boundary. The exploit loop is:

1. Wait for `currentDay > lastMintDay` (day boundary).
2. Call `deposit()` with ETH such that `rsETHAmount >= dailyMintLimit`. The modifier sets `dailyMintAmount = dailyMintLimit`; all further deposits revert.
3. Fill remaining blocks with cheap filler transactions (block stuffing on a low-fee L2).
4. At day N+1, the attacker's stuffed blocks guarantee they are first, exhausting the limit again before any legitimate user can deposit.
5. Repeat indefinitely.

The existing `nonReentrant` and `whenNotPaused` guards do not constrain per-user deposit size or frequency, and the `limitDailyMint` modifier provides no per-address accounting.

## Impact Explanation
**Low. Block stuffing.** Legitimate users are unable to call `deposit()` for the entire daily window, repeated across consecutive days. User ETH is not lost, but the pool is rendered functionally unusable for deposits for as long as the attacker sustains the campaign. This matches the explicitly allowed impact class "Low. Block stuffing."

## Likelihood Explanation
Realistic on any L2 where the base fee is low enough that filling a day's worth of blocks costs less than the fee revenue the attacker sacrifices. The attacker receives `wrsETH` worth approximately `(1 - feeBps/10000) × dailyMintLimit` ETH per day, so the net cost is only `feeBps/10000 × dailyMintLimit` ETH per day plus block-stuffing gas. On chains like Optimism or Base at low-congestion base fees, block stuffing for 24 hours can cost a few dollars, while the fee cost on a large daily limit is also modest. The attack is repeatable indefinitely with no privileged access required.

## Recommendation
1. **Per-user daily sub-limit**: Track `dailyMintAmountPerUser[msg.sender]` and cap each address's daily contribution to a fraction of `dailyMintLimit`.
2. **Minimum deposit size**: Require `msg.value >= minDepositAmount` to raise the capital cost of exhausting the limit in one shot.
3. **Partial fill instead of revert**: Allow a deposit up to the remaining daily limit rather than reverting entirely, so the attacker cannot atomically exhaust the cap in one transaction without leaving residual capacity for other users.

## Proof of Concept
```solidity
// Foundry fork test (L2 with low base fee)
function testDailyMintExhaustionDoS() public {
    uint256 N = 3;
    for (uint256 day = 0; day < N; day++) {
        // 1. Warp to day boundary
        vm.warp(pool.startTimestamp() + day * 1 days + 1);

        // 2. Attacker exhausts daily limit in one deposit
        uint256 needed = computeETHForLimit(pool.dailyMintLimit(), pool.feeBps(), oracle.getRate());
        vm.prank(attacker);
        pool.deposit{value: needed}("attacker");
        // dailyMintAmount == dailyMintLimit

        // 3. Legitimate user deposit reverts
        vm.expectRevert(RSETHPoolV2.DailyMintLimitExceeded.selector);
        vm.prank(user);
        pool.deposit{value: 1 ether}("user");

        // 4. Assert attacker net cost = fee only (profitable on low-fee L2)
        uint256 fee = needed * pool.feeBps() / 10_000;
        assertLt(fee, blockStuffingCostThreshold);
    }
}
```
The loop confirms that across `N` days the attacker's total cost (fee + stuffing gas) remains below the economic threshold, and no legitimate deposit succeeds during any day window.
