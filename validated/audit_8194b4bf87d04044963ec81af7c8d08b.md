### Title
Missing Borsh Migration Guard for `per_domain_thresholds` in `ProposedThresholdParameters` and `ResharingContractState` Causes Permanent Contract Freeze on Upgrade — (File: `crates/contract/src/v3_12_0_state.rs`)

---

### Summary

The `v3_12_0_state.rs` migration module imports `ProtocolContractState` directly from the current codebase. That type transitively includes `ProposedThresholdParameters` (stored inside `ThresholdParametersVotes` → `RunningContractState`) and `ResharingContractState`, both of which now carry a `per_domain_thresholds` field that did not exist in the v3.12.0 layout. Neither struct is shadowed in the migration module with its old layout. The `#[serde(default)]` attribute on `per_domain_thresholds` only protects JSON deserialization; Borsh reads fields sequentially and has no concept of a default value. If the production contract holds any pending parameter-change votes or is in `Resharing` state at upgrade time, both deserialization attempts in `migrate()` fail, the function panics, and the contract is permanently bricked.

---

### Finding Description

**Root cause — `ProposedThresholdParameters`:**

The internal struct in `crates/contract/src/primitives/thresholds.rs` is:

```rust
#[near(serializers=[borsh, json])]
pub struct ProposedThresholdParameters {
    parameters: ThresholdParameters,
    #[serde(default)]                          // ← JSON-only guard
    per_domain_thresholds: BTreeMap<DomainId, ReconstructionThreshold>,
}
``` [1](#0-0) 

The `#[serde(default)]` attribute is a serde/JSON attribute. It has zero effect on Borsh deserialization. Borsh reads fields in declaration order; if the serialized bytes end after `parameters` (as they would for any state written by v3.12.0 before `per_domain_thresholds` was added), `BorshDeserialize` returns an `io::Error`.

`ProposedThresholdParameters` is stored inside `ThresholdParametersVotes`, which is a field of `RunningContractState`:

```rust
pub struct RunningContractState {
    ...
    pub parameters_votes: ThresholdParametersVotes,  // contains ProposedThresholdParameters
    ...
}
``` [2](#0-1) 

Any production state that has at least one pending `vote_new_parameters` entry will have a serialized `ProposedThresholdParameters` without `per_domain_thresholds`.

**Root cause — `ResharingContractState`:**

```rust
#[near(serializers=[borsh, json])]
pub struct ResharingContractState {
    pub previous_running_state: RunningContractState,
    pub reshared_keys: Vec<KeyForDomain>,
    pub resharing_key: KeyEvent,
    pub cancellation_requests: HashSet<AuthenticatedAccountId>,
    pub per_domain_thresholds: BTreeMap<DomainId, ReconstructionThreshold>,  // new field
}
``` [3](#0-2) 

The DTO version carries `#[serde(default)]` on `per_domain_thresholds`, confirming the field was added after an earlier deployment: [4](#0-3) 

**Root cause — migration module does not shadow either struct:**

`v3_12_0_state.rs` imports `ProtocolContractState` from the live codebase:

```rust
use crate::{
    ...
    state::{ProtocolContractState, running::RunningContractState},
    ...
};
``` [5](#0-4) 

The `v3_12_0_state::MpcContract` struct uses this imported `ProtocolContractState` directly:

```rust
pub struct MpcContract {
    protocol_state: ProtocolContractState,   // ← uses CURRENT layout
    ...
}
``` [6](#0-5) 

The module's own guideline says: *"only copy the structures that have changed"*. Neither `ResharingContractState` nor `ProposedThresholdParameters` is shadowed here, so the migration silently uses the new layout to decode old bytes.

**Root cause — `migrate()` has no recovery path:**

```rust
pub fn migrate() -> Result<Self, Error> {
    match try_state_read::<v3_12_0_state::MpcContract>() {
        Ok(Some(state)) => return Ok(state.into()),
        Ok(None) => return Err(InvalidState::ContractStateIsMissing.into()),
        Err(err) => { log!(...); }   // ← falls through
    };

    match try_state_read::<Self>() {
        Ok(Some(state)) => Ok(state),
        Ok(None) => Err(InvalidState::ContractStateIsMissing.into()),
        Err(err) => env::panic_str(&format!("could not deserialize contract state: {err}")),
        //          ^^^^^^^^^^^^^^^^ permanent panic, contract bricked
    }
}
``` [7](#0-6) 

Both deserialization attempts use the same current-layout types. Both fail on old bytes. The second failure calls `env::panic_str`, which aborts the transaction. Because the binary was already replaced in a prior transaction, the contract is left with a new binary that cannot read its own state.

---

### Impact Explanation

A permanently bricked MPC contract means:

- All in-flight `sign`, `request_app_private_key`, and `verify_foreign_transaction` requests are permanently frozen — no MPC node can respond to them.
- No new signature requests can be accepted.
- All cross-chain funds whose release depends on an MPC-issued signature are permanently inaccessible.

This matches the allowed Critical impact: *"permanent freezing of funds controlled by the MPC network, chain-signature contract, or verified foreign-chain flow."*

---

### Likelihood Explanation

- **Pending parameter votes** (`ThresholdParametersVotes` with at least one `ProposedThresholdParameters` entry) are the normal state during any governance operation (`vote_new_parameters`). On a live network with multiple participants, it is routine for some votes to be in flight at any given time.
- **Resharing state** is transient but occurs during every participant-set change.
- The upgrade governance flow (`propose_update` → `vote_update`) does not enforce that the contract must be in a clean state (no pending votes, not resharing) before the binary is replaced.
- No privileged attacker is required; the bug is triggered by the normal upgrade path.

Likelihood: **High** for the pending-votes path; **Medium** for the resharing path.

---

### Recommendation

1. **Shadow both changed structs in `v3_12_0_state.rs`** with their exact pre-`per_domain_thresholds` layouts:

```rust
// v3_12_0_state.rs
#[derive(BorshSerialize, BorshDeserialize)]
pub struct OldProposedThresholdParameters {
    parameters: ThresholdParameters,
    // per_domain_thresholds did NOT exist in 3.12.0
}

#[derive(BorshSerialize, BorshDeserialize)]
pub struct OldResharingContractState {
    previous_running_state: RunningContractState,
    reshared_keys: Vec<KeyForDomain>,
    resharing_key: KeyEvent,
    cancellation_requests: HashSet<AuthenticatedAccountId>,
    // per_domain_thresholds did NOT exist in 3.12.0
}
```

2. **Add a pre-upgrade gate** in the governance upgrade flow that rejects the binary replacement if the contract is in `Resharing` state or has any pending `ThresholdParametersVotes`.

3. **Add a migration integration test** that exercises the upgrade from a state that is in `Resharing` or has pending parameter votes, to catch this class of regression automatically.

---

### Proof of Concept

1. Deploy the v3.12.0 contract binary.
2. Initialize with N participants.
3. Have at least one participant call `vote_new_parameters` (creating a `ProposedThresholdParameters` entry without `per_domain_thresholds` in the Borsh-serialized state).
4. Deploy the new contract binary (the upgrade vote passes).
5. Call `migrate()`.
6. Observe: `try_state_read::<v3_12_0_state::MpcContract>()` fails because `v3_12_0_state::MpcContract` uses the current `ProtocolContractState` → `RunningContractState` → `ThresholdParametersVotes` → `ProposedThresholdParameters` (with `per_domain_thresholds`), but the stored bytes have no such field.
7. `try_state_read::<Self>()` fails for the same reason.
8. `env::panic_str(...)` is called; the migration transaction reverts.
9. The contract binary is the new version but the state is unreadable — the contract is permanently bricked.
