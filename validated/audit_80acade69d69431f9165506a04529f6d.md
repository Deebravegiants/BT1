### Title
Permanent Locking of Storage-Staking Deposits in `propose_update` — (`File: crates/contract/src/update.rs`)

### Summary
`propose_update` is a `#[payable]` function that collects a storage-staking deposit from the proposer to cover on-chain storage for the update entry. When any update is executed via `do_update`, **all** pending entries are cleared from storage, but the deposits paid by every proposer are never returned. The `UpdateEntry` struct does not record the proposer's account ID, making a refund structurally impossible. There is no cancel/withdraw path. Every upgrade cycle permanently locks NEAR tokens in the contract.

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` accepts a deposit, computes the exact required amount, refunds only the excess, and keeps the remainder: [1](#0-0) 

The required deposit is calculated as `storage_byte_cost × bytes_used`, where `bytes_used` includes the full contract WASM blob plus a fixed overhead for 128 participant votes: [2](#0-1) 

The deposit is kept in the contract to cover storage staking for the `UpdateEntry`. When `do_update` is triggered by a threshold vote, it removes the winning entry and then calls `self.entries.clear()` to wipe all competing proposals: [3](#0-2) 

Clearing the entries frees the on-chain storage, but **no refund is issued to any proposer**. The `UpdateEntry` struct stores only `update` and `bytes_used` — the proposer's `AccountId` is never persisted: [4](#0-3) 

The proposer's account ID is captured in `propose_update` only to refund the excess, then discarded: [5](#0-4) 

There is no `cancel_update` endpoint. `remove_update_vote` removes only the vote record, not the entry or the deposit: [6](#0-5) 

### Impact Explanation

Every upgrade cycle permanently locks NEAR tokens in the contract. For a near-maximum-size contract (~1.5 MB), `bytes_used` ≈ 1.6 MB, yielding a required deposit of roughly 16 NEAR (at NEAR's ~10 yoctoNEAR/byte storage cost). When multiple participants each propose competing updates (as shown in the sandbox test where every participant proposes a config update), all of their deposits are locked simultaneously upon execution. The contract's NEAR balance grows monotonically with each upgrade cycle, and the locked funds are irrecoverable — there is no owner withdrawal function, no refund path in `do_update`, and no cancel mechanism. This matches the **Medium** allowed impact: a production accounting invariant (storage-staking deposits must be returned when storage is freed) is permanently broken by normal contract operation, without requiring any privileged access or network-level attack. [7](#0-6) 

### Likelihood Explanation

The trigger is routine: every contract upgrade or config change requires a `propose_update` call with a deposit. The NEAR MPC network is expected to undergo periodic upgrades (TEE image rotations, config changes, code migrations). Each cycle locks the proposer's deposit. Any participant (a current MPC node operator) can trigger this by calling `propose_update` — no special privilege beyond being a voter is required. [8](#0-7) 

### Recommendation

1. Add a `proposer: AccountId` field to `UpdateEntry` so the deposit owner is recorded at proposal time.
2. In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and issue `Promise::new(entry.proposer).transfer(entry.deposit)` for each.
3. Alternatively, add a `cancel_update(id: UpdateId)` endpoint that allows the original proposer to withdraw their entry and receive a full deposit refund.
4. Consider also refunding the winning proposer's deposit after the upgrade executes successfully (via a callback on the deploy/migrate promise).

### Proof of Concept

1. Participant A calls `propose_update` with a 1.5 MB WASM blob, attaching ~16 NEAR.
2. Participants B and C each call `propose_update` with their own proposals, each attaching their required deposits.
3. Threshold participants vote for A's proposal; `vote_update` calls `do_update`.
4. `do_update` removes A's entry, then calls `self.entries.clear()` — B's and C's entries are wiped.
5. No `Promise::transfer` is issued to A, B, or C.
6. All three deposits remain in the contract with no recovery path.

The sandbox test `test_propose_update_config` demonstrates multiple participants each submitting proposals in a single upgrade cycle, confirming the multi-deposit lock scenario is a normal operational pattern: [9](#0-8)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
