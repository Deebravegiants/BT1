### Title
Fee-on-Transfer Token Support Breaks 1:1 Collateral Invariant in `_deposit()` - (File: contracts/L2/RsETHTokenWrapper.sol)

---

### Summary

`RsETHTokenWrapper._deposit()` and `AGETHTokenWrapper._deposit()` both mint wrapper tokens equal to the caller-supplied `_amount` parameter, but the actual tokens received via `safeTransferFrom` may be less than `_amount` if the underlying asset charges a fee on transfer. This permanently undercollateralizes the wrapper, causing later withdrawers to be unable to redeem their wrapper tokens.

---

### Finding Description

Both `RsETHTokenWrapper._deposit()` and `AGETHTokenWrapper._deposit()` follow the same pattern:

```solidity
// contracts/L2/RsETHTokenWrapper.sol, lines 134–141
function _deposit(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();

    ERC20Upgradeable(_asset).safeTransferFrom(msg.sender, address(this), _amount);

    _mint(_to, _amount);   // <-- mints _amount, not actual received balance
    emit Deposit(_asset, msg.sender, _to, _amount);
}
``` [1](#0-0) 

If the `_asset` token deducts a fee during `transferFrom`, the contract receives `_amount - fee` tokens but mints `_amount` of `wrsETH`. The wrapper's total supply of `wrsETH` therefore exceeds its actual token balance.

The `_withdraw()` function burns `_amount` of `wrsETH` and transfers `_amount` of the underlying token back:

```solidity
// contracts/L2/RsETHTokenWrapper.sol, lines 120–128
function _withdraw(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    _burn(msg.sender, _amount);
    ERC20Upgradeable(_asset).safeTransfer(_to, _amount);
    ...
}
``` [2](#0-1) 

Because the contract holds less underlying than the total `wrsETH` supply, the last withdrawers will find the contract balance insufficient and their withdrawals will revert.

The identical pattern exists in `AGETHTokenWrapper._deposit()`: [3](#0-2) 

A secondary instance exists in `LRTDepositPool.depositAsset()`, where `rsethAmountToMint` is computed from `depositAmount` before the transfer occurs, then minted regardless of the actual received amount: [4](#0-3) 

---

### Impact Explanation

The 1:1 collateral invariant of the wrapper is broken. Every deposit with a fee-on-transfer token creates a shortfall equal to the fee amount. As the shortfall accumulates across multiple deposits, the wrapper contract will not hold enough underlying tokens to honor all outstanding `wrsETH` redemptions. Users who attempt to withdraw last will have their transactions revert — their `wrsETH` is burned but no underlying is returned — constituting a **permanent freezing of funds** for those users.

For `LRTDepositPool.depositAsset()`, the protocol mints more `rsETH` than the actual deposited value, diluting all existing `rsETH` holders and creating protocol insolvency over time.

---

### Likelihood Explanation

The `allowedTokens` mapping in `RsETHTokenWrapper` is managed via `addAllowedToken` (restricted to `TIMELOCK_ROLE`) and `reinitialize`. Currently the allowed tokens are alt-rsETH bridge tokens, which do not charge transfer fees. However, the `addAllowedToken` function permits adding any future token: [5](#0-4) 

If any newly added allowed token implements a fee-on-transfer mechanism (e.g., a rebasing or tax token), or if an existing allowed token upgrades to add fees (as USDT has done on other chains), the vulnerability is immediately exploitable by any unprivileged depositor. The entry path requires no special privileges — `deposit()` and `depositTo()` are fully public. [6](#0-5) 

---

### Recommendation

Use the balance-before/balance-after pattern to determine the actual received amount, and mint only that amount:

```solidity
function _deposit(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();

    uint256 balanceBefore = ERC20Upgradeable(_asset).balanceOf(address(this));
    ERC20Upgradeable(_asset).safeTransferFrom(msg.sender, address(this), _amount);
    uint256 actualReceived = ERC20Upgradeable(_asset).balanceOf(address(this)) - balanceBefore;

    _mint(_to, actualReceived);
    emit Deposit(_asset, msg.sender, _to, actualReceived);
}
```

Apply the same fix to `AGETHTokenWrapper._deposit()` and to `LRTDepositPool.depositAsset()` (compute `rsethAmountToMint` after the transfer using the actual received amount).

---

### Proof of Concept

Assume `altRsETH` is an allowed token in `RsETHTokenWrapper` and charges a 1% fee on transfer.

1. Alice calls `deposit(altRsETH, 100e18)`.
2. `safeTransferFrom` deducts 1% fee: wrapper receives `99e18` altRsETH.
3. `_mint(Alice, 100e18)` — Alice holds `100e18` wrsETH; wrapper holds `99e18` altRsETH.
4. Bob calls `deposit(altRsETH, 100e18)` — wrapper now holds `198e18` altRsETH, total wrsETH supply = `200e18`.
5. Alice calls `withdraw(altRsETH, 100e18)` — burns `100e18` wrsETH, transfers `100e18` altRsETH. Wrapper now holds `98e18` altRsETH, Bob holds `100e18` wrsETH.
6. Bob calls `withdraw(altRsETH, 100e18)` — attempts to transfer `100e18` altRsETH but wrapper only holds `98e18`. **Transaction reverts. Bob's funds are frozen.**

The shortfall grows linearly with each deposit, permanently locking the last depositors' funds.
