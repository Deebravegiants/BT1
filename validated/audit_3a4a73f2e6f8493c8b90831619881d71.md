Audit Report

## Title
rsETH Price Inflation via ETH Donation to LRTDepositPool — Direct Theft of Depositor Funds - (File: contracts/LRTDepositPool.sol / contracts/LRTOracle.sol)

## Summary

`LRTDepositPool` exposes an unrestricted `receive()` fallback that allows any caller to inflate `address(this).balance`. Because `getETHDistributionData()` uses the raw contract balance as the canonical ETH accounting figure, and `LRTOracle._updateRsETHPrice()` derives `rsETHPrice` from that figure via a public, permissionless call, an attacker can permanently inflate the stored price. Subsequent depositors receive fewer rsETH tokens than their ETH warrants, while the attacker — holding rsETH minted before the donation — redeems at the inflated rate, extracting a direct profit at victims' expense.

## Finding Description

**Step 1 — Open ETH receiver**

`LRTDepositPool` declares an unrestricted `receive()` fallback at line 58:

```solidity
receive() external payable { }
```

Any address can send arbitrary ETH to the deposit pool at any time with no access control.

**Step 2 — Raw balance used as canonical accounting**

`getETHDistributionData()` at line 480 reads `address(this).balance` directly:

```solidity
ethLyingInDepositPool = address(this).balance;
```

Donated ETH is indistinguishable from legitimately deposited ETH. There is no internal accounting variable that tracks only authorised inflows.

**Step 3 — Price derived from inflated balance**

`LRTOracle._getTotalEthInProtocol()` (lines 331–349) calls `ILRTDepositPool.getTotalAssetDeposits(ETH_TOKEN)`, which calls `getAssetDistributionData` → `getETHDistributionData()`, summing the inflated balance into `totalETHInProtocol`. `_updateRsETHPrice()` then computes at line 250:

```solidity
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

With `rsethSupply` unchanged and `totalETHInProtocol` artificially elevated, `rsETHPrice` is inflated by the full donation amount.

**Step 4 — `updateRSETHPrice()` is public and permissionless**

Line 87:

```solidity
function updateRSETHPrice() public whenNotPaused {
```

The attacker can trigger the price update themselves immediately after the donation, with no waiting period.

**Step 5 — Minting uses the stored inflated price**

`getRsETHAmountToMint()` at line 520 divides by the stored `rsETHPrice`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

A higher `rsETHPrice` means fewer rsETH tokens minted per unit of ETH deposited by subsequent users.

**Step 6 — No effective guard when `pricePercentageLimit == 0`**

`initialize()` (lines 64–68) never sets `pricePercentageLimit`, so it defaults to `0`. The guard at lines 256–257:

```solidity
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
```

short-circuits to `false` unconditionally, and the price update proceeds with no cap regardless of donation size.

## Impact Explanation

**Critical — Direct theft of user funds.**

An attacker who holds rsETH minted before the donation redeems at the inflated `rsETHPrice`, recovering more ETH than they deposited. The shortfall comes directly from subsequent depositors who receive fewer rsETH tokens than the ETH they contributed warrants. Concretely: with a 1000 ETH donation against 1 wei of existing rsETH supply, a subsequent 2000 ETH depositor receives only 1 wei rsETH (integer division floors the result), while the attacker's 1 wei rsETH entitles them to ~1500 ETH on redemption — a ~500 ETH profit extracted from the victim.

## Likelihood Explanation

**High.** The attack requires no privileged access. The only precondition is holding any nonzero amount of rsETH before the donation, which is obtained by a normal `depositETH` call. The donation itself is a plain ETH transfer to `receive()`. The price update is a public call. The only friction is the withdrawal queue delay, which does not prevent the harm to victims and does not reduce the attacker's eventual payout.

## Recommendation

1. **Track deposited ETH separately from raw balance.** Maintain an internal accounting variable (e.g., `totalDepositedETH`) incremented only through authorised deposit paths (`depositETH`, `receiveFromRewardReceiver`, `receiveFromNodeDelegator`, `receiveFromLRTConverter`) and use that variable instead of `address(this).balance` in `getETHDistributionData()`.

2. **Set a non-zero `pricePercentageLimit` at initialisation.** This limits the per-update price increase and forces large manipulations to span multiple cycles, giving operators time to detect and respond.

3. **Seed the protocol with an initial deposit at deployment** so that the ratio of donated ETH to existing TVL is never large enough to cause meaningful rounding loss for subsequent depositors.

## Proof of Concept

```
Initial state:
  rsethSupply = 1 wei  (attacker deposited 1 wei ETH via depositETH)
  totalETHInProtocol = 1 wei
  rsETHPrice = 1e18

Attack:
  Attacker sends 1000 ETH directly to LRTDepositPool via receive()
  Attacker calls updateRSETHPrice()
    → totalETHInProtocol = 1000e18 + 1
    → newRsETHPrice = (1000e18 + 1) * 1e18 / 1  [divWad with rsethSupply=1]
    → rsETHPrice ≈ 1000e36

Alice deposits 2000 ETH:
  rsethAmountToMint = (2000e18 * 1e18) / (1000e36 + 1e18)
                    = 2000e36 / (1000e36 + 1e18)
                    = floor(2000e18 / (1000e18 + 1))
                    = 1 wei rsETH  (integer division; 2*(1000e18+1) > 2000e18)

Total rsETH supply = 1 wei (attacker) + 1 wei (Alice) = 2 wei
Total ETH in protocol ≈ 3000 ETH

Attacker redeems 1 wei rsETH via withdrawal queue:
  ETH received ≈ 3000 ETH * 1/2 = 1500 ETH

Attacker net: +1500 ETH received − 1000 ETH donated − 1 wei deposited ≈ +500 ETH profit
Alice net:    −2000 ETH deposited, holds 1 wei rsETH worth ≈ 1500 ETH → −500 ETH loss
```

Foundry test plan: deploy `LRTDepositPool` and `LRTOracle` with `pricePercentageLimit = 0`; call `depositETH{value: 1}()` as attacker; send 1000 ether to `address(depositPool)` directly; call `updateRSETHPrice()`; call `depositETH{value: 2000 ether}()` as Alice; assert `rsETH.balanceOf(alice) == 1`; assert attacker's redemption value exceeds 1000 ETH.
