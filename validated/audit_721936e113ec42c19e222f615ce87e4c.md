Audit Report

## Title
`protocolUnpausePool` Sets Pause Level to 1 Instead of 0, Leaving Pool Permanently Swap-Blocked After Protocol Unpause — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

## Summary

`protocolUnpausePool` is intended to restore a protocol-paused pool (level 2) to active status (level 0), but it calls `setPause(1)` instead of `setPause(0)`. After the protocol owner calls `protocolUnpausePool`, the pool transitions from level 2 to level 1 (admin-paused) — still paused. `swap` continues to revert with `PoolPaused` because `_checkNotPaused()` reverts on any non-zero `pauseLevel`. The pool admin must separately call `unpausePool` to complete the transition; if the admin is absent, the pool is permanently stuck at level 1.

## Finding Description

`MetricOmmPool.sol` defines three pause levels: [1](#0-0) 

`_checkNotPaused()` reverts on any non-zero level: [2](#0-1) 

`swap` applies `whenNotPaused` unconditionally: [3](#0-2) 

`protocolUnpausePool` correctly asserts `cur == 2` but then calls `setPause(1)` instead of `setPause(0)`: [4](#0-3) 

After this call, `pauseLevel == 1`, which still satisfies `pauseLevel != 0`, so every subsequent `swap` call reverts with `PoolPaused`. The only path to level 0 is the pool admin calling `unpausePool`, which enforces `cur == 1 → 0`: [5](#0-4) 

If the pool was at level 0 before the protocol pause, the admin never issued a pause and may not know they must act. If the admin is unresponsive, the pool is permanently stuck at level 1.

## Impact Explanation

Core swap functionality is broken: every `swap` call reverts with `PoolPaused` after the protocol owner's "unpause" action. All swap volume and fee accrual is lost until the pool admin separately acts. This directly matches the allowed impact: "Broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows." Severity is High — swap is the primary revenue-generating function of the pool, and the breakage is permanent absent admin cooperation.

## Likelihood Explanation

The bug fires unconditionally on every invocation of `protocolUnpausePool`. No special preconditions, no attacker required. Every protocol-pause/unpause governance cycle leaves the pool admin-paused. The protocol owner has no alternative path to restore swaps directly.

## Recommendation

Change `setPause(1)` to `setPause(0)` in `protocolUnpausePool`, and update the revert argument to match:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
-   IMetricOmmPoolFactoryActions(pool).setPause(1);
+   IMetricOmmPoolFactoryActions(pool).setPause(0);
}
```

## Proof of Concept

1. Pool is active: `pauseLevel == 0`.
2. Protocol owner calls `protocolPausePool(pool)` → `setPause(2)` → `pauseLevel == 2`. Swaps revert.
3. Protocol owner calls `protocolUnpausePool(pool)`:
   - Reads `cur == 2` ✓ passes the guard.
   - Calls `setPause(1)` → `pauseLevel == 1`.
4. Any trader calls `swap(...)` → `whenNotPaused` → `_checkNotPaused()` → `pauseLevel != 0` → **`PoolPaused` revert**.
5. Pool admin must separately call `unpausePool(pool)` (checks `cur == 1`) → `setPause(0)` to restore swaps.
6. If admin is absent, pool remains permanently paused at level 1 despite the protocol owner having "unpaused" it.

Foundry test: deploy pool, call `protocolPausePool`, call `protocolUnpausePool`, assert `swap` reverts with `PoolPaused`, assert `pauseLevel == 1`.
