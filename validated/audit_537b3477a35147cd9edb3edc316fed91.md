### Title
Single Attested Participant Can Forge Foreign-Chain Verification via Cross-Request Payload Hash Substitution - (File: crates/contract/src/lib.rs)

### Summary

`respond_verify_foreign_tx` verifies that a signature is valid for a caller-supplied `payload_hash`, but never checks that `payload_hash` was actually derived from the `request` argument in the same call. A single Byzantine attested participant can replay any previously-issued valid signature under a different pending request, causing the contract to deliver a forged verification attestation to the waiting caller.

### Finding Description

`respond_verify_foreign_tx` accepts two independent arguments: a `VerifyForeignTransactionRequest` (identifying which pending request to resolve) and a `VerifyForeignTransactionResponse` (containing `payload_hash` and `signature`). [1](#0-0) 

The contract's only cryptographic check is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
).is_ok()
```

This confirms the signature is a valid MPC-threshold signature over *some* 32-byte hash, but it does **not** confirm that hash was produced by hashing a `ForeignTxSignPayload` whose embedded `request` field matches the `request` argument. After the check passes, the contract unconditionally resolves all yields queued under `request`: [2](#0-1) 

The signed payload structure is:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))
``` [3](#0-2) 

The `request` field inside the payload binds the hash to a specific `tx_id`, chain, and extractor list. The contract never reconstructs or checks this binding; it only checks the outer ECDSA signature.

The SDK-side verifier (`ForeignChainSignatureVerifier::verify_signature`) does perform this binding check client-side: [4](#0-3) 

But this is a caller-side library, not enforced by the contract. Any on-chain consumer that trusts the contract's yield-resume response without re-running the SDK check is vulnerable.

### Impact Explanation

A single Byzantine attested participant can:

1. Observe any previously completed `respond_verify_foreign_tx` call on-chain and extract `{payload_hash: H_A, signature: sig_A}` (both are public).
2. Submit (or wait for) a new `verify_foreign_transaction` request B targeting a **different** `tx_id` (e.g., a non-existent or attacker-controlled transaction).
3. Call `respond_verify_foreign_tx(request=B, response={payload_hash: H_A, signature: sig_A})`.
4. The contract accepts: `sig_A` is a valid threshold signature over `H_A` ✓. It then resolves all yields queued under request B, delivering `{payload_hash: H_A, signature: sig_A}` to every waiting caller.
5. Any on-chain bridge or vault contract that receives this response and does not independently recompute the expected payload hash will treat it as proof that transaction B was verified by the MPC network — when in fact it was never queried.

The fan-out behavior amplifies the damage: a single forged `respond_verify_foreign_tx` call resolves **all** callers waiting on the same request key simultaneously. [5](#0-4) 

The primary use case for `verify_foreign_transaction` is the Omnibridge inbound flow (foreign chain → NEAR), where the MPC attestation is the sole proof that a foreign deposit finalized. A forged attestation for a non-existent deposit directly enables theft of bridged assets.

### Likelihood Explanation

- Requires only a single attested participant (well below the signing threshold).
- The replay material (`payload_hash` + `signature`) is fully public on-chain after any legitimate signing.
- The attacker controls both the submission of request B and the timing of the forged response.
- No cryptographic capability beyond reading on-chain state is needed.
- The intersection of bridge contracts that do not re-verify the payload hash client-side is non-trivial, especially for contracts written before the SDK verifier existed or that omit the check for gas reasons.

### Recommendation

The contract must enforce the binding between `request` and `response.payload_hash`. The cleanest fix is to change `respond_verify_foreign_tx` to accept the full `ForeignTxSignPayload` instead of only `payload_hash`, recompute the hash on-chain, verify the signature against the recomputed hash, and additionally assert that `payload.request == request.request`. This eliminates the substitution surface entirely:

```rust
// Proposed: accept full payload, not just hash
pub fn respond_verify_foreign_tx(
    &mut self,
    request: VerifyForeignTransactionRequest,
    payload: ForeignTxSignPayload,   // <-- full payload, not just hash
    signature: SignatureResponse,
) -> Result<(), Error> {
    // Enforce binding: payload must encode the same request
    let ForeignTxSignPayload::V1(ref inner) = payload;
    if inner.request != request.request {
        return Err(RespondError::PayloadRequestMismatch.into());
    }
    let payload_hash = payload.compute_msg_hash()?;
    // verify signature over recomputed hash ...
}
```

If changing the ABI is undesirable in the short term, the contract can at minimum recompute the Borsh prefix of the expected payload (the `request` portion) and assert that `response.payload_hash` could only have been produced from a payload containing that prefix — though this requires partial-preimage reasoning and is harder to implement correctly.

### Proof of Concept

```
// Step 1: Alice submits a legitimate request for Bitcoin tx 0xAA...AA
verify_foreign_transaction({ tx_id: 0xAA..AA, chain: Bitcoin, extractors: [BlockHash] })
// → MPC nodes verify, produce sig_A over H_A = SHA256(borsh({request:{tx_id:0xAA..AA,...}, values:[block_hash_real]}))
// → respond_verify_foreign_tx(request_A, {payload_hash: H_A, sig: sig_A}) succeeds on-chain
// → H_A and sig_A are now public

// Step 2: Attacker (an attested participant) submits a fake request for non-existent tx 0xBB...BB
verify_foreign_transaction({ tx_id: 0xBB..BB, chain: Bitcoin, extractors: [BlockHash] })
// → request_B is now pending in pending_verify_foreign_tx_requests

// Step 3: Attacker calls respond with the REPLAYED response from Step 1
respond_verify_foreign_tx(
    request = { tx_id: 0xBB..BB, chain: Bitcoin, ... },  // request B
    response = { payload_hash: H_A, signature: sig_A }   // response from request A
)
// Contract checks: verify_ecdsa(sig_A, H_A, root_pk) → OK  ✓
// Contract does NOT check: H_A == SHA256(borsh({request:{tx_id:0xBB..BB,...}, ...}))
// Contract resolves yields for request_B → caller receives {payload_hash: H_A, sig: sig_A}

// Step 4: Bridge contract on NEAR receives the response, checks sig_A over H_A → valid ✓
// Bridge contract does NOT recompute expected hash from tx_id 0xBB..BB
// Bridge contract releases funds for a deposit that never happened
``` [6](#0-5) [7](#0-6)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
