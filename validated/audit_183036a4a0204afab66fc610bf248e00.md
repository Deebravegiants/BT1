### Title
`RSETHPriceFeed.getRoundData()` Returns Inaccurate Historical Price by Applying Current rsETH Rate to Historical ETH/USD Data - (File: contracts/oracles/RSETHPriceFeed.sol)

### Summary
`RSETHPriceFeed.getRoundData()` is supposed to return the historical rsETH/USD price for a given Chainlink round, but it multiplies the historical ETH/USD price by the **current** `rsETHPrice()` instead of the rsETH/ETH rate that existed at that historical round. This is a direct analog to the TWAP time-window error: just as the original bug uses two wrong checkpoint endpoints to compute an average, this function uses a mismatched pair of data points (historical ETH/USD + current rsETH/ETH) to compute a historical price.

### Finding Description
In `RSETHPriceFeed.sol`, the `getRoundData(uint80 _roundId)` function fetches historical ETH/USD data from the Chainlink aggregator for the requested `_roundId`, but then multiplies it by `RS_ETH_ORACLE.rsETHPrice()`, which is the **current** stored rsETH/ETH exchange rate from `LRTOracle`:

```solidity
function getRoundData(uint80 _roundId)
    external
    view
    returns (uint80 roundId, int256 answer, uint256 startedAt, uint256 updatedAt, uint80 answeredInRound)
{
    (roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.getRoundData(_roundId);
    answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;  // ← current rate, not historical
}
``` [1](#0-0) 

The `rsETHPrice` stored in `LRTOracle` is updated only when `updateRSETHPrice()` is called, and it reflects the current exchange rate, not any historical rate:

```solidity
uint256 public override rsETHPrice;
``` [2](#0-1) 

The result is that `getRoundData(_roundId)` returns `currentRsETHRate × historicalETHUSDPrice`, not the actual historical rsETH/USD price at that round. The `latestRoundData()` function is correct (current rate × current ETH/USD), but `getRoundData` is broken for any round other than the most recent one. [3](#0-2) 

### Impact Explanation
Any external contract or protocol that integrates `RSETHPriceFeed` as a Chainlink-compatible feed and calls `getRoundData` for historical price lookups — such as for TWAP calculations, dispute resolution windows, or historical collateral valuation — will receive a fabricated price. The returned answer is the product of two temporally mismatched data points. As rsETH/ETH accrues yield over time, the current rate will always be higher than historical rates, causing `getRoundData` to systematically overstate historical rsETH/USD prices. This matches the Low impact category: **contract fails to deliver promised returns, but doesn't lose value** within the LRT-rsETH protocol itself.

### Likelihood Explanation
`RSETHPriceFeed` is deployed as a standard `AggregatorV3Interface`-compatible contract, explicitly designed for external integration. Any integrating protocol that calls `getRoundData` (a standard Chainlink interface method) will be affected. The likelihood is medium: the function is publicly callable by any external contract, and the Chainlink interface contract is specifically designed to be consumed by third-party protocols.

### Recommendation
Since the LRT-rsETH protocol does not store historical rsETH/ETH rates on-chain, `getRoundData` cannot be implemented correctly without a historical rate registry. Options:
1. Revert in `getRoundData` with a `NotSupported()` error to prevent silent mispricing.
2. Implement a historical rate checkpoint mapping keyed by Chainlink round ID, updated whenever `updateRSETHPrice()` is called.
3. Document clearly that `getRoundData` is not supported and only `latestRoundData` should be used.

### Proof of Concept
- **Round R1** (30 days ago): ETH/USD = $2,000, rsETH/ETH = 1.02 → actual rsETH/USD = **$2,040**
- **Round R2** (now): ETH/USD = $3,000, rsETH/ETH = 1.05 → actual rsETH/USD = **$3,150**

Calling `getRoundData(R1_roundId)` returns:
```
answer = int256(1.05e18) * int256(2000e8) / 1e18 = 2100e8  // $2,100 — WRONG
```
The correct answer is `$2,040`. The function returns a price inflated by the current rsETH appreciation, not the historical one. An integrating protocol performing a 30-day TWAP using `getRoundData` across multiple rounds would compute a systematically overstated average rsETH/USD price, analogous to the original bug's "time window error" where the wrong checkpoint endpoints produce an inaccurate average.
