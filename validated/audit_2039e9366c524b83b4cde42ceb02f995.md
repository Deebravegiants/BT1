### Title
Unbounded O(proposals √ó DReps) Iteration in `ratifyTransition` Enables Epoch-Boundary Resource Exhaustion ‚Äî (`File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs`)

---

### Summary

The Conway `RATIFY` rule processes every active governance proposal at each epoch boundary. For each proposal it calls `dRepAcceptedRatio`, which performs a full traversal of the entire DRep stake-distribution map (`reDRepDistr`), and `spoAcceptedRatio`, which performs a full traversal of the entire pool-stake-distribution map (`individualPoolStake`). Neither the number of active proposals nor the number of registered DReps/pools is bounded by any protocol parameter. An unprivileged actor who pays the required deposits can grow both collections independently, making the epoch-boundary computation grow as **O(proposals √ó (DReps + pools))** ‚Äî an unbounded resource-limit bug directly analogous to the reported Solidity pattern.

---

### Finding Description

`ratifyTransition` is a recursive function that consumes the `RatifySignal` ‚Äî a `StrictSeq` of all active `GovActionState` values ‚Äî one element at a time:

```
gas@GovActionState {‚Ä¶} SSeq.:<| sigs -> do
  ‚Ä¶
  if ‚Ä¶ && acceptedByEveryone env st gas
    then ‚Ä¶ trans @(RATIFY era) $ TRC (env, st', RatifySignal sigs)
    else ‚Ä¶ trans @(RATIFY era) $ TRC (env, st,  RatifySignal sigs)
``` [1](#0-0) 

`acceptedByEveryone` calls three sub-checks for every proposal: [2](#0-1) 

**`dRepAcceptedRatio`** performs `Map.foldlWithKey' accumStake (0,0) reDRepDistr` ‚Äî a full traversal of the DRep distribution for every single proposal: [3](#0-2) 

**`spoAcceptedRatio`** performs `Map.foldlWithKey' accumStake (0,0) individualPoolStake` ‚Äî a full traversal of the pool distribution for every single proposal: [4](#0-3) 

The `reDRepDistr` map is the DRep stake distribution computed by the `DRepPulser`. Its size is proportional to the number of registered DReps: [5](#0-4) 

The `DRepPulser` computes the DRep distribution **incrementally** across blocks, but `ratifyTransition` itself is **not incremental** ‚Äî it processes all proposals in a single synchronous pass at epoch boundary via `finishDRepPulser`: [6](#0-5) 

There is no protocol parameter capping the total number of simultaneously active proposals. The only natural bound is `ppGovActionLifetime` (6 epochs on mainnet), meaning proposals accumulate across 6 epochs before expiring: [7](#0-6) 

There is also no protocol parameter capping the number of registered DReps or stake pools.

---

### Impact Explanation

At every epoch boundary, `finishDRepPulser` ‚Üí `runConwayRatify` ‚Üí `ratifyTransition` must iterate over every active proposal and, for each one, traverse the entire DRep distribution and pool distribution. The total work is:

```
O(active_proposals √ó (|reDRepDistr| + |individualPoolStake|))
```

If an attacker inflates both dimensions, epoch-boundary processing time grows without bound. Nodes that cannot complete the epoch transition within their allotted window may stall, fall behind, or ‚Äî if different implementations apply different timeouts ‚Äî produce divergent ledger states. This matches the **Medium** allowed impact: *"Attacker-controlled proposals exceed intended validation limits."* In the worst case, if nodes diverge on whether the epoch boundary completed successfully, it escalates to a **High** deterministic-disagreement impact.

---

### Likelihood Explanation

The attack requires paying `ppGovActionDeposit` (100 ADA on mainnet) per proposal and `ppDRepDeposit` (500 ADA on mainnet) per DRep registration. These are non-trivial but not prohibitive costs for a well-funded adversary. With a tx size limit of 16 384 bytes and proposals of ~100‚Äì200 bytes each, dozens of proposals can be packed per transaction. Over 6 epochs of accumulation, thousands of proposals can be active simultaneously. DRep registrations are similarly cheap relative to the computational damage they cause. The attack is permissionless and requires no privileged access.

---

### Recommendation

1. **Add a protocol parameter `maxActiveProposals`** (or per-purpose sub-limits) and enforce it in the `GOV` rule when `proposalsAddAction` is called, rejecting new proposals once the cap is reached.
2. **Alternatively, make `ratifyTransition` incremental** (similar to the DRepPulser pattern) so that per-epoch work is bounded by a configurable pulse size rather than the full cross-product.
3. **Add a protocol parameter `maxDRepRegistrations`** or rely on the existing `dRepDeposit` being set high enough to make the attack economically infeasible at scale ‚Äî but document this as a security invariant that governance must maintain.

---

### Proof of Concept

1. Attacker registers `N` DReps, each paying `ppDRepDeposit`. This inflates `reDRepDistr` to size `N`.
2. Over 6 epochs, attacker submits `M` governance proposals (e.g., `InfoAction`), each paying `ppGovActionDeposit`. Proposals accumulate because `InfoAction` is never ratified (no threshold).
3. At each epoch boundary, `finishDRepPulser` calls `runConwayRatify` ‚Üí `ratifyTransition`, which recurses over all `M` proposals. For each proposal, `dRepAcceptedRatio` traverses all `N` entries in `reDRepDistr` and `spoAcceptedRatio` traverses all registered pools.
4. Total work per epoch boundary: `O(M √ó N)`. With `M = 10 000` proposals and `N = 10 000` DReps, this is `10^8` map-lookup operations per epoch boundary ‚Äî sufficient to cause multi-second stalls on commodity hardware, with no upper bound imposed by the ledger rules.

The relevant code path is:

`NEWEPOCH` ‚Üí `finishDRepPulser` [8](#0-7) 
‚Üí `runConwayRatify` ‚Üí `ratifyTransition` [9](#0-8) 
‚Üí `dRepAcceptedRatio` (full `reDRepDistr` scan per proposal) [10](#0-9)

  ConwayEraAccounts era => RatifyEnv era -> GovActionState era -> ProtVer -> Rational

    { reStakePoolDistr = PoolDistr individualPoolStake totalActiveStake
    , reAccounts
    , reStakePools
    }
  GovActionState
    { gasStakePoolVotes
    , gasProposalProcedure = ProposalProcedure {pProcGovAction}
    }
  pv =
    toInteger yesStake %? (unCoin (unNonZero totalActiveStake) - toInteger abstainStake)
    where

dRepAcceptedRatio ::
  forall era.
  RatifyEnv era ->
  Map (Credential DRepRole) Vote ->
  GovAction era ->
  Rational
dRepAcceptedRatio RatifyEnv {reDRepDistr, reDRepState, reCurrentEpoch} gasDRepVotes govAction =
  toInteger yesStake %? toInteger totalExcludingAbstainStake
  where
    accumStake (!yes, !tot) drep (CompactCoin stake) =
      case drep of
        DRepCredential cred ->
          case Map.lookup cred reDRepState of
            Nothing -> (yes, tot) -- drep is not registered, so we don't consider it
            Just drepState
              | reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, so we don't consider it
              | otherwise ->
                  case Map.lookup cred gasDRepVotes of
                    -- drep hasn't voted for this action, so we don't count
                    -- the vote but we consider it in the denominator:
                    Nothing -> (yes, tot + stake)
                    Just VoteYes -> (yes + stake, tot + stake)
                    Just Abstain -> (yes, tot)
                    Just VoteNo -> (yes, tot + stake)
        DRepAlwaysNoConfidence ->
          case govAction of
            NoConfidence _ -> (yes + stake, tot + stake)
            _ -> (yes, tot + stake)
        DRepAlwaysAbstain -> (yes, tot)
    (yesStake, totalExcludingAbstainStake) = Map.foldlWithKey' accumStake (0, 0) reDRepDistr

ratifyTransition ::
  forall era.
  ( Embed (EraRule "ENACT" era) (RATIFY era)
  , State (EraRule "ENACT" era) ~ EnactState era
  , Environment (EraRule "ENACT" era) ~ ()
  , Signal (EraRule "ENACT" era) ~ EnactSignal era
  , ConwayEraPParams era
  , ConwayEraAccounts era
  ) =>
  TransitionRule (RATIFY era)
ratifyTransition = do
  TRC
    ( env@RatifyEnv {reCurrentEpoch}
      , st@( RatifyState
               rsEnactState@EnactState
                 { ensCurPParams
                 , ensTreasury
                 , ensPrevGovActionIds
                 }
               _rsEnacted
               _rsExpired
               rsDelayed
             )
      , RatifySignal rsig
      ) <-
    judgmentContext
  case rsig of
    gas@GovActionState {gasId, gasExpiresAfter} SSeq.:<| sigs -> do
      let govAction = gasAction gas
      if prevActionAsExpected gas ensPrevGovActionIds
        && validCommitteeTerm govAction ensCurPParams reCurrentEpoch
        && not rsDelayed
        && withdrawalCanWithdraw govAction ensTreasury
        && acceptedByEveryone env st gas
        then do
          newEnactState <-
            trans @(EraRule "ENACT" era) $
              TRC ((), rsEnactState, EnactSignal gasId govAction)
          let
            st' =
              st
                & rsEnactStateL .~ newEnactState
                & rsDelayedL .~ delayingAction govAction
                & rsEnactedL %~ (Seq.:|> gas)
          trans @(RATIFY era) $ TRC (env, st', RatifySignal sigs)
        else do
          -- This action hasn't been ratified yet. Process the remaining actions.
          st' <- trans @(RATIFY era) $ TRC (env, st, RatifySignal sigs)
          -- Finally, filter out actions that have expired.
          if gasExpiresAfter < reCurrentEpoch
            then pure $ st' & rsExpiredL %~ Set.insert gasId
            else pure st'
    SSeq.Empty -> pure $ st & rsEnactStateL . ensTreasuryL .~ Coin 0
