### Title
Unauthenticated On-Chain Injection of Fake `PlotNFT2Wallet` via Crafted Singleton Puzzle — (`chia/wallet/wallet_state_manager.py`)

### Summary
An unprivileged attacker can corrupt a victim wallet's pool membership state by creating real on-chain coins with a crafted singleton puzzle that encodes an arbitrary `launcher_id`. The wallet's `determine_coin_type` creates and **persists** a `PlotNFT2Wallet` for any singleton whose `parent_coin_info` equals the `launcher_id` embedded in the puzzle, without verifying that the `launcher_id` corresponds to a real singleton launcher coin on-chain.

### Finding Description

In `determine_coin_type`, the PlotNFT branch:

```python
if uncurried.mod == PlotNFT.singleton_puzzles.singleton_mod:
    ...
    if matched_plotnft_wallet_id is None and coin_spend.coin.parent_coin_info == next_plot_nft.launcher_id:
        self.wallets[matched_plotnft_wallet_id] = await PlotNFT2Wallet.create(...)
``` [1](#0-0) 

The `launcher_id` is extracted from the puzzle's curried args with no on-chain verification:

```python
launcher_id = bytes32(singleton.args.at("frf").as_atom())
``` [2](#0-1) 

The guard `coin_spend.coin.parent_coin_info == next_plot_nft.launcher_id` is intended to detect the "first spend after launch" (where the parent coin's parent is the launcher). An attacker can trivially satisfy this by setting `launcher_id = G.name()` in the puzzle, where G is the grandparent coin they control.

`fetch_coin_spend` does validate the puzzle_reveal against the coin's puzzle_hash: [3](#0-2) 

This means the attacker **cannot** inject a fake puzzle_reveal for a real coin via the network. However, the attacker can create **real on-chain coins** with the crafted singleton puzzle, bypassing this check entirely.

`PlotNFT2Wallet.create` then persists the fake wallet to the database and subscribes to puzzle hashes derived from the fake `launcher_id`: [4](#0-3) 

### Impact Explanation

The fake `PlotNFT2Wallet` is persisted to the `user_store` database and survives wallet restarts. The wallet subscribes to `p2_singleton_puzzle_hash` and `hint` derived from the fake `launcher_id`. Pool membership state (`PoolingShareState`) is written with attacker-controlled values in `coin_added`: [5](#0-4) 

If the attacker uses a pubkey from the victim's derivation store (observable from prior transactions), `coin_added` succeeds fully, writing a `PoolingShareState` entry with an attacker-controlled `pool_puzzle_hash` and `pool_url`. This redirects pool reward payouts and corrupts pool membership state persistently.

### Likelihood Explanation

- Requires spending a small amount of XCH to create on-chain coins (not free, but trivially affordable).
- Victim's puzzle hashes are observable from the blockchain once used.
- No special privileges, keys, or cryptographic breaks required.
- The attack is fully automatable.

### Recommendation

Before creating a `PlotNFT2Wallet`, verify on-chain that the `launcher_id` corresponds to a real singleton launcher coin (i.e., a spent coin with `puzzle_hash == singleton_launcher_hash`). This requires a `get_coin_state([launcher_id])` call and checking that the returned coin has the correct puzzle hash and is spent. The check at line 1002 should be replaced with this on-chain verification rather than the `parent_coin_info` heuristic.

### Proof of Concept

1. Choose any victim wallet with a known puzzle hash `VICTIM_PH`.
2. Create grandparent coin `G` (any puzzle, e.g., anyone-can-spend).
3. Craft parent coin `P` with puzzle `singleton_mod(singleton_struct(G.name(), singleton_launcher_hash, singleton_mod_hash), inner_puz)` where `inner_puz` produces one `CreateCoin` with a valid PlotNFT memo structure (attacker-chosen pubkey, pool config).
4. Spend `P` creating child coin `C` with hint `VICTIM_PH`.
5. The victim wallet syncs, calls `determine_coin_type` on `C`.
6. `coin_spend.coin.parent_coin_info == G.name() == launcher_id` → guard passes.
7. `PlotNFT2Wallet.create` is called with `launcher_id = G.name()`, persisted to DB.
8. Assert: `user_store` contains a `PLOTNFT_2` wallet entry with the fake `launcher_id`. [6](#0-5) [7](#0-6)
