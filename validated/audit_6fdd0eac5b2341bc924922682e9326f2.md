Audit Report

## Title
Excess ETH Permanently Locked in `MultiChainRateProvider` Due to Missing Refund/Withdrawal Mechanism — (`contracts/cross-chain/MultiChainRateProvider.sol`)

## Summary
`updateRate()` is `payable` and accepts arbitrary `msg.value`, but forwards only the on-chain `estimatedFee` to LayerZero for each receiver. Any ETH above the sum of those fees is permanently trapped: neither `MultiChainRateProvider` nor its two concrete implementations (`RSETHMultiChainRateProvider`, `AGETHMultiChainRateProvider`) define a `receive()`, `fallback()`, `withdraw()`, or ETH-rescue function, and the `Recoverable` utility present elsewhere in the codebase is not inherited here.

## Finding Description
Inside `updateRate()` (L108–137), the loop calls `ILayerZeroEndpoint.estimateFees()` on-chain for each receiver and then calls `send{ value: estimatedFee }`, forwarding only that exact amount to the LZ endpoint. [1](#0-0) 

The `payable(msg.sender)` argument passed to `send()` is the LayerZero **destination-chain** refund address — used when the destination executor is overpaid on the remote chain. It has no effect on ETH already held by `MultiChainRateProvider` on the source chain.

After the loop, `msg.value − Σ(estimatedFee_i)` remains in the contract. The contract inherits only `Ownable` and `ReentrancyGuard`: [2](#0-1) 

Neither parent provides ETH rescue. The two concrete implementations add only `getLatestRate()` overrides and no ETH recovery: [3](#0-2) [4](#0-3) 

The codebase does contain a `Recoverable` utility with `recoverETH()`: [5](#0-4) 

But `MultiChainRateProvider` and its concrete implementations do not inherit from it. Contrast with `CrossChainRateProvider`, which passes the full `msg.value` to LZ and lets LZ handle the refund: [6](#0-5) 

## Impact Explanation
Any ETH sent in excess of `Σ estimatedFee` is permanently frozen in the contract with no owner rescue, no `selfdestruct`, and no fallback that could drain it. This matches the allowed impact: **Critical — Permanent freezing of funds**.

## Likelihood Explanation
`updateRate()` has no access control — any account can call it. [7](#0-6) 

Callers are expected to query `estimateTotalFee()` off-chain and send that amount. However, LZ fees fluctuate between the off-chain estimate and on-chain execution, so callers routinely overpay to guarantee the transaction does not revert. The contract provides no mechanism to enforce `msg.value == Σ estimatedFee`, making overpayment a normal operational pattern rather than a user mistake.

## Recommendation
After the loop, refund any unspent ETH to `msg.sender`:

```solidity
uint256 spent;
for (...) {
    spent += estimatedFee;
    ILayerZeroEndpoint(layerZeroEndpoint).send{ value: estimatedFee }(...);
}
uint256 refund = msg.value - spent;
if (refund > 0) {
    (bool ok,) = payable(msg.sender).call{ value: refund }("");
    require(ok, "ETH refund failed");
}
```

Alternatively, inherit `Recoverable` (already present at `contracts/utils/Recoverable.sol`) to allow the owner to rescue stranded ETH, or add a `require(msg.value == totalEstimatedFee)` guard before the loop to reject over-payments outright.

## Proof of Concept
Deploy `RSETHMultiChainRateProvider` with a mock LZ endpoint that returns `estimatedFee = 0.01 ether` per `send()` call and accepts exactly that value. Configure two rate receivers.

```solidity
function test_excessEthLocked() public {
    uint256 estimatedPerChain = 0.01 ether;
    uint256 totalFee = estimatedPerChain * 2;   // 0.02 ether
    uint256 overpay   = totalFee * 10;           // 0.20 ether

    uint256 balBefore = address(provider).balance;
    vm.deal(address(this), overpay);
    provider.updateRate{ value: overpay }();
    uint256 balAfter = address(provider).balance;

    // 0.18 ether is permanently locked
    assertEq(balAfter - balBefore, overpay - totalFee);

    // No withdrawal path exists
    vm.prank(owner);
    vm.expectRevert();
    provider.recoverETH(owner, balAfter); // function does not exist
}
```

The `assertEq` passes, confirming the excess is locked. The `vm.expectRevert()` block confirms no withdrawal path exists on the deployed contract.
