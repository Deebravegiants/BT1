### Title
DRep Deposit Permanently Frozen When Script Credential Becomes Inaccessible - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/GovCert.hs`)

---

### Summary

In the Conway era, a DRep deposit can only be recovered by submitting a `ConwayUnRegDRep` certificate that is witnessed by the DRep's own credential. There is no automatic expiry-and-refund path for DRep registrations. If the DRep credential is a script whose execution conditions can never be satisfied (e.g., a native multi-sig where a required co-signer loses their key or refuses to cooperate), the deposit is permanently locked in the ledger state with no recovery path short of a hard fork.

---

### Finding Description

**Vulnerability class:** Funds/accounting bug — permanent deposit freeze due to exclusive access control on the only release path.

**Root cause — witness requirement (TxCert.hs):**

`getScriptWitnessConwayTxCert` and `conwayGovCertVKeyWitness` both enforce that an `UnRegDRepTxCert` must be witnessed by the DRep credential itself:

```haskell
-- script path
ConwayUnRegDRep cred _ -> credScriptHash cred

-- key path
ConwayUnRegDRep cred _ -> credKeyHashWitness cred
``` [1](#0-0) 

No alternative witness path exists. Any transaction carrying `UnRegDRepTxCert` that cannot satisfy the DRep credential's script will be rejected by the UTXOW rule.

**Root cause — unregistration logic (GovCert.hs):**

The `ConwayUnRegDRep` branch in the GOVCERT rule simply deletes the DRep from `vsDRepsL` and clears its delegations. The deposit is recovered as part of the transaction's value-conservation balance. There is no protocol-level fallback that fires when the DRep credential is permanently inaccessible:

```haskell
ConwayUnRegDRep cred refund -> do
  let mDRepState = Map.lookup cred (certState ^. certVStateL . vsDRepsL)
  ...
  isJust mDRepState ?! (injectFailure . ConwayDRepNotRegistered) cred
  failOnJust drepRefundMismatch $ injectFailure . ConwayDRepIncorrectRefund . Mismatch refund
  ...
  pure $ certState' & certDStateL . accountsL . accountsMapL
           %~ clearDRepDelegations (drepDelegs dRepState)
``` [2](#0-1) 

**Contrast with governance proposals:**

Governance proposal deposits have an automatic epoch-boundary refund path in `returnProposalDeposits` (`Epoch.hs`). When a proposal expires, the deposit is credited to the return-address account if registered, or sent to the treasury if not — no credential witness is required:

```haskell
processProposal gas (!accounts, !unclaimed)
  | (Just _accountState, newAccounts) <- updateLookupAccountState addRefund cred accounts =
      (newAccounts, unclaimed)
  | otherwise = (accounts, Map.insert (gasId gas) (gasDeposit gas) unclaimed)
``` [3](#0-2) 

No equivalent automatic-expiry-and-refund mechanism exists for DRep registrations. DReps can become *inactive* (expired) due to inactivity, but expiry does not trigger any deposit return; the deposit remains locked until the credential holder submits `UnRegDRepTxCert`.

**Attacker-controlled entry path:**

An unprivileged actor who is a co-signer in a native multi-sig DRep credential (a "witness provider" in the scope definition) can permanently freeze the DRep operator's deposit simply by refusing to provide their signature on any future `UnRegDRepTxCert` transaction. The DRep credential type `ScriptHashObj` is explicitly supported for DRep roles:

```haskell
data ConwayGovCert
  = ConwayRegDRep !(Credential DRepRole) !Coin !(StrictMaybe Anchor)
  | ConwayUnRegDRep !(Credential DRepRole) !Coin
  ...
``` [4](#0-3) 

The `Credential DRepRole` type accepts both `KeyHashObj` and `ScriptHashObj`, so registering a DRep with a multi-sig or Plutus script credential is a fully valid, on-chain operation.

---

### Impact Explanation

**High — Permanent freezing of deposits where recovery requires a hard fork.**

Once the DRep credential becomes permanently inaccessible, the deposit (denominated in ADA, set by `ppDRepDepositL`) is irrecoverable without a protocol change. The deposit pot is tracked in `utxosDeposited`; the funds are neither returned to the operator nor redirected to the treasury — they remain in the deposit pot indefinitely, effectively removed from circulation without any accounting correction.

---

### Likelihood Explanation

**Low to Medium.** The scenario requires a DRep to register with a script credential. This is a supported and documented feature (script credentials are explicitly handled in `getScriptWitnessConwayTxCert`). The freeze can be triggered:

1. **Accidentally** — a native multi-sig DRep credential where one required co-signer loses their key.
2. **Intentionally** — a malicious co-signer in a threshold script who refuses to sign any `UnRegDRepTxCert` transaction, permanently blocking the DRep operator's deposit recovery.

The DRep deposit amount is a governance parameter (`ppDRepDepositL`) and can be substantial. Multiple DReps could be affected simultaneously.

---

### Recommendation

Add an automatic deposit-return path for DRep registrations that mirrors the governance-proposal mechanism:

1. **Automatic expiry refund:** When a DRep's expiry epoch is reached and the DRep is pruned from the active set, automatically return the deposit to a pre-registered beneficiary address (analogous to `pProcReturnAddr` in proposals), without requiring a credential witness.
2. **Permissionless unregistration of expired DReps:** Allow any account to submit `UnRegDRepTxCert` for a DRep whose expiry epoch has passed, bypassing the credential-witness requirement, so that the deposit can be recovered even when the credential is inaccessible.

---

### Proof of Concept

1. Register a DRep using a 2-of-2 native multi-sig script credential (keys A and B):
   ```
   RegDRepTxCert (ScriptHashObj multiSigHash) deposit SNothing
   ``` [5](#0-4) 

2. Key A's holder loses their signing key (or refuses to cooperate).

3. Attempt to submit `UnRegDRepTxCert (ScriptHashObj multiSigHash) deposit`. The UTXOW rule requires the multi-sig script to validate; without key A's signature, the script fails and the transaction is rejected. [6](#0-5) 

4. The DRep remains in `vsDRepsL` indefinitely. The deposit remains in `utxosDeposited`. No epoch-boundary rule ever returns it. No governance action can force its return without a hard fork.

5. The only recovery path is a hard fork that adds a protocol-level mechanism to return the deposit without requiring the DRep credential's authorization — exactly the situation described in the "High. Permanent freezing of funds … where recovery requires a hard fork" impact category.

getScriptWitnessConwayTxCert ::
  ConwayTxCert era ->
  Maybe ScriptHash
getScriptWitnessConwayTxCert = \case
  ConwayTxCertDeleg delegCert ->
    case delegCert of
      ConwayRegCert _ SNothing -> Nothing
      ConwayRegCert cred (SJust _) -> credScriptHash cred
      ConwayUnRegCert cred _ -> credScriptHash cred
      ConwayDelegCert cred _ -> credScriptHash cred
      ConwayRegDelegCert cred _ _ -> credScriptHash cred
  -- PoolIds can't be Scripts
  ConwayTxCertPool {} -> Nothing
  ConwayTxCertGov govCert -> govWitness govCert

      ConwayAuthCommitteeHotKey coldCred _hotCred -> credScriptHash coldCred
      ConwayResignCommitteeColdKey coldCred _ -> credScriptHash coldCred
