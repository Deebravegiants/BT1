### Title
Malicious Leader Node Can Replay a Prior Foreign-Chain Attestation to Fulfill a Different Pending `verify_foreign_transaction` Request — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that `response.payload_hash` carries a valid MPC root-key signature, but never checks that `payload_hash` was actually derived from the pending request's own `tx_id` / `extractors`. A single malicious attested participant acting as the signing leader can replay a threshold signature obtained from any prior legitimate foreign-chain verification to resolve a completely different pending request, delivering a forged attestation to the caller.

---

### Finding Description

In `respond_verify_foreign_tx` the contract performs three checks before resolving a pending yield:

1. Caller is an attested participant.
2. `verify_ecdsa_signature(response.signature, response.payload_hash, root_key)` — the signature is valid over the **node-supplied** `payload_hash`.
3. A pending entry for `request` exists. [1](#0-0) 

Critically, the contract never verifies that `response.payload_hash` was computed from `request.request` (the `tx_id`, `extractors`, `payload_version`, etc.). The `payload_hash` is entirely node-supplied and unconstrained by the original request.

Contrast this with the regular `respond` path, where the payload is taken directly from the **request** object:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,          // ← from the request, not the response
    &expected_public_key,
)
``` [2](#0-1) 

For foreign-tx verification the analogous binding is absent. The node-side code computes `payload_hash` as `ForeignTxSignPayload::V1 { request, values }.compute_msg_hash()`, which embeds the original `tx_id` and extracted values: [3](#0-2) 

But the contract never reconstructs or checks this hash — it only checks the signature over whatever hash the caller supplies.

**Attack sequence (single malicious leader, no threshold collusion):**

1. A legitimate `verify_foreign_transaction(tx_id=A)` is processed. The leader participates honestly, obtains the threshold signature `sig_A` over `payload_hash_A = hash(A_data)`, and calls `respond_verify_foreign_tx(request=A, response={payload_hash_A, sig_A})`. Request A is resolved normally.
2. The leader **retains** `{payload_hash_A, sig_A}`.
3. A new `verify_foreign_transaction(tx_id=B)` is submitted and enters the pending map.
4. The malicious leader immediately calls `respond_verify_foreign_tx(request=B_request, response={payload_hash=payload_hash_A, signature=sig_A})`.
5. The contract checks: `verify_ecdsa_signature(sig_A, payload_hash_A, root_key)` → **valid** ✓ (no check that `payload_hash_A` corresponds to `B_request`).
6. The contract resolves B's pending yield with `{payload_hash_A, sig_A}`.
7. The caller for B receives a valid MPC-signed attestation that actually attests to tx_A's data, not tx_B's.

The `verify_foreign_transaction` entry-point confirms the chain is supported at submission time: [4](#0-3) 

but nothing in the response path re-anchors the signed hash to that original request.

---

### Impact Explanation

This is a **forged foreign-chain verification / cross-chain replay**. The caller for tx_B receives a response that passes all on-chain signature checks (`verify_ecdsa_signature` succeeds, the MPC root key is correct) yet attests to a completely different transaction. Any bridge or cross-chain application that gates fund release on a `VerifyForeignTransactionResponse` would accept this forged attestation, enabling **invalid bridge execution or double-spend conditions** — matching the High impact tier.

---

### Likelihood Explanation

The attack requires only a **single Byzantine attested participant** (the signing leader for any round). No threshold collusion is needed: the leader merely stores a threshold signature produced during any prior legitimate signing and reuses it. Leader election is tied to presignature ownership, so any participant can become the leader for some round. The attack is silent (no on-chain evidence distinguishes a replayed response from a legitimate one) and can be executed in the same block as the victim request is submitted.

---

### Recommendation

Bind the signed payload to the originating request on-chain. Two concrete options:

1. **Recompute and compare**: In `respond_verify_foreign_tx`, recompute the expected `payload_hash` from `request.request` and `response`'s extracted values, and assert it equals `response.payload_hash` before accepting the response.
2. **Include request commitment in the signed message**: Extend `ForeignTxSignPayload` to include a commitment to the on-chain request key (e.g., `hash(domain_id || tx_id || extractors)`), and have the contract verify this commitment is present in `payload_hash` before resolving the yield.

Either approach mirrors what `respond` already does for regular sign requests — taking the payload from the request, not the response.

---

    pub fn respond(
        &mut self,
        request: SignatureRequest,
        response: dtos::SignatureResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!("respond: signer={}, request={:?}", &signer, &request);

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain)?;

        let signature_is_valid = match (&response, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                // generate the expected public key
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");
                let affine = *k256::PublicKey::try_from(&secp_pk)
                    .expect("stored key is always valid")
                    .as_affine();
                let expected_public_key =
                    derive_key_secp256k1(&affine, &request.tweak).map_err(RespondError::from)?;

                let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");

                // Check the signature is correct
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    payload_hash,
                    &expected_public_key,
                )
                .is_ok()
            }
            (
                dtos::SignatureResponse::Ed25519 { signature },
                PublicKeyExtended::Ed25519 {
                    edwards_point: public_key_edwards_point,
                    ..
                },
            ) => {
                let derived_public_key_edwards_point = derive_public_key_edwards_point_ed25519(
                    &public_key_edwards_point,
                    &request.tweak,
                );
                let derived_public_key_32_bytes =
                    dtos::Ed25519PublicKey::from(derived_public_key_edwards_point.compress());

                let message = request.payload.as_eddsa().expect("Payload is not EdDSA");

                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    message,
                    &derived_public_key_32_bytes,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_signature_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
