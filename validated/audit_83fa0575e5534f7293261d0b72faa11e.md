### Title
`SyncStateReader::get_compiled_class_hash` Left as `todo!()` Panics on Gateway Stateful Validation of Declare Transactions — (`File: crates/apollo_gateway/src/sync_state_reader.rs`)

---

### Summary

`SyncStateReader` implements the `BlockifierStateReader` trait for the gateway's stateful validation path. Its `get_compiled_class_hash` method is left as `todo!()`, which panics at runtime. Any `DeclareV2`/`DeclareV3` transaction that causes the blockifier to call `get_compiled_class_hash` during stateful validation will panic, causing the gateway to reject the transaction or crash the validation task. This is the direct Sequencer analog of the external bug: a public outer path (`nonblockingLzReceive` / gateway stateful validation) works, but the inner handler (`_nonblockingLzReceive` / `get_compiled_class_hash`) is left unimplemented, so the path that depends on it silently fails or crashes.

---

### Finding Description

In `crates/apollo_gateway/src/sync_state_reader.rs`, the `BlockifierStateReader` implementation for `SyncStateReader` contains:

```rust
fn get_compiled_class_hash(&self, _class_hash: ClassHash) -> StateResult<CompiledClassHash> {
    todo!()
}
``` [1](#0-0) 

Every other method in the same `impl BlockifierStateReader for SyncStateReader` block is fully implemented by delegating to the state sync client. Only `get_compiled_class_hash` is left as a `todo!()` placeholder.

The `SyncStateReader` is the production state reader used by the gateway for stateful transaction validation: [2](#0-1) 

It is constructed and returned by `SyncStateReaderFactory::get_blockifier_state_reader_and_gateway_fixed_block_from_latest_block` for all non-genesis blocks: [3](#0-2) 

The `SyncOrGenesisStateReader` enum delegates `get_compiled_class_hash` directly to `SyncStateReader::get_compiled_class_hash`: [4](#0-3) 

The blockifier calls `get_compiled_class_hash` during `DeclareV2`/`DeclareV3` transaction execution to verify whether the class is already declared with the same compiled class hash. When this call is made against `SyncStateReader`, the `todo!()` macro panics with `"not yet implemented"`.

Note the contrast: `GenesisStateReader` (used only when no blocks exist) has a correct implementation that returns `Ok(CompiledClassHash::default())`: [5](#0-4) 

---

### Impact Explanation

**High — Mempool/gateway admission rejects valid transactions before sequencing.**

Any user who submits a `DeclareV2` or `DeclareV3` transaction to the gateway after at least one block exists will trigger the `todo!()` panic inside the blockifier's stateful validation. In Rust, `todo!()` panics unconditionally. Depending on how the gateway's async runtime handles panics in blocking threads, this either:

1. Crashes the validation task, causing the gateway to return an internal error and reject the transaction, or
2. Propagates to crash the gateway process entirely.

In either case, valid Cairo 1 class declarations are permanently rejected at the gateway admission layer, before they ever reach the mempool or sequencer.

---

### Likelihood Explanation

**High.** The trigger is any `DeclareV2` or `DeclareV3` transaction submitted after genesis. This is a standard, unprivileged user operation. No special permissions, keys, or network access are required. The code path is exercised on every such transaction.

---

### Recommendation

Implement `get_compiled_class_hash` in `SyncStateReader` by querying the state sync client, following the same pattern as the other methods in the same `impl` block. For example:

```rust
fn get_compiled_class_hash(&self, class_hash: ClassHash) -> StateResult<CompiledClassHash> {
    self.runtime
        .block_on(self.state_sync_client.get_compiled_class_hash(self.block_number, class_hash))
        .map_err(|e| StateError::StateReadError(e.to_string()))
}
```

If the state sync client does not yet expose `get_compiled_class_hash`, that method must be added to the `StateSyncClient` trait and implemented before this path can be used safely in production.

---

### Proof of Concept

1. Start the Apollo sequencer node with at least one committed block (so `SyncStateReaderFactory` returns a `SyncStateReader`, not a `GenesisStateReader`).
2. Submit a `DeclareV2` or `DeclareV3` transaction to the gateway's HTTP endpoint.
3. The gateway's stateful validation calls `blockifier::execute_transaction(...)` on the `SyncOrGenesisStateReader`.
4. The blockifier calls `state.get_compiled_class_hash(class_hash)` during declare execution.
5. `SyncOrGenesisStateReader::get_compiled_class_hash` delegates to `SyncStateReader::get_compiled_class_hash`.
6. `todo!()` panics: `"not yet implemented"`.
7. The gateway rejects the transaction with an internal error (or crashes).
