### Title
Attached NEAR Deposit Not Refunded on Existing-Participant Re-Attestation in `submit_participant_info` - (File: crates/contract/src/lib.rs)

---

### Summary

`submit_participant_info` is marked `#[payable]` but only executes deposit-refund logic in one of its two code paths. When an already-registered participant re-submits their attestation (e.g., for periodic renewal), the deposit-handling block is skipped entirely. Any NEAR attached to that call is permanently absorbed by the MPC contract with no refund path.

---

### Finding Description

`submit_participant_info` in `crates/contract/src/lib.rs` accepts NEAR deposits via `#[payable]`. After calling `tee_state.add_participant(...)`, it evaluates:

```rust
let caller_is_not_participant = self.voter_account().is_err();
let is_new_attestation = matches!(
    attestation_insertion_result,
    ParticipantInsertion::NewlyInsertedParticipant
);

let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... charge for storage, refund excess ...
}
``` [1](#0-0) 

When an existing participant re-submits (e.g., for hourly attestation renewal), `add_participant` returns `ParticipantInsertion::UpdatedExistingParticipant`: [2](#0-1) 

This makes both `is_new_attestation = false` and `caller_is_not_participant = false`, so `attestation_storage_must_be_paid_by_caller = false`. The entire deposit-handling block — including the refund — is bypassed. Any NEAR attached to the call is silently retained by the contract.

The `ParticipantInsertion` enum confirms both variants exist and the update path is a normal, expected code path: [3](#0-2) 

The operator documentation explicitly instructs operators to call `submit_participant_info` manually with `--deposit` for first-time registration: [4](#0-3) 

An operator following this pattern for a subsequent re-attestation (e.g., after a TEE upgrade or key rotation) would permanently lose the attached deposit.

---

### Impact Explanation

Any NEAR attached to a re-submission call by an existing participant is permanently frozen inside the MPC chain-signature contract. There is no recovery path: the contract has no admin withdrawal function, and the deposit-refund logic is structurally unreachable for the `UpdatedExistingParticipant` branch. This breaks the production accounting invariant that all excess deposits must be returned to callers, and permanently freezes caller-controlled funds inside the contract.

**Impact: Medium** — balance/accounting invariant break that permanently freezes NEAR tokens in the chain-signature contract without requiring network-level DoS or privileged operator collusion.

---

### Likelihood Explanation

The MPC node daemon itself attaches `0` NEAR for automated re-submissions, so routine automated operation is unaffected. However, the likelihood is non-trivial because:

1. Operators are explicitly documented to call `submit_participant_info` manually with `--deposit` for first-time registration.
2. The same operator, performing a manual re-attestation after a TEE image upgrade or TLS key rotation, would naturally repeat the same invocation pattern with a deposit attached.
3. The function is `#[payable]` with no documentation warning that deposits are non-refundable on re-submission, making the silent absorption invisible to the caller.
4. The node's periodic re-attestation cadence (hourly) means re-submissions are frequent, increasing the surface for accidental deposit attachment.

---

### Recommendation

Add an unconditional refund of any attached deposit when `attestation_storage_must_be_paid_by_caller` is `false`. Since no storage cost is charged in that branch, the entire attached amount should be returned:

```rust
if attestation_storage_must_be_paid_by_caller {
    // existing storage-charge and refund logic
} else {
    // Refund any deposit attached to a re-submission by an existing participant
    let attached = env::attached_deposit();
    if attached > NearToken::from_yoctonear(0) {
        Promise::new(account_id).transfer(attached).detach();
    }
}
```

Alternatively, add a `#[deposit(0)]` guard or explicitly reject non-zero deposits in the re-submission path to make the invariant explicit.

---

### Proof of Concept

1. Participant `alice.near` calls `submit_participant_info` for the first time with `1 NEAR` deposit → `NewlyInsertedParticipant` → deposit handling runs, excess refunded. ✓

2. `alice.near` later calls `submit_participant_info` again (e.g., after a TEE upgrade) with `1 NEAR` deposit:
   - `add_participant` returns `UpdatedExistingParticipant` [5](#0-4) 
   - `is_new_attestation = false`, `caller_is_not_participant = false`
   - `attestation_storage_must_be_paid_by_caller = false`
   - The `if` block at line 826 is **not entered** [6](#0-5) 
   - `env::attached_deposit()` is never read or refunded
   - `1 NEAR` is permanently absorbed by the MPC contract
   - `alice.near` receives `Ok(())` with no indication that funds were lost
