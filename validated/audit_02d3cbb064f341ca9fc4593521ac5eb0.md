Audit Report

## Title
Push-Only Withdrawal Pattern Causes Permanent Fund Freeze After rsETH Is Burned — (File: `contracts/LRTWithdrawalManager.sol`)

## Summary

`LRTWithdrawalManager` separates rsETH burning (`unlockQueue`, Stage 2) from asset delivery (`completeWithdrawal`, Stage 3) across two distinct transactions. rsETH is irreversibly burned in Stage 2 before the user receives their underlying asset. If the push transfer in Stage 3 permanently reverts — because the recipient is a non-payable contract for ETH withdrawals, or a blacklisted address for ERC20 LST withdrawals — the user's rsETH is permanently destroyed and their entitled assets are frozen in the contract with no on-chain recovery path.

## Finding Description

**Stage 2 — `unlockQueue`**: At line 305, `IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned)` permanently destroys the rsETH. At line 307, `unstakingVault.redeem(asset, assetAmountUnlocked)` moves the underlying asset into `LRTWithdrawalManager`. At line 809, `unlockedWithdrawalsCount[asset]++` is incremented. This entire stage is a separate, committed transaction.

**Stage 3 — `_processWithdrawalCompletion`**: At line 705, `userAssociatedNonces[asset][user].popFront()` removes the nonce. At line 712, `delete withdrawalRequests[requestId]` removes the request. At line 717, `unlockedWithdrawalsCount[asset]--` decrements the counter. At line 734, `_transferAsset(asset, user, request.expectedAssetAmount)` pushes the asset.

The `_transferAsset` function at lines 876–883 uses `payable(to).call{value: amount}("")` for ETH and reverts on failure; it uses `IERC20(asset).safeTransfer(to, amount)` for ERC20 and also reverts on failure (e.g., blacklisted recipient).

If `_transferAsset` reverts, the entire Stage 3 transaction reverts — restoring the nonce, the request record, and the counter. However, the rsETH burned in Stage 2 (a prior committed transaction) is **not** restored. The assets remain in `LRTWithdrawalManager` indefinitely.

There is no cancellation mechanism, no pull-pattern alternative, and no admin override. `sweepRemainingAssets` at line 403 checks `if (hasUnlockedWithdrawals(asset)) revert PendingWithdrawalsExist()`, and `hasUnlockedWithdrawals` at line 630 returns `true` whenever `unlockedWithdrawalsCount[asset] > 0`. Since the stuck request keeps the counter above zero and can never be decremented (Stage 3 always reverts), `sweepRemainingAssets` is also permanently blocked for all users of that asset.

`completeWithdrawalForUser` at line 202 calls the same `_processWithdrawalCompletion` and suffers the identical failure.

## Impact Explanation

A user whose address cannot receive the withdrawn asset loses both their rsETH (burned in Stage 2) and their entitled underlying asset (frozen in `LRTWithdrawalManager` with no recovery path). This constitutes **permanent freezing of funds**, a Critical impact under the allowed scope. The `unlockedWithdrawalsCount` counter for that asset can never reach zero, so `sweepRemainingAssets` is also permanently blocked for all other users of that asset, compounding the impact across the entire withdrawal queue for that asset.

## Likelihood Explanation

Two realistic trigger conditions exist, both reachable by an unprivileged external caller:

1. **ETH withdrawal to a non-payable contract**: Any smart contract lacking a `receive()` or `fallback()` function (DeFi vault, DAO treasury, multisig) that calls `initiateWithdrawal(ETH_TOKEN, ...)` will permanently fail Stage 3. This is a realistic integration pattern requiring no special privileges.
2. **ERC20 LST transfer revert**: Any ERC20 LST that reverts on `transfer` to a specific address (e.g., due to operator-controlled restrictions) will cause the same permanent freeze.

Neither condition requires admin compromise, governance capture, or any privileged access — both are reachable through the normal public withdrawal path.

**Likelihood: Low** (edge-case conditions), but the impact is irreversible and zero-recovery when triggered.

## Recommendation

Implement a pull pattern for asset delivery. After `unlockQueue` redeems assets into `LRTWithdrawalManager`, record the entitled amount in a per-user mapping (e.g., `claimableAssets[user][asset]`) instead of pushing immediately in `_processWithdrawalCompletion`. Expose a separate `claimAsset(asset)` function that lets the user pull their allocation at any time. This decouples rsETH burning from asset delivery and eliminates the permanent-freeze risk. Alternatively, wrap `_transferAsset` in a try/catch and store failed deliveries in a claimable mapping as a fallback.

## Proof of Concept

```
1. Victim is a smart contract (no receive()) holding rsETH.
2. Victim calls initiateWithdrawal(ETH_TOKEN, 1e18, "").
   → rsETH transferred to LRTWithdrawalManager (L166); withdrawal request queued.

3. Operator calls unlockQueue(ETH_TOKEN, ...).
   → _unlockWithdrawalRequests marks victim's request as unlocked,
     increments unlockedWithdrawalsCount[ETH_TOKEN] (L809).
   → L305: IRSETH.burnFrom(address(this), rsETHBurned)  ← rsETH permanently destroyed.
   → L307: unstakingVault.redeem(ETH_TOKEN, amount)     ← ETH now in LRTWithdrawalManager.
   Transaction committed. rsETH burn is irreversible.

4. Victim calls completeWithdrawal(ETH_TOKEN, "").
   → _processWithdrawalCompletion reaches L734:
     _transferAsset(ETH_TOKEN, victim, amount)
     → payable(victim).call{value: amount}("") returns (false, "")
     → revert EthTransferFailed() (L879)
   → Entire tx reverts; nonce, request, and counter are restored.
   → rsETH remains burned. ETH remains in LRTWithdrawalManager.

5. Victim retries indefinitely — always reverts.
   Operator calls completeWithdrawalForUser — same result (L202).
   unlockedWithdrawalsCount[ETH_TOKEN] never reaches 0.
   sweepRemainingAssets reverts with PendingWithdrawalsExist() (L403).

Result: Victim's rsETH is permanently destroyed.
        ETH is permanently frozen in LRTWithdrawalManager.
        sweepRemainingAssets is permanently blocked for ETH asset.
        No on-chain recovery path exists.
```

**Foundry test plan**: Deploy `LRTWithdrawalManager` on a local fork. Deploy a `NoReceive` contract (no `receive()`) that calls `initiateWithdrawal`. Call `unlockQueue` as operator. Assert rsETH balance of `LRTWithdrawalManager` decreased by burned amount. Call `completeWithdrawal` from `NoReceive` and assert it reverts with `EthTransferFailed`. Assert ETH balance of `LRTWithdrawalManager` equals the redeemed amount. Assert `unlockedWithdrawalsCount[ETH_TOKEN] > 0`. Assert `sweepRemainingAssets` reverts with `PendingWithdrawalsExist`.
