### Title
Silent `Word16` Offset Overflow in `MultiAsset` Compact Representation Corrupts Native Asset PolicyIDs — (File: `eras/mary/impl/src/Cardano/Ledger/Mary/Value.hs`)

---

### Summary

The `to` function in `eras/mary/impl/src/Cardano/Ledger/Mary/Value.hs` computes `Word16` byte-offsets for policy IDs and asset names when encoding a `MultiAsset` value into its compact binary representation (`CompactValueMultiAsset`). The only guard against overflow is `assert (isMultiAssetSmallEnough ma)` (line 588), which uses `assert` from `Control.Exception` (imported at line 68). In Haskell, `Control.Exception.assert` is **compiled away to a no-op in optimized builds** (i.e., any build with `-O` or `-O2`, which is standard for production). When the total representation size exceeds 65 535 bytes, the `Word16` offsets silently wrap around via `fromIntegral`, causing the `from` function to read policy IDs and asset names from wrong byte positions and reconstruct an incorrect `MaryValue` from the UTxO.

---

### Finding Description

**Root cause — `to` function, lines 580–700:**

The `pidOffsetMap` and `assetNameOffsetMap` are declared as `Map PolicyID Word16` and `Map AssetName Word16` respectively:

```haskell
pidOffsetMap :: Map PolicyID Word16
pidOffsetMap =
  let offsets =
        enumFromThen (fromIntegral abcRegionSize) (fromIntegral (abcRegionSize + pidSize))
   in Map.fromList (zip (Set.toList pids) offsets)
```

```haskell
assetNameOffsetMap :: Map AssetName Word16
assetNameOffsetMap =
  let offsets = scanl (+) (abcRegionSize + pidBlockSize) assetNameLengths
   in fromIntegral <$> Map.fromList (zip assetNames offsets)
```

Both use `fromIntegral` to convert `Int` offsets to `Word16`. When the total representation size (`repSize = abcRegionSize + pidBlockSize + anameBlockSize`) exceeds 65 535, the conversion silently truncates (wraps modulo 65 536). The only guard is:

```haskell
to v@(MaryValue _ ma) = do
  c <- assert (isMultiAssetSmallEnough ma) (toCompact ada)
```

`assert` from `Control.Exception` is a no-op in optimized builds. The `isMultiAssetSmallEnough` predicate (lines 710–712) is never enforced in production code paths.

**Root cause — `from` function, lines 730–785:**

The `from` function reads back the stored `Word16` offsets and uses them as byte positions:

```haskell
convertTriple (p, a, i) =
  ( PolicyID $ ScriptHash $ Hash.UnsafeHash $
      readShortByteString rep (fromIntegral p) (fromIntegral $ Hash.hashSize ...)
  , AssetName $ readShortByteString rep (fromIntegral a) (assetLen a)
  , fromIntegral i
  )
```

If `p` or `a` are overflowed `Word16` values, `fromIntegral p` and `fromIntegral a` yield wrong byte positions within `rep`, causing `readShortByteString` to extract bytes from the wrong region of the compact blob. The reconstructed `PolicyID` and `AssetName` values are therefore incorrect — they contain bytes from a different region of the representation (e.g., from region A containing quantities, or from a different policy ID's bytes).

The `from` function also has `assert (isMultiAssetSmallEnough ma)` at line 734, which is equally a no-op in production.

**The `isMultiAssetSmallEnough` overflow threshold:**

```haskell
isMultiAssetSmallEnough (MultiAsset ma) =
  44 * M.getSum (foldMap' (M.Sum . length) ma) + 28 * length ma <= 65535
```

With all unique policy IDs (`p = n`), overflow occurs at `n > 910` assets. With a single shared policy ID, overflow occurs at `n > 1489` assets with 32-byte names.

**No ledger validation rule enforces this bound.** The check exists only in test generators (`Test/Cardano/Ledger/Mary/Arbitrary.hs`, lines 131–133) and in the disabled `assert` calls.

---

### Impact Explanation

When a `CompactValueMultiAsset` is stored in the UTxO with overflowed `Word16` offsets, the `from` function reconstructs a `MaryValue` with **wrong `PolicyID` values**. This means:

1. **Incorrect native asset attribution**: The ledger attributes minted tokens to policy IDs that do not correspond to the actual minting scripts. A token minted under policy `P1` may be reconstructed as belonging to policy `P2`.
2. **Ledger divergence**: If different nodes or implementations handle the `fromIntegral` truncation differently (e.g., due to different optimization levels or future refactors), they will compute different UTxO states from the same block, causing a permanent fork requiring a hard fork to resolve.
3. **Invalid state transitions**: Spending a UTxO output whose value has been corrupted in this way will produce incorrect balance checks, potentially allowing creation or destruction of native assets without a valid minting policy witness.

This matches the allowed impact: **High — Deterministic disagreement between honest nodes from ledger rule evaluation, serialization, or script/witness validation** and potentially **Critical — Direct loss, creation, or destruction of native assets through an invalid ledger state transition**.

---

### Likelihood Explanation

The `maxValSize` protocol parameter (Alonzo+) limits the CBOR-serialized size of a transaction output value, which indirectly constrains the number of assets per output. With the current mainnet `maxValSize` of 5 000 bytes, a single output cannot contain enough assets to overflow the Word16 limit. However:

- The `maxValSize` parameter is governance-controlled and can be changed via a protocol parameter update. A governance action raising `maxValSize` above ~52 000 bytes would make this exploitable.
- The protection is entirely implicit — the code itself has no hard enforcement. The `assert` guard is provably absent in production builds.
- The test infrastructure explicitly documents the overflow risk (comments at lines 98–123 of `Test/Cardano/Ledger/Mary/Arbitrary.hs`) but the production code relies solely on the disabled `assert`.
- An attacker who can influence governance (or who operates in a future era where `maxValSize` is larger) can craft a transaction output whose compact representation overflows Word16.

---

### Recommendation

1. **Replace `assert` with a hard failure in `to`**: Return `Nothing` (the function already returns `Maybe CompactValue`) when `not (isMultiAssetSmallEnough ma)`, rather than using `assert`.

```haskell
to v@(MaryValue _ ma) = do
  guard (isMultiAssetSmallEnough ma)   -- returns Nothing if too large
  c <- toCompact ada
  ...
```

2. **Add a ledger validation rule** (analogous to `maxValSize`) that explicitly rejects transaction outputs whose `MultiAsset` would overflow the compact representation, independent of `maxValSize`.

3. **Remove the `assert` in `from`** and replace it with a proper error or a safe reconstruction path, since `from` is called on data already stored in the UTxO and a panic there would be worse than a controlled failure.

---

### Proof of Concept

The following Haskell pseudocode demonstrates the overflow path (analogous to the Solidity PoC in the external report):

```haskell
-- Construct a MultiAsset with 1490 assets under one policy, each with a 32-byte name
-- Total compact representation size:
--   abcRegionSize = 12 * 1490 = 17880
--   pidBlockSize  = 28 * 1    = 28
--   anameBlockSize= 32 * 1490 = 47680
--   repSize       = 65588  >  65535  --> Word16 overflow!

let pid    = PolicyID (ScriptHash someHash)
    assets = Map.fromList [(AssetName (BS.replicate 32 (fromIntegral i)), 1) | i <- [0..1489]]
    ma     = MultiAsset (Map.singleton pid assets)
    mv     = MaryValue 1000000 ma

-- isMultiAssetSmallEnough mv == False
-- but assert is a no-op in optimized builds, so `to mv` proceeds

case to mv of
  Just cv ->
    -- cv contains Word16 offsets that have wrapped around
    -- from cv reconstructs a MaryValue with WRONG PolicyIDs
    let mv' = from cv
    in assert (mv /= mv')  -- mv' has corrupted PolicyIDs
  Nothing -> error "unexpected"
```

The `assetNameOffsetMap` for the 1490th asset name would compute an offset of `17880 + 28 + 32*1489 = 65556`, which truncates to `65556 - 65536 = 20` as a `Word16`. The `from` function then reads 28 bytes starting at byte 20 of `rep` as the policy ID — which is actually bytes from region A (the quantity array), not from region D (the policy ID blob). from :: CompactValue -> MaryValue
from (CompactValueAdaOnly c) = MaryValue (fromCompact c) (MultiAsset Map.empty)
from (CompactValueMultiAsset c numAssets rep) =
  let mv@(MaryValue _ ma) = valueFromList (fromCompact c) triples
   in assert (isMultiAssetSmallEnough ma) mv
  where
    n = fromIntegral numAssets

    ba = byteArrayFromShortByteString rep

    getTripleForIndex :: Int -> (Word16, Word16, Word64)
    getTripleForIndex i =
      -- indexByteArray indices are in terms of the size of the value being indexed
      -- rather than byte offsets.
      -- The corresponding byte offsets are:
      -- q: 0 + 8i
      -- pidix: 8n + 2i
      -- anameix: 8n + 2n + 2i
      -- Dividing by the sized (resp 8,2,2) yields the indices below
      let q = BA.indexByteArray ba i
          pidix = BA.indexByteArray ba (4 * n + i)
          anameix = BA.indexByteArray ba (5 * n + i)
       in (pidix, anameix, q)

    -- raw triples :: [(pid offset, asset name offset, quantity)]
    rawTriples :: [(Word16, Word16, Word64)]
    rawTriples = map getTripleForIndex [0 .. (fromIntegral $ numAssets - 1)]

    triples :: [(PolicyID, AssetName, Integer)]
    triples = map convertTriple rawTriples

    -- Asset name length are calculated by subtracting the offset from the
    -- next greater offset (or from the end of the rep, if there is none.)
    -- For an index pointing to the end of the array, the associated
    -- length will be: offset - length(rep) = 0
    assetLens =
      -- This assumes that the triples are ordered by nondecreasing asset name offset
      let ixs = nubOrd $ map (\(_, x, _) -> x) rawTriples
          ixPairs = zip ixs (drop 1 ixs ++ [fromIntegral $ SBS.length rep])
       in Map.fromList $ (\(a, b) -> (a, fromIntegral $ b - a)) <$> ixPairs
    assetLen :: Word16 -> Int
    assetLen ix = fromJust (Map.lookup ix assetLens)

    convertTriple ::
      (Word16, Word16, Word64) -> (PolicyID, AssetName, Integer)
    convertTriple (p, a, i) =
      ( PolicyID $
          ScriptHash $
            Hash.UnsafeHash $
              readShortByteString
                rep
                (fromIntegral p)
                (fromIntegral $ Hash.hashSize ([] :: [ADDRHASH]))
      , AssetName $ readShortByteString rep (fromIntegral a) (assetLen a)
      , fromIntegral i
      )

-- When we generate a number of MultiAssets all at once, that number happens to have
-- an implicit upper limit due to the Cardano.Ledger.Mary.Value.{to,from}-based
-- compacting operation. This operation is also performed when we serialise to and from CBOR.
--
-- Refering to the haddock for 'Cardano.Ledger.Mary.Value.to' we surmise that
--   1. The offsets for AssetName and PolicyID are stored as Word16 (maxBound = 65535).
--   2. All offsets (including those for AssetName and PolicyID) are relative to the whole
--      of the representation (binary blob) rather than the start of their respective regions.
--   3. If the offsets exceed their maxBounds, they will overflow.
--   4. So, we need to ensure that at least the last of the offsets (AssetName offsets) do
--      not exceed 65535.
--   5. With `n` as the total number of assets, `p` the number of policy ids, the inequality to be satisfied is thus:
--           8n -- Word64 asset quantities
--        +  2n -- Word16 policy id offsets
--        +  2n -- Word16 asset name offsets
--        + 28p -- 28-byte policy ids
--        + 32n -- 32-byte asset names (a maximum of 32 bytes)
--        should be <= 65535, assuming the numer of policies to be maximal (i.e. equal to number of assets)
--        65535 / 72 ~ 910.2 is the maximum number of triples to be safely generated.
--        Or, in other words, 44n + 28p <= 65535
--
-- NOTE: There are some conditions due to which exceeding this number may not
-- result in a guaranteed failure to compact without overflow, because, during compacting
--   1. The asset names and policy ids are deduplicated
--   2. Not all generated asset names are 32-bytes long
-- But, exceeding this number does make the probability of causing overflow > 0.
