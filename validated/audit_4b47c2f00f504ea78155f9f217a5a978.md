### Title
Faulty Boolean Logic in `configureAaveIntegration` Causes Permanent Freezing of ETH in Old Aave Pool - (File: contracts/LRTWithdrawalManager.sol)

### Summary
`LRTWithdrawalManager.configureAaveIntegration` contains a faulty compound boolean condition that uses `== address(0)` instead of `!= address(0)` for the `aavePool` check. This mirrors the exact class of bug in the Taiko report: one operand in a compound `&&` condition is inverted, making the guard always evaluate to `false` when all three Aave addresses are already configured. As a result, the intended "withdraw all funds from old pool before reconfiguring" branch is never executed, permanently freezing any ETH deposited in the old Aave pool.

### Finding Description

In `configureAaveIntegration`, the guard that is supposed to detect an existing Aave configuration and drain it before switching reads:

```solidity
// If reconfiguring an existing Aave integration, collect interest and withdraw all funds first
if (address(aaveAWETH) != address(0) && address(aaveWETHGateway) != address(0) && aavePool == address(0)) {
``` [1](#0-0) 

The comment states the intent: "If reconfiguring an existing Aave integration, collect interest and withdraw all funds first." For this to be true, all three addresses must already be set (non-zero). However, the third sub-expression is `aavePool == address(0)` — the exact opposite of what is needed.

**Truth-table when all three are already configured (the reconfiguration scenario):**

| Sub-expression | Value |
|---|---|
| `address(aaveAWETH) != address(0)` | `true` |
| `address(aaveWETHGateway) != address(0)` | `true` |
| `aavePool == address(0)` | **`false`** (aavePool is already set) |
| **Full condition** | **`false`** |

The branch is **never entered** during a legitimate reconfiguration. The correct third operand is `aavePool != address(0)`.

This is structurally identical to the Taiko `notBeforeTag` bug: `(notBeforeTag != 0x17 && notBeforeTag == 0x18)` always evaluates to `false`, just as `aavePool == address(0)` always evaluates to `false` when the pool is already configured.

### Impact Explanation

When the LRT manager legitimately calls `configureAaveIntegration` to switch to a new Aave pool:

1. **`_collectInterestToTreasury()` is skipped** — accrued interest is not swept.
2. **`_withdrawFromAave(aaveBalance)` is skipped** — all ETH principal deposited in the old Aave pool remains there.
3. **Old aWETH approval is not revoked** — the old gateway retains a `type(uint256).max` allowance over the old aWETH token.
4. **`aaveAWETH`, `aaveWETHGateway`, `aavePool`, `aaveDataProvider` are overwritten** with new addresses.

After reconfiguration:
- `aaveAWETH.balanceOf(address(this))` returns `0` (new token, no deposits yet).
- `totalETHDepositedToAave` still holds the old (inflated) value.
- `_checkAaveHealth()` returns `false` (principal > balance by more than 2 wei), blocking `collectInterestToTreasury`.
- `emergencyWithdrawFromAave` reverts with `InsufficientAaveBalance` because the new aWETH balance is 0.
- `_processWithdrawalCompletion` for ETH withdrawals calls `_withdrawFromAave`, which also reverts with `InsufficientAaveBalance`. [2](#0-1) 

There is no function in the contract that accepts an arbitrary aWETH address to drain the old pool. The ETH deposited in the old Aave pool is **permanently inaccessible**, and all pending ETH withdrawal completions are **temporarily frozen** until the Aave integration is disabled and the contract balance is manually replenished.

### Likelihood Explanation

The `configureAaveIntegration` function is explicitly designed to support reconfiguration (it has the comment, the withdrawal logic, and the approval-revocation logic). Reconfiguring Aave is a normal operational event (e.g., upgrading to a new Aave version, switching pools, or updating gateway addresses). Any legitimate reconfiguration by the LRT manager triggers the bug. No malicious intent or key compromise is required.

### Recommendation

Change `aavePool == address(0)` to `aavePool != address(0)` in the reconfiguration guard:

```solidity
// If reconfiguring an existing Aave integration, collect interest and withdraw all funds first
if (address(aaveAWETH) != address(0) && address(aaveWETHGateway) != address(0) && aavePool != address(0)) {
``` [3](#0-2) 

### Proof of Concept

1. Aave integration is configured: `aavePool = 0xPool1`, `aaveWETHGateway = 0xGW1`, `aaveAWETH = 0xAWETH1`. The contract has deposited 100 ETH into Aave (`totalETHDepositedToAave = 100e18`).
2. LRT manager calls `configureAaveIntegration(0xPool2, 0xGW2, 0xAWETH2, 0xDP2)`.
3. The guard evaluates: `0xAWETH1 != 0 (true) && 0xGW1 != 0 (true) && 0xPool1 == 0 (false)` → `false`. The withdrawal branch is skipped.
4. `aavePool`, `aaveWETHGateway`, `aaveAWETH`, `aaveDataProvider` are set to new addresses. `totalETHDepositedToAave` remains `100e18`.
5. `aaveAWETH.balanceOf(address(this))` (on new token) = `0`. `_checkAaveHealth()` returns `false`.
6. Any user calling `completeWithdrawal` for ETH triggers `_withdrawFromAave` → `aaveBalance = 0` → `revert InsufficientAaveBalance()`.
7. The 100 ETH in `0xPool1` is permanently inaccessible — no contract function can withdraw from the old pool. [4](#0-3)

    function _withdrawFromAave(uint256 amount) internal returns (uint256 withdrawnAmount) {
        if (amount == 0) return 0;

        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        if (aaveBalance == 0) revert InsufficientAaveBalance();

        // Only withdraw up to the principal amount (don't use accrued interest for user withdrawals)
        uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave;

        withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
        if (withdrawnAmount == 0) return 0;

        aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
        totalETHDepositedToAave -= withdrawnAmount;

        emit ETHWithdrawnFromAave(withdrawnAmount, totalETHDepositedToAave);
    }
