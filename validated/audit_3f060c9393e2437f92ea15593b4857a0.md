### Title
Unverified `CKDResponse` for `AppPublicKey` Requests Allows Single Byzantine Participant to Corrupt Confidential Key Derivation Output - (File: crates/contract/src/lib.rs)

### Summary

`respond_ckd` in `crates/contract/src/lib.rs` performs no cryptographic verification of the `CKDResponse` when the request uses the `AppPublicKey` variant. Any single attested participant can call `respond_ckd` with arbitrary `big_y` / `big_c` values and the contract will accept and resolve the pending yield with that fabricated output. This is the direct analog of the Uniswap `_increase` missing-authorization finding: a function that should enforce a correctness/ownership invariant before settling state does not, allowing an unprivileged (single-node, sub-threshold) actor to corrupt the output delivered to the requester.

### Finding Description

`respond_ckd` is the on-chain callback through which MPC nodes deliver the result of a Confidential Key Derivation (CKD) computation. The function correctly gates access to attested participants via `assert_caller_is_attested_participant_and_protocol_active()`, but it then branches on the request's `app_public_key` variant:

```rust
// crates/contract/src/lib.rs  lines 675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` enforces the pairing equation `e(big_c, G2) = e(big_y, pk2) · e(H(app_id), pk)`, which is satisfiable only by a node that knows the root BLS12-381 private key share — i.e., it is a cryptographic proof of correct computation. [2](#0-1) 

For `AppPublicKey`, the arm is an empty no-op. The contract immediately proceeds to resolve all pending yields for the request with whatever `CKDResponse` the caller supplied: [3](#0-2) 

Because `resolve_yields_for` fans the response out to every queued yield for that request key, a single fabricated call resolves all outstanding requests for that `CKDRequest`.

### Impact Explanation

**Critical — Confidential key derivation output without the required participant authorization.**

The CKD protocol is designed so that the derived key `x·H(app_id)` (where `x` is the threshold-held root private key) is only recoverable by the requester using their own app private key `a`: the requester computes `big_c − a·big_y = x·H(app_id)`. If a Byzantine participant substitutes arbitrary `(big_y′, big_c′)`, the requester derives a completely wrong key with no on-chain indication of tampering. Downstream consequences include:

- Funds locked under the derived key are permanently inaccessible (the correct key is never delivered).
- If the requester trusts the output and uses the fake key to authorize cross-chain operations, those operations are silently misdirected.

The threshold signing requirement — the core security guarantee of the MPC network — is entirely bypassed for `AppPublicKey` CKD requests: a single node suffices.

### Likelihood Explanation

**Medium.** The attacker must be an attested participant (valid TEE attestation accepted by the contract). This is a meaningful barrier in production. However, only **one** such participant is required — well below the signing threshold — and the attack is a simple, direct on-chain call that can be submitted immediately after a `request_app_private_key` event is observed on-chain, racing the legitimate leader's `respond_ckd`. No off-chain coordination or key-share knowledge is needed.

### Recommendation

Apply the same pairing-based output check to `AppPublicKey` requests. For the `AppPublicKey(pk1)` variant the verifier already has `pk1` (a G1 point) and the root G2 public key; a check of the form `e(big_c, G2) = e(big_y, ?) · e(H(app_id), root_pk)` can be constructed analogously to `ckd_output_check`. Alternatively, require the response to be accompanied by a threshold-aggregated proof (e.g., a BLS multi-signature over the response) so that a single node cannot unilaterally settle the yield.

### Proof of Concept

1. Alice calls `request_app_private_key` with `AppPublicKey(pk1)` on a BLS12-381 domain. The contract enqueues a yield and stores the `CKDRequest`.
2. Eve, a single attested participant, observes the pending request on-chain.
3. Eve calls `respond_ckd(ckd_request, CKDResponse { big_y: [1u8;48].into(), big_c: [2u8;48].into() })` — arbitrary garbage values.
4. The contract passes the `AppPublicKey(_) => {}` branch with no check, then calls `resolve_yields_for`, resolving Alice's yield with the fabricated response.
5. Alice's awaiting transaction receives `(big_y=[1…], big_c=[2…])` and computes `big_c − a·big_y`, obtaining a random G1 point instead of `x·H(app_id)`.
6. The legitimate leader's subsequent `respond_ckd` call finds no pending yield and silently returns `Ok(())` (or an error), leaving Alice with the corrupted key.

This is directly demonstrated by the existing unit test at line 3404 (`respond_ckd__should_succeed_when_response_is_valid_and_request_exists`), which passes `CKDResponse { big_y: [1u8;48], big_c: [2u8;48] }` — structurally invalid values — and the contract accepts them without complaint for the `AppPublicKey` variant. pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
