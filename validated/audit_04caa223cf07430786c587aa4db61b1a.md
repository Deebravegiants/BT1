### Title
Gas Key Transaction Pre-Check Incorrectly Rejects Valid `DepositFailed` Transactions - (File: `chain/chain/src/runtime/mod.rs`)

### Summary
`can_verify_and_charge_tx` in `chain/chain/src/runtime/mod.rs` treats `TxVerdict::DepositFailed` identically to `TxVerdict::Failed`, causing the RPC/admission pre-check to permanently reject gas key transactions that the runtime would successfully process (charging gas from the gas key and gracefully failing the deposit). This is a direct analog to the GolomTrader pre-check bug: the guard condition is stricter than the actual execution path.

### Finding Description
The gas key feature separates gas payment (from the gas key balance) from deposit payment (from the account balance). When a gas key transaction has sufficient gas key balance for gas costs but insufficient account balance for the deposit, `verify_and_charge_gas_key_tx_ephemeral` returns `TxVerdict::DepositFailed`. The runtime is explicitly designed to handle this gracefully: gas is charged from the gas key, the nonce is updated, but the deposit is not made.

The `TxVerdict` enum documents this clearly:

```rust
// runtime/runtime/src/lib.rs:274-281
pub enum TxVerdict {
    /// All checks passed.
    Success(VerificationResult),
    /// Gas key valid with sufficient gas balance, but account can't cover deposit.
    /// Gas key balance is deducted, account balance unchanged.
    DepositFailed { result: VerificationResult, error: InvalidTxError },
    /// Hard failure (bad key, bad nonce, insufficient balance). No state changes.
    Failed(InvalidTxError),
}
```

However, `can_verify_and_charge_tx` â€” the RPC/admission pre-check â€” collapses `DepositFailed` into the same error branch as `Failed`:

```rust
// chain/chain/src/runtime/mod.rs:784-795
match verify_and_charge_gas_key_tx_ephemeral(
    runtime_config,
    &signer,
    &access_key,
    current_nonce,
    &tx,
    &cost,
    block_height,
    pending_constraints,
) {
    TxVerdict::Success(_) => Ok(()),
    TxVerdict::DepositFailed { error, .. } | TxVerdict::Failed(error) => Err(error),
}
``` [1](#0-0) 

The pre-check condition is: `gas_key_balance >= gas_cost AND account_balance >= deposit_cost`. The actual execution condition is only: `gas_key_balance >= gas_cost` (deposit failure is handled gracefully). The pre-check is strictly more restrictive than the execution path, exactly mirroring the GolomTrader pattern.

### Impact Explanation
Valid gas key transactions are permanently rejected at the RPC admission level with `InvalidTxError::NotEnoughBalanceForDeposit`. Users with gas keys that have sufficient balance for gas costs but insufficient account balance for deposits cannot submit their transactions, even though the protocol explicitly supports processing them. The corrupted protocol value is the **RPC admission result**: a transaction that would produce a valid on-chain outcome (gas charged, nonce updated, deposit gracefully failed) is instead returned as an error to the submitter and never enters the mempool.

### Likelihood Explanation
Any unprivileged user who holds a gas key with sufficient gas balance but whose account balance is below the deposit amount will encounter this rejection. This is a realistic scenario: gas keys are funded independently of the account balance, and a user may intentionally submit a transaction knowing the deposit will fail (e.g., to update a nonce or trigger a gas-only side effect). The scenario is explicitly exercised in `test_gas_key_deposit_failed` in `test-loop-tests/src/tests/gas_keys.rs`, which must use `AdvProduceChunksMode::ProduceWithoutTxVerification` to bypass the pre-check and confirm the runtime processes the transaction correctly. [2](#0-1) [3](#0-2) 

### Recommendation
In `can_verify_and_charge_tx`, treat `TxVerdict::DepositFailed` as a success (or a distinct non-error outcome) so that gas key transactions with insufficient deposit balance are admitted to the pool. The deposit failure is handled gracefully by the runtime. The corrected match should be:

```rust
match verify_and_charge_gas_key_tx_ephemeral(...) {
    TxVerdict::Success(_) | TxVerdict::DepositFailed { .. } => Ok(()),
    TxVerdict::Failed(error) => Err(error),
}
``` [4](#0-3) 

### Proof of Concept
1. Create an account with a gas key funded with sufficient balance for gas costs (e.g., 100 milliNEAR).
2. Ensure the account balance is insufficient for a deposit (e.g., account has 1 yoctoNEAR, deposit is 1 NEAR).
3. Submit a gas key transaction with a deposit exceeding the account balance via public RPC.
4. **Observed**: RPC rejects with `NotEnoughBalanceForDeposit` from `can_verify_and_charge_tx` â€” the transaction never enters the mempool.
5. **Expected**: The transaction should be admitted; the runtime would charge gas from the gas key, update the nonce, and produce a `DepositFailed` outcome â€” as confirmed by `test_gas_key_deposit_failed` which bypasses the pre-check using adversarial mode and observes exactly this runtime behavior.
