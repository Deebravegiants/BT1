### Title
Stale Post-Resharing Votes Counted Toward TEE Allowlist Threshold Due to Inconsistent Participant Filtering - (File: `crates/contract/src/tee/proposal.rs`, `crates/contract/src/tee/measurements.rs`)

---

### Summary

`CodeHashesVotes::vote()` and `MeasurementVotes::vote()` count **all** stored votes without filtering by the current participant set. In contrast, `TeeVerifierVotes::vote()`, `ProviderVotes::vote()`, and `vote_update` explicitly filter out stale votes from removed participants before comparing against the threshold. After a resharing event where the `clean_tee_status` cleanup promise fails (an explicitly acknowledged failure mode), votes from removed participants persist and are counted toward the threshold for adding a malicious MPC image hash or OS measurement to the TEE allowlist.

---

### Finding Description

The contract maintains several vote-tracking structures for governance decisions. Two of them — `CodeHashesVotes` and `MeasurementVotes` — use an **unfiltered** count when checking whether the threshold has been reached:

**`CodeHashesVotes::count_votes`** (`crates/contract/src/tee/proposal.rs`, lines 47–52):
```rust
fn count_votes(&self, proposal: &NodeImageHash) -> u64 {
    self.proposal_by_account
        .values()
        .filter(|&prop| prop == proposal)
        .count() as u64
}
```

**`MeasurementVotes::count_votes`** (`crates/contract/src/tee/measurements.rs`, lines 58–66):
```rust
fn count_votes(&self, action: &MeasurementVoteAction) -> u64 {
    u64::try_from(
        self.vote_by_account
            .values()
            .filter(|a| *a == action)
            .count(),
    )
    .expect("participant count should not overflow u64")
}
```

Both iterate over **all** stored votes with no participant-membership check.

In contrast, the three other vote-counting paths in the same contract explicitly filter by current participants before comparing against the threshold:

**`TeeVerifierVotes::vote`** (`crates/contract/src/tee/verifier_votes.rs`, lines 74–77):
```rust
let voter_set = self.pending.vote(participant, proposal_hash);
voter_set.count_for(|p| participants.is_participant_given_participant_id(&p.get()))
```

**`ProviderVotes::vote`** (`crates/contract/src/foreign_chain_rpc.rs`, lines 183–188):
```rust
voter_set.count_for(|(p, c)| {
    *c == chain && participants.is_participant_given_participant_id(&p.get())
})
```

**`vote_update`** (`crates/contract/src/lib.rs`, lines 1361–1374) — with an explicit comment:
```rust
// Filter votes to only count current participants voting for this specific update.
// This ensures correctness even if the cleanup promise in MpcContract::vote_reshared() fails.
let valid_votes_count = running_state
    .parameters
    .participants()
    .participants()
    .iter()
    .filter(|(account_id, _, _)| {
        self.proposed_updates
            .vote_by_participant
            .get(account_id)
            .is_some_and(|voted_id| *voted_id == id)
    })
    .count();
```

The comment on `vote_update` explicitly acknowledges that the `clean_tee_status` cleanup promise (called after resharing via `vote_reshared`) **can fail**, leaving stale votes from removed participants in storage. The fix was applied to `vote_update` but not to `vote_code_hash` or `vote_add_os_measurement` / `vote_remove_os_measurement`.

The `clean_tee_status` function is described as a private endpoint callable only by the contract itself via a promise, and the `vote_update` regression comment confirms this promise is not guaranteed to execute.

---

### Impact Explanation

**`vote_code_hash`** uses `CodeHashesVotes::vote()` (unfiltered) to check whether the threshold is reached for adding a new MPC Docker image hash to the TEE allowlist. If stale votes from removed participants persist after a failed `clean_tee_status` promise, a single new participant's vote can combine with those stale votes to cross the threshold and whitelist a **malicious MPC image hash**. Nodes running that malicious image can then submit attestations that pass the contract's verification, be accepted as participants, and participate in threshold signing or key generation — enabling unauthorized signature issuance or key-share exposure.

**`vote_add_os_measurement`** has the same flaw for OS measurement allowlisting. **`vote_remove_os_measurement`** (which requires ALL participants) is also affected: stale votes from removed participants inflate the count, making it easier to remove a valid measurement and shrink the accepted TEE surface.

This maps to: **Critical — Bypass of threshold-signature requirements or unauthorized access to MPC key shares / signing capability** (malicious image hash whitelisted below true threshold), and **High — participant/attestation authorization bypass** (invalid node accepted into the signing set).

---

### Likelihood Explanation

The `vote_update` fix comment is direct evidence that the `clean_tee_status` promise failure is a **known, realistic condition**, not a theoretical edge case. Any resharing event where the cleanup promise runs out of gas, panics, or is otherwise dropped leaves stale votes in `CodeHashesVotes` and `MeasurementVotes`. A Byzantine participant who was removed in a resharing can pre-position votes before the resharing completes; if cleanup fails, those votes persist and a single colluding remaining participant can trigger the threshold.

---

### Recommendation

Apply the same participant-filtering pattern used in `TeeVerifierVotes::vote`, `ProviderVotes::vote`, and `vote_update` to `CodeHashesVotes::vote` and `MeasurementVotes::vote`. Specifically, pass the current `ThresholdParameters` (or `Participants`) into these methods and filter the stored vote map before counting:

```rust
// In CodeHashesVotes::vote / count_votes:
fn count_votes_for_participants(&self, proposal: &NodeImageHash, participants: &Participants) -> u64 {
    self.proposal_by_account
        .iter()
        .filter(|(voter, prop)| {
            *prop == proposal
                && participants.is_participant_given_participant_id(&voter.get())
        })
        .count() as u64
}
```

Update `vote_code_hash`, `vote_add_os_measurement`, and `vote_remove_os_measurement` in `lib.rs` to pass the current participant set and use the filtered count, mirroring the fix already applied to `vote_update`.

---

### Proof of Concept

1. Network has 5 participants (P1–P5), threshold = 3. P1 and P2 vote for a malicious MPC image hash `H_evil` via `vote_code_hash`. Vote count = 2 (below threshold 3). No hash is added.
2. A resharing removes P1 and P2. New set: {P3, P4, P5}, threshold = 3. `vote_reshared` fires `clean_tee_status` as a promise, but the promise **fails** (e.g., out of gas). Stale votes from P1 and P2 remain in `CodeHashesVotes::proposal_by_account`.
3. P3 calls `vote_code_hash(H_evil)`. `CodeHashesVotes::count_votes` iterates all stored votes: P1's vote + P2's vote + P3's vote = **3**, which equals the threshold.
4. `H_evil` is added to `allowed_docker_image_hashes`. A node running the malicious image submits an attestation, passes verification, and joins the signing set — below the true governance threshold of 3 current participants. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

    /// Vote to add a new OS measurement set to the allowed list. Requires threshold votes.
    #[handle_result]
    pub fn vote_add_os_measurement(
        &mut self,
        measurement: ContractExpectedMeasurements,
    ) -> Result<(), Error> {
        log!(
            "vote_add_os_measurement: signer={}, measurement={:?}",
            env::signer_account_id(),
            measurement,
        );
        self.voter_or_panic();

        let threshold_parameters = self.protocol_state.threshold_parameters_or_panic();

        let participant = AuthenticatedParticipantId::new(threshold_parameters.participants())?;
        let action = MeasurementVoteAction::Add(measurement.clone());
        let votes = self.tee_state.vote_measurement(action, &participant);

        if votes >= self.threshold()?.value() {
            let added = self.tee_state.add_measurement(measurement);
            log!("OS measurement add result: {}", added);
        }

        Ok(())
    }

    /// Vote to remove an OS measurement set from the allowed list. Requires ALL participants
    /// to vote for removal.
    #[handle_result]
    pub fn vote_remove_os_measurement(
        &mut self,
        measurement: ContractExpectedMeasurements,
    ) -> Result<(), Error> {
        log!(
            "vote_remove_os_measurement: signer={}, measurement={:?}",
            env::signer_account_id(),
            measurement,
        );
        self.voter_or_panic();

        let threshold_parameters = self.protocol_state.threshold_parameters_or_panic();

        let participant = AuthenticatedParticipantId::new(threshold_parameters.participants())?;
        let action = MeasurementVoteAction::Remove(measurement.clone());
        let votes = self.tee_state.vote_measurement(action, &participant);

        // Removal requires ALL participants to vote
        let total_participants = threshold_parameters.participants().len() as u64;
        if votes >= total_participants {
            let removed = self.tee_state.remove_measurement(&measurement);
            log!("OS measurement remove result: {}", removed);
        }

        Ok(())
    }
