Audit Report

## Title
Missing Staleness Validation in `ChainlinkPriceOracle.getAssetPrice()` Enables Deposits at Inflated Prices During LST Depeg - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards the `updatedAt` timestamp and `answeredInRound` value, performing no staleness or round-completeness check. When a supported LST depegs on the market while the Chainlink feed lags, any unprivileged caller can invoke `LRTDepositPool.depositAsset()` and receive rsETH minted at the stale (inflated) oracle price, extracting value from existing rsETH holders and driving the protocol toward insolvency.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` reads the feed as:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L49-55
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

The `updatedAt` and `answeredInRound` return values are fully discarded. No heartbeat check, no round-completeness check, and no `price > 0` guard are applied.

This oracle is consumed by `LRTOracle.getAssetPrice()` (L156-158), which is in turn called by `LRTDepositPool.getRsETHAmountToMint()`:

```solidity
// contracts/LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.rsETHPrice()` is a cached value updated only when `updateRSETHPrice()` is called externally. The minting ratio is therefore entirely dependent on the Chainlink feed being current.

The same `getAssetPrice()` is used inside `LRTOracle._getTotalEthInProtocol()` (L339), meaning the stale price also propagates into the TVL calculation used to compute the new rsETH price on the next `updateRSETHPrice()` call.

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` (L26-37) explicitly checks `answeredInRound < roundID`, `timestamp == 0`, and `ethPrice <= 0`, demonstrating the protocol is aware of the pattern but failed to apply it to the primary deposit path.

The `depositAsset()` function (L99-118) is fully permissionless — it requires only `whenNotPaused` and `onlySupportedERC20Token(asset)`, both of which are satisfied under normal operating conditions.

The downside protection in `_updateRsETHPrice()` (L270-281) — which pauses the protocol if the new rsETH price drops beyond `pricePercentageLimit` — does not prevent the initial exploit. It only triggers after the fact when `updateRSETHPrice()` is called, and only if `pricePercentageLimit > 0`. The attacker's deposit and value extraction occur before this call.

## Impact Explanation

**Protocol insolvency (Critical).**

When stETH depegs to 0.95 ETH while the Chainlink feed still reports 1.00 ETH:

1. Attacker deposits 10,000 stETH (true market value: 9,500 ETH) via `depositAsset()`.
2. `getRsETHAmountToMint()` prices stETH at 1.00 ETH per the stale feed, minting rsETH representing 10,000 ETH of claim.
3. The attacker receives ~526 ETH of excess rsETH claim at the expense of existing holders.
4. When `updateRSETHPrice()` is eventually called with the correct stETH price, `_getTotalEthInProtocol()` computes a lower TVL, the new `rsETHPrice` drops, and all existing holders are diluted.
5. Repeated at scale, the protocol's backing collapses below 1:1, making rsETH insolvent.

This matches the allowed impact: **Protocol insolvency (Critical)**.

## Likelihood Explanation

LST depeg events are historically documented (stETH traded at a significant discount during the 2022 Celsius/3AC crisis). Chainlink LST/ETH feeds have heartbeats of up to 24 hours and update only on 0.5% deviation thresholds, meaning the feed can lag the market by minutes to hours during a rapid depeg. The entry path (`depositAsset`) is fully permissionless with no access control. Any MEV bot or informed trader monitoring the spread between the Chainlink feed and DEX spot price can profitably exploit this window. The attack is repeatable until the feed updates or the protocol is manually paused.

## Recommendation

Add staleness and validity checks in `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();
    require(answeredInRound >= roundId, "Stale price");
    require(updatedAt != 0, "Incomplete round");
    require(block.timestamp - updatedAt <= MAX_STALENESS, "Price too old");
    require(price > 0, "Invalid price");
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`MAX_STALENESS` should be configured per-asset based on the Chainlink feed's documented heartbeat (e.g., 3600 seconds for a 1-hour heartbeat feed, 86400 + buffer for a 24-hour heartbeat feed).

## Proof of Concept

**Setup:** Fork mainnet. Deploy or mock a Chainlink stETH/ETH feed that returns a stale price of 1.00e18 with `updatedAt` set to `block.timestamp - 2 hours` (within a 24h heartbeat but stale relative to market).

**Steps:**

1. Set market stETH price to 0.95 ETH on a DEX (or simply observe a real depeg scenario). The Chainlink feed still returns 1.00e18.
2. Attacker calls `LRTDepositPool.depositAsset(stETH, 10_000e18, 0, "")`.
3. `_beforeDeposit` → `getRsETHAmountToMint` → `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` returns 1.00e18 (stale).
4. `rsethAmountToMint = (10_000e18 * 1.00e18) / rsETHPrice`. With `rsETHPrice ≈ 1.02e18`, attacker receives ≈ 9,804 rsETH representing ≈ 10,000 ETH of claim.
5. True ETH value deposited: 9,500 ETH. Excess claim: ≈ 500 ETH extracted from existing holders.
6. Call `LRTOracle.updateRSETHPrice()`. `_getTotalEthInProtocol()` now uses the (still stale) Chainlink price, or if the feed has updated, computes a lower TVL. Either way, the attacker's rsETH represents more ETH than was deposited.
7. Assert: `rsETHPrice` after update is lower than before the attacker's deposit, confirming dilution of existing holders.

**Foundry invariant test:** Assert that for any `depositAsset` call, `totalETHInProtocol / rsETH.totalSupply() >= rsETHPrice_before_deposit`. This invariant breaks when the Chainlink feed is stale.
