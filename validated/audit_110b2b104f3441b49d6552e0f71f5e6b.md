### Title
Reward Tokens Permanently Locked When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

---

### Summary

`KernelDepositPool.sol` implements a Synthetix-style staking rewards contract. When all stakers withdraw their KERNEL tokens during an active reward period, `totalKernelStaked` drops to zero. At that point, `rewardPerToken()` stops accumulating, and all reward tokens scheduled for the remaining duration of the period are permanently locked in the contract. There is no admin recovery function, no sweep mechanism, and no way to redistribute these tokens. The contract itself acknowledges this in its NatSpec but provides no on-chain enforcement or recovery path.

---

### Finding Description

`rewardPerToken()` contains a zero-stake guard:

```solidity
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;  // accumulation halts
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
``` [1](#0-0) 

When `totalKernelStaked == 0`, `rewardPerTokenStored` is frozen. The `rewardRate` and `finishAt` remain set, meaning the contract continues to "promise" rewards at the configured rate, but no user ever accrues them. The tokens for that window accumulate silently in the contract balance.

`initiateWithdrawal` decrements `totalKernelStaked` immediately:

```solidity
balanceOf[msg.sender] -= _amount;
totalKernelStaked -= _amount;
``` [2](#0-1) 

Once the last staker's `claimWithdrawal` completes, `totalKernelStaked` is zero. The contract has no `recoverERC20`, no `sweep`, and no admin function that can extract the stranded reward tokens. The only outbound path for `rewardsToken` is `getReward()`, which only transfers `rewards[msg.sender]` — tokens that were never accrued to anyone cannot be claimed by anyone. [3](#0-2) 

The contract's own NatSpec acknowledges the issue but relies entirely on an off-chain operational promise:

```
* @dev If `totalKernelStaked` ever hits zero during a reward distribution window, any remaining rewards
*      for that period will stay locked in the contract. In this deployment, we're avoiding this issue by
*      ensuring there are always some tokens staked...
``` [4](#0-3) 

There is no on-chain enforcement of this invariant. Any staker can withdraw at any time.

---

### Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

Reward tokens deposited by the admin via `notifyRewardAmount` for a given period are permanently locked in the contract if `totalKernelStaked` reaches zero before `finishAt`. There is no function to recover or redistribute them. The tokens are not lost to an attacker, but they are irreversibly inaccessible to both users and the protocol.

---

### Likelihood Explanation

**Medium.** Any combination of stakers can independently and legitimately call `initiateWithdrawal` followed by `claimWithdrawal` at any time. No coordination or malicious intent is required — rational stakers may simply exit the pool. The `withdrawalDelay` introduces a time buffer but does not prevent the condition. The `notifyRewardAmount` guard (`if (totalKernelStaked == 0) revert NoStakedTokens()`) only prevents starting a new period with zero stake; it does not prevent stake from draining to zero mid-period. [5](#0-4) 

---

### Recommendation

1. **Add a token recovery function** restricted to `DEFAULT_ADMIN_ROLE` that can sweep stranded `rewardsToken` back to the admin or treasury, callable only after `finishAt` has passed and all user `rewards[]` balances are zero (or after a separate accounting of unallocated tokens).

2. **Track unallocated rewards explicitly**: maintain a `unallocatedRewards` counter that accumulates `rewardRate * elapsed` whenever `totalKernelStaked == 0`, and allow the admin to redistribute or recover that amount.

3. **Alternatively**, enforce the invariant on-chain by preventing `initiateWithdrawal` from reducing `totalKernelStaked` to zero while `block.timestamp < finishAt`, or by pausing reward accrual and storing the "missed" window for redistribution when staking resumes.

### Proof of Concept

```
Setup:
  - duration = 100 seconds
  - Admin calls notifyRewardAmount(1000 tokens)
    → rewardRate = 1000 / 100 = 10 tokens/sec
    → finishAt = block.timestamp + 100

T=0:   Alice stakes 1000 KERNEL
T=10:  Alice calls initiateWithdrawal(1000)
         → totalKernelStaked = 0
         → Alice's accrued rewards[Alice] = 10 * 10 = 100 tokens (checkpointed by updateReward)
T=10+withdrawalDelay: Alice calls claimWithdrawal()
         → totalKernelStaked remains 0

T=10 to T=100 (90 seconds):
  rewardPerToken() returns rewardPerTokenStored (frozen)
  rewardRate * 90 = 900 tokens are never accrued to anyone

T=100: finishAt passes. 900 reward tokens remain in the contract.
  Alice calls getReward() → receives 100 tokens (her accrued amount)
  Remaining 900 tokens: no function can extract them.
  rewardsToken.balanceOf(address(KernelDepositPool)) = 900 (permanently locked)
```
