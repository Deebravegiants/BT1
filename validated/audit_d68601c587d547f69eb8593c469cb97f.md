### Title
Harvester Can Unconditionally Redirect Farmer Block Rewards Without Fee-Threshold Enforcement — (`File: chia/farmer/farmer_api.py`)

### Summary
The CHIP-22 fee convention allows third-party harvesters to redirect the farmer's block reward to their own address as a fee, gated by a probabilistic `fee_quality` threshold. However, the farmer never cryptographically enforces this threshold: it only logs a warning when the threshold is violated, then unconditionally uses the harvester-supplied `farmer_reward_address_override` in `DeclareProofOfSpace`. A malicious harvester can redirect 100% of the farmer's XCH block rewards to any address it chooses, for every winning proof, with no on-chain or protocol-level enforcement stopping it.

### Finding Description

**Root cause — no enforcement of the fee threshold:**

In `_process_respond_signatures`, the farmer blindly accepts `response.farmer_reward_address_override` from the harvester's `RespondSignatures` message:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override   # ← no threshold check
    include_source_signature_data = True
``` [1](#0-0) 

The only "guard" is `notify_farmer_reward_taken_by_harvester_as_fee`, called earlier in `new_proof_of_space`:

```python
if new_proof_of_space.farmer_reward_address_override is not None:
    self.farmer.notify_farmer_reward_taken_by_harvester_as_fee(sp, new_proof_of_space)
``` [2](#0-1) 

That function only **logs** a warning when `fee_quality > fee_threshold`; it returns `None`, raises no exception, and sets no flag that would prevent the override from being used: [3](#0-2) 

**Attack path:**

1. Malicious third-party harvester connects to the farmer (normal, expected operation).
2. For every winning proof of space, the harvester sends `NewProofOfSpace` with `farmer_reward_address_override = <attacker_address>` and `fee_info.applied_fee_threshold = 0xFFFFFFFF` (claiming 100% fee rate).
3. Farmer calls `notify_farmer_reward_taken_by_harvester_as_fee`, which logs a warning but does not block.
4. Farmer sends `RequestSignatures` to the harvester; harvester responds with `RespondSignatures` also carrying `farmer_reward_address_override = <attacker_address>`.
5. `_process_respond_signatures` sets `farmer_reward_address = <attacker_address>` unconditionally.
6. Farmer broadcasts `DeclareProofOfSpace` to the full node with the attacker's puzzle hash as the farmer reward destination.
7. The full node creates the block; the 0.25 XCH farmer reward coin is created at the attacker's address.

The `farmer_reward_address_override` field in `RespondSignatures` is also completely independent of the one in `NewProofOfSpace` — the harvester can omit it from `NewProofOfSpace` (avoiding even the log warning) and inject it only in `RespondSignatures`, where there is **zero** check of any kind. [4](#0-3) 

### Impact Explanation
**High — unauthorized payout redirection of XCH block rewards.**

A malicious third-party harvester can redirect every farmer block reward (0.25 XCH per block) to an arbitrary address. The farmer operator, who connected to the harvester expecting the CHIP-22 fee convention to be enforced, loses all block rewards. This matches the allowed High impact category: "Bypass of … pool … authorization that enables … payout redirection."

### Likelihood Explanation
Any operator using a third-party harvester service (a common and growing use case enabled by CHIP-22) is exposed. The harvester needs only to set one field in a protocol message it already sends. No key compromise, no admin access, and no cryptographic break is required.

### Recommendation
1. After computing `fee_quality` in `notify_farmer_reward_taken_by_harvester_as_fee`, return a boolean indicating whether the override is legitimate, and have `new_proof_of_space` (and `_process_respond_signatures`) reject the override — not merely log — when `fee_quality > fee_threshold`.
2. Cross-check `RespondSignatures.farmer_reward_address_override` against the override (if any) declared in the corresponding `NewProofOfSpace` entry stored in `self.farmer.proofs_of_space`, so a harvester cannot inject an override at the `RespondSignatures` stage that was never declared upfront.
3. Consider requiring the harvester to sign the `farmer_reward_address_override` field with the plot key so the farmer can verify the harvester actually controls the destination address.

### Proof of Concept
```
Malicious harvester → NewProofOfSpace(
    farmer_reward_address_override = attacker_ph,   # any address
    fee_info = ProofOfSpaceFeeInfo(applied_fee_threshold = 0xFFFFFFFF)
)

Farmer: notify_farmer_reward_taken_by_harvester_as_fee() → logs warning, returns None
Farmer: sends RequestSignatures to harvester

Malicious harvester → RespondSignatures(
    farmer_reward_address_override = attacker_ph    # same or different address
)

_process_respond_signatures():
    farmer_reward_address = attacker_ph             # ← no threshold check

DeclareProofOfSpace(farmer_reward_address = attacker_ph) → broadcast to full node
Result: 0.25 XCH farmer reward coin created at attacker_ph, not farmer's address
```

    def notify_farmer_reward_taken_by_harvester_as_fee(
        self, sp: farmer_protocol.NewSignagePoint, proof_of_space: harvester_protocol.NewProofOfSpace
    ) -> None:
        """
        Apply a fee quality convention (see CHIP-22: https://github.com/Chia-Network/chips/pull/88)
        given the proof and signage point. This will be tested against the fee threshold reported
        by the harvester (if any), and logged.
        """
        assert proof_of_space.farmer_reward_address_override is not None

        challenge_str = str(sp.challenge_hash)

        ph_prefix = self.config["network_overrides"]["config"][self.config["selected_network"]]["address_prefix"]
        farmer_reward_puzzle_hash = encode_puzzle_hash(proof_of_space.farmer_reward_address_override, ph_prefix)

        self.log.info(
            f"Farmer reward for challenge '{challenge_str}' "
            + f"taken by harvester for reward address '{farmer_reward_puzzle_hash}'"
        )

        fee_quality = calculate_harvester_fee_quality(proof_of_space.proof.proof, sp.challenge_hash)
        fee_quality_rate = float(fee_quality) / float(0xFFFFFFFF) * 100.0

        if proof_of_space.fee_info is not None:
            fee_threshold = proof_of_space.fee_info.applied_fee_threshold
            fee_threshold_rate = float(fee_threshold) / float(0xFFFFFFFF) * 100.0

            if fee_quality <= fee_threshold:
                self.log.info(
                    f"Fee threshold passed for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
            else:
                self.log.warning(
                    f"Invalid fee threshold for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
                self.log.warning(
                    "Harvester illegitimately took a fee reward that "
                    + "did not belong to it or it incorrectly applied the fee convention."
                )
        else:
            self.log.warning(
                "Harvester illegitimately took reward by failing to provide its fee rate "
                + f"for challenge '{challenge_str}'. "
                + f"Fee quality was {fee_quality_rate:.3f}% ({fee_quality} or 0x{fee_quality:08x})"
            )
```
