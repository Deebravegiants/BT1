### Title
Missing Supported-Chain Re-Validation in `respond_verify_foreign_tx` Allows Signature Delivery After Chain Removal - (File: `crates/contract/src/lib.rs`)

---

### Summary

`verify_foreign_transaction()` enforces that the requested foreign chain is in the contract's supported set before queuing a request. `respond_verify_foreign_tx()` — the completion path — performs no equivalent check. If governance removes a chain from the supported set after a request is queued but before a node calls `respond_verify_foreign_tx`, the contract accepts the response and delivers the signature to the caller, bypassing the governance decision.

---

### Finding Description

`verify_foreign_transaction()` performs a supported-chain gate at request time:

```rust
// crates/contract/src/lib.rs  lines 533-542
let requested_chain = request.request.chain();
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) {
    env::panic_str(
        &InvalidParameters::ForeignChainNotSupported {
            requested: requested_chain,
        }
        .to_string(),
    );
}
``` [1](#0-0) 

`respond_verify_foreign_tx()` — the node-facing completion function — performs no analogous check. It only verifies the caller is an attested participant, that the protocol is running-or-resharing, that `accept_requests` is set, and that the cryptographic signature is valid:

```rust
// crates/contract/src/lib.rs  lines 691-754
pub fn respond_verify_foreign_tx(
    &mut self,
    request: VerifyForeignTransactionRequest,
    response: VerifyForeignTransactionResponse,
) -> Result<(), Error> {
    let signer = Self::assert_caller_is_signer();
    self.assert_caller_is_attested_participant_and_protocol_active();
    if !self.protocol_state.is_running_or_resharing() { ... }
    if !self.accept_requests { ... }
    // ← no supported-chain check here
    ...
    pending_requests::resolve_yields_for(...)
}
``` [2](#0-1) 

The supported-chain set is dynamic. It is computed from per-node registrations and the RPC whitelist via `update_available_chains_config_cache`, which filters chains by whether `≥ signing_threshold` active participants cover them:

```rust
// crates/contract/src/foreign_chains_metadata.rs  lines 41-66

``` [3](#0-2) 

A chain can leave the supported set at any time — nodes deregistering coverage, a governance vote removing a chain from the RPC whitelist, or the available-chain threshold no longer being met. None of these events invalidate already-queued requests at the contract level.

---

### Impact Explanation

This maps to **Medium** impact: *request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants*.

When governance removes a chain from the supported set (e.g., because its RPC providers are compromised, the chain has reorganized, or the network has voted to stop supporting it), the intent is that no further verifications for that chain should be honored. The missing check in `respond_verify_foreign_tx` means that any request queued before the removal can still be completed and a valid threshold signature delivered to the caller. The caller receives a cryptographically valid attestation that a foreign-chain transaction was verified, even though the network has explicitly revoked support for that chain. In a bridge context this can enable execution of a cross-chain operation that governance intended to block.

---

### Likelihood Explanation

The race window is realistic in production:

1. A chain is in the supported set; a user submits `verify_foreign_transaction` — the request is queued.
2. Governance removes the chain (e.g., via a vote that drops it from the RPC whitelist, or enough nodes deregister coverage so the threshold is no longer met).
3. MPC nodes that had already begun computing the signature (or completed it before the removal propagated) call `respond_verify_foreign_tx`.
4. The contract accepts the response without re-checking the supported-chain set.

No privileged access beyond a normal user submitting a request is required. The attacker-controlled entry path is simply submitting `verify_foreign_transaction` for a chain that is about to be removed, then waiting for the nodes to respond.

---

### Recommendation

Add a supported-chain re-validation at the start of `respond_verify_foreign_tx`, mirroring the check in `verify_foreign_transaction`:

```rust
pub fn respond_verify_foreign_tx(
    &mut self,
    request: VerifyForeignTransactionRequest,
    response: VerifyForeignTransactionResponse,
) -> Result<(), Error> {
    // ... existing checks ...

    // Re-validate that the chain is still supported at response time.
    let requested_chain = request.request.chain();
    let supported_chains = self.get_supported_foreign_chains();
    if !supported_chains.contains(&requested_chain) {
        return Err(InvalidParameters::ForeignChainNotSupported {
            requested: requested_chain,
        }.into());
    }

    // ... signature verification and resolve_yields_for ...
}
```

This mirrors the pattern used in `verify_foreign_transaction` and ensures that governance decisions to remove a chain take effect immediately for all pending requests, not just new ones.

---

### Proof of Concept

1. Deploy the contract with Bitcoin in the supported set.
2. Call `verify_foreign_transaction` with a Bitcoin request — succeeds, request is queued.
3. Remove Bitcoin from the supported set (e.g., all nodes deregister Bitcoin coverage so `update_available_chains_config_cache` drops it from `available_foreign_chains`).
4. Confirm `verify_foreign_transaction` with a new Bitcoin request now fails with `ForeignChainNotSupported`.
5. Call `respond_verify_foreign_tx` with a valid signature for the original queued request — **succeeds**, delivering the signature to the caller despite Bitcoin no longer being supported.

The contract at `crates/contract/src/lib.rs` lines 691–754 contains no call to `get_supported_foreign_chains()` or any equivalent check, so step 5 succeeds unconditionally as long as the cryptographic signature is valid. [4](#0-3) [5](#0-4)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

    pub(crate) fn update_available_chains_config_cache(
        &mut self,
        active_tls_keys: &BTreeSet<dtos::Ed25519PublicKey>,
        threshold: u64,
    ) {
        let mut chain_to_supporter_count: std::collections::BTreeMap<dtos::ForeignChain, u64> =
            std::collections::BTreeMap::new();
        for tls_key in active_tls_keys {
            let Some(chains) = self.foreign_chains_configs.get(tls_key) else {
                continue;
            };
            for chain in chains.iter() {
                if self.rpc_whitelist.entries.is_whitelisted(chain) {
                    let count = chain_to_supporter_count.entry(*chain).or_default();
                    *count = count
                        .checked_add(1)
                        .expect("supporter count bounded by participant set size");
                }
            }
        }
        self.available_foreign_chains = chain_to_supporter_count
            .into_iter()
            .filter_map(|(chain, count)| (count >= threshold).then_some(chain))
            .collect::<BTreeSet<_>>()
            .into();
    }
