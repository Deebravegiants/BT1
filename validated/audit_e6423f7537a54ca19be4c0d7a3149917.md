### Title
`setAaveIntegrationEnabled(false)` Reverts When Aave Has Insufficient Liquidity, Permanently Trapping Integration State and Freezing ETH - (`contracts/LRTWithdrawalManager.sol`)

---

### Summary

`setAaveIntegrationEnabled(false)` unconditionally attempts to withdraw all aWETH principal from Aave before setting the flag. If the Aave v3 WETH pool has insufficient underlying liquidity (high utilization), the `withdrawETH` call reverts, making it impossible to disable the integration. No alternative recovery path exists in the contract.

---

### Finding Description

In `setAaveIntegrationEnabled(false)`, the disable path is: [1](#0-0) 

1. It reads `aaveBalance = aaveAWETH.balanceOf(address(this))`.
2. If `aaveBalance > 0`, it calls `_collectInterestToTreasury()` (which itself calls `aaveWETHGateway.withdrawETH` for any accrued interest — also reverts under zero liquidity).
3. Then calls `_withdrawFromAave(aaveBalance)`.

Inside `_withdrawFromAave`: [2](#0-1) 

`aaveWETHGateway.withdrawETH` is a direct call to Aave's pool. If the pool's available WETH liquidity is less than `withdrawnAmount` (e.g., utilization is at or near 100%), Aave reverts the transaction. Because `isAaveIntegrationEnabled = enabled` is only written **after** the withdrawal block: [3](#0-2) 

...the flag is never updated. The integration remains stuck at `isAaveIntegrationEnabled == true`.

**No recovery path exists**:
- `emergencyWithdrawFromAave` also calls `_withdrawFromAave` and hits the same revert: [4](#0-3) 
- It also guards on `isAaveIntegrationEnabled == true` (line 552), so it cannot be used once the flag is stuck.
- There is no `forceDisableAaveIntegration` or admin override that skips the withdrawal.

The `getAaveWithdrawableLiquidity()` view function confirms the liquidity check is observable but is never used as a guard before attempting withdrawal: [5](#0-4) 

---

### Impact Explanation

All ETH deposited to Aave is temporarily frozen. The manager cannot disable the integration, cannot redirect new ETH away from Aave, and `completeWithdrawal` for ETH users will also fail if Aave has no liquidity to service individual withdrawals: [6](#0-5) 

The freeze persists until Aave's WETH utilization drops enough for the withdrawal to succeed. **Impact: Medium — Temporary freezing of funds.**

---

### Likelihood Explanation

Aave v3 WETH at 100% utilization is an extreme but realistic market condition (e.g., during a liquidity crunch, a bank-run on WETH borrowing, or a coordinated large borrow). It does not require any compromise of the LRT-rsETH protocol itself — only an external market state. The likelihood is **low** but non-zero, and the scenario is locally testable via a mainnet fork by draining the Aave WETH reserve.

---

### Recommendation

Decouple the flag toggle from the mandatory withdrawal. Allow the manager to set `isAaveIntegrationEnabled = false` independently, and separately attempt (or retry) the withdrawal. For example:

```solidity
if (!enabled) {
    // Best-effort withdrawal; do NOT revert if Aave has no liquidity
    uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
    if (aaveBalance > 0) {
        try this.withdrawFromAaveExternal(aaveBalance) {} catch {}
    }
    _revokeApprovalToAaveWETHGateway();
}
isAaveIntegrationEnabled = enabled; // always set the flag
```

Alternatively, add a separate `forceDisableAaveIntegration()` function (restricted to `PAUSER_ROLE` or admin) that sets the flag without requiring a successful withdrawal, leaving the aWETH balance to be recovered later via a dedicated retry function.

---
