### Title
Script-Hash Stake Credentials Bypass DRep Delegation Requirement for Withdrawals - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs`)

### Summary

`validateWithdrawalsDelegated` in the Conway LEDGER rule uses `credKeyHash` to extract withdrawal credentials, which silently returns `Nothing` for `ScriptHashObj` credentials. As a result, script-hash stake credentials are never checked for DRep delegation, allowing them to withdraw rewards without satisfying the Conway governance participation requirement.

### Finding Description

In Conway era, all stake credentials that perform reward withdrawals must be delegated to a DRep. This is enforced by `validateWithdrawalsDelegated`:

```haskell
validateWithdrawalsDelegated accounts tx =
  let wdrls = unWithdrawals $ tx ^. bodyTxL . withdrawalsTxBodyL
      wdrlsKeyHashes =
        [ kh | (ra, _) <- Map.toList wdrls, Just kh <- [credKeyHash $ ra ^. accountAddressCredentialL]
        ]
      isNotDRepDelegated keyHash = isNothing $ do
        accountState <- lookupAccountState (KeyHashObj keyHash) accounts
        accountState ^. dRepDelegationAccountStateL
      nonExistentDelegations =
        filter isNotDRepDelegated wdrlsKeyHashes
   in failureOnNonEmpty nonExistentDelegations ConwayWdrlNotDelegatedToDRep
``` [1](#0-0) 

The list comprehension on line 481 uses `credKeyHash`, which is defined as:

```haskell
credKeyHash :: Credential r -> Maybe (KeyHash r)
credKeyHash = \case
  KeyHashObj hk -> Just hk
  ScriptHashObj _ -> Nothing   -- silently drops script credentials
``` [2](#0-1) 

Because `credKeyHash` returns `Nothing` for every `ScriptHashObj`, the list comprehension `[ kh | ..., Just kh <- [credKeyHash ...] ]` silently discards all script-hash withdrawal credentials. `wdrlsKeyHashes` therefore contains only key hashes, and the subsequent DRep delegation check is never applied to script-hash credentials.

The `ConwayWdrlNotDelegatedToDRep` failure type itself encodes this narrowing — it wraps `NonEmpty (KeyHash Staking)`, not `NonEmpty (Credential Staking)`: [3](#0-2) 

The existing test named `"Withdraw from delegated and non-delegated staking script"` only exercises the delegated path (it calls `delegateToDRep` before withdrawing); it never asserts that a non-delegated script credential is rejected: [4](#0-3) 

The same structural narrowing is present in the Dijkstra era's `ENTITIES` and `SUBENTITIES` rules, which inherit the `WdrlNotDelegatedToDRep (NonEmpty (KeyHash Staking))` failure type. [5](#0-4) 

### Impact Explanation

**Medium.** An attacker who controls a script-hash stake credential (e.g., a Plutus `alwaysSucceeds` script) can register it, accumulate rewards, and submit a withdrawal transaction without ever delegating to a DRep. The withdrawal passes `validateWithdrawalsDelegated` because the credential is silently excluded from the check. This modifies reward withdrawals outside the design parameters of Conway governance (which mandates DRep delegation as a precondition for withdrawals), satisfying the Medium impact criterion: *attacker-controlled transactions modify withdrawals outside design parameters*.

### Likelihood Explanation

**Medium.** Any user can deploy a script-hash stake credential. The `Credential` type explicitly supports `ScriptHashObj` for staking, and script-hash stake credentials are a documented, first-class feature. No privileged access, key compromise, or consensus majority is required. The attacker only needs to: (1) create a satisfiable script, (2) register the script-hash stake credential, (3) accumulate rewards, and (4) submit a withdrawal without delegating to a DRep.

### Recommendation

Replace the key-hash-only extraction with a check over all credential types. The DRep delegation lookup should handle both `KeyHashObj` and `ScriptHashObj`:

```haskell
validateWithdrawalsDelegated accounts tx =
  let wdrls = unWithdrawals $ tx ^. bodyTxL . withdrawalsTxBodyL
      wdrlsCreds =
        [ cred | (ra, _) <- Map.toList wdrls
               , let cred = ra ^. accountAddressCredentialL ]
      isNotDRepDelegated cred = isNothing $ do
        accountState <- lookupAccountState cred accounts
        accountState ^. dRepDelegationAccountStateL
      nonExistentDelegations =
        filter isNotDRepDelegated wdrlsCreds
   in failureOnNonEmpty nonExistentDelegations ConwayWdrlNotDelegatedToDRep
```

The `ConwayWdrlNotDelegatedToDRep` failure constructor and `WdrlNotDelegatedToDRep` in Dijkstra should be updated to carry `NonEmpty (Credential Staking)` instead of `NonEmpty (KeyHash Staking)` to report the full credential.

### Proof of Concept

1. Deploy a Plutus script that always succeeds (e.g., `alwaysSucceedsNoDatum SPlutusV3`).
2. Derive `cred = ScriptHashObj scriptHash`.
3. Register `cred` as a stake credential — do **not** call `delegateToDRep`.
4. Accumulate rewards via `submitAndExpireProposalToMakeReward cred`.
5. Submit a withdrawal transaction: `Withdrawals [(ra, balance)]`.
6. Observe that the transaction is accepted despite `cred` having no DRep delegation, because `validateWithdrawalsDelegated` never evaluates `isNotDRepDelegated` for `ScriptHashObj` credentials. [6](#0-5)
