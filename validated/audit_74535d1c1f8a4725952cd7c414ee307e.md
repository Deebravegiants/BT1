### Title
Excess Deposit Silently Consumed in `submit_participant_info` Re-Submission Path — (`File: crates/contract/src/lib.rs`)

### Summary

`submit_participant_info` is marked `#[payable]` and accepts arbitrary NEAR deposits, but contains a conditional refund path that is skipped when an existing participant re-submits their attestation. Any deposit attached in that branch is permanently locked in the contract with no refund issued.

### Finding Description

The function `submit_participant_info` computes a boolean gate to decide whether to charge and refund the caller:

```rust
let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... cost check and refund logic
}
``` [1](#0-0) 

When `attestation_storage_must_be_paid_by_caller` evaluates to `false` — i.e., the caller is an existing participant (`caller_is_not_participant = false`) AND the insertion is an update rather than a first-time entry (`is_new_attestation = false`) — the entire deposit-handling block is bypassed. Because the function is `#[payable]`, any NEAR tokens attached to the call are accepted by the runtime but never returned to the caller.

The refund logic inside the `if` block is the only place in the function where `env::attached_deposit()` is read and excess is returned:

```rust
if let Some(diff) = attached.checked_sub(cost)
    && diff > NearToken::from_yoctonear(0)
{
    Promise::new(account_id).transfer(diff).detach();
}
``` [2](#0-1) 

Outside this block, there is no fallback refund. The function signature is `#[payable]` with no unconditional refund guard analogous to `require_deposit()` used by `sign` and `request_app_private_key`: [3](#0-2) 

### Impact Explanation

Any NEAR tokens attached to a re-submission call by an existing participant are permanently locked in the MPC contract. The contract's own documentation and the `require_deposit` helper establish the invariant that excess deposits must be refunded; this path silently violates it. Tokens are not recoverable without a contract upgrade.

This maps to the **Medium** allowed impact: *Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants.*

### Likelihood Explanation

The MPC node software currently attaches `0` deposit for re-submissions (confirmed by the operator guide: *"the node attaches 0, so call `submit_participant_info` manually with `--deposit` once"*). [4](#0-3) 

However, the function is a public, `#[payable]` contract method. A participant operator who manually re-submits (e.g., to force an attestation refresh before expiry) and attaches any deposit — following the same pattern as first-time submission — will silently lose those tokens. The CHANGELOG entry *"Contract was refunding too much of deposit (#1165)"* confirms the team has previously had to correct deposit-refund accounting in this contract, indicating the surface is actively exercised. [5](#0-4) 

### Recommendation

Add an unconditional excess-refund at the end of `submit_participant_info`, or move the deposit read and refund outside the `if attestation_storage_must_be_paid_by_caller` block so that any deposit attached on the re-submission path is always returned:

```rust
// After the conditional storage-charge block:
let attached = env::attached_deposit();
let cost = if attestation_storage_must_be_paid_by_caller { computed_cost } else { NearToken::from_yoctonear(0) };
if let Some(diff) = attached.checked_sub(cost) && diff > NearToken::from_yoctonear(0) {
    Promise::new(account_id).transfer(diff).detach();
}
```

Alternatively, apply the same `require_deposit(0, &account_id)` pattern used by `sign` so that any excess is always refunded regardless of the storage-charge branch taken.

### Proof of Concept

1. Deploy the MPC contract with an existing participant set.
2. Have participant `alice.near` call `submit_participant_info` a second time (re-submission) with `attached_deposit = 1_000_000_000_000_000_000_000_000` yoctoNEAR (1 NEAR).
3. The call succeeds: `is_new_attestation = false`, `caller_is_not_participant = false`, so `attestation_storage_must_be_paid_by_caller = false`.
4. The `if` block is skipped entirely — no refund is issued.
5. Alice's balance decreases by 1 NEAR; the contract balance increases by 1 NEAR with no corresponding state entry or refund promise. [6](#0-5)
