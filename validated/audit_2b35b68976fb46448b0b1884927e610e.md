### Title
`withinUnstakeLimits` Modifier Fails to Consume `availableActiveETHWithdrawals`, Allowing Repeated Unstaking Beyond Intended Limit — (`contracts/LRTConverter.sol`)

### Summary

The `withinUnstakeLimits` modifier in `LRTConverter` only decrements `whitelistedUnstakeAllowance` but never decrements the `availableActiveETHWithdrawals` component of the limit. Because `availableActiveETHWithdrawals` is a live read from `LRTWithdrawalManager.assetsCommitted(ETH_TOKEN)` — which is not modified by `unstakeStEth` — the operator can call `unstakeStEth` repeatedly up to `availableActiveETHWithdrawals` per call, bypassing the intended cumulative cap.

### Finding Description

The modifier reads the limit as:

```
amountToUnstake <= whitelistedUnstakeAllowance + availableActiveETHWithdrawals
```

It then only decrements `whitelistedUnstakeAllowance`: [1](#0-0) 

The `availableActiveETHWithdrawals` value comes from: [2](#0-1) 

This reads `lrtWithdrawalManager.assetsCommitted(LRTConstants.ETH_TOKEN)`. That mapping is only modified by `initiateWithdrawal` (increments) and `_unlockWithdrawalRequests` (decrements) in `LRTWithdrawalManager`: [3](#0-2) [4](#0-3) 

`unstakeStEth` → `_unstakeStEth` only interacts with Lido's withdrawal queue and does not touch `assetsCommitted`: [5](#0-4) 

**Concrete scenario:**

| State | Value |
|---|---|
| `whitelistedUnstakeAllowance` | 0 |
| `assetsCommitted[ETH_TOKEN]` | X |
| `LRTConverter` stETH balance | N·X |

- **Call 1:** `unstakeStEth(X)` — check passes (`X ≤ 0 + X`), `whitelistedUnstakeAllowance` branch skipped (it's 0), `_unstakeStEth(X)` executes, `assetsCommitted` unchanged.
- **Call 2:** `unstakeStEth(X)` — identical state, identical result. Passes again.
- **Call N:** Same. Repeats until the contract's stETH balance is exhausted.

The modifier is supposed to enforce a cumulative cap, but it only enforces a per-call cap for the `availableActiveETHWithdrawals` portion.

### Impact Explanation

The rate-limiting mechanism does not work as intended. An operator can unstake far more stETH than the sum of declared withdrawal intents plus pending user ETH withdrawal requests. The stETH is sent to Lido's withdrawal queue and will eventually return as ETH, so no funds are permanently lost. However, the protocol can be drained of stETH beyond the intended operational limit, leaving the deposit pool with less stETH than expected and potentially disrupting other operations that depend on stETH availability.

**Impact: Low — Contract fails to deliver promised returns, but doesn't lose value.**

### Likelihood Explanation

Requires only an operator calling `unstakeStEth` multiple times. No external conditions, no front-running, no governance capture. The operator role is trusted but the invariant violation is mechanical and unconditional whenever `whitelistedUnstakeAllowance = 0` and `assetsCommitted[ETH_TOKEN] > 0`.

### Recommendation

Track consumed `availableActiveETHWithdrawals` within the modifier. Introduce a storage variable (e.g., `activeETHWithdrawalsConsumed`) that is incremented by `amountToUnstake - whitelistedAmountConsumed` on each call, and decremented when `assetsCommitted` decreases (or reset when withdrawals are unlocked). Alternatively, compute the remaining allowance as `max(0, assetsCommitted[ETH_TOKEN] - activeETHWithdrawalsConsumed)` and decrement `activeETHWithdrawalsConsumed` accordingly.

### Proof of Concept

```solidity
// Setup:
// - whitelistedUnstakeAllowance = 0
// - LRTWithdrawalManager.assetsCommitted[ETH_TOKEN] = 100 ether
// - LRTConverter holds 200 ether worth of stETH

// Call 1 (operator):
lrtConverter.unstakeStEth(100 ether);
// Passes: 100 <= 0 + 100, whitelistedUnstakeAllowance branch skipped
// assetsCommitted[ETH_TOKEN] still = 100 ether

// Call 2 (operator):
lrtConverter.unstakeStEth(100 ether);
// Also passes: 100 <= 0 + 100 (unchanged), no revert
// 200 ether of stETH has been sent to Lido, but only 100 ether of user withdrawals exist
```

The second call should revert under the intended invariant but does not. The `availableActiveETHWithdrawals` component of the limit is never consumed.
