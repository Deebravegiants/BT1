### Title
Unpaid Storage for Pending Sign/CKD/ForeignTx Requests Allows Contract Balance Drain - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `sign`, `request_app_private_key`, and `verify_foreign_transaction` endpoints each create a new entry in their respective `pending_*_requests` `LookupMap` without charging the caller for the on-chain storage cost. Only 1 yoctoNEAR is required per call, while the actual NEAR storage staking cost per new map entry is orders of magnitude higher. An unprivileged attacker can repeatedly submit requests with unique keys to drain the contract's storage balance, eventually preventing the contract from writing new state.

---

### Finding Description

All three user-facing request methods share the same deposit enforcement path:

```rust
// crates/contract/src/lib.rs
const MINIMUM_SIGN_REQUEST_DEPOSIT: NearToken = NearToken::from_yoctonear(1);
const MINIMUM_CKD_REQUEST_DEPOSIT:  NearToken = NearToken::from_yoctonear(1);
```

`check_request_preconditions` enforces only this 1-yoctoNEAR floor and refunds any excess:

```rust
fn check_request_preconditions(..., minimum_deposit: NearToken) -> ... {
    ...
    require_deposit(minimum_deposit, &predecessor);
    ...
}
```

After passing preconditions, `sign` calls `enqueue_yield_request` → `add_signature_request`, which ultimately calls `push_pending_yield`:

```rust
// crates/contract/src/pending_requests.rs
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) {
    let queue = requests.entry(request).or_default(); // ← new storage entry created here
    ...
    queue.push(YieldIndex { data_id });
}
```

`requests.entry(request).or_default()` inserts a new key into the `LookupMap` when the request key is novel. In NEAR, `LookupMap` writes directly to contract storage, and the storage staking cost is deducted from the **contract's own balance**, not the caller's. No storage cost is measured or charged to the caller at any point in the call chain.

A `SignatureRequest` key is composed of `(domain_id, payload, predecessor, path)`. An attacker can trivially generate an unbounded number of unique keys by varying the 32-byte ECDSA payload or the derivation path string. Each unique key creates a new storage entry at the contract's expense.

The entries are eventually freed (on MPC response via `resolve_yields_for` or on yield timeout via `pop_oldest_pending_yield`), but during the window between submission and cleanup the contract's balance is reduced. Because NEAR yield timeouts span hundreds of blocks, an attacker submitting requests at a sustained rate maintains a large pool of live entries, creating a continuous drain.

The same pattern applies to `request_app_private_key` (pending CKD requests) and `verify_foreign_transaction` (pending foreign-tx requests), all of which use the same 1-yoctoNEAR minimum and the same `push_pending_yield` helper.

---

### Impact Explanation

**Medium.** The contract's storage balance is depleted at a rate proportional to the attacker's submission rate. NEAR storage costs ~10 yoctoNEAR per byte; a `SignatureRequest` entry (domain id + 32-byte payload + account id + path + `Vec<YieldIndex>`) is on the order of 100–300 bytes, so each new entry costs the contract ~1,000–3,000 yoctoNEAR while the attacker pays only 1 yoctoNEAR. The amplification factor is ~1,000–3,000×.

If the contract's balance is driven below the storage staking floor, the NEAR runtime will reject any further state writes. This breaks the request-lifecycle and contract execution-flow invariants: new sign requests, new participant attestations, new domain configs, and governance votes would all fail to persist, effectively halting the MPC network's ability to accept new work without any threshold-level collusion.

This matches the allowed Medium impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."*

---

### Likelihood Explanation

**Medium.** The attack requires no special privilege — any NEAR account can call `sign`. The cost to the attacker is 1 yoctoNEAR per request plus transaction fees (~0.0001 NEAR). Submitting a few hundred requests per second is feasible from a single account. The contract's balance is large (production deployment), so a sustained attack over hours to days would be required to exhaust it, but the economic asymmetry makes it profitable for a motivated adversary who benefits from halting the MPC network (e.g., to prevent a competing cross-chain transaction from being signed).

---

### Recommendation

Charge callers for the storage their request entry will consume, analogous to how `submit_participant_info` already does it:

1. Snapshot `env::storage_usage()` before inserting the pending request.
2. After insertion, compute `delta = env::storage_usage() - initial` and `cost = env::storage_byte_cost() * delta`.
3. Require `env::attached_deposit() >= MINIMUM_SIGN_REQUEST_DEPOSIT + cost`; refund any excess.
4. When the entry is cleaned up (on response or timeout), refund the storage deposit to the original caller.

Alternatively, set `MINIMUM_SIGN_REQUEST_DEPOSIT` to a conservatively high value that covers the worst-case storage cost of a pending entry (e.g., 300–500 bytes × storage byte cost), and document this as the storage sponsorship fee.

---

### Proof of Concept

```
1. Attacker calls `sign` 10,000 times with distinct payloads
   (e.g., payload = sha256(i) for i in 0..10000), each with 1 yoctoNEAR deposit.

2. Each call passes check_request_preconditions (deposit ≥ 1 yoctoNEAR ✓).

3. Each call reaches push_pending_yield, which executes:
       requests.entry(unique_request_key).or_default()
   creating 10,000 new LookupMap entries in contract storage.

4. Storage cost per entry ≈ 200 bytes × 10 yoctoNEAR/byte = 2,000 yoctoNEAR,
   borne entirely by the contract.

5. Total drain: 10,000 × 2,000 = 20,000,000 yoctoNEAR = 0.02 NEAR
   Attacker cost: 10,000 × 1 yoctoNEAR + tx fees ≈ negligible.

6. Entries persist until NEAR yield timeout (~200 blocks ≈ 200 s).
   Attacker re-submits at the same rate to maintain the drain indefinitely.

7. Once the contract's balance approaches the storage staking minimum,
   all subsequent state writes (new requests, governance votes, attestations)
   are rejected by the NEAR runtime, halting the MPC network.
```

fn require_deposit(minimum_deposit: NearToken, predecessor: &AccountId) {
    let deposit = env::attached_deposit();
    match deposit.checked_sub(minimum_deposit) {
        None => {
            env::panic_str(
                &InvalidParameters::InsufficientDeposit {
                    attached: deposit.as_yoctonear(),
                    required: minimum_deposit.as_yoctonear(),
                }
                .to_string(),
            );
        }
        Some(diff) => {
            if diff > NearToken::from_yoctonear(0) {
                log!("refund excess deposit {diff} to {predecessor}");
                Promise::new(predecessor.clone()).transfer(diff).detach();
            }
        }
    }
}

pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
}
