### Title
Incomplete Condition Check in `isWithdrawalClaimable()` Causes Misleading Claimability Signal - (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary
`KernelDepositPool.isWithdrawalClaimable()` only checks the time-lock condition, while `claimWithdrawal()` enforces three additional guards. Any caller — including off-chain bots or integrating contracts — that uses `isWithdrawalClaimable()` as a pre-flight check before calling `claimWithdrawal()` will receive a misleading `true` result and subsequently have the on-chain call revert.

### Finding Description
`isWithdrawalClaimable()` is defined as:

```solidity
function isWithdrawalClaimable(uint256 _withdrawalId) external view returns (bool) {
    return block.timestamp >= withdrawals[_withdrawalId].unlockTime;
}
``` [1](#0-0) 

`claimWithdrawal()` enforces four distinct guards before transferring tokens:

```solidity
if (withdrawal.user == address(0)) revert WithdrawalDoesNotExist();
if (withdrawal.user != msg.sender)  revert NotYourWithdrawal();
if (block.timestamp < withdrawal.unlockTime) revert WithdrawalNotReady();
if (withdrawal.claimed) revert WithdrawalAlreadyClaimed();
``` [2](#0-1) 

`isWithdrawalClaimable()` only mirrors the third guard. The other three are absent, producing two concrete false-positive scenarios:

1. **Non-existent withdrawal ID**: `withdrawals[nonExistentId].unlockTime` defaults to `0`. Because `block.timestamp >= 0` is always `true`, `isWithdrawalClaimable()` returns `true` for every ID that was never created. `claimWithdrawal()` then reverts with `WithdrawalDoesNotExist`.

2. **Already-claimed withdrawal**: After `claimWithdrawal()` sets `withdrawal.claimed = true` and removes the ID from `userWithdrawalIds`, the `Withdrawal` struct remains in storage with its original `unlockTime`. A direct call to `isWithdrawalClaimable(id)` still returns `true`, but `claimWithdrawal(id)` reverts with `WithdrawalAlreadyClaimed`. [3](#0-2) 

### Impact Explanation
Any off-chain keeper, integrating contract, or user-facing dApp that calls `isWithdrawalClaimable()` to decide whether to submit `claimWithdrawal()` will waste gas on a guaranteed revert. The function's name and return value constitute a broken contract promise: it signals claimability when the action will fail. No funds are lost, but the protocol fails to deliver the promised view-function guarantee.

**Impact: Low — Contract fails to deliver promised returns, but doesn't lose value.**

### Likelihood Explanation
The function is `external` and named to serve as a pre-flight check. Any integrator, bot, or front-end following the natural pattern of "check then act" will hit this. The non-existent-ID case is trivially reachable by any caller passing an arbitrary `uint256`.

### Recommendation
Align `isWithdrawalClaimable()` with all conditions enforced by `claimWithdrawal()`:

```solidity
function isWithdrawalClaimable(uint256 _withdrawalId) external view returns (bool) {
    Withdrawal storage w = withdrawals[_withdrawalId];
    return w.user != address(0)
        && !w.claimed
        && block.timestamp >= w.unlockTime;
}
```

This mirrors the exact guard set in `claimWithdrawal()` and ensures the view function is a reliable pre-flight check.

### Proof of Concept
1. Alice calls `initiateWithdrawal(100e18)`, receiving `withdrawalId = 5` with `unlockTime = block.timestamp + withdrawalDelay`.
2. After the delay, Alice calls `claimWithdrawal(5)` — succeeds; `withdrawal.claimed` is set to `true`.
3. Bob (or any caller) calls `isWithdrawalClaimable(5)` — returns `true` because `block.timestamp >= unlockTime` is still satisfied.
4. Bob calls `claimWithdrawal(5)` — reverts with `WithdrawalAlreadyClaimed`.

For the non-existent ID case: any caller passes `_withdrawalId = 9999` (never created). `withdrawals[9999].unlockTime == 0`, so `isWithdrawalClaimable(9999)` returns `true`. `claimWithdrawal(9999)` reverts with `WithdrawalDoesNotExist`. [1](#0-0) [4](#0-3)
