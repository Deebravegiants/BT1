Audit Report

## Title
Missing Token Decimal Normalization in `viewSwapRsETHAmountAndFee` Causes Severely Incorrect rsETH Minting for Non-18-Decimal Tokens - (`contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`, `contracts/pools/RSETHPoolV3WithNativeChainBridge.sol`, `contracts/pools/RSETHPool.sol`, `contracts/pools/RSETHPoolNoWrapper.sol`)

## Summary

The `viewSwapRsETHAmountAndFee(uint256 amount, address token)` function in all five pool contracts computes rsETH output as `amountAfterFee * tokenToETHRate / rsETHToETHrate`. The oracle returns `tokenToETHRate` as the price of **one full token** in ETH at 1e18 precision, but `amountAfterFee` is the raw token amount in the token's **smallest unit**. For any token with fewer than 18 decimals, the formula underestimates the rsETH output by a factor of `10^(18 - tokenDecimals)`. A user depositing 1,000 units of a 6-decimal token would receive approximately one trillion times fewer rsETH than the fair value of their deposit, while the pool retains the full token amount.

## Finding Description

The vulnerable formula appears identically in all five contracts:

```solidity
// RSETHPoolV3.sol L324-334, RSETHPoolV3ExternalBridge.sol L442-451,
// RSETHPoolV3WithNativeChainBridge.sol L360-370, RSETHPool.sol L336-346,
// RSETHPoolNoWrapper.sol L301-311
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;
uint256 rsETHToETHrate = getRate();                                    // 1e18-precision
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate(); // 1e18-precision
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;        // ← decimal bug
```

The oracle wrapper `ChainlinkOracleForRSETHPoolCollateral.getRate()` normalises the Chainlink answer to 1e18 precision:

```solidity
// ChainlinkOracleForRSETHPoolCollateral.sol L34
uint256 normalizedPrice = uint256(ethPrice) * 1e18
    / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
```

This `normalizedPrice` represents the ETH value of **one full token** (e.g., 1.0 USDC), not one smallest unit. The formula is dimensionally correct only when `tokenDecimals == 18`, because then `amountAfterFee` in smallest units equals `amountAfterFee / 1e18` full tokens, and the 1e18 factors cancel. For a 6-decimal token, `amountAfterFee` is in units of 1e-6 full tokens, but the formula treats it as if it were in units of 1e-18 full tokens, producing a result `10^12` times too small.

**Exploit path:**
1. Admin legitimately calls `addSupportedToken(token, oracle, bridge)` with a non-18-decimal token (e.g., WBTC at 8 decimals, or any 6-decimal stablecoin-equivalent LST). No decimal restriction exists in `addSupportedToken`.
2. Any unprivileged user calls `deposit(token, amount, referralId)`.
3. The pool transfers the full `amount` of tokens from the user via `safeTransferFrom`.
4. `viewSwapRsETHAmountAndFee` returns an rsETH amount `10^(18-tokenDecimals)` times too small.
5. The pool mints/transfers that negligible rsETH amount to the user.
6. The user's tokens are locked in the pool; their rsETH balance is effectively zero relative to the ETH value deposited.

No existing guard prevents this: `addSupportedToken` only checks that the oracle returns a non-zero rate; it does not validate token decimals. The `deposit` function has no minimum rsETH output check. The `limitDailyMint` modifier also uses the same buggy `viewSwapRsETHAmountAndFee`, so the daily limit tracking is also incorrect for such tokens.

The symmetric bug exists in `viewSwapAssetToPremintedRsETH` (`tokenAmount = rsETHAmount * rsETHToETHrate / tokenToETHRate`), which would return `10^12` times too many tokens for a 6-decimal token, but that function is gated behind `OPERATOR_ROLE`.

## Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

When a non-18-decimal token is supported, every depositor of that token receives a negligible rsETH amount relative to the ETH value they contributed. The contract accepts the full token transfer but mints almost nothing in return. The depositor's funds are locked in the pool while their rsETH balance is effectively zero. The pool itself does not lose value (it holds the tokens), but the user's promised exchange rate is not delivered.

## Likelihood Explanation

**Low.** The bug is latent and only activates when an admin adds a non-18-decimal token via `addSupportedToken`. Currently deployed pools support only wstETH (18 decimals), so no user is currently affected. However, the contracts expose a generic multi-token mechanism with no decimal restriction, and protocol expansion to accept WBTC (8 decimals) or other non-18-decimal LSTs is a plausible future action. No malicious intent is required; the miscalculation is invisible at the Solidity level and would not revert. Once such a token is added, any depositor of that token is immediately and silently harmed.

## Recommendation

Retrieve the token's decimal count and normalise `amountAfterFee` to 18-decimal precision before dividing by the rsETH rate:

```solidity
uint8 tokenDecimals = IERC20Metadata(token).decimals();
rsETHAmount = amountAfterFee * tokenToETHRate * 1e18
              / rsETHToETHrate / (10 ** uint256(tokenDecimals));
```

For 18-decimal tokens the `* 1e18 / 1e18` factor cancels and existing behaviour is unchanged. Apply the symmetric correction to `viewSwapAssetToPremintedRsETH` for non-ETH tokens:

```solidity
tokenAmount = rsETHAmount * rsETHToETHrate * (10 ** uint256(tokenDecimals))
              / tokenToETHRate / 1e18;
```

## Proof of Concept

**Foundry fork test outline (Base mainnet, `RSETHPoolV3ExternalBridge`):**

```solidity
// 1. Fork Base mainnet
// 2. Deploy a mock 6-decimal ERC20 token and a mock oracle returning 3e14 (0.0003 ETH/token)
// 3. Call addSupportedToken(mockToken, mockOracle, mockBridge) as admin
// 4. Mint 1000e6 mockToken to alice; alice approves pool
// 5. alice calls deposit(mockToken, 1000e6, "")
// 6. Assert: alice's wrsETH balance ≈ 2.857e5 (current, wrong)
//            expected ≈ 2.857e17 (correct, ~0.2857 rsETH)
// 7. Error factor: 1e12 — confirmed by dimensional analysis

// Concrete numbers:
// amountAfterFee = 1000e6 (assuming feeBps=0 for clarity)
// tokenToETHRate = 3e14
// rsETHToETHrate = 1.05e18
// current:  1000e6 * 3e14 / 1.05e18 = 285714 (≈ 2.857e5 wei of rsETH)
// correct:  1000e6 * 3e14 * 1e18 / 1.05e18 / 1e6 = 285714285714285714 (≈ 2.857e17 wei of rsETH)
```
