Audit Report

## Title
ETH Deposit Limit Guard Omits Incoming Amount, Allowing Deposits to Exceed the Cap — (File: `contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric check: for ERC-20 assets it correctly evaluates `totalAssetDeposits + amount > limit`, but for ETH it evaluates only `totalAssetDeposits > limit`, silently discarding the `amount` argument. As a result, any depositor can push the protocol's ETH holdings arbitrarily beyond `depositLimitByAsset[ETH_TOKEN]` in a single transaction, because the guard never accounts for the incoming deposit when the asset is ETH.

## Finding Description
The root cause is in `_checkIfDepositAmountExceedesCurrentLimit` at [1](#0-0) :

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct
}
```

The ETH branch (line 679) never adds `amount` to `totalAssetDeposits`. While the protocol is operating normally (i.e., `totalAssetDeposits < depositLimitByAsset`), this branch always returns `false` regardless of how large `amount` is, so the revert at [2](#0-1)  is never triggered for ETH.

The public entry point `depositETH` passes `msg.value` as `depositAmount` to `_beforeDeposit`, but that value is dropped inside the ETH branch: [3](#0-2) 

The deposit cap is stored in `depositLimitByAsset` in `LRTConfig`: [4](#0-3) 

The view helper `getAssetCurrentLimit` correctly computes remaining headroom as `depositLimit - totalAssetDeposits`, confirming the intended semantics: [5](#0-4) 

## Impact Explanation
**Low — Contract fails to deliver promised returns.** The `depositLimitByAsset` cap is the protocol's primary risk-management mechanism for bounding EigenLayer ETH exposure. With the guard broken, the cap is entirely unenforceable for ETH: any depositor can exceed it by an arbitrary amount in a single call. The protocol mints rsETH for every wei deposited, diluting the backing ratio for all existing holders and violating the risk-bounded return guarantee the protocol is designed to provide. No funds are directly stolen, placing this squarely in the Low impact tier.

## Likelihood Explanation
`depositETH` is a public payable function with no role restriction. [6](#0-5)  Any wallet can call it at any time. The only prerequisite is the normal operating state (`totalAssetDeposits < depositLimitByAsset`), which is the expected condition during live operation. No special privileges, front-running, or external conditions are required. Likelihood is **High**.

## Recommendation
Apply the same `+ amount` pattern used for ERC-20 assets to the ETH branch, eliminating the special case entirely:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This aligns the ETH guard with the ERC-20 guard and with the semantics of `getAssetCurrentLimit`.

## Proof of Concept
```
Setup:
  depositLimitByAsset[ETH_TOKEN] = 100 ether
  getTotalAssetDeposits(ETH_TOKEN) = 99 ether  // one wei below cap

Attack:
  attacker calls depositETH{value: 50 ether}()

Execution of _checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 50 ether):
  totalAssetDeposits = 99 ether
  ETH branch: return (99 ether > 100 ether) → false  ← 50 ether never used
  → MaximumDepositLimitReached NOT triggered

Result:
  50 ether deposited, rsETH minted
  getTotalAssetDeposits(ETH_TOKEN) = 149 ether (49 ether over cap)

Contrast (stETH, same conditions):
  return (99 ether + 50 ether > 100 ether) → true → revert ✓
```

A Foundry fuzz test parameterizing `amount` over `[1, type(uint128).max]` with `totalAssetDeposits` set to `depositLimit - 1` would confirm that the ETH branch never reverts while the ERC-20 branch always reverts for `amount > 1`.
