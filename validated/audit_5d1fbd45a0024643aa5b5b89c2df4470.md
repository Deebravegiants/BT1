### Title
User-Controlled `confirmations` Parameter in `verify_foreign_transaction` Can Be Set to Zero, Bypassing Bitcoin Finality Check - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction` public entry point accepts a caller-supplied `confirmations: BlockConfirmations` field inside `BitcoinRpcRequest`. The contract enforces no minimum value. Setting `confirmations: 0` causes every MPC node to pass the finality gate unconditionally, so the network will produce a valid threshold signature attesting to an unconfirmed (0-confirmation) Bitcoin transaction. A bridge contract that trusts the MPC signature without re-inspecting the embedded `confirmations` value can be tricked into releasing NEAR-side funds for a Bitcoin transaction that is subsequently double-spent.

---

### Finding Description

`verify_foreign_transaction` in `MpcContract` accepts a `VerifyForeignTransactionRequestArgs` whose inner `ForeignChainRpcRequest::Bitcoin` variant carries a `confirmations: BlockConfirmations` field typed as a plain `u64`. [1](#0-0) 

The ABI schema exposes this field with `"minimum": 0.0` and no `maximum`, meaning any value from 0 to `u64::MAX` is accepted on-chain without further validation. [2](#0-1) 

The `verify_foreign_transaction` handler performs no lower-bound check on `confirmations` before enqueuing the request: [3](#0-2) 

On the node side, the Bitcoin inspector evaluates:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
``` [4](#0-3) 

When `block_confirmations_threshold` is `0`, the inequality `0 <= any_u64` is always `true`, so the finality gate is unconditionally bypassed. The node proceeds to the canonical-chain check and, if that passes, returns extracted values. All nodes independently reach the same conclusion, the threshold is met, and the MPC network issues a valid signature over `(request, observed_values)` — a payload that permanently encodes `confirmations: 0`.

The `Config` struct contains no `min_bitcoin_confirmations` field, and no other code path in the contract or node enforces a floor. [5](#0-4) 

---

### Impact Explanation

The MPC network's `verify_foreign_transaction` flow is the trust anchor for inbound bridge flows (e.g., Omnibridge: foreign chain → NEAR). A bridge contract that calls `verify_foreign_transaction` and then releases NEAR-side assets upon receiving a valid MPC signature is the intended consumer. If that contract does not re-inspect the `confirmations` field embedded in the signed payload, an attacker can:

1. Broadcast a Bitcoin transaction sending funds to the bridge deposit address.
2. Immediately call `verify_foreign_transaction` with `confirmations: 0` before the transaction is mined.
3. Receive a valid MPC threshold signature attesting to the transaction.
4. Submit the signature to the bridge contract and claim NEAR-side assets.
5. Double-spend the original Bitcoin transaction (replace-by-fee or miner collusion), recovering the Bitcoin while keeping the NEAR-side assets.

This is a **light-client-style verification bypass that causes invalid bridge execution and double-spend conditions**, matching the High impact tier.

---

### Likelihood Explanation

The entry point is fully public and requires only a 1 yoctoNEAR deposit. No privilege, TEE attestation, or participant role is needed. The attacker constructs a standard `VerifyForeignTransactionRequestArgs` JSON with `"confirmations": 0`. The attack is a single transaction. Likelihood is high for any bridge that does not independently validate the `confirmations` field in the signed payload.

---

### Recommendation

1. Add a `min_bitcoin_confirmations: u64` field (and analogous fields for other chains) to the `Config` struct, settable by the contract admin via the existing config-update vote mechanism.
2. In `verify_foreign_transaction`, reject requests whose `confirmations` value falls below the configured minimum before enqueuing the yield.
3. Document the minimum in the ABI schema so SDK consumers and bridge contracts can rely on it as a protocol invariant rather than a caller-supplied hint.

---

### Proof of Concept

```json
// Call verify_foreign_transaction with confirmations: 0
{
  "request": {
    "request": {
      "Bitcoin": {
        "tx_id": "<32-byte tx hash of a 0-confirmation tx>",
        "confirmations": 0,
        "extractors": ["BlockHash"]
      }
    },
    "domain_id": 3,
    "payload_version": 1
  }
}
```

Node-side evaluation path:

```
block_confirmations_threshold = BlockConfirmations(0)
transaction_block_confirmation = BlockConfirmations(0)   // tx not yet mined
0 <= 0  →  true  →  finality gate passes
verify_block_is_canonical(...)  →  passes (tx is in mempool, no block yet)
                                   OR passes for any block once mined
→ MPC signs ForeignTxSignPayloadV1 { request: Bitcoin { confirmations: 0, ... }, values: [...] }
```

The resulting signature is indistinguishable from one produced for a 6-confirmation transaction. A bridge contract that checks only `signature.is_valid()` and not `request.confirmations >= MIN_CONFIRMATIONS` will release funds. [6](#0-5) [7](#0-6)

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }

    async fn extract(
        &self,
        transaction: BitcoinTransactionHash,
        block_confirmations_threshold: BlockConfirmations,
        extractors: Vec<BitcoinExtractor>,
    ) -> Result<Vec<BitcoinExtractedValue>, ForeignChainInspectionError> {
        let request_parameters = GetRawTransactionArgs {
            transaction_hash: TransportBitcoinTransactionHash::from(*transaction),
            verbose: VERBOSE_RESPONSE,
        };

        // TODO(#1978): add retry mechanism if the error from the request is transient
        let rpc_response: GetRawTransactionVerboseResponse = self
            .client
            .request(GET_RAW_TRANSACTION_METHOD, &request_parameters)
            .await?;

        let transaction_block_confirmation = rpc_response.confirmations.into();
        let enough_block_confirmations =
            block_confirmations_threshold <= transaction_block_confirmation;

        if !enough_block_confirmations {
            return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations {
                expected: block_confirmations_threshold,
                got: transaction_block_confirmation,
            });
        }

        self.verify_block_is_canonical(rpc_response.blockhash)
            .await?;

        let extracted_values = extractors
            .iter()
            .map(|extractor| extractor.extract_value(&rpc_response))
            .collect();

        Ok(extracted_values)
    }
