### Title
Permanent `ethValueInWithdrawal` inflation from stETH oracle/Lido ETH divergence causes temporary protocol freeze — (File: `contracts/LRTConverter.sol`)

### Summary
`transferAssetFromDepositPool` snapshots `ethValueInWithdrawal` at the oracle price P1, but `_sendEthToDepositPool` decrements it by the actual ETH received from Lido (P2). When P2 < P1 (slashing or stETH depeg during the withdrawal window), the residual `amount × (P1 − P2) / 1e18` is never cleared, permanently overstating TVL and rsETH price, and setting `highestRsethPrice` too high. A subsequent oracle price normalization then causes a price-decrease check to fire, pausing deposits and withdrawals.

### Finding Description

`LRTConverter.transferAssetFromDepositPool` increments `ethValueInWithdrawal` using the live oracle price at the moment of the call:

```solidity
// LRTConverter.sol:140
ethValueInWithdrawal += (_amount * lrtOracle.getAssetPrice(_asset)) / 1e18;
```

After `unstakeStEth` submits the Lido withdrawal request, `claimStEth` calls `_sendEthToDepositPool(address(this).balance)`. The decrement inside `_sendEthToDepositPool` uses the *actual ETH balance* of the contract — i.e., whatever Lido actually returned:

```solidity
// LRTConverter.sol:255-258
if (ethValueInWithdrawal > _amount) {
    ethValueInWithdrawal -= _amount;
} else {
    ethValueInWithdrawal = 0;
}
```

If the stETH oracle price was P1 = 1.05e18 at transfer-in time and Lido returns ETH at an effective rate of P2 = 1.0e18 (due to slashing or a stETH depeg during the multi-day withdrawal window), the accounting diverges:

| Event | `ethValueInWithdrawal` delta |
|---|---|
| `transferAssetFromDepositPool(stETH, A)` | `+A × 1.05 / 1e18` |
| `claimStEth` → `_sendEthToDepositPool(A × 1.0)` | `−A × 1.0 / 1e18` |
| **Residual** | **`+A × 0.05 / 1e18` (permanent)** |

There is no admin function that directly zeroes or corrects `ethValueInWithdrawal` without a corresponding asset transfer. The residual is permanent.

`LRTDepositPool.getETHDistributionData()` returns `ethValueInWithdrawal` verbatim as `ethLyingInConverter`, which is summed into `getTotalAssetDeposits(ETH_TOKEN)` and ultimately into `_getTotalEthInProtocol()`. The inflated total raises `newRsETHPrice` above its true value, and `highestRsethPrice` is updated to this inflated level.

When the stETH oracle price later normalizes (e.g., drops from 1.05e18 back to 1.0e18), `_getTotalEthInProtocol()` decreases because the remaining stETH holdings are now valued lower. `newRsETHPrice` falls below the artificially elevated `highestRsethPrice`. If the gap exceeds `pricePercentageLimit × highestRsethPrice`, `_updateRsETHPrice` pauses both `LRTDepositPool` and `LRTWithdrawalManager`, freezing all user deposits and withdrawals until an admin manually unpauses.

### Impact Explanation

- `ethValueInWithdrawal` is permanently overstated by `amount × (P1 − P2) / 1e18`.
- `rsETHPrice` and `highestRsethPrice` are permanently inflated by the same phantom ETH.
- On oracle price normalization, `isPriceDecreaseOffLimit` fires, pausing `LRTDepositPool` and `LRTWithdrawalManager` — temporarily freezing all user deposits and withdrawals.
- The phantom residual also slightly over-mints rsETH for subsequent depositors (they receive rsETH priced against an inflated TVL), constituting a minor dilution of existing holders.

**Scoped impact: Medium — Temporary freezing of funds.**

### Likelihood Explanation

- The operator calling `transferAssetFromDepositPool` followed by `unstakeStEth` / `claimStEth` is the **intended, documented operational flow** — no malicious actor is required.
- stETH has historically traded at both premiums and discounts to ETH. A 1–5% premium is not unusual during periods of high demand.
- Lido's withdrawal finalization can take days to weeks; slashing events or a stETH depeg during that window are realistic.
- The `pricePercentageLimit` is a configurable parameter; if set to a tight value (e.g., 1%), even a small phantom residual can trigger the pause.
- The scenario requires no front-running, no key compromise, and no governance capture.

### Recommendation

Decouple the `ethValueInWithdrawal` accounting from the oracle price. Instead of recording the oracle-priced ETH value at transfer-in time, track the **raw token amount** in the converter and price it at query time (consistent with how all other assets are priced in `_getTotalEthInProtocol`). Alternatively, when `_sendEthToDepositPool` is called, zero out the portion of `ethValueInWithdrawal` that corresponds to the completed withdrawal request rather than subtracting the raw ETH received. A dedicated admin function to correct `ethValueInWithdrawal` (e.g., `setEthValueInWithdrawal`) would also serve as a mitigation.
