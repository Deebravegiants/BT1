Audit Report

## Title
Static `queuedWithdrawalsBuffer` Disconnected from `assetsCommitted` Allows Instant Withdrawals to Drain Vault Below Queued Obligations - (File: `contracts/LRTUnstakingVault.sol`, `contracts/LRTWithdrawalManager.sol`)

## Summary

`LRTUnstakingVault.getAssetsAvailableForInstantWithdrawal` reserves vault funds for queued withdrawals using a static, operator-set `queuedWithdrawalsBuffer` that has no on-chain link to `LRTWithdrawalManager.assetsCommitted`. As `assetsCommitted` grows with each `initiateWithdrawal` call, the buffer becomes stale, allowing instant withdrawals to drain the vault below the amount needed to service pending queued requests. Queued users whose rsETH is already locked in `LRTWithdrawalManager` cannot complete their withdrawals until the operator manually replenishes the vault.

## Finding Description

**Root cause:** Two independent accounting systems govern the same vault balance with no synchronization.

`LRTWithdrawalManager` tracks pending obligations in `assetsCommitted[asset]`, incremented on every `initiateWithdrawal` call: [1](#0-0) 

`LRTUnstakingVault.getAssetsAvailableForInstantWithdrawal` reserves vault funds using a completely separate static value: [2](#0-1) 

`queuedWithdrawalsBuffer` is only ever written by a privileged operator call with no automatic update mechanism: [3](#0-2) 

`instantWithdrawal` enforces availability using only this stale buffer: [4](#0-3) 

When `unlockQueue` is later called, it reads the raw vault balance as the total available amount — with no awareness of how much was already consumed by instant withdrawals beyond the buffer: [5](#0-4) 

Inside `_unlockWithdrawalRequests`, if `availableAssetAmount < payoutAmount`, the loop breaks and remaining queued requests stay locked: [6](#0-5) 

**Exploit flow:**
1. Vault holds 1000 ETH. Operator sets `queuedWithdrawalsBuffer[ETH] = 200`.
2. Users call `initiateWithdrawal` totaling 800 ETH. `assetsCommitted[ETH] = 800`. Their rsETH is now held by `LRTWithdrawalManager`.
3. `getAssetsAvailableForInstantWithdrawal` returns `1000 - 200 = 800` (uses static buffer, ignores `assetsCommitted`).
4. Any rsETH holder calls `instantWithdrawal` for 800 ETH. Vault balance drops to 200 ETH.
5. Operator calls `unlockQueue`. `_createUnlockParams` reads `unstakingVault.balanceOf(ETH) = 200`.
6. `_unlockWithdrawalRequests` can only service 200 ETH of the 800 ETH in queued requests; the loop breaks after the first request that exceeds remaining balance.
7. The remaining queued users cannot call `completeWithdrawal` (their nonces remain above `nextLockedNonce`) and cannot recover their rsETH (it was burned at initiation). Their funds are frozen until the operator replenishes the vault via EigenLayer unstaking.

**Existing checks are insufficient:** The `initiateWithdrawal` guard (`getAvailableAssetAmount`) checks against total protocol deposits, not vault balance, so it does not prevent over-commitment of the vault specifically. There is no on-chain mechanism that forces `queuedWithdrawalsBuffer` to track `assetsCommitted`.

## Impact Explanation

Queued withdrawal users have already transferred and burned their rsETH. If the vault is drained by instant withdrawals beyond the stale buffer, those users cannot complete withdrawals and have no on-chain recourse until the operator acts. This is a concrete **temporary freezing of funds** — a Medium-severity impact within the allowed scope.

## Likelihood Explanation

Both preconditions are reachable by unprivileged actors with no admin compromise required:
1. `assetsCommitted` exceeding `queuedWithdrawalsBuffer` occurs naturally as users queue withdrawals without the operator updating the buffer — a routine operational gap.
2. Any rsETH holder can call `instantWithdrawal` to drain the vault down to `vaultBalance - queuedWithdrawalsBuffer`.

No special privileges, oracle manipulation, or governance capture is needed. The attack is repeatable whenever the buffer falls behind `assetsCommitted`.

## Recommendation

Replace the static `queuedWithdrawalsBuffer` with a dynamic query against `assetsCommitted` in `getAssetsAvailableForInstantWithdrawal`:

```solidity
function getAssetsAvailableForInstantWithdrawal(address asset) external view returns (uint256) {
    uint256 vaultBalance = balanceOf(asset);
    ILRTWithdrawalManager wm = ILRTWithdrawalManager(lrtConfig.withdrawManager());
    uint256 committed = wm.assetsCommitted(asset);
    return committed >= vaultBalance ? 0 : vaultBalance - committed;
}
```

This ensures the reserved amount always reflects actual on-chain pending obligations, eliminating the off-chain operational dependency.

## Proof of Concept

**Foundry fork test outline:**

```solidity
function test_instantWithdrawalDrainsQueuedFunds() public {
    // 1. Fund vault with 1000 ETH
    vm.deal(address(unstakingVault), 1000 ether);

    // 2. Operator sets static buffer to 200 ETH
    vm.prank(operator);
    unstakingVault.setQueuedWithdrawalsBuffer(ETH, 200 ether);

    // 3. Multiple users initiate queued withdrawals totaling 800 ETH of rsETH
    //    assetsCommitted[ETH] becomes 800 ETH
    for (uint i = 0; i < 8; i++) {
        vm.prank(queuedUser[i]);
        withdrawalManager.initiateWithdrawal(ETH, 100e18_rsETH, "");
    }

    // 4. Assert getAssetsAvailableForInstantWithdrawal returns 800 (not 200)
    assertEq(unstakingVault.getAssetsAvailableForInstantWithdrawal(ETH), 800 ether);

    // 5. Attacker drains 800 ETH via instantWithdrawal
    vm.prank(attacker);
    withdrawalManager.instantWithdrawal(ETH, attacker_rsETH_800, "");

    // 6. Vault now holds 200 ETH; assetsCommitted still 800 ETH
    assertEq(unstakingVault.balanceOf(ETH), 200 ether);

    // 7. Operator calls unlockQueue — only 200 ETH available, 600 ETH of requests blocked
    vm.prank(operator);
    (uint256 burned, uint256 unlocked) = withdrawalManager.unlockQueue(ETH, type(uint256).max, ...);
    assertLt(unlocked, 800 ether); // Only partial unlock

    // 8. Remaining queued users cannot completeWithdrawal — nonces still locked
    vm.prank(queuedUser[7]);
    vm.expectRevert(WithdrawalLocked.selector);
    withdrawalManager.completeWithdrawal(ETH, "");
}
```
