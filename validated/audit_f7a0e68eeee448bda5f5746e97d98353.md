### Title
`is_proposal_init_valid` enforces only the future half of `block_timestamp_window_seconds`, allowing a proposer to commit blocks with arbitrarily stale timestamps — (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`block_timestamp_window_seconds` is documented as *"Maximum allowed deviation (seconds) of a proposed block's timestamp from the current time"* — a symmetric bound. However, `is_proposal_init_valid` only enforces the **upper** (future) half of that bound. There is no check that `init_proposed.timestamp >= now - block_timestamp_window_seconds`. The only lower bound is `last_block_timestamp`, which can be arbitrarily far in the past (e.g., after a chain halt). A malicious proposer can therefore set `ProposalInit.timestamp` to any value ≥ `last_block_timestamp`, and every honest validator will accept the proposal. The committed block carries a wrong `block_timestamp` in `BlockInfo`, corrupting `get_block_timestamp()` syscall results for every transaction in the block, and can also force the L1 gas price oracle path into its fallback, locking in stale or minimum gas prices.

---

### Finding Description

In `is_proposal_init_valid` the timestamp checks are:

```rust
// validate_proposal.rs:260-284
let now: u64 = clock.unix_now();
let last_block_timestamp =
    proposal_init_validation.previous_proposal_init.as_ref().map_or(0, |info| info.timestamp);

if init_proposed.timestamp < last_block_timestamp {          // ← only lower bound: last block
    return Err(InvalidProposalInit(...));
}
if init_proposed.timestamp > now + proposal_init_validation.block_timestamp_window_seconds {
    return Err(InvalidProposalInit(...));                     // ← upper bound: future direction
}
// *** MISSING: if init_proposed.timestamp < now - block_timestamp_window_seconds { Err(...) }
``` [1](#0-0) 

The production value of `block_timestamp_window_seconds` is **1 second**. [2](#0-1) 

The config description explicitly says "Maximum allowed deviation … from the current time," implying the window should be symmetric. The implementation only enforces the future direction.

The same asymmetry exists in `try_sync`:

```rust
// sequencer_consensus_context.rs:1067-1081
if !(block_number == height
    && timestamp.0 >= last_block_timestamp
    && timestamp.0 <= now + self.config.static_config.block_timestamp_window_seconds)
``` [3](#0-2) 

The manipulated timestamp flows into two critical paths:

**Path 1 — `BlockInfo.block_timestamp`:**  
`convert_to_sn_api_block_info(init)` sets `block_timestamp: BlockTimestamp(init.timestamp)`, which is the value returned by the `get_block_timestamp` syscall for every transaction executed in the block. [4](#0-3) 

**Path 2 — L1 gas price oracle:**  
`get_l1_prices_in_fri_and_wei(l1_gas_price_provider, init_proposed.timestamp, ...)` queries the provider with the proposer-supplied timestamp. If the timestamp is old enough that `timestamp - lag_margin` falls outside the ring buffer, `get_price_info` returns `MissingDataError`, triggering fallback to previous-block prices or configured minimum prices. [5](#0-4) [6](#0-5) 

Because the validator calls `get_l1_prices_in_fri_and_wei` with the **same** proposer-supplied timestamp, both sides compute the same fallback prices and the `within_margin` check passes, so the proposal is accepted with artificially low L1 gas prices. [7](#0-6) 

---

### Impact Explanation

1. **Wrong `block_timestamp` in committed blocks (Critical — wrong syscall result from execution logic):** Every contract in the block that calls `get_block_timestamp()` receives the manipulated value. Time-locked contracts, deadline checks, and time-based access controls can be bypassed or broken. The corrupted timestamp is also embedded in the block header and propagated to the block hash commitment.

2. **Wrong L1 gas prices (Critical — incorrect L1 gas price effect with economic impact):** When the stale timestamp causes the oracle path to fall back to minimum configured prices, all transactions in the block are charged fees based on artificially low L1 gas prices. This creates an economic discrepancy between actual L1 costs and what users are charged.

---

### Likelihood Explanation

Requires a malicious proposer (validator node). In a BFT system, up to ⌊(n−1)/3⌋ validators can be Byzantine, so a malicious proposer is within the standard threat model. In normal operation (blocks every few seconds), `last_block_timestamp ≈ now − few_seconds`, limiting the exploitable window to seconds. The impact is most severe after a chain halt or restart, where `last_block_timestamp` can be hours in the past and the proposer can set a timestamp that is hours stale while still satisfying both existing checks.

---

### Recommendation

Add a past-direction guard in `is_proposal_init_valid` immediately after the existing lower-bound check:

```rust
// After the existing `timestamp < last_block_timestamp` check:
if now >= proposal_init_validation.block_timestamp_window_seconds
    && init_proposed.timestamp
        < now - proposal_init_validation.block_timestamp_window_seconds
{
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "Timestamp is too far in the past: now={}, window={}, proposed={}",
            now,
            proposal_init_validation.block_timestamp_window_seconds,
            init_proposed.timestamp
        ),
    ));
}
```

Apply the same fix to the timestamp check in `try_sync`. Consider whether `block_timestamp_window_seconds` needs a larger value (e.g., one consensus round duration) to tolerate normal clock skew while still bounding the past-direction deviation.

---

### Proof of Concept

1. Chain halts for 1 hour. State: `last_block_timestamp = T`, `now = T + 3600`, `block_timestamp_window_seconds = 1`.
2. Malicious proposer constructs `ProposalInit { timestamp: T + 1, l1_gas_price_fri: <minimum>, ... }`.
3. `is_proposal_init_valid` evaluates:
   - `T + 1 >= T` ✓ (passes existing lower bound)
   - `T + 1 <= (T + 3600) + 1` ✓ (passes existing upper bound)
   - No past-direction check exists → proposal accepted.
4. `get_l1_prices_in_fri_and_wei` is called with `timestamp = T + 1`. The ring buffer contains only recent L1 blocks (timestamps near `T + 3600`). `get_price_info(T + 1)` finds no block with `timestamp ≤ T + 1 − lag_margin` → `MissingDataError` → fallback to minimum prices. Validator computes the same result → `within_margin` passes.
5. Block is committed with `block_timestamp = T + 1` (3599 seconds in the past) and minimum L1 gas prices.
6. All contracts in the block receive `get_block_timestamp() = T + 1` instead of `≈ T + 3600`. Time-dependent logic (e.g., `assert(get_block_timestamp() < deadline)`) is evaluated against the manipulated value.
