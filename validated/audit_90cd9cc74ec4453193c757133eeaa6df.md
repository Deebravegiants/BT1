### Title
Missing `payload_hash`-to-`request` Binding Validation in `respond_verify_foreign_tx` Allows Cross-Request Payload Substitution — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` validates that the submitted signature is cryptographically valid over the caller-supplied `response.payload_hash`, but never checks that `response.payload_hash` was actually derived from the `request` being resolved. A single malicious MPC leader node (strictly below the signing threshold) can obtain a valid threshold signature for one pending foreign-chain verification request and use it to resolve a completely different pending request, delivering a `VerifyForeignTransactionResponse` whose `payload_hash` corresponds to a different transaction than the one the caller requested.

---

### Finding Description

The `verify_foreign_transaction` flow works as follows:

1. A user calls `verify_foreign_transaction(request)`, which stores a pending yield keyed on `VerifyForeignTransactionRequest { request, domain_id, payload_version }`.
2. MPC nodes independently query the foreign chain, build `ForeignTxSignPayload::V1 { request, values }`, compute `payload_hash = SHA-256(borsh(payload))`, and run a threshold signing session over that hash.
3. The leader calls `respond_verify_foreign_tx(request, response)` where `response = { payload_hash, signature }`.

The contract's validation in `respond_verify_foreign_tx` is:

```rust
// crates/contract/src/lib.rs:718-734
let signature_is_valid = match (&response.signature, public_key) {
    (
        dtos::SignatureResponse::Secp256k1(signature_response),
        PublicKeyExtended::Secp256k1 { near_public_key },
    ) => {
        let secp_pk = ...;
        let payload_hash: [u8; 32] = response.payload_hash.0;
        // Check the signature is correct against the root public key
        near_mpc_signature_verifier::verify_ecdsa_signature(
            signature_response,
            &payload_hash,
            &secp_pk,
        )
        .is_ok()
    }
    ...
};
```

The contract checks only:
- Is `response.signature` a valid ECDSA signature over `response.payload_hash` under the root key? ✓
- Does `request` match a pending entry in `pending_verify_foreign_tx_requests`? ✓

**What is never checked**: that `response.payload_hash` equals `ForeignTxSignPayload::V1 { request: request.request, values: [...] }.compute_msg_hash()`. The `payload_hash` is completely unconstrained relative to the `request` being resolved.

The `ForeignTxSignPayload` type that defines what should be signed is:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs:1499-1509
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
}
```

The `payload_hash` is supposed to commit to both the original `request` (tx_id, chain, extractors) and the observed `values`. The contract never enforces this binding.

On the node side, the leader independently builds the payload from the foreign chain query:

```rust
// crates/node/src/providers/verify_foreign_tx/sign.rs:73-86
let response_payload = self
    .execute_foreign_chain_request(
        &foreign_tx_request.request,
        foreign_tx_request.payload_version,
    )
    .await?;
let sign_request = build_signature_request(&foreign_tx_request, &response_payload)?;
```

And `build_signature_request` uses a zero tweak:

```rust
// crates/node/src/providers/verify_foreign_tx/sign.rs:39-47
Ok(SignatureRequest {
    ...
    tweak: Tweak::new([0u8; 32]),  // zero tweak — signed under root key
    ...
})
```

Because the tweak is zero, the signature is under the root public key — the same key used for all foreign-tx signing sessions. This means a valid signature obtained for any foreign-tx request is cryptographically valid over its `payload_hash` under the root key, and the contract will accept it as a response to any other pending foreign-tx request.

---

### Impact Explanation

A malicious MPC leader node can:

1. Observe two concurrent pending requests: **Request A** (for `tx_id=X`) and **Request B** (for `tx_id=Y`).
2. Lead the threshold signing session for Request B. Honest follower nodes independently verify `tx_id=Y` on the foreign chain, compute `payload_B = ForeignTxSignPayload{request=Y, values=[block_hash_Y]}`, and cooperate to produce a valid threshold signature `sig_B` over `hash(payload_B)`.
3. Call `respond_verify_foreign_tx(request=A, response={payload_hash=hash(payload_B), signature=sig_B})`.
4. The contract accepts: `sig_B` is a valid signature over `hash(payload_B)` under the root key ✓, and Request A is pending ✓.
5. Request A's caller receives `VerifyForeignTransactionResponse { payload_hash=hash(payload_B), signature=sig_B }` — a valid MPC signature, but over a payload that encodes `tx_id=Y`, not `tx_id=X`.

Any downstream bridge contract that uses this response to verify that `tx_id=X` was processed (e.g., to release funds or confirm a cross-chain transfer) is deceived. The signature is cryptographically valid, but it attests to the wrong transaction. This enables forged foreign-chain verification and potential double-spend or invalid bridge execution.

---

### Likelihood Explanation

- Requires exactly **one** malicious MPC node acting as signing leader — strictly below the signing threshold.
- Requires two concurrent pending foreign-tx requests (routine in any active bridge deployment).
- The malicious node needs to own a presignature to initiate a signing session, which is a normal operational state.
- No TEE break, no key leak, no threshold collusion, no network-level attack required.
- The attack is fully on-chain and requires only a crafted `respond_verify_foreign_tx` call.

---

### Recommendation

In `respond_verify_foreign_tx`, after verifying the signature, recompute the minimum expected `payload_hash` prefix from the `request` and verify that `response.payload_hash` is consistent with it. Concretely, the contract should verify:

```rust
// Reconstruct the canonical payload hash from the request and verify it matches
// the response.payload_hash before accepting the response.
// Since values are not stored on-chain, the check must be done differently:
// Option A: Store the expected payload_hash at request time (requires nodes to
//           commit to it before signing).
// Option B: Include the request's ForeignChainRpcRequest in the payload_hash
//           derivation and verify the hash prefix matches the request.
```

The most robust fix is to have the contract store a commitment to the expected `payload_hash` (or at minimum the `request` portion of it) when the request is enqueued, and verify that `response.payload_hash` starts with the hash of the request component before accepting the response.

### Proof of Concept

```
1. Alice calls verify_foreign_transaction({ request: Bitcoin(tx_id=X), domain_id=0, payload_version=V1 })
   → Contract stores pending request A keyed on { request=Bitcoin(tx_id=X), domain_id=0, payload_version=V1 }

2. Bob calls verify_foreign_transaction({ request: Bitcoin(tx_id=Y), domain_id=0, payload_version=V1 })
   → Contract stores pending request B keyed on { request=Bitcoin(tx_id=Y), domain_id=0, payload_version=V1 }

3. Malicious leader node leads signing for request B:
   - Queries Bitcoin RPC for tx_id=Y → gets block_hash_Y
   - Builds payload_B = ForeignTxSignPayload::V1 { request=Bitcoin(tx_id=Y), values=[BlockHash(block_hash_Y)] }
   - hash_B = SHA-256(borsh(payload_B))
   - Honest followers independently verify tx_id=Y and cooperate to produce sig_B over hash_B

4. Malicious leader calls:
   respond_verify_foreign_tx(
     request = { request=Bitcoin(tx_id=X), domain_id=0, payload_version=V1 },  // ← request A
     response = { payload_hash=hash_B, signature=sig_B }                        // ← payload for tx Y
   )

5. Contract checks:
   - verify_ecdsa_signature(sig_B, hash_B, root_pk) → OK  ✓
   - pending_verify_foreign_tx_requests.get(request_A) → found  ✓
   → Resolves request A with response { payload_hash=hash_B, signature=sig_B }

6. Alice's callback receives VerifyForeignTransactionResponse {
     payload_hash = hash(ForeignTxSignPayload{ request=Bitcoin(tx_id=Y), values=[block_hash_Y] }),
     signature = valid threshold signature
   }
   Alice's bridge contract believes tx_id=X was verified, but the signature attests to tx_id=Y.
```

The root cause is the missing binding check in `crates/contract/src/lib.rs` at lines 718–734: `response.payload_hash` is verified as a valid signature target but never verified to encode `request.request`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

fn build_signature_request(
    request: &VerifyForeignTxRequest,
    foreign_tx_payload: &dtos::ForeignTxSignPayload,

    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
        foreign_tx_payload.compute_msg_hash()?.into();

        tweak: Tweak::new([0u8; 32]),
        entropy: request.entropy,
        timestamp_nanosec: request.timestamp_nanosec,
        domain: request.domain_id,
    })
