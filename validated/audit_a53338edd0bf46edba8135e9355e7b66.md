### Title
User-Controlled Zero Confirmation Threshold Bypasses Bitcoin Finality Guarantee in `verify_foreign_transaction` - (File: crates/foreign-chain-inspector/src/bitcoin/inspector.rs)

### Summary
The `verify_foreign_transaction` contract endpoint accepts a caller-supplied `confirmations` field in `BitcoinRpcRequest` with no minimum bound enforced on-chain. An unprivileged caller can set `confirmations: 0`, causing MPC nodes to sign a payload for a Bitcoin transaction that has not reached any meaningful finality, enabling a double-spend attack against any bridge contract that relies on the MPC attestation.

### Finding Description

The `verify_foreign_transaction` endpoint accepts a `VerifyForeignTransactionRequestArgs` that, for Bitcoin, contains a `BitcoinRpcRequest` with a `confirmations: BlockConfirmations(u64)` field. This field is entirely caller-controlled and represents the minimum number of Bitcoin block confirmations the MPC nodes must observe before signing.

The contract performs no minimum-bound validation on this value:

```rust
pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
    self.check_request_preconditions(
        request.domain_id,
        DomainPurpose::ForeignTx,
        Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
        MINIMUM_SIGN_REQUEST_DEPOSIT,
    );
    let requested_chain = request.request.chain();
    let supported_chains = self.get_supported_foreign_chains();
    if !supported_chains.contains(&requested_chain) { ... }
    // ← No validation of confirmations field
    ...
}
``` [1](#0-0) 

The `BlockConfirmations` type is a plain `u64` newtype with no invariant enforcing a minimum: [2](#0-1) 

On the node side, `BitcoinInspector::extract` uses the caller-supplied threshold directly in a `<=` comparison against the actual on-chain confirmation count:

```rust
let transaction_block_confirmation = rpc_response.confirmations.into();
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;

if !enough_block_confirmations {
    return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations { ... });
}
``` [3](#0-2) 

When `block_confirmations_threshold = 0`, the condition `0 <= actual_confirmations` is trivially true for any mined transaction (even one with a single confirmation), so the inspector proceeds to sign without any finality guarantee.

### Impact Explanation

An attacker submits `verify_foreign_transaction` with `confirmations: 0` for a Bitcoin transaction they just broadcast. The MPC network observes the transaction in a block (1 confirmation), passes the `0 <= 1` check, and issues a threshold signature over the payload `(request, observed_values)`. The attacker immediately presents this MPC-signed attestation to a NEAR bridge contract to claim bridged funds. The underlying Bitcoin transaction, having only 1 confirmation, is then double-spent via a mining reorg (feasible within 1–2 blocks on Bitcoin). The bridge has already released funds on NEAR based on the MPC attestation, resulting in a direct loss of bridged funds.

This matches the allowed impact: **High — forged foreign-chain verification / light-client-style verification bypass causing double-spend conditions**.

### Likelihood Explanation

The entry path requires only 1 yoctoNEAR deposit and a valid Bitcoin transaction ID. Any unprivileged caller can craft the request. Bitcoin 1-block reorgs occur naturally; deliberate reorg attacks are economically viable for high-value bridge transactions. The attack requires no collusion, no privileged access, and no TEE compromise.

### Recommendation

Enforce a protocol-level minimum confirmation count on-chain inside `verify_foreign_transaction` before the request is enqueued. The minimum should be a governance-controlled parameter (e.g., stored in `Config` and voted on by participants), not a caller-supplied value. Reject any `BitcoinRpcRequest` whose `confirmations` field is below this minimum:

```rust
let min_confirmations = self.config.bitcoin_min_confirmations; // e.g., 6
if let ForeignChainRpcRequest::Bitcoin(ref btc_req) = request.request {
    if btc_req.confirmations < BlockConfirmations(min_confirmations) {
        env::panic_str("confirmations below protocol minimum");
    }
}
```

Similarly, for EVM chains, restrict `EvmFinality` to `Finalized` or `Safe` only, rejecting `Latest` at the contract level.

### Proof of Concept

1. Attacker broadcasts a Bitcoin transaction `TX` sending funds to the bridge deposit address.
2. `TX` is mined into block `B` (1 confirmation).
3. Attacker calls `verify_foreign_transaction` with:
   ```json
   {
     "request": {
       "domain_id": <foreign_tx_domain>,
       "payload_version": "V1",
       "request": {
         "Bitcoin": {
           "tx_id": "<TX hash>",
           "confirmations": 0,
           "extractors": ["BlockHash"]
         }
       }
     }
   }
   ```
   with 1 yoctoNEAR deposit.
4. Contract accepts the request (chain is supported, deposit is sufficient, no confirmation floor check).
5. MPC nodes call `BitcoinInspector::extract(TX, BlockConfirmations(0), [BlockHash])`. The check `0 <= 1` passes. The inspector signs the payload.
6. Attacker receives a valid MPC threshold signature attesting that `TX` was observed with `confirmations: 0`.
7. Attacker submits the MPC signature to the NEAR bridge contract, which releases bridged funds.
8. Attacker simultaneously mines a competing Bitcoin block at height `B`, reorganizing `TX` out of the chain (double-spend).
9. The bridge has released NEAR-side funds; the Bitcoin deposit no longer exists on the canonical chain.
