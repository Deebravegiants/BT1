### Title
Caller-Agnostic `VerifyForeignTransactionRequest` Key Enables Targeted Fan-Out Queue Saturation DoS — (File: `crates/contract/src/pending_requests.rs`, `crates/contract/src/lib.rs`)

---

### Summary

The `pending_verify_foreign_tx_requests` map uses a caller-agnostic key (`VerifyForeignTransactionRequest` = `{ForeignChainRpcRequest, domain_id, payload_version}`) with a hard per-key cap of `MAX_PENDING_REQUEST_FAN_OUT = 128`. There is no eviction strategy: when the cap is reached, new submissions panic immediately. Because the key excludes the caller's account ID, an unprivileged attacker who knows a target foreign transaction ID (public on-chain data) can pre-fill all 128 slots at essentially zero cost (1 yoctoNEAR per call), permanently blocking any subsequent legitimate bridge user from submitting the same verification request until the attacker's yields time out — and the attacker can continuously refill the queue to maintain the block indefinitely.

---

### Finding Description

`VerifyForeignTransactionRequest` is defined as:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [1](#0-0) 

The caller's `AccountId` is **not** part of the key. This is confirmed by the codebase's own test comment: *"a different account would today be blocked from receiving a response by alice's submission."* [2](#0-1) 

The fan-out queue enforces a hard cap of 128 yields per key:

```rust
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;
// ...
if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
    env::panic_str(&RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }.to_string());
}
``` [3](#0-2) 

When the cap is reached, the contract simply panics — there is no eviction of older or lower-priority entries, no per-caller rate limit, and no mechanism to reclaim slots except natural timeout (`pop_oldest_pending_yield` fires only when the NEAR yield-resume callback fires on timeout, approximately every 200 blocks). [4](#0-3) 

The `verify_foreign_transaction` entry point requires only 1 yoctoNEAR deposit: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

An attacker who observes a foreign-chain transaction (e.g., a Bitcoin `tx_id`) that a bridge user is about to verify can:

1. Submit 128 `verify_foreign_transaction` calls for that exact `{tx_id, domain_id, payload_version}` tuple before the legitimate user.
2. The queue for that key is now full.
3. The legitimate user's `verify_foreign_transaction` call panics with `PendingRequestQueueFull`, reverting their NEAR transaction.
4. After ~200 blocks the attacker's yields time out; the attacker immediately resubmits 128 new calls.
5. The legitimate user is permanently blocked from completing their bridge verification at a cost of ~128 yoctoNEAR per 200-block cycle (effectively free).

This breaks the bridge request lifecycle: the user cannot obtain the MPC-signed verification payload needed to complete their cross-chain operation. If the bridge contract on the foreign chain has a time-bounded claim window, the user's bridged funds can be permanently lost.

**Impact category**: Medium — request-lifecycle and contract execution-flow manipulation that breaks production bridge safety invariants, without relying on network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

- Foreign transaction IDs (`tx_id`) are public on every supported chain (Bitcoin, Ethereum, etc.) and visible in the mempool before confirmation.
- The attacker needs only 128 NEAR transactions at 1 yoctoNEAR each — negligible cost.
- No privileged access, TEE compromise, or threshold collusion is required.
- The attack is reachable by any unprivileged NEAR account calling `verify_foreign_transaction`.
- The attacker can sustain the block indefinitely by refilling the queue every ~200 blocks.

---

### Recommendation

1. **Include the caller's `predecessor_account_id` in the `VerifyForeignTransactionRequest` key** (analogous to how `SignatureRequest` incorporates the caller via `tweak`). This isolates each caller's queue slot and prevents cross-caller saturation.
2. **Implement a per-caller slot limit** within the fan-out queue so a single account cannot consume all 128 slots for a given request key.
3. **Increase the deposit requirement** for `verify_foreign_transaction` to make bulk queue-filling economically prohibitive.
4. **Add an eviction strategy**: when the queue is full, evict the oldest entry (or the entry from the account with the most slots) rather than panicking.

---

### Proof of Concept

```
Attacker (any NEAR account):
  for i in 0..128:
    call verify_foreign_transaction({
      request: Bitcoin { tx_id: TARGET_TX_ID, confirmations: 1, extractors: [BlockHash] },
      domain_id: FOREIGN_TX_DOMAIN_ID,
      payload_version: V1,
    }, deposit=1 yoctoNEAR)

Victim (legitimate bridge user):
  call verify_foreign_transaction({ same args })
  → panics: "Pending-request queue is full (limit: 128)"

After ~200 blocks, attacker refills queue → victim blocked again.
Cost to attacker: 128 yoctoNEAR per 200-block cycle ≈ $0.
```

pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
