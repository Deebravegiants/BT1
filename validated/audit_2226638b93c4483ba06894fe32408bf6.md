### Title
Single Attested Participant Can Inject Arbitrary CKD Response for `AppPublicKey` Requests Due to Silently Skipped Output Validation - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract silently skips the output validation check for `AppPublicKey`-variant CKD requests. Unlike `AppPublicKeyPV` requests (which enforce `ckd_output_check`) and signature requests (which enforce cryptographic signature verification against the derived public key), `AppPublicKey` CKD responses are accepted without any content validation. Because `respond_ckd` is callable by **any** single attested participant with no leader restriction, a single Byzantine participant below the signing threshold can race the legitimate leader, inject an arbitrary CKD response, and cause the yield to resolve with an attacker-controlled key derivation output.

---

### Finding Description

In `respond_ckd`, the output check is conditionally applied based on the `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← silently skipped
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}

pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [1](#0-0) 

For the `AppPublicKey` arm the match body is empty — the critical validation is silently skipped — and execution falls through unconditionally to `resolve_yields_for`, which immediately removes the request from the pending map and resumes every queued yield with the caller-supplied `response_bytes`. [2](#0-1) 

Contrast this with `respond` (signature flow), where the contract cryptographically verifies the submitted signature against the expected derived public key before resolving any yield: [3](#0-2) 

No equivalent cryptographic guard exists for `AppPublicKey` CKD responses. The only caller-side restriction is `assert_caller_is_attested_participant_and_protocol_active`, which any single current participant satisfies: [4](#0-3) 

There is no leader-only restriction, no threshold-of-responses requirement, and no content check. The first attested participant to call `respond_ckd` with a matching `CKDRequest` key wins: the pending entry is removed and all waiting yields are resolved with whatever `CKDResponse` the caller supplied.

**Analog to the external report:** `harvest()` silently skipped its critical sub-operation (emptying the strategy) when a cooldown condition was true, allowing `changeAdapter()` to proceed without the required precondition. Here, `respond_ckd` silently skips its critical sub-operation (output validation) when the `AppPublicKey` variant is present, allowing `resolve_yields_for` to proceed and deliver an unvalidated — potentially attacker-controlled — key derivation output.

---

### Impact Explanation

A single Byzantine attested participant (strictly below the signing threshold) can:

1. Monitor public contract state for any pending `AppPublicKey` CKD request.
2. Call `respond_ckd` before the legitimate leader with an arbitrary `CKDResponse`.
3. The contract accepts the response with no validation and resolves all queued yields.
4. The user receives an attacker-chosen confidential key derivation output.
5. The legitimate leader's subsequent `respond_ckd` call fails with `RequestNotFound` — the request has already been drained.

This constitutes **confidential key derivation output without the required participant authorization**: the threshold MPC protocol requires `t`-of-`n` nodes to collaboratively compute the response, but the contract enforces no such threshold for `AppPublicKey` responses, allowing a single node to unilaterally determine the output. Any downstream use of the derived key (encryption, signing, asset custody) is compromised.

---

### Likelihood Explanation

The attack surface is open to any current attested participant. The attacker needs only:

- A valid TEE attestation accepted by the contract (standard operational requirement for MPC nodes).
- The ability to read pending `CKDRequest` keys from public contract state.
- The ability to submit a NEAR transaction calling `respond_ckd` before the legitimate leader.

No threshold collusion, no key material, and no privileged access are required. The race window is the time between a user's `request_app_private_key` call and the legitimate leader's `respond_ckd` call — a window that exists on every CKD request.

---

### Recommendation

1. **Add an output check for `AppPublicKey` variant.** If a cryptographic verifiability property can be defined for `AppPublicKey` (analogous to `ckd_output_check` for `AppPublicKeyPV`), enforce it in the same `match` arm before calling `resolve_yields_for`.

2. **If no on-chain verifiability is possible for `AppPublicKey`, enforce a threshold-of-responses pattern.** Require `t` independent `respond_ckd` calls with matching responses before resolving the yield, mirroring how the off-chain threshold protocol is supposed to work.

3. **Restrict `respond_ckd` to the designated leader.** Include the leader's participant ID in the `CKDRequest` or derive it deterministically, and reject calls from non-leaders, preventing any single participant from racing the legitimate responder.

---

### Proof of Concept

```
1. Attacker A is an attested participant in the current epoch.
2. User U calls request_app_private_key({ app_public_key: AppPublicKey(pk), ... }).
   → Contract stores CKDRequest R in pending_ckd_requests; yield Y is parked.
3. A reads R from public contract state.
4. A calls respond_ckd(R, CKDResponse { /* arbitrary bytes */ }).
   → respond_ckd: AppPublicKey arm → no check → resolve_yields_for removes R,
     resumes Y with attacker-chosen bytes.
5. U's parked call resumes with the attacker-chosen CKDResponse.
6. Legitimate leader L later calls respond_ckd(R, correct_response).
   → resolve_yields_for: R not found → returns Err(RequestNotFound).
   → L's call fails; correct response is never delivered.
7. U uses the attacker-chosen derived key for downstream operations.
```

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
