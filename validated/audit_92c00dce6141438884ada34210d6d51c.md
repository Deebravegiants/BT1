### Title
Fee-on-Transfer ERC20 Assets Cause rsETH Over-Minting and Holder Value Dilution - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool.depositAsset` computes the rsETH mint amount from the caller-supplied `depositAmount` before executing `safeTransferFrom`. If a supported asset charges a transfer fee, the contract receives fewer tokens than `depositAmount`, yet mints rsETH as if the full amount arrived. This inflates rsETH supply beyond its actual backing, permanently diluting all existing rsETH holders.

### Finding Description
In `depositAsset`, the mint calculation and the actual token transfer are decoupled: [1](#0-0) 

`_beforeDeposit` calls `getRsETHAmountToMint(asset, depositAmount)` at line 665, which computes: [2](#0-1) 

This uses the caller-supplied `depositAmount` directly. The `safeTransferFrom` at line 114 then executes, but for a fee-on-transfer token the contract only receives `depositAmount - fee`. Despite this shortfall, `_mintRsETH(rsethAmountToMint)` at line 115 mints rsETH calculated on the full `depositAmount`. [3](#0-2) 

The same pattern exists in `RSETHPoolV3.deposit(address token, uint256 amount, ...)`, where `viewSwapRsETHAmountAndFee(amount, token)` is called with the user-supplied `amount` before the transfer, and `wrsETH.mint` is called with the result: [4](#0-3) 

And in `AGETHTokenWrapper._deposit`, which mints agETH 1:1 with `_amount` regardless of actual received tokens: [5](#0-4) 

### Impact Explanation
The rsETH price is computed in `LRTOracle._updateRsETHPrice` as `totalETHInProtocol / rsethSupply`: [6](#0-5) 

When rsETH is over-minted (supply inflated beyond actual backing), `rsETHPrice` decreases. Every existing rsETH holder's redemption value is permanently reduced. A depositor using a fee-on-transfer asset extracts value from all existing holders on every deposit. At scale this constitutes theft of yield from rsETH holders and can drive the protocol toward insolvency.

**Impact**: High — Theft of unclaimed yield / value from existing rsETH holders.

### Likelihood Explanation
The currently listed assets (stETH, ETHx) do not charge transfer fees. However, the protocol's asset listing mechanism (`LRTConfig.addNewSupportedAsset`) does not enforce any restriction against fee-on-transfer tokens, and the protocol is explicitly designed to expand its supported asset list. If any future supported LST enables a transfer fee (as USDT and USDC have been known to consider), any unprivileged depositor can immediately exploit this path with no special access required.

**Likelihood**: Low — requires a fee-on-transfer token to be added as a supported asset, which is a realistic future event given the protocol's expansion trajectory.

### Recommendation
Measure the actual received amount by comparing balances before and after `safeTransferFrom`, and use the delta for both the rsETH mint calculation and event emission:

```solidity
function depositAsset(...) external ... {
    uint256 balanceBefore = IERC20(asset).balanceOf(address(this));
    IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
    uint256 actualReceived = IERC20(asset).balanceOf(address(this)) - balanceBefore;

    uint256 rsethAmountToMint = _beforeDeposit(asset, actualReceived, minRSETHAmountExpected);
    _mintRsETH(rsethAmountToMint);

    emit AssetDeposit(msg.sender, asset, actualReceived, rsethAmountToMint, referralId);
}
```

Apply the same fix to `RSETHPoolV3.deposit(address token, ...)` and `AGETHTokenWrapper._deposit`. Additionally, enforce in the asset listing checklist that fee-on-transfer tokens are explicitly disallowed as supported assets.

### Proof of Concept
1. A fee-on-transfer LST (e.g., 1% fee) is added as a supported asset in `LRTConfig`.
2. Attacker calls `LRTDepositPool.depositAsset(feeToken, 1000e18, 0, "")`.
3. `_beforeDeposit` computes `rsethAmountToMint` based on `1000e18`.
4. `safeTransferFrom` transfers `1000e18` from attacker; contract receives `990e18` (1% fee deducted).
5. `_mintRsETH` mints rsETH equivalent to `1000e18` of the asset.
6. `LRTOracle._updateRsETHPrice` now computes: `totalETHInProtocol` increased by `990e18 * assetPrice`, but `rsethSupply` increased by the equivalent of `1000e18 * assetPrice`. The new `rsETHPrice` is lower than before.
7. All existing rsETH holders now hold tokens worth less ETH. The attacker received rsETH worth `1000e18` of asset value while only depositing `990e18` worth — the `10e18` difference was extracted from existing holders. [7](#0-6) [8](#0-7)
