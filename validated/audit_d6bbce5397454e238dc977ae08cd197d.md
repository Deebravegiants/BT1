Audit Report

## Title
Token-Specific Fee Defaults to Zero Due to Missing Fallback Logic - (File: `contracts/pools/RSETHPool.sol`)

## Summary
`RSETHPool.viewSwapRsETHAmountAndFee(uint256, address)` reads `tokenFeeBps[token]` directly without falling back to the global `feeBps` when the per-token fee is uninitialized. Because `addSupportedToken()` never initializes `tokenFeeBps[token]`, any deposit made between token registration and an explicit `setTokenFeeBps()` call pays zero fee. The protocol cannot distinguish "not configured" from "explicitly zero."

## Finding Description
`RSETHPool` maintains two fee-rate storage slots: the global `feeBps` used for ETH deposits and `tokenFeeBps[token]` for per-token deposits. The ETH path at line 312 correctly uses `feeBps`. The token path at lines 335–336 reads `tokenFeeBps[token]` with no fallback:

```solidity
uint256 feeBpsForToken = tokenFeeBps[token];
fee = amount * feeBpsForToken / 10_000;
```

`addSupportedToken()` (lines 651–653) registers `supportedTokenOracle[token]` and `tokenBridge[token]` but never writes `tokenFeeBps[token]`, leaving it at the Solidity mapping default of `0`. The public `deposit(address token, uint256 amount, string referralId)` function (lines 284–305) has no guard requiring a non-zero fee configuration; it calls `viewSwapRsETHAmountAndFee(amount, token)` directly. Any depositor calling `deposit()` during the window between `addSupportedToken()` and `setTokenFeeBps()` will have `feeBpsForToken = 0`, yielding `fee = 0` regardless of the global `feeBps` setting.

## Impact Explanation
**Low — Contract fails to deliver promised returns.** The protocol collects zero fee revenue on all token deposits made before `setTokenFeeBps()` is explicitly called. No user funds are lost or frozen; the depositor simply receives the full rsETH amount without fee deduction. The protocol's fee mechanism silently fails for every newly added token until the admin takes a separate, uncoupled action.

## Likelihood Explanation
`addSupportedToken()` and `setTokenFeeBps()` are independent admin calls with no on-chain coupling or ordering enforcement. Any deployment of a new supported token creates an immediate zero-fee window. A depositor needs no special capability — calling `deposit(token, amount, referralId)` during this window is sufficient. The window persists until the admin explicitly calls `setTokenFeeBps()`, which may span multiple blocks or longer.

## Recommendation
Two concrete options:

1. **Fallback to global `feeBps`**: In `viewSwapRsETHAmountAndFee(uint256, address)`, replace the direct read with a fallback: `uint256 feeBpsForToken = tokenFeeBps[token] != 0 ? tokenFeeBps[token] : feeBps;`. Introduce a separate `tokenFeeBpsSet[token]` boolean to allow an explicit zero-fee override.
2. **Initialize on registration**: Add a `feeBps` parameter to `addSupportedToken()` and atomically set `tokenFeeBps[token]` during registration, eliminating any zero-fee window entirely.

## Proof of Concept
1. Deploy `RSETHPool` with `feeBps = 30` (0.3%).
2. Admin calls `addSupportedToken(wstETH, oracle, bridge)`. `tokenFeeBps[wstETH]` is `0` (uninitialized).
3. Before admin calls `setTokenFeeBps(wstETH, 30)`, a depositor calls `deposit(wstETH, 1e18, "ref")`.
4. `viewSwapRsETHAmountAndFee(1e18, wstETH)` executes: `feeBpsForToken = tokenFeeBps[wstETH] = 0`, so `fee = 1e18 * 0 / 10_000 = 0`.
5. The depositor receives the full rsETH amount with zero fee. The protocol earns nothing on this deposit.

**Foundry test sketch:**
```solidity
function test_zeroFeeWindowOnNewToken() public {
    // feeBps = 30 set at deployment
    pool.addSupportedToken(address(wstETH), oracle, bridge);
    // setTokenFeeBps NOT called yet
    vm.prank(depositor);
    pool.deposit(address(wstETH), 1e18, "ref");
    assertEq(pool.feeEarnedInToken(address(wstETH)), 0); // fee is zero
}
