### Title
`stake32Eth` Remains `public` and Bypasses Deposit Root Check, Enabling Withdrawal Credential Hijacking — (File: `contracts/NodeDelegator.sol`)

### Summary
`NodeDelegator.stake32Eth` is a `public` function that performs a 32 ETH validator deposit into EigenLayer without verifying the ETH2.0 deposit contract's current deposit root. Although a protected variant `stake32EthValidated` was added that performs this check, `stake32Eth` was never restricted to `internal`, leaving the unprotected path permanently callable by the LRT operator. An attacker monitoring the mempool can front-run any `stake32Eth` call, register the same validator pubkey with attacker-controlled withdrawal credentials, and cause the protocol's deposit to be treated as a top-up — permanently redirecting the 32 ETH away from the EigenPod.

### Finding Description
`NodeDelegator.sol` exposes two staking entry points:

- `stake32Eth` ( [1](#0-0) ) — `public`, `onlyLRTOperator`, `whenNotPaused`. Calls `_getEigenPodManager().stake{value: 32 ether}(pubkey, signature, depositDataRoot)` with **no deposit root check**.
- `stake32EthValidated` ( [2](#0-1) ) — `external`, **no access control modifiers**. Reads `depositContract.get_deposit_root()`, reverts on mismatch, then delegates to `stake32Eth`.

The protected path was introduced as a workaround for the ETH2.0 deposit contract's inability to distinguish initial deposits from top-up deposits. However, `stake32Eth` was left `public` instead of being changed to `internal`. Any LRT operator can call `stake32Eth` directly, skipping the deposit root guard entirely.

The ETH2.0 deposit contract processes deposits sequentially: the first deposit for a given pubkey sets the withdrawal credentials; all subsequent deposits for the same pubkey are treated as balance top-ups and the supplied withdrawal credentials are silently ignored. [3](#0-2) 

When the operator calls `stake32Eth`, an attacker who observes the pending transaction in the mempool can front-run it by calling the ETH2.0 deposit contract directly with the same `pubkey` but with attacker-controlled withdrawal credentials. The operator's transaction then lands as a top-up: the EigenPod's withdrawal credentials are ignored, and the 32 ETH is permanently bonded to the attacker's key.

The `PubkeyRegistry` check inside `stake32Eth` only prevents the same pubkey from being reused within the protocol; it does not prevent an external actor from depositing to the ETH2.0 contract with that pubkey before the operator's transaction is mined. [4](#0-3) 

### Impact Explanation
**Critical — Direct theft of user funds (32 ETH per validator).**

After the front-run succeeds, `stakedButUnverifiedNativeETH` is incremented by 32 ETH ( [5](#0-4) ) and the protocol accounts for the ETH as restaked. However, the validator's withdrawal credentials on the beacon chain point to the attacker, not the EigenPod. The subsequent `verifyWithdrawalCredentials` call will fail because EigenLayer verifies that the withdrawal credentials match the EigenPod address. The 32 ETH is permanently unrecoverable by the protocol, constituting direct theft of funds at rest.

### Likelihood Explanation
**Medium.** The LRT operator role is permissioned, but `stake32Eth` is a documented, `public` function that predates `stake32EthValidated`. Operators may call it directly — by habit, by tooling that was written before `stake32EthValidated` existed, or simply because it is the simpler call. Any attacker passively watching the mempool for `stake32Eth` calls can execute the front-run with a single direct deposit to the ETH2.0 contract. No special privileges are required of the attacker.

### Recommendation
Change `stake32Eth` from `public` to `internal`, making `stake32EthValidated` the sole external entry point for validator staking. This forces every staking call to pass through the deposit root check.

```solidity
// contracts/NodeDelegator.sol
function stake32Eth(
    bytes calldata pubkey,
    bytes calldata signature,
    bytes32 depositDataRoot
)
-   public
+   internal   // was public; use stake32EthValidated as the external entry point
    whenNotPaused
    onlyLRTOperator   // remove this modifier too if making internal, or keep for safety
{
    ...
}
```

Alternatively, if `stake32Eth` must remain callable externally for legacy reasons, add the same deposit root check inline:

```solidity
IETHPOSDeposit depositContract = _getEigenPodManager().ethPOS();
bytes32 actualRoot = depositContract.get_deposit_root();
if (expectedDepositRoot != actualRoot) {
    revert InvalidDepositRoot(expectedDepositRoot, actualRoot);
}
```

### Proof of Concept

1. LRT operator constructs a call to `NodeDelegator.stake32Eth(pubkey, sig, depositDataRoot)` and broadcasts it.
2. Attacker observes the pending transaction in the mempool, extracts `pubkey`.
3. Attacker calls the ETH2.0 deposit contract directly with the same `pubkey`, a valid BLS signature, and **attacker-controlled withdrawal credentials**, paying 1 ETH (minimum initial deposit) or 32 ETH.
4. Attacker's transaction is mined first (higher gas price). The ETH2.0 contract records the attacker's withdrawal credentials as the canonical credentials for `pubkey`.
5. Operator's `stake32Eth` transaction is mined. `PubkeyRegistry.hasPubkey(pubkey)` returns `false` (the registry only tracks protocol-internal usage), so the check passes. [6](#0-5) 
6. `_getEigenPodManager().stake{value: 32 ether}(pubkey, sig, depositDataRoot)` executes. The ETH2.0 deposit contract treats this as a top-up; the EigenPod's withdrawal credentials are silently discarded. [7](#0-6) 
7. `stakedButUnverifiedNativeETH += 32 ether` is recorded. The protocol believes 32 ETH is restaked.
8. When the operator later calls `verifyWithdrawalCredentials`, EigenLayer's proof verification confirms the beacon chain withdrawal credentials do **not** match the EigenPod — the call reverts. The 32 ETH is permanently lost to the attacker.

        bytes calldata pubkey,

        public
        whenNotPaused

    {
        IPubkeyRegistry pubkeyRegistry = IPubkeyRegistry(lrtConfig.pubkeyRegistry());
        if (pubkeyRegistry.hasPubkey(pubkey)) {
            revert PubkeyAlreadyRegistered();
        }
        pubkeyRegistry.addPubkey(pubkey);

        // tracks staked but unverified native ETH
        stakedButUnverifiedNativeETH += 32 ether;

        _getEigenPodManager().stake{ value: 32 ether }(pubkey, signature, depositDataRoot);

        if (address(eigenPod) == address(0)) {
            eigenPod = _getEigenPodManager().ownerToPod(address(this));
            emit EigenPodCreated(address(eigenPod), address(this));
        }
        emit ETHStaked(pubkey, 32 ether);
    }
