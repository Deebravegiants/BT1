Audit Report

## Title
Permissionless `updateRSETHPrice()` Allows Any Caller to Exhaust Daily Fee Minting Budget, Suppressing Treasury Yield - (File: contracts/LRTOracle.sol)

## Summary
`updateRSETHPrice()` carries no access-control modifier, allowing any unprivileged caller to trigger fee minting and consume the shared `maxFeeMintAmountPerDay` budget. An attacker who calls the function early in a day — when only a fraction of the day's yield has accrued — partially fills the daily cap. Subsequent legitimate calls that would push cumulative minted fees above the cap revert with `DailyFeeMintLimitExceeded`, causing the treasury to forfeit the remaining day's fees. The forfeited fees inflate `rsETHPrice` instead, benefiting all rsETH holders including the attacker. The attack is repeatable every day at gas cost only.

## Finding Description
`updateRSETHPrice()` is declared `public` with only a `whenNotPaused` guard:

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

A privileged variant already exists at L94-96 (`updateRSETHPriceAsManager() external onlyLRTManager`), confirming the design intent for restricted access was partially implemented but not applied to the public entry point.

Inside `_updateRsETHPrice()`, every execution path calls `_checkAndUpdateDailyFeeMintLimit` (L303 for the fee path, L310 for the zero-fee path). That function enforces a hard per-day cap:

```solidity
// L197-210
if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
    revert DailyFeeMintLimitExceeded(...);
}
currentPeriodMintedFeeAmount += feeAmount;
```

The fee is computed from the TVL increase since the last price update:

```solidity
// L244-246
uint256 rewardAmount = totalETHInProtocol - previousTVL;
protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
```

`previousTVL` is derived from `rsethSupply.mulWad(rsETHPrice)` (L234), and `rsETHPrice` is only updated on a successful (non-reverting) call (L313). This means:

- **Attacker call (early in day):** Small yield has accrued → small fee X minted → `currentPeriodMintedFeeAmount = X` → `rsETHPrice` updated to P1.
- **Operator call (later in day):** More yield Y has accrued since P1 → fee Y would be minted → if `X + Y > maxFeeMintAmountPerDay` → entire transaction **reverts** → `rsETHPrice` stays at P1, Y is never collected.
- **Next day:** Period resets, `previousTVL` is still based on P1, so Y is included in the next day's fee calculation. If the attacker repeats the pattern, Y is again partially consumed by the attacker's early call, and the remainder is again blocked.

The `pricePercentageLimit` check (L252-266) does not prevent this: the attacker deliberately calls when yield is small, keeping the price increase within the threshold. If `pricePercentageLimit == 0`, the check is bypassed entirely.

## Impact Explanation
**High — Theft of unclaimed yield.**

The protocol treasury's unclaimed yield (protocol fees) is permanently suppressed. The Y rsETH that the treasury should receive instead remains as unrealized yield in the protocol, inflating `rsETHPrice` and accruing proportionally to all rsETH holders. This is a direct, repeatable transfer of value from the treasury to rsETH holders. The treasury is the victim; rsETH holders (including the attacker) are the beneficiaries. The impact matches the allowed scope: **High. Theft of unclaimed yield.**

## Likelihood Explanation
**Medium.** The attacker requires no privileged access, no capital at risk beyond gas, and no off-chain infrastructure beyond a bot monitoring `totalETHInProtocol` vs. `rsETHPrice`. Holding rsETH to benefit is a low barrier. The attack is most effective when yield accrues unevenly (e.g., large EigenLayer reward distributions late in the day), a realistic on-chain condition. The only constraint is timing: the attacker must call before the operator and before the full day's yield has accrued.

## Recommendation
Gate `updateRSETHPrice()` behind the same role check as the already-existing privileged variant:

```solidity
function updateRSETHPrice() public whenNotPaused onlyLRTOperator {
    _updateRsETHPrice();
}
```

Alternatively, remove the public function entirely and rely solely on `updateRSETHPriceAsManager()`. If permissionless price updates are required for liveness, decouple the fee-minting step from the price-update step so that only authorised callers can trigger fee collection.

## Proof of Concept
**Setup:** `maxFeeMintAmountPerDay = 100 rsETH`, `protocolFeeInBPS = 1000` (10%).

1. Day starts; `currentPeriodMintedFeeAmount = 0`, `rsETHPrice = P0`.
2. **Attacker calls `updateRSETHPrice()` early:** 500 ETH-equivalent yield has accrued → `protocolFeeInETH = 50 ETH` → `rsethAmountToMintAsProtocolFee ≈ 50 rsETH` → minted to treasury. `currentPeriodMintedFeeAmount = 50`. `rsETHPrice = P1`.
3. **600 ETH-equivalent more yield accrues later in the day.**
4. **Operator calls `updateRSETHPrice()`:** `rsethAmountToMintAsProtocolFee ≈ 60 rsETH`. `_checkAndUpdateDailyFeeMintLimit(60)`: `50 + 60 = 110 > 100` → **reverts**. The 60 rsETH in fees are never collected; they inflate `rsETHPrice` instead.
5. **Day 2:** Period resets. `previousTVL` is still based on P1. Attacker repeats step 2 with the deferred 60 rsETH yield + any new early-day yield, again partially filling the cap and blocking the operator's later call.

**Foundry test plan:** Deploy `LRTOracle` with `maxFeeMintAmountPerDay = 100e18`. Warp to day start. Simulate TVL increase of 500 ETH. Call `updateRSETHPrice()` as an unprivileged address; assert it succeeds and `currentPeriodMintedFeeAmount == 50e18`. Simulate additional TVL increase of 600 ETH. Call `updateRSETHPrice()` as the operator; assert it reverts with `DailyFeeMintLimitExceeded`. Assert `rsETHPrice` is unchanged from P1 and treasury balance is still 50 rsETH.
