### Title
Stale Cached Rate in `CrossChainRateReceiver.getRate()` Enables Over-Minting of wrsETH — (File: `contracts/cross-chain/CrossChainRateReceiver.sol`)

---

### Summary

`CrossChainRateReceiver` stores a cached rsETH/ETH exchange rate (`rate`) that is updated asynchronously via LayerZero cross-chain messages. The `getRate()` function returns this cached value unconditionally, with no check against the stored `lastUpdated` timestamp. All L2 pool contracts use this rate to calculate how many wrsETH tokens to mint per deposited ETH. When the cached rate is stale (lower than the true current rate), depositors receive more wrsETH than their deposit is worth, extracting value from the protocol.

---

### Finding Description

`CrossChainRateReceiver` maintains two state variables relevant to this issue:

- `uint256 public rate` — the last received rsETH/ETH rate [1](#0-0) 
- `uint256 public lastUpdated` — the timestamp of the last LayerZero update [2](#0-1) 

The rate is written only inside `lzReceive()`, which is called by the LayerZero endpoint when a cross-chain message arrives from L1: [3](#0-2) 

The public accessor `getRate()` returns the cached value with no staleness guard:

```solidity
function getRate() external view returns (uint256) {
    return rate;
}
``` [4](#0-3) 

Every L2 pool contract calls `IOracle(rsETHOracle).getRate()` inside `viewSwapRsETHAmountAndFee()` to compute the wrsETH minting amount:

```solidity
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [5](#0-4) 

The same pattern appears in `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, and `RSETHPoolNoWrapper`. [6](#0-5) 

Because rsETH is a yield-bearing token, its ETH value increases monotonically over time. If the LayerZero message is delayed (network congestion, relayer downtime, etc.), the cached `rate` is lower than the true current rate. The division `amountAfterFee * 1e18 / rsETHToETHrate` then yields a larger `rsETHAmount` than the depositor's ETH actually backs.

The `RSETHPoolV3` family mints wrsETH directly via `wrsETH.mint(msg.sender, rsETHAmount)`, so there is no finite pool inventory to exhaust — the over-issuance is unbounded by pool balance. [7](#0-6) 

---

### Impact Explanation

**High — Theft of unclaimed yield / protocol insolvency.**

rsETH accrues yield continuously. A stale rate that is even 0.1% below the true rate causes every depositor during the stale window to receive ~0.1% more wrsETH than their ETH is worth. When those depositors later unwrap wrsETH → rsETH and redeem on L1, they extract more ETH than they deposited. The shortfall is borne by all other rsETH holders (dilution of backing). At scale or with a significantly stale rate, this constitutes protocol insolvency.

---

### Likelihood Explanation

**Medium.**

LayerZero message delivery is not guaranteed to be instantaneous. Network congestion, relayer downtime, or a temporary L1/L2 connectivity issue can leave `lastUpdated` hours or days behind. An attacker can passively monitor the on-chain `lastUpdated` value and deposit only when the rate is stale, requiring no privileged access and no front-running. The entry path is the standard public `deposit()` function available to any user.

---

### Recommendation

Add a configurable maximum staleness period and revert in `getRate()` if the cached rate is too old:

```solidity
uint256 public constant MAX_RATE_STALENESS = 24 hours;

function getRate() external view returns (uint256) {
    require(
        block.timestamp - lastUpdated <= MAX_RATE_STALENESS,
        "Rate is stale"
    );
    return rate;
}
```

Alternatively, expose `lastUpdated` as part of the oracle interface and enforce the staleness check at the pool level before computing `rsETHAmount`.

---

### Proof of Concept

1. At `T=0`, `lzReceive` sets `rate = 1.010e18` (rsETH/ETH) and `lastUpdated = T`.
2. LayerZero relayer goes offline. True rsETH rate on L1 grows to `1.015e18` by `T+48h`.
3. Attacker calls `RSETHPoolV3.deposit{value: 10 ether}("")` at `T+48h`.
4. Pool calls `CrossChainRateReceiver.getRate()` → returns stale `1.010e18`.
5. `rsETHAmount = 10e18 * 1e18 / 1.010e18 ≈ 9.901 wrsETH` (correct would be `≈ 9.852 wrsETH`).
6. Attacker receives `≈ 0.049 wrsETH` excess — redeemable for `≈ 0.05 ETH` profit per 10 ETH deposited.
7. Repeated across many depositors or with a larger stale window, the protocol's wrsETH supply becomes unbacked. [8](#0-7) [4](#0-3) [9](#0-8) [10](#0-9)
