Audit Report

## Title
Push-Transfer Revert in `_processWithdrawalCompletion` Permanently Freezes Funds and Blocks `sweepRemainingAssets` - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`_processWithdrawalCompletion` decrements `unlockedWithdrawalsCount[asset]` and deletes the withdrawal request before calling `_transferAsset`. If `_transferAsset` reverts (ETH recipient has a reverting `receive()`, or an LST recipient is blacklisted), the entire transaction rolls back, leaving the counter permanently elevated. Because rsETH is already burned and the underlying asset already redeemed from the vault at `unlockQueue` time, the user's funds are permanently frozen inside `LRTWithdrawalManager` with no admin escape hatch, and `sweepRemainingAssets` is permanently blocked for that asset.

## Finding Description
`unlockQueue` burns rsETH and redeems the underlying asset from the vault before any per-user transfer occurs: [1](#0-0) 

`_unlockWithdrawalRequests` increments the counter for each unlocked request: [2](#0-1) 

`_processWithdrawalCompletion` decrements the counter and deletes the request, then calls `_transferAsset`: [3](#0-2) 

`_transferAsset` for ETH uses a raw `call` that reverts on failure: [4](#0-3) 

For ERC20 LSTs it uses `safeTransfer`, which reverts on blacklist or pause: [5](#0-4) 

Because all state changes (decrement, delete) and the transfer are in one atomic transaction, a revert in `_transferAsset` rolls back everything. The counter stays elevated, the request persists, and `sweepRemainingAssets` is permanently gated: [6](#0-5) 

The operator-callable `completeWithdrawalForUser` routes through the same path and provides no bypass: [7](#0-6) 

No function exists to forcibly clear a stuck `withdrawalRequests` entry or manually decrement `unlockedWithdrawalsCount`.

## Impact Explanation
**Critical — Permanent freezing of funds.** After `unlockQueue`, the user's rsETH is irreversibly burned and the redeemed asset sits inside `LRTWithdrawalManager`. If the push transfer to the user always reverts, those funds are permanently locked with no recovery path. Additionally, **Medium — Permanent freezing of unclaimed yield**: `sweepRemainingAssets` is permanently blocked for the affected asset, freezing any excess balance that should flow to the treasury.

## Likelihood Explanation
**ETH path (deliberate, low-cost attack):** Any unprivileged user deploys a contract with `receive() external payable { revert(); }`, calls `initiateWithdrawal` for ETH with the minimum rsETH amount, and waits for the operator to call `unlockQueue`. The withdrawal is then permanently uncompletable. Cost is negligible. The developer NatDoc comment at L191 ("potential gas grief scenarios are non-impactful for ETH") addresses only gas griefing, not a permanently reverting recipient that freezes the counter and the sweep function.

**LST path (accidental or deliberate):** A user whose address is later added to a compliance blacklist of a supported LST will have their unlocked withdrawal permanently stuck.

Both paths are reachable by an unprivileged external caller with no special preconditions beyond holding a minimal rsETH balance.

## Recommendation
1. **Add an admin rescue function** that can forcibly delete a stuck `withdrawalRequests` entry, decrement `unlockedWithdrawalsCount[asset]`, and redirect the asset to the treasury or a recovery address.
2. **Alternatively**, wrap `_transferAsset` in a try/catch (or use a non-reverting low-level call) and, on failure, credit the amount to a per-user claimable mapping so the user can pull it later — preventing a push-transfer failure from blocking the global counter and the sweep function.

## Proof of Concept
1. Attacker deploys `Blocker` with `receive() external payable { revert(); }`.
2. `Blocker` calls `LRTWithdrawalManager.initiateWithdrawal(ETH_TOKEN, minAmount, "")` — rsETH is escrowed, `assetsCommitted[ETH]` increases.
3. Operator calls `unlockQueue(ETH_TOKEN, ...)` — rsETH is burned via `burnFrom`, ETH is redeemed from `unstakingVault` into `LRTWithdrawalManager`, `unlockedWithdrawalsCount[ETH]` becomes `1`.
4. Anyone calls `completeWithdrawal(ETH_TOKEN, ...)` for `Blocker` — `_transferAsset` calls `payable(Blocker).call{value:amount}("")`, `Blocker` reverts, entire tx reverts; counter stays at `1`, request persists.
5. Operator calls `completeWithdrawalForUser(ETH_TOKEN, Blocker, "")` — same revert.
6. `hasUnlockedWithdrawals(ETH_TOKEN)` returns `true` permanently; `sweepRemainingAssets(ETH_TOKEN)` reverts with `PendingWithdrawalsExist` forever.
7. The ETH amount is permanently locked inside `LRTWithdrawalManager` with no recovery path.

Foundry test outline: deploy `Blocker`, fork mainnet, call `initiateWithdrawal`, advance blocks past `withdrawalDelayBlocks`, call `unlockQueue`, assert `unlockedWithdrawalsCount[ETH] == 1`, call `completeWithdrawal` and expect revert, assert `unlockedWithdrawalsCount[ETH]` still `== 1`, call `sweepRemainingAssets` and expect `PendingWithdrawalsExist`.
