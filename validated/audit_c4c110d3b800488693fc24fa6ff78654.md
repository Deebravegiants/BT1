### Title
Cross-Token Withdrawal Allows Draining of Specific Token Reserves - (File: `contracts/L2/RsETHTokenWrapper.sol`)

### Summary
`RsETHTokenWrapper` supports multiple allowed tokens but issues a single fungible `wrsETH` token with no per-token accounting. A depositor of `tokenA` can freely call `withdraw(tokenB, amount)` to redeem their `wrsETH` for `tokenB`, draining reserves deposited by `tokenB` depositors.

### Finding Description

The `_deposit` function mints `wrsETH` 1:1 for any allowed token: [1](#0-0) 

The `_withdraw` function only checks that the requested `_asset` is in `allowedTokens`, then burns `wrsETH` and transfers that asset â€” with no check that the caller ever deposited that specific token: [2](#0-1) 

There is no per-token deposit ledger anywhere in the contract. The `allowedTokens` mapping only tracks whether a token is permitted, not how much of it any user deposited: [3](#0-2) 

### Impact Explanation

The concrete attack path:
1. `userA` calls `deposit(tokenA, 100e18)` â†’ receives 100e18 `wrsETH`
2. `userB` calls `deposit(tokenB, 100e18)` â†’ receives 100e18 `wrsETH`
3. `userA` calls `withdraw(tokenB, 100e18)` â†’ burns 100e18 `wrsETH`, receives 100e18 `tokenB` (a different token than deposited)
4. `userB` calls `withdraw(tokenB, 100e18)` â†’ reverts; contract holds 0 `tokenB`

`userB` cannot redeem `wrsETH` for the token they deposited. They may be able to withdraw `tokenA` instead, so total value in the system is not destroyed â€” matching the scoped impact: **Low. Contract fails to deliver promised returns, but doesn't lose value.**

### Likelihood Explanation

Likelihood is high once a second allowed token is added via `addAllowedToken`. The `TIMELOCK_ROLE` can add tokens, and the `reinitialize` function already demonstrates the design intent to support multiple tokens. No special permissions are needed by the attacker â€” any ordinary user can execute this. [4](#0-3) 

### Recommendation

Track per-token reserves explicitly. One approach: maintain a `mapping(address token => uint256 reserve)` that increments on `_deposit` and decrements on `_withdraw`, and revert in `_withdraw` if the per-token reserve is insufficient. Alternatively, issue separate wrapper tokens per underlying asset rather than a single fungible `wrsETH`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity 0.8.27;

// Fuzz/unit test (local, no mainnet)
function testCrossTokenWithdrawal() public {
    // Setup: two allowed tokens, both deposited
    wrapper.deposit(address(tokenA), 100e18); // userA deposits tokenA
    vm.prank(userB);
    wrapper.deposit(address(tokenB), 100e18); // userB deposits tokenB

    // userA withdraws tokenB (different from what they deposited)
    wrapper.withdraw(address(tokenB), 100e18);
    assertEq(tokenB.balanceOf(address(this)), 100e18); // userA received tokenB

    // userB can no longer redeem their wrsETH for tokenB
    vm.prank(userB);
    vm.expectRevert(); // ERC20: transfer amount exceeds balance
    wrapper.withdraw(address(tokenB), 100e18);
}
```

The root cause is at [2](#0-1)  â€” `_withdraw` has no mechanism to enforce that the caller redeems the same token type they deposited.
