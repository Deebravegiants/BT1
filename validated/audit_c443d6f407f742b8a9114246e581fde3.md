### Title
Unsafe `int256` to `uint256` Cast of Chainlink Price Allows Massive rsETH Minting - (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` casts the raw `int256 price` returned by `latestRoundData()` directly to `uint256` without any negativity check. If Chainlink returns a negative answer, the cast produces a value near `type(uint256).max`, causing every downstream consumer — including rsETH minting — to use a wildly inflated asset price.

---

### Finding Description

In `contracts/oracles/ChainlinkPriceOracle.sol`, `getAssetPrice()` calls `latestRoundData()` and immediately casts the signed result:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

Chainlink's `latestRoundData()` returns `int256 answer`, which is a signed type. The Chainlink specification does not guarantee the value is always positive; negative values can occur during circuit-breaker events, feed misconfiguration, or for certain synthetic/inverse feeds. Solidity's two's-complement representation means `uint256(-1) == 2**256 - 1 ≈ 1.16e77`.

There is no guard such as `require(price > 0)` before the cast. [2](#0-1) 

A secondary instance exists in `contracts/oracles/RSETHPriceFeed.sol`, where the ETH/USD `answer` (also `int256`) is multiplied by `int256(RS_ETH_ORACLE.rsETHPrice())` and returned as the composite price. A negative ETH/USD answer propagates a negative composite answer to any downstream consumer that casts it to `uint256`. [3](#0-2) 

---

### Impact Explanation

The inflated `assetPrice` flows directly into rsETH minting in `LRTDepositPool`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [4](#0-3) 

With `getAssetPrice()` returning ~`2**256 - 1`, the numerator overflows (Solidity 0.8 reverts on overflow) **or**, if the overflow is avoided by the division, the attacker receives an astronomically large rsETH amount for a minimal deposit. Either outcome is critical: revert-on-overflow permanently freezes deposits for all users, while a successful inflated mint constitutes direct theft of protocol funds by draining the rsETH supply against real ETH backing.

The `LRTOracle._getTotalEthInProtocol()` also calls `getAssetPrice()` per supported asset, so a negative price would corrupt the TVL calculation used to compute `rsETHPrice`, cascading into every subsequent price update. [5](#0-4) 

**Impact: Critical** — direct theft of user funds (inflated rsETH mint) or permanent freezing of deposits (revert on overflow).

---

### Likelihood Explanation

Chainlink feeds for LSTs (stETH/ETH, rETH/ETH, etc.) are unlikely to return negative values under normal mainnet conditions. However:
- The `int256` return type is part of the Chainlink interface contract and is not bounded to positive values by the ABI.
- Any future addition of a synthetic, inverse, or misconfigured feed would immediately trigger this path.
- The `updatePriceFeedFor` function allows the LRT manager to point an asset to any address implementing `AggregatorV3Interface`, including feeds that can return negative values. [6](#0-5) 

**Likelihood: Medium** — not routine under current feeds, but the code path is reachable and the attack surface widens with each new supported asset.

---

### Recommendation

1. Add a positivity check before casting in `ChainlinkPriceOracle.getAssetPrice()`:
   ```solidity
   (, int256 price,,,) = priceFeed.latestRoundData();
   require(price > 0, "ChainlinkOracle: non-positive price");
   return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
   ```
2. In `RSETHPriceFeed`, validate that the underlying ETH/USD `answer` is positive before computing the composite price.

---

### Proof of Concept

1. Assume a Chainlink feed for a supported LST returns `price = -1` (e.g., due to a circuit-breaker or misconfigured feed).
2. `ChainlinkPriceOracle.getAssetPrice(asset)` computes `uint256(-1) * 1e18 / 10**8` ≈ `1.16e69`.
3. A depositor calls `LRTDepositPool.depositAsset(asset, 1 wei, 0, "")`.
4. `getRsETHAmountToMint(asset, 1)` computes `(1 * 1.16e69) / rsETHPrice`. With `rsETHPrice ≈ 1e18`, this yields `≈ 1.16e51` rsETH to mint for 1 wei of deposit.
5. The attacker receives `1.16e51` rsETH, redeemable against the protocol's real ETH backing, draining all depositor funds. [1](#0-0) [7](#0-6)
