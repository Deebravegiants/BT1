### Title
`MerkleDistributor.claim()` Does Not Validate `msg.sender` as Account Owner, Enabling Forced Fee-Bearing Claims - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

---

### Summary

The `claim()` function in `MerkleDistributor` accepts an arbitrary `account` address without verifying that `msg.sender == account`. Any caller who possesses a valid merkle proof for a victim's address can trigger the claim on their behalf, forcing the victim to receive `claimableAmount - fee` tokens instead of the full entitled `claimableAmount`. The fee is permanently diverted to the protocol treasury, constituting theft of unclaimed yield.

---

### Finding Description

`MerkleDistributor.claim()` takes `account` as a caller-supplied parameter and performs no ownership check. The merkle proof is verified against `account`, not `msg.sender`, so any party who obtains the victim's proof data (typically published off-chain in a public merkle tree) can submit the claim on their behalf. [1](#0-0) 

The function then deducts `fee = claimableAmount * feeInBPS / 10_000` and sends only `amountToSend = claimableAmount - fee` to `account`, routing the fee to `protocolTreasury`. [2](#0-1) 

The victim's `userClaims` state is updated to reflect the full `cumulativeAmount`, permanently preventing any future claim for the same index. [3](#0-2) 

By contrast, `KernelMerkleDistributor._processClaim()` explicitly enforces `if (account != msg.sender) revert Unauthorized()`, demonstrating that the protocol is aware of this requirement and intentionally added it in the newer distributor but omitted it in the generic `MerkleDistributor`. [4](#0-3) 

---

### Impact Explanation

**High. Theft of unclaimed yield.**

When `feeInBPS > 0` (up to 10% per `MAX_FEE_IN_BPS = 1000`), an attacker can force any victim's claim, causing the victim to permanently lose up to 10% of their entitled token allocation. The lost tokens are irrecoverable because the claim state is updated to `cumulativeAmount`, blocking any re-claim. The victim's yield is stolen and redirected to the treasury. [5](#0-4) 

---

### Likelihood Explanation

**Medium.**

The merkle proof data for all eligible accounts is necessarily published off-chain (e.g., IPFS, protocol website) so that users can construct their own claims. Any observer can read this data and replay it against any account. The only precondition is that `feeInBPS > 0`, which is a configurable parameter within the contract's normal operating range.

---

### Recommendation

Add an ownership check at the start of `claim()`:

```solidity
if (account != msg.sender) revert Unauthorized();
```

This mirrors the protection already present in `KernelMerkleDistributor._processClaim()`. [4](#0-3) 

---

### Proof of Concept

1. Protocol deploys `MerkleDistributor` with `feeInBPS = 500` (5%).
2. Merkle tree is published; victim Alice is entitled to `1000e18` tokens at `index = 5`.
3. Attacker reads Alice's `(index, account, cumulativeAmount, merkleProof)` from the public tree.
4. Attacker calls `MerkleDistributor.claim(5, Alice, 1000e18, aliceProof)`.
5. Contract computes `fee = 50e18`, sends `950e18` to Alice and `50e18` to treasury. [6](#0-5) 

6. `userClaims[Alice].cumulativeAmount = 1000e18` — Alice can never reclaim the 50e18 fee.
7. Alice permanently loses 50e18 tokens she was entitled to, with no recourse.
