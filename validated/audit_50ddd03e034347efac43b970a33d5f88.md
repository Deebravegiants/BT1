Audit Report

## Title
Stale `rsETHPrice` Read in Deposit Path Allows Depositors to Capture Accrued Yield — (File: `contracts/LRTDepositPool.sol`, `contracts/LRTOracle.sol`)

## Summary
`LRTOracle.rsETHPrice` is a stored state variable updated only when `updateRSETHPrice()` is explicitly called by an off-chain keeper. Neither `LRTDepositPool.depositETH` nor `depositAsset` calls `updateRSETHPrice()` before reading `rsETHPrice` to compute the rsETH mint amount. When yield has accrued since the last keeper update, the stored price is lower than the true current price, causing depositors to receive more rsETH than they are entitled to — permanently diluting the yield that rightfully belongs to existing rsETH holders.

## Finding Description
`LRTOracle` stores the rsETH/ETH exchange rate in `rsETHPrice` (L28), updated only via `_updateRsETHPrice()` called from `updateRSETHPrice()` (L87–89) or `updateRSETHPriceAsManager()` (L94–96). Neither function is called atomically inside any user-facing deposit function.

**Deposit path:**
`depositETH` (L76–93) and `depositAsset` (L99–118) both call `_beforeDeposit` (L648–670), which calls `getRsETHAmountToMint` (L506–521). The mint amount is computed as:
```
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```
(L520). No call to `updateRSETHPrice()` precedes this read.

**Why existing guards are insufficient:**
The `pricePercentageLimit` check inside `_updateRsETHPrice()` (L252–266) means that if the price increase since the last update exceeds the configured threshold, a call to `updateRSETHPrice()` by a non-manager will revert with `PriceAboveDailyThreshold`. This makes the stale-price window worse: during periods of significant yield accrual, normal users are blocked from refreshing the price, yet deposits continue to execute against the stale lower price. The downside-protection pause (L270–282) only triggers on price decreases, leaving the upside-stale case entirely unmitigated.

**Withdrawal path note:** `initiateWithdrawal` also reads `rsETHPrice` without updating it (L168, L593), but a stale lower price in this path harms the withdrawer (they receive less asset), not existing holders. The primary theft vector is the deposit path.

## Impact Explanation
**High — Theft of unclaimed yield.**

Between keeper updates, staking and EigenLayer rewards increase the protocol's ETH TVL, but `rsETHPrice` remains at its last stored (lower) value. A depositor calling `depositETH`/`depositAsset` during this window receives `(amount × assetPrice) / staleLowerPrice` rsETH — more than the correct amount at the true current price. The excess rsETH represents yield that has already accrued to existing rsETH holders. After the keeper calls `updateRSETHPrice()`, the new price is permanently lower than it would have been absent the over-minting, and existing holders cannot recover the diluted yield. This matches the allowed impact: **High — Theft of unclaimed yield**.

## Likelihood Explanation
**Medium.** The keeper updates `rsETHPrice` periodically, not on every block. Any depositor acting in the window between yield accrual and the next keeper update benefits from the stale price. No special privileges, front-running, or adversarial setup is required — the mis-accounting occurs for every deposit made while the price is stale. The `pricePercentageLimit` guard can extend this window by blocking non-manager price refreshes when yield has accumulated beyond the threshold. The longer the keeper interval or the higher the yield rate, the larger the exploitable gap.

## Recommendation
- **Short term:** At the start of `depositETH` and `depositAsset`, call `ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice()` before computing the mint amount. Since `updateRSETHPrice()` is `public` and `whenNotPaused`, this can be done atomically. Note that the `pricePercentageLimit` guard may cause this call to revert for non-manager callers when price movement is large; this edge case must be handled (e.g., by allowing the deposit to proceed with the last known price only when the update reverts due to threshold, or by ensuring the keeper interval is short enough that the threshold is never breached between updates).
- **Long term:** Redesign `LRTOracle` so that `rsETHPrice` is computed on-the-fly as a `view` function using live TVL and supply data, eliminating the possibility of a stale read entirely.

## Proof of Concept
1. Protocol state: TVL = 1000 ETH, rsETH supply = 1000, `rsETHPrice = 1.0e18`. Keeper last ran 12 hours ago.
2. Staking rewards add 10 ETH to TVL (now 1010 ETH). True price = 1.01e18. Keeper has not yet called `updateRSETHPrice()`, so `rsETHPrice` remains `1.0e18`.
3. Attacker calls `depositETH{value: 100 ether}(0, "")`. `getRsETHAmountToMint` computes `(100e18 * 1e18) / 1.0e18 = 100e18` rsETH minted.
4. At the true price of 1.01e18, the attacker should receive `100e18 / 1.01e18 ≈ 99.01e18` rsETH. They receive `100e18` — approximately `0.99e18` rsETH of excess.
5. Keeper calls `updateRSETHPrice()`. New supply = 1100 rsETH, TVL = 1110 ETH → new price = `1110/1100 ≈ 1.009e18` instead of the correct `1010/1000 = 1.01e18` (pre-deposit). Existing holders permanently lose ~0.001 ETH per rsETH of accrued yield to the attacker.
