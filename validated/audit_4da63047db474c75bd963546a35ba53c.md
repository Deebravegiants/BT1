### Title
Rogue or Buggy Extension Cannot Be Removed and Permanently Freezes Pool Operations — (`metric-core/contracts/ExtensionCalling.sol`, `metric-core/contracts/libraries/CallExtension.sol`)

---

### Summary

Extensions are baked into a pool at creation time with no post-deployment removal or replacement path. `CallExtension.callExtension` propagates every revert from an extension directly to the caller with no try/catch isolation. If any registered extension begins reverting — due to a bug, an external dependency failure, or an upgrade by its owner — every pool action that routes through that hook is permanently frozen. Because `removeLiquidity` calls `_beforeRemoveLiquidity` unconditionally and carries no `whenNotPaused` guard, LP funds can be locked with no admin escape hatch.

---

### Finding Description

**Extension immutability.** Extensions are supplied to `MetricOmmPoolFactory.createPool` and written into immutable storage slots (`EXTENSION_1` … `EXTENSION_7`). A grep across the entire repository finds zero occurrences of `setExtension`, `updateExtension`, `removeExtension`, or `replaceExtension`. The only post-deployment admin actions available are fee changes, pause/unpause, and a timelocked price-provider rotation — none of which touch extensions.

**Revert propagation without isolation.** `_callExtensionsInOrder` iterates the packed order word and calls each extension through `CallExtension.callExtension`:

```solidity
// metric-core/contracts/ExtensionCalling.sol
function _callExtensionsInOrder(uint256 order, bytes memory data) private {
    if (order == 0) return;
    while (true) {
        uint256 extensionIndex = order & 0x7;
        if (extensionIndex == 0) break;
        address extension = _extensionAddress(extensionIndex);
        if (extension == address(0)) revert PanicEmptyExtension();
        CallExtension.callExtension(extension, data);   // ← no try/catch
        order >>= 3;
    }
}
```

`CallExtension.callExtension` re-raises the extension's revert verbatim:

```solidity
// metric-core/contracts/libraries/CallExtension.sol
function callExtension(address extension, bytes memory data) internal {
    (bool success, bytes memory result) = extension.call(data);
    if (!success) {
        if (result.length > 0) {
            assembly ("memory-safe") { revert(add(result, 32), mload(result)) }
        }
        revert ExtensionCallFailed();
    }
    ...
}
```

There is no isolation layer. A single reverting extension call aborts the entire transaction.

**Affected entry points.** `MetricOmmPool.removeLiquidity` calls `_beforeRemoveLiquidity` and `_afterRemoveLiquidity` before and after the state change. Critically, `removeLiquidity` carries only `nonReentrant` — it has **no** `whenNotPaused` modifier. This means:

- Pausing the pool (the only admin lever) does **not** unblock `removeLiquidity`.
- If an extension registered for `BEFORE_REMOVE_LIQUIDITY_ORDER` or `AFTER_REMOVE_LIQUIDITY_ORDER` begins reverting, LP positions are permanently unwithdrawable.

`swap` is similarly blocked for any extension on `BEFORE_SWAP_ORDER` or `AFTER_SWAP_ORDER`, and `addLiquidity` for the corresponding add-liquidity hooks.

**Realistic trigger paths.**

1. An extension is an upgradeable proxy; its owner (distinct from the pool admin) upgrades the implementation to always revert.
2. An extension reads from an external oracle or registry that itself becomes unavailable or starts reverting.
3. A latent bug in the extension is triggered by a specific pool state (e.g., a bin index boundary, a zero-share bin, an overflow in extension-side math).

None of these require the pool admin to act maliciously; the pool admin has no power to fix the situation once it occurs.

---

### Impact Explanation

**Critical / High.** If an extension registered for `beforeRemoveLiquidity` or `afterRemoveLiquidity` begins reverting, all LP positions in the pool are permanently frozen. Users cannot recover their token0 or token1 principal. The pool admin cannot remove the extension, cannot bypass it, and cannot use the pause mechanism to restore withdrawals (since `removeLiquidity` ignores the pause flag). This is a direct, permanent loss of user principal with no recovery path.

---

### Likelihood Explanation

**Medium.** Extensions are external contracts with no enforced immutability requirement. Any extension that is an upgradeable proxy, reads from a third-party feed, or contains a latent state-dependent bug can trigger this condition without any privileged action by the pool admin or factory owner. The condition is permanent once triggered.

---

### Recommendation

1. **Add a force-remove path for extensions.** Introduce a factory-level function (with appropriate timelock and access control) that allows the pool admin to zero out a specific extension slot and remove it from all order words. This mirrors the `force` flag recommended in the external report.

2. **Alternatively, wrap extension calls in try/catch.** Change `_callExtensionsInOrder` to catch extension reverts, emit a `ExtensionCallFailed(extensionIndex)` event, and continue. This prevents a single bad extension from bricking the pool, at the cost of silent extension failures (which may be acceptable for non-critical hooks).

3. **At minimum, ensure `removeLiquidity` is never blocked by extensions.** Either remove extension hooks from the withdrawal path, or wrap only the withdrawal-path extension calls in try/catch so that LP funds can always be recovered.

---

### Proof of Concept

```solidity
// 1. Deploy pool with an upgradeable extension registered for beforeRemoveLiquidity.
//    Extension initially behaves correctly.

// 2. Extension owner upgrades the implementation to always revert:
contract MaliciousExtensionV2 is IMetricOmmExtensions {
    function beforeRemoveLiquidity(...) external pure returns (bytes4) {
        revert("bricked");
    }
    // ... other hooks pass through
}

// 3. Any LP who calls pool.removeLiquidity(...) now gets:
//    CallExtension.callExtension -> extension.call -> revert("bricked")
//    -> assembly revert propagated -> transaction reverts
//    LP funds are permanently locked.

// 4. Pool admin calls factory.pausePool(pool) — has no effect on removeLiquidity.
// 5. No setExtension / removeExtension function exists anywhere in the factory.
//    There is no recovery path.
