### Title
User-Controlled `BlockConfirmations` Not Validated Against a Minimum — Allows MPC Signing of Unfinalized Bitcoin Transactions - (File: `crates/contract/src/lib.rs`, `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

The `verify_foreign_transaction` contract entry point accepts a `BitcoinRpcRequest` whose `confirmations: BlockConfirmations` field is a raw `u64` supplied entirely by the unprivileged caller. Neither the contract nor the node enforces a minimum value. A caller can set `confirmations: 0`, causing every MPC node to accept any confirmed Bitcoin transaction (even one with a single confirmation) as meeting the finality threshold, and the network will produce and return a threshold signature attesting to that transaction. A bridge contract that trusts the MPC signature without independently re-checking the `confirmations` value embedded in the signed payload will process a transfer that can subsequently be reversed by a Bitcoin reorg, enabling a double-spend.

---

### Finding Description

`BitcoinRpcRequest` carries a caller-supplied `confirmations: BlockConfirmations(u64)` field. [1](#0-0) 

The ABI schema explicitly permits `0` as a valid value (`"minimum": 0.0`). [2](#0-1) 

The contract's `verify_foreign_transaction` entry point performs no validation of this field — it only checks that the chain is supported, the domain exists, and the deposit/gas requirements are met: [3](#0-2) 

On the node side, `BitcoinInspector::extract` uses the caller-supplied threshold directly in a `<=` comparison against the RPC-reported confirmation count: [4](#0-3) 

When the threshold is `0`, the condition `0 <= transaction_block_confirmation` is trivially true for any confirmed transaction (any `confirmations >= 1`). The node then proceeds to verify canonical-chain membership and sign the payload. The `confirmations` value is committed into the signed payload as part of `ForeignChainRpcRequest`, but the contract never enforces a floor. [5](#0-4) 

This is structurally identical to the Pyth analog: a quality/finality metric that is user-controlled and should be validated against a protocol-enforced minimum is instead passed through unchecked.

---

### Impact Explanation

**Impact: High** — Cross-chain double-spend via forged foreign-chain verification.

A bridge contract built on top of `verify_foreign_transaction` receives a threshold MPC signature over `(BitcoinRpcRequest { confirmations: 0, … }, observed_values)`. If the bridge contract does not independently re-validate the `confirmations` field embedded in the signed payload (a common omission when the MPC network is treated as a trusted oracle), it will release funds on NEAR for a Bitcoin transaction that has only one confirmation. A subsequent Bitcoin reorg (or RBF replacement) reverses the Bitcoin-side payment while the NEAR-side release has already been finalized, producing a double-spend.

---

### Likelihood Explanation

**Likelihood: Low** — Requires a Bitcoin reorg or successful RBF replacement after the MPC signature is obtained, and requires the downstream bridge contract to omit a `confirmations` floor check. Bitcoin reorgs of 1–2 blocks occur rarely but are not unprecedented; RBF is more common. The attacker entry path (calling `verify_foreign_transaction` with `confirmations: 0`) requires no privilege.

---

### Recommendation

Enforce a protocol-level minimum `BlockConfirmations` value. Options in increasing order of flexibility:

1. **Constant in the contract**: Reject any `BitcoinRpcRequest` where `confirmations < MIN_BITCOIN_CONFIRMATIONS` (e.g., 6) at the top of `verify_foreign_transaction`.
2. **Per-chain configuration**: Store a `min_confirmations` alongside the foreign-chain whitelist entry and validate against it on each request.
3. **Node-side enforcement**: Have each MPC node refuse to sign if the requested threshold is below a configured minimum, analogous to the Pyth `minConfidenceRatio` recommendation.

Option 1 is the simplest and closes the gap at the contract layer where it is most auditable:

```rust
const MIN_BITCOIN_CONFIRMATIONS: u64 = 6;

// inside verify_foreign_transaction, before enqueue_yield_request:
if let ForeignChainRpcRequest::Bitcoin(ref btc) = request.request {
    if btc.confirmations.0 < MIN_BITCOIN_CONFIRMATIONS {
        env::panic_str("BitcoinConfirmationsTooLow");
    }
}
```

The same pattern should be applied to `EvmFinality::Latest`, which is the weakest EVM finality level and is equally susceptible to reorgs. [6](#0-5) 

---

### Proof of Concept

1. Attacker controls a Bitcoin address `A` and a NEAR account.
2. Attacker sends 1 BTC from `A` to bridge deposit address `B` (tx `T`).
3. After 1 confirmation, attacker calls:
   ```json
   verify_foreign_transaction({
     "request": {
       "Bitcoin": {
         "tx_id": "<T>",
         "confirmations": 0,
         "extractors": ["BlockHash"]
       }
     },
     "domain_id": <foreign_tx_domain>,
     "payload_version": "V1"
   })
   ```
   with the minimum 1 yoctoNEAR deposit.
4. The contract accepts the request (no `confirmations` floor check). [7](#0-6) 
5. Each MPC node calls `getrawtransaction` for `T`, finds `confirmations = 1 >= 0`, passes the check, verifies canonical chain membership, and contributes a signature share. [4](#0-3) 
6. The threshold signature is returned to the bridge contract, which releases NEAR-side funds.
7. Attacker broadcasts a conflicting Bitcoin transaction (double-spend or RBF), reversing `T`.
8. Attacker has received NEAR-side funds while recovering the BTC.

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }

    async fn extract(
        &self,
        transaction: BitcoinTransactionHash,
        block_confirmations_threshold: BlockConfirmations,
        extractors: Vec<BitcoinExtractor>,
    ) -> Result<Vec<BitcoinExtractedValue>, ForeignChainInspectionError> {
        let request_parameters = GetRawTransactionArgs {
            transaction_hash: TransportBitcoinTransactionHash::from(*transaction),
            verbose: VERBOSE_RESPONSE,
        };

        // TODO(#1978): add retry mechanism if the error from the request is transient
        let rpc_response: GetRawTransactionVerboseResponse = self
            .client
            .request(GET_RAW_TRANSACTION_METHOD, &request_parameters)
            .await?;

        let transaction_block_confirmation = rpc_response.confirmations.into();
        let enough_block_confirmations =
            block_confirmations_threshold <= transaction_block_confirmation;

        if !enough_block_confirmations {
            return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations {
                expected: block_confirmations_threshold,
                got: transaction_block_confirmation,
            });
        }

        self.verify_block_is_canonical(rpc_response.blockhash)
            .await?;

        let extracted_values = extractors
            .iter()
            .map(|extractor| extractor.extract_value(&rpc_response))
            .collect();

        Ok(extracted_values)
    }
