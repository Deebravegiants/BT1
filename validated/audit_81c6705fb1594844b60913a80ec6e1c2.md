### Title
Empty Extractor List Bypasses Foreign-Chain Data Extraction, Enabling Forged Verification Payloads - (File: crates/contract/src/lib.rs, crates/node/src/providers/verify_foreign_tx/sign.rs)

### Summary

Neither the on-chain `verify_foreign_transaction` contract method nor the off-chain `execute_foreign_chain_request` node function validates that the caller-supplied `extractors` list is non-empty. An unprivileged caller can submit a `VerifyForeignTransactionRequestArgs` with `extractors: vec![]` for any real foreign-chain transaction. The MPC network will verify the transaction's existence and finality, then produce a threshold signature over a payload whose `values` field is an empty vector. The resulting signature is cryptographically valid but carries zero extracted data, enabling a caller to present it to a NEAR bridge contract as "proof" of a foreign-chain event without the MPC network having attested to any event content.

### Finding Description

The `verify_foreign_transaction` entry point in the contract performs only two checks before enqueuing the request: gas/deposit preconditions and chain-support membership. [1](#0-0) 

There is no check that `request.request.extractors()` is non-empty. The request is stored verbatim and later picked up by MPC nodes.

On the node side, `execute_foreign_chain_request` dispatches to the per-chain inspector with whatever extractor list the caller provided: [2](#0-1) 

Each chain-specific branch iterates `request.extractors` and calls `inspector.extract(tx_id, finality, extractors)`. When `extractors` is empty the inspector still performs all finality and canonical-chain RPC calls (the transaction is genuinely verified to exist and be final), but returns an empty `Vec<ExtractedValue>`. This is confirmed by the existing test suite: [3](#0-2) [4](#0-3) [5](#0-4) 

The node then constructs and signs:

```rust
ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.clone(),   // includes tx_id and extractors: []
    values: vec![],             // no extracted data
})
``` [6](#0-5) 

The resulting `payload_hash` and threshold signature are returned on-chain as a `VerifyForeignTransactionResponse`. The signature is fully valid — the MPC network signed exactly what it was asked to sign — but the signed payload attests to nothing about the transaction's content.

The SDK's `ForeignChainSignatureVerifier` does compare `payload_hash` against a locally-reconstructed expected payload: [7](#0-6) 

However, this protection exists only in the SDK helper. The MPC contract and node impose no such constraint, and any bridge contract that does not use the SDK or that constructs its expected payload incorrectly (e.g., also using `values: []`) will accept the forged proof.

The design document explicitly states that each extractor must return exactly one typed value and that the extracted values are the mechanism by which NEAR contracts react to foreign-chain events: [8](#0-7) 

Allowing zero extractors violates this invariant at the protocol level.

### Impact Explanation

An attacker who finds any real, finalized transaction on a supported foreign chain (Bitcoin, EVM chains, Starknet, etc.) can obtain a valid threshold MPC signature over `(tx_id, [])`. This signature can be presented to any NEAR bridge contract that uses `verify_foreign_transaction` as its trust anchor. If that contract does not independently reconstruct the expected payload with the correct non-empty `values` and compare hashes, it will accept the empty-values signature as proof that the transaction occurred and contained the expected event data — when the MPC network attested to nothing of the sort. This enables invalid bridge execution: an attacker can claim bridge funds or trigger bridge actions by presenting a signature that proves only that some transaction exists, not that it contained the required transfer event, log, or program invocation. This maps to the "forged foreign-chain verification that causes invalid bridge execution" impact class.

### Likelihood Explanation

The attack requires no privileged access. Any NEAR account can call `verify_foreign_transaction` with a deposit of 1 yoctoNEAR and an empty `extractors` list. The attacker only needs a real transaction on the target chain (trivially obtained — any transaction hash works). The MPC network will process the request and return a valid signature. The attack is therefore reachable by any unprivileged external caller with no threshold collusion required.

### Recommendation

Add a non-empty extractor check in the contract before enqueuing the request:

```rust
// In verify_foreign_transaction, after chain-support check:
let extractor_count = request.request.extractor_count();
if extractor_count == 0 {
    env::panic_str("verify_foreign_transaction: extractors list must not be empty");
}
```

Alternatively, enforce this at the type level by replacing `Vec<EvmExtractor>` with a `NonEmptyVec<EvmExtractor>` (analogous to the existing `NonEmptyBTreeSet` in `crates/near-mpc-bounded-collections`): [9](#0-8) 

Add a matching guard in `execute_foreign_chain_request` on the node side as defense-in-depth. Update the `EvmRpcRequest`, `BitcoinRpcRequest`, `StarknetRpcRequest`, and all other chain request structs to use the non-empty collection type: [10](#0-9) 

### Proof of Concept

**Actors:**
- Alice: operator of a NEAR bridge contract that uses `verify_foreign_transaction` to gate token releases.
- Bob: attacker with no privileged access.

**Setup:**
1. Alice deploys a bridge contract on NEAR. It calls `verify_foreign_transaction` and, on receiving the MPC response, releases tokens if the `payload_hash` in the response matches `SHA256(borsh((request, [])))` — a bug where Alice forgot to include expected extracted values in her local reconstruction.
2. The MPC contract is in Running state with a configured `ForeignTx` domain and Bitcoin whitelisted.

**Exploit:**
1. Bob finds any real, finalized Bitcoin transaction `tx_id = [0xbb; 32]` (e.g., from a block explorer).
2. Bob calls `verify_foreign_transaction` with:
   ```json
   {
     "request": {
       "request": { "Bitcoin": { "tx_id": "0xbb...bb", "confirmations": 1, "extractors": [] } },
       "domain_id": <foreign_tx_domain_id>,
       "payload_version": 1
     }
   }
   ```
3. MPC nodes verify the transaction exists and is final, then sign `ForeignTxSignPayload::V1 { request, values: [] }`.
4. Bob receives a valid `VerifyForeignTransactionResponse` with `payload_hash = SHA256(borsh((request, [])))` and a threshold ECDSA signature.
5. Bob submits this response to Alice's bridge contract.
6. Alice's contract reconstructs `expected_hash = SHA256(borsh((request, [])))`, finds it matches `response.payload_hash`, verifies the signature, and releases tokens — despite the MPC network having attested to nothing about the transaction's content.

**Expected outcome without fix:** Bob receives tokens without the underlying Bitcoin transaction containing the required transfer event.

**Expected outcome with fix:** Step 2 fails at the contract level with "extractors list must not be empty."

                }

            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),

impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }

### Extractors

Extractors are strongly typed, bounded operations defined by the MPC protocol implementation.

* Each `Extractor` identifies a built-in extractor and its parameters.
* Each extractor must return exactly one `ExtractedValue`.
* Extractors must be deterministic and specified independently of provider-specific JSON formatting.
* Initial extractor set is intentionally limited and isolated to avoid ambiguity. We'll add more as we uncover more use cases and needs.
