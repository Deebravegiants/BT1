### Title
Unvalidated CKD Response Accepted for Legacy `AppPublicKey` Variant Allows Single Byzantine Participant to Forge Derived Key Output - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd()` performs **no cryptographic validation** of the submitted `CKDResponse` when the request uses the legacy `AppPublicKey` (privately verifiable) variant. A single Byzantine attested participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary forged `CKDResponse` for any pending CKD request, and the contract will accept and deliver it to the user. This bypasses the threshold requirement for confidential key derivation.

---

### Finding Description

In `respond_ckd()`, the contract branches on the `app_public_key` variant of the pending `CKDRequest`:

```rust
// crates/contract/src/lib.rs, lines 675–682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO VALIDATION
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For the `AppPublicKeyPV` (publicly verifiable) variant, the contract calls `ckd_output_check()` to verify the response against the BLS12-381 master public key on-chain. For the `AppPublicKey` (legacy, privately verifiable) variant, the arm is a no-op — the contract performs **zero validation** of the submitted `CKDResponse` fields (`big_y`, `big_c`).

The contract then unconditionally resolves all queued yields with the unvalidated response:

```rust
// crates/contract/src/lib.rs, lines 684–688
pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

The only gate before this branch is `assert_caller_is_attested_participant_and_protocol_active()`, which requires the caller to be a current attested participant — not a threshold quorum. [3](#0-2) 

By contrast, `respond()` for signature requests always validates the submitted signature cryptographically before resolving yields: [4](#0-3) 

The `AppPublicKey` variant is explicitly documented as still supported (legacy): [5](#0-4) 

---

### Impact Explanation

A single Byzantine attested participant (strictly below the signing threshold) can:

1. Observe any pending `AppPublicKey` CKD request in the contract's `pending_ckd_requests` map.
2. Call `respond_ckd(request, forged_response)` with arbitrary `big_y` and `big_c` values.
3. The contract accepts the forged response and delivers it to all queued callers via `promise_yield_resume`.

The user receives a forged derived key. Since `AppPublicKey` is "privately verifiable," the user can only detect the forgery by independently verifying with their app secret key — but the contract has already committed the forged output on-chain and resolved the yield. The attacker can simultaneously compute the correct derived key (using their legitimate key share) and use it, while the victim operates with a wrong key. This constitutes **unauthorized confidential key derivation output without the required participant authorization**, bypassing the threshold-signature requirement for CKD.

**Impact class: Critical** — matches "Unauthorized transaction execution, threshold signature issuance, or confidential key derivation output without the required participant authorization."

---

### Likelihood Explanation

- The `AppPublicKey` variant is the legacy path and remains fully supported in the contract ABI.
- Any single attested participant (one node operator) can exploit this unilaterally — no collusion required.
- The attacker needs only to be an active, attested participant in the MPC network, which is a realistic adversarial assumption for a Byzantine participant below threshold.
- The pending CKD request map is readable on-chain, so the attacker can trivially identify targets.

---

### Recommendation

Apply the same on-chain cryptographic validation to the `AppPublicKey` variant as is applied to `AppPublicKeyPV`. If the `AppPublicKey` variant cannot be validated on-chain (because the G2 component of the app public key is not available), the contract should either:

1. **Deprecate and remove** the `AppPublicKey` variant, requiring all new requests to use `AppPublicKeyPV` which supports on-chain verification via `ckd_output_check`.
2. Or, if backward compatibility is required, **require threshold quorum** (multiple attested participants must submit matching responses) before resolving yields for `AppPublicKey` requests, analogous to how threshold signing works for `respond()`.

The root cause is the same as the referenced external report: off-chain generated data (`CKDResponse`) is accepted by the contract without sufficient on-chain validation, trusting a single participant's submission rather than enforcing the threshold invariant.

---

### Proof of Concept

1. Alice submits a `request_app_private_key` request using the legacy `AppPublicKey` variant (a single BLS12-381 G1 point).
2. The request is stored in `pending_ckd_requests` on-chain.
3. Byzantine participant Eve (a single attested node) calls `respond_ckd` with the correct `request` key but a forged `CKDResponse { big_y: [0u8; 48], big_c: [0u8; 48] }`.
4. The contract executes lines 675–688: the `AppPublicKey` arm is a no-op, `resolve_yields_for` is called, and Alice's yield is resumed with the forged response.
5. Alice's callback receives the forged derived key. The contract state is now cleared (`pending_ckd_requests` entry removed), so no legitimate node can overwrite the result.
6. Eve, knowing the correct derived key from her own key share, can use it while Alice operates with a wrong key — enabling theft of any assets Alice subsequently controls with the derived key. [6](#0-5) [7](#0-6)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

### Submitting a confidential key derivation (ckd) request

Users can submit a ckd request to the MPC network via the
`request_app_private_key` endpoint of this contract. A **deposit of 1
yoctonear is required** (see [Deposit requirement](#deposit-requirement)).

The ckd request takes the following arguments:

- `derivation_path` (String): the derivation path (used to derive different keys from the same account).
- `app_public_key`: the ephemeral public key for the CKD request. Two formats are supported:
  - **Privately verifiable** (legacy): a single G1 point, e.g. `"bls12381g1:<base58>"` or `{"AppPublicKey": "bls12381g1:<base58>"}`.
  - **Publicly verifiable**: a pair of points `(pk1, pk2) = (a·G1, a·G2)`, passed as `{"AppPublicKeyPV": {"pk1": "bls12381g1:<base58>", "pk2": "bls12381g2:<base58>"}}`. This allows anyone to verify the encrypted result on-chain without the app's secret key.
- `domain_id` (integer): identifies the master key to use for deriving the ckd, and must correspond to bls12381.

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
