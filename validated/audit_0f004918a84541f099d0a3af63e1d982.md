Audit Report

## Title
Missing Slippage Protection on `removeLiquidity` Exposes LPs to Sandwich Attacks on Active-Bin Withdrawals - (`metric-core/contracts/MetricOmmPool.sol`, `metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary
`MetricOmmPool.removeLiquidity` accepts no minimum-output parameters and no deadline. Token amounts returned to the LP are computed as a straight proportional share of the **live** bin balances at execution time. Because the active bin holds both `token0BalanceScaled` and `token1BalanceScaled` simultaneously, and swaps continuously alter their ratio, an attacker can sandwich the removal transaction to force the LP to receive a materially worse token composition with no on-chain guard to revert.

## Finding Description
`MetricOmmPool.removeLiquidity` signature carries only `owner`, `salt`, `deltas`, and `extensionData` — no `minAmount0Out`, `minAmount1Out`, or `deadline`: [1](#0-0) 

Inside `LiquidityLib.removeLiquidity`, the output amounts are computed directly from the current `binState.token0BalanceScaled` and `binState.token1BalanceScaled` with no floor check: [2](#0-1) 

Tokens are then transferred unconditionally to `owner`: [3](#0-2) 

`BinState` stores both `token0BalanceScaled` and `token1BalanceScaled` in the active bin simultaneously: [4](#0-3) 

Swaps within the active bin directly mutate both balances — buying token0 drains `token0BalanceScaled` and fills `token1BalanceScaled`, and vice versa: [5](#0-4) [6](#0-5) 

A grep of all `metric-periphery/**/*.sol` files for `removeLiquidity` returns zero matches — there is no periphery wrapper analogous to `MetricOmmPoolLiquidityAdder` for liquidity removal: [7](#0-6) 

The `msg.sender == owner` enforcement means the LP must submit the transaction directly to the public mempool, making it observable and front-runnable: [8](#0-7) 

## Impact Explanation
An LP removing shares from the active bin receives whatever token mix exists at execution time. A sandwich attacker executes a large swap before the LP's transaction to shift the active bin's cursor, converting most of one token into the other. The LP's burn then executes against the manipulated balances, returning a composition dominated by the cheaper-to-acquire token. The attacker back-runs to restore the price, capturing the round-trip profit minus fees. The LP suffers a direct loss of principal with no on-chain recourse. This constitutes a direct loss of LP assets above Sherlock thresholds for sufficiently large positions.

## Likelihood Explanation
`removeLiquidity` is a public, permissionless function callable by any position owner. Active-bin positions are the normal case for LPs who deposit across a range that includes the current price. The attack requires only two standard swaps in the same block, which is routine on any chain with a public mempool. No privileged access, governance action, or malicious token is required. The attack is repeatable on every removal transaction.

## Recommendation
Add minimum-output parameters and a deadline to `removeLiquidity` at the pool level, or provide a periphery wrapper (analogous to `MetricOmmPoolLiquidityAdder`) that enforces them before delegating to the core:

```solidity
function removeLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    uint256 minAmount0Out,
    uint256 minAmount1Out,
    uint256 deadline,
    bytes calldata extensionData
) external returns (uint256 amount0Removed, uint256 amount1Removed);
```

After `LiquidityLib.removeLiquidity` returns, revert if `amount0Removed < minAmount0Out || amount1Removed < minAmount1Out || block.timestamp > deadline`. This mirrors the protection already present for swaps via `amountOutMinimum` in `MetricOmmSimpleRouter`: [9](#0-8) 

## Proof of Concept
```
Setup:
  - Pool: token0 (WETH) / token1 (USDC), active bin = 0
  - Bin 0 holds 500 WETH (token0) + 500 USDC (token1)
  - LP holds 10,000 shares out of 10,000 total in bin 0
  - LP submits: removeLiquidity(owner, salt, [{binIdx:0, shares:10000}], "")
    expecting ~500 WETH + ~500 USDC

Front-run (attacker, same block):
  1. pool.swap(zeroForOne=false, largeAmount)  // buy token0, sell token1
     → token0BalanceScaled drops to ~100, token1BalanceScaled rises to ~900

LP tx executes:
  2. amount0Scaled = 100 * 10000 / 10000 = 100  → ~100 WETH
     amount1Scaled = 900 * 10000 / 10000 = 900  → ~900 USDC
     LP receives 100 WETH + 900 USDC instead of 500 WETH + 500 USDC
     At WETH > USDC price, LP receives less total value.
     No minAmount0Out guard → transaction does not revert.

Back-run (attacker):
  3. pool.swap(zeroForOne=true) to restore price, capturing the spread.
```

The LP has no recourse because `removeLiquidity` accepts no minimum-output or deadline arguments. [10](#0-9)
