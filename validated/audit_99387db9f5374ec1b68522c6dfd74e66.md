### Title
Stale Governance Votes Persist After Resharing Cancellation, Bypassing Threshold Admission Gate for New Participants - (File: `crates/contract/src/state/resharing.rs`)

---

### Summary

When a resharing is cancelled via `vote_cancel_resharing`, the `previous_running_state` is returned directly with its `parameters_votes` map intact. Votes cast for re-proposals *during* the resharing phase are never cleared. In the subsequent running state, those stale votes satisfy the threshold-admission guard that is supposed to prevent new (non-current) participants from voting until enough current participants have explicitly approved the proposal in the current epoch. This breaks the governance invariant that every resharing epoch requires fresh, explicit approval from the required threshold of current participants.

---

### Finding Description

**Root cause — `vote_cancel_resharing` does not reset `parameters_votes`**

`ResharingContractState::vote_cancel_resharing` clones `previous_running_state` and returns it directly, only patching `previously_cancelled_resharing_epoch_id`: [1](#0-0) 

By contrast, every other path that produces a `RunningContractState` goes through `RunningContractState::new`, which explicitly resets `parameters_votes` to the empty default: [2](#0-1) 

**How stale votes accumulate**

During resharing, participants call `ResharingContractState::vote_new_parameters`, which delegates to `previous_running_state.process_new_parameters_proposal`. That function stores each vote in `previous_running_state.parameters_votes`: [3](#0-2) [4](#0-3) 

**How stale votes bypass the admission gate**

Back in the running state, `process_new_parameters_proposal` enforces that a signer who is *not* a current participant may only vote after `threshold` current participants have already voted for the same proposal: [5](#0-4) 

`n_votes` counts entries in `parameters_votes` whose key is a current participant: [6](#0-5) 

If current participants A and B voted for re-proposal P2 during the resharing phase, their entries remain in `parameters_votes` after cancellation. When new participant E calls `vote_new_parameters` for P2 from the running state, `n_votes(P2, current_participants)` returns 2 (A + B stale) ≥ threshold = 2, so E is admitted without A or B having voted in the current epoch.

**Stale votes also inflate the consensus count**

The final consensus check counts *all* entries in `parameters_votes` that match the proposal: [7](#0-6) 

Stale entries from A and B count toward `n_votes`, so consensus can be reached with fewer fresh votes than the protocol requires.

---

### Impact Explanation

A new participant E, who voted for a re-proposal during a resharing that was subsequently cancelled, can:

1. **Bypass the threshold admission gate** — vote for a new resharing proposal without A and B having explicitly approved it in the current epoch, because their stale votes satisfy the `n_votes < threshold` guard.
2. **Inflate the consensus count** — A and B's stale entries count toward the `new_num_participants == n_votes` check, so a resharing can reach consensus with fewer fresh current-participant approvals than the governance threshold demands.

Concretely, if current participants are {A, B, C} with threshold = 2 and the proposed set is {A, B, C, E} (4 participants), C + E can drive the contract into resharing using A and B's stale votes — even if A and B voted to *cancel* the previous resharing and have not re-approved the proposal. This breaks the participant-state safety invariant that every epoch transition requires explicit, epoch-scoped approval from the required threshold of current participants.

This maps to the **Medium** allowed impact: participant-state manipulation that breaks a production safety/accounting invariant.

---

### Likelihood Explanation

The preconditions are realistic in a production MPC network:

- Resharing attempts fail and are cancelled regularly (network partitions, node upgrades, etc.).
- During resharing, participants routinely vote for re-proposals when the original proposed set becomes unavailable.
- A new participant who voted during the resharing phase is a legitimate actor with a valid TEE attestation; no privileged access is required.
- The attacker only needs to call the public `vote_new_parameters` endpoint from their own NEAR account after cancellation.

---

### Recommendation

In `vote_cancel_resharing`, reset `parameters_votes` before returning the running state, mirroring what `RunningContractState::new` does:

```rust
let mut previous_running_state = self.previous_running_state.clone();
// Clear stale re-proposal votes so the next resharing epoch
// requires fresh, explicit approval from current participants.
previous_running_state.parameters_votes = ThresholdParametersVotes::default();
previous_running_state.previously_cancelled_resharing_epoch_id =
    Some(prospective_epoch_id);
Some(previous_running_state)
``` [8](#0-7)

### Proof of Concept

**Setup**: Running state, participants {A, B, C}, governance threshold = 2.

1. All three vote for P1 (proposed set {A, B, C, D}); resharing starts.
2. During resharing, A and B vote for re-proposal P2 (proposed set {A, B, C, E}, 4 participants). Their votes are stored in `previous_running_state.parameters_votes`.
3. `n_votes(P2, {A,B,C})` = 2 ≥ threshold = 2, so E is admitted; E votes for P2. Count = 3 < 4; consensus not yet reached.
4. A and B vote to cancel resharing (2 votes ≥ threshold = 2). `vote_cancel_resharing` returns `previous_running_state` **with A, B, E's votes for P2 still present**.
5. From the running state, E calls `vote_new_parameters(P2)`:
   - `AuthenticatedAccountId::new(current_participants)` fails for E → admission check triggered.
   - `n_votes(P2, {A,B,C})` = 2 (A stale + B stale) ≥ threshold = 2 → **E is admitted without A or B voting in this epoch**.
   - E's entry replaces its stale entry; count = 3 (A stale + B stale + E fresh).
6. C calls `vote_new_parameters(P2)`: count = 4 = `new_num_participants` → **consensus reached; resharing starts**.

    pub fn vote_cancel_resharing(&mut self) -> Result<Option<RunningContractState>, Error> {
        let previous_running_participants = self.previous_running_state.parameters.participants();
        let authenticated_candidate = AuthenticatedAccountId::new(previous_running_participants)?;
        self.cancellation_requests.insert(authenticated_candidate);

        let cancellation_votes_count = self.cancellation_requests.len() as u64;
        let previous_running_threshold = self.previous_running_state.parameters.threshold();

        let threshold_cancellation_votes_reached: bool =
            cancellation_votes_count >= previous_running_threshold.value();

        let running_state = if threshold_cancellation_votes_reached {
            let mut previous_running_state = self.previous_running_state.clone();
            let prospective_epoch_id = self.prospective_epoch_id();
            previous_running_state.previously_cancelled_resharing_epoch_id =
                Some(prospective_epoch_id);

            Some(previous_running_state)
        } else {
            None
        };

        Ok(running_state)
    }

    /// return the number of votes for `proposal` cast by members of `participants`
    pub fn n_votes(
        &self,
        proposal: &ProposedThresholdParameters,
        participants: &Participants,
    ) -> u64 {
        u64::try_from(
            self.proposal_by_account
                .iter()
                .filter(|&(acc, prop)| {
                    participants
                        .participants()
                        .iter()
                        .any(|(acc_id, _, _)| acc.get() == acc_id)
                        && prop == proposal
                })
                .count(),
        )
        .expect("usize should never fail to convert to u64 on wasm32")
    }
