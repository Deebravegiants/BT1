### Title
Queue-Filling DoS on `verify_foreign_transaction` via Caller-Agnostic Request Key — (File: `crates/contract/src/lib.rs`)

### Summary

The `verify_foreign_transaction` function uses a **caller-agnostic** pending-request key. Any unprivileged account can fill the bounded fan-out queue (cap = 128) for a specific foreign transaction, causing every subsequent legitimate caller's request to fail with `PendingRequestQueueFull` and blocking bridge operations for that transaction.

### Finding Description

**Root cause — caller-agnostic key in `verify_foreign_transaction`**

`verify_foreign_transaction` constructs its pending-request key from `VerifyForeignTransactionRequest`, which contains only `domain_id`, `payload_version`, and the chain-specific `ForeignChainRpcRequest` (e.g., `tx_id`). The caller's account ID is **never included**. [1](#0-0) 

In contrast, `sign()` explicitly binds the key to the caller via `SignatureRequest::new(..., &predecessor, ...)`: [2](#0-1) 

The test suite itself acknowledges the caller-agnostic behavior of `verify_foreign_transaction`: [3](#0-2) 

**The cap that enables the DoS**

`push_pending_yield` enforces a hard ceiling of `MAX_PENDING_REQUEST_FAN_OUT = 128`. Exceeding it causes an immediate contract panic (`PendingRequestQueueFull`): [4](#0-3) 

The comment in that file explicitly names the attacker model: *"Without a cap, an attacker could enqueue enough duplicates to make `respond*` run out of gas and strand every queued caller."* The cap prevents the gas-exhaustion variant but does not prevent a targeted queue-saturation attack against a specific foreign transaction.

**Attack scenario (direct analog to permit frontrunning)**

| Step | Permit frontrunning (external report) | NEAR MPC analog |
|---|---|---|
| 1 | Alice signs a permit off-chain and submits `supplyWithPermit` | Alice submits `verify_foreign_transaction` for a specific Bitcoin tx |
| 2 | Attacker sees the tx in the mempool, calls `permit()` directly | Attacker submits 128 identical `verify_foreign_transaction` requests for the same tx (cost: 128 yoctonear + gas ≈ negligible) |
| 3 | Alice's tx reverts: nonce already consumed | Alice's tx panics: `PendingRequestQueueFull` |
| 4 | Alice must use a different code path | Alice must wait for the queue to drain (MPC response or timeout) and retry |
| 5 | — | Attacker immediately refills the queue after each MPC response, sustaining the DoS indefinitely |

### Impact Explanation

`verify_foreign_transaction` is the on-chain entry point for the Omnibridge inbound flow (foreign chain → NEAR). Blocking it for a specific transaction prevents the bridge from completing, temporarily locking user funds on the foreign chain. The attacker can sustain the block indefinitely at negligible cost by refilling the queue after each MPC response drains it. This is a targeted, sustained disruption of the request lifecycle for specific bridge operations — not a generic network-level DoS.

**Impact class:** Medium — request-lifecycle manipulation that breaks production safety/accounting invariants for bridge operations.

### Likelihood Explanation

- **Cost to attacker:** 128 × 1 yoctonear + gas ≈ effectively zero.
- **Information required:** The foreign-chain `tx_id` is public on the foreign chain and visible in the NEAR mempool.
- **Privileged access required:** None. Any NEAR account can call `verify_foreign_transaction`.
- **Sustainability:** After each MPC response drains the queue, the attacker can immediately resubmit 128 requests before the legitimate user retries.

### Recommendation

Include the caller's `predecessor_account_id` in the `VerifyForeignTransactionRequest` key, mirroring the `sign()` design:

```rust
// In args_into_verify_foreign_tx_request (or at the call site in verify_foreign_transaction):
let request = VerifyForeignTransactionRequest {
    caller: env::predecessor_account_id(),  // add this field
    domain_id: request.domain_id,
    payload_version: request.payload_version,
    request: request.request,
};
```

This ensures each caller's queue slot is independent, so an attacker cannot fill another user's queue.

```rust
/// Maximum number of concurrent yield-resume promises that can be queued for a single
/// request key (i.e. the number of duplicate submissions whose responses fan out from
/// one MPC reply).
///
/// The ceiling is needed because `respond*` drains the entire queue in one call: every
/// queued yield triggers a host-side `promise_yield_resume`, paid for out of the
/// responder's 300 TGas budget. Without a cap, an attacker could enqueue enough
/// duplicates to make `respond*` run out of gas and strand every queued caller.
///
/// 128 is validated empirically by the sandbox test
/// `test_contract_request_duplicate_requests_fan_out`, which fills the queue to this
/// cap across all four signature schemes and confirms `respond*` drains it inside its
/// 300 TGas budget.
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
```
