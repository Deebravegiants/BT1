Audit Report

## Title
L2 Oracle Staleness Enables Structural Over-Issuance of wrsETH Relative to L1 rsETH Backing — (`contracts/pools/RSETHPoolV2ExternalBridge.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`, `contracts/pools/RSETHPool.sol`)

## Summary

The L2 pool contracts mint wrsETH immediately at the L2 oracle rate on every `deposit()` call, while the corresponding L1 rsETH is only minted later at the then-current L1 `LRTOracle.rsETHPrice`. Because the L2 oracle (`InterimRSETHOracle` or `CrossChainRateReceiver`) is push-based and has no staleness enforcement in the pool contracts, any depositor who deposits while the L2 oracle lags the L1 rate receives excess wrsETH that is permanently unbacked. Repeated deposits across the staleness window accumulate a structural deficit constituting protocol insolvency.

## Finding Description

**Root cause:** No staleness check exists in any pool `deposit()` path before minting wrsETH.

**Step 1 — L2 deposit mints wrsETH at L2 oracle rate.**

`RSETHPoolV2ExternalBridge.deposit()` calls `viewSwapRsETHAmountAndFee()`, which computes `rsETHAmount = amountAfterFee * 1e18 / getRate()`, where `getRate()` delegates to `IOracle(rsETHOracle).getRate()`. The result is immediately minted to the depositor with no freshness check. [1](#0-0) 

**Step 2 — The L2 oracle is a push-based contract with no on-chain staleness enforcement.**

Two oracle implementations are used. `InterimRSETHOracle` stores a manually-set `rate` with no timestamp tracking whatsoever — `getRate()` simply returns `rate`. [2](#0-1) 

`CrossChainRateReceiver` stores `lastUpdated` but the pool contracts never read it; `getRate()` returns the stored `rate` regardless of age. [3](#0-2) 

`MultiChainRateProvider.updateRate()` is permissionless but push-based — it must be called externally to propagate the L1 rate to L2. Until it is called, the L2 receiver holds a stale value. [4](#0-3) 

**Step 3 — L1 mints rsETH at the live L1 rate, not the L2 oracle rate.**

`L1Vault.depositETHForL1VaultETH()` calls `lrtDepositPool.getRsETHAmountToMint(ETH_IDENTIFIER, balanceOfETH)`, which uses the live `LRTOracle.rsETHPrice` — a value that has increased since the L2 oracle was last updated. [5](#0-4) 

**Step 4 — rsETHPrice is monotonically increasing.**

`LRTOracle.updateRSETHPrice()` computes `newRsETHPrice = totalETHInProtocol / rsethSupply`. As yield accrues, this value only increases, so a stale L2 oracle always shows a lower rate than the current L1 rate. [6](#0-5) 

**The invariant break (concrete example):**

| Variable | Value |
|---|---|
| L2 oracle rate (stale) | 1.05 ETH/rsETH |
| L1 oracle rate (current) | 1.10 ETH/rsETH |
| ETH deposited | 100 ETH |
| wrsETH minted on L2 | 100 / 1.05 ≈ **95.24 rsETH** |
| rsETH minted on L1 | 100 / 1.10 ≈ **90.91 rsETH** |
| **Unbacked over-issuance** | **4.33 rsETH** |

The `dailyMintLimit` modifier bounds per-day magnitude but does not prevent the structural deficit from accumulating day after day whenever the oracle lags. [7](#0-6) 

## Impact Explanation

**Critical — Protocol insolvency.** The total wrsETH outstanding on L2 structurally exceeds the rsETH that can ever be bridged back to cover it. wrsETH is redeemable for L1 rsETH via `RsETHTokenWrapper`, but the L1 rsETH pool backing it is smaller than the wrsETH supply. The deficit is permanent and accumulates with every deposit made during a staleness window. At scale, the protocol cannot honor all wrsETH redemptions, constituting direct protocol insolvency — an allowed Critical impact.

## Likelihood Explanation

rsETH's exchange rate increases continuously as yield accrues. Any push-based or manually-updated L2 oracle will periodically lag the L1 rate — this is a normal operating condition, not an exceptional event. No oracle compromise, privileged access, or victim mistake is required. A depositor only needs to observe that the L2 oracle rate (readable on-chain via `getRate()`) is below the L1 rate (readable on-chain via `LRTOracle.rsETHPrice()`) and call `deposit()` during that window. The gap between oracle updates can be hours to days, giving ample time for exploitation. The attack is repeatable across every staleness window.

## Recommendation

1. **Add a staleness check in `deposit()`**: read `lastUpdated` from the oracle receiver and revert if `block.timestamp - lastUpdated > MAX_ORACLE_AGE` (e.g., 1 hour).
2. **Enforce a rate ceiling**: reject minting if the L2 oracle rate deviates from a trusted cross-chain rate by more than a configurable threshold (e.g., 0.5%).
3. **Reconcile at bridge time**: track total wrsETH minted per bridge cycle and compare against the rsETH actually minted on L1 after `depositETHForL1VaultETH()` completes; pause L2 minting if a deficit is detected.
4. **Use a pull-based cross-chain oracle** (e.g., LayerZero `lzRead` or Chainlink CCIP rate feed) so the L2 oracle is updated atomically with L1 rate changes rather than relying on periodic pushes.

## Proof of Concept

**Foundry fork test plan:**

```solidity
// Fork test against mainnet + L2 fork
function test_staleOracleOverIssuance() public {
    // 1. Read current L1 rate
    uint256 l1Rate = ILRTOracle(L1_ORACLE).rsETHPrice(); // e.g. 1.10e18

    // 2. Set L2 InterimRSETHOracle to a stale (lower) rate
    vm.prank(managerRole);
    InterimRSETHOracle(l2Oracle).setRate(l1Rate * 95 / 100); // 1.045e18

    // 3. Depositor deposits 100 ETH on L2
    address depositor = address(0xBEEF);
    vm.deal(depositor, 100 ether);
    vm.prank(depositor);
    l2Pool.deposit{value: 100 ether}("ref");
    uint256 wrsETHMinted = wrsETH.balanceOf(depositor);
    // wrsETHMinted ≈ 100e18 / 1.045e18 ≈ 95.69 rsETH

    // 4. Operator bridges ETH to L1
    vm.prank(bridger);
    l2Pool.bridgeAssets(100 ether, 99 ether, nativeFee);

    // 5. Manager mints rsETH on L1 at live rate
    vm.prank(manager);
    l1Vault.depositETHForL1VaultETH();
    uint256 rsETHMintedOnL1 = rsETH.balanceOf(address(l1Vault));
    // rsETHMintedOnL1 ≈ 100e18 / 1.10e18 ≈ 90.91 rsETH

    // 6. Assert structural deficit
    assertGt(wrsETHMinted, rsETHMintedOnL1,
        "L2 wrsETH > L1 rsETH: protocol insolvency confirmed");
    // Deficit ≈ 4.78 rsETH on 100 ETH deposited
}
```

The test requires no privileged exploit — the depositor is unprivileged. The only precondition is that the L2 oracle has not been updated since the last L1 rate increase, which is a routine operating condition.
