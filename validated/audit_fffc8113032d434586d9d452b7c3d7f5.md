### Title
Sub-Transactions Bypass Reference Script Fee in Dijkstra Era — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Ledger.hs`)

---

### Summary

In the Dijkstra era, the minimum-fee calculation for a top-level transaction uses only `txNonDistinctRefScriptsSize` (covering the top-level transaction's inputs), while the reference-script size limit is enforced using `batchNonDistinctRefScriptsSize` (covering the top-level transaction **and all its sub-transactions**). Sub-transactions are processed via `updateUTxOStateNoFees` and are never subject to a fee check. An unprivileged sender can therefore embed sub-transactions that consume large reference scripts without paying the reference-script fee component that would be mandatory for the same scripts used in a top-level transaction.

---

### Finding Description

**Vulnerability class:** fee bypass through an alternative code path — direct analog of the external report's "ragequit vs. formal distribution" pattern.

**Standard path (top-level transaction):**

`getMinFeeTxUtxo` for `DijkstraEra` is bound to `getConwayMinFeeTxUtxo`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
getMinFeeTxUtxo = getConwayMinFeeTxUtxo
``` [1](#0-0) 

`getConwayMinFeeTxUtxo` computes the reference-script fee using only the top-level transaction's inputs:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/UTxO.hs
getConwayMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ txNonDistinctRefScriptsSize utxo tx
``` [2](#0-1) 

`txNonDistinctRefScriptsSize` only unions the top-level transaction's `referenceInputs` and `inputs`:

```haskell
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
``` [3](#0-2) 

**Alternative path (sub-transactions):**

Sub-transactions are processed by `SUBUTXO` via `updateUTxOStateNoFees` — no fee is collected and no fee check is performed:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs
if isValid
  then do
    newState <-
      Shelley.updateUTxOStateNoFees pp utxoState txBody certState ...
    pure $ newState & utxosDonationL <>~ txBody ^. treasuryDonationTxBodyL
  else
    pure utxoState
``` [4](#0-3) 

The `SUBUTXO` rule contains no call to `feesOK`, `validateFeeTooSmallUTxO`, or any reference-script fee check.

**The size limit uses the batch function (top-level + sub-transactions):**

`validateAllRefScriptSize` in the Dijkstra LEDGER rule uses `batchNonDistinctRefScriptsSize`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Ledger.hs
validateAllRefScriptSize pp utxo tx =
  let totalRefScriptSize = batchNonDistinctRefScriptsSize utxo tx
      maxRefScriptSizePerTx = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerTxG
   in failureUnless (totalRefScriptSize <= maxRefScriptSizePerTx) $ ...
``` [5](#0-4) 

`batchNonDistinctRefScriptsSize` aggregates reference-script sizes across the top-level transaction **and all sub-transactions**:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
``` [6](#0-5) 

**The asymmetry:**

| Check | Function used | Covers sub-tx ref scripts? |
|---|---|---|
| Fee calculation | `txNonDistinctRefScriptsSize` | **No** |
| Size limit | `batchNonDistinctRefScriptsSize` | **Yes** |

---

### Impact Explanation

The reference-script fee (`minFeeRefScriptCostPerByte`, tiered via `tierRefScriptFee`) was introduced specifically to prevent DDoS attacks where large, expensive-to-deserialize scripts are used at low cost (documented in ADR 009). The attack occurred on mainnet on June 25, 2024.

By routing reference-script usage through sub-transactions, an attacker can:

1. Submit a top-level transaction with **no** reference scripts → pays **zero** reference-script fee.
2. Embed N sub-transactions each referencing large scripts from UTxO outputs.
3. The total batch size is bounded by `maxRefScriptSizePerTx`, but the reference-script fee component of `getConwayMinFeeTx` is computed only over the top-level transaction's inputs, so it remains **zero**.

The node must still deserialize and validate all reference scripts in all sub-transactions, incurring the same CPU/memory cost as if those scripts appeared in a top-level transaction, but the attacker pays only the linear byte-size fee (`a * txSize + b`) for the serialized sub-transaction bytes — not the exponentially-tiered reference-script fee.

This matches the **Medium** impact: *attacker-controlled transactions modify fees outside design parameters*.

---

### Likelihood Explanation

- Any unprivileged transaction sender can craft a Dijkstra-era top-level transaction with sub-transactions.
- No special keys, governance majority, or privileged access is required.
- The attack is cheap to mount: the attacker pays only the linear size fee, not the tiered reference-script fee.
- The Dijkstra era is in the production codebase and will be deployed on mainnet.

---

### Recommendation

Override `getMinFeeTxUtxo` for `DijkstraEra` to use `batchNonDistinctRefScriptsSize` instead of `txNonDistinctRefScriptsSize`, so that the reference-script fee accounts for all reference scripts used across the top-level transaction and all its sub-transactions:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
getMinFeeTxUtxo pp tx utxo =
  getMinFeeTx pp tx $ batchNonDistinctRefScriptsSize utxo tx
```

This mirrors the existing pattern where `validateAllRefScriptSize` already uses `batchNonDistinctRefScriptsSize` for the size-limit check, ensuring fee and size-limit enforcement are symmetric.

---

### Proof of Concept

1. Deploy a UTxO output containing a large Plutus script (e.g., 100 KiB) as a reference script.
2. Construct a Dijkstra-era top-level transaction with:
   - One ordinary input (to satisfy `validateInputSetEmptyUTxO`)
   - No reference inputs at the top level
   - One sub-transaction whose `referenceInputsTxBodyL` points to the 100 KiB reference-script UTxO
3. Submit the transaction. The ledger accepts it because:
   - `validateAllRefScriptSize` passes (100 KiB ≤ `maxRefScriptSizePerTx`)
   - `validateFeeTooSmallUTxO` passes because `getConwayMinFeeTxUtxo` computes `txNonDistinctRefScriptsSize` over the top-level transaction only (0 bytes of reference scripts) → reference-script fee = 0
4. The node deserializes and validates the 100 KiB script for the sub-transaction at zero reference-script fee cost to the attacker.

Repeating this with many sub-transactions (up to the `maxRefScriptSizePerTx` batch limit) per block allows sustained high-cost validation at minimal fee, re-enabling the DDoS vector that the reference-script fee was designed to close. [1](#0-0) [2](#0-1) [6](#0-5) [5](#0-4) [4](#0-3) [7](#0-6)
