### Title
Hard-Coded Reference Script Size Limits and Fee Parameters in Conway Era Bypass Governance and Can Permanently Freeze Funds - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/PParams.hs`)

### Summary
In the Conway era, four critical reference script parameters — `ppMaxRefScriptSizePerTxG`, `ppMaxRefScriptSizePerBlockG`, `ppRefScriptCostMultiplierG`, and `ppRefScriptCostStrideG` — are hard-coded as compile-time constants in the `ConwayEraPParams ConwayEra` instance, completely bypassing the on-chain governance update mechanism. No governance action (PParamUpdate, DRep vote, SPO vote, or committee approval) can change these values in Conway era. Any UTxO locked by a script that requires reference scripts totaling more than 200 KiB per transaction is permanently unspendable in Conway era, with recovery requiring a hard fork to Dijkstra.

### Finding Description

The `ConwayEraPParams ConwayEra` instance in `eras/conway/impl/src/Cardano/Ledger/Conway/PParams.hs` implements four members of the `ConwayEraPParams` type class as constant `SimpleGetter` values rather than as updatable protocol parameters:

```haskell
ppMaxRefScriptSizePerTxG    = L.to . const $ 200 * 1024          -- 204,800 bytes
ppMaxRefScriptSizePerBlockG = L.to . const $ 1024 * 1024         -- 1,048,576 bytes
ppRefScriptCostMultiplierG  = L.to . const . fromJust $ boundRational 1.2
ppRefScriptCostStrideG      = L.to . const $ knownNonZeroBounded @25_600
``` [1](#0-0) 

These four values are not stored in `ConwayPParams` and have no corresponding `PParamUpdate` lens, unlike every other Conway protocol parameter (e.g., `ppMinFeeRefScriptCostPerByte` at tag 33, `ppGovActionDeposit` at tag 30, etc.). [2](#0-1) 

The hard-coded values are consumed in three enforcement sites:

**1. Per-transaction size enforcement** (`validateRefScriptSize` in `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs`):

```haskell
validateRefScriptSize pp utxo tx =
  let totalRefScriptSize = txNonDistinctRefScriptsSize utxo tx
      maxRefScriptSizePerTx = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerTxG
   in failureUnless (totalRefScriptSize <= maxRefScriptSizePerTx) $ ...
``` [3](#0-2) 

**2. Per-block size enforcement** (`validateBodyRefScriptsSizeTooBig` in `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Bbody.hs`):

```haskell
maxSize = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerBlockG
``` [4](#0-3) 

**3. Fee calculation** (`getConwayMinFeeTx` in `eras/conway/impl/src/Cardano/Ledger/Conway/Tx.hs`):

```haskell
tierRefScriptFee
  (unboundRational $ pp ^. ppRefScriptCostMultiplierG)
  (fromIntegral @Word32 @Int . unNonZero $ pp ^. ppRefScriptCostStrideG)
  refScriptCostPerByte
  refScriptsSize
``` [5](#0-4) 

The ADR explicitly acknowledges this design choice: *"One of the constraints we had to operate under was inability to add any new protocol parameters, since that was a bit too late in the release cycle of the Conway era. In other words we had to hard code some values, which will be turned into proper protocol parameters in the next era."* [6](#0-5) 

In Dijkstra, these four values are promoted to proper protocol parameters (`dppMaxRefScriptSizePerBlock`, `dppMaxRefScriptSizePerTx`, `dppRefScriptCostStride`, `dppRefScriptCostMultiplier`) with full governance update paths. [7](#0-6) 

### Impact Explanation

**Fund Freezing (High):** A script author — an explicitly allowed attacker role — can deploy a Plutus script whose spending condition requires referencing scripts totaling more than 200 KiB per transaction. Users who lock ADA or native assets in UTxOs protected by such a script will find those funds permanently unspendable in Conway era. The `validateRefScriptSize` check is a hard ledger rule: the transaction is rejected unconditionally, with no alternative spending path available through governance. Recovery requires a hard fork to Dijkstra era, where `ppMaxRefScriptSizePerTx` becomes a governable parameter.

**Fee Manipulation Outside Design Parameters (Medium):** The hard-coded multiplier (1.2) and stride (25,600 bytes) govern the exponential tier pricing of reference script fees for all Conway transactions. These values cannot be adjusted by governance even if network conditions change (e.g., hardware improvements reduce deserialization cost, making the multiplier too aggressive, or new attack vectors emerge requiring a tighter curve). This means the fee schedule for reference scripts is permanently fixed in Conway era regardless of actual network economics.

### Likelihood Explanation

The 200 KiB per-transaction limit is generous for most current DApps. However, complex multi-script DeFi protocols (e.g., those combining multiple large PlutusV3 scripts as reference inputs) can approach this boundary. The attacker-controlled entry path is straightforward: a script author deploys a contract whose spending logic references scripts summing to > 200 KiB, and users lock funds in it. The script author need not be malicious — a legitimate DApp developer who miscalculates reference script sizes, or whose DApp grows over time, can inadvertently trigger this condition. The fee parameter issue is always active and affects every Conway transaction using reference scripts.

### Recommendation

The fix is already implemented in Dijkstra: promote all four values to proper protocol parameters with governance update paths. For Conway era, the only available mitigation is to ensure the hard-coded values are set conservatively enough that no legitimate DApp is blocked before the Dijkstra upgrade. Additionally, tooling and documentation should warn DApp developers of the 200 KiB per-transaction hard cap in Conway era so they do not deploy scripts that would freeze user funds.

### Proof of Concept

1. A script author deploys a Plutus script `S` whose spending validator requires reference inputs containing scripts `R1, R2, ..., Rn` where `sum(size(Ri)) > 204,800` bytes.
2. Users lock ADA in UTxOs at the address of `S`.
3. Any attempt to spend those UTxOs in Conway era triggers `validateRefScriptSize`, which reads `ppMaxRefScriptSizePerTxG` — a compile-time constant of `200 * 1024` — and unconditionally rejects the transaction with `ConwayTxRefScriptsSizeTooBig`.
4. No governance action (PParamUpdate proposal, DRep vote, SPO vote, committee approval) can raise this limit in Conway era, because `ppMaxRefScriptSizePerTxG` is not stored in `ConwayPParams` and has no `PParamUpdate` lens.
5. The funds remain frozen until the Dijkstra era upgrade, at which point `ppMaxRefScriptSizePerTx` becomes a governable parameter. [8](#0-7) [1](#0-0) [9](#0-8)
