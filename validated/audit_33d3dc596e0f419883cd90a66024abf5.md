### Title
Resharing Cancellation Can Be Front-Run by Completing `vote_reshared` - (File: `crates/contract/src/state/resharing.rs`)

---

### Summary

The `ResharingContractState` exposes two competing, mutually-unguarded state-exit paths: `vote_cancel_resharing` (threshold votes from old participants) and `vote_reshared` (unanimous votes from all new participants). Because neither path checks the accumulation state of the other, a race condition exists where new participants — including a newly-untrusted or compromised one — can complete the resharing before the old participants' cancellation reaches threshold, permanently installing the untrusted participant set and stripping the old participants of any further governance control over that epoch.

---

### Finding Description

`ResharingContractState` in `crates/contract/src/state/resharing.rs` maintains two independent vote-accumulation paths that both exit the `Resharing` state:

**Path 1 — `vote_cancel_resharing`** (lines 173–196): Accumulates votes from old-epoch participants. Transitions back to the previous `RunningContractState` when `cancellation_votes_count >= previous_running_threshold.value()`. [1](#0-0) 

**Path 2 — `vote_reshared`** (lines 121–164): Accumulates votes from new-epoch participants via `resharing_key.vote_success`. Transitions to a new `RunningContractState` when `count == self.parameters.participants().len()` — i.e., **all** new participants have voted unanimously. [2](#0-1) 

Neither function checks the other's accumulation state. `vote_reshared` does not inspect `cancellation_requests`; `vote_cancel_resharing` does not inspect the `resharing_key` vote count. The contract-level entry points confirm this: [3](#0-2) [4](#0-3) 

The outcome is determined purely by which threshold is reached first in NEAR's sequential transaction ordering. MPC nodes are fully automated: they submit `vote_reshared` immediately after completing the off-chain resharing computation. Old participants must manually coordinate and submit `vote_cancel_resharing` transactions. If all new participants' `vote_reshared` transactions land before the old participants accumulate threshold cancellation votes, the resharing completes and the old participants permanently lose governance control over that epoch.

---

### Impact Explanation

If old participants initiate `vote_cancel_resharing` to remove a participant that has become untrusted (e.g., key compromise discovered mid-resharing), but the new participants' automated `vote_reshared` transactions complete the resharing first, the untrusted participant is permanently installed in the new `RunningContractState`. Once the contract exits `Resharing`, the old participants' subsequent `vote_cancel_resharing` calls fail because the state is no longer `Resharing`. The untrusted participant now holds a valid key share for the new epoch and participates in all future signing operations. This breaks the participant-state safety invariant: the old participants' governance intent to cancel is not respected, and the contract execution flow is manipulated into an outcome the old participants explicitly voted against.

---

### Likelihood Explanation

MPC nodes are automated and submit `vote_reshared` transactions as soon as the off-chain computation completes. The old participants must manually detect the threat, coordinate, and submit `vote_cancel_resharing` transactions — a slower, human-in-the-loop process. The window for cancellation is narrow: once all new participants have submitted their `vote_reshared` transactions (which happens automatically), the resharing completes in the next block that processes the final vote. The scenario is realistic whenever a new participant is discovered to be compromised after the resharing has already progressed to the voting phase.

---

### Recommendation

Add a guard in `vote_reshared` (or in `ResharingContractState::vote_reshared`) that checks whether cancellation votes have already reached threshold. If `cancellation_requests.len() >= previous_running_threshold.value()`, reject the `vote_reshared` call with an appropriate error. Alternatively, introduce a `cancellation_pending` boolean flag on `ResharingContractState` that is set to `true` once threshold cancellation votes are accumulated, and block `vote_reshared` from completing the transition while this flag is set.

---

### Proof of Concept

1. Old participants call `vote_new_parameters` → contract enters `ResharingContractState` with new participant set including participant `P_malicious`.
2. All new participants (including `P_malicious`) run the off-chain MPC resharing computation. Their automated nodes immediately submit `vote_reshared` transactions.
3. Old participants discover `P_malicious` is compromised and begin submitting `vote_cancel_resharing` transactions (requires threshold votes from old set).
4. The new participants' unanimous `vote_reshared` transactions are processed first (automated, no human delay). `vote_success` returns `true` when `count == participants().len()`, triggering transition to new `RunningContractState`. [5](#0-4) 

5. Old participants' subsequent `vote_cancel_resharing` calls fail — the contract is no longer in `Resharing` state.
6. `P_malicious` now holds a valid key share in the new epoch and participates in all future signing and CKD operations. [6](#0-5)

    pub fn vote_reshared(
        &mut self,
        key_event_id: KeyEventId,
    ) -> Result<Option<RunningContractState>, Error> {
        let previous_key = self.previous_keyset().domains[self.reshared_keys.len()].clone();
        if self
            .resharing_key
            .vote_success(&key_event_id, previous_key.key.clone())?
        {
            let new_key = KeyForDomain {
                domain_id: key_event_id.domain_id,
                attempt: key_event_id.attempt_id,
                key: previous_key.key,
            };
            self.reshared_keys.push(new_key);
            if let Some(next_domain) = self
                .previous_running_state
                .domains
                .get_domain_by_index(self.reshared_keys.len())
            {
                self.resharing_key = KeyEvent::new(
                    self.prospective_epoch_id(),
                    next_domain.clone(),
                    self.resharing_key.proposed_parameters().clone(),
                );
            } else {
                // Resharing complete: fold the per-domain threshold updates into
                // the registry and store the proposed parameters. The updates live
                // only on this resharing state, so they are structurally dropped
                // here rather than scrubbed off the stored parameters.
                let new_domains = self
                    .previous_running_state
                    .domains
                    .with_threshold_updates(&self.per_domain_thresholds)?;
                return Ok(Some(RunningContractState::new(
                    new_domains,
                    Keyset::new(self.prospective_epoch_id(), self.reshared_keys.clone()),
                    self.resharing_key.proposed_parameters().clone(),
                    self.previous_running_state.add_domains_votes.clone(),
                )));
            }
        }
        Ok(None)
    }

    pub fn vote_cancel_resharing(&mut self) -> Result<Option<RunningContractState>, Error> {
        let previous_running_participants = self.previous_running_state.parameters.participants();
        let authenticated_candidate = AuthenticatedAccountId::new(previous_running_participants)?;
        self.cancellation_requests.insert(authenticated_candidate);

        let cancellation_votes_count = self.cancellation_requests.len() as u64;
        let previous_running_threshold = self.previous_running_state.parameters.threshold();

        let threshold_cancellation_votes_reached: bool =
            cancellation_votes_count >= previous_running_threshold.value();

        let running_state = if threshold_cancellation_votes_reached {
            let mut previous_running_state = self.previous_running_state.clone();
            let prospective_epoch_id = self.prospective_epoch_id();
            previous_running_state.previously_cancelled_resharing_epoch_id =
                Some(prospective_epoch_id);

            Some(previous_running_state)
        } else {
            None
        };

        Ok(running_state)
    }
