### Title
Pending-block `get_class_at` returns spurious `CLASS_HASH_NOT_FOUND` due to `pending_data`/`pending_classes` update race — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

---

### Summary

`get_class_at(Pending, address)` performs two non-atomic reads from two independently-locked objects. The central sync loop writes `pending_data` first and only later — after an async network download — writes `pending_classes`. Any RPC call that lands in that window returns a spurious `CLASS_HASH_NOT_FOUND` for a class that is correctly present in the pending block.

---

### Finding Description

`get_class_at` is a two-step function:

```
get_class_at(Pending, addr)
  └─ get_class_hash_at(Pending, addr)   // reads self.pending_data  (Arc<RwLock<PendingData>>)
  └─ get_class(Pending, class_hash)     // reads self.pending_classes (Arc<RwLock<PendingClasses>>)
``` [1](#0-0) 

`get_class_hash_at` reads `pending_data.state_update.state_diff.deployed_contracts` under one lock: [2](#0-1) 

`get_class` then reads `pending_classes` under a separate lock. If the class is absent, it silently falls back to `BlockId::Tag(Tag::Latest)` and searches finalized storage: [3](#0-2) 

The central sync loop in `pending_sync.rs` commits `pending_data` atomically at line 166, then **returns** and only afterwards spawns async tasks to download each declared class into `pending_classes`: [4](#0-3) 

The class download tasks write to `pending_classes` only after a network round-trip to the central source: [5](#0-4) 

The gap between these two writes — potentially seconds of network latency — is the race window.

---

### Impact Explanation

During the race window, for any contract whose class was **newly declared** in the same pending block (i.e., not yet in finalized storage):

1. `get_class_hash_at(Pending, addr)` succeeds and returns `class_hash X` (found in `pending_data`).
2. `get_class(Pending, X)` finds nothing in `pending_classes`, falls back to `Latest`, finds nothing in storage, and returns `CLASS_HASH_NOT_FOUND`.

The net result is that `get_class_at(Pending, addr)` returns `CLASS_HASH_NOT_FOUND` for a class that is correctly deployed in the pending block. This is a wrong authoritative value from the pending RPC view, matching the **High** impact category: *"RPC pending view returns an authoritative-looking wrong value."*

---

### Likelihood Explanation

- The window is open for the entire duration of the async class download from the central source (network latency, typically hundreds of milliseconds to seconds).
- Any unprivileged RPC client can trigger it by calling `starknet_getClassAt` with `"block_id": "pending"` immediately after a new pending block containing a fresh class declaration appears.
- No special privileges, keys, or timing coordination beyond a normal RPC call are required.

---

### Recommendation

Perform both reads under a single consistent snapshot. One approach: in `get_class_at`, read `pending_data` and `pending_classes` together under a combined lock or pass the already-resolved `pending_classes` snapshot into `get_class` so the fallback to `Latest` only occurs when the class is absent from **both** `pending_classes` and `pending_data.declared_classes`. Alternatively, `get_class` when called with `Pending` should not fall back to `Latest` if the class_hash appears in `pending_data.declared_classes` but is not yet in `pending_classes` — it should instead return an appropriate "pending class not yet available" response rather than `CLASS_HASH_NOT_FOUND`.

---

### Proof of Concept

```
1. pending_data  ← { deployed_contracts: [addr → class_hash X],
                     declared_classes:   [X → compiled_hash Y] }
   pending_classes ← {}   (download of X not yet complete)

2. RPC client calls starknet_getClassAt("pending", addr)

3. get_class_hash_at("pending", addr)
     reads pending_data.deployed_contracts → returns class_hash X   ✓

4. get_class("pending", X)
     reads pending_classes → X not found
     falls back to BlockId::Latest
     storage has no record of X (newly declared, not finalized)
     → returns CLASS_HASH_NOT_FOUND                                  ✗

5. Caller receives CLASS_HASH_NOT_FOUND despite the class being
   correctly present in the pending block.
```
