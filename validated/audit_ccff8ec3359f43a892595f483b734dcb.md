### Title
Deposits Paid for Non-Executed `propose_update` Proposals Are Permanently Locked When Any Update Executes — (File: `crates/contract/src/update.rs`)

---

### Summary

`propose_update` requires a NEAR deposit to cover storage costs. When any competing update reaches threshold votes and executes, `do_update` clears **all** pending proposals without refunding the deposits paid by other proposers. Those NEAR tokens are permanently locked in the contract with no recovery path.

---

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` (lines 1297–1334) requires a deposit equal to `ProposedUpdates::required_deposit(&update)`, computed as `env::storage_byte_cost() * bytes_used`. Only the excess above the required amount is refunded to the caller; the exact required deposit is retained by the contract.

```rust
// lib.rs:1308-1331
let attached = env::attached_deposit();
let required = ProposedUpdates::required_deposit(&update);
if attached < required { return Err(...); }
let id = self.proposed_updates.propose(update);
// Refund the difference if the proposer attached more than required.
if let Some(diff) = attached.checked_sub(required) && diff > NearToken::from_yoctonear(0) {
    Promise::new(proposer).transfer(diff).detach();
}
```

When `vote_update` (lib.rs:1343–1388) determines that threshold votes have been reached, it calls `do_update`. Inside `do_update` (update.rs:195–227):

```rust
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    // ... builds and returns Promise — no refund issued
}
```

Every pending proposal entry is wiped. The `UpdateEntry` struct stores `bytes_used` but **not** the proposer's account ID or the deposit amount:

```rust
// update.rs:132-135
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
}
```

Because the proposer identity and deposit amount are never recorded, there is no information available to issue refunds even if the code attempted to. The freed storage bytes are reclaimed by the runtime, but the NEAR tokens that paid for them remain in the contract balance indefinitely.

There is no `cancel_proposal` or equivalent function. `remove_update_vote` (lib.rs:1397–1404) only removes the caller's vote record, not the proposal entry or its deposit.

---

### Impact Explanation

Every participant who calls `propose_update` while a competing proposal is also pending loses their full required deposit permanently the moment any other update executes. The contract balance grows by the sum of all non-executed proposal deposits with no mechanism to drain it back to rightful owners. This breaks the accounting invariant that storage-staking deposits must be returned when the storage they cover is freed.

Severity: **Medium** — matches "balance or accounting invariant break" in the allowed impact scope.

---

### Likelihood Explanation

In a production MPC network with multiple participants, it is routine for more than one participant to propose competing contract or config updates simultaneously (e.g., two nodes propose different config versions). Every such race results in permanent deposit loss for the losing proposer(s). The condition is reachable by any current MPC participant without any privileged access beyond their normal voter role.

---

### Recommendation

1. Add `proposer: AccountId` and `deposit: NearToken` fields to `UpdateEntry`.
2. In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and issue `Promise::new(entry.proposer).transfer(entry.deposit)` for each one.
3. Alternatively, maintain a separate `proposer_deposits: LookupMap<UpdateId, (AccountId, NearToken)>` and drain it on every `do_update` call.

---

### Proof of Concept

1. Participant A calls `propose_update` with a contract binary, attaching deposit `D_A` (e.g., several NEAR for a large WASM blob). The contract stores the entry and retains `D_A`.
2. Participant B calls `propose_update` with a config change, attaching deposit `D_B`. The contract stores a second entry and retains `D_B`.
3. Threshold participants vote for B's config update via `vote_update(id_B)`.
4. `vote_update` calls `self.proposed_updates.do_update(&id_B, gas)`.
5. `do_update` executes B's update, then calls `self.entries.clear()` — wiping A's entry — and `self.vote_by_participant.clear()`.
6. No refund is issued for `D_A`. `D_A` is now permanently locked in the contract. A has no recourse: there is no cancel function, no withdraw function, and the proposer identity was never stored. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }
