Audit Report

## Title
Missing Staleness Validation on Chainlink `latestRoundData()` Allows Stale Prices to Drive Incorrect rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards `updatedAt`, `roundId`, and `answeredInRound`, performing no time-based or round-based staleness check. A stale Chainlink LST/ETH price feed is accepted without revert and propagated directly into rsETH minting calculations, allowing any depositor to receive more rsETH than their deposited asset is worth, diluting all existing rsETH holders.

## Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol` at line 52, `getAssetPrice()` captures only the `price` field from `latestRoundData()`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
```

`updatedAt`, `roundId`, and `answeredInRound` are all discarded. No check of the form `block.timestamp - updatedAt > heartbeat` is performed, and no `answeredInRound < roundId` guard is applied.

This price is returned directly to `LRTOracle.getAssetPrice()` (line 157), which delegates to the registered `IPriceFetcher` without any additional validation. `LRTDepositPool.getRsETHAmountToMint()` (line 520) then uses this price:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

This value flows through `_beforeDeposit()` (line 665) into `depositAsset()` (line 111), which is a fully permissionless, public function callable by any address.

The sibling contract `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` (lines 30â€“31) demonstrates the protocol is aware of the staleness check pattern:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
```

This check was applied to the pool collateral oracle but omitted from `ChainlinkPriceOracle`, which is the oracle used for all LST deposit pricing.

## Impact Explanation
If a Chainlink LST/ETH feed (e.g., stETH/ETH, rETH/ETH) goes stale at a price inflated relative to the actual current market price â€” a realistic scenario during an LST depeg event where the oracle lags â€” `getAssetPrice()` returns the stale high price without reverting. A depositor calling `depositAsset()` at that moment receives more rsETH than the deposited asset is actually worth. This dilutes all existing rsETH holders by reducing the ETH backing per rsETH token, constituting **theft of unclaimed yield** (High impact).

## Likelihood Explanation
Chainlink LST/ETH feeds carry 24-hour heartbeats and 0.5% deviation thresholds. During periods of network congestion, oracle node failures, or rapid LST price movement (depeg), the feed can lag beyond the heartbeat. This is a documented, historically observed failure mode for Chainlink integrations. The entry path (`depositAsset`) requires no special role, no governance action, and no oracle operator compromise â€” any external address can trigger it.

## Recommendation
Add both a round-based and a time-based staleness check in `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

if (answeredInRound < roundId) revert StalePrice();
if (block.timestamp - updatedAt > STALENESS_THRESHOLD) revert StalePrice();
require(price > 0, "Invalid price");
```

`STALENESS_THRESHOLD` should be set per feed based on its documented heartbeat (e.g., 86400 seconds + a small buffer for 24-hour feeds), consistent with the pattern already applied in `ChainlinkOracleForRSETHPoolCollateral`.

## Proof of Concept
1. Chainlink stETH/ETH feed last updated at `T - 25h` (heartbeat exceeded). Feed price is `1.01e18` (stale high; actual market is `0.99e18` due to a depeg).
2. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
3. `getRsETHAmountToMint(stETH, 100e18)` calls `lrtOracle.getAssetPrice(stETH)` â†’ `ChainlinkPriceOracle.getAssetPrice(stETH)` â†’ returns `1.01e18` with no revert.
4. `rsethAmountToMint = (100e18 * 1.01e18) / rsETHPrice` â€” attacker receives ~2% more rsETH than the deposited stETH is actually worth.
5. All existing rsETH holders are diluted; the attacker has extracted value at their expense.
6. No admin action, no special role, no oracle operator compromise required.

**Foundry fork test plan**: Fork mainnet at a block where the stETH/ETH Chainlink feed is within its heartbeat. Warp `block.timestamp` forward by 25 hours (beyond the 24-hour heartbeat). Call `depositAsset` with stETH. Observe that `getAssetPrice` returns the pre-warp stale price without reverting, and that `rsethAmountToMint` is inflated relative to the actual current stETH/ETH rate.
