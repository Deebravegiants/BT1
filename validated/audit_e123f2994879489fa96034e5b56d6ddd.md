### Title
Unprotected `init` Function Allows Front-Running to Seize Full MPC Network Control - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `MpcContract::init` function lacks any caller restriction (`#[private]` or equivalent). Because the NEAR deployment workflow calls `init` as a **separate transaction** after deploying the WASM, any unprivileged account can observe the deployment and race to call `init` first with attacker-controlled participant parameters, permanently seizing governance of the MPC network.

---

### Finding Description

`MpcContract::init` is the sole entry point for bootstrapping the MPC contract. It is decorated with `#[init]` (which prevents re-initialization once state exists) but carries **no caller restriction**:

```rust
// Contract developer helper API
#[near]
impl MpcContract {
    #[handle_result]
    #[init]                          // ← no #[private], no predecessor check
    pub fn init(
        parameters: dtos::ThresholdParameters,
        init_config: Option<dtos::InitConfig>,
    ) -> Result<Self, Error> {
``` [1](#0-0) 

Contrast this with `init_running`, which is correctly guarded:

```rust
#[private]   // ← only callable by the contract account itself
#[init]
#[handle_result]
pub fn init_running(...)
``` [2](#0-1) 

The production deployment workflow (documented in localnet, testnet, and TEE cluster scripts) always issues `init` as a **separate transaction** after the WASM deploy:

```bash
near contract call-function as-transaction mpc-contract.test.near init \
  file-args /tmp/init_args.json ...
``` [3](#0-2) [4](#0-3) 

Between the WASM deployment and the `init` call, the contract has code but no state. During this window, any account can call `init` with arbitrary `ThresholdParameters`.

The only validation inside `init` is structural (threshold bounds, participant count). An attacker supplying two attacker-controlled accounts with `threshold = 2` satisfies all constraints (≥ 60% of 2 participants, ≥ 2 absolute minimum): [5](#0-4) 

Once the attacker's `init` succeeds, the NEAR `#[init]` guard causes every subsequent legitimate `init` call to panic, permanently locking the attacker's participant set into the contract.

---

### Impact Explanation

After a successful front-run:

1. **Attacker controls the full participant set.** All governance votes (`vote_new_parameters`, `vote_add_domains`, `vote_pk`, `vote_reshared`) require a threshold of the registered participants. With two attacker accounts at threshold 2, the attacker holds 100% of votes.

2. **Attacker drives key generation.** The attacker calls `vote_add_domains` from both accounts to transition the contract to `Initializing`, then runs DKG nodes they control to generate key shares exclusively held by the attacker.

3. **Attacker issues unauthorized threshold signatures.** With full key-share ownership, the attacker can call `respond` to resolve any `sign` request with a signature derived from keys they control, bypassing the intended multi-party threshold entirely.

4. **Legitimate operators are permanently locked out.** Because `#[init]` prevents re-initialization, there is no recovery path short of redeploying to a new contract account.

This satisfies: *Critical — Unauthorized transaction execution, threshold signature issuance, or confidential key derivation output without the required participant authorization* and *Critical — Bypass of threshold-signature requirements or unauthorized access to MPC key shares.*

---

### Likelihood Explanation

- The deployment scripts universally issue `init` as a separate transaction, creating a reliable race window on every mainnet/testnet deployment.
- NEAR transaction ordering is public and observable; a mempool-watching attacker can detect the WASM deploy and submit the malicious `init` in the next block.
- No special privilege, leaked key, or TEE access is required — only a funded NEAR account.
- The attacker's `init` call is structurally valid and will pass all on-chain validation.

---

### Recommendation

Add `#[private]` to `init`, mirroring the protection already applied to `init_running`:

```rust
#[private]   // restricts caller to env::current_account_id()
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
```

Alternatively, assert inside the function body:

```rust
assert_eq!(
    env::predecessor_account_id(),
    env::current_account_id(),
    "init must be called by the contract account itself"
);
```

The deployment scripts already sign `init` as `mpc-contract.test.near` (the contract account itself), so adding `#[private]` is a zero-friction fix that matches the existing deployment convention. [6](#0-5)

### Proof of Concept

1. Attacker monitors the NEAR blockchain for a transaction deploying `mpc_contract.wasm` to `mpc-contract.near`.
2. In the immediately following block, attacker submits:

```bash
near contract call-function as-transaction mpc-contract.near init \
  json-args '{
    "parameters": {
      "threshold": 2,
      "participants": {
        "next_id": 2,
        "participants": [
          ["attacker1.near", 0, {"tls_public_key": "ed25519:<ATK1_KEY>", "url": "https://attacker.example/1"}],
          ["attacker2.near", 1, {"tls_public_key": "ed25519:<ATK2_KEY>", "url": "https://attacker.example/2"}]
        ]
      }
    }
  }' \
  prepaid-gas '300.0 Tgas' attached-deposit '0 NEAR' \
  sign-as attacker1.near network-config mainnet sign-with-keychain send
```

3. The legitimate operator's `init` call arrives later and panics: `"Cannot initialize a contract that is already initialized"`.
4. Attacker calls `vote_add_domains` from both `attacker1.near` and `attacker2.near`, reaching threshold and transitioning to `Initializing`.
5. Attacker runs two MPC nodes they control, completing DKG and calling `vote_pk` to transition to `Running` with keys exclusively held by the attacker.
6. All subsequent `sign` requests are fulfilled by the attacker's nodes, producing signatures under attacker-controlled key shares. [7](#0-6) [2](#0-1)

        if k.value() > n_shares {
            return Err(InvalidThreshold::MaxRequirementFailed {
                max: n_shares,
                found: k.value(),

        if k.value() < MIN_THRESHOLD_ABSOLUTE {

        let lower_relative_bound = governance_threshold_lower_relative_bound(n_shares);

            return Err(InvalidThreshold::MinRelRequirementFailed {
                required: lower_relative_bound,
                found: k.value(),
            }
            .into());
        }

                max: upper_relative_bound,
                found: k.value(),
            }
            .into());
        }
        Ok(())
    }
