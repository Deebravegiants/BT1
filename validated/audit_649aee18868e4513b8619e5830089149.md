### Title
Malicious Third-Party Harvester Can Unconditionally Redirect Farmer Block Reward via Unvalidated `farmer_reward_address_override` - (File: `chia/farmer/farmer_api.py`)

---

### Summary

The farmer's `_process_respond_signatures()` function unconditionally substitutes any `farmer_reward_address_override` supplied in the harvester's `RespondSignatures` message as the `farmer_reward_address` in `DeclareProofOfSpace`, without enforcing the CHIP-22 fee-quality threshold. The only guard, `notify_farmer_reward_taken_by_harvester_as_fee()`, is advisory-only (log/warn, no rejection). A malicious third-party harvester can therefore redirect the farmer's entire block reward (0.25 XCH per block) to an attacker-controlled puzzle hash with no cryptographic check.

---

### Finding Description

**Root cause — two independent bypass paths**

**Path 1 (override present in both messages):**

In `new_proof_of_space`, when `NewProofOfSpace.farmer_reward_address_override is not None`, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()`:

```python
# farmer_api.py L128-129
if new_proof_of_space.farmer_reward_address_override is not None:
    self.farmer.notify_farmer_reward_taken_by_harvester_as_fee(sp, new_proof_of_space)
```

That function computes `fee_quality` and compares it to the harvester-supplied `applied_fee_threshold`, but **only emits log warnings** on failure — it never returns an error, raises an exception, or sets any flag that would prevent the override from being used:

```python
# farmer.py L920-928
else:
    self.log.warning(
        f"Invalid fee threshold for challenge '{challenge_str}': ..."
    )
    self.log.warning(
        "Harvester illegitimately took a fee reward that "
        + "did not belong to it or it incorrectly applied the fee convention."
    )
```

Execution then continues normally into `_process_respond_signatures()`, which blindly accepts `response.farmer_reward_address_override`:

```python
# farmer_api.py L916-919
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
    include_source_signature_data = True
```

**Path 2 (override absent in `NewProofOfSpace`, injected only in `RespondSignatures`):**

The fee-quality check is gated on `NewProofOfSpace.farmer_reward_address_override is not None`. A harvester can send `NewProofOfSpace` with `farmer_reward_address_override = None` (skipping the check entirely), then inject an arbitrary puzzle hash only in the subsequent `RespondSignatures` message. Because `_process_respond_signatures()` reads `response.farmer_reward_address_override` independently, the attacker-controlled address is accepted with **zero fee-quality evaluation**.

The overridden address is then propagated into `DeclareProofOfSpace.farmer_reward_address` and broadcast to the full node, which embeds it in the foliage block data as the coinbase recipient for the farmer's 1/8 block reward.

---

### Impact Explanation

Every block won by the farmer while the malicious harvester is active sends the farmer's coinbase reward (currently 0.25 XCH per block, ~250,000,000,000 mojos) to the attacker's puzzle hash instead of `farmer_target`. This is a direct, irreversible XCH loss per block — a **payout redirection** of a singleton-adjacent coinbase coin. The farmer has no on-chain recourse once the block is confirmed.

---

### Likelihood Explanation

Any third-party harvester operator (CHIP-22 scenario) can exploit this. The attacker only needs:
1. Plots that pass the proof-of-space filter for at least one signage point.
2. A network connection to the victim farmer.

No farmer private keys, no admin access, no cryptographic break is required. The farmer's own signing key is used to sign the foliage block data that already contains the attacker's puzzle hash, so the block is fully valid on-chain.

---

### Recommendation

In `_process_respond_signatures()`, before substituting `response.farmer_reward_address_override`, re-derive the fee quality from the stored proof and signage-point data and enforce the threshold:

```python
if response.farmer_reward_address_override is not None:
    fee_quality = calculate_harvester_fee_quality(pospace.proof, ...)
    # retrieve stored fee_info from the NewProofOfSpace record
    if stored_fee_info is None or fee_quality > stored_fee_info.applied_fee_threshold:
        self.farmer.log.warning("Rejecting invalid farmer_reward_address_override")
        return None          # drop the block attempt
    farmer_reward_address = response.farmer_reward_address_override
```

Additionally, `notify_farmer_reward_taken_by_harvester_as_fee()` should return a boolean so callers can act on the result rather than only logging.

---

### Proof of Concept

1. Malicious harvester connects to victim farmer.
2. Harvester finds a valid proof of space for a signage point.
3. Harvester sends `NewProofOfSpace` with `farmer_reward_address_override = None` and `fee_info = None` — no fee-quality check is triggered.
4. Farmer sends `RequestSignatures` back to the harvester.
5. Harvester returns `RespondSignatures` with `farmer_reward_address_override = attacker_puzzle_hash`.
6. `_process_respond_signatures()` (L916-918) sets `farmer_reward_address = attacker_puzzle_hash` with no validation.
7. Farmer broadcasts `DeclareProofOfSpace` containing `attacker_puzzle_hash` as the farmer reward address.
8. Full node farms the block; the 0.25 XCH coinbase reward is locked to `attacker_puzzle_hash`.

    def notify_farmer_reward_taken_by_harvester_as_fee(
        self, sp: farmer_protocol.NewSignagePoint, proof_of_space: harvester_protocol.NewProofOfSpace
    ) -> None:
        """
        Apply a fee quality convention (see CHIP-22: https://github.com/Chia-Network/chips/pull/88)
        given the proof and signage point. This will be tested against the fee threshold reported
        by the harvester (if any), and logged.
        """
        assert proof_of_space.farmer_reward_address_override is not None

        challenge_str = str(sp.challenge_hash)

        ph_prefix = self.config["network_overrides"]["config"][self.config["selected_network"]]["address_prefix"]
        farmer_reward_puzzle_hash = encode_puzzle_hash(proof_of_space.farmer_reward_address_override, ph_prefix)

        self.log.info(
            f"Farmer reward for challenge '{challenge_str}' "
            + f"taken by harvester for reward address '{farmer_reward_puzzle_hash}'"
        )

        fee_quality = calculate_harvester_fee_quality(proof_of_space.proof.proof, sp.challenge_hash)
        fee_quality_rate = float(fee_quality) / float(0xFFFFFFFF) * 100.0

        if proof_of_space.fee_info is not None:
            fee_threshold = proof_of_space.fee_info.applied_fee_threshold
            fee_threshold_rate = float(fee_threshold) / float(0xFFFFFFFF) * 100.0

            if fee_quality <= fee_threshold:
                self.log.info(
                    f"Fee threshold passed for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
            else:
                self.log.warning(
                    f"Invalid fee threshold for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
                self.log.warning(
                    "Harvester illegitimately took a fee reward that "
                    + "did not belong to it or it incorrectly applied the fee convention."
                )
        else:
            self.log.warning(
                "Harvester illegitimately took reward by failing to provide its fee rate "
                + f"for challenge '{challenge_str}'. "
                + f"Fee quality was {fee_quality_rate:.3f}% ({fee_quality} or 0x{fee_quality:08x})"
            )
