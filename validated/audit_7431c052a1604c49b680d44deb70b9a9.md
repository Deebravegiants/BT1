### Title
Sub-Transaction Plutus Execution Units Excluded from Fee and Block Limit Metering in Dijkstra Era — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Tx.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

---

### Summary

The Dijkstra era introduces nested (sub-)transactions embedded inside a top-level transaction body. Sub-transactions have no fee field and are processed with `updateUTxOStateNoFees`. Critically, the Plutus execution units declared in sub-transaction redeemers are **not counted** in the top-level transaction's minimum-fee calculation, the per-transaction `maxTxExUnits` check, or the per-block `maxBlockExUnits` check. An unprivileged attacker can craft a top-level transaction whose sub-transactions carry arbitrarily expensive Plutus scripts, paying only the byte-proportional fee while forcing every validating node to perform unbounded Plutus evaluation work.

---

### Finding Description

**Root cause — fee calculation omits sub-transaction execution units**

`getMinFeeTx` for `DijkstraEra` is wired to `getConwayMinFeeTx`: [1](#0-0) 

`getConwayMinFeeTx` computes:

```
alonzoMinFeeTx pp tx  +  refScriptsFee
```

`alonzoMinFeeTx` adds `txscriptfee (pp ^. ppPricesL) (totExUnits tx)` where `totExUnits` is:

```haskell
totExUnits tx = foldMap exunits (tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL)
```

This only folds over the **top-level** transaction's redeemer map. Sub-transactions are stored in `dtbrSubTransactions` inside the top-level body and each carries its own `TxWits` with its own redeemer map — none of which is visited here. [2](#0-1) 

**Root cause — per-transaction ExUnits limit omits sub-transaction redeemers**

In `dijkstraUtxoTransition`, the execution-unit guard is:

```haskell
{- totExunits tx ≤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
```

`validateExUnitsTooBigUTxO` calls `totExUnits tx`, which again only reads the top-level witness set. [3](#0-2) [4](#0-3) 

**Root cause — per-block ExUnits limit omits sub-transaction redeemers**

`validateExUnits` in the BBODY rule sums `foldMap totExUnits txs` over all top-level transactions in the block. Sub-transaction redeemers are again invisible. [5](#0-4) 

**Sub-transactions do carry Plutus scripts**

`DijkstraStAnnSubTx` stores `dsastPlutusScriptsWithContext` and `dsastTxInfoResult`, confirming that sub-transaction Plutus scripts are collected and evaluated during validation. The `SUBUTXO` rule runs full script-related checks (validity interval, bad inputs, output size, etc.) but contains **no** `validateExUnitsTooBigUTxO` call and no fee check. [6](#0-5) [7](#0-6) 

**Sub-transactions have no fee field by design**

`DijkstraSubTxBodyRaw` has no `dtbrFee` field; the required-fields decoder for `SSubTx` lists only `inputs` and `outputs`, not `fee`. State updates for sub-transactions explicitly skip fee collection via `updateUTxOStateNoFees`. [8](#0-7) [9](#0-8) 

---

### Impact Explanation

An attacker submits a top-level Dijkstra transaction containing the maximum number of sub-transactions (bounded only by `maxTxSize`). Each sub-transaction references a Plutus script that is small in bytes but consumes the full `maxTxExUnits` budget when executed. The attacker pays only the byte-proportional fee (`a * txSize + b + refScriptFee`) for the entire batch. Every validating node must execute all sub-transaction Plutus scripts, performing work that is orders of magnitude larger than what the fee compensates. Repeated submission of such transactions can saturate block-validation CPU time, causing block-production delays or chain stalls.

This matches the **Medium** allowed impact: *attacker-controlled transactions exceed intended validation limits*.

---

### Likelihood Explanation

- Requires only an unprivileged transaction sender with enough ADA to cover the byte-based fee.
- The Dijkstra era is active at protocol version 12+; any node running Dijkstra is exposed.
- The attack is deterministic and reproducible; no probabilistic or timing conditions are needed.
- The analogous reference-script deserialization attack (ADR-9) was actually exploited on mainnet in June 2024, demonstrating that resource-underpricing attacks are actively pursued. [10](#0-9) 

---

### Recommendation

1. **Extend `totExUnits` for Dijkstra** to recursively sum execution units from all sub-transaction redeemer maps, so that `getConwayMinFeeTx`, `validateExUnitsTooBigUTxO`, and `validateExUnits` (BBODY) all account for sub-transaction Plutus costs.

2. **Add a per-batch ExUnits cap** analogous to `maxTxExUnits` that covers the combined execution units of the top-level transaction and all its sub-transactions.

3. **Add a per-block sub-transaction ExUnits contribution** to `validateExUnits` so that the block-level budget is not silently bypassed by batches with expensive sub-transactions.

---

### Proof of Concept

```
1. Compile a Plutus V3 script that loops until it exhausts exactly
   (maxTxExUnits.steps - 1) steps — small serialized size (~200 bytes),
   near-maximum execution cost.

2. Deploy the script to a UTxO (reference script or inline).

3. Construct a Dijkstra TopTx whose body contains N sub-transactions
   (N chosen so total byte size ≤ maxTxSize).  Each sub-transaction:
     - spends one UTxO locked by the looping script
     - carries a redeemer with ExUnits = (maxTxExUnits.steps - 1, ...)
     - has no fee field

4. The top-level transaction's redeemer map is empty → totExUnits = 0
   → fee = a * txSize + b  (byte fee only, no script fee)
   → validateExUnitsTooBigUTxO passes (0 ≤ maxTxExUnits)

5. Submit the transaction.  Each validating node must execute N copies
   of the looping script, consuming N * maxTxExUnits of CPU, while the
   attacker paid only the byte-proportional fee.
``` [11](#0-10) [12](#0-11)

It was identified a while ago that there is an overhead associated with deserializing scripts, so much so that it would be possible to create a very large script that was fairly expensive to deserialize, but very cheap to execute. This opened up an attack vector when such a Plutus script would be used as a reference script. This problem was exacerbated by the fact that there was no real limit on the total size of reference scripts that could be used in a transaction, thus being limited only by the size of the transaction itself. Therefore this opened up Cardano to a DDoS attack where an attacker could submit many such transactions that would cost very little, but would be expensive for a `cardano-node` to validate.

In order to prevent such an attack a `"minFeeRefScriptCostPerByte"` protocol parameter was introduced in the Conway era. The idea was fairly simple: we would calculate the total size of reference scripts used by a transaction and multiply it by the value specified by this protocol parameter. Result would be added to the transaction fee. However, in order for this approach to be a definite deterrent of such attacks this parameter would have to be set to a fairly high value. Community was not very keen on having this value set to a high enough value, since that could make reference scripts almost as expensive to use as regular scripts, thus significantly reducing their usability. Unfortunately things did not go exactly as planned, because on [June 25th 2024 an attack like this actually took place](https://cardanospot.io/news/ddos-attack-on-the-cardano-blockchain-mRIKAzZTNnzq5NGd). This attack forced us to make a quick decision on implementing a definite prevention of such attacks, while trying to not have a drastic impact on the common use case that DApp developers rely on so much.
