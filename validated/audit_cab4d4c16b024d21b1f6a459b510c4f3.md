### Title
Unvalidated `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` function accepts a caller-supplied `payload_hash` field inside `VerifyForeignTransactionResponse` and verifies only that the attached signature is cryptographically valid over that hash using the root public key. It never checks that `payload_hash` is actually derived from the pending `request` being resolved. A single Byzantine attested participant can replay a valid `{payload_hash, signature}` pair obtained from any prior on-chain response to resolve a completely different pending request, delivering a forged foreign-chain attestation to the waiting caller.

---

### Finding Description

In `respond_verify_foreign_tx` the contract performs three checks before resolving the yield:

1. Caller is an attested participant (`assert_caller_is_attested_participant_and_protocol_active`).
2. The ECDSA signature is valid over `response.payload_hash` under the domain's root public key.
3. The `request` key exists in `pending_verify_foreign_tx_requests`. [1](#0-0) 

What is **never** checked is whether `response.payload_hash` is the SHA-256 of `ForeignTxSignPayload { request: <the pending request>, values: <observed values> }`. The hash is taken verbatim from the response struct: [2](#0-1) 

The `VerifyForeignTransactionResponse` type simply carries an opaque `Hash256` alongside the signature: [3](#0-2) 

Every prior `respond_verify_foreign_tx` call is permanently visible on-chain, so any attested participant can extract a previously accepted `{payload_hash_B, signature_B}` pair and resubmit it against a different pending request `request_A`. The contract's only guard — that `signature_B` is valid over `payload_hash_B` — passes trivially because the pair was produced by a legitimate MPC session.

By contrast, the regular `respond` function derives the expected public key from the request's tweak before verifying, binding the signature to the specific request: [4](#0-3) 

No equivalent binding exists in `respond_verify_foreign_tx`.

The node-side code confirms that `payload_hash` is computed by the leader from the foreign-chain RPC result and then included in the response submitted on-chain — it is not re-derived by the contract: [5](#0-4) 

---

### Impact Explanation

**High — Forged foreign-chain verification / invalid bridge execution.**

A bridge or omnibridge contract calls `verify_foreign_transaction` for transaction A and awaits the NEAR yield. If a Byzantine leader replays `{payload_hash_B, signature_B}` (from a prior verification of transaction B) as the response to request A, the bridge receives a valid MPC signature over the wrong hash. Because `payload_hash` is the SHA-256 of `ForeignTxSignPayload{request_B, values_B}`, the bridge cannot reconstruct it without knowing `values_B` (the extracted on-chain data that was never returned to the caller). The bridge therefore cannot distinguish a genuine response from a replayed one. If the bridge uses the returned `payload_hash` to gate an asset release or state transition, the attacker can cause it to act on a different transaction than the one it requested — enabling double-spend or invalid inbound bridge execution.

---

### Likelihood Explanation

The attacker must be a single attested MPC participant (below the signing threshold). No threshold collusion is required. The replayed `{payload_hash, signature}` pair is freely available from any prior on-chain `respond_verify_foreign_tx` call. The attack is executable whenever a victim's `verify_foreign_transaction` request is pending and the attacker acts as the responding node before the honest leader does.

---

### Recommendation

The contract should bind the accepted `payload_hash` to the original request. Two complementary approaches:

1. **Include extracted values in the response.** Add `values: Vec<ExtractedValue>` to `VerifyForeignTransactionResponse`. The contract recomputes `expected_hash = ForeignTxSignPayload::V1 { request, values }.compute_msg_hash()` and asserts `response.payload_hash == expected_hash` before accepting the signature.

2. **Commit the request hash into the signed payload.** Modify `ForeignTxSignPayloadV1` to include the `VerifyForeignTransactionRequest` (or its hash) as a mandatory field. The contract can then verify that the `ForeignChainRpcRequest` embedded in the payload matches the pending request, even without knowing the extracted values, by checking the request portion of the preimage.

Either change closes the replay path by making a signature produced for request B cryptographically invalid as a response to request A.

---

### Proof of Concept

```
1. Honest flow: verify_foreign_transaction(request_B) is processed.
   Leader submits respond_verify_foreign_tx(request_B, {payload_hash_B, sig_B}).
   This call is recorded on-chain; {payload_hash_B, sig_B} is public.

2. Victim submits verify_foreign_transaction(request_A).
   request_A is now pending in pending_verify_foreign_tx_requests.

3. Byzantine attested participant calls:
     respond_verify_foreign_tx(request_A, {payload_hash_B, sig_B})

4. Contract checks:
   - Caller is attested participant          ✓
   - verify_ecdsa(sig_B, payload_hash_B, root_pk) == Ok  ✓  (sig_B is genuine)
   - request_A is in pending map             ✓
   - payload_hash_B == hash(request_A, ...)  ✗  (NOT CHECKED)

5. Contract resolves request_A's yield with {payload_hash_B, sig_B}.

6. Victim's bridge contract receives a valid MPC signature over
   payload_hash_B (transaction B's attestation) while believing it
   received attestation for transaction A.
   Bridge cannot detect the mismatch because values_B are unknown.
``` [6](#0-5) [7](#0-6) [8](#0-7)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

fn build_signature_request(
    request: &VerifyForeignTxRequest,
    foreign_tx_payload: &dtos::ForeignTxSignPayload,
) -> anyhow::Result<SignatureRequest> {
    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
        foreign_tx_payload.compute_msg_hash()?.into();
    let payload_bytes: BoundedVec<u8, ECDSA_PAYLOAD_SIZE_BYTES, ECDSA_PAYLOAD_SIZE_BYTES> =
        payload_hash.into();

    Ok(SignatureRequest {
        id: request.id,
        receipt_id: request.receipt_id,
        payload: Payload::Ecdsa(payload_bytes),
        tweak: Tweak::new([0u8; 32]),
        entropy: request.entropy,
        timestamp_nanosec: request.timestamp_nanosec,
        domain: request.domain_id,
    })
