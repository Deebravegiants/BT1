### Title
Expired/Revoked TEE Attestation Not Re-Verified at Execution Time in `is_caller_an_attested_participant()` — (File: crates/contract/src/tee/tee_state.rs)

---

### Summary

The `is_caller_an_attested_participant()` function, which gates all critical MPC protocol operations, only checks that an attestation *exists* in storage and that keys match. It does **not** re-verify whether the stored attestation is still valid (not expired, image hash still on the allowed list, measurements still accepted). This is the direct analog to the ERC-7484 "check at installation, not at execution" pattern: TEE attestation is fully verified at registration time (`submit_participant_info`), but only its *existence* is checked at execution time (`respond`, `vote_pk`, `vote_reshared`, etc.).

---

### Finding Description

At registration time, `submit_participant_info` → `tee_state.add_participant()` performs full attestation verification: [1](#0-0) 

This calls `attestation.verify_locally(...)` which checks expiry, image hash, launcher compose hash, and measurements.

At execution time, `assert_caller_is_attested_participant_and_protocol_active()` calls `is_caller_an_attested_participant()`: [2](#0-1) 

And `is_caller_an_attested_participant()` only checks: [3](#0-2) 

It checks that the attestation **exists** in `stored_attestations` and that the account/key fields match — but it never calls `reverify_participants()`, which is the function that checks expiry, image hash validity, and measurements: [4](#0-3) 

The full re-verification (`reverify_and_cleanup_participants`) is only triggered by an explicit `verify_tee()` call: [5](#0-4) 

`verify_tee()` is not called automatically — it requires a participant to invoke it. The design documentation states it is expected to be called "every 2 days," but this is a social/operational expectation, not a contract-enforced invariant.

The functions gated by `assert_caller_is_attested_participant_and_protocol_active()` — and therefore affected — are: [6](#0-5) 

Specifically: `respond`, `respond_ckd`, `respond_verify_foreign_tx`, `vote_pk`, `vote_reshared`, `vote_abort_key_event_instance`, `start_keygen_instance`, `start_reshare_instance`.

---

### Impact Explanation

**Scenario:**
1. A vulnerability is discovered in the MPC node software. Governance votes to remove the affected image hash from the allowed list via `vote_code_hash`.
2. The compromised node's `VerifiedAttestation` remains in `stored_attestations` — it is only removed when `verify_tee()` is called and triggers `reverify_and_cleanup_participants()`.
3. Until `verify_tee()` is called (window is unbounded; expected ~2 days but not enforced), the compromised node passes `is_caller_an_attested_participant()` because its attestation still *exists* in storage.
4. The compromised node can call `respond()` to submit signature responses, `vote_pk()` to influence key generation, `vote_reshared()` to influence resharing, and `start_keygen_instance()` / `start_reshare_instance()` as leader.

The TEE attestation is the system's guarantee that only nodes running approved software participate in threshold operations. If a node's image hash is revoked but it can still call `respond()` and key-event vote methods, the TEE security boundary is broken for the duration of the window. A node running software with a known vulnerability that enables malicious signature production or key manipulation can continue to participate in the protocol.

This maps to the allowed impact: **participant/attestation authorization bypass** that enables a node with a revoked TEE attestation to continue exercising signing capability and key-event voting rights.

---

### Likelihood Explanation

**Medium.** The attack requires:
1. A governance action to revoke an image hash (realistic — this is the intended upgrade/revocation path).
2. A window before `verify_tee()` is called (realistic — `verify_tee()` is not automatically triggered; it depends on participants calling it).

The window is not bounded by the contract. If participants are slow to call `verify_tee()`, or if the compromised node is the only one monitoring the contract, the window can extend indefinitely. The node's own `monitor_attestation_removal` task would detect removal from `get_tee_accounts()`, but that only fires after `verify_tee()` has already been called and cleaned up the entry.

---

### Recommendation

Add a call to `reverify_participants()` inside `is_caller_an_attested_participant()` (or in `assert_caller_is_attested_participant_and_protocol_active()`) so that the attestation's current validity — expiry, image hash, measurements — is checked at every execution, not just at registration:

```rust
pub(crate) fn is_caller_an_attested_participant(
    &self,
    participants: &Participants,
    tee_upgrade_deadline_duration: Duration,  // pass from config
) -> Result<(), AttestationCheckError> {
    // ... existing key/account checks ...

    // Re-verify the attestation is still valid right now
    let node_id = &attestation.node_id;
    match self.reverify_participants(node_id, tee_upgrade_deadline_duration) {
        TeeQuoteStatus::Valid => Ok(()),
        TeeQuoteStatus::Invalid(reason) => Err(AttestationCheckError::AttestationExpiredOrRevoked(reason)),
    }
}
```

This mirrors the ERC-7484 recommendation: "Query the Registry about module A on execution of A."

---

### Proof of Concept

1. Deploy the contract with 3 participants, all with valid attestations.
2. Governance votes to remove participant 1's image hash from the allowed list.
3. Do **not** call `verify_tee()`.
4. Participant 1's attestation remains in `stored_attestations` (it was stored at registration and has not been cleaned up).
5. Participant 1 calls `vote_pk(key_event_id, public_key)`.
6. `assert_caller_is_attested_participant_and_protocol_active()` calls `is_caller_an_attested_participant()`.
7. `is_caller_an_attested_participant()` finds the attestation in `stored_attestations`, checks keys match → returns `Ok(())`.
8. The vote is accepted, despite participant 1 running software whose image hash is no longer on the allowed list.

The gap is confirmed by the fact that `reverify_participants()` exists and correctly rejects expired/revoked attestations: [7](#0-6) 

but is never called from `is_caller_an_attested_participant()`: [3](#0-2)

    pub(crate) fn reverify_participants(
        &self,
        node_id: &NodeId,
        tee_upgrade_deadline_duration: Duration,
    ) -> TeeQuoteStatus {
        let allowed_mpc_docker_image_hashes =
            self.get_allowed_mpc_docker_image_hashes(tee_upgrade_deadline_duration);
        let allowed_launcher_compose_hashes = self.get_allowed_launcher_compose_hashes();
        let allowed_measurements = self.get_accepted_measurements();

        let participant_attestation = self.stored_attestations.get(&node_id.tls_public_key);
        let Some(participant_attestation) = participant_attestation else {
            return TeeQuoteStatus::Invalid("participant has no attestation".to_string());
        };

        // Verify the attestation quote
        let time_stamp_seconds = Self::current_time_seconds();
        match participant_attestation.verified_attestation.re_verify(
            time_stamp_seconds,
            &allowed_mpc_docker_image_hashes,
            &allowed_launcher_compose_hashes,
            &allowed_measurements,
        ) {
            Ok(()) => TeeQuoteStatus::Valid,
            Err(err) => TeeQuoteStatus::Invalid(err.to_string()),
        }
    }

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }

    pub fn verify_tee(&mut self) -> Result<bool, Error> {
        log!("verify_tee: signer={}", env::signer_account_id());
        // Caller must be a participant (node or operator).
        self.voter_or_panic();
        let ProtocolContractState::Running(running_state) = &mut self.protocol_state else {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        };
        let current_params = running_state.parameters.clone();

        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);

        match self.tee_state.reverify_and_cleanup_participants(
            current_params.participants(),
            tee_upgrade_deadline_duration,
        ) {
            TeeValidationResult::Full => {
                self.accept_requests = true;
                log!("All participants have an accepted Tee status");
                Ok(true)
            }
