Audit Report

## Title
Stale agETH Rate in `CrossChainRateReceiver` Allows Over-Minting of agETH via `AGETHPoolV3.deposit` — (`contracts/agETH/AGETHPoolV3.sol`)

## Summary

`AGETHPoolV3` mints agETH using a rate fetched from `AGETHRateReceiver` (a `CrossChainRateReceiver`) via LayerZero push updates. Because `CrossChainRateReceiver.getRate()` returns the stored rate unconditionally with no staleness guard, and `updateRate()` is not automatic, a stale (under-valued) agETH rate causes every deposit to mint more agETH than the deposited assets are worth. An unprivileged attacker can exploit any rate-lag window to extract unbacked agETH, leading to protocol insolvency.

## Finding Description

**Rate update is push-based and not automatic:**

`MultiChainRateProvider.updateRate()` is permissionless but requires the caller to supply ETH for LayerZero fees. [1](#0-0) 

It pushes the current agETH rate to all registered `CrossChainRateReceiver` contracts on L2 via LayerZero. [2](#0-1) 

**L2 receiver stores rate with no expiry:**

`CrossChainRateReceiver` stores both `rate` and `lastUpdated`, but `getRate()` returns `rate` unconditionally — no staleness guard exists. [3](#0-2) [4](#0-3) 

**`AGETHPoolV3` uses the stale rate directly for minting:**

`AGETHPoolV3.getRate()` calls `IOracle(agETHOracle).getRate()` with no freshness check. [5](#0-4) 

For ETH deposits, `agETHAmount = amountAfterFee * 1e18 / agETHToETHrate`. A stale-low `agETHToETHrate` inflates the output. [6](#0-5) 

For token deposits, `agETHAmount = amountAfterFee * tokenToETHRate / agETHToETHrate`. If the token oracle is current but the agETH oracle is stale-low, the ratio is inflated. [7](#0-6) 

**Exploit arithmetic (token deposit path):**

| Variable | Stale scenario | Correct scenario |
|---|---|---|
| `agETHToETHrate` (L2 receiver) | `1.00e18` (stale) | `1.05e18` (true) |
| `tokenToETHRate` (wstETH oracle) | `1.05e18` (current) | `1.05e18` |
| `agETHAmount` per 1e18 wstETH | `1.05e18` | `1.00e18` |
| Over-mint | **+5%** | — |

For ETH deposits the same over-mint occurs: `1e18 * 1e18 / 1.00e18 = 1e18` vs. the correct `1e18 * 1e18 / 1.05e18 ≈ 0.952e18`.

No existing check in `AGETHPoolV3`, `CrossChainRateReceiver`, or the `IOracle` interface guards against a stale rate.

## Impact Explanation

Every deposit during a stale-rate window mints agETH unbacked by equivalent ETH value. The attacker can immediately sell or redeem the surplus agETH, extracting value from the protocol's reserves. Repeated exploitation proportional to the rate lag causes **protocol insolvency** (Critical). The `lastUpdated` field is available on `CrossChainRateReceiver` but is never consulted by `AGETHPoolV3`. [8](#0-7) 

## Likelihood Explanation

- agETH is yield-bearing, so its rate monotonically increases; staleness always means under-pricing.
- `updateRate()` requires an off-chain keeper supplying ETH; any gap (network congestion, fee shortage, keeper downtime) creates an exploitable window.
- The attacker needs only to compare `CrossChainRateReceiver.lastUpdated` and `CrossChainRateReceiver.rate` on-chain against the live L1 rate, then call `deposit()` when divergence is profitable.
- No privileged access, no front-running dependency, no external protocol compromise required. [9](#0-8) 

## Recommendation

1. **Add a staleness check in `AGETHPoolV3.getRate()`**: revert if `block.timestamp - IOracle(agETHOracle).lastUpdated() > MAX_RATE_AGE`.
2. **Expose `lastUpdated` from `CrossChainRateReceiver`** via the `IOracle` interface (it already stores it at line 16) and check it before minting.
3. **Pause deposits** if the agETH oracle rate is stale beyond a configurable threshold. [5](#0-4) 

## Proof of Concept

```solidity
// Fork test (L2 fork, e.g., Arbitrum)
// Precondition: AGETHRateReceiver.rate = 1.00e18 (stale), true rate = 1.05e18
// wstETH oracle returns 1.05e18

function testStaleRateOverMint() public {
    // Simulate stale receiver: warp time forward without calling updateRate()
    vm.warp(block.timestamp + 7 days);

    uint256 depositAmount = 1e18; // 1 wstETH
    deal(address(wstETH), attacker, depositAmount);

    vm.startPrank(attacker);
    wstETH.approve(address(pool), depositAmount);
    pool.deposit(address(wstETH), depositAmount, "");
    vm.stopPrank();

    uint256 agETHMinted = agETH.balanceOf(attacker);
    // agETHMinted = 1e18 * 1.05e18 / 1.00e18 = 1.05e18
    // Correct amount = 1e18 * 1.05e18 / 1.05e18 = 1.00e18
    assertGt(agETHMinted, 1e18, "Over-minted agETH due to stale rate");
}
```

The `lastUpdated` field is available on `CrossChainRateReceiver` at line 16 but is never consulted by `AGETHPoolV3`. [8](#0-7) [5](#0-4)
