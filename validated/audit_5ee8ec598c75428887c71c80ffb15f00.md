Audit Report

## Title
Synthetic Ratio Precision Loss in `_getBidAndAskPrice()` Silently Produces Zero Mid, Blocking All Pool Swaps — (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

## Summary
In the two-feed synthetic ratio mode, `_getBidAndAskPrice()` computes `mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2)` with `ORACLE_DECIMALS = 1e8`. When the base feed price is less than `1e-8` of the quote feed price, integer division floors the result to zero. No post-division zero-check exists, so `_computeBidAsk(0, spreadBps)` propagates the stall sentinel `(0, type(uint128).max)`, causing `getBidAndAskPrice()` to revert with `FeedStalled`, which the pool re-throws as `PriceProviderFailed` on every `swap()` call.

## Finding Description

In `_getBidAndAskPrice()`, the synthetic ratio is computed at line 267 with no guard afterward:

```solidity
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);  // line 267
spreadBps += spreadBps2;
// no zero-check on mid here
return _computeBidAsk(mid, spreadBps);           // line 271
``` [1](#0-0) 

The existing guard at line 265 only rejects `mid2 == 0` and a failed leg read — it does not protect against `mid` being too small relative to `mid2` to survive the 8-decimal division:

```solidity
if (!ok2 || mid2 == 0) return (0, type(uint128).max);
``` [2](#0-1) 

When `mid == 0` reaches `_computeBidAsk`, `_bandEdge(0, ...)` returns 0 for both `refBid` and `refAsk`, and the guard at line 311 triggers the stall sentinel:

```solidity
if (refBid == 0 || refAsk > type(uint128).max || refBid >= refAsk) {
    return (0, type(uint128).max);
}
``` [3](#0-2) 

`getBidAndAskPrice()` then reverts:

```solidity
if (bid == 0 || ask == type(uint128).max) revert FeedStalled();
``` [4](#0-3) 

`_getBidAndAskPriceX64()` catches and re-throws as `PriceProviderFailed`:

```solidity
} catch (bytes memory reason) {
    revert PriceProviderFailed(reason);
}
``` [5](#0-4) 

`swap()` calls `_getBidAndAskPriceX64()` as its very first action, so every swap reverts for as long as the price ratio remains below `1e-8`:

```solidity
(uint128 bidPriceX64, uint128 askPriceX64) = _getBidAndAskPriceX64();
``` [6](#0-5) 

## Impact Explanation
All swaps on any pool whose `AnchoredPriceProvider` is configured in synthetic ratio mode (`quoteFeedId != bytes32(0)`) are completely blocked whenever the base/quote price ratio falls below `1e-8`. This satisfies the "Broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows" impact gate: LPs cannot rebalance through swaps and traders cannot execute any trade.

## Likelihood Explanation
The condition `mid * 1e8 < mid2` is reachable without any privileged action through normal market prices. Example: base = SHIB at $0.00001 → oracle price `mid = 1_000` (8-decimal); quote = BTC at $60,000 → oracle price `mid2 = 6_000_000_000_000`. Then `Math.mulDiv(1_000, 1e8, 6e12) = 1e11 / 6e12 = 0`. Both individual legs pass all `_readLeg` guards (non-zero, fresh, within price guard bounds), so the only failure point is the missing post-ratio zero-check. Any pool using synthetic ratio mode with a sufficiently asymmetric pair is vulnerable, and no attacker input is required — the condition arises from ordinary market prices.

## Recommendation
Add a zero-check immediately after the ratio division in `_getBidAndAskPrice()`:

```solidity
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
if (mid == 0) return (0, type(uint128).max); // ratio underflows 8-decimal precision
```

This fails closed (identical behavior to a stale feed) rather than silently passing a zero mid into `_computeBidAsk`. Alternatively, scale `mid` to a higher intermediate precision before dividing to preserve sub-`1e-8` ratios, but the zero-check is the minimal safe fix.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.28;
import {Math} from "@openzeppelin/contracts/utils/math/Math.sol";

contract PoC {
    uint256 constant ORACLE_DECIMALS = 1e8;
    function demonstrateZeroRatio() external pure returns (uint256 syntheticMid) {
        uint256 mid  = 1_000;               // SHIB at $0.00001 → 8-decimal oracle price
        uint256 mid2 = 6_000_000_000_000;   // BTC  at $60,000  → 8-decimal oracle price
        syntheticMid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
        // Returns 0: 1e11 / 6e12 = 0
        // → _computeBidAsk(0, spreadBps) → (0, type(uint128).max)
        // → getBidAndAskPrice() reverts FeedStalled
        // → swap() reverts PriceProviderFailed for every caller
    }
}
```

Deploy against a fork, configure an `AnchoredPriceProvider` with `baseFeedId` = SHIB/USD and `quoteFeedId` = BTC/USD, seed the mock oracle with the above prices, and call `pool.swap(...)` — it reverts with `PriceProviderFailed` on every attempt.
