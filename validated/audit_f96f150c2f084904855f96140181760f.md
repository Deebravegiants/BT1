### Title
Cross-Network Signature Replay of `SignedDelegateAction` Due to Missing Chain-Specific Identifier in Signed Payload - (File: `core/primitives/src/signable_message.rs`, `core/primitives/src/action/delegate.rs`)

---

### Summary

The `DelegateAction` (meta-transaction) signed payload in nearcore does not include any chain-specific identifier. The `SignableMessage` discriminant is a fixed NEP number constant, not bound to a chain ID or genesis hash. A `SignedDelegateAction` signed on NEAR mainnet can be replayed on NEAR testnet (or any other NEAR-based network) by an unprivileged attacker, provided the sender account, public key, and nonce match on the target network — a realistic condition for new accounts or fresh keys.

---

### Finding Description

When a user creates a meta-transaction, they sign a `DelegateAction` struct. The signing hash is computed in `DelegateAction::get_nep461_hash()`:

```rust
pub fn get_nep461_hash(&self) -> CryptoHash {
    let signable = SignableMessage::new(&self, SignableMessageType::DelegateAction);
    let bytes = borsh::to_vec(&signable).expect("Failed to deserialize");
    hash(&bytes)
}
```

`SignableMessage` prepends a `MessageDiscriminant` to the payload before hashing. The discriminant is computed as:

```rust
MessageDiscriminant::new_on_chain(NEP_366_META_TRANSACTIONS).unwrap()
// = MIN_ON_CHAIN_DISCRIMINANT + 366 = (1 << 30) + 366
```

This is a **fixed compile-time constant** — it contains no chain ID, no genesis hash, and no network-specific value. The full signed payload is:

```
hash( borsh( (1<<30)+366 || sender_id || receiver_id || actions || nonce || max_block_height || public_key ) )
```

None of these fields are chain-specific. By contrast, regular `SignedTransaction` uses a `block_hash` field that is inherently chain-specific (different chains produce different block hashes), providing natural cross-chain replay protection. `DelegateAction` replaces `block_hash` with `max_block_height` (a plain integer), which is **not** chain-specific.

The same structural gap exists in `DelegateActionV2` / `VersionedDelegateActionPayload::get_nep461_hash()`, which uses `SignableMessageType::DelegateActionV2` — also a fixed constant `(1<<30)+611`.

---

### Impact Explanation

An attacker who observes a `SignedDelegateAction` submitted on NEAR mainnet (all RPC submissions are public) can replay it verbatim on NEAR testnet (or any other NEAR-based network) by submitting it via the standard `broadcast_tx_async` / `broadcast_tx_commit` RPC endpoint. If the replay succeeds, the inner actions execute on the target chain with `sender_id` as the predecessor — meaning the user's account on the target chain is the apparent authorizer.

Concrete corrupted state values:
- The sender's access-key nonce on the target chain is incremented (consuming a nonce the user never intended to use there).
- All inner `actions` execute on the target chain: token transfers, key additions/deletions, contract calls, etc., all attributed to the user's account without their consent.

---

### Likelihood Explanation

The replay requires:
1. **Same account ID on both networks** — NEAR account IDs are human-readable strings (e.g., `alice.near`). Users routinely hold the same account ID on mainnet and testnet.
2. **Same public key on both networks** — Users commonly reuse key pairs across networks.
3. **Matching nonce on the target network** — For a fresh account or a key that has not yet been used for delegate actions on the target network, the nonce starts at 0 (or the last used value). If the user has nonce `N` on mainnet and the attacker replays a delegate action with nonce `N+1`, it will be accepted on testnet if testnet nonce is also `N`.
4. **`max_block_height` not expired on target network** — Both mainnet and testnet block heights are in the tens of millions; a `max_block_height` set for mainnet is almost certainly also valid on testnet.

Conditions 1 and 2 are the norm for developers and power users. Condition 3 is trivially satisfied for any account that has not yet used delegate actions on the target network. Condition 4 is almost always satisfied given the similar block heights across NEAR networks.

---

### Recommendation

Include a chain-specific identifier in the `SignableMessage` or `DelegateAction` payload. The standard approach is to incorporate the genesis block hash or a chain ID into the discriminant or as an explicit field in `DelegateAction`. For example:

- Add a `chain_id: CryptoHash` field to `DelegateAction` (set to the genesis hash of the network), verified at execution time against the node's known genesis hash.
- Or incorporate the genesis hash into the `MessageDiscriminant` computation so the signing domain is network-specific.

Regular `SignedTransaction` already achieves this implicitly via `block_hash`; `DelegateAction` should adopt an equivalent mechanism.

**Exploit path:**
1. Alice signs a `DelegateAction` on NEAR mainnet (e.g., to transfer FT tokens). The relayer submits it; the `SignedDelegateAction` is visible in the transaction.
2. Attacker copies the `SignedDelegateAction` bytes verbatim.
3. Attacker submits it to NEAR testnet via RPC. If Alice has the same account and key on testnet with a matching nonce, the runtime's `apply_delegate_action` accepts it — the signature verifies identically because the signed hash is chain-agnostic.
4. Alice's inner actions execute on testnet without her consent; her testnet nonce is consumed.

impl From<SignableMessageType> for MessageDiscriminant {
    fn from(ty: SignableMessageType) -> Self {
        // unwrapping here is ok, we know the constant NEP numbers used are in range
        match ty {
            SignableMessageType::DelegateAction => {
                MessageDiscriminant::new_on_chain(NEP_366_META_TRANSACTIONS).unwrap()
            }
            SignableMessageType::DelegateActionV2 => {
                MessageDiscriminant::new_on_chain(NEP_611_GAS_KEYS).unwrap()
            }
        }
    }
