### Title
Unvalidated `ProposalInit.builder` Field Allows Byzantine Proposer to Inject Arbitrary Sequencer Address into Block Execution — (`crates/apollo_consensus_orchestrator/src/utils.rs`)

---

### Summary

`ProposalInit` carries a `builder` field (the address of the block builder/sequencer). `convert_to_sn_api_block_info` maps it directly to `BlockInfo.sequencer_address`, which is passed to the batcher for both proposal building and validation. `is_proposal_init_valid` validates many `ProposalInit` fields but never checks `builder`. A Byzantine proposer can therefore set `builder` to any arbitrary address; every honest validator will accept the proposal, execute the block with the attacker-controlled sequencer address, and commit a block whose hash encodes the wrong sequencer.

---

### Finding Description

`ProposalInit` contains two identity fields:

- `proposer` — the consensus participant who proposed the block. This **is** validated in `handle_proposal` against the committee's expected proposer.
- `builder` — the address that builds/sequences the block and receives transaction fees. This is **never** validated anywhere.

`convert_to_sn_api_block_info` consumes `init.builder` directly:

```rust
// crates/apollo_consensus_orchestrator/src/utils.rs  line 329-347
Ok(starknet_api::block::BlockInfo {
    block_number: init.height,
    block_timestamp: BlockTimestamp(init.timestamp),
    sequencer_address: init.builder,   // ← proposer-supplied, unvalidated
    gas_prices: GasPrices { ... },
    use_kzg_da: init.l1_da_mode.is_use_kzg_da(),
    starknet_version: init.starknet_version,
})
``` [1](#0-0) 

This `BlockInfo` is forwarded to the batcher via `ValidateBlockInput.block_info` in `initiate_validation`:

```rust
// crates/apollo_consensus_orchestrator/src/validate_proposal.rs  line 455-466
let input = ValidateBlockInput {
    proposal_id,
    deadline: ...,
    retrospective_block_hash: ...,
    block_info: convert_to_sn_api_block_info(init)?,
};
batcher.validate_block(input.clone()).await...
``` [2](#0-1) 

`is_proposal_init_valid` checks `timestamp`, `starknet_version`, `version_constant_commitment`, `height`, `l1_da_mode`, `l2_gas_price_fri`, L1 gas prices, and `fee_proposal_fri` — but **not** `builder`: [3](#0-2) 

The `proposer` field is validated in `handle_proposal` in the consensus manager:

```rust
// crates/apollo_consensus/src/manager.rs  line 860-866
if proposer != init.proposer {
    warn!("Invalid proposer for height {} and round {}: expected {:?}, got {:?}", ...);
    return Ok(VecDeque::new());
}
``` [4](#0-3) 

There is no equivalent check for `builder`. The `ProposalInit` struct definition confirms both fields exist on the wire: [5](#0-4) [6](#0-5) 

The `builder` field is also included in `PartialBlockHashComponents.sequencer`, which feeds into the `partial_block_hash` and ultimately the `ProposalCommitment`: [7](#0-6) 

Because both proposer and validator derive `ProposalCommitment` from the same proposer-supplied `init.builder`, the final `built_block != received_fin.proposal_commitment` check passes regardless of what `builder` contains. [8](#0-7) 

---

### Impact Explanation

1. **Wrong `get_sequencer_address` syscall result** — `BlockInfo.sequencer_address` is the value returned by the `get_sequencer_address` syscall. Smart contracts that gate logic on the sequencer address (e.g., privileged operations, fee-routing contracts) receive the attacker-controlled value. This is a wrong execution result from syscall logic for accepted input.

2. **Incorrect fee accounting** — Transaction fees are collected to `sequencer_address`. A Byzantine proposer who sets `builder` to their own address diverts all fees from that block to themselves. This is an incorrect fee/balance effect with direct economic impact.

3. **Wrong block hash committed to chain** — `PartialBlockHashComponents.sequencer` is derived from `init.builder`. The `partial_block_hash` and therefore the final `BlockHash` encode the attacker-supplied address. Every downstream consumer of the block hash (L1 anchoring, state sync, proof verification) receives a hash that encodes a false sequencer identity.

---

### Likelihood Explanation

Requires a Byzantine proposer — a consensus participant who deviates from the protocol. This is within the explicit BFT threat model (up to `f` Byzantine nodes). No external unprivileged attacker is needed; any single Byzantine validator who wins a proposal round can trigger this. The attack is silent: honest validators log no warning and commit the block normally.

---

### Recommendation

Add a `builder` check to `is_proposal_init_valid`. The validator node knows its own expected `builder_address` (the same value the proposer reads from `args.builder_address`). Reject any `ProposalInit` whose `builder` does not match the locally configured value:

```rust
if init_proposed.builder != expected_builder_address {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected={:?}, proposed={:?}",
            expected_builder_address, init_proposed.builder
        ),
    ));
}
```

`ProposalInitValidation` should carry the expected `builder` address, populated from the node's static configuration, mirroring how `l1_da_mode` and `l2_gas_price_fri` are already carried there. [9](#0-8) 

---

### Proof of Concept

1. Byzantine proposer node modifies `initiate_build` to set `init.builder = attacker_address` instead of `args.builder_address`.
2. The proposer broadcasts `ProposalPart::Init(init)` with the crafted `builder` field.
3. Each honest validator calls `validate_proposal` → `is_proposal_init_valid` → passes (no `builder` check).
4. `convert_to_sn_api_block_info(init)` produces `BlockInfo { sequencer_address: attacker_address, ... }`.
5. `batcher.validate_block(ValidateBlockInput { block_info, ... })` executes the block with `sequencer_address = attacker_address`.
6. All transaction fees are credited to `attacker_address`; `get_sequencer_address` syscall returns `attacker_address`.
7. `PartialBlockHashComponents { sequencer: attacker_address, ... }` → `partial_block_hash` encodes the wrong sequencer.
8. `ProposalCommitment = Poseidon(partial_block_hash, fee_proposal_fri)` — both proposer and validator compute the same value, so `built_block == received_fin.proposal_commitment` passes.
9. The block is committed to the chain with the wrong sequencer address in the block hash and wrong fee recipient in the state diff. [10](#0-9) [11](#0-10)

#[instrument(level = "warn", skip_all, fields(?proposal_init_validation, ?init_proposed))]
async fn is_proposal_init_valid(
    proposal_init_validation: &ProposalInitValidation,
    init_proposed: &ProposalInit,
    clock: &dyn Clock,
    l1_gas_price_provider: Arc<dyn L1GasPriceProviderClient>,
    gas_price_params: &GasPriceParams,
) -> ValidateProposalResult<()> {
    let now: u64 = clock.unix_now();
    let last_block_timestamp =
        proposal_init_validation.previous_proposal_init.as_ref().map_or(0, |info| info.timestamp);
    if init_proposed.timestamp < last_block_timestamp {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "Timestamp is too old: last_block_timestamp={}, proposed={}",
                last_block_timestamp, init_proposed.timestamp
            ),
        ));
    }
    if init_proposed.timestamp > now + proposal_init_validation.block_timestamp_window_seconds {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "Timestamp is in the future: now={}, block_timestamp_window_seconds={}, \
                 proposed={}",
                now,
                proposal_init_validation.block_timestamp_window_seconds,
                init_proposed.timestamp
            ),
        ));
    }
    if init_proposed.starknet_version != proposal_init_validation.starknet_version {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "starknet_version mismatch: expected={:?}, proposed={:?}",
                proposal_init_validation.starknet_version, init_proposed.starknet_version
            ),
        ));
    }
    // `version_constant_commitment` is proposer-supplied (network-derived). It is not yet a real
    // commitment (see `expected_version_constant_commitment`): the only valid value is the
    // sentinel, so reject anything else. Enforcing the same value the proposer emits keeps the two
    // sides in lockstep, so a real value cannot ship on one side without the other.
    let expected_commitment = expected_version_constant_commitment();
    if init_proposed.version_constant_commitment != expected_commitment {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "version_constant_commitment mismatch: expected={expected_commitment:?}, \
                 proposed={:?}",
                init_proposed.version_constant_commitment
            ),
        ));
    }
    if !(init_proposed.height == proposal_init_validation.height
        && init_proposed.l1_da_mode == proposal_init_validation.l1_da_mode
        && init_proposed.l2_gas_price_fri == proposal_init_validation.l2_gas_price_fri)
    {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            "ProposalInit validation failed".to_string(),
