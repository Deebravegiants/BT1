### Title
Stale rsETH Oracle Price Masked by Fresh ETH/USD `updatedAt` in Composite Price Feed — (`File: contracts/oracles/RSETHPriceFeed.sol`)

---

### Summary

`RSETHPriceFeed.latestRoundData()` and `getRoundData()` return a composite `answer` that multiplies the ETH/USD Chainlink price by `RS_ETH_ORACLE.rsETHPrice()`, but the `updatedAt` timestamp returned is sourced exclusively from the ETH/USD Chainlink feed. If the rsETH oracle price becomes stale (e.g., `LRTOracle.updateRSETHPrice()` is not called for an extended period), downstream consumers checking `updatedAt` for freshness will be misled into trusting an outdated rsETH price.

---

### Finding Description

`RSETHPriceFeed` implements `AggregatorV3Interface` and is designed to be consumed by external DeFi protocols (lending markets, etc.) as a Chainlink-compatible rsETH/USD price feed.

In `latestRoundData()`:

```solidity
function latestRoundData()
    external view
    returns (uint80 roundId, int256 answer, uint256 startedAt, uint256 updatedAt, uint80 answeredInRound)
{
    (roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.latestRoundData();
    answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
}
```

The `answer` is a product of two independent data sources:
1. **ETH/USD** — from `ETH_TO_USD` (a live Chainlink feed, updated every heartbeat)
2. **rsETH/ETH** — from `RS_ETH_ORACLE.rsETHPrice()`, which is the `LRTOracle.rsETHPrice` storage variable

`LRTOracle.rsETHPrice` is a stored state variable updated only when `updateRSETHPrice()` is called (a public function requiring an off-chain keeper). The `CrossChainRateReceiver.rate` (used on L2 deployments) is similarly a stored variable updated only when a LayerZero message arrives.

Neither oracle exposes its own `lastUpdated` timestamp to `RSETHPriceFeed`. The `updatedAt` returned is solely from the ETH/USD Chainlink feed, which is updated independently and frequently. The rsETH oracle's staleness is completely invisible to the caller.

The same flaw exists in `getRoundData()`:

```solidity
(roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.getRoundData(_roundId);
answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
```

---

### Impact Explanation

**Critical — Direct theft of user funds.**

External lending protocols (e.g., Aave, Compound, or any protocol using this feed as a Chainlink oracle) perform a staleness check on `updatedAt`. Because `updatedAt` reflects only the ETH/USD feed's freshness, the staleness check passes even when `rsETHPrice` is arbitrarily old.

Attack scenario: rsETH suffers a slashing event on EigenLayer, causing the true rsETH/ETH rate to drop. If `updateRSETHPrice()` is not called promptly (keeper delay, network congestion, or deliberate griefing of the keeper), `LRTOracle.rsETHPrice` remains at the pre-slashing (inflated) value. `RSETHPriceFeed.latestRoundData()` returns a fresh `updatedAt` (from ETH/USD) with an inflated `answer`. rsETH holders can then borrow against their collateral at the inflated price in any lending protocol using this feed, extracting funds from lenders that cannot be recovered once the price corrects and liquidations are triggered at the true lower value.

---

### Likelihood Explanation

**Low-Medium.** The rsETH oracle is updated by a public function (`updateRSETHPrice()`) that any address can call, but in practice it depends on a keeper. Keeper delays are realistic during network congestion, infrastructure outages, or deliberate transaction spam. The ETH/USD Chainlink feed updates independently on its own heartbeat, so the divergence window (fresh `updatedAt`, stale rsETH price) is realistic and exploitable by any rsETH holder monitoring the oracle state.

---

### Recommendation

`RSETHPriceFeed` should track the rsETH oracle's last update timestamp and return `min(ethToUSD_updatedAt, rsETH_lastUpdated)` as the `updatedAt` value. Concretely:

1. Extend `IRSETHOracle` to expose a `lastUpdated()` view function.
2. In `LRTOracle`, record `block.timestamp` into a `lastUpdated` variable each time `rsETHPrice` is written in `_updateRsETHPrice()`.
3. In `RSETHPriceFeed.latestRoundData()` and `getRoundData()`, override `updatedAt` with `min(updatedAt_from_chainlink, RS_ETH_ORACLE.lastUpdated())`.

This ensures that any consumer performing a staleness check on `updatedAt` will correctly detect when the rsETH component of the price is stale.

---

### Proof of Concept

1. `RSETHPriceFeed` is deployed pointing to `ETH_TO_USD` (Chainlink ETH/USD, heartbeat = 1 hour) and `RS_ETH_ORACLE` (`LRTOracle`).
2. At time T, `LRTOracle.rsETHPrice = 1.05e18` (1 rsETH = 1.05 ETH). `updateRSETHPrice()` is last called at T.
3. At time T+6h, a slashing event reduces the true rsETH/ETH rate to 0.95. The keeper fails to call `updateRSETHPrice()`.
4. At time T+6h, `ETH_TO_USD.latestRoundData()` returns `updatedAt = T+6h` (fresh, within heartbeat).
5. `RSETHPriceFeed.latestRoundData()` returns:
   - `updatedAt = T+6h` ← appears fresh to any consumer
   - `answer = 1.05e18 * ETH_USD_price / 1e18` ← inflated by stale rsETH price
6. A lending protocol checks `block.timestamp - updatedAt < heartbeat` → passes. It prices rsETH collateral at the inflated rate.
7. An attacker deposits rsETH and borrows the maximum allowed at the inflated price, extracting funds from the lending pool before the true price is reflected and liquidations can occur.

**Root cause lines:** [1](#0-0) 

`updatedAt` is taken directly from `ETH_TO_USD.latestRoundData()` and returned unchanged, while `answer` incorporates `RS_ETH_ORACLE.rsETHPrice()` whose staleness is not reflected. [2](#0-1) 

The same pattern in `getRoundData()`. [3](#0-2) 

`updateRSETHPrice()` is a public function requiring an external call to update `rsETHPrice`; no on-chain enforcement of update frequency exists. [4](#0-3) 

`rsETHPrice` is written here with no associated timestamp stored, making it impossible for `RSETHPriceFeed` to know when it was last updated.
