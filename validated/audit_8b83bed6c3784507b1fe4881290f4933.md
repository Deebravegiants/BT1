### Title
`addPubkeys` Permanently Blocks `stake32Eth` for Pre-Registered Pubkeys, Causing ETH to Sit Idle — (`contracts/PubkeyRegistry.sol`)

---

### Summary

The manager-callable `addPubkeys` function in `PubkeyRegistry` writes `true` into the same `pubkeyRegistry` mapping that `stake32Eth` uses as a duplicate-staking guard. Because there is no `removePubkey` function, any pubkey pre-registered via `addPubkeys` is permanently blocked from the `stake32Eth` path, causing the 32 ETH earmarked for that validator to sit idle in the NDC without earning restaking yield.

---

### Finding Description

`PubkeyRegistry` exposes two write paths into the same `mapping(bytes32 => bool) pubkeyRegistry`:

- `addPubkey` — callable only by an LRT NodeDelegator, invoked inside `stake32Eth` **after** the ETH is staked.
- `addPubkeys` — callable by the LRT Manager, with no staking side-effect. [1](#0-0) 

`stake32Eth` in `NodeDelegator` checks `hasPubkey` before doing anything: [2](#0-1) 

If the manager calls `addPubkeys([pubkeyX])`, `pubkeyRegistry[keccak256(pubkeyX)]` is set to `true`. Subsequently, any operator call to `stake32Eth(pubkeyX, ...)` hits the `hasPubkey` guard and reverts with `PubkeyAlreadyRegistered`. The ETH that was transferred to the NDC for that validator can never be staked through the normal path.

There is no `removePubkey` or equivalent function anywhere in the contract, so the block is **irreversible** without an upgrade. [3](#0-2) 

---

### Impact Explanation

ETH sitting in the NDC earmarked for a pre-registered-but-never-staked validator earns zero restaking yield. The funds are not lost (they remain in the NDC), but the protocol fails to deliver the promised restaking returns for those validators. This matches the **Low** scope: *"Contract fails to deliver promised returns, but doesn't lose value."* [4](#0-3) 

---

### Likelihood Explanation

The `addPubkeys` function is a legitimate, documented manager operation. A plausible scenario is a manager pre-registering a batch of validator pubkeys (e.g., to reserve them or migrate them from an external staking setup) before the corresponding 32 ETH deposits are ready. Once called, the staking path for those pubkeys is permanently closed. No key compromise or malicious intent is required — a routine operational call is sufficient.

---

### Recommendation

1. **Add a `removePubkey` / `removePubkeys` function** (manager-gated) to allow reversing pre-registrations that were not accompanied by an actual stake.
2. **Alternatively**, document clearly that `addPubkeys` must only be called for pubkeys that have already been staked outside the protocol (e.g., migrated validators), and add a NatSpec warning that calling it for unstaked pubkeys permanently blocks staking.
3. **Consider separating the two registries**: one for "staked" pubkeys (written only by `stake32Eth`) and one for "blocked/migrated" pubkeys (written by the manager), so `stake32Eth` only checks the former.

    function hasPubkey(bytes calldata pubkey) external view returns (bool) {
        return pubkeyRegistry[keccak256(pubkey)];
    }

    function addPubkey(bytes calldata pubkey) external onlyLRTNodeDelegator {
        pubkeyRegistry[keccak256(pubkey)] = true;
    }

    function addPubkeys(bytes[] calldata pubkeys) external onlyLRTManager {
        for (uint256 i = 0; i < pubkeys.length; i++) {
            pubkeyRegistry[keccak256(pubkeys[i])] = true;
        }
    }
