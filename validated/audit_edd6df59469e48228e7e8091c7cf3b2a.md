### Title
Caller-Controlled Finality Level Accepted Without Minimum Enforcement Enables MPC Attestation of Non-Final Foreign-Chain Transactions - (File: `crates/contract/src/lib.rs`, `crates/foreign-chain-inspector/src/evm/inspector.rs`, `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`)

---

### Summary

The `verify_foreign_transaction` contract entry point accepts a caller-supplied finality parameter (`EvmFinality::Latest`, `BlockConfirmations(1)`, `SolanaFinality::Processed`) with no minimum enforcement at either the contract or node layer. The MPC network will produce a threshold signature attesting to a transaction that has not achieved economic finality on the foreign chain. A reorganization of that chain after signing yields a valid MPC-signed attestation for a transaction that no longer exists on the canonical chain, enabling double-spend against any bridge contract that relies on the MPC attestation.

---

### Finding Description

`verify_foreign_transaction` is the public entry point for the foreign-chain verification flow. After checking domain purpose, gas, deposit, and chain availability, it enqueues the request verbatim — including the caller-supplied finality field — without any validation of the finality level: [1](#0-0) 

The `EvmFinality` enum exposes three variants, including `Latest` (the chain tip, subject to reorganization) and `Safe` (two epochs behind finalized on Ethereum, still reversible under adversarial conditions): [2](#0-1) 

For Bitcoin, `BlockConfirmations` is a plain `u64` with no lower bound: [3](#0-2) 

On the node side, `EvmInspector::verify_finality_level` faithfully uses whatever `EthereumFinality` the request carries. When `Latest` is requested, it queries `eth_getBlockByNumber("latest")` and accepts any receipt whose block number is ≤ the latest block — i.e., any included transaction, regardless of reorg risk: [4](#0-3) 

For Bitcoin, `BitcoinInspector::extract` enforces only `block_confirmations_threshold <= transaction_block_confirmation`. With a caller-supplied threshold of `1` (or even `0`), the check passes for any transaction with one confirmation: [5](#0-4) 

The signed payload (`ForeignTxSignPayloadV1`) includes the full `ForeignChainRpcRequest`, which embeds the finality field. The MPC signs `hash(request_with_finality_latest, observed_values)`. The signature is cryptographically valid and indistinguishable from one produced for a finalized transaction. [6](#0-5) 

No on-chain whitelist entry, no per-chain configuration, and no node-side policy reader enforces a minimum finality level. The design documentation acknowledges "Finality semantics: Finality definitions differ across chains; mapping them correctly is critical" as a risk, but provides no enforcement mechanism: [7](#0-6) 

---

### Impact Explanation

**High — forged foreign-chain verification that causes invalid bridge execution or double-spend conditions.**

The primary stated use case is the Omnibridge inbound flow: a foreign-chain deposit triggers `verify_foreign_transaction`, the MPC attests to the deposit, and NEAR-side funds are released. If the attested transaction is at `EvmFinality::Latest` and the Ethereum chain reorganizes before the NEAR-side settlement is final, the attacker holds both the NEAR-side payout and the original Ethereum funds (the deposit transaction no longer exists on the canonical chain). The MPC-signed attestation remains cryptographically valid and cannot be revoked after issuance.

---

### Likelihood Explanation

**Medium.** Any unprivileged NEAR account can call `verify_foreign_transaction` with `EvmFinality::Latest` or `BlockConfirmations(1)`. The attacker needs to:
1. Submit a transaction on the foreign chain.
2. Immediately call `verify_foreign_transaction` with the lowest finality level before the transaction is deep enough to be safe.
3. Obtain the MPC signature.
4. Trigger or wait for a chain reorganization that removes the original transaction.
5. Present the MPC signature to the bridge contract.

Step 4 is non-trivial on Ethereum post-merge (reorganizations are rare but not impossible, especially for `Latest`-level blocks). On Bitcoin with `BlockConfirmations(1)`, shallow reorgs are a known and historically observed event. The attack is realistic for any bridge that does not independently re-validate the finality level embedded in the signed payload.

---

### Recommendation

1. **Enforce a minimum finality level in the contract**: Reject `verify_foreign_transaction` requests whose finality field is below a governance-controlled minimum (e.g., reject `EvmFinality::Latest` and `EvmFinality::Safe`; require `EvmFinality::Finalized`; require `BlockConfirmations >= 6` for Bitcoin).
2. **Encode the minimum in the on-chain `ChainEntry` / `ForeignChainRpcWhitelist`**: Each whitelisted chain should carry a `min_finality` field voted in by participants, so the contract can enforce it without a code upgrade when chain-specific thresholds change.
3. **Node-side enforcement as defense-in-depth**: Even if the contract enforces minimums, nodes should independently refuse to sign for finality levels below their configured threshold.
4. **Bridge contract guidance**: Document that downstream contracts MUST validate the finality field embedded in the signed payload and MUST NOT accept signatures produced at `Latest` or `Safe` finality for value-bearing operations.

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }

    async fn verify_finality_level(
        &self,
        receipt_block_number: U64,
        finality: EthereumFinality,
    ) -> Result<(), ForeignChainInspectionError> {
        let finality_tag = match finality {
            EthereumFinality::Finalized => FinalityTag::Finalized,
            EthereumFinality::Safe => FinalityTag::Safe,
            EthereumFinality::Latest => FinalityTag::Latest,
        };
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Tag(finality_tag),
            ReturnFullTransactionObjects::from(false),
        );
        let head: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        if head.number < receipt_block_number {
            return Err(ForeignChainInspectionError::NotFinalized);
        }
        Ok(())
    }

        let request_parameters = GetRawTransactionArgs {
            transaction_hash: TransportBitcoinTransactionHash::from(*transaction),
            verbose: VERBOSE_RESPONSE,
        };

        // TODO(#1978): add retry mechanism if the error from the request is transient
        let rpc_response: GetRawTransactionVerboseResponse = self
            .client
            .request(GET_RAW_TRANSACTION_METHOD, &request_parameters)
            .await?;

        let transaction_block_confirmation = rpc_response.confirmations.into();
        let enough_block_confirmations =
            block_confirmations_threshold <= transaction_block_confirmation;

        if !enough_block_confirmations {
            return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations {
                expected: block_confirmations_threshold,
                got: transaction_block_confirmation,
            });
        }

        self.verify_block_is_canonical(rpc_response.blockhash)
            .await?;

        let extracted_values = extractors
            .iter()
            .map(|extractor| extractor.extract_value(&rpc_response))
            .collect();

        Ok(extracted_values)
    }

* **RPC trust and correctness**: Verification relies on centralized RPC providers. A malicious
  or faulty provider could return incorrect data for a subset of nodes.
* **No additional consensus**: Nodes independently query providers and do not participate on failures.
  If a threshold of nodes are misled by providers, the network could sign invalid observations.
* **Provider availability**: Outages or rate limits can cause verification failures and reduced
  signing availability.
* **Finality semantics**: Finality definitions differ across chains; mapping them correctly is critical.
* **Incomplete chain coverage**: A chain is whitelisted as soon as the network votes in its whitelist entry, independent of any single operator. A node that hasn't configured a whitelisted chain is treated like a node that is down *for that chain* — it does not participate in that chain's verification requests, so the foreign-tx **presignatures** it co-owns (the dedicated `ForeignTx` domain) can sit idle and, if offline long enough, be discarded. Its **triples are not stranded** — the node is still up and triples are shared across all CaitSith domains, so it keeps using them; triples only become offline assets if the node is genuinely down. Mitigated operationally by alerting on any active node that does not cover a whitelisted chain, rather than by the protocol.
