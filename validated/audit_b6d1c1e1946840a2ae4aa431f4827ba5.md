### Title
Single Attested Participant Can Forge Foreign-Chain Verification Results via Cross-Request Signature Replay in `respond_verify_foreign_tx` — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is cryptographically valid over the caller-supplied `response.payload_hash`, but never checks that `response.payload_hash` is the correct hash for the given `request`. A single malicious attested participant (strictly below the signing threshold) can reuse a valid MPC signature from any prior foreign-chain verification and attach it to a completely different pending request, causing the contract to resolve that request with a fabricated verification result.

---

### Finding Description

`respond_verify_foreign_tx` performs three checks before resolving a pending yield:

1. Caller is an attested participant (`assert_caller_is_attested_participant_and_protocol_active`).
2. Protocol is running and TEE is accepted.
3. The ECDSA signature in `response.signature` is valid over `response.payload_hash` against the **root** (un-tweaked) public key. [1](#0-0) 

What is **never** checked: that `response.payload_hash` is the SHA-256 of `borsh(ForeignTxSignPayload { request: <the pending request>, values: <any valid extracted values> })`. The contract accepts any `payload_hash` for which a valid root-key signature exists, regardless of whether that hash has any relationship to the pending `request`.

The SDK-side verifier (`ForeignChainSignatureVerifier::verify_signature`) does enforce this binding — it recomputes the expected hash from `(request, expected_extracted_values)` and compares it to `response.payload_hash` — but this check lives entirely off-chain in the caller's contract, not in the MPC contract itself. [3](#0-2) 

**Cross-request replay attack path:**

1. Attacker (a single attested participant) observes any past successful `respond_verify_foreign_tx` call on-chain. The call is public; the response `{payload_hash: H, signature: S}` is visible to all.
2. A victim bridge contract submits a new `verify_foreign_transaction` request `R1` (e.g., to verify a Bitcoin deposit tx).
3. Before the legitimate MPC network responds, the attacker immediately calls `respond_verify_foreign_tx(R1, {payload_hash: H, signature: S})` — reusing the old signature from a completely different request `R2`.
4. The contract checks: is `S` a valid ECDSA signature over `H` under the root key? Yes — it was produced by the MPC network for `R2`. The check passes.
5. The yield for `R1` is resolved with `{payload_hash: H, signature: S}` and returned to the bridge contract. [4](#0-3) 

The bridge contract receives a response that is cryptographically self-consistent (signature is valid over `payload_hash`) but whose `payload_hash` encodes the extracted values of a **different** foreign-chain transaction (`R2`), not the one the bridge actually submitted (`R1`).

---

### Impact Explanation

**Impact: High** — forged foreign-chain verification / invalid bridge execution.

The `verify_foreign_transaction` flow is the on-chain primitive that lets bridge contracts react to foreign-chain events without a trusted relayer. The response's `payload_hash` is the only on-chain commitment to what the MPC network actually observed on the foreign chain (block hash, confirmation count, extracted log values, etc.).

If a bridge contract does not independently recompute and compare the expected `payload_hash` (i.e., does not call `ForeignChainSignatureVerifier::verify_signature` or equivalent), it will accept the fabricated response as proof that `R1`'s foreign-chain transaction was verified, when in fact the MPC network verified `R2`. Concretely:

- A bridge could credit an inbound transfer that never occurred on the foreign chain.
- A bridge could release funds against a transaction that was already used to satisfy a different request (`R2`), enabling a double-spend.
- Any bridge logic that branches on the extracted values (e.g., "block hash matches expected finality checkpoint") can be fed arbitrary values from a prior signing.

This matches the allowed impact: *"Cross-chain replay, forged foreign-chain verification, light-client-style verification bypass, or participant/attestation authorization bypass that causes invalid bridge execution or double-spend conditions."*

---

### Likelihood Explanation

**Likelihood: Medium.**

- The attacker is a single attested MPC participant — a Byzantine node strictly below the signing threshold. No key compromise, no threshold collusion, and no privileged operator access is required.
- The raw material for the attack (a prior `{payload_hash, signature}` pair) is permanently visible on-chain after any successful `respond_verify_foreign_tx` call.
- The race window is the time between a new `verify_foreign_transaction` submission and the legitimate MPC network's response (typically seconds). An attacker who monitors the contract can submit the fabricated response in the same block.
- The attack succeeds only if the receiving bridge contract omits the `payload_hash` binding check. The SDK provides this check, but the contract does not enforce it, so bridge developers who do not read the SDK documentation or who implement their own verification logic are exposed.

---

### Recommendation

The MPC contract should enforce on-chain that `response.payload_hash` is consistent with the pending `request`. Because the `values` (extracted data) are determined off-chain, the contract cannot recompute the full hash. However, it can enforce a weaker but meaningful binding: require the responder to also supply the `values` in the response, recompute `SHA-256(borsh(ForeignTxSignPayload { request, values }))` on-chain, and assert it equals `response.payload_hash`. This eliminates the cross-request replay entirely.

Alternatively, the contract can enforce that `response.payload_hash` has never been used to resolve a different request (a per-hash nonce map), preventing reuse of any prior signature.

At minimum, the SDK's `ForeignChainSignatureVerifier::verify_signature` binding check should be documented as a **mandatory** security invariant for all bridge integrators, not an optional helper.

### Proof of Concept

```
// Setup: prior successful verification of request R2 produced (H, S) on-chain.
// H = SHA-256(borsh(ForeignTxSignPayload { request: R2, values: [BlockHash(0x42...)] }))
// S = valid ECDSA signature over H under the MPC root key

// Step 1: victim bridge submits a new request R1 (different tx_id)
user.call(contract, "verify_foreign_transaction")
    .args_json({ "request": R1_args })
    .deposit(1)
    .transact_async();

// Step 2: attacker (single attested participant) immediately replays old (H, S)
attacker.call(contract, "respond_verify_foreign_tx")
    .args_json({
        "request": R1,          // the NEW pending request
        "response": {
            "payload_hash": H,  // hash from OLD request R2
            "signature": S      // valid MPC signature over H (from R2)
        }
    })
    .transact();

// Result: contract resolves R1's yield with (H, S).
// Bridge receives VerifyForeignTransactionResponse { payload_hash: H, signature: S }.
// Signature verifies. payload_hash does NOT correspond to R1's foreign-chain state.
// If bridge omits the payload_hash binding check, it accepts a false verification.
```
