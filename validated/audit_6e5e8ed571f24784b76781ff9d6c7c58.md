Audit Report

## Title
Permissionless `sendFunds()` With No Caller Incentive Enables Permanent Yield Dilution of Existing rsETH Holders — (File: contracts/FeeReceiver.sol)

## Summary
`FeeReceiver.sendFunds()` has no access control and no caller reward, so MEV/execution-layer ETH accumulates in `FeeReceiver` indefinitely without being reflected in the protocol's TVL. Because `LRTOracle.rsETHPrice` is a stored state variable updated only when `updateRSETHPrice()` is called — and that computation explicitly excludes `FeeReceiver.balance` — any deposit made while rewards sit uncollected mints excess rsETH at an understated price, permanently diluting the yield share of existing holders.

## Finding Description
`FeeReceiver.sendFunds()` is fully permissionless with no incentive:

```solidity
// contracts/FeeReceiver.sol L53-58
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

`getETHDistributionData()` explicitly excludes `FeeReceiver.balance` from TVL accounting — it only sums ETH in the deposit pool itself, NDCs, EigenLayer, the converter, and the unstaking vault:

```solidity
// contracts/LRTDepositPool.sol L464-466
/// @dev rewards are not accounted here
/// it will automatically be accounted once it is moved from feeReceiver/rewardReceiver to depositPool
```

`LRTOracle._getTotalEthInProtocol()` calls `getTotalAssetDeposits()` for each supported asset, which for ETH routes through `getETHDistributionData()`, so `FeeReceiver.balance` is never included:

```solidity
// contracts/LRTOracle.sol L341-343
uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
```

`rsETHPrice` is a stored state variable (`uint256 public override rsETHPrice`) updated only when `updateRSETHPrice()` is called. `updateRSETHPrice()` is itself public and permissionless (`function updateRSETHPrice() public whenNotPaused`). `getRsETHAmountToMint()` reads this stored value:

```solidity
// contracts/LRTDepositPool.sol L519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

**Exploit path:**
1. MEV rewards accumulate passively in `FeeReceiver` (e.g., 10 ETH on a 1 000 ETH TVL base).
2. Attacker calls `LRTOracle.updateRSETHPrice()` (public) — price is computed without the 10 ETH and stored as understated (e.g., 1.000 instead of 1.010 ETH/rsETH).
3. Attacker calls `LRTDepositPool.depositETH{value: 100 ether}(0, "")` — receives `100 / 1.000 = 100 rsETH` instead of the fair `100 / 1.010 ≈ 99.01 rsETH`.
4. Anyone calls `FeeReceiver.sendFunds()` — 10 ETH enters TVL.
5. `updateRSETHPrice()` is called — price rises to reflect the 10 ETH, but the 0.99 excess rsETH minted in step 3 permanently dilutes existing holders' share of those rewards.

No existing guard prevents this: `_beforeDeposit()` only checks deposit limits and a caller-supplied `minRSETHAmountExpected` slippage parameter (which the attacker sets to 0), and there is no freshness check on `rsETHPrice` nor any automatic flush of `FeeReceiver` before minting.

## Impact Explanation
**High — Theft of unclaimed yield.** Existing rsETH holders permanently lose a portion of their accrued MEV/execution-layer yield to depositors who enter during the window when rewards sit in `FeeReceiver`. The excess rsETH minted cannot be burned; the dilution is irreversible. The magnitude scales with `FeeReceiver.balance / TVL` and the volume of deposits during the delay window.

## Likelihood Explanation
**Low.** The protocol team is expected to call `sendFunds()` periodically off-chain, limiting the typical accumulation window. However, there is no on-chain enforcement, no keeper incentive, and no minimum call frequency. During high-MEV periods (e.g., around major on-chain events), rewards can accumulate rapidly. A sophisticated depositor who monitors `FeeReceiver.balance` can time a large deposit to exploit the window before `sendFunds()` is called, and can themselves call `updateRSETHPrice()` immediately before depositing to lock in the understated price. The attack is repeatable and requires no special privileges.

## Recommendation
1. **Auto-flush on deposit:** At the start of `depositETH()` / `depositAsset()`, call `FeeReceiver.sendFunds()` (or an equivalent internal flush) so rewards are always counted in TVL before new rsETH is minted.
2. **Compute price fresh on deposit:** Replace the stored `rsETHPrice` read in `getRsETHAmountToMint()` with a live computation of `_getTotalEthInProtocol() / rsethSupply` that includes any pending `FeeReceiver` balance, or at minimum require that `rsETHPrice` was updated in the same block as the deposit.
3. **Caller incentive for `sendFunds()`:** Reward the caller with a small fixed percentage (e.g., 0.1–0.5 %) of the forwarded balance to create on-chain incentive for timely flushing.

## Proof of Concept
Foundry fork test outline:

```solidity
// 1. Fork mainnet; confirm FeeReceiver has accumulated ETH (or send ETH to it directly).
// 2. Record existing holder's rsETH balance and current rsETHPrice.
// 3. Call lrtOracle.updateRSETHPrice() as attacker — price stored without FeeReceiver balance.
// 4. Call lrtDepositPool.depositETH{value: 100 ether}(0, "") as attacker.
// 5. Assert attacker received more rsETH than 100 ether / fairPrice.
// 6. Call feeReceiver.sendFunds() as attacker.
// 7. Call lrtOracle.updateRSETHPrice() as manager (to bypass pricePercentageLimit if set).
// 8. Assert existing holder's rsETH now represents a smaller share of total ETH than before step 3.
// 9. Compute yield loss per rsETH = (fairPrice - actualPrice) and assert it is > 0.
```

Key invariant violated: `(totalETHInProtocol_after / rsethSupply_after) * existingHolderBalance` must equal or exceed `(totalETHInProtocol_before_deposit + rewardAmount) / (rsethSupply_before_deposit + fairMintAmount) * existingHolderBalance`. The excess rsETH minted in step 4 breaks this invariant permanently.
