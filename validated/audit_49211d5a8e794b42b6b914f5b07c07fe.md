### Title
Missing CKD Output Verification for `AppPublicKey` Variant Allows Single Malicious Node to Forge Derived Key - (File: crates/contract/src/lib.rs)

### Summary

`respond_ckd` in the MPC contract applies a cryptographic pairing-based output check (`ckd_output_check`) only for the `AppPublicKeyPV` variant of `CKDAppPublicKey`. For the `AppPublicKey` variant, the match arm is a no-op — any single attested participant can submit an arbitrary `CKDResponse` and the contract will accept it unconditionally. This is structurally identical to the reported `approve` vs `safeApprove` pattern: one code path applies the safe/verified interface, the other silently skips it.

### Finding Description

In `respond_ckd`, after confirming the caller is an attested participant, the contract branches on the request's `app_public_key` field:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically binds the response to the MPC master key and the app identity. [2](#0-1) 

For `AppPublicKey`, no equivalent check exists. The contract immediately calls `resolve_yields_for`, serializing and returning whatever `CKDResponse` the caller supplied. [3](#0-2) 

The `AppPublicKey` variant is a production feature exercised in e2e tests: [4](#0-3) 

### Impact Explanation

**Critical — confidential key derivation output without required participant authorization.**

The CKD protocol computes `big_y = g1·r`, `big_c = H(pk, app_id)·msk + pk1·r`. A user holding private scalar `s` (with `pk1 = g1·s`) decrypts: `big_c − big_y·s = H(pk, app_id)·msk`.

A malicious attested participant can instead submit a crafted response:
- `big_y = g1·r′` (attacker-chosen `r′`)
- `big_c = g1·k + pk1·r′` (attacker-chosen scalar `k`)

The user decrypts: `g1·k + g1·s·r′ − g1·r′·s = g1·k`. The derived key is `g1·k`, whose discrete log `k` the attacker knows. The attacker can then sign arbitrary transactions on any chain using the address derived from `g1·k`, draining any funds the user sends there.

This bypasses the threshold requirement: the threshold protocol is enforced only off-chain; the contract enforces no equivalent on-chain for `AppPublicKey`.

### Likelihood Explanation

The attacker must be a single TEE-attested participant — below the signing threshold. The pending `CKDRequest` (including `app_id` and `domain_id`) is observable on-chain, so the attacker can race the honest leader. The `AppPublicKey` variant is the default/legacy path used by existing SDK integrations. [5](#0-4) 

### Recommendation

For `AppPublicKey` requests, add an on-chain verifiable binding between the response and the MPC public key. The simplest approach mirrors `respond` and `respond_verify_foreign_tx`: require the leader to include a threshold ECDSA (or BLS) signature over a canonical encoding of `(app_id, big_y, big_c)` under the domain's public key, and verify it in `respond_ckd` before resolving the yield. Alternatively, extend the CKD protocol to always use the `AppPublicKeyPV` shape (which already has a complete on-chain check) and deprecate the unverifiable `AppPublicKey` variant.

### Proof of Concept

1. User calls `request_app_private_key` with `AppPublicKey(pk1)` where `pk1 = g1·s`.
2. Attacker (single attested participant) observes the pending `CKDRequest` on-chain.
3. Attacker picks scalar `k`, computes `r′ = random`, `big_y = g1·r′`, `big_c = g1·k + pk1·r′`.
4. Attacker calls `respond_ckd(request, CKDResponse { big_y, big_c })`.
5. Contract passes the `AppPublicKey(_) => {}` arm with no check and calls `resolve_yields_for`. [6](#0-5) 

6. User receives `(big_y, big_c)` and decrypts: `big_c − big_y·s = g1·k`.
7. Attacker knows `k` and controls the derived key, enabling theft of any funds sent to the derived address.

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
