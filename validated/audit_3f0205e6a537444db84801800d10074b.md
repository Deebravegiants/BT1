Audit Report

## Title
Missing Staleness Checks in `ChainlinkPriceOracle.getAssetPrice()` Enables Over-Minting of rsETH - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards `updatedAt` and `answeredInRound`, performing no heartbeat deadline or round-completeness validation. When a Chainlink LST/ETH feed goes stale at an inflated price, any unprivileged depositor can call `depositAsset()` and receive rsETH minted against the stale price, extracting value from all existing rsETH holders whose proportional claim on protocol TVL is diluted.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` at `contracts/oracles/ChainlinkPriceOracle.sol` L49–55:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All staleness-related return values (`updatedAt`, `answeredInRound`) are discarded. No check of the form `block.timestamp - updatedAt > MAX_DELAY` and no `answeredInRound >= roundId` check are present.

This oracle is consumed by `LRTOracle.getAssetPrice()` (`contracts/LRTOracle.sol` L156–158), which delegates directly to the registered `IPriceFetcher`. That price is then used in `LRTDepositPool.getRsETHAmountToMint()` (`contracts/LRTDepositPool.sol` L519–520):

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`rsETHPrice` is a **stored** value updated only when `updateRSETHPrice()` is explicitly called; it is not recomputed on every deposit. Therefore, if the Chainlink feed for a supported LST goes stale at an inflated price while `rsETHPrice` still reflects the last accurate update, the numerator (`amount * stalePrice`) is inflated but the denominator (`rsETHPrice`) is not, causing the attacker to receive more rsETH than their deposited assets are worth.

The inconsistency is confirmed by `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` L26–37, which **does** implement the missing checks:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The deposit-path oracle lacks all three of these guards.

The `minRSETHAmountExpected` slippage parameter in `depositAsset()` (`contracts/LRTDepositPool.sol` L99–118) protects only the depositor from receiving too little; it does not prevent the protocol from over-minting. The `pricePercentageLimit` circuit-breaker in `_updateRsETHPrice()` (`contracts/LRTOracle.sol` L252–281) only fires when `updateRSETHPrice()` is called and does not gate individual deposits.

## Impact Explanation
When a supported LST depegs (e.g., stETH drops from 1.00 ETH to 0.90 ETH) while the Chainlink feed remains stale at the pre-depeg price, a depositor mints rsETH computed at 1.00 ETH/stETH but contributes assets worth only 0.90 ETH/stETH. The excess rsETH represents a proportional dilution of every existing rsETH holder's claim on the protocol's ETH TVL — a direct transfer of value from at-rest holders to the depositor.

**Impact: High — Theft of unclaimed yield / value from existing rsETH holders.** In a severe depeg (10–20% price drop while feed is stale), the magnitude approaches Critical (direct theft of user funds at-rest).

## Likelihood Explanation
Chainlink feeds operate on heartbeat intervals (e.g., 1 hour for stETH/ETH on mainnet) and deviation thresholds. During low-volatility periods, feeds may not update for the full heartbeat window. During network congestion or oracle node failures, feeds can go stale for extended periods. No privileged access is required — any external caller can invoke `depositAsset()`. The attack is repeatable for the entire duration the feed remains stale. The protocol's deployment on L2 chains (evidenced by `RSETHPool`, `TACWETHBridge`, `RSETHPoolV3ExternalBridge`) introduces additional sequencer-downtime risk that can freeze feed updates.

**Likelihood: Medium** — requires a stale feed window coinciding with an opportunistic depositor, but no special role or victim mistake is needed.

## Recommendation
Add staleness and validity checks to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (block.timestamp - updatedAt > MAX_STALENESS_DELAY) revert StalePrice();
    if (price <= 0) revert InvalidPrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`MAX_STALENESS_DELAY` should be configured per-asset based on the feed's documented heartbeat interval. For L2 deployments, additionally integrate a Chainlink sequencer uptime feed check before consuming any price data.

## Proof of Concept

**Setup (fork test against mainnet or a local fork):**

1. Deploy or fork with stETH/ETH Chainlink feed. Record current `rsETHPrice` (stored, e.g., 1.05e18).

2. Mock the Chainlink feed to return a stale answer: `price = 1.00e18`, `updatedAt = block.timestamp - 2 hours` (past the 1-hour heartbeat), `answeredInRound < roundId`.

3. Attacker calls:
   ```solidity
   LRTDepositPool.depositAsset(stETH, 1000e18, 0, "");
   ```

4. `getRsETHAmountToMint(stETH, 1000e18)` calls `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns `1.00e18` (stale, no revert).

5. `rsethAmountToMint = (1000e18 * 1.00e18) / 1.05e18 ≈ 952.38e18` rsETH minted.

6. At the true market price of 0.90 ETH/stETH, the attacker deposited assets worth 900 ETH but received rsETH redeemable for ~1000 ETH of protocol TVL claim.

7. **Assert**: attacker's rsETH balance × rsETHPrice > market value of deposited stETH. The ~100 ETH difference is extracted from existing rsETH holders' proportional share.

**Foundry invariant test**: Fuzz `depositAsset` calls with a mock feed returning prices in `[0.80e18, 1.00e18]` while `updatedAt` is set to `block.timestamp - 2 hours`. Assert that `attackerRsETH * rsETHPrice <= depositedAmount * trueMarketPrice + epsilon` for all runs. The invariant will break without the staleness fix.
