Audit Report

## Title
Missing `msg.sender` Validation in `MerkleDistributor.claim()` Enables Forced Fee-Bearing Claims on Behalf of Any Account - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary
`MerkleDistributor.claim()` accepts an arbitrary `account` parameter without verifying that `msg.sender == account`. Any caller with a valid merkle proof for a victim can force a claim on the victim's behalf. When `feeInBPS > 0`, this permanently deducts up to 10% of the victim's claimable allocation as a fee, which the victim would have retained had they waited for the fee to be reduced or zeroed.

## Finding Description
`MerkleDistributor.claim()` performs no caller restriction — it only verifies that a valid merkle proof exists for `(index, account, cumulativeAmount)`: [1](#0-0) 

After proof verification, the fee is unconditionally deducted and tokens are sent to `account`: [2](#0-1) 

The user's claim state is permanently updated to `cumulativeAmount`, so the fee-bearing portion is irrecoverable: [3](#0-2) 

By contrast, `KernelMerkleDistributor._processClaim()` explicitly enforces the caller restriction: [4](#0-3) 

This confirms the omission in `MerkleDistributor` is a defect. The fee ceiling is `MAX_FEE_IN_BPS = 1000` (10%): [5](#0-4) 

**Exploit path:**
1. Owner sets `feeInBPS` to any value above 0 (e.g., 1000 = 10%).
2. Victim holds unclaimed tokens and defers claiming, waiting for the fee to be lowered.
3. Attacker obtains the victim's merkle proof from the public proof file.
4. Attacker calls `claim(index, victim, cumulativeAmount, victimProof)`.
5. Contract verifies the proof, deducts the fee, sends `amountToSend` to victim, sends `fee` to `protocolTreasury`.
6. Victim's claim slot is permanently consumed at the current fee rate; the fee amount is irreversibly lost.

## Impact Explanation
**High — Theft of unclaimed yield.** The fee deducted from `claimableAmount` is yield the victim was entitled to but had not yet claimed. By forcing the claim at a moment when `feeInBPS > 0`, the attacker causes the victim to permanently forfeit up to 10% of their claimable allocation. Because `userClaims[account].cumulativeAmount` is updated to `cumulativeAmount`, the victim cannot reclaim the fee-bearing portion. Had the victim claimed when `feeInBPS = 0`, they would have received the full amount. The loss is concrete, measurable, and irreversible.

## Likelihood Explanation
**Medium.** `feeInBPS > 0` is a normal operational state supported by the contract. Merkle proofs for airdrop/distribution contracts are routinely published off-chain (JSON/IPFS) so users can self-claim; an attacker can read them without any privileged access. The attacker needs only to call a public function with no special role. The attack is repeatable for any victim who defers their claim while a fee is active.

## Recommendation
Add a caller restriction identical to the one in `KernelMerkleDistributor._processClaim()`:

```solidity
if (account != msg.sender) {
    revert Unauthorized();
}
```

Insert this check in `MerkleDistributor.claim()` immediately after the `isClaimed` guard (after line 117), before the merkle proof verification. Also add `nonReentrant` to `claim()` as a defence-in-depth measure, consistent with `KernelMerkleDistributor`.

## Proof of Concept
```solidity
// Precondition: feeInBPS = 1000 (10%), victim has 1000e18 tokens claimable.
// Attacker EOA has no privileged role; victim proof is obtained from the public proof file.

merkleDistributor.claim(
    currentIndex,        // valid index
    victim,              // victim's address — NOT msg.sender
    1000e18,             // victim's cumulative amount from the published proof
    victimMerkleProof    // obtained from the public proof file
);

// Result:
//   victim receives 900e18 tokens (10% fee deducted)
//   protocolTreasury receives 100e18 tokens
//   victim loses 100e18 tokens they would have kept if fee had been 0
//   victim's claim slot is permanently consumed at the 10% fee rate
```

**Foundry test plan:** Deploy `MerkleDistributor`, set `feeInBPS = 1000`, construct a merkle tree with a leaf for `victim`, publish the proof, call `claim` from `attacker` address passing `victim` as `account`, assert `victim` received only 90% of allocation and `userClaims[victim].cumulativeAmount` equals the full `cumulativeAmount`.

```text
    function claim(
        uint256 index,
        address account,
        uint256 cumulativeAmount,
        bytes32[] calldata merkleProof
    )
        external
        override
        whenNotPaused
    {
        if (currentMerkleRoot == bytes32(0)) {
            revert ZeroValueProvided();
        }

        if (index == 0 || index > currentIndex) {
            revert InvalidIndex();
        }

        if (isClaimed(index, account)) {
            revert AlreadyClaimed();
        }

        // Verify the merkle proof.
        bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));
        if (!MerkleProofUpgradeable.verify(merkleProof, currentMerkleRoot, node)) {
            revert InvalidMerkleProof();
        }
```
