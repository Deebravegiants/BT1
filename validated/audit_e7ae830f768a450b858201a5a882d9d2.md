### Title
Mempool Eviction Not Rolled Back on Rejected Oversized Transaction Enables Costless Legitimate-Transaction Purge — (`crates/apollo_mempool/src/mempool.rs`)

---

### Summary

`try_make_space` permanently removes gap-account transactions from the mempool to make room for an incoming transaction. If the incoming transaction is still too large to fit after eviction, `handle_capacity_overflow` returns `MempoolError::MempoolFull` and the incoming transaction is rejected — but the already-evicted legitimate transactions are **never restored**. An attacker can exploit this to repeatedly purge legitimate pending transactions from the mempool at zero fee cost, preventing them from ever being sequenced.

---

### Finding Description

In `crates/apollo_mempool/src/mempool.rs`, `handle_capacity_overflow` calls `try_make_space` to evict transactions from accounts with nonce gaps:

```rust
// mempool.rs lines 1047-1051
if !creating_gap && self.try_make_space(required_space) {
    return Ok(());
}
Err(MempoolError::MempoolFull)
``` [1](#0-0) 

Inside `try_make_space`, transactions are removed from the pool unconditionally as the loop runs:

```rust
// mempool.rs lines 1003-1007
let tx = self
    .tx_pool
    .remove(tx_ref.tx_hash)
    .expect("Transaction must exist in the pool.");
total_space_freed += tx.total_bytes();
``` [2](#0-1) 

The function returns `false` when it exhausts all evictable accounts without freeing enough space. At that point, `handle_capacity_overflow` returns `Err(MempoolError::MempoolFull)`, the incoming transaction is rejected — but **all evictions that occurred inside `try_make_space` are permanent**. There is no rollback.

This is explicitly confirmed by the existing test:

```rust
// fee_mempool_test.rs lines 1936-1938
// Transactions tx1 and tx2 are evicted regardless of whether large_tx is accepted or rejected.
// We do not revert the eviction attempt even if adding large_tx ultimately fails.
assert!(!mempool.tx_pool.contains_account(contract_address!("0x1")));
``` [3](#0-2) 

The gateway's stateless validator enforces size limits — `max_signature_length = 4000` felts and `max_calldata_length = 5000` felts — but these allow a single transaction to reach approximately **288 KB** in serialized size (4000 × 32 + 5000 × 32 bytes): [4](#0-3) [5](#0-4) 

An attacker crafts a transaction at or near the maximum allowed size. When the mempool is full with small gap-account transactions whose combined size is less than the attacker's transaction size, `try_make_space` evicts all of them but still cannot satisfy `required_space`, returns `false`, and the attacker's transaction is rejected. The attacker pays **no fee** (the transaction never executes) but has permanently removed legitimate transactions.

---

### Impact Explanation

Valid, pending transactions from legitimate users are evicted from the mempool and never re-queued. They will not be sequenced into any block unless the users resubmit them. By repeating the attack continuously, an attacker can prevent targeted accounts' transactions from ever reaching the batcher, effectively censoring them at the admission layer. This maps directly to the allowed impact: **"Mempool/gateway/RPC admission … rejects valid transactions before sequencing."**

---

### Likelihood Explanation

- The attacker needs no privileged access — any user can submit a transaction to the public gateway.
- The attacker pays no fee: the oversized transaction is rejected before execution, so no STRK/ETH is charged.
- Gap-account transactions are a normal occurrence in any active mempool (accounts submitting future-nonce transactions).
- The attack is repeatable at the cost of only network round-trips.
- The only prerequisite is that the combined size of evictable transactions is smaller than the attacker's transaction, which is achievable when targeting a small set of gap-account transactions.

---

### Recommendation

Make eviction atomic with respect to the admission decision. Two concrete options:

1. **Pre-check before evicting**: Before calling `try_make_space`, verify that the total evictable bytes across all gap accounts is ≥ `required_space`. Only proceed with eviction if the check passes. This avoids partial eviction that cannot satisfy the requirement.

2. **Collect-then-commit**: In `try_make_space`, collect the transactions to evict into a staging list without removing them from the pool. Only commit the removals after confirming `total_space_freed >= required_space`. If the check fails, discard the staging list and return `false` without mutating the pool.

---

### Proof of Concept

The existing test `rejects_or_accepts_tx_based_on_freed_space` in `crates/apollo_mempool/src/fee_mempool_test.rs` (lines 1894–1938) already demonstrates the exact scenario:

1. Fill the mempool to capacity with `tx1` and `tx2` from address `"0x1"` (nonce gap: tx_nonce=1,2 but account_nonce=0 → evictable).
2. Submit `large_tx` from address `"0x0"` with `signature_size=20` felts — large enough to exceed capacity even after evicting `tx1`+`tx2`.
3. Result: `large_tx` is rejected with `MempoolError::MempoolFull`, but `tx1` and `tx2` are permanently gone from the pool.

In production, an attacker repeats step 2 with a near-maximum-size transaction (up to 4000-felt signature + 5000-felt calldata ≈ 288 KB) each time new gap-account transactions appear in the mempool, continuously purging them at zero cost. [3](#0-2) [1](#0-0)

```rust
    pub fn try_make_space(&mut self, required_space: u64) -> bool {
        let mut total_space_freed = 0;
        let mut evicted_txs = Vec::new();

        while total_space_freed < required_space {
            let Some(address) = self.get_evictable_account() else {
                break;
            };

            let txs: Vec<_> = self.tx_pool.account_txs_sorted_by_nonce(address).copied().collect();
            for tx_ref in txs.iter().rev() {
                let tx = self
                    .tx_pool
                    .remove(tx_ref.tx_hash)
                    .expect("Transaction must exist in the pool.");
                total_space_freed += tx.total_bytes();
                evicted_txs.push(*tx_ref);
                metric_count_evicted_txs(1);
                self.decrement_stuck_txs_if_gap_account(address, 1);
                if total_space_freed >= required_space {
                    break;
                }
            }

            // Clean up if account is now empty.
            if !self.tx_pool.contains_account(address) {
                self.accounts_with_gap.swap_remove(&address);
            }
        }

        // Keep the queue consistent with the pool: the evicted txs were removed from the pool, so
        // drop their (now-orphaned) queue references too. In fee mode gap accounts are never
        // queued, so this is a no-op there; in Echonet/FIFO mode they are, and skipping this leaves
        // a dangling reference that panics the next `get_txs`.
        self.tx_queue.remove_txs(&evicted_txs);

        total_space_freed >= required_space
    }

```rust
    fn handle_capacity_overflow(
        &mut self,
        tx: &InternalRpcTransaction,
        account_nonce: Nonce,
        freed_bytes: u64,
    ) -> Result<(), MempoolError> {
        let address = tx.contract_address();

        let account_has_gap = self.accounts_with_gap.contains(&address);
        let account_has_txs = self.tx_pool.contains_account(address);
        let closing_gap = tx.nonce() == account_nonce;
        let creating_gap = (account_has_gap || !account_has_txs) && !closing_gap;

        // Only the net growth must be evicted: an accompanying replacement removal frees
        // `freed_bytes` (0 when there is no replacement).
        let required_space = tx.total_bytes().saturating_sub(freed_bytes);
        if !creating_gap && self.try_make_space(required_space) {
            return Ok(());
        }

        Err(MempoolError::MempoolFull)
    }

                capacity_in_bytes: capacity,
                ..Default::default()

            ..Default::default()

    fn validate_tx_size(&self, tx: &RpcTransaction) -> StatelessTransactionValidatorResult<()> {
        self.validate_tx_extended_calldata_size(tx)?;

        if let RpcTransaction::Invoke(invoke_tx) = tx {
            self.validate_proof_size(invoke_tx)?;

    /// Validates the transaction calldata size.
    /// This includes client-side proof facts when present.
    fn validate_tx_extended_calldata_size(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        let total_length = match tx {
            RpcTransaction::Declare(_) => return Ok(()),

            RpcTransaction::DeployAccount(RpcDeployAccountTransaction::V3(tx)) => {
                tx.constructor_calldata.0.len()
            }

            RpcTransaction::Invoke(RpcInvokeTransaction::V3(tx)) => {
                tx.calldata.0.len() + tx.proof_facts.0.len()
            }
        };

        if total_length > self.config.max_calldata_length {
            return Err(StatelessTransactionValidatorError::CalldataTooLong {
                calldata_length: total_length,
                max_calldata_length: self.config.max_calldata_length,
            });
        }

        Ok(())
    }

    fn validate_tx_signature_size(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        let signature = tx.signature();

        let signature_length = signature.0.len();
        if signature_length > self.config.max_signature_length {
            return Err(StatelessTransactionValidatorError::SignatureTooLong {
                signature_length,
                max_signature_length: self.config.max_signature_length,
            });
        }

        Ok(())
    }
