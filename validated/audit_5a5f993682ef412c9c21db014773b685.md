### Title
Unbounded Gas Consumption in `updateRSETHPrice()` via Nested Loop Leaves `rsETHPrice` Stale, Enabling Incorrect rsETH Minting — (`contracts/LRTOracle.sol`)

---

### Summary

`updateRSETHPrice()` is publicly callable with no role restriction. Its internal call chain executes a nested loop of `O(supportedAssets × nodeDelegatorQueue)` external calls. If the transaction runs out of gas, it reverts without updating `rsETHPrice`. Because the deposit path reads `rsETHPrice` directly with no freshness guard, subsequent deposits mint rsETH at the stale rate, potentially over-minting relative to current TVL and diluting existing holders.

---

### Finding Description

**Entrypoint — no access control:**

`updateRSETHPrice()` carries only a `whenNotPaused` modifier; any EOA or contract can call it. [1](#0-0) 

**Nested external-call loop:**

`_updateRsETHPrice()` calls `_getTotalEthInProtocol()`, which iterates over every supported asset and, for each asset, calls `ILRTDepositPool.getTotalAssetDeposits(asset)`. [2](#0-1) 

`getTotalAssetDeposits` delegates to `getAssetDistributionData`, which itself loops over the entire `nodeDelegatorQueue` and fires three external calls per NDC per asset (`balanceOf`, `getAssetBalance`, `getAssetUnstaking`). [3](#0-2) 

For the ETH asset, `getETHDistributionData` runs a separate loop over `nodeDelegatorQueue` with two more external calls per NDC (`getEffectivePodShares`, `getAssetUnstaking`). [4](#0-3) 

Total external calls per `updateRSETHPrice()` invocation: `O(supportedAssets × nodeDelegatorQueue)`. `maxNodeDelegatorLimit` is admin-adjustable upward without a hard ceiling in the code. [5](#0-4) 

**State write is at the very end:**

`rsETHPrice` is only committed on the last line of `_updateRsETHPrice()`. An OOG revert at any earlier point leaves the stored value unchanged. [6](#0-5) 

**Deposit path reads the stored value with no freshness check:**

`getRsETHAmountToMint` divides by `lrtOracle.rsETHPrice()` — the raw storage slot — with no timestamp or staleness guard. [7](#0-6) 

If `rsETHPrice` is stale-low (TVL grew via rewards since the last successful update), the denominator is smaller than it should be, so `rsethAmountToMint` is larger than it should be. The depositor receives excess rsETH, diluting all existing holders.

---

### Impact Explanation

- **Primary:** Unbounded gas consumption — the loop grows linearly with `supportedAssets × nodeDelegatorQueue`, each iteration making multiple cold external calls. As the protocol scales, `updateRSETHPrice()` can exceed the block gas limit, making the price update permanently unexecutable without admin intervention to shrink the queue.
- **Secondary:** A stale `rsETHPrice` (lower than current TVL-implied price) causes `depositAsset` / `depositETH` to over-mint rsETH, transferring value from existing holders to the depositor. No freshness guard exists anywhere in the deposit path.

---

### Likelihood Explanation

- The protocol already supports multiple LST assets and multiple NDCs; the loop is real and grows with normal protocol operation.
- `updateRSETHPrice()` is called by off-chain keepers; if it starts reverting due to OOG, the price silently goes stale with no on-chain alarm.
- A sophisticated depositor monitoring mempool failures of `updateRSETHPrice()` can front-run a deposit against the stale price.
- No admin action is required by the attacker; the attacker only needs to observe the stale state and deposit.

---

### Recommendation

1. **Cap the loop at the contract level.** Enforce a hard maximum on `nodeDelegatorQueue.length` and `supportedAssetList.length` that is provably safe within the block gas limit, and add a gas-remaining check inside the loop.
2. **Add a price-freshness timestamp.** Record `rsETHPriceLastUpdated = block.timestamp` alongside every successful `rsETHPrice` write. In `getRsETHAmountToMint`, revert (or use a fallback) if `block.timestamp - rsETHPriceLastUpdated > MAX_STALENESS`.
3. **Consider a pull-based architecture.** Compute TVL off-chain and submit a signed result on-chain, eliminating the on-chain loop entirely.

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Fork test — run with: forge test --fork-url $RPC --match-test testStalePrice -vvv

import "forge-std/Test.sol";
import "../contracts/LRTOracle.sol";
import "../contracts/LRTDepositPool.sol";

contract StalePricePoC is Test {
    LRTOracle  oracle;
    LRTDepositPool pool;
    address    victim = makeAddr("victim");
    address    attacker = makeAddr("attacker");

    function setUp() public {
        // ... deploy / fork setup omitted for brevity
    }

    function testStalePrice() public {
        // 1. Record current rsETHPrice (set by last successful update)
        uint256 stalePriceBefore = oracle.rsETHPrice();

        // 2. Simulate OOG by calling updateRSETHPrice with very low gas
        //    (or by vm.roll to a state where the loop is too large)
        (bool ok,) = address(oracle).call{gas: 50_000}(
            abi.encodeWithSelector(oracle.updateRSETHPrice.selector)
        );
        assertFalse(ok, "expected OOG revert");

        // 3. rsETHPrice must be unchanged (stale)
        assertEq(oracle.rsETHPrice(), stalePriceBefore, "price should be stale");

        // 4. Attacker deposits — minted amount uses stale (lower) price
        uint256 mintedWithStale = pool.getRsETHAmountToMint(asset, depositAmt);

        // 5. Force a successful price update (e.g., manager role)
        vm.prank(manager);
        oracle.updateRSETHPriceAsManager();
        uint256 freshPrice = oracle.rsETHPrice();

        // 6. Compute what should have been minted at fresh price
        uint256 mintedWithFresh = (depositAmt * assetPrice) / freshPrice;

        // 7. Attacker received more rsETH than fair share
        assertGt(mintedWithStale, mintedWithFresh, "attacker over-minted");
    }
}
```

        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)

            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
