### Title
Missing Access Control on `sendETHToL1ViaBridge` Enables Linea Bridge Period-Limit Saturation, Permanently Freezing Pool Yield — (`contracts/bridges/LineaMessenger.sol`)

---

### Summary

`LineaMessenger.sendETHToL1ViaBridge` is a permissionless `external payable` function with no role check. Any caller can invoke it to bridge arbitrary ETH through the Linea native bridge. Because the Linea bridge enforces a per-period ETH cap (`limitInWei()`), an attacker can saturate that cap using their own ETH (which they recover on L1 minus only the `minimumFeeInWei`), causing every subsequent call to `bridgeAssetsViaNativeBridge` from the pool to revert for the entire period. Repeating this each period permanently freezes pool yield.

---

### Finding Description

`LineaMessenger` inherits `Recoverable` (which extends `AccessControl`) but `sendETHToL1ViaBridge` carries no `onlyRole` modifier:

```solidity
// contracts/bridges/LineaMessenger.sol:31
function sendETHToL1ViaBridge(address l2bridge, address target, uint256 value)
    external payable nonReentrant {
    ...
    ILineaMessageService(l2bridge).sendMessage{ value: value }(target, minimumFee, bytes(""));
}
```

The `ILineaMessageService` interface explicitly exposes the rate-limiting state:

```solidity
// contracts/interfaces/L2/ILineaMessageService.sol:29,37
function limitInWei() external view returns (uint256);
function currentPeriodAmountInWei() external view returns (uint256);
```

The pool's authorized bridge path calls through this same function:

```solidity
// contracts/pools/RSETHPool.sol:489-491
IL2Messenger(messenger).sendETHToL1ViaBridge{ value: ethBalanceMinusFees }(
    l2Bridge, l1VaultETHForL2Chain, ethBalanceMinusFees
);
```

**Attack steps:**

1. Attacker reads `limitInWei()` and `currentPeriodAmountInWei()` to compute remaining capacity `R`.
2. Attacker calls `LineaMessenger.sendETHToL1ViaBridge(lineaBridge, attacker_l1, R)` with `msg.value == R`. The call succeeds; attacker's ETH is bridged to their own L1 address. Net cost: only `minimumFeeInWei`.
3. Period limit is now saturated.
4. `BRIDGER_ROLE` calls `bridgeAssetsViaNativeBridge()` → `sendETHToL1ViaBridge` → `sendMessage` → **reverts** (period limit exceeded).
5. Pool ETH is stuck until the period resets (typically 24 h on Linea).
6. Attacker repeats step 1–5 at the start of each new period. Cost per period ≈ `minimumFeeInWei` (a few wei/gwei). Pool yield is permanently frozen.

The attacker does **not** lose their principal — it is bridged to themselves — making the attack economically trivial to sustain indefinitely.

---

### Impact Explanation

Pool ETH (accumulated yield from user swaps) cannot be bridged to L1 for as long as the attacker repeats the saturation. The `BRIDGER_ROLE` has no fallback path once the period limit is consumed. Yield accumulates in the L2 pool contract but never reaches the L1 vault, matching **Medium — Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

- Attacker needs only a wallet with enough ETH to equal the remaining period capacity (a one-time float; recovered on L1 each period).
- No privileged access, no front-running dependency, no oracle manipulation.
- The attack is repeatable every period at near-zero net cost.
- Likelihood: **High** — trivially executable by any motivated actor.

---

### Recommendation

Add a role guard to `sendETHToL1ViaBridge` so only the pool (or a designated caller) can invoke it:

```solidity
function sendETHToL1ViaBridge(address l2bridge, address target, uint256 value)
    external payable nonReentrant onlyRole(BRIDGER_ROLE) {
```

Alternatively, restrict `l2bridge` and `target` to a protocol-controlled allowlist set at construction/admin time, preventing arbitrary callers from consuming bridge capacity.
