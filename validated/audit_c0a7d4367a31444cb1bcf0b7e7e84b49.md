Audit Report

## Title
Chainlink `latestRoundData()` Return Values Not Validated for Staleness or Round Completeness - (`contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` discards all `latestRoundData()` return values except `price`, performing no staleness, incomplete-round, or non-negative price checks. A stale or zero price propagates into `LRTOracle._updateRsETHPrice()`, which can trigger the downside-protection auto-pause and temporarily freeze all user deposits and withdrawals. The same stale price also corrupts the rsETH minting ratio in `LRTDepositPool.getRsETHAmountToMint()`.

## Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol` L49–55, `getAssetPrice()` fetches the Chainlink price as:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

`roundId`, `updatedAt`, and `answeredInRound` are silently discarded. No checks are performed for `price > 0`, `answeredInRound >= roundId`, or `updatedAt > 0`.

By contrast, `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` L30–32 — another oracle wrapper in the same repository — correctly validates all three conditions:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

**Exploit path (temporary freeze):**
1. A Chainlink feed for any supported asset enters a stale round (`answeredInRound < roundId`) and returns `price = 0` or a significantly outdated value.
2. Any unprivileged caller invokes `LRTOracle.updateRSETHPrice()` (public, `whenNotPaused` only — no role check).
3. `_updateRsETHPrice()` → `_getTotalEthInProtocol()` → `getAssetPrice(asset)` returns 0 for the stale asset, zeroing out that asset's contribution to `totalETHInProtocol`.
4. `newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply)` is understated.
5. The downside-protection check at L270–281 fires: `diff > pricePercentageLimit.mulWad(highestRsethPrice)` → `lrtDepositPool.pause()`, `withdrawalManager.pause()`, `_pause()` are called, freezing all deposits and withdrawals until an admin manually unpauses.

**Exploit path (incorrect minting):**
A stale non-zero price (e.g., 10% above actual) causes `getRsETHAmountToMint()` at `LRTDepositPool.sol` L520 to over-mint rsETH, diluting existing holders; a stale price below actual causes under-minting, harming the depositor.

Existing guards are insufficient: `updatePriceOracleForValidated()` at `LRTOracle.sol` L102–107 checks the price is between `1e16` and `1e19` at oracle registration time, but does not prevent a registered feed from later returning a stale or zero value at call time.

## Impact Explanation
**Primary — Medium: Temporary freezing of funds.** Any unprivileged caller can trigger `updateRSETHPrice()` at any time. If a supported Chainlink feed returns a stale/zero price, the auto-pause mechanism at `LRTOracle.sol` L277–281 freezes `LRTDepositPool` and `WithdrawalManager`, blocking all user deposits and withdrawals until an admin manually unpauses. This requires `pricePercentageLimit > 0`, a standard production configuration.

**Secondary — Low: Contract fails to deliver promised returns.** A stale non-zero price causes `getRsETHAmountToMint()` to compute an incorrect rsETH amount, either over-minting (diluting existing holders) or under-minting (loss to the depositor), without any revert.

## Likelihood Explanation
Chainlink feeds can return stale data during network congestion, oracle node downtime, or feed deprecation. The `answeredInRound < roundId` staleness condition is documented by Chainlink and has caused real incidents. `updateRSETHPrice()` is callable by any address with no privilege requirement, and `depositAsset()`/`depositETH()` are standard user-facing entry points. No special setup or victim mistake is required.

## Recommendation
Apply the same validation pattern already present in `ChainlinkOracleForRSETHPoolCollateral.sol` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    require(price > 0, "ChainlinkPriceOracle: price <= 0");
    require(answeredInRound >= roundId, "ChainlinkPriceOracle: stale price");
    require(updatedAt > 0, "ChainlinkPriceOracle: incomplete round");

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Additionally, consider a per-feed heartbeat check (`block.timestamp - updatedAt <= MAX_DELAY`) to guard against feeds that are technically complete but temporally stale.
