Looking at the actual code flow carefully:

### Title
FIFO-mode `remove_rejected_txs` calls `remove_by_address`, permanently evicting valid queued transactions for the same address — (`crates/apollo_mempool/src/mempool.rs`, `crates/apollo_mempool/src/fifo_transaction_queue.rs`)

---

### Summary

In Echonet (FIFO) mode, `Mempool::remove_rejected_txs` calls `tx_queue.remove_by_address(address)` after removing a rejected transaction from the pool. `FifoTransactionQueue::remove_by_address` unconditionally purges **all** entries for that address from both `queue` and `staged_txs`. Because FIFO mode enqueues every transaction for an address immediately (regardless of nonce), any future queued transaction H2 belonging to the same address as the rejected H1 is silently dropped from the queue while remaining in `tx_pool`. There is no re-enqueue path in FIFO mode, so H2 is permanently excluded from sequencing.

---

### Finding Description

**FIFO admission invariant:** `add_tx_inner` in FIFO mode inserts every accepted transaction directly into the queue:

```rust
if self.is_fifo() {
    // FIFO mode: add all transactions to the queue immediately, regardless of nonce.
    self.insert_to_tx_queue(tx_reference);
}
``` [1](#0-0) 

This means multiple transactions from the same address can coexist in `FifoTransactionQueue::queue` at different `block_number` slots.

**`remove_by_address` is address-wide, not tx-specific:**

```rust
fn remove_by_address(&mut self, address: ContractAddress) -> bool {
    let len_before = self.queue.len() + self.staged_txs.len();
    self.queue.retain(|tx| tx.tx_reference.address != address);
    self.staged_txs.retain(|tx| tx.tx_reference.address != address);
    ...
}
``` [2](#0-1) 

It removes every entry for the address, not just the one matching the rejected hash.

**`remove_rejected_txs` calls it unconditionally (no FIFO/fee-mode guard):**

```rust
if let Ok(tx) = self.tx_pool.remove(tx_hash) {
    self.tx_queue.remove_by_address(tx.contract_address());
    ...
}
``` [3](#0-2) 

**`rewind_txs` does not protect H2:** `collect_txs_to_rewind` only inspects `staged_txs`. H2 is in `queue` (not staged), so it is invisible to the rewind logic. Because H1 is rejected, address A is excluded from `addresses_to_rewind`, and `staged_txs.clear()` fires, leaving H2 unprotected in `queue`. [4](#0-3) 

**No re-enqueue path exists in FIFO mode:** Unlike fee mode (which calls `enqueue_next_eligible_txs` after each `get_txs` chunk), FIFO mode has no mechanism to re-insert a pool-resident transaction back into the queue. A subsequent `add_tx` for H2 would fail with `DuplicateTransaction`. [5](#0-4) 

**Exact corrupted state:** After `commit_block` with H1 rejected:
- `tx_pool` still contains H2 (only H1 was removed via `tx_pool.remove(H1)`)
- `tx_queue` no longer contains H2 (removed by `remove_by_address`)
- `tx_pool` and `tx_queue` are permanently inconsistent for H2

---

### Impact Explanation

H2 is a valid transaction admitted through the public gateway. It sits in `tx_pool` and is never returned by `get_txs` again, so it is permanently excluded from sequencing. This matches the allowed impact: **"High — Mempool/gateway/RPC admission rejects valid transactions before sequencing."**

---

### Likelihood Explanation

The path is reachable by any unprivileged user who:
1. Submits two transactions (H1, H2) for the same address in FIFO/Echonet mode.
2. H1 is staged by `get_txs` (moved to `staged_txs`).
3. H1 is rejected in the subsequent `commit_block` call.

No validator, operator, or admin privilege is required. The only precondition is that the node runs in Echonet mode and that H1 is rejected during block execution — a normal operational event.

---

### Recommendation

In `remove_rejected_txs`, replace the address-wide `remove_by_address` call with a hash-specific `remove_txs` call in FIFO mode, so only the rejected transaction is evicted from the queue:

```rust
// Instead of:
self.tx_queue.remove_by_address(tx.contract_address());

// Use (FIFO mode):
self.tx_queue.remove_txs(&[TransactionReference::from(&tx)]);
```

`remove_txs` already exists on `TransactionQueueTrait` and operates by hash, leaving unrelated same-address entries intact. [6](#0-5)

    fn remove_txs(&mut self, txs: &[TransactionReference]) -> Vec<TransactionReference> {
        let mut tx_hashes: HashSet<TransactionHash> = txs.iter().map(|tx| tx.tx_hash).collect();
        let mut removed_hashes: HashSet<TransactionHash> = HashSet::with_capacity(tx_hashes.len());
        self.queue.retain(|tx| {
            let tx_hash = tx.tx_reference.tx_hash;
            if tx_hashes.remove(&tx_hash) {
                removed_hashes.insert(tx_hash);
                false
            } else {
                true
            }
        });
        self.staged_txs.retain(|tx| {
            let tx_hash = tx.tx_reference.tx_hash;
            if tx_hashes.remove(&tx_hash) {
                removed_hashes.insert(tx_hash);
                false
            } else {
                true
            }
        });
        txs.iter().copied().filter(|tx| removed_hashes.contains(&tx.tx_hash)).collect()
    }
