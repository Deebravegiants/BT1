### Title
Missing Storage-Cost Enforcement on Attestation Re-Submission by Existing Participants — (File: `crates/contract/src/lib.rs`)

---

### Summary

`submit_participant_info` unconditionally skips the storage-deposit check when an existing participant re-submits their attestation. If the new `NodeAttestation` serialization is larger than the previous one, the contract silently absorbs the extra storage-staking cost from its own balance — the exact analog of the WETHGateway pattern where a missing revert causes the contract to repay from its own assets.

---

### Finding Description

In `crates/contract/src/lib.rs`, `submit_participant_info` records the pre-call storage baseline, calls `tee_state.add_participant(...)` (which overwrites the `stored_attestations` entry), then decides whether to enforce a deposit:

```rust
// lib.rs:823-824
let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {          // lib.rs:826
    let storage_used = env::storage_usage()
        .saturating_sub(initial_storage);               // lib.rs:831
    let cost = env::storage_byte_cost()
        .saturating_mul(storage_used as u128);          // lib.rs:832
    let attached = env::attached_deposit();             // lib.rs:833
    if attached < cost { return Err(...); }             // lib.rs:835-841
    // refund excess …
}
```

When the caller **is** an existing participant **and** the entry already exists (`UpdatedExistingParticipant`), both flags are `false`, so `attestation_storage_must_be_paid_by_caller` is `false` and the entire deposit-enforcement block is bypassed. The state mutation from `add_participant` is already committed at that point; the function returns `Ok(())` with zero deposit required.

`add_participant` in `tee_state.rs` performs an unconditional `IterableMap::insert` that replaces the old `NodeAttestation { node_id, verified_attestation }` value:

```rust
// tee_state.rs:191-197
let insertion = self.stored_attestations.insert(
    tls_pk,
    NodeAttestation { node_id, verified_attestation },
);
Ok(match insertion {
    Some(_) => ParticipantInsertion::UpdatedExistingParticipant,
    None    => ParticipantInsertion::NewlyInsertedParticipant,
})
```

`VerifiedAttestation` is a Borsh-serialized enum that can carry a full TDX attestation report including a variable-length advisory-ID list. A re-submission that carries more advisory IDs, a longer `account_id` string, or any other growth in the serialized `NodeAttestation` increases `env::storage_usage()` relative to `initial_storage`. Because the deposit check is skipped, the contract's own locked balance covers the additional storage-staking cost — exactly the "gateway repays from its own assets" pattern.

The node's `periodic_attestation_submission` task re-submits every hour. Each re-submission that grows the stored entry silently transfers storage-staking cost from the contract's balance to the participant's benefit.

---

### Impact Explanation

**Medium — balance and accounting-invariant violation.**

The contract's design invariant is that callers pay for the storage they consume. When an existing participant re-submits with a larger attestation payload, the contract's own NEAR balance is consumed to stake the extra bytes. Repeated re-submissions (hourly cadence) accumulate this cost. If the contract's balance is driven below the minimum required for its own storage, future state writes (new participant insertions, pending-request entries, etc.) will fail, breaking the liveness of the signing service. This matches the allowed Medium impact: *"Balance … manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

Every active MPC node calls `submit_participant_info` on a 1-hour cadence. A participant whose TDX quote carries a different advisory-ID set on re-submission (e.g., after a firmware update that adds or removes an advisory) will silently grow the stored entry. The participant need not do anything unusual; the growth is a natural consequence of the hardware's attestation output changing between submissions. Because no deposit is required, there is no economic friction preventing repeated re-submissions.

---

### Recommendation

Extend the storage-cost check to cover re-submissions by existing participants. Compute the net storage delta after `add_participant` regardless of `is_new_attestation`:

```rust
let storage_used = env::storage_usage().saturating_sub(initial_storage);
if storage_used > 0 {
    let cost = env::storage_byte_cost().saturating_mul(storage_used as u128);
    let attached = env::attached_deposit();
    if attached < cost {
        return Err(InvalidParameters::InsufficientDeposit {
            attached: attached.as_yoctonear(),
            required: cost.as_yoctonear(),
        }.into());
    }
    // refund excess …
}
```

This preserves the existing "charge nothing if the entry shrinks" semantics (via `saturating_sub`) while ensuring the contract never absorbs growth costs on behalf of a caller.

---

### Proof of Concept

1. Participant `alice.near` is an active MPC participant with a stored `NodeAttestation` of size *S* bytes.
2. `alice.near` calls `submit_participant_info` with a new valid TDX attestation whose serialized `VerifiedAttestation` is *S + Δ* bytes (e.g., one additional advisory ID in the report).
3. `add_participant` overwrites the entry; `env::storage_usage()` increases by *Δ* bytes.
4. `is_new_attestation` is `false` (entry existed); `caller_is_not_participant` is `false` (alice is a participant).
5. `attestation_storage_must_be_paid_by_caller` evaluates to `false`; the deposit block is skipped entirely.
6. The function returns `Ok(())` with zero deposit attached.
7. The contract's balance is reduced by `Δ × storage_byte_cost` to cover the new storage staking obligation.
8. Repeated hourly re-submissions accumulate this drain without any deposit from the participant.
