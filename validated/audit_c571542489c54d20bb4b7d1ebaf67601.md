### Title
`respond_ckd` Skips All Cryptographic Verification for `AppPublicKey` Variant, Allowing a Single Byzantine Participant to Forge CKD Responses - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` in the MPC smart contract performs a cryptographic output check (`ckd_output_check`) only when the request uses the `AppPublicKeyPV` variant. When the request uses the `AppPublicKey` (privately-verifiable) variant, the branch is an explicit no-op and **any `CKDResponse` is accepted unconditionally**. A single Byzantine attested participant can race the legitimate MPC protocol and submit a forged `(big_y, big_c)` pair, delivering a secret of the attacker's choosing to the user — bypassing the threshold requirement entirely.

---

### Finding Description

In `respond_ckd`, the response validation is gated on the variant of `app_public_key` embedded in the stored request:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically binds the response to the network key and the user's public key. [2](#0-1) 

For `AppPublicKey`, there is **no equivalent check**. The contract immediately proceeds to `resolve_yields_for`, which removes the pending request from the map and resumes the yield with whatever bytes were provided. [3](#0-2) 

The unit test for this path confirms the behavior — it passes dummy all-`1` and all-`2` byte arrays as `big_y`/`big_c` and the call succeeds: [4](#0-3) 

The `AppPublicKey` variant is the standard (non-PV) CKD mode and is used in production e2e tests: [5](#0-4) 

---

### Impact Explanation

The CKD protocol is designed so that `t` participants must cooperate to produce a correct `(big_y, big_c)` output. The contract's role is to verify the output before delivering it to the user. For `AppPublicKey`, the contract performs **zero verification**, so a single Byzantine attested participant can:

1. Monitor the chain for a pending `AppPublicKey` CKD request.
2. Immediately call `respond_ckd` with a crafted response — e.g., `big_y = G1_identity`, `big_c = g1 * x` for attacker-chosen scalar `x`.
3. The contract accepts it (no check), removes the pending request, and resumes the user's yield with the forged bytes.
4. The user computes their secret as `big_c − big_y · a = g1·x − 0·a = g1·x`. Since the attacker chose `x`, they know the user's derived secret.

This is a **threshold bypass**: the security guarantee of the MPC network (requiring `t`-of-`n` cooperation) is entirely circumvented for `AppPublicKey` CKD requests by a single participant. The user's confidential derived key is compromised.

The `respond_ckd` function does not restrict which participant may submit the response — any attested participant may call it, not only the designated leader: [6](#0-5) 

---

### Likelihood Explanation

- `AppPublicKey` is the default CKD mode (the `ckd-example-cli` uses it, and the e2e test suite exercises it).
- Any single attested participant who turns Byzantine can exploit this immediately upon seeing a pending request on-chain, before the multi-round MPC protocol completes.
- No collusion above threshold is required — one participant suffices.
- The attack is a simple on-chain transaction with attacker-controlled arguments; no off-chain cryptographic capability is needed.

---

### Recommendation

The `AppPublicKey` variant lacks a G2 component (`pk2`), so the existing pairing-based `ckd_output_check` cannot be applied directly. Mitigations include:

1. **Deprecate `AppPublicKey` in favor of `AppPublicKeyPV`** — require all CKD callers to supply both `pk1` and `pk2`, enabling the contract to always run `ckd_output_check`. The `AppPublicKeyPV` path is already fully verified.
2. **Require threshold-many co-signatures on the response** — have `t` participants each sign the `(big_y, big_c)` output off-chain; the contract verifies the aggregate before accepting.
3. **Restrict `respond_ckd` to the designated leader** — at minimum, prevent non-leader participants from submitting responses, reducing (but not eliminating) the attack surface.

---

### Proof of Concept

**Setup:** Contract is in Running state with participants `[P1, P2, P3]`, threshold `t = 2`. `P1` is Byzantine.

**Step 1 — User submits a CKD request with `AppPublicKey`:**
```
request_app_private_key({
  derivation_path: "my/path",
  app_public_key: AppPublicKey(g1 * a),   // user's G1 public key
  domain_id: 0
})
```
Contract stores the pending request and creates a yield.

**Step 2 — Byzantine `P1` races the MPC protocol:**
```
respond_ckd(
  request = <the pending CKDRequest>,
  response = CKDResponse {
    big_y: G1_identity,          // identity point (all-zero encoding)
    big_c: g1 * x,               // attacker-chosen point
  }
)
```
`P1` is an attested participant → passes `assert_caller_is_attested_participant_and_protocol_active`.
`AppPublicKey` branch → no `ckd_output_check` → accepted.
`resolve_yields_for` removes the request and resumes the yield with the forged bytes.

**Step 3 — User receives the forged response:**
User computes `secret = big_c − big_y · a = g1·x − identity·a = g1·x`.
Attacker knows `x`, therefore knows `secret = g1·x`. User's confidential key is compromised.

The legitimate MPC nodes' subsequent `respond_ckd` call returns `RequestNotFound` — the request was already consumed.

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

/// Verify that a CKD response (AppPublicKey variant) is mathematically correct.
#[tokio::test]
#[expect(non_snake_case)]
async fn ckd_response__passes_cryptographic_verification() {
    // given
    let (cluster, running) =
        common::must_setup_cluster(common::CKD_VERIFICATION_PORT_SEED, |_| {}).await;

    let bls_domain = running
        .domains
        .domains
        .iter()
        .find(|d| {
            Curve::from(d.protocol) == Curve::Bls12381 && matches!(d.purpose, DomainPurpose::CKD)
        })
        .expect("no Bls12381 CKD domain found")
        .clone();

    let mpc_pk = common::must_get_bls_public_key(&running, bls_domain.id);
    let user = cluster.default_user_account().clone();

    let mut rng = rand::rngs::StdRng::seed_from_u64(1);
    let private_key = Scalar::random(&mut rng);
    let app_public_key = CKDAppPublicKey::AppPublicKey(Bls12381G1PublicKey::from(
        &(G1Projective::generator() * private_key),
    ));

    // when
    let outcome = cluster
        .send_ckd_request(bls_domain.id, app_public_key, &user)
        .await
        .expect("CKD request transaction failed");

    // then
    assert!(
        outcome.is_success(),
        "CKD request failed: {:?}",
        outcome.failure_message()
    );

    let response: serde_json::Value = outcome.json().expect("failed to deserialize CKD response");
    let big_y: Bls12381G1PublicKey =
        serde_json::from_value(response["big_y"].clone()).expect("failed to parse big_y");
    let big_c: Bls12381G1PublicKey =
        serde_json::from_value(response["big_c"].clone()).expect("failed to parse big_c");

    assert!(
        verify_ckd(&user, DERIVATION_PATH, &mpc_pk, private_key, &big_y, &big_c)
            .expect("verify_ckd failed"),
        "CKD response failed cryptographic verification"
    );
}
