### Title
Unbounded NFT Wallet Auto-Creation via Incoming NFT Transfers Causes Permanent Wallet DoS - (`File: chia/wallet/wallet_state_manager.py`)

### Summary

`handle_nft()` in `WalletStateManager` unconditionally creates a new `NFTWallet` for every incoming NFT whose `new_did_id` does not match any existing NFT wallet, with no cap. An unprivileged attacker who mints NFTs with distinct DID owner IDs and transfers them to a victim's puzzle hash forces unbounded NFT wallet creation in the victim's wallet state manager, permanently degrading or halting wallet sync processing.

### Finding Description

When the wallet receives a new NFT coin state, `handle_nft()` is called. It extracts `new_did_id` from the NFT's ownership layer and checks whether any existing `NFTWallet` has a matching `did_id`. If none is found and the new p2 puzzle hash belongs to the victim (`new_derivation_record is not None`), a new `NFTWallet` is created unconditionally:

```python
if wallet_identifier is None and new_derivation_record is not None:
    new_nft_wallet: NFTWallet = await NFTWallet.create_new_nft_wallet(
        self, self.main_wallet, did_id=new_did_id, name="NFT Wallet"
    )
    wallet_identifier = WalletIdentifier.create(new_nft_wallet)
``` [1](#0-0) 

There is **no limit** on how many NFT wallets can be auto-created this way. This is in direct contrast to `handle_did()`, which explicitly enforces a configurable `did_auto_add_limit` (default 10):

```python
limit = self.config.get("did_auto_add_limit", 10)
if did_wallet_count < limit:
    did_wallet = await DIDWallet.create_new_did_wallet_from_coin_spend(...)
``` [2](#0-1) [3](#0-2) 

The `new_did_id` is parsed from the NFT's ownership layer puzzle solution via `get_new_owner_did()`. The wallet does **not** verify that the DID actually exists on-chain before creating the NFT wallet. An attacker can therefore mint NFTs with arbitrary fake DID IDs (no real DID creation required) and transfer them to the victim's puzzle hash. Each unique fake DID ID triggers a new `NFTWallet` entry in the victim's wallet state manager and database. [4](#0-3) [5](#0-4) 

### Impact Explanation

Each new `NFTWallet` is persisted to the wallet database via `user_store.create_wallet()` and registered in the in-memory `wallets` dict. With thousands of spurious NFT wallets:

- Every wallet sync cycle iterates over all wallets, causing O(N) overhead per coin state update.
- The wallet database grows unboundedly, causing disk exhaustion.
- The victim's wallet becomes permanently unable to process sync updates in a timely manner, constituting a long-lived DoS of wallet functionality.
- The victim cannot easily remediate: there is no RPC to bulk-delete spurious NFT wallets, and a full wallet resync would re-trigger the same auto-creation.

This maps to: **High — Permanent or long-lived inability for honest wallets to process sync updates under normal network assumptions**, and **High — Corruption of wallet sync state with direct security impact**.

### Likelihood Explanation

The attack requires only:
1. Minting NFTs with distinct fake DID IDs (arbitrary bytes32 values, no real DID needed).
2. Transferring each NFT to the victim's known puzzle hash (hinted).

NFT minting costs approximately 1 mojo per coin plus transaction fees. The attack is automatable and cheap. The victim's puzzle hash is derivable from their public key, which is observable on-chain from any prior transaction. No privileged access is required.

### Recommendation

Apply the same `did_auto_add_limit` pattern to `handle_nft()`. Count existing auto-created NFT wallets (those not explicitly created by the user) and refuse to create new ones beyond a configurable cap. Alternatively, require explicit user opt-in (e.g., an RPC call) before accepting NFTs from unknown DID owners into new wallets, mirroring the `automatically_add_unknown_cats` flag pattern. [6](#0-5) 

### Proof of Concept

1. Attacker observes victim's puzzle hash `victim_ph` from any prior on-chain transaction.
2. Attacker mints N NFTs, each with a distinct fake `owner_did` (e.g., `bytes32([i]*32)` for i in 0..N-1) pointing to `victim_ph` as the p2 puzzle hash, with a hint to `victim_ph`.
3. Attacker submits all N NFT transfer spend bundles to the mempool.
4. Once confirmed, the victim's wallet processes each coin state update via `_add_coin_states()` → `determine_coin_type()` → `handle_nft()`.
5. Each of the N distinct `new_did_id` values causes `create_new_nft_wallet()` to be called, creating N new `NFTWallet` entries.
6. Victim's wallet sync loop now iterates over N+existing wallets on every update; with N in the thousands, the wallet becomes unresponsive. [1](#0-0) [7](#0-6)

    @classmethod
    async def create_new_nft_wallet(
        cls: type[_T_NFTWallet],
        wallet_state_manager: Any,
        wallet: Wallet,
        did_id: bytes32 | None = None,
        name: str | None = None,
    ) -> _T_NFTWallet:
        """
        This must be called under the wallet state manager lock
        """
        self = cls()
        self.standard_wallet = wallet
        if name is None:
            name = "NFT Wallet"
        self.log = logging.getLogger(name if name else __name__)
        self.wallet_state_manager = wallet_state_manager
        self.nft_wallet_info = NFTWalletInfo(did_id)
        info_as_string = json.dumps(self.nft_wallet_info.to_json_dict())
        self.wallet_info = await wallet_state_manager.user_store.create_wallet(
            name, uint32(WalletType.NFT.value), info_as_string
        )
        self.wallet_id = self.wallet_info.id
        self.nft_store = wallet_state_manager.nft_store
        self.log.debug("NFT wallet id: %r and standard wallet id: %r", self.wallet_id, self.standard_wallet.wallet_id)

        await self.wallet_state_manager.add_new_wallet(self)
        self.log.debug("Generated a new NFT wallet: %s", self.__dict__)
        return self
