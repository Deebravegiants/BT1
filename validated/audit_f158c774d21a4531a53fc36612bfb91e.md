Audit Report

## Title
Zero rsETH Minted on Non-Zero Deposit Due to Rounding Down — (`contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolNoWrapper.sol`)

## Summary
The deposit functions in `RSETHPoolV3` and `RSETHPoolNoWrapper` compute `rsETHAmount` via integer division that rounds down. When the deposited amount is small enough that `amountAfterFee * 1e18 < rsETHToETHrate` (ETH path) or `amountAfterFee * tokenToETHRate < rsETHToETHrate` (token path), `rsETHAmount` evaluates to zero. The deposit proceeds without reverting, the user's ETH or token is consumed, and zero rsETH/wrsETH is minted or transferred.

## Finding Description
In `RSETHPoolV3.viewSwapRsETHAmountAndFee` (ETH path, line 307): `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate` rounds down. With `rsETHToETHrate = 1.05e18` and `amountAfterFee = 1 wei`, this yields `1 * 1e18 / 1.05e18 = 0`. The calling `deposit` function at line 256 only checks `if (amount == 0) revert InvalidAmount()` — it validates the input, not the output. It then unconditionally calls `wrsETH.mint(msg.sender, 0)` at line 262. The same pattern exists in `RSETHPoolNoWrapper.deposit` (line 241: `rsETH.safeTransfer(msg.sender, 0)`). Neither pool has a `require(rsETHAmount > 0)` guard nor a `minRsETHAmountExpected` slippage parameter. By contrast, `LRTDepositPool._beforeDeposit` (line 667) correctly guards with `if (rsethAmountToMint < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet()`. ## Impact Explanation
A depositor sends a non-zero ETH or token amount and receives zero rsETH/wrsETH. The deposited ETH accumulates in the pool and is eventually bridged to L1, where it increases the total ETH backing rsETH without increasing the rsETH supply, silently enriching all existing rsETH/wrsETH holders at the depositor's expense. This is a permanent, unrecoverable loss of the depositor's funds. This matches the **Low** impact class: "Contract fails to deliver promised returns."

## Likelihood Explanation
For the ETH path, the rounding threshold is `amountAfterFee < rsETHToETHrate / 1e18`. Since `rsETHToETHrate ≈ 1.0x × 1e18`, the threshold is approximately 1 wei, making accidental triggering negligible for normal deposits. For the token path with low token-to-ETH rates (e.g., `tokenToETHRate = 1e15`), the threshold rises to ~1050 token-wei, which is still small for 18-decimal tokens but more meaningful for 6-decimal tokens. Any unprivileged user can trigger this deliberately or accidentally by depositing a dust amount. No special permissions or conditions are required.

## Recommendation
Add a zero-output guard in every pool deposit function after computing `rsETHAmount`, mirroring the protection already present in `LRTDepositPool._beforeDeposit`:

```solidity
if (rsETHAmount == 0) revert InvalidAmount();
```

Optionally, expose a `minRsETHAmountExpected` parameter so callers can enforce their own slippage tolerance, consistent with `LRTDepositPool.depositETH` / `depositAsset`.

## Proof of Concept
1. Deploy with `rsETHToETHrate = 1.05e18` (5% staking yield accrued) and `feeBps = 0`.
2. Call `RSETHPoolV3.deposit{value: 1}("")` — deposits 1 wei ETH.
3. `viewSwapRsETHAmountAndFee(1)` computes: `fee = 0`, `amountAfterFee = 1`, `rsETHAmount = 1 * 1e18 / 1.05e18 = 0`.
4. `wrsETH.mint(msg.sender, 0)` executes without revert — user receives 0 wrsETH.
5. The 1 wei ETH remains in the pool, to be bridged to L1 and credited to existing holders.
6. Transaction succeeds; user has permanently lost 1 wei with no rsETH received.

For the token path with `tokenToETHRate = 1e15`: any deposit of ≤ 1049 token-wei silently yields 0 rsETH while consuming the depositor's tokens.

```text
    function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
    }
```
