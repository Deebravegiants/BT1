### Title
Presignature Signing-Set Ignores Chain Coverage, Causing Accepted `verify_foreign_transaction` Requests to Permanently Fail - (File: `crates/node/src/providers/verify_foreign_tx/sign.rs`)

---

### Summary

The `make_verify_foreign_tx_leader` function selects its signing participants directly from a pre-computed presignature whose participant set was chosen for **liveness** (online status), not **foreign-chain coverage**. When the presignature's signing set includes a participant that has no RPC inspector configured for the requested chain, that participant silently produces no signature share. If enough non-covering participants are included, the signing threshold is never reached. The generic request queue then retries the request up to `MAX_ATTEMPTS_PER_REQUEST_AS_LEADER = 10` times, consuming a presignature on each attempt, before permanently dropping the request. An unprivileged caller who submits a valid `verify_foreign_transaction` for a chain the contract has marked "available" can have their request permanently fail — a request-lifecycle invariant break the codebase itself documents as an open gap.

---

### Finding Description

**Root cause — participant election decoupled from chain coverage:**

In `make_verify_foreign_tx_leader`, the signing set is taken directly from the stored presignature:

```rust
let (presignature_id, presignature) = domain_data.presignature_store.take_owned().await;
let participants = presignature.participants.clone();
``` [1](#0-0) 

Presignatures are generated by `run_background_presignature_generation`, which selects participants via `select_random_active_participants_including_me` — a purely liveness-based selection that has no knowledge of which chains each participant covers:

```rust
let participants = match client
    .select_random_active_participants_including_me(num_signers, &running_participants)
``` [2](#0-1) 

The design documentation explicitly acknowledges this as an unimplemented requirement:

> Foreign-tx signing must elect participants that **cover** the requested chain (report ≥ `rpc_quorum(C)` providers for `C`), not merely online ones — a non-covering participant produces no share and can stall the request. **Implementation requirement, not current behavior: today the signing set is inherited from a presignature, whose participants were chosen for liveness, not chain coverage.** [3](#0-2) 

**Compounding factor — generic queue retries sub-quorum failures:**

When a non-covering follower reaches `execute_foreign_chain_request`, it hits the inspector lookup:

```rust
let inspector = self
    .inspectors
    .bitcoin
    .as_ref()
    .context("no inspector configured for bitcoin")?;
``` [4](#0-3) 

This returns an error, the follower produces no share, and the signing threshold is not met. The design doc also flags that the generic queue retries every request, including sub-quorum failures, which is explicitly not the intended behavior:

> **This sub-quorum outcome must be terminal — the leader must not re-attempt the request.** Implementation requirement, not current behavior: the generic queue retries every request, so the foreign-tx path must special-case a sub-quorum result as non-retryable. [5](#0-4) 

The queue retries up to `MAX_ATTEMPTS_PER_REQUEST_AS_LEADER = 10` times before dropping the request with `MaxAttemptsExceeded`: [6](#0-5) [7](#0-6) 

**Contract-side availability check does not prevent this:**

The contract's `verify_foreign_transaction` gates on `get_available_foreign_chains()`, which requires ≥ `signing_threshold` active participants to cover the chain. This check passes even when some participants do not cover the chain (up to `n − signing_threshold` non-covering nodes are tolerated). However, the presignature signing set is drawn from all online participants, so a non-covering participant can end up in the signing set even when the chain is "available": [8](#0-7) 

---

### Impact Explanation

A valid `verify_foreign_transaction` request accepted by the contract — with the chain correctly marked "available" — can permanently fail at the node layer because the signing set is not filtered for chain coverage. Each retry consumes a presignature (which requires expensive triple generation to replace). After `MAX_ATTEMPTS_PER_REQUEST_AS_LEADER` retries, the request is permanently dropped and the on-chain yield-resume promise times out. The user's request is never fulfilled despite the contract's availability guarantee being satisfied. This breaks the request-lifecycle invariant: the contract's acceptance of a request is supposed to guarantee it can reach the signing threshold, but the node-side implementation does not honor this guarantee.

---

### Likelihood Explanation

The scenario requires that the presignature's signing set include at least one participant that does not cover the requested chain. Given that:
- The design explicitly tolerates up to `n − signing_threshold` non-covering nodes (e.g., 1 of 4 in a 3-of-4 network),
- Presignature participants are chosen randomly from all online nodes,
- The probability of a non-covering node appearing in a signing set of size `2t+1` is non-negligible when even one node lacks coverage,

this condition arises naturally during any transient period of incomplete coverage — which the design explicitly acknowledges as an expected operational state, not a misconfiguration.

---

### Recommendation

1. **Fix participant election for foreign-tx signing:** When selecting participants for a `verify_foreign_transaction` signing session, filter the candidate set to only include participants that have registered coverage for the requested chain (via `get_available_foreign_chain_by_node()`). This is already identified as the correct fix in the design doc (tracked in issue #3434 / #3477).

2. **Make sub-quorum failures terminal:** The generic request queue must be extended to treat sub-quorum foreign-tx failures as non-retryable, as documented in `docs/design/calculating-supported-foreign-chains.md`. Retrying a request whose signing set structurally cannot reach threshold only wastes presignatures.

3. **Discard presignatures with non-covering participants before use:** Before consuming a presignature for a foreign-tx request, verify that all participants in the presignature cover the requested chain. If not, discard the presignature and select a new one whose participants all cover the chain.

### Proof of Concept

**Setup:** 4 participants (P1–P4), signing threshold = 3. Bitcoin is whitelisted and "available" (P1, P2, P3 cover Bitcoin; P4 does not). A presignature is generated with participants {P1, P2, P4} (P4 was online when the presignature was created).

**Attack path:**

1. Unprivileged caller submits `verify_foreign_transaction({ chain: Bitcoin, ... })` with 1 yoctoNEAR deposit.
2. Contract calls `get_available_foreign_chains()` → Bitcoin is available (3 ≥ threshold 3). Request is accepted and stored in `pending_verify_foreign_tx_requests`.
3. Node leader (P1) picks up the request, calls `domain_data.presignature_store.take_owned().await` → gets presignature with participants {P1, P2, P4}.
4. Leader opens a signing channel with {P1, P2, P4} and calls `execute_foreign_chain_request` for Bitcoin.
5. P4 reaches `self.inspectors.bitcoin.as_ref().context("no inspector configured for bitcoin")?` → returns `Err`, produces no signature share.
6. Leader receives shares from P1 and P2 only (2 shares < threshold 3). Signing fails.
7. Generic queue retries the request. Steps 3–6 repeat up to 10 times, consuming a presignature each time.
8. After 10 attempts, `ComputationProgressStatus::MaxAttemptsExceeded` is returned → request is permanently dropped.
9. The on-chain yield-resume promise times out after 200 blocks. The caller's request is never fulfilled. [9](#0-8) [10](#0-9) [11](#0-10)
