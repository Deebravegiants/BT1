### Title
Direct ETH Transfer to `LRTDepositPool` Inflates `rsETHPrice`, Causing Depositors to Receive Fewer (or Zero) rsETH Tokens - (File: contracts/LRTDepositPool.sol, contracts/LRTOracle.sol)

---

### Summary

`LRTDepositPool.getETHDistributionData()` uses `address(this).balance` directly as the ETH lying in the deposit pool. Because `LRTDepositPool` has an open `receive()` function, any actor can send ETH directly to the contract without minting rsETH. This inflates the reported total ETH in the protocol, which in turn inflates `rsETHPrice` when `updateRSETHPrice()` is called. Subsequent depositors then receive fewer rsETH tokens than they are entitled to — and if the inflation is large enough, they receive zero rsETH while their ETH is accepted.

---

### Finding Description

`LRTDepositPool` exposes an unrestricted `receive()` function: [1](#0-0) 

`getETHDistributionData()` reports the ETH balance of the deposit pool as `address(this).balance`: [2](#0-1) 

This value feeds directly into `getTotalAssetDeposits(ETH)`, which is consumed by `LRTOracle._getTotalEthInProtocol()`: [3](#0-2) 

`_updateRsETHPrice()` then computes the new rsETH price as: [4](#0-3) 

```
newRsETHPrice = (totalETHInProtocol - protocolFeeInETH) / rsethSupply
```

Finally, `getRsETHAmountToMint()` uses this stored price to determine how many rsETH tokens a depositor receives: [5](#0-4) 

```
rsethAmountToMint = (amount * assetPrice) / rsETHPrice
```

Because `rsETHPrice` is inflated by the donated ETH, the numerator `amount * assetPrice` can become smaller than `rsETHPrice`, causing integer division to truncate to zero.

`_beforeDeposit` only reverts if `rsethAmountToMint < minRSETHAmountExpected`: [6](#0-5) 

If a user passes `minRSETHAmountExpected = 0` (the default for many integrations), the call succeeds, ETH is accepted, and `_mintRsETH(0)` is called — minting zero rsETH to the depositor. [7](#0-6) 

---

### Impact Explanation

**Medium — Temporary freezing of funds / contract fails to deliver promised returns.**

- Depositors who do not set a meaningful `minRSETHAmountExpected` slippage guard can have their ETH accepted while receiving zero rsETH in return. Their ETH is permanently absorbed into the protocol TVL, benefiting existing rsETH holders.
- Even without reaching zero, inflated `rsETHPrice` causes all subsequent depositors to receive proportionally fewer rsETH tokens than the fair exchange rate warrants, constituting a loss of value relative to what the protocol promises.
- The `updateRSETHPrice()` function is public and callable by anyone, so the attacker can trigger the price update immediately after the donation. [8](#0-7) 

---

### Likelihood Explanation

**Low.** The attacker must sacrifice real ETH with no direct financial return (unless they hold a large rsETH position and can profit from the price increase on secondary markets). The `pricePercentageLimit` guard in `_updateRsETHPrice()` will revert non-manager callers if the price jump exceeds the configured threshold: [9](#0-8) 

However, if `pricePercentageLimit == 0` (unconfigured), there is no cap. Additionally, an attacker can execute the inflation gradually across multiple transactions, each staying within the limit, to cumulatively distort the price.

---

### Recommendation

1. **Do not use raw `address(this).balance` for TVL accounting.** Track ETH received through legitimate deposit flows in a dedicated storage variable (e.g., `totalDepositedETH`) and use that instead of `address(this).balance` in `getETHDistributionData()`. Any ETH received via `receive()` outside the deposit flow should not count toward TVL.
2. **Reject zero-rsETH mints.** Add a check in `_beforeDeposit` or `_mintRsETH` that reverts if `rsethAmountToMint == 0`, regardless of `minRSETHAmountExpected`.
3. **Ensure `pricePercentageLimit` is always set to a non-zero value** as part of deployment and upgrade procedures.

---

### Proof of Concept

```
State: rsethSupply = 1000 ETH worth, totalETHInProtocol = 1000 ETH, rsETHPrice = 1e18

Step 1: Attacker calls LRTDepositPool.receive() sending 1000 ETH directly.
        address(this).balance increases by 1000 ETH.

Step 2: Attacker (or anyone) calls LRTOracle.updateRSETHPrice().
        totalETHInProtocol = 2000 ETH (includes donated ETH).
        newRsETHPrice = 2000e18 / 1000 = 2e18.
        rsETHPrice is now 2e18.

Step 3: Victim calls LRTDepositPool.depositETH{value: 1 ETH}(0, "").
        rsethAmountToMint = (1e18 * 1e18) / 2e18 = 0.5e18  ← victim gets half the expected rsETH.

Step 4 (extreme): If attacker donates enough to make rsETHPrice >> deposit amount:
        rsethAmountToMint = (1e18 * 1e18) / 1000e18 = 0 (integer truncation).
        minRSETHAmountExpected = 0 → no revert.
        _mintRsETH(0) → victim deposits 1 ETH, receives 0 rsETH.
```
