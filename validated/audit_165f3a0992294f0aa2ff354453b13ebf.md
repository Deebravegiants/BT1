### Title
Missing On-Chain Cryptographic Verification for `CKDAppPublicKey::AppPublicKey` Variant in `respond_ckd` — (`crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function performs no cryptographic verification of the `CKDResponse` when the CKD request uses the `CKDAppPublicKey::AppPublicKey` (non-PV) variant. A single attested MPC participant can submit an arbitrary `(big_y, big_c)` pair and the contract will accept it, resolving the caller's yield with a forged CKD output that is not bound to the MPC network's secret key.

---

### Finding Description

In `respond_ckd`, the match on `request.app_public_key` is asymmetric: [1](#0-0) 

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For the `AppPublicKeyPV` variant, `ckd_output_check` enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(H(pk‖app_id), msk_pk)` using BLS12-381 host functions: [2](#0-1) 

This pairing check cryptographically binds `big_c` and `big_y` to the network's master secret key and the caller's `app_id`. For the `AppPublicKey` variant, the empty arm `{}` means **no such binding is enforced**. The response is immediately forwarded to the yield resolver: [3](#0-2) 

The only guards before reaching this point are:
- `assert_caller_is_signer()` — caller must be an attested participant
- `is_running_or_resharing()` — protocol must be active
- `accept_requests` flag [4](#0-3) 

None of these prevent a single Byzantine attested participant from submitting an arbitrary response.

---

### Impact Explanation

A single attested MPC participant (below the signing threshold) can:

1. Observe a pending CKD request using the `AppPublicKey` variant.
2. Call `respond_ckd(request, CKDResponse { big_y: <arbitrary>, big_c: <arbitrary> })`.
3. The contract accepts and resolves the yield with the forged output.
4. The requesting application receives `(big_y, big_c)` that does not satisfy `big_c = msk·H(pk‖app_id) + big_y·a`, i.e., the output is not bound to the MPC network's secret key.

A concrete maximally-damaging instantiation: the attacker sets `big_y = G1_identity` (the point at infinity). Then the caller computes `big_c − big_y·a = big_c`, a value the attacker chose and knows. The attacker has delivered a derived key whose value they fully control and know, without any MPC computation having occurred.

This matches the allowed critical impact: **unauthorized confidential key derivation output without the required participant authorization**.

---

### Likelihood Explanation

- Requires only a single attested MPC participant to act Byzantine — well below the threshold.
- The `AppPublicKey` (non-PV) variant is a documented, production-reachable code path (it is the legacy/default variant per the `Deserialize` impl which maps plain `Bls12381G1PublicKey` to `AppPublicKey`). [5](#0-4) 

- No special tooling or privileged access beyond attestation is required.
- The attack is a single on-chain transaction.

---

### Recommendation

The `AppPublicKey` variant lacks a G2 component (`pk2`), making the pairing-based `ckd_output_check` inapplicable. The fix must address this at the protocol level:

1. **Deprecate the `AppPublicKey` variant** in production and require all callers to use `AppPublicKeyPV`, which provides on-chain verifiability.
2. **Or**, if the non-PV variant must be retained, enforce threshold participation: require a quorum of attested participants to submit matching `(big_y, big_c)` values before resolving the yield, so no single Byzantine participant can forge the output.
3. **Or**, require the `AppPublicKey` caller to also supply a G2 commitment (converting it to the PV variant) so the pairing check can be applied.

---

### Proof of Concept

Build a unit/sandbox test that:

1. Submits a CKD request with `CKDAppPublicKey::AppPublicKey(some_g1_pk)`.
2. As an attested participant, calls `respond_ckd(request, CKDResponse { big_y: G1_identity_compressed, big_c: G1_generator_compressed })`.
3. Asserts the contract returns `Ok(())` and the yield is resolved with the forged `(big_y, big_c)`.
4. Verifies that `big_c ≠ msk·H(pk‖app_id) + big_y·a` (trivially true since `big_y = 0` and `big_c = g`).

The test at `crates/contract/src/primitives/ckd.rs:563-575` already shows the `AppPublicKey` variant is used with `[1u8; 48]` as a raw key with no verification concern raised, confirming the path is reachable and unguarded. [6](#0-5)

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
