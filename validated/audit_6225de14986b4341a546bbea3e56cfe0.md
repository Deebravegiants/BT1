### Title
`respond_verify_foreign_tx` Does Not Bind `payload_hash` to the Pending Request, Enabling Cross-Request Signature Replay — (`File: crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that the submitted ECDSA signature is valid over the caller-supplied `response.payload_hash`, but never verifies that `payload_hash` is the correct hash for the pending `VerifyForeignTransactionRequest`. A single Byzantine MPC participant can replay a valid `VerifyForeignTransactionResponse` produced for a previous legitimate request (request_A) to satisfy a completely different pending request (request_B), causing the victim caller to receive a forged foreign-chain verification attestation.

### Finding Description

The `respond_verify_foreign_tx` function accepts a `VerifyForeignTransactionResponse` containing a caller-supplied `payload_hash` and a `signature`. The contract verifies only that the signature is valid over the provided `payload_hash` using the domain's root public key: [1](#0-0) 

The `payload_hash` is taken directly from the response argument — it is never cross-checked against the `request` argument that identifies which pending yield to resolve: [2](#0-1) 

The signed payload that MPC nodes produce is `SHA-256(borsh(ForeignTxSignPayloadV1{request, values}))`, where `values` are the extracted foreign-chain observations: [3](#0-2) 

Because `VerifyForeignTransactionResponse` carries only `payload_hash` and `signature` — not the `values` — the contract has no way to recompute the expected hash from the pending `request` and verify it matches: [4](#0-3) 

Contrast this with the regular `respond` function, where the payload hash is derived directly from the stored `request.payload` (not from the response), making replay impossible: [5](#0-4) 

The `respond_verify_foreign_tx` function is callable by any single attested participant — there is no threshold requirement at the contract level: [6](#0-5) 

### Impact Explanation

A Byzantine MPC participant who observed a legitimately produced `VerifyForeignTransactionResponse` for request_A (e.g., Bitcoin tx_A) can submit it as the response for a different pending request_B (e.g., Bitcoin tx_B). The contract accepts it because the signature is valid over `payload_hash_A`. The victim caller of `verify_foreign_transaction(tx_B)` receives `{payload_hash_A, signature_A}` — a valid MPC signature that attests to tx_A's data, not tx_B's. Any downstream NEAR contract (e.g., an Omnibridge) that uses this response to authorize a cross-chain action (e.g., releasing funds) will act on a forged attestation, enabling invalid bridge execution or double-spend conditions.

This matches the allowed impact: **High — forged foreign-chain verification causing invalid bridge execution**.

### Likelihood Explanation

Only a single Byzantine MPC participant (strictly below the signing threshold) is required. The participant needs:
1. A previously produced valid `VerifyForeignTransactionResponse` for any request — trivially available to any participant who took part in that signing round.
2. A pending `verify_foreign_transaction` request for a different transaction — any user can create one.

No threshold collusion, no privileged access, and no network-level attack is required.

### Recommendation

Include the extracted `values` in `VerifyForeignTransactionResponse` and have the contract recompute and verify the `payload_hash` before resolving the yield:

```rust
// In respond_verify_foreign_tx, after receiving request and response:
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.values.clone(), // values added to response DTO
})
.compute_msg_hash()
.map_err(|_| RespondError::PayloadHashComputationFailed)?;

if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

This mirrors how `respond` derives the payload hash from the stored request rather than trusting the caller-supplied value.

### Proof of Concept

1. Alice submits `verify_foreign_transaction(tx_B)` → pending request_B is queued in `pending_verify_foreign_tx_requests`.
2. The MPC network previously produced `response_A = {payload_hash_A, signature_A}` for a legitimate request_A (tx_A). Byzantine participant P captured this response.
3. P calls `respond_verify_foreign_tx(request_B, response_A)`.
4. Contract checks: `verify_ecdsa(signature_A, payload_hash_A, root_pk)` → **valid** (signature was honestly produced for tx_A).
5. Contract calls `resolve_yields_for(&mut self.pending_verify_foreign_tx_requests, &request_B, serialize(response_A))`.
6. Alice's yield resumes with `{payload_hash_A, signature_A}` — a valid MPC signature that attests to tx_A's data, not tx_B's.
7. Alice's downstream contract verifies the signature is valid and proceeds with a bridge action based on the forged attestation. [7](#0-6) [8](#0-7)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
