### Title
Token-2022 Transfer Fee Extension Causes Unbacked NEAR Supply and Irreversible Vault Shortfall in Solana `init_transfer` - (`solana/programs/bridge_token_factory/src/instructions/user/init_transfer.rs`)

### Summary

The Solana bridge program accepts Token-2022 tokens via a generic `TokenInterface` but does not account for the Token-2022 **transfer fee extension**. When a native Token-2022 token with a transfer fee is bridged Solana → NEAR via `init_transfer`, the vault receives `amount - fee` tokens while the Wormhole message reports the full `amount`. NEAR mints `amount` wrapped tokens, creating unbacked supply. On the return path (`finalize_transfer`), the vault cannot cover the full redemption, permanently locking the fee-withheld portion.

### Finding Description

In `init_transfer.rs`, the `process` function performs two independent operations:

1. **Lock tokens in vault** via `transfer_checked` with `payload.amount`:

```rust
transfer_checked(
    CpiContext::new(..., TransferChecked {
        from: self.from.to_account_info(),
        to: vault.to_account_info(),
        ...
    }),
    payload.amount.try_into()...,  // full user-supplied amount
    self.mint.decimals,
)?;
```

2. **Post Wormhole message** with the same `payload.amount`:

```rust
self.common.post_message(payload.serialize_for_near((
    self.common.sequence.sequence,
    self.user.key(),
    self.mint.key(),
))?)?;
```

`serialize_for_near` encodes `self.amount` (i.e., `payload.amount`) verbatim into the cross-chain message.

With the Token-2022 transfer fee extension, `transfer_checked` succeeds but the vault's **spendable** balance increases by only `payload.amount - fee`. The fee is withheld inside the vault's token account and is only recoverable by the fee authority — it is not available for future `finalize_transfer` redemptions. The Wormhole VAA, however, carries the full `payload.amount`, so NEAR mints `payload.amount` wrapped tokens to the user.

The same divergence exists in `finalize_transfer.rs`: `transfer_checked` is called with `data.amount` but the recipient's ATA receives `data.amount - fee`, while the Wormhole confirmation message reports `data.amount` — causing direct loss to the recipient on every NEAR → Solana redemption.

The Solana `SECURITY.md` explicitly documents that **transfer hooks** are unsupported (causing denial, not fund loss), but makes no mention of the **transfer fee extension**, which is a distinct Token-2022 feature that does not require extra account metas and does not cause a runtime failure — it silently reduces the transferred amount. ### Impact Explanation

**Critical — unbacked supply creation (Solana → NEAR):** Each `init_transfer` with a fee-bearing Token-2022 token mints more NEAR-side wrapped tokens than are backed by vault collateral. The shortfall accumulates with every transfer. When users attempt to redeem, `finalize_transfer` will attempt to transfer the full amount from the vault but the vault's spendable balance is insufficient, permanently locking the withheld fee portions and making some redemptions fail.

**High — direct recipient loss (NEAR → Solana):** `finalize_transfer` calls `transfer_checked(vault, recipient_ata, data.amount)`. The recipient's ATA receives `data.amount - fee`. The NEAR side burned `data.amount` tokens and considers the transfer complete. The fee difference is silently lost to the recipient on every outbound redemption.

Both paths map to the allowed impact scope: the first is unauthorized creation of unbacked wrapped assets; the second is balance-accounting divergence that sends value to the wrong party (the fee authority, not the recipient).

### Likelihood Explanation

`log_metadata` is permissionless — any Token-2022 token with a transfer fee extension can be registered. `init_transfer` accepts any `TokenInterface`-compatible token program without inspecting mint extensions. No privileged role is required. Any user who holds a Token-2022 token with a transfer fee can trigger this path by calling `init_transfer` normally. [6](#0-5) 

### Recommendation

Before performing `transfer_checked`, inspect the mint's Token-2022 extensions and reject (or explicitly handle) mints that carry a `TransferFeeConfig` extension. Anchor's `spl_token_2022` crate exposes `StateWithExtensions::unpack` and `get_extension::<TransferFeeConfig>()` for this purpose. Alternatively, document Token-2022 transfer-fee tokens as unsupported (matching the EVM `SECURITY.md` treatment of fee-on-transfer tokens) and add a runtime check that reverts if the extension is present. [7](#0-6) 

### Proof of Concept

1. Deploy a Token-2022 mint with `TransferFeeConfig` set to 1% (100 bps).
2. Register it via `log_metadata` (permissionless), which creates the vault PDA.
3. Call `init_transfer` with `payload.amount = 1_000_000`.
4. `transfer_checked` succeeds; vault's spendable balance increases by `990_000`; `10_000` tokens are withheld inside the vault account.
5. Wormhole VAA carries `amount = 1_000_000`; NEAR mints `1_000_000` wrapped tokens to the user.
6. User bridges back: NEAR burns `1_000_000` tokens and issues a signed `FinalizeTransferPayload { amount: 1_000_000 }`.
7. `finalize_transfer` calls `transfer_checked(vault, recipient_ata, 1_000_000)`.
8. Vault spendable balance is only `990_000` → transfer fails; `10_000` tokens are permanently locked in the vault's withheld balance, and the NEAR-side wrapped tokens are unbacked. [8](#0-7) [9](#0-8)
