### Title
`respond_verify_foreign_tx` Accepts Any Valid Signature Regardless of Request Binding — (`File: crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is cryptographically valid over `response.payload_hash`, but never checks that `response.payload_hash` was actually derived from the `request` argument. A single malicious attested participant (strictly below the signing threshold) can replay a legitimately-produced MPC signature from one pending foreign-tx request to satisfy a completely different pending request, delivering a forged attestation to the caller.

---

### Finding Description

The `respond_verify_foreign_tx` entry point in `crates/contract/src/lib.rs` performs the following validation:

```rust
// lines 718-747
let signature_is_valid = match (&response.signature, public_key) {
    (
        dtos::SignatureResponse::Secp256k1(signature_response),
        PublicKeyExtended::Secp256k1 { near_public_key },
    ) => {
        let payload_hash: [u8; 32] = response.payload_hash.0;
        // Check the signature is correct against the root public key
        near_mpc_signature_verifier::verify_ecdsa_signature(
            signature_response,
            &payload_hash,
            &secp_pk,
        ).is_ok()
    }
    ...
};
if !signature_is_valid {
    return Err(RespondError::InvalidSignature.into());
}
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [1](#0-0) 

The check confirms only: *"is `response.signature` a valid ECDSA signature over `response.payload_hash` under the domain root key?"*

It does **not** confirm: *"does `response.payload_hash` encode the `request` that was submitted?"*

The canonical payload hash is defined as:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))
``` [2](#0-1) 

Because `request` (including `tx_id`, `chain`, `extractors`) is embedded inside the hash preimage, a hash produced for request R2 is cryptographically distinct from one produced for request R1. The contract never recomputes or prefix-checks the hash against the `request` argument it received; it simply trusts whatever `payload_hash` the caller supplies, provided the signature over it is valid.

---

### Impact Explanation

**Impact: High — forged foreign-chain verification enabling invalid bridge execution.**

A single malicious attested participant (below the signing threshold) can:

1. Wait for the MPC network to legitimately complete a `respond_verify_foreign_tx` call for request R2 (e.g., Bitcoin tx_id = B). The final on-chain transaction is public and contains `{payload_hash: H(R2, values2), signature: S2}`.
2. While a different request R1 (tx_id = A) is pending, call `respond_verify_foreign_tx(request = R1, response = {payload_hash: H(R2, values2), signature: S2})`.
3. The contract accepts the call: `S2` is a valid signature over `H(R2, values2)` under the root key. The yields for R1 are resolved and removed from the pending map.
4. The caller of R1 receives a `VerifyForeignTransactionResponse` whose `payload_hash` encodes tx_id = B, not tx_id = A.

If the downstream bridge contract does not independently verify the `payload_hash` against its expected request (which is the common case for contracts that trust the MPC network as an oracle), it will treat the attestation as proof that tx_id = A was finalized on the foreign chain, when in fact it was never verified. This enables:

- **Invalid bridge execution**: minting tokens on NEAR for a deposit that never occurred on the foreign chain.
- **Permanent denial of the legitimate response**: once yields for R1 are drained, the correct response can never be submitted; the legitimate caller's request is permanently consumed.

The NEAR MPC SDK's `ForeignChainSignatureVerifier::verify_signature` does perform the payload-hash binding check:

```rust
let payload_is_correct = expected_payload_hash == response.payload_hash;
if !payload_is_correct {
    return Err(VerifyForeignChainError::IncorrectPayloadSigned { ... });
}
``` [3](#0-2) 

However, this check lives in the SDK used by callers, not in the contract itself. Callers that do not use the SDK, or that trust the contract's implicit guarantee that the response corresponds to their request, are fully exposed.

---

### Likelihood Explanation

The attack requires only a single malicious attested participant — strictly below the signing threshold. No key material needs to be forged; the attacker simply replays a legitimately-produced signature observed from a prior on-chain transaction. The attacker needs:

1. An active attested participant identity (required to call `respond_verify_foreign_tx`).
2. A pending request R1 to target.
3. Any previously-produced valid response for any other request R2 (observable from chain history).

All three conditions are routinely satisfied in production. The Omnibridge inbound flow is the primary use case for `verify_foreign_transaction`, making bridge contracts the highest-value targets.

---

### Recommendation

The contract should bind the response to the request before accepting it. Since the extracted `values` are not stored on-chain, the contract cannot recompute the full `payload_hash`. The fix is to include the extracted values in the response so the contract can verify the binding:

**Short term**: Extend `VerifyForeignTransactionResponse` to include the `Vec<ExtractedValue>` alongside `payload_hash`. In `respond_verify_foreign_tx`, recompute `expected_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))` and assert `expected_hash == response.payload_hash` before accepting the response.

**Long term**: Ensure all downstream bridge contracts use `ForeignChainSignatureVerifier::verify_signature` from the NEAR MPC SDK to independently verify the payload binding, providing defense-in-depth even if the contract-level check is ever bypassed.

---

### Proof of Concept

```
1. Alice submits verify_foreign_transaction(request = R1 { tx_id = A, chain = Bitcoin, extractors = [BlockHash] })
   → R1 is stored in pending_verify_foreign_tx_requests

2. Bob submits verify_foreign_transaction(request = R2 { tx_id = B, chain = Bitcoin, extractors = [BlockHash] })
   → MPC nodes process R2, produce S2 = sign(H(R2, {block_hash: 0xdeadbeef...}))
   → Honest node calls respond_verify_foreign_tx(request=R2, response={payload_hash=H2, signature=S2})
   → This transaction is visible on-chain

3. Malicious attested participant M observes H2 and S2 from step 2.
   M calls respond_verify_foreign_tx(request=R1, response={payload_hash=H2, signature=S2})

4. Contract checks:
   - M is an attested participant ✓
   - verify_ecdsa_signature(S2, H2, root_pk) == Ok ✓  [S2 is a valid signature over H2]
   - R1 exists in pending map ✓
   → resolve_yields_for(R1, {payload_hash=H2, signature=S2})

5. Alice's contract receives VerifyForeignTransactionResponse{payload_hash=H2, signature=S2}
   H2 encodes tx_id=B, not tx_id=A.
   If Alice's bridge contract does not call ForeignChainSignatureVerifier::verify_signature,
   it treats this as proof that tx_id=A was finalized → mints tokens for a non-existent deposit.

6. The correct response for R1 (tx_id=A) can never be submitted; R1 is no longer pending.
``` [4](#0-3) [5](#0-4)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
