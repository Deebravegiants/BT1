### Title
Unverified CKD Response Delivery for `AppPublicKey` Variant Allows Single Attested Participant to Inject Attacker-Controlled Derived Key - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd()` performs cryptographic output verification **only** when the request uses the `CKDAppPublicKey::AppPublicKeyPV` variant. When the request uses the `CKDAppPublicKey::AppPublicKey` variant, the contract accepts and delivers **any** `CKDResponse` without verification. A single Byzantine attested participant (strictly below the signing threshold) can monitor pending CKD requests on-chain and front-run the legitimate MPC nodes' `respond_ckd` call with an attacker-crafted `(big_y, big_c)` pair, delivering a derived key they control to the user.

---

### Finding Description

In `respond_ckd()`, the response verification branch is:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no verification at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

The `AppPublicKey` arm is a no-op: the contract immediately proceeds to `resolve_yields_for`, delivering the caller-supplied `response` to all queued yield promises for that request. [2](#0-1) 

The `AppPublicKeyPV` arm, by contrast, calls `ckd_output_check`, which enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)` — a cryptographic proof that the response was produced using the real MPC master secret. [3](#0-2) 

The `AppPublicKey` variant is the primary/legacy variant used in production (as shown in the e2e test `ckd_response__passes_cryptographic_verification`). [4](#0-3) 

The only gate before the unverified delivery is `assert_caller_is_attested_participant_and_protocol_active()`, which requires the caller to be a registered, TEE-attested participant — a single node, not a threshold quorum. [5](#0-4) 

---

### Impact Explanation

The CKD protocol is designed so that the user's derived application secret key is encrypted to their ephemeral `app_public_key`. The user decrypts it as `derived_key = big_c − app_sk · big_y`. An attacker who controls `(big_y, big_c)` can craft a pair where they know the value the user will compute:

- Choose random `y` and a fake scalar `msk_fake`.
- Set `big_y = g1 · y`, `big_c = app_pk · y + hash_point · msk_fake`.
- The user computes `big_c − app_sk · big_y = hash_point · msk_fake` — a value the attacker knows, not the real MPC-derived key.

The user receives a silently wrong derived key with no indication of tampering. The legitimate MPC response is permanently blocked because `resolve_yields_for` drains all queued yield promises for the request in one call.

This constitutes **unauthorized confidential key derivation output without the required threshold participant authorization** — a Critical impact under the allowed scope.

---

### Likelihood Explanation

- The attacker needs only **one** attested participant account — strictly below the signing threshold.
- Pending CKD requests are visible on-chain (stored in `pending_ckd_requests` and emitted as NEAR indexer events).
- The attacker simply submits `respond_ckd` with a fabricated response before the honest nodes do. No mempool manipulation is needed on NEAR (transactions are ordered by the block producer), but the attacker can submit with maximum gas priority or simply race the honest nodes.
- The `AppPublicKey` variant is the default production path. [6](#0-5) 

---

### Recommendation

Apply the same pairing-based output verification to the `AppPublicKey` variant. For `AppPublicKey(pk1)`, the verifiable condition is equivalent: construct a synthetic `AppPublicKeyPV` with `pk2 = g2 · dlog(pk1)` — but since `dlog` is unknown, the correct approach is to extend the `AppPublicKey` variant to also carry a `pk2` component (making it equivalent to `AppPublicKeyPV`), or to require all new CKD requests to use `AppPublicKeyPV`.

As an immediate mitigation, add a `ckd_output_check`-equivalent for the `AppPublicKey` arm that verifies the response against the known `pk1` using the available pairing equation, or reject `AppPublicKey`-variant requests in `respond_ckd` until a verification path is implemented.

---

### Proof of Concept

1. User calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(pk1)` where `pk1 = g1 · app_sk`.
2. The request is stored in `pending_ckd_requests` and visible on-chain.
3. Byzantine attested participant `evil.near` observes the pending request.
4. `evil.near` chooses `y = 42`, computes `big_y = g1 · 42`, `big_c = pk1 · 42 + hash_point · msk_evil` (for any known `msk_evil`).
5. `evil.near` calls `respond_ckd(request, CKDResponse { big_y, big_c })`.
6. The contract passes `assert_caller_is_attested_participant_and_protocol_active()`, enters the `AppPublicKey(_) => {}` arm with no check, and calls `resolve_yields_for` — delivering the fabricated response to the user. [7](#0-6) 

7. The user decrypts: `big_c − app_sk · big_y = hash_point · msk_evil` — an attacker-known value, not the real MPC-derived key.
8. The legitimate `respond_ckd` from honest nodes arrives too late; the yield queue is already drained and the call returns `RequestNotFound`.

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

/// Verify that a CKD response (AppPublicKey variant) is mathematically correct.
#[tokio::test]
#[expect(non_snake_case)]
async fn ckd_response__passes_cryptographic_verification() {
    // given
    let (cluster, running) =
        common::must_setup_cluster(common::CKD_VERIFICATION_PORT_SEED, |_| {}).await;

    let bls_domain = running
        .domains
        .domains
        .iter()
        .find(|d| {
            Curve::from(d.protocol) == Curve::Bls12381 && matches!(d.purpose, DomainPurpose::CKD)
        })
        .expect("no Bls12381 CKD domain found")
        .clone();

    let mpc_pk = common::must_get_bls_public_key(&running, bls_domain.id);
    let user = cluster.default_user_account().clone();

    let mut rng = rand::rngs::StdRng::seed_from_u64(1);
    let private_key = Scalar::random(&mut rng);
    let app_public_key = CKDAppPublicKey::AppPublicKey(Bls12381G1PublicKey::from(
        &(G1Projective::generator() * private_key),
    ));

    // when
    let outcome = cluster
        .send_ckd_request(bls_domain.id, app_public_key, &user)
        .await
        .expect("CKD request transaction failed");

    // then
    assert!(
        outcome.is_success(),
        "CKD request failed: {:?}",
        outcome.failure_message()
    );

    let response: serde_json::Value = outcome.json().expect("failed to deserialize CKD response");
    let big_y: Bls12381G1PublicKey =
        serde_json::from_value(response["big_y"].clone()).expect("failed to parse big_y");
    let big_c: Bls12381G1PublicKey =
        serde_json::from_value(response["big_c"].clone()).expect("failed to parse big_c");

    assert!(
        verify_ckd(&user, DERIVATION_PATH, &mpc_pk, private_key, &big_y, &big_c)
            .expect("verify_ckd failed"),
        "CKD response failed cryptographic verification"
    );
}
