Audit Report

## Title
No Staleness Validation in `ChainlinkPriceOracle.getAssetPrice()` Allows Stale Price to Drive Inflated rsETH Minting or Protocol Auto-Pause - (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards `updatedAt`, `answeredInRound`, `roundId`, and performs no positivity check on `price`. The raw answer is returned unconditionally. This stale or invalid price propagates into rsETH minting (Path 1: over-minting that dilutes existing holders) and the publicly callable rsETH price update (Path 2: artificially low price triggers the auto-pause circuit breaker).

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` at L49–55 reads only the `answer` field:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

No check is made on `updatedAt` (staleness), `answeredInRound >= roundId` (round completeness), or `price > 0` (validity). By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` in the same repository performs all three checks: [2](#0-1) 

**Path 1 — rsETH minting (`getRsETHAmountToMint`):**

`LRTDepositPool.getRsETHAmountToMint()` computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [3](#0-2) 

If the Chainlink feed for a supported LST is stale (e.g., the asset has depegged but the oracle has not yet updated), the stale high price inflates `rsethAmountToMint`. The depositor receives rsETH calculated at the pre-depeg rate, extracting value from all existing rsETH holders whose claims on underlying ETH are diluted.

**Path 2 — rsETH price update (`_updateRsETHPrice` → `_getTotalEthInProtocol`):**

`_getTotalEthInProtocol()` accumulates:

```solidity
uint256 assetER = getAssetPrice(asset);
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
``` [4](#0-3) 

If any feed returns 0 (e.g., a deprecated or offline feed), `totalETHInProtocol` is underestimated. If the resulting `newRsETHPrice` drop exceeds `pricePercentageLimit`, the downside-protection logic auto-pauses `LRTDepositPool`, `LRTWithdrawalManager`, and `LRTOracle`: [5](#0-4) 

`updateRSETHPrice()` is publicly callable with no access control: [6](#0-5) 

Any external actor can trigger the auto-pause path at any time the oracle is stale.

## Impact Explanation

**Path 1 — High: Theft of unclaimed yield.** A depositor mints rsETH at a pre-depeg exchange rate for a depegged LST. The excess rsETH represents a direct dilution of all existing rsETH holders' claims on the underlying ETH. The value extracted is proportional to the magnitude of the depeg and the deposit size.

**Path 2 — Medium: Temporary freezing of funds.** A single stale or zeroed oracle for any supported LST causes `_updateRsETHPrice()` to compute an artificially low rsETH price, triggering the auto-pause circuit breaker and freezing all deposits and withdrawals until an admin manually unpauses.

## Likelihood Explanation

Chainlink oracles can become stale during network congestion, oracle node downtime, or when an asset's price moves within the deviation threshold. LST depeg events (stETH, rETH) are historically documented. For Path 1, the window between a real depeg and a Chainlink update is narrow for major LSTs with tight deviation thresholds (0.5%), but is wider for smaller or newer LSTs with looser thresholds. For Path 2, `updateRSETHPrice()` is permissionlessly callable, meaning any external actor can trigger the auto-pause path at any time a feed is stale or returns an anomalous value. Both paths require no special privileges.

## Recommendation

Add staleness, round-completeness, and positivity checks to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

if (answeredInRound < roundId) revert StalePrice();
if (updatedAt == 0) revert IncompleteRound();
if (price <= 0) revert InvalidPrice();
// optionally: if (block.timestamp - updatedAt > HEARTBEAT) revert StalePrice();
```

A per-asset configurable heartbeat constant should be stored alongside `assetPriceFeed` to enforce feed-specific staleness windows.

## Proof of Concept

**Path 1:**
1. Assume `stETH` is a supported LST with a Chainlink feed. The feed's last update was several hours ago, after a depeg event moved the price from 1.0 ETH to 0.8 ETH, but within the feed's heartbeat window so no revert occurs.
2. The Chainlink feed still reports `price = 1e8` (stale, pre-depeg).
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
4. `getRsETHAmountToMint` computes `rsethAmountToMint = (100e18 * 1e18) / rsETHPrice` using the stale 1.0 ETH price instead of the real 0.8 ETH price.
5. Attacker receives ~25% more rsETH than the deposited stETH is worth in ETH terms, diluting all existing holders.

**Path 2:**
1. Configure `pricePercentageLimit` to a non-zero value (e.g., 5%).
2. Arrange for a supported LST's Chainlink feed to return `price = 0` (e.g., deprecated feed, offline node).
3. Any external caller invokes `LRTOracle.updateRSETHPrice()`.
4. `_getTotalEthInProtocol()` returns an underestimated value; `newRsETHPrice` drops below the threshold.
5. The `isPriceDecreaseOffLimit` branch triggers, pausing `LRTDepositPool`, `LRTWithdrawalManager`, and `LRTOracle`, freezing all user deposits and withdrawals until admin intervention.
