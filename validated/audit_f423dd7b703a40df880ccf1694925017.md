Audit Report

## Title
Decimal Precision Oversight in Token-to-rsETH Swap Calculation Causes Massive Under-Minting for Non-18-Decimal Tokens - (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPoolV3WithNativeChainBridge.sol, contracts/pools/RSETHPoolNoWrapper.sol)

## Summary
All four L2 pool variants compute rsETH output via `rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate`, where `tokenToETHRate` is always scaled to 1e18 by the oracle regardless of the deposited token's own decimal count. For any token with fewer than 18 decimals, `amountAfterFee` carries fewer raw units than the formula assumes, causing the minted rsETH/wrsETH to be undercounted by exactly `10^(18 - tokenDecimals)`. The depositor's full token balance is transferred and bridged to L1, but they receive a negligible fraction of the rsETH they are owed.

## Finding Description
`ChainlinkOracleForRSETHPoolCollateral.getRate()` normalizes the Chainlink price to a fixed 1e18 scale:

```solidity
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
```

This returns the ETH value of **one whole token** expressed in 1e18 units, independent of the token's own decimal count.

The swap formula in `RSETHPoolV3.viewSwapRsETHAmountAndFee` (and identically in the other three variants) is:

```solidity
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

For this to be dimensionally correct, `amountAfterFee` must be in 1e18 units. The ETH deposit path confirms the intended convention — it explicitly multiplies by 1e18:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;  // RSETHPoolV3.sol L307
```

For a 6-decimal token (USDC), `amountAfterFee` is in 1e6 units, so the result is `10^12` times smaller than correct. `addSupportedToken` in all variants performs no decimal check:

```solidity
function addSupportedToken(address token, address oracle) external onlyRole(TIMELOCK_ROLE) {
    // only checks: non-zero addresses, not already supported, oracle.getRate() != 0
    supportedTokenOracle[token] = oracle;
```

No decimal normalization of `amountAfterFee` exists anywhere in the pool contracts (confirmed: `decimals` appears only in the oracle file within `contracts/pools/`).

Exploit path:
1. Governance calls `addSupportedToken(USDC, usdcOracle)` — a routine collateral expansion.
2. Alice calls `deposit(USDC, N * 1e6, "ref")`.
3. `N * 1e6` USDC is transferred from Alice to the pool and later bridged to L1 at full value.
4. `viewSwapRsETHAmountAndFee` computes `rsETHAmount = (N * 1e6) * (P * 1e18) / (R * 1e18) = N * P / R * 1e6` instead of the correct `N * P / R * 1e18`.
5. Alice is minted `10^12` times fewer wrsETH than owed, with no recourse.

## Impact Explanation
Direct theft of user funds in motion. The depositor's tokens are fully captured by the protocol (transferred and bridged to L1), while the minted rsETH/wrsETH is negligible. For USDC (6 decimals), a depositor loses 99.9999999999% of their deposited value. This matches the Critical impact class: **Direct theft of any user funds, whether at-rest or in-motion**.

## Likelihood Explanation
The bug is latent under the current collateral set (wstETH, WETH — both 18 decimals). It activates the moment governance adds any sub-18-decimal token via `addSupportedToken`. Adding USDC or USDT as collateral is a natural and common protocol expansion. No attacker action is required beyond depositing the newly-supported token; every depositor of that token is automatically and irreversibly harmed. The governance action is legitimate and non-malicious — the bug is in the code, not in governance intent.

## Recommendation
Normalize `amountAfterFee` to 18 decimals before applying the rate formula in all four pool variants:

```solidity
uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 normalizedAmount = amountAfterFee * 10 ** (18 - tokenDecimals);
rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
```

Apply consistently across `RSETHPoolV3.sol`, `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV3WithNativeChainBridge.sol`, and `RSETHPoolNoWrapper.sol`. Alternatively, enforce at `addSupportedToken` time that `IERC20Metadata(token).decimals() == 18`.

## Proof of Concept
Foundry fork test plan:

1. Fork an L2 where `RSETHPoolV3` is deployed.
2. Impersonate the TIMELOCK_ROLE and call `addSupportedToken(USDC, usdcChainlinkOracle)`.
3. As Alice, approve and call `deposit(USDC, 1_000_000e6, "ref")` (1M USDC, ~$1M value).
4. Assert `wrsETH.balanceOf(alice)` ≈ `3e8` (≈ 0.0000003 wrsETH).
5. Assert correct value would be ≈ `3e20` (≈ 300 wrsETH at 0.0003 ETH/USDC).
6. Assert `IERC20(USDC).balanceOf(pool)` increased by `1_000_000e6` (full transfer occurred).
7. The ratio `correctAmount / actualAmount == 1e12` confirms the `10^(18-6)` precision loss.
