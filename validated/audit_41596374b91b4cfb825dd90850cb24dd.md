Audit Report

## Title
Global Shared Daily Mint Rate Limit Can Be Exhausted by a Single Depositor, Temporarily Blocking All Other Users - (File: contracts/RSETH.sol)

## Summary
`RSETH.sol` enforces a single protocol-wide daily mint cap via `currentPeriodMintedAmount`, a global accumulator with no per-user sub-limit. Any depositor whose single transaction maps to an rsETH amount that fills the remaining daily budget will cause every subsequent `depositETH` or `depositAsset` call to revert with `DailyMintLimitExceeded` for up to 24 hours, temporarily freezing the deposit entry path for all other users.

## Finding Description
The `checkDailyMintLimit` modifier in `contracts/RSETH.sol` (L42–56) resets `currentPeriodMintedAmount` once per 24-hour window and then checks and increments it atomically:

```solidity
if (currentPeriodMintedAmount + amount > maxMintAmountPerDay) {
    revert DailyMintLimitExceeded(currentPeriodMintedAmount + amount, maxMintAmountPerDay);
}
currentPeriodMintedAmount += amount;
```

`currentPeriodMintedAmount` is a single storage slot shared across every caller. The `mint` function (L229–240) applies this modifier unconditionally for every minting call. Both `LRTDepositPool.depositETH` (L76–93) and `depositAsset` (L99–118) call the private `_mintRsETH` (L686–690), which calls `RSETH.mint`. There is no per-depositor quota, no maximum single-transaction cap, and no mechanism to reserve capacity for other users. Once `currentPeriodMintedAmount` reaches `maxMintAmountPerDay`, every subsequent deposit reverts until the 24-hour window rolls over.

The same structural pattern exists in `contracts/pools/RSETHPoolV3.sol` (L96–125) via its `limitDailyMint` modifier, which maintains its own `dailyMintAmount` / `dailyMintLimit` pair with identical shared-counter semantics.

## Impact Explanation
The deposit path (`depositETH` / `depositAsset`) is the sole mechanism for external users to obtain rsETH from the L1 protocol. Exhausting the daily mint cap blocks all deposits for up to 24 hours, constituting a **temporary freezing of funds** (Medium severity): users holding ETH or LSTs they intend to deposit are locked out of the protocol for the duration of the blackout window. This matches the allowed impact: **Medium — Temporary freezing of funds**.

## Likelihood Explanation
`maxMintAmountPerDay` is a finite value set by the LRT manager. Any depositor whose intended deposit maps to an rsETH amount ≥ `maxMintAmountPerDay - currentPeriodMintedAmount` triggers the condition. Large institutional depositors or protocols routinely move hundreds or thousands of ETH in a single transaction. A deliberate attacker can also front-run the daily reset (observable on-chain via `periodStartTime`) to immediately exhaust the new period's budget at the start of each window, making the attack repeatable every 24 hours at no net cost (the attacker receives rsETH in return for their deposit).

## Recommendation
- Replace the single global accumulator with a per-depositor sub-limit, or
- Enforce a maximum single-transaction deposit cap so no single call can consume the entire daily budget, or
- Remove the rolling time-window cap from `RSETH.mint` and rely solely on the cumulative per-asset `depositLimitByAsset` ceiling in `LRTConfig`, which does not create a time-based blackout window.

## Proof of Concept
1. `maxMintAmountPerDay` is set to `X` rsETH (e.g., 1000 rsETH).
2. Attacker calls `LRTDepositPool.depositETH{value: V}(0, "")` where `V` is large enough that `getRsETHAmountToMint(ETH_TOKEN, V) >= X`.
3. `_mintRsETH(X)` → `RSETH.mint(attacker, X)` → `checkDailyMintLimit(X)`: `currentPeriodMintedAmount` becomes `X = maxMintAmountPerDay`.
4. Any subsequent call to `depositETH` or `depositAsset` by any user within the same 24-hour window evaluates `currentPeriodMintedAmount + amount > maxMintAmountPerDay` as true for any `amount > 0` and reverts with `DailyMintLimitExceeded`.
5. All users are locked out of deposits for up to 24 hours. The attacker repeats at the next period reset.

**Foundry test plan:** Deploy `RSETH` + `LRTDepositPool` on a local fork; call `setMaxMintAmountPerDay(1000e18)`; have address A call `depositETH` with value sufficient to mint 1000e18 rsETH; assert success; have address B call `depositETH` with any nonzero value; assert revert with `DailyMintLimitExceeded`; `vm.warp(block.timestamp + 1 days + 1)`; assert address B's deposit now succeeds.
