### Title
Forged Foreign-Chain Verification via Attacker-Controlled `payload_hash` in `respond_verify_foreign_tx` — (File: crates/contract/src/lib.rs)

---

### Summary

In `respond_verify_foreign_tx`, the contract verifies the MPC signature against `response.payload_hash`, a value supplied entirely by the responding participant. The contract never checks that this hash corresponds to the pending request's expected payload. A single Byzantine attested participant can replay any previously observed valid `(payload_hash, signature)` pair from on-chain history to resolve an unrelated pending `verify_foreign_transaction` request, causing the contract to deliver a forged verification outcome to the caller.

---

### Finding Description

**Root cause — asymmetry between `respond` and `respond_verify_foreign_tx`**

In `respond` (the regular sign path), the payload that is verified is taken from the *stored request*, not from the response:

```rust
// lib.rs ~L600 — payload comes from the stored request (non-attacker-controlled)
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
).is_ok()
```

In `respond_verify_foreign_tx`, the payload hash is taken from the *response* submitted by the participant:

```rust
// lib.rs ~L726-L734 — payload_hash comes from the response (attacker-controlled)
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
).is_ok()
```

The contract performs no binding check between `response.payload_hash` and the `VerifyForeignTransactionRequest` that is being resolved. The pending-request map is keyed on the request, but the response payload is accepted verbatim.

**Why a replay is possible**

Every successful `respond_verify_foreign_tx` call is a public NEAR transaction. Its `response.payload_hash` and `response.signature` are permanently visible on-chain. Because the contract only checks `verify_ecdsa_signature(sig, payload_hash, root_key)` and not `payload_hash == expected_hash(request)`, any attested participant can:

1. Extract `(H, σ)` from any past successful `respond_verify_foreign_tx` receipt.
2. Call `respond_verify_foreign_tx(request = <victim's pending request>, response = {payload_hash: H, signature: σ})`.
3. The contract accepts the call (σ is valid for H under the root key; the pending request exists).
4. `resolve_yields_for` drains all yields queued under the victim's request, delivering `{payload_hash: H, signature: σ}` to every waiting caller.

The victim's callers receive a response whose `payload_hash` corresponds to a *different* foreign transaction, not the one they requested.

---

### Impact Explanation

Any smart contract that calls `verify_foreign_transaction` and then acts on the returned `VerifyForeignTransactionResponse` without independently recomputing the expected `payload_hash` from the extracted values will accept the forged response as proof that the requested transaction occurred. This enables:

- **Invalid bridge execution**: a bridge contract that gates fund release on a foreign-chain deposit confirmation can be tricked into releasing funds for a deposit that was never made (or was already used).
- **Double-spend**: the same `(H, σ)` pair can be replayed against multiple independent pending requests for different transactions, each of which resolves successfully.

The `near-mpc-sdk` `ForeignChainSignatureVerifier::verify_signature` does check `payload_hash` against the expected payload, but this is an off-chain SDK helper — the on-chain contract does not enforce it, and callers that do not use the SDK are fully exposed.

---

### Likelihood Explanation

- Requires exactly **one** Byzantine attested participant — strictly below the signing threshold.
- No cryptographic capability is needed beyond reading on-chain transaction history.
- The attacker does not need to participate in any MPC round; they only need to submit a single contract call.
- Every successful `respond_verify_foreign_tx` call permanently supplies a reusable `(H, σ)` pair.
- The attack window is open for as long as any `verify_foreign_transaction` request remains pending (up to ~200 blocks per yield).

---

### Recommendation

Bind the verification to the request by computing the expected payload hash inside the contract. Because the contract does not know the `ExtractedValues` at request time, the simplest fix is to include the expected `payload_hash` in the pending-request map entry at submission time — populated by the first honest `respond_verify_foreign_tx` call — and reject any subsequent call that provides a different hash. Alternatively, require the responding participant to include the full `ForeignTxSignPayload` (not just its hash) so the contract can recompute and compare the hash itself.

At minimum, the contract should reject a `respond_verify_foreign_tx` call whose `response.payload_hash` does not match the hash already stored for that request (if one exists), preventing replay of stale signatures across requests.

---

### Proof of Concept

```
1. Alice calls verify_foreign_transaction(T1_request)
   → pending_verify_foreign_tx_requests[T1_request] = [yield_alice]

2. Honest nodes verify T2 (a different transaction) and call:
   respond_verify_foreign_tx(T2_request, {payload_hash: H_T2, signature: σ_T2})
   → This is now visible on-chain.

3. Byzantine attested participant Bob calls:
   respond_verify_foreign_tx(
       request  = T1_request,          // Alice's pending request
       response = {payload_hash: H_T2, // replayed from step 2
                   signature:    σ_T2}
   )

4. Contract checks:
   verify_ecdsa_signature(σ_T2, H_T2, root_key) → OK   ✓
   pending_verify_foreign_tx_requests[T1_request] exists → OK  ✓
   → resolve_yields_for drains yield_alice with {H_T2, σ_T2}

5. Alice's contract receives {payload_hash: H_T2, signature: σ_T2}
   — a valid MPC signature, but over T2's payload, not T1's.
   If Alice's contract gates a fund release on this response without
   recomputing the expected hash, it releases funds for T1 based on
   T2's verification evidence.
```

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
