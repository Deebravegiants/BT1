### Title
Receipt Size Limit Bypass via `promise_return` Post-Validation Mutation - (File: runtime/runtime/src/lib.rs)

### Summary
An unprivileged user can craft a transaction that causes the runtime to produce and propagate a receipt exceeding `max_receipt_size` (4 MiB). The root cause is that `output_data_receivers` are appended to a `promise_return`-ed receipt **after** the receipt size validation check, bypassing the `max_receipt_size` invariant. This is the nearcore analog of the Uniswap gas-griefing bug: in both cases, a bounded-size check is performed on data that is subsequently enlarged by an implicit, unguarded append.

### Finding Description

The NEAR runtime enforces a `max_receipt_size` limit (4 MiB) on newly created receipts. The check is performed inside `apply_action_receipt` in `runtime/runtime/src/lib.rs`:

```rust
if new_result.result.is_ok() {
    if let Err(e) = new_result.new_receipts.iter().try_for_each(|receipt| {
        validate_receipt(
            &apply_state.config.wasm_config.limit_config,
            receipt,
            apply_state.current_protocol_version,
            ValidateReceiptMode::NewReceipt,
        )
    }) {
        new_result.result = Err(ActionErrorKind::NewReceiptValidationError(e).into());
    }
}
``` [1](#0-0) 

However, **after** this validation, when the function call result is `ReturnData::ReceiptIndex` (i.e., the contract called `promise_return`), the runtime mutates the validated receipt by appending the parent receipt's `output_data_receivers` to it:

```rust
if !action_receipt.output_data_receivers().is_empty() {
    if let Ok(ReturnData::ReceiptIndex(receipt_index)) = result.result {
        match result.new_receipts.get_mut(receipt_index as usize)... {
            ReceiptEnum::ActionV2(new_action_receipt) | ... => new_action_receipt
                .output_data_receivers
                .extend_from_slice(&action_receipt.output_data_receivers()),
        }
    }
``` [2](#0-1) 

This post-validation mutation is never re-checked against `max_receipt_size`. A contract that crafts receipt C at exactly `max_receipt_size` and then calls `promise_return(C)` inside a promise DAG `A → B` will cause the runtime to append B's `output_data_receivers` to C, producing a receipt that exceeds the limit. The codebase explicitly acknowledges this bug:

> "There is a bug which allows to create receipts that are above the size limit. Runtime has to handle them gracefully until the receipt size limit bug is fixed. See https://github.com/near/nearcore/issues/12606 for details." [3](#0-2) 

The test `test_max_receipt_size_promise_return` confirms the oversized receipt is actually produced and propagated across shards: [4](#0-3) 

A parallel path exists for `value_return`: a contract returning a value of exactly `max_receipt_size` bytes causes the runtime to wrap it in a `DataReceipt` whose total borsh-serialized size exceeds `max_receipt_size`, also without re-validation: [5](#0-4) 

### Impact Explanation

The `max_receipt_size` limit is a hard protocol invariant introduced to keep `ChunkStateWitness` under ~17 MiB, ensuring chunk validators can validate chunks within time and bandwidth constraints. Violating it:

1. **Corrupted protocol value**: The `max_receipt_size` invariant in the state witness is violated. Receipts larger than 4 MiB are included in `source_receipt_proofs` and propagated to receiving shards.
2. **State witness size guarantee broken**: The state witness size budget (documented as targeting ≤17 MiB) is exceeded, potentially causing chunk validators to reject chunks or experience degraded validation performance.
3. **Cross-shard propagation**: The oversized receipt is sent to the receiving shard as an incoming receipt, confirmed by `assert_oversized_receipt_occurred` traversing the chain and finding the receipt. [6](#0-5) 

### Likelihood Explanation

Any unprivileged user with a deployed contract can trigger this. The attack requires:
1. Deploy a contract (standard operation).
2. Call a method that creates a two-promise DAG `A → B` where A does `promise_return(C)` and C's args are sized to `max_receipt_size - base_receipt_size`.
3. The test contract `max_receipt_size_promise_return_method1` and `max_receipt_size_promise_return_method2` in the test suite demonstrate the exact pattern. [7](#0-6) 

No special privileges are required. The cost is a standard function call transaction with sufficient gas.

### Recommendation

After the `output_data_receivers` are appended to the `promise_return`-ed receipt (lines 1029–1035 of `runtime/runtime/src/lib.rs`), re-validate the mutated receipt's size against `max_receipt_size`. Alternatively, account for the parent's `output_data_receivers` size when validating the child receipt before the `promise_return` path is taken. The same fix should be applied to the `value_return` path where the data receipt wrapping can also exceed `max_receipt_size`.

### Proof of Concept

The nearcore test suite itself provides the proof of concept:

```
test_max_receipt_size_promise_return (test-loop-tests/src/tests/max_receipt_size.rs:130)
test_max_receipt_size_value_return   (test-loop-tests/src/tests/max_receipt_size.rs:216)
```

Both tests confirm that oversized receipts are produced and propagated on-chain. The `assert_oversized_receipt_occurred` helper verifies the oversized receipt appears in the chain's incoming receipt proofs.

**Step-by-step trigger**:
1. Deploy a contract with a method that creates promise DAG `A → B` where A calls `promise_return(C)` and C has args of size `max_receipt_size - base_receipt_size`.
2. Submit the transaction. Receipt C passes `validate_receipt` at its creation size.
3. The runtime appends B's `output_data_receivers` to C post-validation.
4. C's borsh-serialized size now exceeds `max_receipt_size`.
5. C is sent as an outgoing receipt to the target shard, violating the state witness size invariant. [8](#0-7) [9](#0-8)

    mode: ValidateReceiptMode,

            return Err(ReceiptValidationError::ReceiptSizeExceeded {
                size: receipt_size,
                limit: limit_config.max_receipt_size,

```markdown
## State witness size limits

Some limits were introduced to keep the size of `ChunkStateWitness` reasonable.
`ChunkStateWitness` contains all the incoming transactions and receipts that will be processed during chunk application and in theory a single receipt could be tens of megabytes in size. Distributing a `ChunkStateWitness` this large would be troublesome, so we limit the size and number of transactions, receipts, etc. The limits aim to keep the total uncompressed size of `ChunkStateWitness` under 17MiB.

There are two types of size limits:

* Hard limit - the size must be below this limit, anything else is considered invalid
* Soft limit - things are added until the limit is exceeded, after that things stop being added. The last added thing is allowed to slightly exceed the limit.

The limits are:

* `max_transaction_size = 1.5 MiB`
  * All transactions must be below 1.5 MiB, otherwise they'll be considered invalid and rejected.
  * Previously was 4MiB, now reduced to 1.5MiB
* `max_receipt_size - 4 MiB`:
  * All receipts must be below 4 MiB, otherwise they'll be considered invalid and rejected.
  * Previously there was no limit on receipt size. Set to 4MiB, might be reduced to 1.5MiB in the future to match the transaction limit.
* `combined_transactions_size_limit - 4 MiB`
  * Hard limit on total size of transactions from this and previous chunk. `ChunkStateWitness` contains transactions from two chunks, this limit applies to the sum of their sizes.
* `new_transactions_validation_state_size_soft_limit - 500 KiB`
  * Validating new transactions generates storage proof (recorded trie nodes), which has to be limited. Once transaction validation generates more storage proof than this limit, the chunk producer stops adding new transactions to the chunk.
* `per_receipt_storage_proof_size_limit - 4 MB`
  * Executing a receipt generates storage proof. A single receipt is allowed to generate at most 4MB of storage proof. This is a hard limit, receipts which generate more than that will fail.
* `main_storage_proof_size_soft_limit - 4 MB`
  * This is a limit on the total size of storage proof generated by receipts in one chunk. Once receipts generate more storage proof than this limit, the chunk producer stops processing receipts and moves the rest to the delayed queue.
  * It's a soft limit, which means that the total size of storage proof could reach 8 MB (3.99MB + one receipt which generates 4MB of storage proof)
  * Due to implementation details it's hard to find the exact amount of storage proof generated by a receipt, so an upper bound estimation is used instead. This upper bound assumes that every removal generates additional 2000 bytes of storage proof, so receipts which perform a lot of trie removals might be limited more than theoretically applicable.
* `outgoing_receipts_usual_size_limit - 100 KiB`
  * Limit on the size of outgoing receipts to another shard. Needed to keep the size of `source_receipt_proofs` small.
  * On most block heights a shard isn't allowed to send receipts larger than 100 KiB to another shard.
* `outgoing_receipts_big_size_limit - 4.5 MiB`
  * On every block height there's one special "allowed shard" which is allowed to send larger receipts, up to 4.5 MiB in total.
  * A receiving shard will receive receipts from `num_shards - 1` shards using the usual limit and one shard using the big limit.

In total that gives 4 MiB + 500 KiB + 8MB + 5*100 KiB + 4.5 MiB ~= 17 MiB of maximum witness size. Possibly a little more on missing chunks.
