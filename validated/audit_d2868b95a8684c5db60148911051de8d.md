Looking at the exact validation logic in `set_state_header` and the Merkle proof structure to determine whether the pigeonhole argument actually holds.

### Title
Pigeonhole Receipt Proof Check in `set_state_header` Does Not Bind `from_shard_id` to Merkle Leaf Position, Allowing Forged State Sync Headers - (`chain/chain/src/state_sync/adapter.rs`)

### Summary

The uniqueness check on `from_shard_id` in `set_state_header` is not sufficient to enforce that all shards' receipts are present. Because `verify_path` does not check the leaf's position in the Merkle tree, and because `from_shard_id` is never cryptographically bound to the Merkle proof, a malicious peer can supply N distinct `from_shard_id` labels while reusing the same shard's Merkle proof for all N entries. All validation checks pass, and the syncing node applies the chunk with wrong receipts, producing a divergent state root.

### Finding Description

The comment at `adapter.rs:470-474` states:

> "According to Pigeonhole principle, it's enough to ensure all receipt_proofs are distinct to prove that all receipts were received and no receipts were hidden."

This reasoning is **incorrect**. The pigeonhole argument would be valid only if each distinct `from_shard_id` value were cryptographically bound to a distinct leaf position in the block-level Merkle tree. It is not.

**The two-level proof structure:**

Step 4e verifies: `verify_path(*root, proof, &receipts_hash)` where `root` comes from the attacker-controlled `root_proofs[i][j]`. [1](#0-0) 

Step 4f verifies: `verify_path(*block_header.prev_chunk_outgoing_receipts_root(), block_proof, root)` — proving `root` is *somewhere* in the block-level Merkle tree. [2](#0-1) 

**The `verify_path` function** only checks that following the path from the leaf hash reaches the root. It does not check the leaf's index/position: [3](#0-2) 

**`from_shard_id` is only used for the HashSet uniqueness check** (step 4d) and is never used in either Merkle proof call: [4](#0-3) 

**Concrete attack (2 shards, `chunks_included = 2`):**

The block-level Merkle tree has two leaves:
- Position 0: `root_0` = shard 0's `prev_outgoing_receipts_root`
- Position 1: `root_1` = shard 1's `prev_outgoing_receipts_root`

The attacker provides two `ReceiptProof` entries and two `RootProof` entries:

| Entry | `from_shard_id` | receipts | `proof` | `RootProof` |
|-------|----------------|----------|---------|-------------|
| 0 | 0 | shard 0's real receipts | valid path in shard 0's tree | `(root_0, block_proof_0)` |
| 1 | **1** | shard 0's real receipts (duplicated) | same valid path | `(root_0, block_proof_0)` (same) |

Check results:
- **4c** (count): 2 == `chunks_included(2)` ✓
- **4d** (uniqueness): `{0, 1}` are distinct ✓
- **4e entry 0**: `verify_path(root_0, proof_0, hash(ReceiptList(target, receipts_shard_0)))` → ✓
- **4f entry 0**: `verify_path(block_root, block_proof_0, root_0)` → ✓ (`root_0` is a valid leaf)
- **4e entry 1**: same hash, same proof → ✓
- **4f entry 1**: same `root_0`, same `block_proof_0` → ✓ (`root_0` is still a valid leaf at position 0)

All checks pass. Shard 1's receipts are completely absent; shard 0's receipts are applied twice.

**In `set_state_finalize`**, the stored header's receipts are collected without re-validation and passed directly to `apply_chunk`: [5](#0-4) 

The chunk is applied with the wrong receipt set, producing a state root that diverges from the honest network.

### Impact Explanation

The syncing node computes and stores a wrong state root for the synced shard. All subsequent chunk production and validation by this node will be based on the wrong state, causing it to be permanently excluded from consensus. Cross-shard receipts from shard 1 are silently dropped, breaking application-level invariants (e.g., token transfers, contract calls) for any user whose receipts originated from shard 1.

### Likelihood Explanation

Any peer the syncing node contacts for state sync data can execute this attack. State sync is a standard, unauthenticated peer-to-peer protocol path. No validator or admin privileges are required. The attacker only needs to serve a crafted `ShardStateSyncResponseHeader` in response to a legitimate state sync request. All required Merkle proof values are publicly observable on-chain.

### Recommendation

The fix must cryptographically bind each `from_shard_id` to its position in the block-level Merkle tree. Two approaches:

1. **Use `verify_path_with_index`** (which already exists in the codebase) for step 4f, passing the shard index derived from `from_shard_id`. This ensures the `block_proof` path encodes the correct leaf position. [6](#0-5) 

2. **Explicitly look up `root` from the block's chunk headers** using `from_shard_id` (as done in `compute_state_response_header`) and compare it against the attacker-supplied `root`, rather than trusting the attacker-supplied `root` directly. [7](#0-6) 

### Proof of Concept

```rust
// In a test environment with 2 shards:
// 1. Obtain a valid ShardStateSyncResponseHeader from an honest node.
// 2. Extract the first block's receipt_proofs and root_proofs.
// 3. Duplicate entry 0 into entry 1, changing only from_shard_id to 1.
// 4. Call set_state_header with the crafted header.
// 5. Assert it returns Ok(()) — demonstrating the forged header is accepted.
// 6. Call set_state_finalize and observe the resulting state root differs
//    from the honest node's state root for the same shard.

let mut crafted_header = honest_header.clone();
// Duplicate shard 0's receipt proof under from_shard_id=1
let mut receipt_proofs = crafted_header.incoming_receipts_proofs[0].1.clone();
let mut forged_entry = receipt_proofs[0].clone();
forged_entry.1.from_shard_id = ShardId::new(1); // distinct from_shard_id
receipt_proofs.push(forged_entry);
// Also duplicate the root_proof
let root_proof_dup = crafted_header.root_proofs[0][0].clone();
crafted_header.root_proofs[0].push(root_proof_dup);
// set_state_header will accept this
assert!(chain.state_sync_adapter.set_state_header(shard_id, sync_hash, crafted_header).is_ok());
```

/// Verify merkle path for given item and corresponding path.
pub fn verify_path<T: BorshSerialize>(root: MerkleHash, path: &MerklePath, item: T) -> bool {
    verify_hash(root, path, CryptoHash::hash_borsh(item))
}

pub fn verify_hash(root: MerkleHash, path: &MerklePath, item_hash: MerkleHash) -> bool {
    compute_root_from_path(path, item_hash) == root
}

pub fn verify_path_with_index<T: BorshSerialize>(
    root: MerkleHash,
    path: &MerklePath,
    item: T,
    part_idx: u64,
    num_merklized_parts: u64,
) -> bool {
    verify_path_matches_index(path, part_idx, num_merklized_parts) && verify_path(root, path, item)
}

pub fn compute_root_from_path(path: &MerklePath, item_hash: MerkleHash) -> MerkleHash {
    let mut res = item_hash;
    for item in path {
        match item.direction {
            Direction::Left => {
                res = combine_hash(&item.hash, &res);
            }
            Direction::Right => {
                res = combine_hash(&res, &item.hash);
            }
        }
    }
    res
}
