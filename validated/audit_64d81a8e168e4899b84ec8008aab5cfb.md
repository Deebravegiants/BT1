### Title
Harvester `farmer_reward_address_override` Accepted Without Enforcement — (`chia/farmer/farmer_api.py`)

### Summary

In Chia's CHIP-22 third-party harvester fee mechanism, a connected harvester can set `farmer_reward_address_override` to any arbitrary puzzle hash in its `RespondSignatures` message. The farmer unconditionally substitutes this value as the `farmer_puzzle_hash` in `DeclareProofOfSpace`, which the full node then embeds into the block's foliage as the farmer reward recipient. The fee-quality check that is supposed to gate this override is advisory-only (log-only) and never blocks the substitution. Additionally, the override field in `RespondSignatures` is never cross-validated against the override field in the earlier `NewProofOfSpace` message, so a harvester can bypass even the advisory check entirely.

### Finding Description

**Step 1 — The override field is accepted unconditionally in `_process_respond_signatures`:**

In `chia/farmer/farmer_api.py`, lines 916–919, the farmer replaces its own configured `farmer_target` with whatever the harvester supplies in `RespondSignatures.farmer_reward_address_override`:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
    include_source_signature_data = True
```

No allowlist, no comparison to the farmer's own address, no rejection path. [1](#0-0) 

**Step 2 — The fee-quality check in `new_proof_of_space` is log-only:**

When `NewProofOfSpace.farmer_reward_address_override` is set, `notify_farmer_reward_taken_by_harvester_as_fee` is called. If the fee quality fails (i.e., the harvester is not entitled to take the fee), the function only emits a `log.warning`. It does not return an error, does not set a flag, and does not prevent the block from being built with the overridden address:

```python
else:
    self.log.warning(
        "Harvester illegitimately took a fee reward that "
        + "did not belong to it or it incorrectly applied the fee convention."
    )
``` [2](#0-1) 

**Step 3 — The two override fields are never cross-validated:**

The fee-quality check in `new_proof_of_space` reads `new_proof_of_space.farmer_reward_address_override` (from `NewProofOfSpace`). The address actually used in the block is read from `response.farmer_reward_address_override` (from `RespondSignatures`). These are two separate protocol messages. A malicious harvester can set `farmer_reward_address_override = None` in `NewProofOfSpace` (skipping the fee-quality check entirely) and then set `farmer_reward_address_override = <attacker_address>` in `RespondSignatures` (diverting the reward). [3](#0-2) 

**Step 4 — The full node accepts any `farmer_puzzle_hash` without validation:**

In `chia/full_node/full_node_api.py`, `declare_proof_of_space` blindly uses `request.farmer_puzzle_hash` (which is the overridden address) to build the unfinished block:

```python
farmer_ph = request.farmer_puzzle_hash
```

There is no check that this matches the farmer's registered address or any allowlist. [4](#0-3) 

This `farmer_ph` is passed directly to `create_unfinished_block` → `create_foliage`, where it is embedded in `FoliageBlockData.farmer_reward_puzzle_hash`. [5](#0-4) 

### Impact Explanation

Every block a farmer wins carries a farmer reward coin (1.75 XCH at current subsidy). A malicious third-party harvester can redirect this coin to any puzzle hash it controls. The farmer loses 1.75 XCH per farmed block with no on-chain recourse. This is unauthorized reward diversion affecting XCH — Critical under the allowed impact scope.

### Likelihood Explanation

CHIP-22 explicitly supports third-party harvesters (e.g., DrPlotter). Farmers are expected to connect to external harvester services they do not fully control. Any such service that turns malicious, or any service that a farmer is socially-engineered into connecting to, can exploit this path. The attacker needs only a valid harvester connection — no key compromise, no cryptographic break, no admin access.

### Recommendation

1. **Enforce the fee-quality check**: When `fee_quality > fee_threshold` (or when `fee_info` is absent), the farmer should reject the override and use its own `farmer_target` rather than logging a warning and proceeding.
2. **Cross-validate the two override fields**: The `farmer_reward_address_override` in `RespondSignatures` must match the one in the corresponding `NewProofOfSpace` (or be `None` if the earlier message had `None`).
3. **Optionally, add a farmer-side allowlist**: The farmer could maintain a configurable set of puzzle hashes that harvesters are permitted to redirect rewards to, rejecting any override not in that set.

### Proof of Concept

1. Farmer connects to a malicious third-party harvester.
2. Harvester receives a signage point and finds a valid proof of space.
3. Harvester sends `NewProofOfSpace` to the farmer with `farmer_reward_address_override = None` and `fee_info = None` — the fee-quality check in `new_proof_of_space` is skipped entirely. [3](#0-2) 
4. Farmer sends `RequestSignatures` to the harvester.
5. Harvester responds with `RespondSignatures` where `farmer_reward_address_override = <attacker_puzzle_hash>`.
6. Farmer's `_process_respond_signatures` unconditionally sets `farmer_reward_address = <attacker_puzzle_hash>` and emits `DeclareProofOfSpace` with `farmer_puzzle_hash = <attacker_puzzle_hash>`. [6](#0-5) 
7. Full node builds the unfinished block with `farmer_ph = <attacker_puzzle_hash>`, embeds it in `FoliageBlockData`, and the block is finalized. [7](#0-6) 
8. The farmer reward coin (1.75 XCH) is created at `<attacker_puzzle_hash>`. The farmer receives nothing.
