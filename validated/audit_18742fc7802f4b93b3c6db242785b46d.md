### Title
Missing RSA Key Size Validation in `parse_ek_blob` Allows PVE Encryption Under Attacker-Supplied Sub-2048-bit RSA Key - (`src/cbmpc/api/pve_internal.h`)

### Summary

`parse_ek_blob` only validates the blob version, not the RSA key size. An attacker who supplies a serialized `base_pke_ek_blob_v1_t` with `key_type=rsa_oaep_2048` but with an `rsa_ek` field containing a 1024-bit RSA public key will cause the library to encrypt under that weak key, producing a structurally valid but cryptographically weak ciphertext that is accepted without error.

### Finding Description

**Step 1 — Entry point: `parse_ek_blob`** [1](#0-0) 

`parse_ek_blob` deserializes the blob and checks only `version == base_pke_key_blob_version_v1`. There is no check that `blob.rsa_ek.size() * 8 == RSA_KEY_LENGTH` (2048).

**Step 2 — `base_pke_ek_blob_v1_t::convert` dispatches on `key_type` without size validation** [2](#0-1) 

When `key_type == rsa_oaep_2048`, the converter calls `c.convert(rsa_ek)` and returns. The name `rsa_oaep_2048` implies a 2048-bit key, but the actual modulus size is never checked.

**Step 3 — `unified_key_blob_base_pke_t::encrypt` passes the unchecked key directly** [3](#0-2) 

After `parse_ek_blob` returns `SUCCESS`, the code immediately calls `pve_base_pke_rsa().encrypt(pve_keyref(blob.rsa_ek), ...)` with no intervening size check.

**Step 4 — `kem_policy_rsa_oaep_t::encapsulate` uses `size() * 8` from the actual key** [4](#0-3) 

`encapsulate` calls `pub_key.encrypt_oaep_with_seed(...)`, which internally calls `pad_oaep_with_seed(size() * 8, ...)`. [5](#0-4) 

`size() * 8` is derived from the actual modulus of the supplied key. For a 1024-bit key, this is 1024, not 2048. The OAEP padding succeeds (1024-bit key with SHA-256: 128 − 2×32 − 2 = 62 bytes of plaintext capacity, which comfortably holds the 32-byte KEM shared secret), and `encrypt_raw` produces a 128-byte ciphertext. No error is returned.

**The constant `RSA_KEY_LENGTH = 2048` is defined but never enforced on the deserialization path:** [6](#0-5) 

### Impact Explanation

An attacker who controls the `ek` blob passed to any public PVE encrypt API (`cbmpc_pve_encrypt`, `coinbase::api::pve::encrypt`, batch variants, AC variants) can substitute a 1024-bit RSA public key for the expected 2048-bit key. The library encrypts the KEM shared secret under the weak key, produces a structurally valid ciphertext, and returns `SUCCESS`. The honest decryptor (holding the matching 1024-bit private key) can decrypt it. An adversary who can factor the 1024-bit modulus (feasible with current hardware) recovers the KEM shared secret and therefore the plaintext. This satisfies the **High** impact criterion: "Public API reachable validation bypass in PVE creates accepted invalid cryptographic output."

### Likelihood Explanation

Any caller who accepts an `ek` blob from an external source (network peer, file, user input) is directly exploitable. The attacker only needs to craft a valid DER-encoded 1024-bit RSA public key and embed it in a correctly-versioned blob. No threshold collusion or privileged access is required.

### Recommendation

Add an explicit key-size check immediately after deserialization in `parse_ek_blob` (or inside `base_pke_ek_blob_v1_t::convert` for the `rsa_oaep_2048` branch):

```cpp
// In parse_ek_blob, after coinbase::convert(out, ek):
if (static_cast<base_pke_key_type_v1>(out.key_type) == base_pke_key_type_v1::rsa_oaep_2048) {
  if (out.rsa_ek.size() * 8 != RSA_KEY_LENGTH)
    return coinbase::error(E_BADARG, "RSA base PKE key must be 2048 bits");
}
```

The same guard should be applied in `rsa_oaep_hsm_base_pke_t::encrypt` and any other path that extracts `blob.rsa_ek` from a deserialized blob.

### Proof of Concept

1. Generate a 1024-bit RSA key pair.
2. Serialize it into a `base_pke_ek_blob_v1_t` with `version=1`, `key_type=1` (`rsa_oaep_2048`), and `rsa_ek` = the 1024-bit public key.
3. Call `cbmpc_pve_encrypt(nullptr, curve, crafted_ek_blob, label, x, &ct)`.
4. Observe that the call returns `CBMPC_SUCCESS` and `ct` is non-empty (128-byte RSA-OAEP ciphertext instead of 256 bytes).
5. Assert that the call should have returned `E_BADARG` — it does not, confirming the bypass.
