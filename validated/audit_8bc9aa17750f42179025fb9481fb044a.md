### Title
CHIP-22 Fee Quality Gate Not Enforced — Farmer Reward Unconditionally Diverted by Any Connected Harvester — (`chia/farmer/farmer_api.py`, `chia/farmer/farmer.py`)

### Summary

A connected harvester can redirect the farmer's block reward to an arbitrary attacker-controlled address by sending `NewProofOfSpace` with `farmer_reward_address_override` set and `fee_info=None`. The farmer logs a warning but does not reject the proof or abort processing. It then unconditionally uses the override address from the harvester's `RespondSignatures` in `DeclareProofOfSpace`, which is broadcast to the full node.

### Finding Description

**Step 1 — Entry point: `new_proof_of_space`**

When the farmer receives `NewProofOfSpace` with `farmer_reward_address_override is not None`, it calls `notify_farmer_reward_taken_by_harvester_as_fee` and then unconditionally continues to request signatures: [1](#0-0) 

**Step 2 — No enforcement in `notify_farmer_reward_taken_by_harvester_as_fee`**

When `fee_info is None`, the function logs a warning but does **not** return early, raise an exception, or signal rejection in any way. Execution falls through and the caller continues normally: [2](#0-1) 

The same is true when `fee_info` is present but `fee_quality > fee_threshold` — only a warning is logged: [3](#0-2) 

**Step 3 — `RespondSignatures` carries the override**

`RespondSignatures` has a `farmer_reward_address_override` field that the harvester fully controls: [4](#0-3) 

**Step 4 — `_process_respond_signatures` accepts the override unconditionally**

No fee quality check is performed here. The override address is used directly in `DeclareProofOfSpace`: [5](#0-4) 

**Step 5 — `DeclareProofOfSpace` is broadcast to the full node** [6](#0-5) 

### Impact Explanation

Any harvester with an established connection to the farmer can redirect 100% of the farmer's block reward to an attacker-controlled address on every block won, with no fee quality requirement whatsoever. CHIP-22 is explicitly designed to allow third-party harvesters to take a *proportional* fee based on proof quality — the farmer is supposed to enforce this gate and reject overrides that don't meet the threshold. The enforcement is entirely absent; only logging occurs.

This is a **Critical** impact: unconditional farmer reward diversion affecting XCH.

### Likelihood Explanation

The attacker must be a connected harvester (SSL-authenticated connection). CHIP-22 explicitly contemplates third-party harvester services as the threat model — farmers routinely connect to third-party harvesters they do not fully control. The attack requires no leaked keys, no cryptographic break, and no external compromise beyond being a connected harvester. It is trivially reproducible.

### Recommendation

In `new_proof_of_space`, after calling `notify_farmer_reward_taken_by_harvester_as_fee`, the farmer must check whether the fee quality gate passed. If `fee_info is None` or `fee_quality > fee_threshold`, the farmer should **not** proceed with the override — it should either abort the block-making flow for this proof or substitute its own `farmer_target` address and ignore the override. The warning-only behavior in `notify_farmer_reward_taken_by_harvester_as_fee` must be replaced with a boolean return value (or exception) that the caller acts on.

Additionally, `_process_respond_signatures` should independently verify that any `farmer_reward_address_override` in `RespondSignatures` was pre-authorized (i.e., the corresponding `NewProofOfSpace` passed the fee quality gate), rather than accepting it unconditionally.
