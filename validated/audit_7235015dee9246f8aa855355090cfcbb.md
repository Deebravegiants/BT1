### Title
Unreturned Deposit in `submit_participant_info` for Existing-Participant Re-attestation - (File: crates/contract/src/lib.rs)

### Summary

`submit_participant_info` is `#[payable]` and contains deposit-refund logic only inside a conditional block. When an existing participant re-submits their attestation (the `UpdatedExistingParticipant` path), the entire deposit-handling block is skipped, so any NEAR tokens attached to the call are silently absorbed by the contract and never returned to the caller.

### Finding Description

`submit_participant_info` computes a boolean gate to decide whether the caller must pay for storage:

```rust
let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... check deposit >= cost, then refund excess
}
``` [1](#0-0) 

When the condition is `false` — i.e., the caller is already a participant **and** `add_participant` returns `UpdatedExistingParticipant` — the block is never entered. There is no `else` branch and no unconditional refund. Any NEAR tokens attached to the call remain in the contract balance permanently.

The `UpdatedExistingParticipant` variant is a first-class, expected return value of `add_participant`:

```rust
pub(crate) enum ParticipantInsertion {
    NewlyInsertedParticipant,
    UpdatedExistingParticipant,
}
``` [2](#0-1) 

Nodes re-submit their attestations periodically (the codebase references `periodic-attestation-submission` and `attestation-resubmission-interval`). Because a node cannot know in advance whether the contract will classify its submission as new or an update, it must attach a deposit large enough to cover the worst-case storage cost. On every re-attestation cycle where the node is already a participant, that deposit is forfeited.

The analog to the AuraSpell report is direct: `WAuraPools#burn` sends multiple reward tokens to the calling contract, but `AuraSpell#openPositionFarm` only forwards one of them (`AURA`) back to the user. Here, `submit_participant_info` receives a deposit from the caller but only conditionally returns the unused portion — the "other reward tokens" are the deposit on the `UpdatedExistingParticipant` branch.

### Impact Explanation

Every periodic re-attestation call made by an existing participant that includes a non-zero deposit results in a permanent, unrecoverable transfer of NEAR tokens from the participant's account to the contract. Over the lifetime of the network (re-attestation happens on the order of hours), the cumulative loss per node is proportional to `deposit_amount × resubmission_frequency`. This breaks the production accounting invariant that deposits are refunded when no storage is consumed.

This maps to the **Medium** allowed impact: *balance or contract execution-flow manipulation that breaks production safety/accounting invariants*.

### Likelihood Explanation

Nodes cannot query the contract to determine whether their next `submit_participant_info` call will be classified as new or an update before submitting. The safe, correct behavior for a node is to always attach a deposit sufficient to cover storage. The `UpdatedExistingParticipant` path is reached on every re-attestation after the first, which is the common steady-state case for all long-running participants.

### Recommendation

Add an unconditional refund of any attached deposit when `attestation_storage_must_be_paid_by_caller` is `false`:

```rust
} else {
    let attached = env::attached_deposit();
    if attached > NearToken::from_yoctonear(0) {
        Promise::new(account_id).transfer(attached).detach();
    }
}
```

Alternatively, restructure the function to call `require_deposit(NearToken::from_yoctonear(0), &account_id)` unconditionally at the top (which already handles excess-refund logic) and only perform the storage-cost check inside the conditional block.

### Proof of Concept

1. Contract is in Running state with participant `alice.near` already having a stored attestation (`UpdatedExistingParticipant` will be returned on re-submission).
2. `alice.near` calls `submit_participant_info` with a fresh TEE quote and attaches `1_000_000_000_000_000_000_000_000` yoctoNEAR (1 NEAR) to cover potential storage.
3. `add_participant` returns `UpdatedExistingParticipant` because alice is already stored.
4. `caller_is_not_participant` is `false` (alice is a current participant).
5. `attestation_storage_must_be_paid_by_caller` evaluates to `false`.
6. The `if` block is skipped entirely — no deposit check, no refund.
7. The 1 NEAR deposit is permanently retained by the contract.
8. Alice's account balance decreases by 1 NEAR with no corresponding storage allocation or service rendered. [3](#0-2)
