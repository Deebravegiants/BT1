### Title
`burnFrom(self, amount)` Requires Self-Approval While `burn(amount)` Does Not, Creating Inconsistent Burn API — (`contracts/ccip/WrappedRSETH.sol`)

### Summary
`WrappedRSETH` implements `IBurnMintERC20` and exposes two burn paths for registered burners. `burn(uint256)` burns directly from `msg.sender` with no allowance check, but `burnFrom(address, uint256)` (and its alias `burn(address, uint256)`) routes through OZ `ERC20Burnable.burnFrom`, which calls `_spendAllowance(account, msg.sender, amount)`. When `account == msg.sender`, this requires a prior self-approval that is never set by default, causing the call to revert.

### Finding Description
`WrappedRSETH` inherits from `ERC20Burnable` and overrides both `burn(uint256)` and `burnFrom(address, uint256)` with an `onlyBurner` guard. The two paths diverge in their allowance semantics:

- `burn(amount)` → `ERC20Burnable.burn` → `_burn(msg.sender, amount)` — no allowance consumed.
- `burnFrom(account, amount)` → `ERC20Burnable.burnFrom` → `_spendAllowance(account, msg.sender, amount)` → `_burn(account, amount)` — allowance required even when `account == msg.sender`.

A CCIP pool that is a registered burner, holds wrsETH in its own balance, and calls `burnFrom(address(this), amount)` will revert with `ERC20InsufficientAllowance` unless it has previously called `approve(address(this), amount)` on itself. The `burn(address, uint256)` overload is a direct alias for `burnFrom` and shares the same defect: [5](#0-4) 

### Impact Explanation
No funds are lost or frozen permanently. However, the contract fails to deliver the behavior promised by the `IBurnMintERC20` interface: a registered burner cannot burn from their own address via `burnFrom` without an extra self-approval step that is not documented or enforced anywhere. CCIP pools integrating against this contract that follow the `burnFrom(self, amount)` pattern (a common CCIP pool pattern) will silently fail until they add a self-approve, creating an operational inconsistency. Impact: **Low — contract fails to deliver promised returns, but doesn't lose value**.

### Likelihood Explanation
Any CCIP pool registered as a burner that calls `burnFrom(pool_address, amount)` to burn tokens it holds will hit this. The CCIP reference implementation uses this exact pattern. Likelihood is **medium** given the contract is explicitly modeled after the CCIP BurnMintERC677 standard and CCIP pools are the primary consumers.

### Recommendation
Override `burnFrom` to skip the allowance check when `account == msg.sender`:

```solidity
function burnFrom(address account, uint256 amount)
    public
    override(IBurnMintERC20, ERC20Burnable)
    onlyBurner
{
    if (account == msg.sender) {
        // Burner burning their own tokens — no allowance needed
        _burn(account, amount);
    } else {
        super.burnFrom(account, amount);
    }
}
```

This aligns `burnFrom(self, amount)` with the behavior of `burn(amount)` for the self-burn case while preserving allowance enforcement for third-party burns.
