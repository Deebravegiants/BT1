### Title
Single-Peer-Controlled `highest_height` Triggers Unconditional Data Wipe via Stale Sync Hash Detection — (`chain/client/src/sync/state/mod.rs`)

### Summary

A single malicious peer can report an inflated `highest_height` to unconditionally trigger `StateSyncResult::StaleSyncHash`, which propagates to `SyncHandlerRequest::EpochSyncDataReset` and causes the syncing node to wipe its data directory and restart. The stale sync hash check at line 283 uses the raw peer-reported `highest_height` with no quorum requirement, no verification against the node's own verified chain, and no confirmation from multiple independent peers.

### Finding Description

The check in `StateSync::run()` is:

```rust
if highest_height > block_header.height() + chain.epoch_length + STALE_SYNC_HASH_THRESHOLD {
    return Ok(StateSyncResult::StaleSyncHash);
}
``` [1](#0-0) 

`highest_height` is a plain `u64` passed in from `SyncHandler::handle_sync_needed()`, which derives it from the maximum height reported across connected peers — a value any single peer can inflate arbitrarily by announcing a fabricated chain tip in the standard P2P handshake/height-advertisement protocol. There is no cryptographic proof, stake-weighted quorum, or cross-check against the node's own verified header chain before this value is used to make the destructive decision.

The propagation chain is direct and unconditional:

1. `StateSync::run()` returns `StateSyncResult::StaleSyncHash` [2](#0-1) 
2. `SyncHandler::handle_sync_needed()` maps it to `SyncHandlerRequest::EpochSyncDataReset` [3](#0-2) 
3. `ClientActor` receives `EpochSyncDataReset` and initiates data directory wipe + shutdown [4](#0-3) 

### Impact Explanation

The impact is **data destruction**, not ordinary denial of service. The node wipes its data directory and restarts from scratch. Any partially downloaded state sync progress is lost. A persistent attacker who reconnects after each restart can keep the victim node in a permanent wipe-restart loop, making it impossible to ever complete state sync. This is a targeted, repeatable attack against any node in `SyncStatus::StateSync`.

### Likelihood Explanation

Any unprivileged peer can connect to a NEAR node via the standard P2P protocol and announce an arbitrary `highest_height` in its peer info. No validator key, stake, or special privilege is required. The attacker only needs to know (or estimate) the victim's `sync_hash_height` and `epoch_length` to craft the threshold-exceeding value. Both are observable from public chain data.

### Recommendation

The stale sync hash check must not rely on a single peer's reported height. Require either:
- A **quorum** (e.g., majority) of connected peers to independently report a height exceeding the threshold before triggering data reset, or
- Verification against the node's **own verified header chain** (`chain.header_head()`) rather than the raw peer-reported maximum.

The `STALE_SYNC_HASH_THRESHOLD` comment itself acknowledges the threshold must be large enough to avoid false positives from normal network conditions — but it does not address adversarial inflation at all. [5](#0-4) 

### Proof of Concept

Using the `test_far_horizon_stale_sync_hash_detection` test pattern:

1. Place a syncing node in `SyncStatus::StateSync` with a known `sync_hash` at height `H`.
2. Connect a single malicious peer that advertises `highest_height = H + epoch_length + STALE_SYNC_HASH_THRESHOLD + 1`.
3. All other honest peers advertise the real tip, which is within the threshold.
4. On the next `run_sync_step` tick, `StateSync::run()` evaluates line 283 using the single peer's inflated value as `highest_height`.
5. The check passes, `StaleSyncHash` is returned, `EpochSyncDataReset` is issued, and the node wipes its data.

The honest peers' heights are irrelevant because `highest_height` is the **maximum** across all peers — one malicious peer sets the maximum.

/// Number of blocks past epoch_length that triggers stale sync hash detection.
///
/// During state sync, if the network's highest height exceeds the sync hash
/// block's height + epoch_length + this threshold, the sync hash is considered
/// stale and the node triggers a data reset + restart.
///
/// Must be large enough to account for epoch stretching due to missing blocks
/// and finality delays. Epoch boundaries require `last_finalized_height + 3
/// >= estimated_next_epoch_start`, so with sparse block production, epochs
/// can extend beyond epoch_length. A 100-block threshold is safe because a
/// false positive would require 100+ blocks without finality — a catastrophic
/// consensus failure, not normal missing blocks.
///
/// Under `test_features`, the threshold is lowered to 5 so tests with
/// epoch_length=10 can trigger stale sync hash detection without needing
/// hundreds of blocks.
#[cfg(not(feature = "test_features"))]
pub const STALE_SYNC_HASH_THRESHOLD: u64 = 100;
#[cfg(feature = "test_features")]
pub const STALE_SYNC_HASH_THRESHOLD: u64 = 5;
