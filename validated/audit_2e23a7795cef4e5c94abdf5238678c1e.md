### Title
Missing G2 Subgroup Membership Check in `alt_bn128_pairing_check` Allows Forged Pairing Results — (`runtime/near-vm-runner/src/logic/alt_bn128.rs`)

### Summary

`decode_g2` only validates that a supplied G2 point lies on the twist curve E'(Fq2) via `bn::AffineG2::new`. It performs **no prime-order subgroup check**. An attacker can supply a cofactor-order point (a point Q on E'(Fq2) with `[h2]Q = O` but `[r]Q ≠ O`) that passes decoding and causes `bn::pairing_batch` to return `Gt::one()`, making `pairing_check` return `true` for an invalid input.

### Finding Description

The call chain is:

```
contract_call
  → alt_bn128_pairing_check (logic.rs:1119)
    → alt_bn128::pairing_check (alt_bn128.rs:77)
      → decode_g2 (alt_bn128.rs:131)
        → bn::AffineG2::new(x, y)   ← curve-membership only, no subgroup check
      → bn::pairing_batch(&elements) ← no subgroup guard before pairing
```

`decode_g2` accepts any affine point satisfying the twist curve equation: [1](#0-0) 

`pairing_check` then feeds the accepted point directly into `bn::pairing_batch` and compares to `Gt::one()`: [2](#0-1) 

The `bn` workspace dependency is the `substrate-bn` crate, whose `AffineG2::new` only checks the curve equation — it does **not** multiply by `r` or `h2` to verify subgroup membership.

The host-function documentation in `logic.rs` explicitly promises rejection of out-of-subgroup points:

> "If point coordinates are not on curve, **point is not in the subgroup**, scalar is not in the field … the function returns `AltBn128InvalidInput`." [3](#0-2) 

That promise is not implemented.

### Impact Explanation

For BN128, the twist E'(Fq2) has order `h2 · r` where the cofactor `h2 ≈ r` (approximately `2.19 × 10^76`). A cofactor point Q satisfies `[h2]Q = O`.

By bilinearity of the Ate pairing:

```
e(P, Q)^h2 = e(P, [h2]Q) = e(P, O) = 1  (in GT)
```

Since GT has prime order `r` and `gcd(h2, r) = 1`, the only element of GT with order dividing `h2` is the identity. Therefore **`e(P, Q) = 1` for every P ∈ G1 and every cofactor point Q**.

Consequence: calling `alt_bn128_pairing_check` with `(G1_generator, Q_cofactor)` returns `1` (true) unconditionally, regardless of the G1 element. Any on-chain Groth16 verifier, light-client bridge, or epoch-proof verifier built on this host function can be made to accept a completely fabricated proof.

### Likelihood Explanation

The attack is straightforward:
1. Pick any point P on E'(Fq2) (e.g., the standard G2 generator).
2. Compute `Q = [h2]P` using the publicly known BN128 cofactor `h2`.
3. Encode Q as 128 little-endian bytes (two Fq2 coordinates).
4. Construct a pairing-check input `(G1_gen_bytes || Q_bytes)` and call `alt_bn128_pairing_check`.
5. The host function returns `1`.

No privileged access is required. Any NEAR account can deploy a contract or call an existing one that invokes this host function.

### Recommendation

Add an explicit subgroup membership check in `decode_g2` after `AffineG2::new` succeeds. The standard approach is to verify `[r]Q = O` (scalar-multiply by the group order and assert the result is the point at infinity). Alternatively, use the endomorphism-based subgroup check available for BN curves. The BLS12-381 implementation in the same codebase correctly performs this check via `blst_p2_affine_in_g2`: [4](#0-3) 

The analogous guard is absent for alt_bn128 G2.

### Proof of Concept

```rust
// Pseudocode – executable in a NEAR contract
let h2: U256 = /* BN128 G2 cofactor */ "21888242871839275222246405745257275088844257914179612981280795685733913198485".parse().unwrap();
let g2_gen = /* standard G2 generator point on E'(Fq2) */;
let q_cofactor = g2_gen.scalar_mul(h2); // [h2]G2_gen — order divides h2, not in G2
let g1_gen = /* standard G1 generator */;

// Encode as (G1_gen || Q_cofactor) in little-endian Fq / Fq2 bytes (192 bytes total)
let input = encode(g1_gen, q_cofactor);

// Returns 1 (true) — pairing check passes despite Q not being in G2
let result = env::alt_bn128_pairing_check(input);
assert_eq!(result, 1);
```

The pairing returns `Gt::one()` because `e(G1_gen, Q_cofactor) = 1` by the cofactor argument above, so `pairing_check` incorrectly signals a valid proof.

pub(crate) fn pairing_check(
    elements: &[[u8; PAIRING_CHECK_ELEMENT_SIZE]],
) -> Result<bool, InvalidInput> {
    let elements: Vec<(bn::G1, bn::G2)> = elements
        .iter()
        .map(|chunk| {
            let (g1, g2) = stdx::split_array(chunk);
            let g1 = decode_g1(g1)?;
            let g2 = decode_g2(g2)?;
            Ok((g1, g2))
        })
        .collect::<Result<Vec<_>, InvalidInput>>()?;

    let res = bn::pairing_batch(&elements) == bn::Gt::one();

    Ok(res)
}

fn decode_g2(raw: &[u8; 2 * POINT_SIZE]) -> Result<bn::G2, InvalidInput> {
    let (x, y) = stdx::split_array(raw);
    let x = decode_fq2(x)?;
    let y = decode_fq2(y)?;
    if x.is_zero() && y.is_zero() {
        Ok(bn::G2::zero())
    } else {
        bn::AffineG2::new(x, y)
            .map_err(|_err| InvalidInput::new("invalid g2", raw))
            .map(bn::G2::from)
    }
}
