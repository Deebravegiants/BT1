### Title
Caller-Agnostic Fan-Out Queue Key Allows Indefinite DoS of `verify_foreign_transaction` Requests - (File: `crates/contract/src/pending_requests.rs`, `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

### Summary

The `verify_foreign_transaction` endpoint uses a caller-agnostic request key (`VerifyForeignTransactionRequest` contains no submitter account ID). Combined with the hard cap of `MAX_PENDING_REQUEST_FAN_OUT = 128` slots per key, any unprivileged account can saturate the fan-out queue for a specific foreign-chain transaction, permanently blocking legitimate bridge callers from ever queuing their own yield — at a cost of 128 yoctonear per ~200-block cycle.

### Finding Description

`VerifyForeignTransactionRequest` is defined as:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
```

It contains no caller account ID. [1](#0-0) 

This is by design — the contract explicitly fan-outs responses to all callers who submitted the same foreign-chain request, regardless of who they are. The unit test `verify_foreign_transaction__should_queue_duplicates_from_different_callers` confirms that alice and bob both queue under the same map key. [2](#0-1) 

The fan-out queue is bounded at `MAX_PENDING_REQUEST_FAN_OUT = 128`. When the queue is full, `push_pending_yield` panics with `PendingRequestQueueFull`, rejecting the caller's transaction entirely: [3](#0-2) 

The `verify_foreign_transaction` entry point requires only 1 yoctonear deposit: [4](#0-3) 

**Attack path:**

1. Attacker observes (or predicts) a specific `ForeignChainRpcRequest` that a bridge service will submit (e.g., a Bitcoin `tx_id` that is public on-chain).
2. Attacker submits `verify_foreign_transaction` with that exact request 128 times from one or more accounts, paying 128 yoctonear total.
3. The queue for that `VerifyForeignTransactionRequest` key is now full.
4. The legitimate bridge caller submits the same request — the contract panics with `PendingRequestQueueFull` and the yield is never registered.
5. After ~200 blocks, the NEAR runtime fires the yield-timeout callback for each of the attacker's 128 slots, draining the queue via `pop_oldest_pending_yield`. [5](#0-4) 
6. Attacker immediately refills the queue. The cycle repeats indefinitely.

Alternatively, if MPC nodes respond before the timeout, `resolve_yields_for` drains all 128 attacker yields at once — the attacker receives the verification response (harmless to them), and the victim's yield was never registered. The attacker refills immediately. [6](#0-5) 

This is structurally identical to the external report: a bounded post-operation invariant check (min-shares / queue-full) is exploited by an adversary who manipulates shared state to block a legitimate operation, then undoes their manipulation at no net cost.

**Why `sign()` is not equivalently vulnerable:** `SignatureRequest` embeds a `tweak` derived from `(predecessor_id, path)`, making each caller's key unique. An attacker cannot forge a victim's tweak. [7](#0-6) 

### Impact Explanation

A bridge service relying on `verify_foreign_transaction` to confirm foreign-chain deposits or withdrawals before releasing funds can be blocked from processing any specific transaction indefinitely. The victim's yield is never registered, so the MPC network's response never reaches them. Funds awaiting that verification remain frozen for as long as the attacker sustains the attack. This matches **Medium** impact: request-lifecycle manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.

### Likelihood Explanation

**Medium.** The attacker must know the exact `ForeignChainRpcRequest` (chain, tx_id, extractors, finality, domain_id, payload_version) in advance. For bridge services, the foreign-chain transaction ID is public on-chain and the other fields are protocol-fixed, making the full key predictable. The cost is 128 yoctonear (~$0) plus gas per ~200-block cycle (~4 minutes on NEAR), making sustained attacks economically trivial.

### Recommendation

Include the submitter's `predecessor_account_id` in the `VerifyForeignTransactionRequest` key (analogous to how `SignatureRequest` includes the caller via the `tweak`). This makes each caller's queue slot independent, preventing cross-caller queue saturation. If caller-agnostic fan-out is a required feature (so that multiple bridge callers for the same tx share one MPC computation), introduce a per-caller sub-queue or a per-caller slot reservation within the shared queue, so one account cannot consume all 128 slots.

    domain_id: foreign_tx_domain_id,

        tx_id: BitcoinTxId([0xAA; 32]),

        extractors: vec![BitcoinExtractor::BlockHash],

// Attacker fills all 128 slots (128 × 1 yoctonear = essentially free)
for _ in 0..128 {
    attacker_account
        .call(contract.id(), "verify_foreign_transaction")
        .args_json(json!({ "request": target_request }))
        .deposit(NearToken::from_yoctonear(1))
        .max_gas()
        .transact()

    .call(contract.id(), "verify_foreign_transaction")
    .args_json(json!({ "request": target_request }))
    .deposit(NearToken::from_yoctonear(1))
    .max_gas()
    .transact()
    .await?
    .into_result();
assert!(result.unwrap_err().to_string().contains("Pending-request queue is full"));

/// Maximum number of concurrent yield-resume promises that can be queued for a single
/// request key (i.e. the number of duplicate submissions whose responses fan out from
/// one MPC reply).
///
/// The ceiling is needed because `respond*` drains the entire queue in one call: every
/// queued yield triggers a host-side `promise_yield_resume`, paid for out of the
/// responder's 300 TGas budget. Without a cap, an attacker could enqueue enough
/// duplicates to make `respond*` run out of gas and strand every queued caller.
///
/// 128 is validated empirically by the sandbox test
/// `test_contract_request_duplicate_requests_fan_out`, which fills the queue to this
/// cap across all four signature schemes and confirms `respond*` drains it inside its
/// 300 TGas budget.
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
