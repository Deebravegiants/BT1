Audit Report

## Title
Division by Zero in `getRsETHAmountToMint()` When `rsETHPrice` Is Uninitialized Blocks All Deposits - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool.getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()` without a zero-check. `rsETHPrice` in `LRTOracle` defaults to `0` at deployment and is never set during `initialize()`. Any deposit attempted before `updateRSETHPrice()` is called will revert with a division-by-zero panic, blocking all deposit functionality for that window.

## Finding Description
`LRTOracle` declares `rsETHPrice` as a plain storage variable with no initializer: [1](#0-0) 

`LRTOracle.initialize()` sets only `lrtConfig` and emits an event — it does not set `rsETHPrice`: [2](#0-1) 

`_updateRsETHPrice()` does set `rsETHPrice = 1 ether` for the zero-supply case, but only when explicitly called: [3](#0-2) 

`LRTDepositPool.getRsETHAmountToMint()` performs an unchecked division by `rsETHPrice`: [4](#0-3) 

This is called unconditionally from `_beforeDeposit()`: [5](#0-4) 

Which is called by the public, permissionless `depositETH()`: [6](#0-5) 

No guard in `getRsETHAmountToMint()`, `_beforeDeposit()`, or any caller checks `rsETHPrice != 0` before the division. The EVM will panic with division-by-zero, reverting the transaction.

## Impact Explanation
All user deposits (`depositETH`, `depositAsset`) are completely blocked for the entire period during which `rsETHPrice == 0`. This constitutes **temporary freezing of funds** (Medium), as the core deposit flow of the protocol is non-functional. No funds are lost, but the protocol cannot accept any deposits until a separate `updateRSETHPrice()` transaction is mined.

## Likelihood Explanation
The condition is structurally guaranteed at every fresh deployment of `LRTOracle`. The `initialize()` function does not set `rsETHPrice`, so the zero state is the default. Any unprivileged user who calls `depositETH` or `depositAsset` before `updateRSETHPrice()` is called will trigger the revert. No attacker action is required — a normal user attempting a deposit in this window is sufficient. Likelihood is **medium**: the window is short in practice but is guaranteed to exist and requires no special conditions to exploit.

## Recommendation
Initialize `rsETHPrice` to `1 ether` inside `LRTOracle.initialize()`, mirroring the logic already present in `_updateRsETHPrice()` for the zero-supply case:

```solidity
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    lrtConfig = ILRTConfig(lrtConfigAddr);
    rsETHPrice = 1 ether;          // prevent division-by-zero before first update
    highestRsethPrice = 1 ether;
    emit UpdatedLRTConfig(lrtConfigAddr);
}
```

Alternatively, add an explicit zero-check guard in `getRsETHAmountToMint()`:

```solidity
uint256 currentRsETHPrice = lrtOracle.rsETHPrice();
if (currentRsETHPrice == 0) revert RsETHPriceNotInitialized();
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / currentRsETHPrice;
```

## Proof of Concept
1. Deploy `LRTOracle` fresh. `rsETHPrice` storage slot = `0` (never set by `initialize()`).
2. Deploy `LRTDepositPool` pointing to the above oracle.
3. Do **not** call `updateRSETHPrice()`.
4. Call `depositETH{value: 1 ether}(0, "")` as any EOA.
5. Execution reaches `getRsETHAmountToMint()` → `(1e18 * assetPrice) / 0` → EVM division-by-zero panic → transaction reverts.
6. All deposits are blocked until a separate `updateRSETHPrice()` transaction is mined.

Foundry test sketch:
```solidity
function test_divisionByZeroOnFirstDeposit() public {
    // Deploy fresh oracle without calling updateRSETHPrice
    assertEq(lrtOracle.rsETHPrice(), 0);
    vm.expectRevert(); // division by zero panic
    lrtDepositPool.depositETH{value: 1 ether}(0, "");
}
```
