### Title
Unverified `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Replay by a Single Attested Participant — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies only that the submitted signature is cryptographically valid over `response.payload_hash`. It never checks that `response.payload_hash` was actually derived from `ForeignTxSignPayload { request: <the pending request>, values: ... }`. A single attested MPC participant (strictly below the signing threshold) can therefore replay a legitimate MPC signature produced for one foreign-chain request against a completely different pending request, causing the contract to deliver a `VerifyForeignTransactionResponse` whose `payload_hash` encodes a different transaction than the one the user asked to verify.

---

### Finding Description

`respond_verify_foreign_tx` in `crates/contract/src/lib.rs` accepts two caller-supplied arguments:

- `request: VerifyForeignTransactionRequest` — used only as a map key to locate and drain the pending yield queue.
- `response: VerifyForeignTransactionResponse` — contains `payload_hash` (a 32-byte hash) and `signature`.

The only on-chain check performed on `response` is:

```rust
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,   // ← taken verbatim from response, never re-derived
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract never recomputes `SHA-256(borsh(ForeignTxSignPayload::V1 { request: request.request, values: ... }))` and never asserts that the result equals `response.payload_hash`. The `request` argument and the `payload_hash` inside `response` are therefore completely decoupled at the contract level.

The signed payload is defined as:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,   // includes tx_id, chain, extractors
    pub values: Vec<ExtractedValue>,
}
// msg_hash = SHA-256(borsh(ForeignTxSignPayload))
``` [2](#0-1) 

Because `payload_hash` commits to the original `ForeignChainRpcRequest` (including `tx_id`), a hash produced for request X encodes a different transaction than request Y. The contract resolves the yield for Y with X's hash and signature without detecting the mismatch.

The SDK helper `ForeignChainSignatureVerifier::verify_signature` does perform this check client-side:

```rust
let payload_is_correct = expected_payload_hash == response.payload_hash;
if !payload_is_correct {
    return Err(VerifyForeignChainError::IncorrectPayloadSigned { ... });
}
``` [3](#0-2) 

However, this check is entirely optional and lives outside the contract. Any caller contract that omits it — or that simply trusts the contract's yield-resume result — receives a forged attestation.

---

### Impact Explanation

A single attested MPC participant (below the signing threshold) can:

1. Observe any previously completed `respond_verify_foreign_tx` call on-chain and extract its `(payload_hash_X, signature_X)` pair, which was legitimately produced by the MPC threshold for request X (e.g., Bitcoin `tx_id = [0xAA; 32]`).
2. Identify a different pending request Y (e.g., Bitcoin `tx_id = [0xBB; 32]`) in `pending_verify_foreign_tx_requests`.
3. Call `respond_verify_foreign_tx(request = Y, response = { payload_hash: payload_hash_X, signature: signature_X })`.
4. The contract passes the signature check (the signature is genuinely valid over `payload_hash_X`) and resolves Y's yield with the mismatched response.

The caller contract for Y receives a `VerifyForeignTransactionResponse` whose `payload_hash` encodes transaction X, not Y. Any bridge or omnibridge contract that uses this response to gate fund release — without independently recomputing the expected hash — will execute based on the wrong foreign-chain transaction. This maps directly to **forged foreign-chain verification causing invalid bridge execution**, a High-severity impact under the allowed scope.

---

### Likelihood Explanation

- **Attacker role:** A single attested MPC participant. No threshold collusion is required; the attacker only needs to be enrolled in the participant set.
- **Required material:** A previously emitted, on-chain `respond_verify_foreign_tx` call. These are public NEAR transactions; any participant (or observer) can extract `payload_hash` and `signature` from chain history.
- **Precondition:** A concurrent pending request Y must exist. In a live bridge environment with multiple users, this is routine.
- **Complexity:** Trivially low — the attacker constructs one NEAR function call with two JSON arguments.

---

### Recommendation

Inside `respond_verify_foreign_tx`, after verifying the signature, recompute the minimum verifiable prefix of the expected payload hash from the pending `request` and assert consistency. Because `values` are not stored on-chain, the simplest sound fix is to require the caller to also supply `values: Vec<ExtractedValue>` in the response, recompute `SHA-256(borsh(ForeignTxSignPayload::V1 { request: request.request, values }))` on-chain, and assert it equals `response.payload_hash` before resolving the yield. This mirrors what the SDK already does client-side and closes the cross-request replay path entirely.

---

### Proof of Concept

```
// Setup: two pending requests with different tx_ids
Alice calls verify_foreign_transaction(tx_id = [0xAA; 32])  → pending request A
Bob   calls verify_foreign_transaction(tx_id = [0xBB; 32])  → pending request B

// Honest flow for Bob completes first:
MPC threshold signs payload_hash_B = SHA256(borsh(ForeignTxSignPayload { request_B, values_B }))
Honest node calls respond_verify_foreign_tx(request=B, response={payload_hash_B, sig_B})
  → contract resolves B's yield ✓

// Attacker (single attested participant) replays Bob's response against Alice's request:
Attacker calls respond_verify_foreign_tx(
    request  = A,                          // Alice's pending request (tx_id=[0xAA;32])
    response = { payload_hash_B, sig_B }   // Bob's hash+sig (tx_id=[0xBB;32])
)

// Contract checks:
verify_ecdsa_signature(sig_B, payload_hash_B, root_pk) → OK  ✓  (sig is genuinely valid)
resolve_yields_for(pending_map, request=A, response)   → resolves Alice's yield ✓

// Alice's contract receives:
VerifyForeignTransactionResponse {
    payload_hash: payload_hash_B,   // encodes tx_id=[0xBB;32], NOT Alice's [0xAA;32]
    signature:    sig_B,
}
// If Alice's bridge contract does not call ForeignChainSignatureVerifier::verify_signature,
// it accepts the attestation and releases funds for the wrong transaction.
```

The root cause is at `crates/contract/src/lib.rs` lines 718–747: the contract verifies the signature over `response.payload_hash` but never asserts `payload_hash == recompute(request)`. [4](#0-3)
