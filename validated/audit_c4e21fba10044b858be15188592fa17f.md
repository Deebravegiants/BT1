### Title
Queued Future-Round `ProposalInit` Uses Stale Timestamp for L1 Gas Price Validation, Allowing Incorrect Gas Prices in Committed Blocks — (File: `crates/apollo_consensus_orchestrator/src/sequencer_consensus_context.rs`)

---

### Summary

When a `ProposalInit` for a future consensus round arrives, it is stored verbatim in `queued_proposals` without any gas-price or timestamp pre-validation. When the round is eventually reached, `is_proposal_init_valid` uses the **stale `init_proposed.timestamp`** from the queued proposal to look up L1 gas prices from the oracle. Because the timestamp check only rejects timestamps that are too far in the *future* (not timestamps that are stale/past), a malicious proposer can embed a timestamp from a historical moment when L1 gas prices were favorable, causing the validator to compare the proposer's L1 gas prices against those historical prices rather than current market prices. A block with incorrect L1 gas prices is then committed.

---

### Finding Description

**Step 1 — Proposal queued without validation.**

In `validate_proposal`, when `init.round > self.current_round`, the full `ProposalInit` (including its `timestamp`, `l1_gas_price_fri`, `l1_data_gas_price_fri`, `l1_gas_price_wei`, `l1_data_gas_price_wei`) is stored as-is: [1](#0-0) 

No timestamp or gas-price validation is performed at this point.

**Step 2 — Stale timestamp passes the timestamp check at validation time.**

When the round is reached in `set_height_and_round`, the stored proposal is dequeued and `is_proposal_init_valid` is called. The timestamp check is: [2](#0-1) 

The upper-bound check (`timestamp > now + block_timestamp_window_seconds`) only rejects timestamps that are too far **in the future**. For a queued proposal whose timestamp `T_stale` was set to just after the previous block (e.g., `T_prev + 1`), by the time the round is reached at `T_validate >> T_stale`, the condition `T_stale > T_validate + 1` is trivially false. The stale timestamp passes.

The production value of `block_timestamp_window_seconds` is **1 second**: [3](#0-2) 

**Step 3 — Stale timestamp drives the L1 gas price oracle lookup.**

Inside `is_proposal_init_valid`, the L1 gas prices are fetched using `init_proposed.timestamp` — the stale timestamp from the queued proposal: [4](#0-3) 

The oracle provider's `get_price_info` computes a rolling average of L1 gas prices anchored to the requested timestamp: [5](#0-4) 

The staleness guard only rejects timestamps that are too far **in the future** relative to the latest sample: [6](#0-5) 

A past timestamp is not rejected; the provider returns the historical average around `T_stale`.

**Step 4 — Proposer's L1 gas prices are validated against historical (stale) reference prices.**

The validator then checks the proposer's L1 gas prices against the historical reference within a 10% margin: [7](#0-6) 

If the proposer's prices match the historical prices at `T_stale` (within 10%), the check passes — even if current L1 gas prices are significantly different.

**Step 5 — Block is committed with incorrect gas prices.**

The accepted `ProposalInit` is passed to `initiate_validation` → `convert_to_sn_api_block_info`, which embeds the proposer's stale L1 gas prices into the `BlockInfo` used for execution: [8](#0-7) 

These gas prices are then committed into the block hash via `PartialBlockHashComponents::new`: [9](#0-8) 

---

### Impact Explanation

A malicious proposer for a future round can commit a block whose L1 gas prices (`l1_gas_price_fri`, `l1_data_gas_price_fri`, `l1_gas_price_wei`, `l1_data_gas_price_wei`) reflect a historical moment rather than current market conditions. This directly affects:

1. **Fee calculations for L1 handler transactions** — L1 handler fees are computed using the block's L1 gas price. Artificially low prices allow L1 handler transactions to be underpriced, causing economic harm to the protocol.
2. **Block hash integrity** — The block hash commits to gas prices; a block with stale gas prices produces a different hash than one with correct prices, affecting L1 anchoring and proof verification.
3. **Economic impact** — Matches the "Critical. Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact" criterion.

---

### Likelihood Explanation

Requires the attacker to be the designated proposer for a future round. In BFT consensus with up to `f < n/3` Byzantine validators, a single malicious validator can be the designated proposer for a round. The attack is straightforward: send a future-round proposal with a crafted stale timestamp before the round is reached. No special privileges beyond being a scheduled proposer are required.

---

### Recommendation

1. **Validate the timestamp at queue time**, not only at validation time. Reject proposals whose timestamp is not within `[last_block_timestamp, now + block_timestamp_window_seconds]` before queuing them.
2. **Use the current wall-clock time** (not `init_proposed.timestamp`) to look up L1 gas prices during `is_proposal_init_valid`. The block timestamp is proposer-controlled; the oracle lookup should be anchored to the validator's local clock at validation time.
3. **Re-evaluate the timestamp upper-bound check** for queued proposals: when a queued proposal is dequeued, also verify that `init_proposed.timestamp >= now - some_staleness_bound` to reject proposals whose timestamps have become stale.

---

### Proof of Concept

1. Attacker is the designated proposer for round `R` (a future round at the current height).
2. At time `T_queue`, L1 gas prices are high. Attacker observes that at time `T_prev + 1` (just after the previous block), L1 gas prices were low.
3. Attacker sends `ProposalInit` for round `R` with:
   - `timestamp = T_prev + 1` (stale, but passes `>= last_block_timestamp`)
   - `l1_gas_price_fri = low_historical_price` (matching oracle prices at `T_prev + 1`)
   - `l2_gas_price_fri = self.l2_gas_price` (must match exactly; unchanged within a height)
4. Proposal is stored in `queued_proposals` (no validation at queue time).
5. At time `T_validate >> T_queue`, round `R` is reached via `set_height_and_round`.
6. `is_proposal_init_valid` runs:
   - Timestamp check: `T_prev + 1 >= T_prev` ✓ and `T_prev + 1 <= T_validate + 1` ✓ (stale timestamp passes)
   - L1 oracle lookup: uses `T_prev + 1`, returns historical low prices
   - `within_margin(low_historical_price, low_historical_price, 10%)` → ✓
7. Block is committed with `l1_gas_price = low_historical_price`, even though current L1 gas prices are high.
8. All L1 handler transactions in this block are executed with the artificially low L1 gas price, causing incorrect fee accounting.

pub(crate) fn convert_to_sn_api_block_info(
    init: &ProposalInit,
) -> Result<starknet_api::block::BlockInfo, StarknetApiError> {
    if init.l1_gas_price_fri.0 == 0
        || init.l1_gas_price_wei.0 == 0
        || init.l1_data_gas_price_fri.0 == 0
        || init.l1_data_gas_price_wei.0 == 0
        || init.l2_gas_price_fri.0 == 0
    {
        warn!("Zero gas price detected in block info: {:?}", init);
    }

    let l1_gas_price_fri = NonzeroGasPrice::new(init.l1_gas_price_fri)?;
    let l1_data_gas_price_fri = NonzeroGasPrice::new(init.l1_data_gas_price_fri)?;
    let l1_gas_price_wei = NonzeroGasPrice::new(init.l1_gas_price_wei)?;
    let l1_data_gas_price_wei = NonzeroGasPrice::new(init.l1_data_gas_price_wei)?;
    let l2_gas_price_fri = NonzeroGasPrice::new(init.l2_gas_price_fri)?;
    let proposal_init_info = PreviousProposalInitInfo::from(init);
    let eth_to_fri_rate = calculate_eth_to_fri_rate(&proposal_init_info)?;

    let l2_gas_price_wei = NonzeroGasPrice::new(init.l2_gas_price_fri.fri_to_wei(eth_to_fri_rate)?)

                 FRI: {}",
                init.l2_gas_price_fri
            )
        })?;
    Ok(starknet_api::block::BlockInfo {
        block_number: init.height,
        block_timestamp: BlockTimestamp(init.timestamp),
        sequencer_address: init.builder,
