Audit Report

## Title
Decimal Normalisation Missing in Token-to-rsETH Swap — Users Receive ~1e12× Less rsETH for Sub-18-Decimal Tokens - (File: `contracts/pools/RSETHPool.sol`)

## Summary

`viewSwapRsETHAmountAndFee(uint256 amount, address token)` computes `rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate` without first scaling `amountAfterFee` to 18 decimals. For any token with fewer than 18 decimals, the result is `10^(18 − tokenDecimals)` times smaller than the correct value. A user depositing 1 USDC (6 decimals) receives ~285 wei of rsETH instead of ~285,714,285,714,285 wei; their full USDC principal is transferred in and bridged to L1, permanently lost to the user.

## Finding Description

**Root cause — dimensional mismatch in the token path:**

The ETH-path overload (no `token` argument) correctly normalises by multiplying by `1e18`:

```solidity
// RSETHPool.sol L319
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

For 1 ETH (`amountAfterFee = 1e18` wei): `1e18 × 1e18 / rsETHToETHrate` → correct 1e18-scaled rsETH amount.

The token-path overload omits this normalisation:

```solidity
// RSETHPool.sol L346
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

`tokenToETHRate` is returned by `ChainlinkOracleForRSETHPoolCollateral.getRate()` as:

```solidity
// ChainlinkOracleForRSETHPoolCollateral.sol L34
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
```

This is the price of **one whole token** in ETH, scaled to 1e18. For the formula to be dimensionally correct, `amountAfterFee` must also represent whole-token units scaled to 1e18. For 18-decimal tokens this holds (1 token = 1e18 smallest units). For USDC (6 decimals), `amountAfterFee` is 1e12 times too small.

**Numeric proof (1 USDC, feeBps = 0):**

| Variable | Value |
|---|---|
| `amountAfterFee` | `1e6` |
| `tokenToETHRate` | `~3e14` (0.0003 ETH/USDC × 1e18) |
| `rsETHToETHrate` | `~1.05e18` |
| Actual `rsETHAmount` | `1e6 × 3e14 / 1.05e18 ≈ 285 wei` |
| Correct `rsETHAmount` | `1e18 × 3e14 / 1.05e18 ≈ 285,714,285,714,285 wei` |

The user's USDC is transferred in full at `deposit()` line 296 and accumulates in the pool to be bridged to L1, while the user receives only 285 wei of rsETH.

**`addSupportedToken` has no decimal guard:**

```solidity
// RSETHPool.sol L648-650
if (IOracle(oracle).getRate() == 0) {
    revert UnsupportedOracle();
}
```

Any token with a non-zero oracle rate can be listed. No check on `IERC20Metadata(token).decimals()` exists.

**Identical pattern confirmed in all pool variants:**
- `RSETHPool.sol` L346
- `RSETHPoolNoWrapper.sol` L311
- `RSETHPoolV3.sol` L334
- `RSETHPoolV3ExternalBridge.sol` L452
- `RSETHPoolV3WithNativeChainBridge.sol` L370

## Impact Explanation

Once a sub-18-decimal token is listed, every depositor of that token receives `10^(18 − tokenDecimals)` times less rsETH than owed. For USDC/USDT (6 decimals) the shortfall is 1e12×. The deposited tokens are not returned — they accumulate in the pool and are bridged to L1. This is **direct theft of user principal**, matching the Critical impact class: "Direct theft of any user funds, whether at-rest or in-motion, other than unclaimed yield."

## Likelihood Explanation

The bug is latent until `TIMELOCK_ROLE` adds a sub-18-decimal token via `addSupportedToken`. Adding USDC or USDT as collateral is a natural and common DeFi protocol expansion. No attacker action is required once the token is listed — every ordinary depositor is automatically affected. The precondition is a legitimate protocol governance action, not an attack. Likelihood is **Medium**.

## Recommendation

Normalise `amountAfterFee` to 18 decimals before applying the rate ratio in all affected pool variants:

```solidity
import { IERC20Metadata } from "@openzeppelin/contracts/token/ERC20/extensions/IERC20Metadata.sol";

uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 amountAfterFeeNorm = amountAfterFee * 10 ** (18 - tokenDecimals);
rsETHAmount = amountAfterFeeNorm * tokenToETHRate / rsETHToETHrate;
```

Additionally, enforce `IERC20Metadata(token).decimals() <= 18` inside `addSupportedToken` as a safety guard to prevent listing tokens with more than 18 decimals.

## Proof of Concept

Deploy a mock 6-decimal ERC20 and a mock oracle returning `3e14` (0.0003 ETH/token). Call `addSupportedToken` with `TIMELOCK_ROLE`. Then call `viewSwapRsETHAmountAndFee(1e6, mockUSDC)`:

- Returned `rsETHAmount` ≈ 285 wei
- Expected `rsETHAmount` ≈ 285,714,285,714,285 wei
- Ratio: exactly 1e12×

Confirm that `deposit(mockUSDC, 1e6, "")` transfers 1e6 USDC from the caller and sends only 285 wei of rsETH back, with the remaining value permanently locked in the pool and subsequently bridged to L1 via the token bridge.
