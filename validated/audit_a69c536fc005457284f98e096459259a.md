### Title
Withdrawal Requests in `LRTWithdrawalManager` Have No Expiration and No Cancellation Mechanism, Permanently Locking User rsETH and Inflating `assetsCommitted` - (File: contracts/LRTWithdrawalManager.sol)

---

### Summary

`LRTWithdrawalManager.initiateWithdrawal` creates withdrawal requests that persist indefinitely with no expiration timestamp and no user-callable cancellation path. Once a user initiates a withdrawal, their rsETH is transferred into the contract and `assetsCommitted[asset]` is permanently inflated. The user has no recourse to reclaim their rsETH unless the operator calls `unlockQueue`. This is the direct analog of the OlympusGovernance "active proposal does not expire" class: a state is activated, it persists indefinitely, and users cannot exit it unilaterally.

---

### Finding Description

When a user calls `initiateWithdrawal`, the following happens atomically:

1. The user's rsETH is pulled into the contract.
2. `expectedAssetAmount` is computed at the **current** rsETH price and locked into the request struct.
3. `assetsCommitted[asset] += expectedAssetAmount` is permanently incremented.
4. The request is pushed into `userAssociatedNonces[asset][user]`. [1](#0-0) 

The `WithdrawalRequest` struct stores only `rsETHUnstaked`, `expectedAssetAmount`, and `withdrawalStartBlock` — **no deadline or expiry field exists**. [2](#0-1) 

The only path to completion is:

- Operator calls `unlockQueue` → request is unlocked.
- Withdrawal delay passes.
- User calls `completeWithdrawal`. [3](#0-2) 

There is **no `cancelWithdrawal` function** anywhere in the contract. The user cannot unilaterally exit the queue and recover their rsETH. The request and the inflated `assetsCommitted` entry persist until the operator acts.

The `getAvailableAssetAmount` check at initiation time enforces:

```
availableAssetAmount = totalAssets - assetsCommitted[asset]
``` [4](#0-3) 

Every stale, unprocessed request keeps `assetsCommitted[asset]` elevated, shrinking the available window for all subsequent withdrawal initiations by other users.

---

### Impact Explanation

**Impact: Medium — Temporary freezing of funds.**

Two concrete harms arise:

1. **User rsETH is frozen indefinitely.** The user's rsETH is held in the contract with no expiry and no self-service exit. If the operator is slow, paused, or the `LRTUnstakingVault` lacks sufficient assets to satisfy `unlockQueue`, the user's rsETH is locked for an unbounded duration.

2. **`assetsCommitted` inflation blocks other users.** Each unprocessed request keeps `assetsCommitted[asset]` elevated. When `assetsCommitted[asset] ≥ totalAssets`, `getAvailableAssetAmount` returns 0 and every subsequent `initiateWithdrawal` call reverts with `ExceedAmountToWithdraw`. A single large stale request can freeze the withdrawal queue for all other users of that asset. [5](#0-4) 

---

### Likelihood Explanation

**Likelihood: Medium.**

- Any unprivileged user can call `initiateWithdrawal` — no role required.
- The operator's `unlockQueue` is a separate, off-chain-triggered transaction. Protocol inactivity, a paused state, or insufficient vault liquidity can delay or prevent it indefinitely.
- The OlympusGovernance judge accepted the analogous finding on the same reasoning: governance inactivity (no new proposal) keeps the stale proposal active; here, operator inactivity (no `unlockQueue` call) keeps stale withdrawal requests active and rsETH locked.
- The scenario is realistic: EigenLayer withdrawal delays already impose multi-day latency; if the vault is underfunded, `unlockQueue` reverts, and the queue stalls.

---

### Recommendation

1. **Add an expiry timestamp to `WithdrawalRequest`.** For example, `uint256 expiryBlock = withdrawalStartBlock + maxWithdrawalLifetimeBlocks`.
2. **Implement a `cancelWithdrawal` function** that allows a user to cancel their own request after the expiry has passed, returning their rsETH and decrementing `assetsCommitted[asset]`.
3. Alternatively, allow users to cancel at any time before the request is unlocked by the operator, mirroring the OlympusGovernance recommendation to let users reclaim their VOTES tokens.

---

### Proof of Concept

```
1. Alice calls initiateWithdrawal(stETH, 100e18, "").
   - 100 rsETH transferred to LRTWithdrawalManager.
   - assetsCommitted[stETH] += expectedAssetAmount (e.g., 105 stETH).
   - WithdrawalRequest stored with no expiry.

2. Operator does not call unlockQueue for 6 months
   (e.g., vault is underfunded, or protocol is paused).

3. Alice has no cancelWithdrawal function to call.
   Her 100 rsETH remains locked in the contract indefinitely.

4. Bob tries initiateWithdrawal(stETH, 1e18, "").
   getAvailableAssetAmount(stETH) = totalAssets - assetsCommitted[stETH].
   If assetsCommitted >= totalAssets → revert ExceedAmountToWithdraw.
   Bob's withdrawal is also blocked by Alice's stale, unprocessed request.
``` [6](#0-5) [7](#0-6)

        address asset,

        nonReentrant

            revert InvalidAmountToWithdraw();
        }

        IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);

        uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);

        if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();

        // preventing over-withdrawal.
        assetsCommitted[asset] += expectedAssetAmount;

        _addUserWithdrawalRequest(asset, rsETHUnstaked, expectedAssetAmount);

        emit ReferralIdEmitted(referralId);
    }

    function getAvailableAssetAmount(address asset) public view override returns (uint256 availableAssetAmount) {
        ILRTDepositPool lrtDepositPool = ILRTDepositPool(lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL));
        uint256 totalAssets = lrtDepositPool.getTotalAssetDeposits(asset);
        availableAssetAmount = totalAssets > assetsCommitted[asset] ? totalAssets - assetsCommitted[asset] : 0;
    }

    function _processWithdrawalCompletion(address asset, address user, string calldata referralId) internal {
        if (userAssociatedNonces[asset][user].empty()) {
            revert NoWithdrawalRequests(user, asset);
        }

        // Retrieve and remove the oldest withdrawal request for the user.
        uint256 usersFirstWithdrawalRequestNonce = userAssociatedNonces[asset][user].popFront();
        // Ensure the request is already unlocked.
        if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();

        bytes32 requestId = getRequestId(asset, usersFirstWithdrawalRequestNonce);
        WithdrawalRequest memory request = withdrawalRequests[requestId];

        delete withdrawalRequests[requestId];

        // Check that the withdrawal delay has passed since the request's initiation.
        if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();

    function _addUserWithdrawalRequest(address asset, uint256 rsETHUnstaked, uint256 expectedAssetAmount) internal {
        uint256 nextUnusedNonce_ = nextUnusedNonce[asset];

        // Generate a unique identifier for the new withdrawal request.
        bytes32 requestId = getRequestId(asset, nextUnusedNonce_);

        // Create and store the new withdrawal request.
        withdrawalRequests[requestId] = WithdrawalRequest({
            rsETHUnstaked: rsETHUnstaked, expectedAssetAmount: expectedAssetAmount, withdrawalStartBlock: block.number
        });

        // Map the user to the newly created request index and increment the nonce for future requests.
        userAssociatedNonces[asset][msg.sender].pushBack(nextUnusedNonce_);
        nextUnusedNonce[asset] = nextUnusedNonce_ + 1;

        emit AssetWithdrawalQueued(msg.sender, asset, rsETHUnstaked, nextUnusedNonce_);
