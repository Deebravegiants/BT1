### Title
Vault Inflation Attack via Direct ETH Donation Inflates `rsETHPrice`, Enabling Zero-rsETH Minting for Victim Deposits — (`contracts/LRTDepositPool.sol`, `contracts/LRTOracle.sol`)

---

### Summary

An unprivileged attacker can perform a classic first-depositor inflation attack against the LRT-rsETH protocol. By donating ETH directly to `LRTDepositPool` (which is counted in TVL via `address(this).balance`) and then calling the public `updateRSETHPrice()`, the attacker inflates `rsETHPrice` to an arbitrarily large value. Subsequent victim deposits with `minRSETHAmountExpected = 0` mint 0 rsETH, permanently losing the victim's ETH to the protocol — which the attacker can later redeem via their existing rsETH position.

---

### Finding Description

**Step 1 — rsETH minting formula is vulnerable to price inflation.**

`LRTDepositPool.getRsETHAmountToMint` computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [1](#0-0) 

If `rsETHPrice` is inflated to a sufficiently large value, integer division causes `rsethAmountToMint` to round down to 0.

**Step 2 — `rsETHPrice` is derived from raw ETH balance, which is directly donatable.**

`LRTOracle._updateRsETHPrice` computes:

```solidity
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
``` [2](#0-1) 

`totalETHInProtocol` is built by `_getTotalEthInProtocol`, which calls `ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset)`. For ETH, `getTotalAssetDeposits` calls `getETHDistributionData`, which reads:

```solidity
ethLyingInDepositPool = address(this).balance;
``` [3](#0-2) 

Because `LRTDepositPool` has an open `receive()` function, any ETH sent directly to the contract is counted in TVL. An attacker can donate ETH to inflate `totalETHInProtocol` without receiving any rsETH in return. [4](#0-3) 

**Step 3 — `updateRSETHPrice()` is public and `pricePercentageLimit` defaults to 0.**

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [5](#0-4) 

The price-increase guard only activates when `pricePercentageLimit > 0`:

```solidity
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
``` [6](#0-5) 

`pricePercentageLimit` is never set in `initialize`, so it defaults to `0`. With `pricePercentageLimit == 0`, the condition is always `false`, and the price can increase by any arbitrary amount in a single call.

**Step 4 — Zero rsETH minted is not rejected.**

`_beforeDeposit` only reverts if `rsethAmountToMint < minRSETHAmountExpected`:

```solidity
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [7](#0-6) 

When `minRSETHAmountExpected = 0` (the default for many callers), `0 < 0` is `false`, so the deposit proceeds and `RSETH.mint(msg.sender, 0)` is called — a no-op that mints nothing to the victim.

---

### Impact Explanation

**Critical — Direct theft of user funds.**

The victim deposits ETH and receives 0 rsETH. Their ETH is permanently absorbed into the protocol TVL. The attacker, holding the only rsETH in existence, now owns a claim on the entire TVL (their original deposit + the donation + the victim's deposit). Upon withdrawal via `LRTWithdrawalManager`, the attacker redeems their rsETH for the full inflated TVL, stealing the victim's deposit.

---

### Likelihood Explanation

**Medium-High.** The attack requires no frontrunning and no privileged access. The attacker only needs to:
1. Be the first depositor (or reduce supply to 1 wei via deposit + withdrawal).
2. Send ETH directly to `LRTDepositPool.receive()`.
3. Call the public `updateRSETHPrice()`.
4. Wait for a victim to deposit with `minRSETHAmountExpected = 0`.

The `pricePercentageLimit` protection is not set at deployment, leaving the protocol fully exposed at launch. Users who omit or zero-out the slippage parameter (common in integrations and scripts) are silently drained.

---

### Recommendation

1. **Reject zero rsETH mints:** In `_beforeDeposit`, add:
   ```solidity
   if (rsethAmountToMint == 0) revert ZeroRsETHMinted();
   ``` [7](#0-6) 

2. **Set `pricePercentageLimit` in `initialize`:** Ensure a non-zero limit is enforced from deployment. [8](#0-7) 

3. **Exclude raw balance from TVL accounting:** Track deposited ETH via an internal accounting variable rather than `address(this).balance`, so direct donations do not inflate TVL. [3](#0-2) 

4. **Perform a sacrificial initial deposit:** Mint a non-trivial amount of rsETH to the zero address at deployment (analogous to Uniswap V2's minimum liquidity lock) to make the attack economically infeasible.

---

### Proof of Concept

```
Initial state: rsETHPrice = 1e18, rsethSupply = 0

1. Attacker calls depositETH{value: 1 wei}(0, ""):
   rsethAmountToMint = (1 * 1e18) / 1e18 = 1 wei rsETH
   → rsethSupply = 1, totalETHInProtocol = 1 wei

2. Attacker sends 1000 ETH directly to LRTDepositPool (via receive()):
   → address(LRTDepositPool).balance = 1000 ETH + 1 wei
   → totalETHInProtocol = 1000e18 + 1 ≈ 1000e18

3. Attacker calls updateRSETHPrice():
   newRsETHPrice = (1000e18 * 1e18) / 1 = 1000e36
   → rsETHPrice = 1000e36  (pricePercentageLimit = 0, no revert)

4. Victim calls depositETH{value: 999 ETH}(0, ""):
   rsethAmountToMint = (999e18 * 1e18) / 1000e36 = 999e36 / 1000e36 = 0
   → 0 rsETH minted; victim's 999 ETH absorbed into TVL

5. Attacker calls updateRSETHPrice():
   totalETHInProtocol = 1000 ETH + 999 ETH = 1999 ETH
   rsethSupply = 1 wei
   rsETHPrice = 1999e18 * 1e18 / 1 = 1999e36

6. Attacker redeems 1 wei rsETH via LRTWithdrawalManager:
   ETH received = 1 wei * 1999e36 / 1e18 = 1999 ETH
   Net profit = 1999 ETH − 1000 ETH − 1 wei = 999 ETH (victim's full deposit)
```
