Audit Report

## Title
No minimum withdrawal amount enforced by default allows dust requests that permanently strand assets and block sweep - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
`minRsEthAmountToWithdraw[asset]` is never initialised in `initialize`, so it defaults to zero for every asset. The guard in `initiateWithdrawal` accepts any non-zero rsETH amount, allowing dust withdrawal requests. Once the operator calls `unlockQueue`, the rsETH is burned and a tiny asset amount is pulled into the withdrawal manager. Because the gas cost of `completeWithdrawal` exceeds the dust asset value, the user has no economic incentive to claim, leaving the asset stranded and `unlockedWithdrawalsCount` permanently positive, which blocks `sweepRemainingAssets`.

## Finding Description
`minRsEthAmountToWithdraw` is declared as a mapping with no initialisation in `initialize`:

```solidity
// LRTWithdrawalManager.sol line 35
mapping(address asset => uint256) public minRsEthAmountToWithdraw;
```

The guard in `initiateWithdrawal` (lines 162–164) is:

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```

When the mapping value is zero (the Solidity default for `uint256`), the second condition evaluates to `rsETHUnstaked < 0`, which is always false for `uint256`. Any non-zero rsETH amount therefore passes.

The withdrawal lifecycle then proceeds:
1. `initiateWithdrawal` — rsETH is transferred from the user to the contract; `assetsCommitted` is increased.
2. `unlockQueue` (operator-only, lines 301–307) — `_unlockWithdrawalRequests` increments `unlockedWithdrawalsCount[asset]` (line 809) for each processed request, then rsETH held by the contract is burned and the corresponding asset is pulled from `LRTUnstakingVault`.
3. `completeWithdrawal` / `completeWithdrawalForUser` — must be called to transfer the asset to the user and decrement `unlockedWithdrawalsCount[asset]` (line 717).

For a dust request (e.g., 1 wei rsETH → ~1 wei ETH), step 2 permanently destroys the user's rsETH. Step 3 requires a transaction costing tens of thousands of gas on Ethereum mainnet, far exceeding 1 wei in value. Neither the user nor the operator has an economic incentive to call `completeWithdrawal` / `completeWithdrawalForUser` for such a request.

`sweepRemainingAssets` (lines 401–403) reverts with `PendingWithdrawalsExist` whenever `unlockedWithdrawalsCount[asset] > 0`. Unclaimed dust requests keep this counter positive indefinitely, permanently blocking the sweep path for that asset.

The admin setter exists (lines 330–333) but is never called during deployment or initialisation, leaving the minimum at zero until an admin explicitly configures it.

## Impact Explanation
The protocol burns the user's rsETH in step 2 but cannot economically deliver the promised underlying asset because the gas cost of claiming exceeds the asset value. This matches **Low — Contract fails to deliver promised returns**: the user's rsETH is destroyed but the corresponding asset remains uncollectable in the withdrawal manager. The secondary effect is that `sweepRemainingAssets` is permanently blocked for the affected asset while any unclaimed dust withdrawal exists, preventing the protocol from recovering excess balances.

## Likelihood Explanation
The minimum is zero by default and requires an explicit admin call to `setMinRsEthAmountToWithdraw` per asset. Any unprivileged rsETH holder can call `initiateWithdrawal` with 1 wei at any time without any admin action. The condition is reachable on every supported withdrawal asset. The attack is repeatable and costs only 1 wei of rsETH plus gas for `initiateWithdrawal`.

## Recommendation
Enforce a non-zero minimum in `initialize` for each supported asset:

```solidity
// In initialize:
for (uint256 i; i < supportedAssets.length; i++) {
    minRsEthAmountToWithdraw[supportedAssets[i]] = 1e15; // e.g. 0.001 ETH floor
}
```

Additionally, add a zero-check in the setter to prevent the minimum from being reset to zero:

```solidity
function setMinRsEthAmountToWithdraw(address asset, uint256 minRsEthAmountToWithdraw_) external onlyLRTAdmin {
    if (minRsEthAmountToWithdraw_ == 0) revert MinAmountCannotBeZero();
    minRsEthAmountToWithdraw[asset] = minRsEthAmountToWithdraw_;
    emit MinAmountToWithdrawUpdated(asset, minRsEthAmountToWithdraw_);
}
```

## Proof of Concept
1. Deploy `LRTWithdrawalManager`; never call `setMinRsEthAmountToWithdraw` for ETH. `minRsEthAmountToWithdraw[ETH_TOKEN] == 0`.
2. Alice holds 1 wei rsETH and calls `initiateWithdrawal(ETH_TOKEN, 1, "")`. The check `1 < 0` (uint256) is false; the call succeeds. Alice's 1 wei rsETH is transferred to the withdrawal manager.
3. After `withdrawalDelayBlocks`, the operator calls `unlockQueue(ETH_TOKEN, ...)`. `_unlockWithdrawalRequests` processes Alice's request: `payoutAmount ≈ 1 wei ETH`. rsETH is burned (line 305); 1 wei ETH is pulled from `LRTUnstakingVault` (line 307). `unlockedWithdrawalsCount[ETH_TOKEN]` becomes 1.
4. Alice's `completeWithdrawal` would cost ~50,000–100,000 gas (≈ 0.001–0.003 ETH at 20–30 gwei), far exceeding 1 wei. Alice does not call it. The operator similarly has no economic incentive to call `completeWithdrawalForUser` for 1 wei.
5. The 1 wei ETH is permanently stranded. `sweepRemainingAssets(ETH_TOKEN)` reverts with `PendingWithdrawalsExist` because `unlockedWithdrawalsCount[ETH_TOKEN] == 1`.

Foundry test sketch:
```solidity
function test_dustWithdrawalStrandsAsset() public {
    // minRsEthAmountToWithdraw[ETH] == 0 by default
    vm.prank(alice);
    withdrawalManager.initiateWithdrawal(ETH_TOKEN, 1, "");
    vm.roll(block.number + withdrawalDelayBlocks + 1);
    operator.unlockQueue(ETH_TOKEN, 1, ...);
    // Alice does not call completeWithdrawal
    vm.expectRevert(PendingWithdrawalsExist.selector);
    manager.sweepRemainingAssets(ETH_TOKEN);
}
```
