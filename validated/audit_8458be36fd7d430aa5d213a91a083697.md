### Title
Stub `validatePerasCert` Always Returns `True`, Bypassing Peras Certificate Validation in BBODY Rule - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/BlockBody/Internal.hs`)

---

### Summary

The `validatePerasCert` function in the Dijkstra era's block body implementation is a deliberate stub that unconditionally returns `True`, ignoring all arguments. It is called in the `BBODY` transition rule to validate Peras certificates embedded in block bodies. Because the function never performs any cryptographic check, any block producer can include an arbitrary, forged Peras certificate in a block and all honest nodes will accept it without error.

---

### Finding Description

In `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/BlockBody/Internal.hs`, the `validatePerasCert` function is defined as a placeholder:

```haskell
-- | Mocked-up Peras certificate validation routine
--
-- NOTE: this function will be replaced with the real implementation from
-- 'cardano-base' once it's ready.
validatePerasCert :: Nonce -> PerasKey -> PerasCert -> Bool
validatePerasCert _ _ _ = True
``` [1](#0-0) 

This function is exported from `Cardano.Ledger.Dijkstra.BlockBody` and is called directly in the `dijkstraBbodyTransition` rule in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs`:

```haskell
case blockBody ^. perasCertBlockBodyL of
  SNothing -> pure ()
  SJust cert ->
    let nonce = block ^. prevNonceBlockHeaderL
     in validatePerasCert nonce PerasKey cert ?! injectFailure (PerasCertValidationFailed cert nonce)
``` [2](#0-1) 

The `?!` operator in the STS framework fails the transition if the predicate is `False`. Since `validatePerasCert _ _ _ = True` unconditionally, the `PerasCertValidationFailed` predicate failure — which is fully defined and serializable in `DijkstraBbodyPredFailure` — can **never** be triggered. [3](#0-2) 

The `PerasCert` type is a raw `ByteArray` newtype with no structural constraints, meaning any byte sequence is a syntactically valid certificate:

```haskell
newtype PerasCert = PerasCert ByteArray
  deriving (Eq, Show, Generic)
  deriving newtype (EncCBOR, DecCBOR)
``` [4](#0-3) 

The `PerasKey` used in the call site is also a stub with no key material:

```haskell
data PerasKey = PerasKey
  deriving (Eq, Show, Generic, NoThunks)
``` [5](#0-4)

### Impact Explanation

The Peras certificate embedded in a `DijkstraBlockBody` is intended to represent a quorum certificate from the Peras finality committee, attesting that a specific block has been finalized. Because `validatePerasCert` always returns `True`, any block producer can embed an entirely fabricated `PerasCert` (arbitrary bytes) in a block body. All honest nodes running this code will accept the block as valid through the `BBODY` rule without any cryptographic verification of the certificate's authenticity or the claimed quorum.

This means honest nodes accept blocks containing invalid (forged) Peras certificates — matching the allowed impact: **Critical. Honest nodes accept an invalid block or transaction causing permanent ledger divergence requiring a hard fork.** If a future version of the code or an external component (e.g., a Peras-aware light client or finality gadget) performs real certificate validation, it will disagree with nodes running this stub, causing a deterministic split in what is considered a valid chain.

---

### Likelihood Explanation

The entry path requires only that an attacker be a block producer — a role achievable by any stake pool operator, including one with a very small stake. No governance majority, leaked key, or privileged access is needed. The attacker simply constructs a `DijkstraBlockBody` with a `SJust (PerasCert <arbitrary_bytes>)` in the `dbbrPerasCert` field and produces a block. The CBOR encoding/decoding for `PerasCert` imposes no structural constraints beyond being a valid byte array. [6](#0-5) 

---

### Recommendation

Before activating the Dijkstra era on any network, replace the stub `validatePerasCert _ _ _ = True` with the real cryptographic verification from `cardano-base` once it is available. Until then, the Dijkstra era should not be enabled on any network where Peras certificate integrity is a security assumption. Additionally, a compile-time or CI guard (e.g., a `{-# WARNING #-}` pragma similar to the existing `boom` helper) should be added to `validatePerasCert` to prevent accidental deployment of the stub. [7](#0-6) 

---

### Proof of Concept

1. Construct a `DijkstraBlockBody` with `dbbrPerasCert = SJust (PerasCert <random_bytes>)`.
2. Produce a valid Dijkstra-era block using this body (standard block production, no special privileges beyond being a slot leader).
3. Submit the block to the network.
4. All nodes evaluate `dijkstraBbodyTransition`, reach the `perasCertBlockBodyL` check, call `validatePerasCert nonce PerasKey cert`, receive `True`, and accept the block — despite the certificate being cryptographically meaningless.
5. The `PerasCertValidationFailed` failure is never raised. [8](#0-7)
