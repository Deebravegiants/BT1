### Title
Caller Identity Not Bound to `verify_foreign_transaction` Requests — Any Unprivileged Account Can Obtain MPC Threshold Signatures for Arbitrary Foreign-Chain Transactions - (File: crates/contract/src/lib.rs)

### Summary

`verify_foreign_transaction` discards the caller's account ID and does not embed it into the `VerifyForeignTransactionRequest` key. Because the request key is caller-agnostic, any unprivileged NEAR account can submit a `verify_foreign_transaction` call for any foreign-chain transaction ID, cause the MPC network to produce a threshold signature under the shared ForeignTx root key, and receive that signature — without being the legitimate bridge user who initiated the foreign-chain transaction.

### Finding Description

`sign()` and `request_app_private_key()` both bind the caller's identity into the request by passing `&predecessor` to `SignatureRequest::new` and `CKDRequest::new` respectively, making the derived signing key unique per caller. `verify_foreign_transaction` does not follow this pattern.

In `verify_foreign_transaction`, `check_request_preconditions` returns `(DomainConfig, AccountId)`, but the return value is entirely discarded:

```rust
// lib.rs:526-531
self.check_request_preconditions(
    request.domain_id,
    DomainPurpose::ForeignTx,
    Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
    MINIMUM_SIGN_REQUEST_DEPOSIT,
);
// predecessor is never captured
```

The request is then converted via `args_into_verify_foreign_tx_request`, which produces a `VerifyForeignTransactionRequest` containing only `(request, domain_id, payload_version)` — no caller account ID, no tweak:

```rust
// dto_mapping.rs:840-848
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
```

The `VerifyForeignTransactionRequest` struct itself has no identity field:

```rust
// foreign_chain.rs:124-128
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
```

Contrast with `SignatureRequest`, which includes a `tweak` derived from `(predecessor_id, path)`. The ForeignTx path has no such binding.

Additionally, `respond_verify_foreign_tx` verifies the signature against the **root public key**, not a per-caller derived key:

```rust
// lib.rs:728-734
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,   // root key — no tweak applied
)
.is_ok()
```

The design document (`docs/foreign-chain-transactions.md`) explicitly planned for a `derivation_path` in `VerifyForeignTransactionRequestArgs` and a `tweak` in `VerifyForeignTransactionRequest`, but neither was implemented in the production structs. The contract's own test acknowledges the caller-agnostic nature of the request key:

```
// lib.rs:3242-3243
// And: caller bob submits the identical request — a different account would today
// be blocked from receiving a response by alice's submission.
```

### Impact Explanation

Any unprivileged NEAR account can call `verify_foreign_transaction` for any foreign-chain transaction ID (e.g., a Bitcoin `tx_id` belonging to another user's bridge deposit). The MPC network will verify the transaction on-chain and produce a threshold signature under the shared ForeignTx root key. The attacker receives this signature via the yield/resume callback.

Because the ForeignTx domain signs with the root key (no per-caller derivation), the resulting signature is valid for any downstream bridge contract that trusts the MPC ForeignTx domain key. An attacker who front-runs the legitimate bridge user can:

1. Obtain the MPC threshold signature before the legitimate user.
2. Submit the signature to the bridge contract to claim the bridge output, potentially with attacker-controlled parameters (e.g., zero slippage, attacker-specified destination if the bridge contract accepts caller-supplied routing data alongside the MPC proof).
3. Cause the legitimate user's pending yield to also resolve (fan-out), but by then the bridge output may already be consumed, resulting in a failed or griefed redemption.

This constitutes a **participant/attestation authorization bypass that causes invalid bridge execution**, matching the High impact tier.

### Likelihood Explanation

The entry point is a public, payable contract method requiring only 1 yoctoNEAR deposit and sufficient gas. No privileged role, threshold collusion, or TEE access is needed. Any NEAR account can observe a pending foreign-chain transaction (e.g., by watching the Bitcoin mempool or block explorer) and race to call `verify_foreign_transaction` before the legitimate bridge user. The attack is straightforward and requires no special knowledge beyond the target `tx_id`.

### Recommendation

Bind the caller's account ID into the `VerifyForeignTransactionRequest` key, mirroring the pattern used by `sign()`:

1. Capture the `predecessor` returned by `check_request_preconditions`.
2. Add a `predecessor_account_id: AccountId` (or a `tweak: Tweak` derived from `(predecessor_id, derivation_path)`) to both `VerifyForeignTransactionRequestArgs` and `VerifyForeignTransactionRequest`.
3. In `respond_verify_foreign_tx`, verify the signature against the **derived** key (using the stored tweak), not the root key — consistent with how `respond` handles `SignatureRequest`.

This ensures that only the account that submitted the request can receive the resulting signature, and that each caller's ForeignTx key is distinct, eliminating the front-running vector.

### Proof of Concept

```rust
// Any NEAR account can call verify_foreign_transaction for a victim's bridge deposit.
// The attacker observes tx_id = [7u8; 32] on Bitcoin belonging to alice.near.

let attacker = AccountId::from_str("attacker.near").unwrap();
testing_env!(
    VMContextBuilder::new()
        .signer_account_id(attacker.clone())
        .predecessor_account_id(attacker)
        .attached_deposit(NearToken::from_yoctonear(1))
        .build()
);

// Attacker submits alice's tx_id — no ownership check exists.
contract.verify_foreign_transaction(VerifyForeignTransactionRequestArgs {
    domain_id: DomainId::default().0.into(),
    payload_version: ForeignTxPayloadVersion::V1,
    request: dtos::ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
        tx_id: [7u8; 32].into(),   // alice's deposit tx
        confirmations: 2.into(),
        extractors: vec![BitcoinExtractor::BlockHash],
    }),
});
// MPC nodes verify the tx and call respond_verify_foreign_tx.
// Attacker's yield resolves with a valid root-key signature.
// Attacker can now submit this signature to the bridge contract
// before alice.near does, claiming the bridge output.
```

The existing unit test `verify_foreign_transaction__should_queue_duplicates_from_different_callers` (lib.rs:3208–3298) already demonstrates that two different callers submitting the same request both receive the MPC signature — confirming the caller-agnostic request key and the absence of any ownership enforcement.     pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }

    #[test]
    fn verify_foreign_transaction__should_queue_duplicates_from_different_callers() {
        // Given: two different callers will submit the same foreign-tx verification request.
        let mut rng = rand::rngs::StdRng::from_seed([42u8; 32]);
        let (context, mut contract, secret_key) =
            basic_setup_with_protocol(Protocol::CaitSith, DomainPurpose::ForeignTx, &mut rng);
        register_supported_chains(&mut contract, [dtos::ForeignChain::Bitcoin]);
        let SharedSecretKey::Secp256k1(secret_key) = secret_key else {
            unreachable!();
        };

        let request_args = VerifyForeignTransactionRequestArgs {
            domain_id: DomainId::default().0.into(),
            payload_version: ForeignTxPayloadVersion::V1,
            request: dtos::ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
                tx_id: [7u8; 32].into(),
                confirmations: 2.into(),
                extractors: vec![BitcoinExtractor::BlockHash],
            }),
        };
        let request = args_into_verify_foreign_tx_request(request_args.clone());

        // When: caller alice submits the request.
        let alice = AccountId::from_str("alice.near").unwrap();
        testing_env!(
            VMContextBuilder::new()
                .signer_account_id(alice.clone())
                .predecessor_account_id(alice)
                .current_account_id(context.current_account_id.clone())
                .attached_deposit(NearToken::from_yoctonear(1))
                .build()
        );
        contract.verify_foreign_transaction(request_args.clone());

        // And: caller bob submits the identical request — a different account would today
        // be blocked from receiving a response by alice's submission.
        let bob = AccountId::from_str("bob.near").unwrap();
        testing_env!(
            VMContextBuilder::new()
                .signer_account_id(bob.clone())
                .predecessor_account_id(bob)
                .current_account_id(context.current_account_id.clone())
                .attached_deposit(NearToken::from_yoctonear(1))
                .build()
        );
        contract.verify_foreign_transaction(request_args);

        // Then: both yields are queued under the single (caller-agnostic) request key.
        assert_eq!(
            contract
                .pending_verify_foreign_tx_requests
                .get(&request)
                .map(|q| q.len()),
            Some(2),
            "duplicate foreign-tx requests from different callers should fan out",
        );
