### Title
Zero rsETH Minted for Non-Zero Deposit Due to Integer Division Truncation - (`contracts/LRTDepositPool.sol`)

---

### Summary

`getRsETHAmountToMint` uses integer division that can truncate to zero for small deposits when `rsETHPrice` has appreciated above `1e18`. `_beforeDeposit` has no guard against a zero mint result, and the only slippage check (`rsethAmountToMint < minRSETHAmountExpected`) is trivially bypassed by passing `minRSETHAmountExpected = 0`. The deposit then succeeds: the asset is transferred in, and zero rsETH is minted.

---

### Finding Description

**Root cause — `getRsETHAmountToMint`:**

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [1](#0-0) 

When `amount * assetPrice < rsETHPrice`, Solidity integer division truncates the result to `0`.

**`_beforeDeposit` has no zero-mint guard:**

```solidity
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [2](#0-1) 

If `rsethAmountToMint == 0` and `minRSETHAmountExpected == 0`, the condition `0 < 0` is false — no revert occurs.

**The only other deposit-size guard:**

```solidity
if (depositAmount == 0 || depositAmount < minAmountToDeposit) {
    revert InvalidAmountToDeposit();
}
``` [3](#0-2) 

`minAmountToDeposit` is a storage variable that defaults to `0`. When it is `0`, any `depositAmount >= 1` passes this check.

**`depositAsset` then transfers the asset and mints zero:**

```solidity
IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
_mintRsETH(rsethAmountToMint);  // mints 0
``` [4](#0-3) 

---

### Impact Explanation

The depositor's asset is transferred into the pool but zero rsETH is minted. The depositor receives no share of the protocol's yield-bearing collateral. The protocol retains the collateral without issuing a corresponding claim token. This matches the scoped impact: **Low — Contract fails to deliver promised returns, but doesn't lose value**.

---

### Likelihood Explanation

- `rsETHPrice` starts at `1e18` and monotonically increases as rewards accrue. After any meaningful protocol operation, `rsETHPrice > 1e18` is the normal state.
- With `rsETHPrice = 1.05e18` (5% appreciation) and `assetPrice = 1e18`, depositing `1 wei` gives `(1 * 1e18) / 1.05e18 = 0`.
- `minAmountToDeposit` defaults to `0` and may not be set by operators.
- Any caller who passes `minRSETHAmountExpected = 0` (the default for automated integrations or naive users) is vulnerable.
- The oracle validation in `updatePriceOracleForValidated` bounds asset prices to `[1e16, 1e19]`, but this does not prevent the truncation — it only constrains the ratio. [5](#0-4) 

---

### Recommendation

Add an explicit zero-mint guard in `_beforeDeposit`:

```solidity
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

if (rsethAmountToMint == 0) {
    revert MinimumAmountToReceiveNotMet();
}

if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

Alternatively, enforce a non-zero `minAmountToDeposit` that is large enough to guarantee at least 1 wei of rsETH is minted at any realistic rsETHPrice.

---

### Proof of Concept

```solidity
// Preconditions:
//   - rsETHPrice = 1.05e18 (5% appreciation, realistic after rewards)
//   - stETH assetPrice = 1e18
//   - minAmountToDeposit = 0 (default)
//   - minRSETHAmountExpected = 0 (caller passes 0)

// getRsETHAmountToMint(stETH, 1):
//   = (1 * 1e18) / 1.05e18
//   = 0  (integer truncation)

// _beforeDeposit:
//   depositAmount (1) != 0 and >= minAmountToDeposit (0) → passes
//   rsethAmountToMint (0) < minRSETHAmountExpected (0) → false → no revert

// depositAsset:
//   safeTransferFrom(user, pool, 1 wei stETH)  ← asset leaves user
//   mint(user, 0)                               ← zero rsETH minted

// Assert: user's stETH balance decreased by 1, rsETH balance unchanged.
```
