### Title
Missing `payload_hash`-to-`request` Binding in `respond_verify_foreign_tx` Allows Cross-Request Signature Replay - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that the submitted ECDSA signature is valid over `response.payload_hash`, but never verifies that `response.payload_hash` is the hash of a `ForeignTxSignPayload` that encodes the original `request`. A single Byzantine leader node — already below the signing threshold — can reuse a legitimately-obtained threshold signature from one pending request to resolve a completely different pending request, delivering a forged attestation to the caller.

### Finding Description

In `respond_verify_foreign_tx` the contract performs two checks:

1. The caller is an attested participant.
2. The ECDSA signature in `response` is valid over `response.payload_hash` against the domain root public key. [1](#0-0) 

What is **not** checked: that `response.payload_hash` is the SHA-256 Borsh hash of a `ForeignTxSignPayload` whose embedded `ForeignChainRpcRequest` matches the `request` parameter that was used to look up the pending yield.

The `ForeignTxSignPayload` is defined as: [2](#0-1) 

The hash commits to both the original `ForeignChainRpcRequest` **and** the extracted values. The contract holds the original `request` but never checks that it is embedded in `payload_hash`.

The node-side leader constructs `payload_hash` from the result of `execute_foreign_chain_request` and then runs the threshold signing protocol: [3](#0-2) 

Once the leader has a valid threshold signature `sig_B` over `hash_B` (legitimately produced for pending Request B), nothing in the contract prevents the leader from submitting `respond_verify_foreign_tx(request = A, response = {payload_hash: hash_B, signature: sig_B})`. The contract will:

- Find Request A in `pending_verify_foreign_tx_requests` ✓
- Verify `sig_B` over `hash_B` against the root key ✓
- Resolve Request A's yield with `{payload_hash: hash_B, signature: sig_B}` ✓ [4](#0-3) 

The caller of Request A receives a `VerifyForeignTransactionResponse` whose `payload_hash` corresponds to a completely different foreign-chain transaction.

### Impact Explanation

The caller receives a signed attestation that does not correspond to the transaction they requested verification for. If the caller's application uses the SDK's `ForeignChainSignatureVerifier::verify_signature`, it will detect the mismatch: [5](#0-4) 

However, the contract itself has already consumed the pending request and resolved the yield. The caller's deposit is lost and the request must be resubmitted. More critically, any caller application that checks only "is the signature valid over `payload_hash`?" (without reconstructing the expected hash from the original request and observed values) will accept the forged attestation as proof that their transaction was verified — enabling invalid bridge execution or double-spend conditions on the foreign chain side.

### Likelihood Explanation

A single Byzantine leader node — strictly below the signing threshold — can execute this attack. The leader participates in the normal threshold signing protocol for Request B (no collusion required beyond its own role as leader), obtains a valid `(hash_B, sig_B)` pair, and then submits it as the response for Request A. No special access, no key leakage, and no threshold-or-above collusion is needed. The only prerequisite is that two requests are pending simultaneously, which is the normal operating condition of the network.

### Recommendation

The contract should verify that `response.payload_hash` is consistent with the original `request`. Since the contract does not have the extracted `values`, it cannot recompute the full hash. Two mitigations are possible:

1. **Include the request hash in the pending-request key and verify it against `payload_hash`**: Store a commitment to the `ForeignChainRpcRequest` alongside the pending yield, and require that `payload_hash` begins with (or is derived from) a hash of the original request. This is a partial check but closes the cross-request replay.

2. **Have nodes include the extracted values in the response and verify on-chain**: Return `ForeignTxSignPayload` (not just `payload_hash`) in the response, recompute the hash on-chain from `(request, values)`, and verify the signature against the recomputed hash. This is the complete fix but increases response size.

At minimum, add a test that submits a valid signature for Request B as the response for Request A and asserts rejection.

### Proof of Concept

```
1. Alice submits verify_foreign_transaction(Bitcoin tx_id = X) → Request A queued.
2. Bob  submits verify_foreign_transaction(Bitcoin tx_id = Y) → Request B queued.
3. Byzantine leader runs the threshold signing protocol for Request B:
     hash_B = SHA-256(borsh(ForeignTxSignPayloadV1 { request: B, values: [block_hash_Y] }))
     sig_B  = threshold_sign(hash_B)   // legitimate, requires no collusion
4. Leader calls respond_verify_foreign_tx(
       request  = A,                              // matches Alice's pending yield
       response = { payload_hash: hash_B,         // hash of B's payload
                    signature:    sig_B }          // valid sig over hash_B
   )
5. Contract checks:
     - Request A exists in pending map          → ✓
     - verify_ecdsa(sig_B, hash_B, root_pk)    → ✓  (signature IS valid)
     - payload_hash encodes request A?          → NOT CHECKED
6. Alice's yield is resumed with { payload_hash: hash_B, signature: sig_B }.
7. Alice's application receives an attestation for tx_id Y, not X.
   If it skips ForeignChainSignatureVerifier::verify_signature it accepts the forgery.
``` [6](#0-5) [7](#0-6)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
