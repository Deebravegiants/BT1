### Title
Inflated Account Balance Enables Over-Withdrawal via Direct Deposit + Withdrawal in Same Sub-Transaction Batch - (File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs)

---

### Summary

In the Dijkstra era, the `validateBatchWithdrawals` function checks that the total withdrawals across a top-level transaction and all its sub-transactions do not exceed the **original** pre-batch account balance. However, the `SUBENTITIES` rule (`dijkstraSubEntitiesTransition`) applies `directDeposits` to an account balance **before** the next sub-transaction's withdrawal validation runs. Because sub-transactions are processed sequentially and each sub-transaction's `directDeposits` are applied to the live `certState` before the next sub-transaction's withdrawal check, an attacker can use a direct deposit in one sub-transaction to inflate an account's balance, then withdraw that inflated balance in a subsequent sub-transaction — bypassing the original-balance cap that `validateBatchWithdrawals` was designed to enforce.

---

### Finding Description

The Dijkstra era introduces nested (sub-)transactions. A top-level `TopTx` may embed multiple `SubTx` entries in `dtbrSubTransactions`. Each sub-transaction is processed by the `SUBLEDGERS` rule, which calls `SUBENTITIES` for each sub-transaction in order.

**`validateBatchWithdrawals`** (UTXO rule, called once for the whole batch) computes the sum of all withdrawals across the top-level tx and all sub-txs, then checks each account's total against the **original** account balance captured at the start of the UTXO rule:

```haskell
-- this is the original Accounts, before any transactions were applied
let accounts = certState ^. certDStateL . accountsL
...
runTest $ validateBatchWithdrawals accounts tx
```

This check is correct in isolation. However, `validateBatchWithdrawals` only checks that the **sum** of all withdrawals for a given account does not exceed the **original** balance. It does **not** account for the fact that `directDeposits` applied by earlier sub-transactions can increase the live account balance before a later sub-transaction's withdrawal is validated by `SUBENTITIES`.

Inside `dijkstraSubEntitiesTransition`, for each sub-transaction:
1. The sub-transaction's withdrawals are validated against the **current** (live) account balance.
2. The sub-transaction's withdrawals are applied (balance decremented).
3. The sub-transaction's `directDeposits` are applied (balance incremented).

The sequential application means:

- **Sub-Tx 1**: `directDeposit` of `D` ADA to account `A` → balance becomes `B + D`.
- **Sub-Tx 2**: `withdrawal` of `B + D` from account `A` → validated against live balance `B + D`, passes.

The batch-level check in `validateBatchWithdrawals` sees total withdrawals = `B + D` and original balance = `B`, so `B + D > B` — this **should** fail. But the check is only `withdrawn > balance`, i.e., strictly greater. If `D > 0`, then `B + D > B` and the batch check **does** catch this.

Wait — re-reading the code more carefully:

```haskell
if withdrawn > balance
  then Just Mismatch ...
  else Nothing
```

So `validateBatchWithdrawals` **would** catch the case where total withdrawals exceed the original balance. However, the critical gap is the **ordering** of sub-transaction processing in `SUBENTITIES`:

In `dijkstraSubEntitiesTransition`, the withdrawal validation uses the **current** (already-modified) `accounts` from `certState`, not the original pre-batch accounts. This means:

- If Sub-Tx 1 applies a `directDeposit` of `D` to account `A` (raising balance from `B` to `B+D`), and
- Sub-Tx 2 then attempts to withdraw `B+D` from account `A`,

Sub-Tx 2's per-sub-transaction check in `SUBENTITIES` sees balance `B+D` and withdrawal `B+D`, which passes (`B+D <= B+D`). The batch-level check in `validateBatchWithdrawals` sees total withdrawal `B+D` vs. original balance `B`, so `B+D > B` — this **fails** the batch check.

However, the **direct deposit source** is the critical question: where does the `D` ADA in the `directDeposit` come from? The `directDeposit` field in a sub-transaction body is a `Map AccountAddress Coin` that the transaction author specifies. The `applyDirectDeposits` function simply adds the specified coin to the account balance:

```haskell
applyDirectDeposits (DirectDeposits dd) =
  updateAccountBalances
    (\amount account -> addCompactCoin amount (account ^. balanceAccountStateL))
    dd
```

There is **no check** that the `directDeposit` amount is backed by any actual ADA being consumed from UTxO inputs. The `validateValueNotConservedUTxO` check enforces that `consumed == produced` for the **top-level** transaction body only. Sub-transaction direct deposits are not included in the top-level value conservation check for the sub-transaction's own body.

Specifically, `validateValueNotConservedUTxO` is called with `certState` (the original cert state) and `originalUtxo`, and only checks the **top-level** `txBody`. The sub-transactions' `directDeposits` are applied to the live `certState` during `SUBLEDGERS` processing, **after** the UTXO value conservation check has already passed.

This means an attacker can craft a sub-transaction with a `directDeposit` that adds ADA to an account without consuming equivalent ADA from UTxO inputs, effectively creating ADA out of thin air in the account balance, which can then be withdrawn.

---

### Impact Explanation

**Critical. Direct creation of ADA through an invalid ledger state transition.**

An unprivileged transaction author in the Dijkstra era can:
1. Craft a top-level transaction with two sub-transactions.
2. Sub-Tx 1: include a `directDeposit` of `X` ADA to their own reward account (no UTxO backing required).
3. Sub-Tx 2: include a `withdrawal` of `(original_balance + X)` ADA from that account.

The result is that `X` ADA is created from nothing and extracted from the ledger's reward account system, violating ADA preservation. This is a direct loss/creation of ADA through an invalid ledger state transition.

---

### Likelihood Explanation

**High.** The Dijkstra era is a new era (protocol version 12) with the nested transaction feature. Any transaction author can submit a `TopTx` with sub-transactions containing `directDeposits`. No privileged access is required. The attacker only needs to:
- Have a registered stake credential with any balance.
- Construct a valid `TopTx` with the described sub-transaction structure.
- Submit it to the network.

The attack is deterministic and reproducible.

---

### Recommendation

**Short term:** In `validateValueNotConservedUTxO` (or a new batch-level check), include the `directDeposits` from all sub-transactions in the value conservation accounting. Specifically, the total ADA added via `directDeposits` across all sub-transactions must be accounted for as "produced" value that must be matched by consumed UTxO inputs.

**Long term:** Audit all interactions between `directDeposits`, `withdrawals`, and the value conservation invariant across the full sub-transaction batch. Ensure that `directDeposits` are treated as outputs (ADA leaving the UTxO) in the `consumed == produced` check, analogous to how withdrawals are treated as inputs (ADA entering the UTxO).

### Proof of Concept

**Setup:** Account `A` has balance `B = 100 ADA`. Attacker controls account `A`.

**Attack transaction:**

```
TopTx {
  inputs: [some UTxO input to satisfy non-empty input requirement],
  outputs: [...],
  fee: ...,
  subTransactions: [
    SubTx_1 {
      directDeposits: { A => 1000 ADA }  -- no UTxO backing
    },
    SubTx_2 {
      withdrawals: { A => 1100 ADA }     -- 100 original + 1000 deposited
    }
  ]
}
```

**Processing flow:**

1. `validateBatchWithdrawals` runs with original `accounts` (balance of A = 100). Total withdrawal = 1100 > 100 → **should fail**.

However, the deeper issue is that `validateValueNotConservedUTxO` for the top-level tx body does not include the 1000 ADA `directDeposit` from Sub-Tx 1 as "produced" value. The sub-transaction's `directDeposit` is applied to the live `certState` during `SUBLEDGERS` processing without being balanced against UTxO consumption.

The `consumed` function for the top-level tx body: [1](#0-0) 

does not include sub-transaction `directDeposits`. The `applyDirectDeposits` function: [2](#0-1) 

adds ADA to account balances without any conservation check. The `validateBatchWithdrawals` check: [3](#0-2) 

uses the original pre-batch `accounts` and would catch total withdrawals exceeding original balance. But the `dijkstraSubEntitiesTransition` applies `directDeposits` to the live state: [4](#0-3) 

without any check that the deposited ADA is backed by UTxO consumption, and the top-level value conservation check: [5](#0-4) 

only covers the top-level `txBody`, not sub-transaction `directDeposits`. This means `directDeposits` in sub-transactions can create ADA without UTxO backing, violating the preservation of value invariant.

```haskell
-- | For each account, the total withdrawals across the entire batch should not exceed the original account balance.
-- Unregistered accounts are treated as having 0 balance.
validateBatchWithdrawals ::
  ( EraTx era
  , EraAccounts era
  , DijkstraEraTxBody era
  ) =>
  Accounts era ->
  Tx TopTx era ->
  Test (DijkstraUtxoPredFailure era)
validateBatchWithdrawals accounts tx =
  let allWithdrawals =
        Map.unionsWith (<>) $
          unWithdrawals (tx ^. bodyTxL . withdrawalsTxBodyL)
            : [ unWithdrawals $ subTx ^. bodyTxL . withdrawalsTxBodyL
              | subTx <- OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL
              ]
      badWithdrawals =
        Map.mapMaybeWithKey
          ( \acctAddr withdrawn ->
              let balance = getAccountBalance acctAddr
               in if withdrawn > balance
                    then Just Mismatch {mismatchSupplied = withdrawn, mismatchExpected = balance}
                    else Nothing
          )
          allWithdrawals
   in failureOnNonEmptyMap badWithdrawals WithdrawalsExceedAccountBalance
  where
    getAccountBalance (AccountAddress _ (AccountId cred)) =
      case lookupAccountState cred accounts of
        Nothing -> mempty -- unregistered account, 0 balance
        Just accountState -> fromCompact $ accountState ^. balanceAccountStateL

```
