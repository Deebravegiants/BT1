### Title
Stale Votes from Removed Participants Count Toward TEE Image-Hash Governance Thresholds — (File: `crates/contract/src/lib.rs`)

---

### Summary

After a resharing removes participants from the active set, their previously cast votes for TEE governance actions (`vote_code_hash`, `vote_add_launcher_hash`, `vote_remove_launcher_hash`, `vote_add_os_measurement`, `vote_remove_os_measurement`) persist in contract state and are counted toward the threshold. The cleanup of stale votes is performed only via a detached, fire-and-forget promise that can fail silently. Unlike `vote_update`, which explicitly re-filters stored votes against the current participant set before comparing to the threshold, none of the TEE governance voting functions perform this filtering. A Byzantine participant who is removed via resharing can therefore have a pre-cast vote for a malicious MPC Docker image hash persist and contribute to reaching the whitelisting threshold with fewer current-participant votes than required.

---

### Finding Description

**Root cause — raw vote count, no current-participant filter:**

`vote_code_hash` (and the analogous launcher/measurement variants) calls `self.tee_state.vote(code_hash, &participant)` and compares the returned integer directly to `self.threshold()?.value()`: [1](#0-0) 

`TeeState::vote` simply delegates to `CodeHashesVotes::vote` and returns the raw stored count: [2](#0-1) 

The stale-vote cleanup is a separate method that is only invoked via a detached promise spawned after resharing: [3](#0-2) 

That promise is scheduled with `detach()` and a fixed gas budget; if it runs out of gas or is otherwise dropped, the stale votes remain in `self.votes`, `self.launcher_votes`, and `self.measurement_votes` indefinitely: [4](#0-3) 

**Contrast — `vote_update` explicitly filters for current participants:**

The contract's own comment in `vote_update` acknowledges the cleanup promise can fail and compensates by re-filtering stored votes against the live participant set before comparing to the threshold: [5](#0-4) 

No equivalent filter exists in `vote_code_hash`, `vote_add_launcher_hash`, `vote_remove_launcher_hash`, `vote_add_os_measurement`, or `vote_remove_os_measurement`.

---

### Impact Explanation

The TEE image-hash whitelist (`allowed_docker_image_hashes`) is the gate that determines which MPC node binaries are permitted to submit valid attestations and therefore participate in threshold signing. Whitelisting a malicious image hash allows a compromised node to:

1. Submit a valid TEE attestation accepted by `submit_participant_info`.
2. Pass `assert_caller_is_attested_participant_and_protocol_active` in `respond`, `vote_pk`, `vote_reshared`, etc.
3. Participate in threshold signing or key-generation rounds, enabling unauthorized signature issuance or key-share exfiltration.

This maps to **Critical — bypass of threshold-signature requirements / unauthorized access to MPC key shares or signing capability**.

---

### Likelihood Explanation

The preconditions are:

1. A Byzantine participant (strictly below the signing threshold, so no collusion assumption is needed) casts a vote for a malicious code hash before being removed.
2. A resharing removes that participant.
3. The `CLEAN_TEE_STATUS` detached promise fails (e.g., gas budget exhausted) or has not yet executed.
4. One or more remaining current participants vote for the same malicious hash; the stale vote(s) push the total over the threshold.

Step 3 is the critical enabler. The gas budget is a configurable constant (`clean_tee_status_tera_gas`); if it is set too low for the actual participant-set size, the cleanup silently fails and stale votes persist for the lifetime of the contract. Steps 1–2 are normal protocol events (participant rotation via resharing is expected). Step 4 requires only a single colluding or deceived current participant.

---

### Recommendation

Apply the same pattern used in `vote_update`: after recording the new vote, re-derive the valid vote count by iterating over the **current** participant set and counting only those whose stored vote matches the target hash. This makes correctness independent of whether the cleanup promise succeeded:

```rust
// In vote_code_hash, replace:
let votes = self.tee_state.vote(code_hash, &participant);
if votes >= self.threshold()?.value() { ... }

// With:
self.tee_state.vote(code_hash, &participant);
let valid_votes = threshold_parameters
    .participants()
    .participants()
    .iter()
    .filter(|(_, _, info)| {
        self.tee_state.votes.participant_voted_for(&info, &code_hash)
    })
    .count() as u64;
if valid_votes >= self.threshold()?.value() { ... }
```

Apply the same fix to `vote_add_launcher_hash`, `vote_remove_launcher_hash`, `vote_add_os_measurement`, and `vote_remove_os_measurement`.

### Proof of Concept

**Setup:** 3 participants A, B, C; governance threshold = 2.

1. **A (Byzantine) calls `vote_code_hash(malicious_hash)`.**
   `tee_state.votes` now contains `{A → malicious_hash}`. Count = 1 < 2; no whitelisting yet.

2. **Resharing removes A; new participant set = {B, C}.**
   `vote_reshared` completes and schedules `CLEAN_TEE_STATUS` as a detached promise with a fixed gas budget.

3. **`CLEAN_TEE_STATUS` promise fails** (gas exhausted or dropped).
   `tee_state.votes` still contains `{A → malicious_hash}`.

4. **B calls `vote_code_hash(malicious_hash)`.**
   `AuthenticatedParticipantId::new(threshold_parameters.participants())?` succeeds because B is a current participant.
   `tee_state.vote(malicious_hash, B)` stores B's vote and returns count = 2 (A's stale + B's new).
   `2 >= threshold (2)` → `whitelist_tee_proposal(malicious_hash, ...)` executes. [6](#0-5) 

5. **Malicious image is now whitelisted.** A node running the malicious binary can call `submit_participant_info` with a valid attestation, pass `is_caller_an_attested_participant`, and participate in `respond` / `vote_pk` / `vote_reshared`, enabling unauthorized signing or key-share exfiltration. [7](#0-6)

    pub fn vote_code_hash(&mut self, code_hash: NodeImageHash) -> Result<(), Error> {
        log!(
            "vote_code_hash: signer={}, code_hash={:?}",
            env::signer_account_id(),
            code_hash,
        );
        self.voter_or_panic();

        let threshold_parameters = self.protocol_state.threshold_parameters_or_panic();

        let participant = AuthenticatedParticipantId::new(threshold_parameters.participants())?;
        let votes = self.tee_state.vote(code_hash, &participant);

        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);

        // If the vote threshold is met and the new Docker hash is allowed by the TEE's RTMR3,
        // update the state
        if votes >= self.threshold()?.value() {
            self.tee_state
                .whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
        }

        Ok(())
    }

        &mut self,
        tee_proposal: NodeImageHash,
        tee_upgrade_deadline_duration: Duration,
    ) {
        self.votes.clear_votes();
        // Add compose hashes for the new MPC image across all allowed launcher images
        self.allowed_launcher_images
            .add_mpc_image_compose_hashes(&tee_proposal);
        self.allowed_docker_image_hashes
            .insert(tee_proposal, tee_upgrade_deadline_duration);
    }
