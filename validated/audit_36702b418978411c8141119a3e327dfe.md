### Title
Event-Based (No Cryptographic) Output Verification for Legacy CKD `AppPublicKey` Variant Allows Single Byzantine Participant to Deliver Fabricated Confidential Key Derivation Output - (File: crates/contract/src/lib.rs)

### Summary

The `respond_ckd` function in the MPC contract applies a cryptographic pairing check (`ckd_output_check`) only when the request uses the `AppPublicKeyPV` variant. For the legacy `AppPublicKey` variant — the default and most widely documented path — **no cryptographic verification of the response is performed**. A single malicious attested participant (strictly below the signing threshold) can call `respond_ckd` with an arbitrary fabricated `CKDResponse`, and the contract will accept and deliver it to the requesting user. This is the direct structural analog of the "event-based" authentication vulnerability in the external report: authorization is gated only on caller identity (an event/state check), not on cryptographic binding between the response and the actual MPC-derived key material.

### Finding Description

In `respond_ckd`, the response verification is conditional on the `app_public_key` variant stored in the pending request:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` enforces the BLS12-381 pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, cryptographically binding the response to the MPC network's master key and the user's app public key: [2](#0-1) 

For `AppPublicKey`, the arm is an empty no-op. The only guards that run are:
1. `assert_caller_is_signer()` — signer == predecessor
2. `assert_caller_is_attested_participant_and_protocol_active()` — caller is an attested participant
3. `accept_requests` flag check [3](#0-2) 

These are purely identity/state checks — the "event-based" pattern. They confirm *who* is calling, not *what* was computed. After passing them, any `CKDResponse { big_y, big_c }` with arbitrary byte values is serialized and delivered to the user via `resolve_yields_for`. [4](#0-3) 

The `AppPublicKey` variant is the legacy default and is the primary example in the contract README and the `ckd-example-cli`: [5](#0-4) 

**Attack path:**

1. User calls `request_app_private_key` with `AppPublicKey` variant (a single BLS12-381 G1 point as the encryption target). The request is stored in `pending_ckd_requests`.
2. A single malicious attested participant (one node, below threshold) calls `respond_ckd` with a fabricated `CKDResponse { big_y: attacker_chosen_G1, big_c: attacker_chosen_G1 }` before the honest coordinator submits the real response.
3. The contract passes all identity checks, skips the empty `AppPublicKey` arm, and calls `resolve_yields_for`, delivering the fabricated response to the user's pending yield.
4. The user's callback receives the attacker-controlled `big_y` and `big_c` and uses them to derive their application key.

The attacker does not need threshold collusion — a single Byzantine participant suffices.

### Impact Explanation

**Critical.** This is an unauthorized confidential key derivation output delivered without the required participant authorization (threshold MPC computation). The user receives a `CKDResponse` that was not produced by the MPC network's threshold protocol. Concretely:

- **Integrity breach**: The user's derived application key is wrong/attacker-controlled. Any secrets or assets protected by that key are inaccessible or corrupted.
- **Potential confidentiality breach**: If the attacker crafts `big_c` as a known scalar multiple of the user's `app_public_key` (which is public, stored in the request), the resulting decrypted value is predictable to the attacker. The attacker can effectively choose what "derived key" the user obtains, and may be able to learn it.

The `AppPublicKey` variant is the legacy default path, so this affects the majority of production CKD requests.

### Likelihood Explanation

**Medium.** The attacker must be an attested participant in the MPC network — a role that requires on-chain registration and a valid TEE attestation. However, only **one** such participant needs to be malicious (no threshold collusion required). The attack is a simple race: submit a fabricated `respond_ckd` before the honest coordinator. Since the contract resolves the first valid `respond_ckd` call that matches the pending request key, the attacker wins if they act first. The `AppPublicKey` variant is the default and most commonly used path.

### Recommendation

Apply the same `ckd_output_check` pairing verification to the `AppPublicKey` variant. For the legacy single-G1-point variant, a corresponding cryptographic check must be defined and enforced. Until a proper check is available for the legacy variant, consider rejecting new `AppPublicKey` requests or migrating all users to `AppPublicKeyPV`, which already has result-based verification.

The fix mirrors the `AppPublicKeyPV` branch: the response must be cryptographically bound to the MPC network's master public key and the user's app public key before `resolve_yields_for` is called.

### Proof of Concept

The existing unit test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` already demonstrates the vulnerability unintentionally: it passes `CKDResponse { big_y: [1u8; 48], big_c: [2u8; 48] }` — clearly invalid BLS12-381 points — and the contract accepts them without error for the `AppPublicKey` variant: [6](#0-5) 

A malicious participant would:
1. Observe a pending `AppPublicKey` CKD request on-chain.
2. Construct any `CKDResponse` with chosen `big_y` and `big_c` values.
3. Call `respond_ckd(request, fabricated_response)` from their attested participant account.
4. The contract accepts it; the user's yield resolves with the fabricated output.

    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
