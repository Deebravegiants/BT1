### Title
u64 Overflow in P2P Sync Server Block-Number Arithmetic Causes Wrong Block Data Served to Peers - (File: crates/apollo_p2p_sync/src/server/utils.rs)

### Summary

`calculate_block_number` in the P2P sync server computes `query.step * read_blocks_counter` as a native `u64 × u64` multiplication **before** widening to `i128`. Both operands are attacker-controlled `u64` fields from a peer-supplied protobuf `Iteration` message. When their product exceeds `u64::MAX`, the multiplication wraps (release build) or panics (debug build), causing the server to fetch and stream block data for a completely wrong block number.

### Finding Description

`calculate_block_number` is the sole function that maps a peer query to a concrete `BlockNumber` on every iteration of the sync loop:

```rust
// crates/apollo_p2p_sync/src/server/utils.rs  line 15
let blocks_delta: i128 = direction_factor * i128::from(query.step * read_blocks_counter);
//                                                      ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
//                                                      u64 * u64 — overflows before widening
``` [1](#0-0) 

The `Query` struct carries both fields as `u64`:

```rust
pub struct Query {
    pub limit: u64,
    pub step:  u64,   // attacker-supplied
    ...
}
``` [2](#0-1) 

The protobuf wire format maps directly to these fields with no range restriction:

```proto
uint64 limit = 4;
uint64 step  = 5;
``` [3](#0-2) 

The caller iterates `block_counter` from `0..query.limit` and passes each value straight into `calculate_block_number`:

```rust
for block_counter in 0..query.limit {
    let block_number =
        BlockNumber(utils::calculate_block_number(&query, start_block_number, block_counter)?);
    let data_vec = Data::fetch_block_data(block_number, &txn, class_manager_client).await?;
``` [4](#0-3) 

The code itself carries an explicit acknowledgement of the defect:

```rust

``` [5](#0-4) 

### Impact Explanation

**Release build (production):** `u64` wraps silently. Example: `step = 2^63`, `read_blocks_counter = 2` → product wraps to `0` → `blocks_delta = 0` → the server repeatedly fetches and streams data for `start_block` instead of the intended block. The peer receives authoritative-looking headers, state diffs, transactions, classes, and events for the wrong block number. A syncing node that trusts this data will store incorrect chain state.

**Debug build:** Rust's overflow check panics, crashing the `tokio::task::spawn`-ed query handler. Any peer can trigger this with a single crafted query.

The server handles five data types through the same code path (`SignedBlockHeader`, `StateDiffChunk`, `FullTransaction`, class data, `Event`), so all sync protocols are affected. [6](#0-5) 

### Likelihood Explanation

Any network peer can send a `HeadersRequest`, `StateDiffRequest`, `TransactionsRequest`, `ClassesRequest`, or `EventsRequest` with `step` and `limit` chosen to overflow. No authentication or privilege is required. The `TODO` comment confirms the authors are aware the arithmetic is wrong.

### Recommendation

Widen **both** operands before multiplying:

```rust
let blocks_delta: i128 =
    direction_factor * (i128::from(query.step) * i128::from(read_blocks_counter));
```

This keeps the product inside `i128` for all `u64` inputs and removes the overflow entirely.

### Proof of Concept

Send a `HeadersRequest` (or any sync query) with:

```
start_block = 0
direction   = Forward
step        = 9_223_372_036_854_775_808   // 2^63
limit       = 2
```

Iteration 0 (`block_counter = 0`): `0 * step = 0` → serves block 0 ✓  
Iteration 1 (`block_counter = 1`): `1 * step = 2^63` → no overflow, serves block `2^63` → `BlockNotFound` error, query terminates early.

Now use `step = 2^63` and `limit = 3`:

Iteration 2 (`block_counter = 2`): `2 * 2^63 = 2^64` → wraps to `0` in `u64` → `i128::from(0) = 0` → `blocks_delta = 0` → server fetches and streams **block 0 again** as if it were block `2 * step`.

A syncing peer that accepts this response without independent hash verification will record block 0's data at the wrong chain position. [7](#0-6)

pub(crate) fn calculate_block_number(
    query: &Query,
    start_block: u64,
    read_blocks_counter: u64,
) -> Result<u64, P2pSyncServerError> {
    let direction_factor: i128 = match query.direction {
        Direction::Forward => 1,
        Direction::Backward => -1,
    };
    // TODO(shahak): Fix this code.
    let blocks_delta: i128 = direction_factor * i128::from(query.step * read_blocks_counter);
    let block_number: i128 = i128::from(start_block) + blocks_delta;

    u64::try_from(block_number).map_err(|_| P2pSyncServerError::BlockNumberOutOfRange {
        query: query.clone(),
        counter: read_blocks_counter,
    })
}
