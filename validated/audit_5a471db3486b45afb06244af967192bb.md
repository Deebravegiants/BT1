### Title
Forged Foreign-Chain Verification via Caller-Supplied `payload_hash` in `respond_verify_foreign_tx` - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` function in the MPC contract verifies that the submitted signature is valid over the caller-supplied `response.payload_hash`, but **never verifies that `response.payload_hash` was actually computed from the `request` content**. A single Byzantine participant (below the signing threshold) can recycle a valid MPC signature produced for one foreign-chain request and submit it as the response to a completely different pending request, causing the contract to resolve that request with a forged payload hash.

---

### Finding Description

`respond_verify_foreign_tx` performs two independent checks:

1. The signature in `response` is valid over `response.payload_hash` (verified against the MPC domain public key).
2. The `request` argument exists in `pending_verify_foreign_tx_requests` (used as the map key to resolve the yield).

What it **does not** check is whether `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request, values }))` — i.e., whether the hash was actually derived from the specific request being answered.

The signed payload structure is:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,   // ← extracted from the foreign chain

// msg_hash = SHA-256(borsh(ForeignTxSignPayload))
``` [2](#0-1) 

Because `values` (the extracted block hashes, logs, etc.) are not part of the on-chain request and are not included in `VerifyForeignTransactionResponse`, the contract has no way to recompute the expected hash itself — and currently makes no attempt to do so. [3](#0-2) 

The `ForeignChainSignatureVerifier::verify_signature` helper in the SDK **does** perform this cross-check, but it is a client-side library function — the on-chain contract does not call it. [4](#0-3) 

---

### Impact Explanation

A single Byzantine participant (below the signing threshold) can:

1. Observe a legitimately completed `respond_verify_foreign_tx` call for **request A** (e.g., a Bitcoin deposit tx), obtaining the valid pair `(payload_hash_A, sig_A)` from on-chain history.
2. Wait for a user to submit `verify_foreign_transaction(request_B)` for a **different** transaction B (e.g., a larger deposit or a transaction that never actually occurred on the foreign chain).
3. Call `respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: sig_A })`.
4. The contract accepts: `sig_A` is a valid MPC signature over `payload_hash_A`, and `request_B` exists in `pending_verify_foreign_tx_requests`.
5. The yield for `request_B` is resolved with the forged response.

The caller of `verify_foreign_transaction(request_B)` receives `{ payload_hash: payload_hash_A, signature: sig_A }` — a cryptographically valid MPC attestation that actually describes transaction A, not B. Any bridge or application that does not independently recompute the expected `payload_hash` from its own request parameters (using `ForeignChainSignatureVerifier::verify_signature`) will accept this as proof that transaction B was verified, enabling invalid bridge execution or double-spend conditions. [5](#0-4) 

---

### Likelihood Explanation

- **Attacker model**: A single Byzantine participant — explicitly within the allowed scope ("Byzantine participant strictly below the signing threshold"). No threshold collusion is required; the attacker reuses an already-produced MPC signature.
- **Opportunity**: All `respond_verify_foreign_tx` transactions are public on NEAR. Any participant can harvest `(payload_hash, signature)` pairs from past responses.
- **Precondition**: A concurrent or recent pending request for a different transaction must exist. Bridge services that process high volumes of foreign-chain verifications create this condition routinely.
- **Barrier**: The attacker must be an active, attested MPC participant (`assert_caller_is_attested_participant_and_protocol_active`), which limits the pool of potential attackers to the current participant set. [6](#0-5) 

---

### Recommendation

The contract should recompute the expected `payload_hash` from the `request` and the `values` supplied in the response, and reject any response where they do not match. Concretely, extend `VerifyForeignTransactionResponse` to include the `Vec<ExtractedValue>` alongside the hash, then add an on-chain check:

```rust
// Pseudocode for the fix inside respond_verify_foreign_tx
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.values.clone(),   // values now included in response
});
let expected_hash = expected_payload.compute_msg_hash()?;
require!(expected_hash == response.payload_hash, "payload_hash mismatch");
```

This mirrors the check already present in `ForeignChainSignatureVerifier::verify_signature` and closes the gap between the SDK helper and the on-chain enforcement. [7](#0-6) 

---

### Proof of Concept

```
1. Honest participant calls respond_verify_foreign_tx(request_A, {payload_hash_A, sig_A})
   → tx is public on NEAR; attacker records (payload_hash_A, sig_A).

2. User calls verify_foreign_transaction(request_B)
   → request_B enters pending_verify_foreign_tx_requests.

3. Attacker (Byzantine participant) calls:
     respond_verify_foreign_tx(
       request  = request_B,          // valid key → found in map
       response = { payload_hash: payload_hash_A,   // hash of tx A's data
                    signature:    sig_A }            // valid MPC sig over payload_hash_A
     )

4. Contract checks:
   - verify_ecdsa_signature(sig_A, payload_hash_A, mpc_pk) → OK  ✓
   - request_B in pending_verify_foreign_tx_requests         → OK  ✓
   - payload_hash_A == SHA-256(borsh({request_B, values_B})) → NOT CHECKED ✗

5. Yield for request_B resolved with forged response.
   Caller receives (payload_hash_A, sig_A) as "proof" that tx B was verified.
``` [8](#0-7) [9](#0-8)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
