### Title
`sp_catchup_sem` Monopolization via Crafted `NewSignagePointOrEndOfSubSlot` Enables 300s Signage Point Starvation — (`chia/full_node/full_node_api.py`)

### Summary
A single malicious peer can hold `sp_catchup_sem` for up to 300 seconds by advertising a `NewSignagePointOrEndOfSubSlot` with `index_from_challenge=0` and an unknown `prev_challenge_hash`, then responding to each `RequestSignagePointOrEndOfSubSlot` with a structurally valid `RespondEndOfSubSlot` containing a fresh unknown challenge hash. During this hold, all concurrent honest-peer EOS/SP catchup attempts are silently dropped via `LimitedSemaphoreFullError`, starving farmers and harvesters of signage points.

### Finding Description

The handler `FullNodeAPI.new_signage_point_or_end_of_sub_slot` in `chia/full_node/full_node_api.py` contains a sub-slot catch-up loop that:

1. Acquires `sp_catchup_sem` when `index_from_challenge == 0` and the advertised `prev_challenge_hash` is unknown locally.
2. Loops up to 30 times, each time calling `peer.call_api(..., timeout=10)` on the *same peer that sent the trigger message*.
3. Continues the loop as long as each response is a `RespondEndOfSubSlot` whose embedded challenge is also unknown locally.
4. Holds the semaphore for the entire duration of the loop.

No cryptographic validation of the `RespondEndOfSubSlot` payload occurs inside the loop — only a type check (`isinstance`) and a local store lookup. The `num_non_empty_sub_slots_seen` guard (max 3) is bypassable by crafting responses where `challenge_chain_end_of_slot_vdf.number_of_iterations == reward_chain.end_of_slot_vdf.number_of_iterations` (empty sub-slot format).

**Attack sequence:**
1. Attacker connects as a peer (one inbound connection).
2. Attacker sends `NewSignagePointOrEndOfSubSlot(index_from_challenge=0, prev_challenge_hash=<unknown_hash_A>)`.
3. Victim acquires `sp_catchup_sem`, enters loop.
4. Victim sends `RequestSignagePointOrEndOfSubSlot(challenge=<unknown_hash_A>, index=0, ...)` to attacker.
5. Attacker responds with `RespondEndOfSubSlot` whose `challenge_chain_end_of_slot_vdf.challenge = <unknown_hash_B>` (a new unknown hash), with equal iteration counts in both chains.
6. Victim's store lookup for `<unknown_hash_B>` fails; loop continues with `challenge_hash_to_request = <unknown_hash_B>`.
7. Steps 4–6 repeat 30 times (up to 300s total).
8. After 30 iterations, peer is closed — but attacker reconnects and repeats.

During steps 3–7, any honest peer sending a `NewSignagePointOrEndOfSubSlot` that also needs catchup hits `LimitedSemaphoreFullError` and is silently ignored.

### Impact Explanation

Farmers and harvesters depend on the full node receiving and forwarding signage points. If EOS/SP catchup is blocked for 300s per cycle (and the attacker reconnects to repeat), farmers receive no new signage points, halting block production. This is a **High** impact: long-lived inability for farmers/harvesters to process valid signage point updates under normal network assumptions.

### Likelihood Explanation

- Requires only one inbound peer connection — no special privileges, no leaked keys.
- The trigger message (`NewSignagePointOrEndOfSubSlot`) is a standard protocol message any peer can send.
- The attacker's responses are structurally valid (pass the `isinstance` check) and require no valid VDFs or signatures to keep the loop running.
- The attack is repeatable: after the peer is closed at line 720, the attacker reconnects and re-triggers.
- Rate limiting on `NewSignagePointOrEndOfSubSlot` (one message per cycle) does not prevent the attack since only one message is needed per 300s hold.

### Recommendation

1. **Per-peer semaphore scoping**: Track which peer currently holds the semaphore; reject a second trigger from the same peer without acquiring the semaphore.
2. **Timeout the entire catchup block**, not just individual `call_api` calls — e.g., wrap the `async with sp_catchup_sem` block in `asyncio.wait_for` with a total budget (e.g., 30s).
3. **Validate responses cryptographically** before continuing the loop, or at minimum verify that the returned `end_of_slot_bundle` is internally consistent (VDF challenge chain linkage).
4. **Reduce loop bound**: 30 iterations × 10s is excessive; a tighter bound (e.g., 5 iterations × 3s) limits worst-case hold time.
5. **Disconnect on loop exhaustion before releasing semaphore**: Currently `peer.close()` is called after the semaphore is released (line 720 is inside the `async with` block, so the semaphore is still held — but the close should be paired with an immediate return to prevent reconnect-and-repeat).
