### Title
Non-Strict Timestamp Lower-Bound Allows Frozen Block Timestamp in Committed Blocks — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

`is_proposal_init_valid` uses a strict `<` comparison for the lower-bound timestamp check, meaning a `ProposalInit` with `timestamp == last_block_timestamp` passes validation. Every validator node accepts such a proposal, the block is committed with a frozen timestamp, and every `get_block_timestamp()` syscall inside that block returns the same value as the previous block — breaking the strict-monotonicity invariant that time-sensitive contracts depend on.

### Finding Description

In `is_proposal_init_valid`:

```rust
// crates/apollo_consensus_orchestrator/src/validate_proposal.rs  line 263
if init_proposed.timestamp < last_block_timestamp {
    return Err(ValidateProposalError::InvalidProposalInit(...));
}
```

The condition rejects only `timestamp < last_block_timestamp`. A proposer that sets `timestamp == last_block_timestamp` passes this guard unconditionally. [1](#0-0) 

The accepted `ProposalInit.timestamp` is then forwarded without further sanitisation through `convert_to_sn_api_block_info`, which writes it verbatim into `BlockInfo.block_timestamp`:

```rust

The batcher executes every transaction in the block under this `BlockInfo`. The `get_block_timestamp()` syscall (both deprecated and Sierra paths) reads directly from `block_context.block_info.block_timestamp`: [4](#0-3)

### Impact Explanation

Every transaction executed in the frozen-timestamp block receives the same `block_timestamp` value as the previous block from the `get_block_timestamp()` syscall. Smart contracts that use `block_timestamp` to enforce time-based invariants — timelocks, vesting schedules, interest accrual windows, oracle freshness checks — will observe time as frozen. Depending on the contract logic this can:

- Prevent a timelock from ever unlocking (permanent DoS of a contract function).
- Cause an interest or reward calculation to return zero for the entire block, producing wrong storage state.
- Allow a condition guarded by `current_time > last_time + interval` to be permanently unsatisfied.

This maps to **"Critical. Wrong state … or revert result from blockifier/syscall/execution logic for accepted input"** because the `get_block_timestamp()` syscall returns a provably incorrect value for every transaction in the block, and the resulting state diff is committed to storage.

The timestamp is also hashed into the block hash via `PartialBlockHashComponents`: [7](#0-6) 

A frozen timestamp produces a valid but semantically wrong block hash that is accepted by all downstream consumers (RPC, state sync, L1 anchoring).

### Likelihood Explanation

Any validator that is selected as proposer for a round can craft a `ProposalInit` with `timestamp = previous_block_timestamp`. All other validators run `is_proposal_init_valid` and accept it because the check is `<` not `<=`. The block then accumulates 2f+1 precommit votes and is committed. No special privilege beyond being selected as proposer is required; proposer selection rotates through the validator set in normal BFT operation.

### Recommendation

Change the lower-bound check from strict-less-than to less-than-or-equal in `is_proposal_init_valid`:

```rust
// Before (allows equal timestamps):
if init_proposed.timestamp < last_block_timestamp { ... }

// After (enforces strict monotonicity):
if init_proposed.timestamp <= last_block_timestamp { ... }
```

Apply the same fix to the sync path in `try_sync`:

```rust
// Before:
timestamp.0 >= last_block_timestamp

// After:
timestamp.0 > last_block_timestamp
``` [5](#0-4) 

### Proof of Concept

1. Validator A is selected as proposer for block N.
2. The previous committed block N-1 has `timestamp = T`.
3. Validator A constructs `ProposalInit { height: N, timestamp: T, ... }` (equal to previous).
4. Validator A broadcasts the proposal over the P2P network.
5. Each receiving validator calls `is_proposal_init_valid`:
   - `last_block_timestamp = T`
   - `init_proposed.timestamp = T`
   - Check: `T < T` → **false** → no error returned → proposal accepted.
6. `initiate_validation` calls `convert_to_sn_api_block_info` → `BlockInfo { block_timestamp: BlockTimestamp(T), ... }`.
7. The batcher executes all transactions under `block_timestamp = T`.
8. Any contract calling `get_block_timestamp()` receives `T`, identical to block N-1.
9. The block collects 2f+1 precommits and is committed with frozen timestamp `T`.
10. A timelock contract checking `assert(get_block_timestamp() > unlock_time)` where `unlock_time == T` will revert for every transaction in block N, even though wall-clock time has advanced. [8](#0-7) [9](#0-8)

#[instrument(level = "warn", skip_all, fields(?proposal_init_validation, ?init_proposed))]
async fn is_proposal_init_valid(
    proposal_init_validation: &ProposalInitValidation,
    init_proposed: &ProposalInit,
    clock: &dyn Clock,
    l1_gas_price_provider: Arc<dyn L1GasPriceProviderClient>,
    gas_price_params: &GasPriceParams,
) -> ValidateProposalResult<()> {
    let now: u64 = clock.unix_now();
    let last_block_timestamp =
        proposal_init_validation.previous_proposal_init.as_ref().map_or(0, |info| info.timestamp);
    if init_proposed.timestamp < last_block_timestamp {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "Timestamp is too old: last_block_timestamp={}, proposed={}",
                last_block_timestamp, init_proposed.timestamp
            ),
        ));
    }

    let l1_gas_price_wei = NonzeroGasPrice::new(init.l1_gas_price_wei)?;
    let l1_data_gas_price_wei = NonzeroGasPrice::new(init.l1_data_gas_price_wei)?;
