### Title
Unauthenticated Pool URL Redirect Silently Overwrites Farmer's Pool Endpoint, Enabling Pool Reward Diversion - (File: chia/farmer/farmer.py)

### Summary

The Chia farmer unconditionally accepts HTTP 301/308 redirects from a pool server and permanently overwrites the locally-stored `pool_url` in the farmer's `PoolingShareState` config. No cryptographic verification is performed to confirm the redirect is authorized by the pool's owner key. A malicious or compromised pool operator can silently redirect a farmer's partials to an attacker-controlled pool endpoint, enabling pool reward diversion.

### Finding Description

In `chia/farmer/farmer.py`, the method `_pool_get_pool_info` fetches `GET /pool_info` from the configured pool URL. When the HTTP response history contains only 301 or 308 redirects and the final URL differs from the requested URL, the code extracts the final URL as `new_pool_url`: [1](#0-0) 

This result is returned to `update_pool_state`, which then **unconditionally writes the attacker-supplied URL** into the persistent `PoolingShareState` config with no authorization check, no domain validation, and no cryptographic proof that the redirect is sanctioned by the pool's registered owner key: [2](#0-1) 

The `enforce_https` check that exists in `update_pool_state` is applied only to the *old* `pool_config.pool_url` loaded at the top of the loop iteration, **before** the redirect URL is written: [3](#0-2) 

The new URL is written to disk at line 623 without passing through this guard. On the next polling cycle, the farmer will contact the attacker-controlled endpoint for all subsequent pool interactions (`GET /farmer`, `POST /farmer`, `PUT /farmer`, `POST /partial`).

The `set_payout_instructions` and `set_reward_targets` paths that legitimately mutate pool config require local RPC access: [4](#0-3) [5](#0-4) 

But the redirect-based `pool_url` mutation path requires only that the pool's HTTP server issue a 301/308 response — no local access, no owner-key signature, no farmer consent.

### Impact Explanation

Once the farmer's `pool_url` is overwritten to point to an attacker-controlled endpoint:

1. All future `POST /partial` submissions (the farmer's proof-of-work) are sent to the attacker's pool. The attacker's pool can credit points internally but pay out to an address of the attacker's choosing, diverting pool rewards away from the farmer's `payout_instructions`.
2. The farmer's `payout_instructions` (the puzzle hash where the farmer expects to receive XCH rewards) is transmitted to the attacker's pool via `PUT /farmer`. The attacker's pool can acknowledge receipt but ignore the address, paying to an attacker-controlled puzzle hash instead.
3. The farmer has no indication that its pool endpoint has changed; the config update is silent and persistent across restarts.

This matches the **High** allowed impact: *"Bypass of ... pool ... authorization that enables ... payout redirection"* and *"Corruption of ... pool membership state ... with direct security impact."* [6](#0-5) 

### Likelihood Explanation

The attacker must control or compromise the pool's HTTP server — they are the pool operator or an adversary who has gained control of the pool's web layer. The farmer has already established a relationship with this pool (it joined voluntarily). The attack requires only that the pool's `/pool_info` endpoint return a 301/308 redirect to an attacker-controlled URL. No cryptographic material from the farmer is needed. The attack is persistent (survives farmer restarts) and silent (no user-visible warning).

### Recommendation

1. **Require explicit farmer consent for pool URL changes.** The `pool_url` should only be updated through the same on-chain singleton transition path used by `pw_join_pool` (which requires the owner key signature and a relative lock height delay), not via an unauthenticated HTTP redirect.
2. **If redirect-following is intentional for pool migration**, require the redirect destination to be signed by the pool's registered owner public key (the key embedded in the singleton's `PoolState`) before persisting the new URL.
3. **At minimum**, apply the `enforce_https` check to `new_pool_url` before writing it to config, and log a prominent warning requiring operator acknowledgment before the URL is persisted.

### Proof of Concept

1. Farmer joins `https://legitimate-pool.example.com` via `pw_join_pool`.
2. Attacker compromises `legitimate-pool.example.com`'s web server (or is the pool operator).
3. On the next `update_pool_state` polling cycle (every `UPDATE_POOL_INFO_INTERVAL` seconds), the farmer calls `GET https://legitimate-pool.example.com/pool_info`.
4. The pool server responds with HTTP 301 → `https://attacker-pool.evil.com/pool_info`.
5. `_pool_get_pool_info` sets `new_pool_url = "https://attacker-pool.evil.com"`.
6. `update_pool_state` writes `editable_pool_config.pool_url = "https://attacker-pool.evil.com"` to disk.
7. All subsequent partials, farmer registrations, and payout instruction updates are sent to `attacker-pool.evil.com`.
8. The attacker's pool accepts partials, accumulates points, and pays out to the attacker's address, not the farmer's `payout_instructions`. [7](#0-6)

    async def _pool_get_pool_info(self, pool_config: PoolingShareState) -> GetPoolInfoResult | None:
        try:
            async with aiohttp.ClientSession(trust_env=True) as session:
                url = f"{pool_config.pool_url}/pool_info"
                async with session.get(url, ssl=ssl_context_for_root(get_mozilla_ca_crt(), log=self.log)) as resp:
                    if resp.ok:
                        response: dict[str, Any] = json.loads(await resp.text())
                        self.log.info(f"GET /pool_info response: {response}")
                        if "error_code" in response:
                            self.handle_failed_pool_response(
                                pool_config.p2_singleton_puzzle_hash,
                                f"Error in GET /pool_info {pool_config.pool_url}, {response}",
                            )
                            return None
                        try:
                            pool_info = GetPoolInfoResponse.from_json_dict(response)
                        except Exception as e:
                            self.handle_failed_pool_response(
                                pool_config.p2_singleton_puzzle_hash,
                                f"Invalid GET /pool_info response {pool_config.pool_url}, {e}",
                            )
                            return None
                        new_pool_url: str | None = None
                        response_url_str = f"{resp.url}"
                        if (
                            response_url_str != url
                            and len(resp.history) > 0
                            and all(r.status in {301, 308} for r in resp.history)
                        ):
                            new_pool_url = response_url_str.replace("/pool_info", "")

                        return GetPoolInfoResult(pool_info=pool_info, new_pool_url=new_pool_url)
                    else:
                        self.handle_failed_pool_response(
                            pool_config.p2_singleton_puzzle_hash,
                            f"Error in GET /pool_info {pool_config.pool_url}, {resp.status}",
                        )

        except Exception as e:
            self.handle_failed_pool_response(
                pool_config.p2_singleton_puzzle_hash, f"Exception in GET /pool_info {pool_config.pool_url}, {e}"
            )

        return None
