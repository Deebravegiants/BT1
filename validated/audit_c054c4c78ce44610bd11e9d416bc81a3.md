### Title
Missing Permanent Processed-Request Registry Enables Repeated MPC Signing of the Same Foreign Transaction - (File: `crates/contract/src/lib.rs`, `crates/contract/src/pending_requests.rs`)

---

### Summary

The `verify_foreign_transaction` flow in the NEAR MPC contract uses a transient `pending_verify_foreign_tx_requests` map to track in-flight requests. Once a request is resolved via `respond_verify_foreign_tx`, the entry is **permanently removed** from the map with no record that the foreign transaction was ever processed. Any unprivileged caller can immediately re-submit the identical foreign transaction, causing the MPC network to verify it on the foreign chain again and issue a second (or Nth) valid threshold signature over the same payload. This is the direct analog of the RailgunSmartWallet re-entrancy: the "hash" (foreign tx key) is not marked as permanently consumed before the signing side-effect is delivered.

---

### Finding Description

The `MpcContract` state holds three pending-request maps:

```rust
pending_signature_requests: LookupMap<SignatureRequest, Vec<YieldIndex>>,
pending_ckd_requests:       LookupMap<CKDRequest, Vec<YieldIndex>>,
pending_verify_foreign_tx_requests: LookupMap<VerifyForeignTransactionRequest, Vec<YieldIndex>>,
``` [1](#0-0) 

The `resolve_yields_for` helper, called by `respond_verify_foreign_tx`, **removes** the map entry and resumes all queued yields:

```rust
let resumed = requests
    .remove(request)          // ← entry deleted here; no permanent record kept
    .unwrap_or_default()
    .into_iter()
    .map(|YieldIndex { data_id }| {
        env::promise_yield_resume(&data_id, response_bytes.clone());
    })
    .count();
``` [2](#0-1) 

After `remove` completes, the contract has **zero memory** that this `VerifyForeignTransactionRequest` was ever processed. The `VerifyForeignTransactionRequest` key is caller-agnostic — it contains only the foreign-chain RPC request, domain ID, and payload version:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [3](#0-2) 

Unlike `SignatureRequest` (which binds the caller's account ID and derivation path into the key, making each user's request unique), `VerifyForeignTransactionRequest` carries no per-caller nonce. Any account can re-submit the identical foreign transaction after the first response is delivered, and the contract will accept it as a fresh request.

The `verify_foreign_transaction` entry point performs no check against a "previously processed" set: [4](#0-3) 

The node-side `VerifyForeignTransactionRequestStorage.add()` does guard against inserting the same request ID twice into the node's local DB, but this is a per-node, in-memory/local-DB guard — not a contract-level invariant — and it does not prevent the contract from accepting and queuing a new yield for the same foreign transaction after the first one has been fully resolved. [5](#0-4) 

---

### Impact Explanation

**High — Cross-chain replay / double-spend.**

The `verify_foreign_transaction` feature is the MPC network's bridge verification primitive: callers submit a foreign-chain transaction ID, MPC nodes verify it on-chain, and the network returns a threshold ECDSA signature over the extracted payload. Bridge contracts downstream are expected to treat this signature as proof that the foreign transaction occurred and was verified by the MPC network.

Because the contract issues a fresh valid threshold signature on every re-submission of the same foreign transaction ID, an attacker can:

1. Submit `verify_foreign_transaction` for foreign tx `T`.
2. Receive a valid MPC signature `σ₁` over `payload(T)`.
3. Use `σ₁` to claim a bridge transfer on the destination chain.
4. Re-submit the identical `verify_foreign_transaction` for `T` (the contract has no memory of step 1).
5. Receive a second valid MPC signature `σ₂` over `payload(T)` (different presignature → different `(r, s)` values, both valid).
6. Use `σ₂` to claim the same bridge transfer a second time — double-spend.

The MPC network itself is a necessary step in this path: without the second valid signature, the second claim cannot be authorized.

---

### Likelihood Explanation

**Medium.** The re-submission requires only a 1-yoctoNEAR deposit and standard gas — no privileged access, no threshold collusion, no TEE bypass. The only prerequisite is that the first request has been fully resolved (i.e., `respond_verify_foreign_tx` has been called and the yield resumed). Any unprivileged NEAR account can trigger the re-submission. The attack is profitable whenever the downstream bridge contract does not maintain its own "signature already used" registry, which is a realistic assumption for bridge contracts that delegate replay-prevention responsibility to the MPC verification layer.

---

### Recommendation

Introduce a permanent `LookupSet<VerifyForeignTransactionRequest>` (or a hash thereof) in the contract state, populated **before** the yield is resumed in `respond_verify_foreign_tx`. Reject any `verify_foreign_transaction` call whose request key is already present in this set. This mirrors the external report's recommendation: mark the hash as processed before delivering the side-effect (the signature).

```rust
// In MpcContract state:
processed_foreign_tx_requests: LookupSet<VerifyForeignTransactionRequest>,

// In verify_foreign_transaction():
if self.processed_foreign_tx_requests.contains(&request) {
    env::panic_str("ForeignTransactionAlreadyProcessed");
}

// In respond_verify_foreign_tx(), before resolve_yields_for():
self.processed_foreign_tx_requests.insert(request.clone());
pending_requests::resolve_yields_for(...);
```

Alternatively, if unbounded storage growth is a concern, include a per-epoch or per-block nonce in `VerifyForeignTransactionRequest` so that each submission is inherently unique and cannot be replayed.

---

### Proof of Concept

1. Deploy the contract in Running state with a ForeignTx domain and at least one supported chain (e.g., Bitcoin).
2. Call `verify_foreign_transaction` with `request = { chain: Bitcoin, tx_id: [0xAB; 32], confirmations: 1, extractors: [BlockHash] }`. Observe the yield is queued.
3. Call `respond_verify_foreign_tx` with a valid signature. Observe `pending_verify_foreign_tx_requests.get(&request)` returns `None` — the entry is gone.
4. Immediately call `verify_foreign_transaction` again with the **identical** request. Observe the call succeeds and a new yield is queued (`pending_verify_foreign_tx_requests.get(&request)` returns `Some`).
5. Call `respond_verify_foreign_tx` again with a new valid signature (using a different presignature). Observe a second distinct `(r, s)` ECDSA signature is returned for the same foreign transaction payload.
6. Both `σ₁` and `σ₂` are valid MPC threshold signatures over `payload(T)` and can each independently authorize a bridge action — demonstrating the double-spend primitive. [6](#0-5) [7](#0-6)

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
