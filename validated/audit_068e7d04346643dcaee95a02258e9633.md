### Title
Stale Votes from Removed Participants Count Toward Code-Hash Whitelisting Threshold — (`crates/contract/src/tee/proposal.rs`)

---

### Summary

`CodeHashesVotes::count_votes` tallies every stored vote without filtering by the current participant set. After a resharing that removes participants, their prior votes for a `NodeImageHash` remain in storage and are included in the threshold check inside `vote_code_hash`. This is structurally identical to M-03: a value computed (or accumulated) at an earlier point in time is used for an ongoing safety check without accounting for the state change that occurred in between.

---

### Finding Description

`CodeHashesVotes::vote` (called by `TeeState::vote`, called by `MpcContract::vote_code_hash`) returns the raw count of all stored votes for a proposal:

```rust
// crates/contract/src/tee/proposal.rs  lines 47-52
fn count_votes(&self, proposal: &NodeImageHash) -> u64 {
    self.proposal_by_account
        .values()
        .filter(|&prop| prop == proposal)
        .count() as u64
}
```

No participant-membership filter is applied. The returned count is compared directly against the current threshold in `lib.rs`:

```rust
// crates/contract/src/lib.rs  lines 1425-1428
if votes >= self.threshold()?.value() {
    self.tee_state
        .whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
}
```

Contrast this with every other threshold-crossing vote in the contract. `ProviderVotes::vote` filters at count time:

```rust
// crates/contract/src/foreign_chain_rpc.rs  lines 185-187
voter_set.count_for(|(p, c)| {
    *c == chain && participants.is_participant_given_participant_id(&p.get())
})
```

`TeeVerifierVotes::vote` does the same:

```rust
// crates/contract/src/tee/verifier_votes.rs  lines 74-77
let voter_set = self.pending.vote(participant, proposal_hash);
voter_set.count_for(|p| participants.is_participant_given_participant_id(&p.get()))
```

`CodeHashesVotes` has no such guard. It relies entirely on `clean_non_participant_votes` being called after every resharing to purge stale entries. The test suite itself documents this fragility with the comment *"simulating a missed `clean_tee_status`"* and shows that the other vote types remain correct even when cleanup is skipped, while `CodeHashesVotes` does not.

---

### Impact Explanation

If `clean_non_participant_votes` is not executed after a resharing (e.g., the NEAR cross-contract promise carrying the cleanup call fails, or the call is simply not issued on a code path), stale votes from removed participants persist. A single current participant voting for the same `NodeImageHash` can then push the raw count over the threshold, whitelisting a code hash that was never approved by a quorum of the current participant set.

A whitelisted code hash controls which Docker images are permitted to run inside the TEE and access MPC key shares. Whitelisting a malicious image hash is therefore a direct path to **unauthorized access to MPC key shares and signing capability** — Critical impact under the allowed scope.

---

### Likelihood Explanation

The attack requires:
1. At least `T − 1` participants (below the signing threshold) in the pre-resharing set to vote for a target code hash.
2. A resharing that removes those participants.
3. `clean_non_participant_votes` to not execute (promise failure, missing call site, or race condition).
4. One participant in the new set to vote for the same hash.

Steps 1–2 are within reach of a Byzantine minority strictly below the signing threshold. Step 3 is a realistic failure mode on NEAR where cross-contract promise callbacks can fail silently. Step 4 requires only one additional colluding or deceived participant. The combination is realistic and does not require threshold-or-above collusion.

---

### Recommendation

Apply the same participant-membership filter inside `count_votes` that `ProviderVotes` and `TeeVerifierVotes` already use, so correctness does not depend on cleanup being called:

```rust
fn count_votes_for_current(
    &self,
    proposal: &NodeImageHash,
    participants: &Participants,
) -> u64 {
    self.proposal_by_account
        .iter()
        .filter(|(p, prop)| {
            *prop == proposal
                && participants.is_participant_given_participant_id(&p.get())
        })
        .count() as u64
}
```

Pass `threshold_parameters.participants()` (already available at the call site in `vote_code_hash`) into `TeeState::vote` and down to `CodeHashesVotes::vote`. This mirrors the existing pattern in `ProviderVotes` and `TeeVerifierVotes` and makes the threshold check resilient to missed cleanup, exactly as the M-03 fix recalculated `votesDifference` at use-time rather than relying on a stale creation-time snapshot.

---

### Proof of Concept

**Setup**: N = 5, T = 3. Participants P1–P5.

1. P1 and P2 (malicious, below signing threshold) call `vote_code_hash(H_evil)`. `count_votes` returns 2 < 3. Hash not yet whitelisted.
2. Governance resharing removes P1 and P2. New set: {P3, P4, P5}, T = 3.
3. The cross-contract promise carrying `clean_non_participant_votes` fails. P1 and P2's entries remain in `CodeHashesVotes::proposal_by_account`.
4. P3 (one colluding or deceived participant) calls `vote_code_hash(H_evil)`. `count_votes` iterates all stored entries and returns 3 (P1 + P2 + P3) ≥ T = 3.
5. `whitelist_tee_proposal(H_evil, …)` executes. `H_evil` is now an approved MPC node image.
6. A node running the malicious image boots, passes TEE attestation against the whitelisted hash, and gains access to MPC key shares.

The threshold was crossed with only **1 current participant** voting, not the required 3. The root cause — `count_votes` not filtering by current participants — is the direct analog of M-03's `votesDifference` not accounting for veCRV balance decay.         log!(

```rust
            code_hash,

        // update the state

            self.tee_state
                .whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
        }

        Ok(())
    }
```

```rust
    /// Stale CodeHashesVotes entries from removed participants must not count toward
    /// quorum after resharing.
    ///
    /// Scenario (N=5, T=3):
    /// 1. P1 and P2 vote for malicious hash before resharing.
    /// 2. Resharing removes P1 and P2. New set: {P3, P4, P5}.
    /// 3. clean_non_participant_votes removes stale votes.
    /// 4. P3 votes for the same hash — only 1 vote, not 3.
    #[test]
    fn test_clean_non_participant_votes_removes_stale_votes() {
        // Build 5 participants
        let mut all_participants = Participants::new();
        let mut account_ids = Vec::new();
        for i in 0..5 {
            let (account_id, info) = gen_participant(i);
            account_ids.push(account_id.clone());
            all_participants.insert(account_id, info).unwrap();
        }

        let mut tee_state = TeeState::default();

        // P0 and P1 vote for a malicious hash before resharing
        let malicious_hash = NodeImageHash::from([0xAA; 32]);
        for account_id in &account_ids[0..2] {
            let mut ctx = VMContextBuilder::new();
            ctx.signer_account_id(account_id.clone());
            testing_env!(ctx.build());
            let auth_id = AuthenticatedParticipantId::new(&all_participants).unwrap();
            tee_state.votes.vote(malicious_hash, &auth_id);
        }
        assert_eq!(tee_state.votes.proposal_by_account.len(), 2);

        // Resharing removes P0 and P1. New participant set: {P2, P3, P4}.
        let new_participants = all_participants.subset(2..5);

        // Clean non-participants (as done by CLEAN_TEE_STATUS after resharing)
        tee_state.clean_non_participant_votes(&new_participants);

        // Stale votes must be removed
        assert_eq!(tee_state.votes.proposal_by_account.len(), 0);

        // P2 votes for the same malicious hash — should be only 1 vote, not 3
        let p2_account = &account_ids[2];
        let mut ctx = VMContextBuilder::new();
        ctx.signer_account_id(p2_account.clone());
        testing_env!(ctx.build());
        let auth_id = AuthenticatedParticipantId::new(&new_participants).unwrap();
        let vote_count = tee_state.votes.vote(malicious_hash, &auth_id);
        assert_eq!(vote_count, 1, "Only the fresh vote from P2 should count");
    }
```
