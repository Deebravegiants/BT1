### Title
Broken Pigeonhole Invariant in `set_state_header` Allows Malicious Peer to Inject Wrong Receipts and Corrupt Syncing Node's State Root — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

The `from_shard_id` field inside each `ShardProof` is never validated against the current epoch's shard layout during `set_state_header`. The Pigeonhole-principle argument used to guarantee receipt completeness is therefore unsound: a malicious peer can supply N receipt proofs (matching `chunks_included`) with N distinct but epoch-invalid `from_shard_id` values, all backed by recycled valid merkle proofs from real shards, while silently omitting one real shard's receipts. The crafted header passes every check in `set_state_header`, is persisted to `DBCol::StateHeaders`, and is later consumed by `set_state_finalize`, which applies the wrong receipt set and writes a corrupted `ChunkExtra` state root to disk.

---

### Finding Description

**Step 1 – The Pigeonhole argument and its assumption**

`set_state_header` (adapter.rs lines 470–485) relies on the following reasoning:

> "We know there were exactly `block_header.chunks_included` chunks included … According to Pigeonhole principle, it's enough to ensure all receipt_proofs are distinct to prove that all receipts were received and no receipts were hidden." [1](#0-0) 

The argument is valid only if every `from_shard_id` value is a legitimate shard ID in the current epoch's layout. The code never enforces this: `from_shard_id` is inserted into a plain `HashSet<ShardId>` and checked only for uniqueness.

**Step 2 – Merkle proofs do not bind `from_shard_id` to a tree position**

Steps 4e and 4f verify:

```
verify_path(root, proof, receipts_hash)          // receipts ∈ root
verify_path(block_receipts_root, block_proof, root)  // root ∈ block tree
``` [2](#0-1) 

Neither check uses `from_shard_id`. `verify_path` is a pure hash-chain computation with no position encoding. [3](#0-2) 

**Step 3 – Constructing the malicious header**

Suppose the current epoch has shards `[0, 1, 2]` and `chunks_included = 3`. The attacker (a full-node peer) holds all real receipt proofs. They craft:

| j | `from_shard_id` | receipts | `RootProof` |
|---|---|---|---|
| 0 | 0 | real receipts from shard 0 | real proof for shard 0 |
| 1 | 1 | real receipts from shard 1 | real proof for shard 1 |
| 2 | **999** (non-existent) | **shard 0's receipts again** | **shard 0's `RootProof` reused** |

Shard 2's receipts are omitted entirely.

- **4c count check**: `3 == chunks_included` ✓  
- **4d uniqueness**: `{0, 1, 999}` are all distinct ✓  
- **4e**: `verify_path(root_0, proof_0, hash(shard_id, receipts_0))` — valid, same receipts and same root ✓  
- **4f**: `verify_path(block_receipts_root, block_proof_0, root_0)` — valid, shard 0's root is genuinely in the block tree ✓  

The header is accepted and written to `DBCol::StateHeaders`.

**Step 4 – `set_state_finalize` applies the corrupted receipt set**

`collect_receipts_from_response` performs a flat concatenation with no deduplication or shard-layout validation: [4](#0-3) 

The resulting `receipts` vector contains shard 0's receipts twice and shard 2's receipts not at all. This is passed directly to `apply_chunk`: [5](#0-4) 

`apply_chunk` produces a state root that diverges from the honest network's state root. That root is committed to `ChunkExtra` via `apply_chunk_postprocessing` and persisted. [6](#0-5) 

**Step 5 – No downstream catch**

There is no post-`set_state_finalize` comparison of the computed state root against the block header's expected value. The corrupted `ChunkExtra` becomes the authoritative starting state for all subsequent chunk applications on this node.

---

### Impact Explanation

A syncing node (including a validator catching up after downtime) that connects to a malicious peer will store a wrong state root after `set_state_finalize`. Every subsequent chunk application starts from this wrong root, causing the node to produce wrong chunk hashes and block hashes. The node permanently diverges from the honest network and cannot participate in consensus or serve correct RPC responses until it re-syncs from scratch. If multiple validators are simultaneously syncing and connect to the same malicious peer, consensus participation is degraded.

---

### Likelihood Explanation

State sync is a standard, frequently exercised code path. Any peer on the network can serve `ShardStateSyncResponseHeader` messages. The attack requires no special privileges — only knowledge of the real block data (available to any full node) and the ability to modify the `from_shard_id` field and recycle one `RootProof`. No cryptographic forgery is needed; all merkle proofs are genuine proofs for real on-chain data.

---

### Recommendation

Inside the `from_shard_id` uniqueness loop in `set_state_header`, validate that each `from_shard_id` is a member of the current epoch's shard layout (obtained via `epoch_manager.shard_ids(block_header.epoch_id())`). Reject the header if any `from_shard_id` is not in that set. This restores the Pigeonhole argument: N distinct valid shard IDs for N included chunks guarantees full coverage.

Additionally, consider binding the merkle proof to the shard index (using `verify_path_with_index`) so that a proof for shard X cannot be recycled as a proof for shard Y.
