### Title
Storage Deposit Permanently Locked on Attestation Cleanup — (`File: crates/contract/src/lib.rs`)

---

### Summary

`submit_participant_info()` charges the calling node for NEAR storage used by its attestation record. When `clean_invalid_attestations()` (or `clean_tee_status()`) later deletes that record, the storage is freed but the deposit is **never refunded** to the node that paid for it. The deposit is permanently locked inside the contract. This is a direct structural analog to the `depositAndClose()` bug: a record is deleted without returning the associated deposit to its rightful owner.

---

### Finding Description

**Step 1 — Deposit collected on submission**

`submit_participant_info()` measures storage before and after inserting the attestation, computes the cost, and requires the caller to cover it: [1](#0-0) 

The excess is refunded, but the exact storage cost is retained by the contract. The paying account is `account_id` (the node's NEAR account).

**Step 2 — Record deleted, no refund issued**

`clean_invalid_attestations()` is a public endpoint callable by anyone while the protocol is `Running`. It delegates to `tee_state.clean_invalid_attestations()`, which removes expired or stale attestation entries from `stored_attestations`: [2](#0-1) 

There is no `Promise::new(original_depositor).transfer(freed_storage_cost)` anywhere in this path. The freed storage bytes reduce the contract's on-chain storage bill, but the corresponding yoctoNEAR remain inside the contract with no mechanism to recover them.

The same omission exists in `clean_tee_status()`: [3](#0-2) 

**Structural parallel to the original bug**

| Original (`LineOfCredit`) | This codebase |
|---|---|
| `depositAndClose()` deletes debt record | `clean_invalid_attestations()` deletes attestation record |
| `_close()` has no transfer back to lender | cleanup has no transfer back to node |
| Lender's deposit locked in contract | Node's storage deposit locked in contract |
| `close()` (correct path) does transfer | No correct path exists here |

---

### Impact Explanation

Every node that has ever called `submit_participant_info()` and paid a storage deposit loses that deposit permanently once its attestation is cleaned up — whether by natural expiry, a TEE whitelist change, or an adversary triggering `clean_invalid_attestations()` (which is open to any caller). The freed storage accrues to the contract's balance with no accounting path back to the depositor. This breaks the production safety/accounting invariant that storage deposits must be returned when storage is freed.

**Impact: Medium** — balance and participant-state accounting invariant broken; funds permanently locked without any operator misconfiguration required.

---

### Likelihood Explanation

Attestations expire naturally (governed by `tee_upgrade_deadline_duration_seconds`). Every epoch transition that changes the TEE whitelist can invalidate existing attestations. `clean_invalid_attestations()` is callable by any account with no access control while the protocol is `Running`. The deposit lockout therefore occurs in the normal operational lifecycle without any adversarial action.

**Likelihood: High** — triggered by routine attestation expiry and open cleanup calls.

---

### Recommendation

Track the original depositor and the amount paid inside the attestation record. When `clean_invalid_attestations()` or `clean_tee_status()` removes an entry, issue a `Promise::new(original_depositor).transfer(storage_deposit)` to return the freed deposit, mirroring the pattern already used in `require_deposit()` and `propose_update()`: [4](#0-3) [5](#0-4) 

---

### Proof of Concept

1. Node A calls `submit_participant_info()` with a valid TEE attestation, attaching the required storage deposit (e.g., 0.05 NEAR for a typical attestation payload). The contract stores the attestation and retains the deposit.
2. The TEE whitelist is updated (e.g., via `vote_code_hash` reaching threshold), causing Node A's attestation to reference a stale hash and fail re-verification.
3. Any unprivileged account calls `clean_invalid_attestations(max_scan: 100)`.
4. Node A's attestation entry is removed from `tee_state.stored_attestations`. Storage is freed.
5. Node A's 0.05 NEAR storage deposit remains locked in the contract. Node A has no method to reclaim it. The contract's balance increases by the freed storage cost with no corresponding liability entry.

    pub fn clean_invalid_attestations(&mut self, max_scan: u32) -> Result<u32, Error> {
        log!(
            "clean_invalid_attestations: signer={}, max_scan={}",
            env::signer_account_id(),
            max_scan
        );
        // Running-only: keygen / resharing may reference attestations that have not yet
        // been activated, so cleanup is off-limits during those phases.
        if !matches!(self.protocol_state, ProtocolContractState::Running(_)) {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }
        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);
        Ok(self
            .tee_state
            .clean_invalid_attestations(tee_upgrade_deadline_duration, max_scan as usize))
    }
