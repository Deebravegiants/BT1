### Title
Malicious Harvester Can Unconditionally Redirect Farmer Block Rewards via Unvalidated `farmer_reward_address_override` - (File: chia/farmer/farmer_api.py)

### Summary
`FarmerAPI._process_respond_signatures` unconditionally accepts the `farmer_reward_address_override` field from a harvester's `RespondSignatures` protocol message and uses it as the farmer reward puzzle hash in `DeclareProofOfSpace`, with no fee-quality enforcement. Any connected harvester can redirect 100% of the farmer's block reward (XCH) to an arbitrary address it controls.

### Finding Description
CHIP-22 introduced `farmer_reward_address_override` so that third-party harvesters can take a fee by redirecting a portion of the farmer reward. The fee-quality convention (checked in `notify_farmer_reward_taken_by_harvester_as_fee`) is supposed to limit how much the harvester may take. However, the enforcement is entirely absent.

In `_process_respond_signatures`, when the harvester's `RespondSignatures` message contains a non-`None` `farmer_reward_address_override`, the farmer unconditionally replaces its own configured `farmer_target` with the harvester-supplied address:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override   # no validation
    include_source_signature_data = True
``` [1](#0-0) 

This address is then passed directly into `DeclareProofOfSpace` as the `farmer_reward_puzzle_hash`: [2](#0-1) 

The `RespondSignatures` message type carries this field with no cryptographic binding to any fee-quality proof: [3](#0-2) 

The only guard that exists is `notify_farmer_reward_taken_by_harvester_as_fee`, which is triggered by the earlier `NewProofOfSpace` message — not by `RespondSignatures` — and it only **logs** a warning; it never rejects the override or aborts the block-making flow: [4](#0-3) 

The `NewProofOfSpace` path that triggers the logging: [5](#0-4) 

Two compounding gaps:
1. `notify_farmer_reward_taken_by_harvester_as_fee` only logs — it never blocks the override.
2. The `farmer_reward_address_override` in `RespondSignatures` is not even routed through the fee-quality check at all; it is used directly.

### Impact Explanation
A malicious harvester sets `farmer_reward_address_override` to any puzzle hash it controls inside its `RespondSignatures` reply. The farmer uses that address as the farmer reward puzzle hash. The full node creates the coinbase reward coin (currently 0.25 XCH per block) payable to the attacker's address instead of the farmer's configured address. This is unauthorized diversion of XCH block rewards — a Critical/High impact under the allowed scope ("reward diversion … affecting XCH" / "payout redirection").

### Likelihood Explanation
Any harvester that is connected to a farmer can exploit this. In the third-party harvester scenario that CHIP-22 was designed for, the harvester is an external service operated by a different party. A malicious third-party harvester operator can redirect every block reward to their own address without the farmer's knowledge or consent. No leaked keys, admin access, or cryptographic break is required — only a valid network connection as a harvester peer.

### Recommendation
- **Short term**: In `_process_respond_signatures`, enforce the fee-quality threshold before accepting `response.farmer_reward_address_override`. If the quality does not pass the threshold (or `fee_info` is absent), discard the override and use `self.farmer.farmer_target`. Do not merely log a warning.
- **Long term**: Require the harvester to cryptographically sign the `farmer_reward_address_override` value with the plot key (whose public half is already verified via `agg_pk == pospace.plot_public_key`) so that the farmer can verify the override is intentional and bound to the specific proof, not injected after the fact.

### Proof of Concept
1. Mallory operates a harvester that connects to Alice's farmer.
2. Mallory's harvester finds a valid proof of space and sends `NewProofOfSpace` normally (with `farmer_reward_address_override=None` to avoid even the advisory log).
3. Alice's farmer sends `RequestSignatures` back to Mallory's harvester.
4. Mallory's harvester replies with `RespondSignatures` where `farmer_reward_address_override` is set to Mallory's puzzle hash.
5. `_process_respond_signatures` replaces `farmer_reward_address` with Mallory's address — no check, no rejection.
6. The farmer sends `DeclareProofOfSpace` to the full node with Mallory's address as `farmer_reward_puzzle_hash`.
7. The full node creates the block; the 0.25 XCH farmer reward coin is created at Mallory's address.
8. Alice receives nothing for the block she won.

    def notify_farmer_reward_taken_by_harvester_as_fee(
        self, sp: farmer_protocol.NewSignagePoint, proof_of_space: harvester_protocol.NewProofOfSpace
    ) -> None:
        """
        Apply a fee quality convention (see CHIP-22: https://github.com/Chia-Network/chips/pull/88)
        given the proof and signage point. This will be tested against the fee threshold reported
        by the harvester (if any), and logged.
        """
        assert proof_of_space.farmer_reward_address_override is not None

        challenge_str = str(sp.challenge_hash)

        ph_prefix = self.config["network_overrides"]["config"][self.config["selected_network"]]["address_prefix"]
        farmer_reward_puzzle_hash = encode_puzzle_hash(proof_of_space.farmer_reward_address_override, ph_prefix)

        self.log.info(
            f"Farmer reward for challenge '{challenge_str}' "
            + f"taken by harvester for reward address '{farmer_reward_puzzle_hash}'"
        )

        fee_quality = calculate_harvester_fee_quality(proof_of_space.proof.proof, sp.challenge_hash)
        fee_quality_rate = float(fee_quality) / float(0xFFFFFFFF) * 100.0

        if proof_of_space.fee_info is not None:
            fee_threshold = proof_of_space.fee_info.applied_fee_threshold
            fee_threshold_rate = float(fee_threshold) / float(0xFFFFFFFF) * 100.0

            if fee_quality <= fee_threshold:
                self.log.info(
                    f"Fee threshold passed for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
            else:
                self.log.warning(
                    f"Invalid fee threshold for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
                self.log.warning(
                    "Harvester illegitimately took a fee reward that "
                    + "did not belong to it or it incorrectly applied the fee convention."
                )
        else:
            self.log.warning(
                "Harvester illegitimately took reward by failing to provide its fee rate "
                + f"for challenge '{challenge_str}'. "
                + f"Fee quality was {fee_quality_rate:.3f}% ({fee_quality} or 0x{fee_quality:08x})"
            )
