### Title
Batch dlog0/dlog1/dlog2 Transcript Fork Missing Triple Index Allows Proof Reuse Across All N Positions — (`crates/threshold-signatures/src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

### Summary

In `do_generation_many<N>`, the prover and verifier both call `transcript.fork(b"dlog0", &me.bytes())` / `transcript.fork(b"dlog0", &from.bytes())` inside a loop over N triples, without including the loop index in the fork. Because `fork` clones the base transcript without mutating it, every iteration produces an **identical** forked transcript. A Byzantine participant who sets all N of their `e` polynomials to the same polynomial can generate a single dlog0 proof and submit it for all N positions in `phi_proof0_v`; all N verifications pass.

The `fork` implementation clones the transcript and appends a message to the clone; the base `transcript` is never mutated:

```rust
pub fn fork(&self, label: &'static [u8], data: &[u8]) -> Self {
    let mut out = self.clone();
    out.message(label, data);
    out
}
``` [3](#0-2) 

Inside `prove_with_nonce`, the statement (the public point) is absorbed into the transcript before the challenge is derived:

```rust
transcript.message(NEAR_DLOG_STATEMENT_LABEL, &statement.encode()?);
``` [4](#0-3) 

This means the challenge is a function of: `(base transcript state) + fork label + statement + nonce commitment`. If the attacker sets all N `e_i` polynomials to the same polynomial, then `big_e_i.eval_at_zero()` is the same point for every `i`, the statement is identical across all iterations, and the base fork is identical — so a single `(e, s)` proof satisfies all N verifications.

**What the commitment check does and does not prevent:**

The commitment check at lines 389–400 verifies that the revealed `big_e_i` matches the hash commitment made in round 1. If the attacker commits to the same `big_e` for all N positions, this check passes. The protocol has no uniqueness requirement on the committed polynomials. [5](#0-4) 

The share consistency check at lines 466–472 verifies that the received scalar share `a_i` satisfies `G * a_i == big_e.eval_at_participant(me)`. Because the attacker uses the same polynomial for all N triples, the shares are naturally consistent — this check does not detect the attack. [6](#0-5) 

The same structural issue applies to `dlog1` (lines 285–290 / 417–425) and `dlog2` (lines 582–587 / 640–644), which also fork without a triple index. [7](#0-6) [8](#0-7) 

### Impact Explanation

The attacker contributes the same `a` value (and `b` value) to all N triples in the batch. The final `a_total_i = a_attacker + Σ_{honest j} e_{j,i}(j)` differs across triples only by the honest participants' independent contributions. The attacker knows `a_attacker` exactly. Over multiple signing sessions that consume triples from the same batch, the attacker can subtract their known contribution from each observed nonce, obtaining `Σ_{honest j} e_{j,i}(j)` for each triple. This leaks information about honest participants' polynomial evaluations and, with enough signatures, can enable partial or full recovery of the shared signing key. The impact is a key-share correlation attack across all presignatures generated from the batch.

### Likelihood Explanation

The precondition is a single Byzantine participant below the signing threshold — the minimum adversarial assumption in the MPC model. No collusion, no TEE compromise, and no network-level attack is required. The attack is entirely local: the Byzantine participant simply reuses one polynomial and one proof across all N batch positions. The `generate_triple_many` function is the production batch path used in the node. [9](#0-8) 

### Recommendation

Include the triple index `i` in every transcript fork inside the batch loop, for all three proof types:

```rust
// dlog0
dlog::prove_with_nonce(
    &mut transcript.fork(b"dlog0", &[me.bytes(), i.to_le_bytes()].concat()),
    ...
)
// dlog1, dlog2 analogously
```

This makes each forked transcript unique per `(participant, triple_index)`, so a proof generated for position `i` cannot verify at position `j ≠ i` even when the statements are identical.

### Proof of Concept

```rust
// N=4, attacker sets all e polynomials to the same polynomial.
// Step 1: attacker generates one polynomial e.
// Step 2: sets e_0 = e_1 = e_2 = e_3 = e.
// Step 3: generates one nonce (k, K) and one dlog0 proof P for statement e(0)*G.
// Step 4: submits phi_proof0_v = [P, P, P, P].
// Step 5: verifier loop calls transcript.fork(b"dlog0", &from.bytes()) four times
//         with the same base transcript and the same statement e(0)*G each time.
// Step 6: all four dlog::verify calls return true.
// Step 7: all four resulting triples have a_attacker = e(attacker_index).
// Assertion: result[0].a_attacker == result[1].a_attacker
//         == result[2].a_attacker == result[3].a_attacker  // passes
```

The existing test `test_triple_generation_many` only exercises `N=1` and would not catch this. [10](#0-9)

pub fn generate_triple_many<const N: usize, T, R>(
    participants: &[Participant],
    me: Participant,
    threshold: T,
    rng: R,
) -> Result<impl Protocol<Output = TripleGenerationOutputMany> + use<N, T, R>, InitializationError>
where
    T: Into<ReconstructionThreshold>,
    R: CryptoRngCore + Send + 'static,
{
    let (participants, threshold) = validate_triple_inputs(participants, threshold)?;
    let ctx = Comms::with_buffer_capacity(triple_generation_max_incoming_buffer_entries(
        participants.len(),
        N,
    )?);
    let fut = do_generation_many::<N>(ctx.clone(), participants, me, threshold, rng);
    Ok(make_protocol(ctx, fut))
}
