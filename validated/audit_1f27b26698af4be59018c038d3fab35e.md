Audit Report

## Title
Stale Cached `rsETHPrice` Used in Deposit Mint Calculation Enables Yield Theft from Existing rsETH Holders - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTOracle` stores `rsETHPrice` as a manually-updated state variable. `LRTDepositPool.getRsETHAmountToMint` uses this cached value as the denominator when computing rsETH to mint for a new deposit, without triggering a price refresh. Because rsETH is yield-bearing and its true price continuously rises between updates, a depositor can exploit the staleness gap to receive more rsETH than fair value, diluting the accrued yield of all existing rsETH holders.

## Finding Description
`LRTOracle` declares `rsETHPrice` as a plain storage variable updated only when `updateRSETHPrice()` or `updateRSETHPriceAsManager()` is explicitly called: [1](#0-0) [2](#0-1) 

`_updateRsETHPrice()` derives the correct price from live TVL and current rsETH supply, then writes it to storage: [3](#0-2) [4](#0-3) 

`LRTDepositPool.depositAsset` calls `_beforeDeposit` → `getRsETHAmountToMint`, which reads the cached `rsETHPrice` directly with no price refresh: [5](#0-4) [6](#0-5) 

The numerator (`lrtOracle.getAssetPrice(asset)`) is a live Chainlink read; the denominator (`lrtOracle.rsETHPrice()`) is the stale cached value. As staking rewards accrue between updates, the true rsETH price rises but the denominator stays low, causing the division to yield a larger-than-correct rsETH mint amount. The attacker then calls the public `updateRSETHPrice()` to crystallize the gain.

## Impact Explanation
**High — Theft of unclaimed yield.** An attacker who deposits while `rsETHPrice` is stale receives excess rsETH. After calling `updateRSETHPrice()`, the price corrects upward and the attacker holds rsETH worth more than their deposit. The delta is extracted directly from yield that had accrued to pre-existing rsETH holders but was not yet reflected in the cached price. The loss to existing holders equals the attacker's gain, scaled by deposit size and staleness duration.

## Likelihood Explanation
`updateRSETHPrice()` is not called atomically within `depositAsset`. Any interval between two consecutive price updates — which can be hours in practice, especially during low-activity periods or keeper delays — creates an exploitable window. rsETH accrues staking rewards continuously (~4–5% APY across multiple LSTs), so even a 1-hour staleness window produces a measurable gap. The attack requires no special role, no governance capture, and no external protocol compromise — only a public `depositAsset` call followed by a public `updateRSETHPrice` call.

## Recommendation
Call `_updateRsETHPrice()` (or compute the live price inline) before computing `rsethAmountToMint` inside `getRsETHAmountToMint`. Alternatively, derive the mint denominator from a freshly computed `totalETHInProtocol / rsethSupply` ratio at deposit time rather than reading the stored `rsETHPrice` state variable, so the denominator always reflects the current TVL/supply ratio.

## Proof of Concept
1. Assume `rsETHPrice` was last updated 6 hours ago at `1.040 ETH`. Staking rewards have since accrued; true price is `1.041 ETH`. Existing supply: `961.538 rsETH`, true TVL: `1000.96 ETH`.
2. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, ...)`.
3. `getRsETHAmountToMint` computes: `1000e18 * 1e18 / 1.040e18 = 961.538 rsETH` (stale denominator).
4. Fair amount at true price: `1000e18 * 1e18 / 1.041e18 = 960.615 rsETH`. Attacker receives `~0.923 rsETH` excess.
5. Post-deposit state: TVL = `2000.96 ETH`, supply = `1923.076 rsETH`.
6. Attacker calls `updateRSETHPrice()`. New price = `2000.96 / 1923.076 ≈ 1.04054 ETH`.
7. Attacker's rsETH value: `961.538 × 1.04054 ≈ 1000.52 ETH` on a `1000 ETH` deposit — `~0.52 ETH` profit.
8. Existing holders' rsETH value: `961.538 × 1.04054 ≈ 1000.44 ETH`, down from the `1000.96 ETH` they were owed — `~0.52 ETH` loss. The attacker's gain equals the existing holders' yield loss.

Foundry fork test plan: fork mainnet, snapshot `rsETHPrice`, advance time by 6 hours without calling `updateRSETHPrice`, deposit a large stETH amount, record rsETH minted, call `updateRSETHPrice`, assert attacker's rsETH value exceeds deposit value and existing holders' rsETH value decreased relative to pre-attack accrued yield.
