Audit Report

## Title
Public `updateRSETHPrice()` Enables Deposit/Price-Update/Instant-Withdrawal Sandwich to Steal Yield — (File: `contracts/LRTOracle.sol`)

## Summary
`LRTOracle.updateRSETHPrice()` is callable by any address with no access control beyond `whenNotPaused`. Because `rsETHPrice` is a cached state variable that grows stale as underlying LSTs accrue staking rewards, an attacker can deposit at the stale (lower) price to receive inflated rsETH shares, trigger the price update to the current higher value, and immediately call `instantWithdrawal()` to redeem those inflated shares at the updated price — extracting accrued yield that belongs to existing rsETH holders.

## Finding Description

**Root cause:** `updateRSETHPrice()` has no access control:

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [1](#0-0) 

`rsETHPrice` is a cached storage variable updated only when this function is explicitly called. Between calls, LSTs silently accrue staking rewards, causing the protocol's actual ETH value to exceed what `rsETHPrice` reflects.

**Deposit uses stale price:** `getRsETHAmountToMint()` divides by the stored `rsETHPrice`:

```solidity
// contracts/LRTDepositPool.sol L519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [2](#0-1) 

When `rsETHPrice` is stale-low, this mints more rsETH than the depositor is entitled to.

**Instant withdrawal uses the same stored price:** `getExpectedAssetAmount()` also reads `lrtOracle.rsETHPrice()`:

```solidity
// contracts/LRTWithdrawalManager.sol L592-593
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
``` [3](#0-2) 

After the attacker calls `updateRSETHPrice()`, this function pays out at the now-higher price.

**`pricePercentageLimit` guard is insufficient:** The guard only blocks price increases that exceed the configured threshold relative to `highestRsethPrice`:

```solidity
// contracts/LRTOracle.sol L256-257
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
``` [4](#0-3) 

The attack is profitable whenever the accrued yield fits within the allowed band (e.g., a 0.5% yield increase is well within a 1% daily limit). When `pricePercentageLimit == 0`, there is no bound at all.

**Exploit flow (concrete numbers):**

| State | rsETH Supply | ETH in Protocol | rsETHPrice |
|---|---|---|---|
| Before attack | 1,000 | 1,005 (rewards accrued) | 1.000 (stale) |
| After deposit 1,000 stETH | 2,000 | 2,005 | 1.000 (stale) |
| After `updateRSETHPrice()` | 2,000 | 2,005 | 1.0025 |
| After `instantWithdrawal(1,000 rsETH)` | 1,000 | 1,002.5 | 1.0025 |

Attacker deposited 1,000 stETH and withdrew 1,002.5 stETH → **+2.5 stETH profit**. Original holders' 1,000 rsETH dropped from 1,005 ETH to 1,002.5 ETH in value → **−2.5 ETH stolen yield**.

## Impact Explanation

**Theft of unclaimed yield (High).** The attacker extracts yield that was accruing to existing rsETH holders. The profit scales with deposit size and the staleness of `rsETHPrice`. The yield is permanently transferred out of the protocol; existing holders cannot recover it. This matches the allowed impact "Theft of unclaimed yield."

## Likelihood Explanation

- No special privilege is required; any EOA or contract can call `updateRSETHPrice()` and `instantWithdrawal()`.
- `rsETHPrice` naturally becomes stale between keeper calls; the longer the gap, the larger the exploitable discrepancy.
- The only preconditions are: (a) `isInstantWithdrawalEnabled[asset] == true` and (b) the `LRTUnstakingVault` holds sufficient assets for instant withdrawal — both of which are required for the instant withdrawal feature to function at all, making them realistic operational conditions.
- The `pricePercentageLimit` guard does not prevent the attack for yield accruals within the configured band, and provides zero protection when `pricePercentageLimit == 0`.

Likelihood: **Medium** (feature must be enabled; vault must have liquidity).

## Recommendation

1. **Restrict `updateRSETHPrice()`** to a trusted keeper role (e.g., `onlyLRTManager`) so that unprivileged callers cannot trigger price updates at will.
2. **Alternatively, add a per-block update guard** — revert if `lastUpdateBlock == block.number` — to prevent atomic deposit/update/withdraw sequences.
3. **Ensure `pricePercentageLimit` is always set to a non-zero value** to bound the maximum exploitable price jump per update.
4. **Compute rsETH price live** inside `getRsETHAmountToMint()` and `getExpectedAssetAmount()` so deposit and withdrawal always use a consistent, non-cacheable price within a single transaction.

## Proof of Concept

```solidity
// Preconditions:
// - rsETHPrice is stale at 1.000e18; actual protocol value implies 1.005e18
// - isInstantWithdrawalEnabled[stETH] == true
// - LRTUnstakingVault holds >= 1002.5 stETH available for instant withdrawal

// Step 1: Deposit 1000 stETH at stale price
// rsethAmountToMint = 1000e18 * 1e18 / 1.000e18 = 1000e18 rsETH (inflated by ~0.5%)
depositPool.depositAsset(stETH, 1000e18, 0, "");

// Step 2: Trigger price update — permissionless, no role required
lrtOracle.updateRSETHPrice();
// rsETHPrice is now 1.0025e18 (diluted by attacker's deposit, but still above stale price)

// Step 3: Instant withdrawal of all rsETH
// assetAmountUnlocked = 1000e18 * 1.0025e18 / 1e18 = 1002.5e18 stETH
withdrawalManager.instantWithdrawal(stETH, 1000e18, "");

// Net result: deposited 1000 stETH, received 1002.5 stETH → 2.5 stETH profit
// Existing holders' rsETH value reduced by 2.5 ETH
```
