Audit Report

## Title
Shared `RSETH.maxMintAmountPerDay` Limit Allows User Deposits to Permanently Defer Protocol Fee Minting — (`contracts/RSETH.sol`, `contracts/LRTOracle.sol`)

## Summary

`RSETH.mint()` enforces a single `checkDailyMintLimit` modifier against one shared `maxMintAmountPerDay` counter for every caller holding `MINTER_ROLE`. Both user deposits (via `LRTDepositPool._mintRsETH`) and protocol fee mints (via `LRTOracle._updateRsETHPrice`) route through this same gate. When user deposits exhaust `maxMintAmountPerDay` within a 24-hour window, the subsequent `IRSETH.mint(treasury, feeAmount)` call in `_updateRsETHPrice` reverts, blocking `rsETHPrice` from updating and permanently deferring the fee for that period. An unprivileged actor who repeatedly fills the daily cap across consecutive windows can indefinitely suppress protocol fee accrual.

## Finding Description

**Root cause:** `RSETH.mint()` applies `checkDailyMintLimit` unconditionally to all `MINTER_ROLE` callers with no priority or reservation for the fee-mint path.

**User deposit path (confirmed):**
`LRTDepositPool.depositETH()` → `_mintRsETH()` → `IRSETH(rsethToken).mint(msg.sender, rsethAmountToMint)` → `checkDailyMintLimit` increments `currentPeriodMintedAmount`. [1](#0-0) 

**Fee mint path (confirmed):**
`LRTOracle._updateRsETHPrice()` → `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` (oracle-level cap, passes) → `IRSETH(rsETHTokenAddress).mint(treasury, rsethAmountToMintAsProtocolFee)` → `checkDailyMintLimit` **reverts** if `currentPeriodMintedAmount + feeAmount > maxMintAmountPerDay`. [2](#0-1) 

**Why existing checks fail:** `_checkAndUpdateDailyFeeMintLimit` is an oracle-internal fee cap that runs *before* the `RSETH.mint()` call. It does not bypass or interact with `RSETH.checkDailyMintLimit`. When `RSETH.mint()` reverts at the token level, the entire `_updateRsETHPrice()` transaction reverts — `rsETHPrice` is never written (line 313) and the fee is never minted. [3](#0-2) [4](#0-3) 

The two-limit design (`maxFeeMintAmountPerDay` in `LRTOracle` and `maxMintAmountPerDay` in `RSETH`) creates a false sense of separation: the oracle-level gate can pass while the token-level gate blocks the actual mint.

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

Each blocked `updateRSETHPrice()` call defers the fee for that period. Because `rsETHPrice` is not updated on revert, the next successful call recalculates the fee from the same stale price, capturing the deferred TVL growth — so a single blocked day causes deferral, not permanent loss. However, an actor who fills `maxMintAmountPerDay` every day across consecutive 24-hour windows causes indefinite suppression of fee minting, which constitutes permanent freezing of unclaimed yield (protocol treasury yield) in practice. No privileged access is required to sustain this.

## Likelihood Explanation

- Requires `maxMintAmountPerDay` to be set to a finite non-zero value (necessary for the protocol to function at all, since `maxMintAmountPerDay == 0` causes every mint to revert).
- Requires user deposit volume to reach the daily cap before `updateRSETHPrice()` is called — achievable organically at scale or deliberately by a single large depositor.
- No privileged access needed; `depositETH()` and `depositAsset()` are fully public.
- Repeatable every 24-hour window with no additional cost beyond gas and deposit capital (which the attacker retains as rsETH).
- Likelihood: **low-to-medium** under normal conditions; **high** if a motivated actor targets it deliberately.

## Recommendation

Decouple fee minting from the user-deposit daily cap by one of:
1. **Exempt `LRTOracle` from `checkDailyMintLimit`**: add a separate `mintFee(address to, uint256 amount)` entry point in `RSETH` that is gated only by role and `_checkAndUpdateDailyFeeMintLimit`, bypassing `checkDailyMintLimit`.
2. **Reserve capacity**: track user-deposit mints and fee mints separately in `RSETH`, so the fee-mint sub-limit is never consumed by user deposits.
3. **Separate minter roles**: assign `LRTOracle` a distinct role (e.g., `FEE_MINTER_ROLE`) whose `mint` path skips `checkDailyMintLimit`.

## Proof of Concept

```solidity
// Preconditions:
// - maxMintAmountPerDay = 1000 ether (finite, non-zero)
// - periodStartTime is within the current 24h window
// - TVL has grown (staking rewards accrued), so protocolFeeInETH > 0

// Step 1: Attacker (or organic depositors) fills the daily cap
lrtDepositPool.depositETH{value: largeAmount}(0, "");
// RSETH.currentPeriodMintedAmount == maxMintAmountPerDay (1000 ether)

// Step 2: Anyone calls updateRSETHPrice()
lrtOracle.updateRSETHPrice();
// LRTOracle.sol:303 — _checkAndUpdateDailyFeeMintLimit(feeAmount) PASSES
// LRTOracle.sol:306 — IRSETH.mint(treasury, feeAmount)
//   RSETH.sol:50 — currentPeriodMintedAmount + feeAmount > maxMintAmountPerDay
//   => reverts: DailyMintLimitExceeded
// rsETHPrice NOT updated; treasury receives no fee for this period

// Step 3: Repeat every 24h window to permanently defer fee accrual
```

Repeating Step 1 each day keeps `currentPeriodMintedAmount` at the cap before any `updateRSETHPrice()` succeeds, indefinitely suppressing protocol fee minting. [3](#0-2) [5](#0-4)

```text
    modifier checkDailyMintLimit(uint256 amount) {
        // Check if we need to reset the period if it has been more than 24 hours
        if (block.timestamp >= periodStartTime + 1 days) {
            currentPeriodMintedAmount = 0;
            periodStartTime = getCurrentPeriodStartTime();
        }

        // Check if minting would exceed the daily limit
        if (currentPeriodMintedAmount + amount > maxMintAmountPerDay) {
            revert DailyMintLimitExceeded(currentPeriodMintedAmount + amount, maxMintAmountPerDay);
        }

        currentPeriodMintedAmount += amount;
        _;
    }
```
