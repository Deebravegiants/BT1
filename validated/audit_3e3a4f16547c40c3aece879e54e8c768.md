### Title
Fee-on-Transfer Token Accounting Divergence in `initTransfer` Enables Unbacked Cross-Chain Minting — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` accepts a caller-supplied `amount` parameter, transfers that amount from the sender, and emits it verbatim in the `InitTransfer` event — without verifying the actual balance received. For fee-on-transfer (or rebasing-down) ERC20 tokens, the contract receives fewer tokens than `amount`, yet the event records the full `amount`. The NEAR bridge reads that event value as ground truth and mints/releases the full `amount` on the destination chain, creating unbacked wrapped supply.

---

### Finding Description

In `OmniBridge.sol`, the native-ERC20 branch of `initTransfer` is:

```solidity
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount          // caller-controlled; actual received may be less
    );
}
``` [1](#0-0) 

Immediately after, the event is emitted with the same caller-supplied `amount`:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,   // <-- not the actual received balance
    ...
);
``` [2](#0-1) 

There is no balance-before / balance-after check anywhere in `initTransfer`. [3](#0-2) 

On the NEAR side, `fin_transfer_callback` decodes the proof and constructs the `TransferMessage` directly from `init_transfer.amount` — the value that came from the EVM event:

```rust
amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
``` [4](#0-3) 

NEAR then mints or releases the full `amount` to the recipient, even though the EVM bridge only holds `amount − transfer_fee`. The bridge is undercollateralized by exactly the fee taken by the token on every such transfer.

A symmetric but distinct path exists in `finTransfer`: if a rebasing token's supply contracts while it is held by the bridge, the `safeTransfer` call at line 351–354 will revert because the contract no longer holds `payload.amount`, permanently freezing the user's redemption after they have already burned tokens on the source chain. [5](#0-4) 

---

### Impact Explanation

**Fee-on-transfer path (Critical — unbacked supply):** Each `initTransfer` with a fee-on-transfer token mints more wrapped tokens on NEAR than are locked on EVM. An attacker who controls or deploys such a token can drain the bridge's reserves of any other token by cycling the inflated wrapped supply back through the bridge.

**Rebasing-down path (Critical — irreversible fund lock):** A user who bridges a rebasing token from NEAR to EVM burns their NEAR-side tokens. If the EVM-side balance has since decreased due to a rebase, `finTransfer` reverts. The nonce is marked `completedTransfers[nonce] = true` before the transfer executes, so the settlement cannot be retried, permanently locking the user's value. [6](#0-5) 

---

### Likelihood Explanation

Fee-on-transfer tokens are a well-known ERC20 variant (e.g., USDT on some chains, STA, PAXG). Any unprivileged user can call `initTransfer` with such a token — no special role or leaked key is required. The `initTransfer` function is public and only gated by the `PAUSED_INIT_TRANSFER` pause flag. [7](#0-6) 

The rebasing path requires the token to rebase downward after deposit, which is an autonomous on-chain event outside user control, but is a normal operational property of rebasing tokens (e.g., AMPL negative rebase). Likelihood is medium-high for any bridge deployment that does not explicitly block such tokens.

---

### Recommendation

1. **Short term:** In the native-ERC20 branch of `initTransfer`, record the contract's balance before and after the `safeTransferFrom` call and use the delta as the canonical `amount` in the event:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 received = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// use `received` (cast to uint128) in the event and downstream logic
```

2. **Long term:** Maintain an explicit allowlist of supported ERC20 tokens and reject fee-on-transfer and rebasing tokens at the protocol level, or document them as unsupported and enforce the restriction on-chain.

---

### Proof of Concept

1. Attacker deploys `FeeToken`, an ERC20 that deducts 10% on every transfer.
2. Attacker calls `OmniBridge.initTransfer(feeToken, 1000, 0, 0, nearRecipient, "")`.
3. `safeTransferFrom` moves 1000 tokens; `FeeToken` retains 100 as a fee; bridge receives 900.
4. `InitTransfer` event is emitted with `amount = 1000`.
5. Relayer submits the proof to NEAR `fin_transfer` → `fin_transfer_callback`.
6. NEAR mints 1000 wrapped `FeeToken` to `nearRecipient`. Bridge is undercollateralized by 100 per round trip.
7. Attacker repeats, accumulating wrapped tokens backed by nothing. They then bridge a legitimate token from NEAR to EVM, redeeming against the inflated supply, draining the bridge's legitimate ERC20 reserves. [1](#0-0) [8](#0-7)
