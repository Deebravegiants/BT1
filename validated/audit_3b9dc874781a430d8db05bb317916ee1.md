Audit Report

## Title
Unprivileged Caller Can Trigger Protocol-Wide Pause via Live Chainlink Price Drop — (`contracts/LRTOracle.sol`)

## Summary

`updateRSETHPrice()` is a public, permissionless function that internally pauses `LRTOracle`, `LRTDepositPool`, and `LRTWithdrawalManager` whenever the computed rsETH price falls below `highestRsethPrice` by more than `pricePercentageLimit`. Because the price is derived from live Chainlink spot feeds, any caller — including `address(1)` — can invoke the function during a natural market dip to freeze all user-facing deposit and withdrawal operations until an admin manually unpauses each contract.

## Finding Description

`updateRSETHPrice()` carries only the `whenNotPaused` modifier and no caller restriction: [1](#0-0) 

Inside `_updateRsETHPrice()`, the downside-protection block unconditionally pauses three contracts when the price drop exceeds the configured threshold: [2](#0-1) 

The price used for comparison is the live Chainlink spot price, fetched through `_getTotalEthInProtocol()` → `getAssetPrice()` → `IPriceFetcher.getAssetPrice()`: [3](#0-2) 

`LRTDepositPool.pause()` and `LRTWithdrawalManager.pause()` both require `PAUSER_ROLE`: [4](#0-3) [5](#0-4) 

For the oracle's internal calls to succeed, `LRTOracle` must hold `PAUSER_ROLE` on both contracts — which is the required deployment configuration given the code explicitly calls them. Once paused:

- `depositETH()` and `depositAsset()` revert via `whenNotPaused` on `LRTDepositPool`: [6](#0-5) 
- `initiateWithdrawal()` and `completeWithdrawal()` revert via `whenNotPaused` on `LRTWithdrawalManager`: [7](#0-6) 

Unpausing requires an admin to call `unpause()` on each of the three contracts individually: [8](#0-7) [9](#0-8) [10](#0-9) 

The root cause is the absence of any caller restriction on `updateRSETHPrice()`. The pause mechanism is intentional as a safety circuit, but it is incorrectly exposed to any external caller rather than being restricted to a trusted keeper or manager role. The vulnerability does not require oracle manipulation or incorrect oracle data — the oracle is functioning correctly and reporting the actual market price. The issue is purely an access control deficiency on the public function.

The SECURITY.md exclusion for "Incorrect data supplied by third-party oracles" does not apply here because the oracle is supplying correct data. The exclusion for "depegging of an external stablecoin" does not apply because stETH is not a stablecoin and the attack does not require a depeg — any price movement exceeding `pricePercentageLimit` (e.g., 1% intraday volatility) is sufficient.

## Impact Explanation

All user deposits (`depositETH`, `depositAsset`) and withdrawal initiations (`initiateWithdrawal`, `completeWithdrawal`, `instantWithdrawal`) are frozen until admin intervention. No funds are lost or stolen, but users cannot enter or exit the protocol for the duration of the pause. This constitutes **Medium — Temporary freezing of funds**.

## Likelihood Explanation

- The attacker requires no role, no capital, and no special setup — only a call to a public function.
- `pricePercentageLimit` is admin-configurable; at 1% (1e16), the threshold is reachable during normal intraday volatility on stETH/ETH or ETHx/ETH Chainlink feeds.
- The attacker can monitor the Chainlink feed off-chain and call `updateRSETHPrice()` precisely when the feed reports a price at or below the threshold, requiring no mempool manipulation.
- The attack is repeatable: after admin unpauses, the attacker can trigger the pause again on the next qualifying price observation.
- `highestRsethPrice` is a monotonically non-decreasing all-time-high, meaning even a small reversion from a recent peak is sufficient to trigger the check.

## Recommendation

1. **Restrict the caller**: Add `onlyLRTManager` (or a dedicated keeper role) to `updateRSETHPrice()`, keeping the permissionless path only for price reads via `rsETHPrice`.
2. **Use a TWAP or circuit-breaker cooldown**: Require the depressed price to persist for N blocks before triggering a pause, preventing single-block feed anomalies from halting the protocol.
3. **Separate the pause trigger from the price update**: Emit an event and let a privileged keeper decide whether to pause, rather than auto-pausing inside a public function.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Foundry fork test against mainnet fork
// Preconditions:
//   - pricePercentageLimit = 1e16 (1%)
//   - highestRsethPrice already set from prior updateRSETHPrice() calls
//   - LRTOracle holds PAUSER_ROLE on LRTDepositPool and LRTWithdrawalManager

contract AuditPoC is Test {
    LRTOracle oracle = LRTOracle(<deployed_oracle>);
    LRTDepositPool pool = LRTDepositPool(<deployed_pool>);
    LRTWithdrawalManager wm = LRTWithdrawalManager(<deployed_wm>);

    function testUnprivilegedPauseTrigger() public {
        // 1. Mock the stETH/ETH Chainlink feed to return a price 2% below
        //    the current highestRsethPrice (simulating a transient market dip).
        vm.mockCall(
            <chainlink_steth_feed>,
            abi.encodeWithSignature("latestRoundData()"),
            abi.encode(0, <depressed_price>, 0, block.timestamp, 0)
        );

        // 2. Any unprivileged address calls the public function.
        vm.prank(address(1));
        oracle.updateRSETHPrice();

        // 3. All three contracts are now paused.
        assertTrue(oracle.paused());
        assertTrue(pool.paused());
        assertTrue(wm.paused());

        // 4. User deposit reverts.
        vm.expectRevert();
        pool.depositETH{value: 1 ether}(0, "");

        // 5. User withdrawal initiation reverts.
        vm.expectRevert();
        wm.initiateWithdrawal(<asset>, 1 ether, "");
    }
}
```
