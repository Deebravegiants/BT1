### Title
State Mutation Before Deposit Check Allows Storage Cost Bypass in `submit_participant_info` - (File: `crates/contract/src/lib.rs`)

### Summary

`MpcContract::submit_participant_info` mutates persistent contract state by calling `tee_state.add_participant(...)` **before** verifying that the caller has attached sufficient deposit to cover the resulting storage cost. In NEAR, returning `Err(...)` from a `#[handle_result]` function does **not** revert state — only a `panic!` / `env::panic_str()` does. Consequently, any caller who supplies a valid attestation but an insufficient deposit will have their entry permanently written into `tee_state.stored_attestations` without paying for the storage they consumed.

### Finding Description

The vulnerable sequence in `submit_participant_info` is:

```
1. initial_storage = env::storage_usage()          // snapshot taken
2. self.tee_state.add_participant(...)              // ← PERSISTENT STATE MUTATION
3. caller_is_not_participant = self.voter_account().is_err()
4. if attestation_storage_must_be_paid_by_caller {
5.     storage_used = env::storage_usage() - initial_storage
6.     cost = storage_byte_cost * storage_used
7.     if attached < cost { return Err(InsufficientDeposit) }  // ← TOO LATE
8. }
```

Step 2 writes the new `NodeId` + attestation blob into `tee_state.stored_attestations` (a NEAR `IterableMap`, backed by on-chain storage). Steps 3–7 then decide whether payment is required and whether the attached deposit is enough. If the deposit is insufficient, `Err` is returned at step 7 — but NEAR has already committed the storage write from step 2. The entry persists on-chain with no payment collected.

The correct pattern (used elsewhere in the contract, e.g. `propose_update`) is to validate the deposit **before** any state mutation, or to `env::panic_str()` on failure so the runtime rolls back the write. [1](#0-0) 

### Impact Explanation

An attacker who can produce a valid attestation (i.e., a legitimate TEE node) can call `submit_participant_info` with `attached_deposit = 0`. The `add_participant` call succeeds and the entry is committed to `tee_state.stored_attestations`. The subsequent deposit check returns `Err(InsufficientDeposit)`, but the state write is irreversible. Repeating this across many accounts inflates the contract's on-chain storage without any corresponding NEAR payment, breaking the storage-accounting invariant that the contract explicitly tries to enforce. This maps to the **Medium** allowed impact: *"participant-state or contract execution-flow manipulation that breaks production safety/accounting invariants."*

### Likelihood Explanation

Any node operator running a TEE-attested MPC node can trigger this with a single crafted transaction (zero-deposit call to `submit_participant_info`). No collusion, no privileged access, and no threshold cooperation is required. The only prerequisite is a valid TEE attestation, which every legitimate participant already possesses.

### Recommendation

Move the deposit check **before** the call to `add_participant`, or replace `return Err(...)` with `env::panic_str(...)` so that the NEAR runtime rolls back the storage write on failure:

```rust
// Option A: check deposit first (requires pre-computing expected cost)
// Option B: panic instead of returning Err to force state rollback
if attached < cost {
    env::panic_str(&InvalidParameters::InsufficientDeposit { ... }.to_string());
}
```

Alternatively, restructure the function to compute the expected storage cost from the serialized size of the entry **before** inserting it, then verify the deposit, and only then call `add_participant`.

### Proof of Concept

1. Operator runs a valid TEE node and obtains a real attestation `A`.
2. Operator calls `submit_participant_info(A, tls_pk)` with `attached_deposit = 0 yoctoNEAR`.
3. `add_participant` succeeds → entry written to `tee_state.stored_attestations`.
4. `storage_used > 0`, `cost > 0`, `attached = 0 < cost` → `Err(InsufficientDeposit)` returned.
5. Transaction receipt shows failure, but `tee_state.stored_attestations` now contains the entry — verifiable via a view call to `tee_state` or by observing `env::storage_usage()` before and after.
6. Repeat with different TLS keys to grow contract storage indefinitely at zero cost. [2](#0-1) [3](#0-2)
