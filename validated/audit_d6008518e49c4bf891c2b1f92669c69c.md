### Title
Sub-Transaction ExUnits Excluded from Block-Level Resource Limit Check in Dijkstra Era - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs`)

### Summary

In the Dijkstra era, the block-level `maxBlockExUnits` check in `dijkstraBbodyTransition` calls `Alonzo.validateExUnits` which uses `foldMap totExUnits txs` over only the top-level transactions. The `totExUnits` function sums redeemers from the top-level transaction's witness set only, completely ignoring sub-transaction redeemers. Simultaneously, the `SUBUTXO` rule explicitly marks `ExUnitsTooBigUTxO` as "Impossible" and performs no per-sub-transaction ExUnits check. An unprivileged transaction author can therefore embed arbitrarily large ExUnits declarations across many sub-transactions, causing a block's actual Plutus execution budget to far exceed `maxBlockExUnits` while the ledger check passes.

### Finding Description

**Root cause — `totExUnits` does not recurse into sub-transactions:**

`totExUnits` is defined in `eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Tx.hs`:

```haskell
totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL
```

It reads only the top-level transaction's witness-set redeemers. Sub-transactions are stored in the transaction body (`dtbrSubTransactions :: OMap TxId (Tx SubTx era)`), not in the witness set, so they are invisible to `totExUnits`.

**Block-level check uses `totExUnits` exclusively:**

In `dijkstraBbodyTransition` (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs`, line 361):

```haskell
Alonzo.validateExUnits @era txs $ pp ^. ppMaxBlockExUnitsL
```

`txs` is `blockBody ^. txSeqBlockBodyL` — the sequence of top-level transactions. `Alonzo.validateExUnits` computes `foldMap totExUnits txs`, which sums only top-level redeemer ExUnits. Sub-transaction redeemers are never included.

**`SUBUTXO` explicitly omits the per-transaction ExUnits check:**

In `dijkstraSubUtxoTransition` (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`), the transition rule contains no call to `validateExUnitsTooBigUTxO`. The error-mapping function at line 340 confirms this is intentional:

```haskell
ExUnitsTooBigUTxO _ -> error "Impossible: `ExUnitsTooBigUTxO` for SUBUTXO"
```

**Sub-transaction Plutus scripts are executed:**

`DijkstraStAnnSubTx` carries `dsastPlutusScriptsWithContext :: Either (NonEmpty (CollectError era)) [PlutusWithContext]`, and `SUBUTXOW` evaluates those scripts (evidenced by `SubScriptWitnessNotValidatingUTXOW` in `DijkstraSubUtxowPredFailure`). The Plutus interpreter consumes up to the declared ExUnits budget for each sub-transaction script.

**Exploit path:**

1. Attacker constructs a top-level Dijkstra transaction with zero or minimal top-level redeemers (ExUnits = 0).
2. The top-level transaction body embeds N sub-transactions, each containing Plutus scripts with redeemers declaring ExUnits close to `maxTxExUnits` (e.g., `ExUnits {exUnitsMem = maxMem, exUnitsSteps = maxSteps}`).
3. A block producer includes this transaction in a block.
4. `dijkstraBbodyTransition` runs `Alonzo.validateExUnits txs (pp ^. ppMaxBlockExUnitsL)`: the computed total is ≈ 0 (top-level only), so the check passes.
5. `SUBLEDGERS` processes each sub-transaction through `SUBUTXOW` → `SUBUTXO`, executing Plutus scripts up to N × `maxTxExUnits` worth of computation — potentially many multiples of `maxBlockExUnits`.

### Impact Explanation

**Medium.** Attacker-controlled transactions exceed intended validation limits. The `maxBlockExUnits` protocol parameter is designed to bound the CPU and memory cost of validating a block, ensuring block validation time stays within the slot duration. By embedding sub-transactions with large ExUnits declarations, an attacker causes block validation to consume resources far beyond `maxBlockExUnits`, degrading node performance and potentially causing missed slot leadership or chain slowdown. This matches the allowed impact: *"Attacker-controlled transactions exceed intended validation limits."*

### Likelihood Explanation

**Medium.** The Dijkstra era is new and sub-transactions are a novel feature. Any unprivileged user who can submit a valid Dijkstra-era transaction can trigger this. The only prerequisite is constructing a top-level transaction with sub-transactions containing Plutus scripts — a normal, permissionless operation. No privileged access, governance majority, or key compromise is required.

### Recommendation

`Alonzo.validateExUnits` must be extended to aggregate sub-transaction ExUnits. One approach:

```haskell
-- In dijkstraBbodyTransition, replace:
Alonzo.validateExUnits @era txs $ pp ^. ppMaxBlockExUnitsL

-- With a version that sums top-level + all sub-transaction ExUnits:
let totalExUnits = foldMap (\tx ->
      totExUnits tx
        <> foldMap totExUnits (OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL)
    ) txs
in pointWiseExUnits (<=) totalExUnits (pp ^. ppMaxBlockExUnitsL)
     ?! injectFailure (TooManyExUnits ...)
```

Additionally, `dijkstraSubUtxoTransition` should enforce `maxTxExUnits` per sub-transaction, analogous to `validateExUnitsTooBigUTxO` in the top-level `UTXO` rule.

### Proof of Concept

**Setup:** Protocol parameters with `maxBlockExUnits = ExUnits {exUnitsMem = 62_000_000, exUnitsSteps = 20_000_000_000}` and `maxTxExUnits = ExUnits {exUnitsMem = 14_000_000, exUnitsSteps = 10_000_000_000}` (current mainnet values).

**Attack transaction:**
- Top-level tx: no redeemers, `totExUnits = ExUnits 0 0`.
- Sub-transactions: 10 sub-txs, each with one Plutus script redeemer declaring `ExUnits {exUnitsMem = 14_000_000, exUnitsSteps = 10_000_000_000}`.
- Total actual execution budget: `ExUnits {exUnitsMem = 140_000_000, exUnitsSteps = 100_000_000_000}` — roughly 2.3× `maxBlockExUnits` in memory and 5× in steps.

**Ledger check result:**
- `Alonzo.validateExUnits txs ppMax` computes `foldMap totExUnits txs = ExUnits 0 0 ≤ maxBlockExUnits` → **passes**.
- `SUBUTXO` for each sub-tx: no ExUnits check → **passes**.
- Block is accepted; Plutus execution consumes resources far exceeding `maxBlockExUnits`.
