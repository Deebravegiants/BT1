### Title
Cross-Chain Replay of `deployToken` Signature Permanently Blocks Legitimate Token Deployment — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
The `deployToken` signed payload omits the destination chain ID. Because the NEAR MPC uses a single derivation path (`"bridge-1"`) for all chains, the same `nearBridgeDerivedAddress` is shared across every EVM (and Aptos/Starknet) deployment. A `deployToken` signature that is valid on chain A is therefore equally valid on chain B. An unprivileged attacker can extract a publicly-broadcast signature from chain A and replay it on chain B, pre-deploying the token at an attacker-chosen moment and permanently poisoning the `nearToEthToken` mapping on chain B.

### Finding Description
`OmniBridge.deployToken` constructs its signed Borsh payload as:

```
PayloadType.Metadata | token | name | symbol | decimals
```

with no chain identifier anywhere in the preimage. [1](#0-0) 

The same chain-ID-free layout is used in the Aptos `metadata_to_borsh`: [2](#0-1) 

And in the Starknet `deploy_token`, which calls `payload.to_borsh()` without a chain-ID argument (contrast with `fin_transfer`, which explicitly passes `self.omni_bridge_chain_id.read()`): [3](#0-2) 

The `fin_transfer` path on every chain correctly binds the chain ID into the hash (EVM lines 294/297, Starknet line 253, Aptos line 450), but `deployToken` does not. [4](#0-3) 

Because the NEAR bridge signs with a single path: [5](#0-4) 

every chain instance shares the same `nearBridgeDerivedAddress`. A signature `S` produced for `deployToken(token="usdc.near", …)` on chain A is cryptographically indistinguishable from a fresh signature for the same call on chain B.

The only on-chain guard is:

```solidity
require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST");
``` [6](#0-5) 

This check is local to chain B's state. If the token has not yet been deployed on chain B, the replay succeeds unconditionally.

### Impact Explanation
After a successful replay on chain B:

1. `nearToEthToken[metadata.token]` is set to an attacker-timed proxy address `Z`.
2. The `ERR_TOKEN_EXIST` guard now permanently blocks any future legitimate `deployToken` call for that NEAR token ID on chain B.
3. The only way to bridge that token to chain B is to `bind_token` on NEAR against address `Z` — an address that was deployed at the attacker's chosen moment, not the protocol's.
4. If the NEAR operator never `bind_token`s to `Z`, the token's redemption path to chain B is permanently frozen.

This satisfies the allowed impact: **irreversible fund lock / frozen redemption path** — the legitimate deployment slot for the token on chain B is permanently consumed, and any users or protocol value that depends on bridging that token to chain B is unclaimable through the normal flow.

### Likelihood Explanation
- The attacker needs only a valid `deployToken` signature, which is publicly visible in any chain A transaction that legitimately deployed the same token.
- The call to `deployToken` on chain B is permissionless and requires no privileged access.
- The window is any time before the token is legitimately deployed on chain B — which for newly-supported chains or newly-listed tokens can be arbitrarily large.
- No special tooling is required beyond replaying a known calldata blob to a different RPC endpoint.

### Recommendation
Include `omniBridgeChainId` in the `deployToken` Borsh preimage, mirroring the pattern already used in `finTransfer`:

```solidity
// EVM deployToken — add chain binding
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),          // bind to this chain
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

Apply the same fix to Aptos `metadata_to_borsh` and Starknet `MetadataPayload.to_borsh()`. The NEAR MPC signing path must include the destination chain ID in the payload it signs for `deployToken`, matching the existing `TransferMessagePayload` pattern.

### Proof of Concept

1. Token `usdc.near` is legitimately deployed on EVM chain A (e.g., Ethereum mainnet). The `deployToken` transaction is public; extract `signatureData` and `metadata` from it.

2. Chain B (e.g., a new EVM L2 supported by the bridge) has not yet deployed `usdc.near`.

3. Attacker calls `OmniBridge.deployToken(signatureData, metadata)` on chain B using the extracted data.

4. `keccak256(borshEncoded)` on chain B is identical to chain A (no chain ID in preimage). `ECDSA.recover` returns `nearBridgeDerivedAddress`. Signature check passes.

5. A new `BridgeToken` proxy is deployed at address `Z` on chain B. `nearToEthToken["usdc.near"] = Z` and `isBridgeToken[Z] = true` are written.

6. Any subsequent legitimate `deployToken` call for `usdc.near` on chain B hits `require(!isBridgeToken[nearToEthToken["usdc.near"]], "ERR_TOKEN_EXIST")` and reverts permanently.

7. The redemption path for `usdc.near` on chain B is frozen unless the NEAR operator manually `bind_token`s to the attacker-timed address `Z`.
