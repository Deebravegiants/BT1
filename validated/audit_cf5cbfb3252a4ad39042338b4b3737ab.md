### Title
Off-by-One in Governance Action Expiry Check Allows Actions to Survive One Extra Epoch â€” (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs`)

---

### Summary

In `ratifyTransition`, the governance action expiry check uses strict `<` instead of `<=`. Because `reCurrentEpoch` is set to the epoch that just ended at the epoch boundary, an action whose `gasExpiresAfter` equals `reCurrentEpoch` is not expired when it should be. The action survives one extra epoch, during which it can be ratified and enacted even though its lifetime has elapsed.

---

### Finding Description

In `ratifyTransition`, after failing to ratify a governance action, the code checks whether the action has expired:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs, line 357
if gasExpiresAfter < reCurrentEpoch
  then pure $ st' & rsExpiredL %~ Set.insert gasId
  else pure st'
``` [1](#0-0) 

`gasExpiresAfter` is the last epoch during which the action is valid. It is set at proposal submission time as:

```haskell
gasExpiresAfter = addEpochInterval curEpoch expiryInterval
``` [2](#0-1) 

The RATIFY rule runs at the epoch boundary. At the `N â†’ N+1` boundary, `reCurrentEpoch = N` (the epoch that just ended), as confirmed by the test helper:

```haskell
reCurrentEpoch = eNo - 1   -- eNo is the new epoch after the boundary
``` [3](#0-2) 

**The boundary case**: For an action with `gasExpiresAfter = N`, at the `N â†’ N+1` boundary:

- `reCurrentEpoch = N`
- Check: `N < N` â†’ **False** â†’ action is **not** expired

The action should be expired here (it expires *after* epoch N, so the `N â†’ N+1` boundary is the correct removal point), but the strict `<` causes it to survive into epoch N+1.

This is directly inconsistent with the voting guard in the GOV rule, which correctly uses `<=`:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Gov.hs, line 361-362
checkVotesAreNotForExpiredActions curEpoch votes =
  checkDisallowedVotes votes VotingOnExpiredGovAction $ \GovActionState {gasExpiresAfter} _ ->
    curEpoch <= gasExpiresAfter
``` [4](#0-3) 

GOV blocks new votes in epoch N+1 (correct), but RATIFY does not expire the action at the `N â†’ N+1` boundary (incorrect). The action persists into epoch N+1 with its accumulated votes intact and can be ratified at the `N+1 â†’ N+2` boundary.

---

### Impact Explanation

A governance action that should have been removed at the `gasExpiresAfter â†’ gasExpiresAfter+1` epoch boundary instead survives one additional epoch. During that extra epoch:

- No new votes can be cast (GOV correctly blocks them).
- The action retains all previously cast votes.
- RATIFY at the next boundary (`gasExpiresAfter+1 â†’ gasExpiresAfter+2`) evaluates the action for ratification **before** the expiry check. If `acceptedByEveryone` is satisfied and `rsDelayed = False`, the action is **enacted**.

Concretely, any of the following governance actions could be enacted one epoch after their intended expiry:

- `TreasuryWithdrawals` â€” ADA withdrawn from treasury outside design parameters.
- `HardForkInitiation` â€” unauthorized protocol version bump.
- `UpdateCommittee` â€” unauthorized committee composition change.
- `NewConstitution` â€” unauthorized constitution replacement.
- `ParameterChange` â€” unauthorized protocol parameter modification.

This matches the allowed impact: **"Unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action is enacted"** (Critical) and **"Attacker-controlled transactions modify fees, deposits, refunds, rewards, treasury donations, or withdrawals outside design parameters"** (Medium).

---

### Likelihood Explanation

The extra-epoch survival is unconditional â€” every governance action that reaches its `gasExpiresAfter` epoch without being ratified is affected. The concrete exploitation path requires:

1. A governance action accumulates sufficient votes to satisfy `acceptedByEveryone` by epoch `gasExpiresAfter`.
2. The action is **not** ratified in epoch `gasExpiresAfter` â€” this happens naturally when `rsDelayed = True` (set by any concurrently ratified delaying action: `HardForkInitiation`, `UpdateCommittee`, `NoConfidence`, `NewConstitution`).
3. `rsDelayed` is reset at the next epoch boundary (documented in `RatifyState`), so the action is ratified in the extra epoch. [5](#0-4) 

The `rsDelayed` scenario is a normal, expected governance event (e.g., a hard-fork ratified in the same epoch as a treasury withdrawal). No privileged key or supermajority beyond normal governance thresholds is required. Likelihood is **Medium**.

---

### Recommendation

Change the strict `<` to `<=` in `ratifyTransition`:

```haskell
-- Before (buggy):
if gasExpiresAfter < reCurrentEpoch

-- After (correct):
if gasExpiresAfter <= reCurrentEpoch
``` [6](#0-5) 

This aligns the RATIFY expiry check with the GOV voting guard (`curEpoch <= gasExpiresAfter`) and ensures actions are removed at the correct epoch boundary.

### Proof of Concept

**Setup**: `ppGovActionLifetime = 2`, starting at epoch 0.

| Epoch | Event |
|-------|-------|
| 0 | Action A (e.g., `TreasuryWithdrawals`) submitted â†’ `gasExpiresAfter = 2` |
| 1â€“2 | Sufficient votes accumulated; `acceptedByEveryone = True` |
| 2 | Delaying action B (`HardForkInitiation`) also ratified â†’ `rsDelayed = True`; Action A not ratified |
| 2â†’3 boundary | RATIFY: `2 < 2` = False â†’ Action A **not** expired (bug); `rsDelayed` reset |
| 3â†’4 boundary | RATIFY: `rsDelayed = False`, `acceptedByEveryone = True` â†’ Action A **ratified and enacted** |

Without the bug, Action A would have been added to `rsExpired` at the 2â†’3 boundary and never ratified. With the bug, it is enacted at the 3â†’4 boundary â€” one epoch after its intended expiry. [7](#0-6)
