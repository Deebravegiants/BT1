### Title
`respond_verify_foreign_tx` Accepts Caller-Supplied `payload_hash` Without Verifying It Is Derived From the Submitted `request` — (`File: crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted ECDSA signature is valid over the caller-supplied `response.payload_hash`, but **never verifies that `payload_hash` is actually the hash of `(request, actual_extracted_values)`** for the specific `request` being resolved. A single malicious attested MPC node can take a threshold signature legitimately produced for request `R1` and use it to permanently resolve a different pending request `R2`, injecting a wrong `payload_hash` into the yield response. This is the direct analog of the external report's root cause: the code assumes a 1:1 relationship between an input and a downstream value, but never enforces it.

---

### Finding Description

In `respond_verify_foreign_tx` (`crates/contract/src/lib.rs`, lines 692–754), the contract performs the following check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract verifies that `signature` is valid over `payload_hash` using the domain's shared public key. It then resolves all pending yields for `request` with the full `response`:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

**What is missing**: the contract never computes or checks that `response.payload_hash == SHA256(borsh(ForeignTxSignPayloadV1 { request: request, values: actual_values }))`. The `payload_hash` is entirely caller-supplied. The contract assumes a 1:1 relationship between the submitted `request` and the `payload_hash`, but never enforces it.

By contrast, the node-side code correctly derives `payload_hash` from the actual extracted values:

```rust
let response_payload = self.execute_foreign_chain_request(...).await?;
let sign_request = build_signature_request(&foreign_tx_request, &response_payload)?;
``` [3](#0-2) 

And `build_signature_request` computes the hash from the payload:

```rust
let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
    foreign_tx_payload.compute_msg_hash()?.into();
``` [4](#0-3) 

The on-chain contract never replicates this derivation check. The SDK's `ForeignChainSignatureVerifier::verify_signature` does perform this check client-side:

```rust
let expected_payload_hash = expected_payload.compute_msg_hash()?;
let payload_is_correct = expected_payload_hash == response.payload_hash;
``` [5](#0-4) 

But this is a client-side SDK helper, not an on-chain enforcement. The contract itself never performs this check.

---

### Impact Explanation

**Attack scenario (cross-request replay by a single attested node):**

1. User A submits `verify_foreign_transaction(R1)` and User B submits `verify_foreign_transaction(R2)` — both are queued in `pending_verify_foreign_tx_requests`.
2. The MPC network legitimately processes `R1` and produces a threshold signature `sig_H` over `H = hash(ForeignTxSignPayloadV1 { request: R1, values: actual_values_R1 })`. This is normal operation — no collusion required.
3. A single malicious attested node M (below threshold) observes `sig_H` (it participated in the signing protocol for `R1`) and submits `respond_verify_foreign_tx(R2, { payload_hash: H, signature: sig_H })`.
4. The contract checks: (a) M is an attested participant ✓, (b) `sig_H` is valid over `H` ✓, (c) `R2` exists in pending requests ✓. All checks pass.
5. The yield for `R2` is **permanently resolved** with `{ payload_hash: H, signature: sig_H }`.
6. User B's bridge contract receives a response where `payload_hash` is the hash of `R1`'s data, not `R2`'s.
7. If User B's bridge contract uses `ForeignChainSignatureVerifier::verify_signature`, it detects the mismatch and rejects the response — but the yield is already consumed and cannot be re-resolved. User B's request is **permanently frozen**.
8. If User B's bridge contract does not verify `payload_hash` against expected values, it accepts a signature over fabricated foreign-chain data — enabling **invalid bridge execution**.

This matches the allowed impact: "High. Cross-chain replay, forged foreign-chain verification … that causes invalid bridge execution" and "Medium. Balance, request-lifecycle … manipulation that breaks production safety/accounting invariants."

---

### Likelihood Explanation

The attack requires only **one** malicious attested MPC node — well below the signing threshold. The attested node participates in the normal MPC signing protocol for `R1`, observes the resulting threshold signature, and races to submit it as a response for `R2`. No threshold collusion is needed; the threshold signature is produced during legitimate operation. The race window exists because `respond_verify_foreign_tx` is callable by any single attested participant.

---

### Recommendation

The contract should recompute `payload_hash` from the submitted `request` and the extracted values, or at minimum enforce that `payload_hash` is structurally bound to `request`. Since the contract does not receive `values` directly, the simplest fix is to include the `request` in the signed payload in a way the contract can verify on-chain — for example, by requiring the node to also submit the `values` alongside the response, and having the contract compute `expected_hash = SHA256(borsh(ForeignTxSignPayloadV1 { request, values }))` and assert `expected_hash == response.payload_hash` before accepting the response.

Alternatively, the `payload_hash` could be required to commit to the `request` key in a way the contract can check without knowing `values` (e.g., a domain-separated hash that includes the request hash as a prefix), so that a signature for `R1` cannot be replayed for `R2`.

### Proof of Concept

```
1. Deploy contract with two pending verify_foreign_tx requests:
   - R1 = { tx_id: [0x01; 32], chain: Bitcoin, extractors: [BlockHash] }
   - R2 = { tx_id: [0x02; 32], chain: Bitcoin, extractors: [BlockHash] }

2. MPC nodes legitimately sign H = hash(ForeignTxSignPayloadV1 { request: R1, values: [BlockHash([0xAA; 32])] })

3. Malicious attested node M calls:
   respond_verify_foreign_tx(
     request = R2,
     response = { payload_hash: H, signature: sig_H }

4. Contract accepts: sig_H is valid over H, R2 exists in pending map.

5. R2's yield is resolved with { payload_hash: H, signature: sig_H }.

6. User B's bridge contract receives this response. If it calls
   ForeignChainSignatureVerifier::verify_signature, it computes
   expected_hash = hash(ForeignTxSignPayloadV1 { request: R2, values: expected_R2_values })
   and finds expected_hash != H → rejects.
   R2 is permanently frozen; no legitimate response can ever be submitted.

    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }

        let mut hasher = sha2::Sha256::new();

        Ok(Hash256(hasher.finalize().into()))
