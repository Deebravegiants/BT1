Audit Report

## Title
Immutable Extension With Reverting External Dependency Permanently Bricks All Pool Operations, Locking LP Funds — (`metric-core/contracts/ExtensionCalling.sol`)

## Summary

`ExtensionCalling._callExtensionsInOrder` dispatches to each configured extension via `CallExtension.callExtension` with no error isolation. All seven extension addresses are stored as constructor-time `immutable` values with no governance path to replace them. If any extension's external dependency reverts post-deployment, every call to `removeLiquidity`, `addLiquidity`, and `swap` permanently reverts, locking LP principal with no recovery path.

## Finding Description

`_callExtensionsInOrder` iterates through configured extension slots and calls each one:

```solidity
// ExtensionCalling.sol L75-86
function _callExtensionsInOrder(uint256 order, bytes memory data) private {
    if (order == 0) return;
    while (true) {
        uint256 extensionIndex = order & 0x7;
        if (extensionIndex == 0) break;
        address extension = _extensionAddress(extensionIndex);
        if (extension == address(0)) revert PanicEmptyExtension();
        CallExtension.callExtension(extension, data);   // ← no try/catch
        order >>= 3;
    }
}
```

`CallExtension.callExtension` explicitly propagates any revert from the extension back to the caller — there is no swallowing or isolation:

```solidity
// CallExtension.sol L8-17
function callExtension(address extension, bytes memory data) internal {
    (bool success, bytes memory result) = extension.call(data);
    if (!success) {
        if (result.length > 0) {
            assembly ("memory-safe") { revert(add(result, 32), mload(result)) }
        }
        revert ExtensionCallFailed();
    }
    ...
}
```

All seven extension addresses are set once in the constructor and are permanently immutable:

```solidity
// ExtensionCalling.sol L17-23
address internal immutable EXTENSION_1;
...
address internal immutable EXTENSION_7;
```

`removeLiquidity` unconditionally calls both `_beforeRemoveLiquidity` and `_afterRemoveLiquidity` with no `whenNotPaused` guard and no bypass path:

```solidity
// MetricOmmPool.sol L199-212
function removeLiquidity(...) external nonReentrant(PoolActions.REMOVE_LIQUIDITY) ... {
    ...
    _beforeRemoveLiquidity(msg.sender, owner, salt, deltas, extensionData);
    (amount0Removed, amount1Removed) = LiquidityLib.removeLiquidity(...);
    _afterRemoveLiquidity(msg.sender, owner, salt, deltas, amount0Removed, amount1Removed, extensionData);
}
```

`setPause` only sets `pauseLevel`, which is checked by the `whenNotPaused` modifier. Since `removeLiquidity` does not carry `whenNotPaused`, pausing the pool does not help LPs recover funds when extensions revert. There is no factory function to replace or disable an extension slot.

## Impact Explanation

If any extension registered in `BEFORE_REMOVE_LIQUIDITY_ORDER` or `AFTER_REMOVE_LIQUIDITY_ORDER` reverts (e.g., because an external oracle it calls is upgraded, deprecated, or self-destructed), every call to `removeLiquidity` permanently reverts. LP token0/token1 balances tracked in `binTotals` become permanently inaccessible. The same applies to `swap` and `addLiquidity` via their respective hook orders. This is a direct, permanent loss of LP principal with no on-chain recovery path — matching the "broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows" allowed impact.

## Likelihood Explanation

Extensions are set at pool-creation time and are explicitly designed to call external contracts (oracles, velocity guards, whitelists per the protocol design). Any of those external contracts can fail post-deployment through routine upgrades, emergency freezes, or exploits — all unprivileged external events requiring no action from the pool creator after deployment. The trigger is a standard operational event (oracle ABI change, contract pause, self-destruct) that is entirely outside the pool's control.

## Recommendation

1. Wrap extension calls in `try/catch` inside `_callExtensionsInOrder`. On revert, either emit an event and skip the extension, or revert with a structured error the factory can act on.
2. Add a factory-controlled per-extension-slot emergency-disable flag so a broken extension can be bypassed without redeploying the pool.
3. Provide an extension-bypass path for `removeLiquidity` (e.g., when `pauseLevel == 2`) so LPs can always recover principal even if hooks are broken.

## Proof of Concept

1. Pool is deployed with `EXTENSION_1` pointing to a velocity-guard contract that calls an external price oracle at address `O`.
2. `BEFORE_REMOVE_LIQUIDITY_ORDER` encodes extension index 1.
3. Oracle `O` is upgraded; its `getPrice()` selector changes, causing the extension to revert on every call.
4. Any LP calls `removeLiquidity` → `_beforeRemoveLiquidity` → `_callExtensionsInOrder` → `CallExtension.callExtension(EXTENSION_1, ...)` → extension calls `O.getPrice()` → reverts → `callExtension` re-reverts → `removeLiquidity` reverts.
5. `EXTENSION_1` is immutable; no factory function can replace it.
6. `setPause` sets `pauseLevel` but `removeLiquidity` has no `whenNotPaused` guard, so pausing does not create a bypass.
7. All LP token0/token1 balances in `binTotals` are permanently inaccessible. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
