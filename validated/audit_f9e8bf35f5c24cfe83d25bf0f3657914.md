Audit Report

## Title
Excess ETH Permanently Locked in `MultiChainRateProvider.updateRate()` Due to Missing Refund Mechanism - (File: contracts/cross-chain/MultiChainRateProvider.sol)

## Summary
`updateRate()` is a `payable` function that forwards exactly `estimatedFee` per destination chain to LayerZero, but has no mechanism to return any remaining `msg.value` to the caller. The contract declares no `receive()`, no `fallback()`, and no owner-callable ETH sweep, so any ETH sent beyond the sum of per-chain fees is permanently locked in the contract.

## Finding Description
In `contracts/cross-chain/MultiChainRateProvider.sol` at lines 108–137, `updateRate()` is marked `external payable nonReentrant` with no access control. Inside the loop (lines 119–134), it calls `estimateFees()` on-chain to obtain `estimatedFee` for each destination, then calls `ILayerZeroEndpoint.send{ value: estimatedFee }(...)` forwarding exactly that amount. The `_refundAddress` passed to LayerZero is `payable(msg.sender)` (line 128), which causes LayerZero to refund any unused portion of the `estimatedFee` it received — but LayerZero only controls the ETH it was given. Any ETH in `msg.value` that was never forwarded to LayerZero (i.e., `msg.value − Σ estimatedFee`) remains in the `MultiChainRateProvider` contract itself.

The full contract (lines 1–182) contains no `receive()` function, no `fallback()` function, and no owner-callable ETH withdrawal utility. There is no code path that can move this stranded ETH out of the contract. The `estimateTotalFee()` view function (lines 154–173) exists to help callers pre-compute the required amount, but callers who add a safety buffer — a standard defensive practice — will permanently lose the buffer on every call.

## Impact Explanation
Any ETH sent to `updateRate()` in excess of `Σ estimatedFee` is permanently frozen in the contract with no recovery path. This constitutes permanent freezing of funds for every caller who over-estimates the required fee. The impact maps to **Critical — Permanent freezing of funds** per the allowed scope, though the magnitude per call is bounded by the caller's chosen buffer.

## Likelihood Explanation
`updateRate()` has no access control and is expected to be called regularly for rate updates. Callers who wish to ensure the transaction does not revert due to fee fluctuations between the off-chain `estimateTotalFee()` query and the on-chain execution commonly add a small buffer. Every such call with `msg.value > Σ estimatedFee` results in the excess being permanently locked. The function is public, the pattern is common, and the loss accumulates over the protocol's lifetime.

## Recommendation
Add an ETH refund to `msg.sender` at the end of `updateRate()` for any unused balance:

```solidity
function updateRate() external payable nonReentrant {
    // ... existing loop ...
    uint256 remaining = address(this).balance;
    if (remaining > 0) {
        (bool ok,) = payable(msg.sender).call{value: remaining}("");
        require(ok, "ETH refund failed");
    }
}
```

Alternatively, compute the total required fee before the loop and `require(msg.value == totalFee)` to prevent over-payment entirely.

## Proof of Concept
1. Deploy `MultiChainRateProvider` with two registered rate receivers on different chains.
2. Call `estimateTotalFee()` off-chain; receive `X` wei.
3. Call `updateRate{value: X + 1000}()` (1000 wei safety buffer).
4. The loop forwards exactly `estimatedFee_1 + estimatedFee_2 = X` to LayerZero.
5. The remaining 1000 wei stays in the contract.
6. Confirm with `address(multiChainRateProvider).balance == 1000`.
7. Attempt any function call to recover the ETH — none exists. The ETH is permanently locked.
