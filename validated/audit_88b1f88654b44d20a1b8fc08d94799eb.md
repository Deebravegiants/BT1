Audit Report

## Title
1:1 Cross-Token Swap Without Value Equivalence Check Enables Direct Fund Theft - (`contracts/L2/RsETHTokenWrapper.sol`)

## Summary
`RsETHTokenWrapper` supports multiple allowed rsETH-variant tokens and permits any unprivileged caller to deposit one allowed token and withdraw a different allowed token at a strict 1:1 amount ratio with no price equivalence check. If two allowed tokens trade at different market prices — a realistic condition given the protocol explicitly supports multiple bridge variants — an attacker can deposit the cheaper token and withdraw the more expensive one, draining the wrapper's reserves of the higher-valued asset.

## Finding Description
The contract maintains an `allowedTokens` mapping and adds tokens via `_addAllowedToken`, called both at `initialize` and at `reinitialize` (line 47–48), confirming multi-token support is an intentional, deployed feature.

The public `deposit`/`depositTo` and `withdraw`/`withdrawTo` functions carry no access control:

```solidity
// lines 69–94
function deposit(address asset, uint256 _amount) external { _deposit(asset, msg.sender, _amount); }
function depositTo(address asset, address _to, uint256 _amount) external { _deposit(asset, _to, _amount); }
function withdraw(address asset, uint256 _amount) external { _withdraw(asset, msg.sender, _amount); }
function withdrawTo(address asset, address _to, uint256 _amount) external { _withdraw(asset, _to, _amount); }
```

`_deposit` (lines 134–141) accepts any `allowedToken`, transfers `_amount` in, and mints exactly `_amount` of wrsETH. `_withdraw` (lines 120–128) accepts any `allowedToken`, burns `_amount` of wrsETH, and transfers exactly `_amount` of the requested token out. Neither function checks that the deposited and withdrawn tokens have the same market value. The only guard is `if (!allowedTokens[_asset]) revert TokenNotAllowed()`, which both pass as long as both tokens are in the allowed set.

An attacker can atomically: (1) call `deposit(cheapToken, amount)` to receive `amount` wrsETH, then (2) call `withdraw(expensiveToken, amount)` to redeem that wrsETH for the more valuable asset. The wrapper's balance of the expensive token is reduced and replaced with the cheap token, permanently degrading the collateral backing all remaining wrsETH holders.

## Impact Explanation
**Critical — Direct theft of user funds.** The wrapper holds rsETH-variant tokens deposited by users and bridgers. Any price divergence between two allowed tokens, however small or temporary, is immediately exploitable by any unprivileged caller with no special permissions. The stolen value comes directly from other users' deposits; remaining wrsETH is backed only by the cheaper asset after the attack.

## Likelihood Explanation
**Medium.** The protocol explicitly deploys with at least two allowed tokens (canonical rsETH via LayerZero OFT and an alternative bridge rsETH added via `reinitialize`). Different bridge implementations carry different trust assumptions, liquidity profiles, and redemption mechanisms, meaning secondary-market prices routinely diverge — even transiently. No capital is at risk beyond gas; the attacker acquires the cheap token, executes the swap, and profits the spread. The attack is repeatable and requires no privileged access, no oracle manipulation, and no victim interaction.

## Recommendation
1. **Single canonical token per wrapper**: Remove multi-token deposit/withdraw and allow only one rsETH variant per wrapper instance.
2. **Per-token accounting**: Track wrsETH minted per deposited token type and enforce that withdrawals can only redeem the same token type that was deposited.
3. **Oracle-gated swap**: Before a cross-token swap, verify via an on-chain oracle that both tokens are within an acceptable price tolerance.

## Proof of Concept
Preconditions:
- `allowedTokens[rsETH_A] = true` (market price $2,000)
- `allowedTokens[rsETH_B] = true` (market price $2,050)
- Wrapper holds 100 `rsETH_B` (deposited by bridger)

Attack sequence (single transaction):
1. Attacker acquires 100 `rsETH_A` for $200,000.
2. `deposit(rsETH_A, 100e18)` → wrapper receives 100 `rsETH_A`, mints 100 wrsETH to attacker. *(line 134–141)*
3. `withdraw(rsETH_B, 100e18)` → wrapper burns 100 wrsETH, transfers 100 `rsETH_B` to attacker. *(line 120–128)*
4. Attacker holds 100 `rsETH_B` worth $205,000 — net profit $5,000 (2.5%).

Foundry invariant test plan: deploy wrapper with two mock ERC20 tokens both in `allowedTokens`; seed wrapper with token B; fuzz `deposit(tokenA, x)` followed by `withdraw(tokenB, x)`; assert `wrapper.balanceOf(tokenB) >= initial` — this invariant will break immediately.         if (!allowedTokens[_asset]) revert TokenNotAllowed();

        _burn(msg.sender, _amount);

        ERC20Upgradeable(_asset).safeTransfer(_to, _amount);

        emit Withdraw(_asset, msg.sender, _to, _amount);
    }

    /// @notice Deposit tokens into the lockbox
    /// @param _asset The address of the token to deposit
    /// @param _to The address to send the XERC20 to
    /// @param _amount The amount of tokens to deposit
    function _deposit(address _asset, address _to, uint256 _amount) internal {
        if (!allowedTokens[_asset]) revert TokenNotAllowed();

        ERC20Upgradeable(_asset).safeTransferFrom(msg.sender, address(this), _amount);

        _mint(_to, _amount);
        emit Deposit(_asset, msg.sender, _to, _amount);
    }
