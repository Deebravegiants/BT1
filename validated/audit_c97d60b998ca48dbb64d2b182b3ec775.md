### Title
`is_caller_an_attested_participant` Does Not Check Attestation Expiry, Allowing Nodes with Lapsed TEE Attestations to Submit Responses - (File: `crates/contract/src/tee/tee_state.rs`)

---

### Summary

The `is_caller_an_attested_participant` function verifies that a stored attestation *exists* for a calling participant but never calls `re_verify()` to confirm the attestation has not expired. This is the direct NEAR MPC analog of the reported Chainlink pattern: data is trusted without checking whether the underlying source is still live. A node whose TEE attestation has lapsed can continue to call `respond()`, `respond_ckd()`, and `respond_verify_foreign_tx()` — bypassing the TEE security invariant that every active signing participant must be running verified, trusted code.

---

### Finding Description

**Root cause — `is_caller_an_attested_participant` omits the expiry check**

`is_caller_an_attested_participant` (tee_state.rs:469-498) performs four checks:

1. Caller is in the active participant set.
2. A `NodeAttestation` entry exists for the participant's TLS key.
3. The stored `account_id` matches the signer.
4. The stored `account_public_key` matches the signer's Ed25519 key.

It does **not** call `re_verify()`. [1](#0-0) 

By contrast, `reverify_participants` — the function used by `verify_tee()` — explicitly calls `re_verify()` with the current block timestamp, which rejects any attestation whose `expiry_timestamp_seconds` is in the past: [2](#0-1) 

`re_verify()` on a `Dstack` attestation checks `expiry_timestamp_seconds < timestamp_seconds` and returns `Err` if the certificate has expired: [3](#0-2) 

**Propagation — three privileged entry points use the incomplete check**

`assert_caller_is_attested_participant_and_protocol_active` delegates directly to `is_caller_an_attested_participant`: [4](#0-3) 

This guard is the sole attestation check in all three response submission paths:

- `respond()` — line 573 [5](#0-4) 
- `respond_ckd()` — line 666 [6](#0-5) 
- `respond_verify_foreign_tx()` — line 705 [7](#0-6) 

**The `accept_requests` flag does not close the gap**

Each `respond*` function also checks `self.accept_requests`. However, that flag is only set to `false` when `verify_tee()` is explicitly called by a participant and finds insufficient valid attestations. [8](#0-7) 

`verify_tee()` is never called automatically on attestation expiry. Between the moment an attestation lapses and the moment some participant manually invokes `verify_tee()`, `accept_requests` remains `true` and the expired-attestation node can freely call all three `respond*` methods.

---

### Impact Explanation

**Impact: Medium** — participant-state invariant broken without requiring network-level DoS or operator misconfiguration.

The TEE attestation is the on-chain proof that a node is running the approved, unmodified MPC image inside a genuine TDX enclave. The production safety invariant is: *every node that submits a response must hold a currently-valid TEE attestation*. `is_caller_an_attested_participant` is the per-call enforcement point for this invariant, but it enforces only identity binding, not freshness.

A node whose attestation has expired — and which may therefore be running modified or untrusted code — can continue to call `respond()`, `respond_ckd()`, and `respond_verify_foreign_tx()` for the entire window between expiry and the next `verify_tee()` invocation. In the worst case, if no participant calls `verify_tee()`, this window is unbounded.

For `respond_verify_foreign_tx()` in particular, the node's foreign-chain RPC result is trusted as the basis for the signed observation. A node running tampered code could manipulate that result before the threshold signing step, contributing a corrupted observation to the protocol. While a single node cannot forge the final threshold signature alone, it can corrupt its share of the foreign-chain verification, potentially influencing the signed output if it is one of the threshold-many participants selected for that request.

---

### Likelihood Explanation

**Likelihood: Medium.**

- Dstack attestations carry a hardcoded `DEFAULT_EXPIRATION_DURATION_SECONDS` of 7 days (proposed to be lowered to 1 day). [9](#0-8) 
- Honest nodes resubmit every hour, so under normal operation attestations stay fresh. However, a node that deliberately stops resubmitting (or whose PCCS connectivity fails) will have an expired attestation within 1–7 days.
- `verify_tee()` is not triggered automatically; it requires an explicit participant call. There is no on-chain scheduler or heartbeat that enforces periodic re-validation.
- The attacker-controlled entry path is fully reachable by an unprivileged node operator: stop renewing the attestation, wait for expiry, continue calling `respond*`.

---

### Recommendation

Add an expiry check inside `is_caller_an_attested_participant` by calling `re_verify()` on the retrieved `NodeAttestation` before returning `Ok(())`. The required inputs (current block timestamp, allowed image hashes, allowed launcher hashes, allowed measurements) are already available on `TeeState`. Concretely:

```rust
// After confirming key matches, add:
let timestamp_seconds = Self::current_time_seconds();
let allowed_mpc_hashes = self.get_allowed_mpc_docker_image_hashes(tee_upgrade_deadline_duration);
let allowed_launcher_hashes = self.get_allowed_launcher_compose_hashes();
let allowed_measurements = self.get_accepted_measurements();

attestation.verified_attestation
    .re_verify(timestamp_seconds, &allowed_mpc_hashes, &allowed_launcher_hashes, &allowed_measurements)
    .map_err(|_| AttestationCheckError::AttestationExpiredOrInvalid)?;
```

This mirrors the check already performed in `reverify_participants` and closes the gap between the two code paths. [10](#0-9) 

---

### Proof of Concept

1. Node A calls `submit_participant_info` with a valid Dstack attestation whose `expiry_timestamp_seconds = T`. The contract stores it. [11](#0-10) 
2. Block time advances past `T`. The attestation is now expired; `re_verify()` would return `Err`.
3. No participant calls `verify_tee()`, so `accept_requests` remains `true`.
4. Node A (now potentially running modified code) participates in a threshold signing round and calls `respond(request, response)`.
5. The contract executes `assert_caller_is_attested_participant_and_protocol_active()` → `is_caller_an_attested_participant()`. The function finds the stored attestation, confirms key binding, and returns `Ok(())` — **without checking expiry**. [12](#0-11) 
6. `accept_requests` is `true`, so the `respond()` call proceeds and the signature is accepted on-chain. [13](#0-12) 
7. The TEE security invariant is violated: a node with a lapsed (and unverified) TEE status has successfully submitted a protocol response.

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }

    pub fn verify_tee(&mut self) -> Result<bool, Error> {
        log!("verify_tee: signer={}", env::signer_account_id());
        // Caller must be a participant (node or operator).
        self.voter_or_panic();
        let ProtocolContractState::Running(running_state) = &mut self.protocol_state else {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        };
        let current_params = running_state.parameters.clone();

        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);

        match self.tee_state.reverify_and_cleanup_participants(
            current_params.participants(),
            tee_upgrade_deadline_duration,
        ) {
            TeeValidationResult::Full => {
                self.accept_requests = true;
                log!("All participants have an accepted Tee status");
                Ok(true)
            }
            TeeValidationResult::Partial {
                participants_with_valid_attestation,
            } => {
                let remaining = participants_with_valid_attestation.len();
                // Defense in depth: the surviving participant set must keep the full
                // threshold relation intact — the GovernanceThreshold must still sit
                // within its bounds for the smaller set (in particular it must not
                // exceed the remaining participant count or the upper cap) and must
                // remain at least every domain's ReconstructionThreshold (the kickout
                // keeps the existing per-domain thresholds). Otherwise we refuse and
                // wait for manual intervention.
                let max_reconstruction_threshold =
                    max_reconstruction_threshold(running_state.domains.domains());
                if let Err(err) = ThresholdParameters::validate_governance_against_reconstruction(
                    u64::try_from(remaining).expect("participant count fits in u64"),
                    current_params.threshold(),
                    max_reconstruction_threshold,
                ) {
                    log!(
                        "Kicking out participants with an invalid TEE status would break the threshold relation ({:?}); {} participants remain with a valid TEE status. This requires manual intervention. We will not accept new signature requests as a safety precaution.",
                        err,
                        remaining,
                    );
                    self.accept_requests = false;
                    return Ok(false);
                }

                // here, we set it to true, because at this point, we have at least `threshold`
                // number of participants with an accepted Tee status.
                self.accept_requests = true;
