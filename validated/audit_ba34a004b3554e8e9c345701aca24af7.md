### Title
Caller Deposit Permanently Lost on `storage_balance_of` Failure in `nep_141_storage_balance_callback` - (File: `runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs`)

---

### Summary

The `WalletContract` (the globally-deployed ETH-implicit account contract) fails to refund an external caller's attached NEAR deposit when the intermediate `storage_balance_of` cross-contract call fails inside `nep_141_storage_balance_callback`. This is a direct analog to the reported "unhandled error leads to loss of funds" class: a callback that returns early on failure without performing the necessary compensating action (deposit refund), leaving the caller's NEAR permanently locked in the wallet contract.

---

### Finding Description

When an external caller (e.g., a relayer) invokes `rlp_execute` with an attached NEAR deposit to perform an emulated ERC-20 transfer (`EthEmulationKind::ERC20Transfer`), the following promise chain is constructed in `inner_rlp_execute`:

```
token_contract::storage_balance_of(receiver_id)
  → WalletContract::nep_141_storage_balance_callback(...)
    → WalletContract::rlp_execute_callback(caller_deposit)
``` [1](#0-0) 

The external caller's attached deposit is captured as a `CallerDeposit` and threaded through the chain: [2](#0-1) 

`rlp_execute_callback` correctly handles the failure case by issuing a refund promise to the caller: [3](#0-2) 

However, `nep_141_storage_balance_callback` has **two early-return paths** that return an error response without refunding `caller_deposit`:

**Path 1** — `storage_balance_of` call itself fails (`PromiseResult::Failed`): [4](#0-3) 

**Path 2** — `storage_balance_of` returns unparseable data: [5](#0-4) 

In both cases, the function returns `PromiseOrValue::Value(ExecuteResponse { success: false, ... })` without scheduling any refund transfer to `caller_deposit.account_id`. The caller's NEAR is absorbed into the wallet contract's balance with no recovery path.

The same defect exists in `address_check_callback` for the `EOABaseTokenTransfer` path when the address registrar call fails: [6](#0-5) 

---

### Impact Explanation

The corrupted protocol value is the **caller's NEAR account balance**. When an external caller attaches NEAR to `rlp_execute` for an ERC-20 transfer and the intermediate `storage_balance_of` call fails, the attached deposit is permanently locked inside the wallet contract. The caller's balance is decremented on-chain with no refund receipt generated. This is a direct loss-of-funds outcome for any external caller (relayer or user) who attaches NEAR to such a call.

---

### Likelihood Explanation

The `WalletContract` is the globally-deployed contract for all ETH-implicit accounts on NEAR mainnet. The ERC-20 transfer emulation path is a primary use case. The `storage_balance_of` call can fail due to:
- A malicious or buggy NEP-141 token contract that panics on `storage_balance_of`
- The token contract being deleted between transaction submission and execution
- Gas exhaustion on the `storage_balance_of` call (only 5 TGas is allocated: `NEP_141_STORAGE_BALANCE_OF_GAS`)
- Any token contract that does not implement `storage_balance_of` at all

Any unprivileged external caller can trigger this by calling `rlp_execute` with an attached deposit targeting a token contract that causes `storage_balance_of` to fail. No special privileges are required. [7](#0-6) 

---

### Recommendation

In both early-return paths of `nep_141_storage_balance_callback` (and the analogous paths in `address_check_callback`), issue a refund promise to the caller before returning, mirroring the pattern already used in `rlp_execute_callback`:

```rust
PromiseResult::Failed => {
    // Refund the caller's deposit before returning
    if let Some(CallerDeposit { account_id, yocto_near }) = caller_deposit {
        let refund_promise = env::promise_batch_create(&account_id);
        env::promise_batch_action_transfer(
            refund_promise,
            NearToken::from_yoctonear(yocto_near.into()),
        );
    }
    return PromiseOrValue::Value(ExecuteResponse {
        success: false,
        success_value: None,
        error: Some(format!("Call to NEP-141 {token_id}::storage_balance_of failed")),
    });
}
```

Apply the same fix to the deserialization-failure early return and to the equivalent early returns in `address_check_callback`.

---

### Proof of Concept

1. Deploy a NEP-141 token contract whose `storage_balance_of` method panics unconditionally.
2. An external caller (relayer) calls `rlp_execute` on a `WalletContract` with an attached deposit of, say, 1 NEAR, submitting an ERC-20 transfer signed by the wallet owner targeting the malicious token contract.
3. `inner_rlp_execute` detects `ERC20Transfer`, captures `CallerDeposit { account_id: relayer, yocto_near: 1e24 }`, and schedules `storage_balance_of → nep_141_storage_balance_callback`.
4. The malicious token contract's `storage_balance_of` panics → `PromiseResult::Failed`.
5. `nep_141_storage_balance_callback` hits the early return at line 204–209, returning an error response with no refund promise.
6. The relayer's 1 NEAR is permanently absorbed into the wallet contract's balance.
7. Verify: relayer's balance decreased by ~1 NEAR; wallet contract's balance increased by ~1 NEAR; no refund receipt in the execution outcome. [8](#0-7) [9](#0-8)
