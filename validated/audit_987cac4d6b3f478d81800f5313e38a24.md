### Title
`respond_verify_foreign_tx` Validates Signature Against Node-Supplied `payload_hash` Without Binding It to the Original Request - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the MPC signature is valid for the `payload_hash` supplied in the response, but never checks that `payload_hash` was actually derived from the `request` that is being resolved. A single attested MPC participant can replay a legitimately-produced signature from one pending request onto a completely different pending request, causing the contract to deliver a forged attestation to the second user.

---

### Finding Description

The vulnerability class from the reference report is **Invalid Validation**: a check is performed on a composite/provided value instead of on the individual components that should constrain it.

In `respond_verify_foreign_tx` the contract receives two independent arguments:

```
request  : VerifyForeignTransactionRequest   // the original user request
response : VerifyForeignTransactionResponse  // { payload_hash, signature } — fully node-controlled
```

The only cryptographic check performed is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;   // taken verbatim from the response

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,          // ← node-supplied hash, never re-derived from `request`
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract confirms only that `signature` is a valid ECDSA signature over `response.payload_hash` under the MPC public key. It never computes the **expected** hash from the request:

```
expected = SHA-256(borsh(ForeignTxSignPayload::V1 { request, extracted_values }))
```

and never asserts `response.payload_hash == expected`.

The correct derivation is defined in `ForeignTxSignPayload::compute_msg_hash`:

```rust
pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
    let mut hasher = sha2::Sha256::new();
    borsh::BorshSerialize::serialize(self, &mut hasher)?;
    Ok(Hash256(hasher.finalize().into()))
}
``` [2](#0-1) 

The contract has `request` in scope and could recompute the expected hash, but it does not. The `payload_hash` field in `VerifyForeignTransactionResponse` is therefore an unconstrained free variable from the contract's perspective. [3](#0-2) 

---

### Impact Explanation

**High — Forged foreign-chain verification / invalid bridge execution.**

A single attested MPC participant (below threshold) can:

1. Observe that a valid `{payload_hash_A, sig_A}` was produced for pending request A (e.g., by being the leader of that signing round, or by observing the on-chain `respond_verify_foreign_tx` call for A before it is consumed).
2. Identify a different pending request B in `pending_verify_foreign_tx_requests`.
3. Call `respond_verify_foreign_tx(request_B, { payload_hash_A, sig_A })`.

The contract will:
- Confirm the caller is an attested participant ✓
- Verify `sig_A` is valid for `payload_hash_A` under the MPC key ✓ (it is — it was produced for A)
- Resolve the yield for `request_B` and return `{ payload_hash_A, sig_A }` to user B ✓ [4](#0-3) 

User B's contract receives a valid MPC signature, but over the data of a completely different foreign transaction. Any bridge or application contract that trusts the contract's validation (rather than independently re-verifying via the SDK's `ForeignChainSignatureVerifier::verify_signature`) will accept this as a legitimate attestation of their transaction, enabling invalid bridge execution or double-spend conditions.

The client-side SDK does perform the missing check:

```rust
let payload_is_correct = expected_payload_hash == response.payload_hash;
if !payload_is_correct {
    return Err(VerifyForeignChainError::IncorrectPayloadSigned { ... });
}
``` [5](#0-4) 

But the on-chain contract — the authoritative trust anchor — does not enforce this invariant, leaving every caller that does not use the SDK unprotected.

---

### Likelihood Explanation

**Medium-High.** The attacker is a single attested MPC participant, which is a realistic adversary model (Byzantine participant strictly below the signing threshold). No threshold collusion, key leakage, or TEE attack is required. The attacker only needs to observe or participate in one legitimate signing round to obtain a reusable `{payload_hash, sig}` pair, then submit it against any other pending request of the same domain. The window of opportunity exists whenever two or more `verify_foreign_transaction` requests are pending simultaneously, which is the normal operating condition for a bridge.

---

### Recommendation

In `respond_verify_foreign_tx`, independently recompute the expected `payload_hash` from the `request` and the `response`'s extracted values, and assert equality before accepting the response. Concretely, the contract should reconstruct `ForeignTxSignPayload::V1 { request: request.request.clone(), values: ... }` and call `compute_msg_hash()`, then compare the result to `response.payload_hash`. Since the contract does not have access to the raw `extracted_values` (they are not stored on-chain), the minimal fix is to verify the signature against the independently-derived hash rather than the node-supplied one — mirroring exactly what the SDK's `verify_signature` already does:

```rust
// Derive expected hash from the request + response values, not from response.payload_hash
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(), // extracted_values must be added to the response
});
let expected_hash = expected_payload.compute_msg_hash()?;
assert!(expected_hash == response.payload_hash, "payload_hash mismatch");
```

Alternatively, store the expected `payload_hash` on-chain at request submission time (once the node reports the extracted values) and compare against it in `respond_verify_foreign_tx`.

### Proof of Concept

**Setup**: Two users, Alice and Bob, each submit a `verify_foreign_transaction` request for different Bitcoin transactions. A single malicious attested MPC node M participates.

1. Alice submits `verify_foreign_transaction(request_A)` → contract stores pending yield `Y_A`.
2. Bob submits `verify_foreign_transaction(request_B)` → contract stores pending yield `Y_B`.
3. The MPC network legitimately processes request A. Node M, as leader, obtains `{payload_hash_A, sig_A}` — a valid threshold signature over `SHA-256(borsh(ForeignTxSignPayload{request_A, values_A}))`.
4. Node M calls `respond_verify_foreign_tx(request_A, {payload_hash_A, sig_A})` → Alice's yield resolves correctly.
5. Node M immediately calls `respond_verify_foreign_tx(request_B, {payload_hash_A, sig_A})`.
6. The contract checks: `verify_ecdsa_signature(sig_A, payload_hash_A, mpc_pk)` → **passes** (the signature is valid).
7. Bob's yield `Y_B` resolves with `{payload_hash_A, sig_A}`.
8. Bob's bridge contract receives a valid MPC signature, but over Alice's transaction data. If Bob's contract does not call `ForeignChainSignatureVerifier::verify_signature`, it accepts this as proof that Bob's transaction was verified. [6](#0-5) [7](#0-6)
