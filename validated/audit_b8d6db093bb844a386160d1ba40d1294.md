Audit Report

## Title
Permissionless `FeeReceiver.sendFunds()` + `LRTOracle.updateRSETHPrice()` Enables Yield Front-Running Against rsETH Holders - (`contracts/FeeReceiver.sol`, `contracts/LRTOracle.sol`)

## Summary
`FeeReceiver.sendFunds()` carries no access control and can be called by any external account at any time, flushing accumulated MEV/execution-layer rewards into the deposit pool and increasing the protocol's measured ETH TVL. Because `LRTOracle.updateRSETHPrice()` is equally permissionless, an attacker can deposit ETH, trigger reward distribution, update the price, and immediately withdraw at the inflated rsETH price — capturing a portion of pending yield that should have accrued exclusively to pre-existing holders.

## Finding Description

**Root cause 1 — `FeeReceiver.sendFunds()` has no access control.**

`contracts/FeeReceiver.sol` L53-58:
```solidity
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```
Any EOA or contract can call this at any time.

**Root cause 2 — `LRTOracle.updateRSETHPrice()` has no access control.**

`contracts/LRTOracle.sol` L87-89:
```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

**TVL accounting confirms the price impact.**

`getETHDistributionData()` in `contracts/LRTDepositPool.sol` L480 uses `address(this).balance` for the deposit pool's ETH contribution. When `sendFunds()` sends R ETH to the pool via `receiveFromRewardReceiver()`, `address(depositPool).balance` increases by R, which flows directly into `_getTotalEthInProtocol()` in `LRTOracle.sol` L331-348 via `getTotalAssetDeposits(ETH)`.

**Deposit minting uses the stale (pre-reward) price.**

`getRsETHAmountToMint()` in `LRTDepositPool.sol` L520 reads `lrtOracle.rsETHPrice()` — the stored value — not a freshly computed price. So an attacker who deposits before calling `sendFunds()` receives rsETH at the old, lower price.

**Exploit path:**
1. Attacker calls `LRTDepositPool.depositETH{value: A}()` → receives `A / rsETHPrice_old` rsETH at the pre-reward price.
2. Attacker calls `FeeReceiver.sendFunds()` → R ETH moves to deposit pool; TVL becomes `T + A + R`.
3. Attacker calls `LRTOracle.updateRSETHPrice()` → new price ≈ `(T+A+R−fee) / (S + A/rsETHPrice_old)`.
4a. **Instant path** (when `isInstantWithdrawalEnabled[ETH] == true`): attacker calls `LRTWithdrawalManager.instantWithdrawal(ETH, rsETHAmount, "")`. `getExpectedAssetAmount()` at L593 reads the freshly updated `lrtOracle.rsETHPrice()`, so the attacker receives ETH at the inflated price in the same block.
4b. **Standard path**: attacker calls `initiateWithdrawal(ETH, rsETHAmount, "")`. `expectedAssetAmount` is locked at initiation time using the inflated price (L168), and `completeWithdrawal` pays out that locked amount after the 8-day delay.

**The `pricePercentageLimit` guard is insufficient.**

`LRTOracle.sol` L252-266 checks whether the price increase exceeds `pricePercentageLimit * highestRsethPrice` and reverts for non-managers if it does. This guard fails to prevent the attack in two scenarios:
- `pricePercentageLimit == 0` (the check is gated on `pricePercentageLimit > 0`), in which case there is no guard at all.
- The reward R is small relative to TVL (e.g., 50 ETH on a 10,000 ETH TVL = 0.5% increase). If the limit is set to 1%, the price increase is within bounds and the call succeeds for any caller.

## Impact Explanation
**High — Theft of unclaimed yield.**

Existing rsETH holders accumulate yield as MEV/execution-layer rewards accrue in `FeeReceiver`. An attacker who deposits immediately before flushing those rewards and updating the price captures a proportional share of `R` that should belong only to pre-existing holders. The attacker's profit is `A·R/(T+A)` and the existing holders' collective loss is the same amount. When instant withdrawal is enabled, this is executable atomically in a single block with no capital lock-up risk beyond the instant-withdrawal fee.

## Likelihood Explanation
Both functions are callable by any EOA with no preconditions beyond the contract not being paused. MEV bots routinely monitor on-chain state for exactly this pattern. The attack is profitable whenever `A·R/(T+A)` exceeds the instant-withdrawal fee (instant path) or the cost of capital for 8 days (standard path). No privileged access, leaked keys, or oracle compromise is required. The attack is repeatable every time rewards accumulate in `FeeReceiver`.

## Recommendation
1. **Restrict `FeeReceiver.sendFunds()`** to a trusted role (e.g., `MANAGER`):
   ```solidity
   function sendFunds() external onlyRole(LRTConstants.MANAGER) { ... }
   ```
2. **Restrict `LRTOracle.updateRSETHPrice()`** to a keeper/manager role, or enforce a minimum interval between successive public price updates (e.g., 1 hour), so that a deposit immediately followed by a price update cannot be executed atomically or within the same block.
3. Consider a **snapshot-based yield accrual** model where only holders present at the start of a reward epoch receive that epoch's yield, preventing late depositors from capturing pending rewards.

## Proof of Concept
```
// Setup: FeeReceiver holds 50 ETH. Protocol TVL = 10,000 ETH, rsETH supply = 9,500 (price ≈ 1.0526 ETH/rsETH).

1. attacker.depositETH{value: 5000 ETH}(0, "")
   → rsETH minted = 5000 / 1.0526 ≈ 4750 rsETH

2. attacker.sendFunds()   // FeeReceiver.sendFunds()
   → 50 ETH → DepositPool; address(depositPool).balance += 50
   → getTotalAssetDeposits(ETH) increases by 50

3. attacker.updateRSETHPrice()   // LRTOracle.updateRSETHPrice()
   → totalETHInProtocol = 15,050 ETH
   → newRsETHPrice = 15,050 / 14,250 ≈ 1.05614 ETH/rsETH
   → price increase ≈ 0.08% — well within any reasonable pricePercentageLimit

4a. [Instant path, if enabled]
    attacker.instantWithdrawal(ETH, 4750, "")
    → assetAmountUnlocked = 4750 × 1.05614 ≈ 5016.7 ETH
    → after instant fee (e.g. 10 bps = ~5 ETH): attacker receives ≈ 5011.7 ETH
    → net profit ≈ 11.7 ETH stolen from existing holders in one block

4b. [Standard path]
    attacker.initiateWithdrawal(ETH, 4750, "")
    → expectedAssetAmount locked at ≈ 5016.7 ETH
    → after 8-day delay: completeWithdrawal() pays out 5016.7 ETH
    → net profit ≈ 16.7 ETH; existing holders lose their proportional share of the 50 ETH reward
```
