Audit Report

## Title
Hardcoded 18-Decimal Assumption in Token Swap Calculation Causes Severe rsETH Under-Minting for Non-18 Decimal Tokens - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
`RSETHPoolV3.viewSwapRsETHAmountAndFee(uint256 amount, address token)` and `RSETHPool.viewSwapRsETHAmountAndFee(uint256 amount, address token)` compute rsETH output as `amountAfterFee * tokenToETHRate / rsETHToETHrate` without normalizing `amountAfterFee` to 1e18 precision. For any ERC20 token with fewer than 18 decimals, the result is `10^(18 - tokenDecimals)` times too small, causing depositors to receive negligible rsETH while the pool retains their full token deposit.

## Finding Description
In `RSETHPoolV3.viewSwapRsETHAmountAndFee`:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [1](#0-0) 

`tokenToETHRate` is returned by `ChainlinkOracleForRSETHPoolCollateral.getRate()`, which normalizes the Chainlink price to 1e18 precision ŌĆö it represents the ETH value of **1 whole token** (e.g., 1 USDC), not 1 token-unit:

```solidity
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
``` [2](#0-1) 

`rsETHToETHrate` is similarly 1e18-normalized. The formula is therefore dimensionally correct only when `amountAfterFee` is also in 1e18 units (i.e., the token has 18 decimals). The ETH-deposit overload confirms the intended pattern ŌĆö it explicitly multiplies by `1e18`:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [3](#0-2) 

For a 6-decimal token, `amountAfterFee` for 1 token is `1e6`, not `1e18`, so the formula produces a result `1e12` times too small. The identical structural bug exists in `RSETHPool`:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [4](#0-3) 

`addSupportedToken` imposes no decimal restriction ŌĆö any ERC20 address is accepted: [5](#0-4) 

No decimal-normalization guard exists anywhere in the deposit or swap path.

## Impact Explanation
**Critical ŌĆö Direct theft of user funds.**

A user depositing 1 USDC (`amountAfterFee = 1_000_000`, 6 decimals) with `tokenToETHRate = 3.3e14` (Ōēł 0.00033 ETH/USDC) and `rsETHToETHrate = 1.05e18`:

- **Actual formula result:** `1_000_000 ├Ś 3.3e14 / 1.05e18 Ōēł 314 wei rsETH` (Ōēł 3.14├Ś10Ōü╗┬╣ŌüČ rsETH)
- **Correct result:** `3.14e14 wei rsETH` (Ōēł 3.14├Ś10Ōü╗Ōü┤ rsETH)

The user loses a factor of `1e12` in rsETH. The pool retains the full USDC deposit while the user receives essentially zero rsETH ŌĆö constituting direct, irreversible theft of deposited funds at the protocol level. The `deposit` function transfers tokens in before computing the mint amount, so there is no revert path that returns funds to the user. [6](#0-5) 

## Likelihood Explanation
`addSupportedToken` is callable by `TIMELOCK_ROLE` and is explicitly designed to extend the pool to arbitrary ERC20 collateral. A governance decision to onboard any non-18 decimal token (USDC, USDT, WBTC with 8 decimals, etc.) would silently activate this vulnerability for every subsequent depositor of that token. The miscalculation produces no revert, no event anomaly, and no on-chain signal ŌĆö it is invisible until funds are lost. Once a non-18 decimal token is listed, any unprivileged user calling `deposit(token, amount, referralId)` triggers the loss with no further preconditions.

## Recommendation
Normalize `amountAfterFee` to 1e18 precision using `IERC20Metadata(token).decimals()` before applying the swap formula in both contracts:

```solidity
import { IERC20Metadata } from "@openzeppelin/contracts/token/ERC20/extensions/IERC20Metadata.sol";

uint8 tokenDecimals = IERC20Metadata(token).decimals();
rsETHAmount = amountAfterFee * tokenToETHRate * 1e18
              / (rsETHToETHrate * 10**uint256(tokenDecimals));
```

Apply the same fix to `RSETHPool.viewSwapRsETHAmountAndFee`. Also audit `viewSwapAssetToPremintedRsETH` in `RSETHPoolV3`, which contains the symmetric inverse formula and would over-pay tokens for the same class of non-18 decimal assets. [7](#0-6) 

## Proof of Concept
1. `TIMELOCK_ROLE` calls `RSETHPoolV3.addSupportedToken(USDC_ADDRESS, USDC_CHAINLINK_ORACLE)`.
2. User calls `RSETHPoolV3.deposit(USDC_ADDRESS, 1_000_000, "ref")` (depositing 1 USDC).
3. `viewSwapRsETHAmountAndFee(1_000_000, USDC_ADDRESS)` executes:
   - `fee = 0` (assuming `feeBps = 0` for clarity)
   - `amountAfterFee = 1_000_000`
   - `tokenToETHRate = 3.3e14` (from `ChainlinkOracleForRSETHPoolCollateral`)
   - `rsETHToETHrate = 1.05e18`
   - `rsETHAmount = 1_000_000 * 3.3e14 / 1.05e18 = 314 wei`
4. `wrsETH.mint(msg.sender, 314)` ŌĆö user receives 314 wei rsETH.
5. Expected: `3.14e14 wei` rsETH. Actual: `314 wei`. Loss factor: `1e12├Ś`.
6. Pool retains 1 USDC; user holds negligible rsETH.
