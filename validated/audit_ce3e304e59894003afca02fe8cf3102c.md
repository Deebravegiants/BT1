Audit Report

## Title
Pool Admin Can Instantly Set Uncapped Bin Additional Fees to Frontrun Swaps — (`metric-core/contracts/MetricOmmPoolFactory.sol` / `metric-core/contracts/MetricOmmPool.sol`)

## Summary
`MetricOmmPoolFactory.setPoolBinAdditionalFees()` allows a pool admin to set per-bin buy/sell fees (`addFeeBuyE6` / `addFeeSellE6`) to any value up to `type(uint16).max` (65535 ≈ 6.55% in E6 units) with no upper-bound cap and no timelock. Unlike pool-level admin fees, which are guarded by `maxAdminSpreadFeeE6` / `maxAdminNotionalFeeE8`, bin additional fees have no equivalent guard. A malicious pool admin can frontrun a pending swap by spiking the bin fee, causing the trader to receive far fewer tokens than expected, with the excess retained as LP balance.

## Finding Description
`MetricOmmPoolFactory.setPoolBinAdditionalFees()` passes caller-supplied values directly to the pool with no cap validation: [1](#0-0) 

The pool's `setBinAdditionalFees()` performs only a bin-index range check before writing the fee values: [2](#0-1) 

These fees are added directly to `params.baseFeeX64` at swap execution time for both specified-output and specified-input swap paths: [3](#0-2) [4](#0-3) 

By contrast, `setPoolAdminFees()` enforces explicit caps before writing: [5](#0-4) 

No equivalent guard exists for bin additional fees. A grep for `maxAdminBinFee`, `maxBinFee`, or any analogous cap constant returns no matches anywhere in the production contracts. Similarly, while price-provider updates use a propose/execute timelock pattern, no such delay is applied to `setPoolBinAdditionalFees()`.

## Impact Explanation
A trader submitting a swap against the current bin receives fewer output tokens than the oracle price and their `priceLimitX64` slippage tolerance would suggest, because the effective execution fee is silently elevated by up to `65535/1e6 ≈ 6.55%`. The excess input value is retained inside the bin as LP balance, benefiting LPs (including the pool admin if they are also an LP) at the trader's expense. This is a direct loss of user principal on every swap that crosses the manipulated bin. This satisfies the "Admin-boundary break: pool admin exceeds caps" allowed impact criterion, as the design intent (evidenced by caps on all other admin fee parameters) is that admin-controlled fees must be bounded.

## Likelihood Explanation
The pool admin is a semi-trusted role that can act unilaterally and atomically. No timelock, no governance delay, and no cap guard stands between the admin and the fee change. A malicious admin can observe a large pending swap in the mempool and frontrun it in the same block. Even a non-malicious admin who raises fees for legitimate reasons can accidentally frontrun a user whose transaction was already in flight. The attack is repeatable on every swap and requires no special preconditions beyond the admin role.

## Recommendation
1. Add a hard cap on `addFeeBuyE6` and `addFeeSellE6` in `setPoolBinAdditionalFees()`, analogous to `maxAdminSpreadFeeE6` / `maxAdminNotionalFeeE8` already enforced for pool-level admin fees:
   ```solidity
   if (addFeeBuyE6 > maxAdminBinFeeE6) revert AdminFeeTooHigh();
   if (addFeeSellE6 > maxAdminBinFeeE6) revert AdminFeeTooHigh();
   ```
2. Add a timelock or two-step propose/execute pattern for bin additional fee changes, consistent with the timelock already applied to price-provider updates (`proposePoolPriceProvider` / `executePoolPriceProviderUpdate`).

## Proof of Concept
1. Pool is live; `curBinIdx = 0`; `addFeeBuyE6 = 0` (no additional fee).
2. User submits `swap(recipient, false, 1_000e18, priceLimitX64, ...)` — buying token0 with token1.
3. Pool admin observes the pending transaction and frontruns with:
   ```solidity
   factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535);
   ```
4. User's swap executes with effective fee = `baseFee + 0.065535`, paying ~6.55% more token1 than expected for the same token0 output (exact-output) or receiving ~6.55% fewer token0 (exact-input).
5. The extra token1 retained in the bin accrues to LPs; the user has no recourse.
