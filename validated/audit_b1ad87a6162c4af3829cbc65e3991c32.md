Audit Report

## Title
ETH Deposit Limit Check Omits Deposit Amount, Allowing Any Depositor to Exceed the Configured Cap - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses an asymmetric check: the ETH branch tests only whether the current total already exceeds the limit, without adding the incoming deposit amount. This means the check never blocks the transaction that first crosses the boundary — only subsequent ones. Any unprivileged depositor can push ETH TVL above the protocol-configured cap in a single transaction.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682, the function `_checkIfDepositAmountExceedesCurrentLimit` has two code paths:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount NOT added
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // amount added
}
``` [1](#0-0) 

For ERC-20 assets, `amount` is added to `totalAssetDeposits` before comparing against the limit — the correct guard. For ETH, `amount` is silently ignored. When `totalAssetDeposits == limit`, the ETH branch evaluates `limit > limit` → `false`, so no revert occurs and the deposit is accepted.

This check is called unconditionally from `_beforeDeposit` at line 661, which is invoked by the public `depositETH` entry point at line 87. [2](#0-1) [3](#0-2) 

## Impact Explanation
**Low — Contract fails to deliver promised returns (deposit cap not enforced for ETH).**

The ETH deposit cap (`depositLimitByAsset[ETH_TOKEN]`) is a protocol risk-management parameter. When the TVL is exactly at the cap, one deposit of any size is accepted without revert, minting rsETH against excess ETH. After that transaction, `totalAssetDeposits > limit` becomes true and all further ETH deposits are blocked — but the excess ETH has already been accepted and rsETH minted against it. The protocol holds more ETH exposure than its own configuration permits. No direct loss of user funds occurs.

## Likelihood Explanation
The condition is reachable by any unprivileged ETH depositor. No special role, front-running, oracle manipulation, or governance action is required. The only precondition is that ETH deposits are enabled and the current TVL is at or near the configured limit — a normal operational state. Likelihood is **High**.

## Recommendation
Add the incoming `amount` to the ETH branch, mirroring the ERC-20 logic:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [4](#0-3) 

## Proof of Concept

1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 1000 ether`.
2. Protocol accumulates exactly `1000 ether` of ETH TVL (across NodeDelegators and DepositPool).
3. Any user calls `depositETH{value: 500 ether}(0, "")`.
4. Inside `_checkIfDepositAmountExceedesCurrentLimit`:
   - `totalAssetDeposits = 1000 ether`
   - ETH branch: `return (1000 ether > 1000 ether)` → `false` → no revert
5. `_mintRsETH` mints rsETH for 500 ETH; protocol now holds 1500 ETH against a 1000 ETH cap.
6. All subsequent `depositETH` calls revert (`1500 > 1000`), but the 500 ETH excess is permanently accepted.

**Foundry test plan:** Deploy `LRTDepositPool` with a mock config returning `depositLimitByAsset = 1000 ether`. Seed `getTotalAssetDeposits(ETH_TOKEN)` to return exactly `1000 ether`. Call `depositETH{value: 1 ether}` and assert it does not revert. Assert `getTotalAssetDeposits` now exceeds the configured limit. [1](#0-0)
