### Title
Native Script `TimelockMOf` with Negative Threshold Unconditionally Bypasses All Sub-Script Witness Validation - (File: `eras/allegra/impl/src/Cardano/Ledger/Allegra/Scripts.hs`)

---

### Summary

The `evalTimelock` function in `Cardano.Ledger.Allegra.Scripts` accepts a negative integer threshold in the `TimelockMOf` constructor. When the threshold `n` is negative, the `isValidMOf` helper immediately returns `True` without evaluating any sub-scripts, unconditionally satisfying the script regardless of signatures or time-lock conditions. The CDDL specification explicitly allows `int64` (which includes all negative values) for the `script_n_of_k` threshold, and the ledger accepts and processes such scripts without any validation error. This is the direct analog of the reported pattern: a condition-check function that should enforce authorization constraints but instead always returns `True`.

---

### Finding Description

**Root cause ŌĆö `TimelockRaw` data type and `isValidMOf` evaluator:**

The `TimelockRaw` data type stores the M-of-N threshold as a plain `Int`:

```haskell
| TimelockMOf !Int !(StrictSeq (Timelock era))
| -- Note that the Int may be negative in which case (TimelockMOf (-2) [..]) is always True
``` [1](#0-0) 

The `evalTimelock` function evaluates `TimelockMOf` via `isValidMOf`:

```haskell
isValidMOf n SSeq.Empty = n <= 0
isValidMOf n (ts SSeq.:<| tss) =
  n <= 0 || if go ts then isValidMOf (n - 1) tss else isValidMOf n tss
``` [2](#0-1) 

When `n` is any negative integer (e.g., `-1`, `-9223372036854775808`), the guard `n <= 0` is immediately `True` on every recursive call, so the function returns `True` without evaluating a single sub-script. The entire sub-script list ŌĆö which may contain `RequireSignature`, `RequireTimeStart`, `RequireTimeExpire`, or nested `RequireMOf` conditions ŌĆö is silently ignored.

**The CDDL specification explicitly permits negative thresholds:**

```
script_n_of_k = (3, n : int64, [* native_script])
int64 = min_int64 .. max_int64
min_int64 = -9223372036854775808
``` [3](#0-2) 

This is consistent across all post-Allegra eras (Mary, Alonzo, Babbage, Conway, Dijkstra). [4](#0-3) 

**The CBOR decoder accepts negative thresholds without rejection:**

```haskell
decRaw 3 = Ann (SumD TimelockMOf) <*! Ann From <*! D (sequence <$> decCBOR)
``` [5](#0-4) 

There is no range check on the decoded `Int` value. A serialized transaction carrying `[3, -1, [script1, script2]]` is accepted and stored as `TimelockMOf (-1) [script1, script2]`.

**The same bypass exists in Shelley-era `evalMultiSig`:**

```haskell
isValidMOf n StrictSeq.Empty = n <= 0
isValidMOf n (msig StrictSeq.:<| msigs) =
  n <= 0 || if go msig then isValidMOf (n - 1) msigs else isValidMOf n msigs
``` [6](#0-5) 

**Deviation from the formal specification:**

The formal spec defines `MOfN` with `m Ōłł ŌäĢ` (natural numbers), which excludes negative values:

```
evalTimelock ~(MOfN m ts) ~vhks ~_ =
  m Ōēż ╬Ż (card { t s.t. t ŌåÉ ts Ōł¦ evalTimelock t vhks })
``` [7](#0-6) 

The implementation accepts `Int` (which includes all negative values), deviating from the spec's `ŌäĢ` domain.

**Attack path:**

An attacker who is the script author crafts a native script such as:

```
TimelockMOf (-1) [RequireSignature k1, RequireSignature k2, RequireTimeExpire slot100]
```

This script is serialized as `[3, -1, [[0, k1], [0, k2], [5, 100]]]` in CBOR. The script hash `H(S)` is computed from this serialization. The attacker presents this script to a victim as a "2-of-3 multisig with expiry" (misrepresenting the threshold). The victim sends ADA or native assets to the enterprise or base address derived from `H(S)`. The attacker then submits a transaction spending those UTxOs with an empty witness set. The ledger calls `validateTimelock`, which calls `evalTimelock`, which calls `isValidMOf (-1) [...]`, which immediately returns `True`. The transaction is accepted.

The `validateTimelock` entry point confirms this path:

```haskell
validateTimelock tx timelock = evalTimelock vhks (tx ^. bodyTxL . vldtTxBodyL) timelock
  where
    vhks = Set.map witVKeyHash (tx ^. witsTxL . addrTxWitsL)
``` [8](#0-7) 

With an empty `addrTxWitsL`, `vhks` is the empty set. `isValidMOf (-1) [...]` returns `True` before consulting `vhks` at all.

---

### Impact Explanation

An attacker who is the script author can craft a native script with a negative `script_n_of_k` threshold. This script unconditionally evaluates to `True` regardless of the sub-scripts it contains, bypassing all signature and time-lock requirements. Any ADA or native assets locked to such a script address can be spent by any party without providing any witnesses. This constitutes a direct loss of ADA or native assets through a script witness validation bypass. The impact maps to:

> **Medium. Attacker-controlled scripts exceed intended validation limits outside design parameters.**

The formal spec constrains `m Ōłł ŌäĢ`, but the implementation accepts `m Ōłł Ōäż`. The negative-threshold script is an attacker-controlled serialized input that causes the ledger to skip all sub-script evaluation, producing an invalid ledger state transition from the perspective of the script's intended authorization semantics.

---

### Likelihood Explanation

The attack requires the attacker to be the script author and to convince a counterparty to send funds to the malicious script address (e.g., by presenting it as a legitimate multisig). The behavior is explicitly documented in the source code comment (`-- Note that the Int may be negative in which case (TimelockMOf (-2) [..]) is always True`), indicating it is a known but unmitigated property of the evaluator. The CDDL explicitly permits `int64`, so no serialization-layer rejection occurs. Any wallet or dApp that accepts a script address without independently verifying the threshold sign is exposed. Likelihood is **Low-Medium**: the attack requires a social engineering step, but the ledger provides no on-chain protection.

---

### Recommendation

1. **Type-level fix**: Change the threshold field in `TimelockMOf` from `!Int` to `!Natural` (or `!Word64`). This makes negative thresholds unrepresentable at the type level and eliminates the `n <= 0` short-circuit for negative values.
2. **Decoder-level fix**: Add a range check in the `DecCBOR` instance for `TimelockRaw` to reject any decoded threshold less than zero, returning a `DecoderError` for `decRaw 3` when the decoded `Int` is negative.
3. **CDDL fix**: Change `script_n_of_k = (3, n : int64, ...)` to `script_n_of_k = (3, n : uint, ...)` to align the wire format with the formal spec's `m Ōłł ŌäĢ` constraint.

### Proof of Concept

```
-- Craft the malicious script (CBOR: [3, -1, [[0, <keyhash_k1>], [0, <keyhash_k2>]]])
let maliciousScript = TimelockMOf (-1) [RequireSignature k1, RequireSignature k2]

-- Verify it always evaluates to True with empty witnesses
evalTimelock mempty (ValidityInterval SNothing SNothing) maliciousScript
-- => True  (no signatures checked, no time checked)

-- Lock funds to the script address
-- addr = enterpriseAddr (ScriptHashObj (hashScript maliciousScript))

-- Spend without any witnesses: tx with empty addrTxWitsL
-- validateTimelock emptyTx maliciousScript => True
-- Transaction accepted by UTXOW rule
```

The `isValidMOf (-1) [RequireSignature k1, RequireSignature k2]` call hits `n <= 0` on the first invocation and returns `True` immediately, never consulting `k1` or `k2`. [9](#0-8)
