### Title
Premature `valid_proposals` insertion before `ProposalFinMismatch` check allows stale invalid proposals to persist in validator cache — (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

In `validate_proposal`, the proposal is unconditionally inserted into the shared `valid_proposals` cache **before** the `ProposalFinMismatch` commitment check. If the check fails, the proposal is never removed from the cache, leaving a stale entry that `repropose` can later retrieve and re-stream as if it were a successfully validated proposal.

---

### Finding Description

`validate_proposal` (lines 238–249) performs two operations in the wrong order:

```rust
// Update valid_proposals before sending fin to avoid a race condition
// with `repropose` being called before `valid_proposals` is updated.
let mut valid_proposals = args.valid_proposals.lock().unwrap();
valid_proposals.insert_proposal(args.init, content, &args.proposal_id, finished_info); // ← inserted here

// TODO(matan): Switch to signature validation.
if built_block != received_fin.proposal_commitment {          // ← checked here
    CONSENSUS_PROPOSAL_FIN_MISMATCH.increment(1);
    return Err(ValidateProposalError::ProposalFinMismatch);   // ← returns Err, but entry already in cache
}

Ok(built_block)
``` [1](#0-0) 

The developer comment acknowledges the race condition with `repropose` and intentionally inserts before sending the commitment to consensus. However, the commitment equality check (`built_block != received_fin.proposal_commitment`) is placed **after** the insert. When this check fails and `ProposalFinMismatch` is returned, the function exits without cleaning up the entry it just wrote into `valid_proposals`.

The `valid_proposals` map is typed as:

```rust
type HeightToRoundToProposal =
    BTreeMap<BlockNumber, BTreeMap<Round, (ProposalCommitment, ProposalContent)>>;
``` [2](#0-1) 

It stores the **batcher-computed** `ProposalCommitment` alongside the full `ProposalContent`. The `repropose` path looks up proposals by commitment. A stale entry stored here with the batcher's correct commitment `C_batcher` is indistinguishable from a legitimately validated proposal.

The `validate_and_send` wrapper (called from `validate_current_round_proposal`) propagates the error to consensus, so `fin_sender` is never called for the mismatched proposal. Consensus correctly treats the round as failed. But the entry in `valid_proposals` persists across rounds. [3](#0-2) 

---

### Impact Explanation

**Impact: High — Wrong authoritative commitment/block content can be re-proposed and finalized.**

The `valid_proposals` cache is the sole source of truth for `repropose`. When consensus advances to a new round and the same node becomes proposer, it calls `repropose(C_batcher, new_init)`. The lookup finds the stale entry (keyed by `C_batcher`) and re-streams the full transaction content with the batcher's commitment. Other validators execute the same transactions, independently compute `C_batcher`, and accept the proposal. `decision_reached` is then called with `C_batcher`, and the block is committed.

The net effect: a proposal that was explicitly rejected by the `ProposalFinMismatch` guard in round N is silently committed in round N+1, bypassing the invariant that only proposals whose proposer-supplied commitment matches the locally-computed commitment are eligible for finalization.

---

### Likelihood Explanation

**Likelihood: Medium.**

The trigger requires a proposer to send a `ProposalFin` whose `proposal_commitment` field does not match the batcher's output. This is a single-field manipulation on the wire, reachable by any node that holds a proposer slot. The validator node must subsequently win a proposer slot in a later round for the stale entry to be re-proposed. Both conditions are achievable in a live network without any privileged access.

---

### Recommendation

Reorder the operations so the commitment check gates the insert:

```rust
// Check commitment BEFORE inserting into valid_proposals.
if built_block != received_fin.proposal_commitment {
    CONSENSUS_PROPOSAL_FIN_MISMATCH.increment(1);
    return Err(ValidateProposalError::ProposalFinMismatch);
}

// Only insert after the proposal is confirmed valid.
// The race-condition comment still applies: insert before sending fin to consensus.
let mut valid_proposals = args.valid_proposals.lock().unwrap();
valid_proposals.insert_proposal(args.init, content, &args.proposal_id, finished_info);

Ok(built_block)
```

Alternatively, if the pre-insert ordering must be preserved for the race-condition reason, remove the entry on mismatch:

```rust
valid_proposals.insert_proposal(...);
if built_block != received_fin.proposal_commitment {
    valid_proposals.remove_proposal(&args.proposal_id); // compensating removal
    return Err(ValidateProposalError::ProposalFinMismatch);
}
```

### Proof of Concept

1. **Setup**: Node V is a validator for height H, round 0. Node A is the proposer.
2. **Attack**: A sends a valid transaction batch followed by a `ProposalFin` where `proposal_commitment = C_wrong ≠ C_batcher`.
3. **Stale insert**: V's batcher executes the transactions and returns `C_batcher`. `validate_proposal` calls `valid_proposals.insert_proposal(init, content, proposal_id, finished_info)` — entry is now in the cache keyed by `(H, round=0, C_batcher)`.
4. **Check fails**: `C_batcher != C_wrong` → `ProposalFinMismatch` is returned. `fin_sender` is never called. Consensus on V marks round 0 as failed.
5. **Round advance**: Consensus moves to round 1. V wins the proposer slot.
6. **Repropose**: Consensus calls `repropose(C_batcher, init_round_1)`. The lookup in `valid_proposals` finds the stale entry from step 3 and re-streams the full transaction batch with commitment `C_batcher`.
7. **Acceptance**: Other validators independently compute `C_batcher` and send prevotes. Consensus reaches a quorum.
8. **Commit**: `decision_reached(H, C_batcher)` is called. The batcher looks up `C_batcher` in `valid_proposals`, finds the stale entry, and commits the block — finalizing a proposal that was explicitly rejected in round 0. [1](#0-0)
