### Title
Unbounded Loop in `NodeDelegator.getAssetUnstaking()` Propagates to `updateRSETHPrice()` and User Deposit Functions - (File: contracts/NodeDelegator.sol)

### Summary
`NodeDelegator.getAssetUnstaking()` iterates over all EigenLayer queued withdrawals without any limit. This function is called on-chain from `LRTDepositPool.getAssetDistributionData()` and `getETHDistributionData()` for every NDC, which are called from `getTotalAssetDeposits()`, which is called from both the public `LRTOracle.updateRSETHPrice()` and from user-facing `depositETH()`/`depositAsset()`. As the number of queued withdrawals grows, these critical paths will run out of gas.

### Finding Description
`NodeDelegator.getAssetUnstaking()` fetches all queued withdrawals from EigenLayer and iterates over them with a nested loop:

```solidity
// NodeDelegator.sol lines 406-427
function getAssetUnstaking(address asset) external view returns (uint256 amount) {
    (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
        _getDelegationManager().getQueuedWithdrawals(address(this));

    for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
        ...
        for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
            ...
        }
    }
}
```

This function is called on-chain through the following chains:

**Chain 1 (price update):**
`updateRSETHPrice()` → `_updateRsETHPrice()` → `_getTotalEthInProtocol()` → `getTotalAssetDeposits(asset)` (per supported asset) → `getAssetDistributionData(asset)` / `getETHDistributionData()` → `INodeDelegator(nodeDelegatorQueue[i]).getAssetUnstaking(asset)` (per NDC)

**Chain 2 (user deposit):**
`depositETH()` / `depositAsset()` → `_beforeDeposit()` → `getTotalAssetDeposits(asset)` → same path as above

The total number of `getAssetUnstaking()` calls per `updateRSETHPrice()` invocation is `supportedAssets.length × nodeDelegatorQueue.length`. Each call makes an external call to EigenLayer's `getQueuedWithdrawals()` and iterates over all returned withdrawals.

The protocol's own comment in `LRTUnstakingVault.setMaxUncompletedWithdrawalCount()` explicitly acknowledges this gas concern:

```solidity
// LRTUnstakingVault.sol line 151-152
// 120 is the max number of uncompleted withdrawals that allows us to still perform update rsETH price
// Need buffer for theoretical operator forced undelegations (ndc count * asset count = 15)
if (_maxUncompletedWithdrawalCount > 80) {
    revert MaxUncompletedWithdrawalCountTooHigh();
}
```

This confirms the protocol is aware that at 120 uncompleted withdrawals, `updateRSETHPrice()` would run out of gas. The cap of 80 is a fragile mitigation because:
1. `setUncompletedWithdrawalCount()` (callable by LRT Manager) resets `uncompletedWithdrawalCount` to the actual EigenLayer count, which could exceed the cap if withdrawals were queued before the limit was enforced.
2. The cap is a global count across all NDCs, but `getAssetUnstaking()` is called per-NDC per-asset — the total iteration count scales as `assets × NDCs × (withdrawals/NDC)`.
3. As the protocol adds more supported assets or NDCs (up to `maxNodeDelegatorLimit`), the gas cost multiplies even without increasing the withdrawal count.

### Impact Explanation
If `updateRSETHPrice()` runs out of gas, the rsETH price becomes permanently stale. A stale price means:
- `depositETH()`/`depositAsset()` use an outdated exchange rate (incorrect rsETH minting)
- `initiateWithdrawal()` uses an outdated rate (incorrect asset amounts committed)
- Protocol fee minting is blocked

If `depositETH()`/`depositAsset()` themselves run out of gas (since they also call `getTotalAssetDeposits()`), user deposits are frozen. This constitutes a **temporary freeze of funds** (users cannot deposit) and **permanent freezing of unclaimed yield** (fee minting blocked).

### Likelihood Explanation
`updateRSETHPrice()` is a public function callable by any address. The protocol's own comment acknowledges the threshold is 120 withdrawals. With `maxUncompletedWithdrawalCount` capped at 80, the current mitigation provides a buffer, but the gas cost also scales with the number of supported assets and NDCs. Adding 2 more supported assets (from 3 to 5) with 10 NDCs at 80 withdrawals would increase the iteration count by ~67%, potentially crossing the threshold. This is a realistic operational scenario.

### Recommendation
1. Cache the result of `getQueuedWithdrawals()` per NDC rather than calling it once per asset per NDC. Currently, if there are 3 supported assets, `getQueuedWithdrawals()` is called 3 times per NDC.
2. Add a `startIndex` and `count` parameter to `getAssetUnstaking()` analogous to the external report's recommendation, or maintain a per-asset accounting of unstaking amounts updated incrementally rather than computed on-the-fly.
3. Decouple `getTotalAssetDeposits()` from the deposit/price-update hot path by maintaining a cached/incremental total that is updated on state changes.

### Proof of Concept
Call chain demonstrating the unbounded iteration on-chain:

1. Any address calls `LRTOracle.updateRSETHPrice()` (public, no access control)
2. → `_updateRsETHPrice()` → `_getTotalEthInProtocol()`
3. → For each of N supported assets: `ILRTDepositPool.getTotalAssetDeposits(asset)`
4. → `getAssetDistributionData(asset)` iterates over M NDCs
5. → For each NDC: `INodeDelegator(ndc).getAssetUnstaking(asset)`
6. → `_getDelegationManager().getQueuedWithdrawals(ndc)` returns K withdrawals
7. → Nested loop over K withdrawals × strategies per withdrawal

Total external calls to EigenLayer: N × M. Total loop iterations: N × M × K × S (where S = strategies per withdrawal). With N=3 assets, M=10 NDCs, K=8 withdrawals/NDC, S=2 strategies: 480 inner iterations plus 30 external calls, all within a single transaction.     function getAssetUnstaking(address asset) external view returns (uint256 amount) {
        (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
            _getDelegationManager().getQueuedWithdrawals(address(this));

        for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
            IDelegationManager.Withdrawal memory withdrawal = queuedWithdrawals[withdrawalIndex];

            for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
                IStrategy strategy = withdrawal.strategies[strategyIndex];

                address strategyAsset = address(strategy) == address(lrtConfig.beaconChainETHStrategy())
                    ? LRTConstants.ETH_TOKEN
                    : address(strategy.underlyingToken());

                if (strategyAsset != asset) continue;

                uint256 sharesToUnstake = withdrawalShares[withdrawalIndex][strategyIndex];
                amount += strategyAsset == LRTConstants.ETH_TOKEN
                    ? sharesToUnstake
                    : strategy.sharesToUnderlyingView(sharesToUnstake);
            }
        }
    }
