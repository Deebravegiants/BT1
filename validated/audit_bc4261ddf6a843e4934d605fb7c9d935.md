### Title
`respond_verify_foreign_tx` Does Not Cross-Validate `response.payload_hash` Against `request` — (`crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` accepts two correlated inputs — a `VerifyForeignTransactionRequest` (used as the pending-yield lookup key) and a `VerifyForeignTransactionResponse` (containing a caller-supplied `payload_hash` and signature) — but never verifies that `response.payload_hash` is actually derived from the `request` data. A single Byzantine attested MPC participant can supply a valid `(payload_hash, signature)` pair produced for one pending request and use it to resolve the pending yields of a completely different request, delivering a fraudulent `VerifyForeignTransactionResponse` to callers.

---

### Finding Description

In `respond_verify_foreign_tx`, the signature check uses `response.payload_hash` directly as the message hash:

```rust
// crates/contract/src/lib.rs:726-734
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,   // ← taken verbatim from response, never derived from request
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

After the signature check passes, the function resolves pending yields keyed on `request`:

```rust
// crates/contract/src/lib.rs:749-753
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,                              // ← lookup key
    serde_json::to_vec(&response).unwrap(), // ← response delivered verbatim
)
``` [2](#0-1) 

There is no assertion that `response.payload_hash` equals `ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: request.request.clone(), values: <any> }).compute_msg_hash()`. The `request` field is used only as a map key; the `response` (including its `payload_hash`) is delivered verbatim to every queued caller.

Compare this with the `respond` function for ordinary signatures, which derives the expected public key and payload directly from the stored `request` fields and verifies the signature against them — there is no analogous cross-check in `respond_verify_foreign_tx`. [3](#0-2) 

The canonical hash that nodes are supposed to sign is:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload { request, values }))
``` [4](#0-3) 

The contract never recomputes this hash from `request` to compare against `response.payload_hash`.

---

### Impact Explanation

A Byzantine attested MPC participant can:

1. Observe a legitimate MPC response for pending **request B** (Bitcoin `tx_id=Y`), which they participated in signing: `response_B = { payload_hash: H(ForeignTxSignPayload{B, [BlockHash(Y_hash)]}), signature: valid_sig }`.
2. Call `respond_verify_foreign_tx(request = A, response = response_B)` where **request A** (Bitcoin `tx_id=X`) is a different pending entry.
3. The contract verifies: signature is valid over `response_B.payload_hash` ✓ (it is — it was honestly produced for request B).
4. The contract resolves all pending yields for **request A** with `response_B`.
5. Every caller waiting on request A receives `response_B`.

**For bridge contracts that do not independently verify `response.payload_hash`** (i.e., do not call `ForeignChainSignatureVerifier::verify_signature` from the SDK), this constitutes a **forged foreign-chain verification**: the bridge accepts an MPC-signed attestation for the wrong transaction data, enabling invalid bridge execution. [5](#0-4) 

**For bridge contracts that do verify the payload hash**, the impact is a permanent DoS on request A: the yield is consumed with a wrong response and cannot be retried without resubmitting a new `verify_foreign_transaction` call and paying the deposit again. The pending entry is gone. [6](#0-5) 

This maps to the allowed impact: **High — forged foreign-chain verification that causes invalid bridge execution**, and also **Medium — request-lifecycle manipulation that breaks production safety/accounting invariants**.

---

### Likelihood Explanation

The attacker must be a single attested MPC participant (one that has passed TEE attestation and is registered in the protocol). No threshold collusion is required. The attacker needs only to:

1. Be an attested participant — they already are, by definition of being an MPC node.
2. Observe a legitimate response for any pending request B — they participated in signing it.
3. Know of another pending request A — all pending requests are visible on-chain via `get_pending_verify_foreign_tx_request`.

All three conditions are trivially met by any MPC node. The attack requires a single malicious transaction submission.

---

### Recommendation

In `respond_verify_foreign_tx`, recompute the expected payload hash from `request` and verify that `response.payload_hash` matches. Since the extracted `values` are not stored on-chain, the contract cannot fully recompute the hash, but it can enforce that the hash is at least consistent with the request by requiring the caller to also supply the `ForeignTxSignPayload` (including `values`) and recomputing the hash on-chain:

```rust
// Recompute the expected hash from the supplied payload and verify it matches
let expected_hash = foreign_tx_payload.compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
// Also verify that the payload's request field matches the pending request
if foreign_tx_payload.request() != &request.request {
    return Err(RespondError::RequestMismatch.into());
}
```

Alternatively, store the expected payload hash in the pending-request map at enqueue time (submitted by the first responding node and agreed upon by threshold), so the contract can verify `response.payload_hash` against the stored expected hash without needing the full payload.

### Proof of Concept

```
1. Alice submits verify_foreign_transaction(request=A: Bitcoin tx_id=X, extractors=[BlockHash])
   → pending_verify_foreign_tx_requests[A] = [yield_alice]

2. Bob submits verify_foreign_transaction(request=B: Bitcoin tx_id=Y, extractors=[BlockHash])
   → pending_verify_foreign_tx_requests[B] = [yield_bob]

3. MPC nodes honestly compute response_B:
   payload_B = ForeignTxSignPayload{request: B, values: [BlockHash(real_Y_hash)]}
   payload_hash_B = SHA256(borsh(payload_B))
   sig_B = MPC_sign(payload_hash_B)   ← Byzantine node participates honestly here

4. Byzantine attested participant submits:
   respond_verify_foreign_tx(
     request = A,                                    ← request A's pending entry
     response = { payload_hash: payload_hash_B,      ← hash from request B
                  signature: sig_B }                 ← valid sig over payload_hash_B
   )

5. Contract checks:
   - assert_caller_is_attested_participant_and_protocol_active() ✓
   - verify_ecdsa_signature(sig_B, payload_hash_B, root_pk) ✓ (valid)
   - resolve_yields_for(pending[A], response_B) → yield_alice resumed with response_B

6. Alice's bridge contract receives:
   VerifyForeignTransactionResponse {
     payload_hash: payload_hash_B,   ← hash of tx_id=Y data, not tx_id=X
     signature: sig_B
   }

7a. If Alice's bridge does NOT verify payload_hash against expected:
    → Accepts fraudulent attestation for tx_id=X (forged verification)

7b. If Alice's bridge DOES verify:
    → Rejects (hash mismatch), but yield_alice is permanently consumed
    → Alice must resubmit and pay deposit again; request A is DoS'd
```

```rust
/// Resume every yield queued for `request` with `response_bytes`, draining the
/// fan-out map in one pass. Returns `Err(RequestNotFound)` if the map held no entry.
///
/// Resuming a yield that has already timed out is a no-op at the SDK level.
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
