Audit Report

## Title
`DepositAllowlistExtension` Checks `owner` Instead of `sender`, Allowing Any Unprivileged Caller to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual depositor, i.e. `msg.sender` of `addLiquidity`) and gates only on `owner` (the LP-position recipient, a free caller-supplied parameter). Because `owner` is not validated against `msg.sender` anywhere in `MetricOmmPool.addLiquidity`, any non-whitelisted address can bypass the allowlist by nominating any whitelisted address as `owner`. The pool's intent to restrict who may deposit liquidity is completely defeated.

## Finding Description
`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` with no constraint that it equals `msg.sender`, and passes both `msg.sender` (as `sender`) and `owner` to the extension hook: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` encodes both and calls the extension: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` then **discards `sender`** (unnamed first parameter) and checks only `owner`: [3](#0-2) 

The sibling `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual caller): [4](#0-3) 

The existing unit tests confirm the live check is on `owner`, not `sender` — `address(0)` is passed as `sender` and `depositor` as `owner` in both the revert and pass cases: [5](#0-4) 

After the extension check passes, `LiquidityLib.addLiquidity` issues the `metricOmmModifyLiquidityCallback` to `msg.sender` (the actual caller, Bob), requiring Bob to pay the tokens: [6](#0-5) 

LP shares are minted into Alice's position key, but Bob pays and controls the token flow. The `removeLiquidity` path enforces `msg.sender == owner`, so Alice retains the position — but the pool admin's access-control boundary is fully bypassed.

## Impact Explanation
The `DepositAllowlistExtension` is completely ineffective as an access-control gate. Any non-whitelisted address (Bob) can deposit into an allowlisted pool by nominating any whitelisted address (Alice) as `owner`. Bob pays the tokens via callback, pool bin state (`binTotals`, `_binStates`, `curPosInBin`, `curBinIdx`) is mutated without authorization, and the pool admin's intent to restrict who may interact with pool liquidity is fully defeated. This constitutes an admin-boundary break via an unprivileged path — a non-whitelisted caller can unconditionally alter pool bin balances and LP share accounting.

## Likelihood Explanation
Exploitation requires no special privileges. Any address can call `addLiquidity` directly on the pool with a known whitelisted `owner`. Whitelisted addresses are discoverable on-chain from `AllowedToDepositSet` events emitted by `setAllowedToDeposit`. The bypass is trivial, unconditional, and repeatable. [7](#0-6) 

## Recommendation
Check `sender` (the actual depositor) instead of `owner` (the LP-position recipient), mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
// DepositAllowlistExtension.sol — fixed
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The `allowedDepositor` mapping, `setAllowedToDeposit`, and `isAllowedToDeposit` must also be reviewed to ensure the semantics align with checking the actual depositor (`sender`) rather than the position owner.

## Proof of Concept

```
Setup:
  pool admin deploys pool with DepositAllowlistExtension
  pool admin calls: extension.setAllowedToDeposit(pool, alice, true)
  bob is NOT on the allowlist

Attack:
  bob calls pool.addLiquidity(
      owner        = alice,   // whitelisted — passes the check
      salt         = 0,
      deltas       = <any valid delta>,
      callbackData = <bob's payment data>,
      extensionData = ""
  )

Extension check (DepositAllowlistExtension.beforeAddLiquidity):
  sender (bob) is discarded
  allowedDepositor[pool][alice] == true  →  no revert

Result:
  LP shares minted to alice's position key
  pool calls metricOmmModifyLiquidityCallback on bob
  bob pays tokens
  bob has successfully deposited into an allowlisted pool without being on the allowlist

Foundry test plan:
  1. Deploy pool with DepositAllowlistExtension as before-add-liquidity hook
  2. setAllowedToDeposit(pool, alice, true)
  3. vm.prank(bob); pool.addLiquidity(alice, 0, validDelta, "", "")
  4. Assert call succeeds (no revert)
  5. Assert positionBinShares[keccak256(alice, 0, bin)] > 0
```
