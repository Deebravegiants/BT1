### Title
Native ETH `finTransfer` Permanently Reverts Due to Double-Spend of `msg.value` in `finTransferExtension` — (`evm/src/omni-bridge/contracts/OmniBridgeWormhole.sol`)

---

### Summary

`OmniBridgeWormhole.finTransferExtension` forwards the full `msg.value` to Wormhole as a publishing fee. When the transfer is for native ETH (`tokenAddress == address(0)`), `OmniBridge.finTransfer` has already disbursed `payload.amount` ETH to the recipient before calling `finTransferExtension`. The contract therefore lacks sufficient ETH to satisfy `{value: msg.value}` in the Wormhole call, causing every native ETH finalization to revert. The NEAR-side burn is already committed, so affected users permanently lose their bridged ETH.

---

### Finding Description

`OmniBridge.finTransfer` is `payable` and handles native ETH redemptions by sending `payload.amount` ETH directly to the recipient before delegating to the extension hook:

```solidity
// OmniBridge.sol line 317-322
if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();
}
// ...
finTransferExtension(payload);   // line 357
```

`OmniBridgeWormhole.finTransferExtension` then attempts to pay the Wormhole publishing fee using the **full original** `msg.value`:

```solidity
// OmniBridgeWormhole.sol line 109
_wormhole.publishMessage{value: msg.value}(
    wormholeNonce,
    messagePayload,
    _consistencyLevel
);
```

At the point `publishMessage` is called, the contract's ETH balance from this transaction has already been reduced by `payload.amount`. The contract holds only `msg.value − payload.amount` ETH from the current call. Attempting to forward `msg.value` causes the EVM to revert the entire transaction.

The sibling path `initTransferExtension` correctly avoids this by computing a residual `extensionValue` before any ETH is disbursed and passing only that residual to Wormhole:

```solidity
// OmniBridge.sol line 391
extensionValue = msg.value - amount - nativeFee;
// ...
initTransferExtension(..., extensionValue);   // line 415

// OmniBridgeWormhole.sol line 143
_wormhole.publishMessage{value: value}(...)   // value == extensionValue
```

`finTransferExtension` has no equivalent residual computation, making it structurally broken for native ETH.

---

### Impact Explanation

**Frozen redemption path — Critical.**

A user who initiates a native ETH bridge from NEAR causes the NEAR-side contract to burn/lock their tokens. The EVM-side `finTransfer` is the only redemption path. Because every call to `finTransfer` with `tokenAddress == address(0)` reverts in `OmniBridgeWormhole`, the user can never receive their ETH on EVM. The NEAR-side state is already finalized; there is no rollback. The ETH equivalent is permanently unclaimable.

---

### Likelihood Explanation

**High.** Native ETH is an explicitly supported asset class — `initTransfer` and `finTransfer` both contain dedicated `tokenAddress == address(0)` branches. Any relayer attempting to finalize a native ETH transfer will trigger the revert deterministically, regardless of how much ETH they attach. Sending more ETH does not help: `msg.value` grows proportionally, and the shortfall (`payload.amount`) remains constant. The only escape is a pre-existing contract ETH balance ≥ `payload.amount`, which is not guaranteed and would drain the contract's reserve.

---

### Recommendation

Compute the residual Wormhole fee in `finTransfer` before disbursing ETH to the recipient, mirroring the pattern used in `initTransfer`:

```diff
// OmniBridge.sol
function finTransfer(...) external payable ... {
+   uint256 wormholeFee = msg.value;
    ...
    if (payload.tokenAddress == address(0)) {
        (bool success, ) = payload.recipient.call{value: payload.amount}("");
        if (!success) revert FailedToSendEther();
+       wormholeFee = msg.value - payload.amount;
    }
    ...
-   finTransferExtension(payload);
+   finTransferExtension(payload, wormholeFee);
}

// OmniBridgeWormhole.sol
- function finTransferExtension(BridgeTypes.TransferMessagePayload memory payload) internal override {
+ function finTransferExtension(BridgeTypes.TransferMessagePayload memory payload, uint256 wormholeFee) internal override {
    ...
-   _wormhole.publishMessage{value: msg.value}(...);
+   _wormhole.publishMessage{value: wormholeFee}(...);
}
```

### Proof of Concept

1. User bridges 1 ETH from NEAR to EVM via `OmniBridgeWormhole`. NEAR-side tokens are burned.
2. Relayer calls `finTransfer` with `payload.tokenAddress = address(0)`, `payload.amount = 1 ETH`, and `msg.value = 1 ETH + wormhole_fee`.
3. `finTransfer` sends 1 ETH to `payload.recipient` — succeeds. Contract now holds only `wormhole_fee` ETH from this call.
4. `finTransferExtension` executes `_wormhole.publishMessage{value: msg.value}` — attempts to forward `1 ETH + wormhole_fee`, but only `wormhole_fee` is available. EVM reverts.
5. The entire transaction is rolled back. `completedTransfers[nonce]` was set to `true` before the ETH send (line 287), so the nonce is consumed and the transfer is permanently unfinalizeable.

Wait — `completedTransfers[payload.destinationNonce] = true` is set at line 287, **before** the ETH disbursement and the Wormhole call. Because the revert unwinds all state changes, the nonce is **not** permanently consumed (the revert restores it). However, the NEAR-side burn is irreversible, and the relayer will keep retrying and reverting indefinitely, making the redemption path permanently frozen in practice.
