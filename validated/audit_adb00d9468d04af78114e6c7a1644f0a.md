### Title
Deposit Permanently Locked in `submit_participant_info` When Existing Participant Re-Submits Attestation — (File: crates/contract/src/lib.rs)

### Summary
`submit_participant_info` is marked `#[payable]` but only handles the attached deposit inside a conditional branch. When an existing participant re-submits their attestation (not a new entry), the entire attached deposit is silently consumed by the contract with no refund path, permanently locking those NEAR tokens.

### Finding Description
In `submit_participant_info`, after `add_participant` succeeds, the code computes a flag:

```rust
let caller_is_not_participant = self.voter_account().is_err();
let is_new_attestation = matches!(
    attestation_insertion_result,
    ParticipantInsertion::NewlyInsertedParticipant
);

let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... storage cost check and excess refund
}

Ok(())
```

When `attestation_storage_must_be_paid_by_caller` evaluates to `false` — i.e., the caller is an already-accepted participant **and** the attestation is an update (not a new entry) — the `if` block is skipped entirely. `env::attached_deposit()` is never read, never checked, and never refunded. Any NEAR tokens the caller attached are absorbed into the contract balance with no recovery path.

The function is `#[payable]`, so the NEAR runtime accepts any deposit without complaint. The function returns `Ok(())`, so no panic occurs and no rollback is triggered. The deposit is gone. [1](#0-0) 

The analog to the external report is direct: in the `Flip` contract, only the fee portion was forwarded and the rest of the bet stayed locked; here, only the storage-cost branch handles the deposit, and the re-submission branch leaves the entire deposit locked.

### Impact Explanation
Any NEAR tokens attached to a re-submission call by an existing participant are permanently locked in the MPC contract. The contract's NEAR balance grows by the deposit amount with no corresponding liability or refund obligation recorded anywhere. This breaks the production accounting invariant that every deposit to a `#[payable]` method must be either consumed for a declared purpose (storage staking) or refunded to the caller. Repeated re-submissions (the node re-submits every hour via `periodic_attestation_submission`) compound the loss if any deposit is ever attached.

**Impact: Medium** — balance/accounting invariant broken; funds permanently frozen in the contract.

### Likelihood Explanation
The `mpc-node` software currently attaches `0` deposit for re-submissions (the docs explicitly note "the node attaches `0`"). The bug therefore does not fire in normal automated operation. However:

1. The operator guide instructs participants to call `submit_participant_info` **manually with `--deposit`** for first-time joins; a participant who mistakenly re-runs that manual command after already being accepted triggers the lock.
2. Any future change to the node's deposit logic (e.g., attaching a small deposit as a spam guard) would silently lock funds on every hourly re-submission.
3. The function is publicly callable; any account that has previously been accepted as a participant and calls it again with a deposit loses those tokens.

**Likelihood: Low** — requires a manual mis-invocation or a future node-software change, but the code path is reachable today.

### Recommendation
Add an unconditional refund of any excess deposit in the `else` branch (or after the `if` block):

```rust
if attestation_storage_must_be_paid_by_caller {
    let storage_used = env::storage_usage().saturating_sub(initial_storage);
    let cost = env::storage_byte_cost().saturating_mul(storage_used as u128);
    let attached = env::attached_deposit();
    if attached < cost {
        return Err(InvalidParameters::InsufficientDeposit { ... }.into());
    }
    if let Some(diff) = attached.checked_sub(cost)
        && diff > NearToken::from_yoctonear(0)
    {
        Promise::new(account_id).transfer(diff).detach();
    }
} else {
+   // Refund any deposit the caller attached; re-submissions by existing
+   // participants require no storage payment.
+   let attached = env::attached_deposit();
+   if attached > NearToken::from_yoctonear(0) {
+       Promise::new(account_id).transfer(attached).detach();
+   }
}
```

### Proof of Concept
1. Participant `alice.near` has already submitted a valid attestation; `voter_account()` returns `Ok` for her.
2. Alice calls `submit_participant_info` again (e.g., to refresh her attestation) and attaches `1 NEAR` as a deposit.
3. `add_participant` succeeds and returns `ParticipantInsertion::UpdatedParticipant` (not `NewlyInsertedParticipant`).
4. `is_new_attestation = false`, `caller_is_not_participant = false` → `attestation_storage_must_be_paid_by_caller = false`.
5. The `if` block is skipped; `Ok(())` is returned.
6. Alice's `1 NEAR` is now locked in the contract with no refund scheduled and no state entry tracking the obligation. [2](#0-1)

                    attached: attached.as_yoctonear(),
                    required: cost.as_yoctonear(),
