### Title
`danglingVRFKeyHashes` uses `Map.withoutKeys` (hard-delete) instead of count-decrement, silently erasing shared VRF entries and breaking the post-v11 VRF uniqueness invariant — (`eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/PoolReap.hs`)

---

### Summary

In `poolReapTransition`, the `danglingVRFKeyHashes` set is computed as the set of old VRF keys being abandoned by re-registering pools. These are then removed from `psVRFKeyHashes` via `Map.withoutKeys`, which **hard-deletes the entire map entry** regardless of its reference count. When a shared VRF key (count ≥ 2, legal pre-v11) is only partially abandoned (one pool switches away, another retains it), the hard-delete removes the key entirely. After the epoch boundary, the VRF key is absent from `psVRFKeyHashes`, so the post-v11 uniqueness guard (`Map.notMember sppVrf psVRFKeyHashes`) passes for a new pool registration with that same VRF key. Two active pools now share a VRF key in `psStakePools`, silently breaking the invariant the hardfork was designed to enforce.

The `danglingVRFKeyHashes` computation identifies VRF keys that are being abandoned by a pool re-registration: [1](#0-0) 

It merges `psStakePoolsL` and `psFutureStakePoolParamsL` using `Map.dropMissing` on both sides, meaning only pools present in **both** maps are considered. For each such pool, if the current VRF differs from the future VRF, the current (old) VRF is added to the dangling set. Critically, pool B — which only appears in `psStakePools` and has no future params — is silently dropped by `Map.dropMissing`. Its continued use of VRF1 is invisible to this computation.

These dangling hashes are then removed from `psVRFKeyHashes` via a hard-delete: [2](#0-1)

### Impact Explanation

Two active pools share a VRF key used for Ouroboros Praos slot leader election. Both pools produce identical VRF outputs for every slot lottery, meaning both are elected for the same slots. This causes:

1. **Invalid block production attribution**: competing blocks for the same slot; one is orphaned, the other accepted. Rewards are attributed to the accepted block's pool, not the other.
2. **Reward misdirection**: delegators of the orphaned pool's blocks receive no rewards for those slots, constituting a permanent, unrecoverable loss of staking rewards.
3. **Ledger state inconsistency**: `psVRFKeyHashes` no longer accurately reflects `psStakePools`, breaking the invariant that `psVRFKeyHashes` is the authoritative reference count for VRF key usage. Any downstream logic relying on this map is silently incorrect.

This matches: **High. Deterministic disagreement between honest nodes from ledger rule evaluation** (the ledger state is permanently inconsistent after the epoch boundary), and **High. Permanent freezing of funds, deposits, rewards, or withdrawals** (staking rewards are permanently misdirected for affected delegators without any recovery path short of a hard fork to repair `psVRFKeyHashes`).

### Recommendation

Replace the `Map.withoutKeys danglingVRFKeyHashes` hard-delete with the same count-decrement logic used for retired pools. Specifically, in `poolReapTransition`, change:

```haskell
-- CURRENT (incorrect): hard-deletes entire entry regardless of count
. (`Map.withoutKeys` danglingVRFKeyHashes)
```

to:

```haskell
-- CORRECT: decrements count by 1, only removes entry when count reaches 0
. removeVRFKeyHashOccurrences (Set.toList danglingVRFKeyHashes)
``` [2](#0-1) 

This mirrors the correct treatment of `retiredVRFKeyHashes` and ensures that a VRF key shared by N pools is only fully removed from `psVRFKeyHashes` after all N pools have either retired or switched to a different VRF key.

### Proof of Concept

Concrete ledger state trace (all steps are standard pool certificate transactions):

```
Pre-v11 (pvMajor < 11):
  submit RegPool(A, VRF1)  → psStakePools = {A→VRF1}
  submit RegPool(B, VRF1)  → psStakePools = {A→VRF1, B→VRF1}  [allowed pre-v11]
  psVRFKeyHashes = {}

Hardfork to v11:
  populateVRFKeyHashes called
  psVRFKeyHashes = {VRF1→2}

Post-v11:
  submit RegPool(A, VRF2)  [re-registration]
    check: VRF2 ∉ psVRFKeyHashes → passes
    updateFutureVRFKeyHash: Nothing case → insert VRF2→1
  psFutureStakePoolParams = {A→VRF2}
  psVRFKeyHashes = {VRF1→2, VRF2→1}

Epoch boundary (POOLREAP):
  danglingVRFKeyHashes:
    merge psStakePools={A→VRF1, B→VRF1} with psFutureStakePoolParams={A→VRF2}
    pool A: VRF1 ≠ VRF2 → Just VRF1
    pool B: dropMissing (not in psFutureStakePoolParams) → dropped
  danglingVRFKeyHashes = {VRF1}
  Map.withoutKeys {VRF1} applied to psVRFKeyHashes
  psVRFKeyHashes = {VRF2→1}   ← VRF1 fully removed, but B still holds it!
  psStakePools = {A→VRF2, B→VRF1}  ← B still active with VRF1

Post-epoch:
  submit RegPool(C, VRF1)  [new registration]
    check: VRF1 ∉ psVRFKeyHashes={VRF2→1} → True → PASSES
  psStakePools = {A→VRF2, B→VRF1, C→VRF1}  ← B and C share VRF1
  psVRFKeyHashes = {VRF1→1, VRF2→1}  ← count=1 but two pools hold VRF1

Invariant violated: ∃ B≠C active in psStakePools with spsVrf B = spsVrf C = VRF1
```
