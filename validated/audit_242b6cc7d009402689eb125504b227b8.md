### Title
Expired DRep Delegation Satisfies Withdrawal Governance-Participation Check — (`File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs`)

---

### Summary

`validateWithdrawalsDelegated` only checks that a staking key hash has *some* DRep delegation recorded; it never verifies that the delegated DRep is still active (non-expired). An unprivileged transaction sender can therefore withdraw staking rewards while their stake is delegated to an expired DRep, bypassing the governance-participation requirement that the Conway era imposes on withdrawals. The ledger's own test suite confirms this is reachable and currently succeeds.

---

### Finding Description

Conway introduced a rule that key-hash staking credentials must be delegated to a DRep before they can withdraw rewards (`ConwayWdrlNotDelegatedToDRep`). The intent is to ensure every ADA holder participates in governance as a precondition for accessing their rewards.

The enforcement lives in `validateWithdrawalsDelegated`:

```haskell
isNotDRepDelegated keyHash = isNothing $ do
    accountState <- lookupAccountState (KeyHashObj keyHash) accounts
    accountState ^. dRepDelegationAccountStateL
``` [1](#0-0) 

The predicate returns `True` (i.e., "not delegated") only when `dRepDelegationAccountStateL` is `Nothing`. It does **not** inspect the `DRepState` of the referenced DRep to check whether `drepExpiry` has passed. An expired DRep remains in `vsDReps` with its `drepExpiry` field set to a past epoch; the delegation pointer in the account state is still `Just <expired-drep-cred>`, so the check passes.

The `dRepAcceptedRatio` function in the `RATIFY` rule explicitly skips expired DReps:

```haskell
Just drepState
  | reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, so we don't consider it
``` [2](#0-1) 

So the expired DRep's stake contributes nothing to governance ratification, yet the withdrawal check treats the delegation as valid.

Two concrete paths are confirmed by the test suite:

**Path 1 – delegate to an already-expired DRep, then withdraw (regular `it` test, not disabled):**

```haskell
isDRepExpired drep `shouldReturn` True
_ <- delegateToDRep cred (Coin 1_000_000) (DRepCredential drep)
submitTx_ $ mkBasicTx $ mkBasicTxBody
    & withdrawalsTxBodyL .~ Withdrawals [(ra, balance)]
-- succeeds
``` [3](#0-2) 

**Path 2 – DRep expires after delegation, then withdraw (`disableInConformanceIt`, open formal-spec issue #635):**

```haskell
isDRepExpired drep `shouldReturn` True
submitTx_ $ mkBasicTx $ mkBasicTxBody
    & withdrawalsTxBodyL .~ Withdrawals [(ra, balance)]
-- also succeeds
``` [4](#0-3) 

The second path is explicitly flagged as a known divergence from the formal specification (formal-ledger-specifications issue #635) and is currently disabled in conformance testing.

---

### Impact Explanation

**Medium — withdrawals outside design parameters.**

The Conway era's DRep-delegation precondition for withdrawals is a protocol-level governance-participation requirement. By delegating to an expired DRep, any staking-key holder can drain their reward account while their stake is invisible to governance ratification. This:

- Allows reward withdrawals that the protocol design explicitly intended to gate behind active governance participation.
- Reduces the effective active-DRep stake used in `dRepAcceptedRatio`, potentially making it harder for legitimate governance actions to reach their required threshold if many accounts exploit this path.
- Matches the allowed Medium impact: *"attacker-controlled transactions… modify… withdrawals outside design parameters."*

There is no direct ADA creation or destruction, so Critical/High impacts do not apply.

---

### Likelihood Explanation

**Medium.** Any staking-key holder can trigger this with a single transaction after their DRep expires (or by deliberately delegating to a known-expired DRep). No privileged access, key compromise, or consensus majority is required. The entry path is fully unprivileged: submit a `DelegTxCert` to an expired DRep credential, then submit a withdrawal transaction. The ledger's own test at line 148–173 demonstrates this succeeds unconditionally post-bootstrap.

---

### Recommendation

Extend `validateWithdrawalsDelegated` to also verify that the referenced DRep is not expired at the current epoch. Concretely, after resolving `dRepDelegationAccountStateL` to a `DRep` credential, look it up in `vsDReps` and compare `drepExpiry` against `reCurrentEpoch` (the same comparison already used in `dRepAcceptedRatio`). A delegation to `DRepAlwaysAbstain` or `DRepAlwaysNoConfidence` should continue to be accepted unconditionally, as those pseudo-DReps have no expiry. [5](#0-4) 

---

### Proof of Concept

Attacker-controlled entry path (post-bootstrap, protocol version ≥ 10):

1. Register a staking credential `cred` and accumulate rewards.
2. Register a DRep `drep` (or identify any already-expired DRep in `vsDReps`).
3. Wait (or arrange) for `drep` to expire: `reCurrentEpoch > drepExpiry drepState`.
4. Submit a `DelegTxCert cred (DRepCredential drep)` — accepted because `ConwayDelegPredFailure` does not check DRep liveness for delegation.
5. Submit a withdrawal transaction for `cred`'s reward account.

`validateWithdrawalsDelegated` finds `dRepDelegationAccountStateL = Just (DRepCredential drep)` and passes. The withdrawal is applied. The expired DRep's stake is excluded from `dRepAcceptedRatio` at the next ratification epoch, so the withdrawn stake never contributed to governance.

The test at `LedgerSpec.hs:148–173` reproduces this exactly and asserts `submitTx_` succeeds (no predicate failure). [6](#0-5) [7](#0-6)

validateWithdrawalsDelegated ::
  ( EraTx era
  , ConwayEraCertState era
  ) =>
  Accounts era -> Tx l era -> Test (ConwayLedgerPredFailure era)
validateWithdrawalsDelegated accounts tx =
  let wdrls = unWithdrawals $ tx ^. bodyTxL . withdrawalsTxBodyL
      wdrlsKeyHashes =
        [ kh | (ra, _) <- Map.toList wdrls, Just kh <- [credKeyHash $ ra ^. accountAddressCredentialL]
        ]
      isNotDRepDelegated keyHash = isNothing $ do
        accountState <- lookupAccountState (KeyHashObj keyHash) accounts
        accountState ^. dRepDelegationAccountStateL
      nonExistentDelegations =
        filter isNotDRepDelegated wdrlsKeyHashes
   in failureOnNonEmpty nonExistentDelegations ConwayWdrlNotDelegatedToDRep

```haskell
dRepAcceptedRatio RatifyEnv {reDRepDistr, reDRepState, reCurrentEpoch} gasDRepVotes govAction =
  toInteger yesStake %? toInteger totalExcludingAbstainStake
  where
    accumStake (!yes, !tot) drep (CompactCoin stake) =
      case drep of
        DRepCredential cred ->
          case Map.lookup cred reDRepState of
            Nothing -> (yes, tot) -- drep is not registered, so we don't consider it
            Just drepState
              | reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, so we don't consider it
              | otherwise ->
                  case Map.lookup cred gasDRepVotes of
                    -- drep hasn't voted for this action, so we don't count
                    -- the vote but we consider it in the denominator:
                    Nothing -> (yes, tot + stake)
                    Just VoteYes -> (yes + stake, tot + stake)
                    Just Abstain -> (yes, tot)
                    Just VoteNo -> (yes, tot + stake)
        DRepAlwaysNoConfidence ->
          case govAction of
            NoConfidence _ -> (yes + stake, tot + stake)
            _ -> (yes, tot + stake)
        DRepAlwaysAbstain -> (yes, tot)
    (yesStake, totalExcludingAbstainStake) = Map.foldlWithKey' accumStake (0, 0) reDRepDistr
```
