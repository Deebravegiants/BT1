Audit Report

## Title
Excess ETH Permanently Stranded and Stealable via Unrestricted `refundETH()` — (`metric-periphery/contracts/base/PeripheryPayments.sol`, `metric-periphery/contracts/MetricOmmSimpleRouter.sol`)

## Summary
The `pay()` helper in `PeripheryPayments` wraps exactly the required ETH amount when `nativeBalance >= value`, silently leaving any surplus in the contract. Payable swap entry points (`exactOutputSingle`, `exactInputSingle`, `exactInput`, `exactOutput`) issue no automatic refund. The unrestricted `refundETH()` function sends `address(this).balance` to any caller, enabling a front-runner to steal stranded ETH left by a prior swap.

## Finding Description
**Root cause — `pay()` wraps only `value`, not `nativeBalance`:**
In `PeripheryPayments.sol` L75-77, when `nativeBalance >= value`, exactly `value` wei is deposited into WETH and transferred to the pool. The remainder `nativeBalance - value` is never accounted for or returned.

**Trigger path — `exactOutputSingle` with excess ETH:**
A caller of `exactOutputSingle` cannot know `amountIn` before execution and must supply at least `amountInMaximum` as `msg.value`. The swap callback (`_justPayCallback` → `pay()`) wraps only the actual `amountIn`. The gap `msg.value − amountIn` remains in the router after `exactOutputSingle` returns at L146 (`_clearExpectedCallbackPool()`), with no refund.

**Theft vector — `refundETH()` is unrestricted:**
`refundETH()` at `PeripheryPayments.sol` L58-63 sends `address(this).balance` to `msg.sender` with no access control. Any address can call it at any time and receive the full ETH balance, including surplus left by a prior caller.

**Existing guards are insufficient:**
- `receive()` (L32-34) only blocks plain ETH transfers from non-WETH addresses; it does not affect ETH attached to `payable` function calls.
- The `amountInMaximum` check (L145) only prevents overpayment to the pool; it does not refund the caller.
- The test suite demonstrates the intended multicall+`refundETH` pattern (L106-133) but does not enforce it, and no NatSpec warns callers of the requirement.

## Impact Explanation
A user calling `exactOutputSingle{value: X}(...)` directly (not via `multicall`) with `X > amountIn` permanently loses `X − amountIn` ETH if a front-runner calls `refundETH()` first. This is a direct loss of user principal with no protocol-level recovery path. Severity: Medium — direct ETH loss requiring the user to call outside a `multicall + refundETH` bundle.

## Likelihood Explanation
`exactOutputSingle` is a standard named payable entry point. Nothing in the function signature, NatSpec, or revert messages warns callers to batch `refundETH()`. Users interacting via Etherscan, custom scripts, or wallet integrations that do not construct multicalls will routinely leave surplus ETH. MEV bots continuously scan for this pattern. The condition is repeatable on every such call.

## Recommendation
At the end of each payable swap/liquidity function, automatically refund any remaining native balance to `msg.sender`:

```solidity
function exactOutputSingle(ExactOutputSingleParams calldata params)
    external payable returns (uint256 amountIn)
{
    // ... existing logic ...
    _clearExpectedCallbackPool();
    // Refund unused ETH
    uint256 surplus = address(this).balance;
    if (surplus > 0) _transferETH(msg.sender, surplus);
}
```

Apply the same pattern to `exactInputSingle`, `exactInput`, `exactOutput`, `addLiquidityExactShares`, and `addLiquidityWeighted`.

## Proof of Concept
```solidity
// Alice calls exactOutputSingle directly with excess ETH
// Actual amountIn = 1.5 ETH, Alice sends 2 ETH
uint256 amountIn = router.exactOutputSingle{value: 2 ether}(
    IMetricOmmSimpleRouter.ExactOutputSingleParams({
        pool: address(pool),
        tokenIn: address(weth),
        tokenOut: address(token1),
        zeroForOne: true,
        amountOut: 1_000,
        amountInMaximum: 2 ether,
        recipient: alice,
        deadline: block.timestamp + 60,
        priceLimitX64: 0,
        extensionData: ""
    })
);
// amountIn == 1.5 ETH; 0.5 ETH stranded in router

// Bob (front-runner) calls refundETH() and steals Alice's surplus
vm.prank(bob);
router.refundETH();
assertEq(bob.balance, 0.5 ether);
assertEq(address(router).balance, 0);
```

The existing test `test_multicall_ethInput_exactInputSingle_refundsUnusedEth` (L106-133) confirms the multicall+`refundETH` pattern is required but not enforced. No test covers the single-call path where surplus ETH is left unrecovered.
