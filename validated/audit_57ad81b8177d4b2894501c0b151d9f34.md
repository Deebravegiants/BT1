### Title
`RSETHPriceFeed.latestRoundData` Returns ETH/USD `updatedAt` Instead of rsETHPrice Update Timestamp, Enabling Staleness Mismatch — (`contracts/oracles/RSETHPriceFeed.sol`)

### Summary
`RSETHPriceFeed` composes the rsETH/USD price from two independent sources — the Chainlink ETH/USD feed and `LRTOracle.rsETHPrice` — but returns only the ETH/USD feed's `updatedAt` timestamp. Because `LRTOracle` stores no timestamp for when `rsETHPrice` was last written, the combined feed's staleness metadata is structurally decoupled from one of its two price inputs.

### Finding Description
In `latestRoundData` (and `getRoundData`), all five return values are first populated from `ETH_TO_USD`: [1](#0-0) 

Then only `answer` is overwritten with the composed price: [2](#0-1) 

`updatedAt`, `startedAt`, `roundId`, and `answeredInRound` all remain as-is from the ETH/USD feed. The rsETH component of the answer comes from `LRTOracle.rsETHPrice`: [3](#0-2) 

`LRTOracle` tracks no update timestamp for this variable — confirmed: there is no `lastUpdatedAt`, `priceTimestamp`, or equivalent field anywhere in the contract. `rsETHPrice` is written only when `updateRSETHPrice()` or `updateRSETHPriceAsManager()` is explicitly called: [4](#0-3) [5](#0-4) 

The ETH/USD Chainlink feed has its own independent heartbeat (typically ~1 hour on mainnet). `rsETHPrice` has no enforced update cadence. The two can diverge arbitrarily.

### Impact Explanation
Any downstream protocol (lending market, vault, DEX) that consumes `RSETHPriceFeed` as a Chainlink-compatible oracle and applies a heartbeat/staleness check on `updatedAt` will observe the ETH/USD feed's timestamp, not the rsETH component's age. Two failure modes result:

1. **False freshness (mispricing risk):** ETH/USD updated 20 minutes ago; `rsETHPrice` not updated for 48 hours. `updatedAt` passes any reasonable heartbeat check. The downstream protocol prices rsETH/USD using a 2-day-old rsETH/ETH rate. Collateral is mispriced; bad debt or incorrect liquidations follow.

2. **Temporary fund freeze:** ETH/USD has not triggered a Chainlink update (price within deviation band, heartbeat not yet elapsed). `updatedAt` is, say, 90 minutes old. A downstream protocol with a 1-hour heartbeat guard reverts all price-dependent operations (borrows, withdrawals, liquidations) even though `rsETHPrice` was refreshed moments ago. User funds are inaccessible until ETH/USD triggers a new Chainlink round.

The second path directly matches **Medium — Temporary freezing of funds**.

### Likelihood Explanation
- `RSETHPriceFeed` is a production contract explicitly designed for external consumption as a Chainlink-compatible feed.
- Downstream protocols routinely apply `block.timestamp - updatedAt > heartbeat` staleness guards (standard practice since the Euler/Mango incidents).
- ETH/USD Chainlink updates are not guaranteed to be synchronous with `updateRSETHPrice()` calls.
- No on-chain mechanism couples the two update cadences.
- The condition is passively reachable — no attacker action required; normal market conditions (low ETH volatility suppressing Chainlink updates, or delayed keeper calls) are sufficient.

### Recommendation
`RSETHPriceFeed` should track and expose the rsETHPrice update timestamp independently. Options:

1. Add a `uint256 public rsETHPriceUpdatedAt` to `LRTOracle`, written alongside `rsETHPrice` in `_updateRsETHPrice()`, and expose it via `IRSETHOracle`. Then in `latestRoundData`, set `updatedAt = min(ethToUsdUpdatedAt, rsETHPriceUpdatedAt)` so the returned timestamp reflects the staleness of the *older* of the two inputs.

2. Alternatively, have `RSETHPriceFeed` revert if either component is stale, rather than silently forwarding a misleading timestamp.
