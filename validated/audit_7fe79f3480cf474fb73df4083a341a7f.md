### Title
Incomplete CKD Output Verification in `respond_ckd` Allows Single Byzantine Node to Submit Unauthorized Key Derivation Output — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` performs a cryptographic validity check on the CKD response **only** when the request uses the `AppPublicKeyPV` variant. When the `AppPublicKey` variant is used, the check is entirely skipped, allowing any single attested participant — below the signing threshold — to submit an arbitrary `CKDResponse` that the contract accepts without any cryptographic verification.

---

### Finding Description

The `respond_ckd` function contains the following conditional:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

When `AppPublicKeyPV` is used, `ckd_output_check` verifies that the submitted `CKDResponse` (`big_y`, `big_c`) is cryptographically consistent with the app public key and the domain's root BLS key. When `AppPublicKey` is used, **no such verification is performed**. The contract proceeds directly to `resolve_yields_for`, resolving the pending yield with whatever `big_y` / `big_c` the caller supplied. [2](#0-1) 

The only guards that remain for the `AppPublicKey` path are:

- `assert_caller_is_signer()` — caller must be a registered signer account
- `protocol_state.is_running_or_resharing()` — protocol must be active
- `accept_requests` — TEE gate must be open
- `assert_caller_is_attested_participant_and_protocol_active()` — caller must be an attested participant [3](#0-2) 

None of these guards verify the *content* of the response. A single attested participant satisfies all four guards and can submit `CKDResponse { big_y: [arbitrary; 48], big_c: [arbitrary; 48] }` for any pending `AppPublicKey` request.

The unit test itself demonstrates this: it submits `big_y = [1u8; 48]` and `big_c = [2u8; 48]` (values that are not valid BLS points) and asserts the call succeeds: [4](#0-3) 

This is structurally identical to the reported Atlas bug: the safety invariant (`ckd_output_check`) is enforced only when a specific variant (`AppPublicKeyPV`) is present; the other variant (`AppPublicKey`) silently skips it.

---

### Impact Explanation

The CKD protocol is designed to derive an application-specific private key under the MPC network's shared BLS root key. The caller receives `(big_y, big_c)` and uses them to reconstruct the derived secret. If a malicious node controls those values, it can:

1. Choose `big_y` to be a BLS point for which it knows the discrete logarithm.
2. The user derives an application key from the attacker-controlled `big_y`.
3. The attacker recovers the user's application private key, enabling theft of any funds or secrets protected by that key.

This constitutes **unauthorized confidential key derivation output without the required participant authorization** — a single node below the signing threshold produces a CKD result that the threshold protocol was supposed to guarantee.

---

### Likelihood Explanation

- The `AppPublicKey` variant is a first-class production variant exposed in the contract ABI and exercised in sandbox tests.
- Any single attested MPC participant (a Byzantine node strictly below threshold) can call `respond_ckd` with a fabricated response.
- The attack is a simple race: the malicious node watches the chain for a pending `AppPublicKey` CKD request and submits its fake response before the honest leader does. `resolve_yields_for` resolves on the first accepted call, so the honest response arriving later is silently discarded.
- No special tooling, leaked keys, or network-level access is required beyond being an attested participant.

---

### Recommendation

1. **Require `AppPublicKeyPV` for all production CKD requests.** Deprecate or remove the `AppPublicKey` variant so that `ckd_output_check` is always executed.
2. **If `AppPublicKey` must be retained**, add a threshold-signature-based proof of correctness (analogous to how `respond` verifies the ECDSA signature against the derived public key) so that a single node cannot unilaterally resolve a CKD yield.
3. **Restrict `respond_ckd` to a specific protocol phase** (analogous to the Atlas fix that restricted `reconcile()` to the `SolverOperations` phase), ensuring the response can only be submitted after the full threshold computation has completed.

---

```

       app_public_key: AppPublicKey(<any BLS G1 point>),
       derivation_path: "my/path",
       domain_id: <CKD domain>,
   })
   → contract enqueues a pending CKDRequest under the AppPublicKey variant.

2. Malicious attested node calls respond_ckd(
       request = <the pending CKDRequest>,
       response = CKDResponse {
           big_y: Bls12381G1PublicKey([ATTACKER_CONTROLLED; 48]),
           big_c: Bls12381G1PublicKey([ATTACKER_CONTROLLED; 48]),
       }
   )
   → respond_ckd hits the AppPublicKey arm → no ckd_output_check → resolve_yields_for
     resolves the yield with the attacker's values.

3. User's promise callback fires with the attacker-controlled (big_y, big_c).
   User derives their application private key from attacker-controlled material.
   Attacker recovers the key and drains associated funds.
