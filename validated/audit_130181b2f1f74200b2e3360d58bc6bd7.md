Audit Report

## Title
Permissionless `sendFunds()` + `updateRSETHPrice()` Enables Atomic Yield Extraction from Existing rsETH Holders — (`contracts/FeeReceiver.sol`, `contracts/LRTOracle.sol`)

## Summary
`FeeReceiver.sendFunds()` carries no access control, allowing any caller to flush accumulated MEV/EL rewards into `LRTDepositPool` at will. Because `LRTDepositPool.depositETH()` mints rsETH at the last checkpointed `rsETHPrice` (not a live ratio), and `LRTOracle.updateRSETHPrice()` is also permissionless, an attacker can atomically deposit at the stale pre-reward price, flush the rewards, then trigger a price update — capturing a proportional share of yield that should belong exclusively to existing rsETH holders.

## Finding Description
Three independently confirmed code facts combine to create the exploit:

**1. `FeeReceiver.sendFunds()` has no access control** (`contracts/FeeReceiver.sol`, L53–58):
```solidity
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```
Any EOA or contract can call this at any time.

**2. `LRTOracle.updateRSETHPrice()` is permissionless** (`contracts/LRTOracle.sol`, L87–89):
```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```
Only the `whenNotPaused` modifier applies; no role check.

**3. `depositETH()` mints at the stale stored price** (`contracts/LRTDepositPool.sol`, L519–520):
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```
`lrtOracle.rsETHPrice()` is the last written checkpoint, not a live TVL/supply ratio.

**Atomic attack sequence (single transaction):**
1. Call `depositETH{value: D}(0, "")` — mint `D·S/T` rsETH at stale price `P = T/S`.
2. Call `FeeReceiver.sendFunds()` — flush reward `R` into `LRTDepositPool`; `address(this).balance` in the pool increases by `R`, which is immediately counted by `getETHDistributionData()` → `_getTotalEthInProtocol()`.
3. Call `LRTOracle.updateRSETHPrice()` — `_updateRsETHPrice()` computes:
   - `previousTVL = (S + D·S/T) · P = T + D`
   - `rewardAmount = (T + D + R) − (T + D) = R`
   - `newRsETHPrice = (T + D + R − R·f) / (S·(T+D)/T)`

Attacker's rsETH ETH value after step 3:
```
(D·S/T) · newRsETHPrice = D + D·R·(1−f)/(T+D)
```
Net gain: `D·R·(1−f)/(T+D)` ETH — extracted from existing holders, whose collective gain is reduced from `R·(1−f)` to `R·(1−f)·T/(T+D)`.

**Why the `pricePercentageLimit` guard is insufficient** (`contracts/LRTOracle.sol`, L252–266):
The guard only fires when `pricePercentageLimit > 0` AND the price jump exceeds the configured threshold. It fails to block the attack when: (a) `pricePercentageLimit == 0` (guard disabled); (b) the reward batch `R` is small relative to TVL, keeping the price jump below the threshold; (c) the attacker splits the sequence across multiple smaller transactions to stay under the threshold each time.

## Impact Explanation
**High — Theft of unclaimed yield.** Existing rsETH holders lose a fraction `D/(T+D)` of every MEV/EL reward batch to any attacker executing this sequence. The protocol remains solvent (all rsETH is backed), but yield that should accrue to long-term holders is systematically siphoned. The attack is repeatable every time rewards accumulate in `FeeReceiver`.

## Likelihood Explanation
All three functions (`depositETH`, `sendFunds`, `updateRSETHPrice`) are callable by any unprivileged EOA or contract. The attack requires no front-running, no miner cooperation, and no privileged role — it is fully atomic within a single transaction. `FeeReceiver` accumulates rewards continuously from MEV/EL, so the attack surface is persistent and repeatable. The `pricePercentageLimit` guard provides only partial and bypassable protection.

## Recommendation
1. **Add access control to `sendFunds()`** — restrict it to `MANAGER` or `OPERATOR` role so it cannot be called atomically by an attacker.
2. **Alternatively, compute the mint ratio live** — in `getRsETHAmountToMint`, derive the rsETH price from a live `totalETHInProtocol / rsethSupply` ratio rather than the stale stored `rsETHPrice`, so a deposit immediately before a reward flush does not benefit from the flush.
3. **Consider a deposit cooldown or commit-reveal** to prevent atomic sandwich sequences.

## Proof of Concept
Deploy the following contract on a fork with a funded `FeeReceiver`:

```solidity
contract SandwichAttack {
    function attack(address pool, address feeReceiver, address oracle) external payable {
        uint256 priceBefore = IOracle(oracle).rsETHPrice();
        IDepositPool(pool).depositETH{value: msg.value}(0, "");
        IFeeReceiver(feeReceiver).sendFunds();
        IOracle(oracle).updateRSETHPrice();
        uint256 priceAfter = IOracle(oracle).rsETHPrice();
        // assert priceAfter > priceBefore
        // assert IRSETH(rseth).balanceOf(address(this)) * priceAfter / 1e18 > msg.value
    }
}
```

**Foundry fork test plan:**
1. Fork mainnet/testnet at a block where `FeeReceiver` holds accumulated ETH rewards.
2. Deploy `SandwichAttack` and call `attack{value: D}(...)`.
3. Assert `rseth.balanceOf(attacker) * oracle.rsETHPrice() / 1e18 > D`.
4. Compare against a control holder who held equivalent rsETH value before the attack — verify their per-ETH gain is strictly less than the attacker's per-ETH gain, with the difference equal to `D·R·(1−f)/(T+D)`.
