### Title
Hardcoded `DEFAULT_GAS_LIMIT` in Messenger Contracts Cannot Be Updated, Risking Temporary Freezing of Bridged ETH — (File: `contracts/bridges/BaseMessenger.sol`, `contracts/bridges/OptimismMessenger.sol`, `contracts/bridges/UnichainMessenger.sol`)

---

### Summary
`BaseMessenger`, `OptimismMessenger`, and `UnichainMessenger` each declare `DEFAULT_GAS_LIMIT = 200_000` as an immutable `public constant`. This value is passed as the `_minGasLimit` argument to the respective OP-stack `bridgeETHTo` calls. Because it is a compile-time constant, it can never be updated without redeploying the contract. If the L1 execution of the bridged message requires more gas than this fixed limit — due to a change in the L1 target contract, an EVM opcode repricing, or any other network upgrade — the L1 message will fail and the bridged ETH will be temporarily frozen in the bridge until a replay is possible.

---

### Finding Description

All three messenger contracts define the same immutable constant:

```solidity
uint32 public constant DEFAULT_GAS_LIMIT = 200_000;
```

and pass it unconditionally to the bridge:

```solidity
IBaseMessenger(l2bridge).bridgeETHTo{ value: value }(target, DEFAULT_GAS_LIMIT, bytes(""));
// (same pattern in OptimismMessenger and UnichainMessenger)
``` [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

On OP-stack chains, `_minGasLimit` is the gas forwarded to the L1 target when the cross-domain message is relayed. If the L1 target's execution consumes more than 200,000 gas, the relay transaction reverts and the ETH is held in the `OptimismPortal` / `CrossDomainMessenger` until a successful replay is submitted.

Contrast this with `L1VaultV2`, where the equivalent CCIP gas limit is a configurable storage variable (`ccipGasLimit`) that an admin can update via `setCcipGasLimit`:

```solidity
uint256 public ccipGasLimit;
...
function setCcipGasLimit(uint256 _ccipGasLimit) external onlyRole(TIMELOCK_ROLE) { ... }
``` [7](#0-6) [8](#0-7) 

No equivalent update mechanism exists in any of the three messenger contracts. The `sendETHToL1ViaBridge` function carries no access control, so any caller (including unprivileged users) can invoke it with their own ETH:

```solidity
function sendETHToL1ViaBridge(address l2bridge, address target, uint256 value) external payable nonReentrant {
    if (msg.value != value) revert MismatchedMsgValue();
    IBaseMessenger(l2bridge).bridgeETHTo{ value: value }(target, DEFAULT_GAS_LIMIT, bytes(""));
}
``` [9](#0-8) 

---

### Impact Explanation

If the L1 target (e.g., `L1Vault`) is upgraded to a version whose `receive()` or fallback logic consumes more than 200,000 gas, or if an EVM opcode repricing increases the cost of the L1 execution path, every bridge message sent through these messengers will fail on L1. The ETH is held in the OP-stack bridge contracts and cannot be released until a replay with sufficient gas is submitted. Because the constant cannot be changed in the deployed contract, the only remediation path is a full redeployment, which takes time and requires migration. This constitutes **temporary freezing of funds**.

---

### Likelihood Explanation

The OP-stack ecosystem has already undergone multiple gas schedule changes (e.g., EIP-2929, EIP-3529, EIP-1884 repricing). Future EIPs or L1 contract upgrades to `L1Vault` that add logic to `receive()` could push the required gas above 200,000. The three affected chains (Base, Optimism, Unichain) are all OP-stack derivatives and share this risk. The likelihood is **medium**: the current `L1Vault.receive()` is trivially cheap, but the immutability of the constant means any future change creates an unrecoverable failure path without redeployment. [10](#0-9) 

---

### Recommendation

Replace the compile-time `constant` with a mutable storage variable protected by an admin/timelock setter, mirroring the pattern already used for `ccipGasLimit` in `L1VaultV2`:

```solidity
// Before (immutable, unupgradeable)
uint32 public constant DEFAULT_GAS_LIMIT = 200_000;

// After (admin-configurable)
uint32 public gasLimit = 200_000;
function setGasLimit(uint32 _gasLimit) external onlyRole(ADMIN_ROLE) { gasLimit = _gasLimit; }
```

Apply this change to `BaseMessenger`, `OptimismMessenger`, and `UnichainMessenger`.

---

### Proof of Concept

1. Deploy `BaseMessenger` (or `OptimismMessenger` / `UnichainMessenger`) on Base/OP/Unichain.
2. Upgrade the L1 target contract (`L1Vault`) to a version whose `receive()` function performs additional storage writes, pushing its gas cost above 200,000.
3. Call `sendETHToL1ViaBridge(l2bridge, l1Vault, 1 ether)` with `msg.value = 1 ether`.
4. The OP-stack bridge relays the message on L1 with exactly 200,000 gas; the execution reverts due to out-of-gas.
5. The 1 ETH is now locked in the `CrossDomainMessenger` on L1 until a replay with a higher gas limit is submitted — but no on-chain mechanism exists to update `DEFAULT_GAS_LIMIT` in the deployed messenger, so the protocol cannot self-heal without redeployment. [11](#0-10) [12](#0-11)
