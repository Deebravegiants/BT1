### Title
Caller-Agnostic Queue Key in `verify_foreign_transaction` Enables Targeted Queue Saturation to Block Bridge Transaction Verification — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction` pending queue uses a map key that **excludes the caller's account ID**, unlike `sign()` and `request_app_private_key()` which both embed the caller identity via a derived tweak. Any unprivileged caller can saturate the 128-slot per-key queue for any specific foreign chain transaction at negligible cost (~128 yoctoNEAR + gas), permanently blocking legitimate users from queuing their verification requests and preventing bridge transactions from completing.

---

### Finding Description

**Structural asymmetry in key construction:**

`sign()` constructs its pending-map key as `SignatureRequest { tweak, payload, domain_id }` where `tweak = derive_tweak(predecessor_id, path)`. The caller's account ID is baked into the tweak, so every distinct caller produces a distinct key. The per-key cap of `MAX_PENDING_REQUEST_FAN_OUT = 128` is therefore effectively per-caller. [1](#0-0) 

`verify_foreign_transaction()` constructs its pending-map key as `VerifyForeignTransactionRequest { request: ForeignChainRpcRequest, domain_id, payload_version }`. **No caller account ID is present.** All callers submitting the same foreign-tx parameters share a single queue slot. [2](#0-1) 

This is confirmed by the test `verify_foreign_transaction__should_queue_duplicates_from_different_callers`, which explicitly documents that alice and bob queue under the **same** key. [3](#0-2) 

**The cap enforcement:**

`push_pending_yield` enforces `MAX_PENDING_REQUEST_FAN_OUT = 128` per key. Once the queue reaches 128 entries, any further submission panics with `PendingRequestQueueFull`. [4](#0-3) 

**The attack path:**

Because the key is caller-agnostic, an attacker who knows the target `(tx_id, domain_id, payload_version)` can submit 128 identical `verify_foreign_transaction` calls to fill the queue entirely. The legitimate user's subsequent call is rejected with `PendingRequestQueueFull`. After the 200-block yield-resume timeout expires and the attacker's 128 yields drain, the attacker immediately resubmits 128 new requests. The legitimate user is permanently blocked. [5](#0-4) 

The `verify_foreign_transaction` entry point performs no per-caller rate limiting — only a domain check, a gas attachment check, and a 1-yoctoNEAR deposit check. [6](#0-5) 

---

### Impact Explanation

In the Omnibridge inbound flow, a user sends funds to a foreign-chain bridge contract and then calls `verify_foreign_transaction` to obtain the MPC-signed attestation needed to claim the corresponding NEAR-side tokens. If the attacker saturates the queue for the user's specific `(tx_id, domain_id, payload_version)` key, the user can never queue their request. The user cannot change `tx_id` (it is the actual on-chain transaction), and changing `extractors` or `finality` alters the semantics of the verification. The user's foreign-chain funds are effectively frozen for as long as the attacker maintains the queue saturation.

This is **request-lifecycle manipulation that breaks production safety/accounting invariants** without relying on network-level DoS or operator misconfiguration — matching the Medium allowed impact tier.

---

### Likelihood Explanation

The attack cost is approximately 128 yoctoNEAR (≈ 0) plus gas for 128 transactions (a few cents on NEAR) per 200-block cycle. The attacker needs only to know the target `tx_id`, `domain_id`, and `payload_version` — all of which are observable on-chain once the victim submits their first request. The attack is trivially repeatable and requires no privileged access.

---

### Recommendation

1. **Add a per-caller slot within the fan-out queue**: Track how many of the 128 slots each caller occupies and cap per-caller occupancy (e.g., 1–2 slots). This preserves the fan-out behavior while preventing a single attacker from monopolizing the queue.
2. **Alternatively, add a global cap on distinct pending `verify_foreign_transaction` keys** combined with a per-caller admission limit to prevent both targeted saturation and unbounded map growth.
3. **Increase the deposit requirement** for `verify_foreign_transaction` to make queue saturation economically infeasible.

---

pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
