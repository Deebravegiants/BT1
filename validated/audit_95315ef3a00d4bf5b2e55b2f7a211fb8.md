### Title
Missing CKD Response Verification for `AppPublicKey` Variant Allows Single Malicious Participant to Corrupt Derived Key Output - (File: `crates/contract/src/lib.rs`)

### Summary

In `respond_ckd`, the contract performs a cryptographic pairing check (`ckd_output_check`) on the MPC response only for the `AppPublicKeyPV` variant. For the `AppPublicKey` (privately-verifiable) variant, the match arm is empty — no on-chain integrity check is performed. A single malicious attested participant (below threshold) can therefore call `respond_ckd` with an arbitrary fabricated `CKDResponse` for any pending `AppPublicKey` request, and the contract will accept it, drain the yield queue, and deliver the forged output to the user. The legitimate MPC-computed response can never be delivered afterward because the request is removed from the map on the first resolution.

### Finding Description

In `respond_ckd` (lib.rs lines 675–682), the response verification branch is:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically proves the response was computed using the actual MPC master secret. [2](#0-1) 

For `AppPublicKey`, no equivalent check exists. The contract proceeds directly to `resolve_yields_for`, which drains the entire fan-out queue and removes the request from `pending_ckd_requests`. [3](#0-2) 

The `AppPublicKey` variant is actively used in production (e.g., node integration tests use it as the default CKD request type). [4](#0-3) 

The structural analog to the external report is exact: just as `_completeRecipe()` updates `lastCompletionTime` even when no craft succeeds (missing the guard), `respond_ckd` resolves and removes the pending request even when the response has not been verified against the MPC key (missing the check for one variant).

### Impact Explanation

A single Byzantine attested participant (strictly below the signing threshold) can:

1. Observe any pending `AppPublicKey` CKD request in `pending_ckd_requests`.
2. Call `respond_ckd` with an arbitrary `CKDResponse{big_y, big_c}` — any two G1 points.
3. The contract accepts the call (caller is an attested participant, protocol is running), skips verification, and calls `resolve_yields_for`, which resumes all queued yields with the forged bytes and removes the map entry.
4. The user's TEE application receives a `(big_y, big_c)` pair that does not encode `msk · H(app_id)`. Decryption yields a random/attacker-chosen G1 point rather than the correct confidential key.
5. The legitimate MPC-computed response, when it eventually arrives via the honest `respond_ckd` call, finds `RequestNotFound` and is silently dropped.

This permanently corrupts the user's confidential key derivation output without any threshold of participants agreeing on the correct value — breaking the core MPC safety guarantee for the `AppPublicKey` flow.

**Allowed impact match**: Medium — request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants (the threshold guarantee on CKD output integrity) without requiring network-level DoS or operator misconfiguration.

### Likelihood Explanation

- The attacker only needs to be a single attested participant in the MPC network — well below the signing threshold.
- `AppPublicKey` requests are observable on-chain (stored in `pending_ckd_requests`).
- The call to `respond_ckd` with a fabricated response requires no special knowledge: any two valid compressed BLS12-381 G1 points suffice.
- The window is the entire duration a request is pending (up to the yield timeout).

Likelihood: **High** — any single compromised or malicious attested node can exploit this deterministically against any pending `AppPublicKey` CKD request.

### Recommendation

Add an on-chain integrity check for the `AppPublicKey` variant analogous to `ckd_output_check`. Since `AppPublicKey` provides only `pk1` (a G1 point, no G2 counterpart), the full publicly-verifiable pairing equation cannot be applied directly. Options:

1. **Require `AppPublicKeyPV` for all new requests** and deprecate `AppPublicKey`, so the existing `ckd_output_check` covers all cases.
2. **Add a weaker structural check** for `AppPublicKey` responses (e.g., verify that `big_y` and `big_c` are valid, non-identity G1 points in the prime-order subgroup) to at least prevent trivially malformed responses, and document that full integrity is user-side.
3. **Require threshold-many matching `respond_ckd` calls** before resolving the yield, mirroring how threshold signing works, so a single participant cannot unilaterally determine the output.

The most robust fix is option 3 or option 1.

### Proof of Concept

```
1. Alice (TEE app) submits request_app_private_key with AppPublicKey(pk1).
   → Contract stores CKDRequest in pending_ckd_requests.

2. Mallory (single malicious attested participant) observes the pending request.

3. Mallory calls respond_ckd(request, CKDResponse { big_y: G1::generator(), big_c: G1::generator() }).
   → Contract enters respond_ckd.
   → match arm for AppPublicKey: empty, no check.
   → resolve_yields_for drains the queue, resumes Alice's yield with forged bytes.
   → pending_ckd_requests entry removed.

4. Alice's yield callback fires with the forged (big_y, big_c).
   → Alice's TEE app decrypts: big_c - a*big_y = G1 - a*G1 = (1-a)*G1 ≠ msk·H(app_id).
   → Alice receives a garbage key, not her confidential derived key.

5. Honest MPC nodes eventually call respond_ckd with the correct response.
   → resolve_yields_for returns Err(RequestNotFound).
   → Correct response is silently discarded.
``` [5](#0-4) [6](#0-5)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
