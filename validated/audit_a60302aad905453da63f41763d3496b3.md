### Title
Stale DRep Expiry Check in Governance Ratification Allows Unauthorized Governance Actions to Be Enacted - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs`)

---

### Summary

The `dRepAcceptedRatio` function in the Conway `RATIFY` rule checks DRep expiry using the raw stored `drepExpiry` field from `DRepState`, without accounting for the `vsNumDormantEpochs` counter. When dormant epochs accumulate (no governance proposals are submitted for multiple epochs), a DRep's stored expiry can fall below `reCurrentEpoch` even though the DRep is actually still active. This causes active DReps to be silently excluded from the governance vote count, distorting the yes/no ratio and potentially allowing governance actions to be ratified with insufficient legitimate support.

---

### Finding Description

In `dRepAcceptedRatio`, the expiry check is:

```haskell
Just drepState
  | reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, so we don't consider it
``` [1](#0-0) 

The `drepExpiry` stored in `DRepState` is computed at the time of the last DRep update as:

```
storedExpiry = currentEpoch + drepActivity - numDormantEpochs
``` [2](#0-1) 

The **actual** expiry at any later point in time is `storedExpiry + currentNumDormantEpochs`. This is confirmed by `vsActualDRepExpiry` and the test helper `isDRepExpired`:

```haskell
binOpEpochNo (+) (vState ^. vsNumDormantEpochsL) (drep' ^. drepExpiryL) < currentEpoch
``` [3](#0-2) 

However, `RatifyEnv` does **not** include `numDormantEpochs`:

```haskell
data RatifyEnv era = RatifyEnv
  { reInstantStake :: InstantStake era
  , reStakePoolDistr :: PoolDistr
  , reDRepDistr :: Map DRep (CompactForm Coin)
  , reDRepState :: Map (Credential DRepRole) DRepState
  , reCurrentEpoch :: EpochNo
  , reCommitteeState :: CommitteeState era
  , reAccounts :: Accounts era
  , reStakePools :: Map (KeyHash StakePool) StakePoolState
  }
``` [4](#0-3) 

The stored `drepExpiry` is only updated when a DRep registers, updates, votes, or when a transaction containing a new proposal is processed via `updateDormantDRepExpiries`. If none of these events occur across multiple epochs, the stored expiry becomes stale relative to the actual expiry. [5](#0-4) 

The `EPOCH` rule increments `numDormantEpochs` at each epoch boundary when there are no active proposals, but does **not** update the stored DRep expiry: [6](#0-5) 

---

### Impact Explanation

When dormant epochs accumulate and a long-lived governance proposal is pending, DReps whose stored expiry has fallen below `reCurrentEpoch` (but whose actual expiry has not) are silently excluded from the vote count. Specifically:

- A DRep that voted **No** is excluded from the denominator (`tot`), increasing the yes/total ratio.
- A DRep that **did not vote** (counted as No in the denominator) is also excluded, again inflating the ratio.

This can cause a governance proposal — a `ParameterChange`, `TreasuryWithdrawals`, `HardForkInitiation`, `NewConstitution`, or `UpdateCommittee` — to appear to meet the DRep voting threshold when it does not, resulting in an **unauthorized governance action being enacted**.

This matches the allowed impact: *"Unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action is enacted."*

---

### Likelihood Explanation

The conditions required are:

1. A governance proposal is submitted and not immediately ratified (realistic for contested proposals).
2. No new proposals are submitted for several subsequent epochs, causing `numDormantEpochs` to accumulate.
3. The pending proposal's `govActionLifetime` is long enough to survive into the window where the bug manifests.

All three conditions are realistic in normal protocol operation. The `govActionLifetime` parameter is a governance-controlled value and can be set to many epochs. Periods without new proposals are expected. An attacker who submits a proposal and then refrains from submitting further proposals can deliberately engineer this state.

---

### Recommendation

Include `numDormantEpochs` in `RatifyEnv` and use it in the expiry check inside `dRepAcceptedRatio`:

```haskell
| binOpEpochNo (+) reNumDormantEpochs (drepExpiry drepState) < reCurrentEpoch
    -> (yes, tot) -- drep is actually expired
```

Alternatively, pre-compute the actual expiry when constructing the `reDRepState` snapshot for the `DRepPulser`, using `vsActualDRepExpiry` to normalize stored expiry values before snapshotting.

---

### Proof of Concept

**Setup** (`drepActivity = 2`, `govActionLifetime = 4`):

| Epoch | Event | `storedExpiry` | `numDormantEpochs` | `actualExpiry` |
|-------|-------|---------------|-------------------|----------------|
| 1 | DRep registers | 3 (= 1+2−0) | 0 | 3 |
| 2 | Attacker submits proposal; DRep votes No | 3 | 0 | 3 |
| 3 | No proposals | 3 | 1 | 4 |
| 4 | No proposals | 3 | 2 | 5 |
| 5 | No proposals | 3 | 3 | 6 |

**At epoch 5 boundary**, the `DRepPulser` is initialized with `reDRepState` snapshot (`storedExpiry = 3`) and `reCurrentEpoch = 4` (epoch − 1).

**Ratification check**: `4 > 3` → **DRep treated as expired and excluded**.

**Actual state**: `actualExpiry = 3 + 3 = 6 > 4` → **DRep is still active**.

The proposal's `govActionLifetime = 4` means it expires after epoch `2 + 4 = 6`, so it is still live at epoch 5. The DRep's No vote is excluded from the denominator. If the remaining active DReps have a yes ratio that now meets the threshold, the proposal is ratified and enacted — despite lacking sufficient legitimate DRep support.
