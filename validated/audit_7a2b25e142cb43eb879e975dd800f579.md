### Title
`accept_requests = false` Blocks In-Flight `respond()` / `respond_ckd()` / `respond_verify_foreign_tx()` Completions — (`File: crates/contract/src/lib.rs`)

---

### Summary

The `accept_requests` flag, set to `false` by `verify_tee()` when fewer than threshold participants hold valid TEE attestations, is checked not only in the new-request intake path (`check_request_preconditions`) but also inside every node-response method (`respond`, `respond_ckd`, `respond_verify_foreign_tx`). This means that any already-queued, already-computed signature or CKD response cannot be delivered to the contract once `accept_requests` flips to `false`, causing all in-flight yield-resume requests to time out and fail — even though the MPC nodes have already done the cryptographic work and hold a valid answer.

---

### Finding Description

The `MpcContract` stores a boolean field `accept_requests` (line 162) that is set to `false` inside `verify_tee()` (line 1737) when the surviving attested-participant count drops below the governance threshold. [1](#0-0) 

The intent is to stop **new** requests from entering the system. This check is correctly placed in `check_request_preconditions` (line 300), which gates `sign()`, `request_app_private_key()`, and `verify_foreign_transaction()`. [2](#0-1) 

However, the identical guard is also placed inside all three **node-response** methods:

- `respond()` — lines 579–581
- `respond_ckd()` — lines 662–664
- `respond_verify_foreign_tx()` — lines 711–713 [3](#0-2) [4](#0-3) [5](#0-4) 

These methods are the **completion path** for requests that were already accepted and are already being processed by the MPC nodes. Blocking them does not undo the already-accepted work; it only prevents the result from being delivered, causing every pending yield to expire via `fail_on_timeout`. [6](#0-5) 

The `verify_tee()` function is callable by any single participant (`voter_or_panic()` — one vote suffices to trigger the flag flip): [7](#0-6) 

---

### Impact Explanation

Once `accept_requests` is `false`:

1. All pending `sign` / `request_app_private_key` / `verify_foreign_transaction` yield-resume slots are orphaned — the MPC nodes' `respond*` calls are rejected with `TeeValidationFailed`.
2. Each yield eventually fires its timeout callback (`return_signature_and_clean_state_on_success` / `return_ck_and_clean_state_on_success` / `return_verify_foreign_tx_and_clean_state_on_success`), which calls `fail_on_timeout()` and panics the original caller's transaction. [8](#0-7) 

The user's 1 yoctoNEAR deposit is returned, but any cross-chain operation that was waiting on the MPC signature (e.g., a bridge unlock, a foreign-chain transaction that has already been broadcast) fails permanently. This breaks the request-lifecycle and execution-flow invariants of the MPC network: a request that was legitimately accepted and fully computed is silently discarded rather than delivered.

This matches the **Medium** allowed impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."*

---

### Likelihood Explanation

- TEE attestations expire on a 7-day cadence by design. In any production deployment, there will be windows where one or more attestations have lapsed before renewal.
- `verify_tee()` requires only a single participant call — no threshold coordination needed.
- A single Byzantine participant (strictly below the signing threshold) can call `verify_tee()` at the moment attestations are stale, flip `accept_requests = false`, and cause all in-flight requests to time out.
- The node retry logic (`observe_tx_result` in `crates/node/src/indexer/tx_sender.rs`) will keep retrying the `respond` transaction, but every attempt will fail with `TeeValidationFailed` until `accept_requests` is restored, which requires a subsequent `verify_tee()` call returning `true` — something the Byzantine participant can delay by timing their call. [9](#0-8) 

---

### Recommendation

Remove the `accept_requests` guard from `respond()`, `respond_ckd()`, and `respond_verify_foreign_tx()`. The flag's purpose is to prevent **new** requests from entering the queue; it has no legitimate role in blocking the **delivery** of responses to requests that were already accepted. The three response methods already enforce that the caller is an attested participant via `assert_caller_is_attested_participant_and_protocol_active()`, which is the correct security boundary for those paths. [10](#0-9) 

---

### Proof of Concept

1. Contract is `Running`, `accept_requests = true`. Three participants hold valid attestations; threshold = 2.
2. User calls `sign(request)` — accepted, yield-resume slot created, deposit locked.
3. MPC nodes compute the threshold signature; leader prepares a `respond(request, signature)` transaction.
4. Participant A's attestation expires (7-day TTL elapsed).
5. Byzantine participant B (below signing threshold) calls `verify_tee()`. `TeeValidationResult::Partial` fires; surviving count = 2 = threshold, so `accept_requests` is set to `true` and resharing is triggered — **but** if the surviving count is exactly at the threshold boundary and the threshold-relation check fails (e.g., `GovernanceThreshold > remaining`), `accept_requests` is set to `false` (line 1737) and no resharing occurs.
6. Alternatively, if all three attestations expire simultaneously, `TeeValidationResult::Partial` with 0 valid participants sets `accept_requests = false` unconditionally.
7. Leader's `respond(request, signature)` call hits line 579: `if !self.accept_requests { return Err(TeeValidationFailed) }` — rejected.
8. After NEAR's yield timeout (~200 blocks), `return_signature_and_clean_state_on_success` fires with `Err(PromiseError::Failed)`, calls `fail_on_timeout()`, and the user's cross-chain operation fails permanently. [11](#0-10) [12](#0-11)

async fn observe_tx_result(

    match request {
        Respond(respond_args) => {
            // Confirm whether the respond call succeeded by checking whether the
            // pending signature request still exists in the contract state.
            // A successful respond removes the request from contract state.
            let pending_request_response = indexer_state
                .view_client

                .await?;

            let transaction_status = match pending_request_response {
                Some(_) => TransactionStatus::NotExecuted,
                None => TransactionStatus::Executed,
            };

            Ok(transaction_status)
        }
