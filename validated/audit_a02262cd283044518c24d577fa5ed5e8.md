### Title
Malicious DelegateAction Sender Can Steal Relayer's Attached Deposit via Deliberate Inner-Receipt Failure - (File: runtime/runtime/src/actions.rs)

### Summary
When a meta transaction (`DelegateAction`) is submitted by a relayer, the relayer pays all attached deposits for inner actions. If an inner action receipt fails on the receiver's shard, the deposit refund is sent to the `predecessor_id` of the inner receipt — which is the DelegateAction's `sender_id` (Alice), not the relayer who funded it. A malicious sender can deliberately craft a `DelegateAction` with a large deposit that will fail on the receiver's shard, causing the relayer to lose the deposit while the sender gains it. This is a direct analog to the original report's race condition: the malicious party (sender) manipulates state between the initiation of a two-party arrangement and its completion, harming the counterparty (relayer) who cannot recover.

### Finding Description
In `apply_delegate_action` (`runtime/runtime/src/actions.rs:456–469`), when processing a `DelegateAction`, a new inner receipt is created with `predecessor_id = sender_id` (Alice):

```rust
let new_receipt = Receipt::V0(ReceiptV0 {
    predecessor_id: sender_id.clone(),   // ← Alice, not the relayer
    receiver_id: delegate_action.receiver_id().clone(),
    ...
    actions: delegate_action.get_actions(),
});
```

The relayer pays for all deposits in the inner actions (documented: "Relayer prepaid all fees and all things required by actions: attached deposits and attached gas"). [1](#0-0) 

When the inner receipt fails on the receiver's shard, `refund_unspent_gas_and_deposits` (`runtime/runtime/src/lib.rs:1269–1273`) sends the deposit refund to `receipt.balance_refund_receiver()`: [2](#0-1) 

```rust
if deposit_refund > Balance::ZERO {
    result.new_receipts.push(Receipt::new_balance_refund(
        receipt.balance_refund_receiver(),
        deposit_refund,
    ));
}
```

`balance_refund_receiver()` returns `predecessor_id` (Alice), not the relayer: [3](#0-2) 

```rust
pub fn balance_refund_receiver(&self) -> &AccountId {
    self.refund_to().as_ref().unwrap_or_else(|| self.predecessor_id())
}
```

The protocol itself acknowledges this broken invariant: [4](#0-3) 

> "But what if the transaction fails execution on Bob's shard? At this point, the predecessor is `Alice` and therefore she receives the token balance refunded, not the relayer. This is something relayer implementations must be aware of since there is a financial incentive for Alice to submit meta transactions that have high balances attached but will fail on Bob's shard."

The comment in `apply_delegate_action` itself confirms the relayer bears the deposit risk: [5](#0-4) 

### Impact Explanation
The corrupted protocol value is **Alice's NEAR account balance** (increased by the deposit refund) and **the relayer's NEAR account balance** (decreased by the deposit paid but never recovered). For a `DelegateAction` with a 100 NEAR deposit, Alice gains 100 NEAR and the relayer loses 100 NEAR. This is a direct, concrete balance state corruption enforced by the runtime's deposit-refund routing logic.

### Likelihood Explanation
Medium. The attack requires Alice to ensure the inner action fails on the receiver's shard. She can achieve this by:
1. Crafting an inner `FunctionCall` that calls a non-existent method or panics under specific state conditions.
2. Front-running the relayer's submitted transaction with her own transaction that changes state on the receiver's shard (e.g., removes a storage key the function call depends on, or deploys a new contract version that panics).

The relayer cannot prevent this even with perfect pre-submission checks, because Alice can change state after the relayer's check but before the meta transaction executes. NEAR's mempool is public, making front-running feasible. The attack is especially reliable if Alice controls the receiver contract.

### Recommendation
- Route deposit refunds for failed inner receipts in meta transactions to the relayer (`action_receipt.signer_id()`) rather than the inner receipt's `predecessor_id` (Alice). Gas refunds already correctly go to the relayer via `action_receipt.signer_id()` at `lib.rs:1278–1282`; deposit refunds should follow the same logic.
- Alternatively, introduce a `refund_to` override on the inner receipt generated by `apply_delegate_action` that points to the relayer's account, using the existing `ReceiptEnum::ActionV2` `refund_to` field.

### Proof of Concept
1. Alice signs a `DelegateAction` with a 100 NEAR deposit attached to a `FunctionCall` targeting `bob.near::some_method`.
2. Alice gives the signed `DelegateAction` to the relayer. The relayer checks: Alice's key is valid, nonce is correct, `bob.near` exists, deposit is within bounds — all checks pass.
3. Alice submits a separate transaction to `bob.near` that changes contract state so `some_method` will panic (e.g., removes a required storage key, or redeploys the contract).
4. The relayer wraps the `DelegateAction` in a transaction and submits it. The relayer's transaction is processed; the inner receipt is created with `predecessor_id = Alice`, funded with 100 NEAR from the relayer.
5. The inner receipt executes on `bob.near`'s shard. `some_method` panics. The receipt fails.
6. `refund_unspent_gas_and_deposits` fires: `deposit_refund = 100 NEAR`, `receipt.balance_refund_receiver() = Alice`.
7. A `new_balance_refund` receipt transfers 100 NEAR to Alice.
8. Net result: relayer's balance −100 NEAR, Alice's balance +100 NEAR. The relayer also burned gas with no compensation.
