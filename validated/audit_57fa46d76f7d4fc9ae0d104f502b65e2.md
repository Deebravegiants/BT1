Audit Report

## Title
Stale `rsETHPrice` Enables Yield Theft via Deposit Before Public `updateRSETHPrice()` Call - (File: contracts/LRTOracle.sol)

## Summary

`LRTOracle.updateRSETHPrice()` is a public, permissionless function that updates the stored `rsETHPrice` state variable. Because `LRTDepositPool.getRsETHAmountToMint()` uses the stale cached `rsETHPrice` as the denominator while reading live Chainlink prices as the numerator, an attacker can deposit at the stale (lower) price to receive over-minted rsETH, then immediately call `updateRSETHPrice()` to realize the accumulated yield — extracting yield that belongs to existing rsETH holders.

## Finding Description

`LRTOracle.updateRSETHPrice()` is callable by any unprivileged address: [1](#0-0) 

The stored `rsETHPrice` is only updated when this function is called. Between calls, the underlying LST assets (stETH, ETHx, etc.) continue to accrue staking yield, causing their live Chainlink prices to drift upward while `rsETHPrice` remains stale.

When a user deposits, `getRsETHAmountToMint()` computes the mint amount as: [2](#0-1) 

`lrtOracle.getAssetPrice(asset)` reads the **live** Chainlink price, but `lrtOracle.rsETHPrice()` returns the **stale** cached value. If `rsETHPrice` is lower than the true current rate, the denominator is too small and the depositor receives more rsETH than fair value.

The `pricePercentageLimit` guard in `_updateRsETHPrice()` only blocks a public caller when the price increase exceeds the configured threshold: [3](#0-2) 

For normal daily yield accumulation (typically 0.01–0.1% per day), the price increase is well within any reasonable `pricePercentageLimit` (e.g., 1%), so the guard does not block the attacker's call to `updateRSETHPrice()`. When `pricePercentageLimit == 0`, the guard is entirely disabled.

**Exploit flow:**
1. Attacker observes on-chain that live Chainlink asset prices have risen since the last `rsETHPrice` update (fully verifiable on-chain, no off-chain data needed).
2. Attacker calls `LRTDepositPool.depositETH()` or `depositAsset()`. The mint calculation uses the stale (lower) `rsETHPrice`, so the attacker receives more rsETH than fair value.
3. Attacker calls `LRTOracle.updateRSETHPrice()`. The price is updated to reflect the true TVL. The attacker's over-minted rsETH is now worth more ETH per token.
4. Attacker redeems or sells rsETH at the updated price, extracting the yield that accrued since the last update.

The `_beforeDeposit` path confirms no price freshness check exists before minting: [4](#0-3) 

## Impact Explanation

**High — Theft of unclaimed yield.**

Each execution of this pattern extracts a portion of the yield that accrued since the last price update. The profit per attack is bounded by the yield accumulated in the interval (typically daily staking rewards), but the attack is repeatable every update cycle. Existing rsETH holders are diluted: the price appreciation they were entitled to is partially captured by the attacker's over-minted shares. This directly matches the allowed impact: *High — Theft of unclaimed yield*.

## Likelihood Explanation

**Medium.** The attacker requires no special role, no leaked key, and no oracle compromise. The only precondition is that live asset prices have risen since the last `rsETHPrice` update — a condition that is always true after any non-trivial staking reward accrual period and is fully verifiable on-chain. The attack is self-contained, repeatable, and requires only standard public contract calls (`depositETH`/`depositAsset` + `updateRSETHPrice`).

## Recommendation

1. **Atomic price update on deposit:** Call `_updateRsETHPrice()` at the start of `depositETH` and `depositAsset` (before computing `rsethAmountToMint`) so the mint price always reflects the current TVL.
2. **Compute mint amount from live TVL directly:** Replace the stale `rsETHPrice()` denominator in `getRsETHAmountToMint()` with an on-the-fly computation using `_getTotalEthInProtocol()` and `rsETH.totalSupply()`, bypassing the cached value entirely.
3. **Staleness window:** Enforce a maximum staleness bound on `rsETHPrice` (e.g., revert deposits if the price has not been updated within the last N seconds).

## Proof of Concept

**Setup:** `rsETHPrice = 1.01e18` (last updated 24 hours ago). Staking rewards have accrued; true rsETH price is `1.012e18`.

**Steps:**

1. Attacker calls `LRTDepositPool.depositETH{value: 100 ether}(0, "")`.
   - `getAssetPrice(ETH_TOKEN)` → `1e18` (live).
   - `rsETHPrice()` → `1.01e18` (stale).
   - `rsethAmountToMint = 100e18 * 1e18 / 1.01e18 ≈ 99.0099e18` rsETH minted.
   - Fair amount at true price: `100e18 / 1.012e18 ≈ 98.8142e18` rsETH.
   - Over-mint: `≈ 0.1957e18` rsETH.

2. Attacker calls `LRTOracle.updateRSETHPrice()`.
   - Price increase ≈ 0.2%, within `pricePercentageLimit` (e.g., 1%), so call succeeds.
   - `rsETHPrice` updated to reflect true TVL (slightly diluted by attacker's over-mint, e.g., `≈ 1.0118e18`).

3. Attacker redeems `≈ 99.0099e18 rsETH` at `≈ 1.0118e18` ETH/rsETH → `≈ 100.18 ETH`, profiting `≈ 0.18 ETH` at the expense of existing holders.

**Foundry fork test plan:** Fork mainnet, snapshot `rsETHPrice` and live Chainlink prices, advance time by 1 day (to simulate yield accrual without a price update), execute the deposit + `updateRSETHPrice()` sequence, assert that the attacker's rsETH redemption value exceeds their deposit amount and that existing holders' pro-rata ETH value decreased.
