### Title
`directDeposits` Amounts Excluded from Preservation-of-Value Check Enables Unbounded ADA Creation â€” (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The Dijkstra era introduces a `directDeposits` transaction body field that unconditionally adds ADA to registered account balances via `applyDirectDeposits`. However, these amounts are never included in the `produced` side of the preservation-of-value check (`consumed == produced`). Any unprivileged transaction sender can therefore specify arbitrary `directDeposits` amounts, causing ADA to be minted from nothing into account balances, which can then be withdrawn.

---

### Finding Description

**Vulnerability type:** Funds/accounting bug â€” invalid state transition (direct ADA creation).

The Dijkstra era adds `directDeposits` (CBOR map key 25, type `DirectDeposits = Map AccountAddress Coin`) to both top-level and sub-transaction bodies. [1](#0-0) 

`directDeposits` is the symmetric counterpart to `withdrawals`: withdrawals drain account balances into the UTxO (and are correctly included in `consumed`), while `directDeposits` is supposed to fund account balances from the UTxO (and should therefore appear in `produced`).

**Step 1 â€” `directDeposits` is absent from `produced`.**

`dijkstraProducedValue` delegates entirely to `conwayProducedValue` plus sub-transaction produced values. Neither function references `directDeposits`. [2](#0-1) 

`getProducedDijkstraValue` wraps the same logic for both tx levels: [3](#0-2) 

**Step 2 â€” `directDeposits` is absent from `consumed`.**

`getConsumedDijkstraValue` calls `getConsumedMaryValue` (UTxO inputs + withdrawals + refunds + minted tokens) for both the top-level body and each sub-transaction body. No `directDeposits` term is added. [4](#0-3) 

**Step 3 â€” The preservation-of-value check therefore ignores `directDeposits`.**

`dijkstraUtxoTransition` calls `validateValueNotConservedUTxO`, which enforces `consumed == produced`. Because `directDeposits` appears in neither side, the check passes regardless of the amounts specified. [5](#0-4) 

**Step 4 â€” `applyDirectDeposits` unconditionally inflates account balances.**

After the UTXO rule succeeds, the ENTITIES rule applies `applyDirectDeposits` without any amount validation: [6](#0-5) 

The same applies in SUBENTITIES: [7](#0-6) 

The only validations performed on `directDeposits` are a network-ID check and an account-existence check â€” neither constrains the amounts: [8](#0-7) [9](#0-8) 

**Step 5 â€” Account balances are part of total ADA in circulation.**

The Shelley preservation-of-value proof explicitly lists reward accounts (account balances) as one of the six components of total lovelace in any chain state. [10](#0-9) 

Adding to account balances without a corresponding deduction from the UTxO therefore violates the global ADA conservation invariant.

---

### Impact Explanation

An attacker submits a Dijkstra-era transaction whose UTxO inputs and outputs balance normally (so `consumed == produced` holds), but whose `directDeposits` field specifies an arbitrarily large amount to a registered account they control. The ENTITIES rule adds that amount to the account balance. The attacker then withdraws it in a subsequent transaction. ADA has been created from nothing.

**Impact: Critical** â€” Direct creation of ADA through an invalid ledger state transition, matching the allowed impact "Direct loss, creation, or destruction of ADA or native assets through an invalid ledger state transition."

---

### Likelihood Explanation

No privileged access, governance majority, leaked key, or Sybil attack is required. Any transaction sender who controls a registered stake credential can exploit this in a single Dijkstra-era transaction. The only precondition is that the target account is registered, which the attacker can satisfy themselves.

**Likelihood: High.**

---

### Recommendation

Include the sum of all `directDeposits` amounts in the `produced` side of the preservation-of-value calculation, symmetrically with how `withdrawals` is included in `consumed`. Concretely, update `dijkstraProducedValue` and `dijkstraSubTxProducedValue` in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs` to add `inject (fold (unDirectDeposits (txBody ^. directDepositsTxBodyL)))` to the produced value, analogous to how `withdrawals` contributes to `consumed` via `getConsumedMaryValue`.

---

### Proof of Concept

1. Register a stake credential, creating an account with balance 0.
2. Construct a Dijkstra-era transaction `tx` such that:
   - `inputs` and `outputs` balance normally (e.g., spend 10 ADA, return 9 ADA + 1 ADA fee).
   - `directDeposits = { attacker_account â†’ 1_000_000_000_000 }` (1 million ADA).
3. Submit `tx`. `validateValueNotConservedUTxO` passes because `directDeposits` is absent from both `consumed` and `produced`. The ENTITIES rule calls `applyDirectDeposits`, adding 1,000,000 ADA to the attacker's account balance.
4. Submit a withdrawal transaction draining the account balance into a UTxO output.
5. The attacker has extracted 1,000,000 ADA that was never present in any UTxO input, violating ADA conservation.

  forall era l.
  ( DijkstraEraTxBody era
  , EraUTxO era
  , Value era ~ MaryValue
  , STxLevel l era ~ STxBothLevels l era
  ) =>
  PParams era ->
  (Credential Staking -> Maybe Coin) ->
  (Credential DRepRole -> Maybe Coin) ->
  UTxO era ->
  TxBody l era ->
  Value era
getConsumedDijkstraValue pp lookupStakingDeposit lookupDRepDeposit utxo txBody =
  withBothTxLevels
    txBody
    ( \topTxBody ->
        txBodyConsumedValue topTxBody <> subTransactionsConsumedValue topTxBody
    )
    txBodyConsumedValue

      foldMap'
