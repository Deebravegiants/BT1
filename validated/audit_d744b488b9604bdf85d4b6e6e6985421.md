### Title
Single-Vote Abort Allows Any Participant to Indefinitely Stall DKG or Resharing - (File: crates/contract/src/state/key_event.rs)

### Summary
`vote_abort_key_event_instance` requires only **one** attested participant's vote to immediately destroy the active key-event instance (`self.instance = None`). Any single Byzantine participant, strictly below the signing threshold, can call this function in a tight loop after every leader restart, permanently preventing DKG or resharing from completing. This is the structural analog of the PaymentRegistry bug where any single channel counterparty could reset `withdrawalLockedUntil` to zero, forcing the withdrawer to restart the lockdown period indefinitely.

---

### Finding Description

`KeyEvent::vote_abort` in `key_event.rs` unconditionally nullifies the active instance on the first valid vote:

```rust
pub fn vote_abort(&mut self, key_event_id: KeyEventId) -> Result<(), Error> {
    let candidate = self.verify_vote(&key_event_id)?;
    if self.instance.as_ref().unwrap().completed.contains(&candidate) {
        return Err(VoteError::VoteAlreadySubmitted.into());
    }
    self.instance = None;   // ← single vote destroys the entire attempt
    Ok(())
}
``` [1](#0-0) 

The public entry point `vote_abort_key_event_instance` in `lib.rs` only checks that the caller is an attested participant; it imposes no threshold requirement:

```rust
pub fn vote_abort_key_event_instance(&mut self, key_event_id: KeyEventId) -> Result<(), Error> {
    self.assert_caller_is_attested_participant_and_protocol_active();
    self.protocol_state.vote_abort_key_event_instance(key_event_id)
}
``` [2](#0-1) 

This is routed to both `InitializingContractState::vote_abort` and `ResharingContractState::vote_abort`, both of which delegate directly to `KeyEvent::vote_abort`: [3](#0-2) [4](#0-3) 

After the instance is nullified, the leader must call `start_keygen_instance` or `start_reshare_instance` again to create a new attempt with a fresh `key_event_id`. The Byzantine participant simply calls `vote_abort_key_event_instance` again with the new ID, repeating indefinitely.

Contrast this with `vote_cancel_resharing`, which correctly requires **threshold** votes before reverting state:

```rust
let threshold_cancellation_votes_reached: bool =
    cancellation_votes_count >= previous_running_threshold.value();
``` [5](#0-4) 

`vote_abort` has no equivalent threshold gate.

---

### Impact Explanation

**Medium.** A single attested participant (strictly below the signing threshold) can manipulate the key-event execution flow to break the production safety invariant that DKG and resharing complete whenever at least `threshold` participants are honest.

- **During DKG (`Initializing` state):** The MPC network never generates keys, so no signing or CKD service is ever available.
- **During resharing:** The participant set can never be updated. A compromised node that should be rotated out cannot be removed; the network is permanently locked to the old participant set.

Neither outcome requires funds to be directly stolen; the impact is permanent freezing of the protocol's key-lifecycle machinery, which is a production safety/accounting invariant break.

---

### Likelihood Explanation

Any one of the N registered, TEE-attested participants who turns Byzantine (software bug, key compromise, or deliberate malice) can execute this attack. The attacker needs no special privilege beyond being a registered participant. The attack is cheap (one on-chain transaction per leader restart) and requires no coordination. With N participants, the probability that at least one is Byzantine grows with N.

---

### Recommendation

Require a **threshold** of abort votes before nullifying the instance, mirroring the design of `vote_cancel_resharing`:

```rust
pub fn vote_abort(&mut self, key_event_id: KeyEventId) -> Result<(), Error> {
    let candidate = self.verify_vote(&key_event_id)?;
    let instance = self.instance.as_mut().unwrap();
    if instance.abort_votes.contains(&candidate) {
        return Err(VoteError::VoteAlreadySubmitted.into());
    }
    instance.abort_votes.insert(candidate);
    if instance.abort_votes.len() >= self.parameters.threshold().value() as usize {
        self.instance = None;
    }
    Ok(())
}
```

Alternatively, remove `vote_abort_key_event_instance` entirely and rely solely on the existing block-height timeout (`expires_on`) to clear stalled instances, since the leader already retries after timeout. [6](#0-5) 

---

### Proof of Concept

1. The MPC network enters `Resharing` state (e.g., after `vote_new_parameters` reaches threshold).
2. The leader (lowest participant ID) calls `start_reshare_instance` with `key_event_id = {epoch=E, domain=D, attempt=0}`. A `KeyEventInstance` is created with `expires_on = block + timeout_blocks`.
3. Byzantine participant P calls `vote_abort_key_event_instance({epoch=E, domain=D, attempt=0})`. `verify_vote` passes (P is an attested participant, the ID matches). `self.instance = None` is executed immediately.
4. The leader observes no active instance and calls `start_reshare_instance` with `attempt=1`.
5. P calls `vote_abort_key_event_instance({epoch=E, domain=D, attempt=1})`. Instance is destroyed again.
6. Steps 4–5 repeat indefinitely. `next_attempt_id` increments each time but the instance is always destroyed before any honest participant can cast a `vote_reshared`. Resharing never completes. [7](#0-6) [1](#0-0)
