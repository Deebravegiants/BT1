Audit Report

## Title
Unbounded `supportedAssetList` Iteration in `updateRSETHPrice()` Enables Permanent DoS of Price Updates — (File: contracts/LRTOracle.sol)

## Summary
`LRTOracle.updateRSETHPrice()` is a public, permissionless function whose internal call chain iterates over every entry in `supportedAssetList` with no upper bound. For each asset it makes `nodeDelegatorQueue.length` external calls into EigenLayer's `getQueuedWithdrawals()`, each of which iterates over all queued withdrawals and their strategies. As the protocol adds LSTs, cumulative gas grows proportionally and can permanently exceed the block gas limit, making price updates impossible.

## Finding Description
`updateRSETHPrice()` is declared `public whenNotPaused` with no role restriction, so any address may call it.

It delegates to `_updateRsETHPrice()` → `_getTotalEthInProtocol()`, which fetches `lrtConfig.getSupportedAssetList()` and iterates over every element with no cap:

```solidity
// LRTOracle.sol L331-349
for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
    uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
    ...
}
```

`getTotalAssetDeposits()` calls `getAssetDistributionData()`, which loops over every NDC and calls `getAssetUnstaking(asset)` per NDC per asset:

```solidity
// LRTDepositPool.sol L446-456
for (uint256 i; i < ndcsCount;) {
    assetUnstakingFromEigenLayer += INodeDelegator(nodeDelegatorQueue[i]).getAssetUnstaking(asset);
    ...
}
```

`getAssetUnstaking()` calls `delegationManager.getQueuedWithdrawals(address(this))` — a cold external call — and then iterates over every queued withdrawal and every strategy within it:

```solidity
// NodeDelegator.sol L405-427
(IDelegationManager.Withdrawal[] memory queuedWithdrawals, ...) =
    _getDelegationManager().getQueuedWithdrawals(address(this));
for (uint256 withdrawalIndex = 0; ...) {
    for (uint256 strategyIndex = 0; ...) { ... }
}
```

The total gas cost is **O(assets × NDCs × withdrawals\_per\_NDC)**. `maxNodeDelegatorLimit` (default 10) bounds NDCs and `maxUncompletedWithdrawalCount` bounds withdrawals, but `supportedAssetList` in `LRTConfig` has **no cap**:

```solidity
// LRTConfig.sol L114-116
isSupportedAsset[asset] = true;
supportedAssetList.push(asset);   // no length check
```

Each new asset added via `addNewSupportedAsset()` multiplies the number of external EigenLayer calls by `nodeDelegatorQueue.length`, with no mechanism to prevent the cumulative gas from exceeding the block limit.

## Impact Explanation
Once the gas cost of `updateRSETHPrice()` exceeds the block gas limit, the function becomes permanently uncallable (even `updateRSETHPriceAsManager()` shares the same internal path). The stored `rsETHPrice` becomes stale. All deposits via `depositETH()`/`depositAsset()` and withdrawals via `initiateWithdrawal()`/`instantWithdrawal()` continue to use the stale price, creating direct accounting errors and insolvency risk. **Impact: Medium — Unbounded gas consumption leading to permanent DoS of price updates**, which is an explicitly listed allowed impact.

## Likelihood Explanation
Adding supported assets requires `TIME_LOCK_ROLE`, but this is a normal protocol operation — Kelp DAO's stated roadmap is to onboard additional LSTs over time. No adversarial action is needed; the protocol's own growth drives the system toward this condition. With 10 NDCs and 80 queued withdrawals per NDC, adding approximately 15–20 assets is sufficient to approach or exceed the 30 M gas block limit on Ethereum mainnet, given the cost of repeated cold external calls to EigenLayer per asset per NDC. **Likelihood: Medium.**

## Recommendation
Introduce a `maxSupportedAssetsCount` constant (analogous to `maxNodeDelegatorLimit`) enforced inside `_addNewSupportedAsset()` in `LRTConfig`. Alternatively, refactor `getAssetUnstaking()` so that `delegationManager.getQueuedWithdrawals()` is called once per NDC and the result is cached in memory for the duration of the price-update call, reducing the external-call count from O(assets × NDCs) to O(NDCs) and the inner iteration from O(assets × NDCs × withdrawals) to O(NDCs × withdrawals × strategies).

## Proof of Concept
Call chain triggered by any unprivileged address after the protocol has added N assets:

```
updateRSETHPrice()                          // public, no role check (LRTOracle.sol L87)
  └─ _updateRsETHPrice()
       └─ _getTotalEthInProtocol()          // LRTOracle.sol L331
            └─ for each asset in supportedAssetList (no cap, LRTConfig.sol L115):
                 └─ getTotalAssetDeposits(asset)
                      └─ getAssetDistributionData(asset)
                           └─ for each NDC in nodeDelegatorQueue (≤ maxNodeDelegatorLimit):
                                └─ getAssetUnstaking(asset)   // NodeDelegator.sol L405
                                     └─ delegationManager.getQueuedWithdrawals(ndc)  // cold external call
                                          └─ for each queued withdrawal (≤ maxUncompletedWithdrawalCount):
                                               └─ for each strategy in withdrawal.strategies
```

**Foundry fork test plan:**
1. Fork Ethereum mainnet.
2. Deploy or reuse the live LRT contracts.
3. Using `TIME_LOCK_ROLE`, call `addNewSupportedAsset()` 18 times to register 20 total assets (including the 2 initial ones).
4. For each of 10 NDCs, queue 80 withdrawals via `initiateUnstaking()`.
5. Call `updateRSETHPrice()` with `gasleft()` instrumentation and assert that gas consumed exceeds 30 000 000, or that the call reverts with out-of-gas.

With 20 assets, 10 NDCs, and 80 queued withdrawals per NDC:
- Cold external calls to EigenLayer: 20 × 10 = **200**
- Inner strategy iterations: 20 × 10 × 80 = **16 000**

At ~2 100 gas per cold external call and ~800 gas per SLOAD, this exceeds the 30 M block gas limit.
