Audit Report

## Title
Silent Zero-Mint on Small Deposits Permanently Locks User Funds - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool.depositETH()` and `depositAsset()` accept user funds and silently mint zero rsETH whenever the deposit amount is too small for the integer division in `getRsETHAmountToMint()` to produce a non-zero result. Because no explicit zero-shares guard exists and `minRSETHAmountExpected` defaults to `0`, the user's ETH or LST is permanently locked in the pool with no recovery path.

## Finding Description
`getRsETHAmountToMint()` computes:

```solidity
// LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

When `amount * assetPrice < rsETHPrice`, Solidity truncates the result to `0`.

`_beforeDeposit()` (L648–670) has two guards that both fail to catch this:

1. **Entry check (L657):** `depositAmount == 0 || depositAmount < minAmountToDeposit` — `minAmountToDeposit` is never set in `initialize()`, so it defaults to `0`. Any non-zero `depositAmount` passes.
2. **Slippage check (L667):** `rsethAmountToMint < minRSETHAmountExpected` — when the caller passes `minRSETHAmountExpected = 0` (the natural default), `0 < 0` is `false` and no revert occurs.

Execution then reaches `_mintRsETH(0)` (L686–690), which calls `RSETH.mint(msg.sender, 0)` (L229–240). The `checkDailyMintLimit(0)` modifier (RSETH.sol L42–56) does not revert on zero: `currentPeriodMintedAmount + 0 > maxMintAmountPerDay` is false under normal conditions, and `currentPeriodMintedAmount += 0` is a no-op. OpenZeppelin's `ERC20._mint(to, 0)` succeeds silently. For `depositETH`, the ETH is already held by the contract (via `msg.value`). For `depositAsset`, `safeTransferFrom` executes at L114 after `_beforeDeposit` returns. In both cases the user's assets are in the contract and 0 rsETH is minted. There is no user-callable withdrawal function for deposited funds.

## Impact Explanation
The depositor's funds are permanently transferred into `LRTDepositPool` with zero rsETH minted in return and no mechanism to reclaim them. This constitutes **Critical: Permanent freezing of funds** — the depositor's assets are irretrievably locked in the contract.

## Likelihood Explanation
Any unprivileged user calling `depositETH{value: smallAmount}(0, "")` or `depositAsset(asset, smallAmount, 0, "")` triggers this. No special role or precondition is required beyond the rsETH price exceeding the deposit value. As rsETH appreciates over time, the minimum deposit required to mint 1 wei of rsETH grows, widening the vulnerable range. Naive integrators, bots, and users who omit the slippage parameter are realistic victims.

## Recommendation
Add an explicit zero-shares guard in `_beforeDeposit()` immediately after computing `rsethAmountToMint`:

```solidity
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
if (rsethAmountToMint == 0) revert ZeroRsETHMinted();
```

Additionally, enforce a non-zero `minAmountToDeposit` in `initialize()` calibrated to always produce at least 1 wei of rsETH at the current price, and document that callers must supply a non-zero `minRSETHAmountExpected`.

## Proof of Concept
1. rsETH price appreciates to `1.5e18` via `LRTOracle.rsETHPrice()`.
2. User calls `LRTDepositPool.depositETH{value: 1}(0, "")`.
3. `_beforeDeposit(ETH_TOKEN, 1, 0)`: `1 == 0` → false; `1 < 0` → false. Passes.
4. `getRsETHAmountToMint(ETH_TOKEN, 1)` → `(1 * 1e18) / 1.5e18` → `0`.
5. `0 < 0` → false. No revert.
6. `_mintRsETH(0)` → `RSETH.mint(msg.sender, 0)` → `checkDailyMintLimit(0)` passes → `_mint(msg.sender, 0)` → no-op.
7. User's 1 wei ETH is permanently locked; user holds 0 rsETH.

**Foundry fuzz test plan:** Fuzz `depositAmount` over `[1, rsETHPrice/assetPrice - 1]` with `minRSETHAmountExpected = 0`; assert that `rsETH.balanceOf(depositor) > 0` after each call. The assertion will fail for all inputs in that range, confirming the bug.
