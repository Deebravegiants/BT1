### Title
`respond`, `respond_ckd`, and `respond_verify_foreign_tx` Always Enforce the New-Request `accept_requests` Guard When Resolving Already-Queued Pending Requests - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond()`, `respond_ckd()`, and `respond_verify_foreign_tx()` all gate on `self.accept_requests` — the same flag that blocks **new** user sign requests when TEE validation fails. However, these functions are resolving **already-queued** pending requests whose signatures were computed while the network was healthy. When `accept_requests = false` (set by `verify_tee()` because a TEE attestation expiry would break the threshold relation), nodes with still-valid TEE attestations cannot submit computed signatures, causing all in-flight pending requests to time out and permanently fail.

---

### Finding Description

`check_request_preconditions()` — called by `sign()`, `request_app_private_key()`, and `verify_foreign_transaction()` — correctly checks `accept_requests` to block **new** requests when the network is unhealthy: [1](#0-0) 

The same flag is then redundantly checked inside every **response** path: [2](#0-1) [3](#0-2) [4](#0-3) 

`verify_tee()` sets `accept_requests = false` — while leaving the contract in `Running` state — when kicking out expired-attestation participants would break the governance-threshold relation: [5](#0-4) 

In this state `is_running_or_resharing()` returns `true`, the surviving nodes still hold valid TEE attestations, and `assert_caller_is_attested_participant_and_protocol_active()` passes for them: [6](#0-5) 

Yet `respond()` is still blocked by `accept_requests = false`, even though the signature was computed before the attestation expired and is cryptographically valid.

The correct two-tier constraint model (mirroring the external report's initial-vs-maintenance margin distinction) is:

| Operation | Correct guard |
|---|---|
| **New** request (`sign`, `request_app_private_key`, `verify_foreign_transaction`) | `accept_requests` (stricter — "initial margin") |
| **Existing** request (`respond`, `respond_ckd`, `respond_verify_foreign_tx`) | `is_running_or_resharing()` only (less strict — "maintenance margin") |

---

### Impact Explanation

When `accept_requests = false` while the protocol is still `Running`:

1. All pending `sign` / CKD / foreign-tx requests that were already queued in `pending_signature_requests`, `pending_ckd_requests`, or `pending_verify_foreign_tx_requests` cannot be resolved.
2. The yield-resume promises time out, returning a hard failure to every waiting caller.
3. For bridge operations where the foreign-chain side is time-locked (e.g., a Bitcoin HTLC or a time-bounded Ethereum permit), the inability to deliver the signature within the window causes permanent, irrecoverable loss of the locked funds.

This breaks the production safety invariant that *a request accepted and computed by the MPC network must be resolvable*. It matches the allowed **Medium** impact: *"request-lifecycle … manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."*

---

### Likelihood Explanation

TEE attestations have a finite lifetime. In a production deployment with a high governance threshold (e.g., `threshold = n`, all participants required), a single attestation expiry triggers `accept_requests = false` while the contract stays `Running`. The window between a user's `sign()` call and the leader's `respond()` call is on the order of seconds to minutes — well within the attestation-expiry window. No attacker is required; the condition arises naturally from normal TEE certificate rotation.

---

### Recommendation

Remove the `accept_requests` guard from all three `respond*` functions. They should only verify `is_running_or_resharing()` and the attested-participant check, since they are completing requests that were already accepted under the stricter gate:

```rust
// In respond(), respond_ckd(), respond_verify_foreign_tx() — remove:
if !self.accept_requests {
    return Err(TeeError::TeeValidationFailed.into());
}
```

New requests remain protected by `check_request_preconditions()`. Existing in-flight requests can then be resolved regardless of the current `accept_requests` state, exactly as a trader above the maintenance margin should be able to reduce an open position regardless of the initial margin requirement.

---

### Proof of Concept

1. Deploy the contract with **5 participants (A–E)**, governance threshold **5** (100 %).
2. User calls `sign()` — `accept_requests = true`, request queued in `pending_signature_requests`.
3. All 5 nodes participate in the threshold signing protocol and produce a valid signature.
4. Participant E's TEE attestation expires (natural certificate rotation).
5. Any participant calls `verify_tee()`. Because kicking E out leaves 4 participants but the governance threshold is 5 (5 > 4), `validate_governance_against_reconstruction` fails → `accept_requests = false`, contract stays `Running`.
6. Node A (leader, attestation still valid) calls `respond(request, signature)`.
   - `assert_caller_is_attested_participant_and_protocol_active()` → **passes** (A is attested).
   - `is_running_or_resharing()` → **passes** (still `Running`).
   - `!self.accept_requests` → **reverts** with `TeeValidationFailed`.
7. The pending yield-resume promise times out; the user's cross-chain operation fails permanently. [7](#0-6) [8](#0-7) [9](#0-8)

    fn check_request_preconditions(
        &self,
        domain_id: DomainId,
        expected_purpose: DomainPurpose,
        minimum_gas: Gas,
        minimum_deposit: NearToken,
    ) -> (DomainConfig, AccountId) {
        // 1. Look up the domain and check its purpose.
        let domains = match self.protocol_state.domain_registry() {
            Ok(domains) => domains,
            Err(err) => env::panic_str(&err.to_string()),
        };
        let Some(domain_config) = domains.get_domain_by_domain_id(domain_id) else {
            env::panic_str(
                &InvalidParameters::DomainNotFound {
                    provided: domain_id,
                }
                .to_string(),
            );
        };
        if domain_config.purpose != expected_purpose {
            env::panic_str(
                &InvalidParameters::WrongDomainPurpose {
                    domain_id: domain_config.id,
                    expected: expected_purpose,
                    actual: domain_config.purpose,
                }
                .to_string(),
            );
        }
        let domain_config = domain_config.clone();

        // 2. Make sure the call will not run out of gas doing yield/resume logic.
        let prepaid_gas = env::prepaid_gas();
        if prepaid_gas < minimum_gas {
            env::panic_str(
                &InvalidParameters::InsufficientGas {
                    provided: prepaid_gas.as_gas(),
                    required: minimum_gas.as_gas(),
                }
                .to_string(),
            );
        }

        // 3. Require the minimum deposit and refund any excess.
        let predecessor = env::predecessor_account_id();
        require_deposit(minimum_deposit, &predecessor);

        // 4. Refuse the request if the contract is not currently accepting requests
        //    (e.g. because TEE validation has failed).
        if !self.accept_requests {
            env::panic_str(&TeeError::TeeValidationFailed.to_string())
        }

        (domain_config, predecessor)

    pub fn verify_tee(&mut self) -> Result<bool, Error> {
        log!("verify_tee: signer={}", env::signer_account_id());
        // Caller must be a participant (node or operator).
        self.voter_or_panic();
        let ProtocolContractState::Running(running_state) = &mut self.protocol_state else {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        };
        let current_params = running_state.parameters.clone();

        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);

        match self.tee_state.reverify_and_cleanup_participants(
            current_params.participants(),
            tee_upgrade_deadline_duration,
        ) {
            TeeValidationResult::Full => {
                self.accept_requests = true;
                log!("All participants have an accepted Tee status");
                Ok(true)
            }
            TeeValidationResult::Partial {
                participants_with_valid_attestation,
            } => {
                let remaining = participants_with_valid_attestation.len();
                // Defense in depth: the surviving participant set must keep the full
                // threshold relation intact — the GovernanceThreshold must still sit
                // within its bounds for the smaller set (in particular it must not
                // exceed the remaining participant count or the upper cap) and must
                // remain at least every domain's ReconstructionThreshold (the kickout
                // keeps the existing per-domain thresholds). Otherwise we refuse and
                // wait for manual intervention.
                let max_reconstruction_threshold =
                    max_reconstruction_threshold(running_state.domains.domains());
                if let Err(err) = ThresholdParameters::validate_governance_against_reconstruction(
                    u64::try_from(remaining).expect("participant count fits in u64"),
                    current_params.threshold(),
                    max_reconstruction_threshold,
                ) {
                    log!(
                        "Kicking out participants with an invalid TEE status would break the threshold relation ({:?}); {} participants remain with a valid TEE status. This requires manual intervention. We will not accept new signature requests as a safety precaution.",
                        err,
                        remaining,
                    );
                    self.accept_requests = false;
                    return Ok(false);
