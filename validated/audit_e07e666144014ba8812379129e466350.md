### Title
Missing Time-Based Staleness Check in `ChainlinkOracleForRSETHPoolCollateral` Enables Cross-Oracle Rate Divergence Theft — (`contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol`)

---

### Summary

`ChainlinkOracleForRSETHPoolCollateral.getRate()` performs only a round-completeness check (`answeredInRound < roundID`) but omits a wall-clock heartbeat check (`block.timestamp - updatedAt > heartbeat`). When the collateral token's Chainlink feed is stale while the rsETH oracle has updated to a lower current rate, the pool's minting formula over-issues wrsETH relative to the true ETH value of the deposited collateral, enabling an attacker to extract value from the pool.

---

### Finding Description

`getRate()` in `ChainlinkOracleForRSETHPoolCollateral` reads `latestRoundData()` and applies three guards:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L26-36
function getRate() public view returns (uint256) {
    (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();

    if (answeredInRound < roundID) revert StalePrice();   // round-completeness only
    if (timestamp == 0) revert IncompleteRound();
    if (ethPrice <= 0) revert InvalidPrice();
    ...
}
``` [1](#0-0) 

The `answeredInRound < roundID` check only catches an in-progress, incomplete round. It does **not** catch the case where the last completed round is simply old — i.e., the feed has not been updated within its heartbeat window (e.g., 24 h for many LST/ETH feeds). In that scenario `answeredInRound == roundID` passes, but `timestamp` can be arbitrarily far in the past.

This oracle is registered as the `supportedTokenOracle[token]` for collateral tokens (e.g., wstETH, stETH) in every pool variant that accepts non-ETH deposits. The minting formula in all affected pools is:

```solidity
// e.g. RSETHPoolV3.sol L331-334
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate(); // stale-high
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;          // inflated
``` [2](#0-1) 

The rsETH oracle (`InterimRSETHOracle` or `RSETHRateReceiver`) is updated independently and can reflect a current, lower rate while the collateral Chainlink feed remains stale at a higher historical price. [3](#0-2) [4](#0-3) 

The same pattern is present in every pool that supports token deposits:

- `RSETHPool.viewSwapRsETHAmountAndFee(amount, token)` [5](#0-4) 
- `RSETHPoolNoWrapper.viewSwapRsETHAmountAndFee(amount, token)` [6](#0-5) 
- `RSETHPoolV3ExternalBridge.viewSwapRsETHAmountAndFee(amount, token)` [7](#0-6) 
- `RSETHPoolV3WithNativeChainBridge.viewSwapRsETHAmountAndFee(amount, token)` [8](#0-7) 

Note also that the L1 `ChainlinkPriceOracle` used in `getAssetPrice` has the same omission (no staleness check), but that is a separate surface. [9](#0-8) 

---

### Impact Explanation

**For `RSETHPool`** (pre-minted wrsETH transferred from pool balance):

The pool holds a finite supply of pre-minted wrsETH. An attacker deposits stale-priced collateral and receives more wrsETH than the collateral's true ETH value warrants, directly draining the pool's wrsETH reserves. This is **direct theft of funds at-rest** in the pool. [10](#0-9) 

**For `RSETHPoolV3` and variants** (wrsETH minted on demand):

The pool mints new wrsETH backed by the deposited collateral. With a stale-high `tokenToETHRate`, the minted amount exceeds the true ETH value of the collateral. The pool becomes undercollateralized; the attacker can bridge the excess wrsETH to L1 and redeem it for underlying assets, or sell it on secondary markets. This constitutes **protocol insolvency** and/or **direct theft of other depositors' backing**. [11](#0-10) 

The daily mint limit in V3 pools partially caps per-day exposure but does not prevent the vulnerability. [12](#0-11) 

---

### Likelihood Explanation

Chainlink LST/ETH feeds (e.g., stETH/ETH) have heartbeats of up to 24 hours and only update on-chain when price deviation exceeds a threshold (e.g., 0.5%). During market stress — exactly when LST prices are most likely to drop — network congestion or low keeper incentives can cause feeds to lag behind reality for hours. This is a well-documented operational scenario, not an oracle operator compromise. An attacker needs only to monitor for feed staleness and act when the spread between the stale collateral price and the current rsETH rate is profitable. No privileged access is required.

---

### Recommendation

Add a configurable heartbeat parameter and enforce it in `getRate()`:

```solidity
uint256 public immutable heartbeat;

constructor(address _oracle, uint256 _heartbeat) {
    oracle = _oracle;
    heartbeat = _heartbeat;
}

function getRate() public view returns (uint256) {
    (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();

    if (answeredInRound < roundID) revert StalePrice();
    if (timestamp == 0) revert IncompleteRound();
    if (block.timestamp - timestamp > heartbeat) revert StalePrice(); // ADD THIS
    if (ethPrice <= 0) revert InvalidPrice();

    return uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
}
```

Set `heartbeat` to the Chainlink feed's documented heartbeat plus a small buffer (e.g., heartbeat + 1 hour).

### Proof of Concept

Differential fuzz test (local fork, unmodified contracts):

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import "forge-std/Test.sol";

contract MockChainlinkFeed {
    int256 public price;
    uint256 public updatedAt;
    uint8 public dec = 18;

    function setPrice(int256 _price, uint256 _updatedAt) external {
        price = _price;
        updatedAt = _updatedAt;
    }

    function decimals() external view returns (uint8) { return dec; }

    function latestRoundData() external view returns (
        uint80, int256, uint256, uint256, uint80
    ) {
        return (1, price, 0, updatedAt, 1); // answeredInRound == roundID → passes stale check
    }
}

contract MockRsETHOracle {
    uint256 public rate;
    function setRate(uint256 _rate) external { rate = _rate; }
    function getRate() external view returns (uint256) { return rate; }
}

contract StaleOraclePoC is Test {
    function test_crossOracleDivergenceTheft() public {
        MockChainlinkFeed collateralFeed = new MockChainlinkFeed();
        MockRsETHOracle rsETHOracle = new MockRsETHOracle();

        // T-24h: collateral token was worth 1.2 ETH, rsETH was 1.05 ETH
        // Now: collateral token is worth 1.0 ETH (dropped), rsETH is still 1.05 ETH
        // Chainlink feed is stale — still shows 1.2 ETH (24h ago)
        collateralFeed.setPrice(1.2e18, block.timestamp - 24 hours);
        rsETHOracle.setRate(1.05e18);

        // Deploy ChainlinkOracleForRSETHPoolCollateral (no heartbeat check)
        // getRate() returns 1.2e18 — stale, but passes all checks
        uint256 staleTokenRate = 1.2e18;
        uint256 currentRsETHRate = rsETHOracle.getRate();

        uint256 depositAmount = 1e18; // 1 token deposited
        uint256 rsETHMinted = depositAmount * staleTokenRate / currentRsETHRate;
        // rsETHMinted = 1e18 * 1.2e18 / 1.05e18 ≈ 1.142857e18

        uint256 trueTokenValueInETH = 1.0e18; // current market price
        uint256 fairRsETHAmount = trueTokenValueInETH * 1e18 / currentRsETHRate;
        // fairRsETHAmount = 1e18 * 1e18 / 1.05e18 ≈ 0.952380e18

        // Invariant violated: minted > fair by ~19%
        assertGt(rsETHMinted, fairRsETHAmount + 1 wei,
            "Stale oracle allows over-minting of rsETH");

        // Attacker profit in ETH terms (at current rsETH rate):
        uint256 excessRsETH = rsETHMinted - fairRsETHAmount;
        uint256 profitInETH = excessRsETH * currentRsETHRate / 1e18;
        // ~0.19 ETH profit per 1 token deposited
        assertGt(profitInETH, 0);
    }
}
```

The test passes on unmodified code because `answeredInRound (1) == roundID (1)` satisfies the only staleness guard, while `block.timestamp - updatedAt = 24 hours` is never checked. [13](#0-12)
