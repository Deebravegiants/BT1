### Title
Incorrect `gasLimit = 0` in `ScrollMessenger.sendMessage` Causes All Scroll-to-L1 ETH Bridge Operations to Fail - (File: contracts/bridges/ScrollMessenger.sol)

### Summary

`ScrollMessenger.sol` hardcodes `gasLimit = 0` when calling Scroll's native bridge `sendMessage`, while every other messenger in the codebase uses `DEFAULT_GAS_LIMIT = 200_000`. This incorrect parameter causes the cross-chain message to fail at L1 relay, permanently blocking ETH bridging from Scroll L2 to L1 and freezing all user deposits in the bridge or pool.

### Finding Description

The vulnerability class from the reference report is **incorrect parameters in cross-chain call construction** — wrong flags/metadata that cause the operation to revert. The exact analog exists in `ScrollMessenger.sol`.

`ScrollMessenger.sendETHToL1ViaBridge` calls Scroll's native bridge with `gasLimit = 0`:

```solidity
// contracts/bridges/ScrollMessenger.sol:23
IScrollMessenger(l2bridge).sendMessage{ value: value }(target, value, "", 0, msg.sender);
//                                                                         ^
//                                                                  gasLimit = 0
```

The comment in the code claims `"Gas limit is set to 0 to use the default gas limit"`, but the `IScrollMessenger` interface provides no such semantics — `gasLimit` is the explicit gas budget for L1 execution:

```solidity
// contracts/interfaces/L2/IScrollMessenger.sol:71-79
function sendMessage(
    address target,
    uint256 value,
    bytes calldata message,
    uint256 gasLimit,        // Gas limit required to complete the message relay on L1
    address refundAddress
) external payable;
```

Compare with every other messenger in the codebase, all of which use a non-zero gas limit:

| Messenger | Gas Limit Used |
|---|---|
| `BaseMessenger.sol` | `DEFAULT_GAS_LIMIT = 200_000` |
| `OptimismMessenger.sol` | `DEFAULT_GAS_LIMIT = 200_000` |
| `UnichainMessenger.sol` | `DEFAULT_GAS_LIMIT = 200_000` |
| `ScrollMessenger.sol` | **`0`** ← incorrect |

In Scroll's L2ScrollMessenger, `gasLimit` is used to (a) calculate the relay fee (`fee = gasLimit × l2BaseFee`) and (b) set the gas budget for L1 execution. With `gasLimit = 0`, the fee is 0 (so the L2 call succeeds and ETH is locked in the bridge), but the L1 relay executes with 0 gas — insufficient even for the trivial `receive() external payable {}` on `L1Vault`, which requires at minimum 2300 gas for a value-bearing `call`. The L1 relay fails, and the ETH is stuck in the Scroll bridge with no path to recovery since the `gasLimit` is fixed at send time.

### Impact Explanation

Every invocation of `bridgeAssetsViaNativeBridge()` on any pool contract configured with `ScrollMessenger` as its messenger will:

1. Successfully lock ETH in Scroll's L2 bridge (L2 tx succeeds, ETH leaves the pool).
2. Fail at L1 relay due to `gasLimit = 0` being insufficient for execution.
3. Leave the ETH permanently stuck in the Scroll L1 bridge with no on-chain recovery path, since the `gasLimit` cannot be changed for an already-submitted message.

This constitutes **permanent freezing of user funds** — all ETH deposited by users on Scroll L2 that the BRIDGER attempts to move to L1 is irrecoverably locked.

**Impact: Critical — Permanent freezing of funds.**

### Likelihood Explanation

The `ScrollMessenger` is a production contract in the repository. The `bridgeAssetsViaNativeBridge()` function is the standard operational path for moving accumulated ETH from the Scroll L2 pool to L1. Every time the BRIDGER role executes this routine operation, the bug is triggered. There is no conditional path that avoids it — the `gasLimit = 0` is hardcoded unconditionally.

### Recommendation

Apply the same `DEFAULT_GAS_LIMIT` pattern used by all other messengers:

```diff
// contracts/bridges/ScrollMessenger.sol
+   uint32 public constant DEFAULT_GAS_LIMIT = 200_000;

    function sendETHToL1ViaBridge(address l2bridge, address target, uint256 value) external payable nonReentrant {
        if (msg.value != value) revert MismatchedMsgValue();
-       IScrollMessenger(l2bridge).sendMessage{ value: value }(target, value, "", 0, msg.sender);
+       IScrollMessenger(l2bridge).sendMessage{ value: value }(target, value, "", DEFAULT_GAS_LIMIT, msg.sender);
    }
```

### Proof of Concept

1. Admin deploys `ScrollMessenger` and configures a Scroll L2 pool (e.g., `RSETHPoolV3ExternalBridge`) with `messenger = address(scrollMessenger)` and `l2Bridge = scrollL2Bridge`.
2. Users deposit ETH into the pool on Scroll L2. ETH accumulates.
3. BRIDGER calls `pool.bridgeAssetsViaNativeBridge()`.
4. Pool calls `IL2Messenger(messenger).sendETHToL1ViaBridge{value: ethBalanceMinusFees}(l2Bridge, l1VaultETHForL2Chain, ethBalanceMinusFees)`.
5. `ScrollMessenger.sendETHToL1ViaBridge` calls `IScrollMessenger(l2bridge).sendMessage{value: value}(target, value, "", 0, msg.sender)` — ETH is locked in Scroll's L2 bridge, L2 tx succeeds.
6. Scroll's L1 relayer picks up the message and calls `L1ScrollMessenger.relayMessageWithProof(...)` with `gasLimit = 0`.
7. The L1 execution of `L1Vault.receive()` fails (0 gas < 2300 gas minimum for a value-bearing call).
8. ETH is permanently stuck in the Scroll L1 bridge. All user deposits that passed through this path are frozen.

        UtilLib.checkNonZeroAddress(l2Bridge);
        UtilLib.checkNonZeroAddress(messenger);
        UtilLib.checkNonZeroAddress(l1VaultETHForL2Chain);

        if (amount == 0) revert InvalidAmount();

        // bridge up to the ETH balance minus fees
        uint256 ethBalanceMinusFees = getETHBalanceMinusFees();
        if (amount > ethBalanceMinusFees) revert InsufficientETHBalance();

        IL2Messenger(messenger).sendETHToL1ViaBridge{ value: amount }(l2Bridge, l1VaultETHForL2Chain, amount);

        emit BridgedETHToL1ViaNativeBridge(l1VaultETHForL2Chain, amount);
