Audit Report

## Title
User Deposits Exhaust Shared rsETH Daily Mint Cap, Freezing Protocol Fee Yield and Staling Oracle Price — (`contracts/RSETH.sol`, `contracts/LRTOracle.sol`)

## Summary

`RSETH.mint()` applies the `checkDailyMintLimit` modifier to every caller without exception, using a single shared counter (`currentPeriodMintedAmount` vs `maxMintAmountPerDay`). Both user-deposit minting via `LRTDepositPool._mintRsETH()` and protocol-fee minting via `LRTOracle._updateRsETHPrice()` call `RSETH.mint()`, drawing from the same bucket. When user deposits fill the cap, every subsequent `updateRSETHPrice()` call reverts before reaching `rsETHPrice = newRsETHPrice`, freezing unclaimed fee yield and staling the oracle for the remainder of the 24-hour period.

## Finding Description

**Shared counter in `RSETH.sol`:**

The `checkDailyMintLimit` modifier tracks all minting in one counter: [1](#0-0) 

`RSETH.mint()` applies this modifier unconditionally to every caller holding `MINTER_ROLE`: [2](#0-1) 

**User-deposit path** calls `RSETH.mint()` directly: [3](#0-2) 

**Fee-mint path** in `_updateRsETHPrice()` first passes LRTOracle's own fee cap (`_checkAndUpdateDailyFeeMintLimit`), then calls `IRSETH.mint()` — which hits the same shared `checkDailyMintLimit`: [4](#0-3) 

When `currentPeriodMintedAmount` has been filled by user deposits, the condition `currentPeriodMintedAmount + feeAmount > maxMintAmountPerDay` fires at line 50 of `RSETH.sol`, reverting the entire `_updateRsETHPrice()` call before `rsETHPrice = newRsETHPrice` at line 313 of `LRTOracle.sol` is ever reached. [5](#0-4) 

`updateRSETHPriceAsManager()` also delegates to `_updateRsETHPrice()` and is equally blocked: [6](#0-5) 

The two-limit design (`maxFeeMintAmountPerDay` in `LRTOracle` and `maxMintAmountPerDay` in `RSETH`) creates a false sense of separation: fee minting must pass **both** checks, but user deposits only consume the RSETH-level cap, making it possible to crowd out fee minting without touching the oracle-level cap at all.

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield (repeatable per period).**

- Protocol fee rsETH is not minted to the treasury for the blocked period.
- `rsETHPrice` is not updated, staling the oracle for all downstream consumers.
- Both effects repeat every 24-hour period as long as the cap is filled, making the freeze effectively permanent under sustained deposit pressure.
- No user principal is lost, placing this squarely in the "permanent freezing of unclaimed yield" category rather than a critical fund-loss impact.

## Likelihood Explanation

- `depositETH` is permissionless; no role is required.
- `maxMintAmountPerDay` must be a finite non-zero value for the protocol to function normally.
- A single large depositor or the natural accumulation of many depositors in a busy period can exhaust the cap without any malicious intent.
- The condition is observable on-chain: anyone can read `remainingDailyMintLimit()` and time a deposit to fill the cap at the start of a new period.
- Repeatable every period with no recovery path until the cap resets. [7](#0-6) [8](#0-7) 

## Recommendation

Introduce a dedicated `mintFee` entry point in `RSETH` that uses a separate daily counter (or is exempt from `checkDailyMintLimit`) and is restricted to a `FEE_MINTER_ROLE` held only by `LRTOracle`. This ensures user-deposit volume can never crowd out protocol fee minting. Alternatively, reserve a portion of `maxMintAmountPerDay` exclusively for fee minting by tracking user-deposit minting and fee minting in separate counters within `RSETH` itself.

## Proof of Concept

```solidity
// Foundry fork/local test (pseudocode)
function test_depositExhaustsCapBlocksFeeYield() public {
    // 1. Manager sets a finite daily mint cap
    vm.prank(manager);
    rsETH.setMaxMintAmountPerDay(1000 ether);

    // 2. Depositor fills the cap exactly
    uint256 depositAmt = _ethAmountThatMintsExactly(1000 ether);
    vm.deal(attacker, depositAmt);
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: depositAmt}(0, "");

    // Assert cap is full
    assertEq(rsETH.currentPeriodMintedAmount(), 1000 ether);

    // 3. Simulate staking rewards accruing (TVL increases)
    _simulateRewards(10 ether);

    // 4. Anyone calls updateRSETHPrice() — reverts because fee mint
    //    hits checkDailyMintLimit in RSETH.mint()
    vm.expectRevert(
        abi.encodeWithSelector(RSETH.DailyMintLimitExceeded.selector, ...)
    );
    lrtOracle.updateRSETHPrice();

    // 5. rsETHPrice is stale; treasury received 0 fee rsETH
    assertEq(rsETH.balanceOf(treasury), 0);
}
```

The test requires no admin compromise — only a deposit large enough (or aggregated across multiple depositors) to fill `maxMintAmountPerDay` within a single 24-hour period. [9](#0-8) [10](#0-9)

    modifier checkDailyMintLimit(uint256 amount) {
        // Check if we need to reset the period if it has been more than 24 hours
        if (block.timestamp >= periodStartTime + 1 days) {
            currentPeriodMintedAmount = 0;
            periodStartTime = getCurrentPeriodStartTime();
        }

        // Check if minting would exceed the daily limit
        if (currentPeriodMintedAmount + amount > maxMintAmountPerDay) {
            revert DailyMintLimitExceeded(currentPeriodMintedAmount + amount, maxMintAmountPerDay);
        }

        currentPeriodMintedAmount += amount;
        _;
    }
