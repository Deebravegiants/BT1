### Title
Unprivileged `TransferToGasKey` Griefing Permanently Bricks `DeleteKey` and `DeleteAccount` - (`File: runtime/runtime/src/access_keys.rs`)

### Summary

Any unprivileged account can send NEAR tokens directly into a victim's gas key balance via `TransferToGasKey`. Both `action_delete_key` and `action_delete_account` check the raw gas key balance against `GasKeyInfo::MAX_BALANCE_TO_BURN` (1 NEAR) and fail with `GasKeyBalanceTooHigh` if exceeded. Because the balance check uses the live trie value — which any third party can inflate — a griefer can permanently prevent the account owner from deleting their gas key or their account.

### Finding Description

`action_transfer_to_gas_key` in `runtime/runtime/src/access_keys.rs` executes on the receipt receiver (`account_id`) with no restriction on who the sender is:

```rust
pub(crate) fn action_transfer_to_gas_key(
    state_update: &mut TrieUpdate,
    result: &mut ActionResult,
    account_id: &AccountId,
    action: &TransferToGasKeyAction,
) -> Result<(), RuntimeError> {
    let Some(mut access_key) = get_access_key(state_update, account_id, &action.public_key)? ...
    gas_key_info.balance = gas_key_info.balance.checked_add(action.deposit)...
    set_access_key(state_update, account_id.clone(), action.public_key.clone(), &access_key);
``` [1](#0-0) 

Any account can submit a transaction with `receiver_id = victim_account` and `actions = [TransferToGasKey { public_key: victim_gas_key_pk, deposit: 1_NEAR + 1 }]`. The NEAR is deducted from the sender and credited to the victim's gas key balance.

`delete_gas_key` then blocks deletion if the balance exceeds `MAX_BALANCE_TO_BURN`:

```rust
if gas_key_info.balance > GasKeyInfo::MAX_BALANCE_TO_BURN {
    result.result = Err(ActionErrorKind::GasKeyBalanceTooHigh { ... }.into());
    return Ok(());
}
``` [2](#0-1) 

`action_delete_account` applies the same check over the sum of all gas key balances:

```rust
let gas_key_balance_to_burn = compute_gas_key_balance_sum(state_update, account_id)?;
if gas_key_balance_to_burn > GasKeyInfo::MAX_BALANCE_TO_BURN {
    result.result = Err(ActionErrorKind::GasKeyBalanceTooHigh { ... }.into());
    return Ok(());
}
``` [3](#0-2) 

The threshold is 1 NEAR:

```rust
pub const MAX_BALANCE_TO_BURN: Balance = Balance::from_near(1);
``` [4](#0-3) 

The account owner can call `WithdrawFromGasKeyAction` to drain the excess, but the griefer can immediately re-fund via another `TransferToGasKey` transaction, creating a persistent griefing loop. The victim must win a race against the griefer on every block.

### Impact Explanation

The corrupted protocol value is the `GasKeyInfo::balance` field stored in the account's trie access-key entry (a DB entry). The broken operations are:

- `DeleteKey` action result: permanently returns `GasKeyBalanceTooHigh` instead of succeeding.
- `DeleteAccount` action result: permanently returns `GasKeyBalanceTooHigh` instead of succeeding.

An account owner who has a gas key can be permanently prevented from deleting that key or their entire account. If the account is a contract, this also prevents contract self-destruction. The griefer's cost is only the gas fees for repeated `TransferToGasKey` transactions; the NEAR deposited is not lost (it accumulates in the gas key), so the griefer can even recycle the same NEAR by using a different account to re-fund after each withdrawal.

### Likelihood Explanation

The attack requires only:
1. Knowledge of the victim's account ID and gas key public key (both publicly observable on-chain via `view_access_keys`).
2. Enough NEAR to cover gas fees for repeated `TransferToGasKey` transactions (the deposited NEAR itself is not destroyed).

No special privileges are needed. The `TransferToGasKey` action is a standard protocol action available to any account. The attack is cheap, repeatable, and requires no coordination.

### Recommendation

The root cause is that `delete_gas_key` and `action_delete_account` check the raw trie balance, which any third party can inflate. Two mitigations:

1. **Restrict `TransferToGasKey` to the account owner**: Require that the receipt predecessor equals the receipt receiver (i.e., the action can only be self-directed). This matches the design intent — gas keys are funded by the account owner.

2. **Allow deletion regardless of balance, burning or returning the excess**: Instead of blocking deletion when `balance > MAX_BALANCE_TO_BURN`, burn the entire balance unconditionally (or return it to the beneficiary on `DeleteAccount`). The `MAX_BALANCE_TO_BURN` guard was introduced to limit token destruction, but it should not be a hard blocker that an external party can exploit.

### Proof of Concept

1. Alice creates a gas key on her account: `AddKey { public_key: GK, access_key: gas_key_full_access(3) }`.
2. Griefer Bob submits: `Transaction { signer: bob, receiver: alice, actions: [TransferToGasKey { public_key: GK, deposit: 1_001_000_000_000_000_000_000_000 }] }` (1.001 NEAR > `MAX_BALANCE_TO_BURN`).
3. Alice submits `DeleteKey { public_key: GK }` → runtime executes `delete_gas_key` → `gas_key_info.balance (1.001 NEAR) > MAX_BALANCE_TO_BURN (1 NEAR)` → `GasKeyBalanceTooHigh` error → key not deleted.
4. Alice submits `WithdrawFromGasKey { public_key: GK, amount: 0.002 NEAR }` to drain below threshold.
5. Bob immediately re-submits step 2 in the next block.
6. Alice's `DeleteKey` and `DeleteAccount` are permanently blocked as long as Bob repeats step 2.

The existing test `test_delete_account_gas_key_balance_too_high` in `runtime/runtime/src/access_keys.rs` already confirms that a gas key balance above 1 NEAR blocks account deletion — it just does not test the case where the balance was set by a third party. [5](#0-4) [6](#0-5)

pub(crate) fn action_transfer_to_gas_key(
    state_update: &mut TrieUpdate,
    result: &mut ActionResult,
    account_id: &AccountId,
    action: &TransferToGasKeyAction,
) -> Result<(), RuntimeError> {
    let Some(mut access_key) = get_access_key(state_update, account_id, &action.public_key)? else {
        result.result = Err(ActionErrorKind::GasKeyDoesNotExist {
            account_id: account_id.clone(),
            public_key: Box::new(action.public_key.clone()),
        }
        .into());
        return Ok(());
    };
    let Some(gas_key_info) = access_key.gas_key_info_mut() else {
        // Key exists but is not a gas key
        result.result = Err(ActionErrorKind::GasKeyDoesNotExist {
            account_id: account_id.clone(),
            public_key: Box::new(action.public_key.clone()),
        }
        .into());
        return Ok(());
    };

    gas_key_info.balance = gas_key_info.balance.checked_add(action.deposit).ok_or_else(|| {
        RuntimeError::StorageError(StorageError::StorageInconsistentState(
            "gas key balance integer overflow".to_string(),
        ))
    })?;
    set_access_key(state_update, account_id.clone(), action.public_key.clone(), &access_key);
    Ok(())
