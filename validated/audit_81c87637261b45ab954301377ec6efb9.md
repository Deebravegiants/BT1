### Title
Cross-Chain Metadata Signature Replay Allows Unauthorized `deployToken` on Any OmniBridge Chain - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/omni_bridge.cairo`, `aptos/sources/omni_bridge.move`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The `MetadataPayload` borsh encoding used to verify the NEAR MPC signature in every chain's `deployToken` / `deploy_token` entry point contains no chain identifier (no chain ID, no contract address, no program ID). The signed digest is therefore byte-identical across all OmniBridge deployments. An unprivileged attacker who observes a valid `deployToken` signature on any one chain can replay it verbatim on every other chain, deploying the wrapped token without authorization and permanently blocking the legitimate deployment on those chains.

---

### Finding Description

Every `deployToken` path hashes the same borsh layout:

```
[PayloadType::Metadata (0x01)] || [token_string] || [name_string] || [symbol_string] || [decimals]
```

```solidity

``` [2](#0-1)

- EVM `finTransfer` lines 294, 297: `bytes1(omniBridgeChainId)` appears twice in the encoded message. [6](#0-5)

The `MetadataPayload` encoding was never given the same treatment.

---

### Impact Explanation

**Unauthorized token deployment across all chains.** A single NEAR MPC-signed metadata signature is valid on every OmniBridge instance simultaneously. An attacker who extracts the signature from a public `deployToken` transaction on chain A can immediately call `deployToken` on chains B, C, D, … with the identical calldata. Each call succeeds because:

1. The signature verifies against `nearBridgeDerivedAddress` / `near_bridge_derived_address` (same MPC key everywhere).
2. The token has not yet been deployed on those chains, so the "already exists" guard does not fire.

After the replay, the token is registered in each chain's `nearToEthToken` / `near_to_starknet_token` / `near_to_aptos_token` mapping. When the NEAR bridge later attempts the legitimate deployment on those chains, it reverts with `ERR_TOKEN_EXIST` / `ERR_TOKEN_ALREADY_DEPLOYED`, permanently blocking the canonical deployment path.

This satisfies: **High — Acceptance of insufficiently-bound signatures that bypass execution gates**, and **High — Asset-identity / token-mapping divergence that breaks backing guarantees**.

---

### Likelihood Explanation

All OmniBridge `deployToken` transactions are public. The signature is in the calldata and trivially extractable. No privileged access, leaked key, or social engineering is required. The attacker only needs to monitor one chain and submit the same transaction on all others before the legitimate relayer does. The attack is cheap (gas only) and deterministic.

---

### Recommendation

Add the destination chain identifier to the `MetadataPayload` borsh encoding, mirroring the pattern already used in `finTransfer`:

- **EVM**: prepend `bytes1(omniBridgeChainId)` to `borshEncoded` in `deployToken`.
- **Starknet**: change `payload.to_borsh()` to `payload.to_borsh(self.omni_bridge_chain_id.read())` and update `MetadataPayloadImpl::to_borsh` to accept and embed the chain ID.
- **Aptos**: pass `state.chain_id` into `metadata_to_borsh` and embed it in the buffer.
- **Solana**: write `SOLANA_OMNI_BRIDGE_CHAIN_ID` into `serialize_for_near` before the metadata fields.

The NEAR MPC signing path must be updated in parallel to include the target chain ID when producing the metadata signature, so that a signature issued for chain A is cryptographically invalid on chain B.

---

### Proof of Concept

1. Observe a successful `deployToken(signatureData, metadata)` call on the Ethereum OmniBridge (e.g., deploying `token.near` with name "Token", symbol "TKN", decimals 18).
2. Extract `signatureData` and `metadata` from the calldata.
3. Call `deployToken(signatureData, metadata)` on the Arbitrum OmniBridge with the identical arguments.
4. The call succeeds: `ECDSA.recover(keccak256(borshEncoded), signatureData) == nearBridgeDerivedAddress` because the borsh encoding is chain-agnostic.
5. `nearToEthToken["token.near"]` on Arbitrum now points to a freshly deployed `BridgeToken` proxy.
6. Repeat for Base, BNB, Starknet, Solana, Aptos.
7. When the legitimate relayer later attempts to deploy `token.near` on Arbitrum, the call reverts with `ERR_TOKEN_EXIST`, permanently blocking the canonical deployment.

    fn to_borsh(self: @MetadataPayload) -> ByteArray {
        let mut borsh_bytes: ByteArray = Default::default();
        borsh_bytes.append_byte(PayloadType::Metadata.into());
        borsh_bytes.append(@borsh::encode_byte_array(self.token));
        borsh_bytes.append(@borsh::encode_byte_array(self.name));
        borsh_bytes.append(@borsh::encode_byte_array(self.symbol));
        borsh_bytes.append_byte(*self.decimals);
        borsh_bytes
    }
