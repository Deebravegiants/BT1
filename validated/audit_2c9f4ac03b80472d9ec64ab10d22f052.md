Audit Report

## Title
`addLiquidityWeighted` Reverts with Opaque `MinimalLiquidity` When Scaled Shares Fall Below Pool Floor — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary

`_scaleWeightsToShares` guards only against scaled shares rounding to zero, but not against the range `[1, MINIMAL_MINTABLE_LIQUIDITY − 1]`. When a caller's `maxAmountToken0`/`maxAmountToken1` caps produce a scale factor that maps weight shares into this range, the subsequent paying `addLiquidity` call reverts with `MinimalLiquidity` from `LiquidityLib`. The periphery emits no descriptive error and the weighted liquidity path is permanently unusable for that `(max0, max1)` combination.

## Finding Description

In `_scaleWeightsToShares` (lines 226–243 of `MetricOmmPoolLiquidityAdder.sol`):

```solidity
out.shares[i] = Math.mulDiv(w.shares[i], scaleWad, WAD);
if (w.shares[i] != 0 && out.shares[i] == 0) revert SharesRoundedToZero();
```

The only guard rejects the zero case. Any value in `[1, MINIMAL_MINTABLE_LIQUIDITY − 1]` passes silently. [1](#0-0) 

The scaled `LiquidityDelta` is forwarded to `_addLiquidity`, which calls `IMetricOmmPoolActions(pool).addLiquidity(...)`. Inside `LiquidityLib.addLiquidity`, the check at lines 76–79 enforces:

```solidity
uint256 newUserShares = userShares + sharesToAdd;
if (newUserShares < ctx.minimalMintableLiquidity) {
    revert IMetricOmmPoolActions.MinimalLiquidity(newUserShares, ctx.minimalMintableLiquidity);
}
``` [2](#0-1) 

`_addLiquidity` catches the revert and re-throws it raw via assembly, so the caller receives the pool-level `MinimalLiquidity` error with no periphery context: [3](#0-2) 

`MINIMAL_MINTABLE_LIQUIDITY` is an immutable set at pool construction time and is not accessible to `_scaleWeightsToShares` (which is `pure`): [4](#0-3) 

The full call path in `addLiquidityWeighted` is: probe → `_scaleWeightsToShares` → `_addLiquidity` → pool `addLiquidity` → `LiquidityLib` → `MinimalLiquidity` revert: [5](#0-4) 

## Impact Explanation

The `addLiquidityWeighted` function — the primary ergonomic entry point for LPs specifying token budgets — is broken for any `(max0, max1)` combination that produces scaled shares in `[1, MINIMAL_MINTABLE_LIQUIDITY − 1]`. No funds are lost (transient pay context is cleared in the catch block), but the core periphery liquidity flow is rendered unusable for this input range. The caller must abandon the weighted path and manually compute shares for `addLiquidityExactShares`. This meets the "unusable withdraw/swap/liquidity flows" criterion in the allowed impact gate at Medium severity.

## Likelihood Explanation

The condition is reachable by any unprivileged LP without special setup. It occurs whenever `max0` (or `max1`) is small relative to `need0` (or `need1`) but not so small that shares round to zero. With `MINIMAL_MINTABLE_LIQUIDITY = 1000` and weight shares of `5_000_000`, a cap producing `scaleWad ≈ 0.0002e18` yields `out.shares ≈ 1000`; a cap slightly below that yields `out.shares = 999`, which reverts. This is a normal operating range for users depositing small amounts into a pool with large existing liquidity. The condition is deterministic and repeatable for the same inputs.

## Recommendation

In `_scaleWeightsToShares`, after computing `out.shares[i]`, add a check that the result is either zero or at least `MINIMAL_MINTABLE_LIQUIDITY`. Since the function is `pure`, read `MINIMAL_MINTABLE_LIQUIDITY` from the pool via `IMetricOmmPool(pool).getImmutables()` at the call site in `addLiquidityWeighted` and pass it as a parameter. Revert with a descriptive error (e.g., `SharesBelowMinimalLiquidity`) before the paying `addLiquidity` call is attempted:

```solidity
if (w.shares[i] != 0 && out.shares[i] == 0) revert SharesRoundedToZero();
if (out.shares[i] != 0 && out.shares[i] < minimalMintableLiquidity) revert SharesBelowMinimalLiquidity(out.shares[i], minimalMintableLiquidity);
``` [6](#0-5) 

## Proof of Concept

1. Deploy a pool with `MINIMAL_MINTABLE_LIQUIDITY = 1000`.
2. Seed the pool so that a probe with weight `[binIdx=0, shares=5_000_000]` returns `need0 = 500_000`.
3. Call `addLiquidityWeighted` with `maxAmountToken0 = 99` → `scaleWad = mulDiv(99, 1e18, 500_000) ≈ 198e12`.
4. `_scaleWeightsToShares` computes `out.shares[0] = mulDiv(5_000_000, 198e12, 1e18) = 990`.
5. `990 != 0` → `SharesRoundedToZero` does not fire; scaled delta is forwarded.
6. `LiquidityLib.addLiquidity` evaluates `newUserShares = 0 + 990 = 990 < 1000` → reverts `MinimalLiquidity(990, 1000)`.
7. `_addLiquidity` catches and re-throws raw; caller receives opaque pool-level revert with no periphery context. [7](#0-6) [2](#0-1)

  function _scaleWeightsToShares(LiquidityDelta calldata w, uint256 max0, uint256 max1, uint256 need0, uint256 need1)
    internal
    pure
    returns (LiquidityDelta memory out)
  {
    uint256 scaleWad0 = need0 == 0 ? type(uint256).max : Math.mulDiv(max0, WAD, need0);
    uint256 scaleWad1 = need1 == 0 ? type(uint256).max : Math.mulDiv(max1, WAD, need1);
    uint256 scaleWad = scaleWad0 < scaleWad1 ? scaleWad0 : scaleWad1;

    uint256 n = w.binIdxs.length;
    out.binIdxs = new int256[](n);
    out.shares = new uint256[](n);
    for (uint256 i; i < n; i++) {
      out.binIdxs[i] = w.binIdxs[i];
      out.shares[i] = Math.mulDiv(w.shares[i], scaleWad, WAD);
      if (w.shares[i] != 0 && out.shares[i] == 0) revert SharesRoundedToZero();
    }
  }
```
