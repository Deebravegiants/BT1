### Title
No Slippage Control in L2 Pool `deposit()` Functions Allows Users to Receive Fewer rsETH Than Expected - (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPool.sol, contracts/pools/RSETHPoolNoWrapper.sol, contracts/pools/RSETHPoolV3WithNativeChainBridge.sol)

---

### Summary

All L2 pool `deposit()` functions across every pool variant accept only an `amount` and `referralId` — no `minRsETHAmountExpected` parameter exists. The rsETH amount minted is computed at execution time from a live oracle rate. If the oracle rate is updated between transaction submission and execution, users silently receive fewer wrsETH/rsETH tokens than they anticipated, with no on-chain protection.

---

### Finding Description

Every L2 pool `deposit()` function computes the rsETH output at execution time by reading the live oracle rate:

```solidity

    external payable nonReentrant whenNotPaused limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    ...
    wrsETH.mint(msg.sender, rsETHAmount);
}
``` [1](#0-0) 

```solidity

The rsETH amount is derived from `getRate()`, which reads directly from the on-chain oracle:

```solidity
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
``` [3](#0-2) 

```solidity
function getRate() public view returns (uint256) {

``` [4](#0-3) 

The same pattern is present in every pool variant:

- `RSETHPoolV3ExternalBridge.deposit()` (ETH and token overloads) [5](#0-4)

**Contrast with the L1 deposit pool**, which correctly enforces a minimum output:

```solidity

    revert MinimumAmountToReceiveNotMet();
}
``` [10](#0-9) 

The L1 deposit pool has the protection; the L2 pools do not.

---

### Impact Explanation

The rsETH/ETH oracle rate (`rsETHToETHrate`) is a monotonically increasing value — it grows as EigenLayer staking rewards accrue. The L2 oracle is a cross-chain rate pushed from L1 via `MultiChainRateProvider`. When the oracle is updated on-chain (a permissionless or operator-triggered push), the rate jumps to a higher value.

A user who calls `deposit()` on an L2 pool and whose transaction is included in a block after an oracle rate update will receive fewer wrsETH tokens than they observed when constructing the transaction. There is no on-chain guard to revert the transaction if the received amount falls below the user's expectation.

**Impact classification**: Low — the contract fails to deliver the promised return (the rsETH amount the user saw in `viewSwapRsETHAmountAndFee` before submitting), but the deposited ETH/LST is not stolen; it remains in the pool backing the minted rsETH at the new rate.

---

### Likelihood Explanation

- The rsETH oracle rate is updated periodically (every time rewards are processed on L1 and the rate is bridged to L2). This is a routine, recurring event.
- Any user whose transaction is pending in the mempool at the moment of an oracle update is affected.
- No attacker action is required — this is a natural race condition between user deposits and oracle updates.
- L2 chains (Arbitrum, Optimism, Base) have public mempools and predictable block times, making the timing window well-defined.

---

### Recommendation

Add a `minRsETHAmountExpected` parameter to every L2 pool `deposit()` function and revert if the computed output falls below it, mirroring the protection already present in `LRTDepositPool._beforeDeposit()`:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountExpected)
    external payable nonReentrant whenNotPaused limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    ...
    wrsETH.mint(msg.sender, rsETHAmount);
}
```

Apply the same pattern to all token `deposit()` overloads across `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPool`, and `RSETHPoolNoWrapper`.

---

### Proof of Concept

1. Alice calls `viewSwapRsETHAmountAndFee(1 ether)` on `RSETHPoolV3` and observes she will receive `X` wrsETH at the current oracle rate `R`.
2. Alice submits `deposit{value: 1 ether}("ref")`.
3. Before Alice's transaction is included, the L2 oracle is updated: the rate increases from `R` to `R'` (e.g., after a reward accrual event on L1 is bridged over).
4. Alice's transaction executes. `viewSwapRsETHAmountAndFee` now uses `R'`:
   - `rsETHAmount = (1 ether - fee) * 1e18 / R'`
   - Since `R' > R`, `rsETHAmount < X`.
5. Alice receives fewer wrsETH than she expected. There is no revert, no slippage check, and no recourse.

The magnitude of slippage equals the percentage increase in the oracle rate since Alice's transaction was submitted. For a rate update that reflects, say, one week of ~4% APY staking rewards (~0.077%), a 10 ETH deposit would yield approximately 0.0077 ETH worth of rsETH less than expected — a silent, unrecoverable loss for the user.
