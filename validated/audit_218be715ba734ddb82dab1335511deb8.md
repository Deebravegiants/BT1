### Title
Meta-Transaction Deposit Refund Misdirected to Sender Instead of Relayer on Inner Receipt Failure - (File: runtime/runtime/src/actions.rs)

### Summary

In NEAR's meta-transaction (NEP-366) implementation, when a relayer submits a `DelegateAction` containing an inner action with an attached deposit, and that inner action fails on the receiver's shard, the deposit refund is sent to the `DelegateAction` sender (Alice), not to the relayer who actually paid the deposit. An unprivileged user can deliberately craft a `SignedDelegateAction` with a large deposit attached to a function call that will fail, submit it to a relayer, and receive the full deposit refund â€” draining the relayer's balance without their consent.

### Finding Description

`apply_delegate_action()` in `runtime/runtime/src/actions.rs` constructs the inner receipt forwarded to the final receiver with `predecessor_id: sender_id.clone()` (Alice's account ID), not the relayer's account ID. [1](#0-0) 

When the inner receipt fails execution (e.g., calling a non-existent method on a contract), `refund_unspent_gas_and_deposits()` in `runtime/runtime/src/lib.rs` issues a deposit refund to `receipt.balance_refund_receiver()`. [2](#0-1) 

`balance_refund_receiver()` in `core/primitives/src/receipt.rs` returns `refund_to().as_ref().unwrap_or_else(|| self.predecessor_id())`. Because `apply_delegate_action` always creates a `Receipt::V0` (not `ActionReceiptV2`), `refund_to()` always returns `&None`, so the fallback is `predecessor_id()` â€” which is Alice, not the relayer. [3](#0-2) 

The relayer has no protocol-level mechanism to override this: the `refund_to` field only exists on `ActionReceiptV2`, but `apply_delegate_action` unconditionally emits a `Receipt::V0`. The codebase's own documentation explicitly acknowledges this as a financial incentive for Alice to exploit: [4](#0-3) 

### Impact Explanation

The corrupted protocol value is **Alice's account balance** (increased) and **the relayer's account balance** (decreased) by the full deposit amount attached to the inner action. This is a direct, on-chain token transfer from the relayer to Alice, recorded in the trie state root. The relayer's balance DB entry is permanently reduced; Alice's is permanently increased. No validator or node admin privilege is required â€” the attack executes entirely through normal transaction submission.

### Likelihood Explanation

Alice only needs to:
1. Sign a `DelegateAction` containing a `FunctionCall` with a large `deposit` targeting a method that will fail (e.g., a non-existent method, or a method that panics).
2. Submit the `SignedDelegateAction` to any relayer.

The relayer cannot distinguish a deliberately failing inner action from a legitimate one without simulating execution. The financial incentive is proportional to the deposit size Alice attaches. This is a realistic, repeatable attack requiring no special privileges.

### Recommendation

When `apply_delegate_action` constructs the inner receipt, set `predecessor_id` to the relayer's account ID (the outer receipt's `signer_id` / `predecessor_id`) rather than `sender_id`, so that deposit refunds on inner receipt failure flow back to the party who actually paid. Alternatively, emit an `ActionReceiptV2` with `refund_to` set to the relayer's account ID, so `balance_refund_receiver()` correctly routes the refund.

### Proof of Concept

1. Relayer `R` has 1000 NEAR. Alice `A` has 0 NEAR.
2. Alice signs a `DelegateAction`: `sender_id = A`, `receiver_id = some_contract`, inner action = `FunctionCall { method: "nonexistent", deposit: 500 NEAR, gas: 100 TGas }`.
3. Alice submits the `SignedDelegateAction` to relayer `R`.
4. `R` wraps it in a transaction (paying gas + the 500 NEAR deposit from its own balance) and submits to the network.
5. On Alice's shard, `apply_delegate_action` creates an inner `Receipt::V0` with `predecessor_id = A`, `receiver_id = some_contract`.
6. On `some_contract`'s shard, the function call fails (`AccountDoesNotExist` or method not found).
7. `refund_unspent_gas_and_deposits` calls `Receipt::new_balance_refund(receipt.balance_refund_receiver(), 500 NEAR)`. Since the inner receipt is `V0`, `balance_refund_receiver()` returns `predecessor_id()` = Alice.
8. Alice receives 500 NEAR. Relayer `R` is down 500 NEAR with no recourse. [5](#0-4) [6](#0-5) [7](#0-6)
