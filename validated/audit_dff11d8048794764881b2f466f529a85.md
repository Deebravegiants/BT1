### Title
Unconditional Deletion of Shared VRF Key Hash in `poolReapTransition` Breaks Uniqueness Invariant — (`File: eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/PoolReap.hs`)

### Summary

In `poolReapTransition`, VRF key hashes that become "dangling" due to pool re-registration are removed from `psVRFKeyHashes` using `Map.withoutKeys` — an unconditional deletion — rather than the reference-counted `removeVRFKeyHashOccurrences` used for retired pools. When a VRF key is shared by multiple pools (permitted before protocol version 11), this premature full removal allows a new pool to register with that VRF key, placing two active pools on-chain with the same VRF key and breaking the uniqueness invariant introduced at v11.

### Finding Description

`psVRFKeyHashes` in `PState` is a reference-counted map (`Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64)`) tracking how many registered pools use each VRF key. The v11 hard fork populates it via `populateVRFKeyHashes`, correctly counting all pre-existing shared VRF keys. [1](#0-0) 

At each epoch boundary, `poolReapTransition` removes two categories of VRF keys:

**Category 1 — retired pools** (`retiredVRFKeyHashes`, line 177): removed via `removeVRFKeyHashOccurrences`, which **decrements** the reference count and only deletes the entry when it reaches zero. [2](#0-1) [3](#0-2) 

**Category 2 — dangling VRF keys** (`danglingVRFKeyHashes`, lines 138–148): the old VRF key of a pool that re-registered with a new VRF key. These are removed via `Map.withoutKeys` — an **unconditional deletion** that ignores the reference count entirely. [4](#0-3) [2](#0-1) 

Before v11, the `POOL` rule had no duplicate-VRF check: [5](#0-4) 

So two pools could legitimately share VRF key `V1`. After the v11 hard fork, `populateVRFKeyHashes` correctly sets `psVRFKeyHashes[V1] = 2`: [6](#0-5) 

**Exploit sequence:**

1. Pre-v11: Pool A and Pool B both register with VRF key `V1`. `psVRFKeyHashes = {V1: 2}`.
2. v11 hard fork: `populateVRFKeyHashes` runs. `psVRFKeyHashes = {V1: 2}`.
3. Pool A re-registers with fresh VRF key `V2`. `psFutureStakePoolParams = {A: params_with_V2}`, `psVRFKeyHashes = {V1: 2, V2: 1}`.
4. Epoch boundary — `poolReapTransition`:
   - `danglingVRFKeyHashes = {V1}` (pool A's current VRF `V1` ≠ future VRF `V2`).
   - `Map.withoutKeys {V1}` **completely removes** `V1` from `psVRFKeyHashes`.
   - Result: `psVRFKeyHashes = {V2: 1}`. Pool B still uses `V1`, but `V1` is gone from the map.
5. Pool C submits `RegPool` with VRF key `V1`. The check `Map.notMember V1 psVRFKeyHashes` passes (V1 was deleted). Pool C is accepted.
6. Both Pool B and Pool C are now active with VRF key `V1`. [7](#0-6) 

### Impact Explanation

Two active stake pools share the same VRF key. In Ouroboros Praos, the VRF key determines slot leadership. If Pool C's operator possesses the VRF private key for `V1` (which they must, to have registered it), both Pool B and Pool C can produce valid VRF proofs for the same slots. This means both pools can produce valid blocks for the same slot, causing honest nodes to diverge on chain selection — a deterministic disagreement arising directly from an invalid ledger state transition accepted by the `POOL` and `POOLREAP` rules.

**Impact: High — Deterministic disagreement between honest nodes from ledger rule evaluation.**

### Likelihood Explanation

The precondition is that two pools shared a VRF key before protocol version 11. While pool operators typically generate unique VRF keys, the ledger imposed no uniqueness constraint before v11, so shared keys are a valid pre-existing ledger state. Any pool operator who knows a VRF key already in `psVRFKeyHashes` with count ≥ 2 can trigger the bug by re-registering one of those pools with a new VRF key, then registering a new pool with the freed key. The entry path is a standard, unprivileged `RegPool` certificate transaction.

**Likelihood: Low** (requires pre-v11 shared VRF keys to exist on-chain, which is an unusual but valid historical state).

### Recommendation

Replace the unconditional `Map.withoutKeys danglingVRFKeyHashes` with the same reference-counted removal used for retired pools. Additionally, `danglingVRFKeyHashes` should be kept as a list (not a `Set`) to correctly handle the case where multiple pools simultaneously become dangling for the same VRF key:

```haskell

### Proof of Concept

```
Pre-v11 state:
  psStakePools    = { A -> {vrf: V1, ...}, B -> {vrf: V1, ...} }
  psVRFKeyHashes  = {}   (not yet populated)

After v11 hardfork (populateVRFKeyHashes):
  psVRFKeyHashes  = { V1: 2 }

Tx: Pool A re-registers with V2 (RegPool cert, normal unprivileged tx):
  psFutureStakePoolParams = { A -> {vrf: V2, ...} }
  psVRFKeyHashes          = { V1: 2, V2: 1 }

Epoch boundary (POOLREAP):
  danglingVRFKeyHashes = { V1 }   (A's current vrf V1 ≠ future vrf V2)
  Map.withoutKeys { V1 } applied  → V1 completely removed
  psVRFKeyHashes = { V2: 1 }      ← BUG: Pool B still uses V1

Tx: Pool C registers with V1 (RegPool cert, normal unprivileged tx):
  Map.notMember V1 psVRFKeyHashes  → True  (V1 was deleted)
  Pool C accepted with vrf = V1

Final state:
  psStakePools   = { A -> {vrf: V2}, B -> {vrf: V1}, C -> {vrf: V1} }
  psVRFKeyHashes = { V1: 1, V2: 1 }   ← count is wrong; two pools use V1

Both B and C can produce valid VRF proofs for the same slots → fork risk.
```

populateVRFKeyHashes :: PState era -> PState era
populateVRFKeyHashes pState =
  pState
    & psVRFKeyHashesL
      %~ accumulateVRFKeyHashes (pState ^. psStakePoolsL) (^. spsVrfL)
        . accumulateVRFKeyHashes (pState ^. psFutureStakePoolParamsL) (^. sppVrfL)
  where
    accumulateVRFKeyHashes ::
      Map (KeyHash StakePool) a ->
      (a -> VRFVerKeyHash StakePoolVRF) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64)
    accumulateVRFKeyHashes spMap getVrf acc =
      Map.foldr' (addVRFKeyHashOccurrence . getVrf) acc spMap
    addVRFKeyHashOccurrence vrfKeyHash =
      Map.insertWith combine vrfKeyHash (knownNonZeroBounded @1)
      where
        -- Saturates at maxBound: if (+1) would overflow to 0, keep existing value
        combine _ oldVal = fromMaybe oldVal $ mapNonZero (+ 1) oldVal
