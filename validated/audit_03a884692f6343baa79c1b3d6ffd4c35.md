### Title
Unbounded `fee_proposal_fri` accepted during startup window corrupts L2 gas price for all subsequent blocks — (`File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

The `fee_proposal_fri` bounds check inside `is_proposal_init_valid` is gated on `fee_actual` being `Some`. `fee_actual` is `None` for the first `window_size` (10) blocks because `compute_fee_actual` returns `None` when `height < window_size`. During this startup window the bounds check is silently skipped, so any consensus proposer can publish an arbitrary `fee_proposal_fri` value. Those values are recorded in `fee_proposals_window`, become the median (`fee_actual`) once the window fills, and then drive the L2 gas price for every block that follows.

### Finding Description

`is_proposal_init_valid` in `validate_proposal.rs` enforces that `fee_proposal_fri` lies within a geometric margin of `fee_actual`:

```rust
// Validate fee_proposal is within the configured margin of fee_actual.
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    ...bounds check...
}
``` [1](#0-0) 

`fee_actual` is computed by `compute_fee_actual`, which returns `None` whenever `height < window_size` (currently 10):

```rust
let Some(start) = height.0.checked_sub(window_size) else {
    warn!("Cannot compute fee_actual for height {height}: height is below window_size ({window_size})");
    return None;
};
``` [2](#0-1) 

Both the proposer path and the validator path call `compute_fee_actual` and pass the result as `ProposalInitValidation.fee_actual`: [3](#0-2) 

Because the `if let` pattern fails when `fee_actual` is `None`, the entire bounds check is bypassed. A proposer can set `fee_proposal_fri` to `u128::MAX` (or `0`) for each of the first 10 blocks, and every validator will accept the proposal without complaint.

Each accepted `fee_proposal_fri` is stored in `fee_proposals_window` via `record_fee_proposal`: [4](#0-3) 

Once the window fills, `fee_actual` becomes the median of those poisoned values. `fee_actual` then feeds `calculate_next_l2_gas_price`, which sets `self.l2_gas_price` — the L2 gas price used in every subsequent `ProposalInit.l2_gas_price_fri` and validated by all peers: [5](#0-4) 

Additionally, `fee_proposal_fri` is hashed into the `ProposalCommitment` for V0_14_3+ blocks: [6](#0-5) 

So the corrupted value is permanently committed to the chain.

### Impact Explanation

A malicious proposer who controls any of the first 10 block proposals can set `fee_proposal_fri` to an extreme value (e.g., `u128::MAX`). After 10 blocks the median of the poisoned window becomes `fee_actual`, which then drives `l2_gas_price` for all future blocks. This permanently sets the L2 gas price to an extreme value, making all user transactions either free or prohibitively expensive. The corrupted value is also forwarded to the centralized cende pipeline via `prepare_blob_for_next_height`. This matches the Critical impact category: "Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."

### Likelihood Explanation

The window is only 10 blocks, but it occurs at every fresh chain start or after a revert that crosses the window boundary. Any validator who is scheduled to propose during those 10 blocks can trigger this. The code comment explicitly acknowledges the open window ("During initiation (fee_actual is None, <window_size blocks), bounds are not enforced"), confirming the design is aware of but does not guard against the attack.

### Recommendation

Apply a fallback bound during the startup window instead of skipping the check entirely. When `fee_actual` is `None`, bound `fee_proposal_fri` against `l2_gas_price` (the current EIP-1559 base fee) using the same `fee_proposal_margin_ppt`. This mirrors the proposer's own fallback (`compute_proposer_fee_proposal` already freezes at `l2_gas_price` when `fee_actual` is `None`) and closes the window without breaking the startup flow:

```rust
let reference = proposal_init_validation.fee_actual
    .unwrap_or(proposal_init_validation.l2_gas_price_fri); // fallback to current base fee
if let Some(fee_proposal) = init_proposed.fee_proposal_fri {
    let (lower, upper) = fee_proposal_bounds(reference, margin_ppt);
    if fee_proposal.0 < lower || fee_proposal.0 > upper {
        return Err(...);
    }
}
```

### Proof of Concept

1. Chain starts at height 0; `window_size = 10`.
2. A malicious validator is the proposer for block 0.
3. It constructs a `ProposalInit` with `fee_proposal_fri = Some(GasPrice(u128::MAX))`.
4. `is_proposal_init_valid` is called; `proposal_init_validation.fee_actual` is `None` (height 0 < 10).
5. The `if let (Some(fee_actual), Some(fee_proposal))` pattern does **not** match; the bounds check is skipped.
6. The proposal is accepted; `record_fee_proposal(BlockNumber(0), Some(GasPrice(u128::MAX)))` is called.
7. Steps 2–6 repeat for blocks 1–9 (the attacker need not control all 10; controlling the majority of the window suffices to skew the median).
8. At block 10, `compute_fee_actual` returns `Some(GasPrice(u128::MAX))` (or a very large value).
9. `calculate_next_l2_gas_price` sets `self.l2_gas_price ≈ u128::MAX`.
10. All subsequent `ProposalInit.l2_gas_price_fri` values are validated against this corrupted price; every honest proposer must also publish `≈ u128::MAX` or be rejected, permanently breaking the fee market.
