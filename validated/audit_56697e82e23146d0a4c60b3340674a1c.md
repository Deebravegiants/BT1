### Title
`ForeignTxSignPayload` Lacks NEAR Receipt Binding, Enabling Response Replay by a Byzantine Participant — (`crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

The `ForeignTxSignPayloadV1` structure that the MPC network signs contains only the foreign-chain RPC request and extracted values. It does not include any NEAR-specific unique identifier (receipt ID, nonce, or contract address). Simultaneously, `respond_verify_foreign_tx` verifies only that the ECDSA signature is valid for the provided `payload_hash`; it never checks that the hash corresponds to the pending request being resolved. A single Byzantine participant (below the signing threshold) can therefore capture a previously produced valid `VerifyForeignTransactionResponse` and replay it against any future pending request for the same foreign transaction — including one submitted after a foreign-chain reorganization has invalidated the original transaction.

### Finding Description

**Signed payload structure — no NEAR context:**

`ForeignTxSignPayloadV1` is defined as:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
``` [1](#0-0) 

The 32-byte hash that the MPC network actually signs is `SHA-256(borsh(ForeignTxSignPayload))`. Neither the NEAR receipt ID, the NEAR contract address, nor any per-request nonce is committed to in this hash.

**Node-side signing confirms the gap:**

When the node builds the internal `SignatureRequest` for the threshold-signing protocol, it records `id: request.id` (the unique receipt-derived ID) and `receipt_id: request.receipt_id`, but these fields are **not** fed into the `ForeignTxSignPayload` that becomes the signed message:

```rust
