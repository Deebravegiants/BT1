### Title
Attached NEAR Deposit Silently Absorbed Without Refund in `submit_participant_info` Update Path — (File: crates/contract/src/lib.rs)

---

### Summary

`submit_participant_info` is marked `#[payable]` and contains deposit-refund logic, but that logic is gated behind a condition that is **false** for the normal periodic re-attestation path (existing participant updating their attestation). Any NEAR attached in that path is silently absorbed by the contract with no refund and no error.

---

### Finding Description

In `crates/contract/src/lib.rs`, `submit_participant_info` computes a boolean to decide whether to run deposit-handling:

```rust
let caller_is_not_participant = self.voter_account().is_err();
let is_new_attestation = matches!(
    attestation_insertion_result,
    ParticipantInsertion::NewlyInsertedParticipant
);

let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... cost check and refund of excess ...
}

Ok(())
``` [1](#0-0) 

When the caller **is** an existing participant (`caller_is_not_participant = false`) **and** the attestation is an update to an existing entry (`is_new_attestation = false`), `attestation_storage_must_be_paid_by_caller` evaluates to `false`. The deposit-handling block is skipped entirely, and the function returns `Ok(())` without inspecting `env::attached_deposit()` or issuing any refund.

Because the function carries `#[payable]`, the NEAR runtime accepts any attached deposit without complaint. The attached amount is credited to the contract's balance and is never returned. [2](#0-1) 

The two paths that **do** refund correctly are:

- New attestation from any caller (`is_new_attestation = true`)
- Any call from a non-participant (`caller_is_not_participant = true`) [3](#0-2) 

The update path — the one executed every hour by every running node — has no such protection.

---

### Impact Explanation

Any NEAR attached to `submit_participant_info` by an existing participant is permanently absorbed into the contract's balance. The contract has no withdrawal function, so the funds are effectively frozen. This breaks the accounting invariant that `#[payable]` functions must either consume or refund attached deposits, and matches the **Medium** impact class: *balance/accounting invariant broken without relying on network-level DoS or operator misconfiguration*.

---

### Likelihood Explanation

The node software (`periodic_attestation_submission` in `crates/node/src/tee/remote_attestation.rs`) attaches **0 NEAR** by default, so the normal automated path is safe. [4](#0-3) 

However, the operator documentation explicitly instructs first-time joiners to call `submit_participant_info` **manually with a deposit**:

> *"first-time joiners must attach enough yoctoNEAR for storage; the node attaches 0, so call `submit_participant_info` manually with `--deposit` once."* [5](#0-4) 

An operator who later re-submits manually (e.g., after a TEE upgrade or attestation expiry) and attaches a deposit — a reasonable action given the documented precedent — will silently lose that NEAR. The function is `#[payable]`, which signals to callers that deposits are accepted, making the silent absorption a genuine trap.

---

### Recommendation

Add an unconditional deposit refund after the `if attestation_storage_must_be_paid_by_caller` block, or refactor the function so the update path explicitly rejects any non-zero deposit (e.g., by asserting `env::attached_deposit() == 0` when `!attestation_storage_must_be_paid_by_caller`). The pattern already used elsewhere in the contract is the correct model:

```rust
if let Some(diff) = attached.checked_sub(cost)
    && diff > NearToken::from_yoctonear(0)
{
    Promise::new(account_id).transfer(diff).detach();
}
``` [6](#0-5) 

---

### Proof of Concept

1. Contract is in Running state; `node_a.near` is an existing participant with a stored attestation.
2. `node_a.near` calls `submit_participant_info` with `attached_deposit = 1 NEAR` (e.g., operator manually re-attesting after a TEE upgrade, following the documented first-join procedure).
3. `add_participant` returns `ParticipantInsertion::UpdatedExistingParticipant` → `is_new_attestation = false`.
4. `voter_account()` succeeds for `node_a.near` → `caller_is_not_participant = false`.
5. `attestation_storage_must_be_paid_by_caller = false || false = false`.
6. The `if` block is skipped; `Ok(())` is returned.
7. The 1 NEAR is credited to the contract's balance. No refund is issued. The operator's account is permanently short 1 NEAR with no on-chain indication of what happened. [7](#0-6)
