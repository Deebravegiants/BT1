### Title
Reward tokens permanently stuck when all stakers exit during an active reward period — (`contracts/KERNEL/KernelDepositPool.sol`)

### Summary
`KernelDepositPool` guards `notifyRewardAmount` against being called with zero stakers, but provides no equivalent protection for the mid-period case where all stakers exit via `initiateWithdrawal`. When `totalKernelStaked` drops to zero during an active reward window, the `rewardPerToken()` function freezes `rewardPerTokenStored` while `updatedAt` continues to advance, causing all reward tokens emitted during the zero-staker interval to become permanently irrecoverable.

### Finding Description

`KernelDepositPool.notifyRewardAmount` contains an explicit guard at line 570:

```solidity
if (totalKernelStaked == 0) revert NoStakedTokens();
``` [1](#0-0) 

This prevents starting a reward period with no stakers. However, once a reward period is active, any staker can call `initiateWithdrawal`, which **immediately** decrements `totalKernelStaked` before the withdrawal delay elapses:

```solidity
balanceOf[msg.sender] -= _amount;
totalKernelStaked -= _amount;
``` [2](#0-1) 

If all stakers call `initiateWithdrawal`, `totalKernelStaked` reaches zero mid-period. From that point on, every invocation of the `updateReward` modifier executes:

```solidity
rewardPerTokenStored = rewardPerToken();   // returns unchanged stored value (totalKernelStaked == 0)
updatedAt = lastTimeRewardApplicable();    // advances the timestamp
``` [3](#0-2) 

The `rewardPerToken()` function explicitly short-circuits when `totalKernelStaked == 0`:

```solidity
if (totalKernelStaked == 0) {
    return rewardPerTokenStored;
}
``` [4](#0-3) 

Because `rewardPerTokenStored` is frozen while `updatedAt` advances, the reward tokens that should have been distributed during the zero-staker interval are never credited to any account. They accumulate as an untracked surplus inside the contract. There is no admin rescue or sweep function to recover them.

When a new staker eventually joins and a new reward period is started via `notifyRewardAmount`, the `remaining` rollover calculation only accounts for **future** undistributed rewards:

```solidity
uint256 remaining = (finishAt - block.timestamp) * rewardRate;
rewardRate = (receivedAmount + remaining) / duration;
``` [5](#0-4) 

The tokens lost during the zero-staker gap are not included in `remaining` and are permanently stranded.

### Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

Reward tokens emitted during any interval where `totalKernelStaked == 0` are permanently locked in the contract with no recovery path. The magnitude of loss is `rewardRate × (duration of zero-staker period)`. For a typical reward period of days to weeks, this can represent a substantial fraction of the total reward allocation.

### Likelihood Explanation

**Low.** Triggering the loss requires all stakers to exit simultaneously during an active reward window. This is realistic in adversarial conditions (e.g., a coordinated exit, a depeg event for the staking token, or a migration to a new contract version). The `initiateWithdrawal` path is fully permissionless and requires no admin action.

### Recommendation

Two complementary mitigations:

1. **Track unallocated rewards.** In the `updateReward` modifier, when `totalKernelStaked == 0`, accumulate the reward tokens that would have been emitted into a dedicated `unallocatedRewards` variable instead of silently discarding them. Roll this surplus into the next `notifyRewardAmount` call.

2. **Add an admin rescue function.** Provide a `recoverUnallocatedRewards` function restricted to `DEFAULT_ADMIN_ROLE` that transfers any surplus reward tokens (i.e., `rewardsToken.balanceOf(address(this)) - totalTrackedRewards`) to a treasury address, analogous to the recommendation in the reference report.

### Proof of Concept

```solidity
function test_KernelDepositPool_rewardsLost_noStakers() public {
    address admin = address(this);
    address user1 = address(0xa11ce);

    MockERC20 kernelToken = new MockERC20("KERNEL", "K", 18);
    MockERC20 rewardToken  = new MockERC20("Reward", "R", 18);

    KernelDepositPool pool = new KernelDepositPool();
    pool.initialize(admin, address(kernelToken), address(rewardToken));
    pool.setRewardsDuration(7 days);
    pool.setWithdrawalDelay(1);

    // User1 stakes
    kernelToken.mint(user1, 100 ether);
    vm.startPrank(user1);
    kernelToken.approve(address(pool), 100 ether);
    pool.stake(100 ether);
    vm.stopPrank();

    // Admin notifies rewards (stakers present → succeeds)
    rewardToken.mint(admin, 1 ether);
    rewardToken.approve(address(pool), 1 ether);
    pool.notifyRewardAmount(1 ether);

    // User1 immediately initiates withdrawal → totalKernelStaked = 0
    vm.prank(user1);
    pool.initiateWithdrawal(100 ether);

    // Reward period elapses with zero stakers
    skip(7 days);

    // User1 claims withdrawal and tries to collect rewards
    vm.prank(user1);
    pool.claimWithdrawal(1);

    vm.prank(user1);
    pool.getReward();

    // Rewards earned only for the instant before withdrawal; the rest are stuck
    // rewardToken balance of pool ≈ 1 ether (minus dust earned before withdrawal)
    assertGt(rewardToken.balanceOf(address(pool)), 0.99 ether); // nearly all rewards stuck
}
```
