### Title
Missing Per-Asset Chainlink Staleness Check Allows Stale Prices to Silently Affect rsETH Rate and Trigger Protocol Pause - (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

---

### Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards the `updatedAt` return value entirely, applying no staleness threshold to any asset. Because different LST assets (stETH, cbETH, ETHx, rETH, sfrxETH, etc.) have different Chainlink heartbeat intervals, the absence of any per-asset staleness guard means a stale feed for any one asset silently propagates into the rsETH price calculation, potentially triggering the protocol's downside-protection pause and temporarily freezing user funds.

---

### Finding Description
`ChainlinkPriceOracle.getAssetPrice()` retrieves price data via `latestRoundData()` but only consumes the `price` field:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L49-55
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

The `updatedAt` timestamp (4th return value) and `answeredInRound` (5th return value) are both silently discarded. There is no `block.timestamp - updatedAt <= maxStaleness` guard, and no per-asset heartbeat mapping. The analog to the original report is direct: just as the Morpho protocol used a single `TWAP_INTERVAL` for pools with different liquidity profiles, LRT-rsETH applies a single (absent) staleness policy to all assets regardless of their individual Chainlink heartbeat intervals (e.g., ETH/USD: 1 hour; some LST feeds: 24 hours).

By contrast, the pool-level oracle wrapper `ChainlinkOracleForRSETHPoolCollateral.sol` does implement partial checks (`answeredInRound < roundID`, `timestamp == 0`), but the core `ChainlinkPriceOracle` used by `LRTOracle` for rsETH price computation has none.

The stale price flows into `LRTOracle._updateRsETHPrice()` via `_getTotalEthInProtocol()` → `LRTOracle.getAssetPrice()` → `ChainlinkPriceOracle.getAssetPrice()`. If the stale price is sufficiently below the actual price, `newRsETHPrice` drops below `highestRsethPrice` by more than `pricePercentageLimit`, triggering the downside-protection block at lines 270–282 of `LRTOracle.sol`, which calls `lrtDepositPool.pause()` and `withdrawalManager.pause()`.

---

### Impact Explanation
**Medium — Temporary freezing of funds.**

A stale Chainlink price for any supported LST asset causes `_updateRsETHPrice()` to compute an artificially low rsETH price. If the deviation exceeds `pricePercentageLimit`, the protocol auto-pauses both `LRTDepositPool` and `LRTWithdrawalManager`, preventing all deposits and withdrawals until an admin manually unpauses. Users with pending withdrawals or deposits are temporarily frozen with no recourse until admin intervention.

---

### Likelihood Explanation
**Low.** Chainlink feeds do occasionally lag or go stale during periods of network congestion or oracle node downtime. The impact is conditional on the stale price deviating enough to cross `pricePercentageLimit`. However, no attacker action is required — this is a passive, market-condition-triggered failure path reachable by any user calling `updateRSETHPrice()` (a public function).

---

### Recommendation
Add a per-asset configurable staleness threshold to `ChainlinkPriceOracle`, mirroring the fix recommended in the original report:

```solidity
mapping(address asset => uint256 maxStaleness) public assetMaxStaleness;

function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();
    require(answeredInRound >= roundId, "Stale round");
    require(updatedAt != 0, "Incomplete round");
    uint256 staleness = assetMaxStaleness[asset];
    if (staleness > 0) require(block.timestamp - updatedAt <= staleness, "Stale price");
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

The `assetMaxStaleness` mapping should be settable by `onlyLRTManager` and initialized to each asset's Chainlink heartbeat interval at the time the price feed is registered.

---

### Proof of Concept

1. The Chainlink ETHx/ETH feed goes stale (e.g., `updatedAt` is 26 hours ago; heartbeat is 24 hours).
2. Any caller invokes `LRTOracle.updateRSETHPrice()` (public, no access control).
3. `_updateRsETHPrice()` calls `_getTotalEthInProtocol()`, which calls `LRTOracle.getAssetPrice(ethx)`.
4. `LRTOracle.getAssetPrice(ethx)` delegates to `ChainlinkPriceOracle.getAssetPrice(ethx)`.
5. `ChainlinkPriceOracle` returns the stale (lower) price with no staleness check.
6. `totalETHInProtocol` is understated; `newRsETHPrice` is computed below `highestRsethPrice`.
7. If `highestRsethPrice - newRsETHPrice > pricePercentageLimit * highestRsethPrice / 1e18`, the condition at `LRTOracle.sol:277` fires.
8. `lrtDepositPool.pause()` and `withdrawalManager.pause()` are called — all user deposits and withdrawals are frozen until admin unpauses. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
