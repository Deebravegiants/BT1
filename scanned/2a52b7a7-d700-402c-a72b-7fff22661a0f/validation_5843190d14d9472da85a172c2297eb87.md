### Title
Block Stuffing Can Delay `updateRSETHPrice()`, Preventing Automatic Pause and Serving Stale Price via `RSETHPriceFeed` — (`contracts/oracles/RSETHPriceFeed.sol`)

---

### Summary

`RSETHPriceFeed.latestRoundData()` derives the rsETH/USD price by reading `LRTOracle.rsETHPrice` — a stored state variable — and multiplying it by the live ETH/USD Chainlink feed. That stored value is only updated when `LRTOracle.updateRSETHPrice()` (a public, permissionless function) is called. An attacker can use block stuffing to prevent that call from landing, keeping `rsETHPrice` stale while the true on-chain value has moved beyond `pricePercentageLimit`, and simultaneously preventing the automatic pause of `LRTDepositPool` and `LRTWithdrawalManager` that would otherwise trigger.

---

### Finding Description

**`RSETHPriceFeed.latestRoundData()` reads a stored, lazily-updated price:** [1](#0-0) 

The `rsETHPrice()` call resolves to `LRTOracle.rsETHPrice`, a plain storage variable: [2](#0-1) 

This variable is only written inside `_updateRsETHPrice()`, which is reached exclusively through the two public/external entry points: [3](#0-2) 

`updateRSETHPrice()` carries no access control — anyone may call it — but that also means an attacker can block it by filling every block with high-gas transactions (block stuffing), preventing any honest caller's transaction from being included.

**The automatic pause lives entirely inside `_updateRsETHPrice()`:** [4](#0-3) 

If `_updateRsETHPrice()` is never executed, the pause never fires, `rsETHPrice` is never updated, and `RSETHPriceFeed` continues to return the pre-deviation price to every integrated protocol.

There is **no staleness check** anywhere in `RSETHPriceFeed`: [5](#0-4) 

The `updatedAt` timestamp returned is the one from the ETH/USD Chainlink feed, not from the last `updateRSETHPrice()` call, so downstream consumers cannot detect that the rsETH component is stale.

---

### Impact Explanation

During the block-stuffing window:

- `LRTOracle.rsETHPrice` remains at the pre-deviation (inflated) value.
- `RSETHPriceFeed.latestRoundData()` returns `staleRsETHPrice × currentETH/USD`, an inflated rsETH/USD figure.
- `LRTDepositPool` and `LRTWithdrawalManager` are **not paused**, so deposits and withdrawals continue at the stale price.
- Any external protocol (e.g., a lending market) that uses `RSETHPriceFeed` as collateral pricing will accept rsETH at an inflated value, enabling over-collateralized borrowing against an asset whose true backing has dropped.

Impact: **Low — Block stuffing** (contract fails to deliver promised safety guarantees; oracle serves a price that no longer reflects true backing value).

---

### Likelihood Explanation

Block stuffing on Ethereum mainnet is expensive but has been executed in real DeFi exploits. The attacker's profit window only needs to last long enough to exploit the stale price in integrated lending protocols. The cost/benefit ratio is unfavorable for small deviations but becomes viable when the price deviation is large (e.g., a significant LST depeg event), exactly the scenario `pricePercentageLimit` is designed to guard against.

---

### Recommendation

1. **Add a staleness guard in `RSETHPriceFeed`**: record `block.timestamp` each time `LRTOracle.rsETHPrice` is updated and expose it; `latestRoundData()` should revert (or return a sentinel) if the stored price is older than a configurable heartbeat (e.g., 24 hours).
2. **Return the correct `updatedAt`**: `latestRoundData()` should return the timestamp of the last `updateRSETHPrice()` call, not the ETH/USD feed's `updatedAt`, so downstream consumers can apply their own staleness checks.
3. **Consider a push-based pause**: allow any caller to trigger a pause-only path that reads current asset prices and compares against `highestRsethPrice` without updating `rsETHPrice`, so the safety pause can land even if the full price-update transaction is stuffed out.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Fork test (Foundry) — run against a mainnet fork
// 1. Deploy / use existing LRTOracle, RSETHPriceFeed, LRTDepositPool, LRTWithdrawalManager
// 2. Record current rsETHPrice and highestRsethPrice
// 3. Simulate a large drop in underlying asset value (e.g., manipulate stETH oracle price
//    on the fork so that _getTotalEthInProtocol() would return a value >pricePercentageLimit
//    below highestRsethPrice * rsethSupply)
// 4. Do NOT call updateRSETHPrice() — simulate block stuffing by simply skipping the call
// 5. Assert:
//    a. RSETHPriceFeed.latestRoundData().answer == stalePrice (pre-deviation)
//    b. LRTDepositPool.paused() == false
//    c. LRTWithdrawalManager.paused() == false
//    d. LRTOracle.rsETHPrice == stalePrice

function testBlockStuffingStalePrice() public {
    uint256 stalePriceBefore = lrtOracle.rsETHPrice();

    // Simulate asset price drop on fork (manipulate mock oracle)
    mockAssetOracle.setPrice(address(stETH), stalePriceBefore * 80 / 100); // 20% drop

    // Block stuffing: attacker fills blocks — modelled here by simply NOT calling updateRSETHPrice()
    // In a real attack, attacker submits high-gas txs to fill every block for the window duration.

    // Assert RSETHPriceFeed still returns pre-deviation price
    (, int256 answer,,,) = rsETHPriceFeed.latestRoundData();
    assertEq(uint256(answer), stalePriceBefore * ethUsdPrice / 1e18);

    // Assert protocol is NOT paused
    assertFalse(lrtDepositPool.paused());
    assertFalse(lrtWithdrawalManager.paused());

    // Assert stored rsETHPrice is stale
    assertEq(lrtOracle.rsETHPrice(), stalePriceBefore);
}
```

The test passes on unmodified code, confirming that block stuffing successfully prevents the pause and keeps `RSETHPriceFeed` serving the pre-deviation price.
