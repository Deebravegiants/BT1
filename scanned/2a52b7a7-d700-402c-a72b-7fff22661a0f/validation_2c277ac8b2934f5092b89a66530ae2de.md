### Title
Pending Withdrawal Requests Permanently Frozen When Asset Is Removed From Supported List — (`contracts/LRTWithdrawalManager.sol`, `contracts/LRTConfig.sol`)

### Summary
The `unlockQueue` function in `LRTWithdrawalManager` is gated by `onlySupportedAsset(asset)`. If an asset is removed from the supported list via `LRTConfig.removeSupportedAsset` while users have pending (locked) withdrawal requests for that asset, those requests can never be unlocked. Because `completeWithdrawal` requires a request to be unlocked first, and no cancel/refund mechanism exists, the rsETH already transferred into the withdrawal manager at `initiateWithdrawal` is permanently frozen.

### Finding Description

The withdrawal lifecycle in `LRTWithdrawalManager` is a three-step process:

1. **`initiateWithdrawal`** — user's rsETH is pulled into the contract; a locked withdrawal request is created; `assetsCommitted[asset]` is incremented.
2. **`unlockQueue`** — operator-only call that advances `nextLockedNonce[asset]`, burns the rsETH held in the contract, and redeems the underlying asset from the unstaking vault.
3. **`completeWithdrawal`** — user receives the underlying asset; requires the request nonce to be below `nextLockedNonce[asset]` (i.e., already unlocked).

`unlockQueue` carries the `onlySupportedAsset(asset)` modifier: [1](#0-0) 

This modifier resolves to: [2](#0-1) 

`LRTConfig.removeSupportedAsset` (callable by `DEFAULT_ADMIN_ROLE`) deletes `isSupportedAsset[asset]` and `assetStrategy[asset]`: [3](#0-2) 

Its only guard against removal while deposits exist is: [4](#0-3) 

This guard checks the deposit pool's aggregate view of total asset deposits. It does **not** directly check `assetsCommitted[asset]` in the withdrawal manager, nor does it verify whether `nextLockedNonce[asset] < nextUnusedNonce[asset]` (i.e., whether locked-but-not-yet-unlocked withdrawal requests exist). If `maxNegligibleAmount` is set to any non-zero value, or if the deposit pool's `getTotalAssetDeposits` does not include assets held in the `LRTUnstakingVault` that are committed to pending requests, the guard can be bypassed.

After removal, `unlockQueue` reverts with `AssetNotSupported` for that asset. `completeWithdrawal` then permanently reverts with `WithdrawalLocked` because `nextLockedNonce[asset]` can never advance: [5](#0-4) 

There is no `cancelWithdrawal`, no refund path, and no mechanism to recover the rsETH held in the contract for locked requests. The `sweepRemainingAssets` function also carries `onlySupportedAsset(asset)`, so it cannot be used to rescue funds either: [6](#0-5) 

### Impact Explanation

Users who called `initiateWithdrawal` for an asset that is subsequently removed have their rsETH permanently locked inside `LRTWithdrawalManager`. The rsETH was transferred in at step 1 and is only burned at step 2 (`unlockQueue`). With step 2 permanently blocked, the rsETH cannot be recovered by any on-chain path. This constitutes **permanent freezing of user funds**.

### Likelihood Explanation

The scenario requires the admin to call `removeSupportedAsset` for an asset while locked withdrawal requests exist. This is a realistic operational event (e.g., deprecating a poorly-performing LST). The guard in `removeSupportedAsset` provides partial protection when `maxNegligibleAmount == 0` and `getTotalAssetDeposits` fully accounts for committed assets, but the guard is not verified against the withdrawal manager's own state (`assetsCommitted`, `nextLockedNonce`). Any non-zero `maxNegligibleAmount` setting or accounting gap makes the freeze reachable. Likelihood is **Low to Medium**.

### Recommendation

1. In `removeSupportedAsset`, add an explicit check that no locked withdrawal requests exist for the asset:
   ```solidity
   ILRTWithdrawalManager wm = ILRTWithdrawalManager(getContract(LRTConstants.LRT_WITHDRAWAL_MANAGER));
   require(wm.nextLockedNonce(asset) >= wm.nextUnusedNonce(asset), "Pending locked withdrawals exist");
   ```
2. Alternatively, remove the `onlySupportedAsset` modifier from `unlockQueue` so that already-queued requests can always be processed regardless of the asset's current support status.
3. Add a `cancelWithdrawal` function that allows users to reclaim their rsETH if their request has been in the locked queue beyond a maximum waiting period, providing a safety valve regardless of asset status.

### Proof of Concept

1. Asset `stETH` is supported; `maxNegligibleAmount` in `LRTConfig` is set to `1e18` (1 stETH).
2. Alice calls `initiateWithdrawal(stETH, rsETHAmount, "")` — her rsETH is transferred to `LRTWithdrawalManager`; `assetsCommitted[stETH]` increases by `< 1e18` stETH worth.
3. Admin calls `LRTConfig.removeSupportedAsset(stETH, idx)` — the guard passes because `getTotalAssetDeposits(stETH) <= maxNegligibleAmount`; `isSupportedAsset[stETH]` is deleted.
4. Operator attempts `unlockQueue(stETH, ...)` — reverts: `AssetNotSupported`.
5. Alice calls `completeWithdrawal(stETH, "")` — reverts: `WithdrawalLocked` (her nonce is still `>= nextLockedNonce[stETH]`).
6. Alice's rsETH remains in `LRTWithdrawalManager` with no recovery path. [7](#0-6) [8](#0-7) [9](#0-8)

    /// @dev Removes a supported asset
    /// @param asset The asset address
    function removeSupportedAsset(
        address asset,
        uint256 tokenIndex
    )
        external
        onlySupportedAsset(asset)
        onlyRole(DEFAULT_ADMIN_ROLE)
    {
        UtilLib.checkNonZeroAddress(asset);

        if (supportedAssetList[tokenIndex] != asset) {
            revert TokenNotFoundError();
        }

        address depositPool = getContract(LRTConstants.LRT_DEPOSIT_POOL);

        if (ILRTDepositPool(depositPool).getTotalAssetDeposits(asset) > maxNegligibleAmount) {
            revert CannotRemoveAssetWithDeposits(asset);
        }

        delete isSupportedAsset[asset];
        delete assetStrategy[asset];
        depositLimitByAsset[asset] = 0;

        supportedAssetList[tokenIndex] = supportedAssetList[supportedAssetList.length - 1];
        supportedAssetList.pop();

        emit RemovedSupportedAsset(asset);
    }
