### Title
Misleading `onlyManager` Modifier Silently Grants `DEFAULT_ADMIN_ROLE` Access to Manager-Restricted Functions - (File: contracts/offchain/OffchainConfig.sol)

### Summary

`OffchainConfig.sol` defines an `onlyManager` modifier whose name and revert error (`OnlyManagerCanCall`) imply that only the `MANAGER_ROLE` holder may call the guarded functions. In reality, the modifier also passes for any `DEFAULT_ADMIN_ROLE` holder, silently expanding the privileged set beyond what the name communicates.

### Finding Description

The `onlyManager` modifier in `OffchainConfig` is implemented as:

```solidity
modifier onlyManager() {
    if (!hasRole(MANAGER_ROLE, msg.sender) && !hasRole(DEFAULT_ADMIN_ROLE, msg.sender)) {
        revert OnlyManagerCanCall();
    }
    _;
}
``` [1](#0-0) 

The modifier guards five state-changing functions:

- `setBufferPoolLimit(address asset, uint256 limit)` [2](#0-1) 
- `setBufferPoolLimitETH`, `setBufferPoolLimitSTETH`, `setBufferPoolLimitETHX` [3](#0-2) 
- `changeGlobalLimit(uint256 newLimit)` [4](#0-3) 

During `initialize`, both `DEFAULT_ADMIN_ROLE` and `MANAGER_ROLE` are granted to `_initialManager`, and `MANAGER_ROLE`'s admin is set to `DEFAULT_ADMIN_ROLE`:

```solidity
_grantRole(DEFAULT_ADMIN_ROLE, _initialManager);
_grantRole(MANAGER_ROLE, _initialManager);
_setRoleAdmin(MANAGER_ROLE, DEFAULT_ADMIN_ROLE);
_setRoleAdmin(USER_ROLE, MANAGER_ROLE);
``` [5](#0-4) 

After deployment, the `DEFAULT_ADMIN_ROLE` may be transferred to a separate governance multisig or timelock while `MANAGER_ROLE` is held by an operational account. At that point, the `DEFAULT_ADMIN_ROLE` holder — who is not the designated manager — can still invoke all `onlyManager`-guarded functions, contrary to what the modifier name and revert error communicate.

The `onlyAuthorizedUser` modifier has the same pattern — it accepts `USER_ROLE` **or** `MANAGER_ROLE` — but its name is at least partially descriptive. The `onlyManager` case is the direct analog to the reported `onlyDAO` issue because the revert message `OnlyManagerCanCall()` actively misleads callers and auditors about who is authorized.

### Impact Explanation

The `bufferPoolLimits` and `globalLimit` values act as upper bounds on `bufferPoolAmounts`, which offchain scripts read to determine available liquidity for withdrawals. A `DEFAULT_ADMIN_ROLE` holder that is not the intended manager can:

1. Set per-asset `bufferPoolLimits` to zero, preventing the authorized manager from ever setting a non-zero `bufferPoolAmount` for that asset (the `validBufferAmount` modifier would always revert). [6](#0-5) 
2. Reduce `globalLimit` to zero, blocking all `setBufferPoolAmount` calls across every asset.

This can cause offchain withdrawal-management scripts to read zero available liquidity and halt withdrawals, constituting a temporary freeze of withdrawal operations. Impact: **Low — contract fails to deliver promised returns (withdrawal liquidity signalling) without direct loss of funds.**

### Likelihood Explanation

Likelihood is low. It requires the `DEFAULT_ADMIN_ROLE` to be held by an entity distinct from the `MANAGER_ROLE` holder and for that entity to act outside its intended scope. However, the misleading modifier name means the separation of duties is not enforced by code, only by convention, and any future role reassignment silently widens the attack surface without any code change.

### Recommendation

Rename the modifier to `onlyManagerOrAdmin` (or `onlyManagerRole` if the intent is to exclude `DEFAULT_ADMIN_ROLE`). If `DEFAULT_ADMIN_ROLE` access is intentional, update the revert error from `OnlyManagerCanCall` to `OnlyManagerOrAdminCanCall`. If it is unintentional, remove the `DEFAULT_ADMIN_ROLE` branch:

```solidity
modifier onlyManager() {
    if (!hasRole(MANAGER_ROLE, msg.sender)) {
        revert OnlyManagerCanCall();
    }
    _;
}
```

### Proof of Concept

1. Deploy `OffchainConfig` with `_initialManager = address(A)`.
2. Admin `A` calls `grantRole(DEFAULT_ADMIN_ROLE, address(B))` and `revokeRole(DEFAULT_ADMIN_ROLE, address(A))`. Now `B` holds `DEFAULT_ADMIN_ROLE` and `A` holds `MANAGER_ROLE`.
3. `B` calls `changeGlobalLimit(1)` — succeeds despite `B` not holding `MANAGER_ROLE`, because `onlyManager` passes for `DEFAULT_ADMIN_ROLE`.
4. `A` (the legitimate manager) calls `setBufferPoolAmount(ETH, 2)` — reverts with `BufferAmountExceedsGlobalLimit` because `globalLimit` is now `1`.
5. Offchain withdrawal scripts read `bufferPoolAmounts[ETH] == 0` and treat ETH withdrawals as unavailable.
