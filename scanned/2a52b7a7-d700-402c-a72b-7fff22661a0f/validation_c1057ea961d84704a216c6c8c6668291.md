Looking at the code directly:

### Title
Unpermissioned Zero-Value Calls to `sendETHToL1ViaBridge` Enable Block Stuffing — (`contracts/bridges/UnichainMessenger.sol`)

---

### Summary

`UnichainMessenger.sendETHToL1ViaBridge` has no access control and accepts calls where both `msg.value` and `value` are zero. Any unprivileged actor can call it in a tight loop at minimal cost, consuming Unichain block gas and temporarily delaying the `BRIDGER_ROLE`'s ability to execute `RSETHPoolNoWrapper.bridgeAssetsViaNativeBridge`.

---

### Finding Description

`sendETHToL1ViaBridge` is declared `external payable nonReentrant` with no role modifier: [1](#0-0) 

The only guard is the value-match check on line 25:

```solidity
if (msg.value != value) revert MismatchedMsgValue();
```

When both `msg.value` and `value` are `0`, the condition `0 != 0` evaluates to `false` — the check passes silently. Execution then reaches:

```solidity
IUnichainMessenger(l2bridge).bridgeETHTo{ value: 0 }(target, DEFAULT_GAS_LIMIT, bytes(""));
```

Because `l2bridge` is fully attacker-controlled and the call carries `value: 0`, passing any EOA or a cheap stub contract as `l2bridge` causes the EVM `CALL` to succeed (a call to a non-contract address with zero value returns success). The `nonReentrant` modifier only prevents reentrancy within a single invocation; it does not prevent repeated independent transactions.

The legitimate caller path in `RSETHPoolNoWrapper.bridgeAssetsViaNativeBridge` always passes a non-zero `ethBalanceMinusFees`: [2](#0-1) 

There is no corresponding zero-value guard in `UnichainMessenger` itself.

---

### Impact Explanation

An attacker submits a stream of zero-cost (ETH-wise) transactions calling `sendETHToL1ViaBridge(eoa, eoa, 0)` with `msg.value=0`. Each transaction consumes ~30,000–50,000 gas (21k base + nonReentrant overhead + external CALL). On Unichain (OP Stack), gas is cheap, making it economically feasible to fill blocks. This temporarily prevents the `BRIDGER_ROLE` from getting `bridgeAssetsViaNativeBridge` included, delaying ETH bridging from L2 to L1.

Impact: **Low — Block stuffing** (within the defined allowed scope).

---

### Likelihood Explanation

The preconditions are minimal: no ETH required, no special role, no prior state setup. The attacker only needs to pay L2 gas fees. On OP Stack chains the sequencer can in principle reorder transactions, but there is no protocol-level guarantee that the sequencer will prioritize the `BRIDGER_ROLE` call over a flood of attacker transactions. The attack is repeatable across consecutive blocks.

---

### Recommendation

Add access control to `sendETHToL1ViaBridge` so only the trusted pool contract (or a designated role) can invoke it:

```solidity
address public immutable authorizedCaller;

function sendETHToL1ViaBridge(address l2bridge, address target, uint256 value)
    external payable nonReentrant
{
    if (msg.sender != authorizedCaller) revert Unauthorized();
    if (msg.value != value) revert MismatchedMsgValue();
    IUnichainMessenger(l2bridge).bridgeETHTo{ value: value }(target, DEFAULT_GAS_LIMIT, bytes(""));
}
```

Alternatively, add a zero-value guard (`if (value == 0) revert InvalidAmount();`) as a secondary defence, though access control is the stronger fix.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import "forge-std/Test.sol";
import "contracts/bridges/UnichainMessenger.sol";

contract StubBridge {
    // accepts any call, does nothing
    fallback() external payable {}
}

contract BlockStuffingPoC is Test {
    UnichainMessenger messenger;
    StubBridge stub;
    address attacker = address(0xBEEF);

    function setUp() public {
        messenger = new UnichainMessenger();
        stub = new StubBridge();
    }

    function test_zeroValueSpam() public {
        vm.startPrank(attacker);
        // No ETH needed; loop succeeds every iteration
        for (uint256 i = 0; i < 500; i++) {
            messenger.sendETHToL1ViaBridge(address(stub), address(0xDEAD), 0);
        }
        vm.stopPrank();
        // 500 successful calls, each consuming ~30-50k gas, with zero ETH spent
        // A BRIDGER_ROLE tx submitted in the same block may be crowded out
    }
}
```

Each iteration passes the `msg.value == value` check (both zero), calls `stub.bridgeETHTo{value:0}(...)` which succeeds, and returns normally. No ETH is spent by the attacker; only gas fees are paid.
