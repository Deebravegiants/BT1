### Title
Arithmetic Underflow in `_processClaim` Permanently Freezes User Allocations When New Merkle Root Reduces `cumulativeAmount` — (`contracts/KERNEL/KernelMerkleDistributor.sol`)

---

### Summary

`_processClaim` computes the incremental claimable amount as an unchecked subtraction under Solidity 0.8's default overflow protection. If the owner ever publishes a new merkle root in which a user's `cumulativeAmount` is lower than the value already stored in `userClaims[account].cumulativeAmount`, the subtraction reverts with an arithmetic underflow. Because there is no admin function to reset `userClaims` and no on-chain monotonicity guard on `setMerkleRoot`, the user is permanently unable to claim from any future root whose value does not exceed their stored watermark.

---

### Finding Description

In `_processClaim`, the claimable delta is computed at:

```solidity
uint256 claimableAmount = cumulativeAmount - userClaims[account].cumulativeAmount;
``` [1](#0-0) 

`userClaims[account].cumulativeAmount` is written once per successful claim and never decremented or reset:

```solidity
userClaims[account].lastClaimedIndex = index;
userClaims[account].cumulativeAmount = cumulativeAmount;
``` [2](#0-1) 

`setMerkleRoot` accepts any non-zero root with no on-chain validation that the new root's per-user cumulative amounts are monotonically non-decreasing relative to stored state:

```solidity
function setMerkleRoot(bytes32 _merkleRootToSet) external onlyOwner {
    if (_merkleRootToSet == bytes32(0)) { revert ZeroValueProvided(); }
    currentMerkleRoot = _merkleRootToSet;
    currentMerkleRootIndex++;
    currentIndex++;
    emit MerkleRootSet(currentMerkleRootIndex, currentMerkleRoot);
}
``` [3](#0-2) 

The full set of admin functions (`setKernelDepositPool`, `setProtocolTreasury`, `setFeeInBPS`, `setMerkleRoot`, `pause`, `unpause`) contains no mechanism to reset or adjust `userClaims`. [4](#0-3) 

---

### Impact Explanation

Once a user's stored `cumulativeAmount` is 500 and the active root encodes 300 for that user, every call to `claim` or `claimAndStake` for that user reverts with an arithmetic underflow. The user's KERNEL allocation is permanently frozen with no on-chain recovery path. The only escape would be a new root that raises the user's cumulative amount back above 500, which may be impossible if the intended total allocation is 300. This satisfies **Critical — Permanent freezing of funds**.

---

### Likelihood Explanation

The trigger does not require a malicious owner. It can arise from:
- An off-chain merkle generation bug that resets cumulative totals instead of accumulating them.
- A legitimate correction to an over-allocation in a prior epoch.
- A root regeneration after a data pipeline failure that loses historical state.

The owner is a single EOA or multisig with no on-chain constraint preventing this. The likelihood is **Low-to-Medium** (requires an operational error or deliberate correction), but the impact is irreversible once it occurs.

---

### Recommendation

1. **Enforce monotonicity on-chain**: Store a per-user `maxCumulativeAmount` high-water mark and revert in `_processClaim` with a descriptive error (e.g., `CumulativeAmountReduced`) rather than an opaque underflow, so the root can be corrected before any user is harmed.
2. **Add an admin `resetUserClaim(address account)` function** (with appropriate access control and event emission) to allow recovery from erroneous roots.
3. **Validate off-chain** that every new root's per-user cumulative amounts are ≥ the previously stored values before calling `setMerkleRoot`.

---
