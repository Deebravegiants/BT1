### Title
Withdrawal Delay Not Initialized in `KernelDepositPool`, Allowing Immediate Withdrawal Bypass - (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary
`KernelDepositPool.initialize()` never sets `withdrawalDelay`, leaving it at the Solidity default of `0`. Because `setWithdrawalDelay()` explicitly rejects `0` as an invalid value, the zero state is clearly unintended. When `maxNumberOfWithdrawalsPerUser` is configured (making withdrawals possible) but `withdrawalDelay` remains `0`, any staker can initiate and immediately claim a withdrawal in the same block, completely bypassing the intended time-lock.

### Finding Description
`KernelDepositPool.initialize()` sets up roles and token addresses but never assigns `withdrawalDelay`:

```solidity
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    // ...
    _setupRole(DEFAULT_ADMIN_ROLE, _admin);
    kernelToken = IERC20(_kernelToken);
    rewardsToken = IERC20(_rewardToken);
    // withdrawalDelay is never set → defaults to 0
}
``` [1](#0-0) 

`setWithdrawalDelay()` explicitly rejects `0`, confirming it is not a valid operational value:

```solidity
function setWithdrawalDelay(uint256 _withdrawalDelay) external onlyRole(DEFAULT_ADMIN_ROLE) {
    if (_withdrawalDelay == 0) revert InvalidWithdrawalDelay();
    ...
}
``` [2](#0-1) 

When `withdrawalDelay == 0`, `initiateWithdrawal` computes:

```solidity
uint256 unlockTime = block.timestamp + withdrawalDelay; // = block.timestamp + 0
``` [3](#0-2) 

And `claimWithdrawal` checks:

```solidity
if (block.timestamp < withdrawal.unlockTime) { revert WithdrawalNotReady(); }
``` [4](#0-3) 

With `unlockTime == block.timestamp`, the condition `block.timestamp < block.timestamp` is always `false`, so the revert is never triggered. The withdrawal is claimable immediately — even within the same block as initiation.

### Impact Explanation
Any staker can bypass the withdrawal delay entirely: stake KERNEL tokens, initiate a withdrawal, and claim it in the same block. The intended lock-up period — which exists to prevent rapid entry/exit gaming of the reward distribution — is completely unenforced. The contract fails to deliver its promised time-lock guarantee. Stakers who rely on the delay for security (e.g., to prevent reward sniping) receive no protection.

**Impact: Low — Contract fails to deliver promised returns, but doesn't lose value.**

### Likelihood Explanation
The likelihood is medium. The contract requires admin to call `setMaxNumberOfWithdrawalsPerUser` before withdrawals are possible (since it also defaults to `0`, causing `WithdrawalLimitReached` otherwise). However, it is a realistic deployment scenario for an admin to configure `maxNumberOfWithdrawalsPerUser` without separately configuring `withdrawalDelay`, since there is no enforcement or ordering requirement between the two. Any staker can then exploit the zero delay immediately. [5](#0-4) 

### Recommendation
Set a non-zero default for `withdrawalDelay` directly inside `initialize()`, consistent with the validation in `setWithdrawalDelay()` that rejects `0`:

```solidity
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    // ...
    withdrawalDelay = 7 days; // or another protocol-appropriate default
}
```

This mirrors the pattern used in `LRTWithdrawalManager.initialize()`, which correctly sets `withdrawalDelayBlocks = 8 days / 12 seconds` at initialization time. [6](#0-5) 

### Proof of Concept
1. Admin deploys `KernelDepositPool` and calls `setMaxNumberOfWithdrawalsPerUser(5)` but does not call `setWithdrawalDelay`.
2. `withdrawalDelay` remains `0`.
3. Attacker calls `stake(amount)` — `balanceOf[attacker] += amount`.
4. Attacker calls `initiateWithdrawal(amount)` — `unlockTime = block.timestamp + 0 = block.timestamp`.
5. In the same transaction (or any subsequent block), attacker calls `claimWithdrawal(withdrawalId)`.
6. Check: `block.timestamp < withdrawal.unlockTime` → `block.timestamp < block.timestamp` → `false` → no revert.
7. Attacker receives KERNEL tokens immediately with zero delay enforced. [7](#0-6)

    function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
        if (_amount == 0) revert AmountZero();
        if (balanceOf[msg.sender] < _amount) revert InsufficientStakedBalance();
        if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();

        balanceOf[msg.sender] -= _amount;
        totalKernelStaked -= _amount;

        // Create a withdrawal record
        uint256 withdrawalId = ++withdrawalCounter;
        uint256 unlockTime = block.timestamp + withdrawalDelay;

        withdrawals[withdrawalId] = Withdrawal({
            user: msg.sender, amount: _amount, unlockTime: unlockTime, claimed: false, withdrawalId: withdrawalId
        });
        userWithdrawalIds[msg.sender].push(withdrawalId);

        emit WithdrawalInitiated(msg.sender, _amount, withdrawalId, unlockTime);
    }
