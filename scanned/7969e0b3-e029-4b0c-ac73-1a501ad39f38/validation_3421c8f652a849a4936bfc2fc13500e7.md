### Title
Stale `rsETHPrice` Used in Deposit and Withdrawal Calculations Without Prior Update - (`contracts/LRTDepositPool.sol`, `contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTDepositPool.getRsETHAmountToMint()` and `LRTWithdrawalManager.getExpectedAssetAmount()` both read `lrtOracle.rsETHPrice()`, which is a **stored state variable** in `LRTOracle` that is only refreshed when `updateRSETHPrice()` is explicitly called. Neither the deposit path nor the withdrawal path calls `updateRSETHPrice()` before using this value. When the stored price is stale, depositors receive more rsETH than they deserve (diluting existing holders' accrued yield), or withdrawers receive more underlying assets than they are entitled to.

---

### Finding Description

`LRTOracle.rsETHPrice` is a `uint256` state variable declared at line 28 of `LRTOracle.sol`. It is only updated by calling `updateRSETHPrice()` (public) or `updateRSETHPriceAsManager()` (manager-only). Between calls, the stored value drifts below the true current value as staking rewards accrue and the protocol's total ETH grows.

**Deposit path** (`LRTDepositPool.sol`):

```
depositETH() / depositAsset()
  → _beforeDeposit()
    → getRsETHAmountToMint()
      → rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()
```

`lrtOracle.rsETHPrice()` is a plain getter for the stale stored variable. No call to `updateRSETHPrice()` precedes this computation.

**Withdrawal path** (`LRTWithdrawalManager.sol`):

```
initiateWithdrawal() / instantWithdrawal()
  → getExpectedAssetAmount()
    → underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset)
```

Again, `lrtOracle.rsETHPrice()` is read directly from storage without a prior update.

**`unlockQueue()` path** (`LRTWithdrawalManager.sol`):

```
unlockQueue()
  → _createUnlockParams()
    → rsETHPrice: lrtOracle.rsETHPrice()   // stale stored value
```

---

### Impact Explanation

**Scenario A — Stale price is lower than actual (most common: rewards have accrued since last update):**

`rsethAmountToMint = (depositAmount × assetPrice) / staleLowerRsETHPrice`

Because the denominator is smaller than it should be, the depositor receives **more rsETH than their deposit warrants**. This dilutes the share value of all existing rsETH holders, effectively transferring accrued yield from existing holders to the new depositor. This constitutes **theft of unclaimed yield** from existing rsETH holders.

**Scenario B — Stale price is higher than actual (price has not yet reflected a decrease):**

`underlyingToReceive = rsETHUnstaked × staleHigherRsETHPrice / assetPrice`

A withdrawer calling `initiateWithdrawal()` or `instantWithdrawal()` receives **more underlying assets than their rsETH entitles them to**, constituting direct fund theft from the protocol.

---

### Likelihood Explanation

`updateRSETHPrice()` is a public function with no access restriction, so keepers or bots can call it. However, there is no on-chain enforcement that it is called atomically before every deposit or withdrawal. The price drifts continuously as staking rewards accrue (~4% APY on LSTs). Additionally, `updateRSETHPrice()` contains a `pricePercentageLimit` guard that **reverts** if the price increase exceeds the configured threshold (unless the caller is a manager). This means that if the price has drifted significantly, the update itself may be blocked for unprivileged callers, creating a window where the stale price is both exploitable and un-updatable by ordinary users. Any depositor can monitor the on-chain `rsETHPrice` versus the computed current price and time their deposit to capture the discrepancy.

---

### Recommendation

Call `updateRSETHPrice()` at the start of `depositETH()`, `depositAsset()`, `initiateWithdrawal()`, and `instantWithdrawal()` before any price-dependent computation. This mirrors the pattern recommended in the Venus M-02 report: update the price of every relevant asset before using it in accounting.

```solidity
// In LRTDepositPool.depositETH() and depositAsset():
ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice();
uint256 rsethAmountToMint = _beforeDeposit(asset, depositAmount, minRSETHAmountExpected);

// In LRTWithdrawalManager.initiateWithdrawal() and instantWithdrawal():
ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice();
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
```

### Proof of Concept

1. Assume staking rewards have accrued since the last `updateRSETHPrice()` call. The true rsETH price is `1.01e18` but `lrtOracle.rsETHPrice()` still returns `1.00e18`.
2. Attacker calls `depositETH{value: 100 ether}(0, "")` on `LRTDepositPool`.
3. `getRsETHAmountToMint(ETH_TOKEN, 100e18)` computes: `(100e18 × 1e18) / 1.00e18 = 100e18` rsETH.
4. The correct amount at the true price would be: `(100e18 × 1e18) / 1.01e18 ≈ 99.01e18` rsETH.
5. The attacker receives `~0.99e18` extra rsETH, capturing yield that belonged to existing holders.
6. Attacker immediately calls `updateRSETHPrice()` (or waits for it to be called), then redeems at the updated price, extracting the difference.
