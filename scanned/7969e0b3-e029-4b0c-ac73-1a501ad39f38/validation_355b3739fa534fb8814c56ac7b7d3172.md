### Title
`_checkAndUpdateDailyFeeMintLimit` Reverts on Zero `maxFeeMintAmountPerDay`, Permanently Blocking `updateRSETHPrice` and Protocol Fee Collection - (File: contracts/LRTOracle.sol)

### Summary

`LRTOracle._checkAndUpdateDailyFeeMintLimit` does not treat `maxFeeMintAmountPerDay == 0` as "no limit set," unlike the sibling view function `remainingDailyFeeMintLimit` which explicitly guards against this. When `maxFeeMintAmountPerDay` is zero (its default state) and the protocol accrues any fee, every call to the public `updateRSETHPrice()` reverts with `DailyFeeMintLimitExceeded`, permanently blocking price updates and protocol fee collection until an admin intervenes.

### Finding Description

`maxFeeMintAmountPerDay` is a storage variable that defaults to `0` after `initialize()` and is not set by `reinitialize()`. It must be explicitly configured via `setMaxFeeMintAmountPerDay` (LRT manager only). The view function `remainingDailyFeeMintLimit` correctly treats `0` as a sentinel meaning "no limit":

```solidity
function remainingDailyFeeMintLimit() external view returns (uint256) {
    if (maxFeeMintAmountPerDay == 0) return 0;   // ← special-case guard
    ...
}
```

However, the internal write path `_checkAndUpdateDailyFeeMintLimit` has no such guard:

```solidity
function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
    if (block.timestamp >= feePeriodStartTime + 1 days) {
        currentPeriodMintedFeeAmount = 0;
        feePeriodStartTime = getCurrentPeriodStartTime();
    }
    // When maxFeeMintAmountPerDay == 0 and feeAmount > 0:
    // 0 + feeAmount > 0  →  always true  →  always reverts
    if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
        revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
    }
    currentPeriodMintedFeeAmount += feeAmount;
}
```

`_checkAndUpdateDailyFeeMintLimit` is called unconditionally from `_updateRsETHPrice` in two branches:

- **Fee branch** (when `protocolFeeInETH > 0`): called with `rsethAmountToMintAsProtocolFee > 0`
- **No-fee branch** (when `protocolFeeInETH == 0`): called with `0` — this branch is safe

The fee branch is reached whenever TVL increases and the protocol is not paused. In that state, with `maxFeeMintAmountPerDay == 0`, the check `feeAmount > 0` is always true, so `_updateRsETHPrice` always reverts, and therefore `updateRSETHPrice()` (public, no access control) always reverts. ### Impact Explanation

**High — Theft of unclaimed yield / permanent freezing of unclaimed yield.**

While `maxFeeMintAmountPerDay == 0` persists (the default state before manager configuration, or if the manager explicitly sets it to 0 to "disable" the limit), every price update that would generate a protocol fee reverts. Protocol fees that should have been minted to the treasury are never collected. Additionally, the rsETH price stored in `rsETHPrice` becomes stale: deposits via `getRsETHAmountToMint` and withdrawals via `getExpectedAssetAmount` both read the stored `lrtOracle.rsETHPrice()`, so users transact at an outdated rate for the entire duration of the blockage. [5](#0-4) [6](#0-5) 

### Likelihood Explanation

**Medium.** `maxFeeMintAmountPerDay` is `0` by default after `initialize()` and is not set by `reinitialize()`. Any deployment window before `setMaxFeeMintAmountPerDay` is called, or any deliberate manager action to set it to `0` (intending "no limit"), triggers the bug. The public `updateRSETHPrice()` can be called by any external actor, so no privilege is needed to observe or trigger the revert. The condition (TVL increasing) is the normal operating state of the protocol. [7](#0-6) [8](#0-7) [9](#0-8) 

### Recommendation

Mirror the guard already present in `remainingDailyFeeMintLimit`. Add an early return when `maxFeeMintAmountPerDay == 0`, treating zero as "no limit enforced":

```solidity
function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
+   if (maxFeeMintAmountPerDay == 0) return; // 0 means no limit; skip enforcement

    if (block.timestamp >= feePeriodStartTime + 1 days) {
        currentPeriodMintedFeeAmount = 0;
        feePeriodStartTime = getCurrentPeriodStartTime();
    }

    if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
        revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
    }

    currentPeriodMintedFeeAmount += feeAmount;
}
```

### Proof of Concept

1. Deploy `LRTOracle` and call `initialize(lrtConfigAddr)`. `maxFeeMintAmountPerDay` is `0`.
2. Do **not** call `setMaxFeeMintAmountPerDay` (or call it with `0`).
3. Allow TVL to increase (e.g., staking rewards accrue in EigenLayer strategies), so `totalETHInProtocol > previousTVL` and `protocolFeeInBPS > 0`.
4. Call `updateRSETHPrice()` from any EOA.
5. Execution reaches `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` where `rsethAmountToMintAsProtocolFee > 0`.
6. The check `0 + rsethAmountToMintAsProtocolFee > 0` is `true`.
7. Transaction reverts with `DailyFeeMintLimitExceeded(rsethAmountToMintAsProtocolFee, 0)`.
8. `rsETHPrice` is never updated; protocol fees are never minted to treasury.
9. All subsequent calls to `updateRSETHPrice()` revert identically until an admin calls `setMaxFeeMintAmountPerDay(nonZeroValue)`. [10](#0-9) [11](#0-10)
