### Title
Uninitialized `maxNumberOfWithdrawalsPerUser` Permanently Blocks All Withdrawals Until Admin Intervention - (File: `contracts/KERNEL/KernelDepositPool.sol`)

### Summary
`KernelDepositPool.initialize()` never sets `maxNumberOfWithdrawalsPerUser`, leaving it at the Solidity default of `0`. The guard in `initiateWithdrawal` evaluates `userWithdrawalIds[msg.sender].length >= 0`, which is always `true`, causing every withdrawal attempt to revert with `WithdrawalLimitReached` until the admin explicitly calls `setMaxNumberOfWithdrawalsPerUser`.

### Finding Description

`maxNumberOfWithdrawalsPerUser` is declared as a plain `uint256` state variable with no default assignment: [1](#0-0) 

`initialize()` sets `kernelToken`, `rewardsToken`, and roles, but never touches `maxNumberOfWithdrawalsPerUser`: [2](#0-1) 

`initiateWithdrawal` checks: [3](#0-2) 

When `maxNumberOfWithdrawalsPerUser == 0`, the expression `userWithdrawalIds[msg.sender].length >= 0` is always `true` (a `uint256` array length can never be negative), so the function unconditionally reverts with `WithdrawalLimitReached` for every caller.

Critically, `setMaxNumberOfWithdrawalsPerUser` explicitly rejects `0` as a valid value: [4](#0-3) 

This means the only way out of the broken state is for the admin to call `setMaxNumberOfWithdrawalsPerUser` with a value in `[1, MAX_WITHDRAWALS_PER_USER]`. There is no on-chain enforcement that this call happens before or during initialization.

Meanwhile, `stake()` has no such guard and accepts deposits freely: [5](#0-4) 

So users can stake KERNEL tokens into a contract that cannot return them until admin action.

### Impact Explanation
Any staker who deposits KERNEL tokens before the admin calls `setMaxNumberOfWithdrawalsPerUser` is unable to initiate a withdrawal. Their principal is locked in the contract with no on-chain exit path. No funds are lost (the contract holds them correctly), but the promised ability to exit a staking position is not delivered. This matches the **Low** scope: *Contract fails to deliver promised returns, but doesn't lose value*.

### Likelihood Explanation
This is the default post-`initialize` state of every fresh deployment. Any deployment that does not atomically call `setMaxNumberOfWithdrawalsPerUser` in the same transaction (or immediately after) exposes this window. Since `stake()` is open immediately after `initialize()`, users can deposit into a non-withdrawable contract. The condition is deterministic and requires no attacker — it is triggered by the normal user flow of calling `initiateWithdrawal`.

### Recommendation
Set a safe default for `maxNumberOfWithdrawalsPerUser` inside `initialize()`, or add a parameter to `initialize()` for it. For example:

```solidity
function initialize(
    address _admin,
    address _kernelToken,
    address _rewardToken,
    uint256 _maxWithdrawalsPerUser   // add this
) external initializer {
    ...
    require(_maxWithdrawalsPerUser > 0 && _maxWithdrawalsPerUser <= MAX_WITHDRAWALS_PER_USER);
    maxNumberOfWithdrawalsPerUser = _maxWithdrawalsPerUser;
}
```

Alternatively, initialize the variable inline at declaration:
```solidity
uint256 public maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER; // = 100
```

```solidity
import "forge-std/Test.sol";
import "../contracts/KERNEL/KernelDepositPool.sol";
import "@openzeppelin/contracts/token/ERC20/ERC20.sol";

contract MockToken is ERC20 {
    constructor() ERC20("TKN", "TKN") { _mint(msg.sender, 1e24); }
}

contract PoC is Test {
    KernelDepositPool pool;
    MockToken kernel;
    MockToken reward;
    address alice = address(0xA11CE);

    function setUp() public {
        kernel = new MockToken();
        reward = new MockToken();

        pool = new KernelDepositPool();
        // initialize WITHOUT calling setMaxNumberOfWithdrawalsPerUser
        pool.initialize(address(this), address(kernel), address(reward));

        kernel.transfer(alice, 1e18);
        vm.prank(alice);
        kernel.approve(address(pool), type(uint256).max);
    }

    function test_withdrawalBlockedByDefault() public {
        // Alice stakes successfully
        vm.prank(alice);
        pool.stake(1e18);

        // Alice cannot initiate withdrawal — maxNumberOfWithdrawalsPerUser == 0
        vm.prank(alice);
        vm.expectRevert(KernelDepositPool.WithdrawalLimitReached.selector);
        pool.initiateWithdrawal(1e18);

        // Confirm the root cause
        assertEq(pool.maxNumberOfWithdrawalsPerUser(), 0);
    }
}
```

Running this test locally against unmodified code will pass, confirming the revert is deterministic whenever `maxNumberOfWithdrawalsPerUser` has not been set.
