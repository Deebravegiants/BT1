### Title
Missing Minimum Return Check in L2 Pool Deposit Functions Exposes Users to Unbounded Slippage - (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolV2.sol)

---

### Summary

The L2 pool deposit functions in `RSETHPoolV3` and `RSETHPoolV2` do not accept a `minRSETHAmountExpected` parameter, leaving users with no slippage protection. The minted wrsETH amount is computed at execution time from a live oracle rate that can shift between when a user previews their deposit and when the transaction is included, causing users to silently receive fewer wrsETH than they expected.

---

### Finding Description

`LRTDepositPool.depositETH` and `LRTDepositPool.depositAsset` on L1 both accept a `minRSETHAmountExpected` parameter and enforce it:

```solidity
// contracts/LRTDepositPool.sol L648–L669
function _beforeDeposit(address asset, uint256 depositAmount, uint256 minRSETHAmountExpected)
    private view returns (uint256 rsethAmountToMint)
{
    ...
    rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
    if (rsethAmountToMint < minRSETHAmountExpected) {
        revert MinimumAmountToReceiveNotMet();
    }
}
``` [1](#0-0) 

The L2 counterparts provide no equivalent protection. Both `deposit` overloads in `RSETHPoolV3` accept only a `referralId` string (or a token address and amount) and immediately mint whatever the oracle returns at that moment:

```solidity

    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);   // no floor check
    ...
}
``` [2](#0-1) 

```solidity
// contracts/pools/RSETHPoolV3.sol L271–L293

    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);

The minted amount is derived from two oracle calls whose values can change between blocks:

```solidity
// contracts/pools/RSETHPoolV3.sol L299–L335
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (...) {
    uint256 rsETHToETHrate = getRate();                          // rsETH oracle
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}

function viewSwapRsETHAmountAndFee(uint256 amount, address token) public view returns (...) {
    uint256 rsETHToETHrate = getRate();                          // rsETH oracle
    uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate(); // token oracle
    rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
}
``` [4](#0-3) 

`RSETHPoolV2.deposit` has the identical gap:

    wrsETH.mint(msg.sender, rsETHAmount);   // no floor check
    ...
}

---

### Impact Explanation

A user who calls `viewSwapRsETHAmountAndFee` to preview their expected wrsETH output and then submits a deposit transaction may receive materially fewer tokens than previewed if the oracle rate moves between the preview call and transaction inclusion. Because rsETH accrues staking rewards continuously, its ETH-denominated price (`getRate()`) rises over time; any upward tick between preview and execution silently reduces the user's wrsETH output with no on-chain protection. The user has no way to express a minimum acceptable output, so the contract fails to deliver the return the user was shown.

**Impact: Low — Contract fails to deliver promised returns, but does not lose value.**

---

### Likelihood Explanation

The rsETH oracle rate is updated permissionlessly via `updateRSETHPrice()` on L1 and propagated to L2 oracles. Rate updates occur regularly as staking rewards accrue. Any deposit submitted during or just after an oracle update will silently receive fewer wrsETH than the user previewed. No privileged actor is required; the rate shift is a normal protocol event. The affected entry points (`deposit` on `RSETHPoolV3` and `RSETHPoolV2`) are the primary user-facing deposit functions on every supported L2.

---

### Recommendation

Add a `minRSETHAmountExpected` parameter to all `deposit` functions in `RSETHPoolV3` and `RSETHPoolV2`, mirroring the pattern already used in `LRTDepositPool._beforeDeposit`:

```solidity
function deposit(string memory referralId, uint256 minRSETHAmountExpected)
    external payable nonReentrant whenNotPaused limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    ...
}
```

Apply the same pattern to the ERC-20 token overload and to `RSETHPoolV2.deposit`.

---

### Proof of Concept

1. User calls `RSETHPoolV3.viewSwapRsETHAmountAndFee(1 ether)` and sees they will receive `X` wrsETH.
2. Before the user's deposit transaction is included, the L2 rsETH oracle rate ticks upward (e.g., from `1.05e18` to `1.06e18`) due to a routine oracle update.
3. User's `deposit{value: 1 ether}(referralId)` executes. `viewSwapRsETHAmountAndFee` now returns `1e18 * (1 ether - fee) / 1.06e18` instead of `/ 1.05e18` — approximately 0.94% fewer wrsETH.
4. `wrsETH.mint(msg.sender, rsETHAmount)` mints the reduced amount with no revert, no event indicating slippage, and no recourse for the user.
5. The L1 `LRTDepositPool` would have reverted this same scenario via `MinimumAmountToReceiveNotMet`; the L2 pools have no equivalent guard.
