### Title
Missing Pending EigenLayer Withdrawal Check in `updateAssetStrategy` Allows Inconsistent Asset Accounting — (File: contracts/LRTConfig.sol)

---

### Summary

`LRTConfig.updateAssetStrategy` validates that no assets are currently staked in the old strategy via `IStrategy.userUnderlyingView`, but does not check for assets that are actively queued for withdrawal from EigenLayer (pending unstaking). This is the direct analog of the SecurityCouncilManager `replaceCohort` bug: a state-replacement function that lacks validation of concurrent in-flight operations, producing inconsistent accounting of protocol TVL and the rsETH price.

---

### Finding Description

When an NDC initiates unstaking via `initiateUnstaking`, EigenLayer's `DelegationManager.queueWithdrawals` removes the shares from the strategy manager immediately. From that point forward, `IStrategy(oldStrategy).userUnderlyingView(ndc)` returns **0**, even though the underlying assets are still locked inside EigenLayer's withdrawal queue awaiting the delay period.

`updateAssetStrategy` guards against migrating a strategy that still holds funds with exactly this call:

```solidity
// contracts/LRTConfig.sol  lines 151-166
if (assetStrategy[asset] != address(0)) {
    address depositPool = getContract(LRTConstants.LRT_DEPOSIT_POOL);
    address[] memory ndcs = ILRTDepositPool(depositPool).getNodeDelegatorQueue();

    uint256 length = ndcs.length;
    for (uint256 i = 0; i < length;) {
        uint256 ndcBalance = IStrategy(assetStrategy[asset]).userUnderlyingView(ndcs[i]);
        if (ndcBalance > 0) {
            revert CannotUpdateStrategyAsItHasFundsNDCFunds(ndcs[i], ndcBalance);
        }
        unchecked { ++i; }
    }
}
assetStrategy[asset] = strategy;          // line 169
``` [1](#0-0) 

Because `userUnderlyingView` returns 0 for queued-but-not-yet-completed withdrawals, the guard passes and the strategy pointer is overwritten while assets are still in transit from the old strategy.

After the pointer is overwritten, `getAssetDistributionData` calls `INodeDelegator(ndc).getAssetBalance(asset)` on every NDC, which internally reads `lrtConfig.assetStrategy(asset)` — now the **new** strategy — and returns 0 for the old strategy's in-flight shares. The `assetUnstakingFromEigenLayer` component (`getAssetUnstaking`) similarly identifies queued withdrawals by strategy; once the config points to the new strategy, the old strategy's queued withdrawals are invisible to the accounting loop. `getTotalAssetDeposits` aggregates all six components: [3](#0-2) 

With the queued withdrawal invisible, `getTotalAssetDeposits` undercounts the true TVL, and `LRTOracle._updateRsETHPrice` computes a depressed rsETH price: [4](#0-3) 

The contrast with the ETH-path is instructive: `_checkResidueEthBalance` explicitly checks `getAssetUnstaking(ETH_TOKEN) > 0` before allowing an NDC to be removed from the queue, demonstrating the protocol is aware of the pending-withdrawal state — but this guard is absent from `updateAssetStrategy` for LST assets. ---

### Impact Explanation

While the old strategy's withdrawal is in-flight, `rsETHPrice` is artificially depressed. Any user who calls `depositAsset` or `depositETH` during this window receives more rsETH than the true exchange rate warrants, diluting all existing rsETH holders. When the queued withdrawal eventually completes and the assets land in the NDC (counted as `assetLyingInNDCs`), the price recovers — but the extra rsETH minted during the window is never reclaimed. This constitutes **theft of unclaimed yield** from existing rsETH holders (High impact).

---

### Likelihood Explanation

The window opens whenever an operator calls `initiateUnstaking` on an NDC and the admin subsequently calls `updateAssetStrategy` before EigenLayer's withdrawal delay expires. Strategy migrations are a planned operational activity; the admin's natural check ("does the old strategy still hold shares?") passes cleanly, making this a realistic good-faith mistake rather than a malicious act. Likelihood is **Low** but non-negligible during any strategy migration event.

---

### Recommendation

Mirror the pattern already used in `_checkResidueEthBalance` and add a check for pending EigenLayer withdrawals inside `updateAssetStrategy`:

```solidity
// After the existing userUnderlyingView check, add:
uint256 pendingUnstaking = INodeDelegator(ndcs[i]).getAssetUnstaking(asset);
if (pendingUnstaking > 0) {
    revert CannotUpdateStrategyWithPendingWithdrawals(ndcs[i], pendingUnstaking);
}
```

This ensures the strategy pointer is only updated once all in-flight withdrawals from the old strategy have fully settled, eliminating the accounting gap.

---

### Proof of Concept

1. NDC holds 100 stETH staked in `strategyA`. `userUnderlyingView(ndc)` = 100 stETH.
2. Operator calls `initiateUnstaking([strategyA], [shares])`. EigenLayer queues the withdrawal; `userUnderlyingView(ndc)` for `strategyA` drops to **0**. `getAssetUnstaking(stETH)` = 100 stETH (tracked via old strategy).
3. Admin calls `updateAssetStrategy(stETH, strategyB)`. The guard at line 158 reads `userUnderlyingView` = 0 → no revert. `assetStrategy[stETH]` is set to `strategyB`.
4. `getAssetBalance(stETH)` on the NDC now queries `strategyB` → 0. `getAssetUnstaking(stETH)` now queries `strategyB`'s queued withdrawals → 0. The 100 stETH in EigenLayer's queue is invisible.
5. `getTotalAssetDeposits(stETH)` undercounts by 100 stETH. `rsETHPrice` is computed lower than reality.
6. An unprivileged depositor calls `depositAsset(stETH, ...)` and receives rsETH at the depressed price, minting more rsETH than the true rate warrants.
7. EigenLayer's delay expires; the 100 stETH lands in the NDC and is counted as `assetLyingInNDCs`. The price recovers — but the excess rsETH minted in step 6 remains outstanding, permanently diluting prior holders. [6](#0-5) [7](#0-6)
