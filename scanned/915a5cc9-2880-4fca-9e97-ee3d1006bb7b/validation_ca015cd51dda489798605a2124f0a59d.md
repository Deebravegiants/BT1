### Title
Price-Limit Bypass via Unchecked Underflow in `buyToken0InBinSpecifiedOut` When `priceLimitX64 < lowerPriceX64` — (`metric-core/contracts/libraries/SwapMath.sol`)

### Summary

When a `buyToken0` exact-output swap crosses a bin boundary and the user's `priceLimitX64` falls strictly between the first bin's upper price and the second bin's lower price, `buyToken0InBinSpecifiedOut` is called with `priceLimitX64 < lowerPriceX64`. Inside an `unchecked` block, `calculateBinPositionAtPrice` computes `(priceLimitX64 - lowerPriceX64) * MAX_POS_BIN / ...`, which underflows to near-`uint256.max`. The subsequent guard `finalBinPosAtPriceLimit < finalBinPos` is always false (since `finalBinPos ≤ type(uint104).max`), so the price-limit clamp is silently skipped and the swap executes at prices entirely above the stated limit.

### Finding Description

**`calculateBinPositionAtPrice` — unchecked underflow** [1](#0-0) 

The NatDoc at line 107 states the function *requires* `priceX64 ∈ [lowerPriceX64, upperPriceX64]`, but this precondition is not enforced. When `priceX64 < lowerPriceX64`, the subtraction `priceX64 - lowerPriceX64` wraps to near-`uint256.max` inside the `unchecked` block.

**`buyToken0InBinSpecifiedOut` — wrong guard, wrong comparison** [2](#0-1) 

The entry condition `priceLimitX64 < upperPriceX64` (line 383) is satisfied even when `priceLimitX64 < lowerPriceX64`. The resulting `finalBinPosAtPriceLimit` is near-`uint256.max`, so `finalBinPosAtPriceLimit < finalBinPos` (line 387) is always false. The clamp is never applied and the swap proceeds through the entire bin.

**Orchestrator — no pre-call guard for non-empty bins** [3](#0-2) 

The only price-limit check before the loop is `priceLimitX64 <= initialPriceX64` (line 888), which uses the price at the *starting* position. When the loop advances to a new bin (line 933: `lowerPriceX64 = upperPriceX64`), the empty-bin guard at line 895 checks `priceLimitX64 <= upperPriceX64`, but for a **non-empty** bin this block is never entered, so `buyToken0InBinSpecifiedOut` is called without any check that `priceLimitX64 >= lowerPriceX64`. [4](#0-3) 

### Impact Explanation

The trader's price limit is a hard execution bound. When it is silently bypassed, the swap executes at prices in `[lowerPriceX64_bin2, upperPriceX64_bin2]`, all of which are strictly above `priceLimitX64`. The trader pays more token1 than their stated limit permits — a direct loss of user principal. This satisfies the "bad-price execution: unclamped quote reaches a pool swap" impact gate.

### Likelihood Explanation

The condition is reachable by any unprivileged caller of the public `swap` function with `zeroForOne = false`, `amountSpecified < 0` (exact output of token0), and a `priceLimitX64` that falls between two adjacent bin price ranges. No special pool state, trusted role, or non-standard token is required. It occurs naturally whenever the requested output exceeds the first bin's token0 balance.

### Recommendation

**Option A — Guard in the orchestrator**: Before calling `buyToken0InBinSpecifiedOut` on a non-empty bin, add:
```solidity
if (params.priceLimitX64 <= lowerPriceX64) break;
```
This mirrors the existing empty-bin guard and stops the loop before entering a bin whose entire price range is above the limit.

**Option B — Guard inside `buyToken0InBinSpecifiedOut`**: Replace the entry condition at line 383 with:
```solidity
if (priceLimitX64 > lowerPriceX64 && priceLimitX64 < upperPriceX64) {
```
This ensures `calculateBinPositionAtPrice` is only called when `priceLimitX64` is strictly within the bin's range, matching the function's documented precondition.

The symmetric function `buyToken1InBinSpecifiedOut` (line 469) uses `lowerPriceX64 < priceLimitX64` as its guard, which correctly excludes the out-of-range case for the sell direction. The buy direction should be made consistent.
