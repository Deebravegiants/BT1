### Title
Unpermissioned `addLiquidity` Allows Anyone to Inject Dust Shares Into Any Position, Permanently Disrupting the Owner's `removeLiquidity` Exit — (`metric-core/contracts/MetricOmmPool.sol`, `metric-core/contracts/libraries/LiquidityLib.sol`)

---

### Summary

`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address with no check that `msg.sender == owner`. Any caller who implements `IMetricOmmModifyLiquidityCallback` can credit LP shares to a victim's position while paying the required tokens themselves. Because `LiquidityLib.removeLiquidity` enforces a `minimalMintableLiquidity` floor on the *remaining* share count, an attacker who front-runs a victim's "remove-all" transaction with a single dust-share injection causes the victim's transaction to revert with `MinimalLiquidity`, locking the victim's principal in the pool until they discover and correct the share count.

---

### Finding Description

**Asymmetric access control between `addLiquidity` and `removeLiquidity`**

`removeLiquidity` enforces `msg.sender == owner`: [1](#0-0) 

`addLiquidity` has no equivalent guard — any external caller may supply an arbitrary `owner`: [2](#0-1) 

Inside `LiquidityLib.addLiquidity`, the callback is dispatched to `msg.sender` (the attacker's contract), not to `owner`. The attacker pays the tokens; the shares are credited to the victim: [3](#0-2) 

**The `minimalMintableLiquidity` floor in `removeLiquidity` is the trigger**

When a victim tries to remove all `X` shares, `newUserShares` becomes 0, which passes the floor check. But if the attacker has injected even 1 share beforehand, the victim's call to remove `X` shares leaves `1` share — a non-zero value below `minimalMintableLiquidity` — and the call reverts: [4](#0-3) 

**Attack flow**

1. Victim holds `X` shares (where `X ≥ minimalMintableLiquidity`) and broadcasts `removeLiquidity(owner, salt, X shares)`.
2. Attacker deploys a contract implementing `metricOmmModifyLiquidityCallback`, front-runs with `pool.addLiquidity(victim, salt, 1 share, ...)`, and pays the required token amount in the callback.
3. Victim's position now holds `X + 1` shares.
4. Victim's `removeLiquidity(X shares)` leaves `1` share. Since `1 < minimalMintableLiquidity`, the call reverts with `MinimalLiquidity`.
5. Victim's principal remains locked until they discover the injected share and resubmit with the corrected total.

The attacker can repeat this every time the victim resubmits, as long as they are willing to keep spending tokens (which accrue to the victim's position, not the attacker — making this a pure griefing vector with no attacker profit).

---

### Impact Explanation

The victim's `removeLiquidity` flow is broken: a "remove all" exit — the most common LP withdrawal pattern — can be persistently blocked by an unprivileged third party at the cost of a dust token deposit per block. The victim's principal is not stolen but is effectively frozen until they detect the injected shares and resubmit with the exact current total. This satisfies the impact gate criterion of **broken core pool functionality causing unusable withdraw/liquidity flows**.

---

### Likelihood Explanation

- `addLiquidity` is a public, permissionless function with no factory or role gate.
- The attacker only needs to deploy a trivial callback contract and hold a dust amount of the pool's tokens.
- The attack is repeatable: every corrected victim transaction can be front-run again.
- The victim has no on-chain mechanism to prevent share injection into their position.

---

### Recommendation

Add a caller-identity check to `addLiquidity` symmetric with the one already present in `removeLiquidity`:

```solidity
// In MetricOmmPool.addLiquidity, before calling LiquidityLib:
if (msg.sender != owner) revert NotPositionOwner();
```

If the protocol intends to allow approved routers (e.g., `MetricOmmPoolLiquidityAdder`) to add liquidity on behalf of users, introduce an explicit allowlist or a per-owner approval mapping rather than leaving the function fully open. The periphery adder already validates `owner != address(0)` but does not prevent arbitrary third parties from calling the pool directly. [5](#0-4)

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

import {IMetricOmmPoolActions} from "metric-core/interfaces/IMetricOmmPool/IMetricOmmPoolActions.sol";
import {IMetricOmmModifyLiquidityCallback} from "metric-core/interfaces/callbacks/IMetricOmmModifyLiquidityCallback.sol";
import {LiquidityDelta} from "metric-core/types/PoolOperation.sol";
import {IERC20} from "@openzeppelin/contracts/token/ERC20/IERC20.sol";

contract DustInjector is IMetricOmmModifyLiquidityCallback {
    address immutable pool;
    address immutable token0;
    address immutable token1;

    constructor(address _pool, address _token0, address _token1) {
        pool = _pool;
        token0 = _token0;
        token1 = _token1;
        IERC20(_token0).approve(_pool, type(uint256).max);
        IERC20(_token1).approve(_pool, type(uint256).max);
    }

    /// @notice Inject 1 share into victim's position to break their remove-all exit.
    function attack(address victim, uint80 salt, int256 binIdx) external {
        int256[] memory bins = new int256[](1);
        bins[0] = binIdx;
        uint256[] memory shares = new uint256[](1);
        shares[0] = 1; // dust injection
        LiquidityDelta memory delta = LiquidityDelta({binIdxs: bins, shares: shares});
        // No msg.sender == owner check in addLiquidity — succeeds for any victim address.
        IMetricOmmPoolActions(pool).addLiquidity(victim, salt, delta, "", "");
    }

    function metricOmmModifyLiquidityCallback(
        uint256 amount0Delta,
        uint256 amount1Delta,
        bytes calldata
    ) external override {
        // Pay whatever the pool requests; tokens go to victim's position.
        if (amount0Delta > 0) IERC20(token0).transfer(msg.sender, amount0Delta);
        if (amount1Delta > 0) IERC20(token1).transfer(msg.sender, amount1Delta);
    }
}
```

**Expected result:** After `attack()` is mined before the victim's `removeLiquidity(X shares)`, the victim's call reverts with `MinimalLiquidity(1, minimalMintableLiquidity)`. The victim must discover the injected share and resubmit with `X + 1` shares to exit cleanly.
