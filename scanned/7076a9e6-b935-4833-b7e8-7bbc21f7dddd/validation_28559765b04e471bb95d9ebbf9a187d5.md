### Title
Stale `rsETHPrice` Allows Depositors to Extract Yield from Existing rsETH Holders During stETH Rebase Windows — (File: `contracts/LRTOracle.sol`)

---

### Summary

`LRTOracle` stores `rsETHPrice` as a state variable that is only updated when `updateRSETHPrice()` is explicitly called. Because stETH is a rebase token whose on-chain balance increases every ~24 hours, a window exists between each rebase event and the next price update during which `rsETHPrice` is stale-low. Any depositor who calls `depositAsset(stETH, ...)` inside this window receives more rsETH than the true exchange rate warrants, diluting the yield that should accrue to existing rsETH holders.

---

### Finding Description

`LRTOracle._updateRsETHPrice()` computes the new price as:

```
newRsETHPrice = totalETHInProtocol / rsethSupply
```

where `totalETHInProtocol` is derived from live on-chain balances via `_getTotalEthInProtocol()` → `ILRTDepositPool.getTotalAssetDeposits(asset)` → `IERC20(asset).balanceOf(...)`. [1](#0-0) 

stETH is a rebase token: its `balanceOf` increases automatically every ~24 hours without any transaction. After a rebase, `totalETHInProtocol` is immediately higher, but `rsETHPrice` (a stored state variable) remains at its last written value until `updateRSETHPrice()` is called. [2](#0-1) 

`LRTDepositPool.getRsETHAmountToMint()` uses the stale stored price directly:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [3](#0-2) 

Because `getAssetPrice(stETH)` is a live Chainlink feed (stETH/ETH ≈ 1.0, unaffected by rebase) while `rsETHPrice` is stale-low, the denominator is smaller than the true price, so the depositor receives more rsETH than they are entitled to.

When `updateRSETHPrice()` is subsequently called, `rsethSupply` is now inflated by the attacker's extra rsETH. The fee and price computation uses:

```solidity
uint256 previousTVL = rsethSupply.mulWad(rsETHPrice);   // inflated supply × stale price
uint256 rewardAmount = totalETHInProtocol - previousTVL; // smaller reward
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply); // diluted
``` [4](#0-3) 

The resulting `newRsETHPrice` is lower than it would have been without the attacker's deposit. Existing holders' rsETH is worth less ETH than it should be — the rebase yield has been partially transferred to the attacker.

---

### Impact Explanation

**High — Theft of unclaimed yield.**

Existing rsETH holders are entitled to the full benefit of each stETH rebase. A depositor who times their deposit into the stale-price window extracts a portion of that yield. The stolen amount scales with the ratio of the attacker's deposit to the existing TVL. For a 0.011% daily stETH rebase and an attacker depositing 10% of TVL, approximately 10% of the daily rebase yield is redirected to the attacker. At protocol scale (hundreds of millions of TVL), this is material.

---

### Likelihood Explanation

**Low–Medium.** stETH rebases every ~24 hours, creating a recurring window. `updateRSETHPrice()` is public and callable by anyone, so the window closes as soon as any party calls it. However:

- The window is predictable (rebase timing is known).
- A sophisticated attacker can monitor the mempool and front-run the price update transaction.
- The `pricePercentageLimit` check only blocks callers who are not managers when the price increase exceeds the threshold; for normal daily rebases the increase is well below any reasonable limit, so the public `updateRSETHPrice()` path is unblocked. [5](#0-4) 
- The withdrawal delay (`withdrawalDelayBlocks`) does not prevent the dilution — the damage to existing holders occurs at the moment of the inflated mint, not at withdrawal.

---

### Recommendation

1. **Pull-based price freshness**: Before minting rsETH in `_beforeDeposit`, call `updateRSETHPrice()` (or a view equivalent that computes the live price) so that the price used for minting always reflects the current on-chain TVL.
2. **Alternatively**, compute `rsethAmountToMint` using a live TVL snapshot rather than the stored `rsETHPrice`, eliminating the staleness window entirely.
3. **At minimum**, document the staleness window and enforce a maximum age on `rsETHPrice` (e.g., revert deposits if the price has not been updated within the last N blocks).

---

### Proof of Concept

1. At block B, stETH rebases: the protocol's stETH balance increases by δ, so `totalETHInProtocol` increases by δ × `getAssetPrice(stETH)`. `rsETHPrice` is unchanged.
2. Attacker calls `depositAsset(stETH, D, 0, "")` at block B+1 before any price update. They receive `D × getAssetPrice(stETH) / rsETHPrice_stale` rsETH — more than the true-price amount `D × getAssetPrice(stETH) / rsETHPrice_true`. [3](#0-2) 
3. At block B+2, anyone calls `updateRSETHPrice()`. The new price is computed over the now-inflated `rsethSupply`, yielding a lower `newRsETHPrice` than would have resulted without step 2. [6](#0-5) 
4. Existing holders' rsETH is now worth less ETH than it should be. The difference — proportional to the attacker's deposit share — has been transferred to the attacker as excess rsETH.
5. After `withdrawalDelayBlocks`, the attacker redeems their rsETH via `LRTWithdrawalManager`, receiving more stETH than they deposited. [7](#0-6)

    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
