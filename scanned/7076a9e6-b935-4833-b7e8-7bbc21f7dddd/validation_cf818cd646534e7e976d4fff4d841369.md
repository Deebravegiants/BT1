### Title
Reward Dust Permanently Frozen in `notifyRewardAmount` Due to Integer Division Truncation - (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary

`KernelDepositPool.notifyRewardAmount` computes `rewardRate` via integer division, silently discarding the remainder (`receivedAmount % duration`). These truncated tokens are transferred into the contract but are never included in any future `rewardRate` computation, making them permanently irrecoverable by stakers.

### Finding Description

In `notifyRewardAmount`, the reward rate is set as:

```solidity
// Line 580 (new period)
rewardRate = receivedAmount / duration;

// Line 583 (extending active period)
rewardRate = (receivedAmount + remaining) / duration;
```

The remainder — `receivedAmount % duration` in the first branch, or `(receivedAmount + remaining) % duration` in the second — is silently dropped. These tokens sit in the contract's `rewardsToken` balance but are never re-incorporated into any future `rewardRate` calculation.

The reason they cannot be recovered is that `receivedAmount` is computed as a balance delta from the transfer-in pattern:

```solidity
uint256 balanceBefore = rewardsToken.balanceOf(address(this));
rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
uint256 balanceAfter = rewardsToken.balanceOf(address(this));
uint256 receivedAmount = balanceAfter - balanceBefore;
```

Only the newly transferred tokens are counted. The dust already sitting in the contract from prior truncations is invisible to this calculation and is never rolled into a subsequent `rewardRate`. There is no admin sweep function or recovery path for these tokens.

The contract's own NatSpec at lines 18–22 acknowledges the `totalKernelStaked == 0` lock-up risk but makes no mention of the truncation dust, confirming it is an unintended loss. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

Every call to `notifyRewardAmount` permanently freezes up to `duration - 1` wei of reward tokens. For a 7-day duration (`604800` seconds), the maximum dust per call is `604799` wei. Over many reward periods this accumulates monotonically and is irrecoverable. Stakers who call `getReward` receive less than the amount deposited by the admin. This matches **"Permanent freezing of unclaimed yield"** (Medium) or at minimum **"Contract fails to deliver promised returns"** (Low). [4](#0-3) 

### Likelihood Explanation

This occurs deterministically on every invocation of `notifyRewardAmount` unless `receivedAmount` is an exact multiple of `duration`. In practice, reward amounts are not chosen to be exact multiples of the duration in seconds, so truncation happens on virtually every call. Likelihood is **High**. [5](#0-4) 

### Recommendation

After computing `rewardRate`, calculate the dust and carry it forward into the next period's accounting, or transfer it to a designated treasury address. For example:

```solidity
uint256 newRewardRate = totalForPeriod / duration;
uint256 dust = totalForPeriod - (newRewardRate * duration);
if (dust > 0) {
    rewardsToken.safeTransfer(protocolTreasury, dust);
}
rewardRate = newRewardRate;
```

This mirrors the fix applied in the referenced CoreDAO commit, which redirected dust to a system reward contract rather than leaving it stranded.

### Proof of Concept

1. Admin sets `duration = 604800` (7 days).
2. Admin calls `notifyRewardAmount(1_000_001)`.
3. `rewardRate = 1_000_001 / 604800 = 1` (integer division).
4. Total distributed over the period = `1 * 604800 = 604800` tokens.
5. Dust = `1_000_001 - 604800 = 395201` tokens are permanently stuck in the contract.
6. Stakers call `getReward` after the period ends and collectively receive only `604800` tokens, not `1_000_001`.
7. The `395201` dust tokens remain in `rewardsToken.balanceOf(address(this))` forever, as no subsequent `notifyRewardAmount` call includes them in `receivedAmount`. [6](#0-5) [7](#0-6)

    function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
        if (_amount == 0) revert AmountZero();

        // Prevent starting a reward period when no tokens are staked to avoid unallocated rewards
        if (totalKernelStaked == 0) revert NoStakedTokens();

        // Transfer reward tokens into the contract
        uint256 balanceBefore = rewardsToken.balanceOf(address(this));
        rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
        uint256 balanceAfter = rewardsToken.balanceOf(address(this));
        // Calculate the actual amount of tokens received in case of a transfer fee (tax)
        uint256 receivedAmount = balanceAfter - balanceBefore;

        if (block.timestamp >= finishAt) {
            rewardRate = receivedAmount / duration;
        } else {
            uint256 remaining = (finishAt - block.timestamp) * rewardRate;
            rewardRate = (receivedAmount + remaining) / duration;
        }

        if (rewardRate == 0) revert RewardRateZero();

        finishAt = block.timestamp + duration;
        updatedAt = block.timestamp;

        emit NotifyRewardAmount(receivedAmount, finishAt);
    }
