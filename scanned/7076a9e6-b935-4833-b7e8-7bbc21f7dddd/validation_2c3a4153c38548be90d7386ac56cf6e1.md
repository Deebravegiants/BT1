### Title
No Slippage Protection in `deposit()` Functions Allows Users to Receive Fewer rsETH Than Expected - (File: contracts/pools/RSETHPoolV2NBA.sol, contracts/pools/RSETHPoolV2.sol, contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPoolV3WithNativeChainBridge.sol, contracts/pools/RSETHPool.sol, contracts/pools/RSETHPoolNoWrapper.sol, contracts/pools/RSETHPoolV2ExternalBridge.sol)

---

### Summary

Every public `deposit()` function across all pool contracts converts a user-supplied ETH or ERC-20 token amount into rsETH/wrsETH using a live oracle rate fetched at execution time. None of these functions accept a `minAmountReceived` parameter or a deadline. If a transaction remains pending in the mempool while the oracle rate changes, the user receives fewer rsETH tokens than they expected at submission time, with no on-chain mechanism to reject the trade.

---

### Finding Description

All pool contracts share the same pattern. The ETH deposit path in `RSETHPoolV2NBA.sol` is the simplest illustration:

```solidity
function deposit(string memory referralId) external payable nonReentrant whenNotPaused {
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();

    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);

    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);

    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
``` [1](#0-0) 

The output amount is computed entirely from the live oracle rate:

```solidity
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
``` [2](#0-1) 

The same pattern is present in every other pool contract:

- `RSETHPoolV2.sol` `deposit(string)` [3](#0-2) 
- `RSETHPoolV3.sol` `deposit(string)` and `deposit(address,uint256,string)` [4](#0-3) 
- `RSETHPool.sol` `deposit(string)` and `deposit(address,uint256,string)` [5](#0-4) 
- `RSETHPoolNoWrapper.sol` `deposit(string)` and `deposit(address,uint256,string)` [6](#0-5) 

For token deposits, the output depends on two oracle rates simultaneously — the token/ETH rate and the rsETH/ETH rate — amplifying the exposure:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [7](#0-6) 

Neither the ETH path nor the token path accepts a `minAmountReceived` or a deadline argument. There is no on-chain guard that allows the user to express "I want at least X rsETH or revert."

---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

For ETH deposits: rsETH accrues staking rewards monotonically, so its ETH-denominated rate only increases over time. A transaction delayed in the mempool will execute at a higher `rsETHToETHrate`, yielding fewer rsETH tokens than the user observed off-chain. The ETH-equivalent value is approximately preserved, but the user receives a smaller rsETH balance than promised by the off-chain quote.

For token deposits (e.g., wstETH): the token oracle rate can move in either direction. If the token's ETH price falls while the transaction is pending, the user receives fewer rsETH tokens and the shortfall represents a real reduction in value relative to the user's intent at submission time.

In both cases the contract silently executes at a worse rate than the user expected, with no revert path available to the user.

---

### Likelihood Explanation

L2 chains targeted by these pools (Arbitrum, Unichain, and others) can experience mempool congestion or gas price spikes that delay transactions. The rsETH oracle rate updates with every staking reward accrual. Any non-trivial delay between transaction submission and inclusion is sufficient to produce a worse-than-expected output. No privileged access or external compromise is required; any ordinary depositor is exposed on every call.

---

### Recommendation

Add a `minAmountReceived` parameter to every `deposit()` overload and revert if the computed output falls below it:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmount)
    external payable nonReentrant whenNotPaused
{
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();

    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);

+   if (rsETHAmount < minRsETHAmount) revert SlippageExceeded();

    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
```

Apply the same guard to all token deposit overloads across all pool contracts. Optionally add a `deadline` parameter to guard against indefinite mempool delays.

---

### Proof of Concept

1. rsETH/ETH oracle rate is `1.05e18` (1 rsETH = 1.05 ETH). User calls `deposit{value: 1 ether}("ref")` on `RSETHPoolV2NBA` expecting `~0.952 wrsETH` (ignoring fees).
2. Network congestion delays the transaction in the mempool for several hours.
3. Staking rewards accrue; oracle is updated to `1.06e18`.
4. Transaction executes: `rsETHAmount = 1e18 * 1e18 / 1.06e18 ≈ 0.943e18`.
5. User receives `0.943 wrsETH` instead of the `0.952 wrsETH` they expected — a ~0.9% shortfall with no revert path available.

The same scenario applies to token deposits in `RSETHPoolV3.sol`, `RSETHPool.sol`, `RSETHPoolNoWrapper.sol`, `RSETHPoolV3ExternalBridge.sol`, and `RSETHPoolV3WithNativeChainBridge.sol`, where two oracle rates are involved and the shortfall can be larger.
