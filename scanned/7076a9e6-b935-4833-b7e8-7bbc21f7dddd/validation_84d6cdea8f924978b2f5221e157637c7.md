### Title
Missing Minimum Output Amount Protection in L2 Pool Deposit Functions - (File: contracts/pools/RSETHPoolV3ExternalBridge.sol)

### Summary
The `deposit()` functions across all L2 pool variants lack a `minRsETHAmountExpected` parameter, leaving depositors with no slippage protection when exchanging ETH or LSTs for wrsETH/rsETH. This is the direct analog of the GMX `minMarketTokens = 0` pattern: the protocol's own L1 deposit path (`LRTDepositPool`) enforces a minimum output, but the L2 pool entry points do not.

### Finding Description
Every L2 pool deposit function accepts only the asset amount and a referral string; no minimum output amount is accepted or enforced:

- `RSETHPool.deposit(string referralId)` / `deposit(address token, uint256 amount, string referralId)`
- `RSETHPoolV3.deposit(string referralId)` / `deposit(address token, uint256 amount, string referralId)`
- `RSETHPoolNoWrapper.deposit(string referralId)` / `deposit(address token, uint256 amount, string referralId)`
- `RSETHPoolV3ExternalBridge.deposit(string referralId)` / `deposit(address token, uint256 amount, string referralId)`

The rsETH/wrsETH amount minted is computed at execution time via `viewSwapRsETHAmountAndFee`, which reads the pushed oracle rate:

```solidity
// RSETHPoolV3ExternalBridge.sol
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
wrsETH.mint(msg.sender, rsETHAmount);
```

The oracle rate (`rsETHOracle`) is a cross-chain pushed rate from L1. It can be updated at any time by the rate provider. If the rate is updated (rsETH appreciates) between when the user previews the swap off-chain and when the transaction is mined, the user receives fewer wrsETH than expected with no on-chain recourse.

By contrast, the L1 `LRTDepositPool` correctly enforces a minimum:

```solidity
// LRTDepositPool.sol
function depositETH(uint256 minRSETHAmountExpected, string calldata referralId) external payable ...
function depositAsset(address asset, uint256 depositAmount, uint256 minRSETHAmountExpected, ...) external ...
```

and reverts if `rsethAmountToMint < minRSETHAmountExpected`.

### Impact Explanation
A depositor who previews the swap rate and submits a transaction can receive materially fewer wrsETH/rsETH tokens than the previewed amount if the oracle rate is updated before their transaction is included. The user's ETH or LST is consumed but the output is silently reduced. This matches the allowed impact class: **contract fails to deliver promised returns, but doesn't lose value** (Low).

### Likelihood Explanation
The rsETH exchange rate is pushed from L1 to L2 periodically. Any rate update that occurs in the same block or between a user's preview call and their transaction inclusion causes a silent shortfall. This is a normal operational event, not an exceptional one, making it a realistic occurrence for active depositors.

### Recommendation
Add a `minRsETHAmountExpected` parameter to all `deposit()` overloads in every L2 pool contract, mirroring the pattern already used in `LRTDepositPool`:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountExpected)
    external payable nonReentrant whenNotPaused limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountExpected) revert InsufficientOutputAmount();
    wrsETH.mint(msg.sender, rsETHAmount);
    ...
}
```

### Proof of Concept
1. User calls `viewSwapRsETHAmountAndFee(1 ether)` off-chain and observes they will receive `X` wrsETH at the current oracle rate `R`.
2. The L1 rate provider pushes a new, higher rate `R'` (rsETH has appreciated) to the L2 oracle before the user's transaction is mined.
3. User's `deposit{value: 1 ether}("ref")` executes; `viewSwapRsETHAmountAndFee` now uses `R'`, yielding `X' < X` wrsETH.
4. `wrsETH.mint(msg.sender, X')` is called — the user receives fewer tokens than previewed with no revert or warning.
5. Because no minimum is checked, the transaction succeeds silently. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7)
