### Title
Missing Minimum Output Check in Pool `deposit` Functions Exposes Depositors to Oracle Rate Slippage - (File: contracts/pools/RSETHPoolV2.sol, RSETHPoolV3.sol, RSETHPoolV3ExternalBridge.sol, RSETHPoolV3WithNativeChainBridge.sol, RSETHPoolNoWrapper.sol, RSETHPoolV2NBA.sol, RSETHPool.sol)

---

### Summary

Every L2 pool `deposit` function computes the wrsETH/rsETH output amount by reading `getRate()` from an oracle at execution time, but accepts no `minRsETHAmountExpected` parameter. If the oracle rate rises between the moment a user simulates the transaction and the moment it is mined, the user receives fewer tokens than expected with no on-chain protection. By contrast, `LRTDepositPool.depositETH` and `depositAsset` on L1 both accept and enforce a `minRSETHAmountExpected` argument, demonstrating that the protocol is aware of this protection and intentionally provides it on L1 but omits it across all L2 pool variants.

---

### Finding Description

All L2 pool deposit entry points share the same pattern:

```solidity
// RSETHPoolV2.sol – representative example
function deposit(string memory referralId)
    external payable nonReentrant whenNotPaused limitDailyMint(msg.value)
{
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();

    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
```

`viewSwapRsETHAmountAndFee` divides the deposited ETH by the live oracle rate:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

The oracle rate (`rsETHOracle.getRate()`) can change between the block in which the user simulates the call and the block in which the transaction is included. On chains where the oracle is backed by `LRTOracle`, `updateRSETHPrice()` is a **public, permissionless** function:

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

An attacker can front-run a pending deposit by calling `updateRSETHPrice()` in the same block, causing the rsETH price to jump to its true current value (reflecting accrued staking rewards). The depositor's transaction then executes at the higher rate and receives fewer wrsETH tokens than the simulation showed, with no on-chain guard to revert the transaction.

The same gap exists in every pool variant:

| Contract | Affected function(s) |
|---|---|
| `RSETHPoolV2` | `deposit(string)` |
| `RSETHPoolV2NBA` | `deposit(string)` |
| `RSETHPoolV3` | `deposit(string)`, `deposit(address,uint256,string)` |
| `RSETHPoolV3ExternalBridge` | `deposit(string)`, `deposit(address,uint256,string)` |

| `RSETHPool` | `deposit(string)`, `deposit(address,uint256,string)` |

### Impact Explanation

**Low. Contract fails to deliver promised returns, but doesn't lose value.**

A user who previews the deposit with `viewSwapRsETHAmountAndFee` and then submits the transaction may receive materially fewer wrsETH tokens than the preview showed. Because each wrsETH token is worth proportionally more at the higher rate, the ETH-denominated value received is approximately preserved, but the user does not receive the token quantity they expected and planned for. This is the same class of harm described in the reference report: the depositor receives a smaller *value* (token count) than promised, not a smaller number of a fixed-value token.

---

### Likelihood Explanation

**Medium.**

- `LRTOracle.updateRSETHPrice()` is public and callable by any EOA or contract without any role requirement, making it trivially available as a front-running tool.
- Staking rewards accrue continuously, so the true rsETH/ETH rate drifts above the last stored oracle value between updates. Any attacker who monitors the mempool for large deposits can profitably trigger an oracle update immediately before the deposit lands.
- On chains where the oracle is a cross-chain rate receiver, the rate can be updated by the cross-chain messaging layer at any time, creating the same window without requiring attacker action.
- The `LRTDepositPool` on L1 already enforces `minRSETHAmountExpected`, confirming the protocol recognises this risk; the omission in all L2 pools is therefore a consistent gap rather than an intentional design choice.

---

### Recommendation

Add a `minRsETHAmountExpected` parameter to every `deposit` function in all pool contracts and revert if the computed output falls below it, mirroring the existing protection in `LRTDepositPool._beforeDeposit`:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountExpected)
    external payable nonReentrant whenNotPaused limitDailyMint(msg.value)
{
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();

    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountExpected) revert MinimumAmountToReceiveNotMet();

    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
```

### Proof of Concept

1. Alice calls `viewSwapRsETHAmountAndFee(1 ether)` on `RSETHPoolV2` and sees she will receive `950_000_000_000_000_000` wrsETH (rate = 1.052 ETH/rsETH).
2. Alice submits `deposit{value: 1 ether}("ref")`.
3. Bob (attacker) sees Alice's pending transaction in the mempool and front-runs it by calling `LRTOracle.updateRSETHPrice()` (public, no role required). Staking rewards have accrued since the last update; the new rate is 1.060 ETH/rsETH.
4. Alice's transaction executes. `getRate()` now returns `1_060_000_000_000_000_000`.
5. Alice receives `1e18 * 1e18 / 1.060e18 ≈ 943_396_226_415_094_339` wrsETH — roughly 0.7% fewer tokens than the simulation showed.
6. There is no `minRsETHAmountExpected` check; the transaction succeeds silently.
