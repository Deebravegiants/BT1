### Title
Incorrect `feeBps` Upper-Bound Constant in `setFeeBps` Allows Fees Up to 100% Instead of Intended 10% - (File: contracts/pools/RSETHPoolV3ExternalBridge.sol)

---

### Summary
`RSETHPoolV3ExternalBridge.setFeeBps` uses an incorrect cap of `10_000` (100%) instead of `1_000` (10%), which is the consistent limit enforced by every other pool contract in the codebase. This allows the admin to set a fee far above the protocol's intended maximum, causing depositors to receive materially fewer (or zero) wrsETH tokens.

---

### Finding Description
In `RSETHPoolV3ExternalBridge.sol`, the fee-setter function validates the new fee with:

```solidity
if (_feeBps > 10_000) revert InvalidFeeAmount();
``` [1](#0-0) 

Every other pool contract in the repository caps `feeBps` at `1_000` (10%):

- `RSETHPoolV3.sol` line 519: `if (_feeBps > 1000) revert InvalidFeeAmount();` [2](#0-1) 

- `RSETHPoolV2ExternalBridge.sol` line 532: `if (_feeBps > 1000) revert InvalidFeeAmount();` [3](#0-2) 

- `RSETHPoolV3WithNativeChainBridge.sol` line 582: `if (_feeBps > 1000) revert InvalidFeeAmount();` [4](#0-3) 

The fee is applied directly to every user deposit in `viewSwapRsETHAmountAndFee`:

```solidity
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [5](#0-4) 

At `feeBps = 10_000`, `amountAfterFee = 0` and every depositor receives 0 wrsETH while their full ETH is recorded as protocol fees.

---

### Impact Explanation
Any `feeBps` value above `1_000` causes depositors to receive fewer wrsETH tokens than the protocol's intended 10% maximum fee permits. At the extreme (`feeBps = 10_000`), depositors receive **zero** wrsETH for their ETH — their funds are entirely captured as fees and held in the contract. This is a direct breach of the contract's promised exchange rate and constitutes **theft of unclaimed yield / contract fails to deliver promised returns**.

**Impact: Low — Contract fails to deliver promised returns, but doesn't lose value** (funds remain in the contract as fees, recoverable by the fee-withdrawal path, but depositors are systematically shortchanged).

---

### Likelihood Explanation
The admin of `RSETHPoolV3ExternalBridge` may set a fee of, e.g., `2_000` (20%) in good faith, believing the code's own cap of `10_000` represents the protocol maximum — since the code itself does not revert. The inconsistency across contracts makes an accidental misconfiguration realistic. No malicious intent is required; the incorrect constant silently permits out-of-spec values.

---

### Recommendation
Change the cap in `RSETHPoolV3ExternalBridge.setFeeBps` from `10_000` to `1_000` to match all other pool contracts:

```solidity
// Before (incorrect)
if (_feeBps > 10_000) revert InvalidFeeAmount();

// After (correct, consistent with all other pools)
if (_feeBps > 1_000) revert InvalidFeeAmount();
``` [1](#0-0) 

---

### Proof of Concept

1. Admin calls `setFeeBps(5_000)` on `RSETHPoolV3ExternalBridge` — the call succeeds because `5_000 ≤ 10_000`.
2. User calls `deposit("ref")` with `1 ETH`.
3. `viewSwapRsETHAmountAndFee(1e18)` computes:
   - `fee = 1e18 * 5_000 / 10_000 = 0.5e18`
   - `amountAfterFee = 0.5e18`
   - `rsETHAmount = 0.5e18 * 1e18 / rsETHToETHrate` — user receives wrsETH worth only **0.5 ETH** instead of the intended ≥0.9 ETH (at the 10% cap).
4. The same call on any other pool contract (`RSETHPoolV3`, `RSETHPoolV2ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`) would revert with `InvalidFeeAmount` because `5_000 > 1_000`.

The root cause is the single incorrect literal `10_000` at line 745 of `RSETHPoolV3ExternalBridge.sol`, directly analogous to the wrong hardcoded constant in the referenced `KinTokenSale` report.
