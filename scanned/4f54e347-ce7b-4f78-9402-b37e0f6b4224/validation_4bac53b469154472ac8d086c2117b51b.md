### Title
MEV Rewards in `FeeReceiver` Excluded from TVL Enables Theft of Unclaimed Yield — (`contracts/LRTDepositPool.sol`)

---

### Summary

`getETHDistributionData()` in `LRTDepositPool` explicitly excludes the `FeeReceiver` contract balance from the TVL used to price rsETH. Because `FeeReceiver.sendFunds()` has no access control and is callable by anyone, an unprivileged attacker can deposit ETH while MEV rewards sit unaccounted in `FeeReceiver`, then atomically push those rewards into TVL and trigger a price update — stealing a proportional share of existing holders' unclaimed yield.

---

### Finding Description

`LRTOracle._updateRsETHPrice()` computes the rsETH price via `_getTotalEthInProtocol()`, which calls `LRTDepositPool.getTotalAssetDeposits()` → `getETHDistributionData()`.

`getETHDistributionData()` sums ETH across the deposit pool, NDCs, EigenLayer pod shares, the unstaking vault, and the converter — but **never includes the `FeeReceiver` balance**: [1](#0-0) 

The code comment at line 465–466 explicitly acknowledges this omission:

> *"rewards are not accounted here — it will automatically be accounted once it is moved from feeReceiver/rewardReceiver to depositPool"*

The mechanism to move those rewards is `FeeReceiver.sendFunds()`: [2](#0-1) 

`sendFunds()` carries **no access control** — any address can call it at any time. Combined with the publicly callable `updateRSETHPrice()`: [3](#0-2) 

this creates a fully self-contained, permissionless attack path.

**Attack sequence:**

1. MEV rewards accumulate in `FeeReceiver` (not reflected in TVL).
2. `updateRSETHPrice()` is called (by anyone) → stored `rsETHPrice` reflects TVL *without* `FeeReceiver` rewards → price is understated.
3. Attacker calls `LRTDepositPool.depositETH()` → mints rsETH at the understated price, receiving more rsETH than their ETH contribution warrants.
4. Attacker calls `FeeReceiver.sendFunds()` → rewards land in the deposit pool, now part of TVL.
5. Attacker calls `updateRSETHPrice()` → price rises to reflect the newly included rewards.
6. Attacker's rsETH is now worth more than they paid → profit extracted from existing holders' unclaimed yield.

The rsETH price used for minting is the stored `rsETHPrice` value, not a freshly computed one: [4](#0-3) 

So any gap between when rewards arrive in `FeeReceiver` and when they are pushed to the deposit pool is exploitable.

---

### Impact Explanation

Existing rsETH holders lose a proportional share of their MEV/execution-layer rewards to the attacker. The attacker effectively front-runs the reward distribution: by depositing before rewards enter TVL, they acquire a larger rsETH share than they are entitled to, and that excess share is funded entirely by diluting existing holders' reward entitlement.

**Numerical example:**

| State | TVL | rsETH Supply | Price |
|---|---|---|---|
| Before (FeeReceiver has 100 ETH) | 10,000 ETH | 10,000 | 1.000 |
| Attacker deposits 1,000 ETH | 11,000 ETH | 11,000 | 1.000 |
| After `sendFunds()` + price update | 11,100 ETH | 11,000 | 1.00909 |

- Attacker's 1,000 rsETH → worth 1,009.09 ETH → **+9.09 ETH profit**
- Existing 10,000 rsETH → worth 10,090.9 ETH → **−9.09 ETH vs. fair distribution**

Without the attack, existing holders would have received the full 100 ETH reward (price → 1.01). With the attack they receive only 90.91 ETH worth.

**Impact: High — Theft of unclaimed yield.**

---

### Likelihood Explanation

The attack requires no special role, no governance capture, and no external oracle manipulation. The attacker only needs to:

- Monitor the `FeeReceiver` ETH balance (public on-chain state).
- Call three permissionless functions: `depositETH`, `sendFunds`, `updateRSETHPrice`.

MEV rewards accumulate continuously in `FeeReceiver`, so the exploitable window recurs every time rewards are not immediately forwarded. The attack is repeatable and scales with the size of accumulated rewards relative to total TVL.

---

### Recommendation

1. **Include `FeeReceiver` balance in `getETHDistributionData()`** so it is always part of TVL regardless of whether `sendFunds()` has been called.
2. **Or** add access control to `FeeReceiver.sendFunds()` (e.g., `onlyLRTManager`) so the timing of reward inclusion cannot be exploited by an unprivileged caller.
3. **Or** call `sendFunds()` atomically inside `updateRSETHPrice()` so rewards are always flushed before the price is computed.

---

### Proof of Concept

```
1. FeeReceiver.balance = 100 ETH (MEV rewards, not in TVL)
2. LRTOracle.updateRSETHPrice() called → rsETHPrice = 1.0 (TVL = 10,000 ETH, supply = 10,000)
3. Attacker calls LRTDepositPool.depositETH{value: 1000 ether}(0, "")
   → rsethAmountToMint = 1000 * 1e18 / rsETHPrice = 1000 rsETH minted
4. Attacker calls FeeReceiver.sendFunds()
   → 100 ETH transferred to LRTDepositPool (TVL now 11,100 ETH)
5. Attacker calls LRTOracle.updateRSETHPrice()
   → newRsETHPrice = 11,100e18 / 11,000 ≈ 1.00909e18
6. Attacker's 1,000 rsETH is now redeemable for ≈1,009.09 ETH
   → Profit: ~9.09 ETH stolen from existing holders' MEV reward entitlement
```

    /// @dev provides ETH amount distribution data among depositPool, NDCs and eigenLayer
    /// @dev rewards are not accounted here
    /// it will automatically be accounted once it is moved from feeReceiver/rewardReceiver to depositPool
    function getETHDistributionData()
        public
        view
        override
        returns (
            uint256 ethLyingInDepositPool,
            uint256 ethLyingInNDCs,
            uint256 ethStakedInEigenLayer,
            uint256 ethUnstakingFromEigenLayer,
            uint256 ethLyingInConverter,
            uint256 ethLyingInUnstakingVault
        )
    {
        ethLyingInDepositPool = address(this).balance;

        uint256 ndcsCount = nodeDelegatorQueue.length;

        for (uint256 i; i < ndcsCount;) {
            ethLyingInNDCs += nodeDelegatorQueue[i].balance;

            ethStakedInEigenLayer += INodeDelegator(nodeDelegatorQueue[i]).getEffectivePodShares();
            ethUnstakingFromEigenLayer += INodeDelegator(nodeDelegatorQueue[i])
                .getAssetUnstaking(LRTConstants.ETH_TOKEN);
            unchecked {
                ++i;
            }
        }

        address lrtUnstakingVault = lrtConfig.getContract(LRTConstants.LRT_UNSTAKING_VAULT);
        ethLyingInUnstakingVault = lrtUnstakingVault.balance;

        address lrtConverter = lrtConfig.getContract(LRTConstants.LRT_CONVERTER);
        ethLyingInConverter = ILRTConverter(lrtConverter).ethValueInWithdrawal();
    }
