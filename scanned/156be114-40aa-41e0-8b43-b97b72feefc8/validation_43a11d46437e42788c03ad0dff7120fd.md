### Title
Division by Zero in `getRsETHAmountToMint` Due to Uninitialized `rsETHPrice` — (File: `contracts/LRTDepositPool.sol`)

---

### Summary
`LRTDepositPool.getRsETHAmountToMint` directly reads `lrtOracle.rsETHPrice()` — the raw auto-generated getter for a state variable — and uses it as a divisor with no fallback. When `rsETHPrice` is 0 (its default on fresh deployment, or if `updateRSETHPrice()` has never been called), every call to `depositETH` and `depositAsset` reverts with a division-by-zero panic, temporarily freezing all user deposits.

---

### Finding Description

In `LRTDepositPool.getRsETHAmountToMint`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.rsETHPrice()` is the Solidity-generated getter for the `rsETHPrice` public state variable declared in `LRTOracle`:

```solidity
uint256 public override rsETHPrice;
```

This variable is **zero-initialized** by the EVM and is only updated when `_updateRsETHPrice()` is internally invoked — either via the public `updateRSETHPrice()` or the manager-gated `updateRSETHPriceAsManager()`. There is no fallback: if `rsETHPrice == 0`, the division panics unconditionally.

This is structurally identical to the Suzaku analog: a stored/cached value is read directly from storage without any zero-check or fallback, and when that value is 0, a division-by-zero DoS results.

The call chain for every user deposit is:

```
depositETH / depositAsset
  → _beforeDeposit
    → getRsETHAmountToMint
      → lrtOracle.rsETHPrice()   ← direct state read, no fallback
      → division by rsETHPrice   ← panics if 0
```

A secondary trigger exists: if `totalETHInProtocol` drops to 0 while `rsethSupply > 0` (e.g., due to a slash or accounting bug) **and** `pricePercentageLimit == 0` (downside protection disabled), `_updateRsETHPrice` computes `newRsETHPrice = 0 / rsethSupply = 0` and writes it back to `rsETHPrice`, re-triggering the DoS on the next deposit attempt.

---

### Impact Explanation

All user-facing deposit entry points (`depositETH`, `depositAsset`) flow through `getRsETHAmountToMint`. If `rsETHPrice == 0`, every deposit reverts. No user can deposit ETH or any LST. The DoS is temporary — calling `updateRSETHPrice()` restores the price — but until that happens, the protocol fails to accept any new deposits.

**Impact**: Medium — Temporary freezing of funds (deposits blocked for all users).

---

### Likelihood Explanation

On a fresh deployment, `rsETHPrice` is 0 until `updateRSETHPrice()` is explicitly called. If the deployment or upgrade sequence omits this call before deposits are opened, all deposits immediately fail. The secondary trigger (price driven to 0 via insolvency with `pricePercentageLimit == 0`) is less likely but reachable without admin compromise if the accounting invariant breaks.

**Likelihood**: Low — Requires either a deployment sequencing error or a specific combination of insolvency and disabled downside protection.

---

### Recommendation

Add a zero-check fallback in `getRsETHAmountToMint`, mirroring the existing fallback in `_updateRsETHPrice` for the zero-supply case:

```solidity
function getRsETHAmountToMint(address asset, uint256 amount) public view override returns (uint256 rsethAmountToMint) {
    address lrtOracleAddress = lrtConfig.getContract(LRTConstants.LRT_ORACLE);
    ILRTOracle lrtOracle = ILRTOracle(lrtOracleAddress);

    uint256 price = lrtOracle.rsETHPrice();
    if (price == 0) price = 1 ether; // fallback to 1:1 if price not yet initialized

    rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / price;
}
```

Alternatively, expose a `getRsETHPrice()` view function in `LRTOracle` that returns `1 ether` when `rsETHPrice == 0`, and use that instead of the raw state getter.

---

### Proof of Concept

1. Deploy `LRTOracle` and `LRTDepositPool` on a fresh chain. `rsETHPrice` is 0.
2. Do **not** call `updateRSETHPrice()`.
3. Call `depositETH{value: 1 ether}(0, "")` from any user address.
4. Transaction reverts with a division-by-zero panic inside `getRsETHAmountToMint` at the line:

```solidity
