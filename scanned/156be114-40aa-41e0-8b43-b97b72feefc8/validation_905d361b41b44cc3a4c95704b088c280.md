### Title
`AGETHTokenWrapper` altAgETH Reserves Drained via Unbacked `MINTER_ROLE` Mint — (`contracts/agETH/AGETHTokenWrapper.sol`)

---

### Summary

`AGETHTokenWrapper` exposes two independent paths to obtain wrapper agETH: a collateralised path (`deposit()`, which pulls altAgETH 1:1) and an uncollateralised path (`mint()`, restricted to `MINTER_ROLE`). The `withdraw()` function accepts wrapper agETH from **either** source and redeems it for altAgETH held in the contract. When `AGETHPoolV3` holds `MINTER_ROLE` and mints wrapper agETH in exchange for ETH, those tokens carry no altAgETH backing inside the wrapper. Any holder of such tokens can immediately call `withdraw()` to drain altAgETH that was deposited by legitimate users, breaking the 1:1 backing invariant and causing insolvency.

---

### Finding Description

**`AGETHTokenWrapper._withdraw`** (lines 111–119) burns wrapper agETH from the caller and transfers an equal amount of altAgETH out of the contract with no check on whether the burned tokens were originally minted with altAgETH collateral: [1](#0-0) 

**`AGETHTokenWrapper.mint`** (lines 165–167) is gated only by `MINTER_ROLE` and mints wrapper agETH with zero altAgETH deposited: [2](#0-1) 

**`AGETHPoolV3.deposit`** (lines 115–128) calls `agETH.mint(msg.sender, agETHAmount)` directly when a user deposits ETH, requiring `MINTER_ROLE` on the agETH token (i.e., `AGETHTokenWrapper`): [3](#0-2) 

The `IERC20AgETH` interface used by `AGETHPoolV3` matches `AGETHTokenWrapper.mint` exactly: [4](#0-3) 

**`maxAmountToDepositBridgerAsset`** (lines 90–101) confirms the invariant the contract is supposed to maintain — `totalSupply() <= balanceOfAssetInWrapper` — but this is only a view helper for the bridger; it does not gate `withdraw()`: [5](#0-4) 

**Attack sequence (concrete):**

| Step | State after |
|------|-------------|
| User A calls `deposit(altAgETH, 100)` | `totalSupply = 100`, `balance(altAgETH) = 100` |
| Attacker calls `AGETHPoolV3.deposit{value: X}()` → `AGETHTokenWrapper.mint(attacker, 50)` | `totalSupply = 150`, `balance(altAgETH) = 100` |
| Attacker calls `AGETHTokenWrapper.withdraw(altAgETH, 50)` | `totalSupply = 100`, `balance(altAgETH) = 50` |
| User A calls `withdraw(altAgETH, 100)` | **reverts** — only 50 altAgETH remain |

The attacker spent ETH (which stays in `AGETHPoolV3`) and received altAgETH that belonged to User A. The wrapper is now permanently under-collateralised.

---

### Impact Explanation

**Critical — Direct theft of user funds / Protocol insolvency.**

Every altAgETH deposited via `deposit()` is redeemable 1:1. After the attack, the wrapper holds fewer altAgETH than outstanding wrapper agETH, so some depositors cannot redeem. The shortfall is permanent until the bridger manually re-collateralises via `depositBridgerAssets()`, which itself is rate-limited by `maxAmountToDepositBridgerAsset` and requires off-chain coordination.

---

### Likelihood Explanation

The precondition — `AGETHPoolV3` holding `MINTER_ROLE` on `AGETHTokenWrapper` — is the **intended production deployment**: `AGETHPoolV3` cannot function at all without it, since its sole purpose is to mint agETH. Both contracts reside in the same `contracts/agETH/` directory and share the same token interface. Any attacker who can deposit ETH (a public, permissionless action) can execute this immediately after deployment.



---

### Recommendation

`_withdraw` must only allow redemption of altAgETH-backed wrapper agETH. Two complementary fixes:

1. **Track backed supply separately.** Maintain a `uint256 backedSupply` that increments only in `_deposit` and `depositBridgerAssets`, and decrements in `_withdraw`. Gate `_withdraw` so it cannot burn more than `backedSupply`.

2. **Separate the token.** Give `AGETHPoolV3` its own mintable agETH token distinct from `AGETHTokenWrapper`. The wrapper should only ever mint via `_deposit` (altAgETH in → wrapper agETH out) and should not expose a free `mint()` to external minters.

// Pseudocode — run on a local fork or Hardhat/Foundry test

function testDrainAltAgETH() public {
    // Setup: AGETHPoolV3 has MINTER_ROLE on AGETHTokenWrapper (intended deployment)
    wrapper.grantRole(MINTER_ROLE, address(pool));

    // User A deposits 100 altAgETH into the wrapper
    altAgETH.approve(address(wrapper), 100e18);
    wrapper.deposit(address(altAgETH), 100e18);
    // totalSupply = 100e18, wrapper.balanceOf(altAgETH) = 100e18

    // Attacker deposits ETH to AGETHPoolV3 → mints 50 wrapper agETH (no altAgETH deposited)
    pool.deposit{value: ethEquivalentOf50AgETH}("ref");
    // totalSupply = 150e18, wrapper.balanceOf(altAgETH) = 100e18

    // Attacker withdraws 50 altAgETH using the unbacked wrapper agETH
    wrapper.withdraw(address(altAgETH), 50e18);
    // totalSupply = 100e18, wrapper.balanceOf(altAgETH) = 50e18

    // User A's 100 wrapper agETH can now only redeem 50 altAgETH → REVERTS
    vm.expectRevert();
    wrapper.withdraw(address(altAgETH), 100e18);

    // Assert insolvency
    assert(altAgETH.balanceOf(address(wrapper)) < wrapper.totalSupply());
}
