### Title
Balance-Based Token Removal Check Can Be Permanently DOS'd by Direct Token Transfer - (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol)

### Summary
The `removeSupportedToken` function in `RSETHPoolV3.sol` and `RSETHPoolV3ExternalBridge.sol` uses a raw `balanceOf(address(this))` check to gate token delisting. Any unprivileged actor can permanently block this function by sending 1 wei of the token directly to the pool, making the balance check permanently non-zero and preventing the protocol from ever delisting a compromised or deprecated asset.

### Finding Description
`removeSupportedToken` in both pool contracts contains the following guard:

```solidity
if (IERC20(token).balanceOf(address(this)) != 0) revert TokenBalanceNotZero();
``` [1](#0-0) [2](#0-1) 

The intent is to ensure all token balances have been bridged before delisting. However, neither pool contract has any mechanism to reject arbitrary ERC20 transfers. An attacker can call `IERC20(token).transfer(poolAddress, 1)` at any time, permanently setting `balanceOf(address(this)) >= 1`. Because the check uses `!= 0` (strict equality to zero), even a 1-wei donation makes the check fail forever.

This is structurally identical to the reported vulnerability: a raw `balanceOf` is used as a proxy for a state invariant (token fully drained), but the balance is externally manipulable independently of the protocol's own accounting (`feeEarnedInToken[token]`), so the two values can diverge permanently.

### Impact Explanation
The `TIMELOCK_ROLE` admin loses the ability to delist any supported token that has been griefed. If a supported LST (e.g., wstETH) suffers a depeg, a smart-contract exploit, or a regulatory freeze, the protocol cannot remove it from the accepted deposit list. Users continue depositing the compromised token and receiving freshly minted `wrsETH` at the stale oracle rate, diluting the backing of all existing `wrsETH` holders and moving the protocol toward insolvency. The impact is: **Low — contract fails to deliver promised returns** (inability to delist), with a credible path to higher severity if the griefed token subsequently loses value.

### Likelihood Explanation
Extremely high. The attack costs a single ERC20 transfer of 1 wei. It requires no special permissions, no flash loan, and no coordination. It can be executed by any EOA or contract at any time, including immediately after a token is added to the supported list, before the admin ever needs to remove it.

### Recommendation
Replace the raw balance check with an explicit accounting state variable that tracks only protocol-controlled inflows:

```solidity
mapping(address token => uint256 deposited) public protocolTokenBalance;
// increment in deposit(), decrement in moveAssetsForBridging() / bridgeTokens()
```

Then gate removal on `protocolTokenBalance[token] == 0` rather than `balanceOf(address(this)) != 0`. This mirrors the fix described in the external report: use a dedicated state variable rather than a live balance query that is manipulable by external actors.

Alternatively, allow the admin to sweep any residual dust to a treasury address before delisting, so that a 1-wei donation cannot permanently block the operation.

### Proof of Concept
1. Admin calls `addSupportedToken(tokenX, oracle, bridge)` — token X is now supported.
2. Attacker calls `IERC20(tokenX).transfer(poolAddress, 1)` — 1 wei of token X now sits in the pool.
3. Admin bridges all legitimate token X deposits via `bridgeTokens(tokenX)` or `moveAssetsForBridging(tokenX, amount)`. After bridging, `IERC20(tokenX).balanceOf(pool) == 1` (the attacker's dust).
4. Admin calls `removeSupportedToken(tokenX, index)`.
5. The function evaluates `IERC20(tokenX).balanceOf(address(this)) != 0` → `1 != 0` → `true` → reverts with `TokenBalanceNotZero`.
6. Token X can never be delisted. If token X subsequently depegs, users continue depositing it and receiving `wrsETH` at the pre-depeg oracle rate. [1](#0-0) [2](#0-1)

    function removeSupportedToken(address token, uint256 tokenIndex) external onlyRole(TIMELOCK_ROLE) {
        UtilLib.checkNonZeroAddress(token);
        if (supportedTokenList[tokenIndex] != token) revert TokenNotFoundError();
        if (IERC20(token).balanceOf(address(this)) != 0) revert TokenBalanceNotZero();

        delete supportedTokenOracle[token];
        supportedTokenList[tokenIndex] = supportedTokenList[supportedTokenList.length - 1];
        supportedTokenList.pop();
        emit RemovedSupportedToken(token);
    }
