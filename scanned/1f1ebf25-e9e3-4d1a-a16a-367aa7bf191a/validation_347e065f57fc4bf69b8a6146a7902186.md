### Title
Fee-on-Transfer Token Mis-Accounting in Staking Balance Mappings - (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary
`KernelDepositPool.stake()` and `stakeFor()` update `balanceOf` and `totalKernelStaked` using the caller-supplied `_amount` parameter rather than the actual tokens received by the contract. If `kernelToken` is a fee-on-transfer or deflationary token, the contract credits more than it holds, causing insolvency for the last withdrawers and diluted rewards for all stakers.

### Finding Description
In `KernelDepositPool`, both `stake()` and `stakeFor()` follow the pattern:

```solidity
balanceOf[msg.sender] += _amount;
totalKernelStaked += _amount;
kernelToken.safeTransferFrom(msg.sender, address(this), _amount);
``` [1](#0-0) [2](#0-1) 

The internal accounting (`balanceOf[account]` and `totalKernelStaked`) is updated with the nominal `_amount` **before** (or without verifying) the actual tokens received. If `kernelToken` charges a transfer fee, the contract receives `_amount - fee` but records `_amount` in both mappings.

When a user later calls `initiateWithdrawal(_amount)`, the full `_amount` is deducted from `balanceOf` (which was inflated, so this succeeds), and a `Withdrawal` record is created for `_amount`. [3](#0-2) 

When `claimWithdrawal` is called, the contract attempts `kernelToken.safeTransfer(msg.sender, withdrawal.amount)` for the full `_amount`: [4](#0-3) 

Because the contract holds less than the sum of all recorded `balanceOf` values, the last users to claim will find the contract insolvent and their transfers will revert.

Additionally, `totalKernelStaked` is inflated, which suppresses `rewardPerToken()` below its true value, causing all stakers to earn fewer rewards than they are entitled to.

### Impact Explanation
- **Permanent freezing of funds**: The aggregate `balanceOf` across all stakers exceeds the contract's actual `kernelToken` balance. The last stakers to withdraw will be unable to claim their tokens — their `claimWithdrawal` calls revert due to insufficient balance.
- **Theft of unclaimed yield**: An inflated `totalKernelStaked` denominator reduces `rewardPerToken`, meaning every staker earns fewer rewards than they should. The shortfall accumulates in the contract as permanently unclaimable yield.

### Likelihood Explanation
The `kernelToken` address is set at initialization and is not restricted to non-fee-on-transfer tokens. The contract is explicitly modeled after Synthetix `StakingRewards` and is designed to be reusable. If the configured `kernelToken` (or any future redeployment) implements a transfer fee or rebasing mechanism, the vulnerability is immediately exploitable by any depositor calling `stake()` — no privileged access is required.

### Recommendation
Measure the actual received amount using a balance-difference pattern in both `stake()` and `stakeFor()`:

```solidity
function stake(uint256 _amount) external nonReentrant updateReward(msg.sender) {
    if (_amount == 0) revert AmountZero();

    uint256 balanceBefore = kernelToken.balanceOf(address(this));
    kernelToken.safeTransferFrom(msg.sender, address(this), _amount);
    uint256 receivedAmount = kernelToken.balanceOf(address(this)) - balanceBefore;

    balanceOf[msg.sender] += receivedAmount;
    totalKernelStaked += receivedAmount;

    emit Staked(msg.sender, receivedAmount);
}
```

Apply the same fix to `stakeFor()`.

### Proof of Concept
1. Deploy `KernelDepositPool` with a `kernelToken` that charges a 1% transfer fee.
2. Alice calls `stake(1000e18)`. The contract receives `990e18` tokens but records `balanceOf[Alice] = 1000e18` and `totalKernelStaked = 1000e18`.
3. Bob calls `stake(1000e18)`. Contract receives another `990e18`; records `balanceOf[Bob] = 1000e18`, `totalKernelStaked = 2000e18`. Actual contract balance: `1980e18`.
4. Alice calls `initiateWithdrawal(1000e18)` → succeeds (her recorded balance is `1000e18`).
5. Alice calls `claimWithdrawal(id)` → `safeTransfer(Alice, 1000e18)` succeeds. Contract balance: `980e18`.
6. Bob calls `initiateWithdrawal(1000e18)` → succeeds (his recorded balance is `1000e18`).
7. Bob calls `claimWithdrawal(id)` → `safeTransfer(Bob, 1000e18)` **reverts** — contract only holds `980e18`. Bob's `1000e18` KERNEL is permanently frozen.

    function claimWithdrawal(uint256 _withdrawalId) external nonReentrant {
        Withdrawal storage withdrawal = withdrawals[_withdrawalId];

        if (withdrawal.user == address(0)) {
            revert WithdrawalDoesNotExist();
        }

        if (withdrawal.user != msg.sender) {
            revert NotYourWithdrawal();
        }

        if (block.timestamp < withdrawal.unlockTime) {
            revert WithdrawalNotReady();
        }

        if (withdrawal.claimed) {
            revert WithdrawalAlreadyClaimed();
        }

        withdrawal.claimed = true;

        // Remove the withdrawal ID from the user's list of withdrawal IDs
        uint256[] storage userWithdrawalIdsArray = userWithdrawalIds[msg.sender];
        for (uint256 i = 0; i < userWithdrawalIdsArray.length; ++i) {
            if (userWithdrawalIdsArray[i] == _withdrawalId) {
                userWithdrawalIdsArray[i] = userWithdrawalIdsArray[userWithdrawalIdsArray.length - 1];
                userWithdrawalIdsArray.pop();
                break;
            }
        }

        // Transfer the KERNEL tokens from contract to user
        kernelToken.safeTransfer(msg.sender, withdrawal.amount);

        emit WithdrawalClaimed(msg.sender, withdrawal.amount, _withdrawalId);
    }
