### Title
ETH Withdrawal Permanently Frozen When Recipient Contract Rejects ETH - (File: contracts/LRTWithdrawalManager.sol)

### Summary
`LRTWithdrawalManager._processWithdrawalCompletion` uses a push-based ETH transfer to send withdrawn ETH to the user. If the user's address is a contract that rejects ETH (no `receive()` / reverts in fallback), `completeWithdrawal` will always revert. Because the user's rsETH is already burned in a prior `unlockQueue` call, the ETH is permanently frozen in the contract with no recovery path.

### Finding Description
The withdrawal lifecycle is split across two separate transactions:

1. **`unlockQueue`** (operator-called): burns the user's rsETH from the contract. [1](#0-0) 

2. **`completeWithdrawal`** (user-called): pushes ETH to the user via `_transferAsset`. [2](#0-1) 

The push transfer at line 878 is:
```solidity
(bool sent,) = payable(to).call{ value: amount }("");
if (!sent) revert EthTransferFailed();
```

If `to` is a contract without a `receive()` function (e.g., a multisig, a DeFi vault, or a smart contract wallet), the call returns `false` and the entire `_processWithdrawalCompletion` reverts. Because the rsETH was already burned in the prior `unlockQueue` call, the user has permanently lost their rsETH with no way to claim the corresponding ETH. [3](#0-2) 

The operator-callable `completeWithdrawalForUser` does not help — it calls the same `_processWithdrawalCompletion(asset, user, ...)` and still pushes ETH to the same non-receiving `user` address. [4](#0-3) 

The code comment on `completeWithdrawalForUser` acknowledges gas-grief scenarios for ETH but incorrectly dismisses them as "non-impactful": [5](#0-4) 

There is no admin rescue function that can redirect a stuck ETH withdrawal to a different address. Additionally, the stuck withdrawal keeps `unlockedWithdrawalsCount[asset] > 0`, which permanently blocks `sweepRemainingAssets` for that asset. [6](#0-5) 

### Impact Explanation
**Critical — Permanent freezing of funds.**

The user's rsETH is burned in `unlockQueue` before `completeWithdrawal` is ever called. If `completeWithdrawal` always reverts (because the user's address rejects ETH), the ETH is permanently locked in `LRTWithdrawalManager` with no recovery mechanism. The user loses both their rsETH (burned) and their ETH (frozen). There is no admin function to redirect the ETH to a different address for the affected user.

### Likelihood Explanation
**Medium.** Smart contract wallets (Gnosis Safe, account-abstraction wallets), DeFi vaults, and protocol-owned addresses that interact with LRT-rsETH are common in the ecosystem. Many such contracts do not implement a `receive()` function. A user who initiates a withdrawal from such an address will have their funds permanently frozen after the operator runs `unlockQueue`.

### Recommendation
Replace the push-based ETH transfer with a pull-based pattern. Record the owed ETH amount in a mapping and allow the user (or any address they designate) to call a separate `claimETH(address recipient)` function to pull the funds. This decouples the accounting step from the ETH delivery step, ensuring that a non-receiving address cannot cause permanent fund loss.

```solidity
// Example pull pattern
mapping(address => uint256) public pendingETHWithdrawals;

// In _processWithdrawalCompletion, instead of _transferAsset:
pendingETHWithdrawals[user] += request.expectedAssetAmount;

// New function:
function claimETH(address recipient) external nonReentrant {
    uint256 amount = pendingETHWithdrawals[msg.sender];
    if (amount == 0) revert NothingToClaim();
    pendingETHWithdrawals[msg.sender] = 0;
    (bool sent,) = payable(recipient).call{ value: amount }("");
    if (!sent) revert EthTransferFailed();
}
```

### Proof of Concept

1. Alice deploys a smart contract vault (no `receive()`) and calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, ...)` from it. Her rsETH is transferred to `LRTWithdrawalManager`. [7](#0-6) 

2. An operator calls `unlockQueue(ETH_TOKEN, ...)`. Alice's rsETH is burned from the contract; ETH is pulled from `LRTUnstakingVault` into `LRTWithdrawalManager`. [1](#0-0) 

3. Alice calls `completeWithdrawal(ETH_TOKEN, ...)`. `_processWithdrawalCompletion` reaches `_transferAsset(ETH_TOKEN, alice_vault, amount)`. [8](#0-7) 

4. `payable(alice_vault).call{ value: amount }("")` returns `false` because `alice_vault` has no `receive()`. `EthTransferFailed` is thrown; the entire transaction reverts. [9](#0-8) 

5. Alice's rsETH is permanently burned. The ETH remains in `LRTWithdrawalManager` forever. No admin function exists to redirect it. `unlockedWithdrawalsCount[ETH_TOKEN]` stays elevated, also blocking `sweepRemainingAssets`.

    function _processWithdrawalCompletion(address asset, address user, string calldata referralId) internal {
        if (userAssociatedNonces[asset][user].empty()) {
            revert NoWithdrawalRequests(user, asset);
        }

        // Retrieve and remove the oldest withdrawal request for the user.
        uint256 usersFirstWithdrawalRequestNonce = userAssociatedNonces[asset][user].popFront();
        // Ensure the request is already unlocked.
        if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();

        bytes32 requestId = getRequestId(asset, usersFirstWithdrawalRequestNonce);
        WithdrawalRequest memory request = withdrawalRequests[requestId];

        delete withdrawalRequests[requestId];

        // Check that the withdrawal delay has passed since the request's initiation.
        if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();

        unlockedWithdrawalsCount[asset]--;

        // If Aave integration is enabled and asset is ETH, withdraw from Aave if needed
        if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
            uint256 contractBalance = address(this).balance;
            if (contractBalance < request.expectedAssetAmount) {
                uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
                _withdrawFromAave(amountNeeded);

                // Verify we have sufficient balance after withdrawal
                uint256 balanceAfter = address(this).balance;
                if (balanceAfter < request.expectedAssetAmount) {
                    revert InsufficientLiquidityForWithdrawal();
                }
            }
        }

        _transferAsset(asset, user, request.expectedAssetAmount);

        emit ReferralIdEmitted(referralId);
        emit AssetWithdrawalFinalized(user, asset, request.rsETHUnstaked, request.expectedAssetAmount);
    }
