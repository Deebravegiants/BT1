### Title
Incorrect ETH Deposit Limit Check Omits Deposit Amount, Allowing Deposits Beyond Configured Cap - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` uses an asymmetric check for ETH versus ERC20 assets. The ETH branch omits the incoming `amount` from the comparison, allowing any depositor to push total ETH deposits arbitrarily beyond the admin-configured deposit cap.

### Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` (lines 676–682 of `contracts/LRTDepositPool.sol`), the ETH branch checks only whether the *current* total already exceeds the limit, without including the new deposit amount:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // ← `amount` missing
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct for ERC20
}
```

The ERC20 branch correctly evaluates `totalAssetDeposits + amount > depositLimit`. The ETH branch evaluates only `totalAssetDeposits > depositLimit`. As long as the running total has not yet crossed the cap, the check returns `false` regardless of how large `amount` is, and the deposit proceeds unconstrained.

This is the direct analog of the external report's "incorrect assertion" class: a guard condition that is logically incomplete, allowing inputs that the protocol intends to reject.

The vulnerable call path is:

`depositETH` → `_beforeDeposit` → `_checkIfDepositAmountExceedesCurrentLimit` [1](#0-0) [2](#0-1) 

### Impact Explanation
The deposit limit is the protocol's primary on-chain risk-management control for each asset. Bypassing it for ETH allows the protocol to accumulate ETH restaking exposure far beyond what administrators intended. This undermines the promised invariant that total deposits per asset cannot exceed `depositLimitByAsset`. Classified as **Low — contract fails to deliver promised returns, but does not directly lose value**.

### Likelihood Explanation
Any unprivileged depositor can trigger this. The condition is met whenever `totalAssetDeposits <= depositLimit` (i.e., the cap has not yet been independently breached), which is the normal operating state. No special setup is required beyond a standard `depositETH` call with a large `msg.value`.

### Recommendation
Add `amount` to the ETH branch to match the ERC20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept
1. Admin sets ETH deposit limit to 100 ETH via `LRTConfig`.
2. Current `getTotalAssetDeposits(ETH_TOKEN)` = 99 ETH.
3. Attacker calls `depositETH{value: 10_000 ether}(minRSETH, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 10_000 ether)` evaluates `99 ETH > 100 ETH` → `false` → limit not exceeded.
5. Deposit succeeds; total ETH deposits become 10,099 ETH — 100× the intended cap.
6. rsETH is minted to the attacker; the protocol now holds ETH restaking exposure far beyond the configured risk limit. [3](#0-2)
