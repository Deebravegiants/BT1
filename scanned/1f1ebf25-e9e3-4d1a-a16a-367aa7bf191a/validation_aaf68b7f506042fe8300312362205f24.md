### Title
`rsETHPrice` Defaults to `1 ether` When `rsethSupply == 0`, Enabling First-Depositor Theft of Orphaned Protocol Assets - (File: contracts/LRTOracle.sol)

### Summary
`LRTOracle._updateRsETHPrice()` unconditionally sets `rsETHPrice = 1 ether` and returns early whenever `rsethSupply == 0`, without inspecting the actual ETH value held in the protocol. If the protocol holds any assets at that moment (e.g., rewards forwarded by the reward receiver, or ETH sent directly to the deposit pool), the stored price is wrong and the first depositor can mint rsETH at a 1:1 rate and immediately redeem it for the full protocol TVL, stealing the orphaned assets.

### Finding Description
In `_updateRsETHPrice()`:

```solidity
if (rsethSupply == 0) {
    rsETHPrice = 1 ether;
    highestRsethPrice = 1 ether;
    return;          // ← _getTotalEthInProtocol() is never called
}
```

The early return skips the `_getTotalEthInProtocol()` computation entirely. The stored `rsETHPrice` is therefore always `1e18` whenever supply is zero, regardless of how much ETH or LST value the deposit pool, node delegators, or unstaking vault actually hold.

`updateRSETHPrice()` is a **public, permissionless** function — any caller can trigger this reset.

The minting formula in `LRTDepositPool.getRsETHAmountToMint()` divides by the stored price:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

If `rsETHPrice` is `1e18` but the protocol holds, say, 50 ETH of orphaned rewards, a depositor who sends 1 ETH receives 1 rsETH. On the very next `updateRSETHPrice()` call (after supply is non-zero), the price correctly reflects `(1 + 50) / 1 = 51 ETH per rsETH`. The attacker's 1 rsETH is now redeemable for 51 ETH.

Concrete trigger paths for `rsethSupply == 0` with non-zero TVL:

1. **Reward accumulation before first deposit** — `LRTDepositPool.receiveFromRewardReceiver()` is payable and callable by anyone labelled as the reward receiver; ETH can arrive before any rsETH is minted.
2. **Post-full-withdrawal reward drip** — if all rsETH is eventually burned through the withdrawal pipeline but the reward receiver forwards ETH to the pool in the same block or shortly after, the supply is zero while TVL is non-zero.
3. **Direct ETH send** — `LRTDepositPool` has an open `receive()` fallback; any ETH sent directly inflates TVL without minting rsETH.

### Impact Explanation
A depositor who acts as the first minter after `rsethSupply == 0` with non-zero TVL receives rsETH priced at `1e18` and, after the next price update, holds tokens worth the entire protocol TVL. This constitutes **direct theft of any assets that were present in the protocol at the time of the zero-supply price reset** — a Critical impact under the allowed scope.

### Likelihood Explanation
The condition requires `rsethSupply == 0` simultaneously with a non-zero protocol balance. This is unlikely during steady-state operation but is realistic at:
- Protocol launch (before the first deposit, if any ETH is pre-seeded or rewards arrive early).
- Any future period where all rsETH is redeemed and a reward drip arrives before a new depositor.

Because `updateRSETHPrice()` is public and permissionless, an attacker can atomically: (a) observe the zero-supply state, (b) call `updateRSETHPrice()` to lock in `rsETHPrice = 1e18`, and (c) deposit in the same transaction. Likelihood is **Low** but the attack is fully permissionless once the precondition exists.

### Recommendation
Replace the unconditional early return with a check that also inspects the actual TVL:

```solidity
if (rsethSupply == 0) {
    uint256 totalETH = _getTotalEthInProtocol();
    require(totalETH == 0, "Orphaned assets: supply is zero but TVL is non-zero");
    rsETHPrice = 1 ether;
    highestRsethPrice = 1 ether;
    return;
}
```

Alternatively, revert when `rsethSupply == 0` and `totalETH > 0`, forcing governance to resolve the orphaned-asset state before any new minting is allowed.

### Proof of Concept
1. Protocol is live; all rsETH has been redeemed → `rsethSupply == 0`.
2. Reward receiver forwards 50 ETH to `LRTDepositPool` via `receiveFromRewardReceiver()`.
3. Attacker calls `LRTOracle.updateRSETHPrice()` → `rsETHPrice` is set to `1e18`.
4. Attacker calls `LRTDepositPool.depositETH{value: 1 ether}(0, "")` → receives `1e18` rsETH (1 rsETH).
5. Attacker calls `LRTOracle.updateRSETHPrice()` again → now `rsethSupply = 1`, `totalETHInProtocol = 51 ETH`, so `rsETHPrice = 51e18`.
6. Attacker initiates withdrawal of their 1 rsETH → receives 51 ETH, netting 50 ETH profit from the orphaned reward balance. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
