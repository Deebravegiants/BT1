### Title
Mutable `withdrawalDelayBlocks` Retroactively Extends Lock Period for All Pending Withdrawal Requests - (File: `contracts/LRTWithdrawalManager.sol`)

### Summary
`LRTWithdrawalManager` stores only `withdrawalStartBlock` per withdrawal request, but evaluates the delay check at completion time using the **current global** `withdrawalDelayBlocks`. When the manager legitimately increases `withdrawalDelayBlocks`, every already-queued withdrawal request is retroactively subject to the longer delay, temporarily freezing user funds beyond the period promised at initiation.

### Finding Description
When a user calls `initiateWithdrawal`, the contract records `withdrawalStartBlock: block.number` in the `WithdrawalRequest` struct and transfers the user's rsETH into the contract. [1](#0-0) 

The `WithdrawalRequest` struct contains no snapshot of `withdrawalDelayBlocks`: [2](#0-1) 

Both the operator-facing `_unlockWithdrawalRequests` and the user-facing `_processWithdrawalCompletion` evaluate the delay using the **live** global `withdrawalDelayBlocks` at call time: [3](#0-2) [4](#0-3) 

The manager can update `withdrawalDelayBlocks` at any time, up to the 16-day ceiling: [5](#0-4) 

There is no mechanism to freeze the delay value at initiation time, so a post-initiation increase immediately and silently extends the lock on every pending request.

### Impact Explanation
**Medium — Temporary freezing of funds.** A user who calls `initiateWithdrawal` at block N under the default 8-day delay (`withdrawalDelayBlocks = 57,600`) surrenders their rsETH to the contract and expects to complete the withdrawal at block N + 57,600. If the manager raises `withdrawalDelayBlocks` to the 16-day maximum (115,200 blocks) before that block is reached, both `_unlockWithdrawalRequests` (operator path) and `_processWithdrawalCompletion` (user path) will revert or skip the request until block N + 115,200. The user's rsETH is already burned from their wallet and held by the contract; they cannot cancel the request. Funds are frozen for up to an additional 8 days beyond the promised period.

### Likelihood Explanation
**Low.** The trigger requires the `LRTManager` to increase `withdrawalDelayBlocks` while withdrawal requests are pending. This is a legitimate protocol operation (e.g., aligning with EigenLayer's own withdrawal delay changes) that carries no malicious intent, yet its retroactive effect on existing requests is an unintended design consequence — directly analogous to the external report's `voteRewardsCoefficient` change affecting active proposals.

### Recommendation
Snapshot `withdrawalDelayBlocks` into the `WithdrawalRequest` struct at initiation time and use that stored value in both `_unlockWithdrawalRequests` and `_processWithdrawalCompletion`. This ensures that a user's promised withdrawal window is immutable once the request is created, while still allowing the global parameter to govern future requests.

```solidity
struct WithdrawalRequest {
    uint256 rsETHUnstaked;
    uint256 expectedAssetAmount;
    uint256 withdrawalStartBlock;
    uint256 withdrawalDelayBlocksSnapshot; // add this
}
```

### Proof of Concept
1. `withdrawalDelayBlocks = 57,600` (8 days). User calls `initiateWithdrawal(ETH, 1e18, "")` at block 1,000,000. rsETH is transferred to the contract. Request stored with `withdrawalStartBlock = 1,000,000`.
2. Manager calls `setWithdrawalDelayBlocks(115_200)` (16 days) at block 1,050,000 — a routine operational adjustment.
3. At block 1,057,600 (8 days after initiation), user calls `completeWithdrawal(ETH, "")`.
4. `_processWithdrawalCompletion` evaluates: `1,057,600 < 1,000,000 + 115,200` → `1,057,600 < 1,115,200` → **true** → reverts `WithdrawalDelayNotPassed`.
5. The operator's `unlockQueue` call also skips the request for the same reason.
6. User's rsETH remains locked in the contract until block 1,115,200 — 8 days longer than promised at initiation. [6](#0-5) [5](#0-4) [7](#0-6)

        if (withdrawalDelayBlocks_ > 16 days / 12 seconds) revert ExceedWithdrawalDelay();

        withdrawalDelayBlocks = withdrawalDelayBlocks_;
        emit WithdrawalDelayBlocksUpdated(withdrawalDelayBlocks);
    }

    function _processWithdrawalCompletion(address asset, address user, string calldata referralId) internal {
        if (userAssociatedNonces[asset][user].empty()) {
            revert NoWithdrawalRequests(user, asset);
        }

        // Retrieve and remove the oldest withdrawal request for the user.
        uint256 usersFirstWithdrawalRequestNonce = userAssociatedNonces[asset][user].popFront();
        // Ensure the request is already unlocked.
        if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();

        bytes32 requestId = getRequestId(asset, usersFirstWithdrawalRequestNonce);
        WithdrawalRequest memory request = withdrawalRequests[requestId];

        delete withdrawalRequests[requestId];

        // Check that the withdrawal delay has passed since the request's initiation.
        if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();
