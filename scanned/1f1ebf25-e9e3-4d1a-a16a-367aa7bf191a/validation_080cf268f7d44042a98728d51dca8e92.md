### Title
Withdrawal Delay Decrease Does Not Benefit Users With Pending Withdrawals - (`contracts/KERNEL/KernelDepositPool.sol`)

### Summary
`KernelDepositPool` bakes the current `withdrawalDelay` into each withdrawal's `unlockTime` at initiation time. When the admin decreases `withdrawalDelay`, all previously initiated withdrawals remain locked until their original (longer) `unlockTime`, while new withdrawals immediately benefit from the shorter delay. This is a direct analog of M-19.

### Finding Description
In `KernelDepositPool.initiateWithdrawal()`, the unlock timestamp is computed and stored permanently at the moment of initiation:

```solidity
uint256 unlockTime = block.timestamp + withdrawalDelay;   // line 330
withdrawals[withdrawalId] = Withdrawal({
    ..., unlockTime: unlockTime, ...                       // line 332-334
});
``` [1](#0-0) 

`claimWithdrawal()` then enforces this stored value:

```solidity
if (block.timestamp < withdrawal.unlockTime) revert WithdrawalNotReady();  // line 355
``` [2](#0-1) 

The admin can change `withdrawalDelay` at any time via `setWithdrawalDelay()`, with no lower bound other than `> 0` and `<= MAX_WITHDRAWAL_DELAY` (30 days):

```solidity
function setWithdrawalDelay(uint256 _withdrawalDelay) external onlyRole(DEFAULT_ADMIN_ROLE) {
    if (_withdrawalDelay == 0) revert InvalidWithdrawalDelay();
    if (_withdrawalDelay > MAX_WITHDRAWAL_DELAY) revert MaximumWithdrawalDelayExceeded();
    withdrawalDelay = _withdrawalDelay;
    ...
}
``` [3](#0-2) 

Because `unlockTime` is frozen at initiation, a delay reduction has no retroactive effect on pending withdrawals. Users who initiated before the reduction must wait the full original delay; users who initiate after the reduction wait only the new (shorter) delay.

### Impact Explanation
**Temporary freezing of funds (Medium).** A user who initiated a withdrawal when `withdrawalDelay` was at its maximum of 30 days cannot claim their KERNEL tokens until the original `unlockTime` elapses, even if the delay is subsequently reduced to 1 second. Their staked KERNEL is already debited from `balanceOf` at initiation time (`line 325`), so the tokens are neither staked nor claimable for the full original delay period. [4](#0-3) 

### Likelihood Explanation
The scenario requires only a routine, legitimate governance action (reducing the withdrawal delay). No attacker involvement is needed — any user who happened to initiate a withdrawal before the delay was lowered is automatically affected. Given that `withdrawalDelay` is explicitly designed to be adjustable and the maximum is 30 days, a reduction is a plausible operational event.

### Recommendation
At `claimWithdrawal()` time, allow the user to claim if either the stored `unlockTime` has passed **or** `block.timestamp >= withdrawalStartTimestamp + withdrawalDelay` (using the current, potentially shorter delay). Concretely, store `withdrawalStartTimestamp` instead of (or in addition to) `unlockTime`, and check:

```solidity
uint256 effectiveUnlock = withdrawalStartTimestamp + withdrawalDelay; // current delay
if (block.timestamp < effectiveUnlock) revert WithdrawalNotReady();
```

This mirrors the mitigation suggested in M-19: allow users to benefit from the current (lower) delay even for withdrawals initiated under a higher one.

### Proof of Concept
1. Admin sets `withdrawalDelay = 30 days`.
2. Alice calls `initiateWithdrawal(100e18)`. Her withdrawal record stores `unlockTime = block.timestamp + 30 days`. Her KERNEL balance is immediately debited.
3. Admin calls `setWithdrawalDelay(1 days)` — a legitimate governance reduction.
4. Bob calls `initiateWithdrawal(100e18)`. His `unlockTime = block.timestamp + 1 day`.
5. After 2 days: Bob calls `claimWithdrawal()` successfully.
6. Alice calls `claimWithdrawal()` → reverts with `WithdrawalNotReady` because `block.timestamp < unlockTime` (still ~28 days away).
7. Alice's KERNEL remains locked in the contract for ~28 more days despite the protocol now enforcing only a 1-day delay. [5](#0-4) [6](#0-5)

    function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
        if (_amount == 0) revert AmountZero();
        if (balanceOf[msg.sender] < _amount) revert InsufficientStakedBalance();
        if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();

        balanceOf[msg.sender] -= _amount;
        totalKernelStaked -= _amount;

        // Create a withdrawal record
        uint256 withdrawalId = ++withdrawalCounter;
        uint256 unlockTime = block.timestamp + withdrawalDelay;

        withdrawals[withdrawalId] = Withdrawal({
            user: msg.sender, amount: _amount, unlockTime: unlockTime, claimed: false, withdrawalId: withdrawalId
        });
        userWithdrawalIds[msg.sender].push(withdrawalId);

        emit WithdrawalInitiated(msg.sender, _amount, withdrawalId, unlockTime);
    }
