### Title
Aave State Mismatch Between Withdrawal Initiation and Completion Causes Temporary ETH Freezing — (`contracts/LRTWithdrawalManager.sol`)

---

### Summary

`_processWithdrawalCompletion` evaluates `isAaveIntegrationEnabled` at **completion time**, not at **initiation time**. A user who initiated an ETH withdrawal while Aave was disabled can have their completion attempt forced through the Aave withdrawal path after the operator enables Aave and deposits all idle ETH. If Aave WETH liquidity is insufficient at that point, the user's `completeWithdrawal` call reverts and their ETH is temporarily frozen.

---

### Finding Description

The `WithdrawalRequest` struct stores only three fields: [1](#0-0) 

It does **not** record whether `isAaveIntegrationEnabled` was true or false at initiation time. As a result, `_processWithdrawalCompletion` branches on the **current** flag: [2](#0-1) 

When `isAaveIntegrationEnabled=true` and `address(this).balance < request.expectedAssetAmount`, the function calls `_withdrawFromAave`, which calls `aaveWETHGateway.withdrawETH`: [3](#0-2) 

If Aave WETH liquidity is insufficient, `withdrawETH` reverts, propagating the revert up through `_processWithdrawalCompletion`. Because Solidity reverts roll back all state changes, the `delete withdrawalRequests[requestId]` and `popFront()` are undone — the request is not lost — but the user cannot complete the withdrawal until Aave liquidity is restored.

A second compounding factor: `unlockQueue`, when called with Aave enabled, automatically deposits the freshly redeemed ETH back into Aave: [4](#0-3) 

This means that after `unlockQueue` runs with Aave enabled, `address(this).balance` is 0, guaranteeing the Aave withdrawal path is always taken at completion time.

---

### Impact Explanation

**Medium — Temporary freezing of user ETH.**

The user's rsETH was transferred to the contract at `initiateWithdrawal` and burned at `unlockQueue`. The user holds no rsETH and no ETH. If Aave WETH liquidity is insufficient, every `completeWithdrawal` call reverts. The freeze persists until either Aave liquidity is restored or the operator successfully calls `setAaveIntegrationEnabled(false)` — but that function also calls `_withdrawFromAave` internally and would itself revert if Aave is illiquid: [5](#0-4) 

This creates a situation where neither the user nor the operator can unblock the withdrawal while Aave is illiquid.

---

### Likelihood Explanation

**Low.** The scenario requires three independent conditions to align:

1. A user initiates an ETH withdrawal while Aave is disabled.
2. The operator legitimately enables Aave and deposits idle ETH (both are normal operational actions via `setAaveIntegrationEnabled` and `depositIdleETHToAave`).
3. Aave WETH liquidity becomes insufficient before the user completes withdrawal.

Conditions 1 and 2 are plausible during a protocol upgrade that introduces Aave integration. Condition 3 (Aave WETH illiquidity) is rare but not impossible under extreme market stress. The design flaw is the root cause; Aave illiquidity is a necessary amplifier.

---

### Recommendation

Store the Aave integration state at request initiation time in the `WithdrawalRequest` struct:

```solidity
struct WithdrawalRequest {
    uint256 rsETHUnstaked;
    uint256 expectedAssetAmount;
    uint256 withdrawalStartBlock;
    bool aaveEnabledAtInitiation; // add this
}
```

In `_addUserWithdrawalRequest`, record `isAaveIntegrationEnabled` at the time of the request. In `_processWithdrawalCompletion`, use `request.aaveEnabledAtInitiation` instead of the live `isAaveIntegrationEnabled` flag to decide whether to attempt an Aave withdrawal.

Alternatively, ensure that `setAaveIntegrationEnabled(false)` can always succeed even when Aave is illiquid (e.g., by skipping the withdrawal and leaving aWETH in place, with a separate recovery path).

    function _withdrawFromAave(uint256 amount) internal returns (uint256 withdrawnAmount) {
        if (amount == 0) return 0;

        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        if (aaveBalance == 0) revert InsufficientAaveBalance();

        // Only withdraw up to the principal amount (don't use accrued interest for user withdrawals)
        uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave;

        withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
        if (withdrawnAmount == 0) return 0;

        aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
        totalETHDepositedToAave -= withdrawnAmount;

        emit ETHWithdrawnFromAave(withdrawnAmount, totalETHDepositedToAave);
    }
