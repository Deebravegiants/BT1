### Title
Strict Equality Fee Check in `bridgeRsETHToL2UsingCCIP` Enables Block-Stuffing DoS — (`contracts/L1VaultV2.sol`)

---

### Summary

`bridgeRsETHToL2UsingCCIP` re-quotes the CCIP fee **on-chain at execution time** and enforces **strict equality** between `msg.value` and the live fee. Because the CCIP router itself accepts overpayment, the strict equality guard is an unnecessary constraint that makes every bridge call vulnerable to any fee change between the MANAGER's off-chain quote and on-chain execution — including fee changes induced by block stuffing.

---

### Finding Description

The execution path is:

1. MANAGER calls `getCCIPFee(amount)` off-chain (a view call to `ccipRouter.getFee`), records the result as `fee`, and prepares a transaction with `msg.value = fee`.
2. MANAGER submits `bridgeRsETHToL2UsingCCIP(amount)` with that `msg.value`.
3. On-chain, the function re-executes `getCCIPFee(amount)` at line 354, which again calls `ccipRouter.getFee`.
4. The check at line 356 enforces `msg.value == fee` (strict equality). [1](#0-0) 

If the CCIP fee has changed by even 1 wei between the off-chain quote and on-chain execution, the transaction reverts with `IncorrectCCIPFee`.

The `IRouterClient` interface explicitly documents that `ccipSend` **accepts overpayment**: [2](#0-1) 

Despite this, `L1VaultV2` forbids the MANAGER from sending any buffer above the quoted fee. The MANAGER has no way to submit a transaction that is robust to fee fluctuations.

**Block-stuffing trigger:** An attacker fills L1 blocks between the MANAGER's quote and submission. CCIP fees incorporate destination-chain gas costs and can be updated by Chainlink's fee management in response to gas-price conditions. Repeated stuffing windows force repeated reverts, leaving rsETH stranded in `L1Vault` for the duration of the attack.

---

### Impact Explanation

rsETH accumulates in `L1VaultV2` and cannot be bridged to L2 for as long as the attacker sustains the stuffing window. This is **temporary freezing of funds** (rsETH in transit is not lost, but is inaccessible to L2 users during the attack). Classified **Low — Block stuffing** per the allowed scope.

---

### Likelihood Explanation

Block stuffing on Ethereum mainnet is expensive but has been executed in practice (e.g., NFT mint races). The MANAGER's bridge operation is infrequent and predictable (observable mempool), making the timing window feasible. Even without deliberate stuffing, routine Chainlink CCIP fee updates between quote and execution can trigger the same revert path at zero cost to any party.

---

### Recommendation

Replace the strict equality check with a minimum-fee check, mirroring how the CCIP router itself works:

```solidity
// Before
if (msg.value != fee) revert IncorrectCCIPFee();

// After
if (msg.value < fee) revert IncorrectCCIPFee();
```

This allows the MANAGER to include a small buffer above the quoted fee. Any overpayment is accepted by `ccipSend` per the router's documented behaviour. [3](#0-2) 

---

pragma solidity ^0.8.0;

// Foundry fork test (local fork, no mainnet calls)
// Demonstrates that a 1-wei fee increase between quote and send causes IncorrectCCIPFee

contract MockRouter {
    uint256 public feeToReturn;
    function setFee(uint256 f) external { feeToReturn = f; }
    function getFee(uint64, bytes memory) external view returns (uint256) { return feeToReturn; }
    function ccipSend(uint64, bytes calldata) external payable returns (bytes32) { return bytes32(0); }
}

// 1. Deploy L1VaultV2 pointing at MockRouter
// 2. MockRouter.setFee(1 ether)          <- MANAGER quotes off-chain: fee = 1 ether
// 3. MockRouter.setFee(1 ether + 1)      <- attacker stuffs blocks; Chainlink updates fee
// 4. MANAGER calls bridgeRsETHToL2UsingCCIP{value: 1 ether}(amount)
//    -> getCCIPFee returns 1 ether + 1
//    -> msg.value (1 ether) != fee (1 ether + 1)
//    -> reverts IncorrectCCIPFee                  ✓
// 5. rsETH remains in L1VaultV2; repeat steps 3-4 to sustain the freeze
