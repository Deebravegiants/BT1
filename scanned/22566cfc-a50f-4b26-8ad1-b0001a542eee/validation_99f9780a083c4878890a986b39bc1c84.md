### Title
Chainlink Price Feed Staleness Not Validated in `ChainlinkPriceOracle`, Enabling Deposits at Inflated LST Prices - (File: contracts/oracles/ChainlinkPriceOracle.sol)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards both the `updatedAt` timestamp and the `answeredInRound` return value. No staleness check is performed. A stale Chainlink price for a supported LST (e.g., stETH, ethX, sfrxETH) flows directly into rsETH minting math, allowing depositors to receive more rsETH than the current fair value of their deposit warrants, diluting all existing rsETH holders.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the LST/ETH exchange rate as follows:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L49-55
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
``` [1](#0-0) 

The destructuring `(, int256 price,,,)` discards `roundId`, `startedAt`, `updatedAt`, and `answeredInRound`. Neither a round-based staleness check (`answeredInRound >= roundId`) nor a time-based staleness check (`block.timestamp - updatedAt <= MAX_STALENESS`) is performed.

By contrast, the sibling contract `ChainlinkOracleForRSETHPoolCollateral` — used for pool collateral pricing — does perform partial staleness validation:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L27-32
(uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
    AggregatorV3Interface(oracle).latestRoundData();
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
``` [2](#0-1) 

`ChainlinkPriceOracle` has neither of these guards.

The stale price propagates through the following call chain:

1. `LRTOracle.getAssetPrice(asset)` delegates to `IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset)`, which resolves to `ChainlinkPriceOracle.getAssetPrice()`. [3](#0-2) 

2. `LRTOracle._getTotalEthInProtocol()` calls `getAssetPrice(asset)` for every supported LST to compute the protocol's total ETH value. [4](#0-3) 

3. `LRTOracle._updateRsETHPrice()` uses `_getTotalEthInProtocol()` to derive and store `rsETHPrice`. [5](#0-4) 

4. `LRTDepositPool.getRsETHAmountToMint()` uses both `lrtOracle.getAssetPrice(asset)` and `lrtOracle.rsETHPrice()` to determine how many rsETH tokens to mint per deposit. [6](#0-5) 

5. `LRTDepositPool.depositAsset()` and `depositETH()` call `_beforeDeposit()` → `getRsETHAmountToMint()` and then mint rsETH to the depositor. [7](#0-6) 

---

### Impact Explanation

If a supported LST depegs (e.g., stETH during a slashing event or liquidity crisis) but the Chainlink feed has not yet updated within its heartbeat window, `ChainlinkPriceOracle` continues to return the pre-depeg price. A depositor can then:

- Deposit the depegged LST at the stale (inflated) price.
- Receive rsETH calculated as `(depositAmount × staleHighPrice) / rsETHPrice`, which is more rsETH than the deposit's true ETH value warrants.
- When the oracle eventually updates and `rsETHPrice` is recalculated downward to reflect the true LST value, all existing rsETH holders bear the dilution.

This constitutes **theft of yield from existing rsETH holders** — their share of the underlying pool is reduced by the excess rsETH minted to the attacker. Impact: **High**.

---

### Likelihood Explanation

Chainlink LST/ETH feeds (e.g., stETH/ETH) have heartbeat periods of up to 24 hours and a deviation threshold of 0.5%. During market stress — exactly when LST depegs occur — network congestion can delay oracle updates well beyond the heartbeat. The 2022 stETH depeg is a historical precedent. No special permissions are required; any unprivileged depositor can exploit this window. Likelihood: **Medium**.

---

### Recommendation

Add both a round-based and a time-based staleness check in `ChainlinkPriceOracle.getAssetPrice()`, consistent with the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (block.timestamp - updatedAt > MAX_STALENESS_PERIOD) revert StalePrice(); // e.g. 25 hours
    if (price <= 0) revert InvalidPrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`MAX_STALENESS_PERIOD` should be set per feed based on its documented heartbeat (e.g., 25 hours for a 24-hour heartbeat feed).

---

### Proof of Concept

1. Assume stETH/ETH Chainlink feed has a 24-hour heartbeat. The last update was 23 hours ago at price `1.0 ETH`. stETH then depegs to `0.95 ETH` on-chain but the feed has not yet updated.
2. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
3. `getRsETHAmountToMint` computes: `(1000e18 × 1.0e18) / rsETHPrice`. With `rsETHPrice = 1.01e18`, attacker receives `≈990 rsETH`.
4. True fair value of 1000 stETH at `0.95 ETH` is `950 ETH`, which should yield `≈940 rsETH`.
5. Attacker received `≈50 rsETH` excess (~5% of deposit), diluting all existing rsETH holders by that amount.
6. After the oracle updates, `updateRSETHPrice()` recalculates `rsETHPrice` downward, permanently encoding the dilution into the exchange rate for all prior holders. [1](#0-0) [8](#0-7)
