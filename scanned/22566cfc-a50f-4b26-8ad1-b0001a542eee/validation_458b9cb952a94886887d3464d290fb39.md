### Title
FIFO Queue Head Blocking Causes Temporary Freezing of Subsequent Withdrawal Requests - (`contracts/LRTWithdrawalManager.sol`)

---

### Summary

A user who queues a large withdrawal request that lands at the head of the per-asset FIFO queue can cause all subsequent, smaller withdrawal requests for that asset to be temporarily frozen. The root cause is a `break` at line 800 of `_unlockWithdrawalRequests` that halts queue processing the moment the head request's `payoutAmount` exceeds the unstaking vault's current balance, regardless of whether the vault holds enough to satisfy every other request behind it.

---

### Finding Description

**Two separate asset pools are used for two different checks:**

1. **At `initiateWithdrawal` time** (line 170), the guard uses `getAvailableAssetAmount`, which queries `lrtDepositPool.getTotalAssetDeposits(asset)` — the aggregate of all protocol assets including those still deployed in EigenLayer strategies. [1](#0-0) 

2. **At `unlockQueue` time**, `_createUnlockParams` sets `totalAvailableAssets = unstakingVault.balanceOf(asset)` — only the liquid balance sitting in the unstaking vault, which is typically a small fraction of total deposits. [2](#0-1) 

**The blocking loop:**

`_unlockWithdrawalRequests` iterates the queue in strict FIFO order. When the head request's `payoutAmount` exceeds `availableAssetAmount`, the loop unconditionally `break`s: [3](#0-2) 

There is no mechanism to skip the head request and process smaller ones behind it. The entire remaining queue is frozen until the vault accumulates enough assets to cover the head.

**Concrete attack path (no privileges required):**

1. Attacker calls `initiateWithdrawal(asset, largeRsETH, ...)`. This passes because `lrtDepositPool.getTotalAssetDeposits(asset)` is large. The request is assigned nonce `N` and becomes (or will become) the queue head at `nextLockedNonce`.
2. Victim users call `initiateWithdrawal` with small amounts; their requests get nonces `N+1`, `N+2`, …
3. Operator calls `unlockQueue(asset, firstExcludedIndex, ...)` when `unstakingVault.balanceOf(asset)` is in the range `[payoutAmount_large - 1, payoutAmount_large - 1]` — i.e., just below what the head request needs.
4. The loop hits nonce `N`, evaluates `availableAssetAmount < payoutAmount` → `true`, and `break`s immediately.
5. `nextLockedNonce[asset]` is not advanced. Victims' requests remain locked. Their funds are frozen.

---

### Impact Explanation

All withdrawal requests queued behind the oversized head request for that asset are temporarily frozen. Users cannot call `completeWithdrawal` because `usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]` reverts with `WithdrawalLocked`. [4](#0-3) 

The freeze persists until the unstaking vault receives enough assets to cover the head request. Depending on EigenLayer unbonding periods and operator cadence, this can span days to weeks. Impact: **Medium — Temporary freezing of funds**.

---

### Likelihood Explanation

- No special role is required; any rsETH holder can call `initiateWithdrawal`.
- The attacker only needs to queue before the victims (or be the first in a new queue epoch).
- The condition is naturally reachable: the unstaking vault is replenished incrementally as EigenLayer withdrawals complete, so there will routinely be windows where the vault balance is below a large head request.
- The attacker's own funds are locked too, but this is a low-cost griefing attack for a well-capitalised adversary or a competitor.

---

### Recommendation

Replace the hard `break` with a `continue` when the head request cannot be covered, or introduce a "skip-ahead" mechanism for requests whose `payoutAmount` is individually satisfiable by the current vault balance. Alternatively, cap `expectedAssetAmount` at `initiateWithdrawal` time against the unstaking vault balance (not the deposit pool total) so that a request can only be queued when the vault can realistically service it.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Pseudocode for a Foundry fork/local test
function test_largeHeadBlocksSmallTail() public {
    address asset = ETH_TOKEN;
    uint256 largeRsETH = 100 ether; // maps to ~100 ETH payout
    uint256 smallRsETH = 1 ether;   // maps to ~1 ETH payout

    // Step 1: attacker queues large withdrawal first → nonce 0
    vm.prank(attacker);
    withdrawalManager.initiateWithdrawal(asset, largeRsETH, "");

    // Step 2: victim queues small withdrawal → nonce 1
    vm.prank(victim);
    withdrawalManager.initiateWithdrawal(asset, smallRsETH, "");

    // Step 3: advance blocks past withdrawalDelayBlocks
    vm.roll(block.number + withdrawalManager.withdrawalDelayBlocks() + 1);

    // Step 4: vault has 99 ETH — enough for victim, not for attacker
    // (simulate by setting unstakingVault mock balance to 99 ether)
    mockUnstakingVault.setBalance(asset, 99 ether);

    // Step 5: operator calls unlockQueue
    vm.prank(operator);
    withdrawalManager.unlockQueue(asset, 2, minPrice, minPrice, maxPrice, maxPrice);

    // Step 6: nextLockedNonce is still 0 — neither request was unlocked
    assertEq(withdrawalManager.nextLockedNonce(asset), 0);

    // Step 7: victim cannot complete withdrawal
    vm.prank(victim);
    vm.expectRevert(ILRTWithdrawalManager.WithdrawalLocked.selector);
    withdrawalManager.completeWithdrawal(asset, "");
}
```

The assertion at step 6 confirms that even though the vault holds 99 ETH — more than enough for the victim's 1 ETH request — `nextLockedNonce` is never advanced past the attacker's oversized head request, leaving the victim's funds frozen. [5](#0-4)
