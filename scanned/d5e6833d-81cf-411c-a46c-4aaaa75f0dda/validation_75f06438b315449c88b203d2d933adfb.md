### Title
rsETHPrice Inflation via Direct ETH Donation Enables First-Depositor Share Theft - (File: `contracts/LRTOracle.sol`, `contracts/LRTDepositPool.sol`)

---

### Summary

An attacker can become the first depositor in the LRT-rsETH protocol, then directly donate ETH to `LRTDepositPool` and call the public `updateRSETHPrice()` function to inflate the stored `rsETHPrice`. Because `getRsETHAmountToMint` divides by the stored `rsETHPrice`, subsequent depositors who pass `minRSETHAmountExpected = 0` will receive 0 rsETH for their deposits, losing their principal to the attacker. The default configuration (`pricePercentageLimit = 0`) enables this attack without any admin action.

---

### Finding Description

**Step 1 — rsETH minting formula uses a stored, manipulable price.**

`getRsETHAmountToMint` computes:

```
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()
```

`rsETHPrice` is a stored state variable in `LRTOracle`, updated only when `updateRSETHPrice()` is called. [1](#0-0) 

**Step 2 — `updateRSETHPrice()` is public and computes price from `address(this).balance`.**

`_updateRsETHPrice()` calls `_getTotalEthInProtocol()`, which calls `getETHDistributionData()`. That function includes `address(this).balance` (the raw ETH balance of `LRTDepositPool`) in the TVL:

```solidity
ethLyingInDepositPool = address(this).balance;
``` [2](#0-1) 

The new price is then:

```solidity
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
``` [3](#0-2) 

**Step 3 — `LRTDepositPool` accepts unrestricted ETH via `receive()`.**

```solidity
receive() external payable { }
``` [4](#0-3) 

Any address can donate arbitrary ETH to `LRTDepositPool`, which is immediately counted in `totalETHInProtocol`.

**Step 4 — The price-increase guard is disabled by default.**

The only protection against a runaway price increase is:

```solidity
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
``` [5](#0-4) 

`pricePercentageLimit` is never set in `initialize()`, so it defaults to `0`. The guard is therefore permanently disabled until an admin explicitly calls `setPricePercentageLimit`. [6](#0-5) 

**Step 5 — `minRSETHAmountExpected = 0` bypasses the only user-side protection.**

`_beforeDeposit` checks:

```solidity
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [7](#0-6) 

When `minRSETHAmountExpected = 0` and `rsethAmountToMint = 0`, the condition `0 < 0` is false, so the deposit proceeds and the user receives 0 rsETH while their ETH is permanently credited to the protocol.

---

### Impact Explanation

**Critical — Direct theft of user funds.**

A victim who deposits `D` ETH with `minRSETHAmountExpected = 0` after the price has been inflated to `P` receives:

```
rsethAmountToMint = (D * 1e18) / P
```

If `D < P / 1e18`, this rounds to 0. The victim's ETH is absorbed into the protocol's TVL, and the attacker — holding 100% of the rsETH supply — owns the entire pool including the victim's deposit. The attacker can eventually recover all funds through the normal withdrawal path.

Even when `D` is large enough to yield 1 wei rsETH, the victim still loses the fractional remainder (e.g., depositing 1.9 ETH when the price is set to 1 ETH/wei yields only 1 wei rsETH, losing 0.45 ETH to the attacker).

---

### Likelihood Explanation

**Medium.**

- `pricePercentageLimit` is `0` by default, so no admin action is required to enable the attack.
- `minRSETHAmountExpected = 0` is a natural default for users who do not understand slippage, or for integrations that omit the parameter.
- The attack is most effective when the protocol is newly deployed (rsETH supply is minimal), but it can be executed at any time the attacker can acquire a dominant share of rsETH supply.
- The attacker must lock capital (the donated ETH) in the protocol, but recovers it plus the victim's funds through withdrawal.
- Front-running a pending deposit transaction with `minRSETHAmountExpected = 0` is straightforward on any chain with a public mempool.
