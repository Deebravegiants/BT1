### Title
Excess ETH Sent to `updateRate()` Is Permanently Locked in `MultiChainRateProvider` - (File: contracts/cross-chain/MultiChainRateProvider.sol)

### Summary
`MultiChainRateProvider.updateRate()` is a `payable` function that iterates over multiple LayerZero receivers and sends exactly `estimatedFee` to the endpoint for each one. Any ETH in `msg.value` beyond the sum of all per-receiver fees is never refunded and has no recovery path, permanently locking it in the contract.

### Finding Description
`updateRate()` loops over `rateReceivers` and for each entry calls `ILayerZeroEndpoint.send{ value: estimatedFee }(...)`, consuming only the estimated fee per hop. The refund address passed to LayerZero is `payable(msg.sender)`, so LayerZero itself refunds any per-call excess back to the caller. However, the contract never refunds the difference between `msg.value` and the total of all `estimatedFee` values consumed across the loop. That remainder stays in the `MultiChainRateProvider` contract, which has no `receive()` fallback, no `withdraw` function, and no ETH recovery mechanism.

```solidity
// contracts/cross-chain/MultiChainRateProvider.sol
function updateRate() external payable nonReentrant {
    ...
    for (uint256 i; i < rateReceiversLength;) {
        ...
        (uint256 estimatedFee,) = ILayerZeroEndpoint(layerZeroEndpoint)
            .estimateFees(dstChainId, address(this), _payload, false, bytes(""));

        ILayerZeroEndpoint(layerZeroEndpoint).send{ value: estimatedFee }(   // only estimatedFee consumed
            dstChainId, remoteAndLocalAddresses, _payload, payable(msg.sender), address(0x0), bytes("")
        );
        ...
    }
    // msg.value - sum(estimatedFees) is never returned
}
```

Contrast this with `CrossChainRateProvider.updateRate()`, which forwards the entire `msg.value` directly to the endpoint (`send{ value: msg.value }(...)`), so LayerZero's own refund mechanism handles any excess correctly. [1](#0-0) [2](#0-1) 

### Impact Explanation
Any ETH sent above the exact sum of per-receiver `estimatedFee` values is permanently frozen inside `MultiChainRateProvider`. The contract inherits only `Ownable` and `ReentrancyGuard`; it exposes no `withdraw`, no `receive`, and no ETH recovery function. The frozen ETH is irrecoverable.

**Impact: Permanent freezing of funds (Critical per allowed scope).** [3](#0-2) 

### Likelihood Explanation
`updateRate()` is a public, permissionless function. Callers must estimate the total LayerZero fee off-chain before calling. Because fee estimates can shift between the off-chain query and on-chain execution (gas price changes, oracle updates), callers routinely add a safety buffer. Every such over-payment permanently loses the buffer. The function is expected to be called repeatedly by any party wishing to push the rsETH rate cross-chain.

**Likelihood: Medium.**

### Recommendation
After the loop, refund any unspent ETH to `msg.sender`:

```solidity
function updateRate() external payable nonReentrant {
    ...
    for (...) { ... }

    uint256 remaining = address(this).balance;
    if (remaining > 0) {
        (bool ok,) = payable(msg.sender).call{ value: remaining }("");
        require(ok, "ETH refund failed");
    }
    emit RateUpdated(rate);
}
```

Alternatively, compute `totalFee` before the loop using `estimateTotalFee()` and require `msg.value == totalFee`.

### Proof of Concept
1. Call `estimateTotalFee()` off-chain → returns `X` wei.
2. Call `updateRate{ value: X + 1 ether }()` (caller adds a 1 ETH buffer for safety).
3. The loop sends exactly `estimatedFee_i` per receiver; total consumed ≈ `X`.
4. After the function returns, `address(MultiChainRateProvider).balance == 1 ether`.
5. No function exists to recover this ETH; it is permanently locked. [4](#0-3)
