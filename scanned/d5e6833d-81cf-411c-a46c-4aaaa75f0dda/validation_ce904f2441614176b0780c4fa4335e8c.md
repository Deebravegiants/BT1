### Title
Missing Minimum Output Parameter in `instantWithdrawal` Leaves Users Unprotected from Oracle Drift and Fee Deduction - (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary

The `instantWithdrawal` function in `LRTWithdrawalManager` provides no `minAssetAmountExpected` parameter. The gross asset amount is computed from live oracle prices, then a fee is silently deducted, and the net amount is transferred to the user. Because the slippage check is absent entirely, users cannot protect themselves from oracle price movement between submission and execution, nor from the fee reducing their actual receipt below any amount they anticipated.

---

### Finding Description

`instantWithdrawal` follows this sequence:

1. Compute gross output from live oracle prices:
   ```solidity
   uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
   ```
   where `getExpectedAssetAmount` is:
   ```solidity
   underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
   ```

2. Burn rsETH from the caller (irreversible):
   ```solidity
   IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
   ```

3. Deduct the fee **after** the gross amount is fixed:
   ```solidity
   uint256 fee = (assetAmountUnlocked * instantWithdrawalFee) / 10_000;
   uint256 userAmount = assetAmountUnlocked - fee;
   ```

4. Transfer only `userAmount` to the user.

There is no parameter the user can pass to enforce a minimum on `userAmount`. The function signature is:

```solidity
function instantWithdrawal(
    address asset,
    uint256 rsETHUnstaked,
    string calldata referralId
) external ...
```

The code's own NatDoc acknowledges the exposure: *"Uses the fee set at execution time. Managers can raise it right before this call, making withdrawals cost more than expected."* But even without any manager action, oracle prices for rsETH and the underlying asset fluctuate between the block the user signs the transaction and the block it is included. If `rsETHPrice` drops or `getAssetPrice(asset)` rises in that window, `assetAmountUnlocked` shrinks, and the fee is then taken from that already-reduced gross, compounding the shortfall. The user has already committed by burning rsETH and has no recourse.

This is the direct analog of the reported pattern: in the reference issue, `minTokenAmounts_` is checked at the pool-exit level but wstETH is skimmed afterward, making the check useless for the user's actual receipt. Here, there is no check at all on the user's actual receipt after the fee deduction.

---

### Impact Explanation

**Low — Contract fails to deliver promised returns.**

A user who previews the expected output off-chain (e.g., by calling `getExpectedAssetAmount` in the same block) and then submits `instantWithdrawal` may receive materially less than that preview because:

- Oracle prices shift between preview and inclusion.
- The fee is deducted from the (already-shifted) gross amount.

The user's rsETH is burned irreversibly before the net amount is known, so there is no way to abort once the oracle drift is observed. The user loses value relative to their expectation with no on-chain protection.

---

### Likelihood Explanation

**Medium.** Oracle prices for rsETH and LST assets update on every interaction with the oracle contracts and can move meaningfully during periods of market volatility or network congestion. Any user calling `instantWithdrawal` during such a period is exposed. No privileged actor is required; the exposure is inherent to the function's design.

---

### Recommendation

Add a `minAssetAmountExpected` parameter and enforce it **after** the fee deduction:

```solidity
function instantWithdrawal(
    address asset,
    uint256 rsETHUnstaked,
    uint256 minAssetAmountExpected,   // <-- add this
    string calldata referralId
) external ... {
    ...
    uint256 fee = (assetAmountUnlocked * instantWithdrawalFee) / 10_000;
    uint256 userAmount = assetAmountUnlocked - fee;

    if (userAmount < minAssetAmountExpected) revert SlippageExceeded(); // <-- enforce here

    _transferAsset(asset, msg.sender, userAmount);
    ...
}
```

This mirrors the fix recommended in the reference report: allow the user to specify the amount they receive **after** any protocol-side deduction.

---

### Proof of Concept

**Setup:**
- `instantWithdrawalFee = 500` (5 %, the current max is 1000 = 10 %)
- User previews: `getExpectedAssetAmount(ETH, 1e18 rsETH)` → returns `1.05 ETH` (rsETH price = 1.05 ETH/rsETH, asset price = 1 ETH)
- User expects to receive `1.05 ETH × (1 − 0.05) = 0.9975 ETH`

**Attack / adverse scenario (no attacker needed):**
1. Between the user's preview call and block inclusion, the rsETH oracle price updates to `1.00 ETH/rsETH` (a 5 % drop, plausible during volatility).
2. `assetAmountUnlocked = 1e18 * 1.00e18 / 1e18 = 1.00 ETH`
3. `fee = 1.00 ETH * 500 / 10_000 = 0.05 ETH`
4. `userAmount = 0.95 ETH`

The user receives `0.95 ETH` instead of the `0.9975 ETH` they anticipated — a ~4.8 % shortfall — with no on-chain mechanism to have rejected the transaction.

```text
    function instantWithdrawal(
        address asset,
        uint256 rsETHUnstaked,
        string calldata referralId
    )
        external
        nonReentrant
        whenNotPaused
        onlySupportedAsset(asset)
        onlySupportedStrategy(asset)
        onlyInstantWithdrawalAllowed(asset)
    {
        if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
            revert InvalidAmountToWithdraw();
        }
        if (IERC20(lrtConfig.rsETH()).balanceOf(msg.sender) < rsETHUnstaked) revert NotEnoughRsETH();
        uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
        IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
        ILRTUnstakingVault unstakingVault = ILRTUnstakingVault(lrtConfig.getContract(LRTConstants.LRT_UNSTAKING_VAULT));
        if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
            revert CantInstantWithdrawMoreThanAvailable();
        }

        unstakingVault.redeem(asset, assetAmountUnlocked);

        uint256 fee = (assetAmountUnlocked * instantWithdrawalFee) / 10_000;
        uint256 userAmount = assetAmountUnlocked - fee;

        address feeRecipient = instantWithdrawalFeeRecipient;
        if (feeRecipient == address(0)) {
            // Backwards-compatible default: send fees to the protocol treasury
            feeRecipient = lrtConfig.getContract(LRTConstants.PROTOCOL_TREASURY);
        }
        if (fee > 0) {
            _transferAsset(asset, feeRecipient, fee);
            emit InstantWithdrawalFeeCollected(msg.sender, asset, fee);
        }

        _transferAsset(asset, msg.sender, userAmount);
        emit ReferralIdEmitted(referralId);
        emit AssetWithdrawalFinalized(msg.sender, asset, rsETHUnstaked, userAmount);
    }
```

```text
    function getExpectedAssetAmount(
        address asset,
        uint256 amount
    )
        public
        view
        override
        returns (uint256 underlyingToReceive)
    {
        // setup oracle contract
        ILRTOracle lrtOracle = ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE));

        // calculate underlying asset amount to receive based on rsETH amount and asset exchange rate
        underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
    }
```
