### Title
Owner can raise `feeInBPS` at any time to steal unclaimed yield from reward claimants — (File: `contracts/KERNEL/KernelMerkleDistributor.sol`, `contracts/KERNEL/KernelTop100MerkleDistributor.sol`, `contracts/utils/MerkleDistributor/MerkleDistributor.sol`)

---

### Summary

Three merkle distributor contracts share an identical design flaw: the owner can call `setFeeInBPS` at any time — including after a merkle root has been published and users have accrued claimable rewards — to raise the fee up to 10% (1000 BPS). Because the fee is read from live storage at the moment of each `claim` call, the owner can front-run or simply precede a claim wave with a fee increase, diverting up to 10% of every user's earned token rewards to the `protocolTreasury` address. The owner can also freely redirect `protocolTreasury` to any address via `setProtocolTreasury`, completing the theft.

---

### Finding Description

All three contracts compute the user payout at claim time using the current value of `feeInBPS`:

**`MerkleDistributor.claim`** (lines 138–139):
```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;
```

**`KernelMerkleDistributor._processClaim`** (lines 338–339):
```solidity
uint256 fee = (claimableAmount * feeInBPS) / FEE_DENOMINATOR;
uint256 amountToSend = claimableAmount - fee;
```

**`KernelTop100MerkleDistributor.claim`** (lines 328–329):
```solidity
uint256 fee = (claimableAmount * feeInBPS) / FEE_DENOMINATOR;
uint256 amountToSend = claimableAmount - fee;
```

In all three contracts, `feeInBPS` is a mutable state variable with no snapshot taken at the time the merkle root is published. The owner can call `setFeeInBPS` at any time with no restriction on whether a reward epoch is active:

**`MerkleDistributor.setFeeInBPS`** (lines 198–206):
```solidity
function setFeeInBPS(uint256 _feeInBPS) external onlyOwner {
    if (_feeInBPS > MAX_FEE_IN_BPS) revert InvalidFeeInBPS();
    feeInBPS = _feeInBPS;
    emit FeeInBPSUpdated(_feeInBPS);
}
```

The same unrestricted setter exists in `KernelMerkleDistributor` (lines 388–395) and `KernelTop100MerkleDistributor` (lines 426–431). There is no timelock, no minimum notice period, and no requirement that the current reward epoch has ended before the fee can be changed.

The `protocolTreasury` destination is equally unrestricted. `MerkleDistributor.setProtocolTreasury` (lines 172–179), `KernelMerkleDistributor.setProtocolTreasury` (lines 376–381), and `KernelTop100MerkleDistributor.setProtocolTreasury` (lines 416–419) all allow the owner to redirect fee proceeds to any address at any time.

This is structurally identical to the reference vulnerability: a privileged role can reduce user payouts by manipulating a parameter after users have already earned and committed to their rewards, with the difference flowing to an address the privileged role controls.

---

### Impact Explanation

**High — Theft of unclaimed yield.**

The owner can steal up to 10% of all pending token rewards across every active merkle distribution. In `KernelTop100MerkleDistributor`, users are locked into a 30-day vesting schedule (`VESTING_DURATION = 30 days`) and cannot defer their claims indefinitely without forfeiting vested amounts; they are structurally forced to claim during the elevated-fee window. In `KernelMerkleDistributor` and `MerkleDistributor`, users who delay claiming to wait for a fee reduction risk the merkle root being rotated (via `setMerkleRoot`), which would invalidate their prior proof and force them to claim under the new root — still at the elevated fee.

---

### Likelihood Explanation

**Medium.** The attack requires only two sequential, permissionless-to-the-owner transactions (`setProtocolTreasury` + `setFeeInBPS`) with no on-chain barrier. There is no timelock, no multi-sig requirement enforced at the contract level, and no event-based warning system that would give users time to react before their claims are processed. The owner role is a single key (`OwnableUpgradeable`). Any user who claims between the fee-raise transaction and a subsequent fee-reduction transaction pays the inflated fee with no recourse.

---

### Recommendation

Snapshot `feeInBPS` at the time each merkle root is published and bind all claims under that root to the snapshotted value. Alternatively, mirror the reference report's recommendation: restrict `setFeeInBPS` so the fee can only be **decreased** (never increased) once a merkle root is active — i.e., require `_feeInBPS <= feeInBPS` when `currentMerkleRoot != bytes32(0)`. Apply the same restriction to `setProtocolTreasury` while a distribution is live.

---

### Proof of Concept

1. Owner publishes a merkle root via `setMerkleRoot` with `feeInBPS = 0`. Users accrue KERNEL/token rewards and prepare their merkle proofs.
2. Before the claim wave, owner calls `setProtocolTreasury(attackerAddress)`.
3. Owner calls `setFeeInBPS(1000)` (10% — the maximum).
4. Users call `claim(...)`. Inside `_processClaim` / `claim`, the live `feeInBPS = 1000` is read, 10% is deducted from every user's `claimableAmount`, and transferred to `attackerAddress`.
5. Owner has stolen 10% of all claimable rewards. Users received 90% of what the published merkle root entitled them to, with no on-chain recourse.

For `KernelTop100MerkleDistributor`, the attack is amplified: users cannot simply wait because the vesting schedule forces periodic claims, and each claim during the elevated-fee window incurs the 10% haircut.
