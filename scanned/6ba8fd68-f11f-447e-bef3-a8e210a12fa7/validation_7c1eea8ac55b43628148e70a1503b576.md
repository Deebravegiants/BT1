### Title
ETH Deposit Limit Not Enforced Against Deposit Amount - (File: `contracts/LRTDepositPool.sol`)

### Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric limit check: for ERC20 assets it correctly tests `totalAssetDeposits + amount > depositLimit`, but for native ETH it omits the incoming `amount` and only tests `totalAssetDeposits > depositLimit`. Any depositor can therefore push total ETH holdings arbitrarily above the admin-configured cap in a single `depositETH` call.

### Finding Description

`LRTConfig.updateAssetDepositLimit` lets the MANAGER role set a per-asset ceiling (`depositLimitByAsset`) that is meant to cap total protocol exposure. `LRTDepositPool._beforeDeposit` enforces this ceiling by calling `_checkIfDepositAmountExceedesCurrentLimit`:

```solidity
// contracts/LRTDepositPool.sol  L676-L682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct
}
```

For ETH the function answers "is the current total already over the limit?" rather than "would this deposit push the total over the limit?". As long as `totalAssetDeposits < depositLimit` the check returns `false` regardless of how large `amount` is, so `_beforeDeposit` proceeds and `depositETH` mints rsETH for the full deposit.

The call chain is:

```
depositETH(minRSETHAmountExpected, referralId)   [L76-93]
  └─ _beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, ...)  [L87]
       └─ _checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, msg.value)  [L661]
            └─ returns (totalAssetDeposits > limit)   // msg.value never tested
```

The analogous ERC20 path (`depositAsset`) correctly includes `depositAmount` in the comparison and would revert under the same scenario.

### Impact Explanation

**Low** — The contract fails to deliver the promised deposit cap for ETH. The admin-configured `depositLimitByAsset` for ETH is a risk-management control intended to bound the protocol's ETH exposure in EigenLayer. Because the check ignores the incoming deposit amount, a single depositor can push total ETH holdings to an arbitrary multiple of the configured limit in one transaction. The deposit limit is rendered meaningless for ETH, defeating the admin's ability to manage concentration risk.

### Likelihood Explanation

**High** — The condition is met whenever `totalAssetDeposits < depositLimit`, which is the normal operating state of the protocol. No special timing, front-running, or privileged access is required. Any unprivileged depositor calling `depositETH` with a sufficiently large `msg.value` triggers the bypass.

### Recommendation

Include the deposit amount in the ETH branch, matching the ERC20 logic:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept

1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 100 ether` via `LRTConfig.updateAssetDepositLimit`.
2. Current `totalAssetDeposits` for ETH = 99 ether (normal operating state).
3. Attacker calls `depositETH{value: 10_000 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `99 ether > 100 ether` → `false` → no revert.
5. 10 000 ETH is accepted; `totalAssetDeposits` becomes 10 099 ether — 100× the intended cap.
6. The same call with an ERC20 LST would revert at step 4 (`99 + 10_000 > 100` → `true`). [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
