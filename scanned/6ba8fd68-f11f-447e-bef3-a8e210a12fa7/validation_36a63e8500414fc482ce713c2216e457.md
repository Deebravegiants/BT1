### Title
Insufficient Contract Pausability in KernelDepositPool — All User-Facing Fund Functions Are Unpausable - (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary
`KernelDepositPool` has no pause mechanism whatsoever. Unlike `LRTDepositPool`, `LRTWithdrawalManager`, and the L2 pool contracts which all inherit `PausableUpgradeable` and apply `whenNotPaused` to critical functions, `KernelDepositPool` inherits only `AccessControlUpgradeable` and `ReentrancyGuardUpgradeable`. Every user-facing function that moves funds — `stake()`, `initiateWithdrawal()`, `claimWithdrawal()`, and `getReward()` — is permanently unpausable.

### Finding Description
`KernelDepositPool` manages real user funds: KERNEL staking tokens and a separate reward token. The contract declaration is:

```solidity
contract KernelDepositPool is Initializable, AccessControlUpgradeable, ReentrancyGuardUpgradeable {
``` [1](#0-0) 

There is no `PausableUpgradeable` inheritance, no `pause()` / `unpause()` functions, and no `whenNotPaused` modifier anywhere in the file. The four user-accessible fund-handling functions are:

1. `stake()` (line 281) — transfers KERNEL tokens from user into the contract.
2. `initiateWithdrawal()` (line 320) — reduces staked balance and creates a pending withdrawal record.
3. `claimWithdrawal()` (line 344) — transfers KERNEL tokens back to the user.
4. `getReward()` (line 382) — transfers reward tokens to the user. [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

None of these carry any pause guard. Compare this to `LRTDepositPool`, which correctly applies `whenNotPaused` to its analogous `depositETH` and `depositAsset` functions and exposes a `PAUSER_ROLE`-gated `pause()` function: [6](#0-5) [7](#0-6) 

`KernelDepositPool` has no equivalent.

### Impact Explanation
If any vulnerability is discovered in `stake()`, `initiateWithdrawal()`, `claimWithdrawal()`, or `getReward()`, the protocol team has no on-chain mechanism to halt exploitation. All KERNEL tokens staked by users (`totalKernelStaked`) and all reward tokens held in the contract remain continuously accessible to an attacker for as long as the exploit is active. The only recourse would be an upgrade via the proxy, which requires governance delay, giving an attacker an unconstrained window to drain funds.

**Impact: Medium — Temporary (potentially permanent) freezing or theft of user funds due to inability to stop an active exploit.**

### Likelihood Explanation
The contract is a live staking pool holding user KERNEL tokens and reward tokens. Any unprivileged depositor or staker can call all four affected functions. The absence of a pause mechanism is a structural gap, not a conditional one — it applies to every transaction for the lifetime of the contract unless an upgrade is performed. Likelihood is **Medium**: the gap is always present, but exploitation requires a separate vulnerability in the contract logic.

### Recommendation
1. Inherit `PausableUpgradeable` in `KernelDepositPool`.
2. Call `__Pausable_init()` in `initialize()`.
3. Add `whenNotPaused` to `stake()`, `initiateWithdrawal()`, `claimWithdrawal()`, and `getReward()`.
4. Add a `pause()` function gated to a `PAUSER_ROLE` and an `unpause()` function gated to `DEFAULT_ADMIN_ROLE`, consistent with the pattern used in `LRTDepositPool` and `LRTWithdrawalManager`.

### Proof of Concept
1. A vulnerability is found in `claimWithdrawal()` that allows an attacker to claim another user's withdrawal.
2. The protocol team detects the exploit on-chain.
3. They attempt to pause the contract — there is no `pause()` function; the call reverts.
4. The attacker continues draining KERNEL tokens from all pending withdrawals until a proxy upgrade is executed through governance.
5. All KERNEL tokens in `withdrawals` mapping are at risk for the entire governance delay window. [4](#0-3)

        if (_amount == 0) revert AmountZero();
        if (balanceOf[msg.sender] < _amount) revert InsufficientStakedBalance();
        if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();

        balanceOf[msg.sender] -= _amount;
        totalKernelStaked -= _amount;

        // Create a withdrawal record
        uint256 withdrawalId = ++withdrawalCounter;
        uint256 unlockTime = block.timestamp + withdrawalDelay;

        withdrawals[withdrawalId] = Withdrawal({
            user: msg.sender, amount: _amount, unlockTime: unlockTime, claimed: false, withdrawalId: withdrawalId
        });
        userWithdrawalIds[msg.sender].push(withdrawalId);

        emit WithdrawalInitiated(msg.sender, _amount, withdrawalId, unlockTime);
    }

    function claimWithdrawal(uint256 _withdrawalId) external nonReentrant {
        Withdrawal storage withdrawal = withdrawals[_withdrawalId];

        if (withdrawal.user == address(0)) {
            revert WithdrawalDoesNotExist();
        }

        if (withdrawal.user != msg.sender) {
            revert NotYourWithdrawal();
        }

        if (block.timestamp < withdrawal.unlockTime) {
            revert WithdrawalNotReady();
        }

        if (withdrawal.claimed) {
            revert WithdrawalAlreadyClaimed();
        }

        withdrawal.claimed = true;

        // Remove the withdrawal ID from the user's list of withdrawal IDs
        uint256[] storage userWithdrawalIdsArray = userWithdrawalIds[msg.sender];
        for (uint256 i = 0; i < userWithdrawalIdsArray.length; ++i) {
            if (userWithdrawalIdsArray[i] == _withdrawalId) {
                userWithdrawalIdsArray[i] = userWithdrawalIdsArray[userWithdrawalIdsArray.length - 1];
                userWithdrawalIdsArray.pop();
                break;
            }
        }

        // Transfer the KERNEL tokens from contract to user
        kernelToken.safeTransfer(msg.sender, withdrawal.amount);

        emit WithdrawalClaimed(msg.sender, withdrawal.amount, _withdrawalId);
    }
