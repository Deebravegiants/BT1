### Title
`unlockQueue()` Exclusively Gated by Privileged Role with No User Escape Hatch Permanently Freezes Queued rsETH — (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary

Users who call `initiateWithdrawal()` irrevocably transfer their rsETH to `LRTWithdrawalManager`. The only mechanism to advance those requests to a claimable state is `unlockQueue()`, which is restricted to `ASSET_TRANSFER_ROLE` or `OPERATOR_ROLE` holders. No timeout, no user-initiated cancellation, and no fallback path exists. If the privileged role fails to act — due to key loss, operational failure, or any other reason — all queued rsETH is permanently frozen in the contract.

---

### Finding Description

The withdrawal lifecycle in `LRTWithdrawalManager` is a two-step process:

**Step 1 — `initiateWithdrawal()`**: The user's rsETH is pulled into the contract and a `WithdrawalRequest` is recorded.

```solidity
// contracts/LRTWithdrawalManager.sol:166
IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);
```

At this point the rsETH is held by the contract and the user has no way to reclaim it.

**Step 2 — `unlockQueue()`**: A privileged operator must call this function to transition requests from locked to unlocked, enabling users to call `completeWithdrawal()`.

```solidity
// contracts/LRTWithdrawalManager.sol:268-281
function unlockQueue(...) external nonReentrant onlySupportedAsset(asset) whenNotPaused onlyAssetTransferOrOperatorRole
    returns (uint256 rsETHBurned, uint256 assetAmountUnlocked)
```

The `onlyAssetTransferOrOperatorRole` modifier requires the caller to hold `ASSET_TRANSFER_ROLE` or `OPERATOR_ROLE`:

```solidity
// contracts/utils/LRTConfigRoleChecker.sol:48-56
modifier onlyAssetTransferOrOperatorRole() {
    if (
        !IAccessControl(address(lrtConfig)).hasRole(LRTConstants.ASSET_TRANSFER_ROLE, msg.sender)
            && !IAccessControl(address(lrtConfig)).hasRole(LRTConstants.OPERATOR_ROLE, msg.sender)
    ) {
        revert ILRTConfig.CallerNotLRTConfigOperatorOrAssetTransferRole();
    }
    _;
}
```

`unlockQueue()` is the **only** code path that advances `nextLockedNonce[asset]` and increments `unlockedWithdrawalsCount[asset]`, which are the preconditions checked by `completeWithdrawal()`. There is no:
- Time-based fallback allowing users to reclaim rsETH after a deadline
- User-initiated cancellation function
- Emergency withdrawal path for depositors

The `pause()` function (callable by `PAUSER_ROLE`) also blocks `unlockQueue()` via `whenNotPaused`, creating a second independent path to the same freeze outcome.

---

### Impact Explanation

**Critical — Permanent freezing of user funds.**

Once `initiateWithdrawal()` is called, the user's rsETH is held by the contract. If `unlockQueue()` is never called, `completeWithdrawal()` can never succeed for any queued request. The rsETH is permanently locked with no on-chain recovery path available to users.

---

### Likelihood Explanation

**Medium.** The `ASSET_TRANSFER_ROLE` and `OPERATOR_ROLE` are operational keys expected to act regularly. Realistic failure scenarios include: key loss, infrastructure failure, protocol wind-down without a migration path, or a malicious operator deliberately withholding the call. The protocol's own design acknowledges the operator is a required active participant in the withdrawal lifecycle, making this a structural dependency rather than a theoretical edge case.

---

### Recommendation

Add a user-initiated cancellation path that becomes available after a configurable deadline (e.g., `withdrawalDelayBlocks` plus a grace period). After the deadline, users should be able to call a `cancelWithdrawal()` function that returns their rsETH and removes the request from the queue, without requiring any privileged role to act first.

---

### Proof of Concept

1. Alice calls `initiateWithdrawal(stETH, 1e18, "")`. Her 1 rsETH is transferred to `LRTWithdrawalManager` at line 166. `assetsCommitted[stETH]` is incremented.
2. The `ASSET_TRANSFER_ROLE`/`OPERATOR_ROLE` holder loses their key (or goes offline indefinitely). `unlockQueue()` is never called.
3. Alice calls `completeWithdrawal(stETH, "")`. Internally `_processWithdrawalCompletion` checks whether Alice's nonce has been unlocked (i.e., whether `nextLockedNonce[stETH]` has advanced past her nonce). It has not, so the call reverts.
4. Alice has no alternative on-chain path to recover her rsETH. The funds are permanently frozen. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
