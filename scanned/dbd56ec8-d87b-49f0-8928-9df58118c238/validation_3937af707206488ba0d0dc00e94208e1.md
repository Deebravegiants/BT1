### Title
Unchecked `account` Parameter in `MerkleDistributor.claim()` Enables Forced Claims with Fee Extraction — (File: `contracts/utils/MerkleDistributor/MerkleDistributor.sol`)

---

### Summary

`MerkleDistributor.claim()` accepts a caller-supplied `account` address without verifying that `msg.sender == account`. Any unprivileged attacker who possesses a victim's public merkle proof can trigger a claim on the victim's behalf, causing the protocol fee (up to 10%) to be deducted from the victim's allocation without consent. This is structurally analogous to the reported ZK bypass: a verification step that should gate the action (identity check) is simply absent, and the contract proceeds unconditionally.

---

### Finding Description

In `MerkleDistributor.claim()`, the `account` parameter is accepted from the caller without any identity check:

```solidity
// contracts/utils/MerkleDistributor/MerkleDistributor.sol lines 97–147
function claim(
    uint256 index,
    address account,          // ← caller-supplied, never validated against msg.sender
    uint256 cumulativeAmount,
    bytes32[] calldata merkleProof
) external override whenNotPaused {
    ...
    IERC20(token).safeTransfer(account, amountToSend);
    IERC20(token).safeTransfer(protocolTreasury, fee);   // fee deducted from victim
}
```

The sibling contract `KernelMerkleDistributor._processClaim()` correctly enforces the identity check:

```solidity
// contracts/KERNEL/KernelMerkleDistributor.sol lines 311–313
if (account != msg.sender) {
    revert Unauthorized();
}
```

`MerkleDistributor` has no equivalent guard. Because merkle trees and their proofs are published publicly by the protocol's off-chain services (per the README), any attacker can obtain a valid `(index, account, cumulativeAmount, merkleProof)` tuple for any eligible user and submit it themselves.

The fee is bounded by `MAX_FEE_IN_BPS = 1000` (10%) and is configurable up to that ceiling at initialization. When `feeInBPS > 0`, the forced claim permanently deducts `claimableAmount * feeInBPS / 10_000` from the victim's allocation and sends it to `protocolTreasury`. The victim's `lastClaimedIndex` is also updated, preventing any re-claim at the same index.

---

### Impact Explanation

**High — Theft of unclaimed yield.**

A victim with a pending allocation of `X` tokens loses up to `X * 10%` to the protocol treasury without ever authorizing the transaction. The attacker does not receive the fee, but the victim's yield is permanently reduced. This matches the "theft of unclaimed yield" impact category. The loss is irreversible: once `userClaims[account].lastClaimedIndex` is set to `index`, the victim cannot reclaim the fee-deducted portion.

---

### Likelihood Explanation

**High.** The attack requires no privileged access. Merkle proofs are public data derived from the published merkle tree. Any externally owned account can call `claim()` with a victim's proof at any time the contract is unpaused. The attacker bears only gas cost. The attack is most profitable when `feeInBPS` is at its maximum and victim allocations are large.

---

### Recommendation

Add an identity check in `MerkleDistributor.claim()` mirroring the pattern already used in `KernelMerkleDistributor`:

```solidity
if (account != msg.sender) {
    revert Unauthorized();
}
```

This should be placed before the merkle proof verification, consistent with the pattern at `KernelMerkleDistributor.sol` lines 311–313.

---

### Proof of Concept

1. The protocol publishes a merkle tree. Alice's leaf: `keccak256(abi.encodePacked(5, alice, 1000e18))` with proof `P`.
2. `feeInBPS = 1000` (10%), `MAX_FEE_IN_BPS = 1000`.
3. Bob (unprivileged) calls:
   ```solidity
   merkleDistributor.claim(5, alice, 1000e18, P);
   ```
4. `_processClaim` passes: merkle proof is valid, `isClaimed(5, alice)` is false.
5. `claimableAmount = 1000e18 - 0 = 1000e18`.
6. `fee = 1000e18 * 1000 / 10_000 = 100e18` → sent to `protocolTreasury`.
7. `amountToSend = 900e18` → sent to Alice.
8. `userClaims[alice].lastClaimedIndex = 5`, `userClaims[alice].cumulativeAmount = 1000e18`.
9. Alice receives 900e18 instead of 1000e18. She cannot reclaim the 100e18. Bob spent only gas. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```text
    function claim(
        uint256 index,
        address account,
        uint256 cumulativeAmount,
        bytes32[] calldata merkleProof
    )
        external
        override
        whenNotPaused
    {
        if (currentMerkleRoot == bytes32(0)) {
            revert ZeroValueProvided();
        }

        if (index == 0 || index > currentIndex) {
            revert InvalidIndex();
        }

        if (isClaimed(index, account)) {
            revert AlreadyClaimed();
        }

        // Verify the merkle proof.
        bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));
        if (!MerkleProofUpgradeable.verify(merkleProof, currentMerkleRoot, node)) {
            revert InvalidMerkleProof();
        }

        // Calculate the claimable amount
        uint256 claimableAmount = cumulativeAmount - userClaims[account].cumulativeAmount;

        // Ensure there is something to claim
        if (claimableAmount == 0) {
            revert NoTokensToClaim();
        }

        // Update user claim info, and send the token.
        userClaims[account].lastClaimedIndex = index;
        userClaims[account].cumulativeAmount = cumulativeAmount;

        // Send the claimable amount to the user - deducting the fee
        uint256 fee = (claimableAmount * feeInBPS) / 10_000;
        uint256 amountToSend = claimableAmount - fee;

        IERC20(token).safeTransfer(account, amountToSend);

        // Send the fee to the protocol treasury
        IERC20(token).safeTransfer(protocolTreasury, fee);

        emit Claimed(index, account, claimableAmount);
    }
```
