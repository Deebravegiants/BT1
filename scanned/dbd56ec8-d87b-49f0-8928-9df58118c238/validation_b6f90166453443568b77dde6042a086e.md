### Title
LST Tokens Bridged to `L1Vault` Become Permanently Frozen When Asset Is Removed from `LRTConfig` Supported List — (File: contracts/L1Vault.sol)

---

### Summary

`LRTConfig.removeSupportedAsset()` guards against removal only by checking `getTotalAssetDeposits()`, which does not account for LST balances sitting in `L1Vault`. If an LST is removed from the supported list while tokens of that type are in transit inside `L1Vault` (bridged from L2 but not yet deposited into `LRTDepositPool`), those tokens become permanently frozen because `L1Vault.depositAssetForL1Vault()` will revert and no rescue path exists.

---

### Finding Description

The cross-chain LST deposit lifecycle is:

1. User deposits LST (e.g., wstETH) on an L2 pool (`RSETHPoolV3ExternalBridge.deposit(token, amount, ...)`) → wrsETH is minted to the user, fee is charged.
2. The BRIDGER role calls `bridgeTokens(token)` on the L2 pool, which transfers the LST balance to `L1Vault` on L1.
3. The MANAGER role calls `L1Vault.depositAssetForL1Vault(token)`, which calls `LRTDepositPool.depositAsset(token, ...)` to convert the LST into rsETH.

The vulnerability is in step 3 being blocked by a state change that can occur between steps 2 and 3.

`LRTConfig.removeSupportedAsset()` contains the following guard:

```solidity
if (ILRTDepositPool(depositPool).getTotalAssetDeposits(asset) > maxNegligibleAmount) {
    revert CannotRemoveAssetWithDeposits(asset);
}
```

`getTotalAssetDeposits()` sums assets across: the deposit pool itself, node delegators, EigenLayer strategies, the converter, and the unstaking vault. **It does not include the balance of any `L1Vault` contract.** Therefore, if wstETH has been bridged to `L1Vault` but not yet deposited into `LRTDepositPool`, the guard passes and the admin can successfully remove wstETH from the supported list.

After removal, `depositAssetForL1Vault` calls `LRTDepositPool.depositAsset(wstETH, ...)`, which carries the `onlySupportedERC20Token(asset)` modifier. This modifier checks `lrtConfig.isSupportedAsset[asset]`, which is now `false`, causing the call to revert with `AssetNotSupported`. Since `L1Vault` contains no rescue or emergency-withdrawal function for ERC20 tokens, the wstETH is permanently frozen.

---

### Impact Explanation

**Critical. Permanent freezing of funds.**

LST tokens that have been bridged from L2 to `L1Vault` are permanently unrecoverable through any path in the current contract. `L1Vault` exposes no ERC20 rescue function; the only exit path for LST tokens is `depositAssetForL1Vault`, which is now permanently blocked. The wrsETH already minted to users on L2 is no longer backed by a corresponding rsETH position on L1, creating a backing deficit in the wrapper.

---

### Likelihood Explanation

**Low.** The scenario requires the admin to call `removeSupportedAsset` during the window between the BRIDGER bridging the LST to `L1Vault` and the MANAGER depositing it into `LRTDepositPool`. This window exists in normal operations (bridging and depositing are separate, role-gated transactions). The scenario is realistic during planned asset deprecation, where an admin may remove an LST from the supported list without first verifying that all `L1Vault` contracts hold zero balance of that token.

---

### Recommendation

1. **Extend the guard in `removeSupportedAsset`** to also check the balance of each registered `L1Vault` contract for the asset being removed, reverting if any non-negligible balance is found.
2. **Add an ERC20 rescue function to `L1Vault`** (gated by `TIMELOCK_ROLE` or `DEFAULT_ADMIN_ROLE`) to allow recovery of tokens that cannot be deposited due to a changed protocol state.

---

### Proof of Concept

1. User deposits 100 wstETH on `RSETHPoolV3ExternalBridge` → 100 wrsETH minted to user, fee deducted.
2. BRIDGER calls `bridgeTokens(wstETH)` on the L2 pool → 100 wstETH arrives in `L1Vault` on L1.
3. Admin calls `LRTConfig.removeSupportedAsset(wstETH, idx)`:
   - `getTotalAssetDeposits(wstETH)` returns `0` because wstETH is in `L1Vault`, which is not counted.
   - Guard passes; `isSupportedAsset[wstETH]` is deleted.
4. MANAGER calls `L1Vault.depositAssetForL1Vault(wstETH)`:
   - Internally calls `lrtDepositPool.depositAsset(wstETH, 100e18, rsETHAmountToMint, "")`.
   - `onlySupportedERC20Token(wstETH)` checks `lrtConfig.isSupportedAsset[wstETH]` → `false`.
   - Reverts with `AssetNotSupported`.
5. 100 wstETH is permanently frozen in `L1Vault` with no recovery path.

    /// @param token The address of the token to bridge
    function bridgeTokens(address token)
        external
        payable
        nonReentrant
        onlySupportedToken(token)
        onlyRole(BRIDGER_ROLE)
    {
        if (tokenBridge[token] == address(0)) {
            revert MissingBridgeForToken();
        }

        uint256 balance = getTokenBalanceMinusFees(token);

        if (balance == 0) {
            revert ZeroBridgeAmount();
        }

        // Approve the required amount to the bridge
        IERC20(token).safeIncreaseAllowance(tokenBridge[token], balance);

        // Call the bridge contract to transfer the tokens (msg.value is included in case we need to pay for additional
        // bridging fees)
        IL2TokenBridge(tokenBridge[token]).bridgeTokenToL1{ value: msg.value }(l1VaultETHForL2Chain, balance);

        emit BridgedTokenToL1(token, l1VaultETHForL2Chain, balance);
    }
