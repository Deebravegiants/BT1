### Title
ETH Accumulated in `FeeReceiver` Is Excluded from rsETH Price Until `sendFunds()` Is Called — (`contracts/FeeReceiver.sol`)

---

### Summary

`FeeReceiver` accumulates MEV/execution-layer rewards via its `receive()` fallback. These funds are only transferred to `LRTDepositPool` when `sendFunds()` is explicitly called. Until that happens, the ETH sitting in `FeeReceiver` is entirely absent from `getETHDistributionData()`, and therefore from `LRTOracle.rsETHPrice`. Any rsETH holder who redeems during this window receives fewer assets than the true protocol backing warrants.

---

### Finding Description

`FeeReceiver.sendFunds()` is the sole mechanism to move accumulated rewards into the accounting perimeter: [1](#0-0) 

`LRTDepositPool.getETHDistributionData()` counts ETH in the deposit pool itself, each NDC, the unstaking vault, and the converter — but never `FeeReceiver`: [2](#0-1) 

The code even documents this gap explicitly: [3](#0-2) 

`LRTOracle._getTotalEthInProtocol()` derives total protocol ETH exclusively through `getTotalAssetDeposits()` → `getETHDistributionData()`: [4](#0-3) 

`rsETHPrice` is then set from that incomplete total: [5](#0-4) 

---

### Impact Explanation

While `FeeReceiver` holds unforwarded ETH, `rsETHPrice` is understated relative to the true backing. A user who burns rsETH during this window receives fewer ETH than the protocol actually controls on their behalf. The ETH is not lost — it is recovered once `sendFunds()` is called — but the withdrawing user's share of that ETH is permanently diluted because their rsETH has already been burned at the lower price.

**Scope match:** Low — Contract fails to deliver promised returns, but doesn't lose value.

---

### Likelihood Explanation

`sendFunds()` carries no access control and is callable by anyone: [1](#0-0) 

However, there is no on-chain enforcement that it is called before each price update or withdrawal. If the operator bot lags, is paused, or is simply not running, rewards accumulate silently. The longer the gap, the larger the price discrepancy. This is a passive, always-present condition rather than an active exploit.

---

### Recommendation

Include `FeeReceiver.balance` (or the balance of the address registered as `LRTConstants.LRT_FEE_RECEIVER` / `REWARD_RECEIVER`) directly inside `getETHDistributionData()`, so the price oracle always reflects the full protocol-controlled ETH without requiring an intermediate `sendFunds()` call. Alternatively, call `sendFunds()` atomically inside `updateRSETHPrice()` before computing the new price.

---

### Proof of Concept

```solidity
// 1. Deploy FeeReceiver pointing at LRTDepositPool.
// 2. Send 10 ETH to FeeReceiver (simulating MEV rewards).
// 3. Record price1 = LRTOracle.rsETHPrice() — FeeReceiver ETH is absent.
// 4. Call FeeReceiver.sendFunds().
// 5. Call LRTOracle.updateRSETHPrice().
// 6. Record price2 = LRTOracle.rsETHPrice().
// 7. Assert price2 > price1, proportional to the 10 ETH withheld.
// A user who redeemed rsETH between steps 3 and 4 received assets
// priced at price1, not price2, losing the yield represented by
// the unforwarded FeeReceiver balance.
```

```text
    /// @dev provides ETH amount distribution data among depositPool, NDCs and eigenLayer
    /// @dev rewards are not accounted here
    /// it will automatically be accounted once it is moved from feeReceiver/rewardReceiver to depositPool
    function getETHDistributionData()
        public
        view
        override
        returns (
            uint256 ethLyingInDepositPool,
            uint256 ethLyingInNDCs,
            uint256 ethStakedInEigenLayer,
            uint256 ethUnstakingFromEigenLayer,
            uint256 ethLyingInConverter,
            uint256 ethLyingInUnstakingVault
        )
    {
        ethLyingInDepositPool = address(this).balance;

        uint256 ndcsCount = nodeDelegatorQueue.length;

        for (uint256 i; i < ndcsCount;) {
            ethLyingInNDCs += nodeDelegatorQueue[i].balance;

            ethStakedInEigenLayer += INodeDelegator(nodeDelegatorQueue[i]).getEffectivePodShares();
            ethUnstakingFromEigenLayer += INodeDelegator(nodeDelegatorQueue[i])
                .getAssetUnstaking(LRTConstants.ETH_TOKEN);
            unchecked {
                ++i;
            }
        }

        address lrtUnstakingVault = lrtConfig.getContract(LRTConstants.LRT_UNSTAKING_VAULT);
        ethLyingInUnstakingVault = lrtUnstakingVault.balance;

        address lrtConverter = lrtConfig.getContract(LRTConstants.LRT_CONVERTER);
        ethLyingInConverter = ILRTConverter(lrtConverter).ethValueInWithdrawal();
    }
```

```text
    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
```
