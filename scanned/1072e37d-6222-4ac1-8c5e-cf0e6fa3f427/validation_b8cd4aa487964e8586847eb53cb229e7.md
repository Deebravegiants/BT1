### Title
Stale `rsETHPrice` Allows MEV Bots to Steal Yield from Legitimate Stakers Before Price Update - (File: contracts/LRTOracle.sol)

### Summary

The `LRTOracle` stores `rsETHPrice` as a state variable that is only updated when `updateRSETHPrice()` is explicitly called. Between updates, if rewards have accrued (e.g., ETH sitting in `FeeReceiver`), the stored price is stale. Because `LRTDepositPool.depositETH()` mints rsETH using the stale stored price, and both `FeeReceiver.sendFunds()` and `LRTOracle.updateRSETHPrice()` are permissionless, an attacker can deposit at the stale (lower) price, trigger the price update, and exit via DEX at the new higher price — stealing yield from legitimate stakers.

### Finding Description

`LRTOracle.rsETHPrice` is a stored value updated only on explicit calls to `updateRSETHPrice()`:

```solidity
// LRTOracle.sol line 87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

`LRTDepositPool.getRsETHAmountToMint()` uses this stored price to compute how many rsETH tokens to mint:

```solidity
// LRTDepositPool.sol line 520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

If `rsETHPrice` is stale (lower than the actual backing per rsETH), a depositor receives **more rsETH than they are entitled to**.

`FeeReceiver.sendFunds()` is permissionless — anyone can push accumulated MEV/staking rewards from `FeeReceiver` into the deposit pool:

```solidity
// FeeReceiver.sol lines 53-58
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

`updateRSETHPrice()` is also permissionless. Together, these two calls can be sequenced by any external actor to trigger a price update at will.

The `_updateRsETHPrice()` internal function computes `previousTVL` using the **stored** (stale) `rsETHPrice` multiplied by the **current** `rsethSupply` (which already includes the attacker's newly minted rsETH):

```solidity
// LRTOracle.sol lines 234-250
uint256 previousTVL = rsethSupply.mulWad(rsETHPrice);
// ...
if (!protocolPaused && totalETHInProtocol > previousTVL) {
    uint256 rewardAmount = totalETHInProtocol - previousTVL;
    protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
}
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

Because the attacker's deposit is included in `rsethSupply` when `previousTVL` is computed, the reward increment (`totalETHInProtocol - previousTVL`) is correctly attributed to the reward ETH — but the attacker's rsETH position now participates in the price appreciation, diluting the yield of pre-existing stakers.

### Impact Explanation

**High — Theft of unclaimed yield.**

Legitimate stakers lose a proportional share of pending rewards to the attacker. In the concrete example below, original stakers lose 50% of their expected yield. The attack is repeatable every time rewards accumulate in `FeeReceiver` before `updateRSETHPrice()` is called.

### Likelihood Explanation

**Medium.** The conditions required are:
1. Rewards have accumulated in `FeeReceiver` (routine — MEV/staking rewards arrive continuously).
2. `rsETHPrice` has not been updated since those rewards arrived (routine — price updates are periodic, not per-block).
3. Sufficient DEX liquidity exists for rsETH (rsETH is a major LRT with active secondary markets).
4. Flash loan capital is available (universally available on Ethereum).

All four conditions are routinely satisfied. The attack is fully permissionless and automatable by MEV bots.

### Recommendation

Call `updateRSETHPrice()` (or an equivalent internal price snapshot) at the beginning of `depositETH()` and `depositAsset()` in `LRTDepositPool`, so that the price used for minting always reflects the current TVL including any pending rewards. Alternatively, make `FeeReceiver.sendFunds()` access-controlled so that rewards can only be pushed by a trusted operator who also updates the price atomically.

### Proof of Concept

**Setup:**
- `rsETHPrice` = 1.0 ETH/rsETH (stored, not yet updated)
- `rsethSupply` = 1,000 rsETH
- `totalETHInProtocol` = 1,000 ETH (price is accurate)
- `FeeReceiver` holds 100 ETH in accumulated MEV rewards (not yet sent)

**Attack (single transaction via flash loan):**

1. Attacker flash-loans 1,000 ETH.
2. Attacker calls `LRTDepositPool.depositETH{value: 1000 ETH}(0, "")`.
   - `rsETHAmountToMint = 1000e18 * 1e18 / 1e18 = 1000 rsETH` (at stale price 1.0).
   - `rsethSupply` → 2,000 rsETH; deposit pool ETH → 2,000 ETH.
3. Attacker calls `FeeReceiver.sendFunds()`.
   - 100 ETH moves to deposit pool; `totalETHInProtocol` → 2,100 ETH.
4. Attacker calls `LRTOracle.updateRSETHPrice()`.
   - `previousTVL = 2000 rsETH × 1.0 = 2000 ETH`
   - `rewardAmount = 2100 − 2000 = 100 ETH`
   - `protocolFeeInETH = 100 × 10% = 10 ETH` (example fee)
   - `newRsETHPrice = (2100 − 10) / 2000 = 1.045 ETH/rsETH`
5. Attacker sells 1,000 rsETH on a DEX at 1.045 ETH/rsETH → receives **1,045 ETH**.
6. Attacker repays 1,000 ETH flash loan. **Net profit: 45 ETH.**

**Without the attack**, original stakers (1,000 rsETH) would have received 90 ETH of yield (100 ETH reward minus 10 ETH fee), raising their price to 1.09 ETH/rsETH. With the attack, they only receive 45 ETH of yield (price 1.045 ETH/rsETH). **The attacker extracted 45 ETH of yield that belonged to legitimate stakers.** [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
