### Title
Missing Stale Price Check in `ChainlinkPriceOracle.getAssetPrice()` Corrupts rsETH Price Calculation - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all staleness-related return values (`updatedAt`, `answeredInRound`, `roundId`). No zero-price guard and no heartbeat check are applied. This is the exact same vulnerability class as the reference report. The price returned feeds directly into `LRTOracle._updateRsETHPrice()`, which is publicly callable, meaning any unprivileged user can trigger a price update using a stale or zero oracle answer.

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the Chainlink price as follows:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All five return values of `latestRoundData()` are available — `(roundId, answer, startedAt, updatedAt, answeredInRound)` — but only `answer` is used. The contract never checks:
- `price > 0` (Chainlink returns 0 when no answer has been reached)
- `updatedAt` against a maximum staleness threshold (heartbeat)
- `answeredInRound >= roundId` (incomplete round detection) [1](#0-0) 

By contrast, the sibling contract `ChainlinkOracleForRSETHPoolCollateral` in the same repository performs all three checks correctly: [2](#0-1) 

The stale price flows through the following call chain, all of which are reachable by any unprivileged caller:

1. `LRTOracle.updateRSETHPrice()` (public, no access control) → `_updateRsETHPrice()`
2. `_updateRsETHPrice()` calls `_getTotalEthInProtocol()`
3. `_getTotalEthInProtocol()` calls `getAssetPrice(asset)` for every supported asset
4. `getAssetPrice()` delegates to `ChainlinkPriceOracle.getAssetPrice()` which returns the unchecked stale/zero price [3](#0-2) [4](#0-3) 

### Impact Explanation

**Scenario A — Stale low price triggers false downside protection pause:**
If Chainlink's feed goes stale and the last reported price is significantly below the current market price, `_getTotalEthInProtocol()` underestimates TVL. The computed `newRsETHPrice` drops below `highestRsethPrice` by more than `pricePercentageLimit`. The protocol then auto-pauses `LRTDepositPool`, `LRTWithdrawalManager`, and `LRTOracle` itself: [5](#0-4) 

This constitutes **temporary freezing of funds** — all deposits and withdrawals are halted until an admin manually unpauses.

**Scenario B — Zero price causes division-by-zero or zeroed TVL:**
If `price == 0`, `uint256(price) * 1e18 / 10 ** decimals` returns 0. The affected asset contributes 0 ETH to `totalETHInProtocol`, artificially deflating the rsETH price. New depositors receive more rsETH than they are entitled to, diluting existing holders — constituting **theft of unclaimed yield**.

### Likelihood Explanation

`updateRSETHPrice()` is a public function with no access control, callable by any EOA or contract. Chainlink feeds do occasionally go stale during network congestion or oracle node outages. The ETH/LST feeds used by this protocol (stETH/ETH, cbETH/ETH, etc.) have heartbeats of 1–24 hours; a missed update within that window is a realistic, documented Chainlink behavior. No special attacker capability is required — the attacker simply waits for a stale round and calls `updateRSETHPrice()`.

### Recommendation

Apply the same pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > MAX_ORACLE_DELAY) revert StalePrice(); // e.g. 3600s for ETH/USD

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

A per-feed configurable `maxDelay` is preferable since different Chainlink feeds have different heartbeat intervals.

### Proof of Concept

1. Chainlink's ETH/LST feed goes stale (last `updatedAt` > heartbeat ago, or `answeredInRound < roundId`).
2. Attacker (any EOA) calls `LRTOracle.updateRSETHPrice()`.
3. `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)` which returns the stale (artificially low) price without reverting.
4. `newRsETHPrice` is computed as significantly lower than `highestRsethPrice`.
5. The downside protection branch executes: `lrtDepositPool.pause()`, `withdrawalManager.pause()`, `_pause()` — the entire protocol is frozen.
6. All user deposits and withdrawals are blocked until admin intervention. [1](#0-0) [6](#0-5) [5](#0-4)
