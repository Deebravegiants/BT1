### Title
`setToken` Can Be Reset to a Different Address, Permanently Freezing Unclaimed Yield - (File: `contracts/utils/MerkleDistributor/MerkleDistributor.sol`)

### Summary
`MerkleDistributor.setToken` only guards against `address(0)` but does not prevent the owner from replacing an already-set token address with a different one. Because `claim()` always reads the current `token` state variable when transferring rewards, changing the token while users have pending claims causes those claims to revert permanently, freezing all previously deposited reward tokens inside the contract.

### Finding Description
`setToken` in `MerkleDistributor` performs only a zero-address check before overwriting the `token` state variable:

```solidity
function setToken(address _token) external onlyOwner {
    if (_token == address(0)) {
        revert ZeroValueProvided();
    }
    token = _token;          // ← no guard against overwriting an existing token
    emit TokenUpdated(_token);
}
``` [1](#0-0) 

The `claim` function unconditionally uses the current value of `token` when transferring rewards to users:

```solidity
IERC20(token).safeTransfer(account, amountToSend);
IERC20(token).safeTransfer(protocolTreasury, fee);
``` [2](#0-1) 

If the owner calls `setToken(tokenB)` after the contract has already been funded with `tokenA` and a merkle root has been published, every subsequent `claim()` call will attempt to transfer `tokenB`. Since the contract holds `tokenA` (not `tokenB`), all transfers revert. The `tokenA` balance is stranded in the contract with no withdrawal path for users.

The `initialize` function explicitly allows `token` to be `address(0)` at deployment time (comment: *"token can be set later"*), which is the intended first-time use of `setToken`. However, there is no guard preventing a second call once the token is already set and rewards are in-flight. [3](#0-2) 

### Impact Explanation
**Medium — Permanent freezing of unclaimed yield.**

All reward tokens (`tokenA`) deposited into the contract become unclaimable the moment `setToken(tokenB)` is called. Users' `userClaims` accounting (cumulative amounts, indices) remains intact but is rendered useless because every `claim()` call reverts. There is no rescue or sweep function for the stranded `tokenA` balance, so the freeze is permanent unless the owner calls `setToken(tokenA)` again — which itself is an error-prone manual recovery step with no on-chain enforcement. [4](#0-3) 

### Likelihood Explanation
The owner is a single EOA or multisig with unrestricted ability to call `setToken` at any time. The contract is designed to support multiple reward epochs (`setMerkleRoot` increments `currentMerkleRootIndex`), meaning `setToken` is expected to be callable post-deployment. An accidental or mistaken call with a wrong address — or a deliberate upgrade to a new reward token without first draining the old one — directly triggers the freeze. No attacker action is required; the owner's own privileged call is the trigger. [5](#0-4) 

### Recommendation
Mirror the fix recommended in the referenced report: add a guard that prevents overwriting an already-set token address unless the contract holds no balance of the current token (or the token has never been set):

```solidity
function setToken(address _token) external onlyOwner {
    if (_token == address(0)) revert ZeroValueProvided();
    require(
        token == address(0) || IERC20(token).balanceOf(address(this)) == 0,
        "Token already set with outstanding balance"
    );
    token = _token;
    emit TokenUpdated(_token);
}
```

This ensures the token can only be changed when no reward funds are at risk of being frozen.

### Proof of Concept
1. Owner deploys `MerkleDistributor` and calls `setToken(tokenA)`.
2. Owner funds the contract with 1,000 `tokenA` and calls `setMerkleRoot(root1)`, publishing claims for users.
3. Users have not yet claimed their rewards.
4. Owner (accidentally or intentionally) calls `setToken(tokenB)`.
5. Any user calling `claim(index, account, amount, proof)` now hits `IERC20(tokenB).safeTransfer(account, amount)`. The contract holds zero `tokenB`, so `safeTransfer` reverts with an ERC-20 insufficient-balance error.
6. The 1,000 `tokenA` tokens remain permanently locked in the contract with no user-accessible withdrawal path. [1](#0-0)

    function claim(
        uint256 index,
        address account,
        uint256 cumulativeAmount,
        bytes32[] calldata merkleProof
    )
        external
        override
        whenNotPaused
    {
        if (currentMerkleRoot == bytes32(0)) {
            revert ZeroValueProvided();
        }

        if (index == 0 || index > currentIndex) {
            revert InvalidIndex();
        }

        if (isClaimed(index, account)) {
            revert AlreadyClaimed();
        }

        // Verify the merkle proof.
        bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));
        if (!MerkleProofUpgradeable.verify(merkleProof, currentMerkleRoot, node)) {
            revert InvalidMerkleProof();
        }

        // Calculate the claimable amount
        uint256 claimableAmount = cumulativeAmount - userClaims[account].cumulativeAmount;

        // Ensure there is something to claim
        if (claimableAmount == 0) {
            revert NoTokensToClaim();
        }

        // Update user claim info, and send the token.
        userClaims[account].lastClaimedIndex = index;
        userClaims[account].cumulativeAmount = cumulativeAmount;

        // Send the claimable amount to the user - deducting the fee
        uint256 fee = (claimableAmount * feeInBPS) / 10_000;
        uint256 amountToSend = claimableAmount - fee;

        IERC20(token).safeTransfer(account, amountToSend);

        // Send the fee to the protocol treasury
        IERC20(token).safeTransfer(protocolTreasury, fee);

        emit Claimed(index, account, claimableAmount);
    }
