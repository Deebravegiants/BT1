### Title
Stale Cross-Chain rsETH Rate Accepted Without Staleness Check Leads to Protocol Insolvency - (File: contracts/cross-chain/CrossChainRateReceiver.sol)

### Summary
`CrossChainRateReceiver.getRate()` returns the stored `rate` unconditionally, with no check against `lastUpdated`. All L2 deposit pools (`RSETHPoolV3`, `RSETHPoolV2`, `RSETHPoolNoWrapper`, etc.) use this rate to compute how many wrsETH tokens to mint per deposited ETH. Because rsETH appreciates monotonically over time, a stale (too-low) rate causes the pools to over-mint wrsETH relative to the rsETH that will actually be received when the bridged ETH is deposited on L1, creating a permanent insolvency shortfall.

### Finding Description
The cross-chain rate pipeline works as follows:

1. On L1, `RSETHMultiChainRateProvider` (or `RSETHRateProvider`) reads `LRTOracle.rsETHPrice()` and sends it via LayerZero to each L2 `CrossChainRateReceiver` when `updateRate()` is called by anyone willing to pay the gas.
2. On L2, `CrossChainRateReceiver.lzReceive()` stores the received value in `rate` and records `lastUpdated = block.timestamp`.
3. Every L2 pool calls `IOracle(rsETHOracle).getRate()` which resolves to `CrossChainRateReceiver.getRate()`, returning `rate` with no staleness guard.

```
// CrossChainRateReceiver.sol line 103-105
function getRate() external view returns (uint256) {
    return rate;   // lastUpdated is stored but NEVER checked here
}
```

The minting formula in every pool variant is:

```
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

If `rsETHToETHrate` is stale and lower than the true current rate, `rsETHAmount` is inflated. The deposited ETH is later bridged to L1 via `L1Vault`, deposited into `LRTDepositPool`, and the resulting rsETH (computed at the *actual* current rate) is bridged back to back the wrsETH. Because the actual rate is higher than the stale rate, the rsETH received on L1 is less than the wrsETH already minted on L2.

By contrast, `ChainlinkOracleForRSETHPoolCollateral` — used for collateral-token prices in the same pools — explicitly reverts on stale data (`answeredInRound < roundID`, `timestamp == 0`). No equivalent guard exists for the rsETH/ETH rate.

The rate update is permissionless but not automatic; it requires an external caller to pay LayerZero fees. During periods of low activity, network congestion, or LayerZero degradation, the rate can lag by hours or days. rsETH accrues staking rewards continuously, so the true rate always drifts upward from any stale snapshot.

### Impact Explanation
Every deposit made while the rate is stale mints more wrsETH than the ETH deposited can redeem in rsETH. The shortfall is permanent: the L1Vault converts ETH to rsETH at the real rate, producing fewer rsETH tokens than the outstanding wrsETH supply. Repeated deposits under a stale rate accumulate an ever-growing insolvency gap, directly analogous to the bad-debt accumulation described in the reference report. This is **Critical — protocol insolvency**.

### Likelihood Explanation
`updateRate()` is not called automatically; it depends on off-chain actors paying LayerZero fees. rsETH's rate increases with every EigenLayer reward epoch. Even a modest lag of a few hours during high-reward periods produces a measurable over-mint. Any unprivileged depositor can exploit this passively by depositing whenever the on-chain rate lags the true rate — no special access or front-running is required.

### Recommendation
Add a configurable maximum staleness threshold to `CrossChainRateReceiver.getRate()` and revert if `block.timestamp - lastUpdated` exceeds it:

```solidity
uint256 public maxStaleness; // e.g. 24 hours

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

Additionally, consider incentivizing or automating `updateRate()` calls (e.g., via a keeper or Chainlink Automation) so the rate is refreshed at least once per reward epoch.

### Proof of Concept

**Setup:**
- L1 `LRTOracle.rsETHPrice()` = 1.05e18 (rsETH has appreciated 5% since last update)
- L2 `CrossChainRateReceiver.rate` = 1.00e18 (stale — last updated 48 hours ago)
- `CrossChainRateReceiver.lastUpdated` = 48 hours ago (ignored by `getRate()`)

**Execution:**
1. Attacker calls `RSETHPoolV3.deposit{value: 100 ether}("")`.
2. Pool calls `getRate()` → returns stale `1.00e18`.
3. `rsETHAmount = 100e18 * 1e18 / 1.00e18 = 100e18` wrsETH minted to attacker.
4. Bridger calls `moveAssetsForBridging()`, 100 ETH is sent to L1Vault.
5. L1Vault calls `LRTDepositPool.depositETH()` at actual rate 1.05e18 → mints `100e18 * 1e18 / 1.05e18 ≈ 95.24e18` rsETH.
6. 95.24 rsETH is bridged back to L2 to back the wrsETH supply.
7. **Shortfall: 100 wrsETH outstanding, only 95.24 rsETH backing it — 4.76 rsETH deficit per 100 ETH deposited.**

Repeated across all L2 pools and all deposits during the staleness window, the deficit compounds into protocol insolvency.
