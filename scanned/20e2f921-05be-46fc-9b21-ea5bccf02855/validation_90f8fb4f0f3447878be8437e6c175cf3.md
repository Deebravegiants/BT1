### Title
Lack of Slippage Control on `MetricOmmPool.removeLiquidity` - (File: metric-core/contracts/MetricOmmPool.sol)

### Summary

`MetricOmmPool.removeLiquidity` burns LP shares and transfers underlying tokens directly to the owner with no minimum-amount-out parameters. The token amounts returned are computed from live bin balances that change with every swap, so a front-running swap or oracle-driven price move between tx submission and execution can silently reduce the tokens an LP receives with no on-chain protection.

### Finding Description

`removeLiquidity` is the only pool action that transfers real token principal to a user without any slippage guard. Its signature accepts only the shares to burn and opaque extension bytes; it returns `(amount0Removed, amount1Removed)` but provides no way for the caller to specify minimum acceptable amounts:

```solidity
function removeLiquidity(address owner, uint80 salt, LiquidityDelta calldata deltas, bytes calldata extensionData)
    external
    nonReentrant(PoolActions.REMOVE_LIQUIDITY)
    returns (uint256 amount0Removed, uint256 amount1Removed)
``` [1](#0-0) 

Inside `LiquidityLib.removeLiquidity`, the amounts are computed as a proportional share of the live bin balances:

```solidity
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
uint256 amount1Scaled = _checkedMul(binState.token1BalanceScaled, sharesToRemove) / binTotalSharesVal;
``` [2](#0-1) 

`binState.token0BalanceScaled` and `binState.token1BalanceScaled` are mutated by every swap that crosses the bin. A swap that drains token0 from a bin and adds token1 (or vice versa) directly reduces the token0 the LP receives when they remove liquidity from that bin. Because the pool's oracle-driven bid/ask spread can shift the active bin and trigger cascading swaps, the composition of tokens in any given bin can change substantially between the block a `removeLiquidity` tx is submitted and the block it is included.

There is no periphery wrapper for `removeLiquidity` — unlike `addLiquidity` (which has `MetricOmmPoolLiquidityAdder` with `maxAmountToken0`/`maxAmountToken1` caps) and swaps (which have `amountOutMinimum`/`amountInMaximum` in `MetricOmmSimpleRouter`), the LP must call the pool directly with no slippage protection available. [3](#0-2) [4](#0-3) 

### Impact Explanation

An LP who submits a `removeLiquidity` transaction expecting to receive at least `X` token0 and `Y` token1 has no on-chain mechanism to enforce that expectation. A sandwich attacker can:

1. Front-run with a large swap that drains token0 from the target bin (shifting the bin balance heavily toward token1).
2. Allow the LP's `removeLiquidity` to execute, receiving far less token0 than expected.
3. Back-run to restore the price.

The LP suffers a direct loss of principal with no recourse. The loss magnitude scales with the size of the manipulating swap relative to the bin's liquidity depth. This is a medium-severity direct loss of LP assets.

### Likelihood Explanation

Every LP who calls `removeLiquidity` directly on the pool is exposed. Because `msg.sender == owner` is required, the LP cannot route through a protective wrapper. The attack requires only a standard sandwich setup (no privileged access, no malicious setup). The oracle-driven bid/ask spread means price can move legitimately between blocks, making even non-adversarial slippage a routine risk for LPs in volatile markets.

### Recommendation

Add `minAmount0Removed` and `minAmount1Removed` parameters to `removeLiquidity` and revert if the computed amounts fall below them:

```solidity
function removeLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    uint256 minAmount0Removed,
    uint256 minAmount1Removed,
    bytes calldata extensionData
) external nonReentrant(PoolActions.REMOVE_LIQUIDITY) returns (uint256 amount0Removed, uint256 amount1Removed) {
    // ... existing logic ...
    if (amount0Removed < minAmount0Removed || amount1Removed < minAmount1Removed) {
        revert InsufficientOutput(amount0Removed, amount1Removed, minAmount0Removed, minAmount1Removed);
    }
}
```

Alternatively, add a periphery `removeLiquidity` wrapper (analogous to `MetricOmmPoolLiquidityAdder`) that accepts minimum-amount parameters and reverts post-removal if the amounts are insufficient, without requiring a core contract change.

### Proof of Concept

1. LP holds 10 000 shares in bin 0 (the current bin), which currently holds 50 token0 and 50 token1 scaled.
2. LP submits `removeLiquidity` for all 10 000 shares, expecting ~50 token0 and ~50 token1.
3. Attacker front-runs with a large `zeroForOne` swap that drains all token0 from bin 0 and fills it with token1. After the swap, bin 0 holds 0 token0 and 100 token1 scaled.
4. LP's `removeLiquidity` executes: `amount0Scaled = 0 * 10000 / 10000 = 0`, `amount1Scaled = 100 * 10000 / 10000 = 100`. LP receives 0 token0 instead of the expected ~50.
5. Attacker back-runs to restore the price.

The LP lost ~50 token0 of principal with no on-chain protection because `removeLiquidity` accepts no minimum-amount parameters. [5](#0-4) [1](#0-0)
