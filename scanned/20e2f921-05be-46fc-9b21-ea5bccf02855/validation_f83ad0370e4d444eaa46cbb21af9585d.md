### Title
LP Fee Sandwich: Attacker Captures Existing LPs' Swap Fees by Sandwiching Swaps with Proportional Liquidity Add/Remove — (`metric-core/contracts/libraries/LiquidityLib.sol`)

---

### Summary

LP fees (the non-protocol portion of the spread fee) are deposited directly into each bin's token balance during a swap. Because `addLiquidity` and `removeLiquidity` both use the **current** bin balance proportionally, with no time-lock or fee-on-entry, an attacker can sandwich any swap: add a dominant share of liquidity before the swap, let the LP fee land in the bin, then immediately remove that liquidity and extract the fee. Existing LPs lose the fees they earned.

---

### Finding Description

**How LP fees accumulate in bins**

In `SwapMath` (e.g., `buyToken1InBinSpecifiedIn`), the LP fee is the total input minus the protocol-fee portion, and it is added directly to the bin's token balance:

```solidity
// SwapMath.sol ~L780-781
binState.token0BalanceScaled =
  (uint256(binState.token0BalanceScaled) + totalIn0Scaled - protocolFeeAmountScaled).toUint104();
```

The LP fee (`totalIn0Scaled - protocolFeeAmountScaled - in0WithoutFeeScaled`) is therefore immediately part of the bin balance, claimable by whoever holds shares at the moment of removal.

**How liquidity add/remove is priced**

`LiquidityLib.addLiquidity` prices new shares proportionally to the **current** bin balance:

```solidity
// LiquidityLib.sol L109-110
amount0Scaled = Math.ceilDiv(_checkedMul(binState.token0BalanceScaled, sharesToAdd), binTotalSharesVal);
amount1Scaled = Math.ceilDiv(_checkedMul(binState.token1BalanceScaled, sharesToAdd), binTotalSharesVal);
```

`LiquidityLib.removeLiquidity` returns tokens proportionally to the **current** bin balance at removal time:

```solidity
// LiquidityLib.sol L205-206
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
uint256 amount1Scaled = _checkedMul(binState.token1BalanceScaled, sharesToRemove) / binTotalSharesVal;
```

There is no time-lock, no fee-on-entry, and no time-weighted share mechanism on either path.

**The sandwich attack**

Let bin X have balance `B` token0 and `T` total shares (held by honest LPs). An attacker:

1. **Front-runs** the swap by calling `addLiquidity` with `S` shares (where `S >> T`), paying `B * S / T` token0 (proportional to current balance, which does not yet include the fee).
2. The swap executes; LP fee `F` token0 enters the bin. Bin balance is now `B + F`, total shares `T + S`.
3. **Back-runs** with `removeLiquidity` for `S` shares, receiving `(B + F) * S / (T + S)` token0.

**Attacker profit** = `(B + F) * S / (T + S) − B * S / T = F * S * T / (T * (T + S)) ≈ F` when `S >> T`.

Honest LPs receive only `(B + F) * T / (T + S) ≈ B * T / S → 0` of the fee, instead of the full `F` they earned.

The attacker can use a flash loan to make `S` arbitrarily large relative to `T`, capturing nearly 100% of the LP fee at negligible cost beyond gas and flash-loan fees.

---

### Impact Explanation

Existing LP providers lose their earned swap fees to the attacker. The loss is proportional to the attacker's share dominance and the LP fee generated by the sandwiched swap. With a flash loan, the attacker can dominate any bin and extract essentially all LP fees from any swap, making LP provision economically unviable. This is a direct, repeatable loss of LP principal/yield with no protocol-level guard.

---

### Likelihood Explanation

Any swap that generates a non-zero LP fee (i.e., any swap where `spreadFeeE6 < 1e6` so the LP portion is positive) is exploitable. The attacker only needs:
- MEV capability to front-run/back-run in the same block (standard on most EVM chains)
- A flash loan source for the capital (widely available)
- No special permissions — `addLiquidity` and `removeLiquidity` are permissionless

There are no existing guards (no time-lock, no fee-on-entry, no withdrawal delay) that prevent this.

---

### Recommendation

Implement one or more of the following mitigations:

1. **Fee-on-entry**: When adding liquidity to a non-empty bin, charge the new LP a proportional share of the **accrued** LP fees (i.e., the difference between current bin balance and the "principal" balance). This prevents the attacker from entering at the pre-fee price and exiting at the post-fee price.

2. **Withdrawal delay / time-lock**: Require a minimum holding period (e.g., one block or a configurable delay) between `addLiquidity` and `removeLiquidity` for the same position, preventing same-block sandwich attacks.

3. **Snapshot-based fee accounting**: Track a per-share fee accumulator (similar to Uniswap v3's `feeGrowthInside`) so that fees are attributed only to shares that existed at the time of the swap, not to shares added after.

---

### Proof of Concept

```
State: Bin 0, token0 balance = 10,000, totalShares = 1,000 (honest LP)
LP fee per swap ≈ 100 token0 (1% spread, no protocol cut)

Step 1 — Attacker front-runs swap:
  addLiquidity(bin=0, shares=99,000)
  Cost = 10,000 * 99,000 / 1,000 = 990,000 token0
  New state: balance=1,000,000, totalShares=100,000

Step 2 — Swap executes:
  LP fee = 100 token0 → bin balance = 1,000,100

Step 3 — Attacker back-runs:
  removeLiquidity(bin=0, shares=99,000)
  Receives = 1,000,100 * 99,000 / 100,000 = 990,099 token0

Attacker profit = 990,099 − 990,000 = 99 token0 (99% of the 100 token0 LP fee)
Honest LP receives = 1,000,100 * 1,000 / 100,000 = 10,001 token0 (only 1 token0 of the fee)
```

The honest LP earned the fee by providing liquidity throughout the swap period but receives only 1% of it. The attacker, who provided liquidity only for the duration of the sandwich, captures 99%. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
