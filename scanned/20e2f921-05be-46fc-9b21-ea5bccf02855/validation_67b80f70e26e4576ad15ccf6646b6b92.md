### Title
`block.number` Dependency in `PriceVelocityGuardExtension` Causes Swap DoS on L2s — (File: metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol)

### Summary
`PriceVelocityGuardExtension.beforeSwap` uses `block.number` to compute `blockDiff`, which scales the allowed price-change budget. On L2s such as Arbitrum, `block.number` returns the **L1 block number**, so dozens of consecutive L2 blocks share the same value. `blockDiff` collapses to `0` for every swap within the same L1 block, permanently capping the allowed deviation at `maxChangePerBlockE18` with no scaling, and causing every swap whose oracle mid moves more than that cap to revert with `PriceVelocityExceeded`.

### Finding Description
`PriceVelocityGuardExtension` is a production `beforeSwap` hook that enforces:

```
allowedSq = maxChange² × (1 + blockDiff)
```

where `blockDiff = block.number − prevBlock`. [1](#0-0) 

On Arbitrum, `block.number` returns the L1 block number (≈ one new value every 12 s), while L2 blocks are produced every ≈ 0.25 s — roughly 48 L2 blocks per L1 block. After the first swap in a new L1 block sets `lastUpdateBlock = N`, every subsequent swap in the same L1 epoch reads `block.number = N` again, so `blockDiff = 0` and `allowedSq = maxChange²` for all of them. The intended `sqrt(1 + blockDiff)` relaxation never fires. [2](#0-1) 

`setLastMidPrice` (the admin seed function) also stamps `block.number`, so the same collapse applies on the very first swap after an admin price update. [3](#0-2) 

### Impact Explanation
Any pool on Arbitrum (or another L2 where `block.number` reflects L1 cadence) that attaches this extension will have its `beforeSwap` hook revert with `PriceVelocityExceeded` whenever the oracle mid moves by more than `maxChangePerBlockE18` within a single L1 epoch. Because the oracle price can legitimately move several percent during a 12-second L1 block, and because `blockDiff = 0` for ~47 of every 48 L2 blocks, the guard fires on virtually every swap during any period of normal price movement. This renders the swap path of the affected pool unusable — a broken core pool flow per the contest impact gate.

### Likelihood Explanation
The extension is explicitly designed for deployment on any chain the Metric OMM pool supports. Arbitrum is a primary EVM L2 target. A pool admin who calibrates `maxChangePerBlockE18` against L2 block times (the natural unit for a "per-block" parameter) will trigger the bug on every active trading session. Even an admin who calibrates against L1 block times will still see `blockDiff = 0` for the majority of swaps, making the guard far more restrictive than documented.

### Recommendation
Replace `block.number` with `block.timestamp` throughout `PriceVelocityGuardExtension` and rename the parameter to `maxChangePerSecondE18` (or similar). The velocity formula becomes:

```
allowedSq = maxChange² × (1 + timeDiff)   // timeDiff in seconds
```

This is chain-agnostic and matches the `block.timestamp`-based approach already used by `OracleValueStopLossExtension` and the factory timelocks in the same codebase. [4](#0-3) 

Alternatively, document explicitly that the extension must not be deployed on any chain where `block.number` does not increment with every produced block, and add a deployment-time check or NatSpec warning.

### Proof of Concept
1. Deploy a Metric OMM pool on Arbitrum with `PriceVelocityGuardExtension` attached.
2. Pool admin calls `setMaxChangePerBlock(pool, 1e15)` — 0.1 % per block, a reasonable L2-block-time calibration.
3. Pool admin calls `setLastMidPrice(pool, P0)` — stamps `lastUpdateBlock = block.number` (current L1 block N).
4. Within the same L1 block (same `block.number = N`), the oracle mid moves 0.5 % to `P1`.
5. Any user calls `swap(...)`:
   - `blockDiff = N − N = 0`
   - `allowedSq = (1e15)² × 1 = 1e30`
   - `changeE18 = 5e15`, `actualSq = 25e30`
   - `25e30 > 1e30` → revert `PriceVelocityExceeded`
6. All swaps revert for the remaining ~47 L2 blocks of that L1 epoch, even though the price movement is entirely legitimate. The pool is effectively frozen for trading during any period of normal volatility.
