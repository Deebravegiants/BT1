### Title
ETH Deposit Limit Check Omits the Incoming Deposit Amount, Allowing Any User to Permanently Exceed the Cap and Block Future ETH Deposits - (File: contracts/LRTDepositPool.sol)

---

### Summary

`_checkIfDepositAmountExceedesCurrentLimit` in `LRTDepositPool.sol` applies an asymmetric guard: for LST assets it correctly tests `totalAssetDeposits + amount > limit`, but for ETH it only tests `totalAssetDeposits > limit`, completely ignoring the incoming deposit amount. Any unprivileged user can exploit this to deposit an arbitrarily large ETH amount that pushes `totalAssetDeposits` far above `depositLimitByAsset[ETH_TOKEN]`, after which every subsequent ETH deposit reverts with `MaximumDepositLimitReached`, temporarily freezing the ETH deposit path.

---

### Finding Description

`depositETH()` calls `_beforeDeposit()`, which calls `_checkIfDepositAmountExceedesCurrentLimit()`:

```solidity
// contracts/LRTDepositPool.sol
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct for LSTs
}
``` [1](#0-0) 

For ETH the guard only asks "has the cap already been exceeded?" — it never asks "would this deposit exceed the cap?". The deposit is therefore accepted even when `amount` is large enough to push `totalAssetDeposits` far above `depositLimitByAsset[ETH_TOKEN]`. Once that happens, the same check returns `true` for every future caller, reverting all ETH deposits. [2](#0-1) 

The `depositETH` entry point is fully public and requires no special role. [3](#0-2) 

The only recovery path is for the MANAGER to call `updateAssetDepositLimit` in `LRTConfig` to raise the cap. [4](#0-3) 

---

### Impact Explanation

After a single large ETH deposit that crosses the cap, `getTotalAssetDeposits(ETH_TOKEN) > depositLimitByAsset[ETH_TOKEN]` becomes permanently true until an admin intervenes. Every call to `depositETH` reverts with `MaximumDepositLimitReached`. This is a **temporary freezing of the ETH deposit functionality** — matching the "Medium. Temporary freezing of funds." impact tier.

---

### Likelihood Explanation

The attack requires no special role, no governance capture, and no front-running. Any user who holds enough ETH to fill the remaining capacity (or close to it) can trigger this in a single transaction. The ETH deposit path is a primary protocol entry point, making this a realistic and low-cost attack.

---

### Recommendation

Apply the same guard used for LST assets to ETH deposits:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This ensures the incoming deposit amount is always included in the cap check, consistent with the LST branch.

---

### Proof of Concept

**Setup:**
- `depositLimitByAsset[ETH_TOKEN] = 10_000 ether`
- `totalAssetDeposits(ETH_TOKEN) = 9_999 ether` (just below the cap)

**Step 1 — Attacker calls `depositETH` with `msg.value = 5_000 ether`:**

```
_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 5_000 ether)
  → totalAssetDeposits = 9_999 ether
  → ETH branch: 9_999 > 10_000  →  false   ← deposit is allowed
```

The deposit succeeds; `totalAssetDeposits` is now `14_999 ether`.

**Step 2 — Any legitimate user calls `depositETH` with `msg.value = 1 ether`:**

```
_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 1 ether)
  → totalAssetDeposits = 14_999 ether
  → ETH branch: 14_999 > 10_000  →  true   ← revert MaximumDepositLimitReached
```

All future ETH deposits are blocked until the MANAGER raises `depositLimitByAsset[ETH_TOKEN]` above `14_999 ether`. [1](#0-0) [5](#0-4) [6](#0-5)
