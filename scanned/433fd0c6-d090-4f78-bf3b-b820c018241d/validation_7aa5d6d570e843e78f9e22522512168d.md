### Title
Uninitialized `maxMintAmountPerDay` In `RSETH::reinitialize` Causes `RSETH::mint` To Always Revert - (File: contracts/RSETH.sol)

### Summary

The `reinitialize` function in `RSETH.sol` introduces the daily mint-limit feature by setting `periodStartTime`, but fails to set `maxMintAmountPerDay`. Because `maxMintAmountPerDay` defaults to zero, the `checkDailyMintLimit` modifier causes every call to `mint` to revert, permanently blocking all rsETH minting (and therefore all deposits) until an admin separately calls `setMaxMintAmountPerDay`.

### Finding Description

`RSETH.reinitialize` (v2) accepts `_periodStartTime` and `_custodyAddress` and writes them to storage, but never writes `maxMintAmountPerDay`. [1](#0-0) 

After the upgrade executes, `maxMintAmountPerDay` remains at its default value of `0`.

The `checkDailyMintLimit` modifier, applied to every `mint` call, contains the guard:

```solidity
if (currentPeriodMintedAmount + amount > maxMintAmountPerDay) {
    revert DailyMintLimitExceeded(currentPeriodMintedAmount + amount, maxMintAmountPerDay);
}
``` [2](#0-1) 

With `maxMintAmountPerDay == 0`, the condition `amount > 0` is always `true` for any non-zero mint, so the modifier always reverts.

The `mint` function is gated by this modifier: [3](#0-2) 

### Impact Explanation

Every deposit into the LRT protocol triggers `RSETH.mint`. With `maxMintAmountPerDay == 0`, every such call reverts with `DailyMintLimitExceeded`. No user can deposit any asset until an admin separately calls `setMaxMintAmountPerDay`. [4](#0-3) 

This constitutes **temporary freezing of funds** (Medium severity): user funds are not lost, but the deposit path is completely inoperative until the admin remediation step is taken.

### Likelihood Explanation

The condition is triggered immediately after the upgrade is applied, before any user interaction. Any unprivileged depositor who calls the deposit function after the upgrade (but before the admin calls `setMaxMintAmountPerDay`) will have their transaction revert. No special attacker capability is required — the broken state is the default post-upgrade state.

### Recommendation

Include `maxMintAmountPerDay` as a parameter in `reinitialize` and assign it:

```diff
function reinitialize(
    uint256 _periodStartTime,
-   address _custodyAddress
+   address _custodyAddress,
+   uint256 _maxMintAmountPerDay
) external reinitializer(2) onlyLRTManager {
    if (_periodStartTime > block.timestamp || _periodStartTime <= block.timestamp - 1 days) {
        revert PeriodStartTimeShouldBeWithin24Hours();
    }
    periodStartTime = _periodStartTime;
    emit PeriodStartTimeSet(_periodStartTime);
    _setCustodyAddress(_custodyAddress);
+   maxMintAmountPerDay = _maxMintAmountPerDay;
+   emit MaxMintAmountPerDayUpdated(_maxMintAmountPerDay);
}
```

### Proof of Concept

1. The protocol upgrades `RSETH` to v2 and calls `reinitialize(_periodStartTime, _custodyAddress)`.
2. `maxMintAmountPerDay` is never written; it remains `0`.
3. A user calls `LRTDepositPool.depositAsset(...)`.
4. `LRTDepositPool` calls `RSETH.mint(user, amount)`.
5. `checkDailyMintLimit(amount)` evaluates `0 + amount > 0` → `true`.
6. Transaction reverts with `DailyMintLimitExceeded(amount, 0)`.
7. All deposits are blocked until `setMaxMintAmountPerDay` is called by an admin. [5](#0-4) [1](#0-0) [2](#0-1)
