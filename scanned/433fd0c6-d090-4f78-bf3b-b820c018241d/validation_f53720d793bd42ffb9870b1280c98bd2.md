### Title
`PubkeyRegistry.addPubkeys()` Does Not Increment `stakedButUnverifiedNativeETH`, Blocking `verifyWithdrawalCredentials()` for Bulk-Registered Validators - (File: contracts/PubkeyRegistry.sol / contracts/NodeDelegator.sol)

---

### Summary

`PubkeyRegistry` exposes two registration paths for validator pubkeys. The normal path via `NodeDelegator.stake32Eth()` both registers the pubkey and increments `stakedButUnverifiedNativeETH`. The bulk path via `PubkeyRegistry.addPubkeys()` registers pubkeys without touching `stakedButUnverifiedNativeETH`. Because `NodeDelegator.verifyWithdrawalCredentials()` guards on that counter, any validator registered through `addPubkeys()` can never have its withdrawal credentials verified through the protocol, permanently blocking EigenLayer share credit for those validators.

---

### Finding Description

**Registration path 1 — `NodeDelegator.stake32Eth()`** (the normal path):

```
pubkeyRegistry.addPubkey(pubkey);          // marks pubkey used
stakedButUnverifiedNativeETH += 32 ether;  // increments counter
_getEigenPodManager().stake{value: 32 ether}(...);
``` [1](#0-0) 

**Registration path 2 — `PubkeyRegistry.addPubkeys()`** (bulk/manager path):

```solidity
function addPubkeys(bytes[] calldata pubkeys) external onlyLRTManager {
    for (uint256 i = 0; i < pubkeys.length; i++) {
        pubkeyRegistry[keccak256(pubkeys[i])] = true;
    }
}
``` [2](#0-1) 

`addPubkeys()` only sets the registry flag. It never touches `stakedButUnverifiedNativeETH` on any `NodeDelegator`.

**The blocking check in `verifyWithdrawalCredentials()`:**

```solidity
if (stakedButUnverifiedNativeETH < validatorFields.length * (32 ether)) {
    revert InsufficientStakedBalance();
}
``` [3](#0-2) 

Because `stakedButUnverifiedNativeETH` was never incremented for validators registered via `addPubkeys()`, this check always reverts for them.

**No recovery path exists:**

- `stake32Eth()` checks `pubkeyRegistry.hasPubkey(pubkey)` and reverts with `PubkeyAlreadyRegistered` if the pubkey is already in the registry — so the normal staking path is also blocked.
- There is no admin setter for `stakedButUnverifiedNativeETH`.
- The `NodeDelegator` owns the `EigenPod`, so `eigenPod.verifyWithdrawalCredentials()` can only be reached through `NodeDelegator.verifyWithdrawalCredentials()`, which carries the blocking guard. [4](#0-3) 

---

### Impact Explanation

**Medium — Temporary freezing of funds.**

For every validator registered via `addPubkeys()`:

1. The 32 ETH is in the EigenPod but EigenLayer shares are never credited (withdrawal credentials cannot be verified).
2. `getEffectivePodShares()` returns `stakedButUnverifiedNativeETH + withdrawableShare`; since neither term includes these validators, the ETH is invisible to the protocol's TVL. [5](#0-4) 

3. `LRTDepositPool.getETHDistributionData()` calls `getEffectivePodShares()` for every NDC, so the understated TVL flows into `LRTOracle._updateRsETHPrice()`, causing rsETH to be priced lower than it should be — diluting existing holders. [6](#0-5) 

The freeze is temporary only in the sense that a validator can exit the beacon chain and the ETH can later be recovered via checkpoint proofs. For active validators this freeze is indefinite.

---

### Likelihood Explanation

`addPubkeys()` exists specifically for validators staked outside the normal `stake32Eth()` flow (e.g., migration of pre-existing validators, or validators whose ETH was deposited directly to the beacon chain before the PubkeyRegistry was deployed). When the manager uses this function for such validators and subsequently attempts to verify their withdrawal credentials, the bug is triggered deterministically. The function is callable by the LRT manager role, which is a trusted but operationally active role.

---

### Recommendation

`addPubkeys()` should accept a corresponding `NodeDelegator` address and increment `stakedButUnverifiedNativeETH` by `32 ether * pubkeys.length` on that contract, mirroring what `stake32Eth()` does. Alternatively, add an admin-only setter on `NodeDelegator` to adjust `stakedButUnverifiedNativeETH` for migration scenarios, so that validators registered via the bulk path can still pass the guard in `verifyWithdrawalCredentials()`.

---

### Proof of Concept

1. Manager calls `PubkeyRegistry.addPubkeys([validatorPubkey])` to pre-register a validator whose 32 ETH was staked directly to the beacon chain (withdrawal credentials pointing to the NDC's EigenPod).
2. `stakedButUnverifiedNativeETH` on the relevant `NodeDelegator` remains `0`.
3. Operator calls `NodeDelegator.verifyWithdrawalCredentials(...)` with `validatorFields.length = 1`.
4. The check `0 < 1 * 32 ether` is true → reverts with `InsufficientStakedBalance`.
5. Operator attempts `stake32Eth(validatorPubkey, ...)` to increment the counter → reverts with `PubkeyAlreadyRegistered`.
6. The 32 ETH in the EigenPod is permanently excluded from EigenLayer share accounting and from the protocol's TVL until the validator exits the beacon chain. [7](#0-6) [8](#0-7)

        bytes calldata pubkey,

        public
        whenNotPaused

    {
        IPubkeyRegistry pubkeyRegistry = IPubkeyRegistry(lrtConfig.pubkeyRegistry());
        if (pubkeyRegistry.hasPubkey(pubkey)) {
            revert PubkeyAlreadyRegistered();
        }
        pubkeyRegistry.addPubkey(pubkey);

        // tracks staked but unverified native ETH
        stakedButUnverifiedNativeETH += 32 ether;

        _getEigenPodManager().stake{ value: 32 ether }(pubkey, signature, depositDataRoot);

        if (address(eigenPod) == address(0)) {
            eigenPod = _getEigenPodManager().ownerToPod(address(this));
            emit EigenPodCreated(address(eigenPod), address(this));
        }
        emit ETHStaked(pubkey, 32 ether);
    }
