### Title
Immutable `TOKEN_X_SCALE_MULTIPLIER` Becomes Stale After Token Decimal Upgrade, Breaking Swap Conservation — (`metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`MetricOmmPoolFactory._getScaleMultipliers()` reads `IERC20Metadata(token).decimals()` once at pool creation and passes the result to the pool constructor, where it is stored as the immutable `TOKEN_0_SCALE_MULTIPLIER` / `TOKEN_1_SCALE_MULTIPLIER`. Because these values are baked into bytecode, they can never be updated. If USDC (a known upgradeable proxy) later changes its decimal count, every swap and liquidity operation in the pool will use the wrong scaling factor. The `IncorrectDelta` guard in `swap` and the `InsufficientTokenBalance` guard in `addLiquidity` both derive their expected external-unit amounts from the stale multiplier, so they accept far less input than the pool is owed, draining LP assets.

### Impact Explanation

**Swap conservation failure / direct loss of LP principal.**

Scenario: USDC (token0, 6 decimals at creation) / WETH (token1, 18 decimals). Pool stores `TOKEN_0_SCALE_MULTIPLIER = 10^12`. USDC later upgrades to 18 decimals.

Attacker calls `swap(recipient, zeroForOne=true, amountSpecified=10^6, priceLimitX64=0, ...)`:

| Step | Value | Explanation |
|---|---|---|
| `amountInScaled` | `10^12 × 10^6 = 10^18` | Pool treats 10^6 new-decimal USDC units as 1 full USDC worth of internal value |
| Swap math output | ~`10^18` internal WETH units | Correct for 1 USDC → 1 WETH at par |
| `amount0Delta` (external) | `⌈10^18 / 10^12⌉ = 10^6` | Stale divisor makes the expected payment tiny |
| WETH transferred out | ~1 WETH | Pool sends full WETH to recipient |
| Callback pays | `10^6` new-decimal USDC | = `10^-12` USDC in real value |
| `IncorrectDelta` check | **passes** | `balance0Before + 10^6 ≤ balance0After` |

The pool surrenders ~1 WETH in exchange for `10^-12` USDC. The attack is repeatable until all WETH is drained.

---

### Likelihood Explanation

USDC is an upgradeable proxy (Circle has upgraded it before). The external report explicitly identifies this decimal-change vector for USDC. The pool has no mechanism to update or invalidate the immutable multiplier; once the decimal changes, every pool containing USDC is permanently misconfigured. Any unprivileged caller can trigger the exploit via the public `swap` function.

---

### Recommendation

1. **Do not store scale multipliers as immutables.** Store them in mutable storage so a privileged function (factory-gated, with a timelock) can update them if a token's decimals change.
2. **Alternatively**, read `IERC20Metadata(token).decimals()` at swap/liquidity time and assert it matches the stored value; revert if it has changed, preventing silent miscalculation.
3. **At minimum**, document that pools containing upgradeable tokens (USDC, USDT) must be redeployed if decimals change, and add a factory-level emergency pause path.

---

### Proof of Concept

```
Setup:
  token0 = USDC (6 decimals at creation)
  token1 = WETH (18 decimals)
  TOKEN_0_SCALE_MULTIPLIER = 10^12  (immutable, set at construction)
  Pool has 1000 WETH of liquidity.

Step 1: USDC upgrades decimals from 6 → 18.
        TOKEN_0_SCALE_MULTIPLIER is still 10^12 (immutable, cannot change).

Step 2: Attacker calls:
  pool.swap(attacker, zeroForOne=true, amountSpecified=10^6, priceLimitX64=0, ...)

Step 3: Inside _executeSwap:
  amountInScaled = TOKEN_0_SCALE_MULTIPLIER * 10^6 = 10^18
  → swap math treats this as 1 full USDC (correct in old terms, wrong now)
  → computes amountOutScaled ≈ 10^18 (1 WETH worth)

Step 4: deltasScaledToExternal:
  amount0Delta = ⌈10^18 / 10^12⌉ = 10^6   ← stale divisor
  amount1Delta = -(10^18 / 1) = -10^18

Step 5: Pool transfers 10^18 WETH wei (~1 WETH) to attacker.

Step 6: Callback pays 10^6 new-decimal USDC tokens (= 10^-12 real USDC).

Step 7: IncorrectDelta check:
  balance0Before + 10^6 <= balance0After  → passes

Result: Attacker receives ~1 WETH, pays ~10^-12 USDC.
        Repeat until pool is drained.
```

    internal
    view
    returns (uint256 token0ScaleMultiplier, uint256 token1ScaleMultiplier)
  {
    uint8 token0Decimals = IERC20Metadata(token0).decimals();
    uint8 token1Decimals = IERC20Metadata(token1).decimals();
    uint8 internalDecimals = 18;
    if (token0Decimals > internalDecimals) internalDecimals = token0Decimals;
    if (token1Decimals > internalDecimals) internalDecimals = token1Decimals;
    token0ScaleMultiplier = 10 ** (internalDecimals - token0Decimals);
    token1ScaleMultiplier = 10 ** (internalDecimals - token1Decimals);
  }
