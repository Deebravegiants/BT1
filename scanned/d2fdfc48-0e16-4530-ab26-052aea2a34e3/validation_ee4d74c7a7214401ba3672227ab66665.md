### Title
Pool Admin Can Immediately Execute Oracle Swap With Zero `priceProviderTimelock`, Bypassing Intended Delay - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary

`MetricOmmPoolFactory.createPool` accepts `params.priceProviderTimelock = 0` without any minimum-value guard. When the timelock is zero, `proposePoolPriceProvider` sets `pendingPriceProviderExecuteAfter[pool] = block.timestamp + 0 = block.timestamp`, and `executePoolPriceProviderUpdate` checks `block.timestamp < execAfter`, which evaluates to `block.timestamp < block.timestamp` — always false — so it never reverts. The pool admin can therefore propose and execute an oracle replacement atomically in a single transaction, completely bypassing the timelock that is the only protection LPs have against a rogue admin swapping in a manipulated price feed.

### Finding Description

`createPool` stores the caller-supplied `priceProviderTimelock` directly with no lower-bound validation:

```solidity
// MetricOmmPoolFactory.sol line 213
priceProviderTimelock[pool] = params.priceProviderTimelock;
```

`_validatePoolParameters` checks fees, tokens, admin, and initial amounts, but never validates `priceProviderTimelock`:

```solidity
// MetricOmmPoolFactory.sol lines 548-563
function _validatePoolParameters(PoolParameters calldata params) internal view {
    // ... no check on params.priceProviderTimelock ...
}
```

The propose step computes:

```solidity
// MetricOmmPoolFactory.sol line 487
uint256 executeAfter = block.timestamp + timelock;  // = block.timestamp when timelock == 0
```

The execute step enforces:

```solidity
// MetricOmmPoolFactory.sol line 499
if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(execAfter, block.timestamp);
```

When `timelock == 0`, `execAfter == block.timestamp`, so `block.timestamp < block.timestamp` is `false` and the guard never fires. Both calls can be batched in one transaction.

The only other guard is the immutable-mode sentinel (`type(uint256).max`), which is the opposite extreme — it disables oracle rotation entirely. There is no enforced minimum for the mutable path.

### Impact Explanation

A pool admin who created (or controls) a pool with `priceProviderTimelock = 0` can:

1. Deploy a malicious `IPriceProvider` that returns an extreme bid/ask (e.g., bid ≈ 0, ask ≈ ∞).
2. In one atomic transaction: call `proposePoolPriceProvider` then `executePoolPriceProviderUpdate`.
3. The pool's active oracle is now the malicious feed.
4. Execute a swap at the manipulated price, draining LP token balances from the pool.

LP principal is directly at risk. The timelock is the sole mechanism preventing a rogue admin from performing this attack; a zero value renders it inoperative.

### Likelihood Explanation

`createPool` is permissionless. Any actor can deploy a pool with `priceProviderTimelock = 0`, set themselves as admin, and attract liquidity. The attack requires only that the pool admin is malicious and that LPs deposit without verifying the timelock parameter. Because the `PoolCreated` event emits `priceProviderTimelock` as a raw `uint256`, off-chain tooling that does not specifically flag a zero value will not warn users. Likelihood is medium — it requires a malicious admin — but the admin-boundary break is the exact class the contest marks as in-scope.

### Recommendation

Add a minimum timelock floor in `_validatePoolParameters` (or directly in `createPool`) for the mutable path:

```solidity
if (params.priceProviderTimelock != type(uint256).max) {
    if (params.priceProviderTimelock < MIN_PRICE_PROVIDER_TIMELOCK) {
        revert PriceProviderTimelockTooShort();
    }
}
```

`MIN_PRICE_PROVIDER_TIMELOCK` should be a protocol constant (e.g., 24 hours) that gives LPs sufficient time to observe a pending oracle change and exit before it takes effect.

### Proof of Concept

```solidity
// 1. Deploy pool with zero timelock
PoolParameters memory params = _defaultParams();
params.priceProviderTimelock = 0;          // mutable mode, zero delay
params.admin = attacker;
address pool = factory.createPool(params);

// 2. LPs add liquidity (pool looks legitimate)
// ...

// 3. Attacker atomically swaps oracle and executes
vm.startPrank(attacker);

// Deploy malicious oracle returning extreme prices
MaliciousOracle mal = new MaliciousOracle(token0, token1);
mal.setBidAndAskPrice(1, type(uint128).max);   // near-zero bid

// propose — executeAfter = block.timestamp + 0 = block.timestamp

// execute — block.timestamp < block.timestamp is false, no revert

// 4. Swap at manipulated price, draining LP token1 balance

vm.stopPrank();
