### Title
`getExpectedAssetAmount` Does Not Account for `instantWithdrawalFee`, Overstating Net Withdrawable Assets — (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTWithdrawalManager.getExpectedAssetAmount` is the canonical view function for computing how many underlying assets a given rsETH amount converts to. It is used both by the queued-withdrawal path (where no fee applies) and is the natural function for any user or integrator to call before invoking `instantWithdrawal`. However, `instantWithdrawal` silently deducts `instantWithdrawalFee` from the redeemed amount before sending funds to the user. Because `getExpectedAssetAmount` never subtracts this fee, it systematically overstates the net amount a user will actually receive from an instant withdrawal.

---

### Finding Description

`getExpectedAssetAmount` computes a gross asset amount with no awareness of `instantWithdrawalFee`:

```solidity
function getExpectedAssetAmount(address asset, uint256 amount)
    public view override returns (uint256 underlyingToReceive)
{
    ILRTOracle lrtOracle = ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE));
    underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
}
``` [1](#0-0) 

`instantWithdrawal` then applies the fee after redeeming the full gross amount from the vault:

```solidity
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
...
unstakingVault.redeem(asset, assetAmountUnlocked);

uint256 fee = (assetAmountUnlocked * instantWithdrawalFee) / 10_000;
uint256 userAmount = assetAmountUnlocked - fee;
...
_transferAsset(asset, msg.sender, userAmount);
``` [2](#0-1) 

The fee can be set up to 10 % (1 000 bps):

```solidity
function setInstantWithdrawalFee(uint256 feeBasisPoints) external onlyLRTManager {
    if (feeBasisPoints > 1000) revert FeeTooHigh(); // Max 10%
    instantWithdrawalFee = feeBasisPoints;
``` [3](#0-2) 

The mismatch is structurally identical to the referenced report: a public view function (`getExpectedAssetAmount`) promises a gross amount, while the execution path (`instantWithdrawal`) silently reduces the delivered amount by a fee that the view function does not model.

---

### Impact Explanation

Any user or on-chain integrator that calls `getExpectedAssetAmount` to determine the proceeds of an `instantWithdrawal` will receive up to 10 % less than the value returned by the view function. There is no revert — the transaction succeeds — but the contract fails to deliver the amount it implicitly promised through its public interface. This qualifies as **Low — contract fails to deliver promised returns, but doesn't lose value**.

---

### Likelihood Explanation

`instantWithdrawalFee` is a live, settable parameter (currently up to 1 000 bps). Whenever it is non-zero, every caller of `getExpectedAssetAmount` who then invokes `instantWithdrawal` is affected. The discrepancy is permanent and proportional to the fee magnitude; it is not a transient race condition. [4](#0-3) 

---

### Recommendation

Add a dedicated view function that returns the net amount after fee deduction, and document that `getExpectedAssetAmount` is the pre-fee gross figure:

```solidity
function getExpectedAssetAmountAfterInstantWithdrawalFee(
    address asset,
    uint256 rsETHAmount
) external view returns (uint256 netAmount) {
    uint256 gross = getExpectedAssetAmount(asset, rsETHAmount);
    uint256 fee   = (gross * instantWithdrawalFee) / 10_000;
    netAmount = gross - fee;
}
```

Alternatively, add a `bool applyInstantFee` parameter to `getExpectedAssetAmount` so callers can request the net figure directly.

---

### Proof of Concept

1. `instantWithdrawalFee` is set to 500 (5 %).
2. A user holds 10 rsETH. They call `getExpectedAssetAmount(stETH, 10e18)` and receive `X` stETH as the expected output.
3. They call `instantWithdrawal(stETH, 10e18, ...)`.
4. Internally, `assetAmountUnlocked = X`, `fee = X * 500 / 10_000 = 0.05 X`, `userAmount = 0.95 X`.
5. The user receives `0.95 X` stETH — 5 % less than the `X` promised by `getExpectedAssetAmount`.
6. Any smart-contract integrator that used `getExpectedAssetAmount` as a slippage bound or minimum-output check will observe a systematic shortfall on every instant withdrawal. [5](#0-4)
