### Title
Missing Arbitrum Sequencer Uptime Check in Chainlink Oracle Allows Stale-Price Swaps - (File: contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol)

### Summary
`ChainlinkOracleForRSETHPoolCollateral.getRate()` calls Chainlink's `latestRoundData()` on Arbitrum without verifying that the Arbitrum sequencer is live. When the sequencer is offline, Chainlink feeds on Arbitrum can return cached, stale prices that pass the existing round-validity checks. `RSETHPool` — explicitly deployed on Arbitrum — uses this oracle to price every ETH-for-rsETH and token-for-rsETH swap, so any user can execute swaps at incorrect rates during a sequencer outage.

### Finding Description

`RSETHPool` is explicitly documented as the Arbitrum deployment: [1](#0-0) 

Every swap in the pool calls `getRate()` through `rsETHOracle`: [2](#0-1) 

`getRate()` is resolved by `viewSwapRsETHAmountAndFee`, which divides the deposited amount by the oracle rate to compute how many rsETH tokens to transfer: [3](#0-2) 

The oracle backing `rsETHOracle` (and `supportedTokenOracle` for collateral tokens) is `ChainlinkOracleForRSETHPoolCollateral`. Its `getRate()` calls `latestRoundData()` and performs three validity checks — but **none of them detect a sequencer outage**: [4](#0-3) 

The `answeredInRound < roundID` staleness guard only catches rounds that were never answered; it does **not** catch the case where the sequencer is down and the last answered round is simply old. A stale price from before the outage will have `answeredInRound == roundID` and a non-zero `timestamp`, so all three checks pass while the price is arbitrarily outdated.

`RSETHPriceFeed.latestRoundData()` has the identical gap — it forwards the call to the Chainlink ETH/USD feed with no sequencer check: [5](#0-4) 

`ChainlinkPriceOracle.getAssetPrice()` on mainnet also calls `latestRoundData()` with no checks at all, but the sequencer-liveness issue is specific to L2 deployments: [6](#0-5) 

### Impact Explanation

**Medium — Temporary freezing / incorrect fund delivery.**

During an Arbitrum sequencer outage the ETH/USD (or collateral/ETH) Chainlink feed freezes at its last reported value. If the true market price has moved significantly (e.g., ETH drops 10 % while the sequencer is offline), any depositor calling `deposit()` on `RSETHPool` receives rsETH priced at the stale, pre-outage rate. Depending on the direction of price movement:

- **Price dropped while offline**: depositors receive more rsETH than fair value, draining the pool's rsETH reserves at a discount — a direct loss to the pool and its liquidity providers.
- **Price rose while offline**: depositors receive less rsETH than fair value — a loss to the depositor.

Both outcomes represent the contract failing to deliver correct exchange rates, and the first constitutes a theft of pool value reachable by any unprivileged depositor.

### Likelihood Explanation

Arbitrum sequencer outages have occurred historically (e.g., December 2022). The attack requires no special privilege — any user can call `deposit()` during the outage window. The window can last minutes to hours, and the price deviation needed to profit is proportional to the outage duration and market volatility.

### Recommendation

Follow Chainlink's documented pattern for L2 sequencer feeds. Before consuming `latestRoundData()` in `ChainlinkOracleForRSETHPoolCollateral.getRate()` (and in `RSETHPriceFeed.latestRoundData()`), query the Arbitrum sequencer uptime feed and revert if the sequencer is reported as down or if the grace period since restart has not elapsed:

```solidity
// Example sequencer check (Arbitrum sequencer uptime feed)
(, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
if (answer != 0) revert SequencerDown();
if (block.timestamp - startedAt < GRACE_PERIOD) revert GracePeriodNotOver();
```

Reference: https://docs.chain.link/data-feeds/l2-sequencer-feeds#arbitrum

### Proof of Concept

1. Arbitrum sequencer goes offline; ETH price was $3 000 at last update.
2. True ETH market price falls to $2 700 during the outage.
3. Attacker calls `RSETHPool.deposit{value: 10 ether}("")`.
4. `viewSwapRsETHAmountAndFee(10 ether)` calls `getRate()` → `ChainlinkOracleForRSETHPoolCollateral.getRate()` → `latestRoundData()` returns the stale $3 000 price; all three guards pass.
5. `rsETHAmount = 10e18 * 1e18 / staleRate` — the attacker receives rsETH priced at $3 000/ETH while the real rate is $2 700/ETH, extracting ~10 % excess value from the pool.
6. Sequencer comes back online; attacker redeems rsETH at the correct (higher) rsETH/ETH rate, realising a profit at the pool's expense.

    function getRate() public view returns (uint256) {
        (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
            AggregatorV3Interface(oracle).latestRoundData();

        if (answeredInRound < roundID) revert StalePrice();
        if (timestamp == 0) revert IncompleteRound();
        if (ethPrice <= 0) revert InvalidPrice();

        uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());

        return normalizedPrice;
    }
