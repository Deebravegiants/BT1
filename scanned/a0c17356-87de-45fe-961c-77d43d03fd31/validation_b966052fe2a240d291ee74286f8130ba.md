### Title
Unprotected `initialize2()` in NodeDelegator Allows Anyone to Hijack Upgrade Initialization — (File: contracts/NodeDelegator.sol)

---

### Summary
`NodeDelegator.initialize2()` carries only the `reinitializer(2)` guard, which prevents re-execution but imposes **no restriction on who can make the first call**. Any unprivileged address can invoke it during the upgrade window, permanently consuming the reinitializer slot and preventing the admin from ever running the intended upgrade initialization.

---

### Finding Description
The function at issue is:

```solidity
function initialize2() external reinitializer(2) {
    lastNonce = _getNonce();
}
``` [1](#0-0) 

The `reinitializer(2)` modifier (from OpenZeppelin's `Initializable`) only enforces that the function executes exactly once — when the contract's internal version counter equals 1. It does **not** check `msg.sender`. Any external caller can trigger it.

This stands in direct contrast to every other reinitializer in the codebase:

- `LRTOracle.reinitialize()` — guarded by `onlyLRTManager` [2](#0-1) 

- `LRTConverter.initialize2()` — guarded by `onlyLRTAdmin` [3](#0-2) 

- `LRTWithdrawalManager.initialize2()` — guarded by `onlyRole(LRTConstants.UNLOCKED_WITHDRAWAL_INITIALIZER)` [4](#0-3) 

- `LRTWithdrawalManager.initialize3()` — guarded by `onlyLRTManager` [5](#0-4) 

- `RsETHTokenWrapper.reinitialize()` — guarded by `onlyRole(DEFAULT_ADMIN_ROLE)` [6](#0-5) 

`NodeDelegator.initialize2()` is the sole reinitializer in the production scope that has no access control.

---

### Impact Explanation

1. **Permanent slot consumption.** Once any caller executes `initialize2()`, the `reinitializer(2)` slot is consumed forever. The admin can never call it again on that proxy instance.

2. **`lastNonce` set at attacker-chosen timing.** `lastNonce` is set to `_getNonce()` at the block the attacker chooses. If `_getNonce()` is time- or state-dependent, the value written may differ from what the admin intended, desynchronizing the nonce-based replay-protection logic inside `NodeDelegator`. [1](#0-0) 

3. **Upgrade permanently broken for that instance.** The NodeDelegator instance cannot be brought to the intended post-upgrade state without deploying a new proxy, which requires migrating all EigenLayer delegations and staked assets.

**Impact class:** Low–Medium. The contract fails to deliver its promised post-upgrade behavior; depending on how `lastNonce` is consumed downstream, nonce desynchronization could cause operations to be rejected or replayed (temporary freezing of funds).

---

### Likelihood Explanation

The attack window opens the moment `initialize()` completes (version = 1) and closes when the admin's upgrade transaction calling `initialize2()` is mined. A mempool-watching attacker can trivially front-run the admin's upgrade transaction with a higher gas price. NodeDelegator instances are long-lived contracts holding restaked ETH and LSTs, making them high-value targets. **Likelihood: Medium.**

---

### Recommendation

Add the same access control used by every other reinitializer in the codebase:

```solidity
function initialize2() external reinitializer(2) onlyLRTAdmin {
    lastNonce = _getNonce();
}
```

Alternatively, use `onlyLRTManager` consistent with `LRTOracle.reinitialize()`.

---

### Proof of Concept

1. NodeDelegator proxy is deployed; admin calls `initialize(lrtConfigAddr)` → internal version = 1.
2. Admin prepares an upgrade transaction that calls `initialize2()`.
3. Attacker observes the pending upgrade in the mempool and submits their own call to `initialize2()` with higher gas.
4. Attacker's transaction is mined first; `lastNonce = _getNonce()` is written at the attacker's chosen block.
5. Admin's transaction reverts: `reinitializer(2)` has already been consumed.
6. The NodeDelegator instance is permanently stuck in a partially-upgraded state; the admin cannot re-run `initialize2()` without deploying a new proxy and migrating all staked assets.
