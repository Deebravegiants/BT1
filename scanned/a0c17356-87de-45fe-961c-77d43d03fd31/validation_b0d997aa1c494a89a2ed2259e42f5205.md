### Title
Missing On-Chain Incentive for `updateRSETHPrice()` Causes Stale rsETH Price Used in All Deposit and Withdrawal Calculations — (File: contracts/LRTOracle.sol)

---

### Summary

`LRTOracle.updateRSETHPrice()` is a permissionless public function with no embedded on-chain incentive for callers. The protocol depends entirely on an external centralized actor (the Kelp team or a keeper bot) to invoke it. If it is not called, the stored `rsETHPrice` becomes stale and is used verbatim in every deposit mint calculation and every withdrawal payout calculation, causing users to receive incorrect amounts.

---

### Finding Description

`updateRSETHPrice()` is declared `public` with no access restriction and no reward mechanism:

```solidity
// LRTOracle.sol line 87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [1](#0-0) 

The stored `rsETHPrice` is the sole pricing input for two critical user-facing paths:

**Deposit path** — `LRTDepositPool.getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()`:

```solidity
// LRTDepositPool.sol line 520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [2](#0-1) 

**Withdrawal path** — `LRTWithdrawalManager.getExpectedAssetAmount()` multiplies by `lrtOracle.rsETHPrice()`:

```solidity
// LRTWithdrawalManager.sol line 593
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
``` [3](#0-2) 

The same stale value is consumed by `_createUnlockParams()` inside `unlockQueue()`:

```solidity
// LRTWithdrawalManager.sol line 847
rsETHPrice: lrtOracle.rsETHPrice(),
``` [4](#0-3) 

There is no keeper reward, no priority mechanism, and no gas-station subsidy written into the contracts. The only alternative update path is `updateRSETHPriceAsManager()`, which is restricted to `onlyLRTManager`:

```solidity
// LRTOracle.sol line 94-96
function updateRSETHPriceAsManager() external onlyLRTManager {
    _updateRsETHPrice();
}
``` [5](#0-4) 

Additionally, `_updateRsETHPrice()` enforces a `pricePercentageLimit` guard that **reverts** for non-manager callers when the computed new price exceeds the threshold:

```solidity
// LRTOracle.sol lines 260-265
if (isPriceIncreaseOffLimit) {
    if (!IAccessControl(address(lrtConfig)).hasRole(LRTConstants.MANAGER, msg.sender)) {
        revert PriceAboveDailyThreshold();
    }
}
``` [6](#0-5) 

This means that once the true price has drifted beyond the configured threshold, **no unprivileged caller can update the price at all** — the public function reverts — and the price is frozen at the stale value until a manager acts.

---

### Impact Explanation

When `rsETHPrice` is stale and lower than the true value:
- `getRsETHAmountToMint` mints **more rsETH** than the depositor's assets warrant, diluting all existing rsETH holders and effectively transferring a portion of their accrued yield to the new depositor.

When `rsETHPrice` is stale and higher than the true value:
- `getExpectedAssetAmount` returns **more assets** than the rsETH being burned is worth, allowing withdrawers to extract more than their fair share.

In both cases the contract fails to deliver the exchange rate it promises to users. The magnitude is bounded by `pricePercentageLimit` while the public path is still callable, but once the drift exceeds that limit the price is completely frozen for unprivileged callers.

**Impact: Low — Contract fails to deliver promised returns, but does not directly lose principal.**

---

### Likelihood Explanation

The protocol has no on-chain keeper incentive. Any period of operator inattention, infrastructure failure, or key-person unavailability causes the price to go stale. The `pricePercentageLimit` revert makes the situation self-reinforcing: the larger the drift, the longer only a manager can fix it, and the longer the stale price persists for all users.

---

### Recommendation

1. Emit a `PriceUpdateRequired` event or expose a view that signals staleness (e.g., `lastPriceUpdateTimestamp`) so off-chain monitors can detect and act.
2. Consider rewarding callers of `updateRSETHPrice()` with a small portion of the protocol fee minted during the update, creating an embedded on-chain incentive analogous to keeper rewards in other DeFi protocols.
3. Document in the contract NatSpec that the protocol relies on an external actor to call `updateRSETHPrice()` regularly, and specify the expected update cadence, so users can assess the trust assumption.
4. Consider allowing the `pricePercentageLimit` guard to be bypassed by any caller after a configurable staleness window (e.g., 24 hours), rather than requiring a manager, to reduce centralization dependency.

---

### Proof of Concept

1. Protocol is live; `rsETHPrice` was last updated at block N.
2. Underlying LST prices appreciate by 2% over 48 hours; no keeper calls `updateRSETHPrice()`.
3. A new depositor calls `LRTDepositPool.depositAsset()`. `getRsETHAmountToMint()` reads the stale (lower) `rsETHPrice` and mints proportionally more rsETH than the depositor's assets are worth.
4. Existing rsETH holders' share of the pool is diluted; they receive less on withdrawal than they were owed.
5. If the drift exceeds `pricePercentageLimit`, any attempt by an unprivileged caller to invoke `updateRSETHPrice()` reverts at line 264 of `LRTOracle.sol`, leaving the price frozen until a manager intervenes. [7](#0-6)
