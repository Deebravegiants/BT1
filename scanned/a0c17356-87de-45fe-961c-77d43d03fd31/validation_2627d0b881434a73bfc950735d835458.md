### Title
LST Deposit Limit Bypass via LRTConverter Transfer — (`contracts/LRTDepositPool.sol`)

### Summary

When an LST (e.g. stETH) is transferred from `LRTDepositPool` to `LRTConverter` via `transferAssetFromDepositPool`, the LST balance sitting in `LRTConverter` is permanently excluded from `getTotalAssetDeposits(LST)`. This causes `_checkIfDepositAmountExceedesCurrentLimit` to undercount total LST deposits, allowing users to deposit beyond the configured per-asset cap.

### Finding Description

`getAssetDistributionData` for any non-ETH asset explicitly sets `assetLyingInConverter = 0`: [1](#0-0) 

The comment explains the design intent: the LST's value is re-expressed as ETH and tracked via `ethValueInWithdrawal` in `getETHDistributionData`: [2](#0-1) 

`transferAssetFromDepositPool` in `LRTConverter` increments `ethValueInWithdrawal` by the oracle-priced ETH value of the transferred LST and pulls the tokens: [3](#0-2) 

As a result, `getTotalAssetDeposits(stETH)` drops by exactly the transferred amount, because the LST leaves the deposit pool but is not counted anywhere in the stETH accounting path: [4](#0-3) 

`_checkIfDepositAmountExceedesCurrentLimit` then uses this understated total: [5](#0-4) 

And `getAssetCurrentLimit` exposes the inflated remaining capacity publicly: [6](#0-5) 

### Impact Explanation

The per-asset deposit limit (e.g. 100,000 stETH) can be exceeded. If 100,000 stETH has been deposited and 10,000 is moved to `LRTConverter`, `getTotalAssetDeposits(stETH)` returns 90,000, and users can deposit another 10,000 stETH, pushing the actual protocol-held stETH to 110,000.

**Importantly**, rsETH backing is **not** diluted: `_getTotalEthInProtocol` in `LRTOracle` iterates all supported assets and calls `getTotalAssetDeposits` for each, including ETH. For ETH, `getETHDistributionData` includes `ethLyingInConverter = ethValueInWithdrawal`, so the ETH-equivalent value of the converter-held LST is still counted in the rsETH price calculation: [7](#0-6) 

The sole impact is that the **per-asset deposit cap is not enforced** once any LST is routed through `LRTConverter`. This matches the scoped impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

### Likelihood Explanation

`transferAssetFromDepositPool` is a routine operator action (gated by `onlyAssetTransferRole`) performed as part of normal LST-to-ETH conversion operations. Every time it is called, the stETH deposit limit is silently relaxed by the transferred amount. No attacker action is required; the bypass is a passive consequence of normal protocol operation.

### Recommendation

In `getAssetDistributionData`, instead of hardcoding `assetLyingInConverter = 0`, read the actual LST balance held by `LRTConverter`:

```solidity
address lrtConverter = lrtConfig.getContract(LRTConstants.LRT_CONVERTER);
assetLyingInConverter = IERC20(asset).balanceOf(lrtConverter);
```

This ensures `getTotalAssetDeposits(LST)` always reflects the true protocol-wide LST holdings, keeping the deposit limit check accurate regardless of where the LST resides.

### Proof of Concept

```solidity
// Setup: depositLimit for stETH = 100_000e18
// State: 90_000e18 stETH already deposited and sitting in DepositPool

// Step 1: operator moves 10_000 stETH to LRTConverter
lrtConverter.transferAssetFromDepositPool(stETH, 10_000e18);

// Step 2: getTotalAssetDeposits(stETH) now returns 80_000e18 (not 90_000e18)
uint256 total = depositPool.getTotalAssetDeposits(stETH);
assert(total == 80_000e18); // 10_000 stETH in converter is invisible

// Step 3: getAssetCurrentLimit shows 20_000e18 remaining (should be 10_000e18)
uint256 limit = depositPool.getAssetCurrentLimit(stETH);
assert(limit == 20_000e18); // inflated by 10_000e18

// Step 4: user deposits 20_000 stETH — succeeds even though protocol holds 110_000 stETH total
depositPool.depositAsset(stETH, 20_000e18, 0, "");
// actual stETH in protocol = 90_000 (pool/NDCs) + 10_000 (converter) + 20_000 (new) = 120_000
// but limit was 100_000
```
