### Title
Stale `rsETHPrice` Used in Deposit Mint Calculation Allows Excess rsETH Minting — (`contracts/LRTDepositPool.sol`)

---

### Summary

`LRTDepositPool.getRsETHAmountToMint()` divides a **freshly fetched** Chainlink asset price by a **stale cached** `rsETHPrice` state variable. Because `rsETHPrice` is only updated when `updateRSETHPrice()` is explicitly called — and deposit functions never call it — depositors can mint more rsETH than they are entitled to whenever the stored price lags behind the true current price.

---

### Finding Description

`LRTOracle` stores `rsETHPrice` as a plain state variable: [1](#0-0) 

It is only written inside `_updateRsETHPrice()`, which is triggered by the public `updateRSETHPrice()` or the manager-gated `updateRSETHPriceAsManager()`: [2](#0-1) 

The deposit mint calculation in `LRTDepositPool.getRsETHAmountToMint()` uses:
- **numerator**: `lrtOracle.getAssetPrice(asset)` — a live Chainlink call, always fresh
- **denominator**: `lrtOracle.rsETHPrice()` — the stored state variable, potentially stale [3](#0-2) 

Neither `depositETH()` nor `depositAsset()` calls `updateRSETHPrice()` before invoking `_beforeDeposit()` → `getRsETHAmountToMint()`: [4](#0-3) 

The fresh price is computed inside `_updateRsETHPrice()` by aggregating all asset balances and Chainlink prices: [5](#0-4) 

This fresh computation is never performed inline during a deposit — only the stale stored value is used.

**Aggravating condition — price-gate blocks public updates**: When the true price has risen above `pricePercentageLimit` relative to `highestRsethPrice`, the public `updateRSETHPrice()` reverts for any non-manager caller: [6](#0-5) 

During this window the stored `rsETHPrice` is frozen at its old (lower) value until a manager acts. Any depositor can exploit this window to mint at the stale lower price.

---

### Impact Explanation

`rsETHPrice` increases monotonically as staking rewards accrue. A stale (lower) denominator inflates `rsethAmountToMint`:

```
rsethAmountToMint = (amount × freshAssetPrice) / staleRsETHPrice
                                                   ↑ too low → result too high
```

New depositors receive more rsETH shares than the ETH they contribute warrants. This dilutes the claim of existing rsETH holders on the underlying assets — constituting **theft of unclaimed yield** from existing holders.

**Impact: High** — theft of unclaimed yield.

---

### Likelihood Explanation

The staleness window exists continuously between price-update calls. It is especially wide and exploitable when `pricePercentageLimit` blocks the public `updateRSETHPrice()` path, which is a normal operating condition after a period of strong staking rewards. Any unprivileged depositor can observe the on-chain `rsETHPrice` vs. the live computed price and deposit during the gap.

**Likelihood: Medium** — requires no special privilege; the condition arises in normal protocol operation.

---

### Recommendation

`getRsETHAmountToMint()` should compute the rsETH price fresh rather than reading the stored state variable. The simplest safe fix is to call `_getTotalEthInProtocol()` inline (or expose a view-only price computation path) so the denominator reflects the current TVL and supply, matching the freshness of the Chainlink numerator. Alternatively, `depositETH()` and `depositAsset()` should call `updateRSETHPrice()` atomically before computing the mint amount, with appropriate handling for the manager-gated threshold case.

---

### Proof of Concept

1. At time T, `rsETHPrice = 1.05e18` (last updated). Staking rewards have since accrued; true price is `1.06e18`.
2. A user calls `updateRSETHPrice()`. If the increase exceeds `pricePercentageLimit`, the call reverts at: [7](#0-6) 
3. `rsETHPrice` remains `1.05e18` on-chain.
4. The attacker calls `depositAsset(stETH, 1e18, 0, "")`. Inside `getRsETHAmountToMint()`:
   - `getAssetPrice(stETH)` → `1.00e18` (fresh Chainlink)
   - `rsETHPrice()` → `1.05e18` (stale)
   - `rsethAmountToMint = 1e18 × 1e18 / 1.05e18 ≈ 0.952e18`
   - Correct amount at true price: `1e18 × 1e18 / 1.06e18 ≈ 0.943e18`
5. The attacker receives `≈ 0.009e18` excess rsETH per stETH deposited, extracted from existing holders' accrued yield.
6. The manager eventually calls `updateRSETHPriceAsManager()`: [8](#0-7) 
   The price updates, but the attacker has already minted at the stale rate.

        address asset,
        uint256 amount
    )
        public
        view
        override
        returns (uint256 rsethAmountToMint)

        address lrtOracleAddress = lrtConfig.getContract(LRTConstants.LRT_ORACLE);
        ILRTOracle lrtOracle = ILRTOracle(lrtOracleAddress);

        // calculate rseth amount to mint based on asset amount and asset exchange rate
