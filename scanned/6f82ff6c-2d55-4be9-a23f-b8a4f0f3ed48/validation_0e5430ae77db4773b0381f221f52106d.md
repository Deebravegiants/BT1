### Title
Missing Slippage Protection in `instantWithdrawal` Allows Fee Increase Between Submission and Execution - (File: contracts/LRTWithdrawalManager.sol)

### Summary

`instantWithdrawal()` in `LRTWithdrawalManager` applies `instantWithdrawalFee` at execution time with no user-specified minimum output amount. The protocol's own NatSpec acknowledges this gap: *"Uses the fee set at execution time. Managers can raise it right before this call, making withdrawals cost more than expected."* Users have no on-chain mechanism to reject execution if the fee has changed since they submitted the transaction.

### Finding Description

`instantWithdrawal()` burns the caller's rsETH unconditionally at line 229, then computes the fee at line 237 using the current `instantWithdrawalFee` storage variable:

```solidity
// LRTWithdrawalManager.sol:229
IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
...
// LRTWithdrawalManager.sol:237-238
uint256 fee = (assetAmountUnlocked * instantWithdrawalFee) / 10_000;
uint256 userAmount = assetAmountUnlocked - fee;
``` [1](#0-0) 

`instantWithdrawalFee` is a mutable storage variable settable by the LRT manager at any time, up to 1000 bps (10%):

```solidity
// LRTWithdrawalManager.sol:372-374
function setInstantWithdrawalFee(uint256 feeBasisPoints) external onlyLRTManager {
    if (feeBasisPoints > 1000) revert FeeTooHigh(); // Max 10%
    instantWithdrawalFee = feeBasisPoints;
``` [2](#0-1) 

The function signature accepts no `minAmountOut` parameter:

```solidity
function instantWithdrawal(
    address asset,
    uint256 rsETHUnstaked,
    string calldata referralId
)
``` [3](#0-2) 

Because rsETH is burned before the fee is deducted, the user cannot cancel once execution begins. There is no deadline parameter either.

By contrast, `LRTDepositPool.depositETH` and `depositAsset` both accept a `minRSETHAmountExpected` parameter that reverts if the minted amount falls below the user's threshold — demonstrating the pattern is known and used elsewhere in the codebase: [4](#0-3) 

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A user who previews the withdrawal off-chain (e.g., via `getExpectedAssetAmount`) and submits a transaction expecting fee `F` may receive up to 10% fewer assets if the manager raises the fee to 1000 bps before the transaction executes. The user's rsETH is already burned; they cannot recover it. The shortfall flows to the fee recipient, not to an attacker, so this is a loss of expected yield rather than direct theft.

### Likelihood Explanation

**Medium.** The manager can update `instantWithdrawalFee` in a single transaction with no timelock. The NatSpec comment at line 210–211 explicitly documents this as a known gap, indicating the scenario was anticipated. Any pending `instantWithdrawal` transaction in the mempool is exposed whenever the fee changes. [5](#0-4) 

### Recommendation

Add a `minAmountOut` parameter to `instantWithdrawal` and revert if `userAmount < minAmountOut`:

```solidity
function instantWithdrawal(
    address asset,
    uint256 rsETHUnstaked,
    uint256 minAmountOut,       // <-- add
    string calldata referralId
) external ... {
    ...
    uint256 fee = (assetAmountUnlocked * instantWithdrawalFee) / 10_000;
    uint256 userAmount = assetAmountUnlocked - fee;
    if (userAmount < minAmountOut) revert SlippageExceeded();
    ...
}
```

Optionally, add a `deadline` parameter to guard against delayed execution.

### Proof of Concept

1. `instantWithdrawalFee` is currently 0 bps. Alice calls `instantWithdrawal(ETH, 1 ether rsETH, "ref")` expecting to receive `X` ETH with no fee deducted.
2. Before Alice's transaction is mined, the LRT manager calls `setInstantWithdrawalFee(1000)` (10%).
3. Alice's transaction executes: rsETH is burned at line 229, fee is computed as `X * 10% = 0.1X` at line 237, and Alice receives only `0.9X` ETH — 10% less than she expected.
4. Alice has no recourse; her rsETH is already burned. [6](#0-5)
