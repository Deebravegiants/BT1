### Title
Instant Withdrawal Does Not Subtract `assetsCommitted` From Available Vault Balance, Allowing Committed Queued-Withdrawal Assets to Be Drained - (File: contracts/LRTWithdrawalManager.sol)

### Summary
`instantWithdrawal` checks available vault liquidity via `LRTUnstakingVault.getAssetsAvailableForInstantWithdrawal`, which only subtracts the operator-set `queuedWithdrawalsBuffer` from the raw vault balance. It never subtracts `assetsCommitted[asset]` — the amount already locked to pending queued-withdrawal requests. When the buffer is zero (the default) or smaller than `assetsCommitted`, an instant-withdrawal caller can drain assets that are already earmarked for queued withdrawers, causing those requests to be unserviceable until the vault is externally replenished.

### Finding Description
`LRTWithdrawalManager.initiateWithdrawal` correctly prevents over-commitment by checking `getAvailableAssetAmount`, which subtracts `assetsCommitted[asset]` from the protocol-wide total:

```solidity
// LRTWithdrawalManager.sol line 599-603
function getAvailableAssetAmount(address asset) public view override returns (uint256 availableAssetAmount) {
    ILRTDepositPool lrtDepositPool = ILRTDepositPool(lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL));
    uint256 totalAssets = lrtDepositPool.getTotalAssetDeposits(asset);
    availableAssetAmount = totalAssets > assetsCommitted[asset] ? totalAssets - assetsCommitted[asset] : 0;
}
```

Every accepted queued-withdrawal request increments `assetsCommitted`:

```solidity
// LRTWithdrawalManager.sol line 173
assetsCommitted[asset] += expectedAssetAmount;
```

However, `instantWithdrawal` uses a completely different guard:

```solidity
// LRTWithdrawalManager.sol lines 228-233
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
ILRTUnstakingVault unstakingVault = ILRTUnstakingVault(lrtConfig.getContract(LRTConstants.LRT_UNSTAKING_VAULT));
if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
    revert CantInstantWithdrawMoreThanAvailable();
}
```

`getAssetsAvailableForInstantWithdrawal` only subtracts the static `queuedWithdrawalsBuffer`:

```solidity
// LRTUnstakingVault.sol lines 235-238
uint256 vaultBalance = balanceOf(asset);
uint256 reservedBuffer = queuedWithdrawalsBuffer[asset];
availableAmount = reservedBuffer >= vaultBalance ? 0 : vaultBalance - reservedBuffer;
```

`assetsCommitted` is never consulted. The `queuedWithdrawalsBuffer` defaults to zero and must be manually set by an operator; the protocol imposes no requirement that it be set before instant withdrawal is enabled, and it does not track the dynamic `assetsCommitted` value. Consequently, when the buffer is zero or stale, the full vault balance — including every ETH/LST already promised to queued withdrawers — is reported as available for instant withdrawal.

### Impact Explanation
When an instant-withdrawal caller drains vault assets that are already committed to queued-withdrawal requests, the subsequent `unlockQueue` call computes:

```solidity
// LRTWithdrawalManager.sol line 849
totalAvailableAssets: unstakingVault.balanceOf(asset)
```

If the vault balance is now below the payout amount of the next locked request, `_unlockWithdrawalRequests` exits its loop immediately:

```solidity
// LRTWithdrawalManager.sol line 800
if (availableAssetAmount < payoutAmount) break;
```

The queued-withdrawal requests remain locked and unserviceable until the vault is externally replenished (e.g., by completing an EigenLayer unstaking cycle). This constitutes **temporary freezing of funds** for queued withdrawers.

### Likelihood Explanation
- Instant withdrawal must be enabled by an operator (`setInstantWithdrawalEnabled`), which is an expected operational action.
- The `queuedWithdrawalsBuffer` defaults to zero and is not automatically updated as `assetsCommitted` grows; operators have no on-chain prompt to keep it in sync.
- Once instant withdrawal is enabled with an insufficient buffer, any unprivileged user holding rsETH can trigger the condition in a single transaction.
- No special timing, front-running, or privileged access is required beyond the normal instant-withdrawal path.

### Recommendation
In `instantWithdrawal`, subtract `assetsCommitted[asset]` from the vault's available balance before comparing:

```solidity
uint256 vaultAvailable = unstakingVault.getAssetsAvailableForInstantWithdrawal(asset);
uint256 committed = assetsCommitted[asset];
uint256 trueAvailable = vaultAvailable > committed ? vaultAvailable - committed : 0;
if (assetAmountUnlocked > trueAvailable) revert CantInstantWithdrawMoreThanAvailable();
```

Alternatively, expose `assetsCommitted` through the vault interface and incorporate it directly into `getAssetsAvailableForInstantWithdrawal`.
