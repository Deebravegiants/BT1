### Title
Missing Asset Recovery Mechanism in L1Vault Causes Permanent Freezing of Bridged ETH/LST — (`contracts/L1Vault.sol`)

---

### Summary

`L1Vault` receives ETH and LST tokens bridged from L2 and has exactly one exit path for those assets: forwarding them to `LRTDepositPool` via `depositETHForL1VaultETH` or `depositAssetForL1Vault`. No emergency withdrawal, sweep, or recovery function exists. When `LRTDepositPool` is unavailable for any of several realistic reasons, bridged assets are permanently or indefinitely frozen inside `L1Vault` with no on-chain exit.

---

### Finding Description

`L1Vault` is designed as a one-way funnel: assets arrive via the L2 bridge (`receive()`) and must exit through `LRTDepositPool`. The complete set of asset-moving functions in the contract is:

| Function | What it does |
|---|---|
| `depositETHForL1VaultETH` | Sends all ETH to `lrtDepositPool.depositETH` |
| `depositAssetForL1Vault` | Sends all LST to `lrtDepositPool.depositAsset` |
| `unwrapWstETH` | Converts wstETH → stETH (still inside L1Vault) |
| `unwrapWETH` | Converts WETH → ETH (still inside L1Vault) |
| `bridgeRsETHToL2` | Bridges **rsETH** (only after minting succeeds) | [1](#0-0) [2](#0-1) [3](#0-2) 

There is no `emergencyWithdraw`, `sweep`, `rescue`, or any function that moves raw ETH or ERC-20 tokens out of `L1Vault` without going through `LRTDepositPool`.

`LRTDepositPool` has three independent revert conditions that block both deposit paths:

**1. Pool paused** — `whenNotPaused` on both `depositETH` and `depositAsset`: [4](#0-3) [5](#0-4) 

**2. Deposit limit reached** — `_checkIfDepositAmountExceedesCurrentLimit` reverts with `MaximumDepositLimitReached`: [6](#0-5) [7](#0-6) 

**3. Asset de-listed** — `onlySupportedERC20Token(asset)` / `onlySupportedAsset` modifiers revert if the asset is removed from `lrtConfig`: [8](#0-7) 

Scenarios 1 and 2 are temporary (admin can unpause or raise the limit), producing **temporary freezing** (Medium). Scenario 3 is the permanent path: once an LST is de-listed from `lrtConfig` (e.g., EigenLayer drops support, regulatory action, security incident), any balance of that token already sitting in `L1Vault` has no on-chain exit — `depositAssetForL1Vault` reverts, and no other function can move the token out.

---

### Impact Explanation

**Permanent freezing of funds (Critical)** — for the asset de-listing path: any LST bridged to `L1Vault` before or during a de-listing event is irrecoverable on-chain. The asset cannot be sent back to L2, cannot be deposited into the pool, and cannot be swept to any address.

**Temporary freezing of funds (Medium)** — for the pause and deposit-limit paths: bridged ETH/LST is stuck in `L1Vault` for the duration of the pause or until the limit is raised. During that window, L2 users who bridged assets receive no rsETH and cannot reclaim their funds.

---

### Likelihood Explanation

- **Deposit limit reached**: Highly likely during periods of high TVL growth. The protocol already tracks per-asset deposit caps via `lrtConfig.depositLimitByAsset`. If the cap is at TVL and a bridge message arrives, the deposit call reverts.
- **Pool paused**: Routine during upgrades, incidents, or emergency responses. The `PAUSER_ROLE` can pause at any time.
- **Asset de-listed**: Realistic over a multi-year protocol lifetime (EigenLayer strategy changes, regulatory pressure, asset security issues). Any LST in `L1Vault` at the time of de-listing is permanently frozen.

---

### Recommendation

Add a privileged (e.g., `TIMELOCK_ROLE`) emergency recovery function to `L1Vault` that can transfer ETH or ERC-20 tokens to a designated safe address when the normal deposit path is unavailable:

```solidity
function rescueAsset(address token, address to, uint256 amount)
    external
    onlyRole(TIMELOCK_ROLE)
{
    UtilLib.checkNonZeroAddress(to);
    if (token == ETH_IDENTIFIER) {
        (bool ok,) = to.call{value: amount}("");
        require(ok, "ETH transfer failed");
    } else {
        IERC20(token).safeTransfer(to, amount);
    }
}
```

This should be gated behind `TIMELOCK_ROLE` (not `MANAGER_ROLE`) to prevent misuse, and should emit an event for transparency.

// Fork test (Mainnet fork, deposit limit set to current TVL)
function test_permanentFreeze_depositLimitReached() public {
    // 1. Set deposit limit for ETH to current TVL (simulates cap reached)
    vm.prank(lrtAdmin);
    lrtConfig.updateAssetDepositLimit(
        LRTConstants.ETH_TOKEN,
        lrtDepositPool.getTotalAssetDeposits(LRTConstants.ETH_TOKEN)
    );

    // 2. Bridge 1 ETH to L1Vault (simulates L2 bridge message)
    vm.deal(address(l1Vault), 1 ether);
    assertGt(address(l1Vault).balance, 0);

    // 3. MANAGER attempts to deposit — reverts with MaximumDepositLimitReached
    vm.prank(manager);
    vm.expectRevert(/* MaximumDepositLimitReached */);
    l1Vault.depositETHForL1VaultETH();

    // 4. No other function can move ETH out
    // bridgeRsETHToL2 requires rsETH balance — l1Vault has none
    // unwrapWETH only handles WETH, not native ETH
    // No sweep/rescue function exists

    // 5. ETH is permanently stuck
    assertGt(address(l1Vault).balance, 0); // still > 0, no exit
}

function test_permanentFreeze_assetDelisted() public {
    address stETH = lrtConfig.getLSTToken(LRTConstants.ST_ETH_TOKEN);

    // 1. Bridge stETH to L1Vault
    deal(stETH, address(l1Vault), 10 ether);

    // 2. Protocol de-lists stETH
    vm.prank(lrtAdmin);
    lrtConfig.updateAssetStrategy(stETH, address(0)); // de-list

    // 3. depositAssetForL1Vault reverts — onlySupportedERC20Token
    vm.prank(manager);
    vm.expectRevert();
    l1Vault.depositAssetForL1Vault(stETH);

    // 4. stETH balance remains in L1Vault forever
    assertGt(IERC20(stETH).balanceOf(address(l1Vault)), 0);
}
