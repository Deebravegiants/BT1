### Title
Unconditional Zero-Amount Fee Transfer in `claim()` Freezes All Claims When `feeInBPS` Is Zero - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

### Summary
`MerkleDistributor.claim()` unconditionally transfers the computed fee to `protocolTreasury` even when `feeInBPS` is zero, producing a zero-value ERC20 transfer. If the configured `token` is one that reverts on zero-value transfers, every `claim()` call will revert, permanently freezing all unclaimed yield for every eligible recipient.

### Finding Description
In `MerkleDistributor.claim()`, the fee is computed and then transferred to `protocolTreasury` without any guard:

```solidity
// contracts/utils/MerkleDistributor/MerkleDistributor.sol L138-144
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);

// Send the fee to the protocol treasury
IERC20(token).safeTransfer(protocolTreasury, fee);  // ← no zero-check
``` [1](#0-0) 

`feeInBPS` is allowed to be zero — `setFeeInBPS` only rejects values strictly greater than `MAX_FEE_IN_BPS`, and `initialize` imposes the same upper-bound-only constraint: [2](#0-1) [3](#0-2) 

When `feeInBPS == 0`, `fee == 0`, and `IERC20(token).safeTransfer(protocolTreasury, 0)` is called. Tokens that revert on zero-value transfers (e.g., LEND, cUSDCv3, BNB, and others catalogued at https://github.com/d-xo/weird-erc20#revert-on-zero-value-transfers) will cause every `claim()` invocation to revert.

The sibling contracts `KernelMerkleDistributor` and `KernelTop100MerkleDistributor` both correctly guard the treasury transfer:

```solidity
if (fee > 0) {
    kernel.safeTransfer(protocolTreasury, fee);
}
``` [4](#0-3) 

`MerkleDistributor` is missing this guard entirely.

### Impact Explanation
When `feeInBPS` is zero and the distributed `token` reverts on zero-value transfers, the `claim()` function is permanently broken for all users. No eligible recipient can withdraw their allocated tokens. This constitutes **permanent freezing of unclaimed yield** — all rewards committed to the Merkle tree become inaccessible.

### Likelihood Explanation
`feeInBPS` starting at zero (or being set to zero via `setFeeInBPS`) is a normal operational state — zero fee is the natural default when the protocol does not wish to take a cut. The `token` address is configurable via `setToken` and is not restricted to any specific whitelist, making it plausible that a token with zero-transfer revert behavior is used. Both conditions are independently reachable without any privileged compromise beyond normal admin configuration.

### Recommendation
Add a zero-amount guard before the treasury transfer, consistent with the pattern already used in `KernelMerkleDistributor` and `KernelTop100MerkleDistributor`:

```solidity
// Before:
IERC20(token).safeTransfer(protocolTreasury, fee);

// After:
if (fee > 0) {
    IERC20(token).safeTransfer(protocolTreasury, fee);
}
``` [5](#0-4) 

### Proof of Concept
1. Deploy `MerkleDistributor` with `feeInBPS = 0` and `token` set to any ERC20 that reverts on zero-value transfers (e.g., LEND).
2. Owner calls `setMerkleRoot(root)` with a valid Merkle tree containing at least one allocation.
3. An eligible user calls `claim(index, account, cumulativeAmount, merkleProof)` with a valid proof.
4. Execution reaches line 138: `fee = (claimableAmount * 0) / 10_000 = 0`.
5. Line 141 succeeds: `safeTransfer(account, claimableAmount)`.
6. Line 144 reverts: `safeTransfer(protocolTreasury, 0)` — the token rejects the zero-value transfer.
7. The entire transaction reverts. The user's `userClaims` state was already updated at lines 134–135 before the revert unwinds, so the state is rolled back, but the claim remains permanently blocked as long as `feeInBPS == 0` and the token is unchanged. [6](#0-5)
