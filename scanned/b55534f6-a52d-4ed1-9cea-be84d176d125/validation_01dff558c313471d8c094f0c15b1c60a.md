### Title
Stale `queuedWithdrawalsBuffer` Blocks Instant Withdrawals When Vault Balance Falls Below Buffer — (`contracts/LRTUnstakingVault.sol`)

---

### Summary

`setQueuedWithdrawalsBuffer` stores an absolute token amount with no validation against the current vault balance. When subsequent normal protocol operations (`transferAssetToNodeDelegator`, `redeem`) reduce the vault balance below the stored buffer, `getAssetsAvailableForInstantWithdrawal` permanently returns `0`, blocking all instant withdrawals until an operator manually lowers the buffer.

---

### Finding Description

`setQueuedWithdrawalsBuffer` unconditionally writes the caller-supplied value: [1](#0-0) 

`getAssetsAvailableForInstantWithdrawal` then computes: [2](#0-1) 

When `reservedBuffer >= vaultBalance`, the function returns `0`. Two legitimate, role-gated operations can reduce the vault balance after the buffer is set:

- `transferAssetToNodeDelegator` (requires `onlyAssetTransferRole`) moves tokens out of the vault to a node delegator: [3](#0-2) 

- `redeem` (requires `onlyLRTWithdrawalManager`) pays out queued withdrawal completions: [4](#0-3) 

Neither of these operations adjusts `queuedWithdrawalsBuffer`. There is no automatic recalibration, no cap enforced at buffer-set time, and no on-chain mechanism to detect the stale state.

The instant withdrawal path in `LRTWithdrawalManager` hard-reverts when the available amount is insufficient: [5](#0-4) 

---

### Impact Explanation

Instant withdrawal users are denied service for the entire period between the vault balance dropping below the buffer and an operator manually calling `setQueuedWithdrawalsBuffer` again with a lower value. No funds are lost — the vault still holds the balance — but the promised instant-withdrawal service is unavailable. This matches **Low: Contract fails to deliver promised returns, but doesn't lose value**.

---

### Likelihood Explanation

This is a routine operational scenario, not an edge case:

1. Operator sets `buffer = 80 ETH` when vault holds `100 ETH` (reasonable 80% reservation).
2. The withdrawal manager completes several queued withdrawals via `redeem`, draining the vault to `60 ETH`.
3. `getAssetsAvailableForInstantWithdrawal` now returns `0` (`80 >= 60`).
4. All instant withdrawal calls revert with `CantInstantWithdrawMoreThanAvailable`.

No malicious actor is required. No operator compromise is required. The buffer is set correctly at the time; the balance simply moves through normal protocol flows.

---

### Recommendation

Two complementary fixes:

1. **In `setQueuedWithdrawalsBuffer`**: validate that `buffer <= balanceOf(asset)` at set time, or at minimum emit a warning event when `buffer > balanceOf(asset)`.

2. **In `getAssetsAvailableForInstantWithdrawal`**: treat the buffer as a soft reservation — if `reservedBuffer > vaultBalance`, return `vaultBalance` (i.e., the entire balance is available, since there is nothing left to reserve). This makes the buffer a best-effort soft floor rather than a hard floor that can exceed actual balance:

```solidity
availableAmount = vaultBalance > reservedBuffer ? vaultBalance - reservedBuffer : 0;
// Change to:
availableAmount = reservedBuffer >= vaultBalance ? vaultBalance : vaultBalance - reservedBuffer;
```

Or, more explicitly, document that operators must lower the buffer before or immediately after any large vault drain.

---

### Proof of Concept

```solidity
// Fuzz: (initialBalance=100e18, buffer=80e18, drainAmount=30e18)
// 1. Vault receives 100e18 of asset
// 2. Operator calls setQueuedWithdrawalsBuffer(asset, 80e18)
// 3. transferAssetToNodeDelegator drains 30e18 → vault balance = 70e18
// 4. getAssetsAvailableForInstantWithdrawal(asset)
//    reservedBuffer=80e18 >= vaultBalance=70e18 → returns 0
// 5. instantWithdrawal(...) reverts with CantInstantWithdrawMoreThanAvailable
// Assert: availableAmount == 0, but vault.balanceOf(asset) == 70e18 > 0
```

The invariant `buffer <= initialBalance` holds at set time, but `buffer > initialBalance - drainAmount` after the drain, causing the underflow-guard branch to fire and return `0` despite 70 ETH sitting in the vault.
