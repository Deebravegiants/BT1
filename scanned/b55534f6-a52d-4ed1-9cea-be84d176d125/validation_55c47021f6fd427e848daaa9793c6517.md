### Title
Unprivileged `sendFunds()` Allows Anyone to Inflate Deposit Pool ETH Balance and DoS `depositETH()` — (`contracts/FeeReceiver.sol`)

---

### Summary

`FeeReceiver.sendFunds()` has no access control. Any caller can flush MEV rewards into `LRTDepositPool`, inflating `address(this).balance`. Because `getETHDistributionData()` uses the raw contract balance and `_checkIfDepositAmountExceedesCurrentLimit()` for ETH checks `totalAssetDeposits > depositLimit` (without adding `amount`, since `msg.value` is already in the balance), pushing the balance over `depositLimitByAsset[ETH]` causes every subsequent `depositETH()` call to revert with `MaximumDepositLimitReached`.

---

### Finding Description

**`FeeReceiver.sendFunds()` — no access control:** [1](#0-0) 

The function is `external` with no role modifier. Any EOA or contract can call it at any time.

**`LRTDepositPool.receiveFromRewardReceiver()` — no access control:** [2](#0-1) 

Accepts ETH from any caller, not just the registered `FeeReceiver`.

**`getETHDistributionData()` uses raw `address(this).balance`:** [3](#0-2) 

MEV rewards transferred in via `sendFunds()` immediately increase this value.

**`_checkIfDepositAmountExceedesCurrentLimit()` for ETH omits `amount`:** [4](#0-3) 

For ETH, the check is `totalAssetDeposits > depositLimit` — not `totalAssetDeposits + amount > depositLimit`. This is intentional because `depositETH()` is `payable` and `msg.value` is already in `address(this).balance` before `_beforeDeposit` runs. Consequently, if MEV rewards have already pushed `totalAssetDeposits` above the limit, the condition is `true` before any user ETH is added, and **every** `depositETH()` call reverts.

The protocol's own comment acknowledges the accounting boundary: [5](#0-4) 

Rewards in `FeeReceiver` are excluded from TVL; they are included only after `sendFunds()` moves them. An attacker exploits this boundary by triggering the transfer at a moment when the pool is near its limit.

---

### Impact Explanation

**Medium — Temporary freezing of funds (new ETH deposits).**

The claimed impact of "Critical. Protocol insolvency" is overstated. Existing user funds are not at risk; rsETH holders can still redeem. However, all new `depositETH()` calls revert until an admin either raises `depositLimitByAsset[ETH]` or transfers ETH out of the pool to node delegators. This constitutes a temporary but complete DoS on the primary ETH deposit path.

---

### Likelihood Explanation

Moderate. `sendFunds()` is a public function with no guard. A griever monitoring the mempool can call it whenever `FeeReceiver.balance` is large enough to push the pool over its limit. No capital is required by the attacker; the MEV rewards already held by `FeeReceiver` are the weapon.

---

### Recommendation

1. **Add access control to `sendFunds()`** — restrict to `MANAGER` or a dedicated keeper role:
   ```solidity
   function sendFunds() external onlyRole(LRTConstants.MANAGER) { ... }
   ```
2. **Exclude MEV rewards from the deposit-limit check** — track user-deposited ETH separately from reward ETH, or subtract known reward inflows from `totalAssetDeposits` before comparing to `depositLimitByAsset`.
3. **Add a caller whitelist to `receiveFromRewardReceiver()`** — only the registered `FeeReceiver` address should be able to call it.

---

### Proof of Concept

```solidity
// Fork test (Foundry)
function test_sendFunds_dosDepositETH() public {
    // Setup: pool near ETH limit
    uint256 limit = lrtConfig.depositLimitByAsset(LRTConstants.ETH_TOKEN);
    // Assume pool already has (limit - 1 ether) in TVL

    // Fund FeeReceiver with 2 ETH (simulating accumulated MEV rewards)
    vm.deal(address(feeReceiver), 2 ether);

    // Attacker (any EOA) calls sendFunds — no role required
    vm.prank(attacker);
    feeReceiver.sendFunds();

    // Pool balance now exceeds limit
    assertGt(lrtDepositPool.getTotalAssetDeposits(LRTConstants.ETH_TOKEN), limit);

    // Any user depositETH now reverts
    vm.deal(user, 1 ether);
    vm.prank(user);
    vm.expectRevert(ILRTDepositPool.MaximumDepositLimitReached.selector);
    lrtDepositPool.depositETH{value: 1 ether}(0, "");
}
```
