### Title
Division-by-Zero Panic Revert in `getRsETHAmountToMint` When `rsETHPrice` Is Zero — (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool.getRsETHAmountToMint` divides by `lrtOracle.rsETHPrice()` without a zero guard. Because `rsETHPrice` is a plain `uint256` state variable that defaults to `0` and is only populated by a separate `updateRSETHPrice()` call, any deposit attempted before that call executes will trigger a Solidity panic revert (division by zero), making the deposit entry-points completely unusable.

### Finding Description
`LRTDepositPool.getRsETHAmountToMint` computes the rsETH mint amount as:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [1](#0-0) 

`lrtOracle.rsETHPrice()` reads the `rsETHPrice` storage variable of `LRTOracle`:

```solidity
uint256 public override rsETHPrice;
``` [2](#0-1) 

This variable is **not** set in `LRTOracle.initialize`; it remains `0` until `updateRSETHPrice()` / `_updateRsETHPrice()` is called for the first time. `_updateRsETHPrice` only assigns a non-zero value when it runs:

```solidity
if (rsethSupply == 0) {
    rsETHPrice = 1 ether;
    ...
    return;
}
...
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
...
rsETHPrice = newRsETHPrice;
``` [3](#0-2) [4](#0-3) 

Until that function is called, `rsETHPrice == 0`, and the division in `getRsETHAmountToMint` panics.

`getRsETHAmountToMint` is called unconditionally from both public deposit entry-points:

```solidity
rsethAmountToMint = _beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, minRSETHAmountExpected);
``` [5](#0-4) 

```solidity
uint256 rsethAmountToMint = _beforeDeposit(asset, depositAmount, minRSETHAmountExpected);
``` [6](#0-5) 

Both `depositETH` and `depositAsset` are public, `nonReentrant`, `whenNotPaused` functions with no role restriction. [7](#0-6) [8](#0-7) 

### Impact Explanation
Any depositor calling `depositETH` or `depositAsset` while `rsETHPrice == 0` receives a panic revert. No ETH or LST is transferred (the revert happens before any state change), so no funds are lost, but the protocol's core deposit functionality is completely unavailable. This maps to **Low — contract fails to deliver promised returns, but doesn't lose value**.

### Likelihood Explanation
`rsETHPrice` is `0` only in the initial deployment window before the first successful `updateRSETHPrice()` call. Because `updateRSETHPrice()` is `public` and callable by anyone (subject only to `whenNotPaused`), any actor can unblock deposits by calling it first. The window is therefore narrow and self-correcting, making likelihood **Low**.

### Recommendation
Add an explicit zero-value guard in `getRsETHAmountToMint` before performing the division:

```solidity
uint256 currentRsETHPrice = lrtOracle.rsETHPrice();
if (currentRsETHPrice == 0) revert RsETHPriceNotInitialized();
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / currentRsETHPrice;
```

Alternatively, enforce that `rsETHPrice` is initialised to `1 ether` inside `LRTOracle.initialize`, mirroring the behaviour already present in `_updateRsETHPrice` when `rsethSupply == 0`.

### Proof of Concept
1. Deploy `LRTOracle` and `LRTDepositPool` (fresh state; `rsETHPrice == 0`).
2. Do **not** call `updateRSETHPrice()`.
3. Call `depositETH{value: 1 ether}(0, "")` as any EOA.
4. The call reverts with a Solidity panic (0x12 — division or modulo by zero) because `lrtOracle.rsETHPrice()` returns `0` at line 520 of `LRTDepositPool.sol`. [1](#0-0) [2](#0-1)
