### Title
Incorrect Total-Balance Check in `bridgeTokenToL1` Allows Pre-existing Contract Tokens to Subsidize Bridge Input — (File: `contracts/bridges/SonicChainNativeTokenBridge.sol`)

---

### Summary

`SonicChainNativeTokenBridge.bridgeTokenToL1()` validates the post-transfer token balance using the contract's **total** balance rather than the **balance increase** caused by the `safeTransferFrom`. This mirrors the exact root cause described in the external report: a balance check that fails to exclude funds already held by the contract for other purposes, allowing those pre-existing funds to be silently consumed as part of the operation.

---

### Finding Description

In `bridgeTokenToL1`, after pulling tokens from the caller, the contract performs the following check:

```solidity
token.safeTransferFrom(msg.sender, address(this), amount);   // line 83

uint256 balance = token.balanceOf(address(this));            // line 85
if (balance < amount) revert InsufficientBalance();          // line 86
``` [1](#0-0) 

The check at line 86 reads the **total** token balance of the contract, not the delta caused by the transfer. If the contract already holds a residual token balance (e.g., tokens sent directly to the contract, or tokens left over from any prior interaction), the check passes even when `safeTransferFrom` delivered fewer tokens than `amount` — as occurs with fee-on-transfer tokens.

Subsequently, the bridge call consumes exactly `amount` tokens from the contract:

```solidity
uint256 balanceBefore = token.balanceOf(address(this));      // line 108
token.safeIncreaseAllowance(address(sonicBridge), amount);
sonicBridge.withdraw(uid, originalToken, amount);            // line 115
uint256 balanceAfter = token.balanceOf(address(this));
if (balanceBefore - balanceAfter != amount) {                // line 119
    revert BridgeFailed();
}
``` [2](#0-1) 

The post-bridge verification at line 119 confirms that exactly `amount` tokens left the contract — but it does not distinguish between tokens received from the caller and tokens that were already sitting in the contract. The shortfall introduced by the fee-on-transfer is silently covered by the pre-existing balance.

By contrast, `SonicBridgeReceiver.claimAndTransferToVault()` correctly uses the balance-delta pattern:

```solidity
uint256 balanceBefore = IERC20(token).balanceOf(address(this));
ETHEREUM_TOKEN_DEPOSIT.claim(withdrawalId, token, amount, proof);
uint256 balanceAfter = IERC20(token).balanceOf(address(this));
uint256 received = balanceAfter - balanceBefore;
``` [3](#0-2) 

The inconsistency between the two sibling contracts confirms the defect in `SonicChainNativeTokenBridge`.

---

### Impact Explanation

Any tokens that accumulate in `SonicChainNativeTokenBridge` — whether from accidental direct transfers, a future upgrade path, or any other source — can be silently consumed as bridge input by a caller using a fee-on-transfer token. The caller bridges `amount` tokens to L1 while only paying `amount − fee` tokens; the deficit is drawn from the contract's pre-existing balance. Those pre-existing tokens, which should be recoverable by the admin via `recoverTokens`, are instead permanently bridged away and unrecoverable.

Impact category: **Theft of funds held in the contract** (Low — direct theft of any pre-existing token balance held by the bridge contract).

---

### Likelihood Explanation

Two conditions must hold simultaneously:

1. The bridged token must behave as a fee-on-transfer token (or be upgraded to do so).
2. The contract must hold a residual balance of that token at the time of the call.

The token is fixed at construction time and validated against `tokenPairs.mintedToOriginal`, which reduces but does not eliminate the risk of fee-on-transfer behavior. Residual balances can arise from direct transfers or future protocol changes. Likelihood is **Low**.

---

### Recommendation

Record the balance before the transfer and validate the increase, matching the pattern already used in `SonicBridgeReceiver`:

```solidity
uint256 balanceBefore = token.balanceOf(address(this));
token.safeTransferFrom(msg.sender, address(this), amount);
if (token.balanceOf(address(this)) - balanceBefore < amount) {
    revert InsufficientBalance();
}
```

Remove the redundant post-transfer total-balance check at lines 85–86.

---

### Proof of Concept

1. The contract holds `F` pre-existing tokens (e.g., sent directly by any address).
2. The bridged token charges a transfer fee of `F` per transfer.
3. Attacker calls `bridgeTokenToL1(recipient, amount)`.
4. `safeTransferFrom` delivers `amount − F` tokens; contract balance becomes `F + (amount − F) = amount`.
5. Line 86 check: `amount < amount` → false → no revert.
6. `sonicBridge.withdraw(uid, originalToken, amount)` pulls all `amount` tokens from the contract.
7. Line 119 check: `amount − 0 == amount` → passes.
8. Attacker has bridged `amount` tokens to L1 while paying only `amount − F`; the `F` pre-existing tokens are permanently consumed and unrecoverable by the admin. [4](#0-3)

    function bridgeTokenToL1(address recipient, uint256 amount) external payable override nonReentrant {
        UtilLib.checkNonZeroAddress(recipient);

        // recipient parameter is kept for IL2TokenBridge interface compatibility
        // but the actual flow will be: SonicBridge -> SonicBridgeReceiver -> L1Vault
        if (amount == 0) revert InvalidAmount();

        // No additional msg.value is needed for the fees
        if (msg.value != 0) revert NoMsgValueNeeded();

        token.safeTransferFrom(msg.sender, address(this), amount);

        uint256 balance = token.balanceOf(address(this));
        if (balance < amount) revert InsufficientBalance();

        // Get the original token address (validated in constructor)
        address originalToken = tokenPairs.mintedToOriginal(address(token));

        // Generate a unique UID for this transaction
        uint96 uid = uint96(
            uint256(
                keccak256(
                    abi.encodePacked(
                        block.timestamp, block.number, msg.sender, recipient, amount, tx.gasprice, bridgeReceiver
                    )
                )
            ) % type(uint96).max
        );

        // Ensure UID is not zero
        if (uid == 0) {
            uid = uint96(uint256(keccak256(abi.encodePacked(block.number, msg.sender, tx.gasprice, bridgeReceiver))));
        }

        // Store the current token balance before withdrawal
        uint256 balanceBefore = token.balanceOf(address(this));

        // Approve the Sonic bridge to spend the tokens
        token.safeIncreaseAllowance(address(sonicBridge), amount);

        // Initiate withdrawal on Sonic bridge
        // Note: Sonic gateway will only allow SonicBridgeReceiver to claim (same address as this contract)
        sonicBridge.withdraw(uid, originalToken, amount);

        // Verify tokens were transferred/burned
        uint256 balanceAfter = token.balanceOf(address(this));
        if (balanceBefore - balanceAfter != amount) {
            revert BridgeFailed();
        }

        emit TokensBridgedToL1(recipient, amount, uid, bridgeReceiver);
    }
```
