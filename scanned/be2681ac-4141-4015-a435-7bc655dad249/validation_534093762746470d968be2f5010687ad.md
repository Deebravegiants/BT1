### Title
Missing Minimum rsETH Output Protection in Pool `deposit()` Functions - (File: contracts/pools/RSETHPool.sol, contracts/pools/RSETHPoolNoWrapper.sol, contracts/pools/RSETHPoolV2ExternalBridge.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol)

### Summary
All four L2 pool contracts expose public `deposit()` functions that swap ETH (or supported LSTs) for rsETH using a live oracle rate, but accept no `minRsETHAmountExpected` parameter. Users have zero slippage protection: if the oracle rate changes between transaction submission and execution, they silently receive fewer rsETH than they observed off-chain. The `LRTDepositPool` on L1 already implements this protection correctly, confirming the protocol is aware of the requirement.

### Finding Description
Every pool `deposit()` function computes the rsETH output exclusively from the live oracle rate at execution time:

```solidity
// RSETHPool.sol L311-319
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();          // live oracle read
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
```

The `deposit()` entry points in all four contracts call this helper and immediately transfer the result to the caller with no floor check:

- `RSETHPool.deposit(string)` — transfers `wrsETH` to caller [1](#0-0) 
- `RSETHPool.deposit(address,uint256,string)` — transfers `wrsETH` to caller [2](#0-1) 
- `RSETHPoolNoWrapper.deposit(string)` — transfers `rsETH` to caller [3](#0-2) 
- `RSETHPoolNoWrapper.deposit(address,uint256,string)` — transfers `rsETH` to caller [4](#0-3) 
- `RSETHPoolV2ExternalBridge.deposit(string)` — mints `wrsETH` to caller [5](#0-4) 
- `RSETHPoolV3ExternalBridge.deposit(string)` — mints `wrsETH` to caller [6](#0-5) 
- `RSETHPoolV3ExternalBridge.deposit(address,uint256,string)` — mints `wrsETH` to caller [7](#0-6) 

None of these functions accept or enforce a caller-supplied minimum output amount.

By contrast, `LRTDepositPool.depositETH()` on L1 explicitly accepts and enforces `minRSETHAmountExpected`:

```solidity
// LRTDepositPool.sol L648-669
function _beforeDeposit(...) private view returns (uint256 rsethAmountToMint) {
    rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
    if (rsethAmountToMint < minRSETHAmountExpected) {
        revert MinimumAmountToReceiveNotMet();
    }
}
``` [8](#0-7) 

The oracle rate (`rsETHOracle.getRate()`) is a live, updatable value. It increases over time as staking rewards accrue and is updated by a keeper. Any oracle update that lands in the same block as, or just before, a user's pending deposit transaction will cause the user to receive fewer rsETH than they observed when constructing the transaction.

### Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A user calls `viewSwapRsETHAmountAndFee` off-chain to preview their rsETH output, then submits `deposit()`. If the oracle rate is updated before their transaction executes (a routine, non-adversarial event), the user receives fewer rsETH than previewed. The ETH is accepted by the pool with no recourse for the user. The user cannot set a floor on the rsETH they are willing to accept, so they cannot protect themselves from this outcome.

### Likelihood Explanation
The rsETH oracle rate is updated regularly as Ethereum staking rewards accrue. On L2 chains, oracle updates are pushed by off-chain keepers on a schedule. Any user whose transaction is pending in the mempool during an oracle update will be silently under-filled. This is a routine operational event, not an adversarial one, making it a realistic and recurring condition for active depositors.

### Recommendation
Add a `minRsETHAmountExpected` parameter to all `deposit()` overloads in `RSETHPool`, `RSETHPoolNoWrapper`, `RSETHPoolV2ExternalBridge`, and `RSETHPoolV3ExternalBridge`, mirroring the existing pattern in `LRTDepositPool`:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountExpected)
    external payable nonReentrant whenNotPaused
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    ...
}
```

### Proof of Concept
1. Oracle rate is currently `1.05e18` (1 ETH = ~0.952 rsETH).
2. User calls `viewSwapRsETHAmountAndFee(1 ether)` off-chain → expects `~0.952 rsETH`.
3. User submits `RSETHPool.deposit{value: 1 ether}("ref")`.
4. Before the transaction is mined, the keeper updates the oracle to `1.06e18`.
5. Transaction executes: `rsETHAmount = 1e18 * 1e18 / 1.06e18 ≈ 0.943 rsETH`.
6. User receives `~0.943 rsETH` instead of the expected `~0.952 rsETH` — a ~0.9% shortfall — with no revert and no recourse.

The same scenario applies to token deposits in `RSETHPool.deposit(address,uint256,string)` and `RSETHPoolNoWrapper.deposit(address,uint256,string)`, where both the rsETH oracle and the token oracle can shift between preview and execution. [9](#0-8)
