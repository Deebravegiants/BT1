Let me read the full deposit function and check if there's any zero-amount guard after computing `agETHAmount`.

### Title
Integer Division Truncation in Token Deposit Causes Zero agETH Minted for Small Amounts — (`contracts/agETH/AGETHPoolV3.sol`)

---

### Summary

`AGETHPoolV3.deposit(address token, uint256 amount, string referralId)` accepts a user's token transfer and then computes `agETHAmount` via integer division. When `amount` is small enough that `amountAfterFee * tokenToETHRate < agETHToETHrate`, the division truncates to zero. There is no post-computation guard, so `agETH.mint(msg.sender, 0)` is called — the user's tokens are consumed but zero agETH is returned.

---

### Finding Description

In `viewSwapAgETHAmountAndFee(uint256 amount, address token)`: [1](#0-0) 

```
agETHAmount = amountAfterFee * tokenToETHRate / agETHToETHrate;
```

Both rates are 1e18-scaled. Truncation to zero occurs whenever:

```
amountAfterFee * tokenToETHRate < agETHToETHrate
```

The ETH-path overload avoids this by inserting an explicit `1e18` multiplier: [2](#0-1) 

```
agETHAmount = amountAfterFee * 1e18 / agETHToETHrate;
```

The token-path overload has no equivalent scaling, so the guard is absent.

In `deposit(address token, ...)`, the only input validation is `amount == 0`: [3](#0-2) 

The token transfer happens **before** the amount is computed, and there is no check that `agETHAmount > 0` before minting. A standard ERC-20 `mint(addr, 0)` is a no-op that does not revert, so the entire call succeeds silently.

---

### Impact Explanation

A depositor who sends a non-zero but small token amount loses their tokens: the contract accepts the transfer, mints zero agETH, and emits a `SwapOccurred` event with `agETHAmount = 0`. The tokens remain in the pool (no protocol-level loss), but the user receives nothing, violating the invariant that any non-zero deposit must yield a non-zero output.

Scope: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

---

### Likelihood Explanation

The threshold below which truncation occurs is:

```
amountAfterFee < agETHToETHrate / tokenToETHRate
```

- **18-decimal token** (e.g., stETH, tokenToETHRate ≈ 0.9e18, agETHToETHrate ≈ 1.1e18): threshold ≈ 1–2 wei. Only dust amounts are affected; practically negligible.
- **6-decimal token** (e.g., USDC, tokenToETHRate ≈ 3e14 at ~$3 000/ETH, agETHToETHrate ≈ 1.1e18): threshold ≈ `1.1e18 / 3e14 ≈ 3 667` wei ≈ 0.003667 USDC. Any deposit below this amount silently yields zero agETH. This is a realistic user mistake (copy-paste of a small test amount, UI rounding, etc.).

---

### Recommendation

Add a zero-output guard immediately after computing `agETHAmount` in both the token `deposit` function and `viewSwapAgETHAmountAndFee`:

```solidity
(uint256 agETHAmount, uint256 fee) = viewSwapAgETHAmountAndFee(amount, token);
if (agETHAmount == 0) revert InvalidAmount();
```

Alternatively, enforce a protocol-level minimum deposit amount per token that guarantees at least 1 wei of agETH output.

---

### Proof of Concept

```solidity
// Preconditions:
//   tokenToETHRate  = 0.9e18  (token oracle)
//   agETHToETHrate  = 1.1e18  (agETH oracle)
//   feeBps          = 0       (simplify: no fee)
//   amount          = 1       (1 wei of token)

// viewSwapAgETHAmountAndFee(1, token):
//   fee           = 0
//   amountAfterFee = 1
//   agETHAmount   = 1 * 0.9e18 / 1.1e18
//                 = 9e17 / 11e17
//                 = 0   ← integer truncation

// deposit(token, 1, ""):
//   safeTransferFrom(msg.sender, pool, 1)  ← token taken
//   agETH.mint(msg.sender, 0)              ← no-op, no revert
//   emit SwapOccurred(..., agETHAmount=0, ...)
```

A fuzz test asserting `agETHAmount > 0` for all `amount ∈ [1, 1e6]` with `tokenToETHRate < agETHToETHrate` will find failing cases immediately for amounts below `agETHToETHrate / tokenToETHRate`.
