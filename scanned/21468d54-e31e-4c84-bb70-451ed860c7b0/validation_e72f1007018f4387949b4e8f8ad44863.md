### Title
Zero-ETH Bridge Invocation via Missing Zero-Check in `bridgeAssetsViaNativeBridge()` â€” (`contracts/bridges/UnichainMessenger.sol`, `contracts/pools/RSETHPool.sol`, `contracts/pools/RSETHPoolNoWrapper.sol`, `contracts/pools/RSETHPoolV2.sol`)

---

### Summary

When `address(this).balance == feeEarnedInETH`, `getETHBalanceMinusFees()` returns `0`. The pool's `bridgeAssetsViaNativeBridge()` passes this zero value directly to `UnichainMessenger.sendETHToL1ViaBridge{value: 0}(...)`. The messenger's only guard is `msg.value != value`, which passes trivially when both are `0`. The call then forwards `bridgeETHTo{value: 0}(...)` to the Unichain bridge â€” a no-op that wastes L2 (and potentially L1) gas without delivering any ETH to the L1 vault.

---

### Finding Description

`getETHBalanceMinusFees()` is defined identically across the affected pools:

```solidity
function getETHBalanceMinusFees() public view returns (uint256) {
    return address(this).balance - feeEarnedInETH;
}
``` [1](#0-0) [2](#0-1) 

When `address(this).balance == feeEarnedInETH`, this returns `0`. Each pool's `bridgeAssetsViaNativeBridge()` reads this value and passes it directly to the messenger with no zero-amount guard:

```solidity
uint256 ethBalanceMinusFees = getETHBalanceMinusFees();
IL2Messenger(messenger).sendETHToL1ViaBridge{ value: ethBalanceMinusFees }(
    l2Bridge, l1VaultETHForL2Chain, ethBalanceMinusFees
);
``` [3](#0-2) [4](#0-3) [5](#0-4) 

Inside `UnichainMessenger.sendETHToL1ViaBridge`, the only guard is:

```solidity
if (msg.value != value) revert MismatchedMsgValue();
IUnichainMessenger(l2bridge).bridgeETHTo{ value: value }(target, DEFAULT_GAS_LIMIT, bytes(""));
``` [6](#0-5) 

When both `msg.value` and `value` are `0`, the check `0 != 0` is `false` â€” no revert â€” and `bridgeETHTo{value: 0}(...)` is forwarded to the Unichain bridge. The protocol emits `BridgedETHToL1ViaNativeBridge` with `amount = 0`, falsely signaling a successful bridge operation.

Note: `RSETHPoolV2ExternalBridge.bridgeAssetsViaNativeBridge(uint256 amount)` is **not** affected because it takes an explicit amount parameter and explicitly checks `if (amount == 0) revert InvalidAmount()`. [7](#0-6) 

---

### Impact Explanation

The contract fails to deliver its promised behavior (bridging ETH to L1) without losing funds. L2 gas is consumed for a no-op transaction. If the Unichain bridge does not revert on zero-value calls, an L1 message may also be submitted, consuming L1 gas. The `BridgedETHToL1ViaNativeBridge` event is emitted with `amount = 0`, which can mislead off-chain monitoring. No ETH is lost.

**Scope match: Low â€” Contract fails to deliver promised returns, but doesn't lose value.**

---

### Likelihood Explanation

The state `address(this).balance == feeEarnedInETH` is reachable in normal operation: e.g., immediately after all user-deposited ETH has been bridged via LayerZero but accumulated fees remain, or before any user deposits have occurred. The `BRIDGER_ROLE` is an automated keeper that may call `bridgeAssetsViaNativeBridge()` on a schedule without first checking whether there is a non-zero bridgeable balance.

---

### Recommendation

Add a zero-amount guard at the start of each `bridgeAssetsViaNativeBridge()` implementation:

```solidity
uint256 ethBalanceMinusFees = getETHBalanceMinusFees();
if (ethBalanceMinusFees == 0) revert InvalidAmount();
```

Optionally, add the same guard inside `UnichainMessenger.sendETHToL1ViaBridge` as a defense-in-depth measure:

```solidity
if (value == 0) revert InvalidAmount();
```

---

### Proof of Concept

```solidity
// Precondition: pool.balance == pool.feeEarnedInETH (e.g., 1 ether each)
// No zero-check exists, so the following succeeds silently:

vm.prank(bridgerRole);
pool.bridgeAssetsViaNativeBridge();
// UnichainMessenger receives {value: 0}, passes msg.value == value check,
// calls bridgeETHTo{value: 0}(...) on Unichain bridge.
// Assert: L1 vault receives 0 ETH.
// Assert: BridgedETHToL1ViaNativeBridge emitted with amount == 0.
```
