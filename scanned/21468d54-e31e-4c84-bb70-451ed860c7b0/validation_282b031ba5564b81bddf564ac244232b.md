### Title
ETH Deposit Limit Bypass Due to Missing `amount` in Limit Check - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` uses an inconsistent check for ETH deposits: it omits the incoming `amount` from the comparison, allowing any user to deposit ETH that pushes the protocol's total ETH holdings above the configured `depositLimitByAsset` cap.

### Finding Description
The function `_checkIfDepositAmountExceedesCurrentLimit` applies two different formulas depending on the asset:

```solidity
// contracts/LRTDepositPool.sol lines 676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // ← amount NOT included
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← amount included
}
```

For every non-ETH LST, the check correctly evaluates `totalAssetDeposits + amount > limit`. For ETH, the check only evaluates `totalAssetDeposits > limit`, completely ignoring the incoming `amount`. This is called from `_beforeDeposit`, which runs before any state change, so `totalAssetDeposits` reflects the pre-deposit total.

Concrete example:
- `depositLimitByAsset(ETH_TOKEN)` = 1000 ETH
- `getTotalAssetDeposits(ETH_TOKEN)` = 999 ETH
- User calls `depositETH` with `msg.value` = 500 ETH
- Check: `999 > 1000` → `false` → deposit proceeds
- Post-deposit total: 1499 ETH — 499 ETH above the configured limit

The analog to the external report is exact: the wrong expression is used in the limit check (missing `+ amount` for ETH), allowing users to bypass the cap, mirroring the `amount` vs `_amount` confusion in `FertilizerFacet.mintFertilizer`.

### Impact Explanation
Any unprivileged depositor can push the protocol's ETH holdings arbitrarily above the admin-configured deposit limit in a single transaction. The deposit limit is the primary on-chain mechanism for controlling ETH exposure. Bypassing it means the protocol does not enforce its own promised constraints, which is a **Low** impact: the contract fails to deliver its promised behavior (enforced deposit cap) but does not directly lose or freeze user funds.

### Likelihood Explanation
The entry path is `depositETH()`, which is a public payable function with no access control. Any user can call it at any time. The condition is trivially reachable whenever `totalAssetDeposits <= depositLimitByAsset(ETH_TOKEN)`. Likelihood is **High**.

### Recommendation
Include `amount` in the ETH branch of the check, consistent with the non-ETH branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN)` = 1000 ether.
2. Protocol accumulates 999 ether in ETH deposits.
3. Attacker calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500e18)` evaluates `999e18 > 1000e18` → `false`.
5. Deposit succeeds; `getTotalAssetDeposits(ETH_TOKEN)` is now 1499 ether — 49.9% above the limit.
6. Attacker receives rsETH for the full 500 ETH deposit with no revert. [1](#0-0) [2](#0-1) [3](#0-2)
