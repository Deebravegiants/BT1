### Title
`KernelDepositPool.initiateWithdrawal` Always Reverts When `maxNumberOfWithdrawalsPerUser` Is Uninitialized - (File: contracts/KERNEL/KernelDepositPool.sol)

---

### Summary

The `KernelDepositPool.initialize` function does not set `maxNumberOfWithdrawalsPerUser`, leaving it at its default value of `0`. Because `initiateWithdrawal` unconditionally checks `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser`, and `0 >= 0` is always `true`, every call to `initiateWithdrawal` reverts with `WithdrawalLimitReached` until an admin explicitly calls `setMaxNumberOfWithdrawalsPerUser`. This is the direct analog of H-02: a core user-facing function always reverts on first use due to an uninitialized state variable.

---

### Finding Description

`KernelDepositPool.initialize` sets up the staking token, rewards token, and access control, but never assigns `maxNumberOfWithdrawalsPerUser`: [1](#0-0) 

The variable therefore holds its Solidity default of `0`. Every call to `initiateWithdrawal` then hits: [2](#0-1) 

For any caller, `userWithdrawalIds[msg.sender].length` starts at `0`. The condition `0 >= 0` is always `true`, so the function always reverts with `WithdrawalLimitReached` before any withdrawal can be queued. The `stake` function has no such guard and succeeds normally: [3](#0-2) 

This creates a state where users can deposit KERNEL tokens but are permanently blocked from initiating their retrieval until an admin intervenes.

The admin setter enforces `_maxNumberOfWithdrawalsPerUser > 0`, confirming that `0` is an invalid operational value, yet `initialize` leaves it there: [4](#0-3) 

---

### Impact Explanation

Any user who calls `stake` before the admin calls `setMaxNumberOfWithdrawalsPerUser` has their KERNEL tokens locked in the contract with no ability to initiate a withdrawal. The funds are not lost — the admin can unblock the situation — but until that call is made the withdrawal path is completely non-functional. This constitutes a **temporary freezing of funds** (Medium).

---

### Likelihood Explanation

The window exists from the moment `initialize` is called until `setMaxNumberOfWithdrawalsPerUser` is called. Because `initialize` gives no indication that a follow-up call is required, and because `stake` is immediately usable, real users will deposit before the admin completes setup. The likelihood is **medium-high** for any deployment that does not atomically set `maxNumberOfWithdrawalsPerUser` in the same transaction as initialization.

---

### Recommendation

Set a safe non-zero default for `maxNumberOfWithdrawalsPerUser` inside `initialize`, for example:

```solidity
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    ...
    maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER; // default to protocol maximum
}
```

This mirrors the pattern used for `withdrawalDelayBlocks` in `LRTWithdrawalManager.initialize`, which sets a concrete default at construction time. [5](#0-4) 

---

### Proof of Concept

1. Deploy `KernelDepositPool` and call `initialize(admin, kernelToken, rewardToken)`.
2. `maxNumberOfWithdrawalsPerUser` is now `0`.
3. User calls `stake(100e18)` — succeeds; `balanceOf[user] = 100e18`.
4. User calls `initiateWithdrawal(100e18)`:
   - `userWithdrawalIds[user].length` is `0`.
   - Check: `0 >= 0` → `true` → reverts with `WithdrawalLimitReached`.
5. User's 100e18 KERNEL is locked in the contract with no withdrawal path available.
6. Admin must call `setMaxNumberOfWithdrawalsPerUser(N)` to unblock all affected users. [6](#0-5)

    function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
        if (_amount == 0) revert AmountZero();
        if (balanceOf[msg.sender] < _amount) revert InsufficientStakedBalance();
        if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();

        balanceOf[msg.sender] -= _amount;
        totalKernelStaked -= _amount;

        // Create a withdrawal record
        uint256 withdrawalId = ++withdrawalCounter;
        uint256 unlockTime = block.timestamp + withdrawalDelay;

        withdrawals[withdrawalId] = Withdrawal({
            user: msg.sender, amount: _amount, unlockTime: unlockTime, claimed: false, withdrawalId: withdrawalId
        });
        userWithdrawalIds[msg.sender].push(withdrawalId);

        emit WithdrawalInitiated(msg.sender, _amount, withdrawalId, unlockTime);
    }
