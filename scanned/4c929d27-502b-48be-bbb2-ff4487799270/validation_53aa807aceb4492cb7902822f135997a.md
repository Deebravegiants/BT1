### Title
Instant Withdrawal Fee Rounds to Zero for Small Amounts, Enabling Fee-Free Withdrawals - (File: `contracts/LRTWithdrawalManager.sol`)

### Summary

`LRTWithdrawalManager.instantWithdrawal()` computes the protocol fee as `(assetAmountUnlocked * instantWithdrawalFee) / 10_000`. When `assetAmountUnlocked * instantWithdrawalFee < 10_000`, Solidity integer division truncates the result to zero, and the user pays no fee. Because `minRsEthAmountToWithdraw[asset]` defaults to zero (never initialized in `initialize()`), any rsETH holder can call `instantWithdrawal` with arbitrarily small amounts and bypass the fee entirely.

### Finding Description

The fee calculation in `instantWithdrawal` is:

```solidity
uint256 fee = (assetAmountUnlocked * instantWithdrawalFee) / 10_000;
```

`assetAmountUnlocked` is derived from the rsETH amount via:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

For ETH (where `rsETHPrice ≈ assetPrice ≈ 1e18`), `assetAmountUnlocked ≈ rsETHUnstaked`. The fee truncates to zero whenever:

```
rsETHUnstaked * instantWithdrawalFee < 10_000
```

Concrete thresholds:
- `instantWithdrawalFee = 1` (0.01%): any `rsETHUnstaked < 10_000` wei → fee = 0
- `instantWithdrawalFee = 100` (1%): any `rsETHUnstaked < 100` wei → fee = 0
- `instantWithdrawalFee = 1000` (10%): any `rsETHUnstaked < 10` wei → fee = 0

The only guard against dust amounts is:

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```

`minRsEthAmountToWithdraw` is a storage mapping that is **never set in `initialize()`**, so it defaults to `0` for every asset. The condition reduces to `rsETHUnstaked == 0`, meaning any positive amount passes. An attacker can split a large rsETH balance into many calls each below the rounding threshold and withdraw the full underlying asset amount with zero fee.

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose user funds.**

The protocol's instant withdrawal fee mechanism fails to collect fees on sub-threshold amounts. The fee recipient (treasury or configured address) receives zero instead of the promised fee. On mainnet, gas costs per call (~50,000 gas × gas price) far exceed the per-call fee savings (at most a few wei), making large-scale exploitation economically irrational. However, the invariant that every instant withdrawal pays a fee is broken, and the protocol silently under-collects revenue on any dust-sized call.

### Likelihood Explanation

Low. Requires `minRsEthAmountToWithdraw[asset]` to remain unset (zero) for the targeted asset, and the attacker must accept negative net economics (gas > fee saved) on mainnet. The condition is met by default for any newly supported asset until an admin explicitly calls `setMinRsEthAmountToWithdraw`.

### Recommendation

1. **Enforce a minimum withdrawal amount** by requiring `minRsEthAmountToWithdraw[asset] > 0` before enabling instant withdrawal for an asset, or set a protocol-wide floor in `initialize()`.
2. **Round the fee up** instead of down to prevent zero-fee outcomes:
   ```solidity
   uint256 fee = (assetAmountUnlocked * instantWithdrawalFee + 9_999) / 10_000;
   ```
   This mirrors the approach used in OpenZeppelin's `ERC4626Fees._feeOnRaw` which uses `Math.Rounding.Up`.

### Proof of Concept

1. `instantWithdrawalFee` is set to `100` (1%) by the LRT manager.
2. `minRsEthAmountToWithdraw[ETH]` is never set → defaults to `0`.
3. Attacker holds `1_000_000` wei of rsETH.
4. Attacker calls `instantWithdrawal(ETH, 99, "")` repeatedly (~10,101 times).
5. Each call: `assetAmountUnlocked ≈ 99` wei, `fee = (99 * 100) / 10_000 = 0`.
6. Attacker receives `99` wei of ETH per call with zero fee deducted.
7. Total received: `~1_000_000` wei of ETH, fee paid: `0` (should have been `~10_000` wei).
