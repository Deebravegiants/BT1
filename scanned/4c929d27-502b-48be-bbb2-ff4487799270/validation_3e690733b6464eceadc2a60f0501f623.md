### Title
Fee-on-Transfer Token Deposit Mints Excess rsETH, Diluting All Existing Holders — (`contracts/LRTDepositPool.sol`)

---

### Summary

`LRTDepositPool.depositAsset()` calculates the rsETH amount to mint using the caller-supplied `depositAmount` parameter **before** the `safeTransferFrom` call. If the deposited asset is a fee-on-transfer (deflationary) token, the contract receives fewer tokens than `depositAmount`, but mints rsETH as if the full `depositAmount` arrived. This inflates rsETH supply relative to actual TVL, permanently diluting all existing rsETH holders.

---

### Finding Description

In `LRTDepositPool.depositAsset()`:

```solidity
// contracts/LRTDepositPool.sol lines 110-117
uint256 rsethAmountToMint = _beforeDeposit(asset, depositAmount, minRSETHAmountExpected);

IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
_mintRsETH(rsethAmountToMint);
``` [1](#0-0) 

`_beforeDeposit` delegates to `getRsETHAmountToMint`, which computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [2](#0-1) 

Here `amount` is the raw `depositAmount` parameter — not the actual tokens received. For a fee-on-transfer token with a `f%` fee, the contract receives `depositAmount * (1 - f)` tokens but mints rsETH for `depositAmount` tokens. No balance snapshot (before/after transfer) is taken to reconcile the discrepancy.

The oracle's TVL computation in `_getTotalEthInProtocol()` calls `getTotalAssetDeposits(asset)`, which reads `IERC20(asset).balanceOf(address(this))` — the **actual** on-chain balance:

```solidity
assetLyingInDepositPool = IERC20(asset).balanceOf(address(this));
``` [3](#0-2) 

So after the deposit, the oracle's TVL reflects the real (reduced) balance, but rsETH supply is inflated. The next `updateRSETHPrice()` call computes:

```solidity
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
``` [4](#0-3) 

With an inflated `rsethSupply` and a real (lower) `totalETHInProtocol`, `newRsETHPrice` drops, permanently diluting every existing rsETH holder.

---

### Impact Explanation

**Critical — Protocol insolvency / direct theft of existing user funds.**

Every depositor using a fee-on-transfer supported asset extracts value from all existing rsETH holders. The attacker receives rsETH backed by `depositAmount` worth of assets but only contributes `depositAmount * (1 - fee)` worth of assets. The shortfall is socialized across all holders via a lower rsETH price. Repeated deposits compound the insolvency gap until the protocol cannot honor redemptions at the expected rate.

---

### Likelihood Explanation

**Medium.** The `onlySupportedERC20Token` modifier restricts deposits to admin-approved assets, so the vulnerability is only reachable if a fee-on-transfer token is ever added to the supported asset list. The protocol is explicitly designed to be extensible (`addNewSupportedAsset` is a governance function), and there is no on-chain guard preventing a fee-on-transfer token from being added. [5](#0-4)  No admin compromise is required — a well-intentioned governance action adding a new LST that happens to carry transfer fees is sufficient to trigger the bug.

---

### Recommendation

Replace the use of the raw `depositAmount` parameter with a balance-difference pattern to capture the actual received amount:

```solidity
uint256 balanceBefore = IERC20(asset).balanceOf(address(this));
IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
uint256 actualReceived = IERC20(asset).balanceOf(address(this)) - balanceBefore;

uint256 rsethAmountToMint = _beforeDeposit(asset, actualReceived, minRSETHAmountExpected);
_mintRsETH(rsethAmountToMint);
```

Alternatively, enforce at the governance level that only non-rebasing, non-fee-on-transfer tokens may be added as supported assets, and document this constraint explicitly.

---

### Proof of Concept

1. Governance adds `FeeToken` (1% fee-on-transfer) as a supported asset with a valid price oracle.
2. Attacker calls `depositAsset(FeeToken, 1000e18, 0, "")`.
3. `_beforeDeposit` computes `rsethAmountToMint` for `1000e18` tokens at the current rsETH price.
4. `safeTransferFrom` transfers `1000e18` but the contract only receives `990e18` (1% fee burned).
5. `_mintRsETH` mints rsETH equivalent to `1000e18` tokens — 1% excess.
6. `updateRSETHPrice()` is called: `totalETHInProtocol` reflects `990e18` worth of `FeeToken`; `rsethSupply` is inflated by the 1% excess.
7. `newRsETHPrice` decreases proportionally; all pre-existing rsETH holders now hold less ETH value per token.
8. Attacker repeats to compound the dilution, or immediately redeems their excess rsETH for other protocol assets, extracting value from honest depositors.
