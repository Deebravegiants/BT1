### Title
`allowPushers` Signature Replay Re-Establishes Revoked Delegation, Corrupting Oracle Namespace and Causing Stale-Price Execution - (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracle.allowPushers` verifies a pusher's EIP-191 consent signature that includes a `deadline` but **no nonce**. The same signature is valid for every call within the deadline window. A creator who saves a pusher's consent signature can replay it to re-establish delegation immediately after the pusher self-revokes via `revokePusher()`, indefinitely cycling the pusher's namespace remapping back to the creator's namespace without the pusher's ongoing consent.

---

### Finding Description

`allowPushers` signs over:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

There is no nonce, no used-signature registry, and no one-time-use mechanism. `_ensureDeadline` only checks `block.timestamp <= deadline`. [2](#0-1) 

The code comment explicitly acknowledges the deadline is required to prevent re-delegation after revocation, but a deadline alone is insufficient — it only prevents use *after* expiry, not repeated use *before* expiry: [3](#0-2) 

**Attack path:**

1. Creator C obtains pusher P's consent signature with `deadline = now + 1 day`.
2. C calls `allowPushers(deadline, [P], [sig])` → `namespaceRemapping[P] = C`.
3. P calls `revokePusher()` → `namespaceRemapping[P] = address(0)`. [4](#0-3) 

4. C immediately replays the **same** `sig` via `allowPushers(deadline, [P], [sig])` → `namespaceRemapping[P] = C` again.
5. Steps 3–4 repeat until `deadline` expires. P cannot permanently revoke within the deadline window.

When `namespaceRemapping[P] = C`, P's automated `fallback()` pushes resolve to C's namespace instead of P's own: [5](#0-4) 

P's own namespace (creator = P) stops receiving updates. Any pool whose price provider reads from P's namespace now reads a stale slot (`timestamp = last pre-attack value`).

---

### Impact Explanation

- **Oracle namespace corruption**: P's pushes are silently redirected to C's namespace. P's own namespace timestamp freezes.
- **Stale-price execution**: A pool whose `IPriceProvider` reads from P's namespace will receive a stale `refTime`. If the provider's staleness gate is not tight enough, the pool's `swap` path executes at an outdated bid/ask, causing traders to receive incorrect output amounts or LPs to suffer mispriced inventory.
- **Swap DoS**: If the provider does enforce staleness, every `swap` call reverts with `PriceProviderFailed`, making the pool's core swap functionality unusable until P's namespace is refreshed — which P cannot do while the delegation is active. [6](#0-5) 

---

### Likelihood Explanation

- **Precondition**: Creator C must have obtained a valid consent signature from P (normal operational flow when setting up delegation).
- **Trigger**: C saves the signature and replays it after each revocation — a single line of off-chain code.
- **No privilege required**: `allowPushers` is a public function; C is a normal user who legitimately received the signature.
- **Likelihood**: Low-Medium. Requires a malicious creator and an automated pusher whose namespace is consumed by a live pool.

---

### Recommendation

Add a per-pusher nonce to the signed payload and increment it on each successful `allowPushers` call, invalidating all previously issued signatures:

```solidity
mapping(address => uint256) public pusherNonces;

// Inside allowPushers loop:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender,
        pusherNonces[pusher]   // bind to current nonce
    ))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
pusherNonces[pusher]++;        // invalidate this signature
namespaceRemapping[pusher] = msg.sender;
```

This mirrors the fix recommended in the external report: a storage nonce that increments on every call, so each signed consent is single-use.

### Proof of Concept

```solidity
function test_allowPushers_replay_after_revoke() public {
    uint256 pusherKey = 0xBEEF;
    address pusher  = vm.addr(pusherKey);
    address creator = address(0xC0DE);
    uint256 deadline = block.timestamp + 1 days;

    // Pusher signs consent once
    bytes32 digest = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(oracle), deadline, pusher, creator))
    );
    (uint8 v, bytes32 r, bytes32 s) = vm.sign(pusherKey, digest);
    bytes memory sig = abi.encodePacked(r, s, v);

    address[] memory pushers = new address[](1);
    pushers[0] = pusher;
    bytes[] memory sigs = new bytes[](1);
    sigs[0] = sig;

    // Step 1: creator establishes delegation
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs);
    assertEq(oracle.namespaceRemapping(pusher), creator);

    // Step 2: pusher revokes
    vm.prank(pusher);
    oracle.revokePusher();
    assertEq(oracle.namespaceRemapping(pusher), address(0));

    // Step 3: creator replays the SAME signature — re-establishes delegation
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs); // identical call, no revert
    assertEq(oracle.namespaceRemapping(pusher), creator); // delegation restored without pusher consent
}
```
