### Title
Unrestricted `sweepToken` and `unwrapWETH9` Allow Any Caller to Drain Router Token Balances — (File: metric-periphery/contracts/base/PeripheryPayments.sol)

### Summary
`PeripheryPayments.sweepToken` and `PeripheryPayments.unwrapWETH9` are `public payable` with no caller restriction. Any address can call either function at any time, transferring the router's entire token or WETH balance to an arbitrary `recipient`. When users route swap output through the router (the documented WETH-unwrap pattern) in separate transactions instead of a single `multicall`, an attacker can front-run the sweep/unwrap step and steal the stranded tokens.

### Finding Description
Both helpers are unconditionally public:

```solidity
// PeripheryPayments.sol L48-55
function sweepToken(address token, uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceToken = IERC20(token).balanceOf(address(this));
    if (balanceToken < amountMinimum) revert InsufficientToken(token, amountMinimum, balanceToken);
    if (balanceToken > 0) {
        IERC20(token).safeTransfer(recipient, balanceToken);
    }
}

// PeripheryPayments.sol L37-45
function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(recipient, balanceWETH);
    }
}
```

Neither function checks `msg.sender` against any depositor record. The `recipient` parameter is fully caller-controlled.

The router legitimately holds tokens between steps in two documented patterns:

**Pattern 1 — WETH unwrap (single-hop):** A user calls `exactInputSingle` with `recipient: address(router)` to land WETH on the router, then calls `unwrapWETH9` to convert and forward ETH. The test `test_multicall_tokenForWeth_thenUnwrapEth` shows this is the intended flow, but it is only safe when both calls are batched inside a single `multicall`. If the user issues them as two separate transactions, the router holds WETH between them.

**Pattern 2 — multi-hop exact-input with router as intermediate recipient:** In `exactInput`, intermediate hops send output to `address(this)` (the router) before the next hop consumes it. If the final hop's output is directed to `address(router)` for a subsequent sweep, the same two-transaction race applies.

In both cases, once tokens sit on the router between transactions, any caller can invoke `sweepToken(token, 0, attacker)` or `unwrapWETH9(0, attacker)` to drain the full balance to an attacker-controlled address.

`MetricOmmSimpleRouter` and `MetricOmmPoolLiquidityAdder` both inherit `PeripheryPayments` without overriding or restricting these helpers, so both contracts are affected. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
Any tokens stranded on the router — from the WETH-unwrap pattern, from a multi-hop path that routes output through the router, or from any other source — can be stolen by an unprivileged attacker. The attacker calls `sweepToken` or `unwrapWETH9` with their own address as `recipient` and receives the router's entire balance. This is a direct loss of user principal with no recovery path. [4](#0-3) 

### Likelihood Explanation
The WETH-unwrap pattern is explicitly tested and documented as a supported flow. Users who do not know to use `multicall` — or who call the two steps in separate transactions for any reason (wallet UX, gas estimation, partial automation) — will have their tokens stolen. An attacker only needs to watch the mempool for transactions that send output to the router address and front-run the subsequent sweep or unwrap call. No special privilege or setup is required. [5](#0-4) 

### Recommendation
Restrict `sweepToken` and `unwrapWETH9` so they can only be invoked by `msg.sender` within a `multicall` context (e.g., require `msg.sender == address(this)` when called directly, since `multicall` uses `delegatecall`). Alternatively, track a per-caller depositor mapping so only the address that routed output to the router can sweep it. At minimum, add a prominent NatSpec warning that these functions must always be batched inside `multicall` and never called as standalone transactions. [6](#0-5) 

### Proof of Concept

1. Alice calls `exactInputSingle` with `recipient: address(router)` and `tokenOut: WETH`, intending to follow up with `unwrapWETH9`. Her transaction is confirmed; the router now holds `X` WETH belonging to Alice.
2. Bob (attacker) observes Alice's confirmed transaction in the block and immediately calls `router.unwrapWETH9(0, bob)` in the next block (or front-runs Alice's second transaction in the same block).
3. `unwrapWETH9` checks `balanceWETH = IERC20(WETH).balanceOf(address(router))` — finds `X` WETH — withdraws it all, and sends `X` ETH to `bob`.
4. Alice's subsequent `unwrapWETH9` call reverts with `InsufficientWETH(X, 0)`.
5. Alice loses `X` WETH; Bob gains `X` ETH with zero cost beyond gas.

The same attack applies to any ERC-20 via `sweepToken(token, 0, attacker)`. [1](#0-0) [7](#0-6)
