### Title
`removeLiquidity` can be DOSed by frontrunning `addLiquidity` to violate the `MinimalLiquidity` invariant — (`metric-core/contracts/libraries/LiquidityLib.sol`)

### Summary

`addLiquidity` allows any caller to credit shares to an arbitrary `owner`/`salt` position. `removeLiquidity` enforces that if any shares remain after withdrawal they must be ≥ `minimalMintableLiquidity`. An attacker can frontrun a victim's full-withdrawal `removeLiquidity` call by adding a dust amount of shares to the victim's position, causing the remaining-shares check to revert and permanently blocking the victim from exiting.

### Finding Description

`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address with no restriction that `msg.sender == owner`:

```solidity
function addLiquidity(
    address owner,   // ← no ownership check
    uint80 salt,
    LiquidityDelta calldata deltas,
    ...
) external nonReentrant(PoolActions.ADD_LIQUIDITY) ...
```

The `IMetricOmmModifyLiquidityCallback` is invoked on `msg.sender`, so the attacker pays for the tokens, but shares are written to `positionBinShares[keccak256(abi.encode(owner, salt, bin))]` — the victim's key.

`removeLiquidity` enforces:

```solidity
uint256 newUserShares = userShares - sharesToRemove;
if (newUserShares > 0 && newUserShares < ctx.minimalMintableLiquidity) {
    revert MinimalLiquidity(newUserShares, ctx.minimalMintableLiquidity);
}
``` [4](#0-3) 

When a victim submits a full withdrawal (`sharesToRemove == userShares`), `newUserShares` would be 0, which passes the check (`0 > 0` is false). But if the attacker frontruns and adds `delta` shares (where `0 < delta < minimalMintableLiquidity`) to the victim's position, then after the victim's transaction executes:

```
newUserShares = (userShares + delta) - userShares = delta
```

Since `delta > 0 && delta < minimalMintableLiquidity`, the `MinimalLiquidity` revert fires. The attacker can repeat this for every retry the victim makes.

### Impact Explanation

The victim's LP position is frozen: they cannot exit the pool. Their token principal remains locked in the pool indefinitely as long as the attacker continues to frontrun. This constitutes an unusable core liquidity-withdrawal flow and temporary loss of access to user principal.

### Likelihood Explanation

- `addLiquidity` is a public, permissionless function with no owner check — any EOA or contract can trigger it for any `owner`/`salt`.
- The victim's `owner`, `salt`, and `binIdxs` are visible in the pending transaction's calldata on a public mempool.
- The cost to the attacker per frontrun is the token value of `delta` shares (can be as low as 1 scaled unit, i.e., dust), making repeated griefing cheap.
- `removeLiquidity` correctly enforces `msg.sender == owner`, so the victim cannot be forced out; only their exit is blocked. [5](#0-4) 

### Recommendation

Add an ownership check to `addLiquidity` so that only the designated `owner` (or an approved operator) can credit shares to a position:

```solidity
if (msg.sender != owner) revert NotPositionOwner();
```

This mirrors the guard already present in `removeLiquidity` and eliminates the ability for third parties to manipulate another user's share balance.

Alternatively, if permissionless liquidity-on-behalf-of is an intentional design feature, the `MinimalLiquidity` check in `removeLiquidity` should be applied only when the caller is reducing (not fully exiting) their position, i.e., skip the check when `newUserShares == 0`.

### Proof of Concept

1. Victim holds `userShares = 1000` in bin 0 with `salt = 42`. `minimalMintableLiquidity = 100`.
2. Victim submits `removeLiquidity(victim, 42, {binIdxs:[0], shares:[1000]}, "")` to remove all shares. Expected: `newUserShares = 0`, check `0 > 0` → false → passes.
3. Attacker observes the pending transaction in the mempool, reads `owner = victim`, `salt = 42`, `binIdxs = [0]`, `shares = [1000]`.
4. Attacker submits `addLiquidity(victim, 42, {binIdxs:[0], shares:[1]}, ...)` with higher gas. Attacker pays for 1 share worth of tokens; victim's `positionBinShares[key]` becomes `1001`.
5. Victim's transaction executes: `userShares = 1001`, `sharesToRemove = 1000`, `newUserShares = 1`. Check: `1 > 0 && 1 < 100` → **revert `MinimalLiquidity(1, 100)`**.
6. Attacker repeats step 4 for every retry. Victim's funds remain locked. [6](#0-5) [7](#0-6)
