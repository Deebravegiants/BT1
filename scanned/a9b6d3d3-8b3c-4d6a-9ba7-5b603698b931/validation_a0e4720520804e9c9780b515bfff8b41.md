### Title
Unconditional `metricOmmSwapCallback` invocation with zero deltas causes `MetricOmmSimpleRouter` to always revert on no-op swaps - (`metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`MetricOmmPool.swap()` invokes `metricOmmSwapCallback` on `msg.sender` unconditionally after every swap execution, including when `_executeSwap` returns `(0, 0, 0)` (both deltas zero). `MetricOmmSimpleRouter.metricOmmSwapCallback()` contains a guard that reverts with `InvalidSwapDeltas` whenever `amount0Delta <= 0 && amount1Delta <= 0`. When a swap legitimately produces zero deltas (price limit immediately hit), the pool still fires the callback, the router's guard triggers, and the entire transaction reverts â€” making the swap path through the router permanently unusable for that class of inputs.

---

### Finding Description

**Root cause â€” unconditional callback in `MetricOmmPool.swap()`:** [1](#0-0) 

For both `zeroForOne` branches the callback is fired with no guard on whether either delta is non-zero:

```solidity
// zeroForOne branch
uint256 balance0Before = balance0();
IMetricOmmSwapCallback(msg.sender).metricOmmSwapCallback(amount0Delta, amount1Delta, callbackData);
```

**When can both deltas be zero?**

The internal swap helpers explicitly return `(0, 0, 0)` when the price limit is immediately satisfied: [2](#0-1) 

```solidity
if (params.priceLimitX64 >= initialPriceX64) {
    return (0, 0, 0);
}
```

The interface documentation explicitly acknowledges this: *"values that never cross the active path behave as no-op swaps (zero deltas)"*. [3](#0-2) 

**Revert in `MetricOmmSimpleRouter.metricOmmSwapCallback()`:** [4](#0-3) 

```solidity
function metricOmmSwapCallback(int256 amount0Delta, int256 amount1Delta, bytes calldata data) external override {
    if (amount0Delta <= 0 && amount1Delta <= 0) revert InvalidSwapDeltas();
```

When the pool calls this with `(0, 0, callbackData)`, the condition `0 <= 0 && 0 <= 0` is `true`, so `InvalidSwapDeltas` is thrown. Because this revert propagates back through the pool's `swap()` call, the entire transaction reverts â€” including any token transfer the pool may have already sent to `recipient`.

**Execution path:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle()` with a `priceLimitX64` that is at or beyond the current oracle price.
2. Pool calls `_executeSwap` â†’ internal helper returns `(0, 0, 0)`.
3. Pool unconditionally calls `IMetricOmmSwapCallback(msg.sender).metricOmmSwapCallback(0, 0, "")`.
4. Router guard fires: `revert InvalidSwapDeltas()`.
5. Entire transaction reverts. [5](#0-4) 

---

### Impact Explanation

Any swap routed through `MetricOmmSimpleRouter` (the primary user-facing router) that results in zero deltas â€” a documented, reachable state â€” is permanently blocked. This breaks the core swap flow for:

- `exactInputSingle` with a `priceLimitX64` that is immediately satisfied by the current oracle price.
- `exactOutputSingle` with a price limit that prevents any bin from being crossed.
- Multi-hop `exactInput` / `exactOutput` paths where an intermediate hop produces zero deltas.

The swap path through the router is rendered unusable for this class of inputs. No user funds are directly drained, but the swap functionality is broken for a documented, valid input range.

---

### Likelihood Explanation

The zero-delta case is reachable by any unprivileged caller via a standard `exactInputSingle` call. It occurs whenever:

- The caller supplies a `priceLimitX64` that is at or beyond the current oracle mid-price (e.g., a conservative slippage guard that the oracle has already moved past).
- Oracle price moves between transaction submission and execution, pushing the limit into the immediately-satisfied range.

Both scenarios are normal in live trading conditions, making this a medium-likelihood issue.

---

### Recommendation

**Option A (preferred â€” fix in pool):** Skip the callback entirely when both deltas are zero, mirroring the pattern used in `LiquidityLib.addLiquidity` which guards its callback with `if (amount0Added > 0 || amount1Added > 0)`: [6](#0-5) 

```solidity
// In MetricOmmPool.swap(), before firing the callback:
if (amount0Delta != 0 || amount1Delta != 0) {
    IMetricOmmSwapCallback(msg.sender).metricOmmSwapCallback(amount0Delta, amount1Delta, callbackData);
}
```

**Option B (fix in router):** Change the router guard from `<=` to `<` so that the zero-delta case is handled gracefully (no payment needed, return early):

```solidity
if (amount0Delta < 0 && amount1Delta < 0) revert InvalidSwapDeltas();
if (amount0Delta == 0 && amount1Delta == 0) return; // no-op swap, nothing to pay
```
