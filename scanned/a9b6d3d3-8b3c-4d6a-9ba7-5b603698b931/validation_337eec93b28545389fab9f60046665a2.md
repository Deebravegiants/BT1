### Title
Exact-Output Swap Silently Delivers Zero Tokens Instead of Reverting on Insufficient Liquidity — (`metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`MetricOmmPool` silently caps the output of exact-out swaps to whatever liquidity is available (including zero) rather than reverting. A caller who requests a specific output amount receives less — or nothing — while the transaction succeeds and emits a `Swap` event with zero deltas. This is the direct analog of the yAxis controller not reverting on insufficient liquidity.

---

### Finding Description

In `_swapToken1ForToken0SpecifiedOutput`, before the swap loop runs, the requested output is unconditionally clamped to the pool's total available balance:

```solidity
uint256 totalAvailableToken0Scaled = binTotals.scaledToken0;
if (amountOutScaled > totalAvailableToken0Scaled) {
    amountOutScaled = totalAvailableToken0Scaled;   // silent cap — no revert
}
``` [1](#0-0) 

When `totalAvailableToken0Scaled == 0` (all liquidity drained), `amountOutScaled` becomes `0`, the `while` loop body never executes (`amountSpecifiedRemainingScaled == 0`), and the function returns `(0, 0, 0, 0)`. Back in `_executeSwap`, both `amount0DeltaScaled` and `amount1DeltaScaled` are set to zero:

```solidity
amount0DeltaScaled = -int256(amountOutScaled);   // 0
amount1DeltaScaled =  int256(amountInScaled);    // 0
``` [2](#0-1) 

In `swap()`, because `amount0Delta == 0` no token is transferred to the recipient, and the callback is invoked with `(0, 0)` so the caller pays nothing. The `IncorrectDelta` guard is skipped because `amount1Delta == 0`:

```solidity
if (amount1Delta > 0 && balance1Before + uint256(amount1Delta) > balance1()) {
    revert IncorrectDelta();
}
``` [3](#0-2) 

The transaction completes successfully, emitting a `Swap` event with zero amounts. No error is surfaced to the caller.

The symmetric path (`_swapToken0ForToken1SpecifiedOutput`, `zeroForOne = true, amountSpecified < 0`) follows the same pattern.

---

### Impact Explanation

- **Broken exact-out invariant**: callers that rely on receiving exactly the requested amount (e.g., routers, aggregators, on-chain arbitrage bots) silently receive nothing and cannot distinguish a successful zero-fill from a real swap.
- **MEV sandwich / griefing**: an attacker who observes a pending exact-out swap can drain the relevant bin liquidity in a front-run, causing the victim's transaction to silently produce `(0, 0)` deltas. The victim wastes gas; the attacker back-runs to restore liquidity at a profit from the price impact.
- **Slippage protection is ineffective**: `priceLimitX64` guards against price movement, not against a zero-liquidity no-op. If liquidity is drained without crossing the victim's price limit, the price guard does not trigger.

---

### Likelihood Explanation

Any caller submitting an exact-out swap (`amountSpecified < 0`) against a pool whose relevant bin has been drained — whether by a concurrent swap, a front-runner, or natural trading — will silently receive zero output. No special privilege is required; the attack is executable by any EOA or contract that can call `swap`.

---

### Recommendation

After the swap loop completes, compare the actual output delivered against the originally requested amount and revert if the pool could not fill the order:

```solidity
uint256 actualOut = amountOutScaled - state.amountSpecifiedRemainingScaled;
if (actualOut < requestedAmountOutScaled) revert InsufficientLiquidity();
```

Alternatively, expose the partial-fill amount in the return value and let the caller (router/aggregator) enforce its own minimum-output check — but the pool itself must revert by default so that callers without explicit guards are protected.

---

### Proof of Concept

```solidity
// 1. Attacker drains all token0 from the pool via a normal exact-in swap.
pool.swap(attacker, false /*token1→token0*/, type(int128).max, 0, ...);

// 2. Victim submits exact-out swap requesting 1 000 token0.
//    amountSpecified = -1000e18 (exact out)
(int128 d0, int128 d1) = pool.swap(victim, false, -1000e18, 0, ...);
// d0 == 0, d1 == 0  — victim receives nothing, pays nothing, tx succeeds silently.

// 3. Attacker back-runs: re-adds or re-buys liquidity at the shifted price.
```

The victim's transaction succeeds with a `Swap(... 0, 0 ...)` event. No revert, no slippage error, no output.
