### Title
Zero `rsETHAmount` Minted on Dust Deposits Due to Integer Division Truncation — (File: contracts/pools/RSETHPoolV3.sol)

---

### Summary

The `viewSwapRsETHAmountAndFee` function used across all L2 deposit pool variants computes `rsETHAmount` via integer division that can silently truncate to zero for dust-sized deposits. No downstream guard checks for a zero mint amount before calling `wrsETH.mint(msg.sender, rsETHAmount)`. The result is that a depositor's ETH or LST is accepted by the pool while they receive zero `wrsETH` in return.

---

### Finding Description

`viewSwapRsETHAmountAndFee` computes the output amount as:

```solidity
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [1](#0-0) 

Because `rsETHToETHrate` is the rsETH/ETH exchange rate (currently > `1e18` as rsETH appreciates), any deposit where `amountAfterFee * 1e18 < rsETHToETHrate` produces `rsETHAmount = 0` via integer truncation. With `feeBps ≤ 1000` (enforced by `setFeeBps`), `amountAfterFee ≥ 0.9 × amount`, so a 1-wei ETH deposit yields `amountAfterFee = 1`, and `1 × 1e18 / ~1.05e18 = 0`. [2](#0-1) 

The deposit function checks only that `amount != 0`, not that the computed `rsETHAmount != 0`:

```solidity
if (amount == 0) revert InvalidAmount();
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
wrsETH.mint(msg.sender, rsETHAmount);   // rsETHAmount may be 0
``` [3](#0-2) 

`RsETHTokenWrapper.mint` delegates directly to OpenZeppelin `_mint`, which does not revert on a zero amount:

```solidity
function mint(address _to, uint256 _amount) external onlyRole(MINTER_ROLE) {
    _mint(_to, _amount);
}
``` [4](#0-3) 

The same pattern is present in `RSETHPoolV3ExternalBridge.deposit`, `RSETHPoolV3WithNativeChainBridge.deposit`, and `RSETHPoolNoWrapper.deposit` (which calls `rsETH.safeTransfer(msg.sender, 0)` — also a no-op). [5](#0-4) [6](#0-5) 

The same truncation path exists for token deposits:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [7](#0-6) 

For tokens with a rate much lower than `rsETHToETHrate`, larger amounts can also truncate to zero.

---

### Impact Explanation

**Low — Contract fails to deliver promised returns.**

A depositor who sends a dust amount of ETH (or a token with a low oracle rate) has their funds accepted by the pool — `feeEarnedInETH` is incremented by the fee portion and the remainder sits in the pool's ETH balance — while receiving zero `wrsETH`. The ETH is not returned; it is permanently absorbed into the pool's bridgeable balance. The user receives nothing for their deposit.

---

### Likelihood Explanation

**Low.** The condition is only reachable for deposits of 1 wei of ETH (or equivalently tiny token amounts). Normal users depositing meaningful amounts are unaffected. However, the check is entirely absent, so any dust deposit — whether accidental or from a script — silently fails to mint.

---

### Recommendation

Add an explicit zero-output guard in every deposit function immediately after computing `rsETHAmount`:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
if (rsETHAmount == 0) revert InvalidAmount();
```

This mirrors the fix described in the referenced report (checking for zero before minting) and ensures depositors always receive a non-zero token amount or have their transaction reverted cleanly.

---

### Proof of Concept

1. Deploy `RSETHPoolV3` with `feeBps = 0` and an oracle returning `rsETHToETHrate = 1.05e18`.
2. Call `deposit{value: 1}("")` — sending 1 wei of ETH.
3. `viewSwapRsETHAmountAndFee(1)` returns `rsETHAmount = 1 * 1e18 / 1.05e18 = 0`, `fee = 0`.
4. `feeEarnedInETH += 0`; pool ETH balance increases by 1 wei.
5. `wrsETH.mint(msg.sender, 0)` executes without revert; caller's `wrsETH` balance is unchanged.
6. Caller has lost 1 wei with no recourse.
