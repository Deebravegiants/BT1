### Title
`msg.sender` Hardcoded as LayerZero Refund Address Causes Excess Fee Loss for Contract Callers - (File: contracts/bridges/TACWETHBridge.sol)

### Summary
`TACWETHBridge.bridgeTokenToL1` is a publicly callable function that passes `msg.sender` directly as the `_refundAddress` to the LayerZero OFT `send()` call. When the caller is a contract that lacks a `receive()` fallback, any excess native fee refunded by the LayerZero endpoint on the source chain (TAC) is permanently lost.

### Finding Description
`TACWETHBridge.bridgeTokenToL1` has no access control and is callable by any address, including contracts. [1](#0-0) 

Inside the function, the native fee is quoted via `getNativeFee` and the caller is required to send exactly that amount: [2](#0-1) 

The LayerZero OFT `send()` is then invoked with `msg.sender` hardcoded as the `_refundAddress`: [3](#0-2) 

In LayerZero V2, `_refundAddress` is a **source-chain** address that receives any excess native fee not consumed by the endpoint (e.g., when the actual fee is lower than the quoted fee due to gas price fluctuations between `quoteSend` and `send`, or when the endpoint internally adjusts the fee). If `msg.sender` is a contract without a `receive()` function, the ETH transfer to that address fails. Depending on the LayerZero endpoint implementation, this either silently loses the excess fee or reverts the entire bridge call.

The `IOFT.send` interface confirms the third parameter is the refund address: [4](#0-3) 

By contrast, the operator-restricted `bridgeKernelToBSC` in `KernelVaultETH` correctly accepts an explicit `refundAddress` parameter from the caller: [5](#0-4) 

`TACWETHBridge` provides no equivalent parameter, forcing `msg.sender` as the refund destination with no override path.

### Impact Explanation
Any excess native fee refunded by the LayerZero endpoint on TAC is sent to `msg.sender`. If `msg.sender` is a contract without a `receive()` function (e.g., a multisig, a DeFi aggregator, or a smart-contract wallet), the ETH transfer reverts or is silently dropped, permanently losing the refunded amount. This matches **Low — contract fails to deliver promised returns, but doesn't lose value** (the bridged WETH itself is unaffected; only the fee refund is lost).

### Likelihood Explanation
`bridgeTokenToL1` is unrestricted and is the standard entry point for any caller wishing to bridge WETH from TAC to L1. Contract callers — multisigs, aggregators, vaults — are a realistic and common use case for bridge wrappers. The `IL2TokenBridge` interface is also designed to be called by pool contracts: [6](#0-5) 

This makes contract-originated calls a normal, expected interaction pattern.

### Recommendation
Add an explicit `refundAddress` parameter to `bridgeTokenToL1`, defaulting to `msg.sender` only when the caller is an EOA, or require the caller to supply it explicitly:

```solidity
function bridgeTokenToL1(address recipient, uint256 amount, address refundAddress) external payable nonReentrant {
    UtilLib.checkNonZeroAddress(refundAddress);
    // ...
    wethOFT.send{ value: nativeFee }(sendParam, fee, refundAddress);
}
```

This mirrors the pattern already used in `KernelVaultETH.bridgeKernelToBSC`. [7](#0-6) 

### Proof of Concept
1. A multisig contract (no `receive()`) calls `TACWETHBridge.bridgeTokenToL1(recipient, amount)` with `msg.value == nativeFee`.
2. The LayerZero endpoint processes the send and determines the actual fee is slightly less than `nativeFee` (gas price dropped within the block).
3. The endpoint attempts to refund the excess to `msg.sender` (the multisig).
4. The ETH transfer to the multisig fails because it has no `receive()` function.
5. The excess native fee is permanently lost; the multisig receives no refund. [8](#0-7)
