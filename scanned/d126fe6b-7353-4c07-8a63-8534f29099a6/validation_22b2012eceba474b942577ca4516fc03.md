### Title
`PriceVelocityGuardExtension` Uses `block.number` Instead of `block.timestamp` to Measure Price Velocity, Making the Guard Systematically Miscalibrated on L2s - (File: `metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

---

### Summary

`PriceVelocityGuardExtension.beforeSwap` measures elapsed time using `block.number` and a per-block rate parameter (`maxChangePerBlockE18`). Because block production rates differ dramatically across chains (Ethereum ~12 s/block, Arbitrum ~0.25 s/block, Optimism/Base ~2 s/block), the same `maxChangePerBlockE18` value produces a guard that is up to **48× weaker per real-time second** on fast-block L2s than the admin intended. Swaps at oracle prices that moved far beyond the intended velocity envelope are accepted, exposing LPs to bad-price execution.

---

### Finding Description

`PriceVelocityGuardExtension` is a `beforeSwap` hook that caps how fast the oracle mid-price can move between consecutive swaps. The allowed change is computed as:

```
allowedSq = maxChange² × (1 + blockDiff)
```

where `blockDiff = block.number - prevBlock`. [1](#0-0) 

The state struct stores `lastUpdateBlock` as a `uint64` block number, not a timestamp: [2](#0-1) 

Both the `beforeSwap` hook and the admin setter `setLastMidPrice` write `block.number` into `lastUpdateBlock`: [3](#0-2) [4](#0-3) 

The admin configures `maxChangePerBlockE18` via `setMaxChangePerBlock`: [5](#0-4) 

The NatSpec explicitly frames the guard in terms of blocks-per-unit: [6](#0-5) 

Because the parameter is named and documented as "per block," any admin calibrating it will reason about block intervals. On Ethereum mainnet (~12 s/block) a setting of, say, `maxChangePerBlockE18 = 1e16` (1% per block) means ≈0.083% per second. On Arbitrum (~0.25 s/block) the same value means ≈4% per second — **48× looser** for the same real-time window. The error is not correctable after the fact without an admin transaction, and every swap that slips through at a manipulated price is a permanent LP loss.

---

### Impact Explanation

The velocity guard is the sole on-chain rate-of-change check on the oracle mid-price in the swap path. When it is too loose:

1. An attacker (or a fast oracle update) can push the oracle mid-price far beyond the intended envelope within a single real-time second.
2. `beforeSwap` passes, the pool executes the swap at the manipulated bid/ask, and LPs receive less token input (or pay more token output) than the true market price warrants.
3. The loss is permanent — the swap has already settled tokens before any admin can react.

This matches the allowed impact gate: **bad-price execution** (stale/unbounded bid/ask quote reaches a pool swap) and **direct loss of LP assets**.

---

### Likelihood Explanation

- The Metric OMM protocol targets multi-chain deployment; the periphery contracts contain an explicit L2 price provider (`PriceProviderL2.sol`) with L2-aware staleness logic, confirming L2 deployment is in scope.
- Any pool admin who enables `PriceVelocityGuardExtension` on an L2 and calibrates `maxChangePerBlockE18` using mainnet block-time intuition (or copies a mainnet value) will silently deploy a guard that is orders of magnitude weaker than intended.
- No privileged attacker is required: any public swap call triggers `beforeSwap`, and the miscalibrated guard simply does not revert when it should.

---

### Recommendation

Replace `block.number` / `lastUpdateBlock` / `maxChangePerBlockE18` with `block.timestamp` / `lastUpdateTimestamp` / `maxChangePerSecondE18` throughout `PriceVelocityGuardExtension` and `IPriceVelocityGuardExtension`. The velocity formula becomes:

```solidity
uint256 secondsDiff = block.timestamp - prevTimestamp;
uint256 allowedSq = uint256(maxChangePerSecond) * uint256(maxChangePerSecond) * (1 + secondsDiff);
```

`block.timestamp` manipulation by validators is bounded to a few seconds and is non-cumulative, making it far safer than block-number arithmetic for cross-chain rate limiting — exactly the lesson from the Compound `WhitePaperInterestRateModel` bug.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Scenario: pool deployed on Arbitrum (~0.25 s/block).
// Admin sets maxChangePerBlockE18 = 1e16 (1% per block),
// intending to allow ~0.083% per second (mainnet 12 s/block).
// On Arbitrum, 48 blocks pass per 12 seconds.
// allowedSq = (1e16)^2 * (1 + 48) = 1e32 * 49
// sqrt(allowedSq) / 1e18 = 7% — 7× the intended 1% per block.

// Step 1: Admin deploys pool on Arbitrum, attaches PriceVelocityGuardExtension,
//         calls setMaxChangePerBlock(pool, 1e16).
// Step 2: At block N, a swap executes; lastMidPriceX64 = P, lastUpdateBlock = N.
// Step 3: 12 real seconds later (block N+48 on Arbitrum), oracle price has moved 6%.
//         blockDiff = 48
//         changeE18 = 6e16
//         actualSq  = (6e16)^2 = 3.6e33
//         allowedSq = (1e16)^2 * 49 = 4.9e33
//         3.6e33 < 4.9e33 → guard PASSES, swap executes at manipulated price.
// Step 4: On mainnet with the same 12-second gap (1 block):
//         blockDiff = 1
//         allowedSq = (1e16)^2 * 2 = 2e32
//         3.6e33 > 2e32 → guard REVERTS as intended.
//
// LPs on Arbitrum lose funds on every swap that exploits this 7× wider envelope.
```
