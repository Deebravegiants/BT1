### Title
`addLiquidity` accepts `owner = address(0)`, permanently locking deposited tokens - (File: metric-core/contracts/MetricOmmPool.sol)

### Summary

`MetricOmmPool.addLiquidity` does not validate that the `owner` parameter is non-zero. When `owner = address(0)` is supplied, the pool mints LP shares to the zero-address position key and pulls real tokens from `msg.sender` via the modify-liquidity callback. Because `removeLiquidity` enforces `msg.sender == owner`, those shares can never be redeemed, permanently locking the deposited tokens inside the pool.

### Finding Description

`MetricOmmPool.addLiquidity` (L182–196) accepts an arbitrary `owner` address with no zero-address guard:

```solidity
function addLiquidity(
    address owner,          // ← no require(owner != address(0))
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
    if (deltas.binIdxs.length == 0) return (0, 0);
    if (deltas.binIdxs.length != deltas.shares.length) revert LiquidityDeltaLengthMismatch();
    _beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
    (amount0Added, amount1Added) = LiquidityLib.addLiquidity(
        _liquidityContext(), owner, salt, deltas, callbackData, ...
    );
    ...
}
``` [1](#0-0) 

Inside `LiquidityLib.addLiquidity`, the position key is computed as:

```solidity
bytes32 posKey = _positionBinKey(owner, salt, int8(binIdx));
// = keccak256(abi.encode(owner, salt, bin))
``` [2](#0-1) 

When `owner = address(0)`, this key is a valid but permanently inaccessible storage slot. Shares are written to it and bin balances are updated, then the callback pulls real tokens from `msg.sender`: [3](#0-2) 

`removeLiquidity` then enforces:

```solidity
if (msg.sender != owner) revert NotPositionOwner();
``` [4](#0-3) 

Since `msg.sender` can never equal `address(0)` in a live EVM transaction, the shares minted to the zero-address position key are permanently irrecoverable.

The periphery contract `MetricOmmPoolLiquidityAdder` does guard against this via `_validateOwner(owner)`, which reverts with `InvalidPositionOwner` for a zero address: [5](#0-4) 

However, the pool itself has no such guard. Any direct caller — a custom router, an integration contract, or an EOA — that passes `address(0)` as `owner` will lose their deposited tokens permanently. The test suite confirms the periphery check exists but the pool-level check does not: [6](#0-5) 

### Impact Explanation

Tokens deposited by `msg.sender` are pulled via `metricOmmModifyLiquidityCallback` and credited to the pool's bin balances (`binTotals.scaledToken0/1` and per-bin `token0/1BalanceScaled`). The corresponding LP shares are recorded under the zero-address position key. Since `removeLiquidity` requires `msg.sender == owner` and `msg.sender` can never be `address(0)`, the deposited tokens are permanently locked inside the pool — a direct, irreversible loss of user principal.

### Likelihood Explanation

The pool's `addLiquidity` is a public, permissionless function. Any direct caller that bypasses the periphery and passes `owner = address(0)` — whether by mistake, through a buggy integration, or through a contract that uses `address(0)` as a sentinel — will trigger this loss. The periphery guard is not enforced at the pool level.

### Recommendation

Add a zero-address check for `owner` at the top of `MetricOmmPool.addLiquidity`, mirroring the periphery's existing guard:

```solidity
if (owner == address(0)) revert InvalidOwner();
```

### Proof of Concept

1. Alice calls `pool.addLiquidity(address(0), salt, deltas, callbackData, "")` directly (bypassing `MetricOmmPoolLiquidityAdder`).
2. `LiquidityLib.addLiquidity` computes `posKey = keccak256(abi.encode(address(0), salt, binIdx))` and mints shares to that key; bin balances are updated.
3. The callback fires and pulls tokens from Alice.
4. Alice attempts `pool.removeLiquidity(address(0), salt, deltas, "")` — it reverts with `NotPositionOwner` because `msg.sender (Alice) != owner (address(0))`.
5. Alice's tokens are permanently locked in the pool with no recovery path.
