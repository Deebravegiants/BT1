### Title
Missing Slippage Protection on `removeLiquidity` Exposes LPs to Sandwich Attacks - (File: metric-core/contracts/libraries/LiquidityLib.sol)

### Summary

`MetricOmmPool.removeLiquidity` accepts only a share count to burn and provides no `minAmount0`/`minAmount1` floor parameters. Because bin composition changes with every swap, an LP's withdrawal can be front-run to force an unfavorable token mix with no on-chain revert path. The protocol provides maximum-amount caps for `addLiquidity` via `MetricOmmPoolLiquidityAdder`, but provides no equivalent minimum-amount protection for `removeLiquidity`.

### Finding Description

`LiquidityLib.removeLiquidity` computes the tokens owed to the LP as a proportional share of the current bin state:

```solidity
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
uint256 amount1Scaled = _checkedMul(binState.token1BalanceScaled, sharesToRemove) / binTotalSharesVal;
``` [1](#0-0) 

These values are entirely determined by `binState.token0BalanceScaled` and `binState.token1BalanceScaled` at execution time. Every swap that crosses the LP's bin mutates those fields. The pool's `removeLiquidity` entry point accepts no minimum-amount guard:

```solidity
function removeLiquidity(address owner, uint80 salt, LiquidityDelta calldata deltas, bytes calldata extensionData)
    external
    nonReentrant(PoolActions.REMOVE_LIQUIDITY)
    returns (uint256 amount0Removed, uint256 amount1Removed)
``` [2](#0-1) 

Tokens are transferred directly to `owner` with no callback and no post-transfer check against a user-supplied floor: [3](#0-2) 

By contrast, `addLiquidity` is wrapped by `MetricOmmPoolLiquidityAdder.addLiquidityExactShares`, which enforces `maxAmountToken0` / `maxAmountToken1` caps in the callback before any token pull:

```solidity
if (amount0Delta > max0 || amount1Delta > max1) {
    revert MaxAmountExceeded(amount0Delta, amount1Delta, max0, max1);
}
``` [4](#0-3) 

No equivalent periphery wrapper exists for `removeLiquidity`. The asymmetry means deposit-side slippage is guarded while withdrawal-side slippage is completely unguarded.

### Impact Explanation

An LP withdrawing from the active bin (bin 0) or any bin that is reachable by a swap can receive a materially different token composition than anticipated. If the LP expected predominantly token0 but a front-run swap converted the bin's token0 to token1, the LP receives token1 instead and must execute an additional swap to recover their preferred asset, paying spread fees. In the worst case (full bin conversion), the LP receives 100% of the undesired token. Because `removeLiquidity` transfers tokens unconditionally and returns without any floor check, the LP has no on-chain recourse.

### Likelihood Explanation

Any pending `removeLiquidity` transaction in the mempool is visible to MEV searchers. The active bin is the most liquid target; a single swap can shift its entire composition. On high-throughput chains (Base, HyperEVM) targeted by this protocol, block builders routinely reorder transactions. The LP has no deadline parameter and no minimum-amount parameter to prevent execution under adverse conditions.

### Recommendation

Add `minAmount0` and `minAmount1` parameters to `removeLiquidity` (or to a new periphery wrapper `MetricOmmPoolLiquidityRemover`) and revert if the computed `amount0Removed < minAmount0 || amount1Removed < minAmount1`. A deadline parameter should also be added to bound the execution window, mirroring the protection already present in `MetricOmmSimpleRouter` swap functions.

```solidity
function removeLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    uint256 minAmount0,   // <-- add
    uint256 minAmount1,   // <-- add
    uint256 deadline,     // <-- add
    bytes calldata extensionData
) external returns (uint256 amount0Removed, uint256 amount1Removed) {
    require(block.timestamp <= deadline, DeadlineExpired());
    // ... existing logic ...
    if (amount0Removed < minAmount0 || amount1Removed < minAmount1)
        revert InsufficientOutput(amount0Removed, amount1Removed);
}
```

### Proof of Concept

1. LP holds 10 000 shares in bin 0 (current bin), which currently holds 50 % token0 / 50 % token1.
2. LP submits `removeLiquidity(owner, salt, {binIdxs:[0], shares:[10000]}, "")` expecting ≈ 5 000 units of each token.
3. Attacker observes the pending transaction and front-runs with a large `swap(zeroForOne=false, ...)` that exhausts all token1 in bin 0, converting it entirely to token0.
4. LP's `removeLiquidity` executes: `binState.token1BalanceScaled == 0`, so `amount1Scaled = 0`. LP receives only token0 — zero token1. [1](#0-0) 

5. Attacker back-runs with the reverse swap to extract the spread profit.
6. LP receives 100 % token0 with no ability to revert, and must swap token0 → token1 at market cost to restore their intended position. [5](#0-4) [6](#0-5)
