### Title
Updating `LRT_WITHDRAW_MANAGER` in `LRTConfig` Breaks Pending Withdrawal Requests, Freezing User rsETH - (File: contracts/LRTUnstakingVault.sol)

---

### Summary

`LRTConfig.setContract()` can update the `LRT_WITHDRAW_MANAGER` address at any time without checking for pending withdrawal requests. Because `LRTUnstakingVault.onlyLRTWithdrawalManager` reads the withdrawal manager address **dynamically** from `LRTConfig` on every call, updating the address immediately revokes the old withdrawal manager's ability to call `redeem()`. Any user who has already called `initiateWithdrawal()` — transferring their rsETH into the old withdrawal manager — can no longer complete their withdrawal, freezing their rsETH.

---

### Finding Description

`LRTConfig.setContract()` is a general-purpose setter with no guard against updating critical in-flight contract references: [1](#0-0) 

`LRTUnstakingVault` enforces that only the current withdrawal manager (as recorded in `LRTConfig`) may call `redeem()`: [2](#0-1) [3](#0-2) 

The withdrawal lifecycle is:

1. **`initiateWithdrawal`** — user's rsETH is pulled into `LRTWithdrawalManager`; `assetsCommitted` is incremented. [4](#0-3) 

2. **`unlockQueue`** — operator burns the queued rsETH and calls `unstakingVault.redeem()` to pull assets into the withdrawal manager. [5](#0-4) 

3. **`completeWithdrawal`** — user receives the assets from the withdrawal manager.

If the admin calls `LRTConfig.setContract(LRT_WITHDRAW_MANAGER, newAddress)` while step 1 has occurred but step 2 has not, the old withdrawal manager's call to `unstakingVault.redeem()` will revert because `lrtConfig.withdrawManager()` now returns the new address. Step 2 is permanently blocked for all requests queued in the old manager.

Contrast this with `updateAssetStrategy`, which explicitly checks that no funds remain in the old strategy before allowing the update: [6](#0-5) 

No equivalent guard exists for `setContract`.

---

### Impact Explanation

Users who called `initiateWithdrawal` before the withdrawal manager update have their rsETH locked inside the old `LRTWithdrawalManager`. There is no rescue path: the old manager has no function to return rsETH to users, and `unlockQueue` (which burns rsETH and pulls assets) will revert on the `redeem()` call. The rsETH is effectively frozen until the admin intervenes (e.g., reverting the config change or upgrading the old manager contract).

**Impact: Medium — Temporary freezing of funds.**

---

### Likelihood Explanation

Updating the withdrawal manager is a realistic operational event during protocol upgrades. The admin may not be aware that pending withdrawal requests exist at the time of the update, or may not realize that `setContract` takes effect immediately and breaks in-flight withdrawals. No on-chain check prevents this. The scenario is analogous to the RabbitHole M-06 finding where updating the receipt contract address broke existing quests.

---

### Recommendation

Add a guard in `LRTConfig.setContract` (or in a dedicated `setWithdrawalManager` function) that checks whether the current withdrawal manager has any pending (locked) withdrawal requests before allowing the update. Alternatively, mirror the pattern used in `updateAssetStrategy`:

```solidity
function setWithdrawalManagerContract(address newManager) external onlyRole(DEFAULT_ADMIN_ROLE) {
    address current = contractMap[LRTConstants.LRT_WITHDRAW_MANAGER];
    if (current != address(0)) {
        // Ensure no pending locked requests exist in the old manager
        require(!ILRTWithdrawalManager(current).hasPendingLockedRequests(), "Pending withdrawals exist");
    }
    _setContract(LRTConstants.LRT_WITHDRAW_MANAGER, newManager);
}
```

---

### Proof of Concept

1. Alice calls `LRTWithdrawalManager.initiateWithdrawal(stETH, 10e18, ...)`. Her 10 rsETH is transferred to the old withdrawal manager; `assetsCommitted[stETH]` increases.
2. Admin calls `LRTConfig.setContract(LRT_WITHDRAW_MANAGER, newWithdrawalManager)`.
3. Operator calls `oldWithdrawalManager.unlockQueue(stETH, ...)`. Internally this calls `unstakingVault.redeem(stETH, amount)`.
4. `LRTUnstakingVault.onlyLRTWithdrawalManager` evaluates `lrtConfig.withdrawManager()` → returns `newWithdrawalManager`. `msg.sender` is `oldWithdrawalManager`. Reverts with `CallerNotLRTWithdrawalManager`.
5. Alice's 10 rsETH remains locked in the old withdrawal manager indefinitely. She cannot call `completeWithdrawal` (no assets were unlocked) and cannot recover her rsETH (no return path exists in `LRTWithdrawalManager`).
