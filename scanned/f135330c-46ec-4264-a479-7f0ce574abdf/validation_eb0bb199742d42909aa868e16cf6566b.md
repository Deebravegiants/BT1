### Title
Missing Zero-Amount Check After Integer Division Allows ETH/Token Deposits to Yield Zero rsETH — (`contracts/pools/RSETHPoolV3.sol`)

### Summary
The `deposit(string)` (ETH) and `deposit(address,uint256,string)` (token) functions in `RSETHPoolV3.sol` (and all pool variants) perform integer division to compute `rsETHAmount`. When the deposited amount is small enough that the division truncates to zero, the user's ETH or tokens are permanently consumed by the pool while zero `wrsETH` is minted. No guard checks that `rsETHAmount > 0` before accepting the deposit.

### Finding Description
`viewSwapRsETHAmountAndFee` computes the output amount via integer division:

**ETH path:**
```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```
**Token path:**
```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

Because Solidity truncates toward zero, any `amountAfterFee` small enough that `amountAfterFee * 1e18 < rsETHToETHrate` (ETH) or `amountAfterFee * tokenToETHRate < rsETHToETHrate` (token) produces `rsETHAmount = 0`.

The calling `deposit` functions perform no check on this result before consuming the user's funds:

```solidity
// ETH path — RSETHPoolV3.sol
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
wrsETH.mint(msg.sender, rsETHAmount);   // mints 0 — no revert
```

```solidity
// Token path — RSETHPoolV3.sol
IERC20(token).safeTransferFrom(msg.sender, address(this), amount);  // tokens taken
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
feeEarnedInToken[token] += fee;
wrsETH.mint(msg.sender, rsETHAmount);   // mints 0 — no revert
```

Unlike `LRTDepositPool.depositETH`, which accepts a `minRSETHAmountExpected` slippage guard, the pool contracts expose no such parameter, leaving users with no on-chain protection.

The same root cause is present across every pool variant: [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

### Impact Explanation
A depositor who sends a dust amount of ETH or a supported token receives zero `wrsETH` in return. Their funds are permanently retained by the pool (credited to the pool's ETH/token balance, not even to `feeEarnedInETH`/`feeEarnedInToken` unless `feeBps > 0`). The contract fails to deliver the promised exchange. This matches the **Low** impact tier: *"Contract fails to deliver promised returns, but doesn't lose value."*

For ETH: `rsETHToETHrate ≈ 1.05e18`, so any `amountAfterFee` of 1 wei satisfies `1 * 1e18 / 1.05e18 = 0`. For tokens with lower ETH-denominated rates (e.g., a 6-decimal stablecoin where `tokenToETHRate ≈ 4e14`), the zero-output threshold rises to `amountAfterFee < rsETHToETHrate / tokenToETHRate ≈ 2625 units`, meaning deposits below ~0.002625 USDC yield zero rsETH while the tokens are consumed. [1](#0-0) 

### Likelihood Explanation
Any unprivileged depositor can trigger this by sending a sufficiently small amount. It can occur accidentally (dust transactions, UI rounding) or deliberately. No special role or precondition is required beyond calling the public `deposit` functions. [3](#0-2) 

### Recommendation
1. Add a `require(rsETHAmount > 0, "Zero rsETH output")` check in both `deposit` overloads immediately after calling `viewSwapRsETHAmountAndFee`, before transferring tokens or minting.
2. Optionally, expose a `minRsETHAmountExpected` parameter (as `LRTDepositPool.depositETH` does) to give callers explicit slippage protection.

Reference: `LRTDepositPool._beforeDeposit` already enforces this pattern correctly: [5](#0-4) 

### Proof of Concept
**ETH path (RSETHPoolV3):**
1. `rsETHToETHrate = getRate()` returns `1.05e18` (rsETH worth 1.05 ETH).
2. Attacker/user calls `deposit{value: 1}("")` (1 wei).
3. `viewSwapRsETHAmountAndFee(1)`:
   - `fee = 1 * feeBps / 10_000 = 0` (feeBps < 10 000)
   - `amountAfterFee = 1`
   - `rsETHAmount = 1 * 1e18 / 1.05e18 = 0` ← integer truncation
4. `feeEarnedInETH += 0`
5. `wrsETH.mint(msg.sender, 0)` — no revert, 0 tokens minted.
6. User loses 1 wei; pool balance increases by 1 wei.

**Token path (RSETHPoolV3):**
1. Supported token has `tokenToETHRate = 4e14` (e.g., USDC-equivalent), `rsETHToETHrate = 1.05e18`.
2. User calls `deposit(token, 2000, "")` (2000 smallest units).
3. `viewSwapRsETHAmountAndFee(2000, token)`:
   - `fee = 0`
   - `rsETHAmount = 2000 * 4e14 / 1.05e18 = 8e17 / 1.05e18 = 0` ← truncation
4. `IERC20(token).safeTransferFrom(user, pool, 2000)` — tokens taken.
5. `wrsETH.mint(user, 0)` — no revert, 0 tokens minted.
6. User loses 2000 token units permanently. [4](#0-3) [2](#0-1)
