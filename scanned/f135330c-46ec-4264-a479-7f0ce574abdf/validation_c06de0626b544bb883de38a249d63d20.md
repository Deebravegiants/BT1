### Title
Stale Cross-Chain Rate Used in L2 Pool Deposits Enables Excess wrsETH Minting — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

---

### Summary

`CrossChainRateReceiver.getRate()` returns the stored `rate` variable with no staleness check. Every L2 deposit pool (`RSETHPoolNoWrapper`, `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPool`, `RSETHPoolV2ExternalBridge`, `RSETHPoolV2NBA`) calls this function to price deposits. When the cross-chain rate has not been updated recently, the stale (lower-than-actual) rate causes the pool to mint more wrsETH/rsETH than the deposited ETH is worth at the current rate, diluting existing holders and transferring their accrued yield to the depositor.

---

### Finding Description

The `CrossChainRateReceiver` base contract stores two fields: `rate` (the last received rsETH/ETH exchange rate) and `lastUpdated` (the timestamp of the last LayerZero update). [1](#0-0) 

The `getRate()` function unconditionally returns the stored `rate` without consulting `lastUpdated`: [2](#0-1) 

The rate is only refreshed when a LayerZero message arrives via `lzReceive()`, which is triggered by an off-chain call to `MultiChainRateProvider.updateRate()` — a permissionless but manually initiated and fee-paying operation: [3](#0-2) [4](#0-3) 

Every L2 deposit pool calls `getRate()` → `IOracle(rsETHOracle).getRate()` to compute the wrsETH amount to mint:

**`RSETHPoolNoWrapper`:** [5](#0-4) [6](#0-5) 

**`RSETHPoolV3ExternalBridge`:** [7](#0-6) 

**`RSETHPoolV3WithNativeChainBridge`:** [8](#0-7) 

The minting formula is:

```
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate
```

rsETH is a yield-bearing token whose rate monotonically increases over time as staking rewards accrue. If the L2 oracle rate is stale (i.e., lower than the current L1 rate), the denominator `rsETHToETHrate` is smaller than it should be, causing the pool to mint **more** wrsETH than the deposited ETH is worth at the true current rate.

There is also a second layer of caching: `RSETHMultiChainRateProvider.getLatestRate()` reads `ILRTOracle(rsETHPriceOracle).rsETHPrice()`, which is itself a stored value only updated when `updateRSETHPrice()` is called on L1: [9](#0-8) [10](#0-9) 

This means the L2 rate can lag the true rate by the sum of: (L1 oracle update delay) + (LayerZero propagation delay).

---

### Impact Explanation

rsETH accrues staking yield continuously. When the L2 rate is stale-low, a depositor receives more wrsETH than their ETH is worth at the current rate. The excess wrsETH represents a claim on ETH that belongs to existing holders (their accrued yield). When the depositor later redeems or sells their wrsETH, they extract value that should have belonged to prior holders.

**Impact: High — Theft of unclaimed yield from existing wrsETH holders.**

---

### Likelihood Explanation

- The rate update is not automatic; it requires a manual, fee-paying call to `updateRate()` on L1 and a successful LayerZero delivery.
- During periods of network congestion, LayerZero delays, or simply infrequent updates, the L2 rate can lag by hours or days.
- rsETH's rate increases continuously; even a few hours of lag creates a measurable discrepancy.
- Any unprivileged user can monitor the L1 `rsETHPrice` vs. the L2 `CrossChainRateReceiver.rate` and deposit when the gap is largest.
- No special access or front-running is required; the attacker simply waits for a stale window and deposits.

**Likelihood: Medium** — requires a stale rate window, which occurs regularly given the manual update mechanism.

---

### Recommendation

Add a staleness threshold check inside `CrossChainRateReceiver.getRate()`. Revert (or return a sentinel value) if `block.timestamp - lastUpdated` exceeds a configured maximum staleness period (e.g., 24 hours):

```solidity
uint256 public maxStaleness; // e.g., 24 hours

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

This mirrors the pattern already used in `ChainlinkOracleForRSETHPoolCollateral.getRate()`, which checks `answeredInRound < roundID` and `timestamp == 0` to guard against stale Chainlink data: [11](#0-10) 

---

### Proof of Concept

1. L1 `LRTOracle.rsETHPrice` is updated to `1.05e18` (rsETH = 1.05 ETH) via `updateRSETHPrice()`.
2. The `RSETHMultiChainRateProvider.updateRate()` call to propagate this to L2 is delayed (e.g., LayerZero congestion or no one has paid the fee yet).
3. The L2 `CrossChainRateReceiver.rate` still holds the old value `1.04e18`.
4. Attacker calls `RSETHPoolNoWrapper.deposit{value: 100 ether}("")`.
5. Pool computes: `rsETHAmount = 100e18 * 1e18 / 1.04e18 ≈ 96.15 wrsETH`.
6. At the true rate of `1.05e18`, the correct amount would be: `100e18 * 1e18 / 1.05e18 ≈ 95.24 wrsETH`.
7. Attacker receives `≈ 0.91 wrsETH` excess — worth `≈ 0.91 * 1.05 ≈ 0.955 ETH` — extracted from existing holders' yield.
8. The attacker repeats this for every stale window, scaling the attack with deposit size. [2](#0-1) [12](#0-11) [6](#0-5)

    function updateRate() external payable nonReentrant {
        uint256 latestRate = getLatestRate();

        rate = latestRate;

        lastUpdated = block.timestamp;

        bytes memory _payload = abi.encode(latestRate);

        uint256 rateReceiversLength = rateReceivers.length;

        for (uint256 i; i < rateReceiversLength;) {
            uint16 dstChainId = uint16(rateReceivers[i]._chainId);

            bytes memory remoteAndLocalAddresses = abi.encodePacked(rateReceivers[i]._contract, address(this));

            (uint256 estimatedFee,) = ILayerZeroEndpoint(layerZeroEndpoint)
                .estimateFees(dstChainId, address(this), _payload, false, bytes(""));

            ILayerZeroEndpoint(layerZeroEndpoint).send{ value: estimatedFee }(
                dstChainId, remoteAndLocalAddresses, _payload, payable(msg.sender), address(0x0), bytes("")
            );

            unchecked {
                ++i;
            }
        }

        emit RateUpdated(rate);
    }
