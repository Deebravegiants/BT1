### Title
`FeeReceiver.sendFunds()` Missing Access Control Allows Anyone to Force MEV Reward Distribution - (File: contracts/FeeReceiver.sol)

### Summary
`FeeReceiver` accumulates MEV and execution-layer rewards via its `receive()` fallback. The only ETH-out path, `sendFunds()`, carries **no access-control modifier**, so any unprivileged external caller can flush the entire ETH balance into `LRTDepositPool` at will. The protocol loses all ability to control the timing and batching of reward distribution.

### Finding Description
`FeeReceiver` is the designated recipient of validator MEV and execution-layer rewards for the LRT-rsETH protocol. It inherits `AccessControlUpgradeable` and correctly gates its administrative setter (`setDepositPool`) behind `onlyRole(LRTConstants.MANAGER)`. However, the sole ETH-distribution function `sendFunds()` is declared `external` with no role check:

```solidity
/// @dev send all rewards to deposit pool
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
``` [1](#0-0) 

Any EOA or contract can call `sendFunds()` at any block, draining the entire accumulated ETH balance into `LRTDepositPool.receiveFromRewardReceiver()`. [2](#0-1) 

`receiveFromRewardReceiver()` in `LRTDepositPool` is itself an unrestricted `external payable` stub with no `whenNotPaused` guard, so the push succeeds regardless of the deposit pool's pause state. [3](#0-2) 

The contract stores `_legacyProtocolTreasury` and `_legacyProtocolFeePercentInBPS` (set during `initialize`) but `sendFunds()` ignores them entirely, sending 100 % of rewards to the deposit pool with no fee split. [4](#0-3) 

### Impact Explanation
The protocol is designed to control when accumulated MEV rewards are swept into the deposit pool (for gas batching, accounting, and price-impact management). Because `sendFunds()` is permissionless, any caller can:

1. Force an immediate, unscheduled reward flush, preventing the operator from batching multiple reward epochs into a single transaction.
2. Trigger the flush at a moment that maximises rsETH price impact (e.g., just before a large withdrawal is unlocked), disrupting fair accounting for withdrawers.
3. Repeatedly call the function (no-op when balance is zero, but callable at any time) to interfere with off-chain monitoring that relies on the `MevRewardsAddedToTVL` event being emitted only by authorised parties.

The ETH is not stolen — it reaches its intended destination — but the protocol **fails to deliver the promised controlled distribution behaviour**. This maps to **Low: Contract fails to deliver promised returns, but doesn't lose value**.

### Likelihood Explanation
The entry point is a plain `external` function on a live, deployed contract. No special role, token balance, or precondition is required. Any Ethereum account can call it at zero cost beyond gas. Likelihood is **High**.

### Recommendation
Add the `onlyRole(LRTConstants.MANAGER)` modifier (or an equivalent operator role) to `sendFunds()`:

```solidity
function sendFunds() external onlyRole(LRTConstants.MANAGER) {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
``` [1](#0-0) 

This mirrors the access-control pattern already applied to `setDepositPool`. [5](#0-4) 

### Proof of Concept
1. Observe that `FeeReceiver` accumulates ETH from validator MEV rewards via `receive()`.
2. Call `FeeReceiver.sendFunds()` from any EOA (no role required).
3. The entire ETH balance is immediately transferred to `LRTDepositPool` via `receiveFromRewardReceiver{value: balance}()`.
4. The `MevRewardsAddedToTVL` event is emitted, and the operator's intended batching window is bypassed.
5. Repeat at any block to prevent any accumulation of rewards in the contract.
