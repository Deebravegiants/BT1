### Title
Missing Minimum Output Protection in L2 Pool `deposit` Functions Allows Stale-Rate Execution - (File: contracts/pools/RSETHPoolV3.sol)

---

### Summary

All L2 pool `deposit` functions across `RSETHPool`, `RSETHPoolV2`, `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, and `RSETHPoolV3WithNativeChainBridge` lack both a `deadline` parameter and a `minRSETHAmountExpected` (minimum output) parameter. The output amount is computed at execution time from a cross-chain oracle rate that can change between transaction submission and execution. Unlike `LRTDepositPool.depositETH`, which enforces a caller-supplied minimum output, the L2 pools provide no such protection, meaning a pending transaction can silently mint fewer wrsETH than the user expected.

---

### Finding Description

Every L2 pool `deposit` function computes the wrsETH output by calling `viewSwapRsETHAmountAndFee`, which reads the live oracle rate via `getRate()`:

```solidity
// RSETHPoolV3.sol
function deposit(string memory referralId) external payable nonReentrant whenNotPaused limitDailyMint(msg.value, ETH_IDENTIFIER) {
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    wrsETH.mint(msg.sender, rsETHAmount);
}
``` [1](#0-0) 

The rate is fetched from the oracle at execution time with no floor check:

```solidity
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    uint256 rsETHToETHrate = getRate();
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
``` [2](#0-1) 

The same pattern is present in every pool variant: [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

The oracle rate is a cross-chain value pushed from L1 by the `CrossChainRateProvider`. It is monotonically increasing as staking rewards accrue and is updated on a schedule independent of user transactions. There is no on-chain guarantee that the rate seen at submission time equals the rate at execution time.

By contrast, `LRTDepositPool.depositETH` on L1 accepts and enforces a `minRSETHAmountExpected` parameter:

```solidity
function depositETH(uint256 minRSETHAmountExpected, string calldata referralId) external payable {
    uint256 rsethAmountToMint = _beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, minRSETHAmountExpected);
``` [7](#0-6) 

The `_beforeDeposit` check enforces the minimum:

```solidity
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [8](#0-7) 

This protection is entirely absent from all L2 pool deposit paths.

---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A user who submits a deposit transaction expecting `X` wrsETH will receive fewer wrsETH if the oracle rate is updated upward before their transaction is included. Because rsETH is monotonically appreciating, a higher rate means fewer wrsETH minted per ETH deposited. The user's ETH value is not destroyed — they receive fewer but proportionally more valuable wrsETH — however the contract fails to deliver the output amount the user expected and planned around at submission time. There is no mechanism for the user to express or enforce a minimum acceptable output.

---

### Likelihood Explanation

**Low.** The oracle rate is updated by the protocol's cross-chain rate provider on a periodic schedule, not by arbitrary external actors. MEV bots cannot directly manipulate the oracle rate. The scenario requires a transaction to remain pending across an oracle rate update, which is plausible on L2s during gas price spikes or network congestion. The rate update frequency and magnitude determine how often this materialises in practice.

---

### Recommendation

Add a `minRSETHAmountExpected` parameter to all `deposit` functions in every L2 pool contract (`RSETHPool`, `RSETHPoolV2`, `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`), mirroring the protection already present in `LRTDepositPool.depositETH`. Optionally, also add a `deadline` parameter to allow users to bound how long their transaction may remain pending.

---

### Proof of Concept

1. User calls `RSETHPoolV3.deposit{value: 1 ether}("ref")` when `getRate()` returns `1.05e18` (rsETH worth 1.05 ETH), expecting `≈0.952 wrsETH`.
2. Transaction sits in the L2 mempool during a gas price spike.
3. The cross-chain rate provider pushes a new rate: `getRate()` now returns `1.10e18`.
4. The transaction is eventually included. `viewSwapRsETHAmountAndFee` computes `rsETHAmount = 1e18 * 1e18 / 1.10e18 ≈ 0.909e18`.
5. User receives `≈0.909 wrsETH` instead of the `≈0.952 wrsETH` they expected — a ~4.5% shortfall — with no revert and no recourse, because no minimum output was specified.
