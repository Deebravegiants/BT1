### Title
Token Donation Inflates `getTotalAssetDeposits()` via Raw `balanceOf()`, Enabling Deposit-Limit DoS — (File: `contracts/LRTDepositPool.sol`)

---

### Summary

`getAssetDistributionData()` in `LRTDepositPool.sol` measures protocol-held LST assets using raw `IERC20(asset).balanceOf(address(this))` and `IERC20(asset).balanceOf(nodeDelegatorQueue[i])`. Because any address can receive ERC-20 transfers, an unprivileged attacker can donate a small amount of any supported LST directly to the deposit pool or any NodeDelegator, inflating `getTotalAssetDeposits()`. This inflated value is consumed by `_checkIfDepositAmountExceedesCurrentLimit()`, which then permanently returns `true` for that asset, reverting every subsequent deposit with `MaximumDepositLimitReached`.

---

### Finding Description

`getAssetDistributionData()` computes the total protocol balance for a supported LST as:

```solidity
assetLyingInDepositPool = IERC20(asset).balanceOf(address(this));   // line 444
// ...
assetLyingInNDCs += IERC20(asset).balanceOf(nodeDelegatorQueue[i]); // line 448
``` [1](#0-0) 

These raw balances are summed in `getTotalAssetDeposits()`: [2](#0-1) 

That total is then used in `_checkIfDepositAmountExceedesCurrentLimit()`:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    // ...
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [3](#0-2) 

Which is called inside `_beforeDeposit()`, the guard for every `depositAsset()` and `depositETH()` call: [4](#0-3) 

Because `balanceOf()` reflects any tokens held at those addresses — including unsolicited transfers — an attacker who sends even 1 wei of a supported LST directly to `LRTDepositPool` or any registered NodeDelegator can push `totalAssetDeposits` above `depositLimitByAsset`, causing the check to return `true` indefinitely.

The same inflated value also feeds `LRTOracle._getTotalEthInProtocol()`: [5](#0-4) 

which drives `_updateRsETHPrice()`. A sufficiently large donation can push `newRsETHPrice` above `highestRsethPrice * (1 + pricePercentageLimit)`, causing `updateRSETHPrice()` to revert for any non-manager caller: [6](#0-5) 

Additionally, `getAvailableAssetAmount()` in `LRTWithdrawalManager` also consumes `getTotalAssetDeposits()`: [7](#0-6) 

and is used as the gate in `initiateWithdrawal()`: [8](#0-7) 

An inflated value here allows users to commit more `assetsCommitted` than the protocol actually holds, desynchronising the withdrawal queue from real vault liquidity.

---

### Impact Explanation

**Medium — Temporary freezing of funds.**

Once an attacker donates enough of a supported LST to push `totalAssetDeposits` above the configured deposit limit, every call to `depositAsset()` for that token reverts with `MaximumDepositLimitReached`. Users' ETH/LST cannot enter the protocol until an admin raises the deposit cap. Because the donated tokens cannot be removed by the protocol (there is no sweep function for unsolicited ERC-20 balances in `LRTDepositPool`), the DoS persists until governance action. The attacker can repeat the donation after each admin adjustment, making the freeze effectively continuous at low cost.

---

### Likelihood Explanation

**High.** Any ERC-20 holder of a supported LST (stETH, ETHx, sfrxETH, etc.) can execute this attack permissionlessly with a single `transfer()` call. No special role, flash loan, or complex setup is required. The cost is the donated amount, which can be as small as 1 wei above the remaining deposit headroom.

---

### Recommendation

Replace raw `balanceOf()` reads with an internal accounting variable that is incremented only on protocol-controlled inflows (deposits, transfers from NDCs) and decremented on outflows. Alternatively, track a `depositedBalance` mapping that is updated exclusively inside `depositAsset()`, `transferAssetToNodeDelegator()`, and `receiveFromNodeDelegator()`, and use that instead of `balanceOf()` in `getAssetDistributionData()`. This mirrors the fix pattern used in ERC-4626 vaults that maintain a separate `totalAssets` counter rather than relying on `balanceOf(address(this))`.

---

### Proof of Concept

1. Suppose `depositLimitByAsset(stETH) = 10_000e18` and current `getTotalAssetDeposits(stETH) = 9_999e18`.
2. Attacker calls `stETH.transfer(address(lrtDepositPool), 2e18)`.
3. `getAssetDistributionData()` now returns `assetLyingInDepositPool = 2e18` extra, making `getTotalAssetDeposits(stETH) = 10_001e18`.
4. Any user calling `depositAsset(stETH, amount, ...)` hits `_checkIfDepositAmountExceedesCurrentLimit()` → `10_001e18 + amount > 10_000e18` → `true` → revert `MaximumDepositLimitReached`.
5. All stETH deposits are blocked. The donated 2 stETH remain in the pool with no protocol mechanism to eject them.
