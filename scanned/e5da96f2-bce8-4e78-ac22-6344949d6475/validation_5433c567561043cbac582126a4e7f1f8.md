### Title
Missing `amount` in ETH Deposit Limit Check Allows Deposits Beyond Configured Cap - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric check: for LST assets it correctly tests `totalAssetDeposits + amount > depositLimit`, but for ETH it omits `amount` and only tests `totalAssetDeposits > depositLimit`. Any ETH deposit is accepted as long as the running total has not already crossed the limit, regardless of how large the incoming deposit is.

### Finding Description
`_checkIfDepositAmountExceedesCurrentLimit` is the sole guard that enforces the per-asset deposit cap before minting rsETH:

```solidity
// contracts/LRTDepositPool.sol  lines 676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // ← `amount` absent
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct for LSTs
}
``` [1](#0-0) 

For ETH the function returns `true` (i.e., "limit exceeded, revert") only when `totalAssetDeposits` has **already** surpassed the cap. It never accounts for the size of the incoming deposit. Consequently, a depositor can push the protocol's total ETH holdings arbitrarily above `depositLimitByAsset[ETH]` in a single transaction, as long as the running total was at or below the cap before the call.

The `depositETH` entry point is fully public and guarded only by `whenNotPaused`:

```solidity
// contracts/LRTDepositPool.sol  lines 76-93
function depositETH(uint256 minRSETHAmountExpected, string calldata referralId)
    external payable nonReentrant whenNotPaused onlySupportedAsset(LRTConstants.ETH_TOKEN)
{
    uint256 rsethAmountToMint = _beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, minRSETHAmountExpected);
    _mintRsETH(rsethAmountToMint);
    ...
}
``` [2](#0-1) 

`_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit` and reverts on `true`, but because the ETH branch never adds `amount`, the check is always `false` whenever `totalAssetDeposits <= depositLimit`, no matter how large `msg.value` is. [3](#0-2) 

The deposit limit is stored and read from `LRTConfig.depositLimitByAsset`: [4](#0-3) 

### Impact Explanation
The ETH deposit cap — a protocol-level safety control — is not enforced for any individual deposit. An unprivileged depositor can mint rsETH for an arbitrarily large ETH amount in one call, pushing total ETH holdings well beyond the configured limit. This violates the protocol's own invariant and can cause downstream operational issues (e.g., excess ETH sitting undeployed, node-delegator capacity exceeded, or oracle/TVL accounting skewed). Impact: **Low — contract fails to deliver promised returns (deposit limit not enforced)**, with potential escalation to Medium if the excess deposits disrupt node-delegator operations or TVL accounting.

### Likelihood Explanation
The entry path is a single public `payable` function callable by any Ethereum address with no preconditions beyond the protocol not being paused. No special role, front-running, or external dependency is required. Any depositor who sends ETH while `totalAssetDeposits <= depositLimit` bypasses the cap automatically.

### Recommendation
Add `amount` to the ETH branch of `_checkIfDepositAmountExceedesCurrentLimit`, mirroring the LST branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
-   if (asset == LRTConstants.ETH_TOKEN) {
-       return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
-   }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept
Assume `depositLimitByAsset[ETH] = 1000 ether` and `totalAssetDeposits(ETH) = 999 ether`.

1. Attacker calls `depositETH{value: 500 ether}(0, "")`.
2. `_checkIfDepositAmountExceedesCurrentLimit(ETH, 500 ether)` evaluates `999 ether > 1000 ether` → `false`.
3. The check does **not** revert; `_mintRsETH` mints rsETH for 500 ETH.
4. `getTotalAssetDeposits(ETH)` is now `1499 ether`, 49.9% above the configured cap.
5. For comparison, a 500-token LST deposit under the same conditions would evaluate `999 + 500 > 1000` → `true` and revert with `MaximumDepositLimitReached`.
