### Title
`RSETH.burnFrom` Gated by `whenNotPaused` and Transfer Block Can Freeze Pending Withdrawal Queue — (`contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTWithdrawalManager.unlockQueue()` calls `RSETH.burnFrom(address(this), rsETHBurned)` to burn rsETH held by the withdrawal manager. `RSETH.burnFrom` is gated by `whenNotPaused` and internally calls `_enforceNotBlocked(account)`. If rsETH is paused, or if `LRTWithdrawalManager` is placed on the rsETH transfer-block list, `unlockQueue` reverts unconditionally. Because `completeWithdrawal` requires `nextLockedNonce` to have advanced — which only happens inside `unlockQueue` — all pending withdrawal requests become permanently locked for the duration of the block/pause, trapping user rsETH inside `LRTWithdrawalManager` and the corresponding underlying assets inside `LRTUnstakingVault`.

---

### Finding Description

**Step 1 — User commits rsETH to the withdrawal queue.**

`initiateWithdrawal` pulls rsETH from the user into `LRTWithdrawalManager`: [1](#0-0) 

At this point the user no longer holds their rsETH; it sits in the withdrawal manager awaiting the unlock step.

**Step 2 — `unlockQueue` must burn that rsETH to advance the queue.** [2](#0-1) 

**Step 3 — `RSETH.burnFrom` is gated by two independent blockers.** [3](#0-2) 

- **`whenNotPaused`**: any holder of `PAUSER_ROLE` can pause rsETH, causing every `burnFrom` call to revert.
- **`_enforceNotBlocked(account)`**: `blockUserTransfers` can place any address — including `LRTWithdrawalManager` — on a 24-hour (re-applicable) transfer block. When `account` is blocked, `_enforceNotBlocked` reverts with `TransfersBlocked`. [4](#0-3) 

**Step 4 — `completeWithdrawal` is permanently gated behind `nextLockedNonce`.** [5](#0-4) 

`nextLockedNonce[asset]` only advances inside `_unlockWithdrawalRequests`, which is called exclusively from `unlockQueue`. If `unlockQueue` cannot execute, `nextLockedNonce` never moves, and every call to `completeWithdrawal` reverts with `WithdrawalLocked`.

---

### Impact Explanation

All users who have already called `initiateWithdrawal` have their rsETH held inside `LRTWithdrawalManager` and their corresponding underlying assets (ETH/LSTs) reserved in `LRTUnstakingVault`. For the entire duration that rsETH is paused or `LRTWithdrawalManager` is blocked:

- `unlockQueue` reverts → queue cannot advance.
- `completeWithdrawal` reverts → users cannot receive their assets.
- User rsETH is stuck in `LRTWithdrawalManager`; underlying assets are stuck in `LRTUnstakingVault`.

This constitutes **temporary freezing of funds** (Medium severity per the allowed impact scope). If the block is continuously re-applied (each application resets the 24-hour window), the freeze can be extended indefinitely.

---

### Likelihood Explanation

- Pausing rsETH is a routine emergency action available to any `PAUSER_ROLE` holder. A security incident, regulatory event, or oracle anomaly could trigger it.
- `blockUserTransfers` accepts an arbitrary address array; `LRTWithdrawalManager` is not guaranteed to be in `isPermanentlyExempt`. A manager acting on a compliance list could inadvertently include the withdrawal manager address, or do so deliberately.
- The scenario requires no attacker — it is triggered by normal, intended admin operations that have an unintended side-effect on the withdrawal lifecycle.

---

### Recommendation

1. **Exempt `LRTWithdrawalManager` permanently in rsETH.** Call `addPermanentExemptions([address(LRTWithdrawalManager)])` so it can never be placed on the transfer-block list.
2. **Decouple `unlockQueue` from the rsETH pause state.** Give `LRTWithdrawalManager` a special role (e.g., `BURNER_ROLE` with a bypass modifier) that allows burning even when rsETH is paused, analogous to how the original report recommends giving the liquidator a special burn role exempt from pause/blacklist.
3. **Alternatively**, split the unlock step into two phases: (a) advance `nextLockedNonce` and record asset amounts (no burn), and (b) burn rsETH separately, so that a burn failure does not block queue advancement.

---

### Proof of Concept

1. Alice calls `LRTWithdrawalManager.initiateWithdrawal(stETH, 10e18, "")`. Her 10 rsETH is transferred to `LRTWithdrawalManager`.
2. Admin calls `RSETH.pause()` (legitimate emergency response).
3. Operator calls `LRTWithdrawalManager.unlockQueue(stETH, ...)`. The call reaches line 305 and reverts because `RSETH.burnFrom` hits `whenNotPaused`.
4. Alice calls `LRTWithdrawalManager.completeWithdrawal(stETH, "")`. The call reverts at line 707 with `WithdrawalLocked` because `nextLockedNonce[stETH]` was never advanced.
5. Alice's 10 rsETH remains stuck in `LRTWithdrawalManager`; her stETH remains stuck in `LRTUnstakingVault`. No recovery path exists until rsETH is unpaused.

```text
    function _enforceNotBlocked(address account) internal {
        // Addresses that are permanently exempt can never be blocked
        if (isPermanentlyExempt[account]) return;

        // Check if the account has an active transfer block
        uint256 blockedUntil = transfersBlockedUntil[account];
        if (blockedUntil == 0) return;

        if (block.timestamp < blockedUntil) revert TransfersBlocked(account, blockedUntil);

        // Auto-clean up expired block
        delete transfersBlockedUntil[account];
    }
```
