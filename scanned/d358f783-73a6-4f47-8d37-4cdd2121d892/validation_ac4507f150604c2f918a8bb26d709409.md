### Title
Incorrect Reward Duration Extension in `notifyRewardAmount` Always Resets `finishAt` to Full Duration - (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary
`KernelDepositPool.notifyRewardAmount` unconditionally sets `finishAt = block.timestamp + duration` on every call, even when an active reward period is still running. This silently extends the reward window by a full `duration` from the current timestamp rather than preserving the remaining time, causing rewards to be diluted over a longer period than intended and stakers to receive a lower reward rate than expected.

### Finding Description
In `KernelDepositPool.notifyRewardAmount`, when a reward period is still active (`block.timestamp < finishAt`), the function correctly rolls the leftover rewards into the new rate:

```solidity
uint256 remaining = (finishAt - block.timestamp) * rewardRate;
rewardRate = (receivedAmount + remaining) / duration;
```

However, regardless of whether the period is active or expired, line 588 unconditionally resets the finish timestamp:

```solidity
finishAt = block.timestamp + duration;   // always a full new duration
```

If the admin calls `notifyRewardAmount` halfway through a 30-day period (15 days remaining), the period is silently extended to 30 more days from now instead of the 15 days that were left. The leftover rewards are folded into the rate, but the denominator (`duration`) is always the full period length, so the effective rate is lower than if the remaining window had been preserved. [1](#0-0) 

### Impact Explanation
Stakers receive a lower reward rate than the protocol intends to distribute. Every top-up during an active period silently extends the distribution window by a full `duration`, diluting the per-second reward rate. Rewards are not lost (they remain in the contract), but they are spread over a longer horizon than promised. This matches the **Low** impact class: *Contract fails to deliver promised returns, but doesn't lose value.*

### Likelihood Explanation
The admin is expected to call `notifyRewardAmount` periodically to top up rewards. Any call made before the current `finishAt` triggers the extension. This is a routine operational pattern, so the condition is likely to be hit in normal protocol operation.

### Recommendation
When the reward period is still active, preserve the remaining time rather than resetting to a full new `duration`:

```solidity
if (block.timestamp >= finishAt) {
    rewardRate = receivedAmount / duration;
    finishAt = block.timestamp + duration;   // new period only when expired
} else {
    uint256 remaining = (finishAt - block.timestamp) * rewardRate;
    rewardRate = (receivedAmount + remaining) / (finishAt - block.timestamp);
    // finishAt is NOT changed — keep the existing end time
}
```

Alternatively, if a full reset is desired, document it explicitly and ensure the rate calculation uses the correct denominator.

### Proof of Concept

1. Admin calls `notifyRewardAmount(1000e18)` at `t=0`; `duration = 30 days`.
   - `rewardRate = 1000e18 / 30 days`
   - `finishAt = t + 30 days`
2. At `t = 15 days` (halfway), admin calls `notifyRewardAmount(500e18)`.
   - `remaining = 15 days * rewardRate = 500e18`
   - `rewardRate = (500e18 + 500e18) / 30 days` → same rate as step 1
   - `finishAt = t + 30 days = 45 days` ← extended by 30 days, not 15
3. Stakers now wait 30 more days (total 45 days) to receive rewards that should have finished in 30 days. The effective reward rate is halved relative to what a 15-day remaining window would have produced. [2](#0-1)

    function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
        if (_amount == 0) revert AmountZero();

        // Prevent starting a reward period when no tokens are staked to avoid unallocated rewards
        if (totalKernelStaked == 0) revert NoStakedTokens();

        // Transfer reward tokens into the contract
        uint256 balanceBefore = rewardsToken.balanceOf(address(this));
        rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
        uint256 balanceAfter = rewardsToken.balanceOf(address(this));
        // Calculate the actual amount of tokens received in case of a transfer fee (tax)
        uint256 receivedAmount = balanceAfter - balanceBefore;

        if (block.timestamp >= finishAt) {
            rewardRate = receivedAmount / duration;
        } else {
            uint256 remaining = (finishAt - block.timestamp) * rewardRate;
            rewardRate = (receivedAmount + remaining) / duration;
        }

        if (rewardRate == 0) revert RewardRateZero();

        finishAt = block.timestamp + duration;
        updatedAt = block.timestamp;

        emit NotifyRewardAmount(receivedAmount, finishAt);
    }
