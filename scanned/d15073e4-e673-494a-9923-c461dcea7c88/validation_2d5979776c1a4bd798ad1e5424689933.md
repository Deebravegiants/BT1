### Title
ETH Deposit Limit Bypass Due to Missing `amount` in Limit Check - (File: contracts/LRTDepositPool.sol)

### Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric check for ETH vs ERC20 assets. For ERC20 assets the incoming deposit amount is included in the comparison (`totalAssetDeposits + amount > limit`), but for ETH the incoming `amount` is silently dropped (`totalAssetDeposits > limit`). Any unprivileged depositor can therefore deposit an arbitrarily large ETH amount in a single call as long as the current total has not yet crossed the limit, bypassing the protocol's ETH deposit cap entirely.

### Finding Description
In `LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit`, the ETH branch reads:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
}
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
```

The `amount` parameter — which equals `msg.value` passed from `depositETH` — is never added to `totalAssetDeposits` in the ETH path. The check only asks "has the limit already been exceeded?" not "would this deposit exceed the limit?". For ERC20 assets the correct form (`totalAssetDeposits + amount`) is used.

Call chain:
1. `depositETH(minRSETHAmountExpected, referralId)` — `msg.value` is the deposit
2. → `_beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, minRSETHAmountExpected)`
3. → `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, msg.value)`
4. → returns `totalAssetDeposits > depositLimit` — `msg.value` is ignored
5. → `_mintRsETH(rsethAmountToMint)` — rsETH minted for the full `msg.value`

### Impact Explanation
The ETH deposit cap (`depositLimitByAsset[ETH_TOKEN]`) is rendered ineffective. A depositor can send any amount of ETH in a single transaction while the running total is still at or below the limit, receiving rsETH for the full amount. This allows the protocol to accept ETH far beyond its intended cap, potentially causing protocol insolvency or diluting existing rsETH holders if the cap was set for risk-management or backing-ratio reasons.

**Impact: Low → Medium** — Contract fails to deliver its promised deposit-limit guarantee; depending on why the cap exists, this can escalate to temporary fund freezing or insolvency.

### Likelihood Explanation
The entry point is the public, payable `depositETH` function — no special role or privilege is required. Any depositor who observes that `totalAssetDeposits < depositLimit` can immediately exploit this by sending a large `msg.value`. The condition is trivially detectable on-chain.

### Recommendation
Add `amount` to the ETH branch, mirroring the ERC20 branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept
Assume `depositLimitByAsset[ETH_TOKEN] = 1000 ether` and `getTotalAssetDeposits(ETH_TOKEN) = 999 ether`.

1. Attacker calls `depositETH{value: 500 ether}(0, "")`.
2. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500 ether)` evaluates `999 ether > 1000 ether` → `false` → no revert.
3. rsETH is minted for 500 ETH; total ETH in protocol becomes 1499 ETH — 49.9% above the cap.
4. The same call with any ERC20 would have evaluated `999 + 500 > 1000` → `true` → reverted.

The asymmetry is the root cause, directly analogous to the zkSync issue where the limit was checked against one parameter while the actual minted value used another. [1](#0-0) [2](#0-1) [3](#0-2)
