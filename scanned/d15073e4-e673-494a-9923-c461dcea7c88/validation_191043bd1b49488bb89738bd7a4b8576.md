### Title
Vesting Period Elapses During Pause Without Extension Mechanism — (File: `contracts/KERNEL/KernelTop100MerkleDistributor.sol`)

---

### Summary

`KernelTop100MerkleDistributor` uses a fixed 30-day vesting window (`VESTING_DURATION`). Both `claim` and `claimAndStake` are gated by `whenNotPaused`, so users cannot claim during a pause. However, `block.timestamp` continues to advance during the pause, and the vesting deadline (`vestingStartTimestamp + VESTING_DURATION`) is never extended. Critically, `setVestingStartTimestamp` permanently reverts with `VestingAlreadyStarted` once vesting has begun, so the owner has no on-chain mechanism to compensate for the lost time. This is the direct analog of the `hasSaleEnded` bug: a time-based boundary advances through a pause period without any extension, and the state it governs (claimable vested amount) changes irreversibly while users are locked out.

---

### Finding Description

`_getUnclaimedVestedAmount` computes the claimable amount using a live `block.timestamp`:

```solidity
uint256 vestingEndTime = vestingStartTimestamp + VESTING_DURATION;   // fixed, never extended
uint256 currentTime = block.timestamp;
if (currentTime > vestingEndTime) {
    currentTime = vestingEndTime;                                     // caps at end, not at pause start
}
uint256 totalElapsedTime = currentTime - vestingStartTimestamp;
uint256 totalVestedAmount = (userTotalClaimableAmount * totalElapsedTime) / VESTING_DURATION;
``` [1](#0-0) 

During a pause, `block.timestamp` advances normally. The vesting schedule therefore elapses — potentially to completion — while every user-facing claim path is blocked:

```solidity
function claim(...)        external nonReentrant whenNotPaused { ... }
function claimAndStake(...) external nonReentrant whenNotPaused { ... }
``` [2](#0-1) [3](#0-2) 

The only administrative escape valve, `setVestingStartTimestamp`, is permanently disabled once vesting has started:

```solidity
if (vestingStartTimestamp > 0 && block.timestamp >= vestingStartTimestamp) {
    revert VestingAlreadyStarted();
}
``` [4](#0-3) 

There is therefore no on-chain path to extend the vesting window to account for the pause duration.

---

### Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

When the contract is paused mid-vesting:

1. Tokens that vest during the pause period cannot be claimed and therefore cannot be staked in `KernelDepositPool` via `claimAndStake`.
2. `KernelDepositPool` distributes staking rewards proportional to staked balance over time (`rewardRate * elapsed / totalKernelStaked`). Every second a user's tokens sit unclaimed and unstaked is staking yield that is permanently foregone — it accrues to other stakers and is never recoverable.
3. If the pause spans the entire remaining vesting window, the vesting schedule expires while users are locked out. After unpause, all tokens become claimable at once (the cap at `vestingEndTime` ensures full vesting), but the entire period of gradual staking yield is permanently lost.

The KERNEL tokens themselves are not permanently frozen (they remain claimable after unpause), but the staking yield that would have accrued during the pause is irrecoverably lost — matching the "Permanent freezing of unclaimed yield" impact tier.

---

### Likelihood Explanation

**Medium.** The owner can pause the contract at any time for legitimate operational reasons (e.g., a discovered vulnerability, an oracle issue, a bridge incident). The vesting window is only 30 days. A pause of even a few days during peak vesting represents a material fraction of the total yield window. No attacker action is required; the loss is a structural consequence of any legitimate pause during the vesting period.

---

### Recommendation

Track cumulative pause duration and extend the vesting end time on `unpause`:

```solidity
uint256 public totalPausedDuration;
uint256 private _pauseStartTime;

function _pause() internal override {
    _pauseStartTime = block.timestamp;
    super._pause();
}

function _unpause() internal override {
    totalPausedDuration += block.timestamp - _pauseStartTime;
    super._unpause();
}
```

Then in `_getUnclaimedVestedAmount`, replace the fixed `vestingEndTime` with:

```solidity
uint256 vestingEndTime = vestingStartTimestamp + VESTING_DURATION + totalPausedDuration;
```

This mirrors the fix applied to the `SimpleTokenSale` reference bug: the end boundary is extended by the exact duration of the pause so that users receive the full vesting window they were promised.

---

### Proof of Concept

1. Owner deploys `KernelTop100MerkleDistributor` with `vestingStartTimestamp = T`.
2. Vesting begins at `T`. Users start calling `claimAndStake` to gradually stake vested KERNEL and earn `KernelDepositPool` rewards.
3. At `T + 10 days`, owner calls `pause()` for a legitimate security reason.
4. `block.timestamp` advances. At `T + 30 days`, `vestingEndTime` is reached — the entire vesting schedule has elapsed while the contract is paused.
5. Owner calls `setVestingStartTimestamp(T + 20 days)` to try to extend the window → **reverts with `VestingAlreadyStarted`** because `block.timestamp >= vestingStartTimestamp`. [4](#0-3) 
6. Owner calls `unpause()` at `T + 35 days`.
7. Users call `claim` / `claimAndStake`. `_getUnclaimedVestedAmount` caps `currentTime` at `vestingEndTime = T + 30 days`, so all tokens are now fully vested and claimable in one transaction. [5](#0-4) 
8. **Result**: Users recover their principal KERNEL tokens, but 20 days of `KernelDepositPool` staking rewards (from `T+10` to `T+30`) are permanently lost — accrued to other stakers during the period when these users were locked out. [6](#0-5)
