### Title
Missing `eigenPod.recoverTokens()` Wrapper Permanently Freezes ERC20 Tokens Sent to EigenPod — (`contracts/NodeDelegator.sol`)

### Summary
`NodeDelegator` owns its `EigenPod` but exposes no function to call `IEigenPod.recoverTokens()`. Any ERC20 tokens that land at the EigenPod address — whether from a direct airdrop, an AVS reward mechanism that pushes tokens to the pod, or an accidental transfer — are permanently irrecoverable.

### Finding Description
`IEigenPod.recoverTokens(IERC20[] memory tokenList, uint256[] memory amountsToWithdraw, address recipient)` is gated to the pod owner only. The pod owner is `NodeDelegator`. `NodeDelegator` wraps many EigenPod calls (`startCheckpoint`, `verifyWithdrawalCredentials`, `verifyStaleBalance`) but provides no wrapper for `recoverTokens`. The `INodeDelegator` interface also omits this function. Because `NodeDelegator` is the only address that can call `eigenPod.recoverTokens()`, and it never does, any ERC20 balance at the EigenPod is frozen forever.

The normal EigenLayer reward path (`IRewardsCoordinator.processClaim`) is correctly wired: [2](#0-1) 

But this path only covers rewards distributed via the merkle tree. It does not cover ERC20 tokens transferred directly to the EigenPod address (e.g., direct AVS token airdrops, mistaken transfers, or future reward mechanisms that push tokens to the pod rather than through the coordinator).

### Impact Explanation
**Medium — Permanent freezing of unclaimed yield.**

Any ERC20 tokens at the EigenPod address are permanently stuck. The protocol's invariant that all yield earned by the protocol is claimable and reflected in rsETH value is violated for this token class. The tokens cannot be swept to `LRTDepositPool`, cannot be transferred back, and cannot be recovered by any role.

### Likelihood Explanation
Low-to-medium. The scenario requires ERC20 tokens to reach the EigenPod address outside the `IRewardsCoordinator` flow. This can happen via:
- Direct AVS token airdrops to the pod address
- Future AVS reward mechanisms that push tokens to the pod
- Accidental ERC20 transfers to the pod address

The gap is structural and permanent until an upgrade adds the wrapper.

### Recommendation
Add a permissioned wrapper in `NodeDelegator` (callable by `onlyLRTManager` or `onlyLRTOperator`):

```solidity
function recoverEigenPodTokens(
    IERC20[] calldata tokenList,
    uint256[] calldata amounts,
    address recipient
) external onlyLRTManager {
    eigenPod.recoverTokens(tokenList, amounts, recipient);
}
```

Also add the corresponding signature to `INodeDelegator`.
