### Title
Precision Loss in `rewardRate` Calculation Permanently Locks Reward Tokens - (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary
In `KernelDepositPool.notifyRewardAmount`, the `rewardRate` is computed via integer division `receivedAmount / duration`. Solidity truncates the remainder, so `rewardRate * duration < receivedAmount`. The difference â€” up to `duration - 1` wei per call â€” is permanently locked in the contract and can never be distributed to stakers.

### Finding Description
In `notifyRewardAmount` (lines 579â€“584):

```solidity
if (block.timestamp >= finishAt) {
    rewardRate = receivedAmount / duration;
} else {
    uint256 remaining = (finishAt - block.timestamp) * rewardRate;
    rewardRate = (receivedAmount + remaining) / duration;
}
```

Both branches divide by `duration` using integer division. The truncated remainder (`receivedAmount % duration` in the first branch; `(receivedAmount + remaining) % duration` in the second) is never tracked or recoverable. The contract holds these tokens but has no mechanism to distribute or recover them. The `rewardRate` is set once per call and the remainder is silently discarded.

### Impact Explanation
Every call to `notifyRewardAmount` permanently locks up to `duration - 1` wei of reward tokens. For a 7-day duration (`604800` seconds), up to `604799` wei is lost per call. Over repeated reward cycles this accumulates. The locked tokens satisfy the "Permanent freezing of unclaimed yield" criterion: stakers are entitled to these rewards but can never claim them.

**Impact: Medium â€” Permanent freezing of unclaimed yield.**

### Likelihood Explanation
This occurs on every legitimate invocation of `notifyRewardAmount`. No special conditions are required; the truncation is deterministic and unavoidable given the current formula. Likelihood is **High**.

### Recommendation
Track the undistributed remainder and either:
1. Carry it forward into the next reward period by adding it to `receivedAmount` in the next `notifyRewardAmount` call, or
2. Use a higher-precision intermediate (e.g., scale `receivedAmount` by a large factor before dividing, then scale `rewardRate` back when computing `rewardPerToken`).

A minimal fix for the new-period branch:
```solidity
rewardRate = receivedAmount / duration;
// carry forward remainder to avoid locking
uint256 leftover = receivedAmount - rewardRate * duration;
// store leftover and add to next notifyRewardAmount call
```
