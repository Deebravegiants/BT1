### Title
Two Supported Assets Mapped to the Same EigenLayer Strategy Causes Double-Counted TVL and Inflated rsETH Price — (`contracts/LRTConfig.sol`)

### Summary
`LRTConfig.updateAssetStrategy()` does not verify that the strategy address being assigned is not already mapped to a different supported asset. If two assets share the same EigenLayer strategy, `NodeDelegatorHelper.getAssetBalance()` returns the same strategy balance for both assets, causing `_getTotalEthInProtocol()` in `LRTOracle` to double-count that balance and inflate `rsETHPrice`, breaking the share/asset accounting for all depositors.

---

### Finding Description

`LRTConfig.updateAssetStrategy()` only checks that the new strategy is not identical to the asset's *current* strategy:

```solidity
if (assetStrategy[asset] == strategy) {
    revert ValueAlreadyInUse();
}
``` [1](#0-0) 

There is no check that `strategy` is not already stored in `assetStrategy[otherAsset]` for any other supported asset. An admin can therefore call `updateAssetStrategy(assetB, strategyX)` even when `assetStrategy[assetA] == strategyX`.

`NodeDelegatorHelper.getAssetBalance()` resolves the strategy purely from `lrtConfig.assetStrategy(asset)`:

```solidity
function getAssetBalance(ILRTConfig lrtConfig, address asset) internal view returns (uint256) {
    address strategy = lrtConfig.assetStrategy(asset);
    ...
    uint256 withdrawableShare = getWithdrawableShare(lrtConfig, IStrategy(strategy));
    return IStrategy(strategy).sharesToUnderlyingView(withdrawableShare);
}
``` [2](#0-1) 

When both `assetA` and `assetB` point to `strategyX`, `getAssetBalance(assetA)` and `getAssetBalance(assetB)` both return the full balance of `strategyX` for the NDC. `LRTDepositPool.getAssetDistributionData()` sums these independently per asset:

```solidity
assetStakedInEigenLayer += INodeDelegator(nodeDelegatorQueue[i]).getAssetBalance(asset);
``` [3](#0-2) 

`LRTOracle._getTotalEthInProtocol()` then iterates over all supported assets and accumulates `getTotalAssetDeposits(asset) * assetPrice` for each:

```solidity
for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
    address asset = supportedAssets[assetIdx];
    uint256 assetER = getAssetPrice(asset);
    uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
    totalETHInProtocol += totalAssetAmt.mulWad(assetER);
    ...
}
``` [4](#0-3) 

The strategy balance is counted once for `assetA` and again for `assetB`, inflating `totalETHInProtocol` and therefore `rsETHPrice`.

---

### Impact Explanation

An inflated `rsETHPrice` directly harms new depositors: `getRsETHAmountToMint = (amount * assetPrice) / rsETHPrice`. With an artificially high denominator, depositors receive fewer rsETH tokens than they are entitled to for the same deposit amount. Existing rsETH holders benefit at the expense of new entrants. This is a **share/asset mis-accounting** issue — the contract fails to deliver the promised rsETH amount to depositors.

**Impact: Low** — Contract fails to deliver promised returns to depositors; no direct fund theft, but depositors are systematically under-minted.

---

### Likelihood Explanation

`updateAssetStrategy()` is callable only by `DEFAULT_ADMIN_ROLE`. The misconfiguration requires an admin to assign the same EigenLayer strategy contract address to two different supported LST assets. While EigenLayer strategies are typically token-specific, the absence of a uniqueness check means a configuration error (e.g., copy-paste mistake during a strategy migration or protocol upgrade) can silently introduce double-counting with no on-chain revert.

**Likelihood: Low** — Requires an admin misconfiguration, but the missing guard makes it silently possible.

---

### Recommendation

In `LRTConfig.updateAssetStrategy()`, iterate over `supportedAssetList` and revert if any other asset already maps to the proposed strategy address:

```solidity
address[] memory assets = supportedAssetList;
for (uint256 i = 0; i < assets.length; i++) {
    if (assets[i] != asset && assetStrategy[assets[i]] == strategy) {
        revert StrategyAlreadyUsedByAnotherAsset();
    }
}
``` [5](#0-4) 

---

### Proof of Concept

1. Protocol has `assetA` (e.g., stETH) with `assetStrategy[assetA] = strategyX`.
2. Admin calls `updateAssetStrategy(assetB, strategyX)` — passes all existing checks because `assetStrategy[assetB] != strategyX`.
3. NDC holds 1000 stETH deposited into `strategyX`.
4. `getAssetBalance(assetA)` → 1000 (reads `strategyX` shares for NDC).
5. `getAssetBalance(assetB)` → 1000 (reads the same `strategyX` shares for NDC).
6. `getTotalAssetDeposits(assetA)` includes 1000; `getTotalAssetDeposits(assetB)` also includes 1000.
7. `_getTotalEthInProtocol()` adds both: `1000 * priceA + 1000 * priceB` instead of `1000 * priceA`.
8. `rsETHPrice` is inflated; a new depositor calling `depositAsset(assetA, 1000, ...)` receives fewer rsETH than they should. [6](#0-5) [7](#0-6)
