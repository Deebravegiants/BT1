### Title
Chainlink Price Feed Staleness Not Validated in `ChainlinkPriceOracle`, Enabling Stale-Price rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards every return value except `price`. There is no check on `updatedAt` (time-based staleness), no check that `answeredInRound >= roundId` (round completeness), and no check that `price > 0`. A stale or zero price propagates directly into `LRTOracle._updateRsETHPrice()`, which is publicly callable, causing the rsETH exchange rate to be computed from incorrect asset valuations.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the Chainlink price as follows:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

All five return values of `latestRoundData()` — `roundId`, `answer`, `startedAt`, `updatedAt`, `answeredInRound` — are available in the interface declaration, but only `answer` is used: [2](#0-1) 

The missing validations are:
- `updatedAt == 0` or `block.timestamp - updatedAt > heartbeat` (stale round)
- `answeredInRound < roundId` (incomplete round)
- `price <= 0` (invalid answer)

This price is consumed by `LRTOracle.getAssetPrice()`: [3](#0-2) 

Which is called inside `_getTotalEthInProtocol()` for every supported LST asset: [4](#0-3) 

Which feeds directly into `_updateRsETHPrice()`, the function that sets the on-chain `rsETHPrice` used for all deposit minting: [5](#0-4) 

`updateRSETHPrice()` is public with no access control: [6](#0-5) 

By contrast, `ChainlinkOracleForRSETHPoolCollateral` — a sibling oracle in the same repo — does validate `answeredInRound < roundID` and `timestamp == 0`, demonstrating the team is aware of the pattern but did not apply it to `ChainlinkPriceOracle`: [7](#0-6) 

---

### Impact Explanation

**High — Theft of unclaimed yield / share mis-accounting.**

If a Chainlink feed for any supported LST (e.g., stETH/ETH, cbETH/ETH) goes stale with a price lower than the true value:

1. `_getTotalEthInProtocol()` underestimates the protocol TVL.
2. `newRsETHPrice` is computed lower than the true rate.
3. A depositor who calls `deposit()` immediately after `updateRSETHPrice()` receives more rsETH shares than the deposited ETH value warrants.
4. This dilutes the rsETH/ETH rate for all existing holders — a direct transfer of value from existing holders to the new depositor.

If the stale price is higher than true value, the inverse occurs: depositors receive fewer shares, and the rsETH price is inflated, harming depositors.

In either direction, the `pricePercentageLimit` guard only triggers on large deviations; small but economically meaningful staleness (e.g., a feed that has not updated for several hours during a market move) passes through silently.

---

### Likelihood Explanation

**Medium.** Chainlink LST/ETH feeds have heartbeats of 1–24 hours and deviation thresholds of 0.5–1%. During periods of network congestion or Chainlink node issues, feeds can lag. The entry point (`updateRSETHPrice()`) is permissionless, so any depositor can trigger the price update at the moment most favorable to them — immediately after observing a stale feed on-chain.

---

### Recommendation

Add staleness and validity checks in `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

require(price > 0, "Invalid price");
require(answeredInRound >= roundId, "Stale round");
require(updatedAt != 0, "Incomplete round");
require(block.timestamp - updatedAt <= MAX_STALENESS, "Price too stale");

return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

`MAX_STALENESS` should be set per feed based on its documented heartbeat (e.g., 3600 seconds for a 1-hour heartbeat feed, with a small buffer). Store it alongside `assetPriceFeed` in the mapping or as a separate `assetPriceFeedStaleness` mapping.

---

### Proof of Concept

1. Chainlink stETH/ETH feed has not updated for 2 hours (within its 24-hour heartbeat, so no circuit breaker fires, but price has drifted 0.3% below true value).
2. Attacker observes the stale feed on-chain.
3. Attacker calls `LRTOracle.updateRSETHPrice()` — permissionless.
4. `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns the stale low price.
5. `newRsETHPrice` is computed ~0.3% below true value.
6. Attacker immediately calls `LRTDepositPool.depositAsset(stETH, largeAmount)` — receives ~0.3% more rsETH than the deposited value warrants.
7. True price is restored on next Chainlink update; attacker's rsETH is now worth more than deposited, at the expense of existing holders. [8](#0-7) [6](#0-5) [9](#0-8)
