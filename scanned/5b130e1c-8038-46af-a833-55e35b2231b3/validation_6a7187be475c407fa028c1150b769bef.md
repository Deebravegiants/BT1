### Title
Incorrect `setFeeBps` Upper-Bound Constant in `RSETHPoolV3` and `RSETHPoolV2ExternalBridge` Prevents Admin from Setting Fees Above 10% - (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolV2ExternalBridge.sol)

### Summary
`RSETHPoolV3` and `RSETHPoolV2ExternalBridge` enforce a hardcoded fee cap of `1000` bps (10%) inside `setFeeBps`, while every other pool variant in the same codebase caps at `10_000` bps (100%). This inconsistency means the admin of these two pools can never raise the deposit fee high enough to effectively price out or throttle depositors via the fee mechanism, contrary to the capability offered by all sibling contracts.

### Finding Description
In `RSETHPoolV3.setFeeBps`:

```solidity
function setFeeBps(uint256 _feeBps) external onlyRole(DEFAULT_ADMIN_ROLE) {
    if (_feeBps > 1000) revert InvalidFeeAmount();   // 10% cap
    feeBps = _feeBps;
    emit FeeBpsSet(_feeBps);
}
```

In `RSETHPoolV2ExternalBridge.setFeeBps`:

```solidity
function setFeeBps(uint256 _feeBps) external onlyRole(DEFAULT_ADMIN_ROLE) {
    if (_feeBps > 1000) revert InvalidFeeAmount();   // 10% cap
    feeBps = _feeBps;
    emit FeeBpsSet(_feeBps);
}
```

Every other pool variant uses `10_000` as the upper bound:

- `RSETHPool.setFeeBps`: `if (_feeBps > 10_000) revert InvalidFeeAmount();`
- `RSETHPoolV2.setFeeBps`: `if (_feeBps > 10_000) revert InvalidFeeAmount();`
- `RSETHPoolNoWrapper.setFeeBps`: `if (_feeBps > 10_000) revert InvalidFeeAmount();`
- `RSETHPoolV3ExternalBridge.setFeeBps`: `if (_feeBps > 10_000) revert InvalidFeeAmount();`

The fee is applied to every deposit:

```solidity
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

Because `feeBps` is divided by `10_000`, a value of `10_000` would consume the entire deposit (effectively disabling it via fees). The 1000-bps cap in `RSETHPoolV3` and `RSETHPoolV2ExternalBridge` makes this impossible in those contracts.

### Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

The admin of `RSETHPoolV3` and `RSETHPoolV2ExternalBridge` cannot raise the deposit fee above 10%, even though the identical setter in four other pool contracts allows up to 100%. Any depositor interacting with these two pools will always pay at most a 10% fee regardless of the admin's intent, and the admin has no fee-based mechanism to throttle or effectively disable deposits in these specific pools. The protocol does have a separate `pause()` path, but the fee-based control plane is permanently capped at a value inconsistent with the rest of the system.

### Likelihood Explanation
The incorrect constant is already deployed and active. Any deposit into `RSETHPoolV3` or `RSETHPoolV2ExternalBridge` exercises the capped fee path. The admin discovering the need to raise fees above 10% (e.g., during a market stress event or a controlled wind-down) will find the setter silently reverts, with no on-chain indication that the cap is anomalously low relative to sibling contracts.

### Recommendation
Change the fee cap in `RSETHPoolV3.setFeeBps` and `RSETHPoolV2ExternalBridge.setFeeBps` from `1000` to `10_000` to match every other pool variant:

```solidity
if (_feeBps > 10_000) revert InvalidFeeAmount();
```

Introduce a named constant (e.g., `uint256 public constant MAX_FEE_BPS = 10_000;`) shared across all pool contracts to prevent future divergence.

### Proof of Concept
1. Deploy `RSETHPoolV3` (or use the live proxy).
2. As `DEFAULT_ADMIN_ROLE`, call `setFeeBps(1001)` — transaction reverts with `InvalidFeeAmount`.
3. Call `setFeeBps(1000)` — succeeds; fee is now 10%.
4. A depositor calls `deposit{value: 1 ether}("")`; they receive `rsETH` corresponding to `0.9 ETH` worth of value — the maximum fee the admin can ever impose.
5. Repeat steps 2–4 on `RSETHPoolV2ExternalBridge` — identical behaviour.
6. Repeat on `RSETHPoolV3ExternalBridge` with `setFeeBps(1001)` — succeeds, confirming the cap is `10_000` there, proving the inconsistency. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
