### Title
ETH Deposit Limit Bypass at Exact Boundary Due to `>` Instead of `>=` - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` uses a strict `>` comparison for ETH, meaning when `totalAssetDeposits` exactly equals the configured deposit limit, the check incorrectly returns `false` (not exceeded) and allows additional ETH deposits beyond the protocol's configured cap.

### Finding Description
In `_checkIfDepositAmckIfDepositAmountExceedesCurrentLimit`, the ETH branch is:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
}
``` [1](#0-0) 

When `totalAssetDeposits == depositLimit`, the expression `totalAssetDeposits > depositLimit` evaluates to `false`, so the function returns `false` (not exceeded), and `_beforeDeposit` does not revert with `MaximumDepositLimitReached`. The caller's ETH deposit then proceeds, pushing the total above the configured limit.

The non-ETH branch correctly includes the incoming `amount`:

```solidity
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
``` [2](#0-1) 

The ETH branch has no analogous inclusion of `msg.value`, compounding the off-by-one: even at the exact limit, any further ETH deposit is permitted.

### Impact Explanation
The protocol fails to enforce its own ETH deposit cap at the exact boundary. Any depositor can push total ETH deposits above `depositLimitByAsset` by depositing when `totalAssetDeposits == depositLimit`. This violates the protocol's promised deposit ceiling and can cause the protocol to hold more ETH than intended by governance/risk parameters.

**Impact category**: Low — contract fails to deliver promised returns (deposit limit enforcement), but no direct fund loss.

### Likelihood Explanation
Reaching `totalAssetDeposits == depositLimit` exactly is an edge case, but it is reachable in normal operation as deposits accumulate. A depositor monitoring on-chain state can time a deposit to land precisely at the boundary. No special privileges are required — any unprivileged depositor calling `depositETH` can trigger this.

### Recommendation
Change the ETH branch to use `>=`:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits >= lrtConfig.depositLimitByAsset(asset));
}
```

This mirrors the intent of the non-ETH branch and blocks deposits once the limit is reached.

### Proof of Concept
1. Admin sets `depositLimitByAsset[ETH] = 1000 ether`.
2. Users deposit ETH until `getTotalAssetDeposits(ETH) == 1000 ether`.
3. A new depositor calls `depositETH{value: 1 ether}(...)`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `1000 ether > 1000 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for the depositor; total ETH in protocol becomes `1001 ether`, exceeding the configured limit. [3](#0-2) [4](#0-3)
