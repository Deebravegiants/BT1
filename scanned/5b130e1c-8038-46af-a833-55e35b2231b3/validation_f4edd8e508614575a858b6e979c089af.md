### Title
Fee-on-Transfer Token Deposit Mints rsETH Based on Stated Amount, Not Actual Received — (`contracts/LRTDepositPool.sol`)

---

### Summary

`LRTDepositPool.depositAsset` computes `rsethAmountToMint` from the caller-supplied `depositAmount` parameter and mints that amount to the depositor **before** (or without) verifying how many tokens the contract actually received. For any ERC-20 asset that deducts a transfer fee, the contract receives fewer tokens than `depositAmount` but mints rsETH as if the full amount arrived, creating an insolvency condition that compounds with every deposit.

---

### Finding Description

In `LRTDepositPool.depositAsset`:

```solidity
// contracts/LRTDepositPool.sol  lines 99-118
function depositAsset(
    address asset,
    uint256 depositAmount,
    uint256 minRSETHAmountExpected,
    string calldata referralId
) external nonReentrant whenNotPaused onlySupportedERC20Token(asset) {
    // rsethAmountToMint is derived entirely from the caller-supplied depositAmount
    uint256 rsethAmountToMint = _beforeDeposit(asset, depositAmount, minRSETHAmountExpected);

    // actual tokens received = depositAmount − transferFee  (for fee-on-transfer tokens)
    IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);

    // mints based on depositAmount, not actual received
    _mintRsETH(rsethAmountToMint);
    ...
}
```

`_beforeDeposit` calls `getRsETHAmountToMint(asset, depositAmount)`, which prices the mint entirely against the parameter value. [1](#0-0) 

`_mintRsETH` then mints directly to `msg.sender` with no post-transfer balance check. [2](#0-1) 

The same structural flaw is present in every L2 pool variant that accepts ERC-20 tokens:

- `RSETHPoolV3.deposit(address token, ...)` — mints `wrsETH` based on `amount` [3](#0-2) 
- `RSETHPoolV3ExternalBridge.deposit(address token, ...)` [4](#0-3) 
- `RSETHPoolV3WithNativeChainBridge.deposit(address token, ...)` [5](#0-4) 

By contrast, `KernelDepositPool.notifyRewardAmount` demonstrates the correct pattern already used elsewhere in the same codebase — balance-before/after accounting to determine the actual received amount: [6](#0-5) 

New supported assets are added via `LRTConfig.addNewSupportedAsset` (gated to `TIME_LOCK_ROLE`) and via `addSupportedToken` on each L2 pool (gated to `TIMELOCK_ROLE`). Neither function checks whether the token implements a transfer fee. [7](#0-6) [8](#0-7) 

---

### Impact Explanation

**Critical — Protocol insolvency.**

For every deposit of a fee-on-transfer token, the contract receives `depositAmount − fee` but mints rsETH worth `depositAmount`. The depositor immediately holds rsETH backed by more value than the pool actually holds. When they redeem, they drain real assets from the pool that belong to other depositors. Because there is no mechanism to reverse a completed mint, the deficit is permanent and grows with each deposit. This is identical in structure to the phantom-deposit insolvency described in the reference report.

---

### Likelihood Explanation

**Low.**

The currently deployed supported assets (stETH, ETHx) are not fee-on-transfer tokens, so the vulnerability is latent rather than immediately active. Exploitation requires a fee-on-transfer token to be added to the supported asset list through a legitimate governance action (timelock). Tokens such as USDT (when its dormant fee switch is activated), certain DeFi tokens, or future LSTs with protocol fees are realistic candidates as the protocol expands. The admin is not acting maliciously — the code simply has no guard against this token class, and the developers demonstrably know the correct pattern (see `KernelDepositPool`).

---

### Recommendation

Use a balance-before/after pattern to determine the actual amount received, and base the mint on that value:

```solidity
function depositAsset(
    address asset,
    uint256 depositAmount,
    uint256 minRSETHAmountExpected,
    string calldata referralId
) external nonReentrant whenNotPaused onlySupportedERC20Token(asset) {
    uint256 balanceBefore = IERC20(asset).balanceOf(address(this));
    IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
    uint256 actualReceived = IERC20(asset).balanceOf(address(this)) - balanceBefore;

    uint256 rsethAmountToMint = _beforeDeposit(asset, actualReceived, minRSETHAmountExpected);
    _mintRsETH(rsethAmountToMint);

    emit AssetDeposit(msg.sender, asset, actualReceived, rsethAmountToMint, referralId);
}
```

Apply the same fix to all L2 pool `deposit(address token, ...)` functions.

---

### Proof of Concept

1. Governance adds a fee-on-transfer token `FOT` (1% fee) as a supported asset via `LRTConfig.addNewSupportedAsset`.
2. Attacker calls `LRTDepositPool.depositAsset(FOT, 10_000e18, 0, "")`.
3. `_beforeDeposit` computes `rsethAmountToMint` based on `10_000e18`.
4. `safeTransferFrom` transfers `10_000e18` from attacker; contract receives `9_900e18` (1% fee deducted).
5. `_mintRsETH` mints rsETH equivalent to `10_000e18` FOT to the attacker.
6. Attacker holds rsETH backed by `10_000e18` worth of FOT, but the pool only holds `9_900e18`.
7. Attacker redeems all rsETH, withdrawing `10_000e18` worth of assets — draining `100e18` from other depositors.
8. Repeated deposits compound the deficit until the pool is insolvent.
