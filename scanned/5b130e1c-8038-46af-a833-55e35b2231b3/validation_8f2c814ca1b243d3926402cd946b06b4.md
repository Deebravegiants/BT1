### Title
Stale Chainlink Price Accepted Without Staleness Validation in `ChainlinkPriceOracle` - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards the `updatedAt` timestamp and `answeredInRound` fields, accepting arbitrarily stale prices. This stale price propagates directly into rsETH mint calculations for depositors and asset-return calculations for withdrawers, enabling incorrect share issuance or excess asset extraction.

### Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol`, `getAssetPrice()` destructures the Chainlink response as:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

The `updatedAt` (index 3) and `answeredInRound` (index 4) return values are completely ignored. No check of the form `if (answeredInRound < roundId) revert StalePrice()` or `if (block.timestamp - updatedAt > MAX_DELAY) revert()` is present.

The protocol itself demonstrates awareness of this requirement: `ChainlinkOracleForRSETHPoolCollateral` — used for pool collateral pricing — performs both checks:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
``` [2](#0-1) 

Yet the core `ChainlinkPriceOracle` used for LST assets (stETH, cbETH, etc.) has no equivalent guard.

The stale price flows into two critical paths:

**1. Deposit minting** (`LRTDepositPool.getRsETHAmountToMint`):
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [3](#0-2) 

**2. Withdrawal asset calculation** (`LRTWithdrawalManager.getExpectedAssetAmount`):
```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
``` [4](#0-3) 

**3. rsETH price update** (`LRTOracle._updateRsETHPrice`) calls `_getTotalEthInProtocol()` which aggregates `getAssetPrice` for all supported LSTs to compute the protocol's total ETH value, directly setting `rsETHPrice`. [5](#0-4) 

### Impact Explanation
If a supported LST's Chainlink feed goes stale with a price **lower** than the true market price (e.g., during network congestion after a price increase):

- **Deposit path**: `rsethAmountToMint` is reduced — depositors receive fewer rsETH than owed. The protocol under-delivers on promised returns.
- **Withdrawal path**: `underlyingToReceive = rsETHAmount * rsETHPrice / assetPrice` — a stale low `assetPrice` inflates the denominator's inverse, giving withdrawers **more** underlying asset than they are entitled to, draining assets from the pool at the expense of remaining holders.
- **rsETH price update**: A stale low price for a large LST (e.g., stETH) deflates `totalETHInProtocol`, causing `rsETHPrice` to be set below its true value, which then cascades into all subsequent deposit/withdrawal calculations.

Conversely, a stale **high** price (feed not reflecting a price drop) inflates `rsethAmountToMint`, diluting existing rsETH holders.

Impact classification: **Low — Contract fails to deliver promised returns** (under-minting to depositors); with the withdrawal path, escalates to **High — Theft of unclaimed yield** (over-disbursement to withdrawers when asset price is stale-low).

### Likelihood Explanation
Chainlink feeds for LSTs (stETH/ETH, cbETH/ETH) have documented heartbeat intervals (typically 1–24 hours). During Ethereum network congestion or Chainlink node outages, feeds can lag significantly. This is a known, recurring condition — not a theoretical edge case. Any unprivileged depositor or withdrawer can interact with the protocol during a staleness window without any special access.

### Recommendation
Add staleness validation in `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (block.timestamp - updatedAt > MAX_STALENESS_DELAY) revert StalePrice();
    if (price <= 0) revert InvalidPrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`MAX_STALENESS_DELAY` should be set per-asset based on the Chainlink feed's documented heartbeat (e.g., 3600 seconds for a 1-hour heartbeat feed, with a small buffer).

### Proof of Concept

1. Chainlink's stETH/ETH feed goes stale at price `P_stale = 0.98e18` while the true price rises to `P_true = 1.00e18`.
2. A withdrawer calls `LRTWithdrawalManager.unlockQueue(stETH, ...)` which internally calls `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns `0.98e18` (stale).
3. `_calculatePayoutAmount` computes `currentReturn = rsETHUnstaked * rsETHPrice / 0.98e18`, yielding ~2% more stETH than the withdrawer is entitled to at the true price.
4. The excess stETH is transferred out of `LRTUnstakingVault`, permanently reducing the assets backing remaining rsETH holders.
5. No revert occurs because `ChainlinkPriceOracle` performs zero staleness validation. [6](#0-5) [7](#0-6)
