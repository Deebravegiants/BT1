### Title
`MetricOmmSimpleRouter` Acts as an Unprivileged Proxy to Bypass `SwapAllowlistExtension` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap()` gates on `sender`, which is `msg.sender` of the pool's `swap()` call — the direct caller. When `MetricOmmSimpleRouter` mediates a swap, `sender` equals the router's address, not the end user's. If a pool admin allowlists the router (a natural step to enable router-mediated swaps on a permissioned pool), every user of the public, permissionless router bypasses the allowlist entirely.

---

### Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to extensions.**

In `MetricOmmPool.swap()`, the `sender` forwarded to `_beforeSwap` is always `msg.sender`:

```solidity
_beforeSwap(
    msg.sender,   // ← direct caller of the pool
    recipient,
    ...
);
``` [1](#0-0) 

**Step 2 — `ExtensionCalling._beforeSwap` forwards that address verbatim.** [2](#0-1) 

**Step 3 — `SwapAllowlistExtension.beforeSwap` checks `sender`, not the end user.**

```solidity
function beforeSwap(address sender, address, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
``` [3](#0-2) 

Here `msg.sender` is the pool and `sender` is whoever called the pool. When the router calls the pool, `sender` = router address.

**Step 4 — `MetricOmmSimpleRouter` is a public, permissionless contract.**

`exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput` are all callable by any address with no access control. The router calls `pool.swap(...)` directly:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
``` [4](#0-3) 

**Step 5 — Asymmetry with `DepositAllowlistExtension`.**

The deposit allowlist correctly gates on `owner` (the position owner), not `sender` (the adder contract), so the `MetricOmmPoolLiquidityAdder` does not create the same bypass for deposits:

```solidity
function beforeAddLiquidity(address, address owner, ...) external view override returns (bytes4) {
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [5](#0-4) 

The swap extension has no equivalent "swap owner" parameter — it only sees the direct caller.

---

### Impact Explanation

A pool admin who deploys a permissioned pool with `SwapAllowlistExtension` and then allowlists the `MetricOmmSimpleRouter` (intending to enable router-mediated swaps for their allowlisted users) inadvertently opens the pool to **every user of the public router**. Any address — including those the allowlist was designed to exclude — can call `exactInputSingle` or `exactInput` on the router and execute swaps against the pool. The swap itself executes at oracle-fair prices, so LP funds are not directly drained, but the access-control invariant the pool was deployed to enforce is completely nullified. For pools that use the allowlist to enforce KYC, jurisdiction restrictions, or counterparty vetting, this constitutes broken core pool functionality.

---

### Likelihood Explanation

Low-Medium. The trigger requires a pool admin to allowlist the router address. This is a natural operational mistake: an admin who wants "router users" to be able to swap on their permissioned pool will allowlist the router, not realizing the router is fully permissionless. The `SwapAllowlistExtension` provides no warning, and the protocol documentation does not flag this incompatibility. No attacker capital is required; any address can call the router immediately.

---

### Recommendation

1. **Preferred fix**: Change `SwapAllowlistExtension.beforeSwap` to accept an explicit "effective swapper" identity rather than relying on `sender`. One approach: add an optional `swapper` field to `extensionData` that the router populates with `msg.sender` (the end user), and have the extension decode and check that field when present.

2. **Alternative**: Document explicitly that the `MetricOmmSimpleRouter` must never be added to a pool's swap allowlist, and add a factory-level or extension-level guard that rejects the router address as an allowlist entry.

3. **Consistency fix**: Mirror the deposit allowlist pattern — gate on a caller-supplied identity (analogous to `owner`) rather than on `msg.sender` of the pool call.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension.
  - Pool admin calls swapExtension.setAllowedToSwap(pool, address(router), true)
    (intending to allow router-mediated swaps for their users).

Attack:
  - Attacker (not individually allowlisted) calls:
      router.exactInputSingle(ExactInputSingleParams({
          pool: pool,
          recipient: attacker,
          zeroForOne: true,
          amountIn: X,
          ...
      }));
  - Router calls pool.swap(...) with msg.sender = router.
  - SwapAllowlistExtension checks allowedSwapper[pool][router] → true.
  - Swap executes. Attacker receives output tokens.

Result:
  - Attacker bypassed the allowlist. The pool's access-control invariant is broken.
  - No privileged key is needed after the admin's one-time router allowlist entry.
```
