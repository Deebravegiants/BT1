### Title
Griefing DOS on `removeLiquidity` via Unrestricted `addLiquidity` Owner Parameter and `MinimalLiquidity` Dust Guard — (`metric-core/contracts/MetricOmmPool.sol`, `metric-core/contracts/libraries/LiquidityLib.sol`)

---

### Summary

`addLiquidity` accepts an arbitrary `owner` address with no restriction on who the caller is. An attacker can frontrun any victim's `removeLiquidity` call by depositing a minimal number of shares into the victim's position. This leaves the victim's remaining share balance in the forbidden dust range `(0, minimalMintableLiquidity)`, causing the `MinimalLiquidity` revert to fire and permanently blocking the victim from withdrawing their LP assets.

---

### Finding Description

`MetricOmmPool.addLiquidity` is an open operator-pattern function: any `msg.sender` may specify any `owner` address. [1](#0-0) 

The pool's own interface documents this explicitly:

> `msg.sender` pays but need not equal `owner` (operator pattern). [2](#0-1) 

The periphery test `test_exactShares_canAddOnBehalfOfAnotherOwner` confirms any caller can credit shares to an arbitrary address: [3](#0-2) 

`removeLiquidity` enforces a dust guard: if the remaining position shares after a burn are strictly between `0` and `minimalMintableLiquidity`, the call reverts: [4](#0-3) 

The `addLiquidity` path only requires that the *resulting* position after the add is `>= minimalMintableLiquidity`: [5](#0-4) 

Because the victim's existing position already satisfies this floor, an attacker can add as few as **1 share** to the victim's position (since `victimShares + 1 >= minimalMintableLiquidity` is trivially true). After the frontrun, the victim's attempt to remove their original share count leaves exactly `1` share remaining — which is `> 0` and `< minimalMintableLiquidity` — triggering the `MinimalLiquidity` revert.

The attacker can repeat this every time the victim re-queries their balance and resubmits, because the attacker can always frontrun with another 1-share deposit. The cost per frontrun is negligible (1 share worth of tokens, proportional to `initialScaledToken*PerShareE18`).

---

### Impact Explanation

- **Victim's LP assets are locked**: the victim cannot withdraw their principal from the pool.
- **Permanent griefing**: the attacker can sustain the DOS indefinitely at near-zero cost.
- **No privileged access required**: any unprivileged address can call `addLiquidity` with `owner = victim`.
- **Affects all pools without a deposit allowlist extension**, and even pools *with* the `DepositAllowlistExtension` are vulnerable as long as the victim (`owner`) is on the allowlist, since the extension checks `allowedDepositor[pool][owner]`, not the caller. [6](#0-5) 

---

### Likelihood Explanation

- Trigger requires only a standard `addLiquidity` call with `owner = victim` — no special role, no malicious setup.
- Frontrunning is straightforward on any public mempool chain.
- Cost to attacker: a dust token amount per block; cost to victim: permanent loss of liquidity access.
- Likelihood is **High** for any LP holding a position on a pool without strict per-caller deposit gating.

---

### Recommendation

Enforce that `msg.sender == owner` in `removeLiquidity` (already done) **and symmetrically** in `addLiquidity`, or at minimum require explicit consent from `owner` before a third party can credit shares to their position. The operator pattern should require an on-chain approval mapping (similar to ERC-20 `allowance`) so that only addresses explicitly authorized by `owner` can add to their position. Alternatively, the `MinimalLiquidity` check on removal should be skipped when the caller is removing their *entire* position (i.e., `newUserShares == 0`), which is already safe and is the common full-exit case — the dust guard is only meaningful for partial removals.

---

### Proof of Concept

```
Setup:
  - Pool with minimalMintableLiquidity = 1000
  - Victim holds 10_000 shares in bin 4 (salt = 1)

Step 1 (victim submits):
  pool.removeLiquidity(victim, salt=1, [{bin:4, shares:10_000}], "")
  → newUserShares = 0 → passes MinimalLiquidity check → would succeed

Step 2 (attacker frontruns, same block):
  pool.addLiquidity(owner=victim, salt=1, [{bin:4, shares:1}], callbackData, "")
  → newUserShares = 10_001 >= 1000 → passes add check
  → attacker pays ~1 share worth of tokens

Step 3 (victim's original tx executes):
  removeLiquidity removes 10_000 shares
  → newUserShares = 10_001 - 10_000 = 1
  → 1 > 0 && 1 < 1000 → revert MinimalLiquidity(1, 1000)

Step 4 (victim re-queries balance = 10_001, resubmits remove 10_001):
  Attacker frontruns again with 1 share → victim has 10_002
  → remove 10_001 → newUserShares = 1 → revert again

Victim's funds remain locked indefinitely.
``` [4](#0-3) [1](#0-0)
