### Title
Single `MAX_REF_STALENESS` Applied to Both Legs of Synthetic Two-Feed Quote — (`AnchoredPriceProvider.sol`)

---

### Summary

`AnchoredPriceProvider` supports a synthetic ratio mode (`quoteFeedId != 0`) where the final mid is `price(baseFeedId) / price(quoteFeedId)`. Both legs are validated through `_readLeg()`, which applies the single immutable `MAX_REF_STALENESS` to both feeds. The factory envelope that bounds `maxRefStaleness` at deploy time is keyed exclusively on `baseFeedId`'s class — the `quoteFeedId` has no separate staleness bound. When the two feeds have different on-chain update frequencies (heartbeats), the shared threshold is either too tight for the slow feed (causing constant `FeedStalled` reverts) or too loose for the fast feed (silently accepting a stale denominator price), producing a corrupted synthetic ratio that reaches live pool swaps.

---

### Finding Description

In `AnchoredPriceProvider._getBidAndAskPrice()`, both legs of the synthetic quote call the same internal helper:

```solidity
// AnchoredPriceProvider.sol lines 258-272
function _getBidAndAskPrice() internal returns (uint128, uint128) {
    (uint256 mid, uint256 spreadBps, , bool ok) = _readLeg(baseFeedId);
    if (!ok) return (0, type(uint128).max);

    bytes32 _quote = quoteFeedId;
    if (_quote != bytes32(0)) {
        (uint256 mid2, uint256 spreadBps2, , bool ok2) = _readLeg(_quote);  // same MAX_REF_STALENESS
        ...
        mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
    }
    return _computeBidAsk(mid, spreadBps);
}
```

`_readLeg` applies the single `MAX_REF_STALENESS` to whichever `feedId` it receives:

```solidity
// AnchoredPriceProvider.sol lines 277-295
function _readLeg(bytes32 feedId)
    internal returns (uint256 mid, uint256 spreadBps, uint256 refTime, bool ok)
{
    (mid, spreadBps, , refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);
    if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);
    ...
}
```

`MAX_REF_STALENESS` is a single immutable set at construction. The factory validates it only against the `baseFeedId`'s envelope class:

```solidity
// AnchoredProviderFactory.sol lines 171-180
bytes32 classId = feedClass[baseFeedId];          // keyed on base only
if (classId == bytes32(0)) classId = DEFAULT_CLASS;
Envelope storage env = envelopes[classId];
if (
    ...
    || maxRefStaleness < env.stalenessMin || maxRefStaleness > env.stalenessMax
    ...
) revert ParamsOutOfEnvelope();
```

There is no envelope lookup, no class assignment, and no staleness bound for `quoteFeedId`. The deployer passes a single `maxRefStaleness` that is validated against the base feed's class and then applied identically to both legs at runtime.

---

### Impact Explanation

Consider a synthetic BTC/ETH provider built from `baseFeedId = BTC/USD` (Chainlink, 3600 s heartbeat) and `quoteFeedId = ETH/USD` (Pyth, ~10 s heartbeat):

- **If `MAX_REF_STALENESS` is set to 3600 s** (correct for the base feed, within its envelope): the ETH/USD denominator can be up to 3600 s stale without triggering the staleness guard. A 3600 s stale ETH price can deviate by several percent from the current market price, producing a corrupted BTC/ETH ratio. Swaps execute against this stale ratio — traders receive more output tokens than the true price permits, or the pool receives fewer input tokens than it is owed. This is a direct swap conservation failure.

- **If `MAX_REF_STALENESS` is set to 10 s** (correct for the quote feed): the BTC/USD base leg fails its staleness check on almost every block (Chainlink updates every 3600 s), causing `FeedStalled` reverts on every swap — the pool is permanently bricked for swaps.

Neither setting is simultaneously correct for both feeds. The deployer has no mechanism to assign per-leg staleness thresholds.

---

### Likelihood Explanation

Synthetic two-feed mode is an explicitly documented and supported feature (the `quoteFeedId` parameter is described in both the constructor NatSpec and the factory's `createAnchoredProvider` NatSpec as enabling "BTC/USD ÷ ETH/USD = BTC/ETH" style quoting). Any curator who deploys a synthetic provider pairing feeds with different heartbeats — a common real-world scenario — hits this bug. No privileged action beyond normal provider deployment is required. The factory's envelope system gives no warning and does not reject the misconfigured staleness value because it only checks the base feed's class.

---

### Recommendation

Introduce a separate `MAX_QUOTE_REF_STALENESS` immutable for the `quoteFeedId` leg, set independently at construction and validated against a separate envelope class for `quoteFeedId`. Pass it explicitly to `_readLeg` (or use a dedicated `_readQuoteLeg` helper):

```solidity
// Constructor: accept and store a second staleness bound
uint256 public immutable MAX_QUOTE_REF_STALENESS;

// _getBidAndAskPrice: pass per-leg bound
(uint256 mid2, ...) = _readLeg(_quote, MAX_QUOTE_REF_STALENESS);

// _readLeg: accept the bound as a parameter
function _readLeg(bytes32 feedId, uint256 maxStaleness) internal returns (...) {
    ...
    if (_isStale(refTime, block.timestamp, maxStaleness)) return (..., false);
    ...
}
```

The factory should look up the `quoteFeedId`'s class independently and validate `maxQuoteRefStaleness` against that class's envelope before deployment.

---

### Proof of Concept

1. Admin configures `DEFAULT_CLASS` envelope with `stalenessMin = 1 s`, `stalenessMax = 7200 s`.
2. Curator calls `createAnchoredProvider` with:
   - `baseFeedId` = BTC/USD (Chainlink, 3600 s heartbeat)
   - `quoteFeedId` = ETH/USD (Pyth, 10 s heartbeat)
   - `maxRefStaleness` = 3600 s (passes envelope check, correct for base)
3. Provider is deployed with `MAX_REF_STALENESS = 3600`.
4. Pyth ETH/USD feed is last updated 3500 s ago (within `MAX_REF_STALENESS`). True ETH price has moved 4% since then.
5. A trader calls `swap` on the pool. The pool calls `provider.getBidAndAskPrice()`.
6. `_readLeg(quoteFeedId)` reads the 3500 s stale ETH/USD price — staleness check passes (`3500 <= 3600`).
7. Synthetic mid = `BTC_USD_current / ETH_USD_stale` — the denominator is 4% too low, so the ratio is ~4% too high.
8. The pool quotes a BTC/ETH bid/ask that is ~4% above true market. A trader selling BTC for ETH receives ~4% more ETH than the current market price warrants, draining the pool's ETH reserves beyond what the oracle-permitted price allows. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```text
    function _readLeg(bytes32 feedId)
        internal returns (uint256 mid, uint256 spreadBps, uint256 refTime, bool ok)
    {
        (mid, spreadBps, , refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);

        // Stale reference → not ok. Clamping to a stale anchor is the one false-safety case.
        if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);

        // Basic validity — mid positive, spreadBps not the stalled/off-hours marker (the Chainlink oracle
        // writes spreadBps = ORACLE_BPS when an RWA market is closed).
        if (mid == 0 || spreadBps >= ORACLE_BPS) return (mid, spreadBps, refTime, false);

        // Per-leg price guard.
        (uint128 guardMin, uint128 guardMax) = offchainOracle.priceGuard(feedId);
        guardMax = guardMax == 0 ? type(uint128).max : guardMax;
        if (mid < guardMin || mid > guardMax) return (mid, spreadBps, refTime, false);

        ok = true;
    }
```
