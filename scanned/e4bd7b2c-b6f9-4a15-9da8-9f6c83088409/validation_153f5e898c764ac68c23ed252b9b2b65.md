### Title
ETH Deposit Freeze via Self-Destruct Balance Inflation — (`contracts/LRTDepositPool.sol`)

### Summary
`LRTDepositPool.getETHDistributionData()` reads `address(this).balance` to compute the protocol's total ETH holdings. Because Solidity's `address.balance` includes ETH force-sent via `selfdestruct`, an attacker can inflate the reported ETH total above the configured deposit limit, causing every subsequent call to `depositETH()` to revert with `MaximumDepositLimitReached`.

### Finding Description
`getETHDistributionData()` computes the ETH lying in the deposit pool as the raw contract balance:

```solidity
// LRTDepositPool.sol L480
ethLyingInDepositPool = address(this).balance;
```

This value flows into `getTotalAssetDeposits(ETH_TOKEN)` → `_checkIfDepositAmountExceedesCurrentLimit()`:

```solidity
// LRTDepositPool.sol L678-L680
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
}
```

Because `depositETH` is `payable`, `msg.value` is already included in `address(this).balance` when `_beforeDeposit` is called. The check therefore compares the **post-deposit** total against the limit. If an attacker force-sends even 1 wei above the remaining headroom via `selfdestruct`, the condition `totalAssetDeposits > depositLimit` becomes permanently true for any non-zero deposit, and every `depositETH` call reverts.

The same inflation path also affects `nodeDelegatorQueue[i].balance` and `lrtUnstakingVault.balance` in the same function, giving the attacker three separate contracts to target.

### Impact Explanation
All ETH deposits to the protocol are blocked until a privileged admin raises the deposit limit. Because the attacker can repeat the attack each time the limit is raised (at the cost of a small amount of ETH), the freeze can be maintained indefinitely at low cost. Users cannot mint rsETH via ETH, which is the primary deposit path.

**Impact class:** Medium — Temporary freezing of funds (ETH deposit functionality).

### Likelihood Explanation
The attack requires only a `selfdestruct` contract funded with a small amount of ETH (as little as 1 wei above the remaining deposit headroom). No privileged access is needed. The `LRTDepositPool` has a public `receive()` function, confirming it is a known ETH recipient, and its address is publicly known. Any unprivileged actor can execute this.

### Recommendation
Track ETH deposits with an internal accounting variable (e.g., `totalETHDeposited`) that is incremented only through controlled entry points (`depositETH`, `receiveFromNodeDelegator`, `receiveFromLRTConverter`, `receiveFromRewardReceiver`), and use that variable instead of `address(this).balance` in `getETHDistributionData()`. This mirrors the fix pattern from the reference report: cap or ignore untracked balance increases.

### Proof of Concept
1. Observe the current deposit limit `L = lrtConfig.depositLimitByAsset(ETH_TOKEN)` and current total `T = getTotalAssetDeposits(ETH_TOKEN)`. Assume `T < L`.
2. Deploy a contract `Bomb` holding `L - T + 1 wei` of ETH.
3. Call `Bomb.explode(address(lrtDepositPool))` which executes `selfdestruct(lrtDepositPool)`.
4. `address(lrtDepositPool).balance` is now `≥ L + 1 wei` (original balance + forced ETH).
5. Any user calling `depositETH{value: X}(...)` triggers `_beforeDeposit` → `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, X)` → `getTotalAssetDeposits(ETH_TOKEN)` → `getETHDistributionData()` → `ethLyingInDepositPool = address(this).balance` which now exceeds `L`.
6. The check `totalAssetDeposits > depositLimitByAsset(ETH_TOKEN)` returns `true`, and the call reverts with `MaximumDepositLimitReached`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
