### Title
Integer Truncation in `notifyRewardAmount` Permanently Locks Reward Dust - (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary

`KernelDepositPool.notifyRewardAmount` computes `rewardRate` via integer division against `duration`. The truncated remainder (`receivedAmount % duration`) is transferred into the contract but can never be streamed to stakers, permanently locking it as unclaimable dust. This occurs on every reward notification and compounds on top-ups.

### Finding Description

In `KernelDepositPool.notifyRewardAmount`, the reward rate is computed with integer division:

```solidity
// Line 580 — new period
rewardRate = receivedAmount / duration;

// Line 583 — top-up during active period
rewardRate = (receivedAmount + remaining) / duration;
```

The total tokens that will ever be streamed to stakers is `rewardRate * duration`, which equals `(receivedAmount / duration) * duration`. The difference — `receivedAmount % duration` — is transferred into the contract but is never distributed. There is no recovery path: `getReward()` only pays out `rewards[msg.sender]`, which is accumulated solely through `rewardRate`-based accounting. No sweep, rescue, or carry-forward mechanism exists for the truncated remainder.

On a top-up (line 583), the leftover from the current period (`remaining = (finishAt - block.timestamp) * rewardRate`) is folded back in, but the new combined total is again integer-divided, creating a fresh remainder that is also lost.

### Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

Every call to `notifyRewardAmount` locks up to `duration - 1` base units of `rewardsToken` permanently. For a `duration` of 7 days (604,800 seconds) and a 6-decimal token (e.g., USDC), up to ~0.6 USDC is locked per call. Repeated calls accumulate this dust indefinitely with no recovery mechanism.

### Likelihood Explanation

`notifyRewardAmount` is called by the admin as routine protocol operation to distribute staking rewards. It is called every reward period (e.g., weekly). The truncation occurs on every single call unless `receivedAmount` happens to be an exact multiple of `duration`, which is unlikely in practice. The dust accumulates silently and permanently.

### Recommendation

Compute the effective streamable amount before setting `rewardRate`, and either return the remainder to the caller or carry it forward:

```solidity
uint256 effective = (receivedAmount / duration) * duration;
uint256 dust = receivedAmount - effective;
// return dust to msg.sender, or accumulate in a pendingDust variable
rewardRate = effective / duration;
```

This guarantees `rewardRate * duration == effective` and no tokens are silently locked.

### Proof of Concept

1. Admin calls `notifyRewardAmount(1_000_001e6)` with `duration = 604800` (7 days).
2. `rewardRate = 1_000_001e6 / 604800 = 1653439` (truncated).
3. Total streamable = `1653439 * 604800 = 999_999_187_200` units.
4. Dust locked = `1_000_001_000_000 - 999_999_187_200 = 1_812_800` units ≈ 1.81 USDC permanently locked per call.
5. After `finishAt`, all stakers claim their rewards. The contract retains `1_812_800` units with no function to recover them.
6. On the next `notifyRewardAmount` call, a new remainder is created and added to the locked balance.
