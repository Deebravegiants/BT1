### Title
Pool Contracts Lack `receive()`/`fallback()`, Causing Revert When External Bridge Refunds Native ETH — (`contracts/pools/RSETHPoolV3ExternalBridge.sol`, `contracts/pools/RSETHPoolV3WithNativeChainBridge.sol`)

---

### Summary

`RSETHPoolV3ExternalBridge` and `RSETHPoolV3WithNativeChainBridge` forward native ETH to external bridge contracts (`IL2Messenger`, `IL2TokenBridge`) but define no `receive()` or `fallback()` function. If those external contracts attempt to return any unused ETH to the pool (e.g., unused bridging fees, rounding dust), the entire transaction reverts, permanently blocking the bridging path for user-deposited ETH and tokens.

---

### Finding Description

Every pool contract in `contracts/pools/` was confirmed to have **zero** `receive()` or `fallback()` definitions (grep across `contracts/pools/**/*.sol` returned no matches).

Two contracts make outbound calls that forward ETH to external protocols where the pool itself is `msg.sender` and therefore the natural refund target:

**1. `RSETHPoolV3ExternalBridge.bridgeAssetsViaNativeBridge()`**

```solidity
// contracts/pools/RSETHPoolV3ExternalBridge.sol  L651-L663
function bridgeAssetsViaNativeBridge() external nonReentrant onlyRole(BRIDGER_ROLE) {
    ...
    uint256 ethBalanceMinusFees = getETHBalanceMinusFees();

    IL2Messenger(messenger).sendETHToL1ViaBridge{ value: ethBalanceMinusFees }(
        l2Bridge, l1VaultETHForL2Chain, ethBalanceMinusFees
    );
    ...
}
```

The pool sends its entire spendable ETH balance to the `IL2Messenger`. If the messenger refunds any ETH to `msg.sender` (the pool) — due to rounding, unused gas stipend, or protocol-level excess — the call reverts because the pool has no `receive()`.

**2. `RSETHPoolV3ExternalBridge.bridgeTokens()` and `RSETHPoolV3WithNativeChainBridge.bridgeTokens()`**

```solidity
// contracts/pools/RSETHPoolV3ExternalBridge.sol  L715-L739
function bridgeTokens(address token) external payable nonReentrant onlySupportedToken(token) onlyRole(BRIDGER_ROLE) {
    ...
    IL2TokenBridge(tokenBridge[token]).bridgeTokenToL1{ value: msg.value }(l1VaultETHForL2Chain, balance);
    ...
}
```

```solidity
// contracts/pools/RSETHPoolV3WithNativeChainBridge.sol  L553-L577
function bridgeTokens(address token, uint256 amount) external payable nonReentrant onlySupportedToken(token) onlyRole(BRIDGER_ROLE) {
    ...
    IL2TokenBridge(tokenBridge[token]).bridgeTokenToL1{ value: msg.value }(l1VaultETHForL2Chain, amount);
    ...
}
```

`msg.value` is forwarded as a bridging fee. Many L2 token bridges (e.g., Optimism, Base, Arbitrum native bridges) refund excess fee ETH to the immediate caller. Since the pool is the caller and has no `receive()`, any such refund causes a revert.

**Contrast with `bridgeAssets()` (not affected):**

```solidity
// contracts/pools/RSETHPoolV3ExternalBridge.sol  L705-L706
stargatePool.send{ value: nativeFee + amount }(sendParam, fee, msg.sender);
```

Here the third argument (`msg.sender` = the bridger EOA) is the explicit Stargate refund address, so the pool is never the refund target. This path is safe.

---

### Impact Explanation

When `bridgeAssetsViaNativeBridge()` or `bridgeTokens()` reverts:

- ETH and LST tokens deposited by users accumulate in the L2 pool and **cannot be forwarded to L1** for restaking via the intended path.
- The pool's `moveAssetsForBridging()` escape hatch exists but requires the bridger to manually extract funds and bridge them through an out-of-band mechanism, breaking the protocol's automated bridging flow.
- Until the contract is upgraded, every attempt to use the native-bridge path will revert if the bridge issues any ETH refund, constituting a **temporary (potentially extended) freeze of user funds** in the L2 pool.

**Impact: Medium — Temporary freezing of funds.**

---

### Likelihood Explanation

Native L2 bridges (Optimism, Base, Arbitrum) commonly refund excess ETH fee to the immediate caller when the provided `msg.value` exceeds the actual bridge fee consumed. Because the bridger supplies `msg.value` as a fee estimate, any over-estimate triggers a refund to the pool. This is a well-documented behavior of standard L2 bridge contracts and is not an edge case.

**Likelihood: Medium** — occurs whenever the bridger provides any `msg.value` that is not consumed exactly by the bridge.

---

### Recommendation

Add a `receive()` function to `RSETHPoolV3ExternalBridge` and `RSETHPoolV3WithNativeChainBridge` so that ETH refunds from external bridge contracts are accepted rather than causing a revert:

```solidity
/// @dev Accept ETH refunds from external bridge/messenger contracts
receive() external payable { }
```

This mirrors the fix applied in `L1Vault` and `L1VaultV2`, both of which correctly define `receive() external payable { }` for the same reason.

---

### Proof of Concept

1. Deploy `RSETHPoolV3ExternalBridge` on a testnet with a mock `IL2TokenBridge` that refunds 1 wei of excess fee to `msg.sender`.
2. Grant `BRIDGER_ROLE` to an EOA and call `bridgeTokens(token)` with `msg.value = 0.01 ether`.
3. The mock bridge processes the token transfer and calls `(bool ok,) = msg.sender.call{value: 1}("")` to refund 1 wei.
4. The call fails because `RSETHPoolV3ExternalBridge` has no `receive()` function.
5. The entire `bridgeTokens()` transaction reverts; tokens remain stuck in the pool.
6. Repeat for `bridgeAssetsViaNativeBridge()` with a mock `IL2Messenger` that refunds any ETH.

            revert ZeroBridgeAmount();

        address token,
        uint256 amount
    )
        external
        payable
        nonReentrant
        onlySupportedToken(token)
        onlyRole(BRIDGER_ROLE)
    {
        if (tokenBridge[token] == address(0)) revert MissingBridgeForToken();
        if (amount == 0) revert InvalidAmount();

        // bridge up to the token balance minus fees
        uint256 tokenBalanceMinusFees = getTokenBalanceMinusFees(token);
        if (amount > tokenBalanceMinusFees) revert InsufficientBalance();

        // Approve the required amount to the bridge
        IERC20(token).safeIncreaseAllowance(tokenBridge[token], amount);

        // Call the bridge contract to transfer the tokens
        IL2TokenBridge(tokenBridge[token]).bridgeTokenToL1{ value: msg.value }(l1VaultETHForL2Chain, amount);

        emit BridgedTokenToL1(token, l1VaultETHForL2Chain, amount);
    }
