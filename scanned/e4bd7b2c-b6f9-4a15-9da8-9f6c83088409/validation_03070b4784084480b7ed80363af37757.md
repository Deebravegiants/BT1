### Title
Unbounded `supportedAssets` Loop in `LRTOracle._getTotalEthInProtocol()` Causes DoS on Price Updates - (File: contracts/LRTOracle.sol)

### Summary
`LRTOracle._getTotalEthInProtocol()` iterates over the `supportedAssets` array with no maximum length cap. This function is invoked by the public `updateRSETHPrice()` and the manager-only `updateRSETHPriceAsManager()`. As the protocol adds more supported assets over time via `LRTConfig.addNewSupportedAsset()`, the loop grows unboundedly. If the array becomes large enough, every call to `_updateRsETHPrice()` will run out of gas, permanently freezing the rsETH price oracle and blocking fee minting.

### Finding Description

`LRTOracle._getTotalEthInProtocol()` iterates over every entry in `supportedAssets`: [1](#0-0) 

For each asset it calls `ILRTDepositPool.getTotalAssetDeposits(asset)`, which itself calls `getAssetDistributionData()`: [2](#0-1) 

This creates a nested loop: `O(supportedAssets.length × nodeDelegatorQueue.length)`. While `nodeDelegatorQueue` is bounded by `maxNodeDelegatorLimit` (initialized to 10): [3](#0-2) 

`supportedAssets` has **no maximum length check** in `LRTConfig.addNewSupportedAsset()`: [4](#0-3) 

`_getTotalEthInProtocol()` is called by `_updateRsETHPrice()`, which is invoked by both the public `updateRSETHPrice()` and `updateRSETHPriceAsManager()`: [5](#0-4) 

### Impact Explanation

If `supportedAssets` grows large enough that `_getTotalEthInProtocol()` exhausts the block gas limit, **no caller — including the manager — can update the rsETH price**. Consequences:

- `rsETHPrice` becomes permanently stale, breaking the exchange rate used by `depositETH`, `depositAsset`, `initiateWithdrawal`, and `instantWithdrawal`.
- Protocol fee minting via `_checkAndUpdateDailyFeeMintLimit` is blocked.
- The price-deviation circuit breaker (pause-on-drop logic) inside `_updateRsETHPrice()` can no longer trigger, removing a key safety mechanism.

**Impact class**: Unbounded gas consumption → effective DoS of the price oracle → temporary (potentially permanent) freezing of protocol functionality.

### Likelihood Explanation

`addNewSupportedAsset()` is gated by `TIME_LOCK_ROLE` and requires a timelock, so growth is slow and deliberate. However, there is **no hard cap** preventing the array from reaching a gas-exhausting size. As the protocol matures and integrates more LSTs/assets, the array grows monotonically. Once the threshold is crossed, recovery requires a contract upgrade — there is no in-protocol mechanism to shrink `supportedAssets` without removing assets that may still hold user funds.

### Recommendation

Add a maximum supported-asset count check in `LRTConfig._addNewSupportedAsset()`:

```solidity
uint256 public constant MAX_SUPPORTED_ASSETS = 20; // tune to safe gas budget

function _addNewSupportedAsset(address asset, uint256 depositLimit) private {
    if (supportedAssetList.length >= MAX_SUPPORTED_ASSETS) {
        revert MaxSupportedAssetsReached();
    }
    // ... existing logic
}
```

Additionally, consider caching `supportedAssets` length off-chain and monitoring gas consumption of `updateRSETHPrice()` as the asset list grows.

### Proof of Concept

1. Admin (via `TIME_LOCK_ROLE`) calls `LRTConfig.addNewSupportedAsset()` repeatedly, growing `supportedAssets` to N entries.
2. Each entry causes `_getTotalEthInProtocol()` to call `getTotalAssetDeposits()`, which itself loops over up to `maxNodeDelegatorLimit` (10) NDCs — making the total work O(N × 10).
3. At sufficiently large N (exact threshold depends on per-iteration gas cost of external calls to `getAssetPrice`, `balanceOf`, `getAssetBalance`, `getAssetUnstaking`, `getEffectivePodShares`), any call to `updateRSETHPrice()` or `updateRSETHPriceAsManager()` reverts with out-of-gas.
4. `rsETHPrice` is frozen at its last stored value; all deposit/withdrawal pricing and fee minting halt. [6](#0-5) [7](#0-6) [8](#0-7)

    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
