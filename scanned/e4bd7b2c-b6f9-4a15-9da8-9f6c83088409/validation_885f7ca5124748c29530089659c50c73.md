### Title
Rounding to Zero in Pool Token Deposit Calculation Causes Depositor to Lose Tokens Without Receiving wrsETH - (File: contracts/pools/RSETHPoolV3.sol)

### Summary
The `viewSwapRsETHAmountAndFee` function in the pool contracts computes the wrsETH amount to mint for a token deposit using integer division. When the deposit amount is small relative to the rate ratio, the result truncates to zero. The pool contracts transfer the user's tokens in before computing the mint amount, and there is no zero-amount guard on the result, so a depositor can permanently lose their tokens while receiving nothing.

### Finding Description
In `RSETHPoolV3.viewSwapRsETHAmountAndFee(uint256 amount, address token)`:

```solidity
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;
uint256 rsETHToETHrate = getRate();                                    // >= 1e18
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate(); // validated 1e16–1e19
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

When `amountAfterFee * tokenToETHRate < rsETHToETHrate`, Solidity integer division yields `rsETHAmount = 0`. The calling `deposit` function transfers the user's tokens first, then mints `rsETHAmount` (which is 0) with no guard:

```solidity
IERC20(token).safeTransferFrom(msg.sender, address(this), amount); // tokens taken
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
feeEarnedInToken[token] += fee;
wrsETH.mint(msg.sender, rsETHAmount);  // mints 0 — no zero-check
```

The only input validation is `if (amount == 0) revert InvalidAmount()`, which does not prevent a non-zero amount from producing a zero output.

The oracle validation in `updatePriceOracleForValidated` constrains `tokenToETHRate` to `[1e16, 1e19]` and `rsETHToETHrate >= 1e18`. In the worst-case ratio (`tokenToETHRate = 1e16`, `rsETHToETHrate = 1e18`), any deposit where `amountAfterFee < 100` yields `rsETHAmount = 0`. The same pattern exists identically in `RSETHPool`, `RSETHPoolNoWrapper`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPoolV2`, `RSETHPoolV2ExternalBridge`, and `RSETHPoolV2NBA`.

### Impact Explanation
A depositor who sends a small token amount (e.g., 1–99 wei of a token whose oracle rate is at the minimum 1e16) has their tokens permanently transferred into the pool contract while receiving zero wrsETH in return. The tokens are not recoverable by the user; they silently inflate the pool's token balance. This matches the analog: a user participates in the protocol, their assets are consumed, and they receive nothing due to precision truncation.

**Impact: Low — Contract fails to deliver promised returns; depositor loses the deposited token value without receiving the corresponding wrsETH.**

### Likelihood Explanation
Any external caller can invoke `deposit(address token, uint256 amount, string referralId)` with a non-zero but small `amount`. No role or whitelist is required. The condition is triggered whenever `amount * tokenToETHRate < rsETHToETHrate`, which is achievable with small amounts of any supported low-rate token. The likelihood is low in practice because the affected amounts are tiny, but the path is fully permissionless and requires no special conditions beyond a small deposit.

### Recommendation
Add a zero-amount guard on the computed `rsETHAmount` before transferring tokens or after computing the swap output:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
if (rsETHAmount == 0) revert InvalidAmount();
```

Alternatively, enforce a minimum deposit amount per token that guarantees `rsETHAmount >= 1` given the current rate, similar to how `LRTDepositPool` uses `minAmountToDeposit` and `minRSETHAmountExpected`.

### Proof of Concept
Assume:
- `feeBps = 0` (no fee)
- `tokenToETHRate = 1e16` (minimum allowed by oracle validation)
- `rsETHToETHrate = 1e18` (rsETH at 1 ETH, minimum)
- User calls `deposit(token, 50, "")` (50 wei of the token)

Execution:
1. `fee = 50 * 0 / 10_000 = 0`
2. `amountAfterFee = 50`
3. `rsETHAmount = 50 * 1e16 / 1e18 = 5e17 / 1e18 = 0` (integer truncation)
4. `IERC20(token).safeTransferFrom(msg.sender, address(this), 50)` — 50 wei transferred out of user
5. `wrsETH.mint(msg.sender, 0)` — user receives nothing

The user has lost 50 wei of the token with no recourse. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
