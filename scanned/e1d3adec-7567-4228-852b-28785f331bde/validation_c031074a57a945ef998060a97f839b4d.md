### Title
`LRTConverter.ethValueInWithdrawal` Inflated After Lido Slashing Event Causes Protocol Insolvency via Inflated rsETH Price - (File: `contracts/LRTConverter.sol`)

---

### Summary

`LRTConverter.ethValueInWithdrawal` is incremented by the ETH value of stETH at the time of transfer to the converter, but decremented only by the actual ETH received when the Lido withdrawal is claimed. During a Lido slashing event, the claimed ETH is less than the tracked value, leaving a permanent phantom balance in `ethValueInWithdrawal`. This phantom value propagates through `LRTDepositPool.getETHDistributionData()` → `LRTOracle._getTotalEthInProtocol()` → `rsETHPrice`, inflating the rsETH price. Users who initiate withdrawals at the inflated price receive more than their proportional share, draining the protocol and causing insolvency for later withdrawers.

---

### Finding Description

**Root cause — `LRTConverter.sol`**

When stETH is moved from the deposit pool to the converter in preparation for a Lido unstake, `ethValueInWithdrawal` is incremented by the oracle-priced ETH value of the stETH:

```solidity
// contracts/LRTConverter.sol:140
ethValueInWithdrawal += (_amount * lrtOracle.getAssetPrice(_asset)) / 1e18;
``` [1](#0-0) 

The operator then calls `unstakeStEth()` to submit a Lido withdrawal request for that stETH amount. When the claim is later finalized, `claimStEth()` calls `_sendEthToDepositPool(address(this).balance)`, which decrements `ethValueInWithdrawal` by the actual ETH received:

```solidity
// contracts/LRTConverter.sol:255-259
if (ethValueInWithdrawal > _amount) {
    ethValueInWithdrawal -= _amount;
} else {
    ethValueInWithdrawal = 0;
}
``` [2](#0-1) 

If a Lido slashing event occurs between the withdrawal request and the claim, Lido's own FAQ confirms the claimable amount can be reduced below the requested amount. The decrement uses the actual (reduced) ETH received, leaving a phantom residual in `ethValueInWithdrawal` equal to `(stETH_amount × stETH_price / 1e18) − actual_ETH_received`.

**Propagation — `LRTDepositPool.sol`**

`getETHDistributionData()` reads `ethValueInWithdrawal` directly as `ethLyingInConverter`:

```solidity
// contracts/LRTDepositPool.sol:498-499
address lrtConverter = lrtConfig.getContract(LRTConstants.LRT_CONVERTER);
ethLyingInConverter = ILRTConverter(lrtConverter).ethValueInWithdrawal();
``` [3](#0-2) 

This feeds into `getTotalAssetDeposits(ETH_TOKEN)`:

```solidity
// contracts/LRTDepositPool.sol:385-396
return (assetLyingInDepositPool + assetLyingInNDCs + effectiveAssetWithEigenLayer + assetLyingInConverter
        + assetLyingUnstakingVault);
``` [4](#0-3) 

**Propagation — `LRTOracle.sol`**

`_getTotalEthInProtocol()` calls `getTotalAssetDeposits()` for every supported asset, including ETH, and uses the result to compute `rsETHPrice`:

```solidity
// contracts/LRTOracle.sol:341-343
uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
``` [5](#0-4) 

```solidity
// contracts/LRTOracle.sol:250
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
``` [6](#0-5) 

**Impact — `LRTWithdrawalManager.sol`**

`getExpectedAssetAmount()` uses the stored `rsETHPrice` to compute how much asset a user receives per rsETH burned:

```solidity
// contracts/LRTWithdrawalManager.sol:593
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
``` [7](#0-6) 

This inflated `expectedAssetAmount` is locked in at `initiateWithdrawal()` time and paid out at `completeWithdrawal()` time. Early withdrawers receive more than their proportional share of the actual protocol assets.

---

### Impact Explanation

**Protocol insolvency (Critical).**

Concrete scenario:
- Protocol holds 600 stETH (≈ 600 ETH). Alice, Bob, Carol each own 1/3.
- Operator calls `transferAssetFromDepositPool(stETH, 200)` → `ethValueInWithdrawal = 200 ETH`. Total tracked ETH = 600.
- Operator calls `unstakeStEth(200)` — 200 stETH enters Lido withdrawal queue.
- Lido slashing event: all stETH rebases down 10%. Remaining stETH in converter = 0 (already in queue). Remaining stETH in protocol = 400 × 0.9 = 360 stETH. Lido will pay out 200 × 0.9 = 180 ETH.
- **Before claim**: `ethValueInWithdrawal` is still 200. `_getTotalEthInProtocol()` = 360 + 200 = 560 ETH. Actual protocol value = 360 + 180 = 540 ETH.
- `updateRSETHPrice()` is called → `rsETHPrice` reflects 560 ETH / total_supply (inflated).
- Alice calls `initiateWithdrawal()` → `expectedAssetAmount = 1/3 × 560 ≈ 186.67 ETH` (should be 180 ETH).
- Alice completes withdrawal and receives 186.67 ETH.
- Remaining actual assets: 540 − 186.67 = 353.33 ETH for Bob and Carol, but they are owed 360 ETH. The last user cannot be made whole.

---

### Likelihood Explanation

**Low-to-medium.** Lido slashing events are rare but have occurred historically. The protocol actively uses `LRTConverter` to convert stETH to ETH via Lido's withdrawal queue (the `unstakeStEth` / `claimStEth` path is a live operational flow). The window of vulnerability is the period between the withdrawal request and the claim, which can span days to weeks depending on Lido queue depth. No attacker action is required to trigger the slashing; the attacker only needs to observe the slashing event and call `initiateWithdrawal()` before the oracle price is corrected.

---

### Recommendation

In `_sendEthToDepositPool()`, instead of decrementing `ethValueInWithdrawal` by the actual ETH received, reset it to zero (or track per-request amounts and zero out the specific request's tracked value) when a claim is completed. Alternatively, track the stETH amount sent to the Lido queue separately and, upon claim, set `ethValueInWithdrawal` to zero for that request regardless of the received amount — accepting that slashing losses are reflected immediately in the rsETH price rather than being hidden as phantom assets. The protocol should also consider calling `updateRSETHPrice()` immediately after `claimStEth()` so the price oracle reflects the actual post-slashing TVL before any withdrawals can be processed at the stale inflated price.

---

### Proof of Concept

The vulnerability is traceable through four on-chain state transitions:

1. **`LRTConverter.transferAssetFromDepositPool(stETH, X)`**
   - `ethValueInWithdrawal += X * getAssetPrice(stETH) / 1e18`
   - [8](#0-7) 

2. **`LRTConverter.unstakeStEth(X)`** → Lido withdrawal request for X stETH submitted.
   - [9](#0-8) 

3. **Lido slashing event** → Lido will pay out Y < X ETH.

4. **`LRTConverter.claimStEth(requestId, hint)`** → `_sendEthToDepositPool(Y)` → `ethValueInWithdrawal -= Y`
   - Phantom residual = `X * stETH_price / 1e18 − Y` remains in `ethValueInWithdrawal`.
   - [10](#0-9) 

5. **`LRTOracle.updateRSETHPrice()`** → `_getTotalEthInProtocol()` reads the inflated `ethValueInWithdrawal` via `getETHDistributionData()` → `rsETHPrice` is set above actual value.
   - [3](#0-2) 
   - [6](#0-5) 

6. **`LRTWithdrawalManager.initiateWithdrawal(asset, rsETHAmount)`** → `getExpectedAssetAmount()` uses inflated `rsETHPrice` → user locks in an over-valued `expectedAssetAmount`.
   - [11](#0-10) 
   - [12](#0-11) 

7. **`LRTWithdrawalManager.completeWithdrawal()`** → user receives the over-valued amount, draining assets from remaining holders.
   - [13](#0-12)

    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)

            // totalAssetAmt is in 1e18 precision (standard token decimals)

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
