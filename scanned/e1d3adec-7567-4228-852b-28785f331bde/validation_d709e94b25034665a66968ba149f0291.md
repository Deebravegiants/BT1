### Title
ETH Deposit Limit Check Omits Incoming Amount, Allowing Limit Bypass - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` uses an asymmetric check for ETH vs. LST assets. For LST assets it correctly adds the incoming `amount` to `totalAssetDeposits` before comparing against the cap, but for ETH it omits the incoming amount entirely. This means when `totalAssetDeposits == depositLimitByAsset(ETH)`, the check returns `false` (not exceeded) and the deposit proceeds, pushing the total above the configured cap.

### Finding Description
The deposit limit guard in `_checkIfDepositAmountExceedesCurrentLimit` reads:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← missing + amount
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [1](#0-0) 

For ETH the condition is `totalAssetDeposits > limit` (strict greater-than, no incoming amount). For LST it is `totalAssetDeposits + amount > limit`. The ETH branch therefore allows a deposit whenever `totalAssetDeposits == limit`, because the strict inequality is false. After the deposit completes, `address(this).balance` (and thus `totalAssetDeposits`) exceeds the configured cap by exactly `msg.value`.

The entry point is the public `depositETH` function, which has no role restriction:

```solidity
function depositETH(uint256 minRSETHAmountExpected, string calldata referralId)
    external payable nonReentrant whenNotPaused onlySupportedAsset(LRTConstants.ETH_TOKEN)
``` [2](#0-1)

### Impact Explanation
The `depositLimitByAsset` cap is a safety mechanism to bound protocol exposure to a single asset class. When the cap is exactly met, one additional ETH deposit of arbitrary size can still be accepted in a single transaction, minting rsETH against ETH that exceeds the intended ceiling. This violates the protocol's stated deposit limit invariant. The excess ETH is then forwarded to NodeDelegators and staked in EigenLayer, permanently increasing protocol exposure beyond the configured bound.

Impact: **Low** — the contract fails to deliver its promised deposit limit enforcement for ETH, but no direct theft or fund freeze occurs.

### Likelihood Explanation
The condition `totalAssetDeposits == depositLimit` must hold at the moment of the deposit. This is most likely when an admin sets `depositLimitByAsset` to exactly the current total (e.g., to halt new ETH deposits), or when the total happens to land exactly on the limit after prior deposits. An attacker monitoring the chain can observe this state and immediately submit a large ETH deposit before the limit is raised or the state changes.

### Recommendation
Add the incoming `amount` to the ETH branch, matching the LST branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. Through normal deposits, `getTotalAssetDeposits(ETH_TOKEN)` reaches exactly `1000 ether`.
3. Attacker calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `1000 ether > 1000 ether` → `false` → check passes.
5. `_mintRsETH` mints rsETH for 500 ETH.
6. `getTotalAssetDeposits(ETH_TOKEN)` is now `1500 ether`, 50% above the configured cap.
7. The next call correctly reverts (`1500 > 1000`), but the damage is already done.
