### Title
Stale `InterimRSETHOracle` Rate via Block Stuffing Enables Over-Minting of rsETH Against LST Collateral — (`contracts/pools/oracle/InterimRSETHOracle.sol`)

---

### Summary

`InterimRSETHOracle` stores a manually-set rsETH/ETH rate. An attacker can use block stuffing to delay a pending `setRate()` update, then deposit an LST (e.g., WETH) at the stale, lower rate to receive more rsETH than the deposited collateral backs.

---

### Finding Description

`InterimRSETHOracle.getRate()` returns a storage variable `rate` that is updated exclusively by a `MANAGER_ROLE` holder calling `setRate()`. [1](#0-0) 

`RSETHPoolV3.deposit(token, amount, referralId)` computes the rsETH mint amount as:

```
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate
``` [2](#0-1) 

where `rsETHToETHrate` comes from `InterimRSETHOracle.getRate()` and `tokenToETHRate` comes from the token's oracle (e.g., `WETHOracle` always returns `1e18`). [3](#0-2) 

When rsETH appreciates (true rate rises above the stored stale rate), the manager must submit a `setRate()` transaction. An attacker who observes this pending transaction can fill blocks with high-gas-price transactions to prevent its inclusion. During the stuffed window, the attacker deposits WETH at the stale lower rate, receiving:

```
rsETHAmount_stale  = amountAfterFee * 1e18 / stale_rate   (larger)
rsETHAmount_correct = amountAfterFee * 1e18 / true_rate   (smaller)
```

The excess `rsETHAmount_stale - rsETHAmount_correct` is rsETH minted without backing collateral.

---

### Impact Explanation

The minted rsETH is not fully backed by the deposited WETH. Repeated attacks (bounded only by `dailyMintLimit`) progressively dilute the collateral backing of all rsETH holders. This matches the **Low — Contract fails to deliver promised returns** scope and the **Low — Block stuffing** scope. [4](#0-3) 

---

### Likelihood Explanation

- The `InterimRSETHOracle` is explicitly an interim, manually-operated oracle with no on-chain freshness enforcement or heartbeat check. [5](#0-4) 
- The protocol is deployed on L2 chains (evidenced by the bridging architecture), where block stuffing is significantly cheaper than on Ethereum mainnet.
- The attack window only needs to last long enough for the attacker to land one deposit transaction — a single block of stuffing may suffice if the attacker's deposit is pre-signed and ready.
- No privileged role is required on the attacker's side.

---

### Recommendation

1. Add a staleness check: record `lastUpdatedAt` in `InterimRSETHOracle` and revert in `getRate()` if `block.timestamp - lastUpdatedAt > MAX_STALENESS`.
2. Enforce a maximum rate-change per update (circuit breaker) so a single stale period cannot produce a large discrepancy.
3. Migrate to an on-chain, manipulation-resistant oracle (e.g., Chainlink or a TWAP) as soon as possible, as the contract itself acknowledges this is an interim design. [6](#0-5) 

---

### Proof of Concept

```solidity
// Scenario: true rsETH/ETH rate = 1.05e18, stale stored rate = 1e18
// Attacker stuffs blocks to delay setRate(1.05e18)
// Attacker deposits 100e18 WETH (feeBps = 0 for clarity)

uint256 staleRate   = 1.00e18;
uint256 correctRate = 1.05e18;
uint256 deposit     = 100e18;

uint256 rsETH_stale   = deposit * 1e18 / staleRate;   // = 100e18
uint256 rsETH_correct = deposit * 1e18 / correctRate; // ≈ 95.24e18

// Attacker receives ~4.76e18 rsETH unbacked by collateral
assert(rsETH_stale > rsETH_correct);

// Fuzz: for stale_rate in [1e18, 1.1e18], rsETHAmount monotonically decreases as rate increases
// confirming the invariant violation when rate is stale-low
```

The `dailyMintLimit` caps total daily exposure but does not prevent the attack within that cap. [7](#0-6)
