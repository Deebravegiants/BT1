### Title
Unset Default `pricePercentageLimit` Disables Both Upside and Downside Price Deviation Protections - (File: `contracts/LRTOracle.sol`)

### Summary
`pricePercentageLimit` in `LRTOracle` defaults to `0` after initialization and is never set in `initialize`. Both the upside price-spike guard and the downside auto-pause guard explicitly short-circuit when this value is zero, leaving the protocol with no price deviation protection until an admin explicitly calls `setPricePercentageLimit`. This is the direct Solidity analog of the external report's "unspecified default hash function": a security-critical parameter defaults to a value that silently disables the entire protection mechanism.

### Finding Description
`LRTOracle` declares `pricePercentageLimit` as a plain storage variable with no initialization in `initialize`:

```solidity
uint256 public pricePercentageLimit;   // defaults to 0
``` [1](#0-0) 

The `initialize` function sets only `lrtConfig` and emits an event; it never assigns `pricePercentageLimit`: [2](#0-1) 

Inside `_updateRsETHPrice`, both the upside check and the downside check are guarded by `pricePercentageLimit > 0`:

```solidity
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
``` [3](#0-2) 

```solidity
bool isPriceDecreaseOffLimit =
    pricePercentageLimit > 0 && diff > pricePercentageLimit.mulWad(highestRsethPrice);
``` [4](#0-3) 

When `pricePercentageLimit == 0`, both booleans evaluate to `false` unconditionally. The upside branch never reverts with `PriceAboveDailyThreshold`, and the downside branch never calls `lrtDepositPool.pause()`, `withdrawalManager.pause()`, or `_pause()`. [5](#0-4) 

The setter exists but is never called during initialization: [6](#0-5) 

### Impact Explanation
**Downside (High — theft of unclaimed yield):** When the rsETH price drops significantly (e.g., due to a slashing event reflected in the underlying asset oracles), the protocol is designed to auto-pause deposits and withdrawals to protect existing holders. With `pricePercentageLimit == 0`, this auto-pause never fires. Any user can call the public `updateRSETHPrice()`, push the deflated price on-chain, and immediately deposit at the deflated rate to receive more rsETH than they are entitled to. This dilutes existing rsETH holders, constituting theft of their accrued yield.

**Upside (Medium — temporary freezing of unclaimed yield):** Any oracle-reported price spike passes through without triggering `PriceAboveDailyThreshold`. The new inflated price is committed to storage, and subsequent depositors receive fewer rsETH than they should, with no circuit-breaker to halt the mispricing. [7](#0-6) 

### Likelihood Explanation
`updateRSETHPrice()` is a permissionless public function. Any depositor, withdrawer, or external keeper can call it at any time. The unprotected default state persists from deployment until an admin explicitly calls `setPricePercentageLimit`. During this window — which may span the entire operational lifetime of the contract if the admin omits the call — the price deviation protection is completely absent. The likelihood is **Medium**: the window is real and the call path is trivially reachable by any external actor. [8](#0-7) 

### Recommendation
Set a safe non-zero default for `pricePercentageLimit` directly inside `initialize`, analogous to how the external report recommended setting a secure default hash function:

```solidity
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    lrtConfig = ILRTConfig(lrtConfigAddr);
    pricePercentageLimit = 1e16; // 1% default — admin can adjust via setPricePercentageLimit
    emit UpdatedLRTConfig(lrtConfigAddr);
}
```

Alternatively, require `pricePercentageLimit` to be passed as a constructor/initializer argument and validated as non-zero before the contract is considered operational.

### Proof of Concept
1. Deploy `LRTOracle` and call `initialize`. Observe `pricePercentageLimit == 0`.
2. Arrange for the underlying asset oracle to report a price that would cause rsETH price to drop by, say, 20% (e.g., simulate a slashing event in a fork test).
3. Call `updateRSETHPrice()` as an unprivileged EOA.
4. Observe: `isPriceDecreaseOffLimit` evaluates to `false` (because `pricePercentageLimit == 0`), the auto-pause is never triggered, and `rsETHPrice` is updated to the deflated value.
5. Immediately call `LRTDepositPool.depositETH()` with `msg.value`. Because `rsETHPrice` is now deflated, `getRsETHAmountToMint` returns a larger rsETH amount than the depositor is entitled to at the true price, diluting all existing rsETH holders. [9](#0-8) [10](#0-9)
