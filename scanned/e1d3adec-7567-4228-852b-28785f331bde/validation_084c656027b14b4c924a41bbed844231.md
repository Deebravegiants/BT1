### Title
Permanent Freezing of Unclaimed KERNEL Yield for Contract Recipients That Cannot Self-Execute — (`contracts/KERNEL/KernelMerkleDistributor.sol`)

---

### Summary

`KernelMerkleDistributor._processClaim` enforces `account == msg.sender` unconditionally. Any contract address included in the merkle tree that lacks a self-execute mechanism (no `execute`, `call`, or equivalent function) can never satisfy this check, and there is no admin rescue path in this contract. The yield allocated to such addresses is permanently frozen.

---

### Finding Description

`_processClaim` contains a hard gate at lines 311–313:

```solidity
if (account != msg.sender) {
    revert Unauthorized();
}
``` [1](#0-0) 

Both public entry points — `claim` and `claimAndStake` — route through this function with no bypass: [2](#0-1) [3](#0-2) 

The check requires the transaction originator (`msg.sender`) to be exactly the beneficiary address (`account`). For an EOA this is trivially satisfiable. For a smart contract wallet that can dispatch arbitrary calls (e.g., Gnosis Safe, ERC-4337 account), the contract itself becomes `msg.sender` when it calls `claim`, so the check also passes.

However, for any contract address that **cannot initiate external calls** — a pure receiver, a vault with no `execute` function, a contract whose only callable functions are internal to its own logic — there is no code path that makes `msg.sender == account`. Every call to `claim` or `claimAndStake` for such an address reverts with `Unauthorized`, regardless of who calls it.

Critically, unlike `KernelTop100MerkleDistributor`, which has a `withdrawTokens` admin escape hatch: [4](#0-3) 

`KernelMerkleDistributor` has **no equivalent recovery function**. The owner cannot rescue tokens allocated to a frozen address. The KERNEL tokens remain locked in the distributor contract indefinitely. [5](#0-4) 

---

### Impact Explanation

Any KERNEL yield allocated in the merkle tree to a contract address that cannot self-execute is permanently unclaimable. The tokens remain in the distributor with no admin rescue path. This matches the allowed impact: **Medium — Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

The merkle tree is constructed off-chain. Protocol participants who interact via contract addresses (e.g., a DAO treasury, a simple escrow, a yield aggregator vault with no generic `execute`) are realistic recipients. The protocol has no on-chain mechanism to detect or prevent such addresses from being included. Once included and the distribution round advances, the freeze is irreversible.

---

### Recommendation

Two complementary fixes:

1. **Remove or relax the `msg.sender` check.** The merkle proof already cryptographically binds `(index, account, cumulativeAmount)` to the tree root. A third-party caller cannot redirect funds — `kernel.safeTransfer(account, amountToSend)` always sends to the proven `account`. The `account != msg.sender` guard provides no additional security and should be removed.

2. **Add an owner-controlled token rescue function** (mirroring `KernelTop100MerkleDistributor.withdrawTokens`) as a safety net for any future edge cases.
