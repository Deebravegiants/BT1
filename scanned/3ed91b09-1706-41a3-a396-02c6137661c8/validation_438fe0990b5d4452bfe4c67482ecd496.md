Looking at the key code:

### Title
Exact-equality CCIP fee check blocks bridging whenever on-chain fee differs from off-chain quote — (`contracts/L1VaultV2.sol`)

---

### Summary

`bridgeRsETHToL2UsingCCIP` recomputes the CCIP fee on-chain at execution time and requires `msg.value` to match it exactly. Because the MANAGER_ROLE must quote the fee off-chain before submitting the transaction, any block-to-block fee movement causes every bridge attempt to revert with `IncorrectCCIPFee`, temporarily preventing rsETH from being bridged to L2.

---

### Finding Description

Inside `bridgeRsETHToL2UsingCCIP`, the fee is computed on-chain by calling `getCCIPFee(amount)` (which delegates to `ccipRouter.getFee`), and then enforced with a strict equality check:

```solidity
uint256 fee = getCCIPFee(amount);   // on-chain, at execution block
if (msg.value != fee) {
    revert IncorrectCCIPFee();
}
``` [1](#0-0) 

The MANAGER_ROLE must predict this value off-chain before submitting the transaction. CCIP fees are gas-price-sensitive and can change between the block where the quote is read and the block where the transaction is mined. Even a 1-wei difference causes a revert.

Critically, the underlying CCIP router explicitly accepts overpayment:

> *"Note if msg.value is larger than the required fee (from getFee) we accept the overpayment with no refund."* [2](#0-1) 

The contract's own guard is therefore **stricter than the protocol it wraps**, and it blocks the one mitigation a caller would naturally use (sending a small buffer above the quoted fee).

---

### Impact Explanation

While the fee is volatile, every call to `bridgeRsETHToL2UsingCCIP` reverts. rsETH accumulates in `L1VaultV2` and cannot be moved to L2 until the manager happens to submit a transaction in the same block where the fee matches exactly. This constitutes a **temporary freezing of funds** (rsETH stranded in L1VaultV2, unable to be bridged to L2). [3](#0-2) 

---

### Likelihood Explanation

CCIP fees are computed from destination-chain gas prices and token prices, both of which fluctuate block-to-block. During periods of gas volatility (e.g., network congestion, large price swings), the fee returned by `getFee` at block N will routinely differ from the fee at block N+k where k ≥ 1. The MANAGER_ROLE has no atomic quote-and-send path available in the contract, so this condition is regularly reachable in normal operation.

---

### Recommendation

Replace the exact-equality check with a minimum-fee check, and refund any excess ETH to the caller:

```solidity
uint256 fee = getCCIPFee(amount);
if (msg.value < fee) {
    revert IncorrectCCIPFee();
}

// ... ccipSend ...

uint256 excess = msg.value - fee;
if (excess > 0) {
    (bool ok, ) = msg.sender.call{value: excess}("");
    require(ok, "ETH refund failed");
}
```

This aligns with the CCIP router's own overpayment policy and allows the manager to send a small buffer without causing a revert.

import "forge-std/Test.sol";
import "../contracts/L1VaultV2.sol";

contract MockCCIPRouter {
    uint256 public feeToReturn;

    function setFee(uint256 f) external { feeToReturn = f; }

    function getFee(uint64, Client.EVM2AnyMessage memory)
        external view returns (uint256) { return feeToReturn; }

    function ccipSend(uint64, Client.EVM2AnyMessage calldata)
        external payable returns (bytes32) { return bytes32(0); }
}

contract CCIPFeeDoSTest is Test {
    L1VaultV2 vault;
    MockCCIPRouter router;
    address manager = address(0xBEEF);

    function setUp() public {
        // deploy & initialize vault (abbreviated)
        router = new MockCCIPRouter();
        // ... grant MANAGER_ROLE to manager, set bridgeType = CCIP, etc.
    }

    function test_feeFluctuationCausesRevert() public {
        uint256 amount = 1 ether;

        // Step 1: off-chain quote at block N
        router.setFee(1e15);                    // fee = 1e15 wei
        uint256 quotedFee = vault.getCCIPFee(amount); // returns 1e15

        // Step 2: fee changes by 1 wei before tx lands
        router.setFee(1e15 + 1);               // fee = 1e15 + 1 wei at block N+k

        // Step 3: manager submits with the quoted fee — reverts
        vm.prank(manager);
        vm.expectRevert(L1VaultV2.IncorrectCCIPFee.selector);
        vault.bridgeRsETHToL2UsingCCIP{value: quotedFee}(amount);
    }
}
```

The test demonstrates that a 1-wei increase in the CCIP fee between the off-chain quote and on-chain execution is sufficient to revert the bridge call, leaving rsETH stranded in `L1VaultV2`. [1](#0-0) [4](#0-3)

    function bridgeRsETHToL2UsingCCIP(uint256 amount) external payable nonReentrant onlyRole(MANAGER_ROLE) {
        if (bridgeType != BridgeType.CCIP) {
            revert InactiveBridgeType();
        }

        if (rsETH.balanceOf(address(this)) < amount) {
            revert InsufficientRsETHBalance();
        }

        if (amount == 0) {
            revert ZeroAmount();
        }

        uint256 fee = getCCIPFee(amount);

        if (msg.value != fee) {
            revert IncorrectCCIPFee();
        }

        IERC20(address(rsETH)).safeIncreaseAllowance(address(ccipRouter), amount);

        Client.EVM2AnyMessage memory message = getCCIPMessage(amount);

        bytes32 messageId = ccipRouter.ccipSend{ value: msg.value }(destinationChainSelector, message);

        emit BridgedRsETHToL2UsingCCIP(destinationChainSelector, l2Receiver, amount, messageId);
    }
