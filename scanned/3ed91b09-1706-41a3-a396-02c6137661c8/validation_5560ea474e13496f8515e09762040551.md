### Title
ETH Deposit Limit Check Ignores Deposit Amount, Allowing Limit Overshoot - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` uses a strict `>` comparison for ETH that omits the incoming deposit `amount`, while the ERC20 branch correctly includes `amount` in the comparison. This causes the deposit limit enforcement to never properly execute for ETH: a depositor can push total ETH deposits above the configured limit by one deposit.

### Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit`, the ETH branch returns `totalAssetDeposits > depositLimit`, ignoring the `amount` parameter entirely:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount unused
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct
}
``` [1](#0-0) 

The function is called from `_beforeDeposit` with the user-supplied `depositAmount`: [2](#0-1) 

And `_beforeDeposit` is invoked by the public `depositETH`: [3](#0-2) 

**Analog to the reported bug:** Just as `treeRootHash` was computed but never assigned to `storageRoot` (causing the downstream `if (storageRoot != nil)` guard to never trigger), here `amount` is passed to the function but never used in the ETH branch of the comparison — causing the downstream `revert MaximumDepositLimitReached()` guard to never trigger when it should.

**Concrete scenario:**
- `depositLimitByAsset(ETH) = 1000 ETH`, `totalAssetDeposits = 1000 ETH`
- ETH check: `1000 > 1000` → `false` → deposit allowed
- User deposits 500 ETH → `totalAssetDeposits` becomes 1500 ETH, exceeding the limit
- ERC20 check under same conditions: `1000 + 500 > 1000` → `true` → deposit blocked ✓

### Impact Explanation
The deposit limit is a risk-management control that caps protocol exposure to EigenLayer strategies. Because the ETH branch ignores the incoming `amount`, any depositor can push total ETH deposits above the configured limit by one transaction. The excess is bounded only by the size of a single deposit. This constitutes the protocol failing to deliver its promised deposit-cap guarantee without any loss of user funds.

**Impact: Low — Contract fails to deliver promised returns, but doesn't lose value.**

### Likelihood Explanation
The entry path is the public, permissionless `depositETH` function. No special role is required. The condition is triggered whenever `totalAssetDeposits` is at or near the configured limit, which is a normal operational state. Any ETH depositor can exploit this.

### Recommendation
Add `amount` to the ETH branch of the comparison, matching the ERC20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [4](#0-3) 

### Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. Protocol accumulates `totalAssetDeposits(ETH) = 1000 ether` through normal usage.
3. Attacker calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500 ether)` evaluates `1000 > 1000` → `false`.
5. Deposit succeeds; `totalAssetDeposits` becomes 1500 ether — 50% above the configured limit.
6. The same call with any ERC20 at the same threshold would have reverted with `MaximumDepositLimitReached`.
