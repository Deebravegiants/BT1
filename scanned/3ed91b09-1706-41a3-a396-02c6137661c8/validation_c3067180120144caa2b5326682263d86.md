### Title
Multi-Token Over-Collateralization via Per-Token Cap in `maxAmountToDepositBridgerAsset` — (`contracts/L2/RsETHTokenWrapper.sol`)

---

### Summary

`maxAmountToDepositBridgerAsset` enforces a per-token cap of `totalSupply - tokenX.balanceOf(wrapper)`, but does not account for other allowed tokens already held by the wrapper. When two tokens are allowed and one already fully backs the outstanding wrsETH supply, the bridger can deposit up to `totalSupply` of the second token as well, doubling total collateral to `2S` for only `S` wrsETH. The excess deposit has no wrsETH to burn against it and is permanently locked.

---

### Finding Description

`maxAmountToDepositBridgerAsset` is the sole guard on `depositBridgerAssets`: [1](#0-0) 

```solidity
function maxAmountToDepositBridgerAsset(address _asset) public view returns (uint256) {
    if (!allowedTokens[_asset]) return 0;

    uint256 wrsETHSupply = totalSupply();
    uint256 balanceOfAssetInWrapper = ERC20Upgradeable(_asset).balanceOf(address(this));

    if (balanceOfAssetInWrapper > wrsETHSupply) return 0;

    return wrsETHSupply - balanceOfAssetInWrapper;
}
```

The function computes the cap **independently per token**: it subtracts only `_asset`'s own balance from `totalSupply`, ignoring all other allowed tokens already held. With two allowed tokens, the invariant `sum(all_token_balances) <= totalSupply` is never enforced.

`depositBridgerAssets` uses this as its only guard: [2](#0-1) 

`_withdraw` burns wrsETH 1:1 against a single chosen asset: [3](#0-2) 

Once all `S` wrsETH are burned to redeem tokenA, `totalSupply = 0` and the `S` tokenB deposited by the bridger has no wrsETH to burn against it — it is permanently unrecoverable.

---

### Impact Explanation

The `S` tokenB deposited by the bridger is permanently frozen inside the wrapper. There is no recovery path: `_withdraw` requires burning wrsETH, but none exists after users redeem tokenA. No admin function exists to rescue stranded tokens. This matches **Medium — Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

Requires the `BRIDGER_ROLE` to call `depositBridgerAssets(tokenB, S)` when tokenA already fully backs the supply. This can occur:
- Accidentally, if the bridger contract has a routing bug and deposits the wrong allowed token.
- Operationally, if a second token is added via `addAllowedToken` and the bridger deposits it without checking aggregate collateral.

The bridger does not need to be malicious; the contract's own safety check (`maxAmountToDepositBridgerAsset`) silently permits the over-deposit because it is per-token, not aggregate.

---

### Recommendation

Replace the per-token cap with an aggregate cap across all allowed tokens. Track total deposited collateral in a storage variable (incremented in `depositBridgerAssets`, decremented in `_withdraw`), and enforce:

```solidity
return wrsETHSupply - totalBridgerDeposited;
```

Alternatively, enforce the invariant directly:

```solidity
uint256 totalCollateral = sum of balanceOf(wrapper) for all allowed tokens;
return wrsETHSupply > totalCollateral ? wrsETHSupply - totalCollateral : 0;
```

---

### Proof of Concept

```
State: tokenA.balanceOf(wrapper) = S, tokenB.balanceOf(wrapper) = 0, totalSupply = S

1. bridger calls depositBridgerAssets(tokenB, S)
   - maxAmountToDepositBridgerAsset(tokenB) = S - 0 = S  ✓ (no revert)
   - wrapper receives S tokenB
   - State: tokenA = S, tokenB = S, totalSupply = S

2. All S users call withdraw(tokenA, 1) each
   - Each burns 1 wrsETH, receives 1 tokenA
   - After all: tokenA = 0, tokenB = S, totalSupply = 0

3. tokenB is permanently locked:
   - withdraw(tokenB, any) requires burning wrsETH → totalSupply = 0 → reverts
   - No rescue/sweep function exists
   - S tokenB is frozen forever
```

Invariant fuzz assertion that would catch this:

```solidity
assert(tokenA.balanceOf(wrapper) + tokenB.balanceOf(wrapper) <= wrapper.totalSupply());
```
