### Title
Unbounded Nested Loop in `LRTOracle._getTotalEthInProtocol()` Can Permanently Freeze the rsETH Price Oracle — (File: contracts/LRTOracle.sol)

---

### Summary

`LRTOracle._getTotalEthInProtocol()` contains a nested loop that makes O(supportedAssets × nodeDelegators) external calls per invocation. It is called by the public `updateRSETHPrice()` with no gas guard, no pagination, and no upper-bound enforcement. As the protocol adds more supported assets and node delegators, the gas cost of every price-update call grows proportionally. Once it exceeds the block gas limit the function becomes permanently uncallable, freezing the rsETH price oracle, halting fee minting, and disabling the price-drop protection circuit.

---

### Finding Description

`_getTotalEthInProtocol()` iterates over every entry in `supportedAssetList` and, for each asset, calls `ILRTDepositPool.getTotalAssetDeposits(asset)`: [1](#0-0) 

`getTotalAssetDeposits` delegates to `getAssetDistributionData`, which itself loops over the entire `nodeDelegatorQueue` and issues three external calls per NDC per asset: [2](#0-1) 

The ETH path (`getETHDistributionData`) adds another three external calls per NDC: [3](#0-2) 

The result is a nested loop with roughly `3 × assets × NDCs + 3 × NDCs` external calls per `updateRSETHPrice()` invocation. `updateRSETHPrice()` is declared `public` with no access restriction: [4](#0-3) 

`maxNodeDelegatorLimit` is initialized to 10 but is freely raisable by the admin: [5](#0-4) 

New supported assets are added via `addNewSupportedAsset` with no cap on the list length: [6](#0-5) 

There is no gas-limit guard, no pagination, and no maximum on the product `assets × NDCs`. As the protocol legitimately scales, the call graph grows until it hits the 30 M-gas block ceiling, at which point both `updateRSETHPrice()` and `updateRSETHPriceAsManager()` revert on every attempt.

---

### Impact Explanation

When `updateRSETHPrice()` becomes permanently uncallable:

1. **Permanent freezing of unclaimed yield** — the fee-minting path inside `_updateRsETHPrice()` is never reached again; protocol fees accumulate in the TVL but can never be minted to the treasury.
2. **Price-drop protection disabled** — the circuit that pauses `LRTDepositPool` and `LRTWithdrawalManager` on a large price decrease (`isPriceDecreaseOffLimit`) can never fire, leaving the protocol unprotected against slashing events.
3. **Stale rsETH price** — deposits and withdrawals continue to use the last stored `rsETHPrice`, causing share/asset mis-accounting that worsens over time as the true NAV drifts. [7](#0-6) [8](#0-7) 

**Impact rating: Medium** — permanent freezing of unclaimed yield + share/asset mis-accounting.

---

### Likelihood Explanation

The trigger is normal, expected protocol growth: adding more LST assets (each requiring its own oracle and strategy) and increasing `maxNodeDelegatorLimit` to accommodate more restaking capacity. At 10 assets × 20 NDCs the call count is ~600 external calls (~1.3 M gas cold); at 20 assets × 30 NDCs it is ~1,800 calls (~3.8 M gas cold), and overhead compounds this. No extraordinary event is required — only the admin exercising routine governance to scale the protocol. Because there is no safeguard that prevents the product from growing past the block gas limit, the risk is latent and worsens every time a new asset or NDC is added.

**Likelihood rating: Medium.**

---

### Recommendation

1. **Cache the NDC queue once** outside the asset loop so `getNodeDelegatorQueue()` is not re-fetched per asset.
2. **Separate the ETH and LST accounting loops** so each can be called independently and paginated.
3. **Enforce a hard cap** on `maxNodeDelegatorLimit × supportedAssetList.length` in `addNewSupportedAsset` and `updateMaxNodeDelegatorLimit` that keeps the worst-case gas well below the block limit.
4. **Add a gas-remaining guard** at the top of `_getTotalEthInProtocol()` (e.g., `require(gasleft() > MIN_GAS_RESERVE)`) so the function fails fast with a clear error rather than silently reverting mid-loop.

---

### Proof of Concept

1. Admin calls `addNewSupportedAsset` 15 times, bringing `supportedAssetList.length` to 17 (including the two genesis assets).
2. Admin raises `maxNodeDelegatorLimit` to 25 and adds 25 NDCs.
3. Any caller invokes `updateRSETHPrice()`.
4. `_getTotalEthInProtocol()` executes `17 × 25 × 3 = 1,275` cold external calls ≈ 2.7 M gas, plus `25 × 3 = 75` ETH-path calls ≈ 158 k gas, plus loop overhead and storage reads — total approaches or exceeds 30 M gas.
5. The transaction reverts out-of-gas.
6. Every subsequent call to `updateRSETHPrice()` or `updateRSETHPriceAsManager()` also reverts.
7. `rsETHPrice` is permanently frozen at its last stored value; fee minting and price-drop protection are permanently disabled. [9](#0-8) [1](#0-0)

    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
