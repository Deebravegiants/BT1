### Title
Missing `tokenFeeBps` Initialization in `addSupportedToken()` Allows Zero-Fee Token Deposits - (File: contracts/pools/RSETHPool.sol)

---

### Summary

`RSETHPool.sol` maintains a per-token fee mapping `tokenFeeBps` that is never initialized when a new token is added via `addSupportedToken()`. Because Solidity mappings default to zero, any newly added token has an effective fee of 0 bps until the admin separately calls `setTokenFeeBps()`. Any user who deposits that token in the interim window pays no fee, depriving the protocol of its entitled yield.

---

### Finding Description

`RSETHPool.sol` introduces a per-token fee mechanism via the `tokenFeeBps` mapping:

```solidity
mapping(address token => uint256 feeBps) public tokenFeeBps;
```

The fee for a token deposit is computed in `viewSwapRsETHAmountAndFee`:

```solidity
uint256 feeBpsForToken = tokenFeeBps[token];   // reads 0 if never set
fee = amount * feeBpsForToken / 10_000;         // fee == 0
uint256 amountAfterFee = amount - fee;          // full amount passes through
``` [1](#0-0) 

However, `addSupportedToken()` — callable by `TIMELOCK_ROLE` — never writes to `tokenFeeBps[token]`:

```solidity
function addSupportedToken(address token, address oracle, address bridge) external onlyRole(TIMELOCK_ROLE) {
    ...
    supportedTokenList.push(token);
    supportedTokenOracle[token] = oracle;
    tokenBridge[token] = bridge;
    // tokenFeeBps[token] is never initialized → remains 0
    emit AddSupportedToken(token, oracle, bridge);
}
``` [2](#0-1) 

Setting the fee is a separate, independent call restricted to `DEFAULT_ADMIN_ROLE`:

```solidity
function setTokenFeeBps(address token, uint256 _feeBps) external onlyRole(DEFAULT_ADMIN_ROLE) onlySupportedToken(token) {
    if (_feeBps > 10_000) revert InvalidFeeAmount();
    tokenFeeBps[token] = _feeBps;
    emit TokenFeeBpsSet(token, _feeBps);
}
``` [3](#0-2) 

Because `TIMELOCK_ROLE` and `DEFAULT_ADMIN_ROLE` are distinct roles and the two calls are not atomic, there is always a window — potentially spanning multiple blocks — during which the token is live and depositable with zero fees. A bot watching the mempool for `AddSupportedToken` events can front-run or immediately follow the token listing to deposit large amounts at 0 bps.

The deposit path that is exploited:

```solidity
function deposit(address token, uint256 amount, string memory referralId)
    external nonReentrant whenNotPaused onlySupportedToken(token)
{
    if (amount == 0) revert InvalidAmount();
    IERC20(token).safeTransferFrom(msg.sender, address(this), amount);
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
    feeEarnedInToken[token] += fee;   // fee == 0, nothing accrues
    IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount);
    ...
}
``` [4](#0-3) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

The protocol fee (`feeEarnedInToken[token]`) is the yield earned by the pool on token deposits. When `tokenFeeBps[token] == 0`, `fee == 0` and `feeEarnedInToken[token]` never increases. Simultaneously, the depositor receives rsETH computed on the full `amount` rather than `amount - fee`, meaning they receive more rsETH than they are entitled to at the expense of the protocol's fee revenue. This is a direct, quantifiable loss of yield for every deposit made during the zero-fee window.

---

### Likelihood Explanation

**Moderate.** The precondition is that a new token must be listed via `addSupportedToken()`. This is a routine operational action (the contract already has one token listed and the reinitializer pattern shows more are expected). The gap between `addSupportedToken()` and `setTokenFeeBps()` is structurally guaranteed because they are separate transactions by different roles. A mempool-watching bot can exploit this gap with certainty and zero risk, requiring no special privileges.

---

### Recommendation

Initialize `tokenFeeBps[token]` inside `addSupportedToken()` by accepting an initial fee parameter, and enforce a non-zero minimum:

```diff
-function addSupportedToken(address token, address oracle, address bridge) external onlyRole(TIMELOCK_ROLE) {
+function addSupportedToken(address token, address oracle, address bridge, uint256 initialFeeBps) external onlyRole(TIMELOCK_ROLE) {
     UtilLib.checkNonZeroAddress(token);
     UtilLib.checkNonZeroAddress(oracle);
     UtilLib.checkNonZeroAddress(bridge);
+    if (initialFeeBps == 0 || initialFeeBps > 10_000) revert InvalidFeeAmount();
     ...
     supportedTokenList.push(token);
     supportedTokenOracle[token] = oracle;
     tokenBridge[token] = bridge;
+    tokenFeeBps[token] = initialFeeBps;
     emit AddSupportedToken(token, oracle, bridge);
 }
```

This eliminates the zero-fee window entirely and mirrors the pattern used in the external report's recommended fix (enforcing a minimum premium/fee at the point of action, not as a separate step).

---

### Proof of Concept

1. Admin (TIMELOCK_ROLE) calls `addSupportedToken(tokenX, oracleX, bridgeX)`. `tokenFeeBps[tokenX]` is 0.
2. Attacker observes the transaction (on-chain or in mempool) and immediately calls:
   ```solidity
   deposit(tokenX, largeAmount, "");
   ```
3. Inside `viewSwapRsETHAmountAndFee(largeAmount, tokenX)`:
   - `feeBpsForToken = tokenFeeBps[tokenX]` → **0**
   - `fee = largeAmount * 0 / 10_000` → **0**
   - `rsETHAmount = largeAmount * tokenToETHRate / rsETHToETHrate` (full amount, no deduction)
4. `feeEarnedInToken[tokenX] += 0` — protocol earns nothing.
5. Attacker receives rsETH equivalent to the full deposit with zero fee paid.
6. Admin (DEFAULT_ADMIN_ROLE) later calls `setTokenFeeBps(tokenX, 100)` — but the attacker has already deposited at 0 bps. [5](#0-4) [2](#0-1)

    function viewSwapRsETHAmountAndFee(
        uint256 amount,
        address token
    )
        public
        view
        onlySupportedToken(token)
        returns (uint256 rsETHAmount, uint256 fee)
    {
        uint256 feeBpsForToken = tokenFeeBps[token];
        fee = amount * feeBpsForToken / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // rate of token in ETH
        uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
    }
```
