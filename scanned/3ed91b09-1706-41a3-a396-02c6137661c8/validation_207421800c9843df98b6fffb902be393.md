### Title
Stale `InterimRSETHOracle` Rate Allows Excess wrsETH Minting, Enabling Theft of Unclaimed Yield — (`contracts/pools/oracle/InterimRSETHOracle.sol`, `contracts/pools/RSETHPoolV3.sol`)

---

### Summary

`RSETHPoolV3.deposit(token)` computes the wrsETH amount to mint using the `InterimRSETHOracle` rate, which is manually set by a `MANAGER_ROLE` with no on-chain staleness enforcement. When the oracle rate lags below the true rsETH/ETH rate, depositors receive more wrsETH than their collateral warrants, diluting existing rsETH holders. The cross-chain variant (depositing on a stale-rate chain, bridging wrsETH to L1, redeeming at the true rate) is a concrete, user-accessible path to extract this yield differential.

---

### Finding Description

**`WETHOracle.getRate()`** always returns `1e18` — it is a pure constant. [1](#0-0) 

**`InterimRSETHOracle.getRate()`** returns a manually-set `rate` with no timestamp, no heartbeat, and no staleness guard. The only validation is `rate >= 1e18`. [2](#0-1) 

**`RSETHPoolV3.viewSwapRsETHAmountAndFee(amount, token)`** computes:

```
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate
```

For WETH: `tokenToETHRate = 1e18` (WETHOracle), `rsETHToETHrate = InterimRSETHOracle.rate`. [3](#0-2) 

When `InterimRSETHOracle.rate` is stale-low (e.g., `1.001e18` while the true rate is `1.005e18`), the denominator is smaller than it should be, so `rsETHAmount` is larger than the collateral warrants:

- Stale rate `1.001e18`: `1 WETH → 0.999001 wrsETH`
- True rate `1.005e18`: `1 WETH → 0.995025 wrsETH`
- **Excess minted: ~0.003976 wrsETH per WETH**

This excess is minted via `wrsETH.mint(msg.sender, rsETHAmount)` with no further check. [4](#0-3) 

The `RsETHTokenWrapper` is an OFT-compatible token. A user can bridge wrsETH from L2 to L1 without any role restriction, then call `withdraw(asset, amount)` to burn wrsETH 1:1 for canonical rsETH. [5](#0-4) [6](#0-5) 

The excess wrsETH redeemed on L1 at the true rsETH price extracts value that belongs to existing rsETH holders (their accrued yield is diluted).

---

### Impact Explanation

Every unit of excess wrsETH minted against a stale-low oracle represents yield that was already accrued by existing rsETH holders but is instead captured by the depositor. This is a direct, quantifiable theft of unclaimed yield. The impact scales linearly with deposit size and the magnitude of oracle lag.

---

### Likelihood Explanation

`InterimRSETHOracle` is explicitly described as an interim manual solution. The rsETH/ETH rate increases continuously as staking rewards accrue. Any delay between oracle updates (routine operational lag — no malicious intent required) creates a window where the rate is stale-low. Deploying the same oracle across multiple chains with independent update schedules makes divergent rates a near-certainty during normal operations. No privileged access, no key compromise, and no governance action is needed by the attacker.

---

### Recommendation

1. Add a `lastUpdated` timestamp to `InterimRSETHOracle` and enforce a maximum staleness window (e.g., 24 hours) in `RSETHPoolV3.getRate()`, reverting if the rate is stale.
2. Enforce a cross-chain rate consistency check or use a single authoritative oracle source (e.g., LayerZero DVN-attested rate) rather than independent per-chain manual updates.
3. As a short-term mitigation, add a `maxRate` cap in `viewSwapRsETHAmountAndFee` to bound the rsETH minted per unit of collateral.

### Proof of Concept

```solidity
// Fork two chains: Chain A (stale oracle = 1.001e18), Chain B (true oracle = 1.005e18)
// Chain A setup:
InterimRSETHOracle oracleA = new InterimRSETHOracle(admin, 1.001e18); // stale-low
RSETHPoolV3 poolA = ...; // rsETHOracle = oracleA, WETH supported with WETHOracle

// Chain B setup:
InterimRSETHOracle oracleB = new InterimRSETHOracle(admin, 1.005e18); // true rate
RSETHPoolV3 poolB = ...; // rsETHOracle = oracleB

// Step 1: Attacker deposits 1 WETH on Chain A
uint256 wrsETHReceived = poolA.deposit(WETH, 1e18, "");
// wrsETHReceived ≈ 0.999001e18 (should be 0.995025e18 at true rate)
// Excess: ~0.003976e18 wrsETH

// Step 2: Bridge wrsETH from Chain A to L1 (user-accessible LayerZero OFT bridge)
// wrsETH.sendFrom(attacker, L1_CHAIN_ID, attacker, wrsETHReceived, ...)

// Step 3: On L1, unwrap wrsETH to canonical rsETH 1:1
// RsETHTokenWrapper.withdraw(rsETH, wrsETHReceived) → receive 0.999001 rsETH

// Step 4: rsETH on L1 is worth 1.005 ETH each
// 0.999001 rsETH * 1.005 = ~1.004 ETH
// Profit: ~0.004 ETH per WETH deposited, at expense of existing rsETH holders

// Assert: profit > 0
assert(wrsETHReceived > expectedAtTrueRate); // 0.999001e18 > 0.995025e18 ✓
```

The invariant `rsETH minted ≤ collateral_ETH_value / true_global_rsETH_price` is broken because `InterimRSETHOracle` has no staleness enforcement and `viewSwapRsETHAmountAndFee` uses it without any freshness check. [7](#0-6) [8](#0-7)
