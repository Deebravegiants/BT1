### Title
Last Staker Can Permanently Lock Reward Tokens During Active Distribution Window — (`contracts/KERNEL/KernelDepositPool.sol`)

### Summary

`KernelDepositPool` uses a Synthetix-style staking rewards model. When `totalKernelStaked` drops to zero during an active reward period, the `rewardPerToken()` function freezes accumulation while `updatedAt` is still advanced by the `updateReward` modifier. Any rewards that should have accrued during the zero-staked gap are permanently locked in the contract with no mechanism to recover or redistribute them. A malicious last staker can deliberately trigger this by calling `initiateWithdrawal()` for their full balance immediately after a reward period begins.

### Finding Description

`KernelDepositPool.rewardPerToken()` short-circuits when `totalKernelStaked == 0`:

```solidity
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;   // accumulation frozen
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
``` [1](#0-0) 

However, the `updateReward` modifier always advances `updatedAt` regardless of whether `totalKernelStaked` is zero:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();       // frozen if totalKernelStaked == 0
    updatedAt = lastTimeRewardApplicable();        // always advances
    ...
}
``` [2](#0-1) 

When `initiateWithdrawal()` is called, it decrements `totalKernelStaked` and triggers `updateReward`:

```solidity
function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
    ...
    balanceOf[msg.sender] -= _amount;
    totalKernelStaked -= _amount;
    ...
}
``` [3](#0-2) 

If this call brings `totalKernelStaked` to zero, `updatedAt` is set to the current timestamp. When the next staker eventually joins, the reward calculation uses `(lastTimeRewardApplicable() - updatedAt)` — which now excludes the entire zero-staked gap. The `rewardRate * duration_of_gap` tokens are permanently stranded in the contract.

The contract's own NatSpec acknowledges this but relies on an off-chain operational assumption with no code-level enforcement:

> *"If `totalKernelStaked` ever hits zero during a reward distribution window, any remaining rewards for that period will stay locked in the contract. In this deployment, we're avoiding this issue by ensuring there are always some tokens staked..."* [4](#0-3) 

### Impact Explanation

Reward tokens accrued during the zero-staked window are permanently locked in the contract. No staker — present or future — can ever claim them. This constitutes **permanent freezing of unclaimed yield**. The magnitude scales with `rewardRate × withdrawalDelay`, which can be up to 30 days of rewards.

### Likelihood Explanation

The attack requires the attacker to be the last (or only) staker when a reward period is active. This is achievable:

1. Attacker stakes any nonzero amount of KERNEL before `notifyRewardAmount` is called (required because `notifyRewardAmount` reverts on `totalKernelStaked == 0`).
2. Admin calls `notifyRewardAmount`, starting the reward period.
3. Attacker immediately calls `initiateWithdrawal()` for their full balance → `totalKernelStaked = 0`.
4. `updatedAt` advances; rewards for the withdrawal delay window (up to 30 days) are permanently lost.
5. After the delay, attacker reclaims their KERNEL at no net cost beyond gas.

The attacker's only cost is gas and the opportunity cost of locking KERNEL for the withdrawal delay. The attack is cheap, repeatable each reward period, and requires no special privilege.

### Recommendation

1. **Enforce a minimum stake floor in code**: Revert `initiateWithdrawal` if it would bring `totalKernelStaked` to zero while a reward period is active (`block.timestamp < finishAt`).
2. **Alternatively, do not advance `updatedAt` when `totalKernelStaked == 0`**: Only update `updatedAt` inside `updateReward` when `totalKernelStaked > 0`, so the zero-staked gap is not silently skipped.
3. **Add a recovery function**: Allow the admin to reclaim stranded reward tokens after a reward period ends if they were never distributed.

### Proof of Concept

```
State: rewardRate = 1e18 tokens/sec, duration = 30 days, withdrawalDelay = 30 days

1. Attacker stakes 1 wei KERNEL.
2. Admin calls notifyRewardAmount(X) → finishAt = now + 30 days, updatedAt = now.
3. Attacker calls initiateWithdrawal(1):
   - updateReward fires: rewardPerToken() returns rewardPerTokenStored (totalKernelStaked still 1 at read time, but becomes 0 after)
   - totalKernelStaked = 0
   - updatedAt = block.timestamp (T1)
4. 30 days pass. No one stakes.
5. A new staker stakes 1e18 KERNEL at T1 + 30 days:
   - updateReward fires: rewardPerToken() = rewardPerTokenStored + (rewardRate * (finishAt - updatedAt) * 1e18) / totalKernelStaked
   - updatedAt was T1, finishAt = T1 + 30 days → delta = 30 days
   - But this is the FIRST stake after the gap; the new staker earns all remaining rewards
   - The 30 days of rewards that accrued while totalKernelStaked == 0 are gone — rewardPerTokenStored never increased during that window
   - Those tokens remain locked in the contract forever
```
