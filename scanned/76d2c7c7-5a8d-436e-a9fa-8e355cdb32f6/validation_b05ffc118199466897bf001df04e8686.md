### Title
`burn(address,uint256)` Silently Requires Allowance, Breaking CCIP Burn-and-Mint Pool Integration — (`contracts/ccip/WrappedRSETH.sol`)

### Summary
`WrappedRSETH.burn(address,uint256)` delegates unconditionally to `burnFrom`, which calls OZ `ERC20Burnable.burnFrom` and therefore invokes `_spendAllowance`. The Chainlink reference `BurnMintERC677` that `WrappedRSETH` explicitly models calls `_burn` directly, with no allowance check. Any CCIP pool that calls `burn(account, amount)` without a prior `approve` from `account` to the pool will revert, permanently breaking the bridge burn path.

### Finding Description

`WrappedRSETH` declares itself as implementing `IBurnMintERC20` and cites the Chainlink `BurnMintERC677` as its reference: [1](#0-0) 

The two-argument `burn` is implemented as a pure alias to `burnFrom`: [2](#0-1) 

`burnFrom` passes the `onlyBurner` guard and then calls `super.burnFrom`: [3](#0-2) 

`super.burnFrom` is OZ `ERC20Burnable.burnFrom`, which unconditionally calls `_spendAllowance(account, msg.sender, amount)` before `_burn`: [4](#0-3) 

The Chainlink reference `BurnMintERC677.burn(address,uint256)` is:
```solidity
function burn(address account, uint256 amount)
    public override(IBurnMintERC20, ERC20Burnable) onlyBurner {
    _burn(account, amount);   // no _spendAllowance
}
```

The full call chain for a CCIP pool calling `burn(user, amount)`:

```
pool.lockOrBurn(user, amount)
  → WrappedRSETH.burn(user, amount)          // line 122
    → WrappedRSETH.burnFrom(user, amount)    // line 123
      → ERC20Burnable.burnFrom(user, amount) // super call, line 130
        → _spendAllowance(user, pool, amount) // REVERTS — no allowance set
```

The `IBurnMintERC20` interface carries no allowance requirement in its NatSpec for `burn(address,uint256)`: [5](#0-4) 

### Impact Explanation
A CCIP `BurnFromMintTokenPool` (or any pool that calls `burn(account, amount)` without first receiving a `transferFrom`) will revert on every burn attempt. Tokens locked or escrowed on the source chain cannot be released because the corresponding burn on `WrappedRSETH` always fails. This constitutes **permanent freezing of funds** in the CCIP bridge.

### Likelihood Explanation
The Chainlink CCIP `BurnFromMintTokenPool` pattern calls `burn(originalSender, amount)` directly. `WrappedRSETH` is purpose-built for CCIP (it lives in `contracts/ccip/`, implements `IBurnMintERC20`, and cites `BurnMintERC677` as its model). The mismatch is triggered on every cross-chain transfer that routes through a pool using the burn-from pattern, with no admin action required.

### Recommendation
Override `burn(address,uint256)` to call `_burn` directly (matching the Chainlink reference), bypassing the OZ allowance check, while keeping the `onlyBurner` guard:

```solidity
function burn(address account, uint256 amount)
    public virtual override onlyBurner
{
    _burn(account, amount);
}
```

`burnFrom` should remain as-is for callers that explicitly want allowance-based burning.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import "forge-std/Test.sol";
import "../contracts/ccip/WrappedRSETH.sol";

contract BurnMismatchTest is Test {
    WrappedRSETH token;
    address owner  = address(0xA);
    address pool   = address(0xB);   // simulated CCIP burner
    address user   = address(0xC);

    function setUp() public {
        vm.prank(owner);
        token = new WrappedRSETH("Wrapped rsETH", "wrsETH", 18, 0, owner);

        // Grant pool the burner role (normal deployment step)
        vm.prank(owner);
        token.grantBurnRole(pool);

        // Mint tokens to user (simulates destination-chain wrap or prior mint)
        vm.prank(owner);
        token.grantMintRole(owner);
        vm.prank(owner);
        token.mint(user, 1 ether);
    }

    /// @dev Demonstrates that burn(address,uint256) reverts without allowance,
    ///      unlike the Chainlink BurnMintERC677 reference.
    function test_burnFromAccountRevertsWithoutAllowance() public {
        // Pool calls burn(user, amount) — standard CCIP BurnFromMintTokenPool pattern
        vm.prank(pool);
        vm.expectRevert("ERC20: insufficient allowance");
        token.burn(user, 1 ether);
    }

    /// @dev Shows the call succeeds only after the user pre-approves the pool —
    ///      a requirement absent from the Chainlink reference implementation.
    function test_burnFromAccountSucceedsWithAllowance() public {
        vm.prank(user);
        token.approve(pool, 1 ether);   // user must manually approve — not part of CCIP flow

        vm.prank(pool);
        token.burn(user, 1 ether);      // now succeeds
        assertEq(token.balanceOf(user), 0);
    }
}
```

Running `test_burnFromAccountRevertsWithoutAllowance` passes (revert confirmed); `test_burnFromAccountSucceedsWithAllowance` passes only after the extra `approve` step that the CCIP pool never performs.
