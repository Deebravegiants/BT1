### Title
Blocked Spender Can Drain User rsETH via `transferFrom` to an Unblocked Address — (File: `contracts/RSETH.sol`)

---

### Summary

`RSETH.sol` implements a transfer-block mechanism (`blockUserTransfers` / `_enforceNotBlocked`) that only checks the `from` and `to` addresses inside the `_transfer` override. It never checks `msg.sender` (the spender). A blocked spender that holds a pre-existing ERC-20 allowance from a victim can call `transferFrom(victim, unblocked_attacker, amount)` and successfully drain the victim's rsETH, because neither `victim` nor `unblocked_attacker` is blocked.

---

### Finding Description

`RSETH._transfer` is overridden to enforce the block on both endpoints of a transfer:

```solidity
// contracts/RSETH.sol
function _transfer(address from, address to, uint256 amount) internal override {
    _enforceNotBlocked(from);   // checks victim — not blocked
    _enforceNotBlocked(to);     // checks attacker — not blocked
    super._transfer(from, to, amount);
}
```

`approve()` is **not** overridden, so any address — including a blocked one — can hold or grant allowances freely. The standard `ERC20Upgradeable.transferFrom` path is:

```
transferFrom(from, to, amount)
  → _spendAllowance(from, msg.sender, amount)   // msg.sender = blocked spender
  → _transfer(from, to, amount)                 // only checks from & to
```

`msg.sender` (the spender) is never passed to `_enforceNotBlocked`. Therefore:

1. **User** approves a DeFi router/pool: `rsETH.approve(router, MAX)`.
2. **Manager** blocks the router (e.g., OFAC action): `rsETH.blockUserTransfers([router])`.
3. **Blocked router** calls `rsETH.transferFrom(user, attacker, balance)`.
   - `_enforceNotBlocked(user)` → user is not blocked → passes.
   - `_enforceNotBlocked(attacker)` → attacker is not blocked → passes.
   - Transfer succeeds; user's rsETH is stolen.

The block mechanism is supposed to freeze the router's ability to move funds, but it only prevents the router from appearing as `from` or `to` — not from acting as the `msg.sender` spender.

---

### Impact Explanation

**Critical — Direct theft of user rsETH funds.**

Any rsETH holder who has granted an allowance to a subsequently-blocked address (e.g., a compromised L2 pool, a rogue aggregator, or any address the manager later flags) can have their entire approved balance drained to an unblocked attacker address. The block mechanism, which is the protocol's primary on-chain protective response to a compromised spender, is rendered ineffective.

---

### Likelihood Explanation

**Medium.** The scenario requires:
- A user to have an active `approve()` to a spender.
- The manager to later block that spender.

Both conditions are realistic: users routinely grant max-approvals to pools and routers, and the `blockUserTransfers` function exists precisely to respond to compromised or sanctioned addresses. The window between the manager blocking the spender and the user revoking the approval is the attack surface.

---

### Recommendation

Override `transferFrom` (or `_spendAllowance`) in `RSETH.sol` to enforce the block on `msg.sender` as well:

```solidity
function transferFrom(address from, address to, uint256 amount)
    public
    virtual
    override
    returns (bool)
{
    _enforceNotBlocked(msg.sender); // block the spender too
    return super.transferFrom(from, to, amount);
}
```

Alternatively, add the check inside `_transfer` by threading the spender through, or override `_spendAllowance` to call `_enforceNotBlocked(_msgSender())`.

---

### Proof of Concept

```solidity
// Scenario: router is blocked but still drains user rsETH

address router  = makeAddr("router");
address attacker = makeAddr("attacker");

// 1. User approves router
vm.prank(user);
rsETH.approve(router, type(uint256).max);

// 2. Manager blocks the router
vm.prank(manager);

// 3. Blocked router calls transferFrom(user, attacker, balance)

assertEq(rsETH.balanceOf(user), 0);
assertGt(rsETH.balanceOf(attacker), 0);

    /// @notice Block transfers TO and FROM given users for 24 hours
    /// @dev Re-applying the block before expiry refreshes the hold to `block.timestamp + 1 days`
    ///      (i.e. not cumulative; never more than 24h from the latest call). Exempt addresses cannot be blocked.
    ///      Emits {UserTransfersBlocked} only when the timestamp changes.
    /// @param accounts Addresses to block.
    function blockUserTransfers(address[] calldata accounts) external onlyLRTManager {
        uint256 blockedUntil = block.timestamp + 1 days;
        uint256 length = accounts.length;

        for (uint256 i = 0; i < length; ++i) {
            address account = accounts[i];

            if (isPermanentlyExempt[account] || account == address(0)) continue;

            uint256 prevBlockedUntil = transfersBlockedUntil[account];

            if (blockedUntil != prevBlockedUntil) {
                transfersBlockedUntil[account] = blockedUntil;
                emit UserTransfersBlocked(account, blockedUntil);
            }
        }
    }
```

    /// @dev Reverts if `account` is currently blocked (used for transfers, mints, and burns)
    function _enforceNotBlocked(address account) internal {
        // Addresses that are permanently exempt can never be blocked
        if (isPermanentlyExempt[account]) return;

        // Check if the account has an active transfer block
        uint256 blockedUntil = transfersBlockedUntil[account];
        if (blockedUntil == 0) return;

        if (block.timestamp < blockedUntil) revert TransfersBlocked(account, blockedUntil);

        // Auto-clean up expired block
        delete transfersBlockedUntil[account];
    }
```
