### Title
Fee Rate Applied at Claim Time Rather Than Root-Publication Time Allows Owner to Reduce User Payouts Below Implied Amount — (`contracts/KERNEL/KernelMerkleDistributor.sol`)

### Summary
`KernelMerkleDistributor` applies the **current** `feeInBPS` at the moment a user calls `claim`, not the fee that was in effect when the Merkle root was published. Because `setFeeInBPS` takes effect immediately with no timelock or announcement delay, the owner can — intentionally or inadvertently — raise the fee between root publication and user claims, causing users to receive fewer KERNEL tokens than the Merkle leaf implies.

### Finding Description
When a Merkle root is published via `setMerkleRoot`, the leaf encodes `(index, account, cumulativeAmount)`. [4](#0-3) 

The fee deducted from `claimableAmount` is computed using the **live** `feeInBPS` storage variable at claim time: [1](#0-0) 

`setFeeInBPS` can be called by the owner at any time, with no delay, no timelock, and no per-root fee snapshot: [2](#0-1) 

The maximum fee is 10% (`MAX_FEE_IN_BPS = 1000`): [3](#0-2) 

There is no mechanism for users to verify or rely on a specific fee rate being locked to a given root.

### Impact Explanation
Users who observe a 0% fee at root-publication time and delay claiming can receive up to 10% less KERNEL than the leaf's `cumulativeAmount` delta implies. No principal is lost from the contract — the fee goes to `protocolTreasury` — but users fail to receive the full amount they were promised off-chain. This matches the scoped impact: **Low — contract fails to deliver promised returns, but doesn't lose value**.

### Likelihood Explanation
The non-malicious path is realistic: the owner changes `feeInBPS` for a new distribution epoch without realizing it immediately affects all pending claims from the current root. The malicious path (owner deliberately sandwiches root publication with a fee increase) is also technically possible. Either way, no external attacker is needed — only the owner's own admin call.

### Recommendation
1. **Snapshot the fee per root index**: store `feeInBPS` alongside each `setMerkleRoot` call and use the snapshotted value in `_processClaim` for that root's claims.
2. **Alternatively, add a timelock** on `setFeeInBPS` so users have a guaranteed window to claim under the current fee before any increase takes effect.
3. **Emit the effective fee in the `MerkleRootSet` event** so off-chain tooling can surface the exact fee users will pay for a given root.

### Proof of Concept
```solidity
// 1. Owner publishes root with feeInBPS = 0
distributor.setFeeInBPS(0);
distributor.setMerkleRoot(root); // leaf: (1, alice, 1000e18)

// 2. Owner raises fee to 10% before alice claims
distributor.setFeeInBPS(1000);

// 3. Alice claims — receives 900e18 instead of 1000e18
distributor.claim(1, alice, 1000e18, proof);
// _processClaim: fee = (1000e18 * 1000) / 10000 = 100e18
// amountToSend = 900e18  ← 10% less than the leaf implies
assert(kernel.balanceOf(alice) == 900e18); // passes, proving the invariant breaks
``` [5](#0-4)
