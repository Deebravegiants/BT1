### Title
L1Vault LST Tokens Permanently Frozen When Asset Removed from LRTConfig — (`contracts/L1Vault.sol`)

---

### Summary

`LRTConfig.removeSupportedAsset` guards against removal only by checking `LRTDepositPool.getTotalAssetDeposits`, which does **not** account for LST balances sitting in `L1Vault`. An admin can legitimately delist an asset while tokens are bridged-but-undeposited in `L1Vault`, after which `depositAssetForL1Vault` permanently reverts and no recovery path exists.

---

### Finding Description

**Step 1 — Guard in `removeSupportedAsset` is incomplete.**

`LRTConfig.removeSupportedAsset` allows removal only when total deposits are at or below `maxNegligibleAmount`: [1](#0-0) 

The check delegates to `LRTDepositPool.getTotalAssetDeposits`, which sums balances across: `LRTDepositPool`, NodeDelegators, EigenLayer strategies, `LRTConverter`, and `LRTUnstakingVault`: [2](#0-1) 

`L1Vault` is **not** one of the tracked locations. Its balance is invisible to this guard. [3](#0-2) 

**Step 2 — `depositAssetForL1Vault` routes through `onlySupportedERC20Token`.**

`L1Vault.depositAssetForL1Vault` calls `lrtDepositPool.depositAsset`: [4](#0-3) 

`LRTDepositPool.depositAsset` carries the `onlySupportedERC20Token(asset)` modifier: [5](#0-4) 

That modifier checks `lrtConfig.isSupportedAsset(asset)`: [6](#0-5) 

Once `removeSupportedAsset` clears `isSupportedAsset[asset]`, every call to `depositAssetForL1Vault` for that token reverts with `AssetNotSupported`. [7](#0-6) 

**Step 3 — No recovery path in `L1Vault`.**

`L1Vault` exposes no `emergencyWithdraw`, sweep, or token-rescue function. The only asset-movement functions are `depositAssetForL1Vault` (now blocked), `unwrapWstETH` (wstETH only), `unwrapWETH` (WETH only), and `bridgeRsETHToL2` (rsETH only). The stuck LST and all yield it accrues are unrecoverable. [8](#0-7) 

---

### Impact Explanation

Any LST tokens bridged to `L1Vault` but not yet forwarded to `LRTDepositPool` at the time of asset removal are permanently locked. The staking yield those tokens continue to generate (e.g., stETH rebases) is also permanently frozen. This satisfies **Medium — Permanent freezing of unclaimed yield**. Because the principal is also irrecoverable, it additionally satisfies **Critical — Permanent freezing of funds**.

---

### Likelihood Explanation

The scenario requires only two sequential, individually legitimate admin actions:

1. LST tokens arrive in `L1Vault` via the normal bridge flow (routine operation).
2. Admin calls `LRTConfig.removeSupportedAsset` to delist the asset (routine governance action, e.g., deprecating stETH in favour of wstETH).

The guard in `removeSupportedAsset` gives the admin no signal that tokens are sitting in `L1Vault`; the check passes cleanly. No malicious intent, key compromise, or front-running is required. The window is any period between a bridge delivery and the manager calling `depositAssetForL1Vault`.

---

### Recommendation

1. **Extend the deposit-accounting guard** in `LRTConfig.removeSupportedAsset` to also sum `IERC20(asset).balanceOf(l1Vault)` for every registered `L1Vault` instance, or expose a registry of `L1Vault` addresses that `getTotalAssetDeposits` iterates.

2. **Add an emergency token-rescue function** to `L1Vault` (e.g., `rescueToken(address token, address to, uint256 amount)`) gated behind `TIMELOCK_ROLE` or `DEFAULT_ADMIN_ROLE`, so that tokens stranded by any future configuration change can be recovered.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity 0.8.27;

// Fork test (Foundry) — run against a local fork, no mainnet calls
contract L1VaultFreezePoC is Test {
    // Assume deployed instances wired up in setUp()
    LRTConfig   lrtConfig;
    LRTDepositPool depositPool;
    L1Vault     l1Vault;
    IERC20      stETH;
    address     admin;

    function test_permanentFreeze() external {
        // 1. Simulate bridge delivery: stETH lands in L1Vault
        uint256 bridgedAmount = 10 ether;
        deal(address(stETH), address(l1Vault), bridgedAmount);

        // 2. Confirm getTotalAssetDeposits does NOT see L1Vault balance
        uint256 tracked = depositPool.getTotalAssetDeposits(address(stETH));
        // tracked == 0 (or <= maxNegligibleAmount), L1Vault balance invisible

        // 3. Admin removes stETH — guard passes because L1Vault is not counted
        vm.prank(admin);
        lrtConfig.removeSupportedAsset(address(stETH), /*tokenIndex=*/0);

        // 4. Manager tries to deposit — reverts
        vm.prank(address(this)); // MANAGER_ROLE holder
        vm.expectRevert(ILRTConfig.AssetNotSupported.selector);
        l1Vault.depositAssetForL1Vault(address(stETH));

        // 5. No sweep path — stETH balance unchanged, permanently locked
        assertEq(stETH.balanceOf(address(l1Vault)), bridgedAmount);
    }
}
```

        if (wethBalance == 0) {
            revert NoWETHBalance();
