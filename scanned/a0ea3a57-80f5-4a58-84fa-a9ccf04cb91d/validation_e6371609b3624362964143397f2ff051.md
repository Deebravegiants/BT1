### Title
Lido Discounted Withdrawals Leave Phantom Balance in `ethValueInWithdrawal`, Inflating rsETH Price and Causing Protocol Insolvency - (File: contracts/LRTConverter.sol)

---

### Summary

`LRTConverter` tracks stETH pending Lido withdrawal in `ethValueInWithdrawal` using the **requested** ETH value, but reduces it by the **actually received** ETH when claiming. In a Lido slashing event the received amount is less than the requested amount, leaving a phantom balance in `ethValueInWithdrawal`. This phantom balance is included in `LRTDepositPool.getTotalAssetDeposits()`, inflating the rsETH price and ultimately causing protocol insolvency.

---

### Finding Description

The flow has three steps:

**Step 1 – `transferAssetFromDepositPool` records the full oracle-valued ETH equivalent:**

```solidity
ethValueInWithdrawal += (_amount * lrtOracle.getAssetPrice(_asset)) / 1e18;
``` [1](#0-0) 

**Step 2 – `unstakeStEth` submits the Lido withdrawal request for the same amount.** No update to `ethValueInWithdrawal` occurs here. [2](#0-1) 

**Step 3 – `claimStEth` claims ETH from Lido and calls `_sendEthToDepositPool(address(this).balance)`:**

```solidity
function _sendEthToDepositPool(uint256 _amount) internal {
    if (ethValueInWithdrawal > _amount) {
        ethValueInWithdrawal -= _amount;
    } else {
        ethValueInWithdrawal = 0;
    }
    ILRTDepositPool(lrtDepositPoolAddress).receiveFromLRTConverter{ value: _amount }();
}
``` [3](#0-2) 

When Lido applies a slashing discount, `_amount` (received ETH) < `ethValueInWithdrawal` (tracked ETH). The subtraction leaves a residual phantom balance in `ethValueInWithdrawal` that no longer corresponds to any real asset.

This phantom value is then read by `getETHDistributionData()`:

```solidity
address lrtConverter = lrtConfig.getContract(LRTConstants.LRT_CONVERTER);
ethLyingInConverter = ILRTConverter(lrtConverter).ethValueInWithdrawal();
``` [4](#0-3) 

Which feeds directly into `getTotalAssetDeposits(ETH)`:

```solidity
return (assetLyingInDepositPool + assetLyingInNDCs + effectiveAssetWithEigenLayer + assetLyingInConverter
        + assetLyingUnstakingVault);
``` [5](#0-4) 

`getTotalAssetDeposits` is used by `LRTOracle` to compute the rsETH price, and by `LRTWithdrawalManager.getAvailableAssetAmount` to determine how much can be unlocked for withdrawals. [6](#0-5) 

---

### Impact Explanation

- **Inflated rsETH price**: `getTotalAssetDeposits` is overstated by the phantom ETH, so the rsETH/ETH exchange rate is higher than reality. New depositors receive fewer rsETH shares than they are entitled to (they overpay).
- **Protocol insolvency**: The total rsETH supply is backed by fewer real assets than the price implies. When enough users attempt to redeem, the protocol cannot cover all obligations.
- **Withdrawal queue desync**: `getAvailableAssetAmount` returns an inflated figure, potentially unlocking withdrawal requests that cannot actually be fulfilled, leading to failed transfers or permanent freezing of user funds.

Impact classification: **Critical – Protocol insolvency / Permanent freezing of funds**.

---

### Likelihood Explanation

Lido explicitly documents that slashing events can reduce the claimable amount below the requested amount. The LRT-rsETH protocol holds significant stETH and actively unstakes it through `LRTConverter.unstakeStEth`. Any Lido slashing event during an active withdrawal window triggers this bug. This is a realistic, documented risk of the Lido protocol and not a theoretical edge case.

The operator actions (`transferAssetFromDepositPool`, `unstakeStEth`, `claimStEth`) are all part of the normal, expected operational flow. No attacker action is required to trigger the accounting error; the slashing event alone is sufficient.

---

### Recommendation

In `_sendEthToDepositPool`, do not use `ethValueInWithdrawal` as the source of truth for how much was expected. Instead, track the exact ETH value committed per withdrawal request and always reconcile to zero upon claim completion, regardless of the received amount. Alternatively, after each `claimStEth` call, explicitly set `ethValueInWithdrawal = 0` (or reduce it by the originally committed amount, not the received amount) so that any slashing loss is immediately recognized and reflected in the rsETH price rather than silently inflating it.

---

### Proof of Concept

1. Operator calls `transferAssetFromDepositPool(stETH, 1000e18)`.
   - `ethValueInWithdrawal` becomes `1000e18` (assuming 1:1 stETH/ETH price). [7](#0-6) 

2. Operator calls `unstakeStEth(1000e18)` → Lido withdrawal request for 1000 stETH submitted. [8](#0-7) 

3. Lido slashing event occurs; only 900 ETH is claimable.

4. Operator calls `claimStEth(requestId, hint)` → 900 ETH received → `_sendEthToDepositPool(900e18)` called.
   - `ethValueInWithdrawal = 1000e18 - 900e18 = 100e18` (phantom, non-existent ETH). [3](#0-2) 

5. `getETHDistributionData()` now reports `ethLyingInConverter = 100e18` even though no ETH or stETH remains in the converter. [4](#0-3) 

6. rsETH price is inflated by 100 ETH worth. All subsequent deposits mint fewer rsETH shares than deserved. All subsequent withdrawal unlocks are computed against an inflated total, and the protocol is insolvent by 100 ETH.
