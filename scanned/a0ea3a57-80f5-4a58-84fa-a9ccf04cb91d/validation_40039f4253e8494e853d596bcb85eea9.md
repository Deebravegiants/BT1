### Title
RSETH Pause Does Not Restrict ERC20 Transfers, Allowing Bypass During Emergency - (File: contracts/RSETH.sol)

### Summary
The `RSETH` token contract applies `whenNotPaused` only to `mint()` and `burnFrom()`, but the internal `_transfer()` override — which backs all ERC20 `transfer()` and `transferFrom()` calls — contains no pause check. Any rsETH holder can freely transfer tokens even when the contract is paused, bypassing the intended emergency freeze.

### Finding Description
`RSETH` inherits `PausableUpgradeable` and exposes `pause()` / `unpause()` for emergency control. The `mint()` and `burnFrom()` functions are correctly gated:

```solidity
// contracts/RSETH.sol:229-248
function mint(address to, uint256 amount)
    external
    onlyRole(LRTConstants.MINTER_ROLE)
    whenNotPaused          // ← pause enforced
    checkDailyMintLimit(amount)
{ ... }

function burnFrom(address account, uint256 amount)
    external
    onlyRole(LRTConstants.BURNER_ROLE)
    whenNotPaused          // ← pause enforced
{ ... }
```

However, the `_transfer()` override that handles all ERC20 transfers omits the pause check entirely:

```solidity
// contracts/RSETH.sol:287-291
function _transfer(address from, address to, uint256 amount) internal override {
    _enforceNotBlocked(from);
    _enforceNotBlocked(to);
    super._transfer(from, to, amount);   // ← no whenNotPaused
}
```

Because `ERC20Upgradeable.transfer()` and `transferFrom()` both route through `_transfer()`, any rsETH holder can call them freely regardless of the paused state.

A secondary consequence: `LRTWithdrawalManager.initiateWithdrawal()` calls `IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked)` to pull rsETH from the user. Because `_transfer()` ignores the pause, this `safeTransferFrom` succeeds even when RSETH is paused, locking user rsETH inside the withdrawal manager. The subsequent `unlockQueue()` → `burnFrom()` path will then revert (pause blocks burning), leaving the rsETH stranded until the pause is lifted.

### Impact Explanation
When the RSETH contract is paused in response to an emergency:

1. Any rsETH holder can transfer tokens to a new address, defeating the admin's ability to freeze a specific account via `blockUserTransfers` before the attacker relocates funds.
2. rsETH remains usable in external DeFi integrations (DEXes, lending protocols) despite the emergency stop.
3. Users who call `initiateWithdrawal()` while RSETH is paused (and `LRTWithdrawalManager` is not) will have their rsETH locked in the manager with no ability to complete the withdrawal until the pause is lifted — a temporary fund freeze.

**Impact: Low — Contract fails to deliver promised returns (the pause does not fully freeze rsETH as the emergency mechanism implies).**

### Likelihood Explanation
The PAUSER_ROLE activating a pause is a realistic emergency scenario (oracle manipulation, exploit detection). Any rsETH holder — an unprivileged depositor or secondary-market buyer — can immediately call `transfer()` or `transferFrom()` with no special access. The bypass requires zero attacker privilege and is always available whenever the contract is paused.

### Recommendation
Add `whenNotPaused` to the `_transfer()` override so that all token movements are halted during an emergency:

```solidity
function _transfer(address from, address to, uint256 amount) internal override whenNotPaused {
    _enforceNotBlocked(from);
    _enforceNotBlocked(to);
    super._transfer(from, to, amount);
}
```

If transfers must remain permitted during a pause (e.g., for the `recoverFrozenFunds` admin path), use the same pattern already present there — call `super._transfer()` directly — and add `whenNotPaused` to the public-facing override.

### Proof of Concept

1. Admin detects an exploit and calls `RSETH.pause()`.
2. Attacker holds rsETH at address `A`. Admin prepares to call `blockUserTransfers([A])`.
3. Before the block lands, attacker calls `rsETH.transfer(B, balance)` — succeeds because `_transfer()` has no pause check.
4. Admin's `blockUserTransfers([A])` now blocks an empty address; attacker's rsETH is safely at `B`.
5. Alternatively, a user calls `LRTWithdrawalManager.initiateWithdrawal(ETH, amount, "")` while RSETH is paused: `safeTransferFrom` succeeds (line 166 of `LRTWithdrawalManager.sol`), rsETH is locked in the manager, but `unlockQueue()` → `burnFrom()` reverts — funds are temporarily frozen.
