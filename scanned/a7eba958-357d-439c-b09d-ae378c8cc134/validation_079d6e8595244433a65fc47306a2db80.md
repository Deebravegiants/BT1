### Title
Unprotected `reinitializer(2)` in NodeDelegator Allows Any Caller to Permanently Block Upgrade Initialization - (File: contracts/NodeDelegator.sol)

### Summary
`NodeDelegator.initialize2()` is a `reinitializer(2)` function with no access control. Any unprivileged external caller can invoke it before the admin does during an upgrade, permanently consuming the reinitializer slot and setting `lastNonce` at an unintended time, blocking the legitimate upgrade initialization forever.

### Finding Description
`NodeDelegator.initialize2()` is declared as:

```solidity
function initialize2() external reinitializer(2) {
    lastNonce = _getNonce();
}
```

It carries the `reinitializer(2)` modifier, which allows the function to execute exactly once (when `_initialized < 2`) and then permanently marks the contract as initialized to version 2. However, unlike every other reinitializer in the codebase — all of which carry an explicit role check — `initialize2()` has **no access control modifier whatsoever**. Any EOA or contract can call it.

By contrast, all peer reinitializers enforce role restrictions:
- `LRTWithdrawalManager.initialize2()` — `onlyRole(LRTConstants.UNLOCKED_WITHDRAWAL_INITIALIZER)`
- `LRTWithdrawalManager.initialize3()` — `onlyLRTManager`
- `RSETH.reinitialize()` — `onlyLRTManager`
- All `RSETHPool` / `RSETHPoolV3ExternalBridge` reinitializers — `onlyRole(DEFAULT_ADMIN_ROLE)`

`NodeDelegator.initialize2()` is the sole exception.

### Impact Explanation
When the protocol upgrades `NodeDelegator` to a version that requires `initialize2()` to be called, an attacker can front-run the admin's upgrade transaction and call `initialize2()` themselves. The `reinitializer(2)` modifier is consumed immediately; any subsequent call by the admin reverts with `"Initializable: contract is already initialized"`. The result is:

1. `lastNonce` is set to whatever `_getNonce()` returns at the attacker's chosen block, not the block the admin intended.
2. The admin's upgrade initialization is permanently blocked — the slot can never be reclaimed.
3. The NodeDelegator operates with a potentially incorrect `lastNonce`, which is used for nonce-based tracking of EigenLayer interactions. An incorrect starting nonce can desynchronize the contract's internal accounting from the actual on-chain state, causing operational failures in EigenLayer-related flows (deposits, withdrawals, validator credential verification).

Impact classification: **Low to Medium** — the contract fails to initialize correctly after an upgrade, potentially causing it to fail to deliver promised returns or temporarily freeze funds depending on how `lastNonce` diverges from the expected value.

### Likelihood Explanation
The attack requires only watching the mempool for the admin's upgrade transaction and submitting a higher-gas call to `initialize2()` before it lands. This is a straightforward front-run with no capital requirement and no special permissions. Any unprivileged actor can execute it.

### Recommendation
Add an access-control modifier to `initialize2()`, consistent with every other reinitializer in the codebase:

```solidity
function initialize2() external reinitializer(2) onlyLRTAdmin {
    lastNonce = _getNonce();
}
```

### Proof of Concept

1. Protocol deploys a new `NodeDelegator` implementation that introduces `initialize2()`.
2. Admin submits an upgrade transaction (e.g., via `upgradeAndCall` or a separate `initialize2()` call).
3. Attacker observes the pending transaction in the mempool and submits `initialize2()` with higher gas.
4. Attacker's transaction is mined first; `_initialized` is set to `2` and `lastNonce` is set to `_getNonce()` at the attacker's block.
5. Admin's transaction reverts: `"Initializable: contract is already initialized"`.
6. `NodeDelegator` is now permanently stuck with the attacker-timed `lastNonce` value and no path to correct it. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
