### Title
Incomplete Validation of Chainlink `latestRoundData` Return Values Enables Stale Price Usage in Core Protocol Operations - (File: contracts/oracles/ChainlinkPriceOracle.sol)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls Chainlink's `latestRoundData()` but discards all return values except `price`, performing no staleness check (`updatedAt`), no incomplete-round check (`answeredInRound < roundId`), and no non-positive price check. This stale or zero price propagates directly into rsETH minting (deposits) and asset payout calculations (withdrawals), enabling depositors to receive inflated rsETH or withdrawers to receive excess underlying assets at the expense of existing rsETH holders.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the asset/ETH exchange rate used throughout the protocol:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol, line 52
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All five return values of `latestRoundData` — `roundId`, `answer`, `startedAt`, `updatedAt`, `answeredInRound` — are available, but only `answer` (as `price`) is used. The following checks are absent:

- **No staleness check**: `updatedAt` is not compared against `block.timestamp`. If the Chainlink feed has not been updated within its heartbeat window, the stale price is silently accepted.
- **No incomplete-round check**: `answeredInRound >= roundId` is never verified. An in-progress aggregation round can return a partially-computed answer.
- **No non-positive price check**: `price <= 0` is not rejected. A zero price returns `0` from `getAssetPrice`, and a negative price wraps to a huge `uint256` under Solidity's explicit cast semantics, causing an overflow revert.

This price is consumed in three critical paths:

1. **`LRTDepositPool.getRsETHAmountToMint()`** (line 520): `rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()` — governs how many rsETH tokens a depositor receives.
2. **`LRTWithdrawalManager.getExpectedAssetAmount()`** (line 593): `underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset)` — governs how much underlying a withdrawer receives.
3. **`LRTOracle._getTotalEthInProtocol()`** (line 339): `totalETHInProtocol += totalAssetAmt.mulWad(assetER)` — feeds into `_updateRsETHPrice()`, which sets the global `rsETHPrice` used by all subsequent operations.

---

### Impact Explanation

**Stale inflated price at deposit time**: If an LST's price has fallen but the Chainlink feed is stale (still reporting the old higher price), a depositor calling `depositAsset()` receives more rsETH than the deposited asset is worth. This dilutes the holdings of all existing rsETH holders — equivalent to theft of their accrued yield.

**Stale deflated price at withdrawal time**: If an LST's price has risen but the feed is stale (reporting the old lower price), `getExpectedAssetAmount()` returns a larger `underlyingToReceive` (since it divides by a smaller `assetPrice`). A withdrawer calling `initiateWithdrawal()` locks in an inflated payout, draining assets from the protocol at the expense of other users.

**Zero price at deposit time**: If Chainlink's circuit breaker returns `price = 0`, `getAssetPrice` returns `0`, `getRsETHAmountToMint` returns `0`, and if the caller passes `minRSETHAmountExpected = 0`, the deposit succeeds while minting zero rsETH — the depositor's asset is transferred in but no rsETH is issued, constituting a permanent loss of deposited funds.

**Impact classification**: High — theft of unclaimed yield (stale price scenarios); Critical — permanent freezing of deposited funds (zero price scenario).

---

### Likelihood Explanation

Chainlink feeds can become stale during:
- Ethereum network congestion preventing oracle keeper transactions from landing.
- Chainlink node operator outages.
- Low price volatility periods where the deviation threshold is never crossed and the heartbeat window expires.

These are documented, real-world conditions. The vulnerability requires no privileged access — any unprivileged depositor or withdrawer interacting with the protocol during a stale-feed window is affected. The `depositAsset()` and `initiateWithdrawal()` functions are fully public entry points.

---

### Recommendation

Apply the same validation pattern already present in `ChainlinkOracleForRSETHPoolCollateral.getRate()` (lines 30–32) to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (price <= 0) revert InvalidPrice();
    if (answeredInRound < roundId) revert StalePrice();
    if (block.timestamp - updatedAt > STALENESS_THRESHOLD) revert StalePrice(); // e.g. 24 hours

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Apply equivalent checks to `RSETHPriceFeed.latestRoundData()` (line 68), which also calls `ETH_TO_USD.latestRoundData()` without validating any return values.

### Proof of Concept

**Setup**: Chainlink's stETH/ETH feed becomes stale (last updated >24 hours ago). The feed still reports `price = 1.05e18` (stETH at a 5% premium to ETH), but the current market price has dropped to `1.00e18`.

**Attack path**:

1. Attacker observes the stale feed condition on-chain (`updatedAt` is old).
2. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
3. `_beforeDeposit` → `getRsETHAmountToMint(stETH, 100e18)` → `LRTOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` returns `1.05e18` (stale).
4. `rsethAmountToMint = (100e18 * 1.05e18) / rsETHPrice` — attacker receives rsETH valued at 105 ETH for a deposit worth 100 ETH.
5. The 5 ETH surplus in rsETH value is extracted from existing rsETH holders' proportional share of the protocol TVL.
