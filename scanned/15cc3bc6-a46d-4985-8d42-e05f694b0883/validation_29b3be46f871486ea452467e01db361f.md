### Title
Unchecked Chainlink L2 Sequencer Uptime Allows Stale Price Exploitation in RSETHPool Collateral Deposits - (File: contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol)

### Summary
`ChainlinkOracleForRSETHPoolCollateral.getRate()` calls Chainlink's `latestRoundData()` on L2 without verifying the L2 sequencer uptime feed. During or immediately after a sequencer outage, the oracle returns stale pre-downtime prices, enabling an attacker to mint rsETH at an inflated collateral valuation.

### Finding Description
`ChainlinkOracleForRSETHPoolCollateral` is deployed on L2 chains (Arbitrum, etc.) and serves as the price oracle for collateral tokens accepted by the RSETHPool family of contracts. Its `getRate()` function calls `AggregatorV3Interface(oracle).latestRoundData()` and applies three validity checks: `answeredInRound < roundID` (stale round), `timestamp == 0` (incomplete round), and `ethPrice <= 0` (invalid price).

None of these checks detect the L2 sequencer downtime scenario. When the Arbitrum sequencer is offline, Chainlink's L2 data feeds stop updating but retain their last-known values. The existing staleness check (`answeredInRound < roundID`) passes because the round counter itself does not advance during downtime. When the sequencer resumes, there is a window before the feed catches up during which `getRate()` returns the pre-downtime price — which may be materially different from the current market price.

The Chainlink-recommended mitigation is to query a dedicated sequencer uptime feed (e.g., `0xFdB631F5EE196F0ed6FAa767959853A9F217697D` on Arbitrum) and revert if the sequencer is reported as down or has been up for less than a grace period (typically 3600 seconds).

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol
function getRate() public view returns (uint256) {
    (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();

    if (answeredInRound < roundID) revert StalePrice();   // does NOT catch sequencer downtime
    if (timestamp == 0) revert IncompleteRound();
    if (ethPrice <= 0) revert InvalidPrice();
    // ← no sequencer uptime check here
    ...
}
```

### Impact Explanation
An attacker who deposits a collateral token whose Chainlink feed is stale (price higher than current market) receives more rsETH than the deposited collateral is worth. Repeated exploitation drains the protocol's ETH reserves backing rsETH, causing insolvency. Even a single large deposit during a sequencer-restart window can extract significant value. Impact: **Critical — direct theft of user funds / protocol insolvency**.

### Likelihood Explanation
Arbitrum sequencer outages have occurred historically (e.g., December 2021, June 2023). The attack requires: (1) a sequencer outage during which the collateral token's price moves adversely, and (2) the attacker being first to deposit after the sequencer resumes. Both conditions are realistic and have precedent. Likelihood: **Low-Medium** (rare but non-negligible, and the attack is permissionless once the window opens).

### Recommendation
Add a sequencer uptime check in `getRate()` before consuming the Chainlink price, following the Chainlink documentation for L2 sequencer feeds:

```solidity
AggregatorV3Interface internal sequencerUptimeFeed; // set in constructor

function getRate() public view returns (uint256) {
    (, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
    // answer == 0: sequencer is up; answer == 1: sequencer is down
    if (answer != 0) revert SequencerDown();
    if (block.timestamp - startedAt < GRACE_PERIOD) revert GracePeriodNotOver();

    (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();
    if (answeredInRound < roundID) revert StalePrice();
    if (timestamp == 0) revert IncompleteRound();
    if (ethPrice <= 0) revert InvalidPrice();
    ...
}
```

### Proof of Concept

1. Arbitrum sequencer goes offline. During the outage, the price of a supported collateral token (e.g., wstETH) drops 10% on mainnet.
2. Chainlink's L2 feed for that token stops updating; it retains the pre-drop price.
3. Sequencer comes back online. `ChainlinkOracleForRSETHPoolCollateral.getRate()` still returns the pre-drop (inflated) price because `answeredInRound == roundID` and `timestamp != 0`.
4. Attacker calls `RSETHPool.swapToRsETH(wstETH, largeAmount)` (or equivalent deposit function). The pool calls `supportedTokenOracle[wstETH].getRate()` → `ChainlinkOracleForRSETHPoolCollateral.getRate()` → returns stale high price.
5. The pool mints rsETH proportional to the stale valuation, giving the attacker ~10% more rsETH than the collateral is worth.
6. Attacker bridges rsETH to mainnet via the canonical bridge and redeems for ETH at the correct rate, extracting the difference from the protocol's reserves.
