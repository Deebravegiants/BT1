### Title
`emergencyWithdrawFromAave` Does Not Disable Aave Integration, Allowing Immediate Re-Deposit - (File: contracts/LRTWithdrawalManager.sol)

---

### Summary

`LRTWithdrawalManager.emergencyWithdrawFromAave()` withdraws ETH from Aave but never sets `isAaveIntegrationEnabled = false`. Because `unlockQueue()` unconditionally re-deposits unlocked ETH into Aave whenever `isAaveIntegrationEnabled` is `true`, the very next `unlockQueue(ETH, ...)` call silently re-deposits the rescued ETH back into Aave, defeating the emergency withdrawal entirely.

---

### Finding Description

`emergencyWithdrawFromAave()` is the designated escape hatch for pulling ETH out of Aave in an emergency (e.g., Aave exploit, liquidity freeze, or pool insolvency):

```solidity
// contracts/LRTWithdrawalManager.sol:551-563
function emergencyWithdrawFromAave(uint256 amount) external nonReentrant onlyRole(LRTConstants.PAUSER_ROLE) {
    if (!isAaveIntegrationEnabled) revert AaveIntegrationNotEnabled();

    uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
    if (aaveBalance == 0) revert InsufficientAaveBalance();

    _collectInterestToTreasury();
    uint256 withdrawnAmount = _withdrawFromAave(amount);

    emit EmergencyWithdrawFromAave(withdrawnAmount, address(this));
}
```

After this call, `isAaveIntegrationEnabled` remains `true`. [1](#0-0) 

`unlockQueue()` is the normal operational function called by the operator to process pending ETH withdrawal requests. It contains this block:

```solidity
// contracts/LRTWithdrawalManager.sol:309-317
if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN && assetAmountUnlocked > 0) {
    try this.depositToAaveExternal(assetAmountUnlocked) { }
    catch (bytes memory reason) {
        emit AaveDepositFailed(assetAmountUnlocked, reason);
    }
}
``` [2](#0-1) 

Because `isAaveIntegrationEnabled` was never cleared by `emergencyWithdrawFromAave()`, the next `unlockQueue()` call for ETH will re-deposit the freshly rescued ETH straight back into Aave. The operator does not need to do anything wrong — this is the normal, expected operational call.

The `_depositToAave` internal function has no guard against re-depositing after an emergency:

```solidity
// contracts/LRTWithdrawalManager.sol:894-901
function _depositToAave(uint256 amount) internal {
    if (amount == 0) return;
    aaveWETHGateway.depositETH{ value: amount }(aavePool, address(this), 0);
    totalETHDepositedToAave += amount;
    emit ETHDepositedToAave(amount, totalETHDepositedToAave);
}
``` [3](#0-2) 

---

### Impact Explanation

**Medium — Temporary freezing of funds.**

The emergency withdrawal is the protocol's only mechanism to rescue ETH from Aave in a crisis. If the rescued ETH is silently re-deposited on the next `unlockQueue()` call, the emergency is not resolved. Users whose withdrawal requests were unlocked in that same `unlockQueue()` call will have their ETH immediately re-sent to the compromised/frozen Aave pool instead of being held in the contract for them to claim. This constitutes a temporary (potentially extended) freeze of user ETH pending withdrawal.

---

### Likelihood Explanation

**Medium.**

Emergency withdrawals are infrequent by nature, but when they are triggered, the protocol is already in a stressed state. The re-deposit happens automatically on the very next routine `unlockQueue(ETH, ...)` call — a call the operator is expected to make as part of normal withdrawal processing. There is no warning, no revert, and no indication that the re-deposit is happening. The operator is unlikely to know they must also call `setAaveIntegrationEnabled(false)` before resuming operations.

---

### Recommendation

`emergencyWithdrawFromAave()` should atomically disable the Aave integration after withdrawing:

```solidity
function emergencyWithdrawFromAave(uint256 amount) external nonReentrant onlyRole(LRTConstants.PAUSER_ROLE) {
    if (!isAaveIntegrationEnabled) revert AaveIntegrationNotEnabled();

    uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
    if (aaveBalance == 0) revert InsufficientAaveBalance();

    _collectInterestToTreasury();
    _withdrawFromAave(amount);

    // Disable integration so unlockQueue cannot re-deposit
    isAaveIntegrationEnabled = false;
    _revokeApprovalToAaveWETHGateway();
    emit AaveIntegrationEnabled(false);

    emit EmergencyWithdrawFromAave(withdrawnAmount, address(this));
}
```

This mirrors the fix suggested in the reference report: separate the "stop depositing" state from the "allow withdrawals" state, or in this case, ensure the emergency path atomically closes the re-deposit path.

---

### Proof of Concept

1. Aave experiences an issue; PAUSER_ROLE calls `emergencyWithdrawFromAave(type(uint256).max)`.
   - `_withdrawFromAave` pulls all principal ETH back to `LRTWithdrawalManager`. [4](#0-3) 
   - `isAaveIntegrationEnabled` is still `true`. [1](#0-0) 

2. Operator calls `unlockQueue(ETH_TOKEN, ...)` to process pending user withdrawal requests (normal operation). [5](#0-4) 

3. `unlockQueue` redeems ETH from the unstaking vault and then hits the Aave deposit block:
   ```
   isAaveIntegrationEnabled == true  →  depositToAaveExternal(assetAmountUnlocked)
   ``` [2](#0-1) 

4. The ETH rescued in step 1 (now sitting in the contract) is joined by the newly unlocked ETH and the entire batch is deposited back into the compromised Aave pool.

5. Users whose withdrawal requests were just unlocked cannot claim their ETH — it is back in Aave. The emergency withdrawal must be repeated, and the cycle continues until `setAaveIntegrationEnabled(false)` is explicitly called by the manager — a step that is not enforced or documented in the emergency path. [6](#0-5)

    function _withdrawFromAave(uint256 amount) internal returns (uint256 withdrawnAmount) {
        if (amount == 0) return 0;

        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        if (aaveBalance == 0) revert InsufficientAaveBalance();

        // Only withdraw up to the principal amount (don't use accrued interest for user withdrawals)
        uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave;

        withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
        if (withdrawnAmount == 0) return 0;

        aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
        totalETHDepositedToAave -= withdrawnAmount;

        emit ETHWithdrawnFromAave(withdrawnAmount, totalETHDepositedToAave);
    }
```
