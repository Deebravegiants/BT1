### Title
Mutable `withdrawalDelayBlocks` retroactively extends lock period for already-queued withdrawal requests - (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTWithdrawalManager` stores only `withdrawalStartBlock` at queue time but reads the global `withdrawalDelayBlocks` at both unlock and completion time. Because `withdrawalDelayBlocks` is mutable and can be updated by the LRT Manager at any time, an increase to this parameter retroactively extends the lock period for all already-queued withdrawal requests, temporarily freezing user funds beyond the delay that was in effect when they initiated their withdrawal.

---

### Finding Description

When a user calls `initiateWithdrawal`, the internal `_addUserWithdrawalRequest` function creates a `WithdrawalRequest` struct that stores only `withdrawalStartBlock: block.number`. The global `withdrawalDelayBlocks` is **not** snapshotted into the request. [1](#0-0) 

Later, when the manager calls `unlockWithdrawals` → `_unlockWithdrawalRequests`, the delay check reads the **current** global `withdrawalDelayBlocks`: [2](#0-1) 

The same live read occurs in `_processWithdrawalCompletion` when the user calls `completeWithdrawal`: [3](#0-2) 

The LRT Manager can update `withdrawalDelayBlocks` at any time, up to a ceiling of `16 days / 12 seconds` blocks: [4](#0-3) 

The default delay is initialized to `8 days / 12 seconds` blocks: [5](#0-4) 

If the manager legitimately doubles the delay from 8 days to 16 days, every pending withdrawal request that was queued under the 8-day assumption is now subject to the 16-day window. Users who queued their withdrawal expecting to wait 8 days will instead be forced to wait up to 16 days from their `withdrawalStartBlock`.

By contrast, `KernelDepositPool.initiateWithdrawal` correctly snapshots the delay at queue time by computing and storing `unlockTime = block.timestamp + withdrawalDelay` directly in the `Withdrawal` struct, so subsequent changes to `withdrawalDelay` do not affect already-queued requests: [6](#0-5) 

---

### Impact Explanation

**Medium — Temporary freezing of funds.**

Any increase to `withdrawalDelayBlocks` retroactively extends the lock period for all pending withdrawal requests. Users who have already burned their rsETH and are waiting for the delay to expire will find their funds locked for an additional period (up to an extra 8 days at the maximum ceiling). Their rsETH has already been transferred to the contract at `initiateWithdrawal` time, so they cannot cancel and re-queue; they are stuck until the new, longer delay elapses.

---

### Likelihood Explanation

**Medium.** The LRT Manager role is a routine operational role that is expected to tune protocol parameters. A delay increase is a plausible operational action (e.g., in response to a security incident or to align with a new EigenLayer unbonding period). No malicious intent is required; a legitimate parameter update is sufficient to trigger the impact. The number of affected users scales with the volume of pending withdrawals at the time of the update.

---

### Recommendation

Snapshot `withdrawalDelayBlocks` into the `WithdrawalRequest` struct at queue time, mirroring the pattern used by `KernelDepositPool`:

```solidity
struct WithdrawalRequest {
    uint256 rsETHUnstaked;
    uint256 expectedAssetAmount;
    uint256 withdrawalStartBlock;
    uint256 withdrawalDelayBlocks; // snapshot at queue time
}
```

In `_addUserWithdrawalRequest`, populate the field:

```solidity
withdrawalRequests[requestId] = WithdrawalRequest({
    rsETHUnstaked: rsETHUnstaked,
    expectedAssetAmount: expectedAssetAmount,
    withdrawalStartBlock: block.number,
    withdrawalDelayBlocks: withdrawalDelayBlocks  // current value, frozen for this request
});
```

Replace both live reads of the global `withdrawalDelayBlocks` with `request.withdrawalDelayBlocks` in `_unlockWithdrawalRequests` (line 795) and `_processWithdrawalCompletion` (line 715).

---

### Proof of Concept

1. `withdrawalDelayBlocks` is initialized to `8 days / 12 seconds ≈ 57,600 blocks`.
2. Alice calls `initiateWithdrawal(ETH, rsETHAmount, ...)`. Her `WithdrawalRequest` is stored with `withdrawalStartBlock = N`. No delay is snapshotted.
3. At block `N + 40,000` (still within the 8-day window), the LRT Manager calls `setWithdrawalDelayBlocks(16 days / 12 seconds)` — a legitimate operational update.
4. At block `N + 57,601` (past the original 8-day delay), Alice calls `completeWithdrawal`. The check `block.number < request.withdrawalStartBlock + withdrawalDelayBlocks` evaluates as `N+57,601 < N + 115,200` → **reverts with `WithdrawalDelayNotPassed`**.
5. Alice's funds remain locked for an additional ~8 days beyond her original expectation, despite her rsETH having been transferred to the contract at step 2. [3](#0-2) [1](#0-0)
