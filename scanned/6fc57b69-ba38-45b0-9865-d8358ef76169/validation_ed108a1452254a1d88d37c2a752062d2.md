### Title
Frequent `getReward()` calls cause `rewardPerToken()` to round to zero, permanently losing staker rewards — (`contracts/KERNEL/KernelDepositPool.sol`)

---

### Summary

`KernelDepositPool` uses a Synthetix-style reward accumulator. The `updateReward` modifier, triggered by the permissionless `getReward()` function, advances `updatedAt` on every call. On L2 (Arbitrum), where block times are ~0.25 s, any caller — including one with zero staked balance — can call `getReward()` every block, keeping `timeDelta` at 1 second. When `rewardRate * 1 * 1e18 < totalKernelStaked`, the per-token increment rounds to zero and those rewards are permanently lost.

---

### Finding Description

`rewardPerToken()` computes the incremental reward per token as:

```solidity
// KernelDepositPool.sol line 412
rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
    / totalKernelStaked;
```

`DECIMAL_PRECISION = 1e18`. The `updateReward` modifier (line 232–242) is applied to `getReward()` (line 382), which is fully permissionless — no staked balance is required to call it. Each call sets `updatedAt = lastTimeRewardApplicable()` (line 234), resetting the time window.

**Rounding-to-zero condition (1-second window):**

```
rewardRate * 1 * 1e18 < totalKernelStaked
=> rewardRate < totalKernelStaked / 1e18
```

Concrete example:
- `totalKernelStaked = 1e24` (1 million KERNEL, 18 decimals)
- `duration = 30 days = 2,592,000 s`
- Annual reward budget = 2,000 reward tokens → `rewardRate = 2000e18 / 2592000 ≈ 771,604`
- Rounding condition: `771,604 * 1 * 1e18 = 7.7e23 < 1e24` ✓ → increment = 0

Every `getReward()` call in this window discards the accrued rewards for that second. Over a full 30-day period with per-block calls (~345,600 calls/day on Arbitrum), the entire reward budget can be zeroed out.

The `notifyRewardAmount` function also truncates `rewardRate = receivedAmount / duration`, permanently losing up to `duration - 1` wei per reward period, but that is a secondary, smaller issue.

---

### Impact Explanation

Stakers permanently lose their entitled reward tokens. The tokens remain locked in the contract with no recovery path. This matches **"Permanent freezing of unclaimed yield"** (Medium) or **"Theft of unclaimed yield"** (High) depending on scale. Given that the entire reward budget can be consumed by this attack over a reward period, the impact is a definite, total loss of staker yield.

---

### Likelihood Explanation

`getReward()` is permissionless and costs negligible gas on Arbitrum (~$0.01–$0.10 per call). An attacker with no staked balance can call it every block. Even without malicious intent, organic user activity (multiple stakers calling `getReward()` in quick succession) can trigger the same rounding loss at scale. The condition is more easily met as `totalKernelStaked` grows.

---

### Recommendation

1. **Accumulate before advancing the clock:** Only advance `updatedAt` when the increment is non-zero, or accumulate the remainder into a "dust" buffer that is added to the next interval.
2. **Minimum time gate:** Require `block.timestamp - updatedAt >= MIN_ACCUMULATION_INTERVAL` before updating `rewardPerTokenStored`, preventing sub-second or per-block checkpoint spam.
3. **Scale `rewardRate` up:** Store `rewardRate` with extra precision (e.g., multiply by `1e18` at storage time and divide at read time) to reduce the rounding window.

---

```

  totalKernelStaked = 1_000_000e18  (1 million KERNEL)

  for each block (every ~0.25 s, timeDelta = 1 s):

          rewardPerTokenStored unchanged
          updatedAt = block.timestamp  // clock advanced, rewards lost

  rewardPerTokenStored never increases → all stakers earn 0 rewards
