### Title
Race Condition in `blockUserTransfers` Renders `recoverFrozenFunds` Ineffective - (File: `contracts/RSETH.sol`)

### Summary

`RSETH.sol` implements a two-step fund-freeze-and-recovery mechanism: `blockUserTransfers` sets a 24-hour transfer block on a user, and `recoverFrozenFunds` then sweeps that user's entire rsETH balance to a custody address. Because `blockUserTransfers` is a plain on-chain transaction with no atomicity guarantee relative to the user's own transfers, a targeted user who monitors the mempool can front-run the block by calling `transfer` before the block transaction is mined, draining their balance to a fresh address and leaving `recoverFrozenFunds` with nothing to recover.

### Finding Description

`RSETH` overrides ERC-20 `_transfer` to call `_enforceNotBlocked` on both `from` and `to`:

```solidity
// contracts/RSETH.sol:287-291
function _transfer(address from, address to, uint256 amount) internal override {
    _enforceNotBlocked(from);
    _enforceNotBlocked(to);
    super._transfer(from, to, amount);
}
```

The block itself is written by `blockUserTransfers`:

```solidity
// contracts/RSETH.sol:161-177
function blockUserTransfers(address[] calldata accounts) external onlyLRTManager {
    uint256 blockedUntil = block.timestamp + 1 days;
    ...
    transfersBlockedUntil[account] = blockedUntil;
    ...
}
```

And the recovery step reads the balance **after** the block is confirmed:

```solidity
// contracts/RSETH.sol:206-220
function recoverFrozenFunds(address from) external onlyLRTAdmin {
    ...
    uint256 accountBalance = balanceOf(from);
    super._transfer(from, custodyAddress, accountBalance);
    emit FrozenFundsRecovered(from, custodyAddress, accountBalance);
}
```

There is no atomicity between the block write and the user's ability to transfer. The window between the manager broadcasting `blockUserTransfers` and that transaction being mined is exploitable by any user who watches the public mempool.

### Impact Explanation

`recoverFrozenFunds` is the protocol's only mechanism to seize rsETH from a bad actor (e.g., a compromised or fraudulent address). If the targeted user empties their balance before the block lands, `recoverFrozenFunds` recovers zero tokens — the entire fund-recovery guarantee is nullified. The bad actor retains full control of the rsETH at a new address, permanently defeating the intended security control. This maps to **Medium — Temporary freezing of funds** (the freeze is bypassed entirely) and, depending on the context of the recovery (e.g., stolen rsETH being clawed back), can escalate to **Critical — Direct theft of user funds**.

### Likelihood Explanation

Likelihood is **Medium**. Monitoring the public Ethereum mempool is trivial with standard tooling (e.g., Flashbots, ethers.js pending-transaction subscriptions). Any user who has reason to expect a block (e.g., after a protocol incident or public announcement) can automate a front-run transfer. The attacker only needs to pay a higher gas price than the manager's transaction, which is a low bar on Ethereum mainnet.

### Recommendation

Adopt the same mitigation suggested in the reference report: **pause the RSETH contract before calling `blockUserTransfers`**. Because `_transfer` inherits `whenNotPaused` via `PausableUpgradeable`, a confirmed `pause()` call atomically prevents all user transfers, closing the race window. The sequence should be:

1. Call `pause()` with a high gas price and wait for confirmation.
2. Call `blockUserTransfers([target])`.
3. Call `recoverFrozenFunds(target)`.
4. Call `unpause()`.

Alternatively, combine the pause and block into a single atomic multicall, or add a `whenNotPaused` guard directly to `_transfer` (which `RSETH` already supports via `PausableUpgradeable`).

### Proof of Concept

1. Manager broadcasts `blockUserTransfers([Bob])` (standard gas price).
2. Bob's mempool watcher detects the pending transaction.
3. Bob immediately broadcasts `transfer(aliceAddress, balanceOf(Bob))` with a higher gas price.
4. Bob's transfer is mined first — `_enforceNotBlocked` passes because `transfersBlockedUntil[Bob] == 0` at that point.
5. `blockUserTransfers` is mined — `transfersBlockedUntil[Bob]` is now set, but Bob's balance is 0.
6. Admin calls `recoverFrozenFunds(Bob)` — `balanceOf(Bob) == 0`, so `super._transfer` moves 0 tokens to `custodyAddress`.
7. Bob controls all his original rsETH at `aliceAddress`, which was never blocked.

```text
        uint256 blockedUntil = block.timestamp + 1 days;
```

```text
    function _enforceNotBlocked(address account) internal {
        // Addresses that are permanently exempt can never be blocked
        if (isPermanentlyExempt[account]) return;

        // Check if the account has an active transfer block
        uint256 blockedUntil = transfersBlockedUntil[account];
        if (blockedUntil == 0) return;

        if (block.timestamp < blockedUntil) revert TransfersBlocked(account, blockedUntil);

        // Auto-clean up expired block
        delete transfersBlockedUntil[account];
    }
```
