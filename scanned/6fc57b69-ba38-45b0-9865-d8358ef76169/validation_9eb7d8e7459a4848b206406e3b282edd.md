### Title
Missing Time-Based Staleness Check in `ChainlinkOracleForRSETHPoolCollateral.getRate()` Allows Stale Price Consumption During Token Deposits - (File: `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol`)

---

### Summary

`ChainlinkOracleForRSETHPoolCollateral.getRate()` performs partial Chainlink data validation but omits the essential time-based staleness check (`block.timestamp - timestamp > heartbeat`). This allows a stale collateral token price to be consumed by pool deposit flows, enabling depositors to receive more rsETH than the current fair value of their collateral warrants.

---

### Finding Description

`ChainlinkOracleForRSETHPoolCollateral.getRate()` calls `latestRoundData()` and validates three conditions:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [1](#0-0) 

None of these checks verify that the price was updated within an acceptable time window. The `answeredInRound < roundID` check is a deprecated Chainlink pattern that Chainlink itself no longer guarantees to be meaningful — it does not substitute for a `block.timestamp - updatedAt > heartbeat` check. The `timestamp == 0` check only catches a completely uninitialized round, not a round that is hours or days old.

The missing check is:

```solidity
if (block.timestamp - timestamp > HEARTBEAT) revert StalePrice();
```

This is the direct analog of the missing `block.timestamp > 2 hours + baseTimeStamp` condition described in the reference report.

This oracle is registered as `supportedTokenOracle[token]` in the pool contracts and its `getRate()` return value is used directly in `viewSwapRsETHAmountAndFee(amount, token)`:

```solidity
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [2](#0-1) [3](#0-2) 

The stale `tokenToETHRate` is then used to mint rsETH to the depositor:

```solidity
wrsETH.mint(msg.sender, rsETHAmount);
``` [4](#0-3) 

---

### Impact Explanation

If the Chainlink feed for a supported collateral token (e.g., wstETH/ETH) goes stale while the token's real market price has dropped, the oracle continues to report the old, higher price. A depositor calling `deposit(token, amount, referralId)` will receive rsETH calculated at the inflated stale rate, receiving more rsETH than the current fair value of their collateral. This directly drains the pool's pre-minted rsETH reserves at the expense of other pool participants.

Impact classification: **Low to Medium** — the pool's rsETH reserves are depleted by the difference between the stale price and the true price, for every deposit made during the staleness window. This constitutes the contract failing to deliver promised returns / theft of pool assets depending on the magnitude of price drift.

---

### Likelihood Explanation

Chainlink feeds go stale during periods of network congestion, oracle node downtime, or when price movement is slow enough that the deviation threshold is never triggered but the heartbeat window expires. For feeds like wstETH/ETH on L2 networks (where these pool contracts are deployed), heartbeat windows can be 24 hours, making extended staleness realistic. Any unprivileged depositor can exploit this condition the moment it occurs — no special access is required.

---

### Recommendation

Add a configurable heartbeat constant and enforce the time-based staleness check in `getRate()`:

```diff
+uint256 public immutable heartbeat;

+constructor(address _oracle, uint256 _heartbeat) {
     oracle = _oracle;
+    heartbeat = _heartbeat;
 }

 function getRate() public view returns (uint256) {
     (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
         AggregatorV3Interface(oracle).latestRoundData();

     if (answeredInRound < roundID) revert StalePrice();
     if (timestamp == 0) revert IncompleteRound();
+    if (block.timestamp - timestamp > heartbeat) revert StalePrice();
     if (ethPrice <= 0) revert InvalidPrice();

     uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
     return normalizedPrice;
 }
```

### Proof of Concept

1. Admin registers a supported collateral token (e.g., wstETH) in `RSETHPoolV3` with `supportedTokenOracle[wstETH] = address(new ChainlinkOracleForRSETHPoolCollateral(wstETH_ETH_feed))`.
2. The Chainlink wstETH/ETH feed goes stale — last update was 25 hours ago at a price of 1.15 ETH/wstETH. The real current price has dropped to 1.05 ETH/wstETH due to a depeg event, but the feed has not updated.
3. Attacker calls `deposit(wstETH, 100e18, "")` on `RSETHPoolV3`.
4. `viewSwapRsETHAmountAndFee` calls `ChainlinkOracleForRSETHPoolCollateral.getRate()`. The checks pass: `answeredInRound` is not less than `roundID` (same round), `timestamp != 0`, `ethPrice > 0`. No staleness revert occurs.
5. `tokenToETHRate` = 1.15e18 (stale). `rsETHToETHrate` = current rsETH price ≈ 1.05e18.
6. Attacker receives `100e18 * 1.15e18 / 1.05e18 ≈ 109.5 rsETH` for 100 wstETH that is only worth ~105 ETH at current prices.
7. The pool has overpaid ~4.5 rsETH worth of value, which the attacker can redeem or sell. [5](#0-4) [6](#0-5)
