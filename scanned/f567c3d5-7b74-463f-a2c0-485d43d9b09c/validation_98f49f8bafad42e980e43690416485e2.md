### Title
No Minimum rsETH Output in L2 Pool `deposit()` Allows Fee-Change Race to Silently Reduce User Returns - (File: contracts/pools/RSETHPoolV3.sol)

### Summary
The `deposit()` functions in the L2 pool contracts (`RSETHPoolV3`, `RSETHPoolV2`, `RSETHPool`, `RSETHPoolV2ExternalBridge`, `AGETHPoolV3`) accept ETH or tokens and mint rsETH/agETH based on the current `feeBps` and oracle rate, but expose no `minAmountOut` parameter. Because `setFeeBps` is callable by `DEFAULT_ADMIN_ROLE` without a timelock, a fee increase that lands in the same block as a user's deposit silently reduces the rsETH minted, with no on-chain protection for the user.

### Finding Description

`RSETHPoolV3.deposit()` computes the rsETH output as:

```solidity
// contracts/pools/RSETHPoolV3.sol L299-307
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
```

The `feeBps` storage variable is read at execution time. The admin can update it at any moment:

```solidity
// contracts/pools/RSETHPoolV3.sol L518-521
function setFeeBps(uint256 _feeBps) external onlyRole(DEFAULT_ADMIN_ROLE) {
    if (_feeBps > 1000) revert InvalidFeeAmount();
    feeBps = _feeBps;
    emit FeeBpsSet(_feeBps);
}
```

The ETH-deposit entry point carries no slippage guard:

```solidity
// contracts/pools/RSETHPoolV3.sol L246-265
function deposit(string memory referralId) external payable nonReentrant whenNotPaused ... {
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);   // no minAmountOut check
    ...
}
```

The same pattern applies to the token-deposit overload and to `RSETHPoolV2`, `RSETHPool`, `RSETHPoolV2ExternalBridge`, and `AGETHPoolV3`.

By contrast, the L1 `LRTDepositPool` already enforces this protection:

```solidity
// contracts/LRTDepositPool.sol L667-669
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

The L2 pool contracts are missing the equivalent guard entirely.

### Impact Explanation
A depositor who previews the expected rsETH output off-chain (or via `viewSwapRsETHAmountAndFee`) and then submits a `deposit()` transaction can receive materially fewer tokens than expected if `feeBps` is raised before their transaction is mined. With the maximum allowed fee of 1000 bps (10%), a user depositing 10 ETH expecting ~9.09 rsETH at 0 bps could receive ~8.18 rsETH at 1000 bps — a silent 10% shortfall. The contract fails to deliver the promised return without reverting, matching the **Low** impact class: *contract fails to deliver promised returns, but doesn't lose value*.

### Likelihood Explanation
The `DEFAULT_ADMIN_ROLE` can call `setFeeBps` in a single transaction with no timelock delay. Any fee update that is mined in the same block as a pending user deposit triggers the issue. This is a realistic operational scenario (fee adjustment during normal protocol maintenance) and does not require any malicious intent — the admin may simply be raising fees legitimately while users are actively depositing.

### Recommendation
Add a `minRSETHAmountExpected` (or `minAgETHAmountExpected`) parameter to all `deposit()` functions in the L2 pool contracts, mirroring the existing guard in `LRTDepositPool._beforeDeposit()`. Revert if the computed output falls below the caller-supplied minimum. Additionally, consider applying a timelock to `setFeeBps` in contracts where it is currently gated only by `DEFAULT_ADMIN_ROLE`.

### Proof of Concept

1. `feeBps` is currently 0 in `RSETHPoolV3`.
2. Bob calls `viewSwapRsETHAmountAndFee(10 ether)` off-chain → expects `rsETHAmount = X`.
3. Bob submits `deposit{value: 10 ether}("ref")`.
4. Admin submits `setFeeBps(1000)` with a higher gas price; it is mined first in the same block.
5. Bob's `deposit` executes with `feeBps = 1000`: `fee = 1 ether`, `amountAfterFee = 9 ether`, `rsETHAmount = X * 0.9`.
6. Bob receives 10% fewer rsETH than previewed, with no revert and no recourse. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
