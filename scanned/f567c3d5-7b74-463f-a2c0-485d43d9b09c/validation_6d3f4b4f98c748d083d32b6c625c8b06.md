### Title
`withdrawalDelayBlocks` Can Be Increased Unexpectedly, Temporarily Freezing In-Flight Withdrawal Completions — (`contracts/LRTWithdrawalManager.sol`)

---

### Summary

The `LRTWithdrawalManager` allows the LRT manager to change `withdrawalDelayBlocks` at any time with immediate effect. Because the completion check uses the *current* value of `withdrawalDelayBlocks` rather than the value that was in force when the request was created, a manager action that increases the delay retroactively re-locks withdrawal requests that users had already waited out, temporarily freezing their funds.

---

### Finding Description

`setWithdrawalDelayBlocks` writes the new delay directly to storage with no queuing or per-request snapshotting: [1](#0-0) 

```solidity
function setWithdrawalDelayBlocks(uint256 withdrawalDelayBlocks_) external onlyLRTManager {
    // Set an upper limit of no more than 16 days
    if (withdrawalDelayBlocks_ > 16 days / 12 seconds) revert ExceedWithdrawalDelay();
    withdrawalDelayBlocks = withdrawalDelayBlocks_;
    emit WithdrawalDelayBlocksUpdated(withdrawalDelayBlocks);
}
```

Both the user-facing `completeWithdrawal` path and the operator-facing `unlockQueue` path read the *live* `withdrawalDelayBlocks` at execution time: [2](#0-1) 

```solidity
// Check that the withdrawal delay has passed since the request's initiation.
if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();
``` [3](#0-2) 

```solidity
// Check that the withdrawal delay has passed since the request's initiation.
if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) break;
```

The `WithdrawalRequest` struct stores only `withdrawalStartBlock`; it does not capture the delay that was in force at request time: [4](#0-3) 

```solidity
withdrawalRequests[requestId] = WithdrawalRequest({
    rsETHUnstaked: rsETHUnstaked, expectedAssetAmount: expectedAssetAmount, withdrawalStartBlock: block.number
});
```

**Scenario:**

1. Default `withdrawalDelayBlocks` = `8 days / 12 s` ≈ 57,600 blocks.
2. User calls `initiateWithdrawal`, locking their rsETH in the contract.
3. User waits 57,600 blocks. Off-chain check confirms `block.number >= startBlock + withdrawalDelayBlocks` → eligible.
4. Manager calls `setWithdrawalDelayBlocks(16 days / 12 seconds)` (≈ 115,200 blocks) — a legitimate security response to a detected risk.
5. User's `completeWithdrawal` reverts with `WithdrawalDelayNotPassed`. The user's rsETH is already burned from their wallet (transferred to the contract at step 2) and they cannot recover it for another ~8 days.
6. `unlockQueue` is equally blocked for the same requests, so the operator cannot unlock them either.

---

### Impact Explanation

**Temporary freezing of funds.** Users who have already surrendered their rsETH to the contract and waited the full original delay period are unable to receive their underlying asset. Their funds are locked in `LRTWithdrawalManager` for an additional period of up to 8 days (the maximum increase from 8 → 16 days). This matches the "Temporary freezing of funds" impact tier.

---

### Likelihood Explanation

The LRT manager role is an operational role expected to adjust protocol parameters in response to market or security events. Increasing `withdrawalDelayBlocks` is a plausible and legitimate action (e.g., after a vulnerability disclosure, during a market stress event). No compromise or malicious intent is required — the side-effect on in-flight requests is an unintended consequence of the immediate application of the new value. The manager role is distinct from the admin and is held by a smaller set of addresses, making this a realistic operational scenario.

---

### Recommendation

Snapshot `withdrawalDelayBlocks` into each `WithdrawalRequest` at the time `initiateWithdrawal` is called, and use the stored value — not the live global — when checking completion eligibility:

```solidity
struct WithdrawalRequest {
    uint256 rsETHUnstaked;
    uint256 expectedAssetAmount;
    uint256 withdrawalStartBlock;
    uint256 withdrawalDelayBlocksSnapshot; // add this
}
```

In `_addUserWithdrawalRequest`:
```solidity
withdrawalRequests[requestId] = WithdrawalRequest({
    rsETHUnstaked: rsETHUnstaked,
    expectedAssetAmount: expectedAssetAmount,
    withdrawalStartBlock: block.number,
    withdrawalDelayBlocksSnapshot: withdrawalDelayBlocks
});
```

In `_processWithdrawalCompletion` and `_unlockWithdrawalRequests`, replace:
```solidity
if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) ...
```
with:
```solidity
if (block.number < request.withdrawalStartBlock + request.withdrawalDelayBlocksSnapshot) ...
```

This ensures that changes to `withdrawalDelayBlocks` apply only to future requests, preserving the T&Cs under which existing users initiated their withdrawals.

---

### Proof of Concept

```solidity
// 1. User initiates withdrawal
vm.prank(user);
withdrawalManager.initiateWithdrawal(asset, rsETHAmount, "");

// 2. Fast-forward past the original 8-day delay
vm.roll(block.number + (8 days / 12 seconds) + 1);

// 3. Manager increases delay to 16 days (legitimate security action)
vm.prank(manager);
withdrawalManager.setWithdrawalDelayBlocks(16 days / 12 seconds);

// 4. User's completeWithdrawal now reverts — funds frozen for another ~8 days
vm.prank(user);
vm.expectRevert(ILRTWithdrawalManager.WithdrawalDelayNotPassed.selector);
withdrawalManager.completeWithdrawal(asset, "");
```
