### Title
`withdrawalDelayBlocks` Applied Retroactively to Already-Queued Withdrawals Causes Temporary Fund Freeze - (File: contracts/LRTWithdrawalManager.sol)

### Summary
`LRTWithdrawalManager.sol` reads the current `withdrawalDelayBlocks` value at withdrawal completion time rather than snapshotting it at request creation. A manager updating this parameter after users have already initiated withdrawals and transferred their rsETH retroactively extends the delay those users must wait, temporarily freezing their funds.

### Finding Description
When a user calls `initiateWithdrawal`, their rsETH is immediately transferred to the contract and a `WithdrawalRequest` struct is stored with `withdrawalStartBlock = block.number`. However, the delay parameter `withdrawalDelayBlocks` is **not** stored per-request. [1](#0-0) 

At completion time, both `_processWithdrawalCompletion` and `_unlockWithdrawalRequests` read the **current** `withdrawalDelayBlocks` from storage: [2](#0-1) [3](#0-2) 

The manager can update this value at any time via `setWithdrawalDelayBlocks`, up to a ceiling of 16 days: [4](#0-3) 

### Impact Explanation
A user initiates a withdrawal when `withdrawalDelayBlocks` equals 8 days (~57,600 blocks). They wait the full 8 days. Before they call `completeWithdrawal`, the manager legitimately updates `withdrawalDelayBlocks` to 16 days (~115,200 blocks) — for example, to align with a new EigenLayer withdrawal delay. The user's `completeWithdrawal` now reverts with `WithdrawalDelayNotPassed` even though they already waited the required period at request time. Their rsETH is held in the contract and cannot be recovered until the new 16-day delay elapses from their original `withdrawalStartBlock`.

Impact: **Medium — Temporary freezing of funds.** The user's rsETH is locked in the contract for up to an additional 8 days beyond what was promised at request time.

### Likelihood Explanation
The manager role is an operational role expected to adjust protocol parameters in response to external changes (e.g., EigenLayer delay adjustments). A manager increasing `withdrawalDelayBlocks` for legitimate operational reasons would unintentionally retroactively freeze all in-flight withdrawal requests. No malicious intent is required; the design flaw makes any upward adjustment harmful to existing requestors.

### Recommendation
Snapshot `withdrawalDelayBlocks` into the `WithdrawalRequest` struct at the time `initiateWithdrawal` is called, and use the stored value — not the current global value — when checking the delay in `_processWithdrawalCompletion` and `_unlockWithdrawalRequests`.

```solidity
struct WithdrawalRequest {
    uint256 rsETHUnstaked;
    uint256 expectedAssetAmount;
    uint256 withdrawalStartBlock;
    uint256 withdrawalDelayBlocksSnapshot; // add this
}
```

Then in `_addUserWithdrawalRequest`:
```solidity
withdrawalRequests[requestId] = WithdrawalRequest({
    rsETHUnstaked: rsETHUnstaked,
    expectedAssetAmount: expectedAssetAmount,
    withdrawalStartBlock: block.number,
    withdrawalDelayBlocksSnapshot: withdrawalDelayBlocks
});
```

And replace both delay checks with `request.withdrawalDelayBlocksSnapshot`.

### Proof of Concept
1. `withdrawalDelayBlocks` = 57,600 blocks (8 days).
2. Alice calls `initiateWithdrawal(asset, rsETHAmount, "")`. Her rsETH is transferred to the contract. `request.withdrawalStartBlock = N`.
3. 57,600 blocks pass. Alice is now eligible to complete her withdrawal.
4. Manager calls `setWithdrawalDelayBlocks(115200)` (16 days) for a legitimate operational reason.
5. Alice calls `completeWithdrawal(asset, "")`. The check `block.number < N + 115200` is true (only ~57,600 blocks have passed), so the call reverts with `WithdrawalDelayNotPassed`.
6. Alice's rsETH remains locked in `LRTWithdrawalManager` for another ~57,600 blocks (~8 days) beyond what was promised. [5](#0-4)
