### Title
Lack of Slippage Protection in `deposit()` Functions Across L2 Pool Contracts - (`contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`, `contracts/pools/RSETHPoolV3WithNativeChainBridge.sol`)

---

### Summary

The L2 pool contracts (`RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`) expose public `deposit()` functions that compute the wrsETH output amount at execution time using a live oracle rate, but accept no `minRsETHAmountOut` parameter. A user who previews the output via `viewSwapRsETHAmountAndFee()` before submitting a transaction has no on-chain guarantee that the minted amount will meet their expectation. This is the direct analog of the M-03 finding: the L1 `LRTDepositPool` already enforces slippage protection via `minRSETHAmountExpected`, but the L2 pool contracts do not.

---

### Finding Description

`LRTDepositPool.depositETH()` and `LRTDepositPool.depositAsset()` both accept a `minRSETHAmountExpected` parameter and revert inside `_beforeDeposit()` if the computed mint amount falls below it:

```solidity
// contracts/LRTDepositPool.sol L76-87
function depositETH(uint256 minRSETHAmountExpected, string calldata referralId) ...
    uint256 rsethAmountToMint = _beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, minRSETHAmountExpected);
```

```solidity
// contracts/LRTDepositPool.sol L665-668
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

The L2 pool contracts have no equivalent guard. Both overloads of `deposit()` in `RSETHPoolV3` compute the output amount at execution time and mint immediately with no floor check:

```solidity
// contracts/pools/RSETHPoolV3.sol L246-265 (ETH path)
function deposit(string memory referralId) external payable ... {
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);   // no minRsETHAmountOut check
}

// contracts/pools/RSETHPoolV3.sol L271-293 (token path)
function deposit(address token, uint256 amount, string memory referralId) external ... {
    IERC20(token).safeTransferFrom(msg.sender, address(this), amount);
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
    feeEarnedInToken[token] += fee;
    wrsETH.mint(msg.sender, rsETHAmount);   // no minRsETHAmountOut check
}
```

The output is determined entirely by `viewSwapRsETHAmountAndFee()`, which divides the fee-adjusted input by the live oracle rate:

```solidity
// contracts/pools/RSETHPoolV3.sol L299-308
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();          // live oracle read
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
```

`RSETHPoolV3ExternalBridge` and `RSETHPoolV3WithNativeChainBridge` contain identical patterns with no slippage guard.

---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A user calls `viewSwapRsETHAmountAndFee()` off-chain to preview the wrsETH they will receive, then submits `deposit()`. If the oracle rate is updated upward between preview and execution (rsETH appreciates in ETH terms), the user receives fewer wrsETH tokens than previewed. The user has no on-chain mechanism to revert the transaction if the output falls below their acceptable threshold. The deposited ETH or tokens are consumed and the user receives fewer shares than expected with no recourse.

---

### Likelihood Explanation

**Medium.** The rsETH oracle rate increases monotonically over time as staking rewards accrue. On L2 chains, oracle updates are pushed periodically (not per-block), so the rate seen during a preview can differ from the rate at execution time. Any user who previews and then submits a transaction during a period straddling an oracle update will silently receive fewer tokens than expected. No adversarial action is required — normal protocol operation is sufficient to trigger the discrepancy.

---

### Recommendation

Add a `minRsETHAmountOut` parameter to both `deposit()` overloads in `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, and `RSETHPoolV3WithNativeChainBridge`, mirroring the pattern already used in `LRTDepositPool`:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountOut)
    external payable nonReentrant whenNotPaused limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountOut) revert SlippageExceeded();
    ...
}
```

Apply the same guard to the token-path overload. This aligns the L2 pool contracts with the slippage protection already present in the L1 deposit pool.

---

### Proof of Concept

1. User calls `RSETHPoolV3.viewSwapRsETHAmountAndFee(1 ether)` off-chain. Oracle rate is `1.05e18` (1 rsETH = 1.05 ETH). Expected output: `~0.952 wrsETH`.
2. Before the user's transaction is included, the oracle is updated to `1.10e18`.
3. User's `deposit{value: 1 ether}(referralId)` executes. `viewSwapRsETHAmountAndFee` now returns `~0.909 wrsETH` (≈4.5% less than expected).
4. `wrsETH.mint(msg.sender, rsETHAmount)` mints `0.909 wrsETH` with no revert. The user has no recourse.

The same scenario applies to the token-path `deposit(token, amount, referralId)` in all three pool contracts, and to `RSETHPoolV3ExternalBridge` and `RSETHPoolV3WithNativeChainBridge` which share identical logic.
