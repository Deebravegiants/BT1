### Title
Missing `setSupportedTokenOracle` Makes Collateral Token Oracle Permanently Unupdatable While Pool Holds Funds - (File: contracts/agETH/AGETHPoolV3.sol)

### Summary
`AGETHPoolV3` has no function to update the oracle address for an already-supported collateral token. Unlike `RSETHPoolV3`, which has a dedicated `setSupportedTokenOracle` function, `AGETHPoolV3` only allows oracle updates indirectly by removing and re-adding the token — but `removeSupportedToken` is blocked whenever the pool holds any balance of that token. This means that as long as the pool is in normal operation (holding deposited collateral), the oracle for any collateral token is effectively immutable.

### Finding Description
`AGETHPoolV3.addSupportedToken` guards against re-registration:

```solidity
if (supportedTokenOracle[token] != address(0)) {
    revert AlreadySupportedToken();
}
```

The only escape hatch is `removeSupportedToken`, which hard-blocks on any non-zero balance:

```solidity
if (IERC20(token).balanceOf(address(this)) != 0) revert TokenBalanceNotZero();
```

Even after `moveAssetsForBridging(token)` is called, the contract retains `feeEarnedInToken[token]` wei of that token, keeping the balance non-zero unless fees are also explicitly withdrawn first. There is no `setSupportedTokenOracle` function anywhere in `AGETHPoolV3`.

By contrast, `RSETHPoolV3` (the rsETH-side equivalent) ships with an explicit `setSupportedTokenOracle` that allows the oracle to be swapped at any time:

```solidity
function setSupportedTokenOracle(address token, address oracle)
    external onlyRole(TIMELOCK_ROLE) onlySupportedToken(token)
{ ... }
```

`AGETHPoolV3` is missing this function entirely.

### Impact Explanation
If the Chainlink oracle wrapper (`ChainlinkOracleForRSETHPoolCollateral`) used for a collateral token in `AGETHPoolV3` is deprecated, starts reverting, or returns a stale/incorrect rate, the admin cannot swap it for a new oracle while the pool holds any balance of that token. All `deposit(token, amount, referralId)` calls for that token will revert (because `viewSwapAgETHAmountAndFee` calls `IOracle(supportedTokenOracle[token]).getRate()`), causing a temporary freeze of that deposit path. Users holding that collateral token cannot mint agETH through the pool until the pool is fully drained of that token — an operational action that may not be immediately possible.

**Impact: Medium — Temporary freezing of funds / DoS of a deposit path.**

### Likelihood Explanation
Chainlink periodically deprecates price feeds and migrates to new aggregator addresses. The `ChainlinkOracleForRSETHPoolCollateral` wrapper stores the aggregator address as `immutable`, so a deprecated feed cannot be patched in-place; a new wrapper must be deployed and the pool's oracle pointer updated. Because `AGETHPoolV3` provides no direct oracle-update path, this scenario is a realistic operational risk whenever Chainlink retires a feed used by the pool.

### Recommendation
Add a `setSupportedTokenOracle` function to `AGETHPoolV3`, mirroring the one already present in `RSETHPoolV3`:

```solidity
function setSupportedTokenOracle(address token, address oracle)
    external
    onlyRole(DEFAULT_ADMIN_ROLE)
    onlySupportedToken(token)
{
    UtilLib.checkNonZeroAddress(oracle);
    if (IOracle(oracle).getRate() == 0) revert UnsupportedOracle();
    supportedTokenOracle[token] = oracle;
    emit TokenOracleSet(token, oracle);
}
```

### Proof of Concept

1. Admin calls `AGETHPoolV3.addSupportedToken(wstETH, chainlinkWstETHOracle)`.
2. Users deposit wstETH; the pool accumulates a wstETH balance and fee balance.
3. Chainlink deprecates the wstETH/ETH feed; `chainlinkWstETHOracle.getRate()` begins reverting.
4. Every call to `deposit(wstETH, amount, ref)` now reverts because `viewSwapAgETHAmountAndFee` calls `IOracle(supportedTokenOracle[wstETH]).getRate()`.
5. Admin attempts `addSupportedToken(wstETH, newOracle)` → reverts with `AlreadySupportedToken`.
6. Admin attempts `removeSupportedToken(wstETH, idx)` → reverts with `TokenBalanceNotZero` because the pool still holds wstETH fees.
7. The wstETH deposit path is frozen until the bridger drains all wstETH (including fees) — an out-of-band operational action that may not be immediately executable.
