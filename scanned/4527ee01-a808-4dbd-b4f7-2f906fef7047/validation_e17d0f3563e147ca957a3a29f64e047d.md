### Title
Stale Rate Consumed Without Freshness Check Enables Block-Stuffing-Assisted Over-Minting in RSETHPoolV3 — (`contracts/cross-chain/MultiChainRateProvider.sol`)

---

### Summary

`MultiChainRateProvider.updateRate()` is permissionless and pushes the L1 rsETH/ETH rate to destination chains via LayerZero. The destination-chain receiver (`CrossChainRateReceiver`) stores the rate in a plain storage slot with no expiry. `RSETHPoolV3.deposit()` consumes that stored rate with zero staleness validation. An attacker who stuffs destination-chain blocks to delay LayerZero message delivery can deposit at the stale (lower) rate and receive more rsETH than the current L1 rate entitles them to.

---

### Finding Description

**Rate broadcast path (L1 → L2):**

1. `MultiChainRateProvider.updateRate()` — permissionless, reads `getLatestRate()` from `RSETHMultiChainRateProvider` → `ILRTOracle.rsETHPrice()`, then calls `ILayerZeroEndpoint.send()` to each registered receiver. [1](#0-0) 

2. `CrossChainRateReceiver.lzReceive()` — stores the decoded rate into `rate` and records `lastUpdated = block.timestamp`. The public `getRate()` returns `rate` with **no check on `lastUpdated`**. [2](#0-1) 

3. `RSETHPoolV3.viewSwapRsETHAmountAndFee()` — calls `getRate()` on the oracle (the receiver above) and computes `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate`. No staleness guard exists anywhere in this path. [3](#0-2) 

4. `RSETHPoolV3.deposit()` — mints directly from the result of `viewSwapRsETHAmountAndFee`. [4](#0-3) 

**Block-stuffing attack surface:**

`updateRate()` on L1 is permissionless, so anyone can call it. The LayerZero relayer then delivers the message to L2 by calling `lzReceive()`. On low-cost L2 chains (Arbitrum, Optimism, Base, etc.) an attacker can fill every block with high-priority transactions to delay the relayer's `lzReceive()` call. During this window the receiver's `rate` remains at the pre-increase value.

**Invariant broken:**

When rsETH price on L1 increases by `Δ` (e.g. from staking yield accrual), the stale rate `r_stale < r_current`. The deposit formula yields:

```
rsETHAmount_stale  = ETH * 1e18 / r_stale
rsETHAmount_current = ETH * 1e18 / r_current
excess = rsETHAmount_stale - rsETHAmount_current > 0
```

The excess rsETH is unbacked at the current L1 rate, diluting existing holders' yield.

The `limitDailyMint` modifier uses the same stale rate to compute the rsETH cap, so the daily limit is also evaluated against the inflated amount — it does not independently cap the ETH-equivalent value. [5](#0-4) 

---

### Impact Explanation

Every unit of excess rsETH minted at the stale rate represents yield that belongs to existing rsETH holders but is instead captured by the attacker. The daily mint limit bounds the per-day loss but does not prevent it. Across multiple days or multiple destination chains the cumulative theft of unclaimed yield can be material.

---

### Likelihood Explanation

- `updateRate()` is permissionless; the attacker only needs to delay the LayerZero relayer's L2 delivery.
- On low-fee L2 chains, block stuffing is significantly cheaper than on Ethereum mainnet.
- rsETH yield accrues continuously; any non-trivial delay between rate broadcasts creates a profitable window.
- No on-chain staleness check exists to revert or discount the deposit.
- The attack requires no privileged role, no governance capture, and no external protocol compromise.

---

### Recommendation

1. **Add a staleness guard in `CrossChainRateReceiver.getRate()`** — revert if `block.timestamp - lastUpdated > MAX_STALENESS` (e.g. 24 h).
2. **Add a staleness guard in `RSETHPoolV3.viewSwapRsETHAmountAndFee()`** — read `lastUpdated` from the oracle interface and revert if stale before computing the mint amount.
3. **Expose `lastUpdated` via the oracle interface** so pool contracts can enforce freshness independently.

//    with rate = 1.00e18 (stale, pre-yield)

//    but do NOT call lzReceive on the receiver (simulates stuffed blocks)

//    rsETH_stale  = 100e18 * 1e18 / 1.00e18 = 100e18
// 4. If rate were current:
//    rsETH_current = 100e18 * 1e18 / 1.01e18 ≈ 99.0099e18
// 5. Excess minted = 100e18 - 99.0099e18 ≈ 0.9901 rsETH per 100 ETH
//    At 1.01 ETH/rsETH this is ~1 ETH of stolen yield per 100 ETH deposit
// 6. Assert rsETH_stale > rsETH_current — differential confirms invariant break
