### Title
Fee-on-Transfer Token Deposit Mints Excess rsETH Based on Input Amount Rather Than Actual Received Amount - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool.depositAsset` calculates the rsETH to mint using the caller-supplied `depositAmount` before executing `safeTransferFrom`. If a supported asset implements a transfer fee, the contract receives fewer tokens than `depositAmount`, yet mints rsETH as if the full amount arrived. This inflates rsETH supply relative to actual backing assets, causing protocol insolvency for all rsETH holders.

### Finding Description
In `depositAsset`, `_beforeDeposit` computes `rsethAmountToMint` from the raw input `depositAmount`:

```solidity
// LRTDepositPool.sol lines 111–115
uint256 rsethAmountToMint = _beforeDeposit(asset, depositAmount, minRSETHAmountExpected);

IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
_mintRsETH(rsethAmountToMint);
``` [1](#0-0) 

Inside `_beforeDeposit`, the mint amount is derived purely from the caller-controlled `depositAmount`:

```solidity
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
``` [2](#0-1) 

`getRsETHAmountToMint` scales `amount` by the oracle price and current rsETH price — no balance snapshot is taken before or after the transfer to verify what actually arrived. [3](#0-2) 

`addNewSupportedAsset` (callable by `TIME_LOCK_ROLE`) imposes no restriction on fee-on-transfer tokens:

```solidity
function _addNewSupportedAsset(address asset, uint256 depositLimit) private {
    UtilLib.checkNonZeroAddress(asset);
    if (depositLimit == 0) revert InvalidDepositLimit();
    if (isSupportedAsset[asset]) revert AssetAlreadySupported();
    isSupportedAsset[asset] = true;
    ...
}
``` [4](#0-3) 

### Impact Explanation
When a fee-on-transfer LST is a supported asset, every depositor receives rsETH proportional to `depositAmount` while the contract holds only `depositAmount × (1 − fee)`. `getTotalAssetDeposits` reads actual `balanceOf` values: [5](#0-4) 

So the real asset backing per rsETH is permanently lower than the rsETH price implies. Each subsequent depositor mints rsETH at a price computed from an understated asset base, and each redemption drains more real assets than the redeemer's fair share. The cumulative shortfall grows with deposit volume, eventually making the protocol insolvent — existing rsETH holders cannot redeem at par. **Impact: Critical — Protocol insolvency.**

### Likelihood Explanation
The `TIME_LOCK_ROLE` can add any ERC-20 as a supported asset with no fee-on-transfer check. Several LSTs and yield-bearing tokens in the Ethereum ecosystem carry transfer fees or rebasing mechanics that reduce received amounts. A governance proposal to expand the supported asset list to a fee-bearing token — even unintentionally — immediately activates the vulnerability for every subsequent depositor. **Likelihood: Medium.**

### Recommendation
Record the contract's balance of `asset` before and after `safeTransferFrom` and use the delta as the actual deposited amount for rsETH minting:

```solidity
uint256 balanceBefore = IERC20(asset).balanceOf(address(this));
IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
uint256 actualReceived = IERC20(asset).balanceOf(address(this)) - balanceBefore;

uint256 rsethAmountToMint = getRsETHAmountToMint(asset, actualReceived);
require(rsethAmountToMint >= minRSETHAmountExpected, "MinimumAmountToReceiveNotMet");
_mintRsETH(rsethAmountToMint);
```

Alternatively, enforce in `addNewSupportedAsset` that only tokens with no transfer fee are whitelisted.

### Proof of Concept
1. Governance adds `FeeToken` (2% transfer fee) as a supported asset via `addNewSupportedAsset`.
2. Alice calls `depositAsset(FeeToken, 1000e18, 0, "")`.
3. `_beforeDeposit` computes `rsethAmountToMint` based on `1000e18`.
4. `safeTransferFrom` transfers `1000e18` from Alice; contract receives `980e18` (2% fee burned).
5. `_mintRsETH` mints rsETH equivalent to `1000e18` of `FeeToken`.
6. `getTotalAssetDeposits(FeeToken)` returns `980e18` (actual balance).
7. rsETH supply is inflated by the fee amount; the rsETH price is now lower than it should be, diluting all existing holders.
8. Repeated deposits compound the shortfall until the protocol cannot honor redemptions at the stated rsETH price.
