### Title
`setProtocolFeeBps` Should Accrue Rewards First Before Changing the Fee Rate - (File: contracts/LRTConfig.sol)

### Summary

`LRTConfig.setProtocolFeeBps` changes `protocolFeeInBPS` without first triggering `LRTOracle.updateRSETHPrice()`. Because `_updateRsETHPrice` applies the **current** `protocolFeeInBPS` to all rewards accumulated since the last price update, a fee-rate change is retroactively applied to the entire unaccrued window, causing rsETH holders to lose yield they were entitled to.

### Finding Description

`LRTOracle._updateRsETHPrice` computes the protocol fee on every call as:

```
rewardAmount      = totalETHInProtocol - previousTVL   // all rewards since last update
protocolFeeInETH  = rewardAmount * lrtConfig.protocolFeeInBPS() / 10_000
``` [1](#0-0) 

`previousTVL` is `rsethSupply * rsETHPrice`, where `rsETHPrice` is the value stored at the **last** `updateRSETHPrice` call. Every reward that has accrued since that last call is therefore subject to whatever `protocolFeeInBPS` is at the moment the **next** call executes.

`setProtocolFeeBps` in `LRTConfig.sol` simply overwrites `protocolFeeInBPS` with no prior settlement:

```solidity
function setProtocolFeeBps(uint256 _protocolFeeInBPS) external onlyRole(LRTConstants.MANAGER) {
    if (_protocolFeeInBPS > 1500) revert ProtocolFeeExceedsLimit();
    protocolFeeInBPS = _protocolFeeInBPS;   // ← no updateRSETHPrice() first
    emit UpdateFee(_protocolFeeInBPS);
}
``` [2](#0-1) 

`updateRSETHPrice` is a public, permissionless function callable by anyone: [3](#0-2) 

### Impact Explanation

When `protocolFeeInBPS` is raised, the new higher rate is applied retroactively to all rewards that accrued under the old lower rate. rsETH holders receive a lower rsETH price than they are entitled to — the excess fee is minted to the treasury as rsETH, diluting existing holders. This constitutes **theft of unclaimed yield** from rsETH holders.

Conversely, a fee decrease retroactively reduces the fee on the already-accrued window, causing the treasury to receive less than it should.

The maximum fee is capped at 1500 BPS (15%). A change from 0% to 15% applied retroactively over a multi-day unaccrued window on a large TVL can represent a material loss of yield for holders.

### Likelihood Explanation

Fee adjustments are a routine protocol management action. No malicious intent is required — any legitimate call to `setProtocolFeeBps` while rewards have been accumulating since the last price update triggers the retroactive mis-accounting. The longer the gap between price updates and the larger the TVL, the greater the impact.

### Recommendation

Call `updateRSETHPrice()` (or the internal `_updateRsETHPrice()` via a cross-contract call) inside `setProtocolFeeBps` before writing the new fee value, mirroring the fix applied in the JPEG'd reference:

```solidity
function setProtocolFeeBps(uint256 _protocolFeeInBPS) external onlyRole(LRTConstants.MANAGER) {
    if (_protocolFeeInBPS > 1500) revert ProtocolFeeExceedsLimit();
    // Settle pending rewards at the current rate before changing it
    ILRTOracle(getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice();
    protocolFeeInBPS = _protocolFeeInBPS;
    emit UpdateFee(_protocolFeeInBPS);
}
```

### Proof of Concept

1. Protocol TVL is 10,000 ETH. `protocolFeeInBPS = 500` (5%). `rsETHPrice` was last stored 3 days ago.
2. Over those 3 days, staking rewards cause `totalETHInProtocol` to grow to 10,100 ETH (`rewardAmount = 100 ETH`).
3. Manager calls `setProtocolFeeBps(1500)` — fee jumps to 15%. No price update occurs.
4. Anyone calls `updateRSETHPrice()`.
5. `protocolFeeInETH = 100 ETH * 1500 / 10_000 = 15 ETH` is taken as fee.
6. Correct fee at the old rate would have been `100 ETH * 500 / 10_000 = 5 ETH`.
7. rsETH holders collectively lose **10 ETH** of yield, minted instead to the treasury. [4](#0-3) [2](#0-1)
