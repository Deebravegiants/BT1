### Title
Hardcoded 18-Decimal Assumption in Token-to-rsETH Swap Calculation Causes Massive Underpayment for Non-18-Decimal Tokens - (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol)

### Summary
`RSETHPoolV3` and `RSETHPoolV3ExternalBridge` compute the rsETH amount to mint for ERC20 token deposits using a formula that implicitly assumes the deposited token has 18 decimals. If a token with fewer decimals (e.g., USDC with 6 decimals) is added as a supported token, depositors receive a factor of `10^(18 - tokenDecimals)` fewer rsETH than they are owed, while their full token amount is transferred into the pool.

### Finding Description
In `RSETHPoolV3.viewSwapRsETHAmountAndFee(uint256 amount, address token)`:

```solidity
// rate of token in ETH
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();

// Calculate the final rsETH amount
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

The formula treats `amountAfterFee` as if it is already in 1e18 units. For an 18-decimal token this is correct: `(amountAfterFee / 1e18) * (tokenToETHRate / 1e18) / (rsETHToETHrate / 1e18) * 1e18 = amountAfterFee * tokenToETHRate / rsETHToETHrate`. For a 6-decimal token the same arithmetic produces a result that is `1e12` times too small, because `amountAfterFee` is in 1e6 units while the formula still divides by `rsETHToETHrate` (1e18 scale) without compensating for the missing 12 decimal places.

The ETH-deposit path correctly uses `1e18` as the explicit scaling factor:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

but the token-deposit path omits the equivalent normalization step. Neither `addSupportedToken` nor any other function validates that the token has exactly 18 decimals before registering it.

The identical flaw is present in `RSETHPoolV3ExternalBridge.viewSwapRsETHAmountAndFee(uint256 amount, address token)`.

### Impact Explanation
A depositor who sends 1,000 USDC (6 decimals, `amount = 1e9`) to `deposit(usdcAddress, 1e9, referralId)`:

- `amountAfterFee â‰ˆ 1e9`
- `tokenToETHRate â‰ˆ 3.33e14` (1 USDC â‰ˆ 1/3000 ETH, in 1e18 precision)
- `rsETHToETHrate â‰ˆ 1.05e18`
- Actual minted: `1e9 * 3.33e14 / 1.05e18 â‰ˆ 317` wei rsETH
- Correct amount: `1e9 * 1e12 * 3.33e14 / 1.05e18 â‰ˆ 3.17e14` wei rsETH

The user's 1,000 USDC is transferred into the pool (`safeTransferFrom` executes before the mint), but they receive `~1e12` times fewer rsETH than owed. The tokens are effectively confiscated by the pool. This constitutes direct theft of user funds â€” **Critical**.

### Likelihood Explanation
The trigger requires a TIMELOCK_ROLE admin to call `addSupportedToken` with a non-18-decimal token. This is a legitimate protocol operation (not a key compromise or malicious act); the admin may simply not realize the contract lacks decimal normalization. The pools are deployed on multiple L2 chains where USDC (6 decimals) is a natural candidate for future support. No on-chain guard prevents the misconfiguration.

### Recommendation
Normalize the token amount to 18 decimals before applying the rate formula:

```solidity
uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 normalizedAmount = amountAfterFee * 10 ** (18 - tokenDecimals);
rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
```

Alternatively, enforce in `addSupportedToken` that only 18-decimal tokens may be registered:

```solidity
require(IERC20Metadata(token).decimals() == 18, "Only 18-decimal tokens supported");
```

### Proof of Concept

1. Admin calls `RSETHPoolV3.addSupportedToken(usdcAddress, usdcOracle)` â€” no decimal check prevents this.
2. User approves 1,000 USDC and calls `deposit(usdcAddress, 1_000e6, "ref")`.
3. `safeTransferFrom` moves 1,000 USDC into the pool.
4. `viewSwapRsETHAmountAndFee(1_000e6, usdcAddress)` computes:
   - `fee = 1_000e6 * feeBps / 10_000`
   - `amountAfterFee â‰ˆ 1_000e6`
   - `rsETHAmount = 1_000e6 * 3.33e14 / 1.05e18 â‰ˆ 317` wei
5. `wrsETH.mint(msg.sender, 317)` â€” user receives 317 wei rsETH instead of ~3.17e14 wei rsETH.
6. The 1,000 USDC remains in the pool with no mechanism for the user to recover it.
