### Title
Missing Chainlink Price Validation in `ChainlinkPriceOracle.getAssetPrice()` Allows Zero Price to Corrupt rsETH Minting — (`contracts/oracles/ChainlinkPriceOracle.sol`)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` without validating the returned price for zero, negative, or staleness conditions. If Chainlink returns a zero or invalid price, the value propagates unchecked into `LRTDepositPool.getRsETHAmountToMint()`, causing a depositor who passes `minRSETHAmountExpected = 0` to permanently lose their deposited LST while receiving zero rsETH in return. This is the direct analog of the reference report: oracle failure → zero value used → no slippage/output protection → value extraction.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` reads from Chainlink without any of the standard safety checks:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L49-55
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
``` [1](#0-0) 

Missing checks:
- No `price <= 0` guard (zero or negative price passes through)
- No `answeredInRound < roundId` staleness guard
- No `updatedAt == 0` incomplete-round guard

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` — used for L2 pool collateral in the same repository — performs all three checks:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L27-32
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

The unvalidated price flows into `LRTOracle.getAssetPrice()`: [3](#0-2) 

Which is consumed by `LRTDepositPool.getRsETHAmountToMint()`:

```solidity
// contracts/LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [4](#0-3) 

If `getAssetPrice(asset)` returns `0`, then `rsethAmountToMint = 0`. The slippage guard in `_beforeDeposit()` is:

```solidity
// contracts/LRTDepositPool.sol L667
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [5](#0-4) 

When `minRSETHAmountExpected = 0` (the default for many integrations and naive callers), `0 < 0` is false, so the check passes silently. The user's LST is transferred in, and `_mintRsETH(0)` mints nothing. [6](#0-5) 

A secondary impact path exists through `LRTOracle._getTotalEthInProtocol()`, which sums `assetER * totalAssetAmt` for all supported assets: [7](#0-6) 

A zero price for any asset silently zeroes that asset's contribution to TVL, causing `_updateRsETHPrice()` to compute a deflated `rsETHPrice`, which dilutes existing rsETH holders when subsequent deposits occur at the incorrect rate.

---

### Impact Explanation

**Primary path — direct fund loss for depositor:**
A user calls `depositAsset(asset, amount, 0, referralId)` while the Chainlink oracle for `asset` is returning `price = 0` (deprecated feed, oracle pause, or round failure). The user's LST is transferred to the protocol, `rsethAmountToMint` evaluates to `0`, and `_mintRsETH(0)` is called. The user permanently loses their deposited LST with no rsETH issued. This is a direct theft of user funds in motion.

**Secondary path — rsETH price corruption:**
`updateRSETHPrice()` is a public, permissionless function. Anyone can call it while the oracle is returning a zero or stale price. The resulting deflated `rsETHPrice` stored on-chain causes all subsequent depositors to receive excess rsETH, diluting existing holders. If `pricePercentageLimit == 0` (unset), the downside-protection pause is also bypassed. [8](#0-7) 

---

### Likelihood Explanation

Chainlink feeds can return zero or stale prices during: feed deprecation, sequencer downtime on L2 (though this is L1), or oracle contract upgrades. The missing staleness check (`answeredInRound < roundId`) is the more realistic trigger — Chainlink's own documentation recommends this check. The same repository already applies it in `ChainlinkOracleForRSETHPoolCollateral`, confirming developer awareness. Callers passing `minRSETHAmountExpected = 0` is common in contract integrations and naive front-end defaults. Both conditions occurring simultaneously is low-probability but not negligible, consistent with Medium severity.

---

### Recommendation

Apply the same validation pattern already used in `ChainlinkOracleForRSETHPoolCollateral.getRate()` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

if (answeredInRound < roundId) revert StalePrice();
if (updatedAt == 0) revert IncompleteRound();
if (price <= 0) revert InvalidPrice();
```

Additionally, document that callers of `depositAsset()` and `depositETH()` must supply a non-zero `minRSETHAmountExpected`, and consider reverting when `minRSETHAmountExpected == 0`.

---

### Proof of Concept

1. Chainlink feed for `stETH` enters a stale/zero-price state (feed deprecated or round not completed).
2. Attacker (or any user) calls `LRTOracle.updateRSETHPrice()` — public, no access control — which calls `_getTotalEthInProtocol()`. The stETH contribution is `0 * totalStETH = 0`, deflating `rsETHPrice`.
3. Victim calls `LRTDepositPool.depositAsset(stETH, 10 ether, 0, "")`.
4. `getRsETHAmountToMint()` computes `(10e18 * 0) / rsETHPrice = 0`.
5. `_beforeDeposit()` checks `0 < 0` → false → no revert.
6. `stETH.safeTransferFrom(victim, depositPool, 10 ether)` succeeds.
7. `_mintRsETH(0)` mints nothing.
8. Victim has lost 10 stETH with zero rsETH received. [9](#0-8) [1](#0-0)
