### Title
ETH Deposit Limit Bypass Due to Missing Amount in Cap Check - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric cap check: for LST assets it correctly tests `totalAssetDeposits + amount > depositLimit`, but for ETH it omits the incoming `amount` and only tests `totalAssetDeposits > depositLimit`. Any unprivileged depositor can therefore push total ETH holdings past the configured `depositLimitByAsset` cap.

### Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` (lines 676–682 of `contracts/LRTDepositPool.sol`):

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount omitted
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct for LSTs
}
```

The ETH branch returns `true` (i.e., "limit exceeded") only when the *pre-deposit* total already exceeds the cap. It never adds the incoming `msg.value` to the comparison. Consequently, when `totalAssetDeposits == depositLimit - 1 wei`, the check returns `false` and the deposit proceeds, minting rsETH for an arbitrary `msg.value` and pushing the real total to `depositLimit - 1 + msg.value`.

The LST path at line 681 correctly includes `amount` in the comparison, confirming the ETH branch is an oversight rather than intentional design. [1](#0-0) 

The function is called unconditionally from `_beforeDeposit`, which is the sole pre-flight check for `depositETH`: [2](#0-1) 

`depositETH` is a public, permissionless entry point: [3](#0-2) 

### Impact Explanation
`depositLimitByAsset` is the protocol's risk-management ceiling for each accepted asset. Exceeding it for ETH means the protocol mints rsETH beyond the intended cap, accepting ETH it may not be able to deploy to EigenLayer strategies. This causes the protocol to hold idle, un-yielding ETH while rsETH holders expect full yield, meaning the contract fails to deliver its promised returns. In the worst case, if the cap was sized to match available EigenLayer capacity, the surplus ETH cannot be staked, permanently diluting rsETH yield for all holders.

**Impact: Low — Contract fails to deliver promised returns.**

### Likelihood Explanation
The `depositETH` function is public and requires no special role. Any depositor who observes `totalAssetDeposits` approaching `depositLimitByAsset[ETH_TOKEN]` can call `depositETH` with an arbitrarily large `msg.value` in a single transaction. No front-running, governance access, or privileged key is needed. The condition is trivially reachable whenever the ETH deposit limit is close to being filled.

**Likelihood: High.**

### Recommendation
Add the incoming `amount` to the ETH branch of the cap check, mirroring the LST logic:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This aligns the ETH path with the LST path and with `getAssetCurrentLimit`, which already computes remaining capacity as `depositLimit - totalAssetDeposits` without any special ETH exception. [4](#0-3) 

### Proof of Concept

Assume `depositLimitByAsset[ETH_TOKEN] = 100 ether` and `getTotalAssetDeposits(ETH_TOKEN)` currently returns `99.9 ether`.

1. Attacker calls `depositETH{value: 50 ether}(0, "")`.
2. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 50 ether)`.
3. ETH branch evaluates `99.9 ether > 100 ether` → `false` → limit not exceeded.
4. `_mintRsETH` mints rsETH for 50 ETH; total ETH in protocol becomes `149.9 ether`, exceeding the 100 ETH cap by 49.9 ETH.
5. The same call with an LST would have evaluated `99.9 + 50 > 100` → `true` → reverted with `MaximumDepositLimitReached`. [1](#0-0)
