### Title
`getExpectedAssetAmount` Does Not Account for `instantWithdrawalFee`, Overstating Actual Asset Received - (File: contracts/LRTWithdrawalManager.sol)

---

### Summary

`LRTWithdrawalManager.getExpectedAssetAmount` is a public view function that returns the gross asset amount for a given `rsETHUnstaked` input. However, `instantWithdrawal` silently deducts `instantWithdrawalFee` from that gross amount before transferring to the user. Any upstream contract or user that calls `getExpectedAssetAmount` to determine how much asset they will receive from `instantWithdrawal` will receive a value that is too large, causing the actual transfer to fall short of the expected amount.

---

### Finding Description

`getExpectedAssetAmount` computes the asset return purely from oracle prices, with no awareness of `instantWithdrawalFee`:

```solidity
function getExpectedAssetAmount(address asset, uint256 amount)
    public view override returns (uint256 underlyingToReceive)
{
    ILRTOracle lrtOracle = ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE));
    underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
}
``` [1](#0-0) 

Inside `instantWithdrawal`, the same `getExpectedAssetAmount` result is used as the gross amount, and then the fee is deducted before the user receives anything:

```solidity
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
...
unstakingVault.redeem(asset, assetAmountUnlocked);

uint256 fee = (assetAmountUnlocked * instantWithdrawalFee) / 10_000;
uint256 userAmount = assetAmountUnlocked - fee;
...
_transferAsset(asset, msg.sender, userAmount);
``` [2](#0-1) 

The `instantWithdrawalFee` is a configurable basis-point fee (up to 10 %): [3](#0-2) [4](#0-3) 

The result is a structural mismatch: `getExpectedAssetAmount` is the natural view function for "how much will I get?", but it returns the pre-fee gross amount, not the post-fee net amount that `instantWithdrawal` actually delivers.

---

### Impact Explanation

Any on-chain integrator (e.g., a router, aggregator, or vault) that:
1. Calls `getExpectedAssetAmount(asset, rsETHAmount)` to determine the expected output, and
2. Passes that `rsETHAmount` to `instantWithdrawal` expecting to receive the quoted output,

will consistently receive `fee`-basis-points less than the quoted amount. If the integrator enforces a minimum-output check against the `getExpectedAssetAmount` return value, the call will revert every time a non-zero `instantWithdrawalFee` is set, permanently blocking that integration path.

**Impact: Low â€” Contract fails to deliver promised returns, but doesn't lose value.**

---

### Likelihood Explanation

`instantWithdrawalFee` is currently configurable up to 1000 bps (10 %). Whenever it is set to any non-zero value, every caller that relies on `getExpectedAssetAmount` as a quote for `instantWithdrawal` will receive less than quoted. The discrepancy is proportional to the fee and is always present when the fee is non-zero. Any integrator following the natural pattern of "query view â†’ execute" will be affected.

---

### Recommendation

Add a separate view function that returns the net (post-fee) asset amount for `instantWithdrawal`:

```solidity
function getExpectedAssetAmountAfterFee(address asset, uint256 rsETHAmount)
    public view returns (uint256 netAmount)
{
    uint256 gross = getExpectedAssetAmount(asset, rsETHAmount);
    uint256 fee = (gross * instantWithdrawalFee) / 10_000;
    netAmount = gross - fee;
}
```

Alternatively, document clearly in `getExpectedAssetAmount` that it returns the gross amount and does not reflect `instantWithdrawalFee`, and expose the fee-adjusted quote as the canonical function for `instantWithdrawal` callers.

---

### Proof of Concept

1. `instantWithdrawalFee` is set to 100 bps (1 %).
2. User holds 1 000 rsETH. They call `getExpectedAssetAmount(ETH, 1000e18)` â†’ returns `X` ETH (gross).
3. User (or upstream contract) calls `instantWithdrawal(ETH, 1000e18, "")` expecting to receive `X` ETH.
4. Actual transfer: `userAmount = X - X*100/10_000 = 0.99 * X`.
5. The user receives 1 % less than `getExpectedAssetAmount` promised. An upstream contract enforcing `received >= getExpectedAssetAmount(...)` will revert on every call while the fee is non-zero. [5](#0-4)
