### Title
Missing Deadline Check in Pool Deposit Functions Allows Stale-Rate Execution - (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPoolV3WithNativeChainBridge.sol, contracts/pools/RSETHPool.sol, contracts/pools/RSETHPoolNoWrapper.sol)

---

### Summary

All RSETH pool `deposit()` functions lack both a minimum output parameter and a deadline check. A pending deposit transaction can be executed at a later time when the rsETH/ETH oracle rate has increased, causing the user to receive fewer rsETH tokens than they expected at submission time.

---

### Finding Description

Every public-facing `deposit()` function across the pool family accepts only an ETH value (or token amount) and a `referralId`, then computes the rsETH output at execution time by calling `getRate()` on the oracle:

```solidity
// RSETHPoolV3.sol – ETH deposit
function deposit(string memory referralId)
    external payable nonReentrant whenNotPaused
    limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();

    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
``` [1](#0-0) 

The rsETH amount is derived entirely from the live oracle rate at the moment of execution:

```solidity
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
``` [2](#0-1) 

The same pattern is present in every pool variant:

- `RSETHPoolV3ExternalBridge.deposit()` [3](#0-2) 
- `RSETHPoolV3WithNativeChainBridge.deposit()` [4](#0-3) 
- `RSETHPool.deposit()` [5](#0-4) 
- `RSETHPoolNoWrapper.deposit()` [6](#0-5) 

None of these functions accept a `minRsETHAmountExpected` or a `deadline` parameter. The `LRTDepositPool` deposit functions do accept `minRSETHAmountExpected` but also lack a deadline: [7](#0-6) 

---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

When a user submits a deposit transaction, they observe the current rsETH/ETH rate and form an expectation of how many rsETH tokens they will receive. If the transaction is delayed in the mempool (e.g., during network congestion or low gas pricing), the oracle rate may have increased by the time the transaction is mined — because rsETH accrues yield over time, its ETH price monotonically increases. The user then receives fewer rsETH tokens than they expected at submission time, with no on-chain mechanism to revert the transaction. The deposited ETH is not lost (the rsETH received is still worth approximately the deposited ETH), but the user receives fewer rsETH shares than the rate they observed warranted.

Unlike the AfEth case, the oracle rate here is not directly manipulable by MEV bots in a single block (it is not an AMM spot price), so sandwich attacks are not the primary vector. The risk is purely from delayed execution against a monotonically increasing rate.

---

### Likelihood Explanation

**Low-to-Medium.** The rsETH/ETH rate increases continuously as yield accrues. On L2 chains where gas is cheap and transactions can sit in the mempool for extended periods, or during periods of network congestion, a deposit submitted at one rate can execute at a materially worse rate. This is a passive, always-present risk for every depositor across all deployed pool contracts.

---

### Recommendation

Add a `minRsETHAmountExpected` parameter to all pool `deposit()` functions and revert if the computed output falls below it. Additionally, add a `deadline` parameter and revert if `block.timestamp > deadline`:

```diff
- function deposit(string memory referralId) external payable nonReentrant whenNotPaused limitDailyMint(msg.value, ETH_IDENTIFIER) {
+ function deposit(string memory referralId, uint256 minRsETHAmountExpected, uint256 deadline) external payable nonReentrant whenNotPaused limitDailyMint(msg.value, ETH_IDENTIFIER) {
+     if (block.timestamp > deadline) revert StaleDeposit();
      uint256 amount = msg.value;
      if (amount == 0) revert InvalidAmount();
      (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
+     if (rsETHAmount < minRsETHAmountExpected) revert InsufficientOutput();
      feeEarnedInETH += fee;
      wrsETH.mint(msg.sender, rsETHAmount);
      emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
  }
```

Apply the same pattern to the token `deposit(address token, uint256 amount, string memory referralId)` overloads across all pool contracts, and add a `deadline` check to `LRTDepositPool.depositETH()` and `depositAsset()`.

---

### Proof of Concept

1. User observes rsETH/ETH rate = 1.05 ETH per rsETH and submits a `deposit{value: 1 ether}("ref")` transaction expecting ≈ 0.952 rsETH (minus fee).
2. The transaction sits in the mempool for several hours due to low gas price.
3. The rsETH/ETH oracle rate increases to 1.06 ETH per rsETH as yield accrues.
4. The transaction is mined; `viewSwapRsETHAmountAndFee` computes rsETHAmount using the new rate of 1.06, yielding ≈ 0.943 rsETH — approximately 0.009 rsETH less than expected.
5. The user has no recourse; there is no minimum output check and no deadline to trigger a revert.
