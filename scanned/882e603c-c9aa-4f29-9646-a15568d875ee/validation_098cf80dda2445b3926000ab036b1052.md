### Title
Missing Zero-Rate Validation in `viewSwapRsETHAmountAndFee` Silently Returns 0 rsETH to Token Depositors - (File: contracts/pools/RSETHPoolV3.sol)

### Summary
`RSETHPoolV3.viewSwapRsETHAmountAndFee(uint256 amount, address token)` does not validate that the token oracle's rate is non-zero before computing the rsETH output. If the oracle returns 0, the function silently returns `rsETHAmount = 0`. The `deposit` function then transfers the user's tokens into the pool and mints 0 rsETH — the user loses their deposited assets with no on-chain recovery path.

### Finding Description
In `viewSwapRsETHAmountAndFee(uint256 amount, address token)`, the token-to-ETH rate is fetched and used directly in the division without any zero-check:

```solidity
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [1](#0-0) 

If `tokenToETHRate == 0`, the multiplication yields 0 and `rsETHAmount` silently returns 0 with no revert.

The sibling function `viewSwapAssetToPremintedRsETH` in the same contract **does** perform this validation:

```solidity
uint256 tokenToETHRate = token == ETH_IDENTIFIER ? 1e18 : IOracle(supportedTokenOracle[token]).getRate();
if (tokenToETHRate == 0) revert UnsupportedOracle();
``` [2](#0-1) 

This is the direct structural analog to the reported bug: one code path validates and reverts; the other silently returns a zero/default value.

The `deposit(address token, uint256 amount, string memory referralId)` function has no guard on the computed `rsETHAmount` before executing the token transfer and mint:

```solidity
IERC20(token).safeTransferFrom(msg.sender, address(this), amount);
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
feeEarnedInToken[token] += fee;
wrsETH.mint(msg.sender, rsETHAmount);
``` [3](#0-2) 

The same pattern exists in `RSETHPool.viewSwapRsETHAmountAndFee(amount, token)` and `RSETHPoolNoWrapper.viewSwapRsETHAmountAndFee(amount, token)`: [4](#0-3) [5](#0-4) 

The `addSupportedToken` function validates the oracle rate at registration time:

```solidity
if (IOracle(oracle).getRate() == 0) {
    revert UnsupportedOracle();
}
``` [6](#0-5) 

However, this one-time check at registration does not protect against the oracle returning 0 at a later point during a live deposit.

### Impact Explanation
When `tokenToETHRate` is 0 at deposit time:
1. The user's ERC-20 tokens are transferred into the pool (`safeTransferFrom` succeeds).
2. `viewSwapRsETHAmountAndFee` returns `rsETHAmount = 0`.
3. `wrsETH.mint(msg.sender, 0)` succeeds (ERC-20 allows minting 0).
4. The user receives 0 rsETH while their tokens remain locked in the pool.

There is no on-chain user-facing recovery function in any of the pool contracts. The deposited tokens are permanently inaccessible to the user without out-of-band admin intervention. This constitutes **temporary (potentially permanent) freezing of user funds**.

### Likelihood Explanation
The token oracle returning 0 is a realistic failure mode: price feed contracts can return 0 on staleness, circuit-breaker activation, or an upgrade bug. The one-time registration check (`addSupportedToken`) does not prevent this. Any unprivileged depositor who calls `deposit(token, amount, referralId)` during such a window is affected. The same vulnerability is present across `RSETHPool`, `RSETHPoolV3`, and `RSETHPoolNoWrapper`, widening the attack surface across all supported L2 chains.

### Recommendation
- **Short term:** Add a zero-rate guard in `viewSwapRsETHAmountAndFee(uint256 amount, address token)` consistent with `viewSwapAssetToPremintedRsETH`:
  ```solidity
  if (tokenToETHRate == 0) revert UnsupportedOracle();
  ```
  Additionally, add a guard in `deposit` before the token transfer:
  ```solidity
  (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
  if (rsETHAmount == 0) revert InvalidAmount();
  IERC20(token).safeTransferFrom(msg.sender, address(this), amount);
  ```
- **Long term:** Apply the same fix to `RSETHPool` and `RSETHPoolNoWrapper`. Implement property-based tests asserting that `deposit` always reverts when the oracle returns 0.

### Proof of Concept
1. Token `T` is added via `addSupportedToken(T, oracle, bridge)` — oracle returns a valid rate at this point.
2. The oracle for `T` later returns 0 (staleness, circuit-breaker, or upgrade bug).
3. User calls `RSETHPoolV3.deposit(T, 1e18, "ref")`.
4. `limitDailyMint` modifier calls `viewSwapRsETHAmountAndFee(1e18, T)` → `rsETHAmount = 0`; daily limit check passes (0 ≤ limit).
5. `deposit` executes: `safeTransferFrom` moves 1e18 T tokens from user to pool.
6. `viewSwapRsETHAmountAndFee` returns `rsETHAmount = 0`.
7. `wrsETH.mint(user, 0)` succeeds silently.
8. User holds 0 rsETH; 1e18 T tokens are locked in the pool with no user-accessible withdrawal path.
