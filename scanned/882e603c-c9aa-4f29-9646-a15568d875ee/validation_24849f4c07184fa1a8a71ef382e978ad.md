### Title
Token Decimal Mismatch in `viewSwapRsETHAmountAndFee` Causes Massive rsETH Under-Minting for Non-18-Decimal Tokens - (File: contracts/pools/RSETHPoolV3ExternalBridge.sol)

### Summary

The `viewSwapRsETHAmountAndFee(amount, token)` function across all RSETHPool variants computes the rsETH output as `amountAfterFee * tokenToETHRate / rsETHToETHrate` without normalizing `amountAfterFee` to 18 decimals. The oracle always returns a price scaled to 1e18, but `amountAfterFee` is in the token's native decimals. For 18-decimal tokens (wstETH, WETH) the formula is accidentally correct; for any token with fewer decimals (e.g., USDC at 6 decimals) the result is `1e(18-tokenDecimals)` times too small, causing depositors to receive a negligible amount of rsETH relative to the value they deposited.

### Finding Description

`ChainlinkOracleForRSETHPoolCollateral.getRate()` normalizes the Chainlink answer to 18 decimals regardless of the token's own decimal count:

```solidity
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
``` [1](#0-0) 

This returns the ETH value of **one whole token** (e.g., 1 wstETH = 1.2e18). The pool then computes:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [2](#0-1) 

The same pattern is present in every pool variant: [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

For an 18-decimal token the units work out:

```
(N × 1e18) × (P × 1e18) / (R × 1e18) = N×P/R × 1e18  ✓
```

For a 6-decimal token (e.g., USDC):

```
(N × 1e6) × (P × 1e18) / (R × 1e18) = N×P/R × 1e6   ✗  (should be × 1e18)
```

The result is `1e12` times smaller than the correct rsETH amount. A user depositing 1 USDC (~$1) would receive ~317 wei of rsETH instead of ~317e12 wei.

There is no decimal enforcement in `_addSupportedToken`: [7](#0-6) 

The only guard is that the oracle returns a non-zero rate, so any ERC-20 token can be added regardless of its decimal count.

The reverse path `viewSwapAssetToPremintedRsETH` has the symmetric error — it would return `1e12` times **more** tokens than correct, draining the pool:

```solidity
tokenAmount = rsETHAmount * rsETHToETHrate / tokenToETHRate;
``` [8](#0-7) 

### Impact Explanation

Any user calling `deposit(token, amount, referralId)` with a supported non-18-decimal token receives `1e(18-tokenDecimals)` times less rsETH than the fair value of their deposit. For USDC (6 decimals) this is a factor of `1e12`, meaning the user loses essentially all deposited value. This is a direct, permanent loss of user funds with no recovery path once the transaction is confirmed.

**Impact: Critical — direct theft of user funds at-rest.**

### Likelihood Explanation

Currently deployed pools support only 18-decimal LSTs (wstETH, WETH). However:
- The `addSupportedToken` / `_addSupportedToken` functions impose no decimal restriction.
- The protocol's design explicitly supports multi-token deposits across chains.
- Stablecoins (USDC, USDT — 6 decimals) are natural candidates for future collateral.
- Any admin who adds such a token (even with good intent) silently activates the bug for all depositors.

**Likelihood: Medium** — requires a non-18-decimal token to be added, which is a plausible future governance action.

### Recommendation

Normalize `amountAfterFee` to 18 decimals before applying the rate, or equivalently scale the result:

```solidity
import { IERC20Metadata } from "@openzeppelin/contracts/token/ERC20/extensions/IERC20Metadata.sol";

function viewSwapRsETHAmountAndFee(uint256 amount, address token)
    public view returns (uint256 rsETHAmount, uint256 fee)
{
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;

    uint256 rsETHToETHrate = getRate();
    uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();

    uint8 tokenDecimals = IERC20Metadata(token).decimals();
    // Normalize amountAfterFee to 18 decimals
    uint256 normalizedAmount = amountAfterFee * 10 ** (18 - tokenDecimals);

    rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
}
```

Apply the symmetric inverse normalization in `viewSwapAssetToPremintedRsETH` (divide `tokenAmount` by `10 ** (18 - tokenDecimals)`). Also enforce `tokenDecimals <= 18` in `_addSupportedToken`.

### Proof of Concept

1. Admin adds USDC (6 decimals) as a supported token with a valid Chainlink USDC/ETH oracle.
2. User calls `deposit(USDC, 1_000_000, "ref")` (1 USDC).
3. Oracle returns `tokenToETHRate ≈ 3.33e14` (0.000333 ETH per USDC at $3000/ETH).
4. `rsETHToETHrate ≈ 1.05e18`.
5. Actual computation: `1_000_000 * 3.33e14 / 1.05e18 ≈ 317` (317 wei of rsETH).
6. Correct computation: `1_000_000 * 1e12 * 3.33e14 / 1.05e18 ≈ 317e12` (317 trillion wei = 0.000317 rsETH).
7. User receives 317 wei instead of 317e12 wei — a loss of 99.9999999999% of deposited value. The difference (≈ $1 worth of rsETH) remains permanently locked in the pool as excess minted rsETH supply imbalance.
