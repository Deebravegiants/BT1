### Title
ETH Deposit Limit Bypass Due to Missing `amount` in Limit Check - (`contracts/LRTDepositPool.sol`)

### Summary

The `_checkIfDepositAmountExceedesCurrentLimit` function in `LRTDepositPool.sol` applies an inconsistent limit check between ETH and ERC20 assets. For ERC20 tokens, the incoming `amount` is included in the comparison against the deposit cap. For ETH, it is not. This allows any depositor to push ETH deposits arbitrarily beyond the configured `depositLimitByAsset` cap once total deposits are just below the limit.

### Finding Description

`_checkIfDepositAmountExceedesCurrentLimit` contains two branches:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount omitted
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← amount included
}
``` [1](#0-0) 

For ERC20 assets the check is `totalAssetDeposits + amount > limit`, which correctly rejects a deposit that would push the total over the cap. For ETH the check is `totalAssetDeposits > limit`, which only rejects a deposit when the total is **already** over the cap — it does not account for the incoming `amount` at all.

The ETH deposit entry point is `depositETH`, which calls `_beforeDeposit`, which calls this function: [2](#0-1) [3](#0-2) 

The deposit limit is read from `lrtConfig.depositLimitByAsset(asset)` and is the protocol's primary mechanism for capping ETH TVL growth. [4](#0-3) 

### Impact Explanation

Once `totalAssetDeposits` for ETH is anywhere below `depositLimitByAsset`, a depositor can send an arbitrarily large ETH amount in a single call. The check returns `false` (not exceeded) because it only compares the pre-deposit total against the limit, not `total + amount`. The deposit succeeds, rsETH is minted for the full amount, and the total ETH in the protocol can be pushed far beyond the configured cap in a single transaction.

This bypasses the protocol's TVL cap, a risk-management parameter. The impact is that the protocol accepts ETH deposits beyond its intended limit, minting rsETH for amounts the protocol was not designed to handle at that stage. This maps to **Low: Contract fails to deliver promised returns** (the deposit cap is not enforced) with potential escalation to **Medium** if the cap is intended to protect against protocol insolvency or EigenLayer strategy limits.

### Likelihood Explanation

The entry path is fully permissionless: any address can call `depositETH` with any `msg.value`. No special role, timing, or precondition is required beyond the total ETH deposits being below (not already above) the limit. The condition is trivially reachable in normal protocol operation.

### Recommendation

Include `amount` in the ETH branch, matching the ERC20 branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // fixed
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept

1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. Through normal deposits, `getTotalAssetDeposits(ETH_TOKEN)` reaches `999.9 ether`.
3. `getAssetCurrentLimit(ETH_TOKEN)` returns `0.1 ether`, signalling the cap is nearly full.
4. Attacker calls `depositETH{value: 500 ether}(0, "")`.
5. Inside `_checkIfDepositAmountExceedesCurrentLimit`: `totalAssetDeposits (999.9 ether) > limit (1000 ether)` → `false` → deposit is not blocked.
6. `_mintRsETH` mints rsETH for 500 ETH.
7. `getTotalAssetDeposits(ETH_TOKEN)` is now `1499.9 ether`, 50% above the configured cap.

The same call with an ERC20 token would have been blocked: `999.9 + 500 > 1000` → `true` → `MaximumDepositLimitReached`.
