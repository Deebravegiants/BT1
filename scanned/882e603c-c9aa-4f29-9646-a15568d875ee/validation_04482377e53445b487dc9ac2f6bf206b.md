### Title
Unbounded Withdrawal Queue Enables DoS of `unlockQueue` via Dust Withdrawal Requests — (`contracts/LRTWithdrawalManager.sol`)

---

### Summary

When `minRsEthAmountToWithdraw[asset]` is zero (its default value), any user can call `initiateWithdrawal` with a dust amount (e.g., 1 wei of rsETH) an arbitrary number of times, flooding the global withdrawal queue. The operator-facing `unlockQueue` function iterates over this queue in a `while` loop with no per-call cap, so a sufficiently large queue causes the call to exceed the block gas limit, temporarily freezing all pending user withdrawals for that asset.

---

### Finding Description

`LRTWithdrawalManager.sol` manages a global FIFO withdrawal queue per asset. Users call `initiateWithdrawal` to enqueue a request; operators later call `unlockQueue` to mark requests as claimable; users then call `completeWithdrawal` to receive funds.

**Entry point — `initiateWithdrawal`:** [1](#0-0) 

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```

`minRsEthAmountToWithdraw[asset]` is a storage mapping that is **never initialised** in `initialize()`. [2](#0-1) 

Its default value is therefore `0`. When it is `0`, the guard collapses to `rsETHUnstaked == 0`, meaning any positive amount — including 1 wei — passes the check. An attacker can call `initiateWithdrawal` thousands of times with 1 wei rsETH each, paying only gas, and recover the rsETH once each dust request is eventually processed.

**Unbounded iteration — `_unlockWithdrawalRequests`:** [3](#0-2) 

```solidity
while (nextLockedNonce_ < firstExcludedIndex) {
    bytes32 requestId = getRequestId(asset, nextLockedNonce_);
    WithdrawalRequest storage request = withdrawalRequests[requestId];
    if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) break;
    uint256 payoutAmount = _calculatePayoutAmount(request, rsETHPrice, assetPrice);
    if (availableAssetAmount < payoutAmount) break;
    ...
    unchecked { nextLockedNonce_++; }
}
```

The loop iterates from the current `nextLockedNonce` up to `firstExcludedIndex` (caller-supplied). Each iteration performs a `keccak256`, two storage reads, arithmetic, and multiple storage writes — roughly 5 000–10 000 gas per entry. With a 30 M gas block limit, ~3 000–6 000 entries per call is the practical ceiling. An attacker who enqueues tens of thousands of dust requests forces the operator to either:

- Pass a large `firstExcludedIndex` and run out of gas, or
- Pass a small `firstExcludedIndex` and make many calls, each of which only clears a handful of dust entries before reaching legitimate requests.

Because the queue is strictly FIFO (indexed by `nextLockedNonce`), legitimate requests queued after the dust entries cannot be unlocked until all preceding dust entries are processed. [4](#0-3) 

---

### Impact Explanation

Legitimate users' withdrawal requests are blocked behind an unbounded number of dust entries. Operators cannot advance `nextLockedNonce` past the dust entries without exhausting block gas, so `completeWithdrawal` reverts for all affected users (the request is still `>= nextLockedNonce`, i.e., locked). [5](#0-4) 

This constitutes **temporary freezing of funds** (rsETH already transferred to the contract at `initiateWithdrawal` time, line 166, cannot be recovered until the queue is drained).

**Impact: Medium — Temporary freezing of funds.**

---

### Likelihood Explanation

- No special role is required; any rsETH holder can call `initiateWithdrawal`.
- `minRsEthAmountToWithdraw[asset]` is `0` by default and is only set by admin action; many assets may never have it configured.
- The attacker's rsETH is not permanently lost — it is returned when each dust request is eventually processed — so the net cost is only gas, making the attack economically viable on low-fee networks or during periods of low gas prices.
- The attack is straightforward and requires no flash loans, oracle manipulation, or privileged access.

**Likelihood: Medium.**

---

### Recommendation

1. **Enforce a non-zero minimum in `initiateWithdrawal`:** Require `minRsEthAmountToWithdraw[asset] > 0` at initialisation time, or add a protocol-wide floor (e.g., `1e15` rsETH) that cannot be set to zero.

```solidity
// In initialize():
minRsEthAmountToWithdraw[ETH_TOKEN] = 1e15; // example floor
```

2. **Cap the per-call iteration count in `_unlockWithdrawalRequests`:** Add a `maxIterations` parameter or a hard-coded cap so a single `unlockQueue` call processes at most N entries, preventing gas exhaustion regardless of queue depth.

3. **Validate `setMinRsEthAmountToWithdraw` to reject zero:**

```solidity
function setMinRsEthAmountToWithdraw(address asset, uint256 minRsEthAmountToWithdraw_) external onlyLRTAdmin {
    if (minRsEthAmountToWithdraw_ == 0) revert InvalidMinAmount();
    ...
}
```

---

### Proof of Concept

1. Admin deploys `LRTWithdrawalManager`; `minRsEthAmountToWithdraw[ETH_TOKEN]` is `0` (never set).
2. Attacker acquires rsETH (e.g., by depositing ETH into `LRTDepositPool`).
3. Attacker calls `initiateWithdrawal(ETH_TOKEN, 1, "")` in a loop N times (e.g., N = 50 000), each time transferring 1 wei rsETH to the contract. [6](#0-5) 

4. Legitimate users also call `initiateWithdrawal` with real amounts; their requests are queued after the attacker's dust entries.
5. Operator calls `unlockQueue(ETH_TOKEN, nextUnusedNonce, ...)`. The `while` loop in `_unlockWithdrawalRequests` iterates through all 50 000 dust entries before reaching legitimate requests, consuming far more than 30 M gas and reverting. [7](#0-6) 

6. Legitimate users' rsETH (held by the contract since step 3 of their own flow) cannot be unlocked or reclaimed; their funds are temporarily frozen.

```text
    function _unlockWithdrawalRequests(
        address asset,
        uint256 availableAssetAmount,
        uint256 rsETHPrice,
        uint256 assetPrice,
        uint256 firstExcludedIndex
    )
        internal
        returns (uint256 rsETHAmountToBurn, uint256 assetAmountToUnlock)
    {
        // Check that upper limit is in the range of existing withdrawal requests. If it is greater set it to the first
        // nonce with no withdrawal request.
        if (firstExcludedIndex > nextUnusedNonce[asset]) {
            firstExcludedIndex = nextUnusedNonce[asset];
        }

        uint256 nextLockedNonce_ = nextLockedNonce[asset];
        // Revert when trying to unlock a request that has already been unlocked
        if (nextLockedNonce_ >= firstExcludedIndex) revert NoPendingWithdrawals();

        while (nextLockedNonce_ < firstExcludedIndex) {
            bytes32 requestId = getRequestId(asset, nextLockedNonce_);
            WithdrawalRequest storage request = withdrawalRequests[requestId];

            // Check that the withdrawal delay has passed since the request's initiation.
            if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) break;

            // Calculate the amount user will receive
            uint256 payoutAmount = _calculatePayoutAmount(request, rsETHPrice, assetPrice);

            if (availableAssetAmount < payoutAmount) break; // Exit if not enough assets to cover this request

            assetsCommitted[asset] -= request.expectedAssetAmount;
            // Set the amount the user will receive
            request.expectedAssetAmount = payoutAmount;
            rsETHAmountToBurn += request.rsETHUnstaked;
            availableAssetAmount -= payoutAmount;
            assetAmountToUnlock += payoutAmount;

            unlockedWithdrawalsCount[asset]++;

            unchecked {
                nextLockedNonce_++;
            }
        }
        nextLockedNonce[asset] = nextLockedNonce_;
    }
```
