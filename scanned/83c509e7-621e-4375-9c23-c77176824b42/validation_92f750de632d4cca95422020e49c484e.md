### Title
`emergencyWithdrawFromAave` Blocked by Treasury ETH Rejection Due to Unconditional `_collectInterestToTreasury` Call — (`contracts/LRTWithdrawalManager.sol`)

### Summary
`emergencyWithdrawFromAave` unconditionally calls `_collectInterestToTreasury()` before withdrawing principal. If `PROTOCOL_TREASURY` is a contract that reverts on ETH receive, the interest push fails with `TreasuryTransferFailed`, reverting the entire transaction and preventing any principal from being recovered from Aave.

### Finding Description
In `LRTWithdrawalManager.emergencyWithdrawFromAave`:

```solidity
// First collect any accrued interest to treasury
_collectInterestToTreasury();          // line 558 — unconditional

uint256 withdrawnAmount = _withdrawFromAave(amount);  // line 560 — never reached
```

Inside `_collectInterestToTreasury`, once interest has accrued (`aaveBalance > totalETHDepositedToAave`), the function withdraws the interest from Aave and then pushes it to the treasury:

```solidity
(bool sent,) = payable(treasury).call{ value: interestAmount }("");
if (!sent) revert TreasuryTransferFailed();   // line 957–958
```

If the treasury contract reverts on ETH receive, this line reverts the entire call stack. The `_withdrawFromAave(amount)` on line 560 is never executed. All ETH deposited to Aave remains locked there, and the emergency recovery path is completely blocked.

### Impact Explanation
All ETH deposited to Aave via the withdrawal manager is temporarily frozen. The emergency withdrawal function — the only privileged path for a `PAUSER_ROLE` caller to recover principal from Aave — is rendered inoperable. Users with pending ETH withdrawal requests cannot be served from Aave liquidity. The freeze persists until an admin corrects the treasury address, which may not be possible during the emergency window (e.g., if Aave itself is under attack and time is critical).

### Likelihood Explanation
The `PROTOCOL_TREASURY` is a governance-configured address. Multisig wallets (Gnosis Safe) are the standard treasury type and can fail to receive ETH under several realistic conditions: misconfigured fallback, Safe version incompatibility, or a treasury migration where the new address is a contract not yet configured to accept ETH. No attacker action is required; the condition arises from normal operational changes.

### Recommendation
Decouple interest collection from principal recovery in `emergencyWithdrawFromAave`. Use a try/catch or a boolean `skipInterestCollection` parameter:

```solidity
function emergencyWithdrawFromAave(uint256 amount, bool collectInterest) external nonReentrant onlyRole(LRTConstants.PAUSER_ROLE) {
    if (!isAaveIntegrationEnabled) revert AaveIntegrationNotEnabled();
    uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
    if (aaveBalance == 0) revert InsufficientAaveBalance();

    if (collectInterest) {
        _collectInterestToTreasury(); // optional; can be skipped in emergencies
    }

    uint256 withdrawnAmount = _withdrawFromAave(amount);
    emit EmergencyWithdrawFromAave(withdrawnAmount, address(this));
}
```

Alternatively, wrap `_collectInterestToTreasury()` in a try/catch (via an external self-call) so that a treasury failure is logged but does not block principal recovery.

    function _collectInterestToTreasury() internal returns (uint256 interestAmount) {
        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        uint256 principal = totalETHDepositedToAave;

        // Return 0 if no interest or balance is less than principal (accounting for rounding)
        if (aaveBalance <= principal) return 0;

        interestAmount = aaveBalance - principal;

        aaveWETHGateway.withdrawETH(aavePool, interestAmount, address(this));

        address treasury = lrtConfig.getContract(LRTConstants.PROTOCOL_TREASURY);
        (bool sent,) = payable(treasury).call{ value: interestAmount }("");

        emit InterestCollectedToTreasury(interestAmount, treasury);
    }
