### Title
Stale Chainlink Price Accepted in `ChainlinkPriceOracle.getAssetPrice()` Enables Over-Minting of rsETH - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary
`ChainlinkPriceOracle.getAssetPrice()` calls Chainlink's `latestRoundData()` but discards all validity-related return values (`updatedAt`, `answeredInRound`, `roundId`). A stale price for any supported LST asset is silently accepted and propagated into rsETH minting, rsETH price updates, and withdrawal calculations. The same codebase already applies staleness checks in `ChainlinkOracleForRSETHPoolCollateral`, confirming developer awareness of the pattern — but the core oracle used for all LST deposits is unprotected.

### Finding Description
`ChainlinkPriceOracle.getAssetPrice()` fetches the Chainlink price as:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

No check is made on `updatedAt` (staleness) or `answeredInRound < roundId` (incomplete round). By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` in the same repository explicitly guards against both:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
``` [2](#0-1) 

`LRTOracle.getAssetPrice()` delegates directly to `ChainlinkPriceOracle.getAssetPrice()` with no additional validation: [3](#0-2) 

This stale price then flows into three critical paths:

1. **Deposit / rsETH minting** — `LRTDepositPool.getRsETHAmountToMint()` uses `lrtOracle.getAssetPrice(asset)` to compute how many rsETH tokens to mint per unit of deposited LST: [4](#0-3) 

2. **rsETH price update** — `LRTOracle._getTotalEthInProtocol()` sums `getAssetPrice(asset) * totalAssetDeposits` for every supported asset to derive the global rsETH price: [5](#0-4) 

3. **Withdrawal unlock** — `LRTWithdrawalManager._createUnlockParams()` reads `lrtOracle.getAssetPrice(asset)` to determine payout amounts: [6](#0-5) 

### Impact Explanation
**Critical — Direct theft of user funds / protocol insolvency.**

If a Chainlink feed for a supported LST (e.g., stETH/ETH, ETHx/ETH) stops updating and the last reported price is above the true market price, any depositor can call `depositAsset()` and receive rsETH minted at the inflated rate. The excess rsETH represents a claim on ETH that was never deposited, diluting every existing rsETH holder and leaving the protocol under-collateralised. The attacker can immediately redeem the over-minted rsETH for other assets via the withdrawal path, extracting real value.

Conversely, a stale low price causes `_updateRsETHPrice()` to compute an artificially low rsETH price, triggering the downside-protection pause and freezing all user funds until an admin intervenes.

### Likelihood Explanation
Chainlink feeds can stop updating due to network congestion, sequencer downtime, or feed deprecation — all documented failure modes. The affected assets (stETH, ETHx, rETH, swETH) each have their own Chainlink feed; any single feed going stale is sufficient to trigger the vulnerability. No special permissions are required: `depositAsset()` is callable by any address. [7](#0-6) 

### Recommendation
Add staleness and round-completeness checks inside `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```diff
 function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
     AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

-    (, int256 price,,,) = priceFeed.latestRoundData();
+    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();
+    require(answeredInRound >= roundId, "Stale price");
+    require(updatedAt != 0, "Incomplete round");
+    require(block.timestamp - updatedAt <= MAX_PRICE_AGE, "Price too old");
+    require(price > 0, "Invalid price");

     return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
 }
```

`MAX_PRICE_AGE` should be set per-feed based on the Chainlink heartbeat (e.g., 3600 seconds for 1-hour heartbeat feeds, 86 400 seconds for 24-hour feeds).

### Proof of Concept

1. Assume the stETH/ETH Chainlink feed stops updating. Its last reported price is 1.05 ETH per stETH (above the true 1.00 ETH).
2. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
3. `getRsETHAmountToMint` calls `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns 1.05e18 (stale).
4. rsETH minted = `1000e18 * 1.05e18 / rsETHPrice`. Attacker receives ~5% more rsETH than the deposited stETH is worth.
5. Attacker initiates withdrawal of the excess rsETH for a different asset (e.g., ETHx) at the correct price, extracting real ETH value from the pool.
6. Existing rsETH holders are left with a proportionally smaller claim on protocol assets.

The root cause — `(, int256 price,,,) = priceFeed.latestRoundData()` with no validity checks — is entirely within the protocol's own code and requires no external compromise beyond a Chainlink feed ceasing to update. [8](#0-7)
