### Title
`KernelTop100MerkleDistributor` Does Not Enforce Token Funding at Initialization — Claims Fail if Contract Is Underfunded - (File: contracts/KERNEL/KernelTop100MerkleDistributor.sol)

### Summary
`KernelTop100MerkleDistributor.initialize()` sets the merkle root (which encodes all user token allocations) and the vesting schedule, but performs no `transferFrom` and no balance check to ensure the contract actually holds the KERNEL tokens it promises to distribute. The contract relies entirely on the owner manually sending tokens to the contract address after deployment. If the contract is underfunded — even partially — any `claim()` or `claimAndStake()` call for an amount exceeding the available balance will revert, permanently denying users their vested allocations.

### Finding Description
`initialize()` accepts a `_merkleRoot` that cryptographically commits to the total allocation for every eligible user, and a `_vestingStartTimestamp` after which users may begin claiming. However, it never pulls any KERNEL tokens into the contract:

```solidity
// contracts/KERNEL/KernelTop100MerkleDistributor.sol
function initialize(
    address _kernel,
    address _kernelDepositPool,
    address _protocolTreasury,
    uint256 _feeInBPS,
    uint256 _vestingStartTimestamp,
    bytes32 _merkleRoot
) external initializer {
    // ... validation only ...
    kernel = IERC20(_kernel);
    merkleRoot = _merkleRoot;
    vestingStartTimestamp = _vestingStartTimestamp;
    kernel.forceApprove(_kernelDepositPool, type(uint256).max);
    // ← no transferFrom, no balance check, no totalAllocation parameter
}
``` [1](#0-0) 

When a user calls `claim()`, the contract attempts `kernel.safeTransfer(user, amountToSend)`:

```solidity
kernel.safeTransfer(protocolTreasury, fee);
kernel.safeTransfer(user, amountToSend);
``` [2](#0-1) 

If `kernel.balanceOf(address(this))` is less than `amountToSend + fee`, the `safeTransfer` reverts. There is no on-chain invariant tying the merkle root's total committed allocation to the contract's actual token balance. The contract has no `totalAllocation` state variable and no funding check at any point.

The same issue applies to `claimAndStake()`, which also relies on the contract's KERNEL balance: [3](#0-2) 

### Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

Users who hold valid merkle proofs and have vested allocations are unable to receive their KERNEL tokens if the contract is underfunded. Their `amountClaimed` state is updated *before* the transfer, meaning a partial-balance scenario could cause the transfer to revert after state is written — though in practice `safeTransfer` reverts atomically, so state is rolled back. The net effect is that legitimate claimants are denied their promised yield/reward tokens for as long as the contract remains underfunded.

### Likelihood Explanation
The owner must perform a separate, manual `transfer` of KERNEL tokens to the contract address after deployment. There is no on-chain prompt, no required parameter for total allocation, and no guard preventing `initialize()` from completing with a zero balance. A deployment error, miscalculation of total allocation, or simple omission results in a live contract with a valid merkle root but no tokens to distribute.

### Recommendation
Add a `_totalAllocation` parameter to `initialize()` and pull the tokens atomically:

```solidity
function initialize(
    address _kernel,
    ...
    bytes32 _merkleRoot,
    uint256 _totalAllocation   // sum of all leaf amounts in the merkle tree
) external initializer {
    ...
    kernel = IERC20(_kernel);
    ...
    // Pull tokens from deployer atomically
    kernel.safeTransferFrom(msg.sender, address(this), _totalAllocation);
}
```

This mirrors the fix recommended in the reference report: invoke `transferFrom` at allocation time so the contract's balance is guaranteed to match its commitments.

### Proof of Concept
1. Owner deploys `KernelTop100MerkleDistributor` proxy and calls `initialize()` with a valid `_merkleRoot` encoding 1,000,000 KERNEL across 100 users, but forgets to send KERNEL tokens to the contract (or sends fewer than the total).
2. `vestingStartTimestamp` passes; vesting begins.
3. User Alice, with a valid merkle proof for 10,000 KERNEL, calls `claim(10000, proof)`.
4. `_getUnclaimedVestedAmount` returns a non-zero `claimableAmount`.
5. `userClaims[alice].amountClaimed` is updated.
6. `kernel.safeTransfer(alice, amountToSend)` reverts because `kernel.balanceOf(address(this)) == 0`.
7. The entire transaction reverts; Alice cannot claim. All 100 users are blocked until the owner manually funds the contract. [4](#0-3)

    function claim(uint256 amount, bytes32[] calldata merkleProof) external nonReentrant whenNotPaused {
        address user = msg.sender;

        // Verify merkle proof and update user claim data
        _verifyClaimProof(user, amount, merkleProof);

        // Get claimable amount
        uint256 claimableAmount = _getUnclaimedVestedAmount(user, amount);

        if (claimableAmount == 0) {
            revert NoTokensToClaim();
        }

        // Update user claim data
        userClaims[user].lastClaimTimestamp = block.timestamp;
        userClaims[user].amountClaimed += claimableAmount;

        // Calculate fee
        uint256 fee = (claimableAmount * feeInBPS) / FEE_DENOMINATOR;
        uint256 amountToSend = claimableAmount - fee;

        // Transfer tokens
        if (fee > 0) {
            kernel.safeTransfer(protocolTreasury, fee);
        }
        kernel.safeTransfer(user, amountToSend);

        emit Claimed(user, amountToSend);
    }
