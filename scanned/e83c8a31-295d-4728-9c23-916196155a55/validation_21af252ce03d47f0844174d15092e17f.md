### Title
No Minimum Output Slippage Protection in L2 Pool `deposit()` Functions - (File: contracts/pools/RSETHPoolV3.sol, RSETHPoolV3ExternalBridge.sol, RSETHPoolV3WithNativeChainBridge.sol, RSETHPool.sol, RSETHPoolNoWrapper.sol)

### Summary
All L2 pool `deposit()` functions that mint wrsETH to users accept no `minRsETHAmountExpected` parameter. The amount of wrsETH minted is computed entirely from the oracle rate at execution time, with no user-controlled lower bound. If the oracle rate changes between transaction submission and execution, users silently receive fewer wrsETH than they expected for their ETH or LST input. The L1 `LRTDepositPool` already has this protection (`minRSETHAmountExpected`), confirming the design intent was to protect users — but the L2 pools were left unprotected.

### Finding Description
In every L2 pool variant, the ETH and token `deposit()` functions compute the wrsETH output exclusively from the live oracle rate:

```solidity
// RSETHPoolV3.sol (identical pattern in all pool variants)
function deposit(string memory referralId) external payable ... {
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);   // no minimum check
}

function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (...) {
    uint256 rsETHToETHrate = getRate();     // live oracle read
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
```

The oracle rate (`getRate()`) is sourced from a `CrossChainRateReceiver` that is updated asynchronously via LayerZero from L1. The rate strictly increases over time as EigenLayer rewards accrue. Any rate update that lands in the same block as, or just before, a user's deposit transaction causes the user to receive fewer wrsETH than they anticipated when they signed the transaction — with no on-chain mechanism to revert or bound the loss.

By contrast, the L1 deposit pool enforces a hard minimum:

```solidity
// LRTDepositPool.sol
function depositETH(uint256 minRSETHAmountExpected, ...) external payable {
    uint256 rsethAmountToMint = _beforeDeposit(..., minRSETHAmountExpected);
    ...
}
// _beforeDeposit:
if (rsethAmountToMint < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet();
```

The L2 pools have no equivalent guard.

### Impact Explanation
A depositor who submits a transaction expecting `X` wrsETH may receive materially fewer wrsETH if the oracle rate is updated before their transaction executes. The deposited ETH/LST is fully consumed; the shortfall in wrsETH is permanent and unrecoverable. The contract fails to deliver the amount of wrsETH the user was promised at the time of signing. This matches the **Low** impact tier: "Contract fails to deliver promised returns, but doesn't lose value."

### Likelihood Explanation
The rsETH oracle rate is updated periodically from L1 via LayerZero. On any L2 where block times are short and oracle updates are frequent, the probability of a rate update landing in the same block as a user deposit is non-negligible. Users depositing large amounts are most exposed. No privileged access is required — any depositor calling the public `deposit()` function is affected.

### Recommendation
Add a `minRsETHAmountExpected` parameter to all L2 pool `deposit()` functions, mirroring the existing protection in `LRTDepositPool`:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountExpected)
    external payable nonReentrant whenNotPaused ...
{
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(msg.value);
    if (rsETHAmount < minRsETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);
}
```

Apply the same fix to the token `deposit(address token, uint256 amount, ...)` overloads across all pool variants.

### Proof of Concept
1. Oracle rate on L2 is `1.05e18` (1 ETH = 0.952 rsETH). User signs a tx to deposit 10 ETH, expecting ~9.52 wrsETH.
2. Before the user's tx lands, a LayerZero oracle update pushes the rate to `1.10e18`.
3. User's `deposit()` executes: `viewSwapRsETHAmountAndFee(10 ETH)` now returns `~9.09 wrsETH` (using the new rate).
4. User receives ~9.09 wrsETH instead of ~9.52 wrsETH — a ~4.5% shortfall — with no revert and no recourse.
5. The same scenario applies to token deposits (e.g., wstETH) via `deposit(address token, uint256 amount, ...)`.
