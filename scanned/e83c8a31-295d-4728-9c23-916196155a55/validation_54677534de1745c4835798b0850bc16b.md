### Title
`getTokenBalanceMinusFees` uses raw `balanceOf`, allowing directly-transferred tokens to be silently swept to L1 via `bridgeTokens` — (`contracts/pools/RSETHPoolV3ExternalBridge.sol`)

### Summary
`RSETHPoolV3ExternalBridge` tracks only `feeEarnedInToken[token]` but never tracks deposited principal separately. `getTokenBalanceMinusFees` therefore returns the entire contract token balance minus fees. Any ERC-20 tokens sent directly to the pool address (bypassing `deposit`) are silently included in the next `bridgeTokens` call and permanently transferred to the L1 vault, with no recourse for the original sender.

### Finding Description
`deposit(address token, uint256 amount, string referralId)` pulls tokens from the caller and records only the fee portion: [1](#0-0) 

There is no variable tracking the deposited principal. `getTokenBalanceMinusFees` therefore computes: [2](#0-1) 

`bridgeTokens` uses this value as the full bridge amount: [3](#0-2) 

Because `IERC20(token).balanceOf(address(this))` is the raw on-chain balance, any tokens sent directly to the pool address are indistinguishable from legitimately deposited tokens. When `bridgeTokens` is called by the BRIDGER_ROLE as part of routine operations, the entire balance (deposited + donated) minus fees is bridged to `l1VaultETHForL2Chain`. The original sender of the donated tokens receives nothing and has no recovery path.

### Impact Explanation
Tokens sent directly to the pool — whether accidentally or intentionally — are permanently lost to the sender. They are bridged to the L1 vault, a protocol-controlled contract, with no mechanism for the original sender to claim them. This constitutes direct theft of user funds at-rest. The BRIDGER_ROLE does not need to act maliciously; the sweep occurs as a side-effect of normal bridging operations.

**Scoped impact:** Critical — Direct theft of user funds at-rest.

### Likelihood Explanation
ERC-20 tokens can be transferred to any address at any time. Accidental sends to pool contracts are a well-documented real-world occurrence. Additionally, any actor wishing to grief or permanently destroy another user's tokens can do so by sending them to this pool and waiting for the next bridge cycle. The BRIDGER_ROLE calling `bridgeTokens` is a routine, expected operation, so no special trigger is required.

### Recommendation
Introduce a `depositedBalance` mapping that tracks the principal deposited through `deposit()`:

```solidity
mapping(address token => uint256 deposited) public depositedBalance;
```

In `deposit(address token, ...)`, add:
```solidity
depositedBalance[token] += amount;
```

Replace `getTokenBalanceMinusFees` with:
```solidity
function getTokenBalanceMinusFees(address token) public view returns (uint256) {
    return depositedBalance[token] - feeEarnedInToken[token];
}
```

Decrement `depositedBalance[token]` by `balance` after a successful `bridgeTokens` call. Apply the same pattern to `getETHBalanceMinusFees` / `bridgeAssets` / `bridgeAssetsViaNativeBridge` for ETH.
