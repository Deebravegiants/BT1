### Title
Missing `msg.sender` Validation in `MerkleDistributor.claim()` Enables Forced Fee-Bearing Claims on Behalf of Any Account - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

### Summary

`MerkleDistributor.claim()` accepts an arbitrary `account` parameter with no check that `account == msg.sender`. Any unprivileged caller who possesses a valid merkle proof for a victim can force a claim on the victim's behalf. When `feeInBPS > 0`, this permanently extracts a fee from the victim's claimable allocation — tokens the victim would have retained had they waited for the fee to be lowered or zeroed. The sister contract `KernelMerkleDistributor` explicitly guards against this with an `account != msg.sender` revert, confirming the omission in `MerkleDistributor` is unintentional.

### Finding Description

**Root cause — missing caller restriction:**

`MerkleDistributor.claim()` accepts `account` as a caller-supplied parameter and verifies only that a valid merkle proof exists for `(index, account, cumulativeAmount)`. It never checks that `msg.sender == account`. [1](#0-0) 

The leaf is constructed as:

```solidity
bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));
``` [2](#0-1) 

After proof verification, the fee is deducted and tokens are sent to `account`:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;
IERC20(token).safeTransfer(account, amountToSend);
IERC20(token).safeTransfer(protocolTreasury, fee);
``` [3](#0-2) 

The fee ceiling is 10% (`MAX_FEE_IN_BPS = 1000`): [4](#0-3) 

**Contrast with `KernelMerkleDistributor`**, which explicitly enforces the caller restriction:

```solidity
if (account != msg.sender) {
    revert Unauthorized();
}
``` [5](#0-4) 

This inconsistency confirms the omission in `MerkleDistributor` is a defect, not a deliberate design choice.

**Exploit path:**

1. The owner sets `feeInBPS` to any value above 0 (e.g., 1000 = 10%).
2. A victim holds unclaimed tokens and is waiting for the fee to be reduced before claiming.
3. An attacker — who has no privileged access — obtains the victim's merkle proof (proofs are public in standard airdrop deployments).
4. The attacker calls `claim(index, victim, cumulativeAmount, proof)`.
5. The contract verifies the proof, deducts the fee, sends `amountToSend` to the victim, and sends `fee` to `protocolTreasury`.
6. The victim's claim slot is consumed at the current (high) fee rate. The fee amount is permanently lost to the victim.

### Impact Explanation

**Theft of unclaimed yield — High.**

The fee deducted from `claimableAmount` represents yield the victim was entitled to but did not yet claim. By forcing the claim at a moment when `feeInBPS > 0`, the attacker causes the victim to permanently forfeit up to 10% of their claimable allocation. The victim cannot reclaim this amount: `userClaims[account].cumulativeAmount` is updated to `cumulativeAmount`, so the fee-bearing portion is gone. If the victim had waited for the fee to be set to 0, they would have received the full amount. The attacker does not receive the stolen yield (it goes to the treasury), but the victim suffers a direct, irreversible loss of token value.

### Likelihood Explanation

**Medium.** The preconditions are realistic:

- `feeInBPS > 0` is a normal operational state (the contract supports fees up to 10%).
- Merkle proofs for airdrop/distribution contracts are routinely published off-chain (e.g., in a JSON file or IPFS) so that users can self-claim; an attacker can read them without any privileged access.
- The attacker needs only to call a public function with no special role.
- The attack is profitable as a griefing vector whenever the victim is known to be deferring their claim.

### Recommendation

Add a caller restriction identical to the one in `KernelMerkleDistributor`:

```solidity
if (account != msg.sender) {
    revert Unauthorized();
}
```

Insert this check in `MerkleDistributor.claim()` immediately after the `isClaimed` guard, before the merkle proof verification. Additionally, add `nonReentrant` to `claim()` as a defence-in-depth measure, consistent with `KernelMerkleDistributor`.

### Proof of Concept

```solidity
// Attacker EOA, no privileged role required.
// Precondition: feeInBPS = 500 (5%), victim has 1000 tokens claimable.

merkleDistributor.claim(
    currentIndex,          // valid index
    victim,                // victim's address — NOT msg.sender
    1000e18,               // victim's cumulative amount from the published proof
    victimMerkleProof      // obtained from the public proof file
);

// Result:
//   victim receives 950e18 tokens (5% fee deducted)
//   protocolTreasury receives 50e18 tokens
//   victim loses 50e18 tokens they would have kept if fee had been 0
//   victim's claim slot is permanently consumed at the 5% fee rate
```
