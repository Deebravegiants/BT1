Looking at the contract carefully, I can trace the exact execution path described.

**`claimWithdrawal`** (lines 106-123): Claims tokens from the Sonic bridge and leaves them sitting in the contract — it does NOT transfer to `l1Vault`. [1](#0-0) 

**`updateL1Vault`** (lines 139-144): Admin can update `l1Vault` to any new address at any time, with no check for pending (claimed-but-not-transferred) balances. [2](#0-1) 

**`transferToVault`** (lines 128-135): Reads `l1Vault` at call time — not at claim time — and sends the full balance there. [3](#0-2) 

The race condition is real and requires no malicious actors — only a legitimate admin vault migration that happens to occur while tokens are sitting in the contract between the two steps. The admin is not compromised; they are performing a routine upgrade. The contract simply has no mechanism to snapshot the intended destination at claim time.

The atomic alternative `claimAndTransferToVault` (lines 68-99) does not have this problem because it reads and uses `l1Vault` in the same transaction. [4](#0-3)

### Title
Two-Step Claim/Transfer Race Allows `updateL1Vault` to Redirect Claimed Tokens to Wrong Vault — (`contracts/bridges/SonicBridgeReceiver.sol`)

### Summary
`claimWithdrawal` lands tokens in the contract without recording the intended destination vault. If an admin calls `updateL1Vault` before `transferToVault` is called, the tokens are forwarded to the new vault instead of the vault that was the intended recipient at claim time.

### Finding Description
The contract exposes two flows for handling Sonic bridge withdrawals:

1. **Atomic**: `claimAndTransferToVault` — claims and immediately transfers to the current `l1Vault` in one transaction.
2. **Two-step**: `claimWithdrawal` (claim only) + `transferToVault` (transfer only).

In the two-step flow, `claimWithdrawal` marks the withdrawal as claimed and pulls tokens into the contract, but does not record which `l1Vault` was active at that moment. `transferToVault` later reads the storage variable `l1Vault` at execution time:

```solidity
// transferToVault, line 133
IERC20(token).safeTransfer(l1Vault, transferAmount);
```

If `updateL1Vault` is called between these two steps — a routine admin action during a vault migration — the tokens are silently forwarded to the new vault. The original vault, which initiated the Sonic-side burn and expected to receive the bridged assets, receives nothing.

### Impact Explanation
No funds are lost from the protocol's perspective (they arrive at the new vault), but the original L1Vault never receives the tokens it was expecting from the bridge withdrawal it initiated. This breaks the invariant that a Sonic bridge withdrawal must be settled to the vault that was active at burn time. Scoped impact: **Low — Contract fails to deliver promised returns, but doesn't lose value**.

### Likelihood Explanation
Likelihood is low-to-medium. The two-step path is explicitly provided "for manual handling" and is expected to be used in operational scenarios. A vault migration (calling `updateL1Vault`) is a foreseeable admin action. No malicious intent is required from any party — the admin is acting legitimately, and the claimer is acting legitimately. The window exists whenever tokens sit in the contract between the two steps.

### Recommendation
In `claimWithdrawal`, snapshot the current `l1Vault` and store it alongside the withdrawal record, then enforce that `transferToVault` uses the snapshotted address. Alternatively, deprecate the two-step path and require all claims to go through the atomic `claimAndTransferToVault`. At minimum, add a check in `updateL1Vault` that reverts (or emits a warning) if the contract holds a non-zero token balance, signalling pending transfers.
