### Title
`FeeReceiver` ETH Rewards Excluded from `_getTotalEthInProtocol` Before `updateRSETHPrice`, Enabling Yield Theft from Existing rsETH Holders - (File: contracts/LRTOracle.sol)

---

### Summary

`LRTOracle.updateRSETHPrice()` is a permissionless function that computes the rsETH/ETH exchange rate using `_getTotalEthInProtocol()`. That function calls `LRTDepositPool.getTotalAssetDeposits(ETH_TOKEN)` â†’ `getETHDistributionData()`, which explicitly **excludes** ETH sitting in the `FeeReceiver` contract. Because `FeeReceiver.sendFunds()` is also permissionless, an attacker can call `updateRSETHPrice()` while MEV/execution-layer rewards are still sitting in `FeeReceiver`, causing the rsETH price to be set lower than the true value. The attacker then deposits at the understated price to receive excess rsETH, then calls `sendFunds()` to push the rewards in, permanently diluting existing holders.

---

### Finding Description

`FeeReceiver` accumulates MEV and execution-layer ETH rewards via its `receive()` fallback: [1](#0-0) 

`sendFunds()` is the only way to move those rewards into the deposit pool, and it has **no access control** â€” anyone can call it: [2](#0-1) 

`LRTOracle.updateRSETHPrice()` is also permissionless: [3](#0-2) 

Internally it calls `_getTotalEthInProtocol()`, which sums asset balances via `getTotalAssetDeposits`: [4](#0-3) 

`getTotalAssetDeposits` for ETH delegates to `getETHDistributionData()`. The code comment there is explicit: [5](#0-4) 

`getETHDistributionData()` counts only: deposit pool balance, NDC balances, EigenLayer pod shares, unstaking amounts, and the unstaking vault â€” **never** `FeeReceiver.balance`: [6](#0-5) 

The rsETH price is then computed as:

```
newRsETHPrice = (totalETHInProtocol - protocolFeeInETH) / rsethSupply
``` [7](#0-6) 

If `FeeReceiver` holds, say, 10 ETH in accumulated rewards, and `updateRSETHPrice()` is called before `sendFunds()`, those 10 ETH are invisible to the price calculation. The resulting `rsETHPrice` is lower than the true backing per share.

---

### Impact Explanation

When `rsETHPrice` is understated, the deposit function in `LRTDepositPool` mints more rsETH per ETH deposited than the true ratio warrants:

```
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [8](#0-7) 

The attacker receives excess rsETH. Once `sendFunds()` is called and the price corrects upward, the attacker's rsETH is worth more than what they paid. The difference comes directly from the yield that should have accrued to existing rsETH holders. This is **theft of unclaimed yield** (High severity).

---

### Likelihood Explanation

Both `updateRSETHPrice()` and `FeeReceiver.sendFunds()` are permissionless. MEV rewards accumulate continuously in `FeeReceiver` between operator calls to `sendFunds()`. An attacker only needs to:
1. Monitor `FeeReceiver.balance` off-chain.
2. When a meaningful balance has accumulated, call `updateRSETHPrice()` first.
3. Immediately deposit ETH into `LRTDepositPool` at the understated price.
4. Call `sendFunds()` to push rewards in.
5. Wait for the next legitimate `updateRSETHPrice()` call to realize the gain.

No privileged access, no front-running of a specific transaction, and no external protocol dependency is required. Likelihood is **Medium** (requires monitoring and timing, but is fully permissionless).

---

### Recommendation

`FeeReceiver.sendFunds()` should be called atomically inside `updateRSETHPrice()` (or inside `_getTotalEthInProtocol()`) before the price is computed, mirroring the Derby recommendation of calling `claim` before `pushTotalUnderlyingToController`. Alternatively, `getETHDistributionData()` should include `address(feeReceiver).balance` in its accounting so that pending rewards are always reflected in the price, regardless of call ordering.

---

### Proof of Concept

1. Protocol has 1000 ETH total backing, 1000 rsETH supply â†’ price = 1.0 ETH/rsETH.
2. 10 ETH MEV rewards arrive in `FeeReceiver` (not yet sent to deposit pool).
3. Attacker calls `updateRSETHPrice()`:
   - `_getTotalEthInProtocol()` returns 1000 ETH (FeeReceiver excluded).
   - `newRsETHPrice` = 1000 / 1000 = **1.0 ETH/rsETH** (should be 1010/1000 = 1.01).
4. Attacker deposits 10 ETH â†’ mints `10 / 1.0 = 10 rsETH` (correct mint would be `10 / 1.01 â‰ˆ 9.9 rsETH`).
5. Attacker calls `FeeReceiver.sendFunds()` â†’ 10 ETH moves to deposit pool.
6. Next `updateRSETHPrice()` call: total ETH = 1020, supply = 1010 rsETH â†’ price â‰ˆ 1.0099.
7. Attacker holds 10 rsETH worth â‰ˆ 10.099 ETH, having deposited 10 ETH â€” capturing ~0.099 ETH of yield that belonged to the original 1000 rsETH holders. [2](#0-1) [3](#0-2) [5](#0-4)

    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
