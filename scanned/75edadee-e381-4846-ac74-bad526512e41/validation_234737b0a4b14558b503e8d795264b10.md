### Title
Block Stuffing Enables `minimumFeeInWei` Race to Revert `bridgeAssetsViaNativeBridge` — (`contracts/bridges/LineaMessenger.sol`)

---

### Summary

`LineaMessenger.sendETHToL1ViaBridge` reads `minimumFeeInWei` on-chain at execution time and immediately checks `value <= minimumFee`. If an attacker stuffs Linea blocks to delay the BRIDGER_ROLE's `bridgeAssetsViaNativeBridge` transaction past a routine bridge fee update, the pool's ETH balance (fixed at submission time) may now be ≤ the new higher minimum fee, causing the call to revert with `InsufficientAmountForBridge` and temporarily freezing pool ETH.

---

### Finding Description

The call chain is:

**Pool → Messenger → Bridge**

`RSETHPool.bridgeAssetsViaNativeBridge` passes the entire `ethBalanceMinusFees` as both `msg.value` and `value`: [1](#0-0) 

Inside `LineaMessenger.sendETHToL1ViaBridge`, the minimum fee is read live from the bridge at execution time, and the check is: [2](#0-1) 

The `value` parameter is the pool's entire bridgeable ETH balance, determined at the moment `bridgeAssetsViaNativeBridge` is called. If the BRIDGER_ROLE's transaction is delayed (via block stuffing) and the Linea bridge operator raises `minimumFeeInWei` during that window, the on-chain read at line 39 returns the new higher fee. If `ethBalanceMinusFees ≤ newMinimumFee`, line 40 reverts with `InsufficientAmountForBridge`.

The `ILineaMessageService` interface confirms `minimumFeeInWei` is a mutable external view: [3](#0-2) 

There is no slippage/tolerance parameter, no retry mechanism, and no way for the BRIDGER_ROLE to pass a fee cap or minimum-fee snapshot to protect against this race.

---

### Impact Explanation

**Impact: Low — Block stuffing / Temporary freezing of funds.**

The ETH is not lost; it remains in the pool. However, the bridge call reverts and the pool ETH cannot be moved to L1 until either:
- More ETH accumulates in the pool (via user deposits) so that `ethBalanceMinusFees > newMinimumFee`, or
- The Linea bridge operator lowers `minimumFeeInWei` again.

During this window, user withdrawals that depend on L1 liquidity being replenished are delayed.

---

### Likelihood Explanation

**Likelihood: Low.**

Three conditions must align simultaneously:
1. An attacker successfully stuffs enough Linea blocks to delay the BRIDGER_ROLE's transaction (Linea uses a centralized sequencer, making classic block stuffing harder than on Ethereum mainnet, but not impossible via high-gas-price spam).
2. The Linea bridge operator raises `minimumFeeInWei` during that exact window (a routine but infrequent operation).
3. The pool's ETH balance at submission time is small enough to fall below the new minimum fee.

The combination is unlikely in practice but is a concrete, locally testable path on unmodified code.

---

### Recommendation

1. **Accept a caller-supplied `maxFee` parameter** in `sendETHToL1ViaBridge` and revert if `minimumFeeInWei > maxFee`, giving the BRIDGER_ROLE a slippage guard.
2. Alternatively, **re-read `minimumFeeInWei` and pass it as a parameter from the pool**, so the BRIDGER_ROLE can simulate the call off-chain and set a deadline/max-fee before submitting.
3. Consider adding a **minimum bridgeable threshold** in the pool (e.g., require `ethBalanceMinusFees >= 2 × currentMinimumFee`) so the pool only attempts bridging when there is a comfortable margin above the fee floor.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import "forge-std/Test.sol";
import {LineaMessenger} from "contracts/bridges/LineaMessenger.sol";

contract MockLineaBridge {
    uint256 public minimumFeeInWei = 0.001 ether;

    function setMinimumFee(uint256 fee) external {
        minimumFeeInWei = fee;
    }

    function sendMessage(address, uint256, bytes memory) external payable {
        // no-op
    }
}

contract LineaMessengerBlockStuffingTest is Test {
    LineaMessenger messenger;
    MockLineaBridge bridge;
    address admin = address(0xAD);
    address target = address(0xBEEF);

    function setUp() public {
        messenger = new LineaMessenger(admin);
        bridge = new MockLineaBridge();
    }

    function test_blockStuffing_feeRaceReverts() public {
        // Pool has 0.005 ETH, old minimumFee = 0.001 ETH → would succeed
        uint256 poolBalance = 0.005 ether;
        vm.deal(address(this), poolBalance);

        // Simulate: attacker stuffs blocks, bridge operator raises fee to 0.006 ETH
        bridge.setMinimumFee(0.006 ether);

        // BRIDGER_ROLE tx is now included — value (0.005) <= minimumFee (0.006) → revert
        vm.expectRevert(LineaMessenger.InsufficientAmountForBridge.selector);
        messenger.sendETHToL1ViaBridge{value: poolBalance}(
            address(bridge), target, poolBalance
        );

        // ETH is NOT lost — it was never sent (revert), but pool ETH is frozen
        // until fee drops or more ETH accumulates
    }
}
```
