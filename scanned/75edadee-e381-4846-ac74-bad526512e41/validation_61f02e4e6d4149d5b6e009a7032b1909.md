### Title
CCIP Pool Rotation Creates In-Flight Message Execution Window — (`contracts/ccip/WrappedRSETH.sol`)

---

### Summary

`WrappedRSETH` exposes no atomic "rotate minter" operation. Revoking the old pool's mint role and granting it to a new pool are two separate owner transactions. Any CCIP message that was already committed on-chain between those two transactions will revert when the old pool attempts to call `mint`, permanently failing the message and temporarily freezing the user's bridged funds.

---

### Finding Description

`revokeMintRole` and `grantMintRole` are independent, non-atomic owner calls: [1](#0-0) [2](#0-1) 

`mint` enforces the role at call time via `onlyMinter`: [3](#0-2) [4](#0-3) 

The exact sequence that triggers the freeze:

1. A user initiates a cross-chain transfer on L1. The L1 CCIP pool burns/locks the user's rsETH and emits a CCIP message.
2. The CCIP DON commits the message on L2 (it is now on-chain and awaiting execution).
3. The owner calls `revokeMintRole(oldPool)` as part of a pool upgrade — a fully legitimate operational action.
4. The CCIP router attempts to execute the committed message by calling `oldPool.releaseOrMint()`, which internally calls `WrappedRSETH.mint(user, amount)`.
5. `onlyMinter` reverts with `SenderNotMinter(oldPool)` because the role was already removed.
6. CCIP marks the message as `FAILED`. The user's L1 funds are already burned/locked. Recovery requires manual re-execution, but re-execution through the same `oldPool` will continue to revert — the funds are stuck until the owner explicitly restores `oldPool`'s mint role or CCIP governance intervenes.

There is no grace period, no in-flight message check, and no atomic swap primitive in the contract.

---

### Impact Explanation

**Medium — Temporary freezing of funds.**

The user's L1 rsETH is already burned at the point of failure. The L2 tokens are never minted. The funds are inaccessible until an admin manually restores the old pool's role, re-executes the message, and then re-revokes the role — a multi-step recovery that is error-prone and time-consuming. This matches the allowed impact scope: *"Temporary freezing of funds."*

---

### Likelihood Explanation

Pool upgrades are a routine operational event for any live CCIP integration. CCIP message finality windows (especially for chains requiring many block confirmations) can span minutes to hours, making the overlap between a committed-but-unexecuted message and a role revocation highly probable during any upgrade cycle. No attacker action is required; the trigger is a normal admin operation.

---

### Recommendation

Add an atomic `rotateMintRole(address oldMinter, address newMinter)` function that grants the new role before revoking the old one in a single transaction:

```solidity
function rotateMintRole(address oldMinter, address newMinter) external onlyOwner {
    grantMintRole(newMinter);   // new pool can mint immediately
    revokeMintRole(oldMinter);  // old pool revoked only after new is live
}
```

Additionally, document that operators must wait for all in-flight CCIP messages to be executed before revoking the old pool's role, and consider adding an off-chain monitoring check before any role revocation.

// 1. Deploy WrappedRSETH with owner = address(this)

// 3. Simulate a committed CCIP message: record oldPool as the executor
// 4. Owner calls revokeMintRole(oldPool)
// 5. Attempt oldPool.mint(user, 1 ether) — assert revert SenderNotMinter
// 6. Assert user received 0 tokens despite L1 burn already having occurred

function testInFlightFreeze() public {
    WrappedRSETH token = new WrappedRSETH("Wrapped rsETH", "wrsETH", 18, 0, address(this));
    address oldPool = address(0xBEEF);
    address user    = address(0xCAFE);

    token.grantMintRole(oldPool);

    // Simulate: CCIP message committed, not yet executed
    // Owner rotates pool
    token.revokeMintRole(oldPool);

    // CCIP router calls oldPool → oldPool calls mint
    vm.prank(oldPool);
    vm.expectRevert(abi.encodeWithSelector(WrappedRSETH.SenderNotMinter.selector, oldPool));
    token.mint(user, 1 ether);

    // User has 0 tokens; L1 funds already burned → temporary freeze
    assertEq(token.balanceOf(user), 0);
}
