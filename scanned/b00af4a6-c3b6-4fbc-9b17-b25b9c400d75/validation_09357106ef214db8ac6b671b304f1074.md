### Title
Missing Slippage Protection in `removeLiquidity` Exposes LPs to Composition Loss - (File: metric-core/contracts/MetricOmmPool.sol, metric-core/contracts/libraries/LiquidityLib.sol)

### Summary

`MetricOmmPool.removeLiquidity` burns LP shares and returns the proportional token amounts based on the **live bin state at execution time**, but accepts no `minAmount0Out` / `minAmount1Out` guard and no deadline. Because swaps continuously mutate `token0BalanceScaled` and `token1BalanceScaled` inside each bin, the composition an LP receives can differ materially from what they observed when they signed the transaction. No periphery wrapper exists for `removeLiquidity` (unlike `addLiquidity`, which is wrapped by `MetricOmmPoolLiquidityAdder` with explicit `maxAmountToken0` / `maxAmountToken1` caps), so there is no off-chain-to-on-chain slippage fence anywhere in the production surface.

### Finding Description

`MetricOmmPool.removeLiquidity` signature:

```solidity
function removeLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata extensionData
) external nonReentrant(PoolActions.REMOVE_LIQUIDITY)
  returns (uint256 amount0Removed, uint256 amount1Removed)
``` [1](#0-0) 

Inside `LiquidityLib.removeLiquidity`, the payout is computed as:

```solidity
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
uint256 amount1Scaled = _checkedMul(binState.token1BalanceScaled, sharesToRemove) / binTotalSharesVal;
``` [2](#0-1) 

Both `token0BalanceScaled` and `token1BalanceScaled` are mutated by every swap that crosses the bin. A swap selling token0 into the bin increases `token1BalanceScaled` and decreases `token0BalanceScaled`; the inverse swap does the opposite. There is no check after the computation that the returned amounts meet any caller-specified floor.

The `MetricOmmPoolLiquidityAdder` periphery contract provides `maxAmountToken0` / `maxAmountToken1` caps for `addLiquidity` and enforces them in the callback:

```solidity
if (amount0Delta > max0 || amount1Delta > max1) {
    revert MaxAmountExceeded(amount0Delta, amount1Delta, max0, max1);
}
``` [3](#0-2) 

No equivalent periphery wrapper exists for `removeLiquidity`. The `IMetricOmmPoolLiquidityAdder` interface exposes only `addLiquidityExactShares` and `addLiquidityWeighted`. [4](#0-3) 

The `MetricOmmSimpleRouter` adds `amountOutMinimum` / `amountInMaximum` for swaps, but has no liquidity-removal path at all. [5](#0-4) 

### Impact Explanation

An LP who submits `removeLiquidity` targeting a bin that currently holds, say, 50 % token0 / 50 % token1 can have their transaction land after a large swap has drained all token0 from that bin. They then receive 0 token0 and 100 % token1. If token0 is the higher-value leg at that moment, the LP suffers a direct, measurable loss of principal relative to the composition they observed and intended to withdraw. Because `removeLiquidity` transfers tokens immediately to `owner` with no revert path after the amounts are computed, the loss is irreversible within the transaction. [6](#0-5) 

### Likelihood Explanation

Any LP calling `removeLiquidity` directly on the pool (the only available path) is exposed. The trigger is unprivileged: any other user executing a swap against the same bin between the LP's transaction submission and its inclusion can shift the composition. In a live oracle-driven pool with active trading, this is a routine occurrence. The absence of a deadline compounds the risk for transactions that sit in the mempool during periods of high gas prices.

### Recommendation

1. Add `minAmount0Out` and `minAmount1Out` parameters to `MetricOmmPool.removeLiquidity` (or enforce them in `LiquidityLib.removeLiquidity`) and revert if the computed amounts fall below the caller's floor.
2. Add a `deadline` parameter and check `block.timestamp <= deadline` at entry.
3. Add a periphery `MetricOmmPoolLiquidityRemover` wrapper (mirroring `MetricOmmPoolLiquidityAdder`) that exposes these guards to EOA callers.

Example core-level fix:

```solidity
function removeLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    uint256 minAmount0Out,
    uint256 minAmount1Out,
    uint256 deadline,
    bytes calldata extensionData
) external nonReentrant(PoolActions.REMOVE_LIQUIDITY)
  returns (uint256 amount0Removed, uint256 amount1Removed)
{
    require(block.timestamp <= deadline, DeadlineExpired());
    // ... existing logic ...
    require(amount0Removed >= minAmount0Out && amount1Removed >= minAmount1Out, InsufficientOutput());
}
```

### Proof of Concept

1. Pool has bin 0 with `token0BalanceScaled = 1000`, `token1BalanceScaled = 1000`, `binTotalShares = 10000`.
2. LP holds 5000 shares (50 %). They submit `removeLiquidity` expecting ~500 token0 and ~500 token1.
3. Before the LP's transaction is mined, a trader executes a large exact-output swap that buys all 1000 token0 from bin 0, leaving `token0BalanceScaled = 0`, `token1BalanceScaled = 2000` (approximately, after fee accounting).
4. LP's `removeLiquidity` executes:
   - `amount0Scaled = (0 * 5000) / 10000 = 0`
   - `amount1Scaled = (2000 * 5000) / 10000 = 1000`
5. LP receives 0 token0 and ~1000 token1. If token0 > token1 in oracle value, the LP has received less total value than the 500/500 split they intended, with no on-chain recourse. [7](#0-6)
