### Title
Third-party `addLiquidity` on behalf of victim creates dust remainder that permanently blocks `removeLiquidity` — (`metric-core/contracts/libraries/LiquidityLib.sol`)

### Summary

`addLiquidity` allows any caller to credit shares to an arbitrary `owner`. `removeLiquidity` enforces a `MinimalLiquidity` dust floor on the *remaining* shares after a burn. An attacker can front-run a victim's full-withdrawal transaction by adding exactly 1 share to the victim's position, causing the victim's transaction to leave a 1-share remainder that is below the dust floor and reverts. This can be repeated indefinitely, permanently blocking the victim from withdrawing their liquidity.

### Finding Description

`MetricOmmPool.addLiquidity` accepts an `owner` parameter that is independent of `msg.sender` (the "operator pattern"): [1](#0-0) 

The callback is invoked on `msg.sender`, so the attacker pays the tokens, but shares are credited to the victim's `(owner, salt, binIdx)` key: [2](#0-1) 

`removeLiquidity` enforces that any non-zero remainder must be at or above `MINIMAL_MINTABLE_LIQUIDITY`: [3](#0-2) 

The attack sequence:

1. Victim holds `X` shares in bin `B` and submits `removeLiquidity(victim, salt, {X shares}, "")`.
2. Attacker front-runs with `addLiquidity(victim, salt, {1 share in bin B}, callbackData, "")`. The attacker pays the token cost for 1 share; victim now holds `X + 1` shares.
3. Victim's transaction executes: `newUserShares = (X+1) - X = 1`. The check `1 > 0 && 1 < MINIMAL_MINTABLE_LIQUIDITY` is true → reverts with `MinimalLiquidity(1, 1000)`.
4. Attacker repeats on every retry.

The `addLiquidity` dust check on the *add* side does not block this because the victim already holds `X >= MINIMAL_MINTABLE_LIQUIDITY` shares, so `newUserShares = X + 1 >= MINIMAL_MINTABLE_LIQUIDITY` passes: [4](#0-3) 

`removeLiquidity` enforces `msg.sender == owner`, so only the victim can call it, but the victim cannot atomically read their current share count and remove it in a single EOA transaction: [5](#0-4) 

The periphery `addLiquidityExactShares(pool, owner, ...)` overload explicitly supports adding on behalf of another owner and is confirmed by a test: [6](#0-5) 

The position key is `keccak256(abi.encode(owner, salt, bin))`, and the salt is observable on-chain from past `LiquidityAdded` events, so the attacker can always target the correct position: [7](#0-6) 

### Impact Explanation

An EOA victim cannot withdraw their liquidity as long as the attacker keeps front-running. The victim's principal is locked in the pool indefinitely. The attacker's per-front-run cost is the token value of 1 share (very small), making sustained griefing economically viable. The victim's only escape is a contract-based atomic read-and-remove, which is not provided by the protocol's periphery.

### Likelihood Explanation

The attack requires MEV front-running capability (available on most EVM chains) and knowledge of the victim's `(owner, salt, binIdx)` tuple, which is fully public from on-chain events. The per-attack token cost is minimal (1 share). Any pool without a `DEPOSIT_ALLOWLIST_PROVIDER` extension is vulnerable.

### Recommendation

1. **Restrict third-party adds**: In `addLiquidity`, require `msg.sender == owner` unless the owner has explicitly granted operator approval (e.g., via a signed permit or an on-chain allowance mapping).
2. **Alternatively**, add a `removeAllLiquidity` variant that burns the caller's entire share balance in a bin without checking the dust floor on the remainder (since the remainder is 0).
3. **Alternatively**, remove the `MinimalLiquidity` check from `removeLiquidity` (apply it only on `addLiquidity`), so users can always exit to zero regardless of intermediate share counts.

### Proof of Concept

```solidity
// Attacker contract
contract Attacker is IMetricOmmModifyLiquidityCallback {
    function grief(address pool, address victim, uint80 salt, int256 bin) external {
        int256[] memory bins = new int256[](1);
        bins[0] = bin;
        uint256[] memory shares = new uint256[](1);
        shares[0] = 1; // 1 share — passes add-side MinimalLiquidity because victim already has >= 1000
        LiquidityDelta memory d = LiquidityDelta({binIdxs: bins, shares: shares});
        IMetricOmmPool(pool).addLiquidity(victim, salt, d, "", "");
    }
    function metricOmmModifyLiquidityCallback(uint256 a0, uint256 a1, bytes calldata) external {
        // pay tokens
        if (a0 > 0) IERC20(token0).transfer(msg.sender, a0);
        if (a1 > 0) IERC20(token1).transfer(msg.sender, a1);
    }
}

// Scenario:
// 1. victim has 10_000 shares in bin 4
// 2. victim submits removeLiquidity(victim, salt, {10_000 shares}, "")
// 3. attacker.grief(pool, victim, salt, 4)  ← front-runs; victim now has 10_001 shares
// 4. victim's tx: newUserShares = 10_001 - 10_000 = 1 < 1000 → MinimalLiquidity revert
// 5. repeat indefinitely
```
