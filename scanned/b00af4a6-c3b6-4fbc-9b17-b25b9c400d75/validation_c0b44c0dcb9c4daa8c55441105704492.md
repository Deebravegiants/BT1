### Title
`addLiquidity` Executes Deposits Into a Paused Pool Without Checking `pauseLevel` — (`metric-core/contracts/MetricOmmPool.sol`)

### Summary
`MetricOmmPool.addLiquidity` does not apply the `whenNotPaused` guard. When the pool is paused at any level (admin or protocol), swaps are correctly blocked, but liquidity providers can still deposit real tokens into the pool. This is the direct analog of the external report's missing paused-agent check: a state-changing entry point iterates over and commits user funds without first verifying the pool's paused state.

### Finding Description

`MetricOmmPool` maintains a `pauseLevel` variable (0 = active, 1 = admin-paused, 2 = protocol-paused). The `swap` function correctly enforces `whenNotPaused`: [1](#0-0) 

The `_checkNotPaused` guard rejects any call when `pauseLevel != 0`: [2](#0-1) 

However, `addLiquidity` carries no such guard: [3](#0-2) 

`removeLiquidity` also lacks the guard, but omitting it there is a deliberate design choice (LPs must always be able to exit). Omitting it on `addLiquidity` is not: there is no reason to accept new deposits into a pool that has been halted because of an oracle compromise, price anomaly, or active exploit.

The deposit path calls `LiquidityLib.addLiquidity`, which pulls tokens from the caller via a callback and permanently updates `binTotals`, `_binStates`, `_binTotalShares`, and `_positionBinShares` — all real, persistent state changes that commit user principal to the pool. [4](#0-3) 

### Impact Explanation

A user who calls `addLiquidity` while the pool is paused at level 1 or 2 transfers real token balances into a pool that was halted for a safety reason. If the pause was triggered by an oracle manipulation or an active exploit of the bin-price math, those freshly deposited tokens are immediately at risk once the pool is unpaused or if the attacker finds another entry point (e.g., a non-paused extension). This is a direct loss of user principal that bypasses the intended safety boundary of the pause mechanism.

### Likelihood Explanation

The pool's pause mechanism is explicitly designed for emergency use. Any time a protocol-level pause (level 2) is triggered, uninformed LPs — or bots that monitor mempool liquidity opportunities — can still call `addLiquidity`. The missing guard requires no special privilege from the depositor; any address can trigger it.

### Recommendation

Add `whenNotPaused` to `addLiquidity`, mirroring the guard already present on `swap`:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
```

`removeLiquidity` should remain unguarded so LPs can always exit a paused pool.

### Proof of Concept

1. Pool is deployed and operating normally.
2. Protocol detects an oracle anomaly and calls `factory.setPause(poolAddr, 2)` → `pool.setPause(2)`. `pauseLevel` is now 2.
3. `swap` is now blocked by `whenNotPaused` / `_checkNotPaused`.
4. Alice (or a bot) calls `pool.addLiquidity(alice, salt, deltas, ...)`. The call succeeds — no pause check exists.
5. `LiquidityLib.addLiquidity` executes, the callback pulls Alice's tokens, and `binTotals.scaledToken0/1` are updated with her deposit.
6. Alice's tokens are now locked in a pool that was paused due to a safety issue. If the underlying issue (e.g., manipulated oracle price) is exploited before or after unpausing, Alice's principal is drained through a subsequent swap at the bad price. [5](#0-4) [6](#0-5)
