### Title
Blacklisted LP Cannot Withdraw Principal — Permanent Fund Lock in `removeLiquidity` - (File: metric-core/contracts/libraries/LiquidityLib.sol)

---

### Summary

`LiquidityLib.removeLiquidity` unconditionally transfers withdrawn tokens to `owner` via `safeTransfer`. Because `MetricOmmPool.removeLiquidity` enforces `msg.sender == owner` with no alternative recipient parameter, an LP whose address is blacklisted by a pool token (e.g., USDC) after depositing liquidity can never recover their principal. Every call to `removeLiquidity` reverts at the transfer step, permanently locking the LP's funds inside the pool.

---

### Finding Description

`MetricOmmPool.removeLiquidity` enforces that the caller is the position owner:

```solidity
// MetricOmmPool.sol line 206
if (msg.sender != owner) revert NotPositionOwner();
```

It then delegates to `LiquidityLib.removeLiquidity`, which computes the owed token amounts and transfers them directly to `owner`:

```solidity
// LiquidityLib.sol lines 242-247
if (amount0Removed > 0) {
    IERC20(ctx.token0).safeTransfer(owner, amount0Removed);
}
if (amount1Removed > 0) {
    IERC20(ctx.token1).safeTransfer(owner, amount1Removed);
}
```

If `owner` is blacklisted by the token contract (USDC/USDT implement `transfer` reverts for blacklisted addresses), `safeTransfer` reverts. Because the entire transaction reverts atomically, the share-burn state updates (lines 210–214) are also rolled back, leaving the LP's shares intact but permanently unwithdrawable. There is no `removeLiquidityTo(address recipient)` overload, no admin rescue path for LP principal, and no try-catch around the transfer.

---

### Impact Explanation

An LP who provided liquidity to a USDC/USDT pool and is subsequently blacklisted by the token issuer loses their entire deposited principal. The funds remain in the pool's accounting (`binState.token0BalanceScaled`, `binState.token1BalanceScaled`) but can never be extracted. This is a direct, permanent loss of user principal with no recovery path.

---

### Likelihood Explanation

USDC and USDT both implement address blacklisting. The contest README explicitly includes `Blocklists` of ERC20 tokens as in-scope behavior (mirroring the BendDAO judge's rationale). Any LP using a USDC or USDT pool who is later blacklisted — whether due to regulatory action, a compromised key, or mistaken flagging — triggers this condition. The trigger requires no attacker: the LP themselves is the victim.

---

### Recommendation

Add a `recipient` parameter to `removeLiquidity` so the LP can direct withdrawn tokens to an address that is not blacklisted:

```solidity
function removeLiquidity(
    address owner,
    address recipient,   // <-- new
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata extensionData
) external ...
```

Pass `recipient` through to `LiquidityLib.removeLiquidity` and replace `safeTransfer(owner, ...)` with `safeTransfer(recipient, ...)`. The `msg.sender == owner` check still protects against unauthorized withdrawals; only the destination of the tokens changes.

---

### Proof of Concept

1. Pool is deployed with USDC as `token1`.
2. Alice (`owner = alice`) calls `addLiquidity` and deposits 10,000 USDC into bin 0.
3. USDC issuer blacklists `alice`.
4. Alice calls `removeLiquidity(alice, salt, deltas, "")`.
5. `MetricOmmPool` passes the call to `LiquidityLib.removeLiquidity`.
6. Share accounting is computed; `amount1Removed = 10,000`.
7. `IERC20(usdc).safeTransfer(alice, 10_000)` reverts because `alice` is blacklisted.
8. The entire transaction reverts; Alice's shares are not burned.
9. Alice retries — same revert every time. Her 10,000 USDC is permanently locked. [1](#0-0) [2](#0-1) [3](#0-2)

  function removeLiquidity(address owner, uint80 salt, LiquidityDelta calldata deltas, bytes calldata extensionData)
    external
    nonReentrant(PoolActions.REMOVE_LIQUIDITY)
    returns (uint256 amount0Removed, uint256 amount1Removed)
  {
    if (deltas.binIdxs.length == 0) return (0, 0);
    if (deltas.binIdxs.length != deltas.shares.length) revert LiquidityDeltaLengthMismatch();

    );
    _afterRemoveLiquidity(msg.sender, owner, salt, deltas, amount0Removed, amount1Removed, extensionData);
  }
