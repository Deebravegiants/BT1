### Title
Unrestricted `addLiquidity` Caller Can Grief Any LP Position, Blocking Partial Withdrawals via `minimalMintableLiquidity` Trap — (File: `metric-core/contracts/MetricOmmPool.sol`, `metric-core/contracts/libraries/LiquidityLib.sol`)

---

### Summary

`addLiquidity` accepts a caller-supplied `owner` address with no `msg.sender == owner` guard. Any unprivileged caller can add shares to any existing LP position. By adding a dust amount of shares (`1 ≤ delta < minimalMintableLiquidity`), an attacker traps the victim: the victim's original removal amount now leaves a remainder that fails the `minimalMintableLiquidity` check inside `removeLiquidity`, reverting the transaction and blocking partial withdrawals.

---

### Finding Description

In `MetricOmmPool.addLiquidity`, the `owner` parameter is fully caller-controlled with no access restriction:

```solidity
function addLiquidity(
    address owner,   // ← any address; no msg.sender == owner check
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) ...
``` [1](#0-0) 

Inside `LiquidityLib.addLiquidity`, shares are credited to `positionBinShares[_positionBinKey(owner, salt, bin)]` regardless of who the actual caller is:

```solidity
bytes32 posKey = _positionBinKey(owner, salt, int8(binIdx));
...
positionBinShares[posKey] = newUserShares;   // victim's shares updated by attacker
``` [2](#0-1) 

The only guard during `addLiquidity` is:

```solidity
if (newUserShares < ctx.minimalMintableLiquidity) revert MinimalLiquidity(...);
``` [3](#0-2) 

Since the victim already holds `X >= minimalMintableLiquidity` shares, adding `delta = 1` satisfies `X + 1 >= minimalMintableLiquidity` and passes.

During `removeLiquidity`, the victim tries to remove their original `X` shares. The check is:

```solidity
uint256 newUserShares = userShares - sharesToRemove;  // = (X+1) - X = 1
if (newUserShares > 0 && newUserShares < ctx.minimalMintableLiquidity) {
    revert MinimalLiquidity(newUserShares, ctx.minimalMintableLiquidity);  // REVERTS
}
``` [4](#0-3) 

The victim's partial withdrawal is blocked. The victim must remove all `X + delta` shares or none. Note that `removeLiquidity` does enforce `msg.sender == owner`:

```solidity
if (msg.sender != owner) revert NotPositionOwner();
``` [5](#0-4) 

This asymmetry — anyone can add for any owner, but only the owner can remove — is the root cause. It is the direct analog of the TradingVault `userLastDepositTime` bug: a third party mutates the victim's position state without consent, breaking their withdrawal flow.

---

### Impact Explanation

- Any LP's partial withdrawal is griefable for near-zero cost. When a bin has large `binTotalShares`, the cost of 1 share is `ceil(binState.token0BalanceScaled / binTotalShares)` ≈ 1 wei per token.
- Smart contract LPs (e.g., routers, vaults, liquidity managers) that track shares internally and call `removeLiquidity` with a fixed amount will have their calls permanently revert until they are upgraded or the removal amount is adjusted.
- The attack is repeatable: after the victim adjusts their removal amount to account for the extra shares, the attacker can add another dust amount.
- The attacker's deposited tokens are not lost to the protocol — they go to the victim on full withdrawal — but the attacker can sustain the grief indefinitely at minimal cost.
- Core liquidity removal flow is broken for the targeted position. [6](#0-5) 

---

### Likelihood Explanation

- No privilege required: any EOA or contract can call `addLiquidity` with any `owner` address.
- Attack cost is near-zero for bins with high total shares.
- The attack is repeatable after each victim adjustment.
- The position key `keccak256(abi.encode(owner, salt, bin))` is fully predictable from on-chain data. [7](#0-6) 

---

### Recommendation

Add an access control check in `addLiquidity` requiring `msg.sender == owner`, mirroring the guard already present in `removeLiquidity`. Alternatively, implement an operator-approval mechanism (similar to ERC-721 `setApprovalForAll`) so that only the owner or an approved operator can add liquidity on behalf of an owner.

```solidity
// In MetricOmmPool.addLiquidity, before calling LiquidityLib:
if (msg.sender != owner) revert NotPositionOwner();
```

---

### Proof of Concept

1. Victim holds `X = minimalMintableLiquidity` shares in bin `0`, salt `0`.
2. Attacker calls `pool.addLiquidity(owner=victim, salt=0, deltas={binIdxs:[0], shares:[1]}, ...)`.
   - `newUserShares = X + 1 >= minimalMintableLiquidity` → passes the add-side guard.
   - Attacker pays ≈ 1 wei of each token via the `metricOmmModifyLiquidityCallback`.
3. Victim calls `pool.removeLiquidity(owner=victim, salt=0, deltas={binIdxs:[0], shares:[X]}, ...)`.
   - `newUserShares = (X+1) - X = 1`.
   - `1 > 0 && 1 < minimalMintableLiquidity` → **REVERTS** with `MinimalLiquidity`.
4. Victim is forced to remove all `X + 1` shares or none.
5. Attacker repeats step 2 after victim adjusts, sustaining the grief indefinitely. [8](#0-7) [9](#0-8)
