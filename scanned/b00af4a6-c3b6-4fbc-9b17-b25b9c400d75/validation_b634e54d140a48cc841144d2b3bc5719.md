### Title
Excess native ETH sent to `MetricOmmSimpleRouter` during WETH swaps is silently retained and claimable by any caller ŌĆö (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

When a user calls any payable swap entry-point (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) with `tokenIn = WETH` and supplies `msg.value` greater than the actual swap input, the `pay()` helper wraps only the required amount and leaves the surplus native ETH on the router. Because `refundETH()` is an unrestricted public function that forwards the router's **entire** ETH balance to `msg.sender`, any third party can immediately claim the stranded ETH, causing a direct loss of user principal.

---

### Finding Description

`PeripheryPayments.pay()` handles WETH settlement by inspecting the router's live native balance:

```solidity
// metric-periphery/contracts/base/PeripheryPayments.sol  L74-L77
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();   // wraps exactly `value`
    IERC20(WETH).safeTransfer(recipient, value);
    // ŌåÉ nativeBalance - value ETH is silently left on the router
}
``` [1](#0-0) 

When `nativeBalance > value`, only `value` wei is consumed; the remainder is never returned to the caller. The swap functions themselves (`exactInputSingle`, etc.) make no post-swap refund call.

`refundETH()` is a public, caller-attributed function with no access control:

```solidity
// metric-periphery/contracts/base/PeripheryPayments.sol  L58-L63
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);   // sends to whoever calls, not the original payer
    }
}
``` [2](#0-1) 

The protocol's own test suite confirms the design intent: users are expected to bundle `refundETH()` as a second call inside `multicall` to recover surplus ETH. The test `test_multicall_ethInput_exactInputSingle_refundsUnusedEth` explicitly demonstrates this:

```solidity
calls[0] = abi.encodeWithSelector(router.exactInputSingle.selector, ...);
calls[1] = abi.encodeWithSelector(router.refundETH.selector);
router.multicall{value: msgValue}(calls);
``` [3](#0-2) 

However, when a user calls `exactInputSingle` (or any other payable swap function) **directly** ŌĆö not via `multicall` ŌĆö with excess `msg.value`, no automatic refund occurs. The surplus ETH sits on the router until any external account calls `refundETH()` and claims it.

---

### Impact Explanation

Any user who sends `msg.value > amountIn` when `tokenIn == WETH` loses the surplus ETH. A front-running bot (or any other caller) can monitor the router's ETH balance and call `refundETH()` in the same block to steal it. The loss is bounded only by how much excess ETH the victim sent; there is no protocol-level cap. This constitutes a direct loss of user principal with no recovery path once the ETH is claimed by a third party.

---

### Likelihood Explanation

Users routinely send a round-number `msg.value` to cover an uncertain exact-input amount (e.g., sending `1 ETH` for a swap that costs `0.97 ETH`). The `exactOutputSingle` path is especially prone: the user knows only their `amountInMaximum` at call time, not the actual `amountIn`, so over-sending native ETH is a natural usage pattern. MEV bots already monitor router balances on mainnet for exactly this class of stranded-value opportunity.

---

### Recommendation

1. **Auto-refund in each payable entry-point:** After the swap completes, call `_transferETH(msg.sender, address(this).balance)` (guarded by `> 0`) inside `exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput`.
2. **Alternatively, enforce exact ETH:** Revert if `msg.value` exceeds the amount actually consumed by `pay()`, mirroring the original bug's recommended fix of requiring `msg.value == price * amount`.

---

### Proof of Concept

1. Pool has WETH/Token1. `exactInputSingle` price for `amountIn = 1 ETH` of WETH yields some Token1 output.
2. User calls `router.exactInputSingle{value: 1.5 ether}(params)` where `params.amountIn = 1 ether`, `params.tokenIn = WETH`.
3. Inside `_justPayCallback`, `pay(WETH, user, pool, 1 ether)` is called. `nativeBalance = 1.5 ETH >= value = 1 ETH`, so only `1 ETH` is wrapped and forwarded; `0.5 ETH` remains on the router.
4. The swap completes successfully; the user receives their Token1 output.
5. Attacker calls `router.refundETH()` in the same or next block and receives `0.5 ETH`.
6. User has lost `0.5 ETH` with no recourse. [4](#0-3) [5](#0-4) [6](#0-5)

    // If the payer is contract it means we are in the middle of a path. In the middle of a path we operate on ERC20 only.
    if (payer == address(this)) {
      IERC20(token).safeTransfer(recipient, value);
    } else if (token == WETH) {
      uint256 nativeBalance = address(this).balance;
      if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
      } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
      } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
      }
    } else {
      IERC20(token).safeTransferFrom(payer, recipient, value);
    }
  }
