### Title
Missing Minimum-Amount Enforcement in `removeLiquidity()` Enables Front-Running Token-Distribution Manipulation — (File: metric-core/contracts/MetricOmmPool.sol)

---

### Summary

`MetricOmmPool.removeLiquidity()` burns LP shares and returns however many tokens the bins currently hold for those shares, but gives the caller no way to enforce a minimum amount of either token. An attacker who observes the pending transaction can front-run it with a swap that skews the affected bins' token balances, forcing the LP to receive a different token mix than intended. No periphery wrapper adds slippage protection for the remove path (unlike the add path, which has `MetricOmmPoolLiquidityAdder` with explicit `maxAmountToken0`/`maxAmountToken1` caps).

---

### Finding Description

`MetricOmmPool.removeLiquidity()` signature:

```solidity
function removeLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,   // bin indices + shares to burn
    bytes calldata extensionData
) external nonReentrant(PoolActions.REMOVE_LIQUIDITY)
  returns (uint256 amount0Removed, uint256 amount1Removed)
``` [1](#0-0) 

The caller specifies which bins and how many shares to burn. The amounts returned are computed from the bins' **current** `token0BalanceScaled` / `token1BalanceScaled` fields (stored in `_binStates`) proportional to the caller's share of `_binTotalShares`. There is no `minAmount0` / `minAmount1` parameter and no post-execution check. [2](#0-1) 

Contrast this with the **add** path: `MetricOmmPoolLiquidityAdder` stores `maxAmountToken0` / `maxAmountToken1` in transient storage and the callback explicitly reverts if the pool requests more:

```solidity
if (amount0Delta > max0 || amount1Delta > max1) {
    revert MaxAmountExceeded(amount0Delta, amount1Delta, max0, max1);
}
``` [3](#0-2) 

No equivalent periphery contract exists for `removeLiquidity()`.

Additionally, `removeLiquidity()` carries **no** `whenNotPaused` modifier, so the pool cannot be paused to block a front-run window (unlike `swap()`, which is gated by `whenNotPaused`): [4](#0-3) 

---

### Impact Explanation

An LP who calls `removeLiquidity()` expecting approximately X token0 and Y token1 can be forced to receive X′ token0 and Y′ token1 where the ratio is arbitrarily skewed by a preceding swap. While total value at oracle prices is approximately preserved, the token-distribution shift is a real harm when:

- The LP needs a specific token to repay a debt or meet an obligation.
- A timelock or multisig controls the treasury and cannot execute a corrective reverse trade atomically.
- The attacker profits from the directional swap itself (e.g., arbitraging the pool against an external venue), making the attack economically rational even if the LP's total value is unchanged.

This matches the Medium-severity class of the AeraVaultV1 analog: the attacker can force the counterparty to accept a trade they did not consent to.

---

### Likelihood Explanation

- Any LP calling `removeLiquidity()` directly (not through a slippage-protected wrapper) is exposed.
- The attack requires only a standard mempool front-run; no special permissions or privileged roles are needed.
- MEV bots routinely monitor for large liquidity-removal transactions and can execute the sandwich profitably when the pool's spread fee is smaller than the arbitrage opportunity created by the skew.

---

### Recommendation

1. **Add `minAmount0` / `minAmount1` parameters** to `removeLiquidity()` in the pool, reverting if the computed amounts fall below the caller's floor.
2. **Alternatively, create a periphery `MetricOmmPoolLiquidityRemover`** contract (mirroring `MetricOmmPoolLiquidityAdder`) that wraps `removeLiquidity()` and checks returned amounts against caller-supplied minimums before the call returns.
3. Consider whether `removeLiquidity()` should respect `pauseLevel` (at least at level 2) so that a protocol-level pause can protect LPs during known attack windows.

---

### Proof of Concept

1. Pool bin 0 holds 1 WETH and 3 000 USDC; LP owns 50 % of bin 0 shares.
2. LP submits `removeLiquidity()` expecting ≈ 0.5 WETH + 1 500 USDC.
3. Attacker front-runs: swaps 3 000 USDC into the pool, buying all available WETH from bin 0. Bin 0 now holds 0 WETH and 6 000 USDC; cursor is at the upper end of the bin.
4. LP's `removeLiquidity()` executes. `LiquidityLib.removeLiquidity()` computes amounts proportional to current bin balances: LP receives 0 WETH and 3 000 USDC.
5. LP has the same oracle-price value but zero WETH — a token-distribution outcome they did not consent to and cannot correct if a timelock controls their treasury.
6. Attacker optionally reverses the swap in the same block, recovering their WETH at a small spread cost while having forced the LP into an unfavorable position. [1](#0-0) [5](#0-4)
