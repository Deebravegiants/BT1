### Title
`withinUnstakeLimits` Modifier Fails to Decrement `availableActiveETHWithdrawals` Budget Across Calls, Allowing Repeated Unstaking Beyond the Combined Limit - (File: contracts/LRTConverter.sol)

---

### Summary

The `withinUnstakeLimits` modifier in `LRTConverter` enforces a combined cap of `whitelistedUnstakeAllowance + availableActiveETHWithdrawals` per call, but only decrements `whitelistedUnstakeAllowance`. The `availableActiveETHWithdrawals` component — read live from `lrtWithdrawalManager.assetsCommitted(ETH_TOKEN)` — is never decremented by the modifier. Because `assetsCommitted[ETH_TOKEN]` is a state variable in a separate contract that is not modified by `unstakeStEth`, an operator can call `unstakeStEth` N times in succession, each time passing the same `availableActiveETHWithdrawals` value as budget, and queue N × X worth of stETH for Lido withdrawal when only X was permitted.

---

### Finding Description

The modifier reads the active ETH withdrawal budget from `LRTWithdrawalManager`:

```solidity
// LRTConverter.sol lines 63–75
uint256 availableActiveETHWithdrawals = _getActiveETHUserWithdrawals();

if (amountToUnstake > whitelistedUnstakeAllowance + availableActiveETHWithdrawals) {
    revert UnstakeLimitExceeded();
}

// Consume intended withdrawal limit
if (whitelistedUnstakeAllowance > 0) {
    uint256 whitelistedAmountConsumed =
        amountToUnstake > whitelistedUnstakeAllowance ? whitelistedUnstakeAllowance : amountToUnstake;
    whitelistedUnstakeAllowance -= whitelistedAmountConsumed;
}
``` [1](#0-0) 

`_getActiveETHUserWithdrawals()` is a pure read of `assetsCommitted[ETH_TOKEN]` in `LRTWithdrawalManager`:

```solidity
// LRTConverter.sol lines 266–270
function _getActiveETHUserWithdrawals() internal view returns (uint256 activeETHWithdrawals) {
    ILRTWithdrawalManager lrtWithdrawalManager =
        ILRTWithdrawalManager(lrtConfig.getContract(LRTConstants.LRT_WITHDRAW_MANAGER));
    activeETHWithdrawals = lrtWithdrawalManager.assetsCommitted(LRTConstants.ETH_TOKEN);
}
``` [2](#0-1) 

`assetsCommitted[ETH_TOKEN]` is only modified inside `LRTWithdrawalManager` — incremented by `initiateWithdrawal` and decremented by `_unlockWithdrawalRequests`: [3](#0-2) [4](#0-3) 

Neither `unstakeStEth` nor the modifier touches it. Therefore, between two consecutive calls to `unstakeStEth`, `_getActiveETHUserWithdrawals()` returns the same value both times.

**Scenario** (`whitelistedUnstakeAllowance = 0`, `assetsCommitted[ETH_TOKEN] = X`, `LRTConverter` holds `2X` stETH):

| Call | `whitelistedUnstakeAllowance` | `availableActiveETHWithdrawals` | Check `X > 0 + X` | Passes? | stETH queued |
|------|-------------------------------|----------------------------------|---------------------|---------|--------------|
| 1st `unstakeStEth(X)` | 0 | X | false | ✅ | X |
| 2nd `unstakeStEth(X)` | 0 | X (unchanged) | false | ✅ | X |

Total stETH queued for Lido withdrawal: `2X`. Permitted by the invariant: `X`. The limit is bypassed.

The `_unstakeStEth` call itself requires the contract to hold sufficient stETH (Lido's `requestWithdrawals` pulls the tokens), so the practical upper bound is the stETH balance of `LRTConverter`, not an unbounded amount. [5](#0-4) 

---

### Impact Explanation

Excess stETH is queued for Lido withdrawal beyond what is needed to cover active user ETH withdrawal requests. During the Lido withdrawal period (which can span days to weeks), the protocol earns no stETH yield on the over-unstaked portion. The protocol fails to deliver the stETH yield it would otherwise have generated for rsETH holders. No principal is lost and no user funds are frozen, placing this squarely in the **Low — contract fails to deliver promised returns, but doesn't lose value** category.

---

### Likelihood Explanation

The operator role is trusted but is explicitly constrained by `withinUnstakeLimits`. The bug is reachable through ordinary, repeated calls to `unstakeStEth` — no key compromise or governance capture is required. An operator could trigger this accidentally (e.g., retrying a call) or deliberately. Likelihood is **low-to-medium** given the trusted role, but the code path is trivially reachable.

---

### Recommendation

Track cumulative consumption of the `availableActiveETHWithdrawals` budget within the modifier using a transient or persistent counter. The simplest fix is to introduce a storage variable (e.g., `activeETHWithdrawalsConsumed`) that is incremented by the amount attributed to the active-withdrawal budget on each call, and reset when `assetsCommitted[ETH_TOKEN]` decreases (or reset by governance). Alternatively, the modifier can compute how much of `amountToUnstake` is covered by `whitelistedUnstakeAllowance` and require the remainder to be deducted from a persistent "active-withdrawal budget consumed" counter that is bounded by `assetsCommitted[ETH_TOKEN]`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity 0.8.27;

// Pseudocode unit test (Foundry-style)
function test_doubleUnstakeBypassesActiveWithdrawalLimit() public {
    // Setup: whitelistedUnstakeAllowance = 0
    // Mock LRTWithdrawalManager returns assetsCommitted[ETH_TOKEN] = 100 ether
    // LRTConverter holds 200 ether worth of stETH

    uint256 X = 100 ether;

    // First call: should consume the entire active-withdrawal budget
    lrtConverter.unstakeStEth(X); // passes: X <= 0 + X

    // Second call: modifier re-reads assetsCommitted[ETH_TOKEN] = 100 ether (unchanged)
    // Expected: revert UnstakeLimitExceeded
    // Actual: passes again — 200 ether total queued, limit was 100 ether
    lrtConverter.unstakeStEth(X); // BUG: does NOT revert

    // Assert: 2 * X stETH queued on Lido, violating the invariant
}
```

The second call should revert with `UnstakeLimitExceeded` but does not, because `assetsCommitted[ETH_TOKEN]` is never decremented by the modifier between the two calls. [6](#0-5)
