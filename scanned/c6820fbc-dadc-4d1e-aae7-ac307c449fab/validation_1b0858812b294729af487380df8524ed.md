### Title
Inconsistent Zero-Price Handling in `_updateRsETHPrice` Causes Unprotected Deposit Freeze When `pricePercentageLimit` Is Zero - (File: contracts/LRTOracle.sol)

---

### Summary

`LRTOracle._updateRsETHPrice` exhibits inconsistent behavior when `newRsETHPrice` computes to zero. When `pricePercentageLimit > 0`, a zero price triggers an auto-pause and the price is **not** stored. When `pricePercentageLimit == 0` (the default initial state), the same zero price is **stored without any protection**, causing all subsequent L1 and L2 deposits to revert via division-by-zero, freezing user funds.

---

### Finding Description

`_updateRsETHPrice` computes `newRsETHPrice` from `totalETHInProtocol / rsethSupply`. If `totalETHInProtocol` is zero (e.g., because `ChainlinkPriceOracle.getAssetPrice` returns 0 for a supported asset when Chainlink reports a zero price), `newRsETHPrice` becomes 0.

The downside-protection branch at line 270–282 reads:

```solidity
bool isPriceDecreaseOffLimit =
    pricePercentageLimit > 0 && diff > pricePercentageLimit.mulWad(highestRsethPrice);

if (isPriceDecreaseOffLimit) {
    ...
    _pause();
    return;   // ← price NOT stored
}
```

When `pricePercentageLimit > 0`, a zero `newRsETHPrice` satisfies `diff > pricePercentageLimit.mulWad(highestRsethPrice)`, so `isPriceDecreaseOffLimit = true`, the protocol auto-pauses, and `rsETHPrice` is **not** updated to 0.

When `pricePercentageLimit == 0` (the default value after `initialize`, since no value is set there), the short-circuit `pricePercentageLimit > 0` is `false`, so `isPriceDecreaseOffLimit = false`. Execution falls through to:

```solidity
rsETHPrice = newRsETHPrice;   // stores 0
```

`rsETHPrice = 0` is now persisted on-chain.

`ChainlinkPriceOracle.getAssetPrice` has no zero-price guard:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

A Chainlink feed returning 0 (a documented failure mode) propagates silently into `_getTotalEthInProtocol`, yielding `totalETHInProtocol = 0`.

The inconsistency is further visible across the pool layer: `RSETHPoolV3ExternalBridge.viewSwapAssetToPremintedRsETH` explicitly guards against a zero rate:

```solidity
if (rsETHToETHrate == 0) revert UnsupportedOracle();
```

while `viewSwapRsETHAmountAndFee` (the primary deposit path) does not:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;   // panics on zero
```

---

### Impact Explanation

Once `rsETHPrice == 0` is stored:

- **L1 deposits** (`LRTDepositPool.depositETH` / `depositAsset`) call `getRsETHAmountToMint`, which executes `(amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()` — division by zero → revert → **all L1 deposits frozen**.
- **L2 deposits** (`RSETHPoolV3.deposit`, `RSETHPoolV3ExternalBridge.deposit`, etc.) call `viewSwapRsETHAmountAndFee`, which executes `amountAfterFee * 1e18 / rsETHToETHrate` where `rsETHToETHrate = getRate() = rsETHPrice = 0` → division by zero → revert → **all L2 deposits frozen**.

The freeze persists until an admin manually calls `updateRSETHPriceAsManager` after the oracle recovers, or sets a non-zero `pricePercentageLimit`. This is a **temporary freezing of funds** (Medium impact).

---

### Likelihood Explanation

- `updateRSETHPrice()` is `public whenNotPaused` — callable by any external account.
- `pricePercentageLimit` is initialized to 0 (Solidity default) and is only set by an explicit admin call to `setPricePercentageLimit`. Any deployment window before that call is vulnerable.
- Chainlink returning 0 for a price feed is a known, documented failure mode (e.g., during circuit-breaker events or feed deprecation).
- The combination of default-zero `pricePercentageLimit` and a Chainlink zero-price event is realistic, especially shortly after deployment or after a feed is updated.

---

### Recommendation

1. **Add a zero-price guard in `_updateRsETHPrice`** before storing the new price:
   ```solidity
   if (newRsETHPrice == 0) revert InvalidPrice();
   ```
2. **Add a zero-price guard in `ChainlinkPriceOracle.getAssetPrice`**, consistent with `ChainlinkOracleForRSETHPoolCollateral` which already does `if (ethPrice <= 0) revert InvalidPrice()`.
3. **Add a zero-rate guard in `viewSwapRsETHAmountAndFee`** (both ETH and token overloads) to match the explicit check already present in `viewSwapAssetToPremintedRsETH`.
4. **Initialize `pricePercentageLimit` to a non-zero value** in `initialize` so the downside protection is active from deployment.

---

### Proof of Concept

1. Deploy protocol with default `pricePercentageLimit == 0` (no admin call to `setPricePercentageLimit` yet).
2. Chainlink feed for a supported asset (e.g., stETH) returns `0` during a feed anomaly.
3. Anyone calls `LRTOracle.updateRSETHPrice()` (public, no access control).
4. `_getTotalEthInProtocol` returns 0 (zero asset price × any balance = 0).
5. `newRsETHPrice = 0 / rsethSupply = 0`.
6. Downside-protection check: `isPriceDecreaseOffLimit = (0 > 0) && ... = false` → no pause, no early return.
7. `rsETHPrice = 0` is stored at line 313.
8. Any user calling `LRTDepositPool.depositETH` hits `getRsETHAmountToMint` → `/ lrtOracle.rsETHPrice()` → division-by-zero panic → revert.
9. Any L2 user calling `RSETHPoolV3.deposit` hits `viewSwapRsETHAmountAndFee` → `/ rsETHToETHrate` (= 0) → division-by-zero panic → revert.
