### Title
Division by Zero in `getRsETHAmountToMint()` When `rsETHPrice` Is Uninitialized - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool.getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()` without a zero guard. The `rsETHPrice` storage variable in `LRTOracle` is never set in `initialize()` and defaults to `0` until `updateRSETHPrice()` is first called. Any deposit attempted before that initialization step triggers a division-by-zero panic revert, temporarily freezing all deposits.

### Finding Description
In `LRTDepositPool.sol` at line 520:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`rsETHPrice` is a `uint256` public state variable in `LRTOracle` declared at line 28 but **never assigned** in `initialize()`:

```solidity
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    lrtConfig = ILRTConfig(lrtConfigAddr);
    emit UpdatedLRTConfig(lrtConfigAddr);
}
```

It is only set to a non-zero value inside `_updateRsETHPrice()` — either to `1 ether` (when `rsethSupply == 0`) or to `newRsETHPrice` (computed from TVL). Until `updateRSETHPrice()` is called at least once, `rsETHPrice == 0`, and the bare division at line 520 panics.

Additionally, `_updateRsETHPrice()` can write `rsETHPrice = newRsETHPrice` where `newRsETHPrice` is computed as:

```solidity
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

If `totalETHInProtocol` collapses to zero (e.g., extreme slashing) while `pricePercentageLimit == 0` (its default, since it is also never set in `initialize()`), the downside-protection branch does **not** pause and does **not** return early, so `rsETHPrice` is overwritten with `0`. Subsequent deposits then hit the same division-by-zero.

### Impact Explanation
Every call to `depositAsset()` routes through `getRsETHAmountToMint()`. A zero `rsETHPrice` causes an unconditional panic revert, blocking all deposits. This constitutes a **temporary freezing of funds** (users cannot enter the protocol).

### Likelihood Explanation
Two realistic paths exist:

1. **Deployment window**: Between contract initialization and the first `updateRSETHPrice()` call, `rsETHPrice == 0`. Any deposit in this window reverts. Likelihood is low but non-zero, especially under rushed deployments or upgrade sequences.
2. **Slashing + default config**: `pricePercentageLimit` defaults to `0` (never initialized), disabling downside protection. If protocol TVL reaches zero while rsETH supply is positive, `rsETHPrice` is set to `0` and all subsequent deposits revert until an admin manually calls `updateRSETHPrice()` again (which itself requires `whenNotPaused`). Likelihood is low but the impact window can be extended.

### Recommendation
1. Initialize `rsETHPrice` to `1 ether` inside `initialize()` in `LRTOracle.sol`.
2. Initialize `pricePercentageLimit` to a safe non-zero default in `initialize()`.
3. Add an explicit non-zero guard in `getRsETHAmountToMint()`:

```solidity
uint256 price = lrtOracle.rsETHPrice();
require(price > 0, "LRTDepositPool: rsETH price is zero");
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / price;
```

### Proof of Concept
1. Deploy `LRTOracle` and call `initialize(lrtConfigAddr)`. `rsETHPrice` is `0`.
2. Deploy `LRTDepositPool` pointing to the above oracle.
3. Any user calls `depositAsset(stETH, 1 ether)` before `updateRSETHPrice()` is invoked.
4. Internally, `getRsETHAmountToMint(stETH, 1 ether)` executes:
   ```solidity
   rsethAmountToMint = (1e18 * getAssetPrice(stETH)) / 0;  // panic: division by zero
   ```
5. Transaction reverts; no deposit is processed. All users are blocked from depositing until an operator calls `updateRSETHPrice()`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
