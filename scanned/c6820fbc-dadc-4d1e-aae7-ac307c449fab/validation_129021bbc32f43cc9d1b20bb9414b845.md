### Title
Arithmetic Underflow in `remainingDailyMintLimit()` Causes Confusing Revert Without Meaningful Error — (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPoolV3WithNativeChainBridge.sol, contracts/pools/RSETHPoolV2ExternalBridge.sol)

---

### Summary

The `remainingDailyMintLimit()` and `getNextDailyLimitResetTimestamp()` view functions across all four daily-mint-limited L2 pool contracts contain two arithmetic underflow paths that cause opaque Solidity 0.8 panic reverts instead of meaningful errors: (1) `dailyMintLimit - effectiveDailyMintAmount` underflows when the admin reduces `dailyMintLimit` below the current day's accumulated `dailyMintAmount`, and (2) the internal call to `getCurrentDay()` underflows when `block.timestamp < startTimestamp` because these functions lack the timestamp guard that protects the `limitDailyMint` modifier.

---

### Finding Description

**Path 1 — `dailyMintLimit` reduced below `dailyMintAmount`:**

`setDailyMintLimit` allows the `DEFAULT_ADMIN_ROLE` to lower `dailyMintLimit` at any time with no lower-bound check against the already-accumulated `dailyMintAmount`:

```solidity
// RSETHPoolV3.sol line 605-611
function setDailyMintLimit(uint256 _dailyMintLimit) external onlyRole(DEFAULT_ADMIN_ROLE) {
    if (_dailyMintLimit == 0) {
        revert InvalidDailyMintLimit();
    }
    dailyMintLimit = _dailyMintLimit;
    emit DailyMintLimitSet(_dailyMintLimit);
}
```

After such a reduction, `remainingDailyMintLimit()` executes:

```solidity
// RSETHPoolV3.sol lines 345-350
function remainingDailyMintLimit() external view returns (uint256) {
    uint256 effectiveDailyMintAmount = (getCurrentDay() > lastMintDay) ? 0 : dailyMintAmount;
    return dailyMintLimit - effectiveDailyMintAmount;   // ← underflows if dailyMintAmount > dailyMintLimit
}
```

When `effectiveDailyMintAmount > dailyMintLimit`, the subtraction panics with an arithmetic underflow instead of returning `0` or a descriptive error.

**Path 2 — Called before `startTimestamp`:**

`reinitialize` enforces `_startTimestamp > block.timestamp`, so there is always a deployment window where `block.timestamp < startTimestamp`. The `limitDailyMint` modifier correctly guards `getCurrentDay()` with:

```solidity
// RSETHPoolV3.sol lines 97-98
if (block.timestamp < startTimestamp) {
    revert MintBeforeStartTimestamp();
}
```

But `remainingDailyMintLimit()` and `getNextDailyLimitResetTimestamp()` call `getCurrentDay()` directly without this guard:

```solidity
// RSETHPoolV3.sol lines 339-341
function getCurrentDay() public view returns (uint256) {
    return (block.timestamp - startTimestamp) / 1 days;  // ← underflows when block.timestamp < startTimestamp
}
```

Any caller invoking `remainingDailyMintLimit()` or `getNextDailyLimitResetTimestamp()` during this window receives a raw arithmetic panic instead of `MintBeforeStartTimestamp`.

The same two-path underflow is identically reproduced in:
- `RSETHPoolV3ExternalBridge.sol` lines 554–559 and 548–550
- `RSETHPoolV3WithNativeChainBridge.sol` lines 396–401 and 390–392
- `RSETHPoolV2ExternalBridge.sol` lines 380–385 and 374–376

---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

`remainingDailyMintLimit()` and `getNextDailyLimitResetTimestamp()` are public view functions that front-ends, aggregators, and on-chain integrators rely on to determine how much capacity remains before depositing. When either underflow path is triggered, every such caller receives an opaque panic revert with no actionable information. No user funds are at risk and the deposit path itself (`limitDailyMint` modifier) is unaffected, but the protocol's observable state becomes inconsistent and unusable for any consumer of these view functions.

---

### Likelihood Explanation

**Path 1** requires the admin to call `setDailyMintLimit` with a value smaller than the current day's `dailyMintAmount`. This is a realistic operational action (e.g., emergency limit reduction mid-day) and has no on-chain guard preventing it.

**Path 2** is guaranteed to occur on every deployment: `reinitialize` mandates `_startTimestamp > block.timestamp`, so the underflow window always exists between `reinitialize` and `startTimestamp`. Any monitoring script, UI, or contract that polls `remainingDailyMintLimit()` during this window will receive a panic revert.

---

### Recommendation

1. **Guard `getCurrentDay()` calls in view functions** with the same timestamp check used in `limitDailyMint`:

```solidity
function remainingDailyMintLimit() external view returns (uint256) {
    if (block.timestamp < startTimestamp) return dailyMintLimit;
    uint256 effectiveDailyMintAmount = (getCurrentDay() > lastMintDay) ? 0 : dailyMintAmount;
    return dailyMintLimit > effectiveDailyMintAmount ? dailyMintLimit - effectiveDailyMintAmount : 0;
}
```

2. **Use safe subtraction** in `remainingDailyMintLimit()` so that a post-hoc limit reduction returns `0` instead of panicking.

3. **Apply the same fix** to `getNextDailyLimitResetTimestamp()` in all four affected contracts.

---

### Proof of Concept

```
1. Deploy RSETHPoolV3 and call reinitialize(100e18, block.timestamp + 1 days).
   → startTimestamp is now in the future.

2. Call remainingDailyMintLimit() immediately.
   → getCurrentDay() computes (block.timestamp - startTimestamp) which underflows.
   → Transaction reverts with Panic(0x11) instead of MintBeforeStartTimestamp.

3. Advance time past startTimestamp. Users deposit until dailyMintAmount = 80e18.

4. Admin calls setDailyMintLimit(50e18).
   → No check prevents dailyMintLimit < dailyMintAmount.

5. Call remainingDailyMintLimit().
   → Computes 50e18 - 80e18 → underflows.
   → Transaction reverts with Panic(0x11) instead of returning 0 or DailyMintLimitExceeded.
```
