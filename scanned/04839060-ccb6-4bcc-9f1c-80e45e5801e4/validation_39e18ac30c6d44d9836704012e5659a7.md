### Title
No Minimum Output Amount (Slippage Protection) in L2 Pool `deposit` Functions - (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPool.sol, contracts/pools/RSETHPoolNoWrapper.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol)

---

### Summary
All four L2 pool deposit entry points lack a `minAmountOut` / `minRsETHAmountExpected` guard. The rsETH amount minted is computed at execution time from a live oracle rate. If the rate moves between submission and inclusion — due to a normal oracle update, network congestion, or MEV reordering — the depositor silently receives fewer rsETH than anticipated with no on-chain protection. The L1 `LRTDepositPool` already implements this guard correctly, making the omission in the L2 pools an inconsistency with a concrete user-facing impact.

---

### Finding Description

Every L2 pool `deposit` function computes the rsETH output at execution time by calling `viewSwapRsETHAmountAndFee`, which reads `getRate()` from the live oracle:

**`RSETHPoolV3.sol` (ETH path):**
```solidity
function deposit(string memory referralId)
    external payable nonReentrant whenNotPaused
    limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    ...
    wrsETH.mint(msg.sender, rsETHAmount);   // no minAmountOut check
}
``` [1](#0-0)

The slippage check is enforced inside `_beforeDeposit`:
```solidity

    revert MinimumAmountToReceiveNotMet();
}
``` [12](#0-11) 

No equivalent guard exists in any of the four L2 pool contracts.

---

### Impact Explanation

A depositor who previews the expected rsETH output off-chain (e.g., via `viewSwapRsETHAmountAndFee`) and then submits a transaction may receive materially fewer rsETH if the oracle rate is updated before their transaction is included. The depositor's ETH (or LST) is consumed in full; the shortfall in rsETH is permanent and unrecoverable. The contract fails to deliver the return the user was promised at submission time.

**Impact:** Low — Contract fails to deliver promised returns, but the depositor does not lose absolute value (the rsETH received is still priced at the current oracle rate).

---

### Likelihood Explanation

The rsETH oracle rate is updated periodically by the protocol as rewards accrue and asset prices shift. On L2 chains with public mempools (Arbitrum, Optimism, Base, etc.), any pending deposit transaction is visible to MEV searchers. A searcher can reorder or sandwich the deposit with an oracle update transaction (if they control the oracle update timing) or simply benefit from a naturally occurring rate update that lands in the same block. Even without active MEV, a user whose transaction sits in the mempool during a routine oracle update will silently receive fewer rsETH than expected. This is a realistic, recurring condition — not a theoretical edge case.

---

### Recommendation

Add a `minRsETHAmountOut` parameter to all four L2 pool `deposit` functions (both ETH and token variants) and revert if the computed `rsETHAmount` falls below it, mirroring the pattern already used in `LRTDepositPool._beforeDeposit`:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountOut)
    external payable nonReentrant whenNotPaused
    limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountOut) revert SlippageExceeded();
    ...
}
```

Apply the same change to the token-deposit overload and to `RSETHPool`, `RSETHPoolNoWrapper`, and `RSETHPoolV3ExternalBridge`.

---

### Proof of Concept

1. Alice calls `RSETHPoolV3.deposit{value: 10 ether}("ref")` when `getRate()` returns `1.05e18` (1 rsETH = 1.05 ETH). She expects ≈ 9.52 rsETH (after a 0 % fee for simplicity).
2. Before Alice's transaction is mined, the oracle is updated to `1.10e18` (rsETH appreciated due to rewards).
3. Alice's transaction executes: `rsETHAmount = 10e18 * 1e18 / 1.10e18 ≈ 9.09 rsETH`.
4. Alice receives ≈ 0.43 rsETH less than she expected, with no revert and no recourse.
5. Because no `minRsETHAmountOut` parameter exists, Alice had no way to express her acceptable slippage on-chain.

The same scenario applies to token deposits across all four affected pool contracts.

    function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
    }
