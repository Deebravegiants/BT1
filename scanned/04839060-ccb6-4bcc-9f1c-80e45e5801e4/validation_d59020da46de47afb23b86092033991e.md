### Title
Unbounded Nested-Loop Gas Consumption in `updateRSETHPrice()` Blocks Oracle Updates and Deposits — (`contracts/LRTOracle.sol`)

---

### Summary

`updateRSETHPrice()` is a public, permissionless function. Its internal call chain traverses three nested loops — over supported assets, over node delegators, and over EigenLayer queued withdrawals — with no hard cap on the combined iteration count. As the protocol scales normally, the gas cost grows multiplicatively. Once it exceeds the block gas limit the oracle price can never be updated, permanently freezing deposits and corrupting withdrawal accounting.

---

### Finding Description

`updateRSETHPrice()` is declared `public` with only a `whenNotPaused` guard, so any address can call it. [1](#0-0) 

It delegates to `_updateRsETHPrice()`, which calls `_getTotalEthInProtocol()`: [2](#0-1) 

**Loop 1** — iterates over every entry in `supportedAssetList`. For each asset it calls `ILRTDepositPool.getTotalAssetDeposits(asset)`.

`getTotalAssetDeposits` calls `getAssetDistributionData`: [3](#0-2) 

**Loop 2** — iterates over every entry in `nodeDelegatorQueue`. For each NDC it calls `INodeDelegator.getAssetUnstaking(asset)`.

`getAssetUnstaking` in `NodeDelegator.sol` fetches **all** queued withdrawals from EigenLayer and iterates over them: [4](#0-3) 

**Loop 3** — iterates over every `IDelegationManager.Withdrawal` returned by `getQueuedWithdrawals`, and **Loop 4** — iterates over every strategy inside each withdrawal.

The combined iteration count is:

```
supportedAssets.length
  × nodeDelegatorQueue.length
    × queuedWithdrawals.length
      × withdrawal.strategies.length
```

The protocol tracks `maxUncompletedWithdrawalCount` (capped at 80) as a soft counter, but this counter only governs new `queueUnstaking` calls initiated by the protocol. It does **not** bound the number of entries that EigenLayer's `getQueuedWithdrawals` actually returns. Forced undelegations by EigenLayer operators, or any withdrawal queued before the counter was introduced, can push the real count well above 80 without violating any protocol invariant. [5](#0-4) 

The same nested-loop chain is also triggered on every user deposit, because `_beforeDeposit` → `_checkIfDepositAmountExceedesCurrentLimit` → `getTotalAssetDeposits` → `getAssetDistributionData` → `getAssetUnstaking`: [6](#0-5) 

---

### Impact Explanation

If the combined loop count grows large enough to push the transaction past the block gas limit (30 M gas on Ethereum mainnet):

1. **`updateRSETHPrice()` becomes permanently uncallable.** The stored `rsETHPrice` goes stale. Withdrawals computed against a stale price give users incorrect asset amounts — either over- or under-paying.
2. **Every `depositAsset` / `depositETH` call reverts out-of-gas.** Users cannot obtain rsETH; their ETH/LSTs are effectively locked out of the protocol until the condition is resolved.
3. **The oracle's price-deviation circuit-breaker can never fire**, removing the last line of defense against a price manipulation attack.

This maps to **Medium — Unbounded gas consumption** and **Medium — Temporary freezing of funds**.

---

### Likelihood Explanation

The protocol already acknowledges the gas sensitivity of this path (the comment at `LRTUnstakingVault.sol:151` explicitly notes "120 is the max number of uncompleted withdrawals that allows us to still perform update rsETH price"). With `maxNodeDelegatorLimit` defaulting to 10 and up to 10 supported assets, only ~8–10 queued withdrawals per NDC (each with multiple strategies) are needed to push a single `updateRSETHPrice` call past 10 M gas. EigenLayer forced-undelegation events — which are outside the protocol's control — can create exactly this condition without any malicious action by the protocol's own operators.

---

### Recommendation

1. **Cache `getAssetUnstaking` results off-chain or in a separate keeper call** rather than recomputing them inside `updateRSETHPrice` and `depositAsset`.
2. **Decouple the deposit-limit check from the full TVL computation.** Store a running `totalDeposited[asset]` counter that is incremented/decremented on deposit and withdrawal, eliminating the need to iterate over NDCs on every deposit.
3. **Add a hard cap on the number of strategies per queued withdrawal** that `getAssetUnstaking` will process, reverting or skipping beyond the cap.
4. **Bound `supportedAssetList` length** with an explicit maximum, analogous to `maxNodeDelegatorLimit`.

---

### Proof of Concept

```
// Precondition: protocol has N supported assets, M NDCs, each NDC has K EigenLayer
// queued withdrawals with S strategies each.
//
// Gas cost ≈ N × M × K × (SLOAD + external_call_overhead × S)
//
// With N=10, M=10, K=15, S=3:
//   10 × 10 × 15 × (2100 + ~5000×3) ≈ 24 M gas  →  exceeds block limit
//
// Attack:
// 1. Wait for normal protocol operation to accumulate queued EigenLayer withdrawals
//    (or observe a forced-undelegation event).
// 2. Call:
//      lrtOracle.updateRSETHPrice()   // reverts: out of gas
// 3. Attempt a deposit:
//      lrtDepositPool.depositAsset(stETH, 1 ether, 0, "")  // reverts: out of gas
// 4. rsETHPrice is now permanently stale; all deposit/withdrawal accounting is broken.
```

    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
