### Title
Cross-Chain Replay of Merkle Claims Due to Missing `chain.id` in Leaf Hash - (`contracts/utils/MerkleDistributor/MerkleDistributor.sol`, `contracts/KERNEL/KernelMerkleDistributor.sol`, `contracts/KERNEL/KernelTop100MerkleDistributor.sol`)

---

### Summary

The merkle leaf hashes in all three distributor contracts do not include `block.chainid`. If any of these distributors are deployed on multiple chains with the same merkle root — a realistic scenario for a protocol already operating across Ethereum mainnet and multiple L2s — any user with a valid claim on one chain can replay the identical proof on every other chain and drain tokens from each distributor.

---

### Finding Description

All three merkle distributor contracts compute their leaf hashes without binding them to the current chain:

**`MerkleDistributor.sol` and `KernelMerkleDistributor.sol`:**
```solidity
bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));
```

**`KernelTop100MerkleDistributor.sol`:**
```solidity
bytes32 leaf = keccak256(abi.encodePacked(user, amount));
```

Neither hash includes `block.chainid` or the contract address. A merkle proof that is valid on chain A is therefore equally valid on chain B, provided the same root is set. The LRT-rsETH protocol already operates across Ethereum mainnet and multiple L2 chains (Arbitrum, Optimism, Scroll, Linea, etc.), and deploying reward distributors on additional chains with the same root is a natural protocol expansion — not an edge case. ---

### Impact Explanation

**High — Theft of unclaimed yield.**

A user entitled to `X` tokens on chain A can submit the identical `(index, account, cumulativeAmount, merkleProof)` tuple on chain B (and every subsequent chain) and receive `X` tokens from each distributor. The distributor contracts on the replayed chains are drained of tokens they hold for other beneficiaries. The stolen tokens are KERNEL or other reward tokens held in the distributor contracts.

---

### Likelihood Explanation

**Medium.**

The protocol is already multi-chain. Deploying reward distributors on additional chains with the same merkle root is a standard operational pattern (one root computed off-chain, pushed to all chains). No admin key compromise is required — the vulnerability activates the moment the same root is set on two chains, which is the intended deployment procedure. Any user with a valid claim can exploit this immediately upon multi-chain deployment.

---

### Recommendation

Include `block.chainid` and the contract address in every leaf hash:

```solidity
// MerkleDistributor / KernelMerkleDistributor
bytes32 node = keccak256(
    abi.encodePacked(block.chainid, address(this), index, account, cumulativeAmount)
);

// KernelTop100MerkleDistributor
bytes32 leaf = keccak256(
    abi.encodePacked(block.chainid, address(this), user, amount)
);
```

Alternatively, generate separate merkle roots per chain (each root's leaves embed the target chain ID off-chain) and verify on-chain that the root was intended for the current chain.

---

### Proof of Concept

1. Protocol deploys `KernelMerkleDistributor` on Ethereum mainnet and Arbitrum, both initialised with the same `currentMerkleRoot`.
2. Alice has a valid claim: `index=5, account=Alice, cumulativeAmount=1000e18`.
3. Alice calls `claim(5, Alice, 1000e18, proof)` on mainnet → receives 1000 KERNEL.
4. Alice calls `claim(5, Alice, 1000e18, proof)` on Arbitrum with the **identical** arguments → receives another 1000 KERNEL.
5. The Arbitrum distributor's balance is reduced by 1000 KERNEL that belonged to other beneficiaries.

The `isClaimed` check only reads from the local chain's `userClaims` mapping; it has no cross-chain awareness. [4](#0-3) [5](#0-4)
