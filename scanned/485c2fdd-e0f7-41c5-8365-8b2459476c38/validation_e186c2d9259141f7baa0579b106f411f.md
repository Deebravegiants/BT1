### Title
Missing Slippage Protection in L2 Pool `deposit()` Functions - (Files: `contracts/pools/RSETHPool.sol`, `contracts/pools/RSETHPoolV2ExternalBridge.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`, `contracts/pools/RSETHPoolNoWrapper.sol`, `contracts/pools/RSETHPoolV3WithNativeChainBridge.sol`)

---

### Summary

Multiple L2 pool `deposit()` functions accept ETH or supported tokens from users and mint/transfer rsETH (or wrsETH) based on the current oracle rate, but provide **no minimum output amount parameter**. Users who preview their swap via `viewSwapRsETHAmountAndFee()` and then submit a deposit have no on-chain guarantee they will receive at least the previewed amount.

---

### Finding Description

The `deposit()` functions across all L2 pool variants compute the rsETH output using the live oracle rate at execution time:

```solidity
// RSETHPool.sol (same pattern in all variants)
function deposit(string memory referralId) external payable nonReentrant whenNotPaused {
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();

    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
```

The rate is fetched from a trusted oracle:

```solidity
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
```

There is no `minRsETHAmount` parameter in any of these `deposit()` overloads. By contrast, `LRTDepositPool.depositETH()` and `depositAsset()` on L1 correctly accept and enforce a `minRSETHAmountExpected` parameter via `_beforeDeposit()`.

The L2 pool variants affected:
- `RSETHPool.deposit(string)` and `deposit(address, uint256, string)`
- `RSETHPoolV2ExternalBridge.deposit(string)`
- `RSETHPoolV3ExternalBridge.deposit(string)` and `deposit(address, uint256, string)`
- `RSETHPoolNoWrapper.deposit(string)` and `deposit(address, uint256, string)`
- `RSETHPoolV3WithNativeChainBridge.deposit(string)` and `deposit(address, uint256, string)`

---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A user who calls `viewSwapRsETHAmountAndFee()` to preview their swap and then submits a `deposit()` transaction has no on-chain protection if the oracle rate changes between preview and execution. If the rsETH/ETH rate increases (rsETH appreciates) before the transaction is mined — for example, due to a staking-reward oracle update in the same block — the user receives fewer rsETH tokens than the preview indicated, with no ability to revert. The user's ETH is consumed and they receive a worse-than-expected rsETH amount.

---

### Likelihood Explanation

**Medium.** Oracle rate updates occur regularly as staking rewards accrue. Any oracle update that lands in the same block as a user deposit, or any block reordering scenario, can cause the user to receive fewer rsETH than previewed. This is a structural gap present across all L2 pool variants and affects every depositor.

---

### Recommendation

Add a `minRsETHAmount` parameter to all `deposit()` functions and revert if the computed rsETH output falls below it:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmount) external payable nonReentrant whenNotPaused {
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();

    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmount) revert SlippageExceeded();

    feeEarnedInETH += fee;
    IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
```

This mirrors the protection already present in `LRTDepositPool._beforeDeposit()` on L1.

---

### Proof of Concept

1. User calls `viewSwapRsETHAmountAndFee(1 ether)` and sees they will receive `X` wrsETH.
2. User submits `deposit{value: 1 ether}("ref")`.
3. Before the transaction is mined, the rsETH oracle rate updates (rsETH appreciates relative to ETH).
4. `getRate()` now returns a higher value, so `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate` yields `X' < X`.
5. User receives `X'` wrsETH — less than previewed — with no revert and no recourse.
