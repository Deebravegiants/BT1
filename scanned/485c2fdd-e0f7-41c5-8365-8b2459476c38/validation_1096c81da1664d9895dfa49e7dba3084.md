### Title
Integer Division Truncation in `notifyRewardAmount` Permanently Locks Reward Tokens - (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary
`KernelDepositPool.notifyRewardAmount` computes `rewardRate` via integer division (`receivedAmount / duration`), silently discarding the remainder. The truncated tokens are transferred into the contract but never distributed to stakers, permanently locking up to `duration - 1` wei of reward tokens on every reward notification call.

### Finding Description
In `KernelDepositPool.notifyRewardAmount`, the reward rate is set as:

```solidity
if (block.timestamp >= finishAt) {
    rewardRate = receivedAmount / duration;          // truncates remainder
} else {
    uint256 remaining = (finishAt - block.timestamp) * rewardRate;
    rewardRate = (receivedAmount + remaining) / duration;  // truncates remainder
}
``` [1](#0-0) 

Solidity integer division truncates toward zero. The remainder `receivedAmount % duration` (up to `duration - 1` wei) is silently discarded. The full `receivedAmount` was already transferred into the contract at line 574, but only `rewardRate * duration` tokens will ever be distributed. The difference — `receivedAmount % duration` — is permanently locked in the contract with no recovery mechanism. [2](#0-1) 

The `rewardPerToken()` and `earned()` functions accumulate rewards strictly from `rewardRate`, so the truncated remainder is never reachable by any staker. [3](#0-2) 

In the mid-period rollover branch (`else`), the `remaining` value is itself derived from the already-truncated `rewardRate`, compounding the precision loss across successive `notifyRewardAmount` calls. [4](#0-3) 

### Impact Explanation
**Medium — Permanent freezing of unclaimed yield.**

Every call to `notifyRewardAmount` permanently locks up to `duration - 1` wei of reward tokens. With a typical 7-day duration (604,800 seconds), up to 604,799 wei is lost per call. For reward tokens with low decimals (e.g., 6-decimal USDC), this is up to ~0.6 tokens per call. Across repeated reward periods, the locked amount accumulates indefinitely with no admin recovery path. Stakers collectively receive less yield than was deposited into the contract.

### Likelihood Explanation
**High.** The truncation occurs on every `notifyRewardAmount` call unless `receivedAmount` is an exact multiple of `duration` — a condition that is practically never guaranteed. The function is called as part of normal protocol operation each reward period, making this a near-certain, recurring loss.

### Recommendation
Track and redistribute the truncated remainder. One approach is to add the remainder back into the next reward period:

```solidity
if (block.timestamp >= finishAt) {
    rewardRate = receivedAmount / duration;
    // carry forward the dust to the next period via a stored accumulator, or
    // simply accept the dust is bounded and document it
} else {
    uint256 remaining = (finishAt - block.timestamp) * rewardRate;
    rewardRate = (receivedAmount + remaining) / duration;
}
```

A cleaner fix is to scale `rewardRate` by a precision factor (e.g., `1e18`) so that sub-wei amounts are not lost, and adjust `rewardPerToken` and `earned` accordingly. Alternatively, compute the undistributed dust as `receivedAmount - rewardRate * duration` and carry it forward as an implicit addition to the next `notifyRewardAmount` call.

### Proof of Concept

```solidity
// Scenario: duration = 7 days = 604800 seconds
// Admin calls notifyRewardAmount with 1_000_001 wei of reward token

uint256 duration = 604_800;
uint256 receivedAmount = 1_000_001;

uint256 rewardRate = receivedAmount / duration; // = 1 (truncated)
uint256 totalDistributed = rewardRate * duration; // = 604_800
uint256 permanentlyLocked = receivedAmount - totalDistributed; // = 395_201 wei

// 395_201 wei is transferred into the contract but can never be claimed by any staker.
// Over 52 weekly reward periods: 52 * 395_201 = ~20.5M wei permanently locked.
```

The entry path is: admin calls `notifyRewardAmount` (normal operation) → `rewardRate = receivedAmount / duration` truncates → `rewardRate * duration < receivedAmount` → the difference is permanently locked in `KernelDepositPool` with no sweep or recovery function available to any party. [5](#0-4)

    function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
        if (_amount == 0) revert AmountZero();

        // Prevent starting a reward period when no tokens are staked to avoid unallocated rewards
        if (totalKernelStaked == 0) revert NoStakedTokens();

        // Transfer reward tokens into the contract
        uint256 balanceBefore = rewardsToken.balanceOf(address(this));
        rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
        uint256 balanceAfter = rewardsToken.balanceOf(address(this));
        // Calculate the actual amount of tokens received in case of a transfer fee (tax)
        uint256 receivedAmount = balanceAfter - balanceBefore;

        if (block.timestamp >= finishAt) {
            rewardRate = receivedAmount / duration;
        } else {
            uint256 remaining = (finishAt - block.timestamp) * rewardRate;
            rewardRate = (receivedAmount + remaining) / duration;
        }

        if (rewardRate == 0) revert RewardRateZero();

        finishAt = block.timestamp + duration;
        updatedAt = block.timestamp;

        emit NotifyRewardAmount(receivedAmount, finishAt);
    }
