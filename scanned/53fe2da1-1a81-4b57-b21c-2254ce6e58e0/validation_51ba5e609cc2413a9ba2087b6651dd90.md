### Title
`stakedButUnverifiedNativeETH` Not Adjusted for Beacon-Chain Slashing Causes rsETH Price Inflation and Protocol Pause - (File: contracts/NodeDelegator.sol)

### Summary

`NodeDelegator` tracks ETH staked to beacon-chain validators in a local counter `stakedButUnverifiedNativeETH` that is always incremented by exactly 32 ETH per validator and decremented by exactly 32 ETH at credential-verification time. It is never adjusted for beacon-chain slashing. If a validator is slashed before its withdrawal credentials are verified with EigenLayer, the protocol's total-ETH accounting overstates the real backing, inflating the rsETH price during the unverified window. When credentials are eventually verified, the price corrects downward; if the drop exceeds `pricePercentageLimit`, the oracle's downside-protection logic automatically pauses both `LRTDepositPool` and `LRTWithdrawalManager`, temporarily freezing all user funds.

### Finding Description

**Step 1 – Staking inflates the counter unconditionally.**

`stake32Eth` always adds exactly 32 ETH to `stakedButUnverifiedNativeETH`, regardless of any future beacon-chain balance change: [1](#0-0) 

**Step 2 – `getEffectivePodShares` sums the stale counter with EigenLayer's live withdrawable shares.** [2](#0-1) 

`withdrawableShare` is fetched from EigenLayer's `DelegationManager.getWithdrawableShares`, which correctly reflects post-slash balances. But `stakedButUnverifiedNativeETH` is a raw Solidity storage variable with no on-chain link to the beacon chain; it cannot self-correct when a validator is slashed.

**Step 3 – The inflated value propagates to the oracle's TVL calculation.**

`getETHDistributionData` in `LRTDepositPool` calls `getEffectivePodShares()` for every NDC: [3](#0-2) 

`_getTotalEthInProtocol` in `LRTOracle` calls `getTotalAssetDeposits`, which calls `getETHDistributionData`: [4](#0-3) 

**Step 4 – rsETH price is computed from the inflated TVL.** [5](#0-4) 

During the unverified window the price is overstated by `slashed_amount / rsethSupply`, so new depositors receive fewer rsETH than they are entitled to.

**Step 5 – Credential verification corrects the counter, causing a sudden price drop.**

`verifyWithdrawalCredentials` always subtracts the full 32 ETH per validator: [6](#0-5) 

After this call `stakedButUnverifiedNativeETH` falls by 32 ETH while `withdrawableShare` rises by only `(32 - slashed) ETH`, producing a net decrease in `getEffectivePodShares` equal to the slashed amount.

**Step 6 – If the price drop exceeds `pricePercentageLimit`, the oracle auto-pauses the protocol.** [7](#0-6) 

Both `LRTDepositPool` and `LRTWithdrawalManager` are paused, freezing all deposits and withdrawals until an admin manually unpauses.

### Impact Explanation

**Medium – Temporary freezing of funds.**

All user deposits and withdrawals are blocked for an indefinite period after the oracle auto-pause. The pause is triggered automatically by a public call to `updateRSETHPrice()` and requires an admin (`onlyLRTAdmin`) to call `unpause()` on each contract before normal operation resumes. During the unverified window, new depositors also silently receive fewer rsETH than they are owed (Low: contract fails to deliver promised returns).

### Likelihood Explanation

Beacon-chain slashing is a known, documented risk for ETH validators. The Kelp protocol stakes native ETH through `NodeDelegator` validators. The vulnerable window exists between `stake32Eth` and `verifyWithdrawalCredentials`, which can span days to weeks in practice. A single slashed validator among the protocol's validator set is sufficient to trigger the pause path once credentials are verified.

### Recommendation

1. **Track actual beacon-chain balance at verification time.** When `verifyWithdrawalCredentials` is called, read the actual verified balance from the EigenPod proof data and subtract that value (not a hardcoded 32 ETH) from `stakedButUnverifiedNativeETH`. Any shortfall should be recorded separately so the oracle can reflect the real TVL immediately.

2. **Decouple slashing-induced price drops from the auto-pause.** Consider distinguishing between organic price decreases (slashing) and manipulation-driven ones before triggering a full protocol pause, or widen `pricePercentageLimit` to accommodate expected slashing magnitudes.

3. **Add a slashing-reserve mechanism.** Analogous to the frxETH report's recommendation, maintain a reserve fund that can absorb small slashing events without causing a price drop that triggers the pause.

### Proof of Concept

1. Protocol stakes 32 ETH via `stake32Eth` → `stakedButUnverifiedNativeETH = 32 ETH`.
2. Beacon-chain slashes the validator by 1 ETH; actual balance = 31 ETH.
3. `getEffectivePodShares()` = 32 ETH (overstated). rsETH price is inflated. New depositors receive fewer rsETH.
4. Operator calls `verifyWithdrawalCredentials` → `stakedButUnverifiedNativeETH -= 32 ETH`; EigenLayer records 31 ETH as withdrawable shares.
5. `getEffectivePodShares()` drops by 1 ETH. `_getTotalEthInProtocol()` drops by 1 ETH.
6. Anyone calls `updateRSETHPrice()`. `newRsETHPrice < highestRsethPrice`. If `diff > pricePercentageLimit * highestRsethPrice`, the oracle calls `lrtDepositPool.pause()`, `withdrawalManager.pause()`, and `_pause()`.
7. All user deposits and withdrawals are frozen until admin intervention.
