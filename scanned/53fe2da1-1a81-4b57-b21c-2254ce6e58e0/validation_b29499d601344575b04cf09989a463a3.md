### Title
Stale Chainlink Price Accepted in `getAssetPrice` — No `updatedAt` Staleness Check - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `price`. The `updatedAt` timestamp is never validated, so a stale Chainlink answer is silently accepted and propagated into rsETH minting, withdrawal sizing, and TVL accounting.

### Finding Description
In `ChainlinkPriceOracle.getAssetPrice()`, the call to `latestRoundData()` uses a wildcard destructuring that ignores `roundId`, `startedAt`, `updatedAt`, and `answeredInRound`:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

No check of the form `require(block.timestamp - updatedAt <= MAX_DELAY)` is present. Chainlink feeds can lag — during network congestion, sequencer downtime (on L2), or a feed-specific incident, the last stored answer can be hours or days old while `latestRoundData()` continues to return it without reverting.

This price is consumed by:
- `LRTOracle._getTotalEthInProtocol()` → `LRTOracle.getAssetPrice(asset)` → used to compute `rsETHPrice` on every `updateRSETHPrice()` call.
- `LRTDepositPool.getRsETHAmountToMint()` → `lrtOracle.getAssetPrice(asset)` → determines how many rsETH tokens a depositor receives.
- `LRTWithdrawalManager.getExpectedAssetAmount()` → `lrtOracle.getAssetPrice(asset)` → determines how many LST tokens a withdrawer receives.

`ChainlinkOracleForRSETHPoolCollateral` does include staleness guards (`answeredInRound < roundID`, `timestamp == 0`), but `ChainlinkPriceOracle` — the oracle used for core deposit/withdrawal pricing — has none.

### Impact Explanation
**Medium — Temporary freezing of funds / contract fails to deliver promised returns.**

If a supported LST's Chainlink feed goes stale (e.g., ETHx/ETH, stETH/ETH):

1. **Over-minting rsETH on deposit**: If the stale price is inflated relative to the true market price, depositors receive more rsETH than they are entitled to, diluting existing holders (theft of unclaimed yield / share mis-accounting).
2. **Under-paying withdrawers**: If the stale price is deflated, `getExpectedAssetAmount` returns fewer LST tokens than the user's rsETH is worth, causing the contract to fail to deliver promised returns.
3. **Incorrect rsETH price update**: `_updateRsETHPrice()` computes `totalETHInProtocol` using stale per-asset prices, producing a wrong `rsETHPrice` that cascades into all subsequent deposits and withdrawals until corrected.
4. **Incorrect price-deviation guard**: The `pricePercentageLimit` circuit-breaker in `LRTOracle` compares the new computed price against `highestRsethPrice`. A stale input can either suppress a legitimate pause trigger or fire a false pause, temporarily freezing the protocol.

### Likelihood Explanation
Chainlink feeds have historically gone stale during periods of high network congestion and during L2 sequencer outages. The affected assets (ETHx, stETH, etc.) each have their own feed with independent heartbeat windows (typically 1–24 hours). Any single feed stalling within its heartbeat window is sufficient to trigger the issue. No privileged access is required — any user calling `depositAsset`, `initiateWithdrawal`, or `instantWithdrawal` will interact with the stale price.

### Recommendation
Add a staleness check in `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
uint256 public constant MAX_PRICE_AGE = 1 hours; // tune per feed

function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,, uint256 updatedAt,) = priceFeed.latestRoundData();
    require(block.timestamp - updatedAt <= MAX_PRICE_AGE, "Stale price");
    require(price > 0, "Invalid price");
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Consider making `MAX_PRICE_AGE` configurable per asset, since different Chainlink feeds have different heartbeat intervals.

### Proof of Concept

**Root cause** — `updatedAt` is silently discarded: [1](#0-0) 

**Propagation path 1 — deposit minting uses this price:** [2](#0-1) 

**Propagation path 2 — withdrawal sizing uses this price:** [3](#0-2) 

**Propagation path 3 — rsETH price update uses this price for all supported assets:** [4](#0-3) 

**Attack scenario:**
1. Chainlink ETHx/ETH feed goes stale (e.g., last updated 4 hours ago, price was 1.05 ETH/ETHx).
2. True market price drops to 0.95 ETH/ETHx due to a depeg event.
3. An attacker calls `depositAsset(ETHx, largeAmount, 0, "")` — `getRsETHAmountToMint` uses the stale 1.05 price, minting ~10.5% more rsETH than deserved.
4. Attacker immediately calls `initiateWithdrawal(stETH, rsETHAmount, "")` — `getExpectedAssetAmount` uses the (correct) stETH price, receiving full stETH value.
5. Net result: attacker extracts value from existing rsETH holders at the expense of the protocol's TVL integrity.
