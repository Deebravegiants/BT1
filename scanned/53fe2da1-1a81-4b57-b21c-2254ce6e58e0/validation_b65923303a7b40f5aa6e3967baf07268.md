### Title
Single-Step Ownership Transfer Enables Permanent Freezing of Unclaimed KERNEL Yield - (File: contracts/KERNEL/KernelMerkleDistributor.sol)

### Summary
`KernelMerkleDistributor` (and sibling distributor contracts) inherit `OwnableUpgradeable`, which implements a single-step `transferOwnership()`. If the current owner accidentally transfers ownership to an incorrect address, all owner-gated functions become permanently inaccessible with no recovery path, freezing KERNEL tokens held in the contract and halting future reward distribution.

### Finding Description
`KernelMerkleDistributor` inherits `OwnableUpgradeable` directly: [1](#0-0) 

`OwnableUpgradeable.__Ownable_init()` is called during initialization: [2](#0-1) 

OpenZeppelin's `OwnableUpgradeable.transferOwnership()` immediately overwrites `_owner` with the new address in a single transaction — there is no pending-owner confirmation step and no way to cancel or reverse the transfer. The contract exposes no `withdrawTokens` or emergency-recovery function; the only way to move KERNEL tokens out is through the `claim`/`claimAndStake` user paths, which require a valid `currentMerkleRoot` set exclusively by the owner: [3](#0-2) 

The same pattern exists in three additional production contracts:
- `contracts/utils/MerkleDistributor/MerkleDistributor.sol` — holds ERC-20 reward tokens, no `withdrawTokens` [4](#0-3) 
- `contracts/KERNEL/KernelTop100MerkleDistributor.sol` — holds KERNEL tokens; `withdrawTokens` is owner-only, so loss of ownership freezes those funds too [5](#0-4) 
- `contracts/utils/MerkleDistributor/MerkleBlastPointsDistributor.sol` — owner controls the Blast points operator configuration [6](#0-5) 

By contrast, `WrappedRSETH` correctly uses the two-step `ConfirmedOwnerWithProposal`, which requires the new owner to call `acceptOwnership()` before the transfer completes: [7](#0-6) 

The distributor contracts do not apply this safer pattern.

### Impact Explanation
If the owner sends `transferOwnership(wrongAddress)`, the following become permanently inaccessible in `KernelMerkleDistributor`:
- `setMerkleRoot` — no new reward epochs can be published; all future KERNEL yield is frozen for every eligible user.
- `pause`/`unpause` — the contract cannot be halted in an emergency.
- `setKernelDepositPool`, `setProtocolTreasury`, `setFeeInBPS` — protocol configuration is locked.

Because there is no `withdrawTokens` function, any KERNEL tokens already deposited but not yet claimed are permanently locked in the contract. This matches **Medium — Permanent freezing of unclaimed yield**.

### Likelihood Explanation
Ownership transfer is a rare but high-stakes operation. A single typo, clipboard-hijack, or address-confusion error (e.g., confusing a testnet address with a mainnet address) is sufficient. The risk is realistic for any protocol that rotates multisig signers or migrates to a new governance address. The original report was acknowledged at Medium severity for exactly this scenario.

### Recommendation
Replace `OwnableUpgradeable` with `Ownable2StepUpgradeable` across all four distributor contracts. The two-step pattern requires the nominated address to call `acceptOwnership()`, making accidental transfers recoverable before they take effect.

Reference: https://github.com/OpenZeppelin/openzeppelin-contracts-upgradeable/blob/master/contracts/access/Ownable2StepUpgradeable.sol

### Proof of Concept
1. Deploy `KernelMerkleDistributor` and fund it with KERNEL tokens for a reward epoch.
2. Owner calls `transferOwnership(0xDeAdBeEf...)` — a mistyped address.
3. `OwnableUpgradeable` immediately sets `_owner = 0xDeAdBeEf...` with no confirmation required.
4. The real owner attempts `setMerkleRoot(newRoot)` → reverts with `"Ownable: caller is not the owner"`.
5. No `withdrawTokens` exists; users with valid proofs for the next epoch can never claim.
6. All KERNEL tokens remaining in the contract are permanently frozen.
