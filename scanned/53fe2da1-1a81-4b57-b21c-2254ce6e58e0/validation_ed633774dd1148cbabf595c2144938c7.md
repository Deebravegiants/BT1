### Title
rsETH Minted to `msg.sender` Instead of Intended Recipient in `LRTDepositPool.depositETH` / `depositAsset` — (File: `contracts/LRTDepositPool.sol`)

---

### Summary

`LRTDepositPool.depositETH` and `depositAsset` always mint rsETH to `msg.sender` via the private `_mintRsETH` helper. No `receiver` parameter exists. When any smart-contract intermediary (aggregator, router, multisig, wrapper) calls these functions on behalf of a user, rsETH is minted to the intermediary contract rather than to the actual depositor. The user's ETH or LST is consumed but the rsETH is permanently stranded in the intermediary unless that contract has explicit forwarding logic.

---

### Finding Description

`_mintRsETH` unconditionally mints to `msg.sender`:

```solidity
// contracts/LRTDepositPool.sol  L686-690
function _mintRsETH(uint256 rsethAmountToMint) private {
    address rsethToken = lrtConfig.rsETH();
    IRSETH(rsethToken).mint(msg.sender, rsethAmountToMint);
}
```

Both public deposit entry-points delegate to this helper:

- `depositETH` (L76-93) — user sends ETH, rsETH minted to `msg.sender`
- `depositAsset` (L99-118) — user's LST pulled from `msg.sender`, rsETH minted to `msg.sender`

Neither function accepts a `receiver` address. The protocol itself relies on this pattern in `L1Vault` / `L1VaultV2`, which call `depositETH` / `depositAsset` and intentionally collect rsETH at the vault level before bridging. However, any third-party smart contract that integrates with `LRTDepositPool` on behalf of end-users will silently receive the rsETH instead of the user.

---

### Impact Explanation

**Critical — direct loss of user funds.**

A user who deposits ETH or an LST through a smart-contract intermediary (e.g., a DeFi aggregator, a router, a Gnosis Safe, or any wrapper contract) will have their deposit consumed while rsETH is minted to the intermediary. Unless the intermediary has explicit logic to forward rsETH to the user (which generic aggregators do not), the user permanently loses the value of their deposit. There is no recovery path inside `LRTDepositPool`.

---

### Likelihood Explanation

**Medium.** The pattern is triggered by any smart-contract caller of `depositETH` or `depositAsset`. Concrete realistic paths:

1. A DeFi aggregator (1inch, Paraswap, etc.) routes a user's ETH through `depositETH` to obtain rsETH — rsETH lands in the aggregator.
2. A Gnosis Safe or other smart-contract wallet calls `depositAsset` on behalf of a user — rsETH lands in the Safe, not in the user's EOA.
3. A wrapper/pool contract (e.g., a yield optimizer) integrates with `LRTDepositPool` — rsETH is trapped in the wrapper.

The protocol already demonstrates this pattern is reachable: `L1Vault` and `L1VaultV2` call `depositETH` / `depositAsset` and receive rsETH at the contract level. Any external integrator that does not replicate the vault's bridging logic will silently lose user funds.

---

### Recommendation

Add an explicit `receiver` parameter to both `depositETH` and `depositAsset`, and pass it through to `_mintRsETH`:

```solidity
function depositETH(
    uint256 minRSETHAmountExpected,
    string calldata referralId,
    address receiver          // NEW
) external payable ... {
    uint256 rsethAmountToMint = _beforeDeposit(...);
    _mintRsETH(rsethAmountToMint, receiver);   // pass receiver
    emit ETHDeposit(msg.sender, msg.value, rsethAmountToMint, referralId);
}

function _mintRsETH(uint256 rsethAmountToMint, address receiver) private {
    IRSETH(lrtConfig.rsETH()).mint(receiver, rsethAmountToMint);
}
```

For backwards compatibility, a no-`receiver` overload that defaults to `msg.sender` can be retained. Existing callers such as `L1Vault` / `L1VaultV2` should be updated to pass `address(this)` explicitly.

---

### Proof of Concept

1. Deploy a minimal router contract:
   ```solidity
   contract Router {
       ILRTDepositPool pool;
       function depositFor(address user) external payable {
           pool.depositETH{value: msg.value}(0, "");
           // rsETH is now in Router, not in `user`
       }
   }
   ```
2. User calls `Router.depositFor{value: 1 ether}(userAddress)`.
3. `LRTDepositPool._mintRsETH` mints rsETH to `address(Router)` (the `msg.sender`).
4. `user` receives zero rsETH; their 1 ETH is permanently locked in the protocol with no claim.
