### Title
`LRTOracle.rsETHPrice` Freezes When Daily Fee Mint Cap Is Reached, Causing `RSETHRateProvider` to Propagate a Stale Rate Cross-Chain — (`contracts/LRTOracle.sol`, `contracts/cross-chain/RSETHRateProvider.sol`)

---

### Summary

When `maxFeeMintAmountPerDay` is set and the daily fee-mint cap is reached, every public call to `updateRSETHPrice()` reverts before writing the new price to storage. `RSETHRateProvider.getLatestRate()` reads the stale `rsETHPrice` directly, so `CrossChainRateProvider.updateRate()` propagates an outdated rate to destination chains until the next 24-hour window opens.

---

### Finding Description

**Execution path — `updateRSETHPrice()` reverts before storing the new price**

`LRTOracle.updateRSETHPrice()` is public with no role guard (only `whenNotPaused`): [1](#0-0) 

Inside `_updateRsETHPrice()`, the fee-mint limit check is performed **before** the price is committed to storage: [2](#0-1) 

`_checkAndUpdateDailyFeeMintLimit` reverts unconditionally when the cap is exceeded: [3](#0-2) 

Because the revert unwinds the entire call, `rsETHPrice = newRsETHPrice` on line 313 is never reached. The stored `rsETHPrice` remains at its last successfully written value.

**`RSETHRateProvider.getLatestRate()` reads the stale storage slot**

`getLatestRate()` is a pure storage read — it does not trigger a price update: [4](#0-3) 

**`CrossChainRateProvider.updateRate()` sends whatever `getLatestRate()` returns**

`updateRate()` is public and payable with no access control: [5](#0-4) 

Any caller can invoke `updateRate()` at any time, and it will faithfully relay the stale `rsETHPrice` to the destination chain.

**Note on the question's premise:** The question states that `updateRate()` itself triggers `updateRSETHPrice()`. That is incorrect — `updateRate()` only reads the already-stored value. The revert occurs in a separate, independent call to `updateRSETHPrice()`. The underlying vulnerability (stale price propagated cross-chain) is nonetheless real.

**Edge case — `maxFeeMintAmountPerDay == 0` (default)**

The default value of `maxFeeMintAmountPerDay` is `0`. `_checkAndUpdateDailyFeeMintLimit` has no "disabled" guard for this case: [6](#0-5) 

With `maxFeeMintAmountPerDay == 0`, the condition `currentPeriodMintedFeeAmount + feeAmount > 0` is true for any `feeAmount > 0`, so `updateRSETHPrice()` reverts on every call where rewards have accrued and a protocol fee is configured — even before the cap is ever "reached."

---

### Impact Explanation

Cross-chain consumers of `RSETHRateProvider` (e.g., Balancer pools, lending markets, bridges on destination chains) receive a rate that lags behind the true protocol state. Accrued yield is not reflected in the cross-chain rate until the next 24-hour window resets `currentPeriodMintedFeeAmount`. This matches the scoped impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

---

### Likelihood Explanation

- `updateRSETHPrice()` is public; any account can call it.
- `maxFeeMintAmountPerDay` is a live, manager-settable parameter; setting it to a small value (or leaving it at the default `0` while a protocol fee is active) immediately triggers the condition.
- No privileged compromise is required; the condition arises from normal protocol operation once the cap is configured.

---

### Recommendation

Add a short-circuit in `_checkAndUpdateDailyFeeMintLimit` (or in `_updateRsETHPrice`) so that a zero `maxFeeMintAmountPerDay` is treated as "no limit", and restructure `_updateRsETHPrice` to **write `rsETHPrice` before** attempting fee minting, or to skip fee minting (rather than reverting) when the daily cap is exhausted:

```solidity
// Option A: treat 0 as "no limit"
if (maxFeeMintAmountPerDay == 0) return; // no cap configured

// Option B: cap fee, don't revert — write price regardless
if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
    feeAmount = maxFeeMintAmountPerDay - currentPeriodMintedFeeAmount;
}
```

Decoupling the price-write from the fee-mint ensures `rsETHPrice` always reflects the current TVL even when the daily fee budget is exhausted.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Pseudocode for a local/fork test
function test_stalePricePropagatedCrossChain() public {
    // 1. Admin sets a tiny daily fee mint cap
    vm.prank(manager);
    lrtOracle.setMaxFeeMintAmountPerDay(1); // 1 wei cap

    // 2. Simulate reward accrual so protocolFeeInETH > 0
    //    (increase underlying asset value in EigenLayer mock)
    mockEigenLayerIncreaseTVL(1 ether);

    // 3. First updateRSETHPrice call hits the cap and reverts
    vm.expectRevert(ILRTOracle.DailyFeeMintLimitExceeded.selector);
    lrtOracle.updateRSETHPrice();

    // 4. rsETHPrice is unchanged (stale)
    uint256 stalePrice = lrtOracle.rsETHPrice();

    // 5. Compute what the true price should be
    uint256 truePrice = computeExpectedPrice(); // TVL / supply

    assert(stalePrice < truePrice); // stale price < actual price

    // 6. updateRate() propagates the stale price cross-chain
    rsethRateProvider.updateRate{value: 0.01 ether}();
    assertEq(rsethRateProvider.rate(), stalePrice); // not truePrice
}
```

        // Reset the period if it's unset or a day has passed
        if (block.timestamp >= feePeriodStartTime + 1 days) {
            currentPeriodMintedFeeAmount = 0;
            feePeriodStartTime = getCurrentPeriodStartTime();
        }

        // Check if minting would exceed the daily limit
        if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
            revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
        }

        currentPeriodMintedFeeAmount += feeAmount;
    }
