### Title
Missing Before-After Balance Check on Token Deposits Enables rsETH Over-Minting with Fee-on-Transfer Tokens - (File: contracts/LRTDepositPool.sol, contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPool.sol, contracts/pools/RSETHPoolNoWrapper.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPoolV3WithNativeChainBridge.sol)

---

### Summary

Every token-deposit path in the protocol computes the rsETH/wrsETH amount to mint from the caller-supplied `amount` parameter **before** verifying how many tokens were actually received. If a fee-on-transfer token is ever added as a supported asset, the protocol mints more rsETH/wrsETH than the value it actually holds, directly causing protocol insolvency for all existing holders.

---

### Finding Description

The vulnerability class from the reference report is **missing before-after balance checks on token transfers**: a trusted external entity (router/token) can cause the protocol to credit more value than it actually received, because the protocol never compares its balance before and after the transfer.

The exact same root cause exists across the LRT-rsETH deposit paths.

**`LRTDepositPool.depositAsset()`** (contracts/LRTDepositPool.sol L99–L118):

```solidity
function depositAsset(address asset, uint256 depositAmount, ...) external {
    // rsethAmountToMint is computed from the CALLER-SUPPLIED depositAmount
    uint256 rsethAmountToMint = _beforeDeposit(asset, depositAmount, minRSETHAmountExpected);

    // Actual transfer — if token has a fee, contract receives < depositAmount
    IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);

    // Minting uses the pre-computed amount, NOT the actual received amount
    _mintRsETH(rsethAmountToMint);
}
```

`_beforeDeposit` calls `getRsETHAmountToMint(asset, depositAmount)`, which prices the mint entirely from `depositAmount`. No balance snapshot is taken before or after the `safeTransferFrom`. [1](#0-0) 

The same pattern is repeated verbatim in every L2 pool contract:

- **`RSETHPool.deposit(token, amount, referralId)`** — `safeTransferFrom` then `viewSwapRsETHAmountAndFee(amount, token)` then `safeTransfer(wrsETH)`. [2](#0-1) 
- **`RSETHPoolNoWrapper.deposit(token, amount, referralId)`** — identical pattern. [3](#0-2) 
- **`RSETHPoolV3.deposit(token, amount, referralId)`** — `safeTransferFrom` then `viewSwapRsETHAmountAndFee(amount, token)` then `wrsETH.mint`. [4](#0-3) 
- **`RSETHPoolV3ExternalBridge.deposit(token, amount, referralId)`** — same. [5](#0-4) 
- **`RSETHPoolV3WithNativeChainBridge.deposit(token, amount, referralId)`** — same. [6](#0-5) 

In every case, `viewSwapRsETHAmountAndFee(amount, token)` computes the rsETH output from the caller-supplied `amount`, not from the actual balance delta. [7](#0-6) 

Contrast this with `KernelDepositPool.notifyRewardAmount()`, which **does** perform a before-after balance check: [8](#0-7) 

```solidity
uint256 balanceBefore = rewardsToken.balanceOf(address(this));
rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
uint256 balanceAfter = rewardsToken.balanceOf(address(this));
uint256 receivedAmount = balanceAfter - balanceBefore;
```

The deposit paths lack this protection entirely.

---

### Impact Explanation

If any fee-on-transfer token is added as a supported asset (via `LRTConfig` on L1 or `addSupportedToken` on L2 pools), every depositor receives rsETH/wrsETH computed from the gross `amount` while the contract holds only `amount - fee`. The rsETH/wrsETH supply becomes unbacked by the shortfall, diluting all existing holders. Repeated deposits compound the insolvency. This constitutes **protocol insolvency** — a Critical impact per the allowed scope.

---

### Likelihood Explanation

The protocol is designed to support multiple LSTs and bridge tokens. Adding a new supported asset is a routine governance/admin action. Several real-world tokens (e.g., USDT with fees re-enabled, any rebasing token with a transfer tax, or a future LST with a protocol fee on transfer) could trigger this. The entry path requires no privilege: any unprivileged depositor calling `depositAsset()` or `deposit(token, amount, referralId)` with a fee-on-transfer token triggers the over-mint. Likelihood is **Medium** (depends on a supported fee-on-transfer token being added, which is a realistic operational scenario).

---

### Recommendation

Apply before-after balance checks in all token deposit paths, using the actual received amount for rsETH computation:

```solidity
uint256 balanceBefore = IERC20(asset).balanceOf(address(this));
IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
uint256 actualReceived = IERC20(asset).balanceOf(address(this)) - balanceBefore;

uint256 rsethAmountToMint = getRsETHAmountToMint(asset, actualReceived);
require(rsethAmountToMint >= minRSETHAmountExpected, "MinimumAmountToReceiveNotMet");
_mintRsETH(rsethAmountToMint);
```

Apply the same pattern to all pool `deposit(token, amount, referralId)` functions.

---

### Proof of Concept

1. Admin adds a fee-on-transfer token `FeeToken` (1% fee) as a supported asset in `LRTConfig`.
2. Attacker calls `LRTDepositPool.depositAsset(FeeToken, 1000e18, 0, "")`.
3. `_beforeDeposit` computes `rsethAmountToMint` based on `1000e18`.
4. `safeTransferFrom` transfers `1000e18` from attacker; contract receives `990e18` (1% fee deducted by token).
5. `_mintRsETH(rsethAmountToMint)` mints rsETH equivalent to `1000e18` of `FeeToken`.
6. The protocol is now short `10e18` FeeToken worth of backing per deposit.
7. Repeated by any depositor; rsETH becomes progressively undercollateralized, harming all existing rsETH holders.

```text
    function viewSwapRsETHAmountAndFee(
        uint256 amount,
        address token
    )
        public
        view
        onlySupportedToken(token)
        returns (uint256 rsETHAmount, uint256 fee)
    {
        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // rate of token in ETH
        uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```
