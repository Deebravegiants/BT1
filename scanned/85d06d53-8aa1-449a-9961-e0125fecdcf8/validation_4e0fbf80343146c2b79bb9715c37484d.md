### Title
Uninitialized `withdrawalDelay` and `maxNumberOfWithdrawalsPerUser` in `KernelDepositPool` Cause Withdrawal Delay Bypass and Temporary Fund Freeze - (File: contracts/KERNEL/KernelDepositPool.sol)

---

### Summary

`KernelDepositPool` declares two critical state variables — `withdrawalDelay` and `maxNumberOfWithdrawalsPerUser` — that are never assigned in the `initialize()` function, leaving both at their Solidity default of `0`. The setter functions for both variables correctly reject `0` as an invalid value, but the initial deployment state is `0`. This creates two distinct protocol failures: (1) `maxNumberOfWithdrawalsPerUser = 0` causes every call to `initiateWithdrawal()` to revert immediately, freezing all staked KERNEL tokens until admin intervention; and (2) `withdrawalDelay = 0` causes the withdrawal delay window to be zero, allowing immediate claims and bypassing the intended time-lock protection.

---

### Finding Description

In `KernelDepositPool.sol`, the `initialize()` function sets only `kernelToken`, `rewardsToken`, and `DEFAULT_ADMIN_ROLE`: [1](#0-0) 

Neither `withdrawalDelay` nor `maxNumberOfWithdrawalsPerUser` is assigned, so both remain `0`: [2](#0-1) 

**Effect 1 — Permanent revert on `initiateWithdrawal()`:**

`initiateWithdrawal()` guards against exceeding the per-user withdrawal limit: [3](#0-2) 

When `maxNumberOfWithdrawalsPerUser = 0`, the condition `userWithdrawalIds[msg.sender].length >= 0` is always `true` (since `length` is a `uint256` and is always `>= 0`), so every call reverts with `WithdrawalLimitReached()`. No user can ever initiate a withdrawal until admin calls `setMaxNumberOfWithdrawalsPerUser()`.

**Effect 2 — Zero-delay withdrawal window:**

`initiateWithdrawal()` computes the unlock time as: [4](#0-3) 

With `withdrawalDelay = 0`, `unlockTime = block.timestamp`. The `claimWithdrawal()` check `block.timestamp < withdrawal.unlockTime` is immediately false, so withdrawals are claimable in the same block they are initiated, bypassing the intended delay.

Both setter functions correctly reject `0`: [5](#0-4) [6](#0-5) 

But the initial deployment state is `0` because `initialize()` never sets them.

---

### Impact Explanation

**Primary (Medium — Temporary Freezing of Funds):** With `maxNumberOfWithdrawalsPerUser = 0`, all calls to `initiateWithdrawal()` revert unconditionally. Every user who has staked KERNEL tokens via `stake()` or `stakeFor()` is unable to initiate a withdrawal. Their staked KERNEL balance is locked in the contract until the admin calls `setMaxNumberOfWithdrawalsPerUser()` with a non-zero value. This is a temporary but complete freeze of all staked user funds.

**Secondary (Low — Contract Fails to Deliver Promised Returns):** With `withdrawalDelay = 0`, the time-lock protection is absent. Users can stake and immediately claim in the same block, defeating the protocol's stated withdrawal delay guarantee.

---

### Likelihood Explanation

The likelihood is **high** for the freeze scenario. The `initialize()` function is the sole initialization path for an upgradeable proxy. If the admin does not separately call `setMaxNumberOfWithdrawalsPerUser()` after deployment (a step that is not enforced or documented in the initializer), the contract is immediately broken for all withdrawers. There is no on-chain enforcement requiring this post-deployment call. The original Yield report noted the same pattern: a variable uninitialized at declaration with a setter that rejects the default value, creating a window of broken behavior.

---

### Recommendation

1. Initialize both variables to safe defaults inside `initialize()`:
   ```solidity
   withdrawalDelay = 7 days;
   maxNumberOfWithdrawalsPerUser = 10;
   ```
2. Add a minimum threshold check in `setWithdrawalDelay()` (e.g., `>= 1 days`) analogous to the `MAX_WITHDRAWAL_DELAY` upper bound already present.
3. Emit initialization events so off-chain monitoring can confirm correct deployment state.

---

### Proof of Concept

1. Deploy `KernelDepositPool` proxy and call `initialize(admin, kernelToken, rewardToken)`.
2. Admin does **not** call `setMaxNumberOfWithdrawalsPerUser()` (no on-chain enforcement requires it).
3. User calls `stake(1e18)` — succeeds, KERNEL tokens are transferred in.
4. User calls `initiateWithdrawal(1e18)`:
   - `userWithdrawalIds[user].length` = `0`
   - `maxNumberOfWithdrawalsPerUser` = `0`
   - `0 >= 0` → `true` → reverts `WithdrawalLimitReached()`
5. User's KERNEL tokens are frozen in the contract with no recourse until admin intervention. [7](#0-6)

```text
    function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
        if (_amount == 0) revert AmountZero();
        if (balanceOf[msg.sender] < _amount) revert InsufficientStakedBalance();
        if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();

        balanceOf[msg.sender] -= _amount;
        totalKernelStaked -= _amount;

        // Create a withdrawal record
        uint256 withdrawalId = ++withdrawalCounter;
        uint256 unlockTime = block.timestamp + withdrawalDelay;

        withdrawals[withdrawalId] = Withdrawal({
            user: msg.sender, amount: _amount, unlockTime: unlockTime, claimed: false, withdrawalId: withdrawalId
        });
        userWithdrawalIds[msg.sender].push(withdrawalId);

        emit WithdrawalInitiated(msg.sender, _amount, withdrawalId, unlockTime);
```
