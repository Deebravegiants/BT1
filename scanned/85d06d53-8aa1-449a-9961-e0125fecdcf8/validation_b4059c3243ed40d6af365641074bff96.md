### Title
Increasing `withdrawalDelayBlocks` Retroactively Freezes Pending Withdrawal Requests - (File: contracts/LRTWithdrawalManager.sol)

### Summary
`LRTWithdrawalManager.setWithdrawalDelayBlocks` allows the manager to increase the withdrawal delay up to 16 days at any time. Because `completeWithdrawal` and `unlockQueue` both read the **current** `withdrawalDelayBlocks` value rather than the value stored at request creation time, any increase retroactively extends the lock period for all already-queued withdrawal requests, temporarily freezing users' rsETH beyond what they agreed to when they submitted.

### Finding Description
When a user calls `initiateWithdrawal`, their rsETH is immediately transferred to the contract and a `WithdrawalRequest` is stored containing only `rsETHUnstaked`, `expectedAssetAmount`, and `withdrawalStartBlock`. The delay itself is **not** snapshotted into the request. [1](#0-0) 

The manager can change `withdrawalDelayBlocks` at any time with no lower-bound restriction: [2](#0-1) 

Both the operator-facing `_unlockWithdrawalRequests` and the user-facing `_processWithdrawalCompletion` enforce the delay using the **live** storage variable: [3](#0-2) [4](#0-3) 

Because neither path reads a per-request snapshot, any post-submission increase to `withdrawalDelayBlocks` silently extends the lock on every pending request.

### Impact Explanation
Users who have already submitted withdrawal requests and had their rsETH locked in the contract can have their effective wait time extended from the default 8 days up to the maximum 16 days — an additional 8-day freeze — without any on-chain notice or consent. The rsETH is held by the contract and cannot be reclaimed; the user cannot cancel the request. This constitutes a **temporary freezing of funds** (Medium severity per the allowed scope).

### Likelihood Explanation
The `onlyLRTManager` role is a live operational role that legitimately adjusts protocol parameters (e.g., in response to a security event). No key compromise or collusion is required; a routine, well-intentioned delay increase is sufficient to trigger the impact on all in-flight withdrawal requests. The default delay is 8 days and the ceiling is 16 days, so the maximum retroactive extension is 8 days — material for users expecting timely access to their assets.

### Recommendation
Snapshot `withdrawalDelayBlocks` into each `WithdrawalRequest` struct at the time `initiateWithdrawal` is called, and use that stored value in both `_unlockWithdrawalRequests` and `_processWithdrawalCompletion`. This ensures parameter changes only affect future requests, not existing ones — directly analogous to the Gearbox recommendation to restrict retroactive param changes.

### Proof of Concept
1. Protocol is live; `withdrawalDelayBlocks = 57,600` (~8 days at 12 s/block).
2. Alice calls `initiateWithdrawal(stETH, 10 ether, ...)`. Her rsETH is transferred to the contract; `withdrawalStartBlock = N` is stored.
3. Manager calls `setWithdrawalDelayBlocks(115_200)` (~16 days). No restriction prevents this.
4. At block `N + 57,600` (8 days later) Alice calls `completeWithdrawal`. The check `block.number < N + 115_200` is true → `WithdrawalDelayNotPassed` revert.
5. The operator also cannot call `unlockQueue` for Alice's request until block `N + 115_200`.
6. Alice's rsETH remains locked for an additional 8 days she never agreed to, with no recourse. [5](#0-4) [2](#0-1) [4](#0-3)
