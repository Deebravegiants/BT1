### Title
Missing Slippage Protection on `instantWithdrawal()` - (File: `contracts/LRTWithdrawalManager.sol`)

### Summary

`LRTWithdrawalManager.instantWithdrawal()` burns rsETH and transfers assets to the user without accepting a `minAssetAmountExpected` parameter. The deposit-side functions (`LRTDepositPool.depositETH()` and `depositAsset()`) both accept and enforce a `minRSETHAmountExpected` slippage guard, but the corresponding withdrawal path has no equivalent protection.

### Finding Description

The protocol provides a preview function `getExpectedAssetAmount()` that users can call off-chain to estimate how many assets they will receive for a given rsETH amount before calling `instantWithdrawal()`. The preview computes:

```
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

The execution function `instantWithdrawal()` recomputes the same value at transaction time using the live oracle prices, then immediately burns the user's rsETH before transferring assets:

```solidity
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
...
uint256 fee = (assetAmountUnlocked * instantWithdrawalFee) / 10_000;
uint256 userAmount = assetAmountUnlocked - fee;
...
_transferAsset(asset, msg.sender, userAmount);
``` [1](#0-0) 

There is no `minAssetAmountExpected` parameter. If the rsETH oracle price decreases or the asset oracle price increases between the user's preview call and the transaction's execution, the user receives fewer assets than expected with no ability to revert.

By contrast, the deposit path enforces slippage protection:

```solidity
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [2](#0-1) 

The asymmetry is exact: `depositETH()` and `depositAsset()` both accept `minRSETHAmountExpected` [3](#0-2) , while `instantWithdrawal()` accepts no equivalent. [4](#0-3) 

The code's own NatSpec comment at line 210–211 acknowledges a related timing risk for fees, but the asset amount itself is equally unprotected:

> "Uses the fee set at execution time. Managers can raise it right before this call, making withdrawals cost more than expected." [5](#0-4) 

### Impact Explanation

A user who previews via `getExpectedAssetAmount()` and then submits `instantWithdrawal()` may receive materially fewer assets than expected if the oracle updates between preview and execution. The rsETH burn is irreversible — once `burnFrom` executes, the user cannot recover their tokens regardless of the asset amount received. This constitutes a failure to deliver the promised return, with direct value loss to the user.

**Impact: Low** — Contract fails to deliver promised returns; the user receives the current oracle-fair value but not the previewed amount.

### Likelihood Explanation

The LRT oracle (`LRTOracle`) is updated by protocol operators and can change at any time. The rsETH price is a function of total assets under management and total rsETH supply, both of which can shift between blocks. Any user who previews and then submits in a subsequent block is exposed. This is a routine operational condition, not an edge case.

**Likelihood: Medium** — Oracle updates are frequent and the gap between preview and execution is inherent to normal user behavior.

### Recommendation

Add a `minAssetAmountExpected` parameter to `instantWithdrawal()` and revert if the computed `userAmount` falls below it:

```solidity
function instantWithdrawal(
    address asset,
    uint256 rsETHUnstaked,
    uint256 minAssetAmountExpected,  // <-- add this
    string calldata referralId
) external ... {
    ...
    uint256 userAmount = assetAmountUnlocked - fee;
    if (userAmount < minAssetAmountExpected) revert SlippageExceeded();
    ...
}
```

This mirrors the existing pattern in `depositETH()` / `depositAsset()`.

### Proof of Concept

1. User calls `getExpectedAssetAmount(ETH_TOKEN, 1 ether)` → preview returns `1.05 ETH`.
2. Oracle updates: rsETH price drops 3%.
3. User submits `instantWithdrawal(ETH_TOKEN, 1 ether, referralId)`.
4. `getExpectedAssetAmount` now returns `~1.019 ETH`; after a 0.5% instant withdrawal fee, user receives `~1.014 ETH`.
5. No revert occurs. User loses ~3.4% relative to their preview with no recourse, because rsETH was already burned at step 4. [6](#0-5)
