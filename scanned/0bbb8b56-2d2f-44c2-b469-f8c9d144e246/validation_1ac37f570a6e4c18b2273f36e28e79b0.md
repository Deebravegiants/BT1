### Title
`_checkAndUpdateDailyFeeMintLimit` lacks zero-guard, blocking `updateRSETHPrice()` when `maxFeeMintAmountPerDay` is uninitialized and fees are owed — (`contracts/LRTOracle.sol`)

### Summary
When `maxFeeMintAmountPerDay` is zero (its default uninitialized value, or explicitly reset to zero by the manager) and the protocol accrues staking rewards (`totalETHInProtocol > previousTVL`) with a non-zero `protocolFeeInBPS`, every call to `updateRSETHPrice()` reverts with `DailyFeeMintLimitExceeded`. The price oracle is frozen until admin intervention.

### Finding Description
`_checkAndUpdateDailyFeeMintLimit` enforces the check:

```solidity
// LRTOracle.sol line 205
if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
    revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
}
``` [1](#0-0) 

When `maxFeeMintAmountPerDay == 0` and `feeAmount > 0`, the condition `feeAmount > 0` is always true, causing an unconditional revert. The view function `remainingDailyFeeMintLimit()` correctly short-circuits on zero:

```solidity
// LRTOracle.sol line 171
if (maxFeeMintAmountPerDay == 0) return 0;
``` [2](#0-1) 

But `_checkAndUpdateDailyFeeMintLimit` has no equivalent guard. The fee-minting branch in `_updateRsETHPrice` is entered whenever `protocolFeeInETH > 0`:

```solidity
// LRTOracle.sol lines 299–303
if (protocolFeeInETH > 0) {
    uint256 rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice);
    _checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee);
``` [3](#0-2) 

`protocolFeeInETH` is positive whenever the protocol is unpaused and TVL has grown:

```solidity
// LRTOracle.sol lines 244–247
if (!protocolPaused && totalETHInProtocol > previousTVL) {
    uint256 rewardAmount = totalETHInProtocol - previousTVL;
    protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
}
``` [4](#0-3) 

`maxFeeMintAmountPerDay` is never set in `initialize` or `reinitialize`, so it defaults to zero:

```solidity
// LRTOracle.sol lines 64–79 (initialize + reinitialize — no maxFeeMintAmountPerDay assignment)
``` [5](#0-4) 

Both public entry points (`updateRSETHPrice` and `updateRSETHPriceAsManager`) call the same `_updateRsETHPrice`, so neither can succeed under these conditions. [6](#0-5) 

### Impact Explanation
`rsETHPrice` is never written (line 313 is never reached), leaving the oracle price stale. All protocol components that consume `rsETHPrice` — deposits, withdrawals, pool swaps — operate on an incorrect price until admin manually calls `setMaxFeeMintAmountPerDay` with a non-zero value. This constitutes **temporary freezing of the price-update mechanism**, matching the Medium scope target. [7](#0-6) 

### Likelihood Explanation
`maxFeeMintAmountPerDay` is zero by default and is not initialized in either `initialize` or `reinitialize`. Any deployment or upgrade where the manager forgets to call `setMaxFeeMintAmountPerDay` before staking rewards accrue will trigger this. The manager can also explicitly set it to zero (e.g., intending to "disable" fee minting), not realizing it blocks all price updates. The condition is realistic and requires no attacker — only normal protocol operation (TVL growth).

### Recommendation
Add a zero-bypass guard in `_checkAndUpdateDailyFeeMintLimit`, consistent with the existing logic in `remainingDailyFeeMintLimit`:

```solidity
function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
    if (maxFeeMintAmountPerDay == 0) return; // treat 0 as "no limit / fee minting disabled"
    ...
}
```

Alternatively, treat `maxFeeMintAmountPerDay == 0` as "fee minting disabled" and skip the `_checkAndUpdateDailyFeeMintLimit` call entirely in `_updateRsETHPrice` when `maxFeeMintAmountPerDay == 0`.
