### Title
Missing L2 Sequencer Uptime Check in `PriceProviderL2` and `ProtectedPriceProviderL2` Allows Stale-Price Swaps After Sequencer Recovery - (File: `smart-contracts-poc/contracts/PriceProviderL2.sol`, `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

---

### Summary

`PriceProviderL2` and `ProtectedPriceProviderL2` are explicitly designed for L2 chains and implement a `FUTURE_TOLERANCE` parameter to handle sequencer clock skew. However, neither contract checks whether the L2 sequencer itself is currently up or has recently recovered from downtime. When the sequencer goes down and comes back up, oracle data published before the outage can still pass the `_isStale` check (bounded only by `MAX_TIME_DELTA`, which can be up to 7 days), allowing `MetricOmmPool.swap` to execute at prices that no longer reflect current market conditions.

---

### Finding Description

Both L2 price providers implement a staleness check that compares the oracle's `refTime` against `block.timestamp`:

```solidity
// PriceProviderL2.sol and ProtectedPriceProviderL2.sol — identical logic
function _isStale(
    uint256 refTime, uint256 nowTs,
    uint256 maxDelta, uint256 futureTol
) internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) return (refTime - nowTs) > futureTol;
    return (nowTs - refTime) > maxDelta;
}
``` [1](#0-0) 

The `FUTURE_TOLERANCE` parameter (≤ 1 hour) handles the case where the oracle publisher's clock is slightly ahead of the L2 block timestamp — sequencer **clock skew**. This is a different problem from sequencer **downtime**.

When an L2 sequencer (Arbitrum, Optimism, Base, etc.) goes down:
1. No transactions can be processed on-chain.
2. Oracle publishers cannot push new price updates to the L2.
3. The last oracle update (with its `refTime` and price) remains frozen in the oracle contract.

When the sequencer recovers:
1. `block.timestamp` resumes from where it left off.
2. The oracle's `refTime` is from before the outage.
3. If `(block.timestamp - refTime) ≤ MAX_TIME_DELTA`, the staleness check **passes**.
4. `_getBidAndAskPrice()` returns the pre-outage bid/ask to the pool. [2](#0-1) 

The pool's swap path then consumes this stale quote without any additional guard:

```
pool.swap
  → _getBidAndAskPriceX64()
    → IPriceProvider(activePriceProvider).getBidAndAskPrice()   // returns stale bid/ask
  → SwapMath.midAndSpreadFeeX64FromBidAsk(bid, ask)
  → _executeSwap(...)                                           // settles at stale price
  → token transfer to recipient
``` [3](#0-2) 

The constructor enforces `MAX_TIME_DELTA ∈ (0, 7 days]`:

```solidity
if (_maxTimeDelta == 0 || _maxTimeDelta > 7 days) revert MaxTimeDeltaOutOfBounds();
``` [4](#0-3) 

For Chainlink feeds (heartbeat 1 h or 24 h), operators will naturally set `MAX_TIME_DELTA` to match the heartbeat. Any sequencer outage shorter than `MAX_TIME_DELTA` will leave the stale price accepted.

The codebase's own registry already ships a `ChainlinkVerifierL2` contract with `sequencerUptimeFeed` and `GRACE_PERIOD` fields, confirming the team is aware of the pattern — but neither `PriceProviderL2` nor `ProtectedPriceProviderL2` integrates it. [5](#0-4) 

---

### Impact Explanation

A trader can execute `MetricOmmPool.swap` immediately after sequencer recovery using a bid/ask that was valid before the outage but is now stale. If the real market price moved during the outage (e.g., ETH dropped 5%), the trader can buy token0 at the pre-crash ask price and immediately sell at the real market price, extracting value from the pool's LP reserves. This is a direct loss of user principal (LP assets) and matches the allowed impact gate: **bad-price execution — stale bid/ask quote reaches a pool swap**.

---

### Likelihood Explanation

L2 sequencer outages are documented historical events on Arbitrum, Optimism, and Base. The vulnerability requires no privileged access — any public caller can invoke `swap` the moment the sequencer recovers. The exploitation window equals the outage duration, which can range from minutes to hours. For pools using Chainlink feeds with `MAX_TIME_DELTA` set to the feed's heartbeat (1 h or 24 h), any outage shorter than that window is exploitable.

---

### Recommendation

Add a sequencer uptime check inside `_getBidAndAskPrice()` in both `PriceProviderL2` and `ProtectedPriceProviderL2`, following the pattern already present in the registry's `ChainlinkVerifierL2`:

```solidity
// Immutable set at construction
AggregatorV3Interface public immutable sequencerUptimeFeed;
uint256 public constant GRACE_PERIOD = 3600; // 1 hour after recovery

function _checkSequencer() internal view {
    if (address(sequencerUptimeFeed) == address(0)) return;
    (, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
    // answer == 0: sequencer is up; answer == 1: sequencer is down
    if (answer != 0) return (0, type(uint128).max); // sequencer down → fail closed
    if (block.timestamp - startedAt < GRACE_PERIOD) return (0, type(uint128).max); // grace period
}
```

Call `_checkSequencer()` at the top of `_getBidAndAskPrice()` before the oracle read. If the sequencer is down or within the grace period, return the `(0, type(uint128).max)` sentinel so `getBidAndAskPrice()` reverts with `FeedStalled`, blocking all swaps until prices are fresh again.

### Proof of Concept

```
Setup:
  - Deploy PriceProviderL2 on Arbitrum with MAX_TIME_DELTA = 3600 (1 hour)
  - Pool has ETH/USDC, oracle price = $3000

Attack:
  T=0:    Sequencer goes down. Last oracle refTime = T=0, price = $3000.
  T=0..D: Real ETH price drops to $2800 (off-chain). No L2 txs possible.
  T=D:    Sequencer recovers (D = 30 minutes < MAX_TIME_DELTA = 1 hour).
          block.timestamp = D, oracle refTime = 0.
          _isStale(0, D, 3600, FUTURE_TOL) → (D - 0) = 1800 ≤ 3600 → NOT stale.
          getBidAndAskPrice() returns bid/ask based on $3000.
  T=D+1:  Attacker calls pool.swap(zeroForOne=true, ...) — buys ETH at $3000 ask.
          Real market price is $2800. Attacker immediately sells on CEX for $2800.
          Pool LPs absorb the $200/ETH loss.
``` [6](#0-5) [7](#0-6)
