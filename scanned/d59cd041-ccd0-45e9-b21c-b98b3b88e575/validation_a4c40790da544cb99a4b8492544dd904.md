## Full Trace

**Step 1 — Initial state of `LRTOracle.rsETHPrice`**

`rsETHPrice` is a plain `uint256` storage variable. It is never set in `initialize()`. [1](#0-0) 

It only becomes non-zero after the first call to `_updateRsETHPrice()`, which sets it to `1 ether` when supply is zero, or to a computed value otherwise. [2](#0-1) 

Before that first call, `rsETHPrice == 0`.

**Step 2 — `RSETHRateProvider.getLatestRate()` returns 0 verbatim**

`getLatestRate()` reads `rsETHPrice` directly with no zero-guard. [3](#0-2) 

**Step 3 — `CrossChainRateProvider.updateRate()` is permissionless and has no zero-check**

`updateRate()` is `external payable` with no role restriction. It reads `getLatestRate()`, stores it, encodes it, and sends it over LayerZero — all without checking whether the rate is non-zero. [4](#0-3) 

**Step 4 — `CrossChainRateReceiver.lzReceive()` stores the decoded rate unconditionally**

The receiver decodes the payload and writes `rate = _rate` with no zero-check. [5](#0-4) 

**Step 5 — `RSETHPoolV2NBA.deposit()` divides by the rate**

`viewSwapRsETHAmountAndFee()` computes `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate`. If `rsETHToETHrate == 0`, this is a division-by-zero revert. [6](#0-5) 

`deposit()` calls this unconditionally. [7](#0-6) 

---

## Assessment

The full chain is real and unguarded in the code:

| Step | Contract | Guard present? |
|---|---|---|
| `rsETHPrice` starts at 0 | `LRTOracle` | No zero-init in `initialize()` |
| `getLatestRate()` returns 0 | `RSETHRateProvider` | No zero-check |
| `updateRate()` broadcasts 0 | `CrossChainRateProvider` | No zero-check, no access control |
| `lzReceive()` stores 0 | `CrossChainRateReceiver` | No zero-check |
| `deposit()` divides by 0 | `RSETHPoolV2NBA` | No zero-check |

The attack window is the period between deploying `LRTOracle` and the first successful `updateRSETHPrice()` call. Any address can call `updateRate()` during this window. Once `rate = 0` is stored in the receiver, all `deposit()` calls revert until the protocol broadcasts a valid rate. The freeze is **temporary** (recoverable by broadcasting a valid rate), matching the "Medium — Temporary freezing of funds" scope.

This does not require admin compromise, front-running of a specific victim transaction, or any external dependency failure. It requires only calling a public function during an initialization window that the code itself creates.

### Title
Zero `rsETHPrice` broadcast via permissionless `updateRate()` causes division-by-zero in `RSETHPoolV2NBA.deposit()` — (`contracts/cross-chain/RSETHRateProvider.sol`)

### Summary
`LRTOracle.rsETHPrice` defaults to `0` before the first `updateRSETHPrice()` call. `RSETHRateProvider.updateRate()` is permissionless and has no zero-check on the fetched rate. Any caller can broadcast `rate = 0` to `RSETHRateReceiver`, which stores it unconditionally. `RSETHPoolV2NBA.deposit()` then reverts with division-by-zero on every call until a valid rate is re-broadcast.

### Finding Description
`LRTOracle` does not initialize `rsETHPrice` in its `initialize()` function, leaving it at the Solidity default of `0`. [8](#0-7) 

`RSETHRateProvider.getLatestRate()` reads this value directly. [3](#0-2) 

`CrossChainRateProvider.updateRate()` is callable by anyone and performs no zero-check before encoding and sending the rate as a LayerZero payload. [9](#0-8) 

`CrossChainRateReceiver.lzReceive()` decodes and stores the rate without validation. [5](#0-4) 

`RSETHPoolV2NBA.viewSwapRsETHAmountAndFee()` divides by the stored rate with no zero-guard, causing a revert. [10](#0-9) 

### Impact Explanation
All calls to `RSETHPoolV2NBA.deposit()` revert with division-by-zero while `rate == 0` is stored in the receiver. This is a **temporary freezing of funds** (Medium): no funds are lost, but pool deposits are completely blocked until the protocol re-broadcasts a valid rate.

### Likelihood Explanation
The attack window exists between `LRTOracle` deployment and the first `updateRSETHPrice()` call. `updateRate()` requires no role and only a small ETH value for LayerZero gas. The window may be short in practice but is structurally guaranteed by the code.

### Recommendation
Add a non-zero guard in `CrossChainRateProvider.updateRate()`:

```solidity
uint256 latestRate = getLatestRate();
require(latestRate != 0, "Rate cannot be zero");
```

Alternatively, add the same guard in `CrossChainRateReceiver.lzReceive()` before storing the rate, or initialize `rsETHPrice` to `1 ether` in `LRTOracle.initialize()`.

### Proof of Concept
```solidity
// 1. Deploy LRTOracle (do NOT call updateRSETHPrice)
// 2. Deploy RSETHRateProvider pointing to the oracle
// 3. Deploy RSETHRateReceiver
// 4. Set rateReceiver on provider

// Assert initial state
assertEq(ILRTOracle(oracle).rsETHPrice(), 0);

// 5. Anyone calls updateRate with LZ gas fee
rateProvider.updateRate{value: 0.1 ether}();

// 6. Simulate LZ delivery to receiver
// lzReceive decodes payload → rate = 0
// receiver.rate == 0
assertEq(rateReceiver.rate(), 0);

// 7. RSETHPoolV2NBA uses receiver as oracle
// deposit() → viewSwapRsETHAmountAndFee() → getRate() == 0
// → division by zero → revert
vm.expectRevert(); // division by zero
pool.deposit{value: 1 ether}("ref");
```
