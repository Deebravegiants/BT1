### Title
Daily Fee Mint Limit Asymmetry Permanently Freezes rsETH Price Updates During Yield-Generating Periods — (`contracts/LRTOracle.sol`)

---

### Summary

`_updateRsETHPrice()` calls `_checkAndUpdateDailyFeeMintLimit(0)` when there is no fee to mint, but calls `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` (with a positive value) when TVL has increased. Once `currentPeriodMintedFeeAmount == maxFeeMintAmountPerDay`, the zero-fee path passes the limit check while the positive-fee path always reverts. This creates a permanent asymmetry: `updateRSETHPrice()` succeeds when TVL is flat but reverts whenever yield is generated, freezing both the rsETH price and the treasury's unclaimed fee yield.

---

### Finding Description

In `_checkAndUpdateDailyFeeMintLimit`: [1](#0-0) 

The guard is a strict `>` comparison. When `currentPeriodMintedFeeAmount == maxFeeMintAmountPerDay`:

- `feeAmount = 0`: `X + 0 > X` → **false** → passes, price is updated.
- `feeAmount > 0`: `X + ε > X` → **true** → reverts with `DailyFeeMintLimitExceeded`.

In `_updateRsETHPrice()`, the two branches that invoke this check are: [2](#0-1) 

The `else` branch (line 310) unconditionally passes `0`, so any call where TVL has not grown succeeds. Any call where TVL has grown (i.e., real yield exists) reverts.

**Compounding effect:** Because `rsETHPrice` is never updated during yield-generating periods, `previousTVL = rsethSupply × rsETHPrice` stays stale. Each subsequent call accumulates a larger `rewardAmount`, producing an even larger `rsethAmountToMintAsProtocolFee`, making it progressively harder to ever pass the limit check even after the daily period resets.

---

### Impact Explanation

1. **Unclaimed yield frozen for treasury:** The protocol fee rsETH that should be minted to the treasury address is never minted during high-yield periods once the daily cap is hit.
2. **rsETH price staleness:** `rsETHPrice` is not updated, so `RSETHRateProvider.getLatestRate()` returns a stale rate to all cross-chain consumers. [3](#0-2) 

3. **No privileged escape hatch:** `updateRSETHPriceAsManager()` calls the same `_updateRsETHPrice()` internal function and hits the same revert path. The only remediation available to the manager is to call `setMaxFeeMintAmountPerDay()` to raise the cap, then retry — but this is reactive and the yield window is already missed. [4](#0-3) 

---

### Likelihood Explanation

- No attacker action is required. The daily cap being reached is the intended normal operation of the limit mechanism.
- Once reached within a period, any TVL increase (staking rewards, rebasing LSTs) causes every subsequent public `updateRSETHPrice()` call to revert for the rest of that period.
- Because the stale price inflates the next period's accumulated fee, the condition can persist across multiple periods without admin intervention.

---

### Recommendation

Replace the strict `>` with `>=` in the limit check, or skip the `_checkAndUpdateDailyFeeMintLimit(0)` call entirely when `feeAmount == 0` (there is nothing to check). The zero-fee path should not consume a "slot" in the daily accounting at all:

```solidity
// Only enforce the daily limit when there is actually a fee to mint
if (protocolFeeInETH > 0) {
    _checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee);
    ...
}
// No else branch needed — zero-fee updates don't touch the limit
```

Alternatively, change the guard to allow minting exactly up to the cap:

```solidity
// currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay  ← current (broken)
if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay && feeAmount > 0) { ... }
```

```text
    function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
        // Reset the period if it's unset or a day has passed
        if (block.timestamp >= feePeriodStartTime + 1 days) {
            currentPeriodMintedFeeAmount = 0;
            feePeriodStartTime = getCurrentPeriodStartTime();
        }

        // Check if minting would exceed the daily limit
        if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
            revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
        }

        currentPeriodMintedFeeAmount += feeAmount;
    }
```
