### Title
Missing Chainlink Oracle Staleness Check Enables Stale Price Exploitation During Deposits - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `price`, with no check on `updatedAt` (heartbeat timeout), `answeredInRound >= roundId`, or `price > 0`. This is the direct Solidity analog to the reported "missing timeout" pattern: just as the `iso-filecoin` RPC call had no timeout and could return a hung/stale response, this oracle call has no staleness window and will silently accept arbitrarily old price data. A depositor can exploit a stale inflated LST price to mint excess rsETH at the expense of existing holders.

### Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol`, the `getAssetPrice` function fetches the Chainlink price feed result but ignores every return value except `price`:

```solidity
// Line 52 — all staleness fields discarded
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

The five return values from `latestRoundData()` are `(roundId, answer, startedAt, updatedAt, answeredInRound)`. The contract silently drops `updatedAt` (the timestamp of the last oracle update), `answeredInRound` (the round in which the answer was computed), and `startedAt`. No heartbeat window (e.g., `block.timestamp - updatedAt > MAX_DELAY`) is enforced, and no `answeredInRound >= roundId` check is performed.

Contrast this with `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol`, which the same codebase ships and which does validate `answeredInRound < roundID` and `timestamp == 0` — demonstrating the team is aware of the pattern but did not apply it to the core deposit oracle.

The stale price flows directly into the deposit mint calculation:

- `ChainlinkPriceOracle.getAssetPrice()` ← called by
- `LRTOracle.getAssetPrice()` (line 157) ← called by
- `LRTOracle._getTotalEthInProtocol()` (line 339) ← called by
- `LRTOracle._updateRsETHPrice()` ← sets `rsETHPrice`
- `LRTDepositPool.getRsETHAmountToMint()` (line 520): `rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()`

### Impact Explanation
If a supported LST's Chainlink feed goes stale while the real market price of that LST has dropped (e.g., during network congestion, a sequencer outage on an L2 deployment, or a Chainlink node incident), the oracle continues to report the old, higher price. A depositor who calls `depositAsset()` with that LST at the stale inflated price receives more rsETH than the LST is actually worth. This dilutes the rsETH/ETH exchange rate for all existing holders — constituting theft of unclaimed yield from the existing rsETH holder base. The `pricePercentageLimit` guard in `_updateRsETHPrice` only triggers on large single-step price jumps during an oracle update call, not on the per-deposit mint calculation itself.

**Impact: High — Theft of unclaimed yield from existing rsETH holders.**

### Likelihood Explanation
Chainlink feeds have documented heartbeat intervals (e.g., 1 hour for ETH/USD, 24 hours for some LST feeds). During periods of low volatility, feeds may not update for the full heartbeat window. On L2 networks, sequencer downtime can prevent oracle updates entirely. An attacker monitoring mempool or oracle update gaps can time a deposit to exploit a stale price. This is a realistic, externally-triggered condition requiring no privileged access.

**Likelihood: Medium.**

### Recommendation
Add a configurable staleness threshold per asset feed and validate `updatedAt` against it in `getAssetPrice`:

```solidity
mapping(address asset => uint256 heartbeat) public assetHeartbeat;

function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > assetHeartbeat[asset]) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

This mirrors the pattern already used in `ChainlinkOracleForRSETHPoolCollateral.sol` and adds the missing heartbeat window check.

### Proof of Concept

1. Chainlink's stETH/ETH feed has a 24-hour heartbeat. During a period of low volatility, the feed has not updated for 23 hours. The real stETH price has dropped 2% due to a slashing event, but the feed still reports the old price.
2. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0)`.
3. `getRsETHAmountToMint` calls `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns the stale inflated price (e.g., `1.00e18` instead of `0.98e18`).
4. Attacker receives `rsethAmountToMint = (1000e18 * 1.00e18) / rsETHPrice` — approximately 2% more rsETH than the deposited stETH is worth.
5. The attacker immediately redeems or sells the excess rsETH, extracting value from existing holders.
6. No privileged role is required; the only entry point is the public `depositAsset` function.
