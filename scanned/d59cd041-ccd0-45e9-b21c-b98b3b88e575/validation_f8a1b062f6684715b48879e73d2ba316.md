### Title
Stale Chainlink Price Accepted Without Timestamp Validation in `ChainlinkPriceOracle.getAssetPrice()`, Enabling Over-Minting of rsETH - (File: contracts/oracles/ChainlinkPriceOracle.sol)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls Chainlink's `latestRoundData()` but silently discards the `updatedAt` timestamp, accepting any price regardless of age. This stale price flows directly into `LRTDepositPool.getRsETHAmountToMint()`, which determines how many rsETH tokens a depositor receives. If a supported LST depegs while its Chainlink feed is stale, a depositor can receive more rsETH than their deposit is worth, diluting and stealing value from existing rsETH holders.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the Chainlink price as follows:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

The `updatedAt` return value (4th field) is completely discarded. No check of the form `block.timestamp - updatedAt > stalePeriod` is performed. [1](#0-0) 

This is in direct contrast to `ChainlinkOracleForRSETHPoolCollateral.getRate()`, which at least checks `answeredInRound < roundID` and `timestamp == 0`, demonstrating the protocol is aware of staleness concerns in other oracle wrappers. [2](#0-1) 

The stale price propagates through the following call chain:

1. `LRTDepositPool.depositAsset()` → `_beforeDeposit()` → `getRsETHAmountToMint()` [3](#0-2) 
2. `getRsETHAmountToMint()` calls `lrtOracle.getAssetPrice(asset)` live (not a stored value) and divides by the stored `lrtOracle.rsETHPrice()`: [4](#0-3) 
3. `LRTOracle.getAssetPrice()` delegates to `IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset)`, which resolves to `ChainlinkPriceOracle.getAssetPrice()`: [5](#0-4) 
4. The same stale price also corrupts `_getTotalEthInProtocol()` → `_updateRsETHPrice()`, which computes and stores the new `rsETHPrice`: [6](#0-5) 

---

### Impact Explanation

**Impact: Critical — Direct theft of user funds.**

The minting formula is:

```
rsethAmountToMint = (depositAmount × getAssetPrice(asset)) / rsETHPrice
```

If `getAssetPrice(asset)` returns a stale, inflated price for a supported LST (e.g., stETH, rETH) that has depegged, the depositor receives more rsETH than their deposit is worth. When they later redeem that rsETH, they withdraw more ETH than they deposited, with the excess coming from the pool funded by other users. This is a direct, permanent transfer of value from existing rsETH holders to the attacker.

Additionally, if `updateRSETHPrice()` is called while the stale price is active, the stored `rsETHPrice` is also inflated, which can cause the price-deviation circuit breaker to trigger incorrectly (or fail to trigger when it should), compounding the damage. [7](#0-6) 

---

### Likelihood Explanation

Chainlink feeds for LSTs (stETH/ETH, rETH/ETH) have heartbeat intervals of up to 24 hours and deviation thresholds of 0.5%. During periods of network congestion, Chainlink node downtime, or a gradual LST depeg that stays within the deviation band, the feed can remain stale for an extended window. An attacker monitoring on-chain feed timestamps can identify this window and deposit the depegged LST at the stale (inflated) price. This is a realistic, externally-triggered scenario requiring no privileged access.

---

### Recommendation

Add a configurable `stalePeriod` and validate `updatedAt` in `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(, int256 price,, uint256 updatedAt,) = priceFeed.latestRoundData();
if (block.timestamp - updatedAt > stalePeriod) revert StalePrice();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

This mirrors the pattern already partially applied in `ChainlinkOracleForRSETHPoolCollateral`. [2](#0-1) 

---

### Proof of Concept

1. Chainlink's stETH/ETH feed last updated at `T-25h` (beyond the 24h heartbeat), reporting `1.0 ETH` per stETH.
2. stETH depegs to `0.95 ETH` on-chain, but the feed has not updated.
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
4. `getRsETHAmountToMint` computes: `(1000e18 × 1.0e18) / rsETHPrice`. With `rsETHPrice = 1.05e18`, attacker receives `≈952 rsETH`.
5. Actual value deposited: `1000 × 0.95 = 950 ETH`. rsETH received represents `952 × 1.05 = 999.6 ETH` in claims.
6. Attacker redeems rsETH and extracts ~49.6 ETH more than deposited, at the expense of existing rsETH holders.

The entry point `depositAsset()` is permissionless and reachable by any external caller. [3](#0-2) [1](#0-0)
