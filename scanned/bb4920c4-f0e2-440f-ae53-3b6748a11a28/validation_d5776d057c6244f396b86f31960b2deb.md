### Title
MANAGER-callable `setKingToken` strands existing KING yield tokens with no manager-accessible recovery path — (`contracts/king-protocol/TokenSwap.sol`)

---

### Summary

`TokenSwap.setKingToken` is callable by the `MANAGER_ROLE` and atomically redirects the `kingToken` pointer to a new address. Any KING tokens already held by the contract under the old address become unreachable through `withdrawKing`, and the only recovery function (`emergencyWithdraw`) is restricted to `DEFAULT_ADMIN_ROLE`. If the admin is a timelock or multisig, the stranded yield tokens are frozen for an unbounded period.

---

### Finding Description

`setKingToken` performs a bare pointer swap with no balance check and no migration step: [1](#0-0) 

After the call, `kingToken` points to `_kingToken`. `withdrawKing` then reads the balance of the **new** token: [2](#0-1) 

Because the new token has zero balance in the contract, every `withdrawKing` call reverts with `InsufficientBalance`. The old KING tokens remain in the contract but are no longer reachable through any manager-accessible path.

The only escape hatch is `emergencyWithdraw`, which is `onlyAdmin`: [3](#0-2) 

The `MANAGER_ROLE` has no access to `emergencyWithdraw`. [4](#0-3) 

---

### Impact Explanation

Accrued KING yield tokens held by `TokenSwap` are permanently inaccessible to the manager after a `setKingToken` call. If the admin is a timelock or multisig (the standard deployment pattern for privileged roles), recovery is delayed by the governance delay or may never happen if quorum is not reached. This matches the **Medium — Permanent freezing of unclaimed yield** impact category.

---

### Likelihood Explanation

`setKingToken` is a routine operational function intended to be called during a KING token migration. The manager has no on-chain signal that calling it will strand existing balances. There is no balance guard, no two-step migration, and no requirement to drain the contract first. A legitimate, non-malicious manager performing a standard token upgrade triggers the freeze inadvertently.

---

### Recommendation

1. **Restrict `setKingToken` to `onlyAdmin`**, since it has admin-level consequences (stranding funds that only admin can recover).
2. **Or add a balance guard**: revert if `kingToken.balanceOf(address(this)) > 0` before allowing the pointer swap.
3. **Or add an atomic migration step**: transfer the full old-token balance to a safe address (or to the caller) inside `setKingToken` before updating the pointer.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Invariant: after any setKingToken call, the manager must still be able
// to withdraw the pre-existing KING balance.

function test_setKingToken_strands_yield() public {
    // 1. Seed TokenSwap with 1000 old KING tokens (simulating accrued yield)
    oldKing.mint(address(tokenSwap), 1000e18);

    // 2. Manager updates the KING token pointer to a new address
    vm.prank(manager);
    tokenSwap.setKingToken(address(newKing)); // newKing has 0 balance in tokenSwap

    // 3. Manager attempts to withdraw the accrued yield
    vm.prank(manager);
    vm.expectRevert(TokenSwap.InsufficientBalance.selector);
    tokenSwap.withdrawKing(manager, 1000e18); // reverts — kingToken now = newKing

    // 4. Old KING tokens are confirmed still in the contract
    assertEq(oldKing.balanceOf(address(tokenSwap)), 1000e18);

    // 5. Manager cannot call emergencyWithdraw — admin-only
    vm.prank(manager);
    vm.expectRevert("Caller is not admin");
    tokenSwap.emergencyWithdraw(address(oldKing), manager, 1000e18);

    // Yield is frozen until admin acts — with no time-bound guarantee.
}
```

        UtilLib.checkNonZeroAddress(token);
        UtilLib.checkNonZeroAddress(recipient);

            revert ZeroAmount();
        }
