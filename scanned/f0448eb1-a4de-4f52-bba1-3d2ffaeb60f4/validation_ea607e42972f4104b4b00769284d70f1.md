### Title
Changing `withdrawalDelayBlocks` After Withdrawal Initiation Temporarily Freezes User Funds - (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTWithdrawalManager` implements a two-phase withdrawal flow (`initiateWithdrawal` → `unlockQueue` / `completeWithdrawal`). The `WithdrawalRequest` struct stores `withdrawalStartBlock` at initiation time but **never stores the `withdrawalDelayBlocks` value in effect at that moment**. Both `_unlockWithdrawalRequests` and `_processWithdrawalCompletion` read `withdrawalDelayBlocks` from the current mutable state at execution time. If the manager increases `withdrawalDelayBlocks` after users have already initiated withdrawals, those users' rsETH is temporarily frozen in the contract for longer than the delay they agreed to at initiation.

---

### Finding Description

At `initiateWithdrawal`, the user transfers rsETH to the contract and a `WithdrawalRequest` is stored:

```solidity
withdrawalRequests[requestId] = WithdrawalRequest({
    rsETHUnstaked: rsETHUnstaked,
    expectedAssetAmount: expectedAssetAmount,
    withdrawalStartBlock: block.number   // ← only block number stored, NOT the delay
});
``` [1](#0-0) 

The `WithdrawalRequest` struct contains no field for the delay in effect at initiation time:

```solidity
struct WithdrawalRequest {
    uint256 rsETHUnstaked;
    uint256 expectedAssetAmount;
    uint256 withdrawalStartBlock;
}
``` [2](#0-1) 

At `unlockQueue` → `_unlockWithdrawalRequests`, the **current** `withdrawalDelayBlocks` is read:

```solidity
if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) break;
``` [3](#0-2) 

At `completeWithdrawal` → `_processWithdrawalCompletion`, the **current** `withdrawalDelayBlocks` is read again:

```solidity
if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();
``` [4](#0-3) 

The manager can change `withdrawalDelayBlocks` at any time, up to a maximum of 16 days:

```solidity
function setWithdrawalDelayBlocks(uint256 withdrawalDelayBlocks_) external onlyLRTManager {
    if (withdrawalDelayBlocks_ > 16 days / 12 seconds) revert ExceedWithdrawalDelay();
    withdrawalDelayBlocks = withdrawalDelayBlocks_;
``` [5](#0-4) 

---

### Impact Explanation

**Impact: Medium — Temporary freezing of funds.**

If `withdrawalDelayBlocks` is increased after a user calls `initiateWithdrawal`:

1. The user's rsETH is already held by `LRTWithdrawalManager` (transferred at initiation).
2. `unlockQueue` will skip the request because `block.number < withdrawalStartBlock + newDelay`.
3. `completeWithdrawal` will revert with `WithdrawalDelayNotPassed`.
4. The user cannot recover their rsETH or receive the underlying asset until the new (longer) delay passes.

The maximum additional freeze is bounded by the cap of `16 days / 12 seconds` (~115,200 blocks), but even a change from 0 to the maximum imposes a 16-day freeze on users who initiated under the old delay. The rsETH is stuck in the contract and cannot be used or transferred during this period.

---

### Likelihood Explanation

**Likelihood: Low.**

The manager must call `setWithdrawalDelayBlocks` while pending withdrawals exist. This is a realistic operational scenario — for example, if EigenLayer's own withdrawal delay is extended, the protocol may legitimately want to increase `withdrawalDelayBlocks` to match. The action is not malicious; it is a design flaw where the delay is not snapshotted at initiation time.

---

### Recommendation

Store the `withdrawalDelayBlocks` value in effect at initiation time inside the `WithdrawalRequest` struct, and use the stored value (not the current global state) in both `_unlockWithdrawalRequests` and `_processWithdrawalCompletion`.

```solidity
struct WithdrawalRequest {
    uint256 rsETHUnstaked;
    uint256 expectedAssetAmount;
    uint256 withdrawalStartBlock;
    uint256 withdrawalDelayBlocks; // ← snapshot at initiation
}
```

Then replace both usages of the global `withdrawalDelayBlocks` with `request.withdrawalDelayBlocks`.

---

### Proof of Concept

1. `withdrawalDelayBlocks` is currently 0 (or 1 day in blocks).
2. Alice calls `initiateWithdrawal(ETH, 1 ether, "")`. Her rsETH is transferred to the contract. `WithdrawalRequest` is stored with `withdrawalStartBlock = N`.
3. Manager calls `setWithdrawalDelayBlocks(115200)` (≈16 days).
4. Alice calls `completeWithdrawal(ETH, "")` at block `N + 1`. The check `block.number < N + 115200` is true → reverts with `WithdrawalDelayNotPassed`.
5. Alice's rsETH remains locked in `LRTWithdrawalManager` for up to 16 days beyond her original expectation.
6. `unlockQueue` also skips Alice's request for the same reason, so the operator cannot unblock her either. [6](#0-5) [7](#0-6) [8](#0-7) [9](#0-8)
