### Title
Stale Cached `rsETHPrice` Allows Depositors to Mint Excess rsETH at an Outdated Exchange Rate — (File: `contracts/LRTOracle.sol`)

---

### Summary

`LRTOracle` stores `rsETHPrice` as a cached storage variable that is only updated when `updateRSETHPrice()` is explicitly called. The deposit minting formula in `LRTDepositPool.getRsETHAmountToMint()` divides by this cached value. When rewards have accrued (raising the true rsETH/ETH rate) but `updateRSETHPrice()` has not yet been called, the stored `rsETHPrice` is stale-low. Any depositor can exploit this window to receive more rsETH than their deposit is worth, stealing accrued yield from existing rsETH holders.

---

### Finding Description

`rsETHPrice` is declared as a plain storage variable in `LRTOracle`: [1](#0-0) 

It is only written inside `_updateRsETHPrice()`, which is reached exclusively through the two explicit update entry-points: [2](#0-1) 

Neither `depositAsset()` nor `depositETH()` in `LRTDepositPool` calls `updateRSETHPrice()`. The deposit flow is:

`depositAsset()` → `_beforeDeposit()` → `getRsETHAmountToMint()`: [3](#0-2) 

The minting formula is:

```
rsethAmountToMint = (amount × lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()
```

`getAssetPrice(asset)` reads a **live** price from the configured `IPriceFetcher` (e.g., Chainlink or the LST's own `getExchangeRate()`): [4](#0-3) 

But `lrtOracle.rsETHPrice()` returns the **cached** storage value. When staking rewards accrue inside EigenLayer strategies, the true rsETH/ETH rate rises, but `rsETHPrice` remains at its last-written value until someone calls `updateRSETHPrice()`. During this window the denominator is too small, so the formula mints more rsETH than the deposited assets are worth.

`_updateRsETHPrice()` computes the correct new price from live data only at call time: [5](#0-4) [6](#0-5) 

There is no mechanism that forces a price refresh before or during a deposit.

The same stale `rsETHPrice` is also consumed by the L2 pool contracts through `RSETHRateProvider`: [7](#0-6) 

which feeds `RSETHPoolV2.viewSwapRsETHAmountAndFee()` and `RSETHPoolV3ExternalBridge.viewSwapRsETHAmountAndFee()`: [8](#0-7) [9](#0-8) 

---

### Impact Explanation

When `rsETHPrice` is stale-low (rewards have accrued but the price has not been refreshed), a depositor receives more rsETH than their deposit is worth. After `updateRSETHPrice()` is eventually called, the new price is computed over the now-larger rsETH supply, so the price per token is lower than it would have been. Every existing rsETH holder's share of the protocol TVL is diluted. This constitutes **theft of unclaimed yield** from existing holders — classified **High** under the allowed impact scope.

---

### Likelihood Explanation

`updateRSETHPrice()` is not called automatically; it depends on off-chain keepers or manual invocation. Staking rewards accrue continuously inside EigenLayer strategies. Any gap between reward accrual and the next price update creates an exploitable window. Because the function is public, an attacker can also observe the mempool, wait for a large reward accrual, and deposit before the keeper's update transaction is mined — without requiring any privileged access. Likelihood is **Medium**.

---

### Recommendation

1. **Refresh price atomically on deposit**: call `_updateRsETHPrice()` (or an equivalent view-only computation) inside `getRsETHAmountToMint()` so the minting formula always uses a freshly computed rate rather than the cached storage value.
2. **Alternatively**, compute `rsethAmountToMint` directly from `_getTotalEthInProtocol()` and `rsETH.totalSupply()` inline, bypassing the cached variable entirely.
3. **For L2 pools**, ensure the rate pushed by `RSETHRateProvider` is refreshed before each deposit window opens, or add a maximum-age check on the received rate.

---

### Proof of Concept

1. At time T₀, `rsETHPrice = 1.05 ETH` (last stored value). True value after reward accrual is `1.06 ETH`.
2. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
3. `getRsETHAmountToMint` computes: `100e18 × 1e18 / 1.05e18 ≈ 95.24 rsETH` (correct at true price would be `≈ 94.34 rsETH`). Attacker receives ~0.9 rsETH excess.
4. Anyone calls `updateRSETHPrice()`. The new price is computed over the enlarged supply, landing below `1.06 ETH`. Every pre-existing holder's rsETH is worth fractionally less than it should be.
5. Attacker repeats this every reward cycle, extracting yield continuously with no privileged access — only a public `depositAsset()` call. [10](#0-9) [11](#0-10)
