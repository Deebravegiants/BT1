### Title
Pausing `LRTWithdrawalManager` Freezes Funds of Users With In-Flight Withdrawal Requests - (File: contracts/LRTWithdrawalManager.sol)

### Summary

When `LRTWithdrawalManager` is paused by the `PAUSER_ROLE`, users who have already called `initiateWithdrawal()` — and whose rsETH has already been transferred into the contract — cannot call `completeWithdrawal()` or `instantWithdrawal()` to recover their assets. Additionally, the operator cannot call `unlockQueue()` to advance the withdrawal queue. This is a direct analog to the Isomorph H-6 finding: a pause mechanism that is too broad, blocking exit operations that should remain available to protect users who have already committed funds.

### Finding Description

In `LRTWithdrawalManager.initiateWithdrawal()`, the user's rsETH is immediately transferred into the contract:

```solidity
IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);
``` [1](#0-0) 

At this point the user has given up their rsETH and is waiting for the withdrawal to be processed. However, all three functions needed to complete or advance the withdrawal are gated by `whenNotPaused`:

- `completeWithdrawal()` at line 183
- `completeWithdrawalForUser()` at line 192
- `instantWithdrawal()` at line 212
- `unlockQueue()` at line 268 [2](#0-1) [3](#0-2) [4](#0-3) 

The `pause()` function is callable by any address holding `PAUSER_ROLE`: [5](#0-4) 

There is no cancel/refund path in the contract that would allow a user to reclaim their rsETH while the contract is paused. The rsETH sits in the contract, inaccessible to the user, for the entire duration of the pause.

### Impact Explanation

Users who have called `initiateWithdrawal()` have their rsETH locked inside `LRTWithdrawalManager`. While the contract is paused, they cannot:
1. Complete the withdrawal to receive their LST/ETH (`completeWithdrawal` reverts).
2. Use instant withdrawal (`instantWithdrawal` reverts).
3. Have an operator process the queue on their behalf (`unlockQueue` and `completeWithdrawalForUser` both revert).

This constitutes a **temporary freezing of funds** (Medium impact per the allowed scope). If the admin never unpauses, it escalates to permanent freezing (Critical).

### Likelihood Explanation

The `PAUSER_ROLE` is a distinct, lower-privilege role specifically designed to be used in emergencies (e.g., oracle manipulation, exploit detection). It is realistic that the contract is paused for an extended period during an incident. Any user who happened to have an in-flight withdrawal request at the time of the pause is affected with no recourse.

### Recommendation

Decouple the pause guard from exit/completion operations. Specifically, remove the `whenNotPaused` modifier from `completeWithdrawal()`, `completeWithdrawalForUser()`, and `instantWithdrawal()`. These functions only disburse assets that have already been committed by users and do not introduce new risk during a pause. Optionally, also allow `unlockQueue()` to run while paused so the queue can be advanced. Only `initiateWithdrawal()` — which accepts new rsETH commitments — should remain blocked when paused.

### Proof of Concept

1. Alice calls `initiateWithdrawal(stETH, 1e18, "ref")`. Her 1e18 rsETH is transferred to `LRTWithdrawalManager`. [6](#0-5) 
2. The `PAUSER_ROLE` calls `pause()` (e.g., in response to a suspected oracle issue). [5](#0-4) 
3. The operator tries to call `unlockQueue(stETH, ...)` — reverts with `ContractPaused`. [7](#0-6) 
4. Alice tries to call `completeWithdrawal(stETH, "ref")` — reverts with `ContractPaused`. [8](#0-7) 
5. Alice tries `instantWithdrawal(stETH, 1e18, "ref")` — reverts with `ContractPaused`. [3](#0-2) 
6. Alice's 1e18 rsETH remains locked in the contract with no recovery path for the duration of the pause.

        if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
            revert InvalidAmountToWithdraw();
        }

        IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);

        uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);

        if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();

        // preventing over-withdrawal.
        assetsCommitted[asset] += expectedAssetAmount;

        _addUserWithdrawalRequest(asset, rsETHUnstaked, expectedAssetAmount);

        emit ReferralIdEmitted(referralId);
    }
