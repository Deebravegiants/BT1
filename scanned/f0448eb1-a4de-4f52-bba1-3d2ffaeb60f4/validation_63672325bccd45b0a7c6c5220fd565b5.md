### Title
Missing L2 Sequencer Uptime Check in Chainlink Oracle Allows Stale Price Exploitation on L2 Pools - (File: contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol)

---

### Summary
`ChainlinkOracleForRSETHPoolCollateral.getRate()` calls Chainlink's `latestRoundData()` without verifying that the L2 sequencer is live. This oracle is used directly in `RSETHPoolV3` and `RSETHPoolV3ExternalBridge` on L2 chains (Arbitrum, Base, etc.) to price collateral tokens (wstETH, ETHx, WETH) during user deposits. When the sequencer is down, stale prices are silently accepted, enabling a depositor to mint excess wrsETH relative to the actual value of their collateral.

---

### Finding Description

`ChainlinkOracleForRSETHPoolCollateral.getRate()` fetches the collateral token price from a Chainlink feed: [1](#0-0) 

The only staleness guards present are `answeredInRound < roundID` and `timestamp == 0`. Neither of these detects sequencer downtime. When an Arbitrum or Base sequencer goes offline, Chainlink's L2 oracle freezes at the last reported price. The `answeredInRound >= roundID` condition remains satisfied because no new rounds are opened, so the stale price passes all checks.

This oracle is wired into `RSETHPoolV3.viewSwapRsETHAmountAndFee()` as `supportedTokenOracle[token]`: [2](#0-1) 

The result feeds directly into `deposit()`: [3](#0-2) 

The same pattern exists in `RSETHPoolV3ExternalBridge` and `RSETHPoolV3WithNativeChainBridge`, which are deployed on Base and Arbitrum respectively.

From the README, the affected L2 deployments include:
- **Arbitrum** `RSETHPool` with `wstETH Oracle` (`0x6932e85964fA31F70ed9DEbe63D9D969ad00F112`) and `ETHx Oracle` (`0x0c742a74fA8E1a00D6bC4623d485A84dD147864d`)
- **Base** `RSETHPoolV3ExternalBridge` with `wstETH Oracle` (`0xEFbBf9290cDA1c3046211D5464CC52Dae46C544C`)
- **X Layer** `RSETHPoolV3` with `WETHOracle` (`0x6F27976308001119a8e89cB447333DaaA3043CE7`) [4](#0-3) 

---

### Impact Explanation

When the sequencer is down and the price of a collateral token (e.g., wstETH) has dropped on the real market but the oracle still reports the pre-downtime (inflated) price, an attacker can:

1. Deposit wstETH at the stale inflated `tokenToETHRate`.
2. Receive `rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate` — inflated wrsETH.
3. After the sequencer resumes and prices normalize, redeem or sell the excess wrsETH for more ETH/tokens than deposited.

This constitutes direct theft of funds from the pool's liquidity at the expense of other depositors and the protocol. Impact: **Critical — direct theft of user/pool funds**.

---

### Likelihood Explanation

Arbitrum and Base sequencers have experienced documented outages. The attack window is the period between sequencer downtime and price recovery. An attacker monitoring sequencer status (e.g., via Chainlink's sequencer uptime feed) can execute this atomically upon sequencer restart, before the oracle price updates. Likelihood: **Medium** — requires a sequencer outage coinciding with a price move, but both are realistic and have occurred historically.

---

### Recommendation

Add a sequencer uptime check using Chainlink's L2 sequencer feed before consuming price data, following the [Chainlink documentation](https://docs.chain.link/data-feeds/l2-sequencer-feeds#example-code):

```solidity
// Example addition to ChainlinkOracleForRSETHPoolCollateral
AggregatorV3Interface public immutable sequencerUptimeFeed;
uint256 public constant GRACE_PERIOD = 3600; // 1 hour

function getRate() public view returns (uint256) {
    (, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
    // 0 = sequencer up, 1 = sequencer down
    if (answer != 0) revert SequencerDown();
    if (block.timestamp - startedAt < GRACE_PERIOD) revert GracePeriodNotOver();

    (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();
    // ... existing checks
}
```

Also add a heartbeat/max-staleness check (e.g., `block.timestamp - timestamp > MAX_STALENESS`) since the current code has no time-based staleness guard.

### Proof of Concept

1. Arbitrum sequencer goes offline. wstETH price was 1.2 ETH before downtime; real market price drops to 1.0 ETH during downtime.
2. Sequencer restarts. Chainlink oracle on Arbitrum still reports 1.2 ETH (stale).
3. Attacker calls `RSETHPool.deposit(wstETH, 100e18, "")` on Arbitrum.
4. `viewSwapRsETHAmountAndFee` calls `ChainlinkOracleForRSETHPoolCollateral.getRate()` → returns 1.2e18.
5. `rsETHAmount = 100e18 * 1.2e18 / rsETHToETHrate` — attacker receives ~20% more wrsETH than fair value.
6. Oracle updates to 1.0 ETH. Attacker redeems/sells excess wrsETH, extracting ~20 ETH of value from the pool. [1](#0-0) [5](#0-4)

    function getRate() public view returns (uint256) {
        (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
            AggregatorV3Interface(oracle).latestRoundData();

        if (answeredInRound < roundID) revert StalePrice();
        if (timestamp == 0) revert IncompleteRound();
        if (ethPrice <= 0) revert InvalidPrice();

        uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());

        return normalizedPrice;
    }

        uint256 amount,
        address token
    )
        public
        view
        onlySupportedToken(token)

    {
        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;
