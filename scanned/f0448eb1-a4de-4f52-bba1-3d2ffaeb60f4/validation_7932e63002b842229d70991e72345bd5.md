### Title
Permanent Freezing of Undistributed Rewards When `totalKernelStaked` Drops to Zero — (File: contracts/KERNEL/KernelDepositPool.sol)

---

### Summary
When the last staker calls `initiateWithdrawal()` during an active reward period, `totalKernelStaked` drops to zero. The Synthetix-style reward accounting then stops distributing rewards entirely, and the remaining undistributed rewards for that period are permanently locked in the contract with no recovery path.

---

### Finding Description

`KernelDepositPool` distributes rewards via a `rewardPerToken()` accumulator. When `totalKernelStaked == 0`, the function short-circuits and returns the stored value unchanged:

```solidity
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;   // ← no new rewards distributed
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
``` [1](#0-0) 

`initiateWithdrawal()` reduces `totalKernelStaked` without claiming the user's rewards and without any guard preventing the last staker from exiting mid-period:

```solidity
function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
    ...
    balanceOf[msg.sender] -= _amount;
    totalKernelStaked -= _amount;
    ...
}
``` [2](#0-1) 

Once `totalKernelStaked == 0`, the admin cannot restart distribution either, because `notifyRewardAmount()` hard-reverts:

```solidity
if (totalKernelStaked == 0) revert NoStakedTokens();
``` [3](#0-2) 

The contract itself acknowledges the risk in a NatSpec comment but relies on an off-chain deployment assumption rather than any code-level enforcement:

> *"If `totalKernelStaked` ever hits zero during a reward distribution window, any remaining rewards for that period will stay locked in the contract."* [4](#0-3) 

Critically, even if a new staker deposits 1 wei **after** `totalKernelStaked` has already hit zero, the rewards that accrued during the zero-staking window are irrecoverable. The `updateReward` modifier snapshots `updatedAt = lastTimeRewardApplicable()` at the moment of the new stake, permanently skipping the zero-staking interval in the accumulator.

---

### Impact Explanation

Any rewards that were scheduled to be distributed during the period when `totalKernelStaked == 0` are permanently locked in the contract. No user can claim them, and the admin has no recovery function. This matches **Medium — Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

The trigger is a normal, unprivileged user action (`initiateWithdrawal`). It requires only that the last remaining staker exits while a reward period is active — a realistic scenario during low-participation periods, market downturns, or when the reward period is long. The protocol's only mitigation is an off-chain operational promise, which is not enforceable on-chain.

---

### Recommendation

1. **Track undistributed rewards**: When `totalKernelStaked` drops to zero mid-period, accumulate the skipped reward amount in a separate variable and allow the admin to roll it into the next `notifyRewardAmount` call.
2. **Admin rescue function**: Add an admin-only function to recover rewards that were never distributed (i.e., `rewardsToken.balanceOf(address(this)) - totalClaimableRewards`).
3. **Alternatively**: Enforce a minimum stake (e.g., 1 wei held by the protocol itself) so `totalKernelStaked` can never reach zero during an active period.

---

### Proof of Concept

1. User A stakes 100 KERNEL at `t = 0`.
2. Admin calls `notifyRewardAmount(1000)` for a 100-day period → `rewardRate = 10/day`, `finishAt = t+100`.
3. At `t = 50`, User A calls `initiateWithdrawal(100)`:
   - `updateReward` snapshots `rewards[A] = 500`, `updatedAt = t+50`.
   - `totalKernelStaked` → 0.
4. From `t = 50` to `t = 100`, `rewardPerToken()` returns `rewardPerTokenStored` unchanged (short-circuit path). The 500 remaining rewards are never distributed.
5. At `t = 75`, a new staker deposits 1 wei:
   - `updateReward` runs with `totalKernelStaked == 0` → `rewardPerTokenStored` is still unchanged, `updatedAt` advances to `t+75`.
   - The rewards for `t=50` to `t=75` (250 tokens) are permanently skipped.
6. At `t = 100`, the new staker calls `getReward()` and receives only 250 tokens (for `t=75` to `t=100`).
7. The 250 tokens for `t=50` to `t=75` remain locked in the contract forever, with no function able to recover them.

    function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
        if (_amount == 0) revert AmountZero();
        if (balanceOf[msg.sender] < _amount) revert InsufficientStakedBalance();
        if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();

        balanceOf[msg.sender] -= _amount;
        totalKernelStaked -= _amount;

        // Create a withdrawal record
        uint256 withdrawalId = ++withdrawalCounter;
        uint256 unlockTime = block.timestamp + withdrawalDelay;

        withdrawals[withdrawalId] = Withdrawal({
            user: msg.sender, amount: _amount, unlockTime: unlockTime, claimed: false, withdrawalId: withdrawalId
        });
        userWithdrawalIds[msg.sender].push(withdrawalId);

        emit WithdrawalInitiated(msg.sender, _amount, withdrawalId, unlockTime);
    }
