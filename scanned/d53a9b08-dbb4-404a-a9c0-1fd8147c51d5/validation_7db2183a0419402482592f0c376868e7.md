### Title
Block Stuffing on Unichain Enables Stale-Rate Deposit Drain of Pool rsETH Reserves â€” (`contracts/pools/oracle/InterimRSETHOracle.sol`, `contracts/pools/RSETHPoolNoWrapper.sol`)

---

### Summary

`RSETHPoolNoWrapper` (deployed on Unichain) uses `InterimRSETHOracle` for its rsETH/ETH rate, which is updated manually by a privileged `MANAGER_ROLE`. The contract has **no daily mint limit**. An attacker can stuff Unichain blocks to delay a rate-increase transaction, then call `deposit()` repeatedly at the stale lower rate, receiving more rsETH from the pool's pre-funded balance than the deposited ETH justifies at the correct rate.

---

### Finding Description

**`InterimRSETHOracle.getRate()`** returns a manually-set storage variable: [1](#0-0) [2](#0-1) 

There is no on-chain automatic update; the rate only changes when `MANAGER_ROLE` calls `setRate()`. This creates a window between the "correct" rate and the on-chain rate whenever the manager is about to push an update.

**`RSETHPoolNoWrapper.deposit()`** computes the rsETH payout as:

```
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate
``` [3](#0-2) 

A lower (stale) rate produces a **larger** rsETH payout. The function has no daily mint cap: [4](#0-3) 

Compare with `RSETHPoolV2`, `RSETHPoolV3`, and `RSETHPoolV3ExternalBridge`, all of which enforce a `limitDailyMint` modifier: [5](#0-4) 

`RSETHPoolNoWrapper` has **zero occurrences** of `dailyMintLimit` or `limitDailyMint` in its source.

**Attack flow:**

1. Attacker monitors the Unichain mempool for a pending `InterimRSETHOracle.setRate(newRate)` transaction where `newRate > currentRate`.
2. Attacker submits high-gas filler transactions to saturate Unichain blocks, preventing the `setRate()` call from landing (block stuffing).
3. While the rate remains stale at `R_old < R_correct`, attacker calls `RSETHPoolNoWrapper.deposit{value: X}()` repeatedly.
4. Each call yields `X * 1e18 / R_old` rsETH instead of the correct `X * 1e18 / R_correct`.
5. Excess rsETH per deposit = `X * 1e18 * (1/R_old âˆ’ 1/R_correct)`, drawn from the pool's pre-funded rsETH balance.
6. With no daily cap, the attacker can repeat until the pool's rsETH balance is exhausted.

---

### Impact Explanation

The pool's pre-funded rsETH OFT balance is drained faster than the ETH deposited justifies at the correct rate. The protocol receives ETH at face value but has already disbursed excess rsETH that it cannot recover. This violates the invariant that pool rsETH reserves must not be drained below the ETH backing at the correct rate. Impact: **Low â€” Block stuffing / contract fails to deliver promised returns**.

---

### Likelihood Explanation

- Unichain is an OP Stack L2 with relatively low and predictable gas costs, making block stuffing economically feasible when the rate differential is large enough.
- The `InterimRSETHOracle` is explicitly described as an interim manual oracle, meaning rate updates are infrequent and predictable, widening the exploitable window.
- No daily mint limit in `RSETHPoolNoWrapper` means the attacker is only bounded by the pool's rsETH balance and their own ETH capital.
- The attack requires no privileged access and is executable by any EOA.

---

### Recommendation

1. **Add a daily mint limit** to `RSETHPoolNoWrapper` analogous to the `limitDailyMint` modifier present in `RSETHPoolV2`/`RSETHPoolV3`.
2. **Add a rate-staleness check** in `RSETHPoolNoWrapper.deposit()`: record the block timestamp of the last rate update in `InterimRSETHOracle` and revert if the rate is older than a configurable threshold (e.g., 1 hour).
3. **Long-term**: replace `InterimRSETHOracle` with a push-based oracle that includes a heartbeat/staleness guard, removing the manual-update window entirely.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Fork-test on Unichain (local fork, not public mainnet)
// 1. Deploy InterimRSETHOracle with rate = 1.05e18 (stale, should be 1.06e18)
// 2. Deploy RSETHPoolNoWrapper pointing to the oracle, pre-fund with rsETH
// 3. Simulate block stuffing: submit N filler txs to fill blocks,
//    preventing manager's setRate(1.06e18) from landing
// 4. Attacker deposits 10 ETH repeatedly

function testBlockStuffingStaleRateDrain() public {
    uint256 staleRate    = 1.05e18;
    uint256 correctRate  = 1.06e18;
    uint256 depositAmt   = 10 ether;
    uint256 iterations   = 50; // limited by pool rsETH balance

    // Simulate block stuffing: warp time but do NOT call setRate
    // (manager tx is stuck in mempool due to full blocks)

    uint256 totalRsETHReceived;
    uint256 totalETHDeposited;

    for (uint256 i = 0; i < iterations; i++) {
        vm.deal(attacker, depositAmt);
        vm.prank(attacker);
        pool.deposit{value: depositAmt}("ref");
        totalRsETHReceived += rsETH.balanceOf(attacker);
        totalETHDeposited  += depositAmt;
    }

    uint256 correctRsETH = totalETHDeposited * 1e18 / correctRate;
    // Assert attacker received MORE rsETH than correct rate justifies
    assertGt(totalRsETHReceived, correctRsETH);
    // Excess rsETH drained from pool's pre-funded balance
}
```
