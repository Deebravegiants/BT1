### Title
`RSETHPriceFeed.latestRoundData()` Returns ETH/USD `updatedAt` Instead of Composite rsETH/USD Staleness Timestamp — (File: contracts/oracles/RSETHPriceFeed.sol)

---

### Summary

`RSETHPriceFeed` implements `AggregatorV3Interface` and is intended to serve as a Chainlink-compatible rsETH/USD price feed for downstream consumers (e.g., lending protocols). Its `latestRoundData()` and `getRoundData()` functions compute `answer` as a composite of two independent price sources — `rsETH/ETH` from `LRTOracle.rsETHPrice()` and `ETH/USD` from a Chainlink feed — but return `updatedAt` exclusively from the ETH/USD Chainlink feed. The `rsETH/ETH` component has no timestamp tracked in `RSETHPriceFeed`, so the returned `updatedAt` does not reflect when the composite price was last valid. This is a direct structural analog to the reported GASLIMIT bug: a wrong value (ETH/USD update time) is placed in a context field (`updatedAt`) that is supposed to represent a different quantity (composite price freshness).

---

### Finding Description

`RSETHPriceFeed.latestRoundData()` is:

```solidity
function latestRoundData()
    external view
    returns (uint80 roundId, int256 answer, uint256 startedAt, uint256 updatedAt, uint80 answeredInRound)
{
    (roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.latestRoundData();
    answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
}
```

The `answer` is the product of two independent values:
- `ETH/USD` — from the Chainlink `ETH_TO_USD` feed, updated on its own heartbeat schedule.
- `rsETH/ETH` — from `RS_ETH_ORACLE.rsETHPrice()`, which is a stored `uint256` in `LRTOracle` updated only when `updateRSETHPrice()` is called externally.

The `updatedAt` returned is taken entirely from `ETH_TO_USD.latestRoundData()`. It reflects only when the ETH/USD Chainlink feed was last updated. It carries no information about when `LRTOracle.rsETHPrice()` was last refreshed.

`LRTOracle` stores `rsETHPrice` as a plain `uint256` with no associated `lastUpdatedAt` timestamp:

```solidity
uint256 public override rsETHPrice;
```

`updateRSETHPrice()` is a permissionless function but requires an explicit external call. There is no on-chain mechanism that forces it to be called on any schedule. The `rsETH/ETH` rate can therefore be arbitrarily stale while the ETH/USD Chainlink feed continues to update normally.

The same flaw exists in `getRoundData()`:

```solidity
(roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.getRoundData(_roundId);
answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
```

Here, `answer` uses the **current** `rsETHPrice` regardless of which historical round is requested, and `updatedAt` is from the ETH/USD round, not from the rsETH/ETH component.

---

### Impact Explanation

Any downstream protocol that consumes `RSETHPriceFeed` as a Chainlink-compatible feed and applies a staleness guard on `updatedAt` (the standard pattern) will pass the check even when the rsETH/ETH component is stale. The protocol will treat a potentially outdated rsETH/USD price as fresh.

- If `rsETHPrice` has not been updated for an extended period and the true rsETH/ETH rate has moved, the composite `answer` is incorrect while `updatedAt` appears recent.
- Downstream lending protocols that use this feed for collateral valuation or liquidation thresholds will operate on a wrong price without triggering their staleness circuit-breaker.
- The contract fails to deliver the promised semantics of the `AggregatorV3Interface`: that `updatedAt` faithfully represents when the returned `answer` was last computed.

**Impact: Low** — contract fails to deliver promised returns (incorrect `updatedAt` metadata misleads downstream staleness checks). In edge cases where `rsETHPrice` is stale and higher than the true rate (e.g., after a slashing event before `updateRSETHPrice()` is called), downstream protocols could overvalue rsETH collateral, escalating impact.

---

### Likelihood Explanation

`updateRSETHPrice()` is permissionless but not automatically triggered. Any period during which it is not called — network congestion, keeper failure, or simply no one calling it — leaves `rsETHPrice` stale. The ETH/USD Chainlink feed continues updating on its own heartbeat, so `updatedAt` from `latestRoundData()` will appear fresh to any consumer. This is a realistic operational condition, not a theoretical one.

---

### Recommendation

Track the timestamp at which `rsETHPrice` is last written in `LRTOracle`, and expose it. Then in `RSETHPriceFeed.latestRoundData()`, return `min(ethToUsdUpdatedAt, rsETHPriceUpdatedAt)` as `updatedAt` so that consumers receive the correct staleness indicator for the composite price.

```solidity
// In LRTOracle:
uint256 public rsETHPriceUpdatedAt;

// In _updateRsETHPrice(), after writing rsETHPrice:
rsETHPriceUpdatedAt = block.timestamp;

// In RSETHPriceFeed.latestRoundData():
(roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.latestRoundData();
answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
uint256 rsETHUpdatedAt = RS_ETH_ORACLE.rsETHPriceUpdatedAt();
if (rsETHUpdatedAt < updatedAt) updatedAt = rsETHUpdatedAt;
```

### Proof of Concept

1. `updateRSETHPrice()` is last called at time `T`. `LRTOracle.rsETHPrice()` is set to `1.05e18` (rsETH/ETH).
2. 48 hours pass. No one calls `updateRSETHPrice()`. The true rsETH/ETH rate is now `1.06e18` (rewards accrued).
3. The ETH/USD Chainlink feed updates normally every heartbeat. Its `updatedAt` is `T + 48h`.
4. A lending protocol calls `RSETHPriceFeed.latestRoundData()`.
5. Returned: `updatedAt = T + 48h` (from ETH/USD feed — appears fresh), `answer = 1.05e18 * ETH_USD_price / 1e18` (stale rsETH/ETH component).
6. The lending protocol's staleness check passes. It uses the stale rsETH/USD price.
