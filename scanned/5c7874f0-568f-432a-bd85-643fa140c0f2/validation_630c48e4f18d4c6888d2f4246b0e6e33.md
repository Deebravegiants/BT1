### Title
`getAvailableAssetAmount` Aggregates All Protocol Locations While `unlockQueue` Only Draws From `LRTUnstakingVault`, Causing Temporary Freezing of User rsETH — (`contracts/LRTWithdrawalManager.sol`)

---

### Summary

`initiateWithdrawal` gates user commitment using `getAvailableAssetAmount`, which counts assets across **all** protocol locations (DepositPool, NodeDelegators, EigenLayer strategies, Converter, and UnstakingVault). However, `unlockQueue` can only service withdrawals from the `LRTUnstakingVault` balance. This mismatch allows users to lock their rsETH into the withdrawal queue against assets that are not yet reachable, causing their rsETH to be frozen in `LRTWithdrawalManager` until an operator manually completes the multi-step EigenLayer unstaking cycle.

---

### Finding Description

**Step 1 — Commitment gate uses total protocol assets.**

`initiateWithdrawal` calls `getAvailableAssetAmount`: [1](#0-0) 

`getAvailableAssetAmount` delegates to `getTotalAssetDeposits`: [2](#0-1) 

`getTotalAssetDeposits` sums **every** location — DepositPool, NDCs, EigenLayer staked, EigenLayer unstaking, Converter, and UnstakingVault: [3](#0-2) 

**Step 2 — Unlock gate uses only the UnstakingVault balance.**

`unlockQueue` builds its params via `_createUnlockParams`, which sets `totalAvailableAssets` exclusively to `unstakingVault.balanceOf(asset)`: [4](#0-3) 

`_unlockWithdrawalRequests` then hard-stops if the vault balance is insufficient: [5](#0-4) 

**Step 3 — No cancellation path exists.**

Once `initiateWithdrawal` succeeds, the user's rsETH is transferred to `LRTWithdrawalManager` and `assetsCommitted` is incremented. There is no `cancelWithdrawal` function anywhere in the contract, so the rsETH remains locked until the operator completes the full EigenLayer unstaking cycle and calls `unlockQueue`. [6](#0-5) 

---

### Impact Explanation

A user's rsETH is transferred into `LRTWithdrawalManager` at `initiateWithdrawal` time and cannot be recovered until `unlockQueue` succeeds. If the assets backing the withdrawal are in EigenLayer (the protocol's normal operating state), the operator must: (1) call `initiateUnstaking` on a `NodeDelegator`, (2) wait for EigenLayer's withdrawal delay (~7 days), (3) call `completeUnstaking` to move assets to `LRTUnstakingVault`, and (4) call `unlockQueue`. During this entire window the user's rsETH is frozen — they cannot sell it, use it as collateral, or reclaim it. This matches the **Medium — Temporary freezing of funds** impact class.

---

### Likelihood Explanation

The protocol's primary purpose is to restake assets into EigenLayer. In normal operation the vast majority of assets reside in EigenLayer strategies, not in the `LRTUnstakingVault`. Any user who calls `initiateWithdrawal` when the vault holds less than their `expectedAssetAmount` will have their rsETH frozen. This is a routine, predictable state, making the likelihood **medium-high**.

---

### Recommendation

Align the commitment gate with the execution gate. Two options:

1. **Restrict `getAvailableAssetAmount` to UnstakingVault balance only** — change the function to return `unstakingVault.balanceOf(asset) - assetsCommitted[asset]` (adjusted for the buffer). This prevents users from committing against assets that are not yet reachable.

2. **Allow `unlockQueue` to pull from all locations** — extend the unlock path to also draw from the DepositPool and NDC balances, mirroring the broader accounting used at commitment time.

Option 1 is simpler and safer; it makes the commitment gate a strict subset of what the execution gate can service.

---

### Proof of Concept

1. Protocol state: 900 ETH staked in EigenLayer, 100 ETH in `LRTUnstakingVault`. `assetsCommitted[ETH] = 0`.
2. `getTotalAssetDeposits(ETH)` returns 1000 ETH. `getAvailableAssetAmount(ETH)` returns 1000 ETH.
3. User calls `initiateWithdrawal(ETH, X_rsETH)` where `expectedAssetAmount = 1000 ETH`. Check passes (`1000 <= 1000`). User's rsETH is transferred to `LRTWithdrawalManager`. `assetsCommitted[ETH] = 1000`.
4. Operator calls `unlockQueue(ETH, ...)`. `_createUnlockParams` sets `totalAvailableAssets = unstakingVault.balanceOf(ETH) = 100 ETH`.
5. Inside `_unlockWithdrawalRequests`: `payoutAmount ≈ 1000 ETH`, `availableAssetAmount = 100 ETH`. Line 800 fires: `break`. Zero withdrawals are unlocked.
6. User's rsETH remains locked in `LRTWithdrawalManager` with no recourse.
7. Operator must initiate EigenLayer unstaking for 900 ETH, wait ~7 days, complete unstaking, then call `unlockQueue` again — only then is the withdrawal unlocked. [7](#0-6) [8](#0-7)

        ILRTDepositPool lrtDepositPool = ILRTDepositPool(lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL));
        uint256 totalAssets = lrtDepositPool.getTotalAssetDeposits(asset);
        availableAssetAmount = totalAssets > assetsCommitted[asset] ? totalAssets - assetsCommitted[asset] : 0;
    }
