### Title
No Cancel Mechanism for Pending Withdrawal Requests Causes Temporary Fund Freeze - (File: contracts/LRTWithdrawalManager.sol)

### Summary
After a user calls `initiateWithdrawal()`, their rsETH is immediately transferred into `LRTWithdrawalManager` and placed in a locked queue. There is no `cancelWithdrawal()` function anywhere in the contract. The user cannot recover their rsETH until (1) an operator calls `unlockQueue()` and (2) the `withdrawalDelayBlocks` period elapses. During this entire window the user's rsETH is frozen with no exit path, directly analogous to the Y2K epoch-start fund freeze.

### Finding Description
`LRTWithdrawalManager.initiateWithdrawal()` immediately pulls rsETH from the caller into the contract:

```solidity
IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);
``` [1](#0-0) 

The request is then stored in a locked queue via `_addUserWithdrawalRequest()`: [2](#0-1) 

To ever reach `completeWithdrawal()`, two independent conditions must both be satisfied:

**Condition 1 – Operator gate**: An account holding `ASSET_TRANSFER_ROLE` or `OPERATOR_ROLE` must call `unlockQueue()`. This is a privileged, off-chain-triggered action with no deadline. [3](#0-2) 

**Condition 2 – Block delay**: Even after the operator acts, `_processWithdrawalCompletion()` enforces a hard block-number delay (default ≈ 8 days):

```solidity
if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();
``` [4](#0-3) 

The default `withdrawalDelayBlocks` is set at initialization to `8 days / 12 seconds` ≈ 57,600 blocks: [5](#0-4) 

A search of the entire contract reveals **no `cancelWithdrawal()` or equivalent function**. The `instantWithdrawal()` path burns rsETH from `msg.sender`'s wallet balance — it cannot touch rsETH already held by the contract on behalf of a pending request: [6](#0-5) 

Once `initiateWithdrawal()` is called, the user has zero on-chain recourse to recover their rsETH before both conditions are met.

### Impact Explanation
**Medium — Temporary freezing of funds.**

Every user who calls `initiateWithdrawal()` is subject to a mandatory minimum freeze of `withdrawalDelayBlocks` (≈ 8 days) with no cancel option. If the operator is slow, paused, or the contract itself is paused (`whenNotPaused` guards both `initiateWithdrawal` and `completeWithdrawal`), the freeze extends indefinitely. During this window the user cannot:
- Cancel and reclaim their rsETH.
- Use `instantWithdrawal()` (rsETH is no longer in their wallet).
- Transfer or trade the locked rsETH.

If the rsETH/asset exchange rate moves adversely during the lock, the user is stuck in a losing position with no exit — the exact harm described in the reference report.

### Likelihood Explanation
**High.** The freeze is not an edge case; it is the normal path for every withdrawal. The 8-day default delay is long enough for material price movements. The operator dependency adds a second, unbounded dimension to the freeze. Any user who initiates a withdrawal and later wants to reverse course (e.g., due to a price spike in rsETH, a personal liquidity need, or a protocol event) discovers there is no exit.

### Recommendation
Add a `cancelWithdrawal(address asset)` function that:
1. Is callable only while the request is still in the **locked** state (i.e., `usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]`).
2. Removes the request from `userAssociatedNonces` and `withdrawalRequests`.
3. Decrements `assetsCommitted[asset]` by `request.expectedAssetAmount`.
4. Returns the original `rsETHUnstaked` amount to `msg.sender`.

This mirrors the standard pattern of allowing depositors to exit before a state transition they cannot control.

### Proof of Concept
1. Alice calls `initiateWithdrawal(stETH, 10e18, "")`. Her 10 rsETH is transferred to `LRTWithdrawalManager`.
2. The rsETH price rises 5 % the next day. Alice wants to cancel and sell her rsETH at the higher price.
3. Alice searches for a cancel function — none exists.
4. `instantWithdrawal()` is unavailable: Alice's rsETH is held by the contract, not her wallet.
5. Alice must wait for the operator to call `unlockQueue()` **and** for ≈ 57,600 blocks (≈ 8 days) to pass before she can call `completeWithdrawal()`.
6. During the entire 8-day window, Alice's 10 rsETH is frozen with no on-chain exit path. [7](#0-6) [8](#0-7)

    function _processWithdrawalCompletion(address asset, address user, string calldata referralId) internal {
        if (userAssociatedNonces[asset][user].empty()) {
            revert NoWithdrawalRequests(user, asset);
        }

        // Retrieve and remove the oldest withdrawal request for the user.
        uint256 usersFirstWithdrawalRequestNonce = userAssociatedNonces[asset][user].popFront();
        // Ensure the request is already unlocked.
        if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();

        bytes32 requestId = getRequestId(asset, usersFirstWithdrawalRequestNonce);
        WithdrawalRequest memory request = withdrawalRequests[requestId];

        delete withdrawalRequests[requestId];

        // Check that the withdrawal delay has passed since the request's initiation.
        if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();
```

    function _addUserWithdrawalRequest(address asset, uint256 rsETHUnstaked, uint256 expectedAssetAmount) internal {
        uint256 nextUnusedNonce_ = nextUnusedNonce[asset];

        // Generate a unique identifier for the new withdrawal request.
        bytes32 requestId = getRequestId(asset, nextUnusedNonce_);

        // Create and store the new withdrawal request.
        withdrawalRequests[requestId] = WithdrawalRequest({
            rsETHUnstaked: rsETHUnstaked, expectedAssetAmount: expectedAssetAmount, withdrawalStartBlock: block.number
        });

        // Map the user to the newly created request index and increment the nonce for future requests.
        userAssociatedNonces[asset][msg.sender].pushBack(nextUnusedNonce_);
        nextUnusedNonce[asset] = nextUnusedNonce_ + 1;

        emit AssetWithdrawalQueued(msg.sender, asset, rsETHUnstaked, nextUnusedNonce_);
```
