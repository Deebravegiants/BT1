### Title
Unrestricted `sendFunds()` Allows Anyone to Manipulate MEV Reward Distribution Timing to Steal Yield from Existing rsETH Holders - (File: contracts/FeeReceiver.sol)

### Summary
`FeeReceiver.sendFunds()` has no access control, allowing any external caller to trigger the transfer of accumulated MEV/fee rewards into the deposit pool at an arbitrary time. An attacker can exploit this by depositing a large amount of ETH immediately before calling `sendFunds()`, diluting the MEV rewards that should accrue exclusively to pre-existing rsETH holders, and then withdrawing at the inflated price to capture yield they did not earn.

### Finding Description
`FeeReceiver.sol` accumulates ETH from MEV rewards and protocol fees. The `sendFunds()` function is declared `external` with no role check or access modifier:

```solidity
/// @dev send all rewards to deposit pool
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
``` [1](#0-0) 

`receiveFromRewardReceiver` on the deposit pool adds ETH to the protocol's TVL without minting new rsETH, which increases the rsETH/ETH exchange rate. Because `sendFunds()` is callable by anyone, an attacker can choose exactly when this price-increasing event occurs.

The admin-gated functions in the same contract (`setDepositPool`) correctly use `onlyRole(LRTConstants.MANAGER)`, confirming that the absence of a modifier on `sendFunds()` is the anomaly: [2](#0-1) 

### Impact Explanation
**High — Theft of unclaimed yield.**

Concrete scenario with numbers:
1. Protocol state: 100 ETH in deposit pool, 100 rsETH outstanding → price = 1.00 ETH/rsETH. FeeReceiver holds 10 ETH in accumulated MEV rewards (earned by existing stakers).
2. Attacker deposits 100 ETH → receives 100 rsETH at price 1.00. Pool now: 200 ETH, 200 rsETH.
3. Attacker calls `sendFunds()` → 10 ETH pushed into deposit pool. Pool now: 210 ETH, 200 rsETH → price = 1.05 ETH/rsETH.
4. Attacker withdraws 100 rsETH → receives 105 ETH. Net gain: **5 ETH** stolen from pre-existing holders.
5. Pre-existing holders who earned the 10 ETH of MEV rewards now only receive 5 ETH of benefit instead of 10 ETH — a 50% loss of their unclaimed yield.

The attacker's profit scales with the ratio of their deposit to the existing TVL and the size of the accumulated reward balance.

### Likelihood Explanation
**Medium-High.** The attack requires no special permissions, no flash loan (the attacker simply deposits their own capital), and no complex setup. The only cost is the capital required for the deposit and the gas for two transactions. MEV rewards accumulate continuously, so the opportunity is always present. The attacker can monitor the FeeReceiver balance on-chain and execute when the reward balance is large enough to be profitable relative to their deposit size.

### Recommendation
Restrict `sendFunds()` to a trusted role (e.g., `MANAGER` or `OPERATOR`), or implement a time-lock / minimum interval between calls so that the distribution timing cannot be gamed. Alternatively, snapshot rsETH balances at the time rewards are earned (rather than at distribution time) to prevent late depositors from capturing pre-existing rewards.

### Proof of Concept
```
// 1. Observe FeeReceiver.balance = R (large accumulated MEV reward)
// 2. Attacker calls LRTDepositPool.depositETH{value: D}()
//    → receives rsETH at current price P (which does NOT yet include R)
// 3. Attacker calls FeeReceiver.sendFunds()
//    → R ETH pushed to deposit pool; rsETH price rises to P' > P
// 4. Attacker initiates withdrawal of their rsETH
//    → receives D * (P'/P) ETH, pocketing D*(P'-P)/P ETH of yield
//    → yield stolen = R * D / (TVL + D), at the expense of pre-existing holders
```

The entry path is fully permissionless: `FeeReceiver.sendFunds()` at [1](#0-0)  requires no role, no signature, and no precondition beyond having ETH to deposit.
