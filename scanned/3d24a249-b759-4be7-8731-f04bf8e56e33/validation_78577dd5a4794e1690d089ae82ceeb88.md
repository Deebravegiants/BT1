### Title
`CrossChainRateReceiver.getRate()` Returns Arbitrarily Stale Rate Used for wrsETH Minting in Pool Deposits - (File: contracts/cross-chain/CrossChainRateReceiver.sol)

### Summary
`CrossChainRateReceiver` stores a `rate` variable that is updated only when a LayerZero cross-chain message is received. The `getRate()` function returns this stored value with no staleness check against `lastUpdated`. All L2 pool contracts use this oracle as `rsETHOracle` to compute how much wrsETH to mint per ETH deposited. When the stored rate lags behind the true on-chain rsETH/ETH rate, depositors receive more wrsETH than they are entitled to, diluting existing holders.

### Finding Description
`CrossChainRateReceiver` tracks two state variables: `rate` (the rsETH/ETH exchange rate) and `lastUpdated` (the timestamp of the last LayerZero update). [1](#0-0) 

The `getRate()` view function returns `rate` unconditionally: [2](#0-1) 

`lastUpdated` is recorded on every `lzReceive()` call but is never consulted when `getRate()` is called: [3](#0-2) 

`RSETHRateReceiver` is the concrete implementation deployed on L2 chains and is set as `rsETHOracle` in every pool variant: [4](#0-3) 

Every pool's `getRate()` delegates directly to this oracle: [5](#0-4) 

The minting formula in `viewSwapRsETHAmountAndFee` divides by this rate: [6](#0-5) 

The same pattern appears in `RSETHPoolV2`, `RSETHPoolV2ExternalBridge`, `RSETHPoolV3ExternalBridge`, and `RSETHPoolNoWrapper`. [7](#0-6) 

Because rsETH/ETH is a monotonically increasing rate (staking rewards accrue), a stale `rate` is always lower than the true current rate. A lower denominator in `amountAfterFee * 1e18 / rsETHToETHrate` produces a larger `rsETHAmount`, so the depositor receives more wrsETH than the ETH they deposited is actually worth at the current rate.

### Impact Explanation
**High — Theft of unclaimed yield.**

Existing wrsETH holders' backing is diluted. Each wrsETH is redeemable for a pro-rata share of the rsETH held by the wrapper. When new wrsETH is minted at a stale (lower) rate, the new tokens claim a larger share of the pool than the deposited ETH justifies. The difference is extracted from accrued yield that rightfully belongs to prior holders. The magnitude scales with (a) how stale the rate is and (b) the deposit size; both can be large.

### Likelihood Explanation
**Medium.**

The rate is updated only when an off-chain keeper calls `updateRate()` on `RSETHRateProvider`/`RSETHMultiChainRateProvider` and the LayerZero message is delivered on L2. [8](#0-7) 

There is no on-chain enforcement of a minimum update frequency. During periods of high L1 gas costs, keeper inactivity, or LayerZero congestion, the stored rate can lag by hours or days. A sophisticated actor can monitor the divergence between the stale L2 rate and the live mainnet `LRTOracle.rsETHPrice` and time deposits to maximise the overmint. [9](#0-8) 

### Recommendation
Add a configurable `maxRateAge` constant and revert in `getRate()` if `block.timestamp - lastUpdated > maxRateAge`:

```solidity
uint256 public maxRateAge = 24 hours; // configurable by owner

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxRateAge, "Rate is stale");
    return rate;
}
```

This mirrors the fix recommended in the reference report: use the expiry-aware accessor rather than the raw cached value.

### Proof of Concept
1. On mainnet, `LRTOracle.rsETHPrice` is updated to `1.10e18` (reflecting recent staking rewards).
2. The keeper has not yet called `updateRate()` on `RSETHRateProvider`; the L2 `RSETHRateReceiver.rate` still holds `1.05e18` from 48 hours ago.
3. Attacker calls `RSETHPoolV3.deposit{value: 100 ether}("")` on L2.
4. `viewSwapRsETHAmountAndFee(100 ether)` computes `rsETHAmount = 100e18 * 1e18 / 1.05e18 ≈ 95.24 wrsETH`.
5. At the true rate the deposit is worth `100e18 / 1.10e18 ≈ 90.91 wrsETH`.
6. Attacker receives `≈ 4.33 wrsETH` in excess — yield stolen from existing holders — with no special privilege required beyond calling the public `deposit()` function. [10](#0-9) [2](#0-1)
