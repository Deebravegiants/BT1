### Title
`getRsETHAmountToMint`: Division by Zero When `rsETHPrice` Is Uninitialized - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool.getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()` without checking whether the value is zero. Because `rsETHPrice` is a plain storage variable that defaults to `0` and is only populated by a separate `updateRSETHPrice()` call, any deposit attempted before that call reverts with a division-by-zero panic, temporarily freezing all user deposits.

### Finding Description
`LRTDepositPool.getRsETHAmountToMint()` computes the rsETH mint amount as:

```solidity
// LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.rsETHPrice()` reads the public storage variable `rsETHPrice` declared in `LRTOracle.sol`:

```solidity
// LRTOracle.sol L28
uint256 public override rsETHPrice;
```

This variable is never set inside `LRTOracle.initialize()`:

```solidity
// LRTOracle.sol L64-68
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    lrtConfig = ILRTConfig(lrtConfigAddr);
    emit UpdatedLRTConfig(lrtConfigAddr);
}
```

`rsETHPrice` is only written inside `_updateRsETHPrice()`, which is called by the permissionless `updateRSETHPrice()` or the manager-gated `updateRSETHPriceAsManager()`. Until one of those is called, `rsETHPrice == 0`.

`getRsETHAmountToMint()` is called by every deposit path:

- `depositETH()` → `_beforeDeposit()` → `getRsETHAmountToMint()` (L87, L665)
- `depositAsset()` → `_beforeDeposit()` → `getRsETHAmountToMint()` (L111, L665)

Neither `_beforeDeposit()` nor any caller checks whether `rsETHPrice` is zero before the division.

### Impact Explanation
All user-facing deposit functions (`depositETH`, `depositAsset`) revert with a division-by-zero panic whenever `rsETHPrice == 0`. This completely blocks rsETH minting for every depositor until `updateRSETHPrice()` is called. The impact is **Medium: Temporary freezing of funds** — no assets are lost, but the protocol cannot accept deposits during the uninitialized window.

### Likelihood Explanation
The window exists between contract deployment/upgrade and the first successful `updateRSETHPrice()` call. There is no on-chain enforcement that `updateRSETHPrice()` must be called before deposits are opened. A user who deposits in this window (e.g., immediately after a deployment or a re-initialization) will have their transaction revert. The permissionless nature of `updateRSETHPrice()` means any actor can close the window, but the contract itself provides no guarantee.

### Recommendation
Add a zero-check in `getRsETHAmountToMint()` and revert with a descriptive error, or initialize `rsETHPrice` to `1 ether` inside `LRTOracle.initialize()` (matching the behavior already present in `_updateRsETHPrice()` when `rsethSupply == 0`):

```solidity
// Option A – guard in getRsETHAmountToMint
uint256 price = lrtOracle.rsETHPrice();
if (price == 0) revert RSETHPriceNotInitialized();
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / price;

// Option B – initialize in LRTOracle.initialize()
rsETHPrice = 1 ether;
highestRsethPrice = 1 ether;
```

### Proof of Concept
1. Deploy `LRTOracle` and call `initialize()`. `rsETHPrice` is `0`.
2. Deploy `LRTDepositPool` and call `initialize()`. No price update is triggered.
3. A user calls `depositETH{value: 1 ether}(0, "")`.
4. Execution reaches `_beforeDeposit()` → `getRsETHAmountToMint()` → `(1e18 * assetPrice) / 0` → EVM division-by-zero panic, transaction reverts.
5. All deposits are blocked until a separate `updateRSETHPrice()` call sets `rsETHPrice` to a non-zero value.
