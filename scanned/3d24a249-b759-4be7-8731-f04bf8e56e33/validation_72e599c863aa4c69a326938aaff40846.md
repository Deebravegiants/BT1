### Title
Token Decimal Assumption in `viewSwapRsETHAmountAndFee` Causes Incorrect rsETH Minting for Non-18-Decimal Tokens - (File: contracts/pools/RSETHPoolV3.sol)

---

### Summary

`RSETHPoolV3.viewSwapRsETHAmountAndFee(uint256 amount, address token)` computes the rsETH output as `amountAfterFee * tokenToETHRate / rsETHToETHrate` without normalizing `amountAfterFee` to 18 decimals. Both `tokenToETHRate` and `rsETHToETHrate` are 1e18-precision values, so the formula implicitly assumes the deposited token also has 18 decimals. If a supported token has fewer than 18 decimals, depositors receive drastically fewer rsETH than owed; if it has more than 18 decimals, they receive far more, threatening protocol solvency. The same flaw is present in `RSETHPoolV3ExternalBridge` and `RSETHPoolV3WithNativeChainBridge`.

---

### Finding Description

`RSETHPoolV3.viewSwapRsETHAmountAndFee` (token overload) computes:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [1](#0-0) 

`tokenToETHRate` is fetched from `ChainlinkOracleForRSETHPoolCollateral.getRate()`, which normalizes the Chainlink answer to 1e18 precision — i.e., it represents the ETH value of **one token unit expressed in 18-decimal fixed-point**:

```solidity
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
``` [2](#0-1) 

`rsETHToETHrate` is similarly 1e18-precision from `InterimRSETHOracle` or equivalent.

The formula is dimensionally correct **only when `amountAfterFee` is also in 18-decimal units**. For a token with `d` decimals, the correct formula is:

```
rsETHAmount = amountAfterFee * 10^(18-d) * tokenToETHRate / rsETHToETHrate
```

No such normalization exists. The `addSupportedToken` function that registers new tokens performs no decimal check:

```solidity
function addSupportedToken(address token, address oracle) external onlyRole(TIMELOCK_ROLE) {
    ...
    if (IOracle(oracle).getRate() == 0) { revert UnsupportedOracle(); }
    supportedTokenList.push(token);
    supportedTokenOracle[token] = oracle;
    ...
}
``` [3](#0-2) 

The identical flaw exists in `RSETHPoolV3ExternalBridge`: [4](#0-3)

### Impact Explanation

**Underpayment (token decimals < 18, e.g., USDC with 6 decimals):**

- User deposits 1 USDC (`amountAfterFee = 1e6`), `tokenToETHRate ≈ 3e14` (0.0003 ETH/USDC), `rsETHToETHrate ≈ 1.05e18`
- Computed: `1e6 * 3e14 / 1.05e18 ≈ 286` rsETH wei (≈ 2.86e-16 rsETH)
- Correct: `1e18 * 3e14 / 1.05e18 ≈ 2.86e14` rsETH wei (≈ 2.86e-4 rsETH)
- User receives **1e12× fewer rsETH** than owed — a near-total loss of deposited value.

**Overpayment (token decimals > 18):**

- User receives 1e(d-18)× more rsETH than owed, draining the wrapper's pre-minted rsETH supply and causing protocol insolvency.

Impact classification: **Low–Critical** depending on token decimals direction. The underpayment case (most likely with stablecoins) maps to **"contract fails to deliver promised returns"** (Low) or **"temporary freezing of funds"** (Medium). The overpayment case maps to **"protocol insolvency"** (Critical).

---

### Likelihood Explanation

The `addSupportedToken` function is gated by `TIMELOCK_ROLE`, so a non-18-decimal token must first be registered. However:

1. The code is explicitly designed to be extensible to arbitrary ERC20 tokens (the function accepts any `token` address).
2. There is no decimal guard, no documentation warning, and no NatSpec restricting tokens to 18 decimals.
3. Stablecoins (USDC, USDT — 6 decimals) are natural candidates for future supported collateral in an L2 deposit pool.
4. Once added, **any unprivileged depositor** triggers the miscalculation by calling `deposit(token, amount, referralId)`. [5](#0-4) 

---

### Recommendation

Normalize `amountAfterFee` to 18 decimals before applying the rate formula:

```solidity
uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 normalizedAmount = amountAfterFee * 10 ** (18 - tokenDecimals);
rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
```

Alternatively, add a decimal check in `addSupportedToken` that reverts if `IERC20Metadata(token).decimals() != 18`, and document this invariant. Apply the same fix to `RSETHPoolV3ExternalBridge` and `RSETHPoolV3WithNativeChainBridge`.

---

### Proof of Concept

1. Admin (TIMELOCK_ROLE) calls `addSupportedToken(USDC, usdcOracle)` on `RSETHPoolV3`. `usdcOracle.getRate()` returns a non-zero value, so the check passes.
2. Attacker (or innocent user) calls `deposit(USDC, 1_000_000, "ref")` (depositing 1 USDC = 1e6 units).
3. `viewSwapRsETHAmountAndFee(1_000_000, USDC)` executes:
   - `fee = 1_000_000 * feeBps / 10_000` (small)
   - `amountAfterFee ≈ 1_000_000`
   - `tokenToETHRate = usdcOracle.getRate()` → e.g., `3e14` (0.0003 ETH per USDC, 1e18-normalized)
   - `rsETHToETHrate = getRate()` → e.g., `1.05e18`
   - `rsETHAmount = 1_000_000 * 3e14 / 1.05e18 ≈ 286`
4. `wrsETH.mint(msg.sender, 286)` — user receives 286 wei of rsETH instead of ~2.86e14 wei.
5. The 1 USDC is permanently held by the pool; the user has lost essentially all deposited value. [6](#0-5)
