### Title
Missing Cancel Mechanism for Initiated Withdrawal Requests Permanently Freezes User rsETH - (File: contracts/LRTWithdrawalManager.sol)

### Summary
`LRTWithdrawalManager` has no `cancelWithdrawal()` function. Once a user calls `initiateWithdrawal()`, their rsETH is irrevocably transferred into the contract and can only be recovered through the operator-gated `unlockQueue()` path. If that path is unavailable — because the unstaking vault holds insufficient assets, or the operator fails to act — the user's rsETH is permanently frozen with no recourse.

### Finding Description
In `initiateWithdrawal()`, the user's rsETH is pulled into the contract and `assetsCommitted[asset]` is incremented:

```solidity
IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);
...
assetsCommitted[asset] += expectedAssetAmount;
```

Recovery requires a strict three-step sequence:
1. An `ASSET_TRANSFER_ROLE` or `OPERATOR_ROLE` holder calls `unlockQueue()`, which itself requires `params.totalAvailableAssets > 0` in the unstaking vault and prices within caller-supplied bounds.
2. `withdrawalDelayBlocks` must elapse.
3. The user calls `completeWithdrawal()`.

There is no `cancelWithdrawal()` or equivalent escape hatch anywhere in the contract. The `sweepRemainingAssets()` function is manager-only and sends funds to the treasury, not back to users. The contract provides no timeout after which a user may reclaim their rsETH unilaterally.

The `_unlockWithdrawalRequests` loop breaks immediately if the vault has insufficient assets:

```solidity
if (availableAssetAmount < payoutAmount) break;
```

meaning a single under-funded request blocks all subsequent requests in the queue. Combined with the absence of a cancel path, any user whose request sits at or behind such a blocking position has their rsETH frozen indefinitely.

### Impact Explanation
User rsETH transferred into `LRTWithdrawalManager` via `initiateWithdrawal()` is permanently inaccessible to the user if:
- The unstaking vault never accumulates sufficient assets (e.g., EigenLayer slashing, delayed unstaking), or
- The operator role is unavailable (key loss, protocol wind-down, prolonged outage).

Because rsETH is a yield-bearing receipt token, locking it also forfeits any accrued yield for the duration of the freeze. In the worst case (operator permanently unavailable), this constitutes permanent freezing of user principal — a Critical impact. In the expected case (operator eventually acts), it is a temporary freeze — a Medium impact.

### Likelihood Explanation
Any user who calls `initiateWithdrawal()` is immediately exposed. The freeze materialises whenever:
- The unstaking vault balance is zero or below the committed threshold (a normal transient state during EigenLayer withdrawal delays), or
- The operator is slow, misconfigured, or permanently unavailable.

No special permissions or external conditions beyond normal protocol operation are required to enter the frozen state. The absence of a cancel mechanism is a structural design gap, not a conditional one.

### Recommendation
Implement a `cancelWithdrawal()` function that:
1. Verifies the request has not yet been unlocked (`usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]`).
2. Removes the request from `userAssociatedNonces` and `withdrawalRequests`.
3. Decrements `assetsCommitted[asset]` by `request.expectedAssetAmount`.
4. Returns `request.rsETHUnstaked` rsETH to the original requester.

Optionally, enforce a maximum lock duration (e.g., 30 days) after which cancellation is automatically permitted, mirroring the expiry-based cancel pattern recommended in the reference report.

### Proof of Concept
1. Alice calls `initiateWithdrawal(stETH, 10e18, "")`.
   - `10e18` rsETH is transferred from Alice to `LRTWithdrawalManager`. [1](#0-0) 
   - `assetsCommitted[stETH] += expectedAssetAmount`. [2](#0-1) 
2. The `LRTUnstakingVault` holds 0 stETH (EigenLayer withdrawal still pending).
3. Operator calls `unlockQueue(stETH, ...)` → reverts with `AmountMustBeGreaterThanZero`. [3](#0-2) 
4. Alice calls `completeWithdrawal(stETH, "")` → reverts with `WithdrawalLocked` because `nextLockedNonce` was never advanced. [4](#0-3) 
5. There is no `cancelWithdrawal()` function anywhere in the contract. Alice's `10e18` rsETH remains locked indefinitely. [5](#0-4)

    function initiateWithdrawal(
        address asset,
        uint256 rsETHUnstaked,
        string calldata referralId

        external
        override
        nonReentrant
        whenNotPaused
        onlySupportedAsset(asset)

    {
        if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
            revert InvalidAmountToWithdraw();
        }

        IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);

        uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);

        if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();

        // preventing over-withdrawal.
        assetsCommitted[asset] += expectedAssetAmount;

        _addUserWithdrawalRequest(asset, rsETHUnstaked, expectedAssetAmount);

        emit ReferralIdEmitted(referralId);
    }
