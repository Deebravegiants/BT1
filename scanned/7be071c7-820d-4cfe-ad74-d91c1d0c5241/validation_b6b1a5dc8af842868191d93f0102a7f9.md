### Title
Missing Minimum Output Amount (Slippage Protection) in Pool `deposit()` Functions - (File: contracts/pools/RSETHPool.sol, RSETHPoolV2.sol, RSETHPoolV2ExternalBridge.sol, RSETHPoolV2NBA.sol, RSETHPoolV3.sol, RSETHPoolV3ExternalBridge.sol, RSETHPoolV3WithNativeChainBridge.sol, RSETHPoolNoWrapper.sol)

---

### Summary

Every L2 pool `deposit()` function that swaps ETH or a supported token for wrsETH/rsETH computes the output amount from an oracle rate at execution time but provides no parameter for the caller to specify a minimum acceptable output. A user who submits a transaction when the oracle rate is favorable may receive materially fewer rsETH tokens if the rate moves before the transaction is mined. The `LRTDepositPool` on L1 already enforces this protection via `minRSETHAmountExpected`, but none of the L2 pool contracts do.

---

### Finding Description

All L2 pool contracts expose one or two public `deposit()` entry points:

- `deposit(string referralId)` — ETH → wrsETH/rsETH
- `deposit(address token, uint256 amount, string referralId)` — ERC-20 → wrsETH/rsETH

In every case the output amount is computed inside `viewSwapRsETHAmountAndFee`, which reads the live oracle rate at the moment of execution:

```solidity
// RSETHPool.sol (same pattern in all variants)
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount);
``` [1](#0-0) 

```solidity
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();          // live oracle read
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
``` [2](#0-1) 

No `minRsETHAmount` (or equivalent) parameter exists in any of these functions. The same pattern is present in:

- `RSETHPoolV2.sol` `deposit(string)` [3](#0-2)

- `RSETHPoolV3ExternalBridge.sol` `deposit(string)` and `deposit(address,uint256,string)` [7](#0-6)

By contrast, `LRTDepositPool` on L1 already enforces this protection:

```solidity

function depositAsset(address asset, uint256 depositAmount, uint256 minRSETHAmountExpected, ...) external { ... }

    revert MinimumAmountToReceiveNotMet();
}
``` [11](#0-10) 

The L2 pool contracts have no equivalent guard.

---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A user who previews the oracle rate off-chain and submits a `deposit()` transaction may receive fewer wrsETH/rsETH tokens than expected if the oracle rate increases (rsETH appreciates in ETH terms) between submission and execution. The user's ETH/token is consumed but the rsETH minted is worth less than the user anticipated at submission time. The user has no on-chain mechanism to reject the transaction if the output falls below their acceptable threshold.

---

### Likelihood Explanation

The rsETH/ETH oracle rate changes continuously as staking rewards accrue and as the underlying LST basket rebalances. On congested L2s, transactions can sit in the mempool for multiple blocks, during which the rate can shift. The entry path is fully permissionless — any user calling `deposit()` is exposed. No admin compromise or oracle manipulation is required; ordinary rate drift is sufficient.

---

### Recommendation

Add a `minRsETHAmount` parameter to every public `deposit()` function in all pool contracts and revert if the computed output falls below it, mirroring the pattern already used in `LRTDepositPool._beforeDeposit()`:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmount)
    external payable nonReentrant whenNotPaused
{
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();

    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmount) revert MinimumAmountToReceiveNotMet();

    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
```

Apply the same pattern to the token-deposit overload and to all pool contract variants.

---

### Proof of Concept

1. Alice calls `RSETHPoolV3.deposit{value: 10 ether}("ref")`. At the time she signs the transaction, `getRate()` returns `1.05e18` (1 rsETH = 1.05 ETH), so she expects ≈ `9.52 wrsETH` (after fee).
2. Before her transaction is mined, the oracle rate updates to `1.10e18` (rsETH appreciated).
3. Her transaction executes: `rsETHAmount = (10e18 - fee) * 1e18 / 1.10e18 ≈ 9.09 wrsETH`.
4. Alice receives ~4.5% fewer wrsETH tokens than she expected with no ability to revert the transaction.
5. Because no `minRsETHAmount` parameter exists, Alice had no on-chain protection against this outcome. [12](#0-11) [13](#0-12)
