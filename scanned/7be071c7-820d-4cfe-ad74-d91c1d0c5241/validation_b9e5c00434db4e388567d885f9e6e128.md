### Title
Stale Chainlink Price Accepted in `getAssetPrice` Enables rsETH Over-Minting via Cross-Asset Price Skew - (File: contracts/oracles/ChainlinkPriceOracle.sol)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards every field except `answer`. Neither `updatedAt` nor `answeredInRound` is validated. Because `updateRSETHPrice()` is public and the stored `rsETHPrice` is the denominator used when minting rsETH for depositors, a natural Chainlink staleness event on any one supported LST feed is sufficient for an unprivileged depositor to mint rsETH at a deflated price and extract value from existing holders.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` silently accepts whatever `answer` Chainlink returns, regardless of how old it is:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

The five return values of `latestRoundData` are `(roundId, answer, startedAt, updatedAt, answeredInRound)`. The code binds only `price` (the second slot); `updatedAt` and `answeredInRound` are never read, so there is no heartbeat check and no round-completeness check.

This price is consumed by `LRTOracle._getTotalEthInProtocol()`, which sums `assetAmount × assetPrice` across every supported LST to compute the protocol's total ETH value:

```solidity
// contracts/LRTOracle.sol L339
uint256 assetER = getAssetPrice(asset);
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
``` [2](#0-1) 

That total feeds `_updateRsETHPrice()`, which writes the new `rsETHPrice` to storage. `updateRSETHPrice()` is unrestricted — any address may call it:

```solidity
// contracts/LRTOracle.sol L87
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [3](#0-2) 

The stored `rsETHPrice` is then used as the denominator when computing how many rsETH tokens a depositor receives:

```solidity
// contracts/LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [4](#0-3) 

The numerator (`lrtOracle.getAssetPrice(asset)`) is a live call that also goes through the same unchecked `ChainlinkPriceOracle`, so when a single asset's feed goes stale the two sides of the fraction diverge:

- **Stale feed asset (e.g. stETH/ETH):** its stale-low price deflates `totalETHInProtocol`, which deflates `rsETHPrice` stored in `LRTOracle`.
- **Fresh feed asset (e.g. cbETH/ETH):** its live price is unaffected.

An attacker who deposits the fresh-feed asset now computes:

```
rsethAmountToMint = (cbETH_amount × P_cbETH_live) / rsETHPrice_deflated
```

Because `rsETHPrice_deflated < rsETHPrice_real`, the attacker receives more rsETH than the deposited value warrants, diluting every existing rsETH holder.

By contrast, `ChainlinkOracleForRSETHPoolCollateral` — a different oracle wrapper in the same repo — already performs partial staleness checks (`answeredInRound < roundID`, `timestamp == 0`, `ethPrice <= 0`), confirming the team is aware of the pattern but did not apply it to `ChainlinkPriceOracle`: [5](#0-4) 

---

### Impact Explanation

Existing rsETH holders suffer dilution — their proportional claim on the underlying ETH pool shrinks without receiving any compensation. This constitutes direct, at-rest fund loss for every holder at the time of the exploit. The magnitude scales with (a) the size of the stale asset's TVL share and (b) the degree of price deviation before the feed recovers.

**Impact: Critical — direct theft of user funds (dilution of rsETH holders' underlying ETH claim).**

---

### Likelihood Explanation

Chainlink feeds have historically gone stale during periods of network congestion or node operator issues. The protocol supports multiple LST assets, each with its own feed; the probability that at least one feed lags during a high-congestion period is non-trivial. The attacker requires no special role, no front-running, and no oracle manipulation — only the ability to observe on-chain that one feed's `updatedAt` is old and to call two public functions (`updateRSETHPrice` then `depositAsset`).

**Likelihood: Low-to-Medium** (depends on network conditions; no attacker-controlled precondition beyond timing).

---

### Recommendation

1. **Add staleness and round-completeness checks in `ChainlinkPriceOracle.getAssetPrice()`**, mirroring the pattern already present in `ChainlinkOracleForRSETHPoolCollateral`, and additionally enforce a configurable heartbeat window:

```solidity
(uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound)
    = priceFeed.latestRoundData();

require(answeredInRound >= roundId,  "Stale price: incomplete round");
require(updatedAt != 0,              "Stale price: round not complete");
require(block.timestamp - updatedAt <= maxStaleness, "Stale price: heartbeat exceeded");
require(price > 0,                   "Invalid price");
```

2. **Store a per-feed `maxStaleness` value** (e.g. 3600 s for a 1-hour heartbeat feed) set by the LRT manager alongside `updatePriceFeedFor`.

3. **Apply the same fix to `RSETHPriceFeed.latestRoundData()`**, which also forwards the raw ETH/USD answer without staleness validation. [6](#0-5)

### Proof of Concept

**Setup:** Protocol holds 1 000 stETH (stETH/ETH feed = 0.999) and 1 000 cbETH (cbETH/ETH feed = 1.05). rsETH supply = 2 049 rsETH. Real `rsETHPrice` ≈ 1.024 ETH.

**Step 1 — Feed goes stale.** The stETH/ETH Chainlink feed stops updating (network congestion). Its last reported price was 0.999 but the real price has moved to 1.001. The stale answer remains 0.999.

**Step 2 — Attacker calls `updateRSETHPrice()` (public, no role required).**
- `_getTotalEthInProtocol()` computes: `1000 × 0.999 + 1000 × 1.05 = 2049 ETH` (stale stETH price used).
- Suppose real total is `1000 × 1.001 + 1000 × 1.05 = 2051 ETH`.
- `rsETHPrice` is written as `2049 / 2049 = 1.000 ETH` instead of the real `2051 / 2049 ≈ 1.001 ETH`.

**Step 3 — Attacker calls `depositAsset(cbETH, 100e18, ...)`.**
- `rsethAmountToMint = (100 × 1.05) / 1.000 = 105 rsETH`
- Correct amount would be `(100 × 1.05) / 1.001 ≈ 104.9 rsETH`
- Attacker receives ~0.1 rsETH excess per 100 cbETH deposited.

**Step 4 — Feed recovers.** `updateRSETHPrice()` is called again with correct prices. `rsETHPrice` rises back to ~1.001. The attacker's 105 rsETH is now worth more ETH than they deposited, at the expense of all prior holders.

The entry path is entirely unprivileged: `depositAsset` → `_beforeDeposit` → `getRsETHAmountToMint` → `lrtOracle.getAssetPrice` → `ChainlinkPriceOracle.getAssetPrice` → unchecked `latestRoundData`. [1](#0-0) [7](#0-6) [8](#0-7)
