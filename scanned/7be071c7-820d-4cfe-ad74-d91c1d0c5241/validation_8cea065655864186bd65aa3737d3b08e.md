### Title
ETH Deposit Limit Check Ignores Deposit Amount, Allowing Unlimited Cap Bypass - (File: contracts/LRTDepositPool.sol)

---

### Summary

`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric check for ETH deposits: it only tests whether the *current* total already exceeds the configured limit, without adding the incoming deposit amount. Any unprivileged depositor can bypass the ETH deposit cap entirely as long as the running total has not yet crossed the limit, regardless of how large their deposit is.

---

### Finding Description

In `LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` the ETH branch and the ERC-20 branch are structurally different:

```solidity
// contracts/LRTDepositPool.sol  lines 676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount NOT included
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← amount included
}
```

For ETH the function returns `totalAssetDeposits > limit`, which is `false` whenever the current total is still below the limit — regardless of how large `amount` is. For every ERC-20 asset the function correctly adds `amount` before comparing.

This function is the sole deposit-limit guard called from `_beforeDeposit`, which is invoked by the public `depositETH` entry point:

```solidity
// contracts/LRTDepositPool.sol  lines 648-670
function _beforeDeposit(address asset, uint256 depositAmount, uint256 minRSETHAmountExpected)
    private view returns (uint256 rsethAmountToMint)
{
    ...
    if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
        revert MaximumDepositLimitReached();
    }
    ...
}
```

The structural analogy to the reported Lyra issue is exact: the Lyra `canHedge()` check only inspects the *sign* (direction) of the expected hedge without accounting for the *magnitude* of the trade, so a sufficiently large trade in the "right" direction still increases risk. Here, the ETH limit check only inspects whether the *current* total has already crossed the boundary, without accounting for the *magnitude* of the incoming deposit, so a sufficiently large deposit in the "allowed" direction still blows past the cap.

---

### Impact Explanation

The deposit limit is a protocol-level safety mechanism that bounds total ETH exposure (EigenLayer capacity, slashing risk, liquidity constraints). Bypassing it allows the protocol to accumulate ETH far beyond the intended ceiling. No direct fund theft occurs, but the protocol fails to enforce its promised deposit constraint for ETH, and excess deposits increase systemic risk (e.g., inability to fully stake or withdraw if EigenLayer capacity is exceeded).

**Impact: Low — Contract fails to deliver promised returns / enforce promised constraints, but does not directly lose value.**

---

### Likelihood Explanation

The entry path is fully permissionless: any address with sufficient ETH can call `depositETH`. No role, whitelist, or special condition is required. The only prerequisite is that `totalAssetDeposits(ETH_TOKEN)` has not yet crossed the limit — a normal operating condition. The bug is therefore trivially reachable by any depositor at any time before the cap is naturally hit.

---

### Recommendation

Add the deposit amount to the ETH branch, consistent with the ERC-20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

---

### Proof of Concept

1. Assume `depositLimitByAsset(ETH_TOKEN) = 1 000 ETH` and `getTotalAssetDeposits(ETH_TOKEN) = 999 ETH`.
2. Attacker calls `depositETH{value: 10_000 ether}(0, "")`.
3. Inside `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 10_000 ether)`:
   - `totalAssetDeposits = 999 ETH`
   - ETH branch evaluates `999 > 1000` → `false`
   - Function returns `false` → no revert.
4. `_mintRsETH` executes; 10 000 ETH is accepted, bringing the total to 10 999 ETH — **11× the intended cap**.
5. The deposit limit is completely bypassed for ETH while the identical deposit in any supported ERC-20 would have been correctly rejected. [1](#0-0) [2](#0-1) [3](#0-2)
