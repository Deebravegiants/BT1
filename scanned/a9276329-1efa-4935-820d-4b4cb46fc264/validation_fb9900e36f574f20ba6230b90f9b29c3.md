### Title
Token Amount Not Normalized to 18 Decimals Before rsETH Mint Calculation - (File: contracts/LRTDepositPool.sol, contracts/LRTOracle.sol)

### Summary
`LRTDepositPool.getRsETHAmountToMint()` and `LRTOracle._getTotalEthInProtocol()` both use raw token balances in native decimals directly in 1e18-precision arithmetic without normalizing for tokens with fewer than 18 decimals. If any supported asset has non-18 decimals, depositors receive a drastically wrong rsETH amount and the protocol-wide rsETH price is severely miscalculated.

### Finding Description

**Root cause 1 — `LRTDepositPool.getRsETHAmountToMint`:**

```solidity
// contracts/LRTDepositPool.sol line 520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`amount` is the raw deposit quantity in native token decimals. `getAssetPrice` returns a 1e18-scaled price. `rsETHPrice` is 1e18-scaled. For an 18-decimal token the arithmetic cancels correctly. For a 6-decimal token (e.g. USDC), `amount` is 10¹² times smaller than its 1e18-equivalent, so `rsethAmountToMint` is 10¹² times smaller than it should be — the depositor is robbed of virtually all their rsETH. [1](#0-0) 

**Root cause 2 — `LRTOracle._getTotalEthInProtocol`:**

```solidity
// contracts/LRTOracle.sol lines 340-343
// totalAssetAmt is in 1e18 precision (standard token decimals)  ← comment is wrong
uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
```

`getTotalAssetDeposits` returns raw `balanceOf` sums in native token decimals. `mulWad` computes `a * b / 1e18`. For a 6-decimal token this underestimates the ETH value by 10¹², collapsing the computed TVL and therefore the rsETH price toward zero. [2](#0-1) 

**Root cause 3 — `RSETHPoolV3.viewSwapRsETHAmountAndFee` (token overload):**

```solidity
// contracts/pools/RSETHPoolV3.sol line 334
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

`amountAfterFee` is in native token decimals; both rates are 1e18-scaled. For a non-18-decimal token the result is off by `10^(18 - decimals)`. This view function drives the actual `deposit(token)` execution path via the `limitDailyMint` modifier and the mint call. [3](#0-2) 

The same pattern is present in `RSETHPoolNoWrapper`, `RSETHPoolV3ExternalBridge`, and `RSETHPoolV3WithNativeChainBridge`.

### Impact Explanation

If any supported asset has fewer than 18 decimals:

- **Depositors via `LRTDepositPool.depositAsset`** receive `10^(18-decimals)` times fewer rsETH than they are entitled to — direct, permanent loss of deposited value.
- **rsETH price** computed by `_getTotalEthInProtocol` is `10^(18-decimals)` times smaller than reality, corrupting every subsequent mint, withdrawal payout, and fee calculation across the entire protocol — protocol insolvency.
- **L2 pool depositors** via `RSETHPoolV3.deposit(token)` receive the same drastically reduced wrsETH amount.

Impact: **Critical** — direct theft of user funds and protocol insolvency.

### Likelihood Explanation

The protocol's current supported assets (stETH, rETH, cbETH, native ETH) are all 18-decimal, so the bug is dormant today. However:

- `LRTConfig` has no decimal guard when adding a new supported asset.
- The L2 pool contracts expose `addSupportedToken(token, oracle)` with no decimal check.
- There is no on-chain invariant preventing a 6- or 8-decimal token from being added.

A single governance transaction adding a non-18-decimal collateral (e.g. USDC, WBTC) activates the vulnerability for every subsequent depositor. Likelihood: **Low** (requires a governance action), but the action is routine and the consequence is catastrophic.

### Recommendation

Normalize token amounts to 18-decimal precision before any arithmetic. Introduce a helper analogous to ZAROS's `convertTokenAmountToUd60x18`:

```solidity
function _toWad(address asset, uint256 amount) internal view returns (uint256) {
    uint8 decimals = IERC20Metadata(asset).decimals();
    if (decimals == 18) return amount;
    return amount * 10 ** (18 - decimals);
}
```

Apply it in:

- `getRsETHAmountToMint`: `rsethAmountToMint = (_toWad(asset, amount) * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();`
- `_getTotalEthInProtocol`: `totalETHInProtocol += _toWad(asset, totalAssetAmt).mulWad(assetER);`
- `viewSwapRsETHAmountAndFee(amount, token)`: normalize `amountAfterFee` before the division.

Alternatively, enforce at asset-registration time that only 18-decimal tokens may be added as supported assets.

### Proof of Concept

Assume USDC (6 decimals, price = 1e18 / 3000 ≈ 3.33e14 ETH per USDC) is added as a supported asset and rsETHPrice = 1e18.

**Deposit 1,000 USDC (= 1,000e6 raw units):**

```
rsethAmountToMint = (1_000e6 * 3.33e14) / 1e18
                  = 3.33e23 / 1e18
                  = 333_000          // ~333,000 wei of rsETH
```

Correct value should be:
```
rsethAmountToMint = (1_000e18 * 3.33e14) / 1e18
                  = 333_000e18       // ~333,000 rsETH
```

The depositor receives `10^12` times fewer rsETH than owed — 1,000 USDC worth of value is effectively stolen by the protocol.

**TVL miscalculation:** With 1,000,000 USDC deposited, `_getTotalEthInProtocol` computes `1_000_000e6 * 3.33e14 / 1e18 = 333e6` wei ≈ 0.000000333 ETH instead of the correct ≈ 333 ETH, collapsing rsETH price to near zero and triggering cascading insolvency across all rsETH holders.
