### Title
Missing Slippage Protection in `removeLiquidity` Exposes LPs to Front-Running Composition Loss - (File: metric-core/contracts/MetricOmmPool.sol, metric-core/contracts/libraries/LiquidityLib.sol)

---

### Summary

`MetricOmmPool.removeLiquidity` accepts only shares to burn and extension data. It returns whatever token amounts the current bin state yields at execution time, with no caller-supplied minimum output floor. No periphery wrapper exists to add this guard. An LP's pending withdrawal can be front-run by a swap that shifts the bin cursor into the LP's bin, converting the bin's composition before the LP's transaction settles, causing the LP to receive a materially different (and potentially less valuable) token mix than they submitted for.

---

### Finding Description

`removeLiquidity` computes output amounts purely from live bin state at the moment of execution:

```solidity
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
uint256 amount1Scaled = _checkedMul(binState.token1BalanceScaled, sharesToRemove) / binTotalSharesVal;
``` [1](#0-0) 

These values are then transferred directly to the owner with no minimum-output check:

```solidity
if (amount0Removed > 0) { IERC20(ctx.token0).safeTransfer(owner, amount0Removed); }
if (amount1Removed > 0) { IERC20(ctx.token1).safeTransfer(owner, amount1Removed); }
``` [2](#0-1) 

The pool-level entry point passes no slippage parameters and performs no output validation: [3](#0-2) 

The only periphery liquidity helper, `MetricOmmPoolLiquidityAdder`, exclusively wraps `addLiquidity` and provides no `removeLiquidity` path at all: [4](#0-3) 

By contrast, the swap periphery (`MetricOmmSimpleRouter`) does enforce `amountOutMinimum` / `amountInMaximum` on every swap variant: [5](#0-4) [6](#0-5) 

The gap is therefore structural: swaps are slippage-protected at the periphery; liquidity removal is not protected anywhere.

---

### Impact Explanation

When an LP submits `removeLiquidity` for a bin that currently holds, say, 100% token0 (bin above the cursor), a MEV searcher can front-run with a swap that drives the cursor into that bin. The bin's composition shifts: some token0 is sold out and token1 flows in. The LP's transaction then executes against the modified bin state and receives a mix of token0 and token1 instead of pure token0. Because the oracle-anchored price governs the swap, the total value at oracle price is approximately preserved, but:

1. **Composition mismatch**: The LP receives a token they did not intend to hold, forcing an additional swap (with fees and spread) to rebalance.
2. **Oracle staleness amplifies loss**: If the oracle price is stale relative to the market at execution time, the token1 received is worth less than the token0 that was displaced, producing a direct principal loss.
3. **No recourse**: The LP has no on-chain mechanism to revert the transaction if the output falls below their acceptable threshold.

---

### Likelihood Explanation

MEV front-running of pending liquidity-removal transactions is a well-established attack vector on public mempools. Any LP removing liquidity from a bin near the active cursor is a target. The attacker needs only to submit a swap that moves the cursor into the victim's bin before the removal settles. No special privilege is required; any unprivileged address can call `swap`.

---

### Recommendation

Add `uint256 minAmount0Out` and `uint256 minAmount1Out` parameters to `removeLiquidity` (or to a new periphery wrapper), and revert if the computed outputs fall below those floors:

```solidity
function removeLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    uint256 minAmount0Out,   // NEW
    uint256 minAmount1Out,   // NEW
    bytes calldata extensionData
) external nonReentrant(PoolActions.REMOVE_LIQUIDITY) returns (uint256 amount0Removed, uint256 amount1Removed) {
    // ... existing logic ...
    if (amount0Removed < minAmount0Out || amount1Removed < minAmount1Out) {
        revert InsufficientOutput(amount0Removed, amount1Removed, minAmount0Out, minAmount1Out);
    }
}
```

Alternatively, add a `removeLiquidity` wrapper to `MetricOmmPoolLiquidityAdder` (mirroring the `maxAmountToken0`/`maxAmountToken1` pattern already used for `addLiquidity`) that enforces minimum output amounts after the pool call returns.

---

### Proof of Concept

**Setup**: Pool with bins −1, 0, 1. Oracle mid-price places cursor at bin 0. LP holds 10 000 shares in bin 1 (above cursor → 100% token0, 0 token1).

**Step 1 – LP submits removal**: LP calls `removeLiquidity` expecting to receive ~X token0 and 0 token1.

**Step 2 – Attacker front-runs**: Attacker calls `swap` (token1 → token0, exact-input, large amount) via `MetricOmmSimpleRouter.exactInputSingle`. The swap exhausts bin 0 and partially fills bin 1, moving the cursor into bin 1. Bin 1 now holds, e.g., 60% token0 and 40% token1.

**Step 3 – LP's removal executes**:
```
amount0Scaled = binState.token0BalanceScaled * 10_000 / binTotalSharesVal  // ~60% of original
amount1Scaled = binState.token1BalanceScaled * 10_000 / binTotalSharesVal  // ~40% of original
``` [1](#0-0) 

LP receives 60% of the token0 they expected plus an unwanted token1 position. No revert occurs because there is no minimum-output guard anywhere in the call path. [3](#0-2)
