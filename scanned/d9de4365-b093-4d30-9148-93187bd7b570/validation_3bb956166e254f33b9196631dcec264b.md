### Title
Floor Division in `removeLiquidity` Burns LP Shares and Reduces Bin Balances While Returning Zero Native Tokens - (File: `metric-core/contracts/libraries/LiquidityLib.sol`)

### Summary

`LiquidityLib.removeLiquidity` applies two sequential floor divisions when converting burned shares into native token transfers. For tokens with a large `token0ScaleMultiplier` (e.g., USDC, 6 decimals → multiplier = 10¹²), the scaled balance removed from the bin can be non-zero while the native transfer rounds to zero. The LP's shares are burned and the pool's internal accounting is permanently reduced, but no tokens are sent. The stranded scaled balance becomes an untracked surplus that flows to protocol/admin fee collectors on the next `collectFees` call.

### Finding Description

`removeLiquidity` performs two floor divisions in sequence:

**Step 1 — share-to-scaled (lines 205–206):**

```solidity
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
uint256 amount1Scaled = _checkedMul(binState.token1BalanceScaled, sharesToRemove) / binTotalSharesVal;
```

The bin balance and `binTotalShares` are immediately decremented by these floored values (lines 210–213), and the LP's `positionBinShares` is set to `newUserShares` (line 214). The share burn is unconditional.

**Step 2 — scaled-to-native (lines 239–240 → `_deltasScaledToExternal` lines 274–276):**

```solidity
(amount0Removed, amount1Removed) =
    _deltasScaledToExternal(totalToken0ToRemoveScaled, totalToken1ToRemoveScaled, ctx, Math.Rounding.Floor);
```

```solidity
deltaAmount0 = scaledDeltaAmount0 / ctx.token0ScaleMultiplier;
deltaAmount1 = scaledDeltaAmount1 / ctx.token1ScaleMultiplier;
```

If `totalToken0ToRemoveScaled < token0ScaleMultiplier`, `amount0Removed = 0` and the `safeTransfer` at line 242 is skipped entirely. The pool's `binTotals.scaledToken0` was already decremented (line 232), so the scaled units are permanently orphaned — they no longer appear in bin accounting but remain in the ERC-20 balance, surfacing as surplus in `collectFees`:

```solidity
uint256 surplus0Scaled =
    balance0() * TOKEN_0_SCALE_MULTIPLIER - uint256(binTotals.scaledToken0) - notionalFee0AmountScaled;
```

The LP receives nothing; the protocol/admin receives the LP's tokens as fees.

### Impact Explanation

An LP burns valid shares and has their bin balance permanently reduced, but receives zero native tokens. The lost value is silently redirected to fee collectors. This is a direct, unrecoverable loss of LP principal with no revert or warning.

For a USDC/X pool (`token0ScaleMultiplier = 10¹²`): any removal whose `totalToken0ToRemoveScaled` is in the range `[1, 10¹² − 1]` produces `amount0Removed = 0`. The maximum loss per call is just under 1 USDC. Because `addLiquidity` uses `Math.Rounding.Ceil` for the same conversion, an LP who adds a small position pays at least 1 USDC but recovers 0 USDC on removal — a 100% loss of their deposit.

### Likelihood Explanation

- Any LP interacting with a pool whose token has fewer than 18 decimals (USDC, USDT, WBTC) is exposed.
- The `MINIMAL_MINTABLE_LIQUIDITY` guard (line 200) only prevents leaving a dust *remainder*; it does not prevent removing a quantity of shares whose scaled value is below `token0ScaleMultiplier`.
- No privileged access is required; any LP triggers this by calling `removeLiquidity` with a small share count.
- The asymmetric rounding (`ceilDiv` on add, floor on remove) guarantees the LP always pays at least 1 native unit when adding but may receive 0 when removing.

### Recommendation

1. **Revert on zero-output removal**: After computing `amount0Removed` and `amount1Removed`, revert if either is zero while the corresponding scaled amount is non-zero:
   ```solidity
   if (totalToken0ToRemoveScaled > 0 && amount0Removed == 0) revert ZeroTokenOutput();
   if (totalToken1ToRemoveScaled > 0 && amount1Removed == 0) revert ZeroTokenOutput();
   ```

2. **Alternatively**, expose a `minAmount0Out` / `minAmount1Out` parameter so callers can enforce a minimum and the revert is user-controlled.

3. **For the share-to-scaled step**, consider using `Math.ceilDiv` symmetrically with `addLiquidity` so the LP always recovers at least 1 scaled unit per share removed (at the cost of slightly over-paying the LP, which is the safer direction for LP protection).

### Proof of Concept

**Setup**: USDC/ETH pool, `token0 = USDC` (6 decimals), `token0ScaleMultiplier = 10¹²`, `initialAmount0PerShareE18 = 1e6` (1 USDC per share in native units), `MINIMAL_MINTABLE_LIQUIDITY = 1000`.

**Add liquidity** (1000 shares, empty bin):

```
amount0Scaled = ceilDiv(1e18 × 1000, 1e18) = 1000   // scaled units
amount0Added  = ceilDiv(1000, 1e12)         = 1       // 1 USDC paid by LP
```

**Remove liquidity** (all 1000 shares):

```
amount0Scaled   = 1000 × 1000 / 1000 = 1000          // scaled units, bin balance reduced
amount0Removed  = 1000 / 1e12        = 0              // 0 USDC transferred to LP
```

LP paid 1 USDC, received 0 USDC. The 1000 scaled units are orphaned in the pool's ERC-20 balance and will be swept to fee collectors on the next `collectFees` call. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

```text
  function _deltasScaledToExternal(
    uint256 scaledDeltaAmount0,
    uint256 scaledDeltaAmount1,
    PoolContext memory ctx,
    Math.Rounding rounding
  ) internal pure returns (uint256 deltaAmount0, uint256 deltaAmount1) {
    if (rounding == Math.Rounding.Ceil) {
      deltaAmount0 = Math.ceilDiv(scaledDeltaAmount0, ctx.token0ScaleMultiplier);
      deltaAmount1 = Math.ceilDiv(scaledDeltaAmount1, ctx.token1ScaleMultiplier);
    } else {
      deltaAmount0 = scaledDeltaAmount0 / ctx.token0ScaleMultiplier;
      deltaAmount1 = scaledDeltaAmount1 / ctx.token1ScaleMultiplier;
    }
  }
```
