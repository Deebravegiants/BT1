### Title
`sweepRemainingAssets` Does Not Reset `assetsCommitted` After Sweep, Causing Stale Withdrawal Accounting - (File: contracts/LRTWithdrawalManager.sol)

---

### Summary

`LRTWithdrawalManager.sweepRemainingAssets()` guards only against *unlocked* pending withdrawals via `hasUnlockedWithdrawals(asset)`, but does not check for *locked* (initiated, not yet unlocked) withdrawal requests. After the sweep, `assetsCommitted[asset]` is never decremented, leaving it permanently inflated relative to the actual committed state. This causes `getAvailableAssetAmount()` to underreport available capacity, silently blocking new legitimate withdrawal requests.

---

### Finding Description

`sweepRemainingAssets` is intended to drain residual asset balance from `LRTWithdrawalManager` to the treasury once all withdrawals are complete: [1](#0-0) 

The guard is:

```solidity
if (hasUnlockedWithdrawals(asset)) revert PendingWithdrawalsExist();
```

`hasUnlockedWithdrawals` only checks: [2](#0-1) 

This is `unlockedWithdrawalsCount[asset] > 0`. It says nothing about *locked* withdrawal requests — those created by `initiateWithdrawal` that have not yet been processed by `unlockQueue`.

When a user calls `initiateWithdrawal`, the committed amount is recorded: [3](#0-2) 

This `assetsCommitted[asset]` value is **never touched** inside `sweepRemainingAssets`. After the sweep executes, `assetsCommitted[asset]` retains its pre-sweep value even though the function has just transferred the contract's entire balance to the treasury.

`getAvailableAssetAmount` then computes: [4](#0-3) 

With `assetsCommitted[asset]` inflated and stale, `getAvailableAssetAmount` returns a value lower than the true available capacity. Any subsequent call to `initiateWithdrawal` that checks this value: [5](#0-4) 

will revert with `ExceedAmountToWithdraw` even when protocol assets are genuinely available.

---

### Impact Explanation

New withdrawal requests are incorrectly rejected because `assetsCommitted[asset]` is stale after `sweepRemainingAssets`. The protocol fails to deliver its promised withdrawal service to users who have valid rsETH to redeem. No funds are permanently lost — once the locked requests are eventually unlocked and completed, `assetsCommitted` is decremented back toward zero — but during the interim window the withdrawal queue is artificially blocked.

**Impact: Low** — Contract fails to deliver promised returns (withdrawal initiation) without losing value.

---

### Likelihood Explanation

The scenario requires:
1. Some users have initiated withdrawals (locked, not yet unlocked).
2. All previously unlocked withdrawals have been completed (`unlockedWithdrawalsCount[asset] == 0`).
3. The LRT Manager calls `sweepRemainingAssets`, which passes the guard.

Step 3 is a realistic operational action — the manager sees `hasUnlockedWithdrawals` return `false` and reasonably concludes it is safe to sweep. The function provides no indication that locked requests still exist. This is a plausible good-faith mistake, not an adversarial action.

---

### Recommendation

1. Add a check that no locked withdrawal requests exist before sweeping. Compare `nextUnusedNonce[asset]` against `nextLockedNonce[asset]` to detect pending locked requests:

```solidity
if (nextLockedNonce[asset] < nextUnusedNonce[asset]) revert LockedWithdrawalsPending();
```

2. After sweeping, explicitly reset `assetsCommitted[asset]` to zero (or to the sum of remaining locked requests) so that `getAvailableAssetAmount` reflects the true state.

3. Add NatSpec documentation clarifying that `sweepRemainingAssets` is only safe to call when both unlocked **and** locked withdrawal queues are empty.

---

### Proof of Concept

1. Alice calls `initiateWithdrawal(stETH, 1e18, ...)` → `assetsCommitted[stETH] = X`, `unlockedWithdrawalsCount[stETH] = 0`.
2. Manager calls `sweepRemainingAssets(stETH)`:
   - `hasUnlockedWithdrawals(stETH)` → `false` (passes).
   - Residual stETH balance swept to treasury.
   - `assetsCommitted[stETH]` remains `X`.
3. Bob calls `initiateWithdrawal(stETH, 1e18, ...)`:
   - `getAvailableAssetAmount(stETH)` = `totalAssets - X` (understated).
   - Reverts with `ExceedAmountToWithdraw` even though protocol has sufficient stETH.
4. Alice's locked request is eventually unlocked and completed, decrementing `assetsCommitted[stETH]` — but Bob was blocked during the entire interim window.

```text
    function sweepRemainingAssets(address asset)
        external
        nonReentrant
        onlySupportedAsset(asset)
        onlyLRTManager
        returns (uint256 transferredAmount)
    {
        // Check that all withdrawals are completed
        if (hasUnlockedWithdrawals(asset)) revert PendingWithdrawalsExist();

        uint256 balance = _getAssetBalance(asset);
        if (balance == 0) revert AmountMustBeGreaterThanZero();

        // Transfer to treasury
        address treasury = lrtConfig.getContract(LRTConstants.PROTOCOL_TREASURY);
        _transferAsset(asset, treasury, balance);

        emit RemainingAssetsSwept(asset, balance, treasury);
        return balance;
    }
```
