### Title
Cumulative Aave Scaled-Balance Rounding Permanently Breaks `collectInterestToTreasury` — (`contracts/LRTWithdrawalManager.sol`)

---

### Summary

`_checkAaveHealth` tolerates at most a 2-wei deficit between `totalETHDepositedToAave` and `aaveAWETH.balanceOf(address(this))`. Because Aave v3 uses floor-division when minting scaled aTokens, each deposit can introduce a 1-wei rounding loss. This deficit is never reset by withdrawals. After just 3 deposit cycles that each incur a 1-wei rounding loss, the deficit reaches 3 > 2, `_checkAaveHealth` permanently returns `false`, and the operator-callable `collectInterestToTreasury` permanently reverts with `AaveHealthCheckFailed`.

---

### Finding Description

**Root cause — Aave v3 scaled-balance rounding**

Aave v3 aTokens store a *scaled* balance. On deposit of `amount` wei:

```
scaledMinted = floor(amount × RAY / liquidityIndex)
balanceOf    = floor(scaledMinted × liquidityIndex / RAY)
             ≤ amount   (can be amount − 1 due to floor)
```

So `aaveAWETH.balanceOf(address(this))` can be 1 wei less than the `amount` passed to `depositETH`.

**`_depositToAave` records the full amount, not the actual aToken balance received:**

```solidity
// contracts/LRTWithdrawalManager.sol:897-898
aaveWETHGateway.depositETH{ value: amount }(aavePool, address(this), 0);
totalETHDepositedToAave += amount;   // ← records exact ETH sent, not aToken balance
``` [1](#0-0) 

**`_withdrawFromAave` preserves the deficit across withdrawals:**

```solidity
// contracts/LRTWithdrawalManager.sol:912-918
uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave
    ? aaveBalance : totalETHDepositedToAave;
withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
totalETHDepositedToAave -= withdrawnAmount;
``` [2](#0-1) 

Both `totalETHDepositedToAave` and `aaveBalance` decrease by the same `withdrawnAmount`, so the deficit `totalETHDepositedToAave − aaveBalance` is **unchanged** by any withdrawal. It only grows with each new deposit that incurs rounding.

**`_checkAaveHealth` uses a fixed 2-wei tolerance:**

```solidity
// contracts/LRTWithdrawalManager.sol:931
if (principal > aaveBalance && principal - aaveBalance > 2) return false;
``` [3](#0-2) 

After 3 deposits each incurring a 1-wei rounding loss, `deficit = 3 > 2` and this check permanently returns `false`.

**`collectInterestToTreasury` gates on this check and permanently reverts:**

```solidity
// contracts/LRTWithdrawalManager.sol:542
if (!_checkAaveHealth()) revert AaveHealthCheckFailed();
``` [4](#0-3) 

There is no function that resets or corrects `totalETHDepositedToAave`. Once the deficit exceeds 2 wei, the normal interest-collection path is permanently broken.

**Deposit triggers in normal operation:**

Deposits happen via `unlockQueue` (operator-triggered, calls `depositToAaveExternal`) and `depositIdleETHToAave` (operator-triggered). Neither requires attacker action — this degrades under ordinary protocol operation. [5](#0-4) 

---

### Impact Explanation

The operator-callable `collectInterestToTreasury` permanently reverts after ≥3 deposit cycles with 1-wei rounding. Accrued Aave yield cannot be collected through the intended path. Recovery requires privileged emergency intervention (`emergencyWithdrawFromAave` via PAUSER_ROLE, or `setAaveIntegrationEnabled(false)` via LRT manager), both of which have side effects (withdrawing principal or disabling the integration entirely). This constitutes **permanent freezing of unclaimed yield** under the normal operational path.

---

### Likelihood Explanation

Every `unlockQueue` call for ETH deposits to Aave. The rounding error occurs whenever `amount × RAY mod liquidityIndex ≠ 0`, which is true for the vast majority of deposit amounts. With WETH's liquidityIndex on Aave v3 mainnet at approximately `1.0001×10²⁷`, a 1 ETH deposit yields a 1-wei deficit. After 3 such `unlockQueue` calls — a routine occurrence — the health check permanently fails. No attacker action is required; this is a natural consequence of normal protocol operation.

---

### Recommendation

1. **Track actual aToken balance received, not ETH sent.** In `_depositToAave`, record the post-deposit `aaveAWETH.balanceOf(address(this))` delta instead of `amount`:
   ```solidity
   uint256 balanceBefore = aaveAWETH.balanceOf(address(this));
   aaveWETHGateway.depositETH{ value: amount }(aavePool, address(this), 0);
   totalETHDepositedToAave += aaveAWETH.balanceOf(address(this)) - balanceBefore;
   ```
2. **Alternatively**, remove the health check from `collectInterestToTreasury` entirely (it is already absent from all internal callers) and rely on `_collectInterestToTreasury`'s own guard (`if (aaveBalance <= principal) return 0`), which already handles the rounding case gracefully.
3. **At minimum**, increase the tolerance in `_checkAaveHealth` to a value proportional to the number of expected deposit cycles (e.g., track deposit count and scale tolerance accordingly), or use a relative tolerance.

    function _withdrawFromAave(uint256 amount) internal returns (uint256 withdrawnAmount) {
        if (amount == 0) return 0;

        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        if (aaveBalance == 0) revert InsufficientAaveBalance();

        // Only withdraw up to the principal amount (don't use accrued interest for user withdrawals)
        uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave;

        withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
        if (withdrawnAmount == 0) return 0;

        aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
        totalETHDepositedToAave -= withdrawnAmount;

        emit ETHWithdrawnFromAave(withdrawnAmount, totalETHDepositedToAave);
    }
