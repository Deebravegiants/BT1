### Title
Admin Can Permanently Seize Any User's rsETH Balance via `blockUserTransfers` + `recoverFrozenFunds` — (File: contracts/RSETH.sol)

---

### Summary

The `RSETH` token contract contains a built-in two-step confiscation mechanism: the LRT Manager can freeze any user's rsETH transfers for 24 hours, and the LRT Admin can then forcibly transfer the victim's entire rsETH balance to an admin-controlled custody address. This gives protocol owners absolute, unilateral power to steal any holder's funds with no timelock, no governance process, and no user recourse.

---

### Finding Description

Three functions in `contracts/RSETH.sol` compose the attack surface:

**Step 1 — Freeze the target:**
`blockUserTransfers(address[] calldata accounts)` is callable by any address holding the `MANAGER` role. It sets `transfersBlockedUntil[account] = block.timestamp + 1 days` for each supplied address, which causes `_enforceNotBlocked` to revert on any transfer involving that address. [1](#0-0) 

**Step 2 — Drain the target:**
`recoverFrozenFunds(address from)` is callable by any address holding `DEFAULT_ADMIN_ROLE`. It reads the victim's full balance and calls `super._transfer(from, custodyAddress, accountBalance)`, which bypasses the `_enforceNotBlocked` hook entirely (it calls the parent `ERC20Upgradeable._transfer` directly, skipping the overridden `_transfer`). [2](#0-1) 

**Custody address is admin-controlled:**
`setCustodyAddress` is callable by `onlyLRTAdmin` with no timelock, so the admin can point the custody address to any wallet before executing the drain. [3](#0-2) 

The `_enforceNotBlocked` hook that normally prevents blocked-address transfers is only present in the overridden `_transfer`; `super._transfer` skips it entirely, making the bypass intentional by design. [4](#0-3) 

The `MANAGER` and `DEFAULT_ADMIN_ROLE` are granted through `LRTConfig`, which itself has no timelock on role assignment. [5](#0-4) 

---

### Impact Explanation

**Critical — Direct theft of any user's rsETH funds.**

rsETH represents a claim on restaked ETH and LSTs held across `LRTDepositPool`, `NodeDelegator` contracts, and EigenLayer strategies. Stealing rsETH tokens is economically equivalent to stealing the underlying restaked assets. Any rsETH holder — depositor, withdrawer, LP, or secondary-market buyer — can have their entire balance confiscated in two transactions with no on-chain delay.

---

### Likelihood Explanation

**Medium.** The mechanism requires no external exploit, no oracle manipulation, and no reentrancy. It is a deliberate design choice encoded directly in the token contract. The only prerequisite is that the `MANAGER` and `DEFAULT_ADMIN_ROLE` holders act maliciously or are coerced. There is no timelock, no multi-sig requirement enforced at the contract level, and no user opt-out. The `isPermanentlyExempt` whitelist is also controlled by the manager, so the admin can choose which addresses are protected. [6](#0-5) 

---

### Recommendation

1. **Remove `recoverFrozenFunds`** entirely, or restrict it to a governance-controlled timelock contract with a mandatory delay (e.g., 7 days) and public announcement period.
2. **Separate the freeze and confiscation roles** so that no single key or small set of keys can both freeze and drain in the same time window.
3. **Add a timelock** to `setCustodyAddress` so the destination of any confiscated funds cannot be changed atomically with the confiscation itself.
4. **Emit a public event** at freeze time that gives users a window to exit before confiscation becomes possible.

---

### Proof of Concept

```
// Attacker controls MANAGER and DEFAULT_ADMIN_ROLE (or colludes with holders)

// 1. Point custody to attacker wallet (no timelock)
rsETH.setCustodyAddress(attackerWallet);

// 2. Freeze victim (MANAGER role)
address[] memory victims = new address[](1);
victims[0] = victim;
rsETH.blockUserTransfers(victims);
// transfersBlockedUntil[victim] = block.timestamp + 1 days

// 3. Immediately drain victim (DEFAULT_ADMIN_ROLE)
//    recoverFrozenFunds checks block is active, then calls super._transfer
//    bypassing _enforceNotBlocked on the from address
rsETH.recoverFrozenFunds(victim);
// victim's entire rsETH balance is now in attackerWallet
```

Steps 2 and 3 can be executed in the same block. The victim has no on-chain mechanism to prevent or reverse the confiscation. [2](#0-1) [1](#0-0)

    function blockUserTransfers(address[] calldata accounts) external onlyLRTManager {
        uint256 blockedUntil = block.timestamp + 1 days;
        uint256 length = accounts.length;

        for (uint256 i = 0; i < length; ++i) {
            address account = accounts[i];

            if (isPermanentlyExempt[account] || account == address(0)) continue;

            uint256 prevBlockedUntil = transfersBlockedUntil[account];

            if (blockedUntil != prevBlockedUntil) {
                transfersBlockedUntil[account] = blockedUntil;
                emit UserTransfersBlocked(account, blockedUntil);
            }
        }
    }

    function recoverFrozenFunds(address from) external onlyLRTAdmin {
        UtilLib.checkNonZeroAddress(from);
        UtilLib.checkNonZeroAddress(custodyAddress);

        if (isPermanentlyExempt[from]) revert AddressPermanentlyExempt(from);

        uint256 blockedUntil = transfersBlockedUntil[from];
        if (blockedUntil == 0 || block.timestamp >= blockedUntil) revert NoActiveTransferBlock(from);

        uint256 accountBalance = balanceOf(from);

        // Bypass transfer block enforcement when transferring to custody address
        super._transfer(from, custodyAddress, accountBalance);
        emit FrozenFundsRecovered(from, custodyAddress, accountBalance);
    }
