### Title
Unhandled Chainlink `latestRoundData()` Revert Freezes Deposits, Withdrawals, and Price Updates - (File: contracts/oracles/ChainlinkPriceOracle.sol)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` directly without a `try/catch`. If Chainlink's multisig blocks the feed or the aggregator reverts for any reason, every protocol function that depends on asset pricing — deposits, withdrawals, and rsETH price updates — reverts and becomes inaccessible until an admin manually swaps the oracle.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` performs a bare, unguarded call to `priceFeed.latestRoundData()`:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L49-55
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();   // ← no try/catch
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
``` [1](#0-0) 

If this call reverts, the revert propagates up through the entire call stack:

1. `LRTOracle.getAssetPrice()` → `IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset)` reverts. [2](#0-1) 

2. `LRTOracle._getTotalEthInProtocol()` iterates all supported assets and calls `getAssetPrice(asset)` for each — a single broken feed reverts the entire loop. [3](#0-2) 

3. `LRTOracle._updateRsETHPrice()` calls `_getTotalEthInProtocol()`, so both `updateRSETHPrice()` (public) and `updateRSETHPriceAsManager()` revert. [4](#0-3) 

4. `LRTDepositPool.getRsETHAmountToMint()` calls `lrtOracle.getAssetPrice(asset)`, so `depositAsset()` reverts for every depositor. [5](#0-4) 

5. `LRTWithdrawalManager` also calls `lrtOracle.getAssetPrice(asset)` when computing unlock parameters, freezing withdrawals. [6](#0-5) 

A secondary instance exists in `ChainlinkOracleForRSETHPoolCollateral.getRate()`, which also calls `latestRoundData()` without protection and has an **immutable** `oracle` address — meaning there is no admin path to swap the feed without redeploying the contract. [7](#0-6) 

---

### Impact Explanation

**Medium — Temporary freezing of funds.**

All user-facing deposit and withdrawal flows are blocked for every user until an admin calls `updatePriceOracleFor()` to substitute a working oracle. During this window, no new deposits can be made, no withdrawals can be processed, and the rsETH price cannot be updated. The freeze is temporary only because `LRTOracle.updatePriceOracleFor()` allows the admin to swap the oracle without calling `getAssetPrice()` on the new one. [8](#0-7) 

Note: `updatePriceOracleForValidated()` itself calls `getAssetPrice()` on the new oracle as a sanity check, so it would also revert if the new oracle still uses the broken Chainlink feed — the admin must use the unvalidated `updatePriceOracleFor()` path. [9](#0-8) 

---

### Likelihood Explanation

Chainlink's multisig can block access to price feeds at will, as documented by OpenZeppelin. This is not a theoretical edge case — it is an acknowledged operational risk. Any Chainlink feed used by the protocol (stETH/ETH, cbETH/ETH, etc.) is subject to this. The trigger requires no attacker action; it is a unilateral decision by Chainlink's operators. [10](#0-9) 

---

### Recommendation

Wrap the `latestRoundData()` call in a `try/catch` block. On revert, either fall back to a secondary oracle or revert with a descriptive error that allows the caller to handle the failure gracefully:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    try priceFeed.latestRoundData() returns (uint80, int256 price, uint256, uint256, uint80) {
        return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
    } catch {
        revert PriceFeedCallFailed();
    }
}
```

Apply the same pattern to `ChainlinkOracleForRSETHPoolCollateral.getRate()`. Additionally, consider making the `oracle` address in `ChainlinkOracleForRSETHPoolCollateral` mutable with an admin setter so a broken feed can be replaced without redeployment. [11](#0-10) 

---

### Proof of Concept

1. Chainlink's multisig blocks the stETH/ETH price feed used by `ChainlinkPriceOracle`.
2. Any user calls `LRTDepositPool.depositAsset(stETH, amount, minRSETH, referralId)`.
3. `depositAsset` → `_beforeDeposit` → `getRsETHAmountToMint` → `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → `priceFeed.latestRoundData()` reverts.
4. The entire transaction reverts. All deposits for all assets are blocked (because `_getTotalEthInProtocol` iterates all assets and a single revert aborts the loop).
5. `updateRSETHPrice()` also reverts, so the rsETH price becomes permanently stale.
6. Withdrawals via `LRTWithdrawalManager` also revert at the `getAssetPrice` call.
7. The freeze persists until the admin calls `updatePriceOracleFor(stETH, newOracle)`. [1](#0-0) [12](#0-11) [13](#0-12)
