### Title
Rebasing `rewardsToken` Yield Permanently Locked in `KernelDepositPool` — (File: `contracts/KERNEL/KernelDepositPool.sol`)

---

### Summary

`KernelDepositPool` distributes a configurable `rewardsToken` to KERNEL stakers at a fixed `rewardRate` computed at the time of each `notifyRewardAmount` call. If `rewardsToken` is a rebasing token (e.g., stETH — which is explicitly integrated elsewhere in the protocol), the contract's `rewardsToken` balance grows continuously between reward periods due to rebasing. This excess yield is never distributed to stakers and cannot be recovered, because the contract has no sweep, rescue, or excess-withdrawal function for `rewardsToken`, and does not inherit from `Recoverable`.

---

### Finding Description

`KernelDepositPool` is a Synthetix-style staking rewards contract. The admin calls `notifyRewardAmount` to fund a reward period. The function uses a balance-before/after pattern to determine the actual amount received:

```solidity
uint256 balanceBefore = rewardsToken.balanceOf(address(this));
rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
uint256 balanceAfter = rewardsToken.balanceOf(address(this));
uint256 receivedAmount = balanceAfter - balanceBefore;
rewardRate = receivedAmount / duration;
``` [1](#0-0) 

This pattern correctly handles fee-on-transfer tokens, but it does **not** handle rebasing tokens. Between two calls to `notifyRewardAmount`, if `rewardsToken` is rebasing, the contract's balance grows. When the next `notifyRewardAmount` is called, `balanceBefore` already includes the rebased excess, so `receivedAmount` only reflects the newly transferred amount — the rebased excess is silently excluded from the new `rewardRate` and remains stranded in the contract.

Users claim rewards via `getReward()`, which transfers exactly `rewards[msg.sender]` — a value derived solely from the fixed `rewardRate`:

```solidity
rewards[msg.sender] = 0;
rewardsToken.safeTransfer(msg.sender, rewardAmount);
``` [2](#0-1) 

The `earned()` function is also purely rate-based:

```solidity
return (balanceOf[_account] * (rewardPerToken() - userRewardPerTokenPaid[_account]) / DECIMAL_PRECISION)
    + rewards[_account];
``` [3](#0-2) 

No function in `KernelDepositPool` allows withdrawal of excess `rewardsToken` balance. The contract does not inherit from `Recoverable` (unlike other contracts in the codebase such as `SonicChainNativeTokenBridge`), so there is no admin token-rescue path either. [4](#0-3) 

The same issue applies to `kernelToken` (the staking token): if it is rebasing, `claimWithdrawal` returns only the original staked `withdrawal.amount`, and the rebased excess is permanently locked. [5](#0-4) 

---

### Impact Explanation

**Permanent freezing of unclaimed yield (Medium).**

Any rebasing yield that accrues on the contract's `rewardsToken` balance between reward periods is permanently locked. Stakers receive only the flat rate computed at `notifyRewardAmount` time; the rebased surplus is irrecoverable. This directly matches the external report's impact class: users miss out on yield they are entitled to, and the excess remains locked in the contract with no recovery path.

---

### Likelihood Explanation

The protocol explicitly integrates stETH (a canonical rebasing token) in `LRTDepositPool` via `ILido`: [6](#0-5) 

`KernelDepositPool.initialize` accepts any arbitrary `_rewardToken` address with no validation that it is non-rebasing: [7](#0-6) 

If the deployer configures `rewardsToken` as stETH or any other rebasing token, the vulnerability is immediately active. Given the protocol's deep integration with Lido and yield-bearing tokens, this is a realistic deployment scenario.

---

### Recommendation

1. **Add an excess-withdrawal function** restricted to `DEFAULT_ADMIN_ROLE` that computes the difference between `rewardsToken.balanceOf(address(this))` and the total rewards owed (i.e., `rewardRate * (finishAt - block.timestamp)` plus all pending `rewards[user]` balances) and transfers the surplus to a designated treasury.
2. **Alternatively**, inherit from `Recoverable` (already present in the codebase at `contracts/utils/Recoverable.sol`) to allow admin token rescue, with a guard that prevents recovering more than the surplus above owed rewards.
3. **Document** that rebasing tokens are not supported as `rewardsToken` or `kernelToken`, and add an input validation check (e.g., a whitelist or explicit non-rebasing assertion) in `initialize`.

---

### Proof of Concept

1. Admin deploys `KernelDepositPool` with `rewardsToken = stETH`.
2. Admin calls `notifyRewardAmount(1000e18)` — transfers 1000 stETH, sets `rewardRate = 1000e18 / duration`.
3. Over the reward period, stETH rebases: the contract's stETH balance grows from 1000 stETH to, say, 1010 stETH.
4. All stakers call `getReward()` and collectively receive exactly 1000 stETH (the rate-computed amount).
5. The remaining 10 stETH sits in the contract.
6. Admin calls `notifyRewardAmount(500e18)` for the next period: `balanceBefore = 1010e18`, `receivedAmount = 500e18`, `rewardRate = 500e18 / duration`. The 10 stETH rebasing surplus is excluded from the new rate.
7. After all future reward periods end and all users claim, the 10 stETH (plus any further rebasing) remains permanently locked — there is no function to extract it. [8](#0-7)

    function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
        if (_amount == 0) revert AmountZero();

        // Prevent starting a reward period when no tokens are staked to avoid unallocated rewards
        if (totalKernelStaked == 0) revert NoStakedTokens();

        // Transfer reward tokens into the contract
        uint256 balanceBefore = rewardsToken.balanceOf(address(this));
        rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
        uint256 balanceAfter = rewardsToken.balanceOf(address(this));
        // Calculate the actual amount of tokens received in case of a transfer fee (tax)
        uint256 receivedAmount = balanceAfter - balanceBefore;

        if (block.timestamp >= finishAt) {
            rewardRate = receivedAmount / duration;
        } else {
            uint256 remaining = (finishAt - block.timestamp) * rewardRate;
            rewardRate = (receivedAmount + remaining) / duration;
        }

        if (rewardRate == 0) revert RewardRateZero();

        finishAt = block.timestamp + duration;
        updatedAt = block.timestamp;

        emit NotifyRewardAmount(receivedAmount, finishAt);
```
