### Title
Daily Fee Mint Limit Blocks All rsETH Price Updates, Leaving rsETHPrice Stale for Up to 24 Hours — (`contracts/LRTOracle.sol`)

### Summary

When `currentPeriodMintedFeeAmount` is near `maxFeeMintAmountPerDay` and a TVL increase generates a fee that would push the sum over the cap, `_checkAndUpdateDailyFeeMintLimit` reverts inside `_updateRsETHPrice`. Because **both** `updateRSETHPrice()` and `updateRSETHPriceAsManager()` call the same internal function without any bypass, all price updates are blocked for the remainder of the 24-hour period. `rsETHPrice` goes stale, breaking withdrawal queue accounting until the period resets.

### Finding Description

`_updateRsETHPrice` computes a protocol fee whenever TVL increases: [1](#0-0) 

It then converts that fee to rsETH and calls the daily-limit check: [2](#0-1) 

Inside `_checkAndUpdateDailyFeeMintLimit`, if the accumulated amount plus the new fee exceeds the cap, the function reverts unconditionally: [3](#0-2) 

This revert propagates all the way up, so `rsETHPrice = newRsETHPrice` at line 313 is never reached: [4](#0-3) 

Both public entry points delegate to the same internal function with no way to skip the fee-mint check: [5](#0-4) 

The period only resets when `block.timestamp >= feePeriodStartTime + 1 days`: [6](#0-5) 

Until that reset, every call to either `updateRSETHPrice()` or `updateRSETHPriceAsManager()` that encounters a TVL increase will revert, leaving `rsETHPrice` frozen at its last successfully written value.

### Impact Explanation

A stale `rsETHPrice` directly affects withdrawal queue processing: the withdrawal manager uses the oracle price to determine how much ETH to release per rsETH burned. With a stale (lower-than-actual) price, users receive less ETH than they are owed; with a stale (higher-than-actual) price, the protocol overpays. In either case, accurate settlement is impossible until the price is updated. This constitutes **temporary freezing of funds** for up to 24 hours — matching the Medium impact scope.

### Likelihood Explanation

`maxFeeMintAmountPerDay` is a finite manager-set value. Any single large staking reward event (e.g., EigenLayer restaking yield accrual, a large deposit that shifts TVL) can generate a fee that exhausts the remaining daily allowance in one call. This is a normal, non-adversarial protocol condition that requires no attacker and no privileged compromise — it can occur organically during high-yield periods.

### Recommendation

Decouple fee minting from price updating. When `rsethAmountToMintAsProtocolFee` would exceed the remaining daily limit, either:

1. **Cap the minted fee** at the remaining limit and proceed with the price update (accepting partial fee collection for that call), or
2. **Skip fee minting** for that call (mint zero, update price) and emit an event indicating the fee was foregone.

Do not allow the fee-mint cap to abort the price update itself.
