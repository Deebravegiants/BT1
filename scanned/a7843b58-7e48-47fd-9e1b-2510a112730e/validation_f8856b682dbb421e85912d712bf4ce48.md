### Title
Asymmetric Velocity Denominator in `PriceVelocityGuardExtension.beforeSwap` Allows Larger Downward Oracle Price Moves Than Intended - (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

---

### Summary

`PriceVelocityGuardExtension.beforeSwap` computes the oracle mid-price percentage change by always dividing by `prevMid` (the **old** price). This makes the guard asymmetric: an upward move from price A to B is measured as a larger percentage than the equivalent downward move from B back to A. The admin configures a single `maxChangePerBlockE18` threshold believing it applies symmetrically in both directions, but the guard is materially weaker for downward oracle price moves, allowing larger absolute downward moves than the admin intended to permit.

---

### Finding Description

In `PriceVelocityGuardExtension.beforeSwap`:

```solidity
uint256 delta = midPrice > prevMid ? uint256(midPrice - prevMid) : uint256(prevMid - midPrice);
uint256 changeE18 = (delta * 1e18) / uint256(prevMid);   // ← always divides by OLD price
``` [1](#0-0) 

The denominator is always `prevMid`, regardless of direction. This is the direct analog of the external report's `x.mul(ONE).div(y)` where `y` is hardcoded as the denominator.

**Numerical demonstration** (using `maxChangePerBlockE18 = 0.18e18`, i.e. 18% per block):

| Scenario | prevMid | midPrice | delta | changeE18 | Result |
|---|---|---|---|---|---|
| Price rises 100 → 120 | 100 | 120 | 20 | 20/100 = **20%** | **BLOCKED** (>18%) |
| Price falls 120 → 100 | 120 | 100 | 20 | 20/120 = **16.7%** | **ALLOWED** (<18%) |

The same absolute price move (100 ↔ 120) is blocked in one direction and allowed in the other. The admin set 18% intending to cap both directions equally, but the guard silently permits a downward move that corresponds to a 20% upward move — well above the intended cap.

The symmetric fix would use `max(prevMid, midPrice)` as the denominator, giving the same percentage for both directions.

---

### Impact Explanation

The `PriceVelocityGuardExtension` is the only on-chain mechanism that caps how fast the oracle mid-price can move between blocks. Its purpose is to prevent oracle price manipulation from reaching pool swap execution. Because the guard is asymmetric:

1. **Downward oracle price manipulation is under-guarded.** An attacker who can influence the oracle price (e.g., via flash-loan-driven spot manipulation feeding into the price provider) can move the price downward by a larger absolute amount than the admin intended to allow before the guard triggers.

2. **Bad-price execution reaches the pool.** A downward-manipulated oracle price that passes the guard reaches `SwapMath.midAndSpreadFeeX64FromBidAsk` and `_executeSwap`, causing swaps to execute at a price the admin explicitly intended to block. Token0 buyers (zeroForOne=false) receive token0 at an artificially cheap price, draining LP token1 balances.

3. **Admin miscalibration is silent.** The admin sets `maxChangePerBlockE18` believing it is a symmetric cap. There is no documentation or revert indicating the asymmetry. The effective downward cap is always smaller than the configured value, and the gap widens as the price move grows.

---

### Likelihood Explanation

- The `PriceVelocityGuardExtension` is a production periphery contract listed in scope and designed for real pool deployments.
- Any pool that configures this extension is affected.
- The asymmetry is always present — it is not a corner case. Every downward oracle price move is measured with a smaller percentage than the equivalent upward move.
- The asymmetry is most exploitable when the oracle price is volatile or when an attacker can influence the price provider's output (e.g., via a manipulable on-chain price source).

---

### Recommendation

Replace the fixed `prevMid` denominator with `max(prevMid, midPrice)` so the percentage change is symmetric regardless of direction:

```solidity
uint256 delta = midPrice > prevMid
    ? uint256(midPrice - prevMid)
    : uint256(prevMid - midPrice);

// Use the larger price as denominator — symmetric in both directions
uint256 denominator = midPrice > prevMid ? uint256(midPrice) : uint256(prevMid);
uint256 changeE18 = (delta * 1e18) / denominator;
```

This mirrors the external report's recommended fix: always use the bigger price as the denominator so that `|A→B|` and `|B→A|` produce the same `changeE18`, and the admin's configured threshold applies equally in both directions.

### Proof of Concept

**Setup:**
- Pool configured with `PriceVelocityGuardExtension`, `maxChangePerBlockE18 = 0.18e18` (18% per block).
- Admin intends: no oracle move larger than 18% per block in either direction.

**Step 1 — Establish baseline:**
- Block N: swap executes, oracle mid = 120. `lastMidPriceX64 = 120`, `lastUpdateBlock = N`.

**Step 2 — Downward move in block N+1:**
- Oracle mid = 100 (absolute move = 20, which is 20% of 100).
- `delta = 120 - 100 = 20`
- `changeE18 = (20 * 1e18) / 120 = 0.1667e18` (16.67%)
- `actualSq = (0.1667e18)^2 = 0.02779e36`
- `allowedSq = (0.18e18)^2 * (1 + 0) = 0.0324e36`
- `0.02779e36 < 0.0324e36` → **guard passes**, swap executes at oracle mid = 100.

**Step 3 — Equivalent upward move (blocked):**
- If instead oracle mid = 140 (absolute move = 20, which is 20% of 120):
- `delta = 140 - 120 = 20`
- `changeE18 = (20 * 1e18) / 120 = 0.1667e18` (16.67%)
- Same result — also passes! But if oracle mid = 121.6 (1.33% up from 120), `changeE18 = 1.6/120 = 1.33%` — trivially passes.

**Corrected asymmetry demonstration** — upward from 100 to 120:
- `delta = 20`, `changeE18 = (20 * 1e18) / 100 = 0.20e18` (20%)
- `actualSq = 0.04e36 > allowedSq = 0.0324e36` → **BLOCKED**

The same absolute move (100 ↔ 120) is blocked going up but allowed going down, confirming the asymmetric guard allows oracle prices the admin intended to block to reach pool swap execution. [2](#0-1)
