### Title
Attacker Can Permanently DoS Any LP's Full `removeLiquidity` by Frontrunning with a Dust Share Donation — (`metric-core/contracts/libraries/LiquidityLib.sol`)

---

### Summary

`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address with no access control, allowing anyone to donate shares into any LP's position. `removeLiquidity` enforces a `MinimalLiquidity` guard that reverts when the remaining position shares fall in the range `(0, MINIMAL_MINTABLE_LIQUIDITY)`. An attacker can frontrun any full-withdrawal transaction by donating exactly 1 share to the victim's position, causing the victim's `sharesToRemove` to leave exactly 1 dust share, which triggers the revert. The victim's funds are not permanently lost but the withdrawal is blocked until the victim re-queries their balance and retries — and the attacker can repeat the grief indefinitely at minimal cost.

---

### Finding Description

`MetricOmmPool.addLiquidity` takes an explicit `owner` parameter and has no restriction requiring `msg.sender == owner`:

```solidity
// MetricOmmPool.sol
function addLiquidity(
    address owner,   // ← any address; caller pays, but position is credited to owner
    uint80 salt,
    LiquidityDelta calldata deltas,
    ...
``` [1](#0-0) 

`removeLiquidity`, by contrast, enforces `msg.sender == owner` and then calls into `LiquidityLib.removeLiquidity`, which contains:

```solidity
uint256 newUserShares = userShares - sharesToRemove;
if (newUserShares > 0 && newUserShares < ctx.minimalMintableLiquidity) {
    revert IMetricOmmPoolActions.MinimalLiquidity(newUserShares, ctx.minimalMintableLiquidity);
}
``` [2](#0-1) 

The `addLiquidity` path's own `MinimalLiquidity` guard checks the **post-add** share count of the target position:

```solidity
uint256 newUserShares = userShares + sharesToAdd;
if (newUserShares < ctx.minimalMintableLiquidity) {
    revert IMetricOmmPoolActions.MinimalLiquidity(...);
}
``` [3](#0-2) 

Because the victim already holds `N ≥ MINIMAL_MINTABLE_LIQUIDITY` shares, adding 1 share yields `N+1 ≥ MINIMAL_MINTABLE_LIQUIDITY`, so the donation passes the add-side guard. The victim's pending `removeLiquidity(shares=N)` then computes `newUserShares = 1`, which satisfies `0 < 1 < MINIMAL_MINTABLE_LIQUIDITY` and reverts.

---

### Impact Explanation

- Any LP's full-withdrawal transaction can be griefed by a third party for the cost of 1 share worth of tokens (potentially 1 wei of each token for a high-share-count bin) plus gas.
- The victim's principal is not permanently lost — they can re-query their balance and retry — but the attacker can repeat the grief on every retry, forcing the LP to use a private mempool or MEV-protected RPC.
- Smart-contract vaults or routers that call `removeLiquidity` with a cached share count and do not handle `MinimalLiquidity` reverts gracefully can have their withdrawal flow permanently bricked.
- This maps directly to the allowed impact gate: **broken core pool functionality causing unusable withdraw flows**.

---

### Likelihood Explanation

- The attack requires only a standard `addLiquidity` call with `owner = victim`; no special role or privileged access is needed.
- The attacker's cost is bounded by 1 share of tokens (which rounds up to 1 scaled unit, potentially 1 wei after unscaling for high-liquidity bins) plus gas.
- The attacker's donated tokens are credited to the victim's position and recovered by the victim on a successful retry, so the attacker bears a permanent token loss — but it can be negligible.
- Frontrunning is straightforward on any public mempool chain.

---

### Recommendation

**Option A (preferred):** Restrict `addLiquidity` so that `msg.sender == owner` unless an explicit delegation mechanism is in place. This eliminates the donation vector entirely.

**Option B:** In `removeLiquidity`, cap `sharesToRemove` to `userShares` (analogous to the Venus mitigation of capping `repayAmount` to `maxClose`) instead of reverting when the caller requests more than they own. For the `MinimalLiquidity` guard specifically, allow a full-to-zero withdrawal even when the remaining amount would be dust — only block partial withdrawals that leave dust.

**Option C:** Emit a slippage-style parameter (`minSharesAfter`) that the caller sets, and revert only when the actual remaining shares fall below that caller-supplied floor, giving the LP control over the invariant rather than the pool enforcing a fixed floor that a third party can exploit.

---

### Proof of Concept

1. LP calls `addLiquidity(owner=LP, salt=S, shares=[N])` → position `(LP, S, bin)` holds `N = 10_000` shares (`MINIMAL_MINTABLE_LIQUIDITY = 1_000`).
2. LP broadcasts `removeLiquidity(owner=LP, salt=S, shares=[10_000])` to withdraw everything.
3. Attacker sees the pending tx in the mempool and frontruns with `addLiquidity(owner=LP, salt=S, shares=[1])`, paying for 1 share worth of tokens. The add-side guard passes because `10_000 + 1 = 10_001 ≥ 1_000`.
4. LP's `removeLiquidity` executes: `userShares = 10_001`, `sharesToRemove = 10_000`, `newUserShares = 1`.
5. Guard fires: `1 > 0 && 1 < 1_000` → `revert MinimalLiquidity(1, 1_000)`.
6. LP's withdrawal is blocked. Attacker repeats on every retry. [4](#0-3) [5](#0-4)
