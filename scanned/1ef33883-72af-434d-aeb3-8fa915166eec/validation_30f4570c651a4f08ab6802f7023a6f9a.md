### Title
ETH Permanently Frozen When Withdrawer Is a Contract Without `receive()` Function - (File: `contracts/LRTWithdrawalManager.sol`)

### Summary
`LRTWithdrawalManager._transferAsset()` sends ETH via a raw `.call{value: amount}("")`. If the recipient is a contract without a `receive()` or `fallback()` function, the call fails and the transaction reverts. Because rsETH is burned during the earlier `unlockQueue()` step (not during `completeWithdrawal()`), the user's rsETH is permanently gone while the ETH remains locked in the contract with no admin rescue path.

### Finding Description
The withdrawal lifecycle in `LRTWithdrawalManager` is split across three steps:

1. **`initiateWithdrawal()`** — user transfers rsETH to the contract.
2. **`unlockQueue()`** — operator burns the rsETH held by the contract and moves ETH from `LRTUnstakingVault` into `LRTWithdrawalManager`.
3. **`completeWithdrawal()` / `completeWithdrawalForUser()`** — calls `_processWithdrawalCompletion()`, which calls `_transferAsset()` to push ETH to the user.

`_transferAsset()` is implemented as:

```solidity
function _transferAsset(address asset, address to, uint256 amount) internal {
    if (asset == LRTConstants.ETH_TOKEN) {
        (bool sent,) = payable(to).call{ value: amount }("");
        if (!sent) revert EthTransferFailed();
    } else {
        IERC20(asset).safeTransfer(to, amount);
    }
}
```

If `to` is a contract without a `receive()` or `fallback()` function, the `.call` returns `false`, and the function reverts with `EthTransferFailed`. This reverts all state changes in `_processWithdrawalCompletion()` (the nonce pop, the request deletion, the `unlockedWithdrawalsCount` decrement), so the withdrawal request is restored and the user can retry — but every retry will fail identically.

The ETH is now permanently stranded in `LRTWithdrawalManager`:
- `completeWithdrawal()` and `completeWithdrawalForUser()` both send to `user`, so both always revert.
- `sweepRemainingAssets()` is blocked: it checks `hasUnlockedWithdrawals(asset)`, which returns `true` because `unlockedWithdrawalsCount[asset] > 0` for the stuck request.
- There is no admin "force-send to alternate address" path.

The rsETH was already burned in step 2 and cannot be recovered.

### Impact Explanation
**Permanent freezing of funds (Critical).** A user who is a smart contract without a `receive()` function loses both their rsETH (burned in `unlockQueue()`) and their ETH (stuck in `LRTWithdrawalManager` with no rescue path). The ETH cannot be swept by the manager because `sweepRemainingAssets()` is gated on `!hasUnlockedWithdrawals(asset)`.

### Likelihood Explanation
**Medium.** Smart-contract wallets (e.g., Gnosis Safe with custom modules), protocol-owned treasuries, and DeFi integrations routinely interact with staking protocols. Many such contracts do not implement a `receive()` function. The protocol does not warn callers of this restriction, and `initiateWithdrawal()` accepts any `msg.sender` without checking whether it can receive ETH.

### Recommendation
1. **Pull-payment pattern**: Instead of pushing ETH to the user in `completeWithdrawal()`, record the claimable amount and let the user pull it via a separate `claimETH()` function. This decouples delivery failure from state mutation.
2. **WETH fallback**: If the ETH push fails, wrap the ETH as WETH and credit it to the user's address, similar to the `_sendEthOrWeth()` pattern — but ensure the user is warned that WETH may be received.
3. **Alternate recipient parameter**: Allow the user to specify a different recipient address at claim time.

### Proof of Concept

1. A smart-contract vault (no `receive()`) calls `initiateWithdrawal(ETH_TOKEN, 1 ether rsETH, "")`. [1](#0-0) 

2. Operator calls `unlockQueue(ETH_TOKEN, ...)`. rsETH held by the contract is burned; ETH is moved from `LRTUnstakingVault` into `LRTWithdrawalManager`. [2](#0-1) 

3. The vault calls `completeWithdrawal(ETH_TOKEN, "")`. Inside `_processWithdrawalCompletion()`, `_transferAsset()` is reached: [3](#0-2) 

4. `payable(vault).call{value: amount}("")` returns `false` (no `receive()`). `revert EthTransferFailed()` fires, rolling back all state changes. The withdrawal request is restored.

5. Every subsequent call to `completeWithdrawal()` or `completeWithdrawalForUser()` reverts identically — both send to `user`. [4](#0-3) 

6. `sweepRemainingAssets()` is called by the manager to rescue the ETH, but it reverts because `hasUnlockedWithdrawals(ETH_TOKEN)` is `true`: [5](#0-4) 

Result: rsETH is permanently burned, ETH is permanently locked in `LRTWithdrawalManager`, and no recovery path exists.
