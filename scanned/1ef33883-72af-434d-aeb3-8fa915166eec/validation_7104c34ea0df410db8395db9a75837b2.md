### Title
Inverted Deposit Limit Check for ETH Allows Depositors to Exceed the Cap - (File: `contracts/LRTDepositPool.sol`)

---

### Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an incomplete comparison for ETH deposits, omitting the incoming `amount` from the limit check. Any ETH depositor can push total ETH deposits beyond the configured `depositLimitByAsset` cap, while the equivalent LST path correctly includes the incoming amount.

---

### Finding Description

In `LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` (lines 676–682), the function branches on whether the asset is ETH or an ERC-20 LST:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← missing `+ amount`
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct
}
``` [1](#0-0) 

- **ETH branch**: returns `true` only when the total *already* exceeds the limit — it never considers whether the new deposit would push the total over the limit.
- **LST branch**: correctly returns `true` when `totalAssetDeposits + amount > limit`.

This is structurally identical to the external report's bug: the special treatment (the `+ amount` term) is applied to the wrong branch of the conditional.

The ETH deposit entry point is `depositETH`, which is publicly callable by any user:

```solidity
function depositETH(uint256 minRSETHAmountExpected, string calldata referralId)
    external payable nonReentrant whenNotPaused onlySupportedAsset(LRTConstants.ETH_TOKEN)
{
    uint256 rsethAmountToMint = _beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, minRSETHAmountExpected);
    _mintRsETH(rsethAmountToMint);
    ...
}
``` [2](#0-1) 

`_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit` and reverts on `true`. Because the ETH check never includes `amount`, the revert never fires for a deposit that would merely *push* the total over the limit. [3](#0-2) 

---

### Impact Explanation

**Classification: Low — Contract fails to deliver promised returns.**

The deposit limit (`depositLimitByAsset`) is a protocol-enforced risk cap. For ETH, this cap is not enforced at the boundary: any depositor can submit an ETH deposit that pushes `totalAssetDeposits` above the configured limit, as long as the current total has not already crossed it. The protocol mints rsETH for the excess amount and the ETH is accepted into the pool, violating the intended invariant. No direct fund theft occurs, but the protocol takes on more EigenLayer exposure than governance intended.

---

### Likelihood Explanation

**High likelihood.** The entry point (`depositETH`) is permissionless and requires no special role. The condition is triggered whenever `totalAssetDeposits` is at or near the limit — a normal operational state. Any depositor who observes the current total approaching the cap can trivially submit a deposit that exceeds it.

---

### Recommendation

Add `+ amount` to the ETH branch to match the LST branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

The ETH and LST paths should apply the same arithmetic; the branch is unnecessary once corrected.

---

### Proof of Concept

1. Governance sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. Current `getTotalAssetDeposits(ETH_TOKEN)` returns `999 ether`.
3. An LST depositor attempting to deposit 2 ETH-equivalent is blocked: `999 + 2 > 1000 → true → revert`.
4. An ETH depositor calls `depositETH{value: 2 ether}(...)`:
   - `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 2 ether)` evaluates `999 > 1000 → false`.
   - The check passes; rsETH is minted for 2 ETH.
   - `totalAssetDeposits` is now `1001 ether`, exceeding the cap by 1 ETH.
5. The depositor repeats this for arbitrarily large amounts as long as the total has not yet crossed the limit in a prior block. [1](#0-0)
