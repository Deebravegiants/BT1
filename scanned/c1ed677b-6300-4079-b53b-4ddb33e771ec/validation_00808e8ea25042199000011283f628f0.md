### Title
Large Withdrawal Request at Queue Head Blocks All Subsequent Smaller Requests from Being Unlocked - (File: contracts/LRTWithdrawalManager.sol)

### Summary

`LRTWithdrawalManager._unlockWithdrawalRequests()` processes the withdrawal queue in strict FIFO order and unconditionally `break`s when the head-of-queue request requires more assets than the vault currently holds. An unprivileged rsETH holder can place a large withdrawal request early in the queue, causing all subsequent smaller requests — which could individually be fulfilled — to remain locked until the vault accumulates enough assets to cover the large request first.

### Finding Description

`_unlockWithdrawalRequests` iterates from `nextLockedNonce` forward and stops the entire loop the moment it encounters a request whose `payoutAmount` exceeds `availableAssetAmount`:

```solidity
if (availableAssetAmount < payoutAmount) break; // Exit if not enough assets to cover this request
``` [1](#0-0) 

`availableAssetAmount` is sourced from `unstakingVault.balanceOf(asset)` — the vault's liquid balance at the moment `unlockQueue` is called: [2](#0-1) 

Because EigenLayer imposes a multi-day withdrawal delay, the vault accumulates liquid assets gradually. At any given `unlockQueue` call the vault balance is typically a fraction of total committed assets.

An attacker who holds rsETH calls `initiateWithdrawal` with a large `rsETHUnstaked` value. The check at line 170 only prevents committing more than the currently uncommitted total asset pool: [3](#0-2) 

Once the large request is at the front of the global nonce sequence, every subsequent `unlockQueue` call hits the `break` immediately, leaving all later (smaller) requests permanently locked until the vault accumulates enough liquid assets to cover the attacker's request in full. There is no mechanism to skip a request or process out-of-order.

### Impact Explanation

All withdrawal requests queued after the attacker's large request are temporarily frozen. Users who submitted small, individually-fulfillable requests cannot call `completeWithdrawal` because `nextLockedNonce` never advances past the blocking request: [4](#0-3) 

The freeze persists until the vault's liquid balance reaches the attacker's full `payoutAmount`. Given EigenLayer's 7-day+ unstaking cycle and the fact that the vault services many users, this can translate to a multi-day to multi-week delay for all queued users behind the blocker. **Impact: Medium — Temporary freezing of funds.**

### Likelihood Explanation

Any rsETH holder can call `initiateWithdrawal`. The attacker does not lose funds; they eventually receive their withdrawal. The only cost is locking rsETH for the withdrawal delay period, which the attacker would have done anyway. The attack is most effective when the attacker places the request before other users, or when the queue is short. **Likelihood: Medium.**

### Recommendation

Replace the unconditional `break` with a `continue` (skip-and-proceed) so that smaller requests behind an unfulfillable large request can still be unlocked. Alternatively, track a per-request "skipped" flag and allow the operator to explicitly skip a request that cannot currently be fulfilled, advancing `nextLockedNonce` past it. A simpler mitigation is to cap the maximum single withdrawal size relative to the vault's expected liquid balance.

### Proof of Concept

1. Protocol has 1000 ETH total assets; vault holds 200 ETH liquid; `assetsCommitted = 0`.
2. **Attacker** calls `initiateWithdrawal(ETH, rsETH_equivalent_of_900_ETH, ...)`. `assetsCommitted[ETH]` becomes 900 ETH. Attacker's request is at nonce 0.
3. **Alice** calls `initiateWithdrawal(ETH, rsETH_equivalent_of_5_ETH, ...)`. Her request is at nonce 1.
4. **Bob** calls `initiateWithdrawal(ETH, rsETH_equivalent_of_5_ETH, ...)`. His request is at nonce 2.
5. After the withdrawal delay, operator calls `unlockQueue(ETH, 3, ...)`. Vault balance = 200 ETH.
6. Loop iteration 0: `payoutAmount = 900 ETH > 200 ETH` → `break`. [5](#0-4) 
7. `nextLockedNonce[ETH]` stays at 0. Alice and Bob's requests (5 ETH each, easily coverable) remain locked.
8. Alice and Bob call `completeWithdrawal` → revert `WithdrawalLocked` because their nonces (1, 2) are ≥ `nextLockedNonce` (0). [6](#0-5) 
9. The freeze continues until the vault accumulates ≥ 900 ETH liquid — potentially weeks — while Alice and Bob's 5 ETH requests sit idle.
