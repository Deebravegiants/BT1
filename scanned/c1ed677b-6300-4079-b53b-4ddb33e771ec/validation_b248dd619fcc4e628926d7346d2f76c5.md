### Title
Zero-Initialized Rate in `CrossChainRateReceiver` Causes Division-by-Zero Panic in All Pool Deposit Paths Before First LZ Message — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

---

### Summary

`CrossChainRateReceiver.rate` defaults to `0` at deployment. Every pool variant's `viewSwapRsETHAmountAndFee` divides by this value without a zero-guard, causing an arithmetic panic on any deposit call until the first LayerZero rate message is delivered.

---

### Finding Description

`CrossChainRateReceiver` declares `uint256 public rate;` with no initial value, so it is `0` at construction. [1](#0-0) 

`RSETHRateReceiver`'s constructor sets `rateInfo`, `srcChainId`, `rateProvider`, and `layerZeroEndpoint`, but never sets `rate`. [2](#0-1) 

`getRate()` returns the raw stored value with no zero-check. [3](#0-2) 

Every pool's `getRate()` delegates directly to the oracle. [4](#0-3) 

`viewSwapRsETHAmountAndFee` (the deposit path) divides by `rsETHToETHrate` with no zero-guard: [5](#0-4) 

The zero-check `if (rsETHToETHrate == 0) revert UnsupportedOracle()` exists **only** in `viewSwapAssetToPremintedRsETH` (the operator reverse-swap path), not in the deposit path: [6](#0-5) 

The same unguarded division is present across all pool variants: `RSETHPool`, `RSETHPoolV2`, `RSETHPoolV2NBA`, `RSETHPoolV2ExternalBridge`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, and `RSETHPoolNoWrapper`. [7](#0-6) 

---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

When `rate == 0`, any call to `deposit()` panics at the division and the entire transaction reverts. For ETH deposits, `msg.value` is returned to the caller by the revert. For token deposits, `safeTransferFrom` executes before the division, but the panic reverts the whole transaction, so the token transfer is also unwound. **No user funds are frozen or lost**; deposits simply revert cleanly. The condition resolves automatically once the first `lzReceive` call sets `rate` to a non-zero value.

The claimed impact of "Critical. Protocol insolvency" or "permanent freezing of funds" does not apply: no funds are held in the pool during this window, and all deposit transactions revert atomically.

---

### Likelihood Explanation

The window is bounded by the time between pool deployment and the first successful LayerZero rate message. In normal operation this is short. However, if the LZ message is delayed (e.g., insufficient gas fee paid to the provider, LZ relayer lag, or misconfigured `srcChainId`/`rateProvider`), the pool is entirely non-functional for deposits during that period. No attacker action is required; any user calling `deposit` triggers the revert.

---

### Recommendation

Add a zero-rate guard in `viewSwapRsETHAmountAndFee` (both overloads) consistent with the guard already present in `viewSwapAssetToPremintedRsETH`:

```solidity
uint256 rsETHToETHrate = getRate();
if (rsETHToETHrate == 0) revert UnsupportedOracle();
```

Alternatively, initialize `rate` to a safe bootstrap value (e.g., `1e18`) in the `RSETHRateReceiver` constructor, to be overwritten by the first real LZ message.

---

### Proof of Concept

```solidity
// 1. Deploy RSETHRateReceiver — rate == 0
// 2. Deploy RSETHPoolV3 with rsETHOracle = address(rateReceiver)
// 3. Call pool.deposit{value: 1 ether}("ref")
//    → getRate() returns 0
//    → viewSwapRsETHAmountAndFee: `amountAfterFee * 1e18 / 0` → arithmetic panic
//    → transaction reverts, ETH returned to caller
// 4. Simulate lzReceive setting rate = 1.05e18
// 5. Call pool.deposit{value: 1 ether}("ref") again → succeeds
```
