### Title
Cross-Chain Slippage Delta Permanently Frozen in KernelReceiver with No Recovery Path - (File: contracts/KERNEL/KernelReceiver.sol)

---

### Summary

`KernelVaultETH.bridgeKernelToBSC` explicitly permits a `minAmount < amount` slippage gap, but `KernelReceiver` on BSC has no token-rescue function. Any tokens delivered by the OFT bridge in excess of what `_stakeFor` can consume are permanently locked in the contract.

---

### Finding Description

**On Ethereum (`KernelVaultETH`):**

`bridgeKernelToBSC` accepts `minAmount` strictly less than `amount`, explicitly acknowledging that the OFT bridge may deliver fewer tokens than sent: [1](#0-0) 

The OFT `SendParam` is constructed with `amountLD = amount` and `minAmountLD = minAmount`, so the bridge is permitted to deliver anywhere in `[minAmount, amount)`: [2](#0-1) 

**On BSC (`KernelReceiver`):**

`_stakeFor` checks the contract's live balance against the caller-supplied `amount`: [3](#0-2) 

There are exactly two outcomes when slippage occurs and the OFT delivers `N - delta` instead of `N`:

| Operator action | Result |
|---|---|
| Calls `stakeFor(user, N)` | Reverts with `InsufficientKernelBalance`; all `N - delta` tokens remain in contract |
| Calls `stakeFor(user, N - delta)` | Succeeds, but `delta` tokens remain in contract with no exit path |

In both cases the slippage delta is stranded. `KernelReceiver` exposes **no** `recoverTokens`, `emergencyWithdraw`, or any other admin-callable token-transfer function: [4](#0-3) 

The contract does not inherit `Recoverable` (which exists elsewhere in the codebase at `contracts/utils/Recoverable.sol`). The sole token-exit path is `stakerGateway.stakeFor`, which requires a valid user address and a non-zero amount that does not exceed the balance — making the dust delta unroutable to any user without off-chain coordination and a separate bridge top-up.

---

### Impact Explanation

The slippage delta — up to `amount - minAmount` per bridge call — is permanently frozen inside `KernelReceiver`. Over multiple bridge operations the cumulative stuck balance grows. Users whose deposits contributed to the bridged batch receive fewer staked tokens than they deposited, and the shortfall cannot be recovered or redistributed without a contract upgrade.

**Impact: Critical — Permanent freezing of funds** (the delta tokens have no on-chain exit path in the current implementation).

---

### Likelihood Explanation

LayerZero OFT bridges routinely apply dust-rounding or fee deductions, making `delivered < sent` a normal operating condition rather than an edge case. The operator is expected to call `stakeFor` using the original deposit amounts recorded in `KernelVaultETH.userDeposits`, which are the pre-bridge amounts. The mismatch is therefore structurally guaranteed whenever `minAmount < amount` is used.

**Likelihood: Medium** — requires the operator to bridge with `minAmount < amount` (explicitly supported) and for the OFT to apply any fee or rounding.

---

### Recommendation

1. Add a token-rescue function to `KernelReceiver` (or inherit `Recoverable`) so the admin can recover stranded tokens:

```solidity
function recoverTokens(address token, address recipient, uint256 amount)
    external onlyRole(DEFAULT_ADMIN_ROLE) {
    IERC20(token).safeTransfer(recipient, amount);
}
```

2. Alternatively, track the exact amount delivered by the OFT (e.g., via a `lzReceive` hook or an on-chain balance snapshot before/after delivery) and use that figure — not the original deposit amount — when calling `stakeFor`.

3. Consider enforcing `minAmount == amount` (zero slippage tolerance) if the bridge supports it, eliminating the gap entirely.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Preconditions (local fork / unit test):
// - KernelVaultETH deployed on ETH fork; KernelReceiver deployed on BSC fork
// - User deposited N = 1000e18 KERNEL; operator calls bridgeKernelToBSC(N, minAmount=990e18, ...)
// - OFT bridge delivers 995e18 (within [minAmount, amount)) to KernelReceiver

function testSlippageDeltaFrozen() public {
    uint256 N      = 1000e18;
    uint256 delta  = 5e18;          // slippage: OFT delivers N - delta
    uint256 delivered = N - delta;

    // Simulate OFT delivery: mint delivered tokens to KernelReceiver
    deal(address(kernelBSC), address(kernelReceiver), delivered);

    // Operator reads original deposit amount N from KernelVaultETH.userDeposits
    vm.prank(operator);
    vm.expectRevert(KernelReceiver.InsufficientKernelBalance.selector);
    kernelReceiver.stakeFor(user, N);   // reverts: balance(995e18) < N(1000e18)

    // Operator adjusts to delivered amount — succeeds, but delta is now stranded
    vm.prank(operator);
    kernelReceiver.stakeFor(user, delivered);  // stakes 995e18

    // delta = 5e18 remains in KernelReceiver with no exit function
    assertEq(kernelBSC.balanceOf(address(kernelReceiver)), delta);

    // No recoverTokens / emergencyWithdraw exists — tokens are permanently frozen
    // (verified: grep for recoverTokens in KernelReceiver.sol returns 0 matches)
}
```
