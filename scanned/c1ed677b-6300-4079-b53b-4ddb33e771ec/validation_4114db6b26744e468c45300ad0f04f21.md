### Title
Asymmetric Deposit-Limit Check Allows ETH Deposits to Exceed the Configured Cap - (File: contracts/LRTDepositPool.sol)

---

### Summary

`_checkIfDepositAmountExceedesCurrentLimit` applies two structurally different comparisons depending on whether the deposited asset is native ETH or an ERC-20 LST. The ETH branch omits the incoming deposit amount from the comparison, so a depositor can push total ETH holdings above the configured `depositLimitByAsset` cap while the equivalent ERC-20 path correctly blocks the same action.

---

### Finding Description

In `LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` (lines 676–682):

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← no `amount`
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← includes `amount`
}
```

The ERC-20 branch correctly evaluates `totalAssetDeposits + amount > limit`, blocking any deposit that would push the running total over the cap. The ETH branch evaluates only `totalAssetDeposits > limit`, which is `false` whenever the running total is exactly at the cap. Because the check returns `false`, `_beforeDeposit` does not revert, and `depositETH` mints rsETH and accepts the ETH — pushing the total above the limit by the full `msg.value`.

The call chain is:

```
depositETH(minRSETHAmountExpected, referralId)
  └─ _beforeDeposit(ETH_TOKEN, msg.value, minRSETHAmountExpected)
       └─ _checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, msg.value)
            └─ returns false when totalAssetDeposits == depositLimit  ← bypass
```

`getAssetCurrentLimit` (lines 402–409) correctly returns `0` when `totalAssetDeposits == limit`, so the view function and the enforcement function are inconsistent with each other — the same structural asymmetry as the external report's scalar-vs-vector kappa divergence. ### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

The deposit cap is a protocol-level risk-management invariant. Any user can call `depositETH` with an arbitrary `msg.value` at the exact moment `totalAssetDeposits == depositLimit`, bypassing the cap by up to `msg.value`. The protocol accepts ETH and mints rsETH beyond the intended ceiling. No direct fund theft occurs in a single transaction, but the invariant "total ETH deposits ≤ depositLimitByAsset(ETH)" is violated, which can compound with EigenLayer slashing exposure or strategy capacity limits to produce downstream insolvency risk.

---

### Likelihood Explanation

Any unprivileged depositor can trigger this. The precondition — `totalAssetDeposits == depositLimit` — is a natural boundary condition that occurs whenever the pool approaches its cap. Because `depositETH` is a public payable function with no access control, the window is reachable by any ETH holder without any special privilege or front-running requirement. [3](#0-2) 

---

### Recommendation

Include the incoming deposit amount in the ETH branch, mirroring the ERC-20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This makes both code paths enforce the same invariant and aligns the enforcement logic with what `getAssetCurrentLimit` already reports. [5](#0-4) 

---

### Proof of Concept

1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 100 ether`.
2. Existing deposits bring `getTotalAssetDeposits(ETH_TOKEN)` to exactly `100 ether`.
3. `getAssetCurrentLimit(ETH_TOKEN)` returns `0` — the cap appears full.
4. Attacker calls `depositETH{value: 10 ether}(0, "")`.
5. Inside `_checkIfDepositAmountExceedesCurrentLimit`: `totalAssetDeposits (100e18) > depositLimit (100e18)` → `false`. No revert.
6. `_mintRsETH` mints rsETH for the attacker; total ETH deposits are now `110 ether`, 10% above the configured cap.
7. The same call with an ERC-20 LST at the same boundary would revert with `MaximumDepositLimitReached` because `100e18 + 10e18 > 100e18` → `true`. [1](#0-0) [6](#0-5)
