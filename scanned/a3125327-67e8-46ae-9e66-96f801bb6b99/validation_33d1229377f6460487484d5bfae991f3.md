### Title
No Minimum Output Amount (Slippage Protection) in L2 Pool Deposit Functions - (File: contracts/pools/RSETHPool.sol, contracts/pools/RSETHPoolV3.sol)

### Summary
The `deposit()` functions in `RSETHPool.sol` and `RSETHPoolV3.sol` do not accept a minimum output amount parameter, leaving L2 depositors exposed to receiving fewer rsETH tokens than expected when the oracle rate changes between transaction submission and execution.

### Finding Description
The `deposit(string memory referralId)` (ETH) and `deposit(address token, uint256 amount, string memory referralId)` (ERC20) functions in both `RSETHPool.sol` and `RSETHPoolV3.sol` compute the rsETH output amount at execution time using the live oracle rate, with no user-supplied floor:

```solidity
// RSETHPool.sol L265-278
function deposit(string memory referralId) external payable nonReentrant whenNotPaused {
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
``` [1](#0-0) 

The rate is fetched from an external oracle:

```solidity
// RSETHPool.sol L311-319
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
``` [2](#0-1) 

The same pattern is present in `RSETHPoolV3.sol`: [3](#0-2) [4](#0-3) 

By contrast, the L1 `LRTDepositPool` explicitly enforces a caller-supplied minimum:

```solidity
// LRTDepositPool.sol L667-669
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [5](#0-4) 

The L2 pool deposits have no equivalent guard.

### Impact Explanation
When the rsETH/ETH oracle rate rises between the block a user signs their transaction and the block it is included, the user receives fewer rsETH tokens than they observed in the preview. The ETH is consumed in full; the shortfall is silently absorbed. This matches the **Low** impact tier: *"Contract fails to deliver promised returns, but doesn't lose value."*

### Likelihood Explanation
The rsETH oracle rate (`getRate()`) updates with every oracle refresh cycle. On L2 chains with fast block times, the rate can shift between the moment a user calls `viewSwapRsETHAmountAndFee` off-chain and the moment the transaction lands on-chain. No adversarial action is required; ordinary market movement is sufficient. Likelihood is **Medium**.

### Recommendation
Add a `minRsETHAmountExpected` parameter to both `deposit()` overloads in `RSETHPool.sol`, `RSETHPoolV2.sol`, and `RSETHPoolV3.sol`, and revert if the computed `rsETHAmount` falls below it — mirroring the pattern already used in `LRTDepositPool._beforeDeposit()`.

### Proof of Concept
1. User queries `RSETHPool.viewSwapRsETHAmountAndFee(1 ether)` off-chain and observes `rsETHAmount = 0.95 rsETH` at the current oracle rate.
2. User submits `deposit{value: 1 ether}("ref")`.
3. Before the transaction is mined, the oracle rate ticks upward (rsETH appreciates).
4. At execution time `viewSwapRsETHAmountAndFee` returns `rsETHAmount = 0.93 rsETH`.
5. The user receives `0.93 rsETH` — 2% less than expected — with no revert and no recourse, because there is no minimum-output guard in the function.
