### Title
Dust Withdrawal Requests Can Bloat the Queue and Temporarily Freeze Legitimate Users' Funds - (`contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTWithdrawalManager.initiateWithdrawal()` has no effective minimum amount enforcement when `minRsEthAmountToWithdraw[asset]` is at its default value of `0`. An unprivileged rsETH holder can create arbitrarily many near-zero withdrawal requests, bloating the FIFO queue. Because `_unlockWithdrawalRequests()` iterates strictly in nonce order with no ability to skip entries, all dust requests inserted before a legitimate user's request must be processed first, temporarily freezing the legitimate user's funds.

---

### Finding Description

**Root cause — no effective dust floor in `initiateWithdrawal`:**

`minRsEthAmountToWithdraw` is a plain mapping whose default value is `0`:

```solidity
mapping(address asset => uint256) public minRsEthAmountToWithdraw;
``` [1](#0-0) 

The guard in `initiateWithdrawal` is:

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
``` [2](#0-1) 

When `minRsEthAmountToWithdraw[asset] == 0` (the default for every asset until an admin explicitly calls `setMinRsEthAmountToWithdraw`), the condition reduces to `rsETHUnstaked == 0`, so any amount ≥ 1 wei is accepted. There is no on-chain guarantee that a meaningful floor is ever set.

**Queue insertion — every request occupies a sequential nonce:**

Each accepted call appends a new entry to the global nonce counter for that asset:

```solidity
userAssociatedNonces[asset][msg.sender].pushBack(nextUnusedNonce_);
nextUnusedNonce[asset] = nextUnusedNonce_ + 1;
``` [3](#0-2) 

An attacker who submits N dust requests before a legitimate user's request occupies nonces `[0 … N-1]`; the legitimate request lands at nonce `N`.

**Settlement loop — strictly sequential, no skip mechanism:**

`_unlockWithdrawalRequests` advances `nextLockedNonce_` one step at a time:

```solidity
while (nextLockedNonce_ < firstExcludedIndex) {
    bytes32 requestId = getRequestId(asset, nextLockedNonce_);
    WithdrawalRequest storage request = withdrawalRequests[requestId];

    if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) break;

    uint256 payoutAmount = _calculatePayoutAmount(request, rsETHPrice, assetPrice);
    if (availableAssetAmount < payoutAmount) break;

    ...
    unchecked { nextLockedNonce_++; }
}
nextLockedNonce[asset] = nextLockedNonce_;
``` [4](#0-3) 

There is no mechanism to skip a nonce. The operator-supplied `firstExcludedIndex` only sets an upper bound; it cannot jump over dust entries that sit between `nextLockedNonce` and the legitimate request's nonce. Every dust entry must be individually processed before `nextLockedNonce` advances past it.

**Completing the attack path:**

1. Attacker acquires a small amount of rsETH (or already holds it).
2. Attacker calls `initiateWithdrawal(asset, 1_wei, "")` N times, each accepted because `minRsEthAmountToWithdraw[asset] == 0`.
3. Legitimate user calls `initiateWithdrawal` and receives nonce `N`.
4. After the 8-day delay, the operator calls `unlockQueue`. The loop must iterate through all N dust entries before it can unlock nonce `N`.
5. The legitimate user's `completeWithdrawal` reverts with `WithdrawalLocked` until the operator has processed all N dust entries across potentially many `unlockQueue` calls. [5](#0-4) 

---

### Impact Explanation

Legitimate users' withdrawal requests are **temporarily frozen**: `completeWithdrawal` reverts with `WithdrawalLocked` until the operator has drained all preceding dust entries from the queue. The operator must issue many `unlockQueue` calls (each bounded by gas), multiplying operational cost and delay proportionally to the number of dust entries. This maps directly to the "temporary freezing of funds" impact class (Medium).

---

### Likelihood Explanation

- `minRsEthAmountToWithdraw` is `0` by default for every asset; no deployment script or initializer sets it, so the vulnerability is active unless an admin proactively configures it.
- The attacker needs rsETH (a real economic cost) and gas per call, but the minimum rsETH per request is 1 wei, making the per-entry cost negligible relative to the delay imposed.
- The 8-day `withdrawalDelayBlocks` means the attacker must pre-position dust requests before the victim, but this is easy to do continuously.
- The attack requires no privileged access and is reachable by any rsETH holder.

---

### Recommendation

1. **Enforce a non-zero default minimum** in the initializer:
   ```solidity
   minRsEthAmountToWithdraw[ethToken] = MIN_WITHDRAWAL_FLOOR; // e.g. 0.001 ether
   ```
2. **Require the admin to set a floor before an asset is activated** for withdrawals, rather than relying on a post-deployment call.
3. Alternatively, add a **per-block or per-address rate limit** on `initiateWithdrawal` to prevent rapid queue flooding.

### Proof of Concept

```
// Precondition: minRsEthAmountToWithdraw[ETH_TOKEN] == 0 (default)

// Step 1: Attacker creates 5000 dust withdrawal requests
for (uint i = 0; i < 5000; i++) {
    withdrawalManager.initiateWithdrawal(ETH_TOKEN, 1 wei, "");
    // Each call accepted; nonces 0..4999 consumed
}

// Step 2: Victim creates a legitimate withdrawal request
// Victim's request lands at nonce 5000
withdrawalManager.initiateWithdrawal(ETH_TOKEN, 10 ether, "");

// Step 3: After 8-day delay, operator calls unlockQueue
// firstExcludedIndex = 5001; loop must iterate 5001 times
// Each call to unlockQueue can process ~1000 entries before gas limit
// Operator needs ≥5 separate unlockQueue calls to reach nonce 5000

// Step 4: Until all 5000 dust entries are processed, victim's completeWithdrawal reverts:
// "WithdrawalLocked" — nonce 5000 >= nextLockedNonce
withdrawalManager.completeWithdrawal(ETH_TOKEN, ""); // reverts

    function _unlockWithdrawalRequests(
        address asset,
        uint256 availableAssetAmount,
        uint256 rsETHPrice,
        uint256 assetPrice,
        uint256 firstExcludedIndex
    )
        internal
        returns (uint256 rsETHAmountToBurn, uint256 assetAmountToUnlock)
    {
        // Check that upper limit is in the range of existing withdrawal requests. If it is greater set it to the first
        // nonce with no withdrawal request.
        if (firstExcludedIndex > nextUnusedNonce[asset]) {
            firstExcludedIndex = nextUnusedNonce[asset];
        }

        uint256 nextLockedNonce_ = nextLockedNonce[asset];
        // Revert when trying to unlock a request that has already been unlocked
        if (nextLockedNonce_ >= firstExcludedIndex) revert NoPendingWithdrawals();

        while (nextLockedNonce_ < firstExcludedIndex) {
            bytes32 requestId = getRequestId(asset, nextLockedNonce_);
            WithdrawalRequest storage request = withdrawalRequests[requestId];

            // Check that the withdrawal delay has passed since the request's initiation.
            if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) break;

            // Calculate the amount user will receive
            uint256 payoutAmount = _calculatePayoutAmount(request, rsETHPrice, assetPrice);

            if (availableAssetAmount < payoutAmount) break; // Exit if not enough assets to cover this request

            assetsCommitted[asset] -= request.expectedAssetAmount;
            // Set the amount the user will receive
            request.expectedAssetAmount = payoutAmount;
            rsETHAmountToBurn += request.rsETHUnstaked;
            availableAssetAmount -= payoutAmount;
            assetAmountToUnlock += payoutAmount;

            unlockedWithdrawalsCount[asset]++;

            unchecked {
                nextLockedNonce_++;
            }
        }
        nextLockedNonce[asset] = nextLockedNonce_;
    }
```
