### Title
Stale `rsETHPrice` Silently Propagated Through `RSETHPriceFeed.latestRoundData()` When LRTOracle Auto-Pauses on Price Drop - (`contracts/oracles/RSETHPriceFeed.sol`)

---

### Summary

When `LRTOracle._updateRsETHPrice()` detects a price drop beyond `pricePercentageLimit`, it pauses the contract and returns early **without updating `rsETHPrice`**. The public `updateRSETHPrice()` is then blocked by `whenNotPaused`. `RSETHPriceFeed.latestRoundData()` reads the now-stale `rsETHPrice` but returns the ETH/USD Chainlink feed's `updatedAt` timestamp — which is always fresh — masking the staleness from any downstream consumer.

---

### Finding Description

**Root cause — `_updateRsETHPrice()` returns without writing `rsETHPrice`:** [1](#0-0) 

When `isPriceDecreaseOffLimit` is true, the function pauses and `return`s at line 281. The assignment `rsETHPrice = newRsETHPrice` at line 313 is never reached, so `rsETHPrice` is frozen at its pre-drop (higher) value.

**Root cause — `updateRSETHPrice()` is gated by `whenNotPaused`:** [2](#0-1) 

Once paused, no unprivileged caller can invoke `_updateRsETHPrice()`. Only `updateRSETHPriceAsManager()` (manager-only) bypasses the pause. [3](#0-2) 

**Root cause — `RSETHPriceFeed.latestRoundData()` returns stale rsETH price with fresh ETH/USD timestamp:** [4](#0-3) 

`updatedAt` is sourced from `ETH_TO_USD.latestRoundData()` (line 68) — it reflects when Chainlink last updated ETH/USD, not when `rsETHPrice` was last written. Any downstream consumer checking `updatedAt` for staleness will see a fresh timestamp while receiving a stale (inflated) rsETH/USD answer.

---

### Impact Explanation

The stale `rsETHPrice` is **higher** than the actual current price (the price dropped but the stored value was not updated). `RSETHPriceFeed.latestRoundData()` therefore returns an inflated rsETH/USD answer with a fresh-looking `updatedAt`. Downstream integrations (e.g., Aave using this feed as a collateral oracle, or yield distributors computing rsETH-denominated payouts) cannot distinguish this from a live price. During the pause window — which has no on-chain time bound and requires privileged admin/manager action to exit — all yield claims or collateral valuations that depend on a live rsETH/USD price are either frozen (if the consumer has its own change-detection) or computed against an incorrect inflated price.

---

### Likelihood Explanation

- `pricePercentageLimit` is a deployed, settable parameter (`setPricePercentageLimit` is callable by admin).
- LST slashing or a correlated asset price drop is a realistic market event, not an exotic attack.
- `updateRSETHPrice()` is a public, permissionless function — any caller (keeper, MEV bot, or the attacker themselves) triggers the pause once the price has dropped.
- No attacker capability beyond calling a public function is required.

---

### Recommendation

1. **Expose an rsETH-specific `updatedAt`**: Store a `rsETHPriceUpdatedAt` timestamp in `LRTOracle` and update it every time `rsETHPrice` is written. Have `RSETHPriceFeed.latestRoundData()` return `min(ethUsdUpdatedAt, rsETHPriceUpdatedAt)` so downstream staleness checks reflect the true age of the rsETH component.

2. **Allow price refresh during pause via a separate unprivileged path**: Consider allowing `updateRSETHPrice()` to write the new (lower) price even when paused, so the stored value reflects reality, even if deposits/withdrawals remain blocked.

3. **Alternatively, revert in `latestRoundData()` when paused**: If `RS_ETH_ORACLE` is paused, `RSETHPriceFeed.latestRoundData()` should revert rather than silently return a stale answer, forcing downstream consumers to handle the unavailability explicitly.

---
