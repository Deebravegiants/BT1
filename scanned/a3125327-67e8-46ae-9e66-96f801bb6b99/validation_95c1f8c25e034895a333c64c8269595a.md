### Title
Missing `minAmountOut` and `deadline` Parameters in `deposit()` Functions Allow Stale Oracle Rate Execution - (File: contracts/pools/RSETHPool.sol)

### Summary
All `deposit()` functions across the RSETHPool family accept ETH or ERC-20 tokens and return rsETH/wrsETH at the current oracle rate, but provide no `minAmountOut` (slippage guard) and no `deadline` parameter. A transaction left pending in the mempool will execute at a later, higher rsETH/ETH oracle rate, giving the depositor fewer rsETH tokens than they expected at submission time.

### Finding Description
Every public-facing `deposit()` entry point computes the rsETH output at execution time by calling `viewSwapRsETHAmountAndFee()`, which reads the live oracle rate via `getRate()`. Because rsETH is a yield-bearing token, its ETH-denominated rate is monotonically increasing. A user who submits a deposit transaction expecting rate R will receive fewer rsETH if the transaction is included in a block after the rate has risen to R′ > R. No parameter exists to let the user express a minimum acceptable output or a hard expiry for the transaction.

Affected entry points (same pattern in every pool variant):

- `RSETHPool.deposit(string)` / `RSETHPool.deposit(address,uint256,string)` [1](#0-0) 
- `RSETHPoolNoWrapper.deposit(string)` / `RSETHPoolNoWrapper.deposit(address,uint256,string)` [2](#0-1) 
- `RSETHPoolV2.deposit(string)` [3](#0-2) 
- `RSETHPoolV2NBA.deposit(string)` [4](#0-3) 
- `RSETHPoolV2ExternalBridge.deposit(string)` [5](#0-4) 
- `RSETHPoolV3.deposit(string)` / `RSETHPoolV3.deposit(address,uint256,string)` [6](#0-5) 

The rate computation in every variant follows the same pattern — no floor is enforced on the output:

```solidity
// RSETHPool.sol – viewSwapRsETHAmountAndFee (ETH path)
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;
uint256 rsETHToETHrate = getRate();          // live oracle, read at execution time
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [7](#0-6) 

### Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

The depositor's ETH is not stolen; it remains in the pool. However, the user receives fewer rsETH tokens than the rate visible at submission time would have implied. Because the rsETH/ETH rate rises continuously with staking yield, every mempool delay is a guaranteed, if small, shortfall. The user has no on-chain mechanism to reject execution once the rate has moved beyond their tolerance.

### Likelihood Explanation
Any depositor on a congested L2 or during a gas-price spike can have their transaction sit in the mempool for minutes to hours. PoS block proposers are known in advance (6–13 minutes), giving them the ability to selectively delay inclusion. The rsETH rate accrues continuously, so even a short delay produces a measurable, if modest, loss.

### Recommendation
Add a `minRsETHAmount` (minimum acceptable output) and a `deadline` parameter to every `deposit()` overload. Revert if the computed `rsETHAmount` is below `minRsETHAmount` or if `block.timestamp > deadline`. Example:

```solidity
function deposit(
    string memory referralId,
    uint256 minRsETHAmount,
    uint256 deadline
) external payable nonReentrant whenNotPaused {
    if (block.timestamp > deadline) revert DeadlineExpired();
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmount) revert InsufficientOutput();
    ...
}
```

### Proof of Concept
1. Alice calls `RSETHPool.deposit{value: 1 ether}("ref")` when `getRate()` = 1.05 ETH/rsETH, expecting ≈ 0.952 rsETH.
2. The transaction sits in the mempool for 30 minutes (network congestion or deliberate validator delay).
3. The oracle rate has risen to 1.06 ETH/rsETH by the time the transaction is included.
4. Alice receives `1e18 * 1e18 / 1.06e18` ≈ 0.943 rsETH — roughly 0.009 rsETH less than expected, with no on-chain recourse.
5. Alice had no way to set a deadline or a minimum output to prevent this outcome. [8](#0-7) [7](#0-6)

        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
