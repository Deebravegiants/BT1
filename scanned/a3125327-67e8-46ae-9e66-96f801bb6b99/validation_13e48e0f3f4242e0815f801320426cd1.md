### Title
`LRTConfig.setProtocolFeeBps()` Does Not Settle Accrued Yield Before Changing Fee Rate — (`contracts/LRTConfig.sol`)

---

### Summary

`LRTConfig.setProtocolFeeBps()` changes the protocol fee rate used in `LRTOracle._updateRsETHPrice()` without first calling `updateRSETHPrice()` to settle the yield that accrued under the old rate. This causes the new fee rate to be applied retroactively to yield that accrued during the prior period, mirroring the FraxlendPair `changeFee()` bug exactly.

---

### Finding Description

`LRTOracle._updateRsETHPrice()` computes the protocol fee on every call as:

```solidity
uint256 rewardAmount = totalETHInProtocol - previousTVL;
protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
``` [1](#0-0) 

`previousTVL` is reconstructed from `rsETHPrice` (the last stored price) and the current `rsethSupply`. The entire yield since the last price update is taxed at whatever `protocolFeeInBPS` is at the moment `_updateRsETHPrice()` executes. [2](#0-1) 

`protocolFeeInBPS` is stored in `LRTConfig` and is changed by:

```solidity
function setProtocolFeeBps(uint256 _protocolFeeInBPS) external onlyRole(LRTConstants.MANAGER) {
    if (_protocolFeeInBPS > 1500) revert ProtocolFeeExceedsLimit();
    protocolFeeInBPS = _protocolFeeInBPS;
    emit UpdateFee(_protocolFeeInBPS);
}
``` [3](#0-2) 

There is no call to `updateRSETHPrice()` (or any equivalent settlement) inside `setProtocolFeeBps()`. The timeline of the bug:

- **T1**: `rsETHPrice` is last updated; `protocolFeeInBPS = F1`.
- **T2**: Manager calls `setProtocolFeeBps(F2)` (e.g., a fee increase). No price update occurs.
- **T3**: Anyone calls `updateRSETHPrice()`. The entire yield from T1→T3 is taxed at `F2`, even though only the T2→T3 portion should use `F2`. The T1→T2 portion should have been taxed at `F1`. [4](#0-3) 

---

### Impact Explanation

**Fee increase (F2 > F1):** The protocol treasury receives more rsETH than it is entitled to for the T1→T2 period. This excess comes directly from rsETH holders' yield — it is a retroactive overcharge. This constitutes **theft of unclaimed yield** (High severity).

**Fee decrease (F2 < F1):** The treasury receives less than intended for the T1→T2 period. This is a loss of protocol revenue — **contract fails to deliver promised returns** (Low severity).

The fee can be up to 15% (`1500` BPS cap), so the magnitude of the misallocation scales with both the fee delta and the length of time between the last price update and the fee change. [3](#0-2) 

---

### Likelihood Explanation

`updateRSETHPrice()` is a public function with no access restriction — it can be called by anyone at any time. However, there is no on-chain enforcement that it must be called atomically before `setProtocolFeeBps()`. A manager who is unaware of this dependency (as in the FraxlendPair case) will simply call `setProtocolFeeBps()` directly. The longer the interval since the last price update, the larger the misallocated fee amount. This is a realistic operational scenario, not a theoretical one. [5](#0-4) 

---

### Recommendation

Add a call to `_updateRsETHPrice()` (or expose a settlement hook) inside `setProtocolFeeBps()`, or alternatively enforce the settlement inside `LRTOracle` itself. The simplest fix mirrors the FraxlendPair recommendation:

```solidity
// In LRTConfig.sol — or better, in a wrapper that the manager must use:
function setProtocolFeeBps(uint256 _protocolFeeInBPS) external onlyRole(LRTConstants.MANAGER) {
    if (_protocolFeeInBPS > 1500) revert ProtocolFeeExceedsLimit();

    // Settle accrued yield at the current fee rate before changing it
    ILRTOracle(getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice(); // +++ ADD THIS

    protocolFeeInBPS = _protocolFeeInBPS;
    emit UpdateFee(_protocolFeeInBPS);
}
``` [6](#0-5) 

---

### Proof of Concept

1. Protocol is live; `rsETHPrice = 1.05e18`, `protocolFeeInBPS = 500` (5%). Yield has been accruing for 7 days since the last price update; `totalETHInProtocol - previousTVL = 100 ETH`.
2. Manager calls `LRTConfig.setProtocolFeeBps(1000)` (10%) without first calling `updateRSETHPrice()`.
3. Any user calls `LRTOracle.updateRSETHPrice()`.
4. Inside `_updateRsETHPrice()`: `rewardAmount = 100 ETH`, `protocolFeeInETH = 100 * 1000 / 10_000 = 10 ETH`.
5. Correct calculation: 5 ETH should have been taken at 5% for the 7-day period, and 0 ETH at 10% (since the fee just changed). Instead, 10 ETH is taken — **5 ETH of excess fee is minted to the treasury at the expense of rsETH holders**. [7](#0-6)
