### Title
Direct altAgETH Transfer Inflates `balanceOf` Without Minting, Causing `depositBridgerAssets` to Revert — (`contracts/agETH/AGETHTokenWrapper.sol`)

---

### Summary

`maxAmountToDepositBridgerAsset` computes the bridger's deposit cap as `totalSupply() - balanceOf(contract)`. Because `balanceOf` can be inflated by a plain ERC20 `transfer` to the contract (no role required), any altAgETH holder can reduce or zero-out this cap, causing every subsequent `depositBridgerAssets` call to revert with `CannotDeposit`.

---

### Finding Description

The cap logic in `maxAmountToDepositBridgerAsset` is:

```
cap = totalSupply() - balanceOf(address(this))   // returns 0 if balance > supply
``` [1](#0-0) 

`totalSupply()` only increases through `_deposit` (user wraps) or `mint` (MINTER_ROLE pre-mints for bridging). `balanceOf(address(this))` increases through `_deposit` **and** through any raw ERC20 `transfer` to the contract address. There is no mechanism to sweep or account for "donated" tokens.

The intended flow for the bridger is:

1. MINTER_ROLE calls `mint(to, K)` — mints K agETH on L2 without backing. `supply = K`, `balance = 0`.
2. BRIDGER_ROLE calls `depositBridgerAssets(altAgETH, K)` — deposits K altAgETH as backing. `balance = K`, `supply = K`. [2](#0-1) [3](#0-2) 

If an external party directly transfers M altAgETH to the contract before or after step 1, `balance = M` while `supply = K`. The cap becomes `max(0, K - M)`. When M ≥ K the cap is 0 and `depositBridgerAssets(altAgETH, K)` reverts unconditionally. When M < K the cap is `K - M`, so `depositBridgerAssets(altAgETH, K)` still reverts because `K - M < K`. [4](#0-3) 

---

### Impact Explanation

- **Scope**: Low — Contract fails to deliver promised returns, but doesn't lose value.
- The BRIDGER_ROLE's `depositBridgerAssets` call reverts; the bridger cannot deposit the K altAgETH it holds as backing.
- The K agETH already minted via `mint` is still redeemable: when M ≥ K the contract already holds M ≥ K altAgETH, so `withdraw` succeeds for all K agETH holders. No funds are lost.
- However, the bridger's K altAgETH is stranded (cannot be deposited), and the `depositBridgerAssets` function fails to fulfil its documented purpose of collateralising pre-minted agETH.
- The block persists until MINTER_ROLE mints enough additional agETH to push `totalSupply` above `balanceOf` again — a perverse operational workaround.

---

### Likelihood Explanation

- Requires only a standard ERC20 `transfer` to the contract address — no role, no approval beyond the sender's own balance.
- Can occur accidentally (user sends to wrong address) or intentionally (griefing the bridger).
- The altAgETH token is a bridged asset with real liquidity, so the precondition is easily satisfied.

---

### Recommendation

Track "legitimate" altAgETH deposits in a storage variable rather than relying on live `balanceOf`. Replace the cap calculation with:

```solidity
// storage variable incremented in _deposit and depositBridgerAssets, decremented in _withdraw
uint256 public trackedBalance;

function maxAmountToDepositBridgerAsset(address _asset) public view returns (uint256) {
    if (!allowedTokens[_asset]) return 0;
    uint256 supply = totalSupply();
    if (trackedBalance >= supply) return 0;
    return supply - trackedBalance;
}
```

This makes the cap immune to direct transfers.
