### Title
Unrestricted `sendFunds()` in FeeReceiver Allows Anyone to Trigger Reward Distribution - (File: contracts/FeeReceiver.sol)

### Summary
`FeeReceiver.sendFunds()` lacks any access modifier, allowing any external caller to force the entire MEV/execution-layer reward balance into the deposit pool at an arbitrary time, bypassing any intended operational authorization over reward distribution timing.

### Finding Description
`FeeReceiver.sendFunds()` is declared `external` with no role check:

```solidity
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
``` [1](#0-0) 

Every other state-mutating function in `FeeReceiver` is gated: `setDepositPool` requires `MANAGER` role. [2](#0-1) 

`sendFunds()` is the only function that moves funds, yet it has no restriction. The contract is explicitly described as receiving MEV/execution-layer rewards, and the function is the sole mechanism for forwarding those rewards into the protocol TVL. [3](#0-2) 

### Impact Explanation
Any external caller can invoke `sendFunds()` at any time, flushing the entire accumulated reward balance into `LRTDepositPool`. This bypasses any intended operational control over when rewards are added to TVL. Because `LRTOracle._updateRsETHPrice()` computes `totalETHInProtocol` from the deposit pool's live ETH balance, a forced flush immediately before a price update inflates the apparent reward amount, potentially pushing the new rsETH price above `pricePercentageLimit` and causing `updateRSETHPrice()` to revert for non-manager callers. [4](#0-3) 

Funds are not lost — they reach the correct destination — but the protocol loses control over the timing of reward accounting, which is the operational invariant the authorization was meant to protect.

**Impact**: Low — Contract fails to deliver promised returns (controlled reward timing), but does not lose value.

### Likelihood Explanation
The function is `external` with no guard. Any EOA or contract can call it at zero cost. No special knowledge or setup is required. Likelihood is high.

### Recommendation
Add an access modifier restricting `sendFunds()` to an authorized role, consistent with the rest of the contract's access model:

```solidity
function sendFunds() external onlyRole(LRTConstants.MANAGER) {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

### Proof of Concept
1. MEV rewards accumulate in `FeeReceiver` over time.
2. Attacker observes a large accumulated balance.
3. Attacker calls `FeeReceiver.sendFunds()` directly — no role required, call succeeds.
4. The full balance is transferred to `LRTDepositPool` via `receiveFromRewardReceiver`.
5. If `LRTOracle.updateRSETHPrice()` is called immediately after, `totalETHInProtocol` is inflated by the flushed rewards, potentially exceeding `pricePercentageLimit` and reverting for non-manager callers, disrupting normal price update operations. [1](#0-0) [5](#0-4)
