### Title
Off-by-One in `setSlippageTolerance` Allows `slippageTolerance = BASIS_POINTS_DIVISOR`, Permanently Reverting All Bridge Calls Until Reset — (`contracts/bridges/TACWETHBridge.sol`)

---

### Summary

`setSlippageTolerance` uses a strict `>` comparison against `BASIS_POINTS_DIVISOR`, allowing the value `10_000` to be stored. When `slippageTolerance == 10_000`, `getMinAmount` computes `minAmount = 0` and reverts with `InvalidMinAmount`, causing every call to `bridgeTokenToL1` and `getNativeFee` to revert until the admin corrects the value.

---

### Finding Description

`BASIS_POINTS_DIVISOR` is defined as `10_000`. [1](#0-0) 

The validation in `setSlippageTolerance` uses a strict greater-than check: [2](#0-1) 

Because the check is `> BASIS_POINTS_DIVISOR` (not `>=`), the value `10_000` passes validation and is stored in `slippageTolerance`. The same off-by-one exists in the constructor. [3](#0-2) 

`getMinAmount` then computes:

```
minAmount = amount * (10_000 - 10_000) / 10_000 = 0
```

and immediately reverts: [4](#0-3) 

Both `bridgeTokenToL1` and `getNativeFee` call `getMinAmount` unconditionally, so both revert for any nonzero `amount` while this state persists. [5](#0-4) [6](#0-5) 

---

### Impact Explanation

All bridge transfers are blocked until the admin calls `setSlippageTolerance` again with a valid value. No funds are lost (tokens are never transferred before the revert), but the contract fails to deliver its promised bridging service. This matches **Low — Contract fails to deliver promised returns, but doesn't lose value**.

---

### Likelihood Explanation

Triggering this requires the admin to call `setSlippageTolerance(10_000)`. This is not an admin compromise — it is an accidental misconfiguration made possible by the off-by-one bug in validation. A value of `10_000` basis points (100% slippage) is semantically nonsensical and should be rejected by the contract, but is not. The same bug exists in the constructor, meaning the contract could be deployed in this broken state from the start.

---

### Recommendation

Change both the constructor and `setSlippageTolerance` to use `>=` instead of `>`:

```solidity
// constructor
if (_slippageTolerance >= BASIS_POINTS_DIVISOR) revert InvalidSlippageTolerance();

// setSlippageTolerance
if (newSlippageTolerance >= BASIS_POINTS_DIVISOR) revert InvalidSlippageTolerance();
```

This ensures `slippageTolerance` can never equal `BASIS_POINTS_DIVISOR`, preserving the invariant that `getMinAmount` always returns a nonzero value for any nonzero input.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Assume TACWETHBridge is deployed with slippageTolerance = 100 (1%)
// Admin calls:
bridge.setSlippageTolerance(10_000); // passes validation, no revert

// Now any user call reverts:
bridge.bridgeTokenToL1{value: fee}(recipient, 1 ether);
// → getMinAmount(1 ether) → minAmount = 0 → revert InvalidMinAmount

bridge.getNativeFee(1 ether, recipient);
// → getMinAmount(1 ether) → minAmount = 0 → revert InvalidMinAmount
```
