## Analysis

Let me trace the full attack path and verify each component.

### Step 1: Is `sendFunds()` permissionless?

`FeeReceiver.sendFunds()` has **no access control modifier**: [1](#0-0) 

Any EOA or contract can call it at any time.

### Step 2: What does `receiveFromRewardReceiver()` do? [2](#0-1) 

It is a bare `payable` function with no logic — it simply increases `address(this).balance` of `LRTDepositPool`.

### Step 3: How is rsETH minted on deposit?

`depositETH` → `_beforeDeposit` → `getRsETHAmountToMint`: [3](#0-2) 

The formula is:
```
rsethAmountToMint = (depositAmount * getAssetPrice(ETH)) / rsETHPrice()
```

`rsETHPrice()` returns the **stored, stale** value — it is only updated when `updateRSETHPrice()` is explicitly called. [4](#0-3) 

### Step 4: How does `getETHDistributionData` account for ETH? [5](#0-4) 

`ethLyingInDepositPool = address(this).balance` — it uses the **live balance**, which immediately reflects the reward transfer from `sendFunds()`.

### Step 5: The Math

Let:
- `T` = total ETH in protocol before attack
- `S` = rsETH supply
- `R` = accumulated rewards in FeeReceiver
- `D` = attacker's deposit amount
- Stored `rsETHPrice` = `T/S` (stale, not yet reflecting `R`)

**After `sendFunds()` but before `updateRSETHPrice()`:**
- DepositPool balance = `T + R`
- Stored `rsETHPrice` = `T/S` (unchanged)

**Attacker calls `depositETH(D)`:**
- rsETH minted = `D * 1e18 / (T/S)` = `D * S / T`

**After `updateRSETHPrice()` is called:**
- New TVL = `T + R + D`
- New supply = `S + D*S/T`
- New price = `(T+R+D) / (S + D*S/T)`

**Attacker's rsETH value:**
= `(D*S/T) * (T+R+D) / (S + D*S/T)`
= `D * (T+R+D) / (T+D)`
= `D + D*R/(T+D)` **> D**

The attacker captures `D*R/(T+D)` ETH of yield that belonged to existing holders.

**Existing holders' loss:** Without the attack, after oracle update their price would be `(T+R)/S`. With the attack it is `(T+R+D)/(S + D*S/T)`. Since `T < T+R`, the post-attack price is strictly lower — existing holders are diluted.

### Step 6: Does the `pricePercentageLimit` guard prevent this? [6](#0-5) 

This guard only triggers during `updateRSETHPrice()`, **not** during `depositETH()`. The deposit and rsETH minting happen entirely before the oracle update. The guard cannot prevent the yield capture.

### Step 7: Impact Classification

The protocol does **not** become insolvent — total ETH `(T+R+D)` still backs all rsETH `(S + D*S/T)`. The claimed impact of "protocol insolvency" is incorrect. The actual impact is **theft of unclaimed yield** from existing rsETH holders: the attacker front-runs the oracle update to capture MEV/EL rewards that should accrue pro-rata to all existing holders.

### Title
Permissionless `sendFunds()` Enables Front-Running of Oracle Update to Steal Unclaimed MEV/EL Yield from Existing rsETH Holders — (`contracts/FeeReceiver.sol`)

### Summary
`FeeReceiver.sendFunds()` has no access control, allowing any caller to flush accumulated MEV/EL rewards into `LRTDepositPool` at will. Because `LRTDepositPool.depositETH()` prices rsETH using the stale stored `rsETHPrice` (not the live TVL), a caller who triggers `sendFunds()` immediately before depositing receives rsETH priced against the pre-reward TVL, capturing a portion of the unaccounted rewards at the expense of existing holders.

### Finding Description
`FeeReceiver` accumulates MEV and execution-layer rewards as ETH. `sendFunds()` is a public, permissionless function that transfers the entire balance to `LRTDepositPool` via `receiveFromRewardReceiver()`. The deposit pool's `getETHDistributionData()` immediately reflects the new balance (`address(this).balance`), but `LRTOracle.rsETHPrice` is a stored value updated only on explicit calls to `updateRSETHPrice()`. The mint formula `rsethAmountToMint = depositAmount * getAssetPrice(ETH) / rsETHPrice()` therefore uses a stale denominator, issuing more rsETH per ETH deposited than the true post-reward exchange rate warrants. The excess rsETH is backed by the transferred rewards, diluting existing holders.

### Impact Explanation
**High — Theft of unclaimed yield.** Existing rsETH holders lose a fraction of accrued MEV/EL rewards proportional to `D*R/(T+D)` (where `D` is the attacker's deposit, `R` is the reward, `T` is prior TVL). The protocol remains solvent; no funds are frozen. The impact is yield redistribution from existing holders to the attacker-depositor.

### Likelihood Explanation
- `sendFunds()` is callable by any EOA with zero preconditions.
- FeeReceiver accumulates rewards continuously between oracle update cycles.
- The attack requires only two sequential transactions (or a single contract call) and no privileged access, governance capture, or external oracle manipulation.
- Likelihood is **High**.

### Recommendation
1. **Add access control to `sendFunds()`** — restrict it to `MANAGER` or `OPERATOR_ROLE` so only authorized parties can flush rewards.
2. Alternatively, **call `updateRSETHPrice()` atomically inside `receiveFromRewardReceiver()`** so the price is always current before any subsequent deposit can use it.
3. As a defense-in-depth measure, consider requiring `updateRSETHPrice()` to be called within the same block before `depositETH()` is accepted (a "freshness" check on the oracle timestamp).

### Proof of Concept
```solidity
// Fork test (Foundry)
// State: FeeReceiver holds R = 10 ETH in accumulated rewards
//        Protocol TVL T = 1000 ETH, rsETH supply S = 1000e18 (price = 1e18)

function testYieldTheft() external {
    uint256 R = 10 ether;
    uint256 D = 1 ether;

    // Attacker step 1: flush rewards into DepositPool (permissionless)
    feeReceiver.sendFunds();
    // DepositPool.balance is now 1010 ETH, rsETHPrice still = 1e18

    // Attacker step 2: deposit 1 ETH
    uint256 rsethBefore = rsETH.balanceOf(attacker);
    depositPool.depositETH{value: D}(0, "");
    uint256 rsethMinted = rsETH.balanceOf(attacker) - rsethBefore;
    // rsethMinted = 1e18 (1 ETH / 1e18 price = 1 rsETH)

    // Step 3: oracle update
    lrtOracle.updateRSETHPrice();
    // newPrice = (1010 + 1) / (1000 + 1) = 1011/1001 ≈ 1.00999e18

    // Attacker's rsETH value
    uint256 attackerValue = rsethMinted * lrtOracle.rsETHPrice() / 1e18;
    // ≈ 1.00999 ETH > 1 ETH deposited

    // Existing holder loss: pre-attack expected price = 1010/1000 = 1.01e18
    // Actual price after attack = 1.00999e18 < 1.01e18
    // Existing holders lost ~0.00001 ETH per rsETH

    assertGt(attackerValue, D, "attacker captured yield");
}
```
