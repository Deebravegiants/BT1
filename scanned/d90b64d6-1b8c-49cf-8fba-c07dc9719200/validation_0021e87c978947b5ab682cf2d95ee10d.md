### Title
Missing Decimal Normalization for Non-18-Decimal Tokens in `viewSwapRsETHAmountAndFee` Causes Depositors to Receive Near-Zero wrsETH - (File: `contracts/pools/RSETHPoolV3.sol`)

---

### Summary

In `RSETHPoolV3.viewSwapRsETHAmountAndFee(uint256 amount, address token)`, the rsETH minting formula `rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate` silently assumes the deposited token has 18 decimals. When a non-18-decimal token such as USDC (6 decimals) is a supported pool asset, the formula produces a result that is off by a factor of `10^(18 − tokenDecimals)`, causing depositors to receive essentially zero wrsETH while permanently surrendering their full token balance to the pool.

---

### Finding Description

`RSETHPoolV3.viewSwapRsETHAmountAndFee(uint256 amount, address token)` computes the wrsETH amount to mint for a given ERC-20 deposit:

```solidity
// contracts/pools/RSETHPoolV3.sol  L324-L334
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;

uint256 rsETHToETHrate = getRate();                                          // 1e18-scaled
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();     // 1e18-scaled

rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

`ChainlinkOracleForRSETHPoolCollateral.getRate()` normalises the Chainlink answer to a 1e18-scaled price of **one human-readable token unit** in ETH:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol  L34
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
```

For WETH (18 decimals) the formula is dimensionally consistent: `amountAfterFee` is in wei (1e18 units per ETH), and `tokenToETHRate` is `1e18`, so the division by `rsETHToETHrate` (also 1e18-scaled) yields a correct 1e18-scaled rsETH amount.

For USDC (6 decimals), `amountAfterFee` is in micro-USDC (1e6 units per USDC), but `tokenToETHRate` still represents the price of **one full USDC** in ETH scaled by 1e18. The formula therefore produces a result that is `10^12` times smaller than the correct value.

**Concrete example** (1 000 USDC, ETH price $3 000):

| Quantity | Value |
|---|---|
| `amountAfterFee` | `1 000 × 1e6 = 1e9` |
| `tokenToETHRate` (USDC/ETH, 1e18-scaled) | `≈ 3.33 × 10^14` |
| `rsETHToETHrate` | `≈ 1.05 × 10^18` |
| **Computed `rsETHAmount`** | `1e9 × 3.33e14 / 1.05e18 ≈ 317 000` (raw units) |
| **Correct `rsETHAmount`** | `≈ 0.317 rsETH = 3.17 × 10^17` (raw units) |

The depositor receives `317 000` wrsETH raw units (`≈ 3.17 × 10^−13` wrsETH) instead of `0.317 wrsETH` — a loss factor of `10^12`. Their 1 000 USDC is transferred into the pool and is irrecoverable by the depositor.

The same arithmetic error is present in every pool variant that supports non-ETH tokens:

- `RSETHPoolV3.sol` L334
- `RSETHPoolV3ExternalBridge.sol` L452
- `RSETHPoolV3WithNativeChainBridge.sol` L370
- `RSETHPool.sol` L346
- `RSETHPoolNoWrapper.sol` L311
- `AGETHPoolV3.sol` L194

---

### Impact Explanation

**Critical — Direct theft of user funds.**

Any depositor who calls `deposit(token, amount, referralId)` with a supported non-18-decimal token (e.g., USDC) loses their entire deposit. The token is transferred from the depositor to the pool (`safeTransferFrom` at L284), but the minted wrsETH is negligible (effectively zero). The depositor has no mechanism to recover the deposited tokens; they are pooled and eventually moved for bridging by the `BRIDGER_ROLE`. The loss is immediate and permanent upon each such deposit.

---

### Likelihood Explanation

**High.** The pool architecture is explicitly designed to accept arbitrary ERC-20 tokens via `addSupportedToken`. The `ChainlinkOracleForRSETHPoolCollateral` wrapper is purpose-built to support tokens with non-standard Chainlink decimal counts, confirming the protocol's intent to support non-18-decimal assets. USDC is the most common non-18-decimal collateral asset on every chain where these pools are deployed. Any ordinary depositor calling the public `deposit(address token, uint256 amount, string referralId)` function with USDC triggers the bug with no special preconditions.

---

### Recommendation

Normalise `amountAfterFee` to 18 decimals before applying the rate formula:

```solidity
uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 normalizedAmount = amountAfterFee * 10 ** (18 - tokenDecimals);
rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
```

Apply the same fix to `viewSwapAssetToPremintedRsETH` (reverse direction) and to all pool variants listed above.

---

### Proof of Concept

1. Admin calls `addSupportedToken(USDC, chainlinkUSDCOracle)` on `RSETHPoolV3`.
2. Attacker (or any user) approves the pool for 1 000 USDC and calls:
   ```solidity
   pool.deposit(USDC, 1_000e6, "ref");
   ```
3. Inside `deposit`, `safeTransferFrom` moves 1 000 USDC from the caller to the pool.
4. `viewSwapRsETHAmountAndFee(1_000e6, USDC)` computes:
   ```
   rsETHAmount = 1_000e6 * 3.33e14 / 1.05e18 ≈ 317_000   // raw wrsETH units
   ```
5. `wrsETH.mint(msg.sender, 317_000)` mints `≈ 3.17e-13` wrsETH — economically worthless.
6. The depositor has lost 1 000 USDC (≈ $1 000) and received nothing of value.
7. The USDC sits in the pool until `BRIDGER_ROLE` calls `moveAssetsForBridging(USDC, amount)`, at which point it is permanently removed from the depositor's reach. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
