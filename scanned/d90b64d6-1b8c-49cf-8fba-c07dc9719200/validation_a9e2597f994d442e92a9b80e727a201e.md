### Title
Unprotected `updateRSETHPrice()` Allows Anyone to Trigger Protocol-Wide Pause and Manipulate Fee Minting Timing - (File: `contracts/LRTOracle.sol`)

---

### Summary

`LRTOracle.updateRSETHPrice()` is declared `public whenNotPaused` with no role-based access control, allowing any unprivileged external caller to invoke it. This function updates the rsETH/ETH exchange rate stored in `rsETHPrice`, mints protocol fees as rsETH to the treasury, and — critically — can trigger a protocol-wide pause of `LRTDepositPool` and `LRTWithdrawalManager` if the computed price has dropped below the configured `pricePercentageLimit` threshold. A separate privileged entry point `updateRSETHPriceAsManager()` exists with `onlyLRTManager`, but the unprotected `updateRSETHPrice()` remains callable by anyone.

---

### Finding Description

In `contracts/LRTOracle.sol`, the function `updateRSETHPrice()` is declared as:

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [1](#0-0) 

The only guard is `whenNotPaused` — there is no role check. The internal `_updateRsETHPrice()` it delegates to performs three financially significant operations:

**1. Protocol fee minting**: When `totalETHInProtocol > previousTVL`, it computes a fee in ETH and mints rsETH to the treasury. Any caller can trigger this at any time. [2](#0-1) 

**2. Protocol-wide pause**: If the new rsETH price has dropped more than `pricePercentageLimit` below `highestRsethPrice`, the function pauses `LRTDepositPool`, `LRTWithdrawalManager`, and `LRTOracle` itself — freezing all deposits and withdrawals. [3](#0-2) 

**3. Price storage update**: `rsETHPrice` is updated and consumed by `LRTDepositPool.getRsETHAmountToMint()` to determine how much rsETH to mint per deposit. [4](#0-3) 

A privileged counterpart exists but does not replace the unprotected path:

```solidity
function updateRSETHPriceAsManager() external onlyLRTManager {
    _updateRsETHPrice();
}
``` [5](#0-4) 

The `rsETHPrice` stored value is directly used in `LRTDepositPool.getRsETHAmountToMint()` to compute how many rsETH tokens a depositor receives:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [6](#0-5) 

---

### Impact Explanation

**Primary — Temporary freezing of funds (Medium):** If the rsETH price has genuinely dropped below the `pricePercentageLimit` threshold from `highestRsethPrice` (e.g., due to an EigenLayer slashing event or a reduction in underlying LST value), any unprivileged attacker can call `updateRSETHPrice()` to immediately trigger a protocol-wide pause of `LRTDepositPool` and `LRTWithdrawalManager`. This freezes all user deposits and withdrawals until an admin manually unpauses. The attacker controls the timing — they can choose to trigger the pause at the most disruptive moment (e.g., during high withdrawal demand), even if the manager would have waited for conditions to stabilize or used `updateRSETHPriceAsManager()` to handle the price update without triggering the pause path.

**Secondary — Fee minting timing manipulation:** Any caller can trigger protocol fee minting at any time when TVL has increased. While the fee goes to the treasury (not the attacker), the caller bypasses the manager's intended control over when fees are crystallized, which can affect the rsETH price trajectory and the daily fee minting limit accounting tracked in `currentPeriodMintedFeeAmount`. [7](#0-6) 

---

### Likelihood Explanation

**Medium.** The pause-triggering path requires the rsETH price to have dropped below the `pricePercentageLimit` threshold from `highestRsethPrice`. This is a realistic scenario in production given EigenLayer slashing risk or LST depegs. The fee minting manipulation path is always available whenever TVL has increased since the last price update, which is the normal operating state of the protocol. No special privileges, flash loans, or oracle manipulation are required — a simple external call suffices.

---

### Recommendation

1. Remove the `public` `updateRSETHPrice()` entry point entirely, or restrict it with an access control modifier (e.g., `onlyLRTManager` or a dedicated `PRICE_UPDATER_ROLE`).
2. If a permissionless price update path is intentionally desired (e.g., for keeper bots), separate the pause-triggering logic from the permissionless path: only allow unprivileged callers to update the price when it is within safe bounds, and require a privileged role to trigger the pause path.
3. Rename the internal function to `_updateRsETHPrice` (already done) and ensure all external entry points are properly access-controlled, consistent with the existing `updateRSETHPriceAsManager()` pattern.

```solidity

// (e.g., due to EigenLayer slashing reducing total ETH in protocol)

// No role required — anyone can call this

// _updateRsETHPrice() computes newRsETHPrice < highestRsethPrice - threshold
// Execution reaches:

//   _pause(); // pauses LRTOracle itself
//   return;

// Result: LRTDepositPool.depositETH() and depositAsset() are now paused
// LRTWithdrawalManager withdrawals are paused
// All user funds are temporarily frozen until admin unpauses
```
