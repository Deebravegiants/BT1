### Title
Zero-rate oracle causes deposit revert on fresh deployment — (`contracts/agETH/AGETHRateReceiver.sol`, `contracts/agETH/AGETHPoolV3.sol`)

### Summary
`AGETHRateReceiver` inherits `CrossChainRateReceiver`, whose `rate` storage variable initializes to `0`. Before the first LayerZero message is received, `getRate()` returns `0`. `AGETHPoolV3.viewSwapAgETHAmountAndFee` divides by this value without a zero-guard, causing every `deposit()` call to revert with a division-by-zero panic.

### Finding Description
`CrossChainRateReceiver.rate` is a `uint256` that defaults to `0` at deployment. `getRate()` returns it unconditionally. `AGETHPoolV3.viewSwapAgETHAmountAndFee` computes `agETHAmount = amountAfterFee * 1e18 / agETHToETHrate` (line 168) and `agETHAmount = amountAfterFee * tokenToETHRate / agETHToETHrate` (line 194), both of which panic on division by zero. `initialize()` and `setAgETHOracle()` accept any non-zero oracle address without verifying `getRate() != 0`, unlike `addSupportedToken()` which does enforce this check for token oracles.

### Impact Explanation
All ETH and token deposits to `AGETHPoolV3` revert during the window between contract deployment and the first successful `lzReceive` call. The contract advertises `isEthDepositEnabled == true` but cannot fulfill deposits. No funds are lost (reverts return ETH to callers). Impact: **Low — contract fails to deliver promised returns, but doesn't lose value**.

### Likelihood Explanation
This is a near-certain occurrence on every fresh deployment. The LayerZero relayer is not instantaneous; there is always a gap between deployment and the first cross-chain rate update. Any user who attempts a deposit in that window will be silently rejected with a confusing panic revert.

### Recommendation
Add a zero-rate guard in `initialize()` and `setAgETHOracle()`, mirroring the existing check in `addSupportedToken()`:

```solidity
if (IOracle(_agETHOracle).getRate() == 0) revert UnsupportedOracle();
```

Alternatively, add a zero-check inside `viewSwapAgETHAmountAndFee` before dividing, and revert with a descriptive error (e.g., `OracleNotInitialized()`).

### Proof of Concept

```solidity
// Deploy AGETHRateReceiver (rate == 0, lzReceive never called)
// Deploy AGETHPoolV3 with agETHOracle = address(rateReceiver)
// isEthDepositEnabled == true after initialize()

vm.expectRevert(); // division-by-zero panic
pool.deposit{value: 1 ether}("ref");
// Reverts because viewSwapAgETHAmountAndFee divides by getRate() == 0
```
