### Title
Chainlink Oracle Accepts Stale/Invalid Price Data Without Staleness or Round Completeness Checks - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `price`, performing no staleness, round-completeness, or price-validity checks. This stale or zero price propagates directly into rsETH minting and the protocol-wide `rsETHPrice` update.

### Finding Description
`ChainlinkPriceOracle.getAssetPrice()` reads the Chainlink feed as follows:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

All five return values of `latestRoundData()` — `roundId`, `price`, `startedAt`, `updatedAt`, `answeredInRound` — are available, but only `price` is used. The following checks are entirely absent:

- **No heartbeat/staleness check**: `updatedAt` is ignored; a feed that stopped updating hours or days ago is silently accepted.
- **No round-completeness check**: `startedAt` is never verified to be non-zero; a round that never completed (Chainlink returns `startedAt == 0`) is accepted.
- **No stale-round check**: `answeredInRound >= roundId` is never verified; a price answered in a prior round is accepted.
- **No price-validity check**: `price` is never checked to be `> 0`; a zero price returns `0` from `getAssetPrice`, which collapses `totalETHInProtocol` to zero.

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` in the same repository correctly implements all three guards:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

`ChainlinkPriceOracle` is the oracle used for all LST assets (stETH, ETHx, swETH, rETH, etc.) registered in `LRTOracle`.

### Impact Explanation
The stale price flows through two critical paths:

**Path 1 — rsETH minting (depositor-facing):**
`LRTDepositPool.depositAsset()` → `getRsETHAmountToMint()` → `lrtOracle.getAssetPrice(asset)` → `ChainlinkPriceOracle.getAssetPrice()`. [3](#0-2) 

A stale inflated price causes depositors to receive more rsETH than the deposited asset is worth, diluting existing holders (theft of yield/value). A stale deflated price causes depositors to receive fewer rsETH than owed.

**Path 2 — Protocol-wide rsETH price update:**
`LRTOracle.updateRSETHPrice()` → `_updateRsETHPrice()` → `_getTotalEthInProtocol()` → `getAssetPrice(asset)` for every supported asset. [4](#0-3) 

If a stale price causes `totalETHInProtocol` to drop below the previous TVL by more than `pricePercentageLimit`, the protocol auto-pauses (`LRTDepositPool`, `LRTWithdrawalManager`, and `LRTOracle` are all paused), temporarily freezing all user funds. [5](#0-4) 

If `price == 0` (returned by Chainlink during an incomplete round), `getAssetPrice` returns `0`, making the entire asset's contribution to `totalETHInProtocol` zero, which is equivalent to a massive artificial price drop and will trigger the auto-pause.

**Impact classification:** Medium — Temporary freezing of funds (auto-pause triggered by stale/zero price) and incorrect rsETH minting (contract fails to deliver promised returns / theft of unclaimed yield from existing holders).

### Likelihood Explanation
Chainlink feeds can go stale during:
- Network congestion preventing oracle updates within the heartbeat window.
- Chainlink node outages or aggregator contract issues.
- An incomplete round where `startedAt == 0` and `answeredInRound < roundId`.

These are known, documented Chainlink failure modes. The `ChainlinkPriceOracle` is the live, deployed oracle for all LST assets on mainnet (`0x78C12ccE8346B936117655Dd3D70a2501Fd3d6e6`). Any unprivileged user can call `updateRSETHPrice()` or `depositAsset()` at any time, including during a stale-feed window.

### Recommendation
Apply the same guards already present in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price, uint256 startedAt, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    require(price > 0, "invalid price");
    require(startedAt > 0, "round not complete");
    require(answeredInRound >= roundId, "stale price");
    require(block.timestamp - updatedAt <= heartbeat, "stale price");

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

A per-asset `heartbeat` mapping should be added and set to the Chainlink feed's documented heartbeat (e.g., 3600 seconds for most ETH-denominated LST feeds).

### Proof of Concept

1. Chainlink's stETH/ETH feed stops updating (e.g., network congestion). `updatedAt` is now 2+ hours old.
2. An attacker (or any user) calls `LRTDepositPool.depositAsset(stETH, amount, 0, "")`.
3. `_beforeDeposit` → `getRsETHAmountToMint` → `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)`.
4. `latestRoundData()` returns the stale price (e.g., stETH was 1.05 ETH before a depeg event; actual price is now 0.95 ETH).
5. `getAssetPrice` returns `1.05e18` instead of `0.95e18`.
6. `rsethAmountToMint = (amount * 1.05e18) / rsETHPrice` — the depositor receives ~10.5% more rsETH than the asset is worth, at the expense of existing rsETH holders.

Alternatively, for the freeze path:
1. A Chainlink feed returns `price == 0` (incomplete round, `startedAt == 0`).
2. `getAssetPrice` returns `0` for that asset.
3. `_getTotalEthInProtocol` undervalues the protocol TVL.
4. `_updateRsETHPrice` detects a price drop exceeding `pricePercentageLimit` and calls `_pause()`, `lrtDepositPool.pause()`, `withdrawalManager.pause()`.
5. All deposits and withdrawals are frozen until an admin manually unpauses. [6](#0-5) [7](#0-6)
