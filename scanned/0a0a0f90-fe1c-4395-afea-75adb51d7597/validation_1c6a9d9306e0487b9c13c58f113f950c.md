### Title
First Depositor Captures Orphaned Protocol Rewards When rsETH Supply Is Zero - (File: contracts/LRTOracle.sol)

### Summary

`LRTOracle._updateRsETHPrice()` unconditionally resets `rsETHPrice` to `1 ether` when `rsethSupply == 0`, ignoring any ETH or assets already present in the protocol. Combined with the 1:1 mint ratio this produces in `LRTDepositPool.getRsETHAmountToMint()`, the first depositor after a zero-supply period captures all orphaned protocol rewards.

### Finding Description

In `LRTOracle._updateRsETHPrice()`, when `rsethSupply == 0` the function returns early after hard-coding the price to `1 ether`:

```solidity
// LRTOracle.sol lines 218-222
if (rsethSupply == 0) {
    rsETHPrice = 1 ether;
    highestRsethPrice = 1 ether;
    return;
}
```

It never calls `_getTotalEthInProtocol()`, so any ETH sitting in the deposit pool (e.g., sent via the unrestricted `receiveFromRewardReceiver()` payable function, or accumulated from staking rewards) is completely ignored when computing the price.

`LRTDepositPool.getRsETHAmountToMint()` then uses this stale `1 ether` price:

```solidity
// LRTDepositPool.sol lines 519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
// = (amount * 1e18) / 1e18 = amount  →  pure 1:1 mint
```

The first depositor therefore receives rsETH at a 1:1 ratio against ETH, even though the protocol already holds additional ETH that should dilute that ratio. When `updateRSETHPrice()` is next called with a non-zero supply, the price rises to reflect the full TVL, and the first depositor redeems at a profit equal to all orphaned assets.

`receiveFromRewardReceiver()` has no access control:

```solidity
// LRTDepositPool.sol line 61
function receiveFromRewardReceiver() external payable { }
```

Any ETH forwarded here while supply is zero becomes orphaned. `updateRSETHPrice()` is also public and callable by anyone, so the attacker can trigger the price reset themselves.

### Impact Explanation

**Impact: High — Theft of unclaimed yield.**

The first depositor after a zero-supply period steals all ETH that accumulated in the protocol during that period. The stolen amount equals the total orphaned ETH, which can be significant if staking rewards or fee-receiver ETH accumulated over a prolonged zero-supply window.

### Likelihood Explanation

**Likelihood: Low.**

The condition requires `rsETH.totalSupply() == 0` while ETH exists in the protocol. This can occur:
1. During protocol initialization before the first deposit, if the reward receiver or admin forwards ETH to the deposit pool.
2. After a full withdrawal cycle where all rsETH is burned but residual ETH remains (e.g., from `receiveFromRewardReceiver()` called by the fee/reward receiver contract between the last withdrawal and the next deposit).

This matches the likelihood classification of the reference finding.

### Recommendation

In `_updateRsETHPrice()`, do not return early when `rsethSupply == 0`. Instead, compute `totalETHInProtocol` and, if it is non-zero, revert or skip the price update to prevent reward distribution to an empty holder pool:

```solidity
if (rsethSupply == 0) {
    uint256 totalETH = _getTotalEthInProtocol();
    require(totalETH == 0, "Cannot update price: orphaned assets exist with zero supply");
    rsETHPrice = 1 ether;
    highestRsethPrice = 1 ether;
    return;
}
```

Alternatively, restrict `receiveFromRewardReceiver()` to only accept ETH when `rsETH.totalSupply() > 0`.

### Proof of Concept

1. Protocol is live; all rsETH holders redeem → `rsETH.totalSupply() == 0`.
2. The reward receiver contract forwards 100 ETH to `LRTDepositPool` via `receiveFromRewardReceiver()`. The deposit pool now holds 100 ETH.
3. Anyone calls `updateRSETHPrice()`. Because `rsethSupply == 0`, `_updateRsETHPrice()` sets `rsETHPrice = 1 ether` and returns without reading the 100 ETH balance.
4. Alice calls `depositETH{value: 1000 ether}(0, "")`. `getRsETHAmountToMint(ETH, 1000e18)` = `(1000e18 * 1e18) / 1e18` = `1000e18`. Alice receives **1000 rsETH** at 1:1.
5. Protocol now holds 1100 ETH with 1000 rsETH outstanding.
6. Anyone calls `updateRSETHPrice()`. `rsethSupply = 1000e18`, `totalETHInProtocol = 1100e18`. New price = `1100e18 / 1000e18 = 1.1 ether`.
7. Alice initiates withdrawal of 1000 rsETH. At 1.1 ether/rsETH she receives **1100 ETH** — a **100 ETH profit** from the orphaned community rewards. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6)

    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
