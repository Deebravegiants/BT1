### Title
`getAvailableAssetAmount` Does Not Subtract `queuedWithdrawalsBuffer`, Allowing Queued Withdrawals to Deplete Instant-Withdrawal Liquidity — (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTWithdrawalManager.getAvailableAssetAmount` computes available assets using the full `LRTUnstakingVault` balance via `getTotalAssetDeposits`, without subtracting the `queuedWithdrawalsBuffer` that the vault operator has reserved to protect instant-withdrawal liquidity. Any user can queue a withdrawal that consumes the entire buffer. When the operator subsequently calls `unlockQueue`, the vault is drained, and `getAssetsAvailableForInstantWithdrawal` returns zero, temporarily freezing all instant withdrawals.

---

### Finding Description

**Two separate liquidity reservations exist but are not reconciled:**

**1. `queuedWithdrawalsBuffer` in `LRTUnstakingVault`**

An operator manually sets a per-asset buffer to protect a portion of the vault's balance from being consumed by instant withdrawals:

```solidity
// LRTUnstakingVault.sol:229-238
function getAssetsAvailableForInstantWithdrawal(address asset) external view ... returns (uint256 availableAmount) {
    uint256 vaultBalance = balanceOf(asset);
    uint256 reservedBuffer = queuedWithdrawalsBuffer[asset];
    availableAmount = reservedBuffer >= vaultBalance ? 0 : vaultBalance - reservedBuffer;
}
``` [1](#0-0) 

**2. `getAvailableAssetAmount` in `LRTWithdrawalManager`**

This function gates new queued withdrawal requests. It uses `getTotalAssetDeposits`, which includes the **full** vault balance (`assetLyingUnstakingVault`), and only subtracts `assetsCommitted` — it never subtracts `queuedWithdrawalsBuffer`:

```solidity
// LRTWithdrawalManager.sol:599-603
function getAvailableAssetAmount(address asset) public view override returns (uint256 availableAssetAmount) {
    ILRTDepositPool lrtDepositPool = ILRTDepositPool(lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL));
    uint256 totalAssets = lrtDepositPool.getTotalAssetDeposits(asset);
    availableAssetAmount = totalAssets > assetsCommitted[asset] ? totalAssets - assetsCommitted[asset] : 0;
}
``` [2](#0-1) 

**`getTotalAssetDeposits` includes the full vault balance:**

```solidity
// LRTDepositPool.sol:385-397
return (assetLyingInDepositPool + assetLyingInNDCs + effectiveAssetWithEigenLayer
        + assetLyingInConverter + assetLyingUnstakingVault);
``` [3](#0-2) 

Where `assetLyingUnstakingVault = IERC20(asset).balanceOf(lrtUnstakingVault)` — the raw vault balance, buffer included: [4](#0-3) 

**`unlockQueue` drains the vault using the full balance:**

```solidity
// LRTWithdrawalManager.sol:286, 307
UnlockParams memory params = _createUnlockParams(lrtOracle, unstakingVault, asset);
// ...
unstakingVault.redeem(asset, assetAmountUnlocked);
```

`_createUnlockParams` sets `totalAvailableAssets = unstakingVault.balanceOf(asset)` — again the full balance, not the buffer-adjusted amount: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

After the attack sequence completes, `getAssetsAvailableForInstantWithdrawal` returns 0 because the vault balance is 0 while `queuedWithdrawalsBuffer` is still set to its original value. Every call to `instantWithdrawal` reverts with `CantInstantWithdrawMoreThanAvailable`:

```solidity
// LRTWithdrawalManager.sol:231-233
if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
    revert CantInstantWithdrawMoreThanAvailable();
}
``` [7](#0-6) 

**Impact: Temporary freezing of funds** — instant withdrawals are blocked for all users until the operator either deposits more assets into the vault or reduces `queuedWithdrawalsBuffer`. This matches the "Medium — Temporary freezing of funds" impact tier.

---

### Likelihood Explanation

- The `queuedWithdrawalsBuffer` is set during normal protocol operation to protect instant-withdrawal liquidity; it is not an edge-case configuration.
- Any unprivileged user can call `initiateWithdrawal` with an rsETH amount whose `expectedAssetAmount` equals or exceeds the buffer.
- The operator's subsequent `unlockQueue` call (a routine operational step) is the trigger that drains the vault.
- No special permissions, front-running, or oracle manipulation are required.

Likelihood: **Medium** — depends on the vault having assets and the buffer being set (normal state), and a user queuing a withdrawal large enough to consume the buffer.

---

### Recommendation

In `getAvailableAssetAmount`, subtract the `queuedWithdrawalsBuffer` from the vault's contribution to `totalAssets` before computing availability:

```solidity
function getAvailableAssetAmount(address asset) public view override returns (uint256 availableAssetAmount) {
    ILRTDepositPool lrtDepositPool = ILRTDepositPool(lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL));
    ILRTUnstakingVault unstakingVault = ILRTUnstakingVault(lrtConfig.getContract(LRTConstants.LRT_UNSTAKING_VAULT));
    uint256 totalAssets = lrtDepositPool.getTotalAssetDeposits(asset);
    uint256 buffer = unstakingVault.queuedWithdrawalsBuffer(asset);
    uint256 adjustedTotal = totalAssets > buffer ? totalAssets - buffer : 0;
    availableAssetAmount = adjustedTotal > assetsCommitted[asset] ? adjustedTotal - assetsCommitted[asset] : 0;
}
```

Alternatively, expose a `getAssetsAvailableForQueuedWithdrawal` view on `LRTUnstakingVault` that mirrors the buffer-adjusted logic, and use it as the cap in `initiateWithdrawal`.

### Proof of Concept

**Setup:**
- `LRTUnstakingVault` holds 100 ETH (all other protocol locations hold 0 for simplicity).
- Operator sets `queuedWithdrawalsBuffer[ETH] = 80 ETH`.
- `assetsCommitted[ETH] = 0`.

**Step 1 — User queues a withdrawal:**
- `getAvailableAssetAmount(ETH)` = `getTotalAssetDeposits(ETH)` − 0 = 100 ETH (buffer not subtracted).
- User calls `initiateWithdrawal(ETH, rsETHAmount)` where `expectedAssetAmount = 100 ETH`.
- Check passes: 100 ETH ≤ 100 ETH available.
- `assetsCommitted[ETH]` = 100 ETH. [8](#0-7) 

**Step 2 — Operator unlocks the queue:**
- `unlockQueue` reads `totalAvailableAssets = unstakingVault.balanceOf(ETH) = 100 ETH`.
- `_unlockWithdrawalRequests` allocates 100 ETH to the request.
- `unstakingVault.redeem(ETH, 100 ETH)` drains the vault to 0. [9](#0-8) 

**Step 3 — Instant withdrawals are frozen:**
- `getAssetsAvailableForInstantWithdrawal(ETH)` = max(0, 0 − 80) = **0**.
- Every `instantWithdrawal` call reverts with `CantInstantWithdrawMoreThanAvailable`.
- Funds remain frozen until the operator deposits more ETH into the vault or lowers the buffer. [10](#0-9)
