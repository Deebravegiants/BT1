### Title
Unprivileged User Can Permanently Block `removeSupportedToken` via Dust Token Donation - (File: contracts/pools/RSETHPoolV3.sol)

### Summary
`RSETHPoolV3.removeSupportedToken` requires the pool's balance of the token to be exactly zero before removal. Any unprivileged user can send a dust amount of the token directly to the pool contract, causing the zero-balance check to revert and permanently blocking the `TIMELOCK_ROLE` from removing a supported token.

### Finding Description
`RSETHPoolV3.removeSupportedToken` contains the following guard at line 562:

```solidity
function removeSupportedToken(address token, uint256 tokenIndex) external onlyRole(TIMELOCK_ROLE) {
    UtilLib.checkNonZeroAddress(token);
    if (supportedTokenList[tokenIndex] != token) revert TokenNotFoundError();
    if (IERC20(token).balanceOf(address(this)) != 0) revert TokenBalanceNotZero(); // <-- vulnerable line
    ...
}
``` [1](#0-0) 

Because any ERC-20 token's `transfer` function is permissionless (any holder can send tokens to any address), an attacker can call `IERC20(token).transfer(address(RSETHPoolV3), 1)` at any time to ensure the pool's balance is non-zero. This is not a one-time front-run: the attacker can repeat the donation every time the BRIDGER_ROLE drains the balance via `moveAssetsForBridging`, making the DoS persistent at negligible cost (1 wei + gas per transaction). [2](#0-1) 

The `getTokenBalanceMinusFees` helper used by `moveAssetsForBridging` only accounts for fee-tracked amounts; a direct ERC-20 transfer bypasses fee accounting entirely, so the donated dust always keeps `balanceOf(address(this)) != 0`. [3](#0-2) 

### Impact Explanation
**Medium — Temporary (persistent) freezing of protocol governance capability.**

The `removeSupportedToken` function is the only mechanism to delist a collateral token from the L2 pool. If a supported token becomes compromised (e.g., its oracle is manipulated, or the token contract has a backdoor allowing unbounded minting), the protocol's emergency response — removing the token — is blocked indefinitely. Users continue to be able to deposit the compromised token and receive `wrsETH`, diluting the backing of all rsETH holders. The TIMELOCK_ROLE cannot complete the removal without atomically draining the balance in the same transaction, which the current contract design does not support.

### Likelihood Explanation
**Medium.** The attacker needs only to hold a trivial amount of any supported token (stETH, ETHx, rETH, sfrxETH) and call `transfer` to the pool address. No privileged access, no oracle manipulation, and no front-running infrastructure is required — the attacker can pre-seed the balance at any time before or after the governance call is submitted. The cost is 1 wei of the token plus gas.

### Recommendation
Remove the strict zero-balance requirement from `removeSupportedToken`. Instead, either:
- **Short term**: Allow removal when `balanceOf(address(this)) <= someNegligibleDustThreshold` (analogous to `maxNegligibleAmount` already used in `LRTDepositPool._checkResidueLSTBalance`), or
- **Long term**: Separate the "delist from oracle/deposit routing" action from the "drain residual balance" action, so governance can delist a token immediately and drain its balance separately.

### Proof of Concept
1. `RSETHPoolV3` supports token `T` (e.g., stETH).
2. TIMELOCK_ROLE submits `removeSupportedToken(T, idx)` to remove a compromised token.
3. Attacker calls `IERC20(T).transfer(address(RSETHPoolV3), 1)` — costs ~1 wei + gas.
4. `removeSupportedToken` reverts with `TokenBalanceNotZero` because `balanceOf(address(this)) == 1 != 0`.
5. BRIDGER_ROLE calls `moveAssetsForBridging(T, 1)` to drain the 1 wei.
6. Attacker immediately calls `IERC20(T).transfer(address(RSETHPoolV3), 1)` again.
7. Steps 4–6 repeat indefinitely; the token is never removed. [1](#0-0)

    {
        if (amount == 0) revert InvalidAmount();
