### Title
ETH Deposit Limit Check Omits New Deposit Amount, Allowing Unlimited rsETH Minting Beyond Protocol Cap - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` uses an asymmetric check for ETH vs ERC20 assets. For ETH it only tests whether the *current* total already exceeds the limit, without adding the incoming deposit amount. Any unprivileged depositor can therefore push the protocol's ETH holdings arbitrarily beyond the configured cap and receive rsETH for the full amount.

### Finding Description
`_checkIfDepositAmountExceedesCurrentLimit` (LRTDepositPool.sol L676-L682) branches on asset type:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // ← missing `+ amount`
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct for ERC20
}
```

For ETH the function returns `true` (blocking the deposit) only when the running total *already* exceeds the limit. It never tests whether the new `msg.value` would push the total over the limit. For every ERC20 asset the check correctly adds `amount` before comparing.

`depositETH` (L76-L93) calls `_beforeDeposit` → `_checkIfDepositAmountExceedesCurrentLimit`, then calls `_mintRsETH(rsethAmountToMint)` where `rsethAmountToMint` is computed from the full `msg.value`. Because the guard never fires, rsETH is minted for the entire deposited ETH even when it far exceeds the remaining capacity.

This is the direct analog of the Yearn finding: in Yearn, shares were minted for the full requested amount while only a partial amount was actually deposited; here, rsETH is minted for the full deposited amount while the deposit-limit guard that should have rejected or capped the deposit silently passes.

### Impact Explanation
The deposit limit is a protocol safety cap (e.g., to match EigenLayer strategy capacity or risk parameters). Bypassing it means:

1. **Excess ETH cannot be deployed to EigenLayer** if the underlying strategy is at capacity. That ETH sits idle in the DepositPool/NDCs earning no yield, permanently diluting the yield accrued by all rsETH holders — **permanent freezing of unclaimed yield (Medium)**.
2. If the limit was set to prevent over-minting relative to backing capacity, repeated large deposits can push the protocol into a state where rsETH is under-backed by yield-generating assets — **protocol insolvency risk (Critical)**.

### Likelihood Explanation
The entry path is fully permissionless: any address can call `depositETH{value: X}(...)`. No role, signature, or special condition is required. The condition is trivially reachable whenever `getTotalAssetDeposits(ETH) ≤ depositLimitByAsset(ETH)`, which is the normal operating state of the protocol. Likelihood is **High**.

### Recommendation
Add the incoming `amount` to the ETH branch, mirroring the ERC20 logic:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept
1. ETH deposit limit is set to 1 000 ETH; current `getTotalAssetDeposits(ETH)` = 999 ETH.
2. Attacker calls `depositETH{value: 10_000 ether}(0, "")`.
3. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `999 > 1000` → `false` → deposit is not blocked.
4. `getRsETHAmountToMint(ETH, 10_000 ether)` returns rsETH proportional to 10 000 ETH.
5. `_mintRsETH` mints that full amount to the attacker.
6. Protocol now holds 10 999 ETH — 9 999 ETH over the intended cap — with rsETH minted for the entire surplus. The excess ETH cannot be deployed to EigenLayer if the strategy is at capacity, permanently suppressing yield for all rsETH holders. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
