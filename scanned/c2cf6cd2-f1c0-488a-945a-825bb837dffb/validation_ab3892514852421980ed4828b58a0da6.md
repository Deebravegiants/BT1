Looking at the exact code flow in `_updateRsETHPrice`:

### Title
Fee Charged on Slash-Recovery Gains Due to Missing High-Water-Mark Baseline — (`contracts/LRTOracle.sol`)

---

### Summary

`_updateRsETHPrice` computes the fee baseline as `previousTVL = rsethSupply.mulWad(rsETHPrice)`, where `rsETHPrice` is the last stored price. After a small slashing event (below `pricePercentageLimit`), `rsETHPrice` is written down. When TVL recovers, the lower post-slash `rsETHPrice` makes the recovery look like new yield, and a protocol fee is minted on it — stealing principal-recovery gains from rsETH holders.

---

### Finding Description

The fee calculation in `_updateRsETHPrice` uses the last stored `rsETHPrice` as its baseline:

```solidity
uint256 previousTVL = rsethSupply.mulWad(rsETHPrice);   // line 234
...
if (!protocolPaused && totalETHInProtocol > previousTVL) {
    uint256 rewardAmount = totalETHInProtocol - previousTVL;  // line 245
    protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
}
``` [1](#0-0) 

The contract also maintains `highestRsethPrice` — an all-time-high watermark — but it is **only** used for the price-change threshold checks (lines 252–291) and is **never** used as the fee baseline. [2](#0-1) 

The downside-protection path (lines 270–281) pauses the protocol and returns **before** writing `rsETHPrice` only when the price drop exceeds `pricePercentageLimit`:

```solidity
if (isPriceDecreaseOffLimit) {
    ...
    _pause();
    return;   // rsETHPrice is NOT updated — correct
}
``` [3](#0-2) 

For any slash **below** the limit (or when `pricePercentageLimit == 0`), execution continues to line 313 and `rsETHPrice` is written down:

```solidity
rsETHPrice = newRsETHPrice;   // line 313 — price permanently lowered
``` [4](#0-3) 

---

### Impact Explanation

**Concrete four-step scenario** (all values in ETH, `feeRate = 10%`, `pricePercentageLimit = 1%`):

| Step | `totalETHInProtocol` | `rsETHPrice` | `highestRsethPrice` | `previousTVL` | `rewardAmount` | Fee minted |
|------|----------------------|--------------|---------------------|---------------|----------------|------------|
| Initial | 1000 | 1.000 | 1.000 | — | — | — |
| After 0.5% slash | 995 | 0.995 | 1.000 | 1000 | 0 (TVL↓) | 0 |
| After recovery | 1000 | — | 1.000 | **995** | **5 ETH** | **0.5 ETH** |

The 5 ETH recovery is pure principal restoration, not new yield. Yet `protocolFeeInETH = 5 * 10% = 0.5 ETH` worth of rsETH is minted to the treasury, diluting all rsETH holders.

The invariant "fee is only taken on gains above the all-time-high TVL" is broken. `highestRsethPrice` correctly stays at 1.000 throughout, but the fee baseline ignores it.

---

### Likelihood Explanation

- `updateRSETHPrice()` is a **public, permissionless** function (line 87). Any caller can trigger the fee-minting step after TVL recovers.
- Slashing events on Ethereum validators are routine (e.g., double-vote slashings). A 0.1–0.5% TVL reduction is realistic and well below typical `pricePercentageLimit` values.
- If `pricePercentageLimit == 0` (unset), **every** downward price move — however large — goes through this path. [5](#0-4) 

---

### Recommendation

Replace the fee baseline with `highestRsethPrice` instead of `rsETHPrice`:

```solidity
// Use the all-time-high as the fee watermark, not the last stored price
uint256 previousTVL = rsethSupply.mulWad(highestRsethPrice);
```

This ensures fees are only charged on net new gains above the historical peak, never on recovery from losses.

function test_feeOnSlashRecovery() public {
    // Setup: rsethSupply = 1000e18, rsETHPrice = 1e18, highestRsethPrice = 1e18
    // totalETHInProtocol = 1000e18, protocolFeeInBPS = 1000 (10%)
    // pricePercentageLimit = 1e16 (1%)

    // Step 1: Simulate 0.5% slash — totalETHInProtocol drops to 995e18
    mockTotalETH(995e18);
    oracle.updateRSETHPrice();
    // rsETHPrice = 0.995e18, highestRsethPrice = 1e18, no fee minted ✓

    uint256 treasuryBalanceBefore = rsETH.balanceOf(treasury);

    // Step 2: TVL recovers to 1000e18
    mockTotalETH(1000e18);
    oracle.updateRSETHPrice();

    uint256 treasuryBalanceAfter = rsETH.balanceOf(treasury);

    // Fee was minted on 5 ETH recovery (10% of 5 ETH = 0.5 ETH worth of rsETH)
    assertGt(treasuryBalanceAfter, treasuryBalanceBefore, "Fee incorrectly minted on recovery");

    // Assert: no fee should have been minted since TVL only recovered to pre-slash level
    // (not above the all-time-high of 1000e18)
    assertEq(treasuryBalanceAfter, treasuryBalanceBefore, "FAIL: fee stolen from rsETH holders");
}
```

The second `assertEq` will fail on unmodified code, confirming the vulnerability.
