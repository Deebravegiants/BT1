### Title
Push-Payment ETH Transfer in `_processWithdrawalCompletion` Permanently Freezes User ETH After rsETH Is Burned — (`contracts/LRTWithdrawalManager.sol`)

---

### Summary
`LRTWithdrawalManager` separates the rsETH burn step (`unlockQueue`) from the ETH delivery step (`completeWithdrawal`). After rsETH is irreversibly burned and ETH is redeemed into the contract, the only delivery path pushes ETH directly to the user's address. If that address cannot receive ETH, both `completeWithdrawal` and the operator-assisted `completeWithdrawalForUser` permanently revert, and no recovery path exists in the current implementation.

---

### Finding Description

The withdrawal lifecycle has two distinct, non-atomic phases:

**Phase 1 — `unlockQueue` (operator-triggered, irreversible):** [1](#0-0) 

rsETH held by the contract is burned and the corresponding ETH is pulled from `LRTUnstakingVault` into `LRTWithdrawalManager`. Both operations are final; there is no undo path.

**Phase 2 — `completeWithdrawal` / `completeWithdrawalForUser` (push delivery):** [2](#0-1) 

`_transferAsset` for ETH uses a low-level call: [3](#0-2) 

If `to` is a contract without a `receive`/`fallback` function, or one that deliberately reverts on ETH receipt, `sent` is `false` and the function reverts with `EthTransferFailed`. Because all state mutations in `_processWithdrawalCompletion` (nonce pop, request deletion, counter decrement) occur in the same transaction, they are also reverted — the request remains in the queue forever.

The operator-assisted path `completeWithdrawalForUser` passes the same `user` address to `_processWithdrawalCompletion` and therefore suffers the identical failure: [4](#0-3) 

The developer comment on line 191 acknowledges ETH transfer issues but dismisses them as "non-impactful gas grief," overlooking the permanent-freeze scenario that arises after rsETH is already burned.

There is no `cancelWithdrawal`, no alternative recipient parameter, and `sweepRemainingAssets` is gated behind `hasUnlockedWithdrawals(asset) == false`: [5](#0-4) 

So the stuck ETH cannot be recovered through any existing function.

---

### Impact Explanation

After `unlockQueue` executes, the user's rsETH is permanently burned and the corresponding ETH sits in `LRTWithdrawalManager`. If the user's address cannot receive ETH:

- `completeWithdrawal` always reverts.
- `completeWithdrawalForUser` always reverts.
- `sweepRemainingAssets` is blocked because `unlockedWithdrawalsCount[asset]` never reaches zero for that slot.
- The ETH is permanently frozen in the contract with no in-protocol recovery mechanism.

**Impact class**: Critical — Permanent freezing of funds.

---

### Likelihood Explanation

The user must be a smart contract address that rejects ETH. Realistic scenarios:

1. **Accidental**: A multisig or contract wallet (e.g., Safe without a fallback handler) initiates a withdrawal. After `unlockQueue` runs, the ETH cannot be delivered.
2. **Intentional griefing**: A malicious actor deploys a contract with a togglable ETH-rejection flag, calls `initiateWithdrawal`, waits for the operator to run `unlockQueue` (burning rsETH), then enables rejection. The attacker sacrifices their own rsETH value to permanently lock the ETH in the contract and corrupt `unlockedWithdrawalsCount`.

Contract wallets are common among DeFi power users, making scenario 1 a realistic production risk. Likelihood: **Medium**.

---

### Recommendation

1. **Pull-over-push**: Instead of pushing ETH in `_processWithdrawalCompletion`, record the claimable amount in a mapping and let the user pull it in a separate `claimETH()` call. This decouples state finalization from delivery.
2. **Configurable recipient**: Allow the user to specify a recipient address at `completeWithdrawal` time (authenticated by `msg.sender` ownership of the request).
3. **Wrap delivery in try/catch with a claimable fallback**: If the push fails, store the amount in a `pendingClaims[user]` mapping rather than reverting, so the request is finalized and the ETH remains claimable.

---

### Proof of Concept

```
1. Attacker deploys MaliciousReceiver with:
     bool public rejectETH = false;
     receive() external payable { require(!rejectETH, "rejected"); }
     function setReject(bool v) external { rejectETH = v; }

2. MaliciousReceiver calls LRTWithdrawalManager.initiateWithdrawal(ETH_TOKEN, amount, "")
   → rsETH transferred from MaliciousReceiver to WithdrawalManager.

3. Operator calls unlockQueue(ETH_TOKEN, ...) 
   → rsETH burned (line 305), ETH redeemed into WithdrawalManager (line 307).
   → unlockedWithdrawalsCount[ETH_TOKEN]++

4. Attacker calls MaliciousReceiver.setReject(true).

5. Anyone calls completeWithdrawal(ETH_TOKEN, "") or
   operator calls completeWithdrawalForUser(ETH_TOKEN, MaliciousReceiver, "")
   → _transferAsset pushes ETH to MaliciousReceiver
   → MaliciousReceiver.receive() reverts
   → EthTransferFailed() is thrown; entire tx reverts.

6. Step 5 reverts indefinitely.
   → ETH permanently locked in LRTWithdrawalManager.
   → unlockedWithdrawalsCount[ETH_TOKEN] never decrements.
   → sweepRemainingAssets blocked for ETH_TOKEN.
```
