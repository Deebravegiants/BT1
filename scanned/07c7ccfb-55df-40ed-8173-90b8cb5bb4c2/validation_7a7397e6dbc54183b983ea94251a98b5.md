The analog vulnerability exists. The `notifyRewardAmount` function in `KernelDepositPool.sol` has the same precision loss issue — with no residual tracking at all (not even for the simple case).

### Title
Precision Loss in `notifyRewardAmount` Permanently Locks Reward Tokens With No Residual Accounting - (File: `contracts/KERNEL/KernelDepositPool.sol`)

### Summary
`KernelDepositPool.notifyRewardAmount` computes `rewardRate` via integer division but never tracks or recovers the truncated remainder. Every invocation permanently locks up to `duration - 1` wei of reward tokens in the contract, which are never distributed to stakers.

### Finding Description
In `notifyRewardAmount`, the `rewardRate` is set via:

```solidity
// New period
rewardRate = receivedAmount / duration;

// Active period (leftover)
uint256 remaining = (finishAt - block.timestamp) * rewardRate;
rewardRate = (receivedAmount + remaining) / duration;
``` [1](#0-0) 

In both branches, the Solidity integer division truncates the result. The discarded remainder — `receivedAmount % duration` (new period) or `(receivedAmount + remaining) % duration` (active period) — is never stored in any residual variable and never added back into a future distribution. Those tokens remain in the contract balance permanently, silently reducing the total rewards actually distributed to stakers below the amount deposited.

Unlike the `MultiRewards.sol` case described in the external report (which at least introduced a `rewardResidual` fix for the simple case), `KernelDepositPool` has **no residual tracking whatsoever** — neither for the new-period branch nor the leftover branch. [2](#0-1) 

### Impact Explanation
**Low.** The lost amount per call is at most `duration - 1` wei (bounded by the duration value). No user funds are stolen; stakers simply receive slightly less yield than the deposited reward amount implies. The contract fails to deliver the full promised return, but does not lose principal value.

### Likelihood Explanation
**High.** The truncation occurs on every single call to `notifyRewardAmount` for both code paths. Since reward distributions are expected to be repeated periodically, the unrecoverable dust accumulates continuously over the contract's lifetime. [1](#0-0) 

### Recommendation
Track the truncated remainder and roll it into the next distribution, analogous to the `rewardResidual` fix recommended in the external report. Concretely:

1. Add a `uint256 public rewardResidue` state variable.
2. In the new-period branch: `rewardResidue += receivedAmount % duration;` then use `(receivedAmount + rewardResidue) / duration` and reset `rewardResidue` to the new remainder.
3. In the active-period branch: apply the same residual accumulation to `(receivedAmount + remaining)` before dividing by `duration`.

### Proof of Concept
Assume `duration = 7 days = 604800 seconds` and admin calls `notifyRewardAmount` with `receivedAmount = 1_000_000` tokens (no active period):

```
rewardRate = 1_000_000 / 604_800 = 1 (truncated)
tokens actually distributed = 1 * 604_800 = 604_800
tokens permanently locked   = 1_000_000 - 604_800 = 395_200
```

On the next call with the same amount (new period again):

```
another 395_200 tokens locked
```

After N calls, `N * (receivedAmount % duration)` tokens are permanently stranded in the contract, never claimable by any staker. [1](#0-0)

    function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
        if (_amount == 0) revert AmountZero();

        // Prevent starting a reward period when no tokens are staked to avoid unallocated rewards
        if (totalKernelStaked == 0) revert NoStakedTokens();

        // Transfer reward tokens into the contract
        uint256 balanceBefore = rewardsToken.balanceOf(address(this));
        rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
        uint256 balanceAfter = rewardsToken.balanceOf(address(this));
        // Calculate the actual amount of tokens received in case of a transfer fee (tax)
        uint256 receivedAmount = balanceAfter - balanceBefore;

        if (block.timestamp >= finishAt) {
            rewardRate = receivedAmount / duration;
        } else {
            uint256 remaining = (finishAt - block.timestamp) * rewardRate;
            rewardRate = (receivedAmount + remaining) / duration;
        }

        if (rewardRate == 0) revert RewardRateZero();

        finishAt = block.timestamp + duration;
        updatedAt = block.timestamp;

        emit NotifyRewardAmount(receivedAmount, finishAt);
    }
