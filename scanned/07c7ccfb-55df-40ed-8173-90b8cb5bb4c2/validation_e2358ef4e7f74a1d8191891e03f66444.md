### Title
Unbounded Withdrawal Queue Growth Enables Temporary Freezing of Legitimate User Withdrawals - (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary

An unprivileged attacker holding rsETH can flood the global withdrawal queue with many minimum-sized withdrawal requests. Because `nextLockedNonce[asset]` advances strictly sequentially, the operator must process every attacker-inserted entry before any later legitimate user's request can be unlocked. This temporarily freezes legitimate users' withdrawals and, as a secondary effect, can exhaust `assetsCommitted[asset]` capacity to block new withdrawal initiations entirely.

---

### Finding Description

**Root cause — sequential global nonce with no per-user isolation:**

`initiateWithdrawal` is callable by any holder of rsETH. Each call:

1. Transfers rsETH from the caller.
2. Commits `expectedAssetAmount` to `assetsCommitted[asset]`.
3. Pushes the request nonce into `userAssociatedNonces[asset][msg.sender]`.
4. Increments the global `nextUnusedNonce[asset]`. [1](#0-0) 

The only per-request floor is `minRsEthAmountToWithdraw[asset]`, which is an admin-settable value that can be zero or very small. [2](#0-1) 

**The sequential unlock bottleneck:**

`_unlockWithdrawalRequests` advances `nextLockedNonce[asset]` one step at a time through a `while` loop: [3](#0-2) 

`completeWithdrawal` enforces that a user's request nonce must be strictly less than `nextLockedNonce[asset]` before it can be finalised: [4](#0-3) 

Because the nonce space is shared and sequential, every attacker-inserted entry that precedes a legitimate user's nonce must be individually processed by the operator before that user's withdrawal can be completed.

**Attack path:**

1. Attacker deposits ETH/LST into `LRTDepositPool.depositETH` / `depositAsset` to obtain rsETH.
2. Attacker calls `initiateWithdrawal` in a loop, each time with exactly `minRsEthAmountToWithdraw[asset]` rsETH, inserting N entries into the global queue ahead of any future legitimate requests.
3. Legitimate users call `initiateWithdrawal` after the attacker; their nonces are all > attacker nonces.
4. The operator must call `unlockQueue` repeatedly, each time advancing `nextLockedNonce` through a batch of attacker entries, before any legitimate user's nonce becomes unlocked.
5. Each `unlockQueue` call iterates through storage reads/writes per entry; with thousands of attacker entries the operator requires many transactions, each consuming significant gas.
6. Until all attacker entries are processed, every legitimate user's `completeWithdrawal` reverts with `WithdrawalLocked`.

**Secondary effect — capacity exhaustion:**

Each attacker request commits `expectedAssetAmount` to `assetsCommitted[asset]`. If the attacker's aggregate committed amount approaches `getTotalAssetDeposits(asset)`, `getAvailableAssetAmount` returns near-zero, causing new `initiateWithdrawal` calls from legitimate users to revert with `ExceedAmountToWithdraw`. [5](#0-4) [6](#0-5) 

---

### Impact Explanation

**Temporary freezing of funds (Medium).** Legitimate users who have already queued withdrawals cannot call `completeWithdrawal` until the operator has processed all preceding attacker entries. Depending on the number of attacker entries and the operator's batching cadence, this delay can span many blocks or days. The secondary capacity-exhaustion effect can additionally prevent new withdrawal initiations, compounding the freeze.

---

### Likelihood Explanation

Any rsETH holder can execute this attack. rsETH is freely obtainable by depositing ETH or supported LSTs into `LRTDepositPool`. The cost to the attacker is the gas for N `initiateWithdrawal` calls plus the temporary lock-up of their rsETH (which they recover after the withdrawal delay). With a small `minRsEthAmountToWithdraw`, the rsETH cost per entry is minimal. The attack is therefore realistic and low-cost relative to its impact.

---

### Recommendation

1. **Enforce a meaningful minimum per withdrawal request** — set `minRsEthAmountToWithdraw` to a value that makes flooding economically prohibitive.
2. **Decouple per-user queues from the global unlock nonce** — instead of a single sequential `nextLockedNonce`, maintain per-user unlock state so that processing one user's requests does not block another's.
3. **Add a per-address rate limit or maximum pending requests cap** — reject `initiateWithdrawal` if the caller already has more than N pending requests.
4. **Allow `unlockQueue` to skip zero-balance or already-withdrawn entries** — so the operator is not forced to iterate through every attacker entry sequentially.

```solidity
// Attacker floods the queue before a legitimate user
function attack(address asset, uint256 minAmount, uint256 N) external {
    // Attacker already holds N * minAmount rsETH
    for (uint256 i = 0; i < N; i++) {
        withdrawalManager.initiateWithdrawal(asset, minAmount, "");
    }
}

// Legitimate user queues after attacker
function legitimateUser(address asset, uint256 amount) external {
    withdrawalManager.initiateWithdrawal(asset, amount, "");
    // nonce = N + 1; nextLockedNonce is still 0
}

// completeWithdrawal for legitimate user reverts until operator
// has called unlockQueue N+1 times to advance nextLockedNonce past N
function tryComplete(address asset) external {
    // reverts: WithdrawalLocked — usersFirstWithdrawalRequestNonce (N+1) >= nextLockedNonce (0..N)
    withdrawalManager.completeWithdrawal(asset, "");
}
```

```text
    function _addUserWithdrawalRequest(address asset, uint256 rsETHUnstaked, uint256 expectedAssetAmount) internal {
        uint256 nextUnusedNonce_ = nextUnusedNonce[asset];

        // Generate a unique identifier for the new withdrawal request.
        bytes32 requestId = getRequestId(asset, nextUnusedNonce_);

        // Create and store the new withdrawal request.
        withdrawalRequests[requestId] = WithdrawalRequest({
            rsETHUnstaked: rsETHUnstaked, expectedAssetAmount: expectedAssetAmount, withdrawalStartBlock: block.number
        });

        // Map the user to the newly created request index and increment the nonce for future requests.
        userAssociatedNonces[asset][msg.sender].pushBack(nextUnusedNonce_);
        nextUnusedNonce[asset] = nextUnusedNonce_ + 1;
```

```text
    function _unlockWithdrawalRequests(
        address asset,
        uint256 availableAssetAmount,
        uint256 rsETHPrice,
        uint256 assetPrice,
        uint256 firstExcludedIndex
    )
        internal
        returns (uint256 rsETHAmountToBurn, uint256 assetAmountToUnlock)
    {
        // Check that upper limit is in the range of existing withdrawal requests. If it is greater set it to the first
        // nonce with no withdrawal request.
        if (firstExcludedIndex > nextUnusedNonce[asset]) {
            firstExcludedIndex = nextUnusedNonce[asset];
        }

        uint256 nextLockedNonce_ = nextLockedNonce[asset];
        // Revert when trying to unlock a request that has already been unlocked
        if (nextLockedNonce_ >= firstExcludedIndex) revert NoPendingWithdrawals();

        while (nextLockedNonce_ < firstExcludedIndex) {
            bytes32 requestId = getRequestId(asset, nextLockedNonce_);
            WithdrawalRequest storage request = withdrawalRequests[requestId];

            // Check that the withdrawal delay has passed since the request's initiation.
            if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) break;

            // Calculate the amount user will receive
            uint256 payoutAmount = _calculatePayoutAmount(request, rsETHPrice, assetPrice);

            if (availableAssetAmount < payoutAmount) break; // Exit if not enough assets to cover this request

            assetsCommitted[asset] -= request.expectedAssetAmount;
            // Set the amount the user will receive
            request.expectedAssetAmount = payoutAmount;
            rsETHAmountToBurn += request.rsETHUnstaked;
            availableAssetAmount -= payoutAmount;
            assetAmountToUnlock += payoutAmount;

            unlockedWithdrawalsCount[asset]++;

            unchecked {
                nextLockedNonce_++;
            }
        }
        nextLockedNonce[asset] = nextLockedNonce_;
    }
```
