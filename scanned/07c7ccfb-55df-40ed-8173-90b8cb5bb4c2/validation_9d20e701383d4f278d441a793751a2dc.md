### Title
Predictable UID Generation from Manipulable On-Chain Values Enables Bridge Withdrawal Collision — (`contracts/bridges/SonicChainNativeTokenBridge.sol`)

---

### Summary

`SonicChainNativeTokenBridge.bridgeTokenToL1` generates a `uint96 uid` by hashing `block.timestamp`, `block.number`, `msg.sender`, `recipient`, `amount`, `tx.gasprice`, and `bridgeReceiver`. Every one of these inputs is either publicly known before the transaction is mined or directly caller-controlled. Two calls that share the same `msg.sender`, `recipient`, `amount`, and `tx.gasprice` within the same block produce an identical UID. The UID is then passed verbatim to `sonicBridge.withdraw(uid, ...)` and later used as the `withdrawalId` key in `SonicBridgeReceiver.claimAndTransferToVault` on L1. A collision in the UID space causes either a failed bridge submission or a permanently unclaimable L1 withdrawal, freezing the bridged funds.

---

### Finding Description

In `bridgeTokenToL1`, the UID is computed as:

```solidity
uint96 uid = uint96(
    uint256(
        keccak256(
            abi.encodePacked(
                block.timestamp, block.number, msg.sender, recipient, amount, tx.gasprice, bridgeReceiver
            )
        )
    ) % type(uint96).max   // truncation to 96 bits
);
``` [1](#0-0) 

All inputs are observable or attacker-controlled:

| Input | Attacker control |
|---|---|
| `block.timestamp` | Same for all txs in a block; validator-manipulable |
| `block.number` | Same for all txs in a block |
| `msg.sender` | Caller identity |
| `recipient` | Caller-supplied parameter |
| `amount` | Caller-supplied parameter |
| `tx.gasprice` | Caller-supplied |
| `bridgeReceiver` | Immutable constant |

The result is further truncated via `% type(uint96).max`, reducing the output space and introducing modular bias.

The UID is passed directly to the Sonic bridge:

```solidity
sonicBridge.withdraw(uid, originalToken, amount);
``` [2](#0-1) 

On L1, `SonicBridgeReceiver.claimAndTransferToVault` uses the same `withdrawalId` as a key in a `claimedWithdrawals` mapping to prevent replay:

```solidity
if (claimedWithdrawals[withdrawalId]) revert WithdrawalAlreadyClaimed();
claimedWithdrawals[withdrawalId] = true;
``` [3](#0-2) 

If two bridge withdrawals share the same UID, the first claim on L1 marks `claimedWithdrawals[uid] = true`. The second withdrawal — representing real tokens already burned/locked on Sonic — can never be claimed, permanently freezing those funds.

---

### Impact Explanation

**Worst case — Permanent freezing of funds (Critical):** If the Sonic bridge protocol does not enforce UID uniqueness on-chain (i.e., it accepts two `withdraw` calls with the same UID as two independent withdrawal records), both withdrawals are registered with the same `withdrawalId`. On L1, `SonicBridgeReceiver` marks the ID as claimed after the first `claimAndTransferToVault`. The second withdrawal's tokens are permanently unclaimable — the funds are frozen with no recovery path, since `claimedWithdrawals[withdrawalId]` is `true` and there is no admin override for individual withdrawal IDs.

**Conservative case — Temporary freezing of funds (Medium):** If the Sonic bridge rejects duplicate UIDs, the second `sonicBridge.withdraw(uid, ...)` call reverts. Because `safeTransferFrom` already executed before the UID is generated, the entire transaction reverts and the user's tokens are returned. However, the user cannot retry with the same parameters in the same block, and a malicious actor who can predict the UID (e.g., by front-running with identical parameters) can force the victim's transaction to always collide, causing a sustained DoS on bridge withdrawals.

---

### Likelihood Explanation

The collision is reachable by any unprivileged caller of `bridgeTokenToL1`. A realistic collision occurs when:

1. A pool contract (e.g., `RSETHPoolV3`) calls `bridgeTokenToL1` twice in the same block with the same `amount` and `recipient`.
2. A user submits two identical bridge transactions that land in the same block.
3. A malicious actor front-runs a victim's bridge call with an identical transaction (same `msg.sender` is not required — only `recipient`, `amount`, `tx.gasprice` need to match, and `msg.sender` can differ if the pool is the caller).

The `% type(uint96).max` truncation further increases collision probability as bridge usage scales.

---

### Recommendation

Replace the on-chain pseudo-random UID with a monotonically incrementing counter maintained by the contract itself:

```solidity
uint96 private _nextUid;

function bridgeTokenToL1(...) external payable nonReentrant {
    ...
    uint96 uid = ++_nextUid;
    sonicBridge.withdraw(uid, originalToken, amount);
    ...
}
```

This guarantees global uniqueness regardless of block context, caller parameters, or validator manipulation, directly mirroring the external report's recommendation that the server (here, the contract) assign the unique ID rather than deriving it from client-supplied or environment-observable inputs.

---

### Proof of Concept

1. Alice calls `bridgeTokenToL1(recipient=R, amount=A)` with `tx.gasprice=G` at block `N`, timestamp `T`.
2. Bob calls `bridgeTokenToL1(recipient=R, amount=A)` with `tx.gasprice=G` in the same block `N`, timestamp `T`.
3. Both compute:
   ```
   uid = keccak256(T, N, alice_addr, R, A, G, bridgeReceiver) % uint96.max
   uid = keccak256(T, N, bob_addr,   R, A, G, bridgeReceiver) % uint96.max
   ```
   If `msg.sender` differs, the UIDs differ — but if the pool contract is the `msg.sender` for both (e.g., two users trigger bridging via the same pool in the same block), `msg.sender` is identical and the UIDs collide exactly.
4. The Sonic bridge registers two withdrawals under the same `uid`.
5. On L1, `SonicBridgeReceiver.claimAndTransferToVault(uid, ...)` succeeds for the first caller.
6. The second caller's `claimAndTransferToVault(uid, ...)` hits `revert WithdrawalAlreadyClaimed()`.
7. The second user's bridged tokens are permanently frozen in the Sonic bridge with no recovery path. [4](#0-3) [5](#0-4)

    function bridgeTokenToL1(address recipient, uint256 amount) external payable override nonReentrant {
        UtilLib.checkNonZeroAddress(recipient);

        // recipient parameter is kept for IL2TokenBridge interface compatibility
        // but the actual flow will be: SonicBridge -> SonicBridgeReceiver -> L1Vault
        if (amount == 0) revert InvalidAmount();

        // No additional msg.value is needed for the fees
        if (msg.value != 0) revert NoMsgValueNeeded();

        token.safeTransferFrom(msg.sender, address(this), amount);

        uint256 balance = token.balanceOf(address(this));
        if (balance < amount) revert InsufficientBalance();

        // Get the original token address (validated in constructor)
        address originalToken = tokenPairs.mintedToOriginal(address(token));

        // Generate a unique UID for this transaction
        uint96 uid = uint96(
            uint256(
                keccak256(
                    abi.encodePacked(
                        block.timestamp, block.number, msg.sender, recipient, amount, tx.gasprice, bridgeReceiver
                    )
                )
            ) % type(uint96).max
        );

        // Ensure UID is not zero
        if (uid == 0) {
            uid = uint96(uint256(keccak256(abi.encodePacked(block.number, msg.sender, tx.gasprice, bridgeReceiver))));
        }

        // Store the current token balance before withdrawal
        uint256 balanceBefore = token.balanceOf(address(this));

        // Approve the Sonic bridge to spend the tokens
        token.safeIncreaseAllowance(address(sonicBridge), amount);

        // Initiate withdrawal on Sonic bridge
        // Note: Sonic gateway will only allow SonicBridgeReceiver to claim (same address as this contract)
        sonicBridge.withdraw(uid, originalToken, amount);

        // Verify tokens were transferred/burned
        uint256 balanceAfter = token.balanceOf(address(this));
        if (balanceBefore - balanceAfter != amount) {
            revert BridgeFailed();
        }

        emit TokensBridgedToL1(recipient, amount, uid, bridgeReceiver);
    }

    function claimAndTransferToVault(
        uint256 withdrawalId,
        address token,
        uint256 amount,
        bytes calldata proof
    )
        external
        nonReentrant
        onlyRole(CLAIMER_ROLE)
    {
        if (withdrawalId == 0) revert InvalidWithdrawalId();
        if (claimedWithdrawals[withdrawalId]) revert WithdrawalAlreadyClaimed();

        // Mark as claimed before external call to prevent reentrancy
        claimedWithdrawals[withdrawalId] = true;

        // Get balance before claim
        uint256 balanceBefore = IERC20(token).balanceOf(address(this));

        // Claim from Sonic bridge
        ETHEREUM_TOKEN_DEPOSIT.claim(withdrawalId, token, amount, proof);
        emit WithdrawalClaimed(withdrawalId, token, amount);

        // Verify we received the tokens and transfer to L1Vault
        uint256 balanceAfter = IERC20(token).balanceOf(address(this));
        uint256 received = balanceAfter - balanceBefore;
        if (received == 0) revert InsufficientBalance();

        // Transfer tokens to L1Vault
        IERC20(token).safeTransfer(l1Vault, received);
        emit TokensTransferredToVault(token, received, l1Vault);
    }
