### Title
Public `updateRSETHPrice()` Enables MEV Sandwich to Steal Staking Yield from rsETH Holders - (File: contracts/LRTOracle.sol)

---

### Summary

`LRTOracle.updateRSETHPrice()` is a permissionless, publicly callable function. Because the price it writes is consumed directly by both the deposit path (`LRTDepositPool`) and the instant-withdrawal path (`LRTWithdrawalManager`), an MEV searcher can atomically sandwich the price update: deposit ETH at the stale (lower) price, trigger the update, and instantly withdraw at the new (higher) price, capturing the staking-yield increment that should have accrued to all rsETH holders.

---

### Finding Description

`updateRSETHPrice()` carries no access-control modifier — only `whenNotPaused`: [1](#0-0) 

The function recomputes `rsETHPrice` as `(totalETHInProtocol − protocolFee) / rsethSupply`: [2](#0-1) 

This stored value is the sole price source for two user-facing operations:

**Deposit** (`LRTDepositPool.getRsETHAmountToMint`):
```
rsethAmountToMint = (amount × assetPrice) / rsETHPrice
``` [3](#0-2) 

**Instant withdrawal** (`LRTWithdrawalManager.getExpectedAssetAmount`):
```
underlyingToReceive = rsETHUnstaked × rsETHPrice / assetPrice
``` [4](#0-3) 

The instant-withdrawal path burns rsETH and immediately transfers assets from `LRTUnstakingVault`: [5](#0-4) 

Because both operations read the same `rsETHPrice` storage slot and `updateRSETHPrice()` is callable by anyone, the price transition is fully observable and exploitable in a single block.

---

### Impact Explanation

**Impact: High — Theft of unclaimed yield.**

When staking rewards accumulate, `totalETHInProtocol` grows while `rsethSupply` stays constant, so the next `updateRSETHPrice()` call will push `rsETHPrice` upward. An attacker who deposits just before the update and withdraws just after captures the entire yield increment for their deposited principal, leaving other rsETH holders with proportionally less.

Concretely, if the attacker deposits `X` ETH at `oldPrice` and withdraws after the price moves to `newPrice`:

```
rsETH received  = X / oldPrice
ETH returned    = (X / oldPrice) × newPrice
Profit          = X × (newPrice / oldPrice − 1)
```

This profit is extracted directly from the yield that should have been shared across all rsETH holders.

---

### Likelihood Explanation

**Likelihood: Medium.**

- `updateRSETHPrice()` is public with no access control — any EOA or contract can call it.
- Staking rewards accumulate continuously; the price increase is predictable by monitoring EigenLayer strategy balances off-chain.
- `instantWithdrawal` must be enabled per asset (`isInstantWithdrawalEnabled[asset]`), which is a configurable flag. When enabled, the full sandwich is atomic.
- The `pricePercentageLimit` guard only fires when `newRsETHPrice > highestRsethPrice`; if the price is recovering from a prior dip, the guard does not apply and the jump can be larger.
- If `pricePercentageLimit == 0`, there is no cap at all.
- The `instantWithdrawalFee` reduces profit but does not eliminate it; a sufficiently large price jump (e.g., after infrequent updates) makes the attack profitable net of fees. [6](#0-5) 

---

### Recommendation

1. **Restrict `updateRSETHPrice()` to authorized callers** (e.g., `onlyLRTManager` or a keeper role), removing the permissionless entry point. The existing `updateRSETHPriceAsManager()` already demonstrates the intended privileged path.
2. **Add a deposit/withdrawal cooldown or same-block deposit-then-withdraw guard** so that rsETH minted in block N cannot be redeemed until block N+k.
3. **Enforce a minimum `instantWithdrawalFee`** that exceeds the maximum allowed single-update price increment (`pricePercentageLimit`) to ensure the attack is never profitable.
4. **Update the oracle frequently** (at least daily) to minimize the accumulated yield available for extraction in a single sandwich.

---

### Proof of Concept

Assume:
- `rsETHPrice` = 1.01 ETH (stale; rewards have accumulated)
- After `updateRSETHPrice()`, `rsETHPrice` will become 1.02 ETH
- `instantWithdrawal` is enabled for ETH; `instantWithdrawalFee` = 5 bps (0.05%)
- Attacker has 1000 ETH

**Step 1 — Front-run (deposit at old price):**
```
LRTDepositPool.depositETH{value: 1000 ETH}(minRSETH=0, "")
rsETH minted = 1000 / 1.01 ≈ 990.099 rsETH
``` [7](#0-6) 

**Step 2 — Trigger price update:**
```
LRTOracle.updateRSETHPrice()   // permissionless; rsETHPrice → 1.02 ETH
``` [8](#0-7) 

**Step 3 — Back-run (instant withdrawal at new price):**
```
LRTWithdrawalManager.instantWithdrawal(ETH, 990.099 rsETH, "")
ETH returned = 990.099 × 1.02 / 1.0 ≈ 1009.9 ETH
Fee (5 bps)  ≈ 0.505 ETH
Net received ≈ 1009.4 ETH
Profit       ≈ 9.4 ETH  (stolen from other rsETH holders' yield)
``` [9](#0-8) 

All three transactions can be bundled atomically via a private relay (e.g., Flashbots), requiring no flash loan and leaving no mempool footprint.

        address asset,
        uint256 rsETHUnstaked,
        string calldata referralId
    )
        external
        nonReentrant
        whenNotPaused
        onlySupportedAsset(asset)
        onlySupportedStrategy(asset)
        onlyInstantWithdrawalAllowed(asset)
    {
        if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
            revert InvalidAmountToWithdraw();
        }
        if (IERC20(lrtConfig.rsETH()).balanceOf(msg.sender) < rsETHUnstaked) revert NotEnoughRsETH();
        uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
        IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
        ILRTUnstakingVault unstakingVault = ILRTUnstakingVault(lrtConfig.getContract(LRTConstants.LRT_UNSTAKING_VAULT));
        if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
            revert CantInstantWithdrawMoreThanAvailable();
        }

        unstakingVault.redeem(asset, assetAmountUnlocked);

        uint256 fee = (assetAmountUnlocked * instantWithdrawalFee) / 10_000;
        uint256 userAmount = assetAmountUnlocked - fee;

            // Backwards-compatible default: send fees to the protocol treasury
            feeRecipient = lrtConfig.getContract(LRTConstants.PROTOCOL_TREASURY);

        if (fee > 0) {
            _transferAsset(asset, feeRecipient, fee);
            emit InstantWithdrawalFeeCollected(msg.sender, asset, fee);
        }

        _transferAsset(asset, msg.sender, userAmount);
        emit ReferralIdEmitted(referralId);
        emit AssetWithdrawalFinalized(msg.sender, asset, rsETHUnstaked, userAmount);
    }
