### Title
No Staleness Check on Chainlink Price Feed Allows Stale LST Prices to Drive rsETH Over-Minting — (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls Chainlink's `latestRoundData()` but silently discards every return value except `price`. There is no check on `updatedAt`, `answeredInRound`, or `roundId`. If a Chainlink LST/ETH feed stops updating while the real market price has fallen (e.g., an LST depeg), the oracle continues returning the last stale (inflated) price. Any depositor can then call `LRTDepositPool.depositAsset()` and receive more rsETH than the true ETH value of their deposit, diluting all existing rsETH holders.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the price as:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All five return values of `latestRoundData()` — `roundId`, `answer`, `startedAt`, `updatedAt`, `answeredInRound` — are available, but only `answer` is used. The function performs no validation of:

- `answeredInRound >= roundId` — detects an incomplete or in-progress round
- `updatedAt > 0` — detects an uninitialized round
- `block.timestamp - updatedAt <= heartbeat` — detects a time-stale price
- `price > 0` — detects a zero/negative price

This price is consumed directly by `LRTOracle.getAssetPrice()`, which is called in two critical paths:

1. **`LRTDepositPool.getRsETHAmountToMint()`** — determines how many rsETH tokens a depositor receives per unit of LST deposited.
2. **`LRTOracle._updateRsETHPrice()` → `_getTotalEthInProtocol()`** — computes the protocol-wide TVL used to set the stored `rsETHPrice`. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

**High — Theft of unclaimed yield from existing rsETH holders.**

If a supported LST's Chainlink feed goes stale while the real market price has dropped (e.g., an LST depeg event), the stale inflated price is used in:

```
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

A depositor submitting `N` units of a depegged LST (real value: `0.95 ETH` each, stale oracle value: `1.00 ETH` each) receives rsETH priced at `1.00 ETH` per unit. The excess rsETH minted represents a direct claim on the pool's real assets, diluting every existing rsETH holder's redemption value. This is equivalent to yield theft from all current holders. [4](#0-3) 

---

### Likelihood Explanation

**Medium.** Chainlink feeds have documented heartbeat intervals (e.g., 24 hours for stETH/ETH on mainnet). During periods of network congestion, oracle node failure, or a rapid LST depeg event, the feed can remain stale for hours. The attack requires no privileged access — any unprivileged depositor can call `depositAsset()` the moment the feed is stale and the real price has diverged. LST depeg events (stETH in 2022, cbETH volatility) are historically observed. [1](#0-0) 

---

### Recommendation

Add staleness and validity checks in `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

if (answeredInRound < roundId) revert StalePrice();
if (updatedAt == 0) revert IncompleteRound();
if (price <= 0) revert InvalidPrice();
if (block.timestamp - updatedAt > STALENESS_THRESHOLD) revert StalePrice();
```

The `STALENESS_THRESHOLD` should be set per-feed based on its documented heartbeat (e.g., 25 hours for a 24-hour heartbeat feed). Note that `ChainlinkOracleForRSETHPoolCollateral` already implements partial checks (`answeredInRound < roundID`, `timestamp == 0`, `ethPrice <= 0`) but still lacks the time-based staleness check — both contracts should be brought to full parity. [5](#0-4) 

---

### Proof of Concept

1. Chainlink's stETH/ETH feed last updated at `T=0` with price `1.00 ETH`. The feed's heartbeat is 24 hours.
2. At `T=12h`, stETH depegs to `0.90 ETH` on the open market. Chainlink nodes fail to push an update (network congestion).
3. At `T=13h`, an attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18)`.
4. `getRsETHAmountToMint()` calls `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns `1.00e18` (stale).
5. `rsethAmountToMint = (1000e18 * 1.00e18) / rsETHPrice` — attacker receives rsETH valued at `1000 ETH` worth of claims.
6. Real value deposited: `900 ETH`. The `100 ETH` difference is a dilution claim against all existing rsETH holders' redemption value.
7. The attacker can immediately redeem or hold rsETH representing more underlying value than they deposited, at the expense of existing holders. [6](#0-5) [1](#0-0)
