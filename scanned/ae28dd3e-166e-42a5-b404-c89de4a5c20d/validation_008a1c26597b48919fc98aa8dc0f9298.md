### Title
Attacker Can Inflate `LRTDepositPool` ETH Balance to Exceed Deposit Limit, Causing DoS on ETH Deposits - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool` exposes a public `receive()` function, allowing anyone to send ETH directly to the contract. The ETH deposit-limit guard reads `address(this).balance` as part of `totalAssetDeposits`. An unprivileged attacker can donate a small amount of ETH to push `totalAssetDeposits` past `depositLimitByAsset`, permanently blocking all further ETH deposits until an admin manually raises the cap.

### Finding Description
`LRTDepositPool` accepts raw ETH via an unrestricted `receive()` fallback: [1](#0-0) 

`getETHDistributionData()` counts the contract's raw ETH balance as protocol-held ETH: [2](#0-1) 

`getTotalAssetDeposits(ETH_TOKEN)` delegates to `getETHDistributionData()` and sums all components including `ethLyingInDepositPool`: [3](#0-2) 

`_checkIfDepositAmountExceedesCurrentLimit` for ETH ignores the incoming `amount` and checks only whether the already-accumulated total exceeds the cap: [4](#0-3) 

If this returns `true`, `depositETH` reverts unconditionally: [5](#0-4) 

Because `address(this).balance` is used directly—rather than an internal accounting variable—any ETH sent via `receive()` is indistinguishable from legitimately deposited ETH and inflates the cap counter.

### Impact Explanation
Once `totalAssetDeposits > depositLimitByAsset(ETH_TOKEN)`, every call to `depositETH` reverts with `MaximumDepositLimitReached`. This blocks all user ETH deposits and also blocks `L1Vault.depositETHForL1VaultETH`, which calls `lrtDepositPool.depositETH` internally: [6](#0-5) 

The freeze persists until an admin raises the deposit cap. This constitutes **temporary freezing of funds** (Medium).

### Likelihood Explanation
The protocol operates with a finite `depositLimitByAsset` for ETH. When the protocol is near capacity—a common operational state—an attacker needs only to send `depositLimit - totalAssetDeposits + 1 wei` of ETH directly to `LRTDepositPool`. The attacker's ETH is absorbed by the protocol (not recoverable by the attacker), making this a low-cost griefing attack. No special permissions are required; any externally-owned account can call `receive()`.

### Recommendation
Replace the raw `address(this).balance` read in `getETHDistributionData()` with an internal accounting variable (e.g., `ethReceivedViaDeposit`) that is incremented only inside `depositETH` and decremented on transfers out. ETH arriving via `receive()` or `receiveFromNodeDelegator()` should not be counted toward the deposit cap until it is explicitly processed.

### Proof of Concept
1. Assume `depositLimitByAsset(ETH_TOKEN) = 10_000 ETH` and current `totalAssetDeposits = 9_999.9 ETH`.
2. Attacker calls `LRTDepositPool.receive()` with `0.1 ETH + 1 wei`.
3. `address(this).balance` increases; `getETHDistributionData()` now returns `ethLyingInDepositPool` that pushes `totalAssetDeposits > 10_000 ETH`.
4. Any user calling `depositETH(...)` receives `MaximumDepositLimitReached`.
5. `L1Vault.depositETHForL1VaultETH()` also reverts, blocking the cross-chain bridging pipeline.
6. State persists until an admin calls `LRTConfig.setDepositLimitByAsset` to raise the cap.
