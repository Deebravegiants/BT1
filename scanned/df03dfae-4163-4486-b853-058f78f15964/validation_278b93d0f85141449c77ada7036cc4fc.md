### Title
Rewards Permanently Locked When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: `contracts/KERNEL/KernelDepositPool.sol`)

---

### Summary

`KernelDepositPool` allows any staker to call `initiateWithdrawal` and reduce `totalKernelStaked` to zero at any time, including during an active reward distribution window. When this happens, the `rewardPerToken()` function freezes the per-token accumulator and the `updateReward` modifier advances `updatedAt` past the zero-staked interval. All reward tokens that should have been distributed during that interval are permanently locked in the contract with no recovery path.

---

### Finding Description

`KernelDepositPool` is a Synthetix-style staking rewards contract. Rewards accumulate continuously at `rewardRate` tokens per second, pro-rated across `totalKernelStaked`. The critical accounting path is the `updateReward` modifier:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();   // snapshot accumulator
    updatedAt = lastTimeRewardApplicable();    // advance time cursor
    ...
}
```

`rewardPerToken()` handles the zero-staked case by returning the stored value unchanged:

```solidity
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;           // accumulator frozen
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
```

When `totalKernelStaked == 0`:
- `rewardPerTokenStored` is not incremented (rewards stop accruing to anyone).
- `updatedAt` **is** advanced to `lastTimeRewardApplicable()`.

The consequence is that the time interval `[T_zero, T_restake]` is silently consumed: when the next staker arrives, `updatedAt` already equals the current time, so the reward tokens that should have covered that interval are never attributed to anyone. They remain in the contract balance forever because there is no `recoverRewards`, `sweep`, or similar admin function.

`initiateWithdrawal` imposes no guard against driving `totalKernelStaked` to zero during an active period:

```solidity
function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
    ...
    balanceOf[msg.sender] -= _amount;
    totalKernelStaked -= _amount;          // no check: can reach 0 mid-period
    ...
}
```

The contract's own NatSpec acknowledges the risk but relies entirely on an off-chain operational promise:

> *"If `totalKernelStaked` ever hits zero during a reward distribution window, any remaining rewards for that period will stay locked in the contract. In this deployment, we're avoiding this issue by ensuring there are always some tokens staked…"*

No code enforces this invariant.

---

### Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

Every reward token accrued during the zero-staked interval is irrecoverably locked in the contract. The magnitude equals `rewardRate × duration_of_zero_staked_interval`. For a typical 7-day reward window at a meaningful `rewardRate`, this can represent a substantial fraction of the total reward budget. No admin function exists to rescue these tokens.

---

### Likelihood Explanation

Any single unprivileged staker who holds the last remaining stake can trigger this unilaterally by calling `initiateWithdrawal`. The scenario is realistic:

- Reward periods are finite and publicly visible on-chain.
- A rational actor who has already claimed their rewards has no incentive to keep tokens staked.
- A malicious actor can deliberately time the withdrawal to maximise the locked interval (e.g., withdraw at the start of a new reward period and restake just before it ends, locking the bulk of the rewards).

---

### Recommendation

1. **Code-level guard**: In `initiateWithdrawal`, revert (or skip reward-period advancement) when the withdrawal would set `totalKernelStaked` to zero while `block.timestamp < finishAt`.
2. **Recovery function**: Add an admin-callable `recoverUnallocatedRewards()` that transfers any reward tokens in excess of what is owed to current stakers back to a treasury address.
3. **Alternatively**: Mirror the fix pattern from the referenced report — when `totalKernelStaked` reaches zero during an active period, immediately set `finishAt = block.timestamp` to end the reward window and prevent further reward accumulation against an empty pool.

---

### Proof of Concept

1. Admin calls `notifyRewardAmount(1_000e18)` with `duration = 7 days`. `finishAt = T0 + 7 days`, `rewardRate = 1_000e18 / 604_800`.
2. Alice is the sole staker with `balanceOf[Alice] = 1_000e18`, `totalKernelStaked = 1_000e18`.
3. At `T0 + 1 day`, Alice calls `initiateWithdrawal(1_000e18)`. `updateReward` runs: `rewardPerTokenStored` is updated for 1 day of rewards, `updatedAt = T0 + 1 day`. Then `totalKernelStaked = 0`.
4. No one stakes for the next 6 days. Every call that touches `updateReward` (e.g., Alice calling `getReward`) advances `updatedAt` toward `finishAt` while `rewardPerTokenStored` stays frozen.
5. At `T0 + 7 days`, the reward period ends. Alice claims her 1-day share of rewards. The remaining ~857e18 reward tokens (6/7 of the budget) are permanently locked in the contract — no staker ever earned them, and no function can retrieve them. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

    function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
        if (_amount == 0) revert AmountZero();

        // Prevent starting a reward period when no tokens are staked to avoid unallocated rewards
        if (totalKernelStaked == 0) revert NoStakedTokens();

        // Transfer reward tokens into the contract
        uint256 balanceBefore = rewardsToken.balanceOf(address(this));
        rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
        uint256 balanceAfter = rewardsToken.balanceOf(address(this));
        // Calculate the actual amount of tokens received in case of a transfer fee (tax)
        uint256 receivedAmount = balanceAfter - balanceBefore;

        if (block.timestamp >= finishAt) {
            rewardRate = receivedAmount / duration;
        } else {
            uint256 remaining = (finishAt - block.timestamp) * rewardRate;
            rewardRate = (receivedAmount + remaining) / duration;
        }

        if (rewardRate == 0) revert RewardRateZero();

        finishAt = block.timestamp + duration;
        updatedAt = block.timestamp;

        emit NotifyRewardAmount(receivedAmount, finishAt);
    }
