### Title
Merkle Leaf Excludes Token Address, Allowing Token Substitution After `setToken()` - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

### Summary
`MerkleDistributor.claim()` constructs the Merkle leaf without including the token address. Because the contract also exposes a `setToken()` admin function (explicitly designed to be called post-deployment), any existing unclaimed proof remains valid against the new token. Users who delay claiming will receive a different token than the one their proof was generated for.

### Finding Description
In `MerkleDistributor._claim()` (the inline logic inside `claim()`), the leaf node is computed as:

```solidity
bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));
``` [1](#0-0) 

The token address is absent from this encoding. The contract simultaneously exposes `setToken()`, which allows the owner to replace the distributed token at any time:

```solidity
function setToken(address _token) external onlyOwner {
    token = _token;
}
``` [2](#0-1) 

The `initialize()` function even documents this design intent with the comment `// token can be set later`, meaning the contract is explicitly built to support post-deployment token changes. [3](#0-2) 

Because the Merkle proof is bound only to `(index, account, cumulativeAmount)` and not to the token, a proof generated when `token == tokenA` remains cryptographically valid after `token` is changed to `tokenB`. The `claim()` function will then transfer `tokenB` to the user. [4](#0-3) 

### Impact Explanation
Users who have not yet claimed their tokens when `setToken()` is called will receive the new token instead of the one their Merkle allocation was computed for. If the new token has a different value (e.g., a higher-value token replaces a lower-value one), claimants drain the contract of the more valuable asset using proofs that were never authorized for it. If the new token is worth less, users silently lose yield. In either direction the contract fails to deliver the promised token to its rightful recipients.

**Impact: Low — Contract fails to deliver promised returns.**

### Likelihood Explanation
The `setToken()` function is a documented, intentional admin capability (the `initialize()` comment explicitly states "token can be set later"). Any routine token migration, upgrade, or correction by the owner triggers the condition. No attacker action is required beyond submitting a valid claim after the token has been changed.

### Recommendation
Include the token address in the Merkle leaf encoding so that each proof is cryptographically bound to the specific token it was generated for:

```solidity
bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount, token));
```

Alternatively, if the token must remain mutable, snapshot the token address at the time each Merkle root is set and store it per-root, then verify against that snapshot during claim.

### Proof of Concept
1. Owner deploys `MerkleDistributor`, initializes with `token = rsETH`, and calls `setMerkleRoot(root1)` where `root1` encodes Alice's entitlement of `100e18` rsETH at `index = 1`.
2. Alice does not claim immediately.
3. Owner calls `setToken(WETH)` to migrate the distributor to WETH rewards (a legitimate admin operation).
4. Alice calls `claim(1, alice, 100e18, proof)` with her original proof.
5. The leaf `keccak256(abi.encodePacked(1, alice, 100e18))` verifies correctly against `root1` — the token change is invisible to the proof system.
6. `IERC20(token).safeTransfer(alice, amountToSend)` transfers WETH, not rsETH.

Alice receives WETH instead of rsETH. The proof she holds was never authorized for WETH, but the contract accepts it because the token address was never committed into the leaf. [5](#0-4)
