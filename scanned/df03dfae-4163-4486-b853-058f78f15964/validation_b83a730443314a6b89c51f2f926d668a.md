### Title
Hardcoded 18-Decimal Assumption in `_getTotalEthInProtocol()` and `getRsETHAmountToMint()` Causes Incorrect rsETH Minting and Price Miscalculation for Non-18 Decimal Assets - (File: contracts/LRTOracle.sol, contracts/LRTDepositPool.sol)

---

### Summary

`LRTOracle._getTotalEthInProtocol()` and `LRTDepositPool.getRsETHAmountToMint()` both silently assume every supported asset has 18 decimals. If a non-18 decimal token is ever added to the supported asset list, the TVL aggregation and rsETH mint-amount calculations produce values that are off by a factor of `10^(18 - tokenDecimals)`, leading to either massive under-minting (loss of user funds) or over-minting (protocol insolvency), and potentially triggering an erroneous protocol-wide pause.

---

### Finding Description

**Root cause 1 — `LRTOracle._getTotalEthInProtocol()`** [1](#0-0) 

```solidity
uint256 assetER = getAssetPrice(asset);          // 1e18-precision price
uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
// comment: "totalAssetAmt is in 1e18 precision (standard token decimals)"
totalETHInProtocol += totalAssetAmt.mulWad(assetER);   // mulWad = x * y / 1e18
```

`mulWad` divides by `1e18`, which is only correct when `totalAssetAmt` is itself in 1e18 units. The inline comment explicitly documents the assumption: *"standard token decimals"* = 18. For a 6-decimal token (e.g. USDC), `totalAssetAmt` is in 1e6 units, so the contribution to `totalETHInProtocol` is `1e12` times too small.

**Root cause 2 — `LRTDepositPool.getRsETHAmountToMint()`** [2](#0-1) 

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`amount` is in the asset's native decimals. `getAssetPrice` and `rsETHPrice` are both 1e18-precision. For an 18-decimal asset the formula is dimensionally correct. For a 6-decimal asset the result is `1e12` times smaller than the correct rsETH amount (rsETH is 18 decimals).

The same pattern appears in the L2 pool swap helpers: [3](#0-2) 

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

`amountAfterFee` is in the token's native decimals; both rates are 1e18. Same decimal mismatch.

---

### Impact Explanation

**Scenario A — token with fewer than 18 decimals (e.g. 6-decimal USDC)**

- `_getTotalEthInProtocol()` understates TVL by `1e12×`, causing `newRsETHPrice` to drop far below `highestRsethPrice`.
- The downside-protection branch in `_updateRsETHPrice()` fires, pausing `LRTDepositPool`, `LRTWithdrawalManager`, and `LRTOracle` — a **temporary freeze of all user funds**.
- Simultaneously, `getRsETHAmountToMint()` mints `1e12×` fewer rsETH per deposited unit, so any user who deposits before the pause is credited almost nothing — **direct loss of user funds**.

**Scenario B — token with more than 18 decimals**

- TVL is overstated, rsETH price is inflated, and depositors receive `1e(d-18)×` more rsETH than the underlying value warrants — **protocol insolvency / theft of yield from existing holders**.

Impact classification: **Critical** (permanent or temporary fund freeze / direct theft) depending on direction of decimal mismatch.

---

### Likelihood Explanation

Adding a new supported asset requires an `onlyLRTAdmin` call to `LRTConfig`. This is not an attacker-controlled path in isolation, but:

- The admin sanity check in `updatePriceOracleForValidated` only validates that the oracle price is between `1e16` and `1e19`; it does **not** check token decimals.
- No other guard in `LRTConfig.addNewSupportedAsset` enforces 18 decimals.
- A well-intentioned admin adding a new collateral type (e.g. a future LST with non-standard decimals, or a stablecoin) would silently introduce the bug.

Likelihood: **Low** (requires admin to add a non-18 decimal asset, but no on-chain guard prevents it).

---

### Recommendation

1. In `_getTotalEthInProtocol()`, normalize `totalAssetAmt` to 18 decimals before calling `mulWad`:
   ```solidity
   uint8 decimals = IERC20Metadata(asset).decimals();
   uint256 normalizedAmt = totalAssetAmt * 10 ** (18 - decimals);
   totalETHInProtocol += normalizedAmt.mulWad(assetER);
   ```
2. Apply the same normalization in `getRsETHAmountToMint()` and in every L2 pool `viewSwapRsETHAmountAndFee(amount, token)` before using `amount` in arithmetic with 1e18-precision rates.
3. Add an on-chain check in `LRTConfig.addNewSupportedAsset` (or in `updatePriceOracleFor`) that reverts if `IERC20Metadata(asset).decimals() != 18`, or explicitly document and enforce the 18-decimal invariant.

### Proof of Concept

**Setup:** Admin adds a 6-decimal token (e.g. USDC, `decimals = 6`) as a supported asset with a valid oracle returning `1e18` (1 USDC ≈ 1 ETH for simplicity). `rsETHPrice = 1e18`.

**Step 1 — User deposits 1 USDC (= `1e6` raw units):**

```
rsethAmountToMint = (1e6 * 1e18) / 1e18 = 1e6
```

User receives `1e6` wei of rsETH instead of `1e18` wei — **`1e12×` shortfall**.

**Step 2 — Oracle update (`_getTotalEthInProtocol`):**

```
totalAssetAmt = 1e6   (USDC balance in raw units)
assetER       = 1e18
contribution  = mulWad(1e6, 1e18) = 1e6 * 1e18 / 1e18 = 1e6
```

`totalETHInProtocol = 1e6` instead of `1e18`. With `rsethSupply = 1e6` (minted above):

```
newRsETHPrice = 1e6 / 1e6 = 1   (i.e. 1 wei per rsETH)
```

`highestRsethPrice = 1e18`, so `diff = 1e18 - 1 ≈ 1e18`, which exceeds any `pricePercentageLimit` → **protocol paused**, all user funds frozen. [4](#0-3) [5](#0-4) [6](#0-5)
