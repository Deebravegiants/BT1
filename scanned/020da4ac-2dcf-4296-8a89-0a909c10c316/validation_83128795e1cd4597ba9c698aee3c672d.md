### Title
Chainlink Price Oracle Missing Staleness Check Enables Incorrect rsETH Minting and Share Dilution - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards the `updatedAt` and `answeredInRound` return values, providing no protection against stale Chainlink price data. This stale price propagates directly into rsETH minting calculations, allowing depositors to receive inflated rsETH amounts during oracle downtime, diluting existing holders.

### Finding Description
`ChainlinkPriceOracle.getAssetPrice()` fetches the Chainlink price but silently ignores staleness indicators:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

The `updatedAt` timestamp and `answeredInRound` fields are never checked against any maximum staleness threshold. [1](#0-0) 

This stale price is consumed by two critical paths:

**Path 1 — rsETH minting:** `LRTDepositPool.getRsETHAmountToMint()` computes:
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [2](#0-1) 

`lrtOracle.getAssetPrice(asset)` delegates directly to `ChainlinkPriceOracle.getAssetPrice()`. [3](#0-2) 

**Path 2 — rsETH price update:** `_getTotalEthInProtocol()` calls `getAssetPrice(asset)` for every supported asset to compute total TVL, which then drives `_updateRsETHPrice()` including protocol fee minting. [4](#0-3) 

### Impact Explanation
If a Chainlink feed for a supported LST (e.g., stETH, ETHx, sfrxETH) becomes stale and its last reported price is higher than the true current price (e.g., during a depeg event or oracle downtime), any depositor calling `depositAsset()` will receive more rsETH than their deposit is worth. This inflates rsETH supply relative to actual TVL, permanently diluting the share value of all existing rsETH holders — a theft of yield from existing holders. The same stale price fed into `_updateRsETHPrice()` can also cause the protocol fee to be computed on phantom gains, minting unearned fee rsETH to the treasury.

**Impact:** High — theft of unclaimed yield / share dilution of existing rsETH holders.

### Likelihood Explanation
Chainlink feeds have documented downtime during network congestion and can remain at their last reported value for extended periods. LST price feeds (stETH/ETH, ETHx/ETH) are particularly relevant because they can diverge from true value during depeg events — exactly when staleness protection matters most. The entry path requires no special privileges: any user can call `depositAsset()` or `depositETH()` on `LRTDepositPool`. [5](#0-4) 

### Recommendation
Add a configurable `maxStaleness` parameter to `ChainlinkPriceOracle` and validate `updatedAt` in `getAssetPrice()`:

```solidity
(, int256 price,, uint256 updatedAt,) = priceFeed.latestRoundData();
if (block.timestamp - updatedAt > maxStaleness) revert StalePriceFeed();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

A per-asset staleness threshold is preferable since different feeds have different heartbeat intervals (e.g., 1 hour for ETH/USD vs. 24 hours for some LST feeds).

### Proof of Concept
1. Chainlink's stETH/ETH feed stops updating (e.g., network congestion). Last reported price: `1.05e18` (stETH at a premium). True current price: `0.98e18` (stETH has depegged).
2. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
3. `getRsETHAmountToMint` computes: `(1000e18 * 1.05e18) / rsETHPrice` → attacker receives rsETH priced as if stETH = 1.05 ETH.
4. True value deposited is only `980 ETH` worth, but attacker receives rsETH worth `1050 ETH` worth.
5. Attacker immediately redeems or sells rsETH, extracting ~70 ETH of value from existing holders.
6. `_updateRsETHPrice()` called with stale prices also computes inflated TVL, potentially minting unearned protocol fee rsETH to treasury. [6](#0-5)
