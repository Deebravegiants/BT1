### Title
Missing Chainlink `updatedAt` Staleness Check Allows Stale Asset Price to Corrupt rsETH Exchange Rate — (`contracts/oracles/ChainlinkPriceOracle.sol`)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards the `updatedAt` timestamp, accepting arbitrarily old prices. This feeds directly into `LRTOracle._getTotalEthInProtocol()` and therefore into `_updateRsETHPrice()`, which sets the canonical `rsETHPrice` used for all deposits and withdrawals. A stale asset price causes the rsETH exchange rate to be computed incorrectly, enabling depositors to receive more rsETH than their assets are worth (or fewer), at the expense of existing rsETH holders.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the Chainlink round but only reads the `price` field:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol  line 52
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

The `updatedAt` return value is never checked against `block.timestamp`. There is no heartbeat or maximum-age guard. Chainlink feeds can go stale during network congestion, sequencer downtime (on L2), or oracle-node failures; in all such cases the contract will continue to accept the last-known price indefinitely.

This price is consumed by `LRTOracle._getTotalEthInProtocol()`:

```solidity
// contracts/LRTOracle.sol  line 339
uint256 assetER = getAssetPrice(asset);
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
``` [2](#0-1) 

`_getTotalEthInProtocol()` is called by `_updateRsETHPrice()`, which writes the result to the public `rsETHPrice` storage variable:

```solidity
// contracts/LRTOracle.sol  line 250 / 313
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
...
rsETHPrice = newRsETHPrice;
``` [3](#0-2) [4](#0-3) 

`updateRSETHPrice()` is a public, permissionless function, so any caller can trigger the stale-price write:

```solidity
// contracts/LRTOracle.sol  line 87
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [5](#0-4) 

The stored `rsETHPrice` is then consumed by L2 rate providers and L2 pool oracles:

```solidity
// contracts/cross-chain/RSETHMultiChainRateProvider.sol  line 27
function getLatestRate() public view override returns (uint256) {
    return ILRTOracle(rsETHPriceOracle).rsETHPrice();
}
``` [6](#0-5) 

The L2 pools (`RSETHPoolV3`, `RSETHPoolNoWrapper`) use this rate to compute how many rsETH tokens to mint per deposited ETH or collateral token:

```solidity
// contracts/pools/RSETHPoolNoWrapper.sol  line 282-285
uint256 rsETHToETHrate = getRate();          // reads rsETHPrice via oracle
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [7](#0-6) 

A parallel missing-staleness issue exists in `ChainlinkOracleForRSETHPoolCollateral.getRate()`, which checks `answeredInRound < roundID` (in-flight stale) but omits any time-based heartbeat check, so a price that is days old but from the latest round passes silently:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol  line 30-32
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
// ← no: block.timestamp - timestamp > heartbeat check
``` [8](#0-7)

### Impact Explanation

**Scenario A — stale price is lower than the true market price** (e.g., stETH/ETH feed freezes during a stETH appreciation event):

- `_getTotalEthInProtocol()` underestimates TVL.
- `newRsETHPrice` is computed below its true value.
- A depositor calling `updateRSETHPrice()` then depositing receives more rsETH than their ETH is worth, diluting existing holders. This is theft of yield from existing rsETH holders.

**Scenario B — stale price is higher than the true market price** (e.g., feed freezes during a collateral-token depeg):

- `_getTotalEthInProtocol()` overestimates TVL.
- `newRsETHPrice` is inflated.
- New depositors receive fewer rsETH; existing holders are temporarily enriched at the expense of new entrants. The protocol's accounting diverges from reality, risking insolvency when the price is corrected.

Impact classification: **High — theft of unclaimed yield / Low — contract fails to deliver promised returns**, depending on the direction of the stale deviation.

---

### Likelihood Explanation

Chainlink feeds have documented heartbeat windows (e.g., 1 hour for ETH/USD, 24 hours for some LST feeds). Sequencer downtime on L2s (Arbitrum, Optimism) can cause feeds to go stale for hours. The `updateRSETHPrice()` function is permissionless, so an attacker can call it immediately after a feed goes stale and before it recovers, locking in the incorrect price. Likelihood is **Medium** — requires a feed disruption, but no privileged access.

---

### Recommendation

Add a maximum-age check in `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(, int256 price,, uint256 updatedAt,) = priceFeed.latestRoundData();
require(block.timestamp - updatedAt <= MAX_ORACLE_AGE, "Stale price");
```

`MAX_ORACLE_AGE` should be set per feed based on its documented heartbeat (e.g., `3600` for 1-hour feeds, `86400` for 24-hour feeds). Apply the same fix to `ChainlinkOracleForRSETHPoolCollateral.getRate()`.

---

### Proof of Concept

1. Chainlink stETH/ETH feed last updated at `T=0` with price `1.05 ETH`.
2. At `T = 25 hours`, the feed has not updated (heartbeat missed). True stETH price is now `1.00 ETH`.
3. Attacker calls `LRTOracle.updateRSETHPrice()` (permissionless).
4. `_getTotalEthInProtocol()` uses the stale `1.05` price → TVL is 5% overstated.
5. `rsETHPrice` is written with an inflated value.
6. Attacker deposits ETH, receives fewer rsETH than expected (Scenario B), OR:
7. In Scenario A (feed freezes while price should be higher), attacker deposits and receives excess rsETH, redeemable for more ETH than deposited — direct theft from existing holders.
