### Title
Token Decimal Mismatch in `viewSwapRsETHAmountAndFee` Causes Near-Total Loss of Depositor Funds for Sub-18-Decimal Tokens - (File: contracts/pools/RSETHPool.sol, RSETHPoolV3.sol, RSETHPoolV3ExternalBridge.sol, RSETHPoolV3WithNativeChainBridge.sol, RSETHPoolNoWrapper.sol)

---

### Summary

The L2 pool contracts accept arbitrary ERC-20 tokens as collateral for minting rsETH. The `viewSwapRsETHAmountAndFee(uint256 amount, address token)` overload computes the rsETH output as:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

Both `tokenToETHRate` and `rsETHToETHrate` are in 1e18 precision, but `amountAfterFee` is in the token's native decimals. For an 18-decimal token the formula is accidentally correct; for any token with fewer than 18 decimals (e.g., WBTC at 8, USDC at 6) the result is `10^(18 - tokenDecimals)` times too small, causing the depositor to receive essentially zero rsETH while their full token balance is transferred into the pool and subsequently bridged to L1.

---

### Finding Description

Every pool variant that supports ERC-20 token deposits contains the same pattern. In `RSETHPool.sol`:

```solidity
// contracts/pools/RSETHPool.sol  lines 335-346
function viewSwapRsETHAmountAndFee(
    uint256 amount,
    address token
) public view onlySupportedToken(token) returns (uint256 rsETHAmount, uint256 fee) {
    uint256 feeBpsForToken = tokenFeeBps[token];
    fee = amount * feeBpsForToken / 10_000;
    uint256 amountAfterFee = amount - fee;

    uint256 rsETHToETHrate = getRate();                                    // 1e18-precision
    uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate(); // 1e18-precision

    rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;        // BUG: amountAfterFee is in tokenDecimals
}
```

The oracle wrapper `ChainlinkOracleForRSETHPoolCollateral.getRate()` always normalises its output to 1e18:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol  line 34
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
```

So `tokenToETHRate` and `rsETHToETHrate` are both 1e18-scaled, but `amountAfterFee` carries the token's raw decimal representation. The ETH deposit path correctly inserts the missing `1e18` factor:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;   // ETH path – correct
```

The token path omits this normalisation entirely.

The same bug is present in:
- `RSETHPoolNoWrapper.sol` line 311
- `RSETHPoolV3.sol` line 334
- `RSETHPoolV3ExternalBridge.sol` line 452
- `RSETHPoolV3WithNativeChainBridge.sol` line 370

---

### Impact Explanation

**Critical – direct theft of user funds.**

For a token with `d` decimals where `d < 18`, the rsETH minted is `10^(18-d)` times smaller than the correct amount.

Concrete example with WBTC (8 decimals), `tokenToETHRate = 30e18` (30 ETH/BTC), `rsETHToETHrate = 1.05e18`:

| | Correct | Actual |
|---|---|---|
| Deposit | 1 WBTC = 1e8 units | 1 WBTC = 1e8 units |
| Expected rsETH | `1e8 * 30e18 * 1e18 / (1.05e18 * 1e8)` ≈ 28.57e18 | — |
| Actual rsETH | `1e8 * 30e18 / 1.05e18` ≈ 2857 | 2857 wei |
| Loss factor | — | 1e18 / 2857 ≈ 3.5e14× underpayment |

The depositor's WBTC is transferred into the pool (`safeTransferFrom` at line 296 of `RSETHPool.sol`) and later bridged to L1 via `bridgeTokens` / `moveAssetsForBridging`. The user receives 2857 wei of rsETH (≈ $0) in exchange for 1 WBTC (≈ $60,000+). The funds are irrecoverably lost to the user.

---

### Likelihood Explanation

**Medium.** The vulnerability is latent: it is harmless while only 18-decimal tokens (wstETH, ETH) are supported. It activates the moment any token with fewer than 18 decimals is added via `addSupportedToken`. That function is gated behind `TIMELOCK_ROLE`, but it is a routine protocol-extension operation with no on-chain guard against sub-18-decimal tokens. The contracts are explicitly designed to be extensible to new collateral types, and common candidates (WBTC, cbBTC, USDC, USDT) all have fewer than 18 decimals. No attacker action is required once such a token is listed; every ordinary depositor calling `deposit(token, amount, referralId)` is immediately affected.

---

### Recommendation

Normalise `amountAfterFee` to 1e18 before applying the rate ratio, mirroring the ETH path:

```solidity
uint8 tokenDecimals = IERC20Metadata(token).decimals();
rsETHAmount = amountAfterFee * 1e18 / 10**tokenDecimals * tokenToETHRate / rsETHToETHrate;
```

Or equivalently:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate * 1e18 / (rsETHToETHrate * 10**tokenDecimals);
```

Apply the same fix to `viewSwapAssetToPremintedRsETH` (the reverse-swap path) in `RSETHPoolV3WithNativeChainBridge.sol` line 434, which has the symmetric error.

### Proof of Concept

1. Admin calls `addSupportedToken(WBTC, wbtcOracle, wbtcBridge)` on `RSETHPool` (Arbitrum).
2. User calls `deposit(WBTC, 1e8, "ref")` (1 WBTC).
3. `safeTransferFrom` moves 1e8 WBTC units from user to pool.
4. `viewSwapRsETHAmountAndFee(1e8, WBTC)` computes:
   - `fee = 0` (assume 0 bps)
   - `amountAfterFee = 1e8`
   - `tokenToETHRate = 30e18` (Chainlink normalised)
   - `rsETHToETHrate = 1.05e18`
   - `rsETHAmount = 1e8 * 30e18 / 1.05e18 = 2857`
5. Pool transfers 2857 wei of wrsETH to user (≈ $0).
6. User has lost ~$60,000 of WBTC; the WBTC sits in the pool until `bridgeTokens` sends it to L1. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
