### Title
Decimal Normalisation Missing in `viewSwapRsETHAmountAndFee` for Non-18-Decimal Tokens — (`contracts/pools/RSETHPool.sol`)

---

### Summary

`viewSwapRsETHAmountAndFee(uint256 amount, address token)` computes `rsETHAmount` by multiplying the raw token amount directly against a 1e18-precision oracle rate, without first scaling the token amount to 18 decimals. For any supported token with fewer than 18 decimals (e.g., USDC at 6 decimals), the output rsETH amount is 1e12× smaller than the economically correct value, while the fee is correctly deducted in the token's native units. The user loses their deposited tokens and receives a negligible rsETH amount.

---

### Finding Description

The token-deposit path in `RSETHPool.sol` computes the rsETH output as:

```
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate
``` [1](#0-0) 

Both `tokenToETHRate` (from `IOracle(supportedTokenOracle[token]).getRate()`) and `rsETHToETHrate` (from `IOracle(rsETHOracle).getRate()`) are 1e18-precision values. The formula is dimensionally correct **only when `amount` is also expressed in 1e18 units** (i.e., the token has 18 decimals).

Compare with the ETH-deposit path, which explicitly multiplies by `1e18` to compensate:

```
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [2](#0-1) 

For a 6-decimal token the discrepancy is 1e12:

| Variable | 18-decimal token (wstETH) | 6-decimal token (USDC) |
|---|---|---|
| `amount` | 1e18 | 1e6 |
| `tokenToETHRate` | ~1.2e18 | ~3.33e14 |
| `rsETHToETHrate` | ~1.05e18 | ~1.05e18 |
| `rsETHAmount` (actual) | ~1.143e18 ✓ | ~317 ✗ |
| `rsETHAmount` (expected) | ~1.143e18 | ~3.17e14 |

The fee is computed correctly in token-native units (`amount * feeBpsForToken / 10_000`), so the fee accounting is fine, but the rsETH transferred to the user is essentially zero. [3](#0-2) 

The `addSupportedToken` function, gated by `TIMELOCK_ROLE`, performs no decimal check on the token being added: [4](#0-3) 

---

### Impact Explanation

A user depositing 1 USDC (1e6 units) receives ~317 wei of wrsETH instead of ~3.17e14 wei. The deposited USDC (minus the small fee) remains in the pool. The wrsETH that should have been distributed to the user stays in the pool as unearned balance — constituting theft of the user's entitled yield/return. At scale, every deposit of a sub-18-decimal token results in near-total loss for the depositor.

**Impact: High — Theft of unclaimed yield** (the rsETH owed to depositors is never distributed; it remains locked in the pool).

---

### Likelihood Explanation

The precondition is that a token with fewer than 18 decimals is added via `addSupportedToken`. This requires `TIMELOCK_ROLE`, which is an admin action — but it is an entirely plausible and intended operational action (the contract is explicitly designed to support multiple tokens). No attacker compromise is required; any ordinary user depositing a supported sub-18-decimal token triggers the bug automatically.

---

### Recommendation

Normalise `amountAfterFee` to 18 decimals before applying the rate formula:

```solidity
uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 normalizedAmount = amountAfterFee * (10 ** (18 - tokenDecimals));
rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
```

Alternatively, enforce in `addSupportedToken` that only 18-decimal tokens may be added, reverting otherwise.

pragma solidity 0.8.27;

// Setup (local fork or mock):
// - Deploy RSETHPool with a mock rsETHOracle returning 1.05e18
// - Add USDC (6 decimals) as supported token with oracle returning 3.33e14

// - Fund pool with wrsETH

function testDecimalMismatch() public {
    uint256 depositAmount = 1e6; // 1 USDC

    (uint256 rsETHAmount, uint256 fee) =
        pool.viewSwapRsETHAmountAndFee(depositAmount, address(usdc));

    // fee is correctly ~100 USDC-units (1% of 1e6)
    assertApproxEqAbs(fee, 1e4, 10);

    // rsETHAmount should be ~3.17e14 (0.000317 rsETH in 18-decimal units)
    // but is actually ~314 — a factor of ~1e12 too small
    assertGt(rsETHAmount, 1e13, "rsETHAmount is 1e12x too small");
    // This assertion FAILS, proving the bug
}
