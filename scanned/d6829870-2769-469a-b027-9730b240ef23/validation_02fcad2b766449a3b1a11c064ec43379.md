### Title
Single-Request stETH Unstaking Design Causes Linear Increase in ETH Withdrawal Processing Time - (File: contracts/unstaking-adapters/UnstakeStETH.sol)

### Summary
`UnstakeStETH._unstakeStEth()` submits a single Lido withdrawal request for the entire `amountToUnstake`. Lido's `WithdrawalQueue` enforces a hard per-request cap of `1000 stETH` (`MAX_STETH_WITHDRAWAL_AMOUNT`). Any call with `amountToUnstake > 1000 ether` reverts at the Lido level. Additionally, `_claimStEth` only claims one Lido request ID at a time. Together, these constraints force the operator to make N sequential `unstakeStEth` + `claimStEth` call pairs for large conversions, each incurring a full 3–5 day Lido finalization delay, causing a linear increase in the time users must wait to complete ETH withdrawals.

### Finding Description
`UnstakeStETH._unstakeStEth()` always constructs a single-element `amounts` array and passes the full `amountToUnstake` to `withdrawalQueue.requestWithdrawals`:

```solidity
// contracts/unstaking-adapters/UnstakeStETH.sol L48-57
function _unstakeStEth(uint256 amountToUnstake) internal {
    stETH.safeIncreaseAllowance(address(withdrawalQueue), amountToUnstake);

    uint256[] memory amounts = new uint256[](1);
    amounts[0] = amountToUnstake;                          // no chunking

    uint256[] memory requestIds = withdrawalQueue.requestWithdrawals(amounts, address(this));
    emit UnstakeStETHStarted(requestIds[0]);
}
```

Lido's `WithdrawalQueue` enforces `MAX_STETH_WITHDRAWAL_AMOUNT = 1000 * 1e18`. Any single request exceeding this limit causes `requestWithdrawals` to revert. The `withinUnstakeLimits` modifier in `LRTConverter.unstakeStEth` imposes no such cap — it only checks against `whitelistedUnstakeAllowance + assetsCommitted(ETH_TOKEN)`, which can be arbitrarily large.

Similarly, `_claimStEth` only processes one Lido request ID per call:

```solidity
// contracts/unstaking-adapters/UnstakeStETH.sol L59-65
function _claimStEth(uint256 _requestId, uint256 _hint) internal {
    uint256[] memory requestIds = new uint256[](1);
    uint256[] memory hints    = new uint256[](1);
    requestIds[0] = _requestId;
    hints[0]      = _hint;
    withdrawalQueue.claimWithdrawalsTo(requestIds, hints, address(this));
}
```

`LRTConverter.claimStEth` wraps this with a single `(requestId, hint)` pair, so claiming N Lido requests requires N separate transactions.

The consequence is that converting X stETH to ETH requires `ceil(X / 1000)` sequential `unstakeStEth` calls, each producing a separate Lido request that takes 3–5 days to finalize, followed by `ceil(X / 1000)` sequential `claimStEth` calls. The total latency scales linearly with X.

### Impact Explanation
**Medium — Temporary freezing of funds.**

Users who call `LRTWithdrawalManager.initiateWithdrawal(ETH_TOKEN, ...)` depend on the operator converting stETH to ETH via `LRTConverter.unstakeStEth` → `claimStEth` → `unlockQueue`. When the protocol holds a large stETH position (e.g., 10 000 stETH), the operator must submit 10 sequential Lido requests. Each request takes 3–5 days to finalize. Users' ETH withdrawal requests remain locked in the queue until sufficient ETH is available, meaning a 10 000 stETH conversion could delay ETH withdrawals by 30–50 days. The funds are not permanently lost, but they are temporarily inaccessible to users for an unbounded, linearly growing period.

### Likelihood Explanation
**Medium.** rsETH is a large-scale LRT protocol. The protocol routinely holds thousands of stETH. Any time the ETH withdrawal queue requires stETH-to-ETH conversion exceeding 1000 stETH, the linear delay is triggered. This is a normal operational scenario, not an edge case.

### Recommendation
1. **Batch unstaking:** Modify `_unstakeStEth` to automatically split `amountToUnstake` into chunks of at most `1000 ether`, submit all chunks in a single `requestWithdrawals` call (Lido accepts an array), and store all returned `requestIds`.
2. **Batch claiming:** Modify `_claimStEth` (and `LRTConverter.claimStEth`) to accept arrays of `requestIds` and `hints`, forwarding them directly to `withdrawalQueue.claimWithdrawalsTo` in a single call.

### Proof of Concept

**Revert path (> 1000 ETH):**
1. Operator calls `LRTConverter.unstakeStEth(2000 ether)`.
2. `withinUnstakeLimits` passes (assume sufficient allowance).
3. `_unstakeStEth(2000 ether)` builds `amounts = [2000 ether]` and calls `withdrawalQueue.requestWithdrawals([2000 ether], address(this))`.
4. Lido reverts: `RequestAmountTooLarge` — the entire unstaking operation fails.

**Linear delay path (sequential 1000 ETH chunks):**
1. Operator calls `LRTConverter.unstakeStEth(1000 ether)` — Lido request #1 created. Wait 3–5 days.
2. Operator calls `LRTConverter.unstakeStEth(1000 ether)` — Lido request #2 created. Wait another 3–5 days.
3. … repeated 10 times for 10 000 stETH total.
4. Operator calls `LRTConverter.claimStEth(requestId1, hint1)` — only request #1 claimed.
5. … repeated 10 times.
6. Total elapsed time: 30–50 days. Users' ETH withdrawal requests remain locked throughout. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

```text
    modifier withinUnstakeLimits(uint256 amountToUnstake) {
        if (amountToUnstake == 0) {
            revert InvalidAmount();
        }

        uint256 availableActiveETHWithdrawals = _getActiveETHUserWithdrawals();

        if (amountToUnstake > whitelistedUnstakeAllowance + availableActiveETHWithdrawals) {
            revert UnstakeLimitExceeded();
        }

        // Consume intended withdrawal limit
        if (whitelistedUnstakeAllowance > 0) {
            uint256 whitelistedAmountConsumed =
                amountToUnstake > whitelistedUnstakeAllowance ? whitelistedUnstakeAllowance : amountToUnstake;

            whitelistedUnstakeAllowance -= whitelistedAmountConsumed;
        }
        _;
```
