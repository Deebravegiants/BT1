### Title
Missing Chainlink `latestRoundData()` Return Value Validation Allows Stale/Invalid Price to Corrupt rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `price`, performing no staleness, completeness, or sign checks. This is the exact same unfixed pattern as the referenced Issue 94. The corrupted price propagates directly into rsETH minting calculations in `LRTDepositPool`, enabling over-minting (fund theft from existing holders) or under-minting (loss for depositors).

---

### Finding Description

In `contracts/oracles/ChainlinkPriceOracle.sol`, `getAssetPrice()` reads from a Chainlink feed as follows:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

All five return values from `latestRoundData()` — `roundId`, `answer`, `startedAt`, `updatedAt`, `answeredInRound` — are available, but only `price` (`answer`) is used. The following critical checks are absent:

| Missing Check | Risk |
|---|---|
| `price > 0` | Negative `int256` cast to `uint256` wraps to ~`2^256`, causing massive over-minting |
| `updatedAt != 0` | Incomplete/unstarted round returns zero timestamp |
| `answeredInRound >= roundId` | Stale answer from a prior round |
| `block.timestamp - updatedAt <= heartbeat` | Time-based staleness (feed not updated within its heartbeat) |

Contrast this with `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol`, which correctly validates all three conditions on the same `latestRoundData()` call:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

The same project applies proper validation in one oracle wrapper but not the other.

The corrupted price flows through `LRTOracle._getTotalEthInProtocol()`:

```solidity
uint256 assetER = getAssetPrice(asset);
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
``` [3](#0-2) 

This inflated/deflated `totalETHInProtocol` then determines `newRsETHPrice`:

```solidity
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
``` [4](#0-3) 

And directly controls how many rsETH tokens a depositor receives:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [5](#0-4) 

---

### Impact Explanation

**Critical — Direct theft of user funds / protocol insolvency.**

Worst-case scenario: `price` is negative (e.g., a misconfigured or malfunctioning feed returns `-1`). The unchecked cast `uint256(-1)` = `2^256 - 1`. This makes `assetER` astronomically large, inflating `totalETHInProtocol` to near-`uint256` max, which sets `newRsETHPrice` to an enormous value. A depositor calling `depositAsset()` at this moment receives rsETH calculated against the inflated `assetPrice` in the numerator and the inflated `rsETHPrice` in the denominator — the exact ratio depends on which asset is affected, but any imbalance allows an attacker to extract value from the pool.

More realistic scenario: a stale price (feed not updated within its heartbeat, or `answeredInRound < roundId` during Chainlink sequencer issues) causes the rsETH exchange rate to diverge from true asset values. An attacker who monitors the mempool can deposit when the stale price is favorable (e.g., stale high price for the deposited asset) and immediately request withdrawal, extracting ETH value from other depositors.

---

### Likelihood Explanation

**Medium.** Chainlink feeds do go stale during network congestion, sequencer downtime (on L2s), or oracle node failures. The `answeredInRound < roundId` condition is a documented Chainlink staleness indicator. The protocol supports multiple LST assets (stETH, rETH, swETH, ETHx, sfrxETH), each with its own feed — any single feed going stale triggers the vulnerability. The attack requires no special permissions; `depositAsset()` is callable by any user.

---

### Recommendation

Apply the same validation pattern already used in `ChainlinkOracleForRSETHPoolCollateral.sol` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    // Optional: add heartbeat check
    // if (block.timestamp - updatedAt > HEARTBEAT) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

### Proof of Concept

1. Assume `stETH` is a supported asset with a Chainlink feed configured in `ChainlinkPriceOracle`.
2. The Chainlink ETH/stETH feed goes stale (e.g., `answeredInRound < roundId`) and returns a price 10% higher than the true market rate.
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
4. `_beforeDeposit` calls `getRsETHAmountToMint(stETH, 100e18)`.
5. `getRsETHAmountToMint` calls `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns stale inflated price (no revert).
6. Attacker receives ~10% more rsETH than the true ETH value of their deposit.
7. Attacker redeems rsETH via `LRTWithdrawalManager`, extracting ETH from other depositors' share of the pool.

The entry path is fully permissionless: `depositAsset()` → `_beforeDeposit()` → `getRsETHAmountToMint()` → `LRTOracle.getAssetPrice()` → `ChainlinkPriceOracle.getAssetPrice()`. [6](#0-5) [7](#0-6)
