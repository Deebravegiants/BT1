### Title
Aave WETH Pool High Utilization DoSes ETH Withdrawal Completions - (File: contracts/LRTWithdrawalManager.sol)

### Summary
When the Aave integration is enabled in `LRTWithdrawalManager`, the `completeWithdrawal` flow calls `_withdrawFromAave`, which calls `aaveWETHGateway.withdrawETH` without any revert-handling. If Aave's WETH pool is at or near 100% utilization (i.e., most WETH is borrowed out), this external call reverts, causing the entire `completeWithdrawal` transaction to revert. Users with unlocked ETH withdrawal requests are temporarily unable to claim their funds.

### Finding Description
`LRTWithdrawalManager._processWithdrawalCompletion` (called by the public `completeWithdrawal` and `completeWithdrawalForUser`) contains the following block when Aave integration is enabled and the contract's idle ETH balance is insufficient to cover the user's request:

```solidity
// If Aave integration is enabled and asset is ETH, withdraw from Aave if needed
if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
    uint256 contractBalance = address(this).balance;
    if (contractBalance < request.expectedAssetAmount) {
        uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
        _withdrawFromAave(amountNeeded);   // <-- can revert
        ...
    }
}
```

`_withdrawFromAave` unconditionally calls:

```solidity
aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
```

Aave's `withdrawETH` reverts when the underlying WETH balance held by the aWETH contract is less than the requested amount — i.e., when pool utilization is near 100% and most WETH has been borrowed out. There is no `try/catch` or pre-flight liquidity check before this call.

The protocol itself acknowledges this risk for the deposit direction: in `unlockQueue`, the Aave deposit is wrapped in a `try/catch` with an explicit comment "Silently fail if Aave deposit fails (e.g., pool at max capacity)". No equivalent protection exists for the withdrawal direction in `_processWithdrawalCompletion`.

The contract even exposes a view function `getAaveWithdrawableLiquidity` that returns the WETH balance of the aWETH contract, confirming the team is aware that withdrawable liquidity can be constrained — but this check is never performed before calling `_withdrawFromAave` in the withdrawal completion path.

### Impact Explanation
When Aave's WETH pool utilization is near 100%, every call to `completeWithdrawal` for ETH (where the contract's idle balance is insufficient) reverts. Users whose withdrawal requests have already been unlocked by the operator cannot claim their ETH. Their funds remain locked in the contract until Aave's utilization drops or an admin calls `emergencyWithdrawFromAave` (PAUSER_ROLE only). This constitutes a **temporary freezing of funds**.

### Likelihood Explanation
Aave's WETH pool regularly experiences high utilization during periods of market stress or high borrowing demand. An attacker with sufficient capital can also deliberately borrow WETH from Aave to push utilization to 100%, then wait for users to attempt withdrawal completions. The condition is externally triggerable and requires no privileged access. The Aave integration is an active, enabled feature of the protocol.

### Recommendation
Wrap the `_withdrawFromAave` call in `_processWithdrawalCompletion` with a pre-flight liquidity check (using `getAaveWithdrawableLiquidity`) and/or a `try/catch`, mirroring the pattern already used for Aave deposits in `unlockQueue`. If the Aave withdrawal fails or is insufficient, the function should either revert with a descriptive error (so the user knows to retry later) or fall back to whatever idle ETH is available rather than reverting the entire completion.

```solidity
// Before calling _withdrawFromAave, check available liquidity
uint256 aaveLiquidity = IERC20(WETH_ADDRESS).balanceOf(address(aaveAWETH));
if (aaveLiquidity >= amountNeeded) {
    _withdrawFromAave(amountNeeded);
} else {
    revert InsufficientAaveLiquidity();
}
```

Alternatively, cap the withdrawal to `min(amountNeeded, aaveLiquidity)` and revert with `InsufficientLiquidityForWithdrawal` if the resulting balance is still insufficient, rather than letting the Aave call revert opaquely.

### Proof of Concept

1. The Aave integration is enabled (`isAaveIntegrationEnabled = true`) and the `LRTWithdrawalManager` has deposited ETH into Aave via `_depositToAave`.
2. An attacker (or natural market conditions) causes Aave's WETH pool utilization to reach ~100% by borrowing most of the available WETH.
3. A user whose ETH withdrawal request has been unlocked calls `completeWithdrawal(ETH_TOKEN, ...)`.
4. Inside `_processWithdrawalCompletion`, `address(this).balance < request.expectedAssetAmount`, so `_withdrawFromAave` is called.
5. `_withdrawFromAave` calls `aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this))`.
6. Aave reverts because `IERC20(WETH_ADDRESS).balanceOf(address(aaveAWETH)) < withdrawnAmount`.
7. The entire `completeWithdrawal` transaction reverts. The user cannot claim their ETH.
8. This persists for all ETH withdrawal completions until Aave utilization drops or an admin intervenes.

```text
    function _withdrawFromAave(uint256 amount) internal returns (uint256 withdrawnAmount) {
        if (amount == 0) return 0;

        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        if (aaveBalance == 0) revert InsufficientAaveBalance();

        // Only withdraw up to the principal amount (don't use accrued interest for user withdrawals)
        uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave;

        withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
        if (withdrawnAmount == 0) return 0;

        aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
        totalETHDepositedToAave -= withdrawnAmount;

        emit ETHWithdrawnFromAave(withdrawnAmount, totalETHDepositedToAave);
    }
```
