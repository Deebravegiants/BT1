### Title
Prior User's Stranded ETH Silently Consumed to Subsidize Subsequent WETH Swap — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary

The `pay()` function's partial-native branch unconditionally spends all ETH held by the router to co-fund the current caller's WETH swap. Any ETH left in the router from a prior user's transaction is silently transferred to the pool on behalf of the current swapper, causing the prior user to lose their ETH with no recourse.

### Finding Description

`PeripheryPayments.pay()` contains three branches for WETH payment: [1](#0-0) 

When `0 < address(this).balance < value`, the middle branch fires:

1. Wraps **all** of the router's native ETH via `IWETH9.deposit{value: nativeBalance}()`
2. Transfers that WETH to the pool
3. Pulls only the remainder (`value - nativeBalance`) from the actual payer via `safeTransferFrom`

The router's native balance is treated as belonging to the current payer, but it may contain ETH deposited by a completely different user in a prior transaction.

**How ETH accumulates between transactions:**

- A user calls `exactInputSingle` with `tokenIn=WETH` and `msg.value > amountIn`. The `pay()` first branch (`nativeBalance >= value`) wraps only `amountIn`, leaving `msg.value - amountIn` ETH stranded in the router.
- A user calls any payable function with `msg.value > 0` for a non-WETH token; all ETH stays in the router.
- A user uses `multicall` and omits `refundETH()` at the end.

There is no automatic refund at the end of any swap function. `refundETH()` must be explicitly called. [2](#0-1) 

### Impact Explanation

**Direct loss of user principal.** User A's ETH is transferred to the pool to settle User B's swap. User B pays `value - nativeBalance` WETH instead of `value` WETH; the pool receives the correct total, so no pool invariant breaks — but User A's ETH is permanently gone. This meets the Critical/High threshold of direct loss of user funds.

### Likelihood Explanation

The scenario is realistic and requires no attacker coordination:

- Sending slightly more ETH than `amountIn` for a WETH swap is a common UX pattern (users round up to avoid slippage-induced reverts).
- The stranded ETH is visible on-chain; a griever or MEV bot can immediately follow with a WETH `exactInputSingle` to drain it.
- No privileged access, malicious pool, or non-standard token is required.

### Recommendation

In the partial-native branch, do not silently consume the router's entire native balance. Instead, require the caller to have explicitly sent ETH for this transaction (e.g., track `msg.value` in transient storage at entry and use only that amount), or remove the partial-native hybrid path entirely and require callers to either send the full amount as ETH or approve the full amount as WETH.
