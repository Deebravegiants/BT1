### Title
`whenNotPaused` on `completeWithdrawal` Temporarily Freezes Already-Unlocked User Funds - (File: contracts/LRTWithdrawalManager.sol)

### Summary

`LRTWithdrawalManager.completeWithdrawal()` and `completeWithdrawalForUser()` carry the `whenNotPaused` modifier. Once a user's withdrawal has been fully processed through `unlockQueue()` — meaning rsETH has been burned, the asset amount has been finalized, and the withdrawal delay has elapsed — the user's funds sit in the contract ready to be claimed. A pause at this point blocks the final transfer step with no protocol-safety benefit, temporarily freezing funds that are already owed to users.

### Finding Description

The withdrawal lifecycle in `LRTWithdrawalManager` has three distinct phases:

1. **`initiateWithdrawal`** — user locks rsETH; `expectedAssetAmount` is recorded; `withdrawalStartBlock` is set.
2. **`unlockQueue`** — operator burns rsETH, finalizes the payout via `_calculatePayoutAmount`, and moves the request to the "unlocked" state.
3. **`completeWithdrawal`** — user claims the finalized asset amount.

After step 2, the user's rsETH is already burned and the asset amount is irrevocably committed. The only remaining action is the transfer. Yet `completeWithdrawal` (line 183) and `completeWithdrawalForUser` (line 199) both carry `whenNotPaused`:

```solidity
function completeWithdrawal(address asset, string calldata referralId)
    external nonReentrant whenNotPaused { ... }

function completeWithdrawalForUser(address asset, address user, string calldata referralId)
    external nonReentrant whenNotPaused onlyLRTOperator { ... }
```

If the `PAUSER_ROLE` pauses `LRTWithdrawalManager` after `unlockQueue` has run but before users call `completeWithdrawal`, those users cannot retrieve their assets. The `PAUSER_ROLE` is explicitly designed to act quickly on false-positive signals (e.g., a Chainlink feed anomaly), making accidental or malicious pauses a realistic scenario.

This is structurally identical to the Reserve Protocol finding: a pause modifier on a settlement/completion function blocks users from receiving assets that are already owed to them, with no corresponding protocol-safety benefit at that stage.

### Impact Explanation

Users whose withdrawals have been unlocked (rsETH already burned, payout finalized) cannot call `completeWithdrawal`. Their ETH or LST assets are locked inside `LRTWithdrawalManager` for the duration of the pause. This constitutes **temporary freezing of funds** — a Medium-severity impact per the allowed scope.

### Likelihood Explanation

The `PAUSER_ROLE` can pause `LRTWithdrawalManager` at any time via `pause()`. The protocol's own documentation acknowledges false-positive pauses are acceptable. A single PAUSER action — whether accidental or malicious — is sufficient to trigger the freeze. No user error or complex precondition is required beyond the normal withdrawal lifecycle reaching the unlocked state.

### Recommendation

Remove `whenNotPaused` from `completeWithdrawal` and `completeWithdrawalForUser`. Once a withdrawal is unlocked (rsETH burned, payout finalized), the settlement step carries no risk that a pause is designed to mitigate. Allowing completion while paused is safe and mirrors the Reserve Protocol's confirmed mitigation (PR #876) which removed `notTradingPausedOrFrozen` from `settleTrade` for the same reason.

### Proof of Concept

1. Alice calls `initiateWithdrawal(ETH, 1e18 rsETH, ...)`. Her rsETH is transferred to the contract; `expectedAssetAmount` = 1.05 ETH is recorded.
2. Eight days pass. The operator calls `unlockQueue(ETH, ...)`. Alice's rsETH is burned; her request is moved to the unlocked queue with `expectedAssetAmount` = 1.05 ETH finalized. `unlockedWithdrawalsCount[ETH]` increments.
3. A PAUSER calls `LRTWithdrawalManager.pause()` (e.g., responding to a false-positive oracle alert).
4. Alice calls `completeWithdrawal(ETH, ...)`. The call reverts at the `whenNotPaused` modifier — her 1.05 ETH remains locked in the contract even though her rsETH is already burned and the payout is fully committed.
5. The pause is lifted hours or days later. Alice can now claim, but her funds were frozen with no protocol-safety justification.
