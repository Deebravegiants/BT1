### Title
Chainlink Oracle Deprecation Causes Protocol-Wide Temporary Fund Freeze - (File: contracts/oracles/ChainlinkPriceOracle.sol)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `priceFeed.latestRoundData()` with no protection against oracle deprecation. When Chainlink deprecates an oracle by setting its aggregator address to `address(0)`, calls to `latestRoundData()` revert with empty return data (Solidity's ABI decoder fails on a call to a non-contract address). Because `LRTOracle`, `LRTDepositPool`, and `LRTWithdrawalManager` all call `getAssetPrice()` directly — with no try-catch or deprecation guard — a single deprecated Chainlink feed freezes deposits, withdrawal initiation, and withdrawal queue processing across the entire protocol until governance deploys and activates a replacement oracle.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` performs a bare external call to `latestRoundData()`:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol:49-55
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
``` [1](#0-0) 

When Chainlink deprecates a feed it sets the underlying aggregator to `address(0)`. A low-level call to `address(0)` in Solidity succeeds at the EVM level (returns empty bytes) but the ABI decoder then reverts with empty error data when trying to decode the return values. There is no deprecation check, no try-catch, and no staleness guard that would distinguish this from a normal revert.

`LRTOracle.getAssetPrice()` delegates directly to the price fetcher:

```solidity
// contracts/LRTOracle.sol:156-158
function getAssetPrice(address asset) public view onlySupportedOracle(asset) returns (uint256) {
    return IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset);
}
``` [2](#0-1) 

`_getTotalEthInProtocol()` iterates over every supported asset and calls `getAssetPrice()` for each one. A single deprecated feed causes the entire loop to revert:

```solidity
// contracts/LRTOracle.sol:336-344
for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
    address asset = supportedAssets[assetIdx];
    uint256 assetER = getAssetPrice(asset);
    uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
    totalETHInProtocol += totalAssetAmt.mulWad(assetER);
    ...
}
``` [3](#0-2) 

This propagates upward to `updateRSETHPrice()` / `_updateRsETHPrice()`.

`LRTDepositPool.getRsETHAmountToMint()` calls both `lrtOracle.getAssetPrice(asset)` and `lrtOracle.rsETHPrice()`:

```solidity
// contracts/LRTDepositPool.sol:519-521
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [4](#0-3) 

This is called inside `_beforeDeposit()`, which is called by both `depositETH()` and `depositAsset()`, blocking all user deposits.

`LRTWithdrawalManager.getExpectedAssetAmount()` has the same dependency:

```solidity
// contracts/LRTWithdrawalManager.sol:592-594
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
``` [5](#0-4) 

This is called inside `initiateWithdrawal()`, blocking withdrawal initiation. `unlockQueue()` also calls `lrtOracle.getAssetPrice(asset)` via `_createUnlockParams()`:

```solidity
// contracts/LRTWithdrawalManager.sol:847-849
return UnlockParams({
    rsETHPrice: lrtOracle.rsETHPrice(),
    assetPrice: lrtOracle.getAssetPrice(asset),
``` [6](#0-5) 

This blocks operators from unlocking the withdrawal queue, preventing users whose withdrawals are already queued from progressing to `completeWithdrawal()`.

---

### Impact Explanation

**Medium — Temporary freezing of funds.**

A deprecated Chainlink oracle causes:
- All `depositETH()` / `depositAsset()` calls to revert → user deposits frozen.
- All `initiateWithdrawal()` calls to revert → new withdrawal requests blocked.
- All `unlockQueue()` calls to revert → existing queued withdrawals cannot be unlocked, preventing users from reaching `completeWithdrawal()`.
- `updateRSETHPrice()` to revert → rsETH price staleness accumulates.

The freeze persists until governance deploys a replacement oracle and calls `updatePriceFeedFor()` / `updatePriceOracleFor()`. Given that governance actions may be subject to a timelock, the freeze window can be material.

---

### Likelihood Explanation

Chainlink has deprecated oracles in the past by setting the aggregator to `address(0)` (documented in the original Reserve Protocol audit and confirmed by on-chain evidence on Polygon). The LRT-rsETH protocol supports multiple LST assets (stETH, ETHx, sfrxETH, rETH, swETH), each with its own Chainlink feed. Any one of these feeds being deprecated triggers the freeze. The probability increases with the number of supported assets and the protocol's operational lifetime.

---

### Recommendation

1. **Add a deprecation check in `ChainlinkPriceOracle.getAssetPrice()`**: Before calling `latestRoundData()`, verify that `address(priceFeed)` has contract code (e.g., `address(priceFeed).code.length > 0`). Revert with a descriptive error rather than propagating empty revert data.

2. **Wrap `latestRoundData()` in a try-catch**: Catch both empty and non-empty revert data and surface a meaningful error, preventing silent propagation.

3. **Consider a circuit-breaker in `_getTotalEthInProtocol()`**: If a single asset's oracle fails, skip that asset and emit an event rather than reverting the entire price update.

---

### Proof of Concept

1. Chainlink announces deprecation of the ETHx/ETH feed and sets its aggregator to `address(0)`.
2. Any call to `ChainlinkPriceOracle.getAssetPrice(ETHx)` reverts with empty data (ABI decode failure on a call to `address(0)`).
3. `LRTOracle._getTotalEthInProtocol()` reverts → `updateRSETHPrice()` reverts.
4. A user calls `LRTDepositPool.depositAsset(ETHx, ...)` → `_beforeDeposit()` → `getRsETHAmountToMint()` → `lrtOracle.getAssetPrice(ETHx)` → reverts. **Deposits frozen.**
5. A user calls `LRTWithdrawalManager.initiateWithdrawal(ETHx, ...)` → `getExpectedAssetAmount()` → `lrtOracle.getAssetPrice(ETHx)` → reverts. **Withdrawal initiation frozen.**
6. An operator calls `unlockQueue(ETHx, ...)` → `_createUnlockParams()` → `lrtOracle.getAssetPrice(ETHx)` → reverts. **Withdrawal queue processing frozen.**
7. Governance must pass a proposal to update the oracle. Until the new oracle is live, all above operations remain frozen.
