### Title
Missing `cancelWithdrawal()` Causes Temporary rsETH Freeze in `LRTWithdrawalManager` - (File: contracts/LRTWithdrawalManager.sol)

### Summary
`LRTWithdrawalManager.initiateWithdrawal()` irrevocably locks user rsETH in the contract. There is no corresponding `cancelWithdrawal()` function. If the unstaking vault holds insufficient assets for the requested asset (a normal protocol state during EigenLayer withdrawal delays), `unlockQueue()` reverts and the user has no user-controlled path to reclaim their rsETH.

### Finding Description
When a user calls `initiateWithdrawal()`, their rsETH is pulled into `LRTWithdrawalManager` and a withdrawal request is recorded:

```solidity
IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);
// ...
assetsCommitted[asset] += expectedAssetAmount;
_addUserWithdrawalRequest(asset, rsETHUnstaked, expectedAssetAmount);
```

The rsETH remains locked until an operator calls `unlockQueue()`, which internally calls `unstakingVault.balanceOf(asset)` and reverts if the vault holds no assets for that asset:

```solidity
if (params.totalAvailableAssets == 0) revert AmountMustBeGreaterThanZero();
```

The unstaking vault is only funded after EigenLayer queued withdrawals complete and are claimed by `NodeDelegator`. During this window — which can span days to weeks — the operator cannot call `unlockQueue()` even in good faith. The user cannot call `completeWithdrawal()` because the request is still locked (`usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]` check). No `cancelWithdrawal()` function exists anywhere in the contract to return the rsETH to the user.

The contract provides `instantWithdrawal()` as an alternative, but it is gated by `onlyInstantWithdrawalAllowed(asset)` and requires the unstaking vault to have assets available for instant withdrawal — the same condition that is unavailable during EigenLayer withdrawal delays.

### Impact Explanation
**Medium — Temporary freezing of funds.** A user's rsETH is locked in `LRTWithdrawalManager` with no user-controlled exit. The user cannot cancel the request, cannot complete the withdrawal (not yet unlocked), and cannot use instant withdrawal (vault empty). The rsETH remains frozen until EigenLayer withdrawal completes, assets flow into the unstaking vault, the operator calls `unlockQueue()`, and the block delay passes. This can span days to weeks.

### Likelihood Explanation
**Medium.** EigenLayer queued withdrawals have a mandatory delay (currently ~7 days). Any user who calls `initiateWithdrawal()` while the unstaking vault is empty — a routine protocol state — will have their rsETH frozen for the duration. This is a predictable, recurring condition, not an edge case.

### Recommendation
Add a `cancelWithdrawal(address asset)` function that:
1. Verifies the request is still in the locked (not yet unlocked) queue.
2. Removes the request from `userAssociatedNonces` and `withdrawalRequests`.
3. Decrements `assetsCommitted[asset]` by `expectedAssetAmount`.
4. Returns the locked rsETH to the user.

### Proof of Concept
1. User calls `initiateWithdrawal(stETH, 1e18, "")` → 1e18 rsETH transferred to `LRTWithdrawalManager`; `assetsCommitted[stETH]` increases.
2. Unstaking vault holds 0 stETH (EigenLayer withdrawal pending).
3. Operator calls `unlockQueue(stETH, ...)` → reverts at `if (params.totalAvailableAssets == 0) revert AmountMustBeGreaterThanZero()`.
4. User calls `completeWithdrawal(stETH, "")` → reverts at `if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked()`.
5. User calls `instantWithdrawal(stETH, 1e18, "")` → reverts at `onlyInstantWithdrawalAllowed(stETH)` (disabled) or `CantInstantWithdrawMoreThanAvailable` (vault empty).
6. No `cancelWithdrawal()` exists. User's 1e18 rsETH is frozen until EigenLayer withdrawal completes. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
