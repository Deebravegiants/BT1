### Title
Integer Truncation in `rewardPerToken()` Causes Zero Reward Accrual When `rewardRate * elapsed * 1e18 < totalKernelStaked` — (`contracts/KERNEL/KernelDepositPool.sol`)

---

### Summary

`rewardPerToken()` uses integer division that truncates to zero when `rewardRate * elapsed * DECIMAL_PRECISION < totalKernelStaked`. Under a configuration with a small `rewardRate` and a large `totalKernelStaked`, every call to `updateReward` snapshots a zero increment, causing stakers to accrue no rewards for the entire active reward period even though `rewardRate > 0`.

---

### Finding Description

`rewardPerToken()` computes the per-token accumulator as:

```solidity
rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
    / totalKernelStaked;
``` [1](#0-0) 

The numerator is `rewardRate * elapsed * 1e18`. When this product is strictly less than `totalKernelStaked`, Solidity's integer division truncates the entire increment to `0`. The `updateReward` modifier then writes this unchanged value back to `rewardPerTokenStored`:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();   // stores 0-increment
    updatedAt = lastTimeRewardApplicable();    // advances the clock
    ...
}
``` [2](#0-1) 

Crucially, `updatedAt` is always advanced even when the increment is zero, so the "missed" time is not carried forward — it is permanently discarded.

`notifyRewardAmount` computes `rewardRate = receivedAmount / duration` (integer division) and only guards against `rewardRate == 0`:

```solidity
rewardRate = receivedAmount / duration;
...
if (rewardRate == 0) revert RewardRateZero();
``` [3](#0-2) 

There is no guard ensuring `rewardRate * 1 * 1e18 >= totalKernelStaked` (the minimum condition for a single-second increment to be non-zero). A `rewardRate` of `31` passes the guard but is completely ineffective against a `totalKernelStaked` of `1e30`.

---

### Impact Explanation

Stakers earn zero rewards for the entire active reward period. The reward tokens deposited by the admin remain locked in the contract with no mechanism to recover or redistribute them once the period ends (`finishAt` passes). This directly violates the invariant that stakers must accrue rewards continuously proportional to their stake and elapsed time.

Impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

---

### Likelihood Explanation

The condition is reachable through two realistic paths:

1. **Admin misconfiguration:** Admin calls `notifyRewardAmount` with a small absolute amount (e.g., `1e6` wei over `365 days` → `rewardRate ≈ 31`). Any `totalKernelStaked > 31 * 1e18 ≈ 31 KERNEL` causes per-second truncation to zero. The period is 1,015 years before a single unit accrues.

2. **Post-notification whale stake:** Admin notifies with a reasonable amount, then a large staker (or the protocol itself via `stakeFor`) pushes `totalKernelStaked` past the truncation threshold. `stake()` and `stakeFor()` are permissionless (or role-gated but not admin-only), so this can happen after the reward period starts.

The contract's own NatSpec acknowledges the Synthetix-derived design but does not document or guard against this truncation edge case. [4](#0-3) 

---

### Recommendation

Add a minimum-accrual invariant check in `notifyRewardAmount` after computing `rewardRate`:

```solidity
// Ensure at least 1 unit accrues per second given current stake
if (rewardRate * DECIMAL_PRECISION < totalKernelStaked) revert RewardRateTooLow();
```

Alternatively, enforce a minimum reward amount relative to `totalKernelStaked` and `duration`:

```solidity
uint256 minAmount = (totalKernelStaked * duration) / DECIMAL_PRECISION;
if (receivedAmount < minAmount) revert InsufficientRewardAmount();
```

This ensures `rewardRate * 1 * 1e18 >= totalKernelStaked` at the time of notification.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Foundry test — run with: forge test --match-test test_rewardTruncation -vvv

import "forge-std/Test.sol";
import {KernelDepositPool} from "contracts/KERNEL/KernelDepositPool.sol";
import {ERC20Mock} from "openzeppelin/mocks/ERC20Mock.sol";

contract RewardTruncationTest is Test {
    KernelDepositPool pool;
    ERC20Mock kernelToken;
    ERC20Mock rewardToken;

    address admin   = address(0xA);
    address staker  = address(0xB);

    function setUp() public {
        kernelToken = new ERC20Mock();
        rewardToken = new ERC20Mock();

        pool = new KernelDepositPool();
        pool.initialize(admin, address(kernelToken), address(rewardToken));

        // Mint tokens
        kernelToken.mint(staker, 1e30);
        rewardToken.mint(admin,  1e6);   // tiny reward: 1e6 wei

        // Admin sets duration = 365 days
        vm.prank(admin);
        pool.setRewardsDuration(365 days);
    }

    function test_rewardTruncation() public {
        // Staker stakes 1e30 KERNEL
        vm.startPrank(staker);
        kernelToken.approve(address(pool), 1e30);
        pool.stake(1e30);
        vm.stopPrank();

        // Admin notifies 1e6 wei reward → rewardRate = 1e6 / 31_536_000 ≈ 31
        vm.startPrank(admin);
        rewardToken.approve(address(pool), 1e6);
        pool.notifyRewardAmount(1e6);
        vm.stopPrank();

        uint256 rptBefore = pool.rewardPerToken();

        // Advance 1000 blocks (~12 000 seconds)
        vm.warp(block.timestamp + 12_000);

        uint256 rptAfter = pool.rewardPerToken();

        // Increment = rewardRate(31) * 12000 * 1e18 / 1e30
        //           = 372_000 * 1e18 / 1e30 = 3.72e-10 → truncates to 0
        assertEq(rptAfter - rptBefore, 0, "rewardPerToken increment should be 0 due to truncation");

        // Staker earned nothing
        assertEq(pool.earned(staker), 0, "staker should have earned 0 rewards");
    }
}
```

The test demonstrates that after 12,000 seconds (≈1,000 blocks) with `rewardRate = 31` and `totalKernelStaked = 1e30`, `rewardPerToken()` returns an identical value and `earned()` returns `0`, confirming complete reward stalling despite an active reward period.
