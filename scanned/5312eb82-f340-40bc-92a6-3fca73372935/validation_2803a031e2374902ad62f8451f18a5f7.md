### Title
Missing Zero-Check on `rsETHAmount` in L2 Pool Deposit Functions Causes Silent Loss of User Funds - (File: contracts/pools/RSETHPoolNoWrapper.sol)

### Summary
The `viewSwapRsETHAmountAndFee` function in every L2 pool contract performs integer division that can silently round `rsETHAmount` down to zero. When this occurs, the deposit function takes the user's ETH or tokens but transfers/mints zero wrsETH or rsETH in return, with no revert.

### Finding Description
Every L2 pool contract (`RSETHPoolNoWrapper`, `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, and their predecessors) computes the output amount via the same pattern:

**ETH path:**
```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [1](#0-0) 

**Token path:**
```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [2](#0-1) 

Due to Solidity floor division, `rsETHAmount` rounds to zero whenever `amountAfterFee * tokenToETHRate < rsETHToETHrate`. Neither the view function nor the deposit function checks that the result is non-zero before proceeding.

The ETH deposit path: [3](#0-2) 

The token deposit path: [4](#0-3) 

In both cases:
1. The user's ETH or tokens are transferred into the pool.
2. `rsETH.safeTransfer(msg.sender, 0)` (or `wrsETH.mint(msg.sender, 0)`) executes successfully — ERC-20 allows zero-value transfers and mints.
3. No revert occurs; the user receives nothing.

The same root cause is present in `RSETHPoolV3`: [5](#0-4) 

And in `RSETHPoolV3ExternalBridge`: [6](#0-5) 

And in `RSETHPoolV3WithNativeChainBridge`: [7](#0-6) 

The only input guard present is `if (amount == 0) revert InvalidAmount()`, which checks the *input* amount, not the *output* rsETH amount. [8](#0-7) 

By contrast, the L1 `LRTDepositPool` is protected by a caller-supplied `minRSETHAmountExpected` slippage guard and a `minAmountToDeposit` floor: [9](#0-8) 

No equivalent protection exists in any L2 pool.

### Impact Explanation
A depositor who sends a non-zero amount of ETH or a supported token receives zero wrsETH/rsETH. Their assets are permanently absorbed into the pool's balance (eventually bridged to L1 and distributed to all rsETH holders), constituting a direct, irreversible loss of the depositor's principal. This satisfies **Critical — direct theft of user funds in motion**.

### Likelihood Explanation
For the ETH path, rounding to zero requires `amountAfterFee < rsETHToETHrate / 1e18`. Since `rsETHToETHrate` starts at `1e18` and grows, only 1-wei deposits are affected under normal conditions. For the token path, the threshold is `amountAfterFee < rsETHToETHrate / tokenToETHRate`. If a supported token has a significantly lower ETH value than rsETH (e.g., a token worth 0.5 ETH while rsETH is worth 1.05 ETH), deposits of up to 2 wei are silently zeroed. In practice the affected amounts are dust, making this **Low likelihood** under normal oracle conditions. However, because no oracle manipulation is required — the condition is purely arithmetic — any user can trigger it deterministically with a crafted deposit amount.

### Recommendation
Add an explicit zero-check on the computed output amount in every deposit function, reverting before any state change or token transfer:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
if (rsETHAmount == 0) revert InvalidAmount();
```

Apply the same guard to the token deposit overload. This mirrors the `minRSETHAmountExpected` protection already present in `LRTDepositPool._beforeDeposit`.

### Proof of Concept

**Setup:**
- `rsETHToETHrate = 1.05e18` (rsETH has accrued 5% yield)
- Supported token oracle returns `tokenToETHRate = 0.5e18` (token worth 0.5 ETH)
- `feeBps = 5` (0.05% fee)

**Steps:**
1. Alice calls `deposit(token, 2, "ref")` on `RSETHPoolNoWrapper` with `amount = 2 wei`.
2. `fee = 2 * 5 / 10_000 = 0`; `amountAfterFee = 2`.
3. `rsETHAmount = 2 * 0.5e18 / 1.05e18 = 0` (floor division).
4. `IERC20(token).safeTransferFrom(Alice, pool, 2)` — Alice's 2 wei of tokens are taken.
5. `rsETH.safeTransfer(Alice, 0)` — succeeds silently.
6. Alice receives 0 rsETH; her 2 wei are permanently lost to the pool.
7. No revert, no slippage protection, no event anomaly that would alert Alice before submission.
