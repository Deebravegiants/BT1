### Title
Publicly Callable `updateRSETHPrice()` With No Rate Limiting Enables Block-Stuffing DoS via Nested External-Call Loop - (File: contracts/LRTOracle.sol)

---

### Summary

`LRTOracle.updateRSETHPrice()` is an unrestricted public function with no cooldown or rate-limiting mechanism. Each invocation triggers a nested loop that makes O(N × M × 3) external calls (N supported assets × M node delegators × 3 calls per NDC). An attacker can call this function repeatedly within consecutive blocks to fill block gas, temporarily preventing legitimate users from depositing, withdrawing, or otherwise interacting with the protocol.

---

### Finding Description

`updateRSETHPrice()` carries no access control and no minimum-interval guard: [1](#0-0) 

It delegates to `_updateRsETHPrice()`, which calls `_getTotalEthInProtocol()`: [2](#0-1) 

For every supported asset the inner call resolves to `LRTDepositPool.getAssetDistributionData()`, which itself iterates over every node delegator and fires three external calls per NDC: [3](#0-2) 

The three per-NDC calls are:
- `IERC20(asset).balanceOf(NDC)` — cold SLOAD + external call
- `INodeDelegator(NDC).getAssetBalance(asset)` — external call that itself queries an EigenLayer strategy
- `INodeDelegator(NDC).getAssetUnstaking(asset)` — external call into EigenLayer withdrawal accounting

With a realistic deployment of 5 supported assets and 10 node delegators the function issues ≥155 external calls per invocation, consuming on the order of 1–3 M gas. Because the block gas limit is ~30 M, an attacker can pack 10–30 invocations per block, saturating the block and starving all other pending transactions (deposits, withdrawals, oracle updates by legitimate callers).

There is no rate-limiting mechanism analogous to what the external report identifies as the root amplification factor: nothing prevents an attacker from submitting hundreds of calls per minute, exactly mirroring the PoC of 300+ req/min described in the report.

---

### Impact Explanation

**Medium — Temporary freezing of funds.**

During a sustained block-stuffing campaign, `depositETH`, `depositAsset`, `initiateWithdrawal`, and `completeWithdrawal` are all effectively unreachable because the attacker's transactions consume the available block gas. Users cannot move funds in or out of the protocol for the duration of the attack. No funds are permanently lost, but the protocol is rendered non-functional for an economically bounded window.

---

### Likelihood Explanation

**Medium.**

The attacker requires no special role, no oracle access, and no protocol-internal state. The only cost is gas. At 10 gwei and ~1 M gas per call, filling one block costs roughly 0.3 ETH; sustaining the attack for one hour costs on the order of 90 ETH — expensive but within reach of a motivated adversary targeting a protocol holding significant TVL. No rate-limiting or access control exists to raise this cost.

---

### Recommendation

1. **Add a minimum update interval**: record `lastPriceUpdateTimestamp` and revert if `block.timestamp < lastPriceUpdateTimestamp + MIN_UPDATE_INTERVAL` for unprivileged callers.
2. **Restrict public access**: gate `updateRSETHPrice()` to `MANAGER_ROLE` or a dedicated keeper role; expose a separate, rate-limited public wrapper if permissionless updates are required.
3. **Cache intermediate results**: store per-asset totals and invalidate them lazily to reduce the per-call external-call count.

---

### Proof of Concept

```solidity
// Attacker contract — no special privileges required
contract Attacker {
    ILRTOracle oracle;

    constructor(address _oracle) { oracle = ILRTOracle(_oracle); }

    // Call this in a loop (e.g., via a script sending 300+ txs/min)
    // Each call consumes O(N_assets × N_NDCs × 3) external calls (~1–3 M gas)
    function spam() external {
        oracle.updateRSETHPrice(); // public, no access control
    }
}
```

**Attack flow:**

1. Attacker deploys `Attacker` and calls `spam()` in rapid succession across consecutive blocks.
2. Each `updateRSETHPrice()` invocation executes the nested loop in `_getTotalEthInProtocol()` (LRTOracle.sol:336–348) and the inner loop in `getAssetDistributionData()` (LRTDepositPool.sol:447–456), consuming 1–3 M gas.
3. With 10–30 such calls per block the attacker saturates the 30 M block gas limit.
4. Legitimate `depositETH`, `depositAsset`, `initiateWithdrawal`, and `completeWithdrawal` transactions are excluded from blocks for the duration of the attack, constituting a temporary freeze of user funds.
