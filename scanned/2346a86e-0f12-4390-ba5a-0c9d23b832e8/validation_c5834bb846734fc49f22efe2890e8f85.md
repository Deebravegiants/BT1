### Title
Unprivileged `updateRate()` Propagates Stale rsETH Price to L2, Enabling Theft of Unclaimed Yield — (`contracts/cross-chain/RSETHRateProvider.sol`)

---

### Summary

`CrossChainRateProvider.updateRate()` is callable by any address with no access control. It reads `LRTOracle.rsETHPrice`, a **stored state variable** that is only updated when `updateRSETHPrice()` is explicitly called. An attacker can call `updateRate()` in the window after EigenLayer rewards accrue but before the oracle is refreshed, broadcasting a stale (below-true) rate to L2. L2 depositors then receive excess `wrsETH` at the pre-yield rate, diluting existing rsETH holders' accrued staking rewards.

---

### Finding Description

**Step 1 — No access control on `updateRate()`**

`CrossChainRateProvider.updateRate()` is `external payable nonReentrant` with no role check: [1](#0-0) 

Any EOA or contract can call it at any time.

**Step 2 — `getLatestRate()` reads a stored, potentially stale value**

`RSETHRateProvider.getLatestRate()` does not compute a fresh price; it reads the `rsETHPrice` storage slot of `LRTOracle`: [2](#0-1) 

`LRTOracle.rsETHPrice` is a plain `uint256` state variable: [3](#0-2) 

It is only updated inside `_updateRsETHPrice()`, which is triggered by `updateRSETHPrice()` (public) or `updateRSETHPriceAsManager()` (manager-only). Between EigenLayer reward epochs and the next oracle refresh call, `rsETHPrice` is stale — it reflects the pre-reward exchange rate.

**Step 3 — Stale rate is forwarded verbatim to L2**

`updateRate()` encodes `latestRate` and sends it via LayerZero: [4](#0-3) 

`CrossChainRateReceiver.lzReceive()` stores whatever value arrives, with no freshness or sanity check: [5](#0-4) 

**Step 4 — L2 pool mints wrsETH using the stale rate**

`RSETHPoolV3.viewSwapRsETHAmountAndFee()` divides the deposit amount by the oracle rate: [6](#0-5) 

`getRate()` calls `IOracle(rsETHOracle).getRate()`, which resolves to `CrossChainRateReceiver.getRate()` — the just-poisoned value: [7](#0-6) 

A stale rate that is **lower** than the true post-yield rate causes the division to produce a **larger** `rsETHAmount`, minting excess `wrsETH`.

---

### Impact Explanation

Suppose the true post-yield rsETH/ETH rate is 1.05 but the stored `rsETHPrice` is still 1.00 (stale). An attacker who calls `updateRate()` before the oracle is refreshed locks in the 1.00 rate on L2. A depositor of 1 ETH then receives `1/1.00 = 1.0 wrsETH` instead of the correct `1/1.05 ≈ 0.952 wrsETH` — a ~5% excess. The extra `wrsETH` is backed by nothing; when the rate is eventually corrected to 1.05, the attacker's position is worth more than deposited, at the expense of existing rsETH holders whose accrued yield is diluted.

Impact: **High — Theft of unclaimed yield.**

---

### Likelihood Explanation

- EigenLayer distributes rewards on a regular cadence; the stale window is predictable and observable on-chain.
- `updateRate()` requires only ETH for LayerZero gas — no special role, no token approval.
- The attacker can monitor mempool/events for reward distribution and front-run the oracle keeper's `updateRSETHPrice()` call.
- No existing guard (reentrancy lock, pause, daily mint cap) prevents this specific ordering attack.

---

### Recommendation

1. **Add access control to `updateRate()`** — restrict it to a trusted keeper/operator role, or at minimum require that `LRTOracle.rsETHPrice` was updated within a freshness window (e.g., within the last N seconds).
2. **Compute a fresh price before broadcasting** — have `RSETHRateProvider.getLatestRate()` call `LRTOracle.updateRSETHPrice()` (or an equivalent view that computes the live price from current TVL) rather than reading the cached storage slot.
3. **Add a staleness check in `RSETHPoolV3`** — reject deposits if `lastUpdated` on the oracle is older than an acceptable threshold.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Fork test (Foundry) — run against a mainnet fork
// Precondition: EigenLayer rewards have accrued; LRTOracle.rsETHPrice has NOT been refreshed yet.

contract StaleRatePoC is Test {
    RSETHRateProvider provider = RSETHRateProvider(<deployed_address>);
    RSETHPoolV3 pool = RSETHPoolV3(<l2_pool_address>);
    LRTOracle oracle = LRTOracle(<oracle_address>);

    function testStaleRateExploit() external {
        // 1. Record the stale stored price
        uint256 staleRate = oracle.rsETHPrice();

        // 2. Compute the true current price (without writing it)
        //    (simulate by calling updateRSETHPrice in a staticcall or reading TVL directly)
        uint256 trueRate = computeTrueRate(); // e.g., 1.05e18

        assert(staleRate < trueRate); // confirms staleness

        // 3. Attacker broadcasts the stale rate to L2 — no role required
        provider.updateRate{value: 0.01 ether}();
        // LayerZero delivers stale rate to CrossChainRateReceiver on L2

        // 4. Attacker deposits 1 ETH on L2 at the stale rate
        uint256 wrsETHReceived = pool.deposit{value: 1 ether}("ref");

        // 5. Assert excess minting
        uint256 fairAmount = 1e18 * 1e18 / trueRate;   // correct wrsETH
        uint256 staleAmount = 1e18 * 1e18 / staleRate; // actual wrsETH minted
        assertGt(wrsETHReceived, fairAmount, "Excess wrsETH minted at stale rate");
    }
}
```

The test demonstrates that calling `updateRate()` as an unprivileged attacker before `updateRSETHPrice()` is called results in L2 depositors receiving more `wrsETH` than the true L1 backing justifies, constituting theft of unclaimed yield from existing rsETH holders.
