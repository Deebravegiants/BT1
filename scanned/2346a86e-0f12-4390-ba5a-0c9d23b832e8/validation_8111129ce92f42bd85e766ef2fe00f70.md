### Title
Unbounded Withdrawal Queue Growth via Dust Requests Causes Temporary Freezing of Legitimate User Withdrawals - (File: contracts/LRTWithdrawalManager.sol)

---

### Summary

`LRTWithdrawalManager.initiateWithdrawal` has no enforced minimum withdrawal amount by default. An unprivileged rsETH holder can flood the global, FIFO withdrawal queue with dust requests (1 wei rsETH each). Because `_unlockWithdrawalRequests` iterates the queue in strict FIFO order, legitimate users' requests are permanently blocked behind the attacker's dust entries until the operator processes every one of them, causing a temporary (and renewable) freezing of legitimate user funds.

---

### Finding Description

`minRsEthAmountToWithdraw[asset]` is a mapping that defaults to `0` for every asset until an admin explicitly sets it. [1](#0-0) 

The guard in `initiateWithdrawal` is:

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
``` [2](#0-1) 

When `minRsEthAmountToWithdraw[asset] == 0` (the default), the condition `rsETHUnstaked < 0` is always `false` for a `uint256`, so the only effective check is `rsETHUnstaked == 0`. Any amount ≥ 1 wei is accepted.

Each accepted call pushes a new nonce into the global queue and increments `nextUnusedNonce[asset]`: [3](#0-2) 

The operator's `unlockQueue` → `_unlockWithdrawalRequests` processes requests in strict FIFO order via a `while` loop from `nextLockedNonce` to `firstExcludedIndex`: [4](#0-3) 

There is no per-user cap on the number of pending requests in `LRTWithdrawalManager` (unlike `KernelDepositPool` which has `maxNumberOfWithdrawalsPerUser`). The `userAssociatedNonces` deque is bounded only by `uint128` overflow (~3.4 × 10³⁸ entries). [5](#0-4) 

---

### Impact Explanation

An attacker who submits N dust requests ahead of legitimate users forces the operator to process all N requests before any legitimate withdrawal can be unlocked. The operator can batch via `firstExcludedIndex`, but each batch only advances `nextLockedNonce` by the batch size. With 100,000 dust requests and a batch size of 5,000, the operator needs 20 `unlockQueue` transactions before a single legitimate user's request is reached. During this entire period, legitimate users' rsETH is locked in the contract and cannot be redeemed — a **temporary freezing of funds**.

The attacker can renew the attack after each operator sweep by re-depositing and re-queuing, making the freeze continuous.

**Impact: Medium — Temporary freezing of funds.**

---

### Likelihood Explanation

- The attacker only needs rsETH (obtainable by depositing ETH or any supported LST via `LRTDepositPool.depositETH`/`depositAsset`).
- `minRsEthAmountToWithdraw` is `0` by default and requires an explicit admin action to set; many deployments may never set it.
- The attacker's rsETH is eventually returned as the underlying asset after the withdrawal delay, so the net cost is only gas.
- On Ethereum mainnet at 10 gwei, creating 10,000 dust requests costs roughly 10 ETH in gas — economically feasible for a motivated attacker seeking to grief the protocol.

**Likelihood: Medium.**

---

### Recommendation

1. **Enforce a non-zero minimum at initialization.** Set a meaningful `minRsEthAmountToWithdraw` for every supported asset during `initialize` (e.g., 0.001 rsETH). Do not rely on admin post-deployment configuration.
2. **Add a per-user pending request cap** in `LRTWithdrawalManager`, analogous to `maxNumberOfWithdrawalsPerUser` in `KernelDepositPool`. [6](#0-5) 

3. **Validate `expectedAssetAmount > 0`** after the oracle calculation inside `initiateWithdrawal` to reject requests that round down to zero asset value.

---

### Proof of Concept

```solidity
// Attacker script (pseudo-code)
address asset = ETH_TOKEN;
uint256 dustAmount = 1; // 1 wei rsETH

// 1. Deposit ETH to get rsETH
lrtDepositPool.depositETH{value: 1 ether}(0, "");

// 2. Approve withdrawal manager
rsETH.approve(address(withdrawalManager), type(uint256).max);

// 3. Flood the queue with dust requests
for (uint i = 0; i < 100_000; i++) {
    withdrawalManager.initiateWithdrawal(asset, dustAmount, "");
}

// Result: nextUnusedNonce[asset] is now 100_000 ahead of nextLockedNonce.
// Any legitimate user who calls initiateWithdrawal after this point
// gets a nonce >= 100_000. The operator must call unlockQueue ~20 times
// (at 5,000 requests/batch) before a single legitimate request is unlocked.
// During this window, all legitimate users' rsETH is frozen in the contract.
```

The attacker's rsETH is returned as the underlying asset after the withdrawal delay, making the net cost only gas. The attack can be repeated continuously to maintain the freeze.
