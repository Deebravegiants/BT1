### Title
No User-Initiated Cancellation Mechanism for Withdrawal Requests Leaves rsETH Permanently Locked - (File: contracts/LRTWithdrawalManager.sol)

### Summary

`LRTWithdrawalManager` implements a two-step withdrawal flow where a user first calls `initiateWithdrawal()` (transferring rsETH to the contract) and then must wait for a privileged operator to call `unlockQueue()` before the user can call `completeWithdrawal()`. There is no `cancelWithdrawal()` function anywhere in the contract or its interface. If the operator never advances `nextLockedNonce[asset]` via `unlockQueue()`, the user's rsETH is permanently locked in the contract with no on-chain escape path.

### Finding Description

The withdrawal lifecycle in `LRTWithdrawalManager` is:

1. **User** calls `initiateWithdrawal(asset, rsETHUnstaked, referralId)` — rsETH is pulled from the user and held by the contract.
2. **Operator** (`ASSET_TRANSFER_ROLE` or `OPERATOR_ROLE`) calls `unlockQueue(asset, firstExcludedIndex, ...)` — this advances `nextLockedNonce[asset]`, marking requests as claimable.
3. **User** calls `completeWithdrawal(asset, referralId)` — assets are transferred to the user.

Step 1 is irreversible from the user's perspective:

```solidity
// contracts/LRTWithdrawalManager.sol:166
IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);
```

Step 3 is gated on the operator having completed step 2:

```solidity
// contracts/LRTWithdrawalManager.sol:707
if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();
```

`nextLockedNonce[asset]` is only ever advanced inside `_unlockWithdrawalRequests()`, which is only reachable through `unlockQueue()`, a function restricted to `onlyAssetTransferOrOperatorRole`.

A search across the entire contract and its interface confirms there is no `cancelWithdrawal`, no user-callable reset, and no timeout after which a user may reclaim their rsETH unilaterally.

### Impact Explanation

Once `initiateWithdrawal()` is called, the user's rsETH is held by `LRTWithdrawalManager` indefinitely. If the operator role becomes unavailable (key loss, operational shutdown, regulatory freeze, or any other liveness failure), the user has no on-chain mechanism to recover their rsETH. The rsETH is burned from the user's wallet and sits in the contract, but the corresponding asset is never disbursed. This constitutes **temporary freezing of funds** at minimum and **permanent freezing of funds** if the operator is never restored — matching the allowed impact scope at Medium and Critical respectively.

### Likelihood Explanation

The operator is a trusted off-chain role. Liveness failures are realistic: key compromise, infrastructure outage, protocol wind-down, or regulatory action can all prevent `unlockQueue()` from being called. The user has no timeout-based escape hatch and no ability to self-rescue. Any user who has called `initiateWithdrawal()` is exposed for the full duration of the operator's unavailability.

### Recommendation

Add a user-callable `cancelWithdrawal(address asset)` function that:
- Is only callable after a defined timeout (e.g., `withdrawalDelayBlocks` × 2) has elapsed since `withdrawalStartBlock` without the request being unlocked.
- Removes the request from `userAssociatedNonces[asset][msg.sender]`.
- Decrements `assetsCommitted[asset]` by `request.expectedAssetAmount`.
- Returns `request.rsETHUnstaked` of rsETH back to the user.
- Deletes the `withdrawalRequests[requestId]` entry.

This mirrors the mitigation recommended in M-5: a timeout after which the initiating party can reset a stalled request.

### Proof of Concept

1. Alice calls `initiateWithdrawal(stETH, 10e18, "")`. Her 10 rsETH is transferred to `LRTWithdrawalManager`. [1](#0-0) 

2. Her request is stored with `withdrawalStartBlock = block.number` and nonce `N = nextUnusedNonce[stETH]`. `assetsCommitted[stETH]` is incremented. [2](#0-1) 

3. The operator never calls `unlockQueue()`. `nextLockedNonce[stETH]` remains at `N` (or below). [3](#0-2) 

4. Alice calls `completeWithdrawal(stETH, "")`. The call reverts at:
   ```solidity
   if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();
   ``` [4](#0-3) 

5. There is no `cancelWithdrawal()` function in the contract or its interface. Alice's 10 rsETH is permanently locked. [5](#0-4)
