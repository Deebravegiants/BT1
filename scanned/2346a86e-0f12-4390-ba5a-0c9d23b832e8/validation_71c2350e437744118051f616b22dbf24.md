### Title
Index Non-Monotonicity Allows Permanent Freezing of Unclaimed Yield — (`contracts/KERNEL/KernelMerkleDistributor.sol`)

---

### Summary

`KernelMerkleDistributor` tracks per-user claim state via `userClaims[account].lastClaimedIndex`. The `isClaimed` function returns `true` whenever `lastClaimedIndex >= index`. Because `setMerkleRoot` only enforces `index ≤ currentIndex` (not `index > lastClaimedIndex`), a root publisher can publish a new root that assigns a user an index lower than or equal to their previously recorded `lastClaimedIndex`. Every subsequent `claim` call for that user will revert with `AlreadyClaimed`, permanently freezing their unclaimed yield in all future roots.

---

### Finding Description

**`isClaimed` logic:** [1](#0-0) 

`isClaimed(index, account)` returns `true` if `userClaims[account].lastClaimedIndex >= index`. This is a monotonic "already seen" check — it considers any index ≤ the last claimed index as already claimed.

**`_processClaim` guard:** [2](#0-1) 

If `isClaimed` returns `true`, `_processClaim` reverts with `AlreadyClaimed` before any proof verification or amount calculation.

**`setMerkleRoot` index enforcement:** [3](#0-2) 

`setMerkleRoot` increments `currentIndex` by 1 per call and emits the new root. The only index constraint enforced at claim time is: [4](#0-3) 

`index > 0` and `index ≤ currentIndex`. There is **no check** that the index in the new root is strictly greater than the user's `lastClaimedIndex`.

**Concrete exploit path:**

| Step | Action | State |
|---|---|---|
| 1 | Owner calls `setMerkleRoot(root1)` | `currentIndex = 1` |
| 2 | Root-1 encodes user with `index=1, cumulativeAmount=100` | — |
| 3 | User calls `claim(1, user, 100, proof1)` | `lastClaimedIndex=1, cumulativeAmount=100` |
| 4 | Owner calls `setMerkleRoot(root2)` | `currentIndex = 2` |
| 5 | Root-2 encodes user with `index=1, cumulativeAmount=200` (off-by-one in off-chain script) | — |
| 6 | User calls `claim(1, user, 200, proof2)` | `isClaimed(1, user)` → `1 >= 1` → `true` → **`AlreadyClaimed` revert** |

The user's 100 additional tokens in root-2 are permanently inaccessible. Every future root that reuses or assigns any index ≤ 1 for this user will also be permanently frozen.

The `lastClaimedIndex` update: [5](#0-4) 

Once set, `lastClaimedIndex` can only increase (it is only written in `_processClaim` after a successful claim). There is no mechanism to reset it or override it.

---

### Impact Explanation

A user whose index in a newly published root is ≤ their `lastClaimedIndex` can never claim from that root or any future root that assigns them the same or a lower index. The tokens allocated to them in the merkle tree remain locked in the contract with no recovery path. This matches the scoped impact: **Medium — Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

`setMerkleRoot` is `onlyOwner`, so this requires the owner to publish a root with a non-monotonically-increasing user index. This is not "admin compromise" — it is a contract design flaw that provides no on-chain protection against an off-chain root generation bug (e.g., a script that reuses a prior epoch's index, reorders users, or fails to increment indices correctly). Off-chain tooling bugs in merkle root generation are a well-documented class of real-world errors. The contract offers no guard against this class of mistake, making the likelihood **low-to-medium** but the consequence irreversible.

---

### Recommendation

In `_processClaim`, replace the `isClaimed` check with a strict monotonicity requirement:

```solidity
// Instead of: if (isClaimed(index, account)) revert AlreadyClaimed();
if (index <= userClaims[account].lastClaimedIndex) revert AlreadyClaimed();
```

This ensures a new root can only be claimed if its index is strictly greater than the user's last claimed index, making the system robust against off-chain root generation mistakes that assign lower or equal indices.

Alternatively, enforce in `setMerkleRoot` that the root publisher must supply a minimum index floor per epoch, or use a separate bitmap-based claimed tracking keyed by `(merkleRootIndex, account)` rather than a global monotonic counter.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Foundry test sketch (local fork, no mainnet)
function test_permanentFreeze() public {
    // Setup: deploy distributor, fund with KERNEL tokens

    // Step 1: Publish root-1 with user at index=1, cumulative=100e18
    bytes32 leaf1 = keccak256(abi.encodePacked(uint256(1), user, uint256(100e18)));
    bytes32 root1 = leaf1; // single-leaf tree for simplicity
    vm.prank(owner);
    distributor.setMerkleRoot(root1);
    // currentIndex == 1

    // Step 2: User claims from root-1
    vm.prank(user);
    distributor.claim(1, user, 100e18, new bytes32[](0));
    // userClaims[user].lastClaimedIndex == 1

    // Step 3: Publish root-2 — off-chain bug reuses index=1 with higher cumulative
    bytes32 leaf2 = keccak256(abi.encodePacked(uint256(1), user, uint256(200e18)));
    bytes32 root2 = leaf2;
    vm.prank(owner);
    distributor.setMerkleRoot(root2);
    // currentIndex == 2, index=1 still valid (1 <= 2)

    // Step 4: User tries to claim additional 100e18 from root-2 — permanently frozen
    vm.prank(user);
    vm.expectRevert(IMerkleDistributor.AlreadyClaimed.selector);
    distributor.claim(1, user, 200e18, new bytes32[](0));
    // Reverts: isClaimed(1, user) => lastClaimedIndex(1) >= 1 => true
}
```
