### Title
`RSETHPriceFeed.latestRoundData()` Returns ETH/USD `updatedAt` Without Reflecting rsETH Price Staleness â€” (File: contracts/oracles/RSETHPriceFeed.sol)

---

### Summary
`RSETHPriceFeed.latestRoundData()` and `getRoundData()` compose the `answer` from two independent sources â€” the ETH/USD Chainlink feed and `LRTOracle.rsETHPrice()` â€” but return `updatedAt` exclusively from the ETH/USD feed. The rsETH/ETH component's staleness is never reflected in the returned metadata, so any consumer performing a standard Chainlink staleness check will incorrectly conclude the rsETH/USD price is fresh even when `rsETHPrice` has not been updated for an extended period.

---

### Finding Description

`RSETHPriceFeed` is a Chainlink `AggregatorV3Interface`-compatible contract that synthesises an rsETH/USD price by multiplying the ETH/USD Chainlink answer by the rsETH/ETH rate stored in `LRTOracle`:

```solidity
// contracts/oracles/RSETHPriceFeed.sol  lines 63-70
function latestRoundData()
    external view
    returns (uint80 roundId, int256 answer, uint256 startedAt, uint256 updatedAt, uint80 answeredInRound)
{
    (roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.latestRoundData();
    answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
}
```

`updatedAt` is taken verbatim from the ETH/USD Chainlink round. `RS_ETH_ORACLE.rsETHPrice()` is a storage variable in `LRTOracle` that is only updated when `updateRSETHPrice()` is explicitly called:

```solidity
// contracts/LRTOracle.sol  lines 87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

`updateRSETHPrice()` is permissionless but not automatically triggered. There is no on-chain record of when `rsETHPrice` was last written, and `RSETHPriceFeed` never queries such a timestamp. The identical flaw exists in `getRoundData()` (line 58-60).

Two concrete incorrect-value scenarios mirror the original report:

1. **Before the first `updateRSETHPrice()` call** â€” `rsETHPrice` is `0` (Solidity default). `latestRoundData()` returns `answer = 0 * ethPrice / 1e18 = 0`, an entirely incorrect price, while `updatedAt` reflects a recent ETH/USD round, making the feed appear valid.

2. **After `rsETHPrice` becomes stale** â€” `updatedAt` continues to track the ETH/USD heartbeat (typically â‰¤ 1 hour old), so any consumer applying a standard staleness window (e.g., `require(updatedAt >= block.timestamp - 3600)`) will accept the answer as fresh even though the rsETH/ETH component may be hours or days old.

---

### Impact Explanation

`RSETHPriceFeed` is deployed by Kelp DAO as the canonical rsETH/USD price feed for integration with external lending markets (Aave-style protocols) that accept rsETH as collateral. Those protocols call `latestRoundData()` and gate collateral valuation and liquidation logic on the returned `answer` and `updatedAt`.

- **Stale-inflated rsETH price**: If `rsETHPrice` has not been updated after a slashing or depeg event, the feed continues to report a higher-than-actual rsETH/USD price. Borrowers can draw more debt than their rsETH collateral is worth. When the price eventually corrects, positions become undercollateralised and are liquidated â€” direct loss of rsETH holder funds.
- **Zero price on first use**: Before `updateRSETHPrice()` is ever called, `answer = 0`. Any protocol that does not separately guard against a zero answer will treat rsETH as worthless, freezing rsETH collateral or triggering mass liquidations.

Impact classification: **Medium â€” Temporary freezing of funds / contract fails to deliver promised returns**, escalating toward **Critical** (direct theft via forced liquidation) if a stale-inflated price is accepted by a live lending market.

---

### Likelihood Explanation

`updateRSETHPrice()` has no keeper, no Chainlink Automation job, and no on-chain incentive. Any network congestion, keeper outage, or protocol pause that delays the call for longer than the staleness window used by an integrating protocol is sufficient to trigger the issue. The zero-price case is reachable during any fresh deployment before the first price update. Likelihood is **Medium**.

---

### Recommendation

1. Track the timestamp of the last `rsETHPrice` update inside `LRTOracle` (e.g., `uint256 public rsETHPriceLastUpdated`), set it in `_updateRsETHPrice()`.
2. In `RSETHPriceFeed.latestRoundData()` and `getRoundData()`, return `updatedAt = min(ethFeedUpdatedAt, rsETHPriceLastUpdated)` so that any consumer's staleness check reflects the freshness of both components.
3. Revert (or return a sentinel) when `RS_ETH_ORACLE.rsETHPrice() == 0` to prevent the zero-price scenario from propagating silently.

---

### Proof of Concept

1. `LRTOracle` is deployed; `rsETHPrice = 0` (default). `RSETHPriceFeed` is pointed at it.
2. A lending protocol integrates `RSETHPriceFeed` with a 1-hour staleness guard.
3. Before `updateRSETHPrice()` is called, `latestRoundData()` returns `answer = 0`, `updatedAt = <recent ETH/USD timestamp>`. The staleness guard passes; the protocol sees rsETH worth $0 and liquidates all rsETH-collateralised positions.

**Stale-price variant:**

1. `rsETHPrice` is last updated at block T (rsETH/ETH = 1.05).
2. A slashing event reduces the true rate to 0.95, but `updateRSETHPrice()` is not called for 6 hours.
3. ETH/USD Chainlink updates normally; `updatedAt` from `latestRoundData()` is always < 1 hour old.
4. A borrower deposits rsETH, receives a loan sized to the stale 1.05 rate.
5. When `updateRSETHPrice()` is finally called, the position is undercollateralised and liquidated, with the borrower extracting excess value at the expense of the lending protocol's liquidity providers â€” reachable by any unprivileged depositor. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
