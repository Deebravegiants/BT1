### Title
Permanent Freezing of KERNEL Tokens in `KernelReceiver` Due to Missing Error Isolation in `batchStakeFor` and Absent Recovery Mechanism — (`contracts/KERNEL/KernelReceiver.sol`)

---

### Summary

`KernelReceiver.batchStakeFor` propagates any revert from `stakerGateway.stakeFor` without isolation, and the contract contains no token-rescue path. If even one user/amount pair is permanently unacceptable to the external `stakerGateway` (blacklisted address, per-user cap exceeded, zero-amount guard, etc.), the operator cannot stake tokens for that user through any path in the contract, and those KERNEL tokens are locked forever.

---

### Finding Description

`batchStakeFor` iterates over all `(users[i], amounts[i])` pairs and delegates to `_stakeFor`, which calls `stakerGateway.stakeFor` with no `try/catch`: [1](#0-0) 

```solidity
for (uint256 i = 0; i < users.length; ++i) {
    _stakeFor(users[i], amounts[i]);
}
```

Inside `_stakeFor`, the external call is bare — any revert bubbles up and rolls back the entire transaction: [2](#0-1) 

```solidity
stakerGateway.stakeFor(address(kernel), user, amount, "");
```

The operator can fall back to the single-entry `stakeFor(address user, uint256 amount)` to process non-problematic users one by one: [3](#0-2) 

However, for the specific user whose `stakerGateway.stakeFor` **always** reverts, both `batchStakeFor` and the single `stakeFor` will revert identically. The contract exposes **no rescue, sweep, or emergency-withdrawal function** — confirmed by the complete absence of any such path in the contract: [4](#0-3) 

Admin functions are limited to `pause`, `unpause`, and `setStakerGateway`. None of these allow moving stranded ERC-20 tokens out of the contract. Changing `stakerGateway` to a permissive mock would allow draining, but that is an admin-key action outside the trust model and does not constitute a user-accessible recovery path.

---

### Impact Explanation

KERNEL tokens bridged from Ethereum mainnet via LayerZero arrive in `KernelReceiver` and must be restaked through `stakerGateway`. If `stakerGateway` permanently rejects a user (blacklist, per-user cap, protocol pause for that asset, etc.), the corresponding KERNEL balance sits in `KernelReceiver` with no on-chain path to recover or redirect it. This satisfies **Critical — Permanent freezing of funds**.

---

### Likelihood Explanation

The Kernel Protocol's `stakerGateway` is an external, upgradeable contract. Realistic permanent-revert triggers include:

- A user address being blacklisted or sanctioned in the Kernel Protocol after their bridge transaction was already submitted on Ethereum.
- A per-user or per-vault deposit cap being reached between bridge initiation and LayerZero delivery.
- The Kernel Protocol deprecating or pausing the KERNEL staking vault.

These are not hypothetical: cross-chain flows have inherent latency (minutes to hours), during which external protocol state can change. The likelihood is **Medium** — not every deployment will hit this, but the window is real and the consequence is irreversible.

---

### Recommendation

1. **Add per-entry error isolation in `batchStakeFor`** using `try/catch` so a single failing entry does not block the rest:

```solidity
for (uint256 i = 0; i < users.length; ++i) {
    try stakerGateway.stakeFor(address(kernel), users[i], amounts[i], "") {
        emit KernelStakedFor(users[i], amounts[i]);
    } catch {
        emit KernelStakeForFailed(users[i], amounts[i]);
    }
}
```

2. **Add an admin-controlled token rescue function** so that tokens for permanently-rejected users can be returned to them or held in escrow:

```solidity
function rescueTokens(address token, address to, uint256 amount)
    external onlyRole(DEFAULT_ADMIN_ROLE) {
    IERC20(token).safeTransfer(to, amount);
}
```

3. **Track per-user pending balances** so that failed stakes can be retried or refunded individually rather than relying on off-chain accounting.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import "forge-std/Test.sol";
import {KernelReceiver} from "contracts/KERNEL/KernelReceiver.sol";
import {KERNEL} from "contracts/KERNEL/KERNEL.sol";

contract RevertingGateway {
    address public blacklisted;
    constructor(address _blacklisted) { blacklisted = _blacklisted; }
    function stakeFor(address, address receiver, uint256, string calldata) external {
        require(receiver != blacklisted, "blacklisted");
    }
}

contract KernelReceiverFreezeTest is Test {
    KernelReceiver receiver;
    KERNEL kernel;
    address admin  = address(0xA);
    address operator = address(0xB);
    address goodUser = address(0xC);
    address badUser  = address(0xD); // permanently blacklisted

    function setUp() public {
        vm.startPrank(admin);
        kernel = new KERNEL(admin);
        RevertingGateway gateway = new RevertingGateway(badUser);
        receiver = new KernelReceiver();
        receiver.initialize(admin, operator, address(kernel), address(gateway));
        kernel.transfer(address(receiver), 200 ether);
        vm.stopPrank();
    }

    function testBatchPermanentlyFrozen() public {
        address[] memory users   = new address[](2);
        uint256[] memory amounts = new uint256[](2);
        users[0] = goodUser; amounts[0] = 100 ether;
        users[1] = badUser;  amounts[1] = 100 ether;

        vm.prank(operator);
        vm.expectRevert("blacklisted");
        receiver.batchStakeFor(users, amounts);

        // goodUser's 100 KERNEL is also stuck — no recovery path
        assertEq(kernel.balanceOf(address(receiver)), 200 ether);

        // Single stakeFor for badUser also permanently reverts
        vm.prank(operator);
        vm.expectRevert("blacklisted");
        receiver.stakeFor(badUser, 100 ether);

        // No rescue function exists — tokens are frozen forever
    }
}
```

Running this test against unmodified `KernelReceiver.sol` will pass, confirming that:
- `batchStakeFor` reverts entirely on one bad entry.
- The single `stakeFor` also reverts for the same user.
- The contract balance remains non-zero with no on-chain recovery path.
