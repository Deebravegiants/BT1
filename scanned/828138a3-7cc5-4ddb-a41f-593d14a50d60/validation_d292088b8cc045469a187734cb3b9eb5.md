### Title
Permanent `stakedButUnverifiedNativeETH` Inflation When Validator Is Slashed Before Credential Verification — (`contracts/NodeDelegator.sol`)

---

### Summary

`stakedButUnverifiedNativeETH` is incremented by exactly 32 ETH on every `stake32Eth` call and is **only ever decremented** inside `verifyWithdrawalCredentials`. If a validator is slashed on the beacon chain and exits before `verifyWithdrawalCredentials` is called, the EigenPod will reject any future credential verification for that validator (exit epoch already set), leaving `stakedButUnverifiedNativeETH` permanently inflated. `getEffectivePodShares` then returns a value larger than the actual recoverable ETH, inflating the protocol's TVL and `rsETHPrice`.

---

### Finding Description

**Increment path — `stake32Eth`:** [1](#0-0) 

Every call adds exactly 32 ETH to the counter and sends 32 ETH to the beacon deposit contract. There is no corresponding decrement path other than `verifyWithdrawalCredentials`.

**Decrement path — `verifyWithdrawalCredentials`:** [2](#0-1) 

The counter is reduced by `validatorFields.length * 32 ether`. This is the **only** place in the contract that decrements `stakedButUnverifiedNativeETH`.

**EigenPod rejects verification for exited validators:**

The EigenPod interface documents: *"Validators proven via this method MUST NOT have an exit epoch set already."* [3](#0-2) 

A slashed validator is forcibly exited by the beacon chain. Once the exit epoch is set, `eigenPod.verifyWithdrawalCredentials` reverts, making it permanently impossible to decrement `stakedButUnverifiedNativeETH` for that validator.

**`getEffectivePodShares` sums both terms:** [4](#0-3) 

`withdrawableShare` is the EigenLayer-tracked share balance (from `DelegationManager.getWithdrawableShares`). After the slashed validator exits and its remaining ETH is swept to the EigenPod, running a checkpoint credits that ETH as pod-balance shares. At that point both terms are non-zero simultaneously for the same underlying ETH, producing a double-count.

**TVL and price inflation path:**

`getEffectivePodShares` feeds into `getETHDistributionData`: [5](#0-4) 

Which feeds into `getTotalAssetDeposits` → `LRTOracle._getTotalEthInProtocol` → `_updateRsETHPrice`: [6](#0-5) 

An inflated `totalETHInProtocol` produces an inflated `rsETHPrice`. Existing rsETH holders can then redeem at the inflated rate, withdrawing more ETH than the protocol actually holds.

---

### Impact Explanation

- `stakedButUnverifiedNativeETH` is permanently stuck at 32 ETH per affected validator with no on-chain correction path.
- `getEffectivePodShares` overstates recoverable ETH by up to 32 ETH per validator (more if a checkpoint is run, causing a double-count of the swept partial withdrawal).
- `rsETHPrice` is inflated proportionally to the overcount relative to total rsETH supply.
- Existing rsETH holders can redeem at the inflated price, draining ETH that does not exist in the protocol → **protocol insolvency**.
- The `pricePercentageLimit` downside-protection in `LRTOracle` only triggers on price *decreases* below `highestRsethPrice`; it does not prevent the inflated price from being used for redemptions. [7](#0-6) 

---

### Likelihood Explanation

- Beacon chain slashing is rare but has occurred on mainnet (e.g., correlation penalties during client bugs).
- The vulnerable window is the period between `stake32Eth` and `verifyWithdrawalCredentials`. Operators typically verify quickly, but there is no on-chain time constraint enforcing this.
- No attacker action is required to trigger the slash; the attacker only needs to observe the inflated price and redeem existing rsETH holdings.
- The bug is permanent once triggered — there is no admin function to manually correct `stakedButUnverifiedNativeETH`.

---

### Recommendation

1. Add an admin/operator function to manually decrease `stakedButUnverifiedNativeETH` for validators that can be proven to have exited without credential verification (e.g., by submitting a beacon chain proof of the validator's exit epoch).
2. Alternatively, track per-validator state so that a validator whose exit can be proven on-chain can be removed from the unverified counter.
3. Consider adding a staleness check: if a validator pubkey has been in the unverified set for longer than a configurable threshold, flag it for operator review.

```
5. Assert eigenPod.verifyWithdrawalCredentials(...) reverts (exit epoch set).
6. Assert stakedButUnverifiedNativeETH == 32 ETH (unchanged).
7. Call startCheckpoint / verifyCheckpointProofs → withdrawableShare = 28 ETH.
8. Assert getEffectivePodShares() == 60 ETH  // double-count
9. Assert actual recoverable ETH == 28 ETH.
10. Call LRTOracle.updateRSETHPrice() → rsETHPrice inflated.
11. Existing rsETH holder redeems at inflated price, receives more ETH than fair share.
```
