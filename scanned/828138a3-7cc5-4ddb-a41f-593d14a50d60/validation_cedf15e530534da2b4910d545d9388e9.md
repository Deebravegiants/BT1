### Title
Unbounded Withdrawal Queue Spam Enables Permanent DoS of `unlockQueue` When `minRsEthAmountToWithdraw` Is Unset — (`File: contracts/LRTWithdrawalManager.sol`)

---

### Summary

When `minRsEthAmountToWithdraw[asset]` is at its default value of `0`, any rsETH holder can flood the withdrawal queue with an arbitrarily large number of dust (1-wei) withdrawal requests. Because `_unlockWithdrawalRequests` iterates the queue sequentially and there is no cap on total queue entries, the `unlockQueue` operator call becomes unboundedly expensive, permanently blocking legitimate users' withdrawals from being processed.

---

### Finding Description

`initiateWithdrawal` enforces a minimum amount check:

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```

`minRsEthAmountToWithdraw` is a `mapping(address => uint256)` that defaults to `0` for any asset that has not been explicitly configured by the admin via `setMinRsEthAmountToWithdraw`. When the stored value is `0`, the condition `rsETHUnstaked < 0` is always false for any `uint256`, so the only effective guard is `rsETHUnstaked == 0`. This means any amount ≥ 1 wei passes.

The withdrawal queue is a global FIFO ordered by `nextUnusedNonce[asset]`. The unlock loop in `_unlockWithdrawalRequests` must advance `nextLockedNonce_` sequentially through every entry:

```solidity
while (nextLockedNonce_ < firstExcludedIndex) {
    bytes32 requestId = getRequestId(asset, nextLockedNonce_);
    WithdrawalRequest storage request = withdrawalRequests[requestId];
    ...
    unchecked { nextLockedNonce_++; }
}
```

There is no cap on the total number of entries in the queue. The only economic constraint is `assetsCommitted[asset]`, which prevents committing more than the vault's available balance. However, with 1-wei requests, an attacker holding `N` wei of rsETH can create `N` separate queue entries, each committing ~1 wei of assets. For a protocol with significant TVL (e.g., 1,000 ETH in the unstaking vault = 10²¹ wei), the attacker can create an astronomically large number of entries.

---

### Impact Explanation

**Medium — Temporary (potentially permanent) freezing of funds.**

Legitimate users' withdrawal requests are queued behind the attacker's spam entries. The operator's `unlockQueue` call must iterate through all spam entries before reaching legitimate requests. With enough spam entries, no single `unlockQueue` call can advance past them within the block gas limit (~30M gas). Legitimate withdrawals are frozen for as long as the spam entries remain unprocessed, which could be indefinitely if the attacker continuously re-queues after their requests complete.

---

### Likelihood Explanation

**Medium.** The attacker must hold rsETH and pay gas per queued request. However:
- The rsETH is returned to the attacker when their requests are eventually processed — the capital is not lost, only temporarily locked.
- The only cost is gas. At 1-wei per request, the number of entries is bounded by the attacker's rsETH balance in wei, making the attack feasible for any holder of meaningful rsETH.
- `minRsEthAmountToWithdraw` is not set in `initialize`, so any newly supported asset or a fresh deployment is immediately vulnerable.

---

### Recommendation

1. **Enforce a non-zero minimum in `initialize`**: Set a meaningful default for `minRsEthAmountToWithdraw` (e.g., 0.001 ETH worth of rsETH) for all supported assets at initialization time.
2. **Add a per-user or global queue depth cap**: Reject `initiateWithdrawal` if the user already has more than `N` pending requests, or if the global queue depth exceeds a protocol-wide limit.
3. **Require `minRsEthAmountToWithdraw[asset] > 0` before accepting withdrawals**: Add a guard in `initiateWithdrawal` that reverts if the minimum has not been explicitly configured.

---

### Proof of Concept

1. Admin deploys `LRTWithdrawalManager` and adds support for `ETH`. `minRsEthAmountToWithdraw[ETH_TOKEN]` is `0` (default).
2. Attacker holds `X` rsETH (e.g., 1 ETH worth = 10¹⁸ wei).
3. Attacker calls `initiateWithdrawal(ETH_TOKEN, 1, "")` in a loop 10¹⁸ times (or as many as gas allows per block, across many blocks). Each call passes the check `rsETHUnstaked == 0 || rsETHUnstaked < 0` → false, so it succeeds.
4. Each call commits ~1 wei of assets and pushes a nonce into the global queue.
5. Operator calls `unlockQueue(ETH_TOKEN, nextUnusedNonce, ...)`. The `_unlockWithdrawalRequests` loop must iterate through all 10¹⁸ spam entries before reaching any legitimate user's request. The call reverts out-of-gas.
6. Legitimate users' withdrawals are permanently frozen behind the spam queue.
