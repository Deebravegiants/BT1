### Title
`KernelDepositPool` Reward Tokens Permanently Locked When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: `contracts/KERNEL/KernelDepositPool.sol`)

### Summary
When all stakers withdraw during an active reward distribution period, `totalKernelStaked` drops to zero. The `rewardPerToken()` function avoids division by zero by returning `rewardPerTokenStored` unchanged, but this silently stops reward accumulation. The reward tokens allocated for the zero-staking window are permanently locked in the contract with no recovery mechanism.

### Finding Description
`KernelDepositPool` implements a Synthetix-style staking rewards mechanism. The `rewardPerToken()` function guards against division by zero:

```solidity
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;
    }
    return rewardPerTokenStored
        + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
``` [1](#0-0) 

While this prevents a panic revert, it also means `rewardPerTokenStored` does not increase while `totalKernelStaked == 0`. Critically, the `updateReward` modifier still advances `updatedAt = lastTimeRewardApplicable()` on every user action: [2](#0-1) 

So when staking resumes, the reward calculation starts from the new `updatedAt`, permanently skipping the zero-staking window.

`notifyRewardAmount` blocks starting a new period when `totalKernelStaked == 0`: [3](#0-2) 

But it does **not** prevent `totalKernelStaked` from dropping to zero during an active period. Any staker can call `initiateWithdrawal` at any time. When the period eventually ends and a new one is started, `notifyRewardAmount` uses a transfer-in pattern — `receivedAmount` is only the newly transferred amount, not the already-locked tokens sitting in the contract: [4](#0-3) 

The contract's own NatSpec comment explicitly acknowledges this: [5](#0-4) 

There is no `recoverTokens`, `sweepRewards`, or any other admin function to retrieve the locked reward tokens. The only admin functions are `setRewardsDuration`, `notifyRewardAmount`, `setWithdrawalDelay`, and `setMaxNumberOfWithdrawalsPerUser`. [6](#0-5) 

### Impact Explanation
**Permanent freezing of unclaimed yield (High).** Reward tokens allocated for the zero-staking window — equal to `rewardRate × (zeroStakingDuration)` — are permanently locked in `KernelDepositPool` with no on-chain recovery path. These are real ERC-20 reward tokens transferred into the contract by the admin.

### Likelihood Explanation
**Medium.** The scenario requires `totalKernelStaked` to reach zero during an active reward period. This is realistic when:
- There is a single staker who calls `initiateWithdrawal` for their full balance.
- All stakers simultaneously unstake (e.g., in response to a market event or protocol concern).

The staker(s) need not be malicious — ordinary unstaking behavior is sufficient. The entry point `initiateWithdrawal` is permissionless for any staker. [7](#0-6) 

### Recommendation
Modify `notifyRewardAmount` to include the contract's existing excess reward token balance (i.e., `rewardsToken.balanceOf(address(this))` minus already-owed rewards) in the new `rewardRate` calculation, so previously unallocated rewards are automatically rolled forward. Alternatively, add an admin `recoverExcessRewards` function that can sweep unallocated reward tokens to the treasury when `totalKernelStaked == 0` and no active reward period is running.

### Proof of Concept
1. Admin calls `notifyRewardAmount(1000e18)` — `rewardRate = 1000e18 / duration`, `finishAt = T0 + duration`.
2. Single staker calls `initiateWithdrawal(totalStake)` at time `T1` — `totalKernelStaked = 0`, `updatedAt = T1`.
3. From `T1` to `T2` (end of period), every call to `rewardPerToken()` returns `rewardPerTokenStored` (frozen). Reward tokens for this window = `rewardRate × (T2 − T1)` accumulate in the contract but are never attributed to anyone.
4. After the period ends, a user stakes 1 wei — `updateReward` fires, `updatedAt = T2`.
5. Admin calls `notifyRewardAmount(1e18)` — `rewardRate = 1e18 / duration` (only the newly transferred tokens; `receivedAmount = balanceAfter − balanceBefore` excludes the already-present locked tokens).
6. The `rewardRate × (T2 − T1)` reward tokens remain in the contract forever with no mechanism to distribute or recover them.

        duration = _duration;
        emit RewardsDurationUpdated(_duration);
    }

    /**
     * @notice Notifies the contract about a new reward amount
     * @dev Uses a transfer-in pattern to determine the exact reward amount received.
     *      Also, to avoid undistributed rewards when no one is staked, this function reverts if totalKernelStaked is
     *      zero.
     * @param _amount The amount of reward tokens to add
     */
    function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
        if (_amount == 0) revert AmountZero();

        // Prevent starting a reward period when no tokens are staked to avoid unallocated rewards
        if (totalKernelStaked == 0) revert NoStakedTokens();

        // Transfer reward tokens into the contract
        uint256 balanceBefore = rewardsToken.balanceOf(address(this));
        rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
        uint256 balanceAfter = rewardsToken.balanceOf(address(this));
        // Calculate the actual amount of tokens received in case of a transfer fee (tax)
        uint256 receivedAmount = balanceAfter - balanceBefore;

        if (block.timestamp >= finishAt) {
            rewardRate = receivedAmount / duration;
        } else {
            uint256 remaining = (finishAt - block.timestamp) * rewardRate;
            rewardRate = (receivedAmount + remaining) / duration;
        }

        if (rewardRate == 0) revert RewardRateZero();

        finishAt = block.timestamp + duration;
        updatedAt = block.timestamp;

        emit NotifyRewardAmount(receivedAmount, finishAt);
    }
