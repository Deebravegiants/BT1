### Title
Zero-Output Token Deposit Silently Consumes User Funds Without Revert - (File: contracts/pools/RSETHPoolNoWrapper.sol)

---

### Summary

The `deposit(address token, uint256 amount, string memory referralId)` function in `RSETHPoolNoWrapper` (and identically in `RSETHPoolV3` and `RSETHPoolV3ExternalBridge`) pulls the user's full token amount into the pool, then computes `rsETHAmount` via integer division. When the deposit amount is small enough that `amountAfterFee * tokenToETHRate / rsETHToETHrate` truncates to zero, the function does not revert. It proceeds to transfer 0 rsETH to the user, permanently consuming their tokens with no recourse.

---

### Finding Description

In `RSETHPoolNoWrapper.deposit(address token, ...)`:

```solidity
// Line 262 — tokens pulled from user unconditionally
IERC20(token).safeTransferFrom(msg.sender, address(this), amount);

// Lines 264 / 301-311 — rsETHAmount computed via integer division
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
// rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate

// Line 268 — transfer of 0 rsETH succeeds silently
rsETH.safeTransfer(msg.sender, rsETHAmount);
``` [1](#0-0) 

The `viewSwapRsETHAmountAndFee` overload for tokens computes:

```
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate
``` [2](#0-1) 

When `amountAfterFee * tokenToETHRate < rsETHToETHrate`, Solidity integer division yields `rsETHAmount = 0`. There is no guard checking `rsETHAmount > 0` before or after the computation. `safeTransfer(msg.sender, 0)` on a standard ERC-20 returns `true` without reverting, so the entire transaction succeeds: the user's tokens are permanently held by the pool and they receive nothing.

The identical pattern exists in `RSETHPoolV3.deposit(address token, ...)` (line 284 pulls tokens, line 290 mints 0 wrsETH) and `RSETHPoolV3ExternalBridge.deposit(address token, ...)` (line 403 pulls tokens, line 409 mints 0 wrsETH). [3](#0-2) [4](#0-3) 

---

### Impact Explanation

**Critical — Direct theft of user funds.**

A user who deposits a token amount small enough to produce `rsETHAmount = 0` loses their entire deposit. The tokens are absorbed into the pool's balance with no accounting entry for the depositor and no recovery path. This is a permanent, irreversible loss of principal, not merely unclaimed yield.

---

### Likelihood Explanation

**Medium-to-High.** The condition is triggered by ordinary, unprivileged user behavior:

- Any supported token whose ETH rate (`tokenToETHRate`) is significantly lower than `rsETHToETHrate` creates a wide range of "dust" amounts that round to zero. For example, if `tokenToETHRate = 1e15` (token ≈ 0.001 ETH) and `rsETHToETHrate = 1.05e18`, any deposit of fewer than 1 050 token units (with zero fee) yields `rsETHAmount = 0`.
- A non-zero `feeBps` widens the vulnerable range further, because `amountAfterFee` is reduced before division.
- No minimum output parameter exists in the function signature — users have no on-chain slippage protection.
- The only existing guard is `if (amount == 0) revert InvalidAmount()`, which does not protect against sub-threshold amounts. [5](#0-4) 

---

### Recommendation

Add a zero-output guard immediately after computing `rsETHAmount`, mirroring the fix described in the reference report:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
if (rsETHAmount == 0) revert InvalidAmount();
```

Apply the same fix to the ETH deposit path and to the equivalent functions in `RSETHPoolV3` and `RSETHPoolV3ExternalBridge`. Optionally, expose a `minRsETHAmountExpected` parameter (as `LRTDepositPool.depositAsset` already does) so users can enforce their own slippage tolerance. [6](#0-5) 

---

### Proof of Concept

Assume:
- `tokenToETHRate = 1e15` (token oracle reports 0.001 ETH per token unit)
- `rsETHToETHrate = 1.05e18` (rsETH oracle reports 1.05 ETH per rsETH)
- `feeBps = 50` (0.5% fee)

User calls `deposit(token, 1000, "")`:

```
fee           = 1000 * 50 / 10_000 = 5
amountAfterFee = 1000 - 5 = 995
rsETHAmount   = 995 * 1e15 / 1.05e18
              = 9.95e17 / 1.05e18
              = 0  (integer division truncates)
```

Execution flow:
1. `safeTransferFrom(user, pool, 1000)` — 1 000 token units leave the user's wallet. ✓
2. `rsETH.safeTransfer(user, 0)` — succeeds, user receives nothing. ✓
3. Transaction emits `SwapOccurred(user, 0, 5, "")` — no on-chain revert signal.

The user has permanently lost 1 000 token units. The pool's `feeEarnedInToken[token]` is credited 5 units, and the remaining 995 units are silently absorbed into the pool's balance with no corresponding liability. [7](#0-6) [2](#0-1)
