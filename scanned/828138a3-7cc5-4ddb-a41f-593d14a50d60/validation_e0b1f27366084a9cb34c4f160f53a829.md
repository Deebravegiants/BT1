### Title
Increasing `withdrawalDelayBlocks` Retroactively Freezes Existing Pending Withdrawal Requests - (File: contracts/LRTWithdrawalManager.sol)

---

### Summary

The `withdrawalDelayBlocks` global variable in `LRTWithdrawalManager` is used at unlock-time to evaluate whether a withdrawal request's delay has elapsed. Because the stored `withdrawalStartBlock` is fixed at initiation but `withdrawalDelayBlocks` is read from current global state, any upward update to `withdrawalDelayBlocks` by the LRT manager retroactively extends the delay for all existing pending requests, temporarily freezing user funds beyond the delay they agreed to when initiating withdrawal.

---

### Finding Description

When a user calls `initiateWithdrawal()`, the contract records only `withdrawalStartBlock: block.number` in the `WithdrawalRequest` struct: [1](#0-0) 

No snapshot of the current `withdrawalDelayBlocks` is stored alongside the request.

Later, when the operator calls `unlockWithdrawals()` → `_unlockWithdrawalRequests()`, the delay check is: [2](#0-1) 

This reads the **current** global `withdrawalDelayBlocks`, not the value in effect when the request was created. The LRT manager can update this variable at any time via: [3](#0-2) 

The upper bound is `16 days / 12 seconds` blocks. The default set at initialization is `8 days / 12 seconds` blocks: [4](#0-3) 

If the manager increases `withdrawalDelayBlocks` from 8 days to 16 days after users have already initiated withdrawals, those requests will not pass the delay check in `_unlockWithdrawalRequests()` even if the original 8-day delay has already elapsed. The `break` statement at line 795 halts the entire unlock loop, meaning **all** queued requests behind the affected one are also blocked.

---

### Impact Explanation

**Temporary freezing of funds — Medium.**

Users who initiated withdrawals under an 8-day delay expectation can have their rsETH locked in the contract for up to an additional 8 days (the maximum increase possible). During this period:
- The rsETH is held by the `LRTWithdrawalManager` contract (transferred in at `initiateWithdrawal`).
- The user cannot complete the withdrawal (`completeWithdrawal` requires the request to be unlocked first).
- The user cannot reclaim their rsETH through any other path.

The freeze is temporary (bounded by the new delay), placing this squarely in the **Medium — temporary freezing of funds** category.

---

### Likelihood Explanation

The LRT manager role is a normal operational role, not a compromised one. Increasing the withdrawal delay is a plausible legitimate action (e.g., in response to a security incident, a bridge issue, or a protocol upgrade requiring more time). The manager does not need to be malicious — any upward adjustment of `withdrawalDelayBlocks` for any reason retroactively harms all users with pending requests. Given that the protocol is actively used and the manager role is exercised regularly, this scenario is realistic.

---

### Recommendation

Store the `withdrawalDelayBlocks` value at the time of request creation inside the `WithdrawalRequest` struct, and use that stored value in the delay check:

```diff
 struct WithdrawalRequest {
     uint256 rsETHUnstaked;
     uint256 expectedAssetAmount;
     uint256 withdrawalStartBlock;
+    uint256 withdrawalDelayBlocks;
 }
```

In `_addUserWithdrawalRequest`:
```diff
 withdrawalRequests[requestId] = WithdrawalRequest({
     rsETHUnstaked: rsETHUnstaked,
     expectedAssetAmount: expectedAssetAmount,
-    withdrawalStartBlock: block.number
+    withdrawalStartBlock: block.number,
+    withdrawalDelayBlocks: withdrawalDelayBlocks
 });
```

In `_unlockWithdrawalRequests`:
```diff
-if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) break;
+if (block.number < request.withdrawalStartBlock + request.withdrawalDelayBlocks) break;
```

This ensures each request is evaluated against the delay that was in effect when the user committed their rsETH, honoring the terms under which the withdrawal was initiated.

---

### Proof of Concept

1. Protocol initializes with `withdrawalDelayBlocks = 8 days / 12 seconds` (~57,600 blocks).
2. Alice calls `initiateWithdrawal(ETH, 1e18, "")`. Her rsETH is transferred to the contract. Her `WithdrawalRequest` records `withdrawalStartBlock = N`.
3. 57,600 blocks pass. Alice's withdrawal is now eligible to be unlocked under the original terms.
4. Before the operator calls `unlockWithdrawals`, the LRT manager calls `setWithdrawalDelayBlocks(16 days / 12 seconds)` (~115,200 blocks) — a legitimate security measure.
5. The operator calls `unlockWithdrawals`. Inside `_unlockWithdrawalRequests`, the check evaluates: `block.number < N + 115,200`. Since only 57,600 blocks have passed, the condition is true and the loop `break`s immediately.
6. Alice's withdrawal — and every request queued after hers — cannot be unlocked. Alice's rsETH remains frozen in the contract for an additional ~8 days with no recourse.
