### Title
Zero-value `safeTransfer` to `protocolTreasury` in `MerkleDistributor.claim()` can permanently freeze user token claims - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

---

### Summary

`MerkleDistributor.claim()` unconditionally calls `IERC20(token).safeTransfer(protocolTreasury, fee)` even when `fee` rounds down to zero. For tokens that revert on zero-value transfers, every affected user's `claim()` call will revert, permanently freezing their unclaimed token distributions in the contract.

---

### Finding Description

In `MerkleDistributor.claim()`, the fee is computed as:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);

// Send the fee to the protocol treasury
IERC20(token).safeTransfer(protocolTreasury, fee);   // ← unconditional, fee can be 0
``` [1](#0-0) 

`fee` evaluates to zero whenever `claimableAmount * feeInBPS < 10_000`. Two concrete scenarios:

1. **`feeInBPS = 0`** (fee disabled): `fee` is always `0`, so every single `claim()` call unconditionally attempts `safeTransfer(protocolTreasury, 0)`.
2. **Small claim amounts**: With `feeInBPS = 1` (0.01%), any `claimableAmount < 10_000` produces `fee = 0`.

The sibling contracts `KernelMerkleDistributor` and `KernelTop100MerkleDistributor` both correctly guard this transfer:

```solidity
if (fee > 0) {
    kernel.safeTransfer(protocolTreasury, fee);
}
``` [2](#0-1) [3](#0-2) 

`MerkleDistributor` is the only distributor missing this guard.

---

### Impact Explanation

If the distributed token reverts on zero-value transfers (a known behavior in tokens such as LEND, cUSDCv3, and others documented in the weird-ERC20 list), then:

- Every user whose `fee` rounds to zero will have their `claim()` permanently revert.
- The user's `userClaims` state is updated **before** the failing transfer (lines 134–135), meaning the state is already written when the revert unwinds — wait, actually Solidity reverts the entire transaction, so state is not committed. The user can retry, but the retry will always fail under the same conditions.
- The tokens remain locked in the `MerkleDistributor` contract with no recovery path for affected users.

**Impact: Medium — Permanent freezing of unclaimed yield.** [4](#0-3) 

---

### Likelihood Explanation

- `MerkleDistributor` is a **generic** distributor: `token` is set via `setToken()` and can be any ERC20. The protocol has no control over which tokens are used with it in the future.
- `feeInBPS = 0` is a natural operational choice (no fee), which makes `fee = 0` for **every** claim — affecting all users simultaneously.
- Even with non-zero `feeInBPS`, users with small cumulative allocations (common in broad airdrop distributions) will hit the rounding-to-zero case.
- No privileged access is required; any token holder with a valid merkle proof triggers the path.

---

### Recommendation

Mirror the guard already present in `KernelMerkleDistributor` and `KernelTop100MerkleDistributor`:

```diff
- IERC20(token).safeTransfer(protocolTreasury, fee);
+ if (fee > 0) IERC20(token).safeTransfer(protocolTreasury, fee);
``` [5](#0-4) 

---

### Proof of Concept

**Setup:**
- `MerkleDistributor` is deployed with a token that reverts on zero-value transfers (e.g., LEND).
- `feeInBPS = 0` (fee disabled by the owner).

**Execution:**
1. Alice has a valid merkle proof for `cumulativeAmount = 500`.
2. Alice calls `claim(index, alice, 500, proof)`.
3. `claimableAmount = 500 - 0 = 500`.
4. `fee = (500 * 0) / 10_000 = 0`.
5. `IERC20(token).safeTransfer(alice, 500)` succeeds.
6. `IERC20(token).safeTransfer(protocolTreasury, 0)` **reverts** because the token disallows zero-value transfers.
7. The entire transaction reverts. Alice cannot claim her 500 tokens.
8. Alice retries — same result. Her tokens are permanently frozen in the contract.

**Alternative scenario with non-zero fee:**
- `feeInBPS = 1`, `claimableAmount = 9_999`.
- `fee = (9_999 * 1) / 10_000 = 0` (rounds down).
- Same revert path. [6](#0-5)

```text
    /// @inheritdoc IMerkleDistributor
    function claim(
        uint256 index,
        address account,
        uint256 cumulativeAmount,
        bytes32[] calldata merkleProof
    )
        external
        override
        whenNotPaused
    {
        if (currentMerkleRoot == bytes32(0)) {
            revert ZeroValueProvided();
        }

        if (index == 0 || index > currentIndex) {
            revert InvalidIndex();
        }

        if (isClaimed(index, account)) {
            revert AlreadyClaimed();
        }

        // Verify the merkle proof.
        bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));
        if (!MerkleProofUpgradeable.verify(merkleProof, currentMerkleRoot, node)) {
            revert InvalidMerkleProof();
        }

        // Calculate the claimable amount
        uint256 claimableAmount = cumulativeAmount - userClaims[account].cumulativeAmount;

        // Ensure there is something to claim
        if (claimableAmount == 0) {
            revert NoTokensToClaim();
        }

        // Update user claim info, and send the token.
        userClaims[account].lastClaimedIndex = index;
        userClaims[account].cumulativeAmount = cumulativeAmount;

        // Send the claimable amount to the user - deducting the fee
        uint256 fee = (claimableAmount * feeInBPS) / 10_000;
        uint256 amountToSend = claimableAmount - fee;

        IERC20(token).safeTransfer(account, amountToSend);

        // Send the fee to the protocol treasury
        IERC20(token).safeTransfer(protocolTreasury, fee);

        emit Claimed(index, account, claimableAmount);
```
