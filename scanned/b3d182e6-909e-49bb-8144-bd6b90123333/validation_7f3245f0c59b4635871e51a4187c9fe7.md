### Title
Attacker Can Permanently Prevent Supported Token Removal via Dust Donation - (File: contracts/pools/RSETHPoolV3.sol)

### Summary
The `removeSupportedToken` function across all L2 pool contracts uses a strict zero-balance check (`IERC20(token).balanceOf(address(this)) != 0`) as a gate condition before delisting a token. Any unprivileged attacker can donate 1 wei of a supported token to the pool, permanently blocking the `TIMELOCK_ROLE` from removing that token from the supported deposit list.

### Finding Description
In `RSETHPoolV3.sol`, `removeSupportedToken` contains:

```solidity
function removeSupportedToken(address token, uint256 tokenIndex) external onlyRole(TIMELOCK_ROLE) {
    UtilLib.checkNonZeroAddress(token);
    if (supportedTokenList[tokenIndex] != token) revert TokenNotFoundError();
    if (IERC20(token).balanceOf(address(this)) != 0) revert TokenBalanceNotZero(); // ← vulnerable line
    delete supportedTokenOracle[token];
    supportedTokenList[tokenIndex] = supportedTokenList[supportedTokenList.length - 1];
    supportedTokenList.pop();
    emit RemovedSupportedToken(token);
}
``` [1](#0-0) 

The check is a strict equality to zero. Because ERC20 tokens can be freely transferred to any address by anyone, an attacker can call `IERC20(token).transfer(poolAddress, 1)` at any time to ensure `balanceOf(address(this)) >= 1`, causing the removal to revert with `TokenBalanceNotZero`. The attacker can repeat this indefinitely at negligible cost (1 wei + gas per transaction).

The same vulnerable pattern is present in every pool variant in the repository: [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

The analog to the original report is direct: the original issue used a `hasZeroBalance` trigger that an attacker kept non-zero by donating tokens; here the same donation pattern keeps `balanceOf != 0`, permanently blocking the administrative exit path for a token.

### Impact Explanation
The `removeSupportedToken` function is the only on-chain mechanism to delist a token from the pool's accepted deposit list. If the admin cannot remove a token:

- The pool continues to accept user deposits of that token indefinitely.
- If the token's oracle becomes unreliable, is deprecated, or the token itself is compromised, users can still deposit it and receive wrsETH at a potentially incorrect rate.
- The admin's only remaining recourse is to pause the entire pool contract, which also blocks all other legitimate deposits — a disproportionate response.

The contract fails to deliver the promised administrative return (token delisting) without any loss of value to the attacker.

**Impact: Low — Contract fails to deliver promised returns, but doesn't lose value.**

### Likelihood Explanation
The attack requires no privilege, no capital beyond 1 wei of the target token, and no timing precision. Any holder of a supported token (including the token's own deployer or any user who received dust) can execute it. The attacker can front-run every removal attempt. Likelihood is **High** given the trivial cost and permissionless entry path.

### Recommendation
Replace the strict zero check with a configurable dust threshold, mirroring the `maxNegligibleAmount` pattern already used in `LRTDepositPool._checkResidueLSTBalance`: [6](#0-5) 

```solidity
// Instead of:
if (IERC20(token).balanceOf(address(this)) != 0) revert TokenBalanceNotZero();

// Use:
if (IERC20(token).balanceOf(address(this)) > dustThreshold) revert TokenBalanceNotZero();
```

Where `dustThreshold` is a governance-settable value (e.g., `maxNegligibleAmount`). This ensures that a negligible donated balance does not block legitimate token removal, while still preventing removal when meaningful user funds remain in the pool.

### Proof of Concept

1. Pool supports token `T`. Admin decides to delist `T` (e.g., oracle deprecated).
2. Attacker observes the pending `removeSupportedToken(T, idx)` transaction in the mempool.
3. Attacker front-runs with `IERC20(T).transfer(poolAddress, 1)` — cost: 1 wei + gas.
4. Admin's transaction executes: `IERC20(T).balanceOf(address(this))` returns `1 != 0` → reverts with `TokenBalanceNotZero`.
5. Attacker repeats step 3 on every subsequent removal attempt.
6. Token `T` is permanently locked in the supported list. Users continue depositing `T` even after it is deprecated or its oracle becomes stale. [7](#0-6)
