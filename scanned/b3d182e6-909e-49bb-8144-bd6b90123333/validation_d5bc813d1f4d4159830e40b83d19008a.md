### Title
ETH Deposit Limit Bypass Due to Missing `amount` in Limit Check - (`contracts/LRTDepositPool.sol`)

---

### Summary

The `_checkIfDepositAmountExceedesCurrentLimit` function in `LRTDepositPool.sol` applies an asymmetric check: for LST assets it correctly evaluates `totalAssetDeposits + amount > limit`, but for ETH it omits the incoming `amount`, evaluating only `totalAssetDeposits > limit`. This means any ETH deposit passes the limit check as long as the pre-deposit total is at or below the limit, regardless of how large the deposit is.

---

### Finding Description

In `_checkIfDepositAmountExceedesCurrentLimit`:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));  // ← amount not included
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));  // ← correct for LSTs
}
``` [1](#0-0) 

For ETH, the function checks whether the **current** total exceeds the limit, not whether the **post-deposit** total would. The incoming `amount` (`msg.value`) is passed in but never used in the ETH branch. For all LST assets the check is `totalAssetDeposits + amount > limit`, which is correct.

The entry point is `depositETH`, callable by any unprivileged user:

```solidity
function depositETH(uint256 minRSETHAmountExpected, string calldata referralId)
    external payable nonReentrant whenNotPaused onlySupportedAsset(LRTConstants.ETH_TOKEN)
{
    uint256 rsethAmountToMint = _beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, minRSETHAmountExpected);
    _mintRsETH(rsethAmountToMint);
    ...
}
``` [2](#0-1) 

`_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit` and reverts on `true`. Because the ETH branch never adds `amount`, the check always returns `false` whenever `totalAssetDeposits <= depositLimit`, regardless of `msg.value`. [3](#0-2) 

---

### Impact Explanation

The ETH deposit limit (`depositLimitByAsset[ETH_TOKEN]`) is entirely ineffective. A user whose deposit would push the real total above the configured cap can still deposit and receive rsETH. The protocol's risk management invariant for ETH is broken. Deposit limits exist to cap EigenLayer strategy exposure; exceeding them can lead to strategy capacity overflow and, in the worst case, inability to redeem rsETH — a temporary or permanent freeze of user funds.

**Impact: Low** (contract fails to deliver promised returns — the deposit limit guarantee is not upheld for ETH). Escalates to **Medium** (temporary freezing of funds) if EigenLayer strategy capacity is breached as a result.

---

### Likelihood Explanation

The `depositETH` function is permissionless and callable by any user. No special role, governance action, or external condition is required. The bug is triggered on every ETH deposit when `totalAssetDeposits` is at or below the limit, which is the normal operating state of the protocol. Likelihood is **High**.

---

### Recommendation

Add `amount` to the ETH branch of the check, consistent with the LST branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [1](#0-0) 

---

### Proof of Concept

**Setup:**
- `depositLimitByAsset[ETH_TOKEN]` = 1000 ETH
- Current `getTotalAssetDeposits(ETH_TOKEN)` = 1000 ETH (exactly at the limit)
- Attacker calls `depositETH{value: 500 ETH}(...)`

**Execution:**
1. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500e18)` is called.
2. `totalAssetDeposits = 1000e18`.
3. ETH branch evaluates: `1000e18 > 1000e18` → `false`.
4. Check returns `false` → deposit is **not** blocked.
5. `_mintRsETH` mints rsETH for 500 ETH above the limit.

**Expected behavior (LST path for comparison):**
- `totalAssetDeposits + amount = 1500e18 > 1000e18` → `true` → deposit reverts with `MaximumDepositLimitReached`.

The ETH deposit limit is bypassed by exactly `msg.value` on every call when `totalAssetDeposits <= limit`. [4](#0-3)
