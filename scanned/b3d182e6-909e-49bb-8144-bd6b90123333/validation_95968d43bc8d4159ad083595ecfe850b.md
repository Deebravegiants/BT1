### Title
ETH Deposit Limit Bypassed Due to Missing Amount in Limit Check - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric check: for ERC20 assets it correctly adds the incoming deposit amount to the running total before comparing against the configured limit, but for ETH it omits the incoming amount entirely. As a result, the `depositLimitByAsset[ETH_TOKEN]` cap set by the admin is never enforced against the size of a new ETH deposit — only against the pre-existing total — allowing any depositor to push the protocol's ETH holdings arbitrarily beyond the configured ceiling.

### Finding Description
`LRTConfig.depositLimitByAsset` stores a per-asset deposit ceiling that the admin configures via `addNewSupportedAsset` / `updateAssetDepositLimit`. [1](#0-0) 

During every deposit, `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit` to enforce that ceiling. [2](#0-1) 

Inside that helper the ETH branch reads:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
}
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
``` [3](#0-2) 

For every ERC20 asset the new `amount` is included in the comparison. For ETH it is not. The function therefore returns `true` (i.e., "limit exceeded, reject") only when the pre-existing total already exceeds the limit — never because the incoming deposit would push it over. A depositor who calls `depositETH` while `totalAssetDeposits < depositLimit` will always pass the check, regardless of how large `msg.value` is.

The entry path is fully public and permissionless: [4](#0-3) 

### Impact Explanation
The `depositLimitByAsset[ETH_TOKEN]` cap is a risk-management parameter intended to bound the protocol's ETH exposure. Because the check omits the incoming deposit amount, the cap is never actually enforced: a single depositor can, in one transaction, push the ETH total from just below the limit to an arbitrarily large value. The protocol mints rsETH for the full amount and the excess ETH is subsequently delegated to EigenLayer strategies, exceeding the intended ceiling. The deposit limit feature fails to deliver its promised protection for ETH, though no funds are directly stolen or permanently frozen.

**Impact: Low — Contract fails to deliver promised returns, but doesn't lose value.**

### Likelihood Explanation
The vulnerable path is the standard `depositETH` function, callable by any user at any time. No special role, timing, or precondition is required beyond the current ETH total being below the limit (the normal operating state). Likelihood is **High**.

### Recommendation
Include the incoming deposit amount in the ETH branch, mirroring the ERC20 logic:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
```

### Proof of Concept
1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 1000 ether` via `updateAssetDepositLimit`.
2. Current `getTotalAssetDeposits(ETH_TOKEN)` returns `999 ether`.
3. Attacker calls `depositETH{value: 5000 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH, 5000e18)` evaluates `999e18 > 1000e18` → `false` → check passes.
5. `5000 ether` of rsETH is minted; total ETH in protocol becomes `5999 ether`, far exceeding the `1000 ether` limit.
6. The same call with an ERC20 asset would have evaluated `999e18 + 5000e18 > 1000e18` → `true` → correctly rejected. [3](#0-2) [5](#0-4)

        private

    {
        if (depositAmount == 0 || depositAmount < minAmountToDeposit) {
            revert InvalidAmountToDeposit();
        }

        if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
            revert MaximumDepositLimitReached();
        }

        rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

        if (rsethAmountToMint < minRSETHAmountExpected) {
            revert MinimumAmountToReceiveNotMet();
        }
    }
