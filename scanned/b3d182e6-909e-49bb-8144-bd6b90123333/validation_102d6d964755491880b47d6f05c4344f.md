### Title
`removeAllowedToken` Blocks Bridger Collateralization of Outstanding wrsETH, Breaking the Wrapper's Backing Invariant — (`contracts/L2/RsETHTokenWrapper.sol`)

---

### Summary

`removeAllowedToken` sets `allowedTokens[asset] = false` with no check for outstanding uncollateralized wrsETH. Once a token is removed, `maxAmountToDepositBridgerAsset` hard-returns `0` for that asset, causing any subsequent `depositBridgerAssets` call for that asset to revert. wrsETH already minted against that asset remains permanently undercollateralized until governance re-adds the token.

---

### Finding Description

The `mint()` function (called by pool contracts such as `RSETHPoolV3ExternalBridge.deposit()`) mints wrsETH without requiring any collateral to be present in the wrapper at mint time. [1](#0-0) 

The `BRIDGER_ROLE` is expected to later call `depositBridgerAssets` to bring the backing asset into the wrapper and close the collateral gap. [2](#0-1) 

`depositBridgerAssets` gates itself on `maxAmountToDepositBridgerAsset`, which unconditionally returns `0` for any non-allowed token: [3](#0-2) 

Because `0 < amount` is always true for any non-zero deposit, the call reverts with `CannotDeposit()`. [4](#0-3) 

`removeAllowedToken` performs no check on whether outstanding wrsETH is already backed (or unbacked) by the token being removed: [5](#0-4) 

Additionally, `_withdraw` also gates on `allowedTokens`, so wrsETH holders cannot redeem the removed token either: [6](#0-5) 

The combined effect: after `removeAllowedToken(tokenA)`, the wrapper's `totalSupply()` can exceed the sum of all allowed-token balances, and neither the bridger nor users can correct this through normal contract paths.

---

### Impact Explanation

The wrapper's core invariant — that every outstanding wrsETH token is backed 1:1 by an allowed collateral token — is broken. Users holding wrsETH cannot redeem it for `tokenA`, and the bridger cannot deposit `tokenA` to restore backing. This matches **Low: Contract fails to deliver promised returns, but doesn't lose value** — the tokenA is still held in the wrapper (if any was previously deposited), but the protocol can no longer route it correctly, and newly minted-but-unbacked wrsETH has no collateral path.

---

### Likelihood Explanation

The scenario is operationally realistic. The TIMELOCK_ROLE is a legitimate governance role (deployed as a `TimelockController` on Base, Optimism, Scroll, and Linea per the README). A token migration or deprecation — a normal governance action — would trigger `removeAllowedToken`. If any wrsETH was minted by a pool contract before the bridger's collateral deposit landed on-chain, the removal window creates the undercollateralized state. No key compromise or malicious intent is required; a routine governance action is sufficient.

---

### Recommendation

Add a collateral-gap check inside `removeAllowedToken` before allowing removal:

```solidity
function removeAllowedToken(address _asset) external onlyRole(TIMELOCK_ROLE) {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    // Ensure no uncollateralized wrsETH is backed by this asset
    uint256 gap = maxAmountToDepositBridgerAsset(_asset);
    if (gap > 0) revert OutstandingCollateralGap();
    allowedTokens[_asset] = false;
    emit TokenRemoved(_asset);
}
```

This prevents removal while any wrsETH remains unbacked by `_asset`, forcing governance to either wait for the bridger to fully collateralize or burn the outstanding wrsETH first.

### Proof of Concept

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity 0.8.27;

import "forge-std/Test.sol";
import "contracts/L2/RsETHTokenWrapper.sol";
import "@openzeppelin/contracts/token/ERC20/ERC20.sol";

contract MockToken is ERC20 {
    constructor() ERC20("AltRsETH", "altRsETH") { _mint(msg.sender, 1000e18); }
}

contract RemoveAllowedTokenRaceTest is Test {
    RsETHTokenWrapper wrapper;
    MockToken tokenA;

    address admin   = address(0xA);
    address minter  = address(0xB);
    address bridger = address(0xC);
    address timelock = address(0xD);
    address user    = address(0xE);

    function setUp() public {
        tokenA  = new MockToken();
        wrapper = new RsETHTokenWrapper();
        wrapper.initialize(admin, bridger, address(tokenA));

        vm.startPrank(admin);
        wrapper.grantRole(wrapper.MINTER_ROLE(),   minter);
        wrapper.grantRole(wrapper.TIMELOCK_ROLE(), timelock);
        vm.stopPrank();

        // Fund bridger with tokenA
        tokenA.transfer(bridger, 100e18);
        vm.prank(bridger);
        tokenA.approve(address(wrapper), type(uint256).max);
    }

    function test_removeAllowedToken_blocks_bridger_deposit() public {
        // Step 1: MINTER_ROLE mints wrsETH (simulating a pool deposit from L2 user)
        vm.prank(minter);
        wrapper.mint(user, 50e18);

        // Confirm undercollateralized: totalSupply=50, tokenA balance=0
        assertEq(wrapper.totalSupply(), 50e18);
        assertEq(wrapper.maxAmountToDepositBridgerAsset(address(tokenA)), 50e18);

        // Step 2: TIMELOCK_ROLE removes tokenA BEFORE bridger deposits
        vm.prank(timelock);
        wrapper.removeAllowedToken(address(tokenA));

        // Step 3: maxAmountToDepositBridgerAsset now returns 0
        assertEq(wrapper.maxAmountToDepositBridgerAsset(address(tokenA)), 0);

        // Step 4: Bridger's depositBridgerAssets reverts
        vm.prank(bridger);
        vm.expectRevert(RsETHTokenWrapper.CannotDeposit.selector);
        wrapper.depositBridgerAssets(address(tokenA), 50e18);

        // Step 5: Invariant broken — totalSupply > sum of allowed token balances
        // tokenA is no longer allowed, so effective backing = 0
        assertGt(wrapper.totalSupply(), 0); // 50e18 wrsETH outstanding, 0 backing
    }
}
```

Running `forge test --match-test test_removeAllowedToken_blocks_bridger_deposit` on unmodified code will pass, confirming the invariant break.
