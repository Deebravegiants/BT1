### Title
Reward Token Dust Permanently Locked Due to Integer Division Truncation in `notifyRewardAmount` - (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary
In `KernelDepositPool.notifyRewardAmount()`, the `rewardRate` is computed via integer division, silently truncating a remainder of up to `duration - 1` raw token units per call. The truncated tokens are transferred into the contract but are never distributed to stakers and cannot be recovered, permanently locking yield.

### Finding Description
`notifyRewardAmount` computes the per-second reward rate as:

```solidity
// new period (line 580)
rewardRate = receivedAmount / duration;

// mid-period rollover (line 583)
rewardRate = (receivedAmount + remaining) / duration;
```

Solidity integer division truncates the result. The discarded remainder â€” `receivedAmount % duration` (or `(receivedAmount + remaining) % duration`) â€” represents real reward tokens that were transferred into the contract at line 574 but are never accounted for in the `rewardPerToken` accumulator. Because the contract has no `sweep`, `recoverERC20`, or equivalent rescue function, these tokens are permanently locked.

This is the direct analog to the reference bug: just as the reference contract used an average TPS across epochs of different lengths instead of computing `rate_i Ã— time_i` per epoch, `KernelDepositPool` computes a uniform `rewardRate` via lossy integer division, causing the per-period reward budget to be systematically under-distributed. ### Impact Explanation
Every call to `notifyRewardAmount` permanently locks up to `duration - 1` raw token units. There is no recovery path in the contract. The locked amount accumulates across reward periods. For a 7-day `duration` (604 800 seconds) and a 6-decimal reward token, each call can lock up to ~0.6 tokens; over a year of weekly calls that is ~31 tokens permanently frozen. For higher-value or lower-decimal tokens the loss per call is proportionally larger. Impact: **Medium â€” permanent freezing of unclaimed yield**. [2](#0-1) 

### Likelihood Explanation
`notifyRewardAmount` is called by the admin on a regular cadence (weekly/monthly) as part of normal protocol operation. No exploit or compromise is required â€” the truncation occurs on every invocation regardless of inputs. Likelihood: **High**.

### Recommendation
Track the undistributed remainder and carry it forward into the next reward period:

```solidity
uint256 leftover = (block.timestamp >= finishAt)
    ? 0
    : (finishAt - block.timestamp) * rewardRate;
rewardRate = (receivedAmount + leftover) / duration;
// carry forward: store (receivedAmount + leftover) % duration and add to next call
```

Alternatively, add an admin-accessible `recoverERC20` function that is gated to only allow recovery of tokens in excess of `rewardRate * (finishAt - block.timestamp)`.
