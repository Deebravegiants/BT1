### Title
`ChainlinkPriceOracle::getAssetPrice` Does Not Check for Stale Price - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary
`ChainlinkPriceOracle::getAssetPrice` calls Chainlink's `latestRoundData()` but discards every return value except `answer` (price). Neither `updatedAt` nor `answeredInRound` is validated, so a stale price is silently accepted and used to compute how many rsETH tokens a depositor receives.

### Finding Description
In `ChainlinkPriceOracle::getAssetPrice`, the call to `latestRoundData()` uses a wildcard destructuring that ignores `updatedAt` and `answeredInRound`:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52
(, int256 price,,,) = priceFeed.latestRoundData();
``` [1](#0-0) 

No heartbeat check (`block.timestamp - updatedAt > heartbeat`) and no round-completeness check (`answeredInRound < roundId`) are performed. The raw `price` value is returned directly to callers regardless of how old it is.

This contrasts with `ChainlinkOracleForRSETHPoolCollateral::getRate`, which at least checks `answeredInRound < roundID` and `timestamp == 0`, showing the project is aware of the pattern but did not apply it to `ChainlinkPriceOracle`. [2](#0-1) 

### Impact Explanation
`ChainlinkPriceOracle::getAssetPrice` is the terminal price source for the core deposit flow:

```
LRTDepositPool::depositAsset / depositETH
  → _beforeDeposit
    → getRsETHAmountToMint          (LRTDepositPool.sol L520)
      → LRTOracle::getAssetPrice    (LRTOracle.sol L157)
        → ChainlinkPriceOracle::getAssetPrice  (ChainlinkPriceOracle.sol L52)
``` [3](#0-2) [4](#0-3) 

The rsETH mint amount is computed as:

```
rsethAmountToMint = (amount × assetPrice) / rsETHPrice
```

- **Inflated stale price** → depositor mints more rsETH than their assets are worth → existing holders are diluted → **protocol insolvency**.
- **Deflated stale price** → depositor mints fewer rsETH than deserved → **loss of value for the depositor**.

**Impact: Critical** — direct path to protocol insolvency or theft of depositor value.

### Likelihood Explanation
Chainlink feeds have documented heartbeat windows (e.g., 1 hour for ETH/USD, 24 hours for some LST feeds). During periods of L1/L2 congestion, sequencer downtime, or oracle node failure, feeds can go stale within their heartbeat window without triggering a deviation update. Because `ChainlinkPriceOracle` accepts any non-reverted answer unconditionally, any such event immediately propagates a stale price into rsETH minting. No attacker action is required beyond depositing during the staleness window.

### Recommendation
Capture and validate `updatedAt` and `answeredInRound` in `ChainlinkPriceOracle::getAssetPrice`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (price <= 0) revert InvalidPrice();
    if (answeredInRound < roundId) revert StalePrice();
    if (block.timestamp - updatedAt > heartbeat[asset]) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Store a per-asset `heartbeat` value (e.g., in `assetPriceFeed` mapping or a parallel `assetHeartbeat` mapping) set by `onlyLRTManager`, mirroring the approach recommended in the reference report for `MarginCollateralConfiguration::Data`.

### Proof of Concept

1. Chainlink's stETH/ETH feed (or any LST feed used by the protocol) goes stale — e.g., last updated 25 hours ago due to low volatility and no deviation trigger.
2. The actual market price of stETH has dropped 5% since the last update (e.g., due to a slashing event).
3. An attacker calls `LRTDepositPool::depositAsset(stETH, amount, 0)`.
4. `getRsETHAmountToMint` calls `ChainlinkPriceOracle::getAssetPrice` which returns the stale (inflated) price.
5. The attacker receives rsETH computed at the old higher price — more rsETH than their stETH is currently worth.
6. The attacker redeems rsETH, extracting value from honest depositors and pushing the protocol toward insolvency. [1](#0-0) [5](#0-4)
