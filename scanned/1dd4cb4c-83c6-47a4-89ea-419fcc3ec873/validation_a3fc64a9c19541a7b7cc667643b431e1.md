### Title
Missing Zero-Share Guard in `depositToKingProtocol` Allows Silent Loss of Deposited Assets — (`contracts/king-protocol/TokenSwap.sol`)

---

### Summary

`TokenSwap.depositToKingProtocol` calls `kingProtocol.previewDeposit` to obtain the expected share count, stores it in `shareReceived`, but **never validates that `shareReceived > 0`** before executing the actual deposit. If King Protocol returns `(0, amount)` from `previewDeposit` — whether due to 100% fees, extreme fee configuration, or share-price rounding on small amounts — the function proceeds, transfers all tokens to King Protocol, mints zero KING shares to `TokenSwap`, and returns successfully. The deposited assets are permanently lost as fees with no yield-bearing position created.

---

### Finding Description

In `TokenSwap.depositToKingProtocol`:

```
(uint256 expectedShares,) = kingProtocol.previewDeposit(tokens, amounts);
shareReceived = expectedShares;                    // may be 0 — never checked

assetToken.forceApprove(address(kingProtocol), amount);
kingProtocol.deposit(tokens, amounts, address(this));  // executes regardless
assetToken.forceApprove(address(kingProtocol), 0);

emit TokensDeposited(asset, amount, shareReceived, msg.sender);  // emits shareReceived=0
```

The `previewDeposit` call is the contract's own mechanism to determine expected output. Its result is captured but never acted upon as a guard. The deposit executes unconditionally after the preview.

The same flaw exists in `depositMultipleToKingProtocol` at line 210–214.

**Realistic triggers (not requiring full external compromise):**

1. **Fee rounding on small amounts**: If King Protocol's fee is, say, 50% and the deposit is 1 wei, integer division yields `shareToMint = 0`. No extreme state required.
2. **High share price**: If King Protocol's share price is very high relative to the deposit amount, `shareToMint` rounds to 0 via integer arithmetic.
3. **King Protocol fee governance**: King Protocol governance raising fees to a level where `shareToMint = 0` for the deposited amount.

None of these require compromise of `TokenSwap`'s own admin/manager keys.

---

### Impact Explanation

When `expectedShares == 0` and the deposit proceeds:
- All `amount` tokens are transferred from `TokenSwap` to King Protocol (consumed as fees).
- Zero KING shares are minted to `TokenSwap`.
- `kingToken.balanceOf(address(tokenSwap))` is unchanged (still 0 for that deposit).
- The yield-bearing position that should have been created is permanently absent.
- The deposited assets cannot be recovered through `withdrawKing` (nothing to withdraw).
- `emergencyWithdraw` only rescues tokens still held by `TokenSwap`, not tokens already sent to King Protocol.

This matches **Medium — Permanent freezing of unclaimed yield**: all deposited assets are consumed as fees, zero yield-bearing position is created, and the yield is permanently lost.

---

### Likelihood Explanation

- The function is restricted to `onlyAdminOrManager`, which limits the attack surface to operational mistakes or King Protocol state changes.
- However, the admin/manager has no on-chain protection forcing them to check `previewDeposit` output before calling `depositToKingProtocol`. The contract itself is supposed to enforce this invariant.
- Share-price rounding to zero is a realistic scenario for small deposits or high-fee configurations, requiring no external compromise.
- Likelihood: **Low-to-Medium** (requires specific King Protocol state, but no attacker action needed — an honest admin/manager call suffices).

---

### Recommendation

Add a zero-share guard immediately after the `previewDeposit` call in both `depositToKingProtocol` and `depositMultipleToKingProtocol`:

```solidity
(uint256 expectedShares,) = kingProtocol.previewDeposit(tokens, amounts);
if (expectedShares == 0) revert ZeroAmount();
shareReceived = expectedShares;
```

This ensures the invariant "depositing non-zero assets always results in non-zero yield-bearing shares" is enforced at the `TokenSwap` layer, independent of King Protocol's internal state.

---

### Proof of Concept

```solidity
// MockKingProtocol: previewDeposit returns (0, fullAmount), deposit transfers tokens and mints nothing
contract MockKingProtocol {
    function previewDeposit(address[] memory, uint256[] memory amounts)
        external pure returns (uint256 shareToMint, uint256 depositFee) {
        return (0, amounts[0]); // 100% fee
    }
    function deposit(address[] memory tokens, uint256[] memory amounts, address) external {
        // consume all tokens as fees, mint zero shares
        IERC20(tokens[0]).transferFrom(msg.sender, address(this), amounts[0]);
    }
}

// Test
function testZeroShareDeposit() public {
    // Setup: TokenSwap with MockKingProtocol, fund with 1e18 asset tokens
    tokenSwap.depositToKingProtocol(asset, 1e18); // succeeds, no revert
    assertEq(kingToken.balanceOf(address(tokenSwap)), 0); // zero KING shares
    assertEq(assetToken.balanceOf(address(tokenSwap)), 0); // all assets gone
}
```

The call succeeds, emits `TokensDeposited(asset, 1e18, 0, caller)`, and leaves `TokenSwap` with zero KING shares and zero asset tokens.
