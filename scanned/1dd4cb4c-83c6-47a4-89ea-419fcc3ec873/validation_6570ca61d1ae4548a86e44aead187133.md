### Title
Depositors Can Sandwich `updateRSETHPrice()` to Steal Yield from Existing rsETH Holders — (File: `contracts/LRTOracle.sol`, `contracts/LRTDepositPool.sol`, `contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTOracle.rsETHPrice` is a cached value that only updates when `updateRSETHPrice()` is called. Because `updateRSETHPrice()` is a public, permissionless function, an attacker can atomically: (1) deposit at the stale lower price to receive more rsETH than fair value, (2) call `updateRSETHPrice()` to reflect accrued yield, and (3) initiate withdrawal at the newly elevated price. This transfers yield that belongs to existing rsETH holders to the attacker.

---

### Finding Description

`LRTOracle` stores the rsETH/ETH exchange rate in the state variable `rsETHPrice`. [1](#0-0) 

This value is only refreshed when `updateRSETHPrice()` is explicitly called. Critically, this function is **public and permissionless**: [2](#0-1) 

Between calls, the underlying LST prices and staking rewards continue to accrue, making `rsETHPrice` stale. When a deposit is made, `LRTDepositPool.getRsETHAmountToMint` divides by the stale `rsETHPrice`: [3](#0-2) 

A lower stale price means the depositor receives **more rsETH** than the current fair value warrants.

When `_updateRsETHPrice()` subsequently runs, it computes `previousTVL` using the stale `rsETHPrice` multiplied by the now-inflated rsETH supply (which includes the attacker's freshly minted tokens): [4](#0-3) 

The accrued yield (`totalETHInProtocol - previousTVL`) is therefore diluted across the attacker's rsETH as well, and the new price rises to a level that reflects the attacker's share of the yield.

The attacker then calls `initiateWithdrawal()`, which locks in `expectedAssetAmount` using the freshly updated (higher) `rsETHPrice`: [5](#0-4) [6](#0-5) 

At unlock time, `_calculatePayoutAmount` returns `min(expectedAssetAmount, currentReturn)`: [7](#0-6) 

Because `expectedAssetAmount` was locked in at the post-update higher price, the attacker's profit is secured at step 3. The `min()` only protects against further price drops after initiation, not against the gain already captured.

---

### Impact Explanation

**Impact: High — Theft of unclaimed yield.**

Concrete example with numbers:
- Protocol TVL: 1,000 ETH; rsETH supply: 1,000; stale `rsETHPrice` = 1.0 ETH; actual underlying = 1,010 ETH (10 ETH yield accrued but not yet reflected).
- Attacker deposits 1,000 ETH at stale price → minted 1,000 rsETH (fair value would be ~990 rsETH).
- `updateRSETHPrice()` is called: new TVL = 2,010 ETH, supply = 2,000 rsETH → new price = 1.005 ETH.
- Attacker initiates withdrawal: `expectedAssetAmount = 1,000 × 1.005 / 1.0 = 1,005 ETH`.
- Attacker profits **5 ETH**; original holders receive 1,005 ETH instead of 1,010 ETH — **5 ETH of yield stolen**.

The attack scales linearly with deposit size and the length of the stale window. If `instantWithdrawal` is enabled for an asset, the entire cycle (deposit → price update → instant withdrawal) can be executed atomically with a flash loan, eliminating the 8-day capital lockup requirement. [8](#0-7) 

---

### Likelihood Explanation

**Likelihood: Low.**

Without instant withdrawal, the attacker must lock capital for `withdrawalDelayBlocks` (~8 days): [9](#0-8) 

The `pricePercentageLimit` caps the price increase per update cycle, bounding per-attack profit: [10](#0-9) 

However, the attack requires no privileged access, no oracle manipulation, and no external protocol compromise — only the ability to monitor the mempool for stale price windows and submit a deposit transaction. When instant withdrawal is enabled, the attack becomes flash-loan-executable in a single block, raising likelihood significantly.

---

### Recommendation

1. **Enforce a deposit fee or entry delay** proportional to the time since the last price update, so late depositors cannot capture accrued yield for free.
2. **Update `rsETHPrice` atomically inside `depositETH`/`depositAsset`** before computing `rsethAmountToMint`, ensuring deposits always use a fresh price.
3. **Restrict or rate-limit `updateRSETHPrice()`** to authorized callers (e.g., `MANAGER` role or a keeper with a minimum call interval), preventing an attacker from triggering the update at will.
4. If instant withdrawal remains enabled, apply a withdrawal fee large enough to exceed the maximum yield capturable in one update cycle.

---

### Proof of Concept

```
Block N (stale window — yield has accrued, rsETHPrice not yet updated):
  1. Attacker calls LRTDepositPool.depositETH{value: 1000 ether}(0, "")
     → getRsETHAmountToMint uses stale rsETHPrice (1.0 ETH)
     → attacker receives 1000 rsETH (fair value ≈ 990 rsETH)

Block N (same tx or next block):
  2. Attacker calls LRTOracle.updateRSETHPrice()
     → totalETHInProtocol = 2010 ETH, rsethSupply = 2000
     → rsETHPrice updated to 1.005 ETH

Block N:
  3. Attacker calls LRTWithdrawalManager.initiateWithdrawal(ETH, 1000e18, "")
     → expectedAssetAmount = 1000 * 1.005 / 1.0 = 1005 ETH
     → locked in at the post-update price

Block N + withdrawalDelayBlocks (~57,600 blocks):
  4. Operator calls unlockQueue(); attacker calls completeWithdrawal()
     → attacker receives 1005 ETH
     → net profit: 5 ETH stolen from existing rsETH holders

[If instantWithdrawal is enabled for the asset:]
  Steps 1–4 collapse into a single flash-loan transaction:
  flash borrow → depositETH → updateRSETHPrice → instantWithdrawal → repay loan → pocket profit
```

        address asset,
        uint256 rsETHUnstaked,
        string calldata referralId
    )
        external
        nonReentrant
        whenNotPaused
        onlySupportedAsset(asset)
        onlySupportedStrategy(asset)
        onlyInstantWithdrawalAllowed(asset)
    {
        if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
            revert InvalidAmountToWithdraw();
        }
        if (IERC20(lrtConfig.rsETH()).balanceOf(msg.sender) < rsETHUnstaked) revert NotEnoughRsETH();
        uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
        IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
        ILRTUnstakingVault unstakingVault = ILRTUnstakingVault(lrtConfig.getContract(LRTConstants.LRT_UNSTAKING_VAULT));
        if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
            revert CantInstantWithdrawMoreThanAvailable();
        }

        unstakingVault.redeem(asset, assetAmountUnlocked);

        uint256 fee = (assetAmountUnlocked * instantWithdrawalFee) / 10_000;
        uint256 userAmount = assetAmountUnlocked - fee;

        address feeRecipient = instantWithdrawalFeeRecipient;
        if (feeRecipient == address(0)) {
            // Backwards-compatible default: send fees to the protocol treasury
            feeRecipient = lrtConfig.getContract(LRTConstants.PROTOCOL_TREASURY);
        }
        if (fee > 0) {
            _transferAsset(asset, feeRecipient, fee);
            emit InstantWithdrawalFeeCollected(msg.sender, asset, fee);
        }

        _transferAsset(asset, msg.sender, userAmount);
        emit ReferralIdEmitted(referralId);
        emit AssetWithdrawalFinalized(msg.sender, asset, rsETHUnstaked, userAmount);
    }
