### Title
Chainlink Price Feed Used Without Staleness or Validity Checks in `ChainlinkPriceOracle` - (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `priceFeed.latestRoundData()` but discards all round metadata, performing no checks on price validity, round completeness, or staleness. A stale or zero Chainlink price propagates directly into rsETH minting amounts and the protocol's total ETH accounting, causing depositors to receive incorrect rsETH amounts or lose deposited assets entirely.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the Chainlink price as follows:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

The fields `roundId`, `updatedAt`, and `answeredInRound` are all silently discarded. There are no checks for:
- `price > 0` — a zero or negative price is accepted
- `updatedAt != 0` — an incomplete round is accepted
- `answeredInRound >= roundId` — a stale price from a prior round is accepted

This is the oracle used by `LRTOracle.getAssetPrice()` for all supported LST assets:

```solidity
return IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset);
``` [2](#0-1) 

The stale price then flows into two critical paths:

**1. rsETH minting during deposits** (`LRTDepositPool.getRsETHAmountToMint`):
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [3](#0-2) 

**2. Protocol-wide rsETH price update** (`LRTOracle._getTotalEthInProtocol`):
```solidity
uint256 assetER = getAssetPrice(asset);
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
``` [4](#0-3) 

Which feeds `_updateRsETHPrice()` and permanently corrupts the stored `rsETHPrice`: [5](#0-4) 

Notably, the sibling contract `ChainlinkOracleForRSETHPoolCollateral` in the same repository already implements all three required checks correctly:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [6](#0-5) 

This confirms the protocol is aware of the pattern but did not apply it to `ChainlinkPriceOracle`.

---

### Impact Explanation

**Scenario A — Stale price (too low):** A depositor calling `depositAsset()` receives fewer rsETH than their deposit is worth. Existing rsETH holders are not diluted, but the depositor suffers a direct loss of value relative to their deposit.

**Scenario B — Price = 0 (incomplete round or Chainlink node failure):** `getAssetPrice` returns 0. `getRsETHAmountToMint` computes `(amount * 0) / rsETHPrice = 0`. The depositor's LST tokens are transferred in but 0 rsETH is minted — a permanent loss of the deposited assets (critical fund freeze).

**Scenario C — Stale price corrupts `rsETHPrice`:** If `updateRSETHPrice()` is called while a stale price is active, `_getTotalEthInProtocol()` returns an incorrect TVL, and the stored `rsETHPrice` is permanently set to a wrong value. All subsequent deposits and withdrawals use this corrupted rate, causing systematic mis-accounting across all users.

**Impact classification: Medium** — Temporary freezing of funds / contract fails to deliver promised returns, triggered only when Chainlink oracle is in a degraded state.

---

### Likelihood Explanation

Chainlink oracle degradation (node abandonment, chain congestion, sequencer downtime on L2s) is a known and documented risk. The protocol is deployed on multiple chains including Arbitrum and other L2s where sequencer downtime is a realistic event. The vulnerability is passively triggered — no attacker action is required beyond the oracle entering a bad state. Any depositor transacting during that window is affected.

---

### Recommendation

Apply the same staleness and validity checks already present in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    require(price > 0, "ChainlinkPriceOracle: Invalid price");
    require(updatedAt != 0, "ChainlinkPriceOracle: Incomplete round");
    require(answeredInRound >= roundId, "ChainlinkPriceOracle: Stale price");

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

---

### Proof of Concept

1. Chainlink enters a degraded state for a supported LST asset (e.g., stETH/ETH feed). `answeredInRound < roundId` (stale) or `updatedAt == 0` (incomplete round).
2. Any user calls `LRTDepositPool.depositAsset(stETH, amount)`.
3. `LRTDepositPool.getRsETHAmountToMint(stETH, amount)` is called internally.
4. This calls `LRTOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)`.
5. `latestRoundData()` returns a stale/zero price; no revert occurs.
6. If price = 0: `rsethAmountToMint = (amount * 0) / rsETHPrice = 0`. The user's stETH is transferred in, 0 rsETH is minted. Funds are permanently locked.
7. If price is stale and inflated: the user receives more rsETH than deserved, diluting all existing rsETH holders.
8. If `updateRSETHPrice()` is called concurrently, the corrupted price is written to `rsETHPrice` storage, affecting all future protocol interactions. [7](#0-6) [8](#0-7)
