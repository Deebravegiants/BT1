### Title
Missing Fee Initialization for Newly Added Tokens Allows Zero-Fee Deposits - (File: contracts/pools/RSETHPool.sol)

### Summary
In `RSETHPool.sol`, when a new token is added via `addSupportedToken`, the per-token fee mapping `tokenFeeBps[token]` is never initialized and defaults to `0`. Any depositor who calls `deposit(token, amount, referralId)` before the admin separately calls `setTokenFeeBps` will pay zero protocol fees and receive more rsETH than the protocol intends, stealing fee yield from the protocol treasury.

### Finding Description
`RSETHPool.sol` maintains a per-token fee mapping `tokenFeeBps` distinct from the ETH-deposit `feeBps`. [1](#0-0) 

When a new token is added, `addSupportedToken` sets the oracle and bridge but never touches `tokenFeeBps[token]`: [2](#0-1) 

The fee calculation for token deposits reads directly from this uninitialized mapping: [3](#0-2) 

Because `tokenFeeBps[token]` is `0` by default, `fee = amount * 0 / 10_000 = 0`, and the depositor receives the full rsETH amount with no fee deduction. The admin must call `setTokenFeeBps` as a separate transaction to correct this: [4](#0-3) 

There is no on-chain enforcement requiring the fee to be set atomically with token addition. This is the direct analog to the fusion-swap finding: fee parameters are not enforced at the point of the relevant action, allowing users to bypass them.

### Impact Explanation
**High — Theft of unclaimed yield.** Protocol fee revenue for the newly added token is zero during the uninitialized window. All depositors in that window receive rsETH at a 0% fee rate instead of the intended rate. The fees that should have accumulated in `feeEarnedInToken[token]` and been withdrawn to the protocol treasury are permanently lost for those deposits.

### Likelihood Explanation
**Medium.** Adding a new supported token is a routine protocol operation gated by `TIMELOCK_ROLE`. Because the addition goes through a timelock, the pending transaction is publicly visible on-chain before execution. A watching depositor can prepare to front-run the token addition and immediately deposit at 0 fee the moment the timelock executes. The window persists until the admin issues a second transaction calling `setTokenFeeBps`. Even without front-running, any ordinary depositor who interacts with the pool before that second transaction pays no fee.

### Recommendation
Pass a `_feeBps` parameter to `addSupportedToken` and initialize `tokenFeeBps[token] = _feeBps` atomically within the same call, mirroring how `feeBps` is set during `initialize`. Alternatively, add a validation in `viewSwapRsETHAmountAndFee` that reverts if `tokenFeeBps[token]` has never been explicitly set (e.g., via a separate `isTokenFeeSet` mapping).

### Proof of Concept
1. Admin calls `addSupportedToken(token, oracle, bridge)` via the timelock. `tokenFeeBps[token]` remains `0`.
2. A depositor (who observed the pending timelock transaction) immediately calls `deposit(token, largeAmount, "")`.
3. Inside `deposit`, `viewSwapRsETHAmountAndFee(largeAmount, token)` executes:
   - `feeBpsForToken = tokenFeeBps[token]` → `0`
   - `fee = largeAmount * 0 / 10_000` → `0`
   - `amountAfterFee = largeAmount - 0` → `largeAmount`
   - Depositor receives rsETH calculated on the full `largeAmount` with no fee.
4. `feeEarnedInToken[token]` remains `0`; the protocol treasury receives nothing.
5. Admin later calls `setTokenFeeBps(token, intendedFeeBps)`, but all deposits made before this call have already bypassed the fee. [5](#0-4)

        external

        UtilLib.checkNonZeroAddress(token);

        UtilLib.checkNonZeroAddress(bridge);

        if (supportedTokenOracle[token] != address(0)) {
            revert AlreadySupportedToken();
        }
        if (tokenBridge[token] != address(0)) {
            revert AlreadySupportedToken();
        }
        if (IOracle(oracle).getRate() == 0) {
            revert UnsupportedOracle();
        }
        supportedTokenList.push(token);
        supportedTokenOracle[token] = oracle;
        tokenBridge[token] = bridge;

        emit AddSupportedToken(token, oracle, bridge);
