### Title
Missing Zero-Value Check on Incoming Rate in `lzReceive()` Allows Zero Rate to Be Stored — (File: `contracts/cross-chain/CrossChainRateReceiver.sol`)

### Summary
`CrossChainRateReceiver.lzReceive()` decodes and stores an incoming rate from LayerZero without validating that it is non-zero. A zero rate is a special invalid value: if stored, it causes all L2 pool `deposit()` calls to revert, temporarily freezing the deposit functionality for all users.

### Finding Description
In `CrossChainRateReceiver.lzReceive()`, the decoded rate is stored directly with no zero-value guard:

```solidity
uint256 _rate = abi.decode(_payload, (uint256));
rate = _rate;          // ← no check: require(_rate != 0)
lastUpdated = block.timestamp;
``` [1](#0-0) 

The `rate` storage variable is the sole source of truth returned by `getRate()`, which every L2 pool contract calls inside its swap/deposit path. [2](#0-1) 

The concrete rate provider, `RSETHMultiChainRateProvider.getLatestRate()`, reads `rsETHPrice` directly from `LRTOracle`:

```solidity
function getLatestRate() public view override returns (uint256) {
    return ILRTOracle(rsETHPriceOracle).rsETHPrice();
}
``` [3](#0-2) 

`rsETHPrice` is a storage variable that starts at `0` (Solidity default) and is only set after `updateRSETHPrice()` is called. The `updateRate()` function on the provider is `external payable` with no access control, so any caller can invoke it: [4](#0-3) 

If `updateRate()` is called before `updateRSETHPrice()` has ever been called on L1, the provider encodes `rsETHPrice = 0` and sends it via LayerZero. The receiver stores `rate = 0` with no rejection.

Once `rate = 0` is stored, every pool's deposit path fails:

- **Older pools** (`RSETHPoolV2`, `RSETHPoolV2ExternalBridge`): `viewSwapRsETHAmountAndFee()` performs `amountAfterFee * 1e18 / rsETHToETHrate` — a division-by-zero panic. [5](#0-4) 

- **Newer pools** (`RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`): an explicit guard reverts with `UnsupportedOracle`. [6](#0-5) 

By contrast, the Merkle distributor contracts in the same codebase correctly guard against zero roots before proceeding: [7](#0-6) 

The rate receiver has no equivalent protection.

### Impact Explanation
If a zero rate is stored in `CrossChainRateReceiver`, every call to `deposit()` on the associated L2 pool reverts. Users cannot exchange ETH or supported LSTs for rsETH/wrsETH on L2. This is a temporary freeze of the deposit functionality — the contract fails to deliver its promised service, but no funds already held by users are lost. Impact: **Low** ("Contract fails to deliver promised returns, but doesn't lose value").

### Likelihood Explanation
The attack window is the period between L2 pool deployment and the first successful call to `updateRSETHPrice()` on L1. During this window, `rsETHPrice == 0` and `updateRate()` is callable by any unprivileged address. After initialization the price is always ≥ 1 ether, so the window closes permanently. Likelihood: **Low**, but the window is real and exploitable by any external caller during deployment.

### Recommendation
Add a non-zero guard in `lzReceive()` before storing the rate:

```solidity
uint256 _rate = abi.decode(_payload, (uint256));
require(_rate != 0, "CrossChainRateReceiver: zero rate");
rate = _rate;
```

Optionally, add the same guard in `CrossChainRateProvider.updateRate()` to prevent a zero rate from ever being dispatched.

### Proof of Concept
1. Protocol is deployed on L1 and L2; `updateRSETHPrice()` has not yet been called, so `LRTOracle.rsETHPrice == 0`.
2. Attacker calls `RSETHMultiChainRateProvider.updateRate{value: fee}()`.
3. Provider encodes `rsETHPrice = 0` and sends it to `RSETHRateReceiver` via the LayerZero endpoint.
4. `CrossChainRateReceiver.lzReceive()` executes: `rate = 0` is stored with no revert.
5. Any user calling `deposit()` on the L2 pool triggers `viewSwapRsETHAmountAndFee()`, which either panics (division by zero) or reverts with `UnsupportedOracle`.
6. The L2 pool is effectively frozen for deposits until an admin updates the oracle or a valid rate message is received.
