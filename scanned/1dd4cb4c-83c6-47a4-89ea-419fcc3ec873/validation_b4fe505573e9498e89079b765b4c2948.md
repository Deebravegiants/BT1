### Title
Missing Upper Bound on `vestingStartTimestamp` Allows Owner to Indefinitely Delay Token Claims - (File: contracts/KERNEL/KernelTop100MerkleDistributor.sol)

### Summary
`KernelTop100MerkleDistributor.setVestingStartTimestamp` enforces only a non-zero and future-timestamp check, but places no upper bound on how far in the future the vesting start can be set. The owner can push `vestingStartTimestamp` arbitrarily far into the future (e.g., 100 years), preventing all merkle-eligible users from claiming any KERNEL tokens for an indefinite period.

### Finding Description
`setVestingStartTimestamp` (lines 438–453) accepts any `_vestingStartTimestamp > block.timestamp` without restriction:

```solidity
function setVestingStartTimestamp(uint256 _vestingStartTimestamp) external onlyOwner {
    if (_vestingStartTimestamp == 0) {
        revert ZeroValueProvided();
    }
    if (vestingStartTimestamp > 0 && block.timestamp >= vestingStartTimestamp) {
        revert VestingAlreadyStarted();
    }
    if (_vestingStartTimestamp <= block.timestamp) {
        revert VestingStartInThePast();
    }
    vestingStartTimestamp = _vestingStartTimestamp;
    emit VestingStartTimestampSet(vestingStartTimestamp);
}
```

The guard `VestingAlreadyStarted` only fires once `block.timestamp >= vestingStartTimestamp`, meaning the owner can freely update the timestamp to any future value **before** vesting begins. Setting it to `block.timestamp + 100 years` is accepted without revert.

`_getUnclaimedVestedAmount` (lines 255–256) returns 0 whenever `currentTime <= vestingStartTimestamp`:

```solidity
if (currentTime <= startTime || currentTime <= vestingStartTimestamp) {
    return 0;
}
```

Both `claim` and `claimAndStake` call `_getUnclaimedVestedAmount` and revert with `NoTokensToClaim` when the result is 0. Therefore, no user can claim any KERNEL tokens until the vesting start timestamp is reached.

### Impact Explanation
**Medium — Permanent freezing of unclaimed yield.**

All KERNEL tokens allocated to users via the merkle tree are locked in the contract and cannot be claimed. The owner can keep the vesting start perpetually in the future by calling `setVestingStartTimestamp` again before the current timestamp is reached (since `VestingAlreadyStarted` only triggers once vesting has begun). In practice, a single call setting the timestamp 100 years ahead achieves the same effect without further intervention. Users lose access to their entitled KERNEL token distributions for an indefinitely long period.

### Likelihood Explanation
The owner of `KernelTop100MerkleDistributor` is a single EOA or multisig with no on-chain time-lock enforced by this contract. The function is callable at any time before vesting starts. The centralization risk is identical in structure to the referenced report: a single privileged call with no parameter bound is sufficient to freeze all user claims.

### Recommendation
Introduce a maximum allowed vesting start delay from the current timestamp, for example:

```solidity
uint256 public constant MAX_VESTING_START_DELAY = 180 days;

function setVestingStartTimestamp(uint256 _vestingStartTimestamp) external onlyOwner {
    if (_vestingStartTimestamp == 0) revert ZeroValueProvided();
    if (vestingStartTimestamp > 0 && block.timestamp >= vestingStartTimestamp) revert VestingAlreadyStarted();
    if (_vestingStartTimestamp <= block.timestamp) revert VestingStartInThePast();
    if (_vestingStartTimestamp > block.timestamp + MAX_VESTING_START_DELAY) revert VestingStartTooFarInFuture();
    vestingStartTimestamp = _vestingStartTimestamp;
    emit VestingStartTimestampSet(vestingStartTimestamp);
}
```

### Proof of Concept
1. Owner deploys `KernelTop100MerkleDistributor` with `vestingStartTimestamp = block.timestamp + 7 days`.
2. Before 7 days elapse, owner calls `setVestingStartTimestamp(block.timestamp + 36500 days)` (100 years).
3. The call succeeds: all three existing checks pass (non-zero, not yet started, in the future).
4. Any user calls `claim(amount, proof)`. Inside `_getUnclaimedVestedAmount`, `currentTime <= vestingStartTimestamp` evaluates to true, returning 0.
5. `claim` reverts with `NoTokensToClaim`. All KERNEL tokens remain locked in the contract for 100 years. [1](#0-0) [2](#0-1) [3](#0-2)
