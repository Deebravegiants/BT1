### Title
`bridgeAssetsViaNativeBridge()` Missing Zero-Amount Guard Allows Silent No-Op Bridge Call — (`contracts/pools/RSETHPool.sol`)

### Summary

`RSETHPool.bridgeAssetsViaNativeBridge()` does not check that `ethBalanceMinusFees > 0` before forwarding the value to `ArbitrumMessenger.sendETHToL1ViaBridge()`. When `feeEarnedInETH == address(this).balance`, the entire call chain completes without reverting and emits `BridgedETHToL1ViaNativeBridge` with `amount = 0`, while no ETH is actually transferred to L1.

---

### Finding Description

**Call chain:**

1. `BRIDGER_ROLE` calls `bridgeAssetsViaNativeBridge()`.
2. `getETHBalanceMinusFees()` returns `address(this).balance - feeEarnedInETH`. [1](#0-0) 
3. If `feeEarnedInETH == address(this).balance`, this returns `0`. There is **no zero-check** here. [2](#0-1) 
4. `IL2Messenger(messenger).sendETHToL1ViaBridge{ value: 0 }(l2Bridge, l1VaultETHForL2Chain, 0)` is called.
5. Inside `ArbitrumMessenger.sendETHToL1ViaBridge`, the only guard is `if (msg.value != value) revert MismatchedMsgValue()`. With both sides being `0`, this passes. [3](#0-2) 
6. `IArbitrumMessenger(l2bridge).withdrawEth{ value: 0 }(target)` is called. The Arbitrum ArbSys precompile does not revert on a zero-value withdrawal. [4](#0-3) 
7. `BridgedETHToL1ViaNativeBridge(l1VaultETHForL2Chain, 0)` is emitted. [5](#0-4) 

**Contrast with `bridgeTokens()`**, which correctly guards against this:
```solidity
if (balance == 0) {
    revert ZeroBridgeAmount();
}
``` [6](#0-5) 

**How `feeEarnedInETH == address(this).balance` is reached:** Every ETH `deposit()` call adds `fee` to `feeEarnedInETH`. If all non-fee ETH has already been bridged (via `bridgeAssets` or `moveAssetsForBridging`) and only fee ETH remains, or if the pool has received only small deposits whose entire value rounds to fees, this state is reachable without any privileged action. [7](#0-6) 

---

### Impact Explanation

L1Vault receives no ETH while the L2 pool emits a `BridgedETHToL1ViaNativeBridge` success event with `amount = 0`. Off-chain monitoring, accounting systems, and operators relying on this event will incorrectly record a successful bridge transfer. No principal is lost (fees remain in the contract), matching the scoped impact: **Low — contract fails to deliver promised returns, but doesn't lose value**.

---

### Likelihood Explanation

The state is reachable through normal operation (deposits accumulate fees; non-fee ETH is bridged separately). The `BRIDGER_ROLE` calling `bridgeAssetsViaNativeBridge()` when the pool holds only fee ETH is a realistic operational scenario, not an adversarial one. No special permissions beyond `BRIDGER_ROLE` are needed.

---

### Recommendation

Add a zero-amount guard at the start of `bridgeAssetsViaNativeBridge()`, consistent with the pattern already used in `bridgeTokens()`:

```solidity
function bridgeAssetsViaNativeBridge() external nonReentrant onlyRole(BRIDGER_ROLE) {
    UtilLib.checkNonZeroAddress(l2Bridge);
    UtilLib.checkNonZeroAddress(messenger);
    UtilLib.checkNonZeroAddress(l1VaultETHForL2Chain);

    uint256 ethBalanceMinusFees = getETHBalanceMinusFees();
+   if (ethBalanceMinusFees == 0) revert ZeroBridgeAmount();

    IL2Messenger(messenger).sendETHToL1ViaBridge{ value: ethBalanceMinusFees }(
        l2Bridge, l1VaultETHForL2Chain, ethBalanceMinusFees
    );

    emit BridgedETHToL1ViaNativeBridge(l1VaultETHForL2Chain, ethBalanceMinusFees);
}
```

// Setup: feeEarnedInETH == address(rsETHPool).balance

// Assert:
// 1. No revert occurred
// 2. mockBridge received withdrawEth call with value = 0
