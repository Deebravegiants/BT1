### Title
Missing Zero-Address Validation on `target` in `UnichainMessenger.sendETHToL1ViaBridge` - ([File: contracts/bridges/UnichainMessenger.sol])

### Summary
`UnichainMessenger.sendETHToL1ViaBridge` accepts a caller-supplied `target` address with no zero-address guard, unlike the analogous `LineaMessenger` implementation. Because the function is public and has no access control, any caller can invoke it with `target = address(0)`, forwarding ETH to the Unichain bridge destined for the zero address on L1.

### Finding Description

`UnichainMessenger.sendETHToL1ViaBridge` performs only one input check — that `msg.value == value` — and immediately forwards the call to the bridge: [1](#0-0) 

By contrast, `LineaMessenger.sendETHToL1ViaBridge` explicitly validates both `l2bridge` and `target` before proceeding: [2](#0-1) 

`UnichainMessenger` has no role-based access control — it is a fully public helper contract. Any EOA or contract can call `sendETHToL1ViaBridge` with `target = address(0)` and a non-zero `value`, causing `bridgeETHTo(address(0), 200_000, bytes(""))` to be submitted to the Unichain bridge. [3](#0-2) 

### Impact Explanation

The contract's stated purpose is to bridge ETH to a valid L1 recipient. When `target = address(0)` is passed, the bridge message targets the zero address on L1. Depending on the Unichain bridge's L1 finalization logic, the ETH is either locked in the bridge contract or effectively burned at `address(0)`. In either case, the contract fails to deliver its promised return (ETH to a valid L1 vault), matching the **Low** scope: *contract fails to deliver promised returns, but doesn't lose value* from the protocol's perspective (the pool contracts' own ETH is protected by `BRIDGER_ROLE` access control and their own `l1VaultETHForL2Chain` non-zero validation). [4](#0-3) [5](#0-4) 

### Likelihood Explanation

The function is public with no access control. Any caller who sends ETH directly to `UnichainMessenger` (bypassing the pool contracts) can trigger this path. The pool contracts themselves always pass the validated `l1VaultETHForL2Chain` (which is enforced non-zero at setter time), so protocol-held ETH is not at risk through the normal bridging flow. The risk is to direct callers of the messenger contract. [6](#0-5) 

### Recommendation

Add zero-address validation for both `l2bridge` and `target` at the top of `sendETHToL1ViaBridge`, mirroring `LineaMessenger`:

```solidity
function sendETHToL1ViaBridge(address l2bridge, address target, uint256 value) external payable nonReentrant {
    UtilLib.checkNonZeroAddress(l2bridge);
    UtilLib.checkNonZeroAddress(target);
    if (msg.value != value) revert MismatchedMsgValue();
    IUnichainMessenger(l2bridge).bridgeETHTo{ value: value }(target, DEFAULT_GAS_LIMIT, bytes(""));
}
```
