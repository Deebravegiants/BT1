### Title
Missing Chainlink Staleness Check Enables Block-Stuffing-Assisted rsETH Price Suppression — (`contracts/oracles/ChainlinkPriceOracle.sol`)

### Summary
`ChainlinkPriceOracle.getAssetPrice()` accepts any price returned by `latestRoundData()` without validating freshness. Combined with the public `updateRSETHPrice()` entrypoint, an attacker can stuff blocks to delay Chainlink heartbeat updates on a rising LST feed, then call `updateRSETHPrice()` to lock in a stale-low `rsETHPrice`, allowing subsequent deposits to mint excess rsETH.

### Finding Description
`ChainlinkPriceOracle.getAssetPrice()` reads `latestRoundData()` and returns the price without checking `updatedAt` against a maximum staleness threshold or verifying `answeredInRound >= roundId`:

```solidity
// ChainlinkPriceOracle.sol:52-54
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

`updateRSETHPrice()` is public and permissionless. When called, `_updateRsETHPrice()` invokes `_getTotalEthInProtocol()`, which multiplies each asset's total deposit amount by the (potentially stale) Chainlink price. The resulting `newRsETHPrice` is written to storage as `rsETHPrice`.

The downside-protection guard at `LRTOracle.sol:272-274` is gated by `pricePercentageLimit > 0`, but `pricePercentageLimit` is never set in `initialize()` and defaults to `0`, so the auto-pause is never triggered regardless of how far the price drops.

Subsequent calls to `getRsETHAmountToMint()` divide by the suppressed `rsETHPrice`, minting more rsETH per unit of deposited asset than the true TVL ratio warrants, diluting existing holders.

### Impact Explanation
Depositors who transact after the price suppression receive excess rsETH relative to the true protocol TVL / rsETH supply ratio. Existing rsETH holders are diluted — their redemption value per token decreases. The impact is bounded by the magnitude of the stale price deviation and the deposit volume during the suppression window.

**Scoped impact: Low — Block stuffing enabling excess rsETH minting at artificially suppressed price.**

### Likelihood Explanation
Block stuffing on Ethereum mainnet is expensive (requires filling entire blocks at prevailing gas prices for the duration of a Chainlink heartbeat, typically 1 hour for stETH/ETH). However:
- The missing staleness check is a code-level defect independent of block stuffing — any Chainlink feed delay (network congestion, keeper failure) also triggers it.
- `updateRSETHPrice()` is callable by anyone immediately after the stale price is in the feed.
- `pricePercentageLimit` defaults to 0, removing the only on-chain circuit breaker.

Likelihood is low due to block stuffing cost, but the staleness check absence is unconditional.

### Recommendation
1. Add a staleness check in `ChainlinkPriceOracle.getAssetPrice()`:
```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();
require(answeredInRound >= roundId, "Stale round");
require(block.timestamp - updatedAt <= MAX_STALENESS, "Stale price");
```
2. Set a non-zero `pricePercentageLimit` during initialization to ensure the auto-pause circuit breaker is active from deployment.
