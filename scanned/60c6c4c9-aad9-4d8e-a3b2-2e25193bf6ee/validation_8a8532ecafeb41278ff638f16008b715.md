### Title
Fee Change Retroactively Applied to Already-Vesting Token Allocations - (File: contracts/KERNEL/KernelTop100MerkleDistributor.sol)

### Summary
`KernelTop100MerkleDistributor.setFeeInBPS()` allows the owner to change the protocol fee at any time. Because `feeInBPS` is a global variable read at claim time rather than recorded per-user at vesting start, any fee increase immediately applies to tokens that users are already vesting — tokens they were entitled to under the original fee terms. The same pattern exists in `KernelMerkleDistributor` and `MerkleDistributor`.

### Finding Description
Users receive a fixed merkle-proven allocation and vest it linearly over 30 days (`VESTING_DURATION`). When a user calls `claim()` or `claimAndStake()`, the fee is computed on-the-fly using the current global `feeInBPS`:

```solidity
// KernelTop100MerkleDistributor.sol line 328-329
uint256 fee = (claimableAmount * feeInBPS) / FEE_DENOMINATOR;
uint256 amountToSend = claimableAmount - fee;
``` [1](#0-0) 

The owner can raise `feeInBPS` at any time via `setFeeInBPS()`, with no timelock:

```solidity
// KernelTop100MerkleDistributor.sol line 426-432
function setFeeInBPS(uint256 _feeInBPS) external onlyOwner {
    if (_feeInBPS > MAX_FEE_IN_BPS) {
        revert InvalidFeeInBPS();
    }
    feeInBPS = _feeInBPS;
    emit FeeInBPSUpdated(feeInBPS);
}
``` [2](#0-1) 

No per-user fee snapshot is recorded at vesting start. The `UserClaim` struct only stores `lastClaimTimestamp` and `amountClaimed`:

```solidity
struct UserClaim {
    uint256 lastClaimTimestamp;
    uint256 amountClaimed;
}
``` [3](#0-2) 

The identical pattern exists in `KernelMerkleDistributor` (line 338) and `MerkleDistributor` (line 138), where `feeInBPS` is also read globally at claim time with no per-user snapshot. [4](#0-3) [5](#0-4) 

### Impact Explanation
Users who began vesting under a 0% (or low) fee will have a higher fee silently applied to all remaining vested-but-unclaimed tokens. The difference between the old and new fee is transferred to `protocolTreasury` instead of the user. This constitutes theft of unclaimed yield. With `MAX_FEE_IN_BPS = 1000` (10%), up to 10% of a user's entire vesting allocation can be redirected to the treasury without their consent.

**Impact: High — Theft of unclaimed yield.**

### Likelihood Explanation
The owner can call `setFeeInBPS()` at any time with no timelock or delay. All users currently mid-vesting are immediately affected. The function is a standard admin setter with no guard against retroactive application. The 30-day vesting window in `KernelTop100MerkleDistributor` means there is always a window during which active vesting positions exist and are exposed.

### Recommendation
Record the `feeInBPS` value per-user at the time their vesting begins (or at first claim), and apply that snapshot fee for all subsequent claims by that user. Alternatively, apply fee changes only to new vesting positions created after the change, not to existing ones.

```solidity
struct UserClaim {
    uint256 lastClaimTimestamp;
    uint256 amountClaimed;
    uint256 feeInBPSAtStart; // snapshot fee at vesting start
}
```

### Proof of Concept
1. Owner deploys `KernelTop100MerkleDistributor` with `feeInBPS = 0`.
2. Alice has a merkle-proven allocation of 1,000 KERNEL tokens. Vesting starts.
3. After 15 days, Alice has 500 KERNEL vested but has not yet claimed.
4. Owner calls `setFeeInBPS(1000)` (10%).
5. Alice calls `claim()`. Her `claimableAmount = 500`. Fee = 50 KERNEL sent to treasury. Alice receives 450 instead of 500.
6. Alice loses 50 KERNEL (10% of her mid-vesting entitlement) that she never agreed to pay. [6](#0-5)

    function claim(uint256 amount, bytes32[] calldata merkleProof) external nonReentrant whenNotPaused {
        address user = msg.sender;

        // Verify merkle proof and update user claim data
        _verifyClaimProof(user, amount, merkleProof);

        // Get claimable amount
        uint256 claimableAmount = _getUnclaimedVestedAmount(user, amount);

        if (claimableAmount == 0) {
            revert NoTokensToClaim();
        }

        // Update user claim data
        userClaims[user].lastClaimTimestamp = block.timestamp;
        userClaims[user].amountClaimed += claimableAmount;

        // Calculate fee
        uint256 fee = (claimableAmount * feeInBPS) / FEE_DENOMINATOR;
        uint256 amountToSend = claimableAmount - fee;

        // Transfer tokens
        if (fee > 0) {
            kernel.safeTransfer(protocolTreasury, fee);
        }
        kernel.safeTransfer(user, amountToSend);

        emit Claimed(user, amountToSend);
```
