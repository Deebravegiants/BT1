### Title
Composite `RSETHPriceFeed.latestRoundData()` Returns `updatedAt` Solely from the ETH/USD Feed, Masking Staleness of the rsETH/ETH Component - (File: contracts/oracles/RSETHPriceFeed.sol)

---

### Summary

`RSETHPriceFeed` is a Chainlink-compatible composite price feed that multiplies an ETH/USD Chainlink price by the rsETH/ETH rate stored in `LRTOracle`. Its `latestRoundData()` propagates `updatedAt` exclusively from the ETH/USD Chainlink feed, while the `answer` it returns is a product of both the ETH/USD price and `RS_ETH_ORACLE.rsETHPrice()`. Because `LRTOracle.rsETHPrice` is a manually-triggered state variable (updated via `updateRSETHPrice()`), it can lag far behind the ETH/USD feed. Any consumer that checks `updatedAt` to detect staleness will see the ETH/USD heartbeat timestamp and conclude the price is fresh, even when the rsETH/ETH component is arbitrarily old.

---

### Finding Description

`RSETHPriceFeed.latestRoundData()` combines two independent price sources:

1. **ETH/USD Chainlink feed** (`ETH_TO_USD`) — updated automatically on a ~1-hour heartbeat or 0.5% deviation threshold.
2. **rsETH/ETH oracle** (`RS_ETH_ORACLE`, i.e., `LRTOracle.rsETHPrice`) — a state variable updated only when `updateRSETHPrice()` is called by a keeper or any public caller.

The implementation:

```solidity
// contracts/oracles/RSETHPriceFeed.sol  lines 63-70
function latestRoundData()
    external view
    returns (uint80 roundId, int256 answer, uint256 startedAt, uint256 updatedAt, uint80 answeredInRound)
{
    (roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.latestRoundData();
    answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
}
```

`updatedAt` is taken verbatim from `ETH_TO_USD.latestRoundData()` and is never adjusted to reflect when `RS_ETH_ORACLE.rsETHPrice()` was last written. The `answer` is a composite of both sources, but the staleness metadata only covers one of them.

`LRTOracle.rsETHPrice` is a plain storage variable:

```solidity
// contracts/LRTOracle.sol  line 28
uint256 public override rsETHPrice;
```

It is written only inside `_updateRsETHPrice()`, which is called by `updateRSETHPrice()` (public, but not self-triggering) or `updateRSETHPriceAsManager()` (manager-only). [1](#0-0) 

There is no on-chain record of when `rsETHPrice` was last set, so `RSETHPriceFeed` has no way to incorporate that timestamp into `updatedAt`. [2](#0-1) 

---

### Impact Explanation

Any external protocol (lending market, DEX, vault) that consumes `RSETHPriceFeed` and applies a standard Chainlink staleness guard of the form:

```solidity
require(block.timestamp - updatedAt <= MAX_STALENESS, "stale price");
```

will pass the check as long as the ETH/USD Chainlink feed is fresh — regardless of how old `LRTOracle.rsETHPrice` is. If the rsETH/ETH rate has drifted (e.g., due to a slashing event, EigenLayer accounting change, or keeper downtime), the composite answer will be wrong while appearing fresh. This causes the contract to fail to deliver the accurate, timely price it promises as a Chainlink-compatible feed.

**Impact: Low — Contract fails to deliver promised returns, but doesn't lose value (within the LRT-rsETH protocol itself).** [2](#0-1) 

---

### Likelihood Explanation

`updateRSETHPrice()` is public but requires an external call; it is not triggered automatically on-chain. [3](#0-2) 

Keeper downtime, network congestion, or a price-threshold revert (the `PriceAboveDailyThreshold` guard) can all delay updates to `rsETHPrice` for hours or days, while the ETH/USD Chainlink feed continues to update normally. During that window every call to `RSETHPriceFeed.latestRoundData()` returns a fresh-looking `updatedAt` paired with a stale `answer`. This is a realistic operational scenario. [4](#0-3) 

---

### Recommendation

Return the **minimum** of the two relevant timestamps so that consumers see the true staleness of the composite price:

```solidity
function latestRoundData()
    external view
    returns (uint80 roundId, int256 answer, uint256 startedAt, uint256 updatedAt, uint80 answeredInRound)
{
    (roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.latestRoundData();
    uint256 rsETHUpdatedAt = RS_ETH_ORACLE.rsETHUpdatedAt(); // expose this from LRTOracle
    if (rsETHUpdatedAt < updatedAt) updatedAt = rsETHUpdatedAt;
    answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
}
```

This requires `LRTOracle` to store and expose the timestamp of the last `rsETHPrice` write alongside the price itself.

---

### Proof of Concept

1. At `T=0`, `updateRSETHPrice()` is called; `LRTOracle.rsETHPrice` is set to `1.05e18` (rsETH/ETH).
2. At `T=2h`, the ETH/USD Chainlink feed updates normally (heartbeat). `ETH_TO_USD.latestRoundData()` returns `updatedAt = T+2h`.
3. At `T=25h`, the keeper bot is down; `rsETHPrice` has not been updated. A slashing event has reduced the true rsETH/ETH rate to `1.00e18`.
4. A lending protocol calls `RSETHPriceFeed.latestRoundData()`. It receives:
   - `updatedAt = T+24h` (from the most recent ETH/USD heartbeat — appears fresh)
   - `answer = 1.05e18 * ETH_USD_price / 1e18` (stale rsETH component)
5. The lending protocol's staleness guard passes (e.g., `MAX_STALENESS = 2h`, `block.timestamp - updatedAt = 1h`).
6. The protocol prices rsETH 5% higher than its true value, allowing over-borrowing against rsETH collateral. [5](#0-4) [6](#0-5) [7](#0-6)
