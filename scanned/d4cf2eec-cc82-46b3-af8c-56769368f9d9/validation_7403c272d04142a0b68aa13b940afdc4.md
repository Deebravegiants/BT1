### Title
Missing time-based Chainlink staleness check enables deposits at stale collateral prices — (File: `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol`)

---

### Summary
`ChainlinkOracleForRSETHPoolCollateral.getRate()` omits a `block.timestamp - updatedAt > heartbeat` staleness guard. When a Chainlink feed stops updating within its heartbeat window, any L2 pool depositor can swap a supported collateral token (e.g., wstETH) for rsETH at an inflated stale rate, extracting value from existing rsETH holders.

---

### Finding Description
`getRate()` calls `latestRoundData()` and validates only three conditions:

```solidity
if (answeredInRound < roundID) revert StalePrice();   // round-based
if (timestamp == 0) revert IncompleteRound();          // incomplete round
if (ethPrice <= 0) revert InvalidPrice();              // negative price
``` [1](#0-0) 

There is **no** check of the form `block.timestamp - timestamp > heartbeat`. Chainlink feeds update on either a price-deviation threshold or a heartbeat interval (e.g., 24 hours for many LST/ETH feeds). If the price has not deviated enough to trigger a deviation-based update, the feed can remain at an old value for the full heartbeat window while `answeredInRound == roundID` (the last round was answered, so the round-based check passes). During this window the pool uses the stale collateral price to compute `rsETHAmount`:

```solidity
// RSETHPoolV3 — token deposit path
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate(); // stale value
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [2](#0-1) 

The same pattern appears in `RSETHPool.viewSwapRsETHAmountAndFee` and `RSETHPoolNoWrapper.viewSwapRsETHAmountAndFee`. [3](#0-2) [4](#0-3) 

A depositor who observes that the real collateral price has fallen below the stale oracle price can deposit the collateral and receive more rsETH than the collateral is worth.

---

### Impact Explanation
**High — Theft of unclaimed yield / dilution of existing rsETH holders.**

In `RSETHPoolV3`, `wrsETH.mint()` creates new wrapped rsETH backed by the deposited collateral. [5](#0-4) 

If the collateral is overvalued by the stale oracle, the newly minted rsETH is not fully backed, diluting all existing rsETH/wrsETH holders proportionally. In `RSETHPool` and `RSETHPoolNoWrapper`, the pool's pre-minted rsETH inventory is depleted faster than the ETH value received, creating a shortfall that must be covered by the protocol.

---

### Likelihood Explanation
**Medium.** Chainlink heartbeats for LST/ETH feeds can be 24 hours. During periods of market stress — exactly when prices move most — network congestion can delay oracle updates. An attacker monitoring the feed can identify the staleness window and act within it. No privileged access is required; any address can call `deposit()` on the L2 pools. [6](#0-5) 

---

### Recommendation
Add a configurable per-feed heartbeat check inside `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
uint256 public immutable heartbeat; // set in constructor per feed, e.g. 24 hours

function getRate() public view returns (uint256) {
    (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();

    if (answeredInRound < roundID) revert StalePrice();
    if (timestamp == 0) revert IncompleteRound();
    if (block.timestamp - timestamp > heartbeat) revert StalePrice(); // ADD
    if (ethPrice <= 0) revert InvalidPrice();

    return uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
}
``` [7](#0-6) 

---

### Proof of Concept
1. `ChainlinkOracleForRSETHPoolCollateral` is configured for a wstETH/ETH feed with a 24-hour heartbeat.
2. At `T = 0` the feed records `wstETH/ETH = 1.10`; `answeredInRound == roundID`.
3. At `T = 20 h` the real market price drops to `wstETH/ETH = 1.00` (depeg event), but the price deviation is below the Chainlink threshold so no new round is published. The round-based check still passes.
4. Attacker calls `RSETHPoolV3.deposit(wstETH, 1000e18, "")`.
5. `viewSwapRsETHAmountAndFee` computes `tokenToETHRate = 1.10e18` (stale), `rsETHToETHrate = 1.05e18` (current rsETH price).
6. Attacker receives `1000 × 1.10 / 1.05 ≈ 1047 wrsETH` instead of the fair `1000 × 1.00 / 1.05 ≈ 952 wrsETH`.
7. Attacker redeems wrsETH for ~1047 ETH worth of value having deposited only ~1000 ETH worth of wstETH — approximately 47 ETH extracted from existing rsETH holders with no privileged access required.
