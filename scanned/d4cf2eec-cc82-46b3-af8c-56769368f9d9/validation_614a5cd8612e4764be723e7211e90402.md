### Title
Missing Chainlink Staleness Checks in `ChainlinkPriceOracle.getAssetPrice()` Allows Stale Prices to Drive rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards all staleness-related return values (`roundId`, `updatedAt`, `answeredInRound`). A stale Chainlink price flows directly into rsETH minting calculations in `LRTDepositPool`, allowing depositors to receive more rsETH than the true ETH value of their deposit, diluting existing holders.

### Finding Description
In `ChainlinkPriceOracle.getAssetPrice()`, the call to `latestRoundData()` captures only the `price` field:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

The three discarded fields — `roundId`, `updatedAt`, and `answeredInRound` — are precisely the ones needed to detect stale data. No check is made for:
- `answeredInRound >= roundId` (detects stale rounds)
- `updatedAt != 0` (detects incomplete rounds)
- `price > 0` (detects invalid/zero prices)

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` in the same repository performs all three checks, demonstrating the developers were aware of the pattern but failed to apply it to the core oracle:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The stale price propagates through the following call chain:

1. `ChainlinkPriceOracle.getAssetPrice(asset)` → returns stale inflated price
2. `LRTOracle.getAssetPrice(asset)` (line 157) → passes it through
3. `LRTDepositPool.getRsETHAmountToMint()` (line 520):
   ```solidity
   rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
   ```
4. `LRTDepositPool.depositAsset()` / `depositETH()` → mints rsETH to depositor

Because `lrtOracle.getAssetPrice(asset)` is a live call while `lrtOracle.rsETHPrice()` is a stored value updated separately via `updateRSETHPrice()`, a stale inflated Chainlink price inflates the numerator without a corresponding inflation of the denominator, causing the depositor to receive more rsETH than the true ETH value of their deposit.

### Impact Explanation
When a Chainlink feed goes stale with an inflated price, any depositor calling `depositAsset()` or `depositETH()` receives excess rsETH. This excess rsETH represents a claim on protocol TVL that was not backed by a corresponding deposit, diluting the ETH-per-rsETH ratio for all existing holders. This constitutes theft of unclaimed yield from existing rsETH holders.

**Impact: High** — Theft of unclaimed yield from existing rsETH holders.

### Likelihood Explanation
Chainlink feeds do go stale: during network congestion, oracle node outages, or circuit-breaker events, `updatedAt` can lag significantly. The Chainlink documentation explicitly warns about `answeredInRound < roundId` indicating stale data. No privileged access is required — any unprivileged depositor can exploit the window while the feed is stale.

**Likelihood: Medium** — Requires a Chainlink feed to go stale, which is a known and documented failure mode, not a theoretical one.

### Recommendation
Apply the same staleness checks already present in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    require(answeredInRound >= roundId, "Stale price");
    require(updatedAt != 0, "Incomplete round");
    require(price > 0, "Invalid price");

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```
