### Title
Chainlink Price Feed Not Validated for Staleness in `ChainlinkPriceOracle.getAssetPrice` - (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards all staleness-related return values (`roundId`, `updatedAt`, `answeredInRound`). A stale Chainlink price for any supported LST asset flows directly into rsETH minting calculations, allowing depositors to receive more rsETH than they are entitled to (diluting existing holders), or causing the protocol to be incorrectly paused.

---

### Finding Description

In `contracts/oracles/ChainlinkPriceOracle.sol`, `getAssetPrice()` calls `priceFeed.latestRoundData()` but only destructures the `price` field, ignoring all staleness indicators:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52-54
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

No check is made on `answeredInRound >= roundId` (round completeness) or `updatedAt > 0` (round not incomplete). A stale price is silently accepted as valid.

This price propagates through the following call chain:

1. `LRTOracle.getAssetPrice(asset)` → `IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset)` [1](#0-0) 
2. `LRTDepositPool.getRsETHAmountToMint()` uses `lrtOracle.getAssetPrice(asset)` live at deposit time: [2](#0-1) 
3. `LRTOracle._getTotalEthInProtocol()` uses `getAssetPrice(asset)` for every supported LST to compute total protocol TVL: [3](#0-2) 
4. `_updateRsETHPrice()` uses that TVL to compute and store `rsETHPrice`, and triggers the downside-protection pause if the computed price drops too far: [4](#0-3) 

The contrast with `ChainlinkOracleForRSETHPoolCollateral.sol` is instructive — that contract, used for pool collateral pricing, performs all three required staleness checks: [5](#0-4) 

The same discipline was not applied to `ChainlinkPriceOracle`, which is the oracle used for all LST assets in the core deposit/mint path.

---

### Impact Explanation

**Primary impact — rsETH over-minting (dilution of existing holders):**

`getRsETHAmountToMint` computes:
```
rsethAmountToMint = (depositAmount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()
``` [2](#0-1) 

`lrtOracle.getAssetPrice(asset)` is called live and reads the stale Chainlink price. If the stale price is higher than the current actual price (e.g., stETH/ETH feed is stale at 1.05 while the real rate has dropped to 1.02), a depositor receives more rsETH than their deposit is worth. This dilutes all existing rsETH holders' share of protocol TVL — a theft of unclaimed yield from existing holders.

**Secondary impact — incorrect protocol pause (temporary fund freeze):**

`updateRSETHPrice()` is a public, permissionless function. [6](#0-5)  If a stale low price causes `newRsETHPrice` to fall below `highestRsethPrice` by more than `pricePercentageLimit`, the deposit pool and withdrawal manager are both paused, freezing all user funds temporarily. [7](#0-6) 

---

### Likelihood Explanation

Chainlink feeds can go stale during periods of network congestion, oracle node downtime, or low price volatility (heartbeat-only updates). The LST/ETH feeds used by this protocol (stETH/ETH, rETH/ETH, etc.) have heartbeat intervals of up to 24 hours. During any such window, a depositor can observe the stale price and exploit the discrepancy. No special privileges are required — `depositAsset` and `depositETH` are open to any user.

---

### Recommendation

Apply the same staleness validation already present in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    require(answeredInRound >= roundId, "Stale price");
    require(updatedAt > 0, "Incomplete round");
    require(price > 0, "Invalid price");

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

---

### Proof of Concept

1. The Chainlink stETH/ETH feed goes stale; its last reported price is 1.05 ETH (actual current rate: 1.02 ETH).
2. Alice calls `LRTDepositPool.depositAsset(stETH, 100e18, minRSETH, "")`.
3. `_beforeDeposit` calls `getRsETHAmountToMint(stETH, 100e18)`.
4. `getRsETHAmountToMint` calls `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns stale `1.05e18` with no staleness revert.
5. `rsethAmountToMint = (100e18 * 1.05e18) / rsETHPrice` — Alice receives ~2.9% more rsETH than her deposit is worth.
6. Existing rsETH holders are diluted by the excess minted rsETH.

The root cause is exclusively at [8](#0-7)  — no staleness validation on the `latestRoundData()` return values.
