### Title
No Flash Loan Protection and Stale Oracle Rate Enable Excess wrsETH Minting in L2 Pools - (File: contracts/cross-chain/CrossChainRateReceiver.sol, contracts/pools/RSETHPoolV3.sol)

---

### Summary

All L2 liquidity pool variants (`RSETHPoolV3`, `RSETHPool`, `RSETHPoolNoWrapper`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`) impose no restriction on smart contracts calling `deposit()`. The oracle rate consumed by every pool is sourced from `CrossChainRateReceiver.getRate()`, which returns a stored value with **no staleness check**. When the LayerZero-delivered rate lags behind the true L1 rsETH/ETH rate, a smart contract can flash-loan ETH, deposit at the stale (artificially low) rate, receive excess `wrsETH`, and exit — leaving the wrapper undercollateralized.

---

### Finding Description

**Root cause 1 — No EOA restriction on any pool `deposit()` function.**

Every pool variant exposes `deposit()` as a plain `external` function with only `nonReentrant` and `whenNotPaused` guards. No `tx.origin == msg.sender` check or equivalent exists anywhere in the deposit path. [1](#0-0) [2](#0-1) [3](#0-2) 

**Root cause 2 — `CrossChainRateReceiver.getRate()` returns a stored rate with no staleness validation.**

The rate is written by `lzReceive()` whenever a LayerZero message arrives from L1. The `lastUpdated` timestamp is stored but is **never consulted** by `getRate()`. If LayerZero delivery is delayed (network congestion, gas price spikes, or simply infrequent pushes), the stored rate can be materially lower than the true L1 rate. [4](#0-3) 

**Root cause 3 — Mint amount is computed directly from the stale rate.**

Every pool's `viewSwapRsETHAmountAndFee` divides the deposit amount by `rsETHToETHrate`. A stale (lower) rate produces a proportionally larger `rsETHAmount`.

```
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate
``` [5](#0-4) [6](#0-5) 

---

### Impact Explanation

When the L2 oracle rate is stale (lower than the true L1 rate), the pool mints more `wrsETH` than the deposited ETH can back once bridged to L1. The ETH bridged to L1 mints a smaller amount of rsETH than the `wrsETH` already issued on L2, leaving the `RsETHTokenWrapper` undercollateralized. Existing `wrsETH` holders suffer dilution; in the extreme case the wrapper cannot redeem all outstanding `wrsETH` 1-for-1 against rsETH, constituting **protocol insolvency** for the L2 wrapper.

Impact classification: **Medium — temporary or permanent undercollateralization of wrsETH (protocol insolvency risk for L2 wrapper).**

---

### Likelihood Explanation

- LayerZero cross-chain message delivery is not instantaneous and can lag by minutes to hours under congestion.
- The `InterimRSETHOracle` (used on several chains as an interim solution) is manually updated by a manager and can be stale for extended periods.
- No EOA check exists, so any smart contract — including a flash-loan receiver — can call `deposit()` in a single atomic transaction.
- The `limitDailyMint` modifier in V3 pools caps the total daily exposure but does not prevent the attack within that cap. [7](#0-6) [8](#0-7) 

Likelihood: **Medium.**

---

### Recommendation

1. **Add a staleness check in `CrossChainRateReceiver.getRate()`**: revert (or return a sentinel) if `block.timestamp - lastUpdated` exceeds a configurable threshold (e.g., 24 hours).
2. **Add an EOA-only guard** (`require(msg.sender == tx.origin, "no contracts")`) to all `deposit()` entry points, or at minimum enforce it as a configurable flag (analogous to the `preventSmartContracts` pattern in the reference report).
3. **Add a same-block deposit restriction** per address to prevent atomic flash-loan deposit-and-exit sequences even when smart contracts are permitted.

---

### Proof of Concept

**Setup**: L1 rsETH/ETH rate is `1.05e18` (rsETH worth 1.05 ETH). L2 `CrossChainRateReceiver.rate` is stale at `1.00e18` (LayerZero message delayed).

**Attack steps (single transaction)**:

1. Attacker contract calls a flash-loan provider to borrow `10,000 ETH`.
2. Attacker contract calls `RSETHPoolV3.deposit{value: 10000 ether}("")`.
3. Pool computes: `rsETHAmount = 10000e18 * 1e18 / 1.00e18 = 10000 wrsETH` (stale rate).
4. Correct amount at true rate: `10000e18 * 1e18 / 1.05e18 ≈ 9523.8 wrsETH`.
5. Attacker receives **476.2 excess wrsETH** (worth ~500 ETH at true rate).
6. Attacker sells excess `wrsETH` on a secondary market or redeems via the wrapper.
7. Repays flash loan. Net profit ≈ 500 ETH at the expense of existing `wrsETH` holders.

The pool bridges `10,000 ETH` to L1, which mints only `~9,523.8 rsETH`. The wrapper has issued `10,000 wrsETH` but holds backing for only `9,523.8`, leaving it **476.2 rsETH short**. [9](#0-8) [10](#0-9)
