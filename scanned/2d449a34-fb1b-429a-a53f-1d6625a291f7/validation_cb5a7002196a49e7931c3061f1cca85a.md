### Title
Uninitialized `maxNumberOfWithdrawalsPerUser` Permanently Blocks `initiateWithdrawal` Until Admin Intervenes — (`contracts/KERNEL/KernelDepositPool.sol`)

### Summary
`KernelDepositPool.initialize` never sets `maxNumberOfWithdrawalsPerUser`, leaving it at the Solidity default of `0`. Because `initiateWithdrawal` guards with `>=` rather than `>`, the check `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser` evaluates to `0 >= 0 == true` for every caller, causing an immediate `WithdrawalLimitReached` revert. Any tokens staked before the admin calls `setMaxNumberOfWithdrawalsPerUser` — including tokens auto-staked via `KernelMerkleDistributor.claimAndStake` — are temporarily unwithdrawable.

### Finding Description

**Root cause — `KernelDepositPool.sol` line 323:**

```solidity
if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();
```

`maxNumberOfWithdrawalsPerUser` defaults to `0`. A fresh user has `userWithdrawalIds[msg.sender].length == 0`. The condition `0 >= 0` is always `true`, so `initiateWithdrawal` always reverts.

**`initialize` never sets the value:** [1](#0-0) 

There is no assignment to `maxNumberOfWithdrawalsPerUser` anywhere in `initialize`. The only way to set it is via the separate admin call `setMaxNumberOfWithdrawalsPerUser`, which itself rejects `0` as an argument — meaning `0` is exclusively an uninitialized sentinel, not a valid configured value. [2](#0-1) 

**The `claimAndStake` path:**

`KernelMerkleDistributor.claimAndStake` calls `KernelDepositPool.stakeFor`, which credits `balanceOf[account]` with no check on `maxNumberOfWithdrawalsPerUser`. [3](#0-2) [4](#0-3) 

The user's tokens are now staked, but `initiateWithdrawal` is blocked until the admin separately configures the limit. [5](#0-4) 

### Impact Explanation
Tokens staked via `claimAndStake` (or direct `stake`) before admin calls `setMaxNumberOfWithdrawalsPerUser` are temporarily frozen: `balanceOf[user] > 0` but `initiateWithdrawal` always reverts. This matches **Medium — Temporary freezing of funds**. The freeze lifts once the admin configures the limit, so it is not permanent.

### Likelihood Explanation
The `initialize` function is the natural deployment entry point. Nothing in the contract or its deployment flow enforces that `setMaxNumberOfWithdrawalsPerUser` must be called before staking is enabled. The `KernelMerkleDistributor` can be live and accepting `claimAndStake` calls the moment both contracts are deployed and the merkle root is set, with no dependency on the withdrawal limit being configured. This is a realistic deployment ordering gap, not a contrived scenario.

### Recommendation
Set a safe default for `maxNumberOfWithdrawalsPerUser` inside `initialize`, or add a guard in `stakeFor`/`stake` that reverts if `maxNumberOfWithdrawalsPerUser == 0`. For example:

```solidity
// In initialize():
maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER; // or a chosen safe default
```

Alternatively, block staking entirely until the withdrawal limit is configured:

```solidity
// In stake() and stakeFor():
if (maxNumberOfWithdrawalsPerUser == 0) revert WithdrawalLimitNotConfigured();
```

### Proof of Concept

```solidity
// 1. Deploy KernelDepositPool (do NOT call setMaxNumberOfWithdrawalsPerUser)
// 2. Deploy KernelMerkleDistributor, grant it STAKE_FOR_ROLE on KernelDepositPool
// 3. Set a merkle root with a valid leaf for `beneficiary`
// 4. beneficiary calls claimAndStake(index, beneficiary, amount, proof)
//    → stakeFor(beneficiary, amount) succeeds
//    → balanceOf[beneficiary] > 0  ✓
// 5. beneficiary calls initiateWithdrawal(amount)

//    → maxNumberOfWithdrawalsPerUser == 0
//    → 0 >= 0 == true → revert WithdrawalLimitReached  ✓
// 6. Assert: balanceOf[beneficiary] > 0 AND initiateWithdrawal reverts
```
