### Title
Missing Time-Based Staleness Check in Collateral Oracle Enables Over-Minting of wrsETH — (`contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol`)

---

### Summary

`ChainlinkOracleForRSETHPoolCollateral.getRate()` performs only an incomplete-round check (`answeredInRound < roundID`) and a zero-timestamp check, but **never compares `updatedAt` against `block.timestamp`**. On L2 chains where the Chainlink oracle can go stale (sequencer downtime, low gas incentives), a collateral token whose price has dropped since the last oracle update will still report the old, higher rate. `RSETHPoolV3ExternalBridge.deposit(token, amount)` uses this stale-high rate to mint excess wrsETH, creating unbacked supply that dilutes L1 rsETH yield when bridged and unwrapped.

---

### Finding Description

`ChainlinkOracleForRSETHPoolCollateral.getRate()` reads `latestRoundData` and applies three guards:

```solidity
if (answeredInRound < roundID) revert StalePrice();   // incomplete round only
if (timestamp == 0) revert IncompleteRound();          // zero-timestamp only
if (ethPrice <= 0) revert InvalidPrice();              // negative price only
``` [1](#0-0) 

There is **no** `block.timestamp - timestamp > maxStaleness` guard and **no** L2 sequencer uptime feed check. A price that was last updated hours or days ago passes all three checks and is returned as current.

`RSETHPoolV3ExternalBridge.deposit(token, amount)` calls `viewSwapRsETHAmountAndFee(amount, token)`, which fetches the collateral rate from `supportedTokenOracle[token]` (a `ChainlinkOracleForRSETHPoolCollateral` instance) and computes:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [2](#0-1) 

If `tokenToETHRate` is stale-high (collateral price dropped since last oracle update), `rsETHAmount` is inflated. The inflated wrsETH is minted directly to the caller with no further validation: [3](#0-2) 

The daily mint limit (`limitDailyMint`) does not prevent this — it only caps the total wrsETH minted per day, not the per-unit over-minting ratio. [4](#0-3) 

---

### Impact Explanation

Every unit of excess wrsETH minted against a stale-high collateral rate is unbacked. When the attacker bridges the collateral ETH to L1 (via `bridgeAssets` / `bridgeTokens`) and other users unwrap wrsETH to rsETH on L1, the rsETH pool's backing is diluted by the unbacked portion. This is a direct theft of unclaimed yield from all rsETH holders: the attacker extracts value that belongs to the yield accrual of the pool.

---

### Likelihood Explanation

- L2 sequencer downtime is a documented, recurring event on Optimism, Arbitrum, and Base.
- Chainlink L2 feeds stop updating during sequencer outages; `answeredInRound >= roundID` remains true for the last completed round, so the existing check does **not** catch this condition.
- The exploit requires only a public `deposit` call — no privileged role, no governance capture.
- The precondition (collateral price drop during oracle staleness window) is realistic for any non-monotonically-increasing collateral token added via `addSupportedToken`.

---

### Recommendation

Add a configurable maximum staleness age to `ChainlinkOracleForRSETHPoolCollateral` and revert if the price is too old:

```solidity
uint256 public immutable maxStaleness; // e.g. 3600 seconds

function getRate() public view returns (uint256) {
    (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();

    if (answeredInRound < roundID) revert StalePrice();
    if (timestamp == 0) revert IncompleteRound();
    if (ethPrice <= 0) revert InvalidPrice();
    if (block.timestamp - timestamp > maxStaleness) revert StalePrice(); // ADD THIS

    ...
}
```

Additionally, for L2 deployments, integrate a Chainlink L2 Sequencer Uptime Feed and revert if the sequencer is down or has recently restarted (with a grace period).
