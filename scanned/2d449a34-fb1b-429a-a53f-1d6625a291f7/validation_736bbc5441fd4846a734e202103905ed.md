### Title
`RSETHPoolV3.removeSupportedToken` Permanently Blocked by Dust Token Donation - (File: contracts/pools/RSETHPoolV3.sol)

### Summary
`RSETHPoolV3.removeSupportedToken` reverts whenever the contract holds any balance of the token being removed. An unprivileged attacker can send 1 wei of any supported token directly to the pool, permanently preventing the `TIMELOCK_ROLE` from removing that token from the supported list.

### Finding Description
`removeSupportedToken` contains a hard balance check before deleting the token from the supported list: [1](#0-0) 

```solidity
function removeSupportedToken(address token, uint256 tokenIndex) external onlyRole(TIMELOCK_ROLE) {
    UtilLib.checkNonZeroAddress(token);
    if (supportedTokenList[tokenIndex] != token) revert TokenNotFoundError();
    if (IERC20(token).balanceOf(address(this)) != 0) revert TokenBalanceNotZero();
    ...
}
```

The check `IERC20(token).balanceOf(address(this)) != 0` reads the raw on-chain balance of the contract. Because the pool has a public `deposit` function that accepts ERC-20 tokens and accumulates them in the contract, and because ERC-20 tokens can be transferred to any address without consent, an attacker can trivially inflate this balance to a non-zero value.

The pool has no sweep or recovery mechanism for supported tokens that would allow the balance to be drained back to zero before calling `removeSupportedToken`. The only way to reduce the balance is through `moveAssetsForBridging` (BRIDGER_ROLE) or `withdrawFees` (BRIDGER_ROLE), but neither is guaranteed to drain the balance to exactly zero, and an attacker can re-donate after each drain. [2](#0-1) 

### Impact Explanation
The `TIMELOCK_ROLE` is permanently unable to remove a supported token from `RSETHPoolV3`. If a supported token or its oracle becomes compromised, deprecated, or malicious, the protocol cannot delist it. Users will continue to be able to deposit into the problematic token and receive minted `wrsETH`, and the bridger will continue to move those assets to L1. This constitutes a failure to deliver the protocol's promised governance capability without any direct loss of value — matching the **Low** impact tier: *contract fails to deliver promised returns, but doesn't lose value*.

### Likelihood Explanation
The attack requires only a single ERC-20 `transfer` of 1 wei of any supported token to the pool address. The cost to the attacker is negligible. The attack is permissionless, requires no special role, and can be repeated after any partial drain. Likelihood is **High** for any token the attacker wishes to lock in.

### Recommendation
Replace the strict zero-balance check with a tracked internal accounting variable (analogous to the `feeEarnedInToken` pattern already used in the contract). Track the "protocol-owned" balance separately from the raw `balanceOf`, and allow `removeSupportedToken` to proceed when the protocol-tracked balance is zero, sweeping any residual donated balance to the treasury or a recovery address. Alternatively, add an admin-accessible sweep function for supported tokens that can drain the balance to zero before removal.

### Proof of Concept
1. `RSETHPoolV3` is deployed with token `T` as a supported asset.
2. Attacker calls `T.transfer(address(pool), 1)` — costs 1 wei of `T`.
3. `IERC20(T).balanceOf(address(pool))` is now `1`.
4. `TIMELOCK_ROLE` calls `removeSupportedToken(T, index)`.
5. The check `if (IERC20(token).balanceOf(address(this)) != 0) revert TokenBalanceNotZero()` fires.
6. The call reverts. Token `T` cannot be removed.
7. If the bridger calls `moveAssetsForBridging(T, amount)` to drain the balance, the attacker immediately re-donates 1 wei, restoring the block. [3](#0-2) [4](#0-3)
