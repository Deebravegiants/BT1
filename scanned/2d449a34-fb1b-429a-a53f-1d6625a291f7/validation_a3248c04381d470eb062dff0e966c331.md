### Title
`LRTDepositPool.getAssetCurrentLimit` Does Not Account for `RSETH.maxMintAmountPerDay`, Causing Misleading Deposit Limit Reporting - (File: contracts/LRTDepositPool.sol)

---

### Summary

`LRTDepositPool.getAssetCurrentLimit()` is a public view function that reports how much of a given asset a user can still deposit. It only checks the per-asset deposit cap stored in `LRTConfig`, but ignores the separate daily rsETH mint cap enforced by `RSETH.maxMintAmountPerDay`. When the daily mint cap is exhausted, any deposit will revert inside `RSETH.mint()`, yet `getAssetCurrentLimit()` continues to return a non-zero value, misleading users and integrators into submitting transactions that will fail.

---

### Finding Description

`LRTDepositPool.getAssetCurrentLimit()` computes the remaining deposit room solely against `lrtConfig.depositLimitByAsset(asset)`:

```solidity
// contracts/LRTDepositPool.sol:402-409
function getAssetCurrentLimit(address asset) public view override returns (uint256) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)) {
        return 0;
    }
    return lrtConfig.depositLimitByAsset(asset) - totalAssetDeposits;
}
``` [1](#0-0) 

When a deposit is executed, `_mintRsETH()` is called, which calls `IRSETH(rsethToken).mint()`:

```solidity
// contracts/LRTDepositPool.sol:686-690
function _mintRsETH(uint256 rsethAmountToMint) private {
    address rsethToken = lrtConfig.rsETH();
    IRSETH(rsethToken).mint(msg.sender, rsethAmountToMint);
}
``` [2](#0-1) 

`RSETH.mint()` enforces a completely separate daily cap via the `checkDailyMintLimit` modifier:

```solidity
// contracts/RSETH.sol:42-56
modifier checkDailyMintLimit(uint256 amount) {
    if (block.timestamp >= periodStartTime + 1 days) {
        currentPeriodMintedAmount = 0;
        periodStartTime = getCurrentPeriodStartTime();
    }
    if (currentPeriodMintedAmount + amount > maxMintAmountPerDay) {
        revert DailyMintLimitExceeded(currentPeriodMintedAmount + amount, maxMintAmountPerDay);
    }
    currentPeriodMintedAmount += amount;
    _;
}
``` [3](#0-2) 

`RSETH` also exposes `remainingDailyMintLimit()` as a separate view:

```solidity
// contracts/RSETH.sol:265-272
function remainingDailyMintLimit() external view returns (uint256) {
    if (maxMintAmountPerDay == 0) return 0;
    uint256 effectiveDailyMintAmount = (block.timestamp >= periodStartTime + 1 days) ? 0 : currentPeriodMintedAmount;
    return maxMintAmountPerDay > effectiveDailyMintAmount ? maxMintAmountPerDay - effectiveDailyMintAmount : 0;
}
``` [4](#0-3) 

`getAssetCurrentLimit()` never consults `RSETH.remainingDailyMintLimit()` or `RSETH.maxMintAmountPerDay`. The two limits are completely independent, and the view function only reflects one of them. [1](#0-0) 

---

### Impact Explanation

A user or integrator calls `getAssetCurrentLimit(asset)` and receives a positive value, concluding that a deposit of that size is valid. They then call `depositAsset()` or `depositETH()`. If `RSETH.maxMintAmountPerDay` has already been reached for the current period, the call reverts with `DailyMintLimitExceeded` inside `RSETH.mint()`. The user loses gas and receives no rsETH. No funds are permanently lost, but the contract fails to deliver the deposit it appeared to promise. This maps to **Low — Contract fails to deliver promised returns, but doesn't lose value**. [5](#0-4) 

---

### Likelihood Explanation

`maxMintAmountPerDay` is an active operational parameter set by the LRT manager. During high-demand periods (e.g., new LST listings, yield events, or protocol announcements), the daily cap can be exhausted well before the per-asset deposit limit is reached. Any depositor who queries `getAssetCurrentLimit()` as a pre-flight check — a standard pattern for on-chain integrators and front-ends — will receive a misleading result. [6](#0-5) 

---

### Recommendation

Update `getAssetCurrentLimit()` to also account for the remaining rsETH daily mint capacity. The returned value should be the minimum of the per-asset deposit room and the ETH-equivalent of the remaining daily rsETH mint allowance:

```solidity
function getAssetCurrentLimit(address asset) public view override returns (uint256) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)) {
        return 0;
    }
    uint256 assetRoom = lrtConfig.depositLimitByAsset(asset) - totalAssetDeposits;

    // Also cap by remaining RSETH daily mint limit (converted to asset units)
    address rsethToken = lrtConfig.rsETH();
    uint256 remainingRsETH = IRSETH(rsethToken).remainingDailyMintLimit();
    // Convert remainingRsETH to asset units using the oracle
    uint256 rsethPrice = ILRTOracle(...).rsETHPrice();
    uint256 assetPrice = ILRTOracle(...).getAssetPrice(asset);
    uint256 mintRoom = remainingRsETH * rsethPrice / assetPrice;

    return assetRoom < mintRoom ? assetRoom : mintRoom;
}
``` [1](#0-0) 

---

### Proof of Concept

1. `RSETH.maxMintAmountPerDay` is set to `1000e18` rsETH. The current period has already minted `1000e18` rsETH (`currentPeriodMintedAmount == maxMintAmountPerDay`).
2. The per-asset deposit limit for stETH is `10000e18` stETH, and only `5000e18` has been deposited. `getAssetCurrentLimit(stETH)` returns `5000e18`.
3. A user calls `getAssetCurrentLimit(stETH)`, sees `5000e18`, and submits `depositAsset(stETH, 1e18, 0, "")`.
4. `_beforeDeposit` passes (asset limit not exceeded). `_mintRsETH` calls `RSETH.mint()`.
5. `checkDailyMintLimit` reverts with `DailyMintLimitExceeded(1000e18 + rsethToMint, 1000e18)`.
6. The transaction reverts. The user loses gas. `getAssetCurrentLimit` gave a false positive. [7](#0-6) [8](#0-7)
