Looking at the swap path, `pay()` function, and the `IncorrectDelta` guard, I found a direct analog to the external bug.

### Title
ETH Sent Alongside Non-WETH Token Swaps Is Silently Accepted and Permanently Exposed to Theft — (`metric-periphery/contracts/MetricOmmSimpleRouter.sol`, `metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary

All four swap entry points in `MetricOmmSimpleRouter` (`exactInputSingle`, `exactOutputSingle`, `exactInput`, `exactOutput`) are declared `external payable`. The `pay()` helper in `PeripheryPayments` only consumes the router's native ETH balance when `token == WETH`. When `tokenIn != WETH`, the ETH is silently accepted, never used, and left sitting in the router where any third party can drain it via `refundETH()`.

### Finding Description

The `pay()` function branches on `token == WETH`:

```solidity
// PeripheryPayments.sol lines 69-88
function pay(address token, address payer, address recipient, uint256 value) internal {
    if (payer == address(this)) {
        IERC20(token).safeTransfer(recipient, value);
    } else if (token == WETH) {          // ← only branch that uses msg.value / native balance
        uint256 nativeBalance = address(this).balance;
        if (nativeBalance >= value) {
            IWETH9(WETH).deposit{value: value}();
            IERC20(WETH).safeTransfer(recipient, value);
        } ...
    } else {
        IERC20(token).safeTransferFrom(payer, recipient, value);  // ← ETH ignored
    }
}
``` [1](#0-0) 

When `tokenIn != WETH`, the `else` branch executes a plain `safeTransferFrom` and the router's ETH balance is untouched. None of the swap functions check `msg.value` against `tokenIn`:

```solidity
// MetricOmmSimpleRouter.sol lines 67-86
function exactInputSingle(ExactInputSingleParams calldata params) external payable returns (uint256 amountOut) {
    _checkDeadline(params.deadline);
    // ← no guard: if msg.value > 0 && params.tokenIn != WETH, ETH is silently accepted
    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    ...
}
``` [2](#0-1) 

The same omission exists in `exactOutputSingle`, `exactInput`, and `exactOutput`. [3](#0-2) 

The `receive()` guard only blocks bare ETH transfers (no calldata), not ETH attached to a function call:

```solidity
receive() external payable {
    if (msg.sender != WETH) revert NotWETH();
}
``` [4](#0-3) 

`refundETH()` sends the entire ETH balance to `msg.sender`, not to the original depositor:

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);   // ← goes to caller, not original victim
    }
}
``` [5](#0-4) 

### Impact Explanation

Any user who calls a swap function with `msg.value > 0` and `tokenIn != WETH` loses their ETH. The ETH accumulates in the router and is claimable by any address that calls `refundETH()` before the victim does. This is a direct loss of user principal with no revert, no warning, and no automatic refund. The victim must issue a second transaction to recover funds, during which a front-runner or MEV bot can drain the balance.

### Likelihood Explanation

Medium. Users familiar with Uniswap-style routers commonly send native ETH for swaps involving the native token. A user who passes the wrong `tokenIn` address (e.g., the raw token address instead of WETH), or who uses a script that attaches ETH unconditionally, will silently lose funds. The `payable` modifier on all four entry points provides no friction against this mistake.

### Recommendation

Add a guard in each swap entry point that reverts when ETH is sent but the input token is not WETH:

```solidity
if (msg.value > 0 && params.tokenIn != WETH) revert NativeTokenNotAccepted();
```

Alternatively, automatically refund any unused ETH at the end of each swap function (analogous to the external report's fix of returning unused native coin as wrapped native coin).

### Proof of Concept

1. Pool has `token0 = WETH`, `token1 = USDC`.
2. Alice calls `exactInputSingle{value: 1 ether}(params)` where `params.tokenIn = address(USDC)`.
3. The swap executes normally — USDC is pulled from Alice via `safeTransferFrom`; the 1 ETH is accepted by the `payable` function and ignored by `pay()`.
4. Alice's 1 ETH sits in the router.
5. Bob (front-runner) calls `refundETH()` in the next block and receives Alice's 1 ETH.
6. Alice has lost 1 ETH with no revert or on-chain indication of the error.
