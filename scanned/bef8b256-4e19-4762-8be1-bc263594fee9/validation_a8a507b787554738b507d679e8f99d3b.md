### Title
`removeLiquidity` silently skips LP token transfer when floor-rounding produces zero external amount, causing permanent LP principal loss — (`File: metric-core/contracts/libraries/LiquidityLib.sol`)

---

### Summary

`LiquidityLib.removeLiquidity` decrements the pool's internal scaled accounting for every share burned, then converts the total scaled removal to an external token amount using **floor division**. When the scaled amount is smaller than the scale multiplier (a realistic condition for USDC/USDT pools), the external amount rounds to zero. The `if (amount0Removed > 0)` guard then silently skips the ERC-20 transfer — the LP's shares are gone, the pool's accounting is decremented, but no tokens are returned. This is the direct Metric OMM analog of the `payUser` silent-skip bug.

---

### Finding Description

In `LiquidityLib.removeLiquidity`:

**Step 1 — shares burned and bin accounting decremented (state is mutated first):** [1](#0-0) 

```solidity
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
...
binState.token0BalanceScaled -= uint104(amount0Scaled);   // bin accounting decremented
...
binTotalShares[binIdx] = binTotalSharesVal - sharesToRemove;  // shares burned
positionBinShares[posKey] = newUserShares;
```

**Step 2 — pool totals decremented:** [2](#0-1) 

**Step 3 — floor-division conversion to external units:** [3](#0-2) 

```solidity
(amount0Removed, amount1Removed) =
    _deltasScaledToExternal(totalToken0ToRemoveScaled, totalToken1ToRemoveScaled, ctx, Math.Rounding.Floor);
```

`_deltasScaledToExternal` with `Math.Rounding.Floor` performs: [4](#0-3) 

```solidity
deltaAmount0 = scaledDeltaAmount0 / ctx.token0ScaleMultiplier;   // floor — can produce 0
```

**Step 4 — silent skip:** [5](#0-4) 

```solidity
if (amount0Removed > 0) {
    IERC20(ctx.token0).safeTransfer(owner, amount0Removed);   // skipped when == 0
}
```

When `totalToken0ToRemoveScaled < token0ScaleMultiplier`, `amount0Removed = 0` and the transfer is silently skipped. No revert, no distinct event field — the `LiquidityRemoved` event fires with `amount0Removed = 0`, which is indistinguishable from a legitimate zero-token-0 removal (e.g., a bin entirely in token1).

The scale multiplier is set at pool creation as `10^(max(18, decimals) - decimals)`: [6](#0-5) 

For USDC (6 decimals) `token0ScaleMultiplier = 10^12`. For USDT (6 decimals) same. Both are explicitly in scope.

The asymmetry is intentional on the **add** side (ceil protects the pool) but the **remove** side has no floor-to-zero guard: [7](#0-6) 

```solidity
// addLiquidity — ceil, LP always pays ≥ 1 token unit
(amount0Added, amount1Added) =
    _deltasScaledToExternal(totalToken0ToAddScaled, totalToken1ToAddScaled, ctx, Math.Rounding.Ceil);
```

---

### Impact Explanation

- LP burns shares; `binState.token0BalanceScaled` and `binTotals.scaledToken0` are decremented.
- No ERC-20 transfer occurs; the LP receives 0 tokens.
- The pool's actual ERC-20 balance exceeds its internal accounting by `totalToken0ToRemoveScaled` scaled units. These tokens become untracked surplus — effectively locked or swept as protocol fees — permanently out of reach of the LP.
- Maximum loss per `removeLiquidity` call: up to `(token0ScaleMultiplier − 1) / token0ScaleMultiplier` tokens. For USDC this is up to **~1 USDC per call**. Repeated calls accumulate.
- This is a direct loss of LP principal with no revert and no distinguishable on-chain signal.

---

### Likelihood Explanation

- Requires a pool with a token whose decimals < 18 (USDC, USDT — both in scope per the allowed-impact gate).
- Requires the bin's scaled token0 balance at removal time to be < `token0ScaleMultiplier`. This is a normal post-swap state: swaps consume token0 from bins, and a bin that started with 1 USDC can be traded down to 0.5 USDC (= `5 × 10^11` scaled units < `10^12`).
- No special permissions required; any LP calling `removeLiquidity` on a USDC/USDT pool after sufficient swap activity hits this path.

---

### Recommendation

Before performing the transfer guard, check whether the scaled amount is non-zero but the external amount rounded to zero, and revert (or at minimum emit a distinct event):

```solidity
// After _deltasScaledToExternal with Floor:
if (totalToken0ToRemoveScaled > 0 && amount0Removed == 0) {
    revert RoundedToZero(); // LP cannot silently lose principal
}
if (totalToken1ToRemoveScaled > 0 && amount1Removed == 0) {
    revert RoundedToZero();
}
```

Alternatively, use `Math.Rounding.Ceil` on the remove path as well (accepting that the pool may transfer 1 extra token unit), or accumulate sub-unit remainders in a per-position credit that can be claimed later.

---

### Proof of Concept

1. Deploy a pool with USDC (6 decimals) as `token0`; `token0ScaleMultiplier = 10^12`.
2. LP calls `addLiquidity` on a bin above the current bin, paying 1 USDC. The bin receives `10^12` scaled units.
3. Swaps execute, consuming token0 from that bin until `binState.token0BalanceScaled = 5 × 10^11` (0.5 USDC in scaled units).
4. LP calls `removeLiquidity` to burn all shares in that bin.
   - `amount0Scaled = 5 × 10^11 * sharesToRemove / binTotalSharesVal = 5 × 10^11` (removing 100 % of shares).
   - `totalToken0ToRemoveScaled = 5 × 10^11`.
   - `binTotals.scaledToken0` decremented by `5 × 10^11`. ✓ (state mutated)
   - `amount0Removed = 5 × 10^11 / 10^12 = 0`. ← floor rounds to zero
   - `if (0 > 0)` → **transfer skipped silently**.
5. LP's USDC balance is unchanged. The 0.5 USDC is permanently stranded in the pool.
6. `LiquidityRemoved` event fires with `amount0Removed = 0` — no on-chain indication of the loss.
