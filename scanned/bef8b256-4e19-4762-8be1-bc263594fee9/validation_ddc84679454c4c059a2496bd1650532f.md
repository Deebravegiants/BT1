### Title
Attacker Can Grief `removeLiquidity` by Frontrunning with `addLiquidity` to Victim's Position, Triggering `MinimalLiquidity` Revert — (File: `metric-core/contracts/libraries/LiquidityLib.sol`)

---

### Summary

`addLiquidity` permits any `msg.sender` to mint shares into any `owner`'s position. An attacker who monitors the mempool can frontrun a victim's `removeLiquidity` call by adding a single share to the victim's position, causing the `MinimalLiquidity` dust-floor check to revert. This can be repeated indefinitely, permanently denying the victim the ability to fully exit their position.

---

### Finding Description

**Root cause — two independent design choices compose into a griefing vector:**

**1. `addLiquidity` is permissionless with respect to `owner`.** [1](#0-0) 

Any `msg.sender` may call `addLiquidity(owner=<victim>, salt=<victim_salt>, ...)`. The caller pays the tokens; the shares are credited to `owner`. There is no consent check from `owner`.

**2. `removeLiquidity` reverts when remaining shares fall in the dust range `(0, MINIMAL_MINTABLE_LIQUIDITY)`.** [2](#0-1) 

```solidity
if (userShares < sharesToRemove) {
    revert IMetricOmmPoolActions.InsufficientLiquidity(sharesToRemove, userShares);
}
uint256 newUserShares = userShares - sharesToRemove;
if (newUserShares > 0 && newUserShares < ctx.minimalMintableLiquidity) {
    revert IMetricOmmPoolActions.MinimalLiquidity(newUserShares, ctx.minimalMintableLiquidity);
}
```

**Attack flow:**

1. Victim holds `userShares = MINIMAL_MINTABLE_LIQUIDITY` (e.g., 1 000) in bin `B` with salt `S`.
2. Victim submits `removeLiquidity(owner=victim, salt=S, shares=[1000])` to fully exit.
3. Attacker sees the transaction in the mempool and frontruns with `addLiquidity(owner=victim, salt=S, shares=[1])`, paying 1 share worth of tokens.
4. Victim's `userShares` is now 1 001.
5. Victim's `removeLiquidity(1000)` computes `newUserShares = 1001 − 1000 = 1`.
6. `1 > 0 && 1 < 1000` → `MinimalLiquidity` revert.
7. Attacker repeats for every retry.

The victim's salt is publicly observable from the indexed `LiquidityAdded` event: [3](#0-2) 

The `addLiquidity` dust-floor check only guards the *new* total, not the victim's ability to remove: [4](#0-3) 

---

### Impact Explanation

The victim cannot fully exit their position as long as the attacker keeps frontrunning. Their LP tokens remain locked in the pool. Because the attacker's tokens are credited to the victim's position, the victim's claimable balance grows, but they can never withdraw it to zero. Any partial removal that would leave `0 < remaining < MINIMAL_MINTABLE_LIQUIDITY` also reverts, so the victim cannot drain the position in steps either — the attacker simply adds 1 share before each step. This constitutes an **unusable `removeLiquidity` flow** for the targeted position, matching the allowed impact gate.

---

### Likelihood Explanation

Medium. The attacker must spend real tokens per frontrun (1 share worth of collateral each time). The victim can partially mitigate by using a helper contract that reads `userShares` at execution time and removes all shares atomically — but only if the victim's `owner` address is itself a contract. EOA owners have no atomic read-then-remove primitive and are fully exposed. The victim's salt is always recoverable from on-chain events, so target identification costs nothing.

---

### Recommendation

**Option A (preferred):** In `removeLiquidity`, when `sharesToRemove == userShares` (full exit), skip the `MinimalLiquidity` check entirely. A full exit to zero is always safe and should never be blocked.

**Option B:** Add a `removeAllLiquidity` entry point that always removes `positionBinShares[posKey]` regardless of the dust floor, preventing the attacker's injected shares from blocking exit.

**Option C:** Require `msg.sender == owner` for `addLiquidity` (breaks the operator pattern but eliminates the griefing surface entirely).

---

### Proof of Concept

```
Setup:
  MINIMAL_MINTABLE_LIQUIDITY = 1_000
  Victim (EOA) holds 1_000 shares in bin 4, salt = 0

Step 1 — Victim submits:
  pool.removeLiquidity(victim, 0, {binIdxs:[4], shares:[1000]}, "")

Step 2 — Attacker frontruns (pays ~1 share of token0):
  pool.addLiquidity(victim, 0, {binIdxs:[4], shares:[1]}, "", "")
  // positionBinShares[key(victim,0,4)] = 1_001

Step 3 — Victim's tx executes:
  newUserShares = 1001 - 1000 = 1
  1 > 0 && 1 < 1000  →  revert MinimalLiquidity(1, 1000)

Step 4 — Victim retries with shares=[1001]:
  Attacker frontruns again with shares=[1]
  newUserShares = 1002 - 1001 = 1  →  revert

Result: Victim's position is permanently griefed; funds are inaccessible.
```
