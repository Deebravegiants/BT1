### Title
Unprivileged Caller Can Bypass `onlyLRTOperator` via Direct `eigenPod.verifyStaleBalance()` to Force a Checkpoint and Trigger LRTOracle Downside-Protection Pause — (`contracts/NodeDelegator.sol`, `contracts/LRTOracle.sol`)

---

### Summary

`NodeDelegator.startCheckpoint` is guarded by `onlyLRTOperator`, but EigenLayer's `IEigenPod.verifyStaleBalance` is intentionally permissionless. Any caller can invoke `verifyStaleBalance` directly on the EigenPod (bypassing `NodeDelegator` entirely), force-start a checkpoint on a slashed validator, complete it via the also-permissionless `verifyCheckpointProofs`, and then call the public `LRTOracle.updateRSETHPrice()`. If the resulting rsETH price drop exceeds `pricePercentageLimit`, the oracle's downside-protection logic pauses both `LRTDepositPool` and `LRTWithdrawalManager`, temporarily freezing user deposits and withdrawals.

---

### Finding Description

**Guard bypass path:**

`NodeDelegator.startCheckpoint` enforces `onlyLRTOperator`: [1](#0-0) 

However, the EigenPod's `verifyStaleBalance` carries no caller restriction. The interface comment explicitly states:

> "this method allows **anyone** to start a checkpoint as soon as a slashing occurs on the beacon chain." [2](#0-1) 

Similarly, `verifyCheckpointProofs` is permissionless:

> "Anyone can call this method to submit proofs towards the current checkpoint." [3](#0-2) 

**Price impact path:**

After the checkpoint finalizes with a negative `balanceDeltaWei`, EigenLayer calls `recordBeaconChainETHBalanceUpdate`, reducing the NodeDelegator's `beaconChainETHStrategy` shares. `NodeDelegator.getEffectivePodShares()` reads these shares via `NodeDelegatorHelper.getWithdrawableShare()`: [4](#0-3) 

`LRTDepositPool.getETHDistributionData()` aggregates `getEffectivePodShares()` across all NDCs: [5](#0-4) 

`LRTOracle._updateRsETHPrice()` calls `_getTotalEthInProtocol()` → `getTotalAssetDeposits()` → `getETHDistributionData()`, so the reduced shares flow directly into `newRsETHPrice`.

**Pause trigger:**

`LRTOracle._updateRsETHPrice()` contains downside-protection logic: [6](#0-5) 

If `diff > pricePercentageLimit.mulWad(highestRsethPrice)`, it pauses `LRTDepositPool`, `LRTWithdrawalManager`, and itself. `updateRSETHPrice()` is `public whenNotPaused`, callable by anyone: [7](#0-6) 

---

### Impact Explanation

An unprivileged attacker can choose the exact moment to trigger the checkpoint — e.g., right after the initial slash (before correlation penalties finalize), when the balance delta is already negative but the operator has not yet decided it is the right time to checkpoint. This causes a premature rsETH price drop. If `pricePercentageLimit` is configured (which is its intended production use), the oracle pauses deposits and withdrawals. No ETH is stolen, but users are temporarily unable to deposit or withdraw, and the rsETH price is reduced ahead of the operator's intended schedule. This matches **Low — Contract fails to deliver promised returns, but doesn't lose value** (and borderline **Medium — Temporary freezing of funds** if the pause is considered a freeze).

---

### Likelihood Explanation

- Requires a validator to be slashed on the beacon chain (an external event, but slashings do occur in production).
- The attacker must construct valid beacon chain Merkle proofs — technically demanding but well-documented and tooled (EigenLayer's own test suite demonstrates this).
- No privileged access, no front-running dependency, no collusion required.
- `verifyStaleBalance` and `verifyCheckpointProofs` are both callable directly on the EigenPod address, which is public (`NodeDelegator.eigenPod`).

---

### Recommendation

1. **Accept the EigenLayer design**: Acknowledge that `verifyStaleBalance` is intentionally permissionless and that checkpoints can be externally triggered. Document this explicitly.
2. **Decouple the oracle pause from a single price update**: Instead of pausing immediately on a single `updateRSETHPrice` call, require the price to remain below the threshold for a configurable number of blocks/calls before pausing.
3. **Use a TWAP or smoothing window** for `rsETHPrice` so that a single checkpoint-induced drop does not immediately cross the pause threshold.
4. **Restrict `updateRSETHPrice` to privileged callers** (e.g., `onlyLRTOperator` or a keeper role) so that even if a checkpoint is force-started, the attacker cannot also force the oracle update that triggers the pause.

### Proof of Concept

```solidity
// Fork test (Holesky or mainnet fork with a slashed validator)
function testForceCheckpointPause() external {
    // Preconditions:
    // - nodeDelegator.eigenPod() has an ACTIVE validator that is slashed on beacon chain
    // - lrtOracle.pricePercentageLimit() > 0
    // - validator.lastCheckpointedAt < beaconTimestamp

    IEigenPod pod = nodeDelegator.eigenPod();

    // Step 1: unprivileged caller starts checkpoint via verifyStaleBalance
    vm.prank(attacker);
    pod.verifyStaleBalance(beaconTimestamp, stateRootProof, validatorProof);

    // Step 2: unprivileged caller completes checkpoint
    vm.prank(attacker);
    pod.verifyCheckpointProofs(balanceContainerProof, balanceProofs);

    // Step 3: unprivileged caller triggers oracle price update
    vm.prank(attacker);
    lrtOracle.updateRSETHPrice();

    // Assert: protocol is paused
    assertTrue(lrtDepositPool.paused());
    assertTrue(lrtWithdrawalManager.paused());
    assertTrue(lrtOracle.paused());
}
```
