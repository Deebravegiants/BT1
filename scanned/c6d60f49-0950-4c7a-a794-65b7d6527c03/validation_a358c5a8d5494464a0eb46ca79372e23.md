### Title
LRTManager Can Front-Run `instantWithdrawal` to Extract Maximum Fee With No Slippage Protection - (File: contracts/LRTWithdrawalManager.sol)

### Summary
`setInstantWithdrawalFee` takes effect immediately with no timelock or grace period. The `instantWithdrawal` function accepts no minimum-output parameter, so users have no on-chain protection against a fee increase that is applied between their transaction submission and execution. The protocol's own NatSpec comment explicitly acknowledges this: *"Managers can raise it right before this call, making withdrawals cost more than expected."*

### Finding Description
`setInstantWithdrawalFee` is callable by `onlyLRTManager` and writes `instantWithdrawalFee` immediately with no delay:

```solidity
// LRTWithdrawalManager.sol line 372-376
function setInstantWithdrawalFee(uint256 feeBasisPoints) external onlyLRTManager {
    if (feeBasisPoints > 1000) revert FeeTooHigh(); // Max 10%
    instantWithdrawalFee = feeBasisPoints;
    emit InstantWithdrawalFeeUpdated(feeBasisPoints);
}
```

`instantWithdrawal` reads `instantWithdrawalFee` at execution time and applies it to the user's redeemed asset amount with no minimum-output guard:

```solidity
// LRTWithdrawalManager.sol line 237-238
uint256 fee = (assetAmountUnlocked * instantWithdrawalFee) / 10_000;
uint256 userAmount = assetAmountUnlocked - fee;
```

`setInstantWithdrawalFeeRecipient` is also callable by `onlyLRTManager` with no timelock, allowing the manager to redirect fees to an arbitrary address before front-running.

### Impact Explanation
A user who calls `instantWithdrawal` burns rsETH irrevocably (`burnFrom` executes before the fee deduction). If the fee is raised to 1000 BPS (10%) in the same block, the user receives 10% fewer assets than the rsETH they burned was worth. The difference flows to `instantWithdrawalFeeRecipient`, which the manager can pre-set to any address. This constitutes theft of unclaimed yield — **High impact**.

### Likelihood Explanation
The LRTManager role is a live operational key used for routine protocol management. Any holder of that key can execute this attack unilaterally with a single transaction. The protocol's own NatSpec comment at line 210-211 documents the risk, confirming the developers are aware the design permits it. Large instant-withdrawal transactions are visible in the public mempool, making targeting straightforward.

### Recommendation
Add a timelock or grace period before a fee increase takes effect (analogous to the recommendation in the original report). Alternatively, add a `minAssetAmountOut` parameter to `instantWithdrawal` so users can specify the minimum acceptable asset amount and revert if the fee has been raised above their tolerance.

### Proof of Concept
1. LRTManager calls `setInstantWithdrawalFeeRecipient(attackerAddress)` — no timelock, takes effect immediately.
2. User submits `instantWithdrawal(asset, largeRsETHAmount, referralId)` to the mempool.
3. LRTManager front-runs with `setInstantWithdrawalFee(1000)` (10% max).
4. User's transaction executes: rsETH is burned via `burnFrom`, `assetAmountUnlocked` is computed, then `fee = assetAmountUnlocked * 1000 / 10_000` is deducted and sent to `attackerAddress`.
5. User receives 10% fewer assets than their rsETH was worth; the difference is stolen.
6. LRTManager resets `setInstantWithdrawalFee(originalFee)` to avoid detection.
