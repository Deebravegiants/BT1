### Title
ETH Deposit Limit Bypass Due to Missing `amount` in Limit Check — (`File: contracts/LRTDepositPool.sol`)

---

### Summary
`_checkIfDepositAmountExceedesCurrentLimit` in `LRTDepositPool` omits the incoming ETH deposit amount from the cap comparison. For LST assets the check correctly uses `totalAssetDeposits + amount > depositLimit`, but for ETH it uses only `totalAssetDeposits > depositLimit`. Any depositor can therefore push the protocol's ETH balance arbitrarily above the owner-configured deposit limit in a single call to `depositETH`.

---

### Finding Description

`_checkIfDepositAmountExceedesCurrentLimit` branches on the asset type:

```solidity
// contracts/LRTDepositPool.sol  L676-L682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct
}
```

For ETH the parameter `amount` (which equals `msg.value` passed from `depositETH`) is never added to `totalAssetDeposits` before the comparison. The check only reverts when the total already **strictly exceeds** the limit, not when the incoming deposit would push it over. Because `depositETH` sends `msg.value` directly to the contract (ETH is received via `receive()` / `msg.value` and counted in `address(this).balance` which feeds `getETHDistributionData → getTotalAssetDeposits`), the balance is updated **after** the check passes, so the guard is permanently ineffective for any deposit that arrives while `totalAssetDeposits ≤ depositLimit`.

The root cause is structurally identical to the reference report: a value that should reflect the pending operation (the incoming deposit amount) is absent from the limit check, allowing the cap to be bypassed.

---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

The owner/manager sets `depositLimitByAsset[ETH_TOKEN]` via `LRTConfig` to cap how much ETH the protocol accepts. With this bug the cap is unenforceable: a single depositor can send an arbitrarily large ETH amount in one transaction and the check will pass as long as the pre-deposit total has not already exceeded the limit. The protocol will accept more ETH than intended, diluting yield for existing rsETH holders and violating the operator's risk parameters, but no direct theft of funds occurs.

---

### Likelihood Explanation

**High.** The entry point is the public, permissionless `depositETH` function. No special role, front-running, or external dependency is required. Any user who calls `depositETH` while `totalAssetDeposits ≤ depositLimit` can deposit an unbounded amount of ETH in a single transaction.

---

### Recommendation

Include the deposit amount in the ETH branch, mirroring the LST branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

---

### Proof of Concept

1. Owner sets `depositLimitByAsset[ETH_TOKEN] = 1000 ether` via `LRTConfig.updateAssetDepositLimit`.
2. Current `getTotalAssetDeposits(ETH_TOKEN)` returns `999 ether` (just under the limit).
3. Attacker calls `depositETH{value: 10_000 ether}(0, "")`.
4. Inside `_beforeDeposit` → `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 10_000 ether)`:
   - `totalAssetDeposits = 999 ether`
   - ETH branch: `999 ether > 1000 ether` → **false** → no revert.
5. `_mintRsETH` mints rsETH for the full `10_000 ether`.
6. After the call `getTotalAssetDeposits(ETH_TOKEN)` returns `10_999 ether`, **10× the configured limit**, with no revert ever triggered.

The equivalent LST check at line 681 would have returned `true` (`999 + 10_000 > 1000`) and reverted, confirming the ETH branch is uniquely broken. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
