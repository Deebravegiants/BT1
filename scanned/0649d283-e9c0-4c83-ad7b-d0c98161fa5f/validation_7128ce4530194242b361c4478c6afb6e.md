### Title
Chainlink `latestRoundData()` Missing Staleness Checks Allows Stale Price Acceptance - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all staleness-related return values (`updatedAt`, `answeredInRound`, `roundId`). A stale price silently passes through into rsETH minting and withdrawal payout calculations, enabling depositors to mint excess rsETH or withdrawers to receive incorrect asset amounts.

### Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol`, the `getAssetPrice()` function fetches the Chainlink price but ignores every field that would indicate staleness:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
``` [1](#0-0) 

Neither `answeredInRound < roundId` (incomplete/stale round) nor `updatedAt == 0` (unstarted round) nor a heartbeat-based timestamp check is performed. The only guard is `price <= 0`, which a stale-but-positive price bypasses. [2](#0-1) 

This price is returned to `LRTOracle.getAssetPrice()`, which is the sole price source for all supported LST assets. [3](#0-2) 

`LRTOracle.getAssetPrice()` is consumed in two critical paths:

1. **Deposit path** — `LRTDepositPool` calls it to compute how many rsETH tokens to mint per deposited LST.
2. **Withdrawal path** — `LRTWithdrawalManager._createUnlockParams()` calls it to determine the asset payout for a withdrawal request. [4](#0-3) 

Additionally, `LRTOracle._updateRsETHPrice()` aggregates asset prices via `_getTotalEthInProtocol()` to compute the rsETH/ETH exchange rate stored in `rsETHPrice`, which is also used in withdrawal payout calculations. [5](#0-4) 

### Impact Explanation
If a Chainlink feed goes stale (e.g., sequencer downtime, feed deprecation, or network congestion preventing updates), the last reported price is accepted as current. If the stale price is **lower** than the true market price, a depositor receives more rsETH than the deposited LST is worth, diluting existing holders. If the stale price is **higher**, withdrawers receive fewer assets than owed. In either direction, the rsETH/ETH exchange rate stored in `rsETHPrice` is corrupted, causing protocol-wide accounting errors that affect all subsequent depositors and withdrawers. This constitutes theft of yield from existing rsETH holders or underpayment to withdrawers — matching the **High: theft of unclaimed yield** and **Medium: temporary freezing of funds** impact categories.

### Likelihood Explanation
Chainlink feeds have documented heartbeat intervals (e.g., 1 hour for ETH/USD, 24 hours for some LST feeds). Any network disruption, feed migration, or sequencer outage within those windows produces a stale price that this contract silently accepts. The entry path (`depositAsset`, `depositETH`) is fully permissionless and callable by any user at any time, making exploitation straightforward whenever a feed lags.

### Recommendation
Apply the standard three-part staleness guard in `getAssetPrice()`:

```solidity
(uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

require(answeredInRound >= roundId, "Stale round");
require(updatedAt != 0, "Incomplete round");
require(block.timestamp - updatedAt <= MAX_ORACLE_FRESHNESS, "Price too stale");
require(price > 0, "Non-positive price");
```

`MAX_ORACLE_FRESHNESS` should be set per-feed based on its documented heartbeat (e.g., `3600` for a 1-hour heartbeat feed, with a small buffer).

### Proof of Concept

1. A Chainlink LST/ETH feed (e.g., stETH/ETH) stops updating due to a network event. Its `updatedAt` is now 2 hours old while the heartbeat is 1 hour.
2. The true stETH price has dropped 5% during the outage.
3. An attacker calls `LRTDepositPool.depositAsset(stETH, amount, 0)`.
4. `_beforeDeposit` → `getRsETHAmountToMint` → `LRTOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` returns the stale (inflated) price.
5. The attacker receives rsETH computed at the inflated price — more rsETH than the deposited stETH is currently worth.
6. The attacker redeems rsETH once the feed recovers, extracting value from existing holders. [2](#0-1) [3](#0-2)
