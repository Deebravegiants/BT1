Looking at the deposit functions across all pool contracts and comparing them to `LRTDepositPool`, I can identify a clear analog.

### Title
Missing Minimum Output Amount (Slippage Protection) in L2 Pool `deposit()` Functions - (File: contracts/pools/RSETHPool.sol, RSETHPoolNoWrapper.sol, RSETHPoolV2.sol, RSETHPoolV2ExternalBridge.sol, RSETHPoolV3ExternalBridge.sol, RSETHPoolV3WithNativeChainBridge.sol)

### Summary
All L2 pool `deposit()` functions accept user ETH or token inputs and return rsETH/wrsETH based on a live oracle rate, but provide no `minRsETHAmountExpected` parameter. In contrast, `LRTDepositPool.depositETH()` and `LRTDepositPool.depositAsset()` explicitly enforce a caller-supplied minimum output. Users of the L2 pools have no on-chain mechanism to bound the rsETH amount they receive, leaving them exposed to adverse rate movements between transaction submission and execution.

### Finding Description
`LRTDepositPool._beforeDeposit()` enforces a caller-supplied floor:

```solidity
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [1](#0-0) 

Every L2 pool `deposit()` function omits this check entirely. For example, `RSETHPool.deposit()`:

```solidity
function deposit(string memory referralId) external payable nonReentrant whenNotPaused {
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
``` [2](#0-1) 

The same pattern is present in `RSETHPoolNoWrapper.deposit()`: [3](#0-2) 

`RSETHPoolV2.deposit()`: [4](#0-3)

The output amount is computed from `getRate()` at execution time:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [8](#0-7) 

If the oracle rate increases (rsETH appreciates in ETH terms) between when the user signs the transaction and when it is mined, the user receives fewer rsETH with no recourse.

### Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A user who previews `viewSwapRsETHAmountAndFee()` off-chain and submits a deposit may receive materially fewer rsETH/wrsETH than the quoted amount if the oracle rate is updated before their transaction is included. The deposited ETH is not lost (it is correctly accounted for at the new rate), but the user cannot enforce the minimum output they expected. This is the same root cause as the referenced report: a swap function that accepts user funds without a caller-enforced minimum output.

### Likelihood Explanation
**Medium.** Oracle rate updates for rsETH/ETH occur regularly as staking rewards accrue and as the protocol's total asset value changes. On L2 chains with variable block times or during periods of network congestion, a pending deposit transaction can sit in the mempool long enough for the rate to shift. Any user interacting with these pools faces this risk on every deposit, with no opt-in protection available.

### Recommendation
Add a `minRsETHAmountExpected` parameter to all `deposit()` overloads in every L2 pool contract, mirroring the pattern already used in `LRTDepositPool._beforeDeposit()`:

```solidity
function deposit(uint256 minRsETHAmountExpected, string memory referralId)
    external payable nonReentrant whenNotPaused
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    ...
}
```

Apply the same pattern to the token-input `deposit(address token, uint256 amount, ...)` overloads.

### Proof of Concept
1. Alice calls `viewSwapRsETHAmountAndFee(1 ether)` off-chain and sees she will receive `X` wrsETH.
2. Alice submits `RSETHPool.deposit{value: 1 ether}("ref")`.
3. Before Alice's transaction is mined, the rsETH oracle rate is updated (rsETH/ETH rate increases).
4. Alice's transaction executes; `viewSwapRsETHAmountAndFee` now returns `X - delta` wrsETH.
5. Alice receives fewer wrsETH than she expected with no on-chain protection, because `deposit()` accepts any output amount without a floor check.
6. The same scenario applies to token deposits via `deposit(address token, uint256 amount, string memory referralId)` across all affected pool contracts.
