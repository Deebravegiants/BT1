### Title
`instantWithdrawal` drains `LRTUnstakingVault` assets reserved for queued withdrawals due to unsynchronized `queuedWithdrawalsBuffer` - (File: contracts/LRTWithdrawalManager.sol)

---

### Summary

`LRTWithdrawalManager.initiateWithdrawal()` commits future asset obligations into `assetsCommitted[asset]`, but the corresponding protection in `LRTUnstakingVault` â€” `queuedWithdrawalsBuffer[asset]` â€” is a manually-set static value that is never automatically updated. When instant withdrawals are enabled, any rsETH holder can drain the vault down to zero, making it impossible for the operator to call `unlockQueue()` and service the already-queued withdrawal requests.

---

### Finding Description

**Step 1 â€” Queued withdrawal commits assets against total protocol balance, not vault balance.**

`initiateWithdrawal()` checks availability via `getAvailableAssetAmount()`:

```solidity
// LRTWithdrawalManager.sol:599-603
function getAvailableAssetAmount(address asset) public view override returns (uint256 availableAssetAmount) {
    ILRTDepositPool lrtDepositPool = ILRTDepositPool(lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL));
    uint256 totalAssets = lrtDepositPool.getTotalAssetDeposits(asset);
    availableAssetAmount = totalAssets > assetsCommitted[asset] ? totalAssets - assetsCommitted[asset] : 0;
}
```

`getTotalAssetDeposits` aggregates assets across the deposit pool, all node delegators, and EigenLayer strategies â€” not just the `LRTUnstakingVault` balance. The committed amount is then recorded:

```solidity
// LRTWithdrawalManager.sol:170-173
if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();
assetsCommitted[asset] += expectedAssetAmount;
``` [1](#0-0) [2](#0-1) 

**Step 2 â€” `instantWithdrawal` checks only the vault's static buffer, which is never tied to `assetsCommitted`.**

```solidity

`getAssetsAvailableForInstantWithdrawal` in `LRTUnstakingVault`:

```solidity
// LRTUnstakingVault.sol:229-238
function getAssetsAvailableForInstantWithdrawal(address asset) external view ... returns (uint256 availableAmount) {
    uint256 vaultBalance = balanceOf(asset);
    uint256 reservedBuffer = queuedWithdrawalsBuffer[asset];
    availableAmount = reservedBuffer >= vaultBalance ? 0 : vaultBalance - reservedBuffer;
}
``` [3](#0-2) [4](#0-3) 

**Step 3 â€” `queuedWithdrawalsBuffer` is a static operator-set value, default zero, never updated on `initiateWithdrawal`.**

```solidity
// LRTUnstakingVault.sol:199-208
function setQueuedWithdrawalsBuffer(address asset, uint256 buffer) external onlyLRTOperator onlySupportedAsset(asset) {
    queuedWithdrawalsBuffer[asset] = buffer;
    emit QueuedWithdrawalsBufferUpdated(asset, buffer);
}
```

There is no code path that increments `queuedWithdrawalsBuffer` when `assetsCommitted[asset]` grows in `LRTWithdrawalManager`. The two state variables live in separate contracts and are never synchronized. [5](#0-4) [6](#0-5) 

**Step 4 â€” `unlockQueue` uses only the vault balance; a drained vault causes a revert.**

```solidity
// LRTWithdrawalManager.sol:846-851
function _createUnlockParams(...) internal view returns (UnlockParams memory) {
    return UnlockParams({
        ...
        totalAvailableAssets: unstakingVault.balanceOf(asset)
    });
}
```

```solidity
// LRTWithdrawalManager.sol:297
if (params.totalAvailableAssets == 0) revert AmountMustBeGreaterThanZero();
```

If the vault has been emptied by instant withdrawals, `unlockQueue` reverts unconditionally, and no queued withdrawal can be processed. [7](#0-6) [8](#0-7) 

---

### Impact Explanation

Users who called `initiateWithdrawal()` and had their rsETH burned cannot complete their withdrawal. The operator cannot call `unlockQueue()` because the vault balance is zero. The funds are temporarily frozen until the vault is refilled via EigenLayer withdrawal completions or new deposits. This constitutes **temporary freezing of funds** (Medium severity per the allowed impact scope).

---

### Likelihood Explanation

**Medium.** Two conditions must hold simultaneously:

1. `isInstantWithdrawalEnabled[asset]` is `true` â€” a manager decision that is a normal operational state once the feature is live.
2. `queuedWithdrawalsBuffer[asset]` is zero or stale â€” the default value is zero, and the buffer is never automatically updated when new queued withdrawals are added. Any period between a user calling `initiateWithdrawal()` and an operator manually calling `setQueuedWithdrawalsBuffer()` is a window of exposure. Because `initiateWithdrawal()` is permissionless and continuous, this window is always open.

No admin compromise, oracle manipulation, or governance capture is required. Any rsETH holder can trigger the condition.

---

### Recommendation

Replace the manually-managed `queuedWithdrawalsBuffer` with a dynamic check that reads `assetsCommitted[asset]` from `LRTWithdrawalManager` directly inside `getAssetsAvailableForInstantWithdrawal()`. Alternatively, expose a callback from `LRTWithdrawalManager` to `LRTUnstakingVault` that increments/decrements the buffer atomically whenever `assetsCommitted` changes (on `initiateWithdrawal`, `_unlockWithdrawalRequests`, and cancellation paths). This mirrors the fix described in the MozStaking report: the staking contract must mint/burn (or in this case, reserve/release) tokens atomically so it cannot run out.

---

### Proof of Concept

1. Vault holds 200 ETH. `queuedWithdrawalsBuffer[ETH]` = 0 (default). Total protocol assets = 1000 ETH.
2. **User A** calls `initiateWithdrawal(ETH, rsETHFor100ETH)`. Check passes (`1000 - 0 >= 100`). `assetsCommitted[ETH]` = 100 ETH. rsETH is held in `LRTWithdrawalManager`.
3. **User B** (any rsETH holder) calls `instantWithdrawal(ETH, rsETHFor200ETH)`. `getAssetsAvailableForInstantWithdrawal(ETH)` = `200 - 0` = 200 ETH. Check passes. `unstakingVault.redeem(ETH, 200 ETH)` executes. Vault balance = 0.
4. Operator calls `unlockQueue(ETH, ...)`. `_createUnlockParams` reads `unstakingVault.balanceOf(ETH)` = 0. Line 297 reverts: `AmountMustBeGreaterThanZero`.
5. User A's queued withdrawal is frozen. Their rsETH has already been transferred to `LRTWithdrawalManager` and cannot be reclaimed until the vault is refilled. [9](#0-8) [10](#0-9) [11](#0-10)
