### Title
Strict Equality Fee Check in `bridgeRsETHToL2UsingCCIP` Causes Revert on Any CCIP Fee Change, Temporarily Freezing rsETH in L1Vault - (File: contracts/L1VaultV2.sol)

---

### Summary

`bridgeRsETHToL2UsingCCIP` re-computes the CCIP fee on-chain at execution time and enforces strict equality (`msg.value != fee`). Because the CCIP router's `getFee()` is gas-price-sensitive and can change between the manager's off-chain quote and the on-chain transaction, any fee drift causes an `IncorrectCCIPFee` revert. Critically, the CCIP router itself explicitly accepts overpayment, but the contract rejects it, making even a "send more than needed" workaround impossible.

---

### Finding Description

Inside `bridgeRsETHToL2UsingCCIP`, the fee is not passed as a parameter — it is re-fetched from the router on-chain:

```solidity
// contracts/L1VaultV2.sol lines 354-358
uint256 fee = getCCIPFee(amount);   // calls ccipRouter.getFee() at execution time

if (msg.value != fee) {
    revert IncorrectCCIPFee();
}
``` [1](#0-0) 

`getCCIPFee` is a thin wrapper that delegates directly to the router:

```solidity
// contracts/L1VaultV2.sol lines 400-404
function getCCIPFee(uint256 amount) public view returns (uint256) {
    Client.EVM2AnyMessage memory message = getCCIPMessage(amount);
    return ccipRouter.getFee(destinationChainSelector, message);
}
``` [2](#0-1) 

The CCIP router interface explicitly documents that overpayment is accepted:

```solidity
// contracts/external/chainlink/IRouterClient.sol lines 39-40
/// @dev Note if msg.value is larger than the required fee (from getFee) we accept
/// the overpayment with no refund.
``` [3](#0-2) 

The contract's strict `!=` check contradicts this — it rejects both underpayment **and** overpayment, making it impossible for the manager to use a buffer to absorb fee volatility.

---

### Impact Explanation

rsETH minted and held in `L1Vault` cannot be bridged to L2 until the manager submits a transaction with `msg.value` equal to the exact current CCIP fee at the moment of execution. During periods of gas price volatility, every attempt may revert, leaving rsETH temporarily frozen in the vault. The CCIP router's own design (accepting overpayment) provides a natural mitigation path that the contract explicitly blocks.

**Impact: Medium — Temporary freezing of funds.**

---

### Likelihood Explanation

CCIP fees are derived from destination-chain gas prices, which fluctuate block-to-block. The manager must quote off-chain, construct a transaction, and submit it — a process spanning at least one block. Any gas price movement during that window causes a revert. During high-volatility periods (e.g., L2 congestion, network events), this can cause repeated failures across many blocks. Likelihood: **Medium**.

---

### Recommendation

Replace the strict equality check with a lower-bound check, and forward the actual `msg.value` to the router (which accepts overpayment):

```solidity
uint256 fee = getCCIPFee(amount);
if (msg.value < fee) {
    revert IncorrectCCIPFee();
}
// ccipRouter accepts msg.value >= fee; excess is kept by the router or refunded per its policy
bytes32 messageId = ccipRouter.ccipSend{ value: msg.value }(destinationChainSelector, message);
```

This aligns with the CCIP router's documented behavior and eliminates the race condition entirely.
