### Title
Unbounded Gas Consumption in `_getTotalEthInProtocol()` Due to Unbounded `supportedAssetList` Iteration — (`contracts/LRTOracle.sol`)

---

### Summary

`LRTOracle._getTotalEthInProtocol()` iterates over the entire `supportedAssetList` array from `LRTConfig`, and for each asset calls `LRTDepositPool.getTotalAssetDeposits()` which itself iterates over `nodeDelegatorQueue`. There is no upper bound on `supportedAssetList`. As the protocol adds more supported assets over time, the gas cost of `updateRSETHPrice()` grows without limit, eventually causing an out-of-gas revert that permanently prevents price updates.

---

### Finding Description

`LRTConfig.supportedAssetList` is a dynamic array that grows each time `_addNewSupportedAsset()` is called, with no maximum length enforced: [1](#0-0) 

`LRTOracle._getTotalEthInProtocol()` fetches this entire array and loops over every element: [2](#0-1) 

For each asset in the loop, it calls `ILRTDepositPool.getTotalAssetDeposits(asset)`, which calls `getAssetDistributionData()`, which itself loops over the entire `nodeDelegatorQueue`: [3](#0-2) 

The total gas cost is therefore **O(supportedAssets × nodeDelegators)**. `_getTotalEthInProtocol()` is called inside `_updateRsETHPrice()`: [4](#0-3) 

which is triggered by the publicly callable `updateRSETHPrice()`: [5](#0-4) 

---

### Impact Explanation

When `updateRSETHPrice()` reverts due to out-of-gas:

1. **rsETH price becomes permanently stale.** `rsETHPrice` is the denominator used in `getRsETHAmountToMint()` and `_beforeDeposit()`, so all new deposits mint rsETH at an incorrect (stale) rate — a share/asset mis-accounting impact.
2. **Fee minting stops.** Protocol yield accrual via `IRSETH.mint(treasury, ...)` inside `_updateRsETHPrice()` can no longer execute — permanent freezing of unclaimed yield.
3. **Downside price-protection auto-pause stops working.** The circuit-breaker that pauses the protocol on a price drop relies on `_updateRsETHPrice()` completing successfully.

**Impact classification:** Medium — Unbounded gas consumption leading to permanent freezing of unclaimed yield and stale price accounting.

---

### Likelihood Explanation

The protocol is explicitly designed to support multiple LST/LRT assets (stETH, ETHx, and future additions). `addNewSupportedAsset` is gated by `TIME_LOCK_ROLE` and is a routine governance operation, not an attack. As the protocol expands its asset list — a stated design goal — the gas cost of `updateRSETHPrice()` grows linearly per asset and quadratically when combined with NDC count growth. At a realistic scale of 20–30 supported assets and 10 NDCs (the default `maxNodeDelegatorLimit`), the function becomes prohibitively expensive. This is the same root cause confirmed in the YetiFinance M-04 finding.

---

### Recommendation

1. Enforce a maximum length on `supportedAssetList` in `_addNewSupportedAsset()`, analogous to the existing `maxNodeDelegatorLimit` guard on `nodeDelegatorQueue`.
2. Consider caching per-asset TVL and updating it incrementally rather than recomputing the full sum on every `updateRSETHPrice()` call.
3. Alternatively, split `_getTotalEthInProtocol()` into a paginated or off-chain-assisted pattern.

---

### Proof of Concept

**Call chain:**

```
updateRSETHPrice()                          [public, no access control]
  └─ _updateRsETHPrice()
       └─ _getTotalEthInProtocol()
            └─ for each asset in supportedAssetList:          [unbounded]
                 └─ getTotalAssetDeposits(asset)
                      └─ getAssetDistributionData(asset)
                           └─ for each NDC in nodeDelegatorQueue:  [up to maxNodeDelegatorLimit]
                                └─ IERC20.balanceOf()
                                └─ INodeDelegator.getAssetBalance()
                                └─ INodeDelegator.getAssetUnstaking()
```

**Concrete scenario:**
- Protocol adds 30 supported assets over time (no cap enforced).
- `maxNodeDelegatorLimit` is raised to 20 by admin (no cap on this value either).
- Each `updateRSETHPrice()` call now executes 30 × 20 = 600 external calls.
- At ~5,000 gas per external `balanceOf`/`getAssetBalance` call, this exceeds 3,000,000 gas — approaching or exceeding the block gas limit on L2s where the protocol operates.
- `updateRSETHPrice()` reverts with OOG on every call; rsETH price is permanently frozen. [6](#0-5) [7](#0-6) [3](#0-2)

```text
    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
```
