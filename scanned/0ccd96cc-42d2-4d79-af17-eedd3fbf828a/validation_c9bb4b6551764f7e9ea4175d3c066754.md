Looking at the key contracts to trace the exact exploit path.

### Title
Stale `rsETHPrice` Combined with Live `swETH.getRate()` Enables Excess rsETH Minting via Block Stuffing — (`contracts/LRTOracle.sol`, `contracts/oracles/SwETHPriceOracle.sol`)

---

### Summary

`LRTDepositPool.getRsETHAmountToMint()` divides a **live** swETH rate (fetched on-chain from `swETH.getRate()`) by a **stored** `rsETHPrice` that is only updated when `updateRSETHPrice()` is called. An attacker who stuffs blocks to delay that update can deposit swETH while `rsETHPrice` is stale-low, receiving more rsETH than the true backing ratio warrants, diluting existing holders.

---

### Finding Description

**Mint formula** in `LRTDepositPool`:

```solidity
// LRTDepositPool.sol line 520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [1](#0-0) 

- **Numerator** — `lrtOracle.getAssetPrice(swETH)` delegates to `SwETHPriceOracle.getAssetPrice()`, which calls `ISwETH(swETHAddress).getRate()` live every time. [2](#0-1) 

- **Denominator** — `lrtOracle.rsETHPrice()` is a **storage variable** written only inside `_updateRsETHPrice()`, which is triggered by the public `updateRSETHPrice()` or the manager-only `updateRSETHPriceAsManager()`. [3](#0-2) [4](#0-3) 

Because `swETH.getRate()` accrues continuously (it reflects staking rewards) while `rsETHPrice` is only updated on explicit calls, any gap between updates creates a window where the denominator is stale-low relative to the live numerator.

**Block stuffing amplifies this gap.** An attacker fills blocks with high-gas transactions (paying at least the base fee per gas unit) to crowd out `updateRSETHPrice()` calls. During the stuffed window, `swETH.getRate()` keeps rising while `rsETHPrice` stays frozen. After the stuffing ends, the attacker calls `depositAsset(swETH, largeAmount, 0, "")` and receives:

```
rsethMinted = largeAmount × liveSwETHRate / staleRsETHPrice
            > largeAmount × liveSwETHRate / correctRsETHPrice
```

**Additional complication — `pricePercentageLimit` can lock out public callers.** If the accumulated price increase exceeds `pricePercentageLimit`, the public `updateRSETHPrice()` reverts for non-managers:

```solidity
// LRTOracle.sol lines 260-265
if (isPriceIncreaseOffLimit) {
    if (!IAccessControl(address(lrtConfig)).hasRole(LRTConstants.MANAGER, msg.sender)) {
        revert PriceAboveDailyThreshold();
    }
}
``` [5](#0-4) 

This means that after a sufficiently long stuffing window, even the end of the attack does not automatically restore the price — only the manager can do so via `updateRSETHPriceAsManager()`. Until the manager acts, every `depositAsset(swETH)` call continues to mint at the stale-low price.

---

### Impact Explanation

Excess rsETH is minted to the attacker. Because rsETH is a share token backed by a fixed pool of assets, minting more shares than the deposit warrants dilutes every existing holder's claim on the underlying ETH. The attacker extracts value from existing holders proportional to `(correctRsETHPrice − staleRsETHPrice) / correctRsETHPrice × depositAmount`. This maps to **Low — Block stuffing** (attacker pays block-stuffing costs to achieve a profitable mint at a stale price).

---

### Likelihood Explanation

- `updateRSETHPrice()` is public but not called atomically inside `depositAsset()`, so any gap between updates is exploitable without stuffing.
- Block stuffing on Ethereum mainnet is expensive but economically rational for large deposit amounts; the attacker only needs to stuff enough blocks for swETH rate to accrue meaningfully (swETH accrues ~4–5% APR, so even a few hours of stuffing yields a measurable delta).
- The `pricePercentageLimit` guard paradoxically extends the exploitable window by preventing public price correction after the stuffing ends.

---

### Recommendation

1. **Call `updateRSETHPrice()` (or its internal equivalent) at the start of `depositAsset()` and `depositETH()`** so the denominator is always fresh at the moment of minting. This eliminates the stale-price window entirely.
2. Alternatively, compute `rsethAmountToMint` using a freshly computed price (calling `_getTotalEthInProtocol()` inline) rather than the stored `rsETHPrice`.
3. If on-demand price updates are too gas-expensive, add a **maximum staleness check** (e.g., revert if `rsETHPrice` was last updated more than N blocks ago) to bound the exploitable window.

### Proof of Concept

```solidity
// Fork test outline (Foundry)
// 1. Deploy mock swETH whose getRate() returns baseRate + blockNumber * ratePerBlock
// 2. Record rsETHPrice = lrtOracle.rsETHPrice()  (set at last updateRSETHPrice call)
// 3. vm.roll(block.number + N)  // simulate N blocks without updateRSETHPrice
// 4. liveRate = mockSwETH.getRate()  // now higher than at step 2
// 5. attacker calls depositAsset(swETH, AMOUNT, 0, "")
// 6. rsethMinted = lrtDepositPool.getRsETHAmountToMint(swETH, AMOUNT)
//    = AMOUNT * liveRate / staleRsETHPrice
// 7. correctMint = AMOUNT * liveRate / correctRsETHPrice
//    where correctRsETHPrice is computed by calling _getTotalEthInProtocol() / rsethSupply
// 8. assert rsethMinted > correctMint  // attacker received excess rsETH
```

The assertion at step 8 will hold for any `N > 0` where `swETH.getRate()` has increased since the last `updateRSETHPrice()` call, confirming the invariant break without requiring any privileged access.
