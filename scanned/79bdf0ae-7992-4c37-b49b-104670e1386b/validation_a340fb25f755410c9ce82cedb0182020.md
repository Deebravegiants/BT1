### Title
Fee Calculation Rounding Always Truncates Down, Systematically Under-Collecting Protocol Fees - (File: contracts/pools/RSETHPool.sol, RSETHPoolV3.sol, RSETHPoolV3ExternalBridge.sol, RSETHPoolNoWrapper.sol)

---

### Summary

Every L2 pool contract computes the swap fee using integer division that always truncates toward zero. Because Solidity's `/` operator floors the result, the fee charged to each depositor is always slightly less than the mathematically correct value. The shortfall accumulates across every deposit, causing the protocol to permanently under-collect its fee revenue.

---

### Finding Description

All four L2 pool contracts share the same fee computation pattern inside `viewSwapRsETHAmountAndFee`:

```solidity
// ETH path (identical in all four contracts)
fee = amount * feeBps / 10_000;          // integer division → always floors
uint256 amountAfterFee = amount - fee;   // slightly larger than intended
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

The same truncation applies to the token-deposit path in every contract:

```solidity
fee = amount * feeBpsForToken / 10_000;  // also floors
``` [5](#0-4) [6](#0-5) 

Because `fee` is floored, `amountAfterFee` is always at least 1 wei larger than the protocol intends. The depositor therefore receives a marginally larger `rsETHAmount` than the stated fee rate entitles them to, and `feeEarnedInETH` / `feeEarnedInToken` accumulates less than the true fee owed. [7](#0-6) 

The same structural rounding error appears in `LRTOracle._updateRsETHPrice`, where the protocol fee in ETH is also floored:

```solidity
protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
``` [8](#0-7) 

This causes `newRsETHPrice` to be computed slightly higher than it should be, and `rsethAmountToMintAsProtocolFee` to be floored again:

```solidity
uint256 rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice);
``` [9](#0-8) 

`divWad` delegates to OpenZeppelin `mulDiv`, which also truncates toward zero. [10](#0-9) 

---

### Impact Explanation

**Impact: Low — Contract fails to deliver promised returns, but does not lose principal.**

For every deposit the protocol collects up to 1 wei less fee than the configured `feeBps` rate mandates. The shortfall is permanent: the under-collected wei is silently absorbed into the depositor's `amountAfterFee`, inflating the wrsETH minted to them. Across a high-volume L2 pool (thousands of deposits per day), the cumulative under-collection is non-trivial over weeks or months. The fee revenue that flows to the protocol treasury via `withdrawFees` is therefore systematically lower than the rate advertised to governance and users. [11](#0-10) 

---

### Likelihood Explanation

**Likelihood: High.**

The rounding error is triggered on every single deposit call (`deposit(string)` and `deposit(address,uint256,string)`) across all deployed L2 pool contracts. No special conditions, attacker privileges, or unusual inputs are required. Any unprivileged depositor causes the error simply by calling the public deposit function with any non-zero amount that is not an exact multiple of `10_000 / feeBps`. [12](#0-11) 

---

### Recommendation

Round the fee **up** (ceiling division) so that the protocol always collects at least the configured basis-point rate:

```solidity
// Replace:
fee = amount * feeBps / 10_000;

// With ceiling division:
fee = (amount * feeBps + 9_999) / 10_000;
```

Apply the same fix to the `protocolFeeInETH` calculation in `LRTOracle._updateRsETHPrice` and to `rsethAmountToMintAsProtocolFee` (use `mulDiv` with `Math.Rounding.Up`). This ensures rounding always favors the protocol rather than the depositor, consistent with the recommendation in the referenced audit report.

---

### Proof of Concept

**Concrete numeric example (ETH deposit, `feeBps = 30`, `rsETHToETHrate = 1.1e18`):**

```
amount          = 1_000_000_000_000_000_001  (1 ETH + 1 wei)

Correct fee     = 1_000_000_000_000_000_001 * 30 / 10_000
                = 3_000_000_000_000_000.003  → should be 3_000_000_000_000_001 (ceiling)

Actual fee      = 3_000_000_000_000_000      (floored — 1 wei short)

amountAfterFee  = 1_000_000_000_000_000_001 - 3_000_000_000_000_000
                = 997_000_000_000_000_001    (1 wei more than intended)

rsETHAmount     = 997_000_000_000_000_001 * 1e18 / 1.1e18
                = 906_363_636_363_636_364    (user receives 1 extra unit of rsETH
                                              compared to the correct fee path)
```

Per deposit the shortfall is at most 1 wei of fee. With 100,000 deposits per day across all L2 pools, the daily under-collection reaches ~100,000 wei (~0.0000001 ETH). Over a year this compounds to ~36,500,000 wei. At scale, or if `feeBps` is large, the absolute loss grows proportionally.
