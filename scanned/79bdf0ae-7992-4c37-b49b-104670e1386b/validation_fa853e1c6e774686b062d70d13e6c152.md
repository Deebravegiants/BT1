### Title
LineaMessenger `minimumFee` Silently Reduces L1 Delivery While Pool Event Over-Reports Bridged Amount - (File: contracts/bridges/LineaMessenger.sol)

---

### Summary

`RSETHPoolV2ExternalBridge.bridgeAssetsViaNativeBridge` emits `BridgedETHToL1ViaNativeBridge(l1VaultETHForL2Chain, amount)` reporting `amount` ETH bridged, but `LineaMessenger.sendETHToL1ViaBridge` passes `minimumFee` as the relayer fee to Linea's native bridge, causing the L1 vault to receive only `amount - minimumFee`. The `minimumFee` is silently consumed by the bridge relayer and is never tracked or reconciled on either side.

---

### Finding Description

**Step 1 — Pool computes bridgeable amount without accounting for `minimumFee`:**

`getETHBalanceMinusFees()` returns `address(this).balance - feeEarnedInETH`. It subtracts only the pool's own protocol fee accumulator; it does not subtract the upcoming Linea bridge relayer fee. [1](#0-0) 

**Step 2 — Pool calls messenger with full `amount` and emits the same `amount`:**

`bridgeAssetsViaNativeBridge(uint256 amount)` forwards `amount` ETH to the messenger and immediately emits `BridgedETHToL1ViaNativeBridge(l1VaultETHForL2Chain, amount)` — before any knowledge of what the bridge will actually deliver. [2](#0-1) 

**Step 3 — LineaMessenger deducts `minimumFee` from the delivered value:**

Inside `sendETHToL1ViaBridge`, the contract fetches `minimumFee = ILineaMessageService(l2bridge).minimumFeeInWei()` and passes it as the relayer fee argument to `sendMessage`. Linea's bridge contract delivers `msg.value - _fee` = `amount - minimumFee` to the L1 target; the `minimumFee` is paid to the postman/relayer and is gone. [3](#0-2) 

**Step 4 — The interface confirms `_fee` is deducted from the bridged value:**

`ILineaMessageService.sendMessage(address _to, uint256 _fee, bytes memory _calldata)` is documented as "The fee to be paid for the message," meaning the L1 recipient receives `msg.value - _fee`. [4](#0-3) 

**Net result per bridge call:**

| What the pool emits | What L1 vault receives | Untracked loss |
|---|---|---|
| `amount` | `amount - minimumFee` | `minimumFee` |

This discrepancy is permanent and accumulates with every bridge call.

---

### Impact Explanation

If the L1 accounting layer (rsETH rate oracle / TVL tracker) ingests the `BridgedETHToL1ViaNativeBridge` event to credit received ETH, it will credit `amount` while the vault only holds `amount - minimumFee`. The rsETH/ETH rate is computed against an over-stated ETH backing. Users redeeming rsETH are entitled to more ETH than actually exists in the vault; the shortfall (`minimumFee` × number of bridge calls) is permanently frozen — it is accounted for in the rate but can never be delivered. This matches **Medium — Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

- The `BRIDGER_ROLE` operator calls `bridgeAssetsViaNativeBridge` as part of normal, routine operations — no adversarial action is required.
- Linea's `minimumFeeInWei` is a live, non-zero value on mainnet (currently in the range of ~0.001 ETH or higher depending on network conditions).
- The discrepancy is deterministic and occurs on every single native-bridge call; it is not edge-case behavior.

---

### Recommendation

Two complementary fixes:

1. **Deduct `minimumFee` before emitting the event.** Have `LineaMessenger` return the actual delivered amount (`value - minimumFee`) and use that in the pool's `emit`.

2. **Alternatively, subtract `minimumFee` from `amount` before calling the messenger**, so the pool sends `amount - minimumFee` ETH and emits `amount - minimumFee`. The pool should query `ILineaMessageService(l2Bridge).minimumFeeInWei()` (or expose it via the messenger) before computing the bridgeable amount.
