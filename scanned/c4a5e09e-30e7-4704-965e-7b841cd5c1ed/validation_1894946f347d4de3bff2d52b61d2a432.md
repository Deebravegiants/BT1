### Title
Deposit fee (`feeBps`) changeable without timelock enables admin front-running of user deposits with no slippage protection — (`contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPool.sol`)

---

### Summary

All L2 pool `deposit()` functions compute the output `wrsETH` amount using the live `feeBps` storage variable at execution time. Unlike other sensitive configuration functions (oracle changes, enabling deposits, adding tokens), `setFeeBps()` is gated only by `DEFAULT_ADMIN_ROLE` with no timelock. No `minAmountOut` parameter exists in any `deposit()` overload, so users have zero slippage protection. An admin can front-run a pending deposit transaction by raising `feeBps` to the maximum, causing the user to receive far fewer `wrsETH` tokens than expected while the excess is captured as protocol fees.

---

### Finding Description

In `RSETHPoolV3.deposit()` (and every pool variant), the output amount is computed entirely from live contract state:

```solidity
// RSETHPoolV3.sol – deposit(string)
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
wrsETH.mint(msg.sender, rsETHAmount);
```

`viewSwapRsETHAmountAndFee` reads `feeBps` directly from storage:

```solidity
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

`feeBps` is mutable via `setFeeBps()`, which requires only `DEFAULT_ADMIN_ROLE`:

```solidity
// RSETHPoolV3.sol line 518
function setFeeBps(uint256 _feeBps) external onlyRole(DEFAULT_ADMIN_ROLE) {
    if (_feeBps > 1000) revert InvalidFeeAmount();   // max 10%
    feeBps = _feeBps;
    emit FeeBpsSet(_feeBps);
}
```

In the older `RSETHPool.sol`, the cap is 10,000 BPS (100%):

```solidity
// RSETHPool.sol line 574
function setFeeBps(uint256 _feeBps) external onlyRole(DEFAULT_ADMIN_ROLE) {
    if (_feeBps > 10_000) revert InvalidFeeAmount();  // max 100%
    feeBps = _feeBps;
    emit FeeBpsSet(_feeBps);
}
```

By contrast, every other sensitive setter in `RSETHPoolV3` requires `TIMELOCK_ROLE`:

```solidity
function setIsEthDepositEnabled(bool _isEthDepositEnabled) external onlyRole(TIMELOCK_ROLE) { ... }
function setRSETHOracle(address _rsETHOracle) external onlyRole(TIMELOCK_ROLE) { ... }
function addSupportedToken(address token, address oracle) external onlyRole(TIMELOCK_ROLE) { ... }
```

The protocol deliberately placed a timelock on oracle and token-list changes to protect users from sudden parameter shifts, but omitted the same protection for fee changes. The `deposit()` function accepts no `minAmountOut` argument in any variant (`RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPoolNoWrapper`), so users cannot express a minimum acceptable output.

---

### Impact Explanation

**RSETHPool (legacy, 100% cap):** Admin raises `feeBps` to 10,000 BPS immediately before a deposit transaction executes. The entire deposited ETH/token amount is captured as `feeEarnedInETH` / `feeEarnedInToken`. The user receives 0 `wrsETH`. The fee is later withdrawn by the `BRIDGER_ROLE` to an arbitrary receiver. This is **direct theft of user funds** → **Critical**.

**RSETHPoolV3 (current, 10% cap):** Admin raises `feeBps` to 1,000 BPS. A user depositing 100 ETH expecting ~1% fee (10 BPS) instead loses 10 ETH to fees. Repeated across many users this constitutes systematic **theft of deposited value** → **High**.

---

### Likelihood Explanation

The attack requires the `DEFAULT_ADMIN_ROLE` holder to act maliciously or be compromised. However, the protocol's own design acknowledges this risk class by timelocking every other sensitive setter. The absence of a timelock on `setFeeBps` is an inconsistency, not a deliberate trust grant. On L2 networks (Arbitrum, Optimism, Base, Linea) where these pools are deployed, block times are short and mempool visibility is high, making front-running straightforward. Any user depositing a large amount is a viable target.

---

### Recommendation

1. Gate `setFeeBps()` with `TIMELOCK_ROLE` (consistent with all other sensitive setters in the same contract).
2. Add a `minAmountOut` parameter to all `deposit()` overloads and revert if `rsETHAmount < minAmountOut`, giving users slippage protection regardless of fee or rate changes.

---

### Proof of Concept

1. User submits `deposit{value: 100 ether}("ref")` to `RSETHPool` with `feeBps = 10` (0.1% fee, expecting ~99.9 ETH worth of wrsETH).
2. Admin observes the pending transaction in the mempool and front-runs it with `setFeeBps(10_000)` (100% fee). No timelock delay applies.
3. User's deposit executes: `fee = 100 ether * 10_000 / 10_000 = 100 ether`, `amountAfterFee = 0`, `rsETHAmount = 0`.
4. User receives 0 wrsETH; `feeEarnedInETH += 100 ether`.
5. Admin calls `withdrawFees(receiver)` (via `BRIDGER_ROLE`) to extract all 100 ETH.
