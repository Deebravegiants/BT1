### Title
ETH Deposit Limit Bypass via Missing `msg.value` in Limit Check — (`contracts/LRTDepositPool.sol`)

---

### Summary

`_checkIfDepositAmountExceedesCurrentLimit` applies a fundamentally different (and broken) comparison for ETH versus ERC20 assets. The ETH branch omits the incoming deposit amount from the comparison, allowing any EOA to deposit an arbitrary amount of ETH beyond the configured limit in a single transaction.

---

### Finding Description

In `_checkIfDepositAmountExceedesCurrentLimit`:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← no `+ amount`
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct for ERC20
}
``` [1](#0-0) 

For ERC20 assets the check is `totalAssetDeposits + amount > limit` — correctly accounting for the incoming deposit. For ETH the check is only `totalAssetDeposits > limit`, completely ignoring `msg.value`. This means:

- If `totalAssetDeposits == depositLimit - 1 wei`, the guard returns `false` (not exceeded).
- `depositETH` proceeds and mints rsETH for any `msg.value`, even one that is orders of magnitude larger than the remaining headroom.

The `getTotalAssetDeposits` for ETH aggregates the deposit pool balance, NDC balances, EigenLayer pod shares, unstaking amounts, the unstaking vault balance, and `LRTConverter.ethValueInWithdrawal()`: [2](#0-1) 

None of these components are manipulable by an unprivileged caller — `ethValueInWithdrawal` is only modified by `onlyAssetTransferRole` and `onlyLRTOperator` functions in `LRTConverter`. The bypass therefore does not require any privileged collusion; it is purely a function of the broken comparison in the ETH branch.

---

### Impact Explanation

The ETH deposit limit is a risk-management cap (e.g., limiting EigenLayer slashing exposure). Because the incoming `msg.value` is never added to `totalAssetDeposits` before the comparison, a single EOA transaction can push total ETH deposits far beyond the configured limit. The deposited ETH does back the newly minted rsETH (no direct theft), but:

- The protocol's intended risk ceiling is silently violated.
- All subsequent depositors and withdrawers operate against a pool that is over-exposed relative to the governance-approved limit.
- In a slashing event the loss is proportionally larger than the limit was designed to tolerate, which can render the protocol insolvent relative to rsETH redemption obligations.

The impact maps to **Low — Contract fails to deliver promised returns** (deposit limit not enforced) with a realistic path to **Medium — Temporary freezing of funds** if the over-deposit triggers downstream accounting anomalies, and to **Critical — Protocol insolvency** in a combined slashing scenario. The direct, standalone impact without an external slashing event is deposit-limit bypass.

---

### Likelihood Explanation

- Requires no special role, no front-running, no oracle manipulation.
- Any EOA can trigger it by calling `depositETH` with `msg.value` set to any amount while `totalAssetDeposits < depositLimit`.
- The probe condition ("deposit limit minus 1 wei") is the exact boundary that makes the broken check pass.

---

### Recommendation

Add `msg.value` (the `amount` parameter) to `totalAssetDeposits` in the ETH branch, mirroring the ERC20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [3](#0-2)

### Proof of Concept

```solidity
// Assume depositLimit = 100_000 ether, totalAssetDeposits = 99_999.999...999 ether (limit - 1 wei)
// _checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 50_000 ether):
//   totalAssetDeposits (≈ limit - 1 wei) > limit  →  false  →  deposit allowed
// Result: 50_000 ETH deposited, totalAssetDeposits now ≈ 150_000 ether (1.5× the limit)
// rsETH minted for the full 50_000 ETH; deposit limit completely bypassed.

function testETHDepositLimitBypass() public {
    uint256 limit = lrtConfig.depositLimitByAsset(LRTConstants.ETH_TOKEN);
    // Bring total to limit - 1 wei via normal deposits
    vm.deal(alice, limit - 1);
    vm.prank(alice);
    depositPool.depositETH{value: limit - 1}(0, "");

    // Now bypass: deposit 50_000 ETH in one call
    vm.deal(attacker, 50_000 ether);
    vm.prank(attacker);
    depositPool.depositETH{value: 50_000 ether}(0, "");  // succeeds — limit not enforced

    assertGt(depositPool.getTotalAssetDeposits(LRTConstants.ETH_TOKEN), limit);
}
```

The `_checkIfDepositAmountExceedesCurrentLimit` ETH branch returns `false` (not exceeded) for any `msg.value` as long as the pre-call total is at or below the limit, so the deposit and mint proceed unconditionally. [1](#0-0)
