### Title
Permanent Freezing of Unclaimed Token Fees via `removeSupportedToken` Not Deleting `feeEarnedInToken` — (`contracts/pools/RSETHPool.sol`)

---

### Summary

`removeSupportedToken` guards removal with a `balanceOf == 0` check but does **not** delete `feeEarnedInToken[token]`. For a rebasing token whose on-chain balance can drop to zero independently of the accounting variable, this allows the token to be de-listed while accumulated fees remain permanently locked, because the only withdrawal path (`withdrawFees(receiver, token)`) requires the token to still be supported.

---

### Finding Description

**`removeSupportedToken` (lines 660–670):**

```solidity
function removeSupportedToken(address token, uint256 tokenIndex) external onlyRole(TIMELOCK_ROLE) {
    UtilLib.checkNonZeroAddress(token);
    if (supportedTokenList[tokenIndex] != token) revert TokenNotFoundError();
    if (IERC20(token).balanceOf(address(this)) != 0) revert TokenBalanceNotZero();  // L663

    delete supportedTokenOracle[token];   // L665
    delete tokenBridge[token];            // L666
    supportedTokenList[tokenIndex] = supportedTokenList[supportedTokenList.length - 1];
    supportedTokenList.pop();
    emit RemovedSupportedToken(token);
}
``` [1](#0-0) 

The guard at line 663 checks the **raw ERC-20 balance** of the contract, not the accounting variable `feeEarnedInToken[token]`. For a rebasing token (e.g., one that can experience negative rebases), the on-chain balance can fall to zero while `feeEarnedInToken[token]` remains non-zero from previously accumulated swap fees. The function then deletes `supportedTokenOracle[token]` and `tokenBridge[token]` but **never touches `feeEarnedInToken[token]`**.

**`withdrawFees(address, address)` (lines 428–443):**

```solidity

    address receiver,
    address token

    external

    onlySupportedToken(token)   // <-- blocks if oracle == address(0)

    uint256 amountToSendInToken = feeEarnedInToken[token];
    feeEarnedInToken[token] = 0;
    IERC20(token).safeTransfer(receiver, amountToSendInToken);
    emit FeesWithdrawn(amountToSendInToken, token);

**`onlySupportedToken` modifier (lines 100–103):**

```solidity
modifier onlySupportedToken(address token) {
    if (supportedTokenOracle[token] == address(0)) revert UnsupportedToken();
    _;
}
``` [3](#0-2) 

After `removeSupportedToken` executes, `supportedTokenOracle[token] == address(0)`, so every call to `withdrawFees(receiver, token)` reverts with `UnsupportedToken`. There is no rescue function, no admin bypass, and no alternative path to recover `feeEarnedInToken[token]`.

---

### Impact Explanation

Accumulated swap fees denominated in the removed token are permanently frozen in the contract. The `feeEarnedInToken[token]` storage slot retains its non-zero value but is unreachable through any callable function. This matches the allowed impact: **Medium — Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

**Low.** The precondition requires:
1. A rebasing token to be added via `addSupportedToken` (requires `TIMELOCK_ROLE` — a legitimate but deliberate admin action, not a compromise).
2. Swap fees to accumulate in that token.
3. The token's on-chain balance at the contract address to reach exactly zero via a negative rebase event (external, but not impossible for tokens like stETH under slashing scenarios, or any custom rebasing collateral).
4. The `TIMELOCK_ROLE` holder to then call `removeSupportedToken`, which passes the `balanceOf == 0` guard.

The current production tokens (wstETH, etc.) are non-rebasing, so this does not affect the current deployment. However, the contract places no restriction on rebasing tokens being added, and the code defect is unconditional.

---

### Recommendation

In `removeSupportedToken`, add a check that `feeEarnedInToken[token] == 0` before allowing removal, or delete the mapping entry as part of the removal:

```solidity
// Option A: guard
if (feeEarnedInToken[token] != 0) revert FeeBalanceNotZero();

// Option B: clear alongside oracle/bridge
delete feeEarnedInToken[token];
```

Option A is safer because it forces the operator to withdraw fees before de-listing, preserving the invariant that `feeEarnedInToken[token] > 0` implies the token is still supported.
