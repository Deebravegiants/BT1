### Title
Aave Interest Trapped in `_withdrawFromAave` Causes Temporary ETH Withdrawal Freeze - (File: contracts/LRTWithdrawalManager.sol)

### Summary
`LRTWithdrawalManager` deposits unlocked ETH into Aave v3 to earn yield. The internal `_withdrawFromAave` function deliberately caps withdrawals at `totalETHDepositedToAave` (principal only), refusing to touch accrued interest. Once all principal is consumed by sequential user `completeWithdrawal` calls, `totalETHDepositedToAave` reaches zero while aWETH interest balance remains non-zero. Any subsequent user withdrawal that requires ETH from Aave silently receives 0 ETH and reverts with `InsufficientLiquidityForWithdrawal`, temporarily freezing those users' funds even though ETH is sitting in Aave.

### Finding Description
`_withdrawFromAave` computes a `withdrawablePrincipal` ceiling:

```solidity
uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave
    ? aaveBalance
    : totalETHDepositedToAave;

withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
if (withdrawnAmount == 0) return 0;   // silent return, no revert
``` [1](#0-0) 

When `totalETHDepositedToAave == 0` but `aaveAWETH.balanceOf(address(this)) > 0` (interest only remains), `withdrawablePrincipal = 0`, so `withdrawnAmount = 0` and the function returns silently without reverting.

The caller `_processWithdrawalCompletion` then checks:

```solidity
uint256 balanceAfter = address(this).balance;
if (balanceAfter < request.expectedAssetAmount) {
    revert InsufficientLiquidityForWithdrawal();
}
``` [2](#0-1) 

Because `_withdrawFromAave` returned 0, `balanceAfter` is unchanged and the revert fires, blocking the user's withdrawal.

The lifecycle that produces this state:

1. `unlockQueue` redeems ETH from `LRTUnstakingVault` and deposits it to Aave via `_depositToAave`, incrementing `totalETHDepositedToAave`. [3](#0-2) 

2. Users call `completeWithdrawal`; each call that needs Aave liquidity calls `_withdrawFromAave`, decrementing `totalETHDepositedToAave`. [4](#0-3) 

3. After all principal is consumed, `totalETHDepositedToAave == 0`. Aave has accrued interest (aWETH balance > 0), but `_withdrawFromAave` returns 0 for any amount.

4. The only path to recover that interest is `_collectInterestToTreasury`, which sends it to the protocol treasury — not to the withdrawal manager — so it cannot serve pending user withdrawals. [5](#0-4) 

### Impact Explanation
Users with unlocked ETH withdrawal requests whose `expectedAssetAmount` exceeds the contract's idle ETH balance cannot complete their withdrawals. Their rsETH has already been burned (at `unlockQueue` time) and the ETH they are owed is locked in Aave as interest, inaccessible to them. This constitutes a **temporary freeze of user funds** until the operator deposits fresh ETH into the withdrawal manager through another `unlockQueue` cycle.

**Impact: Medium — Temporary freezing of funds.**

### Likelihood Explanation
The Aave integration is an active, enabled feature. As users progressively complete withdrawals, `totalETHDepositedToAave` monotonically decreases toward zero while interest silently accumulates. This is a normal operating condition, not an edge case. Any deployment where the Aave integration is enabled and withdrawal volume is high will eventually reach this state.

**Likelihood: Medium.**

### Recommendation
In `_withdrawFromAave`, when `totalETHDepositedToAave` is zero but `aaveBalance > 0`, allow withdrawal of the full `aaveBalance` (treating it as interest that can serve user withdrawals), or alternatively call `_collectInterestToTreasury` first inside `_processWithdrawalCompletion` before attempting `_withdrawFromAave`, so that the interest is converted to ETH in the contract and available for the user. A simpler fix is to remove the principal cap in `_withdrawFromAave` and instead track interest separately, or allow `_processWithdrawalCompletion` to withdraw up to the full `aaveBalance` when principal is exhausted.

### Proof of Concept
1. Operator calls `unlockQueue(ETH_TOKEN, ...)` — 100 ETH moves from `LRTUnstakingVault` to `LRTWithdrawalManager`, then is deposited to Aave. `totalETHDepositedToAave = 100`.
2. Aave accrues 2 ETH interest. `aaveAWETH.balanceOf(withdrawalManager) = 102`.
3. Users call `completeWithdrawal` repeatedly. Each call triggers `_withdrawFromAave`, decrementing `totalETHDepositedToAave`. After 100 ETH of principal is consumed, `totalETHDepositedToAave = 0`, `aaveBalance = 2`.
4. Next user calls `completeWithdrawal(ETH_TOKEN, ...)` for 1 ETH. Contract ETH balance = 0.
5. `_processWithdrawalCompletion` calls `_withdrawFromAave(1)`.
6. `withdrawablePrincipal = min(2, 0) = 0` → `withdrawnAmount = 0` → returns 0.
7. `balanceAfter = 0 < 1` → reverts `InsufficientLiquidityForWithdrawal`.
8. The user's withdrawal is frozen. The 2 ETH of interest in Aave can only be sent to treasury via `_collectInterestToTreasury`, not to the user. [6](#0-5) [7](#0-6)

    function _withdrawFromAave(uint256 amount) internal returns (uint256 withdrawnAmount) {
        if (amount == 0) return 0;

        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        if (aaveBalance == 0) revert InsufficientAaveBalance();

        // Only withdraw up to the principal amount (don't use accrued interest for user withdrawals)
        uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave;

        withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
        if (withdrawnAmount == 0) return 0;

        aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
        totalETHDepositedToAave -= withdrawnAmount;

        emit ETHWithdrawnFromAave(withdrawnAmount, totalETHDepositedToAave);
    }
