### Title
Missing Chainlink Price Staleness Check in `ChainlinkPriceOracle::getAssetPrice` Allows Stale Prices to Drive rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `price`. It performs no staleness check (`updatedAt`), no round-completeness check (`answeredInRound < roundId`), and no non-positive price check. A stale or frozen Chainlink feed is silently accepted as fresh, causing incorrect rsETH minting amounts for every depositor.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the LST/ETH exchange rate used to compute how many rsETH tokens a depositor receives:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L49-55
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (, int256 price,,,) = priceFeed.latestRoundData();   // ← updatedAt, answeredInRound silently discarded

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`latestRoundData()` returns five values: `(roundId, answer, startedAt, updatedAt, answeredInRound)`. The contract ignores `updatedAt` (the timestamp of the last price update) and `answeredInRound` (the round in which the answer was computed). Neither a time-based staleness check nor a round-completeness check is performed.

The codebase itself demonstrates the correct pattern in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L26-36
function getRate() public view returns (uint256) {
    (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();

    if (answeredInRound < roundID) revert StalePrice();
    if (timestamp == 0) revert IncompleteRound();
    if (ethPrice <= 0) revert InvalidPrice();
    ...
}
```

`ChainlinkPriceOracle` applies none of these guards.

---

### Impact Explanation

`ChainlinkPriceOracle.getAssetPrice()` feeds into two critical paths:

**Path 1 – rsETH minting for depositors:**

`depositAsset()` / `depositETH()` → `_beforeDeposit()` → `getRsETHAmountToMint()` → `lrtOracle.getAssetPrice(asset)` → `ChainlinkPriceOracle.getAssetPrice(asset)`

```solidity
// contracts/LRTDepositPool.sol L519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

If the Chainlink feed for an LST (e.g., stETH, rETH) is stale at a price **below** the true market price, a depositor receives **more rsETH** than their deposit is worth. This dilutes all existing rsETH holders, constituting theft of their accrued yield.

**Path 2 – rsETH price update (TVL computation):**

`updateRSETHPrice()` → `_updateRsETHPrice()` → `_getTotalEthInProtocol()` → `getAssetPrice(asset)` for every supported asset. A stale price here causes the computed TVL to be wrong, leading to incorrect protocol fee minting and a mispriced rsETH exchange rate that persists until the next update.

**Impact classification: High — Theft of unclaimed yield** (dilution of existing rsETH holders via over-minting when a stale low price is consumed).

---

### Likelihood Explanation

Chainlink feeds can go stale during:
- Network congestion preventing oracle keeper transactions
- Sequencer downtime on L2 (though this is L1)
- Chainlink circuit-breaker triggering (price deviation threshold not met, so no update)
- Feed deprecation or migration

The `answeredInRound < roundId` condition can also be true during an incomplete aggregation round. These are known, documented failure modes for Chainlink. The protocol already handles this correctly in `ChainlinkOracleForRSETHPoolCollateral`, confirming awareness of the risk. The omission in `ChainlinkPriceOracle` is therefore an oversight, not a design choice.

---

### Recommendation

Apply the same guards already present in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    // Optionally: if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

A configurable per-feed `MAX_STALENESS` threshold is recommended, as different LST feeds have different heartbeat intervals.

---

### Proof of Concept

1. Chainlink's stETH/ETH feed goes stale (e.g., last updated 4 hours ago at 0.995 ETH/stETH; true price is now 1.002 ETH/stETH).
2. Attacker calls `depositAsset(stETH, 1000e18, 0, "")`.
3. `getRsETHAmountToMint(stETH, 1000e18)` calls `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns stale 0.995e18.
4. `rsethAmountToMint = (1000e18 * 0.995e18) / rsETHPrice` — computed using the stale low price.
5. Because the stale price undervalues stETH, the attacker receives **more rsETH** than 1000 stETH is actually worth at the true rate of 1.002.
6. The excess rsETH represents value extracted from existing rsETH holders, whose share of the underlying TVL is diluted.

No special privileges are required. Any user can call `depositAsset()`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
