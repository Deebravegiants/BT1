### Title
Spam `initiateWithdrawal` with dust amounts fills the global FIFO withdrawal queue, delaying legitimate users' withdrawals - (File: contracts/LRTWithdrawalManager.sol)

---

### Summary
`LRTWithdrawalManager.initiateWithdrawal` has no per-user request cap and defaults `minRsEthAmountToWithdraw[asset]` to zero, allowing any rsETH holder to create an unbounded number of 1-wei withdrawal requests. Because the unlock queue is a strict global FIFO (`nextLockedNonce`), the operator must process every attacker request before any later legitimate request can be unlocked, temporarily freezing legitimate users' withdrawals.

---

### Finding Description

`initiateWithdrawal` enforces only a single size guard:

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```

`minRsEthAmountToWithdraw[asset]` is a storage mapping that is **never initialised** in `initialize`; it starts at `0` for every asset. When it is `0`, the condition collapses to `rsETHUnstaked == 0`, so any non-zero amount — including 1 wei — passes the check. [1](#0-0) [2](#0-1) 

Each accepted request is appended to the **global** FIFO queue via `_addUserWithdrawalRequest`, which increments `nextUnusedNonce[asset]`:

```solidity
userAssociatedNonces[asset][msg.sender].pushBack(nextUnusedNonce_);
nextUnusedNonce[asset] = nextUnusedNonce_ + 1;
``` [3](#0-2) 

The operator's `unlockQueue` → `_unlockWithdrawalRequests` loop advances `nextLockedNonce[asset]` strictly in insertion order:

```solidity
while (nextLockedNonce_ < firstExcludedIndex) {
    ...
    unchecked { nextLockedNonce_++; }
}
nextLockedNonce[asset] = nextLockedNonce_;
``` [4](#0-3) 

A user's request can only be unlocked once `nextLockedNonce[asset]` has advanced past it. There is **no per-user cap** on open requests in `LRTWithdrawalManager` (contrast `KernelDepositPool.maxNumberOfWithdrawalsPerUser`). [5](#0-4) 

An attacker who front-runs a legitimate user's `initiateWithdrawal` transaction with N dust requests inserts N entries ahead of the victim in the global queue. The operator must call `unlockQueue` enough times to advance `nextLockedNonce` past all N attacker entries before the victim's request becomes eligible for unlocking.

Additionally, each dust request increments `assetsCommitted[asset]`:

```solidity
assetsCommitted[asset] += expectedAssetAmount;
``` [6](#0-5) 

If the attacker holds enough rsETH to commit all available assets, `getAvailableAssetAmount` returns 0 and every subsequent `initiateWithdrawal` by any user reverts with `ExceedAmountToWithdraw`, blocking new withdrawal requests entirely until the attacker's requests are processed. [7](#0-6) 

---

### Impact Explanation

**Medium. Temporary freezing of funds.**

Legitimate users who have already transferred rsETH into the contract (or who attempt to do so) cannot have their withdrawal requests unlocked until the operator processes all preceding attacker entries. With a large enough spam batch, the delay can span many operator `unlockQueue` batches across multiple blocks, effectively freezing the victim's rsETH inside the contract for an extended period. In the extreme case where `assetsCommitted` is saturated, no new withdrawal requests can be initiated at all.

---

### Likelihood Explanation

**Low-Medium.**

- `minRsEthAmountToWithdraw[asset]` is `0` by default for every asset; the admin must explicitly call `setMinRsEthAmountToWithdraw` to raise it.
- The attacker needs only 1 wei of rsETH per request; the dominant cost is L1 gas (~100 k–200 k gas per call). A motivated attacker can create hundreds of requests for a few thousand dollars.
- The attack is most effective as a targeted front-run against a specific large withdrawal, analogous to the proposal-spam front-run described in the reference report.

---

### Recommendation

1. **Set a non-zero `minRsEthAmountToWithdraw` for every supported asset** during initialisation (or enforce a protocol-wide floor in `initiateWithdrawal`).
2. **Add a per-user cap** on open (unlocked) withdrawal requests in `LRTWithdrawalManager`, mirroring `KernelDepositPool.maxNumberOfWithdrawalsPerUser`.
3. Consider a **per-block or per-epoch rate limit** on the number of new withdrawal requests to bound queue growth.

---

### Proof of Concept

```
// Precondition: minRsEthAmountToWithdraw[ETH] == 0 (default)
// Victim is about to call initiateWithdrawal(ETH, 10 ether, "")

// Attacker observes victim's pending tx in the mempool and front-runs:
for (uint i = 0; i < 500; i++) {
    // 1 wei rsETH per call; attacker needs 500 wei total
    LRTWithdrawalManager.initiateWithdrawal(ETH, 1, "spam");
}
// Victim's tx lands; victim's nonce = 500

// Operator must now call unlockQueue 5+ times (100 entries/batch)
// to advance nextLockedNonce[ETH] past 500 before victim's
// request at nonce 500 becomes eligible for unlocking.
// Victim's rsETH is locked in the contract throughout.
```
