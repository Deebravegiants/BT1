### Title
Missing Upper-Bound Validation for `feeBps` in `initialize()` Bypasses Fee Cap, Enabling Deposit Freeze or Zero-Yield Minting - (File: contracts/pools/RSETHPoolV2NBA.sol, contracts/pools/RSETHPoolV3WithNativeChainBridge.sol)

---

### Summary

Multiple L2 pool contracts accept an unchecked `_feeBps` parameter during `initialize()`. The post-deployment setter `setFeeBps()` enforces a strict upper bound, but the initializer does not, creating an inconsistency that allows the deployer to set a fee that either silently mints 0 rsETH to every depositor or causes all deposits to revert.

---

### Finding Description

In `RSETHPoolV2NBA.sol` and `RSETHPoolV3WithNativeChainBridge.sol`, the `initialize()` function assigns `feeBps = _feeBps` with no upper-bound check:

```solidity
// RSETHPoolV2NBA.sol – initialize()
function initialize(
    address admin,
    address bridger,
    address _wrsETH,
    uint256 _feeBps,       // ← no upper-bound check
    address _rsETHOracle
) external initializer {
    UtilLib.checkNonZeroAddress(_wrsETH);
    UtilLib.checkNonZeroAddress(_rsETHOracle);
    ...
    feeBps = _feeBps;      // ← stored unchecked
    rsETHOracle = _rsETHOracle;
}
``` [1](#0-0) 

The same pattern exists in `RSETHPoolV3WithNativeChainBridge.sol`: [2](#0-1) 

By contrast, the post-deployment setter enforces a hard cap:

```solidity
function setFeeBps(uint256 _feeBps) external onlyRole(DEFAULT_ADMIN_ROLE) {
    if (_feeBps > 10_000) revert InvalidFeeAmount();
    feeBps = _feeBps;
    emit FeeBpsSet(_feeBps);
}
``` [3](#0-2) 

The fee is applied in `viewSwapRsETHAmountAndFee()`:

```solidity
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;   // underflows if feeBps > 10_000
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [4](#0-3) 

---

### Impact Explanation

Two concrete failure modes arise from an unchecked `_feeBps` at initialization:

1. **`feeBps = 10_000` (100%)**: `fee = amount`, `amountAfterFee = 0`, `rsETHAmount = 0`. Every depositor sends ETH and receives **zero rsETH**. All ETH accumulates as protocol fees and is withdrawable by the `BRIDGER_ROLE`. This constitutes the contract failing to deliver its promised return to users.

2. **`feeBps > 10_000`**: `fee > amount`, causing `amount - fee` to underflow and revert under Solidity 0.8 checked arithmetic. Every call to `deposit()` reverts, **temporarily freezing all user deposits** into the pool until the admin corrects `feeBps` via `setFeeBps()`.

Impact classification: **Medium — Temporary freezing of funds** (scenario 2) / **Low — Contract fails to deliver promised returns** (scenario 1).

---

### Likelihood Explanation

The `initialize()` function is called exactly once, at deployment. The deployer supplies `_feeBps` as a raw constructor argument with no on-chain guard. An accidental off-by-one (e.g., passing `10000` intending basis-point notation but meaning 100%) or a unit confusion (passing `100` for 100% instead of 10%) is realistic. The inconsistency between `initialize()` and `setFeeBps()` makes the omission non-obvious. Likelihood is **Low** (requires deployer error), but the consequence is immediate and affects all subsequent depositors until corrected.

---

### Recommendation

Add the same upper-bound guard used in `setFeeBps()` to both `initialize()` functions:

```diff
function initialize(
    address admin,
    address bridger,
    address _wrsETH,
    uint256 _feeBps,
    address _rsETHOracle
) external initializer {
    UtilLib.checkNonZeroAddress(_wrsETH);
    UtilLib.checkNonZeroAddress(_rsETHOracle);
+   if (_feeBps > 10_000) revert InvalidFeeAmount();
    ...
    feeBps = _feeBps;
```

Apply the same fix to `RSETHPoolV3WithNativeChainBridge.sol`. Consider also adding a zero-address check for `admin` and `bridger` in both initializers, since neither is currently validated. [2](#0-1) 

---

### Proof of Concept

1. Deploy `RSETHPoolV2NBA` proxy and call `initialize(admin, bridger, wrsETH, 10_001, oracle)`.
2. `feeBps` is now `10_001`, which exceeds `10_000`.
3. Any user calls `deposit{value: 1 ether}("ref")`.
4. Inside `viewSwapRsETHAmountAndFee(1 ether)`:
   - `fee = 1 ether * 10_001 / 10_000 = 1.0001 ether`
   - `amountAfterFee = 1 ether - 1.0001 ether` → **underflow → revert**
5. All deposits revert. The pool is non-functional until the admin calls `setFeeBps()` with a valid value.

Alternatively, with `_feeBps = 10_000`:
- `fee = 1 ether`, `amountAfterFee = 0`, `rsETHAmount = 0`
- `wrsETH.mint(msg.sender, 0)` — user receives nothing; ETH is silently captured as fees. [5](#0-4)
