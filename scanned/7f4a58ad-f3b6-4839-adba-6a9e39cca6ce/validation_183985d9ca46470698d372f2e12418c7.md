Looking at the two contracts in detail:

### Title
Unguarded Zero-Value `sendETHToL1ViaBridge` Enables Block Stuffing on Unichain — (`contracts/bridges/UnichainMessenger.sol`)

---

### Summary

`UnichainMessenger.sendETHToL1ViaBridge` has no access control and accepts zero-value calls. An attacker can spam it with a self-controlled `l2bridge` address and `value=0`/`msg.value=0`, filling Unichain blocks at negligible cost and temporarily preventing the `BRIDGER_ROLE` from executing `bridgeAssetsViaNativeBridge` on `RSETHPoolNoWrapper`.

---

### Finding Description

`sendETHToL1ViaBridge` is declared `external payable nonReentrant` with no role gate: [1](#0-0) 

The only guard is the `msg.value == value` parity check. When both are zero, `0 != 0` evaluates to `false` and execution continues: [2](#0-1) 

The `l2bridge` parameter is fully caller-controlled. The attacker deploys a trivial contract implementing `IUnichainMessenger.bridgeETHTo` that accepts zero-value calls and returns immediately: [3](#0-2) 

With a no-op mock bridge, each spam call costs only ~24,000–26,000 gas (21,000 base + execution overhead). On Unichain (OP Stack), gas prices are sub-0.001 gwei, making it trivially cheap to fill a ~30 M gas block (~1,200 calls per block). The `nonReentrant` guard prevents reentrancy within a single call but does not limit separate transactions.

The legitimate `BRIDGER_ROLE` path calls `sendETHToL1ViaBridge` indirectly through `RSETHPoolNoWrapper.bridgeAssetsViaNativeBridge`: [4](#0-3) 

While blocks are stuffed, that transaction cannot be included, temporarily freezing L2 ETH bridging.

---

### Impact Explanation

**Low — Block stuffing.** The attacker can sustain block stuffing for as long as they are willing to pay negligible OP Stack gas fees. During this window, the `BRIDGER_ROLE`'s `bridgeAssetsViaNativeBridge` transaction is excluded from blocks, temporarily freezing the movement of accumulated L2 ETH to L1. No funds are permanently lost; the freeze ends when the attacker stops.

---

### Likelihood Explanation

**Low-to-Medium.** The attack requires no special role, no ETH value (only gas), and only a one-time mock bridge deployment. On Unichain, the per-block cost is negligible. The only practical deterrent is the sequencer's ability to reorder or censor transactions, which is an operational (not contract-level) mitigation.

---

### Recommendation

Add an access control modifier (e.g., `onlyRole(BRIDGER_ROLE)` or a dedicated `MESSENGER_ROLE`) to `sendETHToL1ViaBridge` in `UnichainMessenger`, consistent with how all other bridge-related functions in `RSETHPoolNoWrapper` are protected: [5](#0-4) 

Alternatively, restrict the `l2bridge` parameter to a protocol-configured allowlist so arbitrary mock bridges cannot be injected.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// 1. Deploy a no-op mock bridge
contract MockBridge {
    function bridgeETHTo(address, uint32, bytes memory) external payable {}
}

// 2. Spam loop (run in a local fork test)
contract BlockStuffer {
    address immutable messenger;
    address immutable mockBridge;

    constructor(address _messenger, address _mockBridge) {
        messenger = _messenger;
        mockBridge = _mockBridge;
    }

    // Call this repeatedly (or in a loop) to fill the block
    function spam(uint256 iterations) external {
        for (uint256 i = 0; i < iterations; i++) {
            // msg.value == value == 0 → passes MismatchedMsgValue check
            IUnichainMessenger(messenger).sendETHToL1ViaBridge(
                mockBridge, address(0xdead), 0
            );
        }
    }
}

interface IUnichainMessenger {
    function sendETHToL1ViaBridge(address, address, uint256) external payable;
}
```

**Assert:** After `spam()` exhausts the block gas limit, submit a `bridgeAssetsViaNativeBridge` call from the `BRIDGER_ROLE` account and observe it cannot be mined in the same block due to gas exhaustion.
