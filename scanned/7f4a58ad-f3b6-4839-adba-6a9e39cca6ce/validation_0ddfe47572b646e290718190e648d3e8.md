### Title
Missing Output Amount Validation in RSETHPool Deposit Functions Allows Permanent Loss of Deposited Assets - (File: contracts/pools/RSETHPoolV3.sol)

### Summary
The `deposit` functions in RSETHPool contracts validate that the input `amount != 0` but never validate that the computed output `rsETHAmount > 0`. When a user deposits a small amount of tokens or ETH, integer division in `viewSwapRsETHAmountAndFee` can round `rsETHAmount` down to zero. The transfer of the user's assets succeeds, but 0 rsETH/wrsETH is minted, permanently losing the deposited assets with no reclaim path.

### Finding Description
In `RSETHPoolV3.deposit(address token, uint256 amount, string referralId)`, the only input guard is:

```solidity
if (amount == 0) revert InvalidAmount();
```

After the transfer, the output is computed as:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
```

Inside `viewSwapRsETHAmountAndFee`:

```solidity
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;
uint256 rsETHToETHrate = getRate();
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

When `amountAfterFee * tokenToETHRate < rsETHToETHrate`, Solidity integer division truncates `rsETHAmount` to `0`. The function then executes:

```solidity
wrsETH.mint(msg.sender, rsETHAmount); // mints 0 — no-op
```

The user's tokens are now held by the pool but are not counted in `feeEarnedInToken` (only the `fee` portion is), so the `amountAfterFee` portion is unrecoverable. There is no `minRsETHAmountExpected` slippage guard and no reclaim mechanism.

The same pattern exists in:
- `RSETHPoolV3.deposit(string referralId)` (ETH path) — ETH is sent to the contract, `feeEarnedInETH` is not updated for the `amountAfterFee` portion, and `wrsETH.mint(msg.sender, 0)` is called.
- `RSETHPoolNoWrapper.deposit(address token, uint256 amount, string referralId)` — calls `rsETH.safeTransfer(msg.sender, 0)`, a no-op.
- `RSETHPoolV3ExternalBridge` and `RSETHPoolV3WithNativeChainBridge` deposit functions follow the same pattern.

### Impact Explanation
A user depositing a small amount of tokens (e.g., 1 wei of a token when `rsETHToETHrate > tokenToETHRate`) will have their assets transferred to the pool and permanently lost. The `amountAfterFee` portion is not tracked as fees and cannot be recovered via `withdrawFees`. This constitutes a permanent freezing of user funds with no reclaim path.

### Likelihood Explanation
The condition `amountAfterFee * tokenToETHRate < rsETHToETHrate` is reachable by any unprivileged user depositing a sufficiently small amount. As rsETH accrues value over time (rate increases), the minimum deposit threshold that avoids zero-output grows, making this more likely to affect users who do not account for the current rate. No special permissions are required.

### Recommendation
Add an explicit output amount check after computing `rsETHAmount`:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
if (rsETHAmount == 0) revert InvalidAmount();
```

Alternatively, add a `minRsETHAmountExpected` parameter to all deposit functions so callers can enforce a slippage floor, analogous to the `minRSETHAmountExpected` guard already present in `LRTDepositPool.depositAsset`.

### Proof of Concept
1. Assume `feeBps = 0`, `rsETHToETHrate = 1.05e18` (rsETH worth 1.05 ETH), `tokenToETHRate = 1e18`.
2. User calls `RSETHPoolV3.deposit(token, 1, "")` depositing 1 wei of the token.
3. `viewSwapRsETHAmountAndFee(1, token)` computes: `fee = 0`, `amountAfterFee = 1`, `rsETHAmount = 1 * 1e18 / 1.05e18 = 0`.
4. `IERC20(token).safeTransferFrom(msg.sender, address(this), 1)` — 1 wei transferred to pool.
5. `feeEarnedInToken[token] += 0` — no fee recorded.
6. `wrsETH.mint(msg.sender, 0)` — no-op, user receives nothing.
7. The 1 wei is permanently stuck in the pool's token balance, not recoverable via `withdrawFees` or any other function. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
