### Title
Pre-loaded `wrsETH` inventory in `RSETHPool` has no recovery path, causing permanent fund freeze when pool is deprecated - (File: `contracts/pools/RSETHPool.sol`)

---

### Summary
The `RSETHPool` contract on Arbitrum is pre-loaded with `wrsETH` (i.e., `LZ_RSETH`) tokens that are distributed to depositors. The contract exposes functions to withdraw ETH fees, move ETH for bridging, and bridge supported tokens — but contains **no function to withdraw the `wrsETH` inventory itself**. When the pool is deprecated (a normal protocol lifecycle event, as evidenced by the existence of `RSETHPoolV3`), any remaining `wrsETH` balance is permanently frozen with no recovery path.

---

### Finding Description

`RSETHPool` is the original Arbitrum pool. It holds a pre-loaded balance of `wrsETH` tokens and distributes them to users on each ETH or token deposit:

```solidity
// contracts/pools/RSETHPool.sol
function deposit(string memory referralId) external payable nonReentrant whenNotPaused {
    ...
    IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount);
    ...
}
``` [1](#0-0) 

The contract provides the following asset-exit paths:

| Function | Asset moved |
|---|---|
| `withdrawFees(receiver)` | ETH fees |
| `withdrawFees(receiver, token)` | ERC-20 fees |
| `moveAssetsForBridging(amount)` | ETH (minus fees) |
| `moveAssetsForBridging(token, amount)` | ERC-20 (minus fees) |
| `bridgeAssets(...)` | ETH via Stargate/LZ |
| `bridgeTokens(token)` | ERC-20 via token bridge | [2](#0-1) 

**None of these functions touch the `wrsETH` balance.** The only way `wrsETH` leaves the contract is through `deposit()` calls. There is no `withdrawWrsETH`, no sweep function, and no admin rescue path for the `wrsETH` inventory.

The pool can be permanently disabled via:
- `pause()` (callable by `PAUSER_ROLE`) [3](#0-2) 
- `setIsEthDepositEnabled(false)` (callable by `TIMELOCK_ROLE`) [4](#0-3) 

Once either of these is set, no further `deposit()` calls can drain the `wrsETH` balance, and no other function can recover it.

This is structurally identical to M-3: a contract is pre-funded with tokens, a lifecycle state transition occurs (auction concludes / pool is deprecated), and the remaining pre-funded tokens have no withdrawal path.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

`wrsETH` (`LZ_RSETH`) on Arbitrum is a canonical cross-chain rsETH token backed 1:1 by rsETH on L1. Any `wrsETH` balance remaining in `RSETHPool` at the time of deprecation is permanently inaccessible. Given that `RSETHPoolV3` already exists as the successor contract, the deprecation of `RSETHPool` is not hypothetical — it is an expected protocol lifecycle event.

---

### Likelihood Explanation

**Medium.** The protocol has already deployed `RSETHPoolV3` as the successor to `RSETHPool`. Deprecating `RSETHPool` (by pausing it or disabling ETH deposits) is a routine upgrade step. Any `wrsETH` inventory remaining at that point is permanently frozen. No attacker action is required; the loss is triggered by normal protocol maintenance.

---

### Recommendation

Add a privileged function (e.g., restricted to `TIMELOCK_ROLE` or `DEFAULT_ADMIN_ROLE`) that allows recovery of the `wrsETH` balance:

```solidity
function withdrawWrsETH(address receiver, uint256 amount) external onlyRole(TIMELOCK_ROLE) {
    IERC20(address(wrsETH)).safeTransfer(receiver, amount);
}
```

This mirrors the fix applied in M-3, where the protocol added a post-conclusion withdrawal path for the seller.

---

### Proof of Concept

1. Admin pre-loads `RSETHPool` with 1 000 `wrsETH` tokens (direct transfer).
2. Users deposit ETH over time; 900 `wrsETH` is distributed.
3. Protocol upgrades to `RSETHPoolV3`; admin calls `pause()` on `RSETHPool` to deprecate it.
4. 100 `wrsETH` remains in `RSETHPool`.
5. Admin attempts to recover the 100 `wrsETH` — no function exists to do so.
6. The 100 `wrsETH` (representing real ETH value on L1) is permanently frozen in the contract. [1](#0-0) [5](#0-4)

    /// @param amount The amount of ETH to bridge
    /// @param minAmount The minimum amount of ETH to receive on L1
    /// @param nativeFee The native fee to pay for the bridge
    function bridgeAssets(
        uint256 amount,
        uint256 minAmount,

        external
        payable
        nonReentrant

        // Exclude msg.value so reserved fees can’t be accidentally consumed

            revert InvalidMinAmount();

    /// @dev Bridges tokens to L1 using their corresponding token bridge

    function bridgeTokens(address token)
        external
        payable
        nonReentrant
        onlySupportedToken(token)
        onlyRole(BRIDGER_ROLE)
    {
        if (tokenBridge[token] == address(0)) {
            revert MissingBridgeForToken();

        uint256 balance = getTokenBalanceMinusFees(token);

        if (balance == 0) {
            revert ZeroBridgeAmount();

        // Approve the required amount to the bridge

        IL2TokenBridge(tokenBridge[token]).bridgeTokenToL1{ value: msg.value }(l1VaultETHForL2Chain, balance);

        emit BridgedTokenToL1(token, l1VaultETHForL2Chain, balance);
