### Title
Unbounded ETH Push-Transfer to User-Controlled Contract in `_transferAsset` Can Permanently Freeze ETH Withdrawals - (File: contracts/LRTWithdrawalManager.sol)

### Summary
`LRTWithdrawalManager._transferAsset` makes an unbounded low-level ETH call to a user-controlled address. Because the user's rsETH is burned in a prior `unlockQueue` transaction before `completeWithdrawal` is called, a recipient contract whose `receive()` reverts or exhausts gas causes the ETH to be permanently frozen inside `LRTWithdrawalManager` with no recovery path in the current implementation.

### Finding Description
In `_transferAsset` (line 878), when `asset == LRTConstants.ETH_TOKEN`, the function issues an unbounded low-level call:

```solidity
// contracts/LRTWithdrawalManager.sol:876-883
function _transferAsset(address asset, address to, uint256 amount) internal {
    if (asset == LRTConstants.ETH_TOKEN) {
        (bool sent,) = payable(to).call{ value: amount }("");
        if (!sent) revert EthTransferFailed();
    } else {
        IERC20(asset).safeTransfer(to, amount);
    }
}
```

All remaining gas is forwarded to the recipient's `receive()` / `fallback()`. There is no gas cap and no try/catch — a revert propagates upward and rolls back the entire `_processWithdrawalCompletion` call.

The critical ordering of the withdrawal lifecycle creates the freeze:

| Step | Transaction | Effect |
|---|---|---|
| 1 | `initiateWithdrawal` | rsETH transferred from user → `LRTWithdrawalManager` |
| 2 | `unlockQueue` (operator) | rsETH **burned**; ETH moved from `LRTUnstakingVault` → `LRTWithdrawalManager` |
| 3 | `completeWithdrawal` | ETH pushed to `user` via `_transferAsset` |

After step 2, the rsETH no longer exists. If step 3 always reverts (because the user's contract `receive()` reverts or consumes all gas), the state inside `_processWithdrawalCompletion` is rolled back on every attempt, but the rsETH burn from step 2 is **not** rolled back — it occurred in a separate, already-finalized transaction.
