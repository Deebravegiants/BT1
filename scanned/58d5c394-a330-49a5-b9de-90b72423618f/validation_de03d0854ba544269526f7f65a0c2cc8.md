### Title
Missing Minimum Output Slippage Guard on L2 Pool Deposits - (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPool.sol)

---

### Summary

The L2 pool deposit functions (`RSETHPoolV3.deposit` and `RSETHPool.deposit`) accept no `minRsETH` parameter and perform no minimum-output check before minting rsETH to the caller. The oracle rate used to compute the minted amount can change between transaction submission and execution, leaving depositors with no protection against receiving fewer rsETH tokens than they intended.

---

### Finding Description

`LRTDepositPool` on L1 explicitly accepts a `minRSETHAmountExpected` argument and enforces it inside `_beforeDeposit`:

```solidity
// LRTDepositPool.sol L76-93
function depositETH(uint256 minRSETHAmountExpected, string calldata referralId)
    external payable ...
{
    uint256 rsethAmountToMint = _beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, minRSETHAmountExpected);
    _mintRsETH(rsethAmountToMint);
    ...
}
```

```solidity
// LRTDepositPool.sol L648-670
function _beforeDeposit(..., uint256 minRSETHAmountExpected) private view returns (uint256 rsethAmountToMint) {
    rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
    if (rsethAmountToMint < minRSETHAmountExpected) {
        revert MinimumAmountToReceiveNotMet();
    }
}
```

The L2 pool equivalents carry no such guard. Both ETH and token deposit entry points in `RSETHPoolV3` are:

```solidity
// RSETHPoolV3.sol L246-265
function deposit(string memory referralId)
    external payable nonReentrant whenNotPaused
    limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    ...
    wrsETH.mint(msg.sender, rsETHAmount);   // no minimum check
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}

// RSETHPoolV3.sol L271-293
function deposit(address token, uint256 amount, string memory referralId)
    external nonReentrant whenNotPaused onlySupportedToken(token)
    limitDailyMint(amount, token)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
    ...
    wrsETH.mint(msg.sender, rsETHAmount);   // no minimum check
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId, token);
}
```

The same pattern is present in `RSETHPool.deposit` (lines 265–305).

The minted amount is derived from `viewSwapRsETHAmountAndFee`, which reads the live oracle rate:

```solidity
// RSETHPoolV3.sol L299-308
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();          // live oracle read
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
```

`getRate()` delegates to `rsETHOracle`, a cross-chain rate provider whose value is updated periodically and can increase between the moment a user signs a transaction and the moment it is included in a block. Because no floor is enforced, the user silently receives fewer rsETH tokens than they observed when constructing the transaction.

---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A depositor who previewed a rate of, say, 1.05 ETH/rsETH and submitted a transaction expecting ~952 rsETH for 1 ETH may receive materially fewer tokens if the oracle rate is updated upward before the transaction lands. The deposited ETH is not lost (it is held in the pool and eventually bridged to L1), but the user receives fewer rsETH than they intended with no on-chain recourse.

---

### Likelihood Explanation

The rsETH/ETH exchange rate increases continuously as staking rewards accrue and is pushed on-chain by the cross-chain rate provider on a regular schedule. Any deposit transaction that sits in the mempool across a rate update will be affected. On congested L2s, this window can span several minutes. The entry path is fully permissionless — any depositor calling `deposit()` is exposed.

---

### Recommendation

Add a `minRsETHAmountExpected` parameter to both `deposit` overloads in `RSETHPoolV3`, `RSETHPool`, `RSETHPoolNoWrapper`, and `RSETHPoolV2ExternalBridge`, mirroring the pattern already used in `LRTDepositPool`:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountExpected)
    external payable nonReentrant whenNotPaused
    limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    wrsETH.mint(msg.sender, rsETHAmount);
    ...
}
```

---

### Proof of Concept

1. Oracle rate is 1.05 ETH/rsETH. User calls `RSETHPoolV3.deposit{value: 1 ether}("ref")` expecting ≈ 952 rsETH (after fee).
2. Transaction sits in the mempool. The cross-chain rate provider pushes a new rate of 1.10 ETH/rsETH.
3. Transaction executes. `viewSwapRsETHAmountAndFee(1 ether)` now returns ≈ 909 rsETH.
4. `wrsETH.mint(msg.sender, 909e18)` is called with no revert — the user receives ~43 fewer rsETH than expected (~4.5% shortfall) with no on-chain protection.

**Contrast:** The same user depositing via `LRTDepositPool.depositETH(952e18, "ref")` would have the transaction revert at line 667 (`rsethAmountToMint < minRSETHAmountExpected`), protecting them from the adverse rate.
