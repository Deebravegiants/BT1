### Title
Zero rsETH/wrsETH Minted on Dust Deposits While User's ETH/Tokens Are Consumed - (File: contracts/pools/RSETHPoolV3.sol)

### Summary
The L2 pool `deposit` functions (and `LRTDepositPool.depositAsset`/`depositETH`) perform no check that the computed output amount (`rsETHAmount` / `rsethAmountToMint`) is greater than zero before consuming the caller's ETH or ERC-20 tokens. When a deposit is small enough that integer division truncates the result to zero, the user's funds are silently absorbed by the pool while they receive nothing in return.

### Finding Description

**Root cause — `viewSwapRsETHAmountAndFee` (ETH path):**

```solidity
// RSETHPoolV3.sol lines 299-308
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;   // ← truncates to 0
}
``` [1](#0-0) 

When `amountAfterFee * 1e18 < rsETHToETHrate` (e.g., a 1-wei deposit when the rsETH/ETH rate is ~1.05e18), `rsETHAmount` is 0.

**Root cause — `viewSwapRsETHAmountAndFee` (token path):**

```solidity
// RSETHPoolV3.sol lines 315-335
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;  // ← truncates to 0
``` [2](#0-1) 

**Vulnerable call site — ETH deposit:**

```solidity
// RSETHPoolV3.sol lines 246-265
function deposit(string memory referralId) external payable ... {
    ...
    if (amount == 0) revert InvalidAmount();          // only guards against zero
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);             // mints 0 — no guard
    ...
}
``` [3](#0-2) 

The `limitDailyMint` modifier also silently accepts `rsETHAmount == 0` (it adds 0 to `dailyMintAmount` and the limit check passes), so no revert occurs anywhere in the call stack. [4](#0-3) 

The same pattern is present in every pool variant:

- `RSETHPoolV3ExternalBridge.deposit` — mints 0 wrsETH [5](#0-4) 

- `RSETHPool.deposit` — transfers 0 wrsETH from pool balance [6](#0-5) 

- `RSETHPoolNoWrapper.deposit` — transfers 0 rsETH from pool balance [7](#0-6) 

**L1 analog — `LRTDepositPool.depositAsset`:**

```solidity
// LRTDepositPool.sol lines 506-521
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [8](#0-7) 

If `amount * assetPrice < rsETHPrice`, `rsethAmountToMint = 0`. The `_beforeDeposit` check only reverts if `rsethAmountToMint < minRSETHAmountExpected`; when the caller passes `minRSETHAmountExpected = 0` (the default for many integrations), the deposit proceeds, the asset is pulled from the user, and 0 rsETH is minted. [9](#0-8) 

### Impact Explanation
A depositor who sends a dust amount (e.g., 1 wei of ETH when the rsETH/ETH rate is > 1e18) has their ETH/tokens permanently absorbed by the pool while receiving zero rsETH/wrsETH. The absorbed ETH is later bridged to L1 and credited to the protocol's TVL, diluting the loss across all rsETH holders. The depositor's funds are not recoverable. This matches the **Low** impact tier: *"Contract fails to deliver promised returns, but doesn't lose value"* (the protocol retains the value; the individual user loses it).

### Likelihood Explanation
Any unprivileged caller of `deposit` can trigger this with a carefully chosen dust amount. No special role or precondition is required beyond the pool being unpaused and ETH/token deposits being enabled. The threshold below which `rsETHAmount = 0` grows as the rsETH/ETH rate appreciates over time, making the affected range larger as the protocol matures.

### Recommendation
Add an explicit zero-output guard in each `deposit` function (and in `_beforeDeposit` on L1):

```solidity
// RSETHPoolV3.sol — ETH deposit
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
if (rsETHAmount == 0) revert InvalidAmount();

// LRTDepositPool._beforeDeposit
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
if (rsethAmountToMint == 0) revert InvalidAmountToDeposit();
```

This mirrors the recommended fix from the reference report (`require(tokenAmount > 0)`) and ensures that no deposit can succeed unless the caller receives a non-zero output.

### Proof of Concept

Assume `RSETHPoolV3` is deployed with `feeBps = 0` and the oracle returns `rsETHToETHrate = 1.05e18` (rsETH has appreciated 5%).

1. Attacker calls `deposit{value: 1}("")` (1 wei of ETH).
2. `viewSwapRsETHAmountAndFee(1)`:
   - `fee = 1 * 0 / 10_000 = 0`
   - `amountAfterFee = 1`
   - `rsETHAmount = 1 * 1e18 / 1.05e18 = 0` (integer truncation)
3. `feeEarnedInETH += 0` — no fee recorded.
4. `wrsETH.mint(msg.sender, 0)` — 0 wrsETH minted.
5. The 1 wei of ETH sits in the pool's balance and will be bridged to L1 as protocol TVL.
6. Attacker holds 0 wrsETH and has lost 1 wei with no recourse.

The same arithmetic applies to token deposits and to `LRTDepositPool.depositAsset` when `minRSETHAmountExpected = 0`.
