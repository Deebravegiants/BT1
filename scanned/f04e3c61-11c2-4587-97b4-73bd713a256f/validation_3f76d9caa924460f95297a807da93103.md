### Title
Stale Cross-Chain Rate in L2 Deposit Pools Allows Over-Minting of wrsETH, Causing Protocol Insolvency - (`contracts/cross-chain/CrossChainRateReceiver.sol`)

---

### Summary

The L2 deposit pools (`RSETHPoolV2`, `RSETHPoolV3`, `RSETHPool`, etc.) compute the amount of `wrsETH` to mint using a locally cached `rate` stored in `CrossChainRateReceiver`. This rate is a push-based copy of `LRTOracle.rsETHPrice` on L1, propagated via LayerZero. There is no freshness check on the cached rate. When the L1 rsETH price increases but the L2 rate has not yet been updated, any depositor can mint more `wrsETH` than the deposited ETH can back in rsETH terms, creating a structural shortfall and diluting all existing `wrsETH` holders.

---

### Finding Description

The cross-chain rate synchronization system uses a push-based architecture. On L1, `RSETHMultiChainRateProvider.updateRate()` reads `LRTOracle.rsETHPrice()` and sends it via LayerZero to `CrossChainRateReceiver` on each L2. [1](#0-0) 

The L2 receiver stores the rate and a `lastUpdated` timestamp: [2](#0-1) 

However, `getRate()` returns the stored value with **no staleness check**: [3](#0-2) 

Every L2 pool contract calls this function to price deposits: [4](#0-3) 

The mint amount is computed as:

```
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate
``` [5](#0-4) 

If `rsETHToETHrate` is stale (lower than the true L1 rate), the division yields a **larger** `rsETHAmount`, minting more `wrsETH` than the deposited ETH can back.

The same pattern is present in `RSETHPoolV2`, `RSETHPool`, `RSETHPoolV3ExternalBridge`, and `RSETHPoolV3WithNativeChainBridge`. [6](#0-5) 

The L1 authoritative price lives in `LRTOracle.rsETHPrice`, updated by `updateRSETHPrice()`: [7](#0-6) 

There is no on-chain mechanism that atomically updates both the L1 oracle and the L2 receiver. The two values are always transiently desynced: the L1 price can increase (from staking rewards accrual) while the L2 cached rate lags behind until the off-chain keeper calls `updateRate()` and the LayerZero message is delivered.

---

### Impact Explanation

**Critical — Protocol insolvency.**

`wrsETH` is a 1:1 wrapper for rsETH. When a user deposits ETH on L2, the pool mints `wrsETH` immediately. The ETH is later bridged to L1 and deposited into `LRTDepositPool`, which mints rsETH at the **current** L1 rate: [8](#0-7) 

If the L2 rate was stale (lower than the L1 rate at bridge time), the ETH produces **fewer rsETH** on L1 than the `wrsETH` already minted on L2. The `RsETHTokenWrapper` backing is insufficient: [9](#0-8) 

Every depositor who exploits the stale window extracts value from existing `wrsETH` holders, whose redemption claims are diluted. Repeated exploitation accumulates a permanent insolvency gap.

---

### Likelihood Explanation

**Medium.** The rsETH/ETH rate increases continuously as EigenLayer staking rewards accrue. The rate update is entirely off-chain and push-based — no on-chain mechanism forces the L2 rate to stay current. Any gap between `updateRSETHPrice()` on L1 and the LayerZero message delivery on L2 is a live exploitation window. This window is routine (it exists after every L1 price update) and requires no special permissions to exploit — any address can call `deposit()`.

---

### Recommendation

1. **Short term:** Add a staleness threshold check in `CrossChainRateReceiver.getRate()`. Revert if `block.timestamp - lastUpdated` exceeds a configured maximum (e.g., 24 hours). This prevents deposits when the rate is known to be stale.

2. **Long term:** Consider a pull-based oracle model where the L2 pool fetches the rate on-demand from a trusted source, or implement a circuit-breaker that pauses deposits when the rate has not been updated within the expected window.

---

### Proof of Concept

1. L1 `LRTOracle.rsETHPrice` is updated from `1.05e18` to `1.10e18` via `updateRSETHPrice()`.
2. The off-chain keeper has not yet called `updateRate()`, so `CrossChainRateReceiver.rate` on L2 is still `1.05e18`.
3. Attacker calls `RSETHPoolV3.deposit{value: 1 ether}("ref")` on L2.
4. Pool computes: `rsETHAmount = 1e18 * 1e18 / 1.05e18 = 0.952 wrsETH` (should be `0.909 wrsETH` at the true rate).
5. Attacker receives `0.952 wrsETH` — `4.7%` more than the deposited ETH can back.
6. The 1 ETH is later bridged to L1 and mints only `~0.909 rsETH` at the current rate.
7. The `RsETHTokenWrapper` is short `0.043 rsETH` per 1 ETH deposited during the stale window.
8. Existing `wrsETH` holders bear the loss through dilution of their redemption claims. [10](#0-9) [11](#0-10)
