### Title
Permanent Exemption List Is Add-Only — Compromised Exempt Addresses Cannot Be Blocked or Have Funds Recovered - (File: contracts/RSETH.sol)

---

### Summary

`RSETH.addPermanentExemptions()` adds addresses to the `isPermanentlyExempt` mapping with no corresponding removal function. The NatSpec explicitly labels this "non-reversible." If a permanently exempt address is later compromised, the protocol's two emergency security mechanisms — `blockUserTransfers()` and `recoverFrozenFunds()` — are permanently disabled for that address, with no administrative escape hatch.

---

### Finding Description

`RSETH.sol` implements a transfer-blocking and fund-recovery security system. The Manager can call `blockUserTransfers()` to freeze any address for 24 hours, and the Admin can call `recoverFrozenFunds()` to seize the rsETH balance of a currently-blocked address. To protect trusted integrations (e.g., AMM pools, bridges, wrappers) from being accidentally frozen, the Manager can call `addPermanentExemptions()` to mark addresses as forever unblockable.

The problem is that `isPermanentlyExempt` is a one-way door:

```solidity
// contracts/RSETH.sol:130-154
/// @notice Permanently add accounts to the exempted list (non-reversible)
function addPermanentExemptions(address[] calldata accounts) external onlyLRTManager {
    ...
    if (!isPermanentlyExempt[account]) {
        isPermanentlyExempt[account] = true;
        emit PermanentExemptionAdded(account);
    }
}
```

There is no `removePermanentExemption()` function anywhere in the codebase. All three enforcement points hard-skip exempt addresses:

- `blockUserTransfers()` silently continues past exempt addresses:
  ```solidity
  if (isPermanentlyExempt[account] || account == address(0)) continue;
  ```
- `recoverFrozenFunds()` reverts immediately:
  ```solidity
  if (isPermanentlyExempt[from]) revert AddressPermanentlyExempt(from);
  ```
- `_enforceNotBlocked()` (called on every transfer/mint/burn) returns early:
  ```solidity
  if (isPermanentlyExempt[account]) return;
  ```

---

### Impact Explanation

**Low — Contract fails to deliver promised returns.**

The protocol's security model promises that any suspicious address can be blocked and its rsETH balance recovered. For permanently exempt addresses this promise is irrevocably broken. If a permanently exempt address (e.g., a trusted wrapper contract, an AMM pool, or a bridge) is later exploited or its private key is compromised:

1. `blockUserTransfers([compromisedAddress])` is a no-op — the address is silently skipped.
2. `recoverFrozenFunds(compromisedAddress)` reverts with `AddressPermanentlyExempt`.
3. The attacker controlling the exempt address can freely transfer rsETH without any protocol intervention.

The protocol cannot respond to the incident for that address without a full contract upgrade.

---

### Likelihood Explanation

**Low-to-Medium.** The Manager is expected to exempt multiple trusted integrations over the protocol's lifetime (AMM pools, bridges, wrappers). Any one of those integrations being exploited — a realistic scenario given the history of DeFi hacks — triggers this path. The protocol has no recourse short of a UUPS upgrade.

---

### Recommendation

Replace the boolean flag with a reversible setter, mirroring the fix suggested in M-02:

```solidity
// Replace addPermanentExemptions with a togglable setter:
function setExemptionStatus(address account, bool exempt) external onlyLRTManager {
    isPermanentlyExempt[account] = exempt;
    emit ExemptionStatusUpdated(account, exempt);
}
```

If true permanence is desired for specific addresses (e.g., the zero address), enforce it with an explicit allowlist rather than making the entire mapping irreversible.

---

### Proof of Concept

1. Manager calls `addPermanentExemptions([trustedPool])` — `isPermanentlyExempt[trustedPool] = true`. [1](#0-0) 

2. `trustedPool` is later exploited; attacker controls it.

3. Manager calls `blockUserTransfers([trustedPool])`: [2](#0-1) 
   → Line 168 evaluates `isPermanentlyExempt[trustedPool] == true` → `continue` → address is never blocked.

4. Admin calls `recoverFrozenFunds(trustedPool)`: [3](#0-2) 
   → Line 210 reverts with `AddressPermanentlyExempt`.

5. Attacker freely calls `rsETH.transfer(attacker, balance)` from `trustedPool`; `_enforceNotBlocked` returns early: [4](#0-3) 
   → Transfer succeeds. Protocol has no on-chain remedy.
