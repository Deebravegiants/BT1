### Title
Permissionless `FeeReceiver.sendFunds()` Enables Sandwich Attack to Steal Unclaimed Yield — (`contracts/FeeReceiver.sol`, `contracts/LRTDepositPool.sol`, `contracts/LRTOracle.sol`)

---

### Summary

`FeeReceiver.sendFunds()` has no access control. Any caller can flush accumulated MEV/execution-layer rewards into `LRTDepositPool` at will. Because `rsETHPrice` in `LRTOracle` is a lazily-updated storage variable, a deposit made immediately after `sendFunds()` uses a stale (too-low) price, minting more rsETH than the deposited ETH is worth. When the price is later updated, the attacker's excess rsETH captures value that belonged to existing holders.

---

### Finding Description

**Root cause 1 — `sendFunds()` is permissionless:** [1](#0-0) 

There is no `onlyRole` or any other guard. Any EOA or contract can call it at any time.

**Root cause 2 — TVL is read from live `address(this).balance`, but the price is a stale storage slot:**

`getETHDistributionData()` returns `address(this).balance` directly: [2](#0-1) 

`getRsETHAmountToMint()` divides by the *stored* `rsETHPrice`, which has not been updated since the rewards arrived: [3](#0-2) 

**Root cause 3 — `updateRSETHPrice()` is public:** [4](#0-3) 

Anyone can trigger the price update after depositing.

---

### Impact Explanation

Let the state before the attack be:

| Variable | Value |
|---|---|
| Protocol ETH TVL | X |
| rsETH total supply | S |
| Stored `rsETHPrice` | X/S |
| Pending rewards in FeeReceiver | R |

**Attack steps (single block or consecutive blocks):**

1. Call `FeeReceiver.sendFunds()` → deposit pool balance becomes X + R; stored price still X/S.
2. Call `depositETH(D)` → attacker receives `D * S/X` rsETH (priced at the stale X/S).
3. Call `LRTOracle.updateRSETHPrice()` → new price = `(X + R + D) / (S + D*S/X)`.

**Attacker's rsETH value after step 3:**

```
value = (D*S/X) * (X+R+D)/(S + D*S/X)
      = D * (X+R+D)/(X+D)
      = D + D*R/(X+D)
```

Profit = `D * R / (X + D)` — a positive amount extracted from existing holders' yield.

The `pricePercentageLimit` guard in `_updateRsETHPrice()` only blocks the call if the price jump exceeds the configured threshold: [5](#0-4) 

For typical MEV rewards (small relative to TVL), the jump stays within the limit and the call succeeds. Even if the threshold is breached, the attacker still holds rsETH minted at the stale price; when the manager eventually calls `updateRSETHPriceAsManager()`, the attacker's position is already inflated.

---

### Likelihood Explanation

- `sendFunds()` is callable by any address with zero preconditions.
- `depositETH()` is a standard user-facing function.
- `updateRSETHPrice()` is public.
- No flash-loan is needed; the attacker only needs to hold ETH.
- The attack is profitable whenever the FeeReceiver holds a non-trivial balance, which is a routine operational state (rewards accumulate between keeper calls).
- The only partial mitigation (`pricePercentageLimit`) does not prevent the minting at the stale price; it only delays price recognition, which does not help existing holders.

---

### Recommendation

1. **Add access control to `sendFunds()`** — restrict it to `MANAGER` or `LRT_OPERATOR` role so only trusted keepers can flush rewards.
2. **Alternatively, call `updateRSETHPrice()` atomically inside `receiveFromRewardReceiver()`** — update the stored price before any deposit can use the new balance.
3. **Deposit-time price freshness check** — reject deposits if `rsETHPrice` is older than N blocks/seconds relative to the current TVL.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

// Fork test (Foundry) — run against a mainnet/testnet fork
// Assumes: FeeReceiver holds 10 ETH in rewards, protocol has existing TVL

function testSandwichYield() public {
    address attacker = address(0xBEEF);
    vm.deal(attacker, 1 ether);
    vm.deal(address(feeReceiver), 10 ether); // simulate accumulated rewards

    // Record stale price
    uint256 stalePriceBefore = lrtOracle.rsETHPrice();

    // Step 1: flush rewards — permissionless
    feeReceiver.sendFunds();

    // Step 2: deposit at stale price
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 1 ether}(0, "");

    uint256 rsethMinted = rsETH.balanceOf(attacker);

    // Step 3: update price — permissionless
    lrtOracle.updateRSETHPrice();

    uint256 newPrice = lrtOracle.rsETHPrice();

    // Attacker's rsETH is now worth more than 1 ETH
    uint256 attackerEthValue = rsethMinted * newPrice / 1e18;
    assertGt(attackerEthValue, 1 ether, "attacker profited from stale price");

    // Verify: minted more rsETH than fair share
    uint256 fairRseth = 1 ether * 1e18 / newPrice;
    assertGt(rsethMinted, fairRseth, "excess rsETH minted at stale price");
}
```

The assertion passes because `rsethMinted` was computed using `stalePriceBefore` (lower), while `newPrice` (higher, reflecting the 10 ETH reward) is used to value the position.
