### Title
rsETH Burned Before Availability Check in `instantWithdrawal` Causes Permanent Fund Loss - (File: contracts/LRTWithdrawalManager.sol)

### Summary
`LRTWithdrawalManager.instantWithdrawal()` burns the caller's rsETH tokens before checking whether the `LRTUnstakingVault` has sufficient assets available. If the availability check fails after the burn, the transaction reverts but the rsETH is permanently destroyed with no assets delivered to the user.

### Finding Description
In `instantWithdrawal`, the execution order is:

1. `IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked)` — rsETH is irreversibly burned.
2. `if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) revert CantInstantWithdrawMoreThanAvailable()` — availability is checked only after the burn. [1](#0-0) 

`getAssetsAvailableForInstantWithdrawal` returns `vaultBalance - queuedWithdrawalsBuffer[asset]`, which is a live on-chain value that can change between when the user submits the transaction and when it is mined. [2](#0-1) 

If another user's `instantWithdrawal` is included in the same block first and drains the available balance, or if an operator increases `queuedWithdrawalsBuffer` in the same block, the check at line 231 reverts — but the rsETH burned at line 229 is gone forever. [3](#0-2) 

### Impact Explanation
**Critical — Permanent freezing/theft of user funds.** The user's rsETH is burned (destroyed) and they receive zero assets in return. The loss is irreversible because `burnFrom` has no corresponding mint-back on revert; Solidity reverts do not undo already-executed external calls that themselves succeeded.

### Likelihood Explanation
**Medium-High.** The scenario is realistic whenever:
- Two or more users race to call `instantWithdrawal` for the same asset in the same block, and the vault only has enough balance for one of them.
- An operator calls `setQueuedWithdrawalsBuffer` in the same block as a user's `instantWithdrawal`, reducing available capacity below `assetAmountUnlocked`.

No privileged access or external protocol compromise is required — any unprivileged rsETH holder can be the victim.

### Recommendation
Move the availability check and vault interaction **before** burning rsETH, so that if the check fails the user's tokens are never consumed:

```solidity
function instantWithdrawal(address asset, uint256 rsETHUnstaked, string calldata referralId) external ... {
    ...
    uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
    ILRTUnstakingVault unstakingVault = ILRTUnstakingVault(lrtConfig.getContract(LRTConstants.LRT_UNSTAKING_VAULT));

    // Check BEFORE burning
    if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
        revert CantInstantWithdrawMoreThanAvailable();
    }

    // Burn only after all checks pass
    IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);

    unstakingVault.redeem(asset, assetAmountUnlocked);
    ...
}
```

### Proof of Concept
1. Vault has 10 ETH available for instant withdrawal; `queuedWithdrawalsBuffer[ETH] = 0`.
2. Alice submits `instantWithdrawal(ETH, rsETHForValue10ETH, ...)` — `assetAmountUnlocked = 10 ETH`.
3. Bob submits `instantWithdrawal(ETH, rsETHForValue10ETH, ...)` in the same block.
4. Alice's tx is mined first: rsETH burned, 10 ETH transferred. Vault now has 0 ETH available.
5. Bob's tx is mined second:
   - Line 229: `burnFrom(Bob, rsETHUnstaked)` — Bob's rsETH is burned. ✓
   - Line 231: `assetAmountUnlocked (10 ETH) > getAssetsAvailableForInstantWithdrawal(ETH) (0)` → `revert CantInstantWithdrawMoreThanAvailable`. ✗
6. Bob's rsETH is permanently destroyed; he receives 0 ETH. [4](#0-3)

        external
