### Title
Missing Cancel Mechanism for Pending Withdrawal Requests Causes Indefinite Fund Freeze - (File: `contracts/LRTWithdrawalManager.sol`)

### Summary
`LRTWithdrawalManager` implements a two-step withdrawal flow: `initiateWithdrawal` locks user rsETH in the contract, and `completeWithdrawal` (after `unlockQueue`) returns the underlying asset. There is no cancel or refund path. If the queue cannot advance — because `_unlockWithdrawalRequests` breaks on the first request it cannot service — all subsequent users' rsETH is frozen in the contract with no exit, directly analogous to the rejected-auction pool that can never be closed.

### Finding Description
`initiateWithdrawal` transfers rsETH from the user into the contract and records a `WithdrawalRequest`: [1](#0-0) 

The only path forward is `unlockQueue` → `completeWithdrawal`. `_unlockWithdrawalRequests` iterates the FIFO queue and uses a hard `break` (not `continue`) when the first request it encounters cannot be serviced: [2](#0-1) 

Once the loop breaks on request N, requests N+1, N+2, … are never reached in that call. Because `nextLockedNonce[asset]` is only advanced inside the loop, those later requests remain locked. `completeWithdrawal` enforces that a request must already be unlocked before it can be completed: [3](#0-2) 

No cancel, refund, or skip function exists anywhere in the contract. The state machine for a pending withdrawal request has exactly one exit (unlock → complete) and no alternative path.

### Impact Explanation
Any user whose withdrawal request sits behind a request that cannot be unlocked (because `availableAssetAmount < payoutAmount` for the blocking request) has their rsETH locked in the contract for an indefinite period. The user cannot recover their rsETH, cannot re-initiate a withdrawal for a different asset, and cannot force the queue to skip the blocking entry. This constitutes a **temporary (potentially extended) freezing of user funds** — rsETH is held by the contract with no user-accessible exit.

### Likelihood Explanation
The scenario is reachable by any depositor without privileged access:
1. User A calls `initiateWithdrawal` for a large rsETH amount, placing a large request at the front of the queue.
2. User B calls `initiateWithdrawal` for a smaller amount, placed behind A.
3. The unstaking vault holds assets sufficient to cover B but not A.
4. `unlockQueue` is called; the loop breaks at A's request; B's request is never reached.
5. B's rsETH remains locked until A's request can be serviced (which may require additional EigenLayer unstaking cycles spanning days to weeks).

No admin action or privileged role is required to reach this state. The `initiateWithdrawal` function is open to any user. [4](#0-3) 

### Recommendation
Add a `cancelWithdrawal` function that allows a user to remove their own pending (not yet unlocked) withdrawal request and receive their rsETH back. The function should:
- Verify the request belongs to the caller and has not yet been unlocked (`userNonce >= nextLockedNonce[asset]`).
- Remove the nonce from `userAssociatedNonces[asset][user]`.
- Delete the `withdrawalRequests[requestId]` entry.
- Decrease `assetsCommitted[asset]` by `request.expectedAssetAmount`.
- Transfer `request.rsETHUnstaked` back to the user.

This closes the missing exit path from the "pending" state, directly mirroring the recommendation in the reference report to add a conditional that allows closing a stuck pool.

### Proof of Concept
```
// 1. User A initiates a large withdrawal (front of queue, nonce 0)
withdrawalManager.initiateWithdrawal(stETH, 1_000_000e18, "");

// 2. User B initiates a smaller withdrawal (nonce 1)
withdrawalManager.initiateWithdrawal(stETH, 1e18, "");

// 3. Unstaking vault only has 0.5e18 stETH available
// unlockQueue breaks at nonce 0 (A's request), never reaches nonce 1 (B's request)
withdrawalManager.unlockQueue(stETH, 2, minAsset, minRsETH, maxAsset, maxRsETH);
// → rsETHBurned == 0, assetAmountUnlocked == 0

// 4. User B tries to complete — reverts: WithdrawalLocked
withdrawalManager.completeWithdrawal(stETH, ""); // revert

// 5. No cancel function exists — B's rsETH is frozen indefinitely
``` [5](#0-4) [6](#0-5)
