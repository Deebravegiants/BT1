### Title
Permissionless `sendFunds()` susceptible to block stuffing, temporarily suppressing rsETH rate — (`contracts/FeeReceiver.sol`)

### Summary
`FeeReceiver.sendFunds()` is a permissionless function that forwards accumulated MEV/execution-layer rewards to `LRTDepositPool`. Because the FeeReceiver balance is explicitly excluded from TVL until this call is made, an attacker can perform block stuffing to delay the call, temporarily suppressing the rsETH exchange rate and denying rsETH holders their promised yield accrual for the duration of the attack.

### Finding Description
`FeeReceiver.sendFunds()` carries no role check or access restriction. It is the sole mechanism by which MEV/execution-layer rewards are reflected in the protocol's TVL and, consequently, in the rsETH price computed by `LRTOracle._updateRsETHPrice()`. The `getETHDistributionData()` function in `LRTDepositPool` explicitly documents that FeeReceiver balances are excluded from TVL until forwarded. An attacker who monitors the mempool for a pending `sendFunds()` call (or simply knows the protocol's reward-forwarding schedule) can submit N high-gas-price filler transactions that fill each block's gas limit for multiple consecutive blocks, excluding `sendFunds()` from inclusion. During this window, `FeeReceiver.balance` grows but `LRTDepositPool` receives nothing, so `_getTotalEthInProtocol()` returns a stale (lower) value, and `rsETHPrice` is not updated to reflect the accrued rewards.

### Impact Explanation
- **Temporary freezing of protocol reward funds**: MEV rewards accumulate in `FeeReceiver` but are not forwarded to `LRTDepositPool` for the duration of the attack.
- **Failure to deliver promised returns**: rsETH holders do not see the rate increase they are entitled to during the attack window. No principal is lost; the funds remain in `FeeReceiver` and will eventually be forwarded.
- Matches allowed scope: **Low. Block stuffing** and **Low. Contract fails to deliver promised returns, but doesn't lose value**.

### Likelihood Explanation
Block stuffing on Ethereum mainnet is expensive (attacker must fill ~30M gas per block at competitive base fee + priority fee). The attack is economically rational only when the MEV rewards accumulated in `FeeReceiver` are large enough to justify the cost, or when the attacker profits from a secondary position (e.g., a short on rsETH rate or a competing LRT). The permissionless nature of `sendFunds()` and the explicit TVL exclusion of `FeeReceiver` balance make the precondition structurally sound. Likelihood is **Low** but non-zero.

### Recommendation
Add a time-based automatic accounting mechanism, or include the FeeReceiver balance in `getETHDistributionData()` / `_getTotalEthInProtocol()` directly (reading `address(feeReceiver).balance` as part of TVL), so that accrued rewards are reflected in the rsETH rate regardless of whether `sendFunds()` has been called. Alternatively, restrict `sendFunds()` to a trusted keeper role and ensure the keeper infrastructure is robust against censorship.

### Proof of Concept
```solidity
// Fork test (Foundry)
function testBlockStuffingDelaySendFunds() public {
    // 1. Simulate MEV rewards accumulating in FeeReceiver
    vm.deal(address(feeReceiver), 10 ether);

    // 2. Record current rsETH price (stale, excludes FeeReceiver balance)
    uint256 priceBefore = lrtOracle.rsETHPrice();

    // 3. Simulate N blocks passing without sendFunds() being included
    //    (attacker fills blocks; modeled by simply not calling sendFunds())
    vm.roll(block.number + 50);
    vm.warp(block.timestamp + 600); // ~50 blocks * 12s

    // 4. Assert FeeReceiver still holds the balance
    assertEq(address(feeReceiver).balance, 10 ether);

    // 5. Assert LRTDepositPool ETH balance is unchanged
    assertEq(address(lrtDepositPool).balance, depositPoolBalanceBefore);

    // 6. Update oracle — price does NOT reflect the 10 ETH sitting in FeeReceiver
    lrtOracle.updateRSETHPrice();
    uint256 priceAfterDelay = lrtOracle.rsETHPrice();
    assertEq(priceAfterDelay, priceBefore); // rate unchanged despite 10 ETH in FeeReceiver

    // 7. Now call sendFunds() — price jumps
    feeReceiver.sendFunds();
    lrtOracle.updateRSETHPrice();
    uint256 priceAfterForward = lrtOracle.rsETHPrice();
    assertGt(priceAfterForward, priceBefore); // rate now reflects rewards
}
```

    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
