### Title
Uninitialized `dailyMintLimit` Blocks All Deposits After Upgrade If `reinitialize(3)` Is Skipped — (`contracts/pools/RSETHPoolV2.sol`)

### Summary
If the contract is upgraded to the version introducing `dailyMintLimit` but `reinitialize(uint256,uint256)` (reinitializer version 3) is not called, `dailyMintLimit` remains `0`. Every subsequent `deposit()` call reverts with `DailyMintLimitExceeded`, freezing all user deposits until the admin manually calls `setDailyMintLimit`.

### Finding Description
`RSETHPoolV2` introduces a `limitDailyMint` modifier that enforces `dailyMintAmount + rsETHAmount <= dailyMintLimit`. [5](#0-4)  The only way to set `dailyMintLimit` to a non-zero value post-upgrade is via `reinitialize(3)` or `setDailyMintLimit`. [7](#0-6)  If `reinitialize(3)` is omitted from the upgrade transaction, `dailyMintLimit` stays at its Solidity default of `0`. Because `rsETHAmount > 0` for any non-zero ETH deposit, the condition `0 + rsETHAmount > 0` is always true, and every `deposit()` call reverts. [8](#0-7) 

Additionally, `startTimestamp` also remains `0`, so `getCurrentDay()` returns an astronomically large value (~19,000 days since epoch), but this does not prevent the `DailyMintLimitExceeded` revert — it only resets `dailyMintAmount` to 0 on every call, which makes the limit check even more certain to fail. [4](#0-3) 

### Impact Explanation
All user deposits are blocked from the moment of upgrade until the admin calls `setDailyMintLimit`. This is a **temporary freezing of funds** (Medium). No user ETH already in the contract is lost, but no new deposits can be made during the window.

### Likelihood Explanation
Upgradeable contracts with multi-step initialization are a known source of deployment errors. The `reinitialize(3)` call is a separate transaction from the upgrade itself, and there is no on-chain enforcement requiring it to be called atomically with the upgrade. A deployment script omission or a delayed governance execution is a realistic scenario.

### Recommendation
1. **Atomic initialization**: Perform the upgrade and `reinitialize(3)` call in a single atomic transaction (e.g., via a multisig batch or upgrade script).
2. **Guard in modifier**: Treat `dailyMintLimit == 0` as "no limit enforced" (skip the check), or add an explicit revert with a more descriptive error like `DailyMintLimitNotSet` to distinguish uninitialized state from an exceeded limit.
3. **Deployment checklist**: Enforce that `dailyMintLimit > 0` is verified before the upgraded contract is considered live.

### Proof of Concept
```solidity
// 1. Deploy RSETHPoolV2 (or upgrade existing proxy)
// 2. Do NOT call reinitialize(3)
// assert: dailyMintLimit == 0, startTimestamp == 0

// 3. Any user calls deposit
pool.deposit{value: 1 ether}("");
// → reverts with DailyMintLimitExceeded
//   because: 0 + rsETHAmount > 0 is always true

// 4. Admin calls setDailyMintLimit to unblock
pool.setDailyMintLimit(100 ether); // DEFAULT_ADMIN_ROLE required

// 5. Deposit now succeeds
pool.deposit{value: 1 ether}(""); // succeeds
```

```text
    function reinitialize(
        uint256 _dailyMintLimit,
        uint256 _startTimestamp
    )
        external
        reinitializer(3)
        onlyRole(DEFAULT_ADMIN_ROLE)
    {
        if (_dailyMintLimit == 0) {
            revert InvalidDailyMintLimit();
        }

        // startTimestamp cannot be in the past
        if (block.timestamp > _startTimestamp) {
            revert InvalidStartTimestamp();
        }

        dailyMintLimit = _dailyMintLimit;
        startTimestamp = _startTimestamp;
    }
```
