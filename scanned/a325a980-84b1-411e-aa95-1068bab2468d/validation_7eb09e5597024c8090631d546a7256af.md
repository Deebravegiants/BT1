### Title
Stale `depositPool` in `FeeReceiver` causes MEV rewards to flow to deprecated deposit pool — (`contracts/FeeReceiver.sol`)

---

### Summary

`FeeReceiver` maintains its own `depositPool` storage variable that is independent of the authoritative `LRT_DEPOSIT_POOL` address registered in `LRTConfig`. Because `sendFunds()` always forwards ETH to the local variable and has no access control, any desync between the two — caused by an admin upgrading the deposit pool in `LRTConfig` without a subsequent MANAGER call to `setDepositPool()` — silently routes all accumulated MEV rewards to the old, deprecated pool.

---

### Finding Description

`FeeReceiver.sendFunds()` reads from its own `depositPool` storage slot: [1](#0-0) 

This address is set at initialization and can only be updated by the MANAGER via `setDepositPool()`: [2](#0-1) 

`LRTConfig` maintains a completely separate registry of the canonical deposit pool under `LRT_DEPOSIT_POOL`, updated by the admin via `setContract()`: [3](#0-2) 

There is no on-chain link between these two values. `FeeReceiver` never queries `LRTConfig` for the current deposit pool address. When the admin upgrades the deposit pool (a normal protocol operation), the MANAGER must separately call `FeeReceiver.setDepositPool()`. If that second step is missed or delayed, `sendFunds()` — which is callable by anyone with no access control — will forward all ETH to the old pool.

---

### Impact Explanation

MEV/execution-layer rewards accumulate in `FeeReceiver` and are forwarded to the stale deposit pool. The old pool's `receiveFromRewardReceiver` credits ETH to its own TVL accounting, not the new pool's. The new pool's TVL is understated, so the rsETH exchange rate computed by `LRTOracle` is lower than it should be. rsETH holders receive less yield than promised until the MANAGER manually corrects the address. No ETH is permanently lost, matching **Low: Contract fails to deliver promised returns, but doesn't lose value**.

---

### Likelihood Explanation

Deposit pool upgrades are a realistic operational event (the codebase already supports `LRTConfig.setContract` for this purpose). The two-step update — LRTConfig then FeeReceiver — is not atomic and is not enforced on-chain. A missed or delayed second step is a plausible operational oversight, not a theoretical one. `sendFunds()` is permissionless, so any caller can trigger the misdirection immediately after the LRTConfig update.

---

### Recommendation

Remove the local `depositPool` storage variable entirely. In `sendFunds()`, resolve the deposit pool address dynamically from `LRTConfig` at call time, the same way all other protocol contracts do:

```solidity
address pool = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
ILRTDepositPool(pool).receiveFromRewardReceiver{ value: balance }();
```

This eliminates the dual-state problem and ensures MEV rewards always flow to the currently registered deposit pool without requiring a separate MANAGER update.

---

### Proof of Concept

```solidity
// 1. Deploy LRTConfig, OldPool, NewPool, FeeReceiver(depositPool = OldPool)
// 2. Admin calls LRTConfig.setContract(LRT_DEPOSIT_POOL, address(NewPool))
// 3. MANAGER does NOT call FeeReceiver.setDepositPool(address(NewPool))
// 4. Send 1 ETH to FeeReceiver
// 5. Anyone calls FeeReceiver.sendFunds()
// 6. Assert: OldPool.balance == 1 ether  ✓
//            NewPool.balance == 0         ✓  (TVL unaffected)
//    FeeReceiver.depositPool still == address(OldPool)
``` [4](#0-3) [1](#0-0)

        if (contractMap[key] == val) {
            revert ValueAlreadyInUse();
        }
