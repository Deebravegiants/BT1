### Title
Exact-equality fee check in `bridgeRsETHToL2UsingCCIP()` causes unnecessary reverts when CCIP fee increases between estimation and execution — (`contracts/L1VaultV2.sol`)

### Summary

`bridgeRsETHToL2UsingCCIP()` computes the required CCIP fee on-chain at execution time and enforces `msg.value != fee` with exact equality. Because the CCIP router explicitly accepts overpayment, this check is unnecessarily strict. Any fee increase between the manager's off-chain estimation and on-chain execution causes an `IncorrectCCIPFee` revert, leaving rsETH idle in `L1VaultV2` and delaying L2 users.

### Finding Description

In `bridgeRsETHToL2UsingCCIP()`, the fee is computed on-chain at execution time:

```solidity
uint256 fee = getCCIPFee(amount);   // calls ccipRouter.getFee() at tx execution time

if (msg.value != fee) {             // exact equality — not >= or <=
    revert IncorrectCCIPFee();
}
``` [1](#0-0) 

`getCCIPFee()` delegates directly to `ccipRouter.getFee()`:

```solidity
function getCCIPFee(uint256 amount) public view returns (uint256) {
    Client.EVM2AnyMessage memory message = getCCIPMessage(amount);
    return ccipRouter.getFee(destinationChainSelector, message);
}
``` [2](#0-1) 

The CCIP router's own interface explicitly documents that overpayment is accepted:

```
/// @dev Note if msg.value is larger than the required fee (from getFee) we accept
/// the overpayment with no refund.
``` [3](#0-2) 

The manager's workflow is:
1. Call `getCCIPFee(amount)` off-chain (or in a prior block) to estimate the fee.
2. Submit `bridgeRsETHToL2UsingCCIP(amount)` with `msg.value == estimated_fee`.
3. At execution time, `getCCIPFee(amount)` is called again on-chain. If the CCIP router's fee has increased (e.g., due to destination-chain gas price movement), `msg.value < fee` and the exact-equality check reverts with `IncorrectCCIPFee`.

The root cause is the contract's own `!=` guard, not the external fee change itself. The CCIP router would have accepted the overpayment; the contract rejects it first.

This is structurally different from the LayerZero path, where the manager passes `nativeFee` as a calldata parameter and the contract simply forwards it — the manager always controls the exact value. For CCIP, the fee is re-derived on-chain, making exact equality fragile.

### Impact Explanation

rsETH accumulates in `L1VaultV2` without being bridged. L2 users are delayed in receiving their tokens. No funds are lost; the manager can retry with a fresh fee estimate. This matches **Low — Contract fails to deliver promised returns, but doesn't lose value**.

### Likelihood Explanation

CCIP fees are derived from destination-chain gas prices and token oracle prices, both of which can move between blocks. During periods of elevated L2 gas prices or ETH price volatility, fee increases between estimation and execution are realistic. The manager must retry every time this occurs, and in sustained volatile conditions the bridge can be repeatedly stalled.

### Recommendation

Replace the exact-equality check with a minimum-fee check, consistent with how the CCIP router itself behaves:

```solidity
// Before
if (msg.value != fee) {
    revert IncorrectCCIPFee();
}

// After
if (msg.value < fee) {
    revert IncorrectCCIPFee();
}
```

This allows the manager to send a small buffer above the estimated fee without causing a revert, while still preventing underpayment.

pragma solidity ^0.8.21;

import "forge-std/Test.sol";
import "../contracts/L1VaultV2.sol";

contract MockCCIPRouter {
    uint256 public feeToReturn;

    function setFee(uint256 _fee) external { feeToReturn = _fee; }

    function getFee(uint64, Client.EVM2AnyMessage memory)
        external view returns (uint256) { return feeToReturn; }

    function ccipSend(uint64, Client.EVM2AnyMessage calldata)
        external payable returns (bytes32) { return bytes32(0); }
}

contract BridgeFeeRaceTest is Test {
    L1VaultV2 vault;
    MockCCIPRouter router;

    function setUp() public {
        router = new MockCCIPRouter();
        // ... deploy and initialize vault with router, grant MANAGER_ROLE to address(this)
    }

    function test_feeIncreaseReverts() public {
        uint256 amount = 1 ether;
        // Step 1: manager estimates fee off-chain
        router.setFee(0.01 ether);
        uint256 estimatedFee = vault.getCCIPFee(amount);  // 0.01 ether

        // Step 2: fee increases before tx lands
        router.setFee(0.02 ether);

        // Step 3: manager submits with old estimate — reverts
        vm.expectRevert(L1VaultV2.IncorrectCCIPFee.selector);
        vault.bridgeRsETHToL2UsingCCIP{value: estimatedFee}(amount);
    }
}
```

The test mocks `ccipRouter.getFee()` to return different values at estimation vs. execution time and asserts the `IncorrectCCIPFee` revert, confirming the issue on unmodified production code.
