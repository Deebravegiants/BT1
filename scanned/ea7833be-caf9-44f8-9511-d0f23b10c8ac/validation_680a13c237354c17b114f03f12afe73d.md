### Title
`pricePercentageLimit` Defaults to Zero, Completely Disabling Price Deviation Protection — (File: contracts/LRTOracle.sol)

---

### Summary

`LRTOracle` contains a `pricePercentageLimit` variable intended to act as a circuit breaker against extreme rsETH price movements. Both the upside revert guard and the downside auto-pause guard are gated on `pricePercentageLimit > 0`. Because `pricePercentageLimit` is never initialized in `initialize()` or `reinitialize()`, it defaults to `0`, silently disabling all price deviation protection. This is the direct analog to the reported issue: instead of a threshold that is too high, the threshold here is effectively infinite (disabled entirely).

---

### Finding Description

In `_updateRsETHPrice()`, two circuit breakers exist:

**Upside guard** — reverts if the new rsETH price rises above `pricePercentageLimit` relative to `highestRsethPrice`:

```solidity
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
```

**Downside guard** — auto-pauses the protocol if the new rsETH price falls below `pricePercentageLimit` relative to `highestRsethPrice`:

```solidity
bool isPriceDecreaseOffLimit =
    pricePercentageLimit > 0 && diff > pricePercentageLimit.mulWad(highestRsethPrice);
```

Both conditions short-circuit to `false` whenever `pricePercentageLimit == 0`. [1](#0-0) [2](#0-1) 

The variable is declared as a plain `uint256` storage slot with no default assignment: [3](#0-2) 

Neither `initialize()` nor `reinitialize()` sets `pricePercentageLimit`: [4](#0-3) [5](#0-4) 

The only way to activate the protection is via `setPricePercentageLimit()`, which requires `onlyLRTAdmin`: [6](#0-5) 

Until that call is made, the contract operates with zero price deviation protection.

The rsETH price is computed from raw Chainlink oracle prices with no staleness or deviation check in `ChainlinkPriceOracle.getAssetPrice()`: [7](#0-6) 

This raw price feeds directly into `_getTotalEthInProtocol()` and then into `newRsETHPrice`: [8](#0-7) 

---

### Impact Explanation

With `pricePercentageLimit == 0`:

1. **No upside circuit breaker**: Any arbitrarily large rsETH price increase is accepted and stored. Depositors calling `depositETH` / `depositAsset` receive rsETH minted at `(amount * assetPrice) / rsETHPrice`. An inflated `rsETHPrice` causes depositors to receive fewer rsETH tokens than they are entitled to — share mis-accounting / theft of depositor value. [9](#0-8) 

2. **No downside auto-pause**: A significant price drop (e.g., an underlying LST depeg) does not trigger the auto-pause of `LRTDepositPool` and `LRTWithdrawalManager`. The protocol continues accepting deposits and processing withdrawals at a stale/incorrect rsETH price, leading to temporary freezing of funds or incorrect payouts. [10](#0-9) 

**Impact**: Medium — temporary freezing of funds / share mis-accounting.

---

### Likelihood Explanation

`updateRSETHPrice()` is publicly callable by any address: [11](#0-10) 

Any external caller can trigger a price update at any time. If `pricePercentageLimit` has not been set by the admin (the default state), the circuit breakers are silently absent. A Chainlink oracle returning an anomalous price (e.g., due to a flash-loan-induced price spike on a DEX-backed feed, or a delayed update during a depeg) would propagate directly into `rsETHPrice` with no on-chain guard.

**Likelihood**: Low-to-Medium — depends on whether the admin has set `pricePercentageLimit` post-deployment; the code provides no guarantee it is ever set.

---

### Recommendation

Initialize `pricePercentageLimit` to a conservative non-zero value (e.g., `5e16` for 5%) inside `initialize()` or `reinitialize()`, and add an upper-bound validation in `setPricePercentageLimit()` to prevent the admin from setting it to an excessively large value:

```solidity
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    lrtConfig = ILRTConfig(lrtConfigAddr);
    pricePercentageLimit = 5e16; // 5% default
    emit UpdatedLRTConfig(lrtConfigAddr);
}

function setPricePercentageLimit(uint256 _pricePercentageLimit) external onlyLRTAdmin {
    if (_pricePercentageLimit == 0 || _pricePercentageLimit > 2e17) revert InvalidPricePercentageLimit(); // max 20%
    pricePercentageLimit = _pricePercentageLimit;
    emit PricePercentageLimitUpdate(_pricePercentageLimit);
}
```

### Proof of Concept

1. Deploy `LRTOracle` (or observe the live contract). `pricePercentageLimit` is `0`.
2. Call `updateRSETHPrice()` as any EOA.
3. Inside `_updateRsETHPrice()`, both `isPriceIncreaseOffLimit` and `isPriceDecreaseOffLimit` evaluate to `false` regardless of how extreme `newRsETHPrice` is, because `pricePercentageLimit > 0` is `false`.
4. An anomalous Chainlink price (e.g., stETH/ETH feed returning `0.5e18` during a depeg) flows through `_getTotalEthInProtocol()` → `newRsETHPrice` is halved → no auto-pause fires → `LRTDepositPool` and `LRTWithdrawalManager` remain open → users continue depositing/withdrawing at the incorrect price.
5. Conversely, an inflated oracle price doubles `newRsETHPrice` → no revert → `rsETHPrice` is updated to the inflated value → subsequent depositors receive half the rsETH they should. [12](#0-11) [13](#0-12)
