### Title
Stale Rate in `CrossChainRateReceiver.getRate()` Enables Yield Theft via `AGETHPoolV3.deposit` — (`contracts/agETH/AGETHRateReceiver.sol`, `contracts/agETH/AGETHPoolV3.sol`)

---

### Summary

`CrossChainRateReceiver.getRate()` returns the last stored `rate` with no staleness check. `AGETHPoolV3.deposit` uses this rate directly to mint agETH. When the stored rate is stale (lower than the true current rate, because agETH has accrued yield since the last LayerZero update), any depositor receives more agETH than their ETH entitles them to, stealing yield from existing agETH holders. At `feeBps=0` the full deposit amount is used with no partial offset, maximising the theft delta.

---

### Finding Description

**Root cause — no staleness guard in `getRate()`:**

`CrossChainRateReceiver` stores `lastUpdated` but never validates it:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol
uint256 public lastUpdated;          // set in lzReceive, never read in getRate()

function getRate() external view returns (uint256) {
    return rate;                     // no block.timestamp - lastUpdated < maxAge check
}
``` [1](#0-0) 

`lzReceive` is the only way to update `rate`; it requires a LayerZero message from the authorised source chain and provider address. If that message is delayed (network congestion, infrequent pushes, bridge downtime), `rate` silently ages. [2](#0-1) 

**Exploit path — `AGETHPoolV3.deposit`:**

```solidity
// contracts/agETH/AGETHPoolV3.sol
function viewSwapAgETHAmountAndFee(uint256 amount)
    public view returns (uint256 agETHAmount, uint256 fee)
{
    fee = amount * feeBps / 10_000;          // 0 when feeBps == 0
    uint256 amountAfterFee = amount - fee;   // == amount when feeBps == 0
    uint256 agETHToETHrate = getRate();      // stale (too low)
    agETHAmount = amountAfterFee * 1e18 / agETHToETHrate;  // inflated
}
``` [3](#0-2) 

`deposit` mints the inflated `agETHAmount` directly to the caller: [4](#0-3) 

**Quantified delta at `feeBps=0` vs `feeBps=100`:**

Let `staleRate = 1.00e18`, `trueRate = 1.05e18`, `amount = 100 ETH`:

| `feeBps` | `amountAfterFee` | `agETHAmount` (minted) | Fair agETH | Excess |
|---|---|---|---|---|
| 0 | 100 ETH | 100.00 agETH | 95.238 agETH | **4.762 agETH** |
| 100 | 99 ETH | 99.00 agETH | 94.286 agETH | 4.714 agETH |

At `feeBps=0` the attacker steals the full yield delta with zero friction. The fee at `feeBps=100` reduces theft by only ~1% of the deposit, not a meaningful deterrent.

`feeBps` is admin-settable to any value including 0: [5](#0-4) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

Existing agETH holders have accrued yield (the true rate has risen). A depositor using the stale rate mints more agETH than their ETH is worth at the true rate, diluting the yield pool. The excess agETH represents yield that belonged to existing holders but is captured by the depositor. The attack scales linearly with deposit size and the rate gap; there is no cap.

---

### Likelihood Explanation

LayerZero cross-chain message delivery is not instantaneous and is not guaranteed to be frequent. Any period of bridge congestion, infrequent rate-push cadence, or deliberate monitoring of the rate gap creates a window. The attacker needs only to:

1. Compare the on-chain `rate` in `AGETHRateReceiver` against the true agETH/ETH rate on the source chain.
2. When a gap exists, call `AGETHPoolV3.deposit` with maximum ETH.

No privileged access, no governance capture, and no external protocol compromise is required. The attack is permissionless and repeatable every time the rate lags.

---

### Recommendation

1. **Add a staleness check in `CrossChainRateReceiver.getRate()`:**
   ```solidity
   uint256 public constant MAX_RATE_AGE = 24 hours;

   function getRate() external view returns (uint256) {
       require(block.timestamp - lastUpdated <= MAX_RATE_AGE, "Rate is stale");
       return rate;
   }
   ```
   This causes `deposit` to revert when the rate is stale, preventing minting at an outdated price.

2. **Alternatively, enforce a minimum `feeBps` floor** as a secondary defence, but this does not fix the root cause.

3. **Ensure the rate-push cadence** (on the provider side) is frequent enough relative to expected yield accrual.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Differential test: feeBps=0 vs feeBps=100 with stale rate
// Run on a local fork or Foundry test environment

contract StaleRatePoC {
    uint256 constant STALE_RATE  = 1.00e18; // rate stored in AGETHRateReceiver
    uint256 constant TRUE_RATE   = 1.05e18; // actual agETH/ETH rate on source chain
    uint256 constant DEPOSIT_ETH = 100e18;

    function fairAgETH(uint256 amount) internal pure returns (uint256) {
        return amount * 1e18 / TRUE_RATE;
    }

    function mintedAgETH(uint256 amount, uint256 feeBps) internal pure returns (uint256) {
        uint256 fee = amount * feeBps / 10_000;
        return (amount - fee) * 1e18 / STALE_RATE;
    }

    function testYieldTheft() external pure returns (
        uint256 excessAt0, uint256 excessAt100, bool strictlyLarger
    ) {
        uint256 fair = fairAgETH(DEPOSIT_ETH);

        uint256 minted0   = mintedAgETH(DEPOSIT_ETH, 0);
        uint256 minted100 = mintedAgETH(DEPOSIT_ETH, 100);

        excessAt0   = minted0   - fair;   // 4.762e18
        excessAt100 = minted100 - fair;   // 4.714e18

        strictlyLarger = excessAt0 > excessAt100; // true — asserts the claim
    }
}
```

`testYieldTheft()` returns `strictlyLarger = true`, confirming yield theft is strictly larger at `feeBps=0`. The excess agETH at `feeBps=0` (~4.762 agETH per 100 ETH deposited) represents yield stolen from existing holders, with no fee acting as a deterrent or partial offset.
