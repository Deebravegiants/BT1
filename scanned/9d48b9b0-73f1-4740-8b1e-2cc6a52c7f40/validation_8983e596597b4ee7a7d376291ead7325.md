### Title
Block Stuffing Can Delay Emergency Deposit-Limit or Fee Reduction by Manager — (`contracts/LRTConfig.sol`)

---

### Summary

Both `updateAssetDepositLimit` and `setProtocolFeeBps` are single-SSTORE, no-timelock manager transactions. An attacker with sufficient ETH can fill consecutive blocks to delay these transactions, keeping the old (higher) deposit limit or fee in effect for additional blocks.

---

### Finding Description

`updateAssetDepositLimit` writes one storage slot: [1](#0-0) 

`setProtocolFeeBps` writes one storage slot: [2](#0-1) 

Neither function has a timelock, commit-reveal, or any on-chain mechanism that would guarantee inclusion within a bounded number of blocks. The deposit limit is enforced at deposit time via: [3](#0-2) 

Until the manager's transaction lands, `lrtConfig.depositLimitByAsset(asset)` still returns the old value, so `_beforeDeposit` continues to accept deposits up to the old limit: [4](#0-3) 

An attacker who monitors the mempool for an emergency `updateAssetDepositLimit(asset, 0)` or `setProtocolFeeBps(0)` can submit high-gas filler transactions that consume the entire block gas limit, pushing the manager's transaction out of the block for K blocks. During those K blocks, deposits continue to be accepted at the old limit and fees continue to accrue at the old rate.

---

### Impact Explanation

**Low — Block stuffing / contract fails to deliver promised returns.**

During the delay window the protocol continues to accept deposits into an asset the manager intended to halt, and continues to collect fees at a rate the manager intended to reduce. This matches the allowed scope: *"Low. Block stuffing."* and *"Low. Contract fails to deliver promised returns, but doesn't lose value."*

---

### Likelihood Explanation

Block stuffing on Ethereum mainnet requires filling ~30 M gas per block, which is expensive but not impossible for a well-capitalised attacker who stands to benefit from continued deposits into a compromised asset. The attack requires no privileged access — any EOA can submit filler transactions. The manager has no on-chain fallback that bypasses the mempool for these specific config writes (the `pauseAll()` path at lines 262–285 is itself a mempool transaction and equally subject to stuffing): [5](#0-4) 

---

### Recommendation

1. **Add a `pauseAll`-first policy in runbooks**: In an emergency the PAUSER\_ROLE should call `pauseAll()` *before* or *instead of* updating config values; pausing the deposit pool is the fastest single-transaction mitigation and has the same block-stuffing exposure, but it is a simpler, lower-gas target.
2. **Consider a priority-fee floor or private mempool (Flashbots `eth_sendPrivateTransaction`)** for emergency manager transactions so they cannot be displaced by filler transactions.
3. **Optionally add a `maxDepositLimitReductionDelay`** — if the new limit is lower than the current one, enforce it immediately via a flag rather than a storage write that must be mined.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

// Foundry fork test (local fork, no mainnet testing)
// 1. Fork mainnet state locally.
// 2. Manager submits updateAssetDepositLimit(stETH, 0) with base fee priority.
// 3. Attacker submits N filler transactions each consuming ~29.9 M gas per block
//    (e.g., a tight loop of SSTOREs) with higher effective gas price.
// 4. Assert manager tx is not included for K blocks.
// 5. Assert depositAsset(stETH, amount, ...) still succeeds during those K blocks
//    because depositLimitByAsset[stETH] still equals the old value.
// 6. After K blocks, manager tx lands; assert depositAsset now reverts with
//    MaximumDepositLimitReached.

contract BlockStuffingFiller {
    uint256[500] private _slots;
    function fill() external {
        for (uint256 i; i < 500; ++i) {
            _slots[i] = block.number; // 500 SSTOREs ≈ 10 M gas
        }
    }
}
```

The test measures `lrtConfig.depositLimitByAsset(stETH)` before and after the stuffed blocks to confirm the old limit persists, and calls `depositPool.depositAsset(stETH, amount, 0, "")` within the stuffed window to confirm deposits are accepted at the old limit. [6](#0-5) [7](#0-6)
