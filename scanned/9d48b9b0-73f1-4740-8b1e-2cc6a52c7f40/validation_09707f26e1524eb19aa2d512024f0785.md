### Title
Rebasing Yield Permanently Frozen in Wrapper Contracts Due to 1:1 Burn-Only Withdrawal — (File: `contracts/L2/RsETHTokenWrapper.sol`, `contracts/agETH/AGETHTokenWrapper.sol`)

---

### Summary

`RsETHTokenWrapper` and `AGETHTokenWrapper` hold rebasing tokens (altRsETH / altAgETH) and issue wrapper tokens (wrsETH / wrapped agETH) on a strict 1:1 basis. Because the underlying tokens are rebasing (their balances grow as staking rewards accrue), the actual token balance held by the wrapper will exceed the wrapper's `totalSupply()` over time. The excess rebased tokens have no withdrawal path and are permanently frozen in the contract.

---

### Finding Description

Both wrapper contracts implement the same deposit/withdraw pattern:

**Deposit** — mints exactly `_amount` wrapper tokens for `_amount` underlying tokens transferred in. [1](#0-0) 

**Withdraw** — burns exactly `_amount` wrapper tokens and transfers exactly `_amount` underlying tokens out. [2](#0-1) 

The same logic exists in `RsETHTokenWrapper._withdraw` and `RsETHTokenWrapper._deposit`. [3](#0-2) 

Because the relationship is always 1:1, the maximum amount any user can ever withdraw is bounded by `totalSupply()` of the wrapper token. Any altRsETH / altAgETH that accrues inside the contract through rebasing has no corresponding wrapper token to burn, so it can never leave the contract.

Additionally, the `maxAmountToDepositBridgerAsset` guard explicitly returns `0` (blocking further bridger deposits) whenever `balanceOfAssetInWrapper > wrsETHSupply`: [4](#0-3) [5](#0-4) 

Once a rebase occurs, this check permanently evaluates to `0`, blocking the bridger from depositing additional collateral to back newly bridged wrsETH, compounding the accounting desync.

Neither contract contains a sweep, rescue, or yield-distribution function for excess tokens.

---

### Impact Explanation

All staking yield that accrues to altRsETH / altAgETH tokens held inside the wrapper contracts is permanently frozen. No user, operator, or admin can recover it. This matches the **Medium — Permanent freezing of unclaimed yield** impact category.

---

### Likelihood Explanation

rsETH is explicitly designed to accrue EigenLayer restaking rewards over time; its balance grows automatically without any user action. Every deployment of `RsETHTokenWrapper` or `AGETHTokenWrapper` that holds altRsETH / altAgETH will accumulate frozen yield continuously from the moment of first deposit. No attacker action is required — the rebase is a normal protocol operation.

---

### Recommendation

Adopt one of the following approaches:

1. **Track a stored balance** (analogous to the Curve fix): record the balance at deposit/withdraw time and use the delta to detect and handle rebased amounts — either distributing them to wrapper holders or allowing an admin to sweep them.
2. **Use a share-based accounting model**: instead of minting 1:1, mint shares proportional to the current `balanceOfAssetInWrapper / totalSupply()` ratio (similar to wstETH), so rebased yield is automatically reflected in the redemption rate.
3. **Add a `collectYield` function** restricted to a trusted role that transfers the excess (`balanceOfAsset - totalSupply`) to a treasury or distributes it pro-rata to wrapper holders.

---

### Proof of Concept

1. Alice deposits 100 altRsETH → receives 100 wrsETH. `totalSupply = 100`, `balance = 100`.
2. Time passes; altRsETH rebases. `balance` grows to 110 altRsETH inside the wrapper. `totalSupply` remains 100 wrsETH.
3. Alice calls `withdraw(asset, 100)` → burns 100 wrsETH, receives 100 altRsETH. `totalSupply = 0`, `balance = 10`.
4. The 10 altRsETH representing accrued staking yield is now permanently locked. No function exists to extract it.
5. Simultaneously, `maxAmountToDepositBridgerAsset` returns `0` (step 2 onward), blocking the bridger from depositing collateral for any newly bridged wrsETH. [4](#0-3) [2](#0-1)
