### Title
Merkle Leaf Hash Missing Contract Address Enables Cross-Contract Proof Replay ‚Äî (`contracts/KERNEL/KernelMerkleDistributor.sol`, `contracts/utils/MerkleDistributor/MerkleDistributor.sol`)

---

### Summary

`KernelMerkleDistributor` and `MerkleDistributor` compute their Merkle leaf hashes using an identical encoding that omits `address(this)`. This is the direct structural analog to the ERC-7739 bug: just as SmartSession used its own shared address instead of the individual account address in the domain separator, these distributor contracts omit the contract address from the cryptographic leaf, making proofs valid across any contract that shares the same Merkle root and leaf schema.

---

### Finding Description

Both distributor contracts compute the leaf node identically:

**`KernelMerkleDistributor._processClaim`** (line 320):
```solidity
bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));
``` [1](#0-0) 

**`MerkleDistributor.claim`** (line 120):
```solidity
bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));
``` [2](#0-1) 

Neither encoding includes `address(this)`. A Merkle proof is therefore not cryptographically bound to any specific distributor contract. If the same Merkle root is set in both contracts ‚Äî or across two instances of `MerkleDistributor` deployed for different tokens ‚Äî a proof that is valid in one contract is equally valid in the other.

The protocol's multi-distributor architecture makes this realistic: `KernelMerkleDistributor` distributes KERNEL tokens and `MerkleDistributor` distributes a configurable ERC-20 token. Both are independently administered, both accept the same leaf schema, and both expose a public `claim` function. `MerkleDistributor` additionally has no `msg.sender == account` guard, meaning any caller can submit a replayed proof on behalf of any account. [3](#0-2) [4](#0-3) 

---

### Impact Explanation

If the same Merkle root is active in both contracts simultaneously, an attacker who holds a valid proof `(index, account, cumulativeAmount, proof[])` for `KernelMerkleDistributor` can submit the identical tuple to `MerkleDistributor.claim`, passing proof verification and draining the corresponding token balance from `MerkleDistributor`. The attacker receives tokens they were never allocated, constituting **theft of unclaimed yield** from the second distributor.

---

### Likelihood Explanation

The protocol already deploys multiple distributor contracts with identical leaf schemas. The Merkle roots are set by the owner via `setMerkleRoot`. A realistic trigger is:

1. The protocol reuses the same off-chain distribution dataset for two separate reward campaigns (e.g., KERNEL airdrop and an rsETH reward distribution), producing the same Merkle root.
2. An operator error sets the same root in both contracts.
3. Any user who received a proof for the first contract immediately replays it against the second.

Because `MerkleDistributor` imposes no `msg.sender` restriction, the attacker does not even need to be the beneficiary ‚Äî they can replay proofs for any account and drain the contract. [5](#0-4) 

---

### Recommendation

Include `address(this)` in every Merkle leaf hash so that a proof is cryptographically bound to the specific distributor contract for which it was generated:

```solidity
// KernelMerkleDistributor._processClaim  (line 320)
// MerkleDistributor.claim                (line 120)
bytes32 node = keccak256(abi.encodePacked(address(this), index, account, cumulativeAmount));
```

This mirrors the ERC-7739 fix: use the individual contract's own address (the analog of `verifyingContract`) rather than a shared or absent context. The off-chain Merkle tree generation must be updated to include `address(this)` in the leaf preimage accordingly.

---

### Proof of Concept

1. Owner sets the same `merkleRoot` in both `KernelMerkleDistributor` (K) and `MerkleDistributor` (M).
2. Alice legitimately calls `K.claim(1, alice, 1000e18, proof)` and receives 1000 KERNEL (minus fee). Her `userClaims` entry in K is updated.
3. Attacker (or Alice herself) calls `M.claim(1, alice, 1000e18, proof)` with the identical arguments. Because M's leaf hash is `keccak256(abi.encodePacked(1, alice, 1000e18))` ‚Äî identical to K's ‚Äî `MerkleProofUpgradeable.verify` returns `true`. Alice's `userClaims` entry in M is zero, so `isClaimed` returns `false`. The full `cumulativeAmount` is transferred from M's token balance to Alice.
4. Alice has now claimed 1000 tokens from K and 1000 tokens from M using a single proof, stealing the second allocation entirely. [6](#0-5) [7](#0-6)

    function claim(
        uint256 index,
        address account,
        uint256 cumulativeAmount,
        bytes32[] calldata merkleProof
    )
        external
        override
        whenNotPaused
    {
        if (currentMerkleRoot == bytes32(0)) {
            revert ZeroValueProvided();
        }

        if (index == 0 || index > currentIndex) {
            revert InvalidIndex();
        }

        if (isClaimed(index, account)) {
            revert AlreadyClaimed();
        }

        // Verify the merkle proof.
        bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));
        if (!MerkleProofUpgradeable.verify(merkleProof, currentMerkleRoot, node)) {
            revert InvalidMerkleProof();
        }

        // Calculate the claimable amount
        uint256 claimableAmount = cumulativeAmount - userClaims[account].cumulativeAmount;

        // Ensure there is something to claim
        if (claimableAmount == 0) {
            revert NoTokensToClaim();
        }

        // Update user claim info, and send the token.
        userClaims[account].lastClaimedIndex = index;
        userClaims[account].cumulativeAmount = cumulativeAmount;

        // Send the claimable amount to the user - deducting the fee
        uint256 fee = (claimableAmount * feeInBPS) / 10_000;
        uint256 amountToSend = claimableAmount - fee;

        IERC20(token).safeTransfer(account, amountToSend);

        // Send the fee to the protocol treasury
        IERC20(token).safeTransfer(protocolTreasury, fee);

        emit Claimed(index, account, claimableAmount);
    }
```
