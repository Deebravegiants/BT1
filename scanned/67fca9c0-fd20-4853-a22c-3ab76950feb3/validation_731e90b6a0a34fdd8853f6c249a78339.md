### Title
Stale Chainlink Price Accepted Without Staleness Validation in `ChainlinkPriceOracle.getAssetPrice()` — (File: contracts/oracles/ChainlinkPriceOracle.sol)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `answer`, performing no staleness check (`updatedAt`), no round-completeness check (`answeredInRound < roundId`), and no non-positive price check. A stale or frozen Chainlink feed for any supported LST (e.g., stETH/ETH) will silently propagate an incorrect price into rsETH minting and TVL accounting.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the Chainlink price as follows:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All five return values of `latestRoundData()` — `roundId`, `answer`, `startedAt`, `updatedAt`, `answeredInRound` — are available, but only `answer` is used. The contract never checks:

- `updatedAt == 0` (incomplete round)
- `block.timestamp - updatedAt > heartbeat` (stale price)
- `answeredInRound < roundId` (answer from a prior round)
- `price <= 0` (invalid answer)

This is in direct contrast to `ChainlinkOracleForRSETHPoolCollateral.getRate()`, which is used for pool collateral pricing and **does** perform all three of these checks:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The price returned by `ChainlinkPriceOracle.getAssetPrice()` flows into two critical paths:

1. **`LRTDepositPool.getRsETHAmountToMint()`** — directly determines how many rsETH tokens a depositor receives per unit of LST deposited.
2. **`LRTOracle._getTotalEthInProtocol()`** — aggregates the ETH value of all supported assets to compute the global `rsETHPrice` stored on-chain.

---

### Impact Explanation

If a Chainlink feed for a supported LST (e.g., stETH/ETH) becomes stale while the actual market price has dropped (e.g., a depeg event), the protocol continues to value that LST at the last reported (inflated) price. A depositor can then:

- Deposit depegged LSTs at the stale inflated price.
- Receive more rsETH than the deposited assets are actually worth.
- Redeem or sell that rsETH, extracting value from existing rsETH holders.

This constitutes direct theft of funds from existing rsETH holders (protocol insolvency / share dilution). The `pricePercentageLimit` guard in `_updateRsETHPrice()` only triggers on large single-step price jumps during an oracle update call; it does not protect against a stale feed that simply stops updating.

**Impact: Critical** — Direct theft of funds from existing rsETH holders via inflated mint amounts.

---

### Likelihood Explanation

Chainlink feeds go stale during network congestion, extreme market volatility, or oracle node failures. The stETH depeg in May 2022 and the USDC depeg in March 2023 are real-world precedents. The protocol supports multiple LSTs, each with its own Chainlink feed, increasing the attack surface. No heartbeat or deviation threshold is enforced on-chain, so any feed that stops updating will silently serve its last answer indefinitely.

**Likelihood: Medium** — Requires a Chainlink feed to go stale, which is a known and historically observed failure mode.

---

### Recommendation

Apply the same staleness and validity checks already present in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    // Optionally: if (block.timestamp - updatedAt > MAX_HEARTBEAT) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

---

### Proof of Concept

1. Protocol supports stETH with a Chainlink stETH/ETH feed registered in `ChainlinkPriceOracle`.
2. A market event causes stETH to depeg to 0.90 ETH, but the Chainlink feed stalls and continues to report 1.00 ETH (last answer before the stall).
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18)`.
4. `getRsETHAmountToMint(stETH, 1000e18)` calls `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns stale `1.00e18`.
5. `rsethAmountToMint = (1000e18 * 1.00e18) / rsETHPrice` — attacker receives rsETH valued at 1000 ETH.
6. Actual value deposited is only 900 ETH (at true market price).
7. Attacker redeems or sells rsETH, extracting ~100 ETH of value from existing rsETH holders.

The stale price is accepted silently because `ChainlinkPriceOracle.getAssetPrice()` never validates `updatedAt` or `answeredInRound`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
