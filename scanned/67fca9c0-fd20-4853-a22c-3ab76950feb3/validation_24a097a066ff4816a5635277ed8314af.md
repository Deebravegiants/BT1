### Title
Missing Minimum Output Slippage Protection on `deposit` Functions - (`contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`, `contracts/pools/RSETHPoolV3WithNativeChainBridge.sol`, `contracts/pools/RSETHPoolNoWrapper.sol`)

---

### Summary

All L2 pool `deposit` functions accept ETH or supported tokens and mint `wrsETH`/`rsETH` to the caller, but none accept a `minRsETH` parameter. The minted amount is computed entirely from a live oracle rate that can shift between the user's off-chain simulation and on-chain execution, leaving the user with no way to revert if the received amount falls below their expectation.

---

### Finding Description

Every pool variant exposes two public `deposit` entry points:

- `deposit(string referralId)` — ETH path
- `deposit(address token, uint256 amount, string referralId)` — ERC-20 path

In both cases the output amount is computed by `viewSwapRsETHAmountAndFee`, which reads a live oracle:

```solidity
// RSETHPoolV3.sol lines 299-308
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();          // live oracle call
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
```

The result is immediately minted with no floor check:

```solidity
// RSETHPoolV3.sol lines 258-262
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
wrsETH.mint(msg.sender, rsETHAmount);   // no minRsETH guard
```

The identical pattern appears in:
- `RSETHPoolV3ExternalBridge.sol` lines 377–383 (ETH path) and 405–409 (token path)
- `RSETHPoolV3WithNativeChainBridge.sol` lines 294–298 (ETH path) and 322–326 (token path)
- `RSETHPoolNoWrapper.sol` lines 237–241 (ETH path) and 264–268 (token path)

The oracle rate (`rsETHToETHrate`) reflects the accrued EigenLayer restaking value and increases monotonically over time as staking rewards accumulate. Between a user's off-chain quote and on-chain execution, the rate can tick upward, causing `rsETHAmount` to be lower than simulated. For the token path, both `rsETHToETHrate` and `tokenToETHRate` can shift independently, compounding the divergence.

---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

The user receives fewer `wrsETH`/`rsETH` tokens than their off-chain simulation predicted. Because each token is worth proportionally more ETH at the updated rate, total ETH-denominated value is preserved; however, the user has no on-chain mechanism to enforce a minimum token count and cannot abort a transaction that would leave them with an unacceptably small position. This is directly analogous to the missing `minYT` guard in M-5.

---

### Likelihood Explanation

The rsETH oracle rate increases continuously as EigenLayer staking rewards accrue. Any deposit submitted during a period of rapid rate movement (e.g., immediately after a large reward distribution or rebase event) will silently produce fewer tokens than the user expected. No adversarial action is required; normal protocol operation is sufficient to trigger the discrepancy.

---

### Recommendation

Add a `minRsETH` parameter to all four `deposit` overloads across all pool contracts and revert if the computed amount falls below it:

```diff
- function deposit(string memory referralId) external payable ...
+ function deposit(string memory referralId, uint256 minRsETH) external payable ...
  {
      ...
      (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
+     if (rsETHAmount < minRsETH) revert InsufficientRsETHOut();
      wrsETH.mint(msg.sender, rsETHAmount);
  }
```

Apply the same change to the token-path overload and to all four pool contracts.

---

### Proof of Concept

1. User calls `viewSwapRsETHAmountAndFee(1 ether)` off-chain; oracle returns `1.05e18`, simulation shows `~0.952 wrsETH`.
2. Before the transaction lands, EigenLayer distributes rewards; oracle rate ticks to `1.10e18`.
3. `deposit{value: 1 ether}("ref")` executes; `rsETHAmount = 1e18 * 1e18 / 1.10e18 ≈ 0.909 wrsETH` — ~4.5 % fewer tokens than expected.
4. No revert occurs; user silently receives less than anticipated with no recourse.
