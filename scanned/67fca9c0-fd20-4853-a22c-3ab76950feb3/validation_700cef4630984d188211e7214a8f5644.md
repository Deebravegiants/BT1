### Title
Unbounded `referralId` calldata parameter enables block stuffing via minimal-ETH deposits — (`contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`)

### Summary
`deposit(string memory referralId)` imposes no length limit on the caller-controlled `referralId` string. An attacker can submit a single deposit of 1 wei with a multi-megabyte `referralId`, consuming a disproportionate share of block gas through calldata and event-log costs while minting a negligible amount of rsETH.

### Finding Description
In `RSETHPoolV3` and `RSETHPoolV3ExternalBridge`, the `deposit(string memory referralId)` function accepts an arbitrary-length string from any unprivileged caller. The string is ABI-decoded into memory and then passed verbatim to `emit SwapOccurred(...)`. No guard checks `bytes(referralId).length`. The EVM charges:

- **16 gas per non-zero calldata byte** (EIP-2028)
- **3 gas per 32-byte word** for memory expansion
- **8 gas per byte** of event log data

For a 1 MB `referralId` composed of non-zero bytes:
- Calldata cost: 1,000,000 × 16 = **16,000,000 gas**
- Log data cost: 1,000,000 × 8 = **8,000,000 gas**
- Execution overhead: ~50,000 gas
- **Total ≈ 24,050,000 gas** — roughly 80% of Ethereum's 30M gas block limit in a single transaction.

The attacker's ETH outlay for the deposit itself is 1 wei. The only real cost is the gas fee, which is the same cost any block-stuffer pays — but here the protocol provides a legitimate, always-open entry point that accepts arbitrarily large non-functional calldata with no friction.

### Impact Explanation
A single unprivileged transaction can consume the majority of a block's gas budget via the `referralId` parameter alone, leaving minimal room for other users' transactions. Sustained across multiple blocks, this constitutes block stuffing that delays or prevents other protocol interactions (deposits, withdrawals, bridging). The impact maps to **Medium — Unbounded gas consumption** and **Low — Block stuffing** in the allowed scope.

### Likelihood Explanation
The attack requires no special role, no oracle manipulation, and no governance action. Any address with enough ETH to pay gas fees can execute it. The entry point (`deposit`) is always open when `isEthDepositEnabled == true` and the contract is unpaused — the normal production state. Likelihood is **Medium**.

### Recommendation
Add a maximum length check on `referralId` at the top of both `deposit` overloads:

```solidity
uint256 private constant MAX_REFERRAL_ID_LENGTH = 128; // or a protocol-appropriate bound

function deposit(string memory referralId) external payable ... {
    if (bytes(referralId).length > MAX_REFERRAL_ID_LENGTH) revert ReferralIdTooLong();
    ...
}
```

Apply the same guard to the `deposit(address token, uint256 amount, string memory referralId)` overloads in both contracts and in `RSETHPoolV3WithNativeChainBridge`.

pragma solidity 0.8.27;

import "forge-std/Test.sol";
import "contracts/pools/RSETHPoolV3.sol";

contract ReferralIdGriefTest is Test {
    RSETHPoolV3 pool;

    function setUp() public {
        // deploy/initialize pool with WETHOracle as rsETHOracle,
        // isEthDepositEnabled = true, feeBps = 0, dailyMintLimit = type(uint256).max
        // ... (standard test setup)
    }

    function testBlockStuffingViaReferralId() public {
        // Build a 1 MB referralId of non-zero bytes
        bytes memory bigId = new bytes(1_000_000);
        for (uint256 i = 0; i < bigId.length; i++) bigId[i] = 0x41; // 'A'

        uint256 gasBefore = gasleft();
        pool.deposit{value: 1}(string(bigId));
        uint256 gasUsed = gasBefore - gasleft();

        // Expect ~24M gas consumed for 1 wei deposited
        assertGt(gasUsed, 20_000_000, "single deposit consumed >20M gas via referralId");
    }
}
