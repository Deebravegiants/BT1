### Title
Stale `totalETHDepositedToAave` Residual After Slashing + Reconfiguration Permanently Freezes Unclaimed Yield — (`contracts/LRTWithdrawalManager.sol`)

---

### Summary

When the active Aave pool suffers a loss (slashing/bad debt) such that `aaveAWETH.balanceOf(address(this)) < totalETHDepositedToAave`, and the manager subsequently calls `configureAaveIntegration()` to migrate to a new pool, the `totalETHDepositedToAave` accounting variable is left with a non-zero stale residual equal to the slashing delta. This residual permanently corrupts the interest-accounting invariant for the new pool, causing `collectInterestToTreasury()` to revert indefinitely and `getAccruedInterest()` to return 0 despite real interest existing.

---

### Finding Description

The root cause is in `_withdrawFromAave()`:

```solidity
// contracts/LRTWithdrawalManager.sol:912
uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave
    ? aaveBalance
    : totalETHDepositedToAave;

withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
// ...
totalETHDepositedToAave -= withdrawnAmount;  // line 918
```

When `aaveBalance < totalETHDepositedToAave` (post-slashing), `withdrawablePrincipal = aaveBalance`. The function withdraws `aaveBalance` and decrements `totalETHDepositedToAave` by `aaveBalance`, leaving:

```
totalETHDepositedToAave_after = totalETHDepositedToAave_before - aaveBalance
                               = slashingDelta  (> 0)
```

`configureAaveIntegration()` never resets `totalETHDepositedToAave` to zero before setting new pool addresses:

```solidity
// lines 438-458
_collectInterestToTreasury();          // returns 0 (aaveBalance <= principal)
aaveBalance = aaveAWETH.balanceOf(address(this));
if (aaveBalance > 0) {
    _withdrawFromAave(aaveBalance);    // leaves stale residual
}
// NO RESET of totalETHDepositedToAave
aavePool = aavePool_;
aaveWETHGateway = IWrappedTokenGatewayV3(aaveWETHGateway_);
aaveAWETH = IAToken(aaveAWETH_);
```

After reconfiguration, new deposits increment `totalETHDepositedToAave` on top of the stale residual:

```solidity
// _depositToAave, line 898
totalETHDepositedToAave += amount;
```

So `totalETHDepositedToAave = slashingDelta + newDeposits`.

**`_checkAaveHealth()` then permanently fails:**

```solidity
// lines 927-931
uint256 aaveBalance = aaveAWETH.balanceOf(address(this)); // newDeposits + interest
uint256 principal = totalETHDepositedToAave;              // slashingDelta + newDeposits
// principal - aaveBalance = slashingDelta - interest
// As long as interest < slashingDelta: principal > aaveBalance + 2 → returns false
```

**`collectInterestToTreasury()` reverts:**

```solidity
// lines 542-543
if (!_checkAaveHealth()) revert AaveHealthCheckFailed();
```

**`getAccruedInterest()` returns 0:**

```solidity
// lines 644-646
if (aaveBalance <= totalETHDepositedToAave) return 0;
// newDeposits + interest <= slashingDelta + newDeposits → true while interest < slashingDelta
```

There is no function in the contract that directly resets `totalETHDepositedToAave` to zero. The only path to decrement it is through `_withdrawFromAave`, which itself caps at `min(aaveBalance, totalETHDepositedToAave)`. Once the new pool is active, the stale residual is permanently baked into the accounting.

---

### Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

- `collectInterestToTreasury()` reverts with `AaveHealthCheckFailed` for all time until interest in the new pool exceeds `slashingDelta`.
- `getAccruedInterest()` returns 0 despite real interest existing in the new pool.
- The first `slashingDelta` worth of interest accrued in the new pool is permanently unclaimable to the treasury via the normal path — it is effectively absorbed into the stale principal accounting and never forwarded.
- The ETH is not lost from the contract (it remains available for user withdrawals), but the protocol's yield-collection mechanism is broken for the treasury.

---

### Likelihood Explanation

**Low-Medium.** The preconditions are:

1. An Aave pool used by the contract suffers a loss reducing `aaveAWETH.balanceOf(withdrawalManager)` below `totalETHDepositedToAave`. This is a known tail risk of Aave v3 (bad debt socialization, reserve factor shortfall).
2. The manager calls `configureAaveIntegration()` to migrate — a legitimate and expected operational response to a pool issue.

Neither step requires any compromise. The manager is acting correctly; the bug is in the contract's own accounting.

---

### Recommendation

In `configureAaveIntegration()`, after the full withdrawal from the old pool, explicitly reset `totalETHDepositedToAave` to zero before setting new pool addresses:

```solidity
// After _withdrawFromAave(aaveBalance):
totalETHDepositedToAave = 0;  // Reset: old pool is fully exited
```

Alternatively, in `_withdrawFromAave`, when the intent is a full exit (i.e., `amount >= aaveBalance`), set `totalETHDepositedToAave = 0` directly rather than subtracting, to avoid any residual from slashing:

```solidity
if (withdrawnAmount >= aaveBalance) {
    totalETHDepositedToAave = 0;
} else {
    totalETHDepositedToAave -= withdrawnAmount;
}
```

```text
    function _withdrawFromAave(uint256 amount) internal returns (uint256 withdrawnAmount) {
        if (amount == 0) return 0;

        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        if (aaveBalance == 0) revert InsufficientAaveBalance();

        // Only withdraw up to the principal amount (don't use accrued interest for user withdrawals)
        uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave;

        withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
        if (withdrawnAmount == 0) return 0;

        aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
        totalETHDepositedToAave -= withdrawnAmount;

        emit ETHWithdrawnFromAave(withdrawnAmount, totalETHDepositedToAave);
    }
```
