### Title
Chainlink Price Feed in `ChainlinkPriceOracle#getAssetPrice` Is Not Validated for Staleness, Enabling Incorrect rsETH Minting and Withdrawal Amounts - (File: contracts/oracles/ChainlinkPriceOracle.sol)

---

### Summary

`ChainlinkPriceOracle#getAssetPrice` calls `latestRoundData()` but discards all return values except `answer`. No `updatedAt` heartbeat check, no `answeredInRound < roundId` stale-round check, and no non-positive price guard are present. This stale price propagates directly into every user-facing deposit and withdrawal calculation in the protocol.

---

### Finding Description

In `ChainlinkPriceOracle.sol` line 52, the oracle reads only the raw `price` answer:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

Three validations are absent:

1. **No `updatedAt` staleness check** â€” the feed may have stopped updating (network congestion, Chainlink node downtime, sequencer outage on L2) while the last answer remains frozen on-chain.
2. **No `answeredInRound < roundId` check** â€” a round that was started but not yet answered returns the previous round's price.
3. **No non-positive price guard** â€” a zero or negative `answer` is cast directly to `uint256`, producing a wildly incorrect price.

`getAssetPrice` is consumed by every critical pricing path in the protocol:

- `LRTDepositPool.getRsETHAmountToMint` (line 520) â†’ called inside `_beforeDeposit` â†’ called by the public `depositAsset` and `depositETH` functions.
- `LRTWithdrawalManager.getExpectedAssetAmount` (line 593) â†’ called by the public `initiateWithdrawal` and `instantWithdrawal` functions.
- `LRTOracle._getTotalEthInProtocol` (line 339) â†’ called by the public `updateRSETHPrice` function, which recomputes the global `rsETHPrice` used by all of the above.

---

### Impact Explanation

**Scenario A â€” stale deflated price (LST temporarily depegged, then recovered, feed not yet updated):**
`getAssetPrice` returns a price lower than the true market value. `getRsETHAmountToMint` computes `(depositAmount Ă— staleLowPrice) / rsETHPrice`, minting *more* rsETH than the depositor's assets are worth. This dilutes every existing rsETH holder's share of the underlying pool â€” a direct theft of accrued yield from current holders.

**Scenario B â€” stale inflated price (LST slashed or depegged, feed not yet updated):**
`getAssetPrice` returns a price higher than the true market value. Depositors receive *fewer* rsETH tokens than their assets are worth, and `getExpectedAssetAmount` in `initiateWithdrawal` locks in a withdrawal amount that is too low, causing depositors to receive less than they are owed.

**Scenario C â€” zero/negative answer:**
`uint256(price)` wraps to a huge number or zero, causing either a division-by-zero revert that freezes deposits/withdrawals, or an astronomically incorrect mint amount.

Impact classification: **High â€” theft of unclaimed yield** (Scenario A) and **Medium â€” temporary freezing of funds** (Scenario C).

---

### Likelihood Explanation

Chainlink feeds have documented heartbeat intervals (e.g., 24 hours for ETH/stETH). During periods of high L1 gas prices, sequencer downtime on L2 deployments, or Chainlink node issues, feeds can remain at their last value well beyond the heartbeat. LST assets (stETH, ETHx, cbETH) supported by this protocol are precisely the assets most likely to experience short-term price deviations that a stale feed would miss. The entry path (`depositAsset`, `initiateWithdrawal`) is fully permissionless â€” any user can trigger the mispriced calculation.

---

### Recommendation

Capture and validate `updatedAt` and `answeredInRound` in `getAssetPrice`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (price <= 0) revert InvalidPrice();
    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt < block.timestamp - MAX_STALENESS_DELAY) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`MAX_STALENESS_DELAY` should be set per asset to slightly exceed the Chainlink heartbeat for that feed (e.g., 25 hours for a 24-hour heartbeat feed). Note that `ChainlinkOracleForRSETHPoolCollateral` already implements the `answeredInRound` and `timestamp == 0` checks as a reference pattern, but it also lacks the `updatedAt` heartbeat check.

### Proof of Concept

1. Chainlink's stETH/ETH feed goes stale at price `0.97e18` (a depeg event). The feed is not updated for 30 hours.
2. The actual stETH price recovers to `1.00e18` on-chain.
3. An attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
4. `getRsETHAmountToMint` calls `lrtOracle.getAssetPrice(stETH)` â†’ `ChainlinkPriceOracle.getAssetPrice` â†’ returns stale `0.97e18`.
5. rsETH minted = `(100e18 Ă— 0.97e18) / rsETHPrice`. The attacker receives rsETH priced as if they deposited 97 ETH worth of stETH, but they actually deposited 100 ETH worth.
6. The 3 ETH surplus is extracted from existing rsETH holders' share of the pool.
