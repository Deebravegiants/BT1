### Title
EigenLayer ERC-20 Reward Tokens Permanently Frozen in FeeReceiver — (File: contracts/NodeDelegator.sol)

### Summary
`NodeDelegator.processClaim` forwards EigenLayer reward claims to `lrtConfig.eigenLayerRewardReceiver()`, which resolves to the `FeeReceiver` contract. `FeeReceiver` is designed exclusively for native ETH (MEV/execution-layer rewards) and contains no mechanism to transfer out ERC-20 tokens. Because EigenLayer's `RewardsCoordinator.processClaim` distributes rewards as ERC-20 tokens, any such tokens sent to `FeeReceiver` are permanently irrecoverable.

### Finding Description
`NodeDelegator.processClaim` executes:

```solidity
IRewardsCoordinator(lrtConfig.rewardsCoordinator()).processClaim(claim, lrtConfig.eigenLayerRewardReceiver());
``` [1](#0-0) 

The second argument — `lrtConfig.eigenLayerRewardReceiver()` — is the recipient address for all ERC-20 reward tokens distributed by EigenLayer's `RewardsCoordinator`. This address resolves to `FeeReceiver`, which is explicitly documented as the "RewardReceiver Contract" in `LRTConstants.sol`. [2](#0-1) 

`FeeReceiver`'s entire interface for moving value out of the contract is:

```solidity
receive() external payable { }

function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
``` [3](#0-2) 

`sendFunds()` only reads `address(this).balance` (native ETH) and forwards it to the deposit pool. There is no `safeTransfer`, `transfer`, or any other ERC-20 egress path. There is also no `Recoverable` or sweep function in `FeeReceiver`.

EigenLayer's `RewardsCoordinator` distributes rewards as arbitrary ERC-20 tokens (e.g., EIGEN, stETH, or any token an AVS operator configures). When `processClaim` is called with `FeeReceiver` as the recipient, those ERC-20 tokens are transferred into `FeeReceiver` and have no path out.

### Impact Explanation
Every successful call to `NodeDelegator.processClaim` that includes ERC-20 reward tokens results in those tokens being permanently locked inside `FeeReceiver`. The tokens cannot be recovered by any existing function in the contract. This constitutes **permanent freezing of unclaimed yield** — EigenLayer restaking rewards that belong to the protocol and its depositors are irretrievably lost.

### Likelihood Explanation
`processClaim` is a routine operational function called by the LRT operator to harvest EigenLayer rewards. It is expected to be called regularly as part of normal protocol operations. EigenLayer's reward system is designed to distribute ERC-20 tokens. Every invocation that processes ERC-20 reward tokens silently freezes them. No attacker action is required; the loss occurs through normal, intended protocol operation.

### Recommendation
Either:
1. Add ERC-20 recovery functionality to `FeeReceiver` (a `recoverERC20(address token, uint256 amount)` function callable by the manager), and a forwarding path analogous to `sendFunds()` for ERC-20 tokens; or
2. Replace `eigenLayerRewardReceiver` with a dedicated contract that can handle both ETH and ERC-20 reward tokens and forward them appropriately to the deposit pool or treasury.

### Proof of Concept
1. EigenLayer operator configures an ERC-20 token (e.g., EIGEN) as a reward for restakers.
2. Protocol operator calls `NodeDelegator.processClaim(claim)`.
3. Internally: `RewardsCoordinator.processClaim(claim, address(FeeReceiver))` executes.
4. EigenLayer transfers ERC-20 reward tokens to `FeeReceiver`.
5. `FeeReceiver.sendFunds()` only moves `address(this).balance` (ETH = 0 for this reward); ERC-20 balance is untouched.
6. No function in `FeeReceiver` can move ERC-20 tokens out.
7. ERC-20 reward tokens are permanently frozen. [3](#0-2) [1](#0-0)
