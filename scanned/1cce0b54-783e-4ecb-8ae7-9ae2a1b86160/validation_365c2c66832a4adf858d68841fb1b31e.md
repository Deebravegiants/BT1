### Title
Withdrawal payout recalculated at unlock time using current price, shortchanging users when rsETH price drops — (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary
`LRTWithdrawalManager` records a promised `expectedAssetAmount` at withdrawal initiation time, but `_calculatePayoutAmount` silently overrides that promise at unlock time by taking the **minimum** of the initiation-time amount and the current-price-derived amount. When the rsETH price falls between initiation and unlock, users receive less than what was committed to them.

---

### Finding Description

**Step 1 — Initiation locks in a promise.**
`initiateWithdrawal` computes `expectedAssetAmount` from the live oracle price at the moment of the call and stores it in the `WithdrawalRequest` struct. It also increases `assetsCommitted[asset]` by that amount, signalling a protocol-level commitment to pay it.

```solidity
// LRTWithdrawalManager.sol:168-173
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
...
assetsCommitted[asset] += expectedAssetAmount;
_addUserWithdrawalRequest(asset, rsETHUnstaked, expectedAssetAmount);
```

`getExpectedAssetAmount` uses the live oracle at call time:

```solidity
// LRTWithdrawalManager.sol:593
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

**Step 2 — Unlock silently re-prices.**
When an operator calls `unlockQueue`, `_unlockWithdrawalRequests` calls `_calculatePayoutAmount` with the **current** oracle prices, not the prices from initiation:

```solidity
// LRTWithdrawalManager.sol:798
uint256 payoutAmount = _calculatePayoutAmount(request, rsETHPrice, assetPrice);
```

```solidity
// LRTWithdrawalManager.sol:833-834
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
```

The function returns `min(expectedAssetAmount, currentReturn)`. When the rsETH price has **dropped**, `currentReturn < expectedAssetAmount`, so the user's stored promise is overwritten downward:

```solidity
// LRTWithdrawalManager.sol:804
request.expectedAssetAmount = payoutAmount;  // silently reduced
```

**Step 3 — User receives the reduced amount.**
`completeWithdrawal` → `_processWithdrawalCompletion` transfers `request.expectedAssetAmount` to the user — the already-reduced value.

---

### Impact Explanation
**Low. Contract fails to deliver promised returns, but doesn't lose value.**

Users who initiate withdrawals receive a concrete on-chain commitment (`expectedAssetAmount` stored in `withdrawalRequests`, `assetsCommitted` increased). If the rsETH oracle price decreases before `unlockQueue` is called — due to a slashing event, EigenLayer losses, or a legitimate oracle update — the payout is silently cut to the lower current-price value. The user's rsETH has already been transferred to the contract and is locked; they cannot cancel and re-enter at the new price. The shortfall accrues to the protocol rather than being returned to the user.

---

### Likelihood Explanation
**Medium.** The withdrawal flow has a mandatory delay (`withdrawalDelayBlocks`), creating a guaranteed window between initiation and unlock during which the rsETH price can change. rsETH price decreases are realistic: EigenLayer slashing, NodeDelegator losses, or oracle recalibrations can all reduce `rsETHPrice`. The longer the delay and the larger the position, the larger the potential shortfall.

---

### Recommendation
Honor the `expectedAssetAmount` locked in at initiation time unconditionally. The initiation-time price already reflects the fair value at the moment the user committed their rsETH. If the protocol needs to protect against insolvency when prices drop, it should handle that at the liquidity/vault level rather than silently reducing individual user payouts. Concretely, remove the `min` logic in `_calculatePayoutAmount` and always pay `request.expectedAssetAmount`:

```solidity
function _calculatePayoutAmount(...) private view returns (uint256) {
    // Always honour the amount committed at initiation time.
    return request.expectedAssetAmount;
}
```

If downside protection is required, consider a separate mechanism (e.g., a protocol-funded buffer) rather than passing the loss directly to the withdrawing user.

---

### Proof of Concept

1. rsETH oracle price = 1.05 ETH per stETH. User calls `initiateWithdrawal(stETH, 1e18 rsETH)`.
   - `expectedAssetAmount` = `1e18 * 1.05e18 / 1e18` = **1.05e18 stETH** stored in the request.
   - `assetsCommitted[stETH]` += 1.05e18.
2. A slashing event reduces the rsETH oracle price to 1.00 ETH per stETH before `unlockQueue` is called.
3. Operator calls `unlockQueue(stETH, ...)`.
   - `currentReturn` = `1e18 * 1.00e18 / 1e18` = **1.00e18 stETH**.
   - `_calculatePayoutAmount` returns `min(1.05e18, 1.00e18)` = **1.00e18**.
   - `request.expectedAssetAmount` is overwritten to 1.00e18.
4. User calls `completeWithdrawal` and receives **1.00e18 stETH** — 0.05e18 stETH (≈4.8%) less than the amount committed at initiation.
