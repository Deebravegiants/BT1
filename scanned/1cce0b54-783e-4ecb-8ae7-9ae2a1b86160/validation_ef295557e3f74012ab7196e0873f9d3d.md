### Title
Stale `OneETHPriceOracle` Peg Exploitable via Front-Running Admin Oracle Switch — (`contracts/oracles/OneETHPriceOracle.sol`, `contracts/LRTOracle.sol`, `contracts/LRTDepositPool.sol`)

---

### Summary

`OneETHPriceOracle` is a hardcoded oracle that always returns `1e18` (1 ETH) as the price of any asset. When this oracle is assigned to a non-ETH LST asset and the admin later switches to a real market-rate oracle (because the asset has depegged), an attacker can front-run the oracle-switch transaction to deposit the depegged asset at the inflated pegged price, minting excess rsETH at the expense of all existing rsETH holders.

---

### Finding Description

`OneETHPriceOracle` unconditionally returns `1e18` for any asset: [1](#0-0) 

This oracle can be assigned to any supported asset via `LRTOracle.updatePriceOracleFor()` or `updatePriceOracleForValidated()`, both restricted to `onlyLRTAdmin`: [2](#0-1) [3](#0-2) 

When a user deposits an asset into `LRTDepositPool`, the rsETH minted is calculated as: [4](#0-3) 

This uses `lrtOracle.getAssetPrice(asset)`, which reads directly from whichever oracle is currently assigned: [5](#0-4) 

If `OneETHPriceOracle` is assigned to an LST (e.g., stETH) that has depegged to 0.90 ETH, the protocol still prices it at 1.00 ETH. When the admin submits a transaction to switch to a real Chainlink oracle, an attacker observing the mempool can front-run that transaction by depositing the depegged LST at the inflated pegged price, receiving rsETH worth more than the deposited collateral.

---

### Impact Explanation

**Critical — Direct theft of user funds (protocol insolvency).**

The attacker mints rsETH backed by collateral worth less than the rsETH's implied value. Every existing rsETH holder's share is diluted. For a 10% depeg on a 1,000 ETH deposit, the attacker extracts ~100 ETH of value from the protocol's existing TVL in a single transaction. The `rsETHPrice` used to compute minting is the stored (potentially stale) value, and the asset price is read live from the still-pegged `OneETHPriceOracle` — both inputs are wrong simultaneously during the transition window.

---

### Likelihood Explanation

**Medium.** The conditions required are:

1. `OneETHPriceOracle` is assigned to a non-ETH LST asset (a plausible gas-optimization choice, directly mirroring the SafeguardPool design).
2. That asset depegs materially below 1 ETH.
3. The admin submits an oracle-switch transaction visible in the public mempool.
4. An attacker front-runs it with a `depositAsset()` call.

Steps 3 and 4 are trivially achievable by any MEV bot on Ethereum mainnet. Steps 1 and 2 are the gating conditions, but the existence of `OneETHPriceOracle` in the production codebase as a valid `IPriceFetcher` makes step 1 a realistic deployment choice.

---

### Recommendation

1. **Remove or restrict `OneETHPriceOracle`** from being assignable to any asset other than native ETH, where the 1:1 peg is definitionally correct.
2. If the pegged oracle must remain for gas savings, implement a **timelock** on `updatePriceOracleFor` so that oracle switches cannot be front-run within the same block or within a delay window.
3. Alternatively, enforce that `updatePriceOracleFor` atomically pauses deposits for the affected asset until the next block, preventing any deposit from using the stale pegged price.

---

### Proof of Concept

**Setup:** Admin has assigned `OneETHPriceOracle` to `stETH`. stETH depegs to 0.90 ETH on the market. Admin submits `updatePriceOracleFor(stETH, chainlinkOracle)` to the mempool.

**Attack:**

1. Attacker observes the pending admin transaction.
2. Attacker submits `LRTDepositPool.depositAsset(stETH, 10_000e18, 0, "")` with higher gas, front-running the admin.
3. `getRsETHAmountToMint(stETH, 10_000e18)` executes:
   - `getAssetPrice(stETH)` → `OneETHPriceOracle.getAssetPrice()` → `1e18` (pegged, inflated)
   - `rsETHPrice` → current stored value, e.g., `1.05e18`
   - `rsethAmountToMint = (10_000e18 * 1e18) / 1.05e18 ≈ 9,523 rsETH`
4. The real value of 10,000 stETH at 0.90 ETH = 9,000 ETH → should mint `9,000 / 1.05 ≈ 8,571 rsETH`.
5. Attacker receives **~952 excess rsETH** (~952 ETH of value) extracted from existing holders.
6. Admin's oracle-switch lands in the next block; future deposits now use the correct price.
7. Attacker redeems or sells the excess rsETH. [6](#0-5) [1](#0-0) [3](#0-2)

        uint256 price = IPriceFetcher(priceOracle).getAssetPrice(asset);

        address asset,
        uint256 amount

        public
        view
        override
        returns (uint256 rsethAmountToMint)
    {
        // setup oracle contract
        address lrtOracleAddress = lrtConfig.getContract(LRTConstants.LRT_ORACLE);
        ILRTOracle lrtOracle = ILRTOracle(lrtOracleAddress);
