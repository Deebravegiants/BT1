### Title
No Cancellation Mechanism for Pending Withdrawal Requests Permanently Locks User rsETH - (File: contracts/LRTWithdrawalManager.sol)

### Summary

`LRTWithdrawalManager` provides no function for users to cancel a pending withdrawal request after calling `initiateWithdrawal()`. Once rsETH is transferred into the contract and a request is queued, the only exit path requires an operator to call `unlockQueue()` and then the user to call `completeWithdrawal()`. There is no `cancelWithdrawal()` or equivalent escape hatch. If the queue cannot be unlocked (e.g., the asset is removed from the supported list, blocking `unlockQueue()`), the user's rsETH is permanently frozen with no on-chain recovery path.

### Finding Description

`initiateWithdrawal()` pulls rsETH from the caller and enqueues a `WithdrawalRequest` keyed by a global per-asset nonce:

```
IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);
...
_addUserWithdrawalRequest(asset, rsETHUnstaked, expectedAssetAmount);
``` [1](#0-0) 

The only way to recover those tokens is through `completeWithdrawal()` / `completeWithdrawalForUser()`, both of which require the request to have been unlocked first by an operator calling `unlockQueue()`:

```
if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();
``` [2](#0-1) 

`unlockQueue()` carries the `onlySupportedAsset(asset)` modifier: [3](#0-2) 

If the asset is removed from the supported list (a routine governance action, not a malicious one), `unlockQueue()` reverts for that asset, `nextLockedNonce[asset]` never advances, and every pending request for that asset is permanently locked. No `cancelWithdrawal()` function exists anywhere in the contract or its interface: [4](#0-3) 

Additionally, `completeWithdrawal` is gated by `whenNotPaused`, so a prolonged pause also prevents users from completing already-unlocked requests, with no cancellation fallback. [5](#0-4) 

### Impact Explanation

**Permanent freezing of funds (Critical)** when an asset is deprecated while withdrawal requests are pending: `unlockQueue()` reverts on the `onlySupportedAsset` check, `nextLockedNonce` never advances, and the rsETH held by the contract for those requests can never be returned to users.

**Temporary freezing of funds (Medium)** in normal operation: users who change their mind (e.g., because the rsETH/asset exchange rate moved against them between `initiateWithdrawal` and `unlockQueue`) have no on-chain mechanism to recover their rsETH. They are forced to wait for operator action and accept whatever payout `_calculatePayoutAmount` computes at unlock time, which is capped at `min(expectedAssetAmount, currentReturn)` — meaning they absorb downside but cannot exit early. [6](#0-5) 

### Likelihood Explanation

Asset deprecation is a routine protocol lifecycle event (e.g., removing a poorly-performing LST strategy). Any user who has a pending withdrawal for a deprecated asset is immediately and permanently affected. The scenario requires no attacker — it is triggered by normal governance. Users initiating withdrawals during volatile market conditions and wishing to cancel face this limitation on every withdrawal.

### Recommendation

Add a `cancelWithdrawal(address asset)` function that:
1. Pops the caller's oldest pending (still-locked) nonce from `userAssociatedNonces[asset][msg.sender]`.
2. Deletes the corresponding `withdrawalRequests[requestId]`.
3. Decrements `assetsCommitted[asset]` by `request.expectedAssetAmount`.
4. Returns `request.rsETHUnstaked` rsETH to the caller.

This mirrors the `minimumValidNonce` / counter pattern recommended in M-14 and gives users a trustless escape hatch that does not depend on operator availability or asset support status.

### Proof of Concept

1. Alice calls `initiateWithdrawal(stETH, 10e18, "")`. Her 10 rsETH is transferred to `LRTWithdrawalManager`; a `WithdrawalRequest` is stored at nonce N; `assetsCommitted[stETH]` increases.
2. The protocol governance removes stETH from the supported asset list (e.g., strategy deprecated).
3. An operator attempts `unlockQueue(stETH, ...)` — it reverts because `onlySupportedAsset(stETH)` fails.
4. `nextLockedNonce[stETH]` remains at N; Alice's request is never unlocked.
5. Alice calls `completeWithdrawal(stETH, "")` — it reverts with `WithdrawalLocked`.
6. There is no `cancelWithdrawal` to call. Alice's 10 rsETH is permanently locked in the contract. [7](#0-6) [8](#0-7)
