### Title
Incorrect ETH Deposit Limit Check Omits Current Deposit Amount, Allowing Limit Bypass - (File: contracts/LRTDepositPool.sol)

### Summary
The `_checkIfDepositAmountExceedesCurrentLimit` function applies an asymmetric comparison for ETH versus ERC20 assets. For ETH it checks `totalAssetDeposits > depositLimit` (without adding the incoming `amount`), while for ERC20 it correctly checks `totalAssetDeposits + amount > depositLimit`. This mirrors the same class of incorrect arithmetic check described in the external report: a guard condition that silently passes when it should reject.

### Finding Description
In `LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit`, the ETH branch omits the current deposit amount from the comparison:

```solidity
// contracts/LRTDepositPool.sol lines 676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // ← amount not included
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct for ERC20
}
```

When `totalAssetDeposits == depositLimit`, the ETH check evaluates `depositLimit > depositLimit` → `false`, so `_beforeDeposit` does not revert and the deposit proceeds. After the call, `totalAssetDeposits` becomes `depositLimit + msg.value`, exceeding the configured cap. The ERC20 path correctly evaluates `depositLimit + amount > depositLimit` → `true` and reverts.

This is called unconditionally from `_beforeDeposit`, which is the sole guard for both `depositETH` (public, payable, no role restriction) and `depositAsset`. [1](#0-0)

### Impact Explanation
Any unprivileged depositor can push the protocol's ETH TVL past the admin-configured deposit cap by exactly one deposit amount. The deposit limit is the primary mechanism for bounding protocol exposure; bypassing it causes the contract to mint rsETH beyond the intended ceiling, violating the invariant that `getTotalAssetDeposits(ETH) ≤ depositLimitByAsset(ETH)`. No funds are directly stolen, but the protocol delivers more rsETH than the limit permits.

**Impact: Low** — Contract fails to deliver promised returns (deposit cap enforcement), but no direct value loss.

### Likelihood Explanation
The condition `totalAssetDeposits == depositLimit` is reachable in normal operation whenever the cap is approached. Any depositor monitoring on-chain state can time a call to `depositETH` at that exact moment. No special role, flash loan, or multi-step setup is required — a single `depositETH` call suffices.

### Recommendation
Apply the same `+ amount` inclusion to the ETH branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This unifies the check for both ETH and ERC20 and matches the semantics of `getAssetCurrentLimit`, which already returns `0` when `totalAssetDeposits == depositLimit`. [4](#0-3) 

### Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. Protocol accumulates `totalAssetDeposits(ETH) = 1000 ether` through normal deposits.
3. Attacker calls `depositETH{value: 10 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 10 ether)` evaluates `1000 ether > 1000 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for the attacker.
6. `totalAssetDeposits(ETH)` is now `1010 ether`, 1% above the configured cap.
7. The same step can be repeated on subsequent blocks as long as the limit is not raised, because `totalAssetDeposits > depositLimit` now returns `true` and blocks further deposits — but the damage (one over-limit deposit per cap-crossing event) has already occurred.
