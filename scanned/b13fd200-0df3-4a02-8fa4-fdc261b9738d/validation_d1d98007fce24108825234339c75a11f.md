### Title
Fee Truncates to Zero for Small Claim Amounts Due to Integer Division - (File: contracts/KERNEL/KernelMerkleDistributor.sol)

### Summary
In `KernelMerkleDistributor._processClaim()`, the protocol fee is computed as `(claimableAmount * feeInBPS) / FEE_DENOMINATOR`. Because Solidity integer division always truncates toward zero, any `claimableAmount` small enough that `claimableAmount * feeInBPS < FEE_DENOMINATOR` (10 000) produces `fee = 0`. The protocol treasury then receives nothing for that claim, while the claimer receives the full `claimableAmount` with no deduction.

### Finding Description
`KernelMerkleDistributor._processClaim()` computes the fee and the amount to transfer as follows:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / FEE_DENOMINATOR;   // line 338
uint256 amountToSend = claimableAmount - fee;                    // line 339

if (fee > 0) {
    kernel.safeTransfer(protocolTreasury, fee);                  // line 342
}
```

`FEE_DENOMINATOR = 10_000` and `feeInBPS` is bounded by `MAX_FEE_IN_BPS = 1_000` (10 %). [1](#0-0) [2](#0-1) 

The truncation threshold is `claimableAmount < 10_000 / feeInBPS`:

| `feeInBPS` | fee = 0 when `claimableAmount` < |
|---|---|
| 1 (0.01 %) | 10 000 wei of KERNEL |
| 10 (0.1 %) | 1 000 wei of KERNEL |
| 100 (1 %) | 100 wei of KERNEL |
| 1 000 (10 %) | 10 wei of KERNEL |

`claimableAmount` is the incremental amount since the user's last claim: `cumulativeAmount - userClaims[account].cumulativeAmount`. [3](#0-2) 

The same pattern is present in `MerkleDistributor.sol` and `KernelTop100MerkleDistributor.sol`. [4](#0-3) 

### Impact Explanation
When `fee` truncates to zero the `if (fee > 0)` guard prevents any transfer to `protocolTreasury`. The claimer receives the full `claimableAmount` without paying the configured protocol fee. The protocol fails to collect its promised fee revenue for those claims. No user funds are lost; only the protocol's fee yield is silently skipped.

**Impact: Low — Contract fails to deliver promised returns (fee revenue), but no user value is lost.**

### Likelihood Explanation
KERNEL is an 18-decimal ERC-20 token. The threshold amounts (10–10 000 wei) are sub-dust and would not arise from normal off-chain reward accounting. However, the merkle root is updated periodically; if a user's incremental reward between two consecutive roots is very small (e.g., a dust allocation), the fee silently becomes zero. The user cannot forge a smaller `claimableAmount` than what the merkle tree encodes, but they can time their claim to coincide with a root that carries a dust-sized increment. The likelihood is low but non-zero over a long-running distribution with many small-allocation users.

### Recommendation
1. **Round up the fee** using `Math.mulDiv(claimableAmount, feeInBPS, FEE_DENOMINATOR, Math.Rounding.Up)` so that any non-zero fee always rounds to at least 1 wei.
2. **Enforce a minimum claim amount** (e.g., `claimableAmount >= FEE_DENOMINATOR / feeInBPS`) so that claims too small to generate a fee are rejected or deferred.
3. Apply the same fix to `MerkleDistributor.sol` and `KernelTop100MerkleDistributor.sol` which share the identical pattern.

### Proof of Concept
Assume `feeInBPS = 100` (1 %) and a user's incremental reward between two merkle roots is `99 wei` of KERNEL.

```
claimableAmount = 99
fee = (99 * 100) / 10_000 = 9_900 / 10_000 = 0   ← truncated
amountToSend = 99 - 0 = 99
```

`if (fee > 0)` is false → `protocolTreasury` receives 0. The user claims 99 wei with no fee deduction, violating the 1 % fee invariant. [5](#0-4)
