### Title
`BridgedRsETHToL2` Event Logs Requested Amount Instead of Actual Debited Amount Due to OFT Shared-Decimal Dust Removal — (`contracts/L1Vault.sol`, `contracts/L1VaultV2.sol`)

### Summary
`L1Vault.bridgeRsETHToL2` and `L1VaultV2.bridgeRsETHToL2` discard the `OFTReceipt` returned by `oftAdapter.send()` and emit `BridgedRsETHToL2` with the caller-supplied `amount` rather than `oftReceipt.amountSentLD`. Because LayerZero OFT adapters perform shared-decimal dust removal (rounding `amountLD` down to the nearest `sharedFactor` multiple), `amountSentLD` can be strictly less than `amount`, causing the event to misreport the actual tokens debited.

### Finding Description
rsETH has 18 decimals. LayerZero OFT shared decimals are typically 6, giving a `sharedFactor` of `10^12`. Any `amount` that is not a multiple of `10^12` will have its dust silently truncated by the OFT adapter: the adapter debits `floor(amount / 10^12) * 10^12` tokens from the vault, not `amount`.

The return value of `oftAdapter.send()` carries the true debited amount in `oftReceipt.amountSentLD`, but both vault contracts ignore it entirely:

```solidity
// L1Vault.sol line 254-256
oftAdapter.send{ value: nativeFee }(sendParam, fee, msg.sender);  // return discarded
emit BridgedRsETHToL2(dstLzChainId, l2Receiver, amount, minAmount); // logs raw `amount`
```

The event should log `oftReceipt.amountSentLD` to reflect what was actually bridged.

### Impact Explanation
Off-chain indexers, dashboards, and accounting systems that rely on `BridgedRsETHToL2` to track how much rsETH was sent to L2 will record a value up to `10^12 - 1` wei higher than what was actually debited and credited on L2. Over many bridge calls this accumulates into a persistent discrepancy between L1 event-based accounting and actual L2 token supply. The dust itself remains stranded in the vault (not lost), but the promised event-based accounting invariant is broken. This matches the scoped impact: **Low — contract fails to deliver promised returns, but doesn't lose value**.

### Likelihood Explanation
The condition triggers on any bridge call where `amount % sharedFactor != 0`. The `MANAGER_ROLE` caller is the only actor, but the discrepancy occurs mechanically on any non-aligned amount without any special attacker action. It is a normal operational condition.

### Recommendation
Capture the `OFTReceipt` and emit `amountSentLD` in the event:

```solidity
(, OFTReceipt memory oftReceipt) = oftAdapter.send{ value: nativeFee }(sendParam, fee, msg.sender);
emit BridgedRsETHToL2(dstLzChainId, l2Receiver, oftReceipt.amountSentLD, minAmount);
```

Apply the same fix to `L1VaultV2.bridgeRsETHToL2`.

### Proof of Concept
1. Deploy a mock OFT adapter whose `send()` returns `amountSentLD = amount - (amount % 1e12)` (standard dust removal).
2. Call `bridgeRsETHToL2(amount = 1e18 + 1, ...)`.
3. Observe `BridgedRsETHToL2` event emits `amount = 1e18 + 1`.
4. Assert `oftReceipt.amountSentLD == 1e18` (dust `1` was removed).
5. The event amount `!= oftReceipt.amountSentLD`, confirming the discrepancy.
