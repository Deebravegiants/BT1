Looking at the core accounting functions in `LRTOracle` and `LRTDepositPool`, I can identify a direct analog to the reported vulnerability.

### Title
Hardcoded 18-Decimal Assumption in rsETH Price and Minting Calculations Breaks Protocol for Non-18-Decimal Supported Assets - (`contracts/LRTOracle.sol`, `contracts/LRTDepositPool.sol`)

### Summary
`LRTOracle._getTotalEthInProtocol()` and `LRTDepositPool.getRsETHAmountToMint()` silently assume all supported assets have 18 decimals. No decimal validation is enforced when assets are added via `LRTConfig`. If a non-18-decimal asset is ever added as a supported asset, the rsETH price and minting calculations will be severely miscalculated, leading to incorrect rsETH issuance and a broken protocol exchange rate.

### Finding Description

`LRTOracle._getTotalEthInProtocol()` accumulates the ETH value of all supported assets using a plain `mulWad` (i.e., `x * y / 1e18`):

```solidity
// totalAssetAmt is in 1e18 precision (standard token decimals)
uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
``` [1](#0-0) 

The comment itself documents the assumption: `totalAssetAmt` is treated as if it is always in 1e18 precision. `getTotalAssetDeposits` returns raw ERC20 balances, which are in the token's native decimals — not necessarily 18.

`LRTDepositPool.getRsETHAmountToMint()` performs the same implicit assumption:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [2](#0-1) 

`amount` is the raw deposited token amount in the token's native decimals. `getAssetPrice` returns a price in 1e18 precision. Dividing by `rsETHPrice` (also 1e18) does not normalize for the asset's decimals.

The same pattern appears in `LRTConverter.transferAssetFromDepositPool()`:

```solidity
ethValueInWithdrawal += (_amount * lrtOracle.getAssetPrice(_asset)) / 1e18;
``` [3](#0-2) 

And in `LRTDepositPool.getSwapAssetForETHReturnAmount()`:

```solidity
return lrtOracle.getAssetPrice(fromAsset) * fromAssetAmount / 1e18;
``` [4](#0-3) 

None of these functions query or normalize for the asset's actual decimals. There is no decimal validation anywhere in the asset-addition path.

### Impact Explanation

**Concrete example with a 6-decimal asset (e.g., a hypothetical 6-decimal LST):**

- User deposits 1000 tokens (= `1000e6` raw units)
- Asset price = 0.0003 ETH/token = `3e14` in 1e18 precision
- rsETH price = 1.05 ETH/rsETH = `1.05e18`
- **Actual calculation:** `(1000e6 * 3e14) / 1.05e18` ≈ `285714` (≈ `2.86e5`)
- **Correct result:** `1000 * 0.0003 / 1.05 * 1e18` ≈ `2.86e17`
- **Error factor:** `1e12` — the user receives `1e12` times fewer rsETH than they should

For `_getTotalEthInProtocol`, the TVL would be understated by the same factor, causing the rsETH price to be computed as far lower than reality, which would then allow subsequent depositors to mint rsETH at a massive discount — a direct path to protocol insolvency.

For an asset with more than 18 decimals, the inverse occurs: users receive far more rsETH than they should, draining the protocol.

**Impact:** Protocol insolvency / permanent fund loss for depositors of the affected asset.

### Likelihood Explanation

Low. All currently supported LSTs (stETH, cbETH, rETH, etc.) have 18 decimals. The vulnerability is triggered only if an admin adds a non-18-decimal asset to the supported asset list via `LRTConfig`. This is a misconfiguration risk — the contract provides no guard against it, and an admin may not realize the downstream accounting implications. The external report's analogous finding was rated Medium under the same conditional-on-non-standard-decimals reasoning.

### Recommendation

Add a decimal validation check in the asset-addition path in `LRTConfig` (or wherever `addNewSupportedAsset` is called) to enforce that only 18-decimal assets are accepted:

```solidity
require(IERC20Metadata(asset).decimals() == 18, "Asset must have 18 decimals");
```

Alternatively, normalize `totalAssetAmt` in `_getTotalEthInProtocol` and `amount` in `getRsETHAmountToMint` by scaling to 18 decimals before performing the WAD multiplication:

```solidity
uint8 assetDecimals = IERC20Metadata(asset).decimals();
uint256 normalizedAmt = totalAssetAmt * (10 ** (18 - assetDecimals));
totalETHInProtocol += normalizedAmt.mulWad(assetER);
```

### Proof of Concept

1. Admin calls `LRTConfig.addNewSupportedAsset(hypothetical6DecimalLST, depositLimit)` — no decimal check is performed.
2. A price oracle is configured for the asset returning a price in 1e18 precision.
3. User calls `LRTDepositPool.depositAsset(hypothetical6DecimalLST, 1000e6, 0, "")`.
4. `getRsETHAmountToMint` computes `(1000e6 * assetPrice) / rsETHPrice`, yielding a result `1e12` times smaller than the correct value.
5. User receives near-zero rsETH for their 1000-token deposit.
6. Simultaneously, `_getTotalEthInProtocol` understates TVL by `1e12`, causing `rsETHPrice` to be computed incorrectly, breaking the exchange rate for all subsequent depositors. [5](#0-4) [6](#0-5) [7](#0-6)
