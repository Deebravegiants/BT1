### Title
Stale `rsETHPrice` Used in Deposit and Withdrawal Calculations Without Calling `updateRSETHPrice()` First — (`contracts/LRTDepositPool.sol`, `contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTOracle.rsETHPrice` is a stored state variable that must be explicitly refreshed by calling `updateRSETHPrice()`. Neither `depositAsset()`/`depositETH()` in `LRTDepositPool` nor `initiateWithdrawal()`/`instantWithdrawal()` in `LRTWithdrawalManager` call `updateRSETHPrice()` before reading `rsETHPrice`. As a result, all rsETH minting and withdrawal amount calculations operate on a potentially stale price, allowing an attacker to deposit at an artificially low price and extract yield that belongs to existing rsETH holders.

---

### Finding Description

`LRTOracle` stores the rsETH/ETH exchange rate in a persistent state variable: [1](#0-0) 

This value is only updated when `updateRSETHPrice()` is explicitly called: [2](#0-1) 

The function is `public` — anyone can call it — but it is **not** called automatically before any user-facing operation.

When a user deposits, `_beforeDeposit()` calls `getRsETHAmountToMint()`, which divides by the stored (potentially stale) `rsETHPrice`: [3](#0-2) 

Neither `depositETH()` nor `depositAsset()` call `updateRSETHPrice()` before this calculation: [4](#0-3) [5](#0-4) 

Similarly, `initiateWithdrawal()` calls `getExpectedAssetAmount()` at line 168 without refreshing the price first: [6](#0-5) 

`getExpectedAssetAmount()` reads the same stale stored value: [7](#0-6) 

`instantWithdrawal()` has the same flaw at line 228: [8](#0-7) 

---

### Impact Explanation

**Impact: High — Theft of unclaimed yield.**

When EigenLayer rewards accrue (increasing protocol TVL) but `updateRSETHPrice()` has not yet been called, `rsETHPrice` is lower than the true current price. A depositor who deposits in this window receives more rsETH than they are entitled to. When the price is subsequently updated, the attacker's rsETH is worth more than what they paid, at the direct expense of existing holders whose proportional share of the TVL is diluted. This is a direct theft of accrued yield from existing rsETH holders.

The inverse applies to withdrawals: a user who calls `initiateWithdrawal()` when `rsETHPrice` is stale-low receives a lower `expectedAssetAmount` than they deserve, locking them into an unfavorable rate.

---

### Likelihood Explanation

**Likelihood: Medium.**

`updateRSETHPrice()` is a public function with no access control. An attacker can:
1. Observe that rewards have accrued (TVL has grown) but `rsETHPrice` has not been updated.
2. Call `depositAsset()` or `depositETH()` at the stale lower price, receiving excess rsETH.
3. Immediately call `updateRSETHPrice()` themselves to crystallize the gain.

No privileged access, no front-running of other users, and no external oracle manipulation is required. The attacker controls the entire sequence. The window exists between any two consecutive price updates, which is determined by keeper cadence and is not enforced on-chain.

---

### Recommendation

Call `updateRSETHPrice()` (or an equivalent internal price refresh) at the start of `depositETH()`, `depositAsset()`, `initiateWithdrawal()`, and `instantWithdrawal()` before any price-dependent calculation is performed. This mirrors the fix recommended in the reference report: invoke the state-updating function before the check that depends on it.

```solidity
// Example fix in depositAsset():
function depositAsset(...) external nonReentrant whenNotPaused onlySupportedERC20Token(asset) {
    ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice(); // add this
    uint256 rsethAmountToMint = _beforeDeposit(asset, depositAmount, minRSETHAmountExpected);
    ...
}
```

### Proof of Concept

**Setup:**
- Protocol TVL: 1 000 ETH, rsETH supply: 1 000, stored `rsETHPrice` = 1.000 ETH (last updated).
- EigenLayer rewards accrue: TVL grows to 1 010 ETH. True price = 1.010 ETH. `rsETHPrice` still = 1.000 ETH (stale).

**Attack:**
1. Attacker calls `depositETH{ value: 10 ether }(0, "")`.
2. `getRsETHAmountToMint` computes: `10 ETH * 1e18 / 1.000e18 = 10 rsETH`. (Fair amount at true price: `10 / 1.010 ≈ 9.901 rsETH`.)
3. Attacker receives **10 rsETH** instead of the fair **9.901 rsETH** — an excess of ~0.099 rsETH.
4. Attacker calls `updateRSETHPrice()`. New price = `(1010 + 10) / 1010 ≈ 1.0099 ETH`.
5. Attacker's 10 rsETH is now worth `10 × 1.0099 ≈ 10.099 ETH` — a profit of ~0.099 ETH extracted from existing holders' accrued yield.

The profit scales linearly with deposit size and with the magnitude of accrued-but-unrecorded rewards.
