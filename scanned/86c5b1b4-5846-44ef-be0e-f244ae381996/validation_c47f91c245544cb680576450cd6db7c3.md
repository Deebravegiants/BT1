### Title
Missing Minimum Output (Slippage) Protection in L2 Pool `deposit()` Functions - (File: contracts/pools/RSETHPoolV3.sol, RSETHPoolV3ExternalBridge.sol, RSETHPoolV3WithNativeChainBridge.sol, RSETHPool.sol, RSETHPoolNoWrapper.sol)

---

### Summary

All L2 pool `deposit()` functions lack a `minRSETHAmountExpected` parameter. The rsETH/wrsETH amount minted or transferred is computed at execution time using a live oracle rate. If the oracle rate increases between when a user previews the output via `viewSwapRsETHAmountAndFee()` and when their transaction is included in a block, the user receives fewer rsETH tokens than expected with no ability to revert. The L1 `LRTDepositPool` explicitly guards against this with a `minRSETHAmountExpected` check, but this protection is entirely absent from every L2 pool variant.

---

### Finding Description

Every L2 pool `deposit()` function computes the rsETH output at execution time:

```solidity
// RSETHPoolV3.sol – deposit(string)
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
// ...
wrsETH.mint(msg.sender, rsETHAmount);
```

`viewSwapRsETHAmountAndFee()` divides by the live oracle rate:

```solidity
uint256 rsETHToETHrate = getRate();          // live oracle read
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

Because `rsETHToETHrate` is the denominator, any increase in the rate between preview and execution directly reduces `rsETHAmount`. There is no parameter the caller can supply to bound this reduction.

By contrast, the L1 `LRTDepositPool` enforces a hard floor:

```solidity
// LRTDepositPool.sol – _beforeDeposit()
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

The same gap exists across all L2 pool variants: `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPool`, and `RSETHPoolNoWrapper`.

---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A user calls `viewSwapRsETHAmountAndFee(1 ether)` off-chain, sees they will receive `X` wrsETH, and submits `deposit{value: 1 ether}(referralId)`. If the oracle rate is updated in the same block (or a preceding block in the same mempool window), the user receives `Y < X` wrsETH with no recourse. The ETH is not lost — it is converted at the current rate — but the user receives fewer rsETH tokens than the protocol's own view function promised them. Over time, fewer rsETH tokens means permanently less accrued yield from EigenLayer restaking rewards.

---

### Likelihood Explanation

The rsETH/ETH oracle rate increases monotonically as EigenLayer rewards accrue and is updated by the operator on a regular schedule. Any deposit transaction that lands in the same block as, or after, an oracle rate update will silently receive fewer tokens than the user previewed. This is a routine operational event, not an exceptional one. Users relying on `viewSwapRsETHAmountAndFee()` to size their deposits have no on-chain mechanism to protect themselves.

---

### Recommendation

Add a `minRSETHAmountExpected` parameter to every L2 pool `deposit()` overload and revert if the computed amount falls below it, mirroring the existing L1 guard:

```solidity
function deposit(string memory referralId, uint256 minRSETHAmountExpected)
    external payable nonReentrant whenNotPaused ...
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    wrsETH.mint(msg.sender, rsETHAmount);
}
```

Apply the same change to the token-deposit overload and to all affected pool contracts.

---

### Proof of Concept

1. User calls `viewSwapRsETHAmountAndFee(1 ether)` → sees `rsETHAmount = 950` (rate = ~1.052 ETH/rsETH).
2. User submits `deposit{value: 1 ether}("ref")`.
3. Before the transaction is mined, the operator calls `updateRate()` on the oracle, raising the rate to ~1.100 ETH/rsETH.
4. User's transaction executes: `rsETHAmount = 1e18 * 1e18 / 1.1e18 ≈ 909` — roughly 4% fewer tokens than previewed.
5. No revert occurs; the user silently receives 909 wrsETH instead of the 950 they expected, with no ability to undo the deposit.

The L1 equivalent (`LRTDepositPool.depositETH`) would have reverted at step 4 with `MinimumAmountToReceiveNotMet` if the user had passed `minRSETHAmountExpected = 950`.

    function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
