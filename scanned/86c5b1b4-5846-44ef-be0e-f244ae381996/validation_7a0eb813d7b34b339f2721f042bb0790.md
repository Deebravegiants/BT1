### Title
Stale `rsETHPrice` Enables Over-Minting of rsETH When Asset Oracle Prices Increase — (File: contracts/LRTDepositPool.sol)

---

### Summary
`LRTDepositPool.getRsETHAmountToMint()` computes the rsETH mint amount using a live asset price oracle (`lrtOracle.getAssetPrice(asset)`) divided by a stored, separately-updated `lrtOracle.rsETHPrice()`. When the stored `rsETHPrice` is stale relative to the live asset price — a window that is structurally extended by the `pricePercentageLimit` guard in `LRTOracle._updateRsETHPrice()` — any depositor can mint more rsETH than the fair value of their deposit, diluting existing rsETH holders.

---

### Finding Description

`LRTDepositPool.getRsETHAmountToMint()` computes:

```
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()
``` [1](#0-0) 

`lrtOracle.getAssetPrice(asset)` is a live read from an external price oracle (e.g., Chainlink stETH/ETH), always returning the current market rate. [2](#0-1) 

`lrtOracle.rsETHPrice()` is a **stored state variable** that is only updated when `updateRSETHPrice()` or `updateRSETHPriceAsManager()` is explicitly called. [3](#0-2) 

`_updateRsETHPrice()` contains a guard: when the newly computed rsETH price exceeds `highestRsethPrice` by more than `pricePercentageLimit`, the call **reverts** for any non-manager caller with `PriceAboveDailyThreshold`. [4](#0-3) 

This creates a persistent staleness window:

1. An LST asset price oracle (e.g., stETH/ETH) increases by more than `pricePercentageLimit` (e.g., >1%).
2. Any call to `updateRSETHPrice()` by a non-manager reverts.
3. `rsETHPrice` remains at the old, lower value.
4. `getAssetPrice(asset)` returns the new, higher value.
5. `getRsETHAmountToMint()` returns `amount * newHigherPrice / oldLowerRsETHPrice`, which is greater than the fair mint amount.
6. The depositor receives more rsETH than the fair value of their deposit.

The window persists until a manager calls `updateRSETHPriceAsManager()`. [5](#0-4) 

---

### Impact Explanation

Every rsETH minted above fair value dilutes the share of the protocol's TVL held by existing rsETH holders. The attacker deposits an LST at the inflated `assetPrice / staleRsETHPrice` ratio, receiving excess rsETH. When `rsETHPrice` is eventually corrected upward, the attacker's excess rsETH represents a claim on ETH that was not contributed — extracted from the yield and principal of existing holders.

**Impact: High — Theft of unclaimed yield (dilution of existing rsETH holders' accrued value).**

---

### Likelihood Explanation

- `pricePercentageLimit` is a configurable admin parameter. When set to any non-zero value (e.g., 1% = `1e16`), a natural LST price appreciation event of that magnitude triggers the blocking condition.
- LST/ETH rates (stETH, cbETH, rETH) do experience step-changes of 0.5–1%+ around reward distribution events.
- `updateRSETHPrice()` is public, so the staleness window is normally short; however, the `pricePercentageLimit` guard structurally prevents non-managers from closing it during exactly the events where the mispricing is largest.
- No privileged access, front-running, or oracle manipulation is required. Any depositor can exploit the window by calling `depositAsset()` while `rsETHPrice` is stale. [6](#0-5) 

---

### Recommendation

Before computing `rsethAmountToMint`, atomically refresh `rsETHPrice` within `_beforeDeposit()` (or `getRsETHAmountToMint()`), or derive the mint amount directly from the live TVL and supply without relying on the stored `rsETHPrice` state variable. Alternatively, enforce that `rsETHPrice` was updated within the current block (or within a short staleness bound) before any deposit is accepted.

---

### Proof of Concept

**Setup:**
- `pricePercentageLimit` = `1e16` (1%)
- `rsETHPrice` = `1.04e18` (last stored value)
- `highestRsethPrice` = `1.04e18`
- stETH/ETH oracle price increases to `1.055e18` (a ~1.4% increase, above the limit)

**Step 1:** Attacker (or anyone) calls `updateRSETHPrice()`.
- `newRsETHPrice` ≈ `1.055e18` (computed from live TVL)
- `priceDifference = 1.055e18 - 1.04e18 = 0.015e18`
- `pricePercentageLimit.mulWad(highestRsethPrice) = 0.01 * 1.04e18 = 0.0104e18`
- `0.015e18 > 0.0104e18` → `isPriceIncreaseOffLimit = true`
- Caller is not a manager → **reverts with `PriceAboveDailyThreshold`**
- `rsETHPrice` remains `1.04e18` [7](#0-6) 

**Step 2:** Attacker calls `depositAsset(stETH, 100e18, 0, "")`.
- `getRsETHAmountToMint(stETH, 100e18)` = `100e18 * 1.055e18 / 1.04e18` = **`101.44e18` rsETH**
- Fair mint amount (at correct price `1.055e18`) = `100e18 * 1.055e18 / 1.055e18` = `100e18` rsETH
- **Excess rsETH minted: `1.44e18`** [8](#0-7) 

**Step 3:** Manager eventually calls `updateRSETHPriceAsManager()`, correcting `rsETHPrice` to `1.055e18`. The attacker's `101.44e18` rsETH is now worth `101.44 * 1.055 = 107.02 ETH`, while they deposited `100 stETH` worth `105.5 ETH`. Net gain: **~1.52 ETH** extracted from existing rsETH holders per 100 stETH deposited. [5](#0-4)
