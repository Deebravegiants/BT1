### Title
Division-by-Zero in `viewSwapAgETHAmountAndFee` Blocks All Deposits Before First Cross-Chain Rate Message - (File: contracts/agETH/AGETHRateReceiver.sol)

### Summary
`CrossChainRateReceiver.rate` initializes to `0` and is only updated via `lzReceive`. Before the first LayerZero message arrives, `AGETHPoolV3.deposit()` (both ETH and token variants) will always revert with a Solidity 0.8.x division-by-zero panic, temporarily freezing all pool deposits.

### Finding Description
`CrossChainRateReceiver` declares `rate` as a plain `uint256` state variable with no initial value, so it defaults to `0`. [1](#0-0) 

`getRate()` returns this value unconditionally, with no zero-guard: [2](#0-1) 

`rate` is only ever written inside `lzReceive`, which requires a valid LayerZero message from the source chain: [3](#0-2) 

In `AGETHPoolV3`, both `deposit` overloads call `viewSwapAgETHAmountAndFee`, which performs:

```solidity
agETHAmount = amountAfterFee * 1e18 / agETHToETHrate;
``` [4](#0-3) 

and for token deposits:

```solidity
agETHAmount = amountAfterFee * tokenToETHRate / agETHToETHrate;
``` [5](#0-4) 

When `agETHToETHrate == 0`, Solidity 0.8.x raises `Panic(0x12)` (division by zero), reverting the transaction.

Notably, `addSupportedToken` already guards against a zero rate for token oracles: [6](#0-5) 

...but no equivalent guard exists for `agETHOracle` in `initialize()` or in `deposit()` itself. [7](#0-6) 

### Impact Explanation
All ETH and token deposits into `AGETHPoolV3` are completely blocked from the moment of deployment until the first successful `lzReceive` call populates `rate`. This is a **temporary freezing of funds** — users cannot deposit, and any ETH sent via `deposit` reverts (ETH is returned by the EVM on revert, so no permanent loss, but the service is unavailable).

### Likelihood Explanation
The window is inherent to the deployment lifecycle: the contract is deployed on a destination chain, and the first cross-chain rate message must travel via LayerZero from the source chain. This gap can range from seconds to hours depending on bridge latency, operator scheduling, or network congestion. Any user who attempts a deposit during this window will be affected. No special attacker action is required — the condition is triggered by normal user behavior.

### Recommendation
Add a zero-rate guard in `viewSwapAgETHAmountAndFee` (or in `getRate()` on the pool side), reverting with a descriptive error rather than a panic:

```solidity
uint256 agETHToETHrate = getRate();
if (agETHToETHrate == 0) revert OracleRateNotInitialized();
```

Alternatively, initialize `rate` to a safe sentinel value (e.g., `1e18`) in the `AGETHRateReceiver` constructor, or require the deployer to call `lzReceive` (or a privileged setter) with a valid rate before enabling deposits.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// 1. Deploy AGETHRateReceiver(srcChainId, rateProvider, lzEndpoint)
//    → rate == 0 (default)

// 2. Deploy AGETHPoolV3, initialize with agETHOracle = address(AGETHRateReceiver)

// 3. Call AGETHPoolV3.deposit{value: 1 ether}("")
//    → internally calls viewSwapAgETHAmountAndFee(1 ether)
//    → agETHToETHrate = IOracle(agETHOracle).getRate() == 0
//    → agETHAmount = amountAfterFee * 1e18 / 0  ← Panic(0x12), revert

// 4. Simulate lzReceive: call lzReceive from lzEndpoint with rate = 1e18
//    → rate is now 1e18

// 5. Call AGETHPoolV3.deposit{value: 1 ether}("") again
//    → succeeds, agETHAmount = 1 ether (minus fee)
```

The revert in step 3 and success in step 5 are fully reproducible on a local Hardhat/Foundry fork without any admin compromise or external dependency — only the uninitialized `rate` state is required.
