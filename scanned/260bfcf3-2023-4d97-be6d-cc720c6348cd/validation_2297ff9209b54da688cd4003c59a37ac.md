### Title
Pusher Consent Signature Replay Allows Creator to Re-establish Revoked Delegation — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol::allowPushers`)

---

### Summary

`allowPushers` in `CompressedOracle.sol` verifies a pusher's EIP-191 consent signature but never invalidates it after first use. A creator who holds a valid, non-expired pusher signature can replay it an unlimited number of times within the deadline window. This means a pusher who calls `revokePusher()` to withdraw consent cannot actually stop the creator from re-establishing the delegation until the original deadline expires — breaking the revocation invariant and allowing the creator to continue attributing the pusher's feed writes to their own namespace against the pusher's will.

---

### Finding Description

`allowPushers` verifies the pusher's signed consent and writes `namespaceRemapping[pusher] = msg.sender`:

```solidity
// smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol
function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
    _ensureDeadline(deadline);
    ...
    bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
    );
    require(pusher == ECDSA.recover(hash, signatures[i]));
    namespaceRemapping[pusher] = msg.sender;   // ← written every call
    emit PusherAuthorized(pusher, msg.sender);
}
```

The signed tuple is `(chainid, oracle, deadline, pusher, creator)`. There is no nonce, no used-signature registry, and no per-pusher revocation counter. The same `(deadline, sig)` pair passes `ECDSA.recover` identically on every replay.

`revokePusher` clears the mapping:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
```

But the creator immediately re-writes it by calling `allowPushers` again with the identical signature. The code's own comment acknowledges the risk:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."*

The deadline is treated as the sole mitigation, but it only bounds the replay window — it does not prevent replay within that window. Deadlines are caller-chosen and can be set months or years in the future.

The `fallback` push path resolves the namespace at call time:

```solidity
fallback() override external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0)) creator = msg.sender;
    ...
}
```

So every push the pusher makes after the creator's replay lands in the creator's namespace, not the pusher's own, exactly as before revocation.

---

### Impact Explanation

**Oracle data integrity / bad-price execution path:**

1. A pusher is an automated price-feed bot that continuously pushes data into a creator's namespace via the `fallback` path.
2. The creator's namespace is consumed by a `IPriceProvider` that drives `_getBidAndAskPriceX64()` in `MetricOmmPool.swap`.
3. The pusher discovers the creator is acting maliciously (e.g., selectively withholding or delaying updates to manipulate the bid/ask spread) and calls `revokePusher()`.
4. The creator immediately replays the original consent signature via `allowPushers`. The pusher's bot, still running, continues to push into the creator's namespace.
5. The creator now controls which of the pusher's updates to accept (by controlling the namespace) while the pusher believes they have revoked consent.
6. Stale or selectively-applied oracle data reaches live pool swaps, causing bad-price execution for traders.

The `IncorrectDelta` guard in `swap` only checks token settlement, not oracle freshness. A manipulated but technically valid oracle quote passes all on-chain checks.

---

### Likelihood Explanation

- **Medium.** The creator must have retained the original consent signature (trivially true — they received it when they called `allowPushers` the first time).
- The deadline is creator-chosen; production deployments commonly use multi-month or multi-year deadlines.
- The pusher's bot may continue running after `revokePusher()` if the operator does not immediately halt it (a realistic operational gap).
- No privileged access is required; the replay is a standard public call.

---

### Recommendation

Add a per-pusher revocation nonce to the signed message and increment it on every successful `revokePusher` or `removePushers` call:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender,
        pusherNonce[pusher]   // ← bind to current nonce
    ))
);

// In revokePusher / removePushers:
pusherNonce[pusher]++;        // ← invalidate all prior signatures
namespaceRemapping[pusher] = address(0);
```

This mirrors the Seaport pattern cited in the external report: the pusher can say "I consent to this delegation, but only until I revoke it" — and revocation immediately invalidates all outstanding signatures.

### Proof of Concept

```
1. Pusher P signs:
   sig = sign(keccak256(abi.encode(chainid, oracle, deadline=now+365days, P, C)))

2. Creator C calls allowPushers(deadline, [P], [sig])
   → namespaceRemapping[P] = C  ✓

3. Pusher P calls revokePusher()
   → namespaceRemapping[P] = address(0)  ✓ (P believes they are free)

4. Creator C calls allowPushers(deadline, [P], [sig])  ← SAME sig, deadline still valid
   → ECDSA.recover produces P  ✓ (no nonce, no used-sig check)
   → namespaceRemapping[P] = C  ← re-established without P's knowledge

5. P's bot continues pushing → all writes land in C's namespace
   → C's pool price provider reads C's namespace → manipulated quotes reach swap()
```
