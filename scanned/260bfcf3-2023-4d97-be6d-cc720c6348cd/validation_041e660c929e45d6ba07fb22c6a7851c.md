### Title
Single Broken Extension Permanently DOSes `removeLiquidity`, Locking LP Funds — (`metric-core/contracts/ExtensionCalling.sol`, `metric-core/contracts/libraries/CallExtension.sol`)

---

### Summary

`CallExtension.callExtension` propagates every revert from an extension contract directly to the caller with no error isolation. Because extensions are stored as immutables and cannot be replaced after pool deployment, a single extension that begins reverting on `beforeRemoveLiquidity` or `afterRemoveLiquidity` permanently prevents all LPs from withdrawing their funds. There is no escape hatch: `removeLiquidity` carries no `whenNotPaused` guard, so even a full protocol pause cannot unblock withdrawals.

---

### Finding Description

`ExtensionCalling._callExtensionsInOrder` iterates over every extension index packed into an order word and calls each one through `CallExtension.callExtension`:

```solidity
// ExtensionCalling.sol L75-86
function _callExtensionsInOrder(uint256 order, bytes memory data) private {
    if (order == 0) return;
    while (true) {
        uint256 extensionIndex = order & 0x7;
        if (extensionIndex == 0) break;
        address extension = _extensionAddress(extensionIndex);
        if (extension == address(0)) revert PanicEmptyExtension();
        CallExtension.callExtension(extension, data);   // <-- no try/catch
        order >>= 3;
    }
}
```

`CallExtension.callExtension` re-raises any revert from the extension verbatim:

```solidity
// CallExtension.sol L8-17
function callExtension(address extension, bytes memory data) internal {
    (bool success, bytes memory result) = extension.call(data);
    if (!success) {
        if (result.length > 0) {
            assembly ("memory-safe") { revert(add(result, 32), mload(result)) }
        }
        revert ExtensionCallFailed();
    }
    ...
}
```

`removeLiquidity` calls both `_beforeRemoveLiquidity` and `_afterRemoveLiquidity` through this path:

```solidity
// MetricOmmPool.sol L207-211
_beforeRemoveLiquidity(msg.sender, owner, salt, deltas, extensionData);
(amount0Removed, amount1Removed) = LiquidityLib.removeLiquidity(...);
_afterRemoveLiquidity(msg.sender, owner, salt, deltas, amount0Removed, amount1Removed, extensionData);
```

Extensions are stored as immutables (`EXTENSION_1` … `EXTENSION_7`) set once in the constructor and never changeable. If any extension registered in `BEFORE_REMOVE_LIQUIDITY_ORDER` or `AFTER_REMOVE_LIQUIDITY_ORDER` begins reverting — due to an external dependency failure, a proxy upgrade, or a latent bug triggered by new conditions — every subsequent call to `removeLiquidity` reverts. LP shares and the underlying token balances tracked in `_binStates` / `binTotals` remain permanently locked.

The `pauseLevel` mechanism provides no relief: `removeLiquidity` has no `whenNotPaused` modifier, so pausing the pool at level 1 or 2 does not bypass the extension calls.

---

### Impact Explanation

LP principal is permanently inaccessible. `binTotals.scaledToken0` and `binTotals.scaledToken1` continue to account for the locked tokens, but no withdrawal path exists. The pool's real ERC-20 balances exceed what can ever be claimed, constituting pool insolvency from the LP perspective. `swap` is similarly DOSed if `BEFORE_SWAP_ORDER` or `AFTER_SWAP_ORDER` extensions break, but the more severe consequence is the withdrawal freeze.

---

### Likelihood Explanation

Extensions are external contracts. Realistic failure modes include:

- An extension that consults an external oracle or allowlist contract — if that dependency is paused, upgraded, or decommissioned, the extension reverts on every call.
- An extension implemented as an upgradeable proxy — a botched upgrade introduces a revert path.
- A latent bug in the extension triggered only after a certain pool state is reached (e.g., a specific `curBinIdx` value or cumulative volume threshold).

None of these require the pool admin to act maliciously; they are ordinary operational risks for any contract with external dependencies. Because extensions are immutable, there is zero recovery path once the failure begins.

---

### Recommendation

1. **Wrap extension calls in try/catch** and emit a failure event rather than propagating the revert, at minimum for `beforeRemoveLiquidity` / `afterRemoveLiquidity` hooks, so LPs retain a withdrawal path even when an extension is broken.
2. **Add a `whenNotPaused`-style bypass for `removeLiquidity`** that skips extension calls when the pool is in a protocol-pause state (level 2), giving the protocol owner an emergency escape hatch.
3. **Allow extension replacement via a timelocked factory action**, analogous to the existing `proposePoolPriceProvider` / `executePoolPriceProviderUpdate` pattern, so a broken extension can be swapped out.

---

### Proof of Concept

1. Deploy a pool with an extension `E` registered in `BEFORE_REMOVE_LIQUIDITY_ORDER`.
2. LP Alice calls `addLiquidity`; tokens are deposited and `_binStates` / `binTotals` are updated.
3. Extension `E` begins reverting (e.g., its external oracle dependency is paused).
4. Alice calls `removeLiquidity`. Execution reaches `_beforeRemoveLiquidity` → `_callExtensionsInOrder` → `CallExtension.callExtension(E, ...)` → `E` reverts → `callExtension` re-raises the revert → `removeLiquidity` reverts.
5. Alice's shares remain in `_positionBinShares`; her tokens remain in `binTotals`. No alternative withdrawal function exists. Pausing the pool (level 1 or 2) does not help because `removeLiquidity` has no `whenNotPaused` guard.
