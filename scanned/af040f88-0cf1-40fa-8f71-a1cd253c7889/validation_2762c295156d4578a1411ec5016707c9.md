### Title
Missing Chainlink Price Validation Allows Stale or Invalid Prices to Be Accepted - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `answer`, performing no validation on staleness, round completeness, or price sign. This is the direct analog of the reported Pyth oracle missing-validation pattern, applied to the Chainlink integration that feeds asset prices into rsETH minting and withdrawal payout calculations.

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the Chainlink price as follows:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

The function silently discards `roundId`, `startedAt`, `updatedAt`, and `answeredInRound`. No checks are performed for:

1. **Negative or zero price** — `int256 answer` can be `<= 0` (e.g., during Chainlink circuit-breaker events where the aggregator clamps to `minAnswer`). Casting a negative `int256` to `uint256` produces an astronomically large value.
2. **Stale round** — no check that `answeredInRound >= roundId`.
3. **Incomplete round** — no check that `updatedAt != 0`.
4. **Heartbeat freshness** — no check that `block.timestamp - updatedAt` is within the feed's heartbeat window.

By contrast, the project's own `ChainlinkOracleForRSETHPoolCollateral.getRate()` correctly validates all three conditions:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

### Impact Explanation

`ChainlinkPriceOracle.getAssetPrice()` is consumed by `LRTOracle.getAssetPrice()`: [3](#0-2) 

Which is used in `LRTDepositPool.getRsETHAmountToMint()`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [4](#0-3) 

**Scenario A — Negative price (circuit-breaker clamp):** If Chainlink's aggregator returns a negative `answer` (possible when the underlying feed hits its configured `minAnswer` floor during extreme market events), `uint256(negative_int256)` wraps to a value near `2^256`. The inflated asset price causes `getRsETHAmountToMint` to return an enormous rsETH amount. A depositor calling `depositAsset` at that moment receives vastly more rsETH than the deposited value warrants, directly draining the protocol's backing — **protocol insolvency (Critical)**.

**Scenario B — Stale/zero price:** If the feed returns `0` (e.g., a deprecated or paused feed), `uint256(0)` propagates into the minting formula, causing a division-by-zero revert in `getRsETHAmountToMint` (denominator `lrtOracle.rsETHPrice()` is unaffected, but the numerator being 0 is fine; however if `assetPrice` is used as a divisor in `_calculatePayoutAmount` in the withdrawal manager, it causes revert) — **temporary freezing of funds (Medium)**. [5](#0-4) 

### Likelihood Explanation

Chainlink feeds have documented circuit-breaker behavior where `latestRoundData` returns the `minAnswer` or `maxAnswer` boundary value rather than the true market price during extreme volatility (e.g., LUNA collapse). Staleness occurs during Ethereum network congestion when keepers cannot update feeds within the heartbeat window. Both conditions are realistic on mainnet and require no attacker action — a depositor simply calls `depositAsset` at the wrong moment and the unvalidated price is consumed.

### Recommendation

Apply the same validation pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    require(answeredInRound >= roundId, "Stale price");
    require(updatedAt != 0, "Incomplete round");
    require(price > 0, "Invalid price");
    // Optionally: require(block.timestamp - updatedAt <= HEARTBEAT, "Price too old");

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

### Proof of Concept

1. Chainlink's ETH/stETH or similar feed hits its `minAnswer` circuit-breaker floor and returns `answer = -1` (or any negative value).
2. Any user calls `LRTDepositPool.depositAsset(stETH, amount)`.
3. `getRsETHAmountToMint` calls `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)`.
4. `uint256(int256(-1))` = `2^256 - 1`. The returned "price" is `(2^256 - 1) * 1e18 / 1e18` ≈ `2^256 - 1`.
5. `rsethAmountToMint = (amount * (2^256 - 1)) / rsETHPrice` overflows or returns a massive value (Solidity 0.8 reverts on overflow, but if `amount` is small enough to avoid overflow, the result is still enormous).
6. The depositor receives orders of magnitude more rsETH than their deposit is worth, draining the protocol's collateral backing. [6](#0-5) [7](#0-6)
