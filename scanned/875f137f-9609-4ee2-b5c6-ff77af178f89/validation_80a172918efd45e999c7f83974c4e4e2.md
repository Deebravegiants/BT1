### Title
`instantWithdrawal` Drains `LRTUnstakingVault` Without Respecting `assetsCommitted`, Freezing Queued Withdrawals - (File: contracts/LRTWithdrawalManager.sol)

---

### Summary

`LRTWithdrawalManager` maintains two separate, unsynchronized accounting mechanisms for protecting queued-withdrawal liquidity: `assetsCommitted[asset]` (in `LRTWithdrawalManager`) and `queuedWithdrawalsBuffer[asset]` (in `LRTUnstakingVault`). The `instantWithdrawal` path checks only `queuedWithdrawalsBuffer` — which defaults to zero — and never reads or updates `assetsCommitted`. As a result, instant withdrawers can drain the entire `LRTUnstakingVault` balance, including assets that were already promised to users who initiated queued withdrawals, leaving those users' rsETH locked in the withdrawal manager with no assets available to fulfill their requests.

---

### Finding Description

**Queued path — `initiateWithdrawal`:** [1](#0-0) 

```solidity
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();
// preventing over-withdrawal.
assetsCommitted[asset] += expectedAssetAmount;
```

`getAvailableAssetAmount` computes `lrtDepositPool.getTotalAssetDeposits(asset) - assetsCommitted[asset]`, spanning all protocol assets (deposit pool + node delegators + unstaking vault). Each queued request increments `assetsCommitted`, reserving that amount against future requests. [2](#0-1) 

**Instant path — `instantWithdrawal`:** [3](#0-2) 

```solidity
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
ILRTUnstakingVault unstakingVault = ...;
if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
    revert CantInstantWithdrawMoreThanAvailable();
}
unstakingVault.redeem(asset, assetAmountUnlocked);
```

`getAssetsAvailableForInstantWithdrawal` returns `vaultBalance - queuedWithdrawalsBuffer[asset]`. [4](#0-3) 

`queuedWithdrawalsBuffer` is a manually-set operator parameter that **defaults to zero** for every asset. [5](#0-4) 

`instantWithdrawal` **never reads `assetsCommitted`** and **never updates it**. It simply drains the vault up to `vaultBalance - queuedWithdrawalsBuffer`.

**Queue fulfillment — `unlockQueue`:**

When the operator calls `unlockQueue`, the available assets are taken directly from the vault balance: [6](#0-5) 

```solidity
totalAvailableAssets: unstakingVault.balanceOf(asset)
```

If the vault has been drained by instant withdrawals, `totalAvailableAssets == 0`, and `_unlockWithdrawalRequests` exits immediately without unlocking any queued request. The queued users' rsETH remains locked in the withdrawal manager indefinitely. [7](#0-6) 

**The desync in one equation:**

```
assetsCommitted[asset]  >  LRTUnstakingVault.balanceOf(asset)
```

This state is reachable whenever instant withdrawals consume vault assets that were already promised to queued withdrawal users, because the two accounting variables are never reconciled with each other.

---

### Impact Explanation

Users who called `initiateWithdrawal` have already transferred their rsETH to the withdrawal manager. If the vault is subsequently drained by instant withdrawers, `unlockQueue` cannot process their requests. Their rsETH is locked in the contract until an operator manually replenishes the vault by completing a new EigenLayer unstaking cycle — a multi-day process. This constitutes a **temporary freezing of user funds**.

**Impact: Medium — Temporary freezing of funds.**

---

### Likelihood Explanation

- `isInstantWithdrawalEnabled[asset]` must be `true` (operator-enabled feature, realistic in production).
- `queuedWithdrawalsBuffer[asset]` defaults to `0` for every asset; it must be explicitly set by an operator to provide any protection. If it is never set, or set too low, the vulnerability is fully exposed.
- Any rsETH holder can call `instantWithdrawal` permissionlessly once the feature is enabled.
- The attack requires no special knowledge: a user simply calls `instantWithdrawal` for the full vault balance while queued withdrawal requests exist.

**Likelihood: Medium.**

---

### Recommendation

Synchronize the two accounting mechanisms. In `instantWithdrawal`, check that the requested asset amount does not exceed `getAvailableAssetAmount(asset) - assetsCommitted[asset]` (i.e., assets not already promised to queued users), or alternatively, make `getAssetsAvailableForInstantWithdrawal` aware of `assetsCommitted` so that the vault buffer is automatically set to at least `assetsCommitted[asset]` rather than relying on a manually-configured `queuedWithdrawalsBuffer`. The single source of truth for "how much of the vault is reserved" should be `assetsCommitted`, not a separate manually-maintained buffer.

---

### Proof of Concept

```
State: LRTUnstakingVault holds 100 ETH. queuedWithdrawalsBuffer[ETH] = 0.
       isInstantWithdrawalEnabled[ETH] = true.

1. Alice calls initiateWithdrawal(ETH, rsETH_A)
   → assetsCommitted[ETH] += 80 ETH
   → Alice's rsETH (worth 80 ETH) is locked in LRTWithdrawalManager

2. Bob calls instantWithdrawal(ETH, rsETH_B)
   → getAssetsAvailableForInstantWithdrawal = 100 - 0 = 100 ETH  ✓
   → unstakingVault.redeem(ETH, 100 ETH)  — vault drained to 0
   → assetsCommitted[ETH] unchanged = 80 ETH

3. Operator calls unlockQueue(ETH, ...)
   → totalAvailableAssets = unstakingVault.balanceOf(ETH) = 0
   → _unlockWithdrawalRequests exits immediately (availableAssetAmount < payoutAmount)
   → Alice's request remains locked; her rsETH is stuck in the manager

Result:
  assetsCommitted[ETH] = 80 ETH
  LRTUnstakingVault.balanceOf(ETH) = 0
  Alice cannot complete her withdrawal until the vault is replenished by a new
  EigenLayer unstaking cycle (days of delay, operator-dependent).
```

    function getAssetsAvailableForInstantWithdrawal(address asset)
        external
        view
        onlySupportedAsset(asset)
        returns (uint256 availableAmount)
    {
        uint256 vaultBalance = balanceOf(asset);
        uint256 reservedBuffer = queuedWithdrawalsBuffer[asset];
        availableAmount = reservedBuffer >= vaultBalance ? 0 : vaultBalance - reservedBuffer;
    }
