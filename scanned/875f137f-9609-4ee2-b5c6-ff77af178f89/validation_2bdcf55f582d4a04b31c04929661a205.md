### Title
Stale Cross-Chain rsETH/ETH Rate in `CrossChainRateReceiver` Causes L2 Pool Over-Minting of wrsETH — (File: `contracts/cross-chain/CrossChainRateReceiver.sol`)

---

### Summary

All L2 pool contracts (`RSETHPool`, `RSETHPoolV2`, `RSETHPoolV2ExternalBridge`, `RSETHPoolV2NBA`, `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`) compute the amount of wrsETH to mint/transfer using `getRate()`, which reads a cached `rate` stored in `CrossChainRateReceiver`. This rate is only updated when a LayerZero message arrives from L1. No staleness check is performed at read time. Because rsETH appreciates monotonically, a stale (lower-than-actual) rate causes every depositor to receive more wrsETH than their ETH deposit justifies, diluting existing wrsETH holders and constituting theft of their accrued yield.

---

### Finding Description

`CrossChainRateReceiver` stores the last received rsETH/ETH rate in a public `rate` variable and records the update timestamp in `lastUpdated`. [1](#0-0) 

The `getRate()` function returns `rate` unconditionally, with no check against `lastUpdated`: [2](#0-1) 

The rate is only updated when `lzReceive()` is called by the LayerZero endpoint, which happens only when someone triggers `updateRate()` on the L1 provider (`RSETHMultiChainRateProvider` / `RSETHRateProvider`). Between such triggers, the L2 oracle rate is frozen at its last received value. [3](#0-2) 

Every L2 pool contract calls `IOracle(rsETHOracle).getRate()` to obtain this potentially stale rate and uses it to compute the wrsETH amount:

```
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate
``` [4](#0-3) [5](#0-4) 

Because rsETH is a yield-bearing token whose ETH value increases over time, the actual L1 rate (`LRTOracle.rsETHPrice()`) is always ≥ the last broadcast rate. A stale L2 rate is therefore always ≤ the true rate. Since `rsETHAmount` is inversely proportional to the rate, a stale (lower) rate produces a larger `rsETHAmount` than the deposited ETH actually warrants.

The `RSETHMultiChainRateProvider.getLatestRate()` reads `ILRTOracle(rsETHPriceOracle).rsETHPrice()` — the live L1 price — but this value is only pushed cross-chain on demand: [6](#0-5) 

The `RSETHRateReceiver` (the concrete L2 receiver) inherits `CrossChainRateReceiver` without adding any staleness guard: [7](#0-6) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

When the L2 oracle rate lags behind the true L1 rate, every depositor receives excess wrsETH. For example, if the true rate is 1.05 ETH/rsETH but the stale L2 rate is 1.03 ETH/rsETH, a 1 ETH deposit yields ≈ 0.9709 wrsETH instead of the correct ≈ 0.9524 wrsETH — an excess of ≈ 0.0185 wrsETH per ETH deposited. This excess represents yield that belongs to existing wrsETH holders being silently transferred to new depositors. For `RSETHPoolV2` and later variants that **mint** new wrsETH, this directly dilutes all existing holders. For `RSETHPool` (Arbitrum), which transfers pre-held wrsETH, the pool's reserves are depleted faster than the collected ETH justifies, creating a shortfall borne by the protocol.

---

### Likelihood Explanation

**Medium.** The rate update is not automatic; it requires an off-chain actor to call `updateRate()` on the L1 provider and pay LayerZero messaging fees. In practice, updates are periodic (not per-block). rsETH accrues EigenLayer rewards continuously, so the rate drifts upward between every pair of updates. Any depositor who monitors `CrossChainRateReceiver.lastUpdated` and `LRTOracle.rsETHPrice()` can identify windows where the L2 rate is stale and deposit during those windows to extract excess wrsETH. No special permissions are required; `deposit()` is open to any caller. [8](#0-7) [9](#0-8) 

---

### Recommendation

1. **Add a staleness check in `CrossChainRateReceiver.getRate()`**: revert (or return a sentinel) if `block.timestamp - lastUpdated` exceeds a configurable `maxStaleness` threshold (e.g., 24 hours).
2. **Enforce the staleness check in pool contracts**: before computing `rsETHAmount`, verify that the oracle rate is fresh.
3. **Document the asynchronicity risk** and instruct integrators to monitor `lastUpdated` before depositing large amounts, analogous to the `minAmountLD` guidance in the referenced report.

---

### Proof of Concept

1. At time T₀, `updateRate()` is called on L1. `CrossChainRateReceiver.rate` on L2 is set to `R₀ = 1.03e18` (1.03 ETH per rsETH). `lastUpdated = T₀`.
2. EigenLayer rewards accrue. At time T₁ = T₀ + 48 hours, the true L1 `rsETHPrice` has risen to `R₁ = 1.05e18`. No `updateRate()` has been called; the L2 oracle still returns `1.03e18`.
3. Attacker calls `RSETHPoolV2.deposit{value: 100 ether}("")`.
4. `viewSwapRsETHAmountAndFee(100e18)` computes:
   - `fee = 100e18 * feeBps / 10_000`
   - `rsETHAmount = (100e18 - fee) * 1e18 / 1.03e18 ≈ 97.087 wrsETH`
5. Correct amount at true rate: `(100e18 - fee) * 1e18 / 1.05e18 ≈ 95.238 wrsETH`.
6. Attacker receives ≈ 1.849 excess wrsETH per 100 ETH deposited — yield stolen from existing holders — with no admin involvement and no special role required. [1](#0-0) [5](#0-4)
