### Title
Owner Can Drain KERNEL Reserve via `withdrawTokens`, Permanently Freezing All Unclaimed Vested Yield — (`contracts/KERNEL/KernelTop100MerkleDistributor.sol`)

### Summary
`withdrawTokens` has no guard preventing the owner from withdrawing the KERNEL token itself. After a drain, all subsequent `claim` and `claimAndStake` calls revert due to insufficient balance, permanently freezing every user's unclaimed vested allocation.

### Finding Description
`withdrawTokens` accepts an arbitrary `_token` address and `_amount` with no restriction on withdrawing the distribution token (`kernel`) and no accounting check against total outstanding claimable amounts. [1](#0-0) 

The `claim` function transfers tokens directly from the contract's balance after updating state: [4](#0-3) 

There is no reserve tracking, no minimum balance enforcement, and no mechanism for users to recover their allocation after a drain.

### Impact Explanation
All users with valid merkle allocations who have not yet claimed their full vested amount lose access to those tokens permanently. The `userClaims[user].amountClaimed` state is already updated before the transfer, so even if KERNEL is later re-deposited, users who had their claim revert mid-execution would have corrupted claim state (their `amountClaimed` was incremented but they received nothing). [5](#0-4) 

**Impact: Medium — Permanent freezing of unclaimed yield.**

### Likelihood Explanation
Requires the owner to call `withdrawTokens` with the KERNEL token address, either maliciously or by mistake (e.g., intending to recover accidentally sent tokens but passing `kernel` instead). No external attacker action is needed beyond triggering `claim` afterward. Likelihood is low-to-medium given it is a single privileged call with no confirmation step.

### Recommendation
1. Add a guard in `withdrawTokens` that prevents withdrawing the distribution token:
   ```solidity
   require(_token != address(kernel), "Cannot withdraw distribution token");
   ```
2. Alternatively, track total outstanding claimable KERNEL and enforce a minimum reserve:
   ```solidity
   require(
       IERC20(kernel).balanceOf(address(this)) - _amount >= totalOutstandingClaimable,
       "Insufficient reserve"
   );
   ```
3. Consider removing `kernel` from the set of tokens recoverable via `withdrawTokens` entirely, since the contract's sole purpose is to distribute it.
