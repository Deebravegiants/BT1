### Title
Unpermissioned `updateRSETHPrice()` Allows Any Caller to Trigger a Protocol-Wide Pause, Temporarily Freezing All User Deposits and Withdrawals - (File: `contracts/LRTOracle.sol`)

---

### Summary

`LRTOracle.updateRSETHPrice()` carries no access control — any address can call it. Inside `_updateRsETHPrice()`, when the freshly computed rsETH price has fallen more than `pricePercentageLimit` below `highestRsethPrice`, the function immediately pauses both `LRTDepositPool` and `LRTWithdrawalManager` and self-pauses the oracle. Because the pause path is reachable by an unprivileged caller, an attacker who observes a qualifying price drop can race to call `updateRSETHPrice()` before the protocol team, locking all user deposits and withdrawals until an admin manually unpauses — a process that requires multisig coordination and takes time.

---

### Finding Description

`LRTOracle.updateRSETHPrice()` is declared `public` with only the `whenNotPaused` guard:

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

Inside `_updateRsETHPrice()`, the downside-protection branch executes for any caller:

```solidity
// contracts/LRTOracle.sol L270-281
if (newRsETHPrice < highestRsethPrice) {
    uint256 diff = highestRsethPrice - newRsETHPrice;
    bool isPriceDecreaseOffLimit =
        pricePercentageLimit > 0 && diff > pricePercentageLimit.mulWad(highestRsethPrice);

    if (isPriceDecreaseOffLimit) {
        if (!lrtDepositPool.paused()) lrtDepositPool.pause();
        if (!withdrawalManager.paused()) withdrawalManager.pause();
        _pause();
        return;
    }
```

There is a deliberate asymmetry in the design: for price *increases* beyond the threshold the function reverts for non-managers (`PriceAboveDailyThreshold`), protecting that path. But for price *decreases* beyond the threshold the function executes the pause for **any** caller with no role check.

Once paused:
- `LRTDepositPool.depositETH()` / `depositAsset()` revert on `whenNotPaused`.
- `LRTWithdrawalManager.initiateWithdrawal()`, `completeWithdrawal()`, and `instantWithdrawal()` all revert on `whenNotPaused`.
- The oracle itself is paused, blocking further price updates via the public path.

Unpausing requires an admin (`onlyLRTAdmin`) to call `unpause()` on each contract separately — a multisig operation that introduces meaningful delay.

---

### Impact Explanation

An attacker who successfully races to call `updateRSETHPrice()` when the price-drop condition is met causes a **temporary freeze of all user funds**: no deposits can enter the protocol and no withdrawals (standard, instant, or operator-assisted) can exit. Users with pending withdrawal requests cannot complete them. The freeze persists until the admin multisig executes three separate unpause transactions (`LRTDepositPool.unpause()`, `LRTWithdrawalManager.unpause()`, `LRTOracle.unpause()`).

**Impact: Medium — Temporary freezing of funds.**

---

### Likelihood Explanation

The triggering condition — `newRsETHPrice` dropping more than `pricePercentageLimit` below `highestRsethPrice` — is a real-world event that occurs during EigenLayer slashing, LST de-pegs, or temporary oracle price dips. The attacker does not need to manufacture the condition; they only need to monitor the chain and submit the transaction first. Because `updateRSETHPrice()` is a cheap, permissionless call, a bot can reliably front-run the protocol team's own price-update transaction during any qualifying market event.

**Likelihood: Medium.**

---

### Recommendation

Apply the same role-gating to the downside-pause path that already protects the upside path. The simplest fix is to restrict `updateRSETHPrice()` to authorized callers (manager or operator role), mirroring `updateRSETHPriceAsManager()`:

```solidity
// Before (vulnerable)
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}

// After (recommended)
function updateRSETHPrice() external whenNotPaused onlyLRTOperator {
    _updateRsETHPrice();
}
```

Alternatively, separate the pause-triggering logic from the price-update logic so that the pause can only be initiated by a privileged role, while price reads remain public.

---

### Proof of Concept

1. Protocol is live; `pricePercentageLimit` is set to `1e16` (1%).
2. A slashing event reduces the ETH value of assets held in EigenLayer by 1.5%, causing `newRsETHPrice` to fall 1.5% below `highestRsethPrice`.
3. Attacker computes off-chain that `isPriceDecreaseOffLimit == true`.
4. Attacker calls `LRTOracle.updateRSETHPrice()` (no role required, only costs gas).
5. `_updateRsETHPrice()` executes lines 278–280:
   - `lrtDepositPool.pause()` — all deposits blocked.
   - `withdrawalManager.pause()` — all withdrawals blocked.
   - `_pause()` — oracle paused.
6. Alice, who had queued a withdrawal and whose delay has now elapsed, calls `completeWithdrawal()` — it reverts with `Pausable: paused`.
7. The protocol team must coordinate a multisig to call `unpause()` on three contracts before any user can interact again.
