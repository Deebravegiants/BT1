### Title
No Staleness Check on `latestRoundData()` Allows Stale Price Exploitation - (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `AggregatorV3Interface.latestRoundData()` but silently discards every return value except `price`. Neither `updatedAt` nor `answeredInRound` is validated, so a stale Chainlink answer is accepted as current. This price feeds directly into rsETH mint and redemption accounting, enabling a depositor or withdrawer to extract value from the protocol when a feed goes stale.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` is the price source for every LST supported by the core LRT protocol:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52
(, int256 price,,,) = priceFeed.latestRoundData();
``` [1](#0-0) 

All five return values are available (`roundId`, `answer`, `startedAt`, `updatedAt`, `answeredInRound`), but only `answer` is read. There is no check of the form `block.timestamp - updatedAt < heartbeat` and no check that `answeredInRound >= roundId`.

Compare this with `ChainlinkOracleForRSETHPoolCollateral`, which is used for pool collateral tokens and does validate `answeredInRound < roundID` and `timestamp == 0`:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L27-32
(uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
    AggregatorV3Interface(oracle).latestRoundData();
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
``` [2](#0-1) 

The unprotected `ChainlinkPriceOracle.getAssetPrice()` is consumed by `LRTOracle.getAssetPrice()`: [3](#0-2) 

Which is called inside `_getTotalEthInProtocol()` to compute the total ETH value of all deposited LSTs: [4](#0-3) 

That total feeds `_updateRsETHPrice()`, which sets the global `rsETHPrice` storage variable: [5](#0-4) 

`updateRSETHPrice()` is a public, permissionless function: [6](#0-5) 

The corrupted `rsETHPrice` and per-asset price are then used in two critical user-facing paths:

**Deposit** — `LRTDepositPool.depositAsset()` → `getRsETHAmountToMint()`:
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [7](#0-6) 

**Withdrawal** — `LRTWithdrawalManager.getExpectedAssetAmount()`:
```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
``` [8](#0-7) 

---

### Impact Explanation

**Critical — Direct theft of user funds.**

If a Chainlink feed for a supported LST (e.g., stETH/ETH) becomes stale and the last reported price is inflated relative to the true market price:

1. An attacker calls `updateRSETHPrice()` to lock in the inflated `rsETHPrice`.
2. The attacker calls `depositAsset()` with that LST and receives more rsETH than the LST is worth.
3. The attacker redeems or sells the excess rsETH, extracting value from all other rsETH holders.

Conversely, a stale deflated price allows a withdrawer to receive more LST per rsETH than is fair, directly draining the deposit pool.

---

### Likelihood Explanation

Chainlink feeds can go stale due to network congestion, sequencer downtime (on L2), or oracle node failure. The affected `ChainlinkPriceOracle` is the production oracle for all LSTs on Ethereum mainnet (stETH, ETHx, rETH, swETH, sfrxETH). The entry point `updateRSETHPrice()` and `depositAsset()` are both permissionless. No special conditions beyond a stale feed are required.

---

### Recommendation

Add staleness and round-completeness checks in `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    // heartbeat should be set per feed (e.g., 3600 for 1-hour feeds)
    if (block.timestamp - updatedAt > STALENESS_THRESHOLD) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

---

### Proof of Concept

1. Chainlink stETH/ETH feed goes stale; last reported price is 1.05e18 (true price: 1.00e18).
2. Attacker calls `LRTOracle.updateRSETHPrice()` — `_getTotalEthInProtocol()` uses the stale 1.05 price, inflating total ETH by 5%, raising `rsETHPrice`.
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, ...)` — `getRsETHAmountToMint` computes `(100e18 * 1.05e18) / rsETHPrice`, yielding ~5% more rsETH than fair value.
4. Attacker redeems or sells the excess rsETH, extracting value from existing holders.

The root cause is exclusively in `contracts/oracles/ChainlinkPriceOracle.sol` line 52, where `updatedAt` and `answeredInRound` are silently discarded. [9](#0-8)
