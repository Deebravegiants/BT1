### Title
Stale `rsETHPrice` Used in `depositAsset()` and `initiateWithdrawal()` Without Forcing an Oracle Update - (File: contracts/LRTDepositPool.sol, contracts/LRTWithdrawalManager.sol)

---

### Summary

`LRTDepositPool.depositAsset()` and `LRTWithdrawalManager.initiateWithdrawal()` both compute share/asset amounts using the stored `rsETHPrice` state variable from `LRTOracle` without first calling `updateRSETHPrice()`. Because `updateRSETHPrice()` is public but never invoked atomically inside these user-facing functions, any gap between the last oracle update and the current block allows a depositor to mint more rsETH than the protocol's true backing warrants, extracting value from existing rsETH holders.

---

### Finding Description

`LRTOracle` stores the rsETH/ETH exchange rate in the state variable `rsETHPrice`. [1](#0-0) 

This value is only refreshed when `updateRSETHPrice()` (or its manager variant) is explicitly called: [2](#0-1) 

Neither `LRTDepositPool.depositAsset()` nor `LRTDepositPool.depositETH()` calls `updateRSETHPrice()` before computing the rsETH amount to mint. The mint calculation in `getRsETHAmountToMint()` reads the stored value directly: [3](#0-2) 

Similarly, `LRTWithdrawalManager.initiateWithdrawal()` calls `getExpectedAssetAmount()`, which also reads the stored `rsETHPrice` without triggering an update: [4](#0-3) 

And `instantWithdrawal()` follows the same pattern: [5](#0-4) 

As staking rewards accrue over time, the true rsETH/ETH rate rises continuously, but `rsETHPrice` only reflects this when `updateRSETHPrice()` is called. The longer the gap since the last update, the more undervalued `rsETHPrice` is relative to the true backing.

---

### Impact Explanation

**Impact: High — Theft of unclaimed yield from existing rsETH holders.**

When `rsETHPrice` is stale (lower than the true current value), the mint formula

```
rsethAmountToMint = (depositAmount × assetPrice) / rsETHPrice
```

produces a larger rsETH amount than the depositor's contribution actually warrants. The attacker receives rsETH at a discount, diluting all existing holders. Once `updateRSETHPrice()` is called and the price rises to reflect accumulated rewards, the attacker's rsETH is worth more than what they deposited — the difference is yield stolen from existing holders.

The reverse direction also harms withdrawers: a stale (lower) `rsETHPrice` causes `getExpectedAssetAmount()` to return fewer assets than the withdrawer is entitled to, but this direction benefits the protocol rather than an attacker.

---

### Likelihood Explanation

**Likelihood: Medium.**

`updateRSETHPrice()` is called by off-chain bots on a periodic schedule. Any of the following realistic conditions creates an exploitable window:

- Bot downtime or network congestion delays the update.
- An attacker front-runs a pending `updateRSETHPrice()` transaction by depositing in the same block before the update lands.
- The `pricePercentageLimit` guard causes `updateRSETHPrice()` to revert for non-managers when the price has drifted beyond the threshold, extending the staleness window. [6](#0-5) 

The attacker needs no special role — `depositAsset()` and `initiateWithdrawal()` are fully permissionless.

---

### Recommendation

Call `updateRSETHPrice()` (or an internal equivalent) at the beginning of `depositAsset()`, `depositETH()`, `initiateWithdrawal()`, and `instantWithdrawal()` to ensure the price is fresh before any share/asset calculation. This mirrors the fix described in the external report and eliminates the staleness window entirely.

---

### Proof of Concept

1. Assume `rsETHPrice` was last updated 24 hours ago at `1.010 ETH/rsETH`. Staking rewards have since accrued; the true rate is now `1.011 ETH/rsETH`.
2. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
3. `getRsETHAmountToMint()` computes: `100e18 × 1e18 / 1.010e18 ≈ 99.0099 rsETH` (using stale price).
4. True fair amount at current rate: `100e18 × 1e18 / 1.011e18 ≈ 98.9120 rsETH`.
5. Attacker receives `~0.0979 rsETH` more than entitled — funded by diluting existing holders' share of accumulated yield.
6. Attacker calls `updateRSETHPrice()` (public), price updates to `1.011 ETH/rsETH`.
7. Attacker initiates withdrawal, receiving `99.0099 × 1.011 / 1e18 ≈ 100.098 ETH` worth of stETH — more than the `100 ETH` deposited. [7](#0-6) [8](#0-7)
