### Title
ETH Deposit Limit Bypass Due to Missing `amount` in Comparison - (File: `contracts/LRTDepositPool.sol`)

---

### Summary

`_checkIfDepositAmountExceedesCurrentLimit()` in `LRTDepositPool` uses the wrong variable when checking the ETH deposit limit. The `amount` parameter (the user's new deposit) is omitted from the ETH branch comparison, so any depositor can push the protocol's ETH holdings arbitrarily above the configured cap.

---

### Finding Description

The function `_checkIfDepositAmountExceedesCurrentLimit()` has two branches:

```solidity
// contracts/LRTDepositPool.sol  lines 676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct
}
``` [1](#0-0) 

For every LST the check is `totalAssetDeposits + amount > limit`, which correctly accounts for the incoming deposit. For ETH the check is only `totalAssetDeposits > limit`, which ignores `amount` entirely. As long as the running total has not already exceeded the cap, the condition returns `false` (not exceeded) regardless of how large the new deposit is.

This is called from `_beforeDeposit`, which gates every `depositETH` call:

```solidity
// contracts/LRTDepositPool.sol  lines 648-670
function _beforeDeposit(...) private view returns (uint256 rsethAmountToMint) {
    ...
    if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
        revert MaximumDepositLimitReached();
    }
    ...
}
``` [2](#0-1) 

---

### Impact Explanation

The ETH deposit cap (`depositLimitByAsset[ETH_TOKEN]`) is the protocol's primary mechanism for controlling how much native ETH enters EigenLayer. Because the incoming `amount` is never added to `totalAssetDeposits` before the comparison, a single depositor can send an arbitrarily large ETH deposit in one transaction and bypass the cap entirely, as long as the running total has not already crossed the limit. This causes the protocol to accept and restake more ETH than its risk parameters allow, breaking the promised deposit-limit guarantee.

**Impact**: Low — Contract fails to deliver promised returns (deposit limit is not enforced for ETH).

---

### Likelihood Explanation

Any unprivileged ETH depositor can trigger this in a single `depositETH` call whenever `totalAssetDeposits(ETH) ≤ depositLimitByAsset(ETH)`. No special role, front-running, or oracle manipulation is required. The condition is met during normal protocol operation.

---

### Recommendation

Add `amount` to the ETH branch, mirroring the LST branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

---

### Proof of Concept

1. Admin sets `depositLimitByAsset[ETH] = 1000 ether`.
2. Current `getTotalAssetDeposits(ETH) = 999 ether`.
3. Attacker calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH, 500 ether)` evaluates `999 ether > 1000 ether` → `false` → no revert.
5. 500 ETH is accepted; total becomes 1499 ETH, 49.9% above the configured cap.
6. The same check for any LST would have evaluated `999 + 500 > 1000` → `true` → `MaximumDepositLimitReached` revert. [1](#0-0) [3](#0-2)
