### Title
No Minimum rsETH Output Protection in L2 Pool `deposit` Functions Exposes Depositors to Unlimited Oracle-Rate Slippage - (File: contracts/pools/RSETHPoolV3.sol)

---

### Summary

The `deposit` functions in `RSETHPoolV3`, `RSETHPool`, and `RSETHPoolNoWrapper` calculate the rsETH amount to transfer based on a live oracle rate but accept no `minRsETHAmount` parameter. A depositor has no on-chain mechanism to bound the minimum rsETH they will receive, leaving them fully exposed to any oracle rate movement between transaction submission and execution. The L1 counterpart `LRTDepositPool` already enforces this protection via `minRSETHAmountExpected`, making the omission in the L2 pools a clear inconsistency.

---

### Finding Description

Every L2 pool `deposit` entry point computes the output amount at execution time from the oracle rate:

**`RSETHPoolV3.deposit(string)` (ETH path)** [1](#0-0) 

**`RSETHPoolV3.deposit(address,uint256,string)` (token path)** [2](#0-1) 

The rsETH amount is derived entirely from `viewSwapRsETHAmountAndFee`, which divides the deposit by the live oracle rate: [3](#0-2) 

Neither overload accepts a `minRsETHAmount` argument. The same pattern is present in `RSETHPool.deposit` and `RSETHPoolNoWrapper.deposit`: [4](#0-3) [5](#0-4) 

By contrast, the L1 `LRTDepositPool` explicitly enforces a caller-supplied floor and reverts if the minted amount falls below it: [6](#0-5) [7](#0-6) 

The oracle rate used by the L2 pools (`rsETHOracle.getRate()`) reflects the rsETH/ETH exchange rate. This rate is updated on-chain (the L1 `LRTOracle.updateRSETHPrice()` is a public function callable by anyone) and can increase between the block in which a user signs a transaction and the block in which it is included. When the rate rises, the divisor in `amountAfterFee * 1e18 / rsETHToETHrate` grows, and the user receives fewer rsETH tokens than they observed when constructing the transaction.

---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but does not lose value.**

A depositor who observes a rate of `R` at simulation time and submits a deposit may receive rsETH computed at rate `R + δ` at execution time, yielding fewer tokens than expected with no recourse. The deposited ETH/LST is not returned; the user simply receives a smaller rsETH balance than the off-chain quote indicated. Because the L1 contract already provides this protection and the L2 contracts deliberately omit it, users interacting with L2 pools are silently under-protected relative to the documented interface.

---

### Likelihood Explanation

**Medium.** Every ETH or token deposit through any of the three L2 pool contracts is affected. The oracle rate is updated periodically; any deposit whose execution is delayed by even one oracle update cycle will receive a worse-than-quoted output. No special privileges or capital are required — any ordinary depositor is exposed on every call.

---

### Recommendation

Add a `minRsETHAmount` parameter to all `deposit` overloads in `RSETHPoolV3`, `RSETHPool`, and `RSETHPoolNoWrapper`, mirroring the pattern already used in `LRTDepositPool._beforeDeposit`. Revert if the computed rsETH amount falls below the caller-supplied floor:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmount)
    external payable nonReentrant whenNotPaused
    limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmount) revert InsufficientRsETHOutput();
    ...
}
```

---

### Proof of Concept

1. Alice calls `viewSwapRsETHAmountAndFee(1 ether)` off-chain and observes she will receive `X` rsETH at the current oracle rate `R`.
2. Alice submits `RSETHPoolV3.deposit{value: 1 ether}("ref")` with no minimum bound.
3. Before Alice's transaction is included, the oracle rate is updated to `R + δ` (e.g., via a public `updateRSETHPrice()` call or a scheduled keeper update).
4. Alice's transaction executes: `rsETHAmount = 1e18 * 1e18 / (R + δ)`, which is strictly less than `X`.
5. Alice receives fewer rsETH than she observed, with no on-chain protection and no revert. The shortfall is permanent — she cannot recover the difference.

    function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
    }
