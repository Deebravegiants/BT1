### Title
Zero-Value ERC20 Transfer in `MerkleDistributor#claim` Blocks All Claims When `feeInBPS` Is Zero - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

---

### Summary

`MerkleDistributor.claim()` unconditionally calls `IERC20(token).safeTransfer(protocolTreasury, fee)` even when `fee` is zero. When `feeInBPS` is set to `0` (a valid and explicitly supported state), `fee` evaluates to `0`, and any token that reverts on zero-value transfers (e.g., LEND) will cause every `claim()` call to revert, permanently blocking all users from claiming their allocated tokens.

---

### Finding Description

In `MerkleDistributor.sol`, the `claim()` function computes a protocol fee and sends it to `protocolTreasury`:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);

// Send the fee to the protocol treasury
IERC20(token).safeTransfer(protocolTreasury, fee);  // <-- no zero-check
``` [1](#0-0) 

When `feeInBPS == 0`, `fee` is always `0`, and the second `safeTransfer` attempts a zero-value transfer. Tokens that revert on zero-value transfers (a well-documented class of non-standard ERC20 tokens) will cause the entire `claim()` call to revert.

`feeInBPS = 0` is explicitly permitted by both the initializer and the `setFeeInBPS` setter — neither enforces a non-zero lower bound:

```solidity
if (_feeInBPS > MAX_FEE_IN_BPS) {
    revert InvalidFeeInBPS();
}
``` [2](#0-1) 

```solidity
function setFeeInBPS(uint256 _feeInBPS) external onlyOwner {
    if (_feeInBPS > MAX_FEE_IN_BPS) {
        revert InvalidFeeInBPS();
    }
    feeInBPS = _feeInBPS;
``` [3](#0-2) 

The `claim()` function has no access control — it is callable by any address with a valid Merkle proof:

```solidity
function claim(
    uint256 index,
    address account,
    uint256 cumulativeAmount,
    bytes32[] calldata merkleProof
)
    external
    override
    whenNotPaused
``` [4](#0-3) 

---

### Impact Explanation

When `feeInBPS == 0` and the distributed token reverts on zero-value transfers, **every single `claim()` call reverts**. All users with valid Merkle allocations are unable to claim their tokens. The state update (`userClaims[account].lastClaimedIndex` and `userClaims[account].cumulativeAmount`) is written before the failing transfer, meaning the revert rolls back the state — but users remain permanently blocked from claiming as long as `feeInBPS` stays at `0` and the token exhibits this behavior.

This constitutes **permanent freezing of unclaimed yield** (Medium) for the duration that `feeInBPS == 0`, and **temporary freezing of funds** (Medium) if the owner eventually raises `feeInBPS` above zero.

---

### Likelihood Explanation

- `feeInBPS = 0` is a valid, explicitly supported configuration — the contract can be deployed with it or the owner can set it at any time via `setFeeInBPS(0)`.
- The `MerkleDistributor` is a generic distributor designed to distribute arbitrary ERC20 tokens. The token is configurable post-deployment via `setToken()`, meaning a zero-transfer-reverting token can be set at any time.
- No attacker action is required. The DoS is triggered by the combination of a valid protocol configuration (`feeInBPS == 0`) and a non-standard but real token behavior.

---

### Recommendation

Add a zero-check before the fee transfer to `protocolTreasury`:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);

if (fee > 0) {
    IERC20(token).safeTransfer(protocolTreasury, fee);
}
```

This mirrors the fix recommended in M-32 and is consistent with how `LRTWithdrawalManager.instantWithdrawal()` already guards its fee transfer:

```solidity
if (fee > 0) {
    _transferAsset(asset, feeRecipient, fee);
}
``` [5](#0-4) 

---

### Proof of Concept

1. `MerkleDistributor` is deployed or configured with `feeInBPS = 0` (valid per both `initialize` and `setFeeInBPS`).
2. The distributed `token` is any ERC20 that reverts on zero-value transfers (e.g., LEND).
3. A user calls `claim(index, account, cumulativeAmount, merkleProof)` with a valid proof and a non-zero `claimableAmount`.
4. `fee = (claimableAmount * 0) / 10_000 = 0`.
5. `IERC20(token).safeTransfer(account, claimableAmount)` succeeds.
6. `IERC20(token).safeTransfer(protocolTreasury, 0)` reverts because the token rejects zero-value transfers.
7. The entire transaction reverts. All users are blocked from claiming their tokens. [1](#0-0)
