### Title
Rewards Permanently Frozen When `totalKernelStaked` Drops to Zero Mid-Period — (File: `contracts/KERNEL/KernelDepositPool.sol`)

---

### Summary

`KernelDepositPool` implements a Synthetix-style staking rewards mechanism. When `totalKernelStaked` reaches zero during an active reward distribution window — triggered by any unprivileged user who is the last staker calling `initiateWithdrawal` — the `rewardPerToken()` accumulator freezes while `updatedAt` is silently advanced on the next interaction. All reward tokens that should have been distributed during the zero-staking interval are permanently locked in the contract with no recovery path.

---

### Finding Description

The `rewardPerToken()` view function short-circuits when `totalKernelStaked == 0`:

```solidity
// KernelDepositPool.sol L408-413
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;   // accumulator frozen
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
```

The `updateReward` modifier, which runs before every state-changing function, always advances `updatedAt` regardless of whether `totalKernelStaked` is zero:

```solidity
// KernelDepositPool.sol L232-242
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    updatedAt = lastTimeRewardApplicable();   // always advances
    ...
}
```

**Sequence of events:**

1. Admin calls `notifyRewardAmount` with stakers present → `rewardRate > 0`, `finishAt = now + duration`, `updatedAt = now`.
2. The last staker calls `initiateWithdrawal(_amount)`. The `updateReward` modifier runs first (while `totalKernelStaked > 0`), correctly snapshotting `rewardPerTokenStored` and setting `updatedAt = now`. Then `totalKernelStaked -= _amount` → `totalKernelStaked == 0`.
3. Time passes — `rewardRate` is still non-zero, `finishAt` is still in the future, but no accumulation occurs because `totalKernelStaked == 0`.
4. A new staker calls `stake()`. The `updateReward` modifier fires **before** `totalKernelStaked` is incremented, so `totalKernelStaked` is still 0 at that moment. `rewardPerToken()` returns `rewardPerTokenStored` unchanged, and `updatedAt` is advanced to `lastTimeRewardApplicable()` (current time).
5. The entire reward amount that accrued during the zero-staking interval (`rewardRate × elapsed_time`) is silently consumed — it can never be credited to any user and is permanently stuck in the contract.

The contract's own NatSpec comment acknowledges this at lines 18–22:

> *"If `totalKernelStaked` ever hits zero during a reward distribution window, any remaining rewards for that period will stay locked in the contract."*

However, the mitigation is purely operational ("ensuring there are always some tokens staked for the entire duration of the reward period") and is **not enforced on-chain**. There is no admin sweep or rescue function for reward tokens in the contract. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

Reward tokens transferred into the contract via `notifyRewardAmount` that correspond to the zero-staking interval are permanently irrecoverable. There is no `sweep`, `rescueTokens`, or equivalent admin function. The `getReward()` path only transfers `rewards[msg.sender]`, which is only credited via `earned()` → `rewardPerToken()`, and that accumulator never advanced during the zero-staking window. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

Any unprivileged staker who holds the entire staked supply can unilaterally trigger this by calling `initiateWithdrawal`. Notably, `initiateWithdrawal` immediately decrements `totalKernelStaked` (line 326) — the withdrawal delay only governs when the underlying KERNEL tokens are transferred, not when the accounting drops to zero. This means a single user action is sufficient to freeze rewards for the remainder of the active period. The scenario is realistic whenever a dominant staker exits, or when the staking pool is small and users naturally exit during a reward window. [6](#0-5) 

---

### Recommendation

Enforce on-chain what the NatSpec comment only states operationally. Two complementary fixes:

1. **Pause reward accrual when `totalKernelStaked == 0`:** In the `updateReward` modifier, only advance `updatedAt` when `totalKernelStaked > 0`:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {          // only advance when stakers exist
        updatedAt = lastTimeRewardApplicable();
    }
    ...
}
```

This ensures the time gap during which no one is staked is not silently consumed — the accumulator will catch up correctly once a new staker arrives.

2. **Add an admin token-rescue function** for reward tokens that are provably unallocatable (e.g., after `finishAt` has passed and `totalKernelStaked` was zero for part of the period), so stuck rewards are not permanently lost. [2](#0-1)

### Proof of Concept

```solidity
// Scenario: last staker withdraws mid-period, rewards for remaining window are frozen

// 1. Admin sets up reward period with Alice staked
kernelDepositPool.notifyRewardAmount(1_000e18); // 1000 reward tokens, duration = 30 days
// rewardRate = 1000e18 / 30 days

// 2. Alice is the only staker; she initiates withdrawal after 10 days
// updateReward runs: rewardPerTokenStored advances for 10 days, updatedAt = now
// Then: totalKernelStaked -= aliceAmount → totalKernelStaked == 0
alice.initiateWithdrawal(aliceStake);

// 3. 20 days pass (remaining reward window). totalKernelStaked == 0 the entire time.
// rewardRate * 20 days worth of rewards = ~666e18 tokens are accruing but not credited.

// 4. Bob stakes
bob.stake(1e18);
// updateReward fires: totalKernelStaked still 0 at modifier entry
//   rewardPerToken() → returns rewardPerTokenStored (unchanged, no accumulation)
//   updatedAt = lastTimeRewardApplicable() = finishAt (period just ended)
// totalKernelStaked += 1e18

// 5. Bob claims rewards → 0 (he staked after finishAt effectively)
// Alice claims her 10-day share → ~333e18
// ~666e18 reward tokens are permanently stuck in the contract
uint256 stuck = rewardsToken.balanceOf(address(kernelDepositPool))
              - kernelDepositPool.rewards(alice)
              - kernelDepositPool.rewards(bob);
// stuck ≈ 666e18, permanently unrecoverable

    function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
        if (_amount == 0) revert AmountZero();

        // Prevent starting a reward period when no tokens are staked to avoid unallocated rewards
        if (totalKernelStaked == 0) revert NoStakedTokens();

        // Transfer reward tokens into the contract
        uint256 balanceBefore = rewardsToken.balanceOf(address(this));
        rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
        uint256 balanceAfter = rewardsToken.balanceOf(address(this));
        // Calculate the actual amount of tokens received in case of a transfer fee (tax)
        uint256 receivedAmount = balanceAfter - balanceBefore;

        if (block.timestamp >= finishAt) {
            rewardRate = receivedAmount / duration;
        } else {
            uint256 remaining = (finishAt - block.timestamp) * rewardRate;
            rewardRate = (receivedAmount + remaining) / duration;
        }

        if (rewardRate == 0) revert RewardRateZero();

        finishAt = block.timestamp + duration;
        updatedAt = block.timestamp;

        emit NotifyRewardAmount(receivedAmount, finishAt);
    }
