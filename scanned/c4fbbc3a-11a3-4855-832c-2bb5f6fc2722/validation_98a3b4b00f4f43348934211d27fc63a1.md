### Title
LST Price Oracles Use Only Internal Protocol Exchange Rates Without Market Price Validation, Enabling Deposit of Depegged Assets at Inflated Rates - (File: contracts/oracles/RETHPriceOracle.sol, contracts/oracles/SwETHPriceOracle.sol, contracts/oracles/SfrxETHPriceOracle.sol, contracts/oracles/EthXPriceOracle.sol)

### Summary

All four LST price oracles in the protocol price their respective assets using only the internal protocol exchange rate returned by the LST contract itself, with no secondary market price check. If any supported LST depegs from ETH on the open market, the oracle continues to return the inflated internal rate, allowing any user to deposit the depegged LST into `LRTDepositPool.depositAsset()` and receive rsETH minted at a value far exceeding the actual market value of the deposited asset.

### Finding Description

The four LST price oracles each delegate entirely to a single on-chain source that reflects only the protocol's internal accounting rate — not the market price:

- `RETHPriceOracle.getAssetPrice()` calls `IrETH(rETHAddress).getExchangeRate()` (Rocket Pool's internal rate)
- `SwETHPriceOracle.getAssetPrice()` calls `ISwETH(swETHAddress).getRate()` (Swell's internal rate)
- `SfrxETHPriceOracle.getAssetPrice()` calls `ISfrxETH(sfrxETHContractAddress).pricePerShare()` (Frax's vault internal rate)
- `EthXPriceOracle.getAssetPrice()` calls `IETHXStakePoolsManager(...).getExchangeRate()` (Stader's internal rate)

None of these oracles consult a secondary market price source (e.g., Chainlink, Uniswap TWAP). The internal rates only reflect accrued staking rewards and are monotonically increasing; they do not reflect a market depeg event.

`LRTDepositPool.depositAsset()` is a public, permissionless function. It calls `getRsETHAmountToMint()`, which computes:

```
rsethAmountToMint = (depositAmount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()
```

`lrtOracle.getAssetPrice(asset)` routes to the relevant single-source oracle above. If the market price of the LST has dropped significantly below the internal rate, the numerator is inflated relative to the true ETH value of the deposited tokens, and the user receives more rsETH than the deposited assets are worth.

### Impact Explanation

**Critical — Protocol insolvency.**

An attacker purchases a depegged LST cheaply on the open market (e.g., rETH trading at 0.80 ETH due to a Rocket Pool exploit), then deposits it into `LRTDepositPool.depositAsset()`. The oracle returns the internal rate (e.g., 1.07 ETH/rETH), so the attacker receives rsETH priced at 1.07 ETH per rETH deposited. The attacker immediately redeems or sells the rsETH, extracting value from the protocol. The protocol's TVL is now backed by assets worth less than the rsETH outstanding, causing insolvency and losses for all existing rsETH holders.

### Likelihood Explanation

**Medium.** LST depegs are not theoretical: stETH depegged to ~0.94 ETH during the May 2022 market crisis. Any of the four supported LSTs could depeg due to a smart contract exploit in the underlying protocol, a mass slashing event, or a liquidity crisis. The attack requires no special permissions — any user can call `depositAsset()` — and is immediately profitable whenever the market price of a supported LST falls below its internal protocol rate.

### Recommendation

Implement a dual-oracle check for each LST, analogous to the recommendation in the reference report. For each LST, compare the internal protocol rate against a market price oracle (e.g., a Chainlink LST/ETH feed or a Uniswap V3 TWAP). If the market price deviates below the internal rate by more than a configured threshold (e.g., 2%), halt new deposits of that asset. This prevents both price manipulation and depeg exploitation while preserving normal operation.

### Proof of Concept

1. Assume rETH's Rocket Pool protocol is exploited; rETH market price drops to 0.80 ETH while `rETH.getExchangeRate()` still returns 1.07e18.
2. Attacker buys 1000 rETH on the open market for 800 ETH.
3. Attacker calls `LRTDepositPool.depositAsset(rETH, 1000e18, minRSETH, "")`.
4. `getRsETHAmountToMint` computes: `(1000e18 * 1.07e18) / rsETHPrice` — pricing the deposit as if it is worth 1070 ETH.
5. Attacker receives rsETH representing ~1070 ETH of claim, having deposited assets worth only 800 ETH.
6. Attacker redeems or sells rsETH, extracting ~270 ETH of value from existing rsETH holders.
7. Protocol is left holding 1000 rETH worth 800 ETH against rsETH liabilities of 1070 ETH — insolvent.

    function getRsETHAmountToMint(
        address asset,
        uint256 amount
    )
        public
        view
        override
        returns (uint256 rsethAmountToMint)
    {
        // setup oracle contract
        address lrtOracleAddress = lrtConfig.getContract(LRTConstants.LRT_ORACLE);
        ILRTOracle lrtOracle = ILRTOracle(lrtOracleAddress);

        // calculate rseth amount to mint based on asset amount and asset exchange rate
        rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
    }
