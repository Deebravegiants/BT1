### Title
Dust rsETH Amount Causes Zero-Asset Return in `instantWithdrawal` Due to Division Truncation, Permanently Burning User Funds — (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary

`instantWithdrawal` in `LRTWithdrawalManager` accepts any non-zero rsETH amount (when `minRsEthAmountToWithdraw` is unset, i.e., 0). For dust amounts where `rsETHPrice < assetPrice`, the integer division in `getExpectedAssetAmount` truncates to zero. The function then burns the user's rsETH and transfers zero assets back — a permanent, unrecoverable loss of user funds.

---

### Finding Description

`getExpectedAssetAmount` computes the asset payout as:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
``` [1](#0-0) 

When `amount` is a dust value (e.g., 1 wei) and `rsETHPrice < assetPrice` (e.g., rsETH = 1.00e18, stETH oracle = 1.05e18), the numerator `1 * 1.00e18 = 1e18` is less than the denominator `1.05e18`, so Solidity integer division yields `underlyingToReceive = 0`.

The only guard in `instantWithdrawal` is:

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
``` [2](#0-1) 

`minRsEthAmountToWithdraw[asset]` is a mapping that defaults to `0`, so the only enforced minimum is `rsETHUnstaked != 0`. A value of `1` passes. [3](#0-2) 

After computing `assetAmountUnlocked = 0`, the function immediately and irrevocably burns the user's rsETH:

```solidity
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
``` [4](#0-3) 

Then redeems 0 from the vault, computes `fee = 0`, `userAmount = 0`, and transfers 0 to the user:

```solidity
unstakingVault.redeem(asset, assetAmountUnlocked);   // redeem(asset, 0) — no-op
uint256 fee = (assetAmountUnlocked * instantWithdrawalFee) / 10_000;  // 0
uint256 userAmount = assetAmountUnlocked - fee;  // 0
_transferAsset(asset, msg.sender, userAmount);   // transfer 0
``` [5](#0-4) 

The same truncation-to-zero path exists in `initiateWithdrawal`: `expectedAssetAmount = 0` is stored in the withdrawal request, and during `unlockQueue`, `_calculatePayoutAmount` also returns 0 (same formula), so rsETH is burned via `burnFrom` with 0 assets redeemed from the vault and 0 transferred to the user on `completeWithdrawal`. [6](#0-5) [7](#0-6) 

---

### Impact Explanation

**Critical — Permanent loss of user funds.**

The user's rsETH is burned (destroyed) and they receive zero underlying assets. There is no recovery path: the burn is irreversible, and the contract emits `AssetWithdrawalFinalized` with `amountReceived = 0`, giving no on-chain indication of the error. The lost value equals the market value of the burned rsETH.

---

### Likelihood Explanation

**Low-Medium.**

The condition requires:
1. `minRsEthAmountToWithdraw[asset]` is `0` — this is the **default state** of the mapping; no admin action is needed to reach it.
2. `rsETHPrice < assetPrice` at the time of the call — possible when a supported LST (e.g., stETH) trades at a premium to rsETH's current exchange rate, or during early protocol life when rsETH price has not yet exceeded the LST oracle price.
3. The user submits a dust amount — could happen via a UI bug, a script, or a user testing the minimum.

Condition 1 is always satisfied unless the admin proactively sets a minimum. Conditions 2 and 3 are realistic but not constant.

---

### Recommendation

Add a zero-output guard immediately after computing the expected asset amount in both `instantWithdrawal` and `initiateWithdrawal`:

```solidity
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
if (assetAmountUnlocked == 0) revert InvalidAmountToWithdraw();
```

Additionally, enforce a non-zero `minRsEthAmountToWithdraw` for every supported asset at configuration time, rather than relying on the default-zero mapping value.

---

### Proof of Concept

**Setup:** rsETH oracle price = `1.00e18`, stETH oracle price = `1.05e18`. `minRsEthAmountToWithdraw[stETH] = 0` (default). Instant withdrawal is enabled for stETH.

1. Attacker (or victim) holds `1` wei of rsETH.
2. Calls `instantWithdrawal(stETH, 1, "")`.
3. Check passes: `1 != 0` and `1 >= 0`.
4. `getExpectedAssetAmount(stETH, 1)` → `1 * 1e18 / 1.05e18 = 0`.
5. `burnFrom(msg.sender, 1)` — 1 wei rsETH permanently destroyed.
6. `redeem(stETH, 0)` — no-op.
7. `fee = 0`, `userAmount = 0`.
8. `_transferAsset(stETH, msg.sender, 0)` — 0 stETH transferred.
9. Transaction succeeds. User has lost 1 wei rsETH and received nothing.

The same scenario applies to `initiateWithdrawal` + `completeWithdrawal` with the same price conditions, where the rsETH is burned during `unlockQueue` and the user receives 0 on completion. [8](#0-7) [9](#0-8)
