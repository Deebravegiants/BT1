### Title
Withdrawal `unlockTime` Stored as Absolute Timestamp Prevents Immediate Claims After Delay Reduction - (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary
`KernelDepositPool` stores the absolute `unlockTime` (`block.timestamp + withdrawalDelay`) at the moment `initiateWithdrawal` is called. When the admin later reduces `withdrawalDelay` via `setWithdrawalDelay`, all previously created withdrawal records retain their original, longer `unlockTime`. Users who initiated withdrawals before the reduction cannot claim their KERNEL tokens until the original deadline passes, even though the protocol's current policy would allow earlier access.

### Finding Description
In `KernelDepositPool`, the `Withdrawal` struct stores an absolute `unlockTime`:

```solidity
struct Withdrawal {
    address user;
    uint256 amount;
    uint256 unlockTime;   // absolute timestamp baked in at initiation
    bool claimed;
    uint256 withdrawalId;
}
``` [1](#0-0) 

When a user calls `initiateWithdrawal`, the unlock time is computed once and stored permanently:

```solidity
uint256 unlockTime = block.timestamp + withdrawalDelay;
withdrawals[withdrawalId] = Withdrawal({
    user: msg.sender, amount: _amount, unlockTime: unlockTime, ...
});
``` [2](#0-1) 

`claimWithdrawal` enforces this stored value directly:

```solidity
if (block.timestamp < withdrawal.unlockTime) {
    revert WithdrawalNotReady();
}
``` [3](#0-2) 

The admin can reduce `withdrawalDelay` at any time (down to 1 second) via `setWithdrawalDelay`:

```solidity
function setWithdrawalDelay(uint256 _withdrawalDelay) external onlyRole(DEFAULT_ADMIN_ROLE) {
    if (_withdrawalDelay == 0) revert InvalidWithdrawalDelay();
    if (_withdrawalDelay > MAX_WITHDRAWAL_DELAY) revert MaximumWithdrawalDelayExceeded();
    withdrawalDelay = _withdrawalDelay;
    ...
}
``` [4](#0-3) 

Because `setWithdrawalDelay` does not retroactively update existing `Withdrawal` records, users who initiated withdrawals under the old (longer) delay are stuck waiting for their original `unlockTime`, while users who initiate new withdrawals after the reduction benefit from the shorter delay immediately.

### Impact Explanation
**Medium — Temporary freezing of funds.**

Users who called `initiateWithdrawal` before a delay reduction cannot access their KERNEL tokens until the original `unlockTime` expires. In an emergency scenario where the admin reduces the delay to allow rapid exits (e.g., from 30 days to 1 day), affected users are frozen for up to 30 days while new withdrawers can claim within 1 day. The KERNEL tokens are not lost, but access is temporarily denied contrary to the protocol's updated policy.

### Likelihood Explanation
The admin legitimately calls `setWithdrawalDelay` to adjust protocol parameters — this is an expected operational action, not an attack. Any time the delay is reduced (including emergency reductions), all users with pending withdrawal records are silently disadvantaged. Given that `MAX_WITHDRAWAL_DELAY` is 30 days and the delay can be reduced to 1 second, the maximum freeze extension is nearly 30 days. The likelihood is medium: it requires an admin delay reduction, which is a normal governance action.

### Recommendation
Store the withdrawal initiation timestamp (`withdrawalStart`) instead of the absolute `unlockTime`. Compute the effective unlock time dynamically in `claimWithdrawal` using the current `withdrawalDelay`:

```solidity
struct Withdrawal {
    address user;
    uint256 amount;
    uint256 withdrawalStart;  // timestamp when withdrawal was initiated
    bool claimed;
    uint256 withdrawalId;
}
```

In `initiateWithdrawal`:
```solidity
uint256 withdrawalStart = block.timestamp;
withdrawals[withdrawalId] = Withdrawal({
    user: msg.sender, amount: _amount, withdrawalStart: withdrawalStart, ...
});
```

In `claimWithdrawal`:
```solidity
if (block.timestamp < withdrawal.withdrawalStart + withdrawalDelay) {
    revert WithdrawalNotReady();
}
```

This ensures that any change to `withdrawalDelay` is immediately reflected for all pending withdrawals.

### Proof of Concept

```solidity
function testWithdrawalDelayReduction() public {
    // Admin sets withdrawal delay to 30 days
    vm.prank(admin);
    kernelDepositPool.setWithdrawalDelay(30 days);

    // Alice stakes and initiates withdrawal
    uint256 amount = 100e18;
    vm.startPrank(alice);
    kernelToken.approve(address(kernelDepositPool), amount);
    kernelDepositPool.stake(amount);
    kernelDepositPool.initiateWithdrawal(amount); // unlockTime = now + 30 days
    vm.stopPrank();

    // Admin reduces delay to 1 day (emergency or policy change)
    vm.prank(admin);
    kernelDepositPool.setWithdrawalDelay(1 days);

    // Bob stakes and initiates withdrawal under new delay
    vm.startPrank(bob);
    kernelToken.approve(address(kernelDepositPool), amount);
    kernelDepositPool.stake(amount);
    kernelDepositPool.initiateWithdrawal(amount); // unlockTime = now + 1 day
    vm.stopPrank();

    // Advance 1 day
    vm.warp(block.timestamp + 1 days + 1);

    // Bob can claim immediately
    vm.prank(bob);
    kernelDepositPool.claimWithdrawal(2); // succeeds

    // Alice is still locked for ~29 more days despite the new policy
    vm.prank(alice);
    vm.expectRevert(KernelDepositPool.WithdrawalNotReady.selector);
    kernelDepositPool.claimWithdrawal(1); // reverts — temporary freeze
}
```
