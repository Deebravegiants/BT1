### Title
No Chainlink Price Staleness Check in `ChainlinkPriceOracle` Allows Stale Prices to Drive rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls Chainlink's `latestRoundData()` but discards all return values except `price`, performing no staleness validation. Stale LST/ETH prices flow directly into `LRTOracle._updateRsETHPrice()` and `LRTDepositPool.getRsETHAmountToMint()`, causing incorrect rsETH minting amounts for depositors and incorrect `rsETHPrice` storage updates.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the Chainlink price as:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

The `updatedAt` timestamp and `answeredInRound` fields are silently discarded. No check of the form `updatedAt > block.timestamp - heartbeat` or `answeredInRound >= roundId` is performed. [1](#0-0) 

This is in direct contrast to `ChainlinkOracleForRSETHPoolCollateral.getRate()` in the same codebase, which explicitly reverts on `answeredInRound < roundID` and `timestamp == 0`: [2](#0-1) 

`ChainlinkPriceOracle.getAssetPrice()` is consumed by `LRTOracle.getAssetPrice()`: [3](#0-2) 

Which is called inside `_getTotalEthInProtocol()` to compute the total ETH value of all supported LSTs: [4](#0-3) 

That result drives `_updateRsETHPrice()`, which stores the new `rsETHPrice`: [5](#0-4) 

`rsETHPrice` is then used in `LRTDepositPool.getRsETHAmountToMint()` to determine how many rsETH tokens each depositor receives: [6](#0-5) 

---

### Impact Explanation

**Impact: High — Theft of unclaimed yield / share mis-accounting**

When a Chainlink feed for a supported LST (e.g., stETH, cbETH, rETH) goes stale with a price lower than the true market price:

1. `_getTotalEthInProtocol()` undervalues the protocol TVL.
2. `newRsETHPrice` is computed lower than it should be.
3. `rsETHPrice` is stored at the deflated value.
4. Subsequent depositors calling `depositAsset()` or `depositETH()` receive more rsETH than they are entitled to (numerator `getAssetPrice(asset)` is also stale-low, but the denominator `rsETHPrice` is stale-low by a larger factor across all assets).
5. This dilutes the share value of all existing rsETH holders — a direct theft of their accrued yield.

Conversely, a stale-high price causes depositors to receive fewer rsETH tokens than they are owed, causing them to lose value on deposit. [7](#0-6) 

---

### Likelihood Explanation

- `updateRSETHPrice()` is publicly callable with no access control — any external actor can trigger a price update at any time.
- Chainlink feeds go stale during network congestion, oracle downtime, or when the deviation threshold is not crossed for extended periods.
- The protocol supports multiple LSTs (stETH, cbETH, rETH, sfrxETH, swETH, ETHx), each with its own Chainlink feed — any single feed going stale is sufficient to trigger the issue.
- No heartbeat or `answeredInRound` check exists anywhere in `ChainlinkPriceOracle`, making this a persistent, unconditional gap. [8](#0-7) 

---

### Recommendation

Add staleness validation in `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    // Optionally: if (block.timestamp - updatedAt > HEARTBEAT) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

---

### Proof of Concept

1. Chainlink's stETH/ETH feed goes stale (e.g., `updatedAt` is 4 hours old, `answeredInRound < roundId`).
2. Attacker (or anyone) calls `LRTOracle.updateRSETHPrice()`.
3. `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns stale low price, no revert.
4. `newRsETHPrice` is computed below true value; `rsETHPrice` is stored at the deflated rate.
5. Attacker calls `LRTDepositPool.depositAsset(stETH, largeAmount, 0, "")`.
6. `getRsETHAmountToMint()` computes `(largeAmount * stalePrice) / staleLowRsETHPrice` → mints excess rsETH.
7. Attacker holds inflated rsETH position; existing holders' share value is permanently diluted. [1](#0-0) [9](#0-8)
