### Title
Token Deposits Bypass Protocol Fees Due to Uninitialized `tokenFeeBps` in `addSupportedToken` - (File: contracts/pools/RSETHPool.sol)

### Summary
In `RSETHPool.sol`, the `addSupportedToken` function adds a new supported token without initializing its per-token fee rate (`tokenFeeBps`). Since Solidity mapping values default to 0, any user can deposit the newly added token with zero fee until the admin separately calls `setTokenFeeBps`. This is structurally analogous to the `initiateRepay` vulnerability where `msg.value` is used as the repayment amount without deducting fees — in both cases the user receives the full benefit of their input without the protocol collecting its entitled fee.

### Finding Description
`RSETHPool.sol` maintains a per-token fee mapping `tokenFeeBps` (line 88). When a new token is added via `addSupportedToken` (restricted to `TIMELOCK_ROLE`), this mapping is never initialized for the new token, leaving it at the Solidity default of 0. [1](#0-0) 

The `viewSwapRsETHAmountAndFee(amount, token)` function reads `tokenFeeBps[token]` directly: [2](#0-1) 

With `feeBpsForToken = 0`, `fee = 0` and the user receives rsETH calculated on the full deposit amount — no fee is deducted. The `feeEarnedInToken[token]` counter is never incremented.

The fee can only be set via `setTokenFeeBps`, which requires `DEFAULT_ADMIN_ROLE`: [3](#0-2) 

This is a separate role from `TIMELOCK_ROLE` that adds tokens: [4](#0-3) 

The structural separation of roles guarantees a multi-step gap between token activation and fee enforcement — `addSupportedToken` never sets `tokenFeeBps[token]`, and there is no atomic path to do both in one transaction.

The token deposit function that any user can call: [5](#0-4) 

### Impact Explanation
Every token deposit made between `addSupportedToken` and `setTokenFeeBps` pays zero protocol fee. The protocol permanently loses the fee revenue on those deposits. This constitutes **theft of unclaimed yield (High impact)** — the protocol fails to collect fees it is entitled to, and those fees are effectively transferred to depositors as excess rsETH.

### Likelihood Explanation
**Medium.** Adding a new supported token is a normal, recurring protocol operation. The structural separation of roles (`TIMELOCK_ROLE` for `addSupportedToken`, `DEFAULT_ADMIN_ROLE` for `setTokenFeeBps`) means the fee cannot be set atomically with token addition. Any user monitoring on-chain events (the `AddSupportedToken` event) can immediately exploit the zero-fee window before the admin sets the fee in a subsequent transaction.

### Recommendation
Add a `_feeBps` parameter to `addSupportedToken` and set `tokenFeeBps[token] = _feeBps` within the same function body, eliminating the zero-fee window entirely. Validate `_feeBps <= 10_000` consistent with the existing check in `setTokenFeeBps`.

### Proof of Concept
1. `TIMELOCK_ROLE` calls `addSupportedToken(wstETH, oracle, bridge)` — `tokenFeeBps[wstETH]` remains 0 (unset).
2. Attacker calls `deposit(wstETH, 100e18, "")` — `viewSwapRsETHAmountAndFee` computes `fee = 100e18 * 0 / 10_000 = 0`; attacker receives rsETH for the full 100 wstETH with no fee deduction.
3. `DEFAULT_ADMIN_ROLE` calls `setTokenFeeBps(wstETH, 50)` — too late; all prior deposits in the window paid zero fee.
4. Protocol permanently lost fee revenue on all deposits in the window; `feeEarnedInToken[wstETH]` remains 0 for those deposits.
