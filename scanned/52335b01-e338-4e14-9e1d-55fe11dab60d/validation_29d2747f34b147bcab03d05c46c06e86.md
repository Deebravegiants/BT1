### Title
Excess ETH Permanently Stuck in KernelVaultETH Due to Lax Native Fee Check - (File: contracts/KERNEL/KernelVaultETH.sol)

---

### Summary
`KernelVaultETH.bridgeKernelToBSC` accepts any `msg.value >= nativeFee` but only forwards exactly `nativeFee` to the LayerZero OFT adapter. The excess ETH (`msg.value - nativeFee`) is permanently locked in `KernelVaultETH` because the contract has no ETH recovery mechanism.

---

### Finding Description

In `contracts/KERNEL/KernelVaultETH.sol`, the `bridgeKernelToBSC` function is `payable` and enforces only a lower-bound check on the native fee:

```solidity
// Line 246
if (msg.value < nativeFee) {
    revert InsufficientNativeFee();
}
```

It then forwards only the exact `nativeFee` to the OFT adapter:

```solidity
// Line 264
kernelOftAdapter.send{ value: nativeFee }(sendParam, fee, refundAddress);
```

Any ETH sent above `nativeFee` remains in `KernelVaultETH`. The contract has no `receive()` fallback, no ETH sweep/recovery function, and no refund logic. The only payable entry point is `bridgeKernelToBSC` itself.

Compare this to every other bridge function in the codebase, which all enforce an exact equality check:

- `RSETHPool.bridgeAssets`: `if (msg.value != nativeFee) revert IncorrectNativeFee();`
- `RSETHPoolV2ExternalBridge.bridgeAssets`: `if (msg.value != nativeFee) revert IncorrectNativeFee();`
- `RSETHPoolV3ExternalBridge.bridgeAssets`: `if (msg.value != nativeFee) revert IncorrectNativeFee();`
- `L1VaultV2.bridgeRsETHToL2`: `if (msg.value != nativeFee) revert IncorrectNativeFee();`

`KernelVaultETH` is the only contract that uses `<` instead of `!=`, creating a silent ETH trap. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

---

### Impact Explanation

Any ETH sent in excess of `nativeFee` is permanently frozen inside `KernelVaultETH`. The contract inherits only `Initializable`, `AccessControlUpgradeable`, `PausableUpgradeable`, and `ReentrancyGuardUpgradeable` — none of which provide ETH recovery. There is no `receive()` function, no `withdrawETH`, and no sweep function. The stuck ETH is irrecoverable.

**Impact**: Permanent freezing of funds (operator's excess ETH). [5](#0-4) 

---

### Likelihood Explanation

The `bridgeKernelToBSC` function is called by the `OPERATOR_ROLE`. LayerZero native fees are dynamic and can fluctuate between the time `getNativeFee()` is quoted and the time the transaction is submitted. An operator may intentionally or accidentally send slightly more ETH than the quoted fee to ensure the transaction does not revert due to fee increases. The lax `<` check silently accepts this overpayment and traps the excess. This is a realistic operational scenario, not a theoretical edge case. [6](#0-5) 

---

### Recommendation

Replace the lower-bound check with an exact equality check, consistent with all other bridge functions in the codebase:

```diff
- if (msg.value < nativeFee) {
-     revert InsufficientNativeFee();
- }
+ if (msg.value != nativeFee) {
+     revert InsufficientNativeFee();
+ }
```

Alternatively, if overpayment must be tolerated, add a refund of the excess at the end of the function:

```solidity
if (msg.value > nativeFee) {
    (bool ok,) = payable(msg.sender).call{ value: msg.value - nativeFee }("");
    require(ok, "ETH refund failed");
}
``` [7](#0-6) 

---

### Proof of Concept

1. Operator calls `getNativeFee(amount, minAmount)` and receives `quotedFee = 0.01 ETH`.
2. Operator calls `bridgeKernelToBSC(amount, minAmount, 0.01 ether, refundAddress)` with `msg.value = 0.015 ether` (sending 0.005 ETH extra as buffer).
3. The check `msg.value < nativeFee` passes (`0.015 < 0.01` is false).
4. `kernelOftAdapter.send{ value: 0.01 ether }(...)` is called — only `0.01 ETH` leaves the contract.
5. The remaining `0.005 ETH` stays in `KernelVaultETH` permanently with no way to recover it. [8](#0-7)
