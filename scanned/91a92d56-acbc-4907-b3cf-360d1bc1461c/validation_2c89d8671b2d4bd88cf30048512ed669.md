### Title
Zero-Initialized Rate in `CrossChainRateReceiver` Causes Division-by-Zero DoS on All Pool Deposits Before First LZ Message - (`contracts/cross-chain/CrossChainRateReceiver.sol`)

---

### Summary
`CrossChainRateReceiver.rate` defaults to `0` at deployment. Every pool variant calls `getRate()` → `IOracle(rsETHOracle).getRate()` → `rate` (which is `0`) and then divides by it. In Solidity 0.8.x this is a checked division-by-zero panic revert, so every `deposit()` call reverts until the first LayerZero rate message is delivered.

---

### Finding Description
`CrossChainRateReceiver` declares `uint256 public rate` with no initial value, so it is `0` at construction. [1](#0-0) 

`RSETHRateReceiver`'s constructor sets `rateInfo`, `srcChainId`, `rateProvider`, and `layerZeroEndpoint`, but never sets `rate`. [2](#0-1) 

`getRate()` returns `rate` unconditionally with no zero-check. [3](#0-2) 

Every pool variant (`RSETHPoolV3`, `RSETHPoolV2`, `RSETHPool`, `RSETHPoolNoWrapper`, `RSETHPoolV2ExternalBridge`, `RSETHPoolV2NBA`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`) calls `getRate()` and then performs:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [4](#0-3) [5](#0-4) [6](#0-5) 

When `rsETHToETHrate == 0`, Solidity 0.8.x emits a `Panic(0x12)` (division by zero) and reverts the entire transaction. This affects the `deposit()` entry point in every pool. [7](#0-6) 

Note: the overflow/unbounded-minting branch of the question does **not** apply — Solidity 0.8.x checked arithmetic reverts on division by zero rather than wrapping.

---

### Impact Explanation
**Medium. Temporary freezing of funds.**

All `deposit()` calls revert with a panic until the first `lzReceive` message sets a non-zero rate. No user ETH is permanently lost (the revert returns the ETH), but the pool is completely non-functional for deposits during this window. The window ends as soon as the LZ oracle operator delivers the first rate message; it is therefore temporary, not permanent.

The question's claimed scope of "High. Theft of unclaimed yield" does **not** apply: no yield is stolen and no unbounded minting occurs.

---

### Likelihood Explanation
Moderate. The race window exists between pool deployment/configuration and the first LZ rate delivery. Any user who calls `deposit()` in that window (or if the LZ relayer is delayed) hits the revert. No attacker action is required — ordinary users trigger it naturally.

---

### Recommendation
Add a non-zero rate guard in `getRate()`:

```solidity
function getRate() external view returns (uint256) {
    require(rate != 0, "Rate not initialized");
    return rate;
}
```

Alternatively, set a reasonable initial rate in the `RSETHRateReceiver` constructor, or add a `require(rate != 0)` check inside each pool's `viewSwapRsETHAmountAndFee`.

---

### Proof of Concept

```solidity
// Foundry test (local, no mainnet)
function test_zeroRateDoS() public {
    RSETHRateReceiver receiver = new RSETHRateReceiver(
        1, address(0xBEEF), address(0xDEAD)
    );
    // rate is 0 — no lzReceive called

    RSETHPoolV3 pool = new RSETHPoolV3();
    pool.initialize(admin, bridger, address(wrsETH), 0, address(receiver), true);

    vm.deal(user, 1 ether);
    vm.prank(user);
    vm.expectRevert(); // Panic: division by zero
    pool.deposit{value: 1 ether}("ref");
}
```

Deploy `RSETHRateReceiver` without calling `lzReceive`, configure any pool variant to use it as `rsETHOracle`, call `deposit()` — the call reverts with a division-by-zero panic every time. [8](#0-7)
