Looking at the code, I need to verify the exact flow and whether the pause mechanism truly blocks the deposit path.

### Title
ETH Temporarily Frozen in L1VaultV2 When LRTDepositPool Is Paused Between `unwrapWETH` and `depositETHForL1VaultETH` - (File: contracts/L1VaultV2.sol)

---

### Summary

`L1VaultV2` implements a mandatory two-step flow to convert bridged WETH into rsETH: first `unwrapWETH()` converts WETH to native ETH, then `depositETHForL1VaultETH()` deposits that ETH into `LRTDepositPool`. If `LRTDepositPool` is paused between these two calls, the second step reverts and the ETH is stranded in the vault with no recovery path, because `L1VaultV2` has no re-wrap function and does not inherit `Recoverable`.

---

### Finding Description

**Step 1 — `unwrapWETH()` is irreversible:** [1](#0-0) 

`IWETH(WETH).withdraw(wethBalance)` converts WETH to native ETH. There is no corresponding `wrapETH()` or `recoverETH()` function in `L1VaultV2`, and the contract does not inherit `Recoverable`.

**Step 2 — `depositETHForL1VaultETH()` is blocked when paused:** [2](#0-1) 

The call to `lrtDepositPool.depositETH{value: balanceOfETH}(...)` hits the `whenNotPaused` guard in `LRTDepositPool`: [3](#0-2) 

**Pause is callable by any `PAUSER_ROLE` holder, independently of the L1Vault manager:** [4](#0-3) 

`PAUSER_ROLE` is a distinct role from `L1VaultV2`'s `MANAGER_ROLE`: [5](#0-4) 

**No ETH rescue path in `L1VaultV2`:**

`L1VaultV2` does not inherit `Recoverable` (which provides `recoverETH()`). The only ETH egress is `depositETHForL1VaultETH()`, which itself requires the pool to be unpaused. The vault's `receive()` function only accepts ETH in: [6](#0-5) 

---

### Impact Explanation

ETH that has been unwrapped from WETH sits in `L1VaultV2` with no exit path until `LRTDepositPool` is unpaused. The vault holds bridged user funds intended for rsETH minting; any delay in minting delays the corresponding rsETH bridge back to L2 users. This is **temporary freezing of funds** (Medium).

---

### Likelihood Explanation

`LRTDepositPool` pauses are a normal operational event (security incidents, oracle anomalies, governance actions). The two manager calls are separate transactions, so any pause event occurring in the block window between them triggers the freeze. Pauses can last hours to days. The `PAUSER_ROLE` is held by multiple actors (`LRTConfig.pauseAll` also triggers it), making the window realistic. [7](#0-6) 

---

### Recommendation

Combine the unwrap and deposit into a single atomic function, or add a `wrapETH()` recovery function (calling `IWETH(WETH).deposit{value: ...}()`) so the manager can reverse the unwrap if the deposit is blocked. Alternatively, have `depositETHForL1VaultETH()` check the pause state before unwrapping, or inherit `Recoverable` to allow admin ETH rescue.

---

### Proof of Concept

```solidity
// Fork test (Mainnet fork, unmodified contracts)
function test_ethFrozenOnPausedPool() public {
    // 1. Simulate WETH arriving at vault (e.g., from bridge)
    deal(address(WETH), address(vault), 10 ether);

    // 2. Manager unwraps WETH → ETH
    vm.prank(manager);
    vault.unwrapWETH();
    assertEq(address(vault).balance, 10 ether);

    // 3. Any PAUSER_ROLE holder pauses LRTDepositPool
    vm.prank(pauser);
    lrtDepositPool.pause();
    assertTrue(lrtDepositPool.paused());

    // 4. Manager attempts to deposit ETH → reverts
    vm.prank(manager);
    vm.expectRevert("Pausable: paused");
    vault.depositETHForL1VaultETH();

    // 5. ETH is stuck in vault
    assertGt(address(vault).balance, 0);
}
```

The revert at step 4 is confirmed by `whenNotPaused` on `LRTDepositPool.depositETH` and the absence of any ETH recovery path in `L1VaultV2`. [8](#0-7)
