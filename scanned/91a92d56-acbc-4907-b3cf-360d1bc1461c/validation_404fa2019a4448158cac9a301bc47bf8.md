### Title
Missing Zero-Address Validation in `updateRateReceiver` Allows Silent Rate Broadcast Failure — (`contracts/cross-chain/CrossChainRateProvider.sol`)

---

### Summary

`updateRateReceiver` accepts `address(0)` without reversion. If set, every subsequent `updateRate()` call updates the provider's local state (`rate`, `lastUpdated`) and emits `RateUpdated`, while the LayerZero message is dispatched to `address(0)` on the destination chain — permanently silencing all rate propagation to the real receiver.

---

### Finding Description

`updateRateReceiver` performs no zero-address check: [1](#0-0) 

`updateRate()` then directly encodes `rateReceiver` as the remote destination: [2](#0-1) 

The provider's `rate` and `lastUpdated` are written unconditionally before the LZ send: [3](#0-2) 

The LZ `send()` call then fires with `address(0)` packed as the remote address: [4](#0-3) 

There is no post-send verification. The transaction succeeds, `RateUpdated` is emitted, but the destination `CrossChainRateReceiver` never receives the message and its `rate` / `lastUpdated` remain permanently stale.

---

### Impact Explanation

The provider contract reports a fresh, up-to-date rate locally, but the cross-chain receiver — which downstream integrations (AMMs, lending protocols, yield vaults) query via `getRate()` — is frozen at its last valid value. This is a silent, undetectable divergence: no revert, no error event, no on-chain signal that delivery failed. The contract fails to deliver its core promised return (cross-chain rate propagation) without any loss of funds.

**Scoped impact:** Low — Contract fails to deliver promised returns, but doesn't lose value.

---

### Likelihood Explanation

The precondition is an owner misconfiguration (passing `address(0)` to `updateRateReceiver`), which is an honest operational mistake enabled by the absence of input validation. It does not require a malicious owner or key compromise. The same pattern (`updateLayerZeroEndpoint` at line 57) also lacks a zero-address guard, showing this is a systemic omission rather than an isolated oversight. [5](#0-4) 

---

### Recommendation

Add a zero-address guard to `updateRateReceiver` (and `updateLayerZeroEndpoint`):

```solidity
function updateRateReceiver(address _rateReceiver) external onlyOwner {
    require(_rateReceiver != address(0), "Zero address");
    rateReceiver = _rateReceiver;
    emit RateReceiverUpdated(_rateReceiver);
}
```

Optionally, add a guard in `updateRate()` itself:

```solidity
require(rateReceiver != address(0), "Receiver not set");
```

```solidity
function test_updateRate_silentFailureOnZeroReceiver() public {
    // 1. Owner sets rateReceiver to address(0)

    provider.updateRateReceiver(address(0));

    // 2. Any caller invokes updateRate with sufficient fee
    vm.deal(caller, 1 ether);
    vm.prank(caller);
    provider.updateRate{value: 0.01 ether}();

    // 3. Provider local state updated — appears successful
    assertGt(provider.rate(), 0);
    assertEq(provider.lastUpdated(), block.timestamp);

    // 4. Mock LZ endpoint received destination == address(0)
    assertEq(mockLZ.lastRemoteAddress(), address(0)); // message sent to zero

    // 5. Actual receiver rate unchanged — delivery silently dropped
    assertEq(receiver.rate(), 0); // never updated
}
```

The mock LZ endpoint should accept the `send()` call without reverting (consistent with LZ v1 endpoint behavior, which does not validate the remote address bytes). The test confirms the invariant break: `updateRate()` succeeds while the receiver's rate is never updated.
