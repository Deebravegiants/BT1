### Title
ETH Deposit Limit Bypass Due to Missing Incoming Amount in Validation Check - (File: contracts/LRTDepositPool.sol)

### Summary

`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric validation for ETH vs. ERC20 tokens. For ERC20 assets the incoming deposit amount is included in the limit comparison, but for ETH it is omitted. This allows any depositor to push total ETH deposits past the protocol-enforced `depositLimitByAsset` cap in a single call.

### Finding Description

`_checkIfDepositAmountExceedesCurrentLimit` is the sole gate that enforces the per-asset deposit cap before rsETH is minted:

```solidity
// contracts/LRTDepositPool.sol  L676-L682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ŌåÉ amount NOT included
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ŌåÉ amount included
}
```

For ERC20 assets the check is `totalAssetDeposits + amount > limit`, which correctly rejects a deposit that would breach the cap. For ETH the check is `totalAssetDeposits > limit`, which only rejects deposits when the cap has **already been exceeded**. When `totalAssetDeposits == limit` the function returns `false` (not exceeded), the deposit proceeds, and `totalAssetDeposits` becomes `limit + msg.value`.

The analog to M-7 is direct: in M-7 a value passes validation on its pre-truncation form but the stored (truncated) form violates the protocol-enforced ratio. Here, the incoming ETH amount passes validation because the check omits it, but after the deposit the protocol-enforced cap is violated. In both cases a constraint that is supposed to be enforced at write-time is silently bypassed. [1](#0-0) 

The function is called unconditionally from `_beforeDeposit`, which is the only pre-condition path for both `depositETH` and `depositAsset`: [2](#0-1) 

`depositETH` is a public, payable, non-privileged entry point: [3](#0-2) 

The deposit limit is set by the MANAGER role via `LRTConfig.updateAssetDepositLimit`: [4](#0-3) 

### Impact Explanation

Any unprivileged depositor can mint rsETH backed by ETH in excess of the protocol-enforced cap. The deposit limit exists to bound EigenLayer exposure; breaching it means the protocol holds more ETH in restaking strategies than governance has approved. The bypass is bounded to a single deposit's `msg.value`, but there is no minimum deposit size enforced beyond `minAmountToDeposit`, so a depositor can send an arbitrarily large ETH amount in one transaction. After the bypass the cap is exceeded and subsequent deposits are blocked, but the over-deposited ETH has already been accepted and rsETH minted against it.

Impact classification: **Low ŌĆö contract fails to deliver promised returns** (the deposit cap guarantee is not upheld for ETH).

### Likelihood Explanation

The condition `totalAssetDeposits == depositLimitByAsset(ETH)` is a normal operational state ŌĆö the cap is routinely reached as the protocol fills. Any depositor who calls `depositETH` at that moment triggers the bypass with no special knowledge or privilege. Likelihood is **high** given the frequency with which the limit is at capacity.

### Recommendation

Include the incoming amount in the ETH branch, mirroring the ERC20 branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept

1. Admin sets `depositLimitByAsset[ETH] = 1000 ether`.
2. Users deposit until `getTotalAssetDeposits(ETH) == 1000 ether`.
3. Attacker calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `1000 ether > 1000 ether` ŌåÆ `false` ŌåÆ deposit is not blocked.
5. `_mintRsETH` mints rsETH for 500 ETH; `getTotalAssetDeposits(ETH)` is now `1500 ether`, 50 % above the cap.
6. The same call with an ERC20 asset at the same state would evaluate `1000 ether + 500 ether > 1000 ether` ŌåÆ `true` ŌåÆ reverts with `MaximumDepositLimitReached`. [1](#0-0) [5](#0-4)
