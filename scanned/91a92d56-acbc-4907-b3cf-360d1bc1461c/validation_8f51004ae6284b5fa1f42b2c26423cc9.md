### Title
Missing Storage Gap in `LRTConfigRoleChecker` Base Contract Risks Storage Collision on Upgrade - (File: contracts/utils/LRTConfigRoleChecker.sol)

### Summary
`LRTConfigRoleChecker` is an abstract upgradeable base contract inherited by every core protocol contract (`NodeDelegator`, `LRTDepositPool`, `LRTWithdrawalManager`, `LRTUnstakingVault`, `LRTConverter`, `LRTOracle`, `RSETH`). It declares the `lrtConfig` state variable but contains no `__gap` reserve. If a future upgrade appends a new state variable to `LRTConfigRoleChecker`, the storage layout of every inheriting proxy will silently shift, corrupting critical state variables across the entire protocol. The same pattern exists in `UnstakeStETH`, which the developers themselves annotated with an explicit warning about this exact risk.

### Finding Description
`LRTConfigRoleChecker` is an abstract contract at the root of the inheritance chain for all major protocol contracts. It declares one storage variable with no gap:

```solidity
abstract contract LRTConfigRoleChecker {
    ILRTConfig public lrtConfig;   // slot 0 — no __gap follows
    ...
}
``` [1](#0-0) 

Because no `uint256[N] private __gap` is present, the storage layout of every inheriting proxy is tightly packed immediately after `lrtConfig`. For example, `NodeDelegator` lays out:

- Slot 0: `lrtConfig` (from `LRTConfigRoleChecker`)
- Slot 1: `eigenPod` (NodeDelegator)
- Slot 2: `stakedButUnverifiedNativeETH` (NodeDelegator)
- Slot 3: `__elOperatorDelegatedTo` (NodeDelegator)
- Slot 4: `__legacyExtraStakeToReceive` (NodeDelegator)
- Slot 5: `lastNonce` (NodeDelegator) [2](#0-1) 

If a future upgrade adds even one new variable to `LRTConfigRoleChecker`, every inheriting proxy's storage layout shifts by one slot. The existing on-chain storage values remain at their old slots, so after the upgrade all child-contract variables read from the wrong slots.

The same issue exists in `UnstakeStETH`, which is inherited by `LRTConverter` and carries an explicit developer acknowledgement:

```solidity
/// @dev Upgradeable base contract without storage gaps; adding variables in upgrades can collide with inheritors.
abstract contract UnstakeStETH is Initializable {
    ILidoWithdrawalQueue public withdrawalQueue;
    IERC20 public stETH;
``` [3](#0-2) 

The protocol has already performed multiple upgrades (`initialize2`, `initialize3`, `reinitialize` across `NodeDelegator`, `LRTWithdrawalManager`, `LRTConverter`, `LRTOracle`, `RSETH`, `L1Vault`, `L1VaultV2`), confirming that base-contract evolution is an active practice. [4](#0-3) [5](#0-4) 

### Impact Explanation
`lrtConfig` is the single source of truth for all contract addresses, roles, and asset configurations. Its corruption in every inheriting proxy simultaneously would:

- Cause all role-check modifiers (`onlyLRTManager`, `onlyLRTAdmin`, `onlyLRTOperator`) to revert or pass incorrectly, because they call `IAccessControl(address(lrtConfig)).hasRole(...)`.
- Cause all asset and contract lookups (`lrtConfig.depositPool()`, `lrtConfig.rsETH()`, `lrtConfig.strategyManager()`, etc.) to return wrong or zero addresses.
- Effectively freeze all deposits, withdrawals, and unstaking operations across the protocol.

This maps to **Medium: Temporary freezing of funds** (all user funds locked until a corrective upgrade) with potential escalation to **Critical: Permanent freezing of funds** if the corrupted `lrtConfig` pointer cannot be recovered through a subsequent upgrade. [6](#0-5) 

### Likelihood Explanation
The protocol has a demonstrated pattern of upgrading its base contracts. The absence of `__gap` in `LRTConfigRoleChecker` means any future upgrade that adds a variable to this base contract will trigger the collision across all six inheriting proxies simultaneously. The developer comment in `UnstakeStETH.sol` confirms the team is aware of this pattern in at least one location but has not addressed it in `LRTConfigRoleChecker`. Likelihood is **Low** (requires a future upgrade action that adds a variable to the base contract), but the blast radius and consequence are severe.

### Recommendation
Add a storage gap to `LRTConfigRoleChecker` and all other abstract upgradeable base contracts in the inheritance chain:

```solidity
abstract contract LRTConfigRoleChecker {
    ILRTConfig public lrtConfig;

    // ... existing code ...

    uint256[49] private __gap; // 50 slots total (1 used by lrtConfig)
}
```

Similarly add `__gap` to `UnstakeStETH`:

```solidity
abstract contract UnstakeStETH is Initializable {
    ILidoWithdrawalQueue public withdrawalQueue;
    IERC20 public stETH;

    // ... existing code ...

    uint256[48] private __gap; // 50 slots total (2 used)
}
```

### Proof of Concept
**Current storage layout for `NodeDelegator` proxy:**

| Slot | Variable | Contract |
|------|----------|----------|
| 0 | `lrtConfig` | LRTConfigRoleChecker |
| 1 | `eigenPod` | NodeDelegator |
| 2 | `stakedButUnverifiedNativeETH` | NodeDelegator |
| 3 | `__elOperatorDelegatedTo` | NodeDelegator |

**After upgrade that adds `newVar` to `LRTConfigRoleChecker`:**

| Slot | Expected variable | Actual on-chain value |
|------|------------------|-----------------------|
| 0 | `lrtConfig` | old `lrtConfig` ✓ |
| 1 | `newVar` | old `eigenPod` address ✗ |
| 2 | `eigenPod` | old `stakedButUnverifiedNativeETH` value ✗ |
| 3 | `stakedButUnverifiedNativeETH` | old `__elOperatorDelegatedTo` value ✗ |

All `eigenPod` calls in `NodeDelegator` would return a corrupted address, breaking `verifyWithdrawalCredentials`, `startCheckpoint`, `stake32Eth`, and `completeUnstaking`, freezing all staked ETH. The same shift propagates identically to `LRTDepositPool`, `LRTWithdrawalManager`, `LRTUnstakingVault`, `LRTConverter`, `LRTOracle`, and `RSETH`. [7](#0-6) [8](#0-7) [9](#0-8)
