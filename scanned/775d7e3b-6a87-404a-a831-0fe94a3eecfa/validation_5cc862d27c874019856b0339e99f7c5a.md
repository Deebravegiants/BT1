### Title
Missing `assetsCommitted` Guard in `transferETHToNodeDelegator` Enables Temporary Freezing of Queued ETH Withdrawals — (`contracts/LRTUnstakingVault.sol`)

### Summary

`LRTUnstakingVault.transferETHToNodeDelegator` imposes no constraint against `LRTWithdrawalManager.assetsCommitted[ETH_TOKEN]` or `queuedWithdrawalsBuffer[ETH_TOKEN]`. An `ASSET_TRANSFER_ROLE` caller can drain the entire vault ETH balance to a NodeDelegator even while user ETH withdrawals are queued and committed, causing every subsequent `unlockQueue` call to revert and temporarily freezing all pending ETH withdrawal completions.

### Finding Description

**Root cause — `LRTUnstakingVault.transferETHToNodeDelegator` (lines 132–146):**

```solidity
function transferETHToNodeDelegator(
    uint256 ndcIndex,
    uint256 amount
)
    external
    nonReentrant
    onlyAssetTransferRole
    onlySupportedAsset(LRTConstants.ETH_TOKEN)
{
    ...
    INodeDelegator(nodeDelegator).sendETHFromUnstakingVaultToNDC{ value: amount }();
    emit EthTransferred(nodeDelegator, amount);
}
```

The only guards are `onlyAssetTransferRole` and `onlySupportedAsset`. There is no check against:
- `LRTWithdrawalManager.assetsCommitted[ETH_TOKEN]` — the ETH already promised to queued withdrawal requests
- `queuedWithdrawalsBuffer[ETH_TOKEN]` — the operator-set reserve for queued withdrawals [1](#0-0) 

**Contrast with `getAssetsAvailableForInstantWithdrawal`**, which correctly subtracts `queuedWithdrawalsBuffer` before reporting available ETH — but this guard is never consulted by `transferETHToNodeDelegator`: [2](#0-1) 

**Downstream failure — `LRTWithdrawalManager.unlockQueue` (lines 268–320):**

`_createUnlockParams` reads `totalAvailableAssets` directly from `unstakingVault.balanceOf(asset)`, which for ETH is `address(this).balance`: [3](#0-2) 

If the vault ETH balance is zero after the transfer, the check at line 297 reverts unconditionally:

```solidity
if (params.totalAvailableAssets == 0) revert AmountMustBeGreaterThanZero();
``` [4](#0-3) 

Even if `unlockQueue` somehow proceeded, `vault.redeem(ETH_TOKEN, amount)` would revert because the vault has no ETH to forward: [5](#0-4) 

**`assetsCommitted` is only decremented inside `_unlockWithdrawalRequests`** (line 802), which is only reached via `unlockQueue`. If `unlockQueue` is permanently blocked, `assetsCommitted` is never reduced, and `getAvailableAssetAmount` continues to report zero, blocking new queued withdrawals as well: [6](#0-5) 

### Impact Explanation

All pending ETH withdrawal requests are frozen until an operator manually returns ETH from the NodeDelegator back to the vault. During this window, no user can complete an ETH withdrawal. This matches **Medium — Temporary freezing of funds**.

### Likelihood Explanation

The `ASSET_TRANSFER_ROLE` is an operational role used routinely to move ETH from the vault to NodeDelegators for restaking. The role holder may legitimately call `transferETHToNodeDelegator` with the full vault balance without being aware of `assetsCommitted[ETH_TOKEN]`, since the contract provides no on-chain enforcement. This is an operational mistake enabled by a missing code-level guard, not a scenario requiring a malicious or compromised actor.

### Recommendation

Add a check in `transferETHToNodeDelegator` that prevents transferring ETH that is already committed to queued withdrawals:

```solidity
ILRTWithdrawalManager withdrawalManager =
    ILRTWithdrawalManager(lrtConfig.withdrawManager());
uint256 committed = withdrawalManager.assetsCommitted(LRTConstants.ETH_TOKEN);
uint256 available = address(this).balance > committed
    ? address(this).balance - committed : 0;
if (amount > available) revert InsufficientUncommittedBalance();
```

Alternatively, enforce that `amount` does not exceed `address(this).balance - queuedWithdrawalsBuffer[ETH_TOKEN]`, and ensure the buffer is always kept ≥ `assetsCommitted[ETH_TOKEN]` by updating it atomically in `initiateWithdrawal` / `_unlockWithdrawalRequests`.

### Proof of Concept

```solidity
// State before attack:
// vault.balance = 100 ETH
// assetsCommitted[ETH_TOKEN] = 80 ETH  (80 ETH promised to queued users)
// queuedWithdrawalsBuffer[ETH_TOKEN] = 0 (not set)

// Step 1: ASSET_TRANSFER_ROLE drains the vault (legitimate restaking operation)
vault.transferETHToNodeDelegator(0, 100 ether);
// vault.balance == 0

// Step 2: Operator tries to unlock the queue
// _createUnlockParams → totalAvailableAssets = vault.balanceOf(ETH) = 0
// → revert AmountMustBeGreaterThanZero()
withdrawalManager.unlockQueue(ETH_TOKEN, type(uint256).max, ...);

// Step 3: Users try to complete withdrawals
// completeWithdrawal also fails because unlockQueue never ran,
// so nextLockedNonce was never advanced → revert WithdrawalLocked()

// All 80 ETH of queued user withdrawals are frozen until ETH is
// manually returned from the NodeDelegator to the vault.
``` [1](#0-0) [7](#0-6)
