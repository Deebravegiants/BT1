### Title
Stale rsETH Exchange Rate in `CrossChainRateReceiver.getRate()` Enables Over-Minting of wrsETH After L2 Sequencer Downtime - (File: contracts/cross-chain/CrossChainRateReceiver.sol)

---

### Summary

`CrossChainRateReceiver.getRate()` returns the stored `rate` with no staleness check on `lastUpdated`. The rate is only updated when a LayerZero message arrives from L1. During L2 sequencer downtime, no new LayerZero messages are delivered, so the stored rate becomes arbitrarily stale. When the sequencer recovers, all L2 pool contracts (`RSETHPoolV3`, `RSETHPoolNoWrapper`, etc.) immediately resume using this stale rate to price deposits, allowing users to mint wrsETH at a favorable outdated rate.

---

### Finding Description

`CrossChainRateReceiver` stores the rsETH/ETH exchange rate pushed from L1 via LayerZero:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol
uint256 public rate;
uint256 public lastUpdated;

function lzReceive(...) external {
    ...
    rate = _rate;
    lastUpdated = block.timestamp;
    emit RateUpdated(_rate);
}

function getRate() external view returns (uint256) {
    return rate;  // No staleness check on lastUpdated
}
``` [1](#0-0) 

`getRate()` unconditionally returns `rate` regardless of how old `lastUpdated` is. There is no check of the form `require(block.timestamp - lastUpdated <= MAX_STALENESS)`.

All L2 pool variants consume this rate directly:

```solidity
// contracts/pools/RSETHPoolV3.sol
function getRate() public view returns (uint256) {
    return IOracle(rsETHOracle).getRate();
}
``` [2](#0-1) 

And use it to compute how many wrsETH tokens to mint:

```solidity
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [3](#0-2) 

The same pattern exists in `RSETHPoolNoWrapper`: [4](#0-3) 

Additionally, `ChainlinkOracleForRSETHPoolCollateral.getRate()`, used for collateral token pricing in the same pools, also lacks a Chainlink L2 sequencer uptime feed check and a time-based staleness bound:

```solidity
function getRate() public view returns (uint256) {
    (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();
    if (answeredInRound < roundID) revert StalePrice();
    if (timestamp == 0) revert IncompleteRound();
    // No: block.timestamp - timestamp > heartbeat check
    // No: sequencer uptime feed check
    ...
}
``` [5](#0-4)

### Impact Explanation

rsETH accrues staking rewards over time, so its ETH price (`rsETHToETHrate`) monotonically increases. During sequencer downtime (e.g., on Scroll, Optimism, Base, Linea), LayerZero messages carrying updated rates cannot be delivered. When the sequencer recovers, the stored `rate` in `CrossChainRateReceiver` is lower than the true current rate (it reflects a past, pre-reward-accrual value).

A user who deposits ETH immediately after sequencer recovery receives:

```
rsETHAmount = amountAfterFee * 1e18 / staleRate
```

Since `staleRate < trueRate`, the user receives **more wrsETH than the current fair value**, diluting existing wrsETH holders and extracting value from the protocol. The longer the downtime, the larger the discrepancy and the larger the extractable value.

**Impact: High — Theft of unclaimed yield** (existing wrsETH holders' accrued staking rewards are diluted by over-minted wrsETH).

---

### Likelihood Explanation

- All target L2 chains (Scroll, Optimism, Base, Linea, Arbitrum) use centralized sequencers with documented historical downtime events.
- LayerZero message delivery is contingent on sequencer liveness; downtime directly causes rate staleness.
- No special permissions are required — any unprivileged depositor can call `deposit()` immediately after sequencer recovery.
- The attacker only needs to monitor the sequencer status and submit a deposit transaction as soon as the sequencer resumes.

**Likelihood: Medium** — Sequencer downtime is infrequent but historically documented; exploitation requires only timing awareness, not privileged access.

---

### Recommendation

1. **Add a staleness check in `CrossChainRateReceiver.getRate()`**:
   ```solidity
   uint256 public constant MAX_RATE_STALENESS = 24 hours;

   function getRate() external view returns (uint256) {
       require(block.timestamp - lastUpdated <= MAX_RATE_STALENESS, "Rate is stale");
       return rate;
   }
   ```

2. **Add a Chainlink L2 Sequencer Uptime Feed check in `ChainlinkOracleForRSETHPoolCollateral.getRate()`** per [Chainlink's official L2 guidance](https://docs.chain.link/data-feeds/l2-sequencer-feeds), including a grace period (e.g., 1 hour) after sequencer recovery before accepting prices.

3. **Add a time-based staleness bound** in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:
   ```solidity
   require(block.timestamp - timestamp <= HEARTBEAT + BUFFER, "Stale price");
   ```

### Proof of Concept

1. rsETH rate on L1 is `1.05 ETH` per rsETH. A LayerZero message updates `CrossChainRateReceiver.rate = 1.05e18` on the L2 pool.
2. The L2 sequencer goes down for 6 hours. During this time, rsETH accrues rewards; the true rate rises to `1.06 ETH` per rsETH. No LayerZero message can be delivered.
3. The sequencer recovers. `CrossChainRateReceiver.rate` is still `1.05e18`.
4. Attacker calls `RSETHPoolV3.deposit{value: 100 ether}("")`.
5. Pool computes: `rsETHAmount = 100e18 * 1e18 / 1.05e18 ≈ 95.238 wrsETH`.
6. Fair amount at true rate: `100e18 * 1e18 / 1.06e18 ≈ 94.340 wrsETH`.
7. Attacker receives **~0.898 excess wrsETH** per 100 ETH deposited, extracted from existing holders' accrued yield.
8. The attacker can repeat this until a new LayerZero message arrives and corrects the rate.
