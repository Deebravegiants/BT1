### Title
Off-by-one in slippage guard allows `slippageTolerance = 10000`, permanently breaking `bridgeTokenToL1` until corrected — (`contracts/bridges/TACWETHBridge.sol`)

### Summary
Both the constructor and `setSlippageTolerance` use a strictly-greater-than check (`> BASIS_POINTS_DIVISOR`) instead of greater-than-or-equal (`>= BASIS_POINTS_DIVISOR`). This allows `slippageTolerance` to be set to exactly `10000` (100%). When set, `getMinAmount` computes `minAmount = 0` for every non-zero input and reverts with `InvalidMinAmount`, making `bridgeTokenToL1` uncallable until the admin corrects the value.

### Finding Description
`BASIS_POINTS_DIVISOR` is `10_000`. The intended invariant is that `slippageTolerance` must be strictly less than `10_000` so that `getMinAmount` always returns a positive value. However:

- Constructor guard: [1](#0-0) 
- Setter guard: [2](#0-1) 

Both use `>` not `>=`, so `slippageTolerance = 10000` is accepted as valid.

`getMinAmount` then computes: [3](#0-2) 

With `slippageTolerance = 10000`: `amount * (10000 - 10000) / 10000 = 0`, which triggers the `InvalidMinAmount` revert unconditionally.

`bridgeTokenToL1` calls `getNativeFee` on line 108, which internally calls `getMinAmount`, so the revert occurs before any token transfer: [4](#0-3) 

### Impact Explanation
All calls to `bridgeTokenToL1` revert with `InvalidMinAmount` for any non-zero amount. No WETH can be bridged from TAC to L1. Funds are not permanently lost (the admin can call `setSlippageTolerance` with a corrected value), but the bridge is completely frozen until that correction is made. This matches **Medium — Temporary freezing of funds**.

### Likelihood Explanation
The admin could set `slippageTolerance = 10000` accidentally (e.g., intending to express "no minimum amount enforced" or making a basis-point unit error). The contract provides no protection against this value, and the resulting breakage is non-obvious because the revert originates in `getMinAmount`, not in the setter itself. The same off-by-one exists in the constructor, so the contract can also be deployed in a broken state.

### Recommendation
Change both guards from `>` to `>=`:

```solidity
// Constructor (line 70)
if (_slippageTolerance >= BASIS_POINTS_DIVISOR) {
    revert InvalidSlippageTolerance();
}

// setSlippageTolerance (line 87)
if (newSlippageTolerance >= BASIS_POINTS_DIVISOR) {
    revert InvalidSlippageTolerance();
}
```

This ensures `slippageTolerance` is always in `[0, 9999]`, guaranteeing `getMinAmount` returns a positive value for any non-zero input.
