### Title
Unbounded Gas Consumption in `getETHDistributionData()` / `getAssetDistributionData()` Causes User Deposits and Price Updates to Become Uncallable - (File: contracts/LRTDepositPool.sol)

### Summary
Every user deposit (`depositETH`, `depositAsset`) and every public price update (`updateRSETHPrice`) traverses a nested loop: once over all NodeDelegators, and once per NDC over all queued EigenLayer withdrawals. No gas bound or pagination exists. As the protocol scales within its designed operational parameters, these critical functions can exceed the block gas limit and become permanently uncallable.

### Finding Description

**Call chain for `depositETH()`:**

`depositETH()` → `_beforeDeposit()` → `_checkIfDepositAmountExceedesCurrentLimit()` → `getTotalAssetDeposits(ETH_TOKEN)` → `getETHDistributionData()` [1](#0-0) 

`_checkIfDepositAmountExceedesCurrentLimit` calls `getTotalAssetDeposits`: [2](#0-1) 

`getTotalAssetDeposits` dispatches to `getETHDistributionData()` for ETH: [3](#0-2) 

`getETHDistributionData()` loops over every NDC in `nodeDelegatorQueue` and for each one calls `getEffectivePodShares()` (one external call) and `getAssetUnstaking(ETH_TOKEN)` (one external call + inner loop): [4](#0-3) 

`getAssetUnstaking()` in `NodeDelegator.sol` calls EigenLayer's `getQueuedWithdrawals()` and then iterates over **every queued withdrawal** and **every strategy within each withdrawal**: [5](#0-4) 

**Call chain for `updateRSETHPrice()` (public):**

`updateRSETHPrice()` → `_updateRsETHPrice()` → `_getTotalEthInProtocol()` loops over **every supported asset**, and for each asset calls `getTotalAssetDeposits()`, which in turn loops over every NDC and calls `getAssetUnstaking()`. The total cost is O(assets × NDCs × queued\_withdrawals\_per\_NDC): [6](#0-5) [7](#0-6) 

**No cap on the combined product.** `maxNodeDelegatorLimit` (initially 10, admin-raisable) bounds NDC count, and `maxUncompletedWithdrawalCount` bounds queued withdrawals per NDC, but neither bound is enforced inside the loop itself, and the product of the two multiplied by the number of supported assets (no cap) is unbounded in practice. `supportedAssetList` grows via `addNewSupportedAsset` with no maximum: [8](#0-7) 

### Impact Explanation

**Temporary freezing of funds (Medium).** When the cumulative gas cost of the nested loops exceeds the block gas limit:

- `depositETH()` and `depositAsset()` revert on every call → users cannot deposit ETH or LSTs.
- `updateRSETHPrice()` reverts → the rsETH/ETH exchange rate becomes permanently stale → all subsequent deposit minting and withdrawal calculations use an incorrect price.

Both outcomes constitute temporary (potentially indefinite) freezing of the deposit path and corruption of the price oracle relied upon by the entire protocol.

### Likelihood Explanation

**Medium.** The protocol is architecturally designed to operate with multiple NDCs (up to `maxNodeDelegatorLimit`) and multiple concurrent EigenLayer withdrawal queues per NDC (up to `maxUncompletedWithdrawalCount`). Operators routinely call `initiateUnstaking()` to queue withdrawals. With 10 NDCs each holding 50 queued withdrawals and 5 supported assets, `updateRSETHPrice()` triggers 10 × 50 × 5 = 2,500 inner iterations plus 10 × 5 = 50 external `getQueuedWithdrawals()` calls. Each external call costs thousands of gas; at scale this exceeds Ethereum's ~30M block gas limit. No attacker action is required — normal protocol growth triggers the condition.

### Recommendation

1. **Cache asset totals.** Store `totalAssetDeposits` per asset in a storage variable updated incrementally on deposit/withdrawal/unstaking events rather than recomputing it on every call.
2. **Remove `getAssetUnstaking()` from the hot deposit path.** Maintain a running `assetUnstaking` counter updated in `initiateUnstaking()` and `completeUnstaking()` instead of iterating over EigenLayer's queue on every deposit.
3. **Paginate `_getTotalEthInProtocol()`.** If on-chain enumeration is required, add a `startIndex`/`endIndex` parameter and accumulate the result across multiple transactions.
4. **Cap `supportedAssetList` length** with an explicit maximum enforced in `_addNewSupportedAsset`.

### Proof of Concept

```
Setup:
  - 5 supported assets in LRTConfig.supportedAssetList
  - maxNodeDelegatorLimit = 10, 10 NDCs registered
  - maxUncompletedWithdrawalCount = 50, each NDC has 50 queued withdrawals
  - Each queued withdrawal covers 2 strategies

Gas estimate for depositETH():
  getETHDistributionData() outer loop: 10 NDCs
    per NDC:
      getEffectivePodShares() → getWithdrawableShares() [external call ~5,000 gas]
      getAssetUnstaking(ETH) → getQueuedWithdrawals() [external call ~10,000 gas]
        inner loop: 50 withdrawals × 2 strategies = 100 iterations [~500 gas each]
  Total per depositETH: 10 × (15,000 + 50,000) = 650,000 gas (deposit path alone)

Gas estimate for updateRSETHPrice():
  _getTotalEthInProtocol() outer loop: 5 assets
    per asset: getAssetPrice() [external] + getTotalAssetDeposits() [above loop]
  Total: 5 × 650,000 + oracle calls ≈ 3,500,000+ gas

As maxNodeDelegatorLimit is raised or more withdrawals accumulate,
depositETH() and updateRSETHPrice() exceed the 30M block gas limit,
permanently bricking deposits and price updates.

        public
        view
        override
        onlySupportedAsset(asset)
        returns (
            uint256 assetLyingInDepositPool,
            uint256 assetLyingInNDCs,
            uint256 assetStakedInEigenLayer,
            uint256 assetUnstakingFromEigenLayer,

                ++i;

        assetLyingInConverter = 0; // assets in converter are accounted in their eth value => getETHDistributionData()
        assetLyingUnstakingVault = IERC20(asset).balanceOf(lrtUnstakingVault);

    function getAssetUnstaking(address asset) external view returns (uint256 amount) {
        (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
            _getDelegationManager().getQueuedWithdrawals(address(this));

        for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
            IDelegationManager.Withdrawal memory withdrawal = queuedWithdrawals[withdrawalIndex];

            for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
                IStrategy strategy = withdrawal.strategies[strategyIndex];

                address strategyAsset = address(strategy) == address(lrtConfig.beaconChainETHStrategy())
                    ? LRTConstants.ETH_TOKEN
                    : address(strategy.underlyingToken());

                if (strategyAsset != asset) continue;

                uint256 sharesToUnstake = withdrawalShares[withdrawalIndex][strategyIndex];
                amount += strategyAsset == LRTConstants.ETH_TOKEN
                    ? sharesToUnstake
                    : strategy.sharesToUnderlyingView(sharesToUnstake);
            }
        }
    }

    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
