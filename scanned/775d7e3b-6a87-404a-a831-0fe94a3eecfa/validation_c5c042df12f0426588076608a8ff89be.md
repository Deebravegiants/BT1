### Title
Stale `rsETHPrice` Read in `depositETH` Allows Depositors to Dilute Existing Holders' Yield — (`contracts/LRTDepositPool.sol`)

---

### Summary

`depositETH` mints rsETH using a **stored, periodically-updated** `rsETHPrice` from `LRTOracle`. Because `rsETHPrice` is only refreshed when `updateRSETHPrice()` is called (a public, permissionless function), any depositor who deposits while the stored price is stale (lower than the actual backing ratio) receives more rsETH than the current TVL justifies. This dilutes the accrued yield of all existing rsETH holders.

---

### Finding Description

The mint formula in `getRsETHAmountToMint` is:

```
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()
``` [1](#0-0) 

`lrtOracle.rsETHPrice()` is a **state variable** in `LRTOracle`, written only inside `_updateRsETHPrice()`: [2](#0-1) 

It is updated by the public function `updateRSETHPrice()`: [3](#0-2) 

Between calls to `updateRSETHPrice()`, staking yield accrues (ETH balance in NDCs, EigenLayer pod shares, etc.), so `totalETHInProtocol` grows while `rsETHPrice` stays fixed at the last snapshot. The deposit path has **no freshness check** and **no requirement to update the price first**: [4](#0-3) 

**Concrete arithmetic:**

| State | Value |
|---|---|
| totalETH (with 1 ETH accrued yield) | 100 ETH |
| rsethSupply | 99 rsETH |
| `rsETHPrice` (stale) | 1.000 ETH |
| Fair price (`100/99`) | ≈ 1.0101 ETH |

Attacker deposits 99 ETH using stale price:
- Minted: `99 / 1.000 = 99 rsETH` (fair would be `99 / 1.0101 ≈ 98.01 rsETH`)

After deposit, `updateRSETHPrice()` runs:
- `newRsETHPrice = 199 ETH / 198 rsETH ≈ 1.00505 ETH`

Attacker's 99 rsETH is now worth `99 × 1.00505 ≈ 99.5 ETH` — a **~0.5 ETH profit** on a 99 ETH deposit, extracted entirely from the 1 ETH of yield that existing holders had earned.

The `_updateRsETHPrice` function computes the new price using the **post-deposit** supply and TVL, so the attacker's dilution is baked into the price update: [5](#0-4) 

---

### Impact Explanation

Existing rsETH holders lose a portion of their accrued staking yield proportional to the attacker's deposit size relative to total TVL. The attacker's principal is not at risk; the attack extracts **unclaimed yield** from existing holders. This does not threaten principal or cause insolvency, but it is a repeatable, permissionless extraction of yield.

**Impact: High — Theft of unclaimed yield.**

The claimed Critical impact (direct theft of user funds / principal) is **not reachable**: existing holders' rsETH remains backed above 1 ETH per token; only the yield increment is stolen.

---

### Likelihood Explanation

- `updateRSETHPrice()` is public and permissionless; the attacker controls when to deposit relative to price updates.
- Yield accrues continuously; the stale window is always non-zero between updates.
- No special role, governance access, or oracle manipulation is required.
- The attack is repeatable every price-update cycle.
- Front-running the `updateRSETHPrice()` transaction (or simply depositing in any block before it) is sufficient.

---

### Recommendation

1. **Atomic price refresh on deposit**: Call `_updateRsETHPrice()` (or an equivalent internal read of the live price) at the start of `_beforeDeposit`, so the mint rate always reflects the current TVL.
2. **Alternatively**, compute the mint rate on-the-fly from `totalETHInProtocol / rsethSupply` rather than from the stored `rsETHPrice` snapshot, eliminating the stale-read window entirely.
3. **At minimum**, document the staleness window and enforce a maximum update interval enforced on-chain (e.g., revert deposits if `rsETHPrice` is older than N blocks).

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

// Foundry invariant sketch — run on a local fork
// Demonstrates that depositing before updateRSETHPrice gives more rsETH
// than the fair backing ratio justifies.

contract StaleRsETHPricePoC is Test {
    LRTDepositPool pool;
    LRTOracle oracle;
    IRSETH rsETH;

    function setUp() public {
        // fork + deploy protocol (standard test setup)
    }

    function test_staleRsETHPriceDilutesYield() public {
        // 1. Simulate yield accrual: send 1 ETH directly to a NodeDelegator
        //    so totalETHInProtocol increases without rsETHPrice being updated.
        vm.deal(address(nodeDelegator), address(nodeDelegator).balance + 1 ether);

        uint256 rsETHPriceBefore = oracle.rsETHPrice(); // stale, lower than fair

        // 2. Attacker deposits at stale price
        uint256 depositAmt = 99 ether;
        vm.deal(attacker, depositAmt);
        vm.prank(attacker);
        pool.depositETH{value: depositAmt}(0, "");

        uint256 attackerRsETH = rsETH.balanceOf(attacker);

        // 3. Update price — now includes attacker's deposit
        oracle.updateRSETHPrice();
        uint256 rsETHPriceAfter = oracle.rsETHPrice();

        // 4. Attacker's rsETH value > deposited ETH
        uint256 attackerValueAfter = attackerRsETH * rsETHPriceAfter / 1e18;
        assertGt(attackerValueAfter, depositAmt, "Attacker profited from stale price");

        // 5. Existing holders' effective yield is reduced
        // (compare rsETHPriceAfter to what it would have been without the deposit)
    }
}
```

The invariant `attackerValueAfter <= depositAmt` breaks whenever `rsETHPrice` is stale and yield has accrued, confirming the finding.
