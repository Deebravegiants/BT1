### Title
Missing Negative Price Validation Before `uint256` Cast Causes Revert-Based Freeze of Deposits and Price Updates - (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

### Summary
`ChainlinkPriceOracle.getAssetPrice()` casts the `int256 price` returned by Chainlink's `latestRoundData()` directly to `uint256` without first validating that the price is positive. If Chainlink returns a zero or negative value, the cast produces a massive wrapped value, and the subsequent multiplication overflows in Solidity 0.8, causing a revert. This propagates through `LRTOracle._getTotalEthInProtocol()` → `_updateRsETHPrice()` → `updateRSETHPrice()`, freezing rsETH price updates and blocking all deposits.

### Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol`, `getAssetPrice()` reads the raw `int256 price` from Chainlink and immediately casts it to `uint256`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

There is no guard against `price <= 0`. In Solidity 0.8, the explicit cast `uint256(negative_int256)` does not revert — it wraps to a value near `type(uint256).max`. The subsequent arithmetic `uint256(price) * 1e18` then triggers an overflow revert under Solidity 0.8's checked arithmetic.

By contrast, the sibling oracle `ChainlinkOracleForRSETHPoolCollateral.getRate()` correctly validates the sign before casting:

```solidity
if (ethPrice <= 0) revert InvalidPrice();
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / ...;
``` [2](#0-1) 

`ChainlinkPriceOracle` is the oracle used for all supported LST assets (stETH, ETHx, etc.) in the main protocol. Its output feeds directly into `LRTOracle.getAssetPrice()`: [3](#0-2) 

Which is called inside `_getTotalEthInProtocol()`: [4](#0-3) 

Which is called inside `_updateRsETHPrice()`: [5](#0-4) 

`updateRSETHPrice()` is a public, permissionless function: [6](#0-5) 

### Impact Explanation
If any Chainlink feed configured in `ChainlinkPriceOracle` returns a non-positive `int256` answer (e.g., during a circuit-breaker event, oracle failure, or feed misconfiguration), every call to `updateRSETHPrice()` reverts. Because `rsETHPrice` is stale, `getRsETHAmountToMint()` in `LRTDepositPool` uses the stale price, and any deposit path that triggers a price update will revert. This constitutes a **temporary freezing of funds** (Medium) — users cannot deposit, and the protocol's exchange rate is frozen until the oracle recovers or an admin intervenes.

**Impact: Medium — Temporary freezing of funds.**

### Likelihood Explanation
Chainlink feeds can return non-positive values during circuit breaker events (e.g., when the underlying asset price hits the feed's `minAnswer`/`maxAnswer` bounds). This is a known, documented Chainlink behavior. The `ChainlinkPriceOracle` is the primary oracle for all LST assets in the protocol, making this a realistic trigger path.

### Recommendation
Add a non-positive price check before casting, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
if (price <= 0) revert InvalidPrice();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

### Proof of Concept
1. Chainlink's `latestRoundData()` for a supported asset (e.g., stETH/ETH) returns `price = -1` (or `0`) during a circuit breaker event.
2. `ChainlinkPriceOracle.getAssetPrice(stETH)` executes `uint256(-1)` → `2^256 - 1`, then `(2^256 - 1) * 1e18` overflows → **revert**.
3. `LRTOracle._getTotalEthInProtocol()` reverts.
4. `LRTOracle._updateRsETHPrice()` reverts.
5. `LRTOracle.updateRSETHPrice()` (public) reverts for all callers.
6. `LRTDepositPool.depositAsset()` → `getRsETHAmountToMint()` → `lrtOracle.getAssetPrice(asset)` also reverts.
7. All deposits are frozen until the Chainlink feed recovers or an admin replaces the oracle.
