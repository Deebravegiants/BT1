### Title
Direct ETH Donation to `LRTDepositPool` Inflates `address(this).balance`, Enabling Deposit Limit Griefing and rsETH Price Manipulation - (File: contracts/LRTDepositPool.sol)

---

### Summary
The `LRTDepositPool` contract exposes an open `receive()` function that allows any external actor to send ETH directly. This inflates `address(this).balance`, which is used raw in `getETHDistributionData()` to compute `ethLyingInDepositPool`. That value propagates into `getTotalAssetDeposits()`, which is consumed by both `_checkIfDepositAmountExceedesCurrentLimit()` (deposit gating) and `LRTOracle._getTotalEthInProtocol()` (rsETH price). An unprivileged attacker can exploit this to (a) push total ETH deposits over the configured limit, blocking all new ETH deposits, and (b) inflate the rsETH price so that legitimate depositors receive fewer rsETH than they are entitled to.

---

### Finding Description

`LRTDepositPool` declares an unconditional `receive()`: [1](#0-0) 

`getETHDistributionData()` reads the raw native balance of the contract as the canonical measure of ETH held in the deposit pool: [2](#0-1) 

This value is returned as `ethLyingInDepositPool` and is summed with NDC balances, EigenLayer shares, and unstaking amounts inside `getTotalAssetDeposits()`. The deposit-limit gate then uses this sum: [3](#0-2) 

For ETH the check is `totalAssetDeposits > limit` (not `totalAssetDeposits + depositAmount > limit`), so any donation that pushes the running total above the configured cap immediately blocks every subsequent `depositETH` call with `MaximumDepositLimitReached`.

The same inflated total is consumed by `LRTOracle._getTotalEthInProtocol()`: [4](#0-3) 

A higher `totalETHInProtocol` raises `rsETHPrice`, so the next `updateRSETHPrice()` call records a price that is artificially elevated. Every subsequent depositor then receives fewer rsETH tokens than the actual protocol-owned ETH warrants.

No before/after balance accounting is performed anywhere in this path; the entire ETH balance of the contract — including any externally donated ETH — is treated as protocol-owned collateral.

---

### Impact Explanation

**Temporary freezing of ETH deposits (Medium):** An attacker who donates enough ETH to push `totalAssetDeposits` above `depositLimitByAsset(ETH_TOKEN)` causes every `depositETH` call to revert. The freeze persists until an operator transfers the excess ETH to a NodeDelegator, reducing the pool balance. The attacker forfeits the donated ETH (it remains in the pool), but the cost is bounded by the gap between the current total and the deposit cap.

**Theft of unclaimed yield / under-minting (High):** If `updateRSETHPrice()` is called while the balance is inflated, the recorded rsETH price is higher than the true backing ratio. All depositors who mint rsETH after that price update receive fewer shares than they are owed relative to the ETH they contribute. The difference accrues to existing rsETH holders, constituting a transfer of value from new depositors to incumbents.

---

### Likelihood Explanation

The `receive()` function is unconditionally open; no role, whitelist, or minimum-amount check exists. Any externally-owned account can trigger the inflation with a single ETH transfer. The cost to the attacker equals the ETH donated, which is not destroyed but is locked in the pool until an operator acts. When the deposit cap is close to the current total (a common operational state near capacity), even a small donation suffices to trigger the freeze. The rsETH price manipulation requires only that `updateRSETHPrice()` be called while the inflated balance persists, which is a routine keeper operation.

---

### Recommendation

Replace the raw `address(this).balance` read in `getETHDistributionData()` with a storage-tracked variable (`ethReceivedFromDeposits`) that is incremented only inside the explicit deposit entry points (`depositETH`, `receiveFromNodeDelegator`, `receiveFromRewardReceiver`, `receiveFromLRTConverter`). Alternatively, implement a before/after balance check inside each entry point and credit only the delta, analogous to the fix applied in the referenced TREC-3 report. This ensures that externally donated ETH is never counted as protocol-owned collateral.

---

### Proof of Concept

1. Observe the current `getTotalAssetDeposits(ETH_TOKEN)` value; assume it is `D` and the deposit limit is `L` with `D < L`.
2. Attacker calls `address(lrtDepositPool).call{value: L - D + 1}("")`, sending `L - D + 1` wei directly to `LRTDepositPool`.
3. `address(this).balance` inside `LRTDepositPool` increases by `L - D + 1`.
4. `getETHDistributionData()` now returns `ethLyingInDepositPool = old_balance + (L - D + 1)`.
5. `getTotalAssetDeposits(ETH_TOKEN)` returns `D + (L - D + 1) = L + 1 > L`.
6. Any call to `depositETH(...)` executes `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, amount)`, which evaluates `totalAssetDeposits > depositLimitByAsset(ETH_TOKEN)` → `L + 1 > L` → `true`, and reverts with `MaximumDepositLimitReached`.
7. Simultaneously, the next `LRTOracle.updateRSETHPrice()` call computes `_getTotalEthInProtocol()` using the inflated pool balance, records an elevated `rsETHPrice`, and all subsequent depositors receive fewer rsETH than they are entitled to. [5](#0-4) [6](#0-5) [7](#0-6) [4](#0-3)
