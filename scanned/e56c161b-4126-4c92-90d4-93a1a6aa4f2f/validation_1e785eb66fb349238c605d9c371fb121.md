### Title
Missing Sign Check Before `int256`→`uint256` Cast Allows Chainlink Negative Price to Freeze Protocol Operations - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary
`ChainlinkPriceOracle.getAssetPrice()` casts the `int256 price` returned by Chainlink's `latestRoundData()` directly to `uint256` without verifying the value is positive. If the feed returns a negative answer, the cast produces a value near `2^256`, and the subsequent multiplication overflows (Solidity 0.8.x checked arithmetic), causing a revert. Every protocol path that depends on this oracle — deposits, rsETH price updates, and withdrawals — is then frozen until the feed recovers or the oracle is replaced.

### Finding Description
In `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol  lines 52-54
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

`int256 price` is cast to `uint256` with no sign guard. For any negative `price` (e.g., `-1`), the cast yields `2^256 − 1` via two's-complement reinterpretation. The immediately following `* 1e18` then overflows and reverts under Solidity 0.8.x checked arithmetic.

The sister oracle in the same repository, `ChainlinkOracleForRSETHPoolCollateral`, already applies the correct guard:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol  line 32
if (ethPrice <= 0) revert InvalidPrice();
```

`ChainlinkPriceOracle` has no equivalent check.

### Impact Explanation
`LRTOracle.getAssetPrice()` delegates to `IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset)`, which resolves to `ChainlinkPriceOracle.getAssetPrice()` for every Chainlink-backed LST. [1](#0-0) 

That call is made inside `_getTotalEthInProtocol()`, which is invoked by the public `updateRSETHPrice()` and by every deposit path through `getRsETHAmountToMint()`. [2](#0-1) 

A revert in `getAssetPrice()` propagates upward and freezes:
- `LRTDepositPool.depositAsset()` / `depositETH()` — new deposits impossible.
- `LRTOracle.updateRSETHPrice()` — price cannot be refreshed.
- Any withdrawal path in `LRTWithdrawalManager` that calls `getAssetPrice()` (4 call-sites confirmed by grep).

The freeze persists until the Chainlink feed recovers or an admin replaces the oracle, which may require governance delay.

### Likelihood Explanation
Chainlink's `latestRoundData()` specification does not guarantee a positive `answer`; deprecated or malfunctioning feeds have historically returned zero or negative values. The `ChainlinkPriceOracle` is used for all Chainlink-backed LST assets in the protocol. A single affected feed is sufficient to freeze the entire deposit and price-update flow. No attacker action is required beyond waiting for a feed anomaly; the root cause is entirely within the LRT-rsETH codebase (the missing guard), not in Chainlink itself.

### Recommendation
Add a positivity check before the cast, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
if (price <= 0) revert InvalidPrice();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

### Proof of Concept

1. A Chainlink LST/ETH feed (e.g., stETH/ETH) returns `answer = -1` due to feed deprecation or malfunction.
2. Any caller invokes `LRTDepositPool.depositAsset(stETH, amount, minRSETH, ref)`.
3. Internally, `getRsETHAmountToMint()` → `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)`.
4. `uint256(int256(-1))` = `2^256 − 1`; `(2^256 − 1) * 1e18` overflows → revert.
5. All deposits revert. `updateRSETHPrice()` also reverts. The protocol is operationally frozen until the oracle is replaced via admin action. [3](#0-2) [4](#0-3) [1](#0-0) [5](#0-4)
