### Title
Large `initiateWithdrawal` Commitment Blocks Other Users From Initiating Withdrawals - (File: contracts/LRTWithdrawalManager.sol)

### Summary

`LRTWithdrawalManager.initiateWithdrawal` gates new withdrawal requests against `getAvailableAssetAmount`, which counts **all** protocol assets — including those locked in EigenLayer with a 7-day unbonding period — as "available." A user holding a large rsETH balance can commit to withdrawing nearly all protocol assets, inflating `assetsCommitted[asset]` and reducing the available amount to near-zero. This blocks every other user from initiating a new withdrawal until the large request is processed through `unlockQueue`, which itself requires the assets to be retrieved from EigenLayer (≥7 days).

### Finding Description

`initiateWithdrawal` enforces a capacity check before recording a new request:

```solidity
// LRTWithdrawalManager.sol:168-173
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();
assetsCommitted[asset] += expectedAssetAmount;
```

`getAvailableAssetAmount` computes:

```solidity
// LRTWithdrawalManager.sol:599-603
uint256 totalAssets = lrtDepositPool.getTotalAssetDeposits(asset);
availableAssetAmount = totalAssets > assetsCommitted[asset] ? totalAssets - assetsCommitted[asset] : 0;
```

`getTotalAssetDeposits` aggregates every location where the asset exists in the protocol:

```solidity
// LRTDepositPool.sol:394-396
return (assetLyingInDepositPool + assetLyingInNDCs + effectiveAssetWithEigenLayer
        + assetLyingInConverter + assetLyingUnstakingVault);
```

where `effectiveAssetWithEigenLayer = assetStakedInEigenLayer + assetUnstakingFromEigenLayer`.

Assets staked in EigenLayer are **illiquid** — they require a 7-day unbonding period before they can be moved to the `LRTUnstakingVault`. Despite this, they are counted as "available" for the purpose of accepting new withdrawal commitments. A whale can therefore commit to withdrawing EigenLayer-locked assets, saturating `assetsCommitted[asset]` and causing every subsequent `initiateWithdrawal` call from other users to revert with `ExceedAmountToWithdraw`.

The commitment is only released when `unlockQueue` processes the request:

```solidity
// LRTWithdrawalManager.sol:802
assetsCommitted[asset] -= request.expectedAssetAmount;
```

But `unlockQueue` can only process requests when the assets are physically present in the `LRTUnstakingVault` (`unstakingVault.balanceOf(asset)`), not when they are still in EigenLayer. The operator must first queue the EigenLayer withdrawal, wait the 7-day unbonding period, complete it, and transfer assets to the vault before `unlockQueue` can run. During this entire window, all other users are blocked.

### Impact Explanation

**Temporary freezing of funds.** Any rsETH holder who initiates a withdrawal after the whale's commitment will receive `ExceedAmountToWithdraw` and be unable to enter the withdrawal queue. Their rsETH remains locked in their wallet with no path to redemption until the whale's request is fully processed — a delay of at minimum 7 days (EigenLayer unbonding) plus operator processing time. This matches the "temporary freezing of funds" impact category.

### Likelihood Explanation

The entry point `initiateWithdrawal` is publicly callable by any rsETH holder with no role restriction. A single large holder (e.g., a whale, a protocol treasury, or a DeFi vault that accumulated rsETH) can trigger this unintentionally or deliberately. The condition is straightforward to reach: hold enough rsETH to commit to a large fraction of total protocol assets. Given that EigenLayer assets typically represent the majority of protocol TVL, the blocking window is structurally long.

### Recommendation

Decouple the withdrawal commitment capacity from illiquid EigenLayer assets. `getAvailableAssetAmount` should only count assets that are realistically redeemable within the withdrawal delay window — i.e., assets in the `LRTDepositPool`, NDCs, and `LRTUnstakingVault`. EigenLayer assets should not be counted until they have been queued for withdrawal and are in the `assetUnstakingFromEigenLayer` state, or better yet, only once they have landed in the `LRTUnstakingVault`. Alternatively, document this property clearly so users understand that large pending withdrawals can delay queue entry for others.

### Proof of Concept

1. Protocol state: 10,000 stETH total, 9,500 stETH staked in EigenLayer, 500 stETH in `LRTUnstakingVault`. `assetsCommitted[stETH] = 0`.
2. `getAvailableAssetAmount(stETH)` returns `10,000 - 0 = 10,000`.
3. Whale calls `initiateWithdrawal(stETH, rsETHAmount)` where `rsETHAmount` corresponds to 9,800 stETH. The check passes; `assetsCommitted[stETH] = 9,800`.
4. `getAvailableAssetAmount(stETH)` now returns `10,000 - 9,800 = 200`.
5. Alice calls `initiateWithdrawal(stETH, rsETHAmount)` for 300 stETH. `300 > 200` → `ExceedAmountToWithdraw`. Alice is blocked.
6. Operator must queue 9,300 stETH from EigenLayer, wait 7 days, complete withdrawal, transfer to vault, then call `unlockQueue`. Only after this does `assetsCommitted` decrease and Alice can retry. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
