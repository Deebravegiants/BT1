### Title
No Slippage Protection in RSETHPool Deposit Functions - (File: contracts/pools/RSETHPool.sol)

### Summary
The `deposit()` functions in `RSETHPool` accept ETH or tokens from users and return rsETH based on the current oracle rate, but provide no `minRSETHAmountExpected` parameter. Users cannot protect themselves from receiving fewer rsETH than anticipated if the oracle rate changes between transaction submission and execution.

### Finding Description
`RSETHPool.deposit(string)` (ETH path) and `RSETHPool.deposit(address, uint256, string)` (token path) both compute the rsETH output solely from the live oracle rate at execution time:

```solidity
// ETH deposit – no minimum rsETH guard
function deposit(string memory referralId) external payable nonReentrant whenNotPaused {
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    ...
    IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount);
}

// Token deposit – no minimum rsETH guard
function deposit(address token, uint256 amount, string memory referralId)
    external nonReentrant whenNotPaused onlySupportedToken(token)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
    ...
    IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount);
}
``` [1](#0-0) [2](#0-1) 

The rate is fetched from `rsETHOracle` at call time:

```solidity
uint256 rsETHToETHrate = getRate(); // live oracle read
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [3](#0-2) 

By contrast, the L1-side `LRTDepositPool.depositETH()` and `depositAsset()` both accept a `minRSETHAmountExpected` argument and revert if the minted amount falls below it, giving L1 depositors explicit slippage protection that L2 pool depositors lack. [4](#0-3) [5](#0-4) 

### Impact Explanation
A user who observes a favorable rsETH/ETH rate and submits a deposit transaction may have the oracle update to a higher rate before their transaction is included. They will receive fewer rsETH tokens than they anticipated with no on-chain recourse. The user does not lose ETH value outright, but the contract fails to deliver the rsETH amount the user reasonably expected — matching the **Low** impact tier: *contract fails to deliver promised returns, but doesn't lose value*.

### Likelihood Explanation
The rsETH/ETH rate accretes slowly via staking rewards, so large single-block swings are uncommon. However, oracle updates can occur at any time, and during periods of high network congestion a pending transaction may sit in the mempool long enough for one or more oracle updates to occur. Any user of the L2 pool is exposed on every deposit with no opt-out mechanism.

### Recommendation
Add a `minRSETHAmountExpected` parameter to both `deposit()` overloads and revert if the computed `rsETHAmount` is below it, mirroring the pattern already used in `LRTDepositPool`:

```solidity
function deposit(string memory referralId, uint256 minRSETHAmountExpected)
    external payable nonReentrant whenNotPaused
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    ...
}
```

The same fix should be applied to the analogous `deposit()` functions in `RSETHPoolNoWrapper`, `RSETHPoolV2ExternalBridge`, and `RSETHPoolV3ExternalBridge`.


### Proof of Concept
1. Oracle reports rsETH/ETH rate = 1.05; user submits `deposit{value: 1 ether}("ref")` expecting ≈ 0.952 rsETH (minus fee).
2. Before the transaction is mined, the oracle updates to rate = 1.10.
3. Transaction executes: user receives ≈ 0.909 rsETH — ~4.5 % less than expected.
4. No parameter exists to cause the transaction to revert; the user has no protection.

    function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
    }
