### Title
`BURNER_ROLE` Can Burn Any Amount of rsETH From an Arbitrary Address Without Allowance - (`contracts/RSETH.sol`)

### Summary

`RSETH.burnFrom()` allows any address holding `BURNER_ROLE` to burn an arbitrary amount of rsETH from **any** user address without requiring an ERC-20 allowance or any form of user consent. This is a direct structural analog to M-08.

### Finding Description

`RSETH.burnFrom()` is gated only by `onlyRole(LRTConstants.BURNER_ROLE)` and `whenNotPaused`. It calls `_burn(account, amount)` directly — the internal OpenZeppelin burn that destroys tokens from the target balance — with no `_spendAllowance` check and no requirement that `account == msg.sender`.

```solidity
// contracts/RSETH.sol L245-248
function burnFrom(address account, uint256 amount) external onlyRole(LRTConstants.BURNER_ROLE) whenNotPaused {
    _enforceNotBlocked(account);
    _burn(account, amount);
}
```

The standard ERC-20 `burnFrom` (from OpenZeppelin `ERC20Burnable`) requires the caller to have an allowance from `account`. This implementation bypasses that entirely: the role holder can target **any** non-blocked address and destroy **any** amount up to its full balance in a single call. [1](#0-0) 

### Impact Explanation

A `BURNER_ROLE` holder can permanently destroy the entire rsETH balance of any user. rsETH represents a claim on pooled ETH/LST collateral; burning it without redemption means the user loses their staked value with no recourse. This constitutes **permanent loss of user funds** (Critical — permanent freezing/destruction of funds).

### Likelihood Explanation

Low. Exploitation requires a `BURNER_ROLE` holder to act maliciously or for that key to be compromised. However, the role is not time-locked or subject to a delay, so once a malicious actor holds it, the attack is instant and irreversible. Users have no on-chain mechanism to detect or prevent it before execution.

### Recommendation

1. Remove the ability to burn from arbitrary addresses without allowance. If cross-protocol burn-on-bridge is the intended use case, restrict `burnFrom` to burn only from `msg.sender`, or require a valid ERC-20 allowance via `_spendAllowance(account, msg.sender, amount)` before calling `_burn`.
2. If burning from arbitrary addresses is genuinely required (e.g., for bridge adapters), scope the role more narrowly — e.g., only allow burning from the bridge contract's own escrow address, not from arbitrary user wallets.
3. Consider adding a timelock or multi-sig requirement before `BURNER_ROLE` can be granted.

### Proof of Concept

1. Attacker acquires `BURNER_ROLE` (or the role holder acts maliciously).
2. User Alice holds 100 rsETH.
3. Attacker calls `RSETH.burnFrom(alice, 100e18)`.
4. `_enforceNotBlocked(alice)` passes (Alice is not blocked).
5. `_burn(alice, 100e18)` executes — Alice's 100 rsETH is permanently destroyed with no compensation.
6. Alice's underlying ETH/LST collateral remains in the protocol, now distributed to remaining rsETH holders (diluting her claim to zero). [2](#0-1)
