### Title
`assetsCommitted` Phantom Capacity After rsETH Price Drop Allows Over-Commitment of Withdrawal Requests - (`contracts/LRTWithdrawalManager.sol`)

### Summary

When the rsETH price drops between the time a withdrawal request is initiated and the time it is unlocked, `assetsCommitted` is reduced by the original (higher) `expectedAssetAmount` while only the lower `payoutAmount` is actually redeemed from the unstaking vault. This creates phantom available capacity in `getAvailableAssetAmount`, allowing new withdrawal requests to be initiated against assets that do not exist in the protocol, leading to temporary freezing of funds for users whose requests cannot be fulfilled.

### Finding Description

The vulnerability class from the external report is: a stored rate/amount is used to calculate a payout, but the current rate has dropped below the stored value, causing the accounting to diverge from reality. In LRT-rsETH, the analog is in the `assetsCommitted` accounting within `LRTWithdrawalManager`.

**Step 1 — `initiateWithdrawal` stores `expectedAssetAmount` at the current (higher) price:** [1](#0-0) 

```
expectedAssetAmount = rsETHUnstaked * rsETHPrice_T1 / assetPrice_T1
assetsCommitted[asset] += expectedAssetAmount
```

**Step 2 — `_unlockWithdrawalRequests` (called via `unlockQueue`) reduces `assetsCommitted` by the original `expectedAssetAmount`, but only redeems the lower `payoutAmount`:** [2](#0-1) 

```
payoutAmount = min(expectedAssetAmount, currentReturn)   // currentReturn < expectedAssetAmount when price dropped
assetsCommitted[asset] -= request.expectedAssetAmount    // reduced by ORIGINAL higher amount
assetAmountToUnlock += payoutAmount                      // only lower amount redeemed
``` [3](#0-2) 

**Step 3 — `getAvailableAssetAmount` uses `getTotalAssetDeposits`, which includes the unstaking vault balance:** [4](#0-3) [5](#0-4) 

`getTotalAssetDeposits` explicitly includes `assetLyingUnstakingVault`: [6](#0-5) 

**The accounting divergence:**

After `unlockQueue` is called with a lower rsETH price:
- `totalAssets` (from `getTotalAssetDeposits`) decreases by `payoutAmount` (assets moved from unstaking vault to withdrawal manager)
- `assetsCommitted` decreases by `expectedAssetAmount` (the original higher amount)
- Net change in `getAvailableAssetAmount` = `+(expectedAssetAmount − payoutAmount)` > 0

The assets in the withdrawal manager are **not** counted in `getTotalAssetDeposits` (only deposit pool, NDCs, EigenLayer, converter, and unstaking vault are counted). So the freed capacity `expectedAssetAmount − payoutAmount` is phantom — no actual assets back it.

A new user can now call `initiateWithdrawal` for this phantom capacity: [7](#0-6) 

When the operator later calls `unlockQueue` for this new request, the unstaking vault may not have enough assets, causing the loop to `break` at: [8](#0-7) 

The new user's rsETH is locked in the withdrawal manager with no assets to back the request.

### Impact Explanation

**Medium — Temporary freezing of funds.** Users who initiate withdrawal requests using the phantom available capacity will have their rsETH locked in the `LRTWithdrawalManager` with no corresponding assets in the unstaking vault to fulfill the request. The `unlockQueue` loop will skip their request indefinitely until the unstaking vault is replenished. The rsETH has already been transferred to the withdrawal manager at `initiateWithdrawal` time, so it is inaccessible to the user during this period.

### Likelihood Explanation

The rsETH price can decrease when underlying LST assets (stETH, ETHx, rETH) lose value, or when EigenLayer slashing occurs. The protocol itself acknowledges this possibility — `LRTOracle._updateRsETHPrice` has explicit downside protection logic that pauses the protocol if the price drops beyond a threshold: [9](#0-8) 

However, price drops **within** the `pricePercentageLimit` threshold do not trigger a pause and will silently create phantom capacity. The `unlockQueue` function also accepts operator-supplied price bounds (`minimumRsEthPrice`, `maximumRsEthPrice`) that could allow unlocking at a price lower than the original commitment price: [10](#0-9) 

### Recommendation

In `_unlockWithdrawalRequests`, reduce `assetsCommitted` by `payoutAmount` (the actual amount redeemed) rather than by `request.expectedAssetAmount` (the original commitment):

```solidity
// Before (incorrect):
assetsCommitted[asset] -= request.expectedAssetAmount;

// After (correct):
assetsCommitted[asset] -= payoutAmount;
```

This ensures `getAvailableAssetAmount` accurately reflects the actual assets available in the protocol after a price-drop unlock.

### Proof of Concept

1. rsETH price is `P1 = 1.05e18`, stETH price is `1e18`.
2. User A calls `initiateWithdrawal(stETH, 100e18)`:
   - `expectedAssetAmount = 100e18 * 1.05e18 / 1e18 = 105e18`
   - `assetsCommitted[stETH] += 105e18`
3. rsETH price drops to `P2 = 1.00e18` (within the `pricePercentageLimit` threshold, no pause).
4. Operator calls `unlockQueue(stETH, ...)` at the new price:
   - `currentReturn = 100e18 * 1.00e18 / 1e18 = 100e18`
   - `payoutAmount = min(105e18, 100e18) = 100e18`
   - `assetsCommitted[stETH] -= 105e18` ← reduced by original amount
   - `unstakingVault.redeem(stETH, 100e18)` ← only 100e18 redeemed
5. `getAvailableAssetAmount(stETH)` increases by `105e18 − 100e18 = 5e18` (phantom).
6. User B calls `initiateWithdrawal(stETH, 4.76e18)` (≈ 5e18 / 1.05 rsETH):
   - `expectedAssetAmount ≈ 5e18`
   - `assetsCommitted[stETH] += 5e18`
   - User B's rsETH is transferred to the withdrawal manager.
7. Operator calls `unlockQueue` for User B's request. The unstaking vault has no assets to cover it (they were all redeemed in step 4). The loop breaks at `if (availableAssetAmount < payoutAmount) break`.
8. User B's rsETH is frozen in the withdrawal manager with no path to completion until the unstaking vault is replenished.

        WithdrawalRequest storage request,
        uint256 rsETHPrice,
        uint256 assetPrice
    )
        private
        view
        returns (uint256)
    {
        uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
