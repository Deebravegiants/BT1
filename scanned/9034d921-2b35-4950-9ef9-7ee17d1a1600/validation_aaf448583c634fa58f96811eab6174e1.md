### Title
Missing Token Decimal Normalization in `viewSwapRsETHAmountAndFee` Causes Incorrect rsETH Minting for Non-18-Decimal Tokens - (`contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`, `contracts/pools/RSETHPoolV3WithNativeChainBridge.sol`, `contracts/pools/RSETHPool.sol`, `contracts/pools/RSETHPoolNoWrapper.sol`)

---

### Summary

Every L2 pool contract that accepts ERC-20 token deposits computes the rsETH amount to mint with the formula:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

`tokenToETHRate` is normalised to 1e18 precision and represents the price of **one full token** in ETH. `amountAfterFee` is the raw token amount in the token's **smallest unit** (e.g. 1e6 for 1 USDC). The formula is dimensionally correct only when the token has 18 decimals. For any token with fewer decimals the result is `10^(18 − tokenDecimals)` times too small, causing users to receive a negligible amount of rsETH for a real deposit.

---

### Finding Description

The vulnerable function appears identically across all five pool contracts:

```solidity
// contracts/pools/RSETHPoolV3.sol  (and four other pool contracts)
function viewSwapRsETHAmountAndFee(
    uint256 amount,
    address token
) public view onlySupportedToken(token) returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;

    uint256 rsETHToETHrate = getRate();                                    // 1e18-precision
    uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate(); // 1e18-precision

    rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;        // ← decimal bug
}
```

The oracle wrapper that supplies `tokenToETHRate` normalises the Chainlink answer to 1e18:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol
uint256 normalizedPrice = uint256(ethPrice) * 1e18
    / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
```

`normalizedPrice` is the price of **one full token** (e.g. 1.0 USDC) in ETH, expressed in 1e18 units. It says nothing about the token's own decimal count.

**Dimensional analysis for a 6-decimal token (e.g. USDC):**

| Quantity | Value | Unit |
|---|---|---|
| `amountAfterFee` | `1 000 × 1e6 = 1e9` | USDC smallest unit |
| `tokenToETHRate` | `3e14` | ETH-wei per 1 full USDC |
| `rsETHToETHrate` | `1.05e18` | ETH-wei per 1 rsETH |
| **Current result** | `1e9 × 3e14 / 1.05e18 ≈ 2.857e5` | rsETH smallest unit |
| **Correct result** | `2.857e17` | rsETH smallest unit |

The user receives `1e12` times fewer rsETH than the fair value of their deposit.

The correct formula must normalise `amountAfterFee` to 18-decimal precision:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate * 1e18
              / rsETHToETHrate / (10 ** tokenDecimals);
```

For 18-decimal tokens the `1e18 / 1e18` factor cancels and the formula reduces to the current one, so existing behaviour is unchanged.

The same implicit assumption appears in the reverse-swap view:

```solidity
tokenAmount = rsETHAmount * rsETHToETHrate / tokenToETHRate;
```

For a 6-decimal token this returns `1e12` times **too many** tokens, which would drain the pool if the operator-restricted `swapAssetToPremintedRsETH` were ever called with such a token.

---

### Impact Explanation

When a non-18-decimal token is added to any pool's `supportedTokenOracle` mapping, every depositor of that token receives a negligible rsETH amount relative to the ETH value they contributed. The contract accepts the full token transfer but mints almost nothing in return. This constitutes the contract failing to deliver its promised exchange rate — the depositor's funds are locked in the pool while their rsETH balance is effectively zero.

**Impact: Low — Contract fails to deliver promised returns, but doesn't lose value.**

---

### Likelihood Explanation

The pool contracts expose a generic `addSupportedToken` / `supportedTokenOracle` mechanism with no restriction on token decimals. An admin legitimately expanding the protocol to accept WBTC (8 decimals) or any stablecoin (6 decimals) would silently activate the bug. No malicious intent is required; the miscalculation is invisible at the Solidity level and would not revert.

**Likelihood: Low** — requires a non-18-decimal token to be added as supported, which is not the current configuration but is a plausible future expansion.

---

### Recommendation

Retrieve the token's decimal count and normalise `amountAfterFee` before applying the rate:

```solidity
function viewSwapRsETHAmountAndFee(
    uint256 amount,
    address token
) public view onlySupportedToken(token) returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;

    uint256 rsETHToETHrate = getRate();
    uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
    uint8   tokenDecimals  = IERC20Metadata(token).decimals();

    // Normalise amountAfterFee to 1e18 before dividing by the rsETH rate
    rsETHAmount = amountAfterFee * tokenToETHRate * 1e18
                  / rsETHToETHrate / (10 ** uint256(tokenDecimals));
}
```

Apply the symmetric correction to `viewSwapAssetToPremintedRsETH`.

### Proof of Concept

```
Token: hypothetical 6-decimal LST (e.g. USDC-equivalent collateral)
Deposit: 1 000 tokens  →  amountAfterFee = 1 000 × 1e6 = 1e9

tokenToETHRate  = 3e14   (0.0003 ETH per token, 1e18-normalised)
rsETHToETHrate  = 1.05e18

Current formula:
  rsETHAmount = 1e9 × 3e14 / 1.05e18
              ≈ 2.857e5   (≈ 0.000000000000285 rsETH)

Correct formula (with decimal normalisation):
  rsETHAmount = 1e9 × 3e14 × 1e18 / 1.05e18 / 1e6
              ≈ 2.857e17  (≈ 0.2857 rsETH)

Error factor: 1e12 — the user receives one trillion times fewer rsETH
than the fair value of their deposit.
