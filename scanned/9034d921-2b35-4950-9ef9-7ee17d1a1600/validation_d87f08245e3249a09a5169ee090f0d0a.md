### Title
Asymmetric ETH Deposit-Limit Check Allows Limit Overshoot and Blocks All Subsequent ETH Deposits — (`contracts/LRTDepositPool.sol`)

### Summary

`_checkIfDepositAmountExceedesCurrentLimit` applies a different comparison for ETH vs ERC20 assets. The ETH branch omits `+ amount`, allowing a single deposit to push `totalAssetDeposits` above `depositLimit`, after which every subsequent ETH deposit reverts with `MaximumDepositLimitReached`.

### Finding Description

In `contracts/LRTDepositPool.sol` lines 676–682:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← no + amount
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [1](#0-0) 

- **ERC20 path** (correct): `totalAssetDeposits + amount > depositLimit` — blocks any deposit that would breach the cap.
- **ETH path** (buggy): `totalAssetDeposits > depositLimit` — only blocks deposits *after* the cap is already exceeded; it does not prevent the deposit that *reaches* the cap from overshooting it.

**Concrete trigger sequence:**

1. `getTotalAssetDeposits(ETH_TOKEN) == depositLimit` (cap is exactly met).
2. Any user calls `depositETH` with any `msg.value > 0`.
3. The check evaluates `depositLimit > depositLimit` → `false` → deposit proceeds.
4. After the deposit, `getTotalAssetDeposits(ETH_TOKEN) == depositLimit + msg.value`.
5. Every future call to `depositETH` now evaluates `depositLimit + msg.value > depositLimit` → `true` → `MaximumDepositLimitReached` revert. [2](#0-1) 

### Impact Explanation

**Medium — Temporary freezing of new ETH deposits.**

The impact does **not** reach "Critical — Permanent freezing of funds" because:

- The depositor who triggered the overshoot received rsETH and can redeem it via `LRTWithdrawalManager.initiateWithdrawal` / `completeWithdrawal`. [3](#0-2) 
- ETH already in the pool is accessible to protocol operators (transfer to NodeDelegators, etc.).
- A MANAGER-role call to `updateAssetDepositLimit` can raise the cap and re-enable deposits. [4](#0-3) 

The real impact is: **all new ETH deposits are blocked until an admin manually raises the deposit limit**, which is a temporary but protocol-wide DoS on the ETH deposit path. Additionally, the configured deposit cap is silently violated by exactly one deposit.

### Likelihood Explanation

Moderate. The condition `totalAssetDeposits == depositLimit` is reachable in normal operation (the cap is a round number set by governance; deposits accumulate toward it). No special role, front-running, or external dependency is required — any ordinary user calling `depositETH` at the right moment triggers it. The state is sticky: once overshot, no automatic recovery exists.

### Recommendation

Add `amount` to the ETH branch to match the ERC20 logic:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [5](#0-4)
