### Title
Users Cannot Complete Their Own Withdrawals Without Privileged Operator Action — (`File: contracts/LRTWithdrawalManager.sol`)

### Summary
The `LRTWithdrawalManager` withdrawal flow contains a mandatory privileged intermediate step (`unlockQueue`) that only an `ASSET_TRANSFER_ROLE` or `OPERATOR_ROLE` holder can execute. If this role goes offline, all user withdrawal requests are permanently blocked: users' rsETH is already held in the contract but they cannot retrieve their underlying assets.

### Finding Description
The withdrawal process in `LRTWithdrawalManager` is a three-step flow:

1. **User** calls `initiateWithdrawal` — rsETH is pulled from the user into the contract.
2. **Operator/AssetTransfer role** calls `unlockQueue` — advances `nextLockedNonce[asset]`, burns rsETH, and redeems assets from the vault.
3. **User** calls `completeWithdrawal` — transfers the underlying asset to the user.

Step 2 is gated exclusively by `onlyAssetTransferOrOperatorRole`:

```solidity
// LRTWithdrawalManager.sol line 268–281
function unlockQueue(...) external nonReentrant onlySupportedAsset(asset) whenNotPaused onlyAssetTransferOrOperatorRole ...
```

Inside `_processWithdrawalCompletion` (called by `completeWithdrawal`), there is a hard check:

```solidity
// line 707
if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();
```

`nextLockedNonce[asset]` is only ever advanced inside `_unlockWithdrawalRequests`, which is only called from `unlockQueue`. There is no user-callable path to advance this nonce. Therefore, if the privileged operator never calls `unlockQueue`, every call to `completeWithdrawal` reverts with `WithdrawalLocked`, and the user's rsETH — already transferred to the contract at `initiateWithdrawal` — is irrecoverable without operator action.

### Impact Explanation
Users' rsETH is transferred into the contract at `initiateWithdrawal` (line 166). From that point, the user has no unilateral path to recover either their rsETH or the underlying asset. The funds are temporarily frozen for as long as the privileged role is inactive. This matches **Medium — Temporary freezing of funds**.

### Likelihood Explanation
The `ASSET_TRANSFER_ROLE` and `OPERATOR_ROLE` are off-chain operated keys. Any period of operator inactivity (key rotation, infrastructure failure, regulatory action, or simple negligence) directly freezes all queued user withdrawals. The scenario is realistic and does not require any attacker — it is triggered by ordinary user deposits combined with operator absence.

### Recommendation
Provide a permissionless fallback path that allows a user to self-unlock and complete their own withdrawal request once the standard time-based delay (`withdrawalDelayBlocks`) has elapsed and sufficient assets are available in the vault. The privileged `unlockQueue` can remain for batch efficiency, but users should not be wholly dependent on it to recover their own funds.

### Proof of Concept

1. Alice calls `initiateWithdrawal(stETH, 1e18, "")`. Her rsETH is transferred to the contract. [1](#0-0) 

2. The operator goes offline and never calls `unlockQueue`. `nextLockedNonce[stETH]` remains at its current value (e.g., `0`), while Alice's request nonce is `0`.

3. Alice calls `completeWithdrawal(stETH, "")`. Internally `_processWithdrawalCompletion` is invoked: [2](#0-1) 

4. The check at line 707 evaluates `0 >= 0` → `true` → reverts with `WithdrawalLocked`. [3](#0-2) 

5. Alice has no other callable function to advance `nextLockedNonce`. `unlockQueue` is the sole path and it is restricted: [4](#0-3) 

6. Alice's rsETH and the corresponding underlying asset are frozen until the operator resumes activity.
