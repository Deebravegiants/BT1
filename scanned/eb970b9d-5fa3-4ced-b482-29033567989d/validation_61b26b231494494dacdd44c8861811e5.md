### Title
Protocol Fee Minting in `_updateRsETHPrice()` Blocked When RSETH Daily Mint Limit Is Exhausted by User Deposits - (File: contracts/LRTOracle.sol)

### Summary

`LRTOracle._updateRsETHPrice()` mints protocol fees via `IRSETH.mint()`, which enforces `RSETH.checkDailyMintLimit`. The oracle's own fee-limit guard (`_checkAndUpdateDailyFeeMintLimit`) only checks against the oracle-local `maxFeeMintAmountPerDay` cap and is completely unaware of the RSETH token's separate `maxMintAmountPerDay` cap. When ordinary depositors exhaust the RSETH daily mint quota, every subsequent call to `updateRSETHPrice()` reverts, freezing protocol-fee collection and price updates for the remainder of the day.

### Finding Description

**Two independent daily-mint limits govern rsETH minting:**

1. **`RSETH.checkDailyMintLimit`** — applied inside `RSETH.mint()` to every caller, including both `LRTDepositPool._mintRsETH()` (user deposits) and `LRTOracle._updateRsETHPrice()` (protocol-fee minting). It tracks `currentPeriodMintedAmount` against `maxMintAmountPerDay`. [1](#0-0) 

2. **`LRTOracle._checkAndUpdateDailyFeeMintLimit`** — an oracle-local guard that tracks `currentPeriodMintedFeeAmount` against `maxFeeMintAmountPerDay`. It is checked *before* the actual `IRSETH.mint()` call but is entirely independent of the RSETH token's own cap. [2](#0-1) 

**The vulnerable sequence in `_updateRsETHPrice()`:**

```
_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee);  // passes oracle-local check
IRSETH(rsETHTokenAddress).mint(treasury, rsethAmountToMintAsProtocolFee); // reverts if RSETH cap exhausted
``` [3](#0-2) 

User deposits flow through `LRTDepositPool._mintRsETH()` → `IRSETH.mint()` → `checkDailyMintLimit`, consuming the same `currentPeriodMintedAmount` counter. [4](#0-3) 

Because `_checkAndUpdateDailyFeeMintLimit` never reads or accounts for `RSETH.maxMintAmountPerDay` or `RSETH.currentPeriodMintedAmount`, the oracle believes it is safe to mint fees while the RSETH token will unconditionally revert.

### Impact Explanation

When `RSETH.maxMintAmountPerDay` is set to a finite value and user deposits exhaust it:

- Every call to `updateRSETHPrice()` / `updateRSETHPriceAsManager()` reverts for the rest of the 24-hour window.
- Protocol fees accrued during that window are not minted to the treasury (temporary freeze of unclaimed yield).
- The stored `rsETHPrice` becomes stale, which affects all downstream consumers (withdrawal calculations, L2 rate propagation, etc.).

This maps to **Medium — Temporary freezing of unclaimed yield** and a temporary DoS on price updates.

### Likelihood Explanation

`maxMintAmountPerDay` is a configurable parameter set by the LRT manager. It is designed to be non-zero (a zero value causes `checkDailyMintLimit` to revert on every mint, making the token non-functional). Any day with high deposit volume relative to the configured cap triggers the condition. No privileged access or special conditions beyond normal user deposit activity are required.

### Recommendation

`_updateRsETHPrice()` should check the remaining RSETH daily mint capacity before attempting to mint protocol fees, and gracefully skip fee minting (rather than reverting) if the capacity is insufficient:

```solidity
uint256 remaining = IRSETH(rsETHTokenAddress).remainingDailyMintLimit();
if (rsethAmountToMintAsProtocolFee > remaining) {
    rsethAmountToMintAsProtocolFee = remaining; // or skip entirely
}
if (rsethAmountToMintAsProtocolFee > 0) {
    _checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee);
    IRSETH(rsETHTokenAddress).mint(treasury, rsethAmountToMintAsProtocolFee);
    emit FeeMinted(treasury, rsethAmountToMintAsProtocolFee);
}
```

Alternatively, the RSETH token's `checkDailyMintLimit` should exempt the oracle's fee-minting role from the daily cap, since fee minting is already rate-limited by `_checkAndUpdateDailyFeeMintLimit`.

### Proof of Concept

1. Admin sets `RSETH.maxMintAmountPerDay = 1000e18`.
2. Users deposit ETH/LSTs via `LRTDepositPool.depositETH()` / `depositAsset()`, minting 1000e18 rsETH total. `RSETH.currentPeriodMintedAmount` reaches 1000e18.
3. Staking rewards accrue; `totalETHInProtocol > previousTVL`.
4. Anyone calls `LRTOracle.updateRSETHPrice()`.
5. `_checkAndUpdateDailyFeeMintLimit(feeAmount)` passes (fee is within `maxFeeMintAmountPerDay`).
6. `IRSETH.mint(treasury, feeAmount)` reverts with `DailyMintLimitExceeded` because `currentPeriodMintedAmount + feeAmount > maxMintAmountPerDay`.
7. The entire `updateRSETHPrice()` call reverts. `rsETHPrice` is not updated, and protocol fees are not collected for this period.
8. This repeats for every call until the 24-hour window resets. [5](#0-4) [6](#0-5)
