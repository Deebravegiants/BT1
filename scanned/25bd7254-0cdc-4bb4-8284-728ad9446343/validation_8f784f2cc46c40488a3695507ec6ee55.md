### Title
Stale `rsETHPrice` Used in Deposit Calculations Allows New Depositors to Dilute Existing Holders' Accrued Yield â€” (`contracts/LRTDepositPool.sol`)

---

### Summary

`LRTDepositPool.getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()`, a **stored state variable** that is only updated when `updateRSETHPrice()` is explicitly called. No price update is triggered before deposits. When staking yield accrues between oracle updates, the stored price is lower than the true price, so new depositors receive more rsETH than they are entitled to, directly stealing accrued yield from existing rsETH holders.

---

### Finding Description

`LRTOracle.rsETHPrice` is a state variable set only inside `_updateRsETHPrice()`, which is invoked by the permissionless `updateRSETHPrice()` or the manager-only `updateRSETHPriceAsManager()`. Neither `depositETH` nor `depositAsset` in `LRTDepositPool` calls `updateRSETHPrice()` before computing the mint amount.

The mint calculation is:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.getAssetPrice(asset)` fetches a **live** price from an external price oracle, while `lrtOracle.rsETHPrice()` returns the **last stored** value. [1](#0-0) 

The stored `rsETHPrice` is computed inside `_updateRsETHPrice()` as:

```solidity
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

and written to state only at the end of that function. [2](#0-1) 

Between two consecutive `updateRSETHPrice()` calls, ETH staking rewards continuously increase `totalETHInProtocol` (via EigenLayer pod shares and LST rebasing), but `rsETHPrice` does not move. The gap between the live TVL and the stored price grows monotonically with time.

---

### Impact Explanation

**Theft of unclaimed yield â€” High.**

Concrete scenario (simplified, ignoring protocol fees):

| State | Total ETH | Total rsETH | True price | Stored `rsETHPrice` |
|---|---|---|---|---|
| After Alice deposits 100 ETH | 100 | 100 | 1.000 | 1.000 |
| After 1 day of staking yield | 101 | 100 | **1.010** | **1.000** (stale) |
| Bob deposits 100 ETH at stale price | 201 | **200** | â€” | 1.000 |
| After `updateRSETHPrice()` | 201 | 200 | **1.005** | 1.005 |

- Bob should have received `100 / 1.010 â‰ˆ 99.01 rsETH`; he received `100 / 1.000 = 100 rsETH`.
- After the price update, Alice's 100 rsETH is worth `100 Ã— 1.005 = 100.5 ETH` instead of the `101 ETH` she was owed.
- Bob captured **0.5 ETH of Alice's accrued yield** without contributing to it.

The loss to existing holders scales with (a) the size of the deposit and (b) the staleness of `rsETHPrice`. Because `updateRSETHPrice()` is permissionless, an attacker can deliberately delay calling it, let yield accumulate, deposit a large amount, and then trigger the update â€” all in a single block or across a short window.

---

### Likelihood Explanation

**Medium.** `updateRSETHPrice()` is permissionless and not called on every block, so there is always a non-zero staleness window. The protocol's own fee-minting logic depends on this function being called regularly, but nothing enforces it before deposits. An attacker can observe the on-chain TVL (via `_getTotalEthInProtocol` logic) versus the stored `rsETHPrice` and time a large deposit to maximise the spread. No privileged access is required; any EOA can execute this.

---

### Recommendation

Call `updateRSETHPrice()` (or an equivalent internal price-sync) at the beginning of `depositETH` and `depositAsset`, before `getRsETHAmountToMint` is invoked. This mirrors the VaultV2 pattern of calling `accrueInterest()` before any share-minting operation, ensuring the denominator always reflects the current protocol state.

---

### Proof of Concept

1. Observe that `depositETH` calls `_beforeDeposit` â†’ `getRsETHAmountToMint` with no prior price update. [3](#0-2) 
2. `getRsETHAmountToMint` divides by the stored `rsETHPrice`, not a freshly computed value. [4](#0-3) 
3. `rsETHPrice` is only written inside `_updateRsETHPrice`, which is never invoked from the deposit path. [5](#0-4) 
4. `_getTotalEthInProtocol` (called inside `_updateRsETHPrice`) reads live EigenLayer balances, confirming the true TVL diverges from the stored price as yield accrues. [6](#0-5) 
5. An attacker deposits a large amount while `rsETHPrice` is stale-low, receives excess rsETH, then calls `updateRSETHPrice()` to crystallise the gain â€” all without any privileged role.
