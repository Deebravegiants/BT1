### Title
Unbounded Nested Loops in `updateRSETHPrice()` Can Cause Permanent Oracle Freeze - (File: contracts/LRTOracle.sol)

### Summary

`LRTOracle.updateRSETHPrice()` is a publicly callable function that internally executes nested loops over `supportedAssetList` and `nodeDelegatorQueue`, with no on-chain maximum enforced on either array's length. As the protocol grows, this critical function can exceed the block gas limit, permanently freezing the price oracle and halting all deposits and withdrawals.

### Finding Description

`updateRSETHPrice()` is unrestricted (only `whenNotPaused`) and calls `_getTotalEthInProtocol()`, which loops over every supported asset: [1](#0-0) 

For each asset, it calls `ILRTDepositPool.getTotalAssetDeposits(asset)` → `getAssetDistributionData(asset)`, which itself loops over every NDC in `nodeDelegatorQueue`: [2](#0-1) 

For each NDC, `getAssetUnstaking()` is called, which fetches all queued EigenLayer withdrawals and iterates over their strategies: [3](#0-2) 

The total gas cost scales as `supportedAssets.length × nodeDelegatorQueue.length × queuedWithdrawals × strategies_per_withdrawal`, each step involving external calls. Neither `supportedAssetList` nor `maxNodeDelegatorLimit` has an on-chain hard cap:

- `supportedAssetList` grows via `addNewSupportedAsset()` with no maximum enforced: [4](#0-3) 
- `maxNodeDelegatorLimit` is admin-settable with no upper bound: [5](#0-4) 

The same nested loop is also triggered on every user deposit via `depositAsset()` / `depositETH()` → `_beforeDeposit()` → `getTotalAssetDeposits()`: [6](#0-5) 

### Impact Explanation

If `updateRSETHPrice()` reverts due to out-of-gas, `rsETHPrice` is never updated. All deposits rely on `getRsETHAmountToMint()` → `lrtOracle.rsETHPrice()`, and all withdrawals rely on `getExpectedAssetAmount()` → `lrtOracle.rsETHPrice()`. A stale or permanently frozen oracle effectively halts the entire protocol. Additionally, user deposits themselves loop over `nodeDelegatorQueue` and can independently hit the gas limit. This constitutes **temporary (escalating to permanent) freezing of funds** — Medium severity.

### Likelihood Explanation

The protocol is designed to grow: more LSTs are added as supported assets, more NDCs are deployed to scale ETH restaking, and each NDC accumulates queued EigenLayer withdrawals. No single admin action is malicious; the gas exhaustion emerges from legitimate protocol growth. `updateRSETHPrice()` is callable by anyone, so any user can trigger the revert once the threshold is crossed.

### Recommendation

1. Enforce an on-chain maximum for `supportedAssetList.length` in `_addNewSupportedAsset()`.
2. Enforce an on-chain hard cap for `maxNodeDelegatorLimit` (e.g., `require(maxNodeDelegatorLimit_ <= MAX_NDC_COUNT)`).
3. Consider paginating `_getTotalEthInProtocol()` or caching per-asset TVL snapshots updated lazily, rather than recomputing the full sum on every price update.
4. Add a gas-cost benchmark test that fails if the combined loop exceeds a safe fraction of the block gas limit.

### Proof of Concept

1. Protocol adds 10 supported LST assets via `addNewSupportedAsset()`.
2. Admin sets `maxNodeDelegatorLimit` to 20 and adds 20 NDCs.
3. Each NDC has 5 queued EigenLayer withdrawals with 2 strategies each.
4. Any user calls `updateRSETHPrice()`.
5. `_getTotalEthInProtocol()` executes: 10 assets × 20 NDCs × (1 `getAssetBalance` + 1 `getAssetUnstaking` with 5×2 inner iterations) = hundreds of external SLOAD-heavy calls.
6. Transaction reverts with out-of-gas; `rsETHPrice` is never updated; all subsequent deposits and withdrawals revert at the oracle price fetch.
