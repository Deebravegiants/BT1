### Title
`stakedButUnverifiedNativeETH` Excluded from `getETHDistributionData()` Causes rsETH Price Undervaluation — (File: `contracts/LRTDepositPool.sol`)

---

### Summary

`LRTDepositPool.getETHDistributionData()` omits the `stakedButUnverifiedNativeETH` balance tracked in each `NodeDelegator`. This is ETH that has been sent to beacon-chain validators but whose withdrawal credentials have not yet been verified with EigenLayer. During the gap between staking and verification, this ETH is invisible to the total-assets calculation, causing `LRTOracle._getTotalEthInProtocol()` to undercount protocol ETH, which in turn understates the rsETH price. Any depositor who mints rsETH while unverified ETH exists receives more rsETH than they are entitled to, diluting existing holders.

---

### Finding Description

**Step 1 — ETH disappears from accounting when staked.**

When `NodeDelegator.stake()` is called, ETH is sent to the beacon-chain deposit contract and `stakedButUnverifiedNativeETH` is incremented. At that moment:

- The ETH leaves `nodeDelegatorQueue[i].balance` (raw NDC ETH balance).
- It is not yet credited to EigenLayer pod shares (`getEffectivePodShares()` only reflects verified credentials).
- `stakedButUnverifiedNativeETH` is a public variable on `NodeDelegator` but is never queried by `getETHDistributionData()`.

**Step 2 — `getETHDistributionData()` sums six buckets, none of which cover unverified ETH.** [1](#0-0) 

```
ethLyingInDepositPool  = address(this).balance
ethLyingInNDCs         = Σ nodeDelegatorQueue[i].balance          // raw ETH, already gone
ethStakedInEigenLayer  = Σ getEffectivePodShares()                // only verified shares
ethUnstakingFromEigenLayer = Σ getAssetUnstaking(ETH_TOKEN)
ethLyingInConverter    = ILRTConverter.ethValueInWithdrawal()
ethLyingInUnstakingVault = lrtUnstakingVault.balance
```

`stakedButUnverifiedNativeETH` falls into none of these buckets.

**Step 3 — The undercount propagates to rsETH price.**

`LRTOracle._getTotalEthInProtocol()` calls `getTotalAssetDeposits()` → `getETHDistributionData()` for every supported asset and sums the result. [2](#0-1) 

The understated `totalETHInProtocol` is then used to compute `newRsETHPrice`: [3](#0-2) 

**Step 4 — New depositors exploit the understated price.**

`LRTDepositPool.getRsETHAmountToMint()` uses `rsETHPrice` from the oracle. With an understated price, the formula `rsethAmountToMint = depositAmount / rsETHPrice` yields more rsETH per ETH deposited than the depositor is entitled to. [4](#0-3) 

---

### Impact Explanation

Every rsETH minted against an understated price dilutes the claims of existing holders. The magnitude equals `stakedButUnverifiedNativeETH / totalETHInProtocol`. Given that the protocol regularly stakes 32 ETH increments to validators and the verification window can span multiple days, the unverified balance can be substantial (e.g., hundreds of ETH across multiple NDCs). This constitutes **theft of unclaimed yield** from existing rsETH holders — classified as **High**.

---

### Likelihood Explanation

The condition is triggered every time the operator calls `stake()` on any `NodeDelegator`. Credential verification requires a separate `verifyWithdrawalCredentials()` call after beacon-chain finality, which typically takes at least one to several days. During that window, `updateRSETHPrice()` can be called (it is permissionless), and any user can deposit ETH at the understated price. This is a **routine operational state**, not an edge case.

---

### Recommendation

In `getETHDistributionData()`, add a loop that queries `stakedButUnverifiedNativeETH` from each NDC and adds it to `ethStakedInEigenLayer` (or a dedicated return value):

```solidity
for (uint256 i; i < ndcsCount;) {
    ethLyingInNDCs += nodeDelegatorQueue[i].balance;
    ethStakedInEigenLayer += INodeDelegator(nodeDelegatorQueue[i]).getEffectivePodShares();
+   ethStakedInEigenLayer += INodeDelegator(nodeDelegatorQueue[i]).stakedButUnverifiedNativeETH();
    ethUnstakingFromEigenLayer += INodeDelegator(nodeDelegatorQueue[i])
        .getAssetUnstaking(LRTConstants.ETH_TOKEN);
    unchecked { ++i; }
}
``` [5](#0-4) 

---

### Proof of Concept

1. Protocol has 1000 ETH verified in EigenLayer; rsETH supply = 1000; price = 1.0 ETH/rsETH.
2. Operator calls `NodeDelegator.stake()` for 100 ETH → `stakedButUnverifiedNativeETH = 100`.
3. `updateRSETHPrice()` is called → `_getTotalEthInProtocol()` returns 1000 (misses the 100) → `rsETHPrice = 1000/1000 = 1.0` (should be `1100/1000 = 1.1`).
4. Attacker deposits 100 ETH → receives `100 / 1.0 = 100 rsETH` (should receive `100 / 1.1 ≈ 90.9 rsETH`).
5. After `verifyWithdrawalCredentials()` is called, `_getTotalEthInProtocol()` returns 1200 (1100 original + 100 new deposit); rsETH supply = 1100; price = `1200/1100 ≈ 1.09`.
6. Original holders' rsETH is now worth less than 1.1 ETH/rsETH — the attacker captured ~9 ETH of yield that belonged to them.

    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {

            }
        }
