### Title
`assetsCommitted` Decremented by Stale Original Amount Instead of Actual Payout, Creating Phantom Withdrawal Capacity — (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary

In `LRTWithdrawalManager._unlockWithdrawalRequests`, `assetsCommitted[asset]` is decremented by the **original** committed amount (`request.expectedAssetAmount` before it is overwritten), while the actual payout written back is `payoutAmount` — the minimum of the original and the current market-rate return. When rsETH price has fallen since the request was initiated, `payoutAmount < originalAmount`, and `assetsCommitted` is reduced by more than what is actually disbursed. This inflates the result of `getAvailableAssetAmount`, creating phantom withdrawal capacity that allows new users to queue withdrawals against assets that do not exist in the vault.

---

### Finding Description

`initiateWithdrawal` records the expected asset amount at the current rsETH price and adds it to `assetsCommitted`:

```
assetsCommitted[asset] += expectedAssetAmount;   // line 173
``` [1](#0-0) 

Later, when the operator calls `unlockQueue`, `_unlockWithdrawalRequests` computes the actual payout as `min(originalAmount, currentReturn)` and then:

1. Decrements `assetsCommitted` by the **old** `request.expectedAssetAmount` (line 802).
2. Overwrites `request.expectedAssetAmount` with `payoutAmount` (line 804).

```
assetsCommitted[asset] -= request.expectedAssetAmount;  // uses OLD value
request.expectedAssetAmount = payoutAmount;             // then updates
``` [2](#0-1) 

`_calculatePayoutAmount` returns `min(request.expectedAssetAmount, currentReturn)`: [3](#0-2) 

Whenever `currentReturn < originalAmount` (rsETH price has dropped), `assetsCommitted` is reduced by `originalAmount` but only `payoutAmount` is redeemed from the vault. The vault balance (`assetLyingUnstakingVault`) falls by `payoutAmount`, but `assetsCommitted` falls by `originalAmount`.

`getAvailableAssetAmount` computes:

```
availableAssetAmount = totalAssets - assetsCommitted[asset]
``` [4](#0-3) 

`totalAssets` is sourced from `getTotalAssetDeposits`, which explicitly includes `assetLyingUnstakingVault`: [5](#0-4) 

Net effect after one unlock cycle where `payoutAmount < originalAmount`:

```
getAvailableAssetAmount increases by (originalAmount - payoutAmount)
```

This phantom capacity is real enough to pass the guard in `initiateWithdrawal`:

```
if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();
``` [6](#0-5) 

New users who queue withdrawals against this phantom capacity have their rsETH transferred into the contract and their requests recorded. When the operator later tries to unlock those requests, the vault does not hold the required assets, so `unlockQueue` cannot process them. The users' rsETH sits locked in the withdrawal manager indefinitely.

---

### Impact Explanation

**Medium — Temporary (potentially permanent) freezing of funds.**

Users who initiate withdrawals using phantom capacity have their rsETH locked in `LRTWithdrawalManager` with no path to completion: the vault is short by exactly `originalAmount - payoutAmount` per affected unlock cycle. If the vault is never topped up to cover the shortfall, the freeze becomes permanent. The rsETH is not burned until `unlockQueue` succeeds, so it cannot be recovered by the user.

---

### Likelihood Explanation

**Medium.** rsETH price relative to any LST fluctuates continuously. Any market event that causes the rsETH/LST rate to decline between the time a withdrawal is initiated and the time the operator calls `unlockQueue` triggers the discrepancy. No special permissions or oracle manipulation are required — ordinary price movement is sufficient. The effect compounds across every unlock cycle where a price decline occurred.

---

### Recommendation

Decrement `assetsCommitted` by `payoutAmount` (the actual disbursement), not by the stale `request.expectedAssetAmount`:

```solidity
// Before (wrong):
assetsCommitted[asset] -= request.expectedAssetAmount;
request.expectedAssetAmount = payoutAmount;

// After (correct):
assetsCommitted[asset] -= payoutAmount;
request.expectedAssetAmount = payoutAmount;
```

This ensures `assetsCommitted` always reflects the true outstanding obligation, keeping `getAvailableAssetAmount` accurate.

---

### Proof of Concept

1. rsETH/stETH rate at request time: 1 rsETH = 1.05 stETH. User calls `initiateWithdrawal(stETH, 1e18 rsETH)`. `expectedAssetAmount = 1.05e18`. `assetsCommitted[stETH] += 1.05e18`.

2. rsETH price drops. At unlock time: 1 rsETH = 1.00 stETH. `payoutAmount = min(1.05e18, 1.00e18) = 1.00e18`.

3. `_unlockWithdrawalRequests` executes:
   - `assetsCommitted[stETH] -= 1.05e18` (old value) → `assetsCommitted` drops by 1.05e18
   - `request.expectedAssetAmount = 1.00e18`
   - Vault redeems `1.00e18` stETH → `totalAssets` drops by 1.00e18

4. `getAvailableAssetAmount` = `(totalAssets - 1.00e18) - (assetsCommitted - 1.05e18)` = original + **0.05e18 phantom capacity**.

5. A second user calls `initiateWithdrawal(stETH, X rsETH)` where `expectedAssetAmount ≤ 0.05e18`. The check passes. Their rsETH is transferred in and `assetsCommitted[stETH] += 0.05e18`.

6. The vault holds 0 stETH for this asset. `unlockQueue` cannot process the second user's request. Their rsETH is frozen in the contract. [7](#0-6)
