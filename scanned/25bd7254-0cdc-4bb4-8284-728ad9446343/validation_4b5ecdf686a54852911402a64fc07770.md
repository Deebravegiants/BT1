## Analysis

Let me trace the full attack path through the production code.

### Step 1: How TVL is read live vs. price stored

`LRTOracle.rsETHPrice` is a **stored** value, updated only when `updateRSETHPrice()` is called. [1](#0-0) 

`_getTotalEthInProtocol()` reads **live** on-chain state every time it is called, including `getEffectivePodShares()` on each NodeDelegator. [2](#0-1) 

`getEffectivePodShares()` returns `stakedButUnverifiedNativeETH + withdrawableShare`, where `withdrawableShare` is fetched live from EigenLayer's `DelegationManager.getWithdrawableShares()`. [3](#0-2) 

### Step 2: When does TVL increase without a price update?

When an EigenPod checkpoint finalizes, EigenLayer calls `recordBeaconChainETHBalanceUpdate` on the `EigenPodManager`, which increases `podOwnerDepositShares` for the NodeDelegator. This is immediately reflected in `getWithdrawableShares()` → `getEffectivePodShares()` → `getTotalAssetDeposits()` → `_getTotalEthInProtocol()`. The stored `rsETHPrice` is **not** updated atomically. [4](#0-3) 

### Step 3: Mint formula uses stale price

`getRsETHAmountToMint` divides by `lrtOracle.rsETHPrice()` — the stale stored value: [5](#0-4) 

If `rsETHPrice` is stale-low (TVL has grown but price not yet updated), the depositor receives **more rsETH** than the current TVL justifies.

### Step 4: `updateRSETHPrice()` is permissionless

Anyone can call it: [6](#0-5) 

### Step 5: The attack sequence (no front-running required)

The attacker does not need to race against anyone in the mempool. They can execute all steps themselves:

1. Observe checkpoint finalization on-chain (emitted `CheckpointFinalized` event).
2. Call `depositETH()` — mints rsETH at the stale (pre-reward) price.
3. Call `updateRSETHPrice()` — price rises to reflect the new TVL.
4. Initiate withdrawal — rsETH is now worth more ETH than was deposited.

### Step 6: Math

Let T = TVL before rewards, R = reward, S = rsETH supply, D = attacker deposit.

- Stale price: `T/S`
- Attacker mints: `D·S/T` rsETH
- After price update: new price = `(T+R+D)·T / (S·(T+D))`
- Attacker's rsETH value: `D·(T+R+D)/(T+D)` ETH
- **Attacker profit: `R·D/(T+D)` ETH**
- **Existing holder loss: `R·D/(T+D)` ETH** (they receive `T·R/(T+D)` instead of `R`)

### Step 7: Does `pricePercentageLimit` block this?

The check at line 256–266 reverts `updateRSETHPrice()` for non-managers if the price increase exceeds the limit. [7](#0-6) 

The attacker's deposit D **dilutes** the price increase to `R/(T+D)`. By making D large enough, the attacker keeps the price increase below the threshold. If `pricePercentageLimit == 0`, there is no protection at all. Even if the threshold is hit, the manager will eventually call `updateRSETHPriceAsManager()`, and the attacker still profits from having deposited at the stale price. [8](#0-7) 

### Step 8: Economic constraints

The attacker's profit per unit of capital is `R/(T+D) ≈ R/T` (the protocol's per-period yield rate). Capital must be locked for the EigenLayer withdrawal delay (~7 days). The attack is only profitable for actors with idle capital whose opportunity cost is below the protocol yield rate. This limits realistic likelihood but does not eliminate the vulnerability.

### Title
Stale `rsETHPrice` After EigenPod Checkpoint Finalization Allows Yield Theft via Deposit-Before-PriceUpdate — (`contracts/LRTOracle.sol`, `contracts/LRTDepositPool.sol`)

### Summary
`rsETHPrice` is a stored value updated only by explicit calls to `updateRSETHPrice()`. When an EigenPod checkpoint finalizes, the protocol's live TVL (read via `getEffectivePodShares()`) increases immediately, but the stored price does not. An attacker can deposit ETH at the stale price, then call the permissionless `updateRSETHPrice()`, capturing a portion of the consensus rewards that should have accrued proportionally to all existing rsETH holders.

### Finding Description
`LRTOracle._getTotalEthInProtocol()` reads live EigenLayer share balances on every call, while `rsETHPrice` is a cached value. After a checkpoint finalizes and `podOwnerDepositShares` increases, there is a window — potentially spanning multiple blocks — during which `getTotalAssetDeposits(ETH)` returns a higher value than what `rsETHPrice` reflects. `depositETH()` uses `rsETHPrice` as the denominator for minting, so any deposit during this window mints excess rsETH. The attacker then calls `updateRSETHPrice()` (permissionless, no role required) to crystallize the higher price, and eventually withdraws at a profit.

The `pricePercentageLimit` guard is bypassable: the attacker's own deposit dilutes the price increase, keeping it below the threshold. If `pricePercentageLimit` is zero, there is no guard at all.

### Impact Explanation
Existing rsETH holders receive less yield than they are entitled to. The attacker extracts `R·D/(T+D)` ETH of yield, where R is the checkpoint reward, D is the attacker's deposit, and T is the pre-reward TVL. For large D, this approaches the full reward R.

### Likelihood Explanation
Low-to-Medium. The attack is permissionless and requires no privileged access. It requires capital proportional to the TVL to be economically meaningful, and the capital must be locked for the EigenLayer withdrawal delay. Profitable only for actors with idle capital. Checkpoints are finalized periodically (at least daily in active protocols), so the window recurs regularly.

### Recommendation
1. **Atomic price update on deposit**: Call `_updateRsETHPrice()` at the start of `depositETH()` / `depositAsset()` to ensure the price is always fresh before minting.
2. **Alternatively**: Compute rsETH to mint using live TVL rather than the stored `rsETHPrice`, so the mint ratio always reflects current state.
3. **Minimum**: Emit an event or revert deposits when the live TVL diverges from `rsethSupply * rsETHPrice` by more than a configurable threshold.

### Proof of Concept

```solidity
// Fork test (Foundry, mainnet fork)
function testYieldTheft() public {
    // 1. Setup: protocol has T ETH TVL, S rsETH supply
    uint256 T = 100_000 ether;
    uint256 R = 10 ether; // checkpoint reward

    // 2. Simulate checkpoint finalization: EigenPod calls recordBeaconChainETHBalanceUpdate
    //    podOwnerDepositShares increases by R — live TVL is now T+R
    //    rsETHPrice is still T/S (stale)
    vm.prank(address(eigenPod));
    eigenPodManager.recordBeaconChainETHBalanceUpdate(address(nodeDelegator), T, int256(R));

    // 3. Attacker deposits D ETH at stale price
    uint256 D = 1_000_000 ether;
    vm.deal(attacker, D);
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: D}(0, "");
    uint256 attackerRsETH = rsETH.balanceOf(attacker);

    // 4. Attacker calls updateRSETHPrice — price rises
    uint256 priceBefore = lrtOracle.rsETHPrice();
    lrtOracle.updateRSETHPrice();
    uint256 priceAfter = lrtOracle.rsETHPrice();
    assertGt(priceAfter, priceBefore);

    // 5. Attacker's rsETH is now worth more than D ETH
    uint256 attackerETHValue = attackerRsETH * priceAfter / 1e18;
    assertGt(attackerETHValue, D); // attacker profit > 0

    // 6. Existing holders received less than R ETH of yield
    // (their per-rsETH value increased by less than R/S)
}
```
