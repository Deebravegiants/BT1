### Title
Missing Staleness Check on Cross-Chain Rate Allows Stale Rate to Drive agETH Over-Minting — (`contracts/agETH/AGETHPoolV3.sol`, `contracts/cross-chain/CrossChainRateReceiver.sol`)

---

### Summary

`CrossChainRateReceiver` stores a `lastUpdated` timestamp when a rate is received via LayerZero, but neither `CrossChainRateReceiver.getRate()` nor `AGETHPoolV3.viewSwapAgETHAmountAndFee()` ever validate that timestamp against a maximum staleness threshold. If rate pushes stop for any reason, `AGETHPoolV3` continues minting agETH at an arbitrarily outdated rate, leading to over-minting and protocol insolvency.

---

### Finding Description

The cross-chain rate pipeline works as follows:

1. On mainnet, `AGETHMultiChainRateProvider.updateRate()` reads the live agETH/ETH rate and sends it via LayerZero to destination-chain receivers. This call is `payable` — it requires the caller to supply ETH for LZ fees on every invocation. [1](#0-0) 

2. On the destination chain, `CrossChainRateReceiver.lzReceive()` stores the received rate and records `lastUpdated = block.timestamp`. [2](#0-1) 

3. `CrossChainRateReceiver.getRate()` returns the stored `rate` with **no staleness check** — `lastUpdated` is recorded but never validated. [3](#0-2) 

4. `AGETHPoolV3.viewSwapAgETHAmountAndFee()` calls `getRate()` and uses the result directly to compute how many agETH tokens to mint, again with **no staleness check**. [4](#0-3) 

5. `AGETHPoolV3.deposit()` calls `viewSwapAgETHAmountAndFee()` and immediately mints agETH at the returned amount. [5](#0-4) 

The `lastUpdated` field exists in `CrossChainRateReceiver` but is purely informational — it is never read by `getRate()` or by any caller in the minting path. [6](#0-5) 

---

### Impact Explanation

agETH is a liquid restaking token whose exchange rate vs ETH increases monotonically over time as staking rewards accrue. The minting formula is:

```
agETHAmount = amountAfterFee * 1e18 / agETHToETHrate
``` [7](#0-6) 

If the stored rate is **lower** than the true current rate (i.e., it is stale from before rewards accrued), the denominator is smaller than it should be, so `agETHAmount` is **larger** than it should be. Users receive more agETH per ETH deposited than the protocol's backing can support. Over time this makes agETH undercollateralized, causing insolvency and freezing withdrawals.

Impact: **Critical — Protocol insolvency / permanent freezing of funds.**

---

### Likelihood Explanation

`updateRate()` is a permissionless but **manually triggered, ETH-funded** function. There is no on-chain keeper, no heartbeat enforcement, and no circuit-breaker that pauses deposits when the rate is stale. Any of the following realistic conditions causes a prolonged stale rate:

- The operator's ETH balance for LZ fees is depleted.
- A LayerZero outage delays message delivery.
- The operator simply fails to call `updateRate()` for an extended period.

Because agETH's rate drifts upward continuously, even a few weeks of staleness creates a meaningful over-minting window. The exploit requires no privileged access — any user calling `deposit()` during the stale window benefits (intentionally or not).

---

### Recommendation

Add a configurable `maxStaleness` parameter to `CrossChainRateReceiver` and revert in `getRate()` if `block.timestamp - lastUpdated > maxStaleness`:

```solidity
// CrossChainRateReceiver.sol
uint256 public maxStaleness; // e.g. 24 hours

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

This ensures `AGETHPoolV3.deposit()` reverts automatically when the oracle is stale, protecting the protocol from over-minting without requiring any changes to `AGETHPoolV3` itself.

---

### Proof of Concept

```solidity
// Fork test on destination chain (e.g. Arbitrum)
function test_staleRateCausesOverMint() external {
    // 1. Warp 30 days forward without calling updateRate()
    vm.warp(block.timestamp + 30 days);

    // 2. Actual agETH/ETH rate has increased (e.g. from 1.05e18 to 1.10e18)
    //    but AGETHRateReceiver still holds the old 1.05e18 rate.
    uint256 storedRate = agETHRateReceiver.getRate(); // returns stale 1.05e18

    // 3. User deposits 1 ETH — deposit does NOT revert
    uint256 balanceBefore = agETH.balanceOf(user);
    vm.prank(user);
    agETHPoolV3.deposit{value: 1 ether}("ref");
    uint256 minted = agETH.balanceOf(user) - balanceBefore;

    // 4. User received ~0.952 agETH (1e18/1.05e18) instead of ~0.909 agETH (1e18/1.10e18)
    //    The excess ~0.043 agETH per ETH is unbacked — protocol is undercollateralized.
    assertGt(minted, 1 ether * 1e18 / 1.10e18);
}
```
