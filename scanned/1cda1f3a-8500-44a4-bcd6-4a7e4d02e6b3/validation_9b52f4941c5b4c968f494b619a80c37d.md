### Title
Token Fee Mis-Accounting in `RSETHPoolNoWrapper.deposit()` Permanently Freezes ETH in Pool - (File: contracts/pools/RSETHPoolNoWrapper.sol)

### Summary

`RSETHPoolNoWrapper.deposit(address token, ...)` incorrectly credits token-denominated fees into `feeEarnedInETH` instead of `feeEarnedInToken[token]`. This inflates the ETH fee counter with token-unit values, causing `getETHBalanceMinusFees()` to underflow once `feeEarnedInETH` exceeds `address(this).balance`. Every subsequent call to `bridgeAssets()` and `bridgeAssetsViaNativeBridge()` reverts, permanently trapping all ETH deposited by users inside the pool with no recovery path.

### Finding Description

`RSETHPoolNoWrapper` is the deposit pool for chains without a canonical rsETH wrapper (e.g., Unichain). It accepts both native ETH and supported ERC-20 tokens (e.g., wstETH) from users, giving them rsETH in return. The ETH collected is held in the contract until a BRIDGER_ROLE account calls `bridgeAssets()` or `bridgeAssetsViaNativeBridge()` to move it to L1.

The token deposit path contains a critical accounting error:

```solidity
// RSETHPoolNoWrapper.sol – token deposit
function deposit(address token, uint256 amount, string memory referralId)
    external nonReentrant whenNotPaused onlySupportedToken(token)
{
    ...
    IERC20(token).safeTransferFrom(msg.sender, address(this), amount);
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);

    feeEarnedInETH += fee;          // ← BUG: fee is in token units, not ETH
    // should be: feeEarnedInToken[token] += fee;

    rsETH.safeTransfer(msg.sender, rsETHAmount);
    ...
}
```

`viewSwapRsETHAmountAndFee(amount, token)` computes `fee = amount * feeBps / 10_000`, which is denominated in the deposited token (e.g., wstETH wei), not in ETH wei. Adding this value to `feeEarnedInETH` corrupts the ETH fee counter.

`getETHBalanceMinusFees()` then subtracts this inflated counter from the contract's native ETH balance:

```solidity
function getETHBalanceMinusFees() public view returns (uint256) {
    return address(this).balance - feeEarnedInETH;
}
```

Once `feeEarnedInETH > address(this).balance`, this subtraction underflows (Solidity 0.8 checked arithmetic), reverting every call that depends on it:

```solidity
// bridgeAssets – checks available ETH
if (getETHBalanceMinusFees() - msg.value < amount) revert InsufficientETHBalance();

// bridgeAssetsViaNativeBridge – uses the value directly
uint256 ethBalanceMinusFees = getETHBalanceMinusFees();
IL2Messenger(messenger).sendETHToL1ViaBridge{value: ethBalanceMinusFees}(...);
```

`withdrawFees(address receiver)` also attempts to send `feeEarnedInETH` wei of ETH, which will fail if the contract holds less ETH than the inflated counter. There is no `recoverETH()` or emergency drain function in `RSETHPoolNoWrapper`, so the ETH is permanently locked.

### Impact Explanation

All native ETH deposited by users into `RSETHPoolNoWrapper` accumulates in the contract and can only leave via `bridgeAssets()` or `bridgeAssetsViaNativeBridge()`. Once the token-fee inflation causes `feeEarnedInETH > address(this).balance`, both bridging paths revert unconditionally. The ETH cannot be recovered without a contract upgrade. This constitutes **permanent freezing of funds** held in the pool.

Additionally, `feeEarnedInToken[token]` is never incremented, so `withdrawFees(receiver, token)` always transfers 0 tokens, silently losing all token-denominated fee revenue.

### Likelihood Explanation

The trigger is any ordinary user calling `deposit(token, amount, referralId)`. No special role or privileged access is required. On a chain where wstETH deposits are common and ETH deposits are sparse, `feeEarnedInETH` will exceed `address(this).balance` after a modest volume of token deposits, making this condition easy to reach in normal operation.

### Recommendation

Replace the incorrect fee accumulation in the token deposit path:

```solidity
// Before (wrong):
feeEarnedInETH += fee;

// After (correct):
feeEarnedInToken[token] += fee;
```

This mirrors the correct pattern used in `RSETHPoolV3.sol` and `RSETHPoolV3ExternalBridge.sol`, which both use `feeEarnedInToken[token] += fee` for token deposits.

### Proof of Concept

1. Deploy `RSETHPoolNoWrapper` with `feeBps = 100` (1%) and add wstETH as a supported token.
2. Ensure the pool holds 0 ETH (no ETH deposits yet) and sufficient rsETH for payouts.
3. User calls `deposit(wstETH, 1000e18, "")`.
   - `fee = 1000e18 * 100 / 10_000 = 10e18`
   - `feeEarnedInETH += 10e18` → `feeEarnedInETH = 10e18`
4. `address(this).balance = 0` (no ETH was deposited).
5. BRIDGER calls `bridgeAssets(amount, minAmount, nativeFee)`:
   - `getETHBalanceMinusFees()` → `0 - 10e18` → **underflow revert**.
6. BRIDGER calls `bridgeAssetsViaNativeBridge()`:
   - Same underflow → **revert**.
7. ETH deposited later by ETH depositors is also permanently trapped once `feeEarnedInETH` exceeds the ETH balance. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

    function viewSwapRsETHAmountAndFee(
        uint256 amount,
        address token
    )
        public
        view
        onlySupportedToken(token)
        returns (uint256 rsETHAmount, uint256 fee)
    {
        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // rate of token in ETH
        uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
    }

    {
        // Exclude msg.value so reserved fees can’t be accidentally consumed
        if (getETHBalanceMinusFees() - msg.value < amount) {
            revert InsufficientETHBalance();

            revert InvalidMinAmount();
