### Title
Fee Bypass via Token Deposit Path When `tokenFeeBps` Defaults to Zero - (File: contracts/pools/RSETHPool.sol)

### Summary
`RSETHPool.sol` maintains two separate fee rates: `feeBps` for ETH deposits and `tokenFeeBps[token]` for token deposits. When a new token is added via `addSupportedToken`, `tokenFeeBps[token]` is never initialized and defaults to `0`. Any depositor can bypass the ETH deposit fee entirely by converting ETH to a supported token externally and depositing the token instead, paying zero fee while an equivalent ETH depositor pays `feeBps`.

### Finding Description
`RSETHPool.sol` declares two distinct fee storage variables:

- `feeBps` — applied to all ETH deposits (line 43)
- `tokenFeeBps` — a per-token mapping applied to token deposits (line 88)

The ETH deposit fee calculation:
```solidity
fee = amount * feeBps / 10_000;  // RSETHPool.sol:312
```

The token deposit fee calculation:
```solidity
uint256 feeBpsForToken = tokenFeeBps[token];
fee = amount * feeBpsForToken / 10_000;  // RSETHPool.sol:335-336
```

`tokenFeeBps[token]` is set only via the separate admin function `setTokenFeeBps` (lines 583–594). The `addSupportedToken` function (mirrored from `AGETHPoolV3.sol:272-286`) only sets `supportedTokenOracle[token]` and pushes to `supportedTokenList` — it never initializes `tokenFeeBps[token]`. Solidity's default for unset `uint256` mapping entries is `0`, so every newly added token has a zero fee rate until the admin explicitly calls `setTokenFeeBps`.

During this window (and permanently if the admin never sets it), the two deposit paths for economically equivalent value are:

| Path | Action | Fee paid |
|---|---|---|
| ETH deposit | Send 1 ETH directly | `1e18 * feeBps / 10_000` ETH |
| Token deposit | Swap 1 ETH → stETH externally, deposit stETH | `0` |

Both paths produce the same rsETH/wrsETH output (adjusted by oracle rate), but the token path pays zero fee.

### Impact Explanation
The protocol fails to collect fees consistently across deposit paths. ETH depositors are overcharged relative to token depositors for economically equivalent deposits. Protocol fee revenue is reduced or eliminated for any token whose `tokenFeeBps` has not been explicitly set. This matches **Low — Contract fails to deliver promised returns, but doesn't lose value**: no user funds are stolen, but the protocol does not collect the fees it is designed to charge.

### Likelihood Explanation
Any unprivileged depositor can observe that a supported token has `tokenFeeBps = 0` (public mapping) and immediately route their deposit through the token path. The window exists from the moment `addSuntedToken` is called until `setTokenFeeBps` is called. If the admin never calls `setTokenFeeBps`, the bypass is permanent. No special privileges, front-running, or external protocol compromise is required.

### Recommendation
Initialize `tokenFeeBps[token]` inside `addSupportedToken` to the current `feeBps` value (or require an explicit fee parameter), so newly added tokens are never silently fee-free:

```solidity
function addSupportedToken(address token, address oracle, uint256 _feeBps) external onlyRole(DEFAULT_ADMIN_ROLE) {
    // ... existing checks ...
    supportedTokenList.push(token);
    supportedTokenOracle[token] = oracle;
    tokenFeeBps[token] = _feeBps;   // initialize fee at addition time
    emit AddSupportedToken(token);
}
```

Alternatively, fall back to `feeBps` when `tokenFeeBps[token]` is zero:
```solidity
uint256 feeBpsForToken = tokenFeeBps[token] > 0 ? tokenFeeBps[token] : feeBps;
```

### Proof of Concept
1. Admin calls `addSupportedToken(stETH, stETHOracle)` on `RSETHPool.sol`. `tokenFeeBps[stETH]` is now `0`.
2. Alice wants to deposit 1 ETH worth of value. She checks `tokenFeeBps[stETH]` on-chain — it is `0`.
3. Alice swaps 1 ETH → ~1 stETH on an external DEX (e.g., Curve).
4. Alice calls `deposit(stETH, 1e18, "ref")`. Fee computed: `1e18 * 0 / 10_000 = 0`. She receives the full rsETH amount with zero fee deducted.
5. Bob deposits 1 ETH directly via `deposit("ref")`. Fee computed: `1e18 * feeBps / 10_000 > 0`. Bob receives fewer rsETH than Alice for the same economic input.

Alice pays zero protocol fee; Bob pays the full fee. The protocol collects no fee from Alice's deposit. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
