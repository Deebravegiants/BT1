### Title
Pending Withdrawal Requests Permanently Frozen When Asset Is Removed From Supported List - (File: contracts/LRTWithdrawalManager.sol)

---

### Summary

The `unlockQueue()` function in `LRTWithdrawalManager` carries the `onlySupportedAsset(asset)` modifier. If an asset is removed from the supported list after users have already called `initiateWithdrawal()` — transferring their rsETH into the contract — those withdrawal requests can never be unlocked. `completeWithdrawal()` will permanently revert with `WithdrawalLocked`, freezing the users' rsETH in the contract with no recovery path.

---

### Finding Description

The withdrawal lifecycle in `LRTWithdrawalManager` is a two-phase process:

**Phase 1 — `initiateWithdrawal()`**: The user's rsETH is pulled into the contract and a `WithdrawalRequest` is stored. The request starts in a "locked" state (`usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]`). [1](#0-0) 

**Phase 2 — `unlockQueue()`**: An operator must call this to advance `nextLockedNonce[asset]` past the user's nonce, making the request claimable. This function carries `onlySupportedAsset(asset)`: [2](#0-1) 

**Phase 3 — `completeWithdrawal()`**: The user calls this to receive their asset. It checks that the request has been unlocked: [3](#0-2) 

The critical gap: `LRTConfig.removeSupportedAsset()` only guards against removal when `getTotalAssetDeposits(asset)` in the deposit pool exceeds `maxNegligibleAmount`. It does **not** check for pending withdrawal requests in `LRTWithdrawalManager` (`assetsCommitted[asset]` or `nextLockedNonce[asset] < nextUnusedNonce[asset]`): [4](#0-3) 

Once assets have been moved from the deposit pool through node delegators into the unstaking vault (the normal operational flow), `getTotalAssetDeposits` can return zero for that asset, allowing `removeSupportedAsset()` to succeed even while users have live, locked withdrawal requests. After removal, `unlockQueue()` reverts with `AssetNotSupported` on every call, `nextLockedNonce[asset]` is never advanced, and `completeWithdrawal()` permanently reverts with `WithdrawalLocked`. The rsETH already held by the contract has no exit path.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Users' rsETH is irrecoverably locked inside `LRTWithdrawalManager`. There is no admin escape hatch, no cancel-withdrawal function, and no alternative path to recover the rsETH once the asset is delisted and the queue is frozen. The locked rsETH represents real economic value (it was minted against deposited ETH/LSTs).

---

### Likelihood Explanation

Asset migration or deprecation is a routine protocol operation (e.g., moving from one LST to another, or winding down a strategy). The admin executing `removeSupportedAsset()` would naturally check the deposit pool balance — which the guard enforces — but has no on-chain signal that `LRTWithdrawalManager` holds pending, unprocessed withdrawal requests for that asset. This is an easy mistake to make during a planned asset sunset, making the scenario realistic without requiring any malicious intent.

---

### Recommendation

Add a check in `LRTConfig.removeSupportedAsset()` (or in a pre-removal hook) that verifies no pending withdrawal requests exist for the asset:

```solidity
// In removeSupportedAsset(), before deleting isSupportedAsset[asset]:
address withdrawalManager = getContract(LRTConstants.LRT_WITHDRAWAL_MANAGER);
if (ILRTWithdrawalManager(withdrawalManager).assetsCommitted(asset) > 0) {
    revert PendingWithdrawalsExist();
}
```

Alternatively, remove the `onlySupportedAsset(asset)` modifier from `unlockQueue()` so that already-queued requests for a delisted asset can still be processed to completion.

---

### Proof of Concept

1. Alice calls `initiateWithdrawal(stETH, rsETHAmount, "")`. Her rsETH is transferred to `LRTWithdrawalManager`; `assetsCommitted[stETH]` increases; her nonce is recorded in `userAssociatedNonces[stETH][alice]`. [5](#0-4) 

2. Protocol operators move all stETH from the deposit pool through node delegators into the unstaking vault. `getTotalAssetDeposits(stETH)` in the deposit pool now returns ≤ `maxNegligibleAmount`.

3. Admin calls `LRTConfig.removeSupportedAsset(stETH, index)`. The guard passes; `isSupportedAsset[stETH]` is deleted. [6](#0-5) 

4. Operator attempts `unlockQueue(stETH, ...)`. The `onlySupportedAsset(stETH)` modifier reverts with `AssetNotSupported`. `nextLockedNonce[stETH]` is never advanced. [7](#0-6) 

5. Alice calls `completeWithdrawal(stETH, "")`. `_processWithdrawalCompletion` checks `usersFirstWithdrawalRequestNonce >= nextLockedNonce[stETH]` — true forever — and reverts with `WithdrawalLocked`. [8](#0-7) 

6. Alice's rsETH is permanently locked in `LRTWithdrawalManager` with no recovery mechanism.

```text
    function _processWithdrawalCompletion(address asset, address user, string calldata referralId) internal {
        if (userAssociatedNonces[asset][user].empty()) {
            revert NoWithdrawalRequests(user, asset);
        }

        // Retrieve and remove the oldest withdrawal request for the user.
        uint256 usersFirstWithdrawalRequestNonce = userAssociatedNonces[asset][user].popFront();
        // Ensure the request is already unlocked.
        if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();

        bytes32 requestId = getRequestId(asset, usersFirstWithdrawalRequestNonce);
        WithdrawalRequest memory request = withdrawalRequests[requestId];

        delete withdrawalRequests[requestId];

        // Check that the withdrawal delay has passed since the request's initiation.
        if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();
```

```text
    function removeSupportedAsset(
        address asset,
        uint256 tokenIndex
    )
        external
        onlySupportedAsset(asset)
        onlyRole(DEFAULT_ADMIN_ROLE)
    {
        UtilLib.checkNonZeroAddress(asset);

        if (supportedAssetList[tokenIndex] != asset) {
            revert TokenNotFoundError();
        }

        address depositPool = getContract(LRTConstants.LRT_DEPOSIT_POOL);

        if (ILRTDepositPool(depositPool).getTotalAssetDeposits(asset) > maxNegligibleAmount) {
            revert CannotRemoveAssetWithDeposits(asset);
        }

        delete isSupportedAsset[asset];
        delete assetStrategy[asset];
        depositLimitByAsset[asset] = 0;

        supportedAssetList[tokenIndex] = supportedAssetList[supportedAssetList.length - 1];
        supportedAssetList.pop();

        emit RemovedSupportedAsset(asset);
    }
```
