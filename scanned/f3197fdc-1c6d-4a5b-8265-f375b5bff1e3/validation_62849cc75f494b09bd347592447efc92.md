### Title
Hardcoded 1:1 ETH Rate in `WETHOracle` and `OneETHPriceOracle` Ignores Real Market Depeg — (`contracts/pools/oracle/WETHOracle.sol`, `contracts/oracles/OneETHPriceOracle.sol`)

---

### Summary

Two oracle contracts in the LRT-rsETH protocol unconditionally return `1e18` (i.e., 1:1 with ETH) regardless of actual market conditions. This is the direct structural analog of the reported "stablecoin value is assumed to be constant" vulnerability: if the pegged asset deviates from its assumed price, the protocol continues to mint wrsETH or rsETH at the stale constant rate, enabling depositors to extract more value than they contribute and leaving the pool or protocol undercollateralized.

---

### Finding Description

**`WETHOracle`** (used in L2 pools): [1](#0-0) 

`WETHOracle.getRate()` and `WETHOracle.rate()` unconditionally return `1e18`. This contract is registered as the oracle for WETH deposits in L2 pool variants (`RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPool`, `RSETHPoolNoWrapper`) via the `supportedTokenOracle` mapping. [2](#0-1) 

When a user deposits a supported token, the pool calls: [3](#0-2) 

The `tokenToETHRate` is fetched from `IOracle(supportedTokenOracle[token]).getRate()`. If that oracle is `WETHOracle`, the rate is always `1e18`, so the pool always treats 1 WETH as worth exactly 1 ETH.

**`OneETHPriceOracle`** (used in L1 `LRTOracle`): [4](#0-3) 

`OneETHPriceOracle.getAssetPrice()` unconditionally returns `1e18` for any asset address passed to it. It is registered via `LRTOracle.updatePriceOracleFor` as the price oracle for assets that are assumed to be 1:1 with ETH.

`LRTOracle.getAssetPrice()` delegates directly to the registered oracle: [5](#0-4) 

This price feeds into `_getTotalEthInProtocol()`, which computes the total ETH backing all rsETH: [6](#0-5) 

And into `LRTDepositPool.getRsETHAmountToMint()`: [7](#0-6) 

If the asset assigned `OneETHPriceOracle` depegs, the protocol still mints rsETH at the 1:1 rate.

---

### Impact Explanation

**L2 pools (`WETHOracle`):** If WETH depegs from ETH (e.g., due to a bridge exploit or smart contract failure on the canonical WETH contract), a depositor can send 1 WETH (market value: 0.8 ETH) and receive wrsETH priced at 1 ETH. The pool is immediately undercollateralized. Existing wrsETH holders bear the loss when they redeem, receiving less ETH than their wrsETH represents. This is a direct theft of funds from existing pool participants.

**L1 protocol (`OneETHPriceOracle`):** If an LST assigned `OneETHPriceOracle` depegs, users can deposit devalued LSTs and receive rsETH at the old 1:1 rate. The `_getTotalEthInProtocol()` calculation will overstate TVL, inflating `rsETHPrice` and diluting all existing rsETH holders. This is protocol insolvency.

**Impact classification:** Critical — direct theft of user funds / protocol insolvency.

---

### Likelihood Explanation

- WETH depegging is a low-probability but non-zero event (canonical bridge exploits have occurred on multiple L2s historically).
- LST depegging (stETH, ETHx, rETH, swETH) is a known risk class; all major LSTs have experienced temporary depegs.
- No privileged action is required to exploit once the depeg occurs — any unprivileged depositor can call `deposit()` on the pool or `depositAsset()` on `LRTDepositPool` with the depegged asset.
- The entry path is fully permissionless.

**Likelihood classification:** Low-to-Medium (depeg events are rare but have real historical precedent).

---

### Recommendation

1. Replace `WETHOracle` with a Chainlink ETH/USD or WETH/ETH feed (e.g., via `ChainlinkOracleForRSETHPoolCollateral`) that reflects real market prices, even for assets assumed to be pegged.
2. Replace `OneETHPriceOracle` with a live price feed for any asset that can theoretically depeg. If an asset is truly intended to be 1:1 (e.g., native ETH itself), document and enforce this assumption with circuit-breaker logic.
3. Add a staleness/deviation check: if the oracle-reported price deviates more than a configurable threshold from `1e18`, pause deposits for that asset.

---

### Proof of Concept

**L2 pool exploit path (WETHOracle):**

1. WETH depegs to 0.8 ETH on Arbitrum (e.g., bridge exploit).
2. Attacker calls `RSETHPoolV3.deposit(WETH_ADDRESS, 1e18, "")`.
3. Inside `viewSwapRsETHAmountAndFee(1e18, WETH_ADDRESS)`:
   - `tokenToETHRate = IOracle(supportedTokenOracle[WETH]).getRate()` → `WETHOracle` returns `1e18`.
   - `rsETHAmount = 1e18 * 1e18 / rsETHToETHrate` → attacker receives wrsETH worth ~1 ETH.
4. Attacker contributed WETH worth 0.8 ETH; received wrsETH worth 1 ETH.
5. Pool is undercollateralized by 0.2 ETH per WETH deposited; existing holders absorb the loss on redemption. [8](#0-7) [9](#0-8) 

**L1 protocol exploit path (OneETHPriceOracle):**

1. An LST (e.g., stETH) assigned `OneETHPriceOracle` depegs to 0.8 ETH.
2. Attacker calls `LRTDepositPool.depositAsset(stETH, 1e18, ...)`.
3. `getRsETHAmountToMint(stETH, 1e18)` calls `lrtOracle.getAssetPrice(stETH)` → `OneETHPriceOracle` returns `1e18`.
4. Attacker receives rsETH priced at 1 ETH per stETH deposited, despite stETH being worth only 0.8 ETH.
5. Protocol TVL is overstated; rsETH is underbacked; all existing rsETH holders are diluted. [10](#0-9) [7](#0-6)
