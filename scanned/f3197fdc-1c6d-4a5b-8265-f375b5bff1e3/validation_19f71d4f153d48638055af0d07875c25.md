### Title
ETH Deposit Limit Bypass Due to Incomplete Validation in `_checkIfDepositAmountExceedesCurrentLimit` - (File: contracts/LRTDepositPool.sol)

---

### Summary

`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric check: for ERC-20 assets it correctly tests `totalAssetDeposits + amount > depositLimit`, but for native ETH it silently drops the `amount` parameter and only tests `totalAssetDeposits > depositLimit`. Any depositor can therefore push the ETH balance arbitrarily above the configured cap in a single call.

---

### Finding Description

The function at lines 676–682 of `contracts/LRTDepositPool.sol` reads:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

The ETH branch evaluates whether the limit has **already been exceeded** before this deposit, not whether it **would be exceeded** after it. The `amount` argument — which is `msg.value` forwarded from `depositETH` — is never added to `totalAssetDeposits` in the ETH path.

`depositETH` (lines 76–93) is a public, permissionless, `payable` function. It calls `_beforeDeposit`, which calls `_checkIfDepositAmountExceedesCurrentLimit`. Because the check returns `false` whenever the pre-deposit total is at or below the limit, any single transaction can deposit an arbitrarily large ETH amount and receive a proportional rsETH mint, as long as the total was not already over the cap.

The ERC-20 path (`depositAsset`, lines 99–118) uses the correct `totalAssetDeposits + amount` comparison and is not affected.

---

### Impact Explanation

The deposit limit is the protocol's primary on-chain mechanism for capping ETH exposure and controlling rsETH supply growth. Bypassing it allows a depositor to mint rsETH far beyond the intended ceiling in a single transaction. While the deposited ETH does back the newly minted rsETH (no immediate insolvency), the protocol silently violates its own stated invariant: the deposit limit is never enforced for ETH. This constitutes a **Low** impact — the contract fails to deliver the promised deposit-cap guarantee without losing depositor value directly.

---

### Likelihood Explanation

Likelihood is **High**. `depositETH` is permissionless and payable. No special role, timing, or market condition is required. Any depositor who observes that the ETH total is below the cap can immediately exploit the incomplete check by sending a single large transaction.

---

### Recommendation

Add `amount` to the ETH branch, mirroring the ERC-20 logic:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

---

### Proof of Concept

Assume `depositLimitByAsset[ETH_TOKEN] = 100 ether` and `getTotalAssetDeposits(ETH_TOKEN) = 99 ether`.

1. Attacker calls `depositETH{value: 10_000 ether}(0, "")`.
2. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 10_000 ether)`.
3. Inside the function: `totalAssetDeposits = 99 ether`; ETH branch evaluates `99 ether > 100 ether` → `false`.
4. The check passes; `_mintRsETH` mints rsETH for 10,000 ETH.
5. Post-deposit ETH total is 10,099 ETH — 100× the intended cap — with no revert.

The `amount` parameter is never consulted for ETH, so the deposit limit is entirely ineffective for any single deposit that would push the total over the cap. [1](#0-0) [2](#0-1) [3](#0-2)
