### Title
Missing Arbitrum L2 Sequencer Uptime Check Allows Stale Chainlink Price Exploitation in L2 Pool Deposits - (File: contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol)

---

### Summary

`ChainlinkOracleForRSETHPoolCollateral.getRate()` calls Chainlink's `latestRoundData()` on Arbitrum without verifying that the L2 sequencer is online. When the sequencer is down or in its post-restart grace period, Chainlink feeds on Arbitrum return stale prices. An attacker can exploit this window to deposit collateral tokens (wstETH, ETHx) at a stale, inflated price and receive more wrsETH than the deposited collateral is worth, extracting value from the pool.

---

### Finding Description

`ChainlinkOracleForRSETHPoolCollateral.getRate()` fetches the collateral token/ETH rate directly from a Chainlink aggregator:

```solidity
function getRate() public view returns (uint256) {
    (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();

    if (answeredInRound < roundID) revert StalePrice();
    if (timestamp == 0) revert IncompleteRound();
    if (ethPrice <= 0) revert InvalidPrice();
    ...
}
``` [1](#0-0) 

The existing checks (`answeredInRound < roundID`, `timestamp == 0`) do **not** protect against the Arbitrum sequencer downtime scenario. When the Arbitrum sequencer is offline, Chainlink's L2 price feeds stop updating but continue to return the last known answer without reverting. The `answeredInRound >= roundID` condition remains satisfied because no new round has started — the feed is simply frozen. The `timestamp` field is non-zero (it reflects the last completed round). All three existing guards pass, yet the returned price is stale.

This oracle is deployed on Arbitrum as the `wstETH Oracle` (`0x6932e85964fA31F70ed9DEbe63D9D969ad00F112`) and `ETHx Oracle` (`0x0c742a74fA8E1a00D6bC4623d485A84dD147864d`), and is consumed by the `RSETHPool` (`0x376A7564AF88242D6B8598A5cfdD2E9759711B61`) on Arbitrum. [2](#0-1) 

The pool's token deposit path calls this oracle directly:

```solidity
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [3](#0-2) 

No sequencer uptime feed (e.g., Chainlink's `0xFdB631F5EE196F0ed6FAa767959853A9F217697D` on Arbitrum) is consulted anywhere in the contract suite.

---

### Impact Explanation

**Impact: High — Theft of unclaimed yield / protocol value extraction**

Attack scenario: The Arbitrum sequencer goes down while wstETH/ETH is trading at 1.10 ETH. During the outage, wstETH drops to 1.05 ETH. The sequencer restarts; Chainlink has not yet published a new round. The stale price of 1.10 ETH is still returned and passes all existing guards.

An attacker deposits wstETH at the stale 1.10 rate and receives:
```
rsETHAmount = deposit * 1.10e18 / rsETHToETHrate
```
instead of the correct:
```
rsETHAmount = deposit * 1.05e18 / rsETHToETHrate
```

The attacker receives ~4.8% more wrsETH than the deposited collateral is worth. The excess wrsETH represents a direct extraction of value from the pool's collateral backing, diluting all existing wrsETH holders and the protocol's solvency on that chain.

---

### Likelihood Explanation

**Likelihood: Low-Medium**

Arbitrum sequencer downtime is rare but documented (multiple incidents on record). The attack requires:
1. Sequencer downtime during a period of price movement (common during volatility).
2. The attacker to act within the post-restart grace period before Chainlink publishes a new round (typically minutes).

The attack is permissionless, requires no special role, and is executable by any depositor via the public `deposit(address token, uint256 amount, string referralId)` function. [4](#0-3) 

---

### Recommendation

Add a sequencer uptime check at the top of `ChainlinkOracleForRSETHPoolCollateral.getRate()` using Chainlink's L2 Sequencer Uptime Feed. Revert if the sequencer is down or if the grace period (e.g., 3600 seconds) has not elapsed since restart:

```solidity
// Arbitrum sequencer uptime feed: 0xFdB631F5EE196F0ed6FAa767959853A9F217697D
(, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
if (answer != 0) revert SequencerDown();
if (block.timestamp - startedAt < GRACE_PERIOD) revert GracePeriodNotOver();
```

This pattern is the standard Chainlink recommendation for all L2 deployments.

### Proof of Concept

1. Arbitrum sequencer goes offline. wstETH/ETH moves from 1.10 → 1.05 during the outage.
2. Sequencer restarts. Chainlink wstETH/ETH feed on Arbitrum still returns `answer = 1.10e18` (last round before outage). `answeredInRound == roundID` (no new round started), `timestamp != 0`. All guards in `getRate()` pass.
3. Attacker calls `RSETHPool.deposit(wstETH, 100e18, "")` on Arbitrum.
4. Pool calls `ChainlinkOracleForRSETHPoolCollateral.getRate()` → returns `1.10e18`.
5. `rsETHAmount = 100e18 * 1.10e18 / rsETHToETHrate` — attacker receives ~4.8% excess wrsETH vs. fair value.
6. Attacker bridges wrsETH to L1 and redeems for rsETH, extracting the surplus from the pool's collateral. [1](#0-0) [5](#0-4)
