### Title
`dailyMintLimit` Uninitialized in `initialize` Freezes All Deposits Until Separate `reinitialize` Is Called - (File: contracts/pools/RSETHPoolV2.sol, contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolV3WithNativeChainBridge.sol)

### Summary
Across all L2 pool variants (`RSETHPoolV2`, `RSETHPoolV3`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPoolV2ExternalBridge`, `RSETHPoolV3ExternalBridge`), the `dailyMintLimit` state variable is never assigned in the `initialize` function. It defaults to `0`. The `limitDailyMint` modifier applied to every `deposit` entry point will revert every deposit with `DailyMintLimitExceeded` until a privileged `reinitialize` call is made separately.

### Finding Description
The `initialize` function in `RSETHPoolV3` sets `wrsETH`, `feeBps`, `rsETHOracle`, and `isEthDepositEnabled`, but leaves `dailyMintLimit` and `startTimestamp` at their default value of `0`. [1](#0-0) 

The `limitDailyMint` modifier is applied to both `deposit` overloads: [2](#0-1) 

With `startTimestamp == 0`, the guard `block.timestamp < startTimestamp` evaluates to `block.timestamp < 0`, which is always `false` — so execution falls through to the limit check. With `dailyMintLimit == 0`, the check `dailyMintAmount + rsETHAmount > 0` is `true` for any non-zero deposit, reverting with `DailyMintLimitExceeded`.

The only remediation path is `reinitialize`, which carries a `reinitializer(2)` modifier (callable exactly once) and requires `DEFAULT_ADMIN_ROLE`: [3](#0-2) 

The same pattern is present in `RSETHPoolV2` (`reinitializer(3)`): [4](#0-3) 

And in `RSETHPoolV3WithNativeChainBridge`: [5](#0-4) 

Because `reinitializer(N)` can only be called once, if the admin omits this step during deployment or upgrade, the slot is consumed and `dailyMintLimit` can never be set through that path again. The only remaining escape hatch is `setDailyMintLimit`, which also requires `DEFAULT_ADMIN_ROLE`. [6](#0-5) 

### Impact Explanation
Every public `deposit` call on the affected L2 pools is gated by `limitDailyMint`. With `dailyMintLimit == 0`, no user can deposit ETH or any supported token and receive wrsETH. This constitutes a **temporary freezing of funds** (deposits blocked; no user assets already in the pool are lost, but the protocol fails to deliver its promised service). If `reinitialize` was already consumed by a prior upgrade and `setDailyMintLimit` is also not called, the freeze becomes permanent for that initializer slot. Impact: **Medium — Temporary (potentially permanent) freezing of funds**.

### Likelihood Explanation
The `reinitialize` call is a separate, manually executed transaction that is not enforced by the `initialize` function itself. Any deployment or upgrade that omits this step leaves the contract in the broken state. Given that multiple pool variants share this pattern and the `reinitializer` slot is one-time-use, the probability of at least one deployment window where deposits are blocked is non-trivial. Likelihood: **Low**, but the window is real and the consequence is immediate user-facing breakage.

### Recommendation
Set `dailyMintLimit` (and `startTimestamp`) directly inside the `initialize` function, accepting them as constructor parameters, so the contract is never in a state where `dailyMintLimit == 0` after initialization. Alternatively, add a guard in `limitDailyMint` that skips the limit check when `dailyMintLimit == 0` (treating zero as "no limit"), consistent with how the original Gearbox report recommended setting `expectedLiquidityLimit` in the constructor.

### Proof of Concept
1. Deploy `RSETHPoolV3` proxy and call `initialize(admin, bridger, wrsETH, feeBps, oracle, true)`. Do **not** call `reinitialize`.
2. `dailyMintLimit` is `0`, `startTimestamp` is `0`.
3. Any user calls `deposit{value: 1 ether}("ref")`.
4. `limitDailyMint` executes: `block.timestamp < 0` → false; `rsETHAmount > 0` → `DailyMintLimitExceeded` revert.
5. All deposits are blocked. The `reinitializer(2)` slot is still available, but if a prior upgrade already consumed it, no recovery path exists through `reinitialize`.

```text
    modifier limitDailyMint(uint256 amount, address token) {
        if (block.timestamp < startTimestamp) {
            revert MintBeforeStartTimestamp();
        }

        uint256 rsETHAmount;

        // Calculate the amount of rsETH that will be minted
        if (token == ETH_IDENTIFIER) {
            (rsETHAmount,) = viewSwapRsETHAmountAndFee(amount);
        } else {
            (rsETHAmount,) = viewSwapRsETHAmountAndFee(amount, token);
        }

        uint256 currentDay = getCurrentDay();

        // If the current day is greater than the last mint day, reset the daily mint amount
        if (currentDay > lastMintDay) {
            lastMintDay = currentDay;
            dailyMintAmount = 0;
        }

        // Check if the daily mint amount plus the amount to mint is greater than the daily mint limit
        if (dailyMintAmount + rsETHAmount > dailyMintLimit) {
            revert DailyMintLimitExceeded();
        }

        dailyMintAmount += rsETHAmount;
        _;
    }
```

**File:** contracts/pools/RSETHPoolV3.sol (L179-198)
```text
    function reinitialize(
        uint256 _dailyMintLimit,
        uint256 _startTimestamp
    )
        external
        reinitializer(2)
        onlyRole(DEFAULT_ADMIN_ROLE)
    {
        if (_dailyMintLimit == 0) {
            revert InvalidDailyMintLimit();
        }

        // startTimestamp cannot be in the past
        if (block.timestamp > _startTimestamp) {
            revert InvalidStartTimestamp();
        }

        dailyMintLimit = _dailyMintLimit;
        startTimestamp = _startTimestamp;
    }
```
