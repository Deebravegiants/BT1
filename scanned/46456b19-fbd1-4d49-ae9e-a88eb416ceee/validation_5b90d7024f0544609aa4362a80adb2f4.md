### Title
Token Decimal Assumption in `viewSwapRsETHAmountAndFee` Causes Near-Total Fund Loss for Non-18-Decimal Token Depositors - (File: contracts/pools/RSETHPoolV3ExternalBridge.sol, RSETHPoolV3.sol, RSETHPool.sol, RSETHPoolNoWrapper.sol)

### Summary
Every L2 pool contract computes the rsETH output for ERC-20 token deposits using a formula that silently assumes the deposited token has 18 decimals. If a token with fewer decimals (e.g., USDC with 6) is added as a supported token, depositors receive approximately `10^(18 − tokenDecimals)` times fewer rsETH than they are owed, effectively destroying the deposited value.

### Finding Description
The shared `viewSwapRsETHAmountAndFee(uint256 amount, address token)` function present in all four L2 pool contracts computes:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

`tokenToETHRate` is returned by the token's oracle in 1e18 precision (i.e., "how many ETH per 1 whole token, scaled to 1e18"). `rsETHToETHrate` is also in 1e18 precision. `amountAfterFee` is in the token's **native decimal units**.

For an 18-decimal token (ETH, wstETH) the units cancel correctly. For a 6-decimal token like USDC:

| Variable | Value (example: 1 000 USDC, ETH = $3 000) |
|---|---|
| `amountAfterFee` | `1 000 × 10^6 = 1e9` |
| `tokenToETHRate` | `≈ 3.33e14` (1 USDC = 1/3000 ETH in 1e18 precision) |
| `rsETHToETHrate` | `≈ 1.05e18` |
| **Computed** `rsETHAmount` | `1e9 × 3.33e14 / 1.05e18 ≈ 3.17e5` |
| **Correct** `rsETHAmount` | `≈ 3.17e17` (0.317 rsETH in 1e18 precision) |

The user receives `3.17e5` rsETH raw units (≈ `3.17e-13` rsETH, worth ~$0) instead of `3.17e17` raw units (≈ 0.317 rsETH, worth ~$950). The deposited USDC is transferred into the pool and bridged to L1, while the user's funds are permanently lost.

### Impact Explanation
A user depositing a non-18-decimal token receives `10^(18 − decimals)` times fewer rsETH than owed. For USDC (6 decimals) this is a factor of `1e12`. The deposited tokens are transferred to the pool and bridged to L1; the user cannot recover them. This constitutes **direct, permanent theft of user funds** — Critical severity.

### Likelihood Explanation
The pools expose a generic `addSupportedToken` / `addSupportedToken(token, oracle, bridge)` interface explicitly designed to expand beyond ETH and wstETH. A protocol operator adding USDC, USDT, or any other sub-18-decimal token — a natural expansion — would silently activate the bug for every subsequent depositor. No attacker action is required beyond a normal deposit after the token is listed.

### Recommendation
Normalize `amountAfterFee` to 18 decimals before applying the rate:

```solidity
uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 normalizedAmount = amountAfterFee * 10 ** (18 - tokenDecimals);
rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
```

Apply the same fix to `feeEarnedInToken` accounting and to `getTokenBalanceMinusFees`. Alternatively, enforce in `addSupportedToken` that only 18-decimal tokens may be listed.

### Proof of Concept
1. Admin calls `addSupportedToken(USDC, usdcOracle, usdcBridge)` on `RSETHPoolV3ExternalBridge`.
2. `usdcOracle.getRate()` returns `3.33e14` (1 USDC ≈ 1/3000 ETH in 1e18 precision).
3. User calls `deposit(USDC, 1_000e6, "ref")` (1 000 USDC).
4. `viewSwapRsETHAmountAndFee(1_000e6, USDC)` executes:
   - `fee = 1_000e6 * feeBps / 10_000` (small)
   - `amountAfterFee ≈ 1_000e6`
   - `rsETHAmount = 1_000e6 * 3.33e14 / 1.05e18 ≈ 317_000`
5. `wrsETH.mint(user, 317_000)` — user receives `317_000 / 1e18 ≈ 3.17e-13` rsETH.
6. 1 000 USDC is held in the pool and later bridged to L1 via `bridgeTokens`.
7. User's $1 000 is permanently lost; they hold dust rsETH worth ~$0. [6](#0-5) [7](#0-6)

    function viewSwapRsETHAmountAndFee(
        uint256 amount,
        address token
    )
        public
        view
        onlySupportedToken(token)
        returns (uint256 rsETHAmount, uint256 fee)
    {
        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // rate of token in ETH
        uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
    }
