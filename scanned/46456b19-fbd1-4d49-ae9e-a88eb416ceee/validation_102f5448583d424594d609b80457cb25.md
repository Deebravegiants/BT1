### Title
ETH Temporarily Frozen in L1VaultV2 via Deposit-Limit Griefing Through Permissionless `receive()` — (`contracts/L1VaultV2.sol`)

---

### Summary

`depositETHForL1VaultETH()` always deposits `address(this).balance` in full. Because `L1VaultV2.receive()` is permissionless, any attacker can inflate the vault's ETH balance by a tiny amount, pushing the total above the remaining deposit limit in `LRTDepositPool`. The subsequent call to `depositETH` then reverts with `MaximumDepositLimitReached`, and there is no partial-deposit path or ETH-rescue function in the vault, leaving all vault ETH frozen until an admin raises the limit.

---

### Finding Description

**Step 1 — Permissionless ETH entry point**

`L1VaultV2` accepts arbitrary ETH via its `receive()` fallback: [1](#0-0) 

Any address can call `address(vault).transfer(1 wei)` with no access control.

**Step 2 — Full-balance deposit with no partial option**

`depositETHForL1VaultETH()` unconditionally uses `address(this).balance`: [2](#0-1) 

There is no `amount` parameter; the manager cannot choose to deposit less than the full balance.

**Step 3 — ETH limit check includes the incoming ETH**

Inside `LRTDepositPool.depositETH`, `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit`. For ETH, `totalAssetDeposits` is computed via `getETHDistributionData()`, which reads `address(this).balance` of the deposit pool **after** the ETH has already arrived (the function is `payable`): [3](#0-2) [4](#0-3) 

So the check is effectively: `(priorSystemETH + vaultBalance) > depositLimit`. If the attacker's dust tip pushes `vaultBalance` over the remaining headroom, this returns `true`.

**Step 4 — Revert propagates, ETH is stuck**

`_beforeDeposit` reverts with `MaximumDepositLimitReached`: [5](#0-4) 

Because the vault has no partial-deposit path and no ETH withdrawal function, all ETH (including legitimate L2-bridged ETH) is frozen until an admin increases `depositLimitByAsset(ETH)`.

---

### Impact Explanation

All ETH sitting in `L1VaultV2` — including ETH legitimately bridged from L2 — cannot be deposited into `LRTDepositPool` and therefore cannot be converted to rsETH and bridged back. This is a **temporary freeze of funds** (Medium) because it persists until an admin governance action raises the deposit cap.

---

### Likelihood Explanation

- The attacker needs only to send `1 wei` to the vault when the remaining ETH deposit headroom is less than the vault's current balance.
- No role, approval, or special access is required.
- The condition (deposit limit nearly exhausted) is a normal operational state for a live protocol.
- Cost to the attacker is negligible (1 wei + gas).

---

### Recommendation

1. **Add an `amount` parameter** to `depositETHForL1VaultETH()` so the manager can deposit exactly the amount that fits within the remaining limit, rather than always using `address(this).balance`.
2. **Add an ETH rescue / sweep function** (admin-only) to allow recovery of ETH that cannot be deposited due to limit constraints.
3. Alternatively, inside `depositETHForL1VaultETH()`, cap `balanceOfETH` to `lrtDepositPool.getAssetCurrentLimit(ETH_IDENTIFIER)` before calling `depositETH`.

---

### Proof of Concept

```solidity
// Preconditions:
//   depositLimit = 100 ETH
//   currentSystemETH = 99 ETH  (1 ETH headroom)
//   vault.balance = 1 ETH      (legitimate L2 bridge deposit)

// Attack:
attacker.sendValue(vault, 1 wei);   // vault.balance = 1 ETH + 1 wei

// Manager action:
vault.depositETHForL1VaultETH();
// → balanceOfETH = 1 ETH + 1 wei
// → depositPool.depositETH{value: 1 ETH + 1 wei}(...)
// → totalAssetDeposits = 99 ETH + 1 ETH + 1 wei = 100 ETH + 1 wei
// → _checkIfDepositAmountExceedesCurrentLimit: 100 ETH + 1 wei > 100 ETH → true
// → revert MaximumDepositLimitReached
// All 1 ETH + 1 wei is now frozen in the vault.

// Assert: depositETHForL1VaultETH() reverts; vault.balance unchanged.
```
