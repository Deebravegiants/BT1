### Title
Block Stuffing Can Delay `bridgeAssetsViaNativeBridge` Across Multiple Cycles, Temporarily Freezing Multi-Cycle ETH Deposits — (`contracts/pools/RSETHPoolNoWrapper.sol`)

---

### Summary

An attacker can use block stuffing on Unichain to prevent the `BRIDGER_ROLE` operator from executing `bridgeAssetsViaNativeBridge()` during its scheduled window. Because the contract imposes no on-chain deadline or maximum accumulation cap, deposits from multiple cycles pile up in the pool. The frozen ETH grows linearly with the number of stuffed cycles and is only released once the attacker stops stuffing and the operator successfully submits the transaction.

---

### Finding Description

`RSETHPoolNoWrapper.bridgeAssetsViaNativeBridge()` is the sole on-chain path for moving accumulated user ETH from Unichain to L1 via the native bridge. It is callable only by the `BRIDGER_ROLE` operator and bridges the entire `ethBalanceMinusFees` in a single call: [1](#0-0) 

The call chain is:

```
RSETHPoolNoWrapper.bridgeAssetsViaNativeBridge()
  → IL2Messenger(messenger).sendETHToL1ViaBridge(...)   // UnichainMessenger
    → IUnichainMessenger(l2Bridge).bridgeETHTo(...)      // Unichain native bridge
``` [2](#0-1) [3](#0-2) 

The contract contains **no**:
- On-chain deadline by which bridging must occur
- Maximum ETH accumulation cap per cycle
- Fallback or permissionless bridging path
- Enforcement of bridging frequency

Unichain is an OP-stack L2. Its block gas limit is substantially lower than Ethereum mainnet, making block stuffing significantly cheaper. An attacker can fill consecutive blocks with high-gas-price dummy transactions, consuming the entire block gas budget and preventing the operator's `bridgeAssetsViaNativeBridge` transaction from being included during the scheduled window.

If the operator operates on a fixed schedule (e.g., daily), each stuffed cycle causes one additional cycle of user deposits to remain in the pool. After K stuffed cycles, K × (average daily deposit volume) of ETH is temporarily frozen.

---

### Impact Explanation

**Low — Block stuffing / Temporary freezing of funds.**

User ETH deposited into `RSETHPoolNoWrapper` is not permanently lost: once the attacker stops stuffing, the operator can call `bridgeAssetsViaNativeBridge()` and the full accumulated balance is bridged. However, during the stuffing period, all deposited ETH is frozen on Unichain and unavailable on L1, violating the operational invariant that at most one cycle's worth of deposits is ever pending.

---

### Likelihood Explanation

- Unichain (OP-stack) has a low block gas limit, making block stuffing materially cheaper than on Ethereum mainnet.
- The attacker only needs to sustain stuffing for the duration of the operator's scheduled call window (e.g., a few minutes around the daily trigger).
- No special permissions or insider access are required — any funded account can submit high-gas-price filler transactions.
- The cost scales with the block gas limit and the duration of the window, not with the amount of ETH being frozen, making this economically viable for a motivated attacker targeting a high-TVL pool.

---

### Recommendation

1. **Remove fixed-schedule dependency**: Operate the bridger with a retry loop so that if a transaction is not included within N blocks, it is resubmitted with a higher gas price (priority fee), outbidding the stuffing transactions.
2. **Add a maximum accumulation guard**: Emit an on-chain alert (or revert deposits) if `ethBalanceMinusFees` exceeds a configurable threshold, signalling that bridging is overdue.
3. **Consider a permissionless bridging path**: Allow any caller to trigger bridging after a configurable delay has elapsed since the last bridge call, removing the single-operator dependency.
4. **Monitor mempool inclusion latency**: Off-chain monitoring that detects when a submitted `bridgeAssetsViaNativeBridge` transaction is not mined within an expected window and escalates gas price automatically.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

// Invariant test: simulates K deposit cycles with block stuffing between each.
// Asserts that frozen ETH grows linearly with K and that a single bridge call
// after stuffing ends correctly bridges the full accumulated balance.

contract BlockStuffingPoC {
    RSETHPoolNoWrapper pool;
    address bridger;

    uint256 constant DAILY_DEPOSIT = 10 ether;
    uint256 constant K = 3; // number of stuffed cycles

    function testBlockStuffingAccumulation() external {
        // Simulate K cycles of user deposits with no bridging (stuffing prevents it)
        for (uint256 i = 0; i < K; i++) {
            // Users deposit ETH into the pool each cycle
            vm.deal(address(this), DAILY_DEPOSIT);
            pool.swapETHForRsETH{value: DAILY_DEPOSIT}(...);

            // Attacker stuffs blocks: operator's bridgeAssetsViaNativeBridge tx
            // cannot be included. Simulate by simply NOT calling bridge here.
            vm.roll(block.number + BLOCKS_PER_DAY);
        }

        // After K cycles, pool holds K * DAILY_DEPOSIT (minus fees) unfrozen
        uint256 frozen = pool.getETHBalanceMinusFees();
        assertApproxEqAbs(frozen, K * DAILY_DEPOSIT, FEE_TOLERANCE);
        // frozen grows linearly: invariant "max 1 cycle frozen" is violated

        // Once stuffing ends, operator bridges the full accumulated balance
        vm.prank(bridger);
        pool.bridgeAssetsViaNativeBridge();

        // All K cycles bridged in one call — confirms funds were frozen, not lost
        assertEq(pool.getETHBalanceMinusFees(), 0);
    }
}
```

The test confirms:
- `frozen` grows as `K × DAILY_DEPOSIT`, violating the one-cycle bound.
- A single post-stuffing call to `bridgeAssetsViaNativeBridge()` successfully bridges the entire accumulated balance, confirming the freeze is temporary but real.
