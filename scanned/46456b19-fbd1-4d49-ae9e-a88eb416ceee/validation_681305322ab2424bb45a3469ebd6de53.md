### Title
Cross-Token Value Arbitrage via 1:1 Deposit/Withdraw with Multiple Allowed Tokens — (`contracts/L2/RsETHTokenWrapper.sol`)

---

### Summary

`RsETHTokenWrapper` allows any `allowedToken` to be deposited for wrsETH and any other `allowedToken` to be withdrawn at a strict 1:1 ratio, with no price oracle or rate check. When two tokens of unequal market value are simultaneously in `allowedTokens`, an attacker can deposit the cheaper token and withdraw the more valuable one, draining the contract of its high-value collateral.

---

### Finding Description

`_deposit` and `_withdraw` both operate at a flat 1:1 ratio: [1](#0-0) [2](#0-1) 

There is no exchange rate, no oracle, and no per-asset accounting. The contract assumes all `allowedTokens` are perpetually 1:1 equivalent. The `reinitialize` function explicitly adds a **second** allowed token without removing the first: [3](#0-2) 

This creates a window — potentially permanent if `removeAllowedToken` is not called atomically — where two tokens with divergent values coexist in `allowedTokens`. The `allowedTokens` mapping itself has no value-parity enforcement: [4](#0-3) 

---

### Impact Explanation

**Critical — Protocol insolvency / Direct theft of funds.**

Attack path:
1. `tokenA` (old alt-rsETH, depegged to e.g. 0.5 ETH per unit) and `tokenB` (canonical rsETH, 1 ETH per unit) are both in `allowedTokens`.
2. Attacker calls `deposit(tokenA, N)` → pays N × 0.5 ETH worth, receives N wrsETH.
3. Attacker calls `withdraw(tokenB, N)` → burns N wrsETH, receives N × 1 ETH worth of tokenB.
4. Net gain: N × 0.5 ETH. The contract now holds N tokenA (worth 0.5N ETH) backing wrsETH claims that were previously backed by tokenB (worth N ETH). All remaining tokenB holders face insolvency.

The attack is atomic, requires no privileged role, and is repeatable until all tokenB is drained.

---

### Likelihood Explanation

**High.** The `reinitialize` function is the documented upgrade path and explicitly adds a second token. Any gap between `reinitialize` and a subsequent `removeAllowedToken` (which requires a separate TIMELOCK_ROLE transaction) is exploitable. Additionally, any alt-rsETH token can depeg due to bridge exploits, liquidity crises, or smart contract bugs — all historically observed events in the L2 ecosystem. [5](#0-4) [6](#0-5) 

---

### Recommendation

1. **Enforce single-token invariant**: Allow at most one `allowedToken` at a time, or atomically swap (remove old, add new) in `reinitialize`.
2. **Per-asset accounting**: Track how much of each token was deposited and only allow withdrawal of the same token that was deposited (i.e., per-asset share accounting rather than a fungible wrsETH pool).
3. **Rate oracle**: Integrate a price feed and reject deposits/withdrawals when the token's value deviates beyond a tight band from canonical rsETH.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity 0.8.27;

// Deploy two ERC20 mocks: tokenA (depegged, worth 0.5 ETH) and tokenB (canonical rsETH, worth 1 ETH)
// Both added to allowedTokens (simulating post-reinitialize state before removeAllowedToken)

function testCrossTokenArbitrage() public {
    // Setup: both tokens allowed
    wrapper.addAllowedToken(address(tokenA)); // already added via reinitialize
    // tokenB was the original allowed token

    uint256 N = 100e18;

    // Attacker holds N tokenA (cheap)
    tokenA.mint(attacker, N);
    vm.startPrank(attacker);
    tokenA.approve(address(wrapper), N);

    // Step 1: deposit cheap tokenA, receive N wrsETH
    wrapper.deposit(address(tokenA), N);
    assertEq(wrapper.balanceOf(attacker), N);

    // Step 2: withdraw valuable tokenB using wrsETH
    wrapper.withdraw(address(tokenB), N);
    assertEq(tokenB.balanceOf(attacker), N);

    // Invariant broken: contract holds N tokenA (worth 0.5N ETH)
    // but outstanding wrsETH claims (from other users) expect tokenB (worth N ETH)
    // sum(collateral ETH value) < totalSupply() ETH value => insolvency
    vm.stopPrank();
}
```
