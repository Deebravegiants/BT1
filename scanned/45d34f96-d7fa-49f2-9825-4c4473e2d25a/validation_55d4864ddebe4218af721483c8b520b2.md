### Title
Decimal Precision Mismatch in `viewSwapRsETHAmountAndFee` for Non-18-Decimal Tokens Causes Near-Total Loss of User Deposits - (`contracts/pools/RSETHPool.sol`)

---

### Summary

`viewSwapRsETHAmountAndFee(uint256, address)` computes `rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate` without normalizing for the deposited token's decimal precision. For any supported token with fewer than 18 decimals (e.g., USDC with 6 decimals), the result is `1e12` times smaller than the economically correct value, causing users to receive essentially zero rsETH while the protocol retains the full token deposit.

---

### Finding Description

The ETH deposit path correctly scales the result: [1](#0-0) 

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

The `* 1e18` compensates for the fact that `amountAfterFee` is in wei (18 decimals) and `rsETHToETHrate` is an 18-decimal-precision rate, yielding an 18-decimal rsETH amount.

The token deposit path omits this normalization: [2](#0-1) 

```solidity
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
// Calculate the final rsETH amount
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

This formula is dimensionally correct **only** when `amountAfterFee` is already in 18-decimal units (i.e., the token has 18 decimals). For a 6-decimal token:

| Variable | Value |
|---|---|
| `amount` (1 USDC) | `1e6` |
| `tokenToETHRate` (oracle, 18-dec precision) | `~3e14` (≈ 0.0003 ETH/USDC) |
| `rsETHToETHrate` (oracle, 18-dec precision) | `~1.05e18` |
| **Computed** `rsETHAmount` | `1e6 * 3e14 / 1.05e18 ≈ 285` (wei of rsETH) |
| **Correct** `rsETHAmount` | `~2.83e14` (≈ 0.000283 rsETH) |

The user receives `~285` wei of rsETH instead of `~2.83e14` wei — a factor of `1e12` shortfall. The fee is computed correctly in token units: [3](#0-2) 

```solidity
uint256 feeBpsForToken = tokenFeeBps[token];
fee = amount * feeBpsForToken / 10_000;
uint256 amountAfterFee = amount - fee;
```

So the protocol collects the correct fee in token units and retains the full token deposit, while the user receives a negligible rsETH amount. The same bug exists identically in `RSETHPoolNoWrapper.sol` (line 311) and `RSETHPoolV3.sol` (line 334). [4](#0-3) 

---

### Impact Explanation

A user depositing 1,000 USDC (1e9 units) would receive `~285,000` wei of rsETH (≈ `2.85e-13` rsETH) instead of `~2.83e17` wei (≈ 0.283 rsETH). The protocol retains the full USDC deposit minus the fee. The fee accumulated in `feeEarnedInToken[token]` is proportionally correct in token terms but represents a near-total extraction of value from the user, constituting theft of unclaimed yield (the fee yield is grossly disproportionate to the rsETH value delivered).

**Scoped impact: High — Theft of unclaimed yield.**

---

### Likelihood Explanation

The precondition is that a token with non-18 decimals is added via `addSupportedToken`, which requires `TIMELOCK_ROLE`. This is a normal, intended operational action — the contract is explicitly designed to support multiple tokens. There is no decimal check or guard in `addSupportedToken`: [5](#0-4) 

The only oracle validation is `getRate() != 0`. USDC, USDT, and similar 6-decimal tokens are common candidates for protocol expansion. Once added, any user calling `deposit(token, amount, referralId)` is affected — no attacker action is required beyond making a normal deposit.

---

### Recommendation

Normalize `amountAfterFee` to 18 decimals before applying the rate formula:

```solidity
uint256 tokenDecimals = IERC20Metadata(token).decimals();
uint256 normalizedAmount = amountAfterFee * (10 ** (18 - tokenDecimals));
rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
```

Apply the same fix to `RSETHPoolNoWrapper.sol` and all `RSETHPoolV3*.sol` variants.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Setup: fork Arbitrum, deploy mock 6-decimal token + oracle returning 3e14 (USDC/ETH rate)
// rsETHOracle returns 1.05e18

function testDecimalMismatch() public {
    // Add USDC (6 decimals) as supported token
    pool.addSupportedToken(address(usdc), address(usdcOracle), address(bridge));

    uint256 depositAmount = 1e6; // 1 USDC
    usdc.mint(attacker, depositAmount);
    usdc.approve(address(pool), depositAmount);

    (uint256 rsETHAmount, uint256 fee) = pool.viewSwapRsETHAmountAndFee(depositAmount, address(usdc));

    // fee = 1e6 * feeBps / 10_000 (correct in USDC units)
    // rsETHAmount ≈ 285 wei (should be ~2.83e14 wei)
    assertLt(rsETHAmount, 1000);                  // receives dust
    assertGt(fee, 0);                             // fee is non-zero
    // ratio of fee value to rsETH value is ~1e12x off
    // user loses ~$1 of value, receives ~$0.000000000001 of rsETH
}
```
