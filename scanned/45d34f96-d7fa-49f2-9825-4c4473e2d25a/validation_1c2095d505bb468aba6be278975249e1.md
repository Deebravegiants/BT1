### Title
Missing Chainlink Price Feed Staleness Check Allows Minting rsETH at Stale Prices — (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards the `updatedAt` timestamp, accepting arbitrarily stale prices with no time-based validation. Any depositor can exploit a stale feed to mint rsETH at an inflated rate, diluting existing holders.

---

### Finding Description

In `ChainlinkPriceOracle.getAssetPrice()`, the Chainlink `latestRoundData()` call destructures only the `price` field:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
``` [1](#0-0) 

The `updatedAt` (4th return value) is completely ignored. There is no check of the form:

```solidity
require(block.timestamp - updatedAt <= maxStaleness, "Stale price");
```

This price is then used directly by `LRTDepositPool` to compute how much rsETH to mint for a deposited LST:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [2](#0-1) 

The `LRTOracle` routes asset pricing through `ChainlinkPriceOracle` for supported LSTs (stETH, rETH, cbETH, etc.). If any of those feeds goes stale — returning the last known price indefinitely — the minting calculation uses that stale value.

The analog to the external report is direct: just as a hardcoded Uniswap V3 cardinality of 60 makes the TWAP window too short and chain-agnostic (enabling manipulation), the absence of any staleness bound here makes the oracle window unbounded and chain-agnostic (enabling exploitation of stale data). Both are hardcoded/missing oracle parameters that should be chain-aware and time-bounded.

---

### Impact Explanation

**High — Theft of unclaimed yield / share mis-accounting.**

If an LST's Chainlink feed becomes stale while the real market price has fallen (e.g., a depeg event, oracle downtime, or L2 sequencer outage), an attacker can:

1. Acquire the now-cheaper LST on the open market.
2. Call `LRTDepositPool.depositAsset()` with that LST.
3. Receive rsETH calculated at the stale (higher) price.
4. Immediately redeem or sell rsETH at the correct current rate.

The attacker extracts value from the protocol's backing pool, diluting all existing rsETH holders. The rsETH/ETH exchange rate tracked by `LRTOracle.rsETHPrice()` will subsequently drop, representing a permanent loss to holders.

---

### Likelihood Explanation

**Medium.** Chainlink feeds have documented heartbeat intervals (e.g., 1 hour for ETH/USD on mainnet, 24 hours for some LST feeds). During network congestion, oracle node downtime, or L2 sequencer outages, feeds can lag well beyond their heartbeat. The protocol is deployed on multiple chains (Ethereum, Arbitrum, Base, Optimism, Linea, Scroll), each with different sequencer and oracle reliability characteristics. No privileged access is required — any depositor can trigger this path.

---

### Recommendation

Add a per-asset configurable `maxStaleness` mapping and enforce it in `getAssetPrice()`:

```solidity
mapping(address asset => uint256 maxStaleness) public assetMaxStaleness;

function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,, uint256 updatedAt,) = priceFeed.latestRoundData();

    require(block.timestamp - updatedAt <= assetMaxStaleness[asset], "Stale price");
    require(price > 0, "Invalid price");

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`maxStaleness` should be set per asset to match the Chainlink feed's documented heartbeat, and should be configurable per chain since heartbeat intervals differ across networks — directly mirroring the fix recommended in the external report (chain-specific virtual function instead of a single hardcoded value).

---

### Proof of Concept

1. Assume `stETH` Chainlink feed on Ethereum has a 24-hour heartbeat. The feed last updated 23 hours ago at price `1.05 ETH`. Since then, stETH has depegged to `0.95 ETH` on the market.
2. Attacker buys `1000 stETH` at `0.95 ETH` each (cost: `950 ETH`).
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, referralId)`.
4. `getAssetPrice(stETH)` returns the stale `1.05e18` (no staleness revert).
5. rsETH minted = `1000 * 1.05e18 / rsETHPrice`. If `rsETHPrice ≈ 1.05e18`, attacker receives `≈1000 rsETH`.
6. True backing value added is only `950 ETH`, but `1000 rsETH` worth `≈1050 ETH` was minted.
7. Attacker redeems or sells rsETH, extracting `≈100 ETH` from existing holders. [3](#0-2) [4](#0-3)
