### Title
Stale `rsETHPrice` in `LRTOracle` Used in Deposit Mint Calculation Allows Excess rsETH Minting - (File: `contracts/LRTDepositPool.sol`, `contracts/LRTOracle.sol`)

---

### Summary

`LRTOracle.rsETHPrice` is a cached state variable updated only via a separate, independent `updateRSETHPrice()` transaction. `LRTDepositPool.getRsETHAmountToMint()` reads this cached value directly without refreshing it. When staking rewards accrue and the actual rsETH price rises before `updateRSETHPrice()` is called, the cached price is lower than the true price, causing depositors to receive more rsETH than they are entitled to — diluting existing holders.

---

### Finding Description

`LRTOracle` stores the rsETH/ETH exchange rate in a persistent state variable:

```solidity
uint256 public override rsETHPrice;
``` [1](#0-0) 

This value is only updated when `updateRSETHPrice()` is called as a standalone transaction:

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [2](#0-1) 

The actual price computation happens inside `_updateRsETHPrice()`, which reads live on-chain balances from EigenLayer strategies and computes `newRsETHPrice = totalETHInProtocol / rsethSupply`, then writes it to `rsETHPrice`: [3](#0-2) 

In `LRTDepositPool`, both `depositETH()` and `depositAsset()` call `_beforeDeposit()`, which calls `getRsETHAmountToMint()`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [4](#0-3) 

`lrtOracle.rsETHPrice()` reads the **cached storage value**, not a freshly computed price. There is no call to `updateRSETHPrice()` anywhere inside the deposit flow: [5](#0-4) 

As EigenLayer staking rewards accrue over time, the true rsETH price rises. If `updateRSETHPrice()` has not been called since the last reward accrual, `rsETHPrice` is stale and lower than the actual price. The mint formula then divides by a smaller denominator, producing more rsETH than the deposited value warrants.

An additional complication: `_updateRsETHPrice()` enforces a `pricePercentageLimit` check that **reverts for non-manager callers** if the price increase exceeds the threshold:

```solidity
if (!IAccessControl(address(lrtConfig)).hasRole(LRTConstants.MANAGER, msg.sender)) {
    revert PriceAboveDailyThreshold();
}
``` [6](#0-5) 

This means that when rewards push the price above the threshold, only a manager can update it — extending the stale window and making the vulnerability more exploitable.

---

### Impact Explanation

An attacker who deposits during the stale-price window receives more rsETH than the deposited ETH/LST value justifies. When `updateRSETHPrice()` is eventually called, the attacker's rsETH represents a larger share of the protocol's TVL than they paid for. On redemption, they extract more ETH than they deposited. The excess comes directly from the yield that should have accrued to existing rsETH holders. This is **theft of unclaimed yield** — High severity.

---

### Likelihood Explanation

EigenLayer staking rewards accrue continuously. Every block that passes between a reward accrual event and the next `updateRSETHPrice()` call creates a stale-price window. The `updateRSETHPrice()` function is not called atomically within deposits, and the `pricePercentageLimit` guard can block permissionless updates when the price gap is large, widening the window. An attacker can monitor on-chain balances to detect when `rsETHPrice` is stale and deposit opportunistically.

---

### Recommendation

Call `updateRSETHPrice()` (or inline `_updateRsETHPrice()`) atomically at the start of `depositAsset()` and `depositETH()` before computing `rsethAmountToMint`, so the mint calculation always uses the freshly computed price rather than the cached storage value. Alternatively, replace `lrtOracle.rsETHPrice()` in `getRsETHAmountToMint()` with a live computation equivalent to `_getTotalEthInProtocol() / rsethSupply`.

---

### Proof of Concept

1. At time T, `rsETHPrice = 1.00 ETH` (cached). EigenLayer rewards accrue, making the true price `1.01 ETH`.
2. `updateRSETHPrice()` has not been called yet (or is blocked by `pricePercentageLimit` for non-managers).
3. Attacker calls `depositAsset(stETH, 100e18, 0, "")`.
4. `getRsETHAmountToMint` computes: `100e18 * 1e18 / 1.00e18 = 100 rsETH` (should be `~99.01 rsETH` at the true price).
5. Attacker receives `~0.99 rsETH` excess, representing yield stolen from existing holders.
6. After `updateRSETHPrice()` is called, attacker redeems `100 rsETH` for `~101 ETH`, extracting `~1 ETH` profit at the expense of existing rsETH holders. [7](#0-6) [8](#0-7) [2](#0-1) [9](#0-8)
