### Title
Fee-on-Transfer Token Causes rsETH Over-Minting Relative to Actual Received Assets - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool.depositAsset` computes the rsETH mint amount from the caller-supplied `depositAmount` parameter before the `safeTransferFrom` call, then mints rsETH based on that nominal figure. If the deposited asset is a fee-on-transfer token, the contract receives fewer tokens than `depositAmount`, yet mints rsETH as if the full `depositAmount` arrived. This inflates the rsETH supply relative to the actual collateral held, causing protocol insolvency over time.

### Finding Description
In `depositAsset`, the flow is:

1. `_beforeDeposit(asset, depositAmount, ...)` → calls `getRsETHAmountToMint(asset, depositAmount)` using the raw `depositAmount`.
2. `IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount)` → for a fee-on-transfer token, only `depositAmount * (1 - fee%)` actually lands in the contract.
3. `_mintRsETH(rsethAmountToMint)` → mints rsETH calculated from the full `depositAmount`.

The actual balance increase is never measured; the contract blindly trusts the input parameter.

The same pattern is present in:
- `RSETHPoolV3.deposit(address token, uint256 amount, ...)` — mints wrsETH based on `amount` before checking actual receipt.
- `RSETHPoolV3ExternalBridge.deposit(address token, uint256 amount, ...)` — identical pattern.
- `AGETHPoolV3.deposit(address token, uint256 amount, ...)` — mints agETH based on `amount`.
- `RsETHTokenWrapper._deposit` / `AGETHTokenWrapper._deposit` — mints wrapper tokens 1:1 with `_amount` without checking actual receipt.

The most critical instance is `LRTDepositPool.depositAsset` because it directly inflates the rsETH supply backed by the protocol's core collateral accounting.

### Impact Explanation
Every deposit with a fee-on-transfer token mints more rsETH than the collateral actually received. `getTotalAssetDeposits` reads live `balanceOf` values, so the oracle-computed rsETH price (`rsETHPrice`) will be lower than it should be for honest depositors, but the attacker holds rsETH backed by phantom collateral. When the attacker redeems, the shortfall is borne by all other rsETH holders — **protocol insolvency / theft of other users' funds**.

Impact: **Critical** — direct theft of funds / protocol insolvency.

### Likelihood Explanation
Any unprivileged user can call `depositAsset` with any ERC20 token that is on the supported asset list. If a fee-on-transfer token is ever added to the supported list (e.g., stETH has a 1–2 wei rounding "fee" in some transfer paths; future USDC/USDT fee modes), the vulnerability is immediately exploitable with no special access. The entry path is fully permissionless.

### Recommendation
Measure the actual balance increase around the transfer and use that as the basis for minting:

```solidity
uint256 balanceBefore = IERC20(asset).balanceOf(address(this));
IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
uint256 actualReceived = IERC20(asset).balanceOf(address(this)) - balanceBefore;

uint256 rsethAmountToMint = getRsETHAmountToMint(asset, actualReceived);
require(rsethAmountToMint >= minRSETHAmountExpected, "MinimumAmountToReceiveNotMet");
_mintRsETH(rsethAmountToMint);
```

Apply the same fix to `RSETHPoolV3.deposit`, `RSETHPoolV3ExternalBridge.deposit`, `AGETHPoolV3.deposit`, `RsETHTokenWrapper._deposit`, and `AGETHTokenWrapper._deposit`. Alternatively, explicitly whitelist only non-fee-on-transfer tokens and document this invariant.

### Proof of Concept

**Setup:** Assume a supported LST asset `XYZ` charges a 5% transfer fee.

1. Attacker calls `LRTDepositPool.depositAsset(XYZ, 1000e18, 0, "")`.
2. `_beforeDeposit` computes `rsethAmountToMint = getRsETHAmountToMint(XYZ, 1000e18)` — based on 1000 XYZ.
3. `safeTransferFrom` moves only **950 XYZ** into the contract (50 XYZ taken as fee).
4. `_mintRsETH` mints rsETH equivalent to **1000 XYZ** worth of value.
5. The attacker holds rsETH backed by 1000 XYZ of value but only 950 XYZ exists in the protocol.
6. Repeated deposits drain the collateral pool; honest withdrawers cannot redeem at par.
