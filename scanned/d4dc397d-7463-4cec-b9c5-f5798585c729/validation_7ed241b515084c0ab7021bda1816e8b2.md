### Title
Stale rsETH Rate Broadcast When LRTOracle Is Paused — (`contracts/cross-chain/RSETHRateProvider.sol`)

### Summary
`RSETHRateProvider.updateRate()` is callable by any unprivileged address and reads `LRTOracle.rsETHPrice` directly as a storage variable. When `LRTOracle` is paused, `rsETHPrice` is frozen at its pre-pause value, yet `updateRate()` will still broadcast that frozen value cross-chain via LayerZero, violating the invariant that cross-chain receivers should not receive rates from a degraded/paused oracle.

### Finding Description

**Call chain:**

```
anyone → RSETHRateProvider.updateRate()          [no access control, no pause check]
           └─ getLatestRate()
                └─ ILRTOracle(rsETHPriceOracle).rsETHPrice()   [plain storage read]
                     └─ returns frozen pre-pause value
           └─ ILayerZeroEndpoint.send(...)        [broadcasts stale rate cross-chain]
```

`CrossChainRateProvider.updateRate()` has no caller restriction and no check on the oracle's pause state: [1](#0-0) 

`RSETHRateProvider.getLatestRate()` reads `rsETHPrice` as a raw storage variable — it does not call any function that would revert when paused: [2](#0-1) 

`LRTOracle.rsETHPrice` is a plain `uint256` public state variable: [3](#0-2) 

`LRTOracle.updateRSETHPrice()` is gated by `whenNotPaused`, so once paused, `rsETHPrice` cannot be refreshed: [4](#0-3) 

The pause can be triggered legitimately by `PAUSER_ROLE` (e.g., manual security response) or automatically by `_updateRsETHPrice()` when a price-drop threshold is breached — in the auto-pause case the function returns early *before* writing the new price, so `rsETHPrice` retains the last pre-anomaly value: [5](#0-4) 

After the pause, any caller can invoke `RSETHRateProvider.updateRate()` (paying only the LayerZero fee) and the stale `rsETHPrice` is encoded and sent to the destination chain receiver, which overwrites its stored rate with the frozen value. [6](#0-5) 

### Impact Explanation
Cross-chain receivers (e.g., `RSETHRateReceiver`) serve the stale rate to downstream pools and integrations for the entire duration of the oracle pause. Any protocol that prices rsETH using the receiver's `getRate()` will misprice it relative to the true on-chain value. This fits **Low — contract fails to deliver promised returns, but doesn't lose value**: the rate provider's stated purpose is to relay the *current* rsETH/ETH rate; during a pause it instead relays a frozen, potentially incorrect rate.

### Likelihood Explanation
- The oracle pause is a normal, expected protocol event (both manual and automatic paths exist).
- `updateRate()` requires no role, only ETH for LayerZero fees — any address can trigger it.
- The longer the pause duration, the greater the rate divergence.

### Recommendation
Add a pause-awareness check in `RSETHRateProvider.getLatestRate()` (or in `updateRate()` itself) that reverts if `ILRTOracle(rsETHPriceOracle).paused()` returns `true`. This prevents broadcasting a frozen rate while the source oracle is in a degraded state.

```solidity
function getLatestRate() public view override returns (uint256) {
    require(!ILRTOracle(rsETHPriceOracle).paused(), "Oracle paused");
    return ILRTOracle(rsETHPriceOracle).rsETHPrice();
}
```

The same fix should be applied to `RSETHMultiChainRateProvider.getLatestRate()`, which has the identical pattern. [7](#0-6) 

### Proof of Concept

```solidity
// 1. Deploy LRTOracle, RSETHRateProvider (pointing to oracle), mock LayerZero endpoint
// 2. Call updateRSETHPrice() → rsETHPrice = X (e.g. 1.05e18)
// 3. PAUSER_ROLE calls LRTOracle.pause()
// 4. vm.warp(block.timestamp + 7 days)
//    → true rate would have drifted, but rsETHPrice is still X
// 5. anyEOA calls RSETHRateProvider.updateRate{value: fee}()
//    → getLatestRate() returns X (frozen pre-pause value)
//    → LayerZero endpoint receives payload encoding X
// 6. Assert: broadcast rate == X == pre-pause rsETHPrice
//            broadcast rate != current_true_rate
//    → stale rate successfully propagated cross-chain during oracle pause
```
