### Title
Depositor Can Mint Zero rsETH Shares Due to Rounding in `getRsETHAmountToMint` - (File: contracts/LRTDepositPool.sol)

### Summary
The `getRsETHAmountToMint` function performs integer division that can truncate to zero. When a caller passes `minRSETHAmountExpected = 0`, the protocol accepts the deposit, transfers the user's assets in, and mints 0 rsETH — permanently losing the deposited assets to the protocol with no shares issued.

### Finding Description
In `LRTDepositPool.getRsETHAmountToMint`, the rsETH amount to mint is computed as:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

This is plain integer division. Whenever `amount * assetPrice < rsETHPrice`, the result truncates to `0`.

The only guard against this in `_beforeDeposit` is:

```solidity
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

When `minRSETHAmountExpected = 0` (a valid caller-supplied value), the condition `0 < 0` is `false`, so no revert occurs. The deposit proceeds: assets are transferred from the user into the protocol and `_mintRsETH(0)` is called, minting nothing. There is no explicit `require(rsethAmountToMint > 0)` guard anywhere in the deposit path.

The `minAmountToDeposit` check only guards against a zero or sub-minimum *deposit amount*, not against a zero *output*:

```solidity
if (depositAmount == 0 || depositAmount < minAmountToDeposit) {
    revert InvalidAmountToDeposit();
}
```

`minAmountToDeposit` is initialized to `0` and is only raised by admin action, so by default any non-zero deposit passes this check.

### Impact Explanation
A depositor who passes `minRSETHAmountExpected = 0` and sends a dust-level amount (e.e., 1 wei of ETH or 1 unit of an LST) will have their assets permanently absorbed into the protocol TVL — increasing the value of all existing rsETH holders — while receiving 0 rsETH in return. The depositor suffers a complete loss of the deposited amount. The protocol does not lose value; the loss is borne entirely by the depositor.

**Impact: Low** — Contract fails to deliver promised returns (0 rsETH minted for a non-zero deposit), but the protocol itself does not lose value.

### Likelihood Explanation
The condition `amount * assetPrice < rsETHPrice` is satisfied for very small deposit amounts. Since rsETH price starts at `1e18` and grows with accumulated rewards, the threshold for a zero-mint result is sub-gwei for ETH deposits under normal conditions. The risk increases as rsETH appreciates. Any caller who omits a meaningful `minRSETHAmountExpected` slippage guard (e.g., passes `0`) is exposed. This is a realistic mistake for integrators or unsophisticated users.

**Likelihood: Low** — Requires a dust-level deposit with `minRSETHAmountExpected = 0`.

### Recommendation
Add an explicit zero-output guard in `_beforeDeposit`:

```solidity
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

if (rsethAmountToMint == 0) {
    revert ZeroRsETHMinted();
}

if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

This mirrors the recommendation in the referenced Lido report: add a check so that a user cannot receive zero shares for a non-zero deposit.

### Proof of Concept

1. rsETH price = `1.1e18` (rsETH has appreciated 10% above peg).
2. ETH asset price = `1e18`.
3. `minAmountToDeposit = 0` (default, admin has not set it).
4. Attacker/user calls `depositETH(0, "")` with `msg.value = 1 wei`.
5. `_beforeDeposit` is entered:
   - `depositAmount = 1`, `1 == 0` → false; `1 < 0` → false. Passes.
   - `getRsETHAmountToMint(ETH, 1)` → `(1 * 1e18) / 1.1e18 = 0` (integer truncation).
   - `0 < 0` → false. Passes.
6. `_mintRsETH(0)` is called → `IRSETH.mint(msg.sender, 0)` → 0 rsETH minted.
7. The 1 wei of ETH is now held by `LRTDepositPool`, counted in `getTotalAssetDeposits`, and increases the rsETH price for all existing holders.
8. The depositor holds 0 rsETH and has permanently lost 1 wei.

```text
    function _beforeDeposit(
        address asset,
        uint256 depositAmount,
        uint256 minRSETHAmountExpected
    )
        private
        view
        returns (uint256 rsethAmountToMint)
    {
        if (depositAmount == 0 || depositAmount < minAmountToDeposit) {
            revert InvalidAmountToDeposit();
        }

        if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
            revert MaximumDepositLimitReached();
        }

        rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

        if (rsethAmountToMint < minRSETHAmountExpected) {
            revert MinimumAmountToReceiveNotMet();
        }
    }
```
