### Title
Fee-on-Transfer Token Deposits Cause rsETH Over-Minting Due to Input-Amount vs. Received-Amount Mismatch - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool.depositAsset` computes `rsethAmountToMint` from the caller-supplied `depositAmount` parameter **before** the token transfer executes. If a supported ERC-20 asset charges a transfer fee (fee-on-transfer / deflationary token), the contract receives fewer tokens than `depositAmount`, yet mints rsETH as if the full `depositAmount` arrived. The same pattern is replicated in every L2 pool contract that accepts ERC-20 token deposits.

### Finding Description

In `LRTDepositPool.depositAsset`, the rsETH amount to mint is calculated from the input parameter `depositAmount` in `_beforeDeposit`, and then the token transfer is executed:

```solidity
uint256 rsethAmountToMint = _beforeDeposit(asset, depositAmount, minRSETHAmountExpected);
IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
_mintRsETH(rsethAmountToMint);
```

`_beforeDeposit` calls `getRsETHAmountToMint(asset, depositAmount)`, which computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

If the token deducts a fee on transfer, the contract's actual balance increases by `depositAmount - transferFee`, but `rsethAmountToMint` was computed against the full `depositAmount`. The minted rsETH therefore exceeds the real asset value deposited.

The identical pattern exists in the L2 pool contracts. In `RSETHPoolV3.deposit(address token, uint256 amount, ...)`:

```solidity
IERC20(token).safeTransferFrom(msg.sender, address(this), amount);
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
feeEarnedInToken[token] += fee;
wrsETH.mint(msg.sender, rsETHAmount);
```

`viewSwapRsETHAmountAndFee` uses `amount` (the parameter), not the actual post-transfer balance delta. The same code is duplicated verbatim in `RSETHPoolV3ExternalBridge.deposit` and `RSETHPoolNoWrapper.deposit`.

A secondary consequence: `feeEarnedInToken[token]` is inflated (computed on `amount`, not on the smaller received amount), causing `getTokenBalanceMinusFees` to understate the spendable balance and potentially revert on underflow when the bridger tries to move assets.

### Impact Explanation

Every deposit of a fee-on-transfer token mints rsETH (or wrsETH) in excess of the real asset value received. Over time, the rsETH total supply grows faster than the underlying asset base, making the protocol insolvent. Existing rsETH holders suffer dilution; a sophisticated depositor can repeatedly exploit this to extract value from the protocol at the expense of all other holders. This maps to **Critical – Protocol Insolvency**.

### Likelihood Explanation

The current canonical supported assets (stETH, wstETH, ETH) do not charge transfer fees, so the vulnerability is not immediately exploitable. However, `addSupportedToken` / `addSupportedToken(token, oracle, bridge)` in the L2 pools and the LRTConfig-based asset registry on L1 allow new tokens to be added. No code-level guard prevents a fee-on-transfer token from being registered. Once any such token is added, any depositor can exploit the mismatch. Likelihood is **Medium** (governance-gated token addition is the only barrier).

### Recommendation

Measure the actual received amount by comparing the contract's balance before and after `safeTransferFrom`, and use that delta for all downstream accounting:

```solidity
uint256 balanceBefore = IERC20(asset).balanceOf(address(this));
IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
uint256 actualReceived = IERC20(asset).balanceOf(address(this)) - balanceBefore;

uint256 rsethAmountToMint = _beforeDeposit(asset, actualReceived, minRSETHAmountExpected);
_mintRsETH(rsethAmountToMint);
```

Apply the same pattern in all L2 pool `deposit(token, amount, ...)` functions. Alternatively, explicitly document and enforce that only non-fee-on-transfer tokens may be added as supported assets, and add an on-chain invariant check.

### Proof of Concept

1. A fee-on-transfer token (e.g., 1% fee) is added as a supported asset via `addSupportedToken`.
2. Attacker calls `LRTDepositPool.depositAsset(feeToken, 1000e18, 0, "")`.
3. `_beforeDeposit` computes `rsethAmountToMint` based on `1000e18`.
4. `safeTransferFrom` executes; contract receives `990e18` (1% fee deducted).
5. `_mintRsETH` mints rsETH equivalent to `1000e18` worth of the asset.
6. Attacker received rsETH backed by only `990e18` of real assets — 10 rsETH worth of value extracted from the protocol per iteration.
7. Repeating this drains the protocol's asset-to-rsETH backing ratio, eventually making rsETH redeemable for less than its stated value.

        address lrtOracleAddress = lrtConfig.getContract(LRTConstants.LRT_ORACLE);
        ILRTOracle lrtOracle = ILRTOracle(lrtOracleAddress);

        // calculate rseth amount to mint based on asset amount and asset exchange rate
        rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
    }

**File:** contracts/LRTDepositPool.sol (L648-670)
```text
    function _beforeDeposit(
        address asset,
        uint256 depositAmount,
        uint256 minRSETHAmountExpected
    )
        private
        view
        returns (uint256 rsethAmountToMint)
    {
        if (depositAmount == 0 || depositAmount < minAmountToDeposit) {
            revert InvalidAmountToDeposit();
        }

        if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
            revert MaximumDepositLimitReached();
        }

        rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

        if (rsethAmountToMint < minRSETHAmountExpected) {
            revert MinimumAmountToReceiveNotMet();
        }
    }
```
