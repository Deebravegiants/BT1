### Title
`withinUnstakeLimits` Incorrectly Consumes `whitelistedUnstakeAllowance` When Active ETH Withdrawals Fully Cover the Unstake - (`contracts/LRTConverter.sol`)

### Summary
The `withinUnstakeLimits` modifier always drains `whitelistedUnstakeAllowance` whenever it is non-zero, even when `availableActiveETHWithdrawals` alone is sufficient to authorize the operation. The allowance should only be consumed when active ETH withdrawals are insufficient.

### Finding Description
The modifier at lines 58–77 of `contracts/LRTConverter.sol` performs two independent checks:

1. **Authorization gate** (line 65): reverts if `amountToUnstake > whitelistedUnstakeAllowance + availableActiveETHWithdrawals`. This correctly treats both buckets as a combined pool.

2. **Consumption logic** (lines 70–75): consumes `min(amountToUnstake, whitelistedUnstakeAllowance)` from `whitelistedUnstakeAllowance` whenever `whitelistedUnstakeAllowance > 0`, with **no check** on whether `availableActiveETHWithdrawals` alone already covers `amountToUnstake`. [1](#0-0) 

The condition at line 70 is `if (whitelistedUnstakeAllowance > 0)` — it is entirely blind to the value of `availableActiveETHWithdrawals`. The correct condition should be `if (amountToUnstake > availableActiveETHWithdrawals && whitelistedUnstakeAllowance > 0)`.

`availableActiveETHWithdrawals` is sourced from `lrtWithdrawalManager.assetsCommitted(ETH_TOKEN)`, which tracks pending user ETH withdrawal commitments. [2](#0-1) 

### Impact Explanation
`whitelistedUnstakeAllowance` is depleted faster than intended. Future calls to `unstakeStEth` that should be authorized by the whitelisted budget will revert with `UnstakeLimitExceeded` because the allowance was silently consumed by prior calls that were already fully covered by active ETH withdrawals. No funds are lost, but the contract fails to deliver the promised unstaking capacity. This matches **Low: Contract fails to deliver promised returns, but doesn't lose value**.

### Likelihood Explanation
The trigger requires only two ordinary, non-malicious conditions to hold simultaneously:
- `whitelistedUnstakeAllowance > 0` (a whitelisted user has declared withdrawal intent via `declareWithdrawalIntent`)
- `assetsCommitted[ETH_TOKEN] >= amountToUnstake` (there are active user ETH withdrawal requests in `LRTWithdrawalManager`)

Both are routine operational states. A legitimate LRT operator calling `unstakeStEth` under these conditions will silently exhaust the whitelisted budget.

### Recommendation
Change the consumption block to only draw from `whitelistedUnstakeAllowance` when `availableActiveETHWithdrawals` is insufficient:

```solidity
// Consume whitelisted allowance only for the portion not covered by active withdrawals
if (amountToUnstake > availableActiveETHWithdrawals && whitelistedUnstakeAllowance > 0) {
    uint256 deficit = amountToUnstake - availableActiveETHWithdrawals;
    uint256 whitelistedAmountConsumed =
        deficit > whitelistedUnstakeAllowance ? whitelistedUnstakeAllowance : deficit;
    whitelistedUnstakeAllowance -= whitelistedAmountConsumed;
}
```

### Proof of Concept
```
Initial state:
  whitelistedUnstakeAllowance = 50e18   (W)
  assetsCommitted[ETH_TOKEN]  = 100e18  (A, via LRTWithdrawalManager)

Operator calls: unstakeStEth(50e18)

withinUnstakeLimits executes:
  availableActiveETHWithdrawals = 100e18
  50e18 > 50e18 + 100e18  → false, no revert
  whitelistedUnstakeAllowance > 0  → true
  whitelistedAmountConsumed = min(50e18, 50e18) = 50e18
  whitelistedUnstakeAllowance = 50e18 - 50e18 = 0

Post-call assertion:
  whitelistedUnstakeAllowance == 0   ← WRONG, should still be 50e18
  availableActiveETHWithdrawals (100e18) fully covered the request
  whitelisted budget was silently zeroed out
```

A subsequent `unstakeStEth(1e18)` call with `assetsCommitted[ETH_TOKEN] = 0` will now revert with `UnstakeLimitExceeded` even though the whitelisted budget should have been intact.
