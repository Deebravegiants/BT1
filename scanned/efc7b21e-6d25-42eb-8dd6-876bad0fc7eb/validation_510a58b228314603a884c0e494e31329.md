### Title
Unbounded O(n) Linear Scan in `addWithdrawal` Enables Gas-DoS Under High-Volume Bridging — (`contracts/utils/HashStorage.sol`)

---

### Summary

`HashStorage.addWithdrawal` performs a full linear scan of the `withdrawalTxHashes` storage array on every call to detect duplicates. Because the array is unbounded and no O(1) existence mapping is maintained for pending hashes, gas cost grows proportionally with array length. Under sustained high-volume L2→L1 bridge activity where registration outpaces claim processing, the array can grow to a size where both `addWithdrawal` and `setWithdrawalClaimed` exceed the block gas limit, stalling the entire withdrawal pipeline.

---

### Finding Description

`addWithdrawal` iterates over every element of `withdrawalTxHashes` in storage before pushing a new hash: [1](#0-0) 

```solidity
uint256 length = withdrawalTxHashes.length;
for (uint256 i = 0; i < length; i++) {
    if (withdrawalTxHashes[i] == txHash) {
        revert DuplicateHash();
    }
}
claimedWithdrawals[txHash] = false;
withdrawalTxHashes.push(txHash);
```

Each iteration reads a distinct storage slot. On Ethereum, a cold `SLOAD` costs **2,100 gas**. With a 30M gas block limit, the loop alone exhausts the block at approximately **~14,285 elements** (30,000,000 / 2,100). In practice, with loop overhead, the threshold is lower.

`setWithdrawalClaimed` has the same O(n) scan: [2](#0-1) 

Once the array is large enough, **both** registration and claim processing become impossible in a single transaction, creating a mutual deadlock: you cannot claim to shrink the array because claiming itself OOGs.

The `claimedWithdrawals` mapping already exists but only tracks *claimed* hashes — there is no O(1) lookup for *pending* hashes, which is why the loop is used for duplicate detection. [3](#0-2) 

---

### Impact Explanation

Once the array crosses the gas-DoS threshold:
- `addWithdrawal` reverts OOG → new L2→L1 bridge withdrawals cannot be registered.
- `setWithdrawalClaimed` reverts OOG → existing pending withdrawals cannot be claimed.
- Funds bridged from L2 are temporarily frozen in the pipeline until the array is drained below the threshold, which itself requires successful `setWithdrawalClaimed` calls — a circular dependency.

**Impact: Medium — Temporary freezing of funds / Unbounded gas consumption.**

---

### Likelihood Explanation

The scenario requires only that the operator (acting in good faith) registers hashes faster than they are claimed over a sustained period. This is realistic during:
- Liquidity migrations generating many simultaneous L2→L1 withdrawals.
- Any period where L1 claim processing is delayed (e.g., network congestion, operational lag).

No malicious actor is required. The operator role is a trusted but non-privileged operational role that is expected to call `addWithdrawal` routinely.

---

### Recommendation

Add a `pendingWithdrawals` mapping for O(1) duplicate detection, eliminating the linear scan entirely:

```solidity
mapping(bytes32 txHash => bool isPending) public pendingWithdrawals;

function addWithdrawal(bytes32 txHash) external onlyRole(OPERATOR_ROLE) {
    if (txHash == bytes32(0)) revert InvalidHash();
    if (claimedWithdrawals[txHash]) revert HashAlreadyClaimed();
    if (pendingWithdrawals[txHash]) revert DuplicateHash();   // O(1)

    pendingWithdrawals[txHash] = true;
    withdrawalTxHashes.push(txHash);
    emit WithdrawalAdded(txHash);
}

function setWithdrawalClaimed(bytes32 txHash) external onlyRole(OPERATOR_ROLE) {
    if (txHash == bytes32(0)) revert InvalidHash();
    if (!pendingWithdrawals[txHash]) revert InvalidHash();    // O(1)
    if (claimedWithdrawals[txHash]) revert HashAlreadyClaimed();

    pendingWithdrawals[txHash] = false;
    claimedWithdrawals[txHash] = true;
    // array removal still needed for getWithdrawalTxHashes(), or replace array with enumerable set
    ...
}
```

Alternatively, replace the `bytes32[]` array with OpenZeppelin's `EnumerableSet.Bytes32Set`, which provides O(1) add, remove, and contains operations backed by a mapping.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import "forge-std/Test.sol";
import "contracts/utils/HashStorage.sol";

contract HashStorageGasDoSTest is Test {
    HashStorage hs;
    address admin  = address(0xA);
    address operator = address(0xB);

    function setUp() public {
        hs = new HashStorage(admin, operator, "test");
    }

    function test_gasDoS() public {
        vm.startPrank(operator);

        // Fill array with N hashes without claiming any
        uint256 N = 15_000;
        for (uint256 i = 1; i <= N; i++) {
            hs.addWithdrawal(bytes32(i));
        }

        // Measure gas for the (N+1)-th addWithdrawal
        uint256 gasBefore = gasleft();
        hs.addWithdrawal(bytes32(N + 1));
        uint256 gasUsed = gasBefore - gasleft();

        // At ~2100 gas/SLOAD * 15000 iterations ≈ 31.5M gas > 30M block limit
        assertGt(gasUsed, 30_000_000, "OOG threshold reached");
        vm.stopPrank();
    }
}
```

Run with: `forge test --match-test test_gasDoS --gas-limit 35000000 -vvv`

The test demonstrates that at ~14,000–15,000 unclaimed hashes, `addWithdrawal` exceeds the 30M block gas limit, permanently stalling registration until the backlog is cleared — which itself requires the same OOG-prone `setWithdrawalClaimed`.
