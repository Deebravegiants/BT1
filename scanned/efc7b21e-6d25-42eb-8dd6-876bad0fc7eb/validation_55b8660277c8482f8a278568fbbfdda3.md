### Title
Lack of Slippage Control in L2 Pool `deposit` Functions Allows Users to Receive Fewer wrsETH Than Expected - (`contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`, `contracts/pools/RSETHPoolV3WithNativeChainBridge.sol`, `contracts/pools/RSETHPoolV2ExternalBridge.sol`)

---

### Summary

Multiple L2 pool `deposit` functions compute the amount of wrsETH to mint using a live oracle rate (`getRate()`), but accept no `minRSETHAmountExpected` parameter. If the oracle rate rises between a user's off-chain simulation and on-chain execution, the user receives fewer wrsETH tokens than anticipated with no ability to revert.

---

### Finding Description

Every L2 pool deposit path follows the same pattern: the user sends ETH (or a supported token), the contract calls `viewSwapRsETHAmountAndFee()` which reads `getRate()` from the oracle at execution time, and mints exactly that many wrsETH tokens. No minimum-output guard exists.

- `RSETHPoolV3WithNativeChainBridge.sol` ETH deposit [6](#0-5) 
- `RSETHPoolV3WithNativeChainBridge.sol` token deposit [7](#0-6) 
- `RSETHPoolV2ExternalBridge.sol` ETH deposit [8](#0-7) 

By contrast, the L1 `LRTDepositPool` correctly enforces a `minRSETHAmountExpected` check: [9](#0-8) 

The oracle rate (`rsETHToETHrate`) represents the ETH value of one rsETH unit. Because rsETH is a yield-bearing token, this rate increases monotonically over time as staking rewards accrue. An oracle update that pushes the rate higher between a user's simulation and their transaction execution will reduce the wrsETH minted:

```
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

A higher `rsETHToETHrate` → fewer `rsETHAmount` minted for the same ETH input. [10](#0-9) 

---

### Impact Explanation

Users receive fewer wrsETH tokens than they computed off-chain. The shortfall is permanent — the ETH is already deposited and the mint is final. The user cannot recover the difference. This maps to **Low: Contract fails to deliver promised returns, but doesn't lose value**, since the wrsETH received still represents a valid claim on underlying assets, just at a quantity below the user's expectation.

---

### Likelihood Explanation

The rsETH oracle rate is updated periodically (e.g., on each cross-chain rate push or on-chain oracle refresh). Any deposit transaction that lands in a block immediately after an oracle update will silently receive fewer tokens than the user simulated. This is a routine, predictable on-chain event requiring no attacker — it is triggered by normal protocol operation. Any unprivileged depositor calling `deposit()` on any of the affected L2 pool contracts is exposed.

---

### Recommendation

Add a `minRSETHAmountExpected` parameter to every `deposit` function in the affected L2 pool contracts and revert if the computed `rsETHAmount` falls below it, mirroring the pattern already used in `LRTDepositPool._beforeDeposit()`:

```solidity
if (rsETHAmount < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet();
``` [11](#0-10) 

---

### Proof of Concept

1. Alice calls `RSETHPoolV3.deposit{value: 10 ether}("ref")` off-chain and simulates receiving `X` wrsETH at the current oracle rate `R`.
2. Before Alice's transaction is mined, the rsETH oracle is updated, increasing the rate from `R` to `R'` (where `R' > R`).
3. Alice's transaction executes. `viewSwapRsETHAmountAndFee(10 ether)` now computes:
   - `fee = 10 ether * feeBps / 10_000`
   - `rsETHAmount = (10 ether - fee) * 1e18 / R'`
4. Because `R' > R`, `rsETHAmount < X`. Alice receives fewer wrsETH than expected with no recourse.
5. Alice cannot cancel or adjust — her ETH is already transferred and the mint is complete. [12](#0-11)

**File:** contracts/pools/RSETHPoolV3.sol (L299-308)
```text
    function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
    }
```
