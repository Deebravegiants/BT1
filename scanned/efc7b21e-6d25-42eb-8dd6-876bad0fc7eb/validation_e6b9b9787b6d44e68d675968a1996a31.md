### Title
Instant Withdrawal Drains Vault Assets Committed to Standard Queue, Temporarily Freezing Queued Withdrawers' Funds — (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary

`instantWithdrawal` in `LRTWithdrawalManager` allows any rsETH holder to burn tokens and redeem assets directly from `LRTUnstakingVault`, bypassing the standard withdrawal queue. The only protection against draining assets committed to queued withdrawals is `queuedWithdrawalsBuffer` in `LRTUnstakingVault`, which **defaults to zero** and is **never automatically synchronized** with `assetsCommitted[asset]` in `LRTWithdrawalManager`. When instant withdrawal is enabled for an asset, an attacker can drain vault assets that are already committed to queued withdrawers, causing `unlockQueue` to fail and temporarily freezing those users' funds.

---

### Finding Description

The standard withdrawal flow is: `initiateWithdrawal` → operator calls `unlockQueue` → `completeWithdrawal`. When a user calls `initiateWithdrawal`, `assetsCommitted[asset]` is incremented to reserve the expected payout: [1](#0-0) 

`instantWithdrawal` is a parallel path that burns rsETH and redeems directly from `LRTUnstakingVault`: [2](#0-1) 

The availability check inside `instantWithdrawal` delegates to `LRTUnstakingVault.getAssetsAvailableForInstantWithdrawal`: [3](#0-2) 

This function subtracts `queuedWithdrawalsBuffer[asset]` from the vault balance. That buffer is a **manually set, static value** with a **default of zero**: [4](#0-3) 

There is no code path that automatically updates `queuedWithdrawalsBuffer` when `assetsCommitted[asset]` grows in `LRTWithdrawalManager`. The two accounting systems are completely decoupled. As a result, when instant withdrawal is enabled and the buffer is at its default (zero), the entire vault balance is exposed to instant withdrawals regardless of how much has been committed to the standard queue.

---

### Impact Explanation

**Medium — Temporary freezing of funds.**

When an attacker drains vault assets via `instantWithdrawal`, the operator's subsequent call to `unlockQueue` attempts to call `unstakingVault.redeem(asset, assetAmountUnlocked)`: [5](#0-4) 

If the vault balance is insufficient, this transfer fails and `unlockQueue` reverts. Queued withdrawers cannot progress to `completeWithdrawal` until the vault is replenished (e.g., by completing a new EigenLayer withdrawal cycle, which takes ~7 days). Their rsETH has already been transferred to `LRTWithdrawalManager` and is locked there for the duration.

---

### Likelihood Explanation

**Medium.** The precondition is that a manager has called `setInstantWithdrawalEnabled(asset, true)`. Once enabled, any rsETH holder — with no special role — can call `instantWithdrawal` repeatedly. The buffer defaults to zero, so no additional misconfiguration is required. The attacker only needs enough rsETH to match the vault balance, which is publicly observable on-chain. [6](#0-5) 

---

### Recommendation

1. **Synchronize the buffer automatically.** When `initiateWithdrawal` increments `assetsCommitted[asset]`, call `unstakingVault.setQueuedWithdrawalsBuffer(asset, assetsCommitted[asset])` so the vault always reserves at least the committed amount.
2. **Add an explicit check in `instantWithdrawal`.** Before burning rsETH, verify that the vault balance minus `assetAmountUnlocked` is still sufficient to cover `assetsCommitted[asset]` for the standard queue.
3. **Set a non-zero default buffer** equal to the current `assetsCommitted` value whenever instant withdrawal is enabled for an asset.

---

### Proof of Concept

1. Protocol state: vault holds 100 ETH, EigenLayer holds 100 ETH. Total = 200 ETH.
2. User A calls `initiateWithdrawal(ETH, rsETH_for_100_ETH)`. `assetsCommitted[ETH] = 100 ETH`. `getAvailableAssetAmount` returns 100 ETH (200 − 100), so the request is accepted.
3. Manager calls `setInstantWithdrawalEnabled(ETH, true)`. `queuedWithdrawalsBuffer[ETH]` is still 0 (never set).
4. Attacker calls `instantWithdrawal(ETH, rsETH_for_100_ETH)`. `getAssetsAvailableForInstantWithdrawal(ETH)` returns `100 − 0 = 100 ETH`. Check passes. Vault is drained to 0.
5. Operator calls `unlockQueue(ETH, ...)`. It computes `assetAmountUnlocked = 100 ETH` and calls `unstakingVault.redeem(ETH, 100 ETH)`. The vault has 0 ETH — the call reverts.
6. User A's rsETH is locked in `LRTWithdrawalManager`. They cannot `completeWithdrawal` until the vault is replenished via a new EigenLayer exit (~7 days). [7](#0-6) [8](#0-7)

        address asset,
        uint256 rsETHUnstaked,
        string calldata referralId
    )
        external
        nonReentrant
        whenNotPaused
        onlySupportedAsset(asset)
        onlySupportedStrategy(asset)
        onlyInstantWithdrawalAllowed(asset)
    {
        if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
            revert InvalidAmountToWithdraw();
        }
        if (IERC20(lrtConfig.rsETH()).balanceOf(msg.sender) < rsETHUnstaked) revert NotEnoughRsETH();
        uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
        IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
        ILRTUnstakingVault unstakingVault = ILRTUnstakingVault(lrtConfig.getContract(LRTConstants.LRT_UNSTAKING_VAULT));
        if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
            revert CantInstantWithdrawMoreThanAvailable();
        }

        unstakingVault.redeem(asset, assetAmountUnlocked);

        uint256 fee = (assetAmountUnlocked * instantWithdrawalFee) / 10_000;
        uint256 userAmount = assetAmountUnlocked - fee;

        address feeRecipient = instantWithdrawalFeeRecipient;
        if (feeRecipient == address(0)) {
            // Backwards-compatible default: send fees to the protocol treasury
            feeRecipient = lrtConfig.getContract(LRTConstants.PROTOCOL_TREASURY);
        }
        if (fee > 0) {
            _transferAsset(asset, feeRecipient, fee);
            emit InstantWithdrawalFeeCollected(msg.sender, asset, fee);
        }

        _transferAsset(asset, msg.sender, userAmount);
        emit ReferralIdEmitted(referralId);
        emit AssetWithdrawalFinalized(msg.sender, asset, rsETHUnstaked, userAmount);
    }

        external
        view
        onlySupportedAsset(asset)
        returns (uint256 availableAmount)
    {
        uint256 vaultBalance = balanceOf(asset);
        uint256 reservedBuffer = queuedWithdrawalsBuffer[asset];
        availableAmount = reservedBuffer >= vaultBalance ? 0 : vaultBalance - reservedBuffer;
    }
