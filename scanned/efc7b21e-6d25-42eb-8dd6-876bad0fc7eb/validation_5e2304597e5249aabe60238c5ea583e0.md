### Title
`FeeReceiver` MEV Rewards Excluded from TVL Before `updateRSETHPrice()`, Causing Understated rsETH Price - (File: contracts/LRTOracle.sol, contracts/FeeReceiver.sol, contracts/LRTDepositPool.sol)

---

### Summary

`LRTOracle.updateRSETHPrice()` computes the rsETH/ETH exchange rate using `_getTotalEthInProtocol()`, which calls `LRTDepositPool.getETHDistributionData()`. That function explicitly excludes ETH sitting in `FeeReceiver` (the MEV/execution-layer reward receiver). Because `FeeReceiver.sendFunds()` is never called inside `updateRSETHPrice()`, any accumulated rewards not yet flushed to the deposit pool are invisible to the price calculation. New depositors therefore receive more rsETH than they are entitled to, diluting existing holders' yield.

---

### Finding Description

`FeeReceiver` accumulates ETH from MEV and execution-layer rewards via its `receive()` fallback. The only way those funds enter the TVL calculation is through `sendFunds()`, which transfers the balance to `LRTDepositPool`:

```solidity
// contracts/FeeReceiver.sol:53-58
function sendFunds() external {          // ← no access control
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
``` [1](#0-0) 

`LRTOracle._updateRsETHPrice()` computes total ETH via `_getTotalEthInProtocol()`:

```solidity
// contracts/LRTOracle.sol:331-348
function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
    ...
    uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
    totalETHInProtocol += totalAssetAmt.mulWad(assetER);
    ...
}
``` [2](#0-1) 

`getTotalAssetDeposits` for ETH delegates to `getETHDistributionData()`, which the codebase itself documents as excluding `FeeReceiver`:

```solidity
// contracts/LRTDepositPool.sol:465-466
/// @dev rewards are not accounted here
/// it will automatically be accounted once it is moved from feeReceiver/rewardReceiver to depositPool
``` [3](#0-2) 

The six components summed in `getETHDistributionData()` are: deposit pool balance, NDC balances, EigenLayer shares, queued withdrawals, unstaking vault balance, and converter ETH. `FeeReceiver.balance` is absent from all of them. [4](#0-3) 

`updateRSETHPrice()` is public and callable by anyone:

```solidity
// contracts/LRTOracle.sol:87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [5](#0-4) 

The rsETH mint amount is computed as:

```solidity
// contracts/LRTDepositPool.sol:520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [6](#0-5) 

A lower `rsETHPrice` (because FeeReceiver rewards are excluded) directly inflates `rsethAmountToMint` for every depositor.

---

### Impact Explanation

**High — Theft of unclaimed yield.**

When MEV rewards accumulate in `FeeReceiver` and `updateRSETHPrice()` is called before `sendFunds()`, the rsETH price is understated. Every deposit made at the understated price mints excess rsETH. When `sendFunds()` is eventually called and the price is updated, the excess rsETH already issued to new depositors permanently dilutes the yield owed to existing holders. The protocol fee calculation (`rewardAmount = totalETHInProtocol - previousTVL`) is also understated for the same reason, reducing the fee taken on behalf of the treasury. [7](#0-6) 

---

### Likelihood Explanation

`FeeReceiver` receives ETH continuously from MEV and execution-layer rewards. `updateRSETHPrice()` is a public, permissionless function. Any depositor can call `updateRSETHPrice()` at a moment when `FeeReceiver` holds a large balance (observable on-chain), then immediately deposit to capture the inflated rsETH mint. No privileged access is required. The window exists between every MEV reward receipt and the next `sendFunds()` call.

---

### Recommendation

Call `FeeReceiver.sendFunds()` (or an equivalent internal flush) at the start of `_updateRsETHPrice()` before computing `_getTotalEthInProtocol()`, so that all accrued MEV rewards are included in the TVL used for the exchange rate:

```solidity
function _updateRsETHPrice() internal {
    // Flush pending MEV rewards into the deposit pool first
    IFeeReceiver(lrtConfig.getContract(LRTConstants.REWARD_RECEIVER)).sendFunds();

    address rsETHTokenAddress = lrtConfig.rsETH();
    ...
}
```

This mirrors the recommendation in the reference report: the reward-claiming step must precede the TVL snapshot.

---

### Proof of Concept

1. MEV rewards accumulate: `FeeReceiver.balance = 500 ETH`.
2. Attacker (or anyone) calls `LRTOracle.updateRSETHPrice()`. `_getTotalEthInProtocol()` sums all protocol ETH **excluding** the 500 ETH in `FeeReceiver`. Suppose true TVL is 10 500 ETH and rsETH supply is 10 000; correct price = 1.05 ETH/rsETH. Recorded price = 1.00 ETH/rsETH (500 ETH missing).
3. Attacker calls `LRTDepositPool.depositETH{value: 100 ether}(0, "")`. Mint amount = `100e18 * 1e18 / 1.00e18 = 100 rsETH` instead of the correct `100e18 * 1e18 / 1.05e18 ≈ 95.24 rsETH`. Attacker receives ~4.76 excess rsETH.
4. `sendFunds()` is called; 500 ETH moves to deposit pool.
5. `updateRSETHPrice()` is called again; price rises to reflect the full TVL. Attacker's excess rsETH now represents real value extracted from existing holders. [8](#0-7) [9](#0-8)
