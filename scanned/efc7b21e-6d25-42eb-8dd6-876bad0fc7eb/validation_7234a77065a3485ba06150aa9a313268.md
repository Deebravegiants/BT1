### Title
`withinUnstakeLimits` Modifier Fails to Consume `assetsCommitted` Budget, Enabling Repeated Over-Unstaking of stETH — (`contracts/LRTConverter.sol`)

---

### Summary

The `withinUnstakeLimits` modifier in `LRTConverter` reads `assetsCommitted[ETH_TOKEN]` from `LRTWithdrawalManager` as an upper bound for how much stETH may be unstaked, but **never decrements it**. Only `whitelistedUnstakeAllowance` is consumed. Because the same `assetsCommitted` value is visible on every subsequent call, the operator can invoke `unstakeStEth` an arbitrary number of times against the same user-withdrawal budget, over-unstaking stETH far beyond what users have actually committed to withdraw.

---

### Finding Description

`withinUnstakeLimits` enforces:

```
amountToUnstake ≤ whitelistedUnstakeAllowance + assetsCommitted[ETH_TOKEN]
``` [1](#0-0) 

After the check passes, the modifier subtracts from `whitelistedUnstakeAllowance` only:

```solidity
if (whitelistedUnstakeAllowance > 0) {
    uint256 whitelistedAmountConsumed = ...;
    whitelistedUnstakeAllowance -= whitelistedAmountConsumed;
}
``` [2](#0-1) 

`assetsCommitted[ETH_TOKEN]` is **never touched** by the modifier. It is a live read from `LRTWithdrawalManager`:

```solidity
activeETHWithdrawals = lrtWithdrawalManager.assetsCommitted(LRTConstants.ETH_TOKEN);
``` [3](#0-2) 

`assetsCommitted` is only decremented inside `_unlockWithdrawalRequests`, which is called by `unlockQueue`, not by anything in `LRTConverter`:

```solidity
assetsCommitted[asset] -= request.expectedAssetAmount;
``` [4](#0-3) 

Therefore, between the moment a user calls `initiateWithdrawal` (which increments `assetsCommitted`) and the moment the operator calls `unlockQueue` (which decrements it), the operator can call `unstakeStEth` **N times** with the full committed amount each time, because the guard never records that the budget was already spent.

---

### Impact Explanation

Each `unstakeStEth(X)` call submits a Lido withdrawal request for X stETH. After N calls, N×X stETH is queued for redemption. When Lido pays out, the protocol receives N×X ETH into the deposit pool, but only X ETH worth of user withdrawal claims exist. The surplus ETH is not matched to any user liability, but the stETH that backed other user positions has been permanently removed from the protocol's asset base. Users whose rsETH was backed by that stETH can no longer be made whole — **protocol insolvency**.

---

### Likelihood Explanation

`unstakeStEth` is `onlyLRTOperator`. The exploit does not require a malicious operator; it can be triggered by:

- An operator automation script that retries on failure and calls the function twice for the same pending withdrawal batch.
- An operator calling `unstakeStEth` in two separate transactions before `unlockQueue` is processed (a normal operational pattern when unstaking in tranches).

No user interaction beyond a standard `initiateWithdrawal` is required to set up the precondition.

---

### Recommendation

Track cumulative stETH unstaked against the `assetsCommitted` budget. One approach: maintain a storage variable `ethCommittedConsumedByUnstaking` that is incremented inside `withinUnstakeLimits` by the portion of `amountToUnstake` attributed to `assetsCommitted`, and reset (or decremented) when `unlockQueue` processes the corresponding requests. Alternatively, require that `unstakeStEth` can only be called once per `unlockQueue` cycle by recording the `nextLockedNonce` at the time of unstaking and rejecting re-entry until it advances.

---

### Proof of Concept

```solidity
// Fork test (local fork, no mainnet)
// Precondition: whitelistedUnstakeAllowance == 0

// 1. User initiates ETH withdrawal for X rsETH → assetsCommitted[ETH_TOKEN] = X
withdrawalManager.initiateWithdrawal(ETH_TOKEN, rsETHAmount, "");
uint256 committed = withdrawalManager.assetsCommitted(ETH_TOKEN); // == X

// 2. Operator calls unstakeStEth(X) — passes: X <= 0 + X
converter.unstakeStEth(committed);

// 3. assetsCommitted is still X (modifier never decremented it)
assertEq(withdrawalManager.assetsCommitted(ETH_TOKEN), committed);

// 4. Operator calls unstakeStEth(X) again — passes again: X <= 0 + X
converter.unstakeStEth(committed);

// 5. Total stETH queued for unstaking = 2X, but only X ETH is owed to users
// → over-unstaking by X, protocol insolvency
```

The `withinUnstakeLimits` check at line 65 passes both times because `assetsCommitted` is read-only within the modifier. [5](#0-4)

    modifier withinUnstakeLimits(uint256 amountToUnstake) {
        if (amountToUnstake == 0) {
            revert InvalidAmount();
        }

        uint256 availableActiveETHWithdrawals = _getActiveETHUserWithdrawals();

        if (amountToUnstake > whitelistedUnstakeAllowance + availableActiveETHWithdrawals) {
            revert UnstakeLimitExceeded();
        }

        // Consume intended withdrawal limit
        if (whitelistedUnstakeAllowance > 0) {
            uint256 whitelistedAmountConsumed =
                amountToUnstake > whitelistedUnstakeAllowance ? whitelistedUnstakeAllowance : amountToUnstake;

            whitelistedUnstakeAllowance -= whitelistedAmountConsumed;
        }
        _;
    }
```
