### Title
UID Collision in `bridgeTokenToL1` Due to Block-Level Entropy Reuse Causes Bridge Operation Failure — (File: `contracts/bridges/SonicChainNativeTokenBridge.sol`)

---

### Summary

`SonicChainNativeTokenBridge.bridgeTokenToL1` generates a `uint96 uid` by hashing `block.timestamp`, `block.number`, `msg.sender`, `recipient`, `amount`, `tx.gasprice`, and `bridgeReceiver`. On Sonic chain, all of these values are shared across every transaction in the same block. Two calls with identical parameters in the same block produce the same UID, which is then passed to `sonicBridge.withdraw(uid, ...)`. If the Sonic bridge enforces UID uniqueness (the expected behavior for a deduplication key), the second call reverts after the user's tokens have already been pulled — or, if the bridge silently accepts the duplicate, one withdrawal record is silently overwritten, permanently losing the first user's bridge operation.

---

### Finding Description

In `bridgeTokenToL1` (lines 92–100), the UID is computed as:

```solidity
uint96 uid = uint96(
    uint256(
        keccak256(
            abi.encodePacked(
                block.timestamp, block.number, msg.sender, recipient, amount, tx.gasprice, bridgeReceiver
            )
        )
    ) % type(uint96).max
);
```

Every input to this hash is either a block-level global (`block.timestamp`, `block.number`) or a transaction parameter that can be identical across two separate transactions in the same block (`msg.sender`, `recipient`, `amount`, `tx.gasprice`). `bridgeReceiver` is a contract constant. There is no per-call nonce, no `tx.origin`, and no counter.

The fallback path (line 104) is even weaker — it drops `amount` and `recipient` entirely:

```solidity
uid = uint96(uint256(keccak256(abi.encodePacked(block.number, msg.sender, tx.gasprice, bridgeReceiver))));
```

This means two different callers bridging different amounts in the same block with the same gas price can collide on the fallback UID.

The contract is explicitly deployed on Sonic chain (an L2). Sonic produces blocks rapidly; multiple transactions from the same automated caller (e.g., `RSETHPoolV3AutoBridgedTokens`, referenced in the contract's NatSpec) with the same parameters can land in the same block.

---

### Impact Explanation

**Scenario A — Sonic bridge reverts on duplicate UID (expected behavior):**
The second `bridgeTokenToL1` call reverts at `sonicBridge.withdraw(uid, ...)`. Because `safeTransferFrom` executes before the `withdraw` call (line 83 before line 115), the revert unwinds the token transfer and the user's tokens are returned. The bridge operation silently fails. The caller receives no tokens on L1 and must retry.

**Scenario B — Sonic bridge silently accepts duplicate UID:**
The second `withdraw` call overwrites or shadows the first withdrawal record. The first caller's tokens have already been transferred to the bridge contract and burned/locked, but their withdrawal record is gone. This results in permanent loss of the first caller's funds.

Impact: **Medium — Temporary freezing of funds** (Scenario A, conservative) / **Critical — Permanent freezing of funds** (Scenario B).

---

### Likelihood Explanation

The `SonicChainNativeTokenBridge` is designed to be called by pool contracts (`RSETHPoolV3AutoBridgedTokens`) that may batch or repeat bridge calls. On Sonic chain, fast block times increase the probability that two calls with identical parameters land in the same block. The collision requires only that `msg.sender`, `recipient`, `amount`, and `tx.gasprice` match across two transactions in the same block — a realistic condition for automated pool-level bridging. Likelihood: **Medium**.

---

### Recommendation

Replace the entropy-based UID with a monotonically incrementing per-contract nonce:

```solidity
uint96 private _bridgeNonce;

// inside bridgeTokenToL1:
uint96 uid = ++_bridgeNonce;
```

Alternatively, incorporate `tx.origin` and a contract-level counter to guarantee uniqueness regardless of block-level globals. Do not rely solely on `block.number`, `block.timestamp`, or `tx.gasprice` for uniqueness on any L2.

---

### Proof of Concept

1. Deploy `SonicChainNativeTokenBridge` on Sonic chain.
2. In the same Sonic block, submit two transactions:
   - Tx A: `bridgeTokenToL1(recipientX, 1e18)` from `callerA` with `gasPrice = G`
   - Tx B: `bridgeTokenToL1(recipientX, 1e18)` from `callerA` with `gasPrice = G`
3. Both transactions share `block.timestamp`, `block.number`, `msg.sender = callerA`, `recipient = recipientX`, `amount = 1e18`, `tx.gasprice = G`, `bridgeReceiver = address(this)`.
4. Both compute the identical `uid`.
5. Tx A succeeds; Tx B reverts at `sonicBridge.withdraw(uid, ...)` (Scenario A) or silently corrupts Tx A's withdrawal record (Scenario B).
6. In Scenario A, `callerA`'s second bridge operation is lost with no L1 receipt. In Scenario B, `callerA`'s first bridge operation is permanently lost. [1](#0-0) [2](#0-1)
