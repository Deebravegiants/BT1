### Title
Missing Slippage Protection in L2 Pool `deposit()` Functions - (File: contracts/pools/RSETHPoolV3.sol)

### Summary
Multiple L2 pool `deposit()` functions accept no minimum rsETH output parameter, leaving depositors exposed to oracle rate movements between transaction submission and on-chain execution. The L1 `LRTDepositPool` correctly implements this guard; the L2 pools do not.

### Finding Description

Every L2 pool `deposit()` function computes the rsETH amount to mint at execution time using the live oracle rate, with no floor the caller can enforce:

`RSETHPoolV3.sol` – ETH path: [1](#0-0) 

`RSETHPoolV3.sol` – token path: [2](#0-1) 

The rate used in both paths is fetched from the oracle at execution time: [3](#0-2) 

The same pattern is present in every other L2 pool variant: [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) [9](#0-8) 

By contrast, the L1 `LRTDepositPool` enforces a caller-supplied minimum on every deposit path: [10](#0-9) 

The oracle used by the pools is either a Chainlink wrapper or an admin-set interim oracle: [11](#0-10) [12](#0-11) 

Chainlink prices update on heartbeat or deviation thresholds. For the token-deposit path, the formula is:

```
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate
```

If `tokenToETHRate` drops (collateral token price falls) or `rsETHToETHrate` rises (rsETH accrues yield) between the user's preview call and execution, `rsETHAmount` is lower than the user observed — with no on-chain mechanism to reject the transaction.

### Impact Explanation

A depositor previews the swap via `viewSwapRsETHAmountAndFee`, decides to proceed, and submits the transaction. By the time the transaction executes, the oracle rate has moved and the user receives fewer rsETH tokens than anticipated. The user does not lose ETH value in absolute terms (rsETH is worth more per token), but the contract fails to deliver the token quantity the user acted upon. This maps to:

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

For the token-deposit path with a more volatile collateral oracle, the shortfall can be larger.

### Likelihood Explanation

Chainlink oracles update on a heartbeat (e.g., every 24 h) or on a 0.5 %–1 % deviation threshold. On L2 chains with mempool congestion or during periods of rapid LST price movement, a pending deposit transaction can easily straddle an oracle update. The user has no recourse because no minimum-output guard exists. This is a routine, unprivileged scenario requiring no attacker — only normal oracle operation.

### Recommendation

Add a `minRsETHAmountExpected` parameter to every `deposit()` overload in all L2 pool contracts, mirroring the pattern already used in `LRTDepositPool`:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountExpected)
    external payable nonReentrant whenNotPaused ...
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    ...
}
```

Apply the same guard to the token-deposit overloads.

### Proof of Concept

1. User calls `viewSwapRsETHAmountAndFee(1 ether)` on `RSETHPoolV3` and observes `rsETHAmount = 0.97 wrsETH`.
2. User submits `deposit{value: 1 ether}("ref")`.
3. Before the transaction is mined, the Chainlink oracle for the collateral token ticks down 0.8 % (within normal deviation), or the rsETH oracle ticks up on its heartbeat.
4. `viewSwapRsETHAmountAndFee` now returns `0.962 wrsETH`.
5. The contract mints `0.962 wrsETH` to the user — 0.8 % less than expected — with no revert and no recourse.
6. The same scenario applies to the token-deposit path where both `tokenToETHRate` and `rsETHToETHrate` can move independently, compounding the shortfall.

        (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
            AggregatorV3Interface(oracle).latestRoundData();

        if (answeredInRound < roundID) revert StalePrice();
        if (timestamp == 0) revert IncompleteRound();
        if (ethPrice <= 0) revert InvalidPrice();
