### Title
Pool Admin Can Front-Run Swaps by Instantly Increasing Per-Bin Fees Without Timelock or Cap — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolBinAdditionalFees()` lets the pool admin set per-bin fees (`addFeeBuyE6` / `addFeeSellE6`) to any `uint16` value with no timelock and no cap. A pool admin who spots a pending `swap()` in the mempool can front-run it by spiking the fee on the active bin, forcing the swapper to pay more input tokens than expected. The `priceLimitX64` parameter in `swap()` constrains only the marginal price, not the fee, so it provides no protection against this attack.

---

### Finding Description

**Uncapped, instant per-bin fee mutation**

`setPoolAdminFees()` enforces caps (`maxAdminSpreadFeeE6`, `maxAdminNotionalFeeE8`) before updating the global spread fee: [1](#0-0) 

`setPoolBinAdditionalFees()` has **no equivalent cap check**: [2](#0-1) 

The pool-level handler also performs no cap validation: [3](#0-2) 

`addFeeBuyE6` and `addFeeSellE6` are `uint16`, so the pool admin can set them to `65535` (≈ 6.55 % in E6 units) in a single transaction with no delay.

**Fee is read from storage at swap time**

During `swap()`, the pool reads the live oracle bid/ask and computes `baseFeeX64`, but the per-bin additional fee is loaded from `_binStates` at the moment each bin is processed: [4](#0-3) 

The combined fee is then applied to the gross input: [5](#0-4) 

A fee increase committed in a front-running transaction is therefore fully reflected in the victim's swap.

**`priceLimitX64` does not protect against fee manipulation**

`priceLimitX64` is compared against the interpolated bin-position price to stop the swap from crossing a price boundary: [6](#0-5) 

It has no relationship to the fee multiplier. A swapper can set a tight price limit and still be charged an arbitrarily higher fee on the same output quantity.

---

### Impact Explanation

For an exact-output swap (token1 → token0), the swapper's required token1 input is:

```
amountIn = amountOut × avgPrice × (1 + addFeeBuyE6/1e6 + baseFeeX64/ONE_X64)
```

If the pool admin raises `addFeeBuyE6` from 0 to 65535 immediately before the swap executes, the swapper pays up to ~6.55 % more token1 than they observed when constructing the transaction. The excess fee accrues to LPs in the bin (and to the protocol via `spreadFeeE6`). If the pool admin is also an LP, they capture the excess directly. Even if they are not, the swapper suffers a direct, quantifiable loss of principal with no on-chain recourse.

---

### Likelihood Explanation

The pool admin role is a semi-trusted, per-pool operator (analogous to `MINT_ADMIN_ROLE` in the ERC721Sale report). The attack requires:

1. A pool with a mutable admin (not immutable oracle, not factory owner).
2. The admin monitoring the mempool — trivial on any public EVM chain.
3. A single `setPoolBinAdditionalFees` call to spike the fee, followed by a reset after the victim's transaction confirms.

No timelock, no cap, and no on-chain slippage guard on the fee dimension stand in the way.

---

### Recommendation

1. **Add a cap on per-bin fees** inside `setPoolBinAdditionalFees()` (e.g., bounded by `maxAdminSpreadFeeE6` or a dedicated `maxBinAdditionalFeeE6` constant), mirroring the cap already enforced for global admin fees.
2. **Add a timelock** for per-bin fee increases (a short delay is sufficient to eliminate mempool front-running).
3. Alternatively, expose a `maxTotalFeeE6` parameter on `swap()` so callers can bound the combined fee they accept, analogous to the `maxTotal` fix applied to `ERC721Sale.mint()`.

---

### Proof of Concept

```
Setup:
  - Pool deployed with addFeeBuyE6 = 0 on bin 0 (active bin).
  - Alice quotes a swap: 1000 token1 in → 990 token0 out (fee ≈ 1%).

Attack:
  1. Alice submits swap(recipient, false, -990e18, 0, ...) to the mempool.
  2. Bob (pool admin) sees Alice's tx.
  3. Bob front-runs with:
       factory.setPoolBinAdditionalFees(pool, 0, 65535, 0)
     This sets addFeeBuyE6 = 65535 (≈ 6.55% extra fee on bin 0).
  4. Alice's swap executes. The fee applied is now baseFeeX64 + 6.55%.
     Alice pays ≈ 1066 token1 instead of 1000 — a ~6.6% overcharge.
  5. Bob resets: factory.setPoolBinAdditionalFees(pool, 0, 0, 0).

Result:
  Alice loses ~66 token1 (6.6% of principal) with no on-chain protection.
  The excess accrues to LPs in bin 0; if Bob is an LP, he captures it directly.
```
