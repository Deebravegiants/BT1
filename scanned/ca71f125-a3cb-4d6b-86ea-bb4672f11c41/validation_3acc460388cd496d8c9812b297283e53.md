### Title
`LiquidityLib.addLiquidity` reverts with `toUint104()` overflow when bin's `token0BalanceScaled` is at or near `type(uint104).max` — (`metric-core/contracts/libraries/LiquidityLib.sol`)

### Summary

`LiquidityLib.addLiquidity` computes the proportional token0 deposit for an existing bin and immediately casts the new running total to `uint104` via OpenZeppelin `SafeCast.toUint104()`. There is no guard ensuring the sum fits in 104 bits. When `token0BalanceScaled` is at or near `type(uint104).max`, any call with `sharesToAdd > 0` will revert, permanently bricking `addLiquidity` for that bin.

### Finding Description

In the `binTotalSharesVal > 0` branch of `addLiquidity`:

```solidity
// LiquidityLib.sol line 109
amount0Scaled = Math.ceilDiv(_checkedMul(binState.token0BalanceScaled, sharesToAdd), binTotalSharesVal);
// line 114
binState.token0BalanceScaled = (uint256(binState.token0BalanceScaled) + amount0Scaled).toUint104();
``` [1](#0-0) 

The entire function body is wrapped in `unchecked {}`: [2](#0-1) 

However, `unchecked` in Solidity 0.8+ only suppresses native arithmetic overflow checks — it does **not** suppress reverts from external/internal function calls. `SafeCast.toUint104()` is a function call and will revert unconditionally if its argument exceeds `type(uint104).max`.

The overflow condition:
- Let `token0BalanceScaled = type(uint104).max - 1`, `sharesToAdd = 1`, `binTotalSharesVal = 1`
- `amount0Scaled = ceilDiv((type(uint104).max - 1) * 1, 1) = type(uint104).max - 1`
- Sum = `(type(uint104).max - 1) + (type(uint104).max - 1) = 2 * type(uint104).max - 2 > type(uint104).max` → **revert**

The same pattern applies to `token1BalanceScaled` at line 118: [3](#0-2) 

**How the near-max state is reached**: The first LP (when `binTotalSharesVal == 0`) can set `token0BalanceScaled` to any value up to `type(uint104).max` by choosing `sharesToAdd` appropriately (line 89): [4](#0-3) 

If `initialScaledToken0PerShareE18 = 1e18`, then `amount0Scaled = sharesToAdd`. The first LP can set `token0BalanceScaled = type(uint104).max` exactly. After that, every subsequent `addLiquidity` call for that bin reverts because `amount0Scaled >= 1` and the sum exceeds `type(uint104).max`.

The swap path in `SwapMath.sol` also uses `toUint104()` when accumulating token0 into a bin: [5](#0-4) 

This means swaps would also revert before pushing the balance above max, but a swap that lands exactly at `type(uint104).max` succeeds and leaves the bin in the DoS state.

`BinState.token0BalanceScaled` is typed as `uint104`: [6](#0-5) 

### Impact Explanation

`addLiquidity` is permanently broken for any bin whose `token0BalanceScaled` reaches `type(uint104).max`. No new LP can add liquidity to that bin. Existing LPs can still call `removeLiquidity` (which uses floor division and decreases the balance), but the bin is closed to new deposits. This is broken core pool liquidity functionality meeting the Sherlock medium threshold.

### Likelihood Explanation

Low-to-medium. The near-max state requires either:
1. A deliberate griefing first LP who deposits `type(uint104).max` scaled units of token0 (economically costly but possible for low-decimal or high-supply tokens), or
2. A swap that lands the balance at exactly `type(uint104).max`.

For tokens with small `token0ScaleMultiplier` (e.g., a token with 6 decimals where the multiplier is 1), the scaled balance equals the native balance, making the threshold reachable with far fewer tokens.

### Recommendation

Before writing the updated balance, cap or validate the sum:

```solidity
uint256 newBalance0 = uint256(binState.token0BalanceScaled) + amount0Scaled;
if (newBalance0 > type(uint104).max) revert BinBalanceOverflow();
binState.token0BalanceScaled = uint104(newBalance0);
```

Apply the same fix to `token1BalanceScaled` at line 118 and to the analogous `toUint104()` cast in `SwapMath.buyToken1InBinSpecifiedOut` (line 500-501) and `buyToken0InBinSpecifiedIn`.
