### Title
Double-Counting of Notional Fee Surplus Causes `collectFees` to Over-Allocate, Draining LP Funds — (`metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

In `MetricOmmPool.collectFees`, the `surplus0Scaled` / `surplus1Scaled` values used to compute spread-fee payouts already include the accumulated `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` balances. Because the spread-fee split is computed as a full proportional share of that combined surplus, and the notional fees are then paid out *again* on top, the total attempted payout exceeds the actual surplus. This is confirmed by the protocol's own test, which asserts `total0Attempted > surplus0Scaled`.

---

### Finding Description

`collectFees` computes the surplus as:

```
surplus0Scaled = balance0() * token0ScaleMultiplier - totalScaledToken0InBins
``` [1](#0-0) 

This surplus is the pool's entire non-bin balance, which is composed of two distinct components:

1. **Spread-fee accumulation** — token inflows from the bid/ask spread that accrued to the pool during swaps.
2. **Notional-fee accumulation** — `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled`, which are separately tracked accumulators incremented during every swap. [2](#0-1) 

The spread-fee split is then computed as a full proportional share of the *combined* surplus:

```solidity
uint256 spreadFee0ToAdminScaled    = (surplus0Scaled * adminSpreadFeeE6_)    / spreadSumE6;
uint256 spreadFee0ToProtocolScaled = (surplus0Scaled * protocolSpreadFeeE6_) / spreadSumE6;
``` [3](#0-2) 

Because `adminSpreadFeeE6_ + protocolSpreadFeeE6_ = spreadSumE6`, the two spread-fee lines together consume essentially the *entire* `surplus0Scaled` (minus at most 1 wei of floor rounding). The notional fees are then computed from `notionalFee0AmountScaled` and added on top:

```solidity
uint256 notionalFee0ToProtocolScaled = notionalFee0AmountScaled - notionalFee0ToAdminScaled;
...
uint256 totalFee0ToProtocolScaled = spreadFee0ToProtocolScaled + notionalFee0ToProtocolScaled;
``` [4](#0-3) 

The total attempted payout is therefore:

```
≈ surplus0Scaled  (spread fees)
+ notionalFee0AmountScaled  (notional fees, already counted inside surplus0Scaled)
> surplus0Scaled
```

The protocol's own test explicitly verifies this invariant is broken:

```solidity
assertGt(total0Attempted, surplus0Scaled, "token0 attempted payout exceeds computed surplus");
assertGt(total1Attempted, surplus1Scaled, "token1 attempted payout exceeds computed surplus");
``` [5](#0-4) 

The transfers execute sequentially:

```solidity
transferToken0(adminFeeDestination_, totalFee0ToAdmin);
...
transferToken0(FACTORY, totalFee0ToProtocol);
``` [6](#0-5) 

If the pool's non-bin balance is exactly the surplus (the normal case), the second or third transfer will either revert (DoS of fee collection) or, if the pool holds extra balance from pending LP deposits, will drain LP principal.

---

### Impact Explanation

**Pool insolvency / LP fund drain.** When both `spreadFeeE6 > 0` and `notionalFeeE8 > 0` are configured (the intended production configuration shown in the test setup), every call to `collectFees` attempts to transfer `surplus + notionalAccumulator` tokens out of the pool. The notional accumulator is double-counted: once inside `surplus0Scaled` (which drives the spread-fee computation) and once as an explicit addend. The excess must come from LP-owned bin balances, making the pool insolvent with respect to LP withdrawal claims.

---

### Likelihood Explanation

**High.** The condition requires only that both fee types are non-zero simultaneously, which is the standard production configuration. Any privileged caller (factory owner) invoking `collectFees` triggers the bug. No special market conditions or adversarial inputs are required — the miscalculation is structural and fires on every fee collection when both fee types are active.

---

### Recommendation

Compute the spread-fee base as only the *spread-specific* surplus, excluding the notional accumulator:

```solidity
uint256 spreadSurplus0Scaled = surplus0Scaled - notionalFee0AmountScaled;
uint256 spreadSurplus1Scaled = surplus1Scaled - notionalFee1AmountScaled;

uint256 spreadFee0ToAdminScaled    = spreadSumE6 == 0 ? 0 : (spreadSurplus0Scaled * adminSpreadFeeE6_)    / spreadSumE6;
uint256 spreadFee0ToProtocolScaled = spreadSumE6 == 0 ? 0 : (spreadSurplus0Scaled * protocolSpreadFeeE6_) / spreadSumE6;
```

This mirrors the fix pattern from the external report: use the *remaining* balance for the final component rather than recomputing from the full total.

---

### Proof of Concept

The protocol's own test at `metric-core/test/MetricOmmPool.notionalFee.t.sol:211` reproduces the condition:

1. Configure pool with both `spreadFeeE6 > 0` and `notionalFeeE8 > 0`.
2. Execute several swaps to accumulate both spread surplus and notional fee balances.
3. Compute `surplus0Scaled = balance0 * multiplier - totalScaledToken0InBins`.
4. Mirror the `collectFees` split math.
5. Observe `total0Attempted > surplus0Scaled` — the assertion at line 264 passes, confirming the over-allocation. [7](#0-6) 

In a live pool, the sequential `safeTransfer` calls will either revert (bricking fee collection permanently until the pool receives extra tokens) or, if the pool holds any excess balance, will transfer LP-owned tokens to the fee recipients, directly reducing the token coverage for LP withdrawal claims.
