### Title
SwapAllowlistExtension gates router identity instead of actual swapper, enabling allowlist bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is the `msg.sender` of the pool's `swap` call. When `MetricOmmSimpleRouter` intermediates a swap, `sender` is the router's address, not the actual user. A pool admin who adds the router to the allowlist (the natural action to enable router-mediated swaps) inadvertently grants every user the ability to bypass the per-user swap restriction.

---

### Finding Description

**Root cause — wrong identity checked:**

`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool and checks it against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
``` [1](#0-0) 

The pool passes `msg.sender` of the `swap` call as `sender` to the extension:

```solidity
// MetricOmmPool.swap → ExtensionCalling._beforeSwap
_beforeSwap(msg.sender, recipient, ...);
``` [2](#0-1) [3](#0-2) 

**Router path — actual user identity is lost:**

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly, making itself `msg.sender`:

```solidity
// MetricOmmSimpleRouter.exactInputSingle
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
IMetricOmmPoolActions(params.pool).swap(
    params.recipient, params.zeroForOne, ..., "", params.extensionData
);
``` [4](#0-3) 

The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`. The `extensionData` bytes forwarded by the router are ignored by `SwapAllowlistExtension` (the `bytes calldata` parameter is unnamed and unused).

**Contrast with DepositAllowlistExtension (correct):**

The deposit allowlist correctly gates `owner` (the position beneficiary), not `sender` (the operator/payer), so the operator pattern does not bypass it:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
``` [5](#0-4) 

The swap extension has no equivalent "actual user" field to check.

---

### Impact Explanation

A pool admin deploying `SwapAllowlistExtension` to restrict trading to a curated set of counterparties (e.g., KYC'd addresses, institutional partners) must also add `MetricOmmSimpleRouter` to the allowlist for any user who wants to use the router. Once the router is allowlisted, the restriction is completely nullified: any address can call `exactInputSingle` / `exactInput` / `exactOutput` through the router and the extension will pass. Unauthorized users can drain LP positions at oracle-quoted prices, causing direct loss of LP principal.

---

### Likelihood Explanation

The scenario requires the pool admin to add the router to the allowlist, which is the expected operational action for any pool that supports the periphery router. The admin has no way to simultaneously allow legitimate router users and block unauthorized ones — the two goals are mutually exclusive given the current extension design. Any pool that uses both `SwapAllowlistExtension` and `MetricOmmSimpleRouter` is affected.

---

### Recommendation

The `SwapAllowlistExtension` must check the actual end-user identity, not the intermediary. Two viable approaches:

1. **Extension-data forwarding**: The router encodes `msg.sender` into `extensionData`; the extension decodes and verifies it. The extension must also verify that `sender` is a trusted router before trusting the encoded identity (to prevent spoofing by arbitrary callers).

2. **Separate allowlist entry for routers with user-level enforcement**: Require the router to forward the actual user address as a verifiable field, and have the extension enforce it.

The `DepositAllowlistExtension` pattern (checking `owner`, not `sender`) should be the model: identify the economically relevant actor and gate on that.

---

### Proof of Concept

```
1. Pool admin deploys pool with SwapAllowlistExtension configured.
2. Pool admin calls setAllowedToSwap(pool, router, true)
   — intended to let legitimate users swap via the router.
3. Unauthorized user (not on the allowlist) calls:
       router.exactInputSingle({pool: pool, recipient: attacker, ...})
4. Router calls pool.swap(...) with msg.sender = router.
5. Extension checks allowedSwapper[pool][router] == true → passes.
6. Swap executes; attacker receives output tokens.
   Pool admin's per-user restriction is completely bypassed.
``` [1](#0-0) [6](#0-5) [3](#0-2)
