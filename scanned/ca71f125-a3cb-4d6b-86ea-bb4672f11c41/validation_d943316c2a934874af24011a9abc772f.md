### Title
`collectFees` Drains LP Principal When Both Spread and Notional Fees Are Active — (`metric-core/contracts/MetricOmmPool.sol`)

### Summary

When both spread fees and notional fees are enabled, `collectFees` attempts to pay out more tokens than the actual fee surplus, taking the shortfall directly from LP-owned bin balances and leaving the pool insolvent.

### Finding Description

**Root cause — exact-in swap notional fee accounting**

In `_executeSwap`, for an exact-in swap the notional fee is applied by reducing the output amount sent to the trader:

```solidity
// exact-in zeroForOne: notional on token1 output
uint256 notionalFeeScaled = uint256(-amount1DeltaScaled) * notionalFeeE8 / 1e8;
if (notionalFeeScaled > 0) {
    amount1DeltaScaled = amount1DeltaScaled + int256(notionalFeeScaled); // less negative
    notionalFeeToken1Scaled = (uint256(notionalFeeToken1Scaled) + notionalFeeScaled).toUint128();
}
``` [1](#0-0) 

The pool then updates bin totals by the same reduced delta:

```solidity
binTotals.scaledToken1 = uint128(uint256(binTotals.scaledToken1) - uint256(-amount1DeltaScaled));
``` [2](#0-1) 

Because both `pool.balance1` and `binTotals.scaledToken1` decrease by the same `|amount1DeltaScaled_final|`, the **surplus** (`pool.balance - binTotals`) is unchanged. The notional fee sits inside the bin totals (LP principal), not in the surplus.

**Incorrect payout in `collectFees`**

`collectFees` computes two independent payouts and sums them:

```solidity
uint256 totalFee0ToProtocolScaled = spreadFee0ToProtocolScaled + notionalFee0ToProtocolScaled;
// ...
(uint256 totalFee0ToProtocol, ...) = deltasScaledToExternal(..., Math.Rounding.Floor);
transferToken0(FACTORY, totalFee0ToProtocol);
``` [3](#0-2) 

- `spreadFee0ToProtocolScaled` is drawn from the surplus (correct).
- `notionalFee0ToProtocolScaled` is drawn from `notionalFeeToken0Scaled`, which for exact-in swaps is **not** in the surplus — it is still inside `binTotals`.

The combined transfer therefore exceeds the surplus by exactly the accumulated exact-in notional fee, pulling that shortfall out of LP-owned bin balances.

**Test confirmation**

The test suite explicitly documents this overallocation:

```solidity
assertGt(total0Attempted, surplus0Scaled, "token0 attempted payout exceeds computed surplus");
assertGt(total1Attempted, surplus1Scaled, "token1 attempted payout exceeds computed surplus");
``` [4](#0-3) 

### Impact Explanation

After `collectFees` executes:

- `pool.balance < binTotals` — the pool is insolvent.
- LPs who call `removeLiquidity` after the collection will receive fewer tokens than their shares entitle them to, or the last LP's withdrawal will revert entirely.
- The shortfall equals the sum of all exact-in notional fees collected since the last `collectFees` call, which grows with swap volume.

This matches the **Pool insolvency** allowed impact: balances fail to cover LP claims.

### Likelihood Explanation

The condition is met whenever:
1. A pool is deployed with `notionalFeeE8 > 0` (non-zero notional fee), and
2. At least one exact-in swap has occurred (positive `amountSpecified`), and
3. `collectFees` is called by the factory.

All three are routine, expected operations. No adversarial setup is required. The factory admin calls `collectFees` in good faith; the math is simply wrong.

### Recommendation

For exact-in swaps, the notional fee must be routed so it lands in the surplus (not in the bin totals). Two equivalent fixes:

**Option A — Exclude notional fee from bin-total reduction.** After computing the notional fee, do not reduce `binTotals` by it; only reduce by the net amount actually sent to the trader. The notional fee then sits in the pool balance but not in bin totals, placing it in the surplus where `collectFees` can correctly claim it.

**Option B — Track exact-in notional fees separately and cap `collectFees` payout.** Keep a separate accumulator for exact-in notional fees (which are in bin totals) and exact-out notional fees (which are in surplus). Only pay out the exact-out portion through `collectFees`; the exact-in portion must be settled differently (e.g., by reducing bin balances proportionally at collection time).

### Proof of Concept

1. Deploy a pool with `notionalFeeE8 = 1_000_000` (1%) and `spreadFeeE6 > 0`.
2. Add liquidity across several bins.
3. Execute 8 exact-in swaps in both directions (as in `test_collectProtocolFees_math_overallocates_whenSpreadAndNotionalBothActive`).
4. Read `surplus0Scaled = pool.balance0() * token0ScaleMultiplier - binTotals.scaledToken0`.
5. Read `notional0 = notionalFeeToken0Scaled`.
6. Observe `notional0 > surplus0Scaled` — the notional accumulator exceeds the surplus.
7. Call `collectFees`; the pool transfers `spread_fee + notional_fee` of token0.
8. Verify `pool.balance0() * token0ScaleMultiplier < binTotals.scaledToken0` — pool is now insolvent.
9. Any LP calling `removeLiquidity` for their full share will receive less than owed, or the call reverts. [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7)
