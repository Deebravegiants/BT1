## Analysis

### Verifying the Claimed Code Behavior

**`setMerkleRoot` overwrites the single root slot:** [1](#0-0) 

Every call to `setMerkleRoot` unconditionally overwrites `currentMerkleRoot` and increments both `currentMerkleRootIndex` and `currentIndex`. No historical roots are stored anywhere.

**`_processClaim` index check and proof verification:** [2](#0-1) 

The guard `index > currentIndex` only checks that the index is within the total count of roots ever published. The Merkle proof is then verified exclusively against `currentMerkleRoot` — the single live root.

### Tracing the Exploit Path

| Step | State |
|---|---|
| Owner calls `setMerkleRoot(root_N)` | `currentMerkleRoot = root_N`, `currentIndex = 1` |
| Alice (index=1) delays claiming | — |
| Owner calls `setMerkleRoot(root_N1)` (Alice absent) | `currentMerkleRoot = root_N1`, `currentIndex = 2` |
| Alice calls `claim(1, alice, amount, proof_N)` | `index=1 ≤ currentIndex=2` passes; `isClaimed` passes; proof fails → `InvalidMerkleProof` |
| Alice calls `claim(2, alice, amount, proof_N1)` | Alice has no leaf in tree N+1 → `InvalidMerkleProof` |

Alice is permanently locked out. The tokens remain in the contract (no principal loss), but her promised entitlement is unclaimable.

### Is This Rejected by the Criteria?

The "Reject Immediately" list includes scenarios requiring **admin compromise or malicious maintainer**. This scenario does not require malice — it can occur through:
- A routine operational mistake (accidentally omitting a user when building the new tree)
- A design choice where round N+1 only lists new recipients rather than carrying forward all cumulative entitlements

The contract provides **no on-chain enforcement** that a new root must include all prior recipients, and **no mechanism** for users to claim against a superseded root. The vulnerability is in the contract design, not in admin behavior.

The cumulative-amount design at line 326 (`claimableAmount = cumulativeAmount - userClaims[account].cumulativeAmount`) implies the *intent* is that each new root carries forward all prior entitlements — but this invariant is entirely off-chain and unenforceable. [3](#0-2)

### Title
Single-slot Merkle root overwrite permanently blocks claims from superseded rounds — (`contracts/KERNEL/KernelMerkleDistributor.sol`)

### Summary
`setMerkleRoot` overwrites the single `currentMerkleRoot` storage slot with no historical root retention. A user included in round N who has not yet claimed is permanently denied their entitlement if round N+1 omits their address, because the proof for round N cannot verify against root N+1 and the user has no leaf in tree N+1.

### Finding Description
`currentMerkleRoot` is a single `bytes32` storage variable. [4](#0-3) 

Every `setMerkleRoot` call overwrites it unconditionally. [5](#0-4) 

`_processClaim` verifies the proof only against this single live root. [6](#0-5) 

The index guard `index > currentIndex` does not help: it allows any index up to the total count of roots ever published, but the proof still must match the *current* root, not the root that was active when that index was assigned. [7](#0-6) 

### Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

The KERNEL tokens remain locked in the distributor contract. The affected user's entitlement is permanently unclaimable with no recovery path. No principal is lost from the protocol's perspective, but the user's promised reward is irrecoverable.

### Likelihood Explanation
Moderate. The cumulative-amount design strongly implies the intent that each new root carries forward all prior entitlements. However, this invariant is entirely off-chain. A single tree-building mistake (omitting one address) or a deliberate design choice to publish incremental-only trees permanently forfeits affected users' rewards. No malicious intent is required.

### Recommendation
Store historical roots in a mapping keyed by `currentMerkleRootIndex`:

```solidity
mapping(uint256 => bytes32) public merkleRoots;
```

Allow `claim` to accept a `rootIndex` parameter and verify the proof against `merkleRoots[rootIndex]`, while still enforcing that each `(account, rootIndex)` pair can only be claimed once. Alternatively, enforce on-chain that `currentIndex` is monotonically included in every new tree (i.e., require the new tree to contain every address from the previous tree with at least the same cumulative amount).
