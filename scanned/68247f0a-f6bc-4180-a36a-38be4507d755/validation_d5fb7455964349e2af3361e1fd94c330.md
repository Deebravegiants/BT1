### Title
Token-Specific Fee Defaults to Zero Due to Missing Fallback Logic - (File: `contracts/pools/RSETHPool.sol`)

### Summary
`RSETHPool.viewSwapRsETHAmountAndFee(uint256, address)` reads `tokenFeeBps[token]` directly without falling back to the global `feeBps` when the per-token fee is uninitialized. Because `addSupportedToken()` never initializes `tokenFeeBps[token]`, every token deposit pays zero fee until the admin explicitly calls `setTokenFeeBps()`. The code cannot distinguish between "token fee not configured" and "token fee explicitly set to zero."

### Finding Description
`RSETHPool` maintains two fee-rate storage slots:

- `feeBps` — the global fee rate used for ETH deposits
- `tokenFeeBps[token]` — a per-token fee rate, defaulting to `0` for any newly added token

The ETH deposit path correctly uses the global rate:

```solidity
// viewSwapRsETHAmountAndFee(uint256 amount) — line 312
fee = amount * feeBps / 10_000;
```

The token deposit path reads `tokenFeeBps[token]` directly with no fallback:

```solidity
// viewSwapRsETHAmountAndFee(uint256 amount, address token) — lines 335-336
uint256 feeBpsForToken = tokenFeeBps[token];
fee = amount * feeBpsForToken / 10_000;
```

`addSupportedToken()` registers a token's oracle and bridge but never initializes `tokenFeeBps[token]`:

```solidity
// lines 651-653
supportedTokenList.push(token);
supportedTokenOracle[token] = oracle;
tokenBridge[token] = bridge;
// tokenFeeBps[token] is never set → remains 0
```

`setTokenFeeBps()` exists to configure the rate post-hoc, but there is no enforcement that it must be called before deposits are accepted. The storage pattern is identical to the reported bug class: `tokenFeeBps[token] == 0` is indistinguishable from "not configured" vs. "explicitly zero fee."

### Impact Explanation
**Low — Contract fails to deliver promised returns.**

All token deposits (e.g., wstETH) made after `addSupportedToken()` but before `setTokenFeeBps()` pay zero fee, regardless of the non-zero global `feeBps`. The protocol collects no fee revenue on those deposits. No user funds are at risk, but the protocol's fee mechanism silently fails for every token whose per-token rate has not been explicitly set.

### Likelihood Explanation
**High.** `addSupportedToken()` and `setTokenFeeBps()` are separate, independent admin calls with no on-chain coupling. Any window between token addition and fee configuration — including the initial deployment of a new token — results in fee-free deposits. A depositor does not need to do anything special; calling `deposit(token, amount, referralId)` during this window is sufficient.

### Recommendation
Apply the same fix recommended in the reference report: use a sentinel pattern to distinguish "not set" from "explicitly zero." Two concrete options:

1. **Fallback to global `feeBps`**: if `tokenFeeBps[token] == 0`, use `feeBps` as the default, and introduce a separate boolean `tokenFeeBpsSet[token]` to allow an explicit zero override.
2. **Initialize on registration**: require a fee rate parameter in `addSupportedToken()` and set `tokenFeeBps[token]` atomically, preventing any window of zero-fee deposits.

### Proof of Concept
1. Admin deploys `RSETHPool` with `feeBps = 30` (0.3%).
2. Admin calls `addSupportedToken(wstETH, oracle, bridge)`. `tokenFeeBps[wstETH]` is `0` (uninitialized).
3. Before admin calls `setTokenFeeBps(wstETH, 30)`, a depositor calls `deposit(wstETH, 1e18, "ref")`.
4. `viewSwapRsETHAmountAndFee(1e18, wstETH)` executes: `feeBpsForToken = tokenFeeBps[wstETH] = 0`, so `fee = 1e18 * 0 / 10_000 = 0`.
5. The depositor receives the full rsETH amount with zero fee deducted. The protocol earns nothing on this deposit. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
