### Title
Daily Mint Limit Boundary Straddling Allows 2× Daily Cap Bypass — (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPoolV3WithNativeChainBridge.sol, contracts/pools/RSETHPoolV2.sol, contracts/pools/RSETHPoolV2ExternalBridge.sol)

---

### Summary

The L2 pool contracts enforce a `dailyMintLimit` on wrsETH minting as a safety cap. Because the day boundary is computed from a fixed `startTimestamp` and resets at a predictable on-chain timestamp, an unprivileged depositor can straddle the reset: consume the remaining capacity in the last block of day N, then immediately consume the full new capacity in the first block of day N+1. This allows a single actor to mint up to **2× the intended daily cap** in a matter of seconds, temporarily freezing all other depositors out of the pool for the remainder of the new day.

---

### Finding Description

Every L2 pool contract enforces a daily minting cap through the `limitDailyMint` modifier. The day index is computed as:

```solidity
// RSETHPoolV3.sol, line 340
function getCurrentDay() public view returns (uint256) {
    return (block.timestamp - startTimestamp) / 1 days;
}
```

The modifier resets `dailyMintAmount` to zero whenever `currentDay > lastMintDay`:

```solidity
// RSETHPoolV3.sol, lines 110–123
uint256 currentDay = getCurrentDay();
if (currentDay > lastMintDay) {
    lastMintDay = currentDay;
    dailyMintAmount = 0;
}
if (dailyMintAmount + rsETHAmount > dailyMintLimit) {
    revert DailyMintLimitExceeded();
}
dailyMintAmount += rsETHAmount;
```

The reset timestamp is fully deterministic and publicly readable via `getNextDailyLimitResetTimestamp()`:

```solidity
// RSETHPoolV3.sol, line 354–356
function getNextDailyLimitResetTimestamp() external view returns (uint256) {
    return startTimestamp + (getCurrentDay() + 1) * 1 days;
}
```

Because all deposit activity is public on-chain, an attacker can observe exactly how much of the daily limit remains at any moment. There is no incentive for legitimate users to deposit early — doing so only reveals the remaining capacity to a well-funded attacker. The attacker can:

1. Monitor `remainingDailyMintLimit()` throughout the day.
2. In the last block before `getNextDailyLimitResetTimestamp()`, submit a deposit that consumes the entire remaining capacity (`dailyMintLimit - dailyMintAmount`), blocking all other depositors for the rest of day N.
3. In the very next block (day N+1), submit another deposit consuming the full `dailyMintLimit` of the new day.

The same pattern exists identically in `RSETHPoolV3ExternalBridge` (lines 144–157), `RSETHPoolV3WithNativeChainBridge` (lines 122–135), `RSETHPoolV2` (lines 79–93), and `RSETHPoolV2ExternalBridge` (lines 111–124).

---

### Impact Explanation

**Medium — Temporary freezing of funds.**

After the attacker executes the two-block straddle, `dailyMintAmount` equals `dailyMintLimit` for the new day. Every subsequent `deposit()` call by any other user reverts with `DailyMintLimitExceeded` until the next day boundary. Users who sent ETH or LSTs expecting to receive wrsETH are denied service for up to 24 hours. The daily safety cap — intended to limit protocol exposure to oracle errors or other anomalies — is effectively doubled in a single short window, undermining the protection it was designed to provide.

---

### Likelihood Explanation

**Medium.** The attack requires no special role or privilege — only sufficient ETH/LST to fill the remaining daily capacity. The reset timestamp is deterministic and publicly readable. Any MEV bot or well-capitalized actor can automate the two-transaction sequence. The only cost is the capital locked in wrsETH (which is redeemable), making the attack economically rational for anyone who wants to deny competitors access to the pool or who benefits from being the sole large depositor at the start of a new day.

---

### Recommendation

1. **Short term:** Introduce a per-address sub-limit or a cooldown between large deposits so that a single address cannot consume the entire daily cap in one transaction. Alternatively, reserve a fraction of the daily cap (e.g., 20%) that cannot be consumed in the final hour of each period.
2. **Long term:** Replace the hard daily cap with a rate-limiting mechanism (e.g., a token-bucket or exponential moving average) that smooths consumption over time rather than resetting abruptly, eliminating the exploitable boundary.

---

### Proof of Concept

```
Assume dailyMintLimit = 1000 wrsETH, startTimestamp = T0.

Day N reset occurs at T0 + N * 1 days.

Block at T0 + N * 1 days - 12s (last block of day N):
  - Attacker calls deposit() with enough ETH to mint (1000 - dailyMintAmount) wrsETH.
  - limitDailyMint: currentDay == lastMintDay (still day N), no reset.
  - dailyMintAmount becomes 1000. All other depositors are now blocked for day N.

Block at T0 + N * 1 days (first block of day N+1):
  - Attacker calls deposit() with enough ETH to mint 1000 wrsETH.
  - limitDailyMint: currentDay (N+1) > lastMintDay (N) → reset: dailyMintAmount = 0.
  - dailyMintAmount becomes 1000. All other depositors are now blocked for day N+1.

Net result: attacker minted 2000 wrsETH (2× the daily cap) in ~12 seconds.
All other users are locked out of the pool for the remainder of day N+1.

```text
    /// @notice Gets the current day relative to the start timestamp
    /// @return uint256 The current day relative to the start timestamp
    function getCurrentDay() public view returns (uint256) {
        return (block.timestamp - startTimestamp) / 1 days;
    }

    /// @notice Gets the remaining daily minting limit
    /// @return uint256 The remaining daily minting limit
    function remainingDailyMintLimit() external view returns (uint256) {
        // If we're on a new day but no mint has occurred yet, treat dailyMintAmount as 0
        uint256 effectiveDailyMintAmount = (getCurrentDay() > lastMintDay) ? 0 : dailyMintAmount;

        return dailyMintLimit - effectiveDailyMintAmount;
    }

    /// @notice Gets the next daily mint limit reset timestamp
    /// @return uint256 The next daily mint limit reset timestamp
    function getNextDailyLimitResetTimestamp() external view returns (uint256) {
        return startTimestamp + (getCurrentDay() + 1) * 1 days;
    }
```
