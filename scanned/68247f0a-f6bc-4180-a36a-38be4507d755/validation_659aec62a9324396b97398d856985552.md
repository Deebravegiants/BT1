### Title
Irreversible `removeAllowedToken` With No Recovery Path Permanently Freezes User Funds — (`contracts/agETH/AGETHTokenWrapper.sol`)

---

### Summary

`AGETHTokenWrapper` exposes `removeAllowedToken` callable by `DEFAULT_ADMIN_ROLE` but provides **no `addAllowedToken` counterpart**. Once a token is removed, every user holding wrapper shares backed by that token is permanently locked out of `withdraw`/`withdrawTo`, with no on-chain path to restore access.

---

### Finding Description

`removeAllowedToken` sets `allowedTokens[_asset] = false` and is callable directly by `DEFAULT_ADMIN_ROLE` — with no timelock protection: [1](#0-0) 

`_withdraw` hard-reverts on any token not in `allowedTokens`: [2](#0-1) 

The contract explicitly documents the absence of a re-add path: [3](#0-2) 

There is no `addAllowedToken` function anywhere in `AGETHTokenWrapper`. The only token registration happens at `initialize` time: [4](#0-3) 

**Contrast with the sister contract** `RsETHTokenWrapper`, which has a symmetric `addAllowedToken` (gated by `TIMELOCK_ROLE`) alongside `removeAllowedToken`: [5](#0-4) 

`AGETHTokenWrapper` lacks both the `addAllowedToken` function and the timelock gate on removal.

---

### Impact Explanation

After `removeAllowedToken(altAgETH)` is called:

- All calls to `withdraw(altAgETH, amount)` and `withdrawTo(altAgETH, ...)` revert with `TokenNotAllowed`.
- Users' wrapper shares are non-redeemable; the underlying `altAgETH` balance is stranded in the contract.
- The contract is upgradeable, so a proxy upgrade *could* restore access — but this is an out-of-band, non-atomic recovery that is not guaranteed and introduces additional governance delay and risk.
- Impact: **Medium — temporary (potentially permanent) freezing of all user funds deposited as altAgETH**.

---

### Likelihood Explanation

The `DEFAULT_ADMIN_ROLE` holder may call `removeAllowedToken` for a legitimate operational reason (e.g., the bridged altAgETH token is paused or deprecated) while assuming — by analogy with `RsETHTokenWrapper` — that re-adding is possible. The missing `addAllowedToken` makes the removal irreversible at the contract level. No key compromise is required; a routine admin action suffices.

---

### Recommendation

1. Add an `addAllowedToken` function gated by at least `MANAGER_ROLE` or `DEFAULT_ADMIN_ROLE`, mirroring `RsETHTokenWrapper.addAllowedToken`.
2. Gate `removeAllowedToken` behind a `TIMELOCK_ROLE` (as `RsETHTokenWrapper` does) to give users time to exit before a removal takes effect.
3. Consider emitting a `TokenAdded` event on `initialize` for consistency and off-chain observability.

---

### Proof of Concept

```solidity
// 1. Deploy AGETHTokenWrapper with altAgETH as the allowed token
// 2. User deposits N altAgETH → receives N wrapper shares
// 3. Admin calls removeAllowedToken(altAgETH)
//    → allowedTokens[altAgETH] = false
// 4. User calls withdraw(altAgETH, N)
//    → _withdraw reverts: TokenNotAllowed()
// 5. No addAllowedToken exists → no on-chain recovery path
// 6. User's N wrapper shares are permanently non-redeemable
//    unless a proxy upgrade is performed

assertEq(wrapper.allowedTokens(altAgETH), false);
vm.expectRevert(AGETHTokenWrapper.TokenNotAllowed.selector);
wrapper.withdraw(altAgETH, N);
```

    /// @param _asset The address of the token to remove
    function removeAllowedToken(address _asset) external onlyRole(TIMELOCK_ROLE) {
        if (!allowedTokens[_asset]) revert TokenNotAllowed();

        allowedTokens[_asset] = false;
        emit TokenRemoved(_asset);
    }
