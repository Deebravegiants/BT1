### Title
Token Decimal Mismatch in `viewSwapRsETHAmountAndFee` Causes Severely Incorrect rsETH Minting for Non-18-Decimal Collateral Tokens - (File: contracts/pools/RSETHPoolV3.sol)

---

### Summary

The `viewSwapRsETHAmountAndFee(amount, token)` function used across all L2 pool contracts computes the rsETH/wrsETH output amount without normalizing the deposited token amount to 18 decimals. The formula implicitly assumes the collateral token has 18 decimals. When a non-18-decimal token (e.g., USDC/USDT with 6 decimals) is added as a supported collateral, depositors receive a factor of `10^(18 - tokenDecimals)` fewer rsETH than they are owed, constituting a direct, permanent loss of user funds.

---

### Finding Description

Every L2 pool contract that supports ERC20 token deposits uses the following formula inside `viewSwapRsETHAmountAndFee(uint256 amount, address token)`:

```solidity
// contracts/pools/RSETHPoolV3.sol  lines 324-334
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;

uint256 rsETHToETHrate = getRate();                                        // 1e18-scaled
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();   // 1e18-scaled

rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

The oracle wrapper `ChainlinkOracleForRSETHPoolCollateral.getRate()` always normalises the Chainlink answer to 1e18:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol  line 34
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
```

So `tokenToETHRate` is always expressed as *"ETH per 1 human-readable token"* in 1e18 scale, regardless of the token's on-chain decimal count.

**For an 18-decimal token (wstETH):**
```
amountAfterFee = 1e18   (1 wstETH)
tokenToETHRate = 1.15e18
rsETHToETHrate = 1.05e18
rsETHAmount    = 1e18 * 1.15e18 / 1.05e18 ≈ 1.095e18  ✓
```

**For a 6-decimal token (USDC, 1 USDC ≈ 0.0004 ETH):**
```
amountAfterFee = 1e6    (1 USDC in native units)
tokenToETHRate = 4e14   (0.0004 ETH per 1 USDC, 1e18-scaled)
rsETHToETHrate = 1.05e18
rsETHAmount    = 1e6 * 4e14 / 1.05e18 ≈ 3.8e2   ✗
```

Correct answer: `1 USDC = 0.0004 ETH / 1.05 ≈ 3.81e-4 rsETH = 3.81e14` (18-decimal rsETH units).

The user receives `≈ 3.8e2` instead of `≈ 3.81e14` — a shortfall of **1e12×** (factor of one trillion).

The same structural flaw exists in the reverse-swap view `viewSwapAssetToPremintedRsETH`:

```solidity
// contracts/pools/RSETHPoolV3.sol  line 400
tokenAmount = rsETHAmount * rsETHToETHrate / tokenToETHRate;
```

For a 6-decimal token this returns a value 1e12× too large, causing the operator swap to revert due to insufficient pool balance (DoS on the reverse path).

The identical pattern is present in every pool variant:
- `contracts/pools/RSETHPool.sol` line 346
- `contracts/pools/RSETHPoolNoWrapper.sol` line 311
- `contracts/pools/RSETHPoolV3.sol` line 334
- `contracts/pools/RSETHPoolV3ExternalBridge.sol` line 452
- `contracts/pools/RSETHPoolV3WithNativeChainBridge.sol` line 370
- `contracts/agETH/AGETHPoolV3.sol` line 194

The `addSupportedToken` function enforces no decimal restriction on the token being added:

```solidity
// contracts/pools/RSETHPoolV3.sol  lines 541-554
function addSupportedToken(address token, address oracle) external onlyRole(TIMELOCK_ROLE) {
    UtilLib.checkNonZeroAddress(token);
    UtilLib.checkNonZeroAddress(oracle);
    if (supportedTokenOracle[token] != address(0)) revert AlreadySupportedToken();
    if (IOracle(oracle).getRate() == 0) revert UnsupportedOracle();
    supportedTokenList.push(token);
    supportedTokenOracle[token] = oracle;
    emit AddSupportedToken(token);
}
```

---

### Impact Explanation

**Critical — Direct theft of user funds.**

Any user who calls `deposit(token, amount, referralId)` with a supported non-18-decimal token receives a quantity of rsETH/wrsETH that is `10^(18 - tokenDecimals)` times smaller than the fair value of their deposit. For USDC/USDT (6 decimals) the loss is 99.9999999999% of deposited value. The deposited tokens are held by the pool (or bridged to L1) while the user receives a negligible rsETH balance — a permanent, irrecoverable loss.

---

### Likelihood Explanation

The protocol's `addSupportedToken` path is gated by `TIMELOCK_ROLE` and imposes no decimal check. The pools are explicitly designed to accept arbitrary ERC20 collateral alongside ETH. Stablecoins (USDC, USDT) and other 6-decimal tokens are natural candidates for future collateral expansion on L2s (Arbitrum, Optimism, Unichain) where these pools are deployed. Once such a token is added through normal governance, every subsequent depositor is silently defrauded with no on-chain signal of the error.

---

### Recommendation

Normalise `amountAfterFee` to 18 decimals before applying the rate formula, then scale the result back if needed. For example:

```solidity
import { IERC20Metadata } from "@openzeppelin/contracts/token/ERC20/extensions/IERC20Metadata.sol";

function viewSwapRsETHAmountAndFee(
    uint256 amount,
    address token
) public view onlySupportedToken(token) returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;

    uint256 tokenDecimals = IERC20Metadata(token).decimals();

    uint256 rsETHToETHrate = getRate();
    uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();

    // Normalise amountAfterFee to 18 decimals before rate conversion
    rsETHAmount = amountAfterFee * 10 ** (18 - tokenDecimals) * tokenToETHRate / rsETHToETHrate;
}
```

Apply the symmetric correction in `viewSwapAssetToPremintedRsETH` (divide by `10^(18 - tokenDecimals)` at the end). Alternatively, enforce `IERC20Metadata(token).decimals() == 18` inside `addSupportedToken` if only 18-decimal tokens are intended.

```solidity
// Assume RSETHPoolV3 with USDC (6 decimals) added as supported token
// tokenToETHRate from oracle = 4e14  (0.0004 ETH per 1 USDC, 1e18-scaled)
