### Title
Unrestricted Access to `LidoBridge.bridgeTokenToL1` Bypasses Intended RSETHPool-Only Bridging Flow — (File: `contracts/bridges/LidoBridge.sol`)

---

### Summary

`LidoBridge.bridgeTokenToL1` carries no access control modifier. Any EOA or contract that holds wstETH and has approved `LidoBridge` can invoke it directly, circumventing the `BRIDGER_ROLE`-gated `RSETHPool.bridgeTokens` path entirely.

---

### Finding Description

`LidoBridge.bridgeTokenToL1` is declared `external payable` with only three guards: a non-zero recipient check, a zero-amount check, and a zero-`msg.value` check. [1](#0-0) 

There is no `onlyRole`, `onlyOwner`, or caller-whitelist check of any kind. The function pulls wstETH from `msg.sender` and forwards it to the Lido canonical bridge: [2](#0-1) 

The intended caller is `RSETHPool.bridgeTokens`, which is gated by `BRIDGER_ROLE` and uses `getTokenBalanceMinusFees` to determine the bridgeable amount (pool balance minus accrued fees): [3](#0-2) 

Because `LidoBridge` is a standalone contract with no reference back to `RSETHPool`, a direct caller bypasses all of that logic.

---

### Impact Explanation

The attacker bridges only their **own** wstETH — `safeTransferFrom(msg.sender, ...)` ensures no pool funds are moved. Therefore the protocol does not lose value. However:

- The invariant that all wstETH bridging must flow through `RSETHPool.bridgeTokens` (and thus through `getTokenBalanceMinusFees` accounting) is broken.
- Any party can use `LidoBridge` as a fee-free, permissionless bridge, which is contrary to the contract's documented purpose as a restricted protocol component.
- `feeEarnedInToken` in `RSETHPool` is never updated for these out-of-band bridge calls, so the pool's accounting of bridgeable vs. fee-reserved balances remains correct for pool funds — but the design guarantee is violated.

This maps to **Low — Contract fails to deliver promised returns, but doesn't lose value**. [4](#0-3) 

---

### Likelihood Explanation

Likelihood is **high** — the function is public, requires no special role, and the only precondition is holding wstETH and setting an approval. Any wstETH holder on the deployed L2 can trigger this at will.

---

### Recommendation

Add a caller restriction to `LidoBridge.bridgeTokenToL1`. The simplest approach is to store the authorized caller (the `RSETHPool` address) at construction time and enforce it:

```solidity
address public immutable authorizedCaller;

constructor(address _wstETH, address _lidoBridge, address _authorizedCaller) {
    ...
    authorizedCaller = _authorizedCaller;
}

function bridgeTokenToL1(address recipient, uint256 amount) external payable nonReentrant {
    if (msg.sender != authorizedCaller) revert Unauthorized();
    ...
}
```

Alternatively, inherit `AccessControl` and apply an `onlyRole` modifier consistent with the rest of the protocol.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import "forge-std/Test.sol";
import "contracts/bridges/LidoBridge.sol";

contract LidoBridgeAccessTest is Test {
    LidoBridge bridge;
    address wstETH  = address(0xWSTETH);   // mock ERC-20
    address lidoBridgeMock = address(0xLIDO); // mock IL2ERC20Bridge
    address attacker = address(0xBEEF);
    address attackerL1 = address(0xDEAD);

    function setUp() public {
        bridge = new LidoBridge(wstETH, lidoBridgeMock);
        // fund attacker with wstETH and set approval
        deal(wstETH, attacker, 1 ether);
        vm.prank(attacker);
        IERC20(wstETH).approve(address(bridge), 1 ether);
    }

    function test_unprivilegedCallerCanBridgeDirectly() public {
        vm.prank(attacker);
        // Succeeds with no role check — no revert
        bridge.bridgeTokenToL1(attackerL1, 1 ether);
        // RSETHPool.feeEarnedInToken is never touched — assert separately
    }
}
```

The test passes on unmodified code, confirming the missing access control.
