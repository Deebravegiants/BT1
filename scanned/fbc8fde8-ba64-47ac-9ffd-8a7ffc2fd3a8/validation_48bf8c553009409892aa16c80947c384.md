### Title
Missing Slippage Protection in L2 Pool `deposit` Functions — (`contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`, `contracts/pools/RSETHPoolNoWrapper.sol`, `contracts/pools/RSETHPool.sol`)

### Summary
All L2 pool `deposit` functions accept ETH or LST tokens and return wrsETH/rsETH to the caller, but provide no `minRsETHAmountExpected` parameter. The mainnet `LRTDepositPool` explicitly enforces this guard; the L2 pools do not. A user who submits a deposit transaction cannot bound the minimum rsETH they receive, leaving them exposed to unfavorable oracle-rate movements between submission and execution.

### Finding Description
`LRTDepositPool.depositETH` and `LRTDepositPool.depositAsset` both accept a `minRSETHAmountExpected` argument and revert if the computed mint amount falls below it:

```solidity
// LRTDepositPool.sol:667
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

Every L2 pool `deposit` function omits this parameter entirely. The ETH-deposit path in `RSETHPoolV3`:

```solidity
// RSETHPoolV3.sol:246-265
function deposit(string memory referralId)
    external payable nonReentrant whenNotPaused
    limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);   // no floor check
    ...
}
```

The token-deposit path is identical in structure:

```solidity
// RSETHPoolV3.sol:271-293
function deposit(address token, uint256 amount, string memory referralId)
    external nonReentrant whenNotPaused onlySupportedToken(token)
    limitDailyMint(amount, token)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
    feeEarnedInToken[token] += fee;
    wrsETH.mint(msg.sender, rsETHAmount);   // no floor check
    ...
}
```

The same pattern is present in `RSETHPoolV3ExternalBridge.deposit` (lines 366–384 and 390–412), `RSETHPoolNoWrapper.deposit` (lines 231–244 and 250–271), and `RSETHPool.deposit` (lines 265–278 and 284–305).

The rsETH amount is computed from `getRate()` → `IOracle(rsETHOracle).getRate()`, which is an externally-set oracle. If the oracle rate moves between the time a user signs a transaction and the time it is included in a block, the user silently receives fewer rsETH tokens with no recourse.

### Impact Explanation
A depositor receives fewer wrsETH/rsETH tokens than they observed when constructing the transaction. Because the pool mints (or transfers) exactly the oracle-computed amount with no floor, the contract fails to deliver the return the user expected. No funds are permanently lost, but the user receives less value than promised. This maps to **Low — contract fails to deliver promised returns, but doesn't lose value**.

### Likelihood Explanation
The rsETH/ETH oracle rate is updated periodically (e.g., on each Chainlink heartbeat or deviation threshold). On L2 networks, block times are short and mempool visibility is limited, but oracle updates can still land in the same block as a user deposit, or a user's transaction can be delayed in the mempool while the rate shifts. This is a routine occurrence, not a rare edge case.

### Recommendation
Add a `minRsETHAmountExpected` parameter to every L2 pool `deposit` overload, mirroring the mainnet `LRTDepositPool` pattern:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountExpected)
    external payable nonReentrant whenNotPaused
    limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    ...
}
```

### Proof of Concept
1. User queries `viewSwapRsETHAmountAndFee(1 ether)` off-chain; oracle returns rate R, expected output = X wrsETH.
2. User submits `deposit{value: 1 ether}(referralId)`.
3. Before the transaction is mined, the oracle updates to rate R′ > R (rsETH appreciated vs ETH).
4. `viewSwapRsETHAmountAndFee` now returns X′ < X.
5. `wrsETH.mint(msg.sender, X′)` executes; user receives X′ with no revert.
6. User has no on-chain mechanism to reject this outcome.

    function _beforeDeposit(
        address asset,
        uint256 depositAmount,
        uint256 minRSETHAmountExpected
    )
        private
        view
        returns (uint256 rsethAmountToMint)
    {
        if (depositAmount == 0 || depositAmount < minAmountToDeposit) {
            revert InvalidAmountToDeposit();
        }

        if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
            revert MaximumDepositLimitReached();
        }

        rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

        if (rsethAmountToMint < minRSETHAmountExpected) {
            revert MinimumAmountToReceiveNotMet();
        }
    }
