### Title
L2 Pools Use Stale Cross-Chain Rate During Propagation Window, Allowing Deposits at Below-Current Rate — (`contracts/cross-chain/CrossChainRateProvider.sol`, `contracts/pools/RSETHPoolV2.sol`)

---

### Summary

There is an unbounded time window between when `LRTOracle.updateRSETHPrice()` updates `rsETHPrice` on L1 and when `CrossChainRateProvider.updateRate()` propagates the new rate to L2 via LayerZero. During this window, all three L2 pool contracts (`RSETHPoolV2`, `RSETHPoolV2NBA`, `RSETHPoolV2ExternalBridge`) continue to use the stale (lower) rate stored in `CrossChainRateReceiver.rate`. A user who observes the L1 oracle update can deposit on L2 before the rate propagates and receive more wrsETH than the updated rate would give.

---

### Finding Description

**Rate update path:**

1. `LRTOracle.updateRSETHPrice()` (L1, permissionless) → updates `rsETHPrice` on L1.
2. `CrossChainRateProvider.updateRate()` (L1, permissionless, requires ETH for LZ gas) → reads `getLatestRate()`, sends via LayerZero.
3. `CrossChainRateReceiver.lzReceive()` (L2) → sets `rate = _rate` and `lastUpdated = block.timestamp`.
4. `RSETHPoolV2.deposit()` → `viewSwapRsETHAmountAndFee()` → `getRate()` → `IOracle(rsETHOracle).getRate()` → returns `CrossChainRateReceiver.rate`.

Steps 1 and 2–3 are **separate transactions** with no atomicity guarantee. `updateRSETHPrice()` is permissionless and can be called by anyone. `updateRate()` is also permissionless but requires a separate call and LayerZero message delivery latency. The L2 pool has **no staleness check** on the rate it reads.

**Stale rate usage in deposit math:**

```solidity
// RSETHPoolV2.sol L233
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

If `rsETHToETHrate` is the old (lower) P1 instead of the new P2, the user receives `amountAfterFee * 1e18 / P1 > amountAfterFee * 1e18 / P2` wrsETH.

**`lastUpdated` is stored but never checked:**

`CrossChainRateReceiver` records `lastUpdated` at line 97, but none of the three pool contracts ever read or validate it against a maximum staleness window.

---

### Impact Explanation

The protocol mints more wrsETH than the current backing rate justifies during the propagation delay window. This dilutes existing wrsETH holders by a small amount proportional to the rate increase. No ETH is lost by the protocol (it receives the deposited ETH), but it delivers more wrsETH than the updated rate warrants — matching the "Low: contract fails to deliver promised returns, but doesn't lose value" scope.

---

### Likelihood Explanation

- `updateRSETHPrice()` is permissionless; anyone can trigger a rate update on L1.
- `updateRate()` requires a separate call and LayerZero delivery (minutes of latency minimum).
- The window is predictable and observable on-chain.
- The profit per deposit is bounded by the rate delta (typically small, e.g., daily staking rewards), but the window is always present and requires no special privileges.

---

### Recommendation

Add a maximum staleness check in the L2 pool's `getRate()` or `viewSwapRsETHAmountAndFee()` that reverts (or pauses deposits) if `block.timestamp - CrossChainRateReceiver.lastUpdated > MAX_STALENESS`. This forces deposits to wait until the rate has been propagated within an acceptable window.

Alternatively, expose `lastUpdated` from the oracle interface and enforce it in the pool's deposit path.

---

### Proof of Concept

**Trace through unmodified production code:**

1. **L1**: Anyone calls `LRTOracle.updateRSETHPrice()` — `rsETHPrice` updates from P1 → P2 (P2 > P1). [1](#0-0) 

2. **L1**: `CrossChainRateProvider.updateRate()` has NOT yet been called. `CrossChainRateReceiver.rate` on L2 still equals P1. [2](#0-1) 

3. **L2**: Attacker calls `RSETHPoolV2.deposit{value: X}("")`. Pool calls `viewSwapRsETHAmountAndFee(X)`, which calls `getRate()` → `IOracle(rsETHOracle).getRate()` → returns stale P1. [3](#0-2) 

4. **L2**: `CrossChainRateReceiver.getRate()` returns the stale `rate` with no staleness validation. [4](#0-3) 

5. **L2**: Attacker receives `X * 1e18 / P1` wrsETH. After `updateRate()` propagates, the correct amount would be `X * 1e18 / P2 < X * 1e18 / P1`. The difference is the excess wrsETH minted at the stale rate. [5](#0-4) 

6. The same path applies identically to `RSETHPoolV2NBA.deposit()` and `RSETHPoolV2ExternalBridge.deposit()`, both of which use the same `getRate()` → `IOracle(rsETHOracle).getRate()` pattern with no staleness check. [6](#0-5) [7](#0-6)

```text
    function updateRate() external payable nonReentrant {
        uint256 latestRate = getLatestRate();

        bytes memory remoteAndLocalAddresses = abi.encodePacked(rateReceiver, address(this));

        rate = latestRate;

        lastUpdated = block.timestamp;

        bytes memory _payload = abi.encode(latestRate);

        ILayerZeroEndpoint(layerZeroEndpoint).send{ value: msg.value }(
            dstChainId, remoteAndLocalAddresses, _payload, payable(msg.sender), address(0x0), bytes("")
        );

        emit RateUpdated(rate);
    }
```
