### Title
`updateRSETHPrice()` Reverts When `maxFeeMintAmountPerDay` Is Zero and Protocol Fee Is Non-Zero — (File: `contracts/LRTOracle.sol`)

---

### Summary

`setMaxFeeMintAmountPerDay` accepts zero as a valid value. When `maxFeeMintAmountPerDay == 0` and the protocol has accrued rewards (TVL increased), `_checkAndUpdateDailyFeeMintLimit` unconditionally reverts with `DailyFeeMintLimitExceeded`, permanently blocking `updateRSETHPrice()` until an admin intervenes. This is the direct analog of the StreamingFeeModule bug: a zero-valued limit parameter causes the fee-accrual path to revert, freezing protocol fee yield and leaving the rsETH price oracle stale.

---

### Finding Description

`setMaxFeeMintAmountPerDay` contains no zero-value guard:

```solidity
// contracts/LRTOracle.sol:132-135
function setMaxFeeMintAmountPerDay(uint256 _maxFeeMintAmountPerDay) external onlyLRTManager {
    maxFeeMintAmountPerDay = _maxFeeMintAmountPerDay;   // zero is accepted
    emit MaxFeeMintAmountPerDayUpdated(_maxFeeMintAmountPerDay);
}
```

`maxFeeMintAmountPerDay` also starts at `0` by default (uninitialized storage). The `reinitialize` function only sets `feePeriodStartTime`, never `maxFeeMintAmountPerDay`.

Inside `_checkAndUpdateDailyFeeMintLimit`, the limit check is:

```solidity
// contracts/LRTOracle.sol:205-207
if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
    revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
}
```

When `maxFeeMintAmountPerDay == 0` and `feeAmount > 0`, the condition `0 + feeAmount > 0` is always `true` → always reverts.

`_updateRsETHPrice` calls this function with a non-zero fee whenever TVL has grown:

```solidity
// contracts/LRTOracle.sol:299-311
if (protocolFeeInETH > 0) {
    uint256 rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice);
    _checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee);  // ← reverts
    ...
} else {
    _checkAndUpdateDailyFeeMintLimit(0);  // safe path, never reached when rewards exist
}
```

The `remainingDailyFeeMintLimit()` view function already has a special-case guard for zero (`if (maxFeeMintAmountPerDay == 0) return 0;`), confirming the developers anticipated this state — but the internal `_checkAndUpdateDailyFeeMintLimit` was never given the same treatment.

---

### Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

1. Every call to `updateRSETHPrice()` (public) and `updateRSETHPriceAsManager()` reverts as long as `maxFeeMintAmountPerDay == 0` and rewards are accruing.
2. The protocol fee (`protocolFeeInETH`) that should be minted as rsETH to the treasury is never minted. Because `rsETHPrice` is never updated, the fee calculation on the next successful call will be based on the stale stored price, meaning the accumulated fee is permanently unrecoverable for the periods during which the oracle was broken.
3. The stale `rsETHPrice` causes `getRsETHAmountToMint` in `LRTDepositPool` to over-issue rsETH to new depositors (they pay at the old lower price), diluting existing rsETH holders' share of TVL — a permanent loss of yield for them.

---

### Likelihood Explanation

**Medium.**

- `maxFeeMintAmountPerDay` is `0` by default at deployment/upgrade (uninitialized). If the manager does not call `setMaxFeeMintAmountPerDay` before rewards begin accruing, the oracle breaks immediately on the first price update after TVL grows.
- A manager can also explicitly set it back to `0` at any time (no guard), re-triggering the condition.
- `protocolFeeInBPS` is a live protocol parameter in `LRTConfig`; as long as it is non-zero and TVL grows (the normal operating condition), the revert path is hit.
- `updateRSETHPrice()` is a public function — any external caller will observe the revert.

---

### Recommendation

Add a zero-limit early-return inside `_checkAndUpdateDailyFeeMintLimit`, mirroring the guard already present in `remainingDailyFeeMintLimit`:

```solidity
function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
    if (maxFeeMintAmountPerDay == 0) return; // ← add this guard

    if (block.timestamp >= feePeriodStartTime + 1 days) {
        currentPeriodMintedFeeAmount = 0;
        feePeriodStartTime = getCurrentPeriodStartTime();
    }

    if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
        revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
    }

    currentPeriodMintedFeeAmount += feeAmount;
}
```

Alternatively, add a non-zero check in `setMaxFeeMintAmountPerDay` and ensure `reinitialize` also initialises `maxFeeMintAmountPerDay` to a safe default.

---

### Proof of Concept

1. Deploy `LRTOracle`. `maxFeeMintAmountPerDay` is `0` (default). `protocolFeeInBPS` in `LRTConfig` is non-zero (e.g., 500 bps).
2. Time passes; staking rewards cause `totalETHInProtocol > previousTVL`.
3. Any account calls `updateRSETHPrice()`.
4. Inside `_updateRsETHPrice`: `protocolFeeInETH > 0` → `rsethAmountToMintAsProtocolFee > 0`.
5. `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` executes: `0 + rsethAmountToMintAsProtocolFee > 0` → `revert DailyFeeMintLimitExceeded(...)`.
6. `updateRSETHPrice()` reverts. `rsETHPrice` is never updated. Protocol fee is never minted. All subsequent calls revert identically until a manager calls `setMaxFeeMintAmountPerDay(nonZeroValue)`.
