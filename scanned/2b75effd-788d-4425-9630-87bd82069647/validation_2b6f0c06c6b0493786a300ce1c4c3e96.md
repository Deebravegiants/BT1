### Title
No Minimum Withdrawal Amount Enforced by Default Enables Queue Stuffing and Temporary Freezing of User Funds — (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTWithdrawalManager` declares `minRsEthAmountToWithdraw` as a mapping that defaults to `0` for every asset and is never set during `initialize`. The guard in `initiateWithdrawal` therefore only rejects a zero-amount request; any amount of `1 wei` of rsETH passes. An unprivileged rsETH holder can spam the withdrawal queue with arbitrarily many dust requests, forcing the operator to exhaust many `unlockQueue` calls to drain the queue before legitimate users' requests can be unlocked, temporarily freezing those users' funds.

---

### Finding Description

`minRsEthAmountToWithdraw` is declared as a plain storage mapping: [1](#0-0) 

Its Solidity default value is `0`. The `initialize` function never writes to it: [2](#0-1) 

The only setter is admin-gated: [3](#0-2) 

The guard in `initiateWithdrawal` is:

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
``` [4](#0-3) 

When `minRsEthAmountToWithdraw[asset] == 0` (the default), the condition reduces to `rsETHUnstaked == 0`, so any `rsETHUnstaked >= 1` is accepted. Each accepted call appends a new entry to the global per-asset nonce sequence: [5](#0-4) 

The operator must later drain the queue by calling `unlockQueue`, which internally runs a `while` loop from `nextLockedNonce` to `firstExcludedIndex`: [6](#0-5) 

Because the queue is strictly FIFO (the loop always starts at `nextLockedNonce`), the operator cannot skip the attacker's dust entries; every dust entry must be individually iterated and unlocked before `nextLockedNonce` advances past it. Legitimate users whose requests were queued after the attacker's cannot have their requests unlocked until all preceding dust entries are processed.

---

### Impact Explanation

**Temporary freezing of funds (Medium).** Legitimate rsETH holders who have called `initiateWithdrawal` after the attacker cannot call `completeWithdrawal` until the operator has processed every dust entry ahead of them in the queue. With enough dust entries the operator must issue many `unlockQueue` transactions (each bounded by `firstExcludedIndex`), delaying legitimate withdrawals indefinitely. The attacker's own dust rsETH is also locked for the withdrawal delay period, but the cost to the attacker is only gas for the spam transactions.

Additionally, each dust request increments `assetsCommitted[asset]`, reducing `getAvailableAssetAmount` and potentially blocking new withdrawal requests from other users until the dust is processed: [7](#0-6) 

---

### Likelihood Explanation

**Medium.** The default state of every fresh deployment (or any asset added later) has `minRsEthAmountToWithdraw == 0`. The admin must proactively call `setMinRsEthAmountToWithdraw` for each supported asset to close the window. Any rsETH holder can exploit this at any time the minimum is unset, with no special permissions required.

---

### Recommendation

Set a non-zero `minRsEthAmountToWithdraw` for each supported asset inside `initialize` (or enforce a protocol-wide floor that cannot be set to zero). Alternatively, require `minRsEthAmountToWithdraw[asset] > 0` as a precondition before `initiateWithdrawal` is callable for that asset.

---

### Proof of Concept

1. Admin deploys `LRTWithdrawalManager` and calls `initialize`. `minRsEthAmountToWithdraw[ETH]` is `0` (never set).
2. Attacker acquires a small amount of rsETH (e.g., 10 000 wei worth).
3. Attacker calls `initiateWithdrawal(ETH, 1, "")` 10 000 times in a loop (one wei of rsETH each). Each call passes the `rsETHUnstaked == 0` guard and appends a new entry at nonces `0..9999`.
4. Legitimate user calls `initiateWithdrawal(ETH, 1 ether, "")` — their request lands at nonce `10000`.
5. Operator calls `unlockQueue(ETH, 100, ...)` repeatedly. Each call advances `nextLockedNonce` by at most 100. After 100 operator transactions the attacker's 10 000 dust entries are finally drained.
6. Only then can the legitimate user's request at nonce `10000` be unlocked and `completeWithdrawal` succeed. The user's funds are frozen for the entire duration.
