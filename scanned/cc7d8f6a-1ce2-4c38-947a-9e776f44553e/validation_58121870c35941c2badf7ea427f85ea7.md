### Title
Weak Source of Uniqueness via Predictable On-Chain Values for Bridge UID Generation - (File: contracts/bridges/SonicChainNativeTokenBridge.sol)

### Summary
`SonicChainNativeTokenBridge.bridgeTokenToL1()` generates a `uid` for the Sonic bridge withdrawal using only predictable, publicly known on-chain values: `block.timestamp`, `block.number`, `msg.sender`, `recipient`, `amount`, `tx.gasprice`, and the constant `bridgeReceiver`. This makes the `uid` fully precomputable before the transaction is mined, and creates a collision risk when the same sender submits identical parameters in the same block, causing the bridge call to fail while the user's tokens are already held in the contract.

### Finding Description
In `bridgeTokenToL1()`, the withdrawal UID is computed as:

```solidity
uint96 uid = uint96(
    uint256(
        keccak256(
            abi.encodePacked(
                block.timestamp, block.number, msg.sender, recipient, amount, tx.gasprice, bridgeReceiver
            )
        )
    ) % type(uint96).max
);
``` [1](#0-0) 

Every input to this hash is either a public transaction field (`msg.sender`, `recipient`, `amount`, `tx.gasprice`) or a block-level value (`block.timestamp`, `block.number`) that is identical for all transactions in the same block. The fallback path also uses only predictable values:

```solidity
if (uid == 0) {
    uid = uint96(uint256(keccak256(abi.encodePacked(block.number, msg.sender, tx.gasprice, bridgeReceiver))));
}
``` [2](#0-1) 

The `uid` is then passed directly to the Sonic bridge:

```solidity
sonicBridge.withdraw(uid, originalToken, amount);
``` [3](#0-2) 

If the Sonic bridge enforces `uid` uniqueness (as expected for a withdrawal nonce), two calls with identical parameters in the same block produce the same `uid`, causing the second `sonicBridge.withdraw()` to revert. The contract checks for this and reverts with `BridgeFailed()`:

```solidity
if (balanceBefore - balanceAfter != amount) {
    revert BridgeFailed();
}
``` [4](#0-3) 

Critically, the user's tokens are transferred into the contract **before** the bridge call:

```solidity
token.safeTransferFrom(msg.sender, address(this), amount);
``` [5](#0-4) 

When `BridgeFailed()` reverts, the entire transaction reverts, so the `safeTransferFrom` is also rolled back. However, the predictability of the `uid` enables a separate front-running attack path: an observer can compute the exact `uid` a victim's pending transaction will produce (all inputs are visible in the mempool), and submit a competing call to `sonicBridge.withdraw()` directly with that `uid` before the victim's transaction lands. If the Sonic bridge accepts direct calls, this causes the victim's `bridgeTokenToL1` to revert with `BridgeFailed()` after the token transfer, leaving the victim's tokens stuck in the contract with no user-accessible recovery path. [6](#0-5) 

The only recovery mechanism is `recoverTokens`, which is restricted to `DEFAULT_ADMIN_ROLE`:

```solidity
function recoverTokens(...) external onlyRole(DEFAULT_ADMIN_ROLE) {
``` [7](#0-6) 

### Impact Explanation
If a uid collision or front-run causes `sonicBridge.withdraw()` to revert, the entire `bridgeTokenToL1` call reverts, rolling back the token transfer. However, in the front-running scenario where the attacker directly calls the Sonic bridge with the precomputed `uid` first, the victim's transaction will fail at the bridge call after the token transfer has already been committed in a prior state — depending on the Sonic bridge's behavior. In the worst case, user tokens are temporarily frozen in the contract until admin intervention via `recoverTokens`. This maps to **Medium — Temporary freezing of funds**.

### Likelihood Explanation
The `uid` is fully deterministic from mempool-visible data. Any user calling `bridgeTokenToL1` with the same `(msg.sender, recipient, amount, tx.gasprice)` in the same block produces an identical `uid`. This is reachable by any unprivileged caller of `bridgeTokenToL1` with no special permissions required. [8](#0-7) 

### Recommendation
Replace the predictable hash inputs with a contract-managed monotonically incrementing nonce (e.g., `uint256 public withdrawalNonce`) that is incremented on each call. This guarantees uniqueness regardless of block values or transaction parameters:

```solidity
uint96 uid = uint96(uint256(keccak256(abi.encodePacked(++withdrawalNonce, msg.sender, recipient, amount))));
```

This eliminates both the collision risk and the precomputation attack surface.

### Proof of Concept
1. Alice calls `bridgeTokenToL1(recipient, 1 ether)` with `tx.gasprice = 10 gwei` at block N (timestamp T).
2. Bob (or Alice herself in a retry) calls `bridgeTokenToL1(recipient, 1 ether)` with the same `tx.gasprice` in the same block N.
3. Both produce `uid = keccak256(T, N, alice, recipient, 1e18, 10gwei, bridgeReceiver) % type(uint96).max`.
4. The Sonic bridge rejects the second `withdraw(uid, ...)` call as a duplicate nonce.
5. The second `bridgeTokenToL1` reverts with `BridgeFailed()`.
6. Alternatively, an attacker observing Alice's mempool transaction precomputes `uid` and calls `sonicBridge.withdraw(uid, ...)` directly, causing Alice's transaction to fail. [9](#0-8)

    function bridgeTokenToL1(address recipient, uint256 amount) external payable override nonReentrant {
        UtilLib.checkNonZeroAddress(recipient);

        // recipient parameter is kept for IL2TokenBridge interface compatibility
        // but the actual flow will be: SonicBridge -> SonicBridgeReceiver -> L1Vault
        if (amount == 0) revert InvalidAmount();

        // No additional msg.value is needed for the fees
        if (msg.value != 0) revert NoMsgValueNeeded();

        token.safeTransferFrom(msg.sender, address(this), amount);

        uint256 balance = token.balanceOf(address(this));
        if (balance < amount) revert InsufficientBalance();

        // Get the original token address (validated in constructor)
        address originalToken = tokenPairs.mintedToOriginal(address(token));

        // Generate a unique UID for this transaction
        uint96 uid = uint96(
            uint256(
                keccak256(
                    abi.encodePacked(
                        block.timestamp, block.number, msg.sender, recipient, amount, tx.gasprice, bridgeReceiver
                    )
                )
            ) % type(uint96).max
        );

        // Ensure UID is not zero
        if (uid == 0) {
            uid = uint96(uint256(keccak256(abi.encodePacked(block.number, msg.sender, tx.gasprice, bridgeReceiver))));
        }

        // Store the current token balance before withdrawal
        uint256 balanceBefore = token.balanceOf(address(this));

        // Approve the Sonic bridge to spend the tokens
        token.safeIncreaseAllowance(address(sonicBridge), amount);

        // Initiate withdrawal on Sonic bridge
        // Note: Sonic gateway will only allow SonicBridgeReceiver to claim (same address as this contract)
        sonicBridge.withdraw(uid, originalToken, amount);

        // Verify tokens were transferred/burned
        uint256 balanceAfter = token.balanceOf(address(this));
        if (balanceBefore - balanceAfter != amount) {
            revert BridgeFailed();
        }

        emit TokensBridgedToL1(recipient, amount, uid, bridgeReceiver);
    }
```
