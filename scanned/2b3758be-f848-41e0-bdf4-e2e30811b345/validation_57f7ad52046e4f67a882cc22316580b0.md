Looking at the exact code path in `depositAssetForL1Vault` and `LRTDepositPool.depositAsset`, I need to verify whether the revert is atomic (ruling out allowance accumulation) and whether a genuine no-recovery-path exists.

**Atomic revert confirmed**: `safeIncreaseAllowance` + `depositAsset` are in the same transaction — if `depositAsset` reverts, the allowance increase is also rolled back. The "accumulated allowance" claim in the question is incorrect. However, the core freezing issue is real and independent of that detail.

**No recovery path confirmed**: `grep` for `rescue|recover|emergencyWithdraw|transferToken|sweep` returns zero matches in `L1VaultV2.sol`. The contract has no function to move LSTs out of the vault by any path other than `depositAssetForL1Vault`.

### Title
Deposit-limit saturation permanently blocks LST withdrawal from L1VaultV2 with no recovery path — (`contracts/L1VaultV2.sol`)

### Summary
`depositAssetForL1Vault` is the **only** egress path for LST tokens held in `L1VaultV2`. It does not pre-check the deposit limit before attempting the deposit. When `LRTDepositPool`'s per-asset deposit limit is at capacity, every call to `depositAssetForL1Vault` reverts, and there is no alternative function (rescue, sweep, partial-deposit, or admin withdrawal) to move the tokens out. LSTs bridged from L2 accumulate in the vault with no path to rsETH minting until an admin raises the limit.

### Finding Description

`depositAssetForL1Vault` in `L1VaultV2.sol`:

```
uint256 tokenBalance = IERC20(token).balanceOf(address(this));
uint256 rsETHAmountToMint = lrtDepositPool.getRsETHAmountToMint(token, tokenBalance);
// ↑ oracle-only calculation — does NOT check deposit limit

IERC20(token).safeIncreaseAllowance(address(lrtDepositPool), tokenBalance);
lrtDepositPool.depositAsset(token, tokenBalance, rsETHAmountToMint, "");
// ↑ reverts with MaximumDepositLimitReached when limit is saturated
``` [1](#0-0) 

`getRsETHAmountToMint` is a pure oracle calculation that returns a non-zero value regardless of the deposit limit: [2](#0-1) 

The actual limit check happens inside `_beforeDeposit` → `_checkIfDepositAmountExceedesCurrentLimit`, which reverts the entire transaction: [3](#0-2) [4](#0-3) 

Because the revert is atomic, the allowance increase is also rolled back — the "accumulated allowance" claim in the question is incorrect. The real issue is structural: `L1VaultV2` has **no other function** to move LSTs out. A search for `rescue`, `recover`, `emergencyWithdraw`, `transferToken`, and `sweep` returns zero matches. [5](#0-4) 

### Impact Explanation
LST tokens (stETH, cbETH, ETHx, etc.) bridged from L2 to `L1VaultV2` are temporarily frozen: they cannot be deposited into `LRTDepositPool` (limit reached), cannot be transferred out (no rescue function), and cannot be partially deposited (function always uses full balance). No rsETH is minted for the corresponding L2 users during the freeze. The freeze lifts only when an admin raises `depositLimitByAsset` via `LRTConfig.updateAssetDepositLimit`.

**Impact: Medium — Temporary freezing of funds.**

### Likelihood Explanation
Deposit limits are a live, actively managed protocol parameter initialized at 100,000 ether per asset. The bridge pipeline (L2 → L1VaultV2) operates independently of the deposit pool's remaining capacity. A manager can bridge LSTs to L1VaultV2 at any time, including when the pool is at or near capacity. No on-chain guard prevents this mismatch. The scenario requires no privileged compromise — it is a natural operational state.

### Recommendation

1. **Pre-check the deposit limit** before attempting the deposit in `depositAssetForL1Vault`:

```solidity
uint256 currentLimit = lrtDepositPool.getAssetCurrentLimit(token);
if (currentLimit < tokenBalance) revert DepositLimitReached();
```

2. **Add an emergency token-rescue function** gated to `TIMELOCK_ROLE` so that LSTs can be recovered if the deposit path is blocked for any reason:

```solidity
function rescueToken(address token, address to, uint256 amount)
    external onlyRole(TIMELOCK_ROLE) {
    IERC20(token).safeTransfer(to, amount);
}
```
