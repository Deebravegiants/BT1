### Title
Deposits Proceed with Stale `rsETHPrice` When Oracle Is Paused Independently — (`contracts/LRTDepositPool.sol`, `contracts/LRTOracle.sol`)

### Summary
When `LRTOracle` is paused by the `PAUSER_ROLE` without simultaneously pausing `LRTDepositPool`, user deposits continue to execute using the stale stored `rsETHPrice`. Because `rsETHPrice` is only updated via `updateRSETHPrice()` (which is blocked by `whenNotPaused`), but `LRTDepositPool.depositETH()` / `depositAsset()` carry no check on the oracle's pause state, an unprivileged depositor can mint rsETH at an incorrect exchange rate, stealing yield from existing rsETH holders.

### Finding Description

`LRTOracle` maintains a custom `paused` flag and a `whenNotPaused` modifier that gates `updateRSETHPrice()`: [1](#0-0) [2](#0-1) 

The `PAUSER_ROLE` can pause the oracle independently of the deposit pool: [3](#0-2) 

The deposit pool's `depositETH()` and `depositAsset()` only check the deposit pool's own pause state, not the oracle's: [4](#0-3) [5](#0-4) 

The minting calculation reads the stored `rsETHPrice` directly from the oracle state variable (no pause guard on the read path) and the live `getAssetPrice()` (also no pause guard): [6](#0-5) [7](#0-6) 

`rsETHPrice` is a stored value that is only refreshed by `_updateRsETHPrice()`. When the oracle is paused, this value becomes stale. Meanwhile `getAssetPrice()` continues to return live prices from the underlying price fetcher. The two inputs to the minting formula therefore become desynchronised: a live asset price divided by a stale rsETH price.

The auto-pause path in `_updateRsETHPrice()` does pause all three contracts together: [8](#0-7) 

However, the `PAUSER_ROLE` can also pause the oracle alone (e.g., to halt price updates during a suspected oracle manipulation) without touching the deposit pool. In that window, deposits remain open.

Additionally, `updateRSETHPriceAsManager()` carries no `whenNotPaused` guard, meaning the LRT manager can update the price even during the oracle's emergency state — a secondary pause bypass: [9](#0-8) 

### Impact Explanation

If the oracle is paused while `rsETHPrice` is **lower than the true current value** (e.g., paused before a reward accrual update), the minting formula `(amount × assetPrice) / rsETHPrice` yields a **higher rsETH amount** than warranted. Every deposit in this window mints excess rsETH, diluting the share of all existing holders. When the oracle is later unpaused and the price is updated upward, the attacker's rsETH is worth more than the ETH they deposited — a direct theft of accrued yield from existing holders.

Impact classification: **High — Theft of unclaimed yield.**

### Likelihood Explanation

The `PAUSER_ROLE` is a distinct role from `DEFAULT_ADMIN_ROLE` and is expected to be used operationally (e.g., during suspected oracle anomalies). Pausing only the oracle without the deposit pool is a realistic operational sequence — the role holder may intend to freeze price updates while keeping withdrawals/deposits live, or may simply omit the deposit pool pause. No attacker compromise is required; the depositor only needs to observe that the oracle is paused (public `paused` state variable) and that the deposit pool is not.

### Recommendation

1. In `LRTDepositPool._beforeDeposit()`, add a check that the oracle is not paused before proceeding:
   ```solidity
   IPausable oracle = IPausable(lrtConfig.getContract(LRTConstants.LRT_ORACLE));
   if (oracle.paused()) revert OraclePaused();
   ```
2. Add `whenNotPaused` to `updateRSETHPriceAsManager()` to prevent price updates during the oracle's emergency state.
3. Consider making the oracle's `pause()` function automatically pause the deposit pool and withdrawal manager (mirroring the auto-pause logic already present in `_updateRsETHPrice()`).

### Proof of Concept

1. Protocol is operating normally; `rsETHPrice = 1.05e18` (reflecting accrued rewards).
2. `PAUSER_ROLE` calls `LRTOracle.pause()` — only the oracle is paused; `LRTDepositPool` remains active.
3. Rewards continue to accrue off-chain; the true rsETH price rises toward `1.06e18`, but `rsETHPrice` is frozen at `1.05e18`.
4. Attacker calls `LRTDepositPool.depositETH{value: 100 ether}(0, "")`.
5. `getRsETHAmountToMint` computes: `(100e18 × 1e18) / 1.05e18 ≈ 95.24 rsETH` — more than the `(100e18 × 1e18) / 1.06e18 ≈ 94.34 rsETH` that the correct price would yield.
6. Attacker receives ~0.9 rsETH excess at the expense of existing holders.
7. When the oracle is unpaused and `updateRSETHPrice()` is called, `rsETHPrice` updates to `1.06e18`; the attacker's excess rsETH is now fully backed by the protocol's TVL, diluting every other holder's position.
