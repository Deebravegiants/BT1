### Title
Cumulative Integer-Division Dust in `notifyRewardAmount` Mid-Period Rollover Causes Stakers to Receive Less Than Total Deposited Rewards — (`contracts/KERNEL/KernelDepositPool.sol`)

---

### Summary

`KernelDepositPool.notifyRewardAmount` performs two sequential integer divisions when called mid-period. Each division truncates, and the truncated dust is never re-incorporated into future rate calculations, causing it to accumulate permanently in the contract. The invariant `Σ getReward() == Σ notifyRewardAmount(amount)` is broken.

---

### Finding Description

The relevant logic is in `notifyRewardAmount`:

```solidity
// Line 580 — fresh period: first truncation
rewardRate = receivedAmount / duration;

// Lines 582-583 — mid-period top-up: second truncation
uint256 remaining = (finishAt - block.timestamp) * rewardRate;
rewardRate = (receivedAmount + remaining) / duration;
``` [1](#0-0) 

**Step-by-step dust accumulation:**

**Call 1 (fresh period):**
- `rewardRate = R1 / duration` (floor division)
- Dust locked: `d1 = R1 % duration` (stays in contract, never distributed)
- Over the full period, stakers receive `floor(R1/duration) * duration = R1 - d1`

**Call 2 (mid-period, elapsed `e` seconds):**
- `remaining = (duration - e) * floor(R1/duration)`
- Actual remaining tokens in contract from R1: `R1 - e * floor(R1/duration)`
- The gap between actual remaining and computed `remaining`:
  ```
  actual_remaining - remaining
  = R1 - e*floor(R1/d) - (d-e)*floor(R1/d)
  = R1 - d*floor(R1/d)
  = R1 % d   ← the d1 dust is silently dropped
  ```
- Then `rewardRate = (R2 + remaining) / duration` truncates again, adding `d2 = (R2 + remaining) % duration`

After N top-ups, total dust locked = `d1 + d2 + ... + dN`, where each `di < duration`.

The `rewardPerToken` accumulator compounds this:

```solidity
return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
    / totalKernelStaked;
``` [2](#0-1) 

Since `rewardRate` is already truncated, the per-token accumulation also underestimates, and the dust is never recoverable by any staker.

---

### Impact Explanation

- Tokens are not stolen — they remain in the contract.
- Stakers collectively receive `< Σ notifyRewardAmount(amounts)`.
- The maximum dust per call is `duration - 1` in the reward token's smallest unit.
- For a `duration = 7 days = 604,800 seconds` and a 6-decimal token (e.g., USDC), the maximum dust per call is `~0.6 USDC`. Over N=100 top-ups, this is `~60 USDC` permanently stranded.
- For 18-decimal tokens the per-call dust is negligible in practice, but the invariant is still broken.

**Scope match:** Low — Contract fails to deliver promised returns, but doesn't lose value.

---

### Likelihood Explanation

- `notifyRewardAmount` is a normal admin operation expected to be called repeatedly (e.g., weekly top-ups on a monthly period).
- No attacker action is required — this is a structural property of the contract triggered by normal admin usage.
- The `onlyRole(DEFAULT_ADMIN_ROLE)` gate is not a mitigation; it is the intended caller.

---

### Recommendation

Track undistributed dust explicitly and roll it forward into the next period:

```solidity
// After computing rewardRate:
uint256 distributed = rewardRate * duration;
uint256 dust = (receivedAmount + remaining) - distributed;
// Add dust to a pendingDust accumulator, or include it in the next notifyRewardAmount
```

Alternatively, use a fixed-point scaling approach (multiply `rewardRate` by a large precision factor before dividing) to reduce per-call truncation to sub-wei levels, similar to how `rewardPerToken` already uses `DECIMAL_PRECISION`.

---

### Proof of Concept

```solidity
// Pseudocode fuzz scenario
uint256 duration = 7 days;
uint256 totalDeposited;
uint256 totalClaimed;

// Call 1: deposit 1e6 + 1 tokens (e.g., USDC)
notifyRewardAmount(1_000_001);
totalDeposited += 1_000_001;
// rewardRate = 1_000_001 / 604800 = 1 (dust = 1_000_001 - 604800 = 395201)

// Warp 1 day (86400 seconds)
vm.warp(block.timestamp + 1 days);

// Call 2: deposit another 1e6 tokens mid-period
notifyRewardAmount(1_000_000);
totalDeposited += 1_000_000;
// remaining = (604800 - 86400) * 1 = 518400
// rewardRate = (1_000_000 + 518400) / 604800 = 2 (dust = 1_518_400 - 1_209_600 = 308800)

// Warp to end of new period
vm.warp(block.timestamp + 7 days);

// Single staker claims all
getReward();
totalClaimed = rewards[staker]; // = rewardRate * duration = 2 * 604800 = 1_209_600

// Assert invariant broken:
assert(totalClaimed < totalDeposited);
// gap = 2_000_001 - 1_209_600 = 790_401 units (0.79 USDC) permanently stranded
``` [3](#0-2)

    function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
        if (_amount == 0) revert AmountZero();

        // Prevent starting a reward period when no tokens are staked to avoid unallocated rewards
        if (totalKernelStaked == 0) revert NoStakedTokens();

        // Transfer reward tokens into the contract
        uint256 balanceBefore = rewardsToken.balanceOf(address(this));
        rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
        uint256 balanceAfter = rewardsToken.balanceOf(address(this));
        // Calculate the actual amount of tokens received in case of a transfer fee (tax)
        uint256 receivedAmount = balanceAfter - balanceBefore;

        if (block.timestamp >= finishAt) {
            rewardRate = receivedAmount / duration;
        } else {
            uint256 remaining = (finishAt - block.timestamp) * rewardRate;
            rewardRate = (receivedAmount + remaining) / duration;
        }

        if (rewardRate == 0) revert RewardRateZero();

        finishAt = block.timestamp + duration;
        updatedAt = block.timestamp;

        emit NotifyRewardAmount(receivedAmount, finishAt);
    }
```
