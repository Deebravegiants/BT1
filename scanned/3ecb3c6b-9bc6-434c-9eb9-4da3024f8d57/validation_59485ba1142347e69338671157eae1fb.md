### Title
Unbounded O(n) Linear Search in `HashStorage.addWithdrawal` Causes Denial of Service on Withdrawal Hash Registration - (File: contracts/utils/HashStorage.sol)

### Summary
`HashStorage.addWithdrawal` performs a full linear scan of the `withdrawalTxHashes` storage array on every invocation to detect duplicate hashes. As the number of pending (unclaimed) withdrawal hashes grows, the gas cost of each call increases proportionally. If enough hashes accumulate, the function exceeds the block gas limit and becomes permanently uncallable, preventing operators from registering new L2â†’L1 withdrawal hashes and freezing user withdrawals.

### Finding Description
`HashStorage` is a production contract that tracks L2â†’L1 withdrawal transaction hashes. Operators call `addWithdrawal` to register each new withdrawal hash, and `setWithdrawalClaimed` to remove it once processed.

In `addWithdrawal`, before pushing a new hash, the code performs a full linear scan of the `withdrawalTxHashes` array:

```solidity
uint256 length = withdrawalTxHashes.length;
for (uint256 i = 0; i < length; i++) {
    if (withdrawalTxHashes[i] == txHash) {
        revert DuplicateHash();
    }
}
``` [1](#0-0) 

This is O(n) in the number of currently pending (unclaimed) hashes. The array only shrinks when `setWithdrawalClaimed` is called, which itself also performs an O(n) scan:

```solidity
for (uint256 i = 0; i < length; i++) {
    if (withdrawalTxHashes[i] == txHash) {
        found = true;
        withdrawalTxHashes[i] = withdrawalTxHashes[withdrawalTxHashes.length - 1];
        withdrawalTxHashes.pop();
        break;
    }
}
``` [2](#0-1) 

The contract already maintains a `claimedWithdrawals` mapping, but it cannot be used for O(1) duplicate detection of pending hashes because it stores `false` for both "never added" and "added but not yet claimed" states:

```solidity
mapping(bytes32 txHash => bool isClaimed) public claimedWithdrawals;
``` [3](#0-2) 

No separate O(1) membership structure exists for the pending set.

### Impact Explanation
**Medium â€” Temporary freezing of funds.**

If the `withdrawalTxHashes` array grows large enough (e.g., during a period of high L2 withdrawal activity where claiming lags behind additions), `addWithdrawal` will exceed the block gas limit and revert on every call. Operators will be unable to register new withdrawal hashes. Users who have initiated L2â†’L1 withdrawals will be unable to complete them until the backlog is cleared â€” which itself requires `setWithdrawalClaimed` to succeed, which also has O(n) cost. Both functions can become simultaneously uncallable, creating a deadlock that temporarily freezes user withdrawal funds.

### Likelihood Explanation
**Low-Medium.** The array only holds *unclaimed* hashes, so it shrinks as withdrawals are processed. However, during periods of high cross-chain activity or if the claiming process falls behind (e.g., due to EigenLayer withdrawal delays), the pending set can grow substantially. There is no cap on `withdrawalTxHashes.length`, and no mechanism to batch-process or skip entries. A sustained burst of L2 withdrawals without timely claiming is a realistic operational scenario.

### Recommendation
Replace the linear duplicate-detection loop with an O(1) membership mapping. Introduce a separate `mapping(bytes32 => bool) public pendingWithdrawals` that is set to `true` on `addWithdrawal` and cleared on `setWithdrawalClaimed`. The duplicate check then becomes a single mapping lookup:

```solidity
mapping(bytes32 txHash => bool isPending) public pendingWithdrawals;

function addWithdrawal(bytes32 txHash) external onlyRole(OPERATOR_ROLE) {
    if (txHash == bytes32(0)) revert InvalidHash();
    if (claimedWithdrawals[txHash]) revert HashAlreadyClaimed();
    if (pendingWithdrawals[txHash]) revert DuplicateHash(); // O(1)

    pendingWithdrawals[txHash] = true;
    withdrawalTxHashes.push(txHash);
    emit WithdrawalAdded(txHash);
}
```

This mirrors the fix described in the external report: replacing a linear-search data structure (Move `SimpleMap` / Solidity array scan) with an O(1) lookup structure (Move `Table` / Solidity `mapping`).

### Proof of Concept
1. Operator calls `addWithdrawal` repeatedly for N distinct hashes without any corresponding `setWithdrawalClaimed` calls (simulating a backlog).
2. `withdrawalTxHashes.length` grows to N.
3. The next call to `addWithdrawal` executes the loop N times, reading N storage slots.
4. At approximately N â‰ˆ 3,000â€“5,000 entries (each `SLOAD` costs 2,100 gas cold / 100 gas warm; worst-case ~2,100 Ã— N), the transaction approaches or exceeds the 30M block gas limit.
5. All subsequent `addWithdrawal` calls revert with out-of-gas, and new L2â†’L1 withdrawal hashes cannot be registered. [4](#0-3)
