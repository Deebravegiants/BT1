### Title
ETH Deposit Limit Check Missing `amount` Parameter Allows Limit Bypass - (File: contracts/LRTDepositPool.sol)

### Summary
The `_checkIfDepositAmountExceedesCurrentLimit` function in `LRTDepositPool.sol` applies an asymmetric conditional check for ETH versus ERC20 tokens. For ETH, the incoming deposit `amount` is omitted from the limit comparison, so a single ETH deposit can push total deposits arbitrarily above the configured cap. This is the direct analog of the external report's wrong-condition bug: a subtly incorrect guard that is silently bypassed during normal user interaction, causing an unintended protocol state.

### Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` the ERC20 branch correctly evaluates `totalAssetDeposits + amount > limit`, but the ETH branch evaluates only `totalAssetDeposits > limit`, dropping the `amount` term entirely.

```solidity
// contracts/LRTDepositPool.sol  lines 676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount missing
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [1](#0-0) 

Consequently, whenever `totalAssetDeposits ≤ depositLimit`, the guard returns `false` regardless of how large `amount` is, and `_beforeDeposit` proceeds to mint rsETH. [2](#0-1) 

The public entry point `depositETH` calls `_beforeDeposit` with `msg.value` as `depositAmount`, so any ETH depositor can trigger this path. [3](#0-2) 

The view helper `getAssetCurrentLimit` correctly subtracts `totalAssetDeposits` from the limit and would show a non-zero remaining capacity, giving users and integrators a false sense that the cap is enforced. [4](#0-3) 

### Impact Explanation
The deposit limit is the protocol's primary safety cap on how much ETH can enter `LRTDepositPool`. Bypassing it allows unbounded rsETH minting beyond the intended ceiling, diluting the share value of all existing rsETH holders and exposing the protocol to more EigenLayer slashing risk than governance intended. This matches the "contract fails to deliver promised returns" impact class (Low), with a realistic path to broader accounting desynchronisation if the overshoot is large.

### Likelihood Explanation
No special role or privilege is required. Any user calling `depositETH` with a `msg.value` large enough to exceed the remaining capacity triggers the bypass silently. The condition is met whenever `totalAssetDeposits ≤ depositLimit`, which is the normal operating state of the protocol. The bug is therefore reachable on every ordinary deposit that would otherwise be the limit-crossing deposit.

### Recommendation
Add `amount` to the ETH branch, making it consistent with the ERC20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 1 000 ETH`.
2. Current state: `getTotalAssetDeposits(ETH) = 999 ETH`.
3. Alice calls `depositETH{value: 500 ETH}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH, 500 ETH)` evaluates `999 ETH > 1 000 ETH` → `false`.
5. `_beforeDeposit` does not revert; `_mintRsETH` mints rsETH for 500 ETH.
6. After the call `getTotalAssetDeposits(ETH) = 1 499 ETH` — 499 ETH above the configured cap — with no revert or event indicating the breach. [1](#0-0)
