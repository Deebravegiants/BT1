### Title
`queuedWithdrawalsBuffer` Defaults to Zero, Allowing Instant Withdrawals to Drain Assets Committed to Queued Withdrawals - (File: `contracts/LRTUnstakingVault.sol`)

---

### Summary

The `queuedWithdrawalsBuffer[asset]` mapping in `LRTUnstakingVault` defaults to `0`. When a manager enables instant withdrawal for an asset, the entire vault balance is immediately available for instant withdrawal — including assets already committed to pending queued withdrawal requests. The operator must separately call `setQueuedWithdrawalsBuffer` to reserve assets for queued withdrawals. Until that separate action is taken, instant withdrawal users can drain all vault assets, temporarily freezing funds for queued withdrawal users who have already burned their rsETH.

---

### Finding Description

In `LRTUnstakingVault`, the buffer mapping is declared as:

```solidity
// Portion of the vault reserved for servicing queued withdrawals; unavailable for instant withdrawals.
mapping(address asset => uint256 buffer) public queuedWithdrawalsBuffer;
``` [1](#0-0) 

This defaults to `0` for every asset. The `getAssetsAvailableForInstantWithdrawal` function uses this buffer to compute how much is available for instant withdrawal:

```solidity
uint256 vaultBalance = balanceOf(asset);
uint256 reservedBuffer = queuedWithdrawalsBuffer[asset];
availableAmount = reservedBuffer >= vaultBalance ? 0 : vaultBalance - reservedBuffer;
``` [2](#0-1) 

With `queuedWithdrawalsBuffer[asset] = 0`, `availableAmount = vaultBalance` — the full vault balance is available for instant withdrawal.

When a manager calls `setInstantWithdrawalEnabled(asset, true)`, instant withdrawal is immediately live:

```solidity
modifier onlyInstantWithdrawalAllowed(address asset) {
    if (!isInstantWithdrawalEnabled[asset]) revert InstantWithdrawalNotEnabled();
    _;
}
``` [3](#0-2) 

At this point, `instantWithdrawal` checks:

```solidity
if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
    revert CantInstantWithdrawMoreThanAvailable();
}
unstakingVault.redeem(asset, assetAmountUnlocked);
``` [4](#0-3) 

Since the buffer is `0`, instant withdrawal users can redeem up to the entire vault balance — including assets already committed to queued withdrawal requests (tracked by `assetsCommitted` in `LRTWithdrawalManager`). The `assetsCommitted` mechanism only prevents *new* queued withdrawal requests from over-committing; it does not protect already-committed assets from being drained by instant withdrawals. [5](#0-4) 

When the vault is subsequently drained, the operator's call to `unlockQueue` will revert:

```solidity
if (params.totalAvailableAssets == 0) revert AmountMustBeGreaterThanZero();
``` [6](#0-5) 

because `unstakingVault.balanceOf(asset) = 0`. Queued withdrawal users who have already burned their rsETH cannot complete their withdrawals until the vault is replenished.

The operator can separately set the buffer via:

```solidity
function setQueuedWithdrawalsBuffer(address asset, uint256 buffer) external onlyLRTOperator onlySupportedAsset(asset) {
    queuedWithdrawalsBuffer[asset] = buffer;
    ...
}
``` [7](#0-6) 

But this is a separate transaction from enabling instant withdrawal, creating a window of vulnerability — directly analogous to the Cooler `rollable = true` by default pattern.

---

### Impact Explanation

**Medium — Temporary freezing of funds.**

Queued withdrawal users have already burned their rsETH and are waiting for their withdrawal to be unlocked. If instant withdrawal users drain the vault before the operator sets a non-zero buffer, `unlockQueue` cannot execute, and queued withdrawal users' funds are temporarily frozen. The freeze persists until the protocol replenishes the vault (e.g., by moving assets from a `NodeDelegator`). The rsETH has already been burned, so users cannot reverse their withdrawal request.

---

### Likelihood Explanation

**Medium.** The manager enabling instant withdrawal and the operator setting the buffer are two separate roles performing two separate transactions. In practice, the manager may enable instant withdrawal as part of a feature rollout without the operator having pre-configured the buffer. Any rsETH holder can immediately exploit the zero-buffer window by calling `instantWithdrawal` to drain the vault before the operator acts.

---

### Recommendation

Require the buffer to be set atomically when enabling instant withdrawal, or initialize `queuedWithdrawalsBuffer` to a safe non-zero default. For example, modify `setInstantWithdrawalEnabled` to accept a `buffer` parameter:

```solidity
function setInstantWithdrawalEnabled(address asset, bool enabled, uint256 buffer) external onlySupportedAsset(asset) onlyLRTManager {
    isInstantWithdrawalEnabled[asset] = enabled;
    if (enabled) {
        ILRTUnstakingVault(...).setQueuedWithdrawalsBuffer(asset, buffer);
    }
    emit InstantWithdrawalEnabledUpdated(asset, enabled);
}
```

This ensures the buffer is always configured before instant withdrawal goes live, eliminating the window where the vault can be drained.

---

### Proof of Concept

1. Protocol has 100 stETH in `LRTUnstakingVault`. Users A, B, C have queued withdrawal requests totalling 80 stETH (already burned rsETH, `assetsCommitted[stETH] = 80e18`).
2. Manager calls `setInstantWithdrawalEnabled(stETH, true)`. `queuedWithdrawalsBuffer[stETH]` is still `0`.
3. Attacker (holding rsETH) calls `instantWithdrawal(stETH, largeAmount)`. `getAssetsAvailableForInstantWithdrawal(stETH)` returns `100e18` (full vault balance). Attacker drains 100 stETH from the vault.
4. Operator calls `unlockQueue(stETH, ...)`. `unstakingVault.balanceOf(stETH) = 0` → `params.totalAvailableAssets = 0` → reverts with `AmountMustBeGreaterThanZero`.
5. Users A, B, C cannot complete their withdrawals. Their rsETH is already burned. Funds are temporarily frozen until the vault is replenished.
