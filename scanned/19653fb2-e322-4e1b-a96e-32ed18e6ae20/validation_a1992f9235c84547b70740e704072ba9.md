### Title
Missing `onDropMessage` Hook and `receive()` in `ScrollMessenger` Causes Permanent ETH Freeze on Bridge Failure - (File: `contracts/bridges/ScrollMessenger.sol`)

---

### Summary

`ScrollMessenger` acts as an intermediary that calls the Scroll L2 bridge's `sendMessage`, making itself the `_from` address recorded by the bridge. If the bridged message is later dropped by the Scroll protocol's drop mechanism, the refund is sent back to `ScrollMessenger`. Because `ScrollMessenger` implements neither `onDropMessage` nor a `receive()` / `fallback()` function, the refund reverts and the ETH is permanently frozen inside the Scroll bridge.

---

### Finding Description

When a pool contract (e.g., `RSETHPoolV3ExternalBridge`, `RSETHPoolNoWrapper`, `RSETHPoolV2ExternalBridge`) bridges accumulated ETH to L1 via the Scroll native bridge, it calls:

```solidity
// RSETHPoolV3ExternalBridge.sol:659
IL2Messenger(messenger).sendETHToL1ViaBridge{ value: ethBalanceMinusFees }(
    l2Bridge, l1VaultETHForL2Chain, ethBalanceMinusFees
);
```

`ScrollMessenger.sendETHToL1ViaBridge` then calls the Scroll L2 bridge directly:

```solidity
// ScrollMessenger.sol:23
IScrollMessenger(l2bridge).sendMessage{ value: value }(target, value, "", 0, msg.sender);
```

The caller of `l2bridge.sendMessage` is `ScrollMessenger` itself, so the Scroll bridge records `ScrollMessenger` as `_from`. The fifth argument (`msg.sender`, i.e., the pool contract) is only the `refundAddress` hint, not the authoritative `_from`.

The Scroll bridge's drop mechanism (triggered when a message cannot be relayed on L1) refunds ETH to `_from`. Because the `message` payload is empty (`""`), the bridge performs a plain ETH transfer to `_from` (i.e., `ScrollMessenger`). `ScrollMessenger` has no `receive()` or `fallback()` function and does not implement `onDropMessage`, so the refund call reverts. The ETH remains permanently locked in the Scroll bridge with no recovery path. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

ETH deposited by users into the L2 pool is held in the pool contract until the `BRIDGER_ROLE` calls `bridgeAssetsViaNativeBridge`. At that point the ETH leaves the pool and enters the Scroll bridge. If the corresponding L1 message is dropped (e.g., because `L1Vault.receive()` reverts, the L1Vault is paused, or the message exceeds the Scroll bridge's relay window), the ETH cannot be recovered. Users have already received `wrsETH` / `rsETH` for their deposits, so the protocol becomes insolvent: the minted tokens are no longer fully backed by on-chain ETH.

---

### Likelihood Explanation

**Medium.** The `bridgeAssetsViaNativeBridge` path is a normal operational flow executed by the protocol's bridger. A message drop requires the L1 relay to fail — plausible scenarios include: the `L1Vault` being paused or upgraded between the L2 send and the L1 relay, a revert in `L1Vault.receive()`, or the message exceeding the Scroll bridge's maximum relay window. None of these require attacker action; they are ordinary operational edge cases.

---

### Recommendation

1. Add a `receive()` function to `ScrollMessenger` so that plain-ETH refunds from the Scroll bridge are accepted, and forward the received ETH back to the pool or to a designated recovery address.
2. Implement the `IMessageDropCallback.onDropMessage(bytes calldata message)` interface in `ScrollMessenger` (or in the pool contracts) so that the Scroll bridge's drop hook can execute successfully and return ETH to the correct recipient.
3. Alternatively, pass the pool contract address as the explicit `_from` by having the pool call the Scroll bridge directly (as the Scroll fix PR #1093 did — reverting the bypass and routing through the gateway that already implements `onDropMessage`).

---

### Proof of Concept

1. User calls `RSETHPoolV3ExternalBridge.deposit{value: 10 ether}("ref")` → receives `wrsETH`, pool holds 10 ETH.
2. Bridger calls `RSETHPoolV3ExternalBridge.bridgeAssetsViaNativeBridge()`.
3. Pool calls `ScrollMessenger.sendETHToL1ViaBridge{value: 10 ether}(l2Bridge, l1Vault, 10 ether)`.
4. `ScrollMessenger` calls `L2ScrollMessenger.sendMessage{value: 10 ether}(l1Vault, 10 ether, "", 0, pool)` — `_from` = `ScrollMessenger`.
5. L1 relay fails (e.g., `L1Vault` is paused). Scroll bridge operator calls `dropMessage(ScrollMessenger, l1Vault, 10 ether, nonce, "")`.
6. Scroll bridge attempts `ScrollMessenger.call{value: 10 ether}("")` — reverts (no `receive()`).
7. Drop transaction reverts; 10 ETH is permanently locked in the Scroll bridge. `wrsETH` holders are left with unbacked tokens. [5](#0-4) [6](#0-5)

contract ScrollMessenger is IL2Messenger, ReentrancyGuard {
    /**
     * @notice Bridge ETH from Scroll L2 to Ethereum Mainnet
     * @param l2bridge The address of the L2 bridge on Scroll
     * @param target The address of the target contract on L1
     * @param value The amount of ETH to send
     * @dev Gas limit is set to 0 to use the default gas limit
     */
    function sendETHToL1ViaBridge(address l2bridge, address target, uint256 value) external payable nonReentrant {
        if (msg.value != value) revert MismatchedMsgValue();
        IScrollMessenger(l2bridge).sendMessage{ value: value }(target, value, "", 0, msg.sender);
    }

    function bridgeAssetsViaNativeBridge(uint256 amount) external nonReentrant onlyRole(BRIDGER_ROLE) {
        UtilLib.checkNonZeroAddress(l2Bridge);
        UtilLib.checkNonZeroAddress(messenger);
        UtilLib.checkNonZeroAddress(l1VaultETHForL2Chain);

        if (amount == 0) revert InvalidAmount();

        // bridge up to the ETH balance minus fees
        uint256 ethBalanceMinusFees = getETHBalanceMinusFees();
        if (amount > ethBalanceMinusFees) revert InsufficientETHBalance();

        IL2Messenger(messenger).sendETHToL1ViaBridge{ value: amount }(l2Bridge, l1VaultETHForL2Chain, amount);

        emit BridgedETHToL1ViaNativeBridge(l1VaultETHForL2Chain, amount);
