### Title
Stale `rsETHPrice` Used for Deposit Minting Allows Over-Minting of rsETH at Expense of Existing Holders - (`contracts/LRTOracle.sol` / `contracts/LRTDepositPool.sol`)

---

### Summary

`LRTOracle.rsETHPrice` is a stored, manually-updated value. `LRTDepositPool.getRsETHAmountToMint()` divides by this stored price to determine how many rsETH tokens to mint for a depositor. When the stored price is stale (lower than the actual current value because rewards have accrued but the price has not been updated), any depositor can mint more rsETH than their deposit is worth, diluting existing holders' share of the protocol TVL.

---

### Finding Description

`LRTOracle` stores `rsETHPrice` as a state variable updated only when `updateRSETHPrice()` is explicitly called: [1](#0-0) 

The update function is public but contains a guard that **reverts for non-managers** if the new price exceeds `highestRsethPrice` by more than `pricePercentageLimit`: [2](#0-1) 

When this guard fires, only a privileged manager can push the update through `updateRSETHPriceAsManager()`. Until then, `rsETHPrice` remains at its last stored (stale, lower) value.

`LRTDepositPool.getRsETHAmountToMint()` uses this stored value directly: [3](#0-2) 

The formula is:

```
rsethAmountToMint = (depositAmount × assetPrice) / rsETHPrice
```

`assetPrice` is fetched live from the underlying asset oracle, but `rsETHPrice` is the stale stored value. When actual protocol TVL has grown (staking rewards accrued) but `rsETHPrice` has not been updated, the denominator is artificially low, so the depositor receives more rsETH than their deposit is worth.

The same stale value is consumed by `LRTWithdrawalManager.getExpectedAssetAmount()` and `_createUnlockParams()`: [4](#0-3) [5](#0-4) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

Every rsETH minted at a stale (below-actual) price represents a transfer of value from existing rsETH holders to the new depositor. The new depositor's rsETH represents a larger fraction of the protocol's TVL than their deposit warrants. When `rsETHPrice` is eventually updated upward, the dilution is permanent: existing holders' proportional claim on the TVL has been reduced.

The L2 pools (`RSETHPoolV3`, `RSETHPool`, etc.) and the cross-chain rate receiver also consume `rsETHPrice` via `ILRTOracle(rsETHPriceOracle).rsETHPrice()`: [6](#0-5) [7](#0-6) 

The stale rate is bridged to L2 and used for minting `wrsETH` on every L2 pool deposit, amplifying the over-minting surface across all supported chains.

---

### Likelihood Explanation

**Medium.** Staking rewards accrue continuously. The `pricePercentageLimit` guard (e.g., 1 % = `1e16`) is crossed after roughly 70–125 days of inactivity at typical LST APYs, but can be crossed sooner during periods of elevated rewards or if the manager is slow to act. Once crossed, any public call to `updateRSETHPrice()` reverts with `PriceAboveDailyThreshold`, leaving the price frozen at the stale value. A depositor who monitors the oracle state can exploit the window between the threshold being crossed and the manager's next update.

---

### Recommendation

1. **Compute rsETH price on-the-fly** inside `getRsETHAmountToMint()` and `getExpectedAssetAmount()` using `_getTotalEthInProtocol() / rsETH.totalSupply()` rather than reading the stored `rsETHPrice`, mirroring the OZ `GovernorVotesQuorumFraction` pattern cited in the original report.
2. Alternatively, enforce a **maximum staleness window** (e.g., 24 hours) and revert deposits/withdrawals if `rsETHPrice` has not been updated within that window, forcing the manager to act before users can transact.
3. Separate the fee-minting side-effect (which legitimately needs a discrete update) from the price-reading path used for minting/withdrawal calculations.

---

### Proof of Concept

1. Rewards accrue for several days; actual rsETH price rises from `1.00 ETH` to `1.02 ETH`.
2. A non-manager calls `updateRSETHPrice()`; it reverts with `PriceAboveDailyThreshold` because the 1 % limit is exceeded and `pricePercentageLimit` is set.
3. `rsETHPrice` remains `1.00 ETH` (stale).
4. Attacker calls `LRTDepositPool.depositETH{value: 100 ether}(0, "")`.
5. `getRsETHAmountToMint` computes `100e18 * 1e18 / 1.00e18 = 100 rsETH` instead of the correct `100e18 * 1e18 / 1.02e18 ≈ 98.04 rsETH`.
6. Attacker receives `~1.96 rsETH` more than their deposit warrants, permanently diluting all existing holders.
7. Manager eventually calls `updateRSETHPriceAsManager()`; price jumps to `1.02 ETH`, but the dilution is already baked in. [8](#0-7) [9](#0-8)
