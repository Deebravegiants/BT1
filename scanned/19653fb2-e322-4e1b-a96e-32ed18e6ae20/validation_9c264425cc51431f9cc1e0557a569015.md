### Title
Missing User-Controlled Slippage Protection in L2 Pool `deposit()` Functions - (File: contracts/pools/RSETHPool.sol, contracts/pools/RSETHPoolNoWrapper.sol, contracts/pools/RSETHPoolV2ExternalBridge.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol)

---

### Summary

All four L2 pool contracts expose `deposit()` functions that swap ETH (or supported ERC-20 tokens) for rsETH/wrsETH using a live oracle rate, but accept no `minRsETHAmount` parameter from the caller. A user who submits a deposit transaction cannot enforce a minimum acceptable rsETH output, leaving them exposed to unfavorable rate movements between transaction submission and on-chain execution.

---

### Finding Description

Every L2 pool contract exposes one or two public `deposit()` entry points for unprivileged users:

**RSETHPool** (Arbitrum):
```
function deposit(string memory referralId) external payable   // ETH → rsETH
function deposit(address token, uint256 amount, string memory referralId) external  // token → rsETH
``` [1](#0-0) 

**RSETHPoolNoWrapper** (Unichain / no-wrapper chains):
```
function deposit(string memory referralId) external payable
function deposit(address token, uint256 amount, string memory referralId) external
``` [2](#0-1) 

**RSETHPoolV2ExternalBridge**:
```
function deposit(string memory referralId) external payable
``` [3](#0-2) 

**RSETHPoolV3ExternalBridge**:
```
function deposit(string memory referralId) external payable
function deposit(address token, uint256 amount, string memory referralId) external
``` [4](#0-3) 

In every case the rsETH amount minted is computed entirely from a live oracle rate with no floor enforced by the caller:

```solidity
// RSETHPool / RSETHPoolNoWrapper / RSETHPoolV3ExternalBridge (ETH path)
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;   // rsETHToETHrate from IOracle.getRate()

// token path (RSETHPool / RSETHPoolNoWrapper / RSETHPoolV3ExternalBridge)
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [5](#0-4) 

Neither function signature nor function body contains any `minRsETHAmount` check. The caller has no on-chain mechanism to revert if the oracle rate has moved adversely by the time the transaction is mined.

**Contrast with L1 `LRTDepositPool`**, which correctly accepts and enforces a `minRSETHAmountExpected` parameter in both `depositETH` and `depositAsset`: [6](#0-5) 

The L2 pool contracts are the only user-facing deposit path that lacks this protection.

---

### Impact Explanation

A depositor receives fewer rsETH/wrsETH tokens than they anticipated at the time of signing. Because rsETH is a yield-bearing token, the ETH value is not destroyed, but the user receives a smaller share of the protocol than they intended. This constitutes a failure to deliver the promised return without an actual loss of deposited value.

**Impact: Low** — Contract fails to deliver promised returns, but doesn't lose value.

---

### Likelihood Explanation

Oracle rates (`IOracle.getRate()`) update with every block. On high-throughput L2 chains (Arbitrum, Optimism, Unichain) block times are sub-second to ~2 seconds, meaning the rate seen at signing can differ from the rate at execution. Any period of elevated staking-yield accrual or oracle update causes the rsETH/ETH rate to tick upward, silently reducing the rsETH minted per unit of ETH deposited. No special attacker action is required; ordinary network congestion or a delayed transaction is sufficient.

**Likelihood: High** — Affects every deposit on every L2 pool; rate drift is a normal, continuous occurrence.

---

### Recommendation

Add a `minRsETHAmount` parameter to all `deposit()` overloads in `RSETHPool`, `RSETHPoolNoWrapper`, `RSETHPoolV2ExternalBridge`, and `RSETHPoolV3ExternalBridge`, and revert if the computed `rsETHAmount` falls below it — mirroring the existing pattern in `LRTDepositPool.depositETH` / `depositAsset`.

```solidity
// Example fix for RSETHPoolV3ExternalBridge
function deposit(string memory referralId, uint256 minRsETHAmount)
    external payable nonReentrant whenNotPaused limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();

    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmount) revert InsufficientOutputAmount();

    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
```

Apply the same pattern to the ERC-20 token `deposit()` overloads and to `LRTWithdrawalManager.instantWithdrawal` (which similarly lacks a `minAssetAmount` guard when burning rsETH for an asset). [7](#0-6) 

---

### Proof of Concept

1. Alice calls `RSETHPoolV3ExternalBridge.deposit{value: 1 ether}("ref")` when `rsETHToETHrate = 1.05e18` (rsETH worth 1.05 ETH). Expected rsETH ≈ 0.952.
2. Before Alice's transaction is mined, the oracle updates to `rsETHToETHrate = 1.10e18` (rsETH worth 1.10 ETH).
3. Alice's transaction executes: `rsETHAmount = 1e18 * 1e18 / 1.10e18 ≈ 0.909 rsETH`.
4. Alice receives ~4.5% fewer rsETH tokens than she expected with no on-chain protection and no revert.
5. The same scenario applies to every `deposit()` overload across all four L2 pool contracts. [8](#0-7)

    function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
    }

    /// @dev view function to get the rsETH amount for a given amount of token
    /// @param amount The amount of token
    /// @return rsETHAmount The amount of rsETH that will be received
    /// @return fee The fee that will be charged
    function viewSwapRsETHAmountAndFee(
        uint256 amount,
        address token
    )
        public
        view
        onlySupportedToken(token)
        returns (uint256 rsETHAmount, uint256 fee)
    {
        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // rate of token in ETH
        uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
    }
