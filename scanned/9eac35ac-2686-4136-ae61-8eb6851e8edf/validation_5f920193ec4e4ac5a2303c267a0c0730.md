### Title
Missing Minimum Output Validation in L2 Pool Deposit Functions Exposes Users to Unfavorable Rate Execution - (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPool.sol, contracts/pools/RSETHPoolV2.sol, contracts/pools/RSETHPoolNoWrapper.sol)

---

### Summary
All L2 pool `deposit()` functions lack a `minRSETHAmountExpected` (minimum output) parameter. Users have no on-chain mechanism to enforce a minimum acceptable rsETH amount, leaving them exposed to receiving fewer tokens than anticipated if the oracle rate shifts between transaction submission and execution.

---

### Finding Description
The L1 `LRTDepositPool` correctly implements slippage protection: both `depositETH` and `depositAsset` accept a `minRSETHAmountExpected` argument and revert in `_beforeDeposit` if the computed mint amount falls below it. [1](#0-0) [2](#0-1) 

Every L2 pool deposit function omits this guard entirely:

- `RSETHPoolV3.deposit(string)` and `RSETHPoolV3.deposit(address,uint256,string)` — no `minRSETHAmountExpected` [3](#0-2) [4](#0-3) 

- `RSETHPool.deposit(string)` and `RSETHPool.deposit(address,uint256,string)` — no `minRSETHAmountExpected` [5](#0-4) [6](#0-5) 

- `RSETHPoolV2.deposit(string)` — no `minRSETHAmountExpected` [7](#0-6) 

- `RSETHPoolNoWrapper.deposit(string)` and `RSETHPoolNoWrapper.deposit(address,uint256,string)` — no `minRSETHAmountExpected` [8](#0-7) [9](#0-8) 

The rsETH amount is computed solely from the live oracle rate at execution time: [10](#0-9) 

There is no check that `rsETHAmount` meets any user-specified floor before the mint/transfer executes.

---

### Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A user who previews the exchange rate off-chain (or via `viewSwapRsETHAmountAndFee`) and submits a deposit transaction may receive materially fewer rsETH tokens if the oracle rate updates before their transaction is included. The user's ETH/LST is consumed and they receive rsETH at a rate they did not agree to, with no on-chain recourse. The discrepancy is bounded by the magnitude of the oracle rate movement between submission and inclusion, which under normal conditions is small but non-zero and unpredictable.

---

### Likelihood Explanation
**Medium.** L2 chains (Arbitrum, Optimism, Unichain) have sequencers that can reorder transactions. Oracle rate updates occur regularly as staking rewards accrue and the rsETH/ETH rate rises. Any deposit transaction that is pending in the mempool when an oracle update lands will execute at the new, less favorable rate with no protection. This is a routine, permissionless condition requiring no attacker.

---

### Recommendation
Add a `minRSETHAmountExpected` parameter to all L2 pool `deposit()` functions, mirroring the pattern already used in `LRTDepositPool._beforeDeposit`:

```solidity
// RSETHPoolV3 (and analogously for RSETHPool, RSETHPoolV2, RSETHPoolNoWrapper)
function deposit(string memory referralId, uint256 minRSETHAmountExpected)
    external payable nonReentrant whenNotPaused
    limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    if (!isEthDepositEnabled) revert EthDepositDisabled();
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();

    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);

    if (rsETHAmount < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet();

    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
```

---

### Proof of Concept

1. User calls `viewSwapRsETHAmountAndFee(1 ether)` on `RSETHPoolV3` and sees they will receive `X` rsETH at the current oracle rate.
2. User submits `deposit{value: 1 ether}("ref")`.
3. Before the transaction is included, the rsETH oracle rate is updated (e.g., daily rate accrual), increasing `rsETHToETHrate`.
4. The deposit executes: `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate` yields `X' < X`.
5. User receives `X'` rsETH — less than the amount they observed — with no revert and no recourse.
6. On `LRTDepositPool`, the same scenario would revert with `MinimumAmountToReceiveNotMet` if the user had set `minRSETHAmountExpected = X`, protecting them. No equivalent protection exists in any L2 pool contract.

    function _beforeDeposit(
        address asset,
        uint256 depositAmount,
        uint256 minRSETHAmountExpected
    )
        private
        view
        returns (uint256 rsethAmountToMint)
    {
        if (depositAmount == 0 || depositAmount < minAmountToDeposit) {
            revert InvalidAmountToDeposit();
        }

        if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
            revert MaximumDepositLimitReached();
        }

        rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

        if (rsethAmountToMint < minRSETHAmountExpected) {
            revert MinimumAmountToReceiveNotMet();
        }
    }

    function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
    }
