### Title
Arithmetic Underflow in `bridgeKernelToBSC` When `counter == 0` Blocks All Bridge Operations - (File: `contracts/KERNEL/KernelVaultETH.sol`)

### Summary
`bridgeKernelToBSC` unconditionally computes `lastBridgedDepositId = counter - 1` before bridging. In Solidity 0.8.27, if `counter == 0` this expression underflows and reverts, permanently blocking the bridge path until at least one deposit is recorded via `depositKernel`.

### Finding Description
`counter` is a `uint256` initialized to zero by default and is only incremented inside `_depositKernel`. [1](#0-0) [2](#0-1) 

In `bridgeKernelToBSC`, before calling the OFT adapter, the contract writes:

```solidity
lastBridgedDepositId = counter - 1;   // line 262
``` [3](#0-2) 

There is no guard requiring `counter > 0` before this subtraction. The balance check at line 238 only verifies the contract holds enough KERNEL tokens — it does not ensure any deposit was ever recorded. [4](#0-3) 

**Trigger path:**
1. Anyone transfers KERNEL tokens directly to `KernelVaultETH` (standard ERC-20 `transfer`), bypassing `depositKernel`. `counter` remains `0`.
2. The operator (legitimately) calls `bridgeKernelToBSC` with `amount ≤ kernel.balanceOf(address(this))`.
3. All pre-checks pass (sufficient balance, valid `minAmount`, sufficient `nativeFee`).
4. Execution reaches line 262: `0 - 1` underflows → Solidity 0.8.27 panics with `Panic(0x11)` → revert.
5. The KERNEL tokens held in the vault cannot be bridged.

The same underflow also triggers if the operator calls `bridgeKernelToBSC` before any user has ever deposited, regardless of how tokens arrived.

### Impact Explanation
KERNEL tokens held in the vault are frozen until the underflow condition is resolved. The freeze is **temporary** because a workaround exists (make one deposit via `depositKernel` to advance `counter` to 1), but until that happens the operator cannot bridge any tokens, violating the contract's core invariant that deposited/held KERNEL must be bridgeable to BSC. This maps to **Medium — Temporary freezing of funds**.

### Likelihood Explanation
The scenario is straightforward: any address can send KERNEL tokens directly to the contract at any time. The operator has no way to distinguish this state from a normal state until the bridge call reverts. The contract provides no mechanism to detect or recover from `counter == 0` other than making a new deposit.

### Recommendation
Guard the subtraction or initialize `lastBridgedDepositId` differently:

```solidity
// Option A: guard
if (counter == 0) revert NoDepositsRecorded();
lastBridgedDepositId = counter - 1;

// Option B: use counter directly (set after bridging)
lastBridgedDepositId = counter > 0 ? counter - 1 : 0;
```

Alternatively, initialize `counter` to `1` in `initialize` so that `counter - 1` always yields a valid (zero-based) deposit ID, and adjust `_depositKernel` accordingly.
