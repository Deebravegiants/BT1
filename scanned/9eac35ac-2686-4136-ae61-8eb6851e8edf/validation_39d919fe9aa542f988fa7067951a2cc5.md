### Title
ETH Donation to NodeDelegator Inflates Protocol-Wide ETH Accounting, Triggering Deposit Cap and Blocking New Deposits - (File: contracts/LRTDepositPool.sol)

---

### Summary

`LRTDepositPool.getETHDistributionData()` measures ETH held in NodeDelegator contracts using raw `address.balance`. Because every `NodeDelegator` exposes an unrestricted `receive()` function, any unprivileged caller can donate ETH directly to an NDC, inflating the protocol's reported total ETH deposits above the configured deposit limit and causing every subsequent `depositETH()` call to revert with `MaximumDepositLimitReached`.

---

### Finding Description

`getETHDistributionData()` aggregates ETH across the protocol using raw balance reads:

```solidity
// contracts/LRTDepositPool.sol  line 484-485
for (uint256 i; i < ndcsCount;) {
    ethLyingInNDCs += nodeDelegatorQueue[i].balance;
``` [1](#0-0) 

This value flows directly into `getTotalAssetDeposits(ETH)`, which is consumed by `_checkIfDepositAmountExceedesCurrentLimit()`:

```solidity
// contracts/LRTDepositPool.sol  line 678-679
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
}
``` [2](#0-1) 

`_checkIfDepositAmountExceedesCurrentLimit()` is called inside `_beforeDeposit()`, which gates both `depositETH()` and `depositAsset()`:

```solidity
// contracts/LRTDepositPool.sol  line 661-663
if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
    revert MaximumDepositLimitReached();
}
``` [3](#0-2) 

Every `NodeDelegator` accepts arbitrary ETH via an unrestricted `receive()`:

```solidity
// contracts/NodeDelegator.sol  line 81-83
receive() external payable {
    emit ETHReceived(msg.sender, msg.value);
}
``` [4](#0-3) 

An attacker who sends even a small amount of ETH directly to any NDC permanently inflates `ethLyingInNDCs`, and therefore `getTotalAssetDeposits(ETH)`. If the inflated total crosses `depositLimitByAsset`, every call to `depositETH()` reverts.

This is the direct EVM analog of the Solana report: the Solana code checked `lamports()` (manipulable by sending lamports) instead of `data_len` (not manipulable) to determine account state. Here, the code checks `address.balance` (manipulable by sending ETH) instead of a protocol-internal accounting variable to determine whether the deposit cap has been reached.

---

### Impact Explanation

**Medium — Temporary freezing of ETH deposits.**

All new `depositETH()` calls revert with `MaximumDepositLimitReached` for as long as the inflated balance keeps `totalAssetDeposits > depositLimitByAsset`. The donated ETH is permanently absorbed into the protocol (it cannot be extracted by the attacker), so the freeze persists until an admin raises the deposit limit. Existing rsETH holders and withdrawal flows are unaffected; only new deposits are blocked.

---

### Likelihood Explanation

**Low-Medium.** The attack is permissionless and requires no special role. The only cost to the attacker is the ETH donated (permanently lost). Near a deposit cap, even a 1 wei donation suffices. The attack is most attractive when the protocol is close to its cap and the attacker wishes to prevent competitors from depositing (e.g., to capture a points/reward window). An admin can mitigate by raising the cap, but the window of disruption is real.

---

### Recommendation

Replace raw `address.balance` reads in `getETHDistributionData()` with an internal accounting variable that is incremented only through the protocol's controlled deposit and transfer paths (analogous to checking `data_len` instead of `lamports`). For example, maintain a `mapping(address ndc => uint256 ethTracked)` updated in `transferETHToNodeDelegator` and decremented when ETH leaves the NDC through protocol-sanctioned paths. Alternatively, emit and track ETH movements via events and reconcile off-chain, or add a `maxNegligibleAmount`-style tolerance that ignores small unsolicited donations.

---

### Proof of Concept

1. Protocol has `depositLimitByAsset[ETH] = 10_000 ether` and current `getTotalAssetDeposits(ETH) = 9_999.9 ether`.
2. Attacker calls:
   ```solidity
   (bool ok,) = nodeDelegatorQueue[0].call{value: 0.2 ether}("");
   ```
   `NodeDelegator.receive()` accepts the ETH and emits `ETHReceived`.
3. `getETHDistributionData()` now returns `ethLyingInNDCs` inflated by `0.2 ether`, making `getTotalAssetDeposits(ETH) = 10_000.1 ether`.
4. Any user calling `depositETH(minRSETH, ref)` triggers:
   - `_beforeDeposit` → `_checkIfDepositAmountExceedesCurrentLimit(ETH, amount)`
   - `totalAssetDeposits (10_000.1 ether) > depositLimitByAsset (10_000 ether)` → `true`
   - Reverts with `MaximumDepositLimitReached`.
5. ETH deposits are frozen until an admin raises the limit. [5](#0-4) [6](#0-5)

    function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
        uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
        if (asset == LRTConstants.ETH_TOKEN) {
