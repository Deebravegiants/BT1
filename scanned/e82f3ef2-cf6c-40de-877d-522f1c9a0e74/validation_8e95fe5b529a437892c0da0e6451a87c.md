### Title
If RSETH is paused, `LRTWithdrawalManager.unlockQueue()` will revert, temporarily freezing user funds in the withdrawal queue - (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary

`RSETH.burnFrom()` carries a `whenNotPaused` modifier. `LRTWithdrawalManager.unlockQueue()` calls `burnFrom` unconditionally with no try/catch. If RSETH is paused while users have already deposited rsETH into the withdrawal manager via `initiateWithdrawal()`, the queue cannot be processed and those user funds are temporarily frozen.

---

### Finding Description

The withdrawal lifecycle in LRT-rsETH is two-step:

1. **`initiateWithdrawal()`** — the user transfers rsETH from their wallet into `LRTWithdrawalManager` and a withdrawal request is queued.
2. **`unlockQueue()`** — an operator burns the queued rsETH and redeems the underlying assets from `LRTUnstakingVault`, making them claimable by users via `completeWithdrawal()`.

`RSETH.burnFrom()` is gated by `whenNotPaused`:

```solidity
// contracts/RSETH.sol:245
function burnFrom(address account, uint256 amount) external onlyRole(LRTConstants.BURNER_ROLE) whenNotPaused {
    _enforceNotBlocked(account);
    _burn(account, amount);
}
```

`LRTWithdrawalManager.unlockQueue()` calls this function directly with no error handling:

```solidity
// contracts/LRTWithdrawalManager.sol:305
if (rsETHBurned != 0) IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned);
```

If RSETH is paused at the time `unlockQueue()` is called, the `burnFrom` call reverts and the entire `unlockQueue()` transaction fails. Because rsETH was already transferred into `LRTWithdrawalManager` during `initiateWithdrawal()`, those tokens are stuck in the contract and users cannot progress their withdrawals.

Notably, the same function already uses a try/catch pattern for the Aave deposit step (lines 311–316), demonstrating the team is aware of the pattern — but it was not applied to the `burnFrom` call.

---

### Impact Explanation

**Medium — Temporary freezing of funds.**

Users who have already called `initiateWithdrawal()` have their rsETH held inside `LRTWithdrawalManager`. While RSETH is paused, `unlockQueue()` cannot execute, so:
- No withdrawal requests can be unlocked.
- `completeWithdrawal()` cannot be reached for any new requests.
- The underlying assets (ETH, stETH, ETHx, etc.) remain locked in `LRTUnstakingVault`.

The freeze lasts until RSETH is unpaused. If the pause is prolonged (e.g., during a security incident), users are unable to exit their positions for an extended period.

---

### Likelihood Explanation

**Medium.** RSETH and `LRTWithdrawalManager` have independent pause mechanisms controlled by the same `PAUSER_ROLE`. An emergency scenario — such as a discovered vulnerability in the rsETH token contract, a regulatory freeze, or a bridge exploit — could require pausing RSETH while the withdrawal manager remains operational. At any given time there will be pending withdrawal requests in the queue (the withdrawal delay is up to 8–16 days), making the overlap highly probable during any non-trivial pause event.

---

### Recommendation

Wrap the `burnFrom` call in a try/catch block, consistent with the pattern already used for the Aave deposit in the same function. If the burn fails, the rsETH remains in the contract and can be swept or retried once RSETH is unpaused:

```diff
- if (rsETHBurned != 0) IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned);
+ if (rsETHBurned != 0) {
+     try IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned) {}
+     catch {
+         // burnFrom failed (e.g. RSETH paused); rsETH remains in contract
+         // assets are still redeemed so users can withdraw; rsETH can be burned later
+     }
+ }
```

Alternatively, check whether RSETH is paused before calling `burnFrom` and revert with a descriptive error so operators know to retry after unpausing.

---

### Proof of Concept

1. Alice calls `initiateWithdrawal(stETH, 1e18, "")`. Her 1 rsETH is transferred to `LRTWithdrawalManager`. [1](#0-0) 

2. A security incident causes the `PAUSER_ROLE` holder to call `RSETH.pause()`. [2](#0-1) 

3. An operator calls `unlockQueue(stETH, ...)`. The function reaches line 305 and calls `IRSETH.burnFrom(address(this), rsETHBurned)`. [3](#0-2) 

4. `burnFrom` reverts because `whenNotPaused` is active. [4](#0-3) 

5. The entire `unlockQueue()` transaction reverts. Alice's rsETH remains stuck in `LRTWithdrawalManager` and she cannot complete her withdrawal. The underlying stETH remains locked in `LRTUnstakingVault`. This persists for the entire duration of the RSETH pause. [5](#0-4)

```text
    function unlockQueue(
        address asset,
        uint256 firstExcludedIndex,
        uint256 minimumAssetPrice,
        uint256 minimumRsEthPrice,
        uint256 maximumAssetPrice,
        uint256 maximumRsEthPrice
    )
        external
        nonReentrant
        onlySupportedAsset(asset)
        whenNotPaused
        onlyAssetTransferOrOperatorRole
        returns (uint256 rsETHBurned, uint256 assetAmountUnlocked)
    {
        ILRTOracle lrtOracle = ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE));
        ILRTUnstakingVault unstakingVault = ILRTUnstakingVault(lrtConfig.getContract(LRTConstants.LRT_UNSTAKING_VAULT));

        UnlockParams memory params = _createUnlockParams(lrtOracle, unstakingVault, asset);

        _validatePrices(
            params.rsETHPrice,
            params.assetPrice,
            minimumRsEthPrice,
            maximumRsEthPrice,
            minimumAssetPrice,
            maximumAssetPrice
        );

        if (params.totalAvailableAssets == 0) revert AmountMustBeGreaterThanZero();

        // Updates and unlocks withdrawal requests up to a specified upper limit or until allocated assets are fully
        // utilized.
        (rsETHBurned, assetAmountUnlocked) = _unlockWithdrawalRequests(
            asset, params.totalAvailableAssets, params.rsETHPrice, params.assetPrice, firstExcludedIndex
        );

        if (rsETHBurned != 0) IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned);
        //Take the amount to distribute from vault
        unstakingVault.redeem(asset, assetAmountUnlocked);

        // If Aave integration is enabled and asset is ETH, deposit to Aave
        if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN && assetAmountUnlocked > 0) {
            try this.depositToAaveExternal(assetAmountUnlocked) { }
            catch (bytes memory reason) {
                emit AaveDepositFailed(assetAmountUnlocked, reason);
                // Silently fail if Aave deposit fails (e.g., pool at max capacity)
                // Funds remain in contract for withdrawals
            }
        }

        emit AssetUnlocked(asset, rsETHBurned, assetAmountUnlocked, params.rsETHPrice, params.assetPrice);
    }
```
