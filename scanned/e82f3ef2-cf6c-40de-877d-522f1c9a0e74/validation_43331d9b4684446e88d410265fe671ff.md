### Title
Stale `unlockedWithdrawalsCount` Seeding in `initialize2` Causes Temporary Withdrawal Freeze and Permanent `sweepRemainingAssets` Block — (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTWithdrawalManager.initialize2` seeds the newly-introduced `unlockedWithdrawalsCount` mapping with values determined off-chain. Because `initialize2` is gated by the separate `UNLOCKED_WITHDRAWAL_INITIALIZER` role and is not required to execute atomically with the proxy upgrade, a window exists during which:

1. Any user with a pre-existing unlocked withdrawal who calls `completeWithdrawal` will revert due to an underflow at `unlockedWithdrawalsCount[asset]--` (the counter is still 0), temporarily freezing their funds.
2. If withdrawals are completed between the moment the admin reads the on-chain state to compute the seed values and the moment `initialize2` is mined, the seeded count is stale (too high), permanently preventing `sweepRemainingAssets` from executing.

---

### Finding Description

`LRTWithdrawalManager` was upgraded to introduce `unlockedWithdrawalsCount` as a per-asset counter of withdrawals that have been unlocked but not yet claimed. For existing deployments that already had unlocked-but-unclaimed requests, `initialize2` is provided to retroactively seed those counts: [1](#0-0) 

The function is callable only by `UNLOCKED_WITHDRAWAL_INITIALIZER`, a role distinct from the upgrade admin, meaning it is not guaranteed to execute in the same transaction as the proxy upgrade. This creates two related problems rooted in the same cause — state changes that occur during the upgrade window are not accounted for.

**Problem 1 — Underflow revert (temporary freeze).**
After the new implementation is live but before `initialize2` is called, `unlockedWithdrawalsCount[asset]` is `0` for all assets (Solidity default). `_processWithdrawalCompletion`, called by both `completeWithdrawal` and `completeWithdrawalForUser`, unconditionally decrements this counter: [2](#0-1) 

Solidity 0.8.x reverts on unsigned integer underflow. Any user whose withdrawal was already unlocked (i.e., `nonce < nextLockedNonce[asset]`) and who calls `completeWithdrawal` during this window will have their transaction revert, even though their request is fully eligible. Their rsETH has already been transferred to the contract in `initiateWithdrawal` and cannot be recovered until `initialize2` is called.

**Problem 2 — Stale seed (permanent block of `sweepRemainingAssets`).**
The admin must read the current count of unlocked-but-unclaimed requests off-chain and pass those values to `initialize2`. If any user completes a withdrawal between the moment the admin reads the state and the moment `initialize2` is mined, the seeded value is higher than the true count. Because `initialize2` can only be called once (`reinitializer(2)`) and there is no other mechanism to correct `unlockedWithdrawalsCount`, the counter will never reach zero after all real unlocked withdrawals are claimed. `hasUnlockedWithdrawals` will permanently return `true`: [3](#0-2) 

This permanently blocks `sweepRemainingAssets`: [4](#0-3) 

The same pattern is repeated in `initialize3` for sfrxETH: [5](#0-4) 

---

### Impact Explanation

**Problem 1:** Temporary freezing of user funds. Users with pre-existing unlocked withdrawals cannot claim their assets (rsETH already burned/held by the contract) for the entire duration between the proxy upgrade and the `initialize2` call. The freeze is resolved only when `initialize2` executes, but the duration is unbounded and depends on the operational process.

**Problem 2:** Permanent freezing of unclaimed yield. If even one withdrawal is completed in the window between the admin's off-chain read and `initialize2` execution, `unlockedWithdrawalsCount` is permanently inflated. After all real unlocked withdrawals are claimed, the counter remains non-zero, and `sweepRemainingAssets` is irrecoverably blocked, trapping any residual asset balance in the contract.

---

### Likelihood Explanation

The `UNLOCKED_WITHDRAWAL_INITIALIZER` role is explicitly separate from the upgrade admin role, as confirmed by its distinct constant in `LRTConstants.sol`. This architectural separation makes it operationally likely that `initialize2` is submitted as a separate transaction after the upgrade, creating a real window. The protocol operates on Ethereum mainnet where block times are ~12 seconds; even a single block gap is sufficient for a user to complete a withdrawal. Given that the protocol has active users with pending unlocked withdrawals (the very reason `initialize2` exists), the probability of at least one completion occurring in the window is non-trivial.

---

### Recommendation

**Short term:** Pause `LRTWithdrawalManager` (via `PAUSER_ROLE`) before deploying the new implementation, and unpause only after `initialize2` has been successfully called. This eliminates the window entirely.

**Long term:** Encode `initialize2` as the `data` argument of `upgradeToAndCall` so the seeding executes atomically with the implementation swap. Alternatively, add a guard in `_processWithdrawalCompletion` that skips the decrement (or uses a saturating subtraction) when `unlockedWithdrawalsCount[asset] == 0`, preventing the underflow revert while the counter is being bootstrapped.

---

### Proof of Concept

```
1. Admin deploys new LRTWithdrawalManager implementation (adds unlockedWithdrawalsCount).
2. Proxy is upgraded to the new implementation via upgradeTo().
   → unlockedWithdrawalsCount[ETH] == 0 (default).
   → There exist N unlocked withdrawal requests (nonce < nextLockedNonce[ETH]).
3. Alice (unprivileged user) calls completeWithdrawal(ETH_TOKEN, "ref").
4. _processWithdrawalCompletion executes:
   - userAssociatedNonces check passes (Alice has a request).
   - nonce < nextLockedNonce check passes (request is unlocked).
   - withdrawalDelayBlocks check passes (delay has elapsed).
   - unlockedWithdrawalsCount[ETH]-- → 0 - 1 → REVERT (underflow).
5. Alice's withdrawal is frozen. Her rsETH remains locked in the contract.
6. Admin later calls initialize2(N_ETHx, N_STETH, N_ETH).
   → If N_ETH was read before step 3, it is now stale (one too high).
   → After all remaining N-1 real unlocked withdrawals are claimed,
      unlockedWithdrawalsCount[ETH] == 1 (never reaches 0).
   → sweepRemainingAssets permanently reverts with PendingWithdrawalsExist().
```
