Looking at the code carefully, I need to trace the full accounting path to determine if the described invariant break is real.

**Step 1: What happens when an LST enters the converter?**

In `transferAssetFromDepositPool`, `ethValueInWithdrawal` is set at the price at the moment of transfer: [1](#0-0) 

**Step 2: How is the converter value reflected in distribution data?**

For LST assets, `getAssetDistributionData` explicitly zeroes out the converter portion: [2](#0-1) 

For ETH, `getETHDistributionData` reads the stored (stale) `ethValueInWithdrawal`: [3](#0-2) 

**Step 3: How does `_getTotalEthInProtocol()` aggregate this?**

It iterates all supported assets (including `ETH_TOKEN`) and multiplies `getTotalAssetDeposits(asset) * getAssetPrice(asset)`: [4](#0-3) 

For `ETH_TOKEN`, `getAssetPrice(ETH_TOKEN) = 1e18`, so the ETH contribution from the converter is exactly `ethValueInWithdrawal`. For the LST, the converter portion contributes 0.

**Step 4: The staleness discrepancy**

The total ETH in protocol becomes:

```
(LST not in converter) * P_LST_current + ETH + ethValueInWithdrawal
```

where `ethValueInWithdrawal = LST_amount_in_converter * P_LST_at_transfer_time / 1e18`.

The correct value should be:

```
(LST not in converter) * P_LST_current + ETH + LST_amount_in_converter * P_LST_current / 1e18
```

The difference is `LST_amount_in_converter * (P_LST_current - P_LST_at_transfer_time) / 1e18`.

**Step 5: Impact on `getRsETHAmountToMint`**

`getRsETHAmountToMint` uses the stored `rsETHPrice`: [5](#0-4) 

`rsETHPrice` is computed via `_updateRsETHPrice()` → `_getTotalEthInProtocol()`: [6](#0-5) 

If the LST price **decreases** after transfer to converter, `ethValueInWithdrawal` is **higher** than the current ETH value of the LST in the converter → `_getTotalEthInProtocol()` is **overstated** → `rsETHPrice` is **overstated** → new depositors receive **fewer rsETH** than their collateral's fair ETH value.

**Step 6: Is this a real, reachable path?**

Yes. The operator calls `transferAssetFromDepositPool` (role-gated, but this is the intended operational flow, not an attack). The LST then awaits unstaking (days for stETH via Lido). During this window, the LST price can decrease. When `updateRSETHPrice()` is called (public, permissionless), the stale `ethValueInWithdrawal` inflates `rsETHPrice`. Subsequent depositors of the same or any LST receive fewer rsETH than fair value.

**Step 7: Is there a guard that prevents this?**

No. `ethValueInWithdrawal` is only reduced when ETH is actually sent back via `_sendEthToDepositPool`: [7](#0-6) 

There is no mechanism to re-price `ethValueInWithdrawal` if the LST price changes while awaiting unstaking.

**Conclusion: This is a valid Low severity finding.**

### Title
Stale `ethValueInWithdrawal` in `LRTConverter` causes `rsETHPrice` overstatement when LST price falls during unstaking, shortchanging subsequent depositors — (`contracts/LRTConverter.sol` / `contracts/LRTDepositPool.sol`)

### Summary
When an LST is transferred to `LRTConverter` for unstaking, its ETH value is snapshotted into `ethValueInWithdrawal` at the transfer-time oracle price. If the LST price falls before the unstaking completes, `ethValueInWithdrawal` overstates the converter's contribution to protocol TVL, inflating `rsETHPrice`, and causing subsequent depositors to receive fewer rsETH than their collateral's fair ETH value.

### Finding Description
`transferAssetFromDepositPool` records `ethValueInWithdrawal += (_amount * lrtOracle.getAssetPrice(_asset)) / 1e18` at transfer time. [8](#0-7) 

`getAssetDistributionData` for LST assets returns `assetLyingInConverter = 0`, delegating the converter's value entirely to `getETHDistributionData()`. [9](#0-8) 

`getETHDistributionData` returns the stored (stale) `ethValueInWithdrawal` as `ethLyingInConverter`. [3](#0-2) 

`_getTotalEthInProtocol()` sums `getTotalAssetDeposits(asset) * getAssetPrice(asset)` for all assets. For `ETH_TOKEN`, this includes `ethValueInWithdrawal * 1e18 / 1e18 = ethValueInWithdrawal`. For the LST, the converter portion contributes 0. [4](#0-3) 

`rsETHPrice` is then set to `totalETHInProtocol / rsETH supply`. [10](#0-9) 

`getRsETHAmountToMint` divides by this stored `rsETHPrice`. [5](#0-4) 

If the LST price drops from P1 (at transfer) to P2 < P1 (at `updateRSETHPrice()` call), the overstatement is `LST_amount_in_converter * (P1 - P2) / 1e18`. This inflates `rsETHPrice`, and subsequent depositors receive fewer rsETH than their collateral's fair ETH value.

### Impact Explanation
New LST depositors receive fewer rsETH than the fair ETH value of their deposit during the window when an LST is in the converter and its price has declined. The protocol does not lose funds (existing holders benefit), but the contract fails to deliver the promised exchange rate to new depositors. This matches the Low impact: "Contract fails to deliver promised returns, but doesn't lose value."

### Likelihood Explanation
The operator regularly transfers LSTs to the converter for unstaking (this is the intended operational flow). Lido stETH unstaking takes days. LST prices can and do fluctuate. `updateRSETHPrice()` is public and can be called at any time. The combination is routine and requires no adversarial action.

### Recommendation
Instead of storing the ETH value at transfer time, store the raw LST amount in the converter and compute its current ETH value dynamically in `getETHDistributionData()` using the live oracle price. Alternatively, update `ethValueInWithdrawal` on each `updateRSETHPrice()` call to reflect the current oracle price of the LST held in the converter.

### Proof of Concept
```solidity
// 1. 1000 stETH transferred to converter at price 1.05 ETH/stETH
//    ethValueInWithdrawal = 1000 * 1.05e18 / 1e18 = 1050 ETH

// 2. stETH price drops to 1.00 ETH/stETH

// 3. updateRSETHPrice() is called:
//    _getTotalEthInProtocol() includes ethValueInWithdrawal = 1050 ETH
//    (correct value should be 1000 ETH)
//    rsETHPrice is overstated by 50 ETH / rsETH_supply

// 4. Alice deposits 100 stETH at current price 1.00 ETH/stETH:
//    getRsETHAmountToMint = 100 * 1.00e18 / rsETHPrice_overstated
//    Alice receives fewer rsETH than 100 * 1.00e18 / rsETHPrice_correct

// Invariant test:
// assert: sum(getTotalAssetDeposits(asset) * getAssetPrice(asset)) 
//      == actual ETH value of all protocol-held collateral
// This fails when ethValueInWithdrawal != LST_in_converter * current_price
```
