### Title
No Staleness Validation on Chainlink `latestRoundData()` Allows Stale Prices to Silently Corrupt rsETH Minting and Withdrawal Accounting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls Chainlink's `latestRoundData()` but performs **zero validation** on the returned data — no staleness timeout, no `answeredInRound < roundId` check, no `updatedAt == 0` guard, and no negative-price guard. If Chainlink's aggregator for any supported LST (stETH, ETHx, sfrxETH, etc.) stops updating, the protocol silently consumes the stale price in every deposit, withdrawal, and rsETH price update, leading to incorrect rsETH minting and potential protocol insolvency.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` is the sole Chainlink price adapter used by `LRTOracle` for all supported LST collateral assets:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L49-L55
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (, int256 price,,,) = priceFeed.latestRoundData();   // ← all return values except price discarded

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
``` [1](#0-0) 

The five return values of `latestRoundData()` are `(roundId, answer, startedAt, updatedAt, answeredInRound)`. The implementation discards `roundId`, `startedAt`, `updatedAt`, and `answeredInRound` entirely. There is no check of the form:

- `block.timestamp - updatedAt > TIMEOUT` (staleness)
- `answeredInRound < roundId` (incomplete round)
- `price <= 0` (invalid/negative price)
- `updatedAt == 0` (uninitialized round)

This stale price then propagates through two critical paths:

**Path 1 — rsETH minting** (`LRTDepositPool.getRsETHAmountToMint`):
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [2](#0-1) 

**Path 2 — rsETH price update** (`LRTOracle._getTotalEthInProtocol`):
```solidity
uint256 assetER = getAssetPrice(asset);
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
``` [3](#0-2) 

**Path 3 — withdrawal sizing** (`LRTWithdrawalManager.getExpectedAssetAmount`):
```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
``` [4](#0-3) 

---

### Impact Explanation

**Protocol insolvency (Critical).** If Chainlink's aggregator for a supported LST becomes stale while the real market price has dropped significantly:

1. `getAssetPrice()` returns the old (inflated) price.
2. `_getTotalEthInProtocol()` overstates the protocol's ETH backing.
3. `updateRSETHPrice()` sets `rsETHPrice` too high.
4. New depositors receive fewer rsETH than they deserve (value theft from depositors).
5. Existing holders redeem rsETH at the inflated price, draining more LST than the protocol actually holds.
6. The protocol becomes insolvent as the real collateral value is less than the rsETH liabilities.

Conversely, if the stale price is lower than reality, depositors receive excess rsETH, diluting all existing holders.

---

### Likelihood Explanation

Chainlink aggregators have historically paused or lagged during extreme market volatility (e.g., the March 2020 ETH crash, the LUNA collapse). The LSTs supported by this protocol (stETH, ETHx, sfrxETH) each have their own Chainlink feed, and any one of them going stale triggers the vulnerability. No privileged action is required — any depositor or withdrawer calling the public deposit/withdrawal functions during a stale-price window triggers the mispricing.

---

### Recommendation

Add staleness and validity checks to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already present in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > STALENESS_TIMEOUT) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`STALENESS_TIMEOUT` should be set per-feed based on each Chainlink feed's documented heartbeat (e.g., 3600 s for ETH/USD, 86400 s for LST feeds).

---

### Proof of Concept

1. Chainlink's stETH/ETH aggregator stops updating (e.g., due to a network incident). The last reported price is `1.05e18` (stETH at a 5% premium to ETH).
2. The real stETH price drops to `0.90e18` due to a slashing event, but the feed is frozen.
3. A user calls `LRTDepositPool.depositAsset(stETH, 100e18)`.
4. `getRsETHAmountToMint` calls `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns `1.05e18` (stale).
5. The user receives rsETH priced as if they deposited 105 ETH worth of value, when they only deposited 90 ETH worth.
6. `updateRSETHPrice()` similarly inflates `rsETHPrice` using the stale `1.05e18` rate.
7. Existing rsETH holders redeem at the inflated rate, withdrawing more stETH than the protocol can cover.
8. Protocol becomes insolvent. [1](#0-0) [5](#0-4)
