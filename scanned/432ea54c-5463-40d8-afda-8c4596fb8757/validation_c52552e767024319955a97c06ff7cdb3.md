### Title
Whitelisted User Can Bypass Pause and Drain Pool ETH via `swapAssetToPremintedRsETH` - (File: contracts/pools/RSETHPoolV2ExternalBridge.sol)

---

### Summary

`RSETHPoolV2ExternalBridge` defines a `paused` state enforced by the `whenNotPaused` modifier. The `deposit` function (ETH → wrsETH) is correctly gated by `whenNotPaused`. However, `swapAssetToPremintedRsETH` (rsETH → ETH) is only gated by `onlyOperatorOrWhitelisted(msg.sender)` and **omits** `whenNotPaused`. This means any address holding `WHITELISTED_USER_ROLE` or `OPERATOR_ROLE` can continue to swap rsETH for ETH from the pool even while the contract is paused, directly analogous to the external report's pattern where a user holding both a positive role (whitelisted) and a negative state (paused/blacklisted) bypasses the restriction because only the positive role is checked.

---

### Finding Description

`RSETHPoolV2ExternalBridge` introduces two distinct roles that can coexist on the same address:

1. **Positive role**: `WHITELISTED_USER_ROLE` (or `OPERATOR_ROLE`) — grants access to `swapAssetToPremintedRsETH`.
2. **Negative state**: `paused == true` — is supposed to halt all user-facing operations.

The `deposit` function correctly enforces both constraints:

```solidity
// contracts/pools/RSETHPoolV2ExternalBridge.sol:289
function deposit(string memory referralId) external payable nonReentrant whenNotPaused limitDailyMint(msg.value) {
```

But `swapAssetToPremintedRsETH` only checks the positive role and never checks the negative state:

```solidity
// contracts/pools/RSETHPoolV2ExternalBridge.sol:418-424
function swapAssetToPremintedRsETH(
    address rsETH,
    uint256 rsETHAmount
)
    external
    nonReentrant
    onlyOperatorOrWhitelisted(msg.sender)   // ← only positive role checked
                                             // ← whenNotPaused is ABSENT
{
```

Inside the function, the pool's ETH balance is transferred to the caller:

```solidity
// contracts/pools/RSETHPoolV2ExternalBridge.sol:441-443
if (getETHBalanceMinusFees() < ethAmount) revert InsufficientETHBalanceForReverseSwap();
(bool success,) = payable(msg.sender).call{ value: ethAmount }("");
```

The `WHITELISTED_USER_ROLE` and `OPERATOR_ROLE` are granted by the `DEFAULT_ADMIN_ROLE` via standard `AccessControl` `grantRole`. These roles are entirely independent of the `paused` flag, which is set by the `PAUSER_ROLE`. There is no enforcement that prevents a whitelisted/operator address from acting while the contract is paused — the two states are not mutually exclusive in the access check for this function.

---

### Impact Explanation

When the contract is paused (e.g., due to an oracle anomaly, a security incident, or a rate manipulation event), the intended effect is to halt all user-facing asset flows. However, any address holding `WHITELISTED_USER_ROLE` or `OPERATOR_ROLE` can continue to call `swapAssetToPremintedRsETH`, draining the pool's ETH reserves in exchange for rsETH at the current (potentially manipulated) oracle rate. This:

- Depletes the ETH held in the pool that backs pending wrsETH redemptions.
- Leaves the pool unable to service normal operations once the pause is lifted.
- Constitutes a **temporary (potentially permanent) freezing of depositor funds** and, if the oracle rate is stale or manipulated at pause time, a **theft of pool ETH**.

Impact: **Medium — Temporary freezing of funds** (pool ETH drained during pause, blocking resumption of normal operations); escalates to **High** if the pause was triggered by an oracle manipulation that the attacker exploits via the unguarded swap.

---

### Likelihood Explanation

- The contract has an explicit `PAUSER_ROLE` and a `pause()` function, confirming that pausing is an expected operational action.
- `WHITELISTED_USER_ROLE` and `OPERATOR_ROLE` are separate roles managed independently from the pauser, exactly mirroring the external report's scenario where the Blacklist Manager and Whitelist Manager are different entities.
- Any whitelisted user or operator who is active at the time of a pause can immediately exploit this without any additional setup.
- Likelihood: **Medium** — requires the contract to be paused AND a whitelisted/operator address to act, both of which are realistic operational conditions.

---

### Recommendation

Add the `whenNotPaused` modifier to `swapAssetToPremintedRsETH`:

```solidity
function swapAssetToPremintedRsETH(
    address rsETH,
    uint256 rsETHAmount
)
    external
    nonReentrant
    whenNotPaused                          // ← add this
    onlyOperatorOrWhitelisted(msg.sender)
{
```

---

### Proof of Concept

1. Admin calls `pause()` on `RSETHPoolV2ExternalBridge` (e.g., due to a suspected oracle issue). `paused` is set to `true`.
2. `deposit` is now blocked for all callers — `whenNotPaused` reverts.
3. Alice holds `WHITELISTED_USER_ROLE` and a balance of rsETH on the L2 chain.
4. Alice calls `swapAssetToPremintedRsETH(rsETH, rsETHAmount)`. The modifier `onlyOperatorOrWhitelisted(msg.sender)` passes because Alice has `WHITELISTED_USER_ROLE`. The absent `whenNotPaused` check means `paused == true` is never evaluated.
5. The function transfers Alice's rsETH to the wrapper and sends ETH from the pool to Alice at the current oracle rate.
6. Alice repeats until `getETHBalanceMinusFees() < ethAmount` reverts, fully draining the pool's ETH.
7. When the admin calls `unpause()`, the pool has no ETH to back wrsETH redemptions, freezing depositor funds.

**Root cause**: `swapAssetToPremintedRsETH` at `contracts/pools/RSETHPoolV2ExternalBridge.sol` lines 418–446 checks only `onlyOperatorOrWhitelisted` and omits `whenNotPaused`, while `deposit` at line 289 correctly applies both. The `paused` state and `WHITELISTED_USER_ROLE`/`OPERATOR_ROLE` are not enforced as mutually exclusive for this function. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
