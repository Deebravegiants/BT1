### Title
Token Decimal Mismatch in `viewSwapRsETHAmountAndFee` Enables Excess rsETH Minting for Non-18-Decimal Tokens - (File: contracts/pools/RSETHPoolV3.sol)

---

### Summary
The `viewSwapRsETHAmountAndFee(uint256 amount, address token)` function in all L2 pool contracts computes the rsETH output amount using raw token units without normalizing for the token's decimal precision. When a supported ERC-20 token has fewer than 18 decimals (e.g., USDC with 6), the formula silently overvalues the deposit by up to `10^(18 - tokenDecimals)`, allowing an attacker to mint vastly more rsETH than the deposited collateral warrants.

---

### Finding Description
In `RSETHPoolV3.sol`, `RSETHPool.sol`, `RSETHPoolNoWrapper.sol`, `RSETHPoolV3ExternalBridge.sol`, and `RSETHPoolV3WithNativeChainBridge.sol`, the token-deposit path computes the rsETH output as:

```solidity
// RSETHPoolV3.sol – viewSwapRsETHAmountAndFee(amount, token)
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [1](#0-0) 

The oracle used for collateral tokens is `ChainlinkOracleForRSETHPoolCollateral`, which normalizes the Chainlink answer to 1e18 precision:

```solidity
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
``` [2](#0-1) 

This `normalizedPrice` represents **ETH per 1 full token** (e.g., 4×10¹⁴ for USDC ≈ 0.0004 ETH/USDC). However, `amountAfterFee` is passed in the token's **native atomic units** (e.g., 1×10⁶ for 1 USDC). The formula therefore treats 1×10⁶ USDC atoms as if they were 1×10⁶ units of an 18-decimal token, producing:

```
rsETHAmount = 1e6 * 4e14 / 1e18 = 400 rsETH
```

The correct result is `4×10¹⁴` wei of rsETH (0.0004 rsETH). The overvaluation factor is `10^(18−6) = 10¹²`.

The same arithmetic flaw is present identically in:
- `RSETHPool.sol` [3](#0-2) 
- `RSETHPoolNoWrapper.sol` [4](#0-3) 
- `RSETHPoolV3ExternalBridge.sol` [5](#0-4) 
- `RSETHPoolV3WithNativeChainBridge.sol` [6](#0-5) 

The minted rsETH is issued directly to the caller via `wrsETH.mint(msg.sender, rsETHAmount)`: [7](#0-6) 

---

### Impact Explanation
**Critical — Direct theft of user funds / protocol insolvency.**

An attacker depositing 1 USDC (worth ~0.0004 ETH) receives ~400 rsETH (worth ~400 ETH at a 1:1 rsETH/ETH rate). The attacker can bridge or redeem those rsETH tokens for ETH, extracting value that was never deposited. Because rsETH is a shared liability token backed by the collective TVL, this dilutes every existing rsETH holder and can render the protocol insolvent.

---

### Likelihood Explanation
**Medium.** The pool contracts expose a generic `addSupportedToken` mechanism that accepts any ERC-20 with a configured oracle. The protocol is designed to be extensible beyond LSTs. If any non-18-decimal token (USDC, USDT, WBTC, etc.) is ever added as a supported collateral — a plausible operational decision — the vulnerability is immediately exploitable by any unprivileged depositor calling the public `deposit(address token, uint256 amount, string referralId)` function. No special permissions are required beyond the token being listed.

---

### Recommendation
Normalize `amountAfterFee` to 18-decimal precision before applying the rate formula. Fetch the token's decimals and scale accordingly:

```solidity
uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 normalizedAmount = amountAfterFee * 1e18 / 10**tokenDecimals;
rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
```

Apply this fix consistently across all five affected pool contracts.

---

### Proof of Concept

**Setup:** USDC (6 decimals) is added as a supported token in `RSETHPoolV3` with a Chainlink USDC/ETH oracle. Assume 1 USDC = 0.0004 ETH, so `tokenToETHRate = 4e14`. rsETH/ETH rate = 1e18.

**Attack:**
1. Attacker calls `deposit(USDC, 1e6, "")` — depositing 1 USDC.
2. `viewSwapRsETHAmountAndFee(1e6, USDC)` computes:
   - `fee = 1e6 * feeBps / 10_000` (negligible)
   - `amountAfterFee ≈ 1e6`
   - `tokenToETHRate = 4e14`
   - `rsETHToETHrate = 1e18`
   - `rsETHAmount = 1e6 * 4e14 / 1e18 = 400` rsETH (= 400×10¹⁸ wei)
3. `wrsETH.mint(attacker, 400e18)` is executed.
4. Attacker holds 400 rsETH backed by only 0.0004 ETH of collateral — a 1,000,000× overissuance.
5. Attacker redeems or sells the rsETH, extracting ~400 ETH of value from the protocol. [8](#0-7) [9](#0-8)
