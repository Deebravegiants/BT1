### Title
`emergencyRecover` Cannot Recover ETH Stuck in `SonicBridgeReceiver` - (File: contracts/bridges/SonicBridgeReceiver.sol)

### Summary
`SonicBridgeReceiver.emergencyRecover` unconditionally wraps the `token` argument in `IERC20(token)` calls without handling the native ETH case (`token == address(0)`). If ETH becomes stuck in the contract (e.g., via `selfdestruct` from another contract), the admin has no path to recover it — the function will revert on every attempt.

### Finding Description
`emergencyRecover` is the contract's sole general-purpose asset-recovery escape hatch. Its implementation is:

```solidity
function emergencyRecover(address token, address recipient, uint256 amount) external onlyRole(DEFAULT_ADMIN_ROLE) {
    UtilLib.checkNonZeroAddress(recipient);

    uint256 balance = IERC20(token).balanceOf(address(this));   // reverts if token == address(0)
    uint256 recoverAmount = amount == 0 ? balance : amount;
    if (recoverAmount > balance) revert InsufficientBalance();

    IERC20(token).safeTransfer(recipient, recoverAmount);        // also reverts
}
```

When `token == address(0)` (native ETH), both `IERC20(address(0)).balanceOf(...)` and `IERC20(address(0)).safeTransfer(...)` will revert because there is no code at the zero address. The function provides no ETH-specific branch (no `address(this).balance` check, no low-level `.call{value: ...}("")`).

Contrast this with the sibling contract `SonicChainNativeTokenBridge`, which correctly separates recovery into `recoverTokens` (ERC20) and `recoverETH` (native ETH), and with the `Recoverable` abstract base, which also handles both cases cleanly. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
Any ETH that ends up in `SonicBridgeReceiver` is permanently unrecoverable through the contract's own interface. The contract has no `receive()` function, so ETH cannot arrive via ordinary transfers or `msg.value`; however, a `selfdestruct`-based forced-send from any other contract (including a malicious or buggy one) will deposit ETH with no recourse. Once ETH is in the contract, every call to `emergencyRecover(address(0), ...)` reverts, and there is no alternative ETH-withdrawal path. This constitutes a permanent freeze of those funds. [4](#0-3) 

### Likelihood Explanation
The contract has no `receive()` function and no `payable` functions, so ETH can only arrive via a forced `selfdestruct` send. This is an uncommon but real and well-documented vector (e.g., a self-destructing helper contract, a protocol upgrade gone wrong, or an accidental deployment). The likelihood is low but non-zero, and the consequence once it occurs is irreversible. [5](#0-4) 

### Recommendation
Add an ETH-specific branch to `emergencyRecover`, mirroring the pattern already used in `SonicChainNativeTokenBridge.recoverETH` and `Recoverable.recoverETH`:

```solidity
function emergencyRecover(address token, address recipient, uint256 amount) external onlyRole(DEFAULT_ADMIN_ROLE) {
    UtilLib.checkNonZeroAddress(recipient);

    if (token == address(0)) {
        // Native ETH recovery
        uint256 balance = address(this).balance;
        uint256 recoverAmount = amount == 0 ? balance : amount;
        if (recoverAmount > balance) revert InsufficientBalance();
        (bool sent,) = payable(recipient).call{value: recoverAmount}("");
        if (!sent) revert TransferToVaultFailed();
    } else {
        uint256 balance = IERC20(token).balanceOf(address(this));
        uint256 recoverAmount = amount == 0 ? balance : amount;
        if (recoverAmount > balance) revert InsufficientBalance();
        IERC20(token).safeTransfer(recipient, recoverAmount);
    }
}
```

Alternatively, add a dedicated `recoverETH` function (as done in `SonicChainNativeTokenBridge`) and a `receive()` function so that forced ETH sends do not silently brick the contract. [6](#0-5) [7](#0-6) 

### Proof of Concept
1. A contract holding ETH calls `selfdestruct(payable(address(sonicBridgeReceiver)))`, forcing ETH into `SonicBridgeReceiver`.
2. Admin calls `emergencyRecover(address(0), adminWallet, 0)`.
3. Execution reaches `IERC20(address(0)).balanceOf(address(this))` — this is a call to the zero address, which has no code; the EVM returns empty data, and OpenZeppelin's `SafeERC20` / the ABI decoder reverts.
4. Every subsequent recovery attempt reverts identically. The ETH is permanently frozen. [1](#0-0)
