### Title
Depositor Can DoS Another User's `initiateWithdrawal` by Committing a Tiny Amount First — (`contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTWithdrawalManager.initiateWithdrawal` reverts with `ExceedAmountToWithdraw` when the caller's expected asset amount exceeds `getAvailableAssetAmount`. Because `getAvailableAssetAmount` is derived from the shared, mutable `assetsCommitted` mapping, any other user can frontrun a victim's maximum-sized withdrawal request by submitting a tiny withdrawal first, shrinking the available amount below the victim's threshold and causing the victim's transaction to revert.

---

### Finding Description

`initiateWithdrawal` computes the expected asset amount for the caller's `rsETHUnstaked` and then hard-reverts if it exceeds the currently available asset amount:

```solidity
// LRTWithdrawalManager.sol lines 166–170
IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);

uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);

if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();
``` [1](#0-0) 

`getAvailableAssetAmount` is:

```solidity
availableAssetAmount = totalAssets > assetsCommitted[asset]
    ? totalAssets - assetsCommitted[asset]
    : 0;
``` [2](#0-1) 

Every successful call to `initiateWithdrawal` increases `assetsCommitted[asset]`:

```solidity
assetsCommitted[asset] += expectedAssetAmount;
``` [3](#0-2) 

Because the revert check is strict (`>`) and the available amount is a shared, decreasing counter, a frontrunner who commits even the minimum withdrawal amount (`minRsEthAmountToWithdraw[asset]`) before the victim's transaction is included will reduce `getAvailableAssetAmount` enough to push the victim's `expectedAssetAmount` over the limit, causing the victim's transaction to revert.

---

### Impact Explanation

**Medium — Temporary freezing of funds.**

A victim who attempts to initiate a withdrawal for the full available amount has their transaction reverted. Their rsETH is returned (the whole tx reverts), but they are temporarily blocked from initiating the withdrawal in that block. They must retry with a reduced amount or wait for more assets to become available. A persistent attacker can repeat this pattern every time the victim retries, indefinitely delaying the victim's ability to enter the withdrawal queue and thus temporarily freezing their ability to redeem rsETH.

---

### Likelihood Explanation

`initiateWithdrawal` is a public, permissionless function callable by any rsETH holder. [4](#0-3) 

The minimum withdrawal amount (`minRsEthAmountToWithdraw[asset]`) can be very small (even zero if unset), making the cost of the frontrun negligible. On Ethereum mainnet, public mempool visibility makes this straightforward to execute. The attacker does not lose funds — they simply enter the withdrawal queue themselves with a tiny amount.

---

### Recommendation

Instead of reverting when `expectedAssetAmount > getAvailableAssetAmount(asset)`, cap the withdrawal to the available amount and proceed, similar to the mitigation recommended in the referenced WiseLending report:

```solidity
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
uint256 available = getAvailableAssetAmount(asset);

if (expectedAssetAmount > available) {
    // Cap to available instead of reverting
    expectedAssetAmount = available;
    // Optionally recompute rsETHUnstaked from the capped expectedAssetAmount
}
```

Alternatively, accept a caller-supplied `minExpectedAssetAmount` slippage parameter and revert only if the available amount falls below that floor, giving callers explicit control over their acceptable range.

---

### Proof of Concept

1. The protocol has 100 ETH of available assets for withdrawal (`getAvailableAssetAmount(ETH_TOKEN) == 100 ETH`).
2. Alice submits `initiateWithdrawal(ETH_TOKEN, rsETHFor100ETH, "")` — the maximum possible withdrawal.
3. Bob observes Alice's pending transaction in the mempool and frontruns it with `initiateWithdrawal(ETH_TOKEN, minRsEthAmountToWithdraw[ETH_TOKEN], "")` — a dust-sized withdrawal.
4. Bob's transaction executes first: `assetsCommitted[ETH_TOKEN]` increases by a small amount (e.g., 0.01 ETH worth), so `getAvailableAssetAmount(ETH_TOKEN)` drops to ~99.99 ETH.
5. Alice's transaction executes: `expectedAssetAmount` for `rsETHFor100ETH` is 100 ETH, which now exceeds `getAvailableAssetAmount` (~99.99 ETH).
6. `revert ExceedAmountToWithdraw()` is triggered. [5](#0-4) 
7. Alice's rsETH is returned (tx reverts), but she is blocked from initiating her withdrawal. Bob can repeat this every time Alice retries.
