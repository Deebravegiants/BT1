### Title
Unclaimed KERNEL Reward Tokens Permanently Stranded in KernelMerkleDistributor - (File: contracts/KERNEL/KernelMerkleDistributor.sol)

### Summary

`KernelMerkleDistributor` holds KERNEL tokens for distribution to eligible users via merkle-proof claims. Unlike `KernelTop100MerkleDistributor`, which includes a `withdrawTokens` recovery function, `KernelMerkleDistributor` has no mechanism to recover tokens that are never claimed. Any KERNEL tokens deposited into this contract that go unclaimed are permanently stranded.

### Finding Description

The `KernelMerkleDistributor` contract receives KERNEL tokens and distributes them to users who submit valid merkle proofs. The admin can update the `currentMerkleRoot` via `setMerkleRoot()`, incrementing `currentMerkleRootIndex` and `currentIndex` each time. [1](#0-0) 

The claim model is cumulative: a user's claimable amount is `cumulativeAmount - userClaims[account].cumulativeAmount`, meaning each new root encodes the total entitlement across all epochs. [2](#0-1) 

However, the contract contains **no sweep, withdraw, or recovery function** for the KERNEL token. Scanning the entire admin section confirms only `setKernelDepositPool`, `setProtocolTreasury`, `setFeeInBPS`, `setMerkleRoot`, `pause`, and `unpause` are present. [3](#0-2) 

By contrast, the sibling contract `KernelTop100MerkleDistributor` explicitly includes a `withdrawTokens` function for exactly this purpose: [4](#0-3) 

The protocol also provides a general-purpose `Recoverable` base contract with `recoverTokens` and `recoverETH`, but `KernelMerkleDistributor` does not inherit from it. [5](#0-4) 

### Impact Explanation

KERNEL tokens deposited into `KernelMerkleDistributor` for reward distribution become permanently frozen if:

1. **Users never claim** — wallets are lost, users are inactive, or users are unaware of their eligibility.
2. **Merkle root is updated** — if a new root reduces or removes a user's entitlement relative to what was previously funded, the excess tokens already in the contract have no exit path.
3. **Contract is paused indefinitely** — the `pause()` function blocks all claims; if the contract is never unpaused, all remaining tokens are locked forever.

This matches the **Medium: Permanent freezing of unclaimed yield** impact category.

### Likelihood Explanation

Merkle distributor contracts routinely have unclaimed balances — it is a well-documented operational reality. The fact that the protocol's own `KernelTop100MerkleDistributor` added `withdrawTokens` demonstrates awareness of this risk. The omission in `KernelMerkleDistributor` is a straightforward oversight. Any epoch where not all eligible users claim results in permanently stranded KERNEL tokens.

### Recommendation

Add a token recovery function to `KernelMerkleDistributor`, mirroring the `withdrawTokens` function already present in `KernelTop100MerkleDistributor`:

```solidity
function withdrawTokens(address _token, uint256 _amount, address _recipient) external onlyOwner {
    UtilLib.checkNonZeroAddress(_token);
    UtilLib.checkNonZeroAddress(_recipient);
    if (_amount == 0) revert ZeroValueProvided();
    IERC20(_token).safeTransfer(_recipient, _amount);
    emit TokensWithdrawn(_token, _amount, _recipient);
}
```

Alternatively, inherit from the existing `Recoverable` base contract.

### Proof of Concept

1. Admin funds `KernelMerkleDistributor` with 1,000,000 KERNEL tokens and sets a merkle root covering 1,000 users.
2. Only 600 users claim their tokens; 400 users never interact with the contract.
3. The 400 users' share of KERNEL tokens (potentially hundreds of thousands of tokens) remains in the contract.
4. Admin has no function to call to recover these tokens.
5. Tokens are permanently locked — `kernel.balanceOf(address(kernelMerkleDistributor))` is non-zero forever with no exit path. [6](#0-5) [7](#0-6)

        UtilLib.checkNonZeroAddress(tokenAddress);
