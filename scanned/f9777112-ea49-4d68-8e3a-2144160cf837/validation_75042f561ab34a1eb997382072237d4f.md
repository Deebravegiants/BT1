### Title
ETH Transfer via Low-Level Call to Non-Existent `instantWithdrawalFeeRecipient` Silently Succeeds, Permanently Freezing Fees - (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTWithdrawalManager._transferAsset` sends ETH via a bare low-level `.call{value:}` without verifying the recipient address has deployed code. Per EVM design, low-level calls return `true` for non-existent accounts. If the stored `instantWithdrawalFeeRecipient` address is a `selfdestruct`ed contract, ETH fees are silently sent to a non-existent address with no revert, permanently freezing protocol yield.

---

### Finding Description

`_transferAsset` is the sole ETH-dispatch helper in `LRTWithdrawalManager`:

```solidity
function _transferAsset(address asset, address to, uint256 amount) internal {
    if (asset == LRTConstants.ETH_TOKEN) {
        (bool sent,) = payable(to).call{ value: amount }("");
        if (!sent) revert EthTransferFailed();
    } else {
        IERC20(asset).safeTransfer(to, amount);
    }
}
``` [1](#0-0) 

It checks `sent` but never checks `address(to).code.length > 0`. The Solidity documentation explicitly states: *"The low-level functions `call`, `delegatecall` and `staticcall` return `true` as their first return value if the account called is non-existent, as part of the design of the EVM."*

`_transferAsset` is called in `instantWithdrawal` to dispatch fees to `feeRecipient`, which resolves to the stored `instantWithdrawalFeeRecipient` state variable:

```solidity
address feeRecipient = instantWithdrawalFeeRecipient;
if (feeRecipient == address(0)) {
    feeRecipient = lrtConfig.getContract(LRTConstants.PROTOCOL_TREASURY);
}
if (fee > 0) {
    _transferAsset(asset, feeRecipient, fee);
    emit InstantWithdrawalFeeCollected(msg.sender, asset, fee);
}
``` [2](#0-1) 

`instantWithdrawalFeeRecipient` is a mutable stored address set by LRTManager:

```solidity
function setInstantWithdrawalFeeRecipient(address feeRecipient) external onlyLRTManager {
    UtilLib.checkNonZeroAddress(feeRecipient);
    instantWithdrawalFeeRecipient = feeRecipient;
    emit InstantWithdrawalFeeRecipientUpdated(feeRecipient);
}
``` [3](#0-2) 

If `instantWithdrawalFeeRecipient` is set to a contract (e.g., a dedicated fee collector) that is subsequently `selfdestruct`ed, every subsequent `instantWithdrawal` call will silently send ETH fees to the non-existent address. The call returns `true`, `EthTransferFailed` is never triggered, `InstantWithdrawalFeeCollected` is emitted normally, and the transaction completes — with the fee ETH permanently frozen.

A secondary path with higher impact exists in `_processWithdrawalCompletion`, which also calls `_transferAsset(asset, user, request.expectedAssetAmount)`: [4](#0-3) 

If `user` is a smart-contract wallet that has been `selfdestruct`ed and an operator calls `completeWithdrawalForUser` on their behalf, the user's full withdrawal amount is silently sent to the non-existent address, permanently freezing principal funds.

---

### Impact Explanation

**Primary path (`instantWithdrawalFeeRecipient`):** ETH fees collected across all `instantWithdrawal` calls are permanently frozen at the non-existent address. The protocol emits success events and records no failure. This is **permanent freezing of unclaimed yield** (Medium).

**Secondary path (`_processWithdrawalCompletion` via `completeWithdrawalForUser`):** A user's full ETH withdrawal principal is permanently frozen. The withdrawal is marked complete with no recourse. This is **permanent freezing of funds** (Critical).

---

### Likelihood Explanation

**Low.** Requires `instantWithdrawalFeeRecipient` to be set to a contract address that is subsequently `selfdestruct`ed before the admin updates the stored value. While EIP-6780 restricts `selfdestruct` on Ethereum mainnet post-Dencun (March 2024), the protocol is deployed across multiple chains including L2s (evidenced by the `contracts/L2/` and `contracts/pools/` directories) where this EVM restriction may not apply. The scenario is realistic if a fee collector contract is deprecated via `selfdestruct` and the admin does not immediately update `instantWithdrawalFeeRecipient`. Critically, the silent success means the admin receives no on-chain signal that the address is broken.

---

### Recommendation

Before dispatching ETH via low-level call to any stored/configured address, verify the recipient has deployed code:

```solidity
function _transferAsset(address asset, address to, uint256 amount) internal {
    if (asset == LRTConstants.ETH_TOKEN) {
        if (to.code.length == 0 && to.balance == 0) revert RecipientDoesNotExist();
        (bool sent,) = payable(to).call{ value: amount }("");
        if (!sent) revert EthTransferFailed();
    } else {
        IERC20(asset).safeTransfer(to, amount);
    }
}
```

Alternatively, use OpenZeppelin's `Address.sendValue`, which wraps this check. Also apply the same guard in `Recoverable.recoverETH` and any other ETH dispatch sites. [5](#0-4) 

---

### Proof of Concept

1. LRTManager calls `setInstantWithdrawalFeeRecipient(feeCollector)` where `feeCollector` is a deployed fee-collector contract.
2. `feeCollector` is `selfdestruct`ed (e.g., a critical bug is found and the contract is destroyed to signal users to migrate).
3. Admin does not immediately update `instantWithdrawalFeeRecipient` (or is unaware, since no on-chain failure occurs).
4. A user calls `instantWithdrawal(LRTConstants.ETH_TOKEN, rsETHAmount, "ref")`.
5. Inside `instantWithdrawal`, `fee > 0` so `_transferAsset(ETH_TOKEN, feeCollector, fee)` is called.
6. `payable(feeCollector).call{value: fee}("")` returns `(true, "")` — the account is non-existent, EVM returns `true` by design.
7. `EthTransferFailed` is never triggered; `InstantWithdrawalFeeCollected` is emitted.
8. Fee ETH is permanently frozen at `feeCollector`. This repeats for every subsequent `instantWithdrawal` call until the admin notices and corrects the stored address. [6](#0-5)
