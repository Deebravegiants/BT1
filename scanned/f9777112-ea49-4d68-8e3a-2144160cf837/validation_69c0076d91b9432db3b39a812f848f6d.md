### Title
Slashing-Induced Price Drop Exceeding `pricePercentageLimit` Permanently Freezes Protocol Until Admin Intervention - (File: contracts/LRTOracle.sol)

### Summary
In `LRTOracle._updateRsETHPrice()`, when a slashing event causes `newRsETHPrice` to fall more than `pricePercentageLimit` below `highestRsethPrice`, the function pauses the entire protocol and returns **without updating `rsETHPrice`**. Because `highestRsethPrice` is monotonically non-decreasing and has no admin setter, every subsequent call to `_updateRsETHPrice()` re-triggers the same condition, keeping the protocol frozen indefinitely until the admin disables the limit check entirely.

### Finding Description
`_updateRsETHPrice()` implements a downside-protection guard:

```solidity
bool isPriceDecreaseOffLimit =
    pricePercentageLimit > 0 && diff > pricePercentageLimit.mulWad(highestRsethPrice);

if (isPriceDecreaseOffLimit) {
    if (!lrtDepositPool.paused()) lrtDepositPool.pause();
    if (!withdrawalManager.paused()) withdrawalManager.pause();
    _pause();
    return;          // rsETHPrice is NOT updated
}
``` [1](#0-0) 

The threshold is measured against `highestRsethPrice`, the all-time peak, which is only ever written upward:

```solidity
if (newRsETHPrice > highestRsethPrice) {
    highestRsethPrice = newRsETHPrice;
}
``` [2](#0-1) 

There is no function in the contract that decreases or resets `highestRsethPrice`. After a slashing event that drops the price by more than `pricePercentageLimit` relative to the historical peak:

1. The protocol pauses (deposit pool, withdrawal manager, oracle).
2. `rsETHPrice` is left at its pre-slashing stale value.
3. The public `updateRSETHPrice()` is gated by `whenNotPaused`, so it cannot be called.
4. `updateRSETHPriceAsManager()` (no `whenNotPaused` guard) can be called by a manager, but it invokes the same `_updateRsETHPrice()` path, which will re-trigger `isPriceDecreaseOffLimit` and return early again — the price is still never updated.
5. Because `highestRsethPrice` never decreases, the condition fires on every future call as long as the actual price remains below the threshold. [3](#0-2) 

The only escape is for an admin to call `setPricePercentageLimit(0)` (disabling the guard entirely), manually unpause all three contracts, and then trigger a price update — a multi-step privileged sequence that is not atomic and leaves the protocol frozen in the interim. [4](#0-3) 

### Impact Explanation
All user deposits and withdrawals are frozen for the entire duration between the slashing event and the completion of the admin recovery sequence. Because `highestRsethPrice` is immutable from below and there is no direct setter, the freeze is not self-resolving: it persists indefinitely without privileged intervention. This constitutes **temporary (potentially indefinite) freezing of funds** for all rsETH holders and depositors.

**Severity: Medium — Temporary freezing of funds.**

### Likelihood Explanation
Slashing is a known, realistic event in any ETH-staking protocol. The `pricePercentageLimit` is configurable but is expected to be set to a small value (e.g., 1–5%) to guard against oracle manipulation. A single large slashing incident (or an accumulation of smaller ones between oracle updates) can easily produce a price drop that exceeds this limit relative to `highestRsethPrice`. The entry point (`updateRSETHPrice()`) is public and permissionless, so any user or keeper calling it after a slashing event will trigger the freeze.

### Recommendation
1. **Introduce a `highestRsethPrice` admin setter** (or a mechanism to ratchet it down after confirmed slashing) so the protocol can recover without disabling the guard entirely.
2. Alternatively, measure the downside limit against `rsETHPrice` (the last confirmed price) rather than `highestRsethPrice` (the all-time peak), so that the threshold is not permanently anchored to a historical high that may never be reached again after a slashing event.
3. Ensure `updateRSETHPriceAsManager()` can bypass the pause-and-return path (e.g., with a separate `forceUpdate` flag) so a manager can push the corrected price through without first disabling the entire guard.

### Proof of Concept
1. Protocol operates normally; `highestRsethPrice = 1.10 ETH`, `rsETHPrice = 1.09 ETH`, `pricePercentageLimit = 2e16` (2%).
2. A large slashing event reduces the underlying ETH backing; `_getTotalEthInProtocol()` now yields a price of `0.90 ETH`.
3. Anyone calls `updateRSETHPrice()`.
4. Inside `_updateRsETHPrice()`: `diff = 1.10 - 0.90 = 0.20 ETH`; `pricePercentageLimit.mulWad(highestRsethPrice) = 0.02 * 1.10 = 0.022 ETH`; `0.20 > 0.022` → `isPriceDecreaseOffLimit = true`.
5. Deposit pool, withdrawal manager, and oracle are all paused. Function returns. `rsETHPrice` remains `1.09 ETH` (stale).
6. Manager calls `updateRSETHPriceAsManager()`. Same path executes; same condition fires; same early return. Price still `1.09 ETH`.
7. `highestRsethPrice` is still `1.10 ETH` (no setter). Every future call repeats step 6.
8. All user withdrawals and deposits are frozen until admin executes the multi-step recovery sequence. [5](#0-4) [6](#0-5)
