### Title
`KernelTop100MerkleDistributor::withdrawTokens` Allows Owner to Drain the `kernel` Distribution Token, Stealing All Unclaimed User Yield - (File: contracts/KERNEL/KernelTop100MerkleDistributor.sol)

### Summary
`KernelTop100MerkleDistributor::withdrawTokens` imposes no restriction on which token can be withdrawn. The contract holds KERNEL tokens that users are entitled to claim via merkle proofs over a 30-day vesting schedule. Because `withdrawTokens` accepts any `_token` address with no blocklist, the owner can call it with `_token = address(kernel)` and drain the entire KERNEL balance, permanently preventing all users from claiming their entitled yield.

### Finding Description
`KernelTop100MerkleDistributor` is a vesting merkle distributor that holds KERNEL tokens on behalf of eligible users. Users call `claim` or `claimAndStake` to receive their vested allocation over a 30-day window, verified by a merkle proof.

The admin function `withdrawTokens` is documented as a general token recovery utility:

```solidity
function withdrawTokens(address _token, uint256 _amount, address _recipient) external onlyOwner {
    UtilLib.checkNonZeroAddress(_token);
    UtilLib.checkNonZeroAddress(_recipient);

    if (_amount == 0) {
        revert ZeroValueProvided();
    }

    IERC20(_token).safeTransfer(_recipient, _amount);

    emit TokensWithdrawn(_token, _amount, _recipient);
}
``` [1](#0-0) 

There is no guard of the form `if (_token == address(kernel)) revert()`. The `kernel` state variable is the sole distribution asset of this contract: [2](#0-1) 

All KERNEL tokens pre-funded into the contract to back user claims sit at `address(this)` and are transferred out directly in `claim` and `claimAndStake`: [3](#0-2) 

Because `withdrawTokens` places no restriction on `_token`, the owner can pass `address(kernel)` and transfer the entire KERNEL balance to an arbitrary `_recipient` before any user has claimed.

### Impact Explanation
**High — Theft of unclaimed yield.**

All KERNEL tokens held by the distributor represent unclaimed yield owed to merkle-eligible users. A single `withdrawTokens(address(kernel), kernel.balanceOf(address(this)), attacker)` call drains the full balance. Every subsequent `claim` or `claimAndStake` call reverts due to insufficient balance, permanently denying users their entitled rewards. This maps directly to the "High. Theft of unclaimed yield." impact category.

### Likelihood Explanation
**Low.** The call requires the `onlyOwner` key to be malicious or compromised. This mirrors the exact precondition of the reference finding (`SherpaVault::rescueTokens` also requires owner action). The vulnerability is a missing design-level guard — the contract is intended to protect distribution funds from accidental rescue, yet the guard is absent — making it a structural flaw rather than a pure trust assumption.

### Recommendation
Add an explicit blocklist for the `kernel` token inside `withdrawTokens`, mirroring the fix applied to `SherpaVault`:

```diff
function withdrawTokens(address _token, uint256 _amount, address _recipient) external onlyOwner {
    UtilLib.checkNonZeroAddress(_token);
    UtilLib.checkNonZeroAddress(_recipient);

    if (_amount == 0) {
        revert ZeroValueProvided();
    }

+   if (_token == address(kernel)) revert CannotWithdrawKernelToken();

    IERC20(_token).safeTransfer(_recipient, _amount);

    emit TokensWithdrawn(_token, _amount, _recipient);
}
```

### Proof of Concept
1. Protocol pre-funds `KernelTop100MerkleDistributor` with `N` KERNEL tokens to cover all merkle-eligible users.
2. Vesting starts (`vestingStartTimestamp` is reached). Users begin accruing claimable amounts.
3. Owner calls:
   ```solidity
   withdrawTokens(
       address(kernel),                          // _token = the KERNEL distribution token
       kernel.balanceOf(address(distributor)),   // _amount = entire balance
       attackerAddress                           // _recipient = attacker-controlled address
   );
   ``` [1](#0-0) 
4. The contract's KERNEL balance drops to zero.
5. Every user who calls `claim` or `claimAndStake` now reverts because `kernel.safeTransfer(user, amountToSend)` fails — the contract holds no KERNEL. [4](#0-3) 
6. All unclaimed KERNEL yield is permanently stolen from users with no on-chain recourse.
