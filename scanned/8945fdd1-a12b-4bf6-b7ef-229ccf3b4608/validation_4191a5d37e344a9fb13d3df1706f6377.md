### Title
Missing Slippage Protection in RSETHPool User-Facing Deposit Functions - (File: contracts/pools/RSETHPool.sol)

### Summary
The `deposit()` functions in `RSETHPool.sol` accept ETH or supported tokens from unprivileged users and return pre-minted `wrsETH` at the oracle-determined rate, but provide no `minRSETHAmount` parameter. Users cannot bound the minimum rsETH they receive, exposing every deposit to oracle price movement and MEV sandwich attacks.

### Finding Description
Both public deposit entry points compute the rsETH output entirely from the live oracle rate at execution time, with no caller-supplied floor:

`deposit(string memory referralId)` (ETH path): [1](#0-0) 

`deposit(address token, uint256 amount, string memory referralId)` (token path): [2](#0-1) 

The rate computation in `viewSwapRsETHAmountAndFee` reads `getRate()` from `rsETHOracle` at the moment of execution: [3](#0-2) 

There is no `minRSETHAmount` argument, no deadline, and no post-swap assertion. Whatever the oracle returns at the block the transaction is included in is what the user receives. Compare this with `LRTDepositPool.depositETH`, which does accept a `minRSETHAmountExpected` from the caller: [4](#0-3) [5](#0-4) 

`RSETHPool` provides no equivalent protection.

### Impact Explanation
**Low — Contract fails to deliver promised returns, but does not lose value.**

A user who submits a deposit transaction expecting a rate of `R` may have the transaction mined after an oracle update that lowers the rate to `R'`. The user receives `rsETHAmount = amountAfterFee * 1e18 / R'` instead of the amount they observed off-chain. Because the pool holds pre-minted `wrsETH` and simply transfers it out, the user's ETH/token is consumed and they receive fewer rsETH tokens than anticipated. Funds are not frozen and not stolen outright, but the contract silently delivers a worse outcome than the user agreed to.

### Likelihood Explanation
**Medium.** The `rsETHOracle` rate is updated periodically on-chain. Any oracle update that occurs between a user's transaction submission and its inclusion changes the effective exchange rate. MEV searchers can also observe a pending oracle update and sandwich a user's deposit: submit a deposit just before the oracle update that worsens the rate, then let the user's transaction execute at the worse rate. This is a standard, well-known MEV pattern on L2 pools.

### Recommendation
Add a `minRSETHAmount` parameter to both `deposit` overloads and revert if the computed output falls below it:

```solidity
function deposit(string memory referralId, uint256 minRSETHAmount)
    external payable nonReentrant whenNotPaused
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRSETHAmount) revert SlippageExceeded();
    ...
}
```

This mirrors the protection already present in `LRTDepositPool.depositETH` and `LRTDepositPool.depositAsset`.

### Proof of Concept

1. Oracle currently reports `rsETHToETHrate = 1.05e18` (1 rsETH = 1.05 ETH).
2. User submits `deposit{value: 1 ETH}("ref")` expecting ≈ `0.952 wrsETH`.
3. Before the transaction is mined, the oracle is updated to `rsETHToETHrate = 1.10e18`.
4. `viewSwapRsETHAmountAndFee` now computes `rsETHAmount = 1e18 * 1e18 / 1.10e18 ≈ 0.909 wrsETH`.
5. The user's 1 ETH is consumed and they receive `0.909 wrsETH` — ~4.5% less than expected — with no revert and no recourse. [3](#0-2)

    function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
    }
