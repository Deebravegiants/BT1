### Title
Precision Loss in `rewardPerToken()` Causes Permanent Reward Truncation — (`contracts/KERNEL/KernelDepositPool.sol`)

---

### Summary

Integer division truncation in `rewardPerToken()` permanently destroys reward accrual when `rewardRate * elapsed * DECIMAL_PRECISION < totalKernelStaked`. Because the `updateReward` modifier unconditionally resets `updatedAt` even when the computed increment is zero, the elapsed time that produced no increment is permanently consumed and cannot be recovered in future calculations.

---

### Finding Description

`rewardPerToken()` computes the per-token increment as:

```
(rewardRate * elapsed * DECIMAL_PRECISION) / totalKernelStaked
``` [1](#0-0) 

When `rewardRate * elapsed * 1e18 < totalKernelStaked`, Solidity integer division truncates the result to `0`, so `rewardPerToken()` returns `rewardPerTokenStored` unchanged.

The `updateReward` modifier then:
1. Writes the unchanged value back to `rewardPerTokenStored`
2. **Unconditionally advances `updatedAt`** to `lastTimeRewardApplicable()` [2](#0-1) 

Step 2 is the root cause of permanent loss. The elapsed seconds that produced a zero increment are consumed — `updatedAt` moves forward — so those seconds can never contribute to future reward accumulation. Each subsequent call to any `updateReward`-gated function (stake, stakeFor, initiateWithdrawal, getReward) repeats this cycle, permanently destroying the rewards for each elapsed window.

**Concrete numeric example:**
- `rewardRate = 1` (minimum non-zero; passes the `rewardRate == 0` guard at line 586)
- `totalKernelStaked = 2e18` (2 KERNEL with 18 decimals — a trivially small stake)
- `elapsed = 1` second (one Ethereum block)
- Increment = `(1 * 1 * 1e18) / 2e18 = 0` → truncated [3](#0-2) 

With `rewardRate = receivedAmount / duration`, a `rewardRate` of 1 is reachable when distributing a small reward token amount (e.g., a low-decimal token) over a long duration. The `rewardRate == 0` revert does not prevent `rewardRate = 1`.

More practically, the same truncation occurs at higher `rewardRate` values when `totalKernelStaked` is proportionally larger. For example, `rewardRate = 1000` and `totalKernelStaked = 2e21` (2000 KERNEL) also truncates per-second accrual to zero.

---

### Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

The truncated reward-seconds are not deferred; they are permanently destroyed because `updatedAt` advances past them. Users who should have accrued nonzero rewards receive zero for every elapsed window where the truncation condition holds. The reward tokens remain locked in the contract with no mechanism to redistribute or recover them. The staked KERNEL principal is unaffected (the question's framing of "temporary freezing of funds" is imprecise — it is the yield, not the principal, that is permanently frozen). [4](#0-3) 

---

### Likelihood Explanation

Likelihood is **low-to-medium**. It requires either a very small `rewardRate` (close to 1) or a very large `totalKernelStaked` relative to `rewardRate`. Both conditions are reachable in production without any privileged compromise: any user can stake to increase `totalKernelStaked`, and the admin sets `rewardRate` via `notifyRewardAmount`. The condition worsens as TVL grows, making it more likely over the protocol's lifetime.

---

### Recommendation

Accumulate the remainder rather than discarding it. One standard approach is to scale `rewardRate` by `DECIMAL_PRECISION` at the point it is set in `notifyRewardAmount`, so the division in `rewardPerToken()` is deferred:

```solidity
// In notifyRewardAmount:
rewardRate = (receivedAmount * DECIMAL_PRECISION) / duration;

// In rewardPerToken():
return rewardPerTokenStored + (rewardRate * elapsed) / totalKernelStaked;
```

This shifts the precision loss to a much smaller magnitude. Alternatively, track a `rewardPerTokenRemainder` accumulator and only advance `updatedAt` when the accumulated numerator exceeds `totalKernelStaked`.

    constructor() ERC20("T", "T") { _mint(msg.sender, type(uint256).max); }
}

contract PrecisionLossTest is Test {
    KernelDepositPool pool;
    MockERC20 kernel;
    MockERC20 reward;
    address admin = address(1);
    address alice = address(2);

    function setUp() public {
        kernel = new MockERC20();
        reward = new MockERC20();

        pool = new KernelDepositPool();
        // initialize, grant roles, set duration = 30 days
        // (omitted for brevity — standard setup)

        // Alice stakes 2e18 KERNEL (2 tokens)
        kernel.transfer(alice, 2e18);
        vm.prank(alice);
        kernel.approve(address(pool), 2e18);
        vm.prank(alice);
        pool.stake(2e18);

        // Admin notifies 2_592_000 wei reward over 30 days → rewardRate = 1
        reward.approve(address(pool), 2_592_000);
        vm.prank(admin);
        pool.notifyRewardAmount(2_592_000);
    }

    function testPrecisionLoss() public {
        // Advance 1 second per iteration, trigger updateReward each time
        for (uint i = 0; i < 100; i++) {
            vm.warp(block.timestamp + 1);
            vm.prank(alice);
            pool.getReward(); // triggers updateReward, resets updatedAt
        }

        // After 100 seconds, alice should have earned ~100 wei of reward
        // (rewardRate=1, 100s elapsed, totalStaked=2e18, balanceOf=2e18 → 100 wei)
        // But due to truncation, earned() returns 0 for each 1-second window
        uint256 earned = pool.earned(alice);
        assertEq(earned, 0, "Rewards permanently lost due to truncation");
        // Assert: rewards[alice] is also 0 because each getReward() call found 0
    }
}
```

The test demonstrates that 100 seconds of reward accrual at `rewardRate = 1` with `totalKernelStaked = 2e18` produces zero earned rewards, because each 1-second window truncates to 0 and `updatedAt` advances past it.
