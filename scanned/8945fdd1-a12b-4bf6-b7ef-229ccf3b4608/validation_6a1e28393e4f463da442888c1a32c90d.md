### Title
Missing Minimum Output Slippage Guard in L2 Pool `deposit()` Functions - (File: contracts/pools/RSETHPoolV3.sol, RSETHPoolV3ExternalBridge.sol, RSETHPoolV3WithNativeChainBridge.sol, RSETHPool.sol, RSETHPoolNoWrapper.sol)

---

### Summary
Every L2 liquidity pool `deposit()` function accepts ETH or LST tokens and mints wrsETH/rsETH to the caller based on a live oracle rate, but none of them expose a `minRSETHAmountExpected` parameter. If the oracle rate is updated between the moment a user signs the transaction and the moment it is included on-chain, the user silently receives fewer output tokens than they anticipated. The L1 counterpart (`LRTDepositPool`) already enforces this guard, making the omission on all L2 pools an inconsistency with a concrete user-facing impact.

---

### Finding Description

`LRTDepositPool.depositETH()` and `depositAsset()` on L1 both accept a `minRSETHAmountExpected` argument and revert with `MinimumAmountToReceiveNotMet` when the minted amount falls below it:

```solidity
// LRTDepositPool.sol – L1 deposit with slippage guard
function depositETH(uint256 minRSETHAmountExpected, string calldata referralId)
    external payable ...
{
    uint256 rsethAmountToMint = _beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, minRSETHAmountExpected);
    _mintRsETH(rsethAmountToMint);
}
```

Every L2 pool `deposit()` function omits this parameter entirely. The output amount is computed on-the-fly from the oracle rate at execution time, with no floor check:

```solidity
// RSETHPoolV3.sol – L2 deposit, no slippage guard
function deposit(string memory referralId) external payable ... {
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);   // no minimum enforced
}
```

The same pattern is present in the token-deposit overload and across all pool variants:

| Contract | ETH deposit | Token deposit |
|---|---|---|
| `RSETHPoolV3` | line 246 | line 271 |
| `RSETHPoolV3ExternalBridge` | line 366 | line 390 |
| `RSETHPoolV3WithNativeChainBridge` | line 282 | line 307 |
| `RSETHPool` | line 265 | line 284 |
| `RSETHPoolNoWrapper` | line 231 | line 250 |

The oracle rate is read at execution time via `getRate()`:

```solidity
// RSETHPoolV3.sol
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();          // live oracle read
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
```

If `rsETHToETHrate` increases (rsETH appreciates in ETH terms) between tx submission and inclusion, `rsETHAmount` decreases proportionally and the user has no on-chain mechanism to reject the outcome.

---

### Impact Explanation

**Low – Contract fails to deliver promised returns, but doesn't lose value.**

A user who previews the swap off-chain (e.g., via `viewSwapRsETHAmountAndFee`) and then submits a transaction may receive materially fewer wrsETH/rsETH tokens than the preview indicated. Because each wrsETH token is worth proportionally more ETH at the higher rate, the user's total ETH-denominated value is preserved; however, they receive fewer tokens than they expected and have no on-chain recourse to abort the transaction. This is the same class of missing slippage protection that the L1 pool already guards against.

---

### Likelihood Explanation

The oracle rate is updated periodically by the protocol's cross-chain rate infrastructure. On L2 networks with short block times and active oracle updates, the window between a user's off-chain preview and on-chain execution is sufficient for the rate to shift. No attacker action is required; ordinary oracle maintenance is enough to trigger the discrepancy. The likelihood is **medium** given the frequency of oracle updates across all deployed L2 chains.

---

### Recommendation

Add a `uint256 minRSETHAmountExpected` parameter to every public `deposit()` function in all L2 pool contracts, mirroring the existing guard in `LRTDepositPool._beforeDeposit()`:

```solidity
function deposit(string memory referralId, uint256 minRSETHAmountExpected)
    external payable nonReentrant whenNotPaused ...
{
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(msg.value);
    if (rsETHAmount < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);
}
```

Apply the same pattern to the token-deposit overload and to all five pool variants listed above.

---

### Proof of Concept

1. User calls `viewSwapRsETHAmountAndFee(1 ether)` off-chain; oracle rate is `1.05e18` → preview shows `~0.952 wrsETH`.
2. Before the user's tx is included, the protocol's `CrossChainRateReceiver` updates the pool oracle to `1.10e18`.
3. User's `deposit("ref")` executes; `viewSwapRsETHAmountAndFee` now returns `~0.909 wrsETH`.
4. User receives `0.909 wrsETH` — ~4.5 % fewer tokens than previewed — with no revert and no recourse.
5. Contrast: on L1, the same user would pass `minRSETHAmountExpected = 0.95e18` to `depositETH()`, and `_beforeDeposit` would revert with `MinimumAmountToReceiveNotMet`.         address asset,
        uint256 depositAmount,
        uint256 minRSETHAmountExpected
    )
        private
        view
        returns (uint256 rsethAmountToMint)
    {
        if (depositAmount == 0 || depositAmount < minAmountToDeposit) {
            revert InvalidAmountToDeposit();
        }

        if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
            revert MaximumDepositLimitReached();
        }

        rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

        if (rsethAmountToMint < minRSETHAmountExpected) {
            revert MinimumAmountToReceiveNotMet();
        }
