### Title
Missing Daily Mint Limit in `RSETHPoolNoWrapper` Enables Unlimited rsETH Extraction Against Stale Cross-Chain Oracle Rate - (File: contracts/pools/RSETHPoolNoWrapper.sol)

### Summary
`RSETHPoolNoWrapper` distributes pre-minted rsETH to depositors using a cross-chain oracle rate that is inherently stale. Unlike `RSETHPoolV3`, which enforces a `dailyMintLimit` cap, `RSETHPoolNoWrapper` imposes no per-day ceiling on rsETH outflow. When the L2 oracle rate lags behind the true L1 rate (rsETH has appreciated but the cross-chain message has not yet arrived), any depositor can extract rsETH at a discount — bounded only by the pool's total rsETH balance — with no rate-limiting mechanism to contain the damage.

### Finding Description
`RSETHPoolV3.deposit()` is guarded by the `limitDailyMint` modifier, which computes the rsETH amount that would be minted and reverts if it would exceed `dailyMintLimit` for the current 24-hour window. [1](#0-0) 

`RSETHPoolNoWrapper.deposit()` carries no equivalent guard: [2](#0-1) 

The rsETH amount given to the depositor is computed as:

```
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate
```

where `rsETHToETHrate` comes from `IOracle(rsETHOracle).getRate()`. [3](#0-2) 

The oracle wired to `RSETHPoolNoWrapper` is a cross-chain rate receiver (`RSETHRateReceiver`) that stores the last rate pushed from L1 via LayerZero. The L1 source is `RSETHRateProvider`, which reads `ILRTOracle(rsETHPriceOracle).rsETHPrice()` — a **stored, manually-updated** value in `LRTOracle`. [4](#0-3) [5](#0-4) 

`rsETHPrice` is only refreshed when `updateRSETHPrice()` is called on L1, and the updated value only reaches L2 after a LayerZero message is relayed. Between updates the L2 rate is stale. Because rsETH continuously appreciates (staking rewards accrue), the L2 rate is structurally lower than the true L1 rate during every inter-update window.

### Impact Explanation
When the L2 oracle rate `R_stale` is lower than the true rate `R_true`, a depositor of `A` ETH receives:

```
rsETH_received = A / R_stale   >   A / R_true   (= fair rsETH amount)
```

The excess rsETH — representing the yield that has accrued to existing holders since the last oracle update — is transferred out of the pool's pre-minted rsETH reserves. Because there is no daily mint ceiling, a single depositor can drain the entire pool balance in one transaction, stealing all accrued-but-not-yet-reflected yield from every existing rsETH holder. This maps directly to **High: theft of unclaimed yield**.

### Likelihood Explanation
The cross-chain oracle is structurally stale by design: every period between a LayerZero push from L1 and the next one creates an exploitable window. rsETH appreciates continuously (~4 % APY), so even a 24-hour staleness window produces a ~0.011 % discount per unit. With no daily cap, an attacker can amplify this by depositing an arbitrarily large amount of ETH in a single call, extracting the full accrued-yield surplus from the pool in one transaction. The attack requires no special permissions, no oracle manipulation, and no front-running — only a deposit call during any normal inter-update window.

### Recommendation
Add a `dailyMintLimit` state variable and a `limitDailyMint` modifier to `RSETHPoolNoWrapper`, mirroring the implementation already present in `RSETHPoolV3`. [6](#0-5) [1](#0-0) 

Apply the modifier to both `deposit(string)` and `deposit(address,uint256,string)` in `RSETHPoolNoWrapper`. Additionally, consider adding a staleness check on the cross-chain oracle rate (e.g., revert if the last update timestamp exceeds a configurable threshold).

### Proof of Concept
1. Assume the L2 oracle last received `R_stale = 1.040 ETH/rsETH` 20 hours ago.
2. On L1, staking rewards have pushed the true rate to `R_true = 1.041 ETH/rsETH`.
3. Attacker calls `RSETHPoolNoWrapper.deposit{value: 10_000 ETH}("")`.
4. Contract computes `rsETHAmount = 10_000e18 * 1e18 / 1.040e18 = 9_615.38 rsETH`.
5. Fair amount at true rate: `10_000e18 / 1.041e18 = 9_606.15 rsETH`.
6. Attacker receives **9.23 rsETH extra** (~9.61 ETH of value) in a single call, with no daily cap to prevent it.
7. Attacker repeats or scales the deposit to drain the pool's entire rsETH balance. [2](#0-1) [3](#0-2) [4](#0-3)

```text
    modifier limitDailyMint(uint256 amount, address token) {
        if (block.timestamp < startTimestamp) {
            revert MintBeforeStartTimestamp();
        }

        uint256 rsETHAmount;

        // Calculate the amount of rsETH that will be minted
        if (token == ETH_IDENTIFIER) {
            (rsETHAmount,) = viewSwapRsETHAmountAndFee(amount);
        } else {
            (rsETHAmount,) = viewSwapRsETHAmountAndFee(amount, token);
        }

        uint256 currentDay = getCurrentDay();

        // If the current day is greater than the last mint day, reset the daily mint amount
        if (currentDay > lastMintDay) {
            lastMintDay = currentDay;
            dailyMintAmount = 0;
        }

        // Check if the daily mint amount plus the amount to mint is greater than the daily mint limit
        if (dailyMintAmount + rsETHAmount > dailyMintLimit) {
            revert DailyMintLimitExceeded();
        }

        dailyMintAmount += rsETHAmount;
        _;
    }
```
