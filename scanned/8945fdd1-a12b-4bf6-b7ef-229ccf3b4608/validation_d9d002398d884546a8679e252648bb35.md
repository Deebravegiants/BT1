### Title
Deposit Minimum-Amount Guard Does Not Account for Exchange-Rate Reduction, Enabling Zero-rsETH Deposits That Permanently Absorb User Funds - (File: contracts/pools/RSETHPoolV3.sol)

---

### Summary

Every L2 pool `deposit()` function guards against zero-value inputs with `if (amount == 0) revert InvalidAmount()`, but never verifies that the **output** `rsETHAmount` — computed after fee deduction and exchange-rate division — is also non-zero. Because integer division truncates, a depositor sending a small but non-zero ETH/token amount can receive exactly 0 wrsETH/rsETH while their funds are permanently absorbed into the pool.

---

### Finding Description

In `RSETHPoolV3.deposit()` (ETH path) the full execution path is:

```
// Guard — checks raw input only
if (amount == 0) revert InvalidAmount();           // line 256

// Fee deduction
fee           = amount * feeBps / 10_000;          // line 300
amountAfterFee = amount - fee;                     // line 301

// Exchange-rate conversion — can silently truncate to 0
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;  // line 307

// Mint — succeeds even when rsETHAmount == 0
wrsETH.mint(msg.sender, rsETHAmount);              // line 262
```

`rsETHToETHrate` is the ETH value of 1 rsETH (e.g. `1.05 × 10¹⁸` after 5 % yield accrual). Whenever `amountAfterFee × 10¹⁸ < rsETHToETHrate`, integer division yields `rsETHAmount = 0`. The `limitDailyMint` modifier also computes `rsETHAmount` the same way and adds 0 to `dailyMintAmount`, so it does not revert either.

The identical pattern exists in every pool variant:

| Contract | ETH path | Token path |
|---|---|---|
| `RSETHPoolV3.sol` | line 307 | line 334 |
| `RSETHPool.sol` | line 319 | line 346 |
| `RSETHPoolNoWrapper.sol` | line 285 | line 311 |
| `RSETHPoolV3ExternalBridge.sol` | line 426 | line 450 |
| `RSETHPoolV3WithNativeChainBridge.sol` | line 343 | line 370 |

This is structurally identical to the SecondSwap M-02 root cause: a minimum-amount check is performed on the **pre-reduction** value, while the actual output is computed from a **post-reduction** value that can round to zero.

---

### Impact Explanation

A depositor whose `rsETHAmount` rounds to zero:
- Has their ETH/tokens transferred into the pool (irreversible from the user's side).
- Receives 0 wrsETH/rsETH — the contract delivers nothing in return.
- Cannot recover the deposit; the ETH is eventually bridged to L1 and credited to the protocol treasury, not the depositor.

This is a permanent, direct loss of user funds for any deposit below the rounding threshold.

**Impact: Low** — Contract fails to deliver promised returns; deposited value is lost to the protocol rather than returned.

---

### Likelihood Explanation

The rounding threshold is `amountAfterFee < rsETHToETHrate / 1e18`. At a 5 % yield accrual (`rsETHToETHrate ≈ 1.05 × 10¹⁸`), any ETH deposit of 1 wei with zero fee rounds to zero. As rsETH continues to appreciate the threshold grows, but remains in the sub-wei range for ETH. For token deposits the threshold scales with `rsETHToETHrate / tokenToETHRate`, which can be larger if the deposited token has a lower ETH value than rsETH.

Any unprivileged depositor calling `deposit()` with a dust amount triggers this path. Normal users are unlikely to do so intentionally, but the path is fully reachable without any privilege.

---

### Recommendation

Add a post-calculation guard in every `deposit()` function (and in `viewSwapRsETHAmountAndFee`) to revert when the computed output is zero:

```solidity
if (rsETHAmount == 0) revert InvalidAmount();
```

This mirrors the fix recommended in the SecondSwap report: validate the **output** amount (after all reductions) rather than only the raw input.

---

### Proof of Concept

1. `rsETHToETHrate = 1.05e18` (rsETH has appreciated 5 %).
2. Depositor calls `RSETHPoolV3.deposit{value: 1}("")` — 1 wei ETH.
3. `amount = 1` passes `if (amount == 0)` guard. [1](#0-0) 
4. `feeBps ≤ 1000`, so `fee = 0`, `amountAfterFee = 1`.
5. `rsETHAmount = 1 × 1e18 / 1.05e18 = 0` (integer truncation). [2](#0-1) 
6. `limitDailyMint` adds 0 to `dailyMintAmount` — no revert. [3](#0-2) 
7. `wrsETH.mint(msg.sender, 0)` executes successfully — depositor receives 0 wrsETH. [4](#0-3) 
8. Depositor's 1 wei ETH is permanently absorbed by the pool.

The same path is reachable in `RSETHPool.deposit()` [5](#0-4)  and `RSETHPoolNoWrapper.deposit()` [6](#0-5)  with identical arithmetic. [7](#0-6)
