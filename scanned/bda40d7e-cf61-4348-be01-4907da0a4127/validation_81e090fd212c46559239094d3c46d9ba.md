### Title
Predictable UID Generation via On-Chain Values Causes Bridge Withdrawal Collision - (File: contracts/bridges/SonicChainNativeTokenBridge.sol)

### Summary
`SonicChainNativeTokenBridge.bridgeTokenToL1` generates a `uint96` UID for each Sonic bridge withdrawal using only fully predictable on-chain values (`block.timestamp`, `block.number`, `msg.sender`, `recipient`, `amount`, `tx.gasprice`, `bridgeReceiver`). Two calls in the same block with identical parameters produce an identical UID. When the Sonic bridge receives a duplicate UID, the second withdrawal either reverts (causing the bridge operation to fail after token transfer) or silently overwrites the first withdrawal record, permanently losing the first bridging operation's funds.

### Finding Description
In `bridgeTokenToL1`, the UID passed to `sonicBridge.withdraw` is computed as:

```solidity
uint96 uid = uint96(
    uint256(
        keccak256(
            abi.encodePacked(
                block.timestamp, block.number, msg.sender, recipient, amount, tx.gasprice, bridgeReceiver
            )
        )
    ) % type(uint96).max   // <-- modulo bias: never produces type(uint96).max
);
```

Every input to the hash is either a block-level constant (`block.timestamp`, `block.number`), a fixed immutable (`bridgeReceiver`), or a caller-controlled value (`msg.sender`, `recipient`, `amount`, `tx.gasprice`). There is no nonce, counter, or unpredictable entropy source. Two calls sharing the same block and the same parameter set produce an identical UID. Additionally, the modulo operation uses `type(uint96).max` (`2^96 - 1`) instead of `2^96`, introducing a bias that makes the value `2^96 - 1` unreachable.

The fallback UID at line 104 drops `block.timestamp`, `recipient`, and `amount` from the hash, making it even weaker and more collision-prone. [1](#0-0) 

### Impact Explanation
The Sonic bridge uses the `uid` as a unique identifier for each withdrawal. If two calls produce the same UID:

- **If the Sonic bridge reverts on duplicate UIDs**: The second `sonicBridge.withdraw(uid, ...)` call at line 115 reverts. Because the token transfer at line 83 already succeeded, the entire transaction reverts and the caller's tokens are returned. The bridge operation is silently dropped, causing a **temporary freezing of the bridging flow** until a retry with different parameters (e.g., different gas price or different block) succeeds.
- **If the Sonic bridge silently overwrites on duplicate UIDs**: The first withdrawal record is overwritten by the second, and the first batch of bridged tokens is permanently unclaimable on L1 — a **permanent freezing of funds**. [2](#0-1) 

### Likelihood Explanation
The pool contracts (`RSETHPoolV3WithNativeChainBridge`, `RSETHPool`, `RSETHPoolNoWrapper`, `RSETHPoolV3ExternalBridge`) call `bridgeTokenToL1` via `bridgeTokens`, which is restricted to `BRIDGER_ROLE`. The `msg.sender` is always the pool contract address, `recipient` is always `l1VaultETHForL2Chain`, and `amount` is the full token balance. In the same block, `block.timestamp` and `block.number` are fixed. The only differentiator is `tx.gasprice`. If the BRIDGER_ROLE submits two bridge transactions in the same block with the same gas price (e.g., via a script or keeper bot), the UIDs collide. This is a realistic operational scenario. [3](#0-2) 

### Recommendation
Replace the hash-based UID with a monotonically incrementing counter stored in contract state:

```solidity
uint96 private _uidCounter;

// Inside bridgeTokenToL1:
uint96 uid = ++_uidCounter;
```

This guarantees uniqueness across all calls regardless of block context, eliminates the modulo bias, and removes all predictability concerns. If a hash-based approach is preferred, include `_uidCounter` as an additional input to the hash to ensure uniqueness.

### Proof of Concept
1. BRIDGER_ROLE calls `bridgeTokens(token)` on the pool contract, which calls `bridgeTokenToL1(l1VaultETHForL2Chain, balance)` on `SonicChainNativeTokenBridge`.
2. In the same block, BRIDGER_ROLE submits a second `bridgeTokens(token)` call with the same gas price.
3. Both calls hash `(block.timestamp, block.number, poolAddress, l1VaultETHForL2Chain, balance, tx.gasprice, bridgeReceiver)` — all identical values — producing the same `uid`.
4. The first call succeeds: `sonicBridge.withdraw(uid, originalToken, balance)` registers the withdrawal.
5. The second call passes the same `uid` to `sonicBridge.withdraw`. Depending on the Sonic bridge implementation, this either reverts (dropping the second bridge operation) or overwrites the first withdrawal record (losing the first batch of tokens on L1). [4](#0-3)
