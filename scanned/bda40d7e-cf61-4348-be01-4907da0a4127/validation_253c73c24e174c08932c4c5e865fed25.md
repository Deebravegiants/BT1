### Title
No User-Controlled Minimum Return Amount in Pool Deposit Functions - (`contracts/pools/RSETHPoolNoWrapper.sol`, `contracts/pools/RSETHPoolV3.sol`)

### Summary
The `deposit` functions in `RSETHPoolNoWrapper` and `RSETHPoolV3` compute the rsETH amount returned to the caller using live oracle rates, but accept no `minRsETHAmount` parameter. A price-feed update mined before the user's deposit transaction causes the user to receive fewer rsETH tokens than they anticipated, with no on-chain mechanism to revert the transaction.

### Finding Description
`RSETHPoolNoWrapper.deposit(string)` and `RSETHPoolNoWrapper.deposit(address,uint256,string)` compute the rsETH payout entirely from oracle rates at execution time: [1](#0-0) 

For the token variant, two independent oracle rates are consumed: [2](#0-1) 

The same pattern exists in `RSETHPoolV3`: [3](#0-2) [4](#0-3) 

Neither `deposit` overload in either contract accepts a `minRsETHAmount` parameter. The caller has no way to express a lower bound on the rsETH they are willing to accept.

By contrast, the core `LRTDepositPool` already enforces this pattern correctly via `minRSETHAmountExpected`: [5](#0-4) 

### Impact Explanation
If the rsETH oracle rate rises (or a token oracle rate falls) between the moment the user signs the transaction and the moment it is mined, the computed `rsETHAmount` decreases. The user's ETH or tokens are consumed in full, but the rsETH minted/transferred is less than the user expected. Because the rsETH price is bounded by `pricePercentageLimit` in `LRTOracle`, the per-transaction shortfall is bounded, but it is a real, unrecoverable loss of value for the depositor. The contract fails to deliver the return the user observed when constructing the transaction.

**Impact: Low** — contract fails to deliver promised returns without losing the principal in absolute terms, but the user receives fewer rsETH shares than they are entitled to at the price they observed.

### Likelihood Explanation
The rsETH oracle price is updated by the protocol on a regular cadence via `updateRSETHPrice`. Any deposit transaction that lands in the same block as or after a price update is affected. This is a routine, non-adversarial scenario requiring no special attacker capability — it occurs naturally whenever the mempool is congested or the user's gas price is low. The token-variant path is additionally exposed to Chainlink heartbeat updates for the collateral token oracle.

### Recommendation
**Short term:** Add a `minRsETHAmount` parameter to both `deposit` overloads in `RSETHPoolNoWrapper` and `RSETHPoolV3`, and revert if the computed `rsETHAmount` is below it:

```solidity
require(rsETHAmount >= minRsETHAmount, "Slippage: insufficient rsETH out");
```

**Long term:** Adopt the pattern already used in `LRTDepositPool.depositAsset` and `LRTDepositPool.depositETH` uniformly across all user-facing deposit entry points. Any function that transfers the caller's funds and returns an oracle-priced asset should expose a caller-controlled minimum-output guard.

### Proof of Concept
1. Alice calls `viewSwapRsETHAmountAndFee(1 ether)` on `RSETHPoolNoWrapper` and observes she will receive `X` rsETH at the current oracle rate.
2. Alice submits `deposit{value: 1 ether}("ref")`.
3. Before Alice's transaction is mined, the protocol operator calls `updateRSETHPrice()` on `LRTOracle`, raising the rsETH/ETH rate by the maximum allowed `pricePercentageLimit`.
4. Alice's transaction executes. `viewSwapRsETHAmountAndFee` now returns `X' < X` rsETH because `rsETHToETHrate` is higher.
5. Alice receives `X'` rsETH — fewer shares than she expected — with no revert and no recourse. [6](#0-5) [7](#0-6)
