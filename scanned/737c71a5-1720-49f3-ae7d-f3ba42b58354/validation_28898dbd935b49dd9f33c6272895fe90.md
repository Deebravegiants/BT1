### Title
Unrestricted `addLiquidity` Enables Cheap Griefing Attack That Permanently Blocks LP Full-Withdrawal — (`metric-core/contracts/MetricOmmPool.sol` / `metric-core/contracts/libraries/LiquidityLib.sol`)

---

### Summary

`addLiquidity` imposes no `msg.sender == owner` restriction, so any caller can inject shares into any `(owner, salt, bin)` position. `removeLiquidity` enforces a `MinimalLiquidity` guard that reverts when the post-removal share balance is non-zero but below `MINIMAL_MINTABLE_LIQUIDITY`. An attacker can front-run a victim's "remove all shares" transaction by adding a tiny number of shares to the victim's position, causing the victim's transaction to revert because the residual balance falls below the minimum. The attack costs the attacker only the token value of one share and can be repeated indefinitely.

---

### Finding Description

**Asymmetric access control between add and remove:**

`addLiquidity` accepts an arbitrary `owner` address with no ownership check: [1](#0-0) 

`removeLiquidity` correctly enforces `msg.sender == owner`: [2](#0-1) 

**`MinimalLiquidity` guard in `removeLiquidity`:**

Inside `LiquidityLib.removeLiquidity`, after computing `newUserShares = userShares - sharesToRemove`, the pool reverts if the result is non-zero but below the pool's `MINIMAL_MINTABLE_LIQUIDITY` constant (set to `1000` in all test fixtures): [3](#0-2) 

**`addLiquidity` minimum check does not block the attack:**

When Bob adds shares to Alice's existing position, the add-side check is:

```solidity
uint256 newUserShares = userShares + sharesToAdd;
if (newUserShares < ctx.minimalMintableLiquidity) { revert ... }
``` [4](#0-3) 

Because Alice already holds `>= MINIMAL_MINTABLE_LIQUIDITY` shares, adding even 1 share keeps `newUserShares` above the threshold, so the add succeeds.

**Attack flow:**

1. Alice holds `N` shares (e.g. `N = 10 000`) in bin `b` under `(alice, salt)`.
2. Alice broadcasts `removeLiquidity(alice, salt, {bin: b, shares: 10000}, "")`.
3. Bob sees the transaction in the mempool. He calls `addLiquidity(alice, salt, {bin: b, shares: 1}, ...)`, paying the token cost of 1 share (negligible).
4. Alice's position is now `10 001` shares.
5. Alice's transaction executes: `newUserShares = 10001 - 10000 = 1`. Since `1 > 0 && 1 < 1000`, the pool reverts with `MinimalLiquidity(1, 1000)`.
6. Bob recovers his 1-share deposit by calling `removeLiquidity` on his own position — but he used Alice's `(owner, salt)` key, so the tokens are credited to Alice's position, not Bob's. Bob cannot recover them. **However**, Bob can use a different `salt` value that he controls, or simply accept the cost as a griefing expenditure.

Actually, re-examining: Bob calls `addLiquidity(alice, alice_salt, ...)`. The shares are credited to `positionBinShares[keccak256(alice, alice_salt, bin)]`. Bob cannot call `removeLiquidity` on that key because `removeLiquidity` requires `msg.sender == owner == alice`. So Bob's tokens are permanently donated to Alice's position — but the attack still succeeds in blocking Alice's withdrawal. The cost to Bob is the token value of 1 share per attack round, which is negligible for most pools. [5](#0-4) 

---

### Impact Explanation

An LP who attempts to fully exit a bin position is permanently blocked from doing so as long as an attacker is willing to spend the token cost of 1 share per attempt. The LP's funds are not stolen, but they are rendered inaccessible via the normal `removeLiquidity` path. The LP can only escape by submitting a transaction that removes `N + 1` shares (matching the attacker's donation), but the attacker can repeat the attack for each new attempt. This constitutes broken core pool functionality causing loss of usable withdraw flow, meeting the contest's "broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows" impact gate.

---

### Likelihood Explanation

- The victim's `(owner, salt, bin, shares)` tuple is fully visible in the mempool.
- The attacker needs only a trivial token balance (1 share worth).
- No special role or privileged access is required.
- The attack is repeatable at negligible cost per round.
- Likelihood: **Medium** (requires mempool monitoring and front-running capability, which is standard on most EVM chains).

---

### Recommendation

**Option A (preferred):** Add a `msg.sender == owner` check to `addLiquidity`, consistent with `removeLiquidity`. Third-party deposits on behalf of an owner can be handled via an explicit allowance or extension mechanism.

**Option B:** In `removeLiquidity`, skip the `MinimalLiquidity` check when `sharesToRemove == userShares` (i.e., the user is removing their entire position). A full exit to zero is always safe and should not be blocked by the dust floor.

```solidity
uint256 newUserShares = userShares - sharesToRemove;
// Only enforce minimum if a non-zero residual remains
if (newUserShares > 0 && newUserShares < ctx.minimalMintableLiquidity) {
    revert IMetricOmmPoolActions.MinimalLiquidity(newUserShares, ctx.minimalMintableLiquidity);
}
```

This is already the current code — the fix for Option B is to allow the user to pass `sharesToRemove = userShares` (their full current balance, not the stale value from before the front-run). This requires the caller to read the current on-chain share count immediately before submitting, or to use a "remove all" sentinel value.

**Option C:** Accept a `minSharesRemaining` slippage parameter in `removeLiquidity` so the transaction reverts if the attacker has inflated the position beyond what the user expected.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Assume pool is deployed with MINIMAL_MINTABLE_LIQUIDITY = 1000
// Alice has 10_000 shares in bin 4, salt = 12345

// Step 1: Alice broadcasts this transaction
pool.removeLiquidity(alice, 12345, LiquidityDelta({
    binIdxs: [4],
    shares:  [10_000]
}), "");

// Step 2: Bob front-runs with addLiquidity(owner=alice, salt=12345, shares=1)
// Bob must implement IMetricOmmModifyLiquidityCallback and pay 1 share worth of token0
pool.addLiquidity(alice, 12345, LiquidityDelta({
    binIdxs: [4],
    shares:  [1]          // costs Bob ~1 share of token0, negligible
}), callbackData, "");

// Alice's position is now 10_001 shares.

// Step 3: Alice's transaction executes:
//   newUserShares = 10_001 - 10_000 = 1
//   1 > 0 && 1 < 1000  →  revert MinimalLiquidity(1, 1000)
// Alice's withdrawal is blocked.
``` [1](#0-0) [3](#0-2) [2](#0-1)
