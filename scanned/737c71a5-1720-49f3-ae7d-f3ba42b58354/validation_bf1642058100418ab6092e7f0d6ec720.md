### Title
`isUpdater` Permissions Not Reset on Provider Ownership Transfer, Allowing Stale Updaters to Manipulate Oracle Spread — (`smart-contracts-poc/contracts/PriceProviderFactory.sol`, `PriceProviderFactoryL2.sol`, `AnchoredProviderFactory.sol`)

### Summary

All three factory contracts store a `mapping(address provider => mapping(address updater => bool)) public isUpdater` that is never cleared during `transferProviderOwnership`. A previous owner can pre-grant updater rights to a controlled address before transferring, then use those rights post-transfer to call `setConfidence` and manipulate `confidenceParam` on the provider — widening the bid/ask spread to an extreme or collapsing it to zero, causing bad-price execution or a full pool DoS.

### Finding Description

`PriceProviderFactory`, `PriceProviderFactoryL2`, and `AnchoredProviderFactory` each hold:

```solidity
mapping(address provider => mapping(address updater => bool)) public isUpdater;
``` [1](#0-0) 

The `transferProviderOwnership` function updates `providerOwner` and the `_providersByCreator` enumerable sets, but never touches `isUpdater`:

```solidity
function transferProviderOwnership(address provider, address newOwner) external override onlyProviderOwner(provider) {
    ...
    providerOwner[provider] = newOwner;
    _providersByCreator[previousOwner].remove(provider);
    _providersByCreator[newOwner].add(provider);
    emit ProviderOwnershipTransferred(provider, previousOwner, newOwner);
}
``` [2](#0-1) [3](#0-2) [4](#0-3) 

The updater authorization check used by `setConfidence` is:

```solidity
function _requireUpdater(address provider) internal view {
    if (msg.sender != providerOwner[provider] && !isUpdater[provider][msg.sender])
        revert NotProviderUpdater();
}
``` [5](#0-4) 

Because `isUpdater` is a plain nested mapping (not enumerable), the new owner has no way to enumerate which addresses were granted updater rights. They can only revoke permissions for addresses they already know about.

`setConfidence` calls `PriceProvider.setConfidenceParam`, which sets `confidenceParam` — a multiplier applied directly to the oracle spread before computing bid/ask:

```solidity
uint256 adjustedSpread = spread * confidenceParam;
(uint256 bid, uint256 ask) = _getBidAskFrom(mid, adjustedSpread);
``` [6](#0-5) 

`confidenceParam` is bounded by `CONFIDENCE_MAX = 1_000_000` (100× multiplier). Setting it to the maximum makes the spread 100× wider; setting it to `0` collapses the spread to zero, making `bid == ask`, which triggers the `bidOut >= askOut` guard and returns `(0, type(uint128).max)` — the stalled-feed sentinel — causing `getBidAndAskPrice()` to revert with `FeedStalled`. [7](#0-6) [8](#0-7) 

### Impact Explanation

- **Bad-price execution**: A stale updater sets `confidenceParam` to `CONFIDENCE_MAX`. The spread fed to the pool becomes 100× the oracle spread, producing an extremely wide bid/ask. Swappers receive prices far outside the fair market range, suffering direct principal loss on every swap routed through this provider.
- **Pool DoS**: A stale updater sets `confidenceParam` to `0`. The spread collapses, `bid >= ask`, the provider returns the stalled sentinel, and every swap against the pool reverts with `FeedStalled` — making the pool's swap and liquidity flows completely unusable until the new owner discovers and revokes the stale updater.

Both outcomes are within the allowed impact gate: bad-price execution and broken core pool functionality.

### Likelihood Explanation

The trigger requires a previous owner who deliberately pre-grants updater rights before transferring. This is a semi-trusted actor scenario (the previous owner was trusted enough to own the provider). The `CONFIDENCE_COOLDOWN` of 1 minute limits update frequency but does not prevent sustained manipulation. The new owner has no enumeration path to discover stale updaters, making remediation non-trivial.

### Recommendation

Reset all updater grants during ownership transfer. Because `isUpdater` is not enumerable, the simplest fix is to introduce an `updaterEpoch` counter per provider and include it in the authorization check:

```solidity
mapping(address provider => uint256) public updaterEpoch;
mapping(address provider => mapping(uint256 epoch => mapping(address updater => bool))) public isUpdater;

function transferProviderOwnership(address provider, address newOwner) external ... {
    ...
    updaterEpoch[provider]++;   // invalidates all prior grants atomically
    ...
}
```

Alternatively, store updaters in an enumerable set per provider so the new owner (or the transfer function itself) can iterate and clear them.

### Proof of Concept

1. Alice owns a `PriceProvider` deployed via `PriceProviderFactory`.
2. Alice calls `grantUpdater(provider, alice_bot)` — `isUpdater[provider][alice_bot] = true`.
3. Alice calls `transferProviderOwnership(provider, bob)` — `providerOwner[provider] = bob`, but `isUpdater[provider][alice_bot]` remains `true`.
4. Bob is unaware of `alice_bot`.
5. `alice_bot` calls `factory.setConfidence([provider], [0])`. `_requireUpdater` passes because `isUpdater[provider][alice_bot]` is still `true`.
6. `confidenceParam` is now `0`. The next call to `pool.swap(...)` invokes `priceProvider.getBidAndAskPrice()`, which computes `adjustedSpread = 0`, returns `(0, type(uint128).max)`, and reverts with `FeedStalled` — the pool is DoS'd.
7. Alternatively in step 5, `alice_bot` sets confidence to `CONFIDENCE_MAX = 1_000_000`, making the spread 100× wider. Swappers now execute at severely distorted prices, losing principal on every trade. [9](#0-8) [10](#0-9)
