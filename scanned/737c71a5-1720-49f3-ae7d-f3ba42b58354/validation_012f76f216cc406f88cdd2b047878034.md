Looking at the Metric OMM codebase for an analog to the "wrong party bears accrued costs during a multi-step delay" bug class. The analog maps to fee accounting in `collectFees`: notional fees accrue during swaps but are double-counted when fees are collected, draining LP principal.

---

### Title
`collectFees` double-counts notional fees against the spread surplus, draining LP principal — (`metric-core/contracts/MetricOmmPool.sol`)

### Summary

When both spread fees and notional fees are active, `collectFees` distributes the notional fee amount **twice**: once embedded inside the spread surplus (`balance − binTotals`) and once from the `notionalFeeToken0Scaled`/`notionalFeeToken1Scaled` accumulators. The excess is drawn from LP funds (`binTotals`). The protocol's own test suite explicitly asserts this over-allocation occurs.

### Finding Description

**Swap accounting creates the double-count.** During `_executeSwap`, the notional fee is withheld from the trader's output token but is **not** deducted from `binTotals`: [1](#0-0) 

`binTotals.scaledToken1` is reduced by the **full** pre-notional output. Then the notional fee is applied to `amount1DeltaScaled` (making it less negative) and accumulated: [2](#0-1) 

Because `binTotals` was reduced by the full output but the pool only transferred `output − notionalFee` to the trader, the notional fee sits in the pool balance **above** `binTotals`. It is therefore included in the spread surplus:

```
surplus = balance − binTotals = spread_fee + notional_fee
```

**`collectFees` then distributes both:** [3](#0-2) 

```
total_distributed = surplus + notionalFeeAccumulator
                  = (spread_fee + notional_fee) + notional_fee
                  = spread_fee + 2 × notional_fee
```

Only `spread_fee + notional_fee` is available as fees. The extra `notional_fee` is pulled from `binTotals` — LP principal.

**The protocol's own test documents this invariant break:** [4](#0-3) 

```solidity
assertGt(total0Attempted, surplus0Scaled, "token0 attempted payout exceeds computed surplus");
assertGt(total1Attempted, surplus1Scaled, "token1 attempted payout exceeds computed surplus");
```

The test explicitly asserts the over-allocation happens — it is not a guard, it is documentation of the bug.

### Impact Explanation

LP funds (`binTotals`) are drained by exactly `notionalFeeToken0Scaled + notionalFeeToken1Scaled` on every `collectFees` call when both fee types are active. Since `collectFees` is **permissionless** (any keeper or attacker may call it), the drain can be triggered repeatedly. Over time this accumulates to a direct, measurable loss of LP principal — pool insolvency relative to LP claims.

### Likelihood Explanation

Medium. Requires both `spreadFeeE6 > 0` and `notionalFeeE8 > 0` simultaneously, which is a valid and expected production configuration (the factory supports both). `collectFees` is permissionless per the protocol docs ("any address may call"), so no privileged access is needed to trigger the drain.

### Recommendation

Exclude the notional fee from the spread surplus before computing spread fee distributions. In `collectFees`, subtract the notional accumulators from the surplus before splitting:

```solidity
uint256 spreadSurplus0Scaled = surplus0Scaled - notionalFeeToken0Scaled;
uint256 spreadSurplus1Scaled = surplus1Scaled - notionalFeeToken1Scaled;
// use spreadSurplus* for spread fee math, notionalFeeToken*Scaled for notional fee math
```

Alternatively, deduct the notional fee from `binTotals` at swap time (analogous to how the spread fee is deducted), so it never appears in the surplus.

### Proof of Concept

1. Deploy pool with `adminSpreadFeeE6 = 10_000` and `adminNotionalFeeE8 = 1_000_000` (1%).
2. LP adds liquidity — `binTotals.scaledToken1 = L`.
3. Trader executes exact-in swap: generates `spread_fee` in surplus and `notional_fee` in both surplus and `notionalFeeToken1Scaled`.
4. Record `binTotals.scaledToken1 = L` (unchanged by fee accrual).
5. Call `collectFees` (permissionless).
6. `collectFees` transfers `surplus1 + notionalFeeToken1Scaled = (spread_fee + notional_fee) + notional_fee`.
7. The extra `notional_fee` is pulled from `binTotals` → LP can now only withdraw `L − notional_fee`.
8. Repeat step 3–7 each block; LP principal drains monotonically.

The mathematical proof is already encoded in `test_collectProtocolFees_math_overallocates_whenSpreadAndNotionalBothActive`: [5](#0-4)
