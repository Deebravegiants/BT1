### Title
Attacker Can Consume the Daily Mint Limit to DOS All User Deposits for Up to 24 Hours - (`contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`, `contracts/pools/RSETHPoolV3WithNativeChainBridge.sol`)

---

### Summary

The `limitDailyMint` modifier enforces a shared `dailyMintLimit` across all depositors. Because any unprivileged depositor can consume the entire remaining daily quota in a single transaction, an attacker can front-run victim deposits to fill the limit and cause all subsequent `deposit()` calls to revert with `DailyMintLimitExceeded` for up to 24 hours. The attacker receives wrsETH at fair value in return, so the only cost is opportunity cost and secondary-market slippage.

---

### Finding Description

All three V3 pool variants share an identical `limitDailyMint` modifier:

```solidity
// contracts/pools/RSETHPoolV3.sol  lines 96-125
modifier limitDailyMint(uint256 amount, address token) {
    ...
    if (dailyMintAmount + rsETHAmount > dailyMintLimit) {
        revert DailyMintLimitExceeded();
    }
    dailyMintAmount += rsETHAmount;
    _;
}
```

The modifier is applied to both `deposit(string)` (ETH) and `deposit(address,uint256,string)` (ERC-20) in every pool variant. [1](#0-0) 

`dailyMintAmount` is a single shared counter. There is no per-user quota, no minimum deposit size, and no mechanism to prevent one address from consuming the entire remaining capacity. [2](#0-1) 

The protocol also exposes a public view function that tells any caller exactly how much capacity remains:

```solidity
// contracts/pools/RSETHPoolV3.sol  lines 344-350
function remainingDailyMintLimit() external view returns (uint256) {
    uint256 effectiveDailyMintAmount = (getCurrentDay() > lastMintDay) ? 0 : dailyMintAmount;
    return dailyMintLimit - effectiveDailyMintAmount;
}
``` [3](#0-2) 

**Attack steps:**

1. Attacker calls `remainingDailyMintLimit()` to read the exact remaining quota `R`.
2. Attacker front-runs a victim's `deposit()` by depositing assets worth `R` rsETH (or slightly more than `R - victim_amount`).
3. `dailyMintAmount` is now equal to `dailyMintLimit`.
4. Victim's `deposit()` executes: `dailyMintAmount + rsETHAmount > dailyMintLimit` → reverts with `DailyMintLimitExceeded`.
5. Attacker holds wrsETH received at fair value and can sell on secondary markets to recover capital.
6. The attack can be repeated at the start of every new day (the limit resets when `currentDay > lastMintDay`).

The same modifier and the same attack surface exist verbatim in `RSETHPoolV3ExternalBridge.sol` (lines 130–159) and `RSETHPoolV3WithNativeChainBridge.sol` (lines 108–137). [4](#0-3) [5](#0-4) 

---

### Impact Explanation

All user `deposit()` calls revert for the remainder of the 24-hour window (up to ~24 hours). Users who need to deposit during that window are completely blocked. This constitutes a **temporary freezing of user access to the protocol's core deposit function**, matching the "Temporary freezing of funds" impact category (Medium).

---

### Likelihood Explanation

The attacker receives wrsETH at fair value for their deposit, so the net cost is only opportunity cost plus any secondary-market slippage when unwinding the wrsETH position. The `remainingDailyMintLimit()` view function makes the exact amount to deposit trivially computable on-chain. The attack can be executed by any EOA or contract with sufficient capital, requires no special permissions, and can be repeated every day. Likelihood is **Medium**.

---

### Recommendation

1. **Partial fill instead of revert**: When `dailyMintAmount + rsETHAmount > dailyMintLimit`, mint only up to the remaining limit and refund the excess to the caller, rather than reverting the entire transaction.
2. **Per-block or per-tx rate limiting**: Limit how much a single address can mint within a given window to raise the cost of the attack.
3. **Remove the public `remainingDailyMintLimit()` helper** or make it harder to use atomically (though this is only a minor deterrent).

---

### Proof of Concept

Assume `dailyMintLimit = 1000e18` rsETH and `dailyMintAmount = 0` at the start of a day.

1. Victim submits `deposit{value: 500 ETH}("ref")` targeting ~500e18 rsETH.
2. Attacker reads `remainingDailyMintLimit()` → `1000e18`.
3. Attacker front-runs with `deposit{value: 501 ETH}("ref")` → `dailyMintAmount` becomes `~501e18`; attacker receives `~501e18` wrsETH.
4. Victim's tx executes: `dailyMintAmount(501e18) + rsETHAmount(500e18) = 1001e18 > dailyMintLimit(1000e18)` → `revert DailyMintLimitExceeded()`. [6](#0-5) 
5. Attacker sells `~501e18` wrsETH on a secondary market, recovering most of the capital.
6. At the next day boundary (`getCurrentDay() > lastMintDay`), `dailyMintAmount` resets to 0 and the attacker can repeat. [7](#0-6)

    modifier limitDailyMint(uint256 amount, address token) {
        if (block.timestamp < startTimestamp) {
            revert MintBeforeStartTimestamp();
        }

        uint256 rsETHAmount;

        // Calculate the amount of rsETH that will be minted
        if (token == ETH_IDENTIFIER) {
            (rsETHAmount,) = viewSwapRsETHAmountAndFee(amount);
        } else {
            (rsETHAmount,) = viewSwapRsETHAmountAndFee(amount, token);
        }

        uint256 currentDay = getCurrentDay();

        // If the current day is greater than the last mint day, reset the daily mint amount
        if (currentDay > lastMintDay) {
            lastMintDay = currentDay;
            dailyMintAmount = 0;
        }

        // Check if the daily mint amount plus the amount to mint is greater than the daily mint limit
        if (dailyMintAmount + rsETHAmount > dailyMintLimit) {
            revert DailyMintLimitExceeded();
        }

        dailyMintAmount += rsETHAmount;
        _;
    }
```
