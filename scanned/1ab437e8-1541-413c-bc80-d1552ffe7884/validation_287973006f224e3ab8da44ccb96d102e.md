### Title
Missing Storage Gap in `LRTConfigRoleChecker` Base Contract Enables Storage Collision on Upgrade — (File: contracts/utils/LRTConfigRoleChecker.sol)

---

### Summary

`LRTConfigRoleChecker` is an abstract upgradeable base contract that declares a state variable (`lrtConfig`) but contains no `__gap` reserved storage. It is inherited by every major protocol contract. If a future upgrade adds new state variables to `LRTConfigRoleChecker`, the storage layout of all child contracts will be silently corrupted, overwriting critical variables such as deposit limits, withdrawal queues, and rsETH balances.

---

### Finding Description

`LRTConfigRoleChecker` declares one storage variable at slot 0:

```solidity
ILRTConfig public lrtConfig;
``` [1](#0-0) 

There is no `uint256[N] private __gap;` at the end of this contract. The grep search across all `contracts/**/*.sol` files confirms zero occurrences of `__gap` in any production contract.

This base contract is inherited by every core upgradeable contract in the protocol:

| Child Contract | Critical State at Slot 1 (after `lrtConfig`) |
|---|---|
| `LRTDepositPool` | `maxNodeDelegatorLimit` |
| `NodeDelegator` | `eigenPod` |
| `RSETH` | `maxMintAmountPerDay` |
| `LRTOracle` | `assetPriceOracle` mapping |
| `LRTWithdrawalManager` | `minRsEthAmountToWithdraw` mapping |
| `LRTUnstakingVault` | `__deprecated_sharesUnstaking` mapping |
| `PubkeyRegistry` | `pubkeyRegistry` mapping | [2](#0-1) [3](#0-2) [4](#0-3) 

If a new state variable is added to `LRTConfigRoleChecker` in any future upgrade, the EVM storage layout shifts: the new variable occupies slot 0, `lrtConfig` moves to slot 1, and every subsequent child variable shifts down by one slot. The proxy's existing storage is not rewritten, so all child contracts will silently read `lrtConfig` from slot 0 — which now holds the new variable's raw bytes — and read their own first variable from slot 1, which holds the old `lrtConfig` address value cast as a `uint256`.

---

### Impact Explanation

**Impact: Medium — Temporary or permanent freezing of funds / protocol insolvency.**

A storage collision in `lrtConfig` would cause every role-check modifier (`onlyLRTAdmin`, `onlyLRTManager`, `onlyLRTOperator`, `onlySupportedAsset`) to resolve against a corrupted config address. All privileged operations — deposits, withdrawals, unstaking, oracle updates — would revert or behave incorrectly. Funds already held in `LRTDepositPool`, `LRTUnstakingVault`, and `NodeDelegator` contracts would be inaccessible. Depending on the specific collision, rsETH minting/burning could also be broken, causing protocol insolvency. [5](#0-4) 

---

### Likelihood Explanation

**Likelihood: Low.**

The collision is triggered only when the protocol deploys an upgrade that adds a new state variable to `LRTConfigRoleChecker`. This is a legitimate and expected admin action (the contracts are explicitly designed as upgradeable with `_disableInitializers()` constructors and `reinitializer` functions already in use). The protocol has already performed multiple reinitializations (`initialize2`, `initialize3` in `LRTWithdrawalManager`; `initialize2` in `NodeDelegator`; `reinitialize` in `RSETH` and `LRTOracle`), demonstrating active upgrade cadence. The absence of a `__gap` makes any future base-contract variable addition a silent storage bomb. [6](#0-5) [7](#0-6) 

---

### Recommendation

Add a `__gap` array at the end of `LRTConfigRoleChecker`. The size should account for the one slot already used by `lrtConfig`:

```solidity
abstract contract LRTConfigRoleChecker {
    ILRTConfig public lrtConfig;          // slot 0

    // ... modifiers unchanged ...

    uint256[49] private __gap;            // reserves slots 1–49
}
```

Similarly, add appropriately sized `__gap` arrays to each concrete upgradeable contract (`LRTDepositPool`, `NodeDelegator`, `RSETH`, `LRTOracle`, `LRTWithdrawalManager`, `LRTUnstakingVault`, `PubkeyRegistry`, `FeeReceiver`, `RsETHTokenWrapper`) so that future additions of state variables to those contracts do not collide with storage in any contracts that may inherit from them.

---

### Proof of Concept

1. Current storage layout of `LRTDepositPool` (simplified):
   - Slot 0: `lrtConfig` (from `LRTConfigRoleChecker`)
   - Slot 1: `_paused` (from `PausableUpgradeable`, packed)
   - Slot 2+: `maxNodeDelegatorLimit`, `minAmountToDeposit`, `isNodeDelegator`, `nodeDelegatorQueue`, `maxNegligibleAmount`

2. Protocol upgrades `LRTConfigRoleChecker` to add `address public newConfigVar` before `lrtConfig`.

3. New implementation storage layout:
   - Slot 0: `newConfigVar` (new variable in base)
   - Slot 1: `lrtConfig` (shifted)
   - Slot 2+: all child variables shifted by one

4. The proxy's existing storage is unchanged. `LRTDepositPool.lrtConfig` now reads slot 0, which contains `newConfigVar`'s value (e.g., `address(0)` or garbage). Every call to `onlyLRTAdmin`, `onlyLRTManager`, etc. resolves against `address(0)`, causing all privileged operations to revert permanently.

5. `maxNodeDelegatorLimit` (previously slot 1 in the child) is now read from slot 2, which held `_paused` — a boolean packed as `uint256(1)` — causing `maxNodeDelegatorLimit` to appear as `1`, silently breaking `addNodeDelegatorContractToQueue`. [8](#0-7) [1](#0-0)
