### Title
Temporary Freezing of Remaining Staked KERNEL Balance When All Withdrawal Slots Are Filled With Max-Delay Withdrawals — (`contracts/KERNEL/KernelDepositPool.sol`)

---

### Summary

A user who fills all `maxNumberOfWithdrawalsPerUser` withdrawal slots with max-delay (30-day) withdrawals will have their remaining `balanceOf` staked KERNEL permanently inaccessible for up to 30 days. There is no `cancelWithdrawal` function, no admin escape hatch, and no alternative path to retrieve the remaining staked balance.

---

### Finding Description

`initiateWithdrawal` enforces a hard cap on open (unclaimed) withdrawal slots: [1](#0-0) 

Once the cap is reached, any further call to `initiateWithdrawal` reverts with `WithdrawalLimitReached`. Meanwhile, `claimWithdrawal` enforces a time lock: [2](#0-1) 

The `withdrawalDelay` is admin-configurable up to `MAX_WITHDRAWAL_DELAY = 30 days`: [3](#0-2) 

When a user calls `initiateWithdrawal`, the amount is immediately subtracted from `balanceOf`: [4](#0-3) 

The contract provides **no** `cancelWithdrawal` function and no admin function to forcibly remove a pending withdrawal or override the slot limit. The only functions that touch `balanceOf` are `stake`, `stakeFor`, and `initiateWithdrawal`. Once all slots are filled and all `unlockTime`s are in the future, the remaining `balanceOf[user]` is completely inaccessible. [5](#0-4) 

---

### Impact Explanation

**Medium — Temporary freezing of funds.**

A user's remaining staked KERNEL balance (`balanceOf[user] > 0`) becomes inaccessible for up to `MAX_WITHDRAWAL_DELAY = 30 days`. The user cannot:
- Call `initiateWithdrawal` → reverts `WithdrawalLimitReached`
- Call `claimWithdrawal` on any slot → reverts `WithdrawalNotReady`
- Access the remaining staked balance through any other function

The funds are not permanently lost (they become accessible once the earliest `unlockTime` passes), but the freeze duration can be up to 30 days, which is the defined maximum.

---

### Likelihood Explanation

**Low-Medium.** The scenario requires:
1. Admin has set `withdrawalDelay` to a large value (up to 30 days).
2. A user fills all `maxNumberOfWithdrawalsPerUser` slots (configurable up to `MAX_WITHDRAWALS_PER_USER = 100`) with partial withdrawals, leaving a non-zero `balanceOf`. [6](#0-5) 

This can happen accidentally (a user who routinely initiates many small withdrawals) or deliberately (a user who misunderstands the slot limit). It does not require any external attacker — the user's own actions are sufficient to trigger the freeze.

---

### Recommendation

Add a `cancelWithdrawal(uint256 _withdrawalId)` function that:
1. Verifies the caller owns the withdrawal and it has not been claimed.
2. Removes the withdrawal ID from `userWithdrawalIds[msg.sender]`.
3. Returns the `amount` back to `balanceOf[msg.sender]` and `totalKernelStaked`.

This gives users an escape hatch to free up slots and recover their staked balance without waiting for the delay to expire.

---

### Proof of Concept

```solidity
// Preconditions:
// - withdrawalDelay = 30 days (MAX_WITHDRAWAL_DELAY)
// - maxNumberOfWithdrawalsPerUser = N (e.g., 10)
// - user has staked 1000e18 KERNEL

// Step 1: Fill all N slots with partial withdrawals
for (uint i = 0; i < N; i++) {
    pool.initiateWithdrawal(1e18); // 1 KERNEL each
}
// balanceOf[user] = 1000e18 - N*1e18 > 0
// userWithdrawalIds[user].length == N == maxNumberOfWithdrawalsPerUser
// All unlockTimes = block.timestamp + 30 days

// Step 2: Assert remaining balance is frozen
assert(pool.balanceOf(user) > 0);

// Step 3: initiateWithdrawal reverts
vm.expectRevert(KernelDepositPool.WithdrawalLimitReached.selector);
pool.initiateWithdrawal(1e18);

// Step 4: claimWithdrawal reverts for all slots (unlockTime in future)
uint256[] memory ids = pool.getUserWithdrawalIds(user);
for (uint i = 0; i < ids.length; i++) {
    vm.expectRevert(KernelDepositPool.WithdrawalNotReady.selector);
    pool.claimWithdrawal(ids[i]);
}
// User's remaining staked balance is inaccessible for up to 30 days.
```

    function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
        if (_amount == 0) revert AmountZero();
        if (balanceOf[msg.sender] < _amount) revert InsufficientStakedBalance();
        if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();

        balanceOf[msg.sender] -= _amount;
        totalKernelStaked -= _amount;

        // Create a withdrawal record
        uint256 withdrawalId = ++withdrawalCounter;
        uint256 unlockTime = block.timestamp + withdrawalDelay;

        withdrawals[withdrawalId] = Withdrawal({
            user: msg.sender, amount: _amount, unlockTime: unlockTime, claimed: false, withdrawalId: withdrawalId
        });
        userWithdrawalIds[msg.sender].push(withdrawalId);

        emit WithdrawalInitiated(msg.sender, _amount, withdrawalId, unlockTime);
    }
