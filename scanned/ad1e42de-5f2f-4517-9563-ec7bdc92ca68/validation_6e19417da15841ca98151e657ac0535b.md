### Title
Missing Token Decimal Normalization in L2 Pool `viewSwapRsETHAmountAndFee` Causes Near-Total Loss of Deposited Non-18-Decimal Tokens - (File: `contracts/pools/RSETHPool.sol`, `RSETHPoolV3ExternalBridge.sol`, and all pool variants)

---

### Summary

Every L2 RSETHPool variant computes the rsETH output for a token deposit with the formula:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

`tokenToETHRate` is always normalized to 1e18 precision (price of **1 whole token** in ETH). `amountAfterFee` is in the token's **native decimals**. For 18-decimal tokens the two scales cancel correctly. For any token with fewer decimals (e.g., 8 for WBTC/cbBTC, 6 for USDC) the result is scaled down by `10^(18 − decimals)`, causing the depositor to receive a near-zero rsETH amount while their full token balance is transferred into the pool.

---

### Finding Description

`ChainlinkOracleForRSETHPoolCollateral.getRate()` normalizes the Chainlink answer to 1e18 regardless of the feed's own decimals:

```solidity
uint256 normalizedPrice = uint256(ethPrice) * 1e18
    / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
```

This returns the ETH value of **one whole token** expressed in 1e18 precision (e.g., `30e18` for 1 WBTC = 30 ETH).

The pool's token deposit path then does:

```solidity
// RSETHPool.sol  line 346
// RSETHPoolV3ExternalBridge.sol  line 452
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

For a 18-decimal token `amountAfterFee` is already in 1e18 units, so the division by `rsETHToETHrate` (also 1e18) yields the correct 1e18-precision rsETH amount.

For an 8-decimal token `amountAfterFee` is in 1e8 units. The multiplication by `tokenToETHRate` (1e18) gives a 1e26 intermediate, and dividing by `rsETHToETHrate` (1e18) yields a 1e8-precision result — exactly `10^10` times smaller than the correct 1e18-precision answer.

No decimal normalization step exists anywhere in the deposit path. The `deposit(token, amount, referralId)` function transfers the full `amount` from the user before calling `viewSwapRsETHAmountAndFee`, so the tokens are already gone when the near-zero rsETH is minted.

The same structural flaw is present in every pool variant:

### Impact Explanation

**Critical — direct theft of user funds.**

A depositor who sends 1 WBTC (8 decimals, `amount = 1e8`, market value ≈ 30 ETH) receives approximately `2.94e8` rsETH (≈ `2.94 × 10^{-10}` rsETH in human units) instead of the correct `≈ 29.4e18` rsETH. The deposited WBTC is held by the pool and eventually bridged to L1; the user has no recourse to recover it. The loss ratio is `10^{10}:1`.

---

### Likelihood Explanation

**Medium.** The pools expose an admin-controlled `supportedTokenOracle` mapping that allows any ERC-20 to be listed as a deposit token. The protocol's stated roadmap includes support for LSTs and BTC-backed tokens. Once any non-18-decimal token is listed (a routine governance action, not a compromise), every subsequent depositor of that token is immediately and silently harmed. No attacker action is required beyond calling the public `deposit(token, amount, referralId)` function.

---

### Recommendation

Normalize `amountAfterFee` to 18 decimals before applying the rate ratio:

```solidity
uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 normalizedAmount = amountAfterFee * 1e18 / 10 ** tokenDecimals;
rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
```

Apply the same fix to the fee calculation so `feeEarnedInToken` remains in native token decimals while the rsETH output is in 1e18.

---

### Proof of Concept

**Setup:** Admin lists WBTC (8 decimals) as a supported token in `RSETHPoolV3ExternalBridge` with a `ChainlinkOracleForRSETHPoolCollateral` pointing to the WBTC/ETH feed. WBTC/ETH = 30, so `tokenToETHRate = 30e18`. Current rsETH price = 1.02 ETH, so `rsETHToETHrate = 1.02e18`. `feeBps = 0` for simplicity.

**Attacker (victim) action:** Call `deposit(WBTC, 1e8, "")` — depositing 1 WBTC.

**Execution trace:**

```
amountAfterFee = 1e8
tokenToETHRate = 30e18
rsETHToETHrate = 1.02e18

rsETHAmount = 1e8 * 30e18 / 1.02e18
           = 3e27 / 1.02e18
           ≈ 2.94e8          ← minted to user
```

**Correct value:**

```
normalizedAmount = 1e8 * 1e18 / 1e8 = 1e18   (1 WBTC in 1e18)
rsETHAmount = 1e18 * 30e18 / 1.02e18 ≈ 29.4e18
```

**Result:** User deposits 1 WBTC (≈ 30 ETH of value) and receives `2.94e8` rsETH ≈ `0.000000000294` rsETH. The shortfall is `≈ 29.4e18 − 2.94e8 ≈ 29.4e18` rsETH — effectively the entire deposit value is lost. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
