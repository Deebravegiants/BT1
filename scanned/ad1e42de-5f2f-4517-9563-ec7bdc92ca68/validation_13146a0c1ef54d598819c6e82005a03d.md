### Title
Uninitialized `maxNumberOfWithdrawalsPerUser` Permanently Blocks All Withdrawal Initiations — (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary
`KernelDepositPool.initiateWithdrawal()` always reverts with `WithdrawalLimitReached` immediately after deployment because `maxNumberOfWithdrawalsPerUser` is never set in `initialize()` and defaults to `0`. Every KERNEL staker's withdrawal attempt is blocked until an admin manually calls `setMaxNumberOfWithdrawalsPerUser()`.

### Finding Description
`maxNumberOfWithdrawalsPerUser` is declared as a storage variable but is never assigned in `initialize()`: [1](#0-0) 

The guard in `initiateWithdrawal()` is: [2](#0-1) 

When `maxNumberOfWithdrawalsPerUser == 0` (the default), the condition `userWithdrawalIds[msg.sender].length >= 0` evaluates to `true` for every caller (array length is always `≥ 0`), so every call reverts with `WithdrawalLimitReached` before any withdrawal record is created.

The setter enforces `_maxNumberOfWithdrawalsPerUser != 0`, confirming `0` is an invalid operational value: [3](#0-2) 

The `initialize()` function sets up roles and token addresses but never touches `maxNumberOfWithdrawalsPerUser`: [4](#0-3) 

### Impact Explanation
Any user who has staked KERNEL tokens via `stake()` or `stakeFor()` cannot initiate a withdrawal of their principal. Their staked balance is locked in the contract until an admin calls `setMaxNumberOfWithdrawalsPerUser()`. This constitutes a **temporary freezing of funds** for all KERNEL stakers — matching the "Medium. Temporary freezing of funds" impact tier.

### Likelihood Explanation
The contract is deployed and initialized in a single transaction. Because `maxNumberOfWithdrawalsPerUser` is not set in `initialize()`, the broken state is the default state of every fresh deployment. Any staker who attempts to withdraw before the admin separately calls `setMaxNumberOfWithdrawalsPerUser()` will have their funds frozen. The window of exposure is the entire period between deployment and that admin call.

### Recommendation
Initialize `maxNumberOfWithdrawalsPerUser` to a safe non-zero default inside `initialize()`:

```solidity
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    ...
    maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER; // e.g., 100
}
```

This ensures the contract is immediately operational after deployment without requiring a separate admin transaction.

### Proof of Concept
1. Admin deploys and initializes `KernelDepositPool`. `maxNumberOfWithdrawalsPerUser` is `0`.
2. Alice calls `stake(100e18)`. Her balance is recorded: `balanceOf[Alice] = 100e18`.
3. Alice calls `initiateWithdrawal(100e18)`.
4. The check `userWithdrawalIds[Alice].length >= maxNumberOfWithdrawalsPerUser` evaluates to `0 >= 0 → true`.
5. Transaction reverts with `WithdrawalLimitReached`.
6. Alice's 100e18 KERNEL tokens remain locked in the contract with no way to retrieve them until the admin calls `setMaxNumberOfWithdrawalsPerUser(n)` for some `n > 0`. [5](#0-4)

    function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
        if (_amount == 0) revert AmountZero();
        if (balanceOf[msg.sender] < _amount) revert InsufficientStakedBalance();
        if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();

        balanceOf[msg.sender] -= _amount;
        totalKernelStaked -= _amount;

        // Create a withdrawal record
        uint256 withdrawalId = ++withdrawalCounter;
        uint256 unlockTime = block.timestamp + withdrawalDelay;

        withdrawals[withdrawalId] = Withdrawal({
            user: msg.sender, amount: _amount, unlockTime: unlockTime, claimed: false, withdrawalId: withdrawalId
        });
        userWithdrawalIds[msg.sender].push(withdrawalId);

        emit WithdrawalInitiated(msg.sender, _amount, withdrawalId, unlockTime);
    }
