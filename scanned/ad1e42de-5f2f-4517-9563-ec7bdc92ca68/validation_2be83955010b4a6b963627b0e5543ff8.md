### Title
Stale Chainlink Oracle Price Enables Depositing Depegged LSTs at Inflated Rates to Drain rsETH from L2 Pools - (File: contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol)

---

### Summary

`ChainlinkOracleForRSETHPoolCollateral.getRate()` lacks a time-based staleness check. During an LST depeg event, if Chainlink's price feed stops updating within its heartbeat window, the oracle returns the last known (pre-depeg) price. Any unprivileged depositor can then call `deposit(token, amount, referralId)` on any L2 pool variant, depositing depegged LSTs at the inflated stale price and receiving more rsETH/wrsETH than the deposited LSTs are worth. The protocol is left holding depegged collateral while having issued excess rsETH.

---

### Finding Description

`ChainlinkOracleForRSETHPoolCollateral` is the oracle wrapper used by all L2 pool variants (`RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPool`, `RSETHPoolNoWrapper`) to price supported collateral tokens (e.g., wstETH) when users deposit them in exchange for rsETH or wrsETH.

Its `getRate()` function:

```solidity
function getRate() public view returns (uint256) {
    (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();

    if (answeredInRound < roundID) revert StalePrice();
    if (timestamp == 0) revert IncompleteRound();
    if (ethPrice <= 0) revert InvalidPrice();

    uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
    return normalizedPrice;
}
```

The only staleness guard is `answeredInRound < roundID`. Chainlink has deprecated this field — it is no longer reliably populated and does not catch the scenario where the oracle simply has not been updated within its expected heartbeat period. There is no check of the form `block.timestamp - timestamp > heartbeat`, which is the standard guard against a price feed that has gone silent. [1](#0-0) 

This oracle's `getRate()` return value is consumed directly in `viewSwapRsETHAmountAndFee(amount, token)` across all L2 pool variants:

```solidity
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [2](#0-1) [3](#0-2) [4](#0-3) 

If `tokenToETHRate` is stale and higher than the actual market price of the LST, the depositor receives more rsETH than the LST is worth.

---

### Impact Explanation

**Critical — Direct theft of protocol funds / protocol insolvency.**

The L2 pools hold pre-minted rsETH/wrsETH inventory (or mint it on demand). When a depositor provides a depegged LST priced at a stale inflated rate, they receive rsETH whose redemption value on L1 exceeds the actual value of the deposited LST. The pool accumulates depegged collateral while rsETH holders' claims are diluted. Repeated exploitation across multiple pools and multiple depositors can render the protocol insolvent.

---

### Likelihood Explanation

LST depeg events are not hypothetical — stETH traded at a significant discount to ETH in 2022, and slashing events can cause sudden LST price drops. During a rapid depeg:

1. Chainlink's deviation threshold may not be crossed immediately (e.g., if the depeg is gradual or the threshold is set at 0.5%–1%).
2. Network congestion can delay oracle updates.
3. The heartbeat window (typically 24 hours for LST/ETH feeds) means the oracle can be legally "fresh" by the `answeredInRound` check while being hours behind the market.

An attacker monitoring LST prices on-chain can act within this window. The attack requires no special role — only the ability to call the public `deposit` function.

---

### Recommendation

Add a time-based staleness check in `ChainlinkOracleForRSETHPoolCollateral.getRate()` using a configurable `heartbeat` parameter:

```solidity
uint256 public immutable heartbeat;

constructor(address _oracle, uint256 _heartbeat) {
    oracle = _oracle;
    heartbeat = _heartbeat;
}

function getRate() public view returns (uint256) {
    (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();

    if (answeredInRound < roundID) revert StalePrice();
    if (timestamp == 0) revert IncompleteRound();
    if (block.timestamp - timestamp > heartbeat) revert StalePrice(); // <-- add this
    if (ethPrice <= 0) revert InvalidPrice();

    return uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
}
```

Set `heartbeat` to the Chainlink feed's documented heartbeat (e.g., 86400 seconds for a 24-hour feed) plus a small buffer.

### Proof of Concept

**Setup:** `RSETHPoolV3ExternalBridge` is deployed on Base with wstETH as a supported token. `ChainlinkOracleForRSETHPoolCollateral` wraps the wstETH/ETH Chainlink feed (heartbeat: 24 hours).

**Attack steps:**

1. wstETH depegs on the market to 0.80 ETH (e.g., due to a slashing event). Chainlink's feed last updated 6 hours ago at 1.05 ETH/wstETH and has not yet triggered a new round (deviation threshold not crossed, or network congested).

2. `ChainlinkOracleForRSETHPoolCollateral.getRate()` returns `1.05e18`. The `answeredInRound < roundID` check passes (the round is complete). `timestamp != 0`. `ethPrice > 0`. No revert.

3. Attacker acquires 1000 wstETH at market price (800 ETH worth) and calls:
   ```
   RSETHPoolV3ExternalBridge.deposit(wstETH, 1000e18, "")
   ```

4. `viewSwapRsETHAmountAndFee(1000e18, wstETH)` computes:
   - `tokenToETHRate = 1.05e18` (stale)
   - `rsETHToETHrate = ~1.05e18` (rsETH/ETH rate)
   - `rsETHAmount ≈ 1000 wrsETH` (representing ~1050 ETH of value)

5. Attacker receives ~1000 wrsETH worth ~1050 ETH, having deposited wstETH worth only ~800 ETH. Net profit: ~250 ETH at the pool's expense.

6. Pool holds 1000 wstETH worth 800 ETH but has issued wrsETH claims worth 1050 ETH — a 250 ETH shortfall. [1](#0-0) [5](#0-4) [6](#0-5)
