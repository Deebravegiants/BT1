### Title
Missing Decimal Normalization in Token-to-rsETH Swap Calculation Causes Severe Fund Loss for Non-18-Decimal Token Depositors - (File: contracts/pools/RSETHPoolV3.sol)

---

### Summary

`RSETHPoolV3.viewSwapRsETHAmountAndFee(uint256 amount, address token)` computes the rsETH output as `amountAfterFee * tokenToETHRate / rsETHToETHrate` without normalizing `amountAfterFee` to 18 decimals. The oracle (`ChainlinkOracleForRSETHPoolCollateral`) always returns a price scaled to 18 decimals (price of 1 full token in ETH). For any supported token whose `decimals() < 18`, the product is in `(tokenDecimals + 18)` decimal units instead of `(18 + 18)`, so the result is `10^(18 - tokenDecimals)` times smaller than the correct rsETH amount. A depositor of such a token receives essentially nothing in return for their full token value.

---

### Finding Description

`ChainlinkOracleForRSETHPoolCollateral.getRate()` normalizes the Chainlink price to 18 decimals:

```solidity
uint256 normalizedPrice = uint256(ethPrice) * 1e18
    / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
```

This returns the price of **1 full token** in ETH, expressed in 18-decimal units (e.g., `20e18` for WBTC ≈ 20 ETH).

`RSETHPoolV3.viewSwapRsETHAmountAndFee` then computes:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

For an 18-decimal token (stETH), `amountAfterFee = 1e18` for 1 token, so the product is `1e18 × 1e18 = 1e36`, divided by `rsETHToETHrate (1e18)` → `~0.952e18` rsETH. Correct.

For an 8-decimal token (WBTC), `amountAfterFee = 1e8` for 1 token, so the product is `1e8 × 20e18 = 20e26`, divided by `rsETHToETHrate (1.05e18)` → `~19.05e8` rsETH. This is `10^10` times smaller than the correct `~19.05e18` rsETH. The user deposits 1 WBTC (≈ 20 ETH of value) and receives `~0.0000000019` rsETH.

The same structural flaw exists in `RSETHPool.viewSwapRsETHAmountAndFee(uint256 amount, address token)` and in the reverse direction `viewSwapAssetToPremintedRsETH`, where the inverse error would cause the pool to pay out `10^10` times more tokens than owed.

---

### Impact Explanation

**Critical — Direct theft of user funds.**

Any user who calls `deposit(token, amount, referralId)` with a supported non-18-decimal token receives a rsETH amount that is `10^(18 - tokenDecimals)` times smaller than the correct value. For WBTC (8 decimals), a deposit of 1 WBTC (≈ 20 ETH) yields `~19.05e8` rsETH instead of `~19.05e18` rsETH. The deposited tokens remain in the pool; the user's rsETH balance is negligible. This constitutes permanent loss of the deposited principal.

---

### Likelihood Explanation

**Low-to-Medium.** The vulnerability is latent: it activates the moment any token with `decimals() != 18` is added via `addSupportedToken`. The current deployed supported tokens (stETH, rETH, cbETH, etc.) all have 18 decimals, so no user is currently harmed. However, the contract is explicitly designed to accept arbitrary ERC20 tokens, and the admin can add any token at any time without any decimal check. There is no guard in `addSupportedToken` that enforces `decimals() == 18`. Expansion to tokens such as WBTC (8 decimals) or USDC (6 decimals) is a realistic protocol evolution.

---

### Recommendation

Normalize `amountAfterFee` to 18 decimals before the multiplication, using the token's own `decimals()`:

```solidity
uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 normalizedAmount = amountAfterFee * 10 ** (18 - tokenDecimals);
rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
```

Apply the same fix to `viewSwapAssetToPremintedRsETH` (inverse direction) and to the identical logic in `RSETHPool.viewSwapRsETHAmountAndFee`. Alternatively, enforce `decimals() == 18` inside `addSupportedToken` to restrict the pool to 18-decimal tokens only.

---

### Proof of Concept

**Setup:** Admin calls `addSupportedToken(WBTC, wbtcOracle)` on `RSETHPoolV3`. `wbtcOracle` is a `ChainlinkOracleForRSETHPoolCollateral` wrapping the WBTC/ETH Chainlink feed (8-decimal answer, ~20 ETH per WBTC).

**Oracle rate:** `getRate()` returns `20e8 * 1e18 / 1e8 = 20e18`.

**User deposits 1 WBTC:**
- `amount = 1e8` (WBTC has 8 decimals)
- `fee = 0` (feeBps = 0 for simplicity)
- `amountAfterFee = 1e8`
- `rsETHToETHrate = 1.05e18`
- `tokenToETHRate = 20e18`
- `rsETHAmount = 1e8 * 20e18 / 1.05e18 = ~19.05e8`

**Expected:** `~19.05e18` rsETH (≈ 19.05 rsETH, correct for 20 ETH of WBTC at 1.05 ETH/rsETH).
**Actual:** `~19.05e8` rsETH ≈ `0.0000000019` rsETH.

The user loses ~20 ETH of value. The WBTC remains locked in the pool.

            AggregatorV3Interface(oracle).latestRoundData();

        if (answeredInRound < roundID) revert StalePrice();
        if (timestamp == 0) revert IncompleteRound();
