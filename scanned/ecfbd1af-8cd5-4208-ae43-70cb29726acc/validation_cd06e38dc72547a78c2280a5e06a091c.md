### Title
`RSETHPoolNoWrapper` Can Be Initialized Without Holding rsETH, Causing All User Deposits to Revert — (File: contracts/pools/RSETHPoolNoWrapper.sol)

### Summary
The `RSETHPoolNoWrapper` contract is designed for chains without a local rsETH wrapper and distributes rsETH directly from its own token balance to depositors. Its `initialize` function performs no check that the contract actually holds any rsETH. If the pool is deployed and initialized before being funded with rsETH (via bridging), every call to `deposit()` will revert, making the pool non-functional for all users.

### Finding Description
`RSETHPoolNoWrapper` differs from the other pool variants (`RSETHPoolV3`, `RSETHPoolV2ExternalBridge`, etc.) in a critical way: the other pools call `wrsETH.mint(msg.sender, rsETHAmount)` — they mint new wrapper tokens on demand. `RSETHPoolNoWrapper`, by contrast, holds the canonical rsETH OFT and transfers it from its own balance to depositors.

The `initialize` function sets up roles, oracle, and fee parameters, but performs zero balance verification: [1](#0-0) 

No `rsETH.balanceOf(address(this)) > 0` guard exists. The pool can be fully initialized and appear operational while holding zero rsETH.

The `deposit` function then computes `rsETHAmount` and attempts to transfer rsETH from the contract's balance: [2](#0-1) 

If the pool holds no rsETH at the time of the call, the transfer reverts, and every depositor's transaction fails.

This is the direct analog of the `VestingEscrowSimple` pattern: a contract initialized without verifying it holds the tokens it promises to distribute, causing the primary user-facing function to revert.

### Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

Users who call `deposit()` on an unfunded `RSETHPoolNoWrapper` receive a revert. Their ETH is returned (no permanent loss), but the pool is non-functional. Users are misled into believing the pool is operational (it is initialized and unpaused) when it cannot fulfill any swap.

### Likelihood Explanation
**Low.** The pool is expected to be funded with rsETH through the LayerZero/OFT bridging mechanism after deployment. However, there is a window between initialization and first funding during which the pool is live but non-functional. Any user who deposits during this window will have their transaction revert. Additionally, if the bridging pipeline stalls or the pool is deployed on a new chain without a funded bridge, the pool remains permanently non-functional with no on-chain signal to users.

### Recommendation
Add a balance check at the end of `initialize`, or alternatively, require an initial rsETH transfer as part of initialization:

```solidity
function initialize(...) external initializer {
    // ... existing setup ...
    if (rsETH.balanceOf(address(this)) == 0) revert InsufficientInitialBalance();
}
```

If fee-on-transfer rsETH variants are ever supported, store `rsETH.balanceOf(address(this))` as the authoritative initial balance rather than relying on a parameter.

### Proof of Concept
1. Deploy a UUPS proxy pointing to `RSETHPoolNoWrapper` implementation.
2. Call `initialize(admin, manager, rsETHAddress, feeBps, oracle, true)` — succeeds with zero rsETH balance.
3. Any user calls `deposit{value: 1 ether}("ref")`.
4. `viewSwapRsETHAmountAndFee` computes a non-zero `rsETHAmount`.
5. The subsequent `rsETH.transfer(msg.sender, rsETHAmount)` reverts because `rsETH.balanceOf(address(this)) == 0`.
6. User's ETH is returned, but the pool is non-functional until manually funded. [1](#0-0) [2](#0-1)
