### Title
`maxUncompletedWithdrawalCount` Cap Blocks EigenLayer Unstaking, Temporarily Freezing User Withdrawal Requests — (`contracts/NodeDelegator.sol`, `contracts/LRTUnstakingVault.sol`)

---

### Summary

`NodeDelegator.initiateUnstaking()` enforces a hard cap on the number of in-flight EigenLayer withdrawals via `maxUncompletedWithdrawalCount` (bounded at 80). When this cap is reached, operators cannot queue new EigenLayer withdrawals, blocking the flow of assets from EigenLayer strategies into `LRTUnstakingVault`. Because `LRTWithdrawalManager.unlockQueue()` requires assets to be present in `LRTUnstakingVault`, user withdrawal requests already queued (with rsETH transferred to the contract) cannot be serviced until existing EigenLayer withdrawals complete — a delay of up to 7 days per EigenLayer's withdrawal queue.

---

### Finding Description

The withdrawal lifecycle in LRT-rsETH is:

1. User calls `LRTWithdrawalManager.initiateWithdrawal()` — rsETH is transferred from the user to the `LRTWithdrawalManager` contract and a withdrawal request is recorded.
2. Operator calls `NodeDelegator.initiateUnstaking()` — queues a withdrawal from EigenLayer strategies.
3. After EigenLayer's delay, operator calls `NodeDelegator.completeUnstaking()` — assets arrive in `LRTUnstakingVault`.
4. Operator calls `LRTWithdrawalManager.unlockQueue()` — reads assets from `LRTUnstakingVault` and marks requests as unlocked.
5. User calls `LRTWithdrawalManager.completeWithdrawal()` — receives assets.

Step 2 is gated by the following check in `NodeDelegator.initiateUnstaking()`:

```solidity
if (_getUnstakingVault().uncompletedWithdrawalCount() >= _getUnstakingVault().maxUncompletedWithdrawalCount()) {
    revert MaxUncompletedWithdrawalsReached();
}
``` [1](#0-0) 

`maxUncompletedWithdrawalCount` is set by the LRT Manager and is hard-capped at 80:

```solidity
if (_maxUncompletedWithdrawalCount > 80) {
    revert MaxUncompletedWithdrawalCountTooHigh();
}
maxUncompletedWithdrawalCount = _maxUncompletedWithdrawalCount;
``` [2](#0-1) 

When `uncompletedWithdrawalCount` reaches `maxUncompletedWithdrawalCount`, all calls to `initiateUnstaking` revert. No new assets can be moved from EigenLayer to `LRTUnstakingVault`. Meanwhile, `unlockQueue` in `LRTWithdrawalManager` reads `unstakingVault.balanceOf(asset)` as `totalAvailableAssets`:

```solidity
return UnlockParams({
    ...
    totalAvailableAssets: unstakingVault.balanceOf(asset)
});
``` [3](#0-2) 

If the vault balance is zero (all assets are in EigenLayer), `unlockQueue` reverts with `AmountMustBeGreaterThanZero`:

```solidity
if (params.totalAvailableAssets == 0) revert AmountMustBeGreaterThanZero();
``` [4](#0-3) 

User withdrawal requests already queued (rsETH already transferred to `LRTWithdrawalManager`) cannot be completed. There is no `cancelWithdrawal` function, so users cannot recover their rsETH until the cap situation resolves.

The rsETH is transferred to the contract at `initiateWithdrawal` time:

```solidity
IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);
``` [5](#0-4) 

---

### Impact Explanation

User rsETH is held in `LRTWithdrawalManager` with no cancellation path. When the `maxUncompletedWithdrawalCount` cap is saturated, no new EigenLayer unstaking can be initiated, so no assets flow into `LRTUnstakingVault`, and no queued withdrawal requests can be unlocked or completed. Users experience a **temporary freeze of their funds** for the duration of EigenLayer's withdrawal delay (up to 7 days per batch), which can compound if the cap remains saturated across multiple cycles.

This matches the **Medium — Temporary freezing of funds** impact category.

---

### Likelihood Explanation

As the protocol scales, the number of concurrent EigenLayer unstaking operations grows proportionally with withdrawal demand. The cap of 80 is a fixed ceiling. During periods of elevated withdrawal demand (e.g., market stress, large rsETH holders exiting), operators will naturally saturate the cap through normal operations. Once saturated, every new user withdrawal request enters a queue that cannot be serviced until existing EigenLayer withdrawals complete. This is a realistic, non-adversarial scenario that mirrors the RageTrade senior vault high-utilization scenario.

---

### Recommendation

1. **Monitor and surface the cap utilization** — expose a view function showing `uncompletedWithdrawalCount / maxUncompletedWithdrawalCount` so operators can proactively manage the queue.
2. **Maintain a liquid buffer in `LRTUnstakingVault`** — keep a portion of assets outside EigenLayer to service withdrawal requests without requiring new `initiateUnstaking` calls.
3. **Add a withdrawal cancellation path** — allow users to cancel a queued withdrawal and reclaim their rsETH if the request has not yet been unlocked, preventing indefinite rsETH lockup.
4. **Consider raising or dynamically adjusting the cap** — the comment notes 120 is the gas-safe upper bound; the current hard cap of 80 leaves limited headroom.

---

### Proof of Concept

1. Protocol has 80 in-flight EigenLayer withdrawals (`uncompletedWithdrawalCount == maxUncompletedWithdrawalCount == 80`).
2. Alice calls `LRTWithdrawalManager.initiateWithdrawal(stETH, 10e18, "")` — her 10 rsETH is transferred to `LRTWithdrawalManager`, `assetsCommitted[stETH]` increases.
3. Operator attempts `NodeDelegator.initiateUnstaking([stETHStrategy], [shares])` to bring stETH from EigenLayer to `LRTUnstakingVault` — **reverts** with `MaxUncompletedWithdrawalsReached`.
4. `LRTUnstakingVault.balanceOf(stETH) == 0` (all stETH is in EigenLayer).
5. Operator attempts `LRTWithdrawalManager.unlockQueue(stETH, ...)` — **reverts** with `AmountMustBeGreaterThanZero`.
6. Alice's rsETH remains locked in `LRTWithdrawalManager`. She has no `cancelWithdrawal` option. She must wait until one of the 80 existing EigenLayer withdrawals completes (up to 7 days), freeing a slot for a new `initiateUnstaking` call.
