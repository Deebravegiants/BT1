### Title
Stale Chainlink Price Accepted Without Freshness Validation, Enabling Over-Minting of rsETH - (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards all return values except `price`. No check is made on `updatedAt` (time-based staleness) or `answeredInRound` vs `roundId` (round-completeness). A stale or frozen Chainlink feed will return an outdated price that the protocol accepts as valid, allowing a depositor to mint rsETH at an inflated rate.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the LST/ETH exchange rate used to compute how many rsETH tokens to mint per deposit:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L49-L55
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();   // updatedAt and answeredInRound discarded
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
``` [1](#0-0) 

The `updatedAt` timestamp and `answeredInRound` fields returned by `latestRoundData()` are both silently dropped. There is no guard of the form `require(block.timestamp - updatedAt <= heartbeat)` and no `require(answeredInRound >= roundId)`.

The codebase itself demonstrates the correct pattern in a sibling oracle wrapper:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L27-L32
(uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
    AggregatorV3Interface(oracle).latestRoundData();
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
``` [2](#0-1) 

`ChainlinkPriceOracle` applies none of these checks, making it the weaker of the two oracle wrappers in the codebase.

The stale price propagates through two critical paths:

**Path 1 — rsETH minting:**
`LRTDepositPool.getRsETHAmountToMint()` divides the stale asset price by `rsETHPrice` to compute how many rsETH tokens to mint per deposit unit. [3](#0-2) 

**Path 2 — rsETH price update:**
`LRTOracle._getTotalEthInProtocol()` calls `getAssetPrice(asset)` for every supported LST to compute total protocol TVL, which then sets `rsETHPrice`. [4](#0-3) 

---

### Impact Explanation

**Impact: Critical — Direct theft of user funds / protocol insolvency.**

Concrete scenario:

1. An LST supported by the protocol (e.g., stETH, rETH) experiences a depeg or sharp price drop.
2. The Chainlink feed for that LST goes stale (network congestion, oracle downtime, or the feed's heartbeat window is simply not yet expired).
3. `ChainlinkPriceOracle.getAssetPrice()` returns the old, inflated price.
4. An attacker calls `LRTDepositPool.depositAsset()` with the now-cheaper LST. `getRsETHAmountToMint()` computes `rsethAmountToMint = (depositAmount × stalePriceHigh) / rsETHPrice`, minting more rsETH than the deposited LST is worth.
5. The attacker redeems the excess rsETH for other protocol assets, extracting value from existing rsETH holders.

Additionally, when `updateRSETHPrice()` is called with a stale inflated price, `_getTotalEthInProtocol()` overstates TVL, causing `rsETHPrice` to be set too high. This inflated `rsETHPrice` then causes subsequent legitimate depositors to receive fewer rsETH tokens than they deserve, and may trigger the `PriceAboveDailyThreshold` guard incorrectly. [5](#0-4) 

---

### Likelihood Explanation

**Likelihood: Medium.**

Chainlink feeds have documented heartbeat intervals (e.g., 1 hour for stETH/ETH on mainnet). During periods of low volatility, the feed may not update for the full heartbeat window. During network congestion or oracle downtime, the feed can go stale for longer. The absence of any staleness check means the protocol silently accepts any price, no matter how old. This is a well-known class of vulnerability that has been exploited in production protocols.

---

### Recommendation

Add both a round-completeness check and a time-based staleness check in `ChainlinkPriceOracle.getAssetPrice()`, consistent with the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    // Use a per-feed heartbeat; 1 hour is a conservative default
    if (block.timestamp - updatedAt > 1 hours) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

The heartbeat threshold should be configurable per asset feed, since different Chainlink feeds have different update frequencies.

---

### Proof of Concept

1. Assume stETH/ETH Chainlink feed last updated 2 hours ago (stale, but `latestRoundData()` still returns the old price).
2. The real stETH price has dropped 5% since the last update (e.g., from 1.0 ETH to 0.95 ETH).
3. Attacker acquires 1000 stETH at market price (≈950 ETH worth).
4. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
5. `getRsETHAmountToMint` computes: `rsethAmountToMint = (1000e18 × 1.0e18) / rsETHPrice` — using the stale 1.0 ETH price instead of 0.95 ETH.
6. Attacker receives rsETH worth ~1000 ETH, having deposited only ~950 ETH worth of stETH.
7. Attacker redeems rsETH for ETH or other assets, extracting ~50 ETH from existing rsETH holders.

The entry path is fully permissionless: `depositAsset()` is callable by any address with no role requirement. [6](#0-5) [1](#0-0)
