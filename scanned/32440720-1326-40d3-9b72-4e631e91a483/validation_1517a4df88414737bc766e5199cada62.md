### Title
Missing Circuit Breaker in `CrossChainRateReceiver.lzReceive` Allows Depressed Rate to Over-Mint rsETH on L2 — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

---

### Summary

`CrossChainRateReceiver.lzReceive` unconditionally writes any received rate into storage with no monotonicity guard or circuit breaker. The L1 oracle (`LRTOracle._updateRsETHPrice`) has explicit downside protection that pauses the protocol when the price drops beyond `pricePercentageLimit`. No equivalent protection exists on the L2 receiver. Because `updateRate()` on `CrossChainRateProvider` is permissionless, an attacker can time the cross-chain push to coincide with a transient L1 price dip, locking a depressed rate into the L2 receiver. All L2 pools then mint excess rsETH against that depressed rate until the next update.

---

### Finding Description

**L1 circuit breaker (present):**

In `LRTOracle._updateRsETHPrice`, when the new price drops more than `pricePercentageLimit` below `highestRsethPrice`, the function pauses `LRTDepositPool`, `LRTWithdrawalManager`, and itself, then returns *without* updating `rsETHPrice`. [1](#0-0) 

For drops *within* the limit (or when `pricePercentageLimit == 0`, its default), `rsETHPrice` is updated to the lower value and L1 operations continue. [2](#0-1) 

**L2 receiver (no protection):**

`CrossChainRateReceiver.lzReceive` performs only source-chain and source-address authentication, then writes the payload rate directly: [3](#0-2) 

There is no check against the previous `rate`, no `highestRate` watermark, and no pause trigger.

**Rate provider is permissionless:**

`CrossChainRateProvider.updateRate()` is `external payable` with no role restriction. Anyone can call it at any time to push the current `LRTOracle.rsETHPrice` cross-chain. [4](#0-3) 

`RSETHRateProvider.getLatestRate()` reads the stored `rsETHPrice` from `LRTOracle`: [5](#0-4) 

**L2 pools use the rate directly for minting:**

`RSETHPoolV3.viewSwapRsETHAmountAndFee` computes `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate`. A depressed `rsETHToETHrate` inflates `rsETHAmount` proportionally. [6](#0-5) 

`RSETHPool` uses the identical formula: [7](#0-6) 

---

### Impact Explanation

If the L2 receiver stores a rate of `0.5e18` while the true rate is `1.05e18`, every ETH deposited mints `1e18 / 0.5e18 = 2` rsETH instead of `1e18 / 1.05e18 ≈ 0.952` rsETH — a ~2.1× over-issuance. The excess rsETH is unbacked. When the rate is corrected, existing holders can redeem more ETH than was deposited, draining the pool and causing protocol insolvency. The `dailyMintLimit` in `RSETHPoolV3` provides partial mitigation but does not prevent the attack within its daily cap.

**Impact: Critical — Protocol insolvency.**

---

### Likelihood Explanation

- `updateRate()` requires no role and costs only gas + LayerZero fee — any EOA can call it.
- `updateRSETHPrice()` on L1 is also public when unpaused, so an attacker can sequence both calls atomically (or within the same block) to capture a transient dip.
- If `pricePercentageLimit == 0` (the default until explicitly set by admin), the L1 circuit breaker never fires, and arbitrarily large drops can be pushed cross-chain.
- Even with `pricePercentageLimit` set, drops within the limit are pushed without any L2-side check.

---

### Recommendation

Add a circuit breaker inside `CrossChainRateReceiver.lzReceive` that mirrors the L1 logic:

1. Track `highestRate` (updated only when the new rate exceeds it).
2. Compute the percentage drop from `highestRate`.
3. If the drop exceeds a configurable `rateDropLimit`, either revert/ignore the update or emit an alert and pause the associated L2 pool.
4. Expose an owner-only `setRateDropLimit` and `pause`/`unpause` pair.

Additionally, restrict `CrossChainRateProvider.updateRate()` to a trusted keeper role to prevent adversarial timing of rate pushes.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Pseudocode for a local fork test (Hardhat/Foundry)

// 1. Deploy LRTOracle, RSETHRateProvider, RSETHRateReceiver, RSETHPoolV3
//    with pricePercentageLimit = 0 (default).

// 2. Set initial rsETHPrice = 1.05e18 via updateRSETHPrice().

// 3. Simulate underlying asset price drop so that
//    _getTotalEthInProtocol() / rsethSupply = 0.5e18.

// 4. Call LRTOracle.updateRSETHPrice() → rsETHPrice = 0.5e18
//    (no pause because pricePercentageLimit == 0).

// 5. Call RSETHRateProvider.updateRate{value: lzFee}()
//    → LayerZero delivers payload(0.5e18) to RSETHRateReceiver.lzReceive()
//    → RSETHRateReceiver.rate = 0.5e18  (no check, no pause)

// 6. Attacker calls RSETHPoolV3.deposit{value: 1 ether}("")
//    rsETHAmount = 1e18 * 1e18 / 0.5e18 = 2e18  ← receives 2 rsETH

// 7. True backing per rsETH at correct rate 1.05e18:
//    1 ETH should mint 1e18 / 1.05e18 ≈ 0.952e18 rsETH

// 8. Assert: attacker holds 2e18 rsETH backed by only 1 ETH.
//    Excess = 2e18 - 0.952e18 ≈ 1.048e18 rsETH unbacked → insolvency.
```

The two sequential `lzReceive` calls described in the question (`[1.05e18, 0.5e18]`) are sufficient to demonstrate the invariant break: after the second call, `RSETHRateReceiver.rate == 0.5e18` with no revert, no pause, and no event distinguishable from a legitimate update. [8](#0-7) [9](#0-8)
