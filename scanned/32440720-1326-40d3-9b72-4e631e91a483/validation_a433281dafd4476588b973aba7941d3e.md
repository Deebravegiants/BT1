### Title
Instant Withdrawal Consumes Assets Committed to Queued Withdrawal Users Due to Unsynchronized Accounting - (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary

`instantWithdrawal` in `LRTWithdrawalManager` draws assets from `LRTUnstakingVault` using a manually-set `queuedWithdrawalsBuffer` guard that is never synchronized with `assetsCommitted`. When `queuedWithdrawalsBuffer` is zero (the default) and instant withdrawals are enabled, any rsETH holder can drain vault assets that are already committed to queued withdrawal users, causing those users' `unlockQueue` calls to revert and their funds to be frozen.

---

### Finding Description

The protocol maintains two independent accounting systems for the same pool of assets in `LRTUnstakingVault`:

**System 1 — `assetsCommitted` (in `LRTWithdrawalManager`)**
When a user calls `initiateWithdrawal`, the expected asset amount is added to `assetsCommitted[asset]`: [1](#0-0) 

`getAvailableAssetAmount` uses this to prevent over-commitment of new queued withdrawals: [2](#0-1) 

**System 2 — `queuedWithdrawalsBuffer` (in `LRTUnstakingVault`)**
`instantWithdrawal` checks a completely separate guard — `getAssetsAvailableForInstantWithdrawal` — which uses a manually-set `queuedWithdrawalsBuffer` that defaults to zero: [3](#0-2) [4](#0-3) 

`queuedWithdrawalsBuffer` is set by operators via `setQueuedWithdrawalsBuffer` and is never automatically updated when `assetsCommitted` changes: [5](#0-4) 

**The gap:** `instantWithdrawal` never reads or decrements `assetsCommitted`. When `queuedWithdrawalsBuffer` is zero (the default state), an instant withdrawal user can drain the entire vault balance, including assets that are already committed to queued withdrawal users. The `assetsCommitted` counter remains non-zero, but the vault is empty.

When the operator subsequently calls `unlockQueue` to service the queued requests, it reads `unstakingVault.balanceOf(asset)` as `totalAvailableAssets`: [6](#0-5) 

With the vault empty, `params.totalAvailableAssets == 0` causes an immediate revert at line 297, and `assetsCommitted` is never decremented. The queued withdrawal users' rsETH has already been held by the contract (transferred in `initiateWithdrawal` at line 166), but their assets cannot be unlocked. [7](#0-6) 

---

### Impact Explanation

Queued withdrawal users who called `initiateWithdrawal` have their rsETH locked in `LRTWithdrawalManager` and their expected asset amount committed, but the vault assets backing those commitments have been consumed by instant withdrawal users. The `unlockQueue` call reverts, preventing the withdrawal queue from advancing. This constitutes **temporary (potentially extended) freezing of funds** for queued withdrawal users. Recovery requires the protocol to replenish the vault by completing additional EigenLayer unstaking cycles, which is not guaranteed to happen promptly.

**Severity: Medium** — Temporary freezing of funds.

---

### Likelihood Explanation

The conditions required are:
1. `isInstantWithdrawalEnabled[asset]` is `true` — a normal operational state when the feature is live.
2. `queuedWithdrawalsBuffer[asset]` is `0` — the **default** value; operators must explicitly set it to a non-zero value to provide any protection.
3. Assets are present in `LRTUnstakingVault` while `assetsCommitted[asset] > 0` — a routine state during the withdrawal lifecycle.

All three conditions are simultaneously true in the default operational configuration whenever instant withdrawals are enabled without explicitly configuring the buffer. Any unprivileged rsETH holder can trigger this.

---

### Recommendation

`instantWithdrawal` should enforce that the assets it consumes do not overlap with `assetsCommitted`. The simplest fix is to have `getAssetsAvailableForInstantWithdrawal` (or the check inside `instantWithdrawal`) subtract `assetsCommitted[asset]` from the vault balance, rather than relying on the manually-set `queuedWithdrawalsBuffer`. Alternatively, `instantWithdrawal` should decrement `assetsCommitted` if it is consuming committed assets, but this would require cross-contract coordination. The cleanest solution is to replace the `queuedWithdrawalsBuffer` guard with a direct read of `assetsCommitted` from `LRTWithdrawalManager`.

---

### Proof of Concept

```
1. Protocol has 100 stETH in LRTUnstakingVault.
   queuedWithdrawalsBuffer[stETH] = 0 (default).
   isInstantWithdrawalEnabled[stETH] = true.

2. UserA calls initiateWithdrawal(stETH, rsETH_A):
   → assetsCommitted[stETH] += 100 stETH
   → rsETH_A transferred to LRTWithdrawalManager

3. getAvailableAssetAmount(stETH) = totalAssets - 100 = 0
   → No new queued withdrawals can be initiated (correct).

4. UserB (rsETH holder) calls instantWithdrawal(stETH, rsETH_B):
   → getAssetsAvailableForInstantWithdrawal = 100 - 0 = 100 stETH (buffer is 0)
   → 100 stETH redeemed from LRTUnstakingVault and sent to UserB
   → assetsCommitted[stETH] remains 100 (NOT updated)

5. LRTUnstakingVault now holds 0 stETH.

6. Operator calls unlockQueue(stETH, ...):
   → params.totalAvailableAssets = unstakingVault.balanceOf(stETH) = 0
   → revert AmountMustBeGreaterThanZero()

7. UserA's rsETH is locked in LRTWithdrawalManager.
   UserA cannot complete their withdrawal.
   UserA's funds are frozen until the vault is replenished externally.
``` [8](#0-7) [9](#0-8) [4](#0-3)

    function instantWithdrawal(
        address asset,
        uint256 rsETHUnstaked,
        string calldata referralId
    )
        external
        nonReentrant
        whenNotPaused
        onlySupportedAsset(asset)
        onlySupportedStrategy(asset)
        onlyInstantWithdrawalAllowed(asset)
    {
        if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
            revert InvalidAmountToWithdraw();
        }
        if (IERC20(lrtConfig.rsETH()).balanceOf(msg.sender) < rsETHUnstaked) revert NotEnoughRsETH();
        uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
        IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
        ILRTUnstakingVault unstakingVault = ILRTUnstakingVault(lrtConfig.getContract(LRTConstants.LRT_UNSTAKING_VAULT));
        if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
            revert CantInstantWithdrawMoreThanAvailable();
        }

        unstakingVault.redeem(asset, assetAmountUnlocked);

        uint256 fee = (assetAmountUnlocked * instantWithdrawalFee) / 10_000;
        uint256 userAmount = assetAmountUnlocked - fee;

        address feeRecipient = instantWithdrawalFeeRecipient;
        if (feeRecipient == address(0)) {
            // Backwards-compatible default: send fees to the protocol treasury
            feeRecipient = lrtConfig.getContract(LRTConstants.PROTOCOL_TREASURY);
        }
        if (fee > 0) {
            _transferAsset(asset, feeRecipient, fee);
            emit InstantWithdrawalFeeCollected(msg.sender, asset, fee);
        }

        _transferAsset(asset, msg.sender, userAmount);
        emit ReferralIdEmitted(referralId);
        emit AssetWithdrawalFinalized(msg.sender, asset, rsETHUnstaked, userAmount);
    }

    function getAvailableAssetAmount(address asset) public view override returns (uint256 availableAssetAmount) {
        ILRTDepositPool lrtDepositPool = ILRTDepositPool(lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL));
        uint256 totalAssets = lrtDepositPool.getTotalAssetDeposits(asset);
        availableAssetAmount = totalAssets > assetsCommitted[asset] ? totalAssets - assetsCommitted[asset] : 0;
    }

        external
        view
        onlySupportedAsset(asset)
        returns (uint256 availableAmount)
    {
        uint256 vaultBalance = balanceOf(asset);
        uint256 reservedBuffer = queuedWithdrawalsBuffer[asset];
        availableAmount = reservedBuffer >= vaultBalance ? 0 : vaultBalance - reservedBuffer;
    }
