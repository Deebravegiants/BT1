### Title
`TokenSwap.depositToKingProtocol()` and `depositMultipleToKingProtocol()` Provide Zero Slippage Protection, Enabling Sandwich Attacks on Protocol Deposits - (File: contracts/king-protocol/TokenSwap.sol)

### Summary
`TokenSwap` calls `kingProtocol.previewDeposit()` to compute expected shares before depositing, but never enforces that result as a minimum in the actual `kingProtocol.deposit()` call. An attacker can sandwich the admin/manager's deposit transaction to extract value from the protocol, causing it to receive fewer King Protocol shares than expected.

### Finding Description

In `contracts/king-protocol/TokenSwap.sol`, both `depositToKingProtocol()` and `depositMultipleToKingProtocol()` follow the same vulnerable pattern:

```solidity
// Preview the deposit to get expected shares
(uint256 expectedShares,) = kingProtocol.previewDeposit(tokens, amounts);
shareReceived = expectedShares;   // used only for return value / event

// Approve King Protocol to spend the tokens
assetToken.forceApprove(address(kingProtocol), amount);

// Deposit to King Protocol — NO minimum shares enforced
kingProtocol.deposit(tokens, amounts, address(this));
``` [1](#0-0) 

The `expectedShares` value obtained from `previewDeposit` is assigned to `shareReceived` and returned to the caller, but it is **never used as a guard** on the actual `deposit()` call. The `IKingProtocol.deposit()` interface accepts no `minSharesExpected` parameter:

```solidity
function deposit(address[] memory tokens, uint256[] memory amounts, address recipient) external;
``` [2](#0-1) 

Because `deposit()` returns nothing, the actual shares credited to `TokenSwap` are never compared against `expectedShares`. The same flaw exists in `depositMultipleToKingProtocol()`: [3](#0-2) 

Additionally, `_emitMultipleDepositEvents` always emits `shareReceived = 0` per token, so even the event log is misleading: [4](#0-3) 

### Impact Explanation

**High — Theft of unclaimed yield.**

The `TokenSwap` contract is the protocol's mechanism for converting accepted assets (e.g., stETH, ETH) into King Protocol shares, which ultimately back KING token distributions to users. If an attacker sandwiches a deposit call, the protocol receives fewer shares than the oracle-previewed amount. The difference in shares is captured by the attacker, constituting direct theft of yield that would otherwise accrue to the protocol and its users.

### Likelihood Explanation

**Low.**

The attack requires:
1. The attacker to monitor the mempool for `depositToKingProtocol()` / `depositMultipleToKingProtocol()` calls.
2. The King Protocol to use dynamic pricing (strongly implied by the existence of `previewDeposit` returning variable `expectedShares` and a `depositFee`).
3. The attacker to have sufficient capital to move the King Protocol's exchange rate meaningfully.

These conditions are realistic on Ethereum mainnet where MEV bots routinely monitor privileged protocol calls.

### Recommendation

1. After calling `kingProtocol.deposit()`, measure the actual shares received (e.g., by comparing the KING token balance before and after) and revert if the actual amount falls below a configurable `minSharesExpected` threshold.
2. Alternatively, expose a `minSharesExpected` parameter on both deposit functions so the caller can enforce slippage tolerance off-chain.

```solidity
uint256 sharesBefore = kingToken.balanceOf(address(this));
kingProtocol.deposit(tokens, amounts, address(this));
uint256 sharesAfter = kingToken.balanceOf(address(this));
require(sharesAfter - sharesBefore >= minSharesExpected, "Slippage exceeded");
```

### Proof of Concept

1. Admin/manager submits `depositToKingProtocol(asset, largeAmount)` to the mempool.
2. Attacker's MEV bot detects the pending transaction.
3. **Front-run**: Attacker deposits a large amount into King Protocol, worsening the exchange rate (fewer shares per token).
4. Admin/manager's `depositToKingProtocol()` executes:
   - `previewDeposit()` returns `expectedShares` (already degraded by attacker's front-run).
   - `deposit()` executes at the worsened rate; `TokenSwap` receives fewer shares than the pre-attack fair value.
   - `shareReceived` is set to the stale preview value, masking the actual loss.
5. **Back-run**: Attacker withdraws from King Protocol, restoring the rate and pocketing the profit.
6. Protocol has permanently lost the difference in King Protocol shares, reducing future KING token yield for users. [2](#0-1) [5](#0-4)

    function depositToKingProtocol(
        address asset,
        uint256 amount
    )
        external
        nonReentrant
        whenNotPaused
        onlyAdminOrManager
        returns (uint256 shareReceived)
    {
        if (amount == 0) {
            revert ZeroAmount();
        }

        if (!supportedTokens[asset]) {
            revert UnsupportedAsset();
        }

        IERC20 assetToken = IERC20(asset);
        uint256 contractBalance = assetToken.balanceOf(address(this));

        if (contractBalance < amount) {
            revert InsufficientBalance();
        }

        // Create arrays for the deposit call
        address[] memory tokens = new address[](1);
        uint256[] memory amounts = new uint256[](1);
        tokens[0] = asset;
        amounts[0] = amount;

        // Preview the deposit to get expected shares
        (uint256 expectedShares,) = kingProtocol.previewDeposit(tokens, amounts);
        shareReceived = expectedShares;

        // Approve King Protocol to spend the tokens
        assetToken.forceApprove(address(kingProtocol), amount);

        // Deposit to King Protocol
        kingProtocol.deposit(tokens, amounts, address(this));

        // Reset approval after successful deposit
        assetToken.forceApprove(address(kingProtocol), 0);

        emit TokensDeposited(asset, amount, shareReceived, msg.sender);
    }
```
