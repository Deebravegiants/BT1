### Title
Staking Rewards Permanently Frozen When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: `contracts/KERNEL/KernelDepositPool.sol`)

---

### Summary

`KernelDepositPool.rewardPerToken()` contains an early-return guard `if (totalKernelStaked == 0) return rewardPerTokenStored`. When the last staker calls `initiateWithdrawal()`, `totalKernelStaked` is immediately decremented to zero. From that moment until either a new staker arrives or `finishAt` is reached, all reward tokens that should have been distributed continue to sit in the contract but are never allocated to anyone — they are permanently frozen.

---

### Finding Description

`rewardPerToken()` accumulates rewards proportionally to elapsed time only when `totalKernelStaked > 0`:

```solidity
// KernelDepositPool.sol L408-L414
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;   // ← early return, no accumulation
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
```

`initiateWithdrawal()` immediately subtracts the withdrawn amount from `totalKernelStaked` before the mandatory `withdrawalDelay` elapses:

```solidity
// KernelDepositPool.sol L320-L326
function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
    ...
    balanceOf[msg.sender] -= _amount;
    totalKernelStaked -= _amount;   // ← immediate decrement
    ...
    uint256 unlockTime = block.timestamp + withdrawalDelay;
```

The `updateReward` modifier correctly snapshots the user's earned rewards at the moment of `initiateWithdrawal`. However, once `totalKernelStaked` reaches zero, `rewardPerToken()` returns the frozen `rewardPerTokenStored` for every subsequent call. The `updatedAt` cursor is still advanced by the modifier (`updatedAt = lastTimeRewardApplicable()`), so when a new staker eventually arrives, the time gap during which no one was staked is silently skipped — the rewards for that gap are never distributed and remain locked in the contract forever.

The contract's own NatSpec acknowledges this:

> *"If `totalKernelStaked` ever hits zero during a reward distribution window, any remaining rewards for that period will stay locked in the contract. In this deployment, we're avoiding this issue by ensuring there are always some tokens staked…"*

However, this mitigation is purely operational — there is no on-chain enforcement preventing `totalKernelStaked` from reaching zero. Any unprivileged user who is the last (or sole) staker can trigger this condition unilaterally.

---

### Impact Explanation

**Permanent freezing of unclaimed yield (Medium).**

Reward tokens transferred into the contract via `notifyRewardAmount` and allocated to a reward period are irrecoverably locked whenever `totalKernelStaked` hits zero mid-period. There is no `rescue`, `sweep`, or admin recovery function in `KernelDepositPool`. The frozen tokens cannot be redistributed in a future period because `notifyRewardAmount` uses a transfer-in delta pattern — it only accounts for newly deposited tokens, not the already-resident frozen balance.

---

### Likelihood Explanation

Realistic. The scenario requires only a single unprivileged actor: the last remaining staker. This can occur naturally (all other stakers exit first, leaving one user who then calls `initiateWithdrawal`), or it can be deliberately engineered. The `withdrawalDelay` (up to `MAX_WITHDRAWAL_DELAY = 30 days`) creates a guaranteed window during which `totalKernelStaked == 0` while the reward period is still active, maximising the frozen amount.

---

### Recommendation

Track the cumulative "dead time" during which `totalKernelStaked == 0` and either:

1. **Extend `finishAt`** by the dead-time duration each time `totalKernelStaked` transitions from non-zero to zero, so the full reward budget is eventually distributed; or
2. **Freeze `updatedAt`** when `totalKernelStaked == 0` (do not advance it in the modifier), so the skipped interval is replayed once staking resumes — matching the pattern used by Synthetix's canonical `StakingRewards`.

Option 2 is the minimal change:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    // Only advance the cursor when there are stakers to absorb the rewards
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    ...
}
```

### Proof of Concept

1. Admin calls `notifyRewardAmount(1_000e18)` with `duration = 10 days`. `rewardRate = 1_000e18 / 864_000 ≈ 1157 wei/s`. `finishAt = now + 10 days`.
2. Alice is the sole staker with `balanceOf[Alice] = 1_000e18`, `totalKernelStaked = 1_000e18`.
3. After **5 days**, Alice calls `initiateWithdrawal(1_000e18)`:
   - `updateReward(Alice)` snapshots her 5-day earnings correctly.
   - `totalKernelStaked` becomes **0**.
   - `unlockTime = now + withdrawalDelay`.
4. For the remaining **5 days** (until `finishAt`), every call to `rewardPerToken()` hits the early-return branch and returns the same frozen `rewardPerTokenStored`. `updatedAt` is advanced by the modifier on any interaction, consuming the time window silently.
5. After `withdrawalDelay`, Alice calls `claimWithdrawal` and `getReward` — she correctly receives only her 5-day share.
6. The **5-day reward tranche** (`≈ 500e18` tokens) remains permanently locked in `KernelDepositPool` with no mechanism to recover it. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

    function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
        if (_amount == 0) revert AmountZero();

        // Prevent starting a reward period when no tokens are staked to avoid unallocated rewards
        if (totalKernelStaked == 0) revert NoStakedTokens();

        // Transfer reward tokens into the contract
        uint256 balanceBefore = rewardsToken.balanceOf(address(this));
        rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
        uint256 balanceAfter = rewardsToken.balanceOf(address(this));
        // Calculate the actual amount of tokens received in case of a transfer fee (tax)
        uint256 receivedAmount = balanceAfter - balanceBefore;

        if (block.timestamp >= finishAt) {
            rewardRate = receivedAmount / duration;
        } else {
            uint256 remaining = (finishAt - block.timestamp) * rewardRate;
            rewardRate = (receivedAmount + remaining) / duration;
        }

        if (rewardRate == 0) revert RewardRateZero();

        finishAt = block.timestamp + duration;
        updatedAt = block.timestamp;

        emit NotifyRewardAmount(receivedAmount, finishAt);
