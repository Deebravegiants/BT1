### Title
Manager Can Immediately Raise `instantWithdrawalFee` to Frontrun `instantWithdrawal()`, Stealing User Funds — (`contracts/LRTWithdrawalManager.sol`)

### Summary

`setInstantWithdrawalFee()` takes effect immediately with no timelock, and `instantWithdrawal()` has no `minAmountOut` slippage guard. A `MANAGER` role holder can frontrun a user's instant-withdrawal transaction by raising the fee up to 10 % (1 000 bps), causing the user to receive significantly less than expected after irreversibly burning their rsETH. The protocol's own NatSpec comment explicitly acknowledges this attack path.

---

### Finding Description

`setInstantWithdrawalFee()` is callable by any address holding `MANAGER` role and takes effect in the same block with no delay: [1](#0-0) 

The fee is capped at 1 000 bps (10 %) but has no lower-bound enforcement and no timelock:

```solidity
function setInstantWithdrawalFee(uint256 feeBasisPoints) external onlyLRTManager {
    if (feeBasisPoints > 1000) revert FeeTooHigh(); // Max 10%
    instantWithdrawalFee = feeBasisPoints;
    emit InstantWithdrawalFeeUpdated(feeBasisPoints);
}
```

`instantWithdrawal()` reads `instantWithdrawalFee` at execution time and has **no `minAmountOut` parameter**: [2](#0-1) 

The fee deduction is applied after the rsETH is already burned: [3](#0-2) 

The protocol's own NatSpec comment on `instantWithdrawal()` explicitly acknowledges the attack: [4](#0-3) 

> *"Uses the fee set at execution time. Managers can raise it right before this call, making withdrawals cost more than expected."*

The fee recipient is also manager-controlled via `setInstantWithdrawalFeeRecipient()`: [5](#0-4) 

---

### Impact Explanation

A user who calls `instantWithdrawal()` expecting a fee of, say, 0 bps receives up to 10 % less of their underlying asset than anticipated. Their rsETH is burned before the fee is deducted — there is no revert path once the burn executes. The stolen fee flows to the manager-designated `instantWithdrawalFeeRecipient`. This constitutes **theft of user funds** (High — direct theft of assets in motion).

---

### Likelihood Explanation

The `MANAGER` role is a hot-wallet operational key (not a timelock or multisig by design). The attack requires only a single transaction from the manager, executable in the same block as the victim's withdrawal. The code comment confirms the developers are aware of this exact scenario. Any user calling `instantWithdrawal()` is exposed whenever the fee is not already at its maximum.

---

### Recommendation

1. **Add a `minAmountOut` parameter** to `instantWithdrawal()` so users can specify the minimum asset amount they accept, reverting if the fee-adjusted payout falls below it.
2. **Enforce a timelock** (e.g., 24 h) on `setInstantWithdrawalFee()` changes, consistent with the withdrawal delay already present in the normal withdrawal path (`withdrawalDelayBlocks`).
3. Alternatively, **snapshot the fee at request time** (analogous to how `expectedAssetAmount` is snapshotted in `initiateWithdrawal()`) so users are protected against fee changes between submission and execution.

---

### Proof of Concept

1. Manager calls `setInstantWithdrawalFee(1000)` (10 %) in the same block as (or just before) the victim's transaction.
2. Victim calls `instantWithdrawal(ETH_TOKEN, 10 ether, "")`.
3. `instantWithdrawal()` burns 10 rsETH from the victim.
4. `assetAmountUnlocked` is computed from the oracle price.
5. `fee = assetAmountUnlocked * 1000 / 10_000` — 10 % is deducted.
6. Victim receives only 90 % of the expected ETH; the 10 % fee goes to `instantWithdrawalFeeRecipient` (manager-controlled).
7. The victim has no recourse: rsETH is already burned and there was no `minAmountOut` check. [6](#0-5)

```text
    function instantWithdrawal(
        address asset,
        uint256 rsETHUnstaked,
        string calldata referralId
    )
        external
        nonReentrant
        whenNotPaused
        onlySupportedAsset(asset)
        onlySupportedStrategy(asset)
        onlyInstantWithdrawalAllowed(asset)
    {
        if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
            revert InvalidAmountToWithdraw();
        }
        if (IERC20(lrtConfig.rsETH()).balanceOf(msg.sender) < rsETHUnstaked) revert NotEnoughRsETH();
        uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
        IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
        ILRTUnstakingVault unstakingVault = ILRTUnstakingVault(lrtConfig.getContract(LRTConstants.LRT_UNSTAKING_VAULT));
        if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
            revert CantInstantWithdrawMoreThanAvailable();
        }

        unstakingVault.redeem(asset, assetAmountUnlocked);

        uint256 fee = (assetAmountUnlocked * instantWithdrawalFee) / 10_000;
        uint256 userAmount = assetAmountUnlocked - fee;

        address feeRecipient = instantWithdrawalFeeRecipient;
        if (feeRecipient == address(0)) {
            // Backwards-compatible default: send fees to the protocol treasury
            feeRecipient = lrtConfig.getContract(LRTConstants.PROTOCOL_TREASURY);
        }
        if (fee > 0) {
            _transferAsset(asset, feeRecipient, fee);
            emit InstantWithdrawalFeeCollected(msg.sender, asset, fee);
        }

        _transferAsset(asset, msg.sender, userAmount);
        emit ReferralIdEmitted(referralId);
        emit AssetWithdrawalFinalized(msg.sender, asset, rsETHUnstaked, userAmount);
    }
```
