### Title
Arithmetic Underflow in `bridgeKernelToBSC` Causes Temporary DoS When `counter == 0` — (`contracts/KERNEL/KernelVaultETH.sol`)

---

### Summary

`bridgeKernelToBSC` unconditionally computes `lastBridgedDepositId = counter - 1` before any guard checks that `counter > 0`. Because Solidity 0.8.27 reverts on underflow, any call to `bridgeKernelToBSC` while `counter == 0` will always revert. An unprivileged actor can force this state by transferring KERNEL directly to the vault via a standard ERC20 `transfer`, bypassing `depositKernel` and leaving `counter` at zero while the vault holds a nonzero balance.

---

### Finding Description

In `KernelVaultETH.bridgeKernelToBSC`:

```solidity
// contracts/KERNEL/KernelVaultETH.sol line 262
lastBridgedDepositId = counter - 1;
``` [1](#0-0) 

`counter` starts at `0` and is only incremented inside `_depositKernel`:

```solidity
// line 394
++counter;
``` [2](#0-1) 

The balance check at line 238 only requires `kernel.balanceOf(address(this)) >= amount`, which passes when tokens arrive via a direct ERC20 `transfer` to the vault address — a path that never touches `_depositKernel` and never increments `counter`. [3](#0-2) 

There is no sweep, rescue, or emergency-withdrawal function anywhere in the contract, so the only exit path for KERNEL tokens is `bridgeKernelToBSC`. [4](#0-3) 

---

### Impact Explanation

**Medium — Temporary freezing of funds.**

- While `counter == 0`, every operator call to `bridgeKernelToBSC` reverts with an arithmetic underflow panic, regardless of the vault's KERNEL balance.
- The freeze is **temporary**, not permanent: as soon as any legitimate `depositKernel` call is processed, `counter` becomes `1`, `counter - 1 = 0` is valid, and bridging resumes.
- The directly-transferred tokens themselves are not permanently lost: once `counter > 0`, the operator can include them in the `amount` parameter of a subsequent bridge call. However, they carry no `userDeposits` record, so attribution on BSC is undefined — a secondary accounting issue.
- The "permanent" framing in the question is overstated. The invariant break is real but self-resolving once any normal deposit occurs.

---

### Likelihood Explanation

Low-to-medium. Any holder of KERNEL tokens can execute this with a single `transfer` call. The window of impact is bounded by the time until the first legitimate deposit, but on a freshly deployed vault (or during a quiet period) this could block bridging for an extended period. The attacker loses the transferred tokens (self-inflicted), which limits motivation, but a griefing actor or a mistaken direct transfer achieves the same effect.

---

### Recommendation

Add a guard before the underflow-prone line:

```solidity
// In bridgeKernelToBSC, before line 262:
if (counter == 0) revert NoDepositsTobridge(); // or use a custom error

lastBridgedDepositId = counter - 1;
```

Alternatively, restructure the assignment to only execute when `counter > 0`, or track `lastBridgedDepositId` separately from `counter` so it is not derived by subtraction.
