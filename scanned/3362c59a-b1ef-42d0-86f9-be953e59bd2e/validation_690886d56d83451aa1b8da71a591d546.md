### Title
Exact-Equality Fee Check in `bridgeRsETHToL2UsingCCIP` Causes Repeated Reverts During CCIP Fee Fluctuation — (`contracts/L1VaultV2.sol`)

---

### Summary

`bridgeRsETHToL2UsingCCIP` re-quotes the CCIP fee live inside the transaction and then enforces `msg.value != fee` with exact equality. Because CCIP fees are dynamic (gas-price-sensitive), any drift between the off-chain quote and the on-chain execution block causes every bridge attempt to revert with `IncorrectCCIPFee`, temporarily preventing rsETH from being bridged to L2.

---

### Finding Description

Inside `bridgeRsETHToL2UsingCCIP`, the fee is re-computed on-chain at execution time:

```solidity
uint256 fee = getCCIPFee(amount);   // live call to ccipRouter.getFee(...)

if (msg.value != fee) {
    revert IncorrectCCIPFee();
}
``` [1](#0-0) 

`getCCIPFee` delegates directly to the router:

```solidity
function getCCIPFee(uint256 amount) public view returns (uint256) {
    Client.EVM2AnyMessage memory message = getCCIPMessage(amount);
    return ccipRouter.getFee(destinationChainSelector, message);
}
``` [2](#0-1) 

The MANAGER_ROLE must quote the fee off-chain (also via `getCCIPFee`), then submit the transaction with exactly that value. If the router's `getFee` returns even 1 wei more or less at the execution block, the check fails and the transaction reverts.

Critically, the `IRouterClient` interface itself documents that `ccipSend` **accepts overpayment**:

```
/// @dev Note if msg.value is larger than the required fee (from getFee) we accept
/// the overpayment with no refund.
``` [3](#0-2) 

The contract's exact-equality guard is therefore stricter than the underlying protocol requires, and it directly contradicts the router's own design intent.

---

### Impact Explanation

While rsETH accumulates in `L1VaultV2` (deposited from L2 bridgers), the MANAGER_ROLE cannot move it to L2 whenever CCIP fees are volatile. Every attempt reverts. The funds are not lost, but they are temporarily stranded and cannot be bridged for the duration of fee instability. This matches **Medium — Temporary freezing of funds**.

---

### Likelihood Explanation

CCIP fees are computed from destination-chain gas prices and token prices, both of which fluctuate block-to-block. The window between an off-chain `eth_call` to `getCCIPFee` and the transaction landing on-chain is typically 1–12+ seconds (mempool delay, gas auction). Fee changes of even 1 wei during that window are routine under normal network conditions, making this a high-frequency operational failure, not a theoretical edge case.

---

### Recommendation

Replace the exact-equality check with a minimum-fee check and refund any excess ETH to the caller:

```solidity
uint256 fee = getCCIPFee(amount);

if (msg.value < fee) {
    revert IncorrectCCIPFee();
}

// ... perform ccipSend with `fee`, not `msg.value` ...
bytes32 messageId = ccipRouter.ccipSend{ value: fee }(destinationChainSelector, message);

// Refund overpayment
uint256 excess = msg.value - fee;
if (excess > 0) {
    (bool ok, ) = msg.sender.call{ value: excess }("");
    require(ok, "ETH refund failed");
}
```

This aligns with the router's own documented behavior (overpayment accepted) and eliminates the race condition between quote and execution.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.27;

import "forge-std/Test.sol";
import { L1VaultV2 } from "contracts/L1VaultV2.sol";

contract MockCCIPRouter {
    uint256 public feeToReturn;

    function setFee(uint256 f) external { feeToReturn = f; }

    function getFee(uint64, bytes memory) external view returns (uint256) {
        return feeToReturn;
    }

    function ccipSend(uint64, bytes calldata) external payable returns (bytes32) {
        return bytes32(0);
    }
}

contract CCIPFeeRaceTest is Test {
    L1VaultV2 vault;
    MockCCIPRouter router;
    address manager;

    function setUp() public {
        manager = makeAddr("manager");
        router = new MockCCIPRouter();
        // ... deploy and initialize vault with router, grant MANAGER_ROLE to manager ...
    }

    function test_exactFeeCheckReverts_onFeeChange() public {
        uint256 amount = 1 ether;

        // Step 1: off-chain quote at block N → fee = 0.01 ether
        router.setFee(0.01 ether);
        uint256 quotedFee = vault.getCCIPFee(amount);
        assertEq(quotedFee, 0.01 ether);

        // Step 2: fee changes by 1 wei before execution (block N+k)
        router.setFee(0.01 ether + 1);

        // Step 3: MANAGER submits with the previously quoted fee → reverts
        vm.prank(manager);
        vm.expectRevert(L1VaultV2.IncorrectCCIPFee.selector);
        vault.bridgeRsETHToL2UsingCCIP{ value: quotedFee }(amount);
    }
}
```

The test demonstrates that a 1-wei fee increase between quote and execution is sufficient to permanently block the bridge call, confirming the root cause at [4](#0-3) .
