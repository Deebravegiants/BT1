### Title
rsETH Holders Can Escape Slashing Impact via `instantWithdrawal` Using Stale Oracle Price — (`contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTWithdrawalManager.instantWithdrawal()` calculates the asset payout using the **cached** `rsETHPrice` stored in `LRTOracle`. When an EigenLayer slashing event reduces the protocol's TVL but the oracle has not yet been updated, a user can call `instantWithdrawal` at the stale pre-slashing price and receive more assets than their fair share. The slashing impact is then concentrated on the remaining rsETH holders.

---

### Finding Description

`instantWithdrawal` computes the payout via `getExpectedAssetAmount`:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
``` [1](#0-0) 

`lrtOracle.rsETHPrice()` is a **cached state variable** in `LRTOracle` that is only updated when `updateRSETHPrice()` is explicitly called: [2](#0-1) [3](#0-2) 

When an EigenLayer operator slashing event occurs, `getWithdrawableShares()` in EigenLayer immediately reflects the reduced shares. This means `_getTotalEthInProtocol()` (called inside `updateRSETHPrice`) would return a lower TVL — but only **after** `updateRSETHPrice()` is called. Until then, `rsETHPrice` remains at the pre-slashing value. [4](#0-3) 

The attack window is the gap between the on-chain slashing event and the next `updateRSETHPrice()` call. During this window, `instantWithdrawal` uses the inflated stale price: [5](#0-4) 

Critically, `instantWithdrawal` has **no min-price protection**. By contrast, the queued `initiateWithdrawal` path uses `_calculatePayoutAmount`, which takes `min(expectedAssetAmount, currentReturn)` at unlock time, ensuring slashing is always reflected: [6](#0-5) 

`instantWithdrawal` has no equivalent safeguard. The user burns rsETH and immediately receives assets at the stale price with no recalculation at a later time.

The `pricePercentageLimit` downside-pause mechanism only activates **inside** `updateRSETHPrice()`: [7](#0-6) 

Since the oracle has not been updated yet, the protocol is not paused, and `instantWithdrawal` (which is `whenNotPaused`) proceeds freely.

---

### Impact Explanation

A user who observes an EigenLayer slashing event on-chain before `updateRSETHPrice()` is called can call `instantWithdrawal` and receive assets valued at the pre-slashing rsETH price. The rsETH they burn is priced too high relative to the actual post-slashing TVL. The difference is effectively stolen from remaining rsETH holders, who will see a larger-than-proportional price drop when the oracle is eventually updated.

**Impact: High — theft of user funds from remaining rsETH holders.**

---

### Likelihood Explanation

- EigenLayer operator slashing immediately reduces `getWithdrawableShares()` on-chain, creating an observable discrepancy between actual TVL and the cached `rsETHPrice`.
- `updateRSETHPrice()` is a public function but is not called atomically with slashing; there is always a non-zero window.
- A sophisticated user monitoring EigenLayer's `AllocationManager` or `DelegationManager` events can detect slashing and front-run the oracle update in the same or next block.
- `isInstantWithdrawalEnabled` must be `true` for the target asset, which is a manager-controlled toggle that may be enabled for ETH or LSTs.

**Likelihood: Medium.**

---

### Recommendation

1. **Force a fresh price computation inside `instantWithdrawal`**: Call `updateRSETHPrice()` (or an internal equivalent) before computing `assetAmountUnlocked`, so the payout always reflects the current TVL.
2. **Add a staleness bound**: Revert if `block.timestamp` exceeds the last oracle update timestamp by more than a configurable threshold (e.g., 1 hour).
3. **Apply the same `min(expectedAmount, currentReturn)` guard used in `_calculatePayoutAmount`**: Compute the expected amount at initiation and again at execution, paying the lesser of the two.

---

### Proof of Concept

1. EigenLayer operator is slashed; `getWithdrawableShares()` for the NDC immediately decreases by, say, 5%.
2. `rsETHPrice` in `LRTOracle` is still the pre-slashing value (e.g., `1.05e18`).
3. Attacker calls `instantWithdrawal(ETH, largeRsETHAmount)`.
4. `assetAmountUnlocked = largeRsETHAmount * 1.05e18 / 1e18` — computed at the stale price.
5. Attacker receives the inflated ETH amount from `LRTUnstakingVault`.
6. `updateRSETHPrice()` is called; new price reflects the 5% TVL drop (e.g., `0.9975e18`).
7. Remaining rsETH holders now hold tokens worth less than they should, having absorbed the attacker's share of the slashing loss. [8](#0-7) [9](#0-8)

    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
