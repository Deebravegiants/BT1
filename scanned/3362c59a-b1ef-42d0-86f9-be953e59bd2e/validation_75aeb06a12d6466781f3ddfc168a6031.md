### Title
Mutable `feeInBPS` Applied at Claim Time Silently Reduces Claimants' KERNEL Yield - (File: `contracts/KERNEL/KernelMerkleDistributor.sol`)

---

### Summary

`KernelMerkleDistributor` applies the **current** `feeInBPS` at the moment a user calls `claim` / `claimAndStake`, rather than at the moment the merkle root is published. Because the owner can update `feeInBPS` at any time via `setFeeInBPS`, a fee increase between root publication and user claim silently redirects a portion of users' entitled KERNEL to the protocol treasury — an exact structural analog to the reported "fee fetched at fulfillment time" bug.

---

### Finding Description

The merkle tree encodes each user's **gross cumulative KERNEL amount**. When a user claims, `_processClaim` computes the claimable delta and immediately applies the live `feeInBPS`:

```solidity
// contracts/KERNEL/KernelMerkleDistributor.sol  line 326-342
uint256 claimableAmount = cumulativeAmount - userClaims[account].cumulativeAmount;
...
uint256 fee = (claimableAmount * feeInBPS) / FEE_DENOMINATOR;
uint256 amountToSend = claimableAmount - fee;
if (fee > 0) {
    kernel.safeTransfer(protocolTreasury, fee);
}
``` [1](#0-0) 

The owner can change `feeInBPS` at any time with no time-lock:

```solidity
// contracts/KERNEL/KernelMerkleDistributor.sol  line 388-395
function setFeeInBPS(uint256 _feeInBPS) external onlyOwner {
    if (_feeInBPS > MAX_FEE_IN_BPS) revert InvalidFeeInBPS();
    feeInBPS = _feeInBPS;
    emit FeeInBPSUpdated(feeInBPS);
}
``` [2](#0-1) 

Because the merkle root encodes only gross amounts and the fee is never snapshotted at root-publication time, every pending claimant is silently exposed to whatever `feeInBPS` happens to be in effect when they finally call `claim`.

---

### Impact Explanation

**High — Theft of unclaimed yield.**

Users observe a published merkle root that entitles them to a specific gross KERNEL amount. If `feeInBPS` is raised between root publication and their claim transaction, they receive a smaller net amount than the protocol implicitly promised. The difference flows to `protocolTreasury`, constituting a direct, unannounced reduction of earned yield for every claimant who has not yet claimed.

---

### Likelihood Explanation

The owner has an unrestricted, instant ability to call `setFeeInBPS` at any time. There is no time-lock, no governance delay, and no on-chain notice period. Any routine fee adjustment — even a well-intentioned one — that occurs while a merkle root is live and claimants are pending will silently apply the new rate to all outstanding claims. Given that merkle roots are published periodically and users may not claim immediately, the window of exposure is non-trivial.

---

### Recommendation

Snapshot `feeInBPS` at the time each merkle root is published and store it alongside the root. Apply the snapshotted fee when processing claims against that root, so users are always charged the fee that was in effect when their entitlement was committed on-chain.

```solidity
struct MerkleRootEntry {
    bytes32 root;
    uint256 feeInBPS;   // fee locked at publication time
}
mapping(uint256 => MerkleRootEntry) public merkleRoots;
```

---

### Proof of Concept

1. Owner publishes merkle root at `feeInBPS = 0` (zero fee). Users observe the root and compute they will receive 1 000 KERNEL.
2. Before users claim, owner calls `setFeeInBPS(500)` (5 % fee) — a routine protocol adjustment.
3. User calls `claim(index, account, 1000e18, proof)`.
4. `_processClaim` computes `fee = 1000e18 * 500 / 10_000 = 50e18` and transfers it to `protocolTreasury`.
5. User receives only 950 KERNEL instead of the 1 000 KERNEL the published root entitled them to — 50 KERNEL silently redirected with no user consent or prior notice. [3](#0-2) [2](#0-1)
