The `initialize` function of `KernelDepositPool` does not set `withdrawalDelay`, so it defaults to `0`. Let me confirm the full picture before writing the finding.

### Title
`withdrawalDelay` Defaults to Zero in Initializer, Allowing Immediate Withdrawal of Staked KERNEL Tokens — (File: `contracts/KERNEL/KernelDepositPool.sol`)

---

### Summary
`KernelDepositPool.initialize()` never sets `withdrawalDelay`, so it defaults to `0`. When an admin subsequently enables withdrawals by calling `setMaxNumberOfWithdrawalsPerUser` without also calling `setWithdrawalDelay`, any staker can call `initiateWithdrawal` and `claimWithdrawal` in the same block, bypassing the intended time-lock entirely.

---

### Finding Description
The `initialize` function sets only `kernelToken`, `rewardsToken`, and the admin role. Neither `withdrawalDelay` nor `maxNumberOfWithdrawalsPerUser` is assigned a non-zero value. [1](#0-0) 

`withdrawalDelay` therefore starts at `0`. In `initiateWithdrawal`, the unlock timestamp is computed as:

```solidity
uint256 unlockTime = block.timestamp + withdrawalDelay; // = block.timestamp + 0
``` [2](#0-1) 

`claimWithdrawal` enforces the lock with:

```solidity
if (block.timestamp < withdrawal.unlockTime) revert WithdrawalNotReady();
``` [3](#0-2) 

When `unlockTime == block.timestamp`, the condition `block.timestamp < block.timestamp` is `false`, so the guard does not trigger. Both calls can be made in the same block, making the delay completely ineffective.

Note: `maxNumberOfWithdrawalsPerUser` also defaults to `0`, which causes `initiateWithdrawal` to revert immediately after deployment. However, the realistic deployment sequence is: admin calls `setMaxNumberOfWithdrawalsPerUser` to open withdrawals, then forgets (or delays) calling `setWithdrawalDelay`. During that window the bypass is live. The setter for `withdrawalDelay` explicitly rejects `0`, confirming the protocol intends a non-zero delay — yet the initializer never enforces this. [4](#0-3) 

---

### Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

The withdrawal delay is a security invariant of the staking contract (capped at `MAX_WITHDRAWAL_DELAY = 30 days`). With `withdrawalDelay = 0`, any staker can exit their position in the same block they initiate the withdrawal, defeating the intended lock-up. No direct fund theft occurs because rewards are time-weighted and a same-block stake/unstake earns nothing, but the protocol's stated security property is not upheld.

---

### Likelihood Explanation
**Medium.** The gap between enabling withdrawals (`setMaxNumberOfWithdrawalsPerUser`) and setting the delay (`setWithdrawalDelay`) is a realistic deployment oversight. Both calls are separate admin transactions with no enforced ordering. Any staker active during that window can exploit the zero delay.

---

### Recommendation
Set a safe default for `withdrawalDelay` (e.g., `7 days`) and `maxNumberOfWithdrawalsPerUser` directly inside `initialize`, mirroring how `LRTWithdrawalManager.initialize` hard-codes `withdrawalDelayBlocks`:

```solidity
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    // ... existing checks ...
    withdrawalDelay = 7 days;                  // safe non-zero default
    maxNumberOfWithdrawalsPerUser = 10;        // safe non-zero default
}
``` [5](#0-4)

### Proof of Concept

```
// Assume admin has called setMaxNumberOfWithdrawalsPerUser(10) but NOT setWithdrawalDelay.
// withdrawalDelay == 0 at this point.

1. attacker calls stake(1000e18)
   → balanceOf[attacker] = 1000e18

2. attacker calls initiateWithdrawal(1000e18)
   → unlockTime = block.timestamp + 0 = block.timestamp
   → Withdrawal record created with unlockTime == block.timestamp

3. In the SAME block, attacker calls claimWithdrawal(withdrawalId)
   → check: block.timestamp < block.timestamp  →  false  →  no revert
   → kernelToken.safeTransfer(attacker, 1000e18) executes immediately

Result: attacker recovers all staked KERNEL with zero waiting period,
        bypassing the up-to-30-day delay the protocol intends to enforce.
``` [6](#0-5) [7](#0-6)

    function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
        if (_amount == 0) revert AmountZero();
        if (balanceOf[msg.sender] < _amount) revert InsufficientStakedBalance();
        if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();

        balanceOf[msg.sender] -= _amount;
        totalKernelStaked -= _amount;

        // Create a withdrawal record
        uint256 withdrawalId = ++withdrawalCounter;
        uint256 unlockTime = block.timestamp + withdrawalDelay;

        withdrawals[withdrawalId] = Withdrawal({
            user: msg.sender, amount: _amount, unlockTime: unlockTime, claimed: false, withdrawalId: withdrawalId
        });
        userWithdrawalIds[msg.sender].push(withdrawalId);

        emit WithdrawalInitiated(msg.sender, _amount, withdrawalId, unlockTime);
    }

    function claimWithdrawal(uint256 _withdrawalId) external nonReentrant {
        Withdrawal storage withdrawal = withdrawals[_withdrawalId];

        if (withdrawal.user == address(0)) {
            revert WithdrawalDoesNotExist();
        }

        if (withdrawal.user != msg.sender) {
            revert NotYourWithdrawal();
        }

        if (block.timestamp < withdrawal.unlockTime) {
            revert WithdrawalNotReady();
        }

        if (withdrawal.claimed) {
            revert WithdrawalAlreadyClaimed();
        }

        withdrawal.claimed = true;

        // Remove the withdrawal ID from the user's list of withdrawal IDs
        uint256[] storage userWithdrawalIdsArray = userWithdrawalIds[msg.sender];
        for (uint256 i = 0; i < userWithdrawalIdsArray.length; ++i) {
            if (userWithdrawalIdsArray[i] == _withdrawalId) {
                userWithdrawalIdsArray[i] = userWithdrawalIdsArray[userWithdrawalIdsArray.length - 1];
                userWithdrawalIdsArray.pop();
                break;
            }
        }

        // Transfer the KERNEL tokens from contract to user
        kernelToken.safeTransfer(msg.sender, withdrawal.amount);

        emit WithdrawalClaimed(msg.sender, withdrawal.amount, _withdrawalId);
    }
