### Title
Fee Rounding to Zero Allows Depositors to Bypass Protocol Fees Entirely - (File: contracts/pools/RSETHPoolV3.sol)

### Summary
All L2 pool contracts compute the deposit fee as `fee = amount * feeBps / 10_000` using integer (truncating) division. When `amount * feeBps < 10_000`, the fee rounds to exactly zero. Because the only input guard is `amount != 0`, any depositor can split deposits into chunks below the rounding threshold and pay zero fee on every transaction, fully bypassing the protocol's fee mechanism.

### Finding Description
Every pool contract in the codebase computes the swap fee with the same truncating formula:

```solidity
// RSETHPoolV3.sol – viewSwapRsETHAmountAndFee (ETH path)
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [1](#0-0)

The deposit entry point only rejects a zero amount:

```solidity
if (amount == 0) revert InvalidAmount();
``` [10](#0-9) 

There is no minimum deposit that guarantees `fee >= 1`. The fee-free threshold per transaction is `floor(10_000 / feeBps)` wei. For a representative `feeBps = 5` (0.05 %), any single deposit of ≤ 1 999 wei produces `fee = 0`. A depositor who sends many such transactions receives wrsETH/rsETH at the full 1:1 ETH-equivalent rate with no fee deducted, while `feeEarnedInETH` remains unchanged.

### Impact Explanation
The protocol fails to collect the fee revenue it is designed to earn. Every wei of fee that rounds to zero is yield that should have accrued to the protocol treasury but does not. This matches the **Low** impact tier: "Contract fails to deliver promised returns, but doesn't lose value." User funds are not at risk; only protocol fee revenue is lost.

### Likelihood Explanation
**High.** The attack requires no special privilege, no flash loan, and no coordination. Any depositor on any supported L2 chain can call `deposit()` with a sub-threshold amount. On L2 networks (Arbitrum, Optimism, Base, etc.) where these pools are deployed, per-transaction gas costs are a fraction of a cent, making repeated small deposits economically rational when the fee saved per unit of deposited value exceeds gas cost — a condition that holds for tokens with high unit prices (e.g., wstETH, ETHx) even at very small nominal amounts.

### Recommendation
Add a minimum-fee guard inside `viewSwapRsETHAmountAndFee` (and its token-path overload) so that a non-zero `feeBps` always produces a non-zero fee:

```solidity
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
+   if (feeBps > 0 && fee == 0) revert DepositAmountTooSmall();
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
```

Alternatively, enforce a protocol-wide minimum deposit amount (e.g., `amount >= 10_000 / feeBps`) so that `fee >= 1` is always guaranteed. Apply the same fix to all token-path overloads and to `AGETHPoolV3.viewSwapAgETHAmountAndFee`.

### Proof of Concept
Assume `feeBps = 5` (0.05 %) on `RSETHPoolV3` and `rsETHToETHrate = 1.05e18` (rsETH trades at a 5 % premium to ETH).

1. Attacker calls `deposit{value: 1999}("")` — the maximum sub-threshold amount.
2. `fee = 1999 * 5 / 10_000 = 0` (integer truncation).
3. `amountAfterFee = 1999`.
4. `rsETHAmount = 1999 * 1e18 / 1.05e18 = 1903` (wei of wrsETH minted).
5. `feeEarnedInETH += 0` — no fee recorded.
6. Attacker repeats N times. For N = 500 000 transactions the attacker deposits ≈ 1 ETH and receives the equivalent wrsETH with **zero fee paid**, whereas a single 1 ETH deposit would have incurred `1e18 * 5 / 10_000 = 5e13` wei (0.00005 ETH) in fees.

The same arithmetic applies to every token-path overload across all affected pool contracts. [1](#0-0)
