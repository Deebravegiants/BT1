### Title
Missing Storage Gap in `LRTConfigRoleChecker` Inherited by Multiple Upgradeable Contracts - (File: contracts/utils/LRTConfigRoleChecker.sol)

---

### Summary

`LRTConfigRoleChecker` is a stateful abstract contract (it declares `ILRTConfig public lrtConfig`) that is inherited by every major upgradeable contract in the protocol. It contains no `__gap` storage reservation. If a new storage variable is ever added to `LRTConfigRoleChecker` during an upgrade, the storage layout of every inheriting proxy contract will be silently corrupted, shifting all downstream slot assignments.

---

### Finding Description

`LRTConfigRoleChecker` declares one storage variable:

```solidity
// contracts/utils/LRTConfigRoleChecker.sol line 13
ILRTConfig public lrtConfig;
``` [1](#0-0) 

There is no `uint256[N] private __gap;` anywhere in this contract. Despite this, every major upgradeable production contract inherits from it:

| Contract | Upgradeable bases also inherited |
|---|---|
| `LRTDepositPool` | `PausableUpgradeable`, `ReentrancyGuardUpgradeable` |
| `NodeDelegator` | `PausableUpgradeable`, `ReentrancyGuardUpgradeable` |
| `RSETH` | `ERC20Upgradeable`, `PausableUpgradeable` |
| `LRTWithdrawalManager` | `PausableUpgradeable`, `ReentrancyGuardUpgradeable` |
| `LRTUnstakingVault` | `PausableUpgradeable`, `ReentrancyGuardUpgradeable` |
| `LRTOracle` | `Initializable` |
| `LRTConfig` | `AccessControlUpgradeable` | [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

All of these contracts use `_disableInitializers()` in their constructors and are deployed behind proxies, confirming they are live upgradeable contracts. [7](#0-6) 

The storage layout for each proxy is determined by the full inheritance chain. Because `LRTConfigRoleChecker` has no gap, any future addition of a state variable to it (e.g., a secondary config reference, a new role cache, or a flag) will shift every slot declared in the inheriting contract downward by one or more positions.

---

### Impact Explanation

If `LRTConfigRoleChecker` is modified to add even a single new storage variable in a future upgrade, the following storage slots in every inheriting contract shift:

- **`LRTDepositPool`**: `maxNodeDelegatorLimit`, `minAmountToDeposit`, `isNodeDelegator`, `nodeDelegatorQueue`, `maxNegligibleAmount` — all shift. Deposit limits, node delegator tracking, and ETH routing become corrupted. User deposits and fund transfers break.
- **`LRTWithdrawalManager`**: `minRsEthAmountToWithdraw`, `withdrawalDelayBlocks`, `nextUnusedNonce`, `withdrawalRequests`, `assetsCommitted` — all shift. Withdrawal queue state is corrupted; users cannot claim funds.
- **`RSETH`**: `maxMintAmountPerDay`, `currentPeriodMintedAmount`, `periodStartTime`, `custodyAddress`, `transfersBlockedUntil` — all shift. Token accounting and transfer-block enforcement break.
- **`NodeDelegator`**: `eigenPod`, `stakedButUnverifiedNativeETH` — all shift. EigenLayer restaking state is corrupted.

The worst-case outcome is permanent freezing of all user funds held across these contracts, or uncontrolled minting/burning of rsETH. [8](#0-7) [9](#0-8) [10](#0-9) 

---

### Likelihood Explanation

The protocol is actively developed and has already performed `reinitializer(2)` upgrades on `RSETH` and `LRTOracle`, demonstrating that upgrades are routine. [11](#0-10) [12](#0-11) 

`LRTConfigRoleChecker` is a shared base that a developer might reasonably extend (e.g., to add a secondary config, a cached role address, or a new modifier flag) without recognizing the storage-layout consequence across all seven inheriting proxy contracts. The risk is not hypothetical — it is a latent time-bomb activated by any routine upgrade that touches this base contract.

---

### Recommendation

Add a storage gap to `LRTConfigRoleChecker` sized to reserve enough slots for future additions:

```solidity
abstract contract LRTConfigRoleChecker {
    ILRTConfig public lrtConfig;

    // Reserve storage slots for future upgrades
    uint256[49] private __gap;
    ...
}
```

Similarly audit all other non-OZ abstract contracts in the inheritance chain (e.g., any future base contracts) for the same omission.

---

### Proof of Concept

1. `LRTConfigRoleChecker` occupies slot 0 with `lrtConfig`.
2. `LRTDepositPool` inherits `LRTConfigRoleChecker` → `PausableUpgradeable` → `ReentrancyGuardUpgradeable`, then declares its own variables starting at the next available slot.
3. If `LRTConfigRoleChecker` is upgraded to insert a new variable after `lrtConfig`, the slot previously holding `maxNodeDelegatorLimit` in `LRTDepositPool` now holds the new variable's value, and `maxNodeDelegatorLimit` reads from the slot that previously held `isNodeDelegator`, etc.
4. All role checks, deposit limits, and node-delegator routing in the live proxy immediately read garbage values — no attacker action required beyond the upgrade itself. [1](#0-0) [13](#0-12)
