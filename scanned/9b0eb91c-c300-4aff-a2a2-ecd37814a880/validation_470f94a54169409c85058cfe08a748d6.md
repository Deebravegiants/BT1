### Title
`dailyMintAmount` Not Decremented on wrsETH Burn Enables Daily Limit Exhaustion - (File: contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPoolV3.sol)

---

### Summary

The `dailyMintAmount` supply tracker in `RSETHPoolV3` and `RSETHPoolV3ExternalBridge` is incremented whenever wrsETH is minted via `deposit()`, but is never decremented when wrsETH is burned via `RsETHTokenWrapper._withdraw()`. This asymmetry allows any unprivileged user to permanently exhaust the daily mint quota within a single day at near-zero net cost, blocking all other depositors until the next day's reset.

---

### Finding Description

Both `RSETHPoolV3` and `RSETHPoolV3ExternalBridge` enforce a daily minting cap through the `limitDailyMint` modifier. Every call to `deposit()` increments `dailyMintAmount` by the rsETH equivalent of the deposited asset: [1](#0-0) 

The minted wrsETH is produced by calling `wrsETH.mint()` on the `RsETHTokenWrapper`: [2](#0-1) 

The `RsETHTokenWrapper` exposes a public `withdraw()` / `withdrawTo()` path that burns wrsETH and returns the underlying altRsETH 1:1: [3](#0-2) 

This burn path calls `_burn()` on the ERC-20 (correctly reducing `totalSupply()` of wrsETH) but has **no callback or notification to the pool**. The pool's `dailyMintAmount` variable is never decremented. The pool and the wrapper are two independent contracts with no shared state for the daily counter.

The same flaw exists identically in `RSETHPoolV3`: [4](#0-3) [5](#0-4) 

---

### Impact Explanation

An attacker can exhaust the entire `dailyMintLimit` within a single transaction sequence at near-zero net cost:

1. Call `pool.deposit{value: X}()` → wrsETH minted, `dailyMintAmount += rsETHEquivalent(X)`.
2. Call `wrapper.withdraw(altRsETH, rsETHAmount)` → wrsETH burned, altRsETH returned, `dailyMintAmount` unchanged.
3. Repeat until `dailyMintAmount == dailyMintLimit`.

All subsequent calls to `pool.deposit()` by any user revert with `DailyMintLimitExceeded` until the next day's reset. The attacker recovers their capital as altRsETH (worth ≈ the deposited ETH minus fees), making the attack nearly free.

**Impact: Temporary freezing of funds (Medium)** — legitimate depositors are locked out of the pool for up to 24 hours.

---

### Likelihood Explanation

The attack entry point (`deposit()` + `withdraw()`) is fully public and requires no special role. The attacker only needs ETH to fund the initial deposit; they recover equivalent value as altRsETH. The only cost is gas and the pool fee (`feeBps`). Given that the daily limit is a meaningful protocol control (it exists to cap exposure), exhausting it has real user-facing impact. Likelihood is **Medium**.

---

### Recommendation

Decrement `dailyMintAmount` when wrsETH is burned. Two approaches:

1. **Pool-side tracking**: Add a `burn` hook or a dedicated `burnAndRefund` function in the pool that decrements `dailyMintAmount` before calling `wrapper.withdraw()`. Gate the wrapper's `withdraw()` to only be callable through the pool (breaking change).

2. **Wrapper callback**: Emit a structured event or call a registered pool callback on `_withdraw()` so the pool can decrement its counter. This requires the wrapper to know which pool minted the tokens.

The cleanest long-term fix is to consolidate mint/burn accounting into a single contract that owns both the supply counter and the burn path, mirroring the recommendation in the reference report.

---

### Proof of Concept

```
// Attacker EOA on an L2 with RSETHPoolV3ExternalBridge + RsETHTokenWrapper

// Step 1: Consume the full daily limit
uint256 limit = pool.dailyMintLimit();          // e.g. 100 ether in rsETH terms
uint256 ethNeeded = limit * oracle.getRate() / 1e18;
pool.deposit{value: ethNeeded}("ref");          // dailyMintAmount = limit

// Step 2: Burn wrsETH — daily counter NOT restored
uint256 bal = wrsETH.balanceOf(attacker);
wrsETH.approve(address(wrapper), bal);
wrapper.withdraw(altRsETH, bal);                // dailyMintAmount still == limit

// Step 3: Victim tries to deposit — reverts
pool.deposit{value: 1 ether}("ref");            // ❌ DailyMintLimitExceeded

// Attacker holds altRsETH ≈ ethNeeded in value; net cost ≈ gas + feeBps
```

```text
    modifier limitDailyMint(uint256 amount, address token) {
        if (block.timestamp < startTimestamp) {
            revert MintBeforeStartTimestamp();
        }

        uint256 rsETHAmount;

        // Calculate the amount of rsETH that will be minted
        if (token == ETH_IDENTIFIER) {
            (rsETHAmount,) = viewSwapRsETHAmountAndFee(amount);
        } else {
            (rsETHAmount,) = viewSwapRsETHAmountAndFee(amount, token);
        }

        uint256 currentDay = getCurrentDay();

        // If the current day is greater than the last mint day, reset the daily mint amount
        if (currentDay > lastMintDay) {
            lastMintDay = currentDay;
            dailyMintAmount = 0;
        }

        // Check if the daily mint amount plus the amount to mint is greater than the daily mint limit
        if (dailyMintAmount + rsETHAmount > dailyMintLimit) {
            revert DailyMintLimitExceeded();
        }

        dailyMintAmount += rsETHAmount;
        _;
```
