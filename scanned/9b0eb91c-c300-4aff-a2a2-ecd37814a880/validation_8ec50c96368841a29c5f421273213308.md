### Title
Unbounded Nested-Loop Gas Consumption in `updateRSETHPrice()` Can Permanently DOS Price Updates - (File: contracts/LRTOracle.sol)

---

### Summary

`updateRSETHPrice()` is a public, permissionless function. Its internal call chain traverses nested loops over supported assets, node delegators, queued EigenLayer withdrawals, and withdrawal strategies. As the protocol grows, this call chain can exceed the block gas limit, permanently preventing rsETH price updates and protocol fee minting.

---

### Finding Description

`updateRSETHPrice()` is callable by any address with no role restriction:

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

`_updateRsETHPrice()` calls `_getTotalEthInProtocol()`, which iterates over every supported asset and for each asset calls `ILRTDepositPool.getTotalAssetDeposits(asset)`:

```solidity
for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
    address asset = supportedAssets[assetIdx];
    uint256 assetER = getAssetPrice(asset);
    uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
    totalETHInProtocol += totalAssetAmt.mulWad(assetER);
    unchecked { ++assetIdx; }
}
```

`getTotalAssetDeposits()` calls `getAssetDistributionData()` (or `getETHDistributionData()`), which loops over every NDC in `nodeDelegatorQueue` and for each NDC calls `INodeDelegator.getAssetUnstaking(asset)`:

```solidity
for (uint256 i; i < ndcsCount;) {
    assetLyingInNDCs += IERC20(asset).balanceOf(nodeDelegatorQueue[i]);
    assetStakedInEigenLayer += INodeDelegator(nodeDelegatorQueue[i]).getAssetBalance(asset);
    assetUnstakingFromEigenLayer += INodeDelegator(nodeDelegatorQueue[i]).getAssetUnstaking(asset);
    unchecked { ++i; }
}
```

`getAssetUnstaking()` in `NodeDelegator.sol` calls `delegationManager.getQueuedWithdrawals(address(this))` — an external call that loads all queued withdrawals for that NDC — and then iterates over every withdrawal and every strategy within each withdrawal:

```solidity
for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
    for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
        ...
    }
}
```

The total gas cost is proportional to:

> **`supportedAssets.length` × `nodeDelegatorQueue.length` × `queuedWithdrawals.length` × `strategies.length`**

`supportedAssets` has **no hard cap** — it grows every time admin calls `addNewSupportedAsset()`. `maxNodeDelegatorLimit` is admin-settable with no upper bound. The protocol's own comment in `LRTUnstakingVault.sol` explicitly acknowledges this constraint:

```solidity
// 120 is the max number of uncompleted withdrawals that allows us to still perform update rsETH price
if (_maxUncompletedWithdrawalCount > 80) {
    revert MaxUncompletedWithdrawalCountTooHigh();
}
```

This confirms the team is aware that `updateRSETHPrice()` has a gas ceiling, but the mitigation only caps total withdrawal count — it does not cap `supportedAssets.length` or `nodeDelegatorQueue.length`, and it does not cap the **product** of all four dimensions.

Critically, `updateRSETHPriceAsManager()` (the privileged fallback) calls the **same** `_updateRsETHPrice()` internal function, so it is equally affected and provides no escape hatch.

---

### Impact Explanation

If `updateRSETHPrice()` always reverts due to block gas exhaustion:

1. **Protocol fee minting is permanently frozen.** The fee minting logic is embedded inside `_updateRsETHPrice()`. No fees can be collected.
2. **`rsETHPrice` is permanently stale.** All deposits (`depositETH`, `depositAsset`) and withdrawals (`initiateWithdrawal`, `instantWithdrawal`) use the stored `rsETHPrice`. A diverging stale price causes depositors to receive incorrect rsETH amounts and withdrawers to receive incorrect asset amounts, leading to value extraction from or loss to existing holders.
3. **Price-drop protection never triggers.** The auto-pause on price decrease is inside `_updateRsETHPrice()` and becomes unreachable.

**Impact: Medium — Unbounded gas consumption leading to permanent freezing of unclaimed yield (protocol fees) and temporary freezing of correct price accounting.**

---

### Likelihood Explanation

The protocol is designed to support multiple LST assets and multiple NodeDelegators. The `supportedAssets` list already includes ETH, stETH, and ETHx, and is expected to grow. Each new asset multiplies the number of `getQueuedWithdrawals()` external calls by the number of NDCs. With 10 NDCs, 10 supported assets, 8 queued withdrawals per NDC, and 2 strategies per withdrawal, the call chain executes ~1,600 inner iterations plus 100 external EigenLayer calls — a realistic near-term configuration that approaches block gas limits. The team's own cap comment confirms this is a known operational boundary.

---

### Recommendation

1. **Cache `getAssetUnstaking()` per NDC across all assets.** A single call to `getQueuedWithdrawals(ndc)` can compute the unstaking amount for all assets simultaneously, reducing external calls from `assets × NDCs` to `NDCs`.
2. **Introduce a hard cap on `supportedAssets.length`** analogous to `maxNodeDelegatorLimit`.
3. **Decouple price computation from full TVL traversal.** Store per-asset TVL snapshots updated incrementally (similar to the `pointHistory` fix in the referenced report) rather than recomputing the full sum on every price update call.

---

### Proof of Concept

Call chain triggered by any unprivileged caller:

```
updateRSETHPrice()                          [LRTOracle.sol:87 — public, no auth]
  └─ _updateRsETHPrice()                    [LRTOracle.sol:214]
       └─ _getTotalEthInProtocol()          [LRTOracle.sol:331]
            └─ for each asset (N):          [LRTOracle.sol:336]
                 └─ getTotalAssetDeposits() [LRTDepositPool.sol:385]
                      └─ getAssetDistributionData() / getETHDistributionData()
                           └─ for each NDC (M):  [LRTDepositPool.sol:447 / 484]
                                └─ getAssetUnstaking(asset)  [NodeDelegator.sol:405]
                                     └─ delegationManager.getQueuedWithdrawals()  [external]
                                          └─ for each withdrawal (W):  [NodeDelegator.sol:409]
                                               └─ for each strategy (S):  [NodeDelegator.sol:412]
```

Total external calls: **N × M**. Total inner iterations: **N × M × W × S**. With N=10 assets, M=10 NDCs, W=8 withdrawals/NDC, S=2 strategies: **1,600 iterations + 100 external calls** — a realistic near-term configuration. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

    function getAssetUnstaking(address asset) external view returns (uint256 amount) {
        (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
            _getDelegationManager().getQueuedWithdrawals(address(this));

        for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
            IDelegationManager.Withdrawal memory withdrawal = queuedWithdrawals[withdrawalIndex];

            for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
                IStrategy strategy = withdrawal.strategies[strategyIndex];

                address strategyAsset = address(strategy) == address(lrtConfig.beaconChainETHStrategy())
                    ? LRTConstants.ETH_TOKEN
                    : address(strategy.underlyingToken());

                if (strategyAsset != asset) continue;

                uint256 sharesToUnstake = withdrawalShares[withdrawalIndex][strategyIndex];
                amount += strategyAsset == LRTConstants.ETH_TOKEN
                    ? sharesToUnstake
                    : strategy.sharesToUnderlyingView(sharesToUnstake);
            }
        }
    }
