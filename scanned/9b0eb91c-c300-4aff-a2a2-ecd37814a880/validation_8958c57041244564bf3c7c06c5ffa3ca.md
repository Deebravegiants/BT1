### Title
L2 Pool Depositors Receive Excess wrsETH When Oracle Rate Is Temporarily Depressed — (File: contracts/pools/RSETHPoolV3ExternalBridge.sol)

### Summary
All L2 pool contracts (`RSETHPoolV3ExternalBridge`, `RSETHPoolV3`, `RSETHPool`, `RSETHPoolNoWrapper`) compute the wrsETH/rsETH amount to issue using the live oracle rate at deposit time, with no staleness guard, no minimum-rate floor, and no protocol-side slippage protection. When the L2 oracle rate is temporarily below the true rsETH/ETH rate — a realistic condition given Chainlink heartbeat delays — a depositor receives more wrsETH than the deposited ETH can back at the true rate. The ETH is later bridged to L1 and converted to rsETH at the true (higher) rate, leaving the wrsETH wrapper undercollateralised and stealing value from all other wrsETH holders.

### Finding Description
Every L2 pool's deposit path resolves to `viewSwapRsETHAmountAndFee`, which divides the post-fee ETH amount by the live oracle rate:

```solidity
// RSETHPoolV3ExternalBridge.sol lines 418-427
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();          // live oracle, no staleness check
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}

`getRate()` simply forwards to the configured `rsETHOracle` with no freshness validation:

```solidity

The minted amount is then issued atomically:

The L1 `LRTOracle` has explicit downside protection: if the computed rsETH price falls too far below `highestRsethPrice`, it pauses `LRTDepositPool` and `LRTWithdrawalManager`:

```solidity

No equivalent circuit-breaker exists in any L2 pool contract. The L2 oracle is a separate contract (`rsETHOracle`) whose rate can lag the true rate for up to the Chainlink heartbeat period (typically 24 hours) without triggering any pause.

### Impact Explanation
**High — Theft of unclaimed yield / protocol insolvency for the wrsETH wrapper.**

When the L2 oracle rate is stale-low (e.g., shows 1.00 ETH/rsETH while the true rate is 1.05 ETH/rsETH):

- Attacker deposits 100 ETH → receives `100 / 1.00 = 100 wrsETH`.
- Fair issuance at true rate: `100 / 1.05 ≈ 95.24 wrsETH`.
- ETH is bridged to L1 and deposited into `LRTDepositPool`, yielding `100 / 1.05 ≈ 95.24 rsETH` to back the wrapper.
- The wrapper now has 100 wrsETH outstanding but only 95.24 rsETH backing — a shortfall of ≈4.76 rsETH.
- When the oracle recovers, the attacker's 100 wrsETH is redeemable for 100 rsETH (≈105 ETH), having paid only 100 ETH.
- The 4.76 rsETH shortfall is socialised across all other wrsETH holders, diluting their claims.

### Likelihood Explanation
**Medium.** Chainlink oracles for LST/ETH rates use heartbeat-based updates (commonly 24 hours) and deviation thresholds (commonly 0.5%). Because the rsETH/ETH rate accrues monotonically from staking rewards, the oracle routinely lags the true rate by the full heartbeat window. An attacker needs only to observe that the on-chain oracle rate is below the true rate (computable from L1 `LRTOracle.rsETHPrice()`) and call `deposit()` before the oracle updates. No privileged access, no front-running of a specific transaction, and no external protocol compromise is required.

### Recommendation
1. **Short term**: Add a staleness check in `getRate()` (or in `viewSwapRsETHAmountAndFee`) that reverts if the oracle's `updatedAt` timestamp is older than an acceptable threshold (e.g., 25 hours).
2. **Short term**: Add a `minRateExpected` parameter to `deposit()` so the protocol can enforce a floor rate, analogous to the `minRSETHAmountExpected` slippage guard in `LRTDepositPool.depositAsset`.
3. **Long term**: Mirror the L1 `pricePercentageLimit` circuit-breaker on L2: if the oracle rate drops more than a configured percentage below its historical high, pause L2 deposits automatically.

### Proof of Concept

**Setup**: L2 `rsETHOracle.getRate()` returns `1.00e18` (stale by 20 hours). True rsETH/ETH rate on L1 is `1.05e18`.

**Step 1** — Attacker calls `RSETHPoolV3ExternalBridge.deposit{value: 100 ether}("")`.

`viewSwapRsETHAmountAndFee(100e18)`:
```
fee = 100e18 * feeBps / 10_000          // e.g. 0 if feeBps=0
rsETHAmount = 100e18 * 1e18 / 1.00e18 = 100e18   // 100 wrsETH minted
``` [1](#0-0) 

**Step 2** — `wrsETH.mint(attacker, 100e18)` executes. Attacker holds 100 wrsETH.

**Step 3** — Bridger calls `bridgeAssets(...)`. 100 ETH is sent to L1 `L1Vault`, which calls `LRTDepositPool.depositETH`. At the true rate of 1.05 ETH/rsETH, `LRTDepositPool.getRsETHAmountToMint` yields:

```
rsethAmountToMint = 100e18 * 1e18 / 1.05e18 ≈ 95.24e18   // rsETH minted to wrapper
``` [7](#0-6) 

**Step 4** — Wrapper now holds ≈95.24 rsETH backing 100 wrsETH. Shortfall: ≈4.76 rsETH.

**Step 5** — Oracle updates to `1.05e18`. Attacker redeems 100 wrsETH for ≈100 rsETH (≈105 ETH). Net profit: ≈5 ETH extracted from other wrsETH holders.
