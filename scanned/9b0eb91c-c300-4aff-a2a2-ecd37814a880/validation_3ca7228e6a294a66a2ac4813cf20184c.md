### Title
ETH Deposit Cap Check Missing `amount` in `_checkIfDepositAmountExceedesCurrentLimit()` - (File: `contracts/LRTDepositPool.sol`)

---

### Summary
`_checkIfDepositAmountExceedesCurrentLimit()` applies an asymmetric cap check: for LST assets it correctly tests `totalAssetDeposits + amount > depositLimit`, but for ETH it omits the incoming `amount` and only tests `totalAssetDeposits > depositLimit`. This means any ETH deposit that would push total deposits above the cap is silently accepted, bypassing the protocol's risk-management limit.

---

### Finding Description
In `LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit()`:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // ← amount not included
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct for LSTs
}
```

The ETH branch evaluates `totalAssetDeposits > limit` rather than `totalAssetDeposits + amount > limit`. Consequently:

- When `totalAssetDeposits == depositLimit`, the check returns `false` (not exceeded), so `_beforeDeposit` does not revert and the deposit proceeds, pushing the total to `depositLimit + amount`.
- When `totalAssetDeposits < depositLimit` but `totalAssetDeposits + amount > depositLimit`, the check still returns `false`, again allowing the deposit to exceed the cap by up to `amount`.

This is called from `_beforeDeposit()`, which is the sole pre-flight guard for both `depositETH()` and `depositAsset()`. [1](#0-0) 

`depositETH()` is the public, permissionless entry point that routes through `_beforeDeposit()` → `_checkIfDepositAmountExceedesCurrentLimit()`: [2](#0-1) [3](#0-2) 

`getTotalAssetDeposits(ETH)` aggregates all ETH held across the deposit pool, NDCs, EigenLayer pods, the converter, and the unstaking vault — it is the authoritative total used for the limit comparison: [4](#0-3) 

---

### Impact Explanation
The ETH deposit cap is a protocol-level risk-management parameter set by governance. Bypassing it allows more ETH than intended to flow into EigenLayer strategies, undermining the cap's purpose. No funds are directly stolen or frozen, but the protocol fails to enforce its own promised deposit ceiling for ETH.

**Impact: Low — Contract fails to deliver promised returns, but doesn't lose value.**

---

### Likelihood Explanation
The entry point `depositETH()` is public and requires no special role. Any depositor can trigger this path. The condition is met whenever `totalAssetDeposits` is at or near the limit, which is a normal operational state once the protocol approaches capacity. Likelihood is **High**.

### Recommendation

```diff
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
-       return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
+       return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This makes the ETH branch consistent with the LST branch and correctly accounts for the incoming deposit amount before comparing against the cap.

---

### Proof of Concept

1. Governance sets `depositLimitByAsset[ETH] = 1000 ether`.
2. Protocol accumulates `totalAssetDeposits(ETH) = 1000 ether` (exactly at the cap).
3. User calls `depositETH{value: 100 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH, 100 ether)` evaluates `1000 ether > 1000 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for the user; total ETH in protocol becomes `1100 ether`, 10% above the intended cap.
6. The same user (or any other) can repeat this indefinitely as long as `totalAssetDeposits <= depositLimit` at the time of each call, continuously exceeding the cap.

For LSTs, step 4 would evaluate `1000 ether + 100 ether > 1000 ether` → `true` → revert, correctly blocking the deposit.
