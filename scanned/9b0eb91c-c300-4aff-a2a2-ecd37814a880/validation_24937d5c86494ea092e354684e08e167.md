### Title
Stale Chainlink Price Accepted Without Validation in Core Oracle, Enabling rsETH Price Manipulation - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all validity fields (`updatedAt`, `answeredInRound`, `roundId`), accepting any price — including stale or erroneous ones — without any check. This is the direct analog to the reported vulnerability: the protocol's own L2 Chainlink wrapper (`ChainlinkOracleForRSETHPoolCollateral`) performs staleness and validity checks, but the L1 core oracle does not. A stale or erroneous Chainlink price for any supported LST asset propagates into `rsETHPrice`, which governs both minting and withdrawals.

### Finding Description
`ChainlinkPriceOracle.getAssetPrice()` fetches the Chainlink price as follows:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol line 52
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All five return values of `latestRoundData()` are `(roundId, answer, startedAt, updatedAt, answeredInRound)`. The code destructures only `answer` and silently discards `roundId`, `startedAt`, `updatedAt`, and `answeredInRound`. No check is performed for:
- `answeredInRound >= roundId` (stale round)
- `updatedAt > 0` (incomplete round)
- `updatedAt > block.timestamp - heartbeat` (price freshness)
- `price > 0` (valid price)

By contrast, the protocol's own `ChainlinkOracleForRSETHPoolCollateral.getRate()` — used for L2 pool collateral — performs all three structural checks:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol lines 30-32
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The unvalidated price from `ChainlinkPriceOracle.getAssetPrice()` flows into `LRTOracle._getTotalEthInProtocol()`, which sums `assetPrice × totalAssetDeposits` for every supported LST. This total is then used in `_updateRsETHPrice()` to compute and store `rsETHPrice`. The `pricePercentageLimit` guard in `_updateRsETHPrice()` is initialized to `0` (no limit set in `initialize()`), meaning the condition `pricePercentageLimit > 0 && ...` is always false and provides no protection in the default state.

### Impact Explanation
**Critical — Direct theft of user funds.**

If Chainlink reports a stale or inflated price for any supported LST asset (e.g., stETH/ETH, rETH/ETH):

1. `_getTotalEthInProtocol()` returns an inflated `totalETHInProtocol`.
2. `_updateRsETHPrice()` stores an inflated `rsETHPrice`.
3. `LRTWithdrawalManager.instantWithdrawal()` calls `getExpectedAssetAmount()`, which computes `rsETHUnstaked × rsETHPrice / assetPrice`. With an inflated `rsETHPrice`, the attacker receives more LST than their rsETH is worth.
4. The excess LST is drawn from the pool, directly reducing the assets backing other depositors' rsETH — a theft from remaining holders.

The `_calculatePayoutAmount()` function in the queued withdrawal path also uses `rsETHPrice` directly, so the same inflation affects queued withdrawals unlocked via `unlockQueue()`.

### Likelihood Explanation
Chainlink feeds go stale during network congestion, oracle node outages, or periods of low market activity. The `latestRoundData()` call will return the last stored answer with a stale `updatedAt` timestamp; the contract has no mechanism to detect this. The public `updateRSETHPrice()` function can be called by any address, so an attacker can trigger the price update at the moment a stale inflated price is present. This is a realistic, externally triggerable condition requiring no privileged access.

### Recommendation
Add staleness and validity checks to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    // Optionally: if (block.timestamp - updatedAt > MAX_HEARTBEAT) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Additionally, ensure `pricePercentageLimit` is set to a non-zero value during deployment so the guard in `_updateRsETHPrice()` is active.

### Proof of Concept

1. Chainlink's stETH/ETH feed stops updating (node outage, congestion). The last stored answer is `1.05e18` (stETH = 1.05 ETH), but the true current price has dropped to `1.00e18`.

2. An attacker holds `X` rsETH (acquired at the correct price).

3. The attacker calls `LRTOracle.updateRSETHPrice()` (public, no access control). Inside:
   - `ChainlinkPriceOracle.getAssetPrice(stETH)` returns `1.05e18` (stale, no check).
   - `_getTotalEthInProtocol()` returns an inflated total.
   - `rsETHPrice` is updated upward.

4. The attacker calls `LRTWithdrawalManager.instantWithdrawal(stETH, X, "")`:
   - `getExpectedAssetAmount(stETH, X)` = `X × rsETHPrice_inflated / 1.05e18`.
   - The attacker receives more stETH than their rsETH is worth at the true price.

5. The surplus stETH is taken from the pool, reducing the backing for all other rsETH holders.
