### Title
Missing Staleness Check on Cross-Chain Rate Enables Stale-Rate Deposits via Block Stuffing - (File: contracts/cross-chain/CrossChainRateReceiver.sol, contracts/pools/RSETHPoolV3.sol)

---

### Summary

`CrossChainRateReceiver` stores a `lastUpdated` timestamp when a rate is received from L1 via LayerZero, but neither `CrossChainRateReceiver.getRate()` nor `RSETHPoolV3.getRate()` / `viewSwapRsETHAmountAndFee()` ever consult that timestamp. An attacker who can delay `updateRate()` calls on the source chain — including via block stuffing — can force `RSETHPoolV3.deposit()` to price rsETH using an arbitrarily old rate, causing depositors to receive more or fewer tokens than the current L1 exchange rate warrants.

---

### Finding Description

**Rate update path:**

1. Anyone calls `MultiChainRateProvider.updateRate()` on L1 (permissionless, `payable`). It fetches the current rate, stores it locally, and sends a LayerZero message to every registered `CrossChainRateReceiver`. [1](#0-0) 

2. On the destination chain, `CrossChainRateReceiver.lzReceive()` stores the new rate and records `lastUpdated = block.timestamp`. [2](#0-1) 

3. `CrossChainRateReceiver.getRate()` returns `rate` with **no reference to `lastUpdated`**. [3](#0-2) 

**Consumption path in the pool:**

4. `RSETHPoolV3.getRate()` delegates directly to `IOracle(rsETHOracle).getRate()` — no staleness guard. [4](#0-3) 

5. `viewSwapRsETHAmountAndFee()` (both overloads) uses `getRate()` to compute the rsETH amount, again with no freshness check. [5](#0-4) 

6. `deposit()` calls `viewSwapRsETHAmountAndFee()` and mints directly from the result. [6](#0-5) 

**Block-stuffing vector:**

Because `updateRate()` is permissionless, the only way to prevent a rate refresh is to prevent the transaction from landing. An attacker who stuffs blocks on the source chain for `T` blocks keeps `CrossChainRateReceiver.lastUpdated` frozen at `T_old`. Since rsETH accrues staking yield over time, the stored rate drifts below the true L1 rate. Every `deposit()` call during the stuffing window uses the stale (lower) rate, minting more rsETH per ETH than the current exchange rate justifies. The `lastUpdated` field is never read anywhere in the deposit flow, so the pool has no mechanism to detect or reject the stale value.

---

### Impact Explanation

Depositors receive rsETH calculated from a rate that is lower than the true L1 rsETH/ETH rate. They are over-issued rsETH relative to what the protocol intends. The protocol does not lose principal, but it fails to deliver the correct exchange rate — matching **Low: contract fails to deliver promised returns without principal loss**.

---

### Likelihood Explanation

Block stuffing on Ethereum L1 is expensive but has been demonstrated in practice on low-throughput windows. The attack is more feasible on cheaper L2 source chains. Crucially, the missing staleness check is a latent defect independent of block stuffing: any period of LayerZero relayer downtime, network congestion, or simple keeper inactivity produces the same stale-rate condition with zero attacker cost. The permissionless nature of `updateRate()` does not protect against this because the attacker only needs to delay it, not permanently block it.

---

### Recommendation

Add a configurable `maxStaleness` parameter to `CrossChainRateReceiver` and revert in `getRate()` if `block.timestamp - lastUpdated > maxStaleness`:

```solidity
// CrossChainRateReceiver.sol
uint256 public maxStaleness; // e.g. 24 hours

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

This propagates automatically to `RSETHPoolV3.getRate()` and all callers of `viewSwapRsETHAmountAndFee()` without changes to the pool contract.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Invariant test (Foundry) — run on a local fork
// Asserts: if block.timestamp - CrossChainRateReceiver.lastUpdated > THRESHOLD,
//          RSETHPoolV3.deposit() should revert. Currently it does NOT.

import "forge-std/Test.sol";

interface ICrossChainRateReceiver {
    function lastUpdated() external view returns (uint256);
    function rate() external view returns (uint256);
}

interface IRSETHPoolV3 {
    function deposit(string memory referralId) external payable;
    function getRate() external view returns (uint256);
}

contract StaleRateInvariantTest is Test {
    uint256 constant STALENESS_THRESHOLD = 24 hours;

    ICrossChainRateReceiver receiver = ICrossChainRateReceiver(RECEIVER_ADDR);
    IRSETHPoolV3 pool = IRSETHPoolV3(POOL_ADDR);

    function testStaleRateDepositSucceeds() public {
        // Warp time forward to simulate block stuffing / keeper downtime
        vm.warp(receiver.lastUpdated() + STALENESS_THRESHOLD + 1);

        uint256 staleness = block.timestamp - receiver.lastUpdated();
        assertGt(staleness, STALENESS_THRESHOLD, "rate not stale");

        // deposit() should revert if staleness protection existed — it does NOT
        // The call succeeds, minting rsETH at the old (lower) rate
        vm.deal(address(this), 1 ether);
        pool.deposit{value: 1 ether}("test"); // succeeds — BUG
    }
}
```

The test demonstrates that after warping past the staleness threshold, `deposit()` succeeds and mints rsETH at the stale rate. With a proper staleness guard in `getRate()`, the call would revert. [7](#0-6) [4](#0-3)
