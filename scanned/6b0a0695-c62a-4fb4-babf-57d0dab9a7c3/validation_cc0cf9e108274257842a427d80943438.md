### Title
No Slippage Control on User-Facing `deposit()` Functions - (`contracts/pools/RSETHPoolV2ExternalBridge.sol`, `contracts/pools/RSETHPoolNoWrapper.sol`)

### Summary
The `deposit()` functions in the L2 pool contracts (`RSETHPoolV2ExternalBridge`, `RSETHPoolNoWrapper`) accept ETH from users and mint/transfer rsETH (or wrsETH) based on a live oracle rate, but provide no `minRsETHAmountExpected` parameter. A user's transaction can sit in the mempool while the oracle rate moves unfavorably, causing the user to receive materially fewer rsETH tokens than anticipated with no on-chain protection.

### Finding Description

`RSETHPoolV2ExternalBridge.deposit()` computes the rsETH output entirely from the live oracle rate at execution time:

```solidity
// contracts/pools/RSETHPoolV2ExternalBridge.sol L289-301
function deposit(string memory referralId) external payable nonReentrant whenNotPaused limitDailyMint(msg.value) {
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
``` [1](#0-0) 

The rsETH amount is derived from `viewSwapRsETHAmountAndFee`, which divides by the live oracle rate:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [2](#0-1) 

The identical pattern exists in `RSETHPoolNoWrapper.deposit()`: [3](#0-2) 

Neither function accepts a `minRsETHAmountExpected` parameter. There is also no deadline/expiration check.

By contrast, the L1 `LRTDepositPool.depositAsset()` correctly includes a `minRSETHAmountExpected` guard: [4](#0-3) 

### Impact Explanation

If the rsETH/ETH oracle rate increases between the time a user submits the transaction and when it is mined (e.g., due to mempool congestion, network conditions, or deliberate delay), the user receives fewer rsETH tokens for the same ETH sent. The ETH is irrevocably transferred to the pool; the user cannot recover the difference. This constitutes a failure to deliver the promised return — the user receives a sub-optimal exchange rate with no on-chain recourse.

**Impact: Low** — Contract fails to deliver promised returns, but doesn't lose value (user receives fewer rsETH than expected; the ETH is not stolen but the exchange is unfavorable).

### Likelihood Explanation

Any unprivileged depositor calling `deposit()` on L2 is exposed. L2 networks (Arbitrum, Optimism, etc.) can have variable mempool conditions. The rsETH oracle rate is expected to increase monotonically over time as staking rewards accrue, meaning the rate will always drift upward — making delayed execution consistently unfavorable for depositors. No special conditions or attacker involvement are required; ordinary network congestion is sufficient.

### Recommendation

Add a `minRsETHAmountExpected` parameter to `deposit()` in `RSETHPoolV2ExternalBridge` and `RSETHPoolNoWrapper`, and revert if the computed `rsETHAmount` falls below it:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountExpected)
    external payable nonReentrant whenNotPaused limitDailyMint(msg.value)
{
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountExpected) revert InsufficientRsETHAmount();
    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
```

Optionally, add a `deadline` parameter and check `block.timestamp <= deadline` to guard against stale execution.

### Proof of Concept

1. User observes oracle rate = 1.05 ETH/rsETH and submits `deposit{value: 1 ether}("ref")` expecting ~0.952 wrsETH.
2. Transaction sits in the mempool. Staking rewards accrue; oracle rate updates to 1.10 ETH/rsETH.
3. Transaction executes: `rsETHAmount = 1e18 * 1e18 / 1.10e18 ≈ 0.909e18` — user receives ~0.909 wrsETH instead of ~0.952 wrsETH (~4.5% loss) with no revert.
4. User has no on-chain mechanism to prevent this outcome.
