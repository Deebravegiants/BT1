### Title
`bridgeKernelToBSC` Reverts with Arithmetic Underflow When `counter == 0` — (`contracts/KERNEL/KernelVaultETH.sol`)

---

### Summary

`KernelVaultETH.bridgeKernelToBSC` unconditionally executes `lastBridgedDepositId = counter - 1` before performing the bridge. When `counter` is `0` (its default value, set only by `_depositKernel`), Solidity 0.8.x checked arithmetic causes an underflow revert, making the bridge function permanently uncallable until at least one deposit is recorded through the normal deposit path.

---

### Finding Description

In `contracts/KERNEL/KernelVaultETH.sol`, `counter` is a `uint256` storage variable initialized to `0` by default and incremented only inside `_depositKernel`: [1](#0-0) 

`bridgeKernelToBSC` (operator-restricted) computes: [2](#0-1) 

Under Solidity `0.8.27`, this is checked arithmetic. When `counter == 0`, the subtraction `0 - 1` triggers an `Panic(0x11)` revert before the LayerZero `send` call is ever reached.

The balance guard on line 238 passes independently of `counter`: [3](#0-2) 

So a vault that holds KERNEL tokens acquired via a direct ERC-20 transfer (bypassing `depositKernel`) satisfies the balance check but still reverts at line 262.

`KernelVaultETH` does not inherit `Recoverable` and contains no `recoverTokens`, `emergencyWithdraw`, or equivalent function — confirmed by a full read of the contract. The only egress path for KERNEL tokens is `bridgeKernelToBSC`. [4](#0-3) 

---

### Impact Explanation

**Medium. Temporary freezing of funds.**

The claimed Critical/permanent impact is overstated for two reasons:

1. **Self-resolving**: As soon as any single call to `depositKernel` or `depositKernelFor` succeeds, `counter` becomes `1`, making `counter - 1 = 0` valid. After that, `bridgeKernelToBSC` can bridge the entire vault balance — including any directly-transferred tokens — because the function bridges by amount, not by deposit record.
2. **Upgradeable contract**: `KernelVaultETH` uses the OpenZeppelin upgradeable proxy pattern (`Initializable`), so an admin upgrade can add a recovery path.

The real impact is that KERNEL tokens sent directly to the vault are frozen for the window between the direct transfer and the first legitimate `depositKernel` call. During that window, the operator cannot bridge any vault funds regardless of balance.

---

### Likelihood Explanation

Low-to-medium. The precondition (`counter == 0`, tokens in vault via direct transfer) is most likely to occur:
- At contract deployment before any user has deposited, if the protocol seeds the vault with KERNEL tokens directly.
- If a user accidentally sends KERNEL to the vault address instead of calling `depositKernel`.

The operator calling `bridgeKernelToBSC` in this state is a plausible operational action (e.g., bridging a seed allocation).

---

### Recommendation

Guard against the zero-counter case. The simplest fix is to only assign `lastBridgedDepositId` when `counter > 0`, or use a sentinel value:

```solidity
// Before: always underflows when counter == 0
lastBridgedDepositId = counter - 1;

// After: safe assignment
if (counter > 0) {
    lastBridgedDepositId = counter - 1;
}
```

Alternatively, initialize `counter` to `1` in `initialize()` and use deposit IDs starting from `1`, so `counter - 1` is always valid after initialization.

Also consider adding a `recoverTokens` function (or inheriting `Recoverable`) to provide an admin escape hatch for tokens that arrive outside the normal deposit flow.
