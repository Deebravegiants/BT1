### Title
Zero `rsETHAmount` Deposit Proceeds Without Revert, Consuming User Funds for Zero wrsETH - (File: contracts/pools/RSETHPoolV3.sol)

---

### Summary

All L2 pool deposit functions (`RSETHPoolV2`, `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPoolNoWrapper`) guard against `amount == 0` but never check whether the computed `rsETHAmount` is zero after the rate conversion. Due to integer division truncation in `viewSwapRsETHAmountAndFee`, a non-zero deposit amount can produce `rsETHAmount = 0`. The function then transfers the user's ETH or tokens into the pool, mints 0 wrsETH to the user, and emits a `SwapOccurred` event with `rsETHAmount = 0` — the user permanently loses their deposited assets and receives nothing.

---

### Finding Description

Every pool deposit function follows this pattern (shown for `RSETHPoolV3.sol`):

```solidity
// RSETHPoolV3.sol L246-265 (ETH path)
function deposit(string memory referralId) external payable ... {
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();          // only guards zero input

    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);

    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);             // mints 0 if rsETHAmount == 0
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
```

The rate conversion in `viewSwapRsETHAmountAndFee` uses integer division:

```solidity
// RSETHPoolV3.sol L299-308
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;   // truncates to 0
}
```

For the token path:

```solidity
// RSETHPoolV3.sol L324-334
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;  // truncates to 0
```

Whenever `amountAfterFee * 1e18 < rsETHToETHrate` (ETH path) or `amountAfterFee * tokenToETHRate < rsETHToETHrate` (token path), Solidity integer division yields `rsETHAmount = 0`. There is no subsequent guard:

```solidity
// No check like: if (rsETHAmount == 0) revert InvalidAmount();
```

The same root cause is present identically in:
- `contracts/pools/RSETHPoolV2.sol` L207-219, L225-234
- `contracts/pools/RSETHPoolV3ExternalBridge.sol` L366-412, L418-453
- `contracts/pools/RSETHPoolV3WithNativeChainBridge.sol` L282-329, L335-371
- `contracts/pools/RSETHPoolNoWrapper.sol` L231-271, L277-300

---

### Impact Explanation

When `rsETHAmount` truncates to zero:

- **ETH path**: The user's ETH is accepted by the pool (`msg.value` is consumed), `feeEarnedInETH` is incremented, `wrsETH.mint(user, 0)` is called (no-op), and `SwapOccurred` is emitted with `rsETHAmount = 0`. The user's ETH is permanently in the pool with no wrsETH issued. There is no depositor-facing withdrawal path to recover it.
- **Token path**: `safeTransferFrom` pulls the user's tokens into the pool, `feeEarnedInToken` is incremented, `wrsETH.mint(user, 0)` is called, and `SwapOccurred` is emitted with `rsETHAmount = 0`. Same permanent loss.

The user receives zero shares for a non-zero deposit — the contract fails to deliver its promised return. The deposited assets are irrecoverable by the depositor.

**Impact: Low — Contract fails to deliver promised returns.**

---

### Likelihood Explanation

Since `rsETHToETHrate` is always slightly above `1e18` (rsETH appreciates over time), even a 1-wei ETH deposit satisfies `1 * 1e18 / rsETHToETHrate = 0`. For token deposits, the threshold scales with `rsETHToETHrate / tokenToETHRate`; for tokens with lower ETH value, the zero-output threshold is proportionally larger (e.g., hundreds of wei). Any user who sends a dust deposit — accidentally or through a UI rounding error — triggers this path. No special permissions or oracle manipulation are required; any unprivileged depositor can reach this code path directly.

---

### Recommendation

Add a zero-output guard immediately after computing `rsETHAmount` in every pool deposit function:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
if (rsETHAmount == 0) revert InvalidAmount();
```

This mirrors the fix recommended in the reference report and ensures the function reverts before any state change or event emission when the computed output is zero.

---

### Proof of Concept

Assume `RSETHPoolV3` is deployed with `feeBps = 0` and `rsETHToETHrate = 1.05e18` (rsETH has appreciated 5%).

1. Attacker/user calls `deposit{value: 1}("")` (1 wei ETH).
2. `viewSwapRsETHAmountAndFee(1)`:
   - `fee = 1 * 0 / 10_000 = 0`
   - `amountAfterFee = 1`
   - `rsETHAmount = 1 * 1e18 / 1.05e18 = 0` (integer truncation)
3. `feeEarnedInETH += 0` (no fee, but ETH is in the pool).
4. `wrsETH.mint(msg.sender, 0)` — user receives 0 wrsETH.
5. `emit SwapOccurred(msg.sender, 0, 0, referralId)` — misleading event.
6. User's 1 wei is permanently in the pool; no recovery path exists.

For the token path with a token where `tokenToETHRate = 1e15` (token worth 0.001 ETH) and `rsETHToETHrate = 1.05e18`, any deposit of `amount < 1050` wei produces `rsETHAmount = 0` while the tokens are transferred out of the user's wallet.
