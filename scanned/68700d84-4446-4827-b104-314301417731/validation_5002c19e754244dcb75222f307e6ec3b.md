### Title
Asymmetric `priceLimitX64 = 0` Handling Silently Blocks All `zeroForOne = false` Swaps — (`metric-core/contracts/MetricOmmPool.sol`)

### Summary

When a caller passes `priceLimitX64 = 0` to `swap()` with `zeroForOne = false` (buying token0 with token1), the internal routing functions `_swapToken1ForToken0SpecifiedInput` and `_swapToken1ForToken0SpecifiedOutput` immediately return `(0, 0, 0)` without executing any trade. The same sentinel value `priceLimitX64 = 0` works correctly as "no lower bound" for `zeroForOne = true` swaps, creating a silent, asymmetric failure mode.

### Finding Description

The `swap()` entry point accepts `priceLimitX64 = 0` without validation for either direction.

For `zeroForOne = true` (selling token0, price moves down), the initial guard in `_swapToken0ForToken1SpecifiedInput` is:

```solidity
if (params.priceLimitX64 >= initialPriceX64) {
    return (0, 0, 0);
}
```

With `priceLimitX64 = 0` and any positive `initialPriceX64`, `0 >= initialPriceX64` is **false** → the swap proceeds normally. Inside the loop the per-bin guard is `if (params.priceLimitX64 != 0 && ...)`, so `priceLimitX64 = 0` is treated as "no lower bound" and the swap traverses all bins.

For `zeroForOne = false` (selling token1, price moves up), the initial guard in `_swapToken1ForToken0SpecifiedInput` is:

```solidity
if (params.priceLimitX64 <= initialPriceX64) {
    return (0, 0, 0);
}
```

With `priceLimitX64 = 0` and any positive `initialPriceX64`, `0 <= initialPriceX64` is **always true** → the function returns `(0, 0, 0)` immediately, before any bin is touched. The same early-exit exists in `_swapToken1ForToken0SpecifiedOutput`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

Back in `swap()`, when `_executeSwap` returns `(0, 0, 0)`, the code still calls the swap callback with `(0, 0, callbackData)` and emits a `Swap` event, so the transaction succeeds silently with zero token movement. [5](#0-4) 

### Impact Explanation

Any caller — including the periphery router or an integrator — that passes `priceLimitX64 = 0` for a `zeroForOne = false` swap (the natural "no upper-bound" sentinel, mirroring how `0` works for `zeroForOne = true`) receives `(0, 0)` from `swap()`. The user's transaction does not revert; it silently executes a no-op. The user receives **zero output tokens** instead of the expected amount, and any downstream logic in a multi-hop router that relies on the returned deltas will operate on stale or zero values. This constitutes broken core swap functionality and an unusable swap flow for a valid, unprivileged input.

### Likelihood Explanation

`priceLimitX64 = 0` is the natural "unlimited" sentinel for `zeroForOne = true` and is documented nowhere as invalid for `zeroForOne = false`. Any router, aggregator, or direct caller that symmetrically uses `0` to mean "no price limit" for both directions will silently receive zero output on every `zeroForOne = false` swap. The `swap()` function imposes no validation on `priceLimitX64`. [6](#0-5) 

### Recommendation

Add a direction-aware sentinel: treat `priceLimitX64 = 0` as `type(uint128).max` when `zeroForOne = false` (meaning "no upper bound"), or revert with a descriptive error when `priceLimitX64 = 0` is passed for `zeroForOne = false` so callers are not silently misled. Alternatively, document and enforce that callers must pass `type(uint128).max` for "no upper bound" on `zeroForOne = false` swaps, and add a `require(priceLimitX64 != 0 || !zeroForOne_is_false, ...)` guard.

### Proof of Concept

1. Deploy a pool with token0/token1, add liquidity to bin 0.
2. Call `pool.swap(recipient, false, 1e18, 0, callbackData, "")` — `zeroForOne = false`, `priceLimitX64 = 0`.
3. Observe: `_swapToken1ForToken0SpecifiedInput` hits `if (0 <= initialPriceX64) return (0,0,0)` immediately.
4. `swap()` calls the callback with `(0, 0, callbackData)`, emits `Swap(... 0, 0 ...)`, and returns `(0, 0)`.
5. No token0 is transferred to `recipient`; no token1 is pulled from the caller.
6. Repeat with `zeroForOne = true, priceLimitX64 = 0`: the swap executes normally across all bins, confirming the asymmetry. [7](#0-6) [8](#0-7)
