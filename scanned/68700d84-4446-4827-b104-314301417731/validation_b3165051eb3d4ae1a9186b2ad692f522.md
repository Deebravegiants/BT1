### Title
Zero `priceProviderTimelock` Allows Pool Admin to Bypass Oracle Timelock and Instantly Swap in a Malicious Price Provider — (`File: metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.createPool` accepts `priceProviderTimelock = 0` without any minimum-value guard. When this value is stored, the two-step oracle-rotation flow (`proposePoolPriceProvider` → `executePoolPriceProviderUpdate`) can be completed atomically in a single block, giving LPs zero reaction time before a malicious price provider is active and swap prices are manipulated.

---

### Finding Description

The M01 audit finding (documented in `metric-core/docs/AuditFindings.md`) identified that an admin could arbitrarily replace the pool oracle at runtime. The fix introduced a two-step, timelocked rotation:

1. `proposePoolPriceProvider` records `executeAfter = block.timestamp + priceProviderTimelock[pool]`.
2. `executePoolPriceProviderUpdate` enforces `block.timestamp < execAfter`.

However, `_validatePoolParameters` imposes **no minimum on `priceProviderTimelock`**:

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol  _validatePoolParameters (lines 548-563)
// No check on params.priceProviderTimelock
```

When `priceProviderTimelock = 0` is supplied at pool creation:

```solidity
// proposePoolPriceProvider (line 487)
uint256 executeAfter = block.timestamp + timelock;   // = block.timestamp + 0 = block.timestamp
```

```solidity
// executePoolPriceProviderUpdate (line 499)
if (block.timestamp < execAfter) revert ...;         // block.timestamp < block.timestamp → false → passes
```

Both calls succeed in the **same block** (or even the same transaction via a multicall). The `_validatePriceProvider` check only verifies that the new provider's `token0()`/`token1()` match the pool's tokens — it does not validate price sanity — so a malicious provider that returns extreme bid/ask values passes validation.

---

### Impact Explanation

Once the malicious price provider is active, every subsequent `swap` call reads its manipulated bid/ask. The pool's `SwapMath` uses those prices to compute how many output tokens to send to the recipient. A provider returning a near-zero ask (for a `zeroForOne` swap) or a near-infinite bid (for the reverse) causes the pool to transfer far more output tokens than the input warrants, draining LP principal directly. All token balances held by LPs in every bin are at risk.

---

### Likelihood Explanation

Any permissionless pool creator can set `priceProviderTimelock = 0`. The pool can be presented with attractive parameters (low fees, favorable bin geometry) to attract LP deposits. The malicious oracle rotation can be executed atomically after sufficient TVL accumulates. The on-chain `priceProviderTimelock` value is readable, but there is no UI-level warning, no protocol-level enforcement, and no minimum enforced by the factory. The M01 fix was explicitly intended to give LPs reaction time; a zero timelock silently voids that guarantee.

---

### Recommendation

Add a minimum timelock floor in `_validatePoolParameters`:

```solidity
uint256 internal constant MIN_PRICE_PROVIDER_TIMELOCK = 2 days; // example

// inside _validatePoolParameters:
if (params.priceProviderTimelock != type(uint256).max &&
    params.priceProviderTimelock < MIN_PRICE_PROVIDER_TIMELOCK) {
    revert PriceProviderTimelockTooShort();
}
```

This ensures that even for mutable-oracle pools, LPs always have a meaningful window to observe a pending oracle change and exit before it takes effect.

---

### Proof of Concept

```solidity
// 1. Deploy pool with zero timelock (mutable oracle, no delay)
PoolParameters memory params = ...;
params.priceProviderTimelock = 0;          // NOT type(uint256).max → mutable
address pool = factory.createPool(params);

// 2. LPs deposit; TVL accumulates

// 3. Admin deploys malicious oracle (correct tokens, extreme prices)
MaliciousOracle mal = new MaliciousOracle(token0, token1);
// mal.getBidAsk() returns (1, type(uint128).max)

// 4. Propose + execute in the same block (or same tx via multicall)
vm.prank(admin);
factory.proposePoolPriceProvider(pool, address(mal));
// executeAfter = block.timestamp + 0 = block.timestamp

vm.prank(admin);
factory.executePoolPriceProviderUpdate(pool);
// block.timestamp < block.timestamp → false → succeeds immediately

// 5. Drain: swap with malicious prices active
// Pool sends near-full token1 balance for 1 wei of token0
```

**Root cause**: `_validatePoolParameters` (lines 548–563) has no guard on `params.priceProviderTimelock`, allowing `0` to be stored in `priceProviderTimelock[pool]` (line 213), which makes the `block.timestamp < execAfter` check in `executePoolPriceProviderUpdate` (line 499) trivially false in the same block as `proposePoolPriceProvider` (line 487). ## M01 - Admin can arbitrarily change the pool priceProvider

### Finding summary

`setPriceProvider()` allowed admin to replace oracle at runtime, and `priceProvider` was returned by `getImmutables()`.

### Resolution

Fixed.

Changes:

- Added constructor config `priceProviderTimelock`.
- Added admin flow:
  - `proposePriceProvider(address newPriceProvider)`
  - `executePriceProviderUpdate()`
- Added events:
  - `PriceProviderChangeProposed(currentPriceProvider, newPriceProvider, executeAfter)`
  - `PriceProviderUpdated(newPriceProvider)`

Behavior now:

- `priceProviderTimelock == type(uint256).max` => immutable mode (admin updates disabled).
- `priceProviderTimelock < type(uint256).max` => mutable mode with delayed execution.
- Price provider used by swaps/reports resolves through timelock mode; active provider is read via `MetricOmmPoolStateView.priceProvider(pool)`.
