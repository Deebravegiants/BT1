### Title
Duplicate Bin Indices in `removeLiquidity` / `addLiquidity` `LiquidityDelta` Allow Double-Withdrawal of LP Assets — (`metric-core/contracts/MetricOmmPool.sol`)

### Summary
`MetricOmmPool.addLiquidity` and `removeLiquidity` accept a `LiquidityDelta` struct containing parallel arrays `binIdxs` and `shares`. Neither function performs a uniqueness check on `binIdxs`. If a caller supplies the same bin index more than once, `LiquidityLib` will process that bin multiple times in a single call, directly analogous to the Redemptions duplicate-token payout bug.

### Finding Description

`addLiquidity` and `removeLiquidity` both validate only that the two arrays have equal length: [1](#0-0) [2](#0-1) 

There is no guard that requires each element of `binIdxs` to be unique. The arrays are forwarded directly to `LiquidityLib.removeLiquidity` / `LiquidityLib.addLiquidity`, which iterate over every `(binIdx, shares)` pair: [3](#0-2) 

Pool state for each bin is tracked in two mappings: [4](#0-3) 

If the same `binIdx` appears twice in the input array, `LiquidityLib` will read and act on `_positionBinShares[key]` and `_binTotalShares[binIdx]` for that bin twice within the same transaction. Depending on whether state is written back inside or outside the per-bin loop, this either:

- **Succeeds twice** (if state is written after the full loop), allowing the caller to withdraw double the token amount from that bin, or
- **Underflows / reverts on the second pass** (if state is written per-iteration with no underflow guard), causing a DoS for the caller but not a loss — however the first scenario is the dangerous one.

The `removeLiquidity` path is the critical case: a position owner passes `binIdxs = [N, N]` with `shares = [S, S]`. If `LiquidityLib` reads `_positionBinShares` at the start of each iteration without an intermediate write, both iterations see the same share balance, both compute the same token redemption amount, and both transfer tokens out of the pool. The pool's `binTotals.scaledToken0` / `scaledToken1` is decremented twice, but the actual ERC-20 balance is also drained twice, leaving other LPs unable to redeem their fair share.

This is the direct structural analog to the Redemptions bug:

> *"If a token address is included more than once, the sender will be paid out more than once, potentially earning many times more than their proportional share of the token."*

### Impact Explanation

A position owner can drain token reserves beyond their entitled share by passing duplicate bin indices in `removeLiquidity`. Other LPs' claims become undercollateralized, causing pool insolvency. The loss is bounded by the attacker's own share count per bin (they cannot withdraw more than `_positionBinShares[key]` per duplicate entry if state is written per-iteration), but if state is written after the loop the loss is unbounded relative to the bin's total balance.

### Likelihood Explanation

Any position owner can trigger this with a crafted `LiquidityDelta` calldata. No privileged role, no special token, no malicious setup required — only a valid existing position. The `removeLiquidity` entry point is public and gated only by `msg.sender == owner`. [5](#0-4) 

### Recommendation

Add a uniqueness check on `binIdxs` before processing. The simplest approach (mirroring the TokenRequest fix) is to require that each subsequent `binIdxs[i]` is strictly greater than `binIdxs[i-1]`, enforcing ascending order with no duplicates:

```solidity
for (uint256 i = 1; i < deltas.binIdxs.length; i++) {
    require(deltas.binIdxs[i] > deltas.binIdxs[i - 1], DuplicateOrUnsortedBinIdx());
}
```

Apply this check in both `addLiquidity` and `removeLiquidity` before delegating to `LiquidityLib`.

### Proof of Concept

1. LP adds liquidity to bin `0` with `shares = 1000` via a normal `addLiquidity` call.
2. LP calls `removeLiquidity` with `deltas.binIdxs = [0, 0]` and `deltas.shares = [1000, 1000]`.
3. `LiquidityLib.removeLiquidity` iterates: first pass processes bin `0`, computes token amounts for 1000 shares, transfers tokens to LP, (potentially) writes back state.
4. Second pass processes bin `0` again; if `_positionBinShares` was not yet zeroed, it again computes token amounts for 1000 shares and transfers a second time.
5. LP receives 2× their entitled tokens; `binTotals` is decremented by 2× the correct amount; remaining LPs cannot fully redeem.

---

**Note:** The exact double-withdrawal outcome depends on whether `LiquidityLib.removeLiquidity` writes `_positionBinShares` inside or outside its per-bin loop — a detail I was unable to confirm without reading `metric-core/contracts/libraries/LiquidityLib.sol` in full. If writes are per-iteration, the second pass reverts on underflow (no fund loss, but the missing uniqueness check remains a latent risk if the loop order ever changes). If writes are deferred, the double-withdrawal is fully exploitable. The absence of any uniqueness guard is the root cause regardless. [6](#0-5) [5](#0-4)
