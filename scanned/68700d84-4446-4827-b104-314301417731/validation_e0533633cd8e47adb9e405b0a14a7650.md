### Title
`protocolUnpausePool` Always Leaves Pool in Admin-Paused State Instead of Restoring Active State — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`protocolUnpausePool` is supposed to undo a protocol-level pause and restore the pool to active (`pauseLevel = 0`). Instead it always transitions to `pauseLevel = 1` (admin-paused), which is still a paused state. This mirrors the external report's pattern: a critical administrative function that can never complete its intended state transition due to a wrong constant embedded in the function body.

---

### Finding Description

The pool defines three pause levels:

```
0 = active
1 = paused by admin
2 = paused by protocol
``` [1](#0-0) 

The factory's `protocolPausePool` correctly transitions from level 0 or 1 → 2: [2](#0-1) 

But `protocolUnpausePool` transitions from level 2 → **1** (admin-paused), not → **0** (active): [3](#0-2) 

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    IMetricOmmPoolFactoryActions(pool).setPause(1);   // ← sets admin-paused, not active
}
```

After this call the pool is still paused. The only path to `pauseLevel = 0` is through `unpausePool`, which is restricted to `onlyPoolAdmin`: [4](#0-3) 

The protocol owner has no independent path to restore the pool to active state.

---

### Impact Explanation

`swap` is gated by `whenNotPaused`, which reverts whenever `pauseLevel != 0`: [5](#0-4) [6](#0-5) 

After `protocolUnpausePool` executes, `pauseLevel == 1`, so every swap call reverts. The pool is permanently swap-frozen unless the pool admin separately calls `unpausePool`. If the admin is unresponsive, malicious, or the admin key is lost, the pool is permanently bricked for trading. Protocol fees and LP spread income stop accruing. This is broken core pool functionality with direct loss of protocol revenue and user access.

---

### Likelihood Explanation

The protocol owner is expected to pause and unpause pools during incidents. Every time a pool that was **active (level 0)** is protocol-paused and then protocol-unpaused, the bug fires unconditionally — no special preconditions beyond normal administrative use. Likelihood is **High** given the function is on the normal operational path.

---

### Recommendation

Change the target pause level in `protocolUnpausePool` from `1` to `0`:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
-   IMetricOmmPoolFactoryActions(pool).setPause(1);
+   IMetricOmmPoolFactoryActions(pool).setPause(0);
}
```

If the design intent is a two-step unpause (protocol → admin → active), that must be explicitly documented and a separate `protocolFullUnpausePool` path must be provided so the protocol owner is never dependent on admin cooperation to restore swap functionality.

---

### Proof of Concept

1. Pool is deployed; `pauseLevel = 0` (active). Swaps work.
2. Protocol owner calls `protocolPausePool(pool)` → `pauseLevel = 2`. Swaps revert.
3. Protocol owner calls `protocolUnpausePool(pool)` → `setPause(1)` → `pauseLevel = 1`. Swaps **still revert** (`pauseLevel != 0`).
4. Protocol owner has no further lever — `unpausePool` is `onlyPoolAdmin`. If the admin does not act, every `swap` call reverts indefinitely with `PoolPaused`. [7](#0-6)
