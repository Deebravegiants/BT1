### Title
Precision Truncation in `viewSwapRsETHAmountAndFee` Allows Deposit With Zero rsETH Minted - (File: contracts/pools/RSETHPoolV3.sol)

### Summary
All L2 pool contracts compute `rsETHAmount` via integer division that can truncate to zero for small deposits. No post-calculation guard enforces `rsETHAmount > 0` before the deposit proceeds, so a depositor's ETH or ERC-20 tokens are consumed by the pool while they receive zero wrsETH/rsETH in return.

### Finding Description
`viewSwapRsETHAmountAndFee` in every L2 pool contract computes the rsETH output as:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;   // ETH path
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate; // token path
``` [1](#0-0) [2](#0-1) 

Solidity integer division truncates toward zero. When `amountAfterFee * 1e18 < rsETHToETHrate` (or the token-rate equivalent), the result is `0`. Because `rsETHToETHrate` is always `>= 1e18` (rsETH accrues yield, so its ETH price only grows), any deposit smaller than `rsETHToETHrate / 1e18` wei â€” i.e., smaller than 1 wei of rsETH â€” silently produces `rsETHAmount = 0`.

The `deposit` functions then proceed unconditionally:

```solidity
wrsETH.mint(msg.sender, rsETHAmount);   // mints 0
``` [3](#0-2) [4](#0-3) 

The same pattern is present in every pool variant:

- `RSETHPoolV3ExternalBridge.viewSwapRsETHAmountAndFee` / `deposit` [5](#0-4) [6](#0-5) 

- `RSETHPoolV3WithNativeChainBridge.viewSwapRsETHAmountAndFee` / `deposit` [7](#0-6) 

- `RSETHPoolNoWrapper.viewSwapRsETHAmountAndFee` / `deposit` [8](#0-7) [9](#0-8) 

- `RSETHPool.viewSwapRsETHAmountAndFee` / `deposit` [10](#0-9) [11](#0-10) 

- `RSETHPoolV2.viewSwapRsETHAmountAndFee` / `deposit` [12](#0-11) 

- `RSETHPoolV2ExternalBridge.viewSwapRsETHAmountAndFee` / `deposit` [13](#0-12) 

The same truncation exists in `LRTDepositPool.getRsETHAmountToMint`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [14](#0-13) 

The L1 pool does expose a `minRSETHAmountExpected` slippage guard in `_beforeDeposit`, but it only reverts when `rsethAmountToMint < minRSETHAmountExpected`. If the caller passes `0` as the minimum (the default for many integrations), a zero-mint deposit is accepted silently. [15](#0-14) 

### Impact Explanation
A depositor sends ETH or an ERC-20 token to any L2 pool and receives zero wrsETH/rsETH. The deposited value is absorbed into the pool's balance without being tracked as fees (`feeEarnedInETH` / `feeEarnedInToken` are only incremented by the fee portion, not the full deposit). The user permanently loses their deposited funds while the pool gains untracked ETH/tokens. This matches **Low â€” contract fails to deliver promised returns, but doesn't lose value** from the allowed impact scope.

### Likelihood Explanation
The condition is triggered whenever `deposit amount (wei) * 1e18 < rsETHToETHrate`. Since rsETH/ETH rate starts at `1e18` and only increases with yield accrual, any deposit of exactly `1 wei` already satisfies the condition today. The condition becomes easier to trigger as the rate grows. Any user who accidentally sends a dust amount, or any contract integration that does not enforce a minimum output, will silently lose funds. No special privileges or oracle manipulation are required.

### Recommendation
1. Add a post-calculation guard in every `deposit` function (or in `viewSwapRsETHAmountAndFee`) that reverts when `rsETHAmount == 0`:
   ```solidity
   if (rsETHAmount == 0) revert InvalidAmount();
   ```
2. For `LRTDepositPool`, enforce `minRSETHAmountExpected > 0` in `_beforeDeposit`, or add an explicit `rsethAmountToMint > 0` check independent of the caller-supplied minimum.
3. Document the minimum deposit threshold so integrators can enforce it off-chain.

### Proof of Concept
Assume `rsETHToETHrate = 1.001e18` (rsETH has accrued 0.1% yield â€” realistic after a few months).

1. Attacker/user calls `RSETHPoolV3.deposit{value: 1}("")` (1 wei of ETH).
2. `fee = 1 * feeBps / 10_000` â†’ rounds to `0` for any `feeBps < 10_000`.
3. `amountAfterFee = 1`.
4. `rsETHAmount = 1 * 1e18 / 1.001e18 = 0` (integer division truncates).
5. `wrsETH.mint(msg.sender, 0)` executes without revert.
6. The user's 1 wei is absorbed into the pool's ETH balance; the user holds no wrsETH.

Scaling: at `rsETHToETHrate = 1.1e18` (10% yield), any deposit below `1.1 wei` (i.e., `1 wei`) produces zero rsETH. The same arithmetic applies to ERC-20 token deposits via the token-rate path.
