### Title
Missing `whenNotPaused` on `_transfer` Allows rsETH Transfers When Contract Is Paused - (File: contracts/RSETH.sol)

---

### Summary

`RSETH.sol` inherits `PausableUpgradeable` and applies `whenNotPaused` to `mint` and `burnFrom`, but the overridden `_transfer` function has no pause check. All standard ERC20 transfer paths (`transfer`, `transferFrom`) route through `_transfer`, so rsETH holders can freely move tokens even when the contract is paused.

---

### Finding Description

`RSETH` is an ERC20 token with an explicit pause mechanism. The `mint` function is guarded: [1](#0-0) 

The `burnFrom` function is also guarded: [2](#0-1) 

However, the `_transfer` override â€” which is the single code path executed by every ERC20 `transfer` and `transferFrom` call â€” contains no `whenNotPaused` check: [3](#0-2) 

The contract inherits `PausableUpgradeable` and exposes `pause()` / `unpause()`: [4](#0-3) 

The intent of the pause is to halt all token activity. Because `_transfer` is not guarded, that intent is not fulfilled for the transfer path.

---

### Impact Explanation

When governance pauses RSETH â€” for example, in response to a price manipulation attack, an exploit in a connected contract, or a regulatory freeze â€” rsETH holders can still:

1. Call `transfer()` to move rsETH to any address, including fresh wallets outside the protocol's monitoring.
2. Call `transferFrom()` to move rsETH on behalf of approved addresses.
3. Sell rsETH on any DEX that holds a pool balance, continuing secondary-market activity the pause was meant to stop.

This directly undermines the `blockUserTransfers` / `recoverFrozenFunds` enforcement model: a user who is about to be blocked can race to transfer their balance to an unblocked address before the block is applied, because the pause that would freeze the whole system does not cover transfers.

**Impact: Medium â€” Temporary freezing of funds** (the pause mechanism fails to freeze rsETH transfers as intended, allowing continued movement of protocol tokens during an emergency).

---

### Likelihood Explanation

The `PAUSER_ROLE` exists precisely to halt all activity in emergencies. The missing modifier is a straightforward oversight consistent with the original Aave Lens finding class. Any holder can exploit this the moment a pause is active â€” no special privilege or setup is required.

**Likelihood: High** â€” any rsETH holder can call `transfer` or `transferFrom` at any time, including while the contract is paused.

---

### Recommendation

Add `whenNotPaused` to the `_transfer` override so that all ERC20 transfer paths are blocked when the contract is paused:

```solidity
function _transfer(address from, address to, uint256 amount)
    internal
    override
    whenNotPaused   // <-- add this
{
    _enforceNotBlocked(from);
    _enforceNotBlocked(to);
    super._transfer(from, to, amount);
}
``` [3](#0-2) 

---

### Proof of Concept

1. Governance detects an exploit and calls `RSETH.pause()`.
2. `mint` and `burnFrom` now revert with `EnforcedPause`.
3. Attacker (or any holder) calls `rsETH.transfer(attacker2, balance)`.
4. Execution enters `RSETH._transfer`, which only calls `_enforceNotBlocked` then `super._transfer` â€” no pause check.
5. Transfer succeeds. The attacker's rsETH is now in a fresh address, outside any block list, despite the contract being paused.
6. Governance's `recoverFrozenFunds` call on the original address now finds a zero balance. [5](#0-4) [3](#0-2)
