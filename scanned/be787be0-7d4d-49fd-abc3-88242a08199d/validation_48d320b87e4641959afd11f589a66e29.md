### Title
Dust Withdrawal Requests Cause Unbounded Gas in `unlockQueue`, Potentially Freezing the Withdrawal Queue - (File: contracts/LRTWithdrawalManager.sol)

---

### Summary

`minRsEthAmountToWithdraw` defaults to `0` for every asset in `LRTWithdrawalManager`. Because the only guard in `initiateWithdrawal` is `rsETHUnstaked == 0`, any unprivileged user can queue withdrawal requests of 1 wei of rsETH. The `_unlockWithdrawalRequests` loop processes requests in strict FIFO order and, for dust requests, computes a `payoutAmount` of 0 (integer division underflow), so the early-exit condition `availableAssetAmount < payoutAmount` is never triggered. An attacker who front-fills the queue with thousands of dust entries forces every `unlockQueue` call to iterate over all of them, potentially exhausting the block gas limit and temporarily freezing withdrawals for all legitimate users.

---

### Finding Description

`LRTWithdrawalManager` stores a per-asset minimum:

```solidity
mapping(address asset => uint256) public minRsEthAmountToWithdraw;
``` [1](#0-0) 

This mapping is never initialised in `initialize`; it defaults to `0` for every asset. The admin setter exists but is optional:

```solidity
function setMinRsEthAmountToWithdraw(address asset, uint256 minRsEthAmountToWithdraw_) external onlyLRTAdmin {
    minRsEthAmountToWithdraw[asset] = minRsEthAmountToWithdraw_;
``` [2](#0-1) 

The guard in `initiateWithdrawal` is:

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
``` [3](#0-2) 

When `minRsEthAmountToWithdraw[asset]` is `0`, the condition reduces to `rsETHUnstaked == 0`, so any amount ≥ 1 wei is accepted. Each such call pushes a new entry into the FIFO queue via `_addUserWithdrawalRequest`. [4](#0-3) 

The operator-only `unlockQueue` calls `_unlockWithdrawalRequests`, which iterates from `nextLockedNonce[asset]` up to `firstExcludedIndex`:

```solidity
while (nextLockedNonce_ < firstExcludedIndex) {
    ...
    uint256 payoutAmount = _calculatePayoutAmount(request, rsETHPrice, assetPrice);
    if (availableAssetAmount < payoutAmount) break;
    ...
    unchecked { nextLockedNonce_++; }
}
``` [5](#0-4) 

`_calculatePayoutAmount` for a 1-wei rsETH request computes:

```solidity
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
``` [6](#0-5) 

With `rsETHUnstaked = 1` and any realistic `rsETHPrice / assetPrice` ratio, integer division yields `currentReturn = 0`. Therefore `payoutAmount = 0`, and the break condition `availableAssetAmount < 0` is never true. The loop never exits early and must visit every dust entry up to `firstExcludedIndex`. With enough dust entries, a single call exhausts the block gas limit.

---

### Impact Explanation

If an attacker creates `N` dust withdrawal requests before legitimate users, the operator's `unlockQueue` call must iterate over all `N` entries before reaching any real request. Each iteration performs multiple storage reads/writes (SLOAD/SSTORE on `withdrawalRequests`, `assetsCommitted`, `unlockedWithdrawalsCount`). At a sufficient `N`, the call reverts due to out-of-gas, and no legitimate withdrawal can be unlocked until the dust entries are individually processed — which itself requires many separate `unlockQueue` calls, each costing the operator gas with zero economic return. This constitutes **temporary freezing of funds** (legitimate users cannot complete withdrawals) and **unbounded gas consumption** for the operator.

---

### Likelihood Explanation

The attack requires only rsETH tokens (obtainable by any depositor) and repeated calls to the public `initiateWithdrawal` function. No privileged access is needed. The cost to the attacker is the gas for each `initiateWithdrawal` call plus 1 wei of rsETH per request (which is locked but eventually returned as 0 assets, effectively burning it). The attack is cheap relative to the disruption caused.

---

### Recommendation

1. **Enforce a non-zero minimum at initialisation.** Set a sensible default for `minRsEthAmountToWithdraw` in `initialize` (e.g., `0.001 ether` of rsETH) for every supported asset, rather than relying on an optional admin call.

2. **Reject zero-payout requests at queue time.** In `initiateWithdrawal`, compute `expectedAssetAmount` and revert if it is zero:
   ```solidity
   uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
   if (expectedAssetAmount == 0) revert InvalidAmountToWithdraw();
   ```

3. **Add a per-call iteration cap** in `_unlockWithdrawalRequests` as a defence-in-depth measure to prevent a single `unlockQueue` call from consuming the entire block gas budget regardless of queue contents.

---

### Proof of Concept

1. Admin deploys `LRTWithdrawalManager` and never calls `setMinRsEthAmountToWithdraw` for ETH (default = 0).
2. Attacker acquires rsETH by depositing ETH into `LRTDepositPool`.
3. Attacker calls `initiateWithdrawal(ETH_TOKEN, 1, "")` in a loop (e.g., 50 000 times across multiple transactions). Each call costs ~50k gas and locks 1 wei of rsETH.
4. Legitimate users also call `initiateWithdrawal` with real amounts; their requests are appended after the dust entries.
5. Operator calls `unlockQueue(ETH_TOKEN, 50001, ...)`. The `_unlockWithdrawalRequests` loop iterates 50 000 times, each performing multiple SLOADs/SSTOREs. The call reverts with out-of-gas.
6. Operator reduces `firstExcludedIndex` to 1 000 and calls repeatedly; each call processes only dust entries (payout = 0) and costs ~3–5M gas with zero economic output.
7. Legitimate users' withdrawal requests remain locked until all dust entries are drained, effectively freezing their funds.
