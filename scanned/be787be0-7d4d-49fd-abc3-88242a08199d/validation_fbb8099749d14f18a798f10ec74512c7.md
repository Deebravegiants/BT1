### Title
Uninitialized `maxNumberOfWithdrawalsPerUser` Permanently Blocks All Withdrawals by Default - (`contracts/KERNEL/KernelDepositPool.sol`)

---

### Summary

`maxNumberOfWithdrawalsPerUser` defaults to `0` and is never set in `initialize()`. The guard in `initiateWithdrawal` evaluates `0 >= 0 → true` on the very first call, reverting with `WithdrawalLimitReached` for every user, permanently locking all staked KERNEL until an admin manually calls `setMaxNumberOfWithdrawalsPerUser`.

---

### Finding Description

`maxNumberOfWithdrawalsPerUser` is declared as a plain storage variable with no default: [1](#0-0) 

The `initialize` function sets `kernelToken`, `rewardsToken`, and roles, but never assigns `maxNumberOfWithdrawalsPerUser`: [2](#0-1) 

`initiateWithdrawal` checks:

```solidity
if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();
``` [3](#0-2) 

For a fresh user, `userWithdrawalIds[msg.sender].length == 0` and `maxNumberOfWithdrawalsPerUser == 0`, so `0 >= 0` is `true` and the call always reverts. No withdrawal can ever be initiated until the admin calls `setMaxNumberOfWithdrawalsPerUser`.

The setter enforces a non-zero value, confirming `0` is an invalid operational state: [4](#0-3) 

---

### Impact Explanation

Any user who calls `stake()` successfully transfers KERNEL tokens into the contract. Those tokens are immediately unrecoverable via `initiateWithdrawal` because the limit check always fails. There is no alternative withdrawal path. Funds remain locked until an admin intervenes. If the admin never calls `setMaxNumberOfWithdrawalsPerUser` (e.g., deployment script omits it, admin key is lost, or admin is unaware), the freeze is permanent.

**Impact:** Critical — Permanent freezing of all staked KERNEL user funds.

---

### Likelihood Explanation

The contract is deployed in a broken state by default. Any deployment that does not include an explicit post-`initialize` call to `setMaxNumberOfWithdrawalsPerUser` triggers this condition. This is a realistic omission in deployment scripts. Users can stake immediately after deployment (no admin action required to stake), but cannot withdraw until the admin separately configures the limit.

---

### Recommendation

Set a safe default for `maxNumberOfWithdrawalsPerUser` directly inside `initialize()`:

```solidity
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    // ... existing setup ...
    maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER; // e.g., 100
}
```

This ensures the contract is operational immediately after deployment without requiring a separate admin transaction.

---

### Proof of Concept

```solidity
// Deploy KernelDepositPool via proxy, call initialize() — do NOT call setMaxNumberOfWithdrawalsPerUser
// kernelToken.approve(pool, 1e18);
pool.stake(1e18);
// balanceOf[user] == 1e18, funds are in the contract

pool.initiateWithdrawal(1e18);
// ALWAYS reverts: WithdrawalLimitReached
// because userWithdrawalIds[user].length (0) >= maxNumberOfWithdrawalsPerUser (0) → true

// balanceOf[user] == 0 (decremented on stake), kernelToken still held by contract
// No recovery path exists for the user
```

The call sequence `stake(amount) → initiateWithdrawal(amount)` is entirely permissionless and reproducible on any local fork of the unmodified contract.
