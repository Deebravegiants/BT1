### Title
Unconditional Zero-Value Treasury Transfer in `claim()` Can Permanently Freeze Unclaimed Yield When `feeInBPS=0` and Token Reverts on Zero Transfers — (`contracts/utils/MerkleDistributor/MerkleDistributor.sol`)

### Summary
`MerkleDistributor.claim()` unconditionally calls `IERC20(token).safeTransfer(protocolTreasury, fee)` even when `fee = 0`. When `feeInBPS` is set to zero (a valid configuration) and the distributed token is a non-standard ERC20 that reverts on zero-value transfers, every `claim()` call reverts after the user transfer succeeds, permanently blocking all users from claiming their yield.

### Finding Description
In `claim()`, the fee is computed as `fee = (claimableAmount * feeInBPS) / 10_000`. When `feeInBPS = 0`, `fee = 0`. The subsequent `IERC20(token).safeTransfer(protocolTreasury, fee)` at line 144 is called unconditionally with no `if (fee > 0)` guard. [3](#0-2) 

`setFeeInBPS` explicitly permits zero: [4](#0-3) 

If the token reverts on `transfer(addr, 0)`, the entire transaction reverts. Because the state update (`userClaims[account]`) occurs before the transfers at lines 134–135, the state is also rolled back, leaving the user's claim permanently unprocessable. [5](#0-4) 

### Impact Explanation
**Medium. Permanent freezing of unclaimed yield.** All users with valid Merkle proofs are unable to claim. Tokens remain locked in the distributor contract indefinitely. No admin recovery path exists in the contract (no `emergencyWithdraw` or similar function).

### Likelihood Explanation
Low. Requires the owner to both (a) set `feeInBPS = 0` and (b) configure a token that reverts on zero-value transfers. These are both owner-controlled parameters, making this a self-inflicted misconfiguration scenario rather than an externally exploitable path. The combination is unlikely in a well-operated deployment but is not prevented by any on-chain guard.

### Recommendation
Add a zero-fee guard before the treasury transfer:

```solidity
if (fee > 0) {
    IERC20(token).safeTransfer(protocolTreasury, fee);
}
```

This is the standard pattern and eliminates the issue entirely regardless of token behavior.

### Proof of Concept
1. Deploy `MerkleDistributor` with a mock ERC20 that reverts on `transfer(addr, 0)`.
2. Call `setFeeInBPS(0)`.
3. Set a valid Merkle root and fund the distributor.
4. Call `claim(index, account, cumulativeAmount, proof)` with a valid proof.
5. Observe: step (3) `safeTransfer(account, claimableAmount)` succeeds; step (4) `safeTransfer(protocolTreasury, 0)` reverts; entire transaction reverts.
6. Assert: `isClaimed(index, account)` returns `false`; user balance unchanged; tokens frozen in distributor.
