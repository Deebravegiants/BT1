### Title
Out-of-Order CCIP Execution Combined with Immutable `i_maxSupply` Can Permanently Freeze Bridged rsETH — (`contracts/ccip/WrappedRSETH.sol`, `contracts/L1VaultV2.sol`)

### Summary

`L1VaultV2.getCCIPMessage` unconditionally sets `allowOutOfOrderExecution: true`, permitting the CCIP DON (and any permissionless executor) to deliver messages in any sequence. `WrappedRSETH.i_maxSupply` is an immutable constructor parameter. If two in-flight CCIP bridge messages are delivered in reverse order and the combined mint amount exceeds the remaining cap, the earlier (larger) message permanently fails with `MaxSupplyExceeded`. Because `i_maxSupply` cannot be raised after deployment, the failed message can never be retried successfully, and the rsETH burned on L1 for that message is permanently lost.

### Finding Description

**Step 1 — L1 side (bridge initiation):**
The MANAGER calls `bridgeRsETHToL2UsingCCIP(100)` (M1) and then `bridgeRsETHToL2UsingCCIP(50)` (M2). For each call, the CCIP router burns the corresponding rsETH on L1 and commits a cross-chain message. [4](#0-3) 

**Step 2 — CCIP message construction:**
Every message is built with `allowOutOfOrderExecution: true`, explicitly waiving the sequential delivery guarantee. [5](#0-4) 

**Step 3 — L2 side (out-of-order execution):**
With `allowOutOfOrderExecution: true`, the CCIP OffRamp (or any permissionless executor) may execute M2 before M1. An attacker can accelerate this by block-stuffing L2 to delay M1's inclusion while submitting M2's execution transaction directly. When M2 executes first, `totalSupply` increases by 50.

**Step 4 — M1 permanently fails:**
When M1 subsequently attempts to mint 100 tokens, the check `totalSupply() + 100 > i_maxSupply` reverts with `MaxSupplyExceeded`. Because `i_maxSupply` is immutable, no admin action can raise the cap to allow a retry. [6](#0-5) 

**Step 5 — Permanent fund loss:**
The 100 rsETH burned on L1 for M1 is irrecoverable. There is no refund path in `L1VaultV2` and no mechanism to re-mint the burned rsETH.

### Impact Explanation

**Critical — Permanent freezing of funds.** The rsETH committed to the CCIP bridge for M1 is burned on L1 at send time. If the corresponding L2 mint permanently fails (due to an immutable cap that cannot be raised), those funds are frozen forever. This directly matches the "Critical. Permanent freezing of funds" impact category.

### Likelihood Explanation

**Low.** Three conditions must coincide:
1. `i_maxSupply` must be deployed as a non-zero value close to the current `totalSupply`.
2. Two or more CCIP bridge messages must be in-flight simultaneously with a combined mint amount that exceeds the remaining headroom.
3. The messages must be executed in reverse order (naturally possible due to `allowOutOfOrderExecution: true`, or forced via block stuffing / direct OffRamp calls).

The MANAGER controls bridge call timing, so condition 2 is operationally plausible during normal operations. Condition 3 requires no special privilege because `allowOutOfOrderExecution: true` is already set.

### Recommendation

1. **Remove or conditionally set `allowOutOfOrderExecution`:** Set `allowOutOfOrderExecution: false` unless a specific L2 chain mandates otherwise. This restores sequential delivery guarantees and eliminates the ordering risk entirely.
2. **Remove `i_maxSupply` or make it mutable (owner-only):** An immutable cap combined with a permissionless mint path is inherently fragile. If a cap is required, it should be adjustable by a privileged role so that stuck messages can be unblocked.
3. **Track in-flight amounts:** Maintain an on-chain counter of committed-but-undelivered CCIP amounts and reject new bridge calls that would push `totalSupply + pendingAmount` above `i_maxSupply`.
