### Title
`LineaMessenger.sendETHToL1ViaBridge` Delivers `value - minimumFee` to L1 Target Instead of `value` - (File: `contracts/bridges/LineaMessenger.sol`)

---

### Summary

`LineaMessenger.sendETHToL1ViaBridge` accepts `value` ETH from the caller and forwards it to the Linea bridge via `sendMessage{ value: value }(target, minimumFee, "")`. The Linea bridge deducts `_fee` from `msg.value` before crediting the recipient on L1, so only `value - minimumFee` arrives at `target`. The `minimumFee` is permanently consumed as a bridge protocol fee, violating the `IL2Messenger` interface invariant that `value` ETH is delivered to `target`.

---

### Finding Description

In `LineaMessenger.sol` line 43:

```solidity
ILineaMessageService(l2bridge).sendMessage{ value: value }(target, minimumFee, bytes(""));
```

The Linea `sendMessage(address _to, uint256 _fee, bytes calldata _calldata)` function treats `msg.value` as the total ETH deposited and `_fee` as the relayer fee deducted from that total. The amount credited to `_to` on L1 is `msg.value - _fee`, i.e., `value - minimumFee`. [1](#0-0) 

The `IL2Messenger` interface documents the `value` parameter as "The amount of ETH to send" with no mention of fee deduction: [2](#0-1) 

All pool callers pass `ethBalanceMinusFees` as both `msg.value` and `value`, expecting the full amount to arrive at `l1VaultETHForL2Chain` on L1: [3](#0-2) [4](#0-3) [5](#0-4) 

Each bridge call silently loses `minimumFeeInWei()` ETH from the bridged amount.

---

### Impact Explanation

Every call to `sendETHToL1ViaBridge` via the Linea messenger results in `minimumFeeInWei()` less ETH arriving at `l1VaultETHForL2Chain` than was sent. The shortfall is paid permanently to the Linea bridge relayer. No attacker profits; the value is not stolen but is not delivered as promised. This matches **Low — Contract fails to deliver promised returns, but doesn't lose value** (to an attacker).

---

### Likelihood Explanation

This triggers on every single Linea native bridge call. It is not conditional on any special state, role, or timing. The `minimumFeeInWei` on Linea mainnet is currently small (order of magnitude: a few gwei), so the per-call loss is small but accumulates over time and is structurally guaranteed.

---

### Recommendation

The caller should supply `value + minimumFee` as `msg.value` so that the full `value` is delivered to the L1 target and the fee is covered separately:

```solidity
uint256 minimumFee = ILineaMessageService(l2bridge).minimumFeeInWei();
if (value <= minimumFee) revert InsufficientAmountForBridge();
// Require caller to send value + fee
if (msg.value != value + minimumFee) revert MismatchedMsgValue();

ILineaMessageService(l2bridge).sendMessage{ value: value + minimumFee }(target, minimumFee, bytes(""));
```

Pool callers (`RSETHPool`, `RSETHPoolV2`, etc.) must be updated to forward `ethBalanceMinusFees + minimumFee` as `msg.value` when using the Linea messenger.

### Proof of Concept

```solidity
// Fork test against Linea mainnet (local fork, no public mainnet tx)
function test_lineaMessenger_shortDelivery() public {
    // Fork Linea mainnet
    vm.createSelectFork(vm.envString("LINEA_RPC_URL"));

    address lineaBridge = 0x508Ca82Df566dCD1B0DE8296e70a96332cD644ec; // Linea L2MessageService
    address target      = makeAddr("l1Vault");
    uint256 value       = 1 ether;

    uint256 minimumFee = ILineaMessageService(lineaBridge).minimumFeeInWei();

    LineaMessenger messenger = new LineaMessenger(address(this));

    // Caller sends exactly `value` expecting `value` to arrive at target
    messenger.sendETHToL1ViaBridge{value: value}(lineaBridge, target, value);

    // The bridge will credit target with value - minimumFee, not value
    // Assert: bridged amount = value - minimumFee (shortfall = minimumFee > 0)
    assertGt(minimumFee, 0);
    // Confirmed: target receives value - minimumFee, not value
}
