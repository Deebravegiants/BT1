### Title
Exact-Output Swap Silently Delivers Less Token0 Than Requested When Pool Liquidity Is Insufficient — (`metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`_swapToken1ForToken0SpecifiedOutput` silently clamps the requested exact-output amount to the pool's total available token0 balance instead of reverting. A user who specifies an exact output of token0 that exceeds pool liquidity receives less than requested, yet the swap succeeds and the pool emits no error. This is the direct Metric OMM analog of the MochiVault M-11 bug: a "cap instead of revert" on a user-specified exact amount.

---

### Finding Description

In `_swapToken1ForToken0SpecifiedOutput`, before any bin iteration begins, the function reads `binTotals.scaledToken0` and silently overwrites the caller-supplied `amountOutScaled` if it exceeds the total:

```solidity
// MetricOmmPool.sol lines 866–876
function _swapToken1ForToken0SpecifiedOutput(uint256 amountOutScaled, SwapMath.InternalSwapParams memory params)
    internal
    returns (uint256, uint256, uint256, uint256)
{
    unchecked {
        {
            uint256 totalAvailableToken0Scaled = binTotals.scaledToken0;
            if (amountOutScaled > totalAvailableToken0Scaled) {
                amountOutScaled = totalAvailableToken0Scaled;   // ← silent cap, no revert
            }
        }
``` [1](#0-0) 

The capped value propagates back through `_executeSwap`:

```solidity
// MetricOmmPool.sol lines 716–722
uint256 amountOutScaled = TOKEN_0_SCALE_MULTIPLIER * uint256(-amountSpecified);
uint256 amountInScaled;
(amountInScaled, amountOutScaled, protocolFeeScaled, feeExclusiveInputScaled) =
    _swapToken1ForToken0SpecifiedOutput(amountOutScaled, params);
amount0DeltaScaled = -int256(amountOutScaled);   // now < |amountSpecified|
amount1DeltaScaled =  int256(amountInScaled);
``` [2](#0-1) 

The outer `swap` function then transfers only the capped amount to `recipient`:

```solidity
// MetricOmmPool.sol lines 265–268
if (amount0Delta < 0) {
    transferToken0(recipient, uint256(-amount0Delta));   // less than requested
}
``` [3](#0-2) 

There is no post-swap assertion that `|amount0Delta| == |amountSpecified|` for exact-output swaps. The pool returns `(amount0Delta, amount1Delta)` where `|amount0Delta| < |amountSpecified|` and the transaction succeeds.

---

### Impact Explanation

**Broken exact-output guarantee / partial fill without revert.**

A caller who issues an exact-output swap (`amountSpecified < 0`) expecting to receive a precise quantity of token0 will silently receive less. Any downstream logic that relies on the exact amount (e.g., a multi-hop router, a limit-order settlement, or a liquidation bot that needs a specific token0 quantity) will operate on an incorrect balance. Because the pool does not revert, the caller's slippage guard at the router layer is the only protection — and callers who interact with the pool directly (or whose router checks `amountOut >= minOut` against the returned delta rather than the requested amount) have no protection at all.

The financial loss is: the user's token1 input is consumed for a swap that delivers less token0 than contracted. The user cannot distinguish this from a normal partial fill caused by a price limit, so they may not detect the shortfall.

**Severity: Medium** — direct loss of swap output relative to the user's specified intent; no privileged actor required; reachable whenever pool liquidity is below the requested exact-output amount.

---

### Likelihood Explanation

Any unprivileged caller can trigger this by calling `swap` with `zeroForOne = false` and `amountSpecified < 0` (exact output of token0) where `|amountSpecified| * TOKEN_0_SCALE_MULTIPLIER > binTotals.scaledToken0`. This is a normal market condition (thin liquidity, large order, or a pool that has been partially drained). No malicious setup is required.

---

### Recommendation

Replace the silent cap with an explicit revert when the requested exact-output exceeds available liquidity:

```solidity
uint256 totalAvailableToken0Scaled = binTotals.scaledToken0;
require(amountOutScaled <= totalAvailableToken0Scaled, InsufficientLiquidity());
```

Alternatively, if partial fills are intentional for exact-output swaps, the interface documentation and the `swap` return values must clearly communicate that `|amount0Delta|` may be less than `|amountSpecified|`, and callers must be required to check the returned delta against their minimum acceptable output — enforced at the pool level, not left to the caller.

Also audit `_swapToken0ForToken1SpecifiedOutput` for the symmetric pattern (silent cap on token1 output).

---

### Proof of Concept

1. Deploy a pool with a small token0 balance, e.g. `binTotals.scaledToken0 = 1000`.
2. Call `pool.swap(recipient, false, -5000, type(uint128).max, callbackData, "")` — requesting 5000 token0 out (exact output).
3. Observe: the pool does **not** revert. `_swapToken1ForToken0SpecifiedOutput` silently sets `amountOutScaled = 1000`.
4. `recipient` receives only 1000 token0 (scaled), not 5000.
5. The swap callback is invoked and the caller pays token1 for only 1000 token0 worth of output.
6. The `swap` function returns `amount0Delta = -1000` (not `-5000`), succeeding silently.

The caller requested an exact output of 5000 and received 1000 — a 4000-unit shortfall — with no on-chain error.
