### Title
Payable Swap and Liquidity Functions Accept ETH on Non-WETH Code Paths, Enabling Theft via `refundETH()` — (`metric-periphery/contracts/MetricOmmSimpleRouter.sol`, `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

---

### Summary

`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput` (in `MetricOmmSimpleRouter`) and `addLiquidityExactShares`, `addLiquidityWeighted` (in `MetricOmmPoolLiquidityAdder`) are all declared `payable`. When the input token is not WETH, the internal `pay()` helper takes the ERC-20 `safeTransferFrom` branch and never touches `address(this).balance`. Any ETH attached to the call is silently retained by the contract. Because `refundETH()` unconditionally forwards the entire contract ETH balance to `msg.sender`, a third party can immediately call it and steal the stranded ETH.

---

### Finding Description

`PeripheryPayments.pay()` branches on whether the token is WETH:

```solidity
// metric-periphery/contracts/base/PeripheryPayments.sol  lines 69-88
function pay(address token, address payer, address recipient, uint256 value) internal {
    if (payer == address(this)) {
        IERC20(token).safeTransfer(recipient, value);
    } else if (token == WETH) {          // ← only branch that consumes msg.value
        uint256 nativeBalance = address(this).balance;
        ...
    } else {
        IERC20(token).safeTransferFrom(payer, recipient, value);   // ← ETH ignored
    }
}
```

When `token != WETH`, the `else` branch executes a plain ERC-20 pull and `address(this).balance` is never touched. All four router swap functions and both liquidity-adder entry points are `payable` and call `pay()` through the callback, so any `msg.value` attached to a non-WETH call accumulates in the contract.

`refundETH()` then sends the full balance to whoever calls it next:

```solidity
// metric-periphery/contracts/base/PeripheryPayments.sol  lines 58-63
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);   // ← any caller, not the original payer
    }
}
```

There is no access control, no per-user accounting, and no automatic refund at the end of the swap functions themselves. The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only blocks direct ETH sends; it does not prevent ETH from entering via `msg.value` on `payable` functions.

The interface NatSpec even documents the contradiction:

> "Scope: ERC-20 routes only. No native ETH, WETH wrap/unwrap, on-chain quotes, sweep, or refund helpers."

yet all four swap functions carry the `payable` modifier with no enforcement.

---

### Impact Explanation

A user who calls `exactInputSingle` (or any of the other three swap functions, or either liquidity-adder entry point) with `tokenIn != WETH` and `msg.value > 0` loses that ETH permanently from their perspective. Any address can call `refundETH()` in a subsequent transaction and receive the full stranded balance. This is a direct, irreversible loss of user principal with no recovery path for the victim.

The same theft vector applies to WETH swaps where the user over-sends ETH: `pay()` wraps exactly `value` and leaves the surplus in the contract, which is again claimable by anyone.

---

### Likelihood Explanation

- All four swap functions and both liquidity-adder functions are `payable`, so wallets, aggregators, and scripts that forward ETH will silently succeed.
- The `multicall` wrapper is also `payable`, meaning ETH can arrive through batched calls where one sub-call is non-WETH.
- MEV bots routinely monitor contract ETH balances and can front-run or back-run `refundETH()` calls.
- User error (sending ETH with a non-WETH swap) is a well-documented class of mistake; the PuttyV2 audit found it submitted by 25+ independent researchers.

---

### Recommendation

Add a `msg.value == 0` guard in each code path where ETH is not consumed. The minimal fix mirrors the PuttyV2 mitigation:

```solidity
// In exactInputSingle / exactInput / exactOutputSingle / exactOutput
// before the swap call, when tokenIn != WETH:
if (params.tokenIn != WETH && msg.value != 0) revert UnexpectedETH();
```

Alternatively, remove the `payable` modifier from functions that have no legitimate ETH-consuming path, or add an automatic `refundETH()` call at the end of each swap function so that any surplus is returned to `msg.sender` atomically within the same transaction.

---

### Proof of Concept

1. Pool exists for `USDC/DAI` (neither token is WETH).
2. Alice calls `exactInputSingle({tokenIn: USDC, ...})` with `msg.value = 1 ether` (user error or wallet bug).
3. Inside `_justPayCallback`, `pay(USDC, Alice, pool, amount)` executes the `else` branch — `safeTransferFrom` — and never reads `address(this).balance`. The 1 ETH remains in `MetricOmmSimpleRouter`.
4. Bob observes the contract balance on-chain and calls `refundETH()`.
5. `refundETH()` transfers `address(this).balance` (1 ETH) to Bob. Alice's ETH is gone with no recourse.

```text
  function pay(address token, address payer, address recipient, uint256 value) internal {
    // If the payer is contract it means we are in the middle of a path. In the middle of a path we operate on ERC20 only.
    if (payer == address(this)) {
      IERC20(token).safeTransfer(recipient, value);
    } else if (token == WETH) {
      uint256 nativeBalance = address(this).balance;
      if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
      } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
      } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
      }
    } else {
      IERC20(token).safeTransferFrom(payer, recipient, value);
    }
  }
```
