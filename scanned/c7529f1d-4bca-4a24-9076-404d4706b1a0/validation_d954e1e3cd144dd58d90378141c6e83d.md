### Title
Missing Sequencer Uptime / Grace-Period Check in L2 Price Providers Allows Stale-Price Swap Execution After Sequencer Restart - (`smart-contracts-poc/contracts/PriceProviderL2.sol` and `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

---

### Summary

`PriceProviderL2` and `ProtectedPriceProviderL2` are the L2-specific price providers that feed bid/ask quotes into every `MetricOmmPool` swap on Arbitrum, Base, Optimism, and other L2 networks. Both contracts perform a staleness check (`_isStale`) against `MAX_TIME_DELTA`, but neither checks the Chainlink sequencer uptime feed before consuming oracle data. When the L2 sequencer goes offline and then restarts, oracle data that was stored before the outage can still be within `MAX_TIME_DELTA` and will pass the staleness check, allowing swaps to execute against prices that are economically stale relative to the real market.

---

### Finding Description

Both L2 providers share the same `_isStale` guard:

```solidity
// PriceProviderL2.sol lines 135-150
function _isStale(
    uint256 refTime,
    uint256 nowTs,
    uint256 maxDelta,
    uint256 futureTol
) internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) {
        return (refTime - nowTs) > futureTol;
    }
    return (nowTs - refTime) > maxDelta;
}
```

`MAX_TIME_DELTA` is set at construction and is bounded only by `(0, 7 days]`:

```solidity
// PriceProviderL2.sol line 92
if (_maxTimeDelta == 0 || _maxTimeDelta > 7 days) revert MaxTimeDeltaOutOfBounds();
```

Oracle data is pushed on-chain via `ChainlinkOracle.updateReport()`. When the L2 sequencer goes offline, no new reports can be pushed. When the sequencer restarts, the last stored report's `refTime` may be, for example, 20 minutes old. If `MAX_TIME_DELTA` is set to 1 hour (a common production value), the staleness check passes and the 20-minute-old price is served to the pool as a live quote.

A `grep` across all `.sol` files in the repository for `sequencerUptimeFeed`, `GRACE_PERIOD`, `_checkSequencer`, and `latestRoundData` returns **zero matches**. The `ChainlinkVerifierL2` artifact visible in `registry.json` (which does expose `sequencerUptimeFeed` and `GRACE_PERIOD`) has no corresponding source file in the repository, so it cannot be verified to protect the price-provider read path.

The Chainlink-recommended pattern requires:
1. Calling `sequencerUptimeFeed.latestRoundData()` and asserting `answer == 0` (sequencer is up).
2. Asserting `block.timestamp - startedAt >= GRACE_PERIOD` (e.g., 3600 s) to reject prices published in the window immediately after restart, when the market may have moved significantly during the outage.

Neither step is present in `PriceProviderL2._getBidAndAskPrice()` or `ProtectedPriceProviderL2._getBidAndAskPrice()`.

---

### Impact Explanation

A swap on `MetricOmmPool` calls `_getBidAndAskPriceX64`, which calls `provider.getBidAndAskPrice()`. If the provider returns a pre-outage price that is, say, 5% below the current market (because the market moved during the sequencer downtime), a trader can:

- Execute a `zeroForOne` swap at the stale low ask, receiving token1 at a price that is 5% cheaper than the real market.
- The pool's LPs bear the loss: they deliver token1 at the stale price and receive token0 at the stale price, with no recourse.

This is a direct, quantifiable loss of LP principal proportional to the price movement during the outage and the swap size. It satisfies the "bad-price execution: stale bid/ask quote reaches a pool swap" criterion.

---

### Likelihood Explanation

L2 sequencer outages are documented historical events (Arbitrum, Optimism, Base have each experienced them). The attack requires no privileged access: any public trader can call `swap` immediately after the sequencer restarts, before fresh oracle data is pushed. The window is bounded by how quickly off-chain keepers push a new report, which is typically seconds to minutes — but the attacker can front-run the keeper update in the first block after restart.

---

### Recommendation

Add a sequencer uptime check to both `PriceProviderL2._getBidAndAskPrice()` and `ProtectedPriceProviderL2._getBidAndAskPrice()` (or to a shared base):

```solidity
// Store as immutable at construction (address(0) on L1 = skip check)
AggregatorV3Interface public immutable sequencerUptimeFeed;
uint256 public constant GRACE_PERIOD = 3600; // 1 hour

function _checkSequencer() internal view {
    if (address(sequencerUptimeFeed) == address(0)) return;
    (, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
    // answer == 0: sequencer is up; answer == 1: sequencer is down
    if (answer != 0) revert SequencerDown();
    if (block.timestamp - startedAt < GRACE_PERIOD) revert GracePeriodNotOver();
}
```

Call `_checkSequencer()` at the top of `_getBidAndAskPrice()` before reading oracle data. Return the `(0, type(uint128).max)` sentinel (which propagates to `FeedStalled`) if the check fails.

---

### Proof of Concept

1. Deploy `PriceProviderL2` on Arbitrum with `MAX_TIME_DELTA = 1 hours`.
2. Push a fresh oracle report at `t = 0` (price = 1000 USDC/ETH).
3. Simulate sequencer downtime from `t = 0` to `t = 30 minutes` (no new reports can be pushed; real market moves to 1050 USDC/ETH).
4. At `t = 30 minutes` the sequencer restarts. The stored report has `refTime = 0`, `nowTs = 1800`. `_isStale(0, 1800, 3600, futureTol)` → `(1800 - 0) = 1800 <= 3600` → **not stale**. The stale price 1000 is served.
5. Attacker calls `MetricOmmPool.swap` (zeroForOne, exactIn) and buys ETH at 1000 USDC instead of 1050, extracting 50 USDC per ETH from LP reserves.
6. A keeper pushes the fresh report at `t = 30 minutes + a few seconds`, but the attacker's transaction is already included in the first post-restart block. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
