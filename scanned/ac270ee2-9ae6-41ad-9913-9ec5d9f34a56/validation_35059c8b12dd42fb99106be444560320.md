### Title
Raw Token Balance Check in `_checkResidueLSTBalance` Enables Griefing to Permanently Block NodeDelegator Removal - (`contracts/LRTDepositPool.sol`)

---

### Summary

`_checkResidueLSTBalance` in `LRTDepositPool.sol` uses a raw `IERC20.balanceOf(nodeDelegatorAddress)` call to determine whether a NodeDelegator holds LST funds before allowing its removal from the queue. Because any unprivileged actor can transfer LST tokens directly to a NodeDelegator address, a griefer can permanently prevent `removeNodeDelegatorContractFromQueue` from succeeding, blocking the protocol's ability to retire or replace NodeDelegators.

---

### Finding Description

`_removeNodeDelegatorContractFromQueue` is the internal function that gates NodeDelegator removal. It calls `_checkResidueLSTBalance`, which computes the asset balance as:

```solidity
assetBalance = IERC20(supportedAssets[i]).balanceOf(nodeDelegatorAddress)
    + INodeDelegator(nodeDelegatorAddress).getAssetBalance(supportedAssets[i]);
assetBalance += INodeDelegator(nodeDelegatorAddress).getAssetUnstaking(supportedAssets[i]);

if (assetBalance > maxNegligibleAmount) {
    revert NodeDelegatorHasAssetBalance(supportedAssets[i], assetBalance);
}
``` [1](#0-0) 

The first term, `IERC20(supportedAssets[i]).balanceOf(nodeDelegatorAddress)`, is a raw on-chain token balance that reflects **any** tokens held by the NodeDelegator address, including tokens sent to it externally without going through any protocol function. [2](#0-1) 

This is in contrast to `getAssetBalance` (EigenLayer shares) and `getAssetUnstaking` (queued withdrawals), which are both derived from EigenLayer's internal accounting and cannot be inflated by an external transfer. [3](#0-2) 

The `_removeNodeDelegatorContractFromQueue` function calls both checks before performing the removal: [4](#0-3) 

```solidity
function _removeNodeDelegatorContractFromQueue(address nodeDelegatorAddress) internal {
    uint256 ndcIndex = _getNDCIndex(nodeDelegatorAddress);
    _checkResidueEthBalance(nodeDelegatorAddress);
    _checkResidueLSTBalance(nodeDelegatorAddress);   // <-- griefable
    ...
}
```

An attacker who wants to prevent a NodeDelegator from being removed simply sends any amount of a supported LST (e.g., stETH, cbETH) exceeding `maxNegligibleAmount` directly to the NodeDelegator address. The removal reverts with `NodeDelegatorHasAssetBalance`. If the operator attempts to clear the balance (e.g., by depositing the tokens into EigenLayer via `depositAssetIntoStrategy`), the raw `balanceOf` decreases but `getAssetBalance` increases by the same amount — the sum remains above the threshold. The attacker can also simply re-send tokens after any cleanup, repeating the griefing indefinitely at minimal cost.

---

### Impact Explanation

**Medium — Temporary (but indefinitely repeatable) freezing of protocol management capability.**

The protocol cannot retire, replace, or remove a NodeDelegator while the griefing persists. If a NodeDelegator is compromised, misconfigured, or needs to be rotated, the removal path is blocked. This does not directly freeze user funds in the short term, but it prevents the protocol from responding to operational or security incidents involving a specific NodeDelegator, which can escalate to fund loss if the NodeDelegator itself is compromised and cannot be removed.

---

### Likelihood Explanation

**High.** Any externally reachable actor can call `IERC20.transfer(nodeDelegatorAddress, amount)` for any supported LST. No special role, permission, or protocol interaction is required. The cost is only the token amount sent (which is not lost — it remains in the NodeDelegator — but is effectively locked there as long as the griefing continues). The attacker can repeat the attack after every operator cleanup attempt.

---

### Recommendation

Replace the raw `balanceOf` term with an internal accounting variable that is updated only through protocol-controlled deposit and withdrawal flows. Specifically, track the LST balance held in each NodeDelegator using an internal mapping (e.g., `internalLSTBalance[ndc][asset]`) that is incremented on `transferAssetToNodeDelegator` and decremented on `depositAssetIntoStrategy` or `transferBackToLRTDepositPool`. Use this internal balance instead of `IERC20.balanceOf(nodeDelegatorAddress)` in `_checkResidueLSTBalance`.

Alternatively, add a recovery function that allows the protocol to sweep unexpected (externally sent) token balances from a NodeDelegator to the deposit pool without requiring the NodeDelegator to first be removed from the queue.

---

### Proof of Concept

1. Protocol admin initiates removal of `NodeDelegatorA` (which legitimately holds zero LST balance in EigenLayer and zero raw balance).
2. Attacker front-runs (or simply calls before the admin) `stETH.transfer(NodeDelegatorA, maxNegligibleAmount + 1)`.
3. `_checkResidueLSTBalance` computes `balanceOf(NodeDelegatorA) = maxNegligibleAmount + 1`, which exceeds the threshold.
4. `_removeNodeDelegatorContractFromQueue` reverts with `NodeDelegatorHasAssetBalance`.
5. Operator deposits the stETH into EigenLayer via `depositAssetIntoStrategy`: `balanceOf` drops to 0, but `getAssetBalance` rises by the same amount — total unchanged, still reverts.
6. Operator initiates unstaking from EigenLayer and waits the withdrawal delay. After completion, `getAssetBalance` and `getAssetUnstaking` both drop to 0, and the raw balance is 0.
7. Attacker immediately sends `maxNegligibleAmount + 1` stETH again. Step 4 repeats.
8. The NodeDelegator can never be removed as long as the attacker is willing to spend tokens. [5](#0-4) [4](#0-3)

    /// @dev reverts if NDC has LST balance
    function _checkResidueLSTBalance(address nodeDelegatorAddress) internal view {
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetsLength = supportedAssets.length;

        uint256 assetBalance;
        for (uint256 i; i < supportedAssetsLength; ++i) {
            if (supportedAssets[i] == LRTConstants.ETH_TOKEN) {
                // this function only checks for residual LST balance
                continue;
            }

            assetBalance = IERC20(supportedAssets[i]).balanceOf(nodeDelegatorAddress)
                + INodeDelegator(nodeDelegatorAddress).getAssetBalance(supportedAssets[i]);
            assetBalance += INodeDelegator(nodeDelegatorAddress).getAssetUnstaking(supportedAssets[i]);

            if (assetBalance > maxNegligibleAmount) {
                revert NodeDelegatorHasAssetBalance(supportedAssets[i], assetBalance);
            }
        }
