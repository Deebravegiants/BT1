### Title
`withinUnstakeLimits` Compares stETH Token Amounts Against ETH-Denominated Commitments Without Price Conversion - (`contracts/LRTConverter.sol`)

---

### Summary

The `withinUnstakeLimits` modifier in `LRTConverter.sol` enforces an unstaking budget by comparing `amountToUnstake` (denominated in stETH tokens) directly against `assetsCommitted[ETH_TOKEN]` (denominated in ETH) without any price conversion. When stETH/ETH ‚âÝ 1.0, the limit check is arithmetically incorrect, allowing the operator to queue more ETH value into Lido's withdrawal queue than the protocol's user withdrawal obligations justify.

---

### Finding Description

`unstakeStEth` is gated by the `withinUnstakeLimits(amountToUnstake)` modifier: [1](#0-0) 

The modifier fetches the ETH-denominated pending withdrawal budget: [2](#0-1) 

`assetsCommitted(ETH_TOKEN)` is populated in `LRTWithdrawalManager.initiateWithdrawal` as an ETH amount (the expected ETH payout to users): [3](#0-2) 

The guard at line 65 is:

```solidity
if (amountToUnstake > whitelistedUnstakeAllowance + availableActiveETHWithdrawals) {
    revert UnstakeLimitExceeded();
}
```

`amountToUnstake` is in **stETH token units**; `availableActiveETHWithdrawals` is in **ETH units**. No oracle call or price conversion is performed anywhere in the modifier. The two quantities are only numerically equivalent when stETH/ETH = 1.0 exactly. [4](#0-3) 

---

### Impact Explanation

When stETH/ETH > 1.0 (e.g., 1.05):

- `assetsCommitted[ETH_TOKEN]` = 100e18 (100 ETH owed to users)
- Operator calls `unstakeStEth(100e18)` ‚Äî 100 stETH tokens
- Guard: `100e18 ‚â§ 0 + 100e18` ‚Üí passes
- Actual ETH value queued to Lido: 100 √ó 1.05 = **105 ETH**
- Overage: 5 ETH worth of stETH is unnecessarily locked in Lido's withdrawal queue

No funds are permanently lost ‚Äî the ETH returns on `claimStEth`. However, the protocol's stETH balance is temporarily reduced beyond what user obligations require, and the intended invariant (ETH value of stETH queued ‚â§ ETH committed to users + whitelisted allowance) is violated. This matches **Low: Contract fails to deliver promised returns, but doesn't lose value**.

---

### Likelihood Explanation

The stETH/ETH Chainlink feed has historically returned values both above and below 1.0. The `withinUnstakeLimits` modifier does not consult any oracle at all, so the mismatch exists regardless of the configured price oracle. An operator acting in good faith ‚Äî passing the ETH commitment amount as the stETH amount ‚Äî will trigger this whenever stETH/ETH ‚âÝ 1.0.

---

### Recommendation

Convert `amountToUnstake` to its ETH value before comparing against `availableActiveETHWithdrawals`:

```solidity
uint256 stEthPrice = ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE))
    .getAssetPrice(address(stETH));
uint256 amountToUnstakeInETH = (amountToUnstake * stEthPrice) / 1e18;

if (amountToUnstakeInETH > whitelistedUnstakeAllowance + availableActiveETHWithdrawals) {
    revert UnstakeLimitExceeded();
}
```

The `whitelistedUnstakeAllowance` accounting should similarly be adjusted to track ETH value rather than stETH token amounts.

---

### Proof of Concept

```solidity
// Local fork or mock test ‚Äî no mainnet required
// Setup:
//   stETH oracle price = 1.05e18
//   assetsCommitted[ETH_TOKEN] = 100e18  (users initiated 100 ETH of withdrawals)

// Step 1: users fill assetsCommitted[ETH_TOKEN] = 100e18 via initiateWithdrawal(ETH, ...)
// Step 2: operator calls unstakeStEth(100e18)
//   withinUnstakeLimits check: 100e18 <= 0 + 100e18  ‚Üí does NOT revert
//   _unstakeStEth sends 100 stETH (worth 105 ETH) to Lido queue
// Step 3: assert that 105 ETH of value is now in Lido queue
//         while only 100 ETH was owed to users
//   ‚Üí invariant broken: 5 ETH over-unstaked with no corresponding user obligation
```

The check passes despite the ETH-value overage because the modifier compares raw token units without price normalization. [5](#0-4) [6](#0-5)
