### Title
`ChainlinkPriceOracle#getAssetPrice` Discards All Staleness Data from `latestRoundData`, Allowing rsETH Minting at Stale Prices - (`contracts/oracles/ChainlinkPriceOracle.sol`)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards every return value except `price`. No staleness threshold (`updatedAt`), no round completeness check (`answeredInRound`), and no zero-price guard are applied. This is the direct structural analog to the reported bug: just as using a 15-second TWAP instead of the intended 900-second TWAP produces a price that diverges from the true mark price, using a Chainlink answer with no time-validity window means the price fed into rsETH minting can be arbitrarily stale and diverge from the true current market price.

---

### Finding Description

In `contracts/oracles/ChainlinkPriceOracle.sol` line 52, the implementation is:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All four other return values — `roundId`, `startedAt`, `updatedAt`, `answeredInRound` — are discarded. There is no check of the form `block.timestamp - updatedAt <= heartbeat`, no `answeredInRound >= roundId` guard, and no `price > 0` assertion.

This price is consumed directly by `LRTOracle.getAssetPrice()`, which is the price fetcher registered for each supported LST asset (stETH, ETHx, etc.). That value then flows into `LRTDepositPool.getRsETHAmountToMint()`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

If the Chainlink feed for any supported LST is stale at a price above the true current market price — for example, because the LST depegged but the oracle has not yet updated within its heartbeat window — a depositor will receive more rsETH than the deposited asset is actually worth. The excess rsETH is backed by nothing, diluting all existing rsETH holders.

Note the contrast with `ChainlinkOracleForRSETHPoolCollateral.getRate()`, which at least checks `answeredInRound < roundID` and `timestamp == 0`. `ChainlinkPriceOracle` — the contract used for L1 deposit pricing — applies none of these guards.

---

### Impact Explanation

**Impact: High — Theft of yield / protocol insolvency.**

When a stale high price is used for a deposited LST, the minted rsETH exceeds the ETH value actually added to the protocol. Every existing rsETH holder's share is diluted by the difference. If the depeg is large (e.g., 5–10%) and the oracle is stale for hours, the cumulative extraction across multiple depositors can be material. In an extreme scenario (large depeg, long staleness window), this constitutes partial protocol insolvency.

---

### Likelihood Explanation

**Likelihood: Medium.**

Chainlink LST/ETH feeds on mainnet (e.g., stETH/ETH) have 24-hour heartbeat intervals and 0.5% deviation thresholds. A depeg event of less than 0.5% within a 24-hour window will not trigger an oracle update. Network congestion can further delay updates. LST depeg events (e.g., stETH during the Merge, ETHx during Stader incidents) are historically documented. An unprivileged depositor needs only to call `depositAsset()` during the staleness window — no special access required.

---

### Recommendation

Replace the bare `latestRoundData()` call in `ChainlinkPriceOracle.getAssetPrice()` with full validation:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

require(price > 0, "Invalid price");
require(answeredInRound >= roundId, "Stale round");
require(block.timestamp - updatedAt <= MAX_STALENESS_SECONDS, "Price too old");
```

`MAX_STALENESS_SECONDS` should be set per-feed based on the Chainlink heartbeat (e.g., 86 400 s for a 24-hour feed, with a small buffer). Alternatively, store a per-feed staleness threshold in the mapping alongside the feed address.

---

### Proof of Concept

1. stETH depegs to 0.97 ETH due to a slashing event; the true ETH value of 100 stETH is 97 ETH.
2. The Chainlink stETH/ETH feed has not yet updated (within its 24-hour heartbeat, deviation < 0.5%); it still reports 1.0 ETH per stETH.
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, referralId)`.
4. `getRsETHAmountToMint(stETH, 100e18)` computes:
   - `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → stale `1.0e18`
   - `rsethAmountToMint = 100e18 * 1e18 / rsETHPrice` (priced as if 100 ETH was deposited)
5. Attacker receives rsETH worth 100 ETH, having deposited only 97 ETH worth of stETH.
6. The 3 ETH shortfall is borne by all existing rsETH holders through dilution.
7. The attack is repeatable for the entire duration of the staleness window with no privileged access required.
