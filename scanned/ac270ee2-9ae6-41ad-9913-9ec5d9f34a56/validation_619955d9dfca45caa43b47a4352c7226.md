### Title
ETH Deposit Limit Bypass Due to Missing Amount in Limit Check - (File: contracts/LRTDepositPool.sol)

### Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric check for ETH versus LST assets. For LSTs the incoming `amount` is included in the comparison; for ETH it is omitted. Any depositor can therefore deposit an arbitrarily large amount of ETH in a single call even when the protocol's configured deposit cap has been reached, bypassing the intended safety limit.

### Finding Description
In `LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` (lines 676–682):

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount not included
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← amount included
}
```

For ETH the function returns `true` (i.e., "limit exceeded") only when the **pre-deposit** total already exceeds the cap. It never adds `amount` to the comparison. Consequently, as long as `totalAssetDeposits ≤ depositLimit`, the check always passes regardless of how large `amount` is. A single `depositETH` call with an arbitrarily large `msg.value` will succeed even if it pushes the total far beyond the configured limit.

For LST assets the correct formula `totalAssetDeposits + amount > limit` is used, so the same bug does not affect them.

The entry path is fully permissionless: `depositETH` → `_beforeDeposit` → `_checkIfDepositAmountExceedesCurrentLimit`.

### Impact Explanation
The deposit limit (`depositLimitByAsset`) is the protocol's primary risk-management gate for ETH exposure. Bypassing it allows the protocol to absorb unbounded ETH, minting rsETH beyond the intended ceiling. This violates the protocol's promised cap enforcement and exposes the system to EigenLayer slashing or liquidity risk beyond what governance intended. Impact is classified as **Low — contract fails to deliver promised returns** (the cap guarantee is broken without direct fund loss to existing holders).

### Likelihood Explanation
The path requires no special role, no governance capture, and no external dependency. Any depositor who observes that `totalAssetDeposits` is near the limit can immediately exploit this in a single transaction. Likelihood is **High**.

### Recommendation
Add `amount` to the ETH branch of the check, mirroring the LST branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept
Assume `depositLimitByAsset(ETH) = 1000 ether` and `totalAssetDeposits(ETH) = 999 ether`.

1. Attacker calls `depositETH{value: 500 ether}(0, "")`.
2. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH, 500 ether)`.
3. Inside the function: `totalAssetDeposits = 999 ether`; ETH branch evaluates `999 ether > 1000 ether` → `false` → limit not exceeded.
4. `_mintRsETH` mints rsETH for the full 500 ETH deposit.
5. Post-deposit total is 1499 ETH — 49.9% above the configured cap — with no revert.

The same call with an LST would revert at step 3 because `999 + 500 > 1000` → `true`. [1](#0-0) [2](#0-1)
