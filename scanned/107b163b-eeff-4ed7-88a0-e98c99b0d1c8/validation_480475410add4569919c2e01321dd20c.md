### Title
Withdrawal Payout Capped at Request-Time Price, Stranding Accrued Yield in Vault - (`contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTWithdrawalManager` uses a two-step withdrawal process: `initiateWithdrawal` locks in an `expectedAssetAmount` at the current rsETH price, and `unlockQueue` later settles the request. The `_calculatePayoutAmount` function caps the user's payout at the original `expectedAssetAmount` even when rsETH has appreciated. The full `rsETHUnstaked` is burned, but the user receives less than the current fair value of their rsETH. The excess assets remain in the vault, unattributed to the withdrawing user — a direct analog to M-20's pattern of profits not being credited to the rightful party.

---

### Finding Description

The withdrawal lifecycle in `LRTWithdrawalManager` is:

1. **`initiateWithdrawal`**: User transfers rsETH to the contract. `expectedAssetAmount` is computed at the current rsETH/asset price ratio and stored in the `WithdrawalRequest`. [1](#0-0) 

2. **`unlockQueue` → `_unlockWithdrawalRequests`**: An operator calls this after the withdrawal delay. It calls `_calculatePayoutAmount` to determine the final payout: [2](#0-1) 

The function returns `min(expectedAssetAmount, currentReturn)`. When rsETH price has risen since the request, `currentReturn > expectedAssetAmount`, so the user is capped at the lower, stale `expectedAssetAmount`.

3. **`_unlockWithdrawalRequests`**: The full `rsETHUnstaked` is added to `rsETHAmountToBurn`, but only `payoutAmount` (= `expectedAssetAmount`) is taken from the vault: [3](#0-2) 

4. **`unlockQueue`**: The full `rsETHAmountToBurn` is burned: [4](#0-3) 

5. **`completeWithdrawal` → `_processWithdrawalCompletion`**: User receives only `request.expectedAssetAmount`: [5](#0-4) 

**The accounting gap**: The rsETH burned represents `currentReturn` worth of assets at unlock time. The user receives only `expectedAssetAmount`. The difference `currentReturn - expectedAssetAmount` remains in the `LRTUnstakingVault` — not credited to the withdrawing user, not sent to treasury, simply left as excess backing for remaining rsETH holders.

This is structurally identical to M-20: in a multi-step process, the counterparty's loss (rsETH burned) is fully accounted for, but the user's gain (rsETH appreciation) is not credited to them.

---

### Impact Explanation

**High — Theft of unclaimed yield.**

A user who initiates a withdrawal and waits through the delay period (default 8 days) while rsETH appreciates will have their rsETH burned at full current value but receive only the stale, lower amount locked at request time. The accrued yield between request and unlock is permanently lost to the withdrawing user. The excess assets remain in the vault, silently redistributed to remaining rsETH holders rather than returned to the user who earned them.

Concretely: if a user requests withdrawal of 1 rsETH when rsETH = 1.05 ETH (expecting 1.05 ETH), and rsETH rises to 1.08 ETH by unlock time, the user's 1 rsETH is burned but they receive only 1.05 ETH. The 0.03 ETH difference is stranded in the vault.

---

### Likelihood Explanation

**High.** rsETH price increases continuously as EigenLayer staking rewards accrue and are reflected via `LRTOracle.updateRSETHPrice`. The default withdrawal delay is 8 days: [6](#0-5) 

Over an 8-day window, rsETH price appreciation is a near-certainty under normal protocol operation. Every queued withdrawal processed during a period of rsETH appreciation triggers this loss for the withdrawing user.

---

### Recommendation

In `_calculatePayoutAmount`, return `currentReturn` when rsETH has appreciated (i.e., when `currentReturn > expectedAssetAmount`), so users receive the full current fair value of their rsETH at unlock time. The cap should only apply on the downside (slashing/price drop) to protect the vault from over-commitment:

```solidity
function _calculatePayoutAmount(...) private view returns (uint256) {
    uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
-   return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
+   return currentReturn; // or: return max(expectedAssetAmount, currentReturn) if upside is desired
}
```

If the protocol intentionally caps upside (e.g., to prevent oracle-manipulation profit), the excess assets should at minimum be explicitly credited to the user or returned to the vault's accounting in a way that does not silently dilute the withdrawing user's claim.

---

### Proof of Concept

1. rsETH price = 1.05 ETH/rsETH, stETH price = 1.00 ETH/stETH.
2. User calls `initiateWithdrawal(stETH, 1e18 rsETH)`. `expectedAssetAmount = 1e18 * 1.05 / 1.00 = 1.05e18 stETH`. rsETH is transferred to the contract. `assetsCommitted[stETH] += 1.05e18`.
3. 8 days pass. rsETH price rises to 1.08 ETH/rsETH (staking rewards accrued).
4. Operator calls `unlockQueue(stETH, ...)`. `_calculatePayoutAmount` computes `currentReturn = 1e18 * 1.08 / 1.00 = 1.08e18`. Returns `min(1.05e18, 1.08e18) = 1.05e18`.
5. `rsETHAmountToBurn += 1e18` (full rsETH burned). `assetAmountToUnlock += 1.05e18` (only 1.05e18 stETH taken from vault).
6. User calls `completeWithdrawal`. Receives 1.05e18 stETH.
7. **Result**: User burned 1 rsETH worth 1.08 ETH but received only 1.05 ETH. 0.03 ETH worth of stETH remains in the vault, unattributed to the user who earned it. [2](#0-1) [7](#0-6)
