### Title
`whenNotPaused` on `completeWithdrawal` and `instantWithdrawal` Freezes User Funds When `LRTWithdrawalManager` Is Paused - (File: contracts/LRTWithdrawalManager.sol)

---

### Summary

`LRTWithdrawalManager` applies the `whenNotPaused` modifier to both `completeWithdrawal` and `instantWithdrawal`. When the contract is paused, users who have already locked their rsETH into the withdrawal queue cannot retrieve their entitled assets, temporarily freezing their funds.

---

### Finding Description

`LRTWithdrawalManager` implements a two-step withdrawal process:

1. **`initiateWithdrawal`** — user transfers rsETH into the contract and a `WithdrawalRequest` is queued.
2. **`unlockQueue`** (operator) — rsETH is burned and assets are pulled from `LRTUnstakingVault`.
3. **`completeWithdrawal`** — user claims their LST/ETH.

All three user-facing functions carry `whenNotPaused`:

```solidity
function initiateWithdrawal(...) external nonReentrant whenNotPaused ...
function completeWithdrawal(...) external nonReentrant whenNotPaused { ... }
function instantWithdrawal(...) external nonReentrant whenNotPaused ...
```

Blocking `initiateWithdrawal` when paused is reasonable — it prevents new exposure. However, blocking `completeWithdrawal` and `instantWithdrawal` is not: at the point a user calls `completeWithdrawal`, their rsETH has already been burned and the corresponding LST/ETH has already been pulled from the vault into the `LRTWithdrawalManager` contract. The user is merely claiming assets they are already entitled to. Pausing the contract at any point after `unlockQueue` runs — but before the user calls `completeWithdrawal` — leaves those assets stranded inside the contract with no user-accessible exit path.

Similarly, `unlockQueue` itself carries `whenNotPaused`, so even withdrawals that have not yet been unlocked cannot advance through the queue while paused.

---

### Impact Explanation

Users who have initiated withdrawals and whose requests have been unlocked (rsETH burned, assets sitting in `LRTWithdrawalManager`) cannot call `completeWithdrawal` to retrieve their LST or ETH while the contract is paused. Their funds are temporarily frozen inside the contract with no user-accessible exit. This matches the **Temporary freezing of funds** impact category (Medium).

---

### Likelihood Explanation

The `PAUSER_ROLE` can pause `LRTWithdrawalManager` at any time. Pauses are expected to occur during emergencies — precisely the scenario where users most urgently need to withdraw. The withdrawal queue is a normal, ongoing operational state, so there will routinely be unlocked-but-unclaimed requests in flight. The combination of a routine operational state and an expected emergency action makes this scenario realistic.

---

### Recommendation

Remove `whenNotPaused` from `completeWithdrawal` and `instantWithdrawal`. These functions only disburse assets that users are already entitled to; they do not accept new deposits or create new protocol exposure. Optionally, also remove `whenNotPaused` from `unlockQueue` so the operator can continue processing the queue during a pause, or add a separate `emergencyCompleteWithdrawal` path that bypasses the pause check.

---

### Proof of Concept

1. Alice calls `initiateWithdrawal(stETH, 10e18, ...)`. Her 10 rsETH is transferred into `LRTWithdrawalManager`.
2. Operator calls `unlockQueue(stETH, ...)`. Alice's rsETH is burned; 10 stETH is pulled from `LRTUnstakingVault` into `LRTWithdrawalManager`. `unlockedWithdrawalsCount[stETH]` increments.
3. The withdrawal delay passes.
4. A problem is detected; `PAUSER_ROLE` calls `pause()` on `LRTWithdrawalManager`.
5. Alice calls `completeWithdrawal(stETH, ...)`. The call reverts at the `whenNotPaused` modifier:

```solidity
// contracts/LRTWithdrawalManager.sol:183
function completeWithdrawal(address asset, string calldata referralId) external nonReentrant whenNotPaused {
    _processWithdrawalCompletion(asset, msg.sender, referralId);
}
```

Alice's 10 stETH remains locked inside `LRTWithdrawalManager` with no user-accessible exit until an admin calls `unpause()`. The same applies to `instantWithdrawal`:

    function initiateWithdrawal(
        address asset,
        uint256 rsETHUnstaked,
        string calldata referralId

        external
        override
        nonReentrant
        whenNotPaused
        onlySupportedAsset(asset)
        onlySupportedStrategy(asset)
    {
        if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
            revert InvalidAmountToWithdraw();
        }

        IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);

        uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);

        if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();

        // preventing over-withdrawal.
        assetsCommitted[asset] += expectedAssetAmount;

        _addUserWithdrawalRequest(asset, rsETHUnstaked, expectedAssetAmount);

        emit ReferralIdEmitted(referralId);
    }
