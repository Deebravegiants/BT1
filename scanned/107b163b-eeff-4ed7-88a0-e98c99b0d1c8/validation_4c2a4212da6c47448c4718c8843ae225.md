### Title
Permissionless `FeeReceiver.sendFunds()` Allows Any Attacker to DoS Both Oracle Price Update Paths via `DailyFeeMintLimitExceeded` - (File: contracts/FeeReceiver.sol / contracts/LRTOracle.sol)

---

### Summary

`FeeReceiver.sendFunds()` has no access control. An attacker can send ETH to `FeeReceiver` and call `sendFunds()` to inflate the deposit pool's ETH balance (and therefore TVL). When `_updateRsETHPrice()` is subsequently called, the computed `rsethAmountToMintAsProtocolFee` can exceed `maxFeeMintAmountPerDay`, causing `_checkAndUpdateDailyFeeMintLimit()` to revert with `DailyFeeMintLimitExceeded`. Because both `updateRSETHPrice()` and `updateRSETHPriceAsManager()` call the same `_updateRsETHPrice()` internal function — and neither has a bypass for the daily limit check — both price update paths are blocked. The DoS persists across 24-hour period resets as long as the attacker's ETH remains in the pool, requiring admin intervention to resolve.

---

### Finding Description

**Step 1 — Permissionless ETH injection into TVL**

`FeeReceiver.sendFunds()` carries no role modifier: [1](#0-0) 

It forwards the entire ETH balance to `LRTDepositPool.receiveFromRewardReceiver()`, which is also unrestricted: [2](#0-1) 

**Step 2 — Injected ETH is counted in TVL**

`LRTOracle._getTotalEthInProtocol()` calls `getTotalAssetDeposits(ETH_TOKEN)`, which resolves to `getETHDistributionData()`. That function reads the raw ETH balance of the deposit pool: [3](#0-2) 

So every wei the attacker routes through `sendFunds()` is immediately reflected in `totalETHInProtocol`.

**Step 3 — Inflated TVL produces a fee that exceeds the daily limit**

Inside `_updateRsETHPrice()`, when `totalETHInProtocol > previousTVL`: [4](#0-3) 

The resulting `rsethAmountToMintAsProtocolFee` is passed to `_checkAndUpdateDailyFeeMintLimit()`: [5](#0-4) 

`_checkAndUpdateDailyFeeMintLimit()` reverts unconditionally when the amount exceeds the cap: [6](#0-5) 

**Step 4 — Both price update paths are blocked**

`updateRSETHPrice()` (public, no role) and `updateRSETHPriceAsManager()` (manager-only) both call the same `_updateRsETHPrice()`: [7](#0-6) 

The manager role check inside `_updateRsETHPrice()` only gates the `PriceAboveDailyThreshold` revert path (line 263), not the `_checkAndUpdateDailyFeeMintLimit()` call. There is no manager bypass for the daily limit check, so `updateRSETHPriceAsManager()` also reverts.

**Step 5 — DoS persists beyond the 24-hour window**

`_checkAndUpdateDailyFeeMintLimit()` resets `currentPeriodMintedFeeAmount` to 0 at the start of each new period: [8](#0-7) 

However, the attacker's ETH remains in the deposit pool. On the next period, the fee computation still sees the inflated TVL and produces a fee exceeding `maxFeeMintAmountPerDay`. The DoS therefore repeats every period without any further attacker action, until an admin either increases `maxFeeMintAmountPerDay` or removes the ETH from the pool.

---

### Impact Explanation

All protocol operations that depend on a current rsETH price — deposits, withdrawals, and `unlockQueue` operations — are blocked for as long as the DoS is active. Because the attacker's ETH persists in the pool, the DoS is not self-healing at the 24-hour boundary; it requires privileged admin intervention to resolve. This matches **Medium. Temporary freezing of funds**.

---

### Likelihood Explanation

- The attack requires only ETH (no tokens, no approvals, no special role).
- `sendFunds()` is fully permissionless.
- `maxFeeMintAmountPerDay` is a finite value set by the manager (the feature exists precisely to limit fee minting).
- The cost to the attacker is the ETH sent, which is not lost — it increases TVL and therefore the value of any rsETH the attacker holds, making the attack nearly free in net terms.
- Likelihood: **High**.

---

### Recommendation

1. **Add access control to `sendFunds()`** — restrict it to `MANAGER` or `OPERATOR_ROLE` so only trusted parties can flush rewards into the deposit pool.
2. **Cap rather than revert in `_checkAndUpdateDailyFeeMintLimit()`** — instead of reverting when the fee exceeds the daily limit, mint only up to the remaining limit and carry the excess to the next period. This prevents the daily limit from becoming a DoS vector.
3. **Provide a manager bypass for the daily limit check** — `updateRSETHPriceAsManager()` should be able to proceed even when the daily limit is exceeded, so the manager can always recover the price oracle.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Preconditions:
//   maxFeeMintAmountPerDay = 1 wei
//   rsETHPrice = 1e18, rsethSupply > 0, protocolFeeInBPS > 0
//   feePeriodStartTime set within last 24 h

function testDoS() public {
    // 1. Seed FeeReceiver with 1 ETH
    vm.deal(attacker, 1 ether);
    vm.prank(attacker);
    (bool ok,) = address(feeReceiver).call{value: 1 ether}("");
    require(ok);

    // 2. Flush ETH into deposit pool (permissionless)
    vm.prank(attacker);
    feeReceiver.sendFunds();

    // 3. Public price update reverts
    vm.expectRevert(abi.encodeWithSelector(
        ILRTOracle.DailyFeeMintLimitExceeded.selector
    ));
    lrtOracle.updateRSETHPrice();

    // 4. Manager price update also reverts
    vm.prank(manager);
    vm.expectRevert(abi.encodeWithSelector(
        ILRTOracle.DailyFeeMintLimitExceeded.selector
    ));
    lrtOracle.updateRSETHPriceAsManager();
}
```
