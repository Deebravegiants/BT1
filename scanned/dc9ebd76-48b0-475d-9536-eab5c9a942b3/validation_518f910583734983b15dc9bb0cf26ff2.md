### Title
`ChainlinkPriceOracle.getAssetPrice` Uses Unvalidated Chainlink Price Without Staleness or Validity Checks - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary
`ChainlinkPriceOracle.getAssetPrice` calls Chainlink's `latestRoundData()` but discards all validation return values (`updatedAt`, `answeredInRound`), and does not check whether the returned `price` is positive. This is the oracle used by `LRTOracle.getAssetPrice` to price LST assets (stETH, cbETH, etc.) when computing how much rsETH to mint. A stale or invalid Chainlink price is silently accepted and used to determine the rsETH minting ratio, allowing a depositor to receive more rsETH than the assets they deposit are actually worth.

### Finding Description
`ChainlinkPriceOracle.getAssetPrice` at line 52 reads only the `answer` field from `latestRoundData()`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

The contract ignores `updatedAt` (staleness), `answeredInRound` vs `roundId` (incomplete round), and does not guard against `price <= 0`. This is in direct contrast to `ChainlinkOracleForRSETHPoolCollateral.getRate()`, a sibling oracle in the same repository, which performs all three checks:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The unvalidated price flows into `LRTOracle._getTotalEthInProtocol` (line 339: `uint256 assetER = getAssetPrice(asset)`) and into `LRTDepositPool.getRsETHAmountToMint` (line 520: `rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()`). The minting ratio is therefore `liveChainlinkAssetPrice / storedRsETHPrice`. If the live Chainlink price is stale-high while `rsETHPrice` was last stored at the correct value, the numerator is inflated and the depositor receives excess rsETH.

### Impact Explanation
When a Chainlink feed is stale (e.g., during network congestion, a circuit-breaker event, or a sequencer outage on L2), the asset price returned by `getAssetPrice` may be higher than the true current price. Because `LRTDepositPool.depositAsset` uses this live (stale) price directly while `rsETHPrice` is a stored snapshot that may have been set at the correct price, the minting ratio is skewed in the depositor's favour. The depositor receives more rsETH than the ETH-equivalent value they deposited, diluting all existing rsETH holders. This constitutes theft of yield from existing holders. Additionally, if `price` is ever zero or negative (Chainlink circuit-breaker minimum), `uint256(price)` either produces zero (DoS via division-by-zero in downstream callers) or wraps to `2^256 - 1` (causing arithmetic overflow reverts), temporarily freezing deposits.

### Likelihood Explanation
Chainlink feeds have historically gone stale during periods of high network congestion and have returned clamped/incorrect values during extreme market events (e.g., LUNA collapse, USDC depeg). The `updateRSETHPrice()` function is public and permissionless, meaning any actor can lock in a stale price into `rsETHPrice` at any time. The attack requires no privileged access: any depositor can call `depositAsset` while a feed is stale.

### Recommendation
Apply the same validation pattern already present in `ChainlinkOracleForRSETHPoolCollateral.getRate()` to `ChainlinkPriceOracle.getAssetPrice`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();
    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Additionally, consider adding a configurable `heartbeat` staleness threshold per asset (e.g., `block.timestamp - updatedAt > heartbeat`) to catch feeds that are technically complete but outdated.

### Proof of Concept

1. Assume stETH/ETH Chainlink feed last updated 4 hours ago at `1.05e18` (stale). True current price is `1.00e18`.
2. `rsETHPrice` was last stored correctly at `1.05e18` (1 rsETH = 1.05 ETH).
3. Attacker calls `depositAsset(stETH, 100e18, 0, "")`.
4. `getRsETHAmountToMint` computes: `(100e18 * 1.05e18) / 1.05e18 = 100e18` rsETH minted.
5. Correct amount should be: `(100e18 * 1.00e18) / 1.05e18 ≈ 95.24e18` rsETH.
6. Attacker receives `~4.76 rsETH` excess, diluting all existing holders.
