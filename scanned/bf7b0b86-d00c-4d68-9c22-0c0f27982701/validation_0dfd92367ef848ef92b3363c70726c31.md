### Title
Reward Sniping via Frontrunning `notifyRewardAmount` with No Warmup Period — (File: `contracts/KERNEL/KernelDepositPool.sol`)

### Summary
`KernelDepositPool` has no warmup period before rewards begin accruing. An attacker who holds KERNEL tokens can frontrun the admin's `notifyRewardAmount` transaction, stake a large amount, earn a disproportionate share of the reward stream, and exit immediately — because `withdrawalDelay` is uninitialized (defaults to `0`) until the admin explicitly sets it.

### Finding Description
`KernelDepositPool` is a Synthetix-style staking contract. Rewards drip continuously at `rewardRate` tokens per second over a `duration` window. The `updateReward` modifier snapshots each user's earned rewards at the current `rewardPerToken()` on every state-changing call.

**Root cause 1 — no warmup period:**
`stake()` immediately increases `balanceOf[msg.sender]` and `totalKernelStaked`, so rewards begin accruing in the very next block.

```solidity
// contracts/KERNEL/KernelDepositPool.sol:281-288
function stake(uint256 _amount) external nonReentrant updateReward(msg.sender) {
    if (_amount == 0) revert AmountZero();
    balanceOf[msg.sender] += _amount;
    totalKernelStaked += _amount;
    kernelToken.safeTransferFrom(msg.sender, address(this), _amount);
    emit Staked(msg.sender, _amount);
}
```

**Root cause 2 — `withdrawalDelay` defaults to `0`:**
`initialize()` never sets `withdrawalDelay`, so it is `0` until the admin calls `setWithdrawalDelay`. With `withdrawalDelay = 0`, `unlockTime = block.timestamp`, and `claimWithdrawal` passes its `block.timestamp >= withdrawal.unlockTime` check immediately.

**Root cause 3 — `notifyRewardAmount` is a public mempool transaction:**
The admin must broadcast `notifyRewardAmount` on-chain. Any watcher can see it before it is mined.

```solidity

### Impact Explanation
An attacker who frontruns `notifyRewardAmount` with a large `stake()` call captures a large fraction of `totalKernelStaked` at the moment the reward period begins. They earn rewards proportional to that fraction for every second they remain staked. When `withdrawalDelay = 0`, they can call `initiateWithdrawal` + `claimWithdrawal` + `getReward` in the same block after any number of blocks of accumulation, recovering their principal with no lock-up cost. Legitimate long-term stakers receive a smaller share of the reward pool than they would have without the attack.

**Impact: High — theft of unclaimed yield from legitimate stakers.**

### Likelihood Explanation
- The attacker must already hold KERNEL tokens (no cross-block flash loan is possible), so capital is required.
- `notifyRewardAmount` is an admin-only call that is visible in the public mempool before inclusion.
- `withdrawalDelay` is `0` by default; if the admin has not yet called `setWithdrawalDelay`, the exit is immediate.
- No special permissions are needed beyond holding KERNEL tokens.
- The attack is profitable whenever the reward amount is large relative to gas costs.

**Likelihood: Medium.**

### Recommendation
1. Introduce a per-user warmup period: rewards should not accrue for a configurable number of blocks/seconds after a new stake.
2. Ensure `withdrawalDelay` is set to a meaningful non-zero value in `initialize()` rather than relying on a post-deployment admin call.
3. Alternatively, snapshot eligible stakers at the time `notifyRewardAmount` is called and exclude balances added in the same block.

### Proof of Concept
Assume `withdrawalDelay = 0` (uninitialized), `duration = 7 days`, `notifyRewardAmount(700_000e18)` → `rewardRate = 1e18 / second`.

1. **Block N, tx 1:** Attacker calls `stake(10_000_000e18)`. Suppose existing `totalKernelStaked = 1_000_000e18`. After stake: `totalKernelStaked = 11_000_000e18`. Attacker holds ~91% of the pool.
2. **Block N, tx 2:** Admin's `notifyRewardAmount(700_000e18)` executes. `rewardRate = 1e18/s`, `updatedAt = block.timestamp`.
3. **Block N+k (e.g., k = 86_400 blocks ≈ 1 day):** Attacker calls `initiateWithdrawal(10_000_000e18)`. The `updateReward` modifier snapshots `earned(attacker) ≈ 0.91 × 1e18 × 86_400 ≈ 78_624e18` reward tokens. `balanceOf[attacker]` drops to 0.
4. **Block N+k, same tx or next:** Attacker calls `claimWithdrawal` (passes immediately since `unlockTime = block.timestamp`). Attacker recovers all 10_000_000e18 KERNEL.
5. **Block N+k:** Attacker calls `getReward()`, receiving ~78_624e18 reward tokens.

Legitimate stakers who held throughout the day received only ~9% of that day's rewards instead of ~100%, constituting direct theft of unclaimed yield. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7)

    function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
        if (_amount == 0) revert AmountZero();

        // Prevent starting a reward period when no tokens are staked to avoid unallocated rewards
        if (totalKernelStaked == 0) revert NoStakedTokens();

        // Transfer reward tokens into the contract
        uint256 balanceBefore = rewardsToken.balanceOf(address(this));
        rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
        uint256 balanceAfter = rewardsToken.balanceOf(address(this));
        // Calculate the actual amount of tokens received in case of a transfer fee (tax)
        uint256 receivedAmount = balanceAfter - balanceBefore;

        if (block.timestamp >= finishAt) {
            rewardRate = receivedAmount / duration;
        } else {
            uint256 remaining = (finishAt - block.timestamp) * rewardRate;
            rewardRate = (receivedAmount + remaining) / duration;
        }

        if (rewardRate == 0) revert RewardRateZero();

        finishAt = block.timestamp + duration;
        updatedAt = block.timestamp;

        emit NotifyRewardAmount(receivedAmount, finishAt);
```
