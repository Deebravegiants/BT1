### Title
Admin-Controlled Fixed Slippage With No User Override Enables Fund Freeze or Loss During Bridging - (File: contracts/bridges/TACWETHBridge.sol)

### Summary
`TACWETHBridge.bridgeTokenToL1()` computes `minAmountLD` exclusively from an admin-set `slippageTolerance` state variable. Users have no mechanism to supply their own minimum-amount parameter. If the admin sets `slippageTolerance` to `0` (a valid value per the constructor guard), every bridge call will revert because `minAmountLD == amount` while LayerZero OFT bridges always deduct some dust/fee. If the admin sets it near `BASIS_POINTS_DIVISOR` (10 000), users receive nearly nothing on L1.

### Finding Description
`TACWETHBridge` stores a single global `slippageTolerance` (line 29) that only `DEFAULT_ADMIN_ROLE` can update via `setSlippageTolerance()` (line 86). The public entry point `bridgeTokenToL1()` (line 100) calls `getMinAmount(amount)` (line 123), which computes:

```solidity
uint256 minAmount = amount * (BASIS_POINTS_DIVISOR - slippageTolerance) / BASIS_POINTS_DIVISOR;
```

There is no parameter in `bridgeTokenToL1(address recipient, uint256 amount)` for the caller to supply a custom `minAmountLD`. The user's WETH is pulled from them (line 116) before the bridge call, so if the bridge reverts due to the admin-set tolerance being too tight, the user's tokens are already transferred and the whole transaction reverts — but the user cannot retry with a looser tolerance without waiting for the admin to change the global value.

**Freeze path**: `slippageTolerance = 0` → `minAmountLD = amount`. LayerZero OFT bridges deduct a small amount for cross-chain fees, so `amountReceivedLD < amount`, causing the bridge to revert. All user bridge calls are frozen until admin raises the tolerance.

**Loss path**: `slippageTolerance` set near `10 000` → `minAmountLD ≈ 0`. The bridge accepts any received amount, and users can receive a tiny fraction of their WETH on L1.

### Impact Explanation
- **Freeze path**: Temporary freezing of user funds (Medium). Users cannot bridge their WETH to L1 until the admin intervenes.
- **Loss path**: Direct loss of user funds in motion (Critical). Users receive near-zero WETH on L1 after sending their full balance.

The freeze path is the primary realistic impact; the loss path requires a malicious admin.

### Likelihood Explanation
- The freeze path is realistic: `slippageTolerance = 0` is a valid constructor/setter value (the only guard is `> BASIS_POINTS_DIVISOR`), and an admin could set it believing it means "no slippage allowed" as a safety measure, inadvertently freezing all bridge operations.
- The loss path requires deliberate admin action.
- There is no timelock or multisig enforced on `setSlippageTolerance`, so changes take effect immediately with no user notice.

### Recommendation
1. Add a `minAmountLD` parameter to `bridgeTokenToL1()` so callers can specify their own floor, using the global `slippageTolerance` only as a maximum bound (i.e., revert if the user-supplied `minAmountLD` is below the protocol minimum).
2. Add a lower-bound guard in `setSlippageTolerance()` so `slippageTolerance` cannot be set to `0` (which makes `minAmountLD == amount` and freezes the bridge).
3. Protect `setSlippageTolerance()` with a timelock so users have advance notice of changes.

### Proof of Concept
1. Admin calls `setSlippageTolerance(0)`.
2. User calls `bridgeTokenToL1(recipient, 1 ether)` with the required `msg.value`.
3. `getMinAmount(1 ether)` returns `1 ether` (no slippage allowed).
4. LayerZero OFT bridge deducts a small cross-chain fee, so `amountReceivedLD < 1 ether`.
5. The bridge reverts because `amountReceivedLD < minAmountLD`.
6. The entire transaction reverts; the user cannot bridge until admin changes the global tolerance.

```text
    function setSlippageTolerance(uint256 newSlippageTolerance) external onlyRole(DEFAULT_ADMIN_ROLE) {
        if (newSlippageTolerance > BASIS_POINTS_DIVISOR) {
            revert InvalidSlippageTolerance();
        }
```

```text
    function bridgeTokenToL1(address recipient, uint256 amount) external payable nonReentrant {
        UtilLib.checkNonZeroAddress(recipient);

        if (amount == 0) {
            revert ZeroAmount();
        }

        // Calculate the native fee for bridging
        uint256 nativeFee = getNativeFee(amount, recipient);

        // Check if the msg.value is equal to the native fee for bridging
        if (msg.value != nativeFee) {
            revert InvalidNativeFee();
        }

        // Transfer the tokens to this contract
        IERC20(address(wethOFT)).safeTransferFrom(msg.sender, address(this), amount);

        // Bridge WETH to the L1 recipient
        SendParam memory sendParam = SendParam({
            dstEid: dstLzChainId,
            to: getReceiver(recipient),
            amountLD: amount,
            minAmountLD: getMinAmount(amount),
            extraOptions: bytes(""),
            composeMsg: bytes(""),
            oftCmd: bytes("")
        });

        MessagingFee memory fee = MessagingFee({ nativeFee: nativeFee, lzTokenFee: 0 });

        (, OFTReceipt memory oftReceipt) = wethOFT.send{ value: nativeFee }(sendParam, fee, msg.sender);

        emit BridgedWETHToL1(dstLzChainId, recipient, oftReceipt.amountSentLD, oftReceipt.amountReceivedLD);
    }
```

```text
    function getMinAmount(uint256 amount) public view returns (uint256) {
        if (amount == 0) {
            revert ZeroAmount();
        }

        uint256 minAmount = amount * (BASIS_POINTS_DIVISOR - slippageTolerance) / BASIS_POINTS_DIVISOR;

        if (minAmount == 0) {
            revert InvalidMinAmount();
        }

        return minAmount;
    }
```
