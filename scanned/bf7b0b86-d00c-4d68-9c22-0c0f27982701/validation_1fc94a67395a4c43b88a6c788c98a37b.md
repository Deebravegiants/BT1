### Title
`instantWithdrawal` Bypasses `assetsCommitted` Protection, Enabling Temporary Freeze of Queued Withdrawals — (File: contracts/LRTWithdrawalManager.sol)

### Summary

`LRTWithdrawalManager.instantWithdrawal` drains assets from `LRTUnstakingVault` without checking or decrementing `assetsCommitted`. The only guard is `queuedWithdrawalsBuffer`, which defaults to `0` and is never automatically synchronized with `assetsCommitted`. When instant withdrawals are enabled, any rsETH holder can drain the unstaking vault down to the buffer, leaving committed queued withdrawals unable to be unlocked.

### Finding Description

The withdrawal system uses two separate accounting mechanisms that are not synchronized:

**Mechanism 1 — `assetsCommitted` in `LRTWithdrawalManager`:**
When a user calls `initiateWithdrawal`, the expected asset amount is added to `assetsCommitted[asset]`. [1](#0-0) 

`getAvailableAssetAmount` uses this to prevent over-commitment: [2](#0-1) 

**Mechanism 2 — `queuedWithdrawalsBuffer` in `LRTUnstakingVault`:**
`instantWithdrawal` checks only `getAssetsAvailableForInstantWithdrawal`, which uses a manually set, static buffer: [3](#0-2) 

The `instantWithdrawal` function burns rsETH and redeems assets from the vault, but **never updates `assetsCommitted`**: [4](#0-3) 

The `queuedWithdrawalsBuffer` mapping defaults to `0` (Solidity mapping default). There is no enforcement that it must be set before instant withdrawals are enabled. The operator sets it manually and independently of `assetsCommitted`: [5](#0-4) 

When `unlockQueue` is called, it reads `unstakingVault.balanceOf(asset)` as `totalAvailableAssets`. If the vault has been drained to zero, it reverts: [6](#0-5) 

The `_createUnlockParams` function confirms the dependency on the vault's live balance: [7](#0-6) 

### Impact Explanation

When `isInstantWithdrawalEnabled[asset] = true` and `queuedWithdrawalsBuffer[asset] = 0` (the default), an attacker holding rsETH can drain the entire unstaking vault via `instantWithdrawal`. This causes `unlockQueue` to revert for all subsequent calls, freezing all pending queued withdrawal requests until the operator manually replenishes the vault. Users who called `initiateWithdrawal` and are waiting for `unlockQueue` cannot progress their withdrawal. This is a **temporary freezing of funds** (Medium impact).

### Likelihood Explanation

- `isInstantWithdrawalEnabled` must be set to `true` by the LRT Manager — a normal operational decision when the feature is deployed.
- `queuedWithdrawalsBuffer` defaults to `0` and requires a separate, explicit operator action to set. There is no protocol-level enforcement linking it to `assetsCommitted`. In the absence of documentation or automated tooling enforcing this, the buffer will frequently be `0` or stale.
- The attacker only needs to hold rsETH (any legitimate user) and call `instantWithdrawal` repeatedly until the vault is drained.
- No privileged access, front-running, or external dependency is required.

Likelihood: **Medium**.

### Recommendation

1. In `instantWithdrawal`, compute the available amount for instant withdrawal as `min(getAssetsAvailableForInstantWithdrawal(asset), totalVaultBalance - assetsCommitted[asset])` so that committed queued withdrawals are always protected regardless of the buffer setting.
2. Alternatively, automatically derive `queuedWithdrawalsBuffer` from `assetsCommitted` rather than relying on a manually set static value.
3. Add a check in `setInstantWithdrawalEnabled` that requires `queuedWithdrawalsBuffer[asset] >= assetsCommitted[asset]` before enabling instant withdrawals.

### Proof of Concept

1. Alice calls `initiateWithdrawal(ETH, 100e18, "")` → `assetsCommitted[ETH] = 100e18`.
2. Operator moves 100 ETH to `LRTUnstakingVault` to prepare for `unlockQueue`.
3. Manager calls `setInstantWithdrawalEnabled(ETH, true)`. `queuedWithdrawalsBuffer[ETH]` is `0` (default).
4. Attacker (holding rsETH) calls `instantWithdrawal(ETH, X, "")` where `X` is sized to drain the vault. `getAssetsAvailableForInstantWithdrawal` returns `vaultBalance - 0 = 100 ETH`. The call succeeds and removes all 100 ETH from the vault.
5. Operator calls `unlockQueue(ETH, ...)`. Inside `_createUnlockParams`, `unstakingVault.balanceOf(ETH) = 0`. The function reverts at `if (params.totalAvailableAssets == 0) revert AmountMustBeGreaterThanZero()`.
6. Alice's queued withdrawal is frozen. `assetsCommitted[ETH]` still equals `100e18` but there are no assets in the vault to unlock against. The freeze persists until the operator manually replenishes the vault.

        external
        view
        onlySupportedAsset(asset)
        returns (uint256 availableAmount)
    {
        uint256 vaultBalance = balanceOf(asset);
        uint256 reservedBuffer = queuedWithdrawalsBuffer[asset];
        availableAmount = reservedBuffer >= vaultBalance ? 0 : vaultBalance - reservedBuffer;
    }
