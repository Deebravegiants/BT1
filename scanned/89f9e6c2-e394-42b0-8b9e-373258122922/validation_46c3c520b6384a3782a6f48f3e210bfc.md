### Title
Incorrect Deposit Limit Check for ETH Bypasses Protocol Safety Cap - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` uses an incomplete comparison for ETH deposits: it checks whether the *current* total already exceeds the limit, but omits the incoming deposit amount from the comparison. The ERC-20 path correctly includes `+ amount`. Any unprivileged depositor can push ETH deposits beyond the configured `depositLimitByAsset` cap.

### Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit`, the ETH branch and the ERC-20 branch use structurally different comparisons:

```solidity
// contracts/LRTDepositPool.sol lines 676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← missing `+ amount`
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct
}
```

For ETH the function returns `true` (i.e., "limit exceeded, block the deposit") only when the total *already* exceeds the cap before the new deposit arrives. If `totalAssetDeposits ≤ depositLimit` but `totalAssetDeposits + amount > depositLimit`, the check returns `false` and the deposit is accepted, silently pushing the protocol past its configured limit.

The call chain is fully public and unprivileged:

```
depositETH() [external, line 76]
  → _beforeDeposit() [line 87]
    → _checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, msg.value) [line 661]
```

### Impact Explanation
`depositLimitByAsset` is the protocol's primary risk-management cap on how much of each asset can be absorbed. Bypassing it for ETH means:

- ETH in excess of the intended cap is minted as rsETH and forwarded to `NodeDelegator` → EigenLayer strategies, creating over-exposure beyond what the protocol risk model allows.
- If the cap was set to bound slashing or liquidity risk, exceeding it can contribute to protocol insolvency (rsETH backed by more ETH than the system can safely handle).

Impact: **Critical — Protocol insolvency** (excess ETH is restaked and rsETH is minted against it; the protocol cannot unilaterally reverse this).

### Likelihood Explanation
The condition that triggers the bypass (`totalAssetDeposits ≤ limit` while `totalAssetDeposits + amount > limit`) is the normal operating state whenever the protocol is near its cap. Any depositor who monitors on-chain state can craft a single `depositETH` call that crosses the limit. No special role, front-running, or collusion is required.

Likelihood: **High**.

### Recommendation
Add the incoming `amount` to the ETH branch, matching the ERC-20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept
1. Assume `depositLimitByAsset[ETH_TOKEN] = 1000 ETH` and `getTotalAssetDeposits(ETH_TOKEN) = 999 ETH`.
2. Attacker calls `depositETH{value: 10 ether}(...)`.
3. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `999 > 1000` → `false` → limit not exceeded.
4. `_beforeDeposit` proceeds; rsETH is minted for 10 ETH.
5. `getTotalAssetDeposits` is now 1009 ETH — 9 ETH above the configured cap — with rsETH already minted and ETH already queued for EigenLayer delegation. [1](#0-0) [2](#0-1) [3](#0-2)
