### Title
Sandwich Attack on Public `updateRSETHPrice()` Allows Yield Theft from rsETH Holders - (File: contracts/LRTOracle.sol)

### Summary
`LRTOracle.updateRSETHPrice()` is a permissionless public function that increases `rsETHPrice` when EigenLayer/staking rewards accrue. An unprivileged attacker can atomically (1) deposit ETH at the stale lower price to receive inflated rsETH, (2) call `updateRSETHPrice()` to capture the reward-driven price increase, and (3) immediately withdraw via `instantWithdrawal()` (when enabled) or lock in the higher payout via `initiateWithdrawal()`. The attacker extracts yield that belongs to all existing rsETH holders.

### Finding Description

`LRTOracle.updateRSETHPrice()` carries no access control:

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [1](#0-0) 

`_updateRsETHPrice()` computes the new price as `(totalETHInProtocol - protocolFeeInETH) / rsethSupply`. When EigenLayer rewards or LST rebases have accrued since the last update, `totalETHInProtocol` exceeds `previousTVL`, so `rsETHPrice` increases:

```solidity
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
...
rsETHPrice = newRsETHPrice;
``` [2](#0-1) 

Deposits mint rsETH at the **current** (stale, pre-update) price:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [3](#0-2) 

Withdrawals redeem at the **current** (post-update) price:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
``` [4](#0-3) 

**Attack path A — instant withdrawal (when `isInstantWithdrawalEnabled[asset]` is true):**

1. Bob calls `LRTDepositPool.depositETH()` with X ETH → receives `X / rsETHPrice_old` rsETH.
2. Bob calls `LRTOracle.updateRSETHPrice()` → `rsETHPrice` increases to `rsETHPrice_new`.
3. Bob calls `LRTWithdrawalManager.instantWithdrawal()` → receives `rsETH * rsETHPrice_new / assetPrice` ETH, which is greater than X.

```solidity
function instantWithdrawal(address asset, uint256 rsETHUnstaked, ...) external nonReentrant whenNotPaused
    onlySupportedAsset(asset) onlySupportedStrategy(asset) onlyInstantWithdrawalAllowed(asset)
{
    uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
    IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
    ...
    _transferAsset(asset, msg.sender, userAmount);
``` [5](#0-4) 

**Attack path B — queued withdrawal (always available):**

1. Bob deposits at `rsETHPrice_old`.
2. Bob calls `updateRSETHPrice()` → price rises to `rsETHPrice_new`.
3. Bob calls `initiateWithdrawal()` → `expectedAssetAmount` is locked at `rsETHPrice_new`.
4. After `withdrawalDelayBlocks` (~8 days), Bob calls `completeWithdrawal()` and receives the higher amount.

The `expectedAssetAmount` is locked at initiation time and `_calculatePayoutAmount` returns `min(expectedAssetAmount, currentReturn)`, so as long as the price does not fall below the initiation price before unlock, Bob receives the full inflated amount. [6](#0-5) 

### Impact Explanation

Every time rewards accrue in EigenLayer or through LST rebases, an attacker can capture the entire reward increment for themselves rather than letting it be distributed pro-rata to all rsETH holders. This is direct **theft of unclaimed yield** from existing depositors. The profit scales linearly with the size of the deposit and the magnitude of the reward accrual. With instant withdrawal enabled the attack is atomic and risk-free; with queued withdrawal it carries an 8-day price-stability assumption but is still profitable under normal conditions.

**Impact: High — Theft of unclaimed yield.**

### Likelihood Explanation

`updateRSETHPrice()` is permissionless and must be called regularly to keep the price current. Rewards accrue continuously from EigenLayer strategies and LST rebases. Any on-chain observer can detect when `totalETHInProtocol > previousTVL` (by simulating the call) and execute the sandwich in the same block. No privileged access, leaked keys, or external protocol compromise is required. The attack is repeatable every reward cycle.

### Recommendation

1. **Restrict `updateRSETHPrice()` to authorized callers** (e.g., `onlyLRTManager` or a keeper role), removing the permissionless entry point.
2. **Alternatively, stream rewards over time** (e.g., a `rewardPerSecond` model) so that no single call causes a step-change in `rsETHPrice` that can be sandwiched.
3. **Add a deposit lock-up or withdrawal cooldown** relative to the last `updateRSETHPrice()` call so that a deposit made in the same block as a price update cannot immediately be used for withdrawal.
4. If instant withdrawal remains available, ensure it is gated behind a minimum holding period after deposit.

### Proof of Concept

```
Block N (single transaction or same block):

1. Bob calls LRTDepositPool.depositETH{value: 1000 ETH}(0, "")
   → rsETHPrice = 1.05 ETH/rsETH (stale, rewards not yet applied)
   → Bob receives 1000 / 1.05 ≈ 952.38 rsETH

2. Bob calls LRTOracle.updateRSETHPrice()
   → totalETHInProtocol has grown by 50 ETH in rewards since last update
   → rsETHPrice updates to 1.10 ETH/rsETH

3a. (instant withdrawal enabled)
    Bob calls LRTWithdrawalManager.instantWithdrawal(ETH, 952.38 rsETH, "")
    → assetAmountUnlocked = 952.38 * 1.10 = 1047.6 ETH
    → after instantWithdrawalFee (e.g. 0.1%) Bob receives ≈ 1046.5 ETH
    → Profit ≈ 46.5 ETH stolen from existing holders

3b. (queued withdrawal)
    Bob calls LRTWithdrawalManager.initiateWithdrawal(ETH, 952.38 rsETH, "")
    → expectedAssetAmount locked at 952.38 * 1.10 / 1.0 = 1047.6 ETH
    → After 8 days Bob calls completeWithdrawal() and receives 1047.6 ETH
    → Profit ≈ 47.6 ETH stolen from existing holders
```
