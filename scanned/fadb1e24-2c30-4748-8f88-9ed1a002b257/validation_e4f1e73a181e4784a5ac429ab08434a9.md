### Title
ETH Deposit Limit Check Missing Current Deposit Amount in `_checkIfDepositAmountExceedesCurrentLimit` - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit()` applies an asymmetric check: for ERC20 assets it correctly tests `totalAssetDeposits + amount > limit`, but for ETH it only tests `totalAssetDeposits > limit`, omitting the incoming deposit amount. Any unprivileged depositor can therefore push the protocol's ETH holdings well past the configured cap in a single transaction.

### Finding Description
In `LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit`:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // ← missing + amount
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [1](#0-0) 

For ETH the function returns `true` (limit exceeded) only when `totalAssetDeposits` **already** exceeds the limit. The incoming `amount` is never added to the comparison. For every ERC20 asset the incoming `amount` is correctly included.

Consequence: whenever `totalAssetDeposits ≤ limit`, the ETH branch always returns `false`, allowing the deposit regardless of how large `amount` is. A single depositor can push the ETH total arbitrarily above the configured cap in one call.

The entry path is the public `depositETH()` function, which calls `_beforeDeposit`, which calls `_checkIfDepositAmountExceedesCurrentLimit`: [2](#0-1) [3](#0-2) 

The analog to the external report is direct: `pairTransferERC20From()` handled only `ERC721_ERC20` and silently skipped `ERC1155_ERC20`; here `_checkIfDepositAmountExceedesCurrentLimit` handles ETH with a weaker (incomplete) variant of the same check applied to ERC20 tokens, causing the ETH deposit-limit branch to be functionally broken.

### Impact Explanation
The deposit limit is a protocol safety mechanism controlling how much ETH is accepted and subsequently deployed into EigenLayer strategies. Bypassing it allows:
- rsETH to be minted beyond the intended cap
- ETH to accumulate idle in the deposit pool if EigenLayer capacity is the reason for the cap, reducing yield for all rsETH holders
- The protocol's accounting invariant (`totalAssetDeposits ≤ depositLimitByAsset`) to be violated

**Impact: Low** — the contract fails to deliver promised returns (deposit-limit enforcement, yield optimization) but deposited ETH is not directly lost.

### Likelihood Explanation
Any unprivileged ETH depositor can trigger this in a single call to `depositETH()` whenever `totalAssetDeposits ≤ limit`. No special conditions, governance access, or front-running required. **Likelihood: High.**

### Recommendation
Include the incoming deposit amount in the ETH branch, consistent with the ERC20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept
1. Admin sets ETH deposit limit: `depositLimitByAsset[ETH_TOKEN] = 1000e18`.
2. Current state: `getTotalAssetDeposits(ETH_TOKEN)` = 999e18.
3. Attacker calls `depositETH{value: 500e18}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500e18)` evaluates `999e18 > 1000e18` → `false` → deposit proceeds.
5. Protocol now holds 1499 ETH worth of rsETH minted, 499 ETH above the intended cap.
6. For comparison, an ERC20 depositor with the same state and `amount = 500e18` is correctly rejected: `999e18 + 500e18 > 1000e18` → `true` → `MaximumDepositLimitReached`. [1](#0-0)
