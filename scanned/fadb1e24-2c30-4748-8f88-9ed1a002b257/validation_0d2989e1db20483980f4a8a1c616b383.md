### Title
`_updateRsETHPrice()` reverts entirely when daily fee mint limit is exceeded, leaving `rsETHPrice` stale and allowing depositors to extract yield from existing holders â€” (File: contracts/LRTOracle.sol)

---

### Summary

In `LRTOracle._updateRsETHPrice()`, when the protocol fee on a TVL increase exceeds `maxFeeMintAmountPerDay`, the call to `_checkAndUpdateDailyFeeMintLimit` reverts the entire transaction â€” including the `rsETHPrice = newRsETHPrice` assignment. The stored price is left stale at a value lower than the true TVL-backed price. Any depositor can then mint rsETH at the deflated price and redeem at the correct price once the limit resets, extracting yield that belongs to existing holders.

---

### Finding Description

`_updateRsETHPrice()` computes `newRsETHPrice` correctly from on-chain TVL, then attempts to mint a protocol fee before committing the price:

```
// line 303
_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee);
...
// line 313 â€” never reached if line 303 reverts
rsETHPrice = newRsETHPrice;
``` [1](#0-0) 

`_checkAndUpdateDailyFeeMintLimit` reverts with `DailyFeeMintLimitExceeded` whenever `currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay`: [2](#0-1) 

Because the revert unwinds the entire call, `rsETHPrice` is never updated. The stored price remains at the previous (lower) value even though the actual TVL has increased.

**Default-zero amplification:** `maxFeeMintAmountPerDay` is never set in `initialize()` and defaults to `0`. [3](#0-2) 

When `maxFeeMintAmountPerDay == 0`, the check `feeAmount > 0` is always true for any non-zero fee, so `updateRSETHPrice()` permanently reverts on every TVL increase until the manager explicitly calls `setMaxFeeMintAmountPerDay`. This makes the stale-price window indefinite rather than bounded to one day.

The stale `rsETHPrice` is consumed directly by both the deposit and withdrawal paths:

- **Deposit:** `getRsETHAmountToMint` divides by `lrtOracle.rsETHPrice()` â€” a stale-low price yields more rsETH per unit of asset deposited. [4](#0-3) 

- **Withdrawal:** `getExpectedAssetAmount` multiplies by `lrtOracle.rsETHPrice()` â€” once the price is corrected, the attacker's rsETH redeems for more assets than were deposited in ETH-equivalent terms. [5](#0-4) 

---

### Impact Explanation

An attacker who deposits while `rsETHPrice` is stale-low receives more rsETH than the true TVL entitles them to. After the price is corrected (daily limit resets or manager raises the cap), the attacker's rsETH is worth more than what they deposited. The difference is yield extracted from existing rsETH holders, whose proportional claim on the TVL is diluted by the over-minted rsETH. This is a direct theft of unclaimed yield â€” **High impact**.

---

### Likelihood Explanation

Two independent triggers exist:

1. **`maxFeeMintAmountPerDay == 0` (default):** If `setMaxFeeMintAmountPerDay` has not been called, every `updateRSETHPrice()` call reverts whenever TVL has increased. The attack window is open from deployment until the manager configures the limit.
2. **Daily limit exceeded:** Even with a non-zero limit, a single large TVL increase (e.g., a batch EigenLayer reward distribution, a significant LST price appreciation event) can push the fee above `maxFeeMintAmountPerDay` for that day, blocking the price update for up to 24 hours.

Both scenarios are realistic in normal protocol operation â€” **Medium likelihood**.

---

### Recommendation

Decouple fee minting from the price update. The `rsETHPrice` assignment must execute unconditionally. When the daily fee limit is exceeded, skip or defer the fee mint rather than reverting the entire update:

```solidity
if (protocolFeeInETH > 0) {
    uint256 rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice);
    bool withinLimit = (currentPeriodMintedFeeAmount + rsethAmountToMintAsProtocolFee <= maxFeeMintAmountPerDay);
    if (withinLimit) {
        _updateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee);
        IRSETH(rsETHTokenAddress).mint(treasury, rsethAmountToMintAsProtocolFee);
    }
    // else: fee is deferred; price still updates
}
rsETHPrice = newRsETHPrice; // always executed
```

Additionally, add a guard in `_checkAndUpdateDailyFeeMintLimit` for `maxFeeMintAmountPerDay == 0` to mean "no limit" (consistent with `remainingDailyFeeMintLimit`'s early-return on zero). [6](#0-5)

### Proof of Concept

**Setup:**
- Protocol TVL: 100 000 ETH; rsETH supply: 100 000; `rsETHPrice = 1.00 ETH`; `protocolFeeInBPS = 1000` (10%); `maxFeeMintAmountPerDay` = 0 (default, never set)

**Steps:**

1. EigenLayer distributes rewards; TVL rises to 110 000 ETH. True price = 1.10 ETH.
2. Anyone calls `updateRSETHPrice()`. Inside `_updateRsETHPrice()`:
   - `newRsETHPrice = 1.10 ETH` (computed correctly)
   - `protocolFeeInETH = 10 000 ETH * 10% = 1 000 ETH`
   - `rsethAmountToMintAsProtocolFee = 1 000 / 1.10 â‰ˆ 909 rsETH`
   - `_checkAndUpdateDailyFeeMintLimit(909)` â†’ `909 > 0` â†’ **reverts**
   - `rsETHPrice` stays at `1.00 ETH` [7](#0-6) 

3. Attacker observes on-chain: actual TVL = 110 000 ETH, `rsETHPrice` = 1.00 ETH (stale).
4. Attacker calls `LRTDepositPool.depositETH{value: 10_000 ETH}(0, "")`.
   - `getRsETHAmountToMint` â†’ `10 000 ETH * 1e18 / 1.00e18 = 10 000 rsETH` minted.
   - At the true price of 1.10 ETH, the attacker should have received only `10 000 / 1.10 â‰ˆ 9 091 rsETH`.
   - Attacker receives **909 extra rsETH**. [4](#0-3) 

5. Manager calls `setMaxFeeMintAmountPerDay(large_value)` then `updateRSETHPriceAsManager()`. `rsETHPrice` updates to 1.10 ETH.
6. Attacker calls `initiateWithdrawal(ETH, 10_000 rsETH, "")`.
   - `getExpectedAssetAmount` â†’ `10 000 * 1.10 / 1.00 = 11 000 ETH`.
7. After the delay, attacker calls `completeWithdrawal` and receives **11 000 ETH** â€” a profit of **1 000 ETH** extracted from existing holders. [5](#0-4)
