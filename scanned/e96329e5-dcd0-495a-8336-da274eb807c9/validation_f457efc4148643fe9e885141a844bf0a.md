### Title
Owner of `KernelMerkleDistributor` / `KernelTop100MerkleDistributor` can steal all KERNEL tokens by pointing `kernelDepositPool` to a malicious contract - (File: contracts/KERNEL/KernelMerkleDistributor.sol, contracts/KERNEL/KernelTop100MerkleDistributor.sol)

---

### Summary

The owner of `KernelMerkleDistributor` (and identically `KernelTop100MerkleDistributor`) can drain every KERNEL token held in the distributor by calling `setKernelDepositPool` with a malicious contract address. Because `setKernelDepositPool` immediately grants `type(uint256).max` approval to the new address, any subsequent user call to `claimAndStake` triggers an external call into the malicious contract, which can use that standing approval to `transferFrom` the entire KERNEL balance out of the distributor.

---

### Finding Description

`KernelMerkleDistributor.setKernelDepositPool` is callable by the owner with no validation on the supplied address beyond a non-zero check:

```solidity
// contracts/KERNEL/KernelMerkleDistributor.sol  lines 356-370
function setKernelDepositPool(address _kernelDepositPool) external onlyOwner {
    UtilLib.checkNonZeroAddress(_kernelDepositPool);

    address oldKernelDepositPool = address(kernelDepositPool);
    kernelDepositPool = IKernelDepositPool(_kernelDepositPool);

    kernel.forceApprove(oldKernelDepositPool, 0);
    kernel.forceApprove(_kernelDepositPool, type(uint256).max);   // ← unlimited approval to arbitrary address

    emit KernelDepositPoolUpdated(_kernelDepositPool);
}
``` [1](#0-0) 

The user-facing `claimAndStake` then makes an external call into whatever address `kernelDepositPool` currently points to:

```solidity
// contracts/KERNEL/KernelMerkleDistributor.sol  lines 270-285
function claimAndStake(...) external nonReentrant whenNotPaused {
    uint256 amountToStake = _processClaim(index, account, cumulativeAmount, merkleProof);
    IKernelDepositPool(kernelDepositPool).stakeFor(account, amountToStake);  // ← external call to mutable address
    emit ClaimedAndStaked(index, account, amountToStake);
}
``` [2](#0-1) 

Because the malicious contract already holds `type(uint256).max` approval over the distributor's KERNEL balance, its `stakeFor` implementation can call `kernel.transferFrom(distributor, attacker, kernel.balanceOf(distributor))` and drain every token in the contract — not just the amount the calling user was entitled to.

The identical pattern exists in `KernelTop100MerkleDistributor`: [3](#0-2) [4](#0-3) 

---

### Impact Explanation

All KERNEL tokens held in the distributor — representing the aggregate unclaimed rewards of every eligible user — can be stolen in a single transaction triggered by any user calling `claimAndStake`. The theft is not limited to the caller's own entitlement; the unlimited approval allows the malicious pool to drain the entire contract balance. This constitutes **direct theft of user funds at rest** (Critical).

---

### Likelihood Explanation

Exploitation requires the owner to act maliciously or for the owner key to be compromised. The owner role is a single EOA or multisig with no time-lock on `setKernelDepositPool`, so the change can be made atomically in the same block as a user's `claimAndStake` call. Given that the distributor holds a large aggregate of KERNEL rewards, the financial incentive is high. Likelihood is **Low** (requires privileged role to act maliciously), but the severity makes this a Critical finding consistent with the referenced report's classification.

---

### Recommendation

1. **Validate the new pool address** against a registry of known-good contracts (analogous to the pool-validation fix described in the referenced report for I-9).
2. **Add a time-lock** to `setKernelDepositPool` so users have time to observe and react to a pending change.
3. **Do not grant unlimited approval at setter time.** Instead, approve only the exact amount needed immediately before each `stakeFor` call, and revoke afterwards — or use a pull-based pattern where the pool calls `transferFrom` only for the amount the user is claiming.

---

### Proof of Concept

```
1. Attacker (owner) deploys MaliciousPool:
       function stakeFor(address, uint256) external {
           IERC20(kernel).transferFrom(distributor, attacker, IERC20(kernel).balanceOf(distributor));
       }

2. Owner calls:
       KernelMerkleDistributor.setKernelDepositPool(address(MaliciousPool))
   → kernel.forceApprove(MaliciousPool, type(uint256).max) executes

3. Any user calls:
       KernelMerkleDistributor.claimAndStake(index, account, cumulativeAmount, proof)
   → _processClaim validates proof, computes amountToStake
   → MaliciousPool.stakeFor(account, amountToStake) is called
   → MaliciousPool drains entire KERNEL balance of the distributor via transferFrom

4. All users' unclaimed KERNEL rewards are stolen.
```

        _verifyClaimProof(user, amount, merkleProof);

            kernel.safeTransfer(protocolTreasury, fee);
