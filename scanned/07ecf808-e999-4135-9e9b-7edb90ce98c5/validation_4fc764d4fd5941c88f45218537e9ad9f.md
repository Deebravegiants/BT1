### Title
ETH Deposit Limit Bypass Due to Missing Amount Check in `_checkIfDepositAmountExceedesCurrentLimit` — (File: `contracts/LRTDepositPool.sol`)

---

### Summary

`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` applies two different boundary checks depending on whether the deposited asset is ETH or an ERC20 token. For ERC20 tokens the incoming `amount` is included in the comparison, correctly preventing the limit from being breached. For ETH the `amount` parameter is silently ignored and only the pre-deposit total is tested. This mirrors the original report's pattern exactly: one code path uses a strict check that includes the new value, the other omits it, so the invariant that "total deposits must not exceed the configured limit" is enforced for ERC20 but not for ETH.

---

### Finding Description

`_checkIfDepositAmountExceedesCurrentLimit` (lines 676-682):

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // ← amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← amount included
}
``` [1](#0-0) 

For ETH the function returns `true` (i.e., "limit exceeded, reject") only when the **pre-deposit** total already exceeds the limit. The deposit amount itself is never added to the comparison. Consequently, as long as `totalAssetDeposits ≤ depositLimit` at the moment of the call, any ETH amount — even one that is orders of magnitude larger than the remaining capacity — passes the check and is accepted.

The public view helper `getAssetCurrentLimit` (lines 402-409) correctly computes the remaining capacity as `depositLimit − totalAssetDeposits` and signals to callers that only that much can still be deposited:

```solidity
function getAssetCurrentLimit(address asset) public view override returns (uint256) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)) {
        return 0;
    }
    return lrtConfig.depositLimitByAsset(asset) - totalAssetDeposits;
}
``` [2](#0-1) 

The inconsistency is therefore:

| Path | Check | Behaviour at boundary |
|---|---|---|
| ERC20 deposit | `totalAssetDeposits + amount > limit` | Deposit blocked when it would breach the limit |
| ETH deposit | `totalAssetDeposits > limit` | Deposit allowed even when it massively exceeds the limit |

This is structurally identical to the reference report: one function uses a check that includes the new value (`<=` / `+amount`), the other omits it, so the same invariant is enforced in one path and silently bypassed in the other.

---

### Impact Explanation

The deposit limit is the protocol's primary on-chain mechanism for capping ETH TVL and managing EigenLayer exposure. Bypassing it allows a single depositor to push the ETH TVL arbitrarily above the configured cap in one transaction. Consequences include:

- **Excess EigenLayer risk**: more ETH than the risk-managed cap is delegated to node operators, potentially exposing the protocol to slashing losses beyond what was modelled.
- **rsETH dilution**: the excess mint dilutes existing rsETH holders if the underlying yield does not scale proportionally.
- **Operational desync**: off-chain monitoring and bridging infrastructure sized to the deposit limit may be overwhelmed.

Impact classification: **Low — Contract fails to deliver promised returns** (the deposit-limit guarantee is not upheld for ETH), with potential escalation to **Medium — Temporary freezing of funds** if the excess exposure triggers the oracle's price-drop circuit-breaker and pauses the protocol.

---

### Likelihood Explanation

The entry point is the public, permissionless `depositETH` function. No special role or precondition is required beyond having ETH. The condition that makes the bypass possible (`totalAssetDeposits ≤ depositLimit`) is the normal operating state of the protocol. Any depositor can trigger this at any time.

---

### Recommendation

Include the deposit amount in the ETH branch, matching the ERC20 branch:

```diff
 if (asset == LRTConstants.ETH_TOKEN) {
-    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
+    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
 }
``` [3](#0-2)

### Proof of Concept

1. Protocol is configured with `depositLimitByAsset(ETH) = 100 ETH`.
2. Current state: `totalAssetDeposits(ETH) = 50 ETH` → `getAssetCurrentLimit` returns `50 ETH`.
3. Attacker calls `depositETH{value: 10_000 ETH}(...)`.
4. Inside `_beforeDeposit`, `_checkIfDepositAmountExceedesCurrentLimit(ETH, 10_000e18)` is called.
5. The ETH branch evaluates `50e18 > 100e18` → `false` → function returns `false` → no revert.
6. `_mintRsETH` mints rsETH for the full 10,000 ETH.
7. `totalAssetDeposits(ETH)` is now `10,050 ETH` — 100× the intended cap — while `getAssetCurrentLimit` had advertised only 50 ETH of remaining room. [4](#0-3) [1](#0-0)
