### Title
Assumed 18-Decimal Precision in TVL Accounting and rsETH Minting Breaks Share Valuation for Non-18-Decimal Assets - (File: contracts/LRTOracle.sol, contracts/LRTDepositPool.sol)

### Summary
`LRTOracle._getTotalEthInProtocol()` and `LRTDepositPool.getRsETHAmountToMint()` both silently assume every supported asset has 18 decimals. If any asset with fewer decimals (e.g., 6) is added via the permissioned `addNewSupportedAsset()`, the TVL is underestimated by a factor of `10^(18 - decimals)`, the rsETH price is correspondingly distorted, and depositors of that asset receive a proportionally tiny rsETH amount — effectively losing their funds.

### Finding Description

`LRTConfig.addNewSupportedAsset()` (gated by `TIME_LOCK_ROLE`) imposes no constraint on the decimal count of the asset being added: [1](#0-0) 

Once a non-18-decimal asset is supported, `LRTOracle._getTotalEthInProtocol()` reads its raw on-chain balance and multiplies it by the asset's ETH price using `mulWad` (which divides by `1e18`): [2](#0-1) 

The inline comment even states `"totalAssetAmt is in 1e18 precision (standard token decimals)"` — an assumption that is only valid for 18-decimal tokens. For a 6-decimal asset the raw balance is `1e12` times smaller than its 18-decimal equivalent, so the ETH contribution of that asset to `totalETHInProtocol` is `1e12` times too small.

The same assumption propagates into `LRTDepositPool.getRsETHAmountToMint()`: [3](#0-2) 

`amount` here is the raw token amount in the asset's native decimals. For a 6-decimal asset, `amount * assetPrice / rsETHPrice` yields `1e12` times fewer rsETH than the depositor is entitled to.

The same pattern appears in `LRTConverter.transferAssetFromDepositPool()`: [4](#0-3) 

And in `LRTDepositPool.getSwapAssetForETHReturnAmount()`: [5](#0-4) 

### Impact Explanation

If a 6-decimal asset is added as a supported asset:

1. **Depositor loss (Critical):** A user depositing `1e6` units (= 1 token) of a 6-decimal asset whose ETH price is `3e14` (≈ 0.0003 ETH) receives `1e6 × 3e14 / rsETHPrice` rsETH — roughly `1e12` times less than the correct `1e18 × 3e14 / rsETHPrice`. The depositor's funds are effectively confiscated.

2. **rsETH price distortion (Critical):** `rsETHPrice = totalETHInProtocol / rsethSupply`. With `totalETHInProtocol` underestimated by `1e12`, the rsETH price is artificially depressed, allowing existing holders to redeem at an inflated rate relative to new depositors of the affected asset — a direct wealth transfer from depositors to existing holders.

3. **`ethValueInWithdrawal` accounting error:** `LRTConverter` records the wrong ETH value for assets in withdrawal, further corrupting the TVL used for price computation.

### Likelihood Explanation

`addNewSupportedAsset()` requires `TIME_LOCK_ROLE`, making this a governance-gated action rather than a direct user exploit. However:

- The function contains **no decimal validation** whatsoever.
- The protocol's stated design supports multiple LSTs, and the asset list is expected to grow.
- `updatePriceOracleFor()` (no price-range check) can register an oracle for any asset, bypassing the `1e16–1e19` sanity check in `updatePriceOracleForValidated()`.
- A legitimate governance proposal to add a new collateral type with non-18 decimals would silently break all accounting with no on-chain revert.

Likelihood: **Low-Medium** (requires a governance action, but the code provides no guard against it and the protocol is designed to expand its asset list).

### Recommendation

Normalize all asset amounts to 18-decimal precision before arithmetic. Retrieve the asset's decimals at deposit/accounting time and scale accordingly:

```solidity
// In _getTotalEthInProtocol and getRsETHAmountToMint:
uint8 assetDecimals = IERC20Metadata(asset).decimals();
uint256 normalizedAmount = totalAssetAmt * (10 ** (18 - assetDecimals));
totalETHInProtocol += normalizedAmount.mulWad(assetER);
```

Alternatively, enforce in `_addNewSupportedAsset()` that only 18-decimal assets may be added:

```solidity
require(IERC20Metadata(asset).decimals() == 18, "Only 18-decimal assets supported");
```

### Proof of Concept

Assume a 6-decimal asset `TOKEN6` is added with an ETH price of `3e14` (0.0003 ETH per token), and the current `rsETHPrice` is `1e18`.

**Expected rsETH for depositing 1 TOKEN6 (= `1e6` raw units):**
```
rsethAmountToMint = (1e6 * 1e12) * 3e14 / 1e18 = 3e14   // ~0.0003 rsETH
```

**Actual rsETH minted (current code):**
```solidity
// LRTDepositPool.getRsETHAmountToMint
rsethAmountToMint = (1e6 * 3e14) / 1e18 = 3e2 = 300 wei  // ~0.0000000000003 rsETH
```

The depositor receives `1e12` times fewer rsETH than owed. Their `1e6` TOKEN6 (worth 0.0003 ETH) is credited as if it were worth `3e2 wei` of ETH — a loss of essentially 100% of deposited value. [6](#0-5) [7](#0-6) [8](#0-7)

        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

            }
        }
    }
