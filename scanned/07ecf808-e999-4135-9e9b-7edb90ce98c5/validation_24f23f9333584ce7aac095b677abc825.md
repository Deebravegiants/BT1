`highestRsethPrice` Advances During Paused State via `updateRSETHPriceAsManager()`, Desensitizing Downside-Protection Auto-Pause — (`contracts/LRTOracle.sol`)

### Summary

`updateRSETHPriceAsManager()` carries no `whenNotPaused` guard. When called while `LRTOracle.paused == true`, `_updateRsETHPrice()` correctly skips fee minting but unconditionally advances `highestRsethPrice` if the new computed price exceeds it. Because the downside-protection auto-pause threshold is derived from `highestRsethPrice`, raising it during a paused period permanently weakens the trigger for all future price-drop checks.

### Impact Explanation

No funds are directly lost. The weakened effect is that the downside-protection circuit breaker becomes less sensitive: a price drop that should have triggered an auto-pause (relative to the pre-pause peak) silently passes. This is a failure to deliver the promised safety guarantee — matching **Low: Contract fails to deliver promised returns, but doesn't lose value**.

However, the scenario arises from a plausible, non-malicious operational action: the manager calls `updateRSETHPriceAsManager()` while the protocol is paused (e.g., to refresh the price before unpausing), unaware that doing so permanently raises the auto-pause threshold. ### Recommendation

Skip the `highestRsethPrice` update when the protocol is paused, so the peak reference only advances during fully operational, fee-charging periods:

```solidity
// update highest price if new price exceeds it — only when protocol is active
if (!protocolPaused && newRsETHPrice > highestRsethPrice) {
    highestRsethPrice = newRsETHPrice;
}
```

Alternatively, add `whenNotPaused` to `updateRSETHPriceAsManager()`, though that would remove the manager's ability to update the price during a pause entirely.
