### Title
`LRTOracle.rsETHPrice` Is Never Set During Initialization, Causing Division-by-Zero in All Deposit Paths - (File: `contracts/LRTOracle.sol`, `contracts/LRTDepositPool.sol`)

---

### Summary

`LRTOracle.initialize()` never sets `rsETHPrice`, leaving it at its default value of `0`. Every deposit path in `LRTDepositPool` calls `getRsETHAmountToMint()`, which divides by `lrtOracle.rsETHPrice()`. When `rsETHPrice == 0`, Solidity 0.8's checked arithmetic triggers a division-by-zero panic, reverting every deposit until `updateRSETHPrice()` is explicitly called post-deployment.

---

### Finding Description

`LRTOracle.initialize()` only stores the config address and emits an event. It does not set `rsETHPrice`:

```solidity
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    lrtConfig = ILRTConfig(lrtConfigAddr);
    emit UpdatedLRTConfig(lrtConfigAddr);
}
``` [1](#0-0) 

`rsETHPrice` is a storage variable that defaults to `0`. The only mechanism that sets it is `_updateRsETHPrice()`, which is reached via the public `updateRSETHPrice()` call. That function handles the zero-supply bootstrap case:

```solidity
if (rsethSupply == 0) {
    rsETHPrice = 1 ether;
    highestRsethPrice = 1 ether;
    return;
}
``` [2](#0-1) 

But this bootstrap only executes if `updateRSETHPrice()` is called. If it is not called before the first deposit, `rsETHPrice` remains `0`.

Every deposit — both `depositETH()` and `depositAsset()` — flows through `_beforeDeposit()`, which calls `getRsETHAmountToMint()`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [3](#0-2) 

With `rsETHPrice == 0`, this line triggers a Solidity 0.8 division-by-zero panic, reverting the transaction. [4](#0-3) 

---

### Impact Explanation

All user-facing deposit functions (`depositETH`, `depositAsset`) are completely blocked until `updateRSETHPrice()` is called after deployment. No user funds can enter the protocol. This constitutes **temporary freezing of funds** (Medium severity per the allowed scope), since the condition is correctable by calling the public `updateRSETHPrice()` function, but the protocol is non-functional in the interim.

---

### Likelihood Explanation

The deployment sequence requires the deployer to manually call `updateRSETHPrice()` after `LRTOracle.initialize()` and before opening deposits. There is no on-chain enforcement of this ordering. The `initialize()` function gives no indication that a follow-up call is required. Any deployment that skips this step — or any upgrade that resets the oracle — leaves the protocol in a state where all deposits revert. The likelihood is **medium**: it is a realistic operational omission with no guard preventing it.

---

### Recommendation

Set `rsETHPrice` to `1 ether` inside `LRTOracle.initialize()` to match the bootstrap logic already present in `_updateRsETHPrice()`:

```solidity
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    lrtConfig = ILRTConfig(lrtConfigAddr);
    rsETHPrice = 1 ether;          // bootstrap: 1 rsETH = 1 ETH before any supply
    highestRsethPrice = 1 ether;
    emit UpdatedLRTConfig(lrtConfigAddr);
}
```

This mirrors the pattern in `_updateRsETHPrice()` and eliminates the dependency on a post-deployment manual call.

---

### Proof of Concept

1. Deploy `LRTOracle` proxy and call `initialize(lrtConfigAddr)`. `rsETHPrice` is `0`.
2. Deploy `LRTDepositPool` proxy and call `initialize(lrtConfigAddr)`. Do **not** call `updateRSETHPrice()`.
3. Any user calls `depositETH{value: 1 ether}(0, "")`.
4. Execution reaches `getRsETHAmountToMint(ETH_TOKEN, 1 ether)`.
5. `lrtOracle.rsETHPrice()` returns `0`.
6. `(1 ether * assetPrice) / 0` → Solidity panic (division by zero) → revert.
7. All deposits revert until an actor calls `LRTOracle.updateRSETHPrice()`. [5](#0-4) [6](#0-5)
