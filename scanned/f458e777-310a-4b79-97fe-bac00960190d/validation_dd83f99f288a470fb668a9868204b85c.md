Let me check if there's any staleness validation used elsewhere in the codebase, and look at how `rsETHOracle` is set up in the cross-chain context.

### Title
Stale Cross-Chain Rate in `CrossChainRateReceiver` Enables Over-Minting of wrsETH — (`contracts/cross-chain/CrossChainRateReceiver.sol`, `contracts/pools/RSETHPoolV3.sol`)

---

### Summary

`CrossChainRateReceiver.getRate()` returns the last received `rate` with no staleness check. `RSETHPoolV3.viewSwapRsETHAmountAndFee(amount, token)` divides a live Chainlink `tokenToETHRate` by this potentially stale `rsETHToETHrate`. When the cross-chain rate lags behind the true rsETH/ETH rate (e.g., during LayerZero bridge downtime), the formula over-mints wrsETH relative to the deposited token's actual rsETH value, leaving the pool undercollateralized.

---

### Finding Description

**Root cause — no staleness guard in `CrossChainRateReceiver.getRate()`:**

`CrossChainRateReceiver` stores `lastUpdated` when a rate arrives via `lzReceive`, but `getRate()` returns `rate` unconditionally: [1](#0-0) 

`lastUpdated` is written on receipt but is never read in any validation path anywhere in the contract hierarchy. `RSETHRateReceiver` adds no additional staleness logic: [2](#0-1) 

**Vulnerable formula in `RSETHPoolV3`:**

`viewSwapRsETHAmountAndFee(amount, token)` fetches both rates and computes:

```
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate
``` [3](#0-2) 

- `rsETHToETHrate` comes from `rsETHOracle` → `CrossChainRateReceiver.getRate()` → **no staleness check**
- `tokenToETHRate` comes from `ChainlinkOracleForRSETHPoolCollateral.getRate()` → Chainlink, updated on its own heartbeat [4](#0-3) 

**`ChainlinkOracleForRSETHPoolCollateral` has no time-based staleness check either**, only the `answeredInRound < roundID` guard, so Chainlink prices remain live under normal conditions: [5](#0-4) 

**The `deposit(token, amount, referralId)` path calls this formula directly and mints wrsETH:** [6](#0-5) 

---

### Impact Explanation

When the cross-chain rate is stale-low (e.g., `1.05e18`) while the true rsETH/ETH rate has risen (e.g., `1.10e18`) and Chainlink's wstETH/ETH rate is current (`1.15e18`):

| Variable | Stale scenario | Fair scenario |
|---|---|---|
| `tokenToETHRate` | 1.15e18 | 1.15e18 |
| `rsETHToETHrate` | 1.05e18 (stale) | 1.10e18 (correct) |
| `rsETHAmount` per 1 wstETH | ≈ 1.0952 rsETH | ≈ 1.0454 rsETH |
| Over-mint per wstETH | +0.0498 rsETH | — |

Each deposit mints ~4.8% excess wrsETH. The pool receives wstETH worth less rsETH than it mints. Repeated deposits up to the `dailyMintLimit` drain the pool's backing. The `dailyMintLimit` caps per-day damage but does not prevent the attack — it only bounds the rate of insolvency accumulation: [7](#0-6) 

**Impact: Critical — Protocol insolvency.** wrsETH becomes unbacked; redemptions by later users fail.

---

### Likelihood Explanation

- LayerZero bridge outages are historically documented and require no attacker action to trigger.
- rsETH naturally accrues yield over time, so its rate monotonically increases; any bridge gap creates a growing divergence.
- The attacker only needs to call `deposit(token, amount, referralId)` — a fully public, permissionless function — during a period of bridge lag.
- No privileged role, key compromise, or governance action is required.

---

### Recommendation

1. **Add a staleness check in `CrossChainRateReceiver.getRate()`** using the already-stored `lastUpdated`:

```solidity
uint256 public constant MAX_RATE_AGE = 24 hours; // configurable

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= MAX_RATE_AGE, "Rate is stale");
    return rate;
}
``` [8](#0-7) 

2. **Add a time-based staleness check in `ChainlinkOracleForRSETHPoolCollateral.getRate()`** (e.g., `block.timestamp - timestamp > heartbeat`). [9](#0-8) 

3. Consider pausing deposits automatically when the cross-chain rate is stale.

---

### Proof of Concept

Fork-test against a chain where `RSETHPoolV3` is deployed with `rsETHOracle = RSETHRateReceiver`:

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import "forge-std/Test.sol";

interface IRSETHPoolV3 {
    function deposit(address token, uint256 amount, string memory referralId) external;
    function viewSwapRsETHAmountAndFee(uint256 amount, address token)
        external view returns (uint256 rsETHAmount, uint256 fee);
}

interface IRSETHRateReceiver {
    function rate() external view returns (uint256);
    // owner-only, but we vm.prank the owner in the test
    function lzReceive(uint16, bytes memory, uint64, bytes calldata) external;
}

contract StaleRatePoC is Test {
    IRSETHPoolV3  pool          = IRSETHPoolV3(<POOL_ADDRESS>);
    IRSETHRateReceiver receiver = IRSETHRateReceiver(<RECEIVER_ADDRESS>);
    address wstETH              = <WSTETH_ADDRESS>;
    address receiverOwner       = <OWNER>;
    address lzEndpoint          = <LZ_ENDPOINT>;

    function testStaleRateOverMint() external {
        // 1. Simulate bridge lag: inject a stale low rate (1.00e18) into the receiver
        //    In production this happens naturally when lzReceive stops being called.
        //    Here we warp time forward without sending a new rate.
        uint256 staleRate = 1.00e18;
        // The receiver already holds this rate from a prior lzReceive; we just warp.
        vm.warp(block.timestamp + 2 days); // bridge silent for 2 days

        // 2. Chainlink wstETH/ETH rate is live at ~1.15e18 (read from fork)
        (uint256 rsETHAmountStale,) = pool.viewSwapRsETHAmountAndFee(1e18, wstETH);

        // 3. Compute fair rsETH amount using the true current rate (e.g. 1.10e18)
        uint256 trueRsETHRate = 1.10e18; // from L1 LRTOracle at fork block
        uint256 chainlinkRate = 1.15e18; // from Chainlink wstETH/ETH at fork block
        uint256 fairRsETHAmount = 1e18 * chainlinkRate / trueRsETHRate;

        // 4. Assert over-mint
        assertGt(rsETHAmountStale, fairRsETHAmount,
            "Stale rate should produce more rsETH than fair value");

        emit log_named_uint("rsETH minted (stale rate)", rsETHAmountStale);
        emit log_named_uint("rsETH fair value",          fairRsETHAmount);
        emit log_named_uint("excess rsETH",              rsETHAmountStale - fairRsETHAmount);
    }
}
```

The test demonstrates that `rsETHAmountStale > fairRsETHAmount`, confirming the pool mints unbacked wrsETH whenever the cross-chain rate lags behind the true rsETH/ETH rate.
