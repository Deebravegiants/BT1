### Title
Missing Slippage Protection in L2 Pool `deposit()` Functions — (File: `contracts/pools/RSETHPoolV3.sol`)

### Summary
All L2 RSETHPool `deposit()` variants compute the rsETH output amount from a live oracle rate at execution time but provide no minimum-output parameter, leaving depositors fully exposed to oracle rate changes between transaction submission and execution.

### Finding Description
The `deposit()` functions across every L2 pool variant — `RSETHPool`, `RSETHPoolV2`, `RSETHPoolV3`, `RSETHPoolNoWrapper`, `RSETHPoolV2ExternalBridge`, `RSETHPoolV3ExternalBridge`, and `RSETHPoolV3WithNativeChainBridge` — compute the rsETH amount to mint via `viewSwapRsETHAmountAndFee`, which reads the live oracle rate at execution time:

```solidity
// RSETHPoolV3.sol – deposit (ETH path)
function deposit(string memory referralId) external payable ... {
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    wrsETH.mint(msg.sender, rsETHAmount);          // no minimum check
}

function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (...) {
    uint256 rsETHToETHrate = getRate();            // live oracle read
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
```

There is no `minRSETHAmountExpected` parameter and no minimum-output guard. By contrast, the L1 `LRTDepositPool` explicitly enforces this protection:

```solidity
// LRTDepositPool.sol
function depositETH(uint256 minRSETHAmountExpected, ...) external payable {
    uint256 rsethAmountToMint = _beforeDeposit(..., minRSETHAmountExpected);
    ...
}
// _beforeDeposit:
if (rsethAmountToMint < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet();
```

The oracle rate (`rsETHOracle.getRate()`) is updated by the permissionless `updateRSETHPrice()` on L1, and the updated rate is pushed cross-chain. Any legitimate oracle update that increases the rsETH/ETH rate between a user's transaction submission and its on-chain execution will silently reduce the rsETH the user receives, with no revert path.

### Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A user who previews `viewSwapRsETHAmountAndFee` off-chain and then submits a deposit transaction may receive materially fewer rsETH tokens than the preview indicated if the oracle rate rises before execution. The user's ETH is consumed but the rsETH output is lower than expected, with no on-chain protection.

### Likelihood Explanation
**Medium.** `updateRSETHPrice()` is permissionless and is called regularly (by keepers or any party). Oracle rate increases are normal and expected as EigenLayer rewards accrue. Any deposit transaction pending in the mempool during an oracle update is affected. No special attacker capability is required — the loss occurs from ordinary protocol operation.

### Recommendation
Add a `minRSETHAmountExpected` parameter to all `deposit()` overloads in every L2 pool contract, mirroring the pattern already used in `LRTDepositPool`:

```solidity
function deposit(string memory referralId, uint256 minRSETHAmountExpected)
    external payable nonReentrant whenNotPaused ...
{
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(msg.value);
    if (rsETHAmount < minRSETHAmountExpected) revert SlippageExceeded();
    wrsETH.mint(msg.sender, rsETHAmount);
}
```

### Proof of Concept
1. User calls `viewSwapRsETHAmountAndFee(1 ether)` off-chain; oracle rate = 1.05 ETH/rsETH → preview shows ~0.952 rsETH (after fee).
2. User submits `deposit{value: 1 ether}(referralId)`.
3. Before the tx is mined, a keeper calls `updateRSETHPrice()` on L1, pushing a new rate of 1.10 ETH/rsETH to the L2 oracle.
4. User's tx executes with the new rate → receives ~0.909 rsETH instead of ~0.952 rsETH (~4.5% shortfall).
5. No revert occurs; user has no recourse.
