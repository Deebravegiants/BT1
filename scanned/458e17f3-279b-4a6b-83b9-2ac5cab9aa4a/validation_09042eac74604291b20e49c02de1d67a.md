Looking at the actual code to trace the full call chain and verify whether the two daily mint limits are truly shared or separate.

The code confirms the vulnerability is real. Here is the full analysis:

---

### Title
User Deposits Exhaust Shared RSETH Daily Mint Cap, Permanently Freezing Protocol Fee Yield — (`contracts/RSETH.sol`, `contracts/LRTOracle.sol`)

### Summary

`RSETH.mint()` applies the `checkDailyMintLimit` modifier uniformly to **all** callers. Both user-deposit minting (via `LRTDepositPool._mintRsETH`) and protocol-fee minting (via `LRTOracle._updateRsETHPrice`) draw from the same `maxMintAmountPerDay` bucket in `RSETH.sol`. There is no priority ordering or reserved allocation for fee minting. A depositor who fills the daily cap causes every subsequent `updateRSETHPrice()` call to revert for the remainder of the 24-hour period, freezing unclaimed protocol fee yield and blocking the rsETH price update.

### Finding Description

**Two separate daily limits exist, but fee minting must pass both:**

| Contract | Variable | Tracks |
|---|---|---|
| `RSETH.sol` | `maxMintAmountPerDay` / `currentPeriodMintedAmount` | ALL rsETH minted (users + fees) |
| `LRTOracle.sol` | `maxFeeMintAmountPerDay` / `currentPeriodMintedFeeAmount` | Fee rsETH only |

The fee-mint path in `_updateRsETHPrice()` is:

```
_checkAndUpdateDailyFeeMintLimit(feeAmount)   // LRTOracle's own cap — passes
IRSETH(rsETHTokenAddress).mint(treasury, feeAmount)  // hits RSETH.checkDailyMintLimit — REVERTS
``` [1](#0-0) 

`RSETH.mint()` applies `checkDailyMintLimit` to every caller without exception: [2](#0-1) 

The modifier checks and increments the shared counter: [3](#0-2) 

When a user deposit fills `currentPeriodMintedAmount` to `maxMintAmountPerDay`, the condition on line 50 fires for any subsequent `mint()` call — including the treasury fee mint — reverting the entire `_updateRsETHPrice()` execution before `rsETHPrice = newRsETHPrice` (line 313) is reached. [4](#0-3) 

### Impact Explanation

- Protocol fee yield is not minted for the remainder of the 24-hour period.
- `rsETHPrice` is not updated (the assignment at line 313 is never reached), staling the oracle.
- Both effects are repeatable every period: an attacker (or even an organic large depositor) can fill the cap at the start of each new period.
- Matches **Medium — Permanent freezing of unclaimed yield** (repeatable per period = effectively permanent under sustained pressure).

### Likelihood Explanation

- `maxMintAmountPerDay` must be set to a finite non-zero value by the manager for the protocol to function (if it were 0, `0 + any_amount > 0` would revert all minting, breaking the protocol entirely).
- `depositETH` is permissionless; no role is required.
- `updateRSETHPrice()` is `public whenNotPaused` — anyone can call it and observe the revert.
- A single large depositor, or the natural accumulation of many depositors in a busy period, can exhaust the cap without any malicious intent. [5](#0-4) [6](#0-5) 

### Recommendation

Give fee minting a dedicated, non-shared allocation inside `RSETH.mint()`, or bypass the `checkDailyMintLimit` for callers holding a dedicated `FEE_MINTER_ROLE`. The simplest fix is to add a separate `mintFee` entry point in `RSETH` that is exempt from (or uses a separate counter for) the user-deposit daily cap, so that protocol fee minting can never be crowded out by user activity.

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Pseudocode for a Foundry fork/local test
function test_depositExhaustsCapBlocksFeeYield() public {
    // 1. Manager sets a finite daily mint cap
    vm.prank(manager);
    rsETH.setMaxMintAmountPerDay(1000 ether);

    // 2. Depositor fills the cap exactly
    //    (deposit amount chosen so rsethAmountToMint == maxMintAmountPerDay)
    uint256 depositAmt = _ethAmountThatMintsExactly(1000 ether);
    vm.deal(attacker, depositAmt);
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: depositAmt}(0, "");

    // Assert cap is full
    assertEq(rsETH.currentPeriodMintedAmount(), 1000 ether);

    // 3. Simulate staking rewards accruing (TVL increases)
    _simulateRewards(10 ether);

    // 4. Anyone calls updateRSETHPrice() — reverts because fee mint
    //    hits checkDailyMintLimit in RSETH.mint()
    vm.expectRevert(
        abi.encodeWithSelector(RSETH.DailyMintLimitExceeded.selector, ...)
    );
    lrtOracle.updateRSETHPrice();

    // 5. rsETHPrice is stale; treasury received 0 fee rsETH
    assertEq(rsETH.balanceOf(treasury), 0);
}
```

The test is locally reproducible on unmodified contracts with no admin compromise required — only a sufficiently large (or aggregated) deposit within a single 24-hour period.

    modifier checkDailyMintLimit(uint256 amount) {
        // Check if we need to reset the period if it has been more than 24 hours
        if (block.timestamp >= periodStartTime + 1 days) {
            currentPeriodMintedAmount = 0;
            periodStartTime = getCurrentPeriodStartTime();
        }

        // Check if minting would exceed the daily limit
        if (currentPeriodMintedAmount + amount > maxMintAmountPerDay) {
            revert DailyMintLimitExceeded(currentPeriodMintedAmount + amount, maxMintAmountPerDay);
        }

        currentPeriodMintedAmount += amount;
        _;
    }
