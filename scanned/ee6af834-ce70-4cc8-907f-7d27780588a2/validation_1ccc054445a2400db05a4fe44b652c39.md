### Title
Minimum-Progress Rule + Proportional Scaling Desynchronize `curPosInBin` from `token0BalanceScaled`, Stranding LP Token0 and Causing Zero-Output Swaps — (`metric-core/contracts/libraries/SwapMath.sol`)

---

### Summary

When `curPosInBin = MAX_POS_BIN - 1` and a trader submits a tiny exact-input buy-token0 swap (`zeroForOne = false`, `amountSpecified = 1`), two independent rounding decisions in `computeAnalyticalTargetPosForBuyToken0` and the proportional scaling correction in `buyToken0InBinSpecifiedIn` diverge: the cursor is forced to `MAX_POS_BIN` while `out0Scaled` floors to zero. The pool consumes the trader's token1 input, outputs zero token0, and permanently marks the bin as exhausted for future buy-token0 swaps — even though the bin's full token0 balance remains intact.

---

### Finding Description

**Rounding desynchronization 1 — price at current position**

Inside `computeAnalyticalTargetPosForBuyToken0`, the price at the current bin position is computed with **floor** rounding:

```solidity
uint256 priceAtCurrPosX64 = lowerPriceX64 + (deltaPriceX64 * currBinPos) / MAX_POS_BIN;
``` [1](#0-0) 

The caller `buyToken0InBinSpecifiedIn` computes the same price with **ceil** rounding:

```solidity
uint256 startingPriceX64 =
    calculatePriceAtBinPosition(lowerPriceX64, upperPriceX64, currBinPos, Math.Rounding.Ceil);
``` [2](#0-1) 

The analytical target therefore uses a slightly lower price than the actual swap math, making it slightly over-optimistic about how far the input can move the cursor.

**Rounding desynchronization 2 — minimum progress rule forces `targetPos = MAX_POS_BIN`**

When the quadratic formula rounds `deltaPos` to zero for a tiny input, the minimum-progress guard unconditionally sets `deltaPos = 1`:

```solidity
if (deltaPos == 0 && inputAmount > 0) deltaPos = 1;
``` [3](#0-2) 

With `currBinPos = MAX_POS_BIN - 1`, `maxSubCurrPos = 1`, and `deltaPos = 1`, the clamp `if (deltaPos > maxSubCurrPos)` does **not** fire (equal, not greater), so `targetPos = MAX_POS_BIN`. [4](#0-3) 

**Rounding desynchronization 3 — proportional scaling correction**

Back in `buyToken0InBinSpecifiedIn`, `out0Scaled` is set to the full bin balance (moving from `MAX_POS_BIN - 1` to `MAX_POS_BIN` consumes all remaining token0), and `totalIn1Scaled` is computed for that full output — a large number. The scaling correction then fires:

```solidity
if (totalIn1Scaled > state.amountSpecifiedRemainingScaled) {
    uint256 delta = targetPos - currBinPos;                          // = 1
    uint256 scaledDelta = Math.ceilDiv(delta * state.amountSpecifiedRemainingScaled, totalIn1Scaled);
    if (scaledDelta == 0) scaledDelta = 1;
    targetPos = currBinPos + scaledDelta;                            // stays MAX_POS_BIN

    out0Scaled = (out0Scaled * state.amountSpecifiedRemainingScaled) / totalIn1Scaled;  // floors to 0
    totalIn1Scaled = state.amountSpecifiedRemainingScaled;
}
``` [5](#0-4) 

- `delta = 1`, so `scaledDelta = ceilDiv(1 × 1, large) = 1` → `targetPos` stays at `MAX_POS_BIN`.
- `out0Scaled = (token0Balance × 1) / large_totalIn1 = 0` (floor division, rounds to zero when `token0Balance < totalIn1Scaled`, which holds whenever `price > 1` in internal units — the common case for e.g. ETH/USDC).

**Resulting state inconsistency**

After `_finalizeSwap`, `curPosInBin = MAX_POS_BIN` is persisted: [6](#0-5) 

But `binState.token0BalanceScaled -= 0`, so the bin retains its full token0 balance. [7](#0-6) 

Every subsequent buy-token0 swap checks:

```solidity
if (binState.token0BalanceScaled == 0 || curPosInBinCache >= type(uint104).max) {
``` [8](#0-7) 

`curPosInBinCache = MAX_POS_BIN = type(uint104).max` satisfies the second condition, so the bin is treated as exhausted and skipped — even though it holds its full token0 balance.

### Impact Explanation

1. **Swap conservation failure**: The trader pays 1 unit of scaled token1 and receives 0 token0. The pool accepts the input and emits no output.
2. **Stranded LP token0**: The bin's entire token0 balance becomes inaccessible to all future buy-token0 swaps. LPs can recover it only via `removeLiquidity`, not through normal trading.
3. **Bad-price execution for subsequent traders**: Future buy-token0 swaps skip the stranded bin and fill from the next (higher-priced) bin, paying a worse price than the pool's actual liquidity would permit.

---

### Likelihood Explanation

- Requires `curPosInBin = MAX_POS_BIN - 1`, which is a natural state reached by any buy-token0 swap that nearly exhausts a bin.
- The attacker then calls `swap(recipient, false, 1, type(uint128).max, ...)` — a single 1-wei exact-input swap.
- No privileged access, no special token behavior, no oracle manipulation required.
- The condition `token0Balance < totalIn1Scaled` (needed for `out0Scaled = 0`) holds whenever the token0 price exceeds 1 in internal units, which is the common case for most production pools.

---

### Recommendation

1. **Remove or bound the minimum-progress rule**: Instead of unconditionally forcing `deltaPos = 1`, verify that the implied `out0Scaled` for a 1-step advance is non-zero before applying it. If `out0Scaled` would be zero, return early with no swap rather than advancing the cursor.
2. **Synchronize rounding**: Use the same rounding direction (floor or ceil) for `priceAtCurrPosX64` in `computeAnalyticalTargetPosForBuyToken0` and `startingPriceX64` in `buyToken0InBinSpecifiedIn`.
3. **Guard against zero output**: After the scaling correction, if `out0Scaled == 0`, revert or return `(currBinPos, 0, 0, 0, 0)` rather than consuming input and advancing the cursor.

### Proof of Concept

**Setup**:
- Pool with token0 (e.g. WETH, 18 decimals) and token1 (e.g. USDC, 6 decimals, scaled to 18 internally).
- `upperPriceX64 > ONE_X64` (price of token0 > 1 in internal units — always true for ETH/USDC).
- `curPosInBin = MAX_POS_BIN - 1` (reached naturally by a prior buy-token0 swap).
- `binState.token0BalanceScaled = T` (any non-zero value).

**Attack**:
```
swap(attacker, zeroForOne=false, amountSpecified=1, priceLimitX64=type(uint128).max, ...)
```

**Trace**:
1. `amountInScaled = TOKEN_1_SCALE_MULTIPLIER * 1 = 1` (or `TOKEN_1_SCALE_MULTIPLIER` if multiplier > 1).
2. `computeAnalyticalTargetPosForBuyToken0(MAX_POS_BIN-1, MAX_POS_BIN, 1, T, ...)`:
   - Quadratic gives `deltaPos = 0` → minimum progress → `deltaPos = 1` → `targetPos = MAX_POS_BIN`.
3. `out0Scaled = T * 1 / 1 = T` (full bin balance).
4. `totalIn1Scaled = ceilDiv(T * avgPrice * onePlusFee, ONE_X64²) >> 1`.
5. Scaling correction: `scaledDelta = ceilDiv(1 * 1, large) = 1` → `targetPos = MAX_POS_BIN` (unchanged); `out0Scaled = (T * 1) / large = 0`.
6. Pool stores `curPosInBin = MAX_POS_BIN`, `token0BalanceScaled = T` (unchanged).
7. Trader receives 0 token0, pays 1 unit of token1.
8. All subsequent buy-token0 swaps skip this bin (cursor at `type(uint104).max`), even though it holds `T` units of token0.

**Assertion failures**:
- `out0Scaled == 0` while `totalIn1Scaled == 1 > 0` — conservation violated.
- `curPosInBin == MAX_POS_BIN` while `token0BalanceScaled == T > 0` — cursor/balance invariant violated.
- Next buy-token0 swap fills from a higher-priced bin — bad-price execution.
