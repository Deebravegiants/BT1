### Title
`setPoolAdminFeeDestination` Fails to Collect Accrued Fees Before Changing Destination, Causing Old Destination to Lose Owed Admin Fees - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary
`setPoolAdminFeeDestination` overwrites `poolAdminFeeDestination[pool]` without first flushing accrued spread-fee surplus to the old destination. All surplus tokens that accumulated under the old destination are swept to the new destination on the next `collectPoolFees` call, permanently diverting owed admin fees away from the original recipient.

### Finding Description
The factory exposes two fee-related setters for pool admins. `setPoolAdminFees` correctly calls `collectFees(…, poolAdminFeeDestination[pool])` before updating fee rates, ensuring the old destination receives all fees accrued at the old rates: [1](#0-0) 

`setPoolAdminFeeDestination`, however, directly overwrites `poolAdminFeeDestination[pool]` with no prior collection: [2](#0-1) 

`collectPoolFees` and every internal `collectFees` call read `poolAdminFeeDestination[pool]` at call time: [3](#0-2) 

The spread-fee surplus is computed inside `collectFees` as the live pool balance minus bin totals minus notional fees: [4](#0-3) 

This surplus accumulates continuously as swaps occur. Because the destination is read at collection time, any surplus that built up while the old destination was active will be sent to the new destination on the next collection. The old destination has no recourse.

The protocol documentation acknowledges the destination is a separate treasury entity ("Coordinate with treasury operations"), and the pool creation parameters treat `admin` and `adminFeeDestination` as distinct addresses: [5](#0-4) 

### Impact Explanation
The old admin fee destination loses all admin-share spread fees that accrued between the last `collectPoolFees` call and the `setPoolAdminFeeDestination` call. These are real ERC-20 token balances held by the pool as surplus. The new destination receives tokens it did not earn. The loss is irreversible once the next collection executes.

### Likelihood Explanation
The pool admin is a semi-trusted actor who legitimately calls `setPoolAdminFeeDestination` during treasury rotations. Unlike `setPoolAdminFees`, which enforces collection automatically, `setPoolAdminFeeDestination` provides no such guard. Any treasury rotation without a prior manual `collectPoolFees` call triggers the loss. The permissionless `collectPoolFees` can be front-run by a third party immediately after the destination change, making the race window exploitable.

### Recommendation
Add a fee collection step inside `setPoolAdminFeeDestination` before updating the destination, mirroring the pattern in `setPoolAdminFees`:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
+   // Flush accrued fees to the old destination before switching
+   PoolFeeConfig memory c = poolFeeConfig[pool];
+   IMetricOmmPoolCollectFees(pool).collectFees(
+       c.protocolSpreadFeeE6,
+       c.adminSpreadFeeE6,
+       c.protocolNotionalFeeE8,
+       c.adminNotionalFeeE8,
+       poolAdminFeeDestination[pool]
+   );
    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

### Proof of Concept
1. Pool is created with `adminFeeDestination = oldTreasury`.
2. Swaps occur; spread-fee surplus accumulates in the pool (`balance > binTotals + notionalFees`).
3. Pool admin calls `setPoolAdminFeeDestination(pool, newTreasury)` without first calling `collectPoolFees`.
4. `poolAdminFeeDestination[pool]` is now `newTreasury`.
5. Anyone (keeper, bot, or adversary) calls `collectPoolFees(pool)`.
6. `collectFees` computes the full accumulated surplus and sends the admin share to `newTreasury`.
7. `oldTreasury` receives nothing despite having earned the fees during steps 1–3.
