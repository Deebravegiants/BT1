### Title
Fallback at `buyToken0InBinSpecifiedIn` lines 620-622 charges full remaining input without updating `out0Scaled`, causing trader overpayment — (`metric-core/contracts/libraries/SwapMath.sol`)

---

### Summary

When both the analytical quadratic approximation and the single linear-rescaling refinement underestimate the required token1 input for a given `targetPos`, the fallback at lines 620-622 sets `totalIn1Scaled = state.amountSpecifiedRemainingScaled` (full remaining input) but leaves `out0Scaled` at the value computed for the under-estimated `targetPos`. The subsequent proportional-scaling guard at line 624 is then skipped because `totalIn1Scaled == remaining` (not `>`). The pool charges the trader the full remaining input while delivering only the token0 corresponding to the smaller, under-estimated position.

---

### Finding Description

The exact-input buy-token0 path in `buyToken0InBinSpecifiedIn` uses a two-stage approximation:

**Stage 1 — Analytical quadratic** (lines 581-596): `computeAnalyticalTargetPosForBuyToken0` solves the endpoint-average quadratic. The formula uses `priceAtCurrPosX64` with floor rounding (line 259) while the swap uses ceil rounding (line 548), introducing a systematic downward bias. After computing `out0Scaled` and `totalIn1Scaled` for this `targetPos`, if `totalIn1Scaled < remaining && targetPos < maxFinalBinPos`, the first refinement fires.

**Stage 1 refinement — Linear rescaling** (lines 598-618): `scaledDelta = ceil(delta * remaining / totalIn1Scaled)`. Because the required input is quadratic in position (both `out0Scaled` and `avgPrice` grow with `targetPos`), a linear rescaling of the position delta systematically underestimates the required input when the price gradient is steep. After recomputing `out0Scaled` and `totalIn1Scaled` for the new `targetPos`, if `scaledTarget <= maxFinalBinPos` (line 605 false branch), `targetPos` is still strictly less than `maxFinalBinPos`.

**Stage 2 fallback** (lines 620-622):
```solidity
if (totalIn1Scaled < state.amountSpecifiedRemainingScaled && targetPos < maxFinalBinPos) {
    totalIn1Scaled = state.amountSpecifiedRemainingScaled;   // ← charges full input
}                                                             // ← out0Scaled NOT updated
```

After this assignment `totalIn1Scaled == remaining`, so the proportional-scaling guard at line 624 (`totalIn1Scaled > remaining`) is FALSE and is skipped entirely. `out0Scaled` is never corrected.

Settlement (lines 639-644) then:
- Removes `out0Scaled` (under-estimated) from the bin's token0 balance
- Adds `totalIn1Scaled` (full remaining) to the bin's token1 balance
- Decrements `state.amountSpecifiedRemainingScaled` by the full remaining input
- Credits `state.amountCalculatedScaled` with the under-estimated `out0Scaled`

The excess token1 `(remaining − totalIn1Scaled_before_fallback)` is absorbed by the pool with no corresponding token0 output.

### Impact Explanation

The trader pays `state.amountSpecifiedRemainingScaled` (full remaining input) but receives only `out0Scaled` worth of token0, which corresponds to a `targetPos` strictly less than `maxFinalBinPos`. The excess token1 is credited to the pool's bin balance, permanently benefiting LPs at the trader's expense. This is a direct loss of user principal on every swap that triggers the fallback path. The magnitude scales with the gap between the two underestimates and the input size; with high `addFeeBuyE6` and a steep bin, the gap can be a material fraction of the swap amount.

### Likelihood Explanation

The fallback triggers when:
1. The analytical approximation underestimates `targetPos` (systematic due to floor vs. ceil rounding mismatch at line 259 vs. line 548, and the endpoint-average approximation error).
2. The linear rescaling also underestimates (guaranteed when the price gradient is large enough that the quadratic term dominates, i.e., `deltaPriceX64 / lowerPriceX64` is significant).
3. `scaledTarget <= maxFinalBinPos` (the linear rescaling does not overshoot to the bin boundary).

Conditions 1-3 are simultaneously achievable with any pool that has a non-trivial `addFeeBuyE6` and a bin whose price range spans more than a small fraction of `lowerPriceX64`. These are normal, permissionless pool parameters. No privileged actor is required. [4](#0-3) [5](#0-4) [6](#0-5)

### Recommendation

Replace the fallback at lines 620-622 with a correct resolution. The two sound options are:

**Option A — Consume the full bin:** If both approximations still underestimate and `targetPos < maxFinalBinPos`, the trader has enough input to reach `maxFinalBinPos`. Set `targetPos = maxFinalBinPos` and recompute `out0Scaled` and `totalIn1Scaled` exactly, then let the existing proportional-scaling guard at line 624 handle any overshoot.

**Option B — Accept the underestimate and let the swap continue to the next bin:** Remove the fallback entirely. If `totalIn1Scaled < remaining && targetPos < maxFinalBinPos`, the outer swap loop will advance to the next bin and consume the residual input there. This is the conservative, loss-free path.

Either option eliminates the state where `totalIn1Scaled` is inflated to `remaining` while `out0Scaled` is left at a stale, smaller value.

Apply the same fix symmetrically to `buyToken1InBinSpecifiedIn` lines 758-760. [7](#0-6)
