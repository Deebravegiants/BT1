### Title
Pool Admin Can Bypass OracleValueStopLoss Timelock by Pre-Staging Proposals When Timelock Is Zero — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

### Summary

`OracleValueStopLossExtension` computes `executeAfter` at proposal time using the **current** timelock value. When `timelock == 0`, every proposal receives `executeAfter = block.timestamp`, which `_requireElapsed` accepts immediately and forever. A pool admin can stage drawdown, decay, or watermark proposals while the timelock is zero, then increase the timelock to appear trustworthy, and still execute the pre-staged proposals instantly — bypassing the protection window LPs were promised.

### Finding Description

`_afterTimelock` reads the live timelock and adds it to `block.timestamp`:

```solidity
// OracleValueStopLossExtension.sol
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

When `timelock == 0` this returns exactly `block.timestamp`. Every `propose*` function calls `_afterTimelock` to stamp the pending record:

```solidity
function proposeOracleStopLossDrawdown(address pool_, uint256 newMaxDrawdownE6) external onlyPoolAdmin(pool_) {
    ...
    uint32 executeAfter = _afterTimelock(pool_);   // == block.timestamp when timelock==0
    sched.pendingDrawdownE6 = uint32(newMaxDrawdownE6);
    sched.pendingDrawdownExecuteAfter = executeAfter;
    ...
}
```

`_requireElapsed` uses a strict-less-than check:

```solidity
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(...);
}
```

Because `block.timestamp < block.timestamp` is always `false`, a proposal stamped with `executeAfter = block.timestamp` passes this guard in every future block — including blocks mined long after the timelock has been raised to a non-zero value.

**Attack sequence:**

1. Pool is initialized with `timelock = 0` (no delay).
2. Admin calls `proposeOracleStopLossDrawdown(pool, 0)` → `pendingDrawdownExecuteAfter = block.timestamp_A`.
3. Admin calls `proposeOracleStopLossTimelock(pool, 1 days)` → also stamped with `block.timestamp_A` (timelock still 0).
4. Admin calls `executeOracleStopLossTimelock(pool)` → timelock is now 1 day. LPs observe the change and believe future parameter updates will be delayed by 24 hours.
5. Admin calls `executeOracleStopLossDrawdown(pool)` in the very next block → `block.timestamp_B >= block.timestamp_A`, check passes, drawdown is set to 0 (stop-loss disabled) with zero warning to LPs.

The same pattern applies to `proposeOracleStopLossDecay` and `proposeOracleStopLossHighWatermarks`.

### Impact Explanation

The `OracleValueStopLossExtension` is the primary mechanism protecting LPs from value leakage: it blocks swaps that would push per-share value below the drawdown floor. Disabling it (drawdown → 0) or manipulating watermarks without the expected delay removes LP protection silently. LPs who monitor the timelock value and believe they have a 24-hour exit window are deceived; the admin can drain LP value through adversarial swaps immediately after executing the pre-staged parameter change.

### Likelihood Explanation

Any pool that launches with `timelock = 0` (a common bootstrap configuration, confirmed by the test `test_decayTimelockZeroExecutesImmediately`) is vulnerable. The admin needs only two transactions before raising the timelock. The attack is invisible on-chain until `execute*` is called, at which point LPs have no recourse.

### Recommendation

In every `propose*` function, revert when the current timelock is zero — mirroring the external report's recommendation to block timer-start when the cooldown duration is zero:

```solidity
function proposeOracleStopLossDrawdown(address pool_, uint256 newMaxDrawdownE6) external onlyPoolAdmin(pool_) {
    _validateDrawdown(newMaxDrawdownE6);
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
+   if (oracleStopLossConfig[pool_].timelock == 0) revert OracleStopLossTimelockIsZero(pool_);
    uint32 executeAfter = _afterTimelock(pool_);
    ...
}
```

Apply the same guard to `proposeOracleStopLossDecay`, `proposeOracleStopLossTimelock`, and `proposeOracleStopLossHighWatermarks`. Alternatively, when `timelock == 0`, execute the change immediately inside the `propose*` call rather than staging it, so no stale `executeAfter` can persist across a later timelock increase.
