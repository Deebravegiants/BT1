### Title
ETH Deposit Limit Bypass Due to Missing Amount in Overflow-Equivalent Limit Check — (File: contracts/LRTDepositPool.sol)

### Summary
`_checkIfDepositAmountExceedesCurrentLimit` in `LRTDepositPool.sol` omits the incoming deposit amount from the ETH branch of the limit check. Any depositor can push total ETH holdings arbitrarily above the configured cap in a single transaction, bypassing the protocol's primary deposit-size safety mechanism.

### Finding Description
The function applies two different arithmetic expressions depending on asset type:

```solidity
// contracts/LRTDepositPool.sol L676-L682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount absent
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct
}
```

For every ERC-20 LST the prospective deposit amount is added to the running total before comparing against the cap. For ETH the addition is absent: the guard only fires when the total *already* exceeds the limit. Consequently, while `totalAssetDeposits ≤ depositLimit`, a caller may supply an arbitrarily large `msg.value` and the check returns `false` (not exceeded), allowing the deposit to proceed and the total to overshoot the limit by any magnitude.

The companion view `getAssetCurrentLimit` correctly computes remaining capacity for all assets including ETH:

```solidity
// contracts/LRTDepositPool.sol L402-L409
function getAssetCurrentLimit(address asset) public view override returns (uint256) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)) {
        return 0;
    }
    return lrtConfig.depositLimitByAsset(asset) - totalAssetDeposits;
}
```

The inconsistency between these two functions confirms the omission is unintentional.

### Impact Explanation
The deposit limit is the protocol's primary on-chain control over how much ETH it accepts. Bypassing it allows the pool to accumulate ETH beyond what EigenLayer strategies can absorb, leaving the surplus idle in the deposit pool and not earning restaking yield. rsETH holders who deposited after the limit was breached receive tokens backed by non-yielding ETH, causing the protocol to fail to deliver the promised restaking returns. Classified as **Low — contract fails to deliver promised returns, but does not lose value**.

### Likelihood Explanation
The entry path is fully permissionless: `depositETH` is callable by any address with no role requirement. The condition that enables the bypass (`totalAssetDeposits ≤ depositLimit`) is the normal operating state of the protocol. Any depositor can exploit this at any time without coordination or special privileges.

### Recommendation
Add `amount` to the ETH branch to mirror the ERC-20 logic:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept
Assume `depositLimitByAsset(ETH) = 1 000 ETH` and `getTotalAssetDeposits(ETH) = 999 ETH`.

1. Alice calls `depositETH{value: 5000 ether}(0, "")`.
2. `_checkIfDepositAmountExceedesCurrentLimit(ETH, 5000e18)` evaluates `999e18 > 1000e18` → `false`.
3. `_beforeDeposit` does not revert; rsETH is minted for 5 000 ETH.
4. `getTotalAssetDeposits(ETH)` is now 6 000 ETH — 6× the intended cap — with no protocol-level rejection. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
