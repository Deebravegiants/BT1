### Title
`tokenFeeBps` Defaults to Zero for All Supported ERC20 Tokens, Causing Protocol Fee Revenue Loss - (File: `contracts/pools/RSETHPool.sol`)

### Summary
`RSETHPool.sol` (the Arbitrum L2 pool) introduces a per-token fee mapping `tokenFeeBps` for ERC20 token deposits. However, `addSupportedToken` never initializes this mapping, so `tokenFeeBps[token]` silently defaults to `0` for every supported token. Any depositor calling `deposit(token, amount, referralId)` pays zero fee, and the protocol collects no revenue from ERC20 token deposits unless the admin separately calls `setTokenFeeBps`.

### Finding Description
`RSETHPool.sol` declares a per-token fee mapping at line 88:

```solidity
mapping(address token => uint256 feeBps) public tokenFeeBps;
```

The fee calculation for token deposits in `viewSwapRsETHAmountAndFee(uint256 amount, address token)` reads directly from this mapping:

```solidity
uint256 feeBpsForToken = tokenFeeBps[token];
fee = amount * feeBpsForToken / 10_000;
```

Because Solidity mappings default to zero, `feeBpsForToken` is always `0` for any token unless `setTokenFeeBps` has been explicitly called. The `addSupportedToken` function ŌĆö the only path to register a new token ŌĆö never sets `tokenFeeBps`:

```solidity
function addSupportedToken(address token, address oracle, address bridge) external onlyRole(TIMELOCK_ROLE) {
    ...
    supportedTokenList.push(token);
    supportedTokenOracle[token] = oracle;
    tokenBridge[token] = bridge;
    // tokenFeeBps[token] is never set here
    emit AddSupportedToken(token, oracle, bridge);
}
```

The setter `setTokenFeeBps` is a completely separate, optional admin call. There is no enforcement that it must be called after `addSupportedToken`, and no atomic path to add a token with a non-zero fee in a single transaction. This contrasts with ETH deposits, which correctly use the global `feeBps` variable that is set at initialization.

This is the direct analog of the reported issue: just as the original protocol used a single hardcoded fee value calibrated for one token that was wrong for all others, `RSETHPool.sol` uses a per-token fee that is never initialized, making it effectively zero for every ERC20 token.

### Impact Explanation
Every ERC20 token deposit (e.g., wstETH on Arbitrum) through `RSETHPool.deposit(token, amount, referralId)` is processed with `fee = 0`. The protocol earns no fee revenue from token deposits. Depositors receive the full rsETH amount with no fee deducted, which is more than the protocol intends to give. This constitutes a contract failing to deliver its promised fee-collection behavior.

**Impact: Low** ŌĆö Contract fails to deliver promised returns (fee collection), but no user funds are lost.

### Likelihood Explanation
This affects every ERC20 token deposit on the Arbitrum pool from the moment a token is added until the admin separately calls `setTokenFeeBps`. Since the default state of the mapping is zero and there is no enforcement mechanism, this condition is the default for all tokens. Any unprivileged depositor can exploit this by depositing supported tokens immediately after they are listed, paying zero fees indefinitely if the admin oversight is missed.

### Recommendation
Add a `feeBps` parameter to `addSupportedToken` and set `tokenFeeBps[token]` atomically during token registration:

```solidity
function addSupportedToken(address token, address oracle, address bridge, uint256 _feeBps)
    external onlyRole(TIMELOCK_ROLE)
{
    ...
    if (_feeBps > 10_000) revert InvalidFeeAmount();
    tokenFeeBps[token] = _feeBps;
    ...
}
```

This mirrors how the original report recommends making the fee dynamic per token, ensuring no token can be added without an explicit fee assignment.

### Proof of Concept

1. Admin calls `addSupportedToken(wstETH, oracle, bridge)` ŌĆö `tokenFeeBps[wstETH]` remains `0`.
2. Any user calls `deposit(wstETH, 1e18, "ref")`.
3. `viewSwapRsETHAmountAndFee(1e18, wstETH)` computes:
   - `feeBpsForToken = tokenFeeBps[wstETH]` ŌåÆ `0`

   - `amountAfterFee = 1e18`
4. User receives full rsETH with zero fee deducted; `feeEarnedInToken[wstETH]` stays `0`.
