### Title
Stale Cached Rate in `CrossChainRateReceiver.getRate()` Enables Oracle Drift Arbitrage on L2 Deposits — (File: `contracts/cross-chain/CrossChainRateReceiver.sol`)

---

### Summary

The `CrossChainRateReceiver` stores the rsETH/ETH rate received via LayerZero in a cached `rate` variable. The `getRate()` function returns this cached value with no staleness check, despite the contract tracking `lastUpdated`. L2 deposit pools (`RSETHPoolV3`, `RSETHPoolNoWrapper`) use this oracle to price every deposit. When the true rsETH/ETH rate has risen on L1 but the cross-chain update has not yet been relayed, users can deposit ETH/LST at the stale under-priced rate and receive more rsETH than the true exchange rate warrants, extracting value from the protocol at the expense of existing rsETH holders.

---

### Finding Description

`CrossChainRateReceiver` stores the last received rate and a `lastUpdated` timestamp, but `getRate()` returns the cached `rate` unconditionally:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol
uint256 public rate;
uint256 public lastUpdated;   // stored but never checked

function getRate() external view returns (uint256) {
    return rate;              // no staleness guard
}
```

The rate is only refreshed when `lzReceive()` is called by the LayerZero endpoint, which in turn depends on `RSETHMultiChainRateProvider.getLatestRate()` being triggered on L1:

```solidity
// contracts/cross-chain/RSETHMultiChainRateProvider.sol
function getLatestRate() public view override returns (uint256) {
    return ILRTOracle(rsETHPriceOracle).rsETHPrice();  // cached L1 price
}
```

`LRTOracle.rsETHPrice` is itself a cached value updated only when `updateRSETHPrice()` is called. There are therefore two compounding staleness layers before the rate reaches the L2 pool.

Both `RSETHPoolV3` and `RSETHPoolNoWrapper` call `getRate()` directly to price every deposit:

```solidity
// contracts/pools/RSETHPoolV3.sol
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();          // stale rate
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
```

If `rsETHToETHrate` is stale-low (true rate has grown due to staking rewards), `rsETHAmount` is inflated. The deposit fee (`feeBps`) is the only protection against this drift, but:

1. `feeBps` can be set to `0` by `DEFAULT_ADMIN_ROLE` (no lower bound enforced in `RSETHPoolV3.setFeeBps`).
2. Even at non-zero `feeBps`, if the accumulated drift since the last LayerZero update exceeds `feeBps`, the arbitrage is profitable.

The analogous protection in the reference report is the Redemption Base Fee (50 BPS). Here, `feeBps` plays the same role, but the protocol provides no on-chain guarantee that `feeBps ≥ maximum possible drift`, and there is no maximum-staleness revert in `getRate()`.

---

### Impact Explanation

**High — Theft of unclaimed yield.**

When a user deposits at a stale-low rate they receive more wrsETH/rsETH than the true exchange rate entitles them to. The excess rsETH is effectively extracted from the yield that should accrue to existing holders. The ETH deposited is eventually bridged to L1 and deposited at the true rate, creating a shortfall: the pool has issued more rsETH claims than the underlying ETH justifies. Existing rsETH holders bear the dilution.

---

### Likelihood Explanation

**Medium.**

rsETH/ETH appreciates continuously (~4% APY ≈ 0.011%/day). The LayerZero update is not automatic; it must be triggered by an off-chain keeper calling the rate provider. Any gap between updates (hours to days) accumulates drift. If `feeBps` is low (e.g., 5–10 BPS, which is common for competitive L2 pools) or zero, even a few hours of drift makes the arbitrage profitable. A sophisticated MEV searcher monitoring the `lastUpdated` timestamp and the L1 `rsETHPrice` can time deposits precisely.

---

### Recommendation

1. Add a maximum-staleness check in `CrossChainRateReceiver.getRate()`:
   ```solidity
   uint256 public maxStaleness; // e.g., 24 hours

   function getRate() external view returns (uint256) {
       require(block.timestamp - lastUpdated <= maxStaleness, "Rate stale");
       return rate;
   }
   ```
2. Ensure `feeBps` has a minimum floor that is at least as large as the maximum expected drift over `maxStaleness` (analogous to the Redemption Base Fee recommendation in the reference report).
3. Consider enforcing a non-zero lower bound on `feeBps` in `setFeeBps`.

---

### Proof of Concept

1. At time T, L1 `rsETHPrice = 1.040 ETH/rsETH`. LayerZero relays this to L2; `CrossChainRateReceiver.rate = 1.040e18`, `lastUpdated = T`.
2. 48 hours pass. Staking rewards accrue; true L1 `rsETHPrice` rises to `1.041 ETH/rsETH`. No LayerZero update is triggered.
3. Attacker calls `RSETHPoolV3.deposit{value: 100 ETH}("")` on L2.
   - `rsETHToETHrate = getRate() = 1.040e18` (stale)
   - `feeBps = 5` (0.05%)
   - `fee = 0.05 ETH`, `amountAfterFee = 99.95 ETH`
   - `rsETHAmount = 99.95e18 / 1.040e18 = 96.106 wrsETH`
4. True value of 96.106 wrsETH at the real rate: `96.106 × 1.041 = 100.07 ETH`.
5. Attacker's profit: `100.07 − 100 = 0.07 ETH` (net of fee: `0.07 − 0.05 = 0.02 ETH`).
6. The 100 ETH is bridged to L1 and deposited at the true rate, but the pool has already issued 96.106 wrsETH worth 100.07 ETH — the 0.07 ETH excess is diluted from existing holders. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
