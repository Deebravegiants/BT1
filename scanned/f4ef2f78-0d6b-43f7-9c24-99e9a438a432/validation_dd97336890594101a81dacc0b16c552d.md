### Title
Missing Zero-Rate Validation in Token Deposit Path Silently Mints Zero wrsETH to Depositor — (File: contracts/pools/RSETHPoolV3.sol)

### Summary
`viewSwapRsETHAmountAndFee(uint256 amount, address token)` does not validate that the token oracle's `getRate()` return is non-zero before using it in the rsETH amount calculation. If the oracle returns 0, the deposit proceeds silently: the user's tokens are transferred to the pool and 0 wrsETH is minted. The sibling function `viewSwapAssetToPremintedRsETH()` correctly guards against this with explicit zero-rate checks, exposing an inconsistency that is the direct analog of the reported sentinel-value mishandling pattern.

### Finding Description
The `deposit(address token, uint256 amount, string memory referralId)` function calls `viewSwapRsETHAmountAndFee(amount, token)` to compute the wrsETH amount to mint:

```solidity
function viewSwapRsETHAmountAndFee(uint256 amount, address token)
    public view onlySupportedToken(token)
    returns (uint256 rsETHAmount, uint256 fee)
{
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();
    uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate(); // ← no zero check
    rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
}
``` [1](#0-0) 

If `tokenToETHRate` is 0, `rsETHAmount` evaluates to 0. The `deposit()` function then:
1. Transfers the user's tokens to the pool via `safeTransferFrom`
2. Adds the fee to `feeEarnedInToken[token]`
3. Calls `wrsETH.mint(msg.sender, 0)` — minting nothing to the user [2](#0-1) 

The user's tokens are permanently locked in the pool with no user-accessible recovery mechanism.

By contrast, `viewSwapAssetToPremintedRsETH()` explicitly guards against this exact condition:

```solidity
uint256 rsETHToETHrate = getRate();
if (rsETHToETHrate == 0) revert UnsupportedOracle();
uint256 tokenToETHRate = token == ETH_IDENTIFIER ? 1e18 : IOracle(supportedTokenOracle[token]).getRate();
if (tokenToETHRate == 0) revert UnsupportedOracle();
``` [3](#0-2) 

The `addSupportedToken` function validates `getRate() != 0` only at setup time:

```solidity
if (IOracle(oracle).getRate() == 0) {
    revert UnsupportedOracle();
}
``` [4](#0-3) 

This provides no protection at deposit time if the oracle subsequently returns 0. The `limitDailyMint` modifier also calls `viewSwapRsETHAmountAndFee`, so a 0 `rsETHAmount` additionally bypasses the daily mint limit accounting (`dailyMintAmount += 0`). [5](#0-4) 

### Impact Explanation
A user who deposits a supported token when the token oracle returns 0 loses their entire deposit. The tokens are transferred to the pool, a fee is recorded against the user's deposit, and 0 wrsETH is minted. The user has no on-chain mechanism to recover the deposited tokens. This constitutes permanent freezing and effective loss of user funds — matching the **Critical: Permanent freezing of funds** impact tier.

### Likelihood Explanation
The oracle returning 0 after initial setup is a realistic failure mode: oracles can malfunction, be deprecated, or have their underlying data source return 0 (e.g., a Chainlink feed going stale or being sunset). The `addSupportedToken` guard only validates at setup time. Any subsequent oracle failure that causes `getRate()` to return 0 triggers this path silently, with no revert to protect the depositor.

### Recommendation
Add zero-rate validation in `viewSwapRsETHAmountAndFee(uint256 amount, address token)`, mirroring the guard already present in `viewSwapAssetToPremintedRsETH()`:

```solidity
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
if (tokenToETHRate == 0) revert UnsupportedOracle();
```

Similarly, validate `rsETHToETHrate != 0` (currently only a division-by-zero revert protects this path).

### Proof of Concept
1. A token oracle for a supported token (e.g., wstETH) begins returning 0 due to oracle malfunction or deprecation.
2. A user calls `deposit(wstETH, 1e18, "ref")`.
3. `limitDailyMint` calls `viewSwapRsETHAmountAndFee(1e18, wstETH)` → `rsETHAmount = 1e18 * 0 / rsETHToETHrate = 0`; `dailyMintAmount += 0` (limit bypass).
4. Inside `deposit`: `safeTransferFrom` moves 1e18 wstETH from the user to the pool.
5. `viewSwapRsETHAmountAndFee` again returns `rsETHAmount = 0`.
6. `wrsETH.mint(msg.sender, 0)` is called — user receives 0 wrsETH.
7. User's 1e18 wstETH is permanently locked in the pool with no recovery path.
