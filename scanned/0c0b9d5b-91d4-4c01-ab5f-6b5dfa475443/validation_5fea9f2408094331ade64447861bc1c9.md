### Title
Stale `assetsCommitted` Accounting After EigenLayer Slashing Temporarily Blocks All New `initiateWithdrawal` Calls - (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary

`assetsCommitted[asset]` is a monotonically-increasing accumulator that is only reduced when `unlockQueue` processes queued requests. `getAvailableAssetAmount` computes availability as `getTotalAssetDeposits(asset) - assetsCommitted[asset]`. Because `getTotalAssetDeposits` is a live view that reflects EigenLayer's current withdrawable shares (which decrease under slashing), a slashing event can push `totalAssets` below the already-committed amount, causing `getAvailableAssetAmount` to return 0 and every subsequent `initiateWithdrawal` call to revert with `ExceedAmountToWithdraw`.

---

### Finding Description

**`getAvailableAssetAmount` (line 599–603):**

```solidity
function getAvailableAssetAmount(address asset) public view override returns (uint256 availableAssetAmount) {
    ILRTDepositPool lrtDepositPool = ILRTDepositPool(lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL));
    uint256 totalAssets = lrtDepositPool.getTotalAssetDeposits(asset);
    availableAssetAmount = totalAssets > assetsCommitted[asset] ? totalAssets - assetsCommitted[asset] : 0;
}
``` [1](#0-0) 

**`getTotalAssetDeposits` (line 385–397)** sums live on-chain balances including `assetStakedInEigenLayer` (from `NodeDelegatorHelper.getAssetBalance` → `getWithdrawableShare` on EigenLayer's `DelegationManager`) and `assetUnstakingFromEigenLayer` (from `getAssetUnstaking` → `sharesToUnderlyingView`). Both values reflect EigenLayer's current slashing factor and can decrease at any time without any on-chain action by the protocol. [2](#0-1) [3](#0-2) 

**`assetsCommitted` lifecycle:**
- Increased at `initiateWithdrawal` line 173: `assetsCommitted[asset] += expectedAssetAmount;`
- Decreased only inside `_unlockWithdrawalRequests` line 802: `assetsCommitted[asset] -= request.expectedAssetAmount;` [4](#0-3) [5](#0-4) 

There is no mechanism that adjusts `assetsCommitted` downward when `totalAssets` decreases. The two values are therefore decoupled after the initial commitment.

**`initiateWithdrawal` guard (line 170):**

```solidity
if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();
``` [6](#0-5) 

Once `assetsCommitted[asset] >= totalAssets`, `getAvailableAssetAmount` returns 0, and every call to `initiateWithdrawal` reverts regardless of how small the requested amount is.

---

### Impact Explanation

All new withdrawal initiations for the affected asset are blocked. Users holding rsETH cannot queue a withdrawal through the normal path. The freeze persists until `unlockQueue` is called by an operator, which processes existing queued requests and reduces `assetsCommitted`. The freeze is therefore **temporary** but operator-dependent: if operators are slow or the queue is long, the freeze can last for the full `withdrawalDelayBlocks` window (~8 days by default). [7](#0-6) 

This matches the allowed impact: **Medium — Temporary freezing of funds**.

---

### Likelihood Explanation

EigenLayer's slashing mechanism is live and can reduce `getWithdrawableShare` for any strategy. The protocol explicitly integrates with EigenLayer across multiple `NodeDelegator` contracts. A slashing event that reduces `totalAssets` by even a small percentage (e.g., 5–10%) while `assetsCommitted` is near the previous `totalAssets` ceiling is sufficient to trigger the freeze.

The `LRTOracle` downside-protection auto-pause fires only when the price drop exceeds `pricePercentageLimit`:

```solidity
if (isPriceDecreaseOffLimit) {
    if (!lrtDepositPool.paused()) lrtDepositPool.pause();
    if (!withdrawalManager.paused()) withdrawalManager.pause();
    _pause();
    return;
}
``` [8](#0-7) 

For slashing events below that threshold, the protocol remains unpaused and the broken accounting is exposed. The root cause is the protocol's own accounting design (static accumulator vs. live view), not solely the external slashing event.

---

### Recommendation

1. **Clamp `assetsCommitted` to `totalAssets` inside `getAvailableAssetAmount`**: if `assetsCommitted[asset] > totalAssets`, treat the effective committed amount as `totalAssets` (available = 0) but do not allow the underflow to propagate into future accounting.
2. **Add a reconciliation step**: when `unlockQueue` is called, if `assetsCommitted[asset] > totalAssets`, cap `assetsCommitted[asset]` to `totalAssets` before processing the queue, so the accounting self-heals.
3. **Alternatively**, track `assetsCommitted` as a share of `totalAssets` at commitment time rather than an absolute amount, so it scales proportionally with slashing.

---

### Proof of Concept

```solidity
// Preconditions (fork test or local mock):
// 1. totalAssets for ETH = 100 ether (via EigenLayer strategy)
// 2. Multiple users call initiateWithdrawal until assetsCommitted[ETH] = 95 ether
//    (each call passes the check: expectedAmount <= 100 - committed)

// Trigger:
// 3. EigenLayer slashing reduces getWithdrawableShare by 10%
//    → getTotalAssetDeposits(ETH) now returns 90 ether
//    → assetsCommitted[ETH] = 95 ether > 90 ether

// Assertion:
// 4. getAvailableAssetAmount(ETH) == 0  ✓ (saturating subtraction)
// 5. Any initiateWithdrawal(ETH, minAmount, "") reverts ExceedAmountToWithdraw ✓

// Resolution:
// 6. Operator calls unlockQueue(ETH, ...) → assetsCommitted[ETH] -= unlockedAmount
//    → getAvailableAssetAmount(ETH) > 0 again → initiateWithdrawal succeeds
```

The invariant `getAvailableAssetAmount(asset) > 0` when `totalAssets > 0` and `rsETH supply > 0` is violated between step 3 and step 6, which can span the full `withdrawalDelayBlocks` (~57,600 blocks / ~8 days). [1](#0-0) [9](#0-8) [10](#0-9)
