### Title
Truncated Hash UID Collision in `bridgeTokenToL1` Causes Bridge Transaction Failure ‚Äî (`File: contracts/bridges/SonicChainNativeTokenBridge.sol`)

---

### Summary

`SonicChainNativeTokenBridge.bridgeTokenToL1` generates a `uint96` withdrawal UID by hashing block-scoped inputs and truncating the result. Two transactions sharing the same `msg.sender`, `recipient`, `amount`, and `tx.gasprice` within the same block produce an identical UID. When the Sonic bridge enforces UID uniqueness per caller, the second call reverts after the user's tokens have already been pulled ‚Äî temporarily freezing the bridge path for that user.

---

### Finding Description

The UID is computed as:

```solidity
uint96 uid = uint96(
    uint256(
        keccak256(
            abi.encodePacked(
                block.timestamp, block.number, msg.sender, recipient, amount, tx.gasprice, bridgeReceiver
            )
        )
    ) % type(uint96).max   // ‚Üê should be % (type(uint96).max + 1)
);
``` [1](#0-0) 

Three compounding defects mirror the reported vulnerability class:

1. **Entropy truncation**: The full 256-bit keccak256 output is reduced to 96 bits, shrinking the collision-resistant space from `2^256` to `2^96`.
2. **Wrong modulus**: `% type(uint96).max` is `% (2^96 ‚àí 1)`, not `% 2^96`. The value `2^96 ‚àí 1` is never reachable, and the value `0` is produced with doubled probability (both `0` and `2^96 ‚àí 1` map to `0`).
3. **Block-scoped determinism**: `block.timestamp`, `block.number`, and `bridgeReceiver` are identical for every transaction in the same block. The only differentiating inputs are `msg.sender`, `recipient`, `amount`, and `tx.gasprice`. Two transactions sharing all four produce an identical UID deterministically ‚Äî no brute-force required.

The fallback path for `uid == 0` uses a different but still truncated hash and is not re-checked for zero:

```solidity
if (uid == 0) {
    uid = uint96(uint256(keccak256(abi.encodePacked(block.number, msg.sender, tx.gasprice, bridgeReceiver))));
}
``` [2](#0-1) 

The UID is then passed directly to the Sonic bridge:

```solidity
sonicBridge.withdraw(uid, originalToken, amount);
``` [3](#0-2) 

The `ISonicBridge.withdraw` signature documents `uid` as a **unique identifier** for the withdrawal: [4](#0-3) 

On the L1 side, `SonicBridgeReceiver` enforces strict single-claim semantics per `withdrawalId`:

```solidity
if (claimedWithdrawals[withdrawalId]) revert WithdrawalAlreadyClaimed();
claimedWithdrawals[withdrawalId] = true;
``` [5](#0-4) 

---

### Impact Explanation

**Scenario A ‚Äî Sonic bridge rejects duplicate UIDs (expected behavior):**
The second `bridgeTokenToL1` call reverts at `sonicBridge.withdraw`. Because `safeTransferFrom` already executed before UID generation, the revert unwinds the entire transaction and returns the user's tokens. The user's bridge attempt fails silently; they must retry in a later block or with different parameters. This constitutes **temporary freezing of funds** (Medium).

**Scenario B ‚Äî Sonic bridge accepts duplicate UIDs:**
Both L2 transactions succeed and tokens are burned/transferred. On L1, `SonicBridgeReceiver.claimAndTransferToVault` can only claim one of the two identical `withdrawalId` values; the second call reverts with `WithdrawalAlreadyClaimed`. The second user's tokens are permanently unclaimable ‚Äî **permanent freezing of funds** (Critical). The root cause remains in the LRT-rsETH UID generation, not solely in external bridge behavior.

---

### Likelihood Explanation

The collision condition is fully deterministic and requires no brute-force:

- `block.timestamp` and `block.number` are identical for all transactions in the same block.
- `bridgeReceiver` is always `address(this)` (immutable).
- A single user submitting two identical bridge calls in the same block (e.g., via an automated bridger or a pool contract calling `bridgeTokenToL1` twice) guarantees a collision.
- Two different users bridging the same `amount` with the same `recipient` and `tx.gasprice` in the same block also collide.

On Sonic, blocks are produced at sub-second intervals, making same-block multi-transaction scenarios routine.

---

### Recommendation

Replace the block-scoped, truncated hash with a monotonically incrementing nonce stored in contract state:

```solidity
uint96 private _uidNonce;

function _nextUid() internal returns (uint96) {
    return ++_uidNonce;
}
```

This guarantees strict uniqueness across all calls regardless of block context, eliminates the entropy truncation problem, and removes the incorrect modulus entirely.

---

### Proof of Concept

1. Alice calls `bridgeTokenToL1(L1Vault, 1 ether)` in block N with `tx.gasprice = 1 gwei`.
2. In the same block N, Alice (or Bob with the same parameters) calls `bridgeTokenToL1(L1Vault, 1 ether)` with `tx.gasprice = 1 gwei`.
3. Both calls compute:
   ```
   uid = keccak256(N_timestamp || N || alice || L1Vault || 1e18 || 1e9 || bridge) % (2^96 - 1)
   ```
   producing the **identical** `uid`.
4. The first call succeeds: `sonicBridge.withdraw(uid, token, amount)` registers the withdrawal.
5. The second call hits `sonicBridge.withdraw` with the same `uid`. If the Sonic bridge enforces uniqueness, it reverts ‚Äî the entire transaction reverts, returning Alice's tokens but failing the bridge. If the Sonic bridge does not enforce uniqueness, both withdrawals are registered under the same `uid`, and `SonicBridgeReceiver.claimAndTransferToVault` on L1 can only claim one, permanently locking the second user's funds.
