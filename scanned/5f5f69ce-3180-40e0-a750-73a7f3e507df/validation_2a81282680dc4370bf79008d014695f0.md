### Title
ETH Deposit Limit Not Enforced Due to Missing `+ amount` in Limit Check - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit()` applies an asymmetric check for ETH vs. LST assets. The ETH branch omits the incoming deposit amount from the comparison, allowing any depositor to push total ETH deposits above the configured `depositLimitByAsset` cap.

### Finding Description
In `LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit()`, the LST branch correctly adds the incoming `amount` before comparing to the limit, but the ETH branch does not:

```solidity
// contracts/LRTDepositPool.sol lines 676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // ← missing `+ amount`
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct
}
```

When `totalAssetDeposits == depositLimit`, the ETH check returns `false` (not exceeded), so `_beforeDeposit` does not revert and the deposit proceeds, minting rsETH and pushing total ETH deposits above the configured cap. For LST assets, the same scenario correctly returns `true` and reverts.

This is called from `_beforeDeposit`, which is invoked on every ETH deposit:

```solidity
// contracts/LRTDepositPool.sol lines 661-663
if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
    revert MaximumDepositLimitReached();
}
```

### Impact Explanation
**Low.** The ETH deposit cap (`depositLimitByAsset`) is a protocol-level risk management parameter. Any depositor can bypass it when total ETH deposits are exactly at the limit, minting rsETH beyond the intended ceiling. No direct fund theft occurs, but the protocol fails to deliver its promised deposit constraint, and the excess ETH is staked into EigenLayer beyond the intended exposure limit.

### Likelihood Explanation
**Medium.** The deposit limit is a live on-chain parameter actively managed by the protocol. Any depositor monitoring the pool can trivially trigger this when `totalAssetDeposits == depositLimit`. No special privileges or conditions beyond a normal ETH deposit are required.

### Recommendation
Add `+ amount` to the ETH branch to match the LST branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept
1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 100 ether` via `LRTConfig.updateAssetDepositLimit`.
2. Total ETH deposits reach exactly `100 ether` (e.g., via prior deposits).
3. Attacker calls `depositETH{value: 1 ether}(0)` on `LRTDepositPool`.
4. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 1 ether)`.
5. `totalAssetDeposits = 100 ether`; check evaluates `100 ether > 100 ether` → `false` → no revert.
6. `_mintRsETH` mints rsETH for the attacker; total ETH deposits become `101 ether`, exceeding the cap.
7. Repeat for any subsequent depositor until the limit is far exceeded. [1](#0-0) [2](#0-1) [3](#0-2)
