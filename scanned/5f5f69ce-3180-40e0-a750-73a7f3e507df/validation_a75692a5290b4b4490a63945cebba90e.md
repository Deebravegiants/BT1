### Title
No Cancellation Mechanism for Stuck Withdrawal Requests Causes Temporary Freezing of User rsETH - (File: contracts/LRTWithdrawalManager.sol)

### Summary
`LRTWithdrawalManager` implements a FIFO withdrawal queue where users lock rsETH via `initiateWithdrawal`. The queue is unlocked sequentially by operators via `unlockQueue`. If any request in the queue cannot be fulfilled (e.g., insufficient assets in the unstaking vault due to EigenLayer withdrawal delays), all subsequent requests are also blocked. There is no cancellation function allowing users to recover their locked rsETH, creating a fund-freeze scenario analogous to the Atlas protocol's lack of recovery mechanism.

### Finding Description

The withdrawal lifecycle in `LRTWithdrawalManager` is:

1. User calls `initiateWithdrawal` — rsETH is immediately transferred from the user to the contract and a `WithdrawalRequest` is enqueued.
2. An operator calls `unlockQueue` — iterates the FIFO queue and unlocks requests sequentially.
3. User calls `completeWithdrawal` — receives the underlying asset.

At step 1, rsETH is irrevocably locked: [1](#0-0) 

At step 2, `_unlockWithdrawalRequests` processes requests in strict FIFO order and **breaks** (not continues) on the first request that cannot be covered by available assets: [2](#0-1) 

The critical consequence: if request at nonce N requires more assets than are currently available in the unstaking vault, **all requests at nonces N+1, N+2, … are also blocked**, even if those later requests are individually small enough to be covered. The `nextLockedNonce` pointer does not advance past N, so no subsequent request can ever be unlocked until N is resolved.

There is no `cancelWithdrawal` or equivalent function anywhere in `LRTWithdrawalManager` that would allow a user to reclaim their locked rsETH. The only write paths for a user are `initiateWithdrawal` and `completeWithdrawal`; the latter requires the request to already be unlocked. [3](#0-2) 

### Impact Explanation

A user whose withdrawal request is queued behind a large, unresolvable request has their rsETH permanently locked in `LRTWithdrawalManager` with no on-chain path to recover it. The rsETH has already been burned from the user's perspective (transferred to the contract), and the underlying asset cannot be received until the blocking request is resolved. If the blocking condition persists (e.g., EigenLayer withdrawal delay, slashing reducing vault assets below the large request's threshold), the freeze is indefinite.

**Impact: Medium — Temporary (potentially permanent) freezing of user funds.**

### Likelihood Explanation

The scenario is realistic and requires no privileged action:

1. EigenLayer enforces a withdrawal delay (currently ~7 days on mainnet). During this window, the `LRTUnstakingVault` may hold insufficient liquid assets.
2. Any user can call `initiateWithdrawal` with a large rsETH amount, creating a blocking request at the front of the queue.
3. Any subsequent user calling `initiateWithdrawal` with a smaller amount is queued behind the large request.
4. When the operator calls `unlockQueue`, the loop breaks at the large request, leaving the smaller user's rsETH locked with no recourse.

The entry path is fully unprivileged: `initiateWithdrawal` is callable by any rsETH holder. [4](#0-3) 

### Recommendation

Add a `cancelWithdrawal(address asset)` function that allows a user to remove their oldest pending (still-locked) withdrawal request from the queue and receive their rsETH back. The function must:
- Only allow cancellation of requests that have not yet been unlocked (`userNonce >= nextLockedNonce[asset]`).
- Remove the nonce from `userAssociatedNonces[asset][msg.sender]`.
- Decrease `assetsCommitted[asset]` by `request.expectedAssetAmount`.
- Return `request.rsETHUnstaked` to the user.

This mirrors the Atlas Protocol remediation of adding a refund path when the normal processing pipeline is blocked.

### Proof of Concept

1. Alice calls `initiateWithdrawal(stETH, 10_000e18, "")` — 10,000 rsETH locked, nonce 0 assigned.
2. Bob calls `initiateWithdrawal(stETH, 1e18, "")` — 1 rsETH locked, nonce 1 assigned.
3. The `LRTUnstakingVault` holds only 500 stETH (EigenLayer withdrawal still pending).
4. Operator calls `unlockQueue(stETH, 2, ...)`:
   - Loop hits nonce 0: `payoutAmount` for Alice's 10,000 rsETH >> 500 stETH available → `break`.
   - Nonce 1 (Bob) is never reached; `nextLockedNonce[stETH]` remains 0.
5. Bob calls `completeWithdrawal(stETH, "")` → reverts with `WithdrawalLocked` because nonce 1 >= `nextLockedNonce[stETH]` (= 0). [5](#0-4) 

6. Bob has no function to call to recover his 1 rsETH. His funds remain locked in `LRTWithdrawalManager` indefinitely. [6](#0-5)

    function _processWithdrawalCompletion(address asset, address user, string calldata referralId) internal {
        if (userAssociatedNonces[asset][user].empty()) {
            revert NoWithdrawalRequests(user, asset);
        }

        // Retrieve and remove the oldest withdrawal request for the user.
        uint256 usersFirstWithdrawalRequestNonce = userAssociatedNonces[asset][user].popFront();
        // Ensure the request is already unlocked.
        if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();

        bytes32 requestId = getRequestId(asset, usersFirstWithdrawalRequestNonce);
        WithdrawalRequest memory request = withdrawalRequests[requestId];

        delete withdrawalRequests[requestId];

        // Check that the withdrawal delay has passed since the request's initiation.
        if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();

    function _addUserWithdrawalRequest(address asset, uint256 rsETHUnstaked, uint256 expectedAssetAmount) internal {
        uint256 nextUnusedNonce_ = nextUnusedNonce[asset];

        // Generate a unique identifier for the new withdrawal request.
        bytes32 requestId = getRequestId(asset, nextUnusedNonce_);

        // Create and store the new withdrawal request.
        withdrawalRequests[requestId] = WithdrawalRequest({
            rsETHUnstaked: rsETHUnstaked, expectedAssetAmount: expectedAssetAmount, withdrawalStartBlock: block.number
        });

        // Map the user to the newly created request index and increment the nonce for future requests.
        userAssociatedNonces[asset][msg.sender].pushBack(nextUnusedNonce_);
        nextUnusedNonce[asset] = nextUnusedNonce_ + 1;

        emit AssetWithdrawalQueued(msg.sender, asset, rsETHUnstaked, nextUnusedNonce_);
