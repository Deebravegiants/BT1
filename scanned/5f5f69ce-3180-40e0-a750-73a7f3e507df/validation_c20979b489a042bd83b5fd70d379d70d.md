### Title
ETH Deposit Limit Check Omits Current Deposit Amount, Allowing Limit Bypass - (File: `contracts/LRTDepositPool.sol`)

### Summary

`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric validation: for ERC-20 assets it correctly adds the incoming deposit amount to the running total before comparing against the configured cap, but for ETH it compares only the pre-deposit total. Any depositor can therefore push ETH holdings above the protocol-configured deposit limit by depositing when the running total is exactly at the cap.

### Finding Description

The root cause is in `_checkIfDepositAmountExceedesCurrentLimit` at `contracts/LRTDepositPool.sol` lines 676–682:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // ← amount not included
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct
}
```

For ETH the guard is `totalAssetDeposits > limit`. When `totalAssetDeposits == limit` the expression evaluates to `false`, so `_beforeDeposit` does not revert and `depositETH` proceeds to mint rsETH. After the call, `totalAssetDeposits` becomes `limit + msg.value`, exceeding the cap.

The ERC-20 branch correctly uses `totalAssetDeposits + amount > limit`, which would have caught this case.

The attacker-controlled entry path is fully public:

1. Monitor `getTotalAssetDeposits(ETH_TOKEN)` until it equals `depositLimitByAsset[ETH_TOKEN]`.
2. Call `LRTDepositPool.depositETH{value: X}(minRSETH, referralId)`.
3. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, X)`.
4. The check returns `limit > limit` → `false` → no revert.
5. rsETH is minted for the deposit, and total ETH in the protocol now exceeds the configured cap.

No privileged role is required. The condition (total deposits at the cap) is a normal operational state that occurs whenever the protocol is fully subscribed.

### Impact Explanation

The ETH deposit limit is the protocol's primary risk-management guardrail for native ETH. Bypassing it allows unbounded additional ETH to enter the protocol beyond the intended ceiling, minting rsETH against assets the protocol did not plan to hold. This violates the invariant that `getTotalAssetDeposits(ETH) ≤ depositLimitByAsset[ETH]` and means the contract fails to deliver the promised deposit cap. No funds are directly stolen, placing this in the **Low** category: "Contract fails to deliver promised returns, but doesn't lose value."

### Likelihood Explanation

The condition is reached in normal operation whenever the ETH deposit cap is hit (a common state for a popular restaking protocol). Any depositor who monitors on-chain state can exploit this deterministically with a single transaction. No special timing, front-running, or privileged access is needed.

### Recommendation

Apply the same `+ amount` inclusion used for ERC-20 assets:

```diff
-    if (asset == LRTConstants.ETH_TOKEN) {
-        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
-    }
+    if (asset == LRTConstants.ETH_TOKEN) {
+        return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
+    }
```
