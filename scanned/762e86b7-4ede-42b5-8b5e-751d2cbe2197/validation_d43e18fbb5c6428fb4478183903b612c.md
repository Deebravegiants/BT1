### Title
Stale `rsETHPrice` Used for rsETH Minting After Oracle Unpause — (`contracts/LRTOracle.sol`)

---

### Summary

`LRTOracle` stores `rsETHPrice` as a cached state variable that is only updated via `updateRSETHPrice()`. That function is gated by `whenNotPaused`, so the price cannot be refreshed while the oracle is paused. The `unpause()` function does not trigger a price refresh. Because `LRTDepositPool.getRsETHAmountToMint()` reads the cached `rsETHPrice` directly, any deposits that occur after the oracle is unpaused — but before `updateRSETHPrice()` is called — use a stale exchange rate, causing incorrect rsETH amounts to be minted.

---

### Finding Description

`LRTOracle` maintains a cached `rsETHPrice` state variable: [1](#0-0) 

The public price-update entry point is blocked while the oracle is paused: [2](#0-1) 

The `unpause()` function simply clears the `paused` flag and emits an event — it does not call `_updateRsETHPrice()`: [3](#0-2) 

`LRTDepositPool.getRsETHAmountToMint()` divides by the cached `rsETHPrice` to compute how many rsETH shares to mint: [4](#0-3) 

The critical gap: `PAUSER_ROLE` can pause the oracle (line 138) **without** pausing the deposit pool. During the oracle pause, `rsETHPrice` is frozen. After the oracle is unpaused, the deposit pool remains active and immediately uses the stale price for every deposit until someone calls `updateRSETHPrice()`. [5](#0-4) 

A second trigger path exists via the auto-pause mechanism: when `_updateRsETHPrice()` detects an excessive price drop it pauses the oracle, deposit pool, and withdrawal manager together. When the admin later unpauses the oracle and then the deposit pool, the window between those two `unpause()` calls is safe — but once the deposit pool is unpaused, the stale `rsETHPrice` (frozen at the pre-drop value) is used until `updateRSETHPrice()` is called. [6](#0-5) 

There is no on-chain enforcement that `updateRSETHPrice()` must be called before deposits resume.

---

### Impact Explanation

`getRsETHAmountToMint` computes:

```
rsethAmountToMint = (depositAmount × assetPrice) / rsETHPrice
```

If `rsETHPrice` is stale-low (real price has risen during the pause), depositors receive **more** rsETH than the current TVL justifies, diluting every existing rsETH holder's redemption value — a form of yield theft from existing holders. If `rsETHPrice` is stale-high (real price fell during the pause), depositors receive **fewer** rsETH than they are entitled to, losing value on deposit.

**Impact: Low — Contract fails to deliver promised returns (incorrect share issuance); in the dilution direction this also constitutes theft of unclaimed yield from existing holders.**

---

### Likelihood Explanation

- The `PAUSER_ROLE` is a distinct role from `LRTAdmin`; it can pause the oracle without touching the deposit pool, making the scenario reachable without any admin error.
- Oracle pauses are an expected operational event (emergency response, price anomaly).
- The stale-price window is open from the moment `unpause()` is called until the next successful `updateRSETHPrice()` call, which is a public function that anyone can call but is not atomically enforced.
- The `pricePercentageLimit` check inside `_updateRsETHPrice()` can cause `updateRSETHPrice()` to revert for non-managers if the price moved significantly during the pause, **extending** the stale-price window further. [7](#0-6) 

**Likelihood: Medium.**

---

### Recommendation

In `LRTOracle.unpause()`, call `_updateRsETHPrice()` (or its manager-privileged variant) before clearing the paused flag, so the price is always fresh the moment deposits can resume:

```solidity
function unpause() external whenPaused onlyLRTAdmin {
    _updateRsETHPrice();   // refresh before re-opening
    paused = false;
    emit Unpaused(msg.sender);
}
```

Alternatively, enforce that the deposit pool is always co-paused whenever the oracle is paused, so no deposits can occur with a stale price.

---

### Proof of Concept

1. `rsETHPrice` is `1.1e18` at block N.
2. `PAUSER_ROLE` calls `LRTOracle.pause()`. The deposit pool is **not** paused.
3. Blocks N+1 … N+1000 pass; underlying assets accrue yield. The real rsETH/ETH rate is now `1.2e18`, but `rsETHPrice` is frozen at `1.1e18`.
4. `LRTAdmin` calls `LRTOracle.unpause()`. `rsETHPrice` is still `1.1e18`.
5. Attacker immediately calls `LRTDepositPool.depositETH{value: 1 ether}(0, "")`.
6. `getRsETHAmountToMint` computes: `1e18 × 1e18 / 1.1e18 ≈ 0.909e18 rsETH` minted.
7. Correct amount at real price: `1e18 × 1e18 / 1.2e18 ≈ 0.833e18 rsETH`.
8. Attacker receives ~9% excess rsETH, diluting all existing holders' redemption value.
9. Only after a separate `updateRSETHPrice()` call does `rsETHPrice` correct to `1.2e18`. [2](#0-1) [3](#0-2) [4](#0-3)
