### Title
`updatedAt` Advances Unconditionally When `totalKernelStaked == 0`, Permanently Freezing Unclaimed Yield — (File: `contracts/KERNEL/KernelDepositPool.sol`)

---

### Summary

The `updateReward` modifier in `KernelDepositPool` unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` even when `totalKernelStaked == 0`. Because `rewardPerToken()` returns `rewardPerTokenStored` unchanged when no tokens are staked, the elapsed time window is consumed without distributing any rewards. Those rewards are permanently locked in the contract. An unprivileged staker can deliberately drain `totalKernelStaked` to zero and then trigger repeated `updateReward` calls to advance `updatedAt` all the way to `finishAt`, freezing the entire remaining reward budget.

---

### Finding Description

The `updateReward` modifier executes two unconditional writes before any function body runs:

```solidity
// contracts/KERNEL/KernelDepositPool.sol  lines 232-242
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();   // (A)
    updatedAt = lastTimeRewardApplicable();    // (B)  ← always written
    ...
}
```

`rewardPerToken()` short-circuits when `totalKernelStaked == 0`:

```solidity
// lines 408-414
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;           // no accumulation
    }
    return rewardPerTokenStored
        + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
```

So at step (A), `rewardPerTokenStored` is left unchanged — no rewards are credited to anyone. But at step (B), `updatedAt` is still advanced to `block.timestamp` (or `finishAt` if past). The time interval `[old_updatedAt, block.timestamp]` is silently consumed. Any `rewardRate × elapsed` tokens that should have been earmarked for future stakers are now permanently unaccountable: `updatedAt` has moved past them, so the next `rewardPerToken()` call will never include them in its `(lastTimeRewardApplicable() - updatedAt)` delta.

`initiateWithdrawal` reduces `totalKernelStaked` before returning and is itself guarded by `updateReward`:

```solidity
// lines 320-338
function initiateWithdrawal(uint256 _amount)
    external nonReentrant updateReward(msg.sender)
{
    ...
    balanceOf[msg.sender] -= _amount;
    totalKernelStaked -= _amount;          // can reach 0
    ...
}
```

Once `totalKernelStaked` reaches zero, every subsequent call to any `updateReward`-gated function (`stake`, `stakeFor`, `initiateWithdrawal`, `getReward`, `notifyRewardAmount`) silently advances `updatedAt` without distributing rewards.

---

### Impact Explanation

**Permanent freezing of unclaimed yield — Medium.**

Rewards that accrued during any zero-staking window are permanently locked inside the contract. There is no recovery path: `updatedAt` is a monotonically advancing cursor, and once it passes a time interval with `totalKernelStaked == 0`, the `rewardRate × elapsed` tokens for that interval can never be claimed by anyone. The contract itself acknowledges this in its NatSpec comment (line 18–22) but provides no on-chain enforcement to prevent it.

---

### Likelihood Explanation

Any holder of KERNEL tokens who is also the last (or only) staker can reduce `totalKernelStaked` to zero by calling `initiateWithdrawal`. After that, a low-cost griefing loop is available to any address that holds even 1 wei of KERNEL:

1. `stake(1 wei)` — triggers `updateReward`, advances `updatedAt` to `block.timestamp`
2. `initiateWithdrawal(1 wei)` — triggers `updateReward` again, advances `updatedAt` once more

Repeating this loop across multiple blocks advances `updatedAt` toward `finishAt`, permanently destroying the remaining reward budget. The cost is only gas plus the 1-wei KERNEL token (which is returned after the withdrawal delay). No admin action or privileged role is required after the reward period has started.

---

### Recommendation

Guard the `updatedAt` write so it only advances when tokens are actually staked, mirroring the fix pattern from the referenced report:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {           // ← add this guard
        updatedAt = lastTimeRewardApplicable();
    }
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

This ensures `updatedAt` only moves forward when there are stakers to absorb the rewards, preventing silent time-window consumption.

### Proof of Concept

```
Setup
─────
• rewardRate  = 1e18 tokens/second
• duration    = 1000 seconds
• Alice stakes 100 KERNEL at T=0
• Admin calls notifyRewardAmount() at T=0
  → finishAt = 1000, updatedAt = 0

Step 1 — Alice withdraws at T=100
  initiateWithdrawal(100) triggers updateReward(Alice):
    rewardPerToken() = 0 + (1e18 * (100-0) * 1e18) / 100e18 = 1e18
    rewardPerTokenStored = 1e18
    updatedAt = 100
    rewards[Alice] = 100 * (1e18 - 0) / 1e18 = 100 tokens  ✓
  totalKernelStaked = 0

Step 2 — Bob stakes 1 wei at T=200
  stake(1) triggers updateReward(Bob):
    totalKernelStaked == 0  →  rewardPerToken() returns 1e18 (unchanged)
    rewardPerTokenStored = 1e18  (no change)
    updatedAt = 200          ← advances 100 seconds silently
    rewards[Bob] = 0
  totalKernelStaked = 1

Step 3 — Bob claims at T=300
  getReward() triggers updateReward(Bob):
    rewardPerToken() = 1e18 + (1e18 * (300-200) * 1e18) / 1
                     = 1e18 + 100e18  (Bob earns 100 seconds of full rate)
    rewards[Bob] = 1 * (101e18 - 1e18) / 1e18 = 100 tokens

Result
──────
• Alice earned 100 tokens (T=0..100)  ✓
• Bob earned  100 tokens (T=200..300) ✓
• 100 tokens for T=100..200 are permanently locked in the contract
  (updatedAt jumped from 100→200 with totalKernelStaked==0, consuming
   that window without distributing rewards)

Griefing extension: Bob repeats stake(1)/initiateWithdrawal(1) every block
until updatedAt reaches finishAt=1000, locking all 900 remaining tokens.
