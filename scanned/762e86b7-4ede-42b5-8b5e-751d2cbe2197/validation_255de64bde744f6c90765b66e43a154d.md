### Title
Missing Zero-Value Validation of Token Oracle Rate in `viewSwapRsETHAmountAndFee` Causes Permanent Loss of Deposited Tokens - (File: contracts/pools/RSETHPoolV3.sol, RSETHPoolV3ExternalBridge.sol, RSETHPool.sol, RSETHPoolNoWrapper.sol)

---

### Summary

The `viewSwapRsETHAmountAndFee(uint256 amount, address token)` function in all L2 pool contracts fetches `tokenToETHRate` from the token's oracle but does not validate that the returned value is non-zero before using it in the rsETH mint calculation. If the oracle returns `0`, the arithmetic produces `rsETHAmount = 0`, the user's deposited tokens are transferred into the pool, and zero wrsETH/rsETH is minted to the user. The deposited tokens are permanently unrecoverable by the user.

---

### Finding Description

In `RSETHPoolV3.sol`, `RSETHPoolV3ExternalBridge.sol`, `RSETHPool.sol`, and `RSETHPoolNoWrapper.sol`, the token-deposit variant of `viewSwapRsETHAmountAndFee` reads the token-to-ETH rate from the registered oracle and immediately uses it in a division without any zero-check:

```solidity
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

If `tokenToETHRate == 0`, Solidity 0.8 does not revert (the numerator is zero, not the denominator), so `rsETHAmount` silently evaluates to `0`. The calling `deposit(address token, uint256 amount, string referralId)` function then:

1. Transfers `amount` of the token from the user into the pool via `safeTransferFrom`.
2. Calls `viewSwapRsETHAmountAndFee`, receiving `rsETHAmount = 0`.
3. Mints `0` wrsETH/rsETH to the user.
4. Emits a `SwapOccurred` event with `rsETHAmount = 0`.

The user's tokens are now held by the pool with no corresponding receipt token issued. There is no refund path.

The inconsistency is explicit: the reverse-direction function `viewSwapAssetToPremintedRsETH` in the same contracts **does** guard against a zero token rate:

```solidity
uint256 tokenToETHRate = token == ETH_IDENTIFIER ? 1e18 : IOracle(supportedTokenOracle[token]).getRate();
if (tokenToETHRate == 0) revert UnsupportedOracle();
```

The forward deposit path has no equivalent guard.

The `addSupportedToken` function validates `IOracle(oracle).getRate() != 0` at registration time, but this is a one-time snapshot check. It does not prevent the oracle from returning `0` at a later deposit.

---

### Impact Explanation

**Critical — Permanent freezing of user funds.**

A depositor who calls `deposit(token, amount, referralId)` while the token oracle returns `0` loses their entire deposited amount. The tokens are held by the pool contract, the user holds no receipt token (wrsETH/rsETH) representing a claim, and no contract function allows the user to recover their tokens. The bridger role can move the tokens to L1, but the user has no on-chain claim to them. The loss is permanent from the user's perspective.

---

### Likelihood Explanation

**Low-Medium.** Token price oracles (e.g., Chainlink, custom rate providers) can transiently return `0` due to:
- Oracle contract bugs or upgrades that reset the stored rate.
- A freshly deployed oracle that has not yet received its first price update.
- An oracle that returns `0` as a sentinel for "unavailable" rather than reverting.

The `addSupportedToken` check only validates the rate at the moment of token registration. Any subsequent oracle failure that causes `getRate()` to return `0` is not caught at deposit time. The affected code path is the primary user-facing deposit function, so any depositor transacting during such a window is affected.

---

### Recommendation

Add a zero-value guard for `tokenToETHRate` in `viewSwapRsETHAmountAndFee(uint256 amount, address token)` in all four pool contracts, mirroring the guard already present in `viewSwapAssetToPremintedRsETH`:

```solidity
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
if (tokenToETHRate == 0) revert UnsupportedOracle();
```

Apply the same guard to `rsETHToETHrate` (currently a division-by-zero revert via panic; an explicit revert with a named error is preferable for clarity and gas efficiency).

---

### Proof of Concept

1. A supported token (e.g., wstETH) is registered with a valid oracle via `addSupportedToken`.
2. The oracle contract is upgraded or experiences a failure and begins returning `0` from `getRate()`.
3. A depositor calls `deposit(wstETH, 1e18, "ref")` on `RSETHPoolV3`.
4. `viewSwapRsETHAmountAndFee(1e18, wstETH)` executes:
   - `fee = 1e18 * feeBps / 10_000` (e.g., `1e15` for 10 bps)
   - `amountAfterFee = 1e18 - 1e15 = 0.999e18`
   - `rsETHToETHrate = getRate()` → e.g., `1.05e18` (normal)
   - `tokenToETHRate = IOracle(supportedTokenOracle[wstETH]).getRate()` → **`0`**
   - `rsETHAmount = 0.999e18 * 0 / 1.05e18 = 0`
5. `IERC20(wstETH).safeTransferFrom(msg.sender, address(this), 1e18)` — user's 1 wstETH is taken.
6. `wrsETH.mint(msg.sender, 0)` — user receives nothing.
7. User's 1 wstETH is permanently locked in the pool with no claim.
