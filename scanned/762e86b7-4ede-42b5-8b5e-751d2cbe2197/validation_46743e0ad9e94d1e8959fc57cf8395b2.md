### Title
Multi-Day Price Update Delay Causes `DailyFeeMintLimitExceeded` Revert, Blocking Price Updates and Enabling Dilution of rsETH Holders - (File: contracts/LRTOracle.sol)

### Summary
When `updateRSETHPrice()` is not called for multiple consecutive days (e.g., due to sustained high gas), the accumulated multi-day reward generates a protocol fee that exceeds `maxFeeMintAmountPerDay`. This causes `_checkAndUpdateDailyFeeMintLimit` to revert with `DailyFeeMintLimitExceeded`, blocking all price updates — including the manager path — until an admin manually increases the daily limit. During this window, new depositors mint rsETH at the stale (lower) price, diluting existing holders' share of accumulated rewards.

### Finding Description
In `LRTOracle._updateRsETHPrice()`, the protocol fee is calculated on the **total accumulated reward since the last update**, with no time-weighting:

```solidity
uint256 rewardAmount = totalETHInProtocol - previousTVL;
protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
``` [1](#0-0) 

The fee is then converted to rsETH and passed to the daily limit check:

```solidity
uint256 rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice);
_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee);
``` [2](#0-1) 

Inside `_checkAndUpdateDailyFeeMintLimit`, the period is reset if a day has passed, but the entire N-day accumulated fee is then checked against a **single day's** limit:

```solidity
if (block.timestamp >= feePeriodStartTime + 1 days) {
    currentPeriodMintedFeeAmount = 0;
    feePeriodStartTime = getCurrentPeriodStartTime();
}
if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
    revert DailyFeeMintLimitExceeded(...);
}
``` [3](#0-2) 

Critically, **both** the public `updateRSETHPrice()` and the manager-only `updateRSETHPriceAsManager()` call `_updateRsETHPrice()` internally, so neither path can bypass the daily limit check: [4](#0-3) 

The only recovery path is for an admin to first call `setMaxFeeMintAmountPerDay()` to raise the cap, then call `updateRSETHPriceAsManager()`. During the entire blocked window, `rsETHPrice` remains stale.

New depositors use the stale price directly in `getRsETHAmountToMint`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [5](#0-4) 

Because `rsETHPrice` is lower than the true current value (rewards have accrued but not been reflected), new depositors receive **more rsETH than they are entitled to**, at the expense of existing holders.

### Impact Explanation
The impact is **High — theft of unclaimed yield**. Existing rsETH holders are entitled to the rewards that have accrued since the last price update. When new depositors mint rsETH at the stale (lower) price, they capture a portion of those accrued rewards. Once the price is eventually corrected upward, the over-minted rsETH held by new depositors is worth more than they paid, and existing holders' proportional claim on the TVL is permanently diluted.

### Likelihood Explanation
Low. The protocol relies on an offchain bot to call `updateRSETHPrice()` daily. A multi-day delay would require sustained high gas fees causing the bot's transactions to revert for several consecutive days — the same scenario described in the referenced Lido hurdle rate report. The `maxFeeMintAmountPerDay` is calibrated for single-day fee amounts, so even a 2–3 day delay can push the accumulated fee over the cap.

### Recommendation
1. **Time-weight the fee**: Divide the accumulated fee by the number of elapsed days and apply the daily limit per day, or split the update into multiple sub-steps.
2. **Size `maxFeeMintAmountPerDay` conservatively**: Set it to a multiple of the expected daily fee (e.g., 7×) to absorb short outages without reverting.
3. **Add a manager bypass for the daily cap**: Allow `updateRSETHPriceAsManager()` to skip the `_checkAndUpdateDailyFeeMintLimit` revert (while still recording the mint), so the price can always be corrected even if the cap is exceeded.

### Proof of Concept
| Parameter | Value |
|---|---|
| TVL | 100,000 ETH |
| rsETH supply | 95,238 rsETH |
| `rsETHPrice` (stale) | 1.05 ETH |
| `protocolFeeInBPS` | 1,000 (10%) |
| `maxFeeMintAmountPerDay` | 1 rsETH |
| Daily reward | 10 ETH (~0.01% of TVL) |
| Expected daily fee | 0.952 rsETH (just under cap) |

1. Bot is delayed for **3 days** due to high gas.
2. Accumulated reward = 30 ETH; accumulated fee = 3 ETH = **2.857 rsETH**.
3. `_checkAndUpdateDailyFeeMintLimit(2.857)` reverts: `DailyFeeMintLimitExceeded(2.857, 1)`.
4. Both `updateRSETHPrice()` and `updateRSETHPriceAsManager()` revert.
5. `rsETHPrice` remains at 1.05 ETH; true price ≈ 1.053 ETH.
6. A depositor sends 105 ETH → receives `105 / 1.05 = 100 rsETH` instead of the correct `105 / 1.053 ≈ 99.7 rsETH`.
7. The depositor captures ~0.3% excess rsETH, permanently diluting existing holders' claim on the 30 ETH of accrued rewards.

        // Reset the period if it's unset or a day has passed
        if (block.timestamp >= feePeriodStartTime + 1 days) {
            currentPeriodMintedFeeAmount = 0;
            feePeriodStartTime = getCurrentPeriodStartTime();
        }

        // Check if minting would exceed the daily limit
        if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
            revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
        }

        currentPeriodMintedFeeAmount += feeAmount;
    }
