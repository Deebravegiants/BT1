### Title
Temporary Freeze of ETH Withdrawals After Aave Integration Reconfiguration — (`contracts/LRTWithdrawalManager.sol`)

### Summary
When `configureAaveIntegration` is called to switch to a new Aave pool, all ETH is pulled from the old Aave pool into the contract and `totalETHDepositedToAave` is reset to zero, while `isAaveIntegrationEnabled` remains `true`. Because `_withdrawFromAave` is capped by `totalETHDepositedToAave`, it silently returns 0 for any subsequent on-demand withdrawal attempt. If pending user ETH withdrawals exhaust the contract's idle balance before an operator manually re-deposits to the new pool, every further `completeWithdrawal` call reverts with `InsufficientLiquidityForWithdrawal`, temporarily freezing user funds.

### Finding Description
`configureAaveIntegration` (lines 422–464) handles reconfiguration of an existing Aave integration as follows:

```solidity
// Lines 438-449
if (address(aaveAWETH) != address(0) && ...) {
    uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
    if (aaveBalance > 0) {
        _collectInterestToTreasury();
        aaveBalance = aaveAWETH.balanceOf(address(this));
        if (aaveBalance > 0) {
            _withdrawFromAave(aaveBalance);   // resets totalETHDepositedToAave → 0
        }
    }
}
// Lines 455-458: new pool addresses written
aavePool = aavePool_;
aaveWETHGateway = IWrappedTokenGatewayV3(aaveWETHGateway_);
aaveAWETH = IAToken(aaveAWETH_);
aaveDataProvider = IPoolDataProvider(aaveDataProvider_);
// isAaveIntegrationEnabled is NOT touched — stays true
```

`_withdrawFromAave` (lines 905–921) decrements `totalETHDepositedToAave` by the amount withdrawn, leaving it at or near zero after a full withdrawal:

```solidity
uint256 withdrawablePrincipal =
    aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave;
withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
if (withdrawnAmount == 0) return 0;          // silent no-op when principal == 0
...
totalETHDepositedToAave -= withdrawnAmount;
```

After reconfiguration the contract holds the withdrawn ETH in its native balance, but `totalETHDepositedToAave == 0` and the new Aave pool has received nothing. `isAaveIntegrationEnabled` is still `true`, so `_processWithdrawalCompletion` (lines 719–732) will attempt to pull from Aave whenever the contract balance falls short:

```solidity
if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
    uint256 contractBalance = address(this).balance;
    if (contractBalance < request.expectedAssetAmount) {
        uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
        _withdrawFromAave(amountNeeded);          // returns 0 — totalETHDepositedToAave == 0
        uint256 balanceAfter = address(this).balance;
        if (balanceAfter < request.expectedAssetAmount) {
            revert InsufficientLiquidityForWithdrawal();   // always hits
        }
    }
}
```

Because `_withdrawFromAave` is silently capped at `totalETHDepositedToAave == 0`, it contributes nothing, and the revert fires for every withdrawal whose amount exceeds the remaining idle contract balance.

### Impact Explanation
Users with unlocked ETH withdrawal requests cannot call `completeWithdrawal` or `completeWithdrawalForUser` once the contract's idle ETH balance is exhausted. Their rsETH has already been burned (at `unlockQueue` time) and the ETH owed to them is inaccessible until an operator manually calls `depositIdleETHToAave`. This constitutes a **temporary freezing of user funds** (Medium severity per the allowed impact scope). The duration of the freeze is unbounded by the protocol itself — it depends entirely on operator response time, and there is no on-chain enforcement that the re-deposit happens promptly.

### Likelihood Explanation
The trigger is a legitimate, non-malicious admin action (`configureAaveIntegration`) that is expected to occur during protocol upgrades or Aave pool migrations. No compromise is required. The window between reconfiguration and operator re-deposit is realistic: `depositIdleETHToAave` is a separate, manually-initiated call with no automated trigger. During that window, any user whose unlocked withdrawal amount exceeds the remaining contract balance will be blocked. Given that the Aave integration is designed to hold significant ETH liquidity, the contract balance can be exhausted quickly by a moderate number of concurrent withdrawals.

### Recommendation
After writing the new Aave addresses (lines 455–458), automatically deposit the idle ETH balance into the new pool:

```solidity
uint256 idleBalance = address(this).balance;
if (isAaveIntegrationEnabled && idleBalance > 0) {
    _depositToAave(idleBalance);
}
```

Alternatively, temporarily set `isAaveIntegrationEnabled = false` at the start of `configureAaveIntegration` and require the manager to explicitly re-enable it (and re-deposit) after reconfiguration, preventing the silent `_withdrawFromAave` no-op from blocking user withdrawals.

### Proof of Concept
1. Protocol has `isAaveIntegrationEnabled = true`, `totalETHDepositedToAave = 100 ETH` deposited in old Aave pool.
2. Manager calls `configureAaveIntegration(newPool, newGateway, newAWETH, newDataProvider)`.
   - `_withdrawFromAave(100 ETH)` executes: 100 ETH lands in contract, `totalETHDepositedToAave = 0`.
   - New Aave addresses are set. `isAaveIntegrationEnabled` remains `true`.
3. Users with unlocked ETH withdrawal requests call `completeWithdrawal`. The first ~100 ETH worth of requests succeed (using idle contract balance).
4. The 101st user's request (say, 1 ETH) finds `address(this).balance = 0.5 ETH < 1 ETH`.
   - `_withdrawFromAave(0.5 ETH)` is called; `withdrawablePrincipal = min(0, 0) = 0`; returns 0.
   - `balanceAfter = 0.5 ETH < 1 ETH` → `revert InsufficientLiquidityForWithdrawal`.
5. All subsequent ETH withdrawals revert until an operator calls `depositIdleETHToAave(0.5 ETH)`, updating `totalETHDepositedToAave = 0.5 ETH`, after which `_withdrawFromAave` can service the shortfall. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```text
    function _withdrawFromAave(uint256 amount) internal returns (uint256 withdrawnAmount) {
        if (amount == 0) return 0;

        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        if (aaveBalance == 0) revert InsufficientAaveBalance();

        // Only withdraw up to the principal amount (don't use accrued interest for user withdrawals)
        uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave;

        withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
        if (withdrawnAmount == 0) return 0;
```
