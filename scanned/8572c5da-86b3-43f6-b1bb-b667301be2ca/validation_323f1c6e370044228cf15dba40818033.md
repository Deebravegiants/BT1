### Title
Unrestricted `claim()` Allows Griefing Attack to Temporarily Freeze Victim's Claim Ability — (`contracts/utils/MerkleDistributor/MerkleBlastPointsDistributor.sol`)

### Summary
`MerkleBlastPointsDistributor.claim()` has no `msg.sender == account` guard. Any caller can submit a valid merkle proof on behalf of any other user, advancing that user's `lastClaimedIndex` and cumulative state. If the next published root carries identical cumulative values for the victim (a realistic scenario when a user earns no new points between two root publications), the victim's own subsequent `claim()` call reverts with `NoPointsToClaim`, temporarily freezing their ability to claim at that index.

### Finding Description

The `claim` function accepts an arbitrary `account` parameter and performs no check that `msg.sender == account`: [1](#0-0) 

The only guards are:
- Root is non-zero [2](#0-1) 
- `isClaimed(index, account)` is false [3](#0-2) 
- Merkle proof is valid [4](#0-3) 

After a successful call, the victim's cumulative state is written: [5](#0-4) 

When the owner later publishes root N+1 with the same cumulative `(P, G)` for the victim (no new points earned), the victim's own `claim(N+1, victim, P, G, proof)` computes:

```
claimableBlastPoints = P - P = 0
claimableBlastGold   = G - G = 0
```

and hits the hard revert: [6](#0-5) 

The victim is stuck at index N+1 until a root N+2 is published with strictly higher cumulative values.

### Impact Explanation

The victim cannot claim their Blast Points/Gold at index N+1 despite being validly included in the merkle tree. This constitutes **temporary freezing of unclaimed yield** (Blast Points and Gold have real economic value on the Blast network). The freeze lasts until the owner publishes a subsequent root with a strictly higher cumulative for the victim.

**Matched scope:** Medium — Temporary freezing of funds.

### Likelihood Explanation

- Merkle proofs for any leaf are public (published off-chain alongside the root); the attacker needs no privileged access.
- The scenario where a user's cumulative is unchanged between two consecutive roots is common — the owner may publish roots on a fixed schedule and include all historical users even if some earned nothing in that period.
- The attack costs only gas and requires no capital.
- No admin compromise, front-running dependency, or external protocol failure is needed.

### Recommendation

Add a `msg.sender == account` restriction to `claim()`:

```solidity
if (account != msg.sender) {
    revert Unauthorized();
}
```

This mirrors the guard already present in `KernelMerkleDistributor.sol`: [7](#0-6) 

### Proof of Concept

```solidity
// Setup: currentIndex = 1, root includes victim with (points=100, gold=50)
// Attacker has the public merkle proof for (index=1, victim, 100, 50)

// Step 1: Attacker pre-claims for victim at index 1
distributor.claim(1, victim, 100, 50, attackerHeldProof);
// Succeeds: victim.lastClaimedIndex = 1, cumulative = (100, 50)

// Step 2: Owner publishes root 2 — victim earned no new points, cumulative still (100, 50)
distributor.setMerkleRoot(root2);  // currentIndex = 2

// Step 3: Victim tries to claim at index 2
distributor.claim(2, victim, 100, 50, newProof);
// REVERTS: NoPointsToClaim (100-100=0, 50-50=0)
// Victim is frozen at index 2 until root 3 with strictly higher cumulative is published
```
