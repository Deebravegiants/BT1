### Title
Rewards permanently lost when `totalKernelStaked` drops to zero mid-period via `initiateWithdrawal` - (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary

`KernelDepositPool` uses a Synthetix-style reward accounting where `rewardPerTokenStored` does not increment when `totalKernelStaked == 0`, but the `updatedAt` timestamp still advances. The `notifyRewardAmount` guard prevents starting a reward period with zero stakers, but it does not prevent `totalKernelStaked` from reaching zero mid-period via `initiateWithdrawal`. Any rewards accrued during the zero-supply window are permanently locked in the contract.

### Finding Description

The `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()`: [1](#0-0) 

`rewardPerToken()` returns `rewardPerTokenStored` unchanged when `totalKernelStaked == 0`: [2](#0-1) 

This means that for any time interval `[T1, T2]` during which `totalKernelStaked == 0`, the rewards that should have been distributed (`rewardRate * (T2 - T1)`) are silently skipped — `updatedAt` jumps to `T2` and the interval is permanently erased from accounting.

`notifyRewardAmount` guards against starting a new period with zero stakers: [3](#0-2) 

However, `initiateWithdrawal` immediately decrements `totalKernelStaked` with no floor check: [4](#0-3) 

There is no mechanism to prevent `totalKernelStaked` from reaching zero during an active reward window. The contract's own NatSpec acknowledges this gap and relies on an off-chain operational guarantee: [5](#0-4) 

That guarantee is not enforced on-chain.

### Impact Explanation

Reward tokens sent to the contract via `notifyRewardAmount` that fall within a zero-supply interval are permanently unclaimable. There is no admin recovery function, no sweep mechanism, and no way to re-distribute the lost rewards. The amount lost equals `rewardRate × duration_of_zero_supply_window`. This constitutes **permanent freezing of unclaimed yield**.

### Likelihood Explanation

The scenario requires all current stakers to call `initiateWithdrawal` during an active reward period. This is realistic when:
- A single large staker (e.g., a protocol-controlled address with `STAKE_FOR_ROLE`) holds all staked tokens and withdraws.
- A coordinated exit occurs (e.g., a depeg event or protocol migration).
- The reward period is long (e.g., 30 days) and stakers naturally exit before it ends.

The withdrawal delay only delays token recovery — `totalKernelStaked` drops to zero immediately at `initiateWithdrawal` call time, not at `claimWithdrawal` time.

### Recommendation

Enforce the zero-supply invariant on-chain during `initiateWithdrawal`, or pause reward accrual when `totalKernelStaked` reaches zero by not advancing `updatedAt` in that case. The minimal fix is to skip the `updatedAt` update inside `updateReward` when `totalKernelStaked == 0`:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked != 0) {          // <-- add this guard
        updatedAt = lastTimeRewardApplicable();
    }
    ...
}
```

This ensures the zero-supply interval is not consumed, and rewards resume correctly once stakers return.

### Proof of Concept

1. Admin calls `notifyRewardAmount(1_000e18)` with `duration = 30 days` and `totalKernelStaked = 100e18`. `rewardRate = 1000e18 / 30 days ≈ 385e12 per second`. `finishAt = now + 30 days`.
2. After 10 days, the sole staker calls `initiateWithdrawal(100e18)`. `updateReward` runs: `rewardPerTokenStored` correctly captures 10 days of rewards; `updatedAt = now`. `totalKernelStaked` becomes `0`.
3. 10 more days pass. No one stakes. `rewardPerToken()` returns `rewardPerTokenStored` unchanged on every call, but `updatedAt` will be set to `now` on the next interaction.
4. A new staker calls `stake(100e18)`. `updateReward` runs: `rewardPerToken()` still returns the same `rewardPerTokenStored` (supply was 0), and `updatedAt` is now advanced to the current time. The 10-day gap (`≈ 385e12 × 864000 ≈ 332e18` reward tokens) is permanently lost inside the contract. [2](#0-1) [1](#0-0)
