### Title
Missing Staleness Validation in `ChainlinkPriceOracle.getAssetPrice()` Allows Stale Prices to Drive rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards every return value except `price`. No staleness check (`updatedAt`), no round-completeness check (`answeredInRound >= roundId`), and no non-negative price check are performed. Because this oracle feeds directly into `LRTDepositPool.getRsETHAmountToMint()`, a stale or incomplete Chainlink round causes the protocol to mint rsETH at an incorrect rate for any unprivileged depositor.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` reads the Chainlink feed as:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All five return values of `latestRoundData()` are `(roundId, answer, startedAt, updatedAt, answeredInRound)`. The implementation silently drops `updatedAt` and `answeredInRound`, so:

- **No heartbeat / staleness check**: `block.timestamp - updatedAt > heartbeat` is never evaluated.
- **No round-completeness check**: `answeredInRound < roundId` (indicating an in-progress or failed round) is never caught.
- **No non-negative price guard**: a zero or negative `price` is cast directly to `uint256`, producing a wildly incorrect value.

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` in the same repository does check `answeredInRound < roundID` and `timestamp == 0`, demonstrating the team is aware of these guards but omitted them from the primary oracle.

The stale price flows into rsETH minting via:

```
depositAsset() → _beforeDeposit() → getRsETHAmountToMint()
    → lrtOracle.getAssetPrice(asset)          // live Chainlink call
    → ChainlinkPriceOracle.getAssetPrice()    // no staleness check
rsethAmountToMint = (amount * getAssetPrice(asset)) / rsETHPrice
```

`rsETHPrice` is a stored value updated separately by `updateRSETHPrice()`. If the Chainlink feed for an LST (e.g., stETH/ETH) goes stale while the asset's real price has dropped (depeg scenario), `getAssetPrice()` returns the old, higher price, and the depositor receives more rsETH than the deposited value warrants, diluting all existing rsETH holders.

The same stale price also affects `LRTWithdrawalManager.getExpectedAssetAmount()`:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

A stale high asset price here causes withdrawers to receive fewer underlying tokens than they are owed.

---

### Impact Explanation

**Low — Contract fails to deliver promised returns.**

When a Chainlink feed goes stale:
- Depositors minting rsETH with a depegged LST receive more rsETH than the deposited ETH value warrants, diluting existing holders' share of the backing.
- Withdrawers redeeming rsETH for an LST receive fewer tokens than the rsETH they burn is worth.

The protocol's `pricePercentageLimit` guard in `_updateRsETHPrice()` only applies to the stored `rsETHPrice` update path, not to the live `getAssetPrice()` call used at deposit/withdrawal time, so it does not mitigate this issue.

---

### Likelihood Explanation

**Low.** Chainlink feeds for major LSTs (stETH, rETH) have 24-hour heartbeats and 0.5 % deviation thresholds, making prolonged staleness uncommon under normal conditions. However, during network congestion, oracle node outages, or a rapid LST depeg event (exactly the scenario where correct pricing matters most), the feed can lag significantly before the heartbeat triggers an update.

---

### Recommendation

Add staleness and validity guards to `ChainlinkPriceOracle.getAssetPrice()`, consistent with what `ChainlinkOracleForRSETHPoolCollateral` already does:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice(); // per-feed heartbeat

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

---

### Proof of Concept

1. Chainlink's stETH/ETH feed stops updating (oracle node outage or network congestion). The last reported price is `1.05 ETH` per stETH.
2. stETH depegs on-chain to `0.95 ETH`.
3. An attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
4. `getRsETHAmountToMint` computes: `(1000e18 * 1.05e18) / rsETHPrice` — using the stale `1.05` price instead of the real `0.95`.
5. The attacker receives ~10.5 % more rsETH than the deposited value warrants, at the expense of all existing rsETH holders.
