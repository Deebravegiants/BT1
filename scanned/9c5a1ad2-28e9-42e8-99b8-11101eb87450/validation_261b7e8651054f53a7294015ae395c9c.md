### Title
Unchecked Negative Chainlink Price Cast Causes Arithmetic Overflow Revert, Freezing Deposits and Withdrawals - (File: contracts/oracles/ChainlinkPriceOracle.sol)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` casts the Chainlink `int256 price` directly to `uint256` without first validating it is positive. If Chainlink returns a negative answer (possible during circuit-breaker events or stale-data scenarios), the cast wraps to a value near `type(uint256).max`, and the subsequent multiplication by `1e18` overflows and reverts under Solidity 0.8's checked arithmetic. This revert propagates to every deposit and withdrawal entry point in the protocol.

---

### Finding Description

In `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

Chainlink's `latestRoundData()` returns `int256 answer`. If this value is negative (e.g., `-1`):

1. `uint256(-1)` = `2^256 ŌłÆ 1` ŌĆö type conversion wraps silently; no revert here.
2. `(2^256 ŌłÆ 1) * 1e18` ŌĆö overflows `uint256` ŌåÆ **REVERT** under Solidity 0.8 checked arithmetic.

The revert propagates through the entire price-dependent call graph:

- `LRTOracle.getAssetPrice(asset)` ŌåÆ `LRTOracle._getTotalEthInProtocol()` ŌåÆ `LRTOracle._updateRsETHPrice()` / `updateRSETHPrice()` (public)
- `LRTDepositPool.getRsETHAmountToMint()` ŌåÆ `_beforeDeposit()` ŌåÆ `depositAsset()` / `depositETH()`
- `LRTWithdrawalManager.getExpectedAssetAmount()` ŌåÆ `initiateWithdrawal()`

No validation of `price > 0` exists anywhere in the call chain before the cast.

---

### Impact Explanation

**Medium ŌĆö Temporary freezing of funds.**

All deposits (`depositAsset`, `depositETH`) and all withdrawal initiations (`initiateWithdrawal`) for the affected asset revert for as long as Chainlink returns a non-positive price. The oracle price-update function `updateRSETHPrice()` also reverts, preventing any rsETH price refresh. Users cannot enter or exit the protocol for the affected asset during this window.

---

### Likelihood Explanation

Chainlink price feeds can return non-positive values in documented edge cases: circuit-breaker activation (the feed hits its `minAnswer` floor and returns 0 or a negative sentinel), stale-data propagation, or feed misconfiguration during an upgrade. This is a known operational risk acknowledged in Chainlink's own documentation and in numerous prior audit findings across DeFi. The trigger does not require any privileged actor or governance action ŌĆö it is a passive consequence of Chainlink feed behavior. Likelihood is low-to-medium but non-zero and has materialized in production protocols before.

---

### Recommendation

Validate that `price` is strictly positive before casting:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();
    require(price > 0, "ChainlinkPriceOracle: non-positive price");
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

This converts the overflow-revert into a clean, descriptive revert that does not cascade unexpectedly.

---

### Proof of Concept

1. Chainlink's `latestRoundData()` returns `answer = -1` for a supported asset (e.g., stETH).
2. `ChainlinkPriceOracle.getAssetPrice(stETH)` executes `uint256(-1)` ŌåÆ `2^256 ŌłÆ 1`.
3. `(2^256 ŌłÆ 1) * 1e18` overflows ŌåÆ **revert** (Solidity 0.8 checked arithmetic).
4. `LRTOracle.getAssetPrice(stETH)` reverts.
5. Any user calling `LRTDepositPool.depositAsset(stETH, ...)` receives a revert.
6. Any user calling `LRTWithdrawalManager.initiateWithdrawal(stETH, ...)` receives a revert.
7. `LRTOracle.updateRSETHPrice()` also reverts, blocking price updates.
8. All deposits and withdrawal initiations for stETH are frozen until Chainlink recovers.
