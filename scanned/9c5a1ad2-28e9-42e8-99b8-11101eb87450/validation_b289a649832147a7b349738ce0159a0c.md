### Title
Aave Integration Losses Not Accounted For, Causing Withdrawal Queue Freeze for ETH Withdrawers - (File: contracts/LRTWithdrawalManager.sol)

### Summary
`LRTWithdrawalManager` deposits ETH from the withdrawal queue into Aave v3 for yield. If Aave suffers a loss (i.e., `aaveBalance < totalETHDepositedToAave`), the `_withdrawFromAave` function silently caps the withdrawal at the available balance without reconciling the `totalETHDepositedToAave` accounting variable. This leaves a ghost principal balance recorded in the contract while the actual ETH no longer exists. Users with already-unlocked ETH withdrawal requests are then permanently frozen: `_processWithdrawalCompletion` attempts to pull the shortfall from Aave, fails to recover the full amount, and reverts with `InsufficientLiquidityForWithdrawal`. Early completers receive their full ETH; later completers are frozen with no protocol-level remedy.

### Finding Description

**Root cause — `_withdrawFromAave` does not reconcile losses:**

```solidity
// contracts/LRTWithdrawalManager.sol:905-921
function _withdrawFromAave(uint256 amount) internal returns (uint256 withdrawnAmount) {
    uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
    if (aaveBalance == 0) revert InsufficientAaveBalance();

    // caps at min(aaveBalance, totalETHDepositedToAave)
    uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave;
    withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
    if (withdrawnAmount == 0) return 0;

    aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
    totalETHDepositedToAave -= withdrawnAmount;   // ← only decrements by what was withdrawn, not by the full loss
}
```

If `totalETHDepositedToAave = 100 ETH` and Aave's actual balance is `90 ETH` (10 ETH lost):
- `withdrawablePrincipal = 90`
- `withdrawnAmount = 90`
- `totalETHDepositedToAave` becomes `10` — a ghost value representing ETH that no longer exists in Aave.

The code's own NatSpec acknowledges this: `@dev Returns 0 if aaveBalance < totalETHDepositedToAave (potential accounting issue)`. [1](#0-0) 

**Consequence — `_processWithdrawalCompletion` reverts for all subsequent ETH withdrawers:**

```solidity
// contracts/LRTWithdrawalManager.sol:720-732
if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
    uint256 contractBalance = address(this).balance;
    if (contractBalance < request.expectedAssetAmount) {
        uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
        _withdrawFromAave(amountNeeded);

        uint256 balanceAfter = address(this).balance;
        if (balanceAfter < request.expectedAssetAmount) {
            revert InsufficientLiquidityForWithdrawal();   // ← permanent freeze for this user
        }
    }
}
_transferAsset(asset, user, request.expectedAssetAmount);
``` [2](#0-1) 

Because the entire call reverts, all state mutations (`popFront`, `delete withdrawalRequests`, `unlockedWithdrawalsCount--`) are rolled back. The user's request is preserved but permanently unserviceable until an admin manually deposits ETH to cover the shortfall — a mechanism that does not exist in the protocol.

**"First out" advantage:** Users who call `completeWithdrawal` before the Aave loss event receive their full `expectedAssetAmount`. Users who call after the loss revert indefinitely. The `expectedAssetAmount` was locked at `unlockQueue` time using the then-current rsETH price, so there is no way for later users to accept a reduced payout. [3](#0-2) 

**Aave integration is active and ETH is deposited automatically after every `unlockQueue` call:** [4](#0-3) 

### Impact Explanation
Users with unlocked ETH withdrawal requests are frozen — they cannot complete their withdrawals. The protocol has no on-chain mechanism to reconcile the loss or reduce `expectedAssetAmount` for affected users. This constitutes **temporary (potentially permanent) freezing of user funds** (Medium impact per scope).

### Likelihood Explanation
Aave v3 is a battle-tested protocol, but it is not immune to bad-debt events, governance attacks, or oracle manipulation that could reduce the aWETH balance below the deposited principal. The protocol's own `_checkAaveHealth` function and the `getAccruedInterest` NatSpec both acknowledge the scenario where `aaveBalance < totalETHDepositedToAave` is possible. Likelihood is **Low** but non-zero, and the impact when triggered is significant. [5](#0-4) 

### Recommendation
**Short term:** In `_withdrawFromAave`, when `aaveBalance < totalETHDepositedToAave`, explicitly zero out `totalETHDepositedToAave` after a full withdrawal to prevent ghost accounting. Additionally, in `_processWithdrawalCompletion`, instead of reverting on `InsufficientLiquidityForWithdrawal`, record the shortfall and allow users to claim a proportionally reduced amount, or pause the contract and emit an event for operator intervention.

**Long term:** Implement a loss-socialization mechanism: when Aave returns less than the deposited principal, distribute the loss proportionally across all pending unlocked withdrawal requests rather than freezing later withdrawers entirely.

### Proof of Concept

1. Protocol has 100 ETH in unlocked withdrawal requests, all deposited to Aave via `unlockQueue` → `depositToAaveExternal`. `totalETHDepositedToAave = 100 ETH`.
2. Aave suffers a loss; `aaveAWETH.balanceOf(withdrawalManager) = 90 ETH`.
3. User A calls `completeWithdrawal(ETH, ...)` for 50 ETH. `address(this).balance = 0`, so `_withdrawFromAave(50)` is called. `withdrawablePrincipal = 90`, `withdrawnAmount = 50`. Succeeds. `totalETHDepositedToAave = 50`.
4. User B calls `completeWithdrawal(ETH, ...)` for 50 ETH. `address(this).balance = 0`, so `_withdrawFromAave(50)` is called. `aaveBalance = 40`, `withdrawablePrincipal = 40`, `withdrawnAmount = 40`. `balanceAfter = 40 < 50`. → **`revert InsufficientLiquidityForWithdrawal()`**. User B is frozen.
5. `totalETHDepositedToAave` now shows `10` (ghost), Aave has `0 ETH`. No protocol function exists to cover the 10 ETH shortfall for User B. [6](#0-5) [7](#0-6)

    function _withdrawFromAave(uint256 amount) internal returns (uint256 withdrawnAmount) {
        if (amount == 0) return 0;

        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        if (aaveBalance == 0) revert InsufficientAaveBalance();

        // Only withdraw up to the principal amount (don't use accrued interest for user withdrawals)
        uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave;

        withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
        if (withdrawnAmount == 0) return 0;

        aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
        totalETHDepositedToAave -= withdrawnAmount;

        emit ETHWithdrawnFromAave(withdrawnAmount, totalETHDepositedToAave);
    }

    /// @dev Internal helper to check Aave health - returns false if unhealthy or disabled, doesn't revert
    /// @return healthy True if Aave integration is enabled and balance is healthy
    function _checkAaveHealth() internal view returns (bool healthy) {
        if (!isAaveIntegrationEnabled) return false;
        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        uint256 principal = totalETHDepositedToAave;
        // Allow small rounding differences (up to 2 wei)
