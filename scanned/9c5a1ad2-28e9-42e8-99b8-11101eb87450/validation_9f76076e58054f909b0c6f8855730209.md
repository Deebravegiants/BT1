### Title
Stale Chainlink Price Accepted Without Staleness Validation Enables Arbitrage on rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary
`ChainlinkPriceOracle.getAssetPrice()` calls Chainlink's `latestRoundData()` but silently discards all staleness indicators (`updatedAt`, `answeredInRound`, `roundId`). Chainlink is a push-type oracle that only updates on deviation thresholds or heartbeat intervals — exactly the "fallback oracle not always updated" pattern from M-09. The stale price flows directly into `LRTDepositPool.getRsETHAmountToMint()`, allowing any depositor to exploit the lag between true market price and the stale Chainlink price to receive more rsETH than their deposit is worth, diluting existing rsETH holders.

### Finding Description
`ChainlinkPriceOracle.getAssetPrice()` fetches price data but only extracts the `price` field, discarding all round and timestamp data:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

No check on `updatedAt`, no check on `answeredInRound >= roundId`, no maximum staleness window. Chainlink is a push-type oracle: it only pushes a new price when the deviation threshold (e.g., 0.5% for stETH/ETH) is crossed or the heartbeat (e.g., 24 hours) expires. Between updates, the on-chain price can diverge from the true market price.

This stale price is consumed in two critical paths:

**Path 1 — Direct minting in `LRTDepositPool`:**
```solidity
// contracts/LRTDepositPool.sol
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```
`lrtOracle.getAssetPrice(asset)` calls `ChainlinkPriceOracle.getAssetPrice()` live at deposit time. If the Chainlink price for the deposited asset is stale-high (true price fell but Chainlink hasn't updated), the numerator is inflated and the depositor receives more rsETH than their deposit is worth.

**Path 2 — Stored `rsETHPrice` via `updateRSETHPrice()`:**
`updateRSETHPrice()` is public and callable by anyone. It calls `_updateRsETHPrice()` → `_getTotalEthInProtocol()` → `getAssetPrice(asset)` for each supported asset. A stale Chainlink price here corrupts the stored `rsETHPrice`, which then affects all subsequent minting until the next update.

By contrast, `ChainlinkOracleForRSETHPoolCollateral` (used for pool collateral) does check `answeredInRound < roundID` but still omits the timestamp staleness check (`block.timestamp - updatedAt`), so it is also partially vulnerable.

### Impact Explanation
When a supported asset's Chainlink price is stale-high (true price dropped but Chainlink hasn't updated yet):
- A depositor submits that asset and receives `rsethAmountToMint = amount * staleHighPrice / rsETHPrice`
- The rsETH minted exceeds the true ETH value of the deposit
- The excess rsETH represents value extracted from existing rsETH holders (dilution of their share of protocol TVL)
- This is theft of unclaimed yield from existing holders, bounded by the Chainlink deviation threshold (typically 0.5–1%) but potentially larger during network congestion or market stress when Chainlink updates are delayed for hours

### Likelihood Explanation
Chainlink feeds for LST assets (stETH/ETH, ETH/ETH) have deviation thresholds of 0.5% and heartbeats of 24 hours. During normal conditions, the window is small but non-zero. During market stress (rapid LST price movements, Ethereum network congestion), Chainlink updates can lag significantly. The attack requires no special access — any depositor can call `depositAsset()` or `depositETH()` permissionlessly. The attacker only needs to monitor the gap between Chainlink's on-chain price and the true market price.

### Recommendation
Add staleness validation to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();
require(answeredInRound >= roundId, "Stale round");
require(block.timestamp - updatedAt <= MAX_STALENESS, "Price too old");
```

Define per-feed `MAX_STALENESS` values matching each feed's heartbeat. Also add the timestamp check to `ChainlinkOracleForRSETHPoolCollateral.getRate()`.

### Proof of Concept
1. stETH/ETH Chainlink feed last updated at T=0 with price `0.9995e18` (within 0.5% deviation, no update triggered).
2. At T=2h, true stETH market price drops to `0.9950e18` (0.45% drop — below deviation threshold, Chainlink does not update).
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
4. `getRsETHAmountToMint` calls `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice()` → returns stale `0.9995e18`.
5. `rsethAmountToMint = 1000e18 * 0.9995e18 / rsETHPrice` — computed using inflated stale price.
6. True value of 1000 stETH = 995 ETH; attacker receives rsETH worth ~999.5 ETH.
7. Attacker extracts ~4.5 ETH of value from existing rsETH holders per 1000 stETH deposited.

        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

            // assetER is in 1e18 precision (1.0 = 1e18)

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);
