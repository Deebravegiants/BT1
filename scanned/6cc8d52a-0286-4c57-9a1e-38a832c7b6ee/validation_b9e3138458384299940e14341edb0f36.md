### Title
ETH Deposit Limit Check Omits Incoming Amount, Allowing Deposit Cap Bypass — (File: `contracts/LRTDepositPool.sol`)

### Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric limit check: for ERC20 assets it correctly tests `totalAssetDeposits + amount > depositLimit`, but for ETH it tests only `totalAssetDeposits > depositLimit`, omitting the incoming `amount`. Any depositor can push ETH deposits arbitrarily beyond the configured cap in a single transaction.

### Finding Description
The function at lines 676–682 of `contracts/LRTDepositPool.sol` reads:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // ← amount not included
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← amount included
}
```

For ETH the guard only fires when the limit is **already** exceeded before the deposit arrives. For every ERC20 asset the guard correctly fires when the deposit **would** push the total over the limit. The two branches are inverted relative to their intended semantics, directly analogous to the prefund-calculation inversion in the reference report.

`depositETH` calls `_beforeDeposit` → `_checkIfDepositAmountExceedesCurrentLimit` → reverts with `MaximumDepositLimitReached` only when the check returns `true`. Because the ETH branch never adds `amount`, a depositor whose transaction would push the total from `limit - 1 wei` to `limit + N ETH` passes the check and the full deposit is accepted.

### Impact Explanation
The deposit limit is the protocol's primary mechanism for capping ETH exposure per asset. Bypassing it allows more ETH to be deposited than the protocol intends to deploy into EigenLayer strategies. Excess ETH that cannot be deployed sits idle in the deposit pool, earning no yield, diluting the rsETH exchange rate for all existing holders. The depositor does not lose funds, but the protocol fails to enforce its own promised cap, and all rsETH holders receive lower-than-expected yield on the excess idle ETH.

**Impact: Low** — contract fails to deliver promised returns (deposit limit not enforced for ETH), but no direct loss of principal.

### Likelihood Explanation
The entry path is the public, permissionless `depositETH` function. No special role or precondition is required beyond having ETH to deposit. The condition is met whenever `totalAssetDeposits(ETH) ≤ depositLimit`, which is the normal operating state of the protocol. Likelihood is **High**.

### Recommendation
Add `amount` to the ETH branch to match the ERC20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept
1. Admin sets `depositLimitByAsset(ETH) = 1000 ether`.
2. Current `getTotalAssetDeposits(ETH) = 999 ether`.
3. Depositor calls `depositETH{value: 500 ether}(...)`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH, 500 ether)` evaluates `999 ether > 1000 ether` → `false`.
5. Deposit is accepted; total ETH in protocol becomes `1499 ether` — 499 ETH over the intended cap.
6. The 499 ETH excess sits idle in `LRTDepositPool`, not deployed to EigenLayer, earning no yield, diluting rsETH price for all holders. [1](#0-0) [2](#0-1) [3](#0-2)
