### Title
Zero Fee Applied to Token Deposits Due to Uninitialized `tokenFeeBps` Mapping - (File: contracts/pools/RSETHPool.sol)

### Summary

`RSETHPool.sol` maintains two separate fee variables: a global `feeBps` for ETH deposits and a per-token `tokenFeeBps[token]` mapping for LST token deposits. When a new token is added via `addSupportedToken`, the `tokenFeeBps` entry for that token is never initialized and defaults to zero. Any user depositing that token pays zero protocol fees, while ETH depositors pay the intended `feeBps` rate.

### Finding Description

`RSETHPool` supports two deposit paths:

**ETH deposit** (`deposit(string referralId)`): [1](#0-0) 

Uses `feeBps` (set at initialization), so fees are correctly charged.

**Token deposit** (`deposit(address token, uint256 amount, string referralId)`): [2](#0-1) 

Uses `tokenFeeBps[token]`, a mapping that is **never set** when a token is added:

```solidity
function addSupportedToken(address token, address oracle, address bridge) external onlyRole(TIMELOCK_ROLE) {
    ...
    supportedTokenList.push(token);
    supportedTokenOracle[token] = oracle;
    tokenBridge[token] = bridge;
    // tokenFeeBps[token] is never set — defaults to 0
    emit AddSupportedToken(token, oracle, bridge);
}
``` [3](#0-2) 

Although `setTokenFeeBps` exists as a separate admin call, `addSupportedToken` does not invoke it or require it. The token is immediately usable for deposits with `tokenFeeBps[token] == 0`. [4](#0-3) 

This is structurally identical to the reference bug: one deposit path (ETH) reads the authoritative fee value; the other (token) reads a local mapping that defaults to zero, permanently or until a separate admin action is taken.

Note: `RSETHPoolNoWrapper.sol` does **not** share this bug — its token deposit path correctly uses the global `feeBps`: [5](#0-4) 

### Impact Explanation

Every token deposit made while `tokenFeeBps[token] == 0` pays zero protocol fees. The depositor receives `amountAfterFee * tokenToETHRate / rsETHToETHrate` rsETH where `amountAfterFee == amount` (no deduction). The protocol loses all fee revenue on token deposits — **theft of unclaimed yield** — for as long as the fee is not separately configured. Given that the token is immediately usable after `addSupportedToken`, the window is open from the moment of token listing.

**Impact: High** — Theft of unclaimed yield (protocol fee revenue).

### Likelihood Explanation

`addSupportedToken` is a normal operational action performed by the `TIMELOCK_ROLE`. There is no on-chain enforcement requiring `setTokenFeeBps` to be called before or atomically with token listing. Any token added without a follow-up `setTokenFeeBps` call silently operates at zero fee. This is a realistic operational omission, especially since the ETH fee path works correctly and there is no visible warning.

**Likelihood: Medium.**

### Recommendation

Initialize `tokenFeeBps[token]` inside `addSupportedToken`, defaulting to the global `feeBps` value (or requiring an explicit fee parameter):

```solidity
function addSupportedToken(
    address token,
    address oracle,
    address bridge,
    uint256 _feeBps          // add explicit fee parameter
) external onlyRole(TIMELOCK_ROLE) {
    ...
    tokenFeeBps[token] = _feeBps;
    emit AddSupportedToken(token, oracle, bridge);
    emit TokenFeeBpsSet(token, _feeBps);
}
```

This ensures the fee is always set atomically with token listing, mirroring how `feeBps` is set at initialization for ETH deposits.

### Proof of Concept

1. Admin calls `addSupportedToken(wstETH, oracle, bridge)` — `tokenFeeBps[wstETH]` remains `0`.
2. Attacker (or any user) calls `deposit(wstETH, 100e18, "")`.
3. `viewSwapRsETHAmountAndFee(100e18, wstETH)` computes `fee = 100e18 * 0 / 10_000 = 0`.
4. `amountAfterFee = 100e18` — full amount converted, zero fee collected.
5. Protocol receives 0 fee; `feeEarnedInToken[wstETH]` stays at 0.
6. Compare: an ETH depositor of equivalent value pays `feeBps / 10_000` of their deposit as fee.
