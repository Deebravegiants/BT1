Ristretto encoding is canonical (bijective), so two different 32-byte values cannot decompress to the same point. However, if a peer sends a block with vrf_value bytes that decompress successfully but whose re-compression (point.compress().to_bytes()) differs from the original bytes (which should be impossible for Ristretto but could occur due to a bug in the curve25519-dalek library version used), the random_value = hash(vrf_value.0) stored in the block header would differ from hash(point.compress().to_bytes()), causing nodes that re-derive random_value from the decompressed point to disagree with nodes that use the raw bytes, splitting the epoch rng_seed. Proof idea: fuzz all 32-byte values through CompressedRistretto::decompress() and assert that for all Some(p) results, p.compress().to_bytes() == input bytes.
