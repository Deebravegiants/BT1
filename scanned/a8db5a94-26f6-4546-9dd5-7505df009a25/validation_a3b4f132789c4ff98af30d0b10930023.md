### Title
Missing Chainlink Price Feed Staleness and Validity Checks Allow Stale Prices to Drive Incorrect rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` without validating the returned price for staleness, negativity, or feed reversion. Because this oracle is the direct price source for all LST asset valuations used in rsETH minting and TVL accounting, a stale or invalid price propagates unchecked into the core deposit and price-update flows.

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` at line 52 discards all return values from `latestRoundData()` except `price`:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol:49-55
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Three checks are absent:

1. **No staleness check** — `updatedAt` is silently discarded. There is no comparison of `block.timestamp - updatedAt` against a per-feed heartbeat threshold. Chainlink feeds for LSTs (e.g., stETH/ETH, rETH/ETH) have heartbeats of up to 24 hours; a feed that has not been updated within that window will silently return its last stored answer.

2. **No non-positive price guard** — `price` is cast directly to `uint256` without checking `price > 0`. A zero return produces `rsethAmountToMint = 0`, causing a depositor to lose their LST with no rsETH minted (if `minRSETHAmountExpected` is also 0). A negative return wraps to a huge `uint256`, causing arithmetic overflow and a revert-based DoS.

3. **No try-catch around `latestRoundData()`** — If the Chainlink aggregator's access-control list is modified (e.g., by the multisig that controls the feed), the call reverts, propagating a revert through the entire deposit and price-update path.

The stale-price path is the most impactful. `getAssetPrice()` feeds directly into two critical flows:

**Flow 1 — rsETH minting** (`LRTDepositPool.depositAsset` / `depositETH`):
```solidity
// contracts/LRTDepositPool.sol:519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [1](#0-0) 

**Flow 2 — rsETH price update** (`LRTOracle._getTotalEthInProtocol` → `_updateRsETHPrice`):
```solidity
// contracts/LRTOracle.sol:339-343
uint256 assetER = getAssetPrice(asset);
uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
``` [2](#0-1) 

Both flows consume the raw Chainlink answer with no staleness gate. [3](#0-2) 

### Impact Explanation

**Stale inflated price → theft of unclaimed yield (High)**

If a Chainlink LST/ETH feed goes stale while the LST's actual market price has fallen (e.g., a depeg event), the oracle continues to report the pre-depeg price. A depositor calling `depositAsset()` during this window receives:

```
rsethAmountToMint = (depositAmount × staleHighPrice) / rsETHPrice
```

This is more rsETH than the depositor's assets are worth at the true price. The excess rsETH represents a claim on ETH that belongs to existing rsETH holders — a direct transfer of unclaimed yield (and principal backing) from existing holders to the new depositor.

**Missing try-catch → temporary freezing of funds (Medium)**

If `latestRoundData()` reverts (feed access blocked), every call to `depositAsset()`, `depositETH()`, and `updateRSETHPrice()` reverts. Users cannot deposit, and the rsETH price cannot be updated, until the feed is restored. [4](#0-3) 

### Likelihood Explanation

Chainlink feeds on Ethereum mainnet and L2s (Arbitrum, Optimism, Base — all networks where this protocol is deployed) have documented heartbeat windows of up to 24 hours. Network congestion, gas spikes, or sequencer downtime on L2s can cause feeds to miss their heartbeat. The protocol holds multiple LST assets (stETH, rETH, ETHx, swETH, sfrxETH), each with its own feed and heartbeat. A staleness window coinciding with a depeg or price movement is a realistic, non-adversarial scenario that requires no attacker action beyond a normal deposit call. [5](#0-4) 

### Recommendation

1. **Add a per-feed staleness threshold** and revert if `block.timestamp - updatedAt > staleFeedThreshold`. Each asset's feed has a different heartbeat; the threshold must be set per feed.

2. **Add a non-positive price guard**: `require(price > 0, "Invalid price")`.

3. **Wrap `latestRoundData()` in a try-catch** and revert with a descriptive error if the call fails, rather than propagating an opaque revert.

4. **For L2 deployments**, add a Chainlink L2 sequencer uptime feed check before consuming any price data, as recommended in the [Chainlink L2 sequencer documentation](https://docs.chain.link/data-feeds/l2-sequencer-feeds).

Example staleness check:
```solidity
(uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();
require(price > 0, "Invalid price");
require(answeredInRound >= roundId, "Stale round");
require(block.timestamp - updatedAt <= staleFeedThreshold[asset], "Stale price");
```

### Proof of Concept

1. A Chainlink LST/ETH feed (e.g., stETH/ETH) misses its 24-hour heartbeat due to network congestion. The last stored answer is `1.0 ETH` (pre-depeg).
2. The actual stETH price drops to `0.95 ETH` due to a depeg event, but the feed is not updated.
3. An attacker (or any user) calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
4. `getRsETHAmountToMint()` calls `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns `1.0e18` (stale).
5. `rsethAmountToMint = (100e18 × 1.0e18) / rsETHPrice` — the depositor receives rsETH priced as if stETH = 1 ETH, but only contributed assets worth 0.95 ETH each.
6. The depositor redeems the over-minted rsETH, extracting value from existing rsETH holders.

The root cause is exclusively in `ChainlinkPriceOracle.getAssetPrice()` at line 52, which discards `updatedAt` and performs no staleness validation. [6](#0-5) [7](#0-6) [8](#0-7)
