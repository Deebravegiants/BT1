### Title
Rebasable `rewardsToken` Breaks Reward Accounting in `KernelDepositPool` — (File: `contracts/KERNEL/KernelDepositPool.sol`)

### Summary
`KernelDepositPool` computes a fixed `rewardRate` from a snapshot balance at the time `notifyRewardAmount` is called. If `rewardsToken` is a rebasable token (e.g., stETH, aTokens), the contract's actual token balance drifts independently of the fixed rate, causing reward accounting to diverge from reality. This results in either theft of unclaimed yield (negative rebase) or permanent freezing of unclaimed yield (positive rebase).

### Finding Description
In `notifyRewardAmount`, the contract measures the tokens actually received and derives a fixed `rewardRate`: [1](#0-0) 

`rewardRate` is then used in `rewardPerToken()` to accumulate rewards over time: [2](#0-1) 

And `earned()` uses this to compute each user's claimable amount: [3](#0-2) 

Finally, `getReward()` transfers the computed amount directly: [4](#0-3) 

The `rewardsToken` is set at initialization with no restriction on token type: [5](#0-4) 

If `rewardsToken` is rebasable, the contract's actual balance changes after `notifyRewardAmount` is called, but `rewardRate` remains fixed. The accounting permanently diverges:

- **Negative rebase**: The contract holds fewer tokens than `rewardRate * remainingDuration` promises. Early claimers drain the pool; late claimers' `getReward()` calls revert due to insufficient balance — their earned rewards are permanently lost.
- **Positive rebase**: The contract holds more tokens than the accounting distributes. The surplus is permanently locked with no recovery mechanism.

### Impact Explanation
**High — Theft of unclaimed yield / Permanent freezing of unclaimed yield.**

In the negative-rebase scenario, users who call `getReward()` first receive their full computed reward, draining the contract balance below what is owed to remaining stakers. Subsequent `getReward()` calls revert, permanently destroying those users' earned rewards. In the positive-rebase scenario, the surplus is irrecoverable since there is no admin sweep function for `rewardsToken`.

### Likelihood Explanation
**Medium.** The `rewardsToken` is set once at `initialize` with no type restriction or documentation warning. Any deployment that uses a yield-bearing or rebasable token (stETH, wstETH unwrapped, aTokens, etc.) as the reward token — a plausible choice given the LRT/restaking context — triggers this bug. No attacker action is required; the rebase occurs automatically via the token's own mechanism.

### Recommendation
1. Document and enforce that `rewardsToken` must not be a rebasable token. Add a NatSpec warning to `initialize` and `notifyRewardAmount`.
2. Alternatively, redesign the reward accounting to use share-based accounting (record shares at deposit, convert to amounts at claim time) so that balance changes from rebases are automatically reflected.
3. Consider adding an admin-accessible `recoverERC20` function that excludes `kernelToken` but allows recovery of surplus `rewardsToken` in the positive-rebase case.

### Proof of Concept

**Setup:**
- `rewardsToken` = a rebasable token (e.g., stETH) with current balance 1000 tokens.
- Admin calls `notifyRewardAmount(1000e18)` with `duration = 100 days`.
- `rewardRate` is set to `1000e18 / (100 days)` = ~1.157e14 tokens/second.

**Attack / Failure Scenario (negative rebase):**
1. A protocol-level negative rebase occurs (e.g., slashing event), reducing the contract's `rewardsToken` balance from 1000 to 800 tokens.
2. `rewardRate` remains unchanged; the accounting still promises 1000 tokens total.
3. Alice calls `getReward()` and receives her full computed share (e.g., 500 tokens). Contract balance: 300 tokens.
4. Bob calls `getReward()` for his computed share of 500 tokens. `safeTransfer` reverts — contract only holds 300 tokens.
5. Bob's 500 tokens of earned rewards are permanently frozen/lost. [6](#0-5)

    function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
        if (_amount == 0) revert AmountZero();

        // Prevent starting a reward period when no tokens are staked to avoid unallocated rewards
        if (totalKernelStaked == 0) revert NoStakedTokens();

        // Transfer reward tokens into the contract
        uint256 balanceBefore = rewardsToken.balanceOf(address(this));
        rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
        uint256 balanceAfter = rewardsToken.balanceOf(address(this));
        // Calculate the actual amount of tokens received in case of a transfer fee (tax)
        uint256 receivedAmount = balanceAfter - balanceBefore;

        if (block.timestamp >= finishAt) {
            rewardRate = receivedAmount / duration;
        } else {
            uint256 remaining = (finishAt - block.timestamp) * rewardRate;
            rewardRate = (receivedAmount + remaining) / duration;
        }

        if (rewardRate == 0) revert RewardRateZero();

        finishAt = block.timestamp + duration;
        updatedAt = block.timestamp;

        emit NotifyRewardAmount(receivedAmount, finishAt);
    }
