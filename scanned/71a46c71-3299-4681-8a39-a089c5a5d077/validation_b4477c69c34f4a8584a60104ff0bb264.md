### Title
No Staleness Validation in `ChainlinkPriceOracle.getAssetPrice()` Allows Stale Price to Drive Inflated rsETH Minting or Protocol Auto-Pause - (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards every validity field (`updatedAt`, `answeredInRound`, `roundId`) and does not check for a non-positive answer. The raw `price` value is returned unconditionally. This stale or invalid price propagates into two critical paths: rsETH minting for depositors and the rsETH price update used by the entire protocol.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` reads only the `answer` field from Chainlink:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L49-L55
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
``` [1](#0-0) 

No check is made on:
- `updatedAt` — whether the price was updated within the oracle's heartbeat window (staleness)
- `answeredInRound >= roundId` — whether the round is complete
- `price > 0` — whether the answer is valid

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` in the same repository performs all three checks:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L26-L36
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

The unvalidated price from `ChainlinkPriceOracle` is consumed in two critical paths:

**Path 1 — rsETH minting (`getRsETHAmountToMint`):**

```solidity
// contracts/LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [3](#0-2) 

If the Chainlink feed for an LST asset is stale (e.g., the asset has depegged but the oracle has not yet updated), the stale high price is used to compute `rsethAmountToMint`. A depositor who deposits the depegged LST receives rsETH calculated at the pre-depeg rate, extracting value from all existing rsETH holders.

**Path 2 — rsETH price update (`_updateRsETHPrice` → `_getTotalEthInProtocol`):**

```solidity
// contracts/LRTOracle.sol L339
uint256 assetER = getAssetPrice(asset);
...
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
``` [4](#0-3) 

If the oracle returns 0 (e.g., a Chainlink feed that has gone offline and returns a zeroed answer), `totalETHInProtocol` is underestimated. The resulting `newRsETHPrice` is artificially low. If the drop exceeds `pricePercentageLimit`, the downside-protection logic auto-pauses both `LRTDepositPool` and `LRTWithdrawalManager`:

```solidity
// contracts/LRTOracle.sol L277-L281
if (isPriceDecreaseOffLimit) {
    if (!lrtDepositPool.paused()) lrtDepositPool.pause();
    if (!withdrawalManager.paused()) withdrawalManager.pause();
    _pause();
    return;
}
``` [5](#0-4) 

This freezes all user deposits and withdrawals until an admin manually unpauses.

---

### Impact Explanation

**Path 1 (stale high price):** A depositor mints rsETH at a pre-depeg exchange rate for a depegged LST. The excess rsETH represents a direct dilution of all existing rsETH holders' claims on the underlying ETH — **High: Theft of unclaimed yield**.

**Path 2 (stale zero/low price):** A single stale oracle for any supported LST asset causes `_updateRsETHPrice()` to compute an artificially low rsETH price, triggering the auto-pause circuit breaker and freezing all deposits and withdrawals — **Medium: Temporary freezing of funds**.

---

### Likelihood Explanation

Chainlink oracles can become stale during periods of network congestion, oracle node downtime, or when an asset's price moves within the deviation threshold (no update triggered). LST depeg events (e.g., stETH, rETH) are historically documented. The `updateRSETHPrice()` function is publicly callable, meaning any external actor can trigger the auto-pause path at any time the oracle is stale. [6](#0-5) 

---

### Recommendation

Add staleness, round-completeness, and positivity checks to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

if (answeredInRound < roundId) revert StalePrice();
if (updatedAt == 0) revert IncompleteRound();
if (price <= 0) revert InvalidPrice();
// optionally: if (block.timestamp - updatedAt > HEARTBEAT) revert StalePrice();
```

---

### Proof of Concept

1. Assume `stETH` is a supported LST with a Chainlink feed. The feed's last update was 4 hours ago (within the 24-hour heartbeat but after a depeg event that moved the price from 1.0 ETH to 0.8 ETH).
2. The Chainlink feed still reports `price = 1e8` (stale, pre-depeg).
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
4. `getRsETHAmountToMint` computes: `rsethAmountToMint = (100e18 * 1e18) / rsETHPrice` — using the stale 1.0 ETH price instead of the real 0.8 ETH price.
5. Attacker receives ~25% more rsETH than the deposited stETH is worth in ETH terms, diluting all existing holders.

For Path 2: replace step 2 with a feed returning `price = 0`, call `updateRSETHPrice()` publicly, and observe the protocol auto-pause via the `isPriceDecreaseOffLimit` branch in `_updateRsETHPrice()`. [7](#0-6) [8](#0-7)
