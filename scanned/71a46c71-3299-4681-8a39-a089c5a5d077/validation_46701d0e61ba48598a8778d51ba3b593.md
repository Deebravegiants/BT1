### Title
Missing Maximum Upper Bound on `startTimestamp` in L2 Pool `reinitialize` Can Permanently Freeze All Deposits - (File: contracts/pools/RSETHPoolV3.sol)

### Summary
The `reinitialize` functions across all L2 pool variants validate that `_startTimestamp` is not in the past, but impose **no maximum upper bound**. A fat-finger or misconfiguration setting `_startTimestamp` to a far-future value (e.g., `type(uint256).max`) causes the `limitDailyMint` modifier to permanently revert every deposit call with `MintBeforeStartTimestamp()`. Because no setter for `startTimestamp` exists outside the one-shot reinitializer, the only recovery path is a full contract upgrade.

### Finding Description
In `RSETHPoolV3.reinitialize` (and the identical pattern in `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, and `RSETHPoolV2`), the `_startTimestamp` parameter is validated only against the past:

```solidity
// startTimestamp cannot be in the past
if (block.timestamp > _startTimestamp) {
    revert InvalidStartTimestamp();
}
dailyMintLimit = _dailyMintLimit;
startTimestamp = _startTimestamp;
``` [1](#0-0) 

There is no check that `_startTimestamp` is within a reasonable future window (e.g., `block.timestamp + MAX_FUTURE_OFFSET`). The `limitDailyMint` modifier, applied to every `deposit` entry point, gates all minting on this value:

```solidity
modifier limitDailyMint(uint256 amount, address token) {
    if (block.timestamp < startTimestamp) {
        revert MintBeforeStartTimestamp();
    }
    ...
}
``` [2](#0-1) 

If `startTimestamp` is set to any value far in the future, every call to `deposit` (ETH or token) reverts immediately. Because `reinitialize` is a `reinitializer(2)` one-shot function and no independent `setStartTimestamp` setter exists, the state cannot be corrected without a new proxy upgrade.

### Impact Explanation
All L2 user-facing deposit paths (`deposit(string)` for ETH and `deposit(address,uint256,string)` for tokens) are gated by `limitDailyMint`. A misconfigured `startTimestamp` freezes 100% of new rsETH/wrsETH minting on the affected L2 chain. Existing rsETH holders are unaffected, but no new deposits can be processed until a contract upgrade is executed. This constitutes **temporary (potentially permanent) freezing of funds in motion** — users cannot convert ETH/LSTs to rsETH on L2.

Impact rating: **Medium — Temporary freezing of funds** (upgrade required to recover; no on-chain self-correction path).

### Likelihood Explanation
Low. The `reinitialize` function is restricted to `DEFAULT_ADMIN_ROLE` and is callable only once per contract version. Triggering the freeze requires the admin to supply a grossly incorrect `_startTimestamp` (fat-finger or copy-paste error) during an upgrade transaction. This matches the original report's stated likelihood: a configuration error, not an active exploit.

### Recommendation
Add a maximum future offset check alongside the existing minimum check in every `reinitialize` overload that accepts `_startTimestamp`:

```solidity
uint256 constant MAX_START_TIMESTAMP_OFFSET = 365 days;

if (block.timestamp > _startTimestamp) revert InvalidStartTimestamp();
if (_startTimestamp > block.timestamp + MAX_START_TIMESTAMP_OFFSET) revert InvalidStartTimestamp();
```

Additionally, introduce a dedicated `setStartTimestamp` setter (restricted to `DEFAULT_ADMIN_ROLE` or `TIMELOCK_ROLE`) so that a misconfigured value can be corrected without a full proxy upgrade.

### Proof of Concept
1. Admin calls `RSETHPoolV3.reinitialize(validLimit, type(uint256).max)` during an upgrade.
2. `block.timestamp > type(uint256).max` is false → no revert; `startTimestamp = type(uint256).max` is stored.
3. Any user calls `deposit{value: 1 ether}("ref")`.
4. `limitDailyMint` evaluates `block.timestamp < type(uint256).max` → true → reverts with `MintBeforeStartTimestamp()`.
5. All deposits on this L2 pool are permanently frozen. No on-chain function exists to update `startTimestamp`; only a new proxy upgrade can recover. [5](#0-4) [6](#0-5)
