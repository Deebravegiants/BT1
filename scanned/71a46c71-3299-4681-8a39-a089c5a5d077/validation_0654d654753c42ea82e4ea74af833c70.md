### Step 2: EigenLayer's `stakerDepositShares` behavior when `podOwnerDepositShares < 0`

The `IEigenPodManager` interface explicitly documents two facts:

1. `podOwnerDepositShares` **can go negative** as a deficit when slashing occurs during the withdrawal delay: [3](#0-2) 

2. `stakerDepositShares` **returns 0** when the underlying `podOwnerDepositShares` is negative: [4](#0-3) 

`DelegationManager.getWithdrawableShares` internally calls `stakerDepositShares` on the EigenPodManager to obtain deposit shares, then multiplies by `depositScalingFactor` and `beaconChainSlashingFactor`. When `stakerDepositShares` returns 0, `getWithdrawableShares` returns 0 for the beaconChainETH strategy.

### Step 3: `getEffectivePodShares()` returns only `stakedButUnverifiedNativeETH`

The comment in `NodeDelegator.sol` says "staker balances can no longer be negative" — but this is incorrect per the EigenLayer spec. When `withdrawableShare == 0` (due to deficit) and all validators are verified (`stakedButUnverifiedNativeETH == 0`), `getEffectivePodShares()` returns 0: [5](#0-4) 

### Step 4: `totalETHInProtocol` undercounts → price drop → auto-pause

`LRTDepositPool.getETHDistributionData()` sums `getEffectivePodShares()` across all NDCs: [6](#0-5) 

`LRTOracle._getTotalEthInProtocol()` calls `getTotalAssetDeposits(ETH_TOKEN)` which routes through `getETHDistributionData()`: [7](#0-6) 

`_updateRsETHPrice()` computes `newRsETHPrice` from the undercounted `totalETHInProtocol`. If the drop exceeds `pricePercentageLimit`, it auto-pauses both `LRTDepositPool` and `LRTWithdrawalManager`: [8](#0-7)

### Verdict

### Title
Negative `podOwnerDepositShares` Deficit Causes False Price Drop and Auto-Pause of Deposits and Withdrawals — (`contracts/NodeDelegator.sol`, `contracts/LRTOracle.sol`)

### Summary
When EigenLayer's `podOwnerDepositShares` for an NDC goes negative (a documented deficit state caused by beacon-chain slashing during the withdrawal delay), `IEigenPodManager.stakerDepositShares` returns 0 per spec. This propagates through `DelegationManager.getWithdrawableShares` → `NodeDelegatorHelper.getWithdrawableShare` → `NodeDelegator.getEffectivePodShares()`, causing the NDC to report 0 ETH staked. If all validators are verified (`stakedButUnverifiedNativeETH == 0`), `totalETHInProtocol` is undercounted by the full deficit amount. When `updateRSETHPrice()` is called (public, permissionless), the computed `newRsETHPrice` drops below `highestRsethPrice` by more than `pricePercentageLimit`, triggering auto-pause of `LRTDepositPool` and `LRTWithdrawalManager`.

### Finding Description
The root cause is that `NodeDelegator.getEffectivePodShares()` relies on `DelegationManager.getWithdrawableShares` to measure ETH staked in EigenLayer. This function returns 0 for the beaconChainETH strategy whenever `podOwnerDepositShares < 0`. The developer comment "staker balances can no longer be negative" is incorrect — EigenLayer explicitly allows negative `podOwnerDepositShares` as a transient deficit mechanism. The ETH is not actually lost; it is still in the withdrawal queue and the deficit is resolved upon withdrawal completion. However, the LRT oracle treats the 0 return as a genuine loss of TVL, computes a false price drop, and triggers the auto-pause circuit breaker.

**Concrete scenario:**
- NDC has 100 ETH worth of verified validators (`stakedButUnverifiedNativeETH == 0`)
- Operator calls `initiateUnstaking` / `undelegate`, queuing a withdrawal
- During the ~7-day withdrawal delay, a beacon-chain checkpoint records a slashing event, causing `podOwnerDepositShares` to go negative
- Anyone calls `updateRSETHPrice()` (public)
- `getEffectivePodShares()` returns 0 for this NDC
- If 100 ETH represents >X% of total TVL (where X = `pricePercentageLimit`), the auto-pause fires
- `LRTDepositPool` and `LRTWithdrawalManager` are paused; all user deposits and withdrawals are frozen until admin manually unpauses

### Impact Explanation
Temporary freezing of all user deposits and withdrawals. The pause is not self-resolving — it requires an admin to call `unpause()` on both contracts. During the freeze, users cannot deposit or withdraw. The ETH is not lost, but the accounting deficit creates a false signal that triggers the safety circuit breaker.

### Likelihood Explanation
Beacon-chain slashing is a real, non-negligible event. At scale (many validators), the probability of at least one slashing event coinciding with an active withdrawal delay window is meaningful. No attacker action is required beyond calling the public `updateRSETHPrice()` function at the right moment. The trigger is permissionless.

### Recommendation
`getEffectivePodShares()` should not rely solely on `getWithdrawableShares` for the beaconChainETH strategy. It should also query `podOwnerDepositShares` directly from `IEigenPodManager` and, when negative, add the absolute deficit back to the reported value (since the ETH is still in the withdrawal queue and will be recovered). Alternatively, the auto-pause circuit breaker in `LRTOracle` should exclude transient EigenLayer deficit states from its price-drop calculation, or the oracle should be made aware of queued-but-not-yet-completed EigenLayer withdrawals.
