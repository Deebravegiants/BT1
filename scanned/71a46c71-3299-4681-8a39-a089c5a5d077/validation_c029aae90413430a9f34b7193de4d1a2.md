### Title
Missing Chainlink Price Staleness Check Allows Stale Oracle Price to Inflate rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards the `updatedAt` timestamp and `answeredInRound` fields, accepting arbitrarily stale prices. This stale price flows directly into rsETH minting calculations on mainnet, allowing a depositor to receive more rsETH than their deposited asset is worth when the Chainlink feed is stale.

### Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol`, `getAssetPrice()` fetches the Chainlink price but performs no staleness or validity checks:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

The return values `updatedAt` (timestamp of last update) and `answeredInRound` (round in which the answer was computed) are silently discarded. By contrast, the sibling contract `ChainlinkOracleForRSETHPoolCollateral` explicitly validates all three conditions before using the price:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

The stale price returned by `ChainlinkPriceOracle.getAssetPrice()` is consumed by `LRTOracle.getAssetPrice()`: [3](#0-2) 

Which is then used in two critical paths:

1. `LRTOracle._getTotalEthInProtocol()` → `_updateRsETHPrice()` — sets the global `rsETHPrice` used for all subsequent minting: [4](#0-3) 

2. `LRTDepositPool.getRsETHAmountToMint()` — directly computes how many rsETH tokens a depositor receives: [5](#0-4) 

### Impact Explanation
If a Chainlink feed for a supported LST (e.g., stETH/ETH, cbETH/ETH) becomes stale at a price higher than the current market price — for example, before a depeg event or during oracle downtime — a depositor can call `LRTDepositPool.depositAsset()` and receive rsETH computed against the inflated stale price. Since `rsethAmountToMint = (amount × staleHighPrice) / rsETHPrice`, the depositor receives more rsETH than the deposited asset is worth. This dilutes all existing rsETH holders and can cause protocol insolvency if the stale price diverges significantly from reality.

**Impact: Critical — Protocol insolvency / theft of funds from existing rsETH holders.**

### Likelihood Explanation
Chainlink feeds can go stale during network congestion, oracle node downtime, or circuit-breaker events. The absence of any heartbeat or `answeredInRound` check means the protocol silently accepts any price returned, regardless of age. This is a well-known class of vulnerability with documented real-world occurrences. Any unprivileged depositor can exploit the window between feed staleness and the next valid update.

### Recommendation
Add staleness and validity checks to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    // Optionally: if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

### Proof of Concept
1. Chainlink's stETH/ETH feed goes stale at price `1.05e18` (last update was 4 hours ago; current true price is `0.95e18` after a depeg event).
2. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, ...)`.
3. `getRsETHAmountToMint` calls `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns stale `1.05e18`.
4. `rsethAmountToMint = (100e18 × 1.05e18) / rsETHPrice` — attacker receives rsETH valued at 105 ETH for an asset worth only 95 ETH.
5. Attacker redeems rsETH for 105 ETH worth of assets, extracting 10 ETH from the protocol at the expense of other rsETH holders. [6](#0-5) [7](#0-6)
