### Title
Missing Zero-Amount Check on `rsETHAmount` in L2 Pool `deposit()` Functions - (File: `contracts/pools/RSETHPoolV3.sol`)

### Summary
All L2 pool `deposit()` functions compute `rsETHAmount` via integer division but never verify the result is non-zero before minting or transferring rsETH/wrsETH to the caller. A depositor sending a sufficiently small amount of ETH or tokens will have their funds accepted by the pool while receiving zero rsETH in return, permanently losing their deposit.

### Finding Description
In every L2 pool variant (`RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPool`, `RSETHPoolNoWrapper`, `RSETHPoolV2`, `RSETHPoolV2ExternalBridge`, `RSETHPoolV2NBA`), the output amount is computed inside `viewSwapRsETHAmountAndFee`:

**ETH path:**
```
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate
```

**Token path:**
```
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate
```

Because `rsETHToETHrate` is always `>= 1e18` (rsETH accrues value over time and starts at 1 ETH), any deposit where `amountAfterFee * 1e18 < rsETHToETHrate` truncates to zero. For example, depositing `1 wei` of ETH when `rsETHToETHrate = 1.05e18` yields `rsETHAmount = 1e18 / 1.05e18 = 0`. For token deposits, the threshold scales with `tokenToETHRate / rsETHToETHrate`.

After `viewSwapRsETHAmountAndFee` returns, the deposit function proceeds unconditionally:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
wrsETH.mint(msg.sender, rsETHAmount);   // mints 0 — user gets nothing
```

There is no `if (rsETHAmount == 0) revert` guard anywhere in the deposit path. The only input guard is `if (amount == 0) revert InvalidAmount()`, which does not protect against the truncation-to-zero case.

Contrast this with `LRTDepositPool.depositETH`, which accepts a `minRSETHAmountExpected` slippage parameter and reverts via `MinimumAmountToReceiveNotMet` if the computed amount falls below it. The L2 pool contracts have no equivalent protection.

### Impact Explanation
A depositor who sends a dust amount of ETH or tokens (e.g., 1 wei) will have their funds accepted and permanently credited to the pool's balance (or fee accounting), while receiving zero rsETH/wrsETH. The depositor has no mechanism to recover the lost funds. This matches **Low — Contract fails to deliver promised returns, but doesn't lose value** (the pool retains the ETH; the depositor loses it).

### Likelihood Explanation
This is triggered by any deposit where `amountAfterFee * 1e18 < rsETHToETHrate`. In practice this means dust-level deposits (1–2 wei of ETH). The condition is reachable by any unprivileged depositor calling `deposit()` directly with a small `msg.value`. No special setup or attacker coordination is required; it is a natural consequence of integer division with no floor check.

### Recommendation
Add a zero-amount guard on the computed output, mirroring the fix applied to the CGDA `swapExactAmountOut` function:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
if (rsETHAmount == 0) revert InvalidAmount();
```

This should be applied to both the ETH and token `deposit()` overloads in all L2 pool contracts.

### Proof of Concept
1. Deploy `RSETHPoolV3` with `feeBps = 0` and an oracle returning `rsETHToETHrate = 1.05e18`.
2. Call `deposit{value: 1}("")` (1 wei of ETH).
3. `viewSwapRsETHAmountAndFee(1)` computes `rsETHAmount = 1 * 1e18 / 1.05e18 = 0`.
4. `wrsETH.mint(msg.sender, 0)` executes — caller receives 0 wrsETH.
5. The pool's ETH balance increases by 1 wei; the depositor's balance decreases by 1 wei with no rsETH issued.

The same scenario applies to the token deposit path when `amountAfterFee * tokenToETHRate < rsETHToETHrate`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
