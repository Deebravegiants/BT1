### Title
Daily Mint Limit Permanently Consumed Without Decrement on wrsETH Redemption Enables Low-Cost Deposit DoS - (File: contracts/pools/RSETHPoolV3.sol)

### Summary

The `limitDailyMint` modifier across all L2 pool variants increments `dailyMintAmount` on every `deposit()` call but never decrements it when wrsETH is burned or redeemed. An attacker can deposit ETH to exhaust the daily mint limit, immediately sell the received wrsETH on a secondary market to recover capital, and repeat — blocking all other users from depositing for up to 24 hours at a cost of only the protocol fee and DEX slippage.

### Finding Description

The `limitDailyMint` modifier in `RSETHPoolV3.sol`, `RSETHPoolV2ExternalBridge.sol`, `RSETHPoolV3ExternalBridge.sol`, and `RSETHPoolV3WithNativeChainBridge.sol` enforces a per-day cap on wrsETH minting:

```solidity
// RSETHPoolV3.sol lines 119-124
if (dailyMintAmount + rsETHAmount > dailyMintLimit) {
    revert DailyMintLimitExceeded();
}
dailyMintAmount += rsETHAmount;
```

`dailyMintAmount` is only ever reset to `0` at the start of a new day:

```solidity
// RSETHPoolV3.sol lines 113-116
if (currentDay > lastMintDay) {
    lastMintDay = currentDay;
    dailyMintAmount = 0;
}
```

There is no code path that decrements `dailyMintAmount`. The reverse-swap function `swapAssetToPremintedRsETH()` — which accepts wrsETH back and returns pool assets — does not touch `dailyMintAmount` at all. Critically, this function is restricted to `OPERATOR_ROLE` (or `WHITELISTED_USER_ROLE` in `RSETHPoolV2ExternalBridge`), so it is not a public escape hatch for users.

Because wrsETH is a freely transferable ERC-20 token that trades on secondary markets, an attacker can:

1. Call `deposit()` with enough ETH to push `dailyMintAmount` to `dailyMintLimit`.
2. Sell the received wrsETH on a DEX for ETH, recovering most of the capital.
3. The daily limit is now exhausted; every subsequent `deposit()` call by any user reverts with `DailyMintLimitExceeded` until midnight UTC.

The net cost to the attacker is only `feeBps` (capped at 10 % in `RSETHPoolV3`, and as low as 0 bps if the admin sets it to zero) plus DEX slippage. No flash loan is required because the attacker retains the wrsETH as a liquid asset.

### Impact Explanation

All L2 deposit paths — ETH and supported LST tokens — are gated by `limitDailyMint`. Once `dailyMintAmount == dailyMintLimit`, every call to either `deposit()` overload reverts for every user for the remainder of the 24-hour window. This is a **temporary freezing of the deposit function** lasting up to 24 hours, repeated every day the attacker chooses to execute the attack. Users who wish to enter the protocol during that window are completely blocked.

Impact: **Medium — Temporary freezing of funds** (users' ability to deposit and receive wrsETH is frozen for up to 24 hours per attack cycle).

### Likelihood Explanation

- wrsETH is a liquid token traded on secondary markets; the attacker can recover capital immediately after depositing.
- The attack cost is bounded by `feeBps × dailyMintLimit / 10_000` plus DEX slippage — potentially a fraction of a percent of the limit.
- No privileged role is required; any EOA or contract can call `deposit()`.
- The attack can be repeated every day indefinitely.
- The only deterrent is the upfront capital requirement (equal to the daily mint limit in ETH value), but since the attacker recovers it via the DEX, the recurring cost is only fees and slippage.

Likelihood: **Medium**.

### Recommendation

Decrement `dailyMintAmount` whenever wrsETH is effectively returned to the system. The cleanest fix mirrors the Alchemix recommendation: add an `increase` path to the daily-limit accounting and call it inside `swapAssetToPremintedRsETH()`:

```solidity
// In swapAssetToPremintedRsETH, after computing rsETHAmount returned:
if (getCurrentDay() == lastMintDay && dailyMintAmount >= rsETHAmount) {
    dailyMintAmount -= rsETHAmount;
}
```

Additionally, consider charging a non-zero minimum `feeBps` to raise the cost of cycling attacks, and monitor `dailyMintAmount` approaching `dailyMintLimit` as an on-chain signal.

### Proof of Concept

1. Attacker calls `deposit{value: X}("")` on `RSETHPoolV3` where `X` is sized so that `viewSwapRsETHAmountAndFee(X).rsETHAmount == dailyMintLimit - dailyMintAmount` (exhausting the limit in one transaction).
2. `limitDailyMint` sets `dailyMintAmount = dailyMintLimit`. [1](#0-0) 
3. Attacker receives `wrsETH` tokens and immediately sells them on a DEX for ETH, recovering capital minus fee and slippage.
4. Any subsequent call to `deposit()` by any user reverts: `dailyMintAmount + rsETHAmount > dailyMintLimit`. [2](#0-1) 
5. `dailyMintAmount` is never decremented — `swapAssetToPremintedRsETH()` (the only reverse path) does not touch it and is operator-gated. [3](#0-2) 
6. The block persists until `getCurrentDay() > lastMintDay` resets `dailyMintAmount = 0`. [4](#0-3) 

The same root cause and exploit path apply identically to `RSETHPoolV2ExternalBridge` [5](#0-4) , `RSETHPoolV3ExternalBridge` [6](#0-5) , and `RSETHPoolV3WithNativeChainBridge` [7](#0-6) .

    function swapAssetToPremintedRsETH(
        address rsETH,
        address token,
        uint256 rsETHAmount
    )
        external
        nonReentrant
        onlySupportedTokenOrEth(token)
        onlyRole(OPERATOR_ROLE)
    {
        UtilLib.checkNonZeroAddress(rsETH);

        IRsETHTokenWrapper wrapper = IRsETHTokenWrapper(address(wrsETH));
        IERC20 tokenContract = IERC20(token);

        if (!wrapper.allowedTokens(rsETH)) revert TokenNotAllowedInWrapper();
        if (rsETHAmount == 0) revert InvalidAmount();
        if (rsETHAmount > wrapper.maxAmountToDepositBridgerAsset(rsETH)) revert ExceedsMaxAmountToDepositInWrapper();

        // Get the amount of token to transfer to the user for the given amount of rsETH provided
        uint256 tokenAmount = viewSwapAssetToPremintedRsETH(token, rsETHAmount);

        // Transfer rsETH from sender to the wrapper
        IERC20(rsETH).safeTransferFrom(msg.sender, address(wrapper), rsETHAmount);

        // Transfer the token from the pool to the sender
        if (token == ETH_IDENTIFIER) {
            if (getETHBalanceMinusFees() < tokenAmount) revert InsufficientETHBalanceForReverseSwap();
            (bool success,) = payable(msg.sender).call{ value: tokenAmount }("");
            if (!success) revert TransferFailed();
        } else {
            if (getTokenBalanceMinusFees(token) < tokenAmount) revert InsufficientAssetBalanceForReverseSwap();
            tokenContract.safeTransfer(msg.sender, tokenAmount);
        }

        emit ReverseSwapOccurred(msg.sender, rsETH, token, rsETHAmount, tokenAmount);
    }
