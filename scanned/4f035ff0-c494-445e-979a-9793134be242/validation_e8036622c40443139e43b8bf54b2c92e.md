### Title
Cross-Token Withdrawal Drains Specific Token Balance, Temporarily Freezing Depositor Funds - (File: `contracts/L2/RsETHTokenWrapper.sol`)

### Summary
`RsETHTokenWrapper` mints a single fungible `wrsETH` token regardless of which allowed token is deposited, but allows withdrawal of **any** allowed token against that fungible balance. With two allowed tokens, a holder of `wrsETH` minted from `tokenA` can fully drain the contract's `tokenB` reserves, leaving `tokenB` depositors unable to withdraw their funds.

### Finding Description
The `_deposit` function mints `wrsETH` 1:1 for any allowed token deposited, with no per-token accounting: [1](#0-0) 

The `_withdraw` function only checks that the requested `_asset` is in `allowedTokens`, then burns `wrsETH` and transfers the requested token — again with no per-token accounting: [2](#0-1) 

Because `wrsETH` is fungible across all deposited token types, any `wrsETH` holder can redeem against **any** allowed token's balance, regardless of which token they originally deposited.

The contract supports multiple allowed tokens via `addAllowedToken` (TIMELOCK_ROLE) and `reinitialize`: [3](#0-2) [4](#0-3) 

### Impact Explanation
**Medium — Temporary freezing of funds.**

Concrete call sequence:
1. `addAllowedToken(tokenB)` — admin adds a second allowed token
2. `userB.deposit(tokenB, N)` — user B deposits N tokenB, receives N wrsETH
3. `userA.deposit(tokenA, N)` — user A deposits N tokenA, receives N wrsETH
4. `userA.withdraw(tokenB, N)` — **succeeds**: burns N wrsETH from user A, transfers N tokenB out; contract now holds N tokenA, 0 tokenB
5. `userB.withdraw(tokenB, N)` — **reverts**: `safeTransfer` fails because contract holds 0 tokenB

User B's N tokenB is inaccessible via the intended path. While user B could theoretically call `withdraw(tokenA, N)` to recover value in a different token, this is not guaranteed to be equivalent in value (the two alt-rsETH tokens may trade at different rates), and the protocol's 1:1 deposit/withdraw promise is broken. There is no on-chain recourse for user B to recover their specific deposited token without admin intervention or a new tokenB deposit.

### Likelihood Explanation
**Medium.** The multi-token path is an intended production feature — `reinitialize` was specifically added to introduce a second allowed token, and `addAllowedToken` is a live admin function. Any `wrsETH` holder (including a rational arbitrageur) has direct economic incentive to exploit this if one allowed token trades at a premium to another. [5](#0-4) 

### Recommendation
Track per-token net deposits and enforce that withdrawals of a given token cannot exceed that token's net deposit balance:

```solidity
mapping(address => uint256) public tokenNetDeposits;

// in _deposit:
tokenNetDeposits[_asset] += _amount;

// in _withdraw:
require(tokenNetDeposits[_asset] >= _amount, "Insufficient per-token balance");
tokenNetDeposits[_asset] -= _amount;
```

Alternatively, restrict each `wrsETH` mint to a single canonical token and disallow multi-token support, or issue separate wrapper tokens per deposited asset.

### Proof of Concept

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity 0.8.27;

// Invariant fuzz test: per-token balance in wrapper >= per-token net deposits at all times

function testCrossTokenDrain() public {
    // Setup: two allowed tokens, each worth 1 wrsETH
    wrapper.addAllowedToken(address(tokenB)); // admin adds tokenB

    // userB deposits N tokenB
    tokenB.mint(userB, N);
    vm.prank(userB);
    tokenB.approve(address(wrapper), N);
    vm.prank(userB);
    wrapper.deposit(address(tokenB), N);
    // wrapper holds: N tokenB; wrsETH supply: N (userB holds N)

    // userA deposits N tokenA
    tokenA.mint(userA, N);
    vm.prank(userA);
    tokenA.approve(address(wrapper), N);
    vm.prank(userA);
    wrapper.deposit(address(tokenA), N);
    // wrapper holds: N tokenA, N tokenB; wrsETH supply: 2N

    // userA withdraws tokenB (cross-token drain)
    vm.prank(userA);
    wrapper.withdraw(address(tokenB), N); // succeeds
    // wrapper holds: N tokenA, 0 tokenB; wrsETH supply: N (userB still holds N)

    // userB cannot withdraw their tokenB
    vm.prank(userB);
    vm.expectRevert(); // ERC20: transfer amount exceeds balance
    wrapper.withdraw(address(tokenB), N);

    // Invariant broken: tokenB.balanceOf(wrapper) == 0 < N (userB's net deposit)
    assertGe(tokenB.balanceOf(address(wrapper)), N); // FAILS
}
