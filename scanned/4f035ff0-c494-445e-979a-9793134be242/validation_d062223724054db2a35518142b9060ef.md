### Title
Immutable `i_maxSupply` in `WrappedRSETH` permanently blocks CCIP yield delivery when cap is saturated — (`contracts/ccip/WrappedRSETH.sol`)

---

### Summary

`WrappedRSETH` stores its supply ceiling in an `immutable` variable `i_maxSupply` with no owner function to raise it. Once `totalSupply()` reaches the cap through ordinary deposits or CCIP mints, every subsequent CCIP bridge execution that calls `WrappedRSETH.mint` reverts with `MaxSupplyExceeded`, permanently preventing yield-bearing rsETH from being delivered to L2 holders.

---

### Finding Description

`WrappedRSETH.mint` enforces:

```solidity
// contracts/ccip/WrappedRSETH.sol  line 138
if (i_maxSupply != 0 && totalSupply() + amount > i_maxSupply)
    revert MaxSupplyExceeded(totalSupply() + amount);
``` [1](#0-0) 

`i_maxSupply` is set once in the constructor and declared `immutable`:

```solidity
// contracts/ccip/WrappedRSETH.sol  lines 41, 54
uint256 internal immutable i_maxSupply;
...
i_maxSupply = maxSupply_;
``` [2](#0-1) 

There is no `setMaxSupply`, `increaseMaxSupply`, or any other owner-callable function anywhere in `WrappedRSETH` that could raise the ceiling after deployment. [3](#0-2) 

On the L1 side, `L1VaultV2.bridgeRsETHToL2UsingCCIP` constructs a CCIP message that includes rsETH as a token transfer:

```solidity
// contracts/L1VaultV2.sol  lines 360-364
IERC20(address(rsETH)).safeIncreaseAllowance(address(ccipRouter), amount);
Client.EVM2AnyMessage memory message = getCCIPMessage(amount);
bytes32 messageId = ccipRouter.ccipSend{ value: msg.value }(destinationChainSelector, message);
``` [4](#0-3) 

The CCIP router locks/burns rsETH on L1 and calls `WrappedRSETH.mint` on L2. If that mint reverts, the CCIP message enters a failed state. CCIP's manual-execution retry will also revert for the same reason as long as `totalSupply() == i_maxSupply`, making the failure permanent.

---

### Impact Explanation

All rsETH that was locked/burned on L1 for the failed CCIP message cannot be recovered by users. Because rsETH accrues yield continuously via EigenLayer/Lido restaking, the L1 vault periodically bridges additional rsETH to L2 to reflect that yield. Once the cap is saturated, none of those yield-delivery bridge calls can complete. L2 `WrappedRSETH` holders permanently stop receiving their share of accrued yield. This matches the allowed impact: **Medium — Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

The precondition is that `WrappedRSETH` is deployed with a finite `maxSupply_` (i.e., `maxSupply_ != 0`). The contract explicitly supports this path and the comment on line 19 even advertises it: `"The total supply can be limited during deployment."` [5](#0-4)  Once deployed with a finite cap, organic deposit growth or a deliberate griefing deposit (any address can deposit into the L2 pool) can push `totalSupply` to the ceiling. No privileged access is required.

---

### Recommendation

1. Replace the `immutable` field with a mutable storage variable and add an `onlyOwner` function `setMaxSupply(uint256)` that can only increase (never decrease) the cap.
2. Alternatively, deploy with `maxSupply_ = 0` (unlimited) and enforce supply limits at the pool/bridge layer rather than in the token contract.
3. Add a CCIP message-recovery path (e.g., a fallback that unlocks rsETH on L1 if the L2 mint permanently fails).

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import "forge-std/Test.sol";
import "contracts/ccip/WrappedRSETH.sol";

contract MaxSupplyFreezeTest is Test {
    WrappedRSETH token;
    address minter = address(0xBEEF);
    address user   = address(0xCAFE);

    function setUp() public {
        token = new WrappedRSETH("Wrapped rsETH", "wrsETH", 18, 1000e18, address(this));
        token.grantMintRole(minter);
    }

    function test_ccipMintRevertsAfterCapSaturated() public {
        // Step 1: saturate cap via normal deposits
        vm.prank(minter);
        token.mint(user, 1000e18);          // totalSupply == i_maxSupply

        // Step 2: CCIP bridge tries to deliver yield (+1 wei)
        vm.prank(minter);
        vm.expectRevert(
            abi.encodeWithSelector(WrappedRSETH.MaxSupplyExceeded.selector, 1000e18 + 1)
        );
        token.mint(user, 1);                // reverts — yield permanently frozen

        // Step 3: no admin escape hatch exists — cap cannot be raised
        // token.setMaxSupply(2000e18);     // does not compile; function does not exist
    }
}
```

The test deploys `WrappedRSETH` with `i_maxSupply = 1000e18`, saturates it, then shows that a single-wei CCIP yield mint reverts and cannot be unblocked. [1](#0-0)
