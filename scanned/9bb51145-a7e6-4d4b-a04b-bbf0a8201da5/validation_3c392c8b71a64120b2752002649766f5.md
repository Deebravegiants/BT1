### Title
`highestRsethPrice = 0` Bootstrap DoS Permanently Blocks Non-Manager `updateRSETHPrice()` Until Manager Intervenes — (`contracts/LRTOracle.sol`)

---

### Summary

When `rsETHPrice` and `highestRsethPrice` are both `0` (their default post-deployment values) and `pricePercentageLimit > 0`, the public `updateRSETHPrice()` function will always revert with `PriceAboveDailyThreshold` for any non-manager caller. The manager-only escape hatch `updateRSETHPriceAsManager()` is the only way to bootstrap the price out of this state.

---

### Finding Description

After deployment, both `rsETHPrice` and `highestRsethPrice` are `0`. [1](#0-0) 

When `rsethSupply > 0` (i.e., rsETH has been minted via deposits), the `rsethSupply == 0` early-return branch is skipped: [2](#0-1) 

The `highestRsethPrice == 0` guard then sets `highestRsethPrice = rsETHPrice = 0` — a no-op that leaves `highestRsethPrice` at `0`: [3](#0-2) 

`newRsETHPrice` is computed from real TVL and will be a positive value (e.g., ~1 ether). The threshold check then evaluates:

```
priceDifference = newRsETHPrice - 0 = newRsETHPrice   (> 0)
pricePercentageLimit.mulWad(0)      = 0
isPriceIncreaseOffLimit             = (pricePercentageLimit > 0) && (newRsETHPrice > 0)  → TRUE
``` [4](#0-3) 

For any non-manager caller this triggers the revert: [5](#0-4) 

Because the revert rolls back all state changes (including the `highestRsethPrice = rsETHPrice` assignment at line 225), the contract returns to the exact same `0/0` state after every failed call. Every subsequent non-manager call hits the same path and reverts identically — the state never advances.

---

### Impact Explanation

**Medium. Temporary freezing of funds.**

The public `updateRSETHPrice()` is completely blocked for non-managers until a manager calls `updateRSETHPriceAsManager()`. While the manager can unblock it, until that happens:

- `rsETHPrice` remains `0`, which is used as the basis for `previousTVL` and fee calculations throughout the protocol.
- Any protocol component that reads `rsETHPrice` (e.g., withdrawal share math) operates on a stale/zero price.
- The permissionless price-update guarantee — the entire purpose of the public function — is broken.

The impact is **not** "Critical. Permanent freezing of funds" as claimed, because `updateRSETHPriceAsManager()` provides a manager escape hatch: [6](#0-5) 

Once the manager calls it, `highestRsethPrice` is set to a positive value and non-manager calls resume normally. The freeze is therefore **temporary**, not permanent.

---

### Likelihood Explanation

This is triggered by the normal deployment sequence: deploy → set `pricePercentageLimit` → users deposit (minting rsETH) → anyone calls `updateRSETHPrice()` before the manager has ever called it. No attacker action is required; it is a latent initialization bug that activates automatically on the first real-world call.

---

### Recommendation

In the `highestRsethPrice == 0` bootstrap branch, seed `highestRsethPrice` with `newRsETHPrice` (computed later) rather than the stale `rsETHPrice = 0`. Alternatively, skip the threshold check entirely when `highestRsethPrice == 0`, since there is no meaningful "previous peak" to compare against:

```solidity
if (highestRsethPrice == 0) {
    // No historical peak — skip threshold check, seed with current price
    highestRsethPrice = newRsETHPrice;
    // ... continue to update rsETHPrice and return
}
```

Or, require that `initialize` / `reinitialize` sets `highestRsethPrice` to a non-zero sentinel (e.g., `1 ether`) so the bootstrap state is never `0`.

---

### Proof of Concept

```solidity
// State: rsETHPrice = 0, highestRsethPrice = 0, pricePercentageLimit = 1e16
// rsethSupply > 0 (users have deposited)
// Caller: non-manager EOA

// Step 1: rsethSupply > 0 → skip early return
// Step 2: highestRsethPrice == 0 → highestRsethPrice = rsETHPrice = 0 (no-op)
// Step 3: newRsETHPrice = totalETHInProtocol / rsethSupply ≈ 1e18 (positive)
// Step 4: newRsETHPrice (1e18) > highestRsethPrice (0) → true
// Step 5: priceDifference = 1e18 - 0 = 1e18
// Step 6: pricePercentageLimit.mulWad(0) = 0
// Step 7: 1e18 > 0 → isPriceIncreaseOffLimit = true
// Step 8: caller is not manager → revert PriceAboveDailyThreshold ✓

// State after revert: rsETHPrice = 0, highestRsethPrice = 0 (unchanged)
// → Every subsequent non-manager call repeats steps 1–8 and reverts
```
