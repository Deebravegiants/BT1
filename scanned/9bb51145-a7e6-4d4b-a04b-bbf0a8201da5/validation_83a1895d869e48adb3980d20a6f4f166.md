### Title
ETH Deposit Limit Check Omits New Deposit Amount, Allowing Limit to Be Exceeded - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric ceiling check: for ERC-20 assets it correctly adds the incoming `amount` to `totalAssetDeposits` before comparing against the configured limit, but for ETH it omits the new deposit amount entirely. Any depositor can therefore push total ETH deposits above the protocol-configured ceiling by exactly one deposit, minting rsETH that the limit was designed to prevent.

### Finding Description
`_checkIfDepositAmountExceedesCurrentLimit` (lines 676–682) contains two branches:

```solidity
// contracts/LRTDepositPool.sol  L676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← missing `+ amount`
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

For ETH the guard evaluates `totalAssetDeposits > limit`. When `totalAssetDeposits == limit` this returns `false`, so `_beforeDeposit` does not revert, and `depositETH` proceeds to mint rsETH for the full `msg.value`. After the call `totalAssetDeposits` exceeds the configured ceiling by the deposited amount.

For ERC-20 tokens the guard correctly evaluates `totalAssetDeposits + amount > limit`, preventing any deposit that would breach the ceiling.

The call chain is:
1. `depositETH()` (L76) → `_beforeDeposit()` (L87) → `_checkIfDepositAmountExceedesCurrentLimit()` (L661) → returns `false` when at-limit → no revert
2. `_mintRsETH(rsethAmountToMint)` (L90) mints rsETH above the intended ceiling

### Impact Explanation
The deposit limit (`depositLimitByAsset`) is the protocol's primary risk-management ceiling for ETH exposure. When it is bypassed, rsETH is minted against ETH that the protocol did not intend to accept, understating the risk taken on by the system. This is directly analogous to the external report's pattern: a ceiling check that does not account for the new amount being added, allowing the tracked total to exceed the intended maximum and resulting in tokens minted beyond the sanctioned limit. Impact is classified **Low** — the contract fails to enforce its promised deposit ceiling, but no direct theft occurs in a single transaction.

### Likelihood Explanation
Any unprivileged depositor can trigger this. The condition `totalAssetDeposits == depositLimitByAsset(ETH)` is a natural boundary state that occurs whenever the pool approaches capacity. No special permissions, flash loans, or oracle manipulation are required — a standard `depositETH` call with any non-zero `msg.value` suffices.

### Recommendation
Add the incoming `amount` to the ETH branch of the check, mirroring the ERC-20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. Protocol accumulates `totalAssetDeposits(ETH) = 1000 ether` through normal usage.
3. Attacker calls `depositETH{value: 1 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `1000 ether > 1000 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for 1 ETH; total ETH deposits become 1001 ether, exceeding the configured ceiling.
6. The ERC-20 equivalent path would have reverted at step 4 (`1000 + 1 > 1000` → `true`). [1](#0-0) [2](#0-1) [3](#0-2)
