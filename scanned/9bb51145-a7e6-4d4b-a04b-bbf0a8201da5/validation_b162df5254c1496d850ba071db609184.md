### Title
Zero `rsETHAmount` Minted on Dust Deposits Due to Integer Division Truncation - (File: contracts/pools/RSETHPoolV3ExternalBridge.sol)

### Summary
The `deposit()` functions across multiple L2 pool contracts check that the input `amount` is non-zero but never verify that the computed `rsETHAmount` is non-zero before calling `wrsETH.mint(msg.sender, rsETHAmount)`. Due to integer division truncation in `viewSwapRsETHAmountAndFee`, a depositor sending a dust amount of ETH or tokens receives 0 wrsETH while their deposited assets are permanently absorbed into the pool.

### Finding Description
In `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV3.sol`, `RSETHPoolV3WithNativeChainBridge.sol`, `RSETHPoolNoWrapper.sol`, `RSETHPoolV2ExternalBridge.sol`, and `RSETHPoolV2NBA.sol`, the ETH deposit path is:

```solidity
// RSETHPoolV3ExternalBridge.sol lines 373-383
uint256 amount = msg.value;
if (amount == 0) revert InvalidAmount();

(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
wrsETH.mint(msg.sender, rsETHAmount);   // rsETHAmount can be 0
```

The rate computation in `viewSwapRsETHAmountAndFee` is:

```solidity
// RSETHPoolV3ExternalBridge.sol lines 418-427
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

When `amountAfterFee * 1e18 < rsETHToETHrate`, Solidity integer division truncates to 0. For example, with `rsETHToETHrate = 1.05e18` (a realistic value as rsETH accrues yield), any deposit where `amountAfterFee ≤ 1 wei` produces `rsETHAmount = 0`. The guard `if (amount == 0) revert InvalidAmount()` does not protect against this because `amount` is 1 (non-zero) while the derived `rsETHAmount` is 0.

The same truncation applies to the token deposit path:

```solidity
// RSETHPoolV3ExternalBridge.sol lines 442-452
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

If `tokenToETHRate < rsETHToETHrate` and `amountAfterFee` is small, `rsETHAmount` truncates to 0.

The analog to the original report is exact: the original bug passed `actualTokens = 0` to `lendingPool.deposit()` causing a revert; here `rsETHAmount = 0` is passed to `wrsETH.mint()`, which succeeds silently (standard ERC20 `_mint` with amount 0 does not revert), leaving the depositor with no receipt tokens and no recourse.

### Impact Explanation
A depositor who sends a dust ETH amount (e.g., 1 wei) receives 0 wrsETH. The deposited ETH is retained by the pool and eventually bridged to L1, but the depositor holds no claim on it. This constitutes a permanent loss of the deposited amount for the user. Impact is **Low**: the contract fails to deliver the promised wrsETH receipt tokens for a valid non-zero deposit.

### Likelihood Explanation
Any unprivileged depositor can trigger this by calling `deposit()` with `msg.value = 1 wei`. No special role or precondition is required beyond the pool being unpaused. The condition is deterministic and reproducible on every L2 pool variant. Likelihood is **Low** because the monetary loss per call is negligible (1 wei), but the code path is unconditionally reachable by any external caller.

### Recommendation
Add a post-computation zero-check on `rsETHAmount` in every `deposit()` function, mirroring the pattern used in `LRTDepositPool._beforeDeposit`:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
if (rsETHAmount == 0) revert InvalidAmount();
```

This should be applied to both the ETH and token deposit paths in `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV3.sol`, `RSETHPoolV3WithNativeChainBridge.sol`, `RSETHPoolNoWrapper.sol`, `RSETHPoolV2ExternalBridge.sol`, and `RSETHPoolV2NBA.sol`.

### Proof of Concept
1. Deploy or interact with `RSETHPoolV3ExternalBridge` on a chain where `rsETHToETHrate = 1.05e18` (realistic post-yield value).
2. Call `deposit{value: 1}("")` — `amount = 1`, passes the `amount == 0` guard.
3. `viewSwapRsETHAmountAndFee(1)` computes: `fee = 0`, `amountAfterFee = 1`, `rsETHAmount = 1 * 1e18 / 1.05e18 = 0`.
4. `wrsETH.mint(msg.sender, 0)` executes successfully; depositor receives 0 wrsETH.
5. The 1 wei ETH is credited to the pool's bridgeable balance (`getETHBalanceMinusFees()` increases by 1), permanently inaccessible to the depositor.
