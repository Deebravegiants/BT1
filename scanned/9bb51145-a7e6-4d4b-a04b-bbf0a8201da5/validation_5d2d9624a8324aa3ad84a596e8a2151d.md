### Title
Missing Chainlink Price Feed Staleness Validation Enables Stale-Price rsETH Minting — (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards every validity indicator (`updatedAt`, `answeredInRound`, `roundId`). A stale or frozen Chainlink feed is silently accepted as current. Because `LRTOracle.updateRSETHPrice()` is public and uses this oracle to compute the protocol-wide rsETH/ETH price, any caller can lock in a stale price that lets depositors mint rsETH at an incorrect rate, diluting existing holders.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the Chainlink round data but only reads the `price` field:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol  line 52
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

`updatedAt`, `answeredInRound`, and `roundId` are all silently discarded. No check is made that:
- `answeredInRound >= roundId` (round completeness)
- `updatedAt != 0` (round started)
- `updatedAt` is within an acceptable heartbeat window (staleness)
- `price > 0` (valid answer)

By contrast, the pool-level wrapper `ChainlinkOracleForRSETHPoolCollateral.getRate()` performs all three of these checks before returning a price:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol  lines 30-32
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

The unguarded `ChainlinkPriceOracle` is the oracle used by `LRTOracle` for all L1 LST assets. `LRTOracle.getAssetPrice()` delegates directly to it:

```solidity
// contracts/LRTOracle.sol  line 157
return IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset);
``` [3](#0-2) 

That price feeds `_getTotalEthInProtocol()`, which feeds `_updateRsETHPrice()`, which sets the stored `rsETHPrice` used for all deposit minting:

```solidity
// contracts/LRTOracle.sol  lines 231, 250
uint256 totalETHInProtocol = _getTotalEthInProtocol();
...
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
``` [4](#0-3) 

`updateRSETHPrice()` is an unrestricted public function — any EOA or contract can call it:

```solidity
// contracts/LRTOracle.sol  line 87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [5](#0-4) 

---

### Impact Explanation

**High — Theft of unclaimed yield / share dilution.**

If a Chainlink LST/ETH feed becomes stale at a price lower than the true current price (e.g., during network congestion, feed deprecation, or a sequencer outage on L2), an attacker can:

1. Call `updateRSETHPrice()` while the stale low price is active.
2. The computed `totalETHInProtocol` is artificially depressed, so `newRsETHPrice` is set below its true value.
3. The attacker immediately deposits ETH or an LST into `LRTDepositPool`, which mints rsETH at the artificially low price — receiving more rsETH than the deposited value warrants.
4. When the feed recovers and `rsETHPrice` is corrected upward, the attacker's excess rsETH represents a direct transfer of value from all existing rsETH holders.

The magnitude scales with the price deviation and the attacker's deposit size. A 1% stale deviation on a large deposit is a direct, quantifiable extraction of yield from existing holders.

---

### Likelihood Explanation

**Medium.** Chainlink feeds have heartbeat intervals (commonly 1 hour for LST feeds) and deviation thresholds. Staleness windows do occur during:
- Ethereum network congestion
- Chainlink node outages
- Feed deprecation/migration periods

The attack requires no special role, no governance capture, and no private key compromise. Any depositor can execute it permissionlessly by monitoring feed staleness and calling `updateRSETHPrice()` at the right moment.

---

### Recommendation

Add staleness and validity guards to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`MAX_STALENESS` should be set per-asset based on the Chainlink feed's documented heartbeat interval.

---

### Proof of Concept

1. A Chainlink LST/ETH feed (e.g., stETH/ETH) goes stale at price `P_stale < P_true` (e.g., during congestion).
2. Attacker calls `LRTOracle.updateRSETHPrice()` (public, no access control).
3. `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns `P_stale`.
4. `newRsETHPrice = (stale_TVL) / rsethSupply` — lower than true price.
5. `rsETHPrice` is written to storage at the depressed value.
6. Attacker calls `LRTDepositPool.depositAsset(stETH, largeAmount)`.
7. Deposit pool mints `rsETHAmount = largeAmount * P_true / rsETHPrice_stale` — more rsETH than deserved.
8. Feed recovers; next `updateRSETHPrice()` call raises `rsETHPrice`, but attacker holds excess rsETH backed by other users' value. [6](#0-5) [7](#0-6)

    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
