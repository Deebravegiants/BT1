### Title
Deprecated `answeredInRound < roundID` Staleness Check in `ChainlinkOracleForRSETHPoolCollateral` Can Freeze L2 Pool Deposits During Chainlink Round Gaps - (File: contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol)

---

### Summary

`ChainlinkOracleForRSETHPoolCollateral.getRate()` uses the deprecated `answeredInRound < roundID` check to detect stale prices. This check produces false-positive reverts during normal Chainlink operation — specifically when Chainlink begins a new round (e.g., due to a price deviation trigger) but has not yet finalized the answer. During that window, `roundID` is incremented while `answeredInRound` still reflects the previous round, causing `getRate()` to revert with `StalePrice`. Every L2 deposit pool that routes LST token pricing through this oracle is frozen for the duration of the gap.

---

### Finding Description

`ChainlinkOracleForRSETHPoolCollateral.getRate()` performs three checks on the Chainlink `latestRoundData()` response:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L27-32
(uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
    AggregatorV3Interface(oracle).latestRoundData();

if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [1](#0-0) 

The `answeredInRound < roundID` check was originally intended to detect rounds whose answer was carried over from a prior round. Chainlink has since deprecated this check because it is unreliable: when Chainlink's off-chain network begins a new aggregation round (triggered by a price deviation or heartbeat), `roundID` is incremented immediately, but `answeredInRound` only advances once the round is finalized on-chain. During this in-progress window — which can last seconds to minutes under network congestion — `answeredInRound < roundID` is true even though the price is not stale, causing `getRate()` to revert with `StalePrice`.

This oracle wrapper is registered as `supportedTokenOracle[token]` in every major L2 deposit pool variant: [2](#0-1) 

The `IOracle(supportedTokenOracle[token]).getRate()` call is on the critical path for all LST token deposits. If it reverts, the entire deposit flow for that token is blocked.

Additionally, `ChainlinkPriceOracle.getAssetPrice()` on L1 performs no staleness check at all — it discards `updatedAt` entirely — meaning the L1 rsETH price calculation can silently consume a stale price in the opposite failure mode:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52
(, int256 price,,,) = priceFeed.latestRoundData();
``` [3](#0-2) 

---

### Impact Explanation

When Chainlink begins a new round and `answeredInRound < roundID`, every call to `getRate()` reverts with `StalePrice`. All L2 pool deposit functions that price LST collateral through `ChainlinkOracleForRSETHPoolCollateral` revert for the duration of the gap. Users cannot enter the market. This matches the **temporary freezing of funds** impact class (Medium). [4](#0-3) 

---

### Likelihood Explanation

Chainlink price feeds on Ethereum and L2s update frequently — both on heartbeat and on price-deviation triggers. Under normal gas conditions the in-progress window is short, but under network congestion (documented by the Chainlink team to produce outlier delays of 3× the nominal heartbeat) the window can persist long enough to affect multiple user transactions. This is not a theoretical edge case; it is a documented operational characteristic of Chainlink feeds.

---

### Recommendation

Replace the deprecated `answeredInRound < roundID` check with a time-based heartbeat staleness check, and add the same check to `ChainlinkPriceOracle`:

```solidity
// ChainlinkOracleForRSETHPoolCollateral.sol
uint256 public immutable HEARTBEAT; // e.g. 3600 for a 1-hour feed

function getRate() public view returns (uint256) {
    (, int256 ethPrice,, uint256 updatedAt,) =
        AggregatorV3Interface(oracle).latestRoundData();

    if (updatedAt == 0) revert IncompleteRound();
    if (block.timestamp - updatedAt > HEARTBEAT) revert StalePrice();
    if (ethPrice <= 0) revert InvalidPrice();

    return uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
}
```

Apply the same pattern to `ChainlinkPriceOracle.getAssetPrice()`.

---

### Proof of Concept

1. Chainlink's ETH/USD feed on the target L2 begins a new aggregation round due to a price deviation trigger. `roundID` is incremented to `N+1`; `answeredInRound` remains `N`.
2. During this window, any user calls `deposit(wstETH, amount)` on `RSETHPoolV3ExternalBridge`.
3. The pool calls `IOracle(supportedTokenOracle[wstETH]).getRate()`.
4. Inside `ChainlinkOracleForRSETHPoolCollateral.getRate()`, `latestRoundData()` returns `roundID = N+1`, `answeredInRound = N`.
5. `answeredInRound < roundID` → `true` → `revert StalePrice()`.
6. The deposit reverts. All LST deposits into the pool are frozen until the round finalizes. [4](#0-3) [2](#0-1)
