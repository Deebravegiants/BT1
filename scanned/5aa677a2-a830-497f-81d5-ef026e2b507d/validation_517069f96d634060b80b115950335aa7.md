### Title
Ignored `recipient` Parameter in `bridgeTokenToL1()` Silently Misroutes Bridged Tokens - (File: contracts/bridges/SonicChainNativeTokenBridge.sol)

### Summary
`SonicChainNativeTokenBridge` implements `IL2TokenBridge`, whose interface specifies that `recipient` is "The address of the recipient on L1." However, the implementation silently ignores this parameter ‚Äî tokens are always routed to `L1Vault` via `SonicBridgeReceiver`, regardless of what `recipient` is supplied. Any caller relying on the interface contract to deliver tokens to a specified L1 address will find their tokens permanently diverted to `L1Vault`.

### Finding Description
The `IL2TokenBridge` interface defines:

```solidity
/// @param recipient The address of the recipient on L1
function bridgeTokenToL1(address recipient, uint256 amount) external payable;
``` [1](#0-0) 

The `SonicChainNativeTokenBridge` implementation accepts and validates `recipient` (non-zero check), uses it only for UID entropy and event emission, but never routes tokens to it. The actual destination is always `bridgeReceiver` (i.e., `address(this)` = `SonicBridgeReceiver`), which forwards to `L1Vault`:

```solidity
/// @dev The recipient parameter is ignored as Sonic gateway only allows contract self-claiming
function bridgeTokenToL1(address recipient, uint256 amount) external payable override nonReentrant {
    UtilLib.checkNonZeroAddress(recipient);
    ...
    sonicBridge.withdraw(uid, originalToken, amount); // always routes to bridgeReceiver
    emit TokensBridgedToL1(recipient, amount, uid, bridgeReceiver);
}
``` [2](#0-1) 

The `recipient` is used only in UID generation (line 96) and the event (line 123) ‚Äî never to control where tokens land on L1. [3](#0-2) 

### Impact Explanation
Any external caller (unprivileged user or integrating contract) that calls `bridgeTokenToL1` with their own L1 address as `recipient` ‚Äî relying on the `IL2TokenBridge` interface specification ‚Äî will have their tokens taken from them on L2 and delivered to `L1Vault` instead of their address. The tokens are not lost from the protocol but are permanently inaccessible to the intended recipient without admin intervention. This matches **Low: Contract fails to deliver promised returns, but doesn't lose value** (from the protocol's perspective), though from the caller's perspective it is a fund freeze.

### Likelihood Explanation
Any external contract or user that integrates against the `IL2TokenBridge` interface and calls `bridgeTokenToL1` with a non-`L1Vault` recipient address will trigger this. The interface gives no indication that `recipient` is ignored. The likelihood is low-to-medium since direct user calls to this bridge are possible but the primary callers are pool contracts that pass `L1Vault` as recipient anyway. [4](#0-3) 

### Recommendation
Since the Sonic gateway enforces self-claiming (the bridge contract must be the claimer), the `recipient` parameter cannot be honored. The fix is to remove `recipient` from the function signature and define a Sonic-specific interface that does not include it, or add an explicit revert if `recipient != L1Vault` to prevent callers from being misled. The current design silently accepts and ignores a parameter that the interface contract promises is meaningful.

### Proof of Concept
1. External contract reads `IL2TokenBridge` interface: `bridgeTokenToL1(address recipient, uint256 amount)` ‚Äî `recipient` is documented as "The address of the recipient on L1."
2. External contract approves `SonicChainNativeTokenBridge` to spend 100 tokens and calls `bridgeTokenToL1(externalL1Address, 100e18)`.
3. `SonicChainNativeTokenBridge` passes the non-zero check on `recipient`, pulls 100 tokens from the caller, and calls `sonicBridge.withdraw(uid, originalToken, 100e18)` ‚Äî routing to `bridgeReceiver` (`address(this)`) on L1.
4. On L1, `SonicBridgeReceiver` receives and forwards to `L1Vault`. `externalL1Address` receives nothing.
5. The caller's tokens are permanently inaccessible to them without admin recovery via `recoverTokens`. [5](#0-4)

    /// @notice Initiates a withdrawal of a specified amount of tokens to the L1Vault via SonicBridgeReceiver
    /// @dev The recipient parameter is ignored as Sonic gateway only allows contract self-claiming
    /// @dev The actual recipient will be determined by SonicBridgeReceiver which forwards to L1Vault
    /// @param recipient The intended final recipient (informational only - actual recipient is L1Vault)
    /// @param amount The amount of tokens to bridge to L1
    function bridgeTokenToL1(address recipient, uint256 amount) external payable override nonReentrant {
        UtilLib.checkNonZeroAddress(recipient);

        // recipient parameter is kept for IL2TokenBridge interface compatibility
        // but the actual flow will be: SonicBridge -> SonicBridgeReceiver -> L1Vault
        if (amount == 0) revert InvalidAmount();

        // No additional msg.value is needed for the fees
        if (msg.value != 0) revert NoMsgValueNeeded();

        token.safeTransferFrom(msg.sender, address(this), amount);

        uint256 balance = token.balanceOf(address(this));
        if (balance < amount) revert InsufficientBalance();

        // Get the original token address (validated in constructor)
        address originalToken = tokenPairs.mintedToOriginal(address(token));

        // Generate a unique UID for this transaction
        uint96 uid = uint96(
            uint256(
                keccak256(
                    abi.encodePacked(
                        block.timestamp, block.number, msg.sender, recipient, amount, tx.gasprice, bridgeReceiver
                    )
                )
            ) % type(uint96).max
        );

        // Ensure UID is not zero
        if (uid == 0) {
            uid = uint96(uint256(keccak256(abi.encodePacked(block.number, msg.sender, tx.gasprice, bridgeReceiver))));
        }

        // Store the current token balance before withdrawal
        uint256 balanceBefore = token.balanceOf(address(this));

        // Approve the Sonic bridge to spend the tokens
        token.safeIncreaseAllowance(address(sonicBridge), amount);

        // Initiate withdrawal on Sonic bridge
        // Note: Sonic gateway will only allow SonicBridgeReceiver to claim (same address as this contract)
        sonicBridge.withdraw(uid, originalToken, amount);

        // Verify tokens were transferred/burned
        uint256 balanceAfter = token.balanceOf(address(this));
        if (balanceBefore - balanceAfter != amount) {
            revert BridgeFailed();
        }

        emit TokensBridgedToL1(recipient, amount, uid, bridgeReceiver);
    }
