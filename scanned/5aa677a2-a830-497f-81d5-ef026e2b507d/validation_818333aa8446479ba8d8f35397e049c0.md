### Title
Excess ETH Permanently Locked in `MultiChainRateProvider` Due to `payable` `updateRate()` Not Forwarding Full `msg.value` - (File: contracts/cross-chain/MultiChainRateProvider.sol)

### Summary
`MultiChainRateProvider.updateRate()` is `payable` and accepts ETH from any caller, but internally sends only the on-chain `estimatedFee` per receiver to LayerZero — not `msg.value`. Any ETH in excess of the sum of per-receiver estimated fees is permanently locked in the contract, which has no withdrawal function and no `receive()` fallback.

### Finding Description
`MultiChainRateProvider.updateRate()` is declared `external payable` and is callable by any address. Inside the function, for each rate receiver, the contract calls `ILayerZeroEndpoint.estimateFees()` to compute a fee, then calls `ILayerZeroEndpoint.send{ value: estimatedFee }(...)` — forwarding only `estimatedFee`, not `msg.value`, to LayerZero. [1](#0-0) 

The loop sends exactly `estimatedFee` to LayerZero for each receiver. The refund address in the LayerZero `send` call is `payable(msg.sender)`, so LayerZero refunds any excess from `estimatedFee` back to the caller — but the contract itself only forwards `estimatedFee`, not `msg.value`. Therefore, if `msg.value > Σ estimatedFee_i`, the difference remains in the `MultiChainRateProvider` contract. [2](#0-1) 

The abstract contract defines no withdrawal function and no `receive()` fallback. Concrete implementations do not add one either, as confirmed by `CrossChainRateProvider`, which is the sibling abstract contract. [3](#0-2) 

Note the contrast: `CrossChainRateProvider.updateRate()` correctly passes `msg.value` directly to LayerZero (`send{ value: msg.value }`), so no ETH is stranded there. `MultiChainRateProvider` does not follow this pattern. [4](#0-3) 

### Impact Explanation
Any ETH sent to `MultiChainRateProvider.updateRate()` in excess of the sum of on-chain estimated LayerZero fees is permanently locked in the contract. There is no owner-callable rescue function, no `receive()` fallback that could be exploited to drain it, and no upgrade path described in the contract. The locked ETH is irrecoverable.

This matches the "Permanent freezing of funds" impact class.

### Likelihood Explanation
`updateRate()` is callable by any external address with no access control. Callers are expected to pre-compute the required ETH using `estimateTotalFee()`, but:

1. LayerZero fees fluctuate between the off-chain estimation call and the on-chain execution block.
2. Callers routinely add a buffer (e.g., 10–20%) to avoid reverts, which is standard practice for LZ integrations.
3. The `estimateTotalFee()` view itself re-estimates fees at call time, so even a careful caller who calls it in the same block may see a different fee on-chain due to gas price changes.

Every call to `updateRate()` that includes any buffer results in permanently locked ETH. Given that this function is intended to be called repeatedly (rate updates), the cumulative locked amount grows over time.

### Recommendation
Replace the per-receiver `estimatedFee` forwarding pattern with `msg.value` forwarding (as `CrossChainRateProvider` correctly does), or add a refund of excess ETH to `msg.sender` after the loop, or add an owner-only ETH rescue function. The simplest fix consistent with the sibling contract's design is:

```solidity
// Instead of: send{ value: estimatedFee }(...)
// Use:        send{ value: msg.value }(...) for a single receiver,
// or distribute msg.value proportionally across receivers,
// and refund any remainder to msg.sender after the loop.
```

### Proof of Concept

1. `MultiChainRateProvider` has two rate receivers configured.
2. `estimateTotalFee()` returns 0.01 ETH.
3. Caller calls `updateRate{ value: 0.012 ETH }()` (20% buffer).
4. Inside the loop, `estimatedFee` for receiver 0 = 0.005 ETH → sent to LZ.
5. `estimatedFee` for receiver 1 = 0.005 ETH → sent to LZ.
6. Total forwarded to LZ = 0.010 ETH. Remaining in contract = 0.002 ETH.
7. No function exists to withdraw the 0.002 ETH. It is permanently locked.
8. After N calls with buffers, the contract accumulates N × buffer ETH, all irrecoverable. [5](#0-4)

    function updateRate() external payable nonReentrant {
        uint256 latestRate = getLatestRate();

        rate = latestRate;

        lastUpdated = block.timestamp;

        bytes memory _payload = abi.encode(latestRate);

        uint256 rateReceiversLength = rateReceivers.length;

        for (uint256 i; i < rateReceiversLength;) {
            uint16 dstChainId = uint16(rateReceivers[i]._chainId);

            bytes memory remoteAndLocalAddresses = abi.encodePacked(rateReceivers[i]._contract, address(this));

            (uint256 estimatedFee,) = ILayerZeroEndpoint(layerZeroEndpoint)
                .estimateFees(dstChainId, address(this), _payload, false, bytes(""));

            ILayerZeroEndpoint(layerZeroEndpoint).send{ value: estimatedFee }(
                dstChainId, remoteAndLocalAddresses, _payload, payable(msg.sender), address(0x0), bytes("")
            );

            unchecked {
                ++i;
            }
        }

        emit RateUpdated(rate);
    }
