### Title
Stale `protocolFeeInBPS` Applied Retroactively to Pre-Update Rewards Without Prior Price Checkpoint — (File: `contracts/LRTConfig.sol`)

---

### Summary

`LRTConfig.setProtocolFeeBps()` updates the protocol fee rate without first calling `LRTOracle.updateRSETHPrice()` to settle accrued rewards at the old rate. Because `_updateRsETHPrice()` computes fees over the entire TVL increase since the last price update, a fee-rate change mid-period causes the new rate to be retroactively applied to rewards that accrued under the old rate.

---

### Finding Description

The protocol charges a management fee in `LRTOracle._updateRsETHPrice()`. The fee is computed as a fraction of the TVL increase since the last price update:

```solidity
// LRTOracle.sol line 244-246
if (!protocolPaused && totalETHInProtocol > previousTVL) {
    uint256 rewardAmount = totalETHInProtocol - previousTVL;
    protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
}
``` [1](#0-0) 

`previousTVL` is implicitly `rsethSupply * rsETHPrice` — i.e., the TVL at the time of the **last** `updateRSETHPrice()` call. All rewards accumulated since that call are multiplied by whatever `protocolFeeInBPS` is **at the moment of the next call**.

The setter in `LRTConfig` does not checkpoint fees before changing the rate:

```solidity
// LRTConfig.sol line 196-200
function setProtocolFeeBps(uint256 _protocolFeeInBPS) external onlyRole(LRTConstants.MANAGER) {
    if (_protocolFeeInBPS > 1500) revert ProtocolFeeExceedsLimit();
    protocolFeeInBPS = _protocolFeeInBPS;
    emit UpdateFee(_protocolFeeInBPS);
}
``` [2](#0-1) 

There is no call to `updateRSETHPrice()` before the assignment, so the new rate is applied to the full reward window `[lastPriceUpdate, nextPriceUpdate]`, even though part of that window accrued under the old rate.

---

### Impact Explanation

**High — Theft of unclaimed yield from rsETH holders.**

When `protocolFeeInBPS` is raised, the next `updateRSETHPrice()` call mints more rsETH as protocol fees to the treasury than was legitimately earned at the new rate. The excess fee is extracted from the reward that should have been distributed to rsETH holders as price appreciation. Concretely:

- Suppose rewards of 100 ETH accrued over 24 hours at the old rate of 5% (fee owed = 5 ETH).
- MANAGER raises the rate to 10% without triggering a price update.
- The next `updateRSETHPrice()` call computes fee = 10 ETH, minting 5 ETH worth of extra rsETH to the treasury.
- Existing rsETH holders receive a lower rsETH/ETH price than they were entitled to — their yield is stolen.

The inverse (rate decrease) causes the protocol to under-collect fees, harming the treasury. [3](#0-2) 

---

### Likelihood Explanation

**Medium.** The MANAGER role is expected to adjust `protocolFeeInBPS` as part of normal protocol governance. `updateRSETHPrice()` is a public function called periodically (by bots or users), so the gap between a fee-rate change and the next price update is a realistic and recurring window. No attacker action is required — the miscalculation occurs automatically on the next price update after any fee-rate change. [4](#0-3) 

---

### Recommendation

Call `updateRSETHPrice()` (or the internal `_updateRsETHPrice()`) inside `setProtocolFeeBps()` before changing the rate, so all rewards accrued at the old rate are settled first:

```solidity
function setProtocolFeeBps(uint256 _protocolFeeInBPS) external onlyRole(LRTConstants.MANAGER) {
    if (_protocolFeeInBPS > 1500) revert ProtocolFeeExceedsLimit();
    // Settle accrued fees at the current rate before changing it
    ILRTOracle(getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice();
    protocolFeeInBPS = _protocolFeeInBPS;
    emit UpdateFee(_protocolFeeInBPS);
}
```

---

### Proof of Concept

1. At block `T0`: `updateRSETHPrice()` is called. `rsETHPrice = 1.05 ETH`, `rsethSupply = 1000`, so `previousTVL = 1050 ETH`.
2. Over the next 24 hours, staking rewards push `totalETHInProtocol` to `1150 ETH` (`rewardAmount = 100 ETH`). `protocolFeeInBPS = 500` (5%).
3. At block `T1` (before the next price update): MANAGER calls `setProtocolFeeBps(1000)` (10%). No price update is triggered.
4. At block `T2`: Any user calls `updateRSETHPrice()`.
   - `rewardAmount = 1150 - 1050 = 100 ETH`
   - `protocolFeeInETH = 100 * 1000 / 10_000 = 10 ETH` ← should be 5 ETH
   - Extra 5 ETH worth of rsETH is minted to the treasury.
   - rsETH holders receive a price of `(1150 - 10) / 1000 = 1.14` instead of the correct `(1150 - 5) / 1000 = 1.145`.

The 5 ETH difference in fee is permanently extracted from rsETH holders' yield. [5](#0-4) [2](#0-1)
