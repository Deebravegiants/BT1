### Title
Decimal Mismatch in `RsETHTokenWrapper` and `AGETHTokenWrapper` Deposit/Withdraw Assumes 1:1 Token Correspondence Without Decimal Enforcement - (File: contracts/L2/RsETHTokenWrapper.sol, contracts/agETH/AGETHTokenWrapper.sol)

### Summary
The `_deposit` and `_withdraw` functions in `RsETHTokenWrapper` and `AGETHTokenWrapper` assume a strict 1:1 unit correspondence between the alternative token (`altRsETH`/`altAgETH`) and the wrapper token (`wrsETH`/`agETH`). No check is performed in `_addAllowedToken` or `initialize` to verify that the allowed token's decimals match the wrapper's decimals (18). If an allowed token with a different decimal count is registered, every user who calls `deposit` or `withdraw` will receive a grossly incorrect amount of wrapper tokens minted or burned.

### Finding Description
In `RsETHTokenWrapper._deposit`:

```solidity
function _deposit(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    ERC20Upgradeable(_asset).safeTransferFrom(msg.sender, address(this), _amount);
    _mint(_to, _amount);   // mints exactly _amount of wrsETH, no decimal adjustment
    emit Deposit(_asset, msg.sender, _to, _amount);
}
```

And `_withdraw`:

```solidity
function _withdraw(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    _burn(msg.sender, _amount);                          // burns _amount of wrsETH
    ERC20Upgradeable(_asset).safeTransfer(_to, _amount); // transfers _amount of altRsETH
    emit Withdraw(_asset, msg.sender, _to, _amount);
}
```

Both functions treat `_amount` as interchangeable between the altRsETH token and wrsETH. The wrapper is initialized with `__ERC20_init("rsETHWrapper", "wrsETH")`, which defaults to 18 decimals. The `_addAllowedToken` function performs no decimal check:

```solidity
function _addAllowedToken(address _asset) internal {
    UtilLib.checkNonZeroAddress(_asset);
    if (allowedTokens[_asset]) revert TokenAlreadyAllowed();
    allowedTokens[_asset] = true;
    emit TokenAdded(_asset);
}
```

The same pattern exists identically in `AGETHTokenWrapper._deposit` and `AGETHTokenWrapper._withdraw`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

### Impact Explanation
**Scenario A — altRsETH has fewer decimals (e.g., 6) than wrsETH (18):**
A user depositing 1 altRsETH (= 1e6 raw units) receives only 1e6 raw wrsETH units (= 0.000000000001 wrsETH). The user loses 1 − 1e-12 of their deposited value. This constitutes direct loss of user funds.

**Scenario B — altRsETH has more decimals (e.g., 27) than wrsETH (18):**
A user depositing 1 altRsETH (= 1e27 raw units) receives 1e27 raw wrsETH units (= 1e9 wrsETH). The user extracts 1e9 wrsETH for 1 altRsETH, draining the wrapper and causing protocol insolvency.

Impact: **Critical** — direct theft of user funds (Scenario A) or protocol insolvency (Scenario B).

### Likelihood Explanation
The `addAllowedToken` function is restricted to `TIMELOCK_ROLE`, and `initialize`/`reinitialize` are admin-gated. The vulnerability is triggered if an altRsETH or altAgETH token with a non-18 decimal count is registered — either through an administrative mistake or through a governance process that does not validate decimals. Because the contract provides no on-chain guard, the risk is entirely dependent on off-chain operational care. The original report's accepted risk ("the administrator will pay attention to the parameters") confirms this class of issue is considered real by the protocol team. Likelihood: **Low**, but the absence of any on-chain enforcement makes it a latent systemic risk. [6](#0-5) [7](#0-6) 

### Recommendation
1. In `_addAllowedToken`, enforce that the incoming token's decimals match the wrapper's decimals:
   ```solidity
   require(
       ERC20Upgradeable(_asset).decimals() == decimals(),
       "Decimal mismatch"
   );
   ```
2. Apply the same guard in `AGETHTokenWrapper._addAllowedToken` (implicitly called from `initialize`).
3. Alternatively, derive the wrapper's decimals directly from the first registered token rather than relying on the default 18.

### Proof of Concept
1. Deploy `RsETHTokenWrapper` with a legitimate 18-decimal altRsETH token.
2. Admin calls `addAllowedToken(altRsETH6)` where `altRsETH6` is a 6-decimal token (e.g., a USDC-like bridged variant).
3. User calls `deposit(altRsETH6, 1e6)` (= 1 full token).
4. `_deposit` calls `_mint(user, 1e6)` — user receives 1e6 wrsETH raw units = 0.000000000001 wrsETH.
5. User's 1 altRsETH6 token is locked in the contract, but they hold only a dust amount of wrsETH — funds are effectively lost.

Conversely, with a 27-decimal token, step 4 mints 1e27 wrsETH units (= 1 billion wrsETH) for a single token deposit, draining the entire wrapper supply. [1](#0-0) [4](#0-3)
