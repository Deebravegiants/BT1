### Title
Unvalidated Chainlink `latestRoundData()` Return Values Allow Stale Price Ingestion - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `AggregatorV3Interface.latestRoundData()` but discards all return values except `price`, performing zero validation. A stale or zero price is silently accepted and propagated into rsETH minting and withdrawal calculations.

### Finding Description
`ChainlinkPriceOracle` is the production Chainlink oracle adapter used by `LRTOracle` to price all supported LST assets (stETH, ETHx, rETH, swETH, etc.) against ETH. Its `getAssetPrice` function reads the feed but ignores `roundId`, `updatedAt`, and `answeredInRound`:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L49-55
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();   // ← all validation fields discarded
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

No checks are performed for:
- `roundId == 0` (uninitialized feed)
- `price <= 0` (invalid/negative price)
- `updatedAt == 0` (incomplete round)
- `block.timestamp - updatedAt > TIMEOUT` (stale price)
- `answeredInRound < roundId` (round not answered)

This price flows directly into the core protocol accounting:

1. `LRTOracle.getAssetPrice(asset)` → delegates to `ChainlinkPriceOracle.getAssetPrice()` [1](#0-0) 
2. `LRTDepositPool.getRsETHAmountToMint()` uses `lrtOracle.getAssetPrice(asset)` to compute how many rsETH tokens to mint per deposited LST [2](#0-1) 
3. `LRTWithdrawalManager.getExpectedAssetAmount()` uses `lrtOracle.getAssetPrice(asset)` to compute how much LST a withdrawer receives [3](#0-2) 
4. `LRTOracle._updateRsETHPrice()` calls `_getTotalEthInProtocol()` which aggregates asset prices to compute the rsETH/ETH exchange rate used protocol-wide [4](#0-3) 

The contrast with `ChainlinkOracleForRSETHPoolCollateral` (which does validate `answeredInRound < roundID`, `timestamp == 0`, and `ethPrice <= 0`) confirms the team is aware of the need for validation but did not apply it to the core oracle. [5](#0-4) 

### Impact Explanation
**Critical — Protocol insolvency / permanent fund loss.**

If a Chainlink feed returns a stale price that is significantly lower than the true market price (e.g., during a network outage or sequencer downtime on an L2 where the L1 oracle is read), a depositor calling `depositAsset()` receives an inflated rsETH mint amount:

```
rsethAmountToMint = (depositAmount * staleAssetPrice) / rsETHPrice
```

A stale price that is artificially low causes `staleAssetPrice` to be small, minting fewer rsETH — but a stale price that is artificially high (e.g., from a previous round before a price crash) causes over-minting of rsETH, diluting all existing holders and creating an under-collateralized protocol state. Conversely, a zero or negative price causes a revert or division-by-zero, temporarily freezing deposits. A stale price used in `_updateRsETHPrice()` corrupts the global rsETH/ETH rate, affecting all subsequent mints and withdrawals.

### Likelihood Explanation
**Medium.** Chainlink feeds on Ethereum mainnet are generally reliable, but the protocol is deployed across many chains (Arbitrum, Optimism, Scroll, Base, etc.) where sequencer downtime is a documented risk. During sequencer outages, Chainlink feeds can go stale for hours. The protocol explicitly lists Arbitrum and other L2s as deployment targets. No heartbeat/timeout check exists, so any feed that stops updating is silently accepted.

### Recommendation
Add the following validation to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

if (roundId == 0) revert InvalidRoundId();
if (price <= 0) revert InvalidPrice();
if (updatedAt == 0) revert IncompleteRound();
if (answeredInRound < roundId) revert StalePrice();
if (block.timestamp - updatedAt > STALENESS_TIMEOUT) revert StalePrice();
```

Use per-feed staleness timeouts (e.g., 1 hour for ETH/stETH, 24 hours for slower feeds). For L2 deployments, additionally implement a Chainlink sequencer uptime feed check per [Chainlink documentation](https://docs.chain.link/data-feeds/l2-sequencer-feeds).

### Proof of Concept

1. A Chainlink LST/ETH feed (e.g., stETH/ETH) stops updating due to a sequencer outage on Arbitrum.
2. The feed's `updatedAt` is now 6 hours old, but the last reported price was from before a 5% stETH depeg event.
3. An attacker calls `LRTDepositPool.depositAsset(stETH, amount, ...)` on L1 (where the same stale feed is read via `ChainlinkPriceOracle`).
4. `getAssetPrice(stETH)` returns the pre-depeg price (inflated by ~5%).
5. `getRsETHAmountToMint` computes: `rsethToMint = (amount * inflatedPrice) / rsETHPrice` — minting ~5% more rsETH than the deposited collateral is worth.
6. The attacker immediately withdraws via `LRTWithdrawalManager`, receiving more LST value than deposited, at the expense of existing rsETH holders.

No special privileges are required. Any user can call `depositAsset()` or `depositETH()`. [6](#0-5) [7](#0-6) [1](#0-0)
