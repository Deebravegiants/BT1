### Title
ETH Withdrawal Permanently Frozen When Recipient Contract Reverts on ETH Receive - (File: contracts/LRTWithdrawalManager.sol)

### Summary
`LRTWithdrawalManager._processWithdrawalCompletion()` sends ETH to a user-controlled address via a low-level `call`. If the recipient is a contract that reverts on ETH receive, the withdrawal can never be completed. Because rsETH is burned in a prior, separate `unlockQueue()` transaction, the user's rsETH is permanently destroyed while the corresponding ETH is permanently frozen inside `LRTWithdrawalManager`.

### Finding Description
The withdrawal lifecycle in `LRTWithdrawalManager` spans three separate transactions:

1. **`initiateWithdrawal()`** — user's rsETH is transferred into the contract.
2. **`unlockQueue()`** (operator-only) — rsETH is burned via `IRSETH.burnFrom()` and ETH is pulled from `LRTUnstakingVault` into `LRTWithdrawalManager`.
3. **`completeWithdrawal()` / `completeWithdrawalForUser()`** — ETH is sent to the user.

The ETH transfer in step 3 is performed by `_transferAsset()`:

```solidity
// LRTWithdrawalManager.sol line 877-879
if (asset == LRTConstants.ETH_TOKEN) {
    (bool sent,) = payable(to).call{ value: amount }("");
    if (!sent) revert EthTransferFailed();
}
``` [1](#0-0) 

If `to` is a contract without a `receive()` / `fallback()` function, or one that explicitly reverts on ETH receipt, `sent` is `false` and `EthTransferFailed` is thrown. Because Solidity reverts all state changes in the failing transaction, the withdrawal request record and `unlockedWithdrawalsCount` are restored — but the rsETH burned in the prior `unlockQueue()` transaction is **not** restored. The ETH now sits in `LRTWithdrawalManager` with no mechanism to release it to the user.

The rsETH burn happens in `unlockQueue()`:

```solidity
// LRTWithdrawalManager.sol line 305
if (rsETHBurned != 0) IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned);
``` [2](#0-1) 

The operator-assisted path `completeWithdrawalForUser()` calls the same `_processWithdrawalCompletion()` and therefore also always reverts for such a user:

```solidity
// LRTWithdrawalManager.sol line 202
_processWithdrawalCompletion(asset, user, referralId);
``` [3](#0-2) 

The only potential escape valve, `sweepRemainingAssets()`, is gated by `hasUnlockedWithdrawals(asset)`:

```solidity
// LRTWithdrawalManager.sol line 403
if (hasUnlockedWithdrawals(asset)) revert PendingWithdrawalsExist();
``` [4](#0-3) 

Because the stuck withdrawal keeps `unlockedWithdrawalsCount[asset] > 0`, `sweepRemainingAssets()` is permanently blocked for that asset as well. There is no admin rescue path.

### Impact Explanation
**Critical — Permanent freezing of funds.**

After `unlockQueue()` commits, the user's rsETH is irreversibly burned and the equivalent ETH is held in `LRTWithdrawalManager`. If the user's address cannot accept ETH, both assets are permanently lost: rsETH is gone and ETH is unrecoverable. No admin function exists to force-redirect the ETH to an alternative address.

### Likelihood Explanation
**Medium.**

Smart-contract wallets (multisigs, DAO treasuries, protocol vaults) that interact with LRT-rsETH are common and many do not implement a `receive()` function. A user operating through such a contract who requests an ETH withdrawal will trigger this freeze automatically — no malicious intent is required. The operator's `unlockQueue()` call, which is a routine operational step, is the irreversible trigger.

### Recommendation
Replace the bare `call` with a pull-payment pattern: instead of pushing ETH to the user in `_processWithdrawalCompletion()`, record the owed amount in a claimable balance mapping and let the user (or any address they designate) pull it in a separate `claimETH()` function. Alternatively, allow the user to specify a separate `recipient` address at withdrawal-completion time so they can redirect ETH to an EOA even if the initiating contract cannot receive it.

### Proof of Concept
1. Deploy `MaliciousVault` — a contract with no `receive()` function.
2. From `MaliciousVault`, call `LRTWithdrawalManager.initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. rsETH is now held by the withdrawal manager.
3. Operator calls `unlockQueue(ETH_TOKEN, ...)`. rsETH is burned; ETH is transferred from `LRTUnstakingVault` into `LRTWithdrawalManager`. **This transaction commits.**
4. Anyone calls `completeWithdrawal(ETH_TOKEN, "")` on behalf of `MaliciousVault`. `_transferAsset` executes `payable(MaliciousVault).call{value: amount}("")`; the call returns `false`; `EthTransferFailed` is thrown. Transaction reverts.
5. Step 4 can be repeated indefinitely — it always reverts.
6. `sweepRemainingAssets(ETH_TOKEN)` reverts with `PendingWithdrawalsExist`.
7. rsETH is permanently burned; ETH is permanently frozen in `LRTWithdrawalManager`.
