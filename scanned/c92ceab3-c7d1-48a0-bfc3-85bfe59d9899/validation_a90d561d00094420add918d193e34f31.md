### Title
Aave WETH Liquidity Exhaustion Blocks All ETH `completeWithdrawal` Calls With No Working Escape Hatch — (`contracts/LRTWithdrawalManager.sol`)

---

### Summary

When `isAaveIntegrationEnabled` is `true` and the Aave WETH pool reaches 100% utilization, `_withdrawFromAave` reverts on the `aaveWETHGateway.withdrawETH` call. This revert propagates uncaught through `_processWithdrawalCompletion`, causing every ETH `completeWithdrawal` call to revert. Critically, every administrative escape hatch (`emergencyWithdrawFromAave`, `setAaveIntegrationEnabled(false)`, `configureAaveIntegration`) also calls `_withdrawFromAave` and fails under the same condition, leaving no recovery path until Aave liquidity naturally recovers.

---

### Finding Description

**Withdrawal path — no try/catch:**

In `_processWithdrawalCompletion`, when the contract's ETH balance is insufficient, it calls `_withdrawFromAave` with no error handling: [1](#0-0) 

Inside `_withdrawFromAave`, the call to `aaveWETHGateway.withdrawETH` is bare — any revert from Aave propagates directly: [2](#0-1) 

**Contrast with the deposit path**, which correctly uses `try/catch`: [3](#0-2) 

**All escape hatches are broken under the same condition:**

- `emergencyWithdrawFromAave` (PAUSER_ROLE): calls `_withdrawFromAave` at line 560 — reverts. [4](#0-3) 

- `setAaveIntegrationEnabled(false)` (LRT Manager): calls `_withdrawFromAave` at line 495 — reverts. [5](#0-4) 

- `configureAaveIntegration` (LRT Manager): calls `_withdrawFromAave` at line 447 — reverts. [6](#0-5) 

None of these wrap `_withdrawFromAave` in a try/catch or provide a force-skip path. The manager cannot set `isAaveIntegrationEnabled = false` without first successfully draining Aave, which is impossible when utilization is 100%.

---

### Impact Explanation

**Impact: Medium — Temporary freezing of funds.**

All ETH `completeWithdrawal` calls revert for every user with a pending unlocked ETH withdrawal request, for as long as Aave WETH utilization remains at or near 100%. The funds are not lost (aWETH balance is intact), but they are inaccessible. The "permanent" framing in the question overstates the risk: Aave's interest rate model makes 100% utilization economically self-correcting as borrowing costs spike, so recovery is expected. However, the duration is entirely outside the protocol's control, and there is no administrative action that can unblock withdrawals while the condition persists.

---

### Likelihood Explanation

Aave WETH utilization reaching 100% is a realistic, historically observed market condition (e.g., during high ETH demand periods or market stress). The preconditions — `isAaveIntegrationEnabled = true`, contract ETH balance below the withdrawal amount, and Aave at max utilization — can all occur simultaneously in normal operation, since the protocol actively deposits unlocked ETH into Aave via `unlockQueue`.

---

### Recommendation

1. **Wrap `_withdrawFromAave` in `_processWithdrawalCompletion` with a try/catch.** If the Aave withdrawal fails, fall through to the `InsufficientLiquidityForWithdrawal` revert (already present at line 729) rather than propagating the Aave revert. This preserves the existing error semantics while preventing opaque Aave errors from surfacing.

2. **Decouple `setAaveIntegrationEnabled(false)` from `_withdrawFromAave`.** Allow the manager to set `isAaveIntegrationEnabled = false` even when Aave cannot be drained (e.g., skip the withdrawal if Aave reverts, leaving aWETH in place to be claimed later via a separate function). This restores the escape hatch.

3. **Add a `forceDisableAaveIntegration` function** that sets `isAaveIntegrationEnabled = false` without attempting to withdraw, for use when Aave is illiquid.

---

### Proof of Concept

```solidity
// Fork mainnet with Aave WETH at 100% utilization
// 1. Enable Aave integration and deposit ETH to Aave via unlockQueue
// 2. Drain Aave WETH liquidity (simulate all WETH borrowed)
// 3. Call completeWithdrawal for an unlocked ETH request
//    where address(this).balance < request.expectedAssetAmount
// Expected: reverts with Aave's "not enough liquidity" error
// 4. Call emergencyWithdrawFromAave — also reverts
// 5. Call setAaveIntegrationEnabled(false) — also reverts
// Result: no administrative action can unblock withdrawals

function testAaveLiquidityExhaustionBlocksWithdrawals() public {
    // Setup: fork mainnet, configure Aave integration, unlock ETH withdrawal
    // Drain Aave WETH pool to 0 available liquidity
    vm.mockCall(
        address(aaveWETHGateway),
        abi.encodeWithSelector(IWrappedTokenGatewayV3.withdrawETH.selector),
        abi.encode() // force revert by removing mock or using vm.mockCallRevert
    );
    vm.expectRevert();
    withdrawalManager.completeWithdrawal(LRTConstants.ETH_TOKEN, "");

    vm.expectRevert();
    withdrawalManager.emergencyWithdrawFromAave(type(uint256).max);

    vm.expectRevert();
    withdrawalManager.setAaveIntegrationEnabled(false);
}
```
