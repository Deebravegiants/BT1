### Title
Non-Monotonic Merkle Root Causes Permanent Underflow Revert in `claim()` — (`contracts/utils/MerkleDistributor/MerkleBlastPointsDistributor.sol`)

---

### Summary

`MerkleBlastPointsDistributor.claim()` computes incremental rewards by subtracting the previously stored cumulative amount from the new leaf value. Because Solidity 0.8 reverts on unsigned integer underflow, any Merkle root whose leaf encodes a cumulative amount smaller than the user's stored value causes an irrecoverable revert. The contract has no on-chain monotonicity guard and no recovery path.

---

### Finding Description

After a successful claim at index N, the contract stores:

```
userClaims[account].cumulativeBlastPointAmount = X
userClaims[account].lastClaimedIndex          = N
``` [1](#0-0) 

When the owner later calls `setMerkleRoot()` to publish index M > N, the contract performs no on-chain check that the new leaf values are ≥ the previously stored values. [2](#0-1) 

If the new leaf encodes `cumulativeBlastPointAmount = Y` where `Y < X`, the subtraction at line 117 underflows:

```solidity
uint256 claimableBlastPoints =
    cumulativeBlastPointAmount - userClaims[account].cumulativeBlastPointAmount;
// Y - X  →  arithmetic underflow → revert (Solidity 0.8)
``` [3](#0-2) 

Because the revert happens after the Merkle proof passes but before any state update, the user's `lastClaimedIndex` remains at N. The user cannot claim at M (underflow), and cannot skip to M+1 unless the admin publishes a corrected root. The incremental rewards encoded in the bad root are permanently unclaimable.

The same underflow applies symmetrically to `cumulativeBlastGoldAmount` on line 118.

---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

Blast Points are off-chain; no ERC-20 tokens are transferred by this contract. However, the user is denied the incremental Blast Points/Gold rewards they are entitled to for the period covered by index M. The denial is permanent for that index with no user-side recovery path.

---

### Likelihood Explanation

Requires the off-chain Merkle tree generation pipeline to produce a non-monotonic cumulative value for at least one user — plausible via a data pipeline bug, rebase of the accounting dataset, or an epoch-reset error. It does not require malicious admin intent; an honest off-chain mistake is sufficient. No special permissions are needed by the user to trigger the revert once the bad root is published.

---

### Recommendation

Add an explicit monotonicity guard before the subtraction:

```solidity
if (cumulativeBlastPointAmount < userClaims[account].cumulativeBlastPointAmount) {
    revert InvalidCumulativeAmount();
}
if (cumulativeBlastGoldAmount < userClaims[account].cumulativeBlastGoldAmount) {
    revert InvalidCumulativeAmount();
}
```

This converts a silent permanent lock into a clear, diagnosable revert that the admin can resolve by publishing a corrected root, without permanently blocking the user.
