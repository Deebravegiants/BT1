### Title
Rewards emitted while `totalKernelStaked == 0` during an active reward window are permanently frozen in `KernelDepositPool` — (File: `contracts/KERNEL/KernelDepositPool.sol`)

---

### Summary

In `KernelDepositPool`, the `updateReward` modifier always advances `updatedAt` to `lastTimeRewardApplicable()`, even when `totalKernelStaked == 0`. Because `rewardPerToken()` returns the unchanged `rewardPerTokenStored` when `totalKernelStaked == 0`, any rewards emitted at `rewardRate` during a zero-staked interval are silently skipped and permanently locked in the contract.

---

### Finding Description

`KernelDepositPool` uses a Synthetix-style reward accounting model. The `updateReward` modifier snapshots the global reward index and advances the timestamp on every user interaction:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L232-242
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();   // (A)
    updatedAt = lastTimeRewardApplicable();    // (B)
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

`rewardPerToken()` short-circuits when `totalKernelStaked == 0`:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L408-414
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;   // no accumulation
    }
    return rewardPerTokenStored
        + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
```

Step (A) leaves `rewardPerTokenStored` unchanged, but step (B) still advances `updatedAt`. The time gap during which `totalKernelStaked == 0` is therefore consumed without distributing any rewards. When the next staker arrives, `updateReward` is called before the new tokens are credited, so `totalKernelStaked` is still 0 at that moment, `rewardPerToken()` again returns the stale `rewardPerTokenStored`, and `updatedAt` is pushed forward once more. The rewards that should have accrued over the entire zero-staked interval are permanently irrecoverable.

The `notifyRewardAmount` guard prevents *starting* a new reward period with zero stakers:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L570
if (totalKernelStaked == 0) revert NoStakedTokens();
```

But it provides no protection against all stakers withdrawing *during* an active period. `initiateWithdrawal` decrements `totalKernelStaked` immediately:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L325-326
balanceOf[msg.sender] -= _amount;
totalKernelStaked -= _amount;
```

The contract's own NatSpec comment acknowledges the risk but relies on an off-chain operational promise rather than an on-chain invariant:

```
// contracts/KERNEL/KernelDepositPool.sol L18-22
// If `totalKernelStaked` ever hits zero during a reward distribution window,
// any remaining rewards for that period will stay locked in the contract.
// In this deployment, we're avoiding this issue by ensuring there are always
// some tokens staked...
```

---

### Impact Explanation

Any rewards emitted at `rewardRate` during the interval `[t_zero_staked, t_next_stake]` are permanently frozen in the contract. No user can ever claim them. This matches **Medium — Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

The scenario requires all current stakers to call `initiateWithdrawal` before any new staker arrives. This is realistic during low-activity periods, market downturns, or if the staker set is small. No admin action or privileged key is needed — ordinary stakers trigger it through the public `initiateWithdrawal` entry point.

---

### Recommendation

Two complementary fixes:

1. **Track elapsed zero-staked time and skip it.** When `totalKernelStaked == 0`, do not advance `updatedAt` in the modifier. Change line (B) to:

```solidity
if (totalKernelStaked > 0) {
    updatedAt = lastTimeRewardApplicable();
}
```

This mirrors the approach recommended in the external report: only advance the accounting clock when there are stakers to receive rewards.

2. **Alternatively, carry forward unallocated rewards.** When `notifyRewardAmount` is called again after a zero-staked gap, compute the unallocated rewards and roll them into the new `rewardRate`, so no tokens are stranded.

    function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
        if (_amount == 0) revert AmountZero();

        // Prevent starting a reward period when no tokens are staked to avoid unallocated rewards
        if (totalKernelStaked == 0) revert NoStakedTokens();

        // Transfer reward tokens into the contract
        uint256 balanceBefore = rewardsToken.balanceOf(address(this));
        rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
        uint256 balanceAfter = rewardsToken.balanceOf(address(this));
        // Calculate the actual amount of tokens received in case of a transfer fee (tax)
        uint256 receivedAmount = balanceAfter - balanceBefore;

        if (block.timestamp >= finishAt) {
            rewardRate = receivedAmount / duration;
        } else {
            uint256 remaining = (finishAt - block.timestamp) * rewardRate;
            rewardRate = (receivedAmount + remaining) / duration;
        }

        if (rewardRate == 0) revert RewardRateZero();

        finishAt = block.timestamp + duration;
        updatedAt = block.timestamp;

        emit NotifyRewardAmount(receivedAmount, finishAt);
    }
