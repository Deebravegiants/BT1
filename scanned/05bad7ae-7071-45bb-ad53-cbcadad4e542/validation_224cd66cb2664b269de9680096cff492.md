### Title
`_updateRsETHPrice` price-increase guard is bypassed when rsETH price is below its all-time high, enabling front-running of price updates — (File: `contracts/LRTOracle.sol`)

---

### Summary

`LRTOracle._updateRsETHPrice()` applies the `pricePercentageLimit` guard **only** when `newRsETHPrice > highestRsethPrice`. When the rsETH price is recovering from a prior dip (still below `highestRsethPrice`), the guard is silently skipped, allowing a price jump larger than the configured daily limit to go through unchecked. Because `updateRSETHPrice()` is a public, permissionless function, any attacker can atomically deposit ETH at the stale low price, trigger the price update themselves, and immediately extract the accrued yield — stealing rewards that belong to existing rsETH holders.

---

### Finding Description

In `LRTOracle._updateRsETHPrice()`, the `pricePercentageLimit` check is the protocol's sole mechanism to rate-limit price increases and prevent sandwiching of price updates:

```solidity
// contracts/LRTOracle.sol  lines 252-267
if (newRsETHPrice > highestRsethPrice) {
    uint256 priceDifference = newRsETHPrice - highestRsethPrice;
    bool isPriceIncreaseOffLimit =
        pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);

    if (isPriceIncreaseOffLimit) {
        if (!IAccessControl(address(lrtConfig)).hasRole(LRTConstants.MANAGER, msg.sender)) {
            revert PriceAboveDailyThreshold();
        }
    }
}
```

The outer `if` condition gates the entire check on `newRsETHPrice > highestRsethPrice`. Whenever the current price is below the all-time high (e.g., after a partial slashing event or a temporary dip that did not trigger the pause), the block is never entered, and **no limit is enforced on the magnitude of the price increase**.

This is structurally identical to the reported Bunni H-03 pattern: a protective check that is only applied in one branch of a two-branch condition, leaving the other branch completely unguarded.

The entry path is fully permissionless:

1. `LRTDepositPool.depositETH()` — public, callable by any EOA. [1](#0-0) 
2. `LRTOracle.updateRSETHPrice()` — public, callable by any EOA. [2](#0-1) 
3. `LRTWithdrawalManager.instantWithdrawal()` (if enabled) or secondary-market sale of rsETH.

The price guard that is bypassed: [3](#0-2) 

The price computation that produces the unchecked jump: [4](#0-3) 

The deposit function that uses the stale stored price: [5](#0-4) 

---

### Impact Explanation

When the guard is bypassed, the price can increase by an arbitrarily large percentage in a single `updateRSETHPrice()` call (bounded only by accumulated staking rewards since the last update). An attacker who deposits ETH immediately before triggering this update receives rsETH priced at the old, lower rate. After the update, their rsETH is worth proportionally more ETH than they deposited. The surplus comes directly from the yield that should have accrued to existing rsETH holders — this is **theft of unclaimed yield**.

**Impact: High** — theft of unclaimed yield from existing rsETH holders.

---

### Likelihood Explanation

The preconditions are realistic and observable on-chain:

- The rsETH price must have previously dipped below `highestRsethPrice` without triggering the pause (i.e., the dip was within `pricePercentageLimit`).
- `updateRSETHPrice()` must not have been called for several days, allowing staking rewards to accumulate such that the pending price increase exceeds `pricePercentageLimit`.
- Both conditions are visible on-chain before the attacker acts.

**Likelihood: Medium** — requires a prior dip and a multi-day gap between price updates, both of which are observable and not uncommon in practice.

---

### Recommendation

Replace the comparison against `highestRsethPrice` with a comparison against the **previous stored price** (`rsETHPrice`) so that the daily rate-limit is enforced on every price increase, regardless of whether the new price is above or below the all-time high:

```solidity
// Apply the limit to any upward move, not only moves above the ATH
if (newRsETHPrice > rsETHPrice) {
    uint256 priceDifference = newRsETHPrice - rsETHPrice;
    bool isPriceIncreaseOffLimit =
        pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(rsETHPrice);

    if (isPriceIncreaseOffLimit) {
        if (!IAccessControl(address(lrtConfig)).hasRole(LRTConstants.MANAGER, msg.sender)) {
            revert PriceAboveDailyThreshold();
        }
    }
}
```

This mirrors the fix recommended in the Bunni H-03 report: apply the protective check in **all** cases where a price increase occurs, not only in the subset of cases the original author anticipated.

### Proof of Concept

**Setup:**
- `highestRsethPrice = 1.05e18`
- `rsETHPrice = 1.03e18` (price dipped 1.9% previously, within `pricePercentageLimit`, no pause triggered)
- `pricePercentageLimit = 1e16` (1% per day)
- `updateRSETHPrice()` has not been called for 2 days; accumulated staking rewards will push the price to `1.049e18`

**Attack (single block, or two sequential transactions):**

1. Attacker calls `LRTDepositPool.depositETH{value: 100 ether}(minExpected, "")`.
   - `getRsETHAmountToMint` uses stored `rsETHPrice = 1.03e18`.
   - Attacker receives `100e18 / 1.03e18 ≈ 97.087 rsETH`. [6](#0-5) 

2. Attacker calls `LRTOracle.updateRSETHPrice()`.
   - `newRsETHPrice = 1.049e18`.
   - Check at line 252: `1.049e18 < 1.05e18 = highestRsethPrice` → **guard block is skipped entirely**.
   - `rsETHPrice` is updated to `1.049e18`. [3](#0-2) 

3. Attacker calls `LRTWithdrawalManager.instantWithdrawal(ETH_TOKEN, 97.087 rsETH, "")` (if instant withdrawal is enabled) or sells rsETH on a secondary market.
   - `getExpectedAssetAmount` uses new `rsETHPrice = 1.049e18`.
   - Attacker receives `97.087 × 1.049 ≈ 101.84 ETH`.

**Result:** Attacker deposited 100 ETH and extracted ≈ 101.84 ETH — a profit of ≈ 1.84 ETH. The `pricePercentageLimit` of 1% would have blocked this update (the increase from 1.03 to 1.049 is 1.84%), but the guard was never reached because `1.049 < highestRsethPrice`.
