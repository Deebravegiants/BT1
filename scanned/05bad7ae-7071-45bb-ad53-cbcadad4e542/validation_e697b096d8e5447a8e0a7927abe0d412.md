### Title
No Staleness Check on Cross-Chain Rate Allows Depositors to Mint Excess wrsETH at Stale Rate - (File: contracts/cross-chain/CrossChainRateReceiver.sol)

---

### Summary

The `CrossChainRateReceiver` contract stores a `lastUpdated` timestamp every time a rate is received via LayerZero, but `getRate()` returns the stored `rate` unconditionally without ever checking whether the rate is fresh. All L2 pool variants (`RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolNoWrapper`) use this oracle as `rsETHOracle` and call `getRate()` directly to compute how many wrsETH/rsETH tokens to mint per deposit. When the cross-chain rate is stale, any depositor can mint more tokens than the deposited ETH/LST is actually worth in rsETH terms, diluting existing holders.

---

### Finding Description

`CrossChainRateReceiver.lzReceive()` records `lastUpdated = block.timestamp` each time a rate message arrives from L1 via LayerZero. [1](#0-0) 

However, `getRate()` simply returns the stored `rate` with no freshness guard: [2](#0-1) 

Every pool variant calls this function to price deposits:

```solidity
uint256 rsETHToETHrate = getRate();          // stale rate accepted silently
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [3](#0-2) [4](#0-3) [5](#0-4) 

rsETH is an LRT whose exchange rate monotonically increases over time as EigenLayer staking rewards accumulate. A stale (lower-than-current) rate causes the denominator in the mint formula to be smaller than it should be, so `rsETHAmount` is larger than the deposited ETH actually warrants. The protocol tracks the information needed to detect this (`lastUpdated`) but never uses it.

The analog to the external report is exact: just as the iOS build pipeline trusted a GitHub Action pinned to a mutable tag without verifying the action had not been silently changed, the L2 pools trust the cross-chain rate without verifying it has not gone stale — in both cases the trusted component can diverge from its expected state without any on-chain detection.

---

### Impact Explanation

rsETH's exchange rate grows continuously. If the LayerZero rate update is delayed (network congestion, keeper downtime, or simply no one calling `updateRate()` on `MultiChainRateProvider`), the stored `rate` on L2 is lower than the true L1 rate. A depositor who calls `deposit()` during this window receives more wrsETH than the deposited ETH is worth. When the pool's ETH is eventually bridged to L1 and converted to rsETH, the rsETH minted is less than the wrsETH already outstanding. Existing wrsETH holders bear the shortfall — their shares are diluted. This constitutes **theft of unclaimed yield** from existing holders (High impact per the allowed scope).

---

### Likelihood Explanation

`updateRate()` on `MultiChainRateProvider` is permissionless and must be called manually (or by a keeper) and paid for with ETH for LayerZero fees. [6](#0-5) 

Keeper downtime, LayerZero congestion, or simply a gap between keeper runs creates windows where the rate is stale. The protocol deploys on multiple L2s (Arbitrum, Optimism, Base, Linea, Unichain), multiplying the attack surface. Any depositor can observe `lastUpdated` on-chain and time their deposit to coincide with a stale window.

---

### Recommendation

Add a maximum staleness threshold to `getRate()` in `CrossChainRateReceiver`:

```solidity
uint256 public constant MAX_RATE_STALENESS = 24 hours;

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= MAX_RATE_STALENESS, "Rate is stale");
    return rate;
}
```

This mirrors the standard Chainlink staleness pattern and ensures the pools revert rather than mint at an outdated rate.

---

### Proof of Concept

1. On L1, `LRTOracle.rsETHPrice` is `1.05e18` (rsETH has appreciated 5% since last L2 update).
2. The LayerZero keeper has not called `updateRate()` for 30 hours; `CrossChainRateReceiver.rate` is still `1.00e18` and `lastUpdated` is 30 hours old.
3. Attacker calls `RSETHPoolV3.deposit{value: 100 ether}("")`.
4. Pool computes: `rsETHAmount = 100e18 * 1e18 / 1.00e18 = 100e18` wrsETH minted.
5. Correct amount at current rate: `100e18 * 1e18 / 1.05e18 ≈ 95.24e18` wrsETH.
6. Attacker receives ~4.76 excess wrsETH (≈ 5% overmint) at the expense of existing holders.
7. When the 100 ETH is bridged to L1 and deposited, only ~95.24 rsETH is minted, leaving the pool short by ~4.76 rsETH to cover all outstanding wrsETH. [7](#0-6) [8](#0-7)
