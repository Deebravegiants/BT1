### Title
Unvalidated Chainlink `latestRoundData()` Return Values Enable Stale Price Consumption in Deposit Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards every validation field (`roundId`, `updatedAt`, `answeredInRound`), and performs no positivity check on the returned price. This is the direct analog of the reported vulnerability. The function sits on the critical path for rsETH minting: a stale or zero price propagates through `LRTOracle.getAssetPrice()` into `LRTDepositPool.getRsETHAmountToMint()`, causing incorrect rsETH issuance to any depositor.

### Finding Description
`ChainlinkPriceOracle.getAssetPrice()` fetches the asset/ETH exchange rate used to determine how many rsETH tokens a depositor receives:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52-54
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

All five return values from `latestRoundData()` are available (`roundId`, `answer`, `startedAt`, `updatedAt`, `answeredInRound`), but only `answer` is used. No checks are performed for:
- `answeredInRound < roundId` — detects a stale round where the answer was carried over from a prior round
- `updatedAt == 0` — detects an incomplete/uninitialised round
- `block.timestamp - updatedAt > heartbeat` — detects a feed that has not been updated within its expected heartbeat window
- `price <= 0` — detects an invalid or sentinel price value

The contrast with the project's own `ChainlinkOracleForRSETHPoolCollateral`, which does perform these checks, confirms the team is aware of the requirement:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L27-32
(uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
    AggregatorV3Interface(oracle).latestRoundData();
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

The stale price flows directly into rsETH minting:

```solidity
// contracts/LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [3](#0-2) 

`lrtOracle.getAssetPrice(asset)` delegates to `IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset)`, which resolves to `ChainlinkPriceOracle.getAssetPrice()`. [4](#0-3) 

### Impact Explanation
**Protocol insolvency / share mis-accounting (Critical).**

If the Chainlink feed is stale and the last recorded price is higher than the true current price (e.g., the underlying LST depegged after the last update), every depositor calling `depositAsset` receives more rsETH than the deposited collateral is worth. This dilutes all existing rsETH holders proportionally and, if repeated or large enough, renders the protocol insolvent — the total rsETH supply becomes unbacked.

Conversely, if `price` is zero (possible during an uninitialised or broken round), `uint256(0)` causes `rsethAmountToMint = 0`, meaning the depositor's assets are accepted but zero rsETH is minted — a temporary freeze of the deposited value.

### Likelihood Explanation
Chainlink feeds can go stale during periods of chain congestion, oracle node failures, or when the deviation threshold is never crossed (e.g., a pegged asset that barely moves). The scenario is well-documented and has occurred on mainnet. Any depositor transacting during such a window is affected without any special action on their part.

### Recommendation
Apply the same validation pattern already used in `ChainlinkOracleForRSETHPoolCollateral.getRate()` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

if (answeredInRound < roundId) revert StalePrice();
if (updatedAt == 0) revert IncompleteRound();
if (price <= 0) revert InvalidPrice();
if (block.timestamp - updatedAt > HEARTBEAT) revert StalePrice();
```

`HEARTBEAT` should be set per feed (e.g., 3600 s for a 1-hour heartbeat feed, 86 400 s for a 24-hour feed).

### Proof of Concept

1. Chainlink nodes fail to reach consensus on a new round for an LST/ETH feed (e.g., due to chain congestion). The last recorded price is `1.05e18` (5 % above the true current value of `1.00e18`).
2. A depositor calls `LRTDepositPool.depositAsset(asset, 1000e18)`.
3. `getRsETHAmountToMint` calls `lrtOracle.getAssetPrice(asset)` → `ChainlinkPriceOracle.getAssetPrice(asset)` → returns stale `1.05e18`.
4. `rsethAmountToMint = (1000e18 * 1.05e18) / rsETHPrice` — the depositor receives 5 % more rsETH than the collateral justifies.
5. No revert occurs; no staleness check is present in `ChainlinkPriceOracle`.
6. Existing rsETH holders are diluted; if the true asset price is lower, the protocol's backing ratio falls below 1:1.
