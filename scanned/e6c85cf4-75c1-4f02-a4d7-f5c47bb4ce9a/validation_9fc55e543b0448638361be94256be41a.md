### Title
Missing Minimum Output Slippage Protection in L2 Pool `deposit()` Functions — (`contracts/pools/RSETHPool.sol`, `RSETHPoolV2.sol`, `RSETHPoolNoWrapper.sol`, `RSETHPoolV3.sol`, `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV3WithNativeChainBridge.sol`, `RSETHPoolV2NBA.sol`)

---

### Summary

All L2 pool `deposit()` functions compute the rsETH/wrsETH output amount at execution time using the live oracle rate, but accept no `minAmountOut` parameter. A user who previews the expected output off-chain and then submits a deposit transaction can receive materially fewer tokens than expected if the oracle rate changes before the transaction is mined. The L1 `LRTDepositPool` already has this protection (`minRSETHAmountExpected`), but the L2 pool contracts do not.

---

### Finding Description

In `LRTDepositPool.sol`, both `depositETH()` and `depositAsset()` accept a `minRSETHAmountExpected` parameter and revert if the minted amount falls below it: [1](#0-0) 

This slippage guard is entirely absent from every L2 pool `deposit()` function. For example, in `RSETHPool.sol`:

```solidity
function deposit(string memory referralId) external payable nonReentrant whenNotPaused {
    if (!isEthDepositEnabled) revert EthDepositDisabled();
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();

    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
``` [2](#0-1) 

The output amount is computed entirely from the live oracle rate at execution time: [3](#0-2) 

The same pattern is present in the token-deposit overload: [4](#0-3) 

And identically in `RSETHPoolV2.sol`, `RSETHPoolV2NBA.sol`, `RSETHPoolNoWrapper.sol`, `RSETHPoolV3.sol`, `RSETHPoolV3ExternalBridge.sol`, and `RSETHPoolV3WithNativeChainBridge.sol`: [5](#0-4) [6](#0-5) [7](#0-6) 

The rate used in all these pools is fetched from an external oracle at call time: [8](#0-7) 

---

### Impact Explanation

**Impact: Low** — Contract fails to deliver promised returns, but does not lose the user's principal.

A user who calls `viewSwapRsETHAmountAndFee()` to preview their output and then submits `deposit()` can receive fewer rsETH/wrsETH tokens than expected if the oracle rate increases (rsETH appreciates in ETH terms) between preview and execution. The user's ETH or LST is consumed in full; only the rsETH output is reduced. There is no mechanism to revert or protect the user.

---

### Likelihood Explanation

**Likelihood: Medium** — The rsETH oracle rate is updated periodically by the protocol (via `_updateRsETHPrice()` in `LRTOracle.sol`). [9](#0-8) 

Any oracle update that occurs between a user's off-chain preview and on-chain execution silently changes the output. This is a routine occurrence during normal protocol operation, not a contrived attack scenario. L2 chains with faster block times and active oracle updates make this more frequent.

---

### Recommendation

Add a `minRSETHAmountExpected` parameter to every L2 pool `deposit()` function, mirroring the existing pattern in `LRTDepositPool`:

```solidity
function deposit(string memory referralId, uint256 minRSETHAmountExpected)
    external payable nonReentrant whenNotPaused
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRSETHAmountExpected) revert InsufficientOutputAmount();
    ...
}
```

This applies to all overloads (ETH and token deposits) across all pool contract versions.

---

### Proof of Concept

1. User calls `RSETHPool.viewSwapRsETHAmountAndFee(1 ether)` off-chain. Oracle rate is `1.05 ETH/rsETH`, so expected output is `~0.952 rsETH`.
2. User submits `deposit{value: 1 ether}("ref")`.
3. Before the transaction is mined, the protocol oracle is updated: rsETH price rises to `1.10 ETH/rsETH`.
4. Transaction executes: `rsETHAmount = 1e18 * 1e18 / 1.10e18 ≈ 0.909 rsETH` — approximately 4.5% less than previewed.
5. No revert occurs. The user silently receives fewer tokens than expected with no recourse. [8](#0-7) [1](#0-0)
