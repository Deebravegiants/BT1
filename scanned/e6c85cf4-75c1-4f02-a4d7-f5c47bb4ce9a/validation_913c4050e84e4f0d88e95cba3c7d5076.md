### Title
`reinitialize()` Does Not Reset `lastMintDay` / `dailyMintAmount`, Causing Stale Daily-Mint State to Persist and Blocking All Deposits - (File: contracts/pools/RSETHPoolV3.sol)

---

### Summary

The `reinitialize()` functions across all L2 pool variants update `startTimestamp` and `dailyMintLimit` but leave `lastMintDay` and `dailyMintAmount` untouched. Because `getCurrentDay()` is computed relative to `startTimestamp`, resetting that anchor resets the day counter to zero while the old accumulated values remain. The `limitDailyMint` modifier only clears `dailyMintAmount` when `currentDay > lastMintDay`; after a reinitialisation that advances `startTimestamp`, `currentDay` starts at 0 and `lastMintDay` may be hundreds of days ahead, so the reset condition is never triggered. Any non-zero `dailyMintAmount` from the previous epoch is silently carried forward, potentially blocking every deposit until the new epoch catches up to the old day number.

---

### Finding Description

`getCurrentDay()` is defined as:

```solidity
return (block.timestamp - startTimestamp) / 1 days;
```

The `limitDailyMint` modifier resets the daily counter only when the new day index exceeds the stored one:

```solidity
if (currentDay > lastMintDay) {
    lastMintDay = currentDay;
    dailyMintAmount = 0;
}
```

The `reinitialize` functions that introduce the daily-mint feature set only two variables:

```solidity
dailyMintLimit = _dailyMintLimit;
startTimestamp = _startTimestamp;
// lastMintDay and dailyMintAmount are NOT touched
```

This is present identically in:
- `RSETHPoolV3.reinitialize(uint256,uint256)` (reinitializer version 2)
- `RSETHPoolV2.reinitialize(uint256,uint256)` (reinitializer version 3)
- `RSETHPoolV3ExternalBridge.reinitialize(uint256,uint256)` (reinitializer version 4)
- `RSETHPoolV3WithNativeChainBridge.reinitialize(uint256,uint256)` (reinitializer version 2)

**Concrete scenario**

1. A pool has been live for 365 days. On the last day users mint up to the limit: `lastMintDay = 365`, `dailyMintAmount = dailyMintLimit`.
2. Admin calls `reinitialize(newLimit, T2)` where `T2 ≥ block.timestamp` (required by the guard).
3. After `T2` passes, `getCurrentDay()` returns 0. The condition `0 > 365` is false, so `dailyMintAmount` is never cleared.
4. Every call to `deposit` / `swapETHForRsETH` hits `dailyMintAmount + rsETHAmount > dailyMintLimit` and reverts with `DailyMintLimitExceeded`.
5. This persists until `(block.timestamp - T2) / 1 days > 365`, i.e. for up to 365 days after the new epoch starts.

---

### Impact Explanation

All user deposits into the affected L2 pool are blocked for a period proportional to the old `lastMintDay` value. Users holding ETH or LSTs on L2 cannot swap them for rsETH/wrsETH, effectively freezing the deposit path. This maps to **Medium – temporary freezing of funds**.

---

### Likelihood Explanation

`reinitialize` is a routine upgrade operation that admins are expected to call when deploying the daily-mint feature or adjusting parameters. The bug is silent: no error is emitted, and the pool appears functional (it accepts calls but reverts them). The longer a pool has been running before reinitialisation, the longer the freeze lasts. Likelihood is **Low-Medium** — it requires an admin reinitialisation, but that is a normal, planned action.

---

### Recommendation

Reset the stale daily-mint state inside every `reinitialize` overload that writes `startTimestamp`:

```solidity
dailyMintLimit   = _dailyMintLimit;
startTimestamp   = _startTimestamp;
lastMintDay      = 0;   // reset epoch counter
dailyMintAmount  = 0;   // clear accumulated amount
```

Alternatively, expose a dedicated `setDailyMintLimit` / `setStartTimestamp` setter that always resets both counters atomically, removing the need for versioned reinitializers for this purpose.

---

### Proof of Concept

1. Deploy `RSETHPoolV3` and call `reinitialize(reinitializer(2))` with `startTimestamp = T1`.
2. Advance time to day 100 relative to `T1`; call `deposit` until `dailyMintAmount == dailyMintLimit`. Now `lastMintDay = 100`.
3. Admin calls `reinitialize(reinitializer(2))` — **this reverts** because reinitializer(2) was already consumed. This illustrates that the same versioned reinitializer cannot be called twice; the stale-state bug is baked in at the one-time call.
4. For pools where a higher-version reinitializer is available (e.g. `RSETHPoolV2` version 3, `RSETHPoolV3ExternalBridge` version 4), repeat steps 1–2, then call the higher-version `reinitialize(newLimit, T2)` with `T2 = block.timestamp`.
5. Immediately after `T2`, call `deposit(1 ether)`. Observe revert `DailyMintLimitExceeded` even though the new epoch just started, because `dailyMintAmount` was never cleared.

### Citations

**File:** contracts/pools/RSETHPoolV3.sol (L96-125)
```text
    modifier limitDailyMint(uint256 amount, address token) {
        if (block.timestamp < startTimestamp) {
            revert MintBeforeStartTimestamp();
        }

        uint256 rsETHAmount;

        // Calculate the amount of rsETH that will be minted
        if (token == ETH_IDENTIFIER) {
            (rsETHAmount,) = viewSwapRsETHAmountAndFee(amount);
        } else {
            (rsETHAmount,) = viewSwapRsETHAmountAndFee(amount, token);
        }

        uint256 currentDay = getCurrentDay();

        // If the current day is greater than the last mint day, reset the daily mint amount
        if (currentDay > lastMintDay) {
            lastMintDay = currentDay;
            dailyMintAmount = 0;
        }

        // Check if the daily mint amount plus the amount to mint is greater than the daily mint limit
        if (dailyMintAmount + rsETHAmount > dailyMintLimit) {
            revert DailyMintLimitExceeded();
        }

        dailyMintAmount += rsETHAmount;
        _;
    }
```

**File:** contracts/pools/RSETHPoolV3.sol (L179-198)
```text
    function reinitialize(
        uint256 _dailyMintLimit,
        uint256 _startTimestamp
    )
        external
        reinitializer(2)
        onlyRole(DEFAULT_ADMIN_ROLE)
    {
        if (_dailyMintLimit == 0) {
            revert InvalidDailyMintLimit();
        }

        // startTimestamp cannot be in the past
        if (block.timestamp > _startTimestamp) {
            revert InvalidStartTimestamp();
        }

        dailyMintLimit = _dailyMintLimit;
        startTimestamp = _startTimestamp;
    }
```
