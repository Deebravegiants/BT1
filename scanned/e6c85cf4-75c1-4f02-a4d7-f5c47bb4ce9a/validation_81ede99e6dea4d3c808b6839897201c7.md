### Title
Stale Cross-Chain Rate with No Freshness Check Allows Depositors to Mint Excess rsETH at Outdated Rate — (File: contracts/cross-chain/CrossChainRateReceiver.sol)

---

### Summary
`CrossChainRateReceiver` stores a `lastUpdated` timestamp every time a LayerZero rate message arrives, but `getRate()` returns the cached `rate` unconditionally — no staleness guard exists. All L2 deposit pools (`RSETHPoolV3`, `RSETHPoolNoWrapper`) call this function to price every deposit. When the rate is stale (lower than the true current rate), depositors receive more rsETH than they are entitled to, diluting the yield of all existing rsETH holders.

---

### Finding Description

`CrossChainRateReceiver.lzReceive()` writes both `rate` and `lastUpdated` when a LayerZero message arrives from L1:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol  lines 93-97
uint256 _rate = abi.decode(_payload, (uint256));
rate = _rate;
lastUpdated = block.timestamp;
```

`getRate()` ignores `lastUpdated` entirely:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol  lines 103-105
function getRate() external view returns (uint256) {
    return rate;
}
```

Every L2 pool reads this value to compute how many rsETH tokens to mint:

```solidity
// contracts/pools/RSETHPoolV3.sol  lines 303-307
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

```solidity
// contracts/pools/RSETHPoolNoWrapper.sol  lines 282-285
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

Because rsETH accrues staking rewards over time, its ETH value monotonically increases. A stale (older, lower) rate causes the division to yield a **larger** rsETH amount than the depositor deserves. The attacker does not need to front-run anything; they simply deposit while the oracle is stale.

The rate is pushed from L1 only when `MultiChainRateProvider.updateRate()` is called:

```solidity
// contracts/cross-chain/MultiChainRateProvider.sol  lines 108-137
function updateRate() external payable nonReentrant { ... }
```

This is a permissionless call that requires the caller to supply LayerZero gas fees. If no one calls it for an extended period (e.g., hours or days), the L2 rate drifts below the true L1 rate, and the window for over-minting opens to any depositor.

---

### Impact Explanation

Every rsETH minted against a stale rate is backed by less ETH than the protocol's accounting assumes. When the rate is eventually corrected, the rsETH price on L1 (`LRTOracle.rsETHPrice`) reflects the true TVL divided by the now-inflated supply, so the price per rsETH is lower than it would have been. All pre-existing holders receive fewer ETH-equivalent assets upon withdrawal than they were entitled to. This is a direct, quantifiable theft of accrued yield from existing rsETH holders.

**Impact class**: High — theft of unclaimed yield.

---

### Likelihood Explanation

`updateRate()` is permissionless but requires the caller to pay LayerZero messaging fees. During periods of high gas costs, network congestion, or simply operator inattention, the rate can go un-refreshed for many hours. rsETH's yield accrual rate is roughly 3–5 % APY, meaning a 24-hour stale window creates a ~0.01 % per-deposit over-mint — small per transaction but aggregated across the full daily deposit volume it represents meaningful dilution. Any depositor who monitors `lastUpdated` can exploit this passively without any special privilege or front-running.

**Likelihood class**: Medium.

---

### Recommendation

Add a configurable maximum staleness constant and enforce it inside `getRate()`:

```solidity
uint256 public constant MAX_RATE_STALENESS = 24 hours;

function getRate() external view returns (uint256) {
    require(
        block.timestamp - lastUpdated <= MAX_RATE_STALENESS,
        "CrossChainRateReceiver: rate is stale"
    );
    return rate;
}
```

Additionally, the L2 pool contracts should propagate this revert so that deposits are blocked rather than silently executed at a stale price. The `MultiChainRateProvider.updateRate()` call should be automated (e.g., via a keeper) to ensure the rate is refreshed at least once per day.

---

### Proof of Concept

1. At T=0, L1 rsETH price = 1.00 ETH/rsETH. `updateRate()` is called; L2 `rate` = 1.00e18.
2. 24 hours pass. Staking rewards accrue; L1 rsETH price rises to 1.0027 ETH/rsETH. No one calls `updateRate()`.
3. At T=24h, Alice deposits 100 ETH on L2 (`RSETHPoolV3.deposit()`).
   - `rsETHAmount = 100e18 * 1e18 / 1.00e18 = 100.00 rsETH` (stale rate)
   - Correct amount would be: `100e18 * 1e18 / 1.0027e18 ≈ 99.73 rsETH`
   - Alice receives **0.27 rsETH excess** (~$675 at $2500/ETH).
4. `updateRate()` is eventually called; L2 rate corrects to 1.0027e18.
5. All existing rsETH holders now hold tokens backed by slightly less ETH per token than they were owed — their accrued yield has been partially transferred to Alice. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

        uint256 latestRate = getLatestRate();

        rate = latestRate;

        lastUpdated = block.timestamp;

        bytes memory _payload = abi.encode(latestRate);

        uint256 rateReceiversLength = rateReceivers.length;

        for (uint256 i; i < rateReceiversLength;) {
            uint16 dstChainId = uint16(rateReceivers[i]._chainId);

            bytes memory remoteAndLocalAddresses = abi.encodePacked(rateReceivers[i]._contract, address(this));

            (uint256 estimatedFee,) = ILayerZeroEndpoint(layerZeroEndpoint)
                .estimateFees(dstChainId, address(this), _payload, false, bytes(""));

                dstChainId, remoteAndLocalAddresses, _payload, payable(msg.sender), address(0x0), bytes("")
            );

            unchecked {
                ++i;
            }
