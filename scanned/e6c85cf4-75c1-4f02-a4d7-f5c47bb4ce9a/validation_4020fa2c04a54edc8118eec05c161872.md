### Title
Unbounded Gas Consumption in `_getTotalEthInProtocol` via S × N × M × K Loop Chain — (`contracts/LRTOracle.sol`)

### Summary

`updateRSETHPrice()` calls `_getTotalEthInProtocol()`, which iterates over every supported asset and, for each asset, calls `getAssetUnstaking(asset)` on every NDC. Each `getAssetUnstaking` call makes an external call to EigenLayer's `getQueuedWithdrawals`, then loops over all returned withdrawals and their strategies. The total work is O(S × N × M × K), where S = supported assets, N = NDCs, M = queued withdrawals per NDC, and K = strategies per withdrawal. Neither S nor N has a hard on-chain cap, and the protocol's own gas-budget comment only accounts for M (total withdrawal count), not the S × N multiplier on external calls.

### Finding Description

**Call chain:**

```
updateRSETHPrice()
  → _updateRsETHPrice()
    → _getTotalEthInProtocol()                          [loop: S assets]
      → getTotalAssetDeposits(asset)
        → getAssetDistributionData(asset)               [loop: N NDCs]
          → INodeDelegator.getAssetUnstaking(asset)     [external call per NDC]
            → IDelegationManager.getQueuedWithdrawals() [external call to EigenLayer]
              [loop: M withdrawals × K strategies]
```

In `_getTotalEthInProtocol`, for each of the S supported assets, `getTotalAssetDeposits` is called: [1](#0-0) 

Inside `getAssetDistributionData`, for each of the N NDCs, `getAssetUnstaking` is called: [2](#0-1) 

Inside `getAssetUnstaking`, a full `getQueuedWithdrawals` external call is made, then a nested loop over withdrawals × strategies runs: [3](#0-2) 

**The protocol's own gas-budget comment** in `setMaxUncompletedWithdrawalCount` caps total withdrawals at 80 and notes the concern: [4](#0-3) 

However, this comment only accounts for M (total withdrawal count). The actual number of external `getQueuedWithdrawals` calls is **S × N**, not N. With S = 10 assets and N = 10 NDCs, that is 100 external storage-reading calls to EigenLayer, each returning up to 80 withdrawals. The `maxNodeDelegatorLimit` has no hard on-chain upper bound: [5](#0-4) 

The `maxUncompletedWithdrawalCount` is capped at 80 globally, but this bounds M (total withdrawals), not the S × N external call count. As S and N grow independently (both admin-controlled without hard caps), the gas cost of `updateRSETHPrice` grows without a corresponding on-chain ceiling.

### Impact Explanation

If the gas cost of `_getTotalEthInProtocol` exceeds the block gas limit, every call to `updateRSETHPrice()` reverts OOG. `updateRSETHPriceAsManager()` uses the same `_updateRsETHPrice()` internal function and would also revert. The rsETH price becomes stale and un-updatable until admin intervention (completing withdrawals, removing NDCs, or pausing). This constitutes **temporary freezing of price updates** and breaks the invariant that rsETH price must be updatable.

### Likelihood Explanation

The default `maxNodeDelegatorLimit` is 10 and the protocol currently supports a small number of assets. Within these defaults, OOG is unlikely. However, the admin can legitimately increase `maxNodeDelegatorLimit` (no on-chain cap), and the protocol may add more supported assets over time. The gas analysis embedded in the comment (`// 120 is the max number of uncompleted withdrawals`) does not account for the S multiplier on external calls. As the protocol scales, the risk becomes concrete.

### Recommendation

1. Cache `getQueuedWithdrawals` per NDC and reuse the result across all asset iterations, reducing external calls from S × N to N.
2. Add a hard on-chain cap to `maxNodeDelegatorLimit` (e.g., ≤ 10) enforced in `updateMaxNodeDelegatorLimit`.
3. Revise the gas budget analysis to account for S × N external calls, not just M total withdrawals.

### Proof of Concept

```solidity
// Pseudocode: measure gas of updateRSETHPrice at boundary conditions
// Setup: S=10 supported assets, N=10 NDCs, M=8 withdrawals per NDC (80 total), K=3 strategies each
// Total getQueuedWithdrawals calls: 10 × 10 = 100
// Total inner loop iterations: 10 × 10 × 8 × 3 = 2,400
// Each external call ~10,000 gas → 1,000,000 gas for external calls alone
// Inner iterations ~5,000 gas each → 12,000,000 gas
// Plus _updateRsETHPrice overhead → approaches 30M block limit

// If maxNodeDelegatorLimit is raised to 20 and assets grow to 15:
// External calls: 15 × 20 = 300 → ~3,000,000 gas
// Inner iterations: 15 × 80 × 3 = 3,600 → ~18,000,000 gas
// Total > 21,000,000 gas, exceeding practical tx gas limits
```

The concrete path: deploy with S=10, N=10, M=80 (all on one NDC), K=3; call `updateRSETHPrice()`; measure gas; assert it stays below 30M. At S=15, N=20, the call reverts OOG without any attacker involvement — purely from legitimate protocol growth within admin-settable parameters.

```text
    function getAssetUnstaking(address asset) external view returns (uint256 amount) {
        (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
            _getDelegationManager().getQueuedWithdrawals(address(this));

        for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
            IDelegationManager.Withdrawal memory withdrawal = queuedWithdrawals[withdrawalIndex];

            for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
                IStrategy strategy = withdrawal.strategies[strategyIndex];

                address strategyAsset = address(strategy) == address(lrtConfig.beaconChainETHStrategy())
                    ? LRTConstants.ETH_TOKEN
                    : address(strategy.underlyingToken());

                if (strategyAsset != asset) continue;

                uint256 sharesToUnstake = withdrawalShares[withdrawalIndex][strategyIndex];
                amount += strategyAsset == LRTConstants.ETH_TOKEN
                    ? sharesToUnstake
                    : strategy.sharesToUnderlyingView(sharesToUnstake);
            }
        }
    }
```
