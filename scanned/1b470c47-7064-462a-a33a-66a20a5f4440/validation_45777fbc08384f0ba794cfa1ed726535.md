### Title
ETH Withdrawal Permanently Frozen for Smart Contract Users After rsETH Is Burned — (`contracts/LRTWithdrawalManager.sol`)

### Summary
The `LRTWithdrawalManager` processes ETH withdrawals across two separate transactions. In the first (`unlockQueue`), rsETH is irreversibly burned and ETH is redeemed into the manager. In the second (`completeWithdrawal`), ETH is pushed to the user via a low-level call. If the user's address is a smart contract that cannot receive ETH, the push permanently fails, leaving the ETH frozen in `LRTWithdrawalManager` with no recovery path.

### Finding Description
The withdrawal lifecycle is split across two independent transactions:

**Transaction 1 — `unlockQueue()`**: rsETH held by the manager is burned and the corresponding ETH is pulled from `LRTUnstakingVault` into `LRTWithdrawalManager`. [1](#0-0) 

**Transaction 2 — `completeWithdrawal()` → `_processWithdrawalCompletion()`**: ETH is pushed to the user via `_transferAsset`, which uses a raw `.call{value}`. [2](#0-1) [3](#0-2) 

If the user's address is a smart contract without a `receive()` or `fallback()` function, the `.call` returns `false`, `_transferAsset` reverts with `EthTransferFailed`, and the entire `completeWithdrawal` transaction is rolled back. The withdrawal request is restored in storage, but the rsETH burned in Transaction 1 is **not** restored — it is gone permanently.

There is no alternative delivery path. `completeWithdrawalForUser` (operator-assisted) calls the same `_processWithdrawalCompletion` and hits the same revert: [4](#0-3) 

`sweepRemainingAssets` cannot rescue the ETH because it requires `!hasUnlockedWithdrawals(asset)`, which remains `true` as long as the stuck request exists: [5](#0-4) 

There is no mechanism to redirect the ETH to an alternative address or to re-credit rsETH.

### Impact Explanation
After `unlockQueue` executes, the user's rsETH is permanently burned and the ETH sits in `LRTWithdrawalManager`. If the user's address cannot receive ETH, every subsequent `completeWithdrawal` call reverts. The ETH is permanently frozen — the user has lost their rsETH and cannot recover the underlying ETH. This matches **Critical — Permanent freezing of funds**.

### Likelihood Explanation
Smart contracts that interact with DeFi protocols (multisigs, vaults, DAOs, proxy wallets) are common depositors. A contract without a `receive()` function — or one that conditionally reverts on ETH receipt — will trigger this path. The user need only call `initiateWithdrawal` once; the freeze is triggered automatically when the operator runs `unlockQueue`. Likelihood is **Low** but realistic given the prevalence of smart-contract depositors.

### Recommendation
Replace the push model with a pull model for ETH withdrawals: record the claimable amount in a mapping and let the user (or any address they designate) call a separate `claimETH()` function. Alternatively, allow users to register an alternative recipient address before `completeWithdrawal` is called, or add an admin-callable rescue function that can redirect stuck ETH to a user-specified address when the primary transfer has failed a configurable number of times.

### Proof of Concept
1. A smart contract `Vault` (no `receive()`) calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. rsETH is transferred from `Vault` to `LRTWithdrawalManager`.
2. Operator calls `unlockQueue(ETH_TOKEN, ...)`. rsETH is burned at line 305; ETH is redeemed from `LRTUnstakingVault` into `LRTWithdrawalManager` at line 307.
3. `Vault` calls `completeWithdrawal(ETH_TOKEN, "")`. `_transferAsset` executes `payable(Vault).call{value: amount}("")` — `Vault` has no `receive()`, so the call returns `false`. `EthTransferFailed` is thrown; the transaction reverts.
4. The withdrawal request is restored in storage (revert undoes the `delete` and `popFront`), but the rsETH burned in step 2 is not restored.
5. Every subsequent call to `completeWithdrawal` or `completeWithdrawalForUser` for `Vault` repeats step 3 and reverts. The ETH is permanently frozen in `LRTWithdrawalManager`.
