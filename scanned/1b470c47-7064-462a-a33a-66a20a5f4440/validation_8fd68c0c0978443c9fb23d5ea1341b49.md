### Title
`limitDailyMint` Modifier Hard-Reverts on Shared Daily Cap, Allowing Any Depositor to Grief All Others for Up to 24 Hours - (File: `contracts/pools/RSETHPoolV3.sol`)

---

### Summary

The `limitDailyMint` modifier, present across all L2 pool contracts, unconditionally reverts with `DailyMintLimitExceeded` whenever a deposit would push `dailyMintAmount` past `dailyMintLimit`. Because the cap is a single shared counter, any depositor who consumes the remaining headroom — even legitimately — causes every subsequent depositor's transaction to revert for the rest of the 24-hour window. A malicious actor can exploit this deliberately and at zero net cost (they receive `wrsETH` in return), repeating the attack every day.

---

### Finding Description

The `limitDailyMint` modifier in `RSETHPoolV3.sol` (and identically in `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV3WithNativeChainBridge.sol`, `RSETHPoolV2ExternalBridge.sol`, `RSETHPoolV2.sol`) reads:

```solidity
// contracts/pools/RSETHPoolV3.sol  lines 118-121
if (dailyMintAmount + rsETHAmount > dailyMintLimit) {
    revert DailyMintLimitExceeded();
}
dailyMintAmount += rsETHAmount;
```

There is no partial-fill path. If the requested deposit would exceed the cap by even 1 wei, the entire transaction reverts. The cap is a single global counter shared by all depositors in the same 24-hour window.

The public `deposit()` functions apply this modifier unconditionally:

```solidity
// RSETHPoolV3.sol line 251
limitDailyMint(msg.value, ETH_IDENTIFIER)

// RSETHPoolV3.sol line 280
limitDailyMint(amount, token)
```

An attacker who deposits an amount equal to `dailyMintLimit - dailyMintAmount` (the remaining headroom) atomically sets `dailyMintAmount == dailyMintLimit`. Every subsequent depositor's call reverts until `getCurrentDay()` increments and the counter resets.

---

### Impact Explanation

**Medium — Temporary freezing of funds.**

All honest depositors are locked out of the L2 pool for up to 24 hours per attack cycle. Because the attacker receives `wrsETH` worth approximately the ETH they deposited, their net economic cost is zero. The attack can be repeated every day indefinitely. Users who need to deposit during a specific window (e.g., to capture a rate arbitrage, or to bridge before a deadline) suffer a concrete, time-bounded loss of access to the protocol's core deposit function.

---

### Likelihood Explanation

**Low-Medium.** The attack requires no special privilege — any externally-owned account can call `deposit()`. The attacker bears no net financial loss (they receive `wrsETH`). The only prerequisite is that the daily limit has not already been fully consumed by organic activity. On chains where the daily limit is set conservatively relative to organic volume, the remaining headroom is predictably available near the start of each UTC day, making the attack straightforward to time.

---

### Recommendation

Mirror the M-06 mitigation pattern: instead of reverting when the deposit would exceed the cap, cap the accepted amount to the remaining headroom and either refund the excess (for ETH) or transfer only the capped amount (for ERC-20 tokens):

```solidity
modifier limitDailyMint(uint256 amount, address token) {
    // ... timestamp and day-reset logic unchanged ...

    if (dailyMintAmount + rsETHAmount > dailyMintLimit) {
-       revert DailyMintLimitExceeded();
+       uint256 excess = (dailyMintAmount + rsETHAmount) - dailyMintLimit;
+       rsETHAmount -= excess;
+       amount = /* back-calculate capped input amount from rsETHAmount */;
+       // refund excess ETH / reduce ERC-20 transfer accordingly
    }

    dailyMintAmount += rsETHAmount;
    _;
}
```

Alternatively, expose a `minAcceptedAmount` parameter in each `deposit()` function so callers can specify the minimum they are willing to receive; revert only when the remaining headroom is below that threshold. This preserves the hard-revert UX for users who need an exact fill while eliminating the griefing vector for users who are flexible.

---

### Proof of Concept

Assume `dailyMintLimit = 1000 ether` in `wrsETH` equivalent and `dailyMintAmount = 950 ether` at the start of a block.

1. Alice submits `deposit{value: 60 ether}(referralId)` — she wants to deposit 60 ETH worth of `wrsETH`.
2. Bob (attacker) sees Alice's transaction in the mempool and front-runs it with `deposit{value: 50 ether}(referralId)`.
3. Bob's transaction executes first: `dailyMintAmount` becomes `1000 ether` (the limit).
4. Alice's transaction executes next: `dailyMintAmount + rsETHAmount = 1000 + ~60 > 1000` → `revert DailyMintLimitExceeded()`. Alice's ETH is returned, but she cannot deposit.
5. Alice retries with 50 ETH. Bob front-runs again with any non-zero deposit — the limit is already at `1000 ether`, so even 1 wei causes Alice's retry to revert.
6. Alice must wait until the next UTC day. Bob repeats the attack at the start of each new day at zero net cost.

**Root cause**: [1](#0-0)  — the hard revert on a shared global counter with no partial-fill fallback.

**Entry path**: [2](#0-1)  and [3](#0-2)  — both public `deposit()` overloads apply `limitDailyMint` unconditionally.
