### Title
`deposit()` Functions in L2 Pool Contracts Lack Slippage Protection (Minimum rsETH Output) - (`contracts/pools/RSETHPool.sol`, `RSETHPoolNoWrapper.sol`, `RSETHPoolV2.sol`, `RSETHPoolV2ExternalBridge.sol`, `RSETHPoolV2NBA.sol`, `RSETHPoolV3.sol`, `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV3WithNativeChainBridge.sol`)

---

### Summary

All L2 pool `deposit()` functions that swap ETH or supported tokens for rsETH/wrsETH lack a `minRsETHAmountExpected` parameter. The rsETH amount returned to the user is computed at execution time using a live oracle rate. If the oracle rate changes between transaction submission and execution, the user receives fewer rsETH tokens than anticipated with no on-chain recourse. This contrasts directly with the L1 `LRTDepositPool.sol`, which enforces a `minRSETHAmountExpected` check on every deposit.

---

### Finding Description

Every L2 pool contract exposes one or more `deposit()` entry points for unprivileged users:

**`RSETHPool.sol`** (ETH path):
```solidity
function deposit(string memory referralId) external payable nonReentrant whenNotPaused {
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount);
    ...
}
``` [1](#0-0) 

The rsETH amount is computed as:
```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [2](#0-1) 

where `rsETHToETHrate` is fetched live from the oracle at execution time. The same pattern is present in the token deposit path: [3](#0-2) 

The identical pattern exists across all pool variants:
- `RSETHPoolNoWrapper.sol` ETH and token deposit paths [4](#0-3) 
- `RSETHPoolV2.sol` ETH deposit path [5](#0-4) 
- `RSETHPoolV2ExternalBridge.sol` ETH deposit path [6](#0-5) 
- `RSETHPoolV3.sol` ETH and token deposit paths [7](#0-6) 
- `RSETHPoolV3ExternalBridge.sol` ETH and token deposit paths [8](#0-7) 
- `RSETHPoolV3WithNativeChainBridge.sol` ETH and token deposit paths [9](#0-8) 

None of these functions accept or enforce a minimum rsETH output amount.

**Contrast with L1 `LRTDepositPool.sol`**, which correctly implements slippage protection:
```solidity
function depositETH(uint256 minRSETHAmountExpected, string calldata referralId) external payable ...
``` [10](#0-9) 

The internal `_beforeDeposit` enforces the check:
```solidity
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [11](#0-10) 

The L2 pools provide a helper `getMinAmount()` for off-chain use, but this is purely informational and is never enforced on-chain during `deposit()`. [12](#0-11) 

---

### Impact Explanation

A user submits a `deposit()` transaction with ETH or a supported token. Between submission and on-chain execution, the rsETH oracle rate increases (rsETH appreciates in ETH terms, which is the normal direction for a staking yield token). The user's ETH is consumed by the pool, but they receive fewer rsETH tokens than they expected when they signed the transaction. There is no on-chain mechanism to revert the transaction if the output falls below the user's acceptable threshold. The user's funds are not stolen — they are still represented by the rsETH received — but the user receives less value than they intended to exchange for.

**Impact**: Low — Contract fails to deliver promised returns, but does not lose value.

---

### Likelihood Explanation

The rsETH/ETH oracle rate increases monotonically as staking rewards accrue, making this a persistent and directional risk rather than a random one. During periods of network congestion where transactions remain pending for extended blocks, or when the oracle is updated with a significant rate jump, users depositing large amounts can receive meaningfully fewer rsETH tokens than they simulated off-chain. Any unprivileged user calling `deposit()` on any of the affected L2 pool contracts is exposed.

---

### Recommendation

Add a `minRsETHAmountExpected` parameter to all `deposit()` functions in the L2 pool contracts, mirroring the pattern already implemented in `LRTDepositPool.sol`:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountExpected)
    external payable nonReentrant whenNotPaused
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    ...
}
```

Users who want no protection can pass `0`. This is the industry-standard pattern already used by the same protocol on L1.

---

### Proof of Concept

1. User calls `viewSwapRsETHAmountAndFee(1 ether)` off-chain and sees they will receive `X` wrsETH at the current oracle rate.
2. User submits `deposit{value: 1 ether}("ref")` to `RSETHPoolV3.sol`.
3. Before the transaction is mined, the rsETH oracle rate is updated (rsETH price increases).
4. The transaction executes: `rsETHAmount = amountAfterFee * 1e18 / newHigherRate` yields a value less than `X`.
5. The user receives fewer wrsETH tokens than expected with no revert, and cannot recover the difference. [13](#0-12) [14](#0-13)
