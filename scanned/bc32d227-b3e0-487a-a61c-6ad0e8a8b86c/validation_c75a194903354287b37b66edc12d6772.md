### Title
Unpermissioned `sendFunds()` Allows Any Caller to Exhaust Daily Fee Minting Limit — (`contracts/FeeReceiver.sol` / `contracts/LRTOracle.sol`)

---

### Summary

`FeeReceiver.sendFunds()` carries no access control. Any caller can push accumulated MEV/EL rewards into the deposit pool at the start of a new 24-hour period, then call the equally-permissionless `LRTOracle.updateRSETHPrice()`. The resulting TVL spike causes `_checkAndUpdateDailyFeeMintLimit` to consume some or all of `maxFeeMintAmountPerDay`, blocking legitimate protocol fee minting for the remainder of that day.

---

### Finding Description

**Entry point 1 — `FeeReceiver.sendFunds()`** [1](#0-0) 

The function is `external` with no role modifier. Any EOA or contract can call it at any time, immediately forwarding the entire ETH balance of `FeeReceiver` to the deposit pool.

**Entry point 2 — `LRTOracle.updateRSETHPrice()`** [2](#0-1) 

Also permissionless (`public whenNotPaused`). Any caller can trigger a price update and the associated fee-minting logic.

**Fee computation path**

Inside `_updateRsETHPrice()`, the protocol computes a fee whenever `totalETHInProtocol > previousTVL`: [3](#0-2) 

The ETH pushed by `sendFunds()` is counted in `_getTotalEthInProtocol()` via `getTotalAssetDeposits`, so it directly inflates `totalETHInProtocol` relative to `previousTVL = rsethSupply × rsETHPrice`.

**Daily limit consumption** [4](#0-3) 

`_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` is called unconditionally whenever `protocolFeeInETH > 0`: [5](#0-4) 

If `currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay` the call reverts; otherwise `currentPeriodMintedFeeAmount` is incremented by `feeAmount`. An attacker who times `sendFunds()` so that the resulting `feeAmount ≤ maxFeeMintAmountPerDay` will successfully consume that portion of the daily budget. A second `updateRSETHPrice()` call later in the same period (from normal protocol operations) will then revert with `DailyFeeMintLimitExceeded` if the remaining budget is insufficient.

**Period-reset logic does not protect against this** [6](#0-5) 

The reset only fires when `block.timestamp >= feePeriodStartTime + 1 days`. An attacker who calls at the very start of a new period resets the counter to 0 and then immediately fills it.

---

### Impact Explanation

The daily fee minting limit is a protocol-level cap on how much rsETH can be minted as protocol fees per 24-hour window. By exhausting it at period start, the attacker prevents the protocol from collecting fees on any additional yield that accrues later in the day (e.g., staking rewards, further MEV). Those fees are permanently lost for that period — they are not deferred, because the price update that would have minted them reverts. This constitutes **theft of unclaimed yield** (High).

The fees minted during the attack do go to the treasury, so the attacker gains nothing financially; the harm is entirely to the protocol's fee revenue.

---

### Likelihood Explanation

- Both `sendFunds()` and `updateRSETHPrice()` are callable by any address with no preconditions beyond the contract not being paused.
- FeeReceiver accumulates ETH passively from validator MEV/EL rewards; over a multi-day window it will hold a predictable balance.
- The attacker only needs to monitor the FeeReceiver balance off-chain and call at period boundary when `feeAmount ≤ maxFeeMintAmountPerDay`. This is straightforward to script.
- No privileged key, front-running dependency, or external protocol compromise is required.

---

### Recommendation

1. **Add access control to `sendFunds()`** — restrict it to `MANAGER` or a dedicated keeper role, consistent with how other sensitive state-changing functions in the protocol are guarded.
2. **Alternatively**, decouple TVL accounting from the fee-minting daily limit: accumulate reward ETH in a staging variable and only recognize it during a permissioned price-update call, so an unprivileged caller cannot force fee recognition.
3. Consider whether `updateRSETHPrice()` itself should be restricted to a keeper/manager role, given that it has material economic side-effects (fee minting, price updates, potential auto-pause).

### Proof of Concept

```solidity
// Fork test (Foundry) — illustrative
function test_exhaustDailyFeeMintLimit() public {
    // 1. Simulate MEV accumulation in FeeReceiver
    vm.deal(address(feeReceiver), 10 ether);

    // 2. Warp to the start of a new 24-hour period
    vm.warp(oracle.getCurrentPeriodStartTime() + 1 days);

    // 3. Attacker (unprivileged) pushes ETH into deposit pool
    vm.prank(attacker);
    feeReceiver.sendFunds();

    // 4. Attacker triggers price update — fee is minted, daily limit consumed
    vm.prank(attacker);
    oracle.updateRSETHPrice();

    assertEq(oracle.currentPeriodMintedFeeAmount(), oracle.maxFeeMintAmountPerDay());

    // 5. Legitimate price update later in the same period reverts
    vm.expectRevert(abi.encodeWithSelector(ILRTOracle.DailyFeeMintLimitExceeded.selector, ...));
    oracle.updateRSETHPrice();
}
```

    function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
        // Reset the period if it's unset or a day has passed
        if (block.timestamp >= feePeriodStartTime + 1 days) {
            currentPeriodMintedFeeAmount = 0;
            feePeriodStartTime = getCurrentPeriodStartTime();
        }

        // Check if minting would exceed the daily limit
        if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
            revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
        }

        currentPeriodMintedFeeAmount += feeAmount;
    }
