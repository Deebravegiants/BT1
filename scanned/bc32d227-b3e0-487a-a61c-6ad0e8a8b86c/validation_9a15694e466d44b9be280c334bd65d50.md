### Title
Withdrawal Requests Permanently Tied to Initiating Address With No Re-Assign or Cancel Mechanism — (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTWithdrawalManager` permanently binds every withdrawal request to the address that called `initiateWithdrawal`. There is no function to transfer, re-assign, or cancel a pending request. If the initiating address is a smart contract that later becomes unable to receive ETH (e.g., no `receive()` fallback, self-destructed, or a multisig that loses quorum), the rsETH is already held by the contract and the corresponding underlying assets become permanently frozen with no admin escape hatch.

---

### Finding Description

When `initiateWithdrawal` is called, rsETH is pulled from `msg.sender` into the contract and a withdrawal request nonce is pushed into `userAssociatedNonces[asset][msg.sender]`: [1](#0-0) [2](#0-1) 

The only way to complete the withdrawal is through `_processWithdrawalCompletion`, which pops from `userAssociatedNonces[asset][user]` and then calls `_transferAsset(asset, user, request.expectedAssetAmount)`: [3](#0-2) [4](#0-3) 

The operator-callable `completeWithdrawalForUser` also routes assets to the original `user` address — it cannot redirect to a different recipient: [5](#0-4) 

`_transferAsset` for ETH uses a low-level `call` and **reverts** on failure: [6](#0-5) 

There is no `cancelWithdrawal` function, no re-assign function, and no admin rescue path. `sweepRemainingAssets` is blocked while `unlockedWithdrawalsCount[asset] > 0`: [7](#0-6) 

---

### Impact Explanation

After `unlockQueue` processes the stuck request — burning the rsETH and redeeming the underlying asset from `LRTUnstakingVault` into `LRTWithdrawalManager` — the asset sits in the contract with `unlockedWithdrawalsCount[asset] > 0`. Every attempt to deliver it via `completeWithdrawalForUser` reverts because the target address cannot receive ETH. `sweepRemainingAssets` is gated behind `hasUnlockedWithdrawals == false`, so it cannot be used either. The funds are permanently frozen.

**Impact class**: Permanent freezing of funds (Critical).

---

### Likelihood Explanation

Smart-contract callers of `initiateWithdrawal` are a realistic population: protocol integrators, smart-contract wallets, multisigs, and yield aggregators all interact with withdrawal queues programmatically. A multisig that loses quorum, a contract that lacks a `receive()` fallback for ETH, or a contract that is self-destructed after queuing a withdrawal all trigger this path without any attacker involvement. The entry point (`initiateWithdrawal`) is fully public and permissionless.

**Likelihood**: Low-to-medium (requires a smart-contract initiator that later becomes unable to receive ETH, which is a realistic but not common scenario).

---

### Recommendation

1. Add a `cancelWithdrawal(address asset)` function that returns the held rsETH to the original requester while the request is still in the locked (pre-`unlockQueue`) state.
2. Add a `reassignWithdrawal(address asset, address newRecipient)` function, callable by the original requester or by an admin/operator, that updates the recipient stored for a given nonce.
3. Alternatively, store an explicit `recipient` field in `WithdrawalRequest` and allow the original initiator to update it before the request is unlocked.

---

### Proof of Concept

1. A smart-contract integrator (no `receive()` fallback) calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`.
2. rsETH is transferred to `LRTWithdrawalManager`; the nonce is pushed to `userAssociatedNonces[ETH_TOKEN][integrator]`.
3. The integrator contract is later self-destructed or loses the ability to make calls.
4. An operator calls `unlockQueue` — rsETH is burned, ETH is redeemed from `LRTUnstakingVault` into `LRTWithdrawalManager`, `unlockedWithdrawalsCount[ETH_TOKEN]++`.
5. Operator calls `completeWithdrawalForUser(ETH_TOKEN, integrator, "")` → `_transferAsset` attempts `payable(integrator).call{value: amount}("")` → reverts → transaction rolls back.
6. No other function can move the ETH out: `sweepRemainingAssets` reverts because `hasUnlockedWithdrawals(ETH_TOKEN)` is `true`.
7. ETH is permanently frozen in `LRTWithdrawalManager`. [8](#0-7) [9](#0-8)
