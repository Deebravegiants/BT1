### Title
`feeBps` Lacks Upper-Bound Validation in `initialize()` While `setFeeBps()` Enforces It â€” Multiple Pool Contracts

### Summary

Every L2 pool contract in the LRT-rsETH suite accepts a `_feeBps` parameter in its `initialize()` function and stores it without any upper-bound check. The corresponding `setFeeBps()` setter in each contract correctly enforces a maximum (e.g., `> 10_000` or `> 1000`), but that guard is absent from the initializer. If `feeBps` is set to `10_000` (100%) at deployment, every subsequent depositor receives 0 rsETH/wrsETH for their ETH, with the full deposit silently routed to the fee pool.

---

If `feeBps == 10_000`, then `fee = amount`, `amountAfterFee = 0`, and `rsETHAmount = 0`. The user's ETH is consumed entirely as a fee and credited to `feeEarnedInETH`, which is withdrawable only by `BRIDGER_ROLE`. If `feeBps > 10_000`, the subtraction `amount - fee` underflows and reverts, bricking all deposits.

### Impact Explanation

**Impact: Lowâ€“Medium.** If `feeBps` is initialized to `10_000`, every depositor calling `deposit()` receives 0 rsETH/wrsETH while their ETH is locked in the contract as fees. This constitutes "contract fails to deliver promised returns" (Low) at minimum. In the worst case, before the admin corrects the value via `setFeeBps`, user ETH is temporarily frozen and routed to the fee pool â€” matching "temporary freezing of funds" (Medium). The admin can remediate by calling `setFeeBps(0)`, but deposits made before the fix are already lost to fees.

---

### Likelihood Explanation

**Likelihood: Low.** Requires a human error during deployment â€” the deployer passes an out-of-range `_feeBps`. The asymmetry between the guarded setter and the unguarded initializer makes this a realistic misconfiguration risk, especially across eight contracts with the same pattern. No attacker action is required; the misconfiguration alone triggers the impact for all subsequent depositors.

---

### Recommendation

Apply the same upper-bound check used in `setFeeBps()` inside each `initialize()` function. For example, in `RSETHPoolV2NBA`:

```solidity
if (_feeBps > 10_000) revert InvalidFeeAmount();
feeBps = _feeBps;
```

And in `RSETHPoolV3` / `RSETHPoolV3ExternalBridge` / `RSETHPoolV3WithNativeChainBridge`:

```solidity
if (_feeBps > 1000) revert InvalidFeeAmount();
feeBps = _feeBps;
```

This mirrors the fix applied in the referenced Debt DAO report: enforce the same constraint at initialization time that is already enforced in the update path.

---

### Proof of Concept

1. Deploy `RSETHPoolV2NBA` with `_feeBps = 10_000`.
2. Any user calls `deposit{value: 1 ether}("ref")`.
3. `viewSwapRsETHAmountAndFee(1 ether)` computes `fee = 1 ether * 10_000 / 10_000 = 1 ether`, `amountAfterFee = 0`, `rsETHAmount = 0`.
4. `wrsETH.mint(msg.sender, 0)` is called â€” user receives 0 rsETH.
5. `feeEarnedInETH += 1 ether` â€” the full deposit is credited to the fee pool.
6. Only `BRIDGER_ROLE` can recover the ETH via `withdrawFees()`. [6](#0-5)

    function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {

        uint256 amountAfterFee = amount - fee;

        // Calculate the final rsETH amount

    }
