### Title
ETH Deposit Limit Never Enforced — `amount` Omitted from Cap Check in `_checkIfDepositAmountExceedesCurrentLimit` — (`contracts/LRTDepositPool.sol`)

---

### Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses two structurally different comparisons: the ERC20 branch checks `totalAssetDeposits + amount > depositLimit`, while the ETH branch checks only `totalAssetDeposits > depositLimit`, silently dropping `amount`. As a result, any ETH deposit is permitted as long as the running total has not already exceeded the limit — the incoming deposit amount is never factored in.

---

### Finding Description
In `LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` (line 676–682), the function is supposed to return `true` when a proposed deposit would push total deposits over the configured cap. For ERC20 assets the check is correct:

```solidity
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
```

For ETH the check is:

```solidity
return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
```

Two distinct defects are present simultaneously:

1. **`amount` is not added.** The check evaluates the *current* total, not the *post-deposit* total. A deposit of any size — 1 wei or 1 000 ETH — passes the check identically.
2. **Strict `>` instead of `>=`.** Even if `amount` were zero, a state where `totalAssetDeposits == depositLimit` would still return `false`, allowing one more deposit.

Combined effect: whenever `totalAssetDeposits ≤ depositLimit`, the function returns `false` regardless of `amount`, and `_beforeDeposit` does not revert. A single call to `depositETH` can push total ETH deposits arbitrarily far above `depositLimitByAsset(ETH_TOKEN)`. [1](#0-0) 

The caller path is fully unprivileged:

```
depositETH(minRSETHAmountExpected, referralId)   [external payable, no role check]
  └─ _beforeDeposit(ETH_TOKEN, msg.value, ...)
       └─ _checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, msg.value)
            └─ returns false  ← never reverts for ETH
``` [2](#0-1) [3](#0-2) 

---

### Impact Explanation
The deposit limit for ETH is entirely unenforced. Any unprivileged depositor can call `depositETH` with an arbitrary `msg.value` and receive rsETH in return, regardless of how much ETH has already been deposited. The protocol's TVL cap — a primary risk-management invariant — is silently bypassed for every ETH deposit. rsETH is minted against ETH that exceeds the administrator-configured limit, violating the protocol's stated invariant that no asset's total deposits may exceed `depositLimitByAsset`. This constitutes a contract failing to deliver its promised behavior (enforced deposit caps), and depending on the reason the cap was set (e.g., EigenLayer strategy exposure limits, liquidity constraints), can escalate to protocol insolvency if the underlying strategy cannot absorb the excess.

---

### Likelihood Explanation
The entry point is `depositETH`, which is public, payable, and requires no role. The precondition (`totalAssetDeposits ≤ depositLimit`) is the normal operating state of the protocol. Any depositor — including a bot — can trigger this at any time without coordination or privilege. Likelihood is high.

---

### Recommendation
Replace the ETH branch to mirror the ERC20 branch exactly:

```solidity
// Before (broken):
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
}

// After (correct):
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [4](#0-3)
