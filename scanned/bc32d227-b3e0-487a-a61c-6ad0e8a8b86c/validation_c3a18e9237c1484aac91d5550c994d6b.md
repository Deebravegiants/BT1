### Title
Stale `rsETHPrice` Served as Fresh via `RSETHPriceFeed.latestRoundData` After Auto-Pause â€” (`contracts/oracles/RSETHPriceFeed.sol`)

---

### Summary

When `LRTOracle._updateRsETHPrice()` detects a price drop exceeding `pricePercentageLimit`, it auto-pauses the protocol and **returns early without writing the new (lower) price to `rsETHPrice`**. `RSETHPriceFeed.latestRoundData()` then serves this stale, inflated `rsETHPrice` multiplied by a **fresh** ETH/USD `updatedAt` timestamp, causing external lending protocols (e.g., Aave) to accept the stale price as current.

---

### Finding Description

**Step 1 â€” Auto-pause leaves `rsETHPrice` stale.**

In `LRTOracle._updateRsETHPrice()`, when the computed new price drops more than `pricePercentageLimit` below `highestRsethPrice`, the function pauses the protocol and returns **before** the `rsETHPrice = newRsETHPrice` assignment: [1](#0-0) 

The storage variable `rsETHPrice` retains the pre-pause (inflated) value.

**Step 2 â€” `updateRSETHPriceAsManager()` cannot fix it either.**

`updateRSETHPriceAsManager()` bypasses `whenNotPaused` but calls the same `_updateRsETHPrice()` internally: [2](#0-1) 

As long as the TVL remains depressed (i.e., `isPriceDecreaseOffLimit` stays true), every call to `_updateRsETHPrice()` hits the same early `return` at line 281, so `rsETHPrice` is permanently frozen at the stale value until TVL recovers above the threshold.

**Step 3 â€” `RSETHPriceFeed.latestRoundData()` blindly reads the stale value.** [3](#0-2) 

There is no staleness check on `RS_ETH_ORACLE.rsETHPrice()`. The function reads whatever is stored in `LRTOracle.rsETHPrice` and multiplies it by the live ETH/USD answer.

**Step 4 â€” `updatedAt` is sourced from the ETH/USD feed, not from rsETHPrice.**

The `updatedAt` field returned to callers comes from `ETH_TO_USD.latestRoundData()` (line 68), which reflects the Chainlink ETH/USD heartbeat (typically â‰¤1 hour old). External protocols like Aave use `updatedAt` as their sole freshness signal. Because `updatedAt` is always recent, Aave's staleness guard passes even though the rsETH/ETH component has not been updated since the auto-pause. [4](#0-3) 

---

### Impact Explanation

An attacker holding rsETH can deposit it as collateral on Aave (or any protocol consuming `RSETHPriceFeed`) and borrow against the stale, inflated rsETH/USD price. Because the real value of rsETH has dropped (e.g., 20%), the attacker extracts more debt than the collateral is worth. If the attacker walks away, the lending protocol is left with undercollateralized positions â€” **protocol insolvency** for the lending integration and indirect reputational/financial damage to rsETH.

---

### Likelihood Explanation

The precondition (TVL drop exceeding `pricePercentageLimit`) is realistic: a slashing event on an underlying EigenLayer operator, or a significant price drop in a supported LST (stETH, cbETH), can trigger it. The auto-pause is a public, permissionless path â€” any caller of `updateRSETHPrice()` can trigger it once the TVL condition is met. The attacker does not need any privileged role; they only need to observe the on-chain state and act within the window.

---

### Recommendation

1. **Expose a `rsETHPriceUpdatedAt` timestamp** in `LRTOracle` and update it every time `rsETHPrice` is written. Have `RSETHPriceFeed.latestRoundData()` return `min(ethUsdUpdatedAt, rsETHPriceUpdatedAt)` as `updatedAt`, so external protocols see the true staleness of the composite price.
2. **Allow the price to be written even during auto-pause** (at least downward), so the stored value reflects reality. The auto-pause is a circuit breaker for deposits/withdrawals, not a reason to freeze the price feed at an inflated level.
3. Alternatively, have `RSETHPriceFeed.latestRoundData()` revert (or return `updatedAt = 0`) when `LRTOracle` is paused, so downstream protocols treat the feed as unavailable rather than stale-but-fresh.
