### Title
Daily Fee Mint Limit Causes Complete Price Update Lockout — (`contracts/LRTOracle.sol`)

### Summary

`_checkAndUpdateDailyFeeMintLimit` is called unconditionally inside `_updateRsETHPrice`, and it reverts with `DailyFeeMintLimitExceeded` when accumulated fee minting within a period reaches `maxFeeMintAmountPerDay`. Because both `updateRSETHPrice()` and `updateRSETHPriceAsManager()` route through `_updateRsETHPrice`, neither function can succeed once the limit is hit. `rsETHPrice` is left stale for up to 24 hours with no admin escape path.

### Finding Description

`_updateRsETHPrice` always calls `_checkAndUpdateDailyFeeMintLimit`, regardless of whether the caller is a privileged manager: [1](#0-0) 

Both public entry points delegate to the same internal function with no bypass:

```
updateRSETHPrice()          → _updateRsETHPrice() → _checkAndUpdateDailyFeeMintLimit(fee)
updateRSETHPriceAsManager() → _updateRsETHPrice() → _checkAndUpdateDailyFeeMintLimit(fee)
```

Inside `_checkAndUpdateDailyFeeMintLimit`, the revert fires before any state is written: [2](#0-1) 

The revert at line 206 unwinds the entire call, so `rsETHPrice = newRsETHPrice` at line 313 is never reached: [3](#0-2) 

**Trigger sequence:**

1. Over the course of a 24-hour window, multiple successful price updates accumulate `currentPeriodMintedFeeAmount` close to `maxFeeMintAmountPerDay`.
2. A subsequent TVL increase (e.g., staking rewards accruing in EigenLayer) produces a `rsethAmountToMintAsProtocolFee` that pushes `currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay`.
3. Both `updateRSETHPrice()` and `updateRSETHPriceAsManager()` revert for the remainder of the period.
4. `rsETHPrice` is frozen at its last value until `block.timestamp >= feePeriodStartTime + 1 days`, at which point `_checkAndUpdateDailyFeeMintLimit` resets the counter. [4](#0-3) 

There is no function in `LRTOracle` that updates `rsETHPrice` while skipping the fee-limit check.

### Impact Explanation

**Medium — Temporary freezing of funds.**

`rsETHPrice` is the exchange rate used by `LRTWithdrawalManager` to determine ETH owed to withdrawers. A stale price for up to 24 hours means withdrawal queue processing either uses an incorrect rate or is blocked entirely if the manager enforces price freshness. Depositors and withdrawers cannot get accurate settlement during the lockout window.

### Likelihood Explanation

Likelihood is **medium**. The scenario requires:
- `maxFeeMintAmountPerDay` to be set to a finite value (it is a configurable parameter, expected to be set in production).
- Enough TVL growth within a single 24-hour window to exhaust the daily fee cap.

Both conditions are normal operating states for a live protocol with significant TVL and active staking rewards. No attacker action is required; organic protocol operation is sufficient.

### Recommendation

Decouple the fee-minting limit from the price-update path. Two options:

1. **Skip fee minting (not the price update) when the limit is reached.** Instead of reverting, simply skip the `IRSETH.mint` call and emit an event indicating the fee was waived for the period. The price update proceeds normally.

2. **Add a manager bypass.** In `_updateRsETHPrice`, check whether `msg.sender` holds `MANAGER` role (already done for the price-threshold check at line 263) and skip the fee-limit revert for managers, allowing `updateRSETHPriceAsManager()` to always succeed.

Option 1 is safer because it removes the revert entirely from the price-update hot path.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Invariant: after the daily fee cap is exhausted within a period,
// both updateRSETHPrice() and updateRSETHPriceAsManager() must revert.

function test_dailyFeeLimitBlocksPriceUpdate() public {
    // Setup: maxFeeMintAmountPerDay = 1 ether, feePeriodStartTime = now
    oracle.setMaxFeeMintAmountPerDay(1 ether);

    // Step 1: drive currentPeriodMintedFeeAmount to just below the cap
    // by calling updateRSETHPrice() repeatedly while TVL grows slightly each time
    // until currentPeriodMintedFeeAmount == 0.99 ether

    // Step 2: simulate a large TVL jump so feeAmount = 0.02 ether
    // (pushes total to 1.01 ether > maxFeeMintAmountPerDay)
    mockTVLIncrease(largeAmount);

    // Step 3: both calls revert
    vm.expectRevert(LRTOracle.DailyFeeMintLimitExceeded.selector);
    oracle.updateRSETHPrice();

    vm.expectRevert(LRTOracle.DailyFeeMintLimitExceeded.selector);
    vm.prank(manager);
    oracle.updateRSETHPriceAsManager();

    // Step 4: warp past the period boundary — calls succeed again
    vm.warp(block.timestamp + 1 days);
    oracle.updateRSETHPrice(); // succeeds, resets counter
}
```

The test confirms that no privileged role can bypass the lockout, and the only resolution is waiting for the 24-hour period to expire.

```text
    function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
        // Reset the period if it's unset or a day has passed
        if (block.timestamp >= feePeriodStartTime + 1 days) {
            currentPeriodMintedFeeAmount = 0;
            feePeriodStartTime = getCurrentPeriodStartTime();
        }

        // Check if minting would exceed the daily limit
        if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
            revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
        }

        currentPeriodMintedFeeAmount += feeAmount;
    }
```
