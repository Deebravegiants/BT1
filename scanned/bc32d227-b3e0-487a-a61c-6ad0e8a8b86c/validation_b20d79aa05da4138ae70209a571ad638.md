### Title
Excess Native ETH Sent to `bridgeKernelToBSC` Gets Permanently Locked in Contract - (File: contracts/KERNEL/KernelVaultETH.sol)

### Summary
`KernelVaultETH.bridgeKernelToBSC` uses a `msg.value < nativeFee` (less-than) guard instead of an exact-equality check. Any ETH sent above `nativeFee` is silently accepted but never forwarded or refunded, permanently locking the surplus in the contract with no recovery path.

### Finding Description
`bridgeKernelToBSC` is a `payable` function that accepts a caller-supplied `nativeFee` parameter and forwards exactly that amount to the LayerZero OFT adapter:

```solidity
// KernelVaultETH.sol line 246-248
if (msg.value < nativeFee) {
    revert InsufficientNativeFee();
}
// ...
kernelOftAdapter.send{ value: nativeFee }(sendParam, fee, refundAddress);
```

The guard only rejects under-payment. If `msg.value > nativeFee`, the call succeeds, `nativeFee` wei is forwarded to `kernelOftAdapter`, and the remainder (`msg.value - nativeFee`) stays in the contract. There is no `receive`/`fallback` ETH-sweep, no admin ETH-withdrawal function, and no refund logic anywhere in the contract, so the surplus is permanently unrecoverable.

This contrasts with every other bridge function in the repository (e.g., `RSETHPool.bridgeAssets`, `L1Vault.bridgeRsETHToL2`, `TACWETHBridge.bridgeTokenToL1`) which all use `msg.value != nativeFee` and revert on any mismatch.

### Impact Explanation
Any ETH sent in excess of `nativeFee` is permanently frozen inside `KernelVaultETH`. Because the contract has no ETH recovery mechanism, the surplus cannot be retrieved by any role. This constitutes **permanent freezing of funds**.

### Likelihood Explanation
`bridgeKernelToBSC` is called by the `OPERATOR_ROLE` — a role that is expected to call this function regularly as part of normal protocol operations. LayerZero native fees are not perfectly predictable off-chain; an operator may intentionally over-provision `msg.value` as a safety margin (a common practice), or may simply miscalculate. Because the contract silently accepts the overpayment, the mistake goes undetected until the ETH is noticed as missing. The likelihood is realistic given routine operational use.

### Recommendation
Replace the less-than guard with an exact-equality check, consistent with all other bridge functions in the codebase:

```solidity
// Before (vulnerable)
if (msg.value < nativeFee) {
    revert InsufficientNativeFee();
}

// After (safe)
if (msg.value != nativeFee) {
    revert InsufficientNativeFee();
}
```

Alternatively, refund any surplus: `if (msg.value > nativeFee) { payable(refundAddress).transfer(msg.value - nativeFee); }`.

### Proof of Concept
1. Operator calls `bridgeKernelToBSC(amount, minAmount, 0.01 ether, refundAddress)` but sends `msg.value = 0.02 ether`.
2. Check at line 246: `0.02 ether < 0.01 ether` → false → no revert.
3. Line 264: `kernelOftAdapter.send{ value: 0.01 ether }(...)` — only `nativeFee` is forwarded.
4. Remaining `0.01 ether` stays in `KernelVaultETH` with no withdrawal path.
5. ETH is permanently locked. [1](#0-0) [2](#0-1)
