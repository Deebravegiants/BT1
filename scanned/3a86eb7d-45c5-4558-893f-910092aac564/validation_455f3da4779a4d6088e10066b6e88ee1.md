### Title
Strict `msg.value == getCCIPFee()` Equality Check Enables Block-Stuffing to Repeatedly Revert `bridgeRsETHToL2UsingCCIP` — (`contracts/L1VaultV2.sol`)

---

### Summary

`bridgeRsETHToL2UsingCCIP` re-computes the CCIP fee on-chain at execution time and enforces **exact equality** with `msg.value`. An attacker can stuff blocks to delay the manager's pending transaction while simultaneously driving up gas prices, causing the on-chain fee to exceed the off-chain-quoted value and reverting with `IncorrectCCIPFee`. Because the CCIP router accepts overpayment but the vault does not, the manager has no way to send a fee buffer.

---

### Finding Description

The function `bridgeRsETHToL2UsingCCIP` calls `getCCIPFee(amount)` **inside the transaction** at execution time: [1](#0-0) 

`getCCIPFee` delegates directly to `ccipRouter.getFee()`: [2](#0-1) 

The intended usage pattern is:
1. Manager calls `getCCIPFee(amount)` off-chain to determine `msg.value`.
2. Manager submits `bridgeRsETHToL2UsingCCIP{value: fee_quoted}(amount)`.
3. On-chain, `getCCIPFee` is called again; if `fee_current != fee_quoted`, the tx reverts.

The strict `!=` check means the manager **cannot** send a fee buffer (e.g., `fee_quoted + 1 wei`) to absorb fluctuations. Yet the CCIP router itself explicitly accepts overpayment: [3](#0-2) 

An attacker exploits this by:
1. Observing the manager's pending transaction in the mempool.
2. Submitting many high-gas transactions to fill blocks (block stuffing), delaying the manager's tx.
3. The block-filling activity drives up the base fee / gas price, which CCIP's `getFee` reflects.
4. When the manager's tx finally lands, `getCCIPFee(amount)` returns a higher value → `IncorrectCCIPFee` revert.
5. The attacker repeats this each time the manager resubmits, keeping rsETH frozen in the vault.

---

### Impact Explanation

rsETH accumulates in the vault and cannot be delivered to L2 users for as long as the attacker sustains the block-stuffing campaign. Each revert forces the manager to re-query the fee and resubmit, and the attacker can immediately repeat the attack. This constitutes a **temporary freezing of funds** and delayed delivery to L2 users, matching the scoped impact **Low. Block stuffing**.

---

### Likelihood Explanation

Block stuffing on Ethereum mainnet is expensive but economically rational if the attacker has a competing interest (e.g., holding a short position on rsETH liquidity on L2, or extracting MEV from the delayed bridging). The attack requires no privileged access — only ETH to fill blocks. The manager has no on-chain mechanism to escape the attack (cannot overpay, cannot pass a fee tolerance).

---

### Recommendation

Replace the exact-equality check with a minimum-fee check, allowing the manager to send at least the quoted fee:

```solidity
// Instead of:
if (msg.value != fee) revert IncorrectCCIPFee();

// Use:
if (msg.value < fee) revert IncorrectCCIPFee();
// Refund excess: if (msg.value > fee) { (bool ok,) = msg.sender.call{value: msg.value - fee}(""); require(ok); }
```

This mirrors the CCIP router's own overpayment semantics and lets the manager include a small buffer (e.g., 5–10%) to absorb fee fluctuations between off-chain quote and on-chain execution, eliminating the block-stuffing attack surface.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.21;

import "forge-std/Test.sol";
import "../contracts/L1VaultV2.sol";

contract MockCCIPRouter {
    uint256 public feePerBlock = 0.01 ether;
    uint256 public deployBlock;

    constructor() { deployBlock = block.number; }

    function getFee(uint64, Client.EVM2AnyMessage memory) external view returns (uint256) {
        // Fee increases by feePerBlock for each block elapsed
        return 0.1 ether + (block.number - deployBlock) * feePerBlock;
    }

    function ccipSend(uint64, Client.EVM2AnyMessage calldata) external payable returns (bytes32) {
        return bytes32(0);
    }
}

contract BlockStuffingTest is Test {
    L1VaultV2 vault;
    MockCCIPRouter router;
    address manager = address(0xBEEF);

    function setUp() public {
        router = new MockCCIPRouter();
        // deploy & initialize vault with router, grant MANAGER_ROLE to manager
        // ... (setup omitted for brevity)
    }

    function testFuzz_blockStuffingCausesRevert(uint8 blockDelay) public {
        vm.assume(blockDelay >= 1 && blockDelay <= 50);

        uint256 amount = 1 ether;
        // Step 1: manager queries fee off-chain at current block
        uint256 quotedFee = vault.getCCIPFee(amount);

        // Step 2: attacker stuffs `blockDelay` blocks
        vm.roll(block.number + blockDelay);

        // Step 3: manager submits with stale fee
        vm.prank(manager);
        vm.expectRevert(L1VaultV2.IncorrectCCIPFee.selector);
        vault.bridgeRsETHToL2UsingCCIP{value: quotedFee}(amount);
    }
}
```

For any `blockDelay >= 1` where `MockCCIPRouter.getFee` returns a higher value, the assertion holds: `bridgeRsETHToL2UsingCCIP` reverts with `IncorrectCCIPFee` on every iteration, confirming the block-stuffing griefing path.
