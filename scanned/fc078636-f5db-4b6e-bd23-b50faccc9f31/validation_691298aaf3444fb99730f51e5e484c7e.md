### Title
Withdrawal Not Linked to Deposited Token Allows Cross-Token Drain - (File: contracts/L2/RsETHTokenWrapper.sol)

---

### Summary

`RsETHTokenWrapper._withdraw` only validates that the requested `_asset` is in `allowedTokens`, but never verifies that the withdrawn token corresponds to the token originally deposited. Because the contract explicitly supports multiple allowed tokens (via `reinitialize` and `addAllowedToken`), any holder of `wrsETH` can burn it to redeem whichever allowed token they choose — including one they never deposited — draining other depositors' collateral.

---

### Finding Description

`RsETHTokenWrapper` is a 1:1 wrapper: depositing N units of any `allowedToken` mints N `wrsETH`; burning N `wrsETH` transfers N units of any `allowedToken` back to the caller.

The `_deposit` function:

```solidity
function _deposit(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    ERC20Upgradeable(_asset).safeTransferFrom(msg.sender, address(this), _amount);
    _mint(_to, _amount);
    ...
}
``` [1](#0-0) 

The `_withdraw` function:

```solidity
function _withdraw(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    _burn(msg.sender, _amount);
    ERC20Upgradeable(_asset).safeTransfer(_to, _amount);
    ...
}
``` [2](#0-1) 

The only guard in `_withdraw` is `allowedTokens[_asset]`. There is no record of which token a given `wrsETH` balance was minted against, and no check that the caller withdraws the same token they deposited. The contract explicitly supports multiple allowed tokens: `initialize` registers the first, `reinitialize` (callable by `DEFAULT_ADMIN_ROLE`) registers a second, and `addAllowedToken` (callable by `TIMELOCK_ROLE`) can register further tokens. [3](#0-2) [4](#0-3) 

---

### Impact Explanation

**Critical — direct theft of user funds.**

Once two tokens are simultaneously in `allowedTokens`, any user can deposit the lower-value token and withdraw the higher-value token at a 1:1 nominal ratio, extracting the price difference from other depositors. The entire balance of the more valuable token held by the wrapper (including collateral deposited by the bridger via `depositBridgerAssets`) is at risk. There is no per-token accounting, so the theft is bounded only by the wrapper's balance of the target token.

---

### Likelihood Explanation

**Medium.** The multi-token design is intentional: `reinitialize` exists specifically to add a second altRsETH token to a live deployment, and `addAllowedToken` allows further additions via governance. The moment a second token is active and the two tokens trade at any price difference (common for different bridge representations of the same underlying), the attack is immediately profitable for any unprivileged user with no special access.

---

### Recommendation

Track per-token balances internally. When minting `wrsETH`, record which token was deposited and in what amount. On withdrawal, only allow redeeming the same token that was deposited, or enforce that the total redeemable amount of each token is bounded by the amount of that specific token held as collateral for outstanding `wrsETH`. Alternatively, restrict the wrapper to a single canonical token at any given time and require a migration path when switching.

---

### Proof of Concept

Precondition: `altRsETH1` and `altRsETH2` are both in `allowedTokens`. `altRsETH2` trades at a 1% premium over `altRsETH1`. Other users have deposited 10,000 `altRsETH2` into the wrapper.

1. Attacker calls `deposit(altRsETH1, 10_000)` — deposits 10,000 cheap `altRsETH1`, receives 10,000 `wrsETH`. [5](#0-4) 

2. Attacker calls `withdraw(altRsETH2, 10_000)` — burns 10,000 `wrsETH`, receives 10,000 `altRsETH2`. [6](#0-5) 

3. `_withdraw` passes because `allowedTokens[altRsETH2] == true`. No check verifies that the caller's `wrsETH` was minted from `altRsETH2`. [7](#0-6) 

4. Attacker now holds 10,000 `altRsETH2` (worth ~100 ETH more than the 10,000 `altRsETH1` they deposited). The wrapper is left holding 10,000 `altRsETH1` instead of `altRsETH2`, and the original `altRsETH2` depositors cannot recover their tokens.
