### Title
Stale Merkle Root Overwrites Permanently Lock Unclaimed Blast Points — (`contracts/utils/MerkleDistributor/MerkleBlastPointsDistributor.sol`)

### Summary
`setMerkleRoot` unconditionally overwrites `currentMerkleRoot` with no historical root storage and no claim grace period. Any user who has not yet called `claim` for root R1 before the owner publishes R2 loses their R1 entitlement permanently, because `claim` verifies proofs exclusively against the single live `currentMerkleRoot`.

### Finding Description

`setMerkleRoot` replaces the single stored root and increments `currentIndex`: [1](#0-0) 

`claim` then verifies the caller's proof against only that live root: [2](#0-1) 

There is no mapping of `rootIndex → merkleRoot`, no historical root lookup, and no minimum claim window enforced before a root can be replaced.

**Concrete revert path after R1 → R2 transition:**

| Call | Guard hit | Revert |
|---|---|---|
| `claim(1, user, 100, 0, proof_R1)` | proof verified against `currentMerkleRoot = R2` | `InvalidMerkleProof` |
| `claim(2, user, 100, 0, proof_R2)` | R2 omits user entirely | `InvalidMerkleProof` |
| `claim(2, user, 100, 0, proof_R2)` | R2 carries same cumulative (100, 0) | succeeds — but only if owner correctly carries forward amounts |

The `index > currentIndex` guard at line 101 does not help: after R2 is set, `currentIndex = 2`, so `claim(1, ...)` passes that check but fails proof verification because R1 is gone. [3](#0-2) 

The `isClaimed` check is also irrelevant — it returns `false` for a user who never claimed R1, so it does not block the call; the proof check is what kills it. [4](#0-3) 

### Impact Explanation
Users with valid proofs for a superseded root permanently lose their Blast Points entitlement if the new root omits them or resets their cumulative amount. `userClaims[account].cumulativeBlastPointAmount` remains 0 and no `Claimed` event is ever emitted. This matches **Low — contract fails to deliver promised returns, but doesn't lose value** (the points remain under operator control; they are simply unclaimable by the entitled user).

### Likelihood Explanation
The scenario does not require malicious intent. It arises from ordinary operational patterns:
- Owner publishes R1, discovers an error, and immediately publishes a corrected R2 before users claim.
- Owner runs an automated script that publishes roots on a fixed schedule, and a user misses the window between two consecutive publications.
- R2 is generated from a snapshot that excludes users who were included in R1 (off-chain tooling bug).

No private key compromise, front-running, or governance capture is required — only the owner performing their normal `setMerkleRoot` duty.

### Recommendation
1. **Store historical roots**: use a `mapping(uint256 rootIndex => bytes32 root) public merkleRoots` and verify the proof against `merkleRoots[index]` rather than `currentMerkleRoot`.
2. **Alternatively**, enforce a mandatory claim window (e.g., a `claimDeadline` per root) before a new root can be set, giving users guaranteed time to claim.
3. **At minimum**, document and enforce off-chain that every new root must carry forward all cumulative amounts from the previous root for every user who appeared in it.
