### Title
Stale Chainlink Price Accepted Without Validation Enables Share Mis-Accounting — (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

### Summary

`ChainlinkPriceOracle.getAssetPrice` consumes raw Chainlink `latestRoundData()` output without validating staleness, round completeness, or price sign. This unvalidated oracle value flows directly into rsETH mint calculations and the global rsETH price update, mirroring the external report's pattern of unvalidated input being processed in a critical dispatch path.

### Finding Description

`ChainlinkPriceOracle.getAssetPrice` silently discards all Chainlink safety fields:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol:49-55
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

The return tuple `(roundId, answer, startedAt, updatedAt, answeredInRound)` is destructured with four unnamed blanks. No check is made on:
- `updatedAt` — whether the answer is within an acceptable heartbeat window
- `answeredInRound >= roundId` — whether the round completed
- `price > 0` — whether the answer is positive

The same codebase already implements all three checks in `ChainlinkOracleForRSETHPoolCollateral.getRate` (used for the L2 pool):

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol:27-36
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The unvalidated `ChainlinkPriceOracle` is the oracle registered for L1 LST assets (stETH, rETH, etc.) via `LRTOracle.updatePriceOracleFor`.

### Impact Explanation

The stale price propagates through two critical paths:

**Path 1 — rsETH mint over-issuance:**
`depositAsset` → `_beforeDeposit` → `getRsETHAmountToMint` → `lrtOracle.getAssetPrice(asset)` → `ChainlinkPriceOracle.getAssetPrice`. [1](#0-0) 

If the stale price is inflated (e.g., the feed has not updated during a market drop), a depositor receives more rsETH than the asset is worth, diluting all existing rsETH holders.

**Path 2 — global rsETH price corruption:**
`updateRSETHPrice` → `_getTotalEthInProtocol` → `getAssetPrice` for every supported asset. [2](#0-1) 

A stale price inflates `totalETHInProtocol`, causing `newRsETHPrice` to be computed too high, which in turn causes the fee-minting logic to mint excess rsETH to the treasury, stealing yield from holders. [3](#0-2) 

Impact classification: **High — theft of unclaimed yield / share dilution** for existing rsETH holders.

### Likelihood Explanation

Chainlink feeds have documented heartbeat intervals (e.g., 24 h for stETH/ETH on mainnet). During periods of network congestion, sequencer downtime, or when a feed is deprecated and replaced, `updatedAt` can lag by hours. This is a well-known, historically observed condition, not a theoretical one. No privileged access is required; any depositor can trigger the vulnerable path by calling `depositAsset` while the feed is stale.

### Recommendation

Apply the same validation already present in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

if (answeredInRound < roundId) revert StalePrice();
if (updatedAt == 0) revert IncompleteRound();
if (price <= 0) revert InvalidPrice();
if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();
```

`MAX_STALENESS` should be set per-asset to match the feed's documented heartbeat. [4](#0-3) [5](#0-4) 

### Proof of Concept

1. Chainlink stETH/ETH feed on mainnet enters a stale period (heartbeat exceeded, e.g., during high gas or sequencer lag). `updatedAt` is 25 hours old; actual stETH price has dropped 5% but the feed still reports the old value.
2. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
3. `getRsETHAmountToMint` calls `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice` → returns the stale inflated price (e.g., `1.05e18` instead of `1.00e18`).
4. `rsethAmountToMint = (100e18 * 1.05e18) / rsETHPrice` — attacker receives ~5% more rsETH than the deposited stETH is worth.
5. Attacker later redeems rsETH at the correct rate, extracting value from existing holders.

No admin interaction, no front-running, no brute force required. The attacker only needs to observe that the feed is stale and submit a standard deposit transaction. [6](#0-5) [4](#0-3)
