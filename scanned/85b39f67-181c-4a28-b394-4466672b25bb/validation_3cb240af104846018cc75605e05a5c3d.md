### Title
Missing Minimum Output Check in L2 Pool `deposit` Functions Exposes Depositors to Unfavorable Rate Execution - (File: contracts/pools/RSETHPoolV3.sol)

### Summary
The L2 pool deposit functions in `RSETHPoolV3.sol`, `RSETHPool.sol`, and `RSETHPoolNoWrapper.sol` accept ETH or LST tokens and return rsETH/wrsETH based on a live oracle rate, but provide no `minRsETHAmount` parameter. A depositor has no way to enforce a minimum acceptable output, meaning any oracle rate movement between transaction submission and on-chain execution silently reduces the rsETH they receive with no revert path.

### Finding Description
The L1 `LRTDepositPool.depositETH` and `depositAsset` functions correctly accept a `minRSETHAmountExpected` argument and enforce it inside `_beforeDeposit`:

```solidity
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [1](#0-0) 

The L2 pool equivalents carry no such guard. In `RSETHPoolV3.sol` the ETH deposit path is:

```solidity
function deposit(string memory referralId)
    external payable nonReentrant whenNotPaused
    limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);
    ...
}
``` [2](#0-1) 

`rsETHAmount` is computed entirely from the live oracle rate at execution time:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [3](#0-2) 

The same pattern is present in `RSETHPool.sol` and `RSETHPoolNoWrapper.sol`: [4](#0-3) [5](#0-4) 

The oracle rate (`rsETHOracle`) is a mutable address settable by `TIMELOCK_ROLE`, and the rate it returns can change at any time between when a user previews their deposit and when the transaction is mined. [6](#0-5) 

### Impact Explanation
A depositor who previews a favorable rate and submits a transaction receives whatever rate the oracle reports at execution time, with no on-chain protection. If the rate moves adversely — whether through natural oracle update, network congestion delaying the transaction, or a sequencer reordering — the user receives fewer rsETH/wrsETH than they agreed to. Their deposited ETH/LST is already transferred into the pool with no refund path. This matches the **Low** impact tier: *Contract fails to deliver promised returns, but doesn't lose value* — the deposited ETH is not stolen, but the user receives a worse exchange than they observed and accepted off-chain.

### Likelihood Explanation
L2 oracle rates for rsETH are updated periodically by the protocol's off-chain infrastructure. On chains with variable block times or sequencer reordering, a pending deposit transaction can easily execute against a stale or freshly updated rate. No attacker capability is required; ordinary oracle update timing is sufficient to trigger the discrepancy. Likelihood is **Medium** given the frequency of oracle updates and the absence of any on-chain guard.

### Recommendation
Add a `minRsETHAmount` parameter to all three `deposit` overloads in `RSETHPoolV3.sol`, `RSETHPool.sol`, and `RSETHPoolNoWrapper.sol`, mirroring the existing L1 guard:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmount)
    external payable nonReentrant whenNotPaused
    limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmount) revert MinimumAmountToReceiveNotMet();
    ...
}
```

Apply the same pattern to the token `deposit(address token, uint256 amount, string memory referralId, uint256 minRsETHAmount)` overloads.

### Proof of Concept
1. User calls `viewSwapRsETHAmountAndFee(1 ether)` on `RSETHPoolV3` and observes they will receive `0.95 wrsETH` at the current oracle rate of `~1.05 ETH/rsETH`.
2. User submits `deposit{value: 1 ether}("ref")`.
3. Before the transaction is mined, the oracle rate is updated to `1.15 ETH/rsETH` (a legitimate protocol update reflecting accrued rewards).
4. The transaction executes: `rsETHAmount = 1e18 * 1e18 / 1.15e18 = 0.869 wrsETH`.
5. User receives `0.869 wrsETH` instead of the `0.95 wrsETH` they observed — an ~8.5% shortfall — with no revert and no recourse. The 1 ETH is now locked in the pool. [7](#0-6)

        // Calculate the final rsETH amount
