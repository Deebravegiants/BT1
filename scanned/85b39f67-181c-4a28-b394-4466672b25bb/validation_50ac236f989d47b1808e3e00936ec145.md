### Title
Missing Zero-Address Validation for Critical Address Arguments in Pool Initializers - (File: contracts/pools/RSETHPool.sol)

### Summary
Multiple L2 pool `initialize` functions accept critical address-typed arguments — including `admin`, `manager`, and oracle addresses — without validating them against `address(0)`. The analog of the Pickle Finance report's vulnerability class is present across `RSETHPool`, `RSETHPoolV2`, `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, and `RsETHTokenWrapper`. The most impactful instance is `RSETHPool.initialize`, which validates only `_rsETH` and `_wstETH` while leaving `admin`, `manager`, `_rsETHOracle`, and `_wstETH_ETHOracle` unchecked.

### Finding Description

`RSETHPool.initialize` accepts seven address arguments but only validates two of them: [1](#0-0) 

Only `_rsETH` and `_wstETH` are passed through `UtilLib.checkNonZeroAddress`. The remaining four address arguments — `admin`, `manager`, `_rsETHOracle`, and `_wstETH_ETHOracle` — receive no zero-address validation. [2](#0-1) 

If `_rsETHOracle` is `address(0)`, it is stored directly: [3](#0-2) 

Every subsequent call to `getRate()` then calls `IOracle(address(0)).getRate()`, which reverts, breaking all deposit paths: [4](#0-3) 

If `admin` is `address(0)`, `DEFAULT_ADMIN_ROLE` is granted to the zero address, making `unpause()`, `setFeeBps()`, and role-granting permanently inaccessible with no recovery path. [5](#0-4) 

The same pattern — missing zero-address checks for `admin` and `bridger` — is present in `RSETHPoolV2.initialize`, `RSETHPoolV3.initialize`, `RSETHPoolV3ExternalBridge.initialize`, and `RsETHTokenWrapper.initialize`: [6](#0-5) [7](#0-6) [8](#0-7) [9](#0-8) 

By contrast, contracts such as `LRTDepositPool`, `L1Vault`, and `FeeReceiver` consistently validate all address arguments: [10](#0-9) [11](#0-10) 

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

If `_rsETHOracle` is initialized to `address(0)` in `RSETHPool`, every call to `deposit()` (both ETH and token variants) reverts because `viewSwapRsETHAmountAndFee` calls `getRate()` → `IOracle(address(0)).getRate()`. No user funds are lost, but the pool is rendered permanently non-functional for deposits. If `admin` is simultaneously zero, no privileged role can call `setRSETHOracle` to recover, since `TIMELOCK_ROLE` can only be granted by `DEFAULT_ADMIN_ROLE`.

### Likelihood Explanation

Low. The misconfiguration must occur at deployment time by the deployer passing `address(0)` for a critical argument. This is a deployment error, not an attacker-controlled path. However, the absence of a guard means the EVM will accept and persist the misconfiguration silently, with no on-chain revert to alert the deployer. The risk is elevated by the number of affected contracts (five pool initializers) and the fact that some arguments (`_rsETHOracle`, `_wstETH_ETHOracle`) are not obviously critical to a deployer scanning the parameter list.

### Recommendation

Add `UtilLib.checkNonZeroAddress` calls for every address-typed argument in each affected initializer, consistent with the pattern already used in `L1Vault.initialize` and `LRTDepositPool.initialize`. For `RSETHPool.initialize` specifically:

```solidity
function initialize(
    address admin,
    address manager,
    address _rsETH,
    address _wstETH,
    uint256 _feeBps,
    address _rsETHOracle,
    address _wstETH_ETHOracle
) external initializer {
    UtilLib.checkNonZeroAddress(admin);        // ADD
    UtilLib.checkNonZeroAddress(manager);      // ADD
    UtilLib.checkNonZeroAddress(_rsETH);
    UtilLib.checkNonZeroAddress(_wstETH);
    UtilLib.checkNonZeroAddress(_rsETHOracle); // ADD
    UtilLib.checkNonZeroAddress(_wstETH_ETHOracle); // ADD
    ...
}
```

Apply the same pattern to `RSETHPoolV2.initialize`, `RSETHPoolV3.initialize`, `RSETHPoolV3ExternalBridge.initialize`, and `RsETHTokenWrapper.initialize`.

### Proof of Concept

1. Deploy `RSETHPool` proxy and call `initialize` with `_rsETHOracle = address(0)` and a valid `_rsETH`/`_wstETH`. The transaction succeeds.
2. Call `deposit("ref")` with any ETH value.
3. Execution reaches `viewSwapRsETHAmountAndFee` → `getRate()` → `IOracle(address(0)).getRate()` → reverts with no meaningful error.
4. All deposit paths are permanently broken. If `admin` was also `address(0)`, `setRSETHOracle` is unreachable because `TIMELOCK_ROLE` cannot be granted. [12](#0-11)
