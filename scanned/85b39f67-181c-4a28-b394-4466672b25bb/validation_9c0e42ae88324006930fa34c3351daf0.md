### Title
Stale `rsETHPrice` Used in Deposit Minting Allows Depositors to Extract Excess rsETH from Existing Holders - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool.getRsETHAmountToMint()` divides a live Chainlink asset price by `lrtOracle.rsETHPrice()`, a cached state variable that is only updated when `updateRSETHPrice()` is called in a separate transaction. When staking rewards accrue and the cache is stale, any depositor can mint more rsETH than the current protocol value warrants, diluting existing holders' yield.

### Finding Description
`LRTDepositPool.getRsETHAmountToMint()` computes the rsETH to mint as:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [1](#0-0) 

`lrtOracle.getAssetPrice(asset)` reads a live price from the underlying Chainlink feed via `IPriceFetcher`, while `lrtOracle.rsETHPrice()` reads a **stored state variable** that is only refreshed when `updateRSETHPrice()` / `_updateRsETHPrice()` is explicitly called. [2](#0-1) [3](#0-2) 

`_updateRsETHPrice()` computes the true current price as `(totalETHInProtocol - protocolFee) / rsethSupply`: [4](#0-3) 

As EigenLayer staking rewards and LST appreciation accrue, `totalETHInProtocol` grows, so the true rsETH price rises. But `rsETHPrice` (the stored variable) remains at its last-written value until `updateRSETHPrice()` is called. There is no staleness check or forced price refresh anywhere in the deposit path (`depositETH` → `_beforeDeposit` → `getRsETHAmountToMint`): [5](#0-4) 

Additionally, `_updateRsETHPrice()` contains a `pricePercentageLimit` guard that **reverts for non-managers** if the price increase exceeds the configured threshold: [6](#0-5) 

This means that during periods of significant reward accrual, permissionless keepers cannot update the price at all, extending the staleness window and making the attack more reliable.

### Impact Explanation
When `rsETHPrice` is stale and lower than the true current value, the denominator in `getRsETHAmountToMint` is artificially small, causing the depositor to receive more rsETH than their deposit is worth. Once `updateRSETHPrice()` is eventually called, the new price is computed as `totalETHInProtocol / rsethSupply`; because the supply was inflated by the excess mint, the price per rsETH is lower than it would have been, directly reducing the redemption value of every existing holder's position. This constitutes **theft of unclaimed yield** from existing rsETH holders (High impact).

### Likelihood Explanation
- `updateRSETHPrice()` is a separate permissionless call with no on-chain enforcement that it must precede deposits.
- Staking rewards and LST appreciation accrue continuously, so the stale window opens naturally between keeper calls.
- The `pricePercentageLimit` guard can block permissionless updates during high-reward periods, widening the window.
- Any unprivileged depositor can observe the on-chain `rsETHPrice` vs. the live `_getTotalEthInProtocol()` value and time their deposit accordingly.

### Recommendation
Force a price refresh before computing the mint amount. Either:
1. Call `lrtOracle.updateRSETHPrice()` at the start of `depositETH` / `depositAsset`, or
2. Replace `lrtOracle.rsETHPrice()` in `getRsETHAmountToMint` with a freshly computed value derived from `_getTotalEthInProtocol() / rsethSupply` (i.e., expose a view-only price computation that does not write state).

### Proof of Concept
1. Staking rewards accrue over several hours; `_getTotalEthInProtocol()` grows, making the true rsETH price 1.001 ETH while `rsETHPrice` is still stored as 1.000 ETH.
2. `updateRSETHPrice()` has not been called (or is blocked by `pricePercentageLimit` for non-managers).
3. Attacker calls `depositETH(minRSETHAmountExpected, "")` with 100 ETH.
4. `getRsETHAmountToMint` computes: `(100e18 * 1e18) / 1.000e18 = 100 rsETH` instead of the correct `(100e18 * 1e18) / 1.001e18 ≈ 99.9 rsETH`.
5. Attacker receives ~0.1 rsETH excess, extracted from existing holders' accrued yield.
6. When `updateRSETHPrice()` is eventually called, the inflated supply lowers the price for all holders.
