### Title
Stale `rsETHPrice` Used in Deposit Minting Allows Yield Theft from Existing rsETH Holders - (File: `contracts/LRTDepositPool.sol`, `contracts/LRTOracle.sol`)

---

### Summary

`LRTDepositPool.depositETH()` and `depositAsset()` compute the rsETH mint amount using a **stored, potentially stale** `rsETHPrice` from `LRTOracle` without first refreshing it. When staking rewards have accrued and the true rsETH/ETH rate has risen above the stored value, an attacker can deposit at the outdated (lower) price, receive more rsETH than their deposit warrants, and then trigger a price update. The attacker's surplus rsETH captures yield that rightfully belongs to existing holders.

---

### Finding Description

`LRTDepositPool.getRsETHAmountToMint()` computes the mint amount as:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.getAssetPrice(asset)` is a live Chainlink-backed price. `lrtOracle.rsETHPrice()` is a **stored state variable** that is only updated when `updateRSETHPrice()` is explicitly called.

`updateRSETHPrice()` is public and permissionless, but it is **never called atomically inside `depositETH()` or `depositAsset()`**. There is no freshness check or forced update before the mint calculation.

`_updateRsETHPrice()` computes the correct price as:

```solidity
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

Between two calls to `updateRSETHPrice()`, staking rewards (EigenLayer restaking yield, LST rebases, ETH validator rewards) increase `totalETHInProtocol` while `rsethSupply` stays constant, so the true price rises above the stored `rsETHPrice`. Any deposit made in this window uses the stale lower price, minting more rsETH than the depositor's ETH value justifies.

---

### Impact Explanation

**High — Theft of unclaimed yield.**

Existing rsETH holders accumulate yield as `totalETHInProtocol` grows. That yield is only crystallised into the price when `updateRSETHPrice()` is called. A depositor who acts before that call receives rsETH priced at the old (lower) rate, effectively claiming a share of the unrealised yield. After the price update, the attacker's surplus rsETH represents ETH value extracted from existing holders. The theft scales linearly with deposit size; a flash-loan-sized deposit can capture nearly all accrued-but-unrecorded rewards in a single block.

---

### Likelihood Explanation

**Medium.** `updateRSETHPrice()` is not called on every block; it is triggered off-chain (keeper, bot, or manual call). Any gap between reward accrual and the next price update is an exploitable window. The window is predictable: an attacker can monitor `totalETHInProtocol` vs. `rsETHPrice * rsethSupply` on-chain and act the moment a profitable gap opens. No privileged access is required; `depositETH()` is fully public.

---

### Recommendation

Call `updateRSETHPrice()` (or an equivalent internal price refresh) at the beginning of `depositETH()` and `depositAsset()` before computing `rsethAmountToMint`. This mirrors the fix applied in the DyadStablecoin report (rebase modifier on all value-transfer functions). Alternatively, compute the mint amount using a freshly derived price rather than the stored `rsETHPrice`.

---

### Proof of Concept

```
State before attack:
  totalETHInProtocol = 1100 ETH   (100 ETH staking rewards accrued, not yet reflected)
  rsethSupply        = 1000 rsETH
  rsETHPrice (stored)= 1.000 ETH  (stale — correct value is 1.100 ETH)

Step 1 — Attacker calls depositETH{value: 10_000 ETH}():
  rsethAmountToMint = 10_000 * 1e18 / 1.000e18 = 10_000 rsETH
  (correct amount at true price: 10_000 / 1.100 = 9_090.9 rsETH)
  Surplus minted to attacker: 909.1 rsETH

Step 2 — Anyone calls updateRSETHPrice():
  totalETHInProtocol = 11_100 ETH
  rsethSupply        = 11_000 rsETH
  newRsETHPrice      = 11_100 / 11_000 = 1.00909 ETH

Step 3 — Attacker's position:
  10_000 rsETH × 1.00909 ETH = 10_090.9 ETH
  Deposited: 10_000 ETH
  Profit:       90.9 ETH  (≈ 90.9% of the 100 ETH accrued reward stolen)

Existing holders' loss:
  1000 rsETH × 1.00909 ETH = 1_009.09 ETH
  Should have been: 1000 × 1.100 = 1_100 ETH
  Loss: 90.9 ETH
```

**Entry path:**
1. Monitor `_getTotalEthInProtocol()` vs. `rsETHPrice * rsethSupply` for a profitable gap.
2. Call `LRTDepositPool.depositETH{value: flashLoanAmount}(0, "")`.
3. Call `LRTOracle.updateRSETHPrice()`.
4. Withdraw or hold the surplus rsETH.
