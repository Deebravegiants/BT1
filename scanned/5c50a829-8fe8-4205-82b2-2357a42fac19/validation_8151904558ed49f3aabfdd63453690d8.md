### Title
`addLiquidity` Bypasses `pauseLevel` Guard, Allowing Fund Deposits Into a Paused Pool — (`metric-core/contracts/MetricOmmPool.sol`)

### Summary

The `MetricOmmPool` pause mechanism (`pauseLevel`) is enforced only on `swap` via the `whenNotPaused` modifier. `addLiquidity` and `removeLiquidity` carry no such guard. When the pool admin or protocol owner pauses the pool in response to an emergency, users can still deposit principal through `addLiquidity`, exposing those funds to whatever condition triggered the pause.

### Finding Description

`MetricOmmPool` stores a `pauseLevel` state variable (0 = active, 1 = admin-paused, 2 = protocol-paused). The `_checkNotPaused` helper reverts with `PoolPaused` when `pauseLevel != 0`. [1](#0-0) 

This check is wired into the `whenNotPaused` modifier: [2](#0-1) 

`swap` correctly applies the modifier: [3](#0-2) 

But `addLiquidity` does not: [4](#0-3) 

And `removeLiquidity` does not either: [5](#0-4) 

The factory's `protocolPausePool` and `pausePool` paths both call `setPause` on the pool, which correctly writes `pauseLevel`: [6](#0-5) [7](#0-6) 

The state is written correctly, but `addLiquidity` never reads it. This is the direct analog of the Rubicon `stopped` variable being set but never consulted by the market's offer/buy paths.

### Impact Explanation

When the pool is paused due to an emergency (accounting exploit, oracle manipulation, reentrancy vector), `addLiquidity` remains callable. Users who deposit tokens through `addLiquidity` during a pause:

1. Transfer real token principal into the pool via the swap-callback pattern.
2. Receive shares computed against potentially corrupted `binTotals` / `_binStates`.
3. Cannot recover their deposit if the underlying condition that triggered the pause has already corrupted pool accounting.

This constitutes a direct loss of user principal above Sherlock thresholds in any scenario where the pause was triggered by an accounting or state-corruption bug rather than a pure oracle staleness event.

### Likelihood Explanation

The pool admin (`pausePool`) and protocol owner (`protocolPausePool`) are expected to pause pools in emergencies. The pause mechanism is explicitly documented and tested. Any user who calls `addLiquidity` after observing a pause event (or before they notice it) will deposit funds into a pool that the admin has declared unsafe. The trigger is a valid, non-privileged user action against a pool in a reachable paused state.

### Recommendation

Add `whenNotPaused` to `addLiquidity`:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
```

`removeLiquidity` intentionally omitting the guard is defensible (users must always be able to exit), but `addLiquidity` depositing new principal into a paused pool has no analogous justification.

### Proof of Concept

1. Pool admin calls `factory.pausePool(pool)` → `pauseLevel` is set to `1`.
2. Pool is paused due to an accounting exploit in progress.
3. User calls `pool.addLiquidity(...)` — no revert, because `addLiquidity` has no `whenNotPaused` guard.
4. `LiquidityLib.addLiquidity` updates `_binStates` and `_binTotalShares` against corrupted state, then pulls tokens from the user via callback.
5. User's tokens are now inside the compromised pool; shares are computed against corrupted bin balances.
6. User cannot recover principal equivalent to what was deposited.
