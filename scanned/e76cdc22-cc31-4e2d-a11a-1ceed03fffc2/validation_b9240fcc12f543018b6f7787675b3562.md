### Title
Zero-Fee Deposit Enables Low-Cost DoS of L2 Pool Deposit Function - (File: `contracts/pools/RSETHPoolNoWrapper.sol`)

---

### Summary

When `feeBps` is set to 0 in `RSETHPoolNoWrapper`, an attacker can deposit enough ETH to exhaust the pool's entire pre-loaded rsETH balance at near-zero cost (only gas), permanently blocking all subsequent user deposits until the `BRIDGER_ROLE` manually refills the pool.

---

### Finding Description

`RSETHPoolNoWrapper` is an L2 deposit pool that holds a finite pre-loaded balance of rsETH tokens. When a user calls `deposit()`, the pool transfers rsETH from its own balance to the depositor in exchange for ETH.

The fee calculation in `viewSwapRsETHAmountAndFee` is:

```solidity
// contracts/pools/RSETHPoolNoWrapper.sol
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;          // = 0 when feeBps == 0
    uint256 amountAfterFee = amount - fee;   // = amount (full amount)
    uint256 rsETHToETHrate = getRate();
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
``` [1](#0-0) 

When `feeBps == 0`, the attacker receives the full rsETH equivalent of their ETH with zero fee deducted. The `setFeeBps` setter has no lower-bound guard — only an upper-bound check — meaning 0 is a valid and settable value:

```solidity
function setFeeBps(uint256 _feeBps) external onlyRole(TIMELOCK_ROLE) {
    if (_feeBps > 10_000) revert InvalidFeeAmount();  // no lower bound
    feeBps = _feeBps;
    emit FeeBpsSet(_feeBps);
}
``` [2](#0-1) 

The `deposit()` function unconditionally calls `rsETH.safeTransfer(msg.sender, rsETHAmount)` from the pool's own rsETH balance:

```solidity
function deposit(string memory referralId) external payable nonReentrant whenNotPaused {
    if (!isEthDepositEnabled) revert EthDepositDisabled();
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    rsETH.safeTransfer(msg.sender, rsETHAmount);  // reverts if pool balance == 0
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
``` [3](#0-2) 

There is no minimum deposit amount enforced (only `amount != 0`), and no cap on how much rsETH a single depositor can receive in one call. Once the pool's rsETH balance is drained, every subsequent `deposit()` call reverts on the `safeTransfer`.

The attacker's ETH is not lost — it remains in the pool awaiting bridging to L1, and the attacker holds rsETH of equivalent value. The attacker can later redeem rsETH through the normal withdrawal process. The net cost of the attack is **gas only**.

---

### Impact Explanation

**Temporary freezing of funds / Contract fails to deliver promised returns.**

All L2 users are blocked from depositing ETH into the pool and receiving rsETH until the `BRIDGER_ROLE` manually refills the pool's rsETH balance. This is a core protocol function — the entire purpose of the L2 pool is to let users swap ETH for rsETH. The DoS persists for an indeterminate period (until operator intervention), not just a single block.

---

### Likelihood Explanation

The protocol may legitimately set `feeBps = 0` to bootstrap adoption or run promotional campaigns (exactly the scenario described in the reference report). Once `feeBps = 0`, any attacker with sufficient ETH capital (which is preserved as rsETH — no loss) can execute this attack. The attacker needs no special permissions and the entry path is the public `deposit()` function. [2](#0-1) 

---

### Recommendation

1. **Enforce a minimum non-zero fee when `feeBps` is set**, or add a per-depositor cap on the maximum rsETH receivable in a single transaction, so exhausting the pool's balance requires prohibitive capital.
2. **Add a minimum deposit amount** (analogous to `minAmountToDeposit` in `LRTDepositPool`) to raise the cost of spam deposits.
3. **Alternatively**, do not allow `feeBps` to be set to exactly 0; require a minimum floor (e.g., 1 bps) so that any deposit+hold cycle carries a non-trivial cost. [4](#0-3) 

---

### Proof of Concept

**Setup:** Protocol sets `feeBps = 0` to encourage adoption. Pool holds `P` rsETH tokens.

1. **Attacker** calls `deposit{value: P * rsETHToETHrate / 1e18}("")` — depositing exactly enough ETH to receive all `P` rsETH from the pool.
2. Fee calculation: `fee = amount * 0 / 10_000 = 0`. Attacker receives full `P` rsETH. Pool rsETH balance = 0.
3. **Alice** (legitimate user) calls `deposit{value: 1 ether}("")`. The call reverts because `rsETH.safeTransfer(alice, rsETHAmount)` fails — pool has no rsETH.
4. Attacker holds rsETH (no loss). Their ETH sits in the pool awaiting bridging. The DoS persists until `BRIDGER_ROLE` refills the pool.
5. Net attacker cost: gas only. [3](#0-2) [1](#0-0)
