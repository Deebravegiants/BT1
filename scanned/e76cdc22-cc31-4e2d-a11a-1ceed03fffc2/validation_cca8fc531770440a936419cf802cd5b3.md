### Title
No Minimum Output Check on L2 Pool `deposit()` Functions Exposes Depositors to Oracle-Rate Slippage - (File: contracts/pools/RSETHPoolV3.sol, RSETHPool.sol, RSETHPoolNoWrapper.sol, RSETHPoolV2.sol, RSETHPoolV2ExternalBridge.sol, RSETHPoolV3ExternalBridge.sol, RSETHPoolV3WithNativeChainBridge.sol)

---

### Summary

Every L2 pool `deposit()` function computes the rsETH output amount at execution time using the live oracle rate, but accepts no `minRsETHAmountExpected` parameter. A depositor who previews the output via `viewSwapRsETHAmountAndFee()` and then submits a transaction can receive materially fewer rsETH than expected if the oracle rate is updated before their transaction executes. The L1 `LRTDepositPool` already has this protection; the L2 pools do not.

---

### Finding Description

All L2 pool contracts expose `deposit()` functions that:

1. Accept ETH or a supported token from the caller.
2. Compute `rsETHAmount` by calling `viewSwapRsETHAmountAndFee()`, which reads `getRate()` from the oracle at execution time.
3. Transfer or mint that `rsETHAmount` to the caller with no floor check.

### Impact Explanation

**Low. Contract fails to deliver promised returns, but doesn't lose value.**

A depositor calls `viewSwapRsETHAmountAndFee(amount)` off-chain, sees they will receive `X` rsETH, and submits the transaction. If the oracle rate is updated upward in the same block (or a preceding block on a sequencer chain) before their transaction is included, the actual `rsETHAmount` minted/transferred is less than `X`. The depositor's ETH is consumed but they receive fewer rsETH than they planned for. Because rsETH is a yield-bearing token whose rate only increases, this is a one-directional risk: depositors always risk receiving fewer rsETH than previewed, never more.

---

### Likelihood Explanation

The rsETH oracle rate is updated periodically (at least daily on L2 chains). On L2 sequencer chains (Arbitrum, Optimism, Base, etc.) the sequencer controls transaction ordering, making it straightforward for an oracle update transaction to be ordered ahead of a pending deposit. Any depositor who previews the output and then submits a transaction is exposed. The risk is routine and requires no special attacker capability — it occurs naturally whenever an oracle update and a deposit land in the same block.

---

### Recommendation

Add a `minRsETHAmountExpected` parameter to every L2 pool `deposit()` function, mirroring the existing protection in `LRTDepositPool`:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountExpected)
    external payable nonReentrant whenNotPaused
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    ...
}
```

Apply the same pattern to the token-deposit overload and to all pool variants (`RSETHPool`, `RSETHPoolNoWrapper`, `RSETHPoolV2`, `RSETHPoolV2ExternalBridge`, `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`).

---

### Proof of Concept

1. Alice calls `RSETHPoolV3.viewSwapRsETHAmountAndFee(1 ether)` off-chain. Oracle rate is `1.05e18` (1 rsETH = 1.05 ETH). She expects `(1e18 * 1e18) / 1.05e18 ≈ 0.952e18` rsETH.
2. Alice submits `deposit{value: 1 ether}("ref")`.
3. Before Alice's tx is included, the oracle rate is updated to `1.10e18`.
4. Alice's tx executes: `rsETHAmount = (1e18 * 1e18) / 1.10e18 ≈ 0.909e18` rsETH — ~4.5% less than previewed.
5. Alice has no recourse; the contract accepted her ETH and minted the lower amount with no revert.

The root cause is the absence of a minimum-output guard in `viewSwapRsETHAmountAndFee` / `deposit`, confirmed at: [10](#0-9)
