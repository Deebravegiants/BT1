### Title
User Cannot Cancel Pending Withdrawal Request, Causing Indefinite Locking of rsETH - (File: contracts/LRTWithdrawalManager.sol)

### Summary
`LRTWithdrawalManager.initiateWithdrawal()` transfers the user's rsETH into the contract and creates a locked withdrawal request, but no cancellation function exists. The user is permanently committed to the queue and cannot recover their rsETH until the operator calls `unlockQueue()` and the withdrawal delay passes. This is the direct analog of the SKALE issue: one party can enter a two-party relationship (user initiates, operator must process) but cannot exit it unilaterally.

### Finding Description
When a user calls `initiateWithdrawal()`, their rsETH is immediately pulled from them and held by the contract:

```solidity
IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);
```

The request is then stored in `withdrawalRequests` and the user's nonce is pushed into `userAssociatedNonces`. The only path to recover value is:

1. Operator calls `unlockQueue()` (restricted to `onlyAssetTransferOrOperatorRole`)
2. `withdrawalDelayBlocks` pass
3. User calls `completeWithdrawal()`

There is no `cancelWithdrawal()` or equivalent. The user has no unilateral exit. The asymmetry is identical to the SKALE pattern: the user can enter the queue (analogous to a delegator requesting delegation), but the counterparty (operator, analogous to the validator) holds exclusive control over processing, and the initiating party cannot undo their commitment.

Additionally, `unlockQueue` carries `onlySupportedAsset(asset)`:

```solidity
function unlockQueue(
    address asset,
    ...
)
    external
    nonReentrant
    onlySupportedAsset(asset)
    whenNotPaused
    onlyAssetTransferOrOperatorRole
```

If an asset is delisted from the supported set after a user has initiated a withdrawal for it, `unlockQueue` becomes permanently uncallable for that asset, and the user's rsETH is frozen with no recovery path.

### Impact Explanation
**Medium — Temporary freezing of funds** (escalating to Critical permanent freeze in the asset-delisting edge case).

Under normal operation the freeze is temporary: the user's rsETH is locked from `initiateWithdrawal()` until the operator processes the queue and the delay passes. If the operator halts (protocol deprecated, operational failure, or the asset is removed from the supported list), the rsETH held in the contract becomes unrecoverable because:
- `completeWithdrawal` requires the request to be unlocked (`nextLockedNonce` check)
- `unlockQueue` is gated on `onlySupportedAsset` and operator role
- No cancel path exists

### Likelihood Explanation
**Low-to-Medium.** The operator is expected to run continuously, so the common case is a bounded delay. However, the no-cancel design is reachable by any depositor who calls `initiateWithdrawal()` and then faces a protocol pause, asset delisting, or operator downtime. No special privilege or exploit is required to enter this state; the user triggers it themselves by calling a public function.

### Recommendation
Add a `cancelWithdrawal(address asset)` function that:
1. Pops the user's oldest (or a specified) pending nonce from `userAssociatedNonces`
2. Deletes the corresponding `withdrawalRequests` entry
3. Decrements `assetsCommitted[asset]`
4. Returns the locked rsETH to the user

Restrict cancellation to requests that are still in the locked (not yet unlocked) state to avoid double-spend with already-processed requests.

### Proof of Concept
1. Alice calls `initiateWithdrawal(ETH, 1e18, "")`. Her 1 rsETH is transferred to `LRTWithdrawalManager`; `assetsCommitted[ETH]` increases.
2. The operator role is revoked or the protocol is paused indefinitely (or the ETH asset is removed from the supported list).
3. Alice attempts to call `completeWithdrawal(ETH, "")` — reverts with `WithdrawalLocked` because `nextLockedNonce` has not advanced.
4. Alice searches for a cancel function — none exists in `LRTWithdrawalManager`.
5. Alice's 1 rsETH remains locked in the contract with no recovery path.

```text
    function _processWithdrawalCompletion(address asset, address user, string calldata referralId) internal {
        if (userAssociatedNonces[asset][user].empty()) {
            revert NoWithdrawalRequests(user, asset);
        }

        // Retrieve and remove the oldest withdrawal request for the user.
        uint256 usersFirstWithdrawalRequestNonce = userAssociatedNonces[asset][user].popFront();
        // Ensure the request is already unlocked.
        if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();

        bytes32 requestId = getRequestId(asset, usersFirstWithdrawalRequestNonce);
        WithdrawalRequest memory request = withdrawalRequests[requestId];

        delete withdrawalRequests[requestId];

        // Check that the withdrawal delay has passed since the request's initiation.
        if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();
```

```text
    function _addUserWithdrawalRequest(address asset, uint256 rsETHUnstaked, uint256 expectedAssetAmount) internal {
        uint256 nextUnusedNonce_ = nextUnusedNonce[asset];

        // Generate a unique identifier for the new withdrawal request.
        bytes32 requestId = getRequestId(asset, nextUnusedNonce_);

        // Create and store the new withdrawal request.
        withdrawalRequests[requestId] = WithdrawalRequest({
            rsETHUnstaked: rsETHUnstaked, expectedAssetAmount: expectedAssetAmount, withdrawalStartBlock: block.number
        });

        // Map the user to the newly created request index and increment the nonce for future requests.
        userAssociatedNonces[asset][msg.sender].pushBack(nextUnusedNonce_);
        nextUnusedNonce[asset] = nextUnusedNonce_ + 1;

        emit AssetWithdrawalQueued(msg.sender, asset, rsETHUnstaked, nextUnusedNonce_);
    }
```
