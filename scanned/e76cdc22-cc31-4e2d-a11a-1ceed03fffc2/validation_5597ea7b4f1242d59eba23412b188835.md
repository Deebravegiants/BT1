### Title
Missing Zero-Value Validation in `updateAssetDepositLimit` Freezes All New Deposits - (File: contracts/LRTConfig.sol)

### Summary
`LRTConfig.updateAssetDepositLimit` accepts `depositLimit = 0` without validation, while the initial asset-addition path explicitly rejects it. An accidental zero-value call by the MANAGER role would silently block all new deposits for the affected asset.

### Finding Description
`_addNewSupportedAsset` enforces a non-zero deposit limit at asset registration time:

```solidity
// LRTConfig.sol lines 106-110
function _addNewSupportedAsset(address asset, uint256 depositLimit) private {
    UtilLib.checkNonZeroAddress(asset);
    if (depositLimit == 0) {
        revert InvalidDepositLimit();
    }
```

However, the post-deployment update path `updateAssetDepositLimit` has no equivalent guard:

```solidity
// LRTConfig.sol lines 123-133
function updateAssetDepositLimit(
    address asset,
    uint256 depositLimit
)
    external
    onlyRole(LRTConstants.MANAGER)
    onlySupportedAsset(asset)
{
    depositLimitByAsset[asset] = depositLimit;
    emit AssetDepositLimitUpdate(asset, depositLimit);
}
```

When `depositLimitByAsset[asset]` is set to 0, `LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` evaluates:

- **ETH path** (`LRTDepositPool.sol` line 679): `return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset))` → `totalAssetDeposits > 0` is true whenever any ETH has ever been deposited, so every subsequent `depositETH` call reverts with `MaximumDepositLimitReached`.
- **ERC-20 path** (`LRTDepositPool.sol` line 681): `return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset))` → `totalAssetDeposits + amount > 0` is true for any non-zero deposit, blocking all ERC-20 deposits for that asset.

### Impact Explanation
Setting `depositLimit` to 0 for ETH or any supported LST (stETH, ETHx) immediately prevents all new deposits for that asset via `depositETH` / `depositAsset`. Existing user balances and withdrawal flows are unaffected, but the protocol stops accepting new capital. The misconfiguration is reversible only by a subsequent correct MANAGER call. Impact: **Low — contract fails to deliver promised returns, but does not lose value.**

### Likelihood Explanation
Low. The MANAGER role is trusted but the inconsistency between `_addNewSupportedAsset` (which rejects zero) and `updateAssetDepositLimit` (which does not) creates a latent trap. A scripting error, a UI that defaults to 0, or a copy-paste mistake during a routine limit adjustment could trigger this silently — the transaction succeeds and emits a normal `AssetDepositLimitUpdate` event, giving no immediate on-chain signal that deposits are now frozen.

### Recommendation
Mirror the guard already present in `_addNewSupportedAsset`:

```solidity
function updateAssetDepositLimit(
    address asset,
    uint256 depositLimit
)
    external
    onlyRole(LRTConstants.MANAGER)
    onlySupportedAsset(asset)
{
    if (depositLimit == 0) revert InvalidDepositLimit();
    depositLimitByAsset[asset] = depositLimit;
    emit AssetDepositLimitUpdate(asset, depositLimit);
}
```

Additionally, add a lower-bound check in `LRTWithdrawalManager.setWithdrawalDelayBlocks` to prevent accidentally setting the delay to 0, which would allow users to complete withdrawals in the same block they are initiated.

### Proof of Concept
1. MANAGER calls `LRTConfig.updateAssetDepositLimit(0xEeeeeEeeeEeEeeEeEeEeeEEEeeeeEeeeeeeeEEeE, 0)` — transaction succeeds.
2. `depositLimitByAsset[ETH_TOKEN]` is now `0`.
3. Any depositor calls `LRTDepositPool.depositETH{value: 1 ether}(0, "")`.
4. `_beforeDeposit` → `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 1 ether)` → `totalAssetDeposits > 0` → `true`.
5. `depositETH` reverts with `MaximumDepositLimitReached`.
