### Title
Immutable `i_maxSupply` Cap in `WrappedRSETH` Can Be Exhausted via L2 Pool Deposits, Blocking CCIP Bridge Mints and Temporarily Freezing Bridged Funds â€” (`contracts/ccip/WrappedRSETH.sol`)

---

### Summary

`WrappedRSETH.mint()` enforces an immutable supply cap (`i_maxSupply`) that is shared across all minters â€” both the L2 pool contracts and the CCIP bridge pool. An attacker can exhaust the cap via permissionless `deposit()` calls on the L2 pool, causing subsequent CCIP bridge mint attempts to revert with `MaxSupplyExceeded`, temporarily freezing in-flight bridged rsETH in the CCIP escrow.

---

### Finding Description

`WrappedRSETH` stores a supply ceiling in an `immutable` variable: [1](#0-0) 

The `mint()` function enforces it unconditionally for every caller that holds the minter role: [2](#0-1) 

Because `i_maxSupply` is `immutable`, the owner has no mechanism to raise the ceiling after deployment.

The L2 pool contracts â€” `RSETHPoolV2`, `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge` â€” are all granted the minter role and call `wrsETH.mint()` directly on every user `deposit()`: [3](#0-2) [4](#0-3) 

The CCIP bridge pool is a separate minter that also calls `wrsETH.mint()` when delivering a cross-chain message. Both minter classes draw from the same shared `i_maxSupply` budget with no per-minter allocation.

**Concrete attack path:**

1. Attacker calls `RSETHPoolV2.deposit{value: X}("")` repeatedly (subject to `limitDailyMint` per day, but executable over multiple days) until `totalSupply()` approaches `i_maxSupply`.
2. A legitimate user locks rsETH on L1 via the CCIP bridge; the CCIP OffRamp delivers the message on L2 and calls `wrsETH.mint(user, bridgedAmount)`.
3. The check `totalSupply() + bridgedAmount > i_maxSupply` is true â†’ `MaxSupplyExceeded` is thrown â†’ the CCIP execution reverts.
4. The user's rsETH remains locked in the L1 CCIP escrow. The CCIP message enters a failed state and can only be retried â€” but every retry will revert for as long as the attacker holds their wrsETH and keeps the supply at the cap.

The attacker retains full economic value (their wrsETH is redeemable for ETH via the pool), so there is no cost pressure to release the supply.

---

### Impact Explanation

**Medium â€” Temporary freezing of funds.**

The question claims "Critical: Permanent freezing," but this is overstated:

- CCIP failed messages can be manually re-executed via `OffRamp.manuallyExecute()` once supply headroom is restored.
- Supply headroom can be restored if any wrsETH holder burns tokens (e.g., via pool withdrawal).
- The attacker holds valuable wrsETH and has no incentive to burn it, so the freeze can be sustained indefinitely in practice â€” but it is not cryptographically permanent.

The correct scoped impact is **Medium: Temporary freezing of funds** (in-flight CCIP bridged rsETH is inaccessible for an attacker-controlled duration).

---

### Likelihood Explanation

Likelihood is **Low-to-Medium**:

- Requires `i_maxSupply` to be set to a finite value at deployment (the contract explicitly supports this; `0` means unlimited).
- The `limitDailyMint` modifier on pool contracts slows supply exhaustion but does not prevent it over multiple days.
- The attacker must commit real ETH capital (no direct profit motive), making this a griefing attack with sustained capital lockup cost.

---

### Recommendation

1. **Set `i_maxSupply = 0` (unlimited) at deployment** if a hard cap is not a protocol requirement. The check `if (i_maxSupply != 0 && ...)` already handles this safely.
2. If a cap is required, **replace `immutable` with a governance-controlled setter** (owner-only, with a timelock) so the ceiling can be raised without redeploying the token.
3. **Allocate the cap per minter class** (e.g., a separate sub-limit for the CCIP pool vs. the L2 swap pool) so pool deposits cannot crowd out bridge mints.

// Pseudocode â€” run on a local fork with i_maxSupply = 1000 ether worth of wrsETH

function testMaxSupplyGriefAttack() public {
    // 1. Deploy WrappedRSETH with a finite cap
    WrappedRSETH wrsETH = new WrappedRSETH("wrsETH", "wrsETH", 18, 1000e18, owner);

    // 2. Grant minter role to both pool and mock CCIP pool
    vm.prank(owner);
    wrsETH.grantMintRole(address(pool));
    vm.prank(owner);
    wrsETH.grantMintRole(address(ccipPool));

    // 3. Attacker fills supply via pool deposits over N days
    //    (limitDailyMint is respected; loop advances block.timestamp by 1 day each iteration)
    for (uint i = 0; i < N_DAYS; i++) {
        vm.deal(attacker, dailyLimit);
        vm.prank(attacker);
        pool.deposit{value: dailyLimit}("");
        vm.warp(block.timestamp + 1 days);
    }

    // 4. Assert supply is at or near cap
    assertGe(wrsETH.totalSupply(), wrsETH.maxSupply() - smallDust);

    // 5. Simulate CCIP bridge mint for a legitimate user
    vm.prank(address(ccipPool));
    vm.expectRevert(WrappedRSETH.MaxSupplyExceeded.selector);
    wrsETH.mint(victim, 1e18); // victim's bridged rsETH is now stuck on L1
