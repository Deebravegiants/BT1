### Title
Missing Slippage Protection in L2 Pool `deposit()` Functions Allows Users to Receive Fewer rsETH Than Previewed - (File: contracts/pools/RSETHPool.sol, RSETHPoolV2.sol, RSETHPoolV3.sol, RSETHPoolNoWrapper.sol, RSETHPoolV2ExternalBridge.sol, RSETHPoolV2NBA.sol, RSETHPoolV3ExternalBridge.sol, RSETHPoolV3WithNativeChainBridge.sol)

### Summary

All L2 `RSETHPool` variants expose `deposit()` functions that compute the rsETH output amount using the live oracle rate (`getRate()`) at execution time, but accept no `minRsETHAmount` parameter. Users who preview the rate via `viewSwapRsETHAmountAndFee()` before submitting a transaction have no on-chain guarantee that the rate will not have moved by the time their transaction is mined, causing them to silently receive fewer rsETH tokens than expected.

### Finding Description

Every L2 pool contract follows the same pattern. In `RSETHPool.sol`:

```solidity
function deposit(string memory referralId) external payable nonReentrant whenNotPaused {
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
``` [1](#0-0) 

The rsETH amount is computed inside `viewSwapRsETHAmountAndFee()` as:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

where `rsETHToETHrate = getRate()` reads the live oracle rate at execution time. [2](#0-1) 

The oracle rate is sourced from `rsETHOracle.getRate()`, which on L2 chains reflects the rsETH/ETH price bridged from L1 via `RSETHRateProvider` / `InterimRSETHOracle`. The L1 price is updated by the publicly callable `LRTOracle.updateRSETHPrice()`. [3](#0-2) 

The same pattern is present in every pool variant:

- `RSETHPoolV2.deposit()` [4](#0-3) 
- `RSETHPoolV3.deposit()` [5](#0-4) 
- `RSETHPoolNoWrapper.deposit()` [6](#0-5) 

None of these accept a `minRsETHAmount` parameter. By contrast, the L1 `LRTDepositPool` correctly enforces slippage protection:

```solidity
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [7](#0-6) 

This protection is entirely absent from all L2 pool contracts.

### Impact Explanation

**Low ‚Äî Contract fails to deliver promised returns, but doesn't lose value.**

A user calls `viewSwapRsETHAmountAndFee(1 ether)` and sees they will receive, e.g., `0.9524 wrsETH` at a rate of `1.05e18`. Between that preview and the transaction being mined, the oracle rate is updated to `1.10e18` (a legitimate reward accrual update). The user's transaction executes and they receive only `0.9090 wrsETH` ‚Äî approximately 4.5% fewer tokens than previewed ‚Äî with no revert and no recourse. The ETH value is approximately preserved (each wrsETH is worth more), but the user receives fewer tokens than the interface promised, which can break downstream assumptions (e.g., minimum collateral thresholds, LP positions, or automated strategies).

### Likelihood Explanation

The rsETH/ETH rate increases monotonically over time as staking rewards accrue. `LRTOracle.updateRSETHPrice()` is publicly callable by anyone, meaning any actor can trigger a rate update at any time. On L2 chains with cross-chain oracle bridging, rate updates arrive in discrete jumps. A user who previews the rate and submits a transaction during a period of pending oracle update will silently receive fewer tokens. This is a routine operational condition, not a contrived attack.

### Recommendation

Add a `minRsETHAmount` parameter to all `deposit()` functions in every L2 pool contract, mirroring the protection already present in `LRTDepositPool`:

```diff
- function deposit(string memory referralId) external payable nonReentrant whenNotPaused {
+ function deposit(string memory referralId, uint256 minRsETHAmount) external payable nonReentrant whenNotPaused {
      ...
      (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
+     if (rsETHAmount < minRsETHAmount) revert SlippageExceeded();
      ...
  }
```

Apply the same fix to the token-deposit overloads in `RSETHPool`, `RSETHPoolV3`, `RSETHPoolNoWrapper`, `RSETHPoolV3ExternalBridge`, and `RSETHPoolV3WithNativeChainBridge`.

### Proof of Concept

1. User calls `viewSwapRsETHAmountAndFee(1 ether)` on `RSETHPool`. Oracle rate is `1.05e18`. Preview returns `rsETHAmount = 1e18 * 1e18 / 1.05e18 ‚âà 0.9524e18`.
2. User submits `deposit{value: 1 ether}("ref")`.
3. Before the transaction is mined, anyone calls `LRTOracle.updateRSETHPrice()` on L1, which increases the rate to `1.10e18` and bridges the update to L2.
4. User's transaction executes. `getRate()` now returns `1.10e18`. Actual `rsETHAmount = 1e18 * 1e18 / 1.10e18 ‚âà 0.9090e18`.
5. User receives `~0.9090 wrsETH` instead of the previewed `~0.9524 wrsETH` ‚Äî a ~4.5% shortfall ‚Äî with no revert. [2](#0-1) [3](#0-2)
