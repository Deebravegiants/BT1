### Title
`initiateWithdrawal()` Always Reverts Due to Uninitialized `maxNumberOfWithdrawalsPerUser` — (File: contracts/KERNEL/KernelDepositPool.sol)

---

### Summary

The `KernelDepositPool.sol` contract's `initiateWithdrawal()` function unconditionally reverts with `WithdrawalLimitReached` immediately after deployment because `maxNumberOfWithdrawalsPerUser` is never set in `initialize()`, leaving it at its Solidity default of `0`. Since a `uint256` length is always `>= 0`, the guard fires on every call, permanently blocking all stakers from unstaking until an admin manually configures the parameter.

---

### Finding Description

`initialize()` sets `kernelToken`, `rewardsToken`, and roles, but never assigns `maxNumberOfWithdrawalsPerUser`: [1](#0-0) 

As a result, `maxNumberOfWithdrawalsPerUser` remains `0` after deployment.

`initiateWithdrawal()` contains the following guard: [2](#0-1) 

Because `userWithdrawalIds[msg.sender].length` is a `uint256` (always `>= 0`) and `maxNumberOfWithdrawalsPerUser == 0`, the condition `length >= 0` is **always true**. Every call to `initiateWithdrawal()` reverts with `WithdrawalLimitReached` regardless of how many withdrawals the user actually has open.

The setter explicitly forbids restoring the zero value, so the only escape is an admin action: [3](#0-2) 

Until `setMaxNumberOfWithdrawalsPerUser()` is called, the withdrawal path is entirely closed.

This is structurally analogous to the reference report: a validation check that is supposed to guard against an edge case (too many open withdrawals) is instead applied unconditionally due to an uninitialized state variable, blocking the entire operation rather than only the edge case it was meant to prevent.

---

### Impact Explanation

Any user who has called `stake()` or `stakeFor()` and wishes to recover their KERNEL tokens is blocked. Their tokens are held in the contract with no exit path available to them. The freeze lasts until an admin calls `setMaxNumberOfWithdrawalsPerUser()`. This constitutes a **temporary freezing of funds** for all stakers.

---

### Likelihood Explanation

The contract is deployable and usable (staking works) without the admin ever noticing the missing initialization step, because `stake()` has no dependency on `maxNumberOfWithdrawalsPerUser`. A user who stakes immediately after deployment and then tries to withdraw will be blocked. The missing initialization is not enforced or documented in `initialize()`, making it a realistic oversight.

---

### Recommendation

Initialize `maxNumberOfWithdrawalsPerUser` to a safe non-zero default inside `initialize()`, for example:

```solidity
maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER; // 100
```

This mirrors the pattern used for `withdrawalDelayBlocks` in `LRTWithdrawalManager.initialize()`: [4](#0-3) 

---

### Proof of Concept

1. Deploy `KernelDepositPool` (admin never calls `setMaxNumberOfWithdrawalsPerUser`).
2. Alice calls `stake(100e18)` — succeeds; `balanceOf[Alice] = 100e18`.
3. Alice calls `initiateWithdrawal(100e18)`.
4. Execution reaches:
   ```solidity
   if (userWithdrawalIds[Alice].length >= maxNumberOfWithdrawalsPerUser)
   //  0 >= 0  → true
       revert WithdrawalLimitReached();
   ```
5. Transaction reverts. Alice's 100 KERNEL tokens are locked in the contract with no recourse until an admin intervenes. [5](#0-4) [6](#0-5)

```text
    function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
        if (_amount == 0) revert AmountZero();
        if (balanceOf[msg.sender] < _amount) revert InsufficientStakedBalance();
        if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();

        balanceOf[msg.sender] -= _amount;
        totalKernelStaked -= _amount;

        // Create a withdrawal record
        uint256 withdrawalId = ++withdrawalCounter;
        uint256 unlockTime = block.timestamp + withdrawalDelay;

        withdrawals[withdrawalId] = Withdrawal({
            user: msg.sender, amount: _amount, unlockTime: unlockTime, claimed: false, withdrawalId: withdrawalId
        });
        userWithdrawalIds[msg.sender].push(withdrawalId);

        emit WithdrawalInitiated(msg.sender, _amount, withdrawalId, unlockTime);
    }
```
