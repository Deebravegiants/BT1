### Title
`claim()` and `claimAndStake()` in `KernelMerkleDistributor` Are Restricted to Self-Execution Only, Preventing Third-Party Claiming on Behalf of Users - (File: contracts/KERNEL/KernelMerkleDistributor.sol)

---

### Summary

`KernelMerkleDistributor.claim()` and `claimAndStake()` accept an `account` parameter — implying third-party execution on behalf of a beneficiary — but internally enforce `account == msg.sender`, making it impossible for any relayer, meta-transaction forwarder, or smart contract to claim on behalf of the rightful recipient.

---

### Finding Description

The `IMerkleDistributor` interface declares:

```solidity
function claim(uint256 index, address account, uint256 cumulativeAmount, bytes32[] calldata merkleProof) external;
```

The presence of a distinct `account` parameter (separate from `msg.sender`) signals that the function was designed to allow a third party to submit a valid merkle proof and trigger a claim for `account`. However, inside `_processClaim`, the implementation immediately rejects any caller who is not the account itself:

```solidity
if (account != msg.sender) {
    revert Unauthorized();
}
``` [1](#0-0) 

This check is applied identically for both `claim()` and `claimAndStake()`, since both delegate to `_processClaim`. [2](#0-1) 

The merkle leaf is constructed as:

```solidity
bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));
``` [3](#0-2) 

The proof is cryptographically bound to `account`, so there is no risk of fund theft by a third-party caller — the proof would simply fail for any other address. The `account != msg.sender` guard therefore provides no security benefit while actively breaking the third-party execution model implied by the interface.

---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

- Users who rely on gasless meta-transaction relayers (e.g., EIP-2771 forwarders) cannot claim their KERNEL tokens, because `msg.sender` would be the forwarder, not `account`.
- Smart contract addresses included in the merkle tree (e.g., multisigs that delegate execution, protocol-owned treasuries, or vesting contracts without a generic `execute` method) are permanently unable to claim, resulting in permanent freezing of their unclaimed KERNEL yield.
- The `claimAndStake` path is equally blocked, preventing any third-party "claim-and-stake-for" automation that the `stakeFor` design in `KernelDepositPool` was built to support. [4](#0-3) 

---

### Likelihood Explanation

**Medium.** The merkle distribution is a live reward mechanism targeting a broad set of KERNEL holders. Any recipient whose on-chain address is a smart contract without a direct `claim` call path (multisig, vesting contract, DAO treasury) is concretely affected. Relayer-based UX is common in DeFi and is explicitly supported by the `stakeFor` architecture in `KernelDepositPool`. [5](#0-4) 

---

### Recommendation

Remove the `account != msg.sender` guard from `_processClaim`. The merkle proof already cryptographically binds the claim to `account`; no additional caller restriction is needed. Any caller who supplies a valid proof for `account` can only direct tokens to `account`, so there is no theft vector.

```solidity
// Remove these lines from _processClaim:
// if (account != msg.sender) {
//     revert Unauthorized();
// }
```

This mirrors the fix applied in the referenced PoolTogether mitigation (replacing `msg.sender` with the intended beneficiary address), and aligns `KernelMerkleDistributor` with the third-party execution model already present in `KernelDepositPool.stakeFor`.

---

### Proof of Concept

1. Alice's address `0xAlice` is included in the merkle tree with a claimable amount.
2. Alice uses a Gnosis Safe as her primary wallet. The Safe itself is `0xAlice` in the tree.
3. Alice wants to use a relayer (`0xRelayer`) to submit the claim gaslessly.
4. Relayer calls `claim(index, 0xAlice, amount, proof)` — `msg.sender = 0xRelayer`, `account = 0xAlice`.
5. `_processClaim` hits `if (account != msg.sender)` → `revert Unauthorized()`.
6. Alice's KERNEL tokens remain permanently locked in the distributor, since the relayer path is the only viable execution route for her setup. [6](#0-5)

            revert InvalidIndex();
