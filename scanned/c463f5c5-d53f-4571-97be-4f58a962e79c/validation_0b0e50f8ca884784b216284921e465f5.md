### Title
Reward Token Precision Loss in `notifyRewardAmount` Permanently Locks Undistributed Rewards - (File: contracts/KERNEL/KernelDepositPool.sol)

---

### Summary
`KernelDepositPool.notifyRewardAmount` computes `rewardRate` via integer division, permanently locking `receivedAmount % duration` reward tokens in the contract on every reward epoch. There is no recovery path for these tokens, so stakers collectively receive less than the total rewards deposited.

---

### Finding Description
`notifyRewardAmount` determines the per-second reward rate using integer division:

```solidity
// line 580
rewardRate = receivedAmount / duration;
// or, when a period is still active (line 583):
rewardRate = (receivedAmount + remaining) / duration;
```

The Solidity integer division silently discards the remainder. The total rewards that will ever be distributed over the period is `rewardRate * duration`, which equals `receivedAmount - (receivedAmount % duration)`. The remainder `receivedAmount % duration` is transferred into the contract but never emitted to any staker.

There is no `recoverERC20`, `sweep`, or equivalent function anywhere in `KernelDepositPool` (confirmed by search). The contract comment at lines 17–22 acknowledges the Synthetix derivation and the `totalKernelStaked == 0` edge case, but does not address this truncation loss.

---

### Impact Explanation
Every call to `notifyRewardAmount` permanently locks up to `duration - 1` atomic units of the reward token. For a reward token with low decimals (e.g., 6-decimal USDC) and a 7-day duration (`604800` seconds), the maximum loss per epoch is `604799 / 1e6 ≈ 0.60 tokens`. Across many epochs this accumulates into a non-trivial sum of permanently frozen unclaimed yield. Stakers who call `getReward()` receive the full `rewardRate * elapsed` amount, but the aggregate shortfall relative to what was deposited is never recoverable. This matches the allowed impact: **Low — contract fails to deliver promised returns, but doesn't lose value** (the tokens remain in the contract, just inaccessible).

---

### Likelihood Explanation
The loss is deterministic and occurs on every single invocation of `notifyRewardAmount`. No special conditions are required beyond normal protocol operation. The admin calls this function to start each reward epoch, so the loss accumulates with every epoch.

---

### Recommendation
Carry the remainder forward into the next epoch by adding it to the next `receivedAmount` before computing the new rate, or track it in a dedicated state variable. Alternatively, add an admin-only `recoverERC20` function that can only withdraw tokens not accounted for by `rewardRate * (finishAt - updatedAt)`.

```solidity
// Example fix: carry remainder forward
uint256 leftover = rewardRate * (finishAt - block.timestamp); // remaining from old period
rewardRate = (receivedAmount + leftover) / duration;
// The remainder (receivedAmount + leftover) % duration is now bounded to < duration
// and can be recovered or rolled into the next notifyRewardAmount call
```

---

### Proof of Concept

**Setup:**
- `duration = 7 days = 604800 seconds`
- `rewardsToken` = a 6-decimal token (e.g., USDC)
- Admin calls `notifyRewardAmount(1_000_000)` (1.0 USDC)

**Execution:**
```
receivedAmount = 1_000_000
rewardRate     = 1_000_000 / 604800 = 1  (integer division)
distributed    = 1 * 604800 = 604800 units = 0.6048 USDC
locked forever = 1_000_000 - 604800 = 395200 units = 0.3952 USDC (~39.5% of the deposit)
```

After `finishAt`, all stakers call `getReward()`. They collectively receive `0.6048 USDC`. The remaining `0.3952 USDC` sits in the contract with no extraction path.

Over 52 weekly epochs with the same parameters, the permanently locked amount grows to `52 × 0.3952 ≈ 20.55 USDC` — entirely inaccessible to stakers or the protocol. [1](#0-0) [2](#0-1) [3](#0-2)
