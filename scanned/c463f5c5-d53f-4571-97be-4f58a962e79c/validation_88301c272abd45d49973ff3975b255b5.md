### Title
`updatedAt` advances when `totalKernelStaked == 0`, permanently freezing reward yield — (File: `contracts/KERNEL/KernelDepositPool.sol`)

---

### Summary

The `updateReward` modifier in `KernelDepositPool` unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` even when `totalKernelStaked == 0`. This mirrors the Beedle bug class exactly: a time-tracking variable is advanced without distributing rewards, causing all rewards accrued during the zero-supply window to be permanently stuck in the contract.

---

### Finding Description

The `updateReward` modifier runs before every state-changing function:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    updatedAt = lastTimeRewardApplicable();   // ← always advances
    ...
}
``` [1](#0-0) 

`rewardPerToken()` correctly short-circuits when `totalKernelStaked == 0`, returning the stored value unchanged:

```solidity
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;   // no new rewards distributed
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
``` [2](#0-1) 

However, `updatedAt` is still set to `lastTimeRewardApplicable()` in the modifier regardless. The next time a staker deposits and `rewardPerToken()` is called with `totalKernelStaked > 0`, the formula computes:

```
rewardRate * (lastTimeRewardApplicable() - updatedAt)
```

`updatedAt` already reflects the end of the zero-supply gap, so the rewards that accrued during that entire window are silently skipped and permanently locked in the contract.

The contract's own NatDoc acknowledges this behavior but relies on an off-chain assumption that is not enforced on-chain:

```
* @dev If `totalKernelStaked` ever hits zero during a reward distribution window, any remaining rewards
*      for that period will stay locked in the contract. In this deployment, we're avoiding this issue by
*      ensuring there are always some tokens staked before admin calls the `notifyRewardAmount` function,
*      as well as for the entire duration of the reward period.
``` [3](#0-2) 

The guard in `notifyRewardAmount` prevents starting a reward period with zero supply:

```solidity
if (totalKernelStaked == 0) revert NoStakedTokens();
``` [4](#0-3) 

But there is **no on-chain enforcement** preventing `totalKernelStaked` from dropping to zero *after* a reward period has started. Any staker can call `initiateWithdrawal` at any time:

```solidity
function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
    ...
    balanceOf[msg.sender] -= _amount;
    totalKernelStaked -= _amount;
    ...
}
``` [5](#0-4) 

There is no `rescueTokens` or equivalent admin function in `KernelDepositPool` to recover the stuck `rewardsToken` balance. The `notifyRewardAmount` function uses `safeTransferFrom` to pull a *new* amount and cannot recycle the already-stuck balance. [6](#0-5) 

---

### Impact Explanation

Rewards accrued during any window where `totalKernelStaked == 0` are permanently frozen in the contract. No staker can ever claim them, and no admin function can recover them. This matches **Medium — Permanent freezing of unclaimed yield** in the allowed impact scope.

---

### Likelihood Explanation

Medium. The scenario requires all active stakers to call `initiateWithdrawal` during an active reward period. This is a realistic edge case: reward periods can span weeks (`duration` is admin-configurable), and stakers have full autonomy to unstake at any time. A single coordinated exit by all stakers — or a gradual drain to zero — is sufficient to trigger the loss. No admin compromise or privileged action is required beyond the initial `notifyRewardAmount` call.

---

### Recommendation

Only advance `updatedAt` when rewards are actually being distributed (i.e., when `totalKernelStaked > 0`). Change the modifier to:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

This ensures the time elapsed while no tokens are staked is not consumed, so rewards remain claimable when stakers return.

---

### Proof of Concept

1. Admin calls `notifyRewardAmount(1000e18)` with `totalKernelStaked = 100e18`. `rewardRate`, `finishAt`, and `updatedAt = T0` are set.
2. At time `T1`, all stakers call `initiateWithdrawal(all)`. The last call triggers `updateReward`: `rewardPerTokenStored` is updated correctly for the `T0→T1` window, `updatedAt = T1`, then `totalKernelStaked = 0`.
3. Time advances to `T2` (e.g., 7 days later). Rewards worth `rewardRate * (T2 - T1)` have accrued but cannot be distributed.
4. A new staker calls `stake(1e18)`. `updateReward` fires: `rewardPerToken()` returns `rewardPerTokenStored` unchanged (since `totalKernelStaked` was 0), but `updatedAt` is set to `T2`.
5. `totalKernelStaked = 1e18`. Future rewards are calculated from `T2` onward only.
6. All rewards for the `T1→T2` window (`rewardRate * (T2 - T1)` tokens) are permanently stuck in the contract with no recovery path.

    function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
        if (_amount == 0) revert AmountZero();

        // Prevent starting a reward period when no tokens are staked to avoid unallocated rewards
        if (totalKernelStaked == 0) revert NoStakedTokens();

        // Transfer reward tokens into the contract
        uint256 balanceBefore = rewardsToken.balanceOf(address(this));
        rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
        uint256 balanceAfter = rewardsToken.balanceOf(address(this));
        // Calculate the actual amount of tokens received in case of a transfer fee (tax)
        uint256 receivedAmount = balanceAfter - balanceBefore;

        if (block.timestamp >= finishAt) {
            rewardRate = receivedAmount / duration;
        } else {
            uint256 remaining = (finishAt - block.timestamp) * rewardRate;
            rewardRate = (receivedAmount + remaining) / duration;
        }

        if (rewardRate == 0) revert RewardRateZero();

        finishAt = block.timestamp + duration;
        updatedAt = block.timestamp;

        emit NotifyRewardAmount(receivedAmount, finishAt);
    }
