### Title
ETH Push-Payment in `_processWithdrawalCompletion` Permanently Freezes User ETH — (`contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTWithdrawalManager._processWithdrawalCompletion` pushes ETH directly to the withdrawing user's address via a low-level `call`. If the recipient is a smart contract that reverts on ETH receipt, every attempt to complete the withdrawal reverts, permanently freezing the ETH inside `LRTWithdrawalManager` with no admin rescue path.

---

### Finding Description

`_transferAsset` (the terminal step of every ETH withdrawal) uses a push-payment pattern: [1](#0-0) 

This is called unconditionally at the end of `_processWithdrawalCompletion`: [2](#0-1) 

All state mutations that precede the transfer — `popFront`, `delete withdrawalRequests[requestId]`, and `unlockedWithdrawalsCount[asset]--` — are part of the same transaction: [3](#0-2) 

Because Solidity reverts the entire transaction on failure, when `_transferAsset` reverts (recipient contract rejects ETH), none of the state mutations persist. The withdrawal request remains at the front of the user's queue, `unlockedWithdrawalsCount[asset]` stays elevated, and the ETH that was already pulled from `LRTUnstakingVault` into `LRTWithdrawalManager` during `unlockQueue` sits idle in the contract forever.

The operator-callable `completeWithdrawalForUser` provides no relief — it calls the same internal function and still pushes ETH to the same `user` address: [4](#0-3) 

The only other ETH-exit path, `sweepRemainingAssets`, is gated behind `hasUnlockedWithdrawals`: [5](#0-4) 

Because `unlockedWithdrawalsCount[asset]` can never be decremented for the stuck request, `sweepRemainingAssets` is also permanently blocked for that asset.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once `unlockQueue` has pulled the ETH from `LRTUnstakingVault` into `LRTWithdrawalManager`, that ETH is irrecoverably locked if the recipient contract rejects ETH. There is no admin function to redirect the payment to an alternate address, no escape hatch to clear the stuck `unlockedWithdrawalsCount`, and no way to call `sweepRemainingAssets` while the counter is non-zero. The ETH is permanently frozen inside `LRTWithdrawalManager`.

---

### Likelihood Explanation

**High.**

- **Natural occurrence**: Any smart-contract wallet (e.g., a Gnosis Safe whose fallback/receive is not configured, a multisig, or a proxy without a payable fallback) that initiates an ETH withdrawal will trigger this path without any malicious intent.
- **Deliberate attack**: An attacker deploys a contract that explicitly reverts on ETH receipt, acquires rsETH (via `LRTDepositPool.depositETH` or secondary market), calls `initiateWithdrawal(ETH_TOKEN, ...)`, and waits for the operator to run `unlockQueue`. The ETH is then frozen. The attacker sacrifices the rsETH value but permanently freezes the corresponding ETH in the protocol.

No privileged access is required. The entry path (`initiateWithdrawal` → operator `unlockQueue` → `completeWithdrawal`) is the standard, fully public withdrawal lifecycle. [6](#0-5) 

---

### Recommendation

Replace the push-payment pattern with a pull-payment (withdrawal) pattern for ETH:

1. Instead of calling `payable(to).call{value: amount}("")` inside `_processWithdrawalCompletion`, store the owed amount in a mapping: `pendingEthWithdrawals[user] += amount`.
2. Add a separate `claimEth()` function that lets the user pull their ETH at will.
3. This decouples the accounting step (which must succeed atomically) from the ETH delivery step (which depends on the recipient's fallback behavior).

Alternatively, if the push pattern is retained, add an admin function that allows redirecting a stuck withdrawal to a different address supplied by the user (with appropriate authentication), or a manager-level function to forcibly clear a stuck `unlockedWithdrawalsCount` entry and sweep the ETH to treasury.

---

```solidity
pragma solidity ^0.8.0;

interface ILRTDepositPool {
    function depositETH(uint256 minRSETHAmountExpected, string calldata referralId) external payable;
}

interface ILRTWithdrawalManager {
    function initiateWithdrawal(address asset, uint256 rsETHUnstaked, string calldata referralId) external;
    function completeWithdrawal(address asset, string calldata referralId) external;
}

interface IERC20 {
    function approve(address, uint256) external returns (bool);
    function balanceOf(address) external view returns (uint256);
}

address constant ETH_TOKEN = 0xEeeeeEeeeEeEeeEeEeEeeEEEeeeeEeeeeeeeEEeE;

contract MaliciousNoReceive {
    // Intentionally rejects all ETH
    receive() external payable { revert("no ETH"); }

    function attack(
        address depositPool,
        address withdrawalManager,
        address rsETH,
        uint256 depositAmount
    ) external payable {
        // Step 1: Deposit ETH to get rsETH
        ILRTDepositPool(depositPool).depositETH{value: depositAmount}(0, "");

        // Step 2: Approve withdrawal manager to pull rsETH
        uint256 rsETHBalance = IERC20(rsETH).balanceOf(address(this));
        IERC20(rsETH).approve(withdrawalManager, rsETHBalance);

        // Step 3: Initiate ETH withdrawal
        ILRTWithdrawalManager(withdrawalManager).initiateWithdrawal(ETH_TOKEN, rsETHBalance, "");

        // --- Operator calls unlockQueue off-chain after delay ---
        // ETH is now pulled from LRTUnstakingVault into LRTWithdrawalManager

        // Step 4: completeWithdrawal always reverts because this contract rejects ETH
        // ILRTWithdrawalManager(withdrawalManager).completeWithdrawal(ETH_TOKEN, ""); // <-- always reverts
        // completeWithdrawalForUser also reverts for the same reason
        // sweepRemainingAssets is blocked by unlockedWithdrawalsCount > 0
        // ETH is permanently frozen in LRTWithdrawalManager
    }
}
```

The ETH corresponding to `depositAmount` is permanently frozen in `LRTWithdrawalManager` after the operator runs `unlockQueue`. Neither `completeWithdrawal`, `completeWithdrawalForUser`, nor `sweepRemainingAssets` can recover it. [1](#0-0) [7](#0-6)

```text
    function _processWithdrawalCompletion(address asset, address user, string calldata referralId) internal {
        if (userAssociatedNonces[asset][user].empty()) {
            revert NoWithdrawalRequests(user, asset);
        }

        // Retrieve and remove the oldest withdrawal request for the user.
        uint256 usersFirstWithdrawalRequestNonce = userAssociatedNonces[asset][user].popFront();
        // Ensure the request is already unlocked.
        if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();

        bytes32 requestId = getRequestId(asset, usersFirstWithdrawalRequestNonce);
        WithdrawalRequest memory request = withdrawalRequests[requestId];

        delete withdrawalRequests[requestId];

        // Check that the withdrawal delay has passed since the request's initiation.
        if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();

        unlockedWithdrawalsCount[asset]--;

        // If Aave integration is enabled and asset is ETH, withdraw from Aave if needed
        if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
            uint256 contractBalance = address(this).balance;
            if (contractBalance < request.expectedAssetAmount) {
                uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
                _withdrawFromAave(amountNeeded);

                // Verify we have sufficient balance after withdrawal
                uint256 balanceAfter = address(this).balance;
                if (balanceAfter < request.expectedAssetAmount) {
                    revert InsufficientLiquidityForWithdrawal();
                }
            }
        }

        _transferAsset(asset, user, request.expectedAssetAmount);

        emit ReferralIdEmitted(referralId);
        emit AssetWithdrawalFinalized(user, asset, request.rsETHUnstaked, request.expectedAssetAmount);
    }
```
