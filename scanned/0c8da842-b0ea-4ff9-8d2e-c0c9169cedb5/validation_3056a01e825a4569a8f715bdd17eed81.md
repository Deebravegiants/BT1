### Title
aWETH Balance Held by `LRTWithdrawalManager` Not Included in `getETHDistributionData()` — (File: contracts/LRTDepositPool.sol)

---

### Summary

`LRTDepositPool.getETHDistributionData()` enumerates every ETH-holding location in the protocol to compute the total ETH backing rsETH. It counts native ETH in the deposit pool, NDCs, EigenLayer, `LRTUnstakingVault`, and `LRTConverter`, but it completely omits `LRTWithdrawalManager`. When the Aave integration is active, `LRTWithdrawalManager` converts received ETH into aWETH (an ERC-20 yield-bearing token). That aWETH balance is never included in the oracle's TVL calculation, causing `rsETHPrice` to be understated for the entire duration the ETH sits in Aave.

---

### Finding Description

`LRTDepositPool.getETHDistributionData()` is the single source of truth for total protocol ETH: [1](#0-0) 

It sums six locations:

| Location | How counted |
|---|---|
| `LRTDepositPool` | `address(this).balance` |
| Each NDC | `nodeDelegatorQueue[i].balance` |
| EigenLayer staked | `getEffectivePodShares()` |
| EigenLayer unstaking | `getAssetUnstaking(ETH_TOKEN)` |
| `LRTUnstakingVault` | `lrtUnstakingVault.balance` |
| `LRTConverter` | `ethValueInWithdrawal` |

`LRTWithdrawalManager` is absent from this list entirely.

`LRTWithdrawalManager` has an Aave integration that deposits ETH into Aave v3 via `aaveWETHGateway.depositETH()` and receives aWETH in return: [2](#0-1) 

The aWETH balance is tracked separately: [3](#0-2) 

The ETH flow that creates this gap:

1. `LRTUnstakingVault.redeem()` sends ETH to `LRTWithdrawalManager` (called during `_processWithdrawalCompletion`): [4](#0-3) 

2. `LRTWithdrawalManager._depositToAave()` converts that ETH to aWETH: [5](#0-4) 

After step 2, `lrtUnstakingVault.balance` has decreased (ETH left the vault) and the aWETH now sitting in `LRTWithdrawalManager` is not counted anywhere in `getETHDistributionData()`. The ETH has effectively vanished from the oracle's view.

`getETHDistributionData()` feeds `getTotalAssetDeposits(ETH)`: [6](#0-5) 

Which feeds `_getTotalEthInProtocol()` in `LRTOracle`: [7](#0-6) 

Which is used to compute `rsETHPrice`: [8](#0-7) 

---

### Impact Explanation

When ETH is deposited to Aave by `LRTWithdrawalManager`, `_getTotalEthInProtocol()` returns a value lower than the true protocol TVL. The resulting `rsETHPrice` is understated. Because `getRsETHAmountToMint = amount × assetPrice / rsETHPrice`, new depositors receive more rsETH per ETH than they are entitled to, diluting the value held by existing rsETH holders. Additionally, the protocol fee calculation (`protocolFeeInETH = rewardAmount × feeBPS / 10000`) is computed on a deflated TVL, causing fee under-collection. The mis-accounting persists for as long as ETH remains in Aave (which can be an extended period while withdrawal requests queue up).

**Impact class**: Low — contract fails to deliver promised returns to existing rsETH holders; their share value is diluted by new mints priced against an understated TVL.

---

### Likelihood Explanation

The Aave integration is a deployed, configurable feature (`isAaveIntegrationEnabled`). Any unprivileged user who calls `completeWithdrawal` (or whose withdrawal is processed by the operator) triggers `_processWithdrawalCompletion`, which calls `LRTUnstakingVault.redeem()` and subsequently `_depositToAave()`. No special role is required to initiate the ETH flow that creates the accounting gap. The gap is proportional to the total ETH committed to pending withdrawals that has been deposited to Aave.

---

### Recommendation

Include the aWETH balance of `LRTWithdrawalManager` in `getETHDistributionData()`. Concretely, add a term:

```solidity
address lrtWithdrawalManager = lrtConfig.getContract(LRTConstants.LRT_WITHDRAW_MANAGER);
ethLyingInWithdrawalManager = ILRTWithdrawalManager(lrtWithdrawalManager).getAaveBalance()
    + lrtWithdrawalManager.balance;
```

and include `ethLyingInWithdrawalManager` in the return tuple and in `getTotalAssetDeposits`.

---

### Proof of Concept

1. Aave integration is enabled in `LRTWithdrawalManager` (`isAaveIntegrationEnabled = true`).
2. A batch of users request ETH withdrawals; `assetsCommitted[ETH]` grows to, say, 500 ETH.
3. The operator unlocks the batch; `LRTUnstakingVault.redeem()` is called, moving 500 ETH to `LRTWithdrawalManager`.
4. `_depositToAave(500 ETH)` is called; `LRTWithdrawalManager` now holds 500 aWETH, `lrtUnstakingVault.balance` is 500 ETH lower.
5. `updateRSETHPrice()` is called. `getETHDistributionData()` returns a total 500 ETH less than the true protocol TVL (the aWETH is invisible).
6. `rsETHPrice = (totalETH - 500 ETH) / rsETHSupply` — understated.
7. A new depositor calls `depositETH(1 ETH)`. `getRsETHAmountToMint = 1e18 × 1e18 / understatedPrice` — they receive more rsETH than the true exchange rate entitles them to.
8. Existing holders' rsETH is diluted; the protocol fee is computed on a deflated base. [9](#0-8) [10](#0-9) [2](#0-1)

        bool protocolPaused = lrtDepositPool.paused() || withdrawalManager.paused() || paused;

    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
