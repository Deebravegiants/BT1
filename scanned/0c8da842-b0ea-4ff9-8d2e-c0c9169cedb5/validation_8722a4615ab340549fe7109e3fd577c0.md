### Title
`RSETHPoolV3.deposit()` Lacks Slippage Protection for rsETH Minting, Unlike `LRTDepositPool` - (File: contracts/pools/RSETHPoolV3.sol)

---

### Summary

The `deposit()` functions in the L2 pool contracts (`RSETHPoolV3`, `RSETHPool`, `RSETHPoolNoWrapper`, `RSETHPoolV3ExternalBridge`) do not accept a `minRsETHAmountExpected` parameter. The mainnet `LRTDepositPool.depositETH()` and `depositAsset()` explicitly enforce a minimum rsETH output check. Users depositing on L2 have no on-chain protection against receiving fewer rsETH than previewed if the oracle rate changes between the preview call and transaction execution.

---

### Finding Description

`RSETHPoolV3.deposit()` computes the rsETH amount to mint by calling `viewSwapRsETHAmountAndFee()`, which reads the live rate from `rsETHOracle.getRate()` at execution time:

```solidity
// RSETHPoolV3.sol lines 246-265
function deposit(string memory referralId)
    external payable nonReentrant whenNotPaused
    limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);
    ...
}
``` [1](#0-0) 

The rate computation:

```solidity
// RSETHPoolV3.sol lines 299-308
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
``` [2](#0-1) 

There is no parameter allowing the user to specify a minimum acceptable rsETH output. By contrast, the mainnet `LRTDepositPool` explicitly enforces this:

```solidity
// LRTDepositPool.sol lines 76-93
function depositETH(uint256 minRSETHAmountExpected, string calldata referralId)
    external payable ...
{
    uint256 rsethAmountToMint = _beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, minRSETHAmountExpected);
    _mintRsETH(rsethAmountToMint);
    ...
}
``` [3](#0-2) 

The `_beforeDeposit` internal function enforces the minimum:

```solidity
// LRTDepositPool.sol lines 667-669
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [4](#0-3) 

The same gap exists in the token deposit overload: [5](#0-4) 

And in the analogous contracts `RSETHPool`, `RSETHPoolNoWrapper`, and `RSETHPoolV3ExternalBridge`, all of which expose public `deposit()` functions without a minimum output guard.



---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A user calls `viewSwapRsETHAmountAndFee()` off-chain to preview the rsETH they will receive, then submits the deposit transaction. If the `rsETHOracle` rate is updated (rsETH price increases) before the transaction is mined, the user receives fewer rsETH than previewed with no on-chain recourse. The ETH is accepted and the lower rsETH amount is minted with no revert. The user cannot express a minimum acceptable output, so the contract silently delivers less than the user expected.

---

### Likelihood Explanation

**Low.** The `rsETHOracle` rate reflects the rsETH/ETH exchange rate, which increases monotonically as restaking rewards accrue. Rate updates are infrequent and small under normal conditions. However, the gap is structurally present and becomes more relevant during periods of rapid oracle updates or high network congestion where transactions are delayed. The mainnet contract's explicit `minRSETHAmountExpected` guard demonstrates the protocol's own awareness that this protection is necessary — it was simply not carried over to the L2 pool contracts.

---

### Recommendation

Add a `minRsETHAmountExpected` parameter to all `deposit()` functions in `RSETHPoolV3`, `RSETHPool`, `RSETHPoolNoWrapper`, and `RSETHPoolV3ExternalBridge`, and revert if the computed rsETH amount falls below it:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountExpected)
    external payable nonReentrant whenNotPaused
    limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountExpected) revert SlippageExceeded();
    ...
}
```

This mirrors the protection already present in `LRTDepositPool._beforeDeposit()`.

---

### Proof of Concept

1. User calls `viewSwapRsETHAmountAndFee(1 ether)` off-chain. Oracle rate = 1.05 ETH/rsETH → expected rsETH ≈ 0.952 rsETH.
2. Oracle is updated: rate = 1.10 ETH/rsETH (rsETH appreciated).
3. User's `deposit{value: 1 ether}("ref")` transaction executes.
4. `viewSwapRsETHAmountAndFee` now returns ≈ 0.909 rsETH.
5. `wrsETH.mint(msg.sender, 0.909e18)` is called — user receives ~4.5% fewer rsETH than previewed with no revert.
6. On mainnet, the same user calling `LRTDepositPool.depositETH(0.952e18, "ref")` would revert with `MinimumAmountToReceiveNotMet`. [6](#0-5) [7](#0-6)
