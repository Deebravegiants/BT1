### Title
ETH Withdrawal Completion Permanently Bricked When Aave Reserve Is Paused — (`contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTWithdrawalManager` integrates with Aave v3 to earn yield on ETH that has been unlocked for user withdrawals. When `isAaveIntegrationEnabled` is true, `_processWithdrawalCompletion` mandatorily calls `_withdrawFromAave` whenever the contract's native ETH balance is insufficient to cover a pending withdrawal. If Aave's WETH reserve is paused or frozen, `aaveWETHGateway.withdrawETH` reverts, causing every call to `completeWithdrawal` and `completeWithdrawalForUser` to revert. The intended escape hatch, `emergencyWithdrawFromAave`, routes through the same `_withdrawFromAave` path and therefore also fails. User ETH is permanently locked inside Aave with no on-chain rescue path.

---

### Finding Description

**Step 1 — ETH is deposited into Aave during `unlockQueue`.**

`unlockQueue` (lines 301–317) burns the user's rsETH, pulls ETH from `LRTUnstakingVault`, and then attempts to deposit that ETH into Aave:

```solidity
// LRTWithdrawalManager.sol L305-316
if (rsETHBurned != 0) IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned);
unstakingVault.redeem(asset, assetAmountUnlocked);

if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN && assetAmountUnlocked > 0) {
    try this.depositToAaveExternal(assetAmountUnlocked) { }
    catch (bytes memory reason) {
        emit AaveDepositFailed(assetAmountUnlocked, reason);
        // Silently fail if Aave deposit fails
    }
}
```

The `try/catch` correctly handles a failed Aave deposit — ETH stays in the contract. However, when the deposit *succeeds*, the ETH moves into Aave and `totalETHDepositedToAave` is incremented.

**Step 2 — `completeWithdrawal` mandatorily calls `_withdrawFromAave` with no fallback.**

`_processWithdrawalCompletion` (lines 719–731):

```solidity
if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
    uint256 contractBalance = address(this).balance;
    if (contractBalance < request.expectedAssetAmount) {
        uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
        _withdrawFromAave(amountNeeded);          // <-- no try/catch

        uint256 balanceAfter = address(this).balance;
        if (balanceAfter < request.expectedAssetAmount) {
            revert InsufficientLiquidityForWithdrawal();
        }
    }
}
```

`_withdrawFromAave` (line 917) calls:

```solidity
aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
```

If Aave's WETH reserve is paused, this external call reverts. Because there is no `try/catch` here, the entire `completeWithdrawal` transaction reverts. Every ETH withdrawal whose amount exceeds the contract's idle balance is permanently blocked.

**Step 3 — The emergency escape hatch is equally broken.**

`emergencyWithdrawFromAave` (lines 551–563) calls `_withdrawFromAave` directly:

```solidity
uint256 withdrawnAmount = _withdrawFromAave(amount);
```

It routes through the same `aaveWETHGateway.withdrawETH` call and therefore also reverts when Aave is paused. `setAaveIntegrationEnabled(false)` and `configureAaveIntegration` both call `_withdrawFromAave` as well, closing every admin-controlled rescue path.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

By the time a user calls `completeWithdrawal`, their rsETH has already been burned irreversibly in `unlockQueue`. The corresponding ETH has been deposited into Aave. If Aave's WETH reserve is paused:

- `completeWithdrawal` reverts for every ETH withdrawal that requires Aave liquidity.
- `emergencyWithdrawFromAave` reverts.
- `setAaveIntegrationEnabled(false)` reverts.
- There is no alternative code path to retrieve the ETH.

Users have lost rsETH and cannot receive ETH. The funds are permanently frozen inside Aave with no on-chain rescue.

---

### Likelihood Explanation

**Low–Medium.**

Aave v3 reserves have a `PAUSED` state that can be triggered by the Aave Guardian (a multisig) in response to an emergency (e.g., oracle manipulation, exploit detection). This has occurred on live deployments. The Aave WETH reserve on Ethereum mainnet is a high-value target. The condition is not attacker-controlled, but it is a realistic protocol-level event that has precedent. Once triggered, the freeze persists until Aave governance unpauses the reserve — which may take days or be indefinite if the reserve is deprecated.

---

### Recommendation

Mirror the same `try/catch` pattern used in `unlockQueue` inside `_processWithdrawalCompletion`. If the Aave withdrawal fails, fall back to whatever idle ETH balance is available and revert only if that is also insufficient:

```solidity
if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
    uint256 contractBalance = address(this).balance;
    if (contractBalance < request.expectedAssetAmount) {
        uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
        try this.withdrawFromAaveExternal(amountNeeded) { }
        catch { }   // Aave paused/frozen — fall through to balance check
        uint256 balanceAfter = address(this).balance;
        if (balanceAfter < request.expectedAssetAmount) {
            revert InsufficientLiquidityForWithdrawal();
        }
    }
}
```

Additionally, `emergencyWithdrawFromAave` should be able to force-disable the Aave integration flag even when `_withdrawFromAave` fails, so that subsequent `completeWithdrawal` calls bypass the Aave path entirely and serve users from idle contract balance.

---

### Proof of Concept

1. Aave integration is enabled; `isAaveIntegrationEnabled = true`.
2. Operator calls `unlockQueue(ETH_TOKEN, ...)`. rsETH is burned; ETH is pulled from `LRTUnstakingVault` and successfully deposited into Aave. `totalETHDepositedToAave > 0`; contract's idle ETH balance is 0.
3. Aave Guardian pauses the WETH reserve (realistic emergency action).
4. User calls `completeWithdrawal(ETH_TOKEN, ...)`.
5. `_processWithdrawalCompletion` enters the Aave branch: `address(this).balance (0) < request.expectedAssetAmount`.
6. `_withdrawFromAave(amountNeeded)` is called → `aaveWETHGateway.withdrawETH(...)` reverts because the reserve is paused.
7. `completeWithdrawal` reverts. User cannot retrieve ETH.
8. Admin calls `emergencyWithdrawFromAave(type(uint256).max)` → same `_withdrawFromAave` path → same revert.
9. Admin calls `setAaveIntegrationEnabled(false)` → same `_withdrawFromAave` path → same revert.
10. ETH is permanently frozen in Aave; user's rsETH is already burned. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```text
    function _withdrawFromAave(uint256 amount) internal returns (uint256 withdrawnAmount) {
        if (amount == 0) return 0;

        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        if (aaveBalance == 0) revert InsufficientAaveBalance();

        // Only withdraw up to the principal amount (don't use accrued interest for user withdrawals)
        uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave;

        withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
        if (withdrawnAmount == 0) return 0;

        aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
        totalETHDepositedToAave -= withdrawnAmount;
```
