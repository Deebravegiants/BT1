### Title
Fee Bypass via Integer Truncation in Deposit Fee Calculation - (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPool.sol)

---

### Summary

All L2 deposit pool contracts compute protocol fees using integer division that truncates to zero for sufficiently small deposit amounts. Because the only lower-bound check on a deposit is `amount != 0`, an unprivileged depositor can call `deposit()` repeatedly with dust-sized amounts and pay zero fees on the full notional value deposited.

---

### Finding Description

Every pool variant computes the deposit fee identically:

```solidity
fee = amount * feeBps / 10_000;
```

Solidity performs integer division, so whenever `amount * feeBps < 10_000` the result truncates to `0`. The deposit functions enforce only `if (amount == 0) revert InvalidAmount();` — no minimum deposit amount is enforced beyond zero.

Affected locations:

- `RSETHPoolV3.viewSwapRsETHAmountAndFee(uint256)` — ETH path [1](#0-0) 

- `RSETHPoolV3.viewSwapRsETHAmountAndFee(uint256,address)` — ERC-20 token path [2](#0-1) 

- `RSETHPool.viewSwapRsETHAmountAndFee(uint256)` — ETH path [3](#0-2) 

- `RSETHPool.viewSwapRsETHAmountAndFee(uint256,address)` — per-token fee path [4](#0-3) 

The same pattern is structurally present in `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, and `RSETHPoolNoWrapper` (all share the same `feeBps / 10_000` formula). [5](#0-4) [6](#0-5) [7](#0-6)

### Impact Explanation

**High — Theft of unclaimed yield (protocol fee revenue).**

The protocol collects fees into `feeEarnedInETH` / `feeEarnedInToken[token]` and withdraws them via `withdrawFees`. When fee truncates to zero, those accumulators are never incremented, so the protocol permanently loses the fee revenue it is owed. A depositor who splits a large deposit into many sub-threshold transactions receives the same rsETH/wrsETH output as a single fee-paying deposit but contributes nothing to the fee pool. [10](#0-9) 

---

### Likelihood Explanation

**High.** The attack requires no special role, no front-running, and no external dependency. Any EOA or contract that can call `deposit()` can exploit this. The threshold below which fees round to zero is easily computed from the public `feeBps` state variable. On L2 networks (Arbitrum, Optimism, Base, etc.) gas costs are negligible, making thousands of micro-deposits economically rational. For example, with `feeBps = 5` (0.05%), any single deposit below `10_000 / 5 = 2_000 wei` incurs zero fee; a user depositing 1 ETH in 500_000 iterations of 2_000 wei each pays nothing. [11](#0-10) 

---

### Recommendation

Round the fee calculation **up** (in the protocol's favor) using ceiling division:

```solidity
// Before (rounds down — vulnerable)
fee = amount * feeBps / 10_000;

// After (rounds up — protocol-favoring)
fee = (amount * feeBps + 9_999) / 10_000;
```

Alternatively, enforce a minimum deposit amount such that `amount * feeBps >= 10_000` always holds (i.e., `amount >= 10_000 / feeBps`). Apply the fix uniformly across all pool variants: `RSETHPoolV3`, `RSETHPool`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, and `RSETHPoolNoWrapper`.

---

### Proof of Concept

**Setup**: `RSETHPoolV3` deployed with `feeBps = 5` (0.05%). rsETH/ETH rate = 1.05e18 (i.e., 1 rsETH costs 1.05 ETH).

**Fee-bypass threshold**: `10_000 / 5 = 2_000 wei`. Any single ETH deposit below 2_000 wei produces `fee = 0`.

**Attack**:
1. Attacker wants to deposit 1 ETH (1e18 wei) without paying the 0.05% fee (= 5e14 wei ≈ $1.50 at $3,000/ETH).
2. Instead of one call with `msg.value = 1e18`, attacker calls `deposit(referralId)` 500_000 times, each with `msg.value = 2_000 - 1 = 1_999 wei`.
3. Each call: `fee = 1_999 * 5 / 10_000 = 9_995 / 10_000 = 0`.
4. `feeEarnedInETH` remains 0 throughout; attacker receives full rsETH value on every micro-deposit.
5. Total deposited: `500_000 × 1_999 = 999_500_000 wei ≈ 1 ETH`. Total fee paid: **0 wei**.
6. On L2 (e.g., Optimism), 500_000 transactions cost on the order of a few dollars in gas — far less than the $1.50 fee avoided, and the attack scales linearly with deposit size.

The `feeEarnedInETH` accumulator is never incremented, so `withdrawFees` will yield nothing for this deposited value. [12](#0-11) [13](#0-12)
