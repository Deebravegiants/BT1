### Title
Block Stuffing Enables Attacker to Exhaust Deposit Limit and Revert Victim's `depositAsset` Transaction - (File: contracts/LRTDepositPool.sol)

---

### Summary

The deposit limit check in `_beforeDeposit` is evaluated at transaction execution time with no reservation mechanism. An attacker can use block stuffing to delay a victim's pending `depositAsset` transaction while simultaneously filling the remaining deposit capacity for a given LST asset, causing the victim's transaction to revert with `MaximumDepositLimitReached`.

---

### Finding Description

`depositAsset` calls `_beforeDeposit`, which calls `_checkIfDepositAmountExceedesCurrentLimit`: [1](#0-0) [2](#0-1) 

The check `totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)` is computed at execution time against live on-chain state. There is no mechanism to reserve capacity for a pending transaction, no deadline parameter, and no slippage guard on deposit-limit availability.

The attack sequence:

1. Deposit limit for stETH is `X`. Current total deposits are `X - delta`.
2. Victim submits `depositAsset(stETH, delta)` — valid at submission time.
3. Attacker observes the pending transaction in the mempool and begins block stuffing: submitting high-gas-price filler transactions to consume block space and delay the victim's transaction.
4. Concurrently, the attacker submits their own `depositAsset(stETH, delta)` calls (at higher gas price, or in the same stuffed blocks) to consume the remaining `delta` capacity.
5. When the victim's transaction finally executes, `getTotalAssetDeposits(stETH) + delta > X`, and it reverts with `MaximumDepositLimitReached`.

---

### Impact Explanation

The victim's transaction reverts. The victim's tokens are never transferred into the protocol (the `safeTransferFrom` at line 114 is never reached), so funds are not frozen inside the contract. The impact is a failed deposit and potential denial of access to the protocol if the limit remains full — matching the **Low. Block stuffing** impact category. [3](#0-2) 

---

### Likelihood Explanation

Block stuffing on Ethereum mainnet is expensive (attacker must pay for ~30M gas per block at prevailing gas prices). The attacker also needs sufficient stETH to fill the remaining limit. Critically, the attacker has **no economic profit motive**: they receive rsETH for their own deposits (which they could obtain without the attack) but pay substantial extra gas for the stuffing. This makes the attack economically irrational in most scenarios, keeping likelihood **low**.

---

### Recommendation

1. Add a `deadline` parameter to `depositAsset` so users can bound how long their transaction can remain pending.
2. Alternatively, allow users to specify a minimum remaining deposit capacity they require, reverting with a user-friendly error if capacity has been consumed since signing.
3. Consider a commit-reveal or queue-based deposit reservation scheme if deposit limits are expected to be competitive.

The root cause is the absence of any execution-time reservation or deadline guard in `_beforeDeposit`. [4](#0-3)

        address asset,
        uint256 depositAmount,
        uint256 minRSETHAmountExpected
    )
        private
        view
        returns (uint256 rsethAmountToMint)
    {
        if (depositAmount == 0 || depositAmount < minAmountToDeposit) {
            revert InvalidAmountToDeposit();
        }

        if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
            revert MaximumDepositLimitReached();
        }

        rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

        if (rsethAmountToMint < minRSETHAmountExpected) {
            revert MinimumAmountToReceiveNotMet();
        }
    }

    function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
        uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
        if (asset == LRTConstants.ETH_TOKEN) {
            return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
        }
        return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
    }
