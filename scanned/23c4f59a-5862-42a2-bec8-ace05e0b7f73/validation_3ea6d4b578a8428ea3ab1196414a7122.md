### Title
Owner Can Drain Already-Vested KERNEL Tokens Before Users Claim Them - (File: contracts/KERNEL/KernelTop100MerkleDistributor.sol)

### Summary
The `withdrawTokens` function in `KernelTop100MerkleDistributor` allows the owner to withdraw any amount of KERNEL tokens from the contract with no accounting check against tokens already vested and owed to users. This is a direct analog to the `VestingTrustee.revoke` bug: a privileged function that fails to distinguish between tokens already earned by users and tokens that are genuinely unallocated.

### Finding Description
`KernelTop100MerkleDistributor` distributes KERNEL tokens to eligible users over a 30-day linear vesting schedule. Users accumulate vested-but-unclaimed tokens over time, calculated by `_getUnclaimedVestedAmount`. The contract holds the full token supply for all users.

The `withdrawTokens` admin function performs no accounting against vested-but-unclaimed balances:

```solidity
function withdrawTokens(address _token, uint256 _amount, address _recipient) external onlyOwner {
    UtilLib.checkNonZeroAddress(_token);
    UtilLib.checkNonZeroAddress(_recipient);
    if (_amount == 0) {
        revert ZeroValueProvided();
    }
    IERC20(_token).safeTransfer(_recipient, _amount);
    emit TokensWithdrawn(_token, _amount, _recipient);
}
``` [1](#0-0) 

There is no check against the total vested-but-unclaimed amount across all users. The vesting accounting only tracks `userClaims[user].amountClaimed` per user, with no global counter of total tokens owed. [2](#0-1) 

The vested amount for any user at any point in time is:
```
totalVestedAmount = (userTotalClaimableAmount * totalElapsedTime) / VESTING_DURATION
unclaimedAmount   = totalVestedAmount - userClaim.amountClaimed
``` [3](#0-2) 

Once `withdrawTokens` drains the KERNEL balance, subsequent calls to `claim` or `claimAndStake` will revert at the `kernel.safeTransfer` call, permanently blocking users from receiving their already-vested tokens. [4](#0-3) 

### Impact Explanation
**High — Theft of unclaimed yield.** Users who have vested KERNEL tokens (i.e., time has elapsed past `vestingStartTimestamp`) but have not yet called `claim` lose those tokens permanently if the owner calls `withdrawTokens` with an amount that covers the contract's balance. The tokens are transferred to an arbitrary `_recipient`, constituting direct theft of earned yield from all affected users.

### Likelihood Explanation
The owner is a single EOA or multisig. The function requires no special precondition beyond ownership. A malicious or compromised owner can call it at any time after vesting begins. The absence of any guard (no minimum reserve, no vested-amount check) means even an accidental over-withdrawal permanently harms users. The vesting window is only 30 days, so the window of exposure is bounded but real. [5](#0-4) 

### Recommendation
Track the total KERNEL tokens committed to users (sum of all merkle-leaf amounts loaded at initialization, minus total claimed). In `withdrawTokens`, only allow withdrawal of the surplus above the committed amount:

```solidity
uint256 committed = totalAllocated - totalClaimed; // global counters
require(kernel.balanceOf(address(this)) - _amount >= committed, "Would drain vested tokens");
```

Alternatively, restrict `withdrawTokens` to only allow withdrawal of non-KERNEL tokens, or add a time-lock that prevents withdrawal until after `vestingStartTimestamp + VESTING_DURATION`.

### Proof of Concept
1. Protocol deploys `KernelTop100MerkleDistributor` with 1,000,000 KERNEL for 100 users (10,000 each), `vestingStartTimestamp = T`.
2. At `T + 15 days` (halfway through vesting), each user has vested 5,000 KERNEL but none have claimed yet.
3. Owner calls `withdrawTokens(kernel, 1_000_000e18, attacker)` — no revert, all tokens transferred out.
4. Any user calling `claim(10_000e18, proof)` now reverts at `kernel.safeTransfer(user, amountToSend)` because the contract balance is 0.
5. All 500,000 KERNEL of already-vested tokens (across 100 users) are permanently stolen. [6](#0-5)
