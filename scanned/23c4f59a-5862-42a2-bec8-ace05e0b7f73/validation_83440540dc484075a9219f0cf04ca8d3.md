### Title
Uninitialized `withdrawalDelay` in `KernelDepositPool` Collapses Two-Step Withdrawal Into Immediate Claim — (File: contracts/KERNEL/KernelDepositPool.sol)

---

### Summary

`KernelDepositPool.initialize()` never sets `withdrawalDelay`, leaving it at the Solidity default of `0`. Because `initiateWithdrawal()` computes `unlockTime = block.timestamp + withdrawalDelay`, every withdrawal record is stamped with `unlockTime == block.timestamp`. The guard in `claimWithdrawal()` — `if (block.timestamp < withdrawal.unlockTime)` — is therefore never triggered, allowing any staker to initiate and immediately claim a withdrawal in the same block (or any later block), completely bypassing the intended delay.

---

### Finding Description

`KernelDepositPool.initialize()` sets `kernelToken`, `rewardsToken`, and the admin role, but leaves `withdrawalDelay` at `0`: [1](#0-0) 

`initiateWithdrawal()` then computes the unlock time as:

```solidity
uint256 unlockTime = block.timestamp + withdrawalDelay;
``` [2](#0-1) 

With `withdrawalDelay == 0`, `unlockTime` equals the block timestamp at initiation. `claimWithdrawal()` enforces:

```solidity
if (block.timestamp < withdrawal.unlockTime) {
    revert WithdrawalNotReady();
}
``` [3](#0-2) 

Because `block.timestamp` is never strictly less than itself (same block) or a past timestamp (any later block), this guard never fires. The two-step withdrawal process — initiate, wait for delay, claim — collapses into a single-block operation.

The setter `setWithdrawalDelay()` enforces `_withdrawalDelay != 0`, so once set it cannot be zeroed again, but it is never called during initialization: [4](#0-3) 

A prerequisite is that `maxNumberOfWithdrawalsPerUser` must also be non-zero (otherwise `initiateWithdrawal` reverts). That variable is likewise uninitialized to `0` and must be set by the admin via `setMaxNumberOfWithdrawalsPerUser`. In the realistic deployment window where the admin has called `setMaxNumberOfWithdrawalsPerUser` but not yet `setWithdrawalDelay`, the bypass is live.

---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

The withdrawal delay is an explicit security feature (`MAX_WITHDRAWAL_DELAY = 30 days` is declared as a constant, and the delay is documented as a cooldown). When `withdrawalDelay == 0`, any staker can stake KERNEL tokens and reclaim them in the very next block, defeating the cooldown entirely. No other user's funds are at risk and there is no direct theft, but the protocol's stated guarantee — that staked tokens are subject to a delay before exit — is not enforced.

---

### Likelihood Explanation

**Medium.** The window exists from contract deployment until the admin explicitly calls `setWithdrawalDelay`. Given that `setMaxNumberOfWithdrawalsPerUser` must also be called to enable withdrawals at all, the realistic attack window is the period between those two admin calls. Any staker active during that window can exploit the bypass. No special permissions are required beyond being a normal staker.

---

### Recommendation

Initialize `withdrawalDelay` to a safe non-zero default inside `initialize()`, for example:

```solidity
withdrawalDelay = 7 days; // or a protocol-chosen default
```

Alternatively, add a guard in `initiateWithdrawal()`:

```solidity
if (withdrawalDelay == 0) revert WithdrawalDelayNotSet();
```

This mirrors the fix applied in the referenced Lukso report: perform the safety check unconditionally regardless of the uninitialized state.

---

### Proof of Concept

1. Deploy `KernelDepositPool`. Admin calls `setMaxNumberOfWithdrawalsPerUser(100)` but does **not** call `setWithdrawalDelay`. `withdrawalDelay` remains `0`.
2. Attacker calls `stake(1000e18)`.
3. Attacker calls `initiateWithdrawal(1000e18)`. `unlockTime = block.timestamp + 0 = block.timestamp`.
4. In the same block (or any subsequent block), attacker calls `claimWithdrawal(1)`. The check `block.timestamp < block.timestamp` is `false` → no revert → KERNEL tokens returned immediately.
5. The intended delay period is completely skipped. [5](#0-4) [6](#0-5)

    function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
        if (_amount == 0) revert AmountZero();
        if (balanceOf[msg.sender] < _amount) revert InsufficientStakedBalance();
        if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();

        balanceOf[msg.sender] -= _amount;
        totalKernelStaked -= _amount;

        // Create a withdrawal record
        uint256 withdrawalId = ++withdrawalCounter;
        uint256 unlockTime = block.timestamp + withdrawalDelay;

        withdrawals[withdrawalId] = Withdrawal({
            user: msg.sender, amount: _amount, unlockTime: unlockTime, claimed: false, withdrawalId: withdrawalId
        });
        userWithdrawalIds[msg.sender].push(withdrawalId);

        emit WithdrawalInitiated(msg.sender, _amount, withdrawalId, unlockTime);
    }

    function claimWithdrawal(uint256 _withdrawalId) external nonReentrant {
        Withdrawal storage withdrawal = withdrawals[_withdrawalId];

        if (withdrawal.user == address(0)) {
            revert WithdrawalDoesNotExist();
        }

        if (withdrawal.user != msg.sender) {
            revert NotYourWithdrawal();
        }

        if (block.timestamp < withdrawal.unlockTime) {
            revert WithdrawalNotReady();
        }

        if (withdrawal.claimed) {
            revert WithdrawalAlreadyClaimed();
        }

        withdrawal.claimed = true;

        // Remove the withdrawal ID from the user's list of withdrawal IDs
        uint256[] storage userWithdrawalIdsArray = userWithdrawalIds[msg.sender];
        for (uint256 i = 0; i < userWithdrawalIdsArray.length; ++i) {
            if (userWithdrawalIdsArray[i] == _withdrawalId) {
                userWithdrawalIdsArray[i] = userWithdrawalIdsArray[userWithdrawalIdsArray.length - 1];
                userWithdrawalIdsArray.pop();
                break;
            }
        }

        // Transfer the KERNEL tokens from contract to user
        kernelToken.safeTransfer(msg.sender, withdrawal.amount);

        emit WithdrawalClaimed(msg.sender, withdrawal.amount, _withdrawalId);
    }
