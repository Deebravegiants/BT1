### Title
Unbounded Gas Consumption in `updateRSETHPrice()` via Nested Loops with Redundant External Calls - (File: contracts/LRTOracle.sol / contracts/LRTDepositPool.sol / contracts/NodeDelegator.sol)

---

### Summary

`LRTOracle.updateRSETHPrice()` is a public function that can exceed the block gas limit due to a deeply nested call chain that redundantly invokes `NodeDelegator.getAssetUnstaking()` — and therefore EigenLayer's `getQueuedWithdrawals()` — once per supported asset per NDC, with an inner loop over all queued withdrawals and their strategies.

---

### Finding Description

The call chain is:

```
updateRSETHPrice()                          [public, LRTOracle.sol:87]
  └─ _updateRsETHPrice()                    [LRTOracle.sol:214]
       └─ _getTotalEthInProtocol()          [LRTOracle.sol:331]
            └─ for each supported asset:
                 getTotalAssetDeposits(asset) [LRTDepositPool.sol:385]
                   └─ getAssetDistributionData(asset) [LRTDepositPool.sol:426]
                        └─ for each NDC:
                             getAssetUnstaking(asset) [NodeDelegator.sol:405]
                               └─ getQueuedWithdrawals()  [external call]
                                    └─ for each withdrawal:
                                         for each strategy:
                                              sharesToUnderlyingView() [external call]
```

**Root cause — `_getTotalEthInProtocol()` (LRTOracle.sol lines 336–348):** [1](#0-0) 

For every supported asset it calls `getTotalAssetDeposits()`, which calls `getAssetDistributionData()`: [2](#0-1) 

For every NDC, `getAssetUnstaking()` is called: [3](#0-2) 

`getQueuedWithdrawals()` is an external call to EigenLayer that returns all queued withdrawals for the NDC. The result is then iterated with a nested loop over strategies, each making another external call (`sharesToUnderlyingView`).

**The redundancy**: `getQueuedWithdrawals(ndcAddress)` is called once per asset per NDC. With N supported assets and M NDCs, it is called N × M times, each time fetching and iterating the same set of queued withdrawals. The total gas scales as **O(N × M × W × S)** where W = queued withdrawals per NDC and S = strategies per withdrawal.

The protocol team's own comment in `LRTUnstakingVault.setMaxUncompletedWithdrawalCount()` acknowledges the concern: [4](#0-3) 

However, the comment only accounts for a single pass over withdrawals, not the N × M multiplication introduced by the nested asset and NDC loops.

---

### Impact Explanation

If `updateRSETHPrice()` reverts due to out-of-gas, the rsETH price stored in `LRTOracle.rsETHPrice` cannot be updated. Every downstream function that reads this price — including `depositETH()`, `depositAsset()`, `initiateWithdrawal()`, and `completeWithdrawal()` — will either use a stale price or revert, effectively freezing the protocol. This constitutes **temporary (potentially permanent) freezing of funds** for all depositors and withdrawers.

---

### Likelihood Explanation

The protocol currently supports 5+ assets and up to 10 NDCs (`maxNodeDelegatorLimit = 10`). With `maxUncompletedWithdrawalCount` at 80 and 3 strategies per withdrawal, the number of external calls in a single `updateRSETHPrice()` invocation is approximately:

> 5 assets × 10 NDCs × (1 + 80 × 3) external calls ≈ **12,050 external calls**

Each `CALL` opcode costs ≥ 2,100 gas, plus execution. Memory expansion for large `getQueuedWithdrawals()` return arrays adds further cost. This can realistically approach or exceed Ethereum's 30M block gas limit under normal protocol operation — without any malicious action required.

---

### Recommendation

1. **Cache `getQueuedWithdrawals()` per NDC** and reuse the result across all asset iterations, reducing calls from N × M to M.
2. **Restructure `getAssetUnstaking()`** to accept a pre-fetched withdrawal array, or introduce a single `getAllAssetsUnstaking()` function that iterates assets and withdrawals in a single pass per NDC.
3. **Add an explicit cap on supported assets** analogous to `maxNodeDelegatorLimit`, so the O(N × M × W) bound is enforced at the protocol level.

---

### Proof of Concept

With 5 supported assets, 10 NDCs, and 80 queued withdrawals per NDC (each with 3 strategies):

- `getQueuedWithdrawals()` is called: 5 × 10 = **50 times**
- `sharesToUnderlyingView()` is called: 5 × 10 × 80 × 3 = **12,000 times**
- Estimated gas for external calls alone: 12,050 × ~3,000 gas = **~36M gas** — exceeding the 30M block gas limit

Any caller invoking `updateRSETHPrice()` (which is unrestricted) under these conditions will receive an out-of-gas revert, permanently blocking price updates until the number of queued withdrawals is reduced by operators completing unstaking. [5](#0-4) [6](#0-5) [3](#0-2)

    function getAssetUnstaking(address asset) external view returns (uint256 amount) {
        (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
            _getDelegationManager().getQueuedWithdrawals(address(this));

        for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
            IDelegationManager.Withdrawal memory withdrawal = queuedWithdrawals[withdrawalIndex];

            for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
                IStrategy strategy = withdrawal.strategies[strategyIndex];

                address strategyAsset = address(strategy) == address(lrtConfig.beaconChainETHStrategy())
                    ? LRTConstants.ETH_TOKEN
                    : address(strategy.underlyingToken());

                if (strategyAsset != asset) continue;

                uint256 sharesToUnstake = withdrawalShares[withdrawalIndex][strategyIndex];
                amount += strategyAsset == LRTConstants.ETH_TOKEN
                    ? sharesToUnstake
                    : strategy.sharesToUnderlyingView(sharesToUnstake);
            }
        }
    }
