### Title
Burn-on-Transfer Token in `RsETHTokenWrapper._deposit()` Causes wrsETH Withdrawal Failure and Permanent Fund Freezing - (File: contracts/L2/RsETHTokenWrapper.sol)

### Summary
`RsETHTokenWrapper._deposit()` mints wrsETH 1:1 with the caller-supplied `_amount`, but if the deposited `_asset` is a burn-on-transfer token, the contract receives fewer tokens than `_amount`. The paired `_withdraw()` function then attempts to transfer the full `_amount` back to the user, which reverts because the contract holds less than it minted. The last depositors are permanently unable to redeem their wrsETH.

### Finding Description
`_deposit()` performs a `safeTransferFrom` for `_amount` and immediately mints exactly `_amount` wrsETH:

```solidity
// contracts/L2/RsETHTokenWrapper.sol lines 134-141
function _deposit(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    ERC20Upgradeable(_asset).safeTransferFrom(msg.sender, address(this), _amount);
    _mint(_to, _amount);                          // mints _amount, not actual received
    emit Deposit(_asset, msg.sender, _to, _amount);
}
``` [1](#0-0) 

If `_asset` burns a fraction on transfer, the contract receives `_amount - burn_fee` but records a liability of `_amount` wrsETH. The withdrawal path burns the wrsETH and then tries to send the full `_amount` back:

```solidity
// contracts/L2/RsETHTokenWrapper.sol lines 120-128
function _withdraw(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    _burn(msg.sender, _amount);
    ERC20Upgradeable(_asset).safeTransfer(_to, _amount);   // reverts if balance < _amount
    emit Withdraw(_asset, msg.sender, _to, _amount);
}
``` [2](#0-1) 

No balance-before/balance-after check exists anywhere in the deposit path to detect the shortfall.

The public entry points `deposit()` and `depositTo()` are callable by any address without restriction: [3](#0-2) 

The `maxAmountToDepositBridgerAsset()` view also becomes incorrect: it computes `totalSupply() - balanceOf(asset)`, which overstates the gap because `totalSupply()` is inflated relative to actual holdings, misleading the `RSETHPoolV3.swapAssetToPremintedRsETH()` capacity check. [4](#0-3) 

### Impact Explanation
Every deposit of a burn-on-transfer token creates a deficit: the contract holds `_amount × (1 - burn_rate)` tokens but owes `_amount` wrsETH. Withdrawals succeed in FIFO order until the deficit is reached; the remaining wrsETH holders cannot redeem their tokens. Their wrsETH is permanently frozen because no code path allows the contract to recover the missing tokens. This is **Critical – permanent freezing of funds**.

### Likelihood Explanation
The `allowedTokens` mapping is populated by the admin via `addAllowedToken()` (TIMELOCK_ROLE). The wrapper is designed for "alternative rsETH tokens" from various bridges; if any such bridge-wrapped token implements a burn-on-transfer mechanism (e.g., a fee-on-transfer bridge token or a rebasing token that rounds down), the vulnerability is live. The admin may not audit every token's transfer implementation before whitelisting. Likelihood is **Low** given the current token set, but the code provides zero protection if such a token is ever added.

### Recommendation
Measure the actual amount received by comparing balances before and after the transfer, and mint wrsETH only for the received amount:

```solidity
function _deposit(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    uint256 balanceBefore = ERC20Upgradeable(_asset).balanceOf(address(this));
    ERC20Upgradeable(_asset).safeTransferFrom(msg.sender, address(this), _amount);
    uint256 received = ERC20Upgradeable(_asset).balanceOf(address(this)) - balanceBefore;
    _mint(_to, received);
    emit Deposit(_asset, msg.sender, _to, received);
}
```

### Proof of Concept
1. Admin calls `addAllowedToken(burnToken)` where `burnToken` charges a 1 % burn on every transfer.
2. Alice calls `deposit(burnToken, 100e18)`. Contract receives 99e18 tokens; mints 100e18 wrsETH to Alice.
3. Bob calls `deposit(burnToken, 100e18)`. Contract receives 99e18 tokens; mints 100e18 wrsETH to Bob.
4. Contract state: holds 198e18 `burnToken`, has 200e18 wrsETH outstanding.
5. Alice calls `withdraw(burnToken, 100e18)`. Burns 100e18 wrsETH, transfers 100e18 `burnToken` → succeeds. Contract now holds 98e18 tokens, 100e18 wrsETH outstanding.
6. Bob calls `withdraw(burnToken, 100e18)`. Burns 100e18 wrsETH, attempts `safeTransfer(bob, 100e18)` → **reverts** (only 98e18 available).
7. Bob's 100e18 wrsETH is permanently unredeemable; his funds are frozen.
