### Title
Fee-on-Transfer Token Deposit Mints Excess wrsETH, Causing Pool Insolvency - (File: contracts/pools/RSETHPoolV3.sol)

### Summary
All L2 pool contracts compute the wrsETH/rsETH amount to mint using the caller-supplied `amount` parameter rather than the actual tokens received after a `safeTransferFrom`. For fee-on-transfer tokens, the pool receives fewer tokens than `amount`, yet mints wrsETH calculated on the full `amount`. This permanently under-collateralises the pool and causes insolvency when the bridger attempts to move assets to L1.

### Finding Description

In `RSETHPoolV3.sol`, `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV3WithNativeChainBridge.sol`, and `RSETHPoolNoWrapper.sol`, the ERC-20 token deposit path follows this sequence:

```
IERC20(token).safeTransferFrom(msg.sender, address(this), amount);   // (1)
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token); // (2)
feeEarnedInToken[token] += fee;                                        // (3)
wrsETH.mint(msg.sender, rsETHAmount);                                  // (4)
```

Step (1) transfers `amount` from the caller. For a fee-on-transfer token the contract actually receives `amount - transferFee`. Step (2) computes `rsETHAmount` using the original `amount`, not the actual balance delta. Step (4) mints wrsETH proportional to `amount`.

The result is that the pool's token balance is `amount - transferFee`, but the wrsETH outstanding is backed by `amount`. Every deposit with a fee-on-transfer token widens this gap.

Additionally, `feeEarnedInToken[token]` in step (3) is inflated (computed from `amount`), so `getTokenBalanceMinusFees` — which is used as the bridgeable balance — can underflow or return an incorrect value.

The same pattern is present identically in:
- `RSETHPoolV3ExternalBridge.deposit(address,uint256,string)` [1](#0-0) 
- `RSETHPoolV3WithNativeChainBridge.deposit(address,uint256,string)` [2](#0-1) 
- `RSETHPoolNoWrapper.deposit(address,uint256,string)` [3](#0-2) 

### Impact Explanation

**Impact: Critical — Protocol insolvency.**

The pool holds fewer tokens than the wrsETH it has minted. When the bridger calls `bridgeTokens` or `moveAssetsForBridging`, it can only move the actual (reduced) token balance to L1. The L1 vault therefore receives less collateral than the wrsETH outstanding on L2. Holders of wrsETH cannot all redeem at par; the last redeemers receive nothing. This is a direct, permanent loss of user funds across the entire pool.

The `getTokenBalanceMinusFees` function compounds the issue: because `feeEarnedInToken[token]` is over-counted, the function can revert on underflow or report a falsely low bridgeable balance, potentially freezing the remaining (real) token balance in the pool. [4](#0-3) [5](#0-4) 

### Likelihood Explanation

**Likelihood: Low.**

The `addSupportedToken` function is gated behind `TIMELOCK_ROLE`, so a fee-on-transfer token can only be added through a governance/timelock action. The currently deployed supported tokens (e.g. wstETH) are not fee-on-transfer. However, the protocol is explicitly designed to be extensible to new LSTs, and no code-level guard prevents a fee-on-transfer token from being added. Once such a token is added, any unprivileged depositor can trigger the insolvency by simply calling `deposit`. [6](#0-5) 

### Recommendation

Measure the actual balance delta around the `safeTransferFrom` call and use that as the basis for wrsETH minting:

```solidity
uint256 balanceBefore = IERC20(token).balanceOf(address(this));
IERC20(token).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(token).balanceOf(address(this)) - balanceBefore;

(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(actualReceived, token);
feeEarnedInToken[token] += fee;
wrsETH.mint(msg.sender, rsETHAmount);
```

This fix must be applied consistently across all four pool contracts. Additionally, document that rebasing tokens are not supported, as their dynamic balance changes create a similar mismatch between minted wrsETH and actual pool holdings.

### Proof of Concept

1. Admin adds a fee-on-transfer token `FOT` (1% fee) as a supported token via `addSupportedToken`.
2. Attacker calls `RSETHPoolV3.deposit(FOT, 1000e18, "")`.
3. `safeTransferFrom` moves `1000e18` from attacker; pool receives `990e18` (1% fee taken by token).
4. `viewSwapRsETHAmountAndFee(1000e18, FOT)` computes `rsETHAmount` based on `1000e18`.
5. `wrsETH.mint(attacker, rsETHAmount)` — attacker receives wrsETH backed by `1000e18` worth of FOT, but pool only holds `990e18`.
6. `feeEarnedInToken[FOT]` is inflated by the protocol fee computed on `1000e18`.
7. Repeated deposits widen the gap. When the bridger calls `bridgeTokens(FOT)`, it bridges only `990e18 * N` to L1 while `1000e18 * N` worth of wrsETH is outstanding.
8. The last wrsETH holders cannot redeem; funds are permanently lost. [7](#0-6) [8](#0-7)

```text
    function viewSwapRsETHAmountAndFee(
        uint256 amount,
        address token
    )
        public
        view
        onlySupportedToken(token)
        returns (uint256 rsETHAmount, uint256 fee)
    {
        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // rate of token in ETH
        uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
    }
```
