### Title
Mismatch Between `minAmountToDeposit` and `minRsEthAmountToWithdraw` Leaves Users Unable to Withdraw Small rsETH Positions - (File: contracts/LRTDepositPool.sol / contracts/LRTWithdrawalManager.sol)

### Summary

`LRTDepositPool` enforces a minimum deposit in **asset units** (`minAmountToDeposit`), while `LRTWithdrawalManager` enforces a minimum withdrawal in **rsETH units** (`minRsEthAmountToWithdraw[asset]`). These two thresholds are independently configurable with no cross-validation. As rsETH naturally appreciates over time (it is a yield-bearing token), the same asset deposit produces progressively fewer rsETH, eventually falling below `minRsEthAmountToWithdraw`. Users who deposit exactly `minAmountToDeposit` may receive rsETH that is permanently below the withdrawal minimum, leaving them unable to use either `initiateWithdrawal` or `instantWithdrawal`.

### Finding Description

In `LRTDepositPool._beforeDeposit`, the deposit check enforces a minimum in asset units:

```solidity
if (depositAmount == 0 || depositAmount < minAmountToDeposit) {
    revert InvalidAmountToDeposit();
}
```

The rsETH minted is computed as:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [1](#0-0) [2](#0-1) 

In `LRTWithdrawalManager.initiateWithdrawal` (and `instantWithdrawal`), the withdrawal check enforces a minimum in **rsETH units**:

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
``` [3](#0-2) [4](#0-3) 

The two setter functions have no cross-validation: [5](#0-4) [6](#0-5) 

Because `lrtOracle.rsETHPrice()` increases monotonically as yield accrues to the protocol, the rsETH minted per unit of deposited asset decreases over time. Even if both thresholds are set consistently at deployment, the natural appreciation of rsETH causes the minimum deposit (in rsETH terms) to drift below `minRsEthAmountToWithdraw[asset]` without any admin action.

### Impact Explanation

A user who deposits exactly `minAmountToDeposit` of an asset receives `rsethAmountToMint` rsETH. If `rsethAmountToMint < minRsEthAmountToWithdraw[asset]`, both `initiateWithdrawal` and `instantWithdrawal` revert with `InvalidAmountToWithdraw`. The user cannot redeem their rsETH through the protocol's own withdrawal system. They must rely on secondary market liquidity, which may not exist on all supported L2 chains. The protocol fails to deliver its promised withdrawal functionality for these users.

**Impact**: Low — Contract fails to deliver promised returns, but does not directly lose value.

### Likelihood Explanation

This condition arises naturally without any admin error. As rsETH accrues staking yield, `rsETHPrice` increases, so the same `minAmountToDeposit` produces fewer rsETH over time. On a live protocol where `minAmountToDeposit` and `minRsEthAmountToWithdraw` are set once and not continuously re-synchronized, this drift is expected. Any user depositing near the minimum threshold is at risk.

### Recommendation

Add a cross-validation check in `setMinAmountToDeposit` and `setMinRsEthAmountToWithdraw` to ensure that a deposit of exactly `minAmountToDeposit` always produces at least `minRsEthAmountToWithdraw[asset]` rsETH at the current oracle rate. Alternatively, express `minRsEthAmountToWithdraw` in asset units (consistent with the deposit minimum) and convert at withdrawal time, or enforce that `minRsEthAmountToWithdraw` is always set to a value strictly less than the rsETH equivalent of `minAmountToDeposit`.

### Proof of Concept

1. At deployment: `rsETHPrice = 1e18` (1 rsETH = 1 ETH), `minAmountToDeposit = 1e15` (0.001 ETH), `minRsEthAmountToWithdraw[ETH] = 1e15` (0.001 rsETH). Consistent.
2. After yield accrual: `rsETHPrice = 1.2e18` (rsETH appreciated 20%).
3. User calls `depositETH` with `msg.value = 1e15` (exactly `minAmountToDeposit`). Passes the deposit check.
4. `rsethAmountToMint = (1e15 * 1e18) / 1.2e18 = 8.33e14` rsETH minted to user.
5. User calls `initiateWithdrawal(ETH, 8.33e14, "")`. Check: `8.33e14 < 1e15` → reverts `InvalidAmountToWithdraw`.
6. User calls `instantWithdrawal(ETH, 8.33e14, "")`. Same check → same revert.
7. User's rsETH position is irrecoverable through the protocol's withdrawal system.
