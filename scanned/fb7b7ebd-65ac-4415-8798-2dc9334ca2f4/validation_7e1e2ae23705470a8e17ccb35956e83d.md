### Title
Missing Staleness and Validity Checks on Chainlink `latestRoundData()` Return Values - (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `price`, performing no staleness check, no incomplete-round check, and no non-negative price check. A stale or zero price propagates directly into `LRTOracle._updateRsETHPrice()`, corrupting the rsETH/ETH exchange rate used for every deposit and withdrawal in the protocol.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the LST/ETH price as follows:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

All five return values from `latestRoundData()` — `roundId`, `answer`, `startedAt`, `updatedAt`, and `answeredInRound` — are available, but only `price` (`answer`) is used. The following critical checks are absent:

- `answeredInRound >= roundId` — detects a stale round where the answer was carried over from a prior round.
- `updatedAt != 0` — detects an incomplete/unstarted round.
- `updatedAt >= block.timestamp - MAX_DELAY` — detects a heartbeat miss (feed not updated within its expected interval).
- `price > 0` — detects a zero or negative answer, which Chainlink can return silently on certain failure modes.

This price is consumed by `LRTOracle._getTotalEthInProtocol()`, which iterates over every supported LST asset and multiplies its total deposited amount by the oracle price:

```solidity
uint256 assetER = getAssetPrice(asset);
uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
``` [2](#0-1) 

The result feeds directly into `_updateRsETHPrice()`, which sets the global `rsETHPrice` used to compute how much rsETH to mint for every depositor. [3](#0-2) 

A secondary instance exists in `RSETHPriceFeed.latestRoundData()`, which wraps the ETH/USD Chainlink feed and multiplies it by the rsETH/ETH rate, but performs no validation on the ETH/USD feed's return values before forwarding them to external consumers:

```solidity
(roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.latestRoundData();
answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
``` [4](#0-3) 

Note that `ChainlinkOracleForRSETHPoolCollateral` correctly implements all three checks (`answeredInRound < roundID`, `timestamp == 0`, `ethPrice <= 0`), confirming the team is aware of the pattern but did not apply it consistently. [5](#0-4) 

---

### Impact Explanation

**Primary — `ChainlinkPriceOracle`:**

- If a feed returns a stale price that is artificially low (e.g., a missed heartbeat during a price spike), `totalETHInProtocol` is understated. `_updateRsETHPrice()` computes a `newRsETHPrice` below `highestRsethPrice`, and if the deviation exceeds `pricePercentageLimit`, the downside-protection logic automatically pauses `LRTDepositPool` and `LRTWithdrawalManager`, freezing all user deposits and withdrawals. [6](#0-5) 

- If a feed returns a stale price that is artificially high, `totalETHInProtocol` is overstated, inflating `rsETHPrice`. Depositors who call `depositAsset` immediately after receive more rsETH than the assets are worth, diluting existing holders.

**Secondary — `RSETHPriceFeed`:** External DeFi integrations (Curve pools, money markets) consuming this feed receive a stale rsETH/USD price, enabling mispriced trades or liquidations against rsETH collateral.

Impact classification: **Medium — Temporary freezing of funds** (via spurious downside-protection pause) and **Medium — Oracle/rate abuse** (inflated rsETH minting).

---

### Likelihood Explanation

Chainlink feeds have documented heartbeat intervals (e.g., 1 hour for ETH/stETH). During periods of network congestion, oracle operator issues, or L2 sequencer downtime, feeds can go stale within their heartbeat window while still returning the last cached answer with a stale `updatedAt`. This is a well-known, historically observed failure mode. `updateRSETHPrice()` is a public function callable by any address, so an attacker can trigger the price update at the exact moment a stale price is live. [7](#0-6) 

---

### Recommendation

Apply the same validation pattern already used in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

require(answeredInRound >= roundId, "Stale price");
require(updatedAt != 0, "Incomplete round");
require(price > 0, "Invalid price");
require(block.timestamp - updatedAt <= MAX_STALENESS, "Price too old");
```

Apply the same pattern inside `RSETHPriceFeed.latestRoundData()` before using the ETH/USD answer.

---

### Proof of Concept

1. A supported LST asset (e.g., stETH) has its Chainlink feed go stale — `updatedAt` is 2 hours old but `answeredInRound < roundId`, meaning the answer is carried from a prior round at a price 5% below current market.
2. Any address calls `LRTOracle.updateRSETHPrice()` (public, no access control).
3. `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)`, which returns the stale low price with no revert.
4. `totalETHInProtocol` is understated by ~5% of the stETH TVL.
5. `newRsETHPrice` falls below `highestRsethPrice` by more than `pricePercentageLimit`.
6. `_updateRsETHPrice()` calls `lrtDepositPool.pause()` and `withdrawalManager.pause()`, freezing all user deposits and withdrawals until an admin manually unpauses. [8](#0-7)
