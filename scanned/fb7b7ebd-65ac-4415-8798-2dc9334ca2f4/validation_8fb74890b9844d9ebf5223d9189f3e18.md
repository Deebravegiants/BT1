### Title
Withdrawal Completion Lacks Receiver Parameter Causing Permanent Fund Freeze - (File: contracts/LRTWithdrawalManager.sol)

### Summary
`LRTWithdrawalManager.initiateWithdrawal()` does not accept a `receiver` parameter. The `WithdrawalRequest` struct stores no recipient field, and `_processWithdrawalCompletion()` always transfers assets to the `user` address. Because rsETH is burned in a prior operator-controlled `unlockQueue()` transaction, any subsequent failure of `_transferAsset(asset, user, ...)` in `completeWithdrawal()` permanently freezes the user's underlying assets inside the contract with no recovery path.

### Finding Description

The `WithdrawalRequest` struct contains no `receiver` field:

```solidity
struct WithdrawalRequest {
    uint256 rsETHUnstaked;
    uint256 expectedAssetAmount;
    uint256 withdrawalStartBlock;
}
```

`initiateWithdrawal()` maps the request exclusively to `msg.sender`:

```solidity
userAssociatedNonces[asset][msg.sender].pushBack(nextUnusedNonce_);
```

`_processWithdrawalCompletion()` always sends assets to the `user` parameter (which is always `msg.sender` from `completeWithdrawal`, or the stored `user` from `completeWithdrawalForUser`):

```solidity
_transferAsset(asset, user, request.expectedAssetAmount);
```

The critical two-phase lifecycle is:

1. **`initiateWithdrawal()`** — rsETH is transferred from the user to the contract (`safeTransferFrom(msg.sender, address(this), rsETHUnstaked)`).
2. **`unlockQueue()`** (operator-only, separate tx) — rsETH is **burned** from the contract (`burnFrom(address(this), rsETHBurned)`) and assets are pulled from `LRTUnstakingVault` into `LRTWithdrawalManager`.
3. **`completeWithdrawal()`** — assets are transferred to `user`.

If step 3 reverts (e.g., ETH transfer to a contract that rejects ETH via `call{value: amount}("")`, or an LST token with a transfer restriction on the recipient), the entire transaction reverts. The withdrawal request remains in the queue, but the rsETH burned in step 2 is **irrecoverable**. The assets are permanently locked inside `LRTWithdrawalManager`.

There is no admin function to cancel a stuck withdrawal, no mechanism to redirect assets to an alternative address, and no emergency recovery path for individual user withdrawals. `completeWithdrawalForUser()` (operator-callable) still sends to `user`, providing no relief.

Additionally, `unlockedWithdrawalsCount[asset]` is only decremented on successful completion. A permanently stuck withdrawal keeps this counter non-zero, blocking `sweepRemainingAssets()` for the affected asset indefinitely.

### Impact Explanation

**Permanent freezing of funds.** After `unlockQueue()` burns the user's rsETH and moves the corresponding LST/ETH into `LRTWithdrawalManager`, a user whose address cannot receive the asset loses both their rsETH (burned) and their underlying asset (stuck in the contract). There is no on-chain recovery mechanism. `sweepRemainingAssets()` is also blocked for the asset as long as `unlockedWithdrawalsCount[asset] > 0`.

### Likelihood Explanation

Smart contract users (multisigs, DeFi aggregators, vaults) are common depositors in LRT protocols. A contract that initiates a withdrawal for ETH but whose `receive()` function is later removed, upgraded to revert, or was never present (e.g., a proxy whose implementation changes) will trigger this permanently. For LST assets, any token-level transfer restriction (e.g., ETHx operator-level controls) applied to the recipient after `initiateWithdrawal()` but before `completeWithdrawal()` produces the same outcome. The two-phase design with an operator-controlled `unlockQueue()` step creates a meaningful window between rsETH burn and asset delivery.

### Recommendation

1. Add a `receiver` parameter to `initiateWithdrawal()` and store it in `WithdrawalRequest`:

```solidity
struct WithdrawalRequest {
    uint256 rsETHUnstaked;
    uint256 expectedAssetAmount;
    uint256 withdrawalStartBlock;
    address receiver; // added
}
```

2. In `_processWithdrawalCompletion()`, transfer to `request.receiver` instead of `user`.

3. Add an admin/operator function to forcibly cancel a stuck unlocked withdrawal, returning the committed asset amount to a safe address (e.g., back to the unstaking vault or treasury) and decrementing `unlockedWithdrawalsCount[asset]`.

### Proof of Concept

1. A smart contract `VaultProxy` calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. rsETH is transferred to `LRTWithdrawalManager`. [1](#0-0) 

2. Operator calls `unlockQueue(ETH_TOKEN, ...)`. rsETH is burned from the contract; ETH is moved from `LRTUnstakingVault` into `LRTWithdrawalManager`. `unlockedWithdrawalsCount[ETH_TOKEN]++`. [2](#0-1) 

3. `VaultProxy` is upgraded; its new implementation has no `receive()` function.

4. `VaultProxy` calls `completeWithdrawal(ETH_TOKEN, "")`. `_processWithdrawalCompletion` reaches `_transferAsset(ETH_TOKEN, VaultProxy, amount)`, which executes `payable(VaultProxy).call{value: amount}("")`. This reverts. The entire transaction reverts. [3](#0-2) 

5. Operator calls `completeWithdrawalForUser(ETH_TOKEN, VaultProxy, "")`. Same revert. [4](#0-3) 

6. `VaultProxy`'s rsETH is permanently burned. The ETH is permanently locked in `LRTWithdrawalManager`. `unlockedWithdrawalsCount[ETH_TOKEN]` remains non-zero, blocking `sweepRemainingAssets(ETH_TOKEN)`. [5](#0-4)
