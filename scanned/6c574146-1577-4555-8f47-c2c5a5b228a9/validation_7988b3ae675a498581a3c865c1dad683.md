### Title
Missing Staleness Validation in `ChainlinkPriceOracle.getAssetPrice()` Enables Oracle Price Abuse to Steal Value from rsETH Holders - (`contracts/oracles/ChainlinkPriceOracle.sol`)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards all staleness-related return values (`updatedAt`, `answeredInRound`). There is no check that the price was updated within the feed's heartbeat window, and no check that `answeredInRound >= roundId`. A stale Chainlink price — whether caused by normal deviation-threshold behavior (up to 2% for rETH/ETH) or by a feed outage — is consumed as if it were fresh. Because this oracle feeds directly into `LRTDepositPool.getRsETHAmountToMint()` and `LRTOracle._getTotalEthInProtocol()`, an attacker can deposit an LST whose on-chain price is stale-high relative to the true market price and receive more rsETH than the fair value of their deposit, diluting all existing rsETH holders.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the Chainlink price as follows:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

The five return values of `latestRoundData()` are `(roundId, answer, startedAt, updatedAt, answeredInRound)`. The code binds only `price` (the second slot) and discards the rest. Neither `updatedAt` (the timestamp of the last price update) nor `answeredInRound` (the round in which the answer was computed) is ever inspected. There is no comparison of `updatedAt` against `block.timestamp` and no check that `answeredInRound >= roundId`.

This oracle is the sole price source used by `LRTOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) public view onlySupportedOracle(asset) returns (uint256) {
    return IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset);
}
``` [2](#0-1) 

`LRTOracle._getTotalEthInProtocol()` calls `getAssetPrice()` for every supported LST to compute the total ETH value backing rsETH:

```solidity
uint256 assetER = getAssetPrice(asset);
uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
``` [3](#0-2) 

`LRTDepositPool.getRsETHAmountToMint()` uses both `getAssetPrice()` and the stored `rsETHPrice` (which is itself derived from the same stale prices) to determine how many rsETH tokens to mint per unit of deposited LST:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [4](#0-3) 

This is called unconditionally inside `_beforeDeposit()`, which is the gate for both `depositAsset()` and `depositETH()`:

```solidity
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
``` [5](#0-4) 

By contrast, the newer `ChainlinkOracleForRSETHPoolCollateral` (used for L2 pool collateral tokens) does perform partial staleness checks (`answeredInRound < roundID`, `timestamp == 0`), but still omits the time-based heartbeat check:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
``` [6](#0-5) 

The primary L1 oracle `ChainlinkPriceOracle` has **none** of these checks.

---

### Impact Explanation

When a Chainlink LST/ETH feed is stale and its last reported price is above the current true market price (which is normal behavior within the deviation threshold — up to 2% for rETH/ETH, 1% for cbETH/ETH, 0.5% for stETH/ETH — or worse during a feed outage), an attacker deposits that LST and receives rsETH computed at the inflated stale price. The rsETH minted exceeds the true ETH value contributed. When `updateRSETHPrice()` is next called with fresh prices, the rsETH/ETH rate drops, diluting all existing holders. The attacker can exit (via withdrawal or secondary market) before the price corrects, extracting value from the pool.

The impact is **theft of yield/value from existing rsETH holders** (High), escalating to **protocol insolvency** (Critical) if the feed goes fully stale for an extended period and large deposits are made at the inflated price.

---

### Likelihood Explanation

Chainlink LST/ETH feeds have 24-hour heartbeats and deviation thresholds of 0.5%–2%. Within any 24-hour window the on-chain price can legitimately lag the true price by up to the deviation threshold without triggering an update. This is a **constant, always-present** condition, not a rare edge case. An attacker needs only to monitor the on-chain price vs. the spot price and deposit when the spread exceeds the fee cost. No privileged access is required; `depositAsset()` is a public function callable by any address.

---

### Recommendation

Add a time-based staleness check in `ChainlinkPriceOracle.getAssetPrice()` using a per-feed configurable heartbeat:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

if (answeredInRound < roundId) revert StalePrice();
if (block.timestamp - updatedAt > heartbeat[asset]) revert StalePrice();
if (price <= 0) revert InvalidPrice();
```

Store a `heartbeat` mapping (e.g., 86 400 s for 24-hour feeds) set alongside `assetPriceFeed` in `updatePriceFeedFor()`. Apply the same fix to `ChainlinkOracleForRSETHPoolCollateral.getRate()`.

---

### Proof of Concept

1. At time T, the rETH/ETH Chainlink feed last updated at price `P_stale = 1.09149e18` (true market price has since dropped to `P_true = 1.070e18`, a ~2% gap within the deviation threshold — no on-chain update triggered).
2. Attacker calls `LRTDepositPool.depositAsset(rETH, 1000e18, 0, "")`.
3. `getRsETHAmountToMint` computes: `rsethToMint = 1000e18 * 1.09149e18 / rsETHPrice`. Because `rsETHPrice` was last updated using the same stale feed, the numerator is inflated relative to the true ETH value contributed.
4. Attacker receives rsETH representing ~1091.49 ETH worth of claims, having deposited rETH worth only ~1070 ETH at true market price — a ~2% gain at the expense of existing holders.
5. After `updateRSETHPrice()` is called with fresh prices, the rsETH/ETH rate decreases, and all prior holders are diluted by the attacker's over-minted rsETH.

The attack is amplified at scale: with 10 000 rETH deposited at a 2% stale premium, the attacker extracts ~200 ETH of value from the pool in a single transaction.
