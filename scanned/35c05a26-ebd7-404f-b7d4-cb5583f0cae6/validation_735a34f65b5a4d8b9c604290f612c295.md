### Title
rsETH Price Inflation via Direct ETH Donation Enables First-Depositor Share Theft — (`contracts/LRTDepositPool.sol` / `contracts/LRTOracle.sol`)

### Summary

An attacker can manipulate the stored `rsETHPrice` in `LRTOracle` by donating ETH directly to `LRTDepositPool` (which is counted in `totalETHInProtocol`) and then calling the public `updateRSETHPrice()`. When `pricePercentageLimit` is zero (the default initial state), there is no cap on the price increase. Subsequent depositors who pass `minRSETHAmountExpected = 0` receive zero rsETH while their ETH is absorbed into the pool, all of which is redeemable by the attacker's single wei of rsETH.

### Finding Description

**Root cause 1 — unrestricted ETH donation path:**
`LRTDepositPool` exposes a bare `receive()` that accepts ETH from any sender. [1](#0-0) 

`getETHDistributionData()` counts `address(this).balance` as part of the protocol's total ETH, so any donated ETH immediately inflates `totalETHInProtocol`. [2](#0-1) 

**Root cause 2 — public price update with no default guard:**
`updateRSETHPrice()` is callable by anyone. [3](#0-2) 

The price-increase guard is only active when `pricePercentageLimit > 0`. Because `pricePercentageLimit` is never set in `initialize()`, it defaults to zero, making the guard a no-op for any caller. [4](#0-3) 

**Root cause 3 — rsETH minting uses the stored (manipulable) price:**

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [5](#0-4) 

If `rsETHPrice` has been inflated, the division yields zero for any deposit smaller than the inflated price, and the zero-check `rsethAmountToMint < minRSETHAmountExpected` passes when the caller supplies `minRSETHAmountExpected = 0`. [6](#0-5) 

**Root cause 4 — zero-supply bootstrap sets price to 1 ether:** [7](#0-6) 

This guarantees the attacker can always obtain exactly 1 wei rsETH for 1 wei ETH before any manipulation.

### Impact Explanation

A victim who deposits Y ETH after the price has been inflated to `(1 + X) * 1e18` receives `floor(Y / (1+X))` rsETH. If `Y ≤ X`, this rounds to zero. The victim's Y ETH is permanently absorbed into the pool. The attacker's single wei of rsETH now represents the entire pool (`1 + X + Y` ETH), redeemable through the withdrawal queue. This is direct theft of user funds — **Critical**.

### Likelihood Explanation

- `pricePercentageLimit` is zero by default (no admin action required to leave the guard disabled).
- `updateRSETHPrice()` is permissionlessly callable.
- `minRSETHAmountExpected = 0` is a common default in integrations and scripts.
- No front-running is required; the attacker sets the trap and waits for any depositor.
- The attacker's donated ETH is locked in the protocol, but is recoverable via the withdrawal queue once the attacker redeems their rsETH.

### Recommendation

1. **Set a non-zero `pricePercentageLimit` at initialization** (e.g., 5 % = `5e16`) so that `updateRSETHPrice()` reverts for non-managers on large price jumps.
2. **Restrict `updateRSETHPrice()` to authorized callers** (manager/keeper) or add a minimum-supply guard before allowing price updates.
3. **Reject zero `minRSETHAmountExpected`** in `_beforeDeposit`, or compute a protocol-enforced minimum based on the current price and a slippage tolerance.
4. **Seed the pool during deployment** (mint a non-trivial amount of rsETH to `address(0)`) so the first-depositor attack surface is eliminated.

### Proof of Concept

```
// Initial state: rsETHPrice = 1e18, rsethSupply = 0, pricePercentageLimit = 0

// Step 1: Attacker deposits 1 wei ETH → gets 1 wei rsETH
lrtDepositPool.depositETH{value: 1}(0, "");
// rsethSupply = 1, totalETH = 1

// Step 2: Attacker donates X = 10 ether directly to the deposit pool
(bool ok,) = address(lrtDepositPool).call{value: 10 ether}("");

// Step 3: Attacker triggers price update (public, no guard when pricePercentageLimit == 0)
lrtOracle.updateRSETHPrice();
// newRsETHPrice = (1 + 10e18) * 1e18 / 1 ≈ 10e18 * 1e18
// rsETHPrice is now ~10e18 * 1e18

// Step 4: Victim deposits 5 ether with minRSETHAmountExpected = 0
lrtDepositPool.depositETH{value: 5 ether}(0, "");
// rsethAmountToMint = (5e18 * 1e18) / (10e18 * 1e18) = 0  ← rounds to zero
// Victim receives 0 rsETH; 5 ETH absorbed into pool

// Step 5: Attacker's 1 wei rsETH now redeemable for ~15 ETH via withdrawal queue
// Net attacker gain: 5 ETH (victim's deposit) - 10 ETH (donation, recovered later) = -5 ETH upfront
// but attacker recovers all 15 ETH on redemption → net profit = 5 ETH
``` [8](#0-7) [9](#0-8) [10](#0-9)
