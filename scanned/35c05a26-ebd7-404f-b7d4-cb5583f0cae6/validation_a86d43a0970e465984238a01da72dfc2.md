### Title
`checkDailyMintLimit` Active Before `maxMintAmountPerDay` Is Set Causes All Deposits to Revert on Chain Reorg - (File: contracts/RSETH.sol)

---

### Summary

`RSETH.sol` enforces a daily mint cap via the `checkDailyMintLimit` modifier, which is unconditionally applied to every `mint()` call. However, `maxMintAmountPerDay` is never set during `initialize()` or `reinitialize()` — it defaults to `0` and is configured in a separate, independent transaction via `setMaxMintAmountPerDay()`. When `maxMintAmountPerDay == 0`, the modifier's guard evaluates to `amount > 0`, which is always true, causing every mint to revert. A chain reorganisation that reorders user deposit transactions ahead of the `setMaxMintAmountPerDay()` call produces a window in which all deposits fail.

---

### Finding Description

`RSETH.initialize()` sets up the ERC-20 and pausable state but leaves `maxMintAmountPerDay` at its default value of `0`: [1](#0-0) 

`RSETH.reinitialize()` (the v2 upgrade initialiser) sets `periodStartTime` and `custodyAddress` but still does **not** touch `maxMintAmountPerDay`: [2](#0-1) 

`maxMintAmountPerDay` is only set by a standalone manager call: [3](#0-2) 

The `checkDailyMintLimit` modifier is unconditionally applied to every `mint()` invocation with no bypass flag: [4](#0-3) 

When `maxMintAmountPerDay == 0`, the condition on line 50 reduces to `currentPeriodMintedAmount + amount > 0`, which is always `true` for any non-zero `amount`, causing an unconditional revert with `DailyMintLimitExceeded`.

`mint()` is the terminal step of every user deposit through `LRTDepositPool`: [5](#0-4) 

Both `depositETH()` and `depositAsset()` call `_mintRsETH()` which calls `RSETH.mint()`: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

Any user deposit — ETH or LST — that reaches `RSETH.mint()` while `maxMintAmountPerDay == 0` reverts entirely. Because `depositAsset()` transfers the user's tokens before calling `_mintRsETH()`, a revert at the mint step rolls back the entire transaction, so no funds are permanently lost. The impact is **temporary freezing of user deposits**: no rsETH can be minted and no deposit can succeed until `setMaxMintAmountPerDay` is confirmed on-chain. This matches the **Medium — Temporary freezing of funds** impact category.

---

### Likelihood Explanation

The vulnerable window exists from the moment `RSETH` is initialised (or reinitialised) until `setMaxMintAmountPerDay()` is confirmed. On Ethereum mainnet this window is short, but on the L2 chains where the protocol also operates (Arbitrum, Optimism, etc.) chain reorganisations are more frequent. A reorg that reorders user deposit transactions ahead of the `setMaxMintAmountPerDay()` call — even by a single block — is sufficient to trigger the failure. No attacker action is required; the reorg alone causes the revert. The likelihood is **Low-Medium**.

---

### Recommendation

Set `maxMintAmountPerDay` atomically in the same transaction that activates the daily-limit enforcement. Concretely, add a `_maxMintAmountPerDay` parameter to `reinitialize()` and assign it there alongside `periodStartTime`, eliminating the gap between the constraint being active and its bound being non-zero:

```solidity
function reinitialize(
    uint256 _periodStartTime,
    address _custodyAddress,
    uint256 _maxMintAmountPerDay   // <-- add this
) external reinitializer(2) onlyLRTManager {
    ...
    require(_maxMintAmountPerDay > 0, "zero limit");
    maxMintAmountPerDay = _maxMintAmountPerDay;
    ...
}
```

Alternatively, add a zero-value guard in `checkDailyMintLimit` that skips enforcement when `maxMintAmountPerDay == 0` (treating zero as "uncapped"), consistent with how `remainingDailyMintLimit()` already handles the zero case: [8](#0-7) 

---

### Proof of Concept

**Normal flow (block N):**
1. Manager sends `setMaxMintAmountPerDay(1_000 ether)` → tx A, block N.
2. Alice sends `depositETH{value: 1 ether}(0, "")` → tx B, block N.
3. Tx A is mined first; `maxMintAmountPerDay = 1_000 ether`. Tx B succeeds; Alice receives rsETH.

**Chain-reorg flow (block N):**
1. Same two transactions land in the same block.
2. Reorg reorders tx B before tx A.
3. Tx B executes with `maxMintAmountPerDay == 0`; `checkDailyMintLimit` evaluates `0 + rsethAmountToMint > 0` → `true` → `revert DailyMintLimitExceeded(rsethAmountToMint, 0)`.
4. Alice's deposit reverts. All concurrent depositors (Carol, David, …) are equally affected.
5. Deposits remain frozen until tx A is re-included and confirmed.

    modifier checkDailyMintLimit(uint256 amount) {
        // Check if we need to reset the period if it has been more than 24 hours
        if (block.timestamp >= periodStartTime + 1 days) {
            currentPeriodMintedAmount = 0;
            periodStartTime = getCurrentPeriodStartTime();
        }

        // Check if minting would exceed the daily limit
        if (currentPeriodMintedAmount + amount > maxMintAmountPerDay) {
            revert DailyMintLimitExceeded(currentPeriodMintedAmount + amount, maxMintAmountPerDay);
        }

        currentPeriodMintedAmount += amount;
        _;
    }
