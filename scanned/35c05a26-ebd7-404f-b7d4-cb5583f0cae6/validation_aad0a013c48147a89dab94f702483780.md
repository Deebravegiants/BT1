### Title
`maxNumberOfWithdrawalsPerUser` Uninitialized to Zero Permanently Blocks Withdrawals Until Admin Acts - (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary
`KernelDepositPool.sol` allows users to stake KERNEL tokens immediately after deployment, but `maxNumberOfWithdrawalsPerUser` is never set in `initialize()` and defaults to `0`. Because `initiateWithdrawal()` guards with `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser`, and `0 >= 0` is always true, every call to `initiateWithdrawal()` reverts with `WithdrawalLimitReached` until the admin explicitly calls `setMaxNumberOfWithdrawalsPerUser()`. Any user who stakes before that admin call has their KERNEL tokens frozen in the contract.

### Finding Description
`KernelDepositPool` declares `maxNumberOfWithdrawalsPerUser` as a plain storage variable with no default: [1](#0-0) 

The `initialize()` function sets only `kernelToken`, `rewardsToken`, and the admin role — it never assigns `maxNumberOfWithdrawalsPerUser`: [2](#0-1) 

`stake()` imposes no precondition on `maxNumberOfWithdrawalsPerUser`, so users can deposit freely: [3](#0-2) 

`initiateWithdrawal()` then unconditionally evaluates: [4](#0-3) 

With `maxNumberOfWithdrawalsPerUser == 0`, the expression `userWithdrawalIds[msg.sender].length >= 0` is always `true` (any `uint256` satisfies `>= 0`), so the function always reverts. The only remedy is the admin calling `setMaxNumberOfWithdrawalsPerUser()`, which itself enforces `> 0`: [5](#0-4) 

Note also that `withdrawalDelay` is similarly uninitialized (defaults to `0`), meaning that once `maxNumberOfWithdrawalsPerUser` is fixed, withdrawals created before `setWithdrawalDelay()` is called carry `unlockTime = block.timestamp` and are immediately claimable — bypassing the intended delay protection: [6](#0-5) 

### Impact Explanation
**Medium — Temporary freezing of funds.** Every user who calls `stake()` before the admin calls `setMaxNumberOfWithdrawalsPerUser()` has their KERNEL tokens locked in the contract with no withdrawal path. The freeze persists until the admin acts. There is no loss of principal, but the contract fails to honour the withdrawal promise during this window.

### Likelihood Explanation
The contract is designed so that `notifyRewardAmount()` requires `totalKernelStaked > 0`, meaning users are explicitly expected to stake before the admin completes reward setup. This makes it highly likely that stakers will be present before all admin parameters are configured, including `maxNumberOfWithdrawalsPerUser`. The window between deployment and full admin configuration is a realistic, non-negligible period.

### Recommendation
Initialize both `maxNumberOfWithdrawalsPerUser` and `withdrawalDelay` to safe non-zero defaults inside `initialize()`, or add a guard in `stake()` that reverts if `maxNumberOfWithdrawalsPerUser == 0`, mirroring the pattern used in `notifyRewardAmount()` which already guards against `totalKernelStaked == 0`.

### Proof of Concept
1. Admin deploys and calls `initialize(admin, kernelToken, rewardToken)`. `maxNumberOfWithdrawalsPerUser` remains `0`.
2. User calls `stake(100e18)`. Succeeds — 100 KERNEL transferred to contract, `balanceOf[user] = 100e18`.
3. User calls `initiateWithdrawal(100e18)`.
   - Check: `userWithdrawalIds[user].length` = `0`; `maxNumberOfWithdrawalsPerUser` = `0`; `0 >= 0` → `true` → reverts `WithdrawalLimitReached`.
4. User's 100 KERNEL is frozen until admin calls `setMaxNumberOfWithdrawalsPerUser(N)`.
5. If admin also has not called `setWithdrawalDelay()`, any withdrawal initiated immediately after step 4 will have `unlockTime = block.timestamp`, making it claimable in the same block — bypassing the delay entirely.
