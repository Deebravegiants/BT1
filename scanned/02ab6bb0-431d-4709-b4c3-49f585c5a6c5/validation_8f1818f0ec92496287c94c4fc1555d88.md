### Title
`KernelDepositPool.claimWithdrawal` Restricts Claim to Withdrawal Owner Only, Blocking Approved Operators - (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary
`KernelDepositPool.claimWithdrawal` enforces a strict `withdrawal.user != msg.sender` check that prevents any approved operator or delegate from claiming a withdrawal on behalf of the original staker. This mirrors the root cause in the referenced report: functions that should logically permit approved parties are gated exclusively to the direct owner.

### Finding Description
In `KernelDepositPool.claimWithdrawal`, the authorization check is:

```solidity
if (withdrawal.user != msg.sender) {
    revert NotYourWithdrawal();
}
```

followed by:

```solidity
kernelToken.safeTransfer(msg.sender, withdrawal.amount);
``` [1](#0-0) 

The withdrawal record stores the initiating address as `withdrawal.user`:

```solidity
withdrawals[withdrawalId] = Withdrawal({
    user: msg.sender, amount: _amount, unlockTime: unlockTime, claimed: false, withdrawalId: withdrawalId
});
``` [2](#0-1) 

There is no mechanism for an approved operator, delegate, or keeper to call `claimWithdrawal` on behalf of `withdrawal.user`. The contract provides `stakeFor` (gated by `STAKE_FOR_ROLE`) to stake on behalf of another account, but no analogous `claimWithdrawalFor` path exists for the claim side. [3](#0-2) 

The same pattern appears in `KernelMerkleDistributor._processClaim`, which enforces `account != msg.sender` and reverts with `Unauthorized`, blocking any operator from claiming Merkle-distributed KERNEL on behalf of an account: [4](#0-3) 

### Impact Explanation
**Low. Contract fails to deliver promised returns, but doesn't lose value.**

Any user (EOA or smart contract) that has initiated a withdrawal cannot have an approved operator, keeper bot, or delegate complete the claim on their behalf. For smart contract stakers (e.g., vaults or multisigs that called `initiateWithdrawal` directly) that lack a generic `execute`/`call` function, the KERNEL tokens remain locked in the contract indefinitely with no recovery path, escalating to permanent fund freeze for those specific callers. For EOA users, the impact is limited to inability to automate or delegate the claim step.

### Likelihood Explanation
Medium. Any protocol-level vault, yield aggregator, or smart contract wallet that stakes KERNEL and initiates a withdrawal is directly affected. The `stakeFor` function explicitly anticipates non-EOA stakers (it is role-gated for exactly this use case), making it foreseeable that smart contract callers will also need to claim withdrawals. [5](#0-4) 

### Recommendation
Introduce a `claimWithdrawalFor(uint256 _withdrawalId, address _user)` function (or extend `claimWithdrawal`) that allows an approved operator to trigger the claim while still transferring tokens to `withdrawal.user`. Alternatively, maintain an `approvedClaimer` mapping (similar to EigenLayer's `claimerFor` pattern) that the withdrawal owner can set, and check it alongside `withdrawal.user == msg.sender`.

### Proof of Concept
1. Smart contract vault `V` is granted `STAKE_FOR_ROLE` and calls `stakeFor(V, amount)` to stake KERNEL on its own behalf.
2. `V` calls `initiateWithdrawal(amount)` — `withdrawal.user` is set to `V`.
3. After `withdrawalDelay`, keeper bot `K` attempts to call `claimWithdrawal(withdrawalId)`.
4. The check `withdrawal.user != msg.sender` evaluates to `V != K` → reverts with `NotYourWithdrawal`.
5. If `V` has no generic `execute` function, the KERNEL tokens are permanently locked. [6](#0-5)

    function claimWithdrawal(uint256 _withdrawalId) external nonReentrant {
        Withdrawal storage withdrawal = withdrawals[_withdrawalId];

        if (withdrawal.user == address(0)) {
            revert WithdrawalDoesNotExist();
        }

        if (withdrawal.user != msg.sender) {
            revert NotYourWithdrawal();
        }

        if (block.timestamp < withdrawal.unlockTime) {
            revert WithdrawalNotReady();
        }

        if (withdrawal.claimed) {
            revert WithdrawalAlreadyClaimed();
        }

        withdrawal.claimed = true;

        // Remove the withdrawal ID from the user's list of withdrawal IDs
        uint256[] storage userWithdrawalIdsArray = userWithdrawalIds[msg.sender];
        for (uint256 i = 0; i < userWithdrawalIdsArray.length; ++i) {
            if (userWithdrawalIdsArray[i] == _withdrawalId) {
                userWithdrawalIdsArray[i] = userWithdrawalIdsArray[userWithdrawalIdsArray.length - 1];
                userWithdrawalIdsArray.pop();
                break;
            }
        }

        // Transfer the KERNEL tokens from contract to user
        kernelToken.safeTransfer(msg.sender, withdrawal.amount);

        emit WithdrawalClaimed(msg.sender, withdrawal.amount, _withdrawalId);
```
