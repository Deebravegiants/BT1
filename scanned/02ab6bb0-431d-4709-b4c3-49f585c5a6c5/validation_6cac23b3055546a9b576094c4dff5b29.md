### Title
Incorrect ETH Deposit Limit Check Allows Bypass of `depositLimitByAsset` Cap - (File: contracts/LRTDepositPool.sol)

---

### Summary

`_checkIfDepositAmountExceedesCurrentLimit` in `LRTDepositPool` uses an incorrect formula for ETH deposits: it checks whether the **current** total already exceeds the limit, but never adds the incoming `amount` to the comparison. Any unprivileged depositor can therefore push total ETH deposits arbitrarily above the configured `depositLimitByAsset` cap in a single call.

---

### Finding Description

`_checkIfDepositAmountExceedesCurrentLimit` branches on whether the asset is native ETH:

```solidity
// contracts/LRTDepositPool.sol  L676-L682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount never added
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));  // ← correct for ERC-20
}
```

For every ERC-20 LST the guard correctly evaluates `totalAssetDeposits + amount > limit`. For ETH it evaluates only `totalAssetDeposits > limit`, completely ignoring the incoming `amount`. The function therefore returns `false` (i.e. "not exceeded") for any ETH deposit as long as the running total has not already crossed the limit, regardless of how large the new deposit is.

`depositETH` calls `_beforeDeposit`, which calls this check and reverts with `MaximumDepositLimitReached` only when the function returns `true`. Because `amount` is never included in the ETH branch, the revert never fires for a deposit that would push the total over the limit. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

`depositLimitByAsset[ETH_TOKEN]` is the protocol's safety cap on total ETH exposure. Because the check omits `amount`, a single depositor can send an arbitrarily large ETH value and the cap is never enforced. The protocol will mint rsETH for the full amount and forward the ETH to EigenLayer beyond the intended ceiling. The contract fails to deliver the deposit-limit guarantee it advertises, matching the **Low – contract fails to deliver promised returns** impact tier. [3](#0-2) 

---

### Likelihood Explanation

The entry point is the public, payable `depositETH` function — no role, whitelist, or special condition is required. Any depositor who holds enough ETH and observes that `totalAssetDeposits` is close to (but has not yet crossed) the limit can trigger this in a single transaction. Likelihood is **High**. [2](#0-1) 

---

### Recommendation

Add `amount` to the ETH branch, mirroring the ERC-20 branch:

```diff
- if (asset == LRTConstants.ETH_TOKEN) {
-     return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
- }
+ if (asset == LRTConstants.ETH_TOKEN) {
+     return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
+ }
``` [1](#0-0) 

---

### Proof of Concept

Assume:
- `depositLimitByAsset[ETH_TOKEN] = 1000 ether`
- `getTotalAssetDeposits(ETH_TOKEN) = 999 ether` (one wei below the limit)

A depositor calls `depositETH{value: 500 ether}(0, "")`.

1. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500 ether)`.
2. `totalAssetDeposits = 999 ether`.
3. ETH branch evaluates `999 ether > 1000 ether` → `false`.
4. The check passes; 500 ETH is accepted and rsETH is minted.
5. Total ETH in protocol is now `1499 ether`, 49.9 % above the configured cap.

The same call with any ERC-20 LST would correctly evaluate `999 + 500 > 1000` → `true` and revert with `MaximumDepositLimitReached`. [1](#0-0) [4](#0-3)
