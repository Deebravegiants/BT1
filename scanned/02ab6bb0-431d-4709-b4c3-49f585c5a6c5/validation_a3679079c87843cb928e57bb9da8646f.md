### Title
Stale Cached `rsETHPrice` Used in Deposit Minting Allows Theft of Accrued Yield — (`contracts/LRTOracle.sol` / `contracts/LRTDepositPool.sol`)

---

### Summary

`LRTOracle` stores `rsETHPrice` as a cached state variable that is only updated when `updateRSETHPrice()` is explicitly called. `LRTDepositPool.depositETH()` and `depositAsset()` compute the rsETH amount to mint using this stale cached price without first refreshing it. Because `updateRSETHPrice()` is a permissionless `public` function, an attacker can deposit at the stale (lower) price, then immediately trigger the price update themselves, causing their rsETH to be worth more than they deposited — stealing the accrued yield from existing holders.

---

### Finding Description

`LRTOracle` maintains `rsETHPrice` as a persistent storage variable: [1](#0-0) 

This value is only refreshed when `updateRSETHPrice()` is called: [2](#0-1) 

Critically, this function is `public` with no access control — any caller can invoke it.

`LRTDepositPool.getRsETHAmountToMint()` divides by the cached `rsETHPrice` to determine how many rsETH tokens to mint: [3](#0-2) 

Both `depositETH()` and `depositAsset()` route through `_beforeDeposit()` → `getRsETHAmountToMint()`: [4](#0-3) 

Neither `depositETH()` nor `depositAsset()` calls `updateRSETHPrice()` before computing the mint amount. The deposit flow is entirely `view`-based with respect to the oracle — it reads the cached price and mints accordingly. [5](#0-4) 

Between oracle updates, EigenLayer staking rewards and LST rebasing cause the true ETH value of the protocol to grow. `_updateRsETHPrice()` computes the new price from live balances via `_getTotalEthInProtocol()`: [6](#0-5) 

Until `updateRSETHPrice()` is called, `rsETHPrice` remains below its true value. Any deposit made during this window uses the artificially low denominator, minting more rsETH than the deposit is worth at the true rate.

---

### Impact Explanation

**High — Theft of unclaimed yield.**

When `rsETHPrice` is stale and lower than the true value, a depositor receives more rsETH than their deposit warrants. After the price is updated (by anyone, including the attacker), the attacker's rsETH is redeemable for more ETH than they deposited. The surplus comes directly from the accrued rewards that belong to existing rsETH holders, whose share of the pool is diluted.

Concretely: if `rsETHPrice` is stale at 1.00 ETH but the true value is 1.05 ETH (5% rewards accrued), an attacker depositing 10,000 ETH receives `10,000 / 1.00 = 10,000 rsETH` instead of the correct `10,000 / 1.05 ≈ 9,524 rsETH`. After the price update, the attacker holds ~476 excess rsETH worth ~500 ETH, extracted from existing holders. The profit scales with deposit size and the duration since the last price update.

---

### Likelihood Explanation

**Medium-High.** The `rsETHPrice` is updated by off-chain keepers on a periodic schedule (not on every block). EigenLayer staking rewards and LST rebasing accrue continuously. The gap between updates is a predictable, observable window. An attacker can:

1. Monitor `rsETHPrice` on-chain and compare it to the live TVL computed from `_getTotalEthInProtocol()`.
2. When a meaningful gap exists, take a flash loan and call `depositETH()` at the stale price.
3. Immediately call `updateRSETHPrice()` (permissionless) to crystallize the true price.
4. Hold the over-minted rsETH (or initiate withdrawal via `LRTWithdrawalManager`).

No privileged access is required. The entire setup is executable by any EOA or contract in a single transaction (deposit + price update).

---

### Recommendation

Call `updateRSETHPrice()` (or an internal equivalent) at the start of `depositETH()` and `depositAsset()`, before computing `rsethAmountToMint`, so that the mint calculation always uses the current true price:

```solidity
// In depositETH() and depositAsset(), before _beforeDeposit():
ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice();
uint256 rsethAmountToMint = _beforeDeposit(asset, depositAmount, minRSETHAmountExpected);
```

This mirrors the recommended fix in H-04: update the TVL (here, the price) before any share/token minting occurs.

---

### Proof of Concept

**Setup:** `rsETHPrice` = 1.00 ETH (stale; true value = 1.05 ETH due to accrued rewards). Total rsETH supply = 100,000. Total ETH in protocol (true) = 105,000 ETH.

**Step 1 — Attacker deposits using stale price:**
```
depositETH{value: 100,000 ETH}(minRSETHAmountExpected, "")
→ getRsETHAmountToMint: (100,000e18 * 1e18) / 1.00e18 = 100,000 rsETH minted
```
Correct amount at true price: `100,000 / 1.05 ≈ 95,238 rsETH`. Attacker received **4,762 excess rsETH**.

**Step 2 — Attacker triggers price update:**
```
LRTOracle.updateRSETHPrice()
→ totalETHInProtocol = 105,000 + 100,000 = 205,000 ETH
→ rsethSupply = 100,000 + 100,000 = 200,000
→ newRsETHPrice = 205,000 / 200,000 = 1.025 ETH
```

**Step 3 — Attacker's position:**
- Attacker holds 100,000 rsETH worth `100,000 × 1.025 = 102,500 ETH`
- Attacker deposited 100,000 ETH → **profit ≈ 2,500 ETH**

The profit is extracted from the 5,000 ETH of accrued rewards that belonged to the original 100,000 rsETH holders. With a flash loan, the attacker's capital cost is near zero. [2](#0-1) [7](#0-6) [5](#0-4) [8](#0-7)

```text
    function _updateRsETHPrice() internal {
        address rsETHTokenAddress = lrtConfig.rsETH();
        uint256 rsethSupply = IRSETH(rsETHTokenAddress).totalSupply();

        if (rsethSupply == 0) {
            rsETHPrice = 1 ether;
            highestRsethPrice = 1 ether;
            return;
        }

        if (highestRsethPrice == 0) {
            highestRsethPrice = rsETHPrice;
        }

        uint256 previousPrice = rsETHPrice;

        // get total ETH in the protocol (normalized to 1e18)
        uint256 totalETHInProtocol = _getTotalEthInProtocol();

        // calculate previousTVL using rsethSupply multiplied by rsETHPrice
        uint256 previousTVL = rsethSupply.mulWad(rsETHPrice);

        IPausable lrtDepositPool = IPausable(lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL));
        IPausable withdrawalManager = IPausable(lrtConfig.getContract(LRTConstants.LRT_WITHDRAW_MANAGER));

        // determine if the protocol is active (not paused)
        bool protocolPaused = lrtDepositPool.paused() || withdrawalManager.paused() || paused;

        // only take fee if TVL increased and protocol is not paused
        uint256 protocolFeeInETH = 0;
        if (!protocolPaused && totalETHInProtocol > previousTVL) {
            uint256 rewardAmount = totalETHInProtocol - previousTVL;
            protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
        }

        // compute new rsETH price based on total ETH minus fee
        uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```
