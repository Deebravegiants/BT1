### Title
`feeBps` Not Validated in `initialize` Enables Zero-rsETH Minting or Deposit Revert — (File: contracts/pools/RSETHPoolV3WithNativeChainBridge.sol, contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolV2ExternalBridge.sol, contracts/pools/RSETHPoolV2.sol, contracts/pools/RSETHPool.sol, contracts/pools/RSETHPoolNoWrapper.sol)

---

### Summary

Multiple L2 pool contracts accept an unbounded `_feeBps` parameter in their `initialize` functions with no upper-bound check, while their corresponding `setFeeBps` admin functions do enforce a cap. If `feeBps` is initialized at or above the 10,000 denominator, every user deposit either silently yields 0 rsETH (all ETH consumed as fee) or reverts due to arithmetic underflow, permanently blocking the pool.

---

### Finding Description

Every pool contract computes the swap output via `viewSwapRsETHAmountAndFee`:

```solidity
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;   // underflows if feeBps > 10_000
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

The `setFeeBps` admin functions guard against this:

- `RSETHPoolV3.sol` / `RSETHPoolV3WithNativeChainBridge.sol`: `if (_feeBps > 1000) revert InvalidFeeAmount();`
- `RSETHPool.sol` / `RSETHPoolNoWrapper.sol` / `RSETHPoolV2.sol`: `if (_feeBps > 10_000) revert InvalidFeeAmount();`

But the `initialize` functions in all of these contracts assign `feeBps = _feeBps` with **no corresponding check**:

- `RSETHPoolV3.sol` `initialize` (line 229): `feeBps = _feeBps;` — no guard
- `RSETHPoolV3WithNativeChainBridge.sol` `initialize` (line 266): `feeBps = _feeBps;` — no guard
- `RSETHPoolV2ExternalBridge.sol` `initialize` (line 278): `feeBps = _feeBps;` — no guard
- `RSETHPoolV2.sol` `initialize` (line 196): `feeBps = _feeBps;` — no guard
- `RSETHPool.sol` `initialize` (line 248): `feeBps = _feeBps;` — no guard
- `RSETHPoolNoWrapper.sol` `initialize` (line 214): `feeBps = _feeBps;` — no guard [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) 

The fee arithmetic in `viewSwapRsETHAmountAndFee` uses 10,000 as the denominator across all variants: [9](#0-8) [10](#0-9) [11](#0-10)

### Impact Explanation

| `feeBps` value | Effect on `deposit()` |
|---|---|
| `== 10_000` | `fee == amount`, `amountAfterFee == 0`, `rsETHAmount == 0`; user's ETH is silently consumed entirely as protocol fee, credited to `feeEarnedInETH` |
| `> 10_000` | `amount - fee` underflows; every `deposit()` call reverts; pool is permanently bricked until an upgrade |

In the `feeBps == 10_000` case, every depositor loses 100% of their ETH/LST to fees — a direct theft of user funds. In the `feeBps > 10_000` case, the pool is rendered non-functional for all users.

**Impact: Critical** — direct theft of user funds (feeBps == 10_000) or permanent freezing of funds (feeBps > 10_000).

---

### Likelihood Explanation

The `initialize` function is called exactly once at deployment. A deployer who confuses percentage (100) with basis points (10,000), or who copies a value from a different context, can trivially produce `feeBps >= 10_000`. The inconsistency with `setFeeBps` — which does enforce the cap — makes this a realistic deployment error. The pool contracts are deployed across multiple L2 chains, multiplying the exposure surface.

**Likelihood: Low** — requires a deployment-time misconfiguration, but the missing guard makes it a realistic mistake given the inconsistency with `setFeeBps`.

---

### Recommendation

Add the same upper-bound check to every `initialize` function that `setFeeBps` already enforces. For `RSETHPoolV3` and `RSETHPoolV3WithNativeChainBridge`, apply the stricter 1,000 cap:

```solidity
// RSETHPoolV3 / RSETHPoolV3WithNativeChainBridge
if (_feeBps > 1000) revert InvalidFeeAmount();

// RSETHPool / RSETHPoolNoWrapper / RSETHPoolV2 / RSETHPoolV2ExternalBridge
if (_feeBps >= 10_000) revert InvalidFeeAmount();
```

Apply this check immediately before `feeBps = _feeBps;` in each `initialize` function.

---

### Proof of Concept

1. Deploy `RSETHPoolV3WithNativeChainBridge` and call `initialize` with `_feeBps = 10_000`.
2. `feeBps` is stored as `10_000` — no revert occurs.
3. A user calls `deposit{value: 1 ether}("ref")`.
4. Inside `viewSwapRsETHAmountAndFee(1 ether)`:
   - `fee = 1e18 * 10_000 / 10_000 = 1e18`
   - `amountAfterFee = 1e18 - 1e18 = 0`
   - `rsETHAmount = 0 * 1e18 / rate = 0`
5. `feeEarnedInETH += 1e18` — the user's full ETH is recorded as fee.
6. `wrsETH.mint(msg.sender, 0)` — user receives 0 rsETH.
7. The BRIDGER_ROLE can later call `withdrawFees` and extract the user's ETH.

Contrast: calling `setFeeBps(10_000)` after deployment reverts with `InvalidFeeAmount`, demonstrating the inconsistency. [12](#0-11) [13](#0-12)
