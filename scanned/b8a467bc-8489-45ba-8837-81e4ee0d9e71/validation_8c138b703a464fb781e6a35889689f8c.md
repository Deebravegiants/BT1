### Title
Missing Chainlink Staleness Validation in `ChainlinkPriceOracle` Allows Deposits at Stale Asset Prices - (File: contracts/oracles/ChainlinkPriceOracle.sol)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards every field except the raw `answer`, performing no staleness or validity checks. This is the oracle used by `LRTDepositPool` to price LST assets when minting rsETH. If a Chainlink feed becomes stale at an inflated price, any depositor can mint excess rsETH at the expense of existing holders.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the Chainlink price as follows:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

All five return values of `latestRoundData()` — `roundId`, `answer`, `startedAt`, `updatedAt`, `answeredInRound` — are available, but only `answer` is used. The contract performs no check on:

- `answeredInRound >= roundId` (detects a stale round)
- `updatedAt > 0` (detects an incomplete round)
- `updatedAt` freshness relative to `block.timestamp` (detects a time-stale feed)
- `price > 0` (detects an invalid/negative price)

The protocol's own `ChainlinkOracleForRSETHPoolCollateral`, used for L2 pool collateral, implements exactly these checks:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L27-32
(uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
    AggregatorV3Interface(oracle).latestRoundData();

if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

`ChainlinkPriceOracle` is registered in `LRTOracle` as the `assetPriceOracle` for supported LSTs. `LRTOracle.getAssetPrice()` delegates directly to it: [3](#0-2) 

`LRTDepositPool.getRsETHAmountToMint()` uses this price as the numerator in the rsETH minting formula:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [4](#0-3) 

This is called unconditionally inside `_beforeDeposit()`, which is the shared pre-check for both `depositETH()` and `depositAsset()`: [5](#0-4) 

---

### Impact Explanation

**Impact: High — Theft of unclaimed yield (dilution of existing rsETH holders)**

If a Chainlink LST/ETH feed becomes stale while reporting a price higher than the asset's true current value (e.g., the LST has depegged or dropped in value but the feed has not yet updated), any depositor calling `depositAsset()` will receive:

```
rsethAmountToMint = (amount * P_stale) / rsETHPrice
```

where `P_stale > P_actual`. The depositor receives more rsETH than their deposited assets are worth. Because rsETH is a share token backed by the total ETH value in the protocol, this excess minting dilutes all existing rsETH holders — their proportional claim on the underlying assets decreases. This constitutes theft of accrued yield from existing holders.

---

### Likelihood Explanation

**Likelihood: Low**

Chainlink feeds becoming stale requires an external trigger: network congestion, Chainlink node failures, or a rapid price drop that outpaces the feed's heartbeat/deviation threshold. These events are uncommon but historically documented. The attacker must also be monitoring for such conditions and act within the staleness window. The combination makes this low likelihood but not theoretical.

---

### Recommendation

Apply the same staleness validation pattern already used in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    // Optionally: if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

---

### Proof of Concept

1. A supported LST (e.g., stETH) has a Chainlink feed that last updated at price `P_old = 1.05e18` (1.05 ETH per stETH).
2. The LST depegs; its true price drops to `P_actual = 0.98e18`, but the Chainlink feed has not yet updated (stale within heartbeat window).
3. Current `rsETHPrice = 1.06e18` (stored in `LRTOracle`).
4. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
5. `getRsETHAmountToMint(stETH, 1000e18)` computes:
   - `getAssetPrice(stETH)` → `ChainlinkPriceOracle` returns stale `1.05e18` (no staleness check)
   - `rsethAmountToMint = (1000e18 * 1.05e18) / 1.06e18 ≈ 990.57 rsETH`
6. At true price, the attacker should receive: `(1000e18 * 0.98e18) / 1.06e18 ≈ 924.53 rsETH`.
7. Attacker receives ~66 excess rsETH (~7% overmint), diluting all existing rsETH holders by that amount.
8. No revert occurs because `ChainlinkPriceOracle` performs zero validation on the returned data. [1](#0-0) [6](#0-5)
