### Title
`withdrawTokens()` Does Not Protect the `kernel` Distribution Token from Being Drained — (File: `contracts/KERNEL/KernelTop100MerkleDistributor.sol`)

---

### Summary
`KernelTop100MerkleDistributor.withdrawTokens()` is a token-recovery helper intended for accidentally sent tokens, but it applies no exclusion for the `kernel` token itself — the primary asset held in trust for user claims. An owner who calls it with `address(kernel)` drains the entire distribution pool, permanently freezing every pending user claim.

---

### Finding Description
`withdrawTokens()` (lines 461–472) accepts any `_token` address and transfers `_amount` to `_recipient` with no guard:

```solidity
function withdrawTokens(address _token, uint256 _amount, address _recipient) external onlyOwner {
    UtilLib.checkNonZeroAddress(_token);
    UtilLib.checkNonZeroAddress(_recipient);
    if (_amount == 0) revert ZeroValueProvided();
    IERC20(_token).safeTransfer(_recipient, _amount);
    emit TokensWithdrawn(_token, _amount, _recipient);
}
``` [1](#0-0) 

The contract holds `kernel` tokens as the sole source of funds for all user claims. The `kernel` state variable is set at initialization and never excluded from the recovery path: [2](#0-1) 

User entitlements are recorded off-chain in the merkle tree and on-chain only as `userClaims[user].amountClaimed`. There is no on-chain reserve accounting that would prevent `withdrawTokens` from transferring the entire `kernel` balance.

This is the direct analog of the yveCRV `protectedTokens()` bug: just as `yveCrv` was omitted from the protected array and could therefore be swept, `kernel` is omitted from any exclusion list in `withdrawTokens()` and can therefore be drained.

---

### Impact Explanation
**High — Theft of unclaimed yield.**

All KERNEL tokens held by the distributor represent yield owed to users under the vesting schedule. Draining the contract balance makes every future `claim()` and `claimAndStake()` call revert on the `safeTransfer`, permanently freezing every user's unclaimed KERNEL allocation. [3](#0-2) 

---

### Likelihood Explanation
**Low-to-Medium.** The function is callable only by the owner. However, the function is presented as a generic token-recovery utility with no documentation warning against using it on `kernel`. An operator following a runbook to "recover mistakenly sent tokens" could supply the wrong address. Additionally, any future key-management incident affecting the owner EOA directly enables this path. No external preconditions (oracle, bridge, third-party protocol) are required.

---

### Recommendation
Add an explicit guard at the top of `withdrawTokens()`:

```solidity
if (_token == address(kernel)) revert CannotWithdrawKernelToken();
```

This mirrors the correct fix for the yveCRV bug (adding the managed token to the protected list) and ensures the recovery helper cannot be used — accidentally or otherwise — to drain the distribution reserve.

---

### Proof of Concept

1. Owner (or compromised owner key) calls:
   ```solidity
   distributor.withdrawTokens(
       address(kernel),
       kernel.balanceOf(address(distributor)),
       attacker
   );
   ```
2. `withdrawTokens` performs no exclusion check; `safeTransfer` succeeds.
3. `kernel.balanceOf(distributor)` drops to zero.
4. Every subsequent `claim()` / `claimAndStake()` call reverts at `kernel.safeTransfer(user, amountToSend)` because the balance is zero.
5. All users' vested KERNEL yield is permanently frozen. [1](#0-0) [4](#0-3)
