### Title
Missing Chainlink `latestRoundData` Safety Checks Allow Zero/Stale Price to Permanently Freeze Depositor Funds - (File: contracts/oracles/ChainlinkPriceOracle.sol)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `answer` (the price). It performs no validation on the returned data. If Chainlink returns a zero or stale price, depositors who call `LRTDepositPool.depositETH()` or `LRTDepositPool.depositAsset()` will have their assets transferred into the protocol while receiving 0 rsETH in return — permanently freezing their funds with no recovery path.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` is the price oracle used by `LRTOracle` for all LST collateral assets (stETH, rETH, etc.):

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol#52-54
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All five return values from `latestRoundData()` — `roundId`, `answer`, `startedAt`, `updatedAt`, `answeredInRound` — are available, but only `answer` is used. The following safety checks are entirely absent:

- **`require(price > 0)`** — Chainlink can return `0` during circuit-breaker events or for incomplete rounds. `uint256(0)` propagates silently.
- **`require(updatedAt > 0)`** — `updatedAt == 0` indicates an incomplete/invalid round.
- **`require(answeredInRound >= roundId)`** — Ensures the answer is not carried over from a prior round (stale).
- **No staleness window check** — No `block.timestamp - updatedAt < maxDelay` guard.

By contrast, the same codebase's `ChainlinkOracleForRSETHPoolCollateral.getRate()` implements all three critical checks:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol#30-32
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The inconsistency confirms the protocol is aware of these requirements but failed to apply them to `ChainlinkPriceOracle`.

The zero-price path through the deposit flow is:

1. `ChainlinkPriceOracle.getAssetPrice(asset)` returns `0` (Chainlink returns `price = 0`).
2. `LRTOracle.getAssetPrice(asset)` returns `0`.
3. `LRTDepositPool.getRsETHAmountToMint(asset, amount)` computes `(amount * 0) / rsETHPrice = 0`.
4. `_beforeDeposit` checks `rsethAmountToMint < minRSETHAmountExpected` — passes if user set `minRSETHAmountExpected = 0` (the default/common case).
5. `IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount)` — user's LST is taken.
6. `_mintRsETH(0)` — user receives 0 rsETH.
7. User has no rsETH and therefore no withdrawal claim. Funds are permanently locked.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A depositor who calls `depositAsset()` or `depositETH()` during a period when Chainlink returns `price = 0` for the relevant LST feed will:
- Transfer their full LST/ETH amount into `LRTDepositPool`.
- Receive 0 rsETH in return.
- Have no mechanism to recover their deposit (withdrawal requires holding rsETH).

Additionally, a stale price (e.g., Chainlink hits its deviation threshold and stops updating during a market crash) causes `_getTotalEthInProtocol()` to compute an inflated TVL, leading `updateRSETHPrice()` to set an incorrect rsETH price. This allows a depositor to mint more rsETH than their deposit is worth, diluting all existing rsETH holders — a form of share/asset mis-accounting and theft of yield.

---

### Likelihood Explanation

Chainlink price feeds are known to return `0` or stale data during:
- Network congestion preventing oracle updates.
- Circuit-breaker events (price moves beyond min/max bounds).
- Incomplete rounds (`updatedAt == 0`).

These are documented Chainlink edge cases. The protocol's own `ChainlinkOracleForRSETHPoolCollateral` guards against them, confirming the team is aware of the risk. The `ChainlinkPriceOracle` is the primary oracle for all LST assets in the core deposit path, making this a high-likelihood trigger for any depositor active during such an event.

---

### Recommendation

Apply the same validation pattern already used in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    require(answeredInRound >= roundId, "Stale price");
    require(updatedAt > 0, "Incomplete round");
    require(price > 0, "Invalid price");
    // Optional: require(block.timestamp - updatedAt < MAX_STALENESS, "Price too old");

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

### Proof of Concept

**Zero-price fund freeze:**

1. Chainlink's LST/ETH feed enters a circuit-breaker state and returns `answer = 0`.
2. Attacker (or innocent user) calls:
   ```solidity
   LRTDepositPool.depositAsset(stETH, 10 ether, 0, "ref");
   ```
3. `ChainlinkPriceOracle.getAssetPrice(stETH)` returns `uint256(0) * 1e18 / 1e18 = 0`.
4. `getRsETHAmountToMint(stETH, 10 ether)` returns `(10e18 * 0) / rsETHPrice = 0`.
5. `_beforeDeposit` passes because `0 >= 0` (minRSETHAmountExpected = 0).
6. `safeTransferFrom` moves 10 stETH from user to `LRTDepositPool`.
7. `_mintRsETH(0)` mints nothing.
