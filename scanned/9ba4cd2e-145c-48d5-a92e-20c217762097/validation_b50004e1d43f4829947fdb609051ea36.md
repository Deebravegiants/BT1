### Title
Permanent Freezing of User Funds via Blacklisted Address in `_processWithdrawalCompletion` — (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTWithdrawalManager._processWithdrawalCompletion` directly transfers LST tokens to the user's address. If a supported LST token implements a blacklist and the user's address is blacklisted, the transfer will always revert. Because the user's rsETH was already burned at `initiateWithdrawal` time and the assets were already pulled from `LRTUnstakingVault` at `unlockQueue` time, the user's funds become permanently frozen inside `LRTWithdrawalManager` with no recovery path.

---

### Finding Description

The withdrawal lifecycle in `LRTWithdrawalManager` is a three-step process:

**Step 1 — `initiateWithdrawal` (user-callable):**
The user's rsETH is burned and a withdrawal request is created. [1](#0-0) 

**Step 2 — `unlockQueue` (operator-callable):**
The rsETH held by the contract is burned and the corresponding LST assets are pulled from `LRTUnstakingVault` into `LRTWithdrawalManager`. [2](#0-1) 

**Step 3 — `completeWithdrawal` / `completeWithdrawalForUser`:**
Both call `_processWithdrawalCompletion`, which transfers the LST directly to the user's address via `_transferAsset`. [3](#0-2) 

The `_transferAsset` helper calls `IERC20(asset).safeTransfer(to, amount)` for ERC-20 assets: [4](#0-3) 

If the LST token has a blacklist (e.g., a future supported LST similar to USDC/USDT) and the user's address is blacklisted, `safeTransfer` reverts. The entire `_processWithdrawalCompletion` transaction reverts atomically, restoring the queue state. However, the user's rsETH was already burned in Step 1 and the assets are already sitting in `LRTWithdrawalManager` after Step 2. The user can never receive their assets.

There is no function in the contract that allows redirecting a stuck withdrawal to a different address. The `sweepRemainingAssets` function cannot be used either, because it requires `hasUnlockedWithdrawals(asset)` to be false — but the blacklisted user's unlocked withdrawal keeps `unlockedWithdrawalsCount[asset]` non-zero indefinitely: [5](#0-4) 

Even the operator-callable `completeWithdrawalForUser` cannot rescue the funds, because it still transfers to the original (blacklisted) user address: [6](#0-5) 

---

### Impact Explanation

**Permanent freezing of user funds.** The user's rsETH is burned at `initiateWithdrawal`. After `unlockQueue`, the corresponding LST assets are held in `LRTWithdrawalManager`. If the user is blacklisted by the LST token, neither `completeWithdrawal` nor `completeWithdrawalForUser` can deliver the assets. There is no admin escape hatch to redirect the withdrawal to an alternate address. The assets are permanently locked in the contract.

Additionally, `unlockedWithdrawalsCount[asset]` remains non-zero, blocking `sweepRemainingAssets` for all users of that asset indefinitely.

---

### Likelihood Explanation

**Low-to-medium.** The currently supported assets (stETH, ETHx, sfrxETH) do not implement address blacklists. However, the protocol is explicitly designed to be extensible — new LSTs can be added via `LRTConfig`. Any future supported LST that implements a blacklist (a common compliance feature in regulated token designs) would expose this path. A user could also be blacklisted after initiating a withdrawal, making this an accidental rather than intentional scenario.

---

### Recommendation

Replace the direct-push transfer pattern with a **pull-based claim model**: instead of pushing assets to the user in `_processWithdrawalCompletion`, record the claimable amount in a mapping and let the user (or any address they designate) pull the funds in a separate `claim` call. This isolates any per-user transfer failure and prevents it from blocking the protocol or permanently freezing funds.

Alternatively, add an admin/operator function that allows redirecting a stuck withdrawal to a user-specified alternate address, so that a blacklisted user can still recover their funds via a different wallet.

---

### Proof of Concept

1. Alice holds rsETH and calls `initiateWithdrawal(stETH, amount, "")`. Her rsETH is burned; a withdrawal request is recorded. [7](#0-6) 

2. An operator calls `unlockQueue(stETH, ...)`. The contract burns the rsETH it holds and calls `unstakingVault.redeem(stETH, amount)`, moving the stETH into `LRTWithdrawalManager`. [8](#0-7) 

3. Alice's address is blacklisted by the stETH contract (or a future supported LST with blacklist).

4. Alice calls `completeWithdrawal(stETH, "")`. Inside `_processWithdrawalCompletion`, `_transferAsset` calls `IERC20(stETH).safeTransfer(alice, amount)`, which reverts due to the blacklist. The entire transaction reverts. [9](#0-8) 

5. An operator calls `completeWithdrawalForUser(stETH, alice, "")`. Same result — still transfers to Alice's blacklisted address, still reverts. [6](#0-5) 

6. Alice's rsETH is permanently burned. The stETH sits in `LRTWithdrawalManager` forever. `unlockedWithdrawalsCount[stETH]` remains ≥ 1, blocking `sweepRemainingAssets` for all stETH withdrawers. [4](#0-3)

```text
    function _processWithdrawalCompletion(address asset, address user, string calldata referralId) internal {
        if (userAssociatedNonces[asset][user].empty()) {
            revert NoWithdrawalRequests(user, asset);
        }

        // Retrieve and remove the oldest withdrawal request for the user.
        uint256 usersFirstWithdrawalRequestNonce = userAssociatedNonces[asset][user].popFront();
        // Ensure the request is already unlocked.
        if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();

        bytes32 requestId = getRequestId(asset, usersFirstWithdrawalRequestNonce);
        WithdrawalRequest memory request = withdrawalRequests[requestId];

        delete withdrawalRequests[requestId];

        // Check that the withdrawal delay has passed since the request's initiation.
        if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();

        unlockedWithdrawalsCount[asset]--;

        // If Aave integration is enabled and asset is ETH, withdraw from Aave if needed
        if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
            uint256 contractBalance = address(this).balance;
            if (contractBalance < request.expectedAssetAmount) {
                uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
                _withdrawFromAave(amountNeeded);

                // Verify we have sufficient balance after withdrawal
                uint256 balanceAfter = address(this).balance;
                if (balanceAfter < request.expectedAssetAmount) {
                    revert InsufficientLiquidityForWithdrawal();
                }
            }
        }

        _transferAsset(asset, user, request.expectedAssetAmount);

        emit ReferralIdEmitted(referralId);
        emit AssetWithdrawalFinalized(user, asset, request.rsETHUnstaked, request.expectedAssetAmount);
    }
```
