### Title
Missing Zero-Address Validation for `admin` and `bridger` in `initialize` - (File: contracts/pools/RSETHPoolV2.sol)

### Summary
Multiple L2 pool contracts (`RSETHPoolV2`, `RSETHPool`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`) fail to validate that the `admin` and `bridger` addresses are non-zero in their `initialize` functions. If `admin` is set to `address(0)`, `DEFAULT_ADMIN_ROLE` and `BRIDGER_ROLE` are permanently granted to the zero address, making all admin-gated and bridger-gated functions permanently inaccessible. If `bridger` is also zero, accumulated protocol fees (ETH) held in the contract can never be withdrawn.

### Finding Description
In `RSETHPoolV2.sol`, the `initialize` function accepts `admin` and `bridger` as parameters but only validates `_wrsETH` and `_rsETHOracle` for non-zero:

```solidity
function initialize(
    address admin,
    address bridger,
    address _wrsETH,
    uint256 _feeBps,
    address _rsETHOracle
) external initializer {
    UtilLib.checkNonZeroAddress(_wrsETH);      // checked
    UtilLib.checkNonZeroAddress(_rsETHOracle); // checked
    // admin and bridger NOT checked
    _grantRole(DEFAULT_ADMIN_ROLE, admin);
    _setupRole(BRIDGER_ROLE, admin);
    _setupRole(BRIDGER_ROLE, bridger);
    ...
}
```

The same pattern exists in `RSETHPool.sol` (missing checks for `admin`, `manager`, `_rsETHOracle`, `_wstETH_ETHOracle`), `RSETHPoolV3ExternalBridge.sol`, and `RSETHPoolV3WithNativeChainBridge.sol`.

By contrast, contracts such as `KernelReceiver`, `KernelVaultETH`, `RSETHPoolNoWrapper`, and `L1Vault` correctly validate all address parameters including admin/operator roles.

### Impact Explanation
If `admin` is `address(0)`:
- `DEFAULT_ADMIN_ROLE` is permanently assigned to the zero address — no EOA or contract can ever hold it.
- `BRIDGER_ROLE` is also assigned to zero (since `admin` is granted both roles), and if `bridger` is also zero, no one holds `BRIDGER_ROLE`.
- `withdrawFees(address receiver)` requires `BRIDGER_ROLE` — accumulated ETH fees in the pool are permanently frozen.
- `reinitialize` and other admin-gated functions are permanently inaccessible.

This maps to **Medium — Permanent freezing of unclaimed yield** (accumulated ETH fees cannot be recovered) and **Low — Contract fails to deliver promised returns** (admin management permanently broken).

### Likelihood Explanation
Likelihood is low but non-negligible. Deployment scripts that omit or zero-out an address parameter (e.g., a misconfigured deployment pipeline, a scripting error, or a missing environment variable) would silently succeed — the `initialize` call would not revert, and the misconfiguration would only be discovered post-deployment when admin or fee-withdrawal calls fail. The absence of a revert guard means there is no on-chain safety net.

### Recommendation
Add `UtilLib.checkNonZeroAddress` for `admin` and `bridger` (and any other address parameters) at the top of each affected `initialize` function, consistent with the pattern already used in `KernelReceiver`, `KernelVaultETH`, and `RSETHPoolNoWrapper`:

```solidity
UtilLib.checkNonZeroAddress(admin);
UtilLib.checkNonZeroAddress(bridger);
```
