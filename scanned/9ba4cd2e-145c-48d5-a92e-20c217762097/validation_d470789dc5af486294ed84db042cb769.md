### Title
Manager Can Bypass Withdrawal Delay by Setting `withdrawalDelayBlocks` to Zero - (File: contracts/LRTWithdrawalManager.sol)

### Summary
`LRTWithdrawalManager.setWithdrawalDelayBlocks()` enforces only an upper bound on the withdrawal delay (≤ 16 days) but has no lower bound. The LRT manager can reduce `withdrawalDelayBlocks` to `0` at any time, immediately making all pending withdrawal requests eligible for unlocking and completion — bypassing the 8-day delay that users relied upon when initiating their withdrawals.

### Finding Description
The contract initializes `withdrawalDelayBlocks` to `8 days / 12 seconds` (~57,600 blocks): [1](#0-0) 

The manager can update this value via `setWithdrawalDelayBlocks`, which only enforces a ceiling: [2](#0-1) 

There is no minimum delay enforced. Setting `withdrawalDelayBlocks_ = 0` is accepted without revert.

The delay is checked in two places. In `_unlockWithdrawalRequests` (called by the operator via `unlockQueue`): [3](#0-2) 

And in `_processWithdrawalCompletion` (called by the user via `completeWithdrawal`): [4](#0-3) 

Both checks use the current value of `withdrawalDelayBlocks` at execution time, not the value at the time the request was initiated. Therefore, reducing `withdrawalDelayBlocks` to `0` retroactively removes the delay for all already-queued withdrawal requests.

### Impact Explanation
Users who called `initiateWithdrawal` under the assumption of an 8-day delay period can have their withdrawals unlocked and completed immediately after the manager reduces the delay to zero. The payout is computed at `unlockQueue` time using current oracle prices (`min(expectedAssetAmount, currentReturn)`), meaning any oracle price decline between initiation and the forced-early unlock results in users receiving less than the amount they locked their rsETH for. There is no cancel function, so users cannot exit once their rsETH is locked.

**Impact: Low** — Contract fails to deliver the promised delay period; users may receive less than expected if oracle prices declined, but no direct theft occurs from the delay bypass alone.

### Likelihood Explanation
Requires the LRT manager role to act against user interests (reduce delay to 0 and trigger `unlockQueue` immediately). The manager is a privileged protocol role. This is the same trust assumption as the referenced external report (admin reducing upgrade delay).

### Recommendation
Enforce a minimum value for `withdrawalDelayBlocks`, for example no less than `1 day / 12 seconds`. Additionally, consider applying delay changes only to withdrawal requests initiated *after* the change, not retroactively to already-queued requests, by snapshotting the delay at request creation time.

### Proof of Concept
1. User calls `initiateWithdrawal(asset, rsETHAmount, ...)` at block `N`. rsETH is locked. `withdrawalStartBlock = N`. Expected delay: ~57,600 blocks (8 days).
2. LRT manager calls `setWithdrawalDelayBlocks(0)`.
3. Operator immediately calls `unlockQueue(asset, ...)`. The check `block.number < N + 0` is false at any block ≥ N, so the request is unlocked at the current (potentially lower) oracle price.
4. User's withdrawal is finalized immediately with no delay, receiving `min(expectedAssetAmount, currentOracleReturn)` — potentially less than expected if the rsETH/asset oracle rate moved adversely. [2](#0-1) [3](#0-2) [4](#0-3)
