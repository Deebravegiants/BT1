### Title
`ChainlinkPriceOracle.getAssetPrice()` Omits Staleness and Validity Checks That `ChainlinkOracleForRSETHPoolCollateral.getRate()` Enforces - (File: contracts/oracles/ChainlinkPriceOracle.sol)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` — the oracle used by the core protocol (`LRTOracle` → `LRTDepositPool`) — silently discards every return value from `latestRoundData()` except the raw price. The pool-side oracle `ChainlinkOracleForRSETHPoolCollateral.getRate()` enforces three explicit guards (`answeredInRound < roundID`, `timestamp == 0`, `price <= 0`) on the same Chainlink call. The missing guards in the core-protocol path are the direct analog of the Astaria H-19 pattern: a validity check present in one entry path is absent in another, letting the unguarded path accept data the guarded path would reject.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` reads only the `price` field:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol  line 52
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

`roundId`, `startedAt`, `updatedAt`, and `answeredInRound` are all discarded. No check is made for a stale round, an incomplete round, or a non-positive price.

`ChainlinkOracleForRSETHPoolCollateral.getRate()` — used by the RSETHPool family on L2 — performs three guards on the identical call:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol  lines 27-32
(uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
    AggregatorV3Interface(oracle).latestRoundData();

if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0)            revert IncompleteRound();
if (ethPrice <= 0)             revert InvalidPrice();
```

The two paths that consume Chainlink data are therefore asymmetric:

| Path | Oracle contract | Stale-round guard | Incomplete-round guard | Non-positive-price guard |
|---|---|---|---|---|
| RSETHPool (L2 deposit) | `ChainlinkOracleForRSETHPoolCollateral` | ✅ | ✅ | ✅ |
| LRTDepositPool / LRTOracle (L1 core) | `ChainlinkPriceOracle` | ❌ | ❌ | ❌ |

`ChainlinkPriceOracle.getAssetPrice()` feeds directly into:

1. `LRTOracle.getAssetPrice()` → `LRTDepositPool.getRsETHAmountToMint()` — determines how many rsETH tokens a depositor receives.
2. `LRTOracle._getTotalEthInProtocol()` → `LRTOracle._updateRsETHPrice()` — determines the stored `rsETHPrice` used for all subsequent deposit and withdrawal calculations.
3. `LRTWithdrawalManager._createUnlockParams()` — determines the asset price used to settle withdrawal requests.

---

### Impact Explanation

When a Chainlink feed for a supported LST (e.g., stETH/ETH, cbETH/ETH) goes stale at a price above the current market price — a realistic scenario during network congestion or oracle downtime — `ChainlinkPriceOracle` returns the inflated stale price without reverting. A depositor who monitors for this condition can call `LRTDepositPool.depositAsset()` while the feed is stale, causing `getRsETHAmountToMint()` to compute a larger rsETH amount than the deposited collateral warrants. The excess rsETH is minted at the expense of existing rsETH holders, whose proportional claim on protocol TVL is diluted. This constitutes theft of yield from existing holders.

Additionally, if `price` is zero (e.g., a misconfigured or deprecated feed), `uint256(price)` returns 0, making `rsethAmountToMint = 0` (depositor receives nothing) or causing a division-by-zero in `rsETHPrice` computation, which would revert `_updateRsETHPrice()` and freeze price updates.

**Impact classification**: High — theft of unclaimed yield from existing rsETH holders; potential temporary freeze of price-update functionality.

---

### Likelihood Explanation

Chainlink feeds do go stale: during extreme market volatility, sequencer downtime (on L2), or when a feed is deprecated. The `answeredInRound < roundID` condition is a documented Chainlink staleness indicator. An unprivileged depositor needs only to observe the on-chain state of the Chainlink feed and call `LRTDepositPool.depositAsset()` during the stale window — no special permissions required.

---

### Recommendation

Add the same guards present in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0)            revert IncompleteRound();
    if (price <= 0)                revert InvalidPrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Additionally, consider adding a `block.timestamp - updatedAt > heartbeat` check per feed, as `answeredInRound < roundId` is a deprecated Chainlink staleness signal.

### Proof of Concept

1. A Chainlink LST/ETH feed (e.g., stETH/ETH) stops updating; `answeredInRound < roundId` becomes true.
2. `ChainlinkOracleForRSETHPoolCollateral.getRate()` reverts with `StalePrice()` — RSETHPool deposits are blocked.
3. `ChainlinkPriceOracle.getAssetPrice()` returns the last stale price (e.g., 1.05 ETH per stETH when the real rate is 0.98 ETH) — no revert.
4. Attacker calls `LRTDepositPool.depositAsset(stETH, amount, 0, "")`.
5. `getRsETHAmountToMint()` computes `rsethAmountToMint = amount * 1.05e18 / rsETHPrice`, minting ~7% more rsETH than the deposited collateral is worth.
