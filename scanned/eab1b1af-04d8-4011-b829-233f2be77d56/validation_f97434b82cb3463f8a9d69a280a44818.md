### Title
`getExpectedAssetAmount` Does Not Account for `instantWithdrawalFee`, Misleading Integrators on Actual Proceeds - (File: contracts/LRTWithdrawalManager.sol)

### Summary

`LRTWithdrawalManager.getExpectedAssetAmount` is a public view function exposed in the `ILRTWithdrawalManager` interface that is documented as returning "Amount of underlying to receive." However, when `instantWithdrawalFee > 0`, the actual amount transferred to the user in `instantWithdrawal` is `assetAmountUnlocked - fee`, not `assetAmountUnlocked`. The view function never deducts the fee, so it systematically overstates what a caller will receive via the instant-withdrawal path.

### Finding Description

`getExpectedAssetAmount` computes the gross asset amount from the oracle price ratio:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

`instantWithdrawal` calls this same function to obtain `assetAmountUnlocked`, then silently deducts the fee before transferring to the user:

```solidity
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
// ...
uint256 fee = (assetAmountUnlocked * instantWithdrawalFee) / 10_000;
uint256 userAmount = assetAmountUnlocked - fee;
// ...
_transferAsset(asset, msg.sender, userAmount);
```

The fee can be up to 10% (`feeBasisPoints > 1000` reverts). Any external contract or off-chain system that calls `getExpectedAssetAmount` to simulate an instant withdrawal will receive a value that is up to 10% higher than what is actually transferred.

### Impact Explanation

External smart contracts that integrate with `LRTWithdrawalManager` (e.g., aggregators, yield routers, or collateral managers) may call `getExpectedAssetAmount` to determine how much asset they will receive before executing `instantWithdrawal`. Because the fee is not reflected, the returned value is inflated. A downstream contract that relies on receiving exactly `getExpectedAssetAmount` worth of assets — for example, to repay a flash loan or meet a minimum-output check — will receive less than expected and may revert or behave incorrectly. This matches the "contract fails to deliver promised returns" impact class.

### Likelihood Explanation

`instantWithdrawalFee` is a configurable parameter actively managed by `onlyLRTManager`. The setter enforces a maximum of 1000 bps (10%). Any non-zero fee setting — which is the normal operational state for a fee-bearing instant-withdrawal feature — triggers the discrepancy. The function is part of the public interface (`ILRTWithdrawalManager`) and is the natural entry point for integrators to preview proceeds.

### Recommendation

`getExpectedAssetAmount` should deduct the `instantWithdrawalFee` when computing the expected user proceeds, or a separate `getExpectedInstantWithdrawalAmount` function should be provided that returns the net amount after fee. The standard withdrawal path (no fee) can continue to use the gross amount.

### Proof of Concept

1. `instantWithdrawalFee` is set to 500 bps (5%) by the LRT manager.
2. An integrator calls `getExpectedAssetAmount(stETH, 1e18)` and receives `X`.
3. The integrator calls `instantWithdrawal(stETH, 1e18, "")` expecting to receive `X`.
4. Actual transfer is `X * 0.95` (fee deducted at line 237–238).
