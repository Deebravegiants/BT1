### Title
`FeeReceiver.depositPool` Is Stored at Initialization and Not Dynamically Queried from `LRTConfig`, Causing MEV Rewards to Route to a Stale Address - (File: contracts/FeeReceiver.sol)

---

### Summary

`FeeReceiver` stores the `depositPool` address as a local state variable set at initialization time. It has no reference to `LRTConfig` and therefore cannot dynamically resolve the canonical `LRT_DEPOSIT_POOL` address. If the protocol upgrades or replaces the deposit pool and updates `LRTConfig`, `FeeReceiver` continues routing all MEV/execution-layer rewards to the old address. Because `sendFunds()` is permissionless, any external caller can trigger this stale routing before the admin manually calls `setDepositPool()`.

---

### Finding Description

`FeeReceiver.initialize()` stores the deposit pool address in the `depositPool` state variable:

```solidity
depositPool = _depositPool;
```

The `sendFunds()` function, which is callable by anyone with no access control, forwards the entire ETH balance to this stored address:

```solidity
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

Meanwhile, the canonical deposit pool address is maintained in `LRTConfig.contractMap` under the key `LRT_DEPOSIT_POOL`, and is updatable by the admin via `LRTConfig.setContract()`. Every other core protocol contract (`LRTOracle`, `LRTDepositPool` itself, `NodeDelegator`) dynamically resolves contract addresses from `LRTConfig` at call time. `FeeReceiver` is the sole exception — it holds no reference to `LRTConfig` and has no mechanism to auto-sync with it.

When the protocol upgrades the deposit pool (a new proxy address is registered in `LRTConfig`), `FeeReceiver.depositPool` remains pointing to the old address. The only remediation path is a manual `setDepositPool()` call by the `MANAGER` role. During the window between the `LRTConfig` update and the `FeeReceiver` update, any caller can invoke `sendFunds()` and permanently route accumulated MEV rewards to the deprecated deposit pool.

---

### Impact Explanation

**Impact: Medium — Permanent freezing of unclaimed yield.**

MEV and execution-layer rewards accumulate in `FeeReceiver` and are forwarded to the deposit pool via `sendFunds()`. If `depositPool` is stale (pointing to a deprecated proxy), the ETH is sent to a contract that is no longer managed by the protocol. The deprecated deposit pool has no path to forward those funds to the new canonical pool. The rewards are effectively frozen — they do not increase TVL in the active pool, do not accrue to rsETH holders, and cannot be recovered without a separate admin rescue operation on the deprecated contract.

---

### Likelihood Explanation

**Likelihood: Medium.**

The LRT-rsETH protocol uses upgradeable proxies and `LRTConfig` as a live registry. Upgrading or replacing the deposit pool is a realistic operational event (the codebase already shows `LRTDepositPool` as an upgradeable contract registered in `LRTConfig`). The inconsistency window is created by a normal governance action, not an attack. The trigger (`sendFunds()`) is permissionless, so any external actor — including a griefing depositor or MEV bot — can exploit the window without any special role.

---

### Recommendation

Remove the local `depositPool` state variable from `FeeReceiver` and instead store a reference to `LRTConfig`. Resolve the deposit pool address dynamically at call time:

```solidity
address lrtConfig; // set at initialization

function sendFunds() external {
    uint256 balance = address(this).balance;
    address pool = ILRTConfig(lrtConfig).getContract(LRTConstants.LRT_DEPOSIT_POOL);
    ILRTDepositPool(pool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

This mirrors the pattern used by every other core protocol contract and ensures `FeeReceiver` always routes rewards to the current canonical deposit pool without requiring a separate manual update.

---

### Proof of Concept

1. Protocol deploys `FeeReceiver` with `depositPool = address(OldDepositPool)`.
2. MEV rewards accumulate in `FeeReceiver` over time.
3. Protocol governance upgrades the deposit pool: `LRTConfig.setContract(LRT_DEPOSIT_POOL, address(NewDepositPool))`.
4. Before the admin calls `FeeReceiver.setDepositPool(address(NewDepositPool))`, an external caller (any address) calls `FeeReceiver.sendFunds()`.
5. `sendFunds()` reads `depositPool` (still `OldDepositPool`) and calls `ILRTDepositPool(OldDepositPool).receiveFromRewardReceiver{value: balance}()`.
6. ETH is transferred to the deprecated `OldDepositPool`. It is no longer tracked by the active protocol TVL, does not increase rsETH price, and is not accessible to rsETH holders.
