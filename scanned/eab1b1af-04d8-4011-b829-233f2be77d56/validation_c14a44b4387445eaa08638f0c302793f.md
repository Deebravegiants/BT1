### Title
Reward Token Dust Permanently Locked Due to Integer Division Truncation in `notifyRewardAmount` - (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary
In `KernelDepositPool.sol`, the `notifyRewardAmount` function computes `rewardRate` by dividing the received token amount by `duration` using integer division. The remainder (`receivedAmount % duration`) is silently discarded and permanently locked in the contract, as there is no token recovery mechanism. This is a direct analog to the original finding's pattern of division-induced token loss.

### Finding Description
In `notifyRewardAmount`, the reward rate is computed as:

```solidity
if (block.timestamp >= finishAt) {
    rewardRate = receivedAmount / duration;
} else {
    uint256 remaining = (finishAt - block.timestamp) * rewardRate;
    rewardRate = (receivedAmount + remaining) / duration;
}
``` [1](#0-0) 

Both branches perform integer division. The truncated remainder — up to `duration - 1` tokens — is transferred into the contract but never accounted for in `rewardRate`. Since `rewardRate * duration < receivedAmount` whenever `receivedAmount % duration != 0`, the difference is stranded in the contract's `rewardsToken` balance.

There is no `sweep`, `rescue`, or `recoverTokens` function anywhere in `KernelDepositPool.sol`. [2](#0-1) 

The `duration` field is set in seconds (e.g., 7 days = 604,800 seconds). [3](#0-2) 

With reward tokens denominated in 1e18 units, the remainder per call can be up to `604,799` wei — small per call but accumulating across every `notifyRewardAmount` invocation over the protocol's lifetime.

### Impact Explanation
Every call to `notifyRewardAmount` where `receivedAmount % duration != 0` permanently locks `receivedAmount % duration` reward tokens in the contract. These tokens are transferred in but never distributed to stakers. There is no recovery path. This constitutes **permanent freezing of unclaimed yield** for KERNEL stakers.

### Likelihood Explanation
This occurs in normal protocol operation. `duration` is set in seconds (e.g., 604,800 for 7 days), while reward amounts are in token units (e.g., multiples of 1e18). It is practically certain that `receivedAmount % duration != 0` on every realistic call, since there is no input validation requiring the amount to be a multiple of `duration`. The loss compounds with each reward epoch.

### Recommendation
Add a require check in `notifyRewardAmount` to enforce that the received amount is a multiple of `duration`:

```solidity
require(receivedAmount % duration == 0, "Amount not multiple of duration");
```

Alternatively, track the undistributed dust and roll it into the next reward period explicitly, or add an admin token-rescue function for the `rewardsToken`.

### Proof of Concept
1. Admin sets `duration = 604800` (7 days in seconds) via `setRewardsDuration`.
2. Admin calls `notifyRewardAmount(1e18)` (1 token).
3. `receivedAmount = 1e18`, `duration = 604800`.
4. `rewardRate = 1e18 / 604800 = 1653439153439` (truncated).
5. Total distributed over the period: `1653439153439 * 604800 = 999999999999897600` (≈ 0.9999... tokens).
6. Stranded dust: `1e18 - 999999999999897600 = 102400` wei, permanently locked.
7. Repeated over 100 epochs: `~10,240,000` wei locked with no recovery path. [1](#0-0)

    function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
        if (_amount == 0) revert AmountZero();

        // Prevent starting a reward period when no tokens are staked to avoid unallocated rewards
        if (totalKernelStaked == 0) revert NoStakedTokens();

        // Transfer reward tokens into the contract
        uint256 balanceBefore = rewardsToken.balanceOf(address(this));
        rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
        uint256 balanceAfter = rewardsToken.balanceOf(address(this));
        // Calculate the actual amount of tokens received in case of a transfer fee (tax)
        uint256 receivedAmount = balanceAfter - balanceBefore;

        if (block.timestamp >= finishAt) {
            rewardRate = receivedAmount / duration;
        } else {
            uint256 remaining = (finishAt - block.timestamp) * rewardRate;
            rewardRate = (receivedAmount + remaining) / duration;
        }

        if (rewardRate == 0) revert RewardRateZero();

        finishAt = block.timestamp + duration;
        updatedAt = block.timestamp;

        emit NotifyRewardAmount(receivedAmount, finishAt);
    }
```
