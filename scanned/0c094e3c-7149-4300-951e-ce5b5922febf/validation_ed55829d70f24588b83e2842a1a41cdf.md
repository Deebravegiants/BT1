### Title
Reward Token Dust Permanently Locked Due to Integer Division Truncation in `notifyRewardAmount` - (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary

`KernelDepositPool.notifyRewardAmount` computes `rewardRate` via integer floor division of `receivedAmount / duration`. The remainder `receivedAmount % duration` is never distributed and is permanently locked in the contract with no recovery path. This is structurally identical to the RaptorCast bug: a proportional-allocation formula uses floor division, causing the total actually distributed to be strictly less than the total deposited, with the shortfall irrecoverable.

### Finding Description

In `notifyRewardAmount`, the reward rate is set as:

```solidity
rewardRate = receivedAmount / duration;          // line 580
// or, mid-period:
rewardRate = (receivedAmount + remaining) / duration;  // line 583
``` [1](#0-0) 

The total rewards that will actually be emitted over the full period is `rewardRate * duration`, which equals `receivedAmount - (receivedAmount % duration)`. The remainder `receivedAmount % duration` stays in the contract's `rewardsToken` balance permanently. There is no sweep, rescue, or recovery function for `rewardsToken` in the contract. [2](#0-1) 

The `rewardPerToken` and `earned` accumulators are also subject to floor division:

```solidity
return rewardPerTokenStored
    + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
    / totalKernelStaked;
``` [3](#0-2) 

```solidity
return (balanceOf[_account] * (rewardPerToken() - userRewardPerTokenPaid[_account])
    / DECIMAL_PRECISION) + rewards[_account];
``` [4](#0-3) 

Each of these truncations compounds the under-distribution. The primary and most impactful loss is the `receivedAmount % duration` dust locked at `notifyRewardAmount` time.

### Impact Explanation

Every call to `notifyRewardAmount` permanently locks up to `duration - 1` reward tokens. With the maximum allowed `duration` of `30 days = 2,592,000 seconds`, up to `2,591,999` base units of `rewardsToken` are irrecoverable per call. Over multiple reward periods this accumulates. Stakers collectively receive less yield than was deposited into the contract. The locked tokens cannot be claimed by any party — they are not yield to the treasury, not claimable by stakers, and not recoverable by admin. This constitutes **permanent freezing of unclaimed yield**. [5](#0-4) 

### Likelihood Explanation

This triggers on every `notifyRewardAmount` call unless `receivedAmount` is exactly divisible by `duration`. Since `duration` is a large integer (seconds) and `receivedAmount` is an arbitrary token amount chosen by the admin, exact divisibility is virtually never satisfied in practice. The loss is guaranteed on every reward epoch.

### Recommendation

After computing `rewardRate`, carry the remainder forward into the next period rather than discarding it:

```solidity
if (block.timestamp >= finishAt) {
    rewardRate = receivedAmount / duration;
    // carry dust: add (receivedAmount % duration) to next notifyRewardAmount call
    // or: rewardRate = receivedAmount / duration; leftover = receivedAmount % duration;
} else {
    uint256 remaining = (finishAt - block.timestamp) * rewardRate;
    rewardRate = (receivedAmount + remaining) / duration;
}
```

A cleaner fix is to track `undistributedDust` and add it to `receivedAmount` on the next `notifyRewardAmount` call, ensuring no tokens are ever stranded.

### Proof of Concept

Concrete example:
- `duration = 2,592,000` (30 days in seconds)
- Admin calls `notifyRewardAmount` with `receivedAmount = 1,000,000` tokens (18-decimal token, so `1e6 * 1e18` wei)
- `rewardRate = 1_000_000e18 / 2_592_000 = 385,802,469,135,802` per second
- Total emitted: `385,802,469,135,802 * 2,592,000 = 999,999,999,999,997,184` wei
- Permanently locked: `1,000,000e18 - 999,999,999,999,997,184 = 2,816` wei

For a token with 6 decimals and `receivedAmount = 1_000_000` (1 token):
- `rewardRate = 1_000_000 / 2_592_000 = 0` → `RewardRateZero` revert (edge case caught)

For `receivedAmount = 3_000_000` (3 tokens, 6 decimals):
- `rewardRate = 3_000_000 / 2_592_000 = 1` per second
- Total emitted: `2,592,000`
- Permanently locked: `408,000` base units = 0.408 tokens lost per epoch [6](#0-5)

    function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
        if (_amount == 0) revert AmountZero();

        // Prevent starting a reward period when no tokens are staked to avoid unallocated rewards
        if (totalKernelStaked == 0) revert NoStakedTokens();

        // Transfer reward tokens into the contract
        uint256 balanceBefore = rewardsToken.balanceOf(address(this));
        rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
        uint256 balanceAfter = rewardsToken.balanceOf(address(this));
        // Calculate the actual amount of tokens received in case of a transfer fee (tax)
        uint256 receivedAmount = balanceAfter - balanceBefore;

        if (block.timestamp >= finishAt) {
            rewardRate = receivedAmount / duration;
        } else {
            uint256 remaining = (finishAt - block.timestamp) * rewardRate;
            rewardRate = (receivedAmount + remaining) / duration;
        }

        if (rewardRate == 0) revert RewardRateZero();

        finishAt = block.timestamp + duration;
        updatedAt = block.timestamp;

        emit NotifyRewardAmount(receivedAmount, finishAt);
    }
