### Title
Hardcoded `DECIMAL_PRECISION = 1e18` Causes Complete Reward Accrual Failure When Reward Token Has Fewer Than 18 Decimals - (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary
`KernelDepositPool` hardcodes `DECIMAL_PRECISION = 1e18` as a constant while accepting an arbitrary `rewardsToken` that may have fewer than 18 decimals (e.g., USDC with 6 decimals). When `totalKernelStaked` grows large enough relative to `rewardRate`, the `rewardPerToken()` increment truncates to zero on every call, permanently freezing all unclaimed yield inside the contract.

### Finding Description
`DECIMAL_PRECISION` is declared as a hardcoded `constant` at line 32:

```solidity
uint256 public constant DECIMAL_PRECISION = 1e18;
```

The `rewardsToken` is set freely in `initialize()` with no decimal constraint:

```solidity
rewardsToken = IERC20(_rewardToken);
```

`rewardRate` is computed in `notifyRewardAmount()` in the reward token's native units:

```solidity
rewardRate = receivedAmount / duration;
```

`rewardPerToken()` then uses `DECIMAL_PRECISION` as a fixed scaling factor:

```solidity
return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
    / totalKernelStaked;
```

If `rewardsToken` is USDC (6 decimals), `rewardRate` is expressed in units of `1e6/second`. The numerator of the increment is `rewardRate * timeDelta * 1e18`. The denominator is `totalKernelStaked` in units of `1e18` (KERNEL is 18 decimals). The effective computation per second becomes:

```
rewardRate_usdc * 1e18 / totalKernelStaked_kernel
= (e.g.) 1e6 * 1e18 / totalKernelStaked
```

Once `totalKernelStaked > rewardRate * 1e18`, integer division truncates to 0. For a reward rate of 1 USDC/second, this threshold is just **1,000,000 KERNEL tokens** staked — a realistic TVL for any meaningful protocol deployment.

`earned()` compounds the problem: since `rewardPerToken()` never increases, `rewardPerToken() - userRewardPerTokenPaid[_account]` is always 0, so every user's earned amount stays at 0 regardless of how long they stake.

### Impact Explanation
**Medium — Permanent freezing of unclaimed yield.**

All stakers lose 100% of their accrued rewards once `totalKernelStaked` crosses the truncation threshold. The reward tokens deposited by the admin via `notifyRewardAmount()` remain permanently locked in the contract with no recovery mechanism. No individual user can unilaterally fix this; the entire reward distribution is silently broken.

### Likelihood Explanation
The `rewardsToken` is configurable and the contract is explicitly designed to support arbitrary ERC-20 reward tokens. USDC (6 decimals) is the most common non-18-decimal reward token in DeFi staking contracts. The truncation threshold (e.g., 1M KERNEL for 1 USDC/second) is easily reachable in normal protocol operation. No attacker action is required — organic staking growth triggers the condition.

### Recommendation
Replace the hardcoded `DECIMAL_PRECISION` constant with an immutable variable set in the initializer based on the reward token's decimals:

```solidity
uint256 public immutable DECIMAL_PRECISION;

function initialize(..., address _rewardToken) external initializer {
    ...
    uint8 rewardDecimals = IERC20Metadata(_rewardToken).decimals();
    DECIMAL_PRECISION = 10 ** rewardDecimals;
    ...
}
```

Alternatively, always scale `rewardRate` up by `1e18 / 10**rewardDecimals` at the time of `notifyRewardAmount()` so that internal accounting is always in 1e18 precision regardless of the reward token's native decimals.

### Proof of Concept

**Setup:**
- `rewardsToken` = USDC (6 decimals)
- `kernelToken` = KERNEL (18 decimals)
- `duration` = 30 days = 2,592,000 seconds
- Admin calls `notifyRewardAmount(2_592_000e6)` → `rewardRate = 2_592_000e6 / 2_592_000 = 1e6` (1 USDC/second)
- `totalKernelStaked` = 1,000,001e18 (just over 1M KERNEL)

**`rewardPerToken()` increment per second:**
```
rewardRate * 1 * DECIMAL_PRECISION / totalKernelStaked
= 1e6 * 1 * 1e18 / 1_000_001e18
= 1e24 / 1_000_001e18
= 0  (integer truncation)
```

**Result:** `rewardPerTokenStored` never increases. Every call to `earned(user)` returns `rewards[user]` (unchanged). All stakers receive 0 rewards for the entire 30-day period.
