### Title
Stale Cross-Chain rsETH Rate in `RSETHRateReceiver` Enables Excess wrsETH Minting via Token Deposit — (`contracts/cross-chain/CrossChainRateReceiver.sol`, `contracts/pools/RSETHPoolV3.sol`)

### Summary

`CrossChainRateReceiver.getRate()` returns the last stored rate with no staleness check. `RSETHPoolV3.viewSwapRsETHAmountAndFee(amount, token)` divides a fresh Chainlink `tokenToETHRate` by this potentially stale `rsETHToETHrate`. When the stored rsETH/ETH rate is stale-low (rsETH has appreciated since the last LayerZero update) while the token Chainlink oracle is fresh, an unprivileged depositor receives more wrsETH than the deposited collateral is worth at current market prices.

---

### Finding Description

**Rate update mechanism — no access control, no staleness guard**

`CrossChainRateProvider.updateRate()` is a public payable function with only a `nonReentrant` guard. [1](#0-0) 

The concrete `RSETHRateProvider.getLatestRate()` reads `ILRTOracle(rsETHPriceOracle).rsETHPrice()` at call time and broadcasts it via LayerZero. [2](#0-1) 

On the L2 side, `CrossChainRateReceiver.lzReceive()` stores the received value in `rate` and records `lastUpdated = block.timestamp`. [3](#0-2) 

`CrossChainRateReceiver.getRate()` returns `rate` unconditionally — there is no comparison of `lastUpdated` against any maximum-staleness threshold. [4](#0-3) 

**Deposit math — two oracles, one stale**

`RSETHPoolV3.deposit(token, amount, referralId)` calls `viewSwapRsETHAmountAndFee(amount, token)`: [5](#0-4) 

The formula is:

```
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate
```

- `rsETHToETHrate` comes from `getRate()` → `IOracle(rsETHOracle).getRate()` → `CrossChainRateReceiver.getRate()` (stale, no freshness check).
- `tokenToETHRate` comes from `IOracle(supportedTokenOracle[token]).getRate()` — a Chainlink oracle that updates continuously. [6](#0-5) 

**Exploit scenario (concrete numbers)**

| Variable | Stale state | True current |
|---|---|---|
| rsETH/ETH (`rsETHToETHrate`) | 1.050e18 | 1.100e18 |
| wstETH/ETH (`tokenToETHRate`) | — | 1.080e18 (fresh Chainlink) |

Attacker deposits 1 wstETH (fee = 0 for simplicity):

- **Received**: `1e18 * 1.080e18 / 1.050e18 ≈ 1.0286 wrsETH`
- **Fair value**: `1e18 * 1.080e18 / 1.100e18 ≈ 0.9818 wrsETH`
- **Excess minted**: ~4.8% per deposit, unbounded by repetition

The attacker deposits real wstETH and receives wrsETH that is worth more than the deposited collateral at current rates. When the excess wrsETH is redeemed (via the reverse swap or bridging), it drains pool reserves.

---

### Impact Explanation

The pool mints wrsETH via `wrsETH.mint(msg.sender, rsETHAmount)` with no cap other than the daily mint limit. [7](#0-6) 

The daily mint limit (`dailyMintLimit`) only bounds the total rsETH-equivalent minted per day, not the per-unit over-issuance ratio. An attacker can extract the maximum daily allowance at the favorable stale rate, then repeat the next day. The pool's wstETH reserves are progressively undercollateralized relative to outstanding wrsETH, constituting direct theft of pool reserves.

---

### Likelihood Explanation

- `updateRate()` is a manually triggered, permissionless function. There is no keeper or on-chain automation enforcing a maximum update interval.
- LayerZero cross-chain message delivery introduces additional latency.
- rsETH accrues staking yield continuously, so its price relative to ETH increases monotonically over time. Any gap between updates creates a stale-low condition.
- The attacker requires no special role, no front-running, and no oracle manipulation — only the ability to observe that `CrossChainRateReceiver.lastUpdated` is old and that rsETH has appreciated since then.

---

### Recommendation

1. **Add a staleness check in `CrossChainRateReceiver.getRate()`**: revert (or return 0) if `block.timestamp - lastUpdated > MAX_STALENESS` (e.g., 24 hours).
2. **Add a staleness check in `RSETHPoolV3.getRate()`**: validate the oracle's `lastUpdated` before using the rate in deposit math.
3. Consider using a pull-based oracle (e.g., Chainlink on L2 for rsETH) instead of a push-based cross-chain rate that can go stale.

---

### Proof of Concept

```solidity
// Fork test (L2 fork, e.g., Arbitrum)
// 1. Deploy or use existing RSETHRateReceiver with lastUpdated = block.timestamp - 2 days
//    and rate = 1.05e18 (stale-low; true current rsETH/ETH = 1.10e18)
// 2. Chainlink wstETH/ETH oracle returns 1.08e18 (fresh)
// 3. Attacker calls RSETHPoolV3.deposit(wstETH, 1e18, "")
// 4. Assert:
//    wrsETH_received = 1e18 * 1.08e18 / 1.05e18 ≈ 1.0286e18
//    fair_wrsETH     = 1e18 * 1.08e18 / 1.10e18 ≈ 0.9818e18
//    assert(wrsETH_received > fair_wrsETH);  // excess ~4.8%
// 5. Repeat up to dailyMintLimit to maximize extraction
```

The invariant `wrsETH_received <= deposited_token_value / true_rsETH_rate` is violated whenever the cross-chain rate is stale-low, which occurs naturally without any attacker action.

        uint256 amount,
        address token
    )
        public
        view
        onlySupportedToken(token)
        returns (uint256 rsETHAmount, uint256 fee)
    {
        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // rate of token in ETH
        uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
    }
