### Title
Reward Tokens Permanently Locked When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: `contracts/KERNEL/KernelDepositPool.sol`)

### Summary
`KernelDepositPool` distributes reward tokens to KERNEL stakers over a fixed `duration`. When `totalKernelStaked` drops to zero mid-period — which any staker can cause by calling `initiateWithdrawal` — the `rewardPerToken()` accumulator freezes and all remaining rewards for that period are permanently locked in the contract with no recovery path.

### Finding Description

`rewardPerToken()` contains a zero-guard:

```solidity
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;  // accumulation frozen
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
```

When `totalKernelStaked == 0`, the function returns the stored checkpoint without advancing it. Any rewards that accrue during this zero-staking window (`rewardRate × elapsed`) are never credited to any account and cannot be recovered.

`initiateWithdrawal` immediately reduces `totalKernelStaked` with no floor:

```solidity
balanceOf[msg.sender] -= _amount;
totalKernelStaked -= _amount;
```

The only guard against this scenario is in `notifyRewardAmount`:

```solidity
if (totalKernelStaked == 0) revert NoStakedTokens();
```

This prevents *starting* a new period with zero staked, but does nothing to prevent `totalKernelStaked` from reaching zero *during* an active period. Once it does, the remaining rewards are permanently stranded — there is no `sweep`, `recoverERC20`, or any other admin function that can retrieve them.

The contract's own NatSpec acknowledges this:

> *"If `totalKernelStaked` ever hits zero during a reward distribution window, any remaining rewards for that period will stay locked in the contract."*

The stated mitigation is purely operational ("ensuring there are always some tokens staked"), with no on-chain enforcement.

A second, always-present locking mechanism compounds this: integer truncation in `notifyRewardAmount`:

```solidity
rewardRate = receivedAmount / duration;          // new period
rewardRate = (receivedAmount + remaining) / duration;  // mid-period rollover
```

The remainder `(receivedAmount [+ remaining]) % duration` is transferred in but can never be distributed, permanently locking up to `duration − 1` wei of reward tokens per call.

### Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

Reward tokens deposited by the admin for distribution to stakers become permanently unclaimable. The contract holds the tokens but has no mechanism to distribute or recover them once `totalKernelStaked` hits zero mid-period. The truncation dust accumulates silently across every `notifyRewardAmount` call.

### Likelihood Explanation

**Medium.** Any staker can call `initiateWithdrawal` at any time without restriction. A single-staker scenario (common in early protocol life) means one user exiting mid-period is sufficient. In a multi-staker scenario, coordinated or panic exits (e.g., reward token price collapse) can drain `totalKernelStaked` to zero. No on-chain mechanism prevents this.

### Recommendation

1. **Add a recovery function** (e.g., `recoverUnclaimedRewards`) callable by admin after a period ends, to sweep any reward token surplus back to the treasury.
2. **Alternatively**, track `undistributedRewards` explicitly and roll them into the next period via `notifyRewardAmount`, similar to how `remaining` is already rolled over.
3. For the truncation dust, accumulate the remainder and add it to the next period's `receivedAmount`.

### Proof of Concept

```
1. Admin calls notifyRewardAmount(1_000e18) with duration = 30 days
   → rewardRate = 1_000e18 / 2_592_000 ≈ 385_802_469_135_802 tokens/sec
   → 1_000e18 - rewardRate * 2_592_000 = dust locked immediately (truncation)

2. Alice is the sole staker: balanceOf[alice] = 100e18, totalKernelStaked = 100e18

3. After 15 days, Alice calls initiateWithdrawal(100e18)
   → totalKernelStaked = 0

4. For the remaining 15 days:
   → rewardPerToken() returns frozen rewardPerTokenStored
   → ~500e18 reward tokens accrue at rewardRate but are credited to no one

5. Alice calls getReward() → receives only ~500e18 (15 days of rewards)
6. ~500e18 reward tokens remain in the contract forever, unclaimable by anyone

    function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
        if (_amount == 0) revert AmountZero();

        // Prevent starting a reward period when no tokens are staked to avoid unallocated rewards
        if (totalKernelStaked == 0) revert NoStakedTokens();

        // Transfer reward tokens into the contract
        uint256 balanceBefore = rewardsToken.balanceOf(address(this));
        rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
        uint256 balanceAfter = rewardsToken.balanceOf(address(this));
        // Calculate the actual amount of tokens received in case of a transfer fee (tax)
        uint256 receivedAmount = balanceAfter - balanceBefore;

        if (block.timestamp >= finishAt) {
            rewardRate = receivedAmount / duration;
        } else {
            uint256 remaining = (finishAt - block.timestamp) * rewardRate;
            rewardRate = (receivedAmount + remaining) / duration;
        }

        if (rewardRate == 0) revert RewardRateZero();

        finishAt = block.timestamp + duration;
        updatedAt = block.timestamp;

        emit NotifyRewardAmount(receivedAmount, finishAt);
