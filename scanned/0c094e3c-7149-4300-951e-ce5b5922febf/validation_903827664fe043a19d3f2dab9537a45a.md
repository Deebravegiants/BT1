### Title
Missing Slippage Protection on L2 Pool `deposit()` Functions - (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPoolV3WithNativeChainBridge.sol, contracts/pools/RSETHPool.sol, contracts/pools/RSETHPoolNoWrapper.sol)

---

### Summary

All L2 pool `deposit()` functions that swap ETH or LSTs for wrsETH/rsETH lack a `minRSETHAmountExpected` parameter, meaning depositors cannot specify a minimum amount of rsETH to receive. The mainnet `LRTDepositPool` correctly implements this protection, but the L2 pool variants do not.

---

### Finding Description

The mainnet `LRTDepositPool.depositETH()` and `LRTDepositPool.depositAsset()` both accept a `minRSETHAmountExpected` parameter and revert with `MinimumAmountToReceiveNotMet` if the minted rsETH falls below it: [1](#0-0) [2](#0-1) 

In contrast, every L2 pool `deposit()` function computes the rsETH amount solely from the oracle rate at execution time and immediately mints/transfers it, with no user-specified minimum:

**`RSETHPoolV3`** — ETH deposit: [3](#0-2) 

**`RSETHPoolV3`** — token deposit: [4](#0-3) 

The same pattern is present in `RSETHPoolV3ExternalBridge`: [5](#0-4) 

In `RSETHPoolV3WithNativeChainBridge`: [6](#0-5) 

In `RSETHPool`: [7](#0-6) 

And in `RSETHPoolNoWrapper`: [8](#0-7) 

The rsETH amount is computed as:

```
rsETHAmount = (amount - fee) * 1e18 / rsETHToETHrate
```

where `rsETHToETHrate` is fetched live from the oracle at execution time. [9](#0-8) 

If the oracle rate increases between the time the user signs the transaction and the time it is included in a block, the user receives fewer wrsETH/rsETH than they anticipated, with no on-chain mechanism to revert the transaction.

---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A depositor who previewed the exchange rate off-chain (e.g., via `viewSwapRsETHAmountAndFee`) and submitted a transaction may receive materially fewer wrsETH/rsETH than expected if the oracle rate moves before their transaction executes. The deposited ETH/LST is not returned; it is accepted by the pool at the new, less favorable rate. The user's funds are not stolen, but the contract does not deliver the rsETH amount the user reasonably expected.

---

### Likelihood Explanation

**Medium.** Oracle rates for rsETH are updated periodically on-chain. On L2 networks with public mempools, a user's pending deposit transaction is visible and can be front-run by an oracle update or by a searcher who triggers an oracle update before the deposit lands. Additionally, ordinary network congestion can delay a transaction long enough for the rate to shift. This is a realistic, unprivileged scenario requiring no admin compromise.

---

### Recommendation

Add a `minRSETHAmountExpected` parameter to all L2 pool `deposit()` functions, mirroring the mainnet `LRTDepositPool` pattern:

```solidity
// RSETHPoolV3 (and all analogous L2 pool variants)
function deposit(
    string memory referralId,
    uint256 minRSETHAmountExpected   // <-- add
) external payable nonReentrant whenNotPaused limitDailyMint(msg.value, ETH_IDENTIFIER) {
    if (!isEthDepositEnabled) revert EthDepositDisabled();
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();

    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);

    if (rsETHAmount < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet(); // <-- add

    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
```

Apply the same change to the token-deposit overload and to `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPool`, and `RSETHPoolNoWrapper`.

---

### Proof of Concept

1. User calls `viewSwapRsETHAmountAndFee(1 ether)` on `RSETHPoolV3` and sees they will receive `X` wrsETH at the current oracle rate.
2. User submits `deposit{value: 1 ether}("ref")`.
3. Before the transaction is mined, the rsETH oracle rate increases (e.g., due to a scheduled oracle update or a searcher-triggered update).
4. The deposit executes at the new, higher rate, minting `X' < X` wrsETH to the user.
5. The user has no recourse — the transaction succeeded and their ETH is in the pool.

No special privileges are required. Any depositor on any L2 chain where these pools are deployed is exposed. [3](#0-2) [10](#0-9)

    function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
    }
