### Title
`updateRSETHPrice()` Permanently Blocked When `maxFeeMintAmountPerDay` Is Zero and Protocol Fees Accrue - (File: `contracts/LRTOracle.sol`)

---

### Summary

`LRTOracle.updateRSETHPrice()` is a public function that any caller can invoke to refresh the rsETH/ETH exchange rate. Inside `_updateRsETHPrice()`, when the protocol has a non-zero fee rate and TVL has grown, a fee amount is computed and passed to `_checkAndUpdateDailyFeeMintLimit()`. If `maxFeeMintAmountPerDay` is zero — its default value after `initialize()` — the check `currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay` evaluates to `feeAmount > 0`, which is `true`, causing the entire price update to revert with `DailyFeeMintLimitExceeded`. The rsETH price is then frozen at its last stored value.

---

### Finding Description

`LRTOracle.initialize()` does not set `maxFeeMintAmountPerDay`; it defaults to `0`. The value is only set later via `setMaxFeeMintAmountPerDay()` by the manager. Until that call is made — or if the manager intentionally sets it to `0` to disable fee minting — the following path in `_updateRsETHPrice()` is permanently broken whenever rewards accrue:

```solidity
// contracts/LRTOracle.sol lines 244-246
if (!protocolPaused && totalETHInProtocol > previousTVL) {
    uint256 rewardAmount = totalETHInProtocol - previousTVL;
    protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
}
``` [1](#0-0) 

When `protocolFeeInETH > 0`, the code calls:

```solidity
// contracts/LRTOracle.sol lines 301-303
uint256 rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice);
_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee);
``` [2](#0-1) 

Inside `_checkAndUpdateDailyFeeMintLimit`:

```solidity
// contracts/LRTOracle.sol lines 205-207
if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
    revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
}
``` [3](#0-2) 

When `maxFeeMintAmountPerDay == 0` and `feeAmount > 0`, the condition `feeAmount > 0` is always `true`, so the revert fires unconditionally. The price update never reaches `rsETHPrice = newRsETHPrice`. [4](#0-3) 

The public entry point is:

```solidity
// contracts/LRTOracle.sol line 87
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [5](#0-4) 

`initialize()` never sets `maxFeeMintAmountPerDay`, leaving it at `0`: [6](#0-5) 

The same issue applies when the daily limit is legitimately exhausted mid-day: subsequent calls to `updateRSETHPrice()` that would generate any fee revert for the remainder of the day.

---

### Impact Explanation

The stored `rsETHPrice` is consumed directly by `LRTDepositPool.getRsETHAmountToMint()`:

```solidity
// contracts/LRTDepositPool.sol line 520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [7](#0-6) 

A stale (lower-than-actual) `rsETHPrice` causes depositors to receive more rsETH than they are entitled to, diluting existing holders' accrued yield. Withdrawers receive less than they are owed. The protocol's promised yield accounting is broken for the duration of the blockage.

**Impact: Medium — Temporary freezing of unclaimed yield / contract fails to deliver promised returns.**

---

### Likelihood Explanation

`maxFeeMintAmountPerDay` is `0` by default after `initialize()`. Any deployment where the manager has not yet called `setMaxFeeMintAmountPerDay()` — or has set it to `0` to pause fee minting — is immediately vulnerable as soon as `protocolFeeInBPS > 0` and TVL grows. This is the default post-deployment state and requires no attacker action; any public caller of `updateRSETHPrice()` will trigger the revert.

**Likelihood: Medium.**

---

### Recommendation

Decouple the fee-minting step from the price-update step. When the daily fee limit is zero or exhausted, skip the fee mint but still commit the new price:

```solidity
if (protocolFeeInETH > 0) {
    uint256 rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice);
    // Only attempt fee mint if limit allows; otherwise skip silently
    if (maxFeeMintAmountPerDay > 0 && currentPeriodMintedFeeAmount + rsethAmountToMintAsProtocolFee <= maxFeeMintAmountPerDay) {
        _checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee);
        // mint ...
    }
} else {
    _checkAndUpdateDailyFeeMintLimit(0);
}
// Always update the price
rsETHPrice = newRsETHPrice;
```

Alternatively, validate that `maxFeeMintAmountPerDay` is non-zero in `initialize()` or `reinitialize()` before enabling fee collection.

---

### Proof of Concept

1. Deploy `LRTOracle` and call `initialize(lrtConfigAddr)`. `maxFeeMintAmountPerDay` is `0`.
2. Configure `LRTConfig` with `protocolFeeInBPS = 500` (5% fee).
3. Allow EigenLayer rewards to accrue so that `totalETHInProtocol > previousTVL`.
4. Call `LRTOracle.updateRSETHPrice()` from any EOA.
5. Execution enters `_updateRsETHPrice()`, computes `protocolFeeInETH > 0`, computes `rsethAmountToMintAsProtocolFee > 0`, calls `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)`.
6. Check: `0 + rsethAmountToMintAsProtocolFee > 0` → `true` → reverts with `DailyFeeMintLimitExceeded(rsethAmountToMintAsProtocolFee, 0)`.
7. `rsETHPrice` is never updated. All subsequent deposits and withdrawals use the stale price, over-minting rsETH for depositors and under-paying withdrawers.
