### Title
`CREATE3Factory.deploy` Does Not Bind Salt to `msg.sender`, Enabling Front-Run Deployment Hijacking - (File: contracts/utils/CREATE3Factory.sol)

### Summary
`CREATE3Factory.deploy` is a permissionless function that passes the caller-supplied `salt` directly to `CREATE3.deploy` without incorporating `msg.sender`. Because the resulting deployment address is determined solely by `(factory_address, salt)`, any attacker who observes a pending deployment transaction in the mempool can front-run it with the same salt but arbitrary malicious `creationCode`, seizing the expected address before the legitimate deployer.

### Finding Description
`CREATE3Factory.deploy` at [1](#0-0)  forwards the raw `salt` to `CREATE3.deploy`:

```solidity
function deploy(bytes32 salt, bytes memory creationCode) external payable returns (address deployed) {
    deployed = CREATE3.deploy(salt, creationCode, msg.value);
    emit ContractDeployed(salt, deployed);
}
```

Inside `CREATE3.deploy`, the proxy is deployed via `create2` using the salt verbatim:

```solidity
proxy := create2(0, add(proxyChildBytecode, 32), mload(proxyChildBytecode), salt)
``` [2](#0-1) 

The final deployed address is then computed as the CREATE1 address of that proxy (nonce = 1):

```solidity
return keccak256(abi.encodePacked(hex"d694", proxy, hex"01")).fromLast20Bytes();
``` [3](#0-2) 

Because neither `CREATE3Factory.deploy` nor `CREATE3.deploy` mixes `msg.sender` into the salt, the proxy's CREATE2 address — and therefore the final contract address — is identical for every caller who supplies the same `salt`. There is no access control on `deploy`. [1](#0-0) 

### Impact Explanation
An attacker who front-runs a legitimate `CREATE3Factory.deploy(salt, legitimateCode)` call with `CREATE3Factory.deploy(salt, maliciousCode)` will:
1. Occupy the proxy's CREATE2 slot (proxy already exists; the legitimate tx's `create2` returns `address(0)` and reverts with `"DEPLOYMENT_FAILED"`).
2. Have a malicious contract sitting at the address the protocol intended to use.

If the protocol pre-announces or hard-codes the expected deployment address (e.g., in cross-chain configuration, UI, or other contracts — consistent with the factory's stated purpose: *"This factory can be deployed at the same address on multiple chains"*), users and integrated contracts will interact with the attacker's contract instead. This can result in direct theft of deposited funds. Impact: **Critical**.

### Likelihood Explanation
The factory is deployed on-chain with no access control. Any pending `deploy` transaction is visible in the public mempool. An attacker only needs to submit the same salt with higher gas. No privileged access, leaked keys, or oracle compromise is required. Likelihood: **Medium** (requires mempool monitoring and front-running capability, which is routine on EVM chains).

### Recommendation
Bind the effective salt to `msg.sender` inside `CREATE3Factory.deploy` before passing it to the library:

```solidity
function deploy(bytes32 salt, bytes memory creationCode) external payable returns (address deployed) {
    bytes32 guardedSalt = keccak256(abi.encodePacked(msg.sender, salt));
    deployed = CREATE3.deploy(guardedSalt, creationCode, msg.value);
    emit ContractDeployed(salt, deployed);
}
```

This ensures that two different callers supplying the same `salt` value will produce different proxy addresses, making front-running impossible.

### Proof of Concept
1. The protocol broadcasts `CREATE3Factory.deploy(salt, legitimatePoolCode)` to deploy a new pool at a pre-announced address.
2. An attacker's bot monitors the mempool, extracts `salt`, and submits `CREATE3Factory.deploy(salt, maliciousCode)` with higher gas.
3. The attacker's transaction is mined first. The proxy is deployed at `keccak256(0xFF, factory, salt, PROXY_BYTECODE_HASH)` and the malicious contract lands at the expected address.
4. The protocol's transaction reverts at line 46 of `CREATE3.sol` (`require(proxy != address(0), "DEPLOYMENT_FAILED")`) because the proxy slot is already occupied.
5. Users who deposit ETH/LSTs into the pre-announced address interact with the attacker's contract, losing their funds.
