### Title
No Staleness Check on Chainlink `latestRoundData()` Allows Stale Asset Prices to Drive rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards the `updatedAt` timestamp, meaning a Chainlink feed that has stopped updating will continue to return its last known price indefinitely. This stale price flows directly into rsETH minting arithmetic in `LRTDepositPool`, enabling depositors to receive more rsETH than the true value of their deposit warrants.

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the Chainlink price as follows:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

The five return values of `latestRoundData()` are `(roundId, answer, startedAt, updatedAt, answeredInRound)`. The `updatedAt` field — the on-chain timestamp of the last oracle update — is completely discarded. No check of the form `require(block.timestamp - updatedAt <= maxStaleness)` exists anywhere in the function. [1](#0-0) 

This price is consumed by `LRTOracle.getAssetPrice()`, which is called inside `_getTotalEthInProtocol()` during every `updateRSETHPrice()` execution, and also directly inside `LRTDepositPool.getRsETHAmountToMint()`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [2](#0-1) 

If a Chainlink feed for any supported LST (stETH, ETHx, rETH, swETH, sfrxETH) stops updating — due to oracle downtime, sequencer issues, or extreme market conditions — the last reported price is used as if it were current. Chainlink feeds have heartbeat intervals (e.g., 24 hours for some LST/ETH feeds); a feed can legally go 24 hours without an update and still pass all on-chain checks.

By contrast, `ChainlinkOracleForRSETHPoolCollateral` (used for pool collateral) does at least check `answeredInRound < roundID` and `timestamp == 0`, showing the team is aware of partial staleness guards — but `ChainlinkPriceOracle` has none at all. [3](#0-2) 

### Impact Explanation

If the Chainlink feed for an LST reports a price that is stale-high (e.g., stETH was 1.05 ETH at last update but has since de-pegged to 0.95 ETH):

- `getAssetPrice(stETH)` returns 1.05e18 instead of 0.95e18.
- `getRsETHAmountToMint` mints `amount * 1.05e18 / rsETHPrice` rsETH — ~10.5% more than the depositor's assets are worth.
- Every subsequent depositor and withdrawer is diluted; the protocol becomes insolvent relative to its actual TVL.

Conversely, a stale-low price causes under-minting, directly harming depositors who receive fewer rsETH tokens than their deposit warrants.

**Impact class:** Critical — protocol insolvency / direct theft of funds from existing rsETH holders.

### Likelihood Explanation

Chainlink LST/ETH feeds (e.g., stETH/ETH) have 24-hour heartbeat windows and 0.5% deviation thresholds. During low-volatility periods the feed can go many hours without an update. Any network congestion, oracle node failure, or L1 disruption that prevents a heartbeat update creates a window where stale prices are silently accepted. This is a realistic, non-adversarial scenario that has occurred historically on mainnet.

### Recommendation

Add a configurable `maxStaleness` parameter and validate `updatedAt` in `getAssetPrice()`:

```solidity
uint256 public maxStaleness; // e.g., 3600 seconds

function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();
    require(answeredInRound >= roundId, "Stale round");
    require(block.timestamp - updatedAt <= maxStaleness, "Stale price");
    require(price > 0, "Non-positive price");
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Set `maxStaleness` per feed to slightly exceed each feed's documented heartbeat interval.

### Proof of Concept

1. Chainlink stETH/ETH feed last updated at `T - 20 hours`; its price is 1.05e18 (stETH at a premium).
2. stETH de-pegs; true market price falls to 0.95e18.
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
4. `getRsETHAmountToMint` computes `100e18 * 1.05e18 / rsETHPrice` — minting ~10.5% excess rsETH.
5. Attacker immediately redeems via `LRTWithdrawalManager`, extracting ETH worth more than the 100 stETH deposited at true market value.
6. Existing rsETH holders are diluted; the protocol's ETH backing per rsETH decreases — insolvency. [1](#0-0) [4](#0-3) [5](#0-4) [6](#0-5)
