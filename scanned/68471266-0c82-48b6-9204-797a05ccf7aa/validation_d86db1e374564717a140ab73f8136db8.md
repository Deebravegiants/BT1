### Title
Daily Fee Mint Limit Exhaustion Permanently Blocks `rsETHPrice` Updates Until Next Period - (File: contracts/LRTOracle.sol)

### Summary

`LRTOracle._updateRsETHPrice()` unconditionally calls `_checkAndUpdateDailyFeeMintLimit()` on every invocation. When the daily fee mint budget (`maxFeeMintAmountPerDay`) is exhausted — or is zero (the default uninitialized value) — every call to `updateRSETHPrice()` and `updateRSETHPriceAsManager()` reverts with `DailyFeeMintLimitExceeded`, freezing `rsETHPrice` for up to 24 hours. All downstream consumers (L1 deposit pool, L2 pools, cross-chain rate providers) continue reading the stale price during this window.

### Finding Description

`_updateRsETHPrice()` computes a protocol fee whenever TVL has grown and the protocol is not paused: [1](#0-0) 

It then calls `_checkAndUpdateDailyFeeMintLimit` with the computed fee amount: [2](#0-1) 

`_checkAndUpdateDailyFeeMintLimit` reverts whenever the accumulated fee for the current period would exceed `maxFeeMintAmountPerDay`: [3](#0-2) 

Two distinct trigger conditions exist:

1. **Default zero state.** `maxFeeMintAmountPerDay` is a `uint256` storage variable never set in `initialize()`. [4](#0-3)  Until the manager explicitly calls `setMaxFeeMintAmountPerDay()`, it is `0`. Any non-zero `feeAmount` satisfies `feeAmount > 0`, causing an immediate revert on the very first price update that generates a fee.

2. **Limit exhaustion within a period.** Even after the manager sets a non-zero limit, a single large TVL increase (or several updates within the same 24-hour window) can exhaust `maxFeeMintAmountPerDay`. All subsequent `updateRSETHPrice()` calls revert for the remainder of that period.

Critically, `updateRSETHPriceAsManager()` also routes through `_updateRsETHPrice()`: [5](#0-4) 

There is no bypass path that updates `rsETHPrice` without hitting the fee-mint limit check.

### Impact Explanation

`rsETHPrice` is the denominator used by every deposit and swap calculation in the protocol:

- **L1 deposits** — `LRTDepositPool.getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()`. [6](#0-5) 
- **L2 pools** — `RSETHPoolV3.viewSwapRsETHAmountAndFee()` divides by `getRate()` which reads `rsETHPrice` from the oracle. [7](#0-6) 
- **Cross-chain rate providers** — `RSETHRateProvider` and `RSETHMultiChainRateProvider` propagate `rsETHPrice` to L2 receivers. [8](#0-7) 

A stale (lower-than-actual) `rsETHPrice` causes new depositors to receive more rsETH than their ETH contribution warrants, diluting existing holders' accrued yield. This constitutes **temporary freezing of the exchange rate update mechanism** and **theft of unclaimed yield** from existing rsETH holders for the duration of the stale window (up to 24 hours per occurrence, or indefinitely if `maxFeeMintAmountPerDay` remains `0`).

**Impact: Medium — Temporary freezing of unclaimed yield / stale exchange rate.**

### Likelihood Explanation

- The `maxFeeMintAmountPerDay == 0` default is active from deployment until the manager explicitly calls `setMaxFeeMintAmountPerDay()`. Any protocol deployment where `protocolFeeInBPS > 0` and TVL grows before that call is made will hit this immediately.
- Even after a non-zero limit is set, a single day with significant staking rewards or a large deposit that increases TVL can exhaust the budget, blocking all further updates until midnight of the next period.
- `updateRSETHPrice()` is a public, permissionless function — any external caller (keeper bot, user, MEV searcher) can trigger it, meaning the limit can be consumed by the first caller each day, leaving no budget for subsequent legitimate updates.

**Likelihood: Medium.**

### Recommendation

1. **Guard the fee-mint limit separately from the price update.** Decouple `_checkAndUpdateDailyFeeMintLimit` from the price-write path. If the fee mint limit is exceeded, skip minting the fee (or cap it) but still commit the updated `rsETHPrice` to storage.
2. **Treat `maxFeeMintAmountPerDay == 0` as "unlimited."** Add an early-return guard: `if (maxFeeMintAmountPerDay == 0) return;` at the top of `_checkAndUpdateDailyFeeMintLimit`, consistent with the existing behavior in `remainingDailyFeeMintLimit()`. [9](#0-8) 
3. **Initialize `maxFeeMintAmountPerDay` in `initialize()` or `reinitialize()`** to a safe non-zero value so the contract is never in a state where fee-bearing price updates are impossible.

### Proof of Concept

```
State: maxFeeMintAmountPerDay = 0 (default), protocolFeeInBPS > 0

1. Rewards accrue → totalETHInProtocol > previousTVL
2. Anyone calls updateRSETHPrice()
3. _updateRsETHPrice() computes protocolFeeInETH > 0
4. rsethAmountToMintAsProtocolFee = protocolFeeInETH / newRsETHPrice > 0
5. _checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee) is called
6. Check: 0 + rsethAmountToMintAsProtocolFee > 0  → TRUE
7. revert DailyFeeMintLimitExceeded(...)
8. rsETHPrice is NOT updated; stale price persists

All subsequent calls to updateRSETHPrice() and updateRSETHPriceAsManager()
revert identically. L2 pools and LRTDepositPool continue using the stale
rsETHPrice, minting excess rsETH to new depositors and diluting existing holders.
``` [10](#0-9) [11](#0-10)

    function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
        // Reset the period if it's unset or a day has passed
        if (block.timestamp >= feePeriodStartTime + 1 days) {
            currentPeriodMintedFeeAmount = 0;
            feePeriodStartTime = getCurrentPeriodStartTime();
        }

        // Check if minting would exceed the daily limit
        if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
            revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
        }

        currentPeriodMintedFeeAmount += feeAmount;
    }
```
