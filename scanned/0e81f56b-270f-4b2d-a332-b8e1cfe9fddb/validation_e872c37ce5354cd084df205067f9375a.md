### Title
`CrossChainRateReceiver.getRate()` Returns Stale Rate Without Staleness Check, Enabling Over-Minting of wrsETH on L2 Pools - (File: contracts/cross-chain/CrossChainRateReceiver.sol)

### Summary
`CrossChainRateReceiver.getRate()` returns the stored `rate` without validating `lastUpdated` against any maximum staleness threshold. When this receiver is used as the `rsETHOracle` in L2 pool contracts (e.g., `RSETHPoolV3`), a stale (artificially low) rate causes the pool to mint more `wrsETH` than the depositor is entitled to, enabling yield theft.

### Finding Description
`CrossChainRateReceiver` tracks `lastUpdated` (set in `lzReceive`) but never uses it in `getRate()`:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol
uint256 public lastUpdated;   // line 16 — stored but never validated

function getRate() external view returns (uint256) {
    return rate;              // line 103-104 — no staleness check
}
``` [1](#0-0) [2](#0-1) 

`RSETHRateReceiver` (and `RSETHMultiChainRateReceiver`) inherit this base and are deployed as the `rsETHOracle` for L2 pools. `RSETHPoolV3.getRate()` delegates directly to this oracle:

```solidity
// contracts/pools/RSETHPoolV3.sol
function getRate() public view returns (uint256) {
    return IOracle(rsETHOracle).getRate();   // line 235-237
}
``` [3](#0-2) 

The rate is then used to compute how many `wrsETH` tokens to mint per unit of ETH deposited:

```solidity
// contracts/pools/RSETHPoolV3.sol
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;   // line 304-307
``` [4](#0-3) 

Because `rsETH` is a yield-bearing token, its ETH price increases monotonically over time. If the LayerZero cross-chain update is delayed (congestion, downtime, or simply no caller triggering `updateRate()`), the stored `rate` is lower than the true current rate. A lower denominator in the division above produces a larger `rsETHAmount`, so the depositor receives more `wrsETH` than the ETH they deposited is actually worth.

By contrast, the protocol's own `ChainlinkOracleForRSETHPoolCollateral` explicitly guards against stale data:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
``` [5](#0-4) 

This confirms the protocol is aware of the staleness risk but omitted the equivalent guard in `CrossChainRateReceiver`.

### Impact Explanation
An attacker who deposits ETH while the cross-chain rate is stale receives more `wrsETH` than the current fair value. When they later redeem or bridge that `wrsETH` at the correct (higher) rate, they extract more ETH than they deposited. The excess comes from the pool's liquidity, which represents other users' funds. This constitutes **theft of unclaimed yield** (High) and, at scale, can drain pool liquidity (Critical — direct theft of user funds at rest).

### Likelihood Explanation
LayerZero message delivery is not guaranteed to be instantaneous. Network congestion, bridge downtime, or simply the absence of a keeper calling `updateRate()` can leave the rate stale for hours or days. The `updateRate()` function on `CrossChainRateProvider` is permissionless and requires a fee payment, meaning it may not be called regularly. [6](#0-5) 

The attack requires no special privileges — any address can call `deposit()` on the pool. [7](#0-6) 

### Recommendation
Add a configurable `maxStaleness` parameter to `CrossChainRateReceiver` and revert in `getRate()` if `block.timestamp - lastUpdated > maxStaleness`:

```solidity
uint256 public maxStaleness; // e.g., 24 hours

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Stale rate");
    return rate;
}
```

### Proof of Concept
1. At `t=0`, `CrossChainRateReceiver.rate = 1.05e18` (rsETH/ETH) and `lastUpdated = t0`.
2. LayerZero updates stop. By `t=48h`, the true rsETH rate has grown to `1.06e18` but the receiver still holds `1.05e18`.
3. Attacker calls `RSETHPoolV3.deposit{value: 100 ETH}("")`.
4. Pool computes `rsETHAmount = 100e18 * 1e18 / 1.05e18 ≈ 95.238 wrsETH` (should be `≈ 94.339 wrsETH` at the true rate).
5. Attacker receives `~0.9 wrsETH` excess — redeemable for `~0.954 ETH` at the true rate — at the expense of the pool's liquidity. [8](#0-7) [4](#0-3)
