### Title
Uninitialized Cross-Chain Rate Causes Division-by-Zero, Temporarily Freezing All L2 Deposits - (File: contracts/cross-chain/CrossChainRateReceiver.sol)

### Summary
The `CrossChainRateReceiver.rate` storage variable is initialized to `0` (Solidity default) at deployment. Before the first `lzReceive()` call propagates a valid rate from L1, `getRate()` returns `0`. All L2 pool deposit functions divide by this rate without a zero-guard, causing a division-by-zero revert that temporarily freezes every user deposit on the affected L2 chain.

### Finding Description

`CrossChainRateReceiver` stores the rsETH/ETH exchange rate in a plain `uint256 public rate` that defaults to `0`: [1](#0-0) 

The only way to set a non-zero value is via `lzReceive()`, which is called by the LayerZero endpoint after `MultiChainRateProvider.updateRate()` is executed on L1: [2](#0-1) 

Every L2 pool variant (`RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPool`, `RSETHPoolNoWrapper`) calls `IOracle(rsETHOracle).getRate()` and immediately divides by the returned value with no zero-check: [3](#0-2) [4](#0-3) 

The `deposit()` entry points call `viewSwapRsETHAmountAndFee()` directly (or through the `limitDailyMint` modifier), so every deposit reverts with a division-by-zero panic during the window where `rate == 0`: [5](#0-4) 

Critically, neither `initialize()` nor `setRSETHOracle()` validates that the oracle returns a non-zero rate before accepting it: [6](#0-5) [7](#0-6) 

This is inconsistent with `addSupportedToken()`, which explicitly guards against a zero rate for token oracles: [8](#0-7) 

The same zero-rate guard is absent for the primary `rsETHOracle` path across all pool variants.

### Impact Explanation

During the time window between deployment of a `CrossChainRateReceiver`-backed oracle and receipt of the first cross-chain rate message, every call to `deposit()` (ETH or token) on the affected L2 pool reverts. Users cannot deposit any assets. The pool holds no user funds at this stage, so no funds are lost, but the service is completely unavailable. This maps to **Medium – Temporary freezing of funds**.

### Likelihood Explanation

This window is a normal part of every new L2 pool deployment: the pool is initialized, the oracle contract is deployed, and some non-trivial delay exists before the first `updateRate()` → `lzReceive()` round-trip completes. Any user who attempts a deposit during this window is affected. The window is bounded by operational latency (minutes to hours), not by any on-chain trigger, making it a predictable and repeatable condition on each new chain deployment.

### Recommendation

1. Add a non-zero rate check in `initialize()` and `setRSETHOracle()` for all pool variants, mirroring the guard already present in `addSupportedToken()`:
   ```solidity
   if (IOracle(_rsETHOracle).getRate() == 0) revert UnsupportedOracle();
   ```
2. Add an explicit zero-rate guard inside `viewSwapRsETHAmountAndFee()` before the division, reverting with a descriptive error rather than a bare arithmetic panic:
   ```solidity
   if (rsETHToETHrate == 0) revert UnsupportedOracle();
   ```
3. Consider initializing `CrossChainRateReceiver.rate` to a sentinel non-zero value in a constructor/initializer, or require the deployer to supply an initial rate.

### Proof of Concept

1. Deploy a fresh `CrossChainRateReceiver`-backed oracle on an L2. `rate == 0`.
2. Deploy (or upgrade) an L2 pool pointing to this oracle. `setRSETHOracle()` accepts it without checking the rate.
3. Before `MultiChainRateProvider.updateRate()` is called on L1 (or before the LayerZero message is delivered), call `deposit{value: 1 ether}("ref")` on the pool.
4. Execution reaches `viewSwapRsETHAmountAndFee(1 ether)`:
   ```
   rsETHToETHrate = getRate()  // returns 0
   rsETHAmount = amountAfterFee * 1e18 / 0  // panic: division by zero
   ```
5. Transaction reverts. All deposits are frozen until the first cross-chain rate message arrives. [9](#0-8) [10](#0-9)

        rate = _rate;

        lastUpdated = block.timestamp;

        emit RateUpdated(_rate);
