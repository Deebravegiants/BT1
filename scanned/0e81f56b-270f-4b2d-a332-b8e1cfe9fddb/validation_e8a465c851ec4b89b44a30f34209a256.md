### Title
Paused RSETH Contract Does Not Block `transfer`/`transferFrom` — Pause Bypass in `_transfer` Override - (File: contracts/RSETH.sol)

### Summary
`RSETH` inherits `ERC20Upgradeable` and `PausableUpgradeable` separately but does **not** inherit `ERC20PausableUpgradeable`. The overridden `_transfer` function enforces `_enforceNotBlocked` but omits a `whenNotPaused` guard. As a result, any rsETH holder can call the standard ERC20 `transfer` or `transferFrom` functions to freely move tokens even while the contract is paused, defeating the emergency-freeze mechanism entirely.

### Finding Description
`RSETH._transfer` is overridden to enforce per-address transfer blocks:

```solidity
// contracts/RSETH.sol L287-291
function _transfer(address from, address to, uint256 amount) internal override {
    _enforceNotBlocked(from);
    _enforceNotBlocked(to);
    super._transfer(from, to, amount);
}
```

There is no `whenNotPaused` check here. By contrast, `mint` and `burnFrom` both carry `whenNotPaused`:

```solidity
// L229-240
function mint(...) external onlyRole(...) whenNotPaused checkDailyMintLimit(amount) { ... }

// L245-248
function burnFrom(...) external onlyRole(...) whenNotPaused { ... }
```

Because `RSETH` does not inherit `ERC20PausableUpgradeable` (which enforces `require(!paused())` inside `_beforeTokenTransfer`), the base `ERC20Upgradeable._transfer` path has no pause awareness. The public `transfer` and `transferFrom` entry points both resolve to `RSETH._transfer`, which never consults the paused state.

**Exploit path:**
1. Admin detects an exploit or suspicious activity and calls `pause()` to freeze the contract.
2. A targeted user (or any rsETH holder) calls `transfer(attacker2, balance)` or `transferFrom(victim, attacker2, amount)`.
3. The call succeeds — `_enforceNotBlocked` passes (address not individually blocked), and there is no pause check.
4. Tokens move freely. The admin's subsequent `blockUserTransfers([user])` + `recoverFrozenFunds(user)` finds a zero balance; funds are gone.

### Impact Explanation
The pause mechanism is the protocol's primary emergency-stop control for rsETH. Its failure to cover `transfer`/`transferFrom` means:
- A user under investigation can front-run `blockUserTransfers` by transferring tokens to a fresh address immediately after the contract is paused.
- `recoverFrozenFunds` becomes ineffective because the balance has already moved.
- The protocol cannot achieve a complete token freeze during an incident.

**Impact: Medium — Temporary freezing of funds** (the pause is intended to temporarily freeze all token movements; it fails to do so for the primary ERC-20 transfer paths).

### Likelihood Explanation
The contract is already deployed with a `pause()` function callable by `PAUSER_ROLE`. Any time the contract is paused — a realistic and documented operational scenario — every rsETH holder can exploit this gap with a single standard ERC-20 `transfer` call. No special privileges or complex setup are required.

### Recommendation
Add `whenNotPaused` to the `_transfer` override so that the pause state is enforced on all token movements:

```solidity
function _transfer(address from, address to, uint256 amount) internal override whenNotPaused {
    _enforceNotBlocked(from);
    _enforceNotBlocked(to);
    super._transfer(from, to, amount);
}
```

Alternatively, inherit from `ERC20PausableUpgradeable` instead of plain `ERC20Upgradeable`, which enforces the pause inside `_beforeTokenTransfer` and covers `_transfer`, `_mint`, and `_burn` uniformly.

### Proof of Concept
1. Deploy `RSETH` (or use a fork).
2. Mint rsETH to `alice`.
3. Call `pause()` as `PAUSER_ROLE`.
4. From `alice`, call `rsETH.transfer(bob, aliceBalance)`.
5. Observe: call succeeds, `alice`'s balance is now 0, `bob`'s balance increased — despite the contract being paused.
6. Call `blockUserTransfers([alice])` then `recoverFrozenFunds(alice)` — reverts with `NoActiveTransferBlock` or recovers 0 tokens. [1](#0-0) [2](#0-1) [3](#0-2)
