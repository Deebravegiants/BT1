### Title
Broken or Reverting Extension Permanently Locks LP Funds in `removeLiquidity` With No Emergency Exit - (File: `metric-core/contracts/MetricOmmPool.sol`, `metric-core/contracts/ExtensionCalling.sol`, `metric-core/contracts/libraries/CallExtension.sol`)

---

### Summary

`removeLiquidity` unconditionally invokes `_beforeRemoveLiquidity` and `_afterRemoveLiquidity` extension hooks before and after burning LP shares and transferring tokens. `CallExtension.callExtension` propagates every revert from an extension directly to the caller. Extensions are stored as pool-level immutables and cannot be replaced post-deployment. There is no emergency withdrawal path that bypasses extensions. If any extension registered for the remove-liquidity hooks reverts — whether due to a bug, self-destruct, or the `BaseMetricExtension` base class's default `revert ExtensionNotImplemented()` — all LP funds in that pool are permanently frozen.

`_beforeRemoveLiquidity` and `_afterRemoveLiquidity` both delegate to `_callExtensionsInOrder`, which iterates the packed extension order and calls each registered extension via `CallExtension.callExtension`: [2](#0-1) 

`CallExtension.callExtension` **bubbles every revert from the extension back to the caller** — there is no `try/catch`, no skip-on-failure, no fallback: [3](#0-2) 

Additionally, `callExtension` reverts with `InvalidExtensionResponse` if the returned data is shorter than 32 bytes — which is exactly what happens when calling a self-destructed contract (returns empty data): [4](#0-3) 

**Extensions are immutable.** All seven extension slots are stored as Solidity `immutable` variables set at construction time and cannot be updated: [5](#0-4) 

**`BaseMetricExtension` reverts by default on remove-liquidity hooks.** Any extension inheriting from `BaseMetricExtension` that does not override `beforeRemoveLiquidity` or `afterRemoveLiquidity` will always revert with `ExtensionNotImplemented()`: [6](#0-5) 

**No emergency exit exists.** The pause mechanism intentionally leaves `removeLiquidity` unblocked (only `swap` checks `whenNotPaused`), but there is no corresponding mechanism to bypass a broken extension for withdrawals. The factory has no function to replace an extension on a deployed pool.

---

### Impact Explanation

Any pool deployed with an extension registered for `BEFORE_REMOVE_LIQUIDITY_ORDER` or `AFTER_REMOVE_LIQUIDITY_ORDER` that subsequently reverts will have **all LP principal permanently frozen**. Users cannot call `removeLiquidity` because every call reverts at the extension hook before or after the token transfer. There is no alternative withdrawal path. This is a direct, total loss of user principal for all LPs in the affected pool — a Critical/High impact under the contest gate.

---

### Likelihood Explanation

The trigger does not require a malicious actor. Concrete realistic scenarios:

1. **Accidental misconfiguration**: A developer writes an extension inheriting `BaseMetricExtension`, registers it for `beforeRemoveLiquidity` or `afterRemoveLiquidity`, but forgets to override those functions. The base class reverts with `ExtensionNotImplemented()` on every call, immediately and permanently locking all LP funds.

2. **Extension bug post-deployment**: A production extension (e.g., `OracleValueStopLossSubExtension` or a custom allowlist) contains a bug that causes it to revert under certain market conditions. Because extensions are immutable, there is no recovery path.

3. **Self-destruct**: If an extension contract is self-destructed (possible on chains that still support it), `callExtension` receives empty return data and reverts with `InvalidExtensionResponse`, permanently blocking withdrawals.

The `BaseMetricExtension` default-revert pattern makes scenario 1 a realistic developer error with no on-chain recovery.

---

### Recommendation

1. **Add an emergency withdrawal path** that skips extension hooks. This could be a factory-callable `emergencyRemoveLiquidity` that bypasses `_beforeRemoveLiquidity` / `_afterRemoveLiquidity`, or a pool-level flag set by the factory that disables extension calls for the remove-liquidity path only.

2. **Wrap extension calls in `try/catch`** for the remove-liquidity hooks specifically, so a reverting extension degrades gracefully (emitting an event) rather than permanently locking funds.

3. **Change `BaseMetricExtension` defaults** for `beforeRemoveLiquidity` and `afterRemoveLiquidity` to return the correct selector (pass-through) rather than reverting, so unoverridden hooks do not silently brick withdrawals.

//    -> extension.call(data) returns (true, "") — empty returndata
//    -> result.length < 32 → revert InvalidExtensionResponse()
//    -> same permanent lock outcome.

  function _callExtensionsInOrder(uint256 order, bytes memory data) private {
    if (order == 0) return;

    while (true) {
      uint256 extensionIndex = order & 0x7;
      if (extensionIndex == 0) break;
      address extension = _extensionAddress(extensionIndex);
      if (extension == address(0)) revert PanicEmptyExtension();
      CallExtension.callExtension(extension, data);
      order >>= 3;
    }
  }

  function callExtension(address extension, bytes memory data) internal {
    (bool success, bytes memory result) = extension.call(data);
    if (!success) {
      if (result.length > 0) {
        assembly ("memory-safe") {
          revert(add(result, 32), mload(result))
        }
      }
      revert ExtensionCallFailed();
    }
    if (result.length < 32) {
      revert InvalidExtensionResponse();
    }
    bytes4 returnedSelector;
    assembly ("memory-safe") {
      returnedSelector := mload(add(result, 32))
    }
    bytes4 expectedSelector;
    assembly ("memory-safe") {
      expectedSelector := mload(add(data, 32))
    }
    if (returnedSelector != expectedSelector) {
      revert InvalidExtensionResponse();
    }
  }
