### Title
Uninitialized `rate=0` in `CrossChainRateReceiver` Causes Division-by-Zero, Temporarily Freezing All L2 Pool Deposits - (`contracts/cross-chain/CrossChainRateReceiver.sol`)

### Summary

`CrossChainRateReceiver.rate` is a `uint256` storage variable that defaults to `0` at deployment. The `getRate()` function returns this raw value with no zero-check or staleness guard. Every L2 pool variant (`RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPoolNoWrapper`) calls `IOracle(rsETHOracle).getRate()` and divides by the result to compute the wrsETH amount to mint. When `rate == 0`, this division reverts, freezing all deposits until the first LayerZero rate message arrives.

### Finding Description

`CrossChainRateReceiver` stores the rsETH/ETH exchange rate received via LayerZero:

```solidity
uint256 public rate;   // default = 0

function getRate() external view returns (uint256) {
    return rate;       // no zero-check, no staleness check
}
``` [1](#0-0) [2](#0-1) 

`rate` is only written inside `lzReceive()`, which is called by the LayerZero endpoint after a cross-chain message is relayed: [3](#0-2) 

The concrete receiver, `RSETHRateReceiver`, sets no initial rate in its constructor: [4](#0-3) 

Every L2 pool's `deposit()` path calls `viewSwapRsETHAmountAndFee`, which divides by the oracle rate:

```solidity
uint256 rsETHToETHrate = getRate();          // returns 0 before first LZ message
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;  // division by zero → revert
``` [5](#0-4) [6](#0-5) [7](#0-6) 

The same pattern applies to token deposits: [8](#0-7) 

### Impact Explanation

Every `deposit()` call on any L2 pool whose `rsETHOracle` is a `CrossChainRateReceiver`-derived contract will revert with a division-by-zero panic until the first LayerZero rate message is successfully relayed and `lzReceive()` is executed. This is a **temporary freezing of funds** — users cannot deposit ETH or LSTs into the pool during this window. The window can be extended if LayerZero message delivery is delayed (e.g., due to congestion or insufficient gas on the destination chain).

**Impact**: Medium — Temporary freezing of funds (deposits blocked).

### Likelihood Explanation

This is triggered automatically for every newly deployed L2 pool that uses a `CrossChainRateReceiver` as its oracle, during the period between pool deployment and the first successful `lzReceive()` call. No attacker action is required; any ordinary depositor will encounter the revert. Given the protocol is deployed across 15+ L2 chains, this window occurs on every new deployment or oracle contract redeployment.

### Recommendation

Set a safe non-zero initial rate in the `CrossChainRateReceiver` constructor (or in `RSETHRateReceiver`'s constructor), e.g., `rate = 1 ether` (representing 1:1 rsETH/ETH). Alternatively, add a zero-check guard in `getRate()` that reverts with a descriptive error rather than propagating `0` silently, and add a corresponding zero-check in `viewSwapRsETHAmountAndFee` before dividing.

### Proof of Concept

1. Deploy `RSETHRateReceiver` on an L2 chain. `rate` is `0`.
2. Deploy `RSETHPoolV3` with `rsETHOracle = address(rateReceiver)`.
3. Call `pool.deposit{value: 1 ether}("ref")` before any LayerZero message arrives.
4. `viewSwapRsETHAmountAndFee(1 ether)` calls `getRate()` → returns `0`.
5. `rsETHAmount = amountAfterFee * 1e18 / 0` → EVM division-by-zero panic → revert.
6. All deposits are frozen until `lzReceive()` is called by the LayerZero endpoint. [9](#0-8) [10](#0-9)

        require(msg.sender == layerZeroEndpoint, "Sender should be lz endpoint");

        address srcAddress;
        assembly {
            srcAddress := mload(add(_srcAddress, 20))
        }

        require(_srcChainId == srcChainId, "Src chainId must be correct");
        require(srcAddress == rateProvider, "Src address must be provider");

        uint256 _rate = abi.decode(_payload, (uint256));

        rate = _rate;

        lastUpdated = block.timestamp;

        emit RateUpdated(_rate);
