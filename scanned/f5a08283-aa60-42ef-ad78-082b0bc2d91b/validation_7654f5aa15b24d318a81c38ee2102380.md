### Title
ETH Permanently Frozen When Withdrawal Recipient Contract Lacks `receive()` - (File: `contracts/LRTWithdrawalManager.sol`)

### Summary
The two-phase withdrawal design in `LRTWithdrawalManager` burns rsETH in `unlockQueue` (a separate operator transaction) before ETH is delivered to the user in `completeWithdrawal`. If the withdrawing address is a contract without a `receive()` function, the ETH delivery always reverts, but the rsETH burn is already irreversible. The ETH is permanently locked inside `LRTWithdrawalManager` with no admin escape path.

### Finding Description
The withdrawal lifecycle is split across two transactions:

**Phase 1 – `unlockQueue`** (operator-triggered): [1](#0-0) 
rsETH is burned from the contract and ETH is pulled from `LRTUnstakingVault` into `LRTWithdrawalManager`. This is irreversible.

**Phase 2 – `completeWithdrawal` → `_processWithdrawalCompletion`** (user-triggered): [2](#0-1) 

The final delivery uses `_transferAsset`: [3](#0-2) 

When `to` is a contract without `receive()`, `payable(to).call{value: amount}("")` returns `false`, triggering `revert EthTransferFailed()`. The entire `_processWithdrawalCompletion` transaction reverts, restoring `withdrawalRequests[requestId]` and `unlockedWithdrawalsCount[asset]`. However, the rsETH burned in Phase 1 is **not** restored — it was destroyed in a prior, committed transaction.

Every subsequent call to `completeWithdrawal` or `completeWithdrawalForUser` for this user will revert identically, because both route through `_processWithdrawalCompletion` and send to the same `user` address: [4](#0-3) 

The only potential ETH recovery path, `sweepRemainingAssets`, is blocked while any unlocked withdrawal exists: [5](#0-4) 

Because `unlockedWithdrawalsCount[asset]` is never decremented (the decrement reverts with the rest of the transaction), `hasUnlockedWithdrawals` permanently returns `true` for the affected asset, and `sweepRemainingAssets` permanently reverts. `LRTWithdrawalManager` does not inherit any `Recoverable` base and has no other ETH egress function.

### Impact Explanation
- The user's rsETH is permanently burned with no compensation.
- The corresponding ETH is permanently locked inside `LRTWithdrawalManager`.
- No admin function exists to cancel the stuck withdrawal request, redirect the payout to a different address, or otherwise recover the ETH in the current implementation.
- Impact: **Critical — permanent freezing of user funds**.

### Likelihood Explanation
Any smart contract that holds rsETH and calls `initiateWithdrawal(ETH_TOKEN, …)` without implementing a `receive()` fallback is affected. This includes DeFi aggregators, yield vaults, multisig wallets, and other on-chain integrators — a realistic and growing class of callers for a liquid restaking protocol. The trigger requires no special privilege: the unprivileged depositor initiates the withdrawal, and the operator's routine `unlockQueue` call completes the trap.

### Recommendation
**Short term:** Add a `withdrawalRecipient` mapping that lets users designate a separate payout address, or add an admin-only function to redirect or cancel a stuck withdrawal request and return the ETH to a configurable address.

**Long term:** Consider performing a pre-flight ETH-receivability check (analogous to `_safeMint`'s `_checkOnERC721Received`) at `initiateWithdrawal` time for ETH withdrawals, reverting early if the caller's contract cannot receive ETH. This prevents the two-phase commitment from ever entering an unrecoverable state.

### Proof of Concept
```solidity
// Contract with no receive() function
contract NoReceiveWallet {
    function initiateETHWithdrawal(address withdrawalManager, uint256 rsETHAmount) external {
        // rsETH approved beforehand
        ILRTWithdrawalManager(withdrawalManager)
            .initiateWithdrawal(ETH_TOKEN, rsETHAmount, "");
        // rsETH is now held by LRTWithdrawalManager
    }

    function tryComplete(address withdrawalManager) external {
        // Operator has already called unlockQueue — rsETH is burned, ETH is in LRTWithdrawalManager
        // This always reverts with EthTransferFailed because NoReceiveWallet has no receive()
        ILRTWithdrawalManager(withdrawalManager)
            .completeWithdrawal(ETH_TOKEN, "");
    }
    // No receive() — ETH delivery always fails
    // rsETH is gone; ETH is permanently locked in LRTWithdrawalManager
}
```

1. `NoReceiveWallet` calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")` — rsETH transferred to `LRTWithdrawalManager`. [6](#0-5) 
2. Operator calls `unlockQueue(ETH_TOKEN, …)` — rsETH burned, ETH redeemed into `LRTWithdrawalManager`. [1](#0-0) 
3. `NoReceiveWallet` calls `completeWithdrawal(ETH_TOKEN, "")` — reverts with `EthTransferFailed`. [7](#0-6) 
4. rsETH is permanently burned; ETH is permanently frozen in `LRTWithdrawalManager`; `sweepRemainingAssets` is permanently blocked. [8](#0-7)

```text
    function _processWithdrawalCompletion(address asset, address user, string calldata referralId) internal {
        if (userAssociatedNonces[asset][user].empty()) {
            revert NoWithdrawalRequests(user, asset);
        }

        // Retrieve and remove the oldest withdrawal request for the user.
        uint256 usersFirstWithdrawalRequestNonce = userAssociatedNonces[asset][user].popFront();
        // Ensure the request is already unlocked.
        if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();

        bytes32 requestId = getRequestId(asset, usersFirstWithdrawalRequestNonce);
        WithdrawalRequest memory request = withdrawalRequests[requestId];

        delete withdrawalRequests[requestId];

        // Check that the withdrawal delay has passed since the request's initiation.
        if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();

        unlockedWithdrawalsCount[asset]--;

        // If Aave integration is enabled and asset is ETH, withdraw from Aave if needed
        if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
            uint256 contractBalance = address(this).balance;
            if (contractBalance < request.expectedAssetAmount) {
                uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
                _withdrawFromAave(amountNeeded);

                // Verify we have sufficient balance after withdrawal
                uint256 balanceAfter = address(this).balance;
                if (balanceAfter < request.expectedAssetAmount) {
                    revert InsufficientLiquidityForWithdrawal();
                }
            }
        }

        _transferAsset(asset, user, request.expectedAssetAmount);

        emit ReferralIdEmitted(referralId);
        emit AssetWithdrawalFinalized(user, asset, request.rsETHUnstaked, request.expectedAssetAmount);
    }
```
