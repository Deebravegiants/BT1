### Title
ETH Deposit Cap Completely Bypassed Due to Missing `amount` in Limit Check - (`contracts/LRTDepositPool.sol`)

---

### Summary

`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric check: for ERC-20 assets it correctly adds the incoming `amount` to `totalAssetDeposits` before comparing against the cap, but for ETH it omits `amount` entirely. Any user can therefore deposit an arbitrarily large amount of ETH in a single call as long as the current total does not already exceed the cap.

---

### Finding Description

`depositETH` calls `_beforeDeposit`, which calls `_checkIfDepositAmountExceedesCurrentLimit`:

```solidity
// contracts/LRTDepositPool.sol  L676-L682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct for ERC-20
}
```

For ETH the function returns `true` (i.e., "limit exceeded") only when `totalAssetDeposits` **already** exceeds the cap. The incoming `amount` is never added. Consequently, if `totalAssetDeposits ≤ cap`, the check always returns `false` regardless of how large `amount` is, and the deposit proceeds.

`getTotalAssetDeposits` for ETH sums: ETH in the deposit pool, ETH in NDCs, ETH staked/unstaking in EigenLayer, ETH in the converter, and ETH in the unstaking vault — all legitimate protocol-held ETH. None of these values include the incoming deposit, so the check is structurally incomplete for ETH. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

**Critical — deposit cap bypass enabling unlimited ETH minting of rsETH.**

The deposit cap (`depositLimitByAsset[ETH]`) is the primary risk-limiting mechanism for ETH. With the cap bypassed, a single depositor can send an arbitrarily large ETH value in one transaction, receiving a proportional rsETH mint. This:

- Inflates rsETH supply beyond the intended cap, diluting existing holders.
- Exposes the protocol to unbounded EigenLayer restaking risk that the cap was designed to prevent.
- Mirrors the exact impact class of M-6: a core security invariant (deposit cap) is rendered ineffective. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

**High.** The entry point `depositETH` is public and permissionless. No special role, front-running, or multi-step setup is required. Any user who sends ETH while `totalAssetDeposits ≤ cap` can deposit any amount. This condition is the normal operating state of the protocol (the cap is only breached after the bug is exploited). [2](#0-1) 

---

### Recommendation

Add `amount` to the ETH branch, matching the ERC-20 branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [1](#0-0) 

---

### Proof of Concept

Assume `depositLimitByAsset[ETH] = 1000 ether` and `totalAssetDeposits = 999 ether`.

1. Attacker calls `depositETH{value: 1_000_000 ether}(0, "")`.
2. `_checkIfDepositAmountExceedesCurrentLimit(ETH, 1_000_000 ether)` evaluates `999 ether > 1000 ether` → `false` (not exceeded).
3. `_mintRsETH` mints rsETH proportional to 1,000,000 ETH.
4. `totalAssetDeposits` is now ~1,000,999 ETH — 1000× the intended cap — in a single transaction.

The attacker needs only enough ETH to send; no looping or multi-step exploit is required. [6](#0-5) [1](#0-0)
