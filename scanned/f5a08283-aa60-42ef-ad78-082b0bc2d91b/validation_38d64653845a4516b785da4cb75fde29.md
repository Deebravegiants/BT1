### Title
`_checkAndUpdateDailyFeeMintLimit()` Incorrectly Reverts When `maxFeeMintAmountPerDay` Is Zero, Permanently Blocking Protocol Fee Minting - (File: contracts/LRTOracle.sol)

### Summary
When `maxFeeMintAmountPerDay` is zero (the default storage value, or when explicitly reset to zero by the manager), `_checkAndUpdateDailyFeeMintLimit()` unconditionally reverts on any non-zero fee amount. This causes `updateRSETHPrice()` to revert whenever TVL has grown and protocol fees are owed, permanently freezing unclaimed yield for the protocol treasury until an admin intervenes.

### Finding Description
In `LRTOracle.sol`, `_checkAndUpdateDailyFeeMintLimit()` enforces the daily fee-minting cap:

```solidity
// contracts/LRTOracle.sol lines 204-207
if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
    revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
}
```

When `maxFeeMintAmountPerDay == 0` (the Solidity default for an unset `uint256`), the condition simplifies to:

```
currentPeriodMintedFeeAmount + feeAmount > 0
```

Because `currentPeriodMintedFeeAmount` starts at zero and `feeAmount` is the computed protocol fee (non-zero whenever TVL has increased), this expression is always `true`, causing an unconditional revert.

`_updateRsETHPrice()` calls this function on the fee-minting path:

```solidity
// contracts/LRTOracle.sol lines 299-308
if (protocolFeeInETH > 0) {
    uint256 rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice);
    _checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee); // <-- reverts here
    ...
}
```

The `reinitialize()` function (the upgrade entry-point) only sets `feePeriodStartTime`; it never initialises `maxFeeMintAmountPerDay`:

```solidity
// contracts/LRTOracle.sol lines 72-79
function reinitialize(uint256 _feePeriodStartTime) external reinitializer(2) onlyLRTManager {
    ...
    feePeriodStartTime = _feePeriodStartTime;
    emit FeePeriodStartTimeSet(_feePeriodStartTime);
}
```

`setMaxFeeMintAmountPerDay()` also accepts zero without restriction:

```solidity
// contracts/LRTOracle.sol lines 132-135
function setMaxFeeMintAmountPerDay(uint256 _maxFeeMintAmountPerDay) external onlyLRTManager {
    maxFeeMintAmountPerDay = _maxFeeMintAmountPerDay;
    emit MaxFeeMintAmountPerDayUpdated(_maxFeeMintAmountPerDay);
}
```

The companion view function `remainingDailyFeeMintLimit()` also returns the wrong value (0 instead of "unlimited") when `maxFeeMintAmountPerDay == 0`, mirroring the original report's pattern of a function returning false values for the zero-parameter case:

```solidity
// contracts/LRTOracle.sol lines 170-171
function remainingDailyFeeMintLimit() external view returns (uint256) {
    if (maxFeeMintAmountPerDay == 0) return 0; // wrong: should signal unlimited
    ...
}
```

### Impact Explanation
Every call to the public `updateRSETHPrice()` (or `updateRSETHPriceAsManager()`) reverts as soon as TVL grows above the previous snapshot while `maxFeeMintAmountPerDay == 0`. Consequences:

1. **Protocol fee yield is permanently frozen** — `rsethAmountToMintAsProtocolFee` is never minted to the treasury; the protocol loses all accrued fee revenue for the affected period.
2. **rsETH price becomes stale** — because `rsETHPrice` is only written at the end of a successful `_updateRsETHPrice()` call, the stored price drifts below the true NAV. Depositors receive more rsETH than they are entitled to (at the stale lower price), while withdrawers receive less underlying than they are owed.

This satisfies **Medium — Permanent freezing of unclaimed yield**.

### Likelihood Explanation
`maxFeeMintAmountPerDay` is a plain `uint256` storage slot that is zero by default. The upgrade path (`reinitialize`) does not set it. Any deployment or upgrade that omits a subsequent `setMaxFeeMintAmountPerDay` call leaves the contract in the broken state. Additionally, a manager can reset it to zero at any time with no on-chain guard, re-triggering the condition. The entry path (`updateRSETHPrice`) is permissionless — any external caller can observe the revert.

### Recommendation
Guard `_checkAndUpdateDailyFeeMintLimit` so that `maxFeeMintAmountPerDay == 0` is treated as "no cap" (unlimited), consistent with the typical convention for uncapped limits:

```solidity
function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
    if (maxFeeMintAmountPerDay == 0) return; // 0 = no limit; skip check

    if (block.timestamp >= feePeriodStartTime + 1 days) {
        currentPeriodMintedFeeAmount = 0;
        feePeriodStartTime = getCurrentPeriodStartTime();
    }

    if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
        revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
    }

    currentPeriodMintedFeeAmount += feeAmount;
}
```

Apply the same fix to `remainingDailyFeeMintLimit()` (return `type(uint256).max` when `maxFeeMintAmountPerDay == 0`).

### Proof of Concept
1. Protocol is upgraded; `reinitialize()` is called, setting `feePeriodStartTime` but leaving `maxFeeMintAmountPerDay == 0`.
2. Users deposit ETH/LSTs; TVL grows above the previous snapshot price × supply.
3. Anyone calls `updateRSETHPrice()`.
4. Inside `_updateRsETHPrice()`: `totalETHInProtocol > previousTVL`, so `protocolFeeInETH > 0` and `rsethAmountToMintAsProtocolFee > 0`.
5. `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` executes the check: `0 + rsethAmountToMintAsProtocolFee > 0` → `true`.
6. Transaction reverts with `DailyFeeMintLimitExceeded(rsethAmountToMintAsProtocolFee, 0)`.
7. `rsETHPrice` is never updated; protocol fees are never minted; the treasury receives nothing for the period.
8. The condition persists on every subsequent call until a manager explicitly calls `setMaxFeeMintAmountPerDay` with a non-zero value. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
