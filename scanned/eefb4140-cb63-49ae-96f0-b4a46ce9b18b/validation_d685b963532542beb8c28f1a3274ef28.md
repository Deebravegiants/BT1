### Title
No Withdrawal Cancellation Mechanism Permanently Freezes User rsETH When Protocol Assets Are Insufficient — (`contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTWithdrawalManager.initiateWithdrawal` transfers a user's rsETH into the contract and increments `assetsCommitted[asset]`, but there is no `cancelWithdrawal` function. If the protocol's unstaking vault never accumulates enough assets to satisfy the queued withdrawal (e.g., due to EigenLayer slashing or a prolonged asset shortfall), the user's rsETH is permanently locked in the contract with no recovery path.

---

### Finding Description

When a user calls `initiateWithdrawal`, rsETH is pulled from the user and held in `LRTWithdrawalManager`, and `assetsCommitted[asset]` is incremented by the oracle-priced expected asset amount: [1](#0-0) 

The withdrawal can only be resolved by the operator calling `unlockQueue`, which internally calls `_unlockWithdrawalRequests`. That function breaks out of its loop and leaves the request locked if the unstaking vault's available balance is less than the payout amount: [2](#0-1) 

`assetsCommitted[asset]` is only decremented inside `_unlockWithdrawalRequests` when a request is successfully unlocked: [3](#0-2) 

`getAvailableAssetAmount` gates every new `initiateWithdrawal` call: [4](#0-3) 

There is no `cancelWithdrawal`, no admin function to forcibly return rsETH to a user, and no timeout-based escape hatch. The contract's own NatSpec acknowledges the edge case but provides no remedy: [5](#0-4) 

---

### Impact Explanation

**Temporary / Permanent freezing of funds.**

A user who calls `initiateWithdrawal` surrenders their rsETH to the contract. If the protocol's `LRTUnstakingVault` never holds enough of the requested asset to satisfy the queued request — whether because of EigenLayer slashing, a prolonged withdrawal queue from EigenLayer, or a sustained drop in the asset's oracle price — the rsETH remains locked indefinitely. Because `assetsCommitted[asset]` is never decremented for the stuck request, subsequent users also find `getAvailableAssetAmount` returning zero or a reduced value, compounding the freeze across the withdrawal queue.

---

### Likelihood Explanation

EigenLayer slashing is a documented protocol risk. A partial slash of a NodeDelegator's restaked position reduces `getTotalAssetDeposits(asset)` while leaving `assetsCommitted[asset]` unchanged, immediately making `getAvailableAssetAmount` return zero. Any user who had already called `initiateWithdrawal` before the slash event is now in a state where their withdrawal can never be unlocked unless the protocol is recapitalised — and there is no mechanism for that recapitalisation to unblock the stuck request. The scenario is reachable by any ordinary withdrawer without any privileged action.

---

### Recommendation

1. **Add a `cancelWithdrawal` function** that allows a user to reclaim their rsETH after a configurable timeout (e.g., `withdrawalDelayBlocks` has elapsed and the request is still locked), decrementing `assetsCommitted[asset]` accordingly.
2. **Add an admin/operator escape hatch** to forcibly cancel a specific locked request and return rsETH to the user, mirroring the `depositToBatcher` fix in the referenced report.
3. Consider emitting a dedicated event when a request cannot be unlocked due to insufficient assets, so off-chain monitoring can alert operators promptly.

---

### Proof of Concept

1. User calls `initiateWithdrawal(stETH, 10e18)`. rsETH is transferred to `LRTWithdrawalManager`; `assetsCommitted[stETH] += X`.
2. A NodeDelegator is slashed on EigenLayer; `getTotalAssetDeposits(stETH)` drops below `assetsCommitted[stETH]`.
3. Operator calls `unlockQueue(stETH, ...)`. Inside `_unlockWithdrawalRequests`, `availableAssetAmount < payoutAmount` → `break`. The request remains at `nextLockedNonce`, rsETH is not burned, assets are not redeemed.
4. `getAvailableAssetAmount(stETH)` now returns `0`. No new withdrawal can be initiated.
5. The user has no function to call to recover their rsETH. The funds are frozen. [6](#0-5) [7](#0-6)

    /// asset. Will be finalised by calling `completeWithdrawal` after the manager unlocked the request and the delay
    /// has past. There is an edge case were the user withdraws last underlying asset and that asset gets slashed.
    function initiateWithdrawal(
        address asset,
        uint256 rsETHUnstaked,
        string calldata referralId
    )
        external
        override
        nonReentrant
        whenNotPaused
        onlySupportedAsset(asset)
        onlySupportedStrategy(asset)
    {
        if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
            revert InvalidAmountToWithdraw();
        }

        IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);

        uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);

        if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();

        // preventing over-withdrawal.
        assetsCommitted[asset] += expectedAssetAmount;

        _addUserWithdrawalRequest(asset, rsETHUnstaked, expectedAssetAmount);

        emit ReferralIdEmitted(referralId);
    }

        ILRTDepositPool lrtDepositPool = ILRTDepositPool(lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL));
        uint256 totalAssets = lrtDepositPool.getTotalAssetDeposits(asset);
        availableAssetAmount = totalAssets > assetsCommitted[asset] ? totalAssets - assetsCommitted[asset] : 0;
    }
