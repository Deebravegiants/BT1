### Title
Stale Cross-Chain Rate Used in `AGETHPoolV3.deposit()` Causes Over-Minting of agETH — (`contracts/agETH/AGETHRateReceiver.sol`, `contracts/cross-chain/CrossChainRateReceiver.sol`)

---

### Summary

`CrossChainRateReceiver.getRate()` returns the last stored `rate` with no check against `lastUpdated`. `AGETHPoolV3.deposit()` calls this function unconditionally. If the LayerZero relayer is delayed, the stale (lower) rate causes the pool to mint more agETH per ETH than the current L1 backing justifies.

---

### Finding Description

`CrossChainRateReceiver.getRate()` simply returns `rate`: [1](#0-0) 

`lastUpdated` is recorded on every `lzReceive` call: [2](#0-1) 

But it is never consulted in `getRate()`. There is no `require(block.timestamp - lastUpdated <= maxStaleness)` or equivalent guard anywhere in the receiver.

`AGETHPoolV3.getRate()` delegates directly to this oracle: [3](#0-2) 

`deposit()` calls `viewSwapAgETHAmountAndFee()`, which uses that rate to compute the mint amount: [4](#0-3) 

The mint formula is `agETHAmount = amountAfterFee * 1e18 / agETHToETHrate`. Because agETH appreciates monotonically, a stale (lower) rate produces a larger `agETHAmount` than the current L1 rate would justify.

---

### Impact Explanation

Every depositor who calls `deposit()` while the receiver holds a stale rate receives more agETH than the current agETH/ETH exchange rate backs. The protocol has collected the correct ETH but has minted excess agETH, creating a gap between total agETH supply and its ETH collateral. This matches **Low — contract fails to deliver promised returns, but doesn't lose value** (the ETH is still held; the over-minted agETH dilutes existing holders rather than draining funds outright).

---

### Likelihood Explanation

LayerZero relayer delays are a documented operational reality (network congestion, gas spikes, relayer downtime). The `updateRate()` call on the provider side is also permissionless and must be triggered externally; if no one calls it, the receiver rate can be arbitrarily old. No special privilege is required — any depositor benefits passively by depositing during a stale window.

---

### Recommendation

Add a configurable `maxStaleness` parameter to `CrossChainRateReceiver` and revert in `getRate()` if `block.timestamp - lastUpdated > maxStaleness`:

```solidity
uint256 public maxStaleness; // e.g. 24 hours

function getRate() external view returns (uint256) {
    require(lastUpdated != 0 && block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

This causes `deposit()` to revert rather than mint at a stale rate, protecting the protocol's collateral invariant.

---

### Proof of Concept

```solidity
// Fork test (local/private testnet)
// 1. Deploy AGETHRateReceiver with a mock LZ endpoint.
// 2. Call lzReceive with rate = 1.01e18 (agETH slightly above peg).
// 3. Warp block.timestamp forward by 7 days (no further lzReceive).
// 4. On L1, agETH rate has appreciated to 1.05e18 (currentL1Rate).
// 5. Call AGETHPoolV3.deposit{value: 1 ether}().
// 6. Assert:
//    agETHAmount == 1e18 * 1e18 / 1.01e18  ≈ 0.9901e18   (stale, over-minted)
//    currentCorrect == 1e18 * 1e18 / 1.05e18 ≈ 0.9524e18
//    agETHAmount > currentCorrect  // invariant broken
```

The test passes on unmodified code, confirming the over-mint without any privileged action.
