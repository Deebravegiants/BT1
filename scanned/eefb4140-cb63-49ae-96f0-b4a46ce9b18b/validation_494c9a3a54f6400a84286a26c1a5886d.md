### Title
Fee-on-Transfer Token Support Breaks rsETH Minting Accounting in LRTDepositPool - (File: contracts/LRTDepositPool.sol)

---

### Summary
`LRTDepositPool.depositAsset()` calculates the rsETH amount to mint from the caller-supplied `depositAmount` (nominal), then performs `safeTransferFrom` for that same nominal amount. If any supported LST token charges a fee on transfer, the contract receives fewer tokens than `depositAmount`, but mints rsETH as if the full `depositAmount` arrived. This inflates rsETH supply relative to actual backing assets, causing protocol insolvency.

---

### Finding Description
In `LRTDepositPool.depositAsset()`:

```solidity
// contracts/LRTDepositPool.sol lines 110-115
uint256 rsethAmountToMint = _beforeDeposit(asset, depositAmount, minRSETHAmountExpected);

IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
_mintRsETH(rsethAmountToMint);
```

`_beforeDeposit` calls `getRsETHAmountToMint(asset, depositAmount)`:

```solidity
// contracts/LRTDepositPool.sol lines 519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`rsethAmountToMint` is derived from the nominal `depositAmount`, not from the actual tokens received. If the token deducts a fee during `safeTransferFrom`, the contract receives `depositAmount - fee`, but mints rsETH for the full `depositAmount`. The same pattern exists in `RsETHTokenWrapper._deposit()` and `AGETHTokenWrapper._deposit()`, where `_mint(_to, _amount)` is called after `safeTransferFrom(msg.sender, address(this), _amount)` with no balance-delta check. ---

### Impact Explanation
Every deposit with a fee-on-transfer token mints more rsETH than the actual assets backing it. Over time, `getTotalAssetDeposits()` (which reads real on-chain balances) will be lower than the rsETH `totalSupply` implies, making the protocol insolvent. Existing rsETH holders are diluted; the protocol cannot honor all redemptions at the stated exchange rate. This is **protocol insolvency** (Critical). [5](#0-4) 

---

### Likelihood Explanation
The protocol already supports multiple LSTs (stETH, ETHx, etc.) and the admin can add new supported assets via governance. USDT on Ethereum has fee-on-transfer logic compiled in (fee currently set to 0). If USDT or any future asset with transfer fees is added, or if an existing token enables its fee switch, every deposit call by any unprivileged user immediately triggers the accounting mismatch. The entry path requires no special privileges — `depositAsset()` is public. [6](#0-5) 

---

### Recommendation
Use the balance-delta pattern: record the contract's token balance before and after `safeTransferFrom`, and use the actual delta (not the nominal `depositAmount`) to compute `rsethAmountToMint`:

```solidity
uint256 balanceBefore = IERC20(asset).balanceOf(address(this));
IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
uint256 actualReceived = IERC20(asset).balanceOf(address(this)) - balanceBefore;

uint256 rsethAmountToMint = getRsETHAmountToMint(asset, actualReceived);
if (rsethAmountToMint < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet();
_mintRsETH(rsethAmountToMint);
```

Apply the same fix to `RsETHTokenWrapper._deposit()` and `AGETHTokenWrapper._deposit()`. [3](#0-2) [4](#0-3) 

---

### Proof of Concept
1. Assume a supported LST token `T` enables a 1% transfer fee.
2. Attacker calls `LRTDepositPool.depositAsset(T, 1000e18, 0, "")`.
3. `rsethAmountToMint` is computed from `1000e18` → e.g., `990e18` rsETH (at current rate).
4. `safeTransferFrom` transfers `1000e18` from attacker; contract receives `990e18` (1% fee taken).
5. `_mintRsETH(990e18)` mints rsETH for the full nominal amount.
6. Protocol now holds `990e18` of `T` but has issued rsETH as if it held `1000e18`.
7. Repeated deposits widen the gap; `getTotalAssetDeposits(T)` falls below what rsETH supply implies, making the protocol insolvent. [7](#0-6)
