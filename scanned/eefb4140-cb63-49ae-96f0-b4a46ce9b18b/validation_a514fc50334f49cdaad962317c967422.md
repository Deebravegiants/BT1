### Title
Unsafe `uint256`-to-`int256` Cast in `RSETHPriceFeed` Corrupts Price Answer - (File: contracts/oracles/RSETHPriceFeed.sol)

### Summary
`RSETHPriceFeed.latestRoundData()` and `getRoundData()` both perform an unchecked cast of the `uint256` return value of `RS_ETH_ORACLE.rsETHPrice()` to `int256`. If the rsETH price value ever exceeds `type(int256).max` (`2^255 - 1`), the cast silently wraps to a large negative number, causing the price feed to return a corrupted negative answer to every downstream consumer.

### Finding Description
In `contracts/oracles/RSETHPriceFeed.sol`, both `latestRoundData()` and `getRoundData()` compute the rsETH/USD answer as:

```solidity
answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
```

`RS_ETH_ORACLE.rsETHPrice()` returns a `uint256`. Casting a `uint256` value that exceeds `type(int256).max` (i.e., `2^255 - 1 ≈ 5.79 × 10^76`) to `int256` silently wraps to a large negative number in Solidity, with no revert or error. There is no bounds check before the cast. This is the same vulnerability class as the reference report: an unsigned integer is cast to a signed integer of the same bit-width without validating the upper bound.

### Impact Explanation
`RSETHPriceFeed` is a Chainlink-compatible `AggregatorV3Interface` price feed for rsETH/USD. Any protocol that consumes this feed (Curve pools, lending markets, collateral managers, etc.) would receive a negative `int256 answer`. Downstream consumers that do not independently validate the sign of the returned answer would compute incorrect prices, potentially enabling:
- Collateral mis-valuation in lending protocols (under-collateralized borrows)
- Incorrect swap pricing in AMMs
- Corrupted oracle reads for any protocol integrating rsETH as collateral

The impact is oracle/rate corruption affecting all downstream consumers of this price feed.

### Likelihood Explanation
The rsETH price is stored as a `uint256` denominated in ETH with 18-decimal precision (currently ~1.05 × 10^18). For the cast to overflow, the value would need to exceed `2^255 - 1 ≈ 5.79 × 10^76`, which is astronomically larger than any realistic ETH-denominated price. The likelihood of natural overflow is therefore negligible under normal operation. However, the vulnerability class is identical to the reference report (which was also rated Medium despite low likelihood), and the absence of a bounds check leaves the code structurally unsafe. A bug in `LRTOracle._updateRsETHPrice()` that causes `rsETHPrice` to be set to an extreme value, or a future upgrade that changes the price denomination, could trigger the overflow.

### Recommendation
Validate that `RS_ETH_ORACLE.rsETHPrice()` fits within `type(int256).max` before casting, or use OpenZeppelin's `SafeCast.toInt256()`:

```solidity
import { SafeCast } from "@openzeppelin/contracts/utils/math/SafeCast.sol";

// In latestRoundData() and getRoundData():
answer = SafeCast.toInt256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
```

This will revert with a clear error rather than silently producing a corrupted negative price.

### Proof of Concept

The same Chisel demonstration from the reference report applies to `uint256` → `int256`:

```
➜ int256(uint256(type(int256).max))
Type: int
└ Decimal: 57896044618658097711785492504343953926634992332820282019728792003956564819967

➜ int256(uint256(type(int256).max) + 1)
Type: int
└ Decimal: -57896044618658097711785492504343953926634992332820282019728792003956564819968
```

In `RSETHPriceFeed.sol`, if `RS_ETH_ORACLE.rsETHPrice()` returns `type(int256).max + 1` (or any value above `2^255 - 1`), the expression `int256(RS_ETH_ORACLE.rsETHPrice())` silently evaluates to a large negative number. The subsequent multiplication `* answer / 1e18` then produces a negative `answer`, which is returned to all callers of `latestRoundData()` and `getRoundData()` without any revert.
