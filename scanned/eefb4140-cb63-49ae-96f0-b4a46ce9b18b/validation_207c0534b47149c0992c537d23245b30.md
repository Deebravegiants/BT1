### Title
ETH Deposit Limit Bypass Due to Missing Amount in Limit Check - (`contracts/LRTDepositPool.sol`)

### Summary

`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` applies an inconsistent comparison for ETH versus ERC20 assets. For ERC20 tokens the incoming `amount` is correctly added to `totalAssetDeposits` before comparing against the cap, but for ETH the `amount` parameter is silently ignored. This allows any depositor to push the ETH TVL arbitrarily above the configured `depositLimitByAsset` cap in a single transaction.

### Finding Description

`_checkIfDepositAmountExceedesCurrentLimit` receives the deposit `amount` as a parameter but discards it for the ETH branch:

```solidity
// contracts/LRTDepositPool.sol  lines 676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct for ERC20
}
``` [1](#0-0) 

The function is called from `_beforeDeposit` with `depositAmount = msg.value`:

```solidity
// contracts/LRTDepositPool.sol  lines 661-663
if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
    revert MaximumDepositLimitReached();
}
``` [2](#0-1) 

Because the ETH branch only tests `totalAssetDeposits > limit` (the pre-deposit state), the check passes whenever the current total has not yet crossed the cap — regardless of how large `msg.value` is. A single depositor can therefore push the ETH TVL from `limit - 1 wei` to `limit + N ETH` in one call.

The parallel view function `getAssetCurrentLimit` already treats the boundary correctly (returns `0` when `totalAssetDeposits == limit`), confirming the intended invariant is that deposits must not reach or exceed the cap:

```solidity
// contracts/LRTDepositPool.sol  lines 402-409
function getAssetCurrentLimit(address asset) public view override returns (uint256) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)) {
        return 0;
    }
    return lrtConfig.depositLimitByAsset(asset) - totalAssetDeposits;
}
``` [3](#0-2) 

### Impact Explanation

The `depositLimitByAsset` cap is the protocol's primary mechanism for controlling how much ETH is restaked into EigenLayer strategies. Bypassing it allows:

1. The ETH TVL to exceed the EigenLayer strategy cap, causing subsequent `depositAssetIntoStrategy` calls to revert or silently under-deposit, leaving ETH stranded in the `NodeDelegator`.
2. The rsETH price to be diluted if the excess ETH does not earn restaking rewards (e.g., it sits idle in the NDC).
3. Violation of the protocol's risk parameters, which are set to limit slashing exposure.

Impact: **Low — Contract fails to deliver promised returns** (deposit cap invariant is broken; excess ETH may not be productively restaked, diluting rsETH holders).

### Likelihood Explanation

The entry path is fully permissionless: any address can call `depositETH` with an arbitrarily large `msg.value`. No special role, front-running, or oracle manipulation is required. The condition is triggered whenever `totalAssetDeposits < depositLimitByAsset(ETH)`, which is the normal operating state of the protocol.

### Recommendation

Add the incoming deposit amount to the pre-deposit total for ETH, matching the ERC20 branch:

```diff
- if (asset == LRTConstants.ETH_TOKEN) {
-     return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
- }
+ if (asset == LRTConstants.ETH_TOKEN) {
+     return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
+ }
```

### Proof of Concept

1. Admin sets `depositLimitByAsset(ETH) = 100 ether`.
2. Current `getTotalAssetDeposits(ETH) = 99.9 ether` (just below the cap).
3. Attacker calls `depositETH{value: 50 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH, 50 ether)` evaluates `99.9 ether > 100 ether` → `false` → no revert.
5. 50 ETH is accepted; total ETH in protocol becomes 149.9 ETH — 49.9 ETH above the intended cap.
6. `getAssetCurrentLimit(ETH)` now returns `0`, but the damage is already done; the cap has been exceeded by a factor of ~1.5×.
