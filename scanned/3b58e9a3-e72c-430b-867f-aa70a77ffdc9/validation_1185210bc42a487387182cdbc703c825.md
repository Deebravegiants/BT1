### Title
Burner Role Bypass of ERC20 Allowance Check Breaks CCIP Bridge-Back Burns — (`contracts/ccip/WrappedRSETH.sol`)

---

### Summary

`WrappedRSETH.burnFrom` imposes two independent requirements: the `onlyBurner` role guard **and** the standard ERC20 allowance check inherited from `ERC20Burnable`. The Chainlink CCIP `BurnMintTokenPool` calls `IBurnMintERC20(token).burn(address(this), amount)` — the two-argument variant — which routes through `burnFrom` and unconditionally invokes `_spendAllowance(pool, pool, amount)`. Because the pool holds no self-approval, this call reverts on every bridge-back attempt, permanently locking yield-bearing WrappedRSETH on L2.

---

### Finding Description

`WrappedRSETH` inherits from `ERC20Burnable` and overrides `burnFrom` with an `onlyBurner` guard:

```solidity
// contracts/ccip/WrappedRSETH.sol:129-131
function burnFrom(address account, uint256 amount)
    public override(IBurnMintERC20, ERC20Burnable) onlyBurner {
    super.burnFrom(account, amount);   // ← delegates to ERC20Burnable
}
``` [1](#0-0) 

`super.burnFrom` is `ERC20Burnable.burnFrom`, which unconditionally calls `_spendAllowance` before burning:

```solidity
// lib/openzeppelin-contracts/contracts/token/ERC20/extensions/ERC20Burnable.sol:35-38
function burnFrom(address account, uint256 amount) public virtual {
    _spendAllowance(account, _msgSender(), amount);
    _burn(account, amount);
}
``` [2](#0-1) 

The two-argument `burn(address, uint256)` is a thin alias that also routes through `burnFrom`:

```solidity
// contracts/ccip/WrappedRSETH.sol:122-124
function burn(address account, uint256 amount) public virtual override {
    burnFrom(account, amount);
}
``` [3](#0-2) 

The Chainlink CCIP `BurnMintTokenPool.lockOrBurn` calls `IBurnMintERC20(i_token).burn(address(this), amount)` — the two-argument form defined in the interface:

```solidity
// contracts/ccip/IBurnMintERC20.sol:22
function burn(address account, uint256 amount) external;
``` [4](#0-3) 

When the pool calls `burn(address(this), amount)`:

1. `onlyBurner` passes — pool is a registered burner.
2. `_spendAllowance(pool, pool, amount)` is called — checks `allowance[pool][pool]`.
3. `allowance[pool][pool] == 0` → **ERC20: insufficient allowance** → revert.

The pool never self-approves because CCIP pools are not designed to do so, and the `IBurnMintERC20` contract standard does not require it. The reference implementation (`BurnMintERC677`, cited in the contract's own NatDoc at line 18) avoids this entirely by calling `_burn(account, amount)` directly in `burn(address, uint256)`, bypassing `_spendAllowance`. [5](#0-4) 

---

### Impact Explanation

Every bridge-back (L2 → L1) operation fails at the burn step. Users who hold yield-bearing WrappedRSETH on L2 cannot redeem it for rsETH on L1. The accrued yield embedded in the rsETH exchange rate is permanently inaccessible. This matches **Medium — Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

The condition is triggered on every bridge-back call through the standard CCIP pool interface. No special attacker action is required; the bug fires for any ordinary user attempting to bridge back. Likelihood is **high** given that bridge-back is a core, documented user flow.

---

### Recommendation

Replace `super.burnFrom(account, amount)` with a direct `_burn(account, amount)` call inside `WrappedRSETH.burnFrom`, consistent with the Chainlink reference `BurnMintERC677`:

```solidity
function burnFrom(address account, uint256 amount)
    public override(IBurnMintERC20, ERC20Burnable) onlyBurner {
    _burn(account, amount);   // no allowance check; role guard is sufficient
}
```

The `onlyBurner` modifier already enforces access control. The ERC20 allowance check is redundant and incompatible with the CCIP burn-mint pool pattern.

---

### Proof of Concept

```solidity
// Local unit test (no mainnet required)
function test_burnFrom_burnerNoAllowance_reverts() public {
    address pool = address(0xBEEF);
    address user = address(0xCAFE);
    uint256 amount = 1e18;

    // Setup: grant burner role to pool, mint tokens to user
    vm.prank(owner);
    wrappedRSETH.grantBurnRole(pool);
    vm.prank(owner);
    wrappedRSETH.grantMintRole(owner);
    vm.prank(owner);
    wrappedRSETH.mint(user, amount);

    // Pool has burner role but user has NOT approved pool
    assertEq(wrappedRSETH.allowance(user, pool), 0);

    // Simulate CCIP pool calling burn(address(this), amount) after receiving tokens
    // In practice pool holds the tokens; here we test burnFrom(user, amount) directly
    vm.prank(pool);
    vm.expectRevert("ERC20: insufficient allowance");
    wrappedRSETH.burnFrom(user, amount);

    // Also fails via the two-arg burn alias
    vm.prank(pool);
    vm.expectRevert("ERC20: insufficient allowance");
    wrappedRSETH.burn(user, amount);
}
```

The test confirms that a registered burner cannot burn tokens without a prior ERC20 approval, breaking the invariant that the burner role alone is sufficient — as required by the CCIP burn-mint token standard.
