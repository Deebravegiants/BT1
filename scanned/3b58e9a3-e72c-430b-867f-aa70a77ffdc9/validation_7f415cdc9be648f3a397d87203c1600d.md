### Title
Integer Division in `notifyRewardAmount` Permanently Locks Reward Tokens - (`contracts/KERNEL/KernelDepositPool.sol`)

### Summary
`KernelDepositPool.notifyRewardAmount` computes `rewardRate = receivedAmount / duration` using integer division, permanently locking `receivedAmount % duration` reward tokens in the contract with no recovery path. This is the direct analog of the Allora invariant-breaking bug: in Allora, the pending-reward module account holds more than the sum of all `(rewardPerShare × stake − rewardDebt)`; here, the contract's reward token balance exceeds the total rewards that will ever be distributed to stakers.

### Finding Description

In `notifyRewardAmount`, the reward rate is computed as:

```solidity
// contracts/KERNEL/KernelDepositPool.sol:580
rewardRate = receivedAmount / duration;
```

Solidity integer division truncates, so `rewardRate * duration < receivedAmount` whenever `receivedAmount % duration != 0`. The difference (`receivedAmount % duration`) remains in the contract's ERC-20 balance but is never included in any future `rewardPerToken()` accumulation. There is no `sweep`, `recover`, or `emergencyWithdraw` function for the reward token anywhere in the contract.

When `notifyRewardAmount` is called while a reward period is still active, the issue compounds:

```solidity
// contracts/KERNEL/KernelDepositPool.sol:582-583
uint256 remaining = (finishAt - block.timestamp) * rewardRate;
rewardRate = (receivedAmount + remaining) / duration;
```

`remaining` is computed from the current (already-truncated) `rewardRate`, so accumulated dust from all prior calls is silently excluded from the new `rewardRate` calculation and permanently locked.

This breaks the invariant:

> `rewardRate × (finishAt − block.timestamp) = actual reward token balance held by the contract`

The contract holds more reward tokens than the sum of all `earned(user)` values will ever total, mirroring the Allora invariant break where the pending-reward account holds more than `Σ(rewardPerShare × stake − rewardDebt)`.

### Impact Explanation

Reward tokens equal to `receivedAmount % duration` per `notifyRewardAmount` call are permanently frozen in the contract. Stakers can never claim these tokens. For low-decimal reward tokens (e.g., 6 decimals) with a 7-day duration (`604800` seconds), the locked amount can reach up to `604799` base units per call. Over repeated reward periods, the locked amount accumulates monotonically with no recovery mechanism.

**Impact class**: Permanent freezing of unclaimed yield (Medium).

### Likelihood Explanation

This occurs on every invocation of `notifyRewardAmount`, which is a routine, expected admin operation required to start each reward period. The likelihood is certain whenever the function is called with any `receivedAmount` that is not an exact multiple of `duration`.

### Recommendation

Include the accumulated dust in the next reward period by computing `leftover` from the actual remaining balance rather than from `rewardRate`:

```solidity
function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
    // ... existing checks ...
    uint256 receivedAmount = balanceAfter - balanceBefore;

    uint256 leftover = 0;
    if (block.timestamp < finishAt) {
        leftover = (finishAt - block.timestamp) * rewardRate;
    }
    rewardRate = (receivedAmount + leftover) / duration;
    // ...
}
```

Alternatively, add a `recoverERC20` admin function to allow recovery of tokens that exceed `rewardRate * (finishAt - block.timestamp)`.

### Proof of Concept

1. `duration = 604800` (7 days), `rewardsToken` has 6 decimals.
2. Admin calls `notifyRewardAmount(1_000_000)` (1 token).
3. `rewardRate = 1_000_000 / 604800 = 1` (integer division).
4. Contract will distribute `1 × 604800 = 604800` base units over the period.
5. `1_000_000 − 604800 = 395_200` base units (≈ 0.395 tokens) are permanently locked.
6. Admin calls `notifyRewardAmount` again mid-period with another 1 token:
   - `remaining = (finishAt - block.timestamp) * 1` — does **not** include the 395_200 dust.
   - New `rewardRate = (1_000_000 + remaining) / 604800` — dust from step 5 is excluded again.
7. No function in `KernelDepositPool` allows recovery of the locked tokens. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
        if (_amount == 0) revert AmountZero();

        // Prevent starting a reward period when no tokens are staked to avoid unallocated rewards
        if (totalKernelStaked == 0) revert NoStakedTokens();

        // Transfer reward tokens into the contract
        uint256 balanceBefore = rewardsToken.balanceOf(address(this));
        rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
        uint256 balanceAfter = rewardsToken.balanceOf(address(this));
        // Calculate the actual amount of tokens received in case of a transfer fee (tax)
        uint256 receivedAmount = balanceAfter - balanceBefore;

        if (block.timestamp >= finishAt) {
            rewardRate = receivedAmount / duration;
        } else {
            uint256 remaining = (finishAt - block.timestamp) * rewardRate;
            rewardRate = (receivedAmount + remaining) / duration;
        }

        if (rewardRate == 0) revert RewardRateZero();

        finishAt = block.timestamp + duration;
        updatedAt = block.timestamp;

        emit NotifyRewardAmount(receivedAmount, finishAt);
