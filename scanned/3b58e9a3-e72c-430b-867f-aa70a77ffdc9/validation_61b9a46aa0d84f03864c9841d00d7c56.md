### Title
Stale Chainlink Price Accepted Without Staleness Validation, Enabling Incorrect rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `price`, performing no staleness checks. A stale (inflated) price flows directly into rsETH mint calculations, allowing a depositor to receive more rsETH than the current fair value of their deposit, diluting existing rsETH holders.

### Finding Description
`ChainlinkPriceOracle.getAssetPrice()` fetches the Chainlink price as follows:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

The return values `roundId`, `updatedAt`, and `answeredInRound` are all silently discarded. There is no check that:
- `answeredInRound >= roundId` (stale round detection)
- `updatedAt != 0` (incomplete round detection)
- `price > 0` (negative/zero price guard)

This is in direct contrast to `ChainlinkOracleForRSETHPoolCollateral.sol`, which correctly validates all three conditions before using the price:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

The stale price from `ChainlinkPriceOracle.getAssetPrice()` propagates through `LRTOracle.getAssetPrice()`: [3](#0-2) 

...into `LRTDepositPool.getRsETHAmountToMint()`, which uses it to compute how many rsETH tokens to mint:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [4](#0-3) 

The same stale price also feeds `LRTOracle._getTotalEthInProtocol()`, which is used to compute the global `rsETHPrice` during `updateRSETHPrice()`: [5](#0-4) 

### Impact Explanation
**High — Theft of unclaimed yield / value from existing rsETH holders.**

If a Chainlink LST/ETH feed (e.g., stETH/ETH) goes stale while carrying an inflated price (e.g., the feed stops updating after a price spike), a depositor calling `depositAsset()` receives more rsETH than the current fair value of their deposit. Because rsETH is a share token backed by the total protocol TVL, over-minting rsETH for one depositor directly dilutes the redemption value for all existing rsETH holders — their shares now represent a smaller fraction of the same underlying assets. The magnitude scales with the staleness duration and the size of the deposit.

### Likelihood Explanation
**Medium.** Chainlink feeds can go stale during network congestion, sequencer downtime (on L2s), or when the price deviation threshold is not breached for an extended period. The protocol supports multiple LST assets (stETH, rETH, ETHx, swETH), each with its own feed and heartbeat. The absence of any on-chain staleness guard means any single feed going stale is sufficient to trigger the issue. The attacker entry path requires no special role — any unprivileged depositor can call `depositAsset()` or `depositETH()` during the stale window.

### Recommendation
Apply the same staleness validation already present in `ChainlinkOracleForRSETHPoolCollateral.sol` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

require(price > 0, "Chainlink price <= 0");
require(updatedAt != 0, "Incomplete round");
require(answeredInRound >= roundId, "Stale price");
```

Additionally, consider adding a `block.timestamp - updatedAt <= MAX_STALENESS` check with a per-feed configurable heartbeat threshold.

### Proof of Concept

1. Chainlink stETH/ETH feed stops updating (e.g., due to network congestion). The last reported price is `1.05e18` (stETH at a 5% premium to ETH), but the true current price has dropped to `1.00e18`.
2. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
3. `getRsETHAmountToMint` calls `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns stale `1.05e18`.
4. rsETH minted = `(1000e18 * 1.05e18) / rsETHPrice` — 5% more rsETH than the deposit is worth at current prices.
5. Attacker holds inflated rsETH shares. When the feed updates and `updateRSETHPrice()` is called, the true TVL is lower than the inflated mint assumed, so all existing holders' rsETH is worth proportionally less.
6. Attacker redeems via `LRTWithdrawalManager`, extracting value from existing holders. [6](#0-5) [7](#0-6)
