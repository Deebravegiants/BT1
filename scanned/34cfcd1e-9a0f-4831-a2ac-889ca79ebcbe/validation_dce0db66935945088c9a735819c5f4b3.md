### Title
Token Deposit Fee Defaults to Zero in RSETHPool, Allowing Fee-Free Token Deposits - (File: contracts/pools/RSETHPool.sol)

---

### Summary

In `RSETHPool.sol` (the Arbitrum pool), the per-token fee rate `tokenFeeBps[token]` defaults to 0 for any newly added token. Unlike the ETH deposit fee (`feeBps`), which is set during initialization, the token fee rate is never initialized when a token is added via `addSupportedToken`. Any user can deposit supported tokens and receive rsETH without paying any fee, permanently depriving the protocol of fee revenue unless the admin explicitly calls `setTokenFeeBps` after each token addition.

---

### Finding Description

`RSETHPool` maintains a dedicated `tokenFeeBps` mapping for per-token fee rates, separate from the ETH fee `feeBps`: [1](#0-0) 

The `viewSwapRsETHAmountAndFee(amount, token)` function reads this mapping to compute the fee: [2](#0-1) 

Because Solidity mappings default to `0`, `tokenFeeBps[token]` is `0` for any token that has not had its fee explicitly set. The `addSupportedToken` function adds a token to the pool but never initializes `tokenFeeBps[token]`: [3](#0-2) 

As a result, the `deposit(token, amount, referralId)` function calls `viewSwapRsETHAmountAndFee` and accumulates the returned fee into `feeEarnedInToken[token]`, but since the fee is `0`, nothing is collected: [4](#0-3) 

This is structurally analogous to the NibblVault M-06 bug: a specific execution path (token deposit after `addSupportedToken`) silently skips fee collection. The ETH deposit path does not share this flaw because `feeBps` is set at initialization: [5](#0-4) 

All other pool variants (`RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolNoWrapper`, `RSETHPoolV3WithNativeChainBridge`) use the single shared `feeBps` for both ETH and token deposits, so they are not affected. The vulnerability is isolated to `RSETHPool.sol`. [6](#0-5) 

---

### Impact Explanation

Any user can deposit supported tokens (e.g., wstETH) into the Arbitrum `RSETHPool` and receive rsETH without paying any fee. The protocol collects zero fee revenue from all token deposits until the admin separately calls `setTokenFeeBps`. This constitutes theft of unclaimed yield: fee revenue that the protocol is entitled to collect is permanently lost for the entire window between token addition and explicit fee configuration.

**Impact: High — Theft of unclaimed yield.**

---

### Likelihood Explanation

The condition is triggered automatically every time a new token is added via `addSupportedToken`. There is no atomic mechanism to set the fee at token-addition time. Any depositor can observe the on-chain `AddSupportedToken` event and immediately begin depositing fee-free. The admin must remember to issue a separate `setTokenFeeBps` call; there is no enforcement or default fallback to `feeBps`. [7](#0-6) 

---

### Recommendation

Initialize `tokenFeeBps[token]` to `feeBps` (or an explicit fee parameter) inside `addSupportedToken`, ensuring newly added tokens inherit the protocol's default fee rate atomically:

```solidity
function addSupportedToken(address token, address oracle, address bridge) external onlyRole(TIMELOCK_ROLE) {
    // ... existing checks ...
    supportedTokenList.push(token);
    supportedTokenOracle[token] = oracle;
    tokenBridge[token] = bridge;
+   tokenFeeBps[token] = feeBps; // initialize to default fee
    emit AddSupportedToken(token, oracle, bridge);
}
```

Alternatively, modify `viewSwapRsETHAmountAndFee(amount, token)` to fall back to `feeBps` when `tokenFeeBps[token]` is `0`.

---

### Proof of Concept

1. Admin calls `addSupportedToken(wstETH, oracle, bridge)` — `tokenFeeBps[wstETH]` remains `0`.
2. User calls `deposit(wstETH, 1_000e18, "ref")`.
3. `viewSwapRsETHAmountAndFee(1_000e18, wstETH)` computes:
   - `feeBpsForToken = tokenFeeBps[wstETH] = 0`
   - `fee = 1_000e18 * 0 / 10_000 = 0`
   - `amountAfterFee = 1_000e18`
4. `feeEarnedInToken[wstETH] += 0` — no fee collected.
5. User receives the full rsETH amount with zero fee deducted.
6. Protocol loses all fee revenue from wstETH deposits until admin calls `setTokenFeeBps(wstETH, X)`. [8](#0-7)

```text
    function viewSwapRsETHAmountAndFee(
        uint256 amount,
        address token
    )
        public
        view
        onlySupportedToken(token)
        returns (uint256 rsETHAmount, uint256 fee)
    {
        uint256 feeBpsForToken = tokenFeeBps[token];
        fee = amount * feeBpsForToken / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // rate of token in ETH
        uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
    }
```
