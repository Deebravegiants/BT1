### Title
Token Decimal Mismatch in `viewSwapRsETHAmountAndFee` Causes Depositors of Non-18-Decimal Tokens to Receive Near-Zero wrsETH - (File: contracts/pools/RSETHPoolV3.sol)

### Summary
`viewSwapRsETHAmountAndFee(uint256 amount, address token)` in all pool variants implicitly assumes the deposited token has 18 decimals. When a token with fewer decimals (e.g., USDC/USDT at 6) is deposited, the minted `wrsETH` amount is `10^(18 - tokenDecimals)` times smaller than it should be. For USDC, users receive `1e12` times fewer `wrsETH` than the fair value of their deposit, permanently locking their tokens in the pool.

### Finding Description

`ChainlinkOracleForRSETHPoolCollateral.getRate()` normalizes the Chainlink answer to 18 decimals, returning the price of **1 human-readable token** (not 1 token unit) in ETH:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol:34
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
``` [1](#0-0) 

So `tokenToETHRate = 3e14` means "1 human-readable USDC = 0.0003 ETH". But `amountAfterFee` passed into the swap formula is in **token units** (e.g., `1000e6` for 1000 USDC). The pool's token-path formula is:

```solidity
// contracts/pools/RSETHPoolV3.sol:334
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [2](#0-1) 

Compare with the ETH path, which correctly multiplies by `1e18` to bridge the unit gap:

```solidity
// contracts/pools/RSETHPoolV3.sol:307
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [3](#0-2) 

For the ETH path, `amountAfterFee` is already in wei (18 dec), so the `1e18` factor is the correct normalization. For the token path, `amountAfterFee` is in token units (6 dec for USDC), but no equivalent normalization is applied. The result is that `rsETHAmount` is computed in token-native units rather than 18-decimal wrsETH units.

The same bug is present in every pool variant that accepts ERC-20 token deposits: [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) 

The `deposit(address token, uint256 amount, string referralId)` function is the public entry point that calls `viewSwapRsETHAmountAndFee(amount, token)` and then mints the returned (severely underscaled) amount to the user: [9](#0-8) 

### Impact Explanation

**Critical — permanent freezing of user funds / direct theft of deposited tokens.**

Concrete example with USDC (6 decimals), 1000 USDC deposit, USDC/ETH rate = 0.0003 ETH, rsETH/ETH rate = 1.05:

| | Correct | Actual |
|---|---|---|
| `amountAfterFee` | 1000e6 USDC units | 1000e6 USDC units |
| `tokenToETHRate` | 3e14 (per human-readable USDC) | 3e14 |
| `rsETHAmount` | `1000e6 * 3e14 * 1e12 / 1.05e18 ≈ 2.857e17 wei` | `1000e6 * 3e14 / 1.05e18 ≈ 285714 wei` |
| wrsETH received | ~0.2857 wrsETH | ~0.000000000000285714 wrsETH |

The user's 1000 USDC (≈$1000) is transferred into the pool contract and is inaccessible to them. The `BRIDGER_ROLE` can later sweep it via `moveAssetsForBridging`. The user receives `1e12` times fewer wrsETH than the fair value of their deposit — effectively losing their entire principal.

### Likelihood Explanation

**High.** The `deposit(address token, uint256 amount, string referralId)` function is permissionless and callable by any user. Any supported token with fewer than 18 decimals (USDC, USDT, WBTC, etc.) triggers the bug on every call. No special conditions, timing, or attacker setup is required — a normal depositor using a non-18-decimal token suffers the loss immediately.

### Recommendation

Normalize `amountAfterFee` to 18 decimals before applying the rate, mirroring the ETH path's `* 1e18` factor:

```solidity
uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 amountAfterFeeNormalized = amountAfterFee * 10 ** (18 - tokenDecimals);
rsETHAmount = amountAfterFeeNormalized * tokenToETHRate / rsETHToETHrate;
```

Apply the same fix to `viewSwapAssetToPremintedRsETH` (inverse direction) and to all pool variants (`RSETHPoolV3WithNativeChainBridge`, `RSETHPoolV3ExternalBridge`, `RSETHPool`, `RSETHPoolNoWrapper`, `AGETHPoolV3`).

### Proof of Concept

1. Deploy `RSETHPoolV3` with USDC as a supported token (6 decimals), backed by a `ChainlinkOracleForRSETHPoolCollateral` wrapping the USDC/ETH Chainlink feed.
2. Call `deposit(USDC, 1000e6, "ref")` as an unprivileged user after approving 1000 USDC.
3. Observe: `viewSwapRsETHAmountAndFee(1000e6, USDC)` returns `rsETHAmount ≈ 285714` (wei), not `≈ 2.857e17` (wei).
4. `wrsETH.mint(msg.sender, 285714)` is called — the user receives `~2.857e-13` wrsETH.
5. The 1000 USDC remains in the pool, accessible only to `BRIDGER_ROLE` via `moveAssetsForBridging`.
