### Title
`removeSupportedAsset` Does Not Clear Stale `assetPriceOracle` in `LRTOracle`, Enabling Incorrect rsETH Pricing on Asset Re-addition - (File: contracts/LRTConfig.sol)

### Summary
`LRTConfig.removeSupportedAsset` clears `isSupportedAsset`, `assetStrategy`, and `depositLimitByAsset` for a removed asset but does not clear the corresponding `assetPriceOracle[asset]` entry in `LRTOracle`. If the asset is later re-added via `addNewSupportedAsset`, the stale oracle is silently reactivated and used for all rsETH price calculations without any validation, causing depositors and withdrawers to receive incorrect token amounts.

### Finding Description
`removeSupportedAsset` in `LRTConfig.sol` performs the following cleanup: [1](#0-0) 

It deletes `isSupportedAsset[asset]`, `assetStrategy[asset]`, zeroes `depositLimitByAsset[asset]`, and removes the asset from `supportedAssetList`. However, `assetPriceOracle[asset]` stored in `LRTOracle` is a completely separate mapping that is never touched: [2](#0-1) 

When the asset is later re-added via `_addNewSupportedAsset`, that function only sets `isSupportedAsset`, pushes to `supportedAssetList`, and sets `depositLimitByAsset` â€” it never touches the oracle: [3](#0-2) 

From that point, `_getTotalEthInProtocol` iterates over `supportedAssets` and calls `getAssetPrice(asset)` for the re-added asset: [4](#0-3) 

`getAssetPrice` unconditionally delegates to whatever address is stored in `assetPriceOracle[asset]`: [5](#0-4) 

The stale oracle address â€” which may be deprecated, decommissioned, or returning a price that no longer reflects the asset's true value â€” is used without any re-validation. `updatePriceOracleFor` does enforce a non-zero address check, but only when the asset is currently supported; it does not enforce that the oracle is refreshed on re-addition: [6](#0-5) 

### Impact Explanation
Every user-facing pricing path reads through `assetPriceOracle`. `getRsETHAmountToMint` (used in `depositAsset`) and `getExpectedAssetAmount` (used in `initiateWithdrawal`) both call `lrtOracle.getAssetPrice(asset)`: [7](#0-6) [8](#0-7) 

If the stale oracle overvalues the asset, depositors mint more rsETH than the asset is worth, diluting all existing rsETH holders (theft of yield). If it undervalues, depositors receive fewer rsETH tokens than owed. Either direction breaks the protocol's promised exchange rate for every user interacting with the re-added asset.

### Likelihood Explanation
Low. The trigger requires two sequential admin actions â€” `removeSupportedAsset` followed by `addNewSupportedAsset` â€” without an intervening `updatePriceOracleFor` call. This is an operationally plausible oversight (e.g., during an asset migration or oracle upgrade cycle) but is not directly attacker-controlled. Once the stale oracle is live, any unprivileged depositor or withdrawer is immediately affected.

### Recommendation
In `removeSupportedAsset`, call `LRTOracle.updatePriceOracleFor(asset, address(0))` to explicitly zero out the stale oracle entry. Alternatively, add a guard in `_addNewSupportedAsset` (or in `_getTotalEthInProtocol`) that reverts if `assetPriceOracle[asset]` is still set to a non-zero address, forcing the admin to explicitly configure a fresh oracle before the asset becomes active again.

### Proof of Concept
1. Asset `X` is supported with oracle `O1` set via `updatePriceOracleFor(X, O1)`.
2. Admin calls `removeSupportedAsset(X, idx)` â€” `isSupportedAsset[X]` is deleted, but `assetPriceOracle[X] == O1` remains in `LRTOracle`.
3. Oracle `O1` is deprecated or begins returning a stale/incorrect price.
4. Admin calls `addNewSupportedAsset(X, limit)` â€” `isSupportedAsset[X]` is restored; no oracle update is enforced.
5. A user calls `depositAsset(X, amount, minRSETH, ref)`.
6. `getRsETHAmountToMint(X, amount)` â†’ `lrtOracle.getAssetPrice(X)` â†’ `IPriceFetcher(O1).getAssetPrice(X)` returns the stale/incorrect price.
7. User receives an rsETH amount computed from the wrong exchange rate, either over-minting (diluting all holders) or under-minting (shortchanging the depositor).
