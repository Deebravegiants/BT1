### Title
Incorrect ETH Deposit Limit Check Allows Deposits to Exceed Protocol Cap - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool::_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric conditional for ETH versus ERC20 tokens. The ETH branch omits the incoming deposit amount from the limit comparison, allowing any depositor to push total ETH deposits above the configured cap when the cap is exactly reached.

### Finding Description
The function at `contracts/LRTDepositPool.sol` lines 676–682 reads:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← no `+ amount`
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

For ERC20 assets the prospective deposit amount is included in the comparison (`totalAssetDeposits + amount > limit`), correctly blocking any deposit that would breach the cap. For ETH the new amount is absent: the check only asks whether the cap is *already* exceeded, not whether the incoming deposit *would* exceed it. When `totalAssetDeposits == depositLimit` the ETH branch returns `false` (not exceeded), the deposit is accepted, and `totalAssetDeposits` ends up above the limit by exactly `msg.value`. [1](#0-0) 

The caller path is fully unprivileged: `depositETH` → `_beforeDeposit` → `_checkIfDepositAmountExceedesCurrentLimit`. [2](#0-1) [3](#0-2) 

### Impact Explanation
The deposit limit is a protocol-level risk-management control. Bypassing it allows the total ETH TVL to exceed the intended ceiling, violating the protocol's stated invariant. No funds are directly stolen, but the protocol fails to deliver the promised deposit cap guarantee.

**Impact: Low** — Contract fails to deliver promised returns (deposit limit not enforced for ETH), but no direct value loss occurs.

### Likelihood Explanation
The condition is triggered whenever `totalAssetDeposits` for ETH equals exactly `depositLimitByAsset(ETH_TOKEN)`. Any unprivileged user can call `depositETH` at that moment. Because the deposit pool is a high-traffic entry point and the limit is a fixed value, this boundary condition is reachable in normal operation.

**Likelihood: Medium.**

### Recommendation
Apply the same `+ amount` inclusion used for ERC20 tokens to the ETH branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = X`.
2. Cumulative ETH deposits reach exactly `X` (e.g., via many prior `depositETH` calls).
3. Attacker calls `depositETH{value: Y}(...)` for any `Y > 0`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `X > X` → `false` → deposit is not blocked.
5. `getTotalAssetDeposits(ETH_TOKEN)` now returns `X + Y`, exceeding the configured cap.
6. The ERC20 path would have evaluated `X + Y > X` → `true` → revert, demonstrating the asymmetry. [1](#0-0)
