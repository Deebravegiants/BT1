### Title
Daily Mint Limit State Not Migrated When `startTimestamp` Is Reset During Upgrade - (`contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`, `contracts/pools/RSETHPoolV3WithNativeChainBridge.sol`)

---

### Summary

The `reinitialize` functions that set `startTimestamp` in the L2 pool contracts do not reset `lastMintDay` or `dailyMintAmount`. Because `getCurrentDay()` is computed relative to `startTimestamp`, changing `startTimestamp` in a future upgrade makes the stored `lastMintDay` incompatible with the new epoch's day numbering. If `dailyMintAmount` was at or near `dailyMintLimit` when the upgrade occurred, all user deposits are blocked until `getCurrentDay()` (relative to the new `startTimestamp`) exceeds the stale `lastMintDay` — a delay that can span many days.

---

### Finding Description

`getCurrentDay()` returns a day index relative to `startTimestamp`:

```solidity
function getCurrentDay() public view returns (uint256) {
    return (block.timestamp - startTimestamp) / 1 days;
}
```

The `limitDailyMint` modifier resets `dailyMintAmount` only when `currentDay > lastMintDay`:

```solidity
uint256 currentDay = getCurrentDay();
if (currentDay > lastMintDay) {
    lastMintDay = currentDay;
    dailyMintAmount = 0;
}
if (dailyMintAmount + rsETHAmount > dailyMintLimit) {
    revert DailyMintLimitExceeded();
}
```

The `reinitialize` functions that introduce or update the daily mint limit set a new `startTimestamp` but do **not** reset `lastMintDay` or `dailyMintAmount`:

```solidity
function reinitialize(uint256 _dailyMintLimit, uint256 _startTimestamp)
    external reinitializer(2) onlyRole(DEFAULT_ADMIN_ROLE)
{
    // ...
    dailyMintLimit = _dailyMintLimit;
    startTimestamp = _startTimestamp;
    // lastMintDay and dailyMintAmount are NOT reset
}
```

After the upgrade, `getCurrentDay()` restarts from 0 (or a small number relative to the new `startTimestamp`), while `lastMintDay` retains its large value from the old epoch. The condition `currentDay > lastMintDay` evaluates to `false`, so `dailyMintAmount` is never reset. The stale `dailyMintAmount` from the old epoch persists and is compared against `dailyMintLimit`, blocking all deposits.

This pattern is present identically in three pool contracts:
- `RSETHPoolV3.sol` — `reinitialize(2)` at lines 179–198
- `RSETHPoolV3ExternalBridge.sol` — `reinitialize(4)` at lines 276–295
- `RSETHPoolV3WithNativeChainBridge.sol` — `reinitialize(2)` at lines 218–237

---

### Impact Explanation

Any depositor calling `deposit()` (ETH or token) after a reinitialize that changes `startTimestamp` will have their transaction revert with `DailyMintLimitExceeded` if `dailyMintAmount` was at the limit in the old epoch. The freeze persists for up to `lastMintDay` days (relative to the new `startTimestamp`) — potentially many days — until `getCurrentDay()` finally exceeds `lastMintDay` and the reset triggers. This is a **temporary freezing of user deposits** (Medium impact).

---

### Likelihood Explanation

`RSETHPoolV3ExternalBridge.sol` has already been upgraded through 6 reinitializer versions, demonstrating an active upgrade cadence. Any future reinitializer that changes `startTimestamp` (e.g., to fix a bug, adjust the epoch anchor, or introduce a new minting schedule) will trigger this bug if `dailyMintAmount` was non-zero at the time of the upgrade. The bug is guaranteed to manifest under those conditions, which are realistic given the protocol's upgrade history.

---

### Recommendation

Reset `lastMintDay` and `dailyMintAmount` whenever `startTimestamp` is changed in a reinitializer:

```solidity
function reinitialize(uint256 _dailyMintLimit, uint256 _startTimestamp)
    external reinitializer(N) onlyRole(DEFAULT_ADMIN_ROLE)
{
    if (_dailyMintLimit == 0) revert InvalidDailyMintLimit();
    if (block.timestamp > _startTimestamp) revert InvalidStartTimestamp();

    dailyMintLimit = _dailyMintLimit;
    startTimestamp = _startTimestamp;
    lastMintDay = 0;       // reset epoch-relative day counter
    dailyMintAmount = 0;   // reset accumulated mint amount
}
```

---

### Proof of Concept

Consider `RSETHPoolV3ExternalBridge.sol` deployed on Arbitrum:

1. `reinitialize(4)` is called, setting `startTimestamp = T0` and `dailyMintLimit = 100 ETH`.
2. The contract runs for 200 days. On day 199, the full 100 ETH daily limit is consumed: `lastMintDay = 199`, `dailyMintAmount = 100 ETH`.
3. A bug is found in the pool contract requiring an urgent upgrade. A new `reinitialize(7)` is deployed that sets `startTimestamp = block.timestamp` (= T0 + 200 days) and a new `dailyMintLimit`.
4. After the upgrade, `getCurrentDay() = (block.timestamp - T1) / 1 days = 0`. `lastMintDay = 199`. `0 > 199` is `false` — `dailyMintAmount` is not reset.
5. Every depositor calling `deposit()` hits `dailyMintAmount (100 ETH) + rsETHAmount > dailyMintLimit` and reverts with `DailyMintLimitExceeded`.
6. Deposits remain blocked for 199 days (until `getCurrentDay()` exceeds 199), during which users cannot obtain rsETH/wrsETH from the L2 pool. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
