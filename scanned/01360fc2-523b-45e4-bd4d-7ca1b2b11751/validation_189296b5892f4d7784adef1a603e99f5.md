### Title
`LRTWithdrawalManager.instantWithdrawal()` Lacks Minimum Output Amount, Exposing Users to Unfavorable Oracle Rate Changes - (File: contracts/LRTWithdrawalManager.sol)

### Summary

`LRTWithdrawalManager.instantWithdrawal()` burns the caller's rsETH atomically and computes the asset payout using live oracle prices with no minimum output guard. If the oracle rate moves unfavorably between transaction submission and execution, the user's rsETH is permanently destroyed while they receive less underlying asset than they expected.

### Finding Description

`instantWithdrawal` accepts only `asset`, `rsETHUnstaked`, and `referralId`. It has no `minAssetAmountOut` parameter. [1](#0-0) 

The payout is computed by `getExpectedAssetAmount`, which divides live oracle prices:

```
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
``` [2](#0-1) 

The rsETH burn is executed **before** the asset transfer and before any availability check on the vault: [3](#0-2) 

There is no check of the form `if (assetAmountUnlocked < minAssetAmountOut) revert`. Once line 229 executes, the rsETH is gone regardless of how small `assetAmountUnlocked` turns out to be.

This contrasts directly with `LRTDepositPool.depositETH` and `LRTDepositPool.depositAsset`, which both accept a `minRSETHAmountExpected` parameter and enforce it via `_beforeDeposit`, demonstrating that the protocol is aware of the pattern and applied it on the deposit side but omitted it on the instant-withdrawal side. [4](#0-3) 

The same gap exists across the entire RSETHPool family of L2 deposit contracts (`RSETHPool`, `RSETHPoolV2`, `RSETHPoolV3`, `RSETHPoolNoWrapper`, and their `ExternalBridge` variants). Every `deposit()` overload computes the rsETH output from a live oracle rate and transfers it with no minimum-output guard: [5](#0-4) [6](#0-5) 

### Impact Explanation

**Low — Contract fails to deliver promised returns without losing principal.**

A user who calls `instantWithdrawal` expecting to receive `X` units of an LST or ETH may receive materially less if either `rsETHPrice` decreases or `getAssetPrice(asset)` increases between the block the transaction is submitted and the block it is included. Because the rsETH burn is irreversible and occurs before the payout is transferred, the user has no recourse. The same oracle-rate exposure applies to depositors in the RSETHPool family, who may receive fewer rsETH tokens than the rate quoted at submission time.

### Likelihood Explanation

**Medium.** Both `rsETHPrice` (derived from total protocol assets and total rsETH supply) and LST asset prices (Chainlink feeds) update regularly. On a congested network, a transaction can sit in the mempool for multiple blocks, during which oracle prices can shift. No attacker action is required; normal market volatility is sufficient to trigger the discrepancy. The instant-withdrawal path is specifically designed for time-sensitive users, making them more likely to submit transactions during volatile periods.

### Recommendation

1. Add a `minAssetAmountOut` parameter to `instantWithdrawal` and revert if `assetAmountUnlocked < minAssetAmountOut`, mirroring the `minRSETHAmountExpected` guard already present in `LRTDepositPool`.
2. Add a `minRsETHAmountOut` parameter to every `deposit()` overload in the RSETHPool family and revert if the computed `rsETHAmount` falls below it.

### Proof of Concept

1. User observes `rsETHPrice = 1.05e18` and `getAssetPrice(stETH) = 1.00e18`, expecting `1.05 stETH` for `1 rsETH`.
2. User submits `instantWithdrawal(stETH, 1e18, "")`.
3. Before inclusion, an oracle update sets `rsETHPrice = 1.00e18` (e.g., a slashing event reduces total assets).
4. `getExpectedAssetAmount` returns `1.00e18` (1 stETH) instead of `1.05e18`.
5. Line 229 burns the user's `1e18` rsETH.
6. User receives `1.00 stETH` minus the instant-withdrawal fee — `0.05 stETH` less than expected — with no on-chain protection and no ability to cancel. [7](#0-6)

    function instantWithdrawal(
        address asset,
        uint256 rsETHUnstaked,
        string calldata referralId
    )
        external
        nonReentrant
        whenNotPaused
        onlySupportedAsset(asset)
        onlySupportedStrategy(asset)
        onlyInstantWithdrawalAllowed(asset)
    {
        if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
            revert InvalidAmountToWithdraw();
        }
        if (IERC20(lrtConfig.rsETH()).balanceOf(msg.sender) < rsETHUnstaked) revert NotEnoughRsETH();
        uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
        IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
        ILRTUnstakingVault unstakingVault = ILRTUnstakingVault(lrtConfig.getContract(LRTConstants.LRT_UNSTAKING_VAULT));
        if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
            revert CantInstantWithdrawMoreThanAvailable();
        }

        unstakingVault.redeem(asset, assetAmountUnlocked);

        uint256 fee = (assetAmountUnlocked * instantWithdrawalFee) / 10_000;
        uint256 userAmount = assetAmountUnlocked - fee;

        address feeRecipient = instantWithdrawalFeeRecipient;
        if (feeRecipient == address(0)) {
            // Backwards-compatible default: send fees to the protocol treasury
            feeRecipient = lrtConfig.getContract(LRTConstants.PROTOCOL_TREASURY);
        }
        if (fee > 0) {
            _transferAsset(asset, feeRecipient, fee);
            emit InstantWithdrawalFeeCollected(msg.sender, asset, fee);
        }

        _transferAsset(asset, msg.sender, userAmount);
```
