### Title
Rewards permanently lost when `totalKernelStaked` drops to zero mid-period via `initiateWithdrawal` - (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary

`KernelDepositPool` uses a Synthetix-style reward accumulator. When `totalKernelStaked` is zero, `rewardPerToken()` returns the stored value unchanged, but the `updateReward` modifier still advances `updatedAt` to the current timestamp. Any rewards that accrued during the zero-supply window are permanently unrecoverable.

### Finding Description

`rewardPerToken()` short-circuits when `totalKernelStaked == 0`: [1](#0-0) 

The `updateReward` modifier always writes `updatedAt = lastTimeRewardApplicable()` regardless of whether supply is zero: [2](#0-1) 

`initiateWithdrawal` immediately decrements `totalKernelStaked` (the KERNEL tokens are only returned after a delay, but the accounting balance drops at once): [3](#0-2) 

`notifyRewardAmount` guards against starting a reward period with zero supply, but provides no protection against supply dropping to zero mid-period: [4](#0-3) 

The contract's own NatSpec explicitly acknowledges this residual risk as an off-chain operational assumption rather than an on-chain invariant: [5](#0-4) 

**Attack / scenario trace:**

1. Admin calls `notifyRewardAmount` while `totalKernelStaked > 0` — passes the guard, reward period begins.
2. The sole (or last) staker calls `initiateWithdrawal(fullBalance)` — `totalKernelStaked` drops to `0` immediately; `updateReward` snapshots `rewardPerTokenStored` correctly up to this point and sets `updatedAt = block.timestamp`.
3. Time elapses (up to the 30-day `MAX_WITHDRAWAL_DELAY`) with `totalKernelStaked == 0`. Every external call that triggers `updateReward` (e.g., a new staker's `stake`) finds `totalKernelStaked == 0`, so `rewardPerTokenStored` stays frozen while `updatedAt` keeps advancing.
4. When a new staker eventually arrives, `rewardPerToken()` computes only from the moment they staked, skipping the entire zero-supply window. The `rewardRate * elapsed_zero_supply_seconds` worth of tokens remain locked in the contract with no recovery path.

### Impact Explanation

Reward tokens accrued during the zero-supply interval are permanently frozen inside `KernelDepositPool`. There is no admin sweep, no rescue function, and no mechanism to roll the lost rewards into a future period. The frozen amount scales linearly with `rewardRate` and the duration of the zero-supply window — up to 30 days of rewards per withdrawal cycle.

This matches: **Medium — Permanent freezing of unclaimed yield.**

### Likelihood Explanation

Any single staker who holds 100 % of `totalKernelStaked` can trigger this unilaterally by calling `initiateWithdrawal`. No admin action, no governance, and no coordination is required. The withdrawal delay (up to 30 days) means the zero-supply window can be substantial. The scenario is realistic whenever the pool has few participants or one dominant staker.

### Recommendation

Enforce that `totalKernelStaked > 0` cannot persist during an active reward window, or skip advancing `updatedAt` when supply is zero (mirror the `rewardPerTokenStored` freeze to `updatedAt` as well):

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    // Only advance updatedAt when there is supply to absorb rewards
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

This ensures that time during which no tokens are staked does not consume the reward budget — the rewards remain distributable once stakers return.

### Proof of Concept

```
T=0:  Admin calls notifyRewardAmount(1_000e18), duration=30 days
      → rewardRate = 1_000e18 / 30 days ≈ 385e12 tokens/sec
      → finishAt = T+30d, updatedAt = T=0

T=1d: Alice (sole staker, 100 KERNEL) calls initiateWithdrawal(100e18)
      updateReward runs: rewardPerTokenStored += rewardRate*1d/100e18 (correct)
                         updatedAt = T=1d
      totalKernelStaked = 0

T=1d–T=10d: No stakers. Any call to updateReward finds totalKernelStaked==0,
            rewardPerTokenStored stays frozen, but updatedAt advances each time.

T=10d: Bob stakes 100 KERNEL.
       updateReward runs with totalKernelStaked still 0 (modifier runs before body):
         rewardPerTokenStored unchanged, updatedAt = T=10d

T=10d+: Bob earns rewards only from T=10d onward.
        9 days × rewardRate × 1e18 / 100e18 ≈ 9 × 385e12 × 86400 ≈ 299e18 tokens
        are permanently locked in the contract.
```
