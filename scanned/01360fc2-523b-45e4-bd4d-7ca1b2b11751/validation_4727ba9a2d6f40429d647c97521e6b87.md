### Title
Arithmetic Underflow in `remainingDailyMintLimit()` Causes Uninformative Revert Before `startTimestamp` - (File: contracts/pools/RSETHPoolV2.sol)

### Summary
`remainingDailyMintLimit()` calls `getCurrentDay()`, which performs `block.timestamp - startTimestamp` without first verifying `block.timestamp >= startTimestamp`. Because `reinitialize()` mandates `startTimestamp` be in the future, there is always a guaranteed window where this subtraction underflows and reverts with a generic arithmetic error instead of a meaningful message. The same pattern is replicated across all pool variants.

### Finding Description
`reinitialize()` enforces that `startTimestamp` must be strictly in the future: [1](#0-0) 

This creates a guaranteed pre-activation window where `block.timestamp < startTimestamp`. During this window, `getCurrentDay()` underflows: [2](#0-1) 

The `limitDailyMint` modifier correctly guards against this by checking `block.timestamp < startTimestamp` before ever calling `getCurrentDay()`: [3](#0-2) 

However, `remainingDailyMintLimit()` calls `getCurrentDay()` directly with no such guard: [4](#0-3) 

The same unguarded pattern exists identically in:
- `contracts/pools/RSETHPoolV3.sol` lines 345–350 [5](#0-4) 
- `contracts/pools/RSETHPoolV3ExternalBridge.sol` lines 554–559 [6](#0-5) 
- `contracts/pools/RSETHPoolV3WithNativeChainBridge.sol` lines 396–401 [7](#0-6) 
- `contracts/pools/RSETHPoolV2ExternalBridge.sol` lines 380–385 [8](#0-7) 

`getNextDailyLimitResetTimestamp()` has the same root cause in all variants. [9](#0-8) 

### Impact Explanation
Any caller — off-chain integration, front-end, or on-chain contract — invoking `remainingDailyMintLimit()` or `getNextDailyLimitResetTimestamp()` before `startTimestamp` receives a generic Solidity 0.8 arithmetic underflow revert rather than a meaningful error. The contract fails to deliver its promised view return. No funds are at risk.

**Impact: Low** — Contract fails to deliver promised returns, but doesn't lose value.

### Likelihood Explanation
The underflow is guaranteed to be reachable during the window between `reinitialize()` and `startTimestamp`. Since `reinitialize()` requires `startTimestamp` to be strictly in the future, this window always exists after each reinitialisation. Any unprivileged external caller can trigger it with a simple view call.

### Recommendation
Add a pre-activation guard in `remainingDailyMintLimit()` and `getNextDailyLimitResetTimestamp()` mirroring the one already present in `limitDailyMint`:

```solidity
function remainingDailyMintLimit() external view returns (uint256) {
    if (block.timestamp < startTimestamp) return dailyMintLimit;
    uint256 effectiveDailyMintAmount = (getCurrentDay() > lastMintDay) ? 0 : dailyMintAmount;
    return dailyMintLimit - effectiveDailyMintAmount;
}
```

Apply the same fix to `getNextDailyLimitResetTimestamp()` across all affected pool contracts.

### Proof of Concept
1. Admin calls `reinitialize(1000 ether, block.timestamp + 1 days)` — `startTimestamp` is set to tomorrow.
2. Any user calls `remainingDailyMintLimit()`.
3. `getCurrentDay()` computes `block.timestamp - startTimestamp`; since `block.timestamp < startTimestamp` and both are `uint256`, Solidity 0.8 reverts with an arithmetic underflow panic.
4. The caller receives a generic revert with no actionable error message, while the `limitDailyMint` modifier would have given `MintBeforeStartTimestamp`.
