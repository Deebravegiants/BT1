### Title
Withdrawal Queue Liveness Failure: `getAvailableAssetAmount` Counts Illiquid EigenLayer Assets, Causing Temporary Fund Freeze and Stale-Price Execution - (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTWithdrawalManager.initiateWithdrawal` gates new requests against `getAvailableAssetAmount`, which counts assets locked inside EigenLayer strategies as "available." However, `unlockQueue` can only draw from the liquid `LRTUnstakingVault` balance. When the vault is empty, the queue cannot be unlocked regardless of how much total protocol TVL exists, leaving the user's rsETH frozen in the contract with no cancellation path. During the mandatory EigenLayer unwind delay (≥7 days), the oracle price can move, and `_calculatePayoutAmount` will pay out `min(originalQuote, currentReturn)` — so a price drop during the wait causes the user to receive less than the amount they were quoted at request time.

---

### Finding Description

**Step 1 — `initiateWithdrawal` uses an overstated liquidity figure.**

`getAvailableAssetAmount` calls `lrtDepositPool.getTotalAssetDeposits(asset)`, which aggregates every location the asset can be:

```
assetLyingInDepositPool + assetLyingInNDCs
  + assetStakedInEigenLayer          ← illiquid, 7-day EigenLayer delay
  + assetUnstakingFromEigenLayer     ← illiquid, still in EigenLayer queue
  + assetLyingInConverter
  + assetLyingUnstakingVault
``` [1](#0-0) 

The check in `initiateWithdrawal` passes as long as `expectedAssetAmount ≤ totalAssets − assetsCommitted`, even when the vast majority of those assets are locked in EigenLayer and cannot be redeemed for days. [2](#0-1) 

**Step 2 — `unlockQueue` can only draw from the liquid vault.**

`_createUnlockParams` reads `unstakingVault.balanceOf(asset)`, which is only the ETH/LST physically sitting in `LRTUnstakingVault`. If that balance is zero, `unlockQueue` reverts immediately:

```solidity
if (params.totalAvailableAssets == 0) revert AmountMustBeGreaterThanZero();
``` [3](#0-2) [4](#0-3) 

**Step 3 — rsETH is locked with no cancellation path.**

Once `initiateWithdrawal` executes, the user's rsETH is held by the contract. There is no `cancelWithdrawal` function. The user cannot reclaim their rsETH until an operator (a) queues an EigenLayer withdrawal, (b) waits the EigenLayer delay (~7 days), (c) calls `completeUnstaking` on `NodeDelegator`, and (d) routes funds to the vault, and (e) calls `unlockQueue`. The user's rsETH is frozen for the entire duration. [5](#0-4) 

**Step 4 — Stale-price execution when liquidity finally arrives.**

`expectedAssetAmount` is fixed at request time using the oracle's cached `rsETHPrice` and `getAssetPrice(asset)`. At unlock time, `_calculatePayoutAmount` computes:

```solidity
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
``` [6](#0-5) 

If the rsETH/asset exchange rate has declined during the multi-day unwind, `currentReturn < expectedAssetAmount` and the user receives less than the amount they were quoted. The user accepted the queue delay expecting the originally quoted amount; the protocol delivers a lower figure with no recourse.

---

### Impact Explanation

**Temporary freezing of funds (Medium):** A user who calls `initiateWithdrawal` when the vault is empty but EigenLayer TVL is large will have their rsETH locked in `LRTWithdrawalManager` for the full EigenLayer unwind period (≥7 days on top of `withdrawalDelayBlocks` ≈ 8 days), with no ability to cancel.

**Contract fails to deliver promised returns (Low):** The `expectedAssetAmount` stored at request time is the amount the user was quoted. If the oracle price moves adversely during the extended wait, the user receives a lower payout than quoted, with no slippage protection or cancellation right.

---

### Likelihood Explanation

This is a normal operating condition, not an edge case. The protocol actively moves assets from `LRTDepositPool` to `NodeDelegator` and into EigenLayer strategies. The `LRTUnstakingVault` balance is routinely near zero between unwind cycles. Any user who initiates a withdrawal during such a period will experience the freeze. The EigenLayer withdrawal delay is a protocol-level constant (~7 days), making the window for price movement substantial.

---

### Recommendation

1. **Fix the availability check:** `getAvailableAssetAmount` should count only liquid assets (vault balance + deposit pool balance), not EigenLayer-locked assets. This prevents users from queuing withdrawals that cannot be serviced without an EigenLayer unwind.

2. **Add a cancellation path:** Allow users to cancel a pending (not yet unlocked) withdrawal request and reclaim their rsETH, so they are not indefinitely frozen while waiting for operator action.

3. **Positive-slippage protection:** In `_calculatePayoutAmount`, if the price has moved in the user's favor (`currentReturn > expectedAssetAmount`), consider paying `currentReturn` rather than capping at the original quote, consistent with the recommendation in the reference report.

---

### Proof of Concept

1. Protocol has 1000 stETH: 950 stETH staked in EigenLayer, 50 stETH in `LRTUnstakingVault`, `assetsCommitted[stETH] = 0`.
2. User calls `initiateWithdrawal(stETH, rsETHAmount)` where `expectedAssetAmount = 40 stETH`.
3. `getAvailableAssetAmount` returns `1000 − 0 = 1000 stETH` → check passes. rsETH is transferred to the contract. `assetsCommitted[stETH] = 40`.
4. Operator calls `unlockQueue(stETH, ...)`. `unstakingVault.balanceOf(stETH) = 50 ≥ 40` → succeeds. *(Vault had enough liquid assets this time.)*

**Now the problematic scenario:**

1. Protocol has 1000 stETH: 990 stETH in EigenLayer, 10 stETH in vault, `assetsCommitted[stETH] = 0`.
2. User calls `initiateWithdrawal(stETH, rsETHAmount)` where `expectedAssetAmount = 40 stETH`.
3. `getAvailableAssetAmount` returns `1000 − 0 = 1000 stETH` → check passes. rsETH locked. `assetsCommitted[stETH] = 40`.
4. Operator calls `unlockQueue(stETH, ...)`. `unstakingVault.balanceOf(stETH) = 10 < 40` → `availableAssetAmount < payoutAmount` → loop breaks immediately, nothing unlocked. Reverts if `rsETHBurned == 0` is not handled gracefully, or silently does nothing.
5. Operator must queue EigenLayer withdrawal → wait ≥7 days → `completeUnstaking` → route to vault.
6. During those 7 days, stETH/ETH rate drops 2%. At unlock time, `currentReturn = 39.2 stETH < 40 stETH = expectedAssetAmount`. User receives 39.2 stETH instead of the quoted 40 stETH, with no recourse. [7](#0-6) [8](#0-7)

    /// @notice Calculates the amount of asset available for withdrawal.
    /// @param asset The asset address.
    /// @return availableAssetAmount The asset amount avaialble for withdrawal.
    function getAvailableAssetAmount(address asset) public view override returns (uint256 availableAssetAmount) {
        ILRTDepositPool lrtDepositPool = ILRTDepositPool(lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL));
        uint256 totalAssets = lrtDepositPool.getTotalAssetDeposits(asset);
        availableAssetAmount = totalAssets > assetsCommitted[asset] ? totalAssets - assetsCommitted[asset] : 0;
    }
