### Title
Stale Cached Rate in `CrossChainRateReceiver` Enables Sandwich Attack on Rate Updates — (File: `contracts/cross-chain/CrossChainRateReceiver.sol`)

---

### Summary

The `CrossChainRateReceiver` stores a point-in-time snapshot of the rsETH/ETH exchange rate that is only updated when a LayerZero message arrives via `lzReceive`. All L2 deposit pools (`RSETHPoolV3`, `RSETHPoolNoWrapper`) price every deposit against this cached value. Because the rsETH rate continuously appreciates on L1 (staking rewards accrue), the L2 rate is structurally stale between updates. An unprivileged depositor can exploit the gap between the stale L2 rate and the true L1 rate by depositing ETH just before the cross-chain rate update lands, receiving more rsETH/wrsETH than fair value, and then redeeming at the correct rate on L1.

---

### Finding Description

**Root cause — `CrossChainRateReceiver.sol`**

`CrossChainRateReceiver` holds a single `uint256 public rate` that is written only inside `lzReceive`:

```solidity
// CrossChainRateReceiver.sol L95-L97
rate = _rate;
lastUpdated = block.timestamp;
emit RateUpdated(_rate);
```

There is no interpolation, no staleness guard, and no circuit-breaker on the magnitude of a jump. The `getRate()` view simply returns the raw cached value:

```solidity
// CrossChainRateReceiver.sol L103-L105
function getRate() external view returns (uint256) {
    return rate;
}
```

**Consumption in deposit pools**

Both pool variants delegate all deposit pricing to this cached rate.

`RSETHPoolV3.viewSwapRsETHAmountAndFee` (ETH path):
```solidity
// RSETHPoolV3.sol L303-L307
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

`RSETHPoolNoWrapper.viewSwapRsETHAmountAndFee` (ETH path):
```solidity
// RSETHPoolNoWrapper.sol L282-L285
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

Both `getRate()` implementations simply forward to `IOracle(rsETHOracle).getRate()` with no freshness check:

```solidity
// RSETHPoolV3.sol L235-L237
function getRate() public view returns (uint256) {
    return IOracle(rsETHOracle).getRate();
}
```

**Rate update mechanism — `MultiChainRateProvider.sol`**

The rate is pushed from L1 to all registered L2 receivers by calling `updateRate()`, which anyone can trigger (it is not access-controlled):

```solidity
// MultiChainRateProvider.sol L108-L137
function updateRate() external payable nonReentrant {
    uint256 latestRate = getLatestRate();
    rate = latestRate;
    ...
    ILayerZeroEndpoint(layerZeroEndpoint).send{value: estimatedFee}(...);
}
```

The LayerZero cross-chain delivery introduces an unavoidable delay (minutes to hours). During this window the L2 oracle is stale.

**Attack scenario (RSETHPoolNoWrapper — Arbitrum/Unichain)**

`RSETHPoolNoWrapper.deposit` transfers actual rsETH OFT tokens from the pool's pre-minted reserves directly to the depositor:

```solidity
// RSETHPoolNoWrapper.sol L237-L243
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
rsETH.safeTransfer(msg.sender, rsETHAmount);
```

Timeline:
1. L1 rsETH rate increases from `R_old` to `R_new` (e.g. due to reward accrual or rebalancing).
2. `updateRate()` is called on L1; the LayerZero message is in flight.
3. **Before** the message lands on L2, attacker calls `deposit(referralId)` with `ETH_amount`.
   - Receives `rsETHAmount = ETH_amount * 1e18 / R_old` (more rsETH because `R_old < R_new`).
4. LayerZero message lands; `lzReceive` sets `rate = R_new`.
5. Attacker bridges rsETH OFT back to L1 and redeems at `R_new`.
   - ETH received = `rsETHAmount * R_new / 1e18 = ETH_amount * R_new / R_old > ETH_amount`.

Profit per unit invested = `R_new / R_old − 1`.

The same attack applies to `RSETHPoolV3` (wrsETH minting path), where the attacker deposits ETH, receives excess wrsETH, bridges to L1, and unwraps at the correct rate.

---

### Impact Explanation

**Critical — Direct theft of user/protocol funds.**

In `RSETHPoolNoWrapper`, the pool holds a finite pre-minted rsETH OFT reserve. Every deposit at a stale low rate drains more rsETH from that reserve than the deposited ETH is worth at the true rate. The excess rsETH is extracted by the attacker and redeemed on L1 for ETH, leaving the pool short. Honest depositors who arrive after the rate update receive fewer rsETH per ETH, effectively subsidising the attacker.

In `RSETHPoolV3`, the wrsETH minted in excess of fair value represents an over-claim on the underlying ETH collateral, diluting all existing wrsETH holders.

---

### Likelihood Explanation

**Medium-High.**

- The rsETH rate increases continuously as EigenLayer staking rewards accrue; every rate update creates a window.
- `updateRate()` on `MultiChainRateProvider` is permissionless — anyone can trigger it, making the exact timing of the L2 update observable on-chain.
- LayerZero message delivery latency (typically minutes) is publicly visible; an attacker can monitor the L1 `RateUpdated` event and front-run the L2 `lzReceive` call.
- No staleness check, no slippage guard, and no rate-change circuit-breaker exist in any pool variant.
- The attack requires no special role, no governance capture, and no external protocol compromise — only a standard ETH deposit.

---

### Recommendation

1. **Staleness check**: Revert deposits if `block.timestamp - lastUpdated > MAX_STALENESS` (e.g. 1 hour).
2. **Rate-change circuit-breaker**: Revert or pause deposits if the incoming rate deviates from the previous rate by more than a threshold (e.g. 0.5%).
3. **Rate smoothing**: Interpolate the rate linearly from `rate_old` to `rate_new` over a short window (e.g. 30 minutes) after each `lzReceive`, analogous to ERC-4626 vault share-price smoothing.
4. **Reduce update frequency gap**: Trigger `updateRate()` automatically on every L1 rsETH rate change rather than relying on periodic keeper calls.

---

### Proof of Concept

```
Assumptions:
  R_old = 1.050 ETH/rsETH  (stale L2 rate)
  R_new = 1.055 ETH/rsETH  (true L1 rate after reward accrual)
  ETH deposited = 100 ETH
  feeBps = 0 (for clarity)

Step 1 — deposit before lzReceive:
  rsETHAmount = 100e18 * 1e18 / 1.050e18 = 95.238 rsETH

Step 2 — lzReceive lands, rate = 1.055e18

Step 3 — redeem on L1:
  ETH received = 95.238 * 1.055 = 100.476 ETH

Profit = 0.476 ETH (~0.476%) on 100 ETH in a single block.

At $100M pool TVL and a 0.5% rate gap (plausible after several hours of
reward accrual without an update), attacker profit ≈ $500,000.
```
