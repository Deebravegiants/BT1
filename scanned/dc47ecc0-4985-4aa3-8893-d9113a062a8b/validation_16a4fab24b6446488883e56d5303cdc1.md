### Title
Dust Stake Captures All Remaining Rewards When `totalKernelStaked` Drops to Near Zero - (File: contracts/KERNEL/KernelDepositPool.sol)

---

### Summary

`KernelDepositPool` enforces no minimum stake threshold. When `totalKernelStaked` falls to zero during an active reward period, an attacker can stake 1 wei of KERNEL and earn 100% of all remaining rewards for that period. The protocol's own NatSpec acknowledges this risk but relies on an off-chain deployment assumption rather than any on-chain guard.

---

### Finding Description

The `stake()` function rejects only a zero amount: [1](#0-0) 

The `rewardPerToken()` function stops accumulating when `totalKernelStaked == 0` (rewards are silently frozen in the contract), but resumes the moment any non-zero stake exists: [2](#0-1) 

The `earned()` function then attributes the full per-token accumulation to the sole staker: [3](#0-2) 

When the attacker stakes 1 wei after `totalKernelStaked` has dropped to 0:

- `updateReward` snapshots `updatedAt = block.timestamp`, so the zero-stake gap is skipped.
- From that moment forward, `rewardPerToken()` accumulates as `rewardRate * elapsed * 1e18 / 1`, making `earned(attacker) = rewardRate * elapsed` — the **full** remaining reward budget.

The protocol's own comment acknowledges the risk but provides no on-chain enforcement: [4](#0-3) 

The `notifyRewardAmount` guard prevents starting a new period with zero stakers, but does **not** prevent stakers from withdrawing after a period has started: [5](#0-4) 

The `initiateWithdrawal` function freely reduces `totalKernelStaked` to zero with no floor: [6](#0-5) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

An attacker holding 1 wei of KERNEL (negligible cost) can claim 100% of the remaining reward budget for an active distribution period. Rewards that accrued during the zero-stake gap are permanently frozen in the contract (permanent freezing of unclaimed yield), and all rewards from the moment of the dust stake onward flow entirely to the attacker.

---

### Likelihood Explanation

**Medium.** The attack requires `totalKernelStaked` to reach zero during an active reward period. This is realistic during:

- Low-activity periods when legitimate stakers exit en masse.
- End-of-vesting events where `KernelMerkleDistributor` recipients claim and immediately withdraw.
- Coordinated exits triggered by market conditions.

An attacker can monitor `totalKernelStaked` on-chain and execute the dust stake atomically once the condition is met. The cost is 1 wei of KERNEL plus gas.

---

### Recommendation

Introduce a minimum stake threshold enforced at the contract level, analogous to the Ajna mitigation of a minimum vote power:

```solidity
uint256 public minStakeAmount; // set by admin, e.g. 1e18

function stake(uint256 _amount) external nonReentrant updateReward(msg.sender) {
    if (_amount == 0 || _amount < minStakeAmount) revert AmountTooSmall();
    ...
}
```

Alternatively, when `totalKernelStaked` drops to zero mid-period, pause reward accumulation and redirect unallocated rewards to the protocol treasury rather than leaving them claimable by the next dust staker.

---

### Proof of Concept

**Schematic PoC:**

1. Admin calls `notifyRewardAmount(1_000e18)` with a 30-day period while 100 legitimate stakers hold KERNEL.
2. On day 15, all legitimate stakers call `initiateWithdrawal` and eventually `claimWithdrawal`; `totalKernelStaked` → 0.
3. `rewardPerTokenStored` freezes; 500 KERNEL worth of rewards for days 15–20 accumulate but are not distributed.
4. On day 20, attacker calls `stake(1)` (1 wei of KERNEL).
   - `updateReward` snapshots `updatedAt = block.timestamp` (day 20), skipping the frozen gap.
   - `userRewardPerTokenPaid[attacker] = rewardPerTokenStored` (frozen value).
5. On day 30, attacker calls `getReward()`.
   - `rewardPerToken()` = `rewardPerTokenStored + rewardRate * 10 days * 1e18 / 1`
   - `earned(attacker)` = `1 * (rewardPerToken() - userRewardPerTokenPaid[attacker]) / 1e18` = `rewardRate * 10 days` ≈ **333 KERNEL**
6. Attacker receives 333 KERNEL for a cost of 1 wei of KERNEL. The 167 KERNEL from days 15–20 remain permanently locked. [7](#0-6)

    function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
        if (_amount == 0) revert AmountZero();
        if (balanceOf[msg.sender] < _amount) revert InsufficientStakedBalance();
        if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();

        balanceOf[msg.sender] -= _amount;
        totalKernelStaked -= _amount;

        // Create a withdrawal record
        uint256 withdrawalId = ++withdrawalCounter;
        uint256 unlockTime = block.timestamp + withdrawalDelay;

        withdrawals[withdrawalId] = Withdrawal({
            user: msg.sender, amount: _amount, unlockTime: unlockTime, claimed: false, withdrawalId: withdrawalId
        });
        userWithdrawalIds[msg.sender].push(withdrawalId);

        emit WithdrawalInitiated(msg.sender, _amount, withdrawalId, unlockTime);
    }
