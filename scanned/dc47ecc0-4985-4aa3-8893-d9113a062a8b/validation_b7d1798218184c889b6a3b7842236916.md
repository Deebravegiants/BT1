### Title
`updateRSETHPrice()` Permanently Blocked When Accumulated Fee Exceeds `maxFeeMintAmountPerDay` — (File: contracts/LRTOracle.sol)

---

### Summary

`LRTOracle.updateRSETHPrice()` will revert with `DailyFeeMintLimitExceeded` whenever the protocol fee to be minted in a single call exceeds `maxFeeMintAmountPerDay`. Because a failed update leaves `rsETHPrice` stale, the next call computes an even larger fee (the gap between actual TVL and the stale price widens), making the block self-reinforcing. Protocol fee yield is frozen and the rsETH exchange rate becomes stale until an admin raises the limit — a manual intervention analogous to the contract upgrade required in the reported finding.

---

### Finding Description

`_updateRsETHPrice()` computes a protocol fee and then unconditionally calls `_checkAndUpdateDailyFeeMintLimit`:

```solidity
// contracts/LRTOracle.sol  lines 299-310
if (protocolFeeInETH > 0) {
    uint256 rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice);
    _checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee);   // ← reverts here
    ...
} else {
    _checkAndUpdateDailyFeeMintLimit(0);
}
```

Inside `_checkAndUpdateDailyFeeMintLimit`:

```solidity
// contracts/LRTOracle.sol  lines 205-206
if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
    revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
}
``` [1](#0-0) 

When this reverts, the entire `_updateRsETHPrice()` call is rolled back: `rsETHPrice` is **not** updated and no fee is minted. The stored `rsETHPrice` therefore remains at its previous (lower) value. On the next call, `previousTVL = rsethSupply.mulWad(rsETHPrice)` is computed with the stale price, so `rewardAmount = totalETHInProtocol - previousTVL` is even larger, producing an even larger fee — which again exceeds the limit. The block is self-reinforcing. [2](#0-1) 

`maxFeeMintAmountPerDay` is a storage variable initialised to `0` and set only by the manager:

```solidity
// contracts/LRTOracle.sol  line 35
uint256 public maxFeeMintAmountPerDay;
``` [3](#0-2) 

If it is `0` (unset) or set too conservatively, any non-zero fee causes an immediate revert. The public entry point `updateRSETHPrice()` is callable by anyone:

```solidity
// contracts/LRTOracle.sol  line 87
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [4](#0-3) 

The manager-only variant `updateRSETHPriceAsManager()` calls the same internal function and is equally affected. [5](#0-4) 

---

### Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

1. **Protocol fee yield frozen.** Every call to `updateRSETHPrice()` reverts; the rsETH fee tokens that should be minted to the treasury are never created. The longer the block persists, the larger the deferred fee grows, making recovery harder.

2. **Stale rsETH exchange rate.** `LRTWithdrawalManager.getExpectedAssetAmount()` and `initiateWithdrawal()` read `lrtOracle.rsETHPrice()` directly. A stale (lower) price means users who initiate withdrawals receive fewer underlying assets than the true exchange rate entitles them to — the protocol fails to deliver promised returns.

3. **Downside-protection mechanism disabled.** The auto-pause logic that compares `newRsETHPrice` against `highestRsethPrice` and pauses `LRTDepositPool` and `LRTWithdrawalManager` on a large price drop is inside `_updateRsETHPrice()`. If the function always reverts before reaching that logic, the circuit-breaker never fires. [6](#0-5) 

---

### Likelihood Explanation

**Medium.**

- `maxFeeMintAmountPerDay` defaults to `0`. If the manager sets it to a value that was appropriate at launch but the protocol TVL grows significantly, the daily fee will eventually exceed the cap.
- A single extended period without a price update (e.g., during a pause or network congestion) causes the deferred fee to spike above the limit on the next call.
- No attacker action is required; normal protocol operation (staking rewards accruing) is sufficient to trigger the condition.
- The only remediation path is a manager transaction to raise `maxFeeMintAmountPerDay` — identical in nature to the contract upgrade required in the reported finding. [7](#0-6) 

---

### Recommendation

Replace the hard revert with a cap-and-carry approach: mint only up to `maxFeeMintAmountPerDay` in the current period and defer the remainder, or skip fee minting for the period without reverting so that the price update always succeeds. For example:

```solidity
// Instead of reverting, cap the fee at the remaining daily allowance
uint256 remaining = maxFeeMintAmountPerDay - currentPeriodMintedFeeAmount;
uint256 mintableNow = feeAmount > remaining ? remaining : feeAmount;
currentPeriodMintedFeeAmount += mintableNow;
// return mintableNow to the caller so only the capped amount is minted
```

This ensures `updateRSETHPrice()` always completes, the price is always updated, and the downside-protection circuit-breaker always has the opportunity to fire.

---

### Proof of Concept

1. Manager calls `setMaxFeeMintAmountPerDay(X)` where `X` is a reasonable daily cap at launch.
2. Protocol TVL grows; staking rewards accumulate. No price update is called for 48 hours (e.g., during a pause).
3. Anyone calls `updateRSETHPrice()`.
4. `rewardAmount = totalETHInProtocol - rsethSupply * rsETHPrice` is large (48 h of rewards).
5. `rsethAmountToMintAsProtocolFee` exceeds `X`.
6. `_checkAndUpdateDailyFeeMintLimit` reverts with `DailyFeeMintLimitExceeded`.
7. `rsETHPrice` remains stale; fee is not minted.
8. The next call (even 24 h later after the period resets) computes an even larger `rewardAmount` because `rsETHPrice` is still the old value, so the fee is even larger — the block persists indefinitely.
9. All withdrawals use the stale (lower) rsETH price; users receive fewer assets than entitled. Protocol treasury receives no fee yield. The auto-pause circuit-breaker is inoperative. [8](#0-7)

        // Reset the period if it's unset or a day has passed
        if (block.timestamp >= feePeriodStartTime + 1 days) {
            currentPeriodMintedFeeAmount = 0;
            feePeriodStartTime = getCurrentPeriodStartTime();
