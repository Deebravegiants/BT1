### Title
ETH Deposit Limit Bypass Due to Missing Amount in Pre-Condition Check - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit()` applies an asymmetric validation: for ERC-20 LST deposits it correctly checks `totalAssetDeposits + amount > depositLimit`, but for ETH deposits it omits the incoming `amount` and only checks `totalAssetDeposits > depositLimit`. This means any depositor can push total ETH deposits arbitrarily above the configured cap in a single transaction.

### Finding Description
The function `_checkIfDepositAmountExceedesCurrentLimit` at line 676 of `contracts/LRTDepositPool.sol` contains two branches:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // ← amount NOT included
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← amount included
}
```

For ETH, the check only asks "has the limit already been exceeded?" — it does not ask "will this deposit push the total over the limit?" The `amount` parameter (i.e., `msg.value` from `depositETH`) is silently ignored in the ETH branch. The LST branch correctly adds `amount` to `totalAssetDeposits` before comparing.

This is called from `_beforeDeposit` (line 661), which is the sole guard invoked by `depositETH` (line 87) before minting rsETH.

### Impact Explanation
The ETH deposit cap (`depositLimitByAsset[ETH_TOKEN]`) is the protocol's primary risk-management control over how much ETH enters the system. Bypassing it allows any unprivileged depositor to:

1. Mint rsETH backed by ETH far in excess of the intended cap.
2. Force the protocol to hold more ETH than EigenLayer strategies are configured to absorb, creating an unbacked rsETH supply.
3. Undermine the protocol's solvency invariant — rsETH holders may not be able to redeem at par if the excess ETH cannot be restaked and yield-generating.

**Impact: Medium — contract fails to enforce its promised deposit ceiling; in the worst case, excess unbacked rsETH minting leads to protocol insolvency.**

### Likelihood Explanation
The entry point is `depositETH()`, which is a public, permissionless function callable by any user. No special role, front-running, or oracle manipulation is required. The attacker only needs to send ETH when `totalAssetDeposits ≤ depositLimit`. This is the normal operating state of the protocol, so the condition is trivially satisfied at any time before the cap is reached.

### Recommendation
Include the incoming `amount` in the ETH branch of `_checkIfDepositAmountExceedesCurrentLimit`, mirroring the LST branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This ensures the post-condition — that total ETH deposits remain within the configured limit after the deposit — is enforced for ETH just as it is for LSTs.

### Proof of Concept

1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 1000 ether`.
2. Current `getTotalAssetDeposits(ETH_TOKEN)` returns `999 ether`.
3. Attacker calls `depositETH{value: 5000 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 5000 ether)` evaluates `999 ether > 1000 ether` → `false` (limit not yet exceeded), so the check passes.
5. `_mintRsETH` mints rsETH for 5000 ETH.
6. `getTotalAssetDeposits(ETH_TOKEN)` is now `5999 ether` — nearly 6× the intended cap — with no revert.

The LST path under identical conditions (`totalAssetDeposits = 999 ether`, `amount = 5000 ether`) would correctly evaluate `999 + 5000 > 1000` → `true` and revert with `MaximumDepositLimitReached`. [1](#0-0) [2](#0-1) [3](#0-2)
