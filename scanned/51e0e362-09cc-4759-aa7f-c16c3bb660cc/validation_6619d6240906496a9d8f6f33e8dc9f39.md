### Title
`feeInBPS` Update Retroactively Applies to Pending Merkle Claims, Stealing Unclaimed Yield - (File: `contracts/KERNEL/KernelMerkleDistributor.sol`)

---

### Summary

`KernelMerkleDistributor`, `MerkleDistributor`, and `KernelTop100MerkleDistributor` all apply the **current** `feeInBPS` at the moment a user calls `claim()`, not the fee rate that was in effect when the merkle root encoding their rewards was committed. Because `setFeeInBPS()` can be called at any time with no guard over pending claimable balances, a fee-rate increase between merkle root publication and user claim silently redirects a portion of already-earned, already-committed user rewards to the protocol treasury — constituting theft of unclaimed yield.

---

### Finding Description

The merkle root encodes each user's cumulative claimable amount as an immutable off-chain commitment. Once `setMerkleRoot()` is called, every user's entitlement is fixed. However, the fee deduction is computed at claim time using the live `feeInBPS` storage variable:

**`KernelMerkleDistributor._processClaim()` (line 338):**
```solidity
uint256 fee = (claimableAmount * feeInBPS) / FEE_DENOMINATOR;
uint256 amountToSend = claimableAmount - fee;
``` [1](#0-0) 

**`setFeeInBPS()` (line 388–395) directly overwrites the rate with no guard on pending claims:**
```solidity
function setFeeInBPS(uint256 _feeInBPS) external onlyOwner {
    if (_feeInBPS > MAX_FEE_IN_BPS) {
        revert InvalidFeeInBPS();
    }
    feeInBPS = _feeInBPS;
    emit FeeInBPSUpdated(feeInBPS);
}
``` [2](#0-1) 

The identical pattern exists in `MerkleDistributor`: [3](#0-2) [4](#0-3) 

And in `KernelTop100MerkleDistributor` (which additionally has a 30-day vesting window, making the exposure window even longer): [5](#0-4) [6](#0-5) 

There is no snapshot of `feeInBPS` per merkle root index, no requirement to drain pending claims before updating the fee, and no per-user fee-rate record. The fee applied to a claim is always the value of `feeInBPS` at the block the transaction executes.

---

### Impact Explanation

**High — Theft of unclaimed yield.**

Any KERNEL rewards already committed in a published merkle root but not yet claimed are subject to whatever `feeInBPS` is set to at claim time. If the owner raises `feeInBPS` from 0 to 1000 (the maximum, 10%) after a merkle root is published, every user who has not yet claimed loses 10% of their earned-but-unclaimed rewards to the treasury. For `KernelTop100MerkleDistributor`, the 30-day vesting window means the entire vesting tail of every user's allocation is exposed for the full vesting duration. [7](#0-6) 

---

### Likelihood Explanation

**Medium.** The owner role is a single EOA or multisig with no timelock enforced at the contract level. A fee-rate update is a routine administrative action (e.g., adjusting protocol revenue share). The owner need not act maliciously — a well-intentioned fee change intended to apply only to *future* distributions silently and retroactively penalises all *existing* unclaimed balances. The window of exposure is the entire period between `setMerkleRoot()` and the last user claim, which for `KernelTop100MerkleDistributor` is at minimum 30 days. [8](#0-7) 

---

### Recommendation

1. **Snapshot the fee per merkle root index.** Store `feeInBPS` alongside each merkle root in a mapping keyed by `currentMerkleRootIndex`, and use the snapshotted value when processing claims against that root.
2. **Alternatively**, enforce that `setFeeInBPS()` only takes effect for merkle roots published *after* the update, by recording the root index at which the new fee becomes active.
3. For `KernelTop100MerkleDistributor`, record the `feeInBPS` at the time of `initialize()` and treat it as immutable for the lifetime of that deployment, since the merkle root is fixed at construction.

---

### Proof of Concept

```
State:
  feeInBPS = 0
  currentMerkleRoot = root_A  (encodes: user Alice → 1000 KERNEL cumulative)
  Alice has not yet claimed.

Step 1: Owner calls setFeeInBPS(1000).
  → feeInBPS is now 1000 (10%).

Step 2: Alice calls claim(index=1, account=Alice, cumulativeAmount=1000, proof).
  → claimableAmount = 1000 - 0 = 1000
  → fee = (1000 * 1000) / 10_000 = 100 KERNEL  ← taken from Alice's committed reward
  → amountToSend = 900 KERNEL

Result: Alice receives 900 KERNEL instead of the 1000 KERNEL committed in the merkle root.
        100 KERNEL is redirected to protocolTreasury.
        Alice's unclaimed yield is permanently reduced with no recourse.
``` [9](#0-8)
