### Title
Missing Token Decimal Normalization in `viewSwapRsETHAmountAndFee` Leads to Incorrect rsETH Minting - (`contracts/pools/RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV3.sol`, `RSETHPool.sol`, `RSETHPoolNoWrapper.sol`, `RSETHPoolV3WithNativeChainBridge.sol`)

---

### Summary

All L2 pool variants share a `viewSwapRsETHAmountAndFee(uint256 amount, address token)` function that computes the rsETH output for a deposited token. The formula silently assumes the deposited token has 18 decimals. No decimal normalization is applied, and `addSupportedToken` imposes no decimal constraint. If any non-18-decimal token is registered, every depositor using that token receives a drastically wrong rsETH amount.

---

### Finding Description

The shared swap formula across all pool variants is:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

`tokenToETHRate` is the price of **one full token** (e.g., 1 wstETH) expressed in 1e18 ETH precision, as returned by the oracle's `getRate()`. `rsETHToETHrate` is similarly 1e18-scaled. `amountAfterFee` is the raw token amount in the token's **native decimal units**.

For an 18-decimal token (e.g., wstETH, 1 token = 1e18 units):

```
rsETHAmount = 1e18 * 1.05e18 / 1.05e18 = 1e18  ✓  (1 rsETH)
```

For a hypothetical 6-decimal token worth 1 ETH per token (1 token = 1e6 units, `tokenToETHRate` = 1e18):

```
rsETHAmount = 1e6 * 1e18 / 1e18 = 1e6  ✗  (should be 1e18, off by 1e12)
```

The user receives `10^(18 − tokenDecimals)` times less rsETH than they are owed. Conversely, a 20-decimal token would cause 100× over-minting.

The `addSupportedToken` function in every pool variant performs no decimal check:

```solidity
function addSupportedToken(address token, address oracle, address bridge) external onlyRole(TIMELOCK_ROLE) {
    ...
    if (IOracle(oracle).getRate() == 0) revert UnsupportedOracle();
    supportedTokenList.push(token);
    supportedTokenOracle[token] = oracle;
    tokenBridge[token] = bridge;
    emit AddSupportedToken(token, oracle, bridge);
}
```

The only validation is that the oracle returns a non-zero rate — no check that `IERC20(token).decimals() == 18`.

---

### Impact Explanation

**Impact: Low — Contract fails to deliver promised returns.**

A depositor using a non-18-decimal token receives an rsETH amount that is off by a factor of `10^|18 − tokenDecimals|`. For a 6-decimal token this is a factor of 1e12 under-minting; for a 20-decimal token it is 100× over-minting (draining the pool's pre-minted rsETH). In either direction the depositor or the pool suffers a direct, quantifiable loss proportional to the decimal gap.

---

### Likelihood Explanation

**Likelihood: Low.**

All tokens currently supported by the protocol (wstETH, rETH, WETH, native ETH) have 18 decimals, so the bug is dormant in the current deployment. Triggering it requires a `TIMELOCK_ROLE` holder to call `addSupportedToken` with a non-18-decimal token — an administrative action, not an unprivileged one. However, the protocol provides no on-chain guard against this misconfiguration, making it a latent structural defect that becomes exploitable the moment the token set is expanded.

---

### Recommendation

1. Add a decimal enforcement check inside `_addSupportedToken` (or the inline `addSupportedToken` variants):

```solidity
require(IERC20Metadata(token).decimals() == 18, "Token must have 18 decimals");
```

2. Alternatively, normalize `amountAfterFee` before the multiplication:

```solidity
uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 normalizedAmount = amountAfterFee * 10 ** (18 - tokenDecimals);
rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
```

---

### Proof of Concept

Assume a 6-decimal token `T6` with `tokenToETHRate = 1e18` (1 T6 = 1 ETH) is added via `addSupportedToken`. A user calls `deposit(T6, 1e6, "ref")` (depositing 1 full T6 token).

1. `viewSwapRsETHAmountAndFee(1e6, T6)` is called.
2. `fee = 1e6 * feeBps / 10_000` (negligible).
3. `amountAfterFee ≈ 1e6`.
4. `tokenToETHRate = 1e18`, `rsETHToETHrate ≈ 1e18`.
5. `rsETHAmount = 1e6 * 1e18 / 1e18 = 1e6` wei of rsETH.
6. User receives `1e6` wei of rsETH ≈ `0.000000000000000001` rsETH.
7. Correct amount: `1e18` wei of rsETH (1 rsETH, since 1 T6 = 1 ETH = 1 rsETH at parity).
8. Loss factor: `1e12`.

    function _addSupportedToken(address token, address oracle, address bridge) internal {
        UtilLib.checkNonZeroAddress(token);
        UtilLib.checkNonZeroAddress(oracle);
        UtilLib.checkNonZeroAddress(bridge);

        if (supportedTokenOracle[token] != address(0)) {
            revert AlreadySupportedToken();

        if (tokenBridge[token] != address(0)) {
            revert AlreadySupportedToken();
        }

            revert UnsupportedOracle();
        }
        supportedTokenList.push(token);
        supportedTokenOracle[token] = oracle;
        tokenBridge[token] = bridge;

        emit AddSupportedToken(token, oracle, bridge);
    }
