### Title
Stale `InterimRSETHOracle` Rate Allows Depositors to Mint Unbacked wrsETH, Causing Protocol Insolvency ‚Äî (`contracts/pools/oracle/InterimRSETHOracle.sol`)

### Summary
`InterimRSETHOracle` stores the rsETH/ETH rate as a plain `uint256` with no timestamp or staleness guard. Any window where the true rsETH/ETH rate exceeds the stored value allows any depositor to mint more wrsETH than the deposited collateral is worth at the true rate, violating the full-backing invariant.

### Finding Description
`InterimRSETHOracle.getRate()` returns `rate` unconditionally: [1](#0-0) 

There is no `updatedAt` timestamp, no maximum age check, and no circuit breaker. The rate is updated only when a MANAGER_ROLE holder calls `setRate()`: [2](#0-1) 

`RSETHPoolV3.viewSwapRsETHAmountAndFee(amount, token)` uses this rate as the denominator: [3](#0-2) 

A stale-low `rsETHToETHrate` inflates `rsETHAmount`. The `deposit(token, amount, referralId)` function is callable by any address with no role restriction: [4](#0-3) 

By contrast, `ChainlinkOracleForRSETHPoolCollateral` ‚Äî used for the collateral token oracle ‚Äî does enforce staleness: [5](#0-4) 

This asymmetry means the collateral token rate is always fresh, but the rsETH denominator can be arbitrarily stale.

### Impact Explanation
Each deposit during a staleness window mints `(trueRate / staleRate - 1) * amountAfterFee` excess wrsETH. This wrsETH is redeemable against the pool's collateral. Repeated deposits drain the pool's collateral relative to outstanding wrsETH supply, causing protocol insolvency. The daily mint limit caps per-day damage but does not prevent the exploit across multiple days or at the limit boundary.

### Likelihood Explanation
`InterimRSETHOracle` is explicitly an interim, manually-operated oracle. rsETH/ETH rate increases monotonically as the LRT accrues yield. Any update interval (daily, weekly) creates a predictable staleness window. An attacker only needs to observe that `InterimRSETHOracle.rate < trueRsETHRate` on-chain and call `deposit`. No privileged access, no front-running dependency, no external protocol failure required.

### Recommendation
1. Add an `updatedAt` timestamp to `InterimRSETHOracle` and revert in `getRate()` if `block.timestamp - updatedAt > MAX_STALENESS`:

```solidity
uint256 public updatedAt;
uint256 public constant MAX_STALENESS = 24 hours;

function getRate() external view returns (uint256) {
    if (block.timestamp - updatedAt > MAX_STALENESS) revert StaleRate();
    return rate;
}

function _setRate(uint256 newRate) internal {
    if (newRate < 1e18) revert InvalidRate();
    rate = newRate;
    updatedAt = block.timestamp;
    emit RateUpdated(newRate);
}
```

2. Consider adding a maximum single-update deviation check (e.g., ¬±5%) to prevent a compromised manager from setting an extreme rate.
