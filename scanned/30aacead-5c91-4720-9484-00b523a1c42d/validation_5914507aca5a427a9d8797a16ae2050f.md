### Title
Manager Can Retroactively Extend Withdrawal Delay for Already-Queued Withdrawals With No User Cancel Path - (File: contracts/LRTWithdrawalManager.sol)

### Summary
`LRTWithdrawalManager.setWithdrawalDelayBlocks()` allows the LRT manager to immediately update `withdrawalDelayBlocks` with no timelock. Because `_processWithdrawalCompletion()` and `_unlockWithdrawalRequests()` both read the **current** `withdrawalDelayBlocks` at execution time rather than the value stored at request initiation, any increase retroactively extends the waiting period for all already-queued withdrawals. Users have no cancel function to recover their locked rsETH.

### Finding Description
When a user calls `initiateWithdrawal()`, their rsETH is immediately transferred to the contract and a `WithdrawalRequest` is stored with only `withdrawalStartBlock: block.number`. [1](#0-0) 

The delay check at completion time reads the **live** `withdrawalDelayBlocks` state variable: [2](#0-1) 

The same live read occurs in `_unlockWithdrawalRequests()`, which gates the operator-driven unlock step: [3](#0-2) 

The manager can change `withdrawalDelayBlocks` at any time with no timelock, up to the 16-day ceiling: [4](#0-3) 

The default value is `8 days / 12 seconds` (~57,600 blocks): [5](#0-4) 

There is no `cancelWithdrawal` function anywhere in `LRTWithdrawalManager`. Once `initiateWithdrawal` is called, the user's rsETH is irrecoverably locked until the request is processed.

### Impact Explanation
A user who queues a withdrawal expecting an 8-day delay can have that delay silently doubled to 16 days by a single manager transaction. Their rsETH is already held by the contract and `assetsCommitted` is already incremented, so neither the rsETH nor the underlying asset is accessible to the user during the extended period. This constitutes a **temporary freezing of funds** matching the Medium impact tier.

### Likelihood Explanation
The `onlyLRTManager` role is an operational role expected to be used regularly. The change requires no timelock, no multisig threshold beyond what the manager role already holds, and no on-chain announcement. Any manager key action — whether intentional policy change or error — immediately affects all in-flight withdrawals. Users have no on-chain mechanism to observe and react before the change takes effect on their pending request.

### Recommendation
Store the `withdrawalDelayBlocks` value **at request initiation time** inside the `WithdrawalRequest` struct, so each request is bound to the delay that was in effect when the user committed their rsETH. Alternatively, apply delay changes only to requests created **after** the change, by comparing `request.withdrawalStartBlock` against a checkpoint block stored alongside each delay update.

### Proof of Concept
1. User calls `initiateWithdrawal(asset, rsETHAmount, "")`. rsETH is transferred to the contract; `WithdrawalRequest.withdrawalStartBlock = block.number` is stored. Default `withdrawalDelayBlocks = 57600` (~8 days).
2. Manager calls `setWithdrawalDelayBlocks(115200)` (16 days / 12 s), effective immediately with no timelock.
3. User attempts `completeWithdrawal(asset, "")` after 8 days. `_processWithdrawalCompletion` checks `block.number < request.withdrawalStartBlock + 115200` — the check fails and reverts with `WithdrawalDelayNotPassed`.
4. Operator attempts `unlockQueue(...)` — `_unlockWithdrawalRequests` hits the same check at line 795 and breaks out of the loop, leaving the request locked.
5. User has no `cancelWithdrawal` path. rsETH remains locked in the contract for an additional 8 days beyond the originally expected delay. [4](#0-3) [2](#0-1) [3](#0-2) [1](#0-0)

    function setWithdrawalDelayBlocks(uint256 withdrawalDelayBlocks_) external onlyLRTManager {
        // Set an upper limit of no more than 16 days
        if (withdrawalDelayBlocks_ > 16 days / 12 seconds) revert ExceedWithdrawalDelay();

        withdrawalDelayBlocks = withdrawalDelayBlocks_;
        emit WithdrawalDelayBlocksUpdated(withdrawalDelayBlocks);
    }
