### Title
Unconditional zero-value treasury transfer in `claim()` blocks small-balance claimants when token reverts on zero transfers — (`contracts/utils/MerkleDistributor/MerkleDistributor.sol`)

### Summary
`MerkleDistributor.claim()` unconditionally calls `safeTransfer(protocolTreasury, fee)` even when integer division yields `fee = 0`. If the distributed token reverts on zero-value transfers, every claim where `claimableAmount < 10` at `feeInBPS = 1000` will permanently revert until the fee is lowered or the claimable amount grows, temporarily freezing those users' funds.

### Finding Description
In `MerkleDistributor.claim()`, the fee and transfer logic is:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);
IERC20(token).safeTransfer(protocolTreasury, fee);   // ← no `if (fee > 0)` guard
``` [1](#0-0) 

When `feeInBPS = 1000` (the allowed maximum) and `claimableAmount ∈ [1, 9]`, integer division truncates `fee` to `0`. The unconditional `safeTransfer(protocolTreasury, 0)` then reverts on any ERC20 token that disallows zero-value transfers. Because the user transfer at line 141 executes first, the whole transaction is atomic — the revert rolls back both transfers and the state update, leaving the user unable to claim.

The sibling contracts in the same codebase explicitly guard this call:

```solidity
// KernelMerkleDistributor.sol & KernelTop100MerkleDistributor.sol
if (fee > 0) {
    kernel.safeTransfer(protocolTreasury, fee);
}
``` [2](#0-1) 

The missing guard in `MerkleDistributor.sol` is the internal root cause; the token's zero-transfer behavior is the trigger, not the sole cause.

### Impact Explanation
Users whose incremental `claimableAmount` is in `[1, 9]` (at `feeInBPS = 1000`) cannot execute `claim()`. Their funds are locked in the contract until either `feeInBPS` is reduced by the owner or their cumulative claimable amount grows to ≥ 10. This matches **Medium — Temporary freezing of funds**.

### Likelihood Explanation
- `feeInBPS = 1000` is a valid, owner-settable value explicitly bounded by `MAX_FEE_IN_BPS`.
- The contract is generic (`setToken` allows any ERC20); several deployed ERC20s revert on zero-value transfers (e.g., LEND, BNB, and others).
- No attacker action is required — the condition arises from normal protocol configuration combined with small claimable balances.

### Recommendation
Add the same `if (fee > 0)` guard used in the sibling contracts:

```solidity
if (fee > 0) {
    IERC20(token).safeTransfer(protocolTreasury, fee);
}
``` [3](#0-2) 

### Proof of Concept
1. Deploy a token that reverts on `transfer(to, 0)`.
2. Initialize `MerkleDistributor` with `feeInBPS = 1000`.
3. Create a merkle leaf for a user with `cumulativeAmount = 5`.
4. Call `claim(index, user, 5, proof)`.
5. Observe revert at `safeTransfer(protocolTreasury, 0)`.
6. Reduce `feeInBPS` to `0` and repeat — `claim()` succeeds.
7. Alternatively, set `cumulativeAmount = 10` — `fee = 1 > 0`, `claim()` succeeds.
