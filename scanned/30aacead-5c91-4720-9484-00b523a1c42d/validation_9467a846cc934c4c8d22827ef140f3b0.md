### Title
`PUSH0` Opcode Incompatibility with Linea Due to Solidity 0.8.27 + Cancun EVM Target - (File: `contracts/bridges/LineaMessenger.sol`, `contracts/L2/RsETHTokenWrapper.sol`, `contracts/pools/RSETHPoolV2ExternalBridge.sol`, et al.)

---

### Summary

Every production contract in this repository is compiled with `pragma solidity 0.8.27` and the project's `foundry.toml` explicitly sets `evm_version = "cancun"`. Solidity ≥ 0.8.20 compiled against the Shanghai or later EVM target emits the `PUSH0` opcode (EIP-3855). Linea does not support `PUSH0`, yet the protocol has live deployments on Linea and continues to deploy and upgrade contracts there. Any future deployment or proxy-implementation upgrade on Linea using the current build configuration will produce bytecode that Linea's EVM cannot execute.

---

### Finding Description

**Root cause — Solidity version + EVM target:**

All in-scope production contracts declare:

```solidity
pragma solidity 0.8.27;
``` [1](#0-0) [2](#0-1) [3](#0-2) 

The project's build configuration locks the EVM target to Cancun — a version that explicitly includes `PUSH0`:

```toml
evm_version = "cancun"   # See https://www.evmdiff.com/features?name=PUSH0&kind=opcode
solc_version = "0.8.27"
``` [4](#0-3) 

The inline comment itself references the `PUSH0` opcode page, confirming the developers are aware of the opcode but have not restricted the EVM target for Linea-bound contracts.

**Linea deployment confirmed:**

The README lists multiple live Linea deployments, all of which are upgradeable proxies whose implementation contracts must be redeployed on every upgrade:

- `RsETHTokenWrapper` — `0xD2671165570f41BBB3B0097893300b6EB6101E6C`
- `RSETHPoolV2ExternalBridge` — `0x057297e44A3364139EDCF3e1594d6917eD7688c2`
- `LineaMessenger` — `0x838686d23521435B528F68Cde6404C07ae007299`
- `HashStorage` — `0x02591eB906282Ef44135b4793f7adD0A14d0C618`
- `AGETHRateReceiver`, `AGETHTokenWrapper`, `AGETHPoolV3` [5](#0-4) 

**Why upgrades are the critical path:**

`RsETHTokenWrapper` and `RSETHPoolV2ExternalBridge` are UUPS/transparent upgradeable proxies. Every upgrade requires deploying a new implementation contract. If that implementation bytecode contains `PUSH0` and Linea rejects it, the upgrade transaction reverts, leaving the proxy permanently stuck on the old implementation. Users' funds held in `RSETHPoolV2ExternalBridge` (ETH deposited for rsETH minting) and `RsETHTokenWrapper` (wrapped rsETH tokens) become inaccessible until a compatible implementation can be deployed. [6](#0-5) [7](#0-6) 

---

### Impact Explanation

**Medium — Temporary freezing of funds.**

If Linea's EVM rejects `PUSH0`-containing bytecode at deployment time, any upgrade to the Linea proxy contracts will revert. ETH deposited by users into `RSETHPoolV2ExternalBridge` and rsETH tokens held in `RsETHTokenWrapper` cannot be migrated or recovered until a `PUSH0`-free implementation is compiled and deployed. The freeze is temporary only if the team can quickly recompile with `evm_version = "paris"` or `"london"` and redeploy; until then, user funds are locked. [8](#0-7) 

---

### Likelihood Explanation

The `foundry.toml` default profile — used for all production builds — sets `evm_version = "cancun"` unconditionally. There is no per-chain profile that overrides this for Linea. Every `forge build` or `forge script` invocation targeting Linea will produce Cancun bytecode. The next upgrade cycle for any Linea contract triggers the issue without any attacker involvement. [9](#0-8) 

---

### Recommendation

Add a dedicated Foundry profile for Linea deployments that pins the EVM version to `paris` (the last version before `PUSH0`):

```toml
[profile.linea]
evm_version = "paris"
solc_version = "0.8.27"
optimizer = true
optimizer_runs = 4000
```

Use this profile (`FOUNDRY_PROFILE=linea forge script ...`) for all Linea deployments and upgrades. Alternatively, constrain the Solidity pragma to `^0.8.19` for contracts intended for Linea, or wait until Linea officially supports the Cancun EVM and verify before deploying.

---

### Proof of Concept

1. Run `forge build` with the default profile (no override): the compiler emits Cancun bytecode for all contracts including `LineaMessenger`, `RsETHTokenWrapper`, and `RSETHPoolV2ExternalBridge`.
2. Attempt to deploy the resulting bytecode to Linea mainnet (or a Linea testnet that mirrors mainnet EVM support).
3. The deployment transaction reverts because Linea's EVM does not recognise the `PUSH0` (0x5f) opcode.
4. For existing proxy contracts, attempt an upgrade (`upgradeTo` / `upgradeToAndCall`) pointing to the newly compiled implementation — the inner `CREATE` call reverts for the same reason, leaving the proxy on its old implementation and all user funds inaccessible until a compatible build is produced.
