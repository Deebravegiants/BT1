### Title
Owner Can Permanently Freeze Unclaimed Yield via `setToken` Mid-Distribution — (`contracts/utils/MerkleDistributor/MerkleDistributor.sol`)

---

### Summary

`MerkleDistributor.setToken` can be called by the owner at any time with no guard against active distributions. Calling it while users still have unclaimed yield permanently strands the old token balance in the contract, because `claim()` transfers `IERC20(token)` — the **current** state variable — and there is no sweep/rescue function to recover the old token.

---

### Finding Description

`setToken` unconditionally overwrites the `token` state variable: [1](#0-0) 

There are no checks that:
- the contract's balance of the **old** token is zero, or
- the current distribution round is complete (all users have claimed).

`claim()` then transfers using the **current** `token` address: [2](#0-1) 

After `setToken(newToken)`, the contract holds zero `newToken`, so every subsequent `safeTransfer` reverts. The old token balance is permanently locked because the contract exposes no `sweep`, `rescue`, or `recoverToken` function anywhere in its interface. [3](#0-2) 

The `initialize` comment even acknowledges `setToken` is expected to be called post-deployment ("token can be set later"), making this a foreseeable operational action, not an exotic attack: [4](#0-3) 

---

### Impact Explanation

All users who have not yet claimed when `setToken` is called lose their yield permanently. Their merkle proofs remain valid, `isClaimed` returns false, but every `claim()` call reverts on `safeTransfer` because the contract holds none of the new token. The old token balance is irrecoverable. This matches **Medium — Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

The owner calling `setToken` is a routine admin operation (e.g., migrating to a new token version or correcting a misconfigured address). No malicious intent is required — a well-meaning admin can trigger this accidentally. The missing safeguard (balance check or distribution-complete check) makes this a design flaw, not a "compromise" scenario.

---

### Recommendation

Add a guard in `setToken` that prevents the token from being changed while the contract holds a non-zero balance of the current token:

```solidity
function setToken(address _token) external onlyOwner {
    if (_token == address(0)) revert ZeroValueProvided();
    require(
        IERC20(token).balanceOf(address(this)) == 0,
        "MerkleDistributor: old token balance not zero"
    );
    token = _token;
    emit TokenUpdated(_token);
}
```

Alternatively, add a `recoverToken(address, uint256)` rescue function so stranded balances can always be recovered.

---

### Proof of Concept

```solidity
// 1. Deploy MerkleDistributor with tokenA, fund with 1000e18 tokenA
// 2. setMerkleRoot(...) — distribution begins
// 3. Alice claims successfully (tokenA transferred)
// 4. Owner calls setToken(tokenB) — tokenB balance in contract = 0
// 5. Bob calls claim() with valid merkle proof
//    → IERC20(tokenB).safeTransfer(bob, amount) reverts (insufficient balance)
// 6. tokenA balance remains locked forever — no sweep function exists
// assert: bob.claim() reverts
// assert: tokenA.balanceOf(distributor) > 0  (stranded)
// assert: no function exists to recover tokenA
```
