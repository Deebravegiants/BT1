### Title
Residual LST Funds Permanently Frozen in `LRTWithdrawalManager` After Asset Removal — (`contracts/LRTWithdrawalManager.sol` / `contracts/utils/LRTConfigRoleChecker.sol`)

---

### Summary

`sweepRemainingAssets` is the only function capable of recovering residual LST balances from `LRTWithdrawalManager`. It is gated by `onlySupportedAsset(asset)`, which hard-reverts when the asset is no longer in `LRTConfig.isSupportedAsset`. Because `LRTConfig.removeSupportedAsset` only checks the deposit pool's accounting and not the `LRTWithdrawalManager` balance, an admin can legitimately remove an asset while a residual balance remains in the withdrawal manager, permanently freezing those funds with no recovery path.

---

### Finding Description

**Step 1 — Residual balance accumulates in `LRTWithdrawalManager`.**

After `unlockQueue` is called, assets are pulled from the unstaking vault and held in `LRTWithdrawalManager`: [1](#0-0) 

After all `completeWithdrawal` calls transfer the exact `expectedAssetAmount` to users, a residual balance can remain due to rounding, direct ERC20 transfers, or slashing-adjusted payouts. The protocol explicitly acknowledges this possibility by providing `sweepRemainingAssets`.

**Step 2 — `removeSupportedAsset` does not check `LRTWithdrawalManager` balance.**

The only guard in `removeSupportedAsset` is: [2](#0-1) 

`getTotalAssetDeposits` accounts for assets in the deposit pool and node delegators — not assets already transferred to `LRTWithdrawalManager`. So the admin can pass this check and remove the asset even when a residual balance sits in the withdrawal manager.

**Step 3 — `sweepRemainingAssets` permanently reverts.** [3](#0-2) 

The `onlySupportedAsset(asset)` modifier: [4](#0-3) 

Once `isSupportedAsset[asset]` is deleted (line 86 of `LRTConfig.sol`), every call to `sweepRemainingAssets` reverts with `AssetNotSupported`. There is no upgrade path or bypass.

**Step 4 — No alternative drain path exists.**

Every other outbound transfer function in `LRTWithdrawalManager` is either:
- Gated by `onlySupportedAsset` (`instantWithdrawal` at line 220), or
- Requires an active pending withdrawal request (`completeWithdrawal` reverts with `NoWithdrawalRequests` when the queue is empty), or
- Scoped exclusively to Aave ETH (`emergencyWithdrawFromAave`, `_collectInterestToTreasury`). [5](#0-4) [6](#0-5) 

---

### Impact Explanation

Any residual LST balance held by `LRTWithdrawalManager` at the time an asset is removed from `LRTConfig` becomes permanently unrecoverable. There is no admin escape hatch, no upgrade-triggered drain, and no fallback sweep. This satisfies **Critical — Permanent freezing of funds**.

---

### Likelihood Explanation

- `sweepRemainingAssets` exists precisely because the protocol expects residual balances to occur.
- `removeSupportedAsset` is a routine admin operation when deprecating an asset (e.g., sfrxETH, as evidenced by `initialize3`).
- The guard in `removeSupportedAsset` only inspects the deposit pool; a diligent admin has no on-chain signal that `LRTWithdrawalManager` still holds a balance for the asset being removed.
- No attacker action is required; the freeze is triggered by a normal, well-intentioned admin workflow.

Likelihood: **Low-Medium** (requires the coincidence of a residual balance and an asset removal, but both are expected protocol events).

---

### Recommendation

1. **Remove `onlySupportedAsset` from `sweepRemainingAssets`** and replace it with an explicit non-zero balance check. The function already verifies `balance > 0`; asset support is irrelevant to a treasury sweep.
2. **Add a balance check in `removeSupportedAsset`**: before deleting `isSupportedAsset[asset]`, assert that `LRTWithdrawalManager`'s balance of the asset is zero (or below `maxNegligibleAmount`).

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity 0.8.27;

// Pseudocode unit test (Foundry-style)
function test_permanentFreeze() public {
    address asset = address(stETH);

    // 1. User initiates and operator unlocks a withdrawal, leaving residual in WithdrawalManager
    vm.prank(user);
    withdrawalManager.initiateWithdrawal(asset, rsETHAmount, "");
    vm.roll(block.number + withdrawalDelayBlocks + 1);
    vm.prank(operator);
    withdrawalManager.unlockQueue(asset, 1, minPrice, minRsETH, maxPrice, maxRsETH);
    vm.prank(user);
    withdrawalManager.completeWithdrawal(asset, "");

    // Simulate residual balance (e.g., direct transfer or rounding)
    deal(asset, address(withdrawalManager), 1e15);

    // 2. Admin removes the asset (deposit pool is empty, guard passes)
    vm.prank(admin);
    lrtConfig.removeSupportedAsset(asset, tokenIndex);

    // 3. sweepRemainingAssets permanently reverts
    vm.prank(manager);
    vm.expectRevert(ILRTConfig.AssetNotSupported.selector);
    withdrawalManager.sweepRemainingAssets(asset);

    // 4. Confirm no alternative drain path
    // completeWithdrawal reverts — no pending requests
    vm.prank(user);
    vm.expectRevert();
    withdrawalManager.completeWithdrawal(asset, "");

    // instantWithdrawal reverts — onlySupportedAsset
    vm.prank(user);
    vm.expectRevert(ILRTConfig.AssetNotSupported.selector);
    withdrawalManager.instantWithdrawal(asset, rsETHAmount, "");

    // Residual 1e15 stETH is permanently frozen.
    assertEq(IERC20(asset).balanceOf(address(withdrawalManager)), 1e15);
}
```
