`addNewSupportedAsset` Does Not Enforce a Price Oracle, Causing `updateRSETHPrice()` to Permanently Revert Until Remediated - (File: contracts/LRTConfig.sol)

---

### Summary

`LRTConfig.addNewSupportedAsset` adds an asset to `supportedAssetList` without requiring a corresponding price oracle to be registered in `LRTOracle`. Because `LRTOracle._getTotalEthInProtocol()` unconditionally calls `getAssetPrice(asset)` for every entry in `supportedAssetList`, and `getAssetPrice` reverts when no oracle is configured, the entire `updateRSETHPrice()` function becomes uncallable the moment any oracle-less asset is present. This freezes rsETH yield accrual for all holders until the oracle is separately configured.

---

### Finding Description

`LRTConfig.addNewSupportedAsset` (and its internal helper `_addNewSupportedAsset`) registers an asset as supported by setting `isSupportedAsset[asset] = true` and appending it to `supportedAssetList`, but makes no call to `LRTOracle.updatePriceOracleFor` and imposes no requirement that a price oracle exists for the new asset. [1](#0-0) 

`LRTOracle._getTotalEthInProtocol()` iterates over the full `supportedAssetList` returned by `lrtConfig.getSupportedAssetList()` and calls `getAssetPrice(asset)` for every element: [2](#0-1) 

`getAssetPrice` is guarded by the `onlySupportedOracle` modifier, which reverts with `AssetOracleNotSupported` when `assetPriceOracle[asset] == address(0)`: [3](#0-2) 

Because `_getTotalEthInProtocol` is called inside `_updateRsETHPrice`, which is called by the public `updateRSETHPrice()`, any supported asset without a registered oracle causes every invocation of `updateRSETHPrice()` to revert. The stored `rsETHPrice` is frozen at its last value, and no yield can accrue to rsETH holders until the oracle is configured in a separate transaction. [4](#0-3) 

---

### Impact Explanation

**Medium — Temporary freezing of unclaimed yield.**

While the asset-without-oracle condition persists, `updateRSETHPrice()` is completely uncallable. The rsETH exchange rate is frozen, so staking rewards that have accrued in EigenLayer strategies are not reflected in the rsETH price. All rsETH holders are denied yield accrual for the duration of the misconfiguration window. Additionally, the fee-minting path inside `_updateRsETHPrice` is also blocked, so the protocol treasury accrues no fees during this period. [5](#0-4) 

---

### Likelihood Explanation

The `TIME_LOCK_ROLE` holder follows a natural two-step workflow: add the asset first, then configure its oracle. During the window between those two transactions — which may span multiple blocks or even days if the oracle setup is delayed or forgotten — `updateRSETHPrice()` is broken for the entire protocol. No attacker action is required; the misconfiguration arises from the ordinary governance process. The absence of any enforcement in `addNewSupportedAsset` makes this a realistic operational risk. [6](#0-5) 

---

### Recommendation

Enforce oracle registration atomically with asset support. Either:

1. Require the caller to supply a `priceOracle` address in `addNewSupportedAsset` and call `LRTOracle.updatePriceOracleFor(asset, priceOracle)` inside the same transaction, or
2. Add a guard in `_getTotalEthInProtocol` that skips assets whose `assetPriceOracle` is `address(0)` (treating their contribution as zero), consistent with how `NodeDelegator.depositAssetIntoStrategy` already skips assets with no strategy set.

Option 1 is preferred because it prevents silent misconfiguration at the source.

---

### Proof of Concept

1. `TIME_LOCK_ROLE` calls `LRTConfig.addNewSupportedAsset(newLST, depositLimit)`.
   - `newLST` is appended to `supportedAssetList`; `LRTOracle.assetPriceOracle[newLST]` remains `address(0)`.

2. Any account (including a keeper bot or any user) calls `LRTOracle.updateRSETHPrice()`.

3. Execution enters `_updateRsETHPrice()` → `_getTotalEthInProtocol()`.

4. The loop reaches `newLST` and calls `getAssetPrice(newLST)`.

5. The `onlySupportedOracle` modifier fires: `assetPriceOracle[newLST] == address(0)` → revert `AssetOracleNotSupported`.

6. `updateRSETHPrice()` reverts. `rsETHPrice` remains stale. Yield accrual is frozen for all rsETH holders until `updatePriceOracleFor(newLST, oracle)` is called in a separate transaction. [7](#0-6) [8](#0-7)

    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
