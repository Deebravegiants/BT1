### Title
No Slippage Protection in L2 Pool `deposit()` Functions - (File: contracts/pools/RSETHPoolV3.sol, RSETHPoolV3ExternalBridge.sol, RSETHPoolV3WithNativeChainBridge.sol, RSETHPool.sol, RSETHPoolNoWrapper.sol, RSETHPoolV2.sol, RSETHPoolV2NBA.sol, RSETHPoolV2ExternalBridge.sol)

---

### Summary

Every user-facing `deposit()` function across all L2 RSETH pool contracts swaps ETH or a supported token for rsETH/wrsETH using an oracle-derived rate, but accepts no `minRsETHAmountExpected` parameter. Users have zero slippage protection: if the oracle rate moves adversely between transaction submission and execution, the user receives fewer rsETH than they anticipated with no ability to revert.

---

### Finding Description

The L1 `LRTDepositPool` correctly exposes slippage protection to depositors via `minRSETHAmountExpected`:

```solidity
// LRTDepositPool.sol
function depositETH(uint256 minRSETHAmountExpected, string calldata referralId) external payable ...
function depositAsset(address asset, uint256 depositAmount, uint256 minRSETHAmountExpected, ...) external ...
``` [1](#0-0) [2](#0-1) 

The check is enforced in `_beforeDeposit`:

```solidity
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [3](#0-2) 

By contrast, every L2 pool `deposit()` function omits this parameter entirely. The pattern is identical across all pool variants:

```solidity
// RSETHPoolV3.sol – ETH deposit
function deposit(string memory referralId) external payable nonReentrant whenNotPaused limitDailyMint(...) {
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);   // ← no minimum check
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
``` [4](#0-3) 

```solidity
// RSETHPoolV3.sol – token deposit
function deposit(address token, uint256 amount, string memory referralId) external ... {
    IERC20(token).safeTransferFrom(msg.sender, address(this), amount);
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
    feeEarnedInToken[token] += fee;
    wrsETH.mint(msg.sender, rsETHAmount);   // ← no minimum check
}
``` [5](#0-4) 

The same pattern is present in:

- `RSETHPoolV3ExternalBridge.deposit()` (ETH and token) [6](#0-5) 

- `RSETHPoolV3WithNativeChainBridge.deposit()` (ETH and token) [7](#0-6) 

- `RSETHPool.deposit()` (ETH and token) [8](#0-7) 

- `RSETHPoolNoWrapper.deposit()` (ETH and token) [9](#0-8) 

- `RSETHPoolV2.deposit()` (ETH) [10](#0-9) 

- `RSETHPoolV2NBA.deposit()` (ETH) [11](#0-10) 

- `RSETHPoolV2ExternalBridge.deposit()` (ETH) [12](#0-11) 

The rsETH amount is computed from `viewSwapRsETHAmountAndFee`, which reads the live oracle rate at execution time:

```solidity
uint256 rsETHToETHrate = getRate();   // live oracle read
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [13](#0-12) 

---

### Impact Explanation

**Impact: Low** — Contract fails to deliver promised returns, but does not lose value.

A user who previews the rate off-chain (e.g., via `viewSwapRsETHAmountAndFee`) and submits a transaction may receive materially fewer rsETH than expected if the oracle rate increases (rsETH appreciates in ETH terms) between submission and execution. The user's ETH/tokens are fully consumed but the rsETH minted is less than the user intended to accept. There is no mechanism to revert the transaction at an unfavorable rate. The deposited ETH/tokens are not stolen — they remain in the pool — but the user receives a worse exchange than they agreed to.

---

### Likelihood Explanation

**Likelihood: Medium.**

The oracle rate (`rsETHToETHrate`) is a live read from an external oracle contract at execution time. On L2 chains with fast block times and public mempools, the rate can legitimately shift between when a user signs a transaction and when it is included. Additionally, on chains where the oracle is updated by a keeper, a keeper update landing in the same block just before a user's deposit will silently worsen the user's output. No privileged access is required; any user depositing into any of the affected pools is exposed.

---

### Recommendation

Add a `minRsETHAmountExpected` parameter to every `deposit()` overload in all L2 pool contracts, mirroring the pattern already used in `LRTDepositPool`:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountExpected)
    external payable nonReentrant whenNotPaused ...
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    wrsETH.mint(msg.sender, rsETHAmount);
}
```

---

### Proof of Concept

1. Alice calls `viewSwapRsETHAmountAndFee(1 ether)` off-chain and sees she will receive `X` wrsETH at the current oracle rate.
2. Alice submits `RSETHPoolV3.deposit{value: 1 ether}("ref")`.
3. Before Alice's transaction is included, the oracle rate updates (rsETH appreciates), so `rsETHToETHrate` increases.
4. Alice's transaction executes: `rsETHAmount = (1 ether - fee) * 1e18 / newHigherRate` → Alice receives fewer wrsETH than `X`.
5. Alice has no recourse; the transaction succeeded and her ETH is in the pool.
6. The same scenario applies to token deposits across all affected pool contracts.

    function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
    }
```
