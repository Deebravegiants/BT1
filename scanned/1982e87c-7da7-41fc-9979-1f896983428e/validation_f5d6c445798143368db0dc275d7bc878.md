### Title
No-Staleness-Check on Chainlink Price Feed Enables Inflated Instant Withdrawals - (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` consumes Chainlink `latestRoundData()` without validating the `updatedAt` timestamp or `answeredInRound` field. A stale asset price flows directly into `LRTWithdrawalManager.instantWithdrawal()` via `getExpectedAssetAmount()`, which has no operator-supplied price bounds. When the stored `rsETHPrice` reflects a normal valuation but the live Chainlink feed for a withdrawal asset is stale-low, any user can drain more of that asset than their rsETH entitles them to.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` and silently discards all validity fields:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol  line 52
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

No check is made on `updatedAt` (staleness), `answeredInRound >= roundId` (round completeness), or `price > 0` (valid answer). [1](#0-0) 

This price is consumed in two places that determine how many assets a user receives:

**Path 1 – `instantWithdrawal()` (fully user-controlled, no price bounds):**

`instantWithdrawal()` calls `getExpectedAssetAmount()` at execution time and immediately transfers assets to the caller. There are no `minimumAssetPrice` / `maximumAssetPrice` guard parameters here, unlike `unlockQueue()`. [2](#0-1) 

`getExpectedAssetAmount()` computes:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
``` [3](#0-2) 

`lrtOracle.getAssetPrice(asset)` delegates to `ChainlinkPriceOracle.getAssetPrice()`, which returns the unvalidated stale price. [4](#0-3) 

**Path 2 – `initiateWithdrawal()` (locks in stale-price amount):**

`initiateWithdrawal()` also calls `getExpectedAssetAmount()` at request time and stores the result as `expectedAssetAmount`. [5](#0-4) 

This stored amount is what `_processWithdrawalCompletion()` transfers to the user at completion. [6](#0-5) 

**Divergence between `rsETHPrice` and live `assetPrice`:**

`rsETHPrice` is a stored value updated by calling `updateRSETHPrice()` (public, but a separate transaction). [7](#0-6) 

If `rsETHPrice` was last updated when all prices were normal, but the Chainlink feed for a specific withdrawal asset subsequently goes stale at a lower value, the numerator (`rsETHPrice`) remains correct while the denominator (`assetPrice`) is artificially depressed — inflating `underlyingToReceive`.

---

### Impact Explanation

**Critical – Direct theft of user funds.**

When a Chainlink feed for a supported LST asset returns a stale price lower than the true market price, `getExpectedAssetAmount()` returns an inflated asset amount. Any holder of rsETH can call `instantWithdrawal()` and receive more of that LST than their rsETH is worth, draining the `LRTUnstakingVault` at the expense of other depositors. Because `instantWithdrawal()` burns rsETH and transfers assets atomically with no price-bound parameters, there is no on-chain circuit breaker for this path.

---

### Likelihood Explanation

**Medium.**

Chainlink LST/ETH feeds (e.g., stETH/ETH, cbETH/ETH) have heartbeats of up to 24 hours and deviation thresholds of 0.5–1%. During periods of low volatility the feed can remain at a stale value for many hours. Network congestion or a Chainlink node outage can extend this further. No special privilege or front-running is required; any rsETH holder can observe the stale feed on-chain and call `instantWithdrawal()` immediately.

---

### Recommendation

Add staleness and validity checks inside `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound)
    = priceFeed.latestRoundData();

require(price > 0,                        "Invalid price");
require(answeredInRound >= roundId,        "Stale round");
require(block.timestamp - updatedAt <= MAX_STALENESS, "Stale price");
```

`MAX_STALENESS` should be set per feed based on its documented heartbeat. Additionally, add `minimumAssetPrice` / `maximumAssetPrice` slippage parameters to `instantWithdrawal()` mirroring the guards already present in `unlockQueue()`.

---

### Proof of Concept

1. Chainlink stETH/ETH feed last updated 20 hours ago at `0.9990e18`; true price is now `1.0010e18` (feed is stale-low).
2. `rsETHPrice` was updated 1 hour ago at `1.0500e18` (correct at that time).
3. Attacker holds `100e18` rsETH and calls:
   ```solidity
   instantWithdrawal(stETH, 100e18, "");
   ```
4. `getExpectedAssetAmount(stETH, 100e18)`:
   ```
   = 100e18 * 1.0500e18 / 0.9990e18
   ≈ 105.105e18 stETH
   ```
   Fair value at true price: `100e18 * 1.0500e18 / 1.0010e18 ≈ 104.895e18 stETH`.
5. Attacker receives `≈ 0.21 stETH` extra per 100 rsETH. With a large enough position or repeated calls across multiple stale feeds, this drains the vault at the expense of other depositors.
6. No admin action, no front-running, no oracle operator compromise required — only a naturally occurring stale Chainlink feed.
