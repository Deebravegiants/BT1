### Title
Beacon-Chain Slash on Unverified Validator Inflates `getEffectivePodShares()`, Overstating TVL and Diluting New Depositors — (`contracts/NodeDelegator.sol`)

---

### Summary

`stakedButUnverifiedNativeETH` is incremented by exactly 32 ETH on every `stake32Eth()` call and is only decremented inside `verifyWithdrawalCredentials()`. It is never adjusted for beacon-chain slashing. Because `getEffectivePodShares()` sums this counter with the EigenLayer `withdrawableShare`, a slashed-but-unverified validator causes the protocol to permanently overcount ETH until credentials are verified, inflating the rsETH price and causing new depositors to receive fewer rsETH tokens than the ETH they contribute actually warrants.

---

### Finding Description

**Staking path** — `stake32Eth()` increments `stakedButUnverifiedNativeETH` by 32 ETH and sends 32 ETH to the beacon deposit contract: [1](#0-0) 

**Credential verification path** — `verifyWithdrawalCredentials()` is the *only* place `stakedButUnverifiedNativeETH` is decremented: [2](#0-1) 

**TVL accounting** — `getEffectivePodShares()` adds the unverified counter to the EigenLayer withdrawable share: [3](#0-2) 

For a validator whose credentials have never been verified, EigenLayer holds zero deposit shares for that validator, so `withdrawableShare = 0`. The entire 32 ETH contribution to `getEffectivePodShares()` comes from `stakedButUnverifiedNativeETH`.

**Price propagation** — `getETHDistributionData()` calls `getEffectivePodShares()` for every NDC: [4](#0-3) 

`_getTotalEthInProtocol()` calls `getTotalAssetDeposits()` → `getETHDistributionData()`, and `_updateRsETHPrice()` divides that total by `rsethSupply` to produce the stored `rsETHPrice`: [5](#0-4) 

**Minting** — every deposit divides by the stored `rsETHPrice`: [6](#0-5) 

**The gap** — if a beacon-chain slash reduces the validator's actual balance from 32 ETH to, say, 30 ETH *before* `verifyWithdrawalCredentials()` is called:

| Quantity | Expected | Actual (post-slash, pre-verify) |
|---|---|---|
| `stakedButUnverifiedNativeETH` | 30 ETH | 32 ETH |
| `withdrawableShare` | 0 | 0 |
| `getEffectivePodShares()` | 30 ETH | **32 ETH** |

`updateRSETHPrice()` is a public, permissionless function. Any caller can invoke it while the 2 ETH discrepancy exists, locking in an inflated `rsETHPrice`. New depositors who mint rsETH against this inflated price receive fewer tokens than their ETH warrants. When the slash is eventually realized (after `verifyWithdrawalCredentials()` + checkpoint), the price corrects downward, and those depositors bear the loss.

---

### Impact Explanation

New depositors who mint rsETH while `rsETHPrice` is inflated by the unaccounted slash receive fewer rsETH tokens than the ETH they contribute actually entitles them to. When the price corrects, their rsETH is worth less ETH than they deposited. The difference is effectively transferred to pre-existing holders, constituting direct theft of depositor funds. The magnitude scales with the slash amount (minimum ~1 ETH per validator for an initial slash, up to 16 ETH with correlation penalties) and the number of unverified slashed validators.

---

### Likelihood Explanation

- Beacon-chain slashing is a real, recurring event on mainnet.
- There is always a window between `stake32Eth()` and `verifyWithdrawalCredentials()` during which the validator is active on the beacon chain but unverified in EigenLayer.
- `updateRSETHPrice()` is public and is called regularly (by bots, keepers, or any user), so the inflated price will be stored during this window.
- No attacker action is required beyond depositing ETH after the price is inflated; the harm is automatic.

---

### Recommendation

Decrement `stakedButUnverifiedNativeETH` when a checkpoint proves a validator's balance has dropped below 32 ETH, or cap the per-validator contribution to `getEffectivePodShares()` at the proven balance rather than the fixed 32 ETH. Alternatively, track the actual beacon-chain balance of unverified validators via checkpoint proofs before including them in TVL.

---

### Proof of Concept

```solidity
// Fork test (Holesky or mainnet fork)
// 1. Deploy/use existing NDC; call stake32Eth() for validator V → stakedButUnverifiedNativeETH = 32e18
// 2. Simulate beacon-chain slash: validator V's balance drops to 30 ETH on the beacon chain
//    (verifyWithdrawalCredentials() has NOT been called)
// 3. Call updateRSETHPrice() → rsETHPrice is computed using getEffectivePodShares() = 32e18
//    (inflated by 2e18 vs actual recoverable ETH)
// 4. New depositor calls depositETH(1 ether, ...) → receives fewer rsETH than 1 ETH warrants
// 5. Call verifyWithdrawalCredentials() + startCheckpoint() + verifyCheckpointProofs()
//    → stakedButUnverifiedNativeETH -= 32e18; withdrawableShare += 30e18
// 6. Call updateRSETHPrice() → rsETHPrice drops to reflect actual 30e18
// 7. Assert: depositor's rsETH redeemable value < 1 ETH deposited
//    i.e., rsETHBalance * newRsETHPrice / 1e18 < 1 ether
```
