### Title
ETH Deposit Limit Check Omits Deposit Amount, Allowing Deposit Cap Bypass - (File: contracts/LRTDepositPool.sol)

### Summary
In `LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit`, the ETH branch compares only `totalAssetDeposits` against the deposit limit, omitting the incoming `amount`. The LST branch correctly includes `amount`. This is a direct analog to the reported bug: the wrong variable (or an incomplete expression) is used in the guard condition, allowing ETH deposits to bypass the configured deposit cap.

### Finding Description
`_checkIfDepositAmountExceedesCurrentLimit` applies two different checks depending on asset type:

```solidity
// contracts/LRTDepositPool.sol lines 676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount missing
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct
}
```

For ETH the function returns `true` (block the deposit) only when `totalAssetDeposits` **already exceeds** the limit. It never considers the size of the incoming deposit. For LSTs the check is `totalAssetDeposits + amount > limit`, which correctly blocks any deposit that would push the total over the cap.

Consequences:
1. When `totalAssetDeposits == depositLimit` the ETH check returns `false` (not exceeded), so the deposit proceeds and the total rises to `depositLimit + amount`.
2. When `totalAssetDeposits < depositLimit` but `totalAssetDeposits + amount > depositLimit`, the ETH check still returns `false`, allowing a deposit that overshoots the cap by an arbitrary margin.

The function is called unconditionally from `_beforeDeposit`, which is the sole pre-flight guard for both `depositETH` and `depositAsset`. [1](#0-0) 

### Impact Explanation
The deposit limit is the protocol's primary risk-management control on ETH exposure. Bypassing it allows any depositor to push total ETH holdings arbitrarily above the configured cap, violating the invariant that `getTotalAssetDeposits(ETH) ≤ depositLimitByAsset(ETH)`. Over-exposure beyond the intended cap can lead to the protocol taking on more EigenLayer risk than governance approved, and in an adverse slashing or liquidity event, to insolvency. This maps to **Medium – contract fails to deliver promised returns / deposit cap bypass with potential path to insolvency**. [2](#0-1) 

### Likelihood Explanation
The entry point is the public, permissionless `depositETH` function. No special role or precondition is required beyond having ETH. The condition is triggered whenever `totalAssetDeposits` is at or near the configured limit, which is a normal operational state. Any depositor can exploit this without coordination. [3](#0-2) 

### Recommendation
Add `amount` to the ETH branch, mirroring the LST branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

The ETH/LST split is unnecessary once the expression is unified.

### Proof of Concept
1. Governance sets `depositLimitByAsset(ETH_TOKEN) = 1000 ETH`.
2. Protocol accumulates `totalAssetDeposits(ETH) = 1000 ETH` (limit exactly reached).
3. Attacker calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500e18)` evaluates `1000e18 > 1000e18` → `false` → deposit is not blocked.
5. `_mintRsETH` mints rsETH for the attacker; total ETH in protocol is now 1500 ETH, 50 % above the cap.
6. The same attack works for any deposit size as long as `totalAssetDeposits == depositLimit` at the time of the call. [1](#0-0) [4](#0-3)
