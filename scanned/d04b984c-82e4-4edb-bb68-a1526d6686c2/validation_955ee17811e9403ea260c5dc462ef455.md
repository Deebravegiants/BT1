### Title
Uninitialized `maxNumberOfWithdrawalsPerUser` Blocks All Staker Withdrawals at Deployment - (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary
`KernelDepositPool.initialize()` never sets `maxNumberOfWithdrawalsPerUser`, leaving it at the Solidity default of `0`. Because `initiateWithdrawal` guards with `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser`, and any `uint256` length is `>= 0`, every call to `initiateWithdrawal` reverts with `WithdrawalLimitReached` from the moment the contract is deployed until an admin manually calls `setMaxNumberOfWithdrawalsPerUser`. Users who have already staked KERNEL tokens cannot recover them during this window.

### Finding Description
`KernelDepositPool` tracks the maximum number of concurrent open withdrawals per user in `maxNumberOfWithdrawalsPerUser`:

```solidity
uint256 public maxNumberOfWithdrawalsPerUser;
``` [1](#0-0) 

The `initialize` function sets only `kernelToken` and `rewardsToken`; it never assigns `maxNumberOfWithdrawalsPerUser`:

```solidity
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    ...
    kernelToken = IERC20(_kernelToken);
    rewardsToken = IERC20(_rewardToken);
}
``` [2](#0-1) 

`initiateWithdrawal` enforces the limit before creating any withdrawal record:

```solidity
if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();
``` [3](#0-2) 

With `maxNumberOfWithdrawalsPerUser == 0`, the expression `0 >= 0` is always `true`, so the revert fires unconditionally. No withdrawal record is ever created, and staked KERNEL tokens are locked in the contract.

The setter that would fix this enforces a non-zero value, confirming the intent was always `> 0`:

```solidity
if (_maxNumberOfWithdrawalsPerUser == 0 || _maxNumberOfWithdrawalsPerUser > MAX_WITHDRAWALS_PER_USER) {
    revert InvalidMaxNumberOfWithdrawalsPerUser();
}
``` [4](#0-3) 

The analog to the GMX LIQH-2 bug is direct: in GMX, `processLiquidation` created an order without calling `touch()`, leaving `updatedAtBlock = 0` and making execution impossible. Here, `initialize()` creates the contract without setting `maxNumberOfWithdrawalsPerUser`, leaving it at `0` and making `initiateWithdrawal` impossible to call successfully.

### Impact Explanation
Every user who has staked KERNEL tokens via `stake()` or `stakeFor()` is unable to call `initiateWithdrawal` until an admin manually invokes `setMaxNumberOfWithdrawalsPerUser`. The staked tokens are locked in the contract for the entire duration of this window. This constitutes a **temporary freezing of funds** (Medium severity per the allowed impact scope).

### Likelihood Explanation
The freeze is active from the first block after deployment. Any user who stakes and then attempts to withdraw — a normal, expected user flow — will be immediately affected. No special conditions, attacker knowledge, or race conditions are required. The entry path is fully permissionless: `stake()` is open to any address, and `initiateWithdrawal()` is the natural follow-on action.

### Recommendation
Initialize `maxNumberOfWithdrawalsPerUser` to a safe non-zero default inside `initialize()`, for example:

```solidity
maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER;
```

This mirrors the pattern already used for `withdrawalDelay`, which should also be initialized to a non-zero value in `initialize()` to avoid a separate zero-delay window.

### Proof of Concept
1. Deploy `KernelDepositPool` and call `initialize(admin, kernelToken, rewardToken)`.
2. Confirm `maxNumberOfWithdrawalsPerUser == 0`.
3. Any user calls `stake(1e18)` — succeeds; `balanceOf[user] = 1e18`.
4. Same user calls `initiateWithdrawal(1e18)`.
5. The guard evaluates `userWithdrawalIds[user].length >= maxNumberOfWithdrawalsPerUser` → `0 >= 0` → `true`.
6. Transaction reverts with `WithdrawalLimitReached`.
7. The user's `1e18` KERNEL remains locked in the contract with no path to recovery until an admin calls `setMaxNumberOfWithdrawalsPerUser(N)` where `N > 0`.
