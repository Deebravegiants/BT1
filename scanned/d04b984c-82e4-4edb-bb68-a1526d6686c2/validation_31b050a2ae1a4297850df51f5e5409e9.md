### Title
Fee Rounds to Zero for Small-Decimal Tokens, Enabling Protocol Fee Evasion - (File: contracts/pools/RSETHPool.sol, contracts/pools/RSETHPoolV3.sol, contracts/agETH/AGETHPoolV3.sol)

### Summary
All pool contracts compute the protocol fee using integer division: `fee = amount * feeBps / 10_000`. For tokens with small decimals (e.g., 2–6 decimals), a depositor can choose an `amount` small enough that `amount * feeBps < 10_000`, causing the fee to truncate to zero. The depositor receives the full rsETH/agETH output with no fee paid to the protocol.

### Finding Description
Every `viewSwapRsETHAmountAndFee` and `viewSwapAgETHAmountAndFee` function across the pool suite computes the fee identically:

```solidity
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;
``` [1](#0-0) [2](#0-1) [3](#0-2) 

Solidity integer division truncates toward zero. When `amount * feeBps < 10_000`, `fee` becomes `0` and `amountAfterFee == amount`, so the depositor receives rsETH/agETH calculated on the full deposit with zero fee deducted.

The only guard against a zero deposit is:

```solidity
if (amount == 0) revert InvalidAmount();
``` [4](#0-3) 

There is no minimum-fee check or minimum-amount-relative-to-feeBps check anywhere in the deposit path.

The pools accept arbitrary whitelisted ERC-20 tokens via `addSupportedToken`: [5](#0-4) 

`RSETHPool` additionally stores a per-token fee BPS (`tokenFeeBps[token]`), making the threshold configurable per asset: [6](#0-5) 

### Impact Explanation
When fee rounds to zero, the protocol collects no fee revenue on those deposits. The depositor receives the full rsETH/agETH output as if no fee existed. This is a loss of protocol yield (fee revenue) that belongs to the protocol treasury/fee recipients. Impact: **Low — contract fails to deliver promised returns (fee collection), but no user principal is lost.**

### Likelihood Explanation
The vulnerability is only exploitable when a token with small decimals is whitelisted. The pools are designed to accept arbitrary ERC-20 tokens via admin-controlled `addSupportedToken`. If a 2-decimal token (e.g., EURS) or 6-decimal token (e.g., USDC) is added, the threshold below which fee = 0 is:

- EURS (2 dec), feeBps = 30: `amount ≤ 333` units → `fee = 333 * 30 / 10000 = 0` (i.e., deposits ≤ 3.33 EURS pay zero fee)
- USDC (6 dec), feeBps = 1: `amount ≤ 9999` units → `fee = 0` (i.e., deposits ≤ 0.009999 USDC pay zero fee)

For 18-decimal ETH/LST tokens the threshold is sub-wei and not exploitable in practice. Likelihood is **Low** for the current primary token set, but rises if small-decimal tokens are whitelisted.

### Recommendation
1. Enforce a minimum fee: revert if `fee == 0` when `feeBps > 0`:
   ```solidity
   if (feeBps > 0 && fee == 0) revert FeeTooSmall();
   ```
2. Alternatively, enforce a minimum deposit amount scaled to the token's decimals so that `amount * feeBps >= 10_000` is always satisfied.
3. Restrict supported tokens to those with ≥ 18 decimals, or document and enforce a per-token minimum deposit in the whitelist logic.

### Proof of Concept
Assume EURS (2 decimals) is whitelisted and `feeBps = 30` (0.3%):

1. Attacker calls `deposit(EURS, 333, "ref")` (= 3.33 EURS).
2. `viewSwapRsETHAmountAndFee(333, EURS)` computes:
   - `fee = 333 * 30 / 10_000 = 9990 / 10_000 = 0` (integer truncation)
   - `amountAfterFee = 333 - 0 = 333`
3. Attacker receives rsETH calculated on the full 333 units — zero fee paid.
4. Repeating this across many small deposits drains all fee revenue for that token. [7](#0-6) [8](#0-7)
