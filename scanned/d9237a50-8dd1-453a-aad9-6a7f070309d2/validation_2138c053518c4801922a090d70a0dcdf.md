## Analysis

The attack path is concrete and traceable through unmodified production code. Let me map it precisely.

**Key chain:**

1. `MultiChainRateProvider.updateRate()` — **permissionless** (no role check, only `nonReentrant`) [1](#0-0) 

2. It reads `getLatestRate()` → `ILRTOracle(rsETHPriceOracle).rsETHPrice()` — the **currently stored** `rsETHPrice` in `LRTOracle`. [2](#0-1) 

3. In `LRTOracle._updateRsETHPrice()`, when `newRsETHPrice < highestRsethPrice` but the drop is **within** `pricePercentageLimit`, the code does NOT pause — it emits events and writes the lower price to `rsETHPrice`. Only an out-of-limit drop triggers a pause. [3](#0-2) 

4. `CrossChainRateReceiver.lzReceive()` has **no monotonicity guard** — it blindly stores whatever rate arrives from the authorized provider: [4](#0-3) 

5. `RSETHPoolV2.viewSwapRsETHAmountAndFee()` computes `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate` — a **lower rate yields more wrsETH per ETH**. [5](#0-4) 

---

### Title
Permissionless `updateRate()` with no monotonicity guard in `CrossChainRateReceiver` allows attacker to push a temporarily depressed rsETH price to L2 and extract excess wrsETH — (`contracts/cross-chain/RSETHMultiChainRateProvider.sol` / `contracts/cross-chain/CrossChainRateReceiver.sol`)

### Summary

`MultiChainRateProvider.updateRate()` is callable by anyone. It reads the current `rsETHPrice` from `LRTOracle` and broadcasts it to all L2 `CrossChainRateReceiver` contracts via LayerZero. When a slashing event reduces TVL by an amount within `pricePercentageLimit`, `LRTOracle` writes a lower `rsETHPrice` without pausing. An attacker can immediately call `updateRate()` to push this depressed rate to L2, then deposit ETH on L2 at the artificially low rate, receiving more `wrsETH` than the long-run fair value. When the price recovers, the attacker's excess `wrsETH` dilutes existing holders' yield.

### Finding Description

**Step 1 — Slashing reduces TVL within the limit.**
A validator slashing event reduces `totalETHInProtocol`. `updateRSETHPrice()` (also permissionless) is called. Inside `_updateRsETHPrice()`, `newRsETHPrice < highestRsethPrice` but `diff <= pricePercentageLimit.mulWad(highestRsethPrice)`, so the protocol does **not** pause. It writes the lower price to `rsETHPrice` and emits `RsETHPriceBelowPeak`. [6](#0-5) 

**Step 2 — Attacker calls `updateRate()`.**
`MultiChainRateProvider.updateRate()` has no access control. It reads `rsETHPrice` (now depressed) and sends it to every registered `CrossChainRateReceiver` via LayerZero. [7](#0-6) 

**Step 3 — `CrossChainRateReceiver` stores the lower rate unconditionally.**
`lzReceive()` decodes the payload and writes `rate = _rate` with no check that `_rate >= rate` (previous stored rate). [4](#0-3) 

**Step 4 — Attacker deposits on L2 at the depressed rate.**
`RSETHPoolV2.deposit()` calls `viewSwapRsETHAmountAndFee()`, which divides by the depressed rate, minting more `wrsETH` per ETH than the fair value. The only cap is `dailyMintLimit`, which limits magnitude but does not prevent the exploit. [8](#0-7) 

**Step 5 — Price recovers; attacker profits at existing holders' expense.**
When slashing is resolved (e.g., insurance payout, remaining stake), TVL recovers. The attacker's excess `wrsETH` now represents a larger share of the recovered TVL than the ETH they deposited warranted, diluting the yield that would have accrued to pre-existing holders.

### Impact Explanation

**High — Theft of unclaimed yield.**

Existing `wrsETH` holders are entitled to the full recovery of TVL after a slashing event. By minting excess `wrsETH` at the depressed rate, the attacker captures a portion of that recovery, reducing the per-token value that existing holders receive. The magnitude is bounded by `dailyMintLimit` on L2 and the size of the price dip, but is non-zero and directly extractable.

### Likelihood Explanation

- Slashing events on EigenLayer-restaked validators are a known, realistic occurrence.
- Both `updateRSETHPrice()` and `updateRate()` are permissionless — no privileged access required.
- The attacker only needs to monitor the mempool or oracle state and act within the recovery window.
- No front-running of admin actions is required; the attacker simply calls two public functions in sequence.

### Recommendation

1. **Add a monotonicity guard in `CrossChainRateReceiver.lzReceive()`**: reject any incoming rate that is lower than the currently stored `rate`:
   ```solidity
   require(_rate >= rate, "Rate cannot decrease");
   ``` [9](#0-8) 

2. **Alternatively, add access control to `updateRate()`** in `MultiChainRateProvider` so only a trusted keeper/manager role can push rates cross-chain. [1](#0-0) 

3. **Propagate the L1 pause state to L2**: when `LRTOracle` detects any price decrease (even within the limit), the L2 pool's deposit function should be paused until the price recovers above the previous stored rate.

        uint256 latestRate = getLatestRate();

        rate = latestRate;

        lastUpdated = block.timestamp;

        bytes memory _payload = abi.encode(latestRate);

        uint256 rateReceiversLength = rateReceivers.length;

        for (uint256 i; i < rateReceiversLength;) {
            uint16 dstChainId = uint16(rateReceivers[i]._chainId);

            bytes memory remoteAndLocalAddresses = abi.encodePacked(rateReceivers[i]._contract, address(this));

            (uint256 estimatedFee,) = ILayerZeroEndpoint(layerZeroEndpoint)
                .estimateFees(dstChainId, address(this), _payload, false, bytes(""));

            ILayerZeroEndpoint(layerZeroEndpoint).send{ value: estimatedFee }(
                dstChainId, remoteAndLocalAddresses, _payload, payable(msg.sender), address(0x0), bytes("")
            );

            unchecked {
                ++i;
            }
        }

        emit RateUpdated(rate);
    }
