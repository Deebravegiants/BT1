### Title
Missing Minimum Output Protection in L2 Pool `deposit()` Functions - (`contracts/pools/RSETHPool.sol`, `RSETHPoolNoWrapper.sol`, `RSETHPoolV2ExternalBridge.sol`, `RSETHPoolV3.sol`, `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV3WithNativeChainBridge.sol`)

---

### Summary
All L2 pool `deposit()` functions that swap ETH or supported tokens for wrsETH/rsETH lack a caller-supplied minimum output (`minRsETHAmountExpected`) parameter. The exchange rate is read from an oracle at execution time with no slippage guard. The L1 `LRTDepositPool` correctly enforces this protection; the L2 pools do not.

---

### Finding Description

Every L2 pool variant exposes one or two `deposit()` entry points for unprivileged users. In every case the function:

1. Reads the current rate from `IOracle(rsETHOracle).getRate()` at execution time.
2. Computes `rsETHAmount` from `viewSwapRsETHAmountAndFee()`.
3. Mints or transfers that amount to `msg.sender`.
4. Accepts **no** `minRsETHAmountExpected` argument and performs **no** minimum-output check.

### Impact Explanation

If the oracle rate increases (rsETH appreciates in ETH terms) between the moment a user submits a `deposit()` transaction and the moment it is included in a block, the user receives fewer wrsETH/rsETH than they observed off-chain when constructing the transaction. There is no on-chain mechanism to revert the transaction in this case. The user's ETH or tokens are consumed and they receive a smaller-than-expected rsETH position.

**Impact: Low** — Contract fails to deliver promised returns, but the user does not lose the underlying value (they receive fewer rsETH shares, but those shares represent the correct ETH value at the new rate).

---

### Likelihood Explanation

The rsETH oracle rate is updated by the protocol whenever the underlying EigenLayer position accrues rewards or the LST basket composition changes. These updates happen regularly. Any user whose transaction is pending in the mempool during an oracle update is silently affected. No attacker action is required; the exposure is inherent to normal protocol operation.

**Likelihood: Medium** — Oracle updates are routine; any pending deposit transaction is at risk during an update window.

---

### Recommendation

Add a `uint256 minRsETHAmountExpected` parameter to every L2 pool `deposit()` function and revert if the computed `rsETHAmount` is below that threshold, mirroring the pattern already used in `LRTDepositPool._beforeDeposit()`:

```solidity
if (rsETHAmount < minRsETHAmountExpected) revert MinimumAmountToReceiveNotMet();
```

---

### Proof of Concept

1. User calls `RSETHPoolV3.deposit{value: 1 ether}("ref")` after observing an oracle rate of `1.05e18` (1 ETH → ~0.952 wrsETH).
2. Before the transaction is mined, the protocol updates the oracle to `1.10e18`.
3. The transaction executes: `rsETHAmount = (1e18 - fee) * 1e18 / 1.10e18` — the user receives ~0.909 wrsETH instead of the ~0.952 wrsETH they expected.
4. No revert occurs; the user silently receives ~4.5% fewer tokens than anticipated with no recourse. [10](#0-9)

    function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
    }
