### Title
Unrestricted `sendFunds()` in `FeeReceiver` Allows Anyone to Force Reward Accounting, Enabling Yield Theft via Price Manipulation - (File: contracts/FeeReceiver.sol)

---

### Summary

`FeeReceiver.sendFunds()` carries no access-control modifier. Any external caller can invoke it at will to flush the entire accumulated MEV/execution-layer reward balance into `LRTDepositPool`. Because `LRTOracle.updateRSETHPrice()` is also publicly callable, an attacker can sequence these two calls to artificially inflate the TVL spike seen by the oracle, trigger the `pricePercentageLimit` guard, and deposit rsETH at a stale (under-valued) price — stealing yield from existing rsETH holders.

---

### Finding Description

`FeeReceiver.sendFunds()` is declared `external` with no role check:

```solidity
// contracts/FeeReceiver.sol L53-58
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

The function moves **all** ETH held by the contract into `LRTDepositPool` in a single call. The deposit pool's `receiveFromRewardReceiver()` is itself unrestricted and payable:

```solidity
// contracts/LRTDepositPool.sol L61
function receiveFromRewardReceiver() external payable { }
```

Once the ETH lands in the deposit pool, `address(this).balance` is counted as `ethLyingInDepositPool` inside `getETHDistributionData()`, which feeds `_getTotalEthInProtocol()` in `LRTOracle`:

```solidity
// contracts/LRTDepositPool.sol L480
ethLyingInDepositPool = address(this).balance;
```

`LRTOracle.updateRSETHPrice()` is also publicly callable (`public whenNotPaused`). When it runs, it computes the new price and checks whether the increase exceeds `pricePercentageLimit`:

```solidity
// contracts/LRTOracle.sol L252-266
if (newRsETHPrice > highestRsethPrice) {
    uint256 priceDifference = newRsETHPrice - highestRsethPrice;
    bool isPriceIncreaseOffLimit =
        pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);

    if (isPriceIncreaseOffLimit) {
        if (!IAccessControl(address(lrtConfig)).hasRole(LRTConstants.MANAGER, msg.sender)) {
            revert PriceAboveDailyThreshold();
        }
    }
}
```

If the guard fires, the function reverts and `rsETHPrice` is **not updated** — it remains at the old, lower value. Deposits then use this stale price:

```solidity
// contracts/LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

A lower `rsETHPrice` denominator means the depositor receives **more rsETH per ETH** than the true exchange rate warrants.

---

### Impact Explanation

**High — Theft of unclaimed yield from existing rsETH holders.**

Existing rsETH holders are entitled to the appreciation in rsETH price that accumulated MEV rewards represent. When an attacker forces a large reward dump via `sendFunds()` and then exploits the stale price window to deposit, they capture a disproportionate share of that appreciation. The attacker's rsETH balance is inflated relative to the true TVL, diluting every existing holder's claim on the underlying assets.

---

### Likelihood Explanation

**Medium.**

- `pricePercentageLimit` is a configurable admin parameter; the protocol explicitly exposes `setPricePercentageLimit()` and the oracle code is written to enforce it, making it likely to be set in production.
- MEV/execution-layer rewards accumulate continuously in `FeeReceiver`; over days or weeks the balance can be large enough to push a single-block TVL spike past the daily threshold.
- The attack requires only two public calls (`sendFunds()` + `updateRSETHPrice()`) and a subsequent deposit — no special privileges, no flash loan, no front-running of a specific transaction.

---

### Recommendation

Restrict `sendFunds()` to an authorized role (e.g., `MANAGER` or a dedicated keeper role):

```solidity
function sendFunds() external onlyRole(LRTConstants.MANAGER) {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

Optionally, also restrict `receiveFromRewardReceiver()` in `LRTDepositPool` to only accept calls from the registered `FeeReceiver` address, preventing direct ETH injection from arbitrary callers.

---

### Proof of Concept

```
1. FeeReceiver accumulates 500 ETH in MEV rewards over several weeks.
   Current rsETHPrice = 1.05e18, highestRsethPrice = 1.05e18.
   pricePercentageLimit = 1e16 (1%).

2. Attacker calls FeeReceiver.sendFunds().
   → 500 ETH moves to LRTDepositPool.
   → LRTDepositPool.address(this).balance increases by 500 ETH.

3. Attacker calls LRTOracle.updateRSETHPrice().
   → _getTotalEthInProtocol() returns a value ~500 ETH higher.
   → newRsETHPrice >> 1.05e18 * 1.01 (exceeds 1% threshold).
   → Function reverts with PriceAboveDailyThreshold().
   → rsETHPrice remains at 1.05e18.

4. Attacker calls LRTDepositPool.depositETH{ value: X }(minRSETH, "").
   → getRsETHAmountToMint uses rsETHPrice = 1.05e18 (stale).
   → Attacker receives rsETH priced as if the 500 ETH reward bump never happened.

5. Manager calls LRTOracle.updateRSETHPriceAsManager().
   → rsETHPrice is updated to the true higher value.
   → Attacker's rsETH is now worth more than it should be,
     at the expense of all pre-existing rsETH holders.
