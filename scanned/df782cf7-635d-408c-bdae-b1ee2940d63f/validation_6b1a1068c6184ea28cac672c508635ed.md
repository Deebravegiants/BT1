### Title
`block.number`-Based Velocity Guard Is Ineffective on Optimism, Allowing Uncapped Oracle Price Moves to Reach Pool Swaps — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

### Summary

`PriceVelocityGuardExtension` uses `block.number` to measure elapsed time between oracle price updates and scales the allowed price-change budget by `sqrt(1 + blockDiff)`. On Optimism, every transaction is placed in its own block, so `block.number` increments with transaction throughput rather than wall-clock time. A high-traffic period inflates `blockDiff` far beyond what the admin intended, making the guard arbitrarily permissive and allowing a manipulated oracle mid-price to pass the `beforeSwap` check and reach the pool swap.

### Finding Description

The guard stores the block number of the last price update and computes the allowed squared change as:

```solidity
// PriceVelocityGuardExtension.sol  lines 58, 63, 70
s.lastUpdateBlock = uint64(block.number);
...
uint256 blockDiff = block.number - prevBlock;
...
uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);
``` [1](#0-0) 

The struct stores `lastUpdateBlock` as a `uint64` block number, and `maxChangePerBlockE18` is calibrated by the pool admin in "per-block" units:

```solidity
struct PriceVelocityState {
    uint128 lastMidPriceX64;
    uint64  lastUpdateBlock;
    uint64  maxChangePerBlockE18;
}
``` [2](#0-1) 

On Ethereum mainnet, one block ≈ 12 s, so a `maxChangePerBlockE18` value of, say, 0.5 % per block is a meaningful rate limit. On Optimism, the [L2 sequencer places each transaction in its own block](https://community.optimism.io/docs/developers/build/differences/#block-numbers-and-timestamps), so `block.number` advances with every transaction on the network. Between two swaps separated by only a few seconds of real time, `blockDiff` can be in the hundreds or thousands depending on network load. Because `allowedSq` grows linearly with `blockDiff`, the guard's effective price-change budget becomes unbounded relative to real time, defeating its purpose entirely.

**Exploit path:**

1. Pool admin deploys `PriceVelocityGuardExtension` on Optimism and sets `maxChangePerBlockE18` calibrated to mainnet block times (e.g., 0.5 % / 12 s).
2. During a period of high Optimism network activity (many unrelated transactions), `block.number` advances rapidly.
3. An attacker (or a compromised/stale oracle) submits a swap when `blockDiff` is large. The guard computes `allowedSq = maxChange² × (1 + blockDiff)`, which is now orders of magnitude larger than the admin intended.
4. The `actualSq <= allowedSq` check passes even for a price move that should have been rejected.
5. The manipulated mid-price is accepted by `beforeSwap`, and the pool executes the swap at the bad price, extracting value from LPs. [3](#0-2) 

### Impact Explanation

The velocity guard is the sole on-chain mechanism preventing a sudden large oracle price move from reaching the pool swap. When it is bypassed, the pool executes at an unclamped bid/ask derived from the manipulated mid-price. LPs bear the loss because they are the counterparty to every swap; a sufficiently large price deviation allows a swap that drains one side of the pool at a stale or manipulated rate. This maps directly to the allowed impact category: **bad-price execution — unclamped bid/ask quote reaches a pool swap**.

### Likelihood Explanation

Optimism is an explicitly targeted deployment chain for this protocol. The condition (high `blockDiff` due to network activity) is a normal operating state, not an edge case. No attacker capability beyond submitting a standard swap transaction is required. The only external dependency is that the oracle price has moved (or can be nudged) beyond the admin's intended per-real-time-unit cap, which is a realistic scenario during volatile market conditions.

### Recommendation

Replace `block.number` with `block.timestamp` throughout `PriceVelocityGuardExtension`:

- Rename `lastUpdateBlock` → `lastUpdateTimestamp` (use `uint64`, sufficient for Unix timestamps).
- Rename `maxChangePerBlockE18` → `maxChangePerSecondE18` (or per-millisecond).
- Compute `timeDiff = block.timestamp - prevTimestamp` and scale `allowedSq` by `(1 + timeDiff)`.
- Update `setMaxChangePerBlock` → `setMaxChangePerSecond` and emit the renamed event.

`block.timestamp` on Optimism is the L1-derived timestamp and advances monotonically with real time, making it the correct unit for a rate-of-change guard.

### Proof of Concept

```solidity
// Scenario on Optimism with high network activity
// Admin sets maxChangePerBlockE18 = 5e15 (0.5% per block, assuming 12s blocks)
// Intended max change per second ≈ 0.5% / 12 ≈ 0.042% per second

// Between two swaps 10 seconds apart, 1000 Optimism transactions occur
// blockDiff = 1000
// allowedSq = (5e15)^2 * (1 + 1000) = 25e30 * 1001 ≈ 25e33

// Actual price move of 15% (should be rejected):
// changeE18 = 0.15e18 = 1.5e17
// actualSq  = (1.5e17)^2 = 2.25e34

// 2.25e34 > 2.5e33 → REJECTED on mainnet (correct)
// But with blockDiff=1000: allowedSq = 2.5e33 * 1001 = 2.5025e36
// 2.25e34 < 2.5025e36 → PASSES on Optimism (incorrect)

// The 15% price move is accepted, pool swaps at manipulated price, LPs lose funds.
``` [4](#0-3)
