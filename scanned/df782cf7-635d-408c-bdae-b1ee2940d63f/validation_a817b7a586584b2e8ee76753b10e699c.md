### Title
Missing `address(0)` Validation on `addLiquidity` `owner` Parameter Permanently Locks LP Tokens - (File: metric-core/contracts/libraries/LiquidityLib.sol)

### Summary

`addLiquidity` accepts an arbitrary `owner` address with no zero-address guard. If called with `owner = address(0)`, the caller's tokens are pulled into the pool via callback, shares are minted to the `address(0)` position key, and those tokens are permanently unrecoverable because `removeLiquidity` enforces `msg.sender == owner`, and `msg.sender` can never equal `address(0)` in the EVM.

### Finding Description

`MetricOmmPool.addLiquidity` delegates to `LiquidityLib.addLiquidity`, which mints shares keyed by `_positionBinKey(owner, salt, bin)` = `keccak256(abi.encode(owner, salt, bin))`. [1](#0-0) 

When `owner = address(0)`, the position key resolves to `keccak256(abi.encode(address(0), salt, bin))`. Shares are written to that key and `binTotalShares` is incremented normally. [2](#0-1) 

Tokens are then pulled from `msg.sender` via the modify-liquidity callback and deposited into the pool. [3](#0-2) 

`removeLiquidity` enforces `msg.sender == owner` before burning shares and returning tokens. [4](#0-3) 

Because no EOA or contract can produce a transaction where `msg.sender == address(0)`, the position at `address(0)` can never be unwound. The deposited tokens are permanently locked inside the pool, contributing to `binTotals` and `binState.token(0|1)BalanceScaled` but forever inaccessible to the depositor. [5](#0-4) 

Neither `addLiquidity` in the pool nor `LiquidityLib.addLiquidity` contains any `require(owner != address(0))` guard. [6](#0-5) 

### Impact Explanation

Any tokens deposited under `owner = address(0)` are irrecoverably locked. The pool's `binTotals` and per-bin balances reflect the deposit, so the tokens inflate LP accounting without being claimable by anyone. This is a direct, permanent loss of user principal matching the "direct loss of user principal or owed LP assets" impact gate.

### Likelihood Explanation

The protocol explicitly supports an operator pattern where `msg.sender` (e.g., a router such as `MetricOmmPoolLiquidityAdder`) may differ from `owner`. A router that fails to validate its `owner` argument, or a user who passes a zero address by mistake, triggers the loss with a single transaction. No privileged access is required; any caller of `addLiquidity` can reach this path.

### Recommendation

Add a zero-address check at the top of `LiquidityLib.addLiquidity` (or equivalently in `MetricOmmPool.addLiquidity` before delegation):

```solidity
require(owner != address(0), "owner cannot be zero address");
```

### Proof of Concept

1. Attacker (or mistaken user) calls `pool.addLiquidity(address(0), salt, deltas, callbackData, extensionData)`.
2. `LiquidityLib.addLiquidity` computes `posKey = keccak256(abi.encode(address(0), salt, binIdx))` and mints shares there.
3. The modify-liquidity callback fires on `msg.sender`; tokens are transferred into the pool.
4. `binTotals.scaledToken0 / scaledToken1` increase; `binState.token(0|1)BalanceScaled` increases.
5. Caller attempts `pool.removeLiquidity(address(0), salt, deltas, "")` → reverts `NotPositionOwner` because `msg.sender != address(0)`.
6. Tokens are permanently locked; no recovery path exists.
