### Title
`PriceVelocityGuardExtension` Uses `block.number` for Velocity Enforcement, Which Is Unreliable on L2 Networks — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

---

### Summary

`PriceVelocityGuardExtension.beforeSwap` measures elapsed time between oracle price observations using `block.number`. On Arbitrum, `block.number` returns the most recently synced **L1** block number, which can be stale by many L1 blocks across multiple L2 transactions. This causes two failure modes: (1) legitimate swaps are incorrectly reverted when multiple L2 transactions land in the same L1 block and the oracle price moves normally, and (2) the velocity cap is silently widened when the L1 block number jumps, allowing a price move far larger than the configured per-block limit to pass unchecked.

---

### Finding Description

`PriceVelocityGuardExtension` is a `beforeSwap` hook that caps how fast the oracle mid-price can move between swaps. The allowed change scales as `maxChangePerBlockE18 * sqrt(1 + blockDiff)`, where `blockDiff` is computed as:

```solidity
uint256 blockDiff = block.number - prevBlock;
``` [1](#0-0) 

Both `lastUpdateBlock` and the current block are stored/read as `block.number`:

```solidity
s.lastUpdateBlock = uint64(block.number);
``` [2](#0-1) 

On **Arbitrum**, `block.number` returns the L1 Ethereum block number, which is synced to the sequencer periodically. Multiple L2 transactions can execute within the same L1 block epoch, all seeing the same `block.number`. This produces two broken behaviors:

**Failure mode A — false positive (blockDiff = 0):**
Two consecutive L2 swaps land in the same L1 block epoch. `blockDiff = 0`, so `allowedSq = maxChange² × 1`. If the oracle price has moved legitimately between those two L2 transactions (e.g., a Pyth price update arrived), the guard reverts with `PriceVelocityExceeded` even though real wall-clock time has elapsed and the move is within the intended per-second budget.

**Failure mode B — false negative (inflated blockDiff):**
The L1 block number syncs after a gap (e.g., jumps from 100 to 110). The next swap sees `blockDiff = 10`, so `allowedSq = maxChange² × 11`, permitting ~3.3× the intended single-block price move. An attacker who times a swap immediately after such a sync can push through an oracle price change that the guard was specifically configured to block.

The `setLastMidPrice` admin function also writes `block.number`: [3](#0-2) 

This means even manual admin resets of the baseline are anchored to an unreliable L1 block number.

---

### Impact Explanation

- **Failure mode A** breaks core pool swap functionality: legitimate swaps revert during normal oracle price movement on L2, making the pool unusable for any swap that crosses an oracle update boundary within a single L1 block epoch.
- **Failure mode B** defeats the security invariant the extension exists to enforce: a manipulated or rapidly-moved oracle price that should be blocked by the velocity cap passes through unchecked when `blockDiff` is inflated by an L1 sync jump. LPs are then exposed to a bad-price execution — the exact oracle-manipulation scenario the guard is designed to prevent.

Both impacts are contest-relevant: broken swap functionality and bad-price execution reaching LP balances.

---

### Likelihood Explanation

Arbitrum is a primary deployment target for DeFi protocols. On Arbitrum, L2 blocks are ~0.25 seconds while L1 blocks are ~12 seconds, meaning ~48 L2 transactions share the same `block.number`. Any pool using `PriceVelocityGuardExtension` on Arbitrum will encounter failure mode A on virtually every active trading session. Failure mode B occurs at every L1 block sync boundary, which is a predictable, observable on-chain event that an attacker can monitor and exploit.

---

### Recommendation

Replace `block.number` with `block.timestamp` throughout `PriceVelocityGuardExtension`. Rename `lastUpdateBlock` → `lastUpdateTs`, `maxChangePerBlockE18` → `maxChangePerSecondE18`, and restate the allowed-change formula in seconds:

```solidity
uint256 timeDiff = block.timestamp - prevTs;
uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + timeDiff);
```

`block.timestamp` on Arbitrum and Optimism tracks real wall-clock time reliably and is the correct primitive for any time-based rate-limiting or velocity guard on L2.

---

### Proof of Concept

1. Deploy a Metric OMM pool on Arbitrum with `PriceVelocityGuardExtension` attached, `maxChangePerBlockE18 = X` (e.g., 1% per block = `1e16`).
2. Execute swap A at L2 time T₀; `lastUpdateBlock` is set to L1 block N.
3. A Pyth oracle update arrives 2 seconds later (still within L1 block N). The oracle mid moves by 0.5% — within the intended budget.
4. Execute swap B at L2 time T₀ + 2s. `block.number` is still N, so `blockDiff = 0`, `allowedSq = X²`. The 0.5% move squared exceeds `X²` → `PriceVelocityExceeded` reverts. **Swap is incorrectly blocked.**

For failure mode B:
1. Last swap was at L1 block N. L1 block number then jumps to N+10 (sync event).
2. Execute a swap. `blockDiff = 10`, `allowedSq = X² × 11`. A price move of `X * sqrt(11) ≈ 3.3X` is now permitted — 3.3× the intended single-block cap — bypassing the guard. [4](#0-3)
