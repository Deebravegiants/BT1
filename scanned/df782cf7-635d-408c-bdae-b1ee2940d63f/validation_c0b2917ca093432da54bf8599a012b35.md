Looking at the `OracleValueStopLossExtension` for the direct analog to the Timed uninitialized-timestamp bug.

**Key structural parallel:**
- Timed: `startTime = 0` → `block.timestamp - 0 > duration` → `isTimeEnded()` returns `true` (wrong)
- Metric OMM: `lastDecayTs = 0` → `dt = block.timestamp - 0 = block.timestamp` (≈1.7 × 10⁹) → `_decayed(hwm, rate, dt)` returns `0` (wrong — fully decayed)

**Tracing the root cause:**

`BinHighWatermarks.lastDecayTs` is only written in `_checkAndUpdateWatermarks` (called from `afterSwap`). The admin-facing `executeOracleStopLossHighWatermarks` applies values from `PendingHighWatermarks`, which carries only `token0`, `token1`, `binIdx`, and `executeAfter` — **no `lastDecayTs` field** — so it never initialises `lastDecayTs`. [1](#0-0) 

When the first swap touches a bin whose watermarks were set by admin before any swap:

```
dt = block.timestamp - hwmS.lastDecayTs
   = block.timestamp - 0
   ≈ 1_700_000_000
``` [2](#0-1) 

Then in `_decayed`:

```solidity
uint256 factor = ratePerSecondE8 * dt;   // e.g. 1 * 1.7e9 = 1.7e9 >> E8 = 1e8
if (factor >= E8) return 0;              // always true for any non-zero rate
``` [3](#0-2) 

So `_applyWatermark(metricT0, 0, floorMultiplier)` receives `hwm = 0`. The drawdown floor is `0 × floorMultiplier / E6 = 0`. Any positive metric clears the floor — `breach0 = false` — and the stop-loss does not revert. [4](#0-3) 

**Impact gate check:**
The stop-loss extension is the only on-chain mechanism that blocks swaps causing per-share value loss beyond the configured drawdown. Bypassing it on the first swap allows an attacker to drain bin value in a single transaction without triggering `OracleStopLossTriggered`. After that first swap `lastDecayTs` is written and the protection works normally. The window is narrow but deterministic and unprivileged (any caller can be the first swapper).

**Scope:** `metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol` — production extension, not a test or mock.

### Title
`OracleValueStopLossExtension` stop-loss bypassed on first swap due to uninitialized `lastDecayTs` — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

### Summary
`BinHighWatermarks.lastDecayTs` defaults to `0` and is only written by `_checkAndUpdateWatermarks` (i.e., during `afterSwap`). When an admin sets per-bin watermarks via `executeOracleStopLossHighWatermarks` before any swap has touched the bin, `lastDecayTs` remains `0`. On the first swap, `dt = block.timestamp − 0 ≈ 1.7 × 10⁹ s`, which causes `_decayed` to return `0` for any non-zero watermark, making the stop-loss floor `0` and suppressing the breach flag entirely.

### Finding Description
`_checkAndUpdateWatermarks` computes elapsed time as:

```solidity
uint256 dt = block.timestamp - hwmS.lastDecayTs;   // 0 never written → dt ≈ 1.7e9
```

`_decayed` then evaluates:

```solidity
uint256 factor = ratePerSecondE8 * dt;
if (factor >= E8) return 0;   // true for any ratePerSecondE8 ≥ 1
```

returning `0` regardless of the stored watermark. `_applyWatermark(metric, 0, floorMultiplier)` computes a floor of `0`, so `breach = false` for any positive metric. The stop-loss does not revert.

`PendingHighWatermarks` (the struct used by the admin proposal flow) contains no `lastDecayTs` field, so `executeOracleStopLossHighWatermarks` cannot initialise it. The field is only ever written at line 284:

```solidity
hwmS.lastDecayTs = uint32(block.timestamp);
```

which is inside `_checkAndUpdateWatermarks`, reachable only through `afterSwap`. [5](#0-4) 

### Impact Explanation
An attacker who is the first swapper in a bin after admin has set watermarks can execute a swap of arbitrary size without the stop-loss triggering. This allows per-share value in that bin to fall below the configured drawdown floor in a single transaction, causing direct LP principal loss that the extension was deployed to prevent.

### Likelihood Explanation
The window exists from pool deployment (or from the moment admin sets watermarks) until the first swap touches the protected bin. For newly deployed pools with pre-configured watermarks this window is open from genesis. Any unprivileged user can be the first swapper.

### Recommendation
In `executeOracleStopLossHighWatermarks`, after writing `token0` and `token1` to `highWatermarks[pool][binIdx]`, also write `lastDecayTs = uint32(block.timestamp)`. This mirrors the pattern already used in `_checkAndUpdateWatermarks` and closes the uninitialized-timestamp window.

```solidity
BinHighWatermarks storage hwmS = highWatermarks[pool_][pending.binIdx];
hwmS.token0 = pending.token0;
hwmS.token1 = pending.token1;
hwmS.lastDecayTs = uint32(block.timestamp);   // ← add this line
```

### Proof of Concept
1. Deploy pool with `OracleValueStopLossExtension`, `drawdownE6 = 100_000` (10%), `decayPerSecondE8 = 1`.
2. Admin calls `proposeOracleStopLossHighWatermarks(pool, 0, hwm0, hwm1)` then `executeOracleStopLossHighWatermarks(pool)`. `highWatermarks[pool][0].token0 = hwm0`, `lastDecayTs = 0`.
3. Attacker calls `swap` with a size large enough to push per-share value 50% below `hwm0`.
4. Inside `afterSwap → _checkAndUpdateWatermarks`: `dt = block.timestamp ≈ 1.7e9`, `factor = 1 × 1.7e9 > 1e8`, `_decayed(hwm0, 1, dt) = 0`, `_applyWatermark(metric, 0, 900_000) → breach = false`.
5. Swap completes; `OracleStopLossTriggered` is never emitted. LPs suffer the full 50% per-share loss in that bin.

  function _decayed(uint256 hwm, uint256 ratePerSecondE8, uint256 dt) private pure returns (uint256) {
    if (ratePerSecondE8 == 0 || dt == 0 || hwm == 0) return hwm;
    uint256 factor = ratePerSecondE8 * dt;
    if (factor >= E8) return 0;
    return hwm - (hwm * factor) / E8;
  }
