### Title
Missing Zero-Address Validation in `updateRateProvider` Freezes Cross-Chain Rate Updates - (`contracts/cross-chain/CrossChainRateReceiver.sol`)

---

### Summary

`updateRateProvider` accepts `address(0)` without any guard. Once set, the `lzReceive` check `srcAddress == rateProvider` can never pass for a legitimate provider, silently blocking all future rate updates until the owner corrects the value.

---

### Finding Description

`updateRateProvider` writes `_rateProvider` directly to storage with no zero-address check: [1](#0-0) 

`lzReceive` then enforces an exact equality between the decoded source address and `rateProvider`: [2](#0-1) 

The `srcAddress` is decoded from the 20-byte-offset of `_srcAddress` (the LayerZero-packed remote+local address pair). A real provider address is never `address(0)`, so if `rateProvider` is `address(0)` the require at line 91 always reverts with `"Src address must be provider"`, and no rate update can land.

Contrast with other admin setters in the codebase that use `UtilLib.checkNonZeroAddress`: [3](#0-2) 

`updateRateProvider` is the only admin setter in this file that omits this guard.

---

### Impact Explanation

While `rateProvider` is set to `address(0)`, every `lzReceive` call from the legitimate provider reverts. The `rate` variable is frozen at its last stored value. Any protocol component that reads `getRate()` on the destination chain receives a stale rate for the entire window between the bad call and the corrective call. No funds are lost, but the contract fails to deliver its core promise (a live rsETH/ETH rate). This matches **Low – Contract fails to deliver promised returns, but doesn't lose value**.

Note: the claim of "no recovery without redeployment" is incorrect. The owner can recover by calling `updateRateProvider(correctAddress)` again, since there is no state that prevents re-invocation. [1](#0-0) 

---

### Likelihood Explanation

Requires the owner to pass `address(0)` — either by accident (e.g., a deployment script bug, a misconfigured multisig transaction) or through a UI that does not validate the input. The owner is a trusted role, so deliberate exploitation is out of scope, but accidental misconfiguration is a realistic operational risk for an admin-callable setter with no on-chain guard.

---

### Recommendation

Add a zero-address check consistent with the rest of the codebase:

```solidity
function updateRateProvider(address _rateProvider) external onlyOwner {
    UtilLib.checkNonZeroAddress(_rateProvider);   // add this line
    rateProvider = _rateProvider;
    emit RateProviderUpdated(_rateProvider);
}
``` [1](#0-0)

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import "forge-std/Test.sol";
import "contracts/cross-chain/RSETHRateReceiver.sol";

contract ZeroProviderTest is Test {
    RSETHRateReceiver receiver;
    address owner    = address(this);
    address endpoint = address(0xE0);
    address provider = address(0xA1);

    function setUp() public {
        receiver = new RSETHRateReceiver(1, provider, endpoint);
    }

    function test_zeroProviderFreezesRate() public {
        // Owner accidentally sets provider to address(0)
        receiver.updateRateProvider(address(0));
        assertEq(receiver.rateProvider(), address(0));

        // LayerZero endpoint tries to deliver a rate update from the real provider
        bytes memory srcAddress = abi.encodePacked(provider, address(receiver));
        bytes memory payload    = abi.encode(uint256(1.05e18));

        vm.prank(endpoint);
        vm.expectRevert("Src address must be provider");
        receiver.lzReceive(1, srcAddress, 0, payload);

        // Rate is frozen at 0 (initial value)
        assertEq(receiver.rate(), 0);
    }
}
