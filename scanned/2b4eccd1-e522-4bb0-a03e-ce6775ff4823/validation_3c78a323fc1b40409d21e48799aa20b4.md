### Title
Stale Cross-Chain Rate in `CrossChainRateReceiver` Enables Oracle Price Arbitrage in L2 Pool Token Deposits — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

---

### Summary

The `CrossChainRateReceiver` stores the rsETH/ETH rate received via LayerZero but exposes it through `getRate()` with **no staleness check**. L2 pool contracts (`RSETHPool`, `RSETHPoolV2`, `RSETHPoolV3`, `RSETHPoolNoWrapper`) use this stale cached rate alongside a **live Chainlink rate** for the same token-deposit calculation. The two oracles have structurally different update cadences, creating a price desync that allows any depositor to extract value from the pool.

---

### Finding Description

`CrossChainRateReceiver` stores `lastUpdated` but never enforces it:

```solidity
uint256 public rate;
uint256 public lastUpdated;

function getRate() external view returns (uint256) {
    return rate;   // no staleness check
}
``` [1](#0-0) [2](#0-1) 

The rate is only updated when someone calls `updateRate()` on the `MultiChainRateProvider` on L1 and the LayerZero message is delivered to the receiver. There is no on-chain heartbeat enforcement. [3](#0-2) 

`RSETHMultiChainRateProvider.getLatestRate()` reads the **stored** `LRTOracle.rsETHPrice`, which is itself only updated on-demand: [4](#0-3) 

In `RSETHPoolV3.viewSwapRsETHAmountAndFee` (token variant), two oracles with different update cadences are used **simultaneously** in the same pricing formula:

```solidity
uint256 rsETHToETHrate = getRate();                                    // stale cached rate
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate(); // live Chainlink rate
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [5](#0-4) 

The `tokenToETHRate` comes from `ChainlinkOracleForRSETHPoolCollateral`, which queries Chainlink's `latestRoundData()` live on every call: [6](#0-5) 

Because rsETH is a yield-bearing token whose value increases monotonically over time, a stale `rsETHToETHrate` will always be **lower** than the actual current rate. This causes the formula to produce a larger `rsETHAmount` than the deposited token is worth, minting excess wrsETH to the depositor.

The same pattern exists in `RSETHPool.viewSwapRsETHAmountAndFee` and `RSETHPoolNoWrapper.viewSwapRsETHAmountAndFee`: [7](#0-6) [8](#0-7) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

When the L2 `RSETHRateReceiver` rate is stale, any depositor calling `deposit(token, amount, referralId)` receives more wrsETH than the fair value of their deposited token. The pool's token balance is bridged to L1 and deposited into the LRT protocol, minting less rsETH than the outstanding wrsETH claims. The deficit is borne by all existing wrsETH/rsETH holders, constituting a direct extraction of accrued yield from the protocol.

---

### Likelihood Explanation

**Medium.** The rate becomes stale whenever `updateRate()` is not called on the `MultiChainRateProvider`. This can occur due to:
- LayerZero message delivery delays or failures
- High gas costs discouraging rate updates
- Operational gaps between update calls

No on-chain mechanism forces the rate to be fresh before a deposit is accepted. The longer the staleness window, the larger the extractable profit. At 5% APY, a 30-day staleness creates ~0.4% mispricing; on a $10M deposit that is ~$40K of excess wrsETH minted.

---

### Recommendation

Add a configurable `maxStaleness` parameter and enforce it in `CrossChainRateReceiver.getRate()`:

```solidity
uint256 public maxStaleness; // e.g., 86400 (24 hours)

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

This causes deposits to revert when the rate is stale, preventing mispriced minting until the rate is refreshed.

---

### Proof of Concept

1. The `RSETHRateReceiver` on an L2 chain holds a cached rate of `1.050e18` (rsETH/ETH), last updated 30 days ago.
2. The actual current rsETH/ETH rate on L1 is `1.054e18` (yield has accrued).
3. Attacker calls `RSETHPoolV3.deposit(wstETH, 100e18, "")`.
4. Pool fetches:
   - `rsETHToETHrate = 1.050e18` (stale, from `RSETHRateReceiver`)
   - `tokenToETHRate = 1.15e18` (live wstETH/ETH from Chainlink)
5. Pool computes: `rsETHAmount = 100e18 * 1.15e18 / 1.050e18 ≈ 109.52 rsETH`
6. Fair value at current rate: `100e18 * 1.15e18 / 1.054e18 ≈ 109.11 rsETH`
7. Attacker receives **~0.41 rsETH excess** per 100 wstETH deposited.
8. Attacker bridges wrsETH to L1, unwraps to rsETH, and redeems at the actual rate — extracting value from existing holders.

        uint256 latestRate = getLatestRate();

        rate = latestRate;

        lastUpdated = block.timestamp;

        bytes memory _payload = abi.encode(latestRate);

        uint256 rateReceiversLength = rateReceivers.length;

        for (uint256 i; i < rateReceiversLength;) {
            uint16 dstChainId = uint16(rateReceivers[i]._chainId);

            bytes memory remoteAndLocalAddresses = abi.encodePacked(rateReceivers[i]._contract, address(this));

            (uint256 estimatedFee,) = ILayerZeroEndpoint(layerZeroEndpoint)
                .estimateFees(dstChainId, address(this), _payload, false, bytes(""));

            ILayerZeroEndpoint(layerZeroEndpoint).send{ value: estimatedFee }(
                dstChainId, remoteAndLocalAddresses, _payload, payable(msg.sender), address(0x0), bytes("")
            );

            unchecked {
                ++i;
            }
        }

        emit RateUpdated(rate);
    }
