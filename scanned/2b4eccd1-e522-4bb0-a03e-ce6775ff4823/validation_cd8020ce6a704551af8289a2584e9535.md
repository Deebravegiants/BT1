### Title
Stale `rsETHPrice` Used Alongside Live `getAssetPrice()` in Mint Calculation Allows Depositors to Extract Excess rsETH — (`contracts/LRTDepositPool.sol`)

### Summary

`LRTDepositPool.getRsETHAmountToMint()` computes the rsETH to mint using two price sources that can be inconsistent: a **live** Chainlink-backed `getAssetPrice()` and a **stale cached** `rsETHPrice` stored in `LRTOracle`. Because `rsETHPrice` is only updated when `updateRSETHPrice()` is explicitly called, any depositor who acts in the window between a Chainlink price update and the next `updateRSETHPrice()` call receives more rsETH than their deposit is worth, diluting existing holders.

### Finding Description

`LRTDepositPool.getRsETHAmountToMint()` computes the rsETH mint amount as:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [1](#0-0) 

`lrtOracle.getAssetPrice(asset)` is a **live** call that reads directly from the underlying Chainlink price feed on every invocation:

```solidity
function getAssetPrice(address asset) public view onlySupportedOracle(asset) returns (uint256) {
    return IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset);
}
``` [2](#0-1) 

`lrtOracle.rsETHPrice()`, by contrast, is a **stored state variable** that is only updated when `_updateRsETHPrice()` is explicitly invoked:

```solidity
uint256 public override rsETHPrice;
``` [3](#0-2) 

```solidity
rsETHPrice = newRsETHPrice;
``` [4](#0-3) 

Neither `depositAsset()` nor `depositETH()` calls `updateRSETHPrice()` before computing the mint amount:

```solidity
function depositAsset(...) external nonReentrant whenNotPaused onlySupportedERC20Token(asset) {
    uint256 rsethAmountToMint = _beforeDeposit(asset, depositAmount, minRSETHAmountExpected);
    IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
    _mintRsETH(rsethAmountToMint);
    ...
}
``` [5](#0-4) 

`_beforeDeposit` calls `getRsETHAmountToMint` directly without any price refresh: [6](#0-5) 

The `rsETHPrice` is computed as `totalETHInProtocol / rsethSupply`, where `totalETHInProtocol` itself uses `getAssetPrice()` live. When Chainlink updates an asset price upward, `getAssetPrice()` immediately reflects the new value, but `rsETHPrice` remains at the old lower value until `updateRSETHPrice()` is called. The two values are now inconsistent.

### Impact Explanation

When a supported asset's Chainlink price increases (e.g., stETH/ETH rate rises), `getAssetPrice(stETH)` immediately returns the higher value, but `rsETHPrice` remains at the pre-update level. A depositor calling `depositAsset()` in this window receives:

```
rsethAmountToMint = amount * (new higher assetPrice) / (old lower rsETHPrice)
```

This is more rsETH than the deposit is worth at the current rate. The excess rsETH represents a claim on protocol TVL that was not contributed by the depositor — it is extracted from the yield accrued by existing rsETH holders. This constitutes **theft of unclaimed yield** (High impact per scope).

### Likelihood Explanation

Chainlink price feeds update continuously. The window between a Chainlink heartbeat update and the next `updateRSETHPrice()` call is always non-zero and can span many blocks. Any depositor — with no special privileges — can observe the Chainlink price update on-chain and immediately call `depositAsset()` or `depositETH()` before `updateRSETHPrice()` is called. No front-running of other users is required; the attacker simply acts in the natural staleness window. This is a realistic, repeatable condition.

### Recommendation

Call `updateRSETHPrice()` (or an internal equivalent) at the start of `depositAsset()` and `depositETH()` before computing the mint amount, so that `rsETHPrice` is always consistent with the live asset prices used in the same calculation. Alternatively, compute `rsETHPrice` on-the-fly inside `getRsETHAmountToMint()` using the same live `getAssetPrice()` calls rather than reading the cached storage variable.

### Proof of Concept

1. Protocol state: stETH/ETH Chainlink price = 1.00, `rsETHPrice` = 1.05 (last updated).
2. Chainlink updates stETH/ETH to 1.02. `getAssetPrice(stETH)` now returns 1.02e18.
3. `updateRSETHPrice()` has not yet been called; `rsETHPrice` is still 1.05e18.
4. Attacker calls `depositAsset(stETH, 100e18, ...)`.
5. `getRsETHAmountToMint` computes: `100e18 * 1.02e18 / 1.05e18 ≈ 97.14 rsETH`.
6. Correct amount (if `rsETHPrice` were updated first to reflect the new TVL) would be lower, because the new `rsETHPrice` would be `(totalETHInProtocol_new) / rsethSupply`, which is higher than 1.05 after the price increase.
7. The attacker receives excess rsETH at the expense of existing holders whose share of TVL is diluted. [7](#0-6) [8](#0-7)

    {

            revert InvalidAmountToDeposit();
        }

        if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
            revert MaximumDepositLimitReached();
        }

        rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

        if (rsethAmountToMint < minRSETHAmountExpected) {
            revert MinimumAmountToReceiveNotMet();
        }
    }
