### Title
ETH Deposit Limit Check Ignores Deposit Amount, Allowing Limit Bypass - (File: `contracts/LRTDepositPool.sol`)

### Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` silently drops the `amount` parameter for ETH deposits, checking only whether the *current* total already exceeds the limit rather than whether the *incoming* deposit would push it over. Any depositor can bypass the ETH deposit cap with a single call.

### Finding Description
The function signature accepts `amount` for all assets, but the ETH branch ignores it:

```solidity
// contracts/LRTDepositPool.sol  lines 676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount never used
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct for ERC-20
}
```

For every ERC-20 asset the prospective deposit is added before comparing against the cap. For ETH it is not. The check therefore returns `false` (i.e. "not exceeded") whenever `totalAssetDeposits == depositLimit`, even though the incoming deposit will push the total above the cap. The caller `_beforeDeposit` treats a `false` return as "proceed", so the deposit is accepted and `depositETH` mints rsETH for the full amount. [1](#0-0) 

The entry path is the public, permissionless `depositETH`: [2](#0-1) 

which calls `_beforeDeposit` → `_checkIfDepositAmountExceedesCurrentLimit`. [3](#0-2) 

### Impact Explanation
The ETH deposit cap is the protocol's primary risk-management gate for native ETH exposure. Bypassing it allows the protocol to accumulate more ETH than governance has approved, inflating TVL beyond the intended ceiling and undermining the cap's purpose. Because the bypass is silent (no revert, no event), operators have no on-chain signal that the limit has been crossed. This matches **Low – contract fails to deliver promised returns, but doesn't lose value**: the cap guarantee is broken, but no funds are directly stolen.

### Likelihood Explanation
The condition is met whenever `totalAssetDeposits == depositLimit` exactly. Because ETH deposits are continuous and the total is a running sum, this equality is reached transiently on every deposit that fills the pool to the limit. Any unprivileged depositor who observes (or races to) that moment can push an arbitrarily large ETH deposit through. No special role or prior setup is required.

### Recommendation
Apply the same prospective check used for ERC-20 assets:

```diff
-    if (asset == LRTConstants.ETH_TOKEN) {
-        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
-    }
+    if (asset == LRTConstants.ETH_TOKEN) {
+        return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
+    }
```

### Proof of Concept
1. Governance sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. Legitimate deposits bring `totalAssetDeposits` to exactly `1000 ether`.
3. Attacker calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `1000 ether > 1000 ether` → `false` → "not exceeded".
5. `_beforeDeposit` proceeds; `_mintRsETH` mints rsETH for 500 ETH.
6. `totalAssetDeposits` is now `1500 ether`, 50 % above the approved cap, with no revert or alert.
