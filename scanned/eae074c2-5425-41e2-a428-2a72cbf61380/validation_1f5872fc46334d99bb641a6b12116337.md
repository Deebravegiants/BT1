### Title
Missing Zero-Address Check for `admin` and `bridger` in `initialize()` Leaves Pool Contracts Permanently Un-administrable - (File: contracts/pools/RSETHPoolV2NBA.sol, contracts/pools/RSETHPoolV3WithNativeChainBridge.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPool.sol)

---

### Summary

Multiple L2 pool `initialize()` functions validate token/oracle addresses with `UtilLib.checkNonZeroAddress()` but omit the same check for the `admin` (and `bridger`) role addresses. If `admin` is mistakenly set to `address(0)` at deployment, `DEFAULT_ADMIN_ROLE` is permanently granted to the zero address, making all admin-gated functions unreachable and the contract permanently un-administrable.

---

### Finding Description

In `RSETHPoolV2NBA.initialize()`, `RSETHPoolV3WithNativeChainBridge.initialize()`, `RSETHPoolV3ExternalBridge.initialize()`, and `RSETHPool.initialize()`, the pattern is consistent: token/oracle addresses are validated, but the `admin` (and `bridger`) role addresses are not.

**RSETHPoolV2NBA.sol** (`initialize`, lines 75–97): [1](#0-0) 

Only `_wrsETH` and `_rsETHOracle` are checked; `admin` and `bridger` are passed directly to `_grantRole`/`_setupRole` without any zero-address guard.

**RSETHPoolV3WithNativeChainBridge.sol** (`initialize`, lines 245–268): [2](#0-1) 

Same pattern: `_wrsETH` and `_rsETHOracle` are checked; `admin` and `bridger` are not.

**RSETHPoolV3ExternalBridge.sol** (`initialize`, lines 330–352): [3](#0-2) 

**RSETHPool.sol** (`initialize`, lines 222–251): [4](#0-3) 

Only `_rsETH` and `_wstETH` are checked; `admin` and `manager` are not.

By contrast, contracts such as `RSETHPoolNoWrapper.initialize()` and `L1Vault.initialize()` correctly validate all role addresses: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

If `admin = address(0)` is passed at deployment:

- `DEFAULT_ADMIN_ROLE` is permanently granted to `address(0)`, which no EOA or contract controls.
- All `onlyRole(DEFAULT_ADMIN_ROLE)` functions become permanently inaccessible: `setFeeBps`, `unpause`, `setDailyMintLimit` (V3WithNativeChainBridge), and any future admin-gated operations.
- The contract cannot be upgraded or reconfigured. Fee parameters are frozen at their initial values forever.
- If the contract is ever paused (by a holder of `PAUSER_ROLE`), it can never be unpaused, permanently blocking user deposits.

**Impact: Low** — Contract fails to deliver promised returns (fee updates, oracle updates, unpausing) without direct loss of user funds at rest.

---

### Likelihood Explanation

Likelihood is **Low**. This requires a deployment-time error where `address(0)` is passed as `admin`. It is not exploitable by an external user post-deployment. However, the inconsistency across the codebase (some contracts check, others do not) increases the probability of a copy-paste or scripting error during multi-chain deployment.

---

### Recommendation

Add `UtilLib.checkNonZeroAddress(admin)` and `UtilLib.checkNonZeroAddress(bridger)` (or `manager`) at the top of every `initialize()` function that grants roles, consistent with the pattern already used in `RSETHPoolNoWrapper` and `L1Vault`:

```solidity
UtilLib.checkNonZeroAddress(admin);
UtilLib.checkNonZeroAddress(bridger); // or manager
UtilLib.checkNonZeroAddress(_wrsETH);
UtilLib.checkNonZeroAddress(_rsETHOracle);
```

---

### Proof of Concept

1. Deploy `RSETHPoolV2NBA` via its proxy, calling `initialize(address(0), validBridger, validWrsETH, feeBps, validOracle)`.
2. `_grantRole(DEFAULT_ADMIN_ROLE, address(0))` succeeds silently — no revert.
3. Call `setFeeBps(500)` → reverts with `AccessControl: account ... is missing role DEFAULT_ADMIN_ROLE`.
4. Call `unpause()` → same revert.
5. The contract is permanently un-administrable; no recovery path exists since `initialize` is `initializer`-guarded and cannot be called again. [7](#0-6) [8](#0-7)
