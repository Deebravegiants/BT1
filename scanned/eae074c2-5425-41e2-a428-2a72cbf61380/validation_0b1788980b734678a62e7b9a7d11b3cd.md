### Title
No Slippage Protection in `LRTWithdrawalManager.initiateWithdrawal()` — (File: contracts/LRTWithdrawalManager.sol)

---

### Summary

`LRTWithdrawalManager.initiateWithdrawal()` accepts rsETH from users and records an `expectedAssetAmount` based on oracle prices at request time. However, the actual payout is re-evaluated at unlock time via `_calculatePayoutAmount()`, which can return a strictly lower amount if prices have moved adversely during the mandatory 8-day delay. There is no `minAmountOut` or `deadline` parameter in `initiateWithdrawal()`, and no cancellation mechanism exists, leaving users with no on-chain protection against receiving fewer assets than expected.

---

### Finding Description

`initiateWithdrawal()` transfers rsETH from the user and stores a `WithdrawalRequest` with `expectedAssetAmount` computed from live oracle prices at request time: [1](#0-0) 

When the privileged operator later calls `unlockQueue()`, `_unlockWithdrawalRequests()` invokes `_calculatePayoutAmount()`, which returns the **minimum** of the stored `expectedAssetAmount` and a freshly computed `currentReturn` using oracle prices at unlock time: [2](#0-1) 

If the rsETH price has fallen relative to the asset price during the delay, `currentReturn < expectedAssetAmount`, and `request.expectedAssetAmount` is overwritten with the lower value before rsETH is burned: [3](#0-2) 

The user then receives this reduced amount in `_processWithdrawalCompletion()`: [4](#0-3) 

The default `withdrawalDelayBlocks` is set to 8 days, creating a long window for adverse price movement: [5](#0-4) 

`initiateWithdrawal()` accepts no `minAmountOut` or `deadline` parameter: [6](#0-5) 

No cancellation function exists anywhere in `LRTWithdrawalManager`, so once rsETH is committed, the user cannot exit the position regardless of how unfavorable prices become.

---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A user who initiates a withdrawal expecting `X` units of an LST may receive materially fewer units if the rsETH/LST exchange rate declines during the 8-day delay. The user's rsETH is irrevocably committed at request time with no on-chain mechanism to enforce a minimum acceptable output. The protocol itself does not lose funds; only the withdrawing user bears the shortfall.

---

### Likelihood Explanation

**Medium.** LST exchange rates (stETH, ETHx, etc.) relative to rsETH can shift meaningfully over an 8-day window, particularly during market stress, validator slashing events, or depeg episodes. The operator-controlled `unlockQueue()` call timing adds further uncertainty about exactly when prices are sampled. Any unprivileged rsETH holder who calls `initiateWithdrawal()` is exposed.

---

### Recommendation

1. Add a `minAmountOut` parameter to `initiateWithdrawal()`. Revert if `getExpectedAssetAmount(asset, rsETHUnstaked) < minAmountOut` at request time, and/or revert in `_calculatePayoutAmount()` if the computed payout falls below the user-specified floor.
2. Add a `deadline` (block number or timestamp) parameter to `initiateWithdrawal()` so users can bound how long their request remains valid.
3. Implement a user-callable cancellation function that returns the committed rsETH if the request has not yet been unlocked.

---

### Proof of Concept

1. User calls `initiateWithdrawal(stETH, 1e18, "")` when the oracle gives rsETH/stETH ≈ 1.0 → `expectedAssetAmount = 1e18`.
2. `1e18` rsETH is transferred from the user to the contract and the request is stored.
3. Eight days pass; a validator slashing event causes the rsETH/stETH rate to drop to 0.90.
4. Operator calls `unlockQueue(stETH, ...)` → `_calculatePayoutAmount()` computes `currentReturn = 1e18 * rsETHPrice / assetPrice = 0.9e18`, which is less than `expectedAssetAmount`, so `request.expectedAssetAmount` is set to `0.9e18`.
5. User calls `completeWithdrawal(stETH, "")` → receives `0.9e18 stETH` instead of the `1e18 stETH` they expected.
6. The user had no `minAmountOut` to revert the request, no `deadline` to expire it, and no cancellation path to recover their rsETH — the loss is final. [7](#0-6)
