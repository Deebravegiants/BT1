### Title
Daily Mint Limit Exhaustion Enables Cheap Front-Run DoS on Deposits - (File: contracts/pools/RSETHPoolV3.sol)

### Summary

The `limitDailyMint` modifier in `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, and `RSETHPoolV3WithNativeChainBridge` allows any depositor to exhaust the shared `dailyMintLimit` in a single transaction. An attacker who observes a victim's pending deposit in the public mempool can front-run it with a deposit large enough to fill the remaining daily quota, causing the victim's transaction to revert with `DailyMintLimitExceeded`. Because the attacker receives `wrsETH` in return (no capital loss), and because `feeBps` can be set as low as 0, the attack cost is bounded only by gas — which is negligible on L2 chains where all pool variants are deployed.

### Finding Description

Every deposit path in `RSETHPoolV3` is gated by the `limitDailyMint` modifier:

```solidity
modifier limitDailyMint(uint256 amount, address token) {
    ...
    if (dailyMintAmount + rsETHAmount > dailyMintLimit) {
        revert DailyMintLimitExceeded();
    }
    dailyMintAmount += rsETHAmount;
    _;
}
``` [1](#0-0) 

The modifier increments `dailyMintAmount` before the function body executes. There is no minimum-output parameter (`minAmountOut`) and no per-depositor quota. Any single address can consume the entire remaining daily allowance in one call.

The fee charged to the attacker is:

```solidity
fee = amount * feeBps / 10_000;
``` [2](#0-1) 

`feeBps` is admin-configurable and is bounded only by a maximum of 1000 bps (10%); it can be set to 0:

```solidity
function setFeeBps(uint256 _feeBps) external onlyRole(DEFAULT_ADMIN_ROLE) {
    if (_feeBps > 1000) revert InvalidFeeAmount();
    feeBps = _feeBps;
``` [3](#0-2) 

The attacker receives `wrsETH` equal in value to the deposited ETH minus the fee, so the net capital outlay is only the fee (possibly zero) plus L2 gas. The same pattern exists in `RSETHPoolV3ExternalBridge` and `RSETHPoolV3WithNativeChainBridge`: [4](#0-3) [5](#0-4) 

### Impact Explanation

A victim's deposit reverts with `DailyMintLimitExceeded` and the victim's ETH/LST is returned. However, the victim is blocked from depositing for up to 24 hours (until `getCurrentDay()` advances and `dailyMintAmount` resets). Repeated attacks across consecutive days constitute a sustained denial of service on the deposit path, which is the primary user-facing function of the L2 pool contracts. This maps to **Medium — Temporary freezing of funds** (users cannot convert ETH/LSTs to rsETH for up to 24 hours per attack cycle).

### Likelihood Explanation

All pool variants are deployed on L2 networks (Arbitrum, Optimism, Unichain) where gas costs are negligible. The attacker retains full economic value via `wrsETH`, so the only real cost is the protocol fee. When `feeBps = 0` the attack is essentially free. Even at non-zero fee rates the attacker suffers no net loss because `wrsETH` accrues restaking yield. The attack is executable by any address with sufficient capital to reach the daily limit, and the victim's pending transaction is visible in the public mempool.

### Recommendation

1. **Add a `minAmountOut` parameter** to `deposit` so users can specify the minimum `wrsETH` they accept, enabling them to detect when the daily limit has been exhausted before their transaction is mined.
2. **Enforce a per-address deposit cap** within each daily window so no single depositor can exhaust the entire `dailyMintLimit`.
3. **Enforce a non-zero minimum `feeBps`** (e.g., at least 5 bps) so that exhausting the daily limit always carries a real cost proportional to the amount deposited.
4. Consider using a **private RPC / commit-reveal** pattern for large deposits to prevent mempool-visible front-running.

### Proof of Concept

1. Victim submits `deposit{value: V}(referralId)` to `RSETHPoolV3` with `V` small enough to fit within the remaining daily quota `R`.
2. Attacker observes the pending transaction and submits `deposit{value: R}(referralId)` with a higher gas price, consuming the entire remaining daily quota.
3. Victim's transaction executes after the attacker's and reverts: `dailyMintAmount + rsETHAmount > dailyMintLimit`.
4. Attacker holds `wrsETH` worth approximately `R` ETH minus the fee. Net cost: `R * feeBps / 10_000` + gas (≈ 0 on L2 when `feeBps = 0`).
5. Victim cannot deposit again until the next UTC day boundary (`startTimestamp + (getCurrentDay() + 1) * 1 days`). [6](#0-5) [7](#0-6)

```text
    modifier limitDailyMint(uint256 amount, address token) {
        if (block.timestamp < startTimestamp) {
            revert MintBeforeStartTimestamp();
        }

        uint256 rsETHAmount;

        // Calculate the amount of rsETH that will be minted
        if (token == ETH_IDENTIFIER) {
            (rsETHAmount,) = viewSwapRsETHAmountAndFee(amount);
        } else {
            (rsETHAmount,) = viewSwapRsETHAmountAndFee(amount, token);
        }

        uint256 currentDay = getCurrentDay();

        // If the current day is greater than the last mint day, reset the daily mint amount
        if (currentDay > lastMintDay) {
            lastMintDay = currentDay;
            dailyMintAmount = 0;
        }

        // Check if the daily mint amount plus the amount to mint is greater than the daily mint limit
        if (dailyMintAmount + rsETHAmount > dailyMintLimit) {
            revert DailyMintLimitExceeded();
        }

        dailyMintAmount += rsETHAmount;
        _;
    }
```

```text
    function deposit(string memory referralId)
        external
        payable
        nonReentrant
        whenNotPaused
        limitDailyMint(msg.value, ETH_IDENTIFIER)
    {
        if (!isEthDepositEnabled) revert EthDepositDisabled();
        uint256 amount = msg.value;

        if (amount == 0) revert InvalidAmount();

        (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);

        feeEarnedInETH += fee;

        wrsETH.mint(msg.sender, rsETHAmount);

        emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
    }
