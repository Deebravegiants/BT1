### Title
Deposit Limit DoS via Direct Token Transfer Inflating `getTotalAssetDeposits` - (File: contracts/LRTDepositPool.sol)

### Summary
An unprivileged attacker can send a dust amount of any supported ERC-20 LST (or ETH) directly to `LRTDepositPool`, inflating the raw balance used by `getTotalAssetDeposits`. When the protocol is operating near its configured deposit cap, this pushes `totalAssetDeposits` over `depositLimitByAsset`, causing every subsequent legitimate `depositAsset` / `depositETH` call to revert with `MaximumDepositLimitReached` until an admin raises the cap.

### Finding Description

`_checkIfDepositAmountExceedesCurrentLimit` computes the protocol-wide asset total by calling `getTotalAssetDeposits`, which in turn calls `getAssetDistributionData` / `getETHDistributionData`. Both helpers read the **raw on-chain balance** of the deposit pool:

```solidity
// contracts/LRTDepositPool.sol  line 444
assetLyingInDepositPool = IERC20(asset).balanceOf(address(this));

// contracts/LRTDepositPool.sol  line 480
ethLyingInDepositPool = address(this).balance;
```

These values are then summed and compared against the configured cap:

```solidity
// contracts/LRTDepositPool.sol  line 676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

Because `LRTDepositPool` has an open `receive()` function and accepts arbitrary ERC-20 transfers, any external actor can increase `assetLyingInDepositPool` / `ethLyingInDepositPool` without going through the deposit flow. There is no internal accounting variable that tracks only protocol-originated deposits; the check relies entirely on the live balance.

**Attack path (ERC-20 example):**
1. Deposit limit for stETH is 10 000 stETH; current `totalAssetDeposits` = 9 999.9 stETH.
2. Attacker calls `stETH.transfer(depositPool, 0.1 stETH + 1 wei)`.
3. `totalAssetDeposits` is now `10 000 stETH + 1 wei`.
4. Any user calling `depositAsset(stETH, any_amount, ...)` hits `totalAssetDeposits + amount > depositLimitByAsset` → `MaximumDepositLimitReached`.

**Attack path (ETH):**
Because `depositETH` is `payable`, the user's ETH lands in `address(this).balance` *before* `_beforeDeposit` runs. The ETH check therefore reads the already-inflated balance:

```solidity
// contracts/LRTDepositPool.sol  line 678-679
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
}
```

Attacker sends 1 wei via `receive()` → `ethLyingInDepositPool` increases by 1 wei → if the total was exactly at the cap, every future ETH deposit reverts.

### Impact Explanation

All `depositAsset` / `depositETH` calls for the targeted asset revert until an admin raises `depositLimitByAsset`. Users cannot deposit; rsETH cannot be minted. This constitutes a **temporary freezing of funds** (users' capital is not at risk, but the protocol's deposit functionality is disabled for the affected asset). Impact: **Medium**.

### Likelihood Explanation

The attack is cheap (dust amount of the target token or 1 wei of ETH), permissionless, and requires no special knowledge beyond the current TVL and deposit cap (both readable on-chain). The protocol is designed to operate near its deposit cap as a TVL-control mechanism, making the precondition realistic. Likelihood: **Medium**.

### Recommendation

Replace raw `balanceOf` / `address(this).balance` reads in `getAssetDistributionData` and `getETHDistributionData` with an internal accounting variable (e.g., `depositedByProtocol[asset]`) that is incremented only inside `depositAsset` / `depositETH` and decremented when assets are transferred out via the privileged transfer functions. This mirrors the fix recommended in DEPU-1: do not rely on the contract's live balance for security-critical comparisons.

### Proof of Concept

```
// Precondition: stETH deposit limit = 10_000e18, current totalAssetDeposits = 9_999.9e18

// Step 1 – attacker inflates the pool balance with dust
stETH.transfer(address(lrtDepositPool), 0.1e18 + 1);

// Step 2 – legitimate user attempts to deposit
// totalAssetDeposits is now 10_000e18 + 1 wei
// _checkIfDepositAmountExceedesCurrentLimit:
//   totalAssetDeposits + amount = 10_000e18 + 1 + userAmount > 10_000e18  → true
lrtDepositPool.depositAsset(stETH, 1e18, 0, "");
// ↑ reverts: MaximumDepositLimitReached
```
