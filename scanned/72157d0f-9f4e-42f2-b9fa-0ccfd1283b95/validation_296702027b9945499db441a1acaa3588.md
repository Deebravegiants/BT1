### Title
Missing Minimum Output (Slippage) Protection in `deposit()` Functions Across All L2 Pool Variants - (File: contracts/pools/RSETHPoolV3.sol, RSETHPoolV3ExternalBridge.sol, RSETHPoolV3WithNativeChainBridge.sol, RSETHPool.sol, RSETHPoolNoWrapper.sol)

---

### Summary

All L2 deposit pool variants expose `deposit()` functions that accept ETH or supported LSTs and mint `wrsETH`/`rsETH` to the caller. The minted amount is computed at execution time using the live oracle rate. No `minRsETHAmountOut` parameter exists, so users cannot enforce a minimum acceptable output. If the oracle rate is updated between a user's preview call and their deposit transaction, they silently receive fewer tokens than expected with no recourse.

---

### Finding Description

Every `deposit()` function across all pool variants computes the output amount by reading the live oracle rate at execution time:

```solidity
// RSETHPoolV3.sol – deposit(string)
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
wrsETH.mint(msg.sender, rsETHAmount);
```

`viewSwapRsETHAmountAndFee` divides the post-fee input by the current oracle rate:

```solidity
uint256 rsETHToETHrate = getRate();          // live oracle read
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

The `rsETHOracle` on each L2 is a cross-chain rate receiver that is updated periodically from L1. If a rate update is pushed (increasing `rsETHToETHrate`) between the block in which the user previewed the swap and the block in which their `deposit()` executes, the user receives fewer `wrsETH`/`rsETH` than the preview indicated, with no on-chain protection.

The same pattern is present identically in all five pool contracts:

| Contract | ETH deposit | Token deposit |
|---|---|---|
| `RSETHPoolV3` | line 246–265 | line 271–293 |
| `RSETHPoolV3ExternalBridge` | line 366–384 | line 390–412 |
| `RSETHPoolV3WithNativeChainBridge` | line 282–301 | line 307–329 |
| `RSETHPool` | line 265–278 | line 284–305 |
| `RSETHPoolNoWrapper` | line 231–244 | line 250–271 |

None of these functions accept a `minRsETHAmountOut` parameter.

---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but does not lose value.**

A user who calls `viewSwapRsETHAmountAndFee` off-chain to preview their swap, then submits a `deposit()` transaction, may receive fewer `wrsETH`/`rsETH` tokens than the preview showed. Because the rsETH rate is monotonically non-decreasing (it reflects accrued yield), the user's ETH value is preserved in the fewer-but-more-valuable tokens; however, the user has no on-chain mechanism to reject the transaction if the output falls below their acceptable threshold. This breaks the standard swap UX guarantee and can cause downstream issues for users who planned to use a specific token quantity (e.g., as collateral).

---

### Likelihood Explanation

The rsETH exchange rate is pushed from L1 to each L2 via a cross-chain rate provider. Rate updates are periodic and can land in any block. Any user whose `deposit()` transaction is included in the same block as, or immediately after, a rate update will receive fewer tokens than their preview indicated. This is a normal operating condition, not an edge case, and requires no adversarial action.

---

### Recommendation

Add a `minRsETHAmountOut` parameter to all `deposit()` overloads and revert if the computed output falls below it:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountOut)
    external payable nonReentrant whenNotPaused
    limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountOut) revert SlippageExceeded();
    wrsETH.mint(msg.sender, rsETHAmount);
    ...
}
```

Apply the same pattern to the token-deposit overload and to all five pool contracts.

---

### Proof of Concept

1. Alice calls `viewSwapRsETHAmountAndFee(1 ether)` off-chain. With `rsETHToETHrate = 1.05e18`, she expects `~0.952 wrsETH`.
2. Before Alice's transaction is mined, the L2 oracle receives a rate update pushing `rsETHToETHrate` to `1.06e18`.
3. Alice's `deposit{value: 1 ether}("ref")` executes. The contract computes `rsETHAmount = 1e18 * 1e18 / 1.06e18 ≈ 0.943 wrsETH`.
4. Alice receives `~0.009 wrsETH` fewer than expected with no revert and no warning.
