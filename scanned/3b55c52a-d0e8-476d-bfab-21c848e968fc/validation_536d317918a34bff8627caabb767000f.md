### Title
Chainlink `minAnswer`/`maxAnswer` Not Validated in `ChainlinkPriceOracle` — Inflated LST Price Enables rsETH Over-Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but never validates the returned price against the feed's `minAnswer`/`maxAnswer` circuit breakers. If a supported LST asset crashes below the Chainlink feed's `minAnswer`, the oracle silently returns the artificially floored price. This inflated price is then used by `LRTDepositPool` to mint rsETH, allowing an attacker to deposit a worthless (or near-worthless) LST and receive rsETH minted at the stale high price — draining value from all other rsETH holders.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the raw Chainlink answer with no bounds check:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

There is no check that `price` is within `[minAnswer, maxAnswer]`. Chainlink aggregators with active circuit breakers will clamp the returned value to `minAnswer` during a price crash, so the contract will consume a stale, inflated price without any indication of error.

This price is consumed by `LRTOracle.getAssetPrice()`: [2](#0-1) 

Which is called by `LRTDepositPool.getRsETHAmountToMint()`: [3](#0-2) 

Which drives the rsETH minting amount in `depositAsset()`: [4](#0-3) 

The same root cause also exists in `ChainlinkOracleForRSETHPoolCollateral.getRate()`, which is used as the collateral token oracle in L2 pools (`RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`) to price non-ETH tokens during deposit: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

**Critical — Direct theft of user funds / protocol insolvency.**

If a supported LST (e.g., stETH, rETH) suffers a flash crash below the Chainlink feed's `minAnswer`:

1. `ChainlinkPriceOracle.getAssetPrice(stETH)` returns `minAnswer` (e.g., $1,000) instead of the real price (e.g., $100).
2. `getRsETHAmountToMint` mints rsETH at 10× the real value of the deposited stETH.
3. The attacker redeems rsETH for ETH, extracting value from all other depositors.
4. The protocol's TVL accounting (`_getTotalEthInProtocol`) also uses this inflated price, causing `rsETHPrice` to be overstated, compounding the damage. [7](#0-6) 

---

### Likelihood Explanation

Chainlink still maintains active `minAnswer`/`maxAnswer` circuit breakers on many LST/ETH feeds (particularly on non-mainnet EVM chains). The protocol is deployed across multiple EVM-compatible chains. A flash crash or depeg event for any supported LST is a realistic market scenario (cf. stETH depeg in 2022). The attacker entry path requires only a standard `depositAsset()` call — no special role or privilege.

---

### Recommendation

After calling `latestRoundData()`, validate the returned price against the aggregator's `minAnswer` and `maxAnswer`:

```solidity
IChainlinkAggregator aggregator = IChainlinkAggregator(priceFeed.aggregator());
int192 minAnswer = aggregator.minAnswer();
int192 maxAnswer = aggregator.maxAnswer();
if (price <= minAnswer || price >= maxAnswer) revert PriceOutOfBounds();
```

Apply the same fix to `ChainlinkOracleForRSETHPoolCollateral.getRate()`.

---

### Proof of Concept

1. Assume stETH is a supported asset with a Chainlink ETH/stETH feed that has `minAnswer = 0.9e18` (90% of ETH).
2. stETH depegs to `0.1e18` (10% of ETH) due to a slashing event.
3. Chainlink feed clamps and returns `0.9e18` as the price.
4. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
5. `getRsETHAmountToMint` computes: `(1000e18 * 0.9e18) / rsETHPrice` — 9× more rsETH than the real value.
6. Attacker receives 9× over-minted rsETH and redeems it, draining ETH from the protocol at the expense of honest depositors. [8](#0-7) [9](#0-8)
