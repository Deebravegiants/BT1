### Title
`_updateRsETHPrice()` Reverts on Daily Fee Mint Limit Exceeded, Permanently Blocking rsETH Price Updates Within the Period - (File: contracts/LRTOracle.sol)

---

### Summary

`LRTOracle._updateRsETHPrice()` unconditionally calls `_checkAndUpdateDailyFeeMintLimit()` at the end of every execution path. When the computed protocol fee in rsETH exceeds `maxFeeMintAmountPerDay`, this internal check reverts with `DailyFeeMintLimitExceeded`, causing the entire price update to revert. Both the public `updateRSETHPrice()` and the manager-only `updateRSETHPriceAsManager()` share the same internal path, so neither can succeed. The stored `rsETHPrice` becomes stale for the remainder of the day, the protocol fee is permanently lost, and the automatic downside-protection pause cannot fire.

---

### Finding Description

In `_updateRsETHPrice()`, after computing `newRsETHPrice` and `protocolFeeInETH`, the function always ends with a call to `_checkAndUpdateDailyFeeMintLimit`:

```solidity
// contracts/LRTOracle.sol lines 298-311
if (protocolFeeInETH > 0) {
    uint256 rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice);
    _checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee);   // ← can revert
    ...
} else {
    _checkAndUpdateDailyFeeMintLimit(0);
}
``` [1](#0-0) 

`_checkAndUpdateDailyFeeMintLimit` reverts unconditionally when the fee amount exceeds the remaining daily budget:

```solidity
// contracts/LRTOracle.sol lines 205-206
if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
    revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
}
``` [2](#0-1) 

Because the revert is inside `_updateRsETHPrice()`, it propagates through both public entry points:

- `updateRSETHPrice()` — callable by anyone, no access control [3](#0-2) 
- `updateRSETHPriceAsManager()` — callable only by LRT manager [4](#0-3) 

Neither path can bypass the limit check. The stored `rsETHPrice` is only written at line 313 (`rsETHPrice = newRsETHPrice`), which is never reached when the revert fires. [5](#0-4) 

The stale `rsETHPrice` is then consumed by every downstream consumer:

- `LRTDepositPool.getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()` [6](#0-5) 
- `LRTWithdrawalManager.getExpectedAssetAmount()` multiplies by `lrtOracle.rsETHPrice()` [7](#0-6) 
- All L2 pool `viewSwapRsETHAmountAndFee()` variants divide by `getRate()` which reads the same stored price [8](#0-7) 

Additionally, the downside-protection auto-pause (lines 270–281) is part of `_updateRsETHPrice()` and is never reached when the function reverts, so a significant price drop cannot trigger the automatic pause. [9](#0-8) 

---

### Impact Explanation

**Temporary freezing of funds / permanent loss of unclaimed yield (Medium).**

For the entire 24-hour period in which the limit is exceeded:

1. `rsETHPrice` is frozen at its last successful value. Deposits mint rsETH at the wrong rate; withdrawals redeem at the wrong rate. Users who deposit while the true price is higher receive more rsETH than they are entitled to, diluting existing holders.
2. The protocol fee for that day is permanently unrecoverable — the fee is computed from the TVL delta at the time of the call, and once the period rolls over the delta is recalculated from the new (now-updated) stale price, erasing the missed fee.
3. The automatic downside-protection pause cannot fire during the blocked period, leaving the protocol exposed to a price-drop scenario without its safety net.

---

### Likelihood Explanation

`maxFeeMintAmountPerDay` is set by the LRT manager to a finite value. The fee amount minted per call is `protocolFeeInETH / newRsETHPrice`, where `protocolFeeInETH = (totalETHInProtocol - previousTVL) * protocolFeeInBPS / 10_000`. During periods of high yield (e.g., large EigenLayer rewards, LST price spikes, or simply a long gap between price updates), the single-call fee can exceed the daily cap. Because `updateRSETHPrice()` is public and permissionless, any caller — including a bot or a user — triggers the revert, and no privileged actor can override it through the same code path.

---

### Recommendation

Decouple the fee-minting limit check from the price-update logic. Options:

1. **Cap, don't revert**: In `_checkAndUpdateDailyFeeMintLimit`, mint only up to the remaining daily budget and silently skip the excess rather than reverting.
2. **Separate transactions**: Update `rsETHPrice` unconditionally; emit an event for the unclaimed fee; allow a separate privileged call to mint the fee when capacity is available.
3. **Try/catch**: Wrap the fee-minting block in a try/catch so that a limit breach does not roll back the price update.

---

### Proof of Concept

1. Manager calls `setMaxFeeMintAmountPerDay(1e18)` (1 rsETH cap per day).
2. EigenLayer rewards accrue; `totalETHInProtocol` grows by 200 ETH relative to `previousTVL`.
3. With `protocolFeeInBPS = 1000` (10 %), `protocolFeeInETH = 20 ETH`.
4. At `newRsETHPrice ≈ 1.05e18`, `rsethAmountToMintAsProtocolFee ≈ 19.05 rsETH > 1 rsETH`.
5. `_checkAndUpdateDailyFeeMintLimit(19.05e18)` reverts with `DailyFeeMintLimitExceeded`.
6. `updateRSETHPrice()` reverts; `rsETHPrice` is not updated.
7. For the next ~24 hours, all deposits and withdrawals use the stale price; the 20 ETH fee is permanently lost; the auto-pause cannot fire. [2](#0-1) [10](#0-9) [11](#0-10)

    function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
        // Reset the period if it's unset or a day has passed
        if (block.timestamp >= feePeriodStartTime + 1 days) {
            currentPeriodMintedFeeAmount = 0;
            feePeriodStartTime = getCurrentPeriodStartTime();
        }

        // Check if minting would exceed the daily limit
        if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
            revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
        }

        currentPeriodMintedFeeAmount += feeAmount;
    }
```
