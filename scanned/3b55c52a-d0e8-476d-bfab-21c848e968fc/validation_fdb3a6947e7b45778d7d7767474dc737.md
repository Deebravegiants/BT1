### Title
Immutable `rsETHPriceOracle` in Rate Providers Cannot Sync When `LRTOracle` Is Updated in `LRTConfig` - (File: contracts/cross-chain/RSETHMultiChainRateProvider.sol)

### Summary
`RSETHMultiChainRateProvider` and `RSETHRateProvider` cache the `LRTOracle` address as an `immutable` variable set at deployment. `LRTConfig` is the authoritative registry for the `LRTOracle` address and exposes an admin function to update it. Because the rate providers use `immutable`, they can never sync to a new oracle, causing L2 chains to receive stale rsETH/ETH exchange rates while the rest of the L1 protocol uses the updated oracle.

### Finding Description
`RSETHMultiChainRateProvider` stores `rsETHPriceOracle` as `immutable` and reads from it unconditionally:

```solidity
// contracts/cross-chain/RSETHMultiChainRateProvider.sol
address public immutable rsETHPriceOracle;

constructor(address _rsETHPriceOracle, address _layerZeroEndpoint) {
    rsETHPriceOracle = _rsETHPriceOracle;
    ...
}

function getLatestRate() public view override returns (uint256) {
    return ILRTOracle(rsETHPriceOracle).rsETHPrice();
}
``` [1](#0-0) 

`RSETHRateProvider` has the identical pattern: [2](#0-1) 

`LRTConfig` is the protocol's authoritative registry for the `LRTOracle` address. The admin can update it at any time:

```solidity
function setContract(bytes32 contractKey, address contractAddress)
    external onlyRole(DEFAULT_ADMIN_ROLE) {
    _setContract(contractKey, contractAddress);
}
``` [3](#0-2) 

The rest of the L1 protocol reads the oracle address **dynamically** from `LRTConfig` on every call, so it always uses the current oracle:

```solidity
address lrtOracleAddress = lrtConfig.getContract(LRTConstants.LRT_ORACLE);
ILRTOracle lrtOracle = ILRTOracle(lrtOracleAddress);
``` [4](#0-3) 

The rate providers have no setter and no upgrade path for `rsETHPriceOracle`. Once deployed, they are permanently bound to the oracle address given at construction, with no mechanism to sync to a new oracle registered in `LRTConfig`.

### Impact Explanation
When the admin updates `LRTOracle` in `LRTConfig`, the L1 deposit pool immediately uses the new oracle for all L1 minting. However, `RSETHMultiChainRateProvider` and `RSETHRateProvider` continue to push rates derived from the old oracle to every connected L2 chain. L2 pool contracts (`RSETHPoolV3`, `RSETHPoolV2`, etc.) use these pushed rates to calculate how much rsETH to mint for depositors. Users on L2 receive rsETH amounts calculated from a stale or deprecated oracle — either over-minted (protocol insolvency risk) or under-minted (user loss). The contract fails to deliver the promised exchange rate.

**Impact: Low — Contract fails to deliver promised returns; elevated to Medium/Critical if the deprecated oracle diverges significantly from the true price.**

### Likelihood Explanation
The protocol has already undergone multiple upgrade cycles (evidenced by `reinitializer(2)` through `reinitializer(6)` across pool contracts), demonstrating active development. An oracle upgrade — e.g., to add fee minting logic, fix a price calculation bug, or replace a deprecated price feed — is a realistic admin action. The desync is silent: no revert, no event, no on-chain signal that the rate providers are out of sync. The admin must manually remember to redeploy both rate providers and update every `CrossChainRateReceiver` on every L2 chain, a multi-step cross-chain operation that is easy to miss.

### Recommendation
Replace the `immutable` keyword with a mutable state variable and add an owner-gated setter:

```solidity
address public rsETHPriceOracle;

function setRsETHPriceOracle(address _rsETHPriceOracle) external onlyOwner {
    rsETHPriceOracle = _rsETHPriceOracle;
}
```

Alternatively, have the rate providers read the oracle address dynamically from `LRTConfig` on every `getLatestRate()` call, eliminating the cached reference entirely.

### Proof of Concept
1. Admin deploys `RSETHMultiChainRateProvider` with `rsETHPriceOracle = address(oldLRTOracle)`.
2. Admin calls `LRTConfig.setContract(LRTConstants.LRT_ORACLE, address(newLRTOracle))` to upgrade the oracle (e.g., to add fee minting).
3. `LRTDepositPool` now uses `newLRTOracle` for all L1 rsETH minting — correct behavior.
4. `RSETHMultiChainRateProvider.getLatestRate()` still calls `ILRTOracle(oldLRTOracle).rsETHPrice()`.
5. The old oracle's `rsETHPrice` is no longer updated (or diverges from the new oracle's price).
6. Every L2 chain receives stale rates via LayerZero; `CrossChainRateReceiver.rate` reflects the old oracle's last pushed value.
7. An L2 user calls `RSETHPoolV3.deposit()`, which uses the stale rate to compute rsETH amount — receiving more or fewer tokens than the current L1 exchange rate warrants.
