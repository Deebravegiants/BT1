### Title
Mid-Epoch `totalKernelStaked` Drop to Zero Permanently Strands Reward Tokens — (`contracts/KERNEL/KernelDepositPool.sol`)

---

### Summary

`KernelDepositPool` uses the standard Synthetix staking pattern where `rewardPerToken()` returns `rewardPerTokenStored` unchanged whenever `totalKernelStaked == 0`. The guard in `notifyRewardAmount` prevents starting a reward epoch with zero stakers, but there is no on-chain enforcement preventing all stakers from calling `initiateWithdrawal` mid-epoch, which immediately zeroes `totalKernelStaked`. Any `rewardRate × elapsed_zero_staked_seconds` worth of reward tokens then accumulates in the contract with no recipient and no recovery path.

---

### Finding Description

**`rewardPerToken()` silently skips rewards when `totalKernelStaked == 0`:** [1](#0-0) 

When `totalKernelStaked == 0`, the function returns `rewardPerTokenStored` unchanged. The `updateReward` modifier still advances `updatedAt` to `lastTimeRewardApplicable()`: [2](#0-1) 

This means the elapsed time is consumed (the clock advances) but the corresponding `rewardRate × Δt` rewards are never credited to any `rewardPerTokenStored` increment — they are silently stranded.

**`initiateWithdrawal` immediately decrements `totalKernelStaked`:** [3](#0-2) 

The staked balance is reduced at initiation time, not at claim time. If all stakers call `initiateWithdrawal` during an active reward period, `totalKernelStaked` drops to zero immediately.

**`notifyRewardAmount` guard is insufficient:** [4](#0-3) 

This only prevents starting an epoch with zero stakers. It does not prevent the total from reaching zero mid-epoch.

**No recovery function exists.** The contract has no admin rescue path for stranded reward tokens. The last admin function is `setMaxNumberOfWithdrawalsPerUser` at line 610, and there is no `recoverERC20` or equivalent. [5](#0-4) 

**The contract itself acknowledges this limitation in its NatSpec, but only as an off-chain operational commitment:** [6](#0-5) 

The comment says the team will "ensure there are always some tokens staked... for the entire duration of the reward period," but this is not enforced on-chain.

---

### Impact Explanation

Rewards for the zero-staked window — `rewardRate × N_zero_seconds` — are permanently locked in the contract. No staker can ever claim them, and no admin function can recover them. The contract fails to deliver the promised reward distribution for that window. Reward tokens are not lost from the system (the contract holds them), but they are permanently unclaimable, matching **Low: Contract fails to deliver promised returns, but doesn't lose value**.

---

### Likelihood Explanation

Any combination of user actions that drives `totalKernelStaked` to zero mid-epoch triggers this. It does not require malicious intent — normal user behavior (all current stakers deciding to exit during a reward period) is sufficient. The `withdrawalDelay` does not help because `totalKernelStaked` is decremented at `initiateWithdrawal` time, not at `claimWithdrawal` time.

---

### Recommendation

1. **Add a recovery function** so the admin can reclaim stranded reward tokens after an epoch ends:
   ```solidity
   function recoverStrandedRewards() external onlyRole(DEFAULT_ADMIN_ROLE) {
       require(block.timestamp > finishAt, "Epoch not finished");
       uint256 stranded = rewardsToken.balanceOf(address(this)) - totalAccruedButUnclaimed();
       rewardsToken.safeTransfer(msg.sender, stranded);
   }
   ```
2. **Or enforce the invariant on-chain** by reverting `initiateWithdrawal` if it would drop `totalKernelStaked` to zero during an active reward period (`block.timestamp < finishAt`).
3. **Or carry forward** unallocated rewards into the next epoch inside `notifyRewardAmount` by comparing `rewardsToken.balanceOf(address(this))` against the expected distributed amount.

---

### Proof of Concept

```solidity
// 1. Admin calls notifyRewardAmount(1000e18) with 1 staker (Alice, 100 KERNEL)
//    rewardRate = 1000e18 / duration

// 2. Alice calls initiateWithdrawal(100) → totalKernelStaked = 0

// 3. Warp N seconds (totalKernelStaked == 0 the whole time)
//    rewardPerToken() returns rewardPerTokenStored unchanged each call
//    updatedAt advances, consuming N seconds of the epoch

// 4. Bob stakes 100 KERNEL → totalKernelStaked = 100
//    updateReward runs: rewardPerToken() still returns rewardPerTokenStored
//    (because updatedAt was already advanced past the zero window)

// 5. Warp to finishAt

// 6. Bob calls getReward() → receives rewardRate * (duration - N) / 100 * 100
//    Alice calls getReward() → receives rewardRate * 0 (she had zero balance after step 2)

// Total claimed = rewardRate * (duration - N)
// Total deposited = rewardRate * duration
// Stranded = rewardRate * N  ← permanently locked, no recovery path
```
