### Title
Missing `l1VaultETHForL2Chain` Zero-Address Guard in `bridgeTokens` Allows Permanent Token Loss - (File: `contracts/pools/RSETHPoolNoWrapper.sol`)

---

### Summary

`RSETHPoolNoWrapper.bridgeTokens()` passes `l1VaultETHForL2Chain` directly as the L1 recipient to `IL2TokenBridge.bridgeTokenToL1()` without first verifying it is non-zero. If `reinitialize(2)` has not yet been executed, `l1VaultETHForL2Chain` retains its default value of `address(0)`, and a `BRIDGER_ROLE` caller invoking `bridgeTokens()` will irrecoverably send all pooled tokens to the zero address on L1.

---

### Finding Description

`bridgeTokens` in `RSETHPoolNoWrapper.sol` performs three guards before bridging:

1. `tokenBridge[token] != address(0)` — reverts with `MissingBridgeForToken` [1](#0-0) 
2. `balance > 0` — reverts with `ZeroBridgeAmount` [2](#0-1) 
3. Approves and calls the bridge [3](#0-2) 

There is **no guard** on `l1VaultETHForL2Chain`. The call at line 517 is:

```solidity
IL2TokenBridge(tokenBridge[token]).bridgeTokenToL1{ value: msg.value }(l1VaultETHForL2Chain, balance);
``` [4](#0-3) 

`l1VaultETHForL2Chain` is only set during `reinitialize(address,address,uint32)` (reinitializer version 2): [5](#0-4) 

Before that reinitializer is executed, the storage slot holds `address(0)` (Solidity default). The setter `setL1VaultETHForL2Chain` prevents resetting it to zero, but provides no protection against the initial unset state. [6](#0-5) 

**The inconsistency is explicit in the same contract.** The sibling function `bridgeAssetsViaNativeBridge()` correctly guards against this exact condition:

```solidity
UtilLib.checkNonZeroAddress(l1VaultETHForL2Chain);
``` [7](#0-6) 

`bridgeTokens` omits this check entirely.

---

### Impact Explanation

If `bridgeTokens` is called while `l1VaultETHForL2Chain == address(0)`, the bridge contract receives `address(0)` as the L1 recipient. Tokens are transferred out of the pool and credited to the zero address on L1 — an address no one controls. The loss is permanent and unrecoverable. All token balances held in the pool at the time of the call are destroyed.

**Impact: Critical — Permanent freezing/loss of funds.**

---

### Likelihood Explanation

The scenario is reachable during any deployment window between `initialize()` and `reinitialize(2)`. Specifically:

1. Proxy is deployed; `initialize()` is called — `l1VaultETHForL2Chain` is `address(0)`.
2. `addSupportedToken(token, oracle, bridge)` is called by `TIMELOCK_ROLE` — token and bridge are live.
3. Users deposit tokens into the pool.
4. `reinitialize(2)` has not yet been executed (e.g., delayed governance, multi-step upgrade, or operational oversight).
5. `BRIDGER_ROLE` calls `bridgeTokens(token)` — all guards pass, tokens are sent to `address(0)`.

No malicious intent is required. A legitimate `BRIDGER_ROLE` operator acting in good faith, unaware that `reinitialize(2)` has not been called, triggers the loss. The contract provides no revert to warn them.

---

### Recommendation

Add the same zero-address guard used in `bridgeAssetsViaNativeBridge` at the top of `bridgeTokens`:

```solidity
function bridgeTokens(address token) external payable nonReentrant onlySupportedToken(token) onlyRole(BRIDGER_ROLE) {
+   UtilLib.checkNonZeroAddress(l1VaultETHForL2Chain);
    if (tokenBridge[token] == address(0)) revert MissingBridgeForToken();
    ...
}
```

The same fix should be applied to the identical pattern in `RSETHPool.sol` at line 567 and `RSETHPoolV3ExternalBridge.sol` at line 737. [8](#0-7) [9](#0-8)
