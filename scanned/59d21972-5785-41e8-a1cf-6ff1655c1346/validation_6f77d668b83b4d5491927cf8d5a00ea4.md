### Title
Uninitialized `maxNumberOfWithdrawalsPerUser` Permanently Freezes All Staked KERNEL — (`contracts/KERNEL/KernelDepositPool.sol`)

---

### Summary

`maxNumberOfWithdrawalsPerUser` is never set in `initialize()`, so it defaults to `0`. The guard in `initiateWithdrawal` evaluates `0 >= 0 = true` on every call, permanently reverting with `WithdrawalLimitReached` until an admin explicitly calls `setMaxNumberOfWithdrawalsPerUser`. If that call is omitted, all staked KERNEL is permanently frozen.

---

### Finding Description

`initialize()` sets `kernelToken`, `rewardsToken`, and roles, but never assigns `maxNumberOfWithdrawalsPerUser`: [1](#0-0) 

The state variable therefore holds its Solidity default of `0`: [2](#0-1) 

`initiateWithdrawal` checks:

```solidity
if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();
``` [3](#0-2) 

On the very first call, `userWithdrawalIds[msg.sender].length == 0` and `maxNumberOfWithdrawalsPerUser == 0`, so `0 >= 0` is `true` and the function always reverts. There is no fallback withdrawal path; `claimWithdrawal` requires a prior successful `initiateWithdrawal`.

The setter enforces a non-zero value, so it cannot restore the zero state once set — but it also cannot be called with `0` to reproduce the broken state intentionally: [4](#0-3) 

---

### Impact Explanation

**Correct scope: Critical — Permanent freezing of funds.**

The stated scope in the question ("High. Theft of unclaimed yield") is inaccurate for two reasons:

1. **Staked KERNEL is permanently frozen**: no user can ever call `initiateWithdrawal` successfully, so principal is irrecoverable.
2. **Yield is NOT frozen**: `getReward()` is independent of `maxNumberOfWithdrawalsPerUser` and can be called at any time. [5](#0-4) 

The actual impact is **Critical — Permanent freezing of all staked KERNEL** for every user, not theft of unclaimed yield.

---

### Likelihood Explanation

Likelihood is **medium-high**. The contract is deployable and stakeable immediately after `initialize()`. Nothing in the deployment sequence enforces that `setMaxNumberOfWithdrawalsPerUser` is called first. A single missed deployment step causes a protocol-wide, irreversible fund freeze. This is not admin compromise — it is a missing safe default in the initializer.

---

### Recommendation

Set a safe default in `initialize()`:

```solidity
maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER; // 100
``` [6](#0-5) 

---

### Proof of Concept

```solidity
// Deploy KernelDepositPool via proxy; call initialize() but NOT setMaxNumberOfWithdrawalsPerUser
// kernelToken.approve(pool, 1e18);
pool.stake(1e18);                        // succeeds
pool.initiateWithdrawal(1e18);           // reverts: WithdrawalLimitReached
// assert pool.balanceOf(user) == 0 (deducted on stake, but withdrawal impossible)
// assert pool.rewards(user) > 0 (claimable via getReward, but principal is frozen)
```

The revert is deterministic: `userWithdrawalIds[user].length` (0) `>=` `maxNumberOfWithdrawalsPerUser` (0) is always `true` until the admin calls `setMaxNumberOfWithdrawalsPerUser`. [7](#0-6)

    function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
        if (_amount == 0) revert AmountZero();
        if (balanceOf[msg.sender] < _amount) revert InsufficientStakedBalance();
        if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();

        balanceOf[msg.sender] -= _amount;
        totalKernelStaked -= _amount;

        // Create a withdrawal record
        uint256 withdrawalId = ++withdrawalCounter;
        uint256 unlockTime = block.timestamp + withdrawalDelay;

        withdrawals[withdrawalId] = Withdrawal({
            user: msg.sender, amount: _amount, unlockTime: unlockTime, claimed: false, withdrawalId: withdrawalId
        });
        userWithdrawalIds[msg.sender].push(withdrawalId);

        emit WithdrawalInitiated(msg.sender, _amount, withdrawalId, unlockTime);
    }
```
