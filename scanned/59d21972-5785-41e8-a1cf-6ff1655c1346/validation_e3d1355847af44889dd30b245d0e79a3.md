### Title
Stale Chainlink Price Accepted Without Validation in `ChainlinkPriceOracle` Enables Over-Minting of rsETH - (File: contracts/oracles/ChainlinkPriceOracle.sol)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards every return value except `price`. It never checks `updatedAt` for staleness, never verifies `answeredInRound >= roundId`, and never asserts `price > 0`. When a Chainlink feed stops updating while the real LST price has fallen, the stale (inflated) price is fed into `LRTDepositPool.getRsETHAmountToMint()`, allowing a depositor to receive more rsETH than the true value of their deposit warrants, directly diluting every existing rsETH holder.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` is the sole price source for LSTs that use Chainlink feeds (e.g. stETH/ETH):

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol  lines 49-55
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();   // updatedAt, answeredInRound silently discarded
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

The returned `updatedAt` timestamp and `answeredInRound` are silently discarded. No staleness window (e.g. `block.timestamp - updatedAt > heartbeat`) is enforced, and no `price > 0` guard exists.

This price is consumed by two critical paths:

**Path 1 – rsETH minting in `LRTDepositPool`:**

```solidity
// contracts/LRTDepositPool.sol  line 520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.getAssetPrice(asset)` delegates to `ChainlinkPriceOracle.getAssetPrice()`. `lrtOracle.rsETHPrice()` is a **stored** value updated only when `updateRSETHPrice()` is called.

**Path 2 – rsETH price update in `LRTOracle._updateRsETHPrice()`:**

```solidity
// contracts/LRTOracle.sol  lines 339-343
uint256 assetER = getAssetPrice(asset);          // stale price used here too
uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
```

**Attack window**: Between the moment a Chainlink feed goes stale (real price has dropped) and the moment `updateRSETHPrice()` is called to refresh the stored `rsETHPrice`, the numerator `getAssetPrice(asset)` returns the stale inflated price while the denominator `rsETHPrice` still reflects the old correct (lower) price. The ratio is therefore inflated, and the depositor receives more rsETH than their deposit is worth.

---

### Impact Explanation

**Impact: High — Theft of unclaimed yield / dilution of existing rsETH holders.**

Every rsETH token represents a proportional claim on the protocol's total ETH TVL. When a depositor mints rsETH at an inflated price, they acquire a larger share of the TVL than they contributed. All pre-existing rsETH holders are diluted: their tokens now represent a smaller fraction of the same TVL. The attacker can later redeem (via `LRTWithdrawalManager`) for more ETH than they deposited. In a severe depeg scenario (e.g. stETH depegs significantly while the feed is stale), the magnitude could reach Critical — direct theft of funds from existing holders.

---

### Likelihood Explanation

**Likelihood: Medium.**

Chainlink feeds have defined heartbeat intervals (e.g. 24 hours for stETH/ETH on mainnet). During low-volatility periods the feed may not update for the full heartbeat window. Additionally, as the external reference report explicitly notes, a block-stuffing attack can prevent Chainlink keepers from submitting updates before the heartbeat expires, making the stale-price window attacker-controllable. No special privilege is required to exploit the window once it exists: any address can call `LRTDepositPool.depositAsset()`.

---

### Recommendation

Add staleness and validity checks inside `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound)
    = priceFeed.latestRoundData();

require(price > 0, "Invalid price");
require(answeredInRound >= roundId, "Stale round");
require(block.timestamp - updatedAt <= STALENESS_THRESHOLD, "Stale price");
```

`STALENESS_THRESHOLD` should be set per-feed based on its documented heartbeat (e.g. 25 hours for a 24-hour heartbeat feed). Store it in a mapping alongside `assetPriceFeed`.

---

### Proof of Concept

1. Chainlink stETH/ETH feed last updated at `T=0` with price `1.00 ETH`. Heartbeat is 24 h.
2. At `T=12h`, real stETH price drops to `0.95 ETH` (5% depeg). The feed has not updated yet (within heartbeat).
3. `LRTOracle.rsETHPrice` was last set at `T=0` when stETH was `1.00 ETH`.
4. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
5. `getRsETHAmountToMint` computes: `(1000e18 * 1.00e18) / rsETHPrice` — using the stale `1.00 ETH` price instead of the real `0.95 ETH`.
6. Attacker receives `~52.6` extra rsETH (≈5% over-mint on 1000 stETH deposit) compared to what the real price would yield.
7. Each existing rsETH holder's claim on the TVL is proportionally reduced.
8. When the feed eventually updates and `updateRSETHPrice()` is called, `rsETHPrice` drops to reflect the real TVL, but the attacker's over-minted rsETH remains outstanding — the loss is borne by existing holders. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
