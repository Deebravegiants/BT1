### Title
Hardcoded 12-second block time assumption in withdrawal delay enforcement allows premature withdrawals if Ethereum block time changes - (File: contracts/LRTWithdrawalManager.sol)

---

### Summary
`LRTWithdrawalManager` encodes the Ethereum mainnet 12-second block time as a compile-time constant in two places: the default initialization of `withdrawalDelayBlocks` and the hard upper-limit guard in `setWithdrawalDelayBlocks`. Because the upper-limit guard is baked into bytecode, it cannot be updated without a contract upgrade. If Ethereum's block time decreases (as it has historically), the real-time duration enforced by `withdrawalDelayBlocks` shrinks proportionally, allowing users to complete withdrawals sooner than the protocol intends and preventing the manager from restoring the intended delay without a full upgrade.

---

### Finding Description

`LRTWithdrawalManager.initialize` sets the default withdrawal delay:

```solidity
withdrawalDelayBlocks = 8 days / 12 seconds;   // line 94 → 57,600 blocks
```

`setWithdrawalDelayBlocks` enforces a hard ceiling:

```solidity
if (withdrawalDelayBlocks_ > 16 days / 12 seconds) revert ExceedWithdrawalDelay();  // line 340 → 115,200 blocks
```

Both values assume exactly 12 s per block. The delay is enforced in two places using `block.number`:

- `_addUserWithdrawalRequest` (line 752): records `withdrawalStartBlock: block.number`
- `_processWithdrawalCompletion` (line 715): `if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();`
- `_unlockWithdrawalRequests` (line 795): same check gates the operator unlock loop

The upper-limit constant `16 days / 12 seconds` is evaluated at compile time and embedded in bytecode. Unlike `withdrawalDelayBlocks` (which the manager can update), this ceiling cannot be adjusted without redeploying the contract.

If Ethereum's average block time decreases — as it did when transitioning from PoW (≈13 s) to PoS (≈12 s), and as is plausible with future protocol changes — the real-time duration represented by any given block count shrinks proportionally:

| Block time | 57,600 blocks (default) | 115,200 blocks (ceiling) |
|---|---|---|
| 12 s (current) | 8 days | 16 days |
| 6 s (hypothetical) | 4 days | 8 days |
| 3 s (hypothetical) | 2 days | 4 days |

At 6 s blocks the manager cannot set a delay longer than 8 real-time days even if the protocol requires 16, because the ceiling check rejects any value above 115,200 blocks.

---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

The withdrawal delay is a security mechanism intended to give the protocol time to detect and respond to anomalies before funds leave. If block times decrease, the effective delay shrinks without any action by the manager, and the hardcoded ceiling prevents the manager from compensating by raising `withdrawalDelayBlocks` beyond the now-shorter ceiling. Users can call `completeWithdrawal` and receive their assets sooner than the protocol's security model intends. No direct fund theft occurs, but the protocol's promised minimum holding period is silently violated.

---

### Likelihood Explanation

Ethereum block time changes are infrequent but precedented (13 s → 12 s at the Merge). Future proposals (e.g., single-slot finality, faster slot times) could reduce it further. The risk is not theoretical: the exact same class of change has already occurred once on Ethereum mainnet. Because the ceiling is immutable in bytecode, no operational response is possible short of a contract upgrade.

---

### Recommendation

1. Replace the compile-time constant in `setWithdrawalDelayBlocks` with a configurable state variable for the maximum delay, updatable by the admin:
   ```solidity
   uint256 public maxWithdrawalDelayBlocks;
   ```
2. Provide an admin function to update `maxWithdrawalDelayBlocks` when block times change, analogous to how the report recommends making `_K` configurable.
3. Consider emitting an event whenever `withdrawalDelayBlocks` is updated so off-chain monitors can detect unexpected changes.

---

### Proof of Concept

1. Ethereum block time decreases to 6 s (e.g., via a future protocol upgrade).
2. The manager attempts to restore the intended 8-day delay by calling `setWithdrawalDelayBlocks(8 days / 6 seconds)` = `setWithdrawalDelayBlocks(115_200)`.
3. The call reverts: `115_200 > 16 days / 12 seconds` → `115_200 > 115_200` is false — this particular value passes, but any value representing more than 8 real-time days at 6 s blocks (i.e., > 115,200 blocks) is rejected.
4. The manager is therefore capped at a maximum real-time delay of 8 days instead of the intended 16 days, and the default 57,600-block delay now represents only 4 days.
5. A user who called `initiateWithdrawal` can call `completeWithdrawal` after only ~4 real-time days instead of the intended 8, bypassing the protocol's intended security window.
