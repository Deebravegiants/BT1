### Title
Zero `tokenFeeBps` Default Allows Feeless Token Swaps for Newly Added Tokens - (File: contracts/pools/RSETHPool.sol)

---

### Summary

`RSETHPool.sol` (the Arbitrum L2 pool) maintains a per-token fee mapping `tokenFeeBps[token]`. When a new token is added via `addSupportedToken()`, this mapping is never initialized and defaults to `0`. Any user who deposits that token before the admin separately calls `setTokenFeeBps()` receives rsETH with zero protocol fee deducted — a feeless swap. This is a direct analog to the external report's root cause: a special protocol state (empty pool / newly added token) where fee enforcement is absent.

---

### Finding Description

`RSETHPool.sol` introduces per-token fee basis points via the `tokenFeeBps` mapping:

```solidity
mapping(address token => uint256 feeBps) public tokenFeeBps;
```

The fee for a token deposit is computed in `viewSwapRsETHAmountAndFee`:

```solidity
uint256 feeBpsForToken = tokenFeeBps[token];
fee = amount * feeBpsForToken / 10_000;
uint256 amountAfterFee = amount - fee;
...
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [1](#0-0) 

When a new token is registered via `addSupportedToken()`, the function sets the oracle and bridge but **never sets `tokenFeeBps[token]`**:

```solidity
supportedTokenList.push(token);
supportedTokenOracle[token] = oracle;
tokenBridge[token] = bridge;
// tokenFeeBps[token] is never set → defaults to 0
``` [2](#0-1) 

Setting the fee requires a separate, independent admin call to `setTokenFeeBps()`: [3](#0-2) 

Because `tokenFeeBps[token]` is `0` by default, every deposit of the newly added token before `setTokenFeeBps()` is called computes `fee = 0`, and the depositor receives the full rsETH amount with no fee withheld. The `deposit(address token, ...)` function passes this zero fee directly to `feeEarnedInToken[token] += 0`, so no protocol revenue is recorded. [4](#0-3) 

---

### Impact Explanation

Every token deposit made in the window between `addSupportedToken()` and `setTokenFeeBps()` pays zero protocol fee. The attacker receives more rsETH than they are entitled to (the full token value converted at oracle rate, with no fee deducted), while the protocol collects nothing. This constitutes **theft of unclaimed yield** — protocol fee revenue that should have been collected is permanently lost for all deposits in that window.

**Impact: High — Theft of unclaimed yield.**

---

### Likelihood Explanation

This condition is triggered every time a new ERC-20 token is added to the Arbitrum `RSETHPool`. The two governance actions (`addSupportedToken` via `TIMELOCK_ROLE` and `setTokenFeeBps` via `DEFAULT_ADMIN_ROLE`) are separate transactions with no atomicity guarantee. A mempool observer can detect the `AddSupportedToken` event and immediately front-run or race the `setTokenFeeBps` call with a large deposit, extracting the full fee amount. No special privileges are required — any user can call `deposit(token, amount, referralId)` once the token is listed.

**Likelihood: High** — the window is created by normal protocol operation (adding tokens) and is trivially exploitable by any depositor watching on-chain events.

---

### Recommendation

Initialize `tokenFeeBps[token]` atomically inside `addSupportedToken()` by accepting a `_feeBps` parameter:

```solidity
function addSupportedToken(
    address token,
    address oracle,
    address bridge,
    uint256 _feeBps          // ← add this
) external onlyRole(TIMELOCK_ROLE) {
    ...
    if (_feeBps > 10_000) revert InvalidFeeAmount();
    supportedTokenList.push(token);
    supportedTokenOracle[token] = oracle;
    tokenBridge[token] = bridge;
    tokenFeeBps[token] = _feeBps;   // ← set atomically
    emit AddSupportedToken(token, oracle, bridge);
}
```

This eliminates the race window entirely by ensuring the fee is set in the same transaction that enables deposits for the token.

---

### Proof of Concept

1. Admin calls `addSupportedToken(wstETH, oracle, bridge)` on `RSETHPool` (Arbitrum). `tokenFeeBps[wstETH]` is `0`.
2. Before admin calls `setTokenFeeBps(wstETH, 50)` (0.5% fee), attacker calls:
   ```solidity
   RSETHPool.deposit(wstETH, 1_000_000e18, "");
   ```
3. Inside `viewSwapRsETHAmountAndFee(1_000_000e18, wstETH)`:
   - `feeBpsForToken = tokenFeeBps[wstETH] = 0`
   - `fee = 1_000_000e18 * 0 / 10_000 = 0`
   - `amountAfterFee = 1_000_000e18`
   - `rsETHAmount = 1_000_000e18 * tokenToETHRate / rsETHToETHrate` (full value, no fee)
4. Attacker receives rsETH equivalent to the full 1,000,000 wstETH value. At 0.5% fee, the protocol loses 5,000 wstETH worth of fee revenue on this single deposit.
5. Admin's subsequent `setTokenFeeBps(wstETH, 50)` call is too late — the feeless swap already occurred. [5](#0-4) [6](#0-5)
