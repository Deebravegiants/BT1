### Title
Unguarded `CREATE3Factory.deploy()` Allows Salt Hijacking to Deploy Malicious Contracts at Protocol-Expected Addresses — (File: contracts/utils/CREATE3Factory.sol)

---

### Summary

The `CREATE3Factory` contract exposes a public `deploy(bytes32 salt, bytes memory creationCode)` function with no access control and no namespacing of the `salt` by `msg.sender`. Because the deterministic deployment address is derived solely from `address(CREATE3Factory)` and `salt`, any external caller can occupy any salt before the protocol does. An attacker who monitors the mempool (or predicts the salt off-chain) can front-run a legitimate protocol deployment, placing a malicious contract at the address the protocol and its users expect to be the authentic contract.

---

### Finding Description

`CREATE3Factory.deploy()` delegates directly to `CREATE3.deploy(salt, creationCode, msg.value)`:

```solidity
// contracts/utils/CREATE3Factory.sol
function deploy(bytes32 salt, bytes memory creationCode) external payable returns (address deployed) {
    deployed = CREATE3.deploy(salt, creationCode, msg.value);
    emit ContractDeployed(salt, deployed);
}
```

Inside `CREATE3.deploy`, the proxy is created with:

```solidity
proxy := create2(0, add(proxyChildBytecode, 32), mload(proxyChildBytecode), salt)
```

The resulting address is:

```
proxy  = keccak256(0xFF ++ address(CREATE3Factory) ++ salt ++ PROXY_BYTECODE_HASH)
target = keccak256(0xd694 ++ proxy ++ 0x01)
```

The `salt` is used verbatim — it is **not** hashed together with `msg.sender`. There is no `onlyOwner`, no role check, and no salt-reservation mechanism. Any EOA or contract can call `deploy` with an arbitrary salt at any time.

---

### Impact Explanation

The protocol uses `CREATE3Factory` to deploy contracts at deterministic, cross-chain-consistent addresses (the canonical use-case for CREATE3). If an attacker deploys a malicious contract at the salt the protocol intends to use before the protocol does:

1. The protocol's own deployment transaction reverts (`DEPLOYMENT_FAILED` — the proxy address is already occupied).
2. The malicious contract sits at the address that off-chain tooling, other protocol contracts, or users already trust.
3. Any user or contract that sends funds to, or calls, the expected address interacts with the attacker's contract instead.

Depending on which contract is being deployed (L2 pool, bridge receiver, token wrapper, rate provider), the attacker can drain deposited ETH/LSTs, mint unbacked rsETH/wrsETH, or silently discard bridged assets.

**Impact: Critical** — direct theft of user funds in motion (deposits, bridge receipts) or at rest (pool balances).

---

### Likelihood Explanation

- The salt is typically a human-readable string (e.g., `keccak256("RSETHPoolV3")`) or a known constant, making it predictable from public documentation or prior deployments on other chains.
- The attack requires only a standard front-run on the deployment transaction, which is straightforward on any EVM chain with a public mempool.
- The attacker needs no special role, no governance power, and no existing protocol interaction — a single `deploy()` call suffices.

**Likelihood: Medium** — requires mempool visibility or salt prediction, both of which are realistic for a motivated attacker targeting a high-value deployment.

---

### Recommendation

Namespace the salt by `msg.sender` inside `CREATE3Factory.deploy()`:

```solidity
function deploy(bytes32 salt, bytes memory creationCode) external payable returns (address deployed) {
    bytes32 guardedSalt = keccak256(abi.encodePacked(msg.sender, salt));
    deployed = CREATE3.deploy(guardedSalt, creationCode, msg.value);
    emit ContractDeployed(salt, deployed);
}
```

This ensures that only the original deployer can occupy a given logical salt, mirroring the "Universal Links" recommendation in the source report (binding the identifier to a sole owner). Alternatively, add an `onlyOwner` / role guard so only the protocol deployer can call `deploy`.

---

### Proof of Concept

1. Protocol announces it will deploy `RSETHPoolV3` via `CREATE3Factory` with `salt = keccak256("RSETHPoolV3")` on a new L2.
2. Attacker calls:
   ```solidity
   CREATE3Factory.deploy(
       keccak256("RSETHPoolV3"),
       type(MaliciousPool).creationCode   // drains all deposited ETH to attacker
   );
   ```
3. `MaliciousPool` is now live at the address `getDeployed(keccak256("RSETHPoolV3"))`.
4. Protocol's own deployment reverts with `DEPLOYMENT_FAILED`.
5. Users who deposit ETH into the announced address interact with `MaliciousPool`; their funds are stolen.

```text
    function deploy(bytes32 salt, bytes memory creationCode, uint256 value) internal returns (address deployed) {
        bytes memory proxyChildBytecode = PROXY_BYTECODE;

        address proxy;
        assembly {
            // Deploy a new contract with our pre-made bytecode via CREATE2.
            // We start 32 bytes into the code to avoid copying the byte length.
            proxy := create2(0, add(proxyChildBytecode, 32), mload(proxyChildBytecode), salt)
        }
        require(proxy != address(0), "DEPLOYMENT_FAILED");

        deployed = getDeployed(salt);
        (bool success,) = proxy.call{ value: value }(creationCode);
        require(success && deployed.code.length != 0, "INITIALIZATION_FAILED");
```

```text
    function getDeployed(bytes32 salt) internal view returns (address) {
        address proxy = keccak256(abi.encodePacked(bytes1(0xFF), address(this), salt, PROXY_BYTECODE_HASH)).
            // Prefix:
            // Creator:
            // Salt:
            // Bytecode hash:
            fromLast20Bytes();

        return keccak256(abi.encodePacked(hex"d694", proxy, hex"01")) // Nonce of the proxy contract (1)
            // 0xd6 = 0xc0 (short RLP prefix) + 0x16 (length of: 0x94 ++ proxy ++ 0x01)
            // 0x94 = 0x80 + 0x14 (0x14 = the length of an address, 20 bytes, in hex)
            .fromLast20Bytes();
    }
```
