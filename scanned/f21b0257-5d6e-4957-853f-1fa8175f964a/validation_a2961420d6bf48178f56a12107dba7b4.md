### Title
Missing `__gap` Storage Variable in Upgradeable Base Contract `LRTConfigRoleChecker` Enables Silent Storage Collision on Future Upgrades - (File: contracts/utils/LRTConfigRoleChecker.sol)

### Summary

`LRTConfigRoleChecker` is an abstract base contract inherited by every core protocol contract (`LRTDepositPool`, `NodeDelegator`, `RSETH`, `LRTOracle`, `LRTWithdrawalManager`, `LRTUnstakingVault`). It declares one state variable (`lrtConfig`) but contains no `__gap` reserved storage array. None of the concrete inheriting contracts declare a `__gap` either. This means any future upgrade that inserts a new state variable into `LRTConfigRoleChecker` will silently shift the storage layout of all six inheriting contracts, corrupting every state variable they hold.

This demonstrates the protocol regularly performs upgrades. If any future upgrade adds a state variable to `LRTConfigRoleChecker`, the Solidity compiler will silently shift every downstream storage slot in all six inheriting contracts. The proxy storage retains the old layout, so the new implementation reads wrong values from every slot.

### Impact Explanation

If a new variable is prepended or inserted into `LRTConfigRoleChecker`, `lrtConfig` shifts from slot 0 to slot 1. The proxy storage still holds the old `lrtConfig` address at slot 0 and the first concrete contract variable (e.g., `maxNodeDelegatorLimit` in `LRTDepositPool`, a `uint256`) at slot 1. The new implementation reads slot 1 as `lrtConfig`, interpreting a numeric value as an address. This corrupts:

- All role checks (`onlyLRTAdmin`, `onlyLRTManager`, `onlyLRTOperator`) across all six contracts — access control is bypassed or permanently broken.
- Oracle price reads in `LRTOracle` and `LRTDepositPool.getRsETHAmountToMint`, causing incorrect rsETH minting amounts.
- Withdrawal processing in `LRTWithdrawalManager` and `LRTUnstakingVault`, potentially freezing user funds.
- The `RSETH` mint/burn role checks, enabling unauthorized minting or blocking legitimate minting.

The impact is **temporary or permanent freezing of funds** and potential **theft** depending on which slot is misread as `lrtConfig`.

### Likelihood Explanation

The likelihood is **Low** in isolation — it requires a future upgrade that adds a variable to `LRTConfigRoleChecker`. However, the protocol's demonstrated pattern of multiple reinitializations across contracts, combined with the absence of any `__gap` guard and no automated storage layout validation tooling visible in the repository, makes this a realistic risk over the protocol's upgrade lifecycle. The Solidity compiler produces no warning for this class of error.

### Recommendation

1. Add a `__gap` array to `LRTConfigRoleChecker` sized to reserve 49 slots (50 total including `lrtConfig`):

```solidity
uint256[49] private __gap;
```

2. Add `__gap` arrays to each concrete upgradeable contract (`LRTDepositPool`, `NodeDelegator`, `RSETH`, `LRTOracle`, `LRTWithdrawalManager`, `LRTUnstakingVault`, `L1Vault`, `L1VaultV2`) sized so that the total slots used by the contract (inherited + own variables + gap) equals a fixed round number (e.g., 50).

3. Integrate OpenZeppelin's `upgrades` plugin storage layout checks into the CI pipeline to catch layout incompatibilities before deployment.

### Proof of Concept

**Current storage layout of `LRTDepositPool` (simplified):**

| Slot | Variable | Contract |
|------|----------|----------|
| 0 | `lrtConfig` | `LRTConfigRoleChecker` |
| 1 | `_paused` (packed) | `PausableUpgradeable` |
| ... | `__gap[49]` | `PausableUpgradeable` |
| 51 | `_status` | `ReentrancyGuardUpgradeable` |
| ... | `__gap[49]` | `ReentrancyGuardUpgradeable` |
| 101 | `maxNodeDelegatorLimit` | `LRTDepositPool` |
| 102 | `minAmountToDeposit` | `LRTDepositPool` |
| 103 | `isNodeDelegator` (mapping) | `LRTDepositPool` |
| 104 | `nodeDelegatorQueue` (array) | `LRTDepositPool` |
| 105 | `maxNegligibleAmount` | `LRTDepositPool` |

**After upgrade adding `address newVar` to `LRTConfigRoleChecker`:**

| Slot | Variable (new impl reads) | Proxy storage actually holds |
|------|--------------------------|------------------------------|
| 0 | `newVar` | old `lrtConfig` address |
| 1 | `lrtConfig` ← **CORRUPTED** | `_paused` packed bool |

The new implementation reads slot 1 (which holds `_paused`, a packed bool from `PausableUpgradeable`) as the `lrtConfig` address. Every subsequent call to `onlyLRTAdmin`, `onlyLRTManager`, `getRsETHAmountToMint`, etc. uses a garbage address, breaking all protocol operations. No attacker action is required beyond the upgrade itself — the corruption is automatic and silent. [1](#0-0) [12](#0-11)
