### Title
`ethValueInWithdrawal` Accrues Phantom ETH Value on Round-Trip Asset Transfers at Different Oracle Prices — (`contracts/LRTConverter.sol`)

---

### Summary

`LRTConverter.transferAssetFromDepositPool` records the ETH value of an asset at the oracle price at call time, while `transferAssetToDepositPool` subtracts the ETH value at the oracle price at *its* call time. When the oracle price falls between the two calls, a permanent positive residual accumulates in `ethValueInWithdrawal`, inflating the protocol's reported TVL and therefore `rsETHPrice`.

---

### Finding Description

`transferAssetFromDepositPool` adds to `ethValueInWithdrawal` using the spot oracle price at the moment of the call: [1](#0-0) 

`transferAssetToDepositPool` subtracts from `ethValueInWithdrawal` using the spot oracle price at the moment of *that* call: [2](#0-1) 

Because the two calls happen at different times, any price movement between them creates a permanent discrepancy. If the price was `P_high` on entry and `P_normal < P_high` on return, the residual `amount * (P_high − P_normal) / 1e18` is never cleared.

`ethValueInWithdrawal` is consumed directly as `ethLyingInConverter` in `getETHDistributionData`: [3](#0-2) 

This feeds into `_getTotalEthInProtocol` in `LRTOracle`, which sums all asset deposits multiplied by their current oracle prices: [4](#0-3) 

The inflated `totalETHInProtocol` is then used to compute `rsETHPrice`: [5](#0-4) 

Note that for non-ETH assets, `assetLyingInConverter` is explicitly zeroed out with the comment "assets in converter are accounted in their eth value => getETHDistributionData()": [6](#0-5) 

This means the phantom residual in `ethValueInWithdrawal` is the *only* accounting for converter assets, and it is never corrected by any subsequent operation.

The `_sendEthToDepositPool` path (used after a successful unstake claim) correctly subtracts the actual ETH received: [7](#0-6) 

But `transferAssetToDepositPool` — the path for returning an asset *without* unstaking — uses the current oracle price, not the original recorded price, creating the mismatch.

---

### Impact Explanation

The residual permanently inflates `ethValueInWithdrawal`, which inflates `totalETHInProtocol`, which inflates `rsETHPrice`. New depositors calling `getRsETHAmountToMint` receive fewer rsETH tokens than they are entitled to for the same ETH value deposited, for the lifetime of the protocol (until a manual correction). No funds are lost from the protocol, but depositors receive less than the promised return. This matches **Low — Contract fails to deliver promised returns, but doesn't lose value**.

---

### Likelihood Explanation

The `ASSET_TRANSFER_ROLE` is a privileged but operationally active role used for routine asset management. The scenario does not require malicious intent: an operator legitimately moves an LST (e.g., ETHx) to the converter for unstaking, the unstaking is cancelled or not needed, and the asset is returned after the oracle price has declined due to normal market volatility. LST oracle prices (e.g., ETHx/ETH) do fluctuate. The residual accumulates silently with no on-chain alert. Repeated round-trips compound the error.

---

### Recommendation

Record the ETH value at the time of `transferAssetFromDepositPool` and subtract that *same recorded value* (not a re-priced value) when `transferAssetToDepositPool` is called. One approach: maintain a per-asset mapping of recorded ETH value sent to the converter, and subtract the proportional recorded value on return rather than re-querying the oracle.

---

### Proof of Concept

```solidity
// Preconditions:
// - operator has ASSET_TRANSFER_ROLE
// - ETHx is a supported asset
// - DepositPool has approved LRTConverter for ETHx

// Step 1: oracle price = P_high = 1.05e18
// operator calls:
lrtConverter.transferAssetFromDepositPool(ETHx, 100e18);
// ethValueInWithdrawal += 100e18 * 1.05e18 / 1e18 = 105e18

// Step 2: oracle price drops to P_normal = 1.00e18 (normal market movement)
// operator calls:
lrtConverter.transferAssetToDepositPool(ETHx, 100e18);
// assetValue = 100e18 * 1.00e18 / 1e18 = 100e18
// ethValueInWithdrawal = 105e18 - 100e18 = 5e18  ← phantom residual

// Step 3: rsETHPrice is now inflated by 5e18 / rsethSupply
// New depositors receive fewer rsETH than entitled.

// Assert: ethValueInWithdrawal == 0 after round-trip at different prices → FAILS
```

    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
