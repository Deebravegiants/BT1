### Title
Missing Timestamp-Based Staleness Check in Chainlink Oracle Allows Stale Rate to Drive wrsETH Minting — (File: `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol`)

---

### Summary

`ChainlinkOracleForRSETHPoolCollateral.getRate()` validates the Chainlink round answer with `answeredInRound < roundID` and `timestamp == 0`, but never checks `block.timestamp - timestamp > maxStaleness`. This is the direct Solidity analog of the external report's trust-boundary violation: an external data source (Chainlink price feed) is consumed without sufficient validation before being used in a critical financial calculation. A stale collateral-token rate flows unguarded into the wrsETH minting formula in `RSETHPoolV3` and `RSETHPoolV3ExternalBridge`, allowing any depositor to mint wrsETH at an incorrect exchange rate.

---

### Finding Description

`ChainlinkOracleForRSETHPoolCollateral.getRate()` calls `latestRoundData()` and performs three checks:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [1](#0-0) 

The `answeredInRound < roundID` guard is the legacy Chainlink staleness check that Chainlink itself has deprecated; it does **not** catch the common case where the feed simply has not been updated for an extended period (e.g., price movement stayed within the deviation threshold, so no new round was triggered). The correct guard — `block.timestamp - timestamp > MAX_STALENESS` — is entirely absent.

This stale rate is then consumed without any further validation in both L2 pool contracts:

```solidity
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [2](#0-1) [3](#0-2) 

The same stale rate also feeds the `limitDailyMint` modifier's pre-check, meaning the daily cap accounting is also computed on the stale price. [4](#0-3) 

---

### Impact Explanation

**Impact: High — Theft of unclaimed yield / over-issuance of wrsETH**

If the collateral token's real market price has dropped since the last Chainlink update (e.g., wstETH/ETH fell 2 % but the feed has not refreshed), `tokenToETHRate` is inflated relative to reality. A depositor calling `deposit(token, amount, referralId)` receives:

```
rsETHAmount = amountAfterFee * (stale, inflated tokenToETHRate) / rsETHToETHrate
```

This mints more wrsETH than the deposited collateral is worth. The excess wrsETH represents a direct extraction of value from existing rsETH holders (dilution of the backing ratio), equivalent to theft of yield.

Conversely, if the stale price is lower than actual, users are under-minted — the contract fails to deliver promised returns.

---

### Likelihood Explanation

**Likelihood: Medium**

Chainlink feeds for LST/ETH pairs (e.g., wstETH/ETH) have heartbeat intervals of 24 hours and deviation thresholds of 0.5 %–1 %. During periods of low volatility the feed can go many hours without an update. Network congestion or sequencer issues on L2s can further delay updates. An attacker monitoring the feed can time a deposit precisely when the stale price is most favorable. No privileged access is required; any EOA can call `deposit`.

---

### Recommendation

Add a configurable maximum staleness check in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
uint256 public constant MAX_STALENESS = 24 hours; // tune per feed heartbeat

function getRate() public view returns (uint256) {
    (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();

    if (answeredInRound < roundID) revert StalePrice();
    if (timestamp == 0) revert IncompleteRound();
    if (block.timestamp - timestamp > MAX_STALENESS) revert StalePrice(); // ← add this
    if (ethPrice <= 0) revert InvalidPrice();
    ...
}
```

Make `MAX_STALENESS` immutable or admin-settable per oracle instance, sized to the specific feed's heartbeat.

---

### Proof of Concept

1. Chainlink wstETH/ETH feed last updated at `T=0` with rate `1.15e18` (wstETH = 1.15 ETH).
2. At `T = 23h`, real wstETH price drops to `1.10 ETH` due to market movement, but the feed has not triggered a new round (price stayed within deviation threshold).
3. Attacker calls `RSETHPoolV3.deposit(wstETH, 100e18, "")`.
4. `viewSwapRsETHAmountAndFee` fetches `tokenToETHRate = 1.15e18` (stale).
5. `rsETHAmount = 100e18 * 1.15e18 / rsETHToETHrate` — attacker receives wrsETH priced as if they deposited 115 ETH worth of collateral, when they only deposited 110 ETH worth.
6. Attacker immediately bridges/redeems, extracting ~4.5 % excess value from the pool at the expense of existing rsETH holders.

No admin access, no governance, no oracle operator compromise required — only a public `deposit` call timed to a stale feed window. [5](#0-4) [6](#0-5) [7](#0-6)
