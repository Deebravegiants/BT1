### Title
ETH Push-Transfer to Non-Payable Contract Permanently Freezes Withdrawal Funds - (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTWithdrawalManager` uses a push-payment pattern to deliver ETH to withdrawing users. Because rsETH is burned in a prior transaction (`unlockQueue`) and the ETH transfer to the user occurs in a later transaction (`completeWithdrawal`), a user whose address is a contract without a `receive()` or `fallback()` function will have their rsETH permanently destroyed while the corresponding ETH becomes irrecoverable inside the withdrawal manager.

---

### Finding Description

The withdrawal lifecycle in `LRTWithdrawalManager` spans three separate transactions:

**Step 1 — `initiateWithdrawal`**: rsETH is pulled from the user into the withdrawal manager. [1](#0-0) 

**Step 2 — `unlockQueue`**: rsETH held by the manager is burned, and the corresponding ETH is redeemed from `LRTUnstakingVault` into the manager. This is irreversible. [2](#0-1) 

**Step 3 — `completeWithdrawal` → `_processWithdrawalCompletion`**: The withdrawal request is deleted and ETH is pushed to the user via `_transferAsset`. [3](#0-2) 

The ETH transfer inside `_transferAsset` is:

```solidity
(bool sent,) = payable(to).call{ value: amount }("");
if (!sent) revert EthTransferFailed();
``` [4](#0-3) 

If `to` is a contract without a `receive()` or `fallback()` function, the low-level call returns `false`, the function reverts, and the withdrawal request is restored in storage. However, the rsETH burned in Step 2 is **not** restored — it was destroyed in a prior, already-finalized transaction. The ETH now sits in the withdrawal manager with no mechanism to deliver it to the user, and no per-user rescue path exists.

The `completeWithdrawalForUser` operator path suffers the same failure: it also calls `_processWithdrawalCompletion(asset, user, referralId)` and pushes ETH to the same non-payable `user` address. [5](#0-4) 

The only sweep mechanism, `sweepRemainingAssets`, is blocked while any unlocked withdrawal exists for the asset, so the stuck ETH cannot be recovered by the manager either. [6](#0-5) 

---

### Impact Explanation

**Critical — Permanent freezing of user funds.**

After `unlockQueue` executes, the user's rsETH is gone. If the ETH push to the user's contract address always reverts, the ETH is permanently locked inside `LRTWithdrawalManager` with no on-chain path to recover it for that user. The user suffers a total loss of their staked position.

---

### Likelihood Explanation

**Low–Medium.** The trigger requires the withdrawing address to be a smart contract without a `receive()` or `fallback()` function. This is realistic for:

- Protocol-owned treasury or DAO contracts that hold rsETH but were not designed to receive raw ETH.
- Automated vault or strategy contracts that deposit into KelpDAO on behalf of users.
- Any contract that calls `initiateWithdrawal` but whose ETH-receive capability was not verified.

No attacker action is required; the freeze is triggered accidentally by the affected user's own contract architecture, matching the "accidentally triggered" scenario described in the reference report.

---

### Recommendation

1. **Pull-payment pattern**: Instead of pushing ETH to the user in `_processWithdrawalCompletion`, record a claimable balance and let the user pull it with a separate `claimETH()` call. This decouples delivery failure from the withdrawal accounting.
2. **Pre-flight check**: Before accepting an ETH withdrawal request in `initiateWithdrawal`, verify that the caller can receive ETH (e.g., check `address(msg.sender).code.length == 0` or require a test transfer).
3. **Admin rescue**: Add a privileged function that can re-route stuck ETH for a specific request ID to an alternate address supplied by the user, callable only after a timeout.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

// A contract with NO receive() function — cannot accept raw ETH pushes
contract NoReceiveWallet {
    function initiateWithdraw(address withdrawalManager, uint256 rsETHAmount) external {
        // Approve rsETH to withdrawal manager first (omitted for brevity)
        ILRTWithdrawalManager(withdrawalManager)
            .initiateWithdrawal(ETH_TOKEN, rsETHAmount, "ref");
    }

    function tryComplete(address withdrawalManager) external {
        // This will ALWAYS revert with EthTransferFailed because
        // this contract has no receive() function.
        ILRTWithdrawalManager(withdrawalManager)
            .completeWithdrawal(ETH_TOKEN, "ref");
    }
}
```

**Attack trace:**

1. `NoReceiveWallet.initiateWithdraw(...)` — rsETH transferred to `LRTWithdrawalManager`. ✓
2. Operator calls `unlockQueue(ETH_TOKEN, ...)` — rsETH **burned**, ETH redeemed into manager. ✓ (irreversible)
3. `NoReceiveWallet.tryComplete(...)` — `_transferAsset` pushes ETH to `NoReceiveWallet`, call returns `false`, reverts with `EthTransferFailed`. Withdrawal request restored in storage, but rsETH is already gone.
4. Step 3 can be retried infinitely; it will always revert. ETH is permanently frozen. [7](#0-6)

        if (userAssociatedNonces[asset][user].empty()) {
            revert NoWithdrawalRequests(user, asset);
        }

        // Retrieve and remove the oldest withdrawal request for the user.
        uint256 usersFirstWithdrawalRequestNonce = userAssociatedNonces[asset][user].popFront();
        // Ensure the request is already unlocked.
        if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();

        bytes32 requestId = getRequestId(asset, usersFirstWithdrawalRequestNonce);
        WithdrawalRequest memory request = withdrawalRequests[requestId];

        delete withdrawalRequests[requestId];

        // Check that the withdrawal delay has passed since the request's initiation.
        if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();

        unlockedWithdrawalsCount[asset]--;

        // If Aave integration is enabled and asset is ETH, withdraw from Aave if needed
        if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
            uint256 contractBalance = address(this).balance;
            if (contractBalance < request.expectedAssetAmount) {
                uint256 amountNeeded = request.expectedAssetAmount - contractBalance;

                uint256 balanceAfter = address(this).balance;
                if (balanceAfter < request.expectedAssetAmount) {
                    revert InsufficientLiquidityForWithdrawal();
                }
            }
        }

        _transferAsset(asset, user, request.expectedAssetAmount);

        emit ReferralIdEmitted(referralId);
        emit AssetWithdrawalFinalized(user, asset, request.rsETHUnstaked, request.expectedAssetAmount);
    }
