### Title
Stale Cross-Chain Oracle Rate Allows Over-Minting of wrsETH on L2 Pools - (File: contracts/cross-chain/CrossChainRateReceiver.sol)

### Summary
The `CrossChainRateReceiver` contract stores a `lastUpdated` timestamp when the rsETH/ETH rate is received from L1 via LayerZero, but `getRate()` returns the stored `rate` unconditionally without any staleness check. All L2 pool variants (`RSETHPoolV3`, `RSETHPoolV2`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`) call `getRate()` on this oracle to compute how many wrsETH tokens to mint per deposited ETH. When the L2 oracle rate lags behind the true L1 rsETH price, depositors receive more wrsETH than the current fair value entitles them to, extracting yield from existing rsETH holders.

### Finding Description

`CrossChainRateReceiver.lzReceive` updates `rate` and `lastUpdated` when a LayerZero message arrives from L1:

```solidity
rate = _rate;
lastUpdated = block.timestamp;
``` [1](#0-0) 

However, `getRate()` returns the stored value with no staleness guard:

```solidity
function getRate() external view returns (uint256) {
    return rate;
}
``` [2](#0-1) 

`RSETHRateReceiver` inherits this implementation directly and is the oracle wired into every L2 pool as `rsETHOracle`. [3](#0-2) 

Every L2 pool's deposit path calls `getRate()` to obtain `rsETHToETHrate` and then divides:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [4](#0-3) 

The same pattern appears in `RSETHPoolV3ExternalBridge` and `RSETHPoolV3WithNativeChainBridge`: [5](#0-4) 

Because rsETH accrues value over time (rewards flow into the L1 protocol and `LRTOracle._updateRsETHPrice` raises `rsETHPrice`), the true L1 rate is monotonically non-decreasing: [6](#0-5) 

If the L2 oracle has not been refreshed, `rsETHToETHrate` is lower than the current L1 value. A lower denominator yields a larger `rsETHAmount`, so the depositor receives more wrsETH than the ETH they deposited is actually worth at the current exchange rate.

### Impact Explanation

Every wrsETH token represents a proportional claim on the L1 rsETH pool. Over-minting wrsETH at a stale rate dilutes the claims of all existing wrsETH/rsETH holders. The attacker redeems or sells the surplus wrsETH, extracting ETH value that rightfully belongs to prior depositors as accrued yield. This is a direct, quantifiable theft of unclaimed yield — **High severity**.

### Likelihood Explanation

The L2 oracle is updated only when a LayerZero message is relayed from L1. Any gap between L1 price updates and L2 oracle refreshes (network congestion, relayer downtime, infrequent pushes) creates an exploitable window. Because rsETH price increases continuously with EigenLayer rewards, even a few hours of staleness produces a measurable discrepancy. An attacker can monitor `lastUpdated` on-chain and deposit precisely when the gap is largest.

### Recommendation

Add a configurable maximum staleness threshold and revert in `getRate()` if the rate is too old:

```solidity
uint256 public maxStaleness; // e.g. 1 hours

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

This mirrors the mitigation suggested in the reference report: replace the exploitable variable with a trusted, bounded value. Alternatively, the L2 pool contracts themselves should check `lastUpdated` before accepting a deposit.

### Proof of Concept

1. L1 `LRTOracle.updateRSETHPrice()` is called; `rsETHPrice` rises from `1.05e18` to `1.10e18`.
2. The LayerZero relayer has not yet delivered the updated rate to `RSETHRateReceiver` on L2; `rate` still reads `1.05e18`.
3. Attacker calls `RSETHPoolV3.deposit{value: 1 ether}("")`.
4. `viewSwapRsETHAmountAndFee(1 ether)` computes `rsETHAmount = 1e18 * 1e18 / 1.05e18 ≈ 0.9524 wrsETH`.
5. Fair amount at current rate: `1e18 * 1e18 / 1.10e18 ≈ 0.9091 wrsETH`.
6. Attacker receives **≈ 0.0433 wrsETH surplus** — worth ≈ 0.0476 ETH at the true rate — at the expense of existing holders.
7. Attacker bridges or redeems the wrsETH on L1 for the full current rsETH value, realising the profit. [7](#0-6) [8](#0-7)
