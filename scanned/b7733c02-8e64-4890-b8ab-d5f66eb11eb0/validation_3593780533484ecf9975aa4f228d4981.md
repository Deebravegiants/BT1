### Title
Stale Chainlink Price Accepted in `ChainlinkPriceOracle` While `ChainlinkOracleForRSETHPoolCollateral` Validates Freshness — Allows Over-Minting of rsETH - (File: contracts/oracles/ChainlinkPriceOracle.sol)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all staleness indicators, while the sibling contract `ChainlinkOracleForRSETHPoolCollateral.getRate()` explicitly validates `answeredInRound`, `timestamp`, and price sign. Because `updateRSETHPrice()` is a public function, any caller can push a stale-price-derived `rsETHPrice` into `LRTOracle`, causing `LRTDepositPool.getRsETHAmountToMint()` to over-mint rsETH to new depositors and dilute existing holders.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` reads from Chainlink but silently ignores every staleness signal:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

The `roundId`, `updatedAt`, and `answeredInRound` fields are all discarded. No check for `answeredInRound < roundId` (stale round), `updatedAt == 0` (incomplete round), or `price <= 0` (invalid price) is performed.

In contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` — used for pool collateral pricing — performs all three checks:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L27-32
(uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
    AggregatorV3Interface(oracle).latestRoundData();

if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

`ChainlinkPriceOracle` feeds into `LRTOracle.getAssetPrice()`: [3](#0-2) 

Which is called by `_getTotalEthInProtocol()`: [4](#0-3) 

Which is called by `_updateRsETHPrice()`, triggered by the **public** `updateRSETHPrice()`: [5](#0-4) 

The resulting `rsETHPrice` is then used directly in `LRTDepositPool.getRsETHAmountToMint()`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [6](#0-5) 

---

### Impact Explanation

When a Chainlink feed for a supported LST asset (e.g., stETH/ETH, rETH/ETH) goes stale with a price lower than the true market price, an attacker can:

1. Call the public `updateRSETHPrice()` to commit the stale, artificially-low asset price into `rsETHPrice`.
2. A lower `totalETHInProtocol` → lower `rsETHPrice` → `getRsETHAmountToMint` returns a larger rsETH amount per unit of ETH deposited.
3. Deposit ETH or LSTs via `LRTDepositPool.depositETH()` / `depositAsset()` to receive more rsETH than the true exchange rate warrants.
4. Once the oracle recovers and `rsETHPrice` corrects upward, redeem the over-minted rsETH for more underlying value than was deposited.

This constitutes **theft of unclaimed yield** from existing rsETH holders through share dilution. The `pricePercentageLimit` guard only blocks extreme deviations; a stale price within a few percent of the true price passes silently.

**Impact: High** — Theft of unclaimed yield from existing rsETH holders.

---

### Likelihood Explanation

Chainlink feeds can go stale during periods of network congestion, sequencer downtime (on L2), or oracle operator issues. The attack requires no special permissions — `updateRSETHPrice()` is callable by any EOA. The attacker only needs to observe a stale feed and act within the staleness window. This is a realistic, externally-reachable condition.

**Likelihood: Medium** — Requires a Chainlink feed to go stale, which is an uncommon but documented occurrence.

---

### Recommendation

Add staleness validation to `ChainlinkPriceOracle.getAssetPrice()` consistent with the checks already present in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Additionally, consider adding a `block.timestamp - updatedAt > MAX_STALENESS_PERIOD` check with a configurable heartbeat per feed.

---

### Proof of Concept

1. Assume stETH/ETH Chainlink feed has a heartbeat of 24 hours. The feed goes stale (last update was 23h 59m ago) with a price of `0.999e18` while the true price is `1.001e18`.
2. Attacker calls `LRTOracle.updateRSETHPrice()` (public, no access control).
3. `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns stale `0.999e18` with no revert.
4. `totalETHInProtocol` is underestimated → `newRsETHPrice` is set lower than true value (e.g., `1.049e18` instead of `1.051e18`).
5. Attacker calls `LRTDepositPool.depositETH{value: 100 ether}(0, "")`.
6. `getRsETHAmountToMint` computes: `100e18 * 1e18 / 1.049e18 ≈ 95.33 rsETH` instead of the correct `100e18 * 1e18 / 1.051e18 ≈ 95.15 rsETH` — attacker receives ~0.18 rsETH excess.
7. Chainlink feed updates; `rsETHPrice` corrects to `1.051e18`. Attacker redeems 95.33 rsETH for more ETH than deposited, extracting value from existing holders. [1](#0-0) [5](#0-4) [7](#0-6)
