### Title
Stale Cross-Chain Rate in `CrossChainRateReceiver` Enables Profitable Deposit-Bridge-Redeem Round-Trip — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

### Summary
`CrossChainRateReceiver.getRate()` returns the stored `rate` with no staleness check. `RSETHPoolNoWrapper.deposit()` uses this rate to price rsETH swaps. When the L2 rate lags behind the true L1 rate, an attacker can deposit ETH at the stale (underpriced) rate, receive excess rsETH, bridge it to L1 as an OFT, and redeem at the true L1 rate for a net ETH profit equal to the accrued yield differential.

### Finding Description

**Root cause — no staleness guard in `getRate()`:**

`CrossChainRateReceiver` stores `lastUpdated` but never validates it:

```solidity
// CrossChainRateReceiver.sol line 103-105
function getRate() external view returns (uint256) {
    return rate;   // no check against lastUpdated
}
``` [1](#0-0) 

The rate is only refreshed when a LayerZero message arrives via `lzReceive`. Between updates, `rate` can be arbitrarily stale. [2](#0-1) 

**Deposit pricing uses the stale rate directly:**

```solidity
// RSETHPoolNoWrapper.sol line 282-285
uint256 rsETHToETHrate = getRate();          // → stale CrossChainRateReceiver.rate
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [3](#0-2) 

If `R_stale < R_true`, the division yields more rsETH than the depositor deserves at the true rate.

**L1 redemption uses the live oracle price:**

`LRTWithdrawalManager.getExpectedAssetAmount()` prices rsETH at `lrtOracle.rsETHPrice()` — the live L1 rate:

```solidity
// LRTWithdrawalManager.sol line 593
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
``` [4](#0-3) 

The `_calculatePayoutAmount` min() guard does not help here — it caps at the *initially expected amount*, which is already computed at the true L1 rate, so the attacker receives the full true-rate value of their excess rsETH. [5](#0-4) 

**Rate provider confirms the L1 source:**

`RSETHRateProvider.getLatestRate()` reads `ILRTOracle.rsETHPrice()` on L1 and sends it cross-chain. Any delay in LayerZero delivery creates a window where `R_stale < R_true`. [6](#0-5) 

### Impact Explanation

**Profit per round-trip:**

```
rsETH received on L2  = E * 1e18 / R_stale
ETH redeemed on L1    = (E * 1e18 / R_stale) * R_true / 1e18
                      = E * R_true / R_stale  >  E
Net profit            = E * (R_true - R_stale) / R_stale
```

This profit is extracted from the protocol's accrued staking yield — the exact definition of **theft of unclaimed yield (High)**.

### Likelihood Explanation

- Cross-chain rate updates have inherent latency; rsETH rate increases monotonically with staking rewards, so `R_stale < R_true` is the normal state between updates.
- `deposit()` is permissionless and unpaused by default.
- Bridging rsETH OFT from L2 to L1 is permissionless (OFT standard).
- `instantWithdrawal()` on L1 is permissionless for rsETH holders.
- No admin compromise, front-running, or brute-force is required.
- The attack window exists continuously between every pair of rate updates.

### Recommendation

Add a maximum staleness check in `CrossChainRateReceiver.getRate()`:

```solidity
uint256 public constant MAX_RATE_AGE = 24 hours;

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= MAX_RATE_AGE, "Rate is stale");
    return rate;
}
```

This causes `deposit()` to revert when the rate is stale, preventing the exploit window. Alternatively, `RSETHPoolNoWrapper` can enforce the staleness check before pricing swaps.

### Proof of Concept

Fork test outline (local/private testnet, no public mainnet):

1. Deploy or fork with `CrossChainRateReceiver` holding `R_stale = 1.05e18`.
2. Set true L1 rate to `R_true = 1.10e18` (simulate elapsed staking time without a rate update).
3. Attacker calls `RSETHPoolNoWrapper.deposit{value: 10 ether}("")`.
   - Receives `10e18 * 1e18 / 1.05e18 ≈ 9.524 rsETH` (excess vs true rate of `9.09 rsETH`).
4. Attacker bridges rsETH OFT to L1 (standard OFT `send()`).
5. Attacker calls `LRTWithdrawalManager.instantWithdrawal(ETH, 9.524e18 rsETH, "")`.
   - Receives `9.524e18 * 1.10e18 / 1e18 ≈ 10.476 ETH`.
6. Assert `netProfit = 10.476 - 10 = 0.476 ETH > 0`. ✓

The invariant "deposit-bridge-redeem round-trip must not yield more ETH than deposited" is broken.
