### Title
`withinUnstakeLimits` Incorrectly Consumes `whitelistedUnstakeAllowance` When Active ETH Withdrawals Are Sufficient - (`contracts/LRTConverter.sol`)

### Summary
The `withinUnstakeLimits` modifier always deducts from `whitelistedUnstakeAllowance` whenever it is non-zero, regardless of whether `availableActiveETHWithdrawals` alone would have authorized the operation. This causes the whitelisted budget to be depleted faster than intended.

### Finding Description
In `LRTConverter.withinUnstakeLimits`, the authorization check (line 65) correctly treats the two budgets as additive:

```solidity
if (amountToUnstake > whitelistedUnstakeAllowance + availableActiveETHWithdrawals) {
    revert UnstakeLimitExceeded();
}
``` [1](#0-0) 

However, the consumption block (lines 70–75) does not first check whether `availableActiveETHWithdrawals >= amountToUnstake`:

```solidity
if (whitelistedUnstakeAllowance > 0) {
    uint256 whitelistedAmountConsumed =
        amountToUnstake > whitelistedUnstakeAllowance ? whitelistedUnstakeAllowance : amountToUnstake;
    whitelistedUnstakeAllowance -= whitelistedAmountConsumed;
}
``` [2](#0-1) 

The invariant that `whitelistedUnstakeAllowance` should only be consumed for the portion of `amountToUnstake` that exceeds `availableActiveETHWithdrawals` is never enforced. `availableActiveETHWithdrawals` is sourced from `lrtWithdrawalManager.assetsCommitted(ETH_TOKEN)`: [3](#0-2) 

`assetsCommitted[ETH_TOKEN]` is incremented by `initiateWithdrawal` in `LRTWithdrawalManager` and decremented as withdrawals are unlocked: [4](#0-3) 

### Impact Explanation
`whitelistedUnstakeAllowance` is depleted faster than intended. Future calls to `unstakeStEth` that should be authorized via the whitelisted budget will revert with `UnstakeLimitExceeded`, preventing legitimate unstaking operations. No funds are lost, but the contract fails to deliver the promised unstaking capacity. This matches **Low — Contract fails to deliver promised returns, but doesn't lose value**.

### Likelihood Explanation
The precondition (`whitelistedUnstakeAllowance > 0` AND `availableActiveETHWithdrawals >= amountToUnstake`) is a normal operational state: whitelisted users routinely call `declareWithdrawalIntent` to top up the allowance, and `assetsCommitted[ETH_TOKEN]` grows with every user ETH withdrawal request. No adversarial action is required; the bug triggers on every normal operator call to `unstakeStEth` under these conditions. [5](#0-4) 

### Recommendation
Only consume `whitelistedUnstakeAllowance` for the amount that exceeds `availableActiveETHWithdrawals`:

```solidity
if (amountToUnstake > availableActiveETHWithdrawals && whitelistedUnstakeAllowance > 0) {
    uint256 excessAmount = amountToUnstake - availableActiveETHWithdrawals;
    uint256 whitelistedAmountConsumed =
        excessAmount > whitelistedUnstakeAllowance ? whitelistedUnstakeAllowance : excessAmount;
    whitelistedUnstakeAllowance -= whitelistedAmountConsumed;
}
```
