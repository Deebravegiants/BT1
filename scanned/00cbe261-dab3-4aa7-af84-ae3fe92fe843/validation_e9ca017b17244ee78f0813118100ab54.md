### Title
Pusher Consent Signature Replay Allows Creator to Forcibly Re-Establish Revoked Delegation — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.allowPushers` accepts a pusher's EIP-191 consent signature that is bound to `(chainid, oracle, deadline, pusher, creator)` but contains **no revocation nonce**. After a pusher calls `revokePusher()`, the creator can immediately replay the original consent signature — which remains cryptographically valid until the deadline — to silently re-establish the delegation. The pusher's revocation is therefore ineffective for the entire window `[revocation_block, deadline]`.

---

### Finding Description

`allowPushers` constructs the signed digest as:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

The only replay guard is the deadline: once `block.timestamp > deadline` the call reverts via `_ensureDeadline`. There is no per-pusher nonce, no revocation counter, and no on-chain record that the pusher has ever revoked. After `revokePusher()` clears `namespaceRemapping[pusher]` to `address(0)`:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [2](#0-1) 

…the creator can call `allowPushers` again with the **identical** `(deadline, pushers, signatures)` tuple. The signature still recovers to `pusher`, the deadline is still in the future, and `namespaceRemapping[pusher]` is written back to `creator`. The pusher is re-delegated without their knowledge or consent.

The code comment itself acknowledges the risk but mischaracterises the deadline as a complete fix:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [3](#0-2) 

The deadline prevents replay **after expiry**, but not replay **between revocation and expiry** — which is the exact window the comment claims to protect.

---

### Impact Explanation

`CompressedOracleV1` is the price source for `MetricOmmPool` swaps. The pool calls `_getBidAndAskPriceX64()` → `IPriceProvider.getBidAndAskPrice()` → oracle read on every swap. [4](#0-3) 

If a pusher's signing key is compromised and the attacker begins pushing manipulated prices into the creator's namespace, the pusher's only on-chain remedy is `revokePusher()`. A creator colluding with the attacker (or the attacker who also controls the creator key) can replay the original consent signature in the same block or any subsequent block before the deadline, nullifying the revocation. Manipulated oracle data then reaches `_getBidAndAskPriceX64`, drives `SwapMath.midAndSpreadFeeX64FromBidAsk`, and causes swaps to execute at attacker-controlled prices — a direct loss of user principal through bad-price execution.

Additionally, a pusher who legitimately wants to redirect their pushes to their own namespace (to operate an independent feed) is permanently blocked from doing so until the deadline expires, because every push they make is immediately re-captured by the creator's namespace after the creator replays the delegation.

---

### Likelihood Explanation

- The consent signature is a standard EIP-191 message; any creator who stored it (e.g., in a transaction log or off-chain database) can replay it at any time before the deadline.
- Deadlines are typically set days to weeks in the future (the test suite uses `block.timestamp + 1 days`).
- No special privilege beyond being the original creator is required; the creator is a semi-trusted role, not a fully trusted admin.
- The attack is a single `allowPushers` call with already-known calldata — zero additional cost beyond gas.

---

### Recommendation

Introduce a per-pusher revocation nonce stored on-chain and include it in the signed digest:

```solidity
mapping(address => uint256) public pusherRevocationNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender,
        pusherRevocationNonce[pusher]   // <-- add nonce
    ))
);

// In revokePusher:
namespaceRemapping[msg.sender] = address(0);
pusherRevocationNonce[msg.sender]++;    // <-- invalidate all prior signatures
```

This is the direct analog of the CoreWallet fix (commit `6b3784e`): binding the signature to a context-specific counter so that revocation atomically invalidates all previously issued consent signatures.

### Proof of Concept

```
Step 1 — Setup:
  Pusher signs consent: keccak256(chainid, oracle, deadline=T+30d, pusher, creator)
  Creator calls allowPushers(deadline, [pusher], [sig])
  → namespaceRemapping[pusher] = creator  ✓

Step 2 — Pusher revokes:
  Pusher calls revokePusher()
  → namespaceRemapping[pusher] = address(0)  ✓

Step 3 — Creator replays (block.timestamp < deadline):
  Creator calls allowPushers(deadline, [pusher], [sig])   // SAME calldata
  → _ensureDeadline passes (deadline not expired)
  → ECDSA.recover returns pusher  (signature still valid)
  → namespaceRemapping[pusher] = creator  ← revocation undone

Step 4 — Pusher pushes data (intending own namespace):
  fallback() reads namespaceRemapping[pusher] = creator
  → data lands in creator's namespace, not pusher's own
  → pool reads creator's namespace → manipulated price → bad-price swap execution
```
