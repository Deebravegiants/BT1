### Title
Fee-on-Transfer Token Support Causes rsETH Over-Minting and Protocol Insolvency - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool.depositAsset()` calculates the rsETH amount to mint from the caller-supplied `depositAmount` parameter before the token transfer occurs, and never reconciles against the actual received balance. If any supported LST (all of which are upgradeable proxy contracts) is upgraded to introduce a transfer fee or deflationary mechanic, the protocol mints more rsETH than the actual backing assets, leading to protocol insolvency.

### Finding Description
In `LRTDepositPool.depositAsset()`, the rsETH mint amount is computed from the input `depositAmount` on line 111, the token is pulled on line 114, and rsETH is minted on line 115:

```solidity
uint256 rsethAmountToMint = _beforeDeposit(asset, depositAmount, minRSETHAmountExpected);
IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
_mintRsETH(rsethAmountToMint);
``` [1](#0-0) 

`_beforeDeposit` calls `getRsETHAmountToMint(asset, depositAmount)`, which computes the mint amount purely from the caller-supplied `depositAmount`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [2](#0-1) 

There is no balance-before/balance-after check. The code blindly trusts that `safeTransferFrom(..., depositAmount)` results in the contract receiving exactly `depositAmount` tokens.

The same pattern is present across all L2 pool contracts. In `RSETHPoolV3.deposit(token, amount, referralId)`, `RSETHPoolNoWrapper.deposit(token, amount, referralId)`, `RSETHPool.deposit(token, amount, referralId)`, and `RSETHPoolV3ExternalBridge.deposit(token, amount, referralId)`, the rsETH/wrsETH amount is computed from the input `amount` before the transfer, and the mint/transfer is executed against that pre-computed value: [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

All supported LSTs (stETH, wstETH, cbETH, etc.) are upgradeable proxy contracts. The protocol has no mechanism to detect a token upgrade, no automated pause on upgrade detection, and no post-transfer balance reconciliation. This is the direct structural analog to the Polkaswap finding.

### Impact Explanation
**Critical — Protocol insolvency.**

If a supported LST is upgraded to introduce a transfer fee (e.e., 1% burn on transfer), every deposit call mints rsETH based on the gross `depositAmount` while the pool only receives `depositAmount * (1 - fee)`. The rsETH supply grows faster than the actual backing. Since `rsETHPrice` is computed as `totalETHInProtocol / totalRsETHSupply`, the price is diluted for all existing rsETH holders. An attacker who repeatedly deposits and redeems extracts value from the pool at the expense of all other holders, eventually draining the protocol's solvency margin.

### Likelihood Explanation
**Medium.** The supported LSTs (stETZ, wstETH, cbETH) are all deployed behind upgradeable proxy patterns and have governance-controlled upgrade paths. The code has no structural defense (balance diff check) against this class of token behavior. The vulnerability is latent and becomes exploitable the moment any supported token's transfer semantics change — an event that requires no attacker action beyond normal depositing.

### Recommendation
Replace the input-amount-based mint calculation with a balance-before/balance-after pattern:

```solidity
uint256 balanceBefore = IERC20(asset).balanceOf(address(this));
IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
uint256 actualReceived = IERC20(asset).balanceOf(address(this)) - balanceBefore;
uint256 rsethAmountToMint = getRsETHAmountToMint(asset, actualReceived);
```

Apply the same fix to all L2 pool `deposit(token, amount, referralId)` functions. Additionally, implement an upgrade-detection mechanism (e.g., monitoring proxy implementation slots) and add an emergency pause path that can be triggered when a supported token's implementation changes.

### Proof of Concept
1. Protocol supports `wstETH` (an upgradeable proxy) as a deposit asset.
2. `wstETH` governance upgrades the implementation to introduce a 2% transfer fee.
3. Attacker calls `LRTDepositPool.depositAsset(wstETH, 100e18, 0, "")`.
4. `_beforeDeposit` computes `rsethAmountToMint` based on `100e18`.
5. `safeTransferFrom` executes; pool receives `98e18` wstETH (2% burned).
6. `_mintRsETH(rsethAmountToMint)` mints rsETH equivalent to `100e18` wstETH.
7. Attacker holds rsETH backed by only `98e18` wstETH — a 2% over-issuance per deposit.
8. Repeated deposits inflate rsETH supply beyond actual backing, diluting all holders and eventually making the protocol insolvent. [7](#0-6)
