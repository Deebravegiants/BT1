### Title
Unbounded Nested Loop in `getAssetUnstaking` Causes `depositETH`/`depositAsset`/`updateRSETHPrice` to Revert as Queued Withdrawals Accumulate - (File: contracts/NodeDelegator.sol)

---

### Summary

`NodeDelegator.getAssetUnstaking()` contains a nested loop over all queued EigenLayer withdrawals and their strategies. This function is called for every NDC inside `LRTDepositPool.getAssetDistributionData()`, which is invoked on every user deposit and on every public `updateRSETHPrice()` call. As the number of queued withdrawals grows toward the protocol-configured maximum, the gas cost of these critical paths grows proportionally and can exceed the block gas limit, causing permanent reversion.

---

### Finding Description

`NodeDelegator.getAssetUnstaking()` fetches all queued withdrawals from EigenLayer and iterates over them in a nested loop:

```solidity
// NodeDelegator.sol L406-L426
(IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
    _getDelegationManager().getQueuedWithdrawals(address(this));

for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
    IDelegationManager.Withdrawal memory withdrawal = queuedWithdrawals[withdrawalIndex];
    for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
        ...
    }
}
```

This function is called from `LRTDepositPool.getAssetDistributionData()` once per NDC in the queue:

```solidity
// LRTDepositPool.sol L447-L456
for (uint256 i; i < ndcsCount;) {
    assetLyingInNDCs += IERC20(asset).balanceOf(nodeDelegatorQueue[i]);
    assetStakedInEigenLayer += INodeDelegator(nodeDelegatorQueue[i]).getAssetBalance(asset);
    assetUnstakingFromEigenLayer += INodeDelegator(nodeDelegatorQueue[i]).getAssetUnstaking(asset);
    ...
}
```

`getAssetDistributionData` is called from `getTotalAssetDeposits`, which is called from:

1. **`depositETH` / `depositAsset`** (user-facing) → `_beforeDeposit` → `_checkIfDepositAmountExceedesCurrentLimit` → `getTotalAssetDeposits`
2. **`updateRSETHPrice()`** (public) → `_getTotalEthInProtocol` → `getTotalAssetDeposits` (once per supported asset)
3. **`initiateWithdrawal()`** (user-facing) → `getAvailableAssetAmount` → `getTotalAssetDeposits`

The total gas cost scales as:
`O(supportedAssets × NDCs × queuedWithdrawals × strategiesPerWithdrawal)`

`maxUncompletedWithdrawalCount` is capped at 80 (per `LRTUnstakingVault.setMaxUncompletedWithdrawalCount`), and `maxNodeDelegatorLimit` starts at 10. With multiple supported assets, the nested iteration becomes very expensive.

---

### Impact Explanation

**Medium — Temporary freezing of funds / Unbounded gas consumption.**

When queued withdrawals accumulate toward the protocol maximum (80 total across NDCs), the gas cost of `depositETH`, `depositAsset`, and `initiateWithdrawal` can exceed the block gas limit, causing all three to revert. Users are unable to deposit or initiate withdrawals. `updateRSETHPrice()` also reverts, breaking the oracle and blocking the entire withdrawal unlock flow (`unlockQueue` depends on a valid rsETH price). This constitutes a temporary freeze of user funds and protocol operations.

---

### Likelihood Explanation

**Medium.** The protocol is designed to hold up to 80 uncompleted withdrawals simultaneously (a normal operational state during large unstaking events). With 10 NDCs each potentially holding multiple strategies per withdrawal, the nested loop gas cost is substantial and predictable. No attacker action is required — normal protocol operation (initiating unstaking from EigenLayer) naturally fills the queue. The condition is reachable without any privileged compromise.

---

### Recommendation

1. Cache the result of `getAssetUnstaking` off-chain and provide it as a parameter, or introduce a dedicated accounting variable that is updated incrementally on each `initiateUnstaking` / `completeUnstaking` call rather than recomputed on every read.
2. Alternatively, decouple the `getAssetUnstaking` call from the hot deposit/withdrawal path by maintaining a running `assetUnstaking[asset]` storage variable in `NodeDelegator` that is incremented in `initiateUnstaking` and decremented in `completeUnstaking`.
3. Add an explicit gas guard or cap on the number of queued withdrawals iterated per call.

---

### Proof of Concept

**Call chain for `depositETH`:**

1. User calls `LRTDepositPool.depositETH(minRSETHAmountExpected, referralId)` [1](#0-0) 

2. → `_beforeDeposit` → `_checkIfDepositAmountExceedesCurrentLimit` → `getTotalAssetDeposits` [2](#0-1) 

3. → `getAssetDistributionData` loops over all NDCs calling `getAssetUnstaking` per NDC [3](#0-2) 

4. → `NodeDelegator.getAssetUnstaking` executes a nested loop over all queued withdrawals × strategies [4](#0-3) 

**Call chain for `updateRSETHPrice` (public):**

1. Anyone calls `LRTOracle.updateRSETHPrice()` [5](#0-4) 

2. → `_getTotalEthInProtocol` loops over all supported assets, calling `getTotalAssetDeposits` per asset [6](#0-5) 

3. → same `getAssetDistributionData` → `getAssetUnstaking` nested loop as above.

**Bounding parameters (worst case):**
- `maxNodeDelegatorLimit` = 10 NDCs [7](#0-6) 
- `maxUncompletedWithdrawalCount` ≤ 80 [8](#0-7) 
- Supported assets: unbounded (admin-controlled)
- Strategies per withdrawal: unbounded

With 5 supported assets, 10 NDCs, and 80 total queued withdrawals each with 2 strategies, the inner loop executes up to `5 × 10 × 80 × 2 = 8,000` iterations, each involving external EigenLayer calls, well within range to exhaust block gas.

```text
    function getAssetUnstaking(address asset) external view returns (uint256 amount) {
        (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
            _getDelegationManager().getQueuedWithdrawals(address(this));

        for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
            IDelegationManager.Withdrawal memory withdrawal = queuedWithdrawals[withdrawalIndex];

            for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
                IStrategy strategy = withdrawal.strategies[strategyIndex];

                address strategyAsset = address(strategy) == address(lrtConfig.beaconChainETHStrategy())
                    ? LRTConstants.ETH_TOKEN
                    : address(strategy.underlyingToken());

                if (strategyAsset != asset) continue;

                uint256 sharesToUnstake = withdrawalShares[withdrawalIndex][strategyIndex];
                amount += strategyAsset == LRTConstants.ETH_TOKEN
                    ? sharesToUnstake
                    : strategy.sharesToUnderlyingView(sharesToUnstake);
            }
        }
    }
```
