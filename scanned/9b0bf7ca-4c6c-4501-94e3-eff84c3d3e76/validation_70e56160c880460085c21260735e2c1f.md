### Title
Caller-Controlled Asset Selection in `withdraw()` Enables Cross-Token Theft from Wrapper — (`contracts/L2/RsETHTokenWrapper.sol` and `contracts/agETH/AGETHTokenWrapper.sol`)

---

### Summary

`RsETHTokenWrapper` and `AGETHTokenWrapper` allow any holder of the wrapper token (wrsETH / agETH) to redeem it for **any** allowed underlying token by supplying an arbitrary `asset` address. Because the contract never records which token was originally deposited, a user who deposited a cheap or depegged rsETH variant can withdraw the more valuable variant, draining it from the wrapper.

---

### Finding Description

Both wrapper contracts maintain an `allowedTokens` mapping that can contain **multiple** distinct ERC-20 tokens (different bridge variants of rsETH or agETH). The `_deposit` function mints wrapper shares 1:1 for whichever allowed token is supplied, and the `_withdraw` function burns wrapper shares and transfers whichever allowed token the caller specifies — with no linkage between the two.

`RsETHTokenWrapper._withdraw` (lines 120–128):

```solidity
function _withdraw(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();   // only checks allowlist

    _burn(msg.sender, _amount);

    ERC20Upgradeable(_asset).safeTransfer(_to, _amount);   // sends attacker-chosen token
    ...
}
``` [1](#0-0) 

`RsETHTokenWrapper._deposit` (lines 134–141) mints wrsETH 1:1 for any allowed token, with no record of which token was deposited: [2](#0-1) 

The `reinitialize` function explicitly supports adding a second allowed token, confirming the multi-token scenario is intended: [3](#0-2) 

The identical pattern exists in `AGETHTokenWrapper._withdraw` and `AGETHTokenWrapper._deposit`: [4](#0-3) [5](#0-4) 

---

### Impact Explanation

**Critical — direct theft of user funds.**

Any user can drain the more valuable rsETH (or agETH) variant held in the wrapper by depositing a cheaper/depegged variant and withdrawing the premium one. The wrapper's entire balance of the premium token is at risk. This is a direct cross-asset theft vector with no admin action required.

---

### Likelihood Explanation

**High.** The entry path is fully permissionless — `withdraw(asset, amount)` and `withdrawTo(asset, _to, amount)` are public with no role restriction. The only precondition is that two allowed tokens with a price discrepancy exist simultaneously in the wrapper, which is the explicit design (multiple bridge variants of rsETH are supported). Price discrepancies between bridge variants are common and observable on-chain, making this trivially exploitable by any MEV bot or attacker monitoring the allowedTokens list. [6](#0-5) 

---

### Recommendation

Track which token each user deposited and enforce that the same token is withdrawn, **or** restrict the wrapper to a single canonical token at any given time. If multi-token support is required, maintain per-user per-token deposit accounting and validate on withdrawal:

```solidity
// Option A: single-token enforcement
require(_asset == canonicalToken, "Wrong asset");

// Option B: per-deposit accounting
mapping(address user => mapping(address asset => uint256 balance)) public depositedBalance;
// on deposit: depositedBalance[_to][_asset] += _amount;
// on withdraw: require(depositedBalance[msg.sender][_asset] >= _amount);
```

---

### Proof of Concept

1. Two allowed tokens exist: `rsETH_A` (depegged, worth 0.90 ETH each) and `rsETH_B` (canonical, worth 1.00 ETH each). Both are in `allowedTokens`.
2. Attacker acquires `1000 rsETH_A` at a discount (cost: ~900 ETH equivalent).
3. Attacker calls `deposit(rsETH_A, 1000e18)` → receives `1000e18` wrsETH.
4. Attacker calls `withdraw(rsETH_B, 1000e18)` → burns `1000e18` wrsETH, receives `1000 rsETH_B` (worth ~1000 ETH equivalent).
5. Net profit: ~100 ETH. The wrapper's `rsETH_B` reserve is drained by the difference.

The same attack applies identically to `AGETHTokenWrapper` via its public `withdraw(asset, _amount)` function. [7](#0-6)
