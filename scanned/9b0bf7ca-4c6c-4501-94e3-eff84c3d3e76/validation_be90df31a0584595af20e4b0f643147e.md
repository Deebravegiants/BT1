### Title
Unconditional Zero-Amount Fee Transfer in `MerkleDistributor.claim()` Blocks All Claims When `feeInBPS` Is Zero - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

### Summary
`MerkleDistributor.claim()` unconditionally calls `IERC20(token).safeTransfer(protocolTreasury, fee)` even when `fee == 0`. When `feeInBPS` is set to zero, this produces a zero-amount transfer to `protocolTreasury`. Any token that reverts on zero-amount transfers will cause every `claim()` call to revert, permanently freezing all unclaimed yield in the distributor. The sibling contracts `KernelMerkleDistributor` and `KernelTop100MerkleDistributor` already apply the correct `if (fee > 0)` guard; `MerkleDistributor` is the only one that does not.

### Finding Description
In `contracts/utils/MerkleDistributor/MerkleDistributor.sol`, the `claim` function computes a fee and then unconditionally transfers it:

```solidity
// lines 138–144
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);

// Send the fee to the protocol treasury
IERC20(token).safeTransfer(protocolTreasury, fee);   // ← no zero-guard
```

`feeInBPS` is settable by the owner via `setFeeInBPS`. The setter only enforces an upper bound (`> MAX_FEE_IN_BPS`), so `feeInBPS = 0` is a valid and reachable state:

```solidity
// lines 198–205
function setFeeInBPS(uint256 _feeInBPS) external onlyOwner {
    if (_feeInBPS > MAX_FEE_IN_BPS) {
        revert InvalidFeeInBPS();
    }
    feeInBPS = _feeInBPS;
    emit FeeInBPSUpdated(_feeInBPS);
}
```

When `feeInBPS == 0`, `fee = 0`, and `safeTransfer(protocolTreasury, 0)` is executed. Tokens that implement a zero-transfer guard (e.g., `if (amount == 0) revert ZeroTransferAmount()`) will revert here, causing the entire `claim` transaction to fail. Because the user's claim state is updated *before* the transfers (lines 134–135), a revert rolls back the state update, but the user can never successfully claim — every attempt will hit the same revert.

The two sibling contracts in the same repository already apply the correct fix:

`KernelMerkleDistributor` (lines 341–343):
```solidity
if (fee > 0) {
    kernel.safeTransfer(protocolTreasury, fee);
}
```

`KernelTop100MerkleDistributor` (lines 332–334):
```solidity
if (fee > 0) {
    kernel.safeTransfer(protocolTreasury, fee);
}
```

`MerkleDistributor` is the only distributor in the codebase that omits this guard.

### Impact Explanation
**Medium — Permanent freezing of unclaimed yield.**

When `feeInBPS == 0` and the configured `token` reverts on zero-amount transfers, every call to `claim()` reverts. No user can ever withdraw their allocated tokens. The tokens remain locked in the distributor contract indefinitely with no recovery path available to users (only the owner can pause/unpause or change the token, but changing the token does not release already-allocated claims).

### Likelihood Explanation
**Medium.**

- `feeInBPS = 0` is an explicitly reachable configuration (no lower-bound check in `setFeeInBPS`). A zero fee is a natural operational choice (e.g., a promotional distribution with no protocol cut).
- The `token` field is fully configurable by the owner and can be set to any ERC-20. Tokens that revert on zero-amount transfers are common in production (USDT on some chains, custom protocol tokens with explicit zero-transfer guards, etc.).
- The combination of both conditions is realistic and requires no attacker action — it is a normal operational state.

### Recommendation
Add a zero-amount guard before the fee transfer, consistent with the pattern already used in `KernelMerkleDistributor` and `KernelTop100MerkleDistributor`:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);

if (fee > 0) {
    IERC20(token).safeTransfer(protocolTreasury, fee);
}
```

### Proof of Concept

1. Owner deploys `MerkleDistributor` with a token that reverts on zero-amount transfers and sets `feeInBPS = 0`.
2. Owner sets a valid Merkle root containing user allocations.
3. User calls `claim(index, account, cumulativeAmount, merkleProof)` with a valid proof.
4. Execution reaches line 138: `fee = (claimableAmount * 0) / 10_000 = 0`.
5. `IERC20(token).safeTransfer(account, claimableAmount)` succeeds.
6. `IERC20(token).safeTransfer(protocolTreasury, 0)` reverts with `ZeroTransferAmount` (or equivalent).
7. The entire transaction reverts. The user's claim state is not updated.
8. Every subsequent `claim()` call by any user hits the same revert. All yield is permanently frozen.
