### Title
`feeBps` Not Validated in `initialize()` Allows 100% Fee Extraction from Depositors - (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPoolV2ExternalBridge.sol, contracts/pools/RSETHPoolNoWrapper.sol, contracts/agETH/AGETHPoolV3.sol)

---

### Summary

Multiple L2 pool contracts accept a `_feeBps` parameter in their `initialize()` function and assign it directly to `feeBps` without any upper-bound validation. The corresponding `setFeeBps()` setter enforces a strict maximum (1,000 bps = 10% in several contracts), but the initializer bypasses this check entirely. If `feeBps` is initialized to 10,000 (100%), every depositor's ETH is consumed entirely as fees, yielding 0 rsETH/agETH tokens in return.

---

### Finding Description

In `RSETHPoolV3.initialize()`, `feeBps` is assigned directly from the constructor argument with no upper-bound check:

```solidity
feeBps = _feeBps;   // no validation
``` [1](#0-0) 

Yet the setter `setFeeBps()` enforces a hard cap of 1,000 bps (10%):

```solidity
function setFeeBps(uint256 _feeBps) external onlyRole(DEFAULT_ADMIN_ROLE) {
    if (_feeBps > 1000) revert InvalidFeeAmount();
``` [2](#0-1) 

The same pattern exists across:

- `RSETHPoolV3ExternalBridge.initialize()` (sets `feeBps = _feeBps` unchecked) vs `setFeeBps()` capped at 1,000 bps [3](#0-2) 

- `RSETHPoolV2ExternalBridge.initialize()` (sets `feeBps = _feeBps` unchecked) vs `setFeeBps()` capped at 1,000 bps [4](#0-3) [5](#0-4) 

- `RSETHPoolNoWrapper.initialize()` (sets `feeBps = _feeBps` unchecked) vs `setFeeBps()` capped at 10,000 bps [6](#0-5) [7](#0-6) 

- `AGETHPoolV3.initialize()` (sets `feeBps = _feeBps` unchecked) [8](#0-7) 

- `RSETHPoolV2NBA.initialize()` (sets `feeBps = _feeBps` unchecked) [9](#0-8) 

---

### Impact Explanation

**Impact: High — Theft of user funds (ETH/LST deposits)**

When `feeBps` is initialized to 10,000 (100%), the fee calculation inside `viewSwapRsETHAmountAndFee` returns `fee = amount` and `rsETHAmount = 0`. Every depositor sends ETH and receives 0 rsETH/agETH. The entire deposited ETH accumulates in `feeEarnedInETH` and is withdrawable by the `BRIDGER_ROLE`. This constitutes direct theft of depositor funds at-rest.

For contracts where `setFeeBps` caps at 1,000 bps, any initialization value between 1,001 and 10,000 bps silently bypasses the intended 10% ceiling, allowing the protocol to extract between 10% and 100% of every deposit.

---

### Likelihood Explanation

**Likelihood: Low**

Exploitation requires a misconfiguration error by the deploying admin at initialization time. The initializer is called once during proxy deployment. There is no on-chain enforcement preventing `_feeBps = 10_000` from being passed. The risk is a deployment-time mistake or a compromised deployment script, not an ongoing attacker-controlled path. This matches the external report's characterization of "low likelihood, high impact."

---

### Recommendation

Apply the same upper-bound check used in `setFeeBps()` inside each `initialize()` function. For example, in `RSETHPoolV3`:

```solidity
if (_feeBps > 1000) revert InvalidFeeAmount();
feeBps = _feeBps;
```

This mirrors the pattern already correctly applied in `KernelMerkleDistributor.initialize()`:

```solidity
if (_feeInBPS > MAX_FEE_IN_BPS) {
    revert InvalidFeeInBPS();
}
``` [10](#0-9) 

---

### Proof of Concept

1. Deploy `RSETHPoolV3` proxy and call `initialize(admin, bridger, wrsETH, 10_000, oracle, true)` — `feeBps` is set to 10,000 with no revert.
2. Any user calls `deposit{value: 1 ether}("ref")`.
3. `viewSwapRsETHAmountAndFee(1 ether)` computes `fee = 1 ether * 10_000 / 10_000 = 1 ether`, `rsETHAmount = 0`.
4. `feeEarnedInETH += 1 ether`; user receives 0 rsETH tokens.
5. Bridger calls `withdrawFees()` and extracts the 1 ETH.
6. Contrast: calling `setFeeBps(10_000)` post-deployment reverts with `InvalidFeeAmount` — the setter correctly blocks this value, but the initializer does not.
