### Title
ETH Deposit Cap Check Omits New Deposit Amount, Allowing Cap Bypass - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric cap check: for ERC20 assets it correctly tests `totalAssetDeposits + amount > depositLimit`, but for ETH it only tests `totalAssetDeposits > depositLimit`, ignoring the incoming `msg.value`. Any ETH deposit that would push the total over the cap is silently accepted.

### Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit`, the ETH branch reads:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
}
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
```

The `amount` parameter (which equals `msg.value` for ETH deposits) is passed into the function but is never used in the ETH branch. The ERC20 branch correctly adds `amount` to `totalAssetDeposits` before comparing against the limit. The ETH branch does not. [1](#0-0) 

The public entry point `depositETH` passes `msg.value` as `depositAmount` to `_beforeDeposit`, which forwards it to this check: [2](#0-1) [3](#0-2) 

### Impact Explanation
The ETH deposit cap (`lrtConfig.depositLimitByAsset(ETH_TOKEN)`) is a protocol-level risk control. Because the incoming deposit amount is excluded from the cap check, a depositor can push total ETH deposits arbitrarily above the configured limit in a single transaction. This causes the protocol to mint more rsETH than the cap was designed to allow, violating the invariant that total ETH deposits ≤ `depositLimitByAsset`. Depending on why the cap was set (e.g., EigenLayer strategy limits, liquidity constraints), this can lead to the protocol holding more ETH than it can safely delegate or redeem, degrading the protocol's ability to deliver promised returns.

**Impact: Low** — Contract fails to deliver promised returns (cap invariant is broken); no direct fund theft or freeze in isolation.

### Likelihood Explanation
The `depositETH` function is a public, permissionless, payable function callable by any user. No special role or precondition is required beyond having ETH. The only prerequisite is that `totalAssetDeposits` is currently below the cap (i.e., the cap has not already been exceeded). This is the normal operating state of the protocol, making the condition trivially satisfiable.

### Recommendation
Apply the same `+ amount` inclusion used for ERC20 assets to the ETH branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This unifies the logic and ensures the incoming deposit is always counted before comparing against the limit.

### Proof of Concept

1. Assume `depositLimitByAsset(ETH_TOKEN) = 100 ether` and `getTotalAssetDeposits(ETH_TOKEN) = 99 ether`.
2. Attacker calls `depositETH{value: 50 ether}(0, "")`.
3. Inside `_checkIfDepositAmountExceedesCurrentLimit`:
   - `totalAssetDeposits = 99 ether`
   - ETH branch: `return (99 ether > 100 ether)` → `false` → cap check passes.
   - The `50 ether` `amount` argument is never evaluated.
4. `_mintRsETH` executes; the contract now holds `149 ether` of ETH, 49 ETH above the configured cap.
5. For ERC20 the same deposit would correctly return `(99 + 50 > 100)` → `true` → revert. [4](#0-3) [5](#0-4)
