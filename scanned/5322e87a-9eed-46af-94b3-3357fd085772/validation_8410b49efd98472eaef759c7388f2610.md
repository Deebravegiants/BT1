### Title
Unprivileged Caller Can Trigger Protocol-Wide Pause via Public `updateRSETHPrice()` - (File: contracts/LRTOracle.sol)

### Summary
`LRTOracle.updateRSETHPrice()` is declared `public` with no access control, allowing any external actor to invoke the rsETH price update logic. When the computed price has dropped beyond `pricePercentageLimit` relative to `highestRsethPrice`, the function automatically pauses the `LRTDepositPool`, `LRTWithdrawalManager`, and the oracle itself. An unprivileged attacker can call this function at a strategically chosen moment to freeze all deposits and withdrawals.

### Finding Description
`updateRSETHPrice()` carries only the `whenNotPaused` modifier and is otherwise unrestricted:

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [1](#0-0) 

Inside `_updateRsETHPrice()`, when the newly computed price falls more than `pricePercentageLimit` below `highestRsethPrice`, the function unconditionally pauses the deposit pool, withdrawal manager, and oracle — regardless of who the caller is:

```solidity
if (isPriceDecreaseOffLimit) {
    if (!lrtDepositPool.paused()) lrtDepositPool.pause();
    if (!withdrawalManager.paused()) withdrawalManager.pause();
    _pause();
    return;
}
``` [2](#0-1) 

The protocol provides a privileged bypass path (`updateRSETHPriceAsManager()`) that is restricted to `onlyLRTManager`, confirming the design intent that sensitive price updates should be gated: [3](#0-2) 

However, the public `updateRSETHPrice()` path has no such gate. Any EOA or contract can call it at any time the oracle is not already paused.

### Impact Explanation
When the price condition is met (a real price drop beyond the configured threshold), an attacker can be the first to call `updateRSETHPrice()`, triggering the automatic pause of:
- `LRTDepositPool` — blocking all new deposits
- `LRTWithdrawalManager` — blocking all pending and new withdrawals
- `LRTOracle` itself — blocking further price updates via the public path

This constitutes a **temporary freezing of funds** for all depositors and withdrawers. The pause can only be lifted by `onlyLRTAdmin` calling `unpause()` on each contract individually. [4](#0-3) 

The `rsethAmountToMint` calculation in `LRTDepositPool.getRsETHAmountToMint()` reads `lrtOracle.rsETHPrice()` directly, so a stale or paused oracle also affects the correctness of any deposits that do occur before the pause is lifted: [5](#0-4) 

### Likelihood Explanation
The price of rsETH can legitimately drop relative to `highestRsethPrice` due to EigenLayer slashing events, LST depeg, or temporary accounting discrepancies. When such a condition exists — even transiently — any observer can front-run the legitimate keeper/manager and call `updateRSETHPrice()` first. The attacker needs no capital, no special role, and no privileged access. The only prerequisite is that the on-chain price condition is satisfied, which is a realistic market scenario.

### Recommendation
Restrict `updateRSETHPrice()` to a trusted role (e.g., `onlyLRTManager` or a dedicated `KEEPER_ROLE`), consistent with the existing `updateRSETHPriceAsManager()` pattern. Alternatively, separate the price-reading logic from the pause-triggering logic so that the automatic pause can only be triggered by a privileged caller, while read-only price queries remain public.

### Proof of Concept
1. Admin sets `pricePercentageLimit` to `1e16` (1%).
2. `highestRsethPrice` is `1.05 ether` (set during a prior update).
3. A slashing event causes the actual computed price to drop to `1.03 ether` (≈1.9% drop, exceeding the 1% limit).
4. Attacker calls `LRTOracle.updateRSETHPrice()` directly — no role required.
5. `_updateRsETHPrice()` computes `newRsETHPrice = 1.03 ether`, detects `isPriceDecreaseOffLimit = true`, and executes:
   - `lrtDepositPool.pause()`
   - `withdrawalManager.pause()`
   - `_pause()` on the oracle
6. All deposits and withdrawals are now frozen. The attacker spent only gas.
