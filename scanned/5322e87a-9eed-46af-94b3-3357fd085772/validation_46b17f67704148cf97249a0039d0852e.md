### Title
Fee-on-Transfer Token Mis-Accounting Inflates rsETH Minted Beyond Actual Received Assets - (File: contracts/LRTDepositPool.sol)

---

### Summary

`LRTDepositPool.depositAsset()` computes the rsETH amount to mint from the caller-supplied `depositAmount` **before** the token transfer occurs, then mints that full amount regardless of how many tokens the contract actually receives. If a supported asset is a Fee-on-Transfer (FoT) token, the contract receives fewer tokens than `depositAmount`, but mints rsETH as if it received the full amount, inflating rsETH supply beyond its real backing and causing protocol insolvency for later redeemers.

---

### Finding Description

In `LRTDepositPool.depositAsset()`, the execution order is:

1. `_beforeDeposit(asset, depositAmount, minRSETHAmountExpected)` is called, which internally calls `getRsETHAmountToMint(asset, depositAmount)` — computing `rsethAmountToMint = (depositAmount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()` using the **nominal** `depositAmount`.
2. `IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount)` executes the transfer. For a FoT token, the contract receives `depositAmount - fee`, not `depositAmount`.
3. `_mintRsETH(rsethAmountToMint)` mints rsETH calculated from the full `depositAmount`. [1](#0-0) 

The rsETH mint quantity is thus overstated by the FoT fee amount. The `getAssetDistributionData` function correctly reads actual `balanceOf` for accounting, so the real backing is lower than the rsETH supply implies. [2](#0-1) 

The same pattern exists in every L2 pool's token `deposit()` function: `safeTransferFrom(msg.sender, address(this), amount)` followed immediately by `viewSwapRsETHAmountAndFee(amount, token)` using the nominal `amount`. [3](#0-2) [4](#0-3) 

---

### Impact Explanation

**Critical — Protocol Insolvency.**

Each FoT deposit mints rsETH backed by fewer assets than accounted. The rsETH price (`rsETHPrice`) is derived from `getTotalAssetDeposits`, which sums actual on-chain balances. After a FoT deposit, the actual balance is `depositAmount - fee`, but rsETH equivalent to `depositAmount` was minted. This dilutes all existing rsETH holders and, if repeated, progressively undercollateralizes the protocol. When users attempt to redeem rsETH, the last redeemers will find insufficient assets to cover their claims — a direct insolvency condition. [5](#0-4) 

---

### Likelihood Explanation

**Low.** The currently supported LST assets (stETH, wstETH, ETHx, etc.) are not FoT tokens. However, the protocol has an admin-controlled mechanism to add new supported assets via `LRTConfig`. If a FoT token is ever whitelisted — whether intentionally or by mistake — any unprivileged depositor can immediately exploit this path. No special permissions are required to call `depositAsset()`. [6](#0-5) 

---

### Recommendation

Apply a balance-before/balance-after check in `depositAsset()` and use the actual received amount for rsETH minting:

```solidity
uint256 balanceBefore = IERC20(asset).balanceOf(address(this));
IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
uint256 actualReceived = IERC20(asset).balanceOf(address(this)) - balanceBefore;

uint256 rsethAmountToMint = _beforeDeposit(asset, actualReceived, minRSETHAmountExpected);
_mintRsETH(rsethAmountToMint);
```

Apply the same fix to all L2 pool `deposit(address token, uint256 amount, ...)` functions in `RSETHPoolV3`, `RSETHPool`, `RSETHPoolNoWrapper`, `RSETHPoolV3ExternalBridge`, and `RSETHPoolV3WithNativeChainBridge`. Alternatively, document explicitly that FoT and rebasing tokens must never be added as supported assets.

---

### Proof of Concept

1. Governance adds a FoT token (e.g., 1% fee on transfer) as a supported asset in `LRTConfig`.
2. Attacker calls `LRTDepositPool.depositAsset(fotToken, 1000e18, 0, "")`.
3. `_beforeDeposit` computes `rsethAmountToMint` based on `1000e18`.
4. `safeTransferFrom` executes; contract receives `990e18` (1% fee taken).
5. `_mintRsETH` mints rsETH equivalent to `1000e18` worth of the asset.
6. Attacker holds rsETH backed by `1000e18` but only `990e18` tokens exist in the pool.
7. Repeated deposits progressively widen the gap between rsETH supply and actual backing.
8. When legitimate users redeem rsETH, the last redeemer cannot withdraw their full entitlement — funds are permanently frozen for them. [7](#0-6) [8](#0-7)
