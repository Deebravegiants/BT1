### Title
Protocol Does Not Update `rsETHPrice` After Changing Asset Price Oracle - (File: contracts/LRTOracle.sol)

### Summary
`LRTOracle.updatePriceOracleFor` replaces the live price oracle for a supported asset but never calls `_updateRsETHPrice()` to refresh the cached `rsETHPrice` storage variable. Because `LRTDepositPool.getRsETHAmountToMint` divides the live new-oracle asset price by the stale cached `rsETHPrice`, every deposit made in the window between the oracle swap and the next `updateRSETHPrice()` call uses a mismatched numerator/denominator, minting an incorrect amount of rsETH.

### Finding Description

`LRTOracle` stores a cached exchange rate in the storage variable `rsETHPrice`. [1](#0-0) 

This cache is refreshed only when `_updateRsETHPrice()` is explicitly invoked. The admin function `updatePriceOracleFor` (and its validated wrapper `updatePriceOracleForValidated`) writes a new oracle address into `assetPriceOracle[asset]` but never triggers a cache refresh: [2](#0-1) 

`_updateRsETHPrice` internally calls `_getTotalEthInProtocol`, which reads live prices through `getAssetPrice(asset)`: [3](#0-2) 

`LRTDepositPool.getRsETHAmountToMint` combines a **live** asset price (from the newly-installed oracle) with the **stale** cached `rsETHPrice`: [4](#0-3) 

The numerator (`lrtOracle.getAssetPrice(asset)`) immediately reflects the new oracle, while the denominator (`lrtOracle.rsETHPrice()`) still reflects the old oracle's valuation. This mismatch persists until any caller invokes the public `updateRSETHPrice()`.

### Impact Explanation

**Low — Contract fails to deliver promised returns.**

- If the replacement oracle reports a **higher** asset price than the old one, `rsETHPrice` is understated relative to the new valuation. Depositors receive **more** rsETH than the protocol's true NAV justifies, diluting existing rsETH holders.
- If the replacement oracle reports a **lower** asset price, depositors receive **less** rsETH than they are entitled to, under-compensating them.

The magnitude scales with (a) the price delta between old and new oracle and (b) the deposit volume during the staleness window. For a large-TVL protocol with significant stETH or cbETH holdings, even a 2–5% oracle price difference produces a material per-deposit error.

### Likelihood Explanation

Oracle upgrades are a routine operational event (e.g., migrating from one Chainlink feed to another, or switching to a new price fetcher contract). The admin does not need to act maliciously; the staleness is an automatic consequence of any legitimate oracle replacement. The public `updateRSETHPrice()` is not guaranteed to be called atomically in the same transaction, so the window is real.

### Recommendation

Call `_updateRsETHPrice()` (or expose a public wrapper) at the end of `updatePriceOracleFor` and `updatePriceOracleForValidated`, mirroring the BendDAO fix pattern:

```diff
 function updatePriceOracleFor(address asset, address priceOracle) public onlyLRTAdmin {
     if (lrtConfig.isSupportedAsset(asset)) {
         UtilLib.checkNonZeroAddress(priceOracle);
     }
     assetPriceOracle[asset] = priceOracle;
+    _updateRsETHPrice();          // refresh cached rsETHPrice immediately
     emit AssetPriceOracleUpdate(asset, priceOracle);
 }
```

### Proof of Concept

1. Protocol is live; `rsETHPrice` = 1.05 ETH (cached from last `updateRSETHPrice()` call).
2. Admin calls `updatePriceOracleFor(stETH, newOracle)`. The new oracle returns 1.10 ETH/stETH (old oracle returned 1.05 ETH/stETH). `rsETHPrice` is **not** refreshed.
3. Alice calls `LRTDepositPool.depositAsset(stETH, 100e18, minExpected)`.
4. `getRsETHAmountToMint` computes: `(100e18 × 1.10e18) / 1.05e18 ≈ 104.76 rsETH`.
5. Correct amount (with refreshed `rsETHPrice` ≈ 1.10 ETH) would be: `(100e18 × 1.10e18) / 1.10e18 = 100 rsETH`.
6. Alice receives ~4.76 extra rsETH, diluting all existing rsETH holders until `updateRSETHPrice()` is called. [5](#0-4) [6](#0-5)
