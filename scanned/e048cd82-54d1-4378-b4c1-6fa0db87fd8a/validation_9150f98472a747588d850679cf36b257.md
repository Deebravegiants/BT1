### Title
Missing Chainlink Price Validity Checks in `ChainlinkPriceOracle` Allows Stale/Invalid Prices to Corrupt rsETH Exchange Rate - (File: contracts/oracles/ChainlinkPriceOracle.sol)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls Chainlink's `latestRoundData()` but silently discards all validity fields (`roundId`, `updatedAt`, `answeredInRound`), performing no staleness, incomplete-round, or non-positive price checks. The same codebase already implements all three checks in `ChainlinkOracleForRSETHPoolCollateral`, making the omission in `ChainlinkPriceOracle` an inconsistency with a concrete security impact. Because `updateRSETHPrice()` is a public function callable by anyone, an attacker can trigger a price update at the exact moment a Chainlink feed is stale or returning an invalid value, permanently corrupting the stored `rsETHPrice` and causing incorrect rsETH minting for depositors and incorrect payout calculations for withdrawers.

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the Chainlink price as follows:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All five return values of `latestRoundData()` — `roundId`, `answer`, `startedAt`, `updatedAt`, `answeredInRound` — are available, but only `answer` is used. No check is made for:
- `answeredInRound < roundId` (stale round)
- `updatedAt == 0` (incomplete round)
- `price <= 0` (invalid/negative price)

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` in the same repository performs all three checks:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L27-L32
(uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
    AggregatorV3Interface(oracle).latestRoundData();
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The vulnerable `getAssetPrice()` is consumed by `LRTOracle.getAssetPrice()`, which delegates directly to the registered `IPriceFetcher`:

```solidity
// contracts/LRTOracle.sol L156-L158
function getAssetPrice(address asset) public view onlySupportedOracle(asset) returns (uint256) {
    return IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset);
}
```

`LRTOracle.getAssetPrice()` is called inside `_getTotalEthInProtocol()`, which is called by `_updateRsETHPrice()`, which is called by the **public** `updateRSETHPrice()`:

```solidity
// contracts/LRTOracle.sol L87-L89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

Inside `_updateRsETHPrice()`, the stale/invalid asset price directly determines `totalETHInProtocol`, which in turn determines the new `rsETHPrice` written to storage:

```solidity
// contracts/LRTOracle.sol L231, L250
uint256 totalETHInProtocol = _getTotalEthInProtocol();
...
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

The corrupted `rsETHPrice` is then used:
1. **Deposits**: `_beforeDeposit` → `getRsETHAmountToMint` uses the stored `rsETHPrice` to determine how many rsETH tokens to mint per deposited asset.
2. **Withdrawals**: `LRTWithdrawalManager._createUnlockParams()` reads `lrtOracle.getAssetPrice(asset)` directly to compute `_calculatePayoutAmount`, determining how much asset a withdrawer receives.

---

### Impact Explanation

- **Stale-low asset price**: `totalETHInProtocol` is underestimated → `rsETHPrice` is set artificially low → subsequent depositors receive more rsETH than they are entitled to, diluting existing rsETH holders (theft of yield/value from existing holders).
- **Stale-high asset price**: `totalETHInProtocol` is overestimated → `rsETHPrice` is set artificially high → subsequent depositors receive fewer rsETH than they are entitled to (loss for depositors).
- **Zero or negative price** (Chainlink can return 0 on circuit-breaker events): `uint256(price)` wraps to a very large number or is 0, causing catastrophic mispricing of `rsETHPrice`, potentially triggering the downside-protection pause or minting unbounded protocol fees.
- **Withdrawal payouts**: `_calculatePayoutAmount` uses `lrtOracle.getAssetPrice(asset)` directly; a stale asset price causes withdrawers to receive more or fewer assets than they are owed.

Impact classification: **High** — theft of unclaimed yield / incorrect rsETH minting that permanently dilutes existing holders, and incorrect withdrawal payouts.

---

### Likelihood Explanation

Chainlink feeds can temporarily return stale data during network congestion, sequencer downtime (on L2), or when a feed's heartbeat has not been met. `updateRSETHPrice()` is a public function with no access control, so any external caller — including a bot or an attacker — can invoke it at the precise moment a feed is stale. This is a realistic, externally triggerable condition requiring no privileged access.

---

### Recommendation

Apply the same validity checks already present in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Additionally, consider adding a configurable heartbeat check (`block.timestamp - updatedAt > heartbeat`) for each feed.

---

### Proof of Concept

1. Chainlink's LST/ETH feed enters a stale state (e.g., `answeredInRound < roundId`) due to network congestion.
2. Attacker calls `LRTOracle.updateRSETHPrice()` (public, no access control).
3. `_updateRsETHPrice()` calls `_getTotalEthInProtocol()`.
4. `_getTotalEthInProtocol()` calls `LRTOracle.getAssetPrice(asset)` → `ChainlinkPriceOracle.getAssetPrice(asset)` → returns stale price with no revert.
5. `totalETHInProtocol` is computed using the stale price.
6. `newRsETHPrice` is written to `rsETHPrice` storage with the corrupted value.
7. Subsequent depositors calling `depositAsset()` receive an incorrect number of rsETH tokens based on the corrupted `rsETHPrice`, diluting or shortchanging them.
8. Withdrawers processed via `unlockQueue()` receive incorrect asset payouts via `_calculatePayoutAmount`, which also reads `lrtOracle.getAssetPrice(asset)` through the same vulnerable path.
