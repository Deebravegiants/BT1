### Title
Daily Fee Mint Cap Blocks `_updateRsETHPrice`, Freezing `rsETHPrice` for Up to 24 Hours - (File: contracts/LRTOracle.sol)

---

### Summary

`_updateRsETHPrice()` calls `_checkAndUpdateDailyFeeMintLimit()` **before** writing `rsETHPrice = newRsETHPrice`. When `currentPeriodMintedFeeAmount + rsethAmountToMintAsProtocolFee > maxFeeMintAmountPerDay`, the check reverts with `DailyFeeMintLimitExceeded`, unwinding the entire transaction and leaving `rsETHPrice` at its stale value. Both public entry points (`updateRSETHPrice()` and `updateRSETHPriceAsManager()`) share this path, so neither can update the price until the 24-hour period resets or an admin raises the cap.

---

### Finding Description

In `LRTOracle.sol`, `_updateRsETHPrice()` computes a new price and a protocol fee, then enforces the daily fee cap before committing the price:

```
// line 301-303
uint256 rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice);
_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee);  // ← can revert here
...
rsETHPrice = newRsETHPrice;   // line 313 — never reached on revert
```

`_checkAndUpdateDailyFeeMintLimit()` reverts unconditionally when the cap is exceeded:

```solidity
// line 205-206
if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
    revert DailyFeeMintLimitExceeded(...);
}
```

Because `rsETHPrice = newRsETHPrice` (line 313) is placed **after** the fee-limit check, any revert from the check leaves `rsETHPrice` frozen at its previous value. The period resets only when `block.timestamp >= feePeriodStartTime + 1 days` (line 199), so the freeze can last up to 24 hours.

The scenario arises organically: across a day of normal operation, each successful price update that captures a TVL gain increments `currentPeriodMintedFeeAmount`. Once the running total is close to `maxFeeMintAmountPerDay`, the next update that would mint even a small fee reverts entirely.

---

### Impact Explanation

While `updateRSETHPrice()` is blocked, `rsETHPrice` is stale. `LRTDepositPool.getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()`:

```solidity
// LRTDepositPool.sol line 520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

A stale (lower-than-correct) `rsETHPrice` causes depositors to receive **more rsETH than they are entitled to**, diluting existing holders. The oracle freeze lasts up to 24 hours, during which every deposit misprices rsETH. This constitutes a temporary functional freeze of the oracle — the protocol cannot correctly account for TVL changes — matching the **Medium: Temporary freezing of funds** scope.

---

### Likelihood Explanation

The condition is reachable through normal protocol operation without any attacker:

1. TVL grows steadily throughout a day (staking rewards, new deposits).
2. Each `updateRSETHPrice()` call that captures a TVL gain mints a small fee and increments `currentPeriodMintedFeeAmount`.
3. When the running total approaches `maxFeeMintAmountPerDay`, the next fee-generating update reverts.
4. `updateRSETHPriceAsManager()` also calls `_updateRsETHPrice()` and is equally blocked.

The only on-chain mitigation requires the LRT manager to call `setMaxFeeMintAmountPerDay()` to raise the cap and then call `updateRSETHPriceAsManager()` — active human intervention within the freeze window.

---

### Recommendation

Decouple the price write from the fee-mint check. Two clean options:

**Option A — Write price first, then attempt fee mint:**
Move `rsETHPrice = newRsETHPrice` before the fee-limit check. If the cap is exceeded, skip the fee mint (or emit an event) but still commit the updated price.

**Option B — Cap the fee instead of reverting:**
In `_checkAndUpdateDailyFeeMintLimit`, instead of reverting, clamp `feeAmount` to the remaining daily allowance and return the clamped value. The caller mints only the clamped amount and still updates the price.

Either approach ensures the price oracle is never blocked by the fee accounting subsystem.

```text
    function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
        // Reset the period if it's unset or a day has passed
        if (block.timestamp >= feePeriodStartTime + 1 days) {
            currentPeriodMintedFeeAmount = 0;
            feePeriodStartTime = getCurrentPeriodStartTime();
        }

        // Check if minting would exceed the daily limit
        if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
            revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
        }

        currentPeriodMintedFeeAmount += feeAmount;
    }
```
