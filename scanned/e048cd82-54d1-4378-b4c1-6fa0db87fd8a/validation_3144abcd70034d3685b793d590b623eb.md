### Title
Reward Accumulator Rounding to Zero While `updatedAt` Advances Causes Permanent Yield Freeze - (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary
`KernelDepositPool.rewardPerToken()` computes a per-token increment that can round to zero under realistic conditions (large `totalKernelStaked`, low `rewardRate`, short `timeDelta`). The `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` even when `rewardPerTokenStored` did not change. Any user can trigger this modifier by calling `stake`, `initiateWithdrawal`, or `getReward`, keeping `timeDelta` perpetually small and causing all deposited reward tokens to be permanently frozen in the contract.

### Finding Description
`rewardPerToken()` computes:

```solidity
// KernelDepositPool.sol line 412-413
return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
    / totalKernelStaked;
```

If `rewardRate * timeDelta * 1e18 < totalKernelStaked`, Solidity integer division truncates the increment to zero, leaving `rewardPerTokenStored` unchanged.

The `updateReward` modifier then unconditionally writes:

```solidity
// KernelDepositPool.sol line 233-234
rewardPerTokenStored = rewardPerToken();   // unchanged (still zero increment)
updatedAt = lastTimeRewardApplicable();    // advances regardless
```

Because `updatedAt` advances even when no rewards were credited, the time window for those seconds is permanently consumed. Subsequent calls to `rewardPerToken()` compute `timeDelta` from the new (advanced) `updatedAt`, so the lost seconds can never be recovered.

Every public state-changing function that carries `updateReward` — `stake`, `initiateWithdrawal`, `getReward` — is an unprivileged entry point that any user can call to trigger this path.

### Impact Explanation
Reward tokens transferred into the contract via `notifyRewardAmount` are permanently locked. `rewardPerTokenStored` never increases, so `earned()` always returns zero for all stakers, and `getReward()` transfers nothing. There is no admin sweep or recovery function for reward tokens. The impact is **permanent freezing of unclaimed yield** (Medium per scope), escalating toward a critical freeze of all reward funds if the condition persists for the full reward duration.

### Likelihood Explanation
The condition is realistic and can occur naturally or be forced:

- **Natural:** KERNEL is an 18-decimal token. If 1 billion KERNEL are staked (`totalKernelStaked = 1e27`) and the admin distributes 1,000 KERNEL over 30 days (`rewardRate = 1e21 / 2_592_000 ≈ 3.86e14`), then with a 1-second `timeDelta`: `3.86e14 * 1 * 1e18 / 1e27 = 0`.
- **Forced:** Any staker can call `stake(1)` or `getReward()` every block (cheap on L2) to keep `timeDelta = 1`, ensuring the numerator never exceeds `totalKernelStaked`.

### Recommendation
In the `updateReward` modifier, only advance `updatedAt` if `rewardPerTokenStored` actually changed:

```solidity
modifier updateReward(address _account) {
    uint256 newRewardPerToken = rewardPerToken();
    if (newRewardPerToken != rewardPerTokenStored) {
        rewardPerTokenStored = newRewardPerToken;
        updatedAt = lastTimeRewardApplicable();
    }
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

Alternatively, enforce a minimum `rewardRate` relative to `totalKernelStaked` inside `notifyRewardAmount` so that at least 1 wei of `rewardPerTokenStored` accrues per second.

### Proof of Concept

1. Admin calls `notifyRewardAmount` with 1,000 KERNEL over 30 days → `rewardRate ≈ 3.86e14`.
2. 1 billion KERNEL are staked → `totalKernelStaked = 1e27`.
3. Attacker (or any normal user) calls `stake(1)` every block (1-second blocks on L2).
4. Each call triggers `updateReward`:
   - `rewardPerToken()` = `rewardPerTokenStored + (3.86e14 * 1 * 1e18) / 1e27` = `rewardPerTokenStored + 0` (truncated).
   - `rewardPerTokenStored` stays at 0.
   - `updatedAt` advances by 1 second.
5. After the full 30-day reward period, `rewardPerTokenStored` is still 0. All stakers call `getReward()` and receive 0. The 1,000 KERNEL reward tokens remain permanently locked in the contract.
