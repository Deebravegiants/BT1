### Title
Unbounded Withdrawal Queue Stuffing Enables Temporary Freezing of Legitimate User Withdrawals - (File: contracts/LRTWithdrawalManager.sol)

### Summary
`LRTWithdrawalManager.initiateWithdrawal` imposes no per-user limit on the number of queued withdrawal requests. Because the global queue is processed strictly in FIFO order via `_unlockWithdrawalRequests`, an attacker who deposits the minimum rsETH amount and calls `initiateWithdrawal` many times across multiple addresses can fill the queue with dust requests, forcing the operator to exhaust those entries before any later legitimate user's request can be unlocked.

### Finding Description
`initiateWithdrawal` accepts any number of calls from any address, subject only to a minimum rsETH amount check:

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```

Each call appends a new entry to the global nonce sequence via `_addUserWithdrawalRequest`:

```solidity
userAssociatedNonces[asset][msg.sender].pushBack(nextUnusedNonce_);
nextUnusedNonce[asset] = nextUnusedNonce_ + 1;
```

The operator unlocks requests by calling `unlockQueue`, which internally runs `_unlockWithdrawalRequests`. That function iterates from the current `nextLockedNonce[asset]` forward:

```solidity
while (nextLockedNonce_ < firstExcludedIndex) {
    bytes32 requestId = getRequestId(asset, nextLockedNonce_);
    WithdrawalRequest storage request = withdrawalRequests[requestId];
    ...
    unchecked { nextLockedNonce_++; }
}
nextLockedNonce[asset] = nextLockedNonce_;
```

The loop always starts at the current head of the global queue and cannot skip entries. There is no mechanism to jump over attacker-controlled nonces. Every attacker request at nonces 0…N-1 must be processed before the legitimate user's request at nonce N can be unlocked.

### Impact Explanation
Legitimate users who have called `initiateWithdrawal` after the attacker cannot have their requests unlocked until all preceding attacker requests are processed. `completeWithdrawal` enforces that the user's oldest nonce is already below `nextLockedNonce[asset]`:

```solidity
if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();
```

Until the operator works through the stuffed queue, every honest user whose nonce falls after the attacker's block is unable to claim their ETH/LST. This constitutes a **temporary freezing of funds** (Medium severity). Additionally, each `unlockQueue` call that must traverse thousands of attacker entries constitutes **unbounded gas consumption** (Medium severity), making the operator's queue-processing transactions prohibitively expensive or impossible within a single block gas limit.

### Likelihood Explanation
The attack requires the attacker to hold rsETH equal to `N × minRsEthAmountToWithdraw[asset]`. The attacker does not lose these funds permanently — they are committed to the queue and will eventually be returned — so the economic cost is the opportunity cost of locked capital plus gas. Using multiple addresses (each depositing the minimum) multiplies the queue depth at proportional cost. Given that the protocol targets significant TVL and the attacker's capital is recoverable, the likelihood is **medium**.

### Recommendation
1. Enforce a per-user cap on the number of pending withdrawal requests per asset (analogous to `maxNumberOfWithdrawalsPerUser` already present in `KernelDepositPool.initiateWithdrawal` at line 323).
2. Alternatively, allow `unlockQueue` to accept a starting nonce so the operator can skip a contiguous range of attacker requests and unlock later legitimate requests out-of-order (requires per-request rather than global nonce tracking).
3. Increase `minRsEthAmountToWithdraw` to raise the capital cost of stuffing.

### Proof of Concept
1. Attacker acquires `K × minRsEthAmountToWithdraw[ETH]` rsETH across K addresses.
2. Each address calls `initiateWithdrawal(ETH, minRsEthAmountToWithdraw[ETH], "")`, inserting K entries at nonces 0…K-1.
3. Honest user calls `initiateWithdrawal`, receiving nonce K.
4. Operator calls `unlockQueue(ETH, K+1, ...)`. `_unlockWithdrawalRequests` loops from nonce 0 to K, consuming O(K) storage reads and writes. For large K this exceeds the block gas limit, causing the transaction to revert.
5. Even if the operator batches calls (e.g., `firstExcludedIndex = 100` per call), the honest user at nonce K cannot be unlocked until all K attacker nonces have been processed in prior calls.
6. The honest user's `completeWithdrawal` reverts with `WithdrawalLocked` until step 5 completes.

```text
    function _addUserWithdrawalRequest(address asset, uint256 rsETHUnstaked, uint256 expectedAssetAmount) internal {
        uint256 nextUnusedNonce_ = nextUnusedNonce[asset];

        // Generate a unique identifier for the new withdrawal request.
        bytes32 requestId = getRequestId(asset, nextUnusedNonce_);

        // Create and store the new withdrawal request.
        withdrawalRequests[requestId] = WithdrawalRequest({
            rsETHUnstaked: rsETHUnstaked, expectedAssetAmount: expectedAssetAmount, withdrawalStartBlock: block.number
        });

        // Map the user to the newly created request index and increment the nonce for future requests.
        userAssociatedNonces[asset][msg.sender].pushBack(nextUnusedNonce_);
        nextUnusedNonce[asset] = nextUnusedNonce_ + 1;

        emit AssetWithdrawalQueued(msg.sender, asset, rsETHUnstaked, nextUnusedNonce_);
```
