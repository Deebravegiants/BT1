### Title
Uninitialized `withdrawalDelay` and `maxNumberOfWithdrawalsPerUser` in `KernelDepositPool` Allow Withdrawal Delay Bypass and Cause Temporary Fund Freeze - (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary

`KernelDepositPool.initialize()` sets only the token addresses and admin role, leaving both `withdrawalDelay` and `maxNumberOfWithdrawalsPerUser` at their Solidity default of zero. Because `initiateWithdrawal()` uses both variables without first verifying they have been configured, the contract is immediately exploitable after deployment: the zero `maxNumberOfWithdrawalsPerUser` causes every withdrawal initiation to revert (freezing staked KERNEL), while a zero `withdrawalDelay` means any withdrawal that does succeed carries an `unlockTime` equal to `block.timestamp`, making the delay check in `claimWithdrawal()` trivially passable in the very next block.

### Finding Description

`KernelDepositPool.initialize()` only sets `kernelToken`, `rewardsToken`, and the admin role:

```solidity
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    // ...
    kernelToken = IERC20(_kernelToken);
    rewardsToken = IERC20(_rewardToken);
    // withdrawalDelay          → 0 (default)
    // maxNumberOfWithdrawalsPerUser → 0 (default)
}
``` [1](#0-0) 

**Root cause 1 — `maxNumberOfWithdrawalsPerUser == 0` freezes all withdrawals**

`initiateWithdrawal()` guards against exceeding the per-user limit:

```solidity
if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();
``` [2](#0-1) 

When `maxNumberOfWithdrawalsPerUser == 0`, the condition evaluates to `0 >= 0` → `true`, so every call to `initiateWithdrawal()` reverts with `WithdrawalLimitReached()`. No user can unstake their KERNEL tokens until the admin calls `setMaxNumberOfWithdrawalsPerUser()`.

**Root cause 2 — `withdrawalDelay == 0` nullifies the time-lock**

`initiateWithdrawal()` computes the unlock timestamp as:

```solidity
uint256 unlockTime = block.timestamp + withdrawalDelay;
``` [3](#0-2) 

When `withdrawalDelay == 0`, `unlockTime == block.timestamp`. The guard in `claimWithdrawal()` is:

```solidity
if (block.timestamp < withdrawal.unlockTime) revert WithdrawalNotReady();
``` [4](#0-3) 

Because `block.timestamp` is always `≥ block.timestamp`, this check is always false, so the withdrawal is immediately claimable in the same or any subsequent block — the intended delay is completely bypassed.

Neither `initiateWithdrawal()` nor `stake()` validates that these parameters have been configured before allowing user operations. [5](#0-4)

### Impact Explanation

**Primary — Medium: Temporary freezing of funds.**
Between deployment and the admin's call to `setMaxNumberOfWithdrawalsPerUser()`, every staked KERNEL token is frozen. Users who have already called `stake()` cannot retrieve their principal. The window is bounded only by admin responsiveness; there is no on-chain guarantee it will be closed promptly.

**Secondary — Low: Contract fails to deliver promised returns.**
If `setWithdrawalDelay()` is also not called before users begin staking, any withdrawal initiated during that window carries a zero delay. Those users permanently bypass the lockup that later stakers must respect, violating the protocol's fairness invariant.

### Likelihood Explanation

The contract is deployed and initialized in a single transaction that does not set either parameter. Any user who monitors the mempool or simply acts before the admin's follow-up configuration transactions can exploit both conditions. The attack requires no special privilege, no flash loan, and no complex setup — only the ability to call `stake()` and `initiateWithdrawal()` (or observe that `initiateWithdrawal()` reverts) immediately after deployment.

### Recommendation

Set both parameters to safe non-zero defaults inside `initialize()`:

```solidity
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    // ...
    withdrawalDelay = 7 days;                    // enforce a meaningful lockup
    maxNumberOfWithdrawalsPerUser = 10;           // allow withdrawals from day one
}
```

Alternatively, add explicit guards in `initiateWithdrawal()`:

```solidity
if (withdrawalDelay == 0) revert WithdrawalDelayNotSet();
if (maxNumberOfWithdrawalsPerUser == 0) revert WithdrawalLimitNotSet();
```

### Proof of Concept

```
// Scenario A — fund freeze (maxNumberOfWithdrawalsPerUser == 0)
1. Admin deploys and calls initialize() — withdrawalDelay = 0, maxNumberOfWithdrawalsPerUser = 0.
2. Alice calls stake(100e18) — succeeds, balanceOf[Alice] = 100e18.
3. Alice calls initiateWithdrawal(100e18):
   → userWithdrawalIds[Alice].length = 0
   → 0 >= 0 (maxNumberOfWithdrawalsPerUser) → true → revert WithdrawalLimitReached()
4. Alice's 100e18 KERNEL is frozen until admin calls setMaxNumberOfWithdrawalsPerUser(N).

// Scenario B — delay bypass (withdrawalDelay == 0, after admin sets maxNumberOfWithdrawalsPerUser)
1. Admin calls setMaxNumberOfWithdrawalsPerUser(10) but forgets setWithdrawalDelay().
2. Bob calls stake(100e18) at block T.
3. Bob calls initiateWithdrawal(100e18):
   → unlockTime = block.timestamp + 0 = block.timestamp (T)
4. Bob calls claimWithdrawal(id) in the next block (T+1):
   → block.timestamp (T+1) < unlockTime (T) → false → no revert
   → Bob receives 100e18 KERNEL immediately, bypassing any intended lockup.
```

    function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
        if (_amount == 0) revert AmountZero();
        if (balanceOf[msg.sender] < _amount) revert InsufficientStakedBalance();
        if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();

        balanceOf[msg.sender] -= _amount;
        totalKernelStaked -= _amount;

        // Create a withdrawal record
        uint256 withdrawalId = ++withdrawalCounter;
        uint256 unlockTime = block.timestamp + withdrawalDelay;

        withdrawals[withdrawalId] = Withdrawal({
            user: msg.sender, amount: _amount, unlockTime: unlockTime, claimed: false, withdrawalId: withdrawalId
        });
        userWithdrawalIds[msg.sender].push(withdrawalId);

        emit WithdrawalInitiated(msg.sender, _amount, withdrawalId, unlockTime);
    }
