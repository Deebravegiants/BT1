### Title
`feeBps` Initialized Without Bounds Check Enables Deposit Freeze or Zero-rsETH Minting - (`contracts/pools/RSETHPoolV3.sol`)

---

### Summary

Multiple L2 pool contracts accept `_feeBps` in their `initialize()` function without any upper-bound validation, while the corresponding `setFeeBps()` setter enforces a strict cap. If `feeBps` is initialized at or above `10_000` (100%), every subsequent `deposit()` call either reverts with arithmetic underflow (freezing all deposits) or silently mints zero rsETH to the depositor (draining their ETH into the contract).

---

### Finding Description

In `RSETHPoolV3.initialize()`, `feeBps` is stored directly from the constructor argument with no range check:

```solidity
// contracts/pools/RSETHPoolV3.sol:229
feeBps = _feeBps;   // no bounds check
```

The post-deployment setter enforces a cap of 1,000 bps (10%):

```solidity
// contracts/pools/RSETHPoolV3.sol:518-521
function setFeeBps(uint256 _feeBps) external onlyRole(DEFAULT_ADMIN_ROLE) {
    if (_feeBps > 1000) revert InvalidFeeAmount();
    feeBps = _feeBps;
```

The fee is consumed inside `viewSwapRsETHAmountAndFee`, which is called by every `deposit()` path:

```solidity
// contracts/pools/RSETHPoolV3.sol:300-307
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;   // underflows if feeBps > 10_000
...
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

The same pattern exists in:
- `RSETHPoolNoWrapper.initialize()` / `setFeeBps()` (cap: 10,000)
- `RSETHPoolV3WithNativeChainBridge.initialize()` / `setFeeBps()` (cap: 1,000)
- `RSETHPoolV2NBA.initialize()` / `setFeeBps()` (cap: 10,000)

---

### Impact Explanation

**Scenario A — `feeBps > 10,000`:**
`fee = amount * feeBps / 10_000 > amount`, so `amountAfterFee = amount - fee` underflows and reverts under Solidity 0.8 checked arithmetic. Every `deposit()` call reverts, freezing all user deposits until the admin calls `setFeeBps()`.

**Scenario B — `feeBps = 10,000` (100%):**
`fee = amount`, `amountAfterFee = 0`, `rsETHAmount = 0`. The depositor's ETH (or ERC-20 token) is transferred into the contract and `wrsETH.mint(msg.sender, 0)` is called — the user receives nothing. The ETH accumulates in the contract and is only accessible to the `BRIDGER_ROLE`, not the depositor. This constitutes direct, permanent loss of user funds.

---

### Likelihood Explanation

The `initialize()` function is called once at deployment. A deployer who accidentally passes `_feeBps = 10_000` (a natural-looking "100%" value) or any value above `10_000` triggers the vulnerability. The inconsistency between the initializer (no check) and the setter (strict cap) makes this a realistic deployment mistake. The setter's own guard proves the protocol considers values above the cap invalid — the initializer simply omits that guard.

---

### Recommendation

Apply the same bounds check in `initialize()` that `setFeeBps()` already enforces:

```solidity
// RSETHPoolV3.initialize()
if (_feeBps > 1000) revert InvalidFeeAmount();
feeBps = _feeBps;
```

Apply the equivalent cap to every pool contract's initializer, matching the cap used in its own `setFeeBps()`.

---

### Proof of Concept

1. Deploy `RSETHPoolV3` proxy and call `initialize(admin, bridger, wrsETH, 10_000, rsETHOracle, true)`.
2. `feeBps` is now `10_000`; `setFeeBps()` would have rejected this value.
3. Any user calls `deposit{value: 1 ether}("ref")`.
4. Inside `viewSwapRsETHAmountAndFee(1 ether)`:
   - `fee = 1e18 * 10_000 / 10_000 = 1e18`
   - `amountAfterFee = 1e18 - 1e18 = 0`
   - `rsETHAmount = 0`
5. `wrsETH.mint(msg.sender, 0)` executes — user's 1 ETH is in the contract, user receives 0 rsETH.
6. Repeat for every depositor; all ETH accumulates in the pool, redeemable only by `BRIDGER_ROLE`.
