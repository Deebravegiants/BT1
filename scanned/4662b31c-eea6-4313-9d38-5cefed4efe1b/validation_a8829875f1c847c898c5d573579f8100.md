### Title
Missing Zero-Value Guard in Messenger Contracts Allows Silent No-Op Bridge Calls - (`contracts/bridges/ArbitrumMessenger.sol`, `BaseMessenger.sol`, `OptimismMessenger.sol`, `UnichainMessenger.sol`, `ScrollMessenger.sol`)

---

### Summary

Five of the six messenger contracts lack a `value > 0` guard. When called with `value = 0` and `msg.value = 0`, the `MismatchedMsgValue` check passes (since `0 == 0`), and the underlying bridge is invoked with zero ETH. `LineaMessenger` explicitly has this guard, proving developer intent — the other five are inconsistently missing it.

---

### Finding Description

Every messenger contract's `sendETHToL1ViaBridge` contains only one validation:

```solidity
if (msg.value != value) revert MismatchedMsgValue();
``` [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

When `value = 0` and `msg.value = 0`, the condition `msg.value != value` evaluates to `false`, so no revert occurs. The bridge call proceeds with `{ value: 0 }`.

By contrast, `LineaMessenger` explicitly guards against this:

```solidity
if (value == 0) revert ZeroAmount();
if (msg.value != value) revert MismatchedMsgValue();
``` [6](#0-5) 

This inconsistency confirms the zero-value check is intentional design — it is simply absent from the other five implementations.

Additionally, none of the five affected messenger contracts have any access control (no `BRIDGER_ROLE` or equivalent modifier), so any external address can invoke `sendETHToL1ViaBridge` directly, not just the pool contracts that hold the `BRIDGER_ROLE`. [7](#0-6) 

---

### Impact Explanation

A zero-value call passes the only guard and forwards `{ value: 0 }` to the underlying bridge. Depending on the bridge implementation:

- If the bridge reverts on zero-value: the transaction reverts, wasting gas but causing no fund loss.
- If the bridge accepts zero-value: a zero-ETH bridge message is dispatched — a silent no-op that delivers nothing to L1.

No ETH from the messenger's balance is at risk because the bridge call uses `{ value: value }` (i.e., `{ value: 0 }`), not `{ value: address(this).balance }`. The impact is therefore **Low: contract fails to deliver promised returns, but doesn't lose value**.

---

### Likelihood Explanation

The messenger contracts are permissionless — any EOA or contract can call `sendETHToL1ViaBridge` directly. The zero-value path requires no special setup: simply call with `value = 0` and send `msg.value = 0`. The precondition is trivially satisfiable.

---

### Recommendation

Add a zero-value guard to all five affected messengers, mirroring `LineaMessenger`:

```solidity
if (value == 0) revert ZeroAmount();
if (msg.value != value) revert MismatchedMsgValue();
```

Optionally, add a `ZeroAmount` error to `IL2Messenger` so all implementations share a consistent interface. [8](#0-7) 

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import "forge-std/Test.sol";
import "contracts/bridges/ArbitrumMessenger.sol";

contract MockArbitrumBridge {
    uint256 public lastValue;
    function withdrawEth(address) external payable {
        lastValue = msg.value; // records 0 — no revert
    }
}

contract ZeroValueTest is Test {
    ArbitrumMessenger messenger;
    MockArbitrumBridge bridge;

    function setUp() public {
        messenger = new ArbitrumMessenger();
        bridge = new MockArbitrumBridge();
    }

    function test_zeroValueBypassesCheck() public {
        // value=0, msg.value=0 — MismatchedMsgValue does NOT revert
        messenger.sendETHToL1ViaBridge{value: 0}(address(bridge), address(this), 0);
        assertEq(bridge.lastValue(), 0); // bridge called with 0 ETH — silent no-op
    }
}
```

The same test applies to `BaseMessenger`, `OptimismMessenger`, `UnichainMessenger`, and `ScrollMessenger` by substituting the respective mock bridge interface.
