### Title
No Staleness Check on Chainlink Price Feed Allows Stale Prices to Drive Incorrect rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards every return value except `price`, performing zero staleness validation. Any Chainlink feed outage causes the protocol to mint rsETH at an incorrect rate, either diluting existing holders (inflated stale price) or short-changing depositors (deflated stale price).

---

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` is the canonical price source for all supported LST assets (stETH, rETH, etc.) in the LRT-rsETH protocol. The function calls `latestRoundData()` but discards `roundId`, `updatedAt`, and `answeredInRound` entirely:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
```

There is no check of the form `block.timestamp - updatedAt > threshold`, no `answeredInRound < roundId` guard, and no per-asset heartbeat mapping. The contract accepts whatever price the feed last reported, regardless of how old it is.

This is strictly worse than the reference vulnerability, which at least had a single shared threshold. Here there is no threshold at all.

The price returned by `getAssetPrice()` flows directly into rsETH minting math via `LRTOracle` → `LRTDepositPool.getRsETHAmountToMint()`. A stale price therefore directly controls how many rsETH tokens a depositor receives.

---

### Impact Explanation

**Inflated stale price scenario (High impact — theft of unclaimed yield):**
If a Chainlink feed for a supported LST freezes at a price above the true market price (e.g., the ETH/USD feed's documented 6-hour delay), a depositor calling `depositAsset()` will have their deposit valued higher than it actually is. They receive excess rsETH, diluting the proportional claim of every existing rsETH holder on the underlying TVL. This constitutes theft of unclaimed yield from existing holders.

**Deflated stale price scenario (Low impact — contract fails to deliver promised returns):**
If the feed freezes below the true price, depositors receive fewer rsETH than they are entitled to, causing a loss to the depositor without a corresponding gain to the protocol.

Both scenarios are reachable by any unprivileged depositor with no special access.

---

### Likelihood Explanation

Chainlink feed outages are documented real-world events. The ETH/USD feed experienced a 6-hour delay (https://cryptobriefing.com/chainlink-experiences-6-hour-delay-eth-price-feed/). The LRT-rsETH protocol supports multiple LST assets, each with its own feed and heartbeat. The absence of any staleness check means every supported feed is permanently exposed. Likelihood is **medium**: feed outages are rare but have occurred, and the protocol's multi-asset design multiplies the attack surface.

---

### Recommendation

1. Add a per-asset staleness threshold mapping to `ChainlinkPriceOracle`:
   ```solidity
   mapping(address asset => uint256 maxAge) public stalenessThreshold;
   ```
2. Enforce it in `getAssetPrice()`:
   ```solidity
   (, int256 price,, uint256 updatedAt,) = priceFeed.latestRoundData();
   if (block.timestamp - updatedAt > stalenessThreshold[asset]) revert StalePriceFeed();
   ```
3. Set each threshold slightly above the feed's documented heartbeat (e.g., 1 hour + buffer for ETH/USD, 24 hours + buffer for slower feeds).
4. Apply the same fix to `ChainlinkOracleForRSETHPoolCollateral.getRate()`, which relies only on the deprecated `answeredInRound < roundID` check and has no time-based guard.

---

### Proof of Concept

**Root cause — zero staleness validation:** [1](#0-0) 

`updatedAt`, `roundId`, and `answeredInRound` are all silently discarded. The returned `price` is used unconditionally.

**Price flows into rsETH minting:** [2](#0-1) 

`LRTOracle` delegates asset pricing to whichever oracle is registered, including `ChainlinkPriceOracle`, with no additional staleness layer.

**Secondary instance — `ChainlinkOracleForRSETHPoolCollateral` has no time-based check:** [3](#0-2) 

Only `answeredInRound < roundID` is checked (a deprecated Chainlink pattern); `block.timestamp - timestamp` is never validated against any threshold.

**Attack path:**
1. A Chainlink LST/ETH feed goes stale at an inflated price (e.g., stETH/ETH freezes 2% above true value during a network disruption).
2. Any user calls `LRTDepositPool.depositAsset(stETH, amount)`.
3. `LRTOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` returns the stale inflated price with no revert.
4. The user's deposit is over-valued; they receive excess rsETH.
5. Every existing rsETH holder's proportional claim on TVL is diluted by the excess minted tokens.
