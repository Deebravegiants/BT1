### Title
Decimal Mismatch in Token-to-rsETH Swap Formula Causes Massive Under-Minting for Non-18-Decimal Tokens - (File: contracts/pools/RSETHPoolV3.sol)

### Summary
The `viewSwapRsETHAmountAndFee(amount, token)` function in all L2 pool contracts computes the rsETH output by multiplying the raw token amount directly against a 1e18-normalised oracle rate, without adjusting for the token's actual decimal count. If any supported ERC-20 token has fewer than 18 decimals, depositors permanently lose the overwhelming majority of their deposited value.

### Finding Description
Every L2 pool variant (`RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolNoWrapper`, `RSETHPool`, `RSETHPoolV3WithNativeChainBridge`) computes the rsETH mint amount for a token deposit as:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

`tokenToETHRate` is always returned normalised to 1e18 by `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
```

`rsETHToETHrate` is likewise 1e18-normalised. The formula therefore implicitly assumes `amountAfterFee` is also expressed in 18 decimals. When a token with `d < 18` decimals is deposited, `amountAfterFee` carries only `d` decimal places, so the result is `10^(18-d)` times smaller than the correct rsETH amount.

The same inversion applies to the reverse-swap path `viewSwapAssetToPremintedRsETH`:

```solidity
tokenAmount = rsETHAmount * rsETHToETHrate / tokenToETHRate;
```

Here the output `tokenAmount` is inflated by `10^(18-d)`, allowing the caller to drain the pool's token balance.

### Impact Explanation
For a 6-decimal token (e.g. USDC):

| Direction | Expected | Actual | Error factor |
|---|---|---|---|
| Deposit 1000 USDC → rsETH | ~0.317 rsETH | ~3.17×10⁻¹⁰ rsETH | ×10⁻⁹ |
| Swap 1 rsETH → USDC | ~3150 USDC | ~3.15×10¹⁵ USDC | ×10¹² |

A depositor permanently loses their entire token balance while receiving a negligible rsETH amount. The reverse-swap path drains the pool's entire token reserve. Both outcomes constitute direct, permanent theft of user funds — **Critical** impact.

### Likelihood Explanation
The pools expose an `addSupportedToken` function gated to the admin/timelock role. No currently deployed supported token has fewer than 18 decimals (wstETH, WETH, etc. are all 18-decimal). However, the protocol is explicitly designed to be extensible to new collateral types, and the code contains no guard preventing a sub-18-decimal token from being registered. The moment any such token is added through a legitimate governance action, every depositor using that token is immediately affected. Likelihood is **Medium** (requires one governance action; no attacker capability needed beyond that).

### Recommendation
Normalise `amountAfterFee` to 18 decimals before applying the rate formula:

```solidity
uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 normalizedAmount = amountAfterFee * 10 ** (18 - tokenDecimals);
rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
```

Apply the inverse adjustment in `viewSwapAssetToPremintedRsETH` to convert the 18-decimal result back to the token's native decimals. Alternatively, add an explicit `require(IERC20Metadata(token).decimals() == 18)` check inside `addSupportedToken` to prevent non-18-decimal tokens from ever being registered.

### Proof of Concept

**Entry path**: `deposit(token, amount, referralId)` in any L2 pool contract with a registered non-18-decimal token.

**Step-by-step**:
1. Admin legitimately calls `addSupportedToken(usdcAddress, chainlinkUsdcOracle, bridge)`.
2. User calls `deposit(usdcAddress, 1000e6, "ref")` (1000 USDC).
3. `viewSwapRsETHAmountAndFee(1000e6, usdcAddress)` executes:
   - `fee = 1000e6 * feeBps / 10_000` (small)
   - `amountAfterFee ≈ 1000e6`
   - `tokenToETHRate = chainlink_USDC_ETH * 1e18 / 1e8 ≈ 3.33e14` (USDC ≈ 1/3000 ETH)
   - `rsETHToETHrate ≈ 1.05e18`
   - `rsETHAmount = 1000e6 * 3.33e14 / 1.05e18 ≈ 3.17e8`
4. User receives `3.17e8` wei of wrsETH (≈ 3.17×10⁻¹⁰ rsETH) instead of the correct `≈ 3.17e17` (≈ 0.317 rsETH).
5. User's 1000 USDC is permanently held by the pool; they have no recourse.
