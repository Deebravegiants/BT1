### Title
Temporary Freezing of User rsETH: `getAvailableAssetAmount` Counts EigenLayer-Queued Assets Not Yet in `LRTUnstakingVault` — (`contracts/LRTWithdrawalManager.sol`)

### Summary
`initiateWithdrawal` permits users to commit withdrawals against assets that are still queued in EigenLayer's withdrawal system (`assetUnstakingFromEigenLayer`), but `unlockQueue` can only service those requests from the physical balance of `LRTUnstakingVault`. The user's rsETH is locked in `LRTWithdrawalManager` with no cancellation mechanism until EigenLayer withdrawal completes.

### Finding Description
`getAvailableAssetAmount` computes availability as:

```
totalAssets (from getTotalAssetDeposits) - assetsCommitted[asset]
```

`getTotalAssetDeposits` sums six buckets including `assetUnstakingFromEigenLayer` — assets for which EigenLayer withdrawal has been queued but not yet completed and transferred to `LRTUnstakingVault`. [1](#0-0) 

`initiateWithdrawal` accepts the request and increments `assetsCommitted[asset]` if `expectedAssetAmount <= getAvailableAssetAmount(asset)`. [2](#0-1) 

However, `unlockQueue` derives `totalAvailableAssets` exclusively from `unstakingVault.balanceOf(asset)` — the vault's actual on-hand balance: [3](#0-2) 

If the vault balance is zero (assets still in EigenLayer queue), `unlockQueue` reverts: [4](#0-3) 

Even with a partial vault balance, `_unlockWithdrawalRequests` exits early when `availableAssetAmount < payoutAmount`: [5](#0-4) 

There is no `cancelWithdrawal` function in `LRTWithdrawalManager`, so the user cannot reclaim their rsETH.

### Impact Explanation
User rsETH is transferred into `LRTWithdrawalManager` at `initiateWithdrawal` time and cannot be recovered until `unlockQueue` successfully processes the request. If the committed assets are still in EigenLayer's queue, `unlockQueue` will fail or process zero requests, leaving the user's rsETH locked for the duration of EigenLayer's withdrawal delay (currently ~7 days on mainnet, configurable by EigenLayer governance). **Impact: Medium — Temporary freezing of funds.**

### Likelihood Explanation
This is reachable in normal protocol operation. Operators routinely queue EigenLayer withdrawals to service user unstaking. Any user who calls `initiateWithdrawal` while the relevant assets are in EigenLayer's queue (but not yet in the vault) will experience this lockup. No special permissions or adversarial setup are required — only normal timing between EigenLayer withdrawal initiation and completion.

### Recommendation
`getAvailableAssetAmount` should exclude `assetUnstakingFromEigenLayer` from the available amount, or alternatively use `unstakingVault.balanceOf(asset)` (minus already-committed amounts) as the availability check in `initiateWithdrawal`, consistent with what `unlockQueue` actually uses. This ensures users can only commit against assets that are physically present and redeemable from the vault.
