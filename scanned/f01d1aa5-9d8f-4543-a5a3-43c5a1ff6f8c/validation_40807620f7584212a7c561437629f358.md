### Title
Sequential Division Precision Loss in Protocol Fee Minting Causes Systematic Treasury Underpayment - (File: contracts/LRTOracle.sol)

### Summary
`LRTOracle._updateRsETHPrice()` computes the rsETH protocol fee mint amount through three sequential integer divisions. Each division truncates its remainder, and the errors compound in the same direction on every call, guaranteeing the protocol treasury is always minted fewer rsETH tokens than it is mathematically owed.

### Finding Description
Inside `_updateRsETHPrice()`, the protocol fee is computed in three chained steps, each performing an integer division that discards its remainder:

**Step 1** — `protocolFeeInETH` is computed with a division that truncates:
```solidity
// contracts/LRTOracle.sol line 246
protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
```

**Step 2** — `newRsETHPrice` is computed using the already-truncated `protocolFeeInETH` as a subtrahend, then divided again (`divWad` = `a * 1e18 / b`):
```solidity
// contracts/LRTOracle.sol line 250
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

**Step 3** — `rsethAmountToMintAsProtocolFee` is computed by dividing the truncated `protocolFeeInETH` by the inflated `newRsETHPrice`:
```solidity
// contracts/LRTOracle.sol line 301
uint256 rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice);
```

The compounding effect:
- Step 1 truncation makes `protocolFeeInETH` smaller than the true value → the numerator of Step 3 is too small.
- Because `protocolFeeInETH` is too small, `totalETHInProtocol - protocolFeeInETH` is too large → `newRsETHPrice` (the denominator of Step 3) is too large.
- Both errors push `rsethAmountToMintAsProtocolFee` downward simultaneously, guaranteeing the treasury always receives less than the correct fee.

The same two-division pattern also appears in every L2 pool's `viewSwapRsETHAmountAndFee`:
```solidity
// contracts/pools/RSETHPool.sol lines 312, 319
fee = amount * feeBps / 10_000;          // Division 1: fee truncated downward
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;  // Division 2: uses inflated amountAfterFee
```
causing depositors to receive marginally more rsETH than they should on every swap, at the expense of the fee pool.

The consolidated fix for `_updateRsETHPrice()` is to defer all divisions to the end:
```solidity
// Multiply first, divide once
uint256 numerator = rewardAmount.mulWad(uint256(lrtConfig.protocolFeeInBPS()) * 1e14).mulWad(rsethSupply);
uint256 denominator = totalETHInProtocol
    - (rewardAmount * lrtConfig.protocolFeeInBPS() / 10_000);
rsethAmountToMintAsProtocolFee = numerator / denominator;
```

### Impact Explanation
On every `updateRSETHPrice()` call where `totalETHInProtocol > previousTVL`, the protocol treasury is minted fewer rsETH tokens than it is entitled to. The per-call shortfall is bounded by a small number of wei, but it is strictly one-directional and accumulates monotonically across the protocol's lifetime. No user funds are at risk; the loss is entirely borne by the protocol treasury in the form of uncollected protocol yield.

**Impact: Low** — Contract fails to deliver the protocol's promised fee returns to the treasury without losing user value.

### Likelihood Explanation
`updateRSETHPrice()` is a public, permissionless function callable by any address. It is expected to be invoked frequently by keepers or any participant. Every invocation where rewards have accrued triggers the three-step division chain. Likelihood is **High**.

### Recommendation
Refactor `_updateRsETHPrice()` to consolidate all multiplications before the final division, eliminating intermediate truncations. Compute `rsethAmountToMintAsProtocolFee` directly from the primary inputs (`rewardAmount`, `protocolFeeInBPS`, `rsethSupply`, `totalETHInProtocol`) in a single formula. Apply the same principle to `viewSwapRsETHAmountAndFee` across all pool contracts by computing `rsETHAmount = amount * (10_000 - feeBps) * 1e18 / (10_000 * rsETHToETHrate)` without the intermediate `fee` subtraction step.

### Proof of Concept

Assume:
- `rewardAmount = 1e18 + 3` (1 ETH + 3 wei, chosen so `rewardAmount * 1000 % 10_000 = 3000`, causing truncation)
- `protocolFeeInBPS = 1000` (10%)
- `totalETHInProtocol = 100_001e18 + 3`
- `rsethSupply = 95_000e18`

**Step 1 (current):**
`protocolFeeInETH = ((1e18 + 3) * 1000) / 10_000 = (1e21 + 3000) / 10_000 = 1e17` (truncates 3000/10_000 → loses 0.3 wei, rounded to 0)

**Step 1 (correct):**
True fee = `(1e18 + 3) * 1000 / 10_000 = 1e17 + 0.3 wei` → should be `1e17 + 1` after rounding

**Step 3 (current):**
`rsethAmountToMintAsProtocolFee = 1e17 * 1e18 / newRsETHPrice`

Because `protocolFeeInETH` is truncated, `newRsETHPrice` is inflated (denominator too large) and `protocolFeeInETH` is deflated (numerator too small). Both errors reduce the final mint amount below the correct value. Over thousands of calls, the treasury accumulates a deficit of protocol fees that it can never recover.
