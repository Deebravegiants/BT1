### Title
Missing Chainlink Price Validation Allows Circuit-Breaker Price Abuse in Deposits - (`contracts/oracles/ChainlinkPriceOracle.sol`)

### Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but performs zero validation on the returned price: no staleness check, no negative/zero price guard, and no min/max circuit-breaker bounds check. The original report's root cause is that cached Chainlink `minAnswer`/`maxAnswer` bounds become stale when the aggregator is rotated; the analog here is that no bounds are checked at all, leaving the protocol fully exposed to Chainlink circuit-breaker prices.

### Finding Description
`ChainlinkPriceOracle.getAssetPrice()` reads the raw Chainlink price and returns it without any sanity checks:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

There is no check that `price > 0`, no heartbeat/staleness check on `updatedAt`, and no comparison against the aggregator's `minAnswer`/`maxAnswer` circuit-breaker bounds. Chainlink price feeds have hard-coded circuit breakers: when an asset's true market price falls below `minAnswer`, the feed returns `minAnswer` rather than the real price. This is exactly the LUNA-style scenario Chainlink documents warn about.

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` in the same repo does validate `answeredInRound < roundID`, `timestamp == 0`, and `ethPrice <= 0`, demonstrating the team is aware of these checks but omitted them from the primary oracle path. [2](#0-1) 

The unvalidated price flows directly into `LRTOracle.getAssetPrice()`: [3](#0-2) 

Which is consumed by `LRTDepositPool.getRsETHAmountToMint()`: [4](#0-3) 

And by `_getTotalEthInProtocol()` used in every `updateRSETHPrice()` call: [5](#0-4) 

### Impact Explanation
**Critical — Protocol insolvency / direct theft of user funds.**

If a supported LST (e.g., stETH, rETH) depegs severely, Chainlink's circuit breaker clamps the reported price at `minAnswer` (e.g., 0.9 ETH) while the true market price is far lower (e.g., 0.3 ETH). An attacker can:

1. Buy the depegged LST cheaply on the open market (0.3 ETH per token).
2. Call `depositAsset()` on `LRTDepositPool` — the protocol prices the LST at 0.9 ETH via the unvalidated oracle.
3. Receive rsETH minted at the inflated 0.9 ETH rate.
4. Redeem rsETH for other protocol assets at fair value.

The attacker extracts 3× the value they deposited, draining honest depositors' collateral and rendering the protocol insolvent. Additionally, if `price` is 0 or negative (possible during oracle failure), `uint256(price)` either returns 0 (causing division-by-zero or zero rsETH minted, freezing deposits) or wraps to `2^256 - 1` (causing unbounded rsETH minting).

### Likelihood Explanation
**Medium.** Chainlink circuit breakers are a documented, real-world behavior triggered during extreme market events (the LUNA collapse is the canonical example). The protocol supports multiple LSTs, each with its own Chainlink feed and circuit-breaker bounds. Any one of them hitting a circuit breaker during a depeg event triggers this path. No privileged access is required — any depositor can exploit it.

### Recommendation
In `ChainlinkPriceOracle.getAssetPrice()`, add the following checks after calling `latestRoundData()`:

1. **Staleness**: `require(updatedAt >= block.timestamp - HEARTBEAT, "Stale price")`.
2. **Negative/zero price**: `require(price > 0, "Invalid price")`.
3. **Circuit-breaker bounds**: Fetch `aggregator.minAnswer()` and `aggregator.maxAnswer()` dynamically on every call (not cached at construction, per the original report's lesson) and assert `price > minAnswer && price < maxAnswer`.

### Proof of Concept

1. Assume stETH is a supported asset with a Chainlink feed whose `minAnswer` = 0.9e18 (90% of ETH).
2. stETH depegs to 0.3 ETH on the open market; Chainlink circuit breaker activates, feed returns 0.9e18.
3. Attacker buys 1000 stETH for 300 ETH.
4. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
5. `getRsETHAmountToMint` calls `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns 0.9e18 (circuit-breaker value, no validation).
6. rsETH minted = `1000e18 * 0.9e18 / rsETHPrice` — attacker receives rsETH worth ~900 ETH.
7. Attacker redeems rsETH for 900 ETH worth of other assets, netting ~600 ETH profit at the expense of honest depositors. [1](#0-0) [6](#0-5)
