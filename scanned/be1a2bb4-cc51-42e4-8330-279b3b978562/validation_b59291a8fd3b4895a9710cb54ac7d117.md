### Title
Asymmetric Fee Structure Between ETH and Token Deposits Allows Fee Bypass - (File: contracts/pools/RSETHPool.sol)

### Summary
`RSETHPool.sol` (Arbitrum pool) uses two separate fee-rate variables: a global `feeBps` for ETH deposits and a per-token `tokenFeeBps[token]` mapping for LST/ERC-20 deposits. The `tokenFeeBps` mapping is never initialized when a token is added and defaults to `0`. Because depositing ETH and depositing an equivalent-value LST (e.g. wstETH) both produce the same rsETH output, users can route through the token-deposit path to pay zero fee instead of `feeBps`, directly reducing protocol revenue.

### Finding Description
`RSETHPool.sol` maintains two distinct fee-rate variables: [1](#0-0) [2](#0-1) 

ETH deposits use `feeBps`: [3](#0-2) 

Token deposits use `tokenFeeBps[token]`: [4](#0-3) 

`tokenFeeBps` is a plain mapping that is **never written** during `addSupportedToken` or in any initializer, so it silently defaults to `0` for every token: [5](#0-4) 

The admin can later call `setTokenFeeBps`, but until that call is made the asymmetry is live and exploitable: [6](#0-5) 

Because `rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate`, depositing `X ETH` worth of wstETH yields the same rsETH as depositing `X ETH` directly, yet the wstETH path charges `fee = 0` while the ETH path charges `fee = X * feeBps / 10_000`. The two paths are economically equivalent but structurally asymmetric in fee cost — the direct analog of the external report's root cause.

### Impact Explanation
Any depositor can avoid the protocol deposit fee entirely by depositing a supported LST (wstETH, ETHx, stETH, etc.) instead of ETH. For every unit of rsETH minted through the token path, the protocol collects zero fee instead of `feeBps` basis points. This is a direct, ongoing loss of protocol fee revenue — **theft of unclaimed yield** — with no cap on the amount that can be extracted.

### Likelihood Explanation
The condition is trivially met: every token added via `addSupportedToken` starts with `tokenFeeBps = 0`. Any unprivileged depositor who holds or can acquire a supported LST (all of which are freely tradeable on-chain) can exploit this immediately. No special role, oracle manipulation, or governance action is required. The pool is live on Arbitrum mainnet and the entry point (`deposit(address token, uint256 amount, string referralId)`) is fully public.

### Recommendation
Require an explicit fee-rate argument when adding a supported token, or default `tokenFeeBps[token]` to `feeBps` inside `addSupportedToken`. At minimum, add a check that `tokenFeeBps[token] != 0` (or that it has been explicitly set) before allowing deposits for that token.

### Proof of Concept
Assume `feeBps = 100` (1 %) and wstETH/ETH oracle rate = 1.2e18.

**ETH path** — deposit 1.2 ETH:
- `fee = 1.2e18 * 100 / 10_000 = 0.012 ETH` → treasury receives 0.012 ETH
- `rsETHAmount = (1.2e18 - 0.012e18) * 1e18 / rsETHToETHrate`

**wstETH path** — deposit 1 wstETH (≡ 1.2 ETH):
- `feeBpsForToken = tokenFeeBps[wstETH] = 0`
- `fee = 0` → treasury receives **0**
- `rsETHAmount = 1e18 * 1.2e18 / rsETHToETHrate` (identical output)

The depositor receives the same rsETH in both cases but pays 0.012 ETH less in fees via the wstETH path. At scale (e.g. 1 000 wstETH deposit) the protocol loses 10 wstETH (~12 ETH) in fee revenue per transaction.
