### Title
Daily Mint Limit Bypass via `RSETHPoolNoWrapper.deposit()` — Missing Rate-Limiting Check in Parallel Deposit Code Path (File: `contracts/pools/RSETHPoolNoWrapper.sol`)

---

### Summary

`RSETHPoolNoWrapper` is a parallel deposit code path for chains without an rsETH wrapper (e.g., Arbitrum, Unichain). Every other pool variant enforces a `limitDailyMint` modifier on its `deposit()` functions. `RSETHPoolNoWrapper` omits this check entirely, allowing any unprivileged depositor to drain the pool's pre-minted rsETH balance in a single transaction, bypassing the intended daily distribution cap.

---

### Finding Description

All pool contracts in the protocol implement a daily minting limit as a critical rate-limiting security control. The `limitDailyMint` modifier tracks how much rsETH has been distributed in the current day and reverts if the cap is exceeded.

**Contracts with `limitDailyMint` enforced on `deposit()`:**

- `RSETHPoolV3.deposit()` — `limitDailyMint(msg.value, ETH_IDENTIFIER)` [1](#0-0) 
- `RSETHPoolV3ExternalBridge.deposit()` — `limitDailyMint(msg.value, ETH_IDENTIFIER)` [2](#0-1) 
- `RSETHPoolV3WithNativeChainBridge.deposit()` — `limitDailyMint(msg.value, ETH_IDENTIFIER)` [3](#0-2) 
- `RSETHPoolV2ExternalBridge.deposit()` — `limitDailyMint(msg.value)` [4](#0-3) 
- `RSETHPoolV2.deposit()` — `limitDailyMint(msg.value)` [5](#0-4) 

**`RSETHPoolNoWrapper` — both `deposit()` overloads are missing `limitDailyMint`:**

```solidity
function deposit(string memory referralId) external payable nonReentrant whenNotPaused {
``` [6](#0-5) 

```solidity
function deposit(address token, uint256 amount, string memory referralId)
    external nonReentrant whenNotPaused onlySupportedToken(token) {
``` [7](#0-6) 

`RSETHPoolNoWrapper` holds a stock of canonical rsETH OFT tokens and transfers them directly to depositors at the oracle rate. [8](#0-7)  Neither deposit function checks `dailyMintLimit`, `dailyMintAmount`, or `lastMintDay` — and the contract declares none of these state variables, confirming the check was never implemented for this code path.

---

### Impact Explanation

An unprivileged depositor can call `RSETHPoolNoWrapper.deposit()` with an arbitrarily large ETH amount (or supported token amount) in a single transaction, receiving all pre-minted rsETH held by the pool at the oracle rate. This:

1. **Drains the pool's rsETH balance in one transaction**, bypassing the daily cap that all other pool variants enforce.
2. **Temporarily freezes the deposit functionality** for all other users on that chain — subsequent depositors receive nothing until the pool is refilled with rsETH by the bridger.
3. **Bypasses the intended security invariant** that no single day should see more than `dailyMintLimit` rsETH distributed from any L2 pool.

Impact classification: **Medium — Temporary freezing of funds** (deposit functionality unavailable to other users until the pool is manually refilled).

---

### Likelihood Explanation

The entry point is fully public and requires no special role. Any depositor with sufficient ETH (or a supported token) on the affected chain (Arbitrum, Unichain) can trigger this in a single transaction. No front-running, governance capture, or privileged access is required.

---

### Recommendation

Add the `limitDailyMint` modifier (and the corresponding state variables: `dailyMintLimit`, `dailyMintAmount`, `lastMintDay`, `startTimestamp`) to both `deposit()` overloads in `RSETHPoolNoWrapper`, mirroring the implementation in `RSETHPoolV3`. The modifier logic is identical across all other pool variants and should be shared or replicated here:

```solidity
function deposit(string memory referralId)
    external payable nonReentrant whenNotPaused
    limitDailyMint(msg.value, ETH_IDENTIFIER)   // ADD THIS
{
    ...
}

function deposit(address token, uint256 amount, string memory referralId)
    external nonReentrant whenNotPaused onlySupportedToken(token)
    limitDailyMint(amount, token)               // ADD THIS
{
    ...
}
```

A `reinitializer` should be used to set `dailyMintLimit` and `startTimestamp` on upgrade.

---

### Proof of Concept

1. Identify the `RSETHPoolNoWrapper` deployment on Arbitrum or Unichain.
2. Query `rsETH.balanceOf(poolAddress)` — suppose it holds `X` rsETH.
3. Compute the ETH required to receive `X` rsETH at the current oracle rate: `ethNeeded = X * getRate() / 1e18`.
4. Call `RSETHPoolNoWrapper.deposit{value: ethNeeded}("")`.
5. Observe that the entire rsETH balance is transferred to the caller in one transaction, with no `DailyMintLimitExceeded` revert.
6. Subsequent depositors calling `deposit()` receive 0 rsETH (pool is empty) until the bridger refills it.

This demonstrates the daily cap is fully bypassed through the `RSETHPoolNoWrapper` code path, while the identical deposit action on `RSETHPoolV3` would revert with `DailyMintLimitExceeded` once the cap is reached. [9](#0-8)
