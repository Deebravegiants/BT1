### Title
Stale `rsETHPrice` Allows Depositors to Mint Excess rsETH, Stealing Accrued Yield from Existing Holders - (File: contracts/LRTDepositPool.sol / contracts/LRTOracle.sol)

### Summary
`LRTOracle.rsETHPrice` is a stored value updated only on explicit calls to the public `updateRSETHPrice()`. `LRTDepositPool` reads this stale value directly when minting rsETH. When protocol rewards accrue (TVL rises), the stored price lags behind the true value, allowing any depositor to mint more rsETH than they are entitled to, then trigger the price update — capturing yield that belongs to existing holders.

### Finding Description
`LRTDepositPool.getRsETHAmountToMint()` computes the rsETH to mint as:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [1](#0-0) 

`lrtOracle.rsETHPrice()` is a **cached storage variable** — it is only updated when `updateRSETHPrice()` is explicitly called:

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [2](#0-1) 

The true price is computed inside `_updateRsETHPrice()` from live TVL:

```solidity
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
``` [3](#0-2) 

Between reward accruals (EigenLayer restaking rewards, stETH rebases, etc.) and the next `updateRSETHPrice()` call, `rsETHPrice` is **lower than the true value**. Because the mint formula divides by `rsETHPrice`, a lower denominator yields a larger `rsethAmountToMint`. The attacker deposits during this window, receives excess rsETH, then calls `updateRSETHPrice()` themselves to crystallise the gain.

The `pricePercentageLimit` guard only blocks price increases **above** the configured threshold from non-manager callers:

```solidity
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
if (isPriceIncreaseOffLimit) {
    if (!IAccessControl(address(lrtConfig)).hasRole(LRTConstants.MANAGER, msg.sender)) {
        revert PriceAboveDailyThreshold();
    }
}
``` [4](#0-3) 

For any price increase **within** the threshold (routine daily reward accrual), any address can call `updateRSETHPrice()` and complete the attack. The attack is repeatable every reward cycle.

### Impact Explanation
Every time rewards accrue and `rsETHPrice` is stale, an attacker can deposit and receive rsETH at a discount relative to the true NAV. When the price is subsequently updated, the attacker's rsETH is worth more than what they paid. The surplus comes directly from the share of existing rsETH holders, whose proportional claim on the protocol's TVL is diluted. This constitutes **theft of unclaimed yield** (High impact).

### Likelihood Explanation
- `updateRSETHPrice()` is not called atomically with deposits; there is always a window between reward accrual and the next price update.
- The attacker needs no special role — only a standard EOA and ETH/LST to deposit.
- Reward accrual events (EigenLayer distributions, stETH rebases) are publicly observable on-chain, making the timing straightforward to monitor.
- The `pricePercentageLimit` caps per-update profit but does not prevent the attack; it can be repeated each period.

Likelihood: **Medium** (requires monitoring but no privileged access).

### Recommendation
Force a price update before computing the mint amount in `depositETH` and `depositAsset`, or at minimum inside `getRsETHAmountToMint`:

```solidity
function getRsETHAmountToMint(address asset, uint256 amount) public view override returns (uint256) {
    address lrtOracleAddress = lrtConfig.getContract(LRTConstants.LRT_ORACLE);
    ILRTOracle lrtOracle = ILRTOracle(lrtOracleAddress);
    // Ensure price is fresh before minting
    lrtOracle.updateRSETHPrice(); // make this a state-changing call, not view
    rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
}
```

Alternatively, enforce a maximum staleness window on `rsETHPrice` and revert deposits if the price has not been updated within that window.

### Proof of Concept
1. EigenLayer rewards accrue; `_getTotalEthInProtocol()` would now return a value higher than `rsethSupply * rsETHPrice` (i.e., true price > stored `rsETHPrice`).
2. Attacker calls `LRTDepositPool.depositETH{value: 100 ether}(minAmount, "")`. Because `rsETHPrice` is stale-low, the attacker receives `rsETHAmount = 100e18 * 1e18 / staledLowPrice` — more rsETH than the true NAV warrants.
3. Attacker calls `LRTOracle.updateRSETHPrice()`. The price rises to the true value; the attacker's rsETH balance now represents more ETH than they deposited.
4. Attacker calls `LRTWithdrawalManager.initiateWithdrawal()` and eventually `completeWithdrawal()`, recovering more ETH than deposited. The difference is extracted from existing holders' proportional share. [5](#0-4) [6](#0-5)

    function _updateRsETHPrice() internal {
        address rsETHTokenAddress = lrtConfig.rsETH();
        uint256 rsethSupply = IRSETH(rsETHTokenAddress).totalSupply();

        if (rsethSupply == 0) {
            rsETHPrice = 1 ether;
            highestRsethPrice = 1 ether;
            return;
        }

        if (highestRsethPrice == 0) {
            highestRsethPrice = rsETHPrice;
        }

        uint256 previousPrice = rsETHPrice;

        // get total ETH in the protocol (normalized to 1e18)
        uint256 totalETHInProtocol = _getTotalEthInProtocol();

        // calculate previousTVL using rsethSupply multiplied by rsETHPrice
        uint256 previousTVL = rsethSupply.mulWad(rsETHPrice);

        IPausable lrtDepositPool = IPausable(lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL));
        IPausable withdrawalManager = IPausable(lrtConfig.getContract(LRTConstants.LRT_WITHDRAW_MANAGER));

        // determine if the protocol is active (not paused)
        bool protocolPaused = lrtDepositPool.paused() || withdrawalManager.paused() || paused;

        // only take fee if TVL increased and protocol is not paused
        uint256 protocolFeeInETH = 0;
        if (!protocolPaused && totalETHInProtocol > previousTVL) {
            uint256 rewardAmount = totalETHInProtocol - previousTVL;
            protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
        }

        // compute new rsETH price based on total ETH minus fee
        uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
