### Title
Aave WETH Reserve Deactivation Permanently Blocks All Queued ETH Withdrawals With No Admin Escape Hatch - (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary

When `isAaveIntegrationEnabled` is `true` and ETH has been deposited to Aave, any transition of the Aave WETH reserve to `isActive=false` (a legitimate Aave governance risk-parameter action) causes every call to `completeWithdrawal(ETH_TOKEN)` to revert with no recovery path available to the protocol. Critically, every admin escape hatch (`emergencyWithdrawFromAave`, `setAaveIntegrationEnabled(false)`) also routes through the same failing `_withdrawFromAave` call, leaving all queued ETH withdrawal funds frozen until Aave governance re-activates the reserve.

---

### Finding Description

**Vulnerable code path:**

`completeWithdrawal` → `_processWithdrawalCompletion` → `_withdrawFromAave` → `aaveWETHGateway.withdrawETH`

In `_processWithdrawalCompletion`, when `isAaveIntegrationEnabled && asset == ETH_TOKEN` and the contract's native ETH balance is insufficient to cover the request, the code calls `_withdrawFromAave` with **no try/catch**: [1](#0-0) 

`_withdrawFromAave` then calls `aaveWETHGateway.withdrawETH` with **no try/catch**: [2](#0-1) 

If the Aave WETH reserve is inactive, `withdrawETH` reverts at the Aave pool level, propagating the revert all the way up through `completeWithdrawal`.

**Contrast with the deposit path:** `unlockQueue` wraps its Aave deposit in a `try/catch`, explicitly tolerating Aave unavailability and emitting `AaveDepositFailed`: [3](#0-2) 

No equivalent defensive pattern exists for withdrawals.

**All admin escape hatches are also broken:**

- `emergencyWithdrawFromAave` (callable by `PAUSER_ROLE`) calls `_withdrawFromAave` internally — same revert: [4](#0-3) 

- `setAaveIntegrationEnabled(false)` calls `_withdrawFromAave` **before** setting `isAaveIntegrationEnabled = false`, so it also reverts and the flag is never cleared: [5](#0-4) 

There is no function that sets `isAaveIntegrationEnabled = false` without first attempting to drain Aave. Once the reserve is inactive, the protocol is stuck.

---

### Impact Explanation

All queued ETH withdrawal requests that require Aave liquidity (i.e., where `address(this).balance < request.expectedAssetAmount`) become permanently uncompletable until Aave governance re-activates the WETH reserve. This affects every user in the ETH withdrawal queue simultaneously. The funds are not lost (aWETH balances remain), but they are inaccessible — matching **Medium: Temporary freezing of funds**.

---

### Likelihood Explanation

Aave governance has historically set reserves to `isActive=false` for risk management (e.g., deprecated assets, emergency risk actions). The WETH reserve is high-value and unlikely to be deactivated under normal conditions, making this a low-probability event. However, the complete absence of any working recovery path means even a brief deactivation freezes all queued ETH withdrawals with zero protocol recourse. Likelihood: **Low**. Combined with the broad impact across all queued users: **Medium** overall.

---

### Recommendation

1. **Wrap `_withdrawFromAave` in `_processWithdrawalCompletion` with try/catch**, mirroring the deposit path. On failure, leave the request in place (do not `delete withdrawalRequests[requestId]` or decrement `unlockedWithdrawalsCount`) and revert with a descriptive error so users can retry later.

2. **Add a force-disable function** that sets `isAaveIntegrationEnabled = false` without attempting to withdraw from Aave, callable by `PAUSER_ROLE` or admin. This allows the protocol to unblock the withdrawal path even when Aave is unavailable, with the Aave balance recoverable separately once the reserve is re-activated.

3. **Check `isActive` via `aaveDataProvider.getReserveConfigurationData`** before attempting any Aave interaction, and skip/fallback gracefully if the reserve is inactive.

---

    function _withdrawFromAave(uint256 amount) internal returns (uint256 withdrawnAmount) {
        if (amount == 0) return 0;

        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        if (aaveBalance == 0) revert InsufficientAaveBalance();

        // Only withdraw up to the principal amount (don't use accrued interest for user withdrawals)
        uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave;

        withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
        if (withdrawnAmount == 0) return 0;

        aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
        totalETHDepositedToAave -= withdrawnAmount;

        emit ETHWithdrawnFromAave(withdrawnAmount, totalETHDepositedToAave);
    }
