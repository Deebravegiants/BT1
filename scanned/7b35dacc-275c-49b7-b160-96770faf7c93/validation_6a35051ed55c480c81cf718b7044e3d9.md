### Title
Withdrawal Queue Stuffing via Zero-Default `minRsEthAmountToWithdraw` Allows Temporary Freezing of User Funds - (File: contracts/LRTWithdrawalManager.sol)

---

### Summary

`minRsEthAmountToWithdraw[asset]` is never initialized in `LRTWithdrawalManager.initialize()`, leaving it at 0 for every asset. Combined with `minAmountToDeposit` also defaulting to 0 in `LRTDepositPool`, an unprivileged attacker can cheaply create an unbounded number of dust withdrawal requests (1 wei rsETH each). Because the global withdrawal queue is processed in strict FIFO order via a monotonically increasing `nextLockedNonce[asset]`, the attacker's dust requests occupy the front of the queue, forcing the operator to exhaust them before legitimate users' requests can be unlocked. After the 8-day delay the attacker recovers all their assets, making the attack essentially free (gas only).

---

### Finding Description

**Zero minimum withdrawal amount (root cause):**

`LRTWithdrawalManager.initialize()` sets only `withdrawalDelayBlocks` and `lrtConfig`; `minRsEthAmountToWithdraw` is never touched, so it defaults to `0` for every asset. [1](#0-0) 

The guard in `initiateWithdrawal` therefore reduces to `rsETHUnstaked == 0`, allowing any amount ≥ 1 wei:

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
``` [2](#0-1) 

**Zero minimum deposit amount (enables free rsETH acquisition):**

`LRTDepositPool.minAmountToDeposit` is also never set in `initialize()` (defaults to 0), so the deposit guard reduces to `depositAmount == 0`, letting the attacker deposit 1 wei of ETH to receive ~1 wei of rsETH: [3](#0-2) 

**FIFO queue with no skip mechanism (impact amplifier):**

Each call to `initiateWithdrawal` increments the global `nextUnusedNonce[asset]` and stores the request at that slot: [4](#0-3) 

`_unlockWithdrawalRequests` iterates from `nextLockedNonce[asset]` upward; it cannot skip any slot — it must process every request in insertion order before advancing past it: [5](#0-4) 

Legitimate users who call `initiateWithdrawal` after the attacker receive nonces that are numerically higher than all attacker nonces. The operator is forced to drain the attacker's dust requests first.

---

### Impact Explanation

**Temporary freezing of funds (Medium).** When a legitimate user calls `initiateWithdrawal`, their rsETH is immediately transferred to the withdrawal manager: [6](#0-5) 

Those funds remain locked until the operator calls `unlockQueue` and advances `nextLockedNonce` past the user's nonce. If the attacker has pre-filled N dust slots, the operator must process all N slots first. With a sufficiently large N (e.g., tens of thousands of 1-wei requests), the operator may require many separate `unlockQueue` transactions (each bounded by the block gas limit), delaying legitimate users' access to their funds for an extended and attacker-controlled period beyond the intended 8-day delay.

Additionally, each operator `unlockQueue` call that processes attacker dust requests constitutes **unbounded gas consumption** (Medium), as the loop length is proportional to the number of attacker-inserted entries.

---

### Likelihood Explanation

**Medium.** The attack path is fully permissionless: `depositETH` and `initiateWithdrawal` are both public and require no role. The attacker's only cost is gas — all deposited ETH and rsETH is recovered after the 8-day delay. On L1 Ethereum, creating thousands of dust requests is expensive but feasible for a motivated attacker; on any L2 deployment of the pool the cost drops dramatically. No admin compromise, oracle manipulation, or external dependency failure is required.

---

### Recommendation

1. **Set a non-zero default for `minRsEthAmountToWithdraw`** for each supported asset inside `initialize()`, or enforce a single protocol-wide floor (e.g., `0.001 ether` of rsETH).
2. **Set a non-zero default for `minAmountToDeposit`** in `LRTDepositPool.initialize()` to prevent dust rsETH acquisition.
3. Consider adding a **per-user request cap** (e.g., max 10 pending requests per asset per user) to bound the queue pollution any single address can cause.

---

### Proof of Concept

```
// Step 1 – Attacker acquires dust rsETH (minAmountToDeposit == 0)
LRTDepositPool.depositETH{value: 1 wei}(0, "");
// Attacker now holds ~1 wei rsETH

// Step 2 – Attacker stuffs the queue (minRsEthAmountToWithdraw[ETH] == 0)
// Repeat N times (e.g., N = 50,000):
rsETH.approve(address(withdrawalManager), 1);
LRTWithdrawalManager.initiateWithdrawal(ETH_TOKEN, 1, "");
// Each call increments nextUnusedNonce[ETH] and places a dust request at nonces 0..N-1

// Step 3 – Legitimate user initiates withdrawal; receives nonce N
// Their rsETH is locked in the withdrawal manager

// Step 4 – Operator must call unlockQueue repeatedly to drain nonces 0..N-1
// before nonce N (legitimate user) can be unlocked

// Step 5 – After withdrawalDelayBlocks (~8 days), attacker calls completeWithdrawal
// N times and recovers all assets. Net cost: gas only.
```

The legitimate user's rsETH remains frozen for the entire duration it takes the operator to process N dust entries, which is attacker-controlled and unbounded.
