### Title
`bridgeKernelToBSC` Reverts with Arithmetic Underflow When `counter == 0`, Blocking Bridge of Directly-Transferred Tokens - (File: `contracts/KERNEL/KernelVaultETH.sol`)

---

### Summary

`bridgeKernelToBSC` unconditionally computes `counter - 1` to record `lastBridgedDepositId`. When `counter == 0` (no deposit has ever been recorded through the contract's deposit functions), this subtraction underflows and reverts under Solidity 0.8 checked arithmetic, even if the vault holds a non-zero KERNEL balance from a direct ERC20 transfer.

---

### Finding Description

`counter` is a `uint256` that starts at `0` and is only incremented inside `_depositKernel`. [1](#0-0) 

`bridgeKernelToBSC` contains no guard requiring `counter > 0` before executing: [2](#0-1) 

All earlier checks in the function — balance, `minAmount`, and `nativeFee` — can pass cleanly when tokens were sent directly via ERC20 `transfer`: [3](#0-2) 

Execution then reaches line 262 and reverts with a Solidity 0.8 arithmetic underflow panic (`0 - 1`).

---

### Impact Explanation

The operator cannot bridge any KERNEL tokens that arrived in the vault via direct ERC20 transfer while `counter == 0`. Those tokens are inaccessible through the bridge until at least one deposit is recorded via `depositKernel` or `depositKernelFor`, which increments `counter` to 1. The tokens are not permanently destroyed, but the contract fails to deliver its stated behavior (operator can bridge vault holdings) for this reachable state. This matches **Low — contract fails to deliver promised returns, but doesn't lose value**.

---

### Likelihood Explanation

The scenario is straightforward: any party can send KERNEL tokens directly to the vault address before any deposit is made (e.g., an airdrop, a mistaken transfer, or an intentional donation). The operator then calls `bridgeKernelToBSC` and receives an unexpected revert. No privileged compromise is required beyond the operator's own legitimate role.

---

### Recommendation

Guard the assignment or handle the zero case explicitly:

```solidity
lastBridgedDepositId = counter > 0 ? counter - 1 : 0;
```

Or add a precondition revert with a descriptive error if bridging with no recorded deposits is considered invalid:

```solidity
if (counter == 0) revert NoDepositsRecorded();
lastBridgedDepositId = counter - 1;
```

The first option is preferable if directly-donated tokens should be bridgeable; the second if `lastBridgedDepositId` must always correspond to a real deposit record.
