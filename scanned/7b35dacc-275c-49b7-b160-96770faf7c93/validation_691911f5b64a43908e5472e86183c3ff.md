### Title
ETH Transfer Griefing in `_processWithdrawalCompletion` Permanently Freezes User Funds — (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTWithdrawalManager` pushes ETH directly to the withdrawing user via a low-level `.call{value:}`. If the recipient is a smart contract that reverts on ETH receipt, the withdrawal can never be completed. Because rsETH is burned and ETH is redeemed from the vault *before* the user-facing transfer, the funds are permanently frozen with no admin recovery path.

---

### Finding Description

`_processWithdrawalCompletion` is the shared internal function invoked by both `completeWithdrawal` (user-initiated) and `completeWithdrawalForUser` (operator-initiated). At its final step it calls `_transferAsset`, which for ETH performs:

```solidity
(bool sent,) = payable(to).call{ value: amount }("");
if (!sent) revert EthTransferFailed();
``` [1](#0-0) 

If `to` is a smart contract whose `receive()` reverts (or is absent), `sent` is `false`, `EthTransferFailed` is thrown, and the entire transaction reverts. The withdrawal request is restored to its unlocked state, and `unlockedWithdrawalsCount[asset]` remains positive.

The critical asymmetry is that rsETH is burned and ETH is pulled from the unstaking vault during the *earlier* `unlockQueue` call, not during `completeWithdrawal`:

```solidity
if (rsETHBurned != 0) IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned);
unstakingVault.redeem(asset, assetAmountUnlocked);
``` [2](#0-1) 

By the time `completeWithdrawal` is called, the rsETH is already gone and the ETH already sits inside `LRTWithdrawalManager`. The only potential recovery function, `sweepRemainingAssets`, is gated behind `hasUnlockedWithdrawals(asset) == false`:

```solidity
if (hasUnlockedWithdrawals(asset)) revert PendingWithdrawalsExist();
``` [3](#0-2) 

Because the stuck request keeps `unlockedWithdrawalsCount[asset] > 0`, this gate is never cleared. There is no admin function to redirect the ETH to an alternate address or force-delete the request.

The same `_transferAsset` helper is also used in `instantWithdrawal` to push ETH to `feeRecipient` before pushing to `msg.sender`:

```solidity
if (fee > 0) {
    _transferAsset(asset, feeRecipient, fee);
``` [4](#0-3) 

If `feeRecipient` is a non-ETH-receiving contract, every instant ETH withdrawal is DoS'd for all users until the manager changes the recipient — a separate griefing surface.

---

### Impact Explanation

**Permanent freezing of funds (Critical).** The user's rsETH is irreversibly burned during `unlockQueue`. The ETH equivalent is locked inside `LRTWithdrawalManager` indefinitely because:
- `completeWithdrawal` always reverts for that user.
- `completeWithdrawalForUser` (operator path) also reverts.
- `sweepRemainingAssets` is blocked while any unlocked withdrawal exists.
- No admin escape hatch exists to redirect or force-settle the stuck ETH.

---

### Likelihood Explanation

**Low.** The affected user must be a smart contract without a working ETH `receive()` function. Realistic cases include: multisig wallets lacking a `receive` fallback, DeFi vaults or aggregators that route withdrawals through a contract, or contracts upgraded post-initiation to remove ETH acceptance. The scenario is uncommon but entirely plausible in a protocol that accepts smart-contract depositors.

---

### Recommendation

Replace the push-ETH pattern with a pull (claim) pattern for ETH withdrawals:

1. In `_processWithdrawalCompletion`, instead of calling `_transferAsset(asset, user, amount)`, credit `pendingETHClaims[user] += amount` and emit an event.
2. Add a separate `claimETH()` function that lets the user pull their credited balance at will.
3. Apply the same pattern to the `feeRecipient` transfer in `instantWithdrawal` and to `_collectInterestToTreasury`.

This eliminates the ability of any single recipient to block their own (or others') withdrawal execution.

---

### Proof of Concept

1. A smart contract wallet `VaultProxy` (no `receive()`) calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. rsETH is transferred to `LRTWithdrawalManager`.
2. Operator calls `unlockQueue(ETH_TOKEN, ...)`. rsETH is burned; ETH is redeemed from `LRTUnstakingVault` into `LRTWithdrawalManager`. [2](#0-1) 
3. `VaultProxy` calls `completeWithdrawal(ETH_TOKEN, "")`. Internally `_processWithdrawalCompletion` reaches `_transferAsset(ETH_TOKEN, VaultProxy, amount)`.
4. `payable(VaultProxy).call{value: amount}("")` returns `(false, "")` because `VaultProxy` has no `receive()`.
5. `EthTransferFailed` is thrown; the entire transaction reverts. The withdrawal request is restored. [5](#0-4) 
6. Operator attempts `completeWithdrawalForUser(ETH_TOKEN, VaultProxy, "")` — same revert.
7. `unlockedWithdrawalsCount[ETH_TOKEN]` remains ≥ 1; `sweepRemainingAssets` reverts with `PendingWithdrawalsExist`. [3](#0-2) 
8. `VaultProxy`'s rsETH is permanently burned; the ETH equivalent is permanently locked in `LRTWithdrawalManager` with no recovery path.
