### Title
RSETHPool `deposit` Functions Lack Minimum Output Slippage Protection - (`contracts/pools/RSETHPool.sol`, `contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`)

---

### Summary
All L2 RSETHPool `deposit` functions accept ETH or ERC-20 tokens from users and mint wrsETH based on the current oracle rate, but provide **no `minRsETHAmountExpected` parameter**. Users have zero on-chain protection against receiving fewer wrsETH tokens than anticipated if the oracle rate moves between transaction submission and execution.

---

### Finding Description

Every RSETHPool variant exposes two public `deposit` entry points ŌĆö one for native ETH and one for supported ERC-20 tokens. Both compute the wrsETH output exclusively from the live oracle rate at execution time and immediately transfer that amount to the caller, with no floor check:

```solidity
// RSETHPool.sol ŌĆō ETH path (identical pattern in V2, V3, NoWrapper, ExternalBridge variants)
function deposit(string memory referralId) external payable nonReentrant whenNotPaused {
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();

    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount);   // ŌåÉ no minimum check
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
``` [1](#0-0) 

The rate computation divides the post-fee input by the oracle's rsETH/ETH rate:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [2](#0-1) 

The same pattern is replicated across every pool variant: [3](#0-2) [4](#0-3) 

By contrast, the L1 `LRTDepositPool` correctly accepts a `minRSETHAmountExpected` argument and reverts if the minted amount falls below it:

```solidity
function depositETH(uint256 minRSETHAmountExpected, string calldata referralId) external payable ...
``` [5](#0-4) 

```solidity
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [6](#0-5) 

The RSETHPool contracts never perform this check. The slippage protection is entirely absent ŌĆö analogous to M-16 where slippage was applied to the wrong quantity (assets instead of shares), the net effect is identical: the user's actual output can be materially lower than what they observed off-chain before submitting.

---

### Impact Explanation

**Low ŌĆö Contract fails to deliver promised returns, but doesn't lose value.**

A user who previews `viewSwapRsETHAmountAndFee` off-chain and then submits a deposit transaction may receive fewer wrsETH tokens than expected if the oracle rate increases (rsETH appreciates in ETH terms) before the transaction is mined. The deposited ETH/tokens are not returned; the user simply receives a smaller wrsETH balance than anticipated, with no on-chain mechanism to revert the transaction.

---

### Likelihood Explanation

**Low-Medium.** The rsETH/ETH oracle rate is updated by the protocol as staking rewards accrue and is not trivially manipulable. However, during periods of network congestion, pending transactions can sit in the mempool for extended periods, during which the oracle rate may update. Users on L2 chains (Arbitrum, Base, Optimism, etc.) interacting with these pools have no way to bound their minimum output, making this a persistent, if gradual, risk for every depositor.

---

### Recommendation

Add a `minRsETHAmountExpected` parameter to both `deposit` overloads in every RSETHPool variant, mirroring the pattern already used in `LRTDepositPool`:

```solidity
function deposit(
    string memory referralId,
    uint256 minRsETHAmountExpected   // ŌåÉ add this
) external payable nonReentrant whenNotPaused {
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();

    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountExpected) revert MinimumAmountToReceiveNotMet();

    feeEarnedInETH += fee;
    IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
```

Apply the same change to the ERC-20 token `deposit` overload across all pool variants.

---

### Proof of Concept

1. Alice calls `viewSwapRsETHAmountAndFee(1 ether)` off-chain and sees she will receive `X` wrsETH at the current oracle rate.
2. Alice submits `deposit{value: 1 ether}("ref")` to `RSETHPool`.
3. Before Alice's transaction is mined, the rsETH oracle rate increases (e.g., a large staking reward is processed on L1 and the oracle is updated on L2).
4. Alice's transaction executes with the new, higher rate: `rsETHAmount = 1e18 * 1e18 / newHigherRate < X`.
5. Alice receives fewer wrsETH than she observed, with no revert and no recourse. The difference is permanently lost relative to her expectation, and the ETH is already in the pool. [1](#0-0) [2](#0-1)
