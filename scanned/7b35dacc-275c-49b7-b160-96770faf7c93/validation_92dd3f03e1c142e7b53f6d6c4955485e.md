### Title
Single-Step Ownership Transfer Enables Permanent Loss of Admin Control and Token Theft - (File: contracts/KERNEL/KernelTop100MerkleDistributor.sol)

### Summary
Multiple production contracts in the LRT-rsETH repository inherit from `OwnableUpgradeable` or `Ownable`, both of which implement a single-step ownership transfer. There is no two-step `acceptOwnership()` pattern. If the current owner accidentally calls `transferOwnership()` with a mistyped or uncontrolled address, ownership is immediately and irrevocably transferred. The most critical instance is `KernelTop100MerkleDistributor`, which holds KERNEL tokens for user distribution and exposes a privileged `withdrawTokens()` function that allows the owner to drain the entire token balance.

### Finding Description
The following production contracts use single-step ownership with no `acceptOwnership()` guard:

- `contracts/KERNEL/KernelTop100MerkleDistributor.sol` — inherits `OwnableUpgradeable`
- `contracts/KERNEL/KernelMerkleDistributor.sol` — inherits `OwnableUpgradeable`
- `contracts/utils/MerkleDistributor/MerkleDistributor.sol` — inherits `OwnableUpgradeable`
- `contracts/utils/MerkleDistributor/MerkleBlastPointsDistributor.sol` — inherits `OwnableUpgradeable`
- `contracts/cross-chain/CrossChainRateProvider.sol` — inherits `Ownable`
- `contracts/cross-chain/CrossChainRateReceiver.sol` — inherits `Ownable`
- `contracts/cross-chain/MultiChainRateProvider.sol` — inherits `Ownable`

The only contract that correctly implements a two-step pattern is `contracts/ccip/WrappedRSETH.sol`, which inherits `ConfirmedOwnerWithProposal` and requires the nominee to call `acceptOwnership()`.

The root cause is in `OwnableUpgradeable.transferOwnership()`:

```solidity
function transferOwnership(address newOwner) public virtual onlyOwner {
    require(newOwner != address(0), "Ownable: new owner is the zero address");
    _transferOwnership(newOwner); // immediately writes newOwner into _owner
}
``` [1](#0-0) 

The only guard is a zero-address check. Any non-zero address — including a mistyped EOA, a contract that cannot call back, or an attacker-controlled address — is accepted immediately and permanently.

In `KernelTop100MerkleDistributor`, the `onlyOwner` surface includes:

```solidity
function withdrawTokens(address _token, uint256 _amount, address _recipient) external onlyOwner {
    ...
    IERC20(_token).safeTransfer(_recipient, _amount);
    ...
}
``` [2](#0-1) 

```solidity
function pause() external onlyOwner { _pause(); }
function unpause() external onlyOwner { _unpause(); }
``` [3](#0-2) 

### Impact Explanation
Two distinct impact paths exist once ownership is transferred to an uncontrolled address:

**Path A — Attacker-controlled address:** The new owner immediately calls `withdrawTokens(kernelToken, totalBalance, attackerAddress)`, draining all KERNEL tokens held in the contract that were allocated for user vesting claims. This is a direct, complete theft of user funds at rest. Impact: **Critical**.

**Path B — Dead/uncontrolled address (typo):** No one can call `withdrawTokens` to recover the tokens, `pause`/`unpause` to manage the contract, `setMerkleRoot` (in `KernelMerkleDistributor`/`MerkleDistributor`) to update distribution, or `setVestingStartTimestamp`. All KERNEL tokens locked in the contract become permanently irrecoverable. Impact: **Critical — permanent freezing of funds**.

### Likelihood Explanation
Ownership transfers are infrequent, high-stakes operations typically performed manually. A single character typo in a 42-character hex address produces a valid non-zero address that passes the only guard. The absence of a confirmation step means there is no recovery path. This is a well-documented operational risk for protocols that manage token treasuries.

### Recommendation
Replace `OwnableUpgradeable` with `Ownable2StepUpgradeable` (already present in the repository at `lib/openzeppelin-contracts-upgradeable/contracts/access/Ownable2StepUpgradeable.sol`) for all affected contracts. For non-upgradeable contracts (`CrossChainRateProvider`, `CrossChainRateReceiver`, `MultiChainRateProvider`), replace `Ownable` with `Ownable2Step` (already present at `lib/openzeppelin-contracts/contracts/access/Ownable2Step.sol`). The pattern used in `ConfirmedOwnerWithProposal` for `WrappedRSETH` is the correct reference implementation already in scope.

### Proof of Concept
1. Deploy `KernelTop100MerkleDistributor` with KERNEL tokens loaded for user distribution.
2. Owner calls `transferOwnership(0xDEAD...BEEF)` — a mistyped address that is non-zero.
3. `OwnableUpgradeable.transferOwnership` passes the zero-address check and immediately sets `_owner = 0xDEAD...BEEF`.
4. **Path A:** If `0xDEAD...BEEF` is attacker-controlled, attacker calls `withdrawTokens(kernelAddress, totalBalance, attacker)` — all KERNEL tokens are transferred out.
5. **Path B:** If `0xDEAD...BEEF` is uncontrolled, `pause`, `unpause`, `withdrawTokens`, and `setVestingStartTimestamp` are all permanently bricked. All KERNEL tokens in the contract are frozen forever, and users can never claim their vested allocations. [4](#0-3) [5](#0-4) [1](#0-0) [6](#0-5)
