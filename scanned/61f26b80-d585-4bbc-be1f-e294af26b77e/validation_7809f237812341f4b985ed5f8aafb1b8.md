### Title
Global `withdrawalDelayBlocks` Update Retroactively Affects All Pending Withdrawal Requests - (File: contracts/LRTWithdrawalManager.sol)

### Summary
`setWithdrawalDelayBlocks` updates the single global `withdrawalDelayBlocks` variable, which is then applied to **all existing pending withdrawal requests** at unlock and completion time. A manager increasing the delay retroactively extends the lock period for every user who already submitted a withdrawal request under the prior, shorter delay — temporarily freezing their funds beyond the terms they accepted.

### Finding Description
`withdrawalDelayBlocks` is a single global state variable initialized to `8 days / 12 seconds` (~57,600 blocks). [1](#0-0) 

The manager can update it at any time via `setWithdrawalDelayBlocks`, up to a ceiling of 16 days: [2](#0-1) 

This global value is consumed in two critical paths. First, in `_unlockWithdrawalRequests`, which gates whether a queued request is eligible to be unlocked: [3](#0-2) 

Second, in `_processWithdrawalCompletion`, which gates whether a user can actually claim their assets: [4](#0-3) 

Neither path reads a per-request cached delay. Both read the live global `withdrawalDelayBlocks` at execution time. The `WithdrawalRequest` struct stores only `withdrawalStartBlock`, not the delay that was in effect when the request was created: [5](#0-4) 

### Impact Explanation
When the manager increases `withdrawalDelayBlocks` — for example, from 8 days to 16 days — every pending withdrawal request that was submitted under the old 8-day delay is retroactively subject to the new 16-day delay. Users who already burned or locked their rsETH and were waiting for the 8-day window to expire will find their funds frozen for up to an additional 8 days beyond what the protocol promised at request time. This is **temporary freezing of funds (Medium)**.

### Likelihood Explanation
The manager role is expected to call `setWithdrawalDelayBlocks` for legitimate operational reasons (e.g., aligning with EigenLayer's own withdrawal delay changes, or responding to a security event). No malicious intent is required. Any such call, even well-intentioned, silently retroacts against all outstanding requests. The protocol has no mechanism to distinguish "requests submitted before the change" from "requests submitted after."

### Recommendation
Cache `withdrawalDelayBlocks` inside the `WithdrawalRequest` struct at the time `initiateWithdrawal` is called, analogous to how `withdrawalStartBlock` is already stored. Replace all reads of the global `withdrawalDelayBlocks` in `_unlockWithdrawalRequests` and `_processWithdrawalCompletion` with the per-request cached value. This ensures each request is evaluated against the delay that was in effect when the user committed their rsETH.

### Proof of Concept
1. `withdrawalDelayBlocks` = 57,600 (8 days). User submits `initiateWithdrawal` at block **N**; `withdrawalStartBlock` = N. User expects to claim at block N + 57,600.
2. At block N + 50,000, manager calls `setWithdrawalDelayBlocks(115200)` (16 days) for a legitimate operational reason.
3. At block N + 57,600, user calls `completeWithdrawal`. The check `block.number < request.withdrawalStartBlock + withdrawalDelayBlocks` evaluates as `N+57,600 < N+115,200` → **reverts with `WithdrawalDelayNotPassed`**.
4. The same check in `_unlockWithdrawalRequests` prevents the operator from unlocking the request.
5. User's funds remain frozen for an additional ~57,600 blocks (~8 days) beyond the delay they accepted when initiating the withdrawal. [2](#0-1) [3](#0-2) [4](#0-3)

    function setWithdrawalDelayBlocks(uint256 withdrawalDelayBlocks_) external onlyLRTManager {
        // Set an upper limit of no more than 16 days
        if (withdrawalDelayBlocks_ > 16 days / 12 seconds) revert ExceedWithdrawalDelay();

        withdrawalDelayBlocks = withdrawalDelayBlocks_;
        emit WithdrawalDelayBlocksUpdated(withdrawalDelayBlocks);
    }
