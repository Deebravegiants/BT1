### Title
Reward Tokens Permanently Locked When `totalKernelStaked` Drops to Zero Mid-Period — (`contracts/KERNEL/KernelDepositPool.sol`)

---

### Summary

When the last staker calls `initiateWithdrawal`, `totalKernelStaked` reaches zero mid-reward-period. Because `rewardPerToken()` returns the frozen `rewardPerTokenStored` whenever `totalKernelStaked == 0`, all reward tokens that would have accrued during the zero-staked window are permanently unclaimable. The contract itself acknowledges this in its NatSpec but relies on an off-chain operational guarantee that is not enforced by any code guard.

---

### Finding Description

**Root cause — `rewardPerToken()` freezes at zero supply:** [1](#0-0) 

When `totalKernelStaked == 0`, the function short-circuits and returns `rewardPerTokenStored` unchanged. Time continues to pass, `rewardRate` continues to be non-zero, but no accumulation occurs.

**`updateReward` snapshots state before the balance change:** [2](#0-1) 

The modifier runs before `initiateWithdrawal` decrements `totalKernelStaked`. So at the moment of the call, `rewardPerTokenStored` is correctly updated for the attacker's earned rewards up to that point, and `updatedAt` is set to `lastTimeRewardApplicable()`. After the modifier, `totalKernelStaked` drops to zero.

**`initiateWithdrawal` drains `totalKernelStaked`:** [3](#0-2) 

No guard prevents `totalKernelStaked` from reaching zero during an active reward period.

**Re-staking resets `updatedAt` without recovering the lost window:**

When the attacker later calls `stake(1)`, `updateReward` runs again. At that moment `totalKernelStaked` is still 0, so `rewardPerToken()` still returns the frozen `rewardPerTokenStored`. Then `updatedAt` is set to the current timestamp. After the stake, `totalKernelStaked = 1`. All future `rewardPerToken()` calls will compute accrual starting from the new `updatedAt`, permanently skipping the zero-staked window.

**The contract explicitly acknowledges this design gap:** [4](#0-3) 

The mitigation described ("ensuring there are always some tokens staked") is a purely operational promise with zero on-chain enforcement.

---

### Impact Explanation

The correct scoped impact is **Medium — Permanent freezing of unclaimed yield**, not "High — Theft of unclaimed yield."

The rewards accrued during the zero-staked window (`(t_restake - t_withdraw) * rewardRate` tokens) are permanently locked in the contract. No address can ever claim them. The attacker does **not** receive those locked tokens; they are destroyed from the perspective of distribution. The attacker's post-re-stake capture of future rewards is simply normal sole-staker behavior (100% share with 1 wei), not a gain of the frozen tokens.

---

### Likelihood Explanation

Likelihood is **Low-to-Medium**. It requires the attacker to be the sole remaining staker, which is a non-trivial precondition in a live deployment. However, it requires no privileged role, no oracle manipulation, and no external dependency — only a normal `initiateWithdrawal` call. The operational mitigation (admin ensures staking before `notifyRewardAmount`) does not prevent a staker from withdrawing after the period starts.

---

### Recommendation

Enforce the invariant in code rather than relying on operational discipline. Options:

1. **Pause reward accrual cleanly**: In `rewardPerToken()`, when `totalKernelStaked == 0`, also freeze `updatedAt` so that when staking resumes, the skipped window is not silently discarded. This is already done for `rewardPerTokenStored` but `updatedAt` is reset on re-stake, losing the gap.
2. **Recover undistributed rewards**: Track the duration of zero-staked windows and roll those tokens into the next period via a `notifyRewardAmount`-style recalculation.
3. **Prevent full withdrawal during active period**: Add a guard in `initiateWithdrawal` that reverts if the withdrawal would set `totalKernelStaked` to zero while `block.timestamp < finishAt`.

---

### Proof of Concept

```solidity
// Fork test outline (no mainnet — local Foundry test)
function test_rewardsFrozenAtZeroStake() public {
    // Setup: admin sets duration, attacker stakes 1000e18 KERNEL
    // Admin calls notifyRewardAmount(1000e18 rewardToken)
    // Warp to midpoint of period
    vm.warp(block.timestamp + duration / 2);

    uint256 rptBefore = pool.rewardPerToken();

    // Attacker initiates full withdrawal → totalKernelStaked = 0
    vm.prank(attacker);
    pool.initiateWithdrawal(1000e18);

    // Warp forward — rewards should accrue but don't
    vm.warp(block.timestamp + duration / 4);
    assertEq(pool.rewardPerToken(), rptBefore); // frozen ✓

    // Attacker re-stakes 1 wei
    vm.prank(attacker);
    pool.stake(1);

    // Warp to end of period
    vm.warp(pool.finishAt());

    // earned(attacker) covers only post-re-stake window
    // Rewards for the zero-staked window (duration/4 * rewardRate) are unclaimable
    uint256 earned = pool.earned(attacker);
    uint256 expectedFull = pool.rewardRate() * pool.duration();
    assertLt(earned, expectedFull); // gap exists ✓

    // Confirm tokens are stuck: rewardsToken balance > claimable
    assertGt(rewardsToken.balanceOf(address(pool)), earned);
}
```

    function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
        if (_amount == 0) revert AmountZero();
        if (balanceOf[msg.sender] < _amount) revert InsufficientStakedBalance();
        if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();

        balanceOf[msg.sender] -= _amount;
        totalKernelStaked -= _amount;

        // Create a withdrawal record
        uint256 withdrawalId = ++withdrawalCounter;
        uint256 unlockTime = block.timestamp + withdrawalDelay;

        withdrawals[withdrawalId] = Withdrawal({
            user: msg.sender, amount: _amount, unlockTime: unlockTime, claimed: false, withdrawalId: withdrawalId
        });
        userWithdrawalIds[msg.sender].push(withdrawalId);

        emit WithdrawalInitiated(msg.sender, _amount, withdrawalId, unlockTime);
    }
