### Title
Deposit Limit Reached Causes Temporary Freezing of Bridged LST in L1Vault - (File: contracts/L1Vault.sol)

---

### Summary

`L1Vault.depositAssetForL1Vault` unconditionally forwards the vault's entire LST balance into `LRTDepositPool.depositAsset`. If `depositLimitByAsset[token]` is at or near capacity at the time of the call, `_beforeDeposit` reverts with `MaximumDepositLimitReached`, leaving the already-bridged LST stranded in `L1Vault` with no alternative exit path.

---

### Finding Description

The flow is:

1. The L2 bridge delivers LST tokens directly to `L1Vault` (the contract has a `receive()` fallback and accepts ERC-20 transfers).
2. The MANAGER calls `L1Vault.depositAssetForL1Vault(token)`.

Inside that function:

```solidity
// L1Vault.sol:169-179
uint256 tokenBalance = IERC20(token).balanceOf(address(this));
uint256 rsETHAmountToMint = lrtDepositPool.getRsETHAmountToMint(token, tokenBalance);
if (rsETHAmountToMint == 0) { revert InvalidMinRSETHAmountExpected(); }
IERC20(token).safeIncreaseAllowance(address(lrtDepositPool), tokenBalance);
lrtDepositPool.depositAsset(token, tokenBalance, rsETHAmountToMint, "");
``` [1](#0-0) 

`depositAsset` calls `_beforeDeposit`, which calls `_checkIfDepositAmountExceedesCurrentLimit`:

```solidity
// LRTDepositPool.sol:676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [2](#0-1) 

If `getTotalAssetDeposits(token) + tokenBalance > depositLimitByAsset[token]`, `_beforeDeposit` reverts:

```solidity
// LRTDepositPool.sol:661-663
if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
    revert MaximumDepositLimitReached();
}
``` [3](#0-2) 

Critically, `getTotalAssetDeposits` sums balances across `LRTDepositPool`, NDCs, EigenLayer strategies, the converter, and the unstaking vault — **it does not include tokens sitting in `L1Vault`**:

```solidity
// LRTDepositPool.sol:444
assetLyingInDepositPool = IERC20(asset).balanceOf(address(this)); // address(this) = LRTDepositPool
``` [4](#0-3) 

So the bridged LST is invisible to the limit check until it is successfully deposited. If the limit is already saturated by other depositors, the call reverts and the LST is stuck.

`L1Vault` has no rescue path: no function to return tokens to L2, no partial-deposit logic, no bypass for bridge-originated funds. The only resolution is an admin raising `depositLimitByAsset[token]` via `LRTConfig.updateAssetDepositLimit`:

```solidity
// LRTConfig.sol:123-133
function updateAssetDepositLimit(address asset, uint256 depositLimit)
    external
    onlyRole(LRTConstants.MANAGER)
    onlySupportedAsset(asset)
{
    depositLimitByAsset[asset] = depositLimit;
    ...
}
``` [5](#0-4) 

---

### Impact Explanation

Bridged LST tokens are temporarily frozen in `L1Vault`. The L2 user has already burned their L2 representation of the asset; the corresponding rsETH cannot be minted and bridged back until an admin intervenes. This matches **Medium — Temporary freezing of funds**.

---

### Likelihood Explanation

The deposit limit is a live, adjustable parameter. As the protocol approaches capacity for a given LST (a normal operational state), any bridge delivery that pushes `totalAssetDeposits + bridgedAmount` over the limit triggers the freeze. No attacker action is required; organic deposit activity is sufficient. The MANAGER must notice the stuck balance and raise the limit, introducing an unbounded delay.

---

### Recommendation

Before accepting a bridge delivery (or at the start of `depositAssetForL1Vault`), check the remaining capacity and either:

- **Revert early with a clear error** if capacity is insufficient, so the bridge can handle the failure on the L2 side before funds are irrecoverably sent.
- **Add a rescue function** (e.g., `rescueToken`) callable by `TIMELOCK_ROLE` that can transfer stuck ERC-20s out of `L1Vault` to a safe address, providing an emergency exit without requiring a deposit-limit change.
- **Pre-check capacity off-chain** in the bridge relay and delay the bridge transaction until capacity exists.

---

### Proof of Concept

```solidity
// Pseudocode unit test (local fork, no mainnet)
function test_depositAssetForL1Vault_revertsWhenLimitReached() public {
    address token = address(stETH);
    uint256 currentDeposits = lrtDepositPool.getTotalAssetDeposits(token);
    uint256 limit = lrtConfig.depositLimitByAsset(token);

    // Set limit to exactly current deposits (capacity = 0)
    vm.prank(manager);
    lrtConfig.updateAssetDepositLimit(token, currentDeposits);

    // Simulate bridge: send 1 ether of LST to L1Vault
    deal(token, address(l1Vault), 1 ether);

    // MANAGER attempts to deposit — must revert
    vm.prank(manager);
    vm.expectRevert(ILRTDepositPool.MaximumDepositLimitReached.selector);
    l1Vault.depositAssetForL1Vault(token);

    // LST is now frozen in L1Vault with no exit path
    assertEq(IERC20(token).balanceOf(address(l1Vault)), 1 ether);
}
```

The revert path is: `depositAssetForL1Vault` → `depositAsset` → `_beforeDeposit` → `_checkIfDepositAmountExceedesCurrentLimit` returns `true` → `revert MaximumDepositLimitReached`. [6](#0-5) [7](#0-6)
