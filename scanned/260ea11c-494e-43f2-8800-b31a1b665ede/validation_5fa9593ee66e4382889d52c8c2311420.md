### Title
Pausing `LRTWithdrawalManager` Blocks Completion of Already-Unlocked Withdrawals With No Emergency Recovery Path - (File: contracts/LRTWithdrawalManager.sol)

---

### Summary

The `LRTWithdrawalManager` contract's `pause()` function can be called by the `PAUSER_ROLE` at any time. When called after `unlockQueue()` has already been executed — a state where user rsETH has been burned and assets have been transferred into the contract — the `whenNotPaused` guard on `completeWithdrawal()` blocks all remaining withdrawal paths. There is no emergency withdrawal function for users to bypass the pause, leaving their assets temporarily frozen in the contract with no self-service recovery path.

---

### Finding Description

The withdrawal lifecycle in `LRTWithdrawalManager` is a three-step process:

1. **`initiateWithdrawal()`** — User transfers rsETH to the contract; a withdrawal request is recorded.
2. **`unlockQueue()`** — An operator burns the rsETH held by the contract and transfers the corresponding assets from `LRTUnstakingVault` into `LRTWithdrawalManager`. Requests are marked unlocked.
3. **`completeWithdrawal()`** — User calls this to receive their assets.

The critical issue is that `completeWithdrawal()` carries the `whenNotPaused` modifier:

```solidity
function completeWithdrawal(address asset, string calldata referralId) external nonReentrant whenNotPaused {
    _processWithdrawalCompletion(asset, msg.sender, referralId);
}
``` [1](#0-0) 

After step 2 completes, the user's rsETH is permanently burned and the corresponding assets sit inside `LRTWithdrawalManager`. If `pause()` is called at this point:

- `completeWithdrawal()` reverts due to `whenNotPaused`.
- `completeWithdrawalForUser()` also reverts due to `whenNotPaused`.
- `instantWithdrawal()` also reverts due to `whenNotPaused`.
- There is **no** `emergencyWithdraw()` or any other function that allows users to retrieve their assets while the contract is paused. [2](#0-1) 

The `pause()` function is callable by any address holding `PAUSER_ROLE`, with no state-based guard:

```solidity
function pause() external onlyRole(LRTConstants.PAUSER_ROLE) {
    _pause();
}
``` [3](#0-2) 

The only recovery path is for the `LRT_ADMIN` to call `unpause()`, which is a separate role from `PAUSER_ROLE`:

```solidity
function unpause() external onlyLRTAdmin {
    _unpause();
}
``` [4](#0-3) 

A secondary variant of the same class exists: if `RSETH` is paused (independently, by its own `PAUSER_ROLE`), then `unlockQueue()` fails because it calls `IRSETH.burnFrom()`, which carries `whenNotPaused` on the RSETH contract. This blocks the queue from advancing, leaving rsETH permanently held in `LRTWithdrawalManager` with no cancel path for users. [5](#0-4) [6](#0-5) 

---

### Impact Explanation

After `unlockQueue()` executes, user rsETH is burned and the corresponding LST/ETH assets are held inside `LRTWithdrawalManager`. If the contract is paused before users call `completeWithdrawal()`, those assets are frozen with no self-service recovery path. Users cannot cancel, cannot complete, and cannot use instant withdrawal. The freeze persists until the `LRT_ADMIN` unpauses — a role distinct from `PAUSER_ROLE`, meaning the entity that caused the freeze cannot undo it unilaterally.

**Impact: Medium — Temporary freezing of funds.**

---

### Likelihood Explanation

The `PAUSER_ROLE` is expected to pause the contract in response to security incidents. This is a routine operational action. Because `unlockQueue()` is called regularly by operators to service the withdrawal queue, there will frequently be a window where assets are sitting in `LRTWithdrawalManager` awaiting `completeWithdrawal()` calls. Any pause during this window freezes those assets. The likelihood is **medium**: the scenario requires no attacker, only a legitimate pause during a routine operational state.

---

### Recommendation

1. **Add an emergency withdrawal path** that bypasses the pause for users whose requests are already unlocked (i.e., `usersFirstWithdrawalRequestNonce < nextLockedNonce[asset]`). This mirrors the fix applied in the referenced Trader Joe report, which added LP token recovery to `emergencyWithdraw()`.

2. **Alternatively**, remove `whenNotPaused` from `completeWithdrawal()` for requests that are already in the unlocked state, since the rsETH has already been burned and the assets are already committed to the user.

3. For the RSETH-pause variant: ensure that `unlockQueue()` does not depend on `RSETH.burnFrom()` being unpaused, or document clearly that pausing RSETH will stall the withdrawal queue and strand user rsETH in the manager contract.

---

### Proof of Concept

```
1. Alice calls initiateWithdrawal(ETH, 1e18, "ref")
   → 1e18 rsETH transferred from Alice to LRTWithdrawalManager
   → Withdrawal request recorded at nonce N

2. Operator calls unlockQueue(ETH, N+1, ...)
   → 1e18 rsETH burned from LRTWithdrawalManager (RSETH.burnFrom)
   → ETH transferred from LRTUnstakingVault into LRTWithdrawalManager
   → Request N marked unlocked (nextLockedNonce[ETH] = N+1)

3. PAUSER_ROLE calls pause() on LRTWithdrawalManager
   → paused = true

4. Alice calls completeWithdrawal(ETH, "ref")
   → REVERTS: EnforcedPause() (whenNotPaused guard)

5. Alice calls instantWithdrawal(ETH, ...)
   → REVERTS: EnforcedPause()

6. No other function allows Alice to recover her ETH.
   Her rsETH is already burned. Her ETH sits in LRTWithdrawalManager.
   She must wait for LRT_ADMIN (a separate role) to call unpause().
```
