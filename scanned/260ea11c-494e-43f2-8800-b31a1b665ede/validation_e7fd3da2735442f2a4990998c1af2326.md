### Title
Dust aWETH Stranded After `setAaveIntegrationEnabled(false)` Due to aToken Rebasing Between Balance Snapshot and Withdrawal — (`contracts/LRTWithdrawalManager.sol`)

---

### Summary

When `setAaveIntegrationEnabled(false)` is called, the two-step drain (collect interest → withdraw principal) leaves a dust aWETH balance because `_withdrawFromAave` hard-caps its withdrawal at `totalETHDepositedToAave`. Any aWETH that accrues between the `balanceOf` snapshot in `_collectInterestToTreasury` and the actual `withdrawETH` call is not covered by either step, and with `isAaveIntegrationEnabled = false` all recovery paths are gated behind the flag.

---

### Finding Description

**Step 1 — `_collectInterestToTreasury()`** [1](#0-0) 

The function snapshots `aaveBalance = P + I` and computes `interestAmount = I`. It then calls `withdrawETH(aavePool, I, address(this))`. Because aTokens are rebasing (their balance grows every block via the Aave liquidity index), by the time the `withdrawETH` call executes, the actual aWETH balance has grown to `P + I + δ`. After burning `I` aWETH, the remaining balance is `P + δ`. `totalETHDepositedToAave` is never updated in this function — it stays at `P`.

**Step 2 — `_withdrawFromAave(P + δ)`** [2](#0-1) 

`withdrawablePrincipal = min(P + δ, totalETHDepositedToAave) = min(P + δ, P) = P`. The function withdraws exactly `P`, leaving `δ` aWETH in the contract. `totalETHDepositedToAave` becomes `0`.

**Step 3 — Integration disabled with dust remaining** [3](#0-2) 

`isAaveIntegrationEnabled` is set to `false`. All recovery paths (`emergencyWithdrawFromAave`, `collectInterestToTreasury`, `depositIdleETHToAave`) gate on `isAaveIntegrationEnabled`: [4](#0-3) 

The dust `δ` aWETH is inaccessible until an admin explicitly re-enables the integration.

### Impact Explanation

The contract fails to deliver the full promised returns: dust interest (the `δ` accrued during the transaction) is not swept to the treasury and is not returned to the contract's ETH balance. No principal is lost. This matches **Low — Contract fails to deliver promised returns, but doesn't lose value**.

The dust is technically recoverable by calling `setAaveIntegrationEnabled(true)` followed by `collectInterestToTreasury()` (since `totalETHDepositedToAave = 0 < δ = aaveBalance`, the dust is treated as pure interest). However, the invariant that a single `setAaveIntegrationEnabled(false)` call fully drains aWETH is broken, and the recovery requires an additional privileged transaction that the protocol may not perform.

---

### Likelihood Explanation

This occurs on **every** `setAaveIntegrationEnabled(false)` call when interest has accrued, because aToken balances increase continuously per block. The `δ` is typically 1–2 wei per block at normal Aave utilization rates, but the invariant break is deterministic and reproducible.

---

### Recommendation

After `_collectInterestToTreasury()` returns, re-read the aWETH balance and pass it to `_withdrawFromAave` without the `totalETHDepositedToAave` cap when the intent is a full drain. Alternatively, add a dedicated full-drain path in `_withdrawFromAave` (e.g., when `amount == type(uint256).max`) that withdraws `aaveBalance` directly and sets `totalETHDepositedToAave = 0`, bypassing the principal cap:

```solidity
if (!enabled) {
    uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
    if (aaveBalance > 0) {
        _collectInterestToTreasury();
        // Re-read after interest collection; withdraw ALL remaining aWETH
        aaveBalance = aaveAWETH.balanceOf(address(this));
        if (aaveBalance > 0) {
            // Full drain: bypass principal cap
            aaveWETHGateway.withdrawETH(aavePool, aaveBalance, address(this));
            totalETHDepositedToAave = 0;
        }
    }
    _revokeApprovalToAaveWETHGateway();
}
```
