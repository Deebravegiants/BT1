### Title
rsETH Share Price Inflation via ETH Donation Enables Zero-Mint Deposit Theft - (File: contracts/LRTDepositPool.sol)

### Summary
An attacker can inflate the stored `rsETHPrice` in `LRTOracle` by donating ETH directly to `LRTDepositPool` (via its open `receive()`) and calling the public `updateRSETHPrice()`, causing a subsequent depositor who passes `minRSETHAmountExpected = 0` to receive 0 rsETH while losing all deposited funds.

### Finding Description

`LRTDepositPool.getRsETHAmountToMint()` computes shares to mint as:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [1](#0-0) 

`rsETHPrice` is a **stored value** in `LRTOracle`, updated only when `updateRSETHPrice()` is called (a public, permissionless function): [2](#0-1) 

The price is recomputed as:

```solidity
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
``` [3](#0-2) 

`totalETHInProtocol` is derived from `_getTotalEthInProtocol()`, which includes `address(this).balance` of the deposit pool: [4](#0-3) 

`LRTDepositPool` accepts ETH from **anyone** via: [5](#0-4) 

This creates the donation vector. The price-increase guard in `_updateRsETHPrice()` only activates when `pricePercentageLimit > 0`, which is **not set in `initialize()`** and defaults to 0: [6](#0-5) 

After the price is inflated, `_beforeDeposit` computes `rsethAmountToMint` and checks:

```solidity
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [7](#0-6) 

If the victim passes `minRSETHAmountExpected = 0`, the check passes even when `rsethAmountToMint == 0`. `_mintRsETH` has no zero-amount guard: [8](#0-7) 

**Attack path (with `pricePercentageLimit == 0`):**

1. Attacker deposits 1 wei ETH → receives 1 wei rsETH (price = 1e18)
2. Attacker sends 2e18 ETH directly to `LRTDepositPool` via `receive()`
3. Attacker calls `updateRSETHPrice()` → `totalETHInProtocol ≈ 2e18`, `rsethSupply = 1 wei` → `newRsETHPrice = 2e18 * 1e18 / 1 = 2e36`
4. Victim calls `depositETH(0, "")` with 1e18 ETH → `rsethAmountToMint = (1e18 * 1e18) / 2e36 = 0` → victim receives 0 rsETH, loses 1e18 ETH
5. Attacker's 1 wei rsETH now backs ~3e18 ETH total TVL; attacker redeems and recovers victim's deposit

### Impact Explanation
**Critical — direct theft of user funds.** The victim's entire ETH deposit is transferred to `LRTDepositPool` and 0 rsETH is minted. The attacker's single wei of rsETH represents a claim on the entire pool TVL including the victim's deposit. The attacker recovers their donated ETH plus the victim's deposit upon withdrawal.

### Likelihood Explanation
**Medium.** Two conditions must hold simultaneously:
1. `pricePercentageLimit` is 0 — this is the **default state** since `initialize()` never sets it, making every fresh deployment vulnerable until an admin explicitly configures it.
2. The victim passes `minRSETHAmountExpected = 0` — common when users interact directly with the contract, use scripts without slippage logic, or when frontends omit the parameter.

The attack is a straightforward front-run requiring no special privileges and only ETH capital equal to the victim's deposit amount.

### Recommendation
1. **Enforce a non-zero minimum rsETH output**: revert in `_beforeDeposit` if `rsethAmountToMint == 0`.
2. **Set `pricePercentageLimit` in `initialize()`** to a safe non-zero value so the price cannot be inflated arbitrarily in a single call by an unprivileged caller.
3. **Add virtual balances** to the TVL calculation (analogous to OZ ERC4626's `_decimalsOffset`) so a small rsETH supply cannot be exploited to produce extreme per-share prices.

### Proof of Concept

```
State: pricePercentageLimit == 0 (default), minAmountToDeposit == 0 (default)

1. attacker.depositETH{value: 1}(0, "")
   → rsETH supply = 1 wei, TVL = 1 wei, rsETHPrice = 1e18

2. attacker sends 2e18 ETH to LRTDepositPool (receive())
   → TVL = 2e18 + 1 wei

3. attacker calls lrtOracle.updateRSETHPrice()
   → newRsETHPrice = (2e18+1) * 1e18 / 1 ≈ 2e36
   → rsETHPrice stored = 2e36

4. victim.depositETH{value: 1e18}(0, "")
   → rsethAmountToMint = (1e18 * 1e18) / 2e36 = 0
   → _mintRsETH(0) called — victim receives 0 rsETH
   → victim loses 1e18 ETH

5. attacker's 1 wei rsETH backs TVL ≈ 3e18 ETH
   → attacker redeems, recovering victim's 1e18 ETH
```
