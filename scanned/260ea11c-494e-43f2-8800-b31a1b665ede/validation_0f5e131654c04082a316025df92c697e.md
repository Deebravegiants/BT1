### Title
Unbounded `supportedAssetList` Iteration in `LRTOracle._getTotalEthInProtocol()` Can Permanently Block rsETH Price Updates - (File: contracts/LRTOracle.sol)

### Summary
`LRTConfig.supportedAssetList` has no hard cap on the number of entries. `LRTOracle._getTotalEthInProtocol()` iterates over every supported asset and, for each one, calls `LRTDepositPool.getTotalAssetDeposits()` which itself iterates over `nodeDelegatorQueue`. As the protocol adds more supported assets through normal operation, the gas cost of this nested iteration grows without bound. Because `updateRSETHPrice()` is a public function that calls `_getTotalEthInProtocol()`, once the list is large enough the function will revert on every call, permanently freezing the rsETH price and blocking fee minting.

### Finding Description
`LRTConfig._addNewSupportedAsset()` pushes to `supportedAssetList` with no upper-bound check:

```solidity
// contracts/LRTConfig.sol
function _addNewSupportedAsset(address asset, uint256 depositLimit) private {
    ...
    isSupportedAsset[asset] = true;
    supportedAssetList.push(asset);   // no cap enforced
    ...
}
```

`LRTOracle._getTotalEthInProtocol()` iterates over every element of this list and, for each asset, makes multiple external calls through `getTotalAssetDeposits()` → `getAssetDistributionData()` → a loop over `nodeDelegatorQueue`:

```solidity
// contracts/LRTOracle.sol  L331-L349
function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
    address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
    address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
    uint256 supportedAssetCount = supportedAssets.length;

    for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
        address asset = supportedAssets[assetIdx];
        uint256 assetER = getAssetPrice(asset);                                    // external call
        uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr)
            .getTotalAssetDeposits(asset);                                         // external call → inner loop
        totalETHInProtocol += totalAssetAmt.mulWad(assetER);
        unchecked { ++assetIdx; }
    }
}
```

`getTotalAssetDeposits()` calls `getAssetDistributionData()` which itself loops over `nodeDelegatorQueue` making three external calls per NDC per asset:

```solidity
// contracts/LRTDepositPool.sol  L446-L456
uint256 ndcsCount = nodeDelegatorQueue.length;
for (uint256 i; i < ndcsCount;) {
    assetLyingInNDCs += IERC20(asset).balanceOf(nodeDelegatorQueue[i]);
    assetStakedInEigenLayer += INodeDelegator(nodeDelegatorQueue[i]).getAssetBalance(asset);
    assetUnstakingFromEigenLayer += INodeDelegator(nodeDelegatorQueue[i]).getAssetUnstaking(asset);
    unchecked { ++i; }
}
```

The public entry point is:

```solidity
// contracts/LRTOracle.sol  L87-L89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

`updateRSETHPriceAsManager()` (the manager-only variant) calls the same `_updateRsETHPrice()` and is equally affected.

### Impact Explanation
If `supportedAssetList` grows large enough that the nested iteration exceeds the block gas limit, every call to `updateRSETHPrice()` and `updateRSETHPriceAsManager()` will revert. The stored `rsETHPrice` becomes permanently stale. Downstream effects:

- `getRsETHAmountToMint()` uses the stale price, causing incorrect rsETH minting for all depositors.
- `getExpectedAssetAmount()` in `LRTWithdrawalManager` uses the stale price, causing incorrect withdrawal amounts.
- Protocol fee minting via `_checkAndUpdateDailyFeeMintLimit()` is permanently blocked.
- The price-deviation circuit-breaker (auto-pause on large price drops) is permanently disabled.

This matches **Medium – Unbounded gas consumption** and **Medium – Permanent freezing of unclaimed yield** (fee minting).

### Likelihood Explanation
Adding a supported asset requires `TIME_LOCK_ROLE`, which is normal protocol governance, not an attacker action. Each supported asset adds roughly one `getAssetPrice` external call plus `3 × nodeDelegatorCount` external calls per iteration. With a 30 M gas block limit and ~10 NDCs, each new asset adds on the order of 50 k–100 k gas to `_getTotalEthInProtocol()`. This means roughly 300–600 supported assets would be needed to hit the limit — unlikely in the near term but possible as the protocol expands to support more LSTs/LRTs. There is no code-level cap preventing it.

### Recommendation
1. **Enforce a hard cap** on `supportedAssetList` length inside `_addNewSupportedAsset()`, analogous to `maxNodeDelegatorLimit` in `LRTDepositPool`.
2. **Alternatively**, refactor `_getTotalEthInProtocol()` to accept a subset of assets so the price update can be batched across multiple transactions.
3. **Or**, cache per-asset TVL values that are updated lazily on deposit/withdrawal rather than recomputing the full sum on every price update call.

### Proof of Concept
1. Admin (via `TIME_LOCK_ROLE`) calls `addNewSupportedAsset()` repeatedly, adding N LST/LRT tokens to `supportedAssetList`.
2. Admin adds several NDCs to `nodeDelegatorQueue` (up to `maxNodeDelegatorLimit`).
3. Any unprivileged caller calls `updateRSETHPrice()`.
4. `_getTotalEthInProtocol()` executes `N × (1 + 3 × ndcCount)` external calls.
5. Once N is large enough, the transaction reverts with out-of-gas.
6. All subsequent calls to `updateRSETHPrice()` and `updateRSETHPriceAsManager()` also revert; `rsETHPrice` is frozen at its last value permanently. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
