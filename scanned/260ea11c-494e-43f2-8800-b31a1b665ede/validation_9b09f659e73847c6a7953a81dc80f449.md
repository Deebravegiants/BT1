### Title
Missing Authorization on `sendFunds()` Allows Anyone to Trigger MEV Reward Distribution - (File: `contracts/FeeReceiver.sol`)

### Summary
`FeeReceiver.sendFunds()` has no access control, allowing any external caller to trigger the transfer of all accumulated MEV/execution-layer rewards from `FeeReceiver` to `LRTDepositPool` at any time of their choosing.

### Finding Description
`FeeReceiver` is the protocol's MEV and execution-layer reward accumulator. Its `sendFunds()` function is documented as the mechanism to forward accumulated ETH rewards to the deposit pool, which in turn increases the protocol TVL and the rsETH price. However, the function carries no access control modifier whatsoever:

```solidity
/// @dev send all rewards to deposit pool
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

Every other state-changing function in `FeeReceiver` is gated (`setDepositPool` requires `MANAGER` role), but `sendFunds()` is callable by any EOA or contract. The analog to the original report is direct: a function intended to be called only by a privileged protocol actor (the operator/manager) has no `onlyRole` or equivalent guard, making it freely callable by anyone.

The `receiveFromRewardReceiver()` target in `LRTDepositPool` is equally unguarded (`external payable { }`), so the full call chain executes without revert for any caller.

### Impact Explanation
An unprivileged caller can force the immediate distribution of all pending MEV rewards sitting in `FeeReceiver` into the deposit pool at a moment of their choosing. This artificially inflates the rsETH price at that instant. A caller who holds rsETH and is about to complete a withdrawal (via `LRTWithdrawalManager.completeWithdrawal`) can front-run their own withdrawal with a `sendFunds()` call, receiving a larger asset payout than they would have received had the rewards been distributed on the protocol's normal schedule. The funds are not stolen outright, but the caller extracts yield that would otherwise have been distributed proportionally to all rsETH holders over time.

**Impact: Low — Contract fails to deliver promised returns (proportional, time-averaged reward distribution) but does not lose value.**

### Likelihood Explanation
The function is `external` with zero preconditions. Any on-chain actor can call it at any block. The call costs only gas. The incentive exists whenever a meaningful ETH balance has accumulated in `FeeReceiver` and the caller holds rsETH they intend to withdraw.

### Recommendation
Add an access control guard matching the rest of the contract's privileged functions:

```solidity
function sendFunds() external onlyRole(LRTConstants.MANAGER) {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

### Proof of Concept
1. `FeeReceiver` has accumulated `X` ETH in MEV rewards (balance > 0).
2. Attacker holds rsETH and has a pending withdrawal request that is ready to complete.
3. Attacker calls `FeeReceiver.sendFunds()` — no role check, executes unconditionally.
4. `X` ETH is transferred to `LRTDepositPool`, increasing `getTotalAssetDeposits(ETH_TOKEN)` and therefore the rsETH price returned by `LRTOracle.rsETHPrice()`.
5. Attacker immediately calls `LRTWithdrawalManager.completeWithdrawal(asset, referralId)`.
6. `_processWithdrawalCompletion` pays out `request.expectedAssetAmount` (locked at initiation time), so in the queued-withdrawal path the attacker does not gain extra assets from this specific call. However, in the `instantWithdrawal` path, `getExpectedAssetAmount` is computed at execution time using the now-inflated rsETH price, so the attacker receives more underlying assets than they would have without the forced `sendFunds()` call.
