### Title
Owner Can Delay Vesting Start Timestamp Before Vesting Begins, Breaking Promised Claim Schedule - (`contracts/KERNEL/KernelTop100MerkleDistributor.sol`)

### Summary

`setVestingStartTimestamp` only guards against changes **after** vesting has started. It places no upper bound on the new timestamp, allowing the owner to push `vestingStartTimestamp` arbitrarily far into the future before vesting begins. Because `_getUnclaimedVestedAmount` anchors all vesting math to the current `vestingStartTimestamp`, any claim attempt between the original T1 and the new T2 returns zero, silently breaking the schedule users were promised at initialization.

### Finding Description

`setVestingStartTimestamp` enforces two checks: [1](#0-0) 

```solidity
if (vestingStartTimestamp > 0 && block.timestamp >= vestingStartTimestamp) {
    revert VestingAlreadyStarted();
}
if (_vestingStartTimestamp <= block.timestamp) {
    revert VestingStartInThePast();
}
```

The first check only blocks changes once `block.timestamp >= vestingStartTimestamp`. If `block.timestamp < T1`, the owner can freely call `setVestingStartTimestamp(T2)` with any `T2 > block.timestamp`, including `T2 >> T1`. There is no check that `T2 <= T1` (i.e., no constraint that the new timestamp cannot be later than the original).

`_getUnclaimedVestedAmount` then anchors all vesting math to the live `vestingStartTimestamp`: [2](#0-1) 

```solidity
if (currentTime <= startTime || currentTime <= vestingStartTimestamp) {
    return 0;
}
uint256 totalElapsedTime = currentTime - vestingStartTimestamp;
uint256 totalVestedAmount = (userTotalClaimableAmount * totalElapsedTime) / VESTING_DURATION;
```

If `vestingStartTimestamp` is moved from T1 to T2 (T2 > T1), any call to `claim` or `claimAndStake` at a time `t` where `T1 ≤ t < T2` hits the `currentTime <= vestingStartTimestamp` branch and returns 0, causing `NoTokensToClaim` to revert. [3](#0-2) 

### Impact Explanation

Users who observed the initialization event (`VestingStartTimestampSet(T1)`) and planned their claim schedule accordingly receive zero tokens when they attempt to claim during `[T1, T2)`. No tokens are lost — they remain claimable after T2 — but the contract fails to deliver the returns it promised at the time it was initialized. This matches the **Low** scope: *Contract fails to deliver promised returns, but doesn't lose value*.

### Likelihood Explanation

The owner must call `setVestingStartTimestamp` before `block.timestamp` reaches the original `vestingStartTimestamp`. This is an explicit admin action, but it requires no key compromise — it is a reachable code path using the contract's own interface. The function exists precisely to allow pre-start adjustments, and there is no timelock or governance delay protecting it. Operational delays (e.g., token not ready, deployment issues) are realistic triggers.

### Recommendation

Add a constraint that the new timestamp cannot exceed the previously set one, or remove the ability to push the timestamp forward entirely:

```solidity
// Option A: only allow moving the timestamp earlier
if (_vestingStartTimestamp >= vestingStartTimestamp) {
    revert CannotDelayVesting();
}

// Option B: disallow any change once vestingStartTimestamp is set
if (vestingStartTimestamp != 0) {
    revert VestingStartAlreadySet();
}
```

Alternatively, protect the function with a timelock so users have advance notice of any change.

### Proof of Concept

```solidity
// 1. Deploy with vestingStartTimestamp = block.timestamp + 1 days (T1)
// 2. Warp to block.timestamp + 12 hours  (before T1, vesting not started)
// 3. Owner calls setVestingStartTimestamp(block.timestamp + 2 days) (T2 > T1)
//    → passes: block.timestamp (12h) < vestingStartTimestamp (T1 = 1 day)
// 4. Warp to T1 + 1 hour  (original schedule would have 1h of vesting elapsed)
// 5. User calls claim(amount, proof)
//    → _getUnclaimedVestedAmount: currentTime (T1+1h) <= vestingStartTimestamp (T2=2days) → returns 0
//    → reverts with NoTokensToClaim
// 6. Assert: user receives 0 tokens despite original schedule having started 1 hour ago
``` [4](#0-3) [5](#0-4)

        if (_vestingStartTimestamp == 0) {
            revert ZeroValueProvided();
        }

        if (vestingStartTimestamp > 0 && block.timestamp >= vestingStartTimestamp) {
            revert VestingAlreadyStarted();
        }

        if (_vestingStartTimestamp <= block.timestamp) {
            revert VestingStartInThePast();
        }

        vestingStartTimestamp = _vestingStartTimestamp;
