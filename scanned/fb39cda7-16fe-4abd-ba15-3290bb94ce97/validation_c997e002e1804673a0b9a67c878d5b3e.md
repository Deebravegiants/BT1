### Title
Chainlink Price Feed Staleness Not Validated in `getAssetPrice()` - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `price`, performing no staleness or validity checks. This stale price feeds directly into rsETH minting calculations, allowing depositors to receive incorrect rsETH amounts.

### Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol`, the `getAssetPrice()` function fetches the Chainlink price feed but ignores `updatedAt`, `answeredInRound`, and `roundId`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
``` [1](#0-0) 

No check is made that:
- `block.timestamp - updatedAt` is within an acceptable heartbeat window
- `answeredInRound >= roundId` (round completeness)
- `price > 0` (valid answer)

By contrast, `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` — a sibling oracle in the same repo — explicitly checks `answeredInRound < roundID` and `timestamp == 0`: [2](#0-1) 

The stale price from `ChainlinkPriceOracle` propagates through `LRTOracle.getAssetPrice()`: [3](#0-2) 

And is consumed directly in `LRTDepositPool` to compute rsETH minting amounts: [4](#0-3) 

### Impact Explanation
If the Chainlink feed goes stale with a price lower than the current market price (e.e., during network congestion or oracle downtime), a depositor calling `depositAsset()` receives more rsETH than the fair value of their deposit. This dilutes existing rsETH holders and, at scale, drives the protocol toward insolvency. In the opposite direction (stale price higher than market), depositors receive fewer rsETH tokens than owed — the contract fails to deliver promised returns. Both directions represent a direct mis-accounting of share/asset value.

**Impact: Low → potential Critical** depending on price deviation magnitude and deposit volume during the stale window.

### Likelihood Explanation
Chainlink feeds have documented heartbeat intervals (e.g., 1 hour for ETH/stETH). During periods of L1 congestion or Chainlink node issues, feeds can lag. No external attacker action is required — any depositor transacting during a stale window is affected. The missing check is a straightforward code omission, not a configuration issue.

### Recommendation
Add staleness and validity checks in `ChainlinkPriceOracle.getAssetPrice()`, consistent with the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > MAX_PRICE_AGE) revert StalePrice(); // e.g. MAX_PRICE_AGE = 3600

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

### Proof of Concept
1. Chainlink's stETH/ETH feed goes stale (last update 2 hours ago, price = 0.98 ETH).
2. Market price has since recovered to 1.00 ETH.
3. Attacker (or any user) calls `LRTDepositPool.depositAsset(stETH, 1000e18)`.
4. `LRTOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` returns `0.98e18` (stale).
5. `rsethAmountToMint = (1000e18 * 0.98e18) / rsETHPrice` — user receives rsETH priced at 0.98 ETH/stETH instead of 1.00 ETH/stETH, minting ~2% excess rsETH.
6. No revert occurs; the stale price is accepted silently. [1](#0-0) [3](#0-2) [5](#0-4)
