Looking at the code carefully, I need to trace the exact flow in `bridgeAssets` and determine whether Stargate's protocol fee creates a real underbacking invariant break.

**Key code path in `bridgeAssets`:** [1](#0-0) 

The function:
1. Validates `minAmount <= amount` and `minAmount != 0` — but does **not** require `minAmount == amount`
2. Sends `nativeFee + amount` ETH to Stargate
3. Records `oftReceipt.amountReceivedLD` in `latestTxReceipt` — but takes **no corrective action** if `amountReceivedLD < amount`

**The invariant break:**

When a user deposits ETH, rsETH OFT is distributed based on the full deposited ETH (minus pool `feeBps`): [2](#0-1) 

The rsETH amount is calculated from `amount` (full ETH), so the L1 must receive exactly `amount` ETH to maintain backing. However, Stargate V2's `StargatePoolNative.send` always deducts a protocol fee from `amountLD`, delivering `amountReceivedLD = amountLD - protocolFee` to the destination. Since `minAmountLD` is only required to be `> 0` and `<= amount`, the BRIDGER_ROLE can (and in normal operation, will) set `minAmount` below `amount`, allowing Stargate to retain the difference. [3](#0-2) 

**Is this "admin/operator compromise"?**

No. The BRIDGER_ROLE cannot avoid this — Stargate **always** takes a protocol fee. Even a well-intentioned BRIDGER_ROLE caller setting `minAmount = amount - 1` will still result in `amountReceivedLD < amount`. The contract structurally never compensates for the Stargate fee shortfall.

**Is this "external dependency behavior is the only cause"?**

No. The root cause is the contract's design: it distributes rsETH OFT based on the full deposited ETH, but bridges ETH through a fee-taking protocol without adjusting for the fee. The contract could have accounted for this (e.g., by using `amountReceivedLD` to back-calculate rsETH distribution, or by requiring `minAmount == amount`), but it does not.

**Conclusion: This is a valid Low vulnerability.**

### Title
Stargate Protocol Fee Causes Permanent rsETH OFT Underbacking on L2 — (`contracts/pools/RSETHPoolNoWrapper.sol`)

### Summary
`RSETHPoolNoWrapper.bridgeAssets` sends `amount` ETH to Stargate but L1Vault receives `amountReceivedLD < amount` due to Stargate's protocol fee. Since rsETH OFT was already distributed to depositors based on the full `amount`, the L2 rsETH OFT supply permanently exceeds the ETH backing in L1's `LRTDepositPool`.

### Finding Description
When users deposit ETH via `deposit()`, rsETH OFT is minted proportional to the full deposited ETH (minus pool `feeBps`). The accumulated ETH is later bridged to L1 via `bridgeAssets(amount, minAmount, nativeFee)`.

In `bridgeAssets`, the `SendParam` is constructed with `amountLD = amount` and `minAmountLD = minAmount`, where `minAmount` is only required to satisfy `0 < minAmount <= amount`: [4](#0-3) 

Stargate V2's `StargatePoolNative` deducts a protocol fee from `amountLD`, so `oftReceipt.amountReceivedLD < amount`. The contract records this in `latestTxReceipt` but performs no corrective action: [5](#0-4) 

The gap `amount - amountReceivedLD` is retained by Stargate, never reaching L1Vault. Since rsETH OFT was already distributed against the full `amount`, the canonical rsETH token becomes permanently underbacked by exactly this fee amount per bridge operation.

### Impact Explanation
Every `bridgeAssets` call leaks a small amount of ETH backing to Stargate fees. Over time, the cumulative shortfall means the rsETH OFT supply on L2 exceeds the ETH held in L1's `LRTDepositPool`. The contract fails to deliver the promised 1:1 ETH backing for rsETH OFT. No ETH is stolen by an attacker — it goes to Stargate — so this fits **Low: Contract fails to deliver promised returns, but doesn't lose value**.

### Likelihood Explanation
This occurs on every single `bridgeAssets` call as long as Stargate charges a protocol fee (which is always). The BRIDGER_ROLE cannot avoid it — the fee is inherent to the Stargate protocol. Likelihood is **High** (certain to occur in normal operation).

### Recommendation
After calling `stargatePool.send`, compare `oftReceipt.amountReceivedLD` against `amount`. If `amountReceivedLD < amount`, either:
- Revert if the shortfall exceeds an acceptable threshold (e.g., require `amountReceivedLD >= minAmount` where `minAmount` is set tightly), or
- Track the cumulative shortfall and reduce future rsETH OFT distributions accordingly, or
- Pre-deduct the expected Stargate fee from the rsETH OFT amount given to depositors (quote the fee via `quoteSend` before distribution).
