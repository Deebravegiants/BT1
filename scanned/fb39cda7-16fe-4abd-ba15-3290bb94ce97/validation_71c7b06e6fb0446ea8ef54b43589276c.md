### Title
Deposit Limit Not Enforced at Exact Boundary for ETH Deposits - (File: contracts/LRTDepositPool.sol)

### Summary
In `LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit`, the ETH branch uses a strict `>` comparison instead of `>=` when checking against the deposit limit. When `totalAssetDeposits` equals exactly `depositLimitByAsset(ETH_TOKEN)`, the check returns `false` (not exceeded), allowing further ETH deposits even though the cap has been fully consumed.

### Finding Description
`_checkIfDepositAmountExceedesCurrentLimit` has two branches: one for ETH and one for ERC-20 assets.

```solidity
// contracts/LRTDepositPool.sol  lines 676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // <-- strict >
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

The function is the sole enforcement gate called from `_beforeDeposit`:

```solidity
// contracts/LRTDepositPool.sol  lines 661-663
if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
    revert MaximumDepositLimitReached();
}
```

When `totalAssetDeposits == depositLimitByAsset(ETH_TOKEN)`, the expression `totalAssetDeposits > depositLimit` evaluates to `false`, so `_beforeDeposit` does not revert and the deposit proceeds. The view helper `getAssetCurrentLimit` already returns `0` in this state (it uses the same `>` but the subtraction `depositLimit - totalAssetDeposits` yields `0`), creating an observable inconsistency: the public view reports zero remaining capacity while the enforcement gate still admits deposits.

The correct check should be `>=` so that a fully-consumed limit is treated as exceeded:

```solidity
return (totalAssetDeposits >= lrtConfig.depositLimitByAsset(asset));
```

### Impact Explanation
Any unprivileged depositor calling `depositETH` when `totalAssetDeposits == depositLimit` bypasses the protocol's ETH deposit cap. The deposit limit is the primary mechanism preventing the protocol from accepting more ETH than it is configured to handle. Bypassing it at the exact boundary allows rsETH to be minted against ETH that exceeds the intended ceiling, violating the protocol's stated deposit constraints. This maps to **Low — contract fails to deliver promised returns, but doesn't lose value**.

### Likelihood Explanation
The boundary condition `totalAssetDeposits == depositLimit` can be reached in two realistic ways: (a) an admin sets `depositLimitByAsset` to exactly the current total TVL (e.g., to freeze new deposits), or (b) a sequence of deposits naturally lands the total at the limit. In scenario (a) the bypass is immediately exploitable by any depositor. Likelihood is **Low** overall but non-negligible when admins use the limit as a hard stop.

### Recommendation
Change the strict inequality to a non-strict one in the ETH branch of `_checkIfDepositAmountExceedesCurrentLimit`:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits >= lrtConfig.depositLimitByAsset(asset));
}
```

This aligns the enforcement gate with the semantics of `getAssetCurrentLimit`, which already treats `totalAssetDeposits == depositLimit` as zero remaining capacity.

### Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = X`.
2. Through normal deposits, `getTotalAssetDeposits(ETH_TOKEN)` reaches exactly `X`.
3. `getAssetCurrentLimit(ETH_TOKEN)` returns `0` — signalling the cap is full.
4. A depositor calls `depositETH{value: 1 ether}(0, "")`.
5. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `X > X` → `false` → no revert.
6. `_mintRsETH` mints rsETH for the depositor, pushing total ETH deposits to `X + 1 ether` above the configured limit. [1](#0-0) [2](#0-1) [3](#0-2)
