### Title
`highestRsethPrice` Never Resets Downward Enables Permanent Griefing Loop via Public `updateRSETHPrice()` — (`contracts/LRTOracle.sol`)

### Summary
After a legitimate price drop (e.g., slashing), `highestRsethPrice` remains at its historical peak. Because `updateRSETHPrice()` is unrestricted and the downside-protection check always compares the current price against the all-time high, any public caller can re-trigger an auto-pause immediately after each admin unpause, indefinitely.

### Finding Description

`_updateRsETHPrice()` contains two relevant blocks:

**Downside protection** — pauses if the drop from the all-time high exceeds `pricePercentageLimit`: [1](#0-0) 

**`highestRsethPrice` update** — only ever increases: [2](#0-1) 

There is no admin setter for `highestRsethPrice` and no code path that resets it downward. The only reset is when `rsethSupply == 0` (line 220), which is not a realistic post-slashing state. [3](#0-2) 

`updateRSETHPrice()` is `public whenNotPaused` — callable by any EOA or contract: [4](#0-3) 

**Griefing loop (concrete):**
1. `highestRsethPrice = 1.2e18`, `pricePercentageLimit = 1e16` (1%), real price stabilizes at `1.0e18` after slashing.
2. Admin unpauses.
3. Attacker calls `updateRSETHPrice()`. `diff = 0.2e18 > 1e16 * 1.2e18 / 1e18 = 1.2e16` → `isPriceDecreaseOffLimit = true` → protocol auto-pauses again.
4. Repeat from step 2 indefinitely.

`updateRSETHPriceAsManager()` (manager-only, no `whenNotPaused`) calls the same `_updateRsETHPrice()` and would also trigger the same pause, so the manager cannot bypass this to update the price reference. [5](#0-4) 

### Impact Explanation

Every call to `updateRSETHPrice()` after a legitimate price drop exceeding `pricePercentageLimit` re-pauses `LRTDepositPool`, `WithdrawalManager`, and `LRTOracle` itself. Deposits and withdrawals are frozen. The admin's only workaround is to call `setPricePercentageLimit(0)`, which disables the downside-protection feature entirely — a forced trade-off between resuming operations and maintaining the safety mechanism. This does not cause insolvency or direct fund theft, so the correct impact is **Medium — Temporary freezing of funds** (not Critical/Protocol insolvency as the question title claims). [6](#0-5) 

### Likelihood Explanation

The entry point is fully public with no role restriction. After any slashing event that drops the price by more than `pricePercentageLimit` from the historical peak, the condition is permanently satisfied until `pricePercentageLimit` is zeroed out. Any caller — including a griefing bot — can exploit this with a single transaction after each admin unpause.

### Recommendation

When the admin unpauses after a confirmed legitimate price drop, `highestRsethPrice` should be reset to the current `rsETHPrice`. Add an admin function:

```solidity
function resetHighestRsethPrice() external onlyLRTAdmin {
    highestRsethPrice = rsETHPrice;
    emit HighestRsethPriceReset(rsETHPrice);
}
```

Alternatively, reset `highestRsethPrice` inside `unpause()` so the reference point is always anchored to the price at the time of resumption. [7](#0-6) 

### Proof of Concept

```solidity
// Setup:
// highestRsethPrice = 1.2e18 (set historically)
// pricePercentageLimit = 1e16 (1%)
// After slashing, oracle returns price = 1.0e18
// Protocol was paused; admin calls unpause()

for (uint i = 0; i < 3; i++) {
    // Protocol is unpaused at start of each iteration
    assert(!oracle.paused());

    // Any public caller triggers re-pause
    oracle.updateRSETHPrice(); // diff=0.2e18 > 1.2e16 → auto-pause

    assert(oracle.paused()); // paused again
    assert(depositPool.paused());
    assert(withdrawalManager.paused());

    // Admin unpauses all three for next iteration
    oracle.unpause();
    depositPool.unpause();
    withdrawalManager.unpause();
}
// Protocol is permanently griefable; highestRsethPrice unchanged at 1.2e18
```
