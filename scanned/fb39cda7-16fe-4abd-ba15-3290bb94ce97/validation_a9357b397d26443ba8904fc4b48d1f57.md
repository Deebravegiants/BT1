### Title
Phantom `totalETHDepositedToAave` Remainder After Loss-Scenario Reconfiguration Silently Absorbs Future Treasury Yield — (`contracts/LRTWithdrawalManager.sol`)

---

### Summary

When `configureAaveIntegration` is called while `aaveAWETH.balanceOf() < totalETHDepositedToAave` (an Aave loss state the code explicitly acknowledges), `_withdrawFromAave` reduces `totalETHDepositedToAave` only by the amount actually withdrawn — not by the full principal. This leaves a non-zero "phantom principal" in `totalETHDepositedToAave` after all funds have been physically removed from Aave. Subsequent deposits to the new pool increment this already-inflated counter, causing `_collectInterestToTreasury` to silently absorb the first `lossAmount` of real interest and never forward it to the treasury.

---

### Finding Description

**Root cause — `_withdrawFromAave` caps the accounting decrement at `aaveBalance`, not at `totalETHDepositedToAave`:** [1](#0-0) 

```solidity
// Only withdraw up to the principal amount (don't use accrued interest for user withdrawals)
uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave
    ? aaveBalance          // ← loss scenario: caps at aaveBalance, not totalETHDepositedToAave
    : totalETHDepositedToAave;

withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
totalETHDepositedToAave -= withdrawnAmount;   // ← only subtracts aaveBalance, not the full principal
```

In a loss scenario (`aaveBalance = 90`, `totalETHDepositedToAave = 100`):
- `withdrawablePrincipal = 90`
- `withdrawnAmount = 90`
- `totalETHDepositedToAave = 100 − 90 = 10` ← **phantom principal remains**

**Trigger path — `configureAaveIntegration` calls `_withdrawFromAave` without resetting the counter afterward:** [2](#0-1) 

`_collectInterestToTreasury` returns 0 (since `aaveBalance ≤ principal`), then `_withdrawFromAave(aaveBalance)` drains all physical funds but leaves `totalETHDepositedToAave = 10`. The new pool addresses are then written over the old ones with no reset of the counter. [3](#0-2) 

**Effect on future interest collection:** [4](#0-3) 

After reconfiguration, a 100 ETH deposit makes `totalETHDepositedToAave = 110`. Aave balance starts at 100. Until the balance exceeds 110, `_collectInterestToTreasury` returns 0 — the first 10 ETH of real yield is silently swallowed. The code's own NatDoc acknowledges this: *"Returns 0 if aaveBalance < totalETHDepositedToAave (potential accounting issue)"*. [5](#0-4) 

The same flaw exists in `setAaveIntegrationEnabled(false)`, which follows the identical `_collectInterestToTreasury → _withdrawFromAave` pattern. [6](#0-5) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

The treasury is permanently deprived of the first `lossAmount` of interest earned in the new pool. The yield is not lost to users; it stays in the contract and is eventually used to service withdrawals or sits idle — but it is never routed to the treasury as the protocol invariant requires. The magnitude equals the Aave loss at the time of reconfiguration (could be any amount above the 2-wei health-check tolerance).

---

### Likelihood Explanation

Likelihood is **low-to-medium**. Aave v3 aWETH principal loss is rare but explicitly anticipated by the codebase (`_checkAaveHealth`, the NatDoc comment on `getAccruedInterest`). The manager role is required, but the manager acts in good faith (e.g., migrating to a new Aave pool version). No malicious intent is needed — the bug fires automatically whenever reconfiguration occurs during any loss state, however small.

---

### Recommendation

After `_withdrawFromAave` completes a full-balance withdrawal during reconfiguration or disabling, explicitly reset the counter:

```solidity
// In configureAaveIntegration and setAaveIntegrationEnabled, after _withdrawFromAave:
if (aaveAWETH.balanceOf(address(this)) == 0) {
    totalETHDepositedToAave = 0;
}
```

Alternatively, `_withdrawFromAave` itself could detect the full-drain case and zero the counter unconditionally when `amount >= aaveBalance`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.21;

// Pseudocode for a local fork/unit test

function test_phantomPrincipalAfterLossReconfiguration() public {
    // Setup: 100 ETH deposited to Aave
    // totalETHDepositedToAave = 100e18

    // Simulate Aave loss: mock aWETH.balanceOf returns 90e18
    mockAWETH.setBalance(address(withdrawalManager), 90e18);

    // Manager reconfigures (e.g., migrating to new pool)
    vm.prank(manager);
    withdrawalManager.configureAaveIntegration(newPool, newGateway, newAWETH, newDataProvider);

    // After reconfiguration: totalETHDepositedToAave should be 0, but is 10e18
    assertEq(withdrawalManager.totalETHDepositedToAave(), 0); // FAILS: actual = 10e18

    // New deposit of 100 ETH
    vm.prank(operator);
    withdrawalManager.depositIdleETHToAave(100e18);
    // totalETHDepositedToAave = 110e18, aaveBalance = 100e18

    // Simulate 5 ETH interest accrual
    mockAWETH.setBalance(address(withdrawalManager), 105e18);

    // Collect interest — treasury should receive 5 ETH
    uint256 treasuryBefore = treasury.balance;
    vm.prank(operator);
    withdrawalManager.collectInterestToTreasury(); // reverts (unhealthy) or returns 0

    // Treasury received 0 ETH instead of 5 ETH
    assertEq(treasury.balance - treasuryBefore, 5e18); // FAILS: actual = 0
}
```
