### Title
Missing zero-rate guard in `setAgETHOracle` causes `deposit()` to panic with division-by-zero when `AGETHRateReceiver` is freshly deployed — (`contracts/agETH/AGETHPoolV3.sol`)

### Summary
`AGETHPoolV3.setAgETHOracle()` accepts any non-zero address without verifying the oracle returns a non-zero rate. A freshly deployed `AGETHRateReceiver` starts with `rate == 0`. During the window before the first LayerZero message is delivered (naturally or extended by block stuffing), every call to `deposit()` panics with an uninformative division-by-zero error instead of a descriptive revert.

### Finding Description
`CrossChainRateReceiver.rate` is a `uint256` storage variable that defaults to `0` on deployment. `AGETHRateReceiver` sets no initial rate in its constructor. When an admin deploys a new `AGETHRateReceiver` and calls `setAgETHOracle(newReceiver)`, the oracle immediately becomes active with `rate == 0`.

`AGETHPoolV3.deposit()` calls `viewSwapAgETHAmountAndFee()`, which fetches `agETHToETHrate = getRate()` and computes:

```solidity
agETHAmount = amountAfterFee * 1e18 / agETHToETHrate; // panics if agETHToETHrate == 0
```

This is a Solidity panic `0x12`, not a custom error. The inconsistency is clear: `addSupportedToken()` explicitly guards against zero-rate oracles for token oracles, but `setAgETHOracle()` has no equivalent check.

Block stuffing can extend this zero-rate window beyond the natural LZ delivery delay, but the vulnerability exists without it.

### Impact Explanation
Temporary freezing of ETH deposits via uninformative panic. All `deposit()` calls revert with `panic(0x12)` until a valid rate is delivered via LayerZero or the admin points `agETHOracle` to a working oracle. Impact is **temporary** (recovery exists), classifying as **Low — Block stuffing / temporary deposit freeze**.

### Likelihood Explanation
Requires an admin to deploy a new `AGETHRateReceiver` and call `setAgETHOracle()` before the first LZ message arrives. This is a realistic operational scenario (oracle redeployment). Block stuffing on L2s (Scroll, Linea) is difficult for external attackers but the natural delivery window alone is sufficient to trigger the bug.

### Recommendation
Add a zero-rate guard in `setAgETHOracle()`, mirroring the check already present in `addSupportedToken()`:

```solidity
function setAgETHOracle(address _agETHOracle) external onlyRole(DEFAULT_ADMIN_ROLE) {
    UtilLib.checkNonZeroAddress(_agETHOracle);
    if (IOracle(_agETHOracle).getRate() == 0) revert UnsupportedOracle();
    agETHOracle = _agETHOracle;
    emit OracleSet(_agETHOracle);
}
```

Additionally, add a zero-rate guard in `viewSwapAgETHAmountAndFee()` as a defense-in-depth measure:

```solidity
if (agETHToETHrate == 0) revert UnsupportedOracle();
```
