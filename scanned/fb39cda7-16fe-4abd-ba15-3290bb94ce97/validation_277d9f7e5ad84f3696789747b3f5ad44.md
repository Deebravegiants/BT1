### Title
No Slippage Protection in RSETHPool `deposit` Functions Allows Users to Receive Less rsETH Than Expected - (File: contracts/pools/RSETHPool.sol)

### Summary
All `deposit` functions across the RSETHPool contract family lack a `minRSETHAmountExpected` parameter, giving depositors no protection against oracle rate changes between transaction submission and execution. In volatile conditions, users can receive significantly fewer rsETH tokens than they anticipated.

### Finding Description
The `deposit(string memory referralId)` and `deposit(address token, uint256 amount, string memory referralId)` functions in `RSETHPool.sol` (and its variants `RSETHPoolV3WithNativeChainBridge.sol`, `RSETHPoolV2.sol`, `RSETHPoolV3.sol`, `RSETHPoolNoWrapper.sol`) compute the rsETH output amount solely from the oracle rate at execution time, with no caller-supplied minimum:

```solidity
// RSETHPool.sol line 265-278
function deposit(string memory referralId) external payable nonReentrant whenNotPaused {
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount);
    ...
}
``` [1](#0-0) 

The rsETH amount is derived from `viewSwapRsETHAmountAndFee`, which calls `getRate()` on the oracle:

```solidity
// RSETHPool.sol line 311-319
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
``` [2](#0-1) 

The same pattern is present in `RSETHPoolV3WithNativeChainBridge.sol`: [3](#0-2) [4](#0-3) 

If the oracle-reported rsETH/ETH rate rises between when a user submits the transaction and when it is included in a block, the user receives fewer rsETH tokens than they expected, with no on-chain mechanism to revert the transaction.

By contrast, the upstream `LRTDepositPool.depositETH` correctly enforces a caller-supplied minimum: [5](#0-4) 

### Impact Explanation
Users depositing ETH or supported LSTs into any RSETHPool contract during volatile oracle conditions receive fewer rsETH tokens than they intended. Their ETH is consumed and cannot be recovered. This maps to **Low: Contract fails to deliver promised returns, but doesn't lose value** — the protocol retains the ETH correctly, but the user's rsETH entitlement is silently reduced below their expectation.

### Likelihood Explanation
The rsETH oracle rate is updated by an off-chain operator and can move in discrete steps (as shown in the external report's DEX price charts). Any deposit transaction pending in the mempool during an oracle update is exposed. This is a normal operating condition, not an edge case, making it a realistic and recurring risk for any depositor.

### Recommendation
Add a `minRSETHAmountExpected` parameter to all `deposit` functions and revert if the computed rsETH amount falls below it:

```solidity
function deposit(string memory referralId, uint256 minRSETHAmountExpected)
    external payable nonReentrant whenNotPaused
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRSETHAmountExpected) revert SlippageExceeded();
    ...
}
```

Apply the same change to the token `deposit` overload and to all pool variants (`RSETHPoolV2.sol`, `RSETHPoolV3.sol`, `RSETHPoolV3WithNativeChainBridge.sol`, `RSETHPoolNoWrapper.sol`).

### Proof of Concept
1. User calls `RSETHPool.deposit{value: 10 ether}("ref")` when the oracle reports 1 rsETH = 1.00 ETH, expecting ~10 rsETH.
2. Before the transaction is mined, the oracle operator updates the rate to 1 rsETH = 1.10 ETH (a 10% jump, consistent with the volatile price movements shown in the external report).
3. `viewSwapRsETHAmountAndFee(10 ether)` now returns `10e18 * 1e18 / 1.10e18 ≈ 9.09 rsETH`.
4. The user receives ~9.09 rsETH instead of ~10 rsETH, losing ~0.91 rsETH of value with no recourse, because no minimum was enforced. [2](#0-1) [1](#0-0)
