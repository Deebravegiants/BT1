### Title
Critical Protocol Configuration Functions Controlled by `DEFAULT_ADMIN_ROLE` Instead of `TIME_LOCK_ROLE`, Bypassing Timelock Protection - (File: contracts/LRTConfig.sol, contracts/LRTOracle.sol)

### Summary
The LRT-rsETH protocol explicitly defines a `TIME_LOCK_ROLE` for timelock-protected operations and applies it to `addNewSupportedAsset()`. However, equally or more critical functions — `updatePriceOracleFor()`, `setContract()`, and `setRSETH()` — are gated only by `DEFAULT_ADMIN_ROLE` (a multisig or EOA), bypassing the timelock entirely. This mirrors the original report's pattern: a less-secure role controls functions that should be under a more secure governance mechanism.

### Finding Description
`LRTConstants.TIME_LOCK_ROLE` is defined and applied to `addNewSupportedAsset()` in `LRTConfig.sol`: [1](#0-0) [2](#0-1) 

This establishes the protocol's intent: operations that alter the protocol's asset registry should go through a timelock. Yet the following functions — which are at least as critical — are gated only by `DEFAULT_ADMIN_ROLE`:

**`LRTOracle.updatePriceOracleFor()`** — sets the price oracle for any asset, directly determining the rsETH/ETH exchange rate used for all deposits and withdrawals: [3](#0-2) 

**`LRTConfig.setContract()`** — replaces any core contract address (e.g., `LRT_ORACLE`, `LRT_DEPOSIT_POOL`, `LRT_WITHDRAW_MANAGER`) stored in `contractMap`: [4](#0-3) 

**`LRTConfig.setRSETH()`** — replaces the rsETH token address itself: [5](#0-4) 

The `onlyLRTAdmin` modifier resolves to `DEFAULT_ADMIN_ROLE` in `LRTConfig`, which is a multisig or EOA — not a timelock contract: [6](#0-5) 

The `DEFAULT_ADMIN_ROLE` is granted at initialization to a single `admin` address with no enforced delay: [7](#0-6) 

### Impact Explanation
**Impact: Low** — The protocol fails to deliver its promised security posture. The existence of `TIME_LOCK_ROLE` signals to users and integrators that sensitive configuration changes are subject to a delay, giving them time to react. By controlling `updatePriceOracleFor()`, `setContract()`, and `setRSETH()` through `DEFAULT_ADMIN_ROLE` alone, the protocol can silently swap oracles or core contract addresses with no on-chain delay. Users cannot anticipate or exit before such changes take effect. No direct fund theft occurs from the design flaw alone, but the promised governance security guarantee is not delivered.

### Likelihood Explanation
**Likelihood: High** — These functions are the routine operational levers for protocol maintenance (oracle updates, contract upgrades). They will always be exercised by the `DEFAULT_ADMIN_ROLE` holder without timelock protection, as the current code provides no alternative path. The inconsistency is structural and permanent unless remediated.

### Recommendation
Apply `onlyRole(LRTConstants.TIME_LOCK_ROLE)` (or an equivalent timelock-enforced modifier) to `updatePriceOracleFor()`, `setContract()`, and `setRSETH()`, consistent with the timelock protection already applied to `addNewSupportedAsset()`. This aligns the security posture of all critical configuration functions with the protocol's stated governance model.

### Proof of Concept
1. `TIME_LOCK_ROLE` is defined in `LRTConstants` and applied to `addNewSupportedAsset()` — establishing the protocol's intent for timelock-gated configuration.
2. `updatePriceOracleFor()` in `LRTOracle.sol` (line 113) uses `onlyLRTAdmin`, which checks `DEFAULT_ADMIN_ROLE` in `LRTConfig` — no timelock.
3. `setContract()` in `LRTConfig.sol` (line 237) uses `onlyRole(DEFAULT_ADMIN_ROLE)` — no timelock.
4. `setRSETH()` in `LRTConfig.sol` (line 215) uses `onlyRole(DEFAULT_ADMIN_ROLE)` — no timelock.
5. The `DEFAULT_ADMIN_ROLE` is granted at `initialize()` to a plain `admin` address with no enforced delay, meaning any holder of this key can execute these changes atomically, with no window for users to react.
