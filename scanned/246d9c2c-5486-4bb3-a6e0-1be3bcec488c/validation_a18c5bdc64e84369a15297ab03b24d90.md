### Title
Dust Deposit Mints Zero rsETH, Permanently Losing User Funds - (File: contracts/pools/RSETHPoolV3.sol)

---

### Summary

`RSETHPoolV3.sol`'s `deposit` functions only guard against `amount == 0` but impose no minimum deposit floor. For any dust-sized deposit, the integer division in `viewSwapRsETHAmountAndFee` silently rounds `rsETHAmount` down to zero. The user's ETH or tokens are transferred into the pool, `wrsETH.mint(msg.sender, 0)` is called (which succeeds silently in OpenZeppelin ERC20), and the user receives nothing. The deposited value is absorbed into the pool's bridgeable balance with no user claim, constituting a permanent loss.

---

### Finding Description

Both public `deposit` entry points in `RSETHPoolV3` check only for a strict zero amount:

```solidity
// ETH path
if (amount == 0) revert InvalidAmount();
``` [1](#0-0) 

```solidity
// Token path
if (amount == 0) revert InvalidAmount();
``` [2](#0-1) 

The rsETH amount to mint is computed in `viewSwapRsETHAmountAndFee`:

```solidity
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [3](#0-2) 

For the token path:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [4](#0-3) 

Because `rsETHToETHrate` is denominated in 1e18 (rsETH trades at a premium to ETH, e.g. ~1.05e18), any `amountAfterFee < rsETHToETHrate / 1e18` (i.e., any deposit below ~1 wei of rsETH value) produces `rsETHAmount = 0` via integer truncation. The `limitDailyMint` modifier runs first and silently adds 0 to `dailyMintAmount`:

```solidity
dailyMintAmount += rsETHAmount;   // += 0, no-op
_;
``` [5](#0-4) 

Execution then falls through to the function body, which passes the `amount == 0` guard (since `amount > 0`), and calls:

```solidity
wrsETH.mint(msg.sender, rsETHAmount);   // mint(msg.sender, 0)
``` [6](#0-5) 

OpenZeppelin's `ERC20._mint` does not revert on a zero amount — it emits a `Transfer` event and returns. The user's ETH is now held by the pool:

```solidity
feeEarnedInETH += fee;   // += 0
``` [7](#0-6) 

No fee is recorded, no rsETH is minted, and the deposited ETH is silently absorbed into the pool's bridgeable balance (`getETHBalanceMinusFees()`), which is eventually bridged to L1. The user has no on-chain claim to recover it.

By contrast, `LRTDepositPool.sol` correctly enforces a `minAmountToDeposit` floor:

```solidity
if (depositAmount == 0 || depositAmount < minAmountToDeposit) {
    revert InvalidAmountToDeposit();
}
``` [8](#0-7) 

`RSETHPoolV3` has no equivalent guard.

---

### Impact Explanation

A user who deposits a dust amount (any value whose `rsETHAmount` truncates to 0) permanently loses their ETH or tokens. The funds flow into the pool's bridgeable balance and are eventually sent to L1 with no attribution to the depositor. There is no recovery path. This matches **"Contract fails to deliver promised returns"** (Low) at minimum, and arguably **"Permanent freezing of funds"** (Critical) from the individual user's perspective since the user's ETH is irrecoverably absorbed.

---

### Likelihood Explanation

Requires a user to submit a dust-sized deposit — possible via user input error (wrong decimal units), a buggy front-end, or an automated integration that computes a residual amount. Likelihood: **Low** (mirrors the original report's likelihood of 1/10).

---

### Recommendation

Add a post-calculation guard before minting, analogous to `LRTDepositPool`'s `minRSETHAmountExpected` slippage check:

```solidity
if (rsETHAmount == 0) revert InvalidAmount();
```

Apply this to both the ETH and token `deposit` paths in `RSETHPoolV3.sol` immediately after calling `viewSwapRsETHAmountAndFee`.

---

### Proof of Concept

1. `rsETHToETHrate` = 1.05e18 (rsETH at a 5% premium to ETH).
2. User calls `RSETHPoolV3.deposit{value: 1}("")` (1 wei ETH).
3. `limitDailyMint` modifier: `rsETHAmount = 1 * 1e18 / 1.05e18 = 0`; `dailyMintAmount += 0` (no-op).
4. Function body: `amount = 1`, passes `if (amount == 0)` check.
5. `viewSwapRsETHAmountAndFee(1)` → `fee = 0`, `rsETHAmount = 0`.
6. `feeEarnedInETH += 0`.
7. `wrsETH.mint(msg.sender, 0)` — succeeds, emits `Transfer(address(0), user, 0)`.
8. User's 1 wei is now in the pool's ETH balance; user holds 0 rsETH and has no recovery path.
