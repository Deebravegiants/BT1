### Title
Stale `rsETHPrice` Allows Depositors to Mint Excess rsETH Before Public Price Update, Diluting Existing Holders - (File: `contracts/LRTOracle.sol`, `contracts/LRTDepositPool.sol`)

---

### Summary

`LRTOracle.updateRSETHPrice()` is an unrestricted public function. Between keeper-driven updates, `rsETHPrice` is stale. `LRTDepositPool.getRsETHAmountToMint()` uses this stale stored price as the denominator when minting rsETH. An attacker can deposit at the stale (lower) price to receive more rsETH than fair value, then immediately call `updateRSETHPrice()` to crystallise the price increase, profiting at the expense of existing rsETH holders.

---

### Finding Description

`LRTOracle.updateRSETHPrice()` is declared `public` with no role restriction:

```solidity
// contracts/LRTOracle.sol:87
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

`_updateRsETHPrice()` recomputes the price as `totalETHInProtocol / rsethSupply`, where `totalETHInProtocol` includes all accrued staking rewards that have not yet been reflected in the stored `rsETHPrice`. Between keeper updates, `rsETHPrice` is therefore stale and lower than the true value.

`LRTDepositPool.getRsETHAmountToMint()` uses this stale stored price as the denominator:

```solidity
// contracts/LRTDepositPool.sol:520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

Because `rsETHPrice` is stale-low, the division yields a larger-than-fair `rsethAmountToMint`. The attacker's steps are:

1. Observe that staking rewards have accrued (on-chain `totalETHInProtocol` is higher than `rsethSupply × rsETHPrice_stored`).
2. Call `LRTDepositPool.depositETH()` or `depositAsset()` at the stale low price, receiving excess rsETH.
3. Call `LRTOracle.updateRSETHPrice()` to push the price to its true higher value.
4. The attacker now holds rsETH minted at a discount; existing holders' proportional claim on protocol TVL is permanently diluted.

This is the direct analog of the UnionDAO `exchangeRateStored()` bug: a stored exchange rate that lags behind actual accrued value, exploitable by depositing before the rate update and exiting (or simply holding) after it.

---

### Impact Explanation

Every unit of excess rsETH minted to the attacker represents a unit of protocol TVL that was owed to pre-existing rsETH holders. The attacker captures a portion of the accrued staking yield that should have been distributed pro-rata to existing holders. This is a direct, quantifiable theft of unclaimed yield (High impact). The magnitude per attack is bounded by the `pricePercentageLimit` guard in `_updateRsETHPrice()`, but the attack is repeatable across every reward accrual cycle.

---

### Likelihood Explanation

`updateRSETHPrice()` carries no access control. Any EOA or contract can call it. The price is stale between keeper updates, which occur on a regular schedule. The attacker needs only to monitor the on-chain TVL (via `LRTDepositPool.getTotalAssetDeposits()` and asset price oracles) to detect when `totalETHInProtocol > rsethSupply × rsETHPrice`. This is trivially observable. No privileged access, leaked keys, or external protocol compromise is required. Likelihood is **Medium-High**.

---

### Recommendation

1. **Remove the public access to `updateRSETHPrice()`** or restrict it to `onlyLRTManager`, matching the pattern of `updateRSETHPriceAsManager()`.
2. **Update the price atomically inside `depositETH`/`depositAsset`** before computing `rsethAmountToMint`, so the minting always uses the freshest price.
3. Alternatively, implement a deposit fee or a time-weighted price to make single-block exploitation unprofitable (analogous to the exit-fee recommendation in the UnionDAO discussion).

---

### Proof of Concept

```
Precondition: rsETHPrice stored = 1.00 ETH/rsETH (stale)
              actual TVL / rsethSupply = 1.01 ETH/rsETH (rewards accrued)

Step 1: Attacker calls depositETH(100 ETH)
        rsethAmountToMint = 100e18 * 1e18 / 1.00e18 = 100 rsETH   ← excess
        Fair amount       = 100e18 * 1e18 / 1.01e18 ≈ 99.01 rsETH

Step 2: Attacker calls LRTOracle.updateRSETHPrice()
        rsETHPrice updated to 1.01 ETH/rsETH

Result: Attacker holds 100 rsETH worth 101 ETH at the new price.
        Existing holders' rsETH is diluted by ~0.99 rsETH worth of TVL.
        Attacker profit ≈ 1 ETH per 100 ETH deposited per reward cycle.

    function _updateRsETHPrice() internal {
        address rsETHTokenAddress = lrtConfig.rsETH();
        uint256 rsethSupply = IRSETH(rsETHTokenAddress).totalSupply();

        if (rsethSupply == 0) {
            rsETHPrice = 1 ether;
            highestRsethPrice = 1 ether;
            return;
        }

        if (highestRsethPrice == 0) {
            highestRsethPrice = rsETHPrice;
        }

        uint256 previousPrice = rsETHPrice;

        // get total ETH in the protocol (normalized to 1e18)
        uint256 totalETHInProtocol = _getTotalEthInProtocol();

        // calculate previousTVL using rsethSupply multiplied by rsETHPrice
        uint256 previousTVL = rsethSupply.mulWad(rsETHPrice);

        IPausable lrtDepositPool = IPausable(lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL));
        IPausable withdrawalManager = IPausable(lrtConfig.getContract(LRTConstants.LRT_WITHDRAW_MANAGER));

        // determine if the protocol is active (not paused)
        bool protocolPaused = lrtDepositPool.paused() || withdrawalManager.paused() || paused;

        // only take fee if TVL increased and protocol is not paused
        uint256 protocolFeeInETH = 0;
        if (!protocolPaused && totalETHInProtocol > previousTVL) {
            uint256 rewardAmount = totalETHInProtocol - previousTVL;
            protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
        }

        // compute new rsETH price based on total ETH minus fee
        uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);

    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
