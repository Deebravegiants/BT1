### Title
`instantWithdrawal()` Checks `queuedWithdrawalsBuffer` (Default Zero) Instead of `assetsCommitted`, Allowing Vault Drainage That Freezes Queued Withdrawal Users' Funds - (File: contracts/LRTWithdrawalManager.sol)

---

### Summary

`LRTWithdrawalManager.instantWithdrawal()` enforces an availability check against `LRTUnstakingVault.getAssetsAvailableForInstantWithdrawal()`, which subtracts a manually-set `queuedWithdrawalsBuffer` from the vault balance. This buffer defaults to zero and is never automatically synchronized with `assetsCommitted` — the on-chain accounting variable that tracks how much of the vault is already promised to users who have queued standard withdrawals. As a result, an unprivileged caller can drain the `LRTUnstakingVault` of assets that are already committed to queued withdrawal users, leaving those users unable to complete their withdrawals.

---

### Finding Description

The protocol supports two withdrawal paths:

**Queued withdrawal** (`initiateWithdrawal` → `unlockQueue` → `completeWithdrawal`):
- User transfers rsETH to `LRTWithdrawalManager`.
- `assetsCommitted[asset]` is incremented by `expectedAssetAmount`.
- Assets sit in `LRTUnstakingVault` until the operator calls `unlockQueue()`, which pulls them into `LRTWithdrawalManager` for final claim.

**Instant withdrawal** (`instantWithdrawal`):
- User burns rsETH directly.
- Assets are pulled immediately from `LRTUnstakingVault` via `unstakingVault.redeem()`.

The availability guard in `instantWithdrawal()` is:

```solidity
if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
    revert CantInstantWithdrawMoreThanAvailable();
}
``` [1](#0-0) 

`getAssetsAvailableForInstantWithdrawal()` computes:

```solidity
uint256 vaultBalance = balanceOf(asset);
uint256 reservedBuffer = queuedWithdrawalsBuffer[asset];
availableAmount = reservedBuffer >= vaultBalance ? 0 : vaultBalance - reservedBuffer;
``` [2](#0-1) 

`queuedWithdrawalsBuffer[asset]` is a manually-set operator value that **defaults to zero** for every asset: [3](#0-2) 

It is only updated when the operator explicitly calls `setQueuedWithdrawalsBuffer()`: [4](#0-3) 

By contrast, `assetsCommitted[asset]` in `LRTWithdrawalManager` is automatically incremented every time a user queues a withdrawal: [5](#0-4) 

These two accounting variables are **never automatically synchronized**. When `queuedWithdrawalsBuffer` is zero (the default), `getAssetsAvailableForInstantWithdrawal()` returns the full vault balance, even if the entire balance is already committed to queued withdrawal users via `assetsCommitted`.

The queued withdrawal path correctly guards against over-commitment: [6](#0-5) 

But the instant withdrawal path does not use `assetsCommitted` at all — it only uses the unrelated, manually-set `queuedWithdrawalsBuffer`.

---

### Impact Explanation

When `queuedWithdrawalsBuffer` is zero (the default state for all assets), an attacker holding rsETH can call `instantWithdrawal()` to drain the entire `LRTUnstakingVault` balance for a given asset. Assets that were already promised to queued withdrawal users (tracked by `assetsCommitted`) are consumed. When the operator subsequently calls `unlockQueue()`, the vault is empty and cannot service the queued requests. The queued withdrawal users' rsETH is already held in `LRTWithdrawalManager` (transferred during `initiateWithdrawal()`), and they cannot recover it until the vault is replenished — which may never happen. This constitutes **temporary or permanent freezing of user funds**, and in practice the attacker receives assets that were reserved for other users.

**Impact: Medium — Temporary (potentially permanent) freezing of funds for queued withdrawal users.**

---

### Likelihood Explanation

- `queuedWithdrawalsBuffer` defaults to zero for every asset. No operator action is required to create the vulnerable state; it is the initial state.
- Any user holding rsETH can call `instantWithdrawal()` permissionlessly whenever `isInstantWithdrawalEnabled[asset]` is true.
- The attacker only needs to observe that `assetsCommitted[asset] > 0` and that the vault holds the corresponding assets, then call `instantWithdrawal()` before `unlockQueue()` is called.
- No front-running of a specific transaction is required; the window is open for the entire period between `initiateWithdrawal()` and `unlockQueue()`.

**Likelihood: Medium.**

---

### Recommendation

Replace the `queuedWithdrawalsBuffer`-based check in `instantWithdrawal()` with a check that accounts for `assetsCommitted`. One approach: expose `assetsCommitted[asset]` from `LRTWithdrawalManager` and use it inside `getAssetsAvailableForInstantWithdrawal()`, or perform the check directly in `instantWithdrawal()`:

```solidity
uint256 vaultBalance = unstakingVault.balanceOf(asset);
uint256 committed = assetsCommitted[asset];
uint256 available = committed >= vaultBalance ? 0 : vaultBalance - committed;
if (assetAmountUnlocked > available) revert CantInstantWithdrawMoreThanAvailable();
```

This mirrors the pattern used in `initiateWithdrawal()` and ensures that assets committed to queued withdrawal users are never consumed by instant withdrawals.

---

### Proof of Concept

1. Alice calls `initiateWithdrawal(stETH, 10e18, "")`. Her rsETH is transferred to `LRTWithdrawalManager`. `assetsCommitted[stETH] = 10e18`. The corresponding stETH sits in `LRTUnstakingVault`.
2. `queuedWithdrawalsBuffer[stETH]` is 0 (default, never set by operator).
3. Bob calls `instantWithdrawal(stETH, rsETHEquivalentOf10stETH, "")`.
   - `getAssetsAvailableForInstantWithdrawal(stETH)` returns `vaultBalance - 0 = 10e18`. Check passes.
   - `unstakingVault.redeem(stETH, 10e18)` drains the vault.
   - Bob receives 10 stETH (minus fee).
4. Operator calls `unlockQueue(stETH, ...)`. `params.totalAvailableAssets == 0`. Reverts or unlocks nothing.
5. Alice calls `completeWithdrawal(stETH, "")`. The withdrawal manager has no stETH to send. Alice's rsETH remains locked in `LRTWithdrawalManager` indefinitely.
