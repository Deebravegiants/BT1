### Title
`LRTOracle.updatePriceOracleFor` Missing Sanity Check Allows Setting Faulty Oracle, Freezing Deposits and Withdrawals - (File: contracts/LRTOracle.sol)

---

### Summary

`LRTOracle.updatePriceOracleFor` sets a new asset price oracle without validating that the oracle is functional. If a faulty oracle is installed (even by honest mistake), every downstream call to `getAssetPrice` reverts, permanently breaking deposits, withdrawals, and rsETH price updates until the oracle is replaced.

---

### Finding Description

`LRTOracle` exposes two oracle-setter functions:

- `updatePriceOracleForValidated` (lines 101–108): calls `IPriceFetcher(priceOracle).getAssetPrice(asset)` and reverts if the returned price is outside `[1e16, 1e19]`. [1](#0-0) 
- `updatePriceOracleFor` (lines 113–119): **`public`**, only checks non-zero address for supported assets, then unconditionally writes `assetPriceOracle[asset] = priceOracle`. No liveness check is performed. [2](#0-1) 

Because `updatePriceOracleFor` is `public`, any `onlyLRTAdmin` caller can invoke it directly, bypassing the validation in `updatePriceOracleForValidated`. If a faulty oracle (one whose `getAssetPrice` always reverts) is written for any supported asset, the following chain breaks:

1. `getAssetPrice(asset)` reverts — it delegates unconditionally to `IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset)`. [3](#0-2) 
2. `_getTotalEthInProtocol()` reverts — it iterates all supported assets and calls `getAssetPrice` for each. [4](#0-3) 
3. `_updateRsETHPrice()` → `updateRSETHPrice()` reverts — rsETH price is frozen at its last value. [5](#0-4) 
4. `LRTDepositPool.getRsETHAmountToMint()` reverts — calls `lrtOracle.getAssetPrice(asset)` directly. [6](#0-5) 
5. `LRTDepositPool.depositAsset()` and `depositETH()` revert — both call `_beforeDeposit` → `getRsETHAmountToMint`. [7](#0-6) 
6. `LRTWithdrawalManager.unlockWithdrawals()` reverts — calls `lrtOracle.getAssetPrice(asset)` inside `_createUnlockParams`. [8](#0-7) 

The analogy to the reported Euler bug is direct: just as `setInterestRateModel` resets the interest rate to 0 and then calls the new IRM without guarding against a reverting model, `updatePriceOracleFor` writes the new oracle address without verifying it is callable, leaving the protocol in a broken state if the oracle is faulty.

---

### Impact Explanation

**Medium — Temporary freezing of funds.**

All user-facing deposit and withdrawal paths revert until an admin replaces the faulty oracle. The rsETH price update mechanism is also frozen. Users with pending withdrawal requests cannot have them unlocked. No funds are permanently lost, but the protocol is operationally halted for all depositors and withdrawers.

---

### Likelihood Explanation

The `onlyLRTAdmin` role is trusted, but the existence of `updatePriceOracleForValidated` as a safer sibling function demonstrates that the team already recognizes the need for a liveness check. An admin may accidentally call the unvalidated `updatePriceOracleFor` directly (e.g., via a script or multisig transaction that targets the wrong function selector), or may set an oracle contract that is temporarily functional at the time of the call but later becomes permanently reverting (e.g., a misconfigured Chainlink feed). This mirrors the Euler finding where governance is trusted but the protocol should still guard against honest errors.

---

### Recommendation

Change `updatePriceOracleFor` from `public` to `internal` so it can only be reached through `updatePriceOracleForValidated`, which already performs the liveness check. If a bypass path is intentionally needed (e.g., to set `address(0)` for unsupported assets), add a separate, clearly scoped function for that case rather than leaving the unvalidated path open for supported assets.

---

### Proof of Concept

```
1. Admin calls updatePriceOracleFor(stETH, faultyOracle)
   where faultyOracle.getAssetPrice() always reverts.

2. User calls LRTDepositPool.depositAsset(stETH, 1e18, 0).
   → _beforeDeposit(stETH, 1e18, 0)
   → getRsETHAmountToMint(stETH, 1e18)
   → lrtOracle.getAssetPrice(stETH)
   → IPriceFetcher(faultyOracle).getAssetPrice(stETH)  ← REVERTS
   → entire deposit reverts.

3. Anyone calls updateRSETHPrice().
   → _updateRsETHPrice()
   → _getTotalEthInProtocol()
   → getAssetPrice(stETH)  ← REVERTS
   → rsETH price is permanently frozen.

4. Operator calls LRTWithdrawalManager.unlockWithdrawals(stETH, ...).
   → _createUnlockParams(lrtOracle, unstakingVault, stETH)
   → lrtOracle.getAssetPrice(stETH)  ← REVERTS
   → no withdrawal requests can be unlocked.
```

All three critical protocol paths — deposit, price update, and withdrawal unlock — are frozen until an admin issues a corrective `updatePriceOracleFor` call with a valid oracle.

    function updatePriceOracleForValidated(address asset, address priceOracle) external onlyLRTAdmin {
        // Sanity check: oracle price must have precision between 1e16 and 1e19
        uint256 price = IPriceFetcher(priceOracle).getAssetPrice(asset);
        if (price > 1e19 || price < 1e16) {
            revert InvalidPriceOracle();
        }
        updatePriceOracleFor(asset, priceOracle);
    }
