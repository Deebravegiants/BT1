### Title
Auto-pause Loop: `highestRsethPrice` Never Reset on Unpause Causes Repeated Re-pause — (`contracts/LRTOracle.sol`)

---

### Summary

When `_updateRsETHPrice()` triggers the downside auto-pause, it returns early at line 281 without writing `rsETHPrice` or resetting `highestRsethPrice`. After admin calls `unpause()`, the next public call to `updateRSETHPrice()` recomputes `newRsETHPrice` against the same stale `highestRsethPrice`. If the underlying asset price has not fully recovered above `highestRsethPrice`, the auto-pause fires again immediately, creating a loop that admin `unpause()` alone cannot break.

---

### Finding Description

**Auto-pause path (lines 270–281):** [1](#0-0) 

When `isPriceDecreaseOffLimit` is true, the function pauses `lrtDepositPool`, `withdrawalManager`, and `LRTOracle` itself, then **returns early**. Neither `rsETHPrice` nor `highestRsethPrice` is updated.

**`unpause()` (lines 143–146):** [2](#0-1) 

`unpause()` only sets `paused = false`. It does not reset `highestRsethPrice`, does not unpause `lrtDepositPool` or `withdrawalManager`, and provides no mechanism to adjust the peak price reference.

**`highestRsethPrice` update path (lines 294–296):** [3](#0-2) 

`highestRsethPrice` is only ever updated upward (or reset to `1 ether` when supply is zero). There is no admin function to lower or reset it.

**Re-pause condition (lines 273–274):** [4](#0-3) 

The condition `diff > pricePercentageLimit.mulWad(highestRsethPrice)` uses the same stale `highestRsethPrice` on every call. If the underlying price recovers to, say, 1.5% below the peak while the threshold is 1%, the condition remains true indefinitely.

**`updateRSETHPriceAsManager()` does not help (lines 94–96):** [5](#0-4) 

The manager-only bypass only skips the **upside** revert at line 264. The downside auto-pause block at lines 277–281 has no role check — it fires for any caller, including the manager. [6](#0-5) 

---

### Impact Explanation

Deposits and withdrawals are blocked for the duration of the loop. `lrtDepositPool` and `withdrawalManager` are paused by the auto-pause trigger and are not unpaused by `LRTOracle.unpause()`. Even after the admin unpauses all three contracts individually, the next public `updateRSETHPrice()` call re-pauses them if the price is still below `highestRsethPrice` by more than `pricePercentageLimit`. This constitutes **temporary freezing of funds** — user deposits and withdrawals are inaccessible until either the price fully recovers above `highestRsethPrice` or the admin takes additional action (e.g., calling `setPricePercentageLimit(0)`).

---

### Likelihood Explanation

This is realistic in any scenario where a price drop triggers the auto-pause but the price only partially recovers. For example, with a 1% threshold: if `highestRsethPrice = 1.05 ETH` and the price drops to `1.02 ETH` (triggering auto-pause), then recovers to `1.04 ETH` (still 0.95% below peak — within threshold), the loop does not occur. But if it recovers to only `1.035 ETH` (still 1.4% below peak), the loop fires on every public call. This is a common real-world scenario for LST price fluctuations.

---

### Recommendation

Reset `highestRsethPrice` to the current `rsETHPrice` inside `unpause()`, so that the post-unpause price baseline reflects the actual current price rather than the pre-crash peak:

```solidity
function unpause() external whenPaused onlyLRTAdmin {
    paused = false;
    highestRsethPrice = rsETHPrice; // reset peak to current price
    emit Unpaused(msg.sender);
}
```

Alternatively, add a dedicated admin function `resetHighestRsethPrice()` to allow explicit resetting without coupling it to the unpause flow.

---

### Proof of Concept

**State-test sequence (local fork or unit test on unmodified code):**

```
Setup:
  highestRsethPrice = 1.05e18
  rsETHPrice        = 1.05e18
  pricePercentageLimit = 1e16  (1%)

Step 1 — Trigger auto-pause:
  Underlying oracle drops → newRsETHPrice = 1.02e18
  diff = 0.03e18 > 0.01 * 1.05e18 = 0.0105e18  → isPriceDecreaseOffLimit = true
  → lrtDepositPool.paused = true
  → withdrawalManager.paused = true
  → LRTOracle.paused = true
  → return (rsETHPrice stays 1.05e18, highestRsethPrice stays 1.05e18)

Step 2 — Admin unpauses all three contracts:
  LRTOracle.unpause()          → LRTOracle.paused = false
  lrtDepositPool.unpause()     → lrtDepositPool.paused = false
  withdrawalManager.unpause()  → withdrawalManager.paused = false
  highestRsethPrice            → still 1.05e18 (unchanged)

Step 3 — Underlying oracle partially recovers:
  newRsETHPrice = 1.035e18
  diff = 1.05e18 - 1.035e18 = 0.015e18
  threshold = 0.01 * 1.05e18 = 0.0105e18
  0.015e18 > 0.0105e18 → isPriceDecreaseOffLimit = true

Step 4 — Non-manager calls updateRSETHPrice():
  whenNotPaused passes (paused = false)
  → auto-pause fires again at lines 277-281
  → lrtDepositPool.paused = true, withdrawalManager.paused = true, LRTOracle.paused = true

Assert: protocol is paused again despite admin unpause and partial price recovery.
Loop: repeat from Step 2 indefinitely until price > 1.05e18 or admin calls setPricePercentageLimit(0).
```
