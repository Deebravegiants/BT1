### Title
Stale `rsETHPrice` After LST Upward Depeg Allows Excess Value Extraction via `instantWithdrawal` — (`contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTWithdrawalManager.instantWithdrawal()` computes the asset payout using the **stored/cached** `rsETHPrice` from `LRTOracle` combined with the **live** oracle price for the underlying LST. When an LST depegs upwards, `updateRSETHPrice()` is called and `rsETHPrice` is inflated to reflect the higher TVL. If the LST price then normalises before the next `updateRSETHPrice()` call, the stale inflated `rsETHPrice` is used against the now-lower live asset price, allowing a caller to extract more underlying LST than their rsETH is actually worth — draining value from remaining rsETH holders.

---

### Finding Description

`LRTOracle` stores `rsETHPrice` as a state variable that is only updated when `updateRSETHPrice()` / `_updateRsETHPrice()` is explicitly called. The internal `_getTotalEthInProtocol()` reads **live** oracle prices for each supported LST at the moment of the call. [1](#0-0) [2](#0-1) 

When `updateRSETHPrice()` is called while an LST (e.g. stETH) is trading at a premium (upward depeg), `newRsETHPrice` is computed from the inflated TVL and written to `rsETHPrice`. [3](#0-2) [4](#0-3) 

`instantWithdrawal` then computes the payout as:

```
assetAmountUnlocked = rsETHUnstaked * lrtOracle.rsETHPrice()   // ← stale, inflated
                    / lrtOracle.getAssetPrice(asset)            // ← live, now normalised
``` [5](#0-4) [6](#0-5) 

Unlike the queued-withdrawal path (`initiateWithdrawal` → `unlockQueue`), which caps the payout at `min(expectedAssetAmount, currentReturn)` via `_calculatePayoutAmount`, `instantWithdrawal` applies **no such cap** — it pays out the full computed amount directly. [7](#0-6) 

---

### Impact Explanation

**High — Theft of unclaimed yield / direct theft of funds from other rsETH holders.**

In the window between the LST price normalising and the next `updateRSETHPrice()` call, every `instantWithdrawal` call redeems rsETH at an inflated rate. The excess LST paid out comes from the pool's reserves, reducing the backing for all remaining rsETH holders. The attacker burns rsETH worth `X` ETH but receives LST worth `X + δ` ETH, where `δ` is proportional to the magnitude of the depeg and the amount withdrawn.

---

### Likelihood Explanation

**Medium.**

- LST upward depegs are historically observed (stETH has traded at a premium to ETH on secondary markets).
- `updateRSETHPrice()` is a public, permissionless function; any actor can call it while the price is inflated.
- The LST oracle price can normalise quickly (within the same block or next few blocks), creating a window before the next `updateRSETHPrice()` call.
- `instantWithdrawal` must be enabled per-asset by the manager (`isInstantWithdrawalEnabled[asset]`), which is a realistic operational configuration.
- No special privilege is required to call `instantWithdrawal`. [8](#0-7) 

---

### Recommendation

In `instantWithdrawal`, compute the payout using a **freshly computed** rsETH price (i.e., call `_getTotalEthInProtocol()` inline or require `updateRSETHPrice()` to have been called in the same block), rather than relying on the stored `rsETHPrice`. Alternatively, cap the payout at `min(rsETHUnstaked * rsETHPrice / assetPrice, rsETHUnstaked * freshRsETHPrice / assetPrice)` — analogous to the `_calculatePayoutAmount` guard already present in the queued-withdrawal path. A simpler mitigation is to require that `rsETHPrice` was updated within the current block before any `instantWithdrawal` is processed.

---

### Proof of Concept

1. Protocol holds 100 stETH. stETH oracle price = 1.000 ETH. `rsETHPrice` = 1.020 ETH (reflecting accumulated rewards). 100 rsETH outstanding.
2. stETH depegs upwards: Chainlink feed reports stETH = 1.050 ETH.
3. Anyone calls `updateRSETHPrice()`. `_getTotalEthInProtocol()` returns `100 × 1.050 = 105 ETH`. `newRsETHPrice = 105 / 100 = 1.050 ETH` (ignoring fee). `rsETHPrice` is written as `1.050 ETH`. [9](#0-8) 
4. stETH price normalises: Chainlink feed reports stETH = 1.000 ETH. `rsETHPrice` is still `1.050` (stale).
5. Attacker holds 10 rsETH (worth `10 × 1.000 = 10 ETH` at fair value, since stETH is back to 1.0).
6. Attacker calls `instantWithdrawal(stETH, 10 rsETH)`:
   - `assetAmountUnlocked = 10 × 1.050 / 1.000 = 10.5 stETH`
   - Attacker receives 10.5 stETH worth **10.5 ETH**.
   - Attacker burned rsETH worth only **10 ETH**.
   - **0.5 ETH stolen** from remaining rsETH holders. [10](#0-9) 
7. When `updateRSETHPrice()` is next called, it detects the TVL drop (now `89.5 × 1.000 = 89.5 ETH` for 90 rsETH outstanding), reducing `rsETHPrice` to `~0.994 ETH` — all remaining holders are diluted.

    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);
```
